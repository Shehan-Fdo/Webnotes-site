## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Secure SDLC and DevSecOps overview**

Secure Software Development Life Cycle (SDLC) and DevSecOps integrate security into software delivery instead of adding it after release. The goal is to reduce risk early, make security repeatable, and ensure that controls scale with modern delivery pipelines.

A secure approach covers:
- **Governance**: policies, standards, baselines, and procedures that define how software is built and released.
- **Design**: security requirements, threat modeling, privacy-by-design, and secure architecture choices.
- **Build and test**: code review, static analysis, dependency scanning, dynamic testing, and manual validation.
- **Release and operations**: signing, integrity verification, environment promotion, secrets control, monitoring, and rollback.

DevSecOps is not a separate SDLC; it is a way of embedding security into DevOps workflows so that security becomes automated, measurable, and shared across development, operations, and security teams.

**Governance artifacts and secure development policy**

Security governance gives the SDLC its rules. In practice, you define what must happen, who approves it, and what evidence proves compliance.

Key governance artifacts include:
- **Security policy**: high-level management intent, such as requiring secure coding, vulnerability remediation, and release approval.
- **Standards**: mandatory technical requirements, such as minimum password length, approved cryptographic algorithms, or required logging fields.
- **Baselines**: approved starting configurations, such as hardened container images or secure CI/CD runner settings.
- **Procedures**: step-by-step instructions, such as how to request a production deployment or how to rotate a secret.
- **Guidelines**: recommended practices that are flexible, such as preferred code review patterns or secure branching conventions.

These artifacts matter because they translate security objectives into enforceable controls. Without them, teams may implement inconsistent protections, and auditors cannot easily verify that security is repeatable.

A practical governance model often includes:
- Security requirements in the project charter or product backlog.
- Mandatory architecture and design reviews for high-risk changes.
- Defined approval gates for production releases.
- Exception handling for risk acceptance, with time limits and compensating controls.
- Evidence retention for audits, incident response, and compliance.

> A policy says what must be done; a standard says how it must be done; a procedure says how to do it step by step.

**Secure development methodologies and how security adapts**

Different development models change where and how you place security activities.

- **Waterfall**
  - Security activities are often phase-gated.
  - Requirements, design, implementation, testing, and release happen sequentially.
  - Security reviews fit naturally at requirements and design, with formal test phases later.
  - Risk: defects discovered late are expensive to fix.

- **Iterative and incremental**
  - Security work repeats each iteration.
  - You refine requirements, design, and tests as the product evolves.
  - Threat models and test cases should be updated continuously.

- **Spiral**
  - Risk-driven cycles make security a strong fit.
  - Each loop identifies and mitigates the highest risks first.
  - Useful for high-assurance or high-risk systems because security analysis is built into each cycle.

- **Agile**
  - Security is embedded in user stories, acceptance criteria, and sprint planning.
  - Security tasks become backlog items.
  - Definition of done includes security checks, such as code review, SAST, and test evidence.
  - Security champions help teams make secure decisions quickly.

- **DevOps / DevSecOps**
  - Security controls are automated in pipelines and infrastructure as code.
  - Feedback is fast, and controls are applied continuously.
  - Security teams provide guardrails, not just approvals.

A useful exam distinction is that the more iterative the model, the more security must be continuous and automated. In waterfall, security can be more formal and phase-based; in agile and DevSecOps, it must be lightweight, repeatable, and integrated into delivery.

**Secure design principles and patterns**

Security begins in design. Good design reduces the number of controls you need later and lowers the chance of systemic failure.

Important principles include:
- **Least privilege**: grant only the access required for the task.
- **Defense in depth**: use multiple layers so one failure does not expose the system.
- **Complete mediation**: check authorization every time a resource is accessed, not just once.
- **Fail secure**: when a system fails, it defaults to a safe state rather than an open one.
- **Secure defaults**: the system should be safe immediately after deployment, with no risky optional settings enabled by default.
- **Economy of mechanism**: keep designs simple to reduce attack surface and implementation errors.
- **Separation of duties**: split critical actions across roles to reduce fraud and misuse.
- **Minimize attack surface**: remove unnecessary services, ports, permissions, and code paths.
- **Psychological acceptability**: security controls should be usable enough that users do not bypass them.

Common secure design patterns:
- **Input validation and output encoding** to reduce injection and cross-site scripting risk.
- **Centralized authentication and authorization** to avoid inconsistent access decisions.
- **Tokenization or encryption** for sensitive data at rest and in transit.
- **Brokered access** through APIs or gateways instead of direct database exposure.
- **Immutable infrastructure** where servers are replaced rather than patched in place, improving consistency.

**Requirements engineering and privacy-by-design**

Security requirements should be explicit, testable, and traceable. If a requirement cannot be verified, it is hard to enforce.

Good security requirements:
- State the asset or data type being protected.
- Define the control objective.
- Include measurable acceptance criteria.
- Map to a risk, regulation, or business need.

Examples:
- “Customer PII must be encrypted in transit using TLS 1.2+ and at rest using approved encryption.”
- “Administrative actions must be logged with user ID, timestamp, source IP, and action outcome.”
- “Secrets must not be stored in source code, build logs, or tickets.”

Privacy-by-design is closely related to security-by-design. It emphasizes:
- **Data minimization**: collect only what you need.
- **Purpose limitation**: use data only for the stated purpose.
- **Retention limits**: delete or anonymize data when it is no longer needed.
- **Access limitation**: restrict who can view personal data.
- **Transparency**: document how data is used and protected.

These principles reduce legal exposure and limit the blast radius of a breach.

**Threat modeling and design review**

Threat modeling identifies likely attack paths before code is written or before major changes are released. It is one of the highest-value security activities because it informs architecture, controls, and tests.

Common approaches include:
- **STRIDE**: Spoofing, Tampering, Repudiation, Information disclosure, Denial of service, Elevation of privilege.
- **Attack trees**: break a goal into subgoals and attack paths.
- **Data flow diagrams**: show trust boundaries, data stores, and external entities.

A practical threat modeling workflow:
1. Identify assets, trust boundaries, and entry points.
2. Enumerate threats against each component and data flow.
3. Rank risks by likelihood and impact.
4. Define mitigations and assign owners.
5. Convert mitigations into backlog items and test cases.

Threat modeling should be repeated when architecture changes, new dependencies are added, or sensitive data flows change.

**Secure coding practices and code review**

Secure coding reduces defects before they reach testing or production. It is most effective when paired with review and automation.

Core practices:
- Validate all input and encode all output.
- Use parameterized queries to prevent SQL injection.
- Avoid hard-coded credentials and secrets.
- Handle errors safely without leaking stack traces or sensitive data.
- Use approved cryptographic libraries and modes.
- Apply secure memory handling where relevant.
- Log security-relevant events without exposing secrets or personal data.

Code review should look for:
- Authentication and authorization logic errors.
- Unsafe deserialization.
- Injection risks.
- Insecure direct object references.
- Weak randomness.
- Missing error handling.
- Race conditions and concurrency issues.
- Unsafe use of third-party libraries.

A secure review process often combines:
- Peer review for logic and maintainability.
- Security review for high-risk changes.
- Mandatory review for sensitive modules, such as auth, payments, and key management.

**Automated testing: SAST, DAST, SCA, and more**

Automation gives you scale and consistency, but it does not replace human judgment.

Common tools and what they find:
- **SAST** (Static Application Security Testing): analyzes source code or binaries without running the app; good for insecure patterns and taint flows.
- **DAST** (Dynamic Application Security Testing): tests a running application from the outside; good for runtime issues and exposed endpoints.
- **SCA** (Software Composition Analysis): identifies vulnerable open-source and third-party components.
- **IAST** (Interactive Application Security Testing): combines runtime instrumentation with testing to improve accuracy.
- **Fuzzing**: sends malformed or unexpected inputs to find crashes and parsing flaws.
- **Secrets scanning**: detects credentials, API keys, and tokens in code and history.

A balanced pipeline uses multiple methods because each finds different classes of defects. For example, SAST may catch insecure coding patterns early, SCA may flag a vulnerable library, and DAST may reveal an authentication bypass in a deployed test environment.

Example pipeline stage:
```yaml
stages:
  - lint
  - test
  - sast
  - sca
  - build
  - sign
  - deploy_test
  - dast
  - approve_release
```

Example SAST and dependency scanning commands:
```bash
semgrep --config=auto src/
npm audit --production
pip-audit
trivy fs .
```

**Manual security testing and penetration testing**

Automated tools are not enough. Manual testing is essential for business logic flaws, chained vulnerabilities, and abuse cases that scanners miss.

Use manual testing for:
- Authentication and session management weaknesses.
- Authorization bypass and privilege escalation.
- Workflow abuse, such as coupon fraud or transaction manipulation.
- Multi-step attacks that require human reasoning.
- Validation of compensating controls and edge cases.

Penetration testing is especially useful:
- Before major releases.
- After significant architecture changes.
- For internet-facing systems.
- When regulatory or contractual requirements demand independent validation.

Manual testing complements automation; it does not replace it. A strong program uses both.

**Secrets management and credential hygiene**

Secrets are one of the most common causes of pipeline compromise. Treat them as sensitive assets with lifecycle controls.

Best practices:
- Store secrets in a dedicated vault or secret manager.
- Use short-lived credentials where possible.
- Rotate secrets regularly and immediately after suspected exposure.
- Grant CI/CD systems only the minimum permissions needed.
- Separate human credentials from machine credentials.
- Avoid storing secrets in source code, container images, tickets, chat, or logs.
- Mask secrets in build output and application logs.
- Use environment-specific secrets rather than shared global values.

Example with HashiCorp Vault:
```bash
vault kv put secret/payments api_key="REDACTED"
vault kv get secret/payments
```

Example Kubernetes secret reference:
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app
spec:
  containers:
    - name: app
      image: example/app:1.0
      env:
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: db-secret
              key: password
```

Important controls for CI/CD credentials:
- Use workload identity or federated identity instead of long-lived keys.
- Scope tokens to a single repository, environment, or job.
- Revoke credentials automatically after job completion when possible.
- Prevent secrets from being echoed in shell scripts or debug logs.

> If a secret reaches a repository history or public log, assume it is compromised and rotate it immediately.

**Build integrity, code signing, and release signing**

Integrity controls prove that code and artifacts have not been altered. This is critical in modern supply chains where build systems, registries, and deployment targets are all potential attack points.

Key concepts:
- **Code signing**: signs source code, commits, or release packages to prove origin and integrity.
- **Build signing**: signs compiled artifacts, containers, or packages produced by the build system.
- **Release signing**: signs the final release bundle or manifest before deployment.
- **Verification**: checks the signature before use or promotion.

Practical uses:
- Verify that a package came from the expected publisher.
- Ensure a container image matches the approved build output.
- Block deployment of tampered artifacts.

Example GPG signing:
```bash
gpg --armor --detach-sign release.tar.gz
gpg --verify release.tar.gz.asc release.tar.gz
```

Example container image signing with Cosign:
```bash
cosign sign ghcr.io/example/app:1.2.3
cosign verify ghcr.io/example/app:1.2.3
```

For stronger assurance, use:
- Protected signing keys in hardware security modules (HSMs) or managed key services.
- Separate build and signing steps.
- Immutable artifact repositories.
- Provenance metadata such as build attestations.

**Supply chain security and third-party risk**

Modern applications depend heavily on third-party code, managed services, and external suppliers. Secure procurement is therefore part of secure SDLC.

Supplier and procurement controls should include:
- Security due diligence before onboarding a vendor.
- Contract clauses for vulnerability disclosure, patch timelines, breach notification, and audit rights.
- Requirements for secure development practices and evidence of testing.
- Review of data handling, subcontractors, and geographic processing locations.
- Validation of software provenance and update mechanisms.
- Ongoing monitoring of supplier risk.

For software and services, ask:
- Does the vendor publish security advisories and support patching?
- Are updates signed and verifiable?
- Does the service support logging, access control, and encryption?
- What data does the supplier store, process, or transmit?
- Can you terminate access and retrieve or delete data securely?

Third-party libraries and managed services should be tracked in a software bill of materials (SBOM) where possible. An SBOM helps you identify exposure when a new vulnerability affects a dependency.

**Container and image supply chain security**

Containers add speed, but they also add new trust decisions. You must control what goes into the image, where it comes from, and whether it is allowed to run.

Important controls:
- Use trusted base images from reputable sources.
- Pin image versions or digests instead of floating tags.
- Scan images for vulnerabilities and misconfigurations.
- Remove unnecessary packages and tools from runtime images.
- Build images in isolated, reproducible pipelines.
- Sign images and verify signatures before deployment.
- Restrict registry access and harden the registry itself.
- Use admission control to block unsigned or unapproved images.

Example Dockerfile hardening:
```dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
USER 10001
CMD ["python", "app.py"]
```

Example Kubernetes admission policy concept:
- Allow only images signed by the organization.
- Deny images from unapproved registries.
- Require vulnerability scan results below a defined threshold.

**Environment separation and promotion controls**

Environment separation reduces the chance that test activity affects production and limits the spread of sensitive data.

Typical environments:
- **Development**: rapid change, limited data, developer access.
- **Test/QA**: functional and security testing, controlled test data.
- **Staging/UAT**: production-like validation, release rehearsal, business sign-off.
- **Production**: live customer data and strict access controls.

Promotion controls should ensure:
- Artifacts move forward, not rebuilt differently in each environment.
- The same signed artifact is promoted from test to stage to prod.
- Access to production is restricted and logged.
- Test data is sanitized or synthetic.
- Configuration differences are controlled and documented.

A common anti-pattern is “works in dev, rebuilt in prod.” That breaks traceability and weakens integrity. Prefer immutable artifacts and environment-specific configuration injected at deploy time.

Example deployment promotion flow:
```text
commit -> build -> scan -> sign -> deploy dev -> test -> deploy stage -> approve -> deploy prod
```

**Release management and change control**

Secure release management ensures that only approved, tested, and traceable changes reach production.

Controls include:
- Change requests for significant releases.
- Approval gates based on risk and impact.
- Separation of duties between developer, tester, and releaser.
- Rollback plans and backout procedures.
- Maintenance windows for high-risk changes.
- Emergency change process with post-implementation review.

Release criteria often include:
- No critical unresolved vulnerabilities.
- Required tests passed.
- Security sign-off completed for sensitive changes.
- Artifact signatures verified.
- Monitoring and rollback plans in place.

**Security metrics, KPIs, and KRIs**

Metrics help you measure whether the SDLC is actually improving security. Use them to manage performance and maturity, not just to collect numbers.

Common measures:
- **KPI** (Key Performance Indicator): measures performance against a goal.
- **KRI** (Key Risk Indicator): measures exposure or increasing risk.

Useful SDLC/DevSecOps metrics:
- Mean time to remediate vulnerabilities.
- Percentage of builds passing security gates.
- Number of critical findings per release.
- Percentage of repositories with secret scanning enabled.
- Dependency update lag for critical libraries.
- Percentage of production deployments using signed artifacts.
- Number of exceptions or risk acceptances older than policy allows.
- Coverage of threat modeling for high-risk applications.

Interpret metrics carefully:
- A low number of findings may mean good security or weak scanning.
- A high number of blocked releases may indicate either strong controls or poor upstream quality.
- Trend lines are often more useful than single values.

**End-to-end workflow example**

A strong secure SDLC turns a finding into a controlled action, not just a report.

Example flow:
1. During design review, you identify that a payment API exposes sensitive customer identifiers.
2. You record the issue as a risk and create a backlog item with acceptance criteria.
3. The developer implements tokenization and authorization checks.
4. A peer review confirms the fix and checks for regression risk.
5. SAST and SCA run in the pipeline and pass.
6. DAST validates the endpoint in staging.
7. A manual tester verifies that unauthorized users cannot enumerate identifiers.
8. The release artifact is signed and promoted through controlled environments.
9. Production deployment requires approval and signature verification.
10. Monitoring watches for abnormal access patterns after release.

This workflow shows how security moves from design to backlog to code to test to release gate. That is the core DevSecOps mindset.

**Practical CI/CD example**

A simple GitHub Actions-style pipeline can enforce several controls at once:

```yaml
name: secure-build

on:
  push:
    branches: [main]

jobs:
  build-test-scan:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      id-token: write
    steps:
      - uses: actions/checkout@v4

      - name: Run unit tests
        run: pytest -q

      - name: Static analysis
        run: semgrep --config=auto .

      - name: Dependency scan
        run: pip-audit

      - name: Build image
        run: docker build -t example/app:${{ github.sha }} .

      - name: Sign image
        run: cosign sign example/app:${{ github.sha }}

      - name: Deploy to staging
        run: ./deploy.sh staging
```

This example is simplified, but it illustrates the principle: security checks are part of the pipeline, not a separate manual afterthought.

**Common exam distinctions**

- **Secure SDLC** focuses on embedding security into each development phase.
- **DevSecOps** emphasizes automation, collaboration, and continuous feedback.
- **SAST** analyzes code without execution; **DAST** tests a running app.
- **Code signing** proves origin and integrity; **encryption** protects confidentiality.
- **Policy** sets direction; **standard** sets mandatory specifics; **procedure** tells you how to execute.
- **Pen testing** complements automated scanning; it does not replace it.
- **Environment separation** and **promotion controls** protect release integrity and reduce production risk.

**Quick reference checklist**

- Define security policies, standards, baselines, and procedures.
- Map security activities to the chosen development model.
- Perform threat modeling for high-risk changes.
- Write testable security and privacy requirements.
- Use secure coding standards and mandatory review.
- Run SAST, DAST, SCA, secrets scanning, and fuzzing where appropriate.
- Include manual security testing and penetration testing for critical systems.
- Store secrets in a vault and rotate them regularly.
- Sign code, builds, and releases; verify signatures before deployment.
- Control third-party risk through procurement and supplier review.
- Harden container supply chains and verify image provenance.
- Separate dev, test, stage, and prod with controlled promotion.
- Track KPIs and KRIs to measure effectiveness and maturity.
