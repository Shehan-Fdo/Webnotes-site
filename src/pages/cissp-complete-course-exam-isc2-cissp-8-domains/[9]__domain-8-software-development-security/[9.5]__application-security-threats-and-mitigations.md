## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Application security threat landscape**

Application security focuses on the risks introduced by software design, implementation, deployment, and integration. For CISSP, you need to recognize common attack classes, understand why they happen, and map them to preventive, detective, and corrective controls.

A useful exam mindset is to ask:
- What trust boundary is crossed?
- What input is attacker-controlled?
- What data is exposed, modified, or executed?
- Which control fails: validation, authorization, session handling, cryptography, or supply chain trust?

**Secure coding principles**

Secure development is not just about fixing bugs; it is about designing systems that fail safely and resist misuse.

- **Fail-safe defaults** mean access is denied unless explicitly allowed.
- **Complete mediation** means every access to a resource is checked, not just the first one.
- **Separation of duties** reduces the chance that one compromise can change code, approve it, and deploy it.
- **Defense in depth** layers controls so one failure does not become a breach.
- **Least privilege** limits what code, services, and identities can do.
- **Minimize attack surface** by removing unused endpoints, features, libraries, and permissions.

These principles apply directly to application controls such as authorization checks, input validation, secure session handling, and deployment pipelines.

**Common application security weaknesses**

Many exam questions map to well-known web and software flaws. You should be able to identify the weakness, the likely impact, and the best mitigation.

- **Injection** occurs when untrusted input changes the meaning of a command or query.
  - SQL injection targets databases.
  - OS command injection targets the shell.
  - LDAP injection, NoSQL injection, and XPath injection follow the same pattern.
  - Mitigate with parameterized queries, allowlists, context-aware escaping, and least-privilege database accounts.

- **Cross-site scripting (XSS)** lets attacker-controlled script run in a victim’s browser.
  - Stored XSS persists in the application.
  - Reflected XSS returns input immediately.
  - DOM-based XSS happens in client-side code.
  - Mitigate with output encoding, input validation, safe DOM APIs, and a strong Content Security Policy (CSP).

- **Cross-site request forgery (CSRF)** tricks a logged-in user’s browser into sending an unwanted request.
  - Mitigate with anti-CSRF tokens, same-site cookies, origin checks, and reauthentication for sensitive actions.

- **Insecure deserialization** happens when untrusted serialized data is accepted and object creation triggers dangerous behavior.
  - Risks include remote code execution, privilege escalation, and logic abuse.
  - Mitigate by avoiding native deserialization of untrusted data, using signed or authenticated payloads, allowlisting types, and disabling dangerous gadget chains.

- **XML External Entity (XXE)** attacks exploit XML parsers that resolve external entities.
  - Impacts include file disclosure, server-side request forgery (SSRF), and denial of service.
  - Mitigate by disabling external entity resolution, using secure parser settings, and preferring safer data formats when possible.

- **Path traversal** uses `../` sequences or encoded variants to escape intended directories.
  - Impacts include reading configuration files, keys, or source code.
  - Mitigate with canonicalization, allowlisted file names, sandboxing, and strict path resolution.

- **File inclusion vulnerabilities** allow attacker-controlled file paths to be loaded.
  - **Local file inclusion (LFI)** loads local files and may lead to code execution if combined with upload or log poisoning.
  - **Remote file inclusion (RFI)** loads remote content and is often more severe.
  - Mitigate by avoiding dynamic includes, using allowlists, and disabling remote includes.

- **Template injection** occurs when attacker input is interpreted by a template engine.
  - Server-side template injection can lead to data exposure or code execution.
  - Mitigate by separating data from template logic, escaping output, and avoiding unsafe evaluation features.

- **Buffer overflow** remains relevant in native code and embedded systems.
  - It can overwrite memory and alter control flow.
  - Mitigate with bounds checking, safe languages where possible, compiler protections, ASLR, DEP, and stack canaries.

- **Race conditions** occur when timing affects security decisions.
  - Examples include TOCTOU (time-of-check to time-of-use) file access issues.
  - Mitigate with atomic operations, locking, and rechecking critical state at use time.

- **Business logic flaws** abuse intended functionality rather than technical bugs.
  - Examples include coupon abuse, payment manipulation, and workflow skipping.
  - Mitigate with server-side validation, fraud controls, and abuse testing.

**Authentication threats and account abuse**

Authentication attacks focus on gaining access by guessing, reusing, or coercing credentials and factors.

- **Brute force** tries many passwords against one account.
  - Detect with failed login spikes and lockout patterns.
  - Mitigate with rate limiting, progressive delays, MFA, and strong password policy.

- **Password spraying** tries a few common passwords across many accounts.
  - This avoids lockout thresholds that target one account.
  - Detect with distributed failures across many users from shared sources or patterns.
  - Mitigate with MFA, breached-password checks, and anomaly detection.

- **Credential stuffing** uses stolen username/password pairs from other breaches.
  - It succeeds when users reuse passwords.
  - Mitigate with MFA, bot detection, breached credential monitoring, and passwordless or phishing-resistant authentication.

- **Account enumeration** reveals whether an account exists.
  - Signals include different error messages, timing, password reset behavior, or MFA prompts.
  - Mitigate with uniform responses, consistent timing, and generic recovery flows.

- **MFA fatigue / push bombing** overwhelms users with repeated push prompts until one is approved.
  - Mitigate with number matching, phishing-resistant MFA, rate limits on prompts, and user training.

- **Session hijacking** steals or reuses a valid session token.
  - Mitigate with secure cookie flags, short lifetimes, reauthentication, device binding where appropriate, and token rotation.

- **Replay attacks** reuse captured authentication material.
  - Mitigate with nonces, timestamps, one-time tokens, and mutual challenge-response where appropriate.

**Session and token storage tradeoffs**

Where you store session state depends on the application type and threat model.

- **Browser-based traditional web apps**
  - Prefer server-side sessions with opaque session IDs in `HttpOnly`, `Secure`, `SameSite` cookies.
  - `HttpOnly` reduces theft via XSS because JavaScript cannot read the cookie.
  - `SameSite=Lax` or `Strict` helps reduce CSRF risk.
  - Avoid storing long-lived bearer tokens in `localStorage` because XSS can read them.

- **Single-page applications (SPAs)**
  - SPAs often use access tokens and refresh tokens.
  - Storing tokens in `localStorage` is convenient but increases XSS impact.
  - Storing refresh tokens in `HttpOnly` cookies can reduce exposure, but you must manage CSRF and token rotation carefully.
  - A common pattern is short-lived access tokens in memory and refresh tokens in secure cookies.

- **Mobile apps**
  - Use platform secure storage such as iOS Keychain or Android Keystore-backed storage.
  - Avoid plain text files, shared preferences, or embedded secrets.
  - Assume rooted or jailbroken devices may expose local secrets; design for token revocation and short lifetimes.

- **API clients and service-to-service apps**
  - Use short-lived tokens, mutual TLS where appropriate, and secret managers for long-term credentials.
  - Prefer workload identity over static API keys when possible.

> If you must choose between cookie storage and browser storage, remember the tradeoff: cookies reduce script access but can increase CSRF exposure; web storage avoids automatic sending but is easier to steal with XSS.

**Authorization failures in applications and APIs**

Authorization is often the real failure behind “authentication-looking” incidents.

- **Broken object level authorization (BOLA)** occurs when a user can access another user’s object by changing an identifier.
  - Example: `/api/orders/12345` returns another customer’s order.
  - Mitigate with per-object authorization checks on every request.

- **Broken function level authorization (BFLA)** occurs when a user can call an administrative or privileged function.
  - Example: a normal user invokes `/api/admin/export`.
  - Mitigate with role-based access control (RBAC), attribute-based access control (ABAC), and server-side enforcement.

- **Excessive data exposure** happens when APIs return more fields than the client needs.
  - Example: a profile endpoint returns password hashes, internal flags, or full PII.
  - Mitigate with response shaping, field-level authorization, and least-data principles.

- **Mass assignment / over-posting** occurs when the API binds user input to sensitive fields.
  - Example: a user sets `isAdmin=true` in a JSON payload.
  - Mitigate with explicit DTOs (data transfer objects), allowlisted fields, and server-side validation.

- **Insecure direct object references (IDOR)** are a common symptom of missing object-level authorization.
  - Treat IDOR as an authorization bug, not just an input validation issue.

**API-specific threats and controls**

APIs need security controls beyond traditional web pages because they are machine-to-machine, highly automated, and often expose structured data.

- Validate requests against an **API schema** such as OpenAPI.
  - Reject unknown fields when possible.
  - Enforce types, ranges, formats, and required fields.
  - Use schema validation at the gateway and in the application.

- Apply **rate limiting** and abuse detection.
  - Protect login, password reset, OTP, and search endpoints.
  - Use per-user, per-IP, and per-token controls.

- Use **authentication** and **authorization** separately.
  - Authentication proves identity.
  - Authorization decides what that identity can do.

- Protect against **excessive data exposure**.
  - Return only the fields needed by the client.
  - Use separate endpoints or projections for different roles.

- Prevent **BOLA** and **BFLA** with server-side checks on every object and function.
  - Do not trust client-side role claims alone.
  - Re-evaluate permissions after token refresh or privilege changes.

- Secure API tokens.
  - Use short-lived access tokens.
  - Rotate refresh tokens.
  - Revoke compromised tokens quickly.

- Log security-relevant API events.
  - Include subject, object, action, result, and correlation ID.
  - Avoid logging secrets or full sensitive payloads.

**Cryptographic application flaws**

Cryptography fails most often through poor implementation, poor key handling, or misuse of algorithms.

- **Weak random number generation** undermines tokens, session IDs, salts, and keys.
  - Use cryptographically secure random number generators (CSPRNGs).
  - Never use predictable values like timestamps, counters, or `Math.random()` for security tokens.

- **Hardcoded secrets** expose passwords, API keys, and signing keys in source code or binaries.
  - Mitigate with secret managers, environment injection, and build-time scanning.
  - Rotate exposed secrets immediately.

- **Improper key management** includes weak storage, poor rotation, and excessive access.
  - Store keys in hardware security modules (HSMs), key management services (KMSs), or protected vaults.
  - Separate key usage from key administration.
  - Rotate keys based on policy and compromise risk.

- **Insecure storage of sensitive data** includes plaintext storage of passwords, tokens, or personal data.
  - Hash passwords with a modern adaptive algorithm such as bcrypt, scrypt, Argon2, or PBKDF2 with appropriate parameters.
  - Encrypt sensitive data at rest when needed, but remember encryption does not replace access control.
  - Minimize retention and mask data in logs and backups.

- **Weak or deprecated algorithms** such as MD5, SHA-1 for signatures, or obsolete ciphers can fail under attack.
  - Use approved modern algorithms and modes.
  - Prefer authenticated encryption such as AES-GCM or ChaCha20-Poly1305 where appropriate.

- **Improper certificate validation** can enable man-in-the-middle attacks.
  - Validate the full chain, hostname, and revocation requirements where applicable.
  - Do not disable TLS verification in production.

**Content Security Policy and browser-side defenses**

CSP is a browser control that reduces the impact of XSS and some injection issues, but it is not a substitute for secure coding.

A practical CSP often includes:
- `default-src 'self'`
- `script-src 'self' 'nonce-<random>'`
- `object-src 'none'`
- `base-uri 'self'`
- `frame-ancestors 'none'`
- `img-src 'self' data:`
- `connect-src 'self' https://api.example.com`

Example header:
```http
Content-Security-Policy: default-src 'self'; script-src 'self' 'nonce-r4nd0m'; object-src 'none'; base-uri 'self'; frame-ancestors 'none'; connect-src 'self' https://api.example.com
```

Nonce-based usage:
```html
<script nonce="r4nd0m">
  initApp();
</script>
```

Hash-based usage:
```http
Content-Security-Policy: script-src 'self' 'sha256-Base64EncodedHashHere'
```

Important limitations:
- CSP does not fix server-side injection.
- CSP is weaker if you allow `unsafe-inline` or broad wildcards.
- CSP can be bypassed if the application already allows script injection through trusted domains or JSONP-like patterns.
- CSP should be paired with output encoding, input validation, and secure templating.

**Secure serialization and deserialization controls**

Serialization converts objects into a storable or transferable format; deserialization reconstructs them. The risk appears when untrusted data is accepted and object behavior is triggered.

- Prefer simple, explicit formats such as JSON with strict schema validation.
- Avoid deserializing arbitrary object graphs from untrusted sources.
- Use allowlists for permitted classes or types.
- Sign or authenticate serialized data when integrity matters.
- Disable polymorphic type resolution unless absolutely required.
- Treat deserialization endpoints as high risk if they accept files, cookies, or message queue payloads.

A safe pattern is to deserialize into a plain data structure, validate it, and then map it into application objects.

**Software supply chain controls**

Application security includes the code you write and the code you import.

- **Dependency scanning** identifies vulnerable libraries and transitive dependencies.
  - Set policy thresholds for critical and high severity issues.
  - Decide whether to block builds, warn, or require exception approval.
  - Re-scan regularly because new CVEs (Common Vulnerabilities and Exposures) appear after release.

- **Repository trust controls** reduce tampering.
  - Restrict who can merge to protected branches.
  - Require code review for sensitive paths.
  - Use signed commits or signed tags where supported.
  - Verify provenance for build artifacts when possible.

- **CI/CD secret scanning** detects leaked credentials in source, logs, and pipeline variables.
  - Scan pull requests and main branches.
  - Block merges when secrets are detected.
  - Rotate any exposed secret immediately.

- **Build integrity** matters.
  - Use reproducible or controlled builds where possible.
  - Pin dependencies and verify checksums.
  - Separate build, test, and release permissions.

- **Third-party risk** extends to packages, containers, and plugins.
  - Review package maintainers, update cadence, and trust model.
  - Prefer minimal dependencies.
  - Remove unused packages and transitive bloat.

**Practical secure coding examples**

Parameterized SQL query:
```python
cursor.execute(
    "SELECT id, email FROM users WHERE id = ?",
    (user_id,)
)
```

Unsafe string concatenation:
```python
query = "SELECT * FROM users WHERE id = " + user_id
```

Allowlisted file access:
```python
allowed = {"report.pdf", "invoice.pdf"}
if filename not in allowed:
    raise ValueError("Invalid file")
path = f"/srv/downloads/{filename}"
```

Secure random token generation:
```python
import secrets
token = secrets.token_urlsafe(32)
```

Password hashing:
```python
from argon2 import PasswordHasher
ph = PasswordHasher()
hash_value = ph.hash(password)
```

**Detection, prevention, and response mapping**

For exam questions, map each threat to the right control type.

- **Preventive controls**
  - Input validation
  - Output encoding
  - Parameterized queries
  - MFA
  - Secure defaults
  - Least privilege
  - CSP
  - Schema validation
  - Secret management
  - Secure coding standards

- **Detective controls**
  - WAF (Web Application Firewall) alerts
  - SIEM (Security Information and Event Management) correlation
  - Log analysis
  - Anomaly detection
  - Dependency scanning
  - Secret scanning
  - Code review
  - SAST/DAST/IAST results

- **Corrective controls**
  - Patch vulnerable libraries
  - Rotate keys and tokens
  - Revoke sessions
  - Roll back bad deployments
  - Disable exposed endpoints
  - Force password resets when compromise is confirmed

**Exam-oriented threat-to-control mapping**

| Threat | Primary risk | Best preventive control | Helpful detective control |
|---|---|---|---|
| SQL injection | Unauthorized database access | Parameterized queries | Query anomaly alerts |
| XSS | Script execution in browser | Output encoding, CSP | Client-side error monitoring |
| CSRF | Unauthorized state change | Anti-CSRF tokens, SameSite cookies | Unusual action logs |
| BOLA | Unauthorized object access | Per-object authorization | Access pattern review |
| BFLA | Unauthorized privileged function | Server-side RBAC/ABAC | Admin action auditing |
| Credential stuffing | Account takeover | MFA, breached-password checks | Bot detection |
| Password spraying | Distributed login abuse | Rate limiting, MFA | Failure trend analysis |
| Insecure deserialization | RCE or logic abuse | Avoid unsafe deserialization | Payload inspection |
| XXE | File disclosure, SSRF | Disable external entities | Parser error monitoring |
| Path traversal | File disclosure | Canonicalization, allowlists | File access logs |
| Template injection | Code execution or data leak | Safe templating, escaping | WAF and app logs |
| Hardcoded secrets | Secret exposure | Secret manager, scanning | Repo and CI secret scans |

**Real-world deployment guidance**

Security controls work best when they are layered into the development lifecycle.

1. Define secure coding standards and threat modeling requirements early.
2. Enforce code review for security-sensitive changes.
3. Run SAST (Static Application Security Testing), dependency scanning, and secret scanning in CI.
4. Add DAST (Dynamic Application Security Testing) against staging environments.
5. Protect production with WAF rules, rate limits, and strong logging.
6. Rotate secrets and patch dependencies on a defined schedule.
7. Re-test after every major change or incident.

**Common exam traps**

- A WAF is useful, but it does not replace secure code.
- Encryption at rest does not fix broken authorization.
- MFA reduces account takeover risk, but it does not stop all session theft or push fatigue attacks by itself.
- `HttpOnly` cookies reduce JavaScript theft, but they do not eliminate CSRF.
- Input validation helps, but output encoding is still required for XSS defense.
- Rate limiting helps against brute force, but password spraying and credential stuffing need broader detection and MFA.
- A secure API needs both schema validation and authorization checks; one without the other is incomplete.

**Quick memory anchors**

- **Input validation** stops malformed data early.
- **Output encoding** protects the interpreter that renders data.
- **Authorization** decides who can do what.
- **Authentication** proves who you are.
- **Cryptography** protects data, but only when keys and randomness are managed correctly.
- **Supply chain security** protects the code you did not write.
- **Defense in depth** is the default answer when one control is not enough.
