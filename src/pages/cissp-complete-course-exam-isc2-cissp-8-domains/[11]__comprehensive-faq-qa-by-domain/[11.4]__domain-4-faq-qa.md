## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Domain 4 overview**

Domain 4 focuses on communications and network security: how you design, protect, monitor, and troubleshoot networks without assuming the network itself is trustworthy. For CISSP, you need both conceptual understanding and practical recognition of where controls sit in the stack.

A useful exam mindset is to ask:
- What is the control protecting?
- Where is it placed in the network path?
- Does it prevent, detect, or respond?
- Does it protect confidentiality, integrity, availability, or all three?

**Q&A: Core network security devices**

**Q: What does a firewall do?**

A firewall enforces policy between network zones. It can allow, deny, or inspect traffic based on addresses, ports, protocols, application identity, or session state.

Common firewall types:
- **Packet-filtering firewall**: checks header fields such as source/destination IP, port, and protocol.
- **Stateful firewall**: tracks connection state and allows return traffic for established sessions.
- **Next-generation firewall (NGFW)**: adds application awareness, intrusion prevention, and often user or content controls.
- **Web application firewall (WAF)**: protects HTTP/HTTPS applications by filtering malicious web requests such as SQL injection or cross-site scripting.
- **Host-based firewall**: runs on an endpoint or server and protects that system locally.

Placement matters:
- Perimeter firewalls protect the boundary between trusted and untrusted networks.
- Internal firewalls support segmentation between business units, tiers, or trust zones.
- Host firewalls protect workloads even if the network perimeter is bypassed.

> A firewall is not automatically a security solution just because it exists. Its value depends on rule quality, logging, review, and placement.

**Q: How is a proxy different from a firewall?**

A proxy sits between client and server and relays traffic on behalf of one side. It can hide internal clients, enforce content policy, cache responses, and inspect application traffic.

Proxy types:
- **Forward proxy**: protects clients by mediating outbound access.
- **Reverse proxy**: protects servers by mediating inbound access to web applications.
- **Transparent proxy**: intercepts traffic without explicit client configuration.

Why you use a proxy:
- To centralize authentication and content filtering.
- To reduce direct exposure of internal hosts.
- To cache frequently requested content.
- To terminate TLS and inspect application-layer traffic where policy allows.

**Q: What is a WAF and when do you use it?**

A WAF protects web applications specifically. It understands HTTP methods, headers, cookies, and request bodies better than a generic firewall.

Use a WAF when:
- You expose web apps to untrusted users.
- You need compensating controls for legacy applications.
- You want to block common web attacks before they reach the app.

A WAF does not replace secure coding, input validation, or patching.

**Q: What is a load balancer?**

A load balancer distributes traffic across multiple servers to improve availability and performance.

Common functions:
- **Health checks** to remove failed nodes from rotation.
- **Session persistence** when an application requires sticky sessions.
- **TLS termination** to offload encryption from back-end servers.
- **Layer 4 or Layer 7 balancing** depending on whether it routes by transport or application data.

Exam point:
- A load balancer improves availability, but it is not a substitute for redundancy in the application, storage, and identity layers.

**Q: What does a content filter do?**

A content filter blocks or allows traffic based on content categories, file types, URLs, keywords, or reputation.

Typical uses:
- Blocking malicious downloads.
- Enforcing acceptable use policies.
- Reducing phishing and malware exposure.
- Controlling web access by category.

Content filters often appear in secure web gateways, email security gateways, and DNS filtering platforms.

**Q: What is a secure web gateway?**

A secure web gateway inspects outbound web traffic and enforces policy for web access. It often combines URL filtering, malware scanning, TLS inspection, and data loss prevention features.

**Q: How does a network access control system help?**

Network access control (NAC) checks whether a device meets policy before or after it joins the network.

NAC can:
- Authenticate users and devices.
- Check posture such as patch level or antivirus status.
- Place noncompliant devices into a remediation VLAN.
- Enforce role-based access to network segments.

**Q: What is the difference between IDS and IPS?**

An intrusion detection system (IDS) detects suspicious activity and alerts. An intrusion prevention system (IPS) detects and blocks suspicious activity inline.

Key difference:
- IDS is usually passive.
- IPS is usually inline and can drop traffic.

CISSP angle:
- IPS can affect availability if it blocks legitimate traffic.
- IDS supports visibility and incident response.

**Q: Where do honeypots fit?**

A honeypot is a decoy system designed to attract attackers and generate alerts. It is useful for detection, research, and deception, but it is not a primary control for production protection.

**Q&A: Segmentation and network design**

**Q: Why is segmentation important?**

Segmentation limits lateral movement, reduces attack surface, and helps enforce least privilege across network zones.

Common segmentation methods:
- **VLANs** for logical separation at Layer 2.
- **Subnets** for IP-based separation.
- **DMZs** for public-facing services.
- **Microsegmentation** for workload-level policy, often in virtualized or cloud environments.
- **Air gaps** for extreme isolation where connectivity is intentionally absent.

**Q: What is a VLAN?**

A VLAN (Virtual Local Area Network) creates separate broadcast domains on the same physical switching infrastructure.

Benefits:
- Reduces broadcast traffic.
- Separates departments or trust zones.
- Supports cleaner policy enforcement.

Important exam detail:
- A VLAN is not security by itself. If routing and ACLs permit it, traffic can still move between VLANs.

**Q: What is VLAN trunking and what are the risks?**

A trunk carries traffic for multiple VLANs between switches or between a switch and a router. It uses tagging so the receiving device knows which VLAN each frame belongs to.

Risks:
- Misconfigured trunks can expose unintended VLANs.
- Native VLAN misuse can create confusion or attack paths.
- VLAN hopping attacks may exploit trunk negotiation or tagging weaknesses if switch security is poor.

Defenses:
- Disable unused ports.
- Set access ports explicitly.
- Restrict allowed VLANs on trunks.
- Avoid dynamic trunk negotiation where not needed.
- Use a non-default native VLAN and keep it unused for user traffic.

**Q: What is inter-VLAN routing?**

Inter-VLAN routing allows traffic to move between VLANs through a Layer 3 device such as a router or multilayer switch.

Security implication:
- Once you enable inter-VLAN routing, you must enforce policy with ACLs, firewall rules, or segmentation controls.
- Without policy, VLANs become organizational labels rather than security boundaries.

**Q: What is a DMZ?**

A DMZ (demilitarized zone) is a network segment that hosts public-facing services such as web, mail, or DNS servers while isolating them from the internal network.

Typical design:
- Internet-facing traffic enters the DMZ.
- The DMZ has tightly controlled access to internal systems.
- Internal systems should not be directly reachable from the Internet.

**Q: What is microsegmentation?**

Microsegmentation applies fine-grained policy to workloads, often using software-defined networking, host agents, or virtualization controls.

Use it when:
- You need east-west traffic control inside data centers or cloud environments.
- You want to isolate application tiers more tightly than VLANs alone can provide.
- You need policy that follows workloads rather than physical ports.

**Q: What are common segmentation mistakes?**

- Treating VLANs as a complete security boundary.
- Allowing broad inter-VLAN routing without ACLs.
- Forgetting management networks and backup networks.
- Leaving default gateway access too open.
- Failing to document trust zones and exceptions.

**Q&A: Firewalls, ACLs, and policy enforcement**

**Q: How do ACLs differ from firewalls?**

An ACL (access control list) is a rule set on a router, switch, or host that permits or denies traffic based on criteria such as IP address, protocol, or port.

Differences:
- ACLs are often simpler and closer to the device.
- Firewalls usually provide richer state tracking and inspection.
- ACLs are common for routing interfaces and segmentation boundaries.

**Q: What is the best practice for firewall rule design?**

- Start with a deny-by-default stance.
- Allow only required traffic.
- Document the business justification for each rule.
- Review and remove stale rules regularly.
- Log important denies and high-risk allows.
- Place rules in a predictable order to avoid shadowing.

**Q: What is a common firewall policy pattern for a server tier?**

A typical pattern is:
- Internet to reverse proxy or WAF: allow only required web ports.
- Reverse proxy to application tier: allow only application-specific ports.
- Application tier to database tier: allow only database ports from approved sources.
- All other traffic: deny.

Example Linux `iptables` pattern for a simple host firewall:
```bash
iptables -P INPUT DROP
iptables -P FORWARD DROP
iptables -P OUTPUT ACCEPT

iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
iptables -A INPUT -i lo -j ACCEPT
iptables -A INPUT -p tcp --dport 22 -s 10.10.10.0/24 -j ACCEPT
iptables -A INPUT -p tcp --dport 443 -j ACCEPT
```

This example shows a host-centric policy, not a complete perimeter design. In real environments, you also consider outbound filtering, logging, and any NAT or routing requirements.

**Q&A: VPNs and remote access**

**Q: What is a VPN?**

A VPN (Virtual Private Network) creates an encrypted tunnel over an untrusted network so traffic can traverse public infrastructure securely.

Main goals:
- Confidentiality through encryption.
- Integrity through authentication and message protection.
- Remote access or site-to-site connectivity.

**Q: What is the difference between site-to-site and remote-access VPNs?**

- **Site-to-site VPN**: connects two networks, such as branch office to headquarters.
- **Remote-access VPN**: connects an individual user or device to the enterprise network.

**Q: What is the difference between IPsec and SSL/TLS VPNs?**

| Type | Typical Use | Notes |
|---|---|---|
| IPsec VPN | Site-to-site and some remote access | Operates at Layer 3 and is common for network-to-network tunnels |
| SSL/TLS VPN | Remote access | Uses TLS and is often easier through firewalls and NAT |

IPsec commonly uses:
- IKE (Internet Key Exchange) for negotiation.
- ESP (Encapsulating Security Payload) for encryption and integrity.
- AH (Authentication Header) for integrity only, though it is less common in modern deployments.

SSL/TLS VPN commonly uses:
- A web portal or client software.
- TLS for transport security.
- Per-user authentication and posture checks.

**Q: What is split tunneling and why is it risky?**

Split tunneling sends some traffic through the VPN and some traffic directly to the Internet.

Benefits:
- Reduces load on the corporate VPN concentrator.
- Improves user performance for non-corporate traffic.

Risks:
- Creates a path between the enterprise network and the user’s local network.
- Can expose the corporate network to local threats on the user’s home or public network.
- Can complicate monitoring and data loss prevention.

Use split tunneling only when policy and risk analysis support it.

**Q: What is always-on VPN?**

Always-on VPN keeps the tunnel active whenever the device has network connectivity. It is useful for managed endpoints that must remain under enterprise policy outside the office.

**Q: What remote-access design considerations matter on the exam?**

- Strong authentication, ideally MFA (multifactor authentication).
- Device posture checks for managed endpoints.
- Least-privilege access to internal resources.
- Logging of user, device, and session activity.
- Clear handling of split tunneling, DNS, and local resource access.
- High availability for VPN gateways.

**Q&A: Routing and secure path control**

**Q: Why does routing matter to security?**

Routing determines where traffic can go. A routing mistake can expose internal networks, create loops, blackhole traffic, or send traffic through an untrusted path.

**Q: What is route summarization?**

Route summarization combines multiple routes into a shorter advertisement.

Benefits:
- Reduces routing table size.
- Limits route churn.
- Can reduce the amount of topology detail exposed to other networks.

**Q: What is route redistribution and why is it risky?**

Route redistribution shares routes between different routing protocols.

Risks:
- It can introduce loops if not controlled.
- It can leak routes between domains that should stay separate.
- It can create unstable routing if metrics and filters are poorly designed.

Defenses:
- Filter redistributed routes.
- Tag routes to prevent re-advertisement loops.
- Use route maps or policy controls.
- Test changes in a lab before production.

**Q: What is BGP and why is it important?**

BGP (Border Gateway Protocol) is the routing protocol used between autonomous systems on the Internet.

Security concerns:
- Route hijacking or route leaks can redirect traffic.
- Misconfiguration can expose or blackhole services.
- BGP trust is policy-based, not inherently secure.

Common protections:
- Prefix filtering.
- Max-prefix limits.
- Authentication where supported.
- RPKI (Resource Public Key Infrastructure) and route origin validation.
- Monitoring for unexpected route announcements.

**Q: What is RPKI?**

RPKI helps validate whether an autonomous system is authorized to originate a prefix. It reduces the risk of route hijacking, though it does not solve every routing problem.

**Q: What is a routing loop?**

A routing loop occurs when packets circulate between routers because each router believes another path is correct. Loops waste bandwidth and can cause outages.

**Q&A: DNS and name resolution**

**Q: Why is DNS security important?**

DNS (Domain Name System) translates names into IP addresses. If DNS is compromised, users can be redirected to malicious destinations even when other controls appear normal.

**Q: What are common DNS record types?**

- **A**: maps a name to an IPv4 address.
- **AAAA**: maps a name to an IPv6 address.
- **CNAME**: aliases one name to another.
- **MX**: identifies mail exchangers.
- **NS**: identifies authoritative name servers.
- **PTR**: reverse lookup from IP to name.
- **TXT**: stores text data, often used for SPF, DKIM, or verification.
- **SRV**: identifies service location for some applications.

**Q: What is DNS cache poisoning?**

DNS cache poisoning inserts false DNS data into a resolver cache so clients receive malicious or incorrect answers.

Defenses:
- Use DNSSEC (Domain Name System Security Extensions) where appropriate.
- Patch resolvers and authoritative servers.
- Randomize source ports and query IDs where supported.
- Restrict recursion to trusted clients.
- Monitor for unusual TTLs, answers, or upstream behavior.

**Q: What is DNSSEC?**

DNSSEC adds digital signatures to DNS data so resolvers can verify authenticity and integrity.

Important limitation:
- DNSSEC does not encrypt DNS traffic.
- It protects against tampering and spoofing, not confidentiality.

**Q: What are DNS over TLS and DNS over HTTPS?**

- **DoT** (DNS over TLS) encrypts DNS queries using TLS, commonly on port 853.
- **DoH** (DNS over HTTPS) carries DNS over HTTPS, commonly on port 443.

Why they matter:
- They reduce passive observation and some tampering on the path.
- They can complicate enterprise monitoring and policy enforcement if unmanaged.

**Q: What is split-horizon DNS?**

Split-horizon DNS returns different answers depending on where the query comes from. Internal clients may receive private addresses while external clients receive public addresses.

Use it to:
- Keep internal topology hidden.
- Support internal-only services.
- Avoid exposing private IPs externally.

**Q&A: Wireless security**

**Q: What are the main wireless security standards?**

- **WPA2**: widely deployed, uses AES-based protection in modern configurations.
- **WPA3**: improves authentication and resistance to offline guessing.
- **Enterprise mode**: uses 802.1X and RADIUS for per-user authentication.
- **Personal mode**: uses a pre-shared key, which is simpler but weaker operationally for enterprises.

**Q: What is 802.1X?**

802.1X is port-based network access control used for wired and wireless authentication. It typically involves:
- A supplicant on the client.
- An authenticator such as an access point or switch.
- An authentication server such as RADIUS.

**Q: What is a rogue access point?**

A rogue access point is an unauthorized AP connected to the network or operating nearby. It can bypass policy and create a hidden entry point.

**Q: What is an evil twin attack?**

An evil twin is a malicious AP that imitates a legitimate SSID to trick users into connecting. Attackers may capture credentials or force traffic through a hostile network.

Defenses:
- Use strong enterprise authentication.
- Train users to verify trusted SSIDs.
- Use wireless intrusion detection or prevention.
- Prefer certificate-based authentication where possible.

**Q: What is a deauthentication attack?**

A deauthentication attack sends forged management frames to disconnect clients from an AP. Attackers may use it to force reconnection to a rogue AP or disrupt service.

Defenses:
- Protected management frames where supported.
- Wireless intrusion detection.
- Strong monitoring for unusual disconnect patterns.

**Q: Does MAC filtering secure wireless networks?**

No. MAC filtering is weak because MAC addresses can be observed and spoofed. It may be used as a minor administrative control, but not as a primary security measure.

**Q: Why do site surveys and RF planning matter?**

A site survey helps you place APs, tune power levels, and reduce dead zones or excessive overlap.

Benefits:
- Better coverage and roaming.
- Reduced interference.
- Lower risk of signal leakage outside the intended area.

**Q: What wireless deployment practices are exam-relevant?**

- Use separate SSIDs only when needed; too many SSIDs increase management overhead.
- Place guest wireless in a separate zone with Internet-only access.
- Use RADIUS-backed enterprise authentication for corporate devices.
- Disable legacy protocols and weak ciphers.
- Monitor for rogue APs and unauthorized bridges.

**Q&A: Network time and synchronization**

**Q: Why is time synchronization a security issue?**

Logs, certificates, authentication systems, and incident timelines depend on accurate time. If time drifts, correlation becomes unreliable and some security controls may fail.

**Q: What is NTP?**

NTP (Network Time Protocol) synchronizes clocks across systems.

Security considerations:
- Restrict who can query or serve time.
- Use authenticated NTP where supported.
- Protect time servers from spoofing and abuse.
- Monitor for large time offsets.

**Q: What happens if time drifts too far?**

- Log entries become hard to correlate.
- Kerberos and other time-sensitive authentication systems may fail.
- Certificate validation can break.
- Incident response timelines become less trustworthy.

**Q&A: Monitoring, logging, and response**

**Q: What network data should you monitor?**

- Firewall allow and deny logs.
- IDS/IPS alerts.
- VPN authentication and session logs.
- DNS queries and resolver anomalies.
- Proxy and web gateway activity.
- Wireless controller and AP events.
- NetFlow or similar flow telemetry.
- Router and switch configuration changes.

**Q: How does network telemetry support a SIEM?**

A SIEM (Security Information and Event Management) platform collects, normalizes, correlates, and alerts on logs from many sources.

Network telemetry helps the SIEM:
- Correlate a suspicious login with unusual outbound traffic.
- Tie DNS lookups to later connections.
- Detect lateral movement across segments.
- Identify policy violations and repeated scanning.

**Q: What does SOAR add?**

SOAR (Security Orchestration, Automation, and Response) automates response workflows such as:
- Blocking an IP at the firewall.
- Disabling a VPN account.
- Quarantining a host through NAC.
- Opening an incident ticket and notifying responders.

**Q: What is NetFlow or flow telemetry used for?**

Flow telemetry summarizes conversations between endpoints rather than capturing full packets. It is useful for:
- Traffic baselining.
- Detecting unusual destinations or volumes.
- Investigating lateral movement.
- Supporting capacity planning.

**Q&A: Practical configuration examples**

**Q: What does a simple Linux firewall baseline look like?**

```bash
# Allow loopback and established traffic
iptables -P INPUT DROP
iptables -P FORWARD DROP
iptables -P OUTPUT ACCEPT

iptables -A INPUT -i lo -j ACCEPT
iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT

# Allow SSH only from admin subnet
iptables -A INPUT -p tcp -s 10.10.10.0/24 --dport 22 -j ACCEPT

# Allow HTTPS to a local service
iptables -A INPUT -p tcp --dport 443 -j ACCEPT
```

Use this as a conceptual baseline only. In production, you also define logging, IPv6 policy, persistence, and change control.

**Q: What does a basic `nftables` policy look like?**

```bash
nft add table inet filter
nft 'add chain inet filter input { type filter hook input priority 0; policy drop; }'
nft add rule inet filter input ct state established,related accept
nft add rule inet filter input iif lo accept
nft add rule inet filter input tcp dport 22 ip saddr 10.10.10.0/24 accept
nft add rule inet filter input tcp dport 443 accept
```

**Q: What is a practical VPN design example?**

A common remote-access design is:
- Users authenticate with MFA.
- The VPN gateway terminates TLS or IPsec.
- NAC checks device posture after connection.
- Users receive access only to approved subnets.
- DNS and logging are forced through enterprise controls.

Example conceptual split:
- Finance users: access only finance apps.
- IT admins: access management subnets and jump hosts.
- Contractors: access a single application through a reverse proxy or ZTNA-style gateway.

**Q: What is a practical wireless enterprise example?**

A typical enterprise wireless design:
- Corporate SSID uses WPA2-Enterprise or WPA3-Enterprise with 802.1X.
- Guest SSID maps to a separate VLAN and Internet-only firewall policy.
- RADIUS assigns roles based on user or device group.
- Wireless IDS watches for rogue APs and evil twins.
- AP placement comes from a site survey, not guesswork.

**Q: What is a practical routing security example?**

For an edge router:
- Filter inbound and outbound prefixes.
- Limit accepted routes with max-prefix.
- Summarize internal routes before advertising them externally.
- Use route maps to prevent accidental redistribution.
- Monitor for unexpected BGP announcements.

**Q&A: Quick exam distinctions**

**Q: Is NAT a security control?**

No. NAT (Network Address Translation) changes addresses and can reduce direct exposure of internal IPs, but it is not a security control by itself. Do not treat NAT as a substitute for firewall policy, segmentation, or authentication.

**Q: Does IPv6 remove the need for security controls?**

No. IPv6 still needs segmentation, firewalling, monitoring, and secure configuration. Neighbor Discovery Protocol (NDP) replaces ARP in IPv6 networks, so you must understand IPv6-specific control points.

**Q: Is encryption the same as authentication?**

No. Encryption protects confidentiality. Authentication verifies identity. Some protocols provide both, but they are distinct goals.

**Q: What is the difference between confidentiality, integrity, and availability in network controls?**

- **Confidentiality**: keep traffic private with encryption, access control, and segmentation.
- **Integrity**: prevent unauthorized modification with signatures, authentication, and secure routing.
- **Availability**: keep services reachable with redundancy, load balancing, DDoS controls, and resilient design.

**Q: What is the most common CISSP trap in Domain 4?**

Assuming a single control solves the problem. In practice, secure network design layers controls:
- Segmentation limits reach.
- Firewalls enforce policy.
- VPNs protect transit.
- Monitoring detects abuse.
- Routing and DNS controls reduce misdirection.
- Wireless and time controls close common gaps.

**High-yield review points**

- **Firewalls** enforce policy; WAFs protect web apps; proxies mediate traffic.
- **Segmentation** reduces lateral movement, but VLANs alone are not enough.
- **VPNs** protect traffic in transit; split tunneling changes risk.
- **Routing** mistakes can expose or blackhole traffic.
- **DNS** is a high-value target; protect recursion, cache integrity, and records.
- **Wireless** needs enterprise authentication, rogue AP detection, and RF planning.
- **Time sync** supports logs and authentication; drift breaks trust.
- **SIEM/SOAR** turn network telemetry into detection and response.

**Final exam reminder**

When a question asks for the “best” control, choose the one that matches the threat, the layer, and the operational goal. CISSP often rewards the answer that is preventive, least disruptive, and aligned with policy rather than the most technically aggressive option.
