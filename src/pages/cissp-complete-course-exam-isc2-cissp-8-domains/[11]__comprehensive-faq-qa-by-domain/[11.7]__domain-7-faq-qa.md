## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Domain 7 overview: Security operations**

Domain 7 focuses on keeping security controls effective in day-to-day operations. It covers how you detect, respond to, recover from, and learn from security events while maintaining business continuity and evidence integrity. In CISSP terms, this domain is less about designing controls and more about operating them reliably under real-world constraints.

A strong answer usually balances **confidentiality, integrity, and availability** with operational practicality. You often need to choose the least disruptive action that still reduces risk, preserves evidence, and supports recovery.

**Security operations management**

Security operations management is the discipline of running controls consistently over time. It includes patching, configuration control, asset tracking, and change governance. These are common CISSP topics because many incidents happen not from exotic attacks, but from weak operational hygiene.

- **Patch management** reduces exposure to known vulnerabilities by applying vendor fixes in a controlled way.
- **Configuration management** keeps systems in a known-good state and prevents drift from approved baselines.
- **Asset management** identifies what exists, where it is, who owns it, and what risk it carries.
- **Change management** ensures modifications are reviewed, tested, approved, and traceable.

A practical patch workflow often looks like this:

```bash
# Linux example: identify pending updates
apt list --upgradable

# Apply security updates on Debian/Ubuntu
sudo apt update
sudo apt upgrade -y

# RHEL/CentOS family
sudo dnf updateinfo list security
sudo dnf update -y
```

For CISSP exam purposes, remember that patching is not just “install updates.” You should consider:
- **Risk-based prioritization** for internet-facing or critical systems.
- **Maintenance windows** to reduce business disruption.
- **Rollback plans** in case a patch breaks a service.
- **Verification** after deployment to confirm the fix is effective.

Configuration management is usually built around baselines and drift detection. A baseline defines the approved state of a system, such as enabled services, firewall rules, registry settings, or file permissions. Drift occurs when a system deviates from that baseline.

```bash
# Example: inspect running services and listening ports
systemctl list-units --type=service --state=running
ss -tulpn
```

**Asset management and ownership**

Asset management supports both security and accountability. If you do not know what you have, you cannot patch it, monitor it, or recover it effectively.

- Track **hardware**, **software**, cloud resources, and data repositories.
- Assign an **owner** for each asset so decisions can be made quickly.
- Classify assets by business criticality and sensitivity.
- Maintain lifecycle records for procurement, deployment, transfer, retirement, and disposal.

A common exam theme is that asset inventory must include shadow IT and ephemeral cloud assets, not just traditional servers. In cloud environments, tagging and automated discovery help maintain visibility.

**Change management**

Change management reduces operational risk by controlling how systems evolve. It is especially important when changes affect security controls, availability, or compliance.

Typical change steps:
1. Submit a change request with scope, risk, and rollback plan.
2. Review impact on security, operations, and business continuity.
3. Test in a nonproduction environment when possible.
4. Approve through the change advisory process.
5. Implement during the scheduled window.
6. Validate the result and document the outcome.

> Emergency changes may bypass normal timing, but they should still be documented, reviewed after the fact, and approved retroactively.

**Incident response governance**

Incident response is a managed process, not an ad hoc reaction. CISSP questions often test whether you understand roles, escalation, declaration, evidence handling, and post-incident learning.

A typical incident response team includes:
- **Incident commander** to coordinate actions and decisions.
- **Security analysts** to triage alerts and investigate.
- **System owners** to provide context and authorize service-specific actions.
- **Legal counsel** to advise on privilege, notification, and evidence handling.
- **HR** when employee misconduct is involved.
- **Public relations** for external messaging.
- **IT operations** for containment, recovery, and restoration.

Incident declaration criteria usually include:
- Confirmed or strongly suspected policy violation.
- Evidence of unauthorized access, malware, data loss, or service compromise.
- Material business impact or regulatory exposure.
- Need for coordinated response across multiple teams.

The declaration decision matters because it triggers formal workflows such as evidence preservation, communications control, and executive notification. A false alarm may remain a security event; a confirmed compromise becomes an incident.

**Incident response lifecycle**

The standard lifecycle is often described as preparation, detection and analysis, containment, eradication, recovery, and lessons learned. The exact labels vary, but the operational logic is consistent.

- **Preparation** includes playbooks, tooling, training, and contact lists.
- **Detection and analysis** confirm what happened and how severe it is.
- **Containment** limits spread and protects critical assets.
- **Eradication** removes the root cause, such as malware or unauthorized accounts.
- **Recovery** restores systems to normal operation.
- **Lessons learned** improve controls and response quality.

Containment decisions are a common scenario-based question. Short-term containment may involve isolating a host from the network, disabling an account, or blocking an IP address. Long-term containment may involve rebuilding systems, rotating credentials, or segmenting affected networks.

**Evidence preservation and chain of custody**

Forensics is only useful if evidence remains trustworthy. The goal is to preserve integrity so the evidence can support internal action, legal proceedings, or regulatory review.

Key concepts:
- **Chain of custody** documents who collected, handled, transferred, stored, and analyzed evidence.
- **Legal hold** instructs relevant parties to preserve records and prevent deletion or alteration.
- **Admissibility** depends on relevance, authenticity, integrity, and proper handling.

A practical evidence workflow:
1. Identify potential evidence sources such as disks, memory, logs, cloud audit trails, and email.
2. Photograph or document the scene when relevant.
3. Collect volatile evidence first if needed, such as RAM or active network connections.
4. Use write blockers or forensic methods to avoid altering originals.
5. Hash evidence before and after transfer to verify integrity.
6. Record every handoff in the chain-of-custody log.
7. Store evidence securely with access controls and retention rules.

Common hash commands:

```bash
# Create a SHA-256 hash for integrity verification
sha256sum evidence.img > evidence.img.sha256

# Verify later
sha256sum -c evidence.img.sha256
```

> If evidence handling is sloppy, the organization may lose the ability to use it in disciplinary, civil, or criminal proceedings.

**Logging and monitoring**

Logging supports detection, investigation, compliance, and accountability. Monitoring turns raw logs into actionable security insight. CISSP exam questions often focus on what to log, how to review it, and how to tune alerts.

Useful log sources include:
- Authentication and authorization logs
- Endpoint detection and response telemetry
- Firewall, proxy, DNS, and VPN logs
- Server and application logs
- Cloud control-plane audit logs
- Database audit logs
- Physical access logs

Log analysis use cases:
- Detect brute-force login attempts or impossible travel.
- Correlate a suspicious process with a network connection.
- Identify privilege escalation or account misuse.
- Reconstruct a timeline of an incident.
- Support compliance audits and nonrepudiation.

Alert tuning is essential because too many alerts create fatigue. A tuned monitoring program balances:
- **False positives**, which waste analyst time.
- **False negatives**, which miss real threats.
- **Thresholds** that reflect normal behavior.
- **Context** such as asset criticality and user role.

A simple SIEM-style query example:

```sql
SELECT user, src_ip, COUNT(*) AS failures
FROM auth_logs
WHERE event_type = 'login_failure'
  AND timestamp > NOW() - INTERVAL '15 minutes'
GROUP BY user, src_ip
HAVING COUNT(*) >= 10;
```

Audit and log review responsibilities are usually shared:
- Security operations reviews alerts and investigates anomalies.
- System administrators ensure logs are generated, forwarded, and retained.
- Compliance or audit functions verify that logging meets policy and regulatory requirements.
- Management ensures resources exist for review, retention, and escalation.

Important logging practices:
- Synchronize time with **NTP** (Network Time Protocol) so events can be correlated.
- Protect logs from tampering with access controls and centralized storage.
- Define retention periods based on legal, regulatory, and business needs.
- Monitor for log gaps, which may indicate failure or attacker suppression.

**Forensics operations**

Forensics is the disciplined collection and analysis of digital evidence. It supports incident response, internal investigations, and legal action. The key principle is to preserve original evidence and work from verified copies whenever possible.

Common forensic tasks:
- Disk imaging and hash verification
- Memory capture and analysis
- Timeline reconstruction
- Malware triage
- Artifact review from browsers, registry, event logs, and cloud services

Practical considerations:
- Volatile data disappears quickly, so capture it early when relevant.
- Use least-invasive methods first.
- Document tools, versions, timestamps, and actions taken.
- Keep analysis separate from original evidence storage.

A common scenario is deciding whether to power off a compromised server. If the system contains volatile evidence such as encryption keys, active sessions, or malware in memory, you may preserve memory first. If the system is causing active damage, containment may take priority. CISSP expects you to weigh evidence preservation against business risk.

**Backup and recovery**

Backups protect availability and support disaster recovery, ransomware recovery, and operational rollback. The exam often tests whether you understand strategy, testing, and restore validation rather than just backup types.

Common backup strategies:
- **Full backup** copies all selected data.
- **Incremental backup** copies changes since the last backup of any type.
- **Differential backup** copies changes since the last full backup.
- **Grandfather-father-son (GFS)** uses daily, weekly, and monthly retention tiers.
- **3-2-1 rule** keeps 3 copies of data, on 2 different media, with 1 copy offsite.
- A stronger modern variant adds an **immutable** or offline copy to resist ransomware.

Restore priorities should follow business impact and recovery objectives:
- **RTO** (Recovery Time Objective): how quickly service must be restored.
- **RPO** (Recovery Point Objective): how much data loss is acceptable.

Backup testing methods:
- **File-level restore test** to confirm individual files can be recovered.
- **Bare-metal restore test** to validate full system recovery.
- **Partial application restore** to verify databases or app components.
- **Disaster recovery exercise** to test people, process, and technology together.
- **Checksum or hash validation** to confirm backup integrity.

> A backup that has never been restored is only an assumption, not a control.

Example backup rotation concept:

```text
Daily: incremental backups
Weekly: full backup
Monthly: archived full backup
Offsite: encrypted copy replicated to a separate location
```

Recovery validation should confirm:
- Data integrity after restore
- Application functionality
- Permissions and access controls
- Dependency services such as DNS, identity, and certificates
- Logging and monitoring are restored too

**Physical security and environmental controls**

Physical security protects facilities, equipment, and people. CISSP questions often ask you to match controls to threats such as fire, flood, theft, power loss, or unauthorized entry.

Facility safeguards include:
- Badges, guards, locks, mantraps, and visitor escort procedures
- CCTV (closed-circuit television) and intrusion detection
- Rack locks and secure server rooms
- Asset tagging and secure storage for removable media

Environmental controls are equally important:
- **HVAC** (heating, ventilation, and air conditioning) to maintain temperature and humidity
- Fire detection and suppression systems appropriate to the environment
- Water leak detection in raised-floor or basement facilities
- Uninterruptible power supplies (**UPS**) for short outages
- Generators for extended outages
- Surge protection and power conditioning

A practical facility design often separates critical systems from public areas and places sensitive equipment away from water sources, exterior walls, and single points of failure. Environmental monitoring should alert on temperature spikes, humidity changes, smoke, and power anomalies.

**Media handling and disposal**

Media protection extends the security lifecycle beyond active use. Removable media, backup tapes, SSDs, and retired laptops can all leak sensitive data if not handled correctly.

Good practices:
- Encrypt portable media.
- Restrict use of removable storage where possible.
- Maintain checkout and return records.
- Sanitize or destroy media before disposal or reuse.
- Verify destruction methods match the sensitivity of the data.

Sanitization methods may include:
- Clearing for low-risk reuse
- Purging for stronger protection
- Physical destruction for highly sensitive media

**Operational security scenarios**

CISSP often presents tradeoff questions where no option is perfect. The best answer usually preserves evidence, limits damage, and follows policy.

Scenario examples:
- If ransomware is detected on one workstation, isolate it first, preserve logs and memory if feasible, then assess spread before restoring from clean backups.
- If a privileged account is compromised, disable the account, rotate credentials, review logs for lateral movement, and preserve evidence before making broad changes.
- If a patch causes service instability, roll back using the documented change plan and validate whether the security risk requires a compensating control.

**High-yield distinctions**

| Topic | Key idea | Common exam trap |
|---|---|---|
| Incident vs event | An incident has confirmed or likely adverse impact | Treating every alert as an incident |
| Containment vs eradication | Containment limits spread; eradication removes root cause | Reimaging before preserving evidence |
| Backup vs archive | Backup supports recovery; archive supports retention | Using backups as long-term records |
| Logs vs monitoring | Logs are records; monitoring is active review and alerting | Assuming logging alone detects attacks |
| Physical vs environmental security | Physical controls stop access; environmental controls protect from facility hazards | Ignoring power, fire, and HVAC risks |

**CISSP-style quick checks**

- If the question asks for the first priority during an active compromise, think **containment and safety** before cleanup.
- If legal action is possible, think **evidence preservation** and chain of custody.
- If a system is critical, think about **restore validation** and dependency testing, not just file recovery.
- If alerts are overwhelming, think **tuning and baselining**, not disabling monitoring.
- If a control must survive ransomware, think **offline or immutable backups**.

**Practical command and configuration examples**

Centralized logging with syslog:

```conf
# /etc/rsyslog.d/50-forward.conf
*.* @@logserver.example.com:514
```

NTP configuration example:

```conf
# /etc/chrony.conf
server time1.example.com iburst
server time2.example.com iburst
```

Basic Linux file integrity check:

```bash
find /etc -type f -exec sha256sum {} \; > etc-baseline.sha256
sha256sum -c etc-baseline.sha256
```

These examples matter because Domain 7 is operational. The exam expects you to understand how controls behave in practice, not just how they are defined.

**Final exam mindset**

When you see a Domain 7 question, ask:
- What is the operational risk?
- What action preserves evidence?
- What action reduces impact fastest without creating new risk?
- What control supports recovery and future prevention?
- What process ensures accountability and traceability?

If you answer in that order, you usually align with CISSP’s management-oriented perspective.
