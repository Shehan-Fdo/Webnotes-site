## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Domain 3 overview**

Domain 3 focuses on security architecture and engineering: how you design systems so they remain secure under normal operation, failure, and attack. CISSP questions often test *trade-offs* rather than isolated definitions, so you should connect design principles, cryptography, hardware trust, physical controls, and platform choices to business requirements.

A useful exam mindset is: identify the asset, the threat, the trust boundary, and the control that best reduces risk with the least operational impact.

**What is a security model?**

A **security model** is a formal way to describe, reason about, or enforce a security policy. It gives structure to rules such as who can read, write, or modify data, and under what conditions.

- It is not the same as the policy itself.
- It helps you analyze whether a design meets the policy.
- Common examples include Bell-LaPadula, Biba, Clark-Wilson, and Brewer-Nash.

**What is the difference between a security policy and a security model?**

A **policy** states the organization’s rules and objectives. A **model** provides the formal framework used to express or evaluate those rules.

- Policy answers: “What do we want?”
- Model answers: “How do we describe or reason about it?”
- Mechanisms answer: “How do we enforce it?”

This distinction matters on the exam because a model can support multiple policies, and a policy can be implemented through multiple mechanisms.

**Which classic security models should I know?**

The most common CISSP models are:

- **Bell-LaPadula (BLP)**: confidentiality-focused
  - No read up
  - No write down
- **Biba**: integrity-focused
  - No read down
  - No write up
- **Clark-Wilson**: integrity through well-formed transactions and separation of duties
- **Brewer-Nash**: Chinese Wall model, prevents conflicts of interest
- **Graham-Denning**: focuses on secure creation and deletion of subjects and objects
- **Harrison-Ruzzo-Ullman**: analyzes access rights and leakage conditions

For exam questions, match the model to the goal:
- Confidentiality → Bell-LaPadula
- Integrity → Biba or Clark-Wilson
- Conflict of interest → Brewer-Nash

**How do I remember Bell-LaPadula and Biba?**

A simple memory aid:

- **Bell-LaPadula** protects secrecy:
  - “No read up”
  - “No write down”
- **Biba** protects integrity:
  - “No read down”
  - “No write up”

If a question mentions classified data, Bell-LaPadula is usually the better fit. If it mentions trusted data or preventing contamination, Biba is more likely.

**What is the difference between fail-safe and fail-secure?**

These terms are often confused.

- **Fail-safe** means the system defaults to a safe state when it fails.
- **Fail-secure** means the system preserves security on failure, often by denying access or locking down.

In physical security, a door lock may be designed to fail-secure so that a power loss does not unlock the door. In life-safety systems, some devices may fail-safe to allow evacuation.

> On the exam, read the scenario carefully. A “safe state” can mean different things depending on whether the priority is life safety or security.

**What are the core security design principles?**

Security design principles reduce the chance that a single flaw becomes a full compromise.

- **Least privilege**: give only the access required
- **Need to know**: limit access to information required for the task
- **Separation of duties**: split critical tasks across multiple people or systems
- **Defense in depth**: layer controls so one failure does not expose everything
- **Economy of mechanism**: keep designs simple
- **Fail securely**: deny access or preserve security when errors occur
- **Complete mediation**: check every access, not just the first one
- **Open design**: security should not depend on secrecy of the design
- **Least common mechanism**: minimize shared components between users
- **Psychological acceptability**: controls should be usable enough to be followed
- **Work factor**: make attacks more expensive than the value of the target
- **Compromise recording**: detect and log security-relevant events

These principles often appear as scenario clues. For example, a payment system that requires two separate approvals for refunds is using separation of duties.

**What is the difference between fault tolerance, redundancy, high availability, and graceful degradation?**

These terms are related but not identical.

- **Fault tolerance**: the system continues operating despite component failure
- **Redundancy**: duplicate components or paths exist to absorb failure
- **High availability (HA)**: the system is designed to minimize downtime
- **Graceful degradation**: the system continues in reduced mode rather than failing completely

Examples:
- A clustered database with automatic failover uses redundancy and supports HA.
- A web service that disables nonessential features during load spikes is degrading gracefully.
- A dual power supply in a server is a redundancy control that supports fault tolerance.

**What is a common architecture pattern for availability?**

A typical resilient design includes:

- Multiple application instances
- Load balancers
- Redundant network paths
- Replicated storage
- Automated failover
- Monitoring and alerting

Example topology:

```text
Users -> Load Balancer -> App Node A
                      -> App Node B
                      -> App Node C

App Nodes -> Replicated Database Cluster
App Nodes -> Shared Secrets/Config Store
```

This design reduces single points of failure, but it also increases complexity. More components mean more configuration risk, so secure management is essential.

**What is the difference between symmetric and asymmetric cryptography?**

- **Symmetric cryptography** uses the same key for encryption and decryption.
- **Asymmetric cryptography** uses a public/private key pair.

Symmetric cryptography is faster and is usually used for bulk data. Asymmetric cryptography is slower but solves key distribution and supports digital signatures.

Common examples:
- Symmetric: AES, ChaCha20
- Asymmetric: RSA, ECC
- Key exchange: Diffie-Hellman, Elliptic Curve Diffie-Hellman (ECDH)

**When should I use AES, RSA, ECC, or Diffie-Hellman?**

Use the algorithm that matches the job:

- **AES**: encrypting data at rest or in transit
- **RSA**: legacy encryption, signatures, and key transport in some systems
- **ECC**: signatures and key exchange with smaller keys than RSA
- **Diffie-Hellman**: establishing a shared secret over an untrusted network

Practical guidance:
- Use **AES-GCM** or another authenticated encryption mode when you need confidentiality plus integrity.
- Use **ECC** when you want strong security with smaller keys and better performance on constrained devices.
- Use **RSA** where compatibility matters, but prefer modern alternatives when possible.

**What are common key sizes and what do they imply?**

Key size is not the only measure of strength, but it matters.

- **AES-128** is widely considered strong for most uses.
- **AES-256** provides a larger security margin, though it is not automatically “twice as secure.”
- **RSA-2048** is common, but larger keys may be required for longer-term protection.
- **ECC-256** is often compared roughly to RSA-3072 in security strength.

For the exam, remember that larger keys generally increase computational cost. The best choice depends on performance, interoperability, and required security lifetime.

**What is the role of IVs, nonces, and padding?**

These terms are common crypto pitfalls.

- **IV (Initialization Vector)**: adds randomness or uniqueness to encryption, depending on the mode
- **Nonce**: a number used once, often to ensure uniqueness and prevent replay
- **Padding**: fills the final block when a block cipher mode requires fixed-size blocks

Important points:
- Reusing an IV or nonce in the wrong mode can break confidentiality or integrity.
- CBC mode requires unpredictable IVs.
- GCM requires a unique nonce for each key; nonce reuse is dangerous.
- Padding errors can create oracle attacks if error messages leak information.

**What is the difference between block ciphers and stream ciphers?**

- **Block ciphers** encrypt fixed-size blocks of data.
- **Stream ciphers** encrypt data one bit or byte at a time using a keystream.

Examples:
- Block cipher: AES
- Stream cipher: ChaCha20

Use cases:
- Block ciphers are common in storage and many protocol designs.
- Stream ciphers can be efficient in software and low-latency environments.

**What is authenticated encryption?**

Authenticated encryption provides confidentiality and integrity together.

- **AES-GCM** is a common authenticated encryption mode.
- It helps detect tampering and prevents silent modification of ciphertext.

This is important because encryption alone does not guarantee integrity. A system may hide data but still allow undetected manipulation unless integrity protection is included.

**What are common cryptographic implementation mistakes?**

Common mistakes include:

- Reusing IVs or nonces
- Using weak randomness
- Hardcoding keys in source code
- Storing keys next to encrypted data without protection
- Using outdated algorithms such as MD5 or SHA-1 for security-sensitive purposes
- Failing to validate certificates properly
- Ignoring downgrade attacks
- Rolling your own cryptography instead of using vetted libraries
- Using encryption without integrity protection where tampering matters

A practical example of weak randomness:

```bash
# Bad idea: predictable seed or homegrown randomness
python -c "import random; print(random.randint(1, 1000000))"
```

Use a cryptographically secure random number generator instead.

**What is salting, and how is it different from peppering?**

- **Salt** is a unique random value stored with the password hash.
- **Pepper** is a secret value stored separately from the password database.

Why they matter:
- Salt prevents identical passwords from producing identical hashes.
- Pepper adds another secret layer that helps if the password database is stolen.

Example password storage flow:

```text
password + salt + pepper -> hash -> store hash and salt
```

Best practice:
- Use a slow password hashing algorithm such as Argon2, bcrypt, or scrypt.
- Never store plaintext passwords.
- Do not use fast general-purpose hashes like SHA-256 alone for password storage.

**What is a MAC, and how is it different from a digital signature?**

A **MAC (Message Authentication Code)** provides integrity and authenticity with a shared secret key.

- Fast and efficient
- Both parties must share the same secret
- Does not provide nonrepudiation

A **digital signature** uses asymmetric cryptography.

- Verifiable by anyone with the public key
- Supports stronger attribution than a shared-secret MAC
- Often used for code signing, document signing, and certificate chains

**Does a digital signature guarantee nonrepudiation?**

It supports nonrepudiation, but it does not guarantee it by itself.

Nonrepudiation is a legal and operational property, not just a cryptographic one. It depends on:

- Strong identity proofing
- Private key protection
- Audit logs
- Time synchronization
- Certificate lifecycle controls
- Evidence handling and legal context

If a private key is compromised or identity proofing is weak, the evidentiary value of the signature can be challenged.

**What is PKI (Public Key Infrastructure)?**

**PKI** is the framework for issuing, managing, validating, and revoking digital certificates.

Core components:
- Certificate Authority (CA)
- Registration Authority (RA)
- Certificate repository
- Revocation services
- Trust anchors
- Policies and procedures

A certificate binds a public key to an identity, but trust depends on the entire chain and validation process.

**What should I know about certificate formats and trust chains?**

Common certificate and key formats include:

- **X.509**: standard certificate format
- **PEM**: Base64-encoded text format, often used for certificates and keys
- **DER**: binary encoding of X.509
- **PFX/P12**: container that can hold a certificate and private key

Trust chain concepts:
- A leaf certificate is validated against intermediate CAs and a trusted root CA.
- The root CA is usually stored in a trust store.
- If any certificate in the chain is invalid, expired, or untrusted, validation fails.

Common failure causes:
- Expired certificate
- Missing intermediate certificate
- Hostname mismatch
- Untrusted root
- Revoked certificate
- Incorrect key usage or extended key usage
- Clock skew on the client

**How does revocation checking work?**

Revocation checking determines whether a certificate should still be trusted.

Common methods:
- **CRL (Certificate Revocation List)**: a published list of revoked certificates
- **OCSP (Online Certificate Status Protocol)**: real-time status query
- **OCSP stapling**: the server provides a signed OCSP response to reduce client lookups

Practical considerations:
- CRLs can be large and slow to check.
- OCSP improves freshness but depends on network availability.
- Some clients use soft-fail behavior if revocation status cannot be checked, which can weaken security.

**What is a hardware security module (HSM)?**

An **HSM** is a tamper-resistant device used to generate, store, and use cryptographic keys securely.

Use cases:
- Root CA key protection
- Payment systems
- Code signing
- High-value key custody
- Key generation and signing operations

Why it matters:
- Keys are harder to extract than from general-purpose servers.
- HSMs can enforce policy, such as key usage restrictions and dual control.

**What are TPMs and secure enclaves?**

- **TPM (Trusted Platform Module)**: a hardware chip that stores keys, measures boot integrity, and supports platform attestation
- **Secure enclave**: a protected execution environment isolated from the main OS

Typical uses:
- Disk encryption key protection
- Measured boot
- Device identity
- Attestation
- Sensitive computation in isolated memory

These technologies help establish trust in the platform, but they do not make the entire system invulnerable. Firmware, supply chain, and software vulnerabilities still matter.

**What is trusted computing and measured boot?**

Trusted computing aims to establish a chain of trust from hardware upward.

- **Secure boot** verifies each boot stage before execution.
- **Measured boot** records hashes of boot components for later attestation.
- **Attestation** lets another system verify the platform state.

This is useful when you need to know not just that a system booted, but that it booted into a known-good state.

**What are the main physical security controls?**

Physical security protects facilities, equipment, and people.

Common controls:
- **Perimeter fencing**
- **Lighting**
- **Guards**
- **CCTV (Closed-Circuit Television)**
- **Badges and access cards**
- **Locks**
- **Mantraps**
- **Biometrics**
- **Visitor escort procedures**
- **Alarm systems**
- **Asset tagging**
- **Security zones**

A layered facility design might look like this:

```text
Perimeter fence -> Parking lot -> Building entrance -> Reception
-> Badge-controlled door -> Mantrap -> Server room -> Cage -> Rack
```

**What is a mantrap and why is it used?**

A **mantrap** is a controlled entry area with two interlocking doors.

- Only one door opens at a time.
- It reduces tailgating and piggybacking.
- It is common in data centers and high-security areas.

Mantraps are especially useful when you need to verify identity before granting access to a sensitive zone.

**What is the difference between tailgating and piggybacking?**

- **Tailgating**: an unauthorized person follows an authorized person through a controlled entry
- **Piggybacking**: the authorized person knowingly allows another person to enter

Both are access control failures, but piggybacking involves consent and is often harder to detect.

**What should I know about locks, badges, and biometrics?**

- **Locks** protect doors and cabinets; their strength depends on the lock type and installation quality.
- **Badges** provide identity and access control, but they can be lost, stolen, or cloned.
- **Biometrics** verify a physical or behavioral trait.

Biometric concepts:
- **FAR (False Acceptance Rate)**: unauthorized person accepted
- **FRR (False Rejection Rate)**: authorized person rejected
- **CER (Crossover Error Rate)**: point where FAR and FRR intersect

Biometrics are best used as part of multi-factor authentication, not as the only control in high-risk environments.

**What is CCTV used for in security architecture?**

CCTV supports deterrence, detection, and investigation.

Good practice:
- Cover entry points, sensitive areas, and blind spots
- Ensure time synchronization
- Retain footage according to policy
- Protect recordings from tampering
- Use signage where required by law or policy

CCTV is not a substitute for access control; it is a monitoring and evidence control.

**What are the major fire suppression options?**

Fire suppression choices depend on the environment and the assets being protected.

- **Water sprinklers**: effective for many fires, but can damage electronics
- **Dry pipe systems**: reduce accidental water discharge in cold or sensitive areas
- **Clean agents**: reduce fire without leaving residue
- **CO2 systems**: effective but dangerous in occupied spaces
- **Foam systems**: useful for flammable liquids
- **Inert gas systems**: reduce oxygen concentration to suppress fire

Important distinction:
- **CO2** is generally unsuitable for occupied spaces because it can displace oxygen and create a life-safety hazard.
- Many **clean agents** are designed for occupied spaces when properly engineered and installed.

Example exam answer:
- A server room needing fire suppression with minimal residue often points to a clean agent system, but you must still consider occupancy, ventilation, and environmental constraints.

**What should I know about power protection?**

Power issues can cause outages, data loss, and hardware damage.

Controls:
- **UPS (Uninterruptible Power Supply)**: short-term battery backup and power conditioning
- **Generator**: long-duration backup power
- **Surge suppressor**: protects against voltage spikes
- **Power conditioning**: smooths irregular power
- **Dual power supplies**: increase resilience

A common design is:
- Utility power -> UPS -> critical equipment
- Utility power -> generator -> UPS recharge / extended operation

**What is the difference between virtualization and containers?**

- **Virtual machines (VMs)** virtualize hardware and run separate guest operating systems.
- **Containers** share the host kernel and isolate applications at the process level.

Security implications:
- VMs provide stronger isolation boundaries in many cases.
- Containers are lighter and faster but depend heavily on host hardening and orchestration security.
- A container escape can affect the host and other workloads.

**What are common container security concerns?**

- Untrusted base images
- Image drift and lack of signing
- Overprivileged containers
- Mounted host paths
- Exposed secrets in environment variables
- Weak orchestration RBAC (Role-Based Access Control)
- Insecure registries
- Shared kernel risk
- Poor network segmentation between pods or services

Practical controls:
- Use minimal base images
- Scan and sign images
- Run as non-root
- Drop unnecessary Linux capabilities
- Store secrets in a dedicated secrets manager
- Restrict admission to approved images
- Apply network policies

Example Kubernetes hardening snippet:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app
spec:
  containers:
    - name: app
      image: registry.example.com/app:1.2.3
      securityContext:
        runAsNonRoot: true
        allowPrivilegeEscalation: false
        readOnlyRootFilesystem: true
```

**What is the cloud shared responsibility model?**

The cloud shared responsibility model divides security duties between the cloud provider and the customer.

A practical way to think about it:
- The provider secures the underlying cloud infrastructure.
- You secure what you deploy, configure, and store in the cloud.

Responsibilities vary by service model:
- **IaaS (Infrastructure as a Service)**: you manage more of the OS, applications, identity, and data
- **PaaS (Platform as a Service)**: the provider manages more of the platform stack
- **SaaS (Software as a Service)**: the provider manages the application platform, but you still manage identity, data, access, and governance

> Do not assume your responsibility simply disappears as you move from IaaS to SaaS. Identity, data classification, configuration, and compliance remain important in every model.

**What are cloud-native security concerns?**

Cloud environments introduce control plane and orchestration risks.

Key concerns:
- Misconfigured storage buckets
- Overly permissive IAM (Identity and Access Management)
- Exposed management APIs
- Weak secrets handling
- Insecure metadata service access
- Poor tenant isolation
- Unreviewed infrastructure-as-code changes
- Lack of logging and monitoring
- Supply chain risk in images and dependencies

Practical controls:
- Use least privilege IAM
- Enable logging and alerting
- Protect secrets with managed vaults
- Sign and scan images
- Review infrastructure as code before deployment
- Restrict administrative access to the control plane

**What is system security architecture?**

System security architecture is the way components, interfaces, trust boundaries, and controls are arranged to meet security requirements.

You should think about:
- Where trust begins and ends
- Which components are exposed
- How data moves between zones
- Which controls are preventive, detective, and corrective
- How failure affects confidentiality, integrity, and availability

A secure architecture usually includes:
- Segmentation
- Strong identity and access control
- Secure defaults
- Logging and monitoring
- Key management
- Patch and configuration management
- Resilience and recovery planning

**What are embedded, IoT, and mobile security considerations?**

These platforms often have constrained resources and different trust assumptions.

Embedded and IoT:
- Limited CPU, memory, and storage
- Long patch cycles
- Physical exposure
- Default credentials
- Weak update mechanisms
- Supply chain risk

Mobile:
- Device encryption
- Secure boot
- App sandboxing
- Mobile device management (MDM)
- App signing and permission control
- Risk from jailbreaking or rooting

Exam clue:
- If the device is physically accessible and hard to patch, focus on tamper resistance, secure update channels, and minimal exposed services.

**What is the difference between authentication, authorization, and accounting?**

- **Authentication**: prove who you are
- **Authorization**: determine what you can do
- **Accounting**: record what you did

These are often grouped as **AAA**. In practice, logging and audit trails support accountability and incident response.

**What is the role of logging and monitoring in architecture?**

Logging and monitoring support detection, investigation, and compliance.

Good logs should be:
- Time-synchronized
- Centralized
- Protected from tampering
- Retained according to policy
- Useful for correlation across systems

Useful sources:
- Authentication logs
- Privileged access logs
- Firewall and IDS/IPS logs
- Cloud control plane logs
- Application audit logs
- Endpoint telemetry

**What is the difference between encryption at rest, in transit, and in use?**

- **At rest**: data stored on disk, object storage, backups, or archives
- **In transit**: data moving across networks
- **In use**: data being processed in memory or CPU registers

Examples:
- Disk encryption protects data at rest.
- TLS (Transport Layer Security) protects data in transit.
- Secure enclaves and confidential computing aim to reduce exposure in use.

**What is TLS used for?**

**TLS** protects network communications with confidentiality, integrity, and server authentication, and optionally client authentication.

Practical points:
- Use modern TLS versions and strong cipher suites.
- Disable weak protocols and ciphers.
- Validate certificates correctly.
- Protect against downgrade attacks.
- Use mutual TLS when both endpoints must authenticate each other.

Example OpenSSL test:

```bash
openssl s_client -connect example.com:443 -servername example.com
```

**What are common secure system design trade-offs?**

Security architecture is usually about balancing competing goals:

- Stronger isolation vs. operational complexity
- More logging vs. privacy and storage cost
- Faster performance vs. stronger cryptography
- Centralized control vs. resilience
- Usability vs. strict enforcement

A good design is not the one with the most controls; it is the one that best fits the risk, environment, and business need.

**How do I approach Domain 3 scenario questions?**

Use a repeatable method:

1. Identify the primary security objective: confidentiality, integrity, availability, or safety.
2. Determine the asset and the threat.
3. Look for the control that best matches the objective.
4. Eliminate answers that solve the wrong problem.
5. Prefer layered, practical controls over single-point fixes.

Example:
- If the question asks for protection against unauthorized disclosure of classified data, choose a confidentiality control such as Bell-LaPadula-style enforcement, encryption, or access restriction.
- If it asks for preventing unauthorized modification of financial records, choose integrity controls such as separation of duties, hashing, signatures, or Clark-Wilson-style transaction controls.

| Topic | Best fit | Key idea |
|---|---|---|
| Bell-LaPadula | Confidentiality | No read up, no write down |

**What are common exam traps in Domain 3?**

- Confusing policy with model
- Treating encryption as a complete integrity solution
- Assuming signatures automatically guarantee nonrepudiation in all contexts
- Mixing up fail-safe and fail-secure
- Choosing a control that is technically strong but operationally unrealistic
- Ignoring occupancy when selecting fire suppression
- Assuming cloud responsibility disappears in SaaS
- Overlooking certificate chain and revocation issues
- Forgetting that containers share the host kernel
- Choosing a single control when the scenario needs defense in depth

**Practical CLI and configuration examples**

Check certificate details:

```bash
openssl x509 -in server.crt -text -noout
```

Verify a certificate chain:

```bash
openssl verify -CAfile root-ca.pem -untrusted intermediate.pem server.crt
```

Inspect a TLS connection:

```bash
openssl s_client -connect mail.example.com:443 -servername mail.example.com -showcerts
```

Generate a strong password hash in a Linux environment:

```bash
python3 - <<'PY'
import bcrypt
pw = b"ExamplePassword123!"
print(bcrypt.hashpw(pw, bcrypt.gensalt()).decode())
PY
```

Example systemd hardening snippet:

```ini
[Service]
NoNewPrivileges=true
PrivateTmp=true
ProtectSystem=strict
ProtectHome=true
ReadWritePaths=/var/lib/myapp
```

**Final exam reminders**

Domain 3 rewards precise thinking. When you see a question, ask whether it is about:
- a formal model,
- a cryptographic primitive,
- a platform trust mechanism,
- a physical control,
- or a resilient architecture pattern.

If two answers seem plausible, choose the one that best matches the stated objective and the least disruptive secure design.
