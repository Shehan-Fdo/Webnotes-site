## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Domain 8 overview**

Domain 8 focuses on software development security: how you build, verify, release, and maintain software so it resists abuse and supports business goals. On the CISSP exam, you are expected to understand the full lifecycle, not just secure coding. That includes governance, requirements, design, implementation, testing, deployment, and change control.

A useful way to think about this domain is: security must be designed in early, validated continuously, and preserved through release and maintenance. If you wait until production to “add security,” you usually create cost, delay, and residual risk.

**Software development lifecycle models**

Different development models change when security activities happen and how much feedback you get before release.

- **Waterfall** is linear and phase-driven.
  - Best when requirements are stable and changes are costly.
  - Security reviews often happen at phase gates, which can delay discovery of defects.
- **Agile** is iterative and incremental.
  - Security work is added to each sprint through user stories, acceptance criteria, and backlog items.
  - Works well when you need frequent feedback and continuous reprioritization.
- **DevOps** combines development and operations to shorten delivery cycles.
  - Security must integrate into automation, infrastructure, and release pipelines.
  - The goal is fast delivery with repeatable controls.
- **DevSecOps** extends DevOps by embedding security into the pipeline.
  - Security testing, policy checks, and approval gates become part of the delivery workflow.
  - This is often the most exam-relevant modern model because it treats security as continuous.
- **Spiral** emphasizes risk analysis in repeated cycles.
  - Useful for high-risk or uncertain projects because each loop identifies and reduces major risks.
  - Strong fit when threat analysis and prototyping matter early.
- **Prototyping** builds an early model to clarify requirements.
  - Helps users validate functionality and usability.
  - Security risk: prototypes are often rushed, underprotected, and mistakenly promoted into production.
- **Rapid Application Development (RAD)** prioritizes speed and user feedback.
  - Useful for time-sensitive business needs.
  - Security can suffer if governance, testing, and architecture reviews are skipped.

> For the exam, match the model to the risk profile: high uncertainty favors spiral or prototyping; high delivery speed favors Agile/DevOps; high assurance demands stronger gates and evidence.

**Governance artifacts and control hierarchy**

CISSP often tests the difference between policy, standard, procedure, and guideline. These terms matter because they define how security requirements become enforceable.

- **Policy** is high-level management intent.
  - Example: “All customer data must be protected in transit and at rest.”
- **Standard** is mandatory and specific.
  - Example: “TLS 1.2+ must be used for all external services.”
- **Procedure** is step-by-step instruction.
  - Example: “Rotate application secrets every 90 days using the secrets vault workflow.”
- **Guideline** is recommended but not mandatory.
  - Example: “Prefer parameterized queries over string concatenation.”

Other important governance artifacts include:

- **Coding standards**
  - Define secure coding expectations such as input validation, error handling, logging, and memory safety.
  - Reduce inconsistency across teams and make code review more effective.
- **Secure baselines**
  - Establish approved configurations for operating systems, containers, middleware, and application frameworks.
  - Help prevent drift and support repeatable hardening.
- **Architecture standards**
  - Define approved patterns such as zero trust, least privilege, and segmentation.
- **Exception/risk acceptance records**
  - Document when a control is not implemented and who accepted the risk.
- **Security requirements traceability**
  - Maps requirements to design, code, tests, and release evidence.

**Requirements engineering and security validation**

Security requirements should be specific, testable, and traceable. Vague statements like “the app must be secure” are not useful.

Good security requirements often include:

- Authentication requirements
  - Example: “Privileged users must use multifactor authentication (MFA).”
- Authorization requirements
  - Example: “Users may access only records in their assigned tenant.”
- Logging requirements
  - Example: “Administrative actions must be logged with user ID, timestamp, source IP, and action result.”
- Availability requirements
  - Example: “The service must recover within 15 minutes after a single-node failure.”
- Privacy requirements
  - Example: “Sensitive fields must be masked in non-production environments.”

Validation methods include:

- **Requirements review**
  - Confirms the requirement is complete, unambiguous, and testable.
- **Traceability matrix**
  - Links each requirement to design elements, test cases, and release evidence.
- **Acceptance criteria**
  - Defines what must be true for a story or feature to be accepted.
- **Misuse/abuse cases**
  - Describes how an attacker might misuse the feature.
- **Security user stories**
  - Example: “As a security admin, I want session timeout enforced so that idle sessions cannot be hijacked.”
- **Control testing**
  - Verifies the implemented control behaves as intended.

**Threat modeling**

Threat modeling identifies what can go wrong before code is written or released. It helps you prioritize controls based on realistic attack paths.

Common outputs include:

- **Data flow diagrams (DFDs)**
  - Show processes, data stores, external entities, and trust boundaries.
  - Useful for identifying where data crosses security boundaries.
- **Threat lists**
  - Enumerate likely threats against each component or flow.
- **Risk ratings**
  - Prioritize threats by likelihood and impact.
- **Mitigation plans**
  - Map threats to controls, owners, and deadlines.

Common methods and frameworks:

- **STRIDE**
  - Spoofing
  - Tampering
  - Repudiation
  - Information disclosure
  - Denial of service
  - Elevation of privilege
  - Best used to classify threats against components and flows.
- **Attack trees**
  - Start with an attacker goal and branch into possible paths.
  - Useful for visualizing alternative attack strategies and control points.
- **PASTA** (Process for Attack Simulation and Threat Analysis)
  - Risk-centric and business-impact oriented.
  - Useful when you want threat modeling tied to business objectives.
- **OCTAVE** (Operationally Critical Threat, Asset, and Vulnerability Evaluation)
  - Asset and risk focused.
  - Often used for organizational risk assessment rather than only application design.

A practical DFD review often asks:

- What are the trust boundaries?
- Where does untrusted input enter?
- Where is sensitive data stored or transmitted?
- Which components can be spoofed or tampered with?
- What happens if a service is unavailable?

Example DFD thinking for a web app:

- Browser sends login request to web tier.
- Web tier calls authentication service.
- Authentication service queries identity store.
- Session token is returned to browser.
- Trust boundaries exist between browser and web tier, and between web tier and identity services.

**Secure design principles**

Security design should reduce attack surface and limit blast radius.

- **Least privilege**
  - Give each component only the permissions it needs.
- **Defense in depth**
  - Use multiple independent controls so one failure does not expose the system.
- **Fail secure**
  - Default to denial when a control fails.
- **Economy of mechanism**
  - Keep design simple to reduce bugs and review complexity.
- **Complete mediation**
  - Check authorization every time a resource is accessed.
- **Separation of duties**
  - Split sensitive actions across roles or approvals.
- **Minimize attack surface**
  - Remove unused services, ports, endpoints, and libraries.
- **Secure by default**
  - Ship with safe settings, not permissive ones.

**Secure coding and common application controls**

Secure coding standards translate design intent into implementation rules. They are especially important for web and API security.

Common controls and patterns:

- **Input validation**
  - Validate type, length, format, range, and business logic.
  - Use allowlists when possible.
- **Output encoding**
  - Encode data based on context: HTML, URL, JavaScript, or SQL.
  - Prevents injection into interpreters and browsers.
- **Parameterized queries**
  - Prevent SQL injection by separating code from data.
  - Example:
    ```python
    cursor.execute(
        "SELECT * FROM users WHERE email = ? AND status = ?",
        (email, "active")
    )
    ```
- **Cross-site scripting (XSS) prevention**
  - Encode output, sanitize rich text, and use Content Security Policy (CSP).
  - Avoid unsafe DOM manipulation.
- **Cross-site request forgery (CSRF) prevention**
  - Use anti-CSRF tokens, SameSite cookies, and reauthentication for sensitive actions.
  - CSRF matters when browsers automatically send credentials.
- **Server-side request forgery (SSRF) prevention**
  - Restrict outbound requests, validate destinations, and block access to internal metadata services.
  - Use egress filtering and URL allowlists.
- **Session management**
  - Use random, high-entropy session IDs.
  - Regenerate session IDs after login and privilege change.
  - Set `HttpOnly`, `Secure`, and `SameSite` cookie flags.
  - Enforce idle and absolute timeouts.
- **Authentication**
  - Prefer MFA for privileged access.
  - Use strong password hashing such as bcrypt, scrypt, or Argon2 for stored passwords.
  - Avoid rolling your own authentication protocol.
- **Authorization**
  - Enforce server-side authorization on every request.
  - Use role-based access control (RBAC), attribute-based access control (ABAC), or policy-based access control as appropriate.
- **Error handling**
  - Return generic errors to users.
  - Log detailed diagnostics securely without exposing secrets.
- **Logging**
  - Log security-relevant events such as login failures, privilege changes, and configuration changes.
  - Protect logs from tampering and unauthorized access.

Example secure session cookie:

```http
Set-Cookie: sessionid=abc123; HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=1800
```

**Software assurance techniques**

Software assurance is broader than testing. It includes methods that increase confidence the software behaves securely and predictably.

- **Static application security testing (SAST)**
  - Analyzes source code or binaries without executing them.
  - Good for finding insecure patterns early in the SDLC.
  - Can produce false positives and requires tuning.
- **Dynamic application security testing (DAST)**
  - Tests a running application from the outside.
  - Good for finding runtime issues such as authentication flaws or injection behavior.
  - May miss code paths not reachable during testing.
- **Interactive application security testing (IAST)**
  - Combines runtime instrumentation with testing.
  - Can provide more context than SAST or DAST alone.
- **Software composition analysis (SCA)**
  - Identifies third-party and open-source dependencies and known vulnerabilities.
  - Also helps with license compliance.
- **Fuzz testing**
  - Sends malformed or unexpected inputs to discover crashes and logic flaws.
  - Useful for parsers, APIs, and file handlers.
- **Penetration testing**
  - Simulates attacker behavior to validate exploitability and impact.
  - Best used after other assurance activities, not as the only control.
- **Code review**
  - Manual review catches logic errors, insecure assumptions, and business-rule bypasses.
- **Formal verification**
  - Uses mathematical methods to prove properties of code or systems.
  - High assurance, but expensive and usually limited to critical components.
- **Unit, integration, and regression testing**
  - Unit tests validate small functions.
  - Integration tests validate component interactions.
  - Regression tests ensure fixes do not break existing behavior.

A practical exam distinction:

- **SAST** finds issues before execution.
- **DAST** finds issues during execution.
- **SCA** finds dependency and license issues.
- **Formal verification** proves properties, but usually only for narrow, critical code.

**Example assurance workflow**

1. Write security requirements and acceptance criteria.
2. Review architecture and threat model.
3. Run SAST and SCA in the build pipeline.
4. Execute unit and integration tests.
5. Run DAST against a staging environment.
6. Perform manual code review for high-risk changes.
7. Approve release only after defects and exceptions are resolved.

**Configuration management and change control**

Configuration management preserves integrity across code, infrastructure, and releases. It is essential for traceability and rollback.

Core concepts:

- **Version control**
  - Use systems like Git to track changes, branches, merges, and history.
  - Protect main branches with reviews and status checks.
- **Baseline management**
  - A baseline is an approved reference state for code, configuration, or infrastructure.
  - Changes should be measured against the baseline.
- **Change control**
  - Formal review and approval process for modifications.
  - Helps prevent unauthorized or poorly tested changes.
- **Release management**
  - Coordinates packaging, deployment, and rollback.
  - Ensures the right version reaches the right environment.

Good practices:

- Require pull requests and peer review for sensitive code.
- Use signed commits or protected branches where appropriate.
- Tag releases and keep immutable build artifacts.
- Track configuration as code when possible.
- Maintain rollback plans and tested restore procedures.

Example Git workflow:

```bash
git checkout -b feature/session-hardening
git add app/auth/session.py
git commit -m "Harden session timeout and cookie flags"
git push origin feature/session-hardening
```

Example branch protection concepts:

- Require at least one or two approvals.
- Require passing tests before merge.
- Block direct pushes to production branches.
- Record change tickets for high-risk releases.

**Data protection in development**

Sensitive data must be protected in code, test, and deployment workflows.

Key practices:

- **Encryption at rest and in transit**
  - Use strong, approved algorithms and protocols.
  - Protect databases, backups, and message queues.
- **Key management**
  - Store keys in a dedicated key management system or hardware security module (HSM) when appropriate.
  - Rotate keys according to policy.
  - Separate key access from application access.
- **Secrets management**
  - Store API keys, passwords, and tokens in a secrets vault, not in source code.
  - Inject secrets at runtime using environment variables or secret mounts.
- **Test data handling**
  - Use masked, synthetic, or tokenized data in non-production environments.
  - Avoid copying production data unless it is approved, minimized, and protected.
- **Environment separation**
  - Keep development, test, staging, and production isolated.
  - Prevent test systems from reaching production data unless explicitly controlled.

Example secret retrieval pattern:

```bash
export DB_PASSWORD="$(vault kv get -field=password secret/app/db)"
```

> Never commit secrets to source control, even in private repositories. Assume code can be copied, mirrored, or exposed later.

**Open-source and third-party component governance**

Modern applications depend on libraries, containers, frameworks, and services you did not write. That creates supply chain risk.

Important controls:

- **Dependency pinning**
  - Lock versions to reduce unexpected changes.
  - Use lockfiles and reproducible builds.
- **License compliance**
  - Verify that open-source licenses are compatible with your distribution and business model.
  - Track obligations such as attribution, disclosure, or copyleft requirements.
- **Vendor and package trust**
  - Prefer trusted registries and signed artifacts when available.
  - Review maintainer reputation and update history.
- **Supply chain risk management**
  - Watch for typosquatting, dependency confusion, malicious updates, and compromised maintainers.
- **Inventory**
  - Maintain a software bill of materials (SBOM) when possible.
  - Know what components are in the build.
- **Patch management**
  - Track vulnerabilities in dependencies and update on a risk-based schedule.

Common supply chain threats:

- Malicious package upload
- Tampered build artifact
- Compromised CI/CD pipeline
- Dependency confusion
- Unmaintained library with known vulnerabilities

**Secure deployment and operations**

Security does not end at code merge. Deployment and operations must preserve the intended security posture.

Key topics:

- **Environment separation**
  - Development, test, staging, and production should be isolated.
  - Production data should not be casually copied into lower environments.
- **Production hardening**
  - Remove debug endpoints, default credentials, sample apps, and unnecessary services.
  - Apply secure baselines to hosts, containers, and cloud services.
- **Blue-green deployment**
  - Maintain two production-like environments.
  - Switch traffic to the new version after validation.
  - Supports fast rollback.
- **Canary release**
  - Release to a small subset of users first.
  - Limits blast radius and helps detect issues early.
- **Rollback planning**
  - Keep previous versions and tested rollback procedures.
  - Roll back quickly when security or stability issues appear.
- **Monitoring and alerting**
  - Watch for authentication anomalies, error spikes, and suspicious traffic.
  - Feed logs into a SIEM (Security Information and Event Management) platform when appropriate.

Example deployment decision points:

- Use **blue-green** when you need near-instant rollback.
- Use **canary** when you want gradual exposure and real-user validation.
- Use **feature flags** when you want to decouple deployment from release.

**Common web and API security examples**

CISSP questions often describe a vulnerability and ask for the best control.

- **XSS**
  - Attackers inject script into content viewed by other users.
  - Prevent with output encoding, sanitization, and CSP.
- **CSRF**
  - Attackers trick a logged-in user’s browser into sending unwanted requests.
  - Prevent with anti-CSRF tokens and SameSite cookies.
- **SSRF**
  - Attackers cause the server to make requests to internal or sensitive endpoints.
  - Prevent with destination allowlists, network segmentation, and metadata service protections.
- **Broken access control**
  - Users access data or functions outside their authorization.
  - Prevent with server-side authorization checks and object-level access control.
- **Session fixation/hijacking**
  - Attackers reuse or steal session identifiers.
  - Prevent with session regeneration, secure cookies, and timeout controls.

Example secure authentication and authorization pattern:

```python
def get_invoice(user, invoice_id):
    invoice = db.fetch_invoice(invoice_id)
    if invoice.tenant_id != user.tenant_id:
        raise ForbiddenError()
    return invoice
```

This pattern matters because authentication alone does not prove authorization. A user can be validly logged in and still be unauthorized for a specific object.

**Exam-oriented distinctions**

- **Policy vs standard vs procedure**
  - Policy states intent.
  - Standard defines mandatory specifics.
  - Procedure explains how to do it.
- **SAST vs DAST**
  - SAST inspects code before execution.
  - DAST attacks the running app.
- **Threat modeling vs penetration testing**
  - Threat modeling predicts likely attacks.
  - Pen testing validates exploitability.
- **Baseline vs benchmark**
  - A baseline is your approved reference state.
  - A benchmark is often a comparison point or measurement standard.
- **Authentication vs authorization**
  - Authentication proves identity.
  - Authorization determines access rights.
- **Blue-green vs canary**
  - Blue-green swaps entire environments.
  - Canary exposes a small subset first.

**High-value exam reminders**

- Security is most effective when it starts in requirements and design.
- Threat modeling should happen before implementation and again when the design changes.
- Secure coding is necessary but not sufficient; governance, testing, and deployment controls matter too.
- Third-party dependencies are part of your attack surface.
- Production hardening and rollback planning are operational security controls, not optional extras.
- The best answer often emphasizes prevention, traceability, and least privilege over reactive fixes.
