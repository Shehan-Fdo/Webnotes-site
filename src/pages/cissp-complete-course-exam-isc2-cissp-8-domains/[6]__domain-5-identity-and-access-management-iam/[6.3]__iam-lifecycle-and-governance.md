**IAM governance overview**

Identity and access management (IAM) governance defines how identities are created, verified, granted access, reviewed, changed, and removed across their lifecycle. For CISSP, the focus is not just on tools, but on control objectives: least privilege, separation of duties, accountability, traceability, and timely deprovisioning.

A mature IAM program answers four questions:

- Who is the identity?
- What access should that identity have?
- Who approved it and why?
- When must it be reviewed or removed?

Governance spans people, process, and technology. In large enterprises, it often includes centralized policy with decentralized ownership. Security defines standards and risk controls, while business, data, and system owners typically approve access based on ownership and business need. Security may also approve or review high-risk, privileged, or exception-based access depending on policy.

**Identity governance model**

IAM governance can be centralized, decentralized, or hybrid.

- **Centralized IAM** uses one team or platform to manage identity policy, provisioning, and reviews.
  - Best for consistency, auditability, and scale.
  - Common in enterprises with shared services and standardized onboarding.
- **Decentralized IAM** lets business units manage more of their own access decisions.
  - Faster for local needs.
  - Higher risk of inconsistent controls and entitlement sprawl.
- **Hybrid IAM** centralizes policy and core controls while delegating approvals and application ownership.
  - Common in large organizations with many applications and regulatory obligations.

In practice, governance is usually layered:

- Corporate IAM team sets standards, workflows, and tooling.
- HR, procurement, or vendor management provides authoritative lifecycle triggers.
- Application owners define entitlements and approve access.
- Security and compliance monitor exceptions, privileged access, and review completion.

**Authoritative sources of identity**

An authoritative source is the system of record that establishes identity attributes and lifecycle events. HR is often the primary source for employees, but it is not the only one.

Common authoritative sources include:

- **HR systems** for employee hire, transfer, leave, and termination events.
- **Directory services** such as Active Directory (AD) or LDAP for account state and group membership.
- **CMDB** (Configuration Management Database) for asset ownership and service relationships.
- **Vendor management systems** for contractors and third parties.
- **Student information systems** in education environments.
- **External identity proofing** sources for remote or high-assurance onboarding.

When sources conflict, governance must define precedence. For example:

- HR may be authoritative for employment status.
- A directory may be authoritative for technical account state.
- An application owner may be authoritative for app-specific entitlements.
- A vendor sponsor may be authoritative for third-party access duration.

> If the source of truth is unclear, provisioning and deprovisioning become inconsistent, which creates orphaned accounts and audit findings.

**Identity lifecycle stages**

The identity lifecycle usually includes:

1. **Joiner**: account creation and baseline access at onboarding.
2. **Mover**: access changes due to role, department, location, or contract changes.
3. **Leaver**: access removal at termination, contract end, or role exit.
4. **Reverification**: periodic confirmation that the identity and access remain valid.
5. **Recovery**: identity restoration after lockout, compromise, or credential loss.

Each stage needs controls, approvals, and logging.

**Provisioning controls**

Provisioning is the creation and assignment of accounts, roles, groups, and entitlements. Good provisioning is driven by workflow, not manual ad hoc changes.

Typical enterprise targets include:

- **AD/LDAP** for on-premises directory accounts and group membership.
- **Cloud IAM** such as AWS IAM, Microsoft Entra ID, or Google Cloud IAM.
- **SaaS** platforms such as Okta, Salesforce, ServiceNow, or Microsoft 365.
- **Privileged access management** (PAM) systems for elevated credentials and just-in-time access.

A common automated flow uses HR as the trigger and SCIM for downstream provisioning.

```text
HR hire event
  -> IAM workflow
    -> create directory account
    -> assign baseline groups
    -> provision SaaS apps
    -> enroll MFA
    -> notify manager
```

SCIM, the System for Cross-domain Identity Management, is widely used to automate user and group lifecycle across SaaS platforms.

Example SCIM-style payload:

```json
{
  "userName": "jane.doe@example.com",
  "name": {
    "givenName": "Jane",
    "familyName": "Doe"
  },
  "active": true,
  "emails": [
    {
      "value": "jane.doe@example.com",
      "primary": true
    }
  ]
}
```

Provisioning should be role-based where possible, with manual exceptions tracked and reviewed.

**Access models and role engineering**

Role engineering reduces complexity by mapping access to business functions rather than individuals.

- **RBAC** (Role-Based Access Control) assigns access based on job role.
  - Good for stable, repeatable access patterns.
  - Example: `Finance_AP_Clerk` gets invoice entry and vendor lookup.
- **ABAC** (Attribute-Based Access Control) evaluates attributes such as department, location, device trust, time, or data sensitivity.
  - Good for dynamic environments and fine-grained policy.
  - Example: allow access only if `department=Legal` and `device_compliant=true`.
- **Rule-based access** uses explicit rules, often based on conditions like time of day, network zone, or transaction type.
  - Useful for constrained workflows and legacy systems.

Role engineering goals:

- Minimize unique entitlements.
- Eliminate privilege creep.
- Group access by function, not by person.
- Keep roles understandable to auditors and owners.

A practical RBAC example in Linux-like systems:

```bash
# Create a role group for database operators
groupadd db-operators
usermod -aG db-operators alice
usermod -aG db-operators bob
```

For CISSP, remember that roles are governance constructs as much as technical ones. A role should have an owner, a business purpose, and a review cadence.

**Access request types and approvals**

Not all access requests are equal. Governance should distinguish request classes and approval paths.

- **Standard access**
  - Preapproved, low-risk access tied to a role.
  - Often auto-provisioned after manager approval or workflow validation.
- **Elevated access**
  - Higher-risk access such as admin rights, production access, or sensitive data access.
  - Usually requires owner approval, security review, and sometimes change management.
- **Emergency access**
  - Break-glass access for urgent incidents.
  - Time-limited, heavily logged, and reviewed after use.
- **Third-party access**
  - Vendor, partner, or contractor access.
  - Requires sponsor, expiration, and tighter monitoring.
- **Temporary access**
  - Granted for a project, leave coverage, or short-term task.
  - Must expire automatically.

A good approval model separates responsibilities:

- Manager approves business need.
- Data or system owner approves access to the resource.
- Security reviews policy exceptions, privileged access, or high-risk cases.
- Compliance may require evidence for regulated systems.

This is not “security never approves access.” Rather, security usually defines control requirements and may approve exceptions or privileged access depending on policy.

**Provisioning and deprovisioning in common platforms**

Enterprise IAM often spans multiple control planes.

- **Active Directory / LDAP**
  - Used for authentication, group membership, and policy enforcement in many on-prem environments.
  - Deprovisioning should disable the account, remove group memberships, and revoke related access paths.
- **Microsoft Entra ID**
  - Common for cloud identity, SSO, conditional access, and SaaS provisioning.
  - Supports lifecycle automation, access reviews, and privileged identity management.
- **AWS IAM**
  - Uses users, groups, roles, and policies.
  - Best practice is to prefer roles and federation over long-lived IAM users.
- **Okta-style identity platforms**
  - Centralize SSO, lifecycle workflows, MFA, and app provisioning.
  - Often integrate with HR and SCIM-enabled SaaS.

Example AWS CLI commands for lifecycle control:

```bash
aws iam delete-login-profile --user-name alice
aws iam detach-user-policy --user-name alice --policy-arn arn:aws:iam::123456789012:policy/DeveloperAccess
aws iam remove-user-from-group --user-name alice --group-name Developers
aws iam delete-access-key --user-name alice --access-key-id AKIAEXAMPLE123456
```

For cloud governance, prefer:

- Federation over local accounts.
- Short-lived credentials over static keys.
- Role assumption over direct privilege assignment.
- Automated deprovisioning over manual cleanup.

**Deprovisioning, suspension, disablement, and deletion**

CISSP often tests the difference between account states.

- **Suspend**
  - Temporarily stop access while preserving the identity record.
  - Useful during investigation, leave, or pending review.
- **Disable**
  - Prevent interactive use of the account.
  - Often used for termination or policy violations.
- **Lock**
  - Usually blocks authentication after failed attempts or administrative action.
  - May not stop all access methods if other credentials or tokens remain valid.
- **Expire**
  - Set an end date after which the account is no longer valid.
  - Common for contractors and temporary staff.
- **Delete**
  - Remove the account record after retention requirements are met.
  - Usually the last step, not the first.

> Password locking alone is not a complete offboarding control. You must also revoke active sessions, tokens, keys, certificates, and alternate authentication paths.

A practical offboarding sequence is:

1. Disable or suspend the account.
2. Revoke sessions, tokens, API keys, certificates, and MFA factors as applicable.
3. Remove group memberships and role assignments.
4. Transfer ownership of data, mailboxes, and shared resources.
5. Preserve logs and evidence.
6. Delete the account only after retention and legal hold requirements are satisfied.

Example Linux-oriented controls:

```bash
# Disable password-based login and expire the account
passwd -l alice
chage -E 0 alice

# Check account status
passwd -S alice
chage -l alice
```

Use these carefully: password locking does not stop SSH keys, Kerberos tickets, SSO sessions, or other non-password methods by itself.

**Third-party, contractor, and guest lifecycle**

Third-party identities need tighter governance because they often bypass normal employee controls.

Key controls include:

- **Sponsor assignment**
  - A business sponsor owns the need for access.
  - The sponsor confirms continued business justification.
- **Expiration**
  - Access should end automatically on contract end or project completion.
- **Scope limitation**
  - Grant only the minimum systems and data required.
- **Segregation**
  - Separate third-party access from internal admin paths where possible.
- **Monitoring**
  - Review activity more frequently than employee access.

For guests and partners:

- Use time-bound accounts.
- Require MFA where feasible.
- Restrict access to specific applications or portals.
- Avoid broad network access unless justified.

**Password lifecycle and account recovery**

Password governance is broader than password complexity.

It includes:

- Initial enrollment and secure delivery.
- Reset and recovery workflows.
- MFA enrollment and reset.
- Identity verification before reset.
- Compromise response.
- Inactivity handling.

Good recovery controls require proof of identity before resetting credentials. For high-risk accounts, use stronger verification than knowledge-based questions, which are often weak.

Common controls:

- Help desk identity verification with documented procedure.
- Manager approval for certain resets.
- Out-of-band verification for privileged users.
- Forced password change after reset.
- MFA re-enrollment after suspected compromise.

Example password policy commands on Linux:

```bash
# Force password change at next login
chage -d 0 alice

# Set maximum password age to 90 days
chage -M 90 alice

# Set warning period to 14 days
chage -W 14 alice
```

For CISSP, understand that password policy is only one part of credential governance. Session tokens, API keys, recovery codes, and certificates also need lifecycle control.

**Federation and trust frameworks**

Federation lets one organization trust identity assertions from another system. The exact roles depend on the protocol.

- In **SAML** (Security Assertion Markup Language):
  - An identity provider (IdP) authenticates the user and issues an assertion.
  - A service provider (SP) consumes the assertion and grants access.
- In **OAuth 2.0**:
  - The focus is authorization delegation, not primary authentication.
  - An authorization server issues tokens to a client for access to a resource server.
- In **OpenID Connect (OIDC)**:
  - OIDC adds authentication on top of OAuth 2.0.
  - The identity provider role is often implemented by the authorization server.
  - The relying party or client consumes identity claims.

This protocol distinction matters. Do not assume the SAML IdP/SP mapping applies unchanged to OAuth or OIDC.

Typical federation benefits:

- Fewer passwords.
- Centralized authentication policy.
- Better SSO (Single Sign-On) user experience.
- Easier deprovisioning when the identity source is authoritative.

Typical risks:

- Trust misconfiguration.
- Overly broad token scopes.
- Weak assertion validation.
- Session persistence after account disablement if tokens are not revoked.

Example SAML-style trust relationship:

```text
User -> IdP authenticates -> SAML assertion -> SP validates signature -> SP grants session
```

Example OIDC flow conceptually:

```text
User -> Authorization Server authenticates -> ID token + access token -> Client uses token -> Resource Server authorizes access
```

**Access reviews and recertification**

Periodic access review is a core governance control. It confirms that access remains appropriate over time.

Review scope should vary by risk:

- **Privileged accounts**: review more frequently, often monthly or quarterly.
- **Contractors and third parties**: review at shorter intervals and at contract milestones.
- **Sensitive data access**: review more often than standard business access.
- **Standard employee access**: review quarterly, semiannually, or annually depending on risk and policy.

Reviewers should validate:

- Identity still exists and is active.
- Access matches job function.
- Privileges are still needed.
- Exceptions are documented.
- Dormant or orphaned access is removed.

Useful metrics include:

- Recertification completion rate.
- Number of overdue reviews.
- Orphaned accounts discovered.
- Stale entitlements removed.
- Deprovisioning SLA compliance.
- Number of emergency access events.
- Percentage of access granted through standard roles versus exceptions.

A simple review workflow:

1. Generate entitlement report.
2. Send to owner, manager, or sponsor.
3. Review each access item.
4. Approve, revoke, or escalate exceptions.
5. Record evidence and remediation dates.
6. Track completion and overdue items.

**Monitoring and governance metrics**

IAM governance needs measurable controls, not just policy statements.

Track:

- **Orphaned accounts**
  - Accounts with no known owner or no valid employment relationship.
- **Stale entitlements**
  - Access not used for a defined period but still assigned.
- **Dormant accounts**
  - Accounts inactive beyond policy thresholds.
- **Privilege creep**
  - Accumulated access beyond current job needs.
- **Provisioning SLA**
  - Time from approved request to access availability.
- **Deprovisioning SLA**
  - Time from termination trigger to access removal.
- **Review completion**
  - Percentage of access reviews completed on time.

Example audit query concept for stale accounts:

```sql
SELECT username, last_login, status
FROM iam_accounts
WHERE status = 'active'
  AND last_login < CURRENT_DATE - INTERVAL '90 days';
```

Monitoring should feed remediation, not just reporting. If a metric is consistently poor, the workflow or source system is likely broken.

**Common enterprise scenarios**

In a cloud-first enterprise:

- HR termination triggers an IAM workflow.
- Entra ID disables the user and revokes sessions.
- SCIM removes the user from SaaS apps.
- AWS roles are removed or federation access is blocked.
- Access reviews confirm no lingering privileged assignments.

In a regulated environment:

- Access to production and sensitive data requires owner approval and sometimes security review.
- Emergency access uses break-glass accounts with logging and post-event review.
- Contractors receive time-limited access with sponsor accountability.
- Quarterly recertification is common for privileged and sensitive systems.

In a merger or acquisition:

- Identity sources may conflict.
- Governance must define which directory or HR system is authoritative during transition.
- Temporary federation or trust bridging may be used while systems are consolidated.
- Access reviews become critical because inherited entitlements are often excessive.

**Exam distinctions to remember**

- **Authentication** proves identity; **authorization** determines access.
- **Provisioning** creates access; **deprovisioning** removes it.
- **Locking** a password is not the same as disabling the account.
- **Expiration** is time-based; **disablement** is administrative.
- **Federation** is trust between identity domains; protocol details matter.
- **RBAC** is role-driven; **ABAC** is attribute-driven.
- **Security** often defines policy and reviews exceptions, while owners approve access based on ownership and risk.

**Practical command examples**

Linux account lifecycle example:

```bash
# Create a user
useradd -m alice

# Set password
passwd alice

# Lock account for investigation
passwd -l alice

# Expire account immediately
chage -E 0 alice

# Force password change on next login after reactivation
chage -d 0 alice
```

AWS IAM cleanup example:

```bash
aws iam list-access-keys --user-name alice
aws iam delete-access-key --user-name alice --access-key-id AKIAEXAMPLE123456
aws iam list-attached-user-policies --user-name alice
aws iam detach-user-policy --user-name alice --policy-arn arn:aws:iam::123456789012:policy/ReadOnlyAccess
```

SCIM lifecycle concept:

```http
POST /scim/v2/Users
PATCH /scim/v2/Users/2819c223-7f76-453a-919d-413861904646
DELETE /scim/v2/Users/2819c223-7f76-453a-919d-413861904646
```

**Key takeaways**

IAM lifecycle and governance is about controlling identity from creation to retirement with clear ownership, automation, and review. The strongest programs use authoritative sources, role engineering, automated provisioning, time-bound access, periodic recertification, and complete deprovisioning that revokes all access paths, not just passwords.
