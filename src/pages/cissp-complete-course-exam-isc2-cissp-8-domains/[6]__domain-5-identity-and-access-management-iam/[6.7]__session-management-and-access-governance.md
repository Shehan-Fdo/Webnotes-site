## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Session management fundamentals**

A session is the temporary security context that ties a user, device, or service to authenticated access after login. In CISSP scenarios, you evaluate how the session is created, maintained, protected, and terminated across web apps, mobile apps, APIs, and privileged administrative tools.

Session controls matter because authentication is not a one-time event. After login, the application must continue to prove that the requester is still authorized, that the session has not been hijacked, and that the risk level still matches the action being performed.

- **Authentication** proves identity at login; session management preserves that identity across requests.
- **Authorization** can change during a session if roles, entitlements, or risk posture change.
- **Confidentiality** and **integrity** of session state are the main goals of session protection.
- **Availability** matters too, because overly aggressive timeouts can break business workflows.

**Session lifecycle and core controls**

A secure session lifecycle usually includes creation, validation, renewal, inactivity handling, and termination. The implementation differs by platform, but the security goals stay the same: prevent fixation, prevent theft, limit replay, and ensure invalid sessions cannot be reused.

Common controls include:

- **Session identifiers** that are random, unpredictable, and unique.
- **Secure cookies** for browser-based sessions.
- **Idle timeout** to end sessions after inactivity.
- **Absolute timeout** to cap total session lifetime.
- **Reauthentication** for sensitive actions such as payment changes, privilege elevation, or export of regulated data.
- **Server-side invalidation** so logout and revocation actually terminate access.

A practical web session cookie should usually include flags like `Secure`, `HttpOnly`, and `SameSite`:

```http
Set-Cookie: sessionid=8f3a9c2d7e4b1a0f; Path=/; Secure; HttpOnly; SameSite=Lax
```

- **Secure** sends the cookie only over HTTPS.
- **HttpOnly** blocks JavaScript access and reduces theft from script injection.
- **SameSite** reduces cross-site request forgery (CSRF) risk by limiting cross-site cookie use.
- **Path** and `Domain` should be as narrow as possible to reduce exposure.

> A cookie without `Secure` or `HttpOnly` is a common exam red flag because it is easier to intercept or steal.

**Session timeout tuning by environment**

CISSP questions often ask you to choose the right timeout type for the environment. The best answer depends on user risk, device exposure, and business impact.

- **Web sessions**
  - Use short idle timeouts for public or shared devices.
  - Use longer idle timeouts for low-risk internal portals if business needs justify it.
  - Use absolute timeouts to force periodic reauthentication even if the user stays active.
  - Use step-up authentication for sensitive transactions rather than forcing full logout for every action.

- **Mobile sessions**
  - Mobile apps often need longer idle windows because users switch apps frequently.
  - Prefer token refresh with device binding or app attestation where available.
  - Use biometric or PIN reauthentication for high-risk actions instead of constant full login.
  - Assume the device may be lost, rooted, or jailbroken, so token storage must be protected by the platform keystore.

- **Privileged/admin sessions**
  - Use the shortest practical idle timeout.
  - Use absolute timeout plus reauthentication for privileged commands.
  - Require step-up authentication before administrative changes.
  - Record or proxy the session when the action affects critical systems.

How to choose timeout types in exam scenarios:

1. Use **idle timeout** when the main risk is unattended access.
2. Use **absolute timeout** when you need a hard upper bound on session lifetime.
3. Use **reauthentication timeout** when the action is sensitive, even if the session is still active.
4. Use a combination when the environment is high risk, such as admin consoles or regulated data systems.

A common pattern is:

- Idle timeout for inactivity.
- Absolute timeout for maximum age.
- Reauthentication for step-up actions.

**Token-based session management**

Modern applications often use tokens instead of classic server-stored sessions. The most common pattern is an access token plus a refresh token.

- **Access token**
  - Short-lived credential used to call APIs.
  - Often a bearer token, meaning possession is enough to use it.
  - Should expire quickly to reduce replay risk.

- **Refresh token**
  - Longer-lived credential used to obtain new access tokens.
  - Should be stored more carefully than access tokens.
  - Should be revocable and rotated when possible.

A typical OAuth 2.0 / OpenID Connect flow uses:

- Short-lived access tokens for API calls.
- Refresh tokens for session continuity.
- Revocation or rotation when compromise is suspected.

Example token refresh request:

```bash
curl -X POST https://idp.example.com/oauth/token \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  -d 'grant_type=refresh_token&refresh_token=eyJhbGciOi...&client_id=webapp'
```

Important token security concepts:

- **Bearer-token risk** means anyone who steals the token can use it until it expires or is revoked.
- **Replay** is a major concern because bearer tokens are reusable.
- **Token binding** and **DPoP** (Demonstration of Proof-of-Possession) reduce replay by tying token use to a client-held key.
- **Audience** and **issuer** validation prevent tokens from being accepted by the wrong service.
- **Scope** limits what the token can do.

JWT session pitfalls are common in modern web apps:

- A JWT (JSON Web Token) is often self-contained, so the server may not check a session table on every request.
- Stateless validation improves scale, but it makes immediate invalidation harder.
- If a JWT is stolen, it may remain valid until expiration unless you add revocation controls.
- Long-lived JWTs are risky because they behave like portable bearer credentials.

Server-side revocation strategies for distributed systems include:

- **Short token lifetimes** so stolen tokens age out quickly.
- **Revocation lists / blacklists** for known-compromised tokens.
- **Token versioning** tied to the user record so password resets or role changes invalidate older tokens.
- **Session stores** in Redis or similar shared caches for centralized invalidation.
- **Introspection endpoints** so resource servers can check token status with the identity provider.
- **Event-driven logout** to propagate revocation across services.

Example of a server-side session store pattern:

```yaml
session:
  backend: redis
  ttl_seconds: 1800
  absolute_ttl_seconds: 28800
  cookie_name: sessionid
  cookie_secure: true
  cookie_httponly: true
  cookie_samesite: Lax
```

**Session attacks and defenses**

CISSP exam items often test whether you can identify the attack and choose the right control.

- **Session hijacking**
  - Attacker steals a valid session identifier and impersonates the user.
  - Defend with HTTPS, secure cookies, short lifetimes, and device/risk checks.

- **Session fixation**
  - Attacker forces a known session ID before login and reuses it after authentication.
  - Defend by regenerating the session ID after successful login and privilege changes.

- **Session sidejacking**
  - Attacker captures session traffic on an unencrypted or weakly protected network.
  - Defend with TLS, HSTS, and secure transport on all endpoints.

- **Cookie poisoning**
  - Attacker modifies cookie values to alter application behavior or escalate privileges.
  - Defend with server-side validation, signed cookies, and integrity checks.

- **Clickjacking-related session abuse**
  - Attacker tricks a user into clicking hidden UI elements in an authenticated session.
  - Defend with `X-Frame-Options`, `Content-Security-Policy: frame-ancestors`, and anti-CSRF controls.

- **Cross-site request forgery (CSRF)**
  - Attacker causes a browser to send authenticated requests without user intent.
  - Defend with anti-CSRF tokens, SameSite cookies, and origin checks.

Example hardening headers:

```http
Content-Security-Policy: frame-ancestors 'none'
X-Frame-Options: DENY
Strict-Transport-Security: max-age=31536000; includeSubDomains
```

**Practical web and reverse proxy configuration**

Session security often depends on the web server, reverse proxy, and application all working together.

Nginx example:

```nginx
server {
    listen 443 ssl http2;
    server_name app.example.com;

    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    add_header X-Frame-Options "DENY" always;
    add_header Content-Security-Policy "frame-ancestors 'none'" always;

    location / {
        proxy_pass http://app_backend;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-Proto https;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

Apache example:

```apache
Header always set Strict-Transport-Security "max-age=31536000; includeSubDomains"
Header always set X-Frame-Options "DENY"
Header always set Content-Security-Policy "frame-ancestors 'none'"
```

Reverse proxy and load balancer considerations:

- **Session affinity** can keep a user pinned to one backend, which helps if the app stores sessions locally.
- **Shared session stores** are better for horizontal scaling and failover.
- **Sticky sessions** can become a single point of failure if the backend dies.
- **Distributed logout** requires a shared revocation mechanism, not just local memory.

**Mobile and API session considerations**

Mobile apps and APIs often use token-based authentication rather than browser cookies. The main risks are token theft from device compromise, insecure local storage, and overlong token lifetimes.

Good practices include:

- Store secrets in the platform **keystore** or secure enclave when possible.
- Avoid embedding long-lived secrets in the app binary.
- Use refresh token rotation and revoke the old token after each use.
- Bind tokens to device or client context when supported.
- Require reauthentication for high-risk actions, even if the access token is still valid.

For APIs, prefer:

- Short-lived access tokens.
- Audience-restricted tokens.
- mTLS (mutual TLS) or proof-of-possession where appropriate.
- Centralized revocation and audit logging.

**Access governance overview**

Access governance is the policy and process layer that ensures the right identities have the right access for the right reasons, and only for as long as needed. It connects identity lifecycle, role design, entitlement review, and audit evidence.

Core governance goals:

- Enforce least privilege.
- Support segregation of duties (SoD).
- Provide traceable approvals and evidence.
- Remove access when it is no longer justified.
- Detect and remediate privilege creep.

**Identity lifecycle and account governance**

A mature program follows the joiner-mover-leaver model:

- **Joiner**
  - New hire or new identity creation.
  - Provision only the access needed for the initial role.
  - Use standard role templates where possible.

- **Mover**
  - Role, team, location, or manager changes.
  - Reevaluate access and remove obsolete entitlements.
  - Avoid accumulating access from prior roles.

- **Leaver**
  - Termination, resignation, contract end, or death.
  - Disable accounts promptly.
  - Revoke tokens, sessions, keys, and badges.
  - Transfer ownership of shared resources and mailboxes as policy allows.

Deprovisioning workflow should include:

1. Disable interactive login immediately.
2. Revoke active sessions and refresh tokens.
3. Remove or disable privileged roles.
4. Rotate shared secrets if the user had access to them.
5. Archive evidence and preserve logs per retention policy.

Orphaned and dormant accounts are governance risks:

- **Orphaned account** has no clear owner or business justification.
- **Dormant account** has not been used for a defined period.
- Both should be reviewed, disabled, or removed according to policy.

**Access control models and entitlement management**

Access governance aligns with the access control model used by the organization.

- **RBAC** (Role-Based Access Control)
  - Access is assigned by job role.
  - Good for standardization and scale.
  - Role mining helps identify common entitlement patterns.

- **ABAC** (Attribute-Based Access Control)
  - Access decisions use attributes such as department, device posture, location, and time.
  - Good for dynamic and context-aware decisions.

- **Rule-based access**
  - Access is granted by explicit rules, such as time-of-day or network zone.
  - Often used alongside RBAC or ABAC.

Entitlement management focuses on the specific permissions attached to an identity:

- Application roles.
- Database privileges.
- Cloud IAM policies.
- Shared folder access.
- API scopes.

Role mining is useful when you need to reduce entitlement sprawl, but it should not replace business review. A mined role still needs validation against actual job duties and SoD constraints.

**Privileged access management**

Privileged access management (PAM) protects high-impact accounts such as domain admins, cloud admins, database admins, and emergency break-glass accounts.

Key PAM controls:

- **Just-in-time (JIT)** access grants privilege only when needed and for a limited duration.
- **Just-enough access** gives only the minimum permissions required for the task.
- **Approval workflows** require authorization before elevation.
- **Session recording** captures privileged activity for audit and investigation.
- **Credential vaulting** stores privileged secrets securely and rotates them regularly.
- **Check-out / check-in** workflows reduce standing privilege.

A typical privileged workflow:

1. User requests elevation for a specific task.
2. Manager or system policy approves the request.
3. PAM grants time-limited access.
4. Session is proxied or recorded.
5. Access expires automatically and evidence is retained.

PAM is especially important for break-glass accounts:

- Keep them tightly controlled.
- Monitor use continuously.
- Test them periodically.
- Store credentials offline or in a hardened vault.
- Review every use after the incident.

**Access certification and recertification**

Access certification is the periodic review of who has access and whether that access is still justified. CISSP questions often focus on governance quality, not just the existence of a review.

Good certification practices include:

- **Reviewer independence** so the approver is not the same person who requested the access.
- **Evidence-based attestation** using role, ticket, business justification, and usage data.
- **Exception handling** for temporary business needs, with expiration dates.
- **Remediation tracking** to ensure revoked access is actually removed.
- **Escalation paths** when reviewers do not respond.

Common review scopes:

- User access to business applications.
- Privileged access.
- Service accounts.
- Third-party/vendor access.
- Dormant or unused entitlements.

A strong review process does not just ask “Do you approve this?” It asks:

- Is the access still needed?
- Is the role still correct?
- Does this violate SoD?
- Is there a compensating control?
- Has the exception expired?

**Audit, compliance, and control objectives**

Access governance supports audit and compliance by producing evidence that controls operate consistently.

Typical control objectives include:

- **Least privilege** enforcement.
- **Segregation of duties** to prevent fraud or self-approval.
- **Periodic access review** and recertification.
- **Timely deprovisioning** after termination or role change.
- **Evidence retention** for investigations and audits.
- **Control testing** to verify that reviews, revocations, and logging actually work.

Useful evidence artifacts:

- Access request tickets.
- Approval records.
- Session logs and audit trails.
- Privileged session recordings.
- Revocation timestamps.
- Recertification reports.
- Exception registers with expiry dates.

Retention should match legal, regulatory, and business requirements. If logs are too short-lived, you may fail an audit or lose forensic evidence.

**Practical governance examples**

Example IAM policy concept for role-based access:

```json
{
  "role": "finance_analyst",
  "permissions": [
    "read:ledger",
    "export:reports"
  ],
  "conditions": {
    "mfa_required": true,
    "device_compliant": true
  }
}
```

Example deprovisioning automation outline:

```bash
#!/usr/bin/env bash
user="$1"

disable_account "$user"
revoke_sessions "$user"
revoke_refresh_tokens "$user"
remove_group_memberships "$user"
log_action "$user" "deprovisioned"
```

Example session invalidation strategy for a distributed app:

- Store session state in Redis with TTL.
- Keep a token revocation list for compromised JWT IDs (`jti`).
- Rotate signing keys carefully and publish key IDs (`kid`).
- Force reauthentication after password reset or privilege change.

**Exam decision cues**

When you see a CISSP question, look for the control that best matches the risk:

- Choose **idle timeout** for unattended workstation risk.
- Choose **absolute timeout** for maximum session age.
- Choose **reauthentication** for sensitive actions.
- Choose **token rotation** and **revocation** for bearer-token compromise.
- Choose **PAM** for admin and high-impact access.
- Choose **recertification** for periodic governance validation.
- Choose **deprovisioning** for leavers and dormant accounts.
- Choose **RBAC** for stable job-based access, **ABAC** for dynamic context-based access.

A good answer usually balances security and usability. Overly strict controls can create workarounds, while weak controls create persistent access risk.

**Quick reference**

- **Session management** protects authenticated state after login.
- **Bearer tokens** are powerful but risky because possession equals access.
- **JWTs** scale well but complicate immediate revocation.
- **PAM** reduces standing privilege and improves accountability.
- **Joiner-mover-leaver** processes keep access aligned with employment status.
- **Recertification** proves access is still justified.
- **SoD** prevents one identity from controlling conflicting steps in a process.
