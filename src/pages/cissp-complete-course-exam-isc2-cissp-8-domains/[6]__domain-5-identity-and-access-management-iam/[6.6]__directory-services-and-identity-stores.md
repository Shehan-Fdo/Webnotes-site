**Directory services and identity stores: core purpose**

Directory services centralize identity data so systems can authenticate users, authorize access, and apply policy consistently. In CISSP terms, they support **identification**, **authentication**, **authorization**, and **accountability** across on-premises, cloud, and hybrid environments.

A directory is not just a database of users. It is a structured identity system that supports:
- Fast lookup of accounts, groups, devices, and service principals
- Policy enforcement through group membership and attributes
- Authentication protocols and trust relationships
- Delegated administration and lifecycle management

Common examples include:
- **Microsoft Active Directory Domain Services (AD DS)** for Windows-centric enterprise identity
- **LDAP**-based directories such as OpenLDAP or 389 Directory Server
- **Cloud directories** such as Microsoft Entra ID, which focus on cloud authentication and federation
- **RADIUS** and **TACACS+** as related identity/authentication stores for network access and device administration

**Directory fundamentals and terminology**

A directory service stores objects in a hierarchical namespace and exposes them through a query protocol or authentication interface. The most common directory model is LDAP, but CISSP expects you to understand the broader identity architecture.

Key terms:
- **Object**: An entry such as a user, group, computer, printer, or service account
- **Attribute**: A property of an object, such as `cn`, `mail`, `memberOf`, or `uidNumber`
- **Schema**: The rules that define object classes and allowed attributes
- **DN (Distinguished Name)**: The unique path to an object, such as `cn=Alice,ou=Users,dc=example,dc=com`
- **RDN (Relative Distinguished Name)**: The local name component of a DN
- **OU (Organizational Unit)**: A container used for delegation and policy scoping in AD
- **Group**: A collection of principals used for authorization
- **Service account**: A non-human identity used by applications or services

A directory is optimized for reads and lookups, not transactional updates. That makes it ideal for authentication and authorization decisions, but not for application data storage.

**LDAP, X.500, and directory structure**

LDAP (Lightweight Directory Access Protocol) is the dominant protocol for querying and modifying directory data. It was designed as a lighter interface to X.500-style directory concepts.

Important structural concepts:
- **Naming context**: A contiguous subtree of directory data
- **Directory partition**: In AD, a logical division of the directory database
- **Base DN**: The starting point for a search
- **Search scope**: `base`, `one`, or `sub`
- **Filter**: A query expression such as `(sAMAccountName=alice)`

In Active Directory, directory partitions are commonly described as:
- **Domain partition**: User, group, and computer objects for a domain
- **Configuration partition**: Forest-wide configuration data
- **Schema partition**: Object classes and attributes
- **Application partitions**: Custom data replicated to selected domain controllers

X.500 concepts still matter because they explain how directories organize data:
- **Naming contexts** define replication and search boundaries
- **Aliases** provide alternate names for objects
- **Referrals** point a client to another directory server or naming context when the requested data is not held locally

Referrals and aliases are often confused:
- **Alias**: A directory object that represents another object
- **Referral**: A server response that tells the client where to continue the search

In LDAP, referrals help clients locate data across partitions or servers. In X.500-style systems, aliases can simplify naming, but they can also create ambiguity if not managed carefully.

**Active Directory architecture**

Active Directory Domain Services is the most common enterprise directory in Windows environments. It combines directory storage, authentication, policy, and replication.

Core AD components:
- **Domain Controller (DC)**: A server that hosts the directory and authenticates users
- **Forest**: The top-level security boundary containing one or more domains
- **Tree**: A set of domains sharing a contiguous DNS namespace
- **Domain**: Administrative and replication boundary for objects and policies
- **Site**: A set of IP subnets used to optimize replication and logon traffic
- **Global Catalog (GC)**: A partial, searchable replica of objects from all domains in the forest

The **Global Catalog** is especially important:
- It stores a subset of attributes for all objects in the forest
- It supports forest-wide searches and universal group membership lookups
- It helps users log on when they belong to groups from multiple domains
- It reduces the need to query remote domain controllers for common searches

> A GC outage can slow logons, break universal group resolution, and disrupt forest-wide searches even if individual domain controllers are still online.

Replication and topology matter because directory services are infrastructure dependencies. If replication is delayed or broken, authentication and authorization can become inconsistent across sites.

**Authentication mechanisms: NTLM vs Kerberos**

Windows environments commonly use **Kerberos** and, in legacy or fallback cases, **NTLM (NT LAN Manager)**.

Kerberos is the preferred protocol because it is stronger, more scalable, and supports mutual authentication. NTLM remains relevant for compatibility, but it is weaker and more vulnerable to relay and pass-the-hash style abuse.

Comparison:
- **Kerberos**
  - Uses tickets instead of sending passwords repeatedly
  - Supports mutual authentication
  - Scales well in domain environments
  - Depends on time synchronization and a Key Distribution Center (KDC)
- **NTLM**
  - Uses challenge-response authentication
  - Does not provide the same level of mutual authentication
  - Is more susceptible to relay attacks
  - Often appears when Kerberos cannot be used

Kerberos ticket flow:
1. The client sends an **AS-REQ (Authentication Service Request)** to the KDC to request a Ticket Granting Ticket (TGT).
2. The KDC replies with an **AS-REP (Authentication Service Reply)** containing the TGT and session key, encrypted appropriately.
3. The client uses the TGT to request access to a service by sending a **TGS-REQ (Ticket Granting Service Request)**.
4. The KDC returns a **TGS-REP (Ticket Granting Service Reply)** with a service ticket.
5. The client presents the service ticket to the target service for access.

Kerberos concepts you should know:
- **KDC (Key Distribution Center)**: The service that issues tickets
- **TGT (Ticket Granting Ticket)**: Proof that the user authenticated to the KDC
- **Service ticket**: Ticket for a specific service
- **SPN (Service Principal Name)**: The Kerberos identity for a service
- **Mutual authentication**: Both client and server verify each other
- **Clock skew**: Time mismatch that can break ticket validation

Practical implication:
- If Kerberos fails because of DNS, time drift, or SPN issues, clients may fall back to NTLM.
- Reducing NTLM usage improves security posture.

**Delegation, trust, and authorization design**

Directory services are not only about login. They also define how access is delegated and how trust is established across boundaries.

Common delegation and trust concepts:
- **OU delegation**: Assign administrative control over a subset of objects without granting domain-wide rights
- **Trust relationship**: A mechanism that allows authentication or authorization across domains or forests
- **One-way trust**: One domain trusts another, but not vice versa
- **Two-way trust**: Both domains trust each other
- **Transitive trust**: Trust can extend through intermediate trusted domains
- **Non-transitive trust**: Trust is limited to the directly trusted domain

Delegation types in Kerberos:
- **Unconstrained delegation**: A service can impersonate a user to any service after receiving a ticket; this is high risk
- **Constrained delegation**: A service can impersonate a user only to specific services
- **Resource-based constrained delegation (RBCD)**: The target resource controls which principals may delegate to it

Constrained delegation is preferred because it limits blast radius. Use it when a front-end service must access a back-end service on behalf of a user, such as a web application calling a database.

**Identity stores beyond AD and LDAP**

Not all identity stores are full directories. CISSP expects you to distinguish between directories, authentication stores, and federation services.

Examples:
- **RADIUS**
  - Centralized authentication, authorization, and accounting for network access
  - Common for VPNs, Wi-Fi, and network access control
  - Often integrates with a directory for user validation
- **TACACS+**
  - Common for device administration
  - Separates authentication, authorization, and accounting more cleanly than RADIUS
  - Often used for router and switch management
- **Entra ID / Azure AD**
  - Cloud identity provider focused on authentication, SSO, conditional access, and federation
  - Not a direct replacement for on-prem AD DS in all cases
  - Uses modern protocols such as SAML, OAuth 2.0, and OpenID Connect
- **Identity governance tools**
  - Manage access reviews, role lifecycle, and entitlement certification
  - They complement directories rather than replace them

Use the right tool for the job:
- Use a **directory** for identity lookup and group-based authorization
- Use **federation** for cross-domain or cross-organization SSO
- Use **identity governance** for access certification, role mining, and lifecycle controls
- Use **RADIUS/TACACS+** for network and device access control

**Federation and cloud identity**

Federation lets one identity provider assert identity to another service provider. This reduces password sprawl and supports single sign-on (SSO).

Common federation roles:
- **Identity Provider (IdP)**: Authenticates the user and issues assertions or tokens
- **Service Provider (SP)**: Consumes the assertion and grants access
- **Relying Party**: The application or service that trusts the IdP

Common protocols:
- **SAML** for enterprise web SSO
- **OAuth 2.0** for delegated authorization
- **OpenID Connect (OIDC)** for authentication built on OAuth 2.0

Cloud identity introduces lifecycle edge cases:
- **Source of authority**: The system that owns the master copy of an identity attribute
- **Attribute precedence**: Rules that determine whether on-premises or cloud values win during sync
- **Synchronization conflicts**: Changes in one system may overwrite or be blocked by another
- **Deprovisioning delays**: Accounts may remain active temporarily after HR or IAM changes
- **Orphaned accounts**: Accounts that remain after the user leaves or the source record is removed

Hybrid identity requires careful design:
- Decide which attributes are mastered on-premises and which are mastered in the cloud
- Avoid duplicate or conflicting identities
- Monitor sync health and stale accounts
- Define offboarding workflows that disable access quickly across all connected systems

**Password storage and credential protection**

Directories store credentials in protected form, not as plaintext. The exact mechanism varies by platform, but the security goals are consistent: resist offline cracking, replay, and credential theft.

Core concepts:
- **Salted hash**: A password hash combined with a unique random value to defeat precomputed rainbow tables
- **Password policy**: Rules for length, complexity, history, and expiration
- **Account lockout**: Temporary or permanent lock after repeated failed logons
- **MFA (Multi-Factor Authentication)**: Requires more than one factor, such as password plus token or biometric
- **Credential stuffing defense**: Detects repeated use of leaked credentials across services
- **Brute-force defense**: Rate limiting, lockout, and anomaly detection

Practical controls:
- Enforce long passphrases rather than arbitrary complexity alone
- Use MFA for privileged and remote access
- Monitor for impossible travel, unusual device posture, and repeated failures
- Disable legacy authentication where possible
- Protect password reset and recovery workflows

> Password policy alone does not stop credential stuffing. You also need MFA, monitoring, and rate limiting.

**Hardening Active Directory and LDAP**

Directory hardening reduces the risk of credential interception, privilege escalation, and unauthorized modification.

Common controls:
- **LDAP signing**: Ensures LDAP messages are signed to prevent tampering
- **LDAP channel binding**: Binds the authentication exchange to the TLS channel to reduce relay risk
- **LDAPS**: LDAP over TLS for encryption in transit
- **SMB signing**: Helps protect related Windows authentication traffic
- **NTLM restriction**: Reduces fallback to weaker authentication
- **Privileged Access Workstations (PAWs)**: Dedicated admin systems for sensitive tasks

Example GPO-based hardening ideas:
- Require LDAP signing on domain controllers
- Require LDAP channel binding where supported
- Restrict NTLM where feasible
- Enforce stronger password and lockout policies
- Audit privileged group membership changes

Example PowerShell for service account review:
```powershell
Get-ADUser -Filter 'ServicePrincipalName -like "*"' -Properties ServicePrincipalName,PasswordLastSet |
  Select-Object Name,Enabled,PasswordLastSet,ServicePrincipalName
```

Example of delegating an OU in AD:
```powershell
dsacls "OU=Finance,DC=example,DC=com" /G "EXAMPLE\Helpdesk:CC;user"
dsacls "OU=Finance,DC=example,DC=com" /G "EXAMPLE\Helpdesk:WP;telephoneNumber"
```

Example of checking LDAP over TLS connectivity:
```bash
ldapsearch -H ldaps://dc1.example.com -D "cn=admin,dc=example,dc=com" -W -b "dc=example,dc=com" "(uid=alice)"
```

**Service accounts and privileged identity management**

Service accounts are often overprivileged and poorly rotated, making them a common attack target. Treat them as high-value identities.

Good practices:
- Use **least privilege** for each service account
- Prefer **managed service accounts** or **group managed service accounts** where available
- Avoid interactive logon for service accounts
- Rotate secrets regularly or use automatic rotation
- Scope access to only the required hosts and services
- Monitor for unusual logon patterns and privilege changes

Examples of service account risks:
- Hardcoded passwords in scripts or configuration files
- Shared accounts used by multiple applications
- Excessive domain admin rights for convenience
- Forgotten accounts that remain enabled after application retirement

**Availability, redundancy, and resilience**

Directory services are critical infrastructure. If they fail, authentication, authorization, and application access can fail across the enterprise.

Resilience strategies:
- Deploy multiple domain controllers per domain and site
- Place global catalog servers where needed for logon performance
- Use site-aware replication and subnet mapping
- Back up directory data and system state regularly
- Test authoritative and non-authoritative restore procedures
- Protect time synchronization, DNS, and replication health

Operational impacts of directory outages:
- Users may not be able to log on
- Applications that depend on LDAP bind or group lookup may fail
- VPN, Wi-Fi, and device admin access may stop if they rely on the directory
- Password resets and account unlocks may be unavailable

Backup considerations:
- Back up the directory-integrated service, not just the OS
- Validate restore procedures in a lab
- Protect backups with the same rigor as production credentials
- Understand tombstone and replication behavior before restoring objects

**Trust models, federation, and when to use governance tools**

CISSP often tests whether you can choose the right identity control for the problem.

Use a directory when you need:
- Centralized identity lookup
- Group-based authorization
- Authentication for domain-joined systems
- Attribute-based queries and policy scoping

Use federation when you need:
- Cross-organization SSO
- Reduced password duplication
- Token-based access to cloud or partner services

Use identity governance when you need:
- Access certification
- Role lifecycle management
- Joiner-mover-leaver workflows
- Segregation of duties reviews

Trust model distinctions:
- **Directory trust** enables authentication across domains or forests
- **Federation trust** enables token acceptance between organizations or platforms
- **Governance trust** is procedural and policy-based, not a runtime authentication mechanism

**Practical examples and troubleshooting cues**

Common LDAP query example:
```bash
ldapsearch -x -H ldap://ldap.example.com -b "dc=example,dc=com" "(memberOf=cn=Finance,ou=Groups,dc=example,dc=com)"
```

Common AD query example:
```powershell
Get-ADGroupMember "Finance" | Select-Object Name,SamAccountName,ObjectClass
```

Check Kerberos tickets on a Windows client:
```cmd
klist
```

Purge tickets during troubleshooting:
```cmd
klist purge
```

Useful troubleshooting signals:
- Kerberos failures often point to DNS, time skew, SPN mismatch, or delegation issues
- NTLM fallback can indicate Kerberos misconfiguration or unsupported service paths
- Replication issues can cause inconsistent group membership or stale attributes
- GC problems can affect universal group resolution and forest searches
- LDAP bind failures may indicate signing, channel binding, certificate, or firewall issues

**Exam focus summary**

For CISSP, remember these distinctions:
- A **directory** stores identities and attributes; an **identity governance** tool manages lifecycle and access reviews
- **Kerberos** is ticket-based and preferred in modern Windows domains; **NTLM** is legacy and weaker
- **Global Catalog** supports forest-wide searches and universal group membership
- **Referrals** direct clients to another server; **aliases** are alternate directory objects
- **Constrained delegation** is safer than unconstrained delegation
- **RADIUS** and **TACACS+** are related identity/authentication stores for network and device access
- **Cloud identity** introduces sync, precedence, and deprovisioning challenges that do not exist in purely on-prem directories
- **Availability** matters because directory outages can cascade into enterprise-wide access failures
