## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Identity, federation, and SSO overview**

Identity and access management (IAM) separates *authentication* from *authorization* and extends trust across systems. In practice, you use federation and SSO to let a user authenticate once with an identity provider (IdP) and then access multiple service providers (SPs) or relying parties without repeated logins.

- **Authentication** proves who the user is.
- **Authorization** decides what the user can do.
- **Federation** extends identity trust across organizational or domain boundaries.
- **SSO** reduces repeated prompts by reusing a trusted session or token.

A common enterprise pattern is:
- A workforce user authenticates to Microsoft Entra ID, Okta, Ping, ADFS, or another IdP.
- The IdP issues a signed assertion or token.
- The application trusts that token and creates a local session.

This model improves usability and centralizes policy, but it also concentrates risk. If the IdP is compromised, many downstream applications can be affected.

**Core identity concepts**

Identity systems rely on a few recurring building blocks:

- **Identity proofing** establishes that a real-world subject is who they claim to be before account creation.
- **Authentication assurance** reflects how strongly the identity was verified at login time.
- **Claims** are statements about a subject, such as email, group membership, role, or department.
- **Attributes** are the data elements used to build claims and authorization decisions.
- **Trust decisions** depend on both the token content and the assurance level behind it.

<dl>
<dt><strong>Identity proofing</strong></dt>
<dd>Verification performed before issuing an identity, often using documents, HR records, or in-person validation.</dd>
<dt><strong>Assurance</strong></dt>
<dd>The confidence level that the authenticated subject is the claimed identity.</dd>
<dt><strong>Claims</strong></dt>
<dd>Portable identity statements consumed by applications for access control or personalization.</dd>
</dl>

For CISSP-style questions, remember that a signed token is not automatically sufficient for every resource. The application must also decide whether the assurance level, audience, issuer, and lifetime are acceptable.

**Federation trust establishment and lifecycle**

Federation is not just a protocol exchange; it is a managed trust relationship. The lifecycle usually includes onboarding, metadata exchange, certificate management, operational monitoring, and offboarding.

- **Onboarding** starts with partner identification, business approval, and technical alignment on protocol, endpoints, and attributes.
- **Metadata exchange** publishes the IdP or SP configuration so each side can validate endpoints, certificates, entity IDs, and supported bindings.
- **Certificate rollover** updates signing or encryption keys before expiration to avoid outages.
- **Trust revocation** removes or disables trust when a partner is compromised, contractually terminated, or no longer authorized.
- **Offboarding** includes disabling metadata, removing certificates, revoking access, and deprovisioning accounts or entitlements.

A typical trust setup process looks like this:

1. Exchange entity metadata or configuration details.
2. Validate the issuer, audience, and endpoint URLs.
3. Import signing certificates or public keys.
4. Map claims to local roles or groups.
5. Test login, logout, and error handling.
6. Schedule certificate rotation and review trust periodically.

Example SAML metadata concepts:
```xml
<EntityDescriptor entityID="https://idp.example.com">
  <IDPSSODescriptor>
    <KeyDescriptor use="signing">
      <KeyInfo>
        <X509Data>
          <X509Certificate>MIIC...</X509Certificate>
        </X509Data>
      </KeyInfo>
    </KeyDescriptor>
    <SingleSignOnService
      Binding="urn:oasis:names:tc:SAML:2.0:bindings:HTTP-Redirect"
      Location="https://idp.example.com/sso"/>
  </IDPSSODescriptor>
</EntityDescriptor>
```

> Trust should be treated as a lifecycle, not a one-time setup. Expired certificates, stale metadata, and orphaned partner accounts are common operational failures.

**SSO patterns and protocol flow**

SSO can be implemented with cookies, Kerberos, SAML, OAuth 2.0, OpenID Connect (OIDC), or proprietary brokers. The key idea is that the user authenticates once and then reuses a trusted artifact.

- **IdP-initiated flow** starts at the identity provider, which sends the user to the application with a signed assertion or token.
- **SP-initiated flow** starts at the application, which redirects the user to the IdP for authentication.
- **Front-channel communication** uses the user agent, usually the browser, to carry messages between parties.
- **Back-channel communication** uses direct server-to-server calls, often for token exchange, introspection, or logout coordination.

Front-channel flows are convenient but exposed to browser-based threats such as replay, interception, and CSRF (Cross-Site Request Forgery). Back-channel flows are generally more controlled and are often preferred for sensitive exchanges.

**SAML essentials**

Security Assertion Markup Language (SAML) is an XML-based federation standard commonly used for enterprise SSO. It is especially common in workforce environments.

- **IdP** authenticates the user and issues a SAML assertion.
- **SP** consumes the assertion and creates a local session.
- **Assertion** contains authentication statements, attributes, and conditions.
- **Audience restriction** limits which SP can accept the assertion.
- **Conditions** can include time validity and not-before/not-after windows.

SAML bindings and profiles matter on exams:
- **HTTP-Redirect** and **HTTP-POST** are common browser bindings.
- **Artifact binding** can move the assertion exchange partly to a back-channel.
- **IdP-initiated SSO** is simpler operationally but can be harder to secure and audit than SP-initiated flows.
- **SP-initiated SSO** gives the application more control over relay state and request correlation.

Token binding concepts to know:
- The assertion is signed by the IdP.
- The SP validates the signature, issuer, audience, and time conditions.
- The assertion may be encrypted if it contains sensitive attributes.
- The browser is only a transport path; it should not be trusted as the security boundary.

Example SAML service provider settings:
```xml
<SPSSODescriptor AuthnRequestsSigned="true" WantAssertionsSigned="true">
  <AssertionConsumerService
    Binding="urn:oasis:names:tc:SAML:2.0:bindings:HTTP-POST"
    Location="https://sp.example.com/saml/acs"
    index="1"/>
</SPSSODescriptor>
```

Practical SAML deployment notes:
- Use signed assertions and, where appropriate, signed authentication requests.
- Validate `AudienceRestriction`, `Recipient`, `InResponseTo`, and timestamps.
- Keep clock skew small and monitor NTP (Network Time Protocol) drift.
- Map only the attributes you need; avoid over-sharing PII (personally identifiable information).

**OAuth 2.0 authorization framework**

OAuth 2.0 is an authorization framework, not an authentication protocol. It delegates access to APIs and resources using access tokens.

- **Resource owner** is usually the user.
- **Client** is the application requesting access.
- **Authorization server** authenticates the user and issues tokens.
- **Resource server** hosts the protected API or data.

The most important practical distinction:
- **Access token** authorizes API access and is presented to the resource server.
- **Refresh token** is used to obtain new access tokens without re-prompting the user.
- **ID token** is an OIDC artifact that proves authentication to the client; it is not an OAuth access token.

OAuth security threats and mitigations are exam-relevant:
- **Authorization code interception** can occur if an attacker steals the code before the client exchanges it.
  - Use **PKCE** (Proof Key for Code Exchange) to bind the authorization code to the original client.
- **CSRF** can trick a browser into completing a login or authorization flow unexpectedly.
  - Use a strong `state` value and validate it on return.
- **Token leakage** can happen through logs, browser history, referrers, or insecure storage.
  - Prefer short-lived access tokens and secure storage.
- **Redirect URI manipulation** can send tokens or codes to an attacker-controlled endpoint.
  - Register exact redirect URIs and validate them strictly.

Why PKCE matters:
- It was designed to protect public clients such as mobile apps and single-page applications that cannot safely store a client secret.
- The client creates a random verifier and sends a derived challenge during authorization.
- When redeeming the code, the client proves possession of the original verifier.
- This reduces the value of a stolen authorization code.

Example OAuth 2.0 authorization request:
```text
GET /authorize?
  response_type=code&
  client_id=app123&
  redirect_uri=https%3A%2F%2Fapp.example.com%2Fcallback&
  scope=openid%20profile%20email&
  state=af0ifjsldkj&
  code_challenge=E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbuGJSstw-cM&
  code_challenge_method=S256
```

Example token exchange:
```bash
curl -X POST https://auth.example.com/token \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  -d 'grant_type=authorization_code' \
  -d 'code=SplxlOBeZQQYbYS6WxSbIA' \
  -d 'redirect_uri=https://app.example.com/callback' \
  -d 'client_id=app123' \
  -d 'code_verifier=dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk'
```

**OpenID Connect essentials**

OpenID Connect (OIDC) adds authentication on top of OAuth 2.0. It is widely used for modern web, mobile, and API ecosystems.

- **ID token** is a JWT (JSON Web Token) that contains identity claims.
- **UserInfo endpoint** can provide additional claims after authentication.
- **Discovery** lets clients find endpoints and supported features dynamically.
- **JWKS** (JSON Web Key Set) publishes public keys used to verify token signatures.
- **JWK** (JSON Web Key) is a single key representation.

OIDC discovery example:
```text
GET /.well-known/openid-configuration
```

Typical discovery response fields include:
- `issuer`
- `authorization_endpoint`
- `token_endpoint`
- `userinfo_endpoint`
- `jwks_uri`
- `response_types_supported`

JWKS example:
```json
{
  "keys": [
    {
      "kty": "RSA",
      "kid": "2026-01",
      "use": "sig",
      "alg": "RS256",
      "n": "....",
      "e": "AQAB"
    }
  ]
}
```

Validation vs introspection:
- **JWT validation** is local verification of signature, issuer, audience, expiration, and nonce where applicable.
- **Token introspection** is a back-channel call to the authorization server to ask whether a token is active and what it contains.
- Use JWT validation when the token is self-contained and the key material is trusted.
- Use introspection when you need near-real-time revocation or opaque tokens.

OIDC flow awareness:
- **Authorization code flow** is the preferred browser-based flow.
- **Implicit flow** is generally deprecated for new applications because tokens are exposed in the browser more directly.
- **Hybrid flow** combines code and token responses and may appear in legacy or specialized deployments.

Practical OIDC validation checklist:
- Verify `iss`, `aud`, `exp`, `iat`, and `nonce`.
- Fetch keys from `jwks_uri` and match by `kid`.
- Reject tokens signed with unexpected algorithms.
- Enforce exact redirect URI matching.
- Use `state` for CSRF protection and request correlation.

**Kerberos authentication**

Kerberos is a ticket-based network authentication protocol commonly used in Windows domains and mixed enterprise environments. It relies on a trusted third party called the Key Distribution Center (KDC).

Core components:
- **Authentication Server (AS)** issues the Ticket Granting Ticket (TGT).
- **Ticket Granting Server (TGS)** issues service tickets.
- **KDC** often refers to AS + TGS together.
- **Realm** is the Kerberos administrative domain.

Typical ticket exchange sequence:
1. The client authenticates to the AS and requests a TGT.
2. The AS returns a TGT and a session key.
3. The client presents the TGT to the TGS for a specific service ticket.
4. The TGS returns a service ticket and session key.
5. The client presents the service ticket to the application server.
6. The server validates the ticket and grants access.

Important Kerberos concepts:
- **Mutual authentication** helps both client and server verify each other.
- **Time synchronization** is critical because tickets are time-bound.
- **Delegation** can allow a service to act on behalf of a user in limited cases.
- **Cross-realm trust** allows authentication across Kerberos realms.

Attack considerations:
- **Pass-the-ticket** uses a stolen Kerberos ticket to impersonate a user.
- **Golden ticket** abuse involves forging tickets when the attacker has the krbtgt key in Active Directory.
- **Kerberoasting** targets service account tickets to crack weak service account passwords offline.

Operational controls:
- Protect domain controller and KDC credentials.
- Enforce strong service account passwords or managed service accounts.
- Monitor unusual ticket lifetimes, service ticket requests, and delegation use.
- Keep clocks synchronized with reliable NTP.

**LDAP and directory services**

Lightweight Directory Access Protocol (LDAP) is used to query and modify directory information. It is not the same as authentication, though it is often used to support authentication and authorization workflows.

Directory basics:
- The directory is hierarchical, often organized as a tree.
- Entries are identified by a Distinguished Name (DN).
- Attributes describe each entry, such as `cn`, `uid`, `mail`, or `memberOf`.
- Object classes define what attributes an entry can have.

Example directory hierarchy:
```text
dc=example,dc=com
├── ou=People
│   ├── uid=alice
│   └── uid=bob
└── ou=Groups
    └── cn=admins
```

LDAP transport and security:
- **LDAP** commonly uses port 389.
- **LDAPS** uses TLS (Transport Layer Security) on port 636.
- **StartTLS** upgrades a plain LDAP connection to TLS.
- Prefer TLS-protected LDAP for credentials and sensitive directory queries.

Practical distinction:
- **Authentication** verifies credentials, often by binding to LDAP.
- **Authorization** uses directory attributes or group membership to decide access.
- **Directory services** store identity data and relationships, but they do not by themselves enforce application access.

Example LDAP query:
```bash
ldapsearch -H ldaps://ldap.example.com:636 \
  -D "uid=alice,ou=People,dc=example,dc=com" \
  -W \
  -b "dc=example,dc=com" "(memberOf=cn=admins,ou=Groups,dc=example,dc=com)"
```

**MFA and step-up authentication**

Multi-factor authentication (MFA) strengthens federation by requiring more than one factor, such as something you know, have, or are. In federated environments, MFA is often enforced at the IdP rather than separately in each application.

- **Step-up authentication** increases assurance when a user requests a sensitive action.
- You may require step-up for payroll changes, admin consoles, wire transfers, or privileged API calls.
- The application can request a higher assurance level or a fresh authentication event.

Common patterns:
- A user signs in with password plus authenticator app.
- A privileged action triggers a second factor or reauthentication.
- The IdP issues a token with an assurance claim that reflects the stronger authentication.

Example policy logic:
- Low-risk app access: password plus remembered device may be acceptable.
- High-risk transaction: require phishing-resistant MFA such as FIDO2/WebAuthn.
- Admin access: require recent authentication and device compliance.

**Claims, attributes, and authorization mapping**

Claims are the bridge between identity and access decisions. The IdP asserts facts, and the application or policy engine interprets them.

- Map only trusted claims to local roles.
- Prefer group or role claims over hard-coded user lists.
- Normalize attribute names across partners to reduce ambiguity.
- Treat external claims as input, not as absolute truth.

Example claim mapping:
```json
{
  "sub": "00u12345",
  "email": "alice@example.com",
  "groups": ["finance", "vpn-users"],
  "amr": ["pwd", "mfa"],
  "acr": "urn:mace:incommon:iap:silver"
}
```

Possible local mapping:
- `finance` group maps to read-only finance application access.
- `vpn-users` maps to remote access entitlement.
- `acr` value maps to a minimum assurance requirement.

**SCIM provisioning and lifecycle automation**

System for Cross-domain Identity Management (SCIM) automates user and group provisioning between identity systems and applications. It complements SSO by handling account lifecycle, not just login.

- **Provisioning** creates accounts and assigns initial attributes.
- **Deprovisioning** disables or removes access when a user leaves.
- **Just-in-time (JIT) provisioning** creates the account on first login.
- **Authoritative source** is usually HR or the primary directory.

Example SCIM workflow:
1. HR marks an employee as active.
2. The identity platform creates the user in the SaaS application.
3. Group membership is synchronized.
4. When the employee leaves, the identity platform disables the account and removes entitlements.

Example SCIM request:
```bash
curl -X POST https://api.example.com/scim/v2/Users \
  -H 'Authorization: Bearer <token>' \
  -H 'Content-Type: application/scim+json' \
  -d '{
    "userName": "alice@example.com",
    "name": { "givenName": "Alice", "familyName": "Ng" },
    "active": true
  }'
```

**Enterprise deployment examples**

Real deployments often combine multiple standards:
- An enterprise IdP such as Microsoft Entra ID, Okta, Ping Identity, ADFS, or Keycloak authenticates users.
- A federation broker or identity hub translates between protocols, such as SAML to OIDC.
- SaaS applications consume SAML assertions or OIDC ID tokens.
- APIs use OAuth 2.0 access tokens.
- SCIM handles account lifecycle synchronization.

Example SAML-to-local-role mapping:
```yaml
issuer: https://idp.example.com
audience: https://app.example.com
attributeMapping:
  email: mail
  department: dept
  groups: memberOf
roleMapping:
  finance: finance_read
  admins: app_admin
```

Example OIDC client configuration:
```yaml
client_id: app123
issuer: https://login.example.com
redirect_uris:
  - https://app.example.com/callback
scopes:
  - openid
  - profile
  - email
  - groups
response_type: code
pkce: required
```

**Trust, revocation, and operational security**

Federation trust must be monitored continuously. Certificates expire, keys rotate, and partners change.

- **Certificate rollover** should be tested before production cutover.
- **Revocation** may be implemented by disabling metadata, rotating keys, or removing trust configuration.
- **Session revocation** is harder than token issuance; short lifetimes reduce exposure.
- **Logout** in federated systems is often partial because browser sessions, IdP sessions, and app sessions may not end at the same time.

Operational controls:
- Use short token lifetimes for high-risk applications.
- Rotate signing keys on a schedule and publish new JWKS entries before retiring old keys.
- Review partner access periodically.
- Remove stale claims and unused trust relationships.
- Log authentication events, token issuance, and administrative changes.

> Do not assume logout means global revocation. In many systems, the IdP session, application session, and refresh token remain independent.

**Common exam comparisons**

| Topic | Key point | Typical use |
|---|---|---|
| SAML | XML-based federation with signed assertions | Enterprise browser SSO |
| OAuth 2.0 | Authorization framework | API delegation |
| OIDC | Authentication layer on OAuth 2.0 | Modern login and identity claims |
| Kerberos | Ticket-based mutual authentication | Windows domain SSO |
| LDAP | Directory access protocol | Identity lookup and group membership |

**Practical troubleshooting checklist**

When federation fails, check the basics first:
- Verify system time and NTP synchronization.
- Confirm issuer, audience, and redirect URI values.
- Check certificate validity and key IDs.
- Inspect browser redirects and relay state.
- Validate claim mappings and required attributes.
- Review logs for signature failures, expired tokens, and clock skew.

Useful commands:
```bash
# Inspect OIDC discovery
curl https://login.example.com/.well-known/openid-configuration

# Fetch JWKS
curl https://login.example.com/.well-known/jwks.json

# Decode a JWT for inspection
python3 - <<'PY'
import base64, json
token = "header.payload.signature"
header, payload, _ = token.split(".")
for part in (header, payload):
    pad = "=" * (-len(part) % 4)
    print(json.dumps(json.loads(base64.urlsafe_b64decode(part + pad)), indent=2))
PY
```

**High-value CISSP takeaways**

- Federation extends trust; it does not eliminate risk.
- SSO improves usability but increases the blast radius of IdP compromise.
- SAML is common for enterprise browser SSO; OIDC is common for modern apps and APIs.
- OAuth 2.0 is for authorization, not authentication.
- Kerberos depends on tickets, time, and trusted realms.
- LDAP supports directory lookups and group-based decisions, especially over TLS.
- MFA and step-up authentication raise assurance for sensitive actions.
- Lifecycle management matters as much as login: onboarding, rotation, revocation, and offboarding are all part of secure identity operations.
