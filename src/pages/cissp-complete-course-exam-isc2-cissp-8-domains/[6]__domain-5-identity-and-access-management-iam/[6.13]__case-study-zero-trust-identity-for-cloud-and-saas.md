## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Zero Trust identity in cloud and SaaS**

Zero Trust assumes no implicit trust based on network location, device ownership, or tenant boundary. In cloud and SaaS environments, identity becomes a major decision point, but it is not the only one. Effective designs combine **identity**, device posture, application context, network signals, and policy enforcement to decide whether access is allowed, limited, or denied.

For CISSP, focus on the governance and control objectives:

- Verify every access request.
- Grant the minimum access needed.
- Continuously reassess trust.
- Log and review decisions for accountability.

A practical Zero Trust identity program covers the full identity lifecycle, federation, privileged access, workload identity, and monitoring across cloud tenants and SaaS applications.

**Identity lifecycle management**

Identity lifecycle management is the foundation of access control in cloud and SaaS. You need reliable processes for **joiner, mover, leaver** events so accounts are created, changed, and removed in sync with employment status and role changes.

- **Joiner**: create the account, assign baseline access, and enforce MFA or passwordless enrollment.
- **Mover**: remove old entitlements before adding new ones to avoid privilege accumulation.
- **Leaver**: disable accounts quickly, revoke sessions, rotate shared secrets, and remove app assignments.

Automated provisioning reduces delay and human error. In SaaS, **SCIM** (System for Cross-domain Identity Management) is commonly used to synchronize users, groups, and sometimes roles from the identity provider to the application.

```http
POST /scim/v2/Users HTTP/1.1
Host: app.example.com
Authorization: Bearer <provisioning-token>
Content-Type: application/scim+json

{
  "userName": "jane.doe@example.com",
  "name": {
    "givenName": "Jane",
    "familyName": "Doe"
  },
  "active": true,
  "emails": [
    {
      "value": "jane.doe@example.com",
      "primary": true
    }
  ]
}
```

- Use **deprovisioning** controls that disable access in the IdP and downstream SaaS.
- Reconcile orphaned accounts created outside the normal workflow.
- Review group membership and app assignments after role changes.
- Prefer automated workflows over manual ticket-based creation for scale and auditability.

> If a user leaves but SaaS sessions remain active, the account may still be usable until tokens expire or are revoked. Immediate session revocation matters.

**Authentication methods and passwordless access**

Zero Trust does not require passwords, but it does require strong authentication. Traditional passwords remain vulnerable to phishing, reuse, and credential stuffing. Modern deployments increasingly use **passwordless authentication** with **FIDO2** (Fast Identity Online 2) and **passkeys**.

Passkeys are cryptographic credentials stored in a platform authenticator or security key. They are generally phishing-resistant because they bind authentication to the legitimate origin, but they are not always equivalent to MFA by themselves. Depending on the authenticator and policy, a passkey may provide single-factor cryptographic authentication or multi-factor authentication when combined with device unlock or another factor.

- **FIDO2/WebAuthn**: standards for phishing-resistant authentication.
- **Passkeys**: user-friendly FIDO credentials that can sync across devices in some ecosystems.
- **MFA**: multi-factor authentication; still useful, but not all MFA methods are equally resistant to phishing.

For exam purposes, distinguish these points:

- SMS codes are weak against interception and SIM swap.
- Push approvals can be vulnerable to fatigue attacks unless number matching or context is used.
- TOTP (time-based one-time password) is better than passwords alone, but still phishable.
- FIDO2/passkeys are preferred for phishing resistance.

```text
Preferred authentication order for high-risk SaaS:
1. FIDO2 security key or passkey
2. Device-bound certificate or managed device trust
3. TOTP as fallback only
4. SMS only as last resort or not at all
```

**Federation and trust boundaries**

Cloud and SaaS commonly rely on federation so the application trusts an external identity provider (IdP) instead of storing separate credentials. The main protocols are **SAML** (Security Assertion Markup Language) and **OIDC** (OpenID Connect), which is built on OAuth 2.0.

In a standard OIDC authorization code flow:

1. The user authenticates at the IdP.
2. The client receives an **authorization code**.
3. The client sends the code to the token endpoint.
4. The IdP returns tokens, typically an **ID token** and an **access token**.

This sequence matters because the code is not the token itself. The code is exchanged for tokens, which reduces exposure in the browser.

```text
Client -> IdP authorization endpoint
Client receives authorization code
Client -> IdP token endpoint
IdP returns ID token + access token
```

Federation trust introduces specific risks:

- **Assertion signing** must be enforced so the service can verify the token or SAML assertion has not been altered.
- **Issuer validation** must confirm the token came from the expected IdP.
- **Audience validation** must ensure the token is meant for the target application.
- **Certificate rotation** is required so signing keys do not become stale or compromised.
- **IdP compromise** can affect many downstream SaaS apps at once because the IdP becomes a high-value trust anchor.

Practical controls include:

- Short token lifetimes.
- Refresh token protection and revocation.
- Conditional access based on device, location, and risk.
- Monitoring for unusual federation events and admin consent grants.

**Authorization and least privilege**

Authentication answers “who are you?” Authorization answers “what can you do?” In Zero Trust, authorization should be enforced at the **resource** and **API** level, not only at the network perimeter.

Use these controls:

- **Least privilege**: grant only the permissions needed for the task.
- **Role-based access control** (RBAC): assign permissions through roles.
- **Attribute-based access control** (ABAC): use user, device, location, and resource attributes.
- **Entitlement management**: govern who can request, approve, and receive access.
- **Access reviews**: periodically recertify access to remove stale entitlements.

A common SaaS mistake is overbroad group assignment. For example, assigning a user to a global admin or tenant-wide editor role when they only need access to one application workspace creates unnecessary risk.

```yaml
# Illustrative conditional access policy pattern
policy:
  name: "Require phishing-resistant auth for admin apps"
  users:
    include: ["AllUsers"]
    exclude: ["BreakGlassAccounts"]
  apps:
    include: ["AdminPortal", "FinanceSaaS"]
  conditions:
    device:
      compliant: true
    risk:
      signInRisk: "medium_or_higher"
  grant:
    requireAuthenticationStrength: "phishing-resistant"
    requireMFA: true
```

- Use **access reviews** for privileged groups, guest users, and dormant accounts.
- Remove standing privilege where possible and use just-in-time elevation.
- Separate administrative roles from daily user accounts.
- Log approval decisions for audit and accountability.

**Cloud tenant isolation and SaaS sharing risks**

Cloud and SaaS environments often span multiple tenants, subscriptions, or workspaces. Tenant isolation reduces the chance that one organization can access another organization’s data or administrative plane.

Key risks include:

- Misconfigured external sharing in SaaS collaboration tools.
- Guest users retaining access after project completion.
- Cross-tenant trust settings that are broader than intended.
- Shared admin roles that span multiple tenants or business units.

Controls to apply:

- Restrict external sharing to approved domains or groups.
- Review guest access and external collaboration settings regularly.
- Use separate admin roles for each tenant or environment.
- Require stronger authentication for cross-tenant access.
- Monitor invitations, consent grants, and sharing link creation.

> A common exam trap is assuming “cloud tenant” automatically means isolation is complete. Misconfiguration can still expose data across tenants or to external guests.

**Workload identity and non-human access**

Zero Trust identity also applies to workloads, not just people. Applications, scripts, containers, and automation pipelines need identities that can authenticate without shared passwords.

Common workload identity patterns:

- **Service principals**: application identities with assigned permissions.
- **Managed identities**: cloud-managed identities attached to compute resources, reducing secret handling.
- **Workload federation**: external workloads authenticate using federated identity instead of long-lived secrets.

Practical distinctions:

- Use **service principals** when an application needs a stable identity and the platform supports app registrations.
- Use **managed identities** when the cloud provider can issue and rotate credentials automatically.
- Use **workload federation** for CI/CD pipelines or multi-cloud workloads that should avoid stored secrets.

Example conceptual comparison:

| Pattern | Secret handling | Typical use | Risk |
|---|---:|---|---|
| Service principal | Often requires secret or certificate | App-to-API access | Secret sprawl if unmanaged |
| Managed identity | No app-managed secret | Cloud VM, function, container | Limited to supported platforms |
| Workload federation | No long-lived secret | CI/CD, external IdP trust | Trust policy must be tightly scoped |

Practical example for a cloud-native app:

```bash
# Illustrative cloud CLI example; exact syntax varies by provider
# Create an app identity
cloud iam app create --name billing-api

# Assign least-privilege access to one resource group or API scope
cloud iam role assign \
  --principal billing-api \
  --role "Reader" \
  --scope "/subscriptions/123/resourceGroups/billing"

# Prefer managed identity where supported
cloud compute identity assign --resource vm-01 --system-assigned
```

**Break-glass and emergency access**

Break-glass accounts are emergency access accounts used when normal identity systems fail or are unavailable. They are essential in Zero Trust because strict controls can otherwise lock administrators out during an outage or IdP compromise.

Good break-glass design includes:

- Strong, unique credentials stored securely offline or in a hardened vault.
- Exclusion from normal conditional access policies only when necessary.
- Very limited membership and tightly controlled use.
- Alerting on every sign-in and every privilege use.
- Periodic testing to confirm the account still works.

Use break-glass accounts for emergencies, not convenience. They should be monitored more aggressively than standard admin accounts.

**Monitoring, logging, and auditability**

Zero Trust depends on continuous verification. Identity events must be logged, correlated, and reviewed so you can detect abuse, misconfiguration, and policy drift.

Monitor for:

- Failed logins and password spray attempts.
- Impossible travel or unusual geolocation.
- New device enrollment or new authenticator registration.
- Privilege escalation and admin consent grants.
- SCIM provisioning failures and orphaned accounts.
- Guest invitations, external sharing, and cross-tenant access changes.
- Token anomalies such as unusual audience, issuer, or lifetime patterns.

Useful log sources include:

- IdP sign-in logs.
- SaaS audit logs.
- Cloud control plane logs.
- Privileged access management logs.
- Endpoint posture and device compliance logs.

```bash
# Example log review workflow using a generic SIEM query style
search source=IdPLogs action=signin
| where risk_level >= "medium"
| stats count by user, app, ip_address
```

- Correlate identity events with device and network telemetry.
- Retain logs long enough to support investigations and compliance.
- Protect logs from tampering and unauthorized deletion.
- Review privileged actions separately from routine user activity.

**Conditional access and policy enforcement**

Conditional access is a practical Zero Trust control that evaluates context before granting access. Policies can require stronger authentication, block risky sessions, or limit access to compliant devices.

Common policy inputs:

- User role or group membership.
- Device compliance or managed status.
- Location, IP reputation, or country.
- Sign-in risk or user risk.
- Application sensitivity.
- Session type and data sensitivity.

Example policy logic:

```text
If user is admin
and app is sensitive
and device is not compliant
then block access or require phishing-resistant MFA
```

Use session controls when you need to allow access but reduce exposure:

- Restrict download on unmanaged devices.
- Require reauthentication for sensitive actions.
- Limit clipboard, print, or upload actions in browser sessions.
- Force step-up authentication for privileged operations.

**CISSP governance and exam mapping**

For CISSP, connect technical controls to governance outcomes:

- **Accountability**: every identity action should be attributable to a person or workload.
- **Auditability**: logs and reviews prove who accessed what and when.
- **Least privilege**: permissions are minimized and reviewed.
- **Separation of duties**: no single identity should control all critical steps.
- **Risk management**: controls should match the sensitivity of the app, data, and tenant.

Typical exam themes include:

- Choosing federation over local accounts for SaaS scale.
- Using SCIM for lifecycle automation.
- Preferring phishing-resistant authentication for privileged access.
- Reviewing guest access and cross-tenant sharing.
- Securing break-glass accounts without making them routine admin accounts.
- Validating SAML/OIDC trust and limiting blast radius if the IdP is compromised.

**Quick comparison of common identity controls**

| Control | Primary purpose | Best use case |
|---|---|---|
| SAML | Federated SSO | Enterprise SaaS login |
| OIDC | Modern federated authentication | Web and mobile apps |
| SCIM | Provisioning and deprovisioning | SaaS lifecycle sync |
| FIDO2/passkeys | Phishing-resistant auth | Privileged and high-risk access |
| Conditional access | Context-aware enforcement | Zero Trust policy decisions |
| Access reviews | Entitlement governance | Privileged and guest access |

**Practical implementation checklist**

1. Inventory all cloud and SaaS identities, including guests and service accounts.
2. Centralize authentication in a trusted IdP where feasible.
3. Enforce phishing-resistant MFA or passkeys for privileged users.
4. Automate joiner, mover, and leaver workflows with SCIM or equivalent APIs.
5. Restrict tenant sharing, guest access, and admin consent.
6. Validate federation settings, signing keys, issuer, and audience.
7. Separate break-glass accounts and monitor them continuously.
8. Review entitlements regularly and remove stale access.
9. Log identity, admin, and provisioning events to a SIEM.
10. Test incident response for IdP outage and IdP compromise scenarios.

**Key takeaways**

Zero Trust identity for cloud and SaaS is not just about logging in securely. It is about governing the full identity lifecycle, enforcing least privilege at the resource level, validating federation trust, protecting non-human identities, and continuously monitoring for misuse. For CISSP, the strongest answers usually emphasize governance, risk reduction, accountability, and auditability rather than a single technology.
