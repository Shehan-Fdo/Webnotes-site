## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Privileged Access Management (PAM) fundamentals**

Privileged Access Management is the set of controls you use to govern accounts that can make high-impact changes to systems, applications, and data. These accounts are attractive targets because they can disable security tools, exfiltrate data, create backdoors, or alter logs. In CISSP terms, PAM is about reducing the likelihood, duration, and blast radius of privileged misuse.

PAM is not just about administrator usernames. It includes human admins, service accounts, application identities, emergency access, and cloud control-plane roles. The goal is to ensure that privileged access is granted only when needed, for the minimum time required, with strong accountability.

- **Least privilege** limits each account to the minimum permissions required for the task.
- **Just-in-time (JIT)** access grants elevated rights only for a short approved window.
- **Just-enough administration (JEA)** restricts what an admin can do, even when elevated.
- **Separation of duties** prevents one person from controlling an entire sensitive process.
- **Accountability** requires logging, attribution, and review of privileged actions.

A practical PAM architecture often includes a privileged access gateway, a vault, session recording, approval workflows, and rotation of shared credentials. In mature environments, administrators do not know the actual root or domain admin password; they request access through the PAM platform and receive a time-bound session or a checked-out credential.

**Core PAM controls**

PAM works best when you combine technical controls with process controls. The technical side reduces exposure; the process side ensures privileged use is authorized and auditable.

- **Unique admin accounts** separate administrative activity from daily user activity.
- **MFA (multifactor authentication)** should protect privileged logons.
- **Session recording** captures commands, screen activity, and command output for review.
- **Approval workflows** add human validation for sensitive actions.
- **Credential checkout** can require ticket numbers, justification, and expiration.
- **Break-glass accounts** provide emergency access when normal controls fail.

> Critical warning: shared administrator passwords without vaulting or rotation create a high-risk accountability gap. If multiple people know the same secret, attribution becomes weak and compromise impact increases.

**Privileged account types and use cases**

Different privileged identities require different controls. Treat them according to their purpose and risk.

| Account type | Typical use | Key control focus |
|---|---|---|
| Human admin account | Interactive administration | MFA, JIT, session logging |
| Service account | Runs a service or daemon | Non-interactive use, rotation, scoped permissions |
| Application account | Application-to-database or API access | Secret storage, rotation, least privilege |
| Emergency account | Recovery during outages | Strong protection, sealed storage, monitored use |
| Root / domain admin | Full system or directory control | Vaulting, approval, recording, limited use |

Service accounts are often overprivileged because they are created for convenience and then forgotten. They should not be used for interactive logons unless there is a documented exception. Application accounts should be constrained to the specific resource and action set they need, such as read-only database access or a single API scope.

**Secrets management overview**

Secrets management is the discipline of storing, distributing, using, rotating, and retiring sensitive values such as passwords, API keys, tokens, certificates, SSH keys, and encryption keys. A secret is any value that must remain confidential to preserve security.

Secrets management differs from PAM in emphasis:

- PAM focuses on controlling privileged identities and sessions.
- Secrets management focuses on protecting the sensitive values those identities and applications use.

In practice, the two overlap heavily. A PAM platform may store privileged passwords, while a secrets manager may provide dynamic database credentials to applications. Both aim to remove hard-coded secrets, reduce exposure, and support rotation.

**Common secret types**

- **Passwords** for admin, service, and application accounts.
- **API keys** used by applications or automation tools.
- **Tokens** such as OAuth access tokens, refresh tokens, or session tokens.
- **Certificates and private keys** used for TLS, mutual TLS, or code signing.
- **SSH keys** for remote administration and automation.
- **Encryption keys** used for data protection and key wrapping.

Secrets should never be embedded in source code, scripts, container images, or configuration files in plaintext. If a secret must be present at runtime, prefer injection from a secure store or environment-specific secret mechanism.

**Secret lifecycle management**

A strong secret lifecycle reduces the chance that a leaked credential remains useful.

1. **Generate** the secret using a secure random source or approved key generation process.
2. **Store** it in a vault, HSM, or other protected repository.
3. **Distribute** it only to authorized workloads or users.
4. **Use** it through controlled retrieval or injection.
5. **Rotate** it on schedule or after suspected compromise.
6. **Revoke** it when no longer needed.
7. **Audit** access and changes throughout the lifecycle.

Rotation matters because secrets eventually leak through logs, memory dumps, backups, screenshots, or developer mistakes. Short-lived credentials reduce the value of theft and simplify incident response.

**Vaulting and secure storage**

A vault is a protected system designed to store secrets securely and control access to them. Vaults typically provide encryption at rest, access policies, audit logs, versioning, and rotation workflows. Some vaults also support dynamic secret generation, where the system creates a credential on demand and expires it automatically.

Common storage patterns include:

- **Centralized vaults** for enterprise-wide secret governance.
- **Cloud secret managers** integrated with IAM and workload identity.
- **Hardware Security Modules (HSMs)** for protecting cryptographic keys.
- **Key management services (KMS)** for envelope encryption and key lifecycle control.

A good vault design separates the secret value from the application code and from the operator who deploys the application. Access should be policy-driven and logged. If possible, the application should authenticate to the vault using workload identity rather than a long-lived bootstrap password.

**Dynamic secrets and ephemeral credentials**

Dynamic secrets are generated on demand and expire automatically after a short time. They are especially useful for databases, cloud roles, and temporary administrative access.

Benefits include:

- Reduced standing privilege.
- Automatic expiration.
- Easier revocation.
- Better auditability because each issuance is unique.

Example use cases:

- A web application requests a database credential valid for 15 minutes.
- A contractor receives a time-bound cloud role for a maintenance window.
- A build pipeline receives a short-lived token to pull artifacts from a repository.

Dynamic credentials are preferable to static shared passwords when the platform supports them. They reduce the risk that a stolen secret remains valid for weeks or months.

**Practical CLI examples**

The exact commands depend on the platform, but the patterns are consistent.

```bash
# View current privileged group membership on Linux
id
groups

# Check sudo privileges
sudo -l

# Use sudo for a single command
sudo systemctl restart nginx

# Inspect SSH authorized keys
cat ~/.ssh/authorized_keys

# Search for hard-coded secrets in a repository
grep -RInE 'password|secret|api[_-]?key|token' .
```

For cloud environments, you often use identity federation and short-lived tokens rather than static keys.

```bash
# Example: obtain a short-lived cloud session via CLI
aws sts get-session-token --duration-seconds 900

# Example: assume a role for temporary access
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/PrivilegedOps \
  --role-session-name incident-response
```

For secret managers, retrieval is usually done at runtime.

```bash
# Example: retrieve a secret from a vault
vault kv get -field=password secret/database/prod

# Example: write a secret
vault kv put secret/database/prod password='S3cure-Example-Value'
```

**Configuration patterns**

A common pattern is to inject secrets into applications at startup rather than storing them in files. Environment variables are convenient, but they are not a perfect secret boundary because they can be exposed through process inspection or debugging tools. Use them carefully and prefer platform-native secret injection when available.

```yaml
# Example Kubernetes secret reference pattern
apiVersion: v1
kind: Pod
metadata:
  name: app
spec:
  containers:
    - name: web
      image: example/web:1.0
      env:
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: db-credentials
              key: password
```

A stronger pattern is to use workload identity and fetch the secret at runtime from a vault with short-lived authorization.

```hcl
# Example Vault policy concept
path "secret/data/database/prod" {
  capabilities = ["read"]
}
```

**PAM workflow example**

A typical privileged access workflow looks like this:

1. You authenticate to the PAM portal with MFA.
2. You request access to a privileged account or session.
3. The request is approved automatically or by a manager, depending on policy.
4. The PAM system checks out a credential or brokers a session.
5. The session is recorded and time-limited.
6. The credential is rotated after use or on checkout return.
7. Logs are reviewed and correlated with a ticket or change record.

This workflow reduces the chance of standing access and improves traceability. It also supports incident response because you can identify who used which privilege, when, and for what purpose.

**Secrets in DevOps and CI/CD**

Continuous integration and continuous delivery (CI/CD) pipelines often need secrets for source control, artifact repositories, cloud APIs, and deployment targets. These secrets are high-value because pipelines can reach many systems quickly.

Good practices include:

- Store secrets in the pipeline platform’s secret store or an external vault.
- Use short-lived tokens where possible.
- Scope secrets to a single repository, environment, or job.
- Mask secrets in logs and prevent echoing them to the console.
- Rotate secrets after pipeline compromise or staff changes.
- Avoid placing secrets in build artifacts or container layers.

A common mistake is baking credentials into container images. Once an image is published, the secret may persist in registries, caches, and derived images. Prefer runtime injection or workload identity instead.

**Cloud and infrastructure considerations**

Cloud platforms shift privileged access from local admin accounts to identity and access management (IAM) roles, policies, and service principals. The same principles still apply: least privilege, short duration, logging, and rotation.

- **IAM roles** are preferable to long-lived access keys for workloads and operators.
- **Federation** lets you map an external identity provider to cloud access.
- **Conditional access** can require device posture, location, or risk checks.
- **Audit logs** should capture role assumption, policy changes, and secret access.
- **Resource-based policies** can reduce the need for broad account-level privilege.

For infrastructure as code, protect state files and variables because they may contain secrets. Terraform state, for example, can store sensitive values unless you design carefully and secure the backend.

```bash
# Example: inspect Terraform state for sensitive exposure risk
terraform state pull | grep -i password
```

**Monitoring, logging, and detection**

PAM and secrets management are only effective if you can detect misuse. Logging should cover authentication, authorization, secret retrieval, checkout, rotation, and administrative changes.

Monitor for:

- Privileged logons outside normal hours.
- Repeated failed access to vaults or admin portals.
- Secret retrieval from unusual hosts or geographies.
- Disabled logging or tampered audit settings.
- Unexpected rotation failures or stale credentials.
- Use of break-glass accounts without corresponding incident records.

Session recording is especially valuable because it provides behavioral evidence, not just authentication metadata. For command-line sessions, capture commands and outputs where feasible. For GUI administration, record screen activity and timestamps.

**Break-glass and emergency access**

Break-glass accounts are emergency credentials used when normal authentication, federation, or PAM workflows are unavailable. They are necessary, but they must be tightly controlled.

- Store them in a sealed, monitored, and access-controlled location.
- Use them only for emergencies with documented justification.
- Rotate them immediately after use.
- Test them periodically to ensure they still work.
- Alert on every use.

Break-glass access should not become a convenience backdoor. If it is used routinely, your normal access design is too restrictive or unreliable.

**Common risks and anti-patterns**

- **Shared credentials** make attribution difficult and increase compromise impact.
- **Hard-coded secrets** in code or scripts are easy to leak.
- **Overprivileged service accounts** expand the blast radius of compromise.
- **Long-lived API keys** remain valid long after the original need ends.
- **Manual password spreadsheets** create uncontrolled distribution.
- **Unlogged emergency access** weakens incident reconstruction.
- **Infrequent rotation** leaves stolen secrets usable for too long.

> Critical warning: if a secret appears in source control, assume it is compromised. Deleting the file does not remove it from history, forks, caches, or backups.

**Exam-focused distinctions**

CISSP questions often test whether you can distinguish related controls.

- **PAM** governs privileged identities and sessions.
- **Secrets management** governs sensitive values used by humans and machines.
- **MFA** strengthens authentication but does not replace least privilege.
- **Encryption** protects data confidentiality but does not control who should access it.
- **Logging** supports accountability but does not prevent misuse by itself.
- **Rotation** limits exposure time but does not fix excessive privilege.

When choosing the best answer, prefer controls that reduce standing privilege, enforce accountability, and minimize secret exposure. If a scenario involves admin access, think PAM. If it involves passwords, tokens, keys, or certificates, think secrets management. If it involves both, the best answer often combines vaulting, rotation, and time-bound access.

**Quick reference checklist**

- Separate admin and user accounts.
- Enforce MFA for privileged access.
- Use JIT or time-bound elevation.
- Vault secrets and rotate them regularly.
- Prefer dynamic credentials over static shared passwords.
- Record privileged sessions and review logs.
- Protect CI/CD, cloud roles, and infrastructure state.
- Use break-glass access only for emergencies.
- Remove secrets from code, images, and repositories.
- Revoke access promptly when roles change or projects end.
