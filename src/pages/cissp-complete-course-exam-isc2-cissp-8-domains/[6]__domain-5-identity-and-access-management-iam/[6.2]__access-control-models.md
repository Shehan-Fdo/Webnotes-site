## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Access control fundamentals**

Access control determines who can access what, under which conditions, and with what level of privilege. In CISSP terms, you are usually evaluating three things:

- **Subject**: the active entity requesting access, such as a user, process, service account, or device.
- **Object**: the resource being protected, such as a file, database row, API, printer, or directory.
- **Operation**: the action requested, such as read, write, execute, delete, or approve.

A useful mental model is the decision flow below:

```text
Subject -> Request -> Policy Check -> Decision -> Object Access
             |             |
             |             +--> Allow / Deny / Conditional Allow
             +--> Identity, attributes, roles, context, labels
```

Access control models define the logic used to make that decision. CISSP expects you to distinguish not only the classic models, but also how they are implemented together in real environments.

**Access Control Matrix as the foundation**

The **Access Control Matrix** is the conceptual foundation behind many access control discussions. It maps subjects to objects and lists the permitted operations for each pairing.

- Rows represent **subjects**.
- Columns represent **objects**.
- Cells contain allowed **rights** or permissions.

A simple example:

| Subject | File A | File B | Printer |
|---|---:|---:|---:|
| Alice | read, write | read | print |
| Bob | read | none | print |
| ServiceAcct | read | read, write | none |

This matrix is rarely implemented directly because it becomes large and sparse. Instead, systems store it in more efficient forms:

- **ACLs (Access Control Lists)** store permissions with the object.
- **Capability lists** store permissions with the subject.

That distinction matters for exams:

- **ACLs** answer: “Who can access this object?”
- **Capability lists** answer: “What can this subject access?”

In practice, file systems often use ACLs, while some distributed systems and older capability-based designs emphasize subject-held tokens or capabilities.

**Discretionary Access Control (DAC)**

**DAC** lets the object owner decide who gets access. It is “discretionary” because the owner can delegate permissions.

Typical characteristics:

- The owner controls permissions.
- Permissions are often granted to users or groups.
- It is flexible and easy to use.
- It is weaker against privilege propagation and accidental sharing.

Common examples:

- A user shares a folder with another user.
- A database owner grants `SELECT` on a table.
- A file owner changes permissions with `chmod` or ACL tools.

Linux example:

```bash
chmod 640 report.txt
setfacl -m u:alice:rw report.txt
getfacl report.txt
```

Why DAC matters:

- It is common in general-purpose operating systems and collaboration environments.
- It scales reasonably for small teams, but administrative control can become inconsistent in large enterprises.
- It is often the least restrictive of the classic models.

Exam angle:

- DAC is associated with user-controlled sharing.
- It is not the best choice when strict centralized control is required.

**Mandatory Access Control (MAC)**

**MAC** uses centrally enforced labels and rules. Users and owners do not decide access on their own; the system enforces policy.

Core ideas:

- Subjects and objects receive security labels.
- The system compares labels against a policy.
- Access is allowed only if the policy permits it.

Common examples:

- Military classification such as `Secret` and `Top Secret`.
- SELinux and AppArmor-style mandatory controls.
- Data classification systems in highly regulated environments.

Linux SELinux example:

```bash
sestatus
ls -Z /var/www/html
chcon -t httpd_sys_content_t /var/www/html/index.html
```

Why MAC matters:

- It provides strong centralized control.
- It reduces the risk of users bypassing policy.
- It is more complex to administer than DAC.

Exam angle:

- MAC is often described as label-based and non-discretionary.
- It is common in environments where confidentiality and policy enforcement are critical.

**Lattice-based access control**

**Lattice-based access control** is a specific structure often associated with MAC. It is more precise than simply saying “MAC uses labels.”

A lattice is a partially ordered set of security levels and categories. Access decisions follow dominance rules.

Example:

- `Top Secret` dominates `Secret`
- `Secret` dominates `Confidential`
- A subject must dominate an object to read it
- Additional categories may also be required, such as `NATO`, `ProjectX`, or `Finance`

Why this is distinct:

- General MAC is the broader model.
- Lattice-based access control is the formal structure often used to implement MAC in high-assurance systems.
- CISSP may use wording that implies ordered security levels and dominance relationships.

Practical interpretation:

- A user with `Secret` clearance cannot read `Top Secret` data.
- A user with `Top Secret` clearance may still be blocked if they lack the required category.

This model is especially relevant when exam questions mention:

- security labels
- dominance
- compartments/categories
- multilevel security

**Role-Based Access Control (RBAC)**

**RBAC** assigns permissions to roles, and users inherit permissions by being assigned to roles. It is one of the most common enterprise models because it aligns access with job function.

Core elements:

- **Users** are assigned to roles.
- **Roles** are assigned permissions.
- **Permissions** are tied to operations on objects.

Example:

- `HelpDesk` role can reset passwords.
- `HR` role can access employee records.
- `FinanceApprover` role can approve payments.

Active Directory example:

- Users are placed into security groups.
- Groups are mapped to file shares, applications, or application roles.
- This is a practical RBAC implementation even if the platform uses the term “group” more often than “role.”

Example PowerShell:

```powershell
Add-ADGroupMember -Identity "HR-ReadOnly" -Members "alice"
Get-ADGroupMember -Identity "HR-ReadOnly"
```

RBAC subtypes and exam distinctions:

- **Flat RBAC**: simple user-to-role assignment with permissions on roles.
- **Hierarchical RBAC**: roles inherit permissions from other roles.
- **Constrained RBAC**: adds restrictions such as separation of duties.
- **Symmetric RBAC**: roles can be assigned based on permissions and vice versa in some models, though this is less commonly emphasized in CISSP.

**RBAC hierarchies**

Hierarchies reduce duplication by letting senior roles inherit junior role permissions.

Example:

- `Employee` role has basic access.
- `Manager` inherits `Employee` permissions.
- `Director` inherits `Manager` permissions.

Benefits:

- Simplifies administration.
- Reflects organizational structure.
- Reduces repeated permission assignment.

Risk:

- Over-inheritance can create excessive privilege if role design is poor.

**Constrained RBAC and separation of duties**

**Separation of duties (SoD)** prevents one person from controlling an entire sensitive process. In RBAC, this is often enforced through constraints.

Examples:

- The person who creates a purchase order cannot approve it.
- The person who develops code cannot deploy it to production alone.
- The person who initiates a payment cannot also release it.

Types of constraints:

- **Static separation of duties**: a user cannot be assigned conflicting roles at the same time.
- **Dynamic separation of duties**: a user may hold both roles, but cannot activate both in the same session.

Why this matters:

- It reduces fraud and abuse.
- It supports compliance requirements.
- It is a common CISSP exam concept because it combines access control with governance.

**Attribute-Based Access Control (ABAC)**

**ABAC** makes decisions using attributes about the subject, object, action, and environment. It is more flexible than RBAC because it can evaluate context in real time.

Typical attributes:

- **Subject attributes**: department, clearance, device posture, location, employment status.
- **Object attributes**: data classification, owner, sensitivity, project tag.
- **Action attributes**: read, write, approve, export.
- **Environmental attributes**: time of day, network zone, risk score, MFA status.

A common policy example:

- Allow access if the user is in `Finance`
- and the resource is tagged `Payroll`
- and the request comes from a managed device
- and the user has completed MFA
- and the request occurs during business hours

ABAC is often described in CISSP with three policy components:

- **PEP (Policy Enforcement Point)**: intercepts the request and enforces the decision.
- **PDP (Policy Decision Point)**: evaluates policy and returns allow/deny.
- **PIP (Policy Information Point)**: supplies attribute data used in the decision.

Simple flow:

```text
User Request -> PEP -> PDP -> PIP data lookup -> Decision -> PEP enforces
```

Practical cloud example:

- AWS IAM, Azure Conditional Access, and similar systems often use attribute-like conditions.
- You can require MFA, device compliance, source IP range, or session risk before allowing access.

Example policy style:

```json
{
  "Effect": "Allow",
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::finance-bucket/*",
  "Condition": {
    "Bool": { "aws:MultiFactorAuthPresent": "true" },
    "StringEquals": { "aws:PrincipalTag/Department": "Finance" }
  }
}
```

Why ABAC matters:

- It scales better than role explosion in large, dynamic environments.
- It supports fine-grained decisions.
- It is ideal when access depends on context, not just job title.

**Rule-based access control**

**Rule-based access control** uses if-then rules to grant or deny access. In CISSP wording, this is often treated as a policy mechanism rather than a primary classic model like DAC, MAC, RBAC, or ABAC.

Examples:

- Allow access only during business hours.
- Deny access from untrusted networks.
- Permit printing only on weekdays.
- Block access if the device is not compliant.

Important distinction:

- Rule-based control can look similar to ABAC because both use conditions.
- ABAC is broader and attribute-driven across subject, object, action, and environment.
- Rule-based control is often narrower and policy-centric, sometimes layered on top of another model.

Exam tip:

- If the question emphasizes fixed policy rules, think rule-based.
- If it emphasizes multiple attributes and context-aware decisions, think ABAC.

**Comparing DAC, MAC, RBAC, and ABAC**

The exam often asks you to choose the best model for a scenario. Focus on administration overhead, scalability, and typical use cases.

| Model | Who decides access | Admin overhead | Scalability | Typical use case |
|---|---|---:|---:|---|
| DAC | Object owner | Low to medium | Moderate | Small teams, file sharing, general OS permissions |
| MAC | Central policy/system | High | Moderate | Military, government, high-security environments |
| RBAC | Role administrators | Medium | High | Enterprises, business applications, help desk, HR |
| ABAC | Policy engine using attributes | Medium to high initially, lower at scale | Very high | Cloud, zero trust, dynamic enterprise access |

Deeper exam-prep distinctions:

- **DAC** is easy to deploy but can become chaotic as sharing grows.
- **MAC** is strongest for centralized control but is rigid and expensive to manage.
- **RBAC** is the most intuitive for organizations with stable job functions.
- **ABAC** is best when access depends on many changing conditions and you need fine-grained control.

Typical organizational fit:

- Small office file sharing: DAC
- Classified government systems: MAC or lattice-based MAC
- Corporate business apps: RBAC
- Cloud and hybrid environments: ABAC or hybrid RBAC + ABAC

**Hybrid access control in real environments**

Most real systems combine models rather than using just one.

Common combinations:

- **RBAC + ABAC**: role grants baseline access, attributes add conditions.
- **MAC + DAC**: classification controls the data, while owners may manage sharing within allowed boundaries.
- **RBAC + rule-based controls**: role determines eligibility, rules restrict time, location, or device.
- **ACLs + RBAC**: groups or roles are mapped to object ACL entries.

Enterprise examples:

- An HR application uses RBAC for job function, but ABAC conditions require managed devices and MFA.
- A file server uses ACLs for folder permissions, while the operating system enforces MAC labels on sensitive directories.
- A cloud platform grants a role to a workload, then applies conditional access based on source network and token claims.

Why hybrid designs are common:

- No single model solves every requirement.
- Organizations need both usability and precision.
- Security teams often layer controls to reduce risk without creating unmanageable policy sprawl.

**ACLs and capability lists in relation to the matrix**

The access control matrix is the conceptual model; ACLs and capability lists are implementation strategies.

**ACLs**

- Stored with the object.
- Good for answering “who can access this file?”
- Common in file systems, directories, and network devices.

Example:

```text
File: payroll.xlsx
ACL:
- FinanceGroup: read
- PayrollAdmins: read, write
- Auditor: read
```

**Capability lists**

- Stored with the subject.
- Good for answering “what can this process or user access?”
- Often used in systems where possession of a capability token implies authority.

Example:

```text
User Alice capability list:
- File A: read
- Printer 2: print
- API token: invoke /reports
```

Exam distinction:

- ACLs are object-centric.
- Capability lists are subject-centric.
- Both are ways to implement the abstract matrix.

**Practical enterprise examples**

**Active Directory and RBAC**

In Windows environments, groups often function as roles.

- `Finance-Read`
- `Finance-Approve`
- `IT-Admin`

You assign users to groups and map groups to resources. This is easier to manage than assigning permissions individually.

Example:

```powershell
New-ADGroup -Name "Finance-Read" -GroupScope Global
Add-ADGroupMember -Identity "Finance-Read" -Members "jdoe"
```

**Cloud IAM and ABAC**

Cloud platforms often support tag-based or condition-based access.

- A resource may have tags like `Department=Finance`.
- A principal may have claims like `Department=Finance`.
- A policy allows access only when tags match and MFA is present.

This is especially useful in multi-account or multi-project environments where roles alone would explode in number.

**SELinux and MAC**

SELinux labels processes and files with security contexts.

- A web server process may only read web content.
- Even if a user has file permissions, SELinux can still block access.
- This is a classic example of MAC overriding discretionary permissions.

**AppArmor and MAC**

AppArmor uses path-based profiles to confine applications.

- It limits what a process can read, write, or execute.
- It is often easier to deploy than more complex mandatory systems.
- It is still a form of mandatory control because the policy is centrally enforced.

**Database access**

Databases often combine models:

- RBAC for application roles like `report_reader` and `db_admin`
- DAC-like ownership for schema objects
- Rule-based restrictions for maintenance windows
- ABAC-like conditions in modern data platforms based on tenant, region, or sensitivity

**How to choose the right model**

Use the model that best matches the control problem:

1. If the owner should decide sharing, use **DAC**.
2. If the system must enforce labels and classification, use **MAC** or lattice-based control.
3. If access follows job function, use **RBAC**.
4. If access depends on dynamic context and multiple attributes, use **ABAC**.
5. If you need time, location, or device restrictions layered on top, add **rule-based** controls or ABAC conditions.
6. If the environment is real, expect a **hybrid** design.

**Exam traps and common confusions**

- **RBAC vs ABAC**
  - RBAC is role-centered.
  - ABAC is attribute-centered.
  - If the policy changes often based on context, ABAC is usually the better answer.

- **MAC vs lattice-based MAC**
  - MAC is the general model.
  - Lattice-based control is the formal ordered structure often used in high-security MAC systems.

- **Rule-based vs ABAC**
  - Rule-based control is often a policy overlay.
  - ABAC is a full model using multiple attributes and a policy engine.

- **ACL vs capability list**
  - ACLs are attached to objects.
  - Capability lists are attached to subjects.

- **RBAC and separation of duties**
  - RBAC alone does not guarantee SoD.
  - You need constraints to prevent conflicting role combinations.

**Quick reference summary**

- **DAC**: owner-controlled, flexible, lower security consistency.
- **MAC**: centrally enforced labels, strong control, less flexible.
- **Lattice-based control**: ordered security levels and dominance rules.
- **RBAC**: permissions tied to roles, best for stable organizations.
- **ABAC**: decisions based on attributes and context, best for dynamic environments.
- **Rule-based**: policy conditions, often layered rather than standalone.
- **ACLs**: object-side permission lists.
- **Capability lists**: subject-side permission lists.
- **Hybrid models**: common in enterprise and cloud systems.

**Practical study mindset**

When you read a CISSP scenario, ask:

- Who controls the permission?
- Is access based on ownership, labels, roles, or attributes?
- Is the policy static or context-aware?
- Is the system using one model or a combination?
- Does the question emphasize administration, scalability, or enforcement strength?

That approach helps you map the scenario to the correct model instead of memorizing definitions in isolation.
