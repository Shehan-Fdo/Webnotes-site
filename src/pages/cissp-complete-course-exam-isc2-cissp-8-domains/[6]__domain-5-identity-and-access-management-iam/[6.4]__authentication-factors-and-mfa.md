## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Authentication fundamentals**

Authentication answers the question: “Who are you?” It verifies an identity claim before access is granted. In CISSP terms, it is distinct from identification, authorization, and accounting, and exam questions often test those boundaries.

- **Identification** is the claim of identity, such as entering a username or presenting an ID badge.
- **Authentication** is the proof of that claim, such as a password, token, or biometric.
- **Authorization** determines what an authenticated subject can do.
- **Accounting** records actions for audit, traceability, and nonrepudiation support.

A common AAA model is:

```text
Identification -> Authentication -> Authorization -> Accounting
```

In practice, you may see these steps overlap in modern systems, but the conceptual separation still matters for design and exam questions. For example, a user can be identified by email, authenticated with a passkey, authorized to access a finance app, and accounted for through logs and session records.

**Authentication factor categories**

Authentication factors are usually grouped into three categories:

- **Something you know**: password, PIN, passphrase, answers to knowledge-based questions.
- **Something you have**: smart card, hardware token, mobile authenticator app, cryptographic key, passkey device.
- **Something you are**: fingerprint, face geometry, iris pattern, voice pattern.

A factor is only useful if it is independent enough from the others. CISSP questions often focus on whether the factors are truly different categories, not just different secrets.

- Password + PIN is not MFA in the strict sense because both are **something you know**.
- Password + security questions is also not strong MFA for the same reason.
- Password + fingerprint is MFA because it combines knowledge and inherence.
- Smart card + PIN is MFA because it combines possession and knowledge.
- Passkey on a device protected by local biometrics can be MFA depending on the implementation, because the cryptographic authenticator is something you have and the biometric unlocks local use.

> A key exam trap: two checks from the same category do not satisfy MFA, even if they feel like two steps.

**MFA and why it matters**

Multi-factor authentication (MFA) reduces the chance that a single stolen credential leads to compromise. It is especially important for remote access, privileged accounts, cloud consoles, and high-value transactions.

MFA is strongest when the factors are:

- Independent
- Resistant to replay
- Bound to the intended service
- Hard to phish or intercept
- Usable enough that users do not bypass it

MFA is not a silver bullet. Attackers target the weakest link, which is often the recovery process, the help desk, or the session after authentication.

**Valid and invalid factor combinations**

Use this distinction carefully in exam scenarios.

| Combination | MFA? | Why |
|---|---:|---|
| Password + PIN | No | Both are knowledge factors |
| Password + security question | No | Both are knowledge factors |
| Password + OTP from phone app | Yes | Knowledge + possession |
| Password + fingerprint | Yes | Knowledge + inherence |
| Smart card + PIN | Yes | Possession + knowledge |
| Smart card + SMS code | Yes, but weaker | Possession + possession-ish delivery path; SMS is vulnerable to interception and SIM swap |
| Face unlock + fingerprint | No | Both are inherence factors |
| Passkey + device PIN | Often yes | Device-bound possession plus local unlock, depending on implementation |
| Two passwords for two systems | No | Still knowledge only |

For CISSP, the safest interpretation is that MFA requires at least two different factor categories. Two items from the same category do not satisfy the intent of MFA.

**Authentication vs adaptive authentication**

Adaptive authentication, also called risk-based authentication, changes the authentication challenge based on context and risk signals. It is distinct from static MFA because it evaluates the situation before deciding whether to require more proof.

Common risk signals include:

- New device or browser
- Impossible travel
- Unusual geolocation
- Tor or anonymizing network use
- Suspicious IP reputation
- Time of day anomalies
- Privilege escalation attempts
- High-risk transaction amount
- Behavioral deviations, such as typing cadence or mouse movement

A typical flow is:

1. User signs in with a normal factor set.
2. The system evaluates context and risk.
3. If risk is low, access continues.
4. If risk is elevated, the system triggers **step-up authentication**.
5. The user must provide a stronger factor or additional verification.

Example policy logic:

```text
IF user = finance_admin AND device = unmanaged AND location = new_country
THEN require phishing-resistant MFA + manager approval
```

Adaptive authentication is useful because it balances security and usability. You do not want to force the highest friction challenge for every login if the risk is low, but you do want stronger proof when the context changes.

**MFA strength and phishing resistance**

Not all MFA methods are equally resistant to attack. The exam often distinguishes between legacy OTP-based methods and modern phishing-resistant methods.

- **OTP** means one-time password, usually a short-lived code.
- **TOTP** means time-based one-time password.
- **HOTP** means HMAC-based one-time password.

OTP-based MFA improves security over passwords alone, but it can still be phished in real time. A user can enter the code into a fake login page, and the attacker can immediately relay it to the real service.

Common bypass and attack techniques include:

- **Man-in-the-middle phishing kits** that proxy the login session and capture credentials plus OTPs in real time.
- **Session hijacking** after authentication, where the attacker steals the session cookie or token.
- **Token replay** when a captured token is reused before it expires.
- **Push fatigue** or MFA bombing, where repeated prompts pressure the user into approving a login.
- **SIM swap** attacks against SMS-based OTP delivery.
- **Help desk social engineering** to reset factors or enroll a new device.

Because of these threats, CISSP favors phishing-resistant MFA for high-risk use cases. Examples include:

- FIDO2 security keys
- WebAuthn passkeys
- Certificate-based authentication
- Smart cards with private keys protected by hardware

**Bearer tokens and proof-of-possession tokens**

Token-based authentication is common in modern systems, especially APIs and single sign-on. The exam may ask whether a token is a bearer token or a proof-of-possession token.

- A **bearer token** grants access to whoever presents it. If stolen, it can often be replayed until it expires or is revoked.
- A **proof-of-possession** token is bound to a key or device, so the attacker needs more than the token value alone.

Bearer tokens are weaker if stolen because possession of the string is enough. Proof-of-possession designs reduce replay risk by requiring cryptographic proof tied to the original holder.

Example conceptual difference:

```text
Bearer token: "Whoever has this token can use it."
PoP token: "You must prove you hold the private key associated with this token."
```

In web and API environments, you often see bearer-style access tokens in OAuth 2.0 deployments. Stronger designs may use sender-constrained tokens, mutual TLS, or DPoP-style proof mechanisms to reduce replay risk.

**Biometrics and exam tradeoffs**

Biometrics are convenient, but they are not secrets in the same way passwords are. You cannot change your fingerprint if it is compromised, so biometrics are usually best treated as a local unlock mechanism for a cryptographic authenticator rather than a standalone remote factor.

Important biometric concepts:

- **False Acceptance Rate (FAR)** is the chance that an unauthorized person is accepted.
- **False Rejection Rate (FRR)** is the chance that an authorized person is rejected.
- Lowering FAR usually increases FRR.
- Lowering FRR usually increases FAR.

This is the classic security-versus-usability tradeoff. A high-security environment may tolerate more false rejections to reduce unauthorized access. A consumer device may tolerate more false acceptances if convenience is prioritized, though that is less desirable for sensitive systems.

Practical interpretation:

- For a high-risk admin console, you want a stricter threshold and lower FAR.
- For a low-risk convenience unlock, you may accept a slightly higher FRR to improve usability.
- For remote authentication, biometrics alone are usually insufficient because they can be spoofed, replayed, or captured from a sensor interface.

Biometric systems also need liveness detection and anti-spoofing controls, but these are not perfect. A fingerprint reader on a laptop is best used to unlock a private key stored in secure hardware, not as the only proof of identity over the network.

**Passkeys, FIDO2, and WebAuthn**

Passkeys are a modern phishing-resistant authentication approach built on public-key cryptography. In many deployments, the authenticator generates a key pair and the private key never leaves the device or secure enclave.

Core ideas:

- The service stores the public key.
- The device stores or protects the private key.
- The user unlocks the authenticator locally with a PIN or biometric.
- The browser or app uses WebAuthn to complete the challenge.

A typical enrollment flow looks like this:

1. You register a new authenticator with the service.
2. The service sends a registration challenge.
3. The device creates a key pair for that site.
4. The public key and attestation data are sent back.
5. The service stores the public key for future logins.

A typical login flow looks like this:

1. You enter your username or choose the account.
2. The service sends a challenge.
3. The authenticator signs the challenge with the private key.
4. The browser returns the assertion to the service.
5. The service verifies the signature and grants access.

Example WebAuthn-style conceptual flow:

```javascript
const credential = await navigator.credentials.create({
  publicKey: {
    challenge: crypto.getRandomValues(new Uint8Array(32)),
    rp: { name: "Example Corp" },
    user: { id: userId, name: "alice@example.com", displayName: "Alice" },
    pubKeyCredParams: [{ type: "public-key", alg: -7 }]
  }
});
```

For exam purposes, the key point is that passkeys and FIDO2 reduce phishing risk because the credential is origin-bound. A fake site cannot easily use the authenticator for the real site.

**Recovery and backup factor risks**

Recovery is one of the most attacked parts of an authentication system. If an attacker cannot break MFA directly, they may target account recovery, backup codes, or support workflows.

Hardening options include:

- Requiring backup factor registration at enrollment
- Limiting how many backup factors can be added
- Requiring reauthentication before adding a new factor
- Sending alerts when factors change
- Delaying factor changes for high-risk accounts
- Requiring manager or security approval for privileged users
- Storing recovery codes offline in a password manager or secure vault
- Treating recovery codes like passwords, because they often are equivalent to account access

Help desk identity proofing is a major attack surface. If support staff can reset MFA after weak verification, the strongest technical controls can be bypassed socially.

Good recovery design usually includes:

- Strong identity proofing for resets
- Audit logging of all recovery actions
- Out-of-band notification to existing trusted channels
- Time delays for sensitive changes
- Escalation paths for privileged accounts
- Separation of duties for support and approval

> If recovery is easier than login, attackers will target recovery.

**Authentication protocols and common examples**

Different protocols implement authentication in different ways, but the exam usually cares more about the security properties than the brand names.

- **Kerberos** uses tickets and a trusted Key Distribution Center (KDC) to support single sign-on in enterprise environments.
- **RADIUS** and **TACACS+** are common for network access and administrative authentication.
- **LDAP** can support directory-based authentication, often with bind operations.
- **SAML** supports federated authentication for browser-based single sign-on.
- **OAuth 2.0** is primarily authorization, not authentication, though it is often paired with OpenID Connect for login.
- **OpenID Connect (OIDC)** adds authentication on top of OAuth 2.0.

A common exam pitfall is confusing OAuth with authentication. OAuth grants delegated access to resources; it does not by itself prove identity for login.

**Practical CLI and configuration examples**

You may not configure MFA directly from the command line in every environment, but real-world administration often includes identity provider setup, SSH hardening, and policy enforcement.

Example SSH configuration that prefers public key authentication and disables password login:

```sshconfig
PasswordAuthentication no
PubkeyAuthentication yes
KbdInteractiveAuthentication no
AuthenticationMethods publickey
```

Example PAM (Pluggable Authentication Modules) concept for Linux MFA integration:

```text
auth required pam_u2f.so cue
auth required pam_google_authenticator.so nullok
```

Example Azure AD / Entra-style conditional access concept:

```text
IF user = privileged_role
AND device = not_compliant
THEN block access
ELSE IF risk = medium
THEN require MFA
```

Example RADIUS-style network access policy:

```text
if NAS-Port-Type = Wireless-802.11 and User-Group = Admin
then require MFA and device compliance
```

Example recovery code handling guidance:

```bash
# Store recovery codes in an encrypted password manager vault
# Do not store them in plain text files, email, or shared chat
chmod 600 recovery-codes.txt
```

**Choosing the right factor set**

Factor choice depends on the threat model, user population, and operational constraints.

- For consumer convenience, password plus push approval may be acceptable, but it is vulnerable to push fatigue and phishing.
- For enterprise remote access, use phishing-resistant MFA such as FIDO2 or smart cards.
- For privileged administrators, prefer hardware-backed factors and step-up controls.
- For high-assurance environments, combine strong identity proofing, hardware keys, and strict recovery controls.

A useful rule is to match the factor to the risk:

- Low risk: simple MFA may be enough.
- Medium risk: use app-based OTP or push with additional controls.
- High risk: use phishing-resistant MFA and adaptive policies.
- Very high risk: require hardware-backed, origin-bound authentication plus strong recovery governance.

**Common exam traps**

- Two factors from the same category do not count as MFA.
- Biometrics are not ideal as the only remote factor.
- OAuth is not authentication by itself.
- OTPs are better than passwords alone, but they are still phishable.
- A stolen bearer token can often be replayed.
- Recovery and help desk workflows are part of the authentication attack surface.
- Adaptive authentication is not the same as static MFA.
- Authorization is not authentication.

**Quick comparison of factor properties**

| Factor type | Strengths | Weaknesses | Best use |
|---|---|---|---|
| Password | Familiar, cheap | Phishable, reused, guessed | Baseline access with compensating controls |
| PIN | Simple, local unlock | Short, guessable | Device unlock, not sole remote auth |
| OTP app | Better than password alone | Phishable, replayable in real time | Moderate-risk MFA |
| SMS OTP | Easy to deploy | SIM swap, interception | Legacy fallback only |
| Hardware key | Phishing-resistant | Cost, logistics | High-value accounts |
| Biometrics | Convenient | FAR/FRR tradeoff, spoofing, non-revocable | Local unlock |
| Passkey | Strong, user-friendly | Enrollment and recovery complexity | Modern primary MFA |

**Study takeaway**

For CISSP, think in terms of categories, attack resistance, and operational risk. The best answer is usually the one that combines independent factors, resists phishing and replay, and includes secure recovery. Strong authentication is not just about the login prompt; it includes enrollment, step-up decisions, session protection, and recovery governance.
