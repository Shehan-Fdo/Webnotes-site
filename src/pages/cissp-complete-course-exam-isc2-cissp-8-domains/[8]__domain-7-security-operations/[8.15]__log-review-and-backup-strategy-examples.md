**Log review and backup strategy: why they matter**

Log review and backup strategy are core operational security controls. Log review helps you detect suspicious activity, support incident response, and prove compliance. Backup strategy protects availability, supports disaster recovery (DR), and reduces the impact of ransomware, deletion, corruption, and hardware failure.

For CISSP, focus on the relationship between **visibility** and **recoverability**:
- Logs tell you what happened, when, where, and by whom.
- Backups let you restore systems and data after loss or compromise.
- Both require retention, integrity, access control, and testing to be useful.

**Log review fundamentals**

Logs are only valuable if they are collected, protected, and reviewed in a way that supports detection and investigation. You should think in terms of source, normalization, correlation, and retention.

Common log sources include:
- **Operating system logs** such as authentication events, privilege use, service starts, and process failures.
- **Application logs** such as transaction errors, access denials, and application-specific audit trails.
- **Security device logs** such as firewall denies, IDS/IPS alerts, VPN sessions, and proxy activity.
- **Identity logs** such as directory service authentication, password resets, and MFA events.
- **Cloud logs** such as API calls, control plane actions, and storage access events.

A practical review process usually includes:
- Establishing what events are normal for each system.
- Prioritizing high-risk events such as failed logons, privilege escalation, account creation, and configuration changes.
- Correlating events across systems to identify patterns that a single log source would miss.
- Preserving logs in a tamper-resistant location with synchronized time.

**Time synchronization and log integrity**

Accurate timestamps are essential for correlation. If systems do not share a common time source, incident timelines become unreliable.

Use Network Time Protocol (NTP) or a managed time service to keep systems aligned:
```bash
# Linux example
timedatectl set-ntp true
timedatectl status

# Query time sources
chronyc sources -v
```

Protect log integrity by:
- Restricting write access to log collectors and administrators.
- Forwarding logs to a centralized system so local attackers cannot easily erase evidence.
- Using append-only storage, immutable buckets, or write-once media where appropriate.
- Hashing or signing logs when the environment requires stronger nonrepudiation.

> If an attacker can alter both the system and its local logs, you lose trust in the evidence. Centralized collection reduces that risk but does not eliminate it.

**What to look for during log review**

Effective review is not random browsing. It is a risk-based process that looks for indicators of compromise, policy violations, and operational anomalies.

Examples of suspicious or high-value events:
- Repeated failed authentication followed by a success.
- Logins from unusual geographies or impossible travel patterns.
- Privileged account use outside normal maintenance windows.
- New services, scheduled tasks, or startup items.
- Unexpected changes to security groups, firewall rules, or IAM permissions.
- Large data transfers, especially to external destinations.
- Disabled logging, cleared event logs, or stopped agents.

A simple Linux review workflow might include:
```bash
# Authentication events
grep -i "failed\|invalid\|accepted" /var/log/auth.log

# Systemd journal review
journalctl -p warning..alert --since "today"

# Search for service changes
journalctl | grep -i "started\|stopped\|failed"
```

On Windows, common review locations include:
- **Security** log for authentication and audit events.
- **System** log for service and driver issues.
- **Application** log for application errors.
- Event Viewer or centralized SIEM dashboards for filtering and correlation.

**Centralized logging and SIEM**

A Security Information and Event Management (SIEM) platform aggregates logs, normalizes them, and applies correlation rules and alerts. This is useful when you need cross-system visibility, long retention, and faster detection.

Typical SIEM benefits:
- Central search across many hosts and applications.
- Correlation of weak signals into stronger alerts.
- Dashboards for compliance and operations.
- Alerting for threshold-based or behavior-based anomalies.

Common implementation pattern:
```text
Endpoint / server / firewall
        -> log forwarder
        -> collector
        -> SIEM
        -> alerting / case management / archive
```

Practical considerations:
- Filter out low-value noise before storage if policy allows.
- Keep raw logs for investigations and normalized data for search.
- Protect the SIEM itself with strong access control and monitoring.
- Define retention periods based on legal, regulatory, and business needs.

**Backup strategy fundamentals**

A backup strategy defines what you back up, how often, where it is stored, how long it is retained, and how quickly it can be restored. The best strategy is driven by business requirements, not by storage convenience.

Key terms:
- **Recovery Point Objective (RPO):** the maximum acceptable data loss measured in time.
- **Recovery Time Objective (RTO):** the maximum acceptable downtime before service must be restored.
- **Retention:** how long backup copies are kept.
- **Restore testing:** verification that backups can actually be recovered.

A backup plan should answer:
- What data and systems are critical?
- How much data loss is acceptable?
- How long can the business tolerate downtime?
- Which backups are needed for operational recovery versus legal hold or archival?

**Backup types and when to use them**

Different backup types trade speed, storage, and restore complexity.

| Type | Description | Strengths | Tradeoffs |
|---|---|---|---|
| Full | Copies all selected data | Simple restore, complete baseline | Uses more time and storage |
| Incremental | Copies changes since the last backup of any type | Fast, efficient storage use | Restore may require multiple backup sets |
| Differential | Copies changes since the last full backup | Faster restore than incremental | Grows larger until next full backup |
| Snapshot | Point-in-time copy at storage or VM layer | Fast creation, useful for short-term recovery | Not always a substitute for backup |
| Image-based | Captures a full system image | Good for bare-metal recovery | Larger and less granular |

Examples:
- Use **full backups** weekly and **incremental backups** daily when storage efficiency matters.
- Use **differential backups** when you want simpler restores than incremental chains.
- Use **snapshots** for quick rollback, but do not rely on them alone for long-term protection.
- Use **image-based backups** for servers that must be rebuilt quickly after failure.

**The 3-2-1 backup rule**

A common resilience pattern is the 3-2-1 rule:
- Keep **3** copies of data.
- Store them on **2** different media or storage types.
- Keep **1** copy offsite.

A modern extension is the 3-2-1-1-0 approach:
- **1** copy is offline or immutable.
- **0** backup verification errors after testing.

This matters because ransomware often targets online backups first. An offline or immutable copy gives you a recovery path when primary and secondary systems are compromised.

**Backup storage and protection options**

Backups are only useful if they survive the same threats as production systems.

Protect backup data with:
- **Encryption at rest** to reduce exposure if media is stolen.
- **Encryption in transit** when sending backups to remote repositories.
- **Access control** limiting who can create, delete, or restore backups.
- **Immutability** such as object lock or WORM (Write Once, Read Many) storage.
- **Air gaps** for especially critical environments.

Example backup command using `rsync` for a simple file-level copy:
```bash
rsync -aHAX --delete /srv/data/ /backup/data/
```

Example with `tar` for a compressed archive:
```bash
tar -czf /backup/home-$(date +%F).tar.gz /home
```

Example with `restic` for encrypted, deduplicated backups:
```bash
export RESTIC_REPOSITORY=/mnt/backup/restic
export RESTIC_PASSWORD='use-a-strong-secret'
restic backup /etc /home
restic snapshots
```

**Backup scheduling and retention**

Backup frequency should match the rate of change and the acceptable RPO. High-change systems need more frequent backups or replication.

A practical schedule example:
- **Databases:** frequent transaction log backups plus nightly full backups.
- **File servers:** nightly incremental backups and weekly full backups.
- **Virtual machines:** daily image backups with periodic application-consistent snapshots.
- **Critical SaaS exports:** scheduled exports or API-based backups if supported.

Retention should reflect:
- Operational recovery needs.
- Legal and regulatory requirements.
- Storage cost and lifecycle management.
- The need to preserve historical versions for rollback.

Common retention patterns:
- Daily backups kept for 7 to 30 days.
- Weekly backups kept for several months.
- Monthly or yearly archives kept for compliance.

**Backup consistency and application awareness**

A backup can be technically complete but still unusable if the application state is inconsistent. This is especially important for databases and transactional systems.

Use the right consistency model:
- **Crash-consistent** backups capture the system as if power failed at that moment.
- **Application-consistent** backups coordinate with the application to flush buffers and complete transactions.
- **Quiesced** backups temporarily pause or reduce activity for a cleaner capture.

Examples:
- For a database, use native backup tools or application-aware snapshots.
- For a virtual machine, use guest tools or VSS (Volume Shadow Copy Service) on Windows to improve consistency.
- For file shares, ensure open files and metadata are handled correctly.

**Restore testing and validation**

Backups are not proven until you restore them. Testing should be routine, documented, and representative of real recovery scenarios.

Restore validation should include:
1. Confirming the backup set is readable.
2. Restoring to an isolated test environment.
3. Verifying file integrity, permissions, and application function.
4. Measuring actual restore time against the RTO.
5. Recording failures and correcting the backup process.

Useful validation examples:
```bash
# Verify a restic backup can be listed
restic snapshots

# Restore a specific path
restic restore latest --target /tmp/restore-test

# Check archive contents
tar -tzf /backup/home-2026-04-26.tar.gz | head
```

> A backup that has never been restored is only an assumption, not a control.

**Log review and backup strategy together**

These two controls reinforce each other. Logs help you detect the need for recovery, and backups help you recover from the event you detected.

Examples of combined use:
- Review authentication logs to identify unauthorized access, then restore affected systems from a known-good backup.
- Use backup job logs to detect failed or incomplete backups before a disaster occurs.
- Correlate SIEM alerts with backup repository access logs to spot tampering.
- Preserve logs separately from production backups so an attacker cannot destroy both evidence and recovery data.

A strong operational pattern is:
- Send logs to a centralized, protected platform.
- Back up the log repository itself.
- Store backup metadata and job history separately from the backup payload.
- Test both log retrieval and data restoration.

**Practical examples of backup strategy choices**

Small environment example:
- Nightly incremental file backups.
- Weekly full backups.
- One offsite encrypted copy.
- Monthly restore test.

Medium enterprise example:
- Centralized log collection into a SIEM.
- Tiered backup schedules by system criticality.
- Immutable backup repository for ransomware resilience.
- Quarterly disaster recovery exercises.

High-availability environment example:
- Continuous replication for short RPO.
- Scheduled backups for point-in-time recovery.
- Application-consistent snapshots for databases.
- Separate archival storage for compliance retention.

**Common pitfalls**

- Relying on snapshots as the only backup.
- Keeping backups online and writable.
- Failing to test restores.
- Ignoring time synchronization across systems.
- Retaining logs too briefly to support investigations.
- Storing logs and backups in the same failure domain.
- Backing up data without backing up configuration, keys, or metadata needed to restore services.

**Exam focus reminders**

For CISSP, choose the answer that best supports business continuity, integrity, and recoverability.

Remember:
- **Logs** support detection, investigation, and accountability.
- **Backups** support restoration and continuity.
- **Retention** must satisfy business, legal, and regulatory needs.
- **Testing** is mandatory for confidence in recovery.
- **Centralization** improves visibility, but protection of the central system is critical.

**Quick reference checklist**

- Define RPO and RTO for each critical system.
- Collect logs from endpoints, servers, network devices, and cloud services.
- Synchronize time across all systems.
- Protect logs from tampering and unauthorized deletion.
- Use a backup strategy that includes full, incremental, or differential methods as appropriate.
- Keep at least one offline or immutable copy.
- Encrypt backups and restrict access.
- Test restores regularly and document results.
- Separate log storage, backup storage, and production systems where possible.
