## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Change and configuration management overview**

Change and configuration management protects the integrity, availability, and traceability of systems by ensuring that modifications are authorized, tested, documented, and reversible. In CISSP terms, this topic sits at the intersection of governance, operations, and secure engineering: you want controlled change without blocking necessary business evolution.

A useful way to think about the topic is:

- **Change management** governs *how* changes are proposed, reviewed, approved, tested, implemented, and reviewed.
- **Configuration management** governs *what* the approved baseline is and how you track, verify, and restore it.
- **Asset management** tracks ownership, value, and lifecycle of assets; it is related, but not the same as configuration management.

The exam often tests whether you can distinguish these disciplines and identify the right control for a scenario. If the question is about authorization, testing, rollback, and release governance, think change management. If it is about baselines, configuration items, dependencies, and drift, think configuration management.

**Why it matters in security**

Uncontrolled change is a common cause of outages, security regressions, and audit findings. Even a well-intentioned patch can break authentication, weaken logging, or expose a service if it bypasses review. Strong change and configuration controls reduce operational risk by making change predictable and auditable.

Key security outcomes include:

- **Integrity** of systems and data through controlled baselines.
- **Traceability** of who changed what, when, why, and with what approval.
- **Recoverability** through rollback and backout planning.
- **Compliance** through evidence of review, testing, and post-implementation validation.

**Formal change management process flow**

CISSP expects you to know the standard change control lifecycle. The exact labels vary by organization, but the core flow is consistent:

1. **Request**
   - A change request is submitted with business justification, scope, affected systems, risk, and desired timing.
   - Requests may come from operations, development, security, audit, or the business.
   - Common artifacts include a change ticket, RFC (Request for Change), or service request.

2. **Impact analysis**
   - You assess technical, security, operational, legal, and business impact.
   - You identify affected services, dependencies, downtime, data sensitivity, and recovery requirements.
   - You estimate risk, effort, cost, and the likelihood of failure.

3. **Approval**
   - The appropriate authority reviews the request, risk, and impact.
   - Approval may come from a change manager, CAB (Change Advisory Board), system owner, security, or business owner depending on the change type.
   - Emergency changes may use expedited approval, but they still require accountability and later review.

4. **Testing**
   - You validate the change in a controlled environment before production.
   - Testing may include functional testing, security testing, regression testing, and rollback testing.
   - For software, this often includes code review, automated tests, and CI/CD (Continuous Integration/Continuous Delivery or Deployment) pipeline gates.

5. **Implementation**
   - You deploy the change according to the approved plan and maintenance window.
   - Implementation should follow documented steps, including communication and monitoring.
   - For high-risk changes, use phased rollout, canary deployment, or blue-green deployment where appropriate.

6. **Documentation**
   - You update tickets, baselines, diagrams, runbooks, CMDB (Configuration Management Database) records, and version history.
   - Documentation should reflect the final state, not just the intended state.
   - This step supports auditability and future troubleshooting.

7. **Post-implementation review**
   - You confirm the change achieved the intended outcome and did not introduce unacceptable side effects.
   - Review includes validation of service health, security controls, logs, and user impact.
   - Failed or emergency changes should be analyzed for lessons learned and process improvement.

> A change is not complete when it is deployed; it is complete when it is validated, documented, and accepted into the controlled baseline.

**Change types and governance**

Different change types require different levels of control. The exam may ask you to choose the most appropriate governance path.

- **Standard change**
  - Low risk, repeatable, pre-approved, and documented.
  - Examples: routine certificate renewal, approved workstation image update, scheduled patch cycle.
  - Often handled through a predefined workflow with minimal manual review.

- **Normal change**
  - Requires assessment and approval before implementation.
  - Examples: firewall rule changes, database schema updates, identity provider configuration changes.
  - Usually reviewed by a CAB or equivalent authority.

- **Emergency change**
  - Used to restore service or address urgent security exposure.
  - Examples: zero-day patching, disabling a compromised account, blocking active malicious traffic.
  - Requires accelerated approval and a retrospective review to ensure accountability.

Emergency changes are necessary, but they are also a governance risk. Excessive emergency change frequency can indicate weak planning, poor maintenance discipline, or unstable architecture.

**Configuration management fundamentals**

Configuration management establishes and maintains the approved state of systems. In CISSP, the classic framework includes four core functions:

- **Identification**
  - Define configuration items (CIs) and establish baselines.
  - Identify what must be controlled: servers, network devices, applications, containers, cloud resources, scripts, policies, and documentation.
  - Assign unique identifiers, versions, owners, and relationships.

- **Control**
  - Ensure changes to CIs are authorized and tracked.
  - Control includes approval workflows, access restrictions, version control, and baseline enforcement.
  - The goal is to prevent unauthorized or accidental drift.

- **Status accounting**
  - Record and report the current and historical state of CIs and changes.
  - Track version, owner, location, approval status, deployment state, and exceptions.
  - This is the evidence trail that supports audits and troubleshooting.

- **Verification and audit**
  - Compare the actual environment to the approved baseline.
  - Verify that the configuration matches policy and that changes were implemented correctly.
  - Audits may be manual, automated, or both.

These four terms are exam favorites. If a question asks about “what function records the current state and change history,” the answer is status accounting. If it asks about “confirming the environment matches the baseline,” think verification and audit.

**Configuration items, baselines, and dependency mapping**

A **configuration item (CI)** is any component that must be managed to deliver a service. CIs are not limited to hardware; they can include software, cloud resources, documentation, and security controls.

Examples of CIs:

- Physical servers, switches, firewalls, and storage arrays
- Virtual machines, containers, and Kubernetes objects
- Applications, libraries, and middleware
- IAM (Identity and Access Management) roles, policies, and groups
- Certificates, secrets, and keys
- Infrastructure-as-code templates and deployment scripts
- Network diagrams, runbooks, and recovery procedures

A **baseline** is the approved reference state of one or more CIs. Baselines are essential because they let you detect drift and restore a known-good configuration.

Dependency mapping is critical because changes rarely affect only one item. A CI may depend on:

- Upstream services such as authentication, DNS, or logging
- Downstream consumers such as APIs, batch jobs, or reporting systems
- Shared infrastructure such as load balancers, databases, or message queues
- Security controls such as certificates, firewall rules, or IAM permissions

Dependency mapping improves:

- **Impact analysis** by showing what may break if a CI changes.
- **Rollback planning** by identifying what must be restored together.
- **Recovery validation** by confirming all dependent services work after the change.

Example dependency chain:

```text
Web app
  -> API gateway
    -> Authentication service
      -> Directory service
    -> Database
    -> Central logging
```

If you patch the authentication service, you must consider token issuance, session validation, API access, and monitoring. A rollback plan that restores only the app but not the auth service may leave the environment inconsistent.

**Change control artifacts**

Good change control depends on consistent artifacts. These are common in mature environments:

- **RFC / change ticket**
  - Describes the change, justification, risk, owner, and schedule.
- **Impact analysis**
  - Documents affected systems, dependencies, and business consequences.
- **Test plan**
  - Defines how success and failure will be measured.
- **Rollback or backout plan**
  - Explains how to return to the previous state if the change fails.
- **Implementation plan**
  - Lists the exact steps, timing, and responsible parties.
- **Communication plan**
  - Identifies stakeholders, outage notices, and escalation contacts.
- **Post-implementation review**
  - Captures results, incidents, and lessons learned.

**Rollback, backout, and recovery validation**

Rollback planning is a major exam theme because it shows whether you understand operational resilience. A rollback plan is not just “revert the change”; it must be realistic, tested, and time-bounded.

Strong rollback strategy includes:

- **Versioning**
  - Keep prior versions of code, configs, schemas, and images.
  - Tag releases so you can identify the exact known-good state.
- **Backout steps**
  - Define the sequence to undo the change safely.
  - Include dependencies, order of operations, and required approvals.
- **Data considerations**
  - Some changes are not fully reversible, especially schema migrations or data transformations.
  - You may need forward-fix procedures, backups, or restore points instead of a simple revert.
- **Recovery validation**
  - Verify service health after rollback.
  - Confirm authentication, logging, monitoring, and business transactions work as expected.

Examples of rollback approaches:

- Revert to the previous container image tag.
- Restore a VM snapshot taken before the change.
- Redeploy the previous application release from artifact storage.
- Restore a database backup if a schema change is destructive.
- Reapply the previous firewall policy from version control.

> Not every change is safely reversible. Schema migrations, key rotations, and data deletions may require restore procedures rather than a simple rollback.

**Version control and release governance**

Versioning is the backbone of traceability. It lets you answer: what changed, when, by whom, and from which baseline?

Practical versioning practices:

- Use semantic or release-based versioning for software and infrastructure artifacts.
- Store configuration files in version control systems such as Git.
- Tag approved releases and link them to change tickets.
- Keep immutable build artifacts so production matches what was tested.

Example Git-based workflow:

```bash
git checkout -b change/firewall-rule-update
git commit -am "Add outbound rule for payment gateway"
git tag -a v2.4.1 -m "Approved release for CAB-1842"
git push origin change/firewall-rule-update --tags
```

Release governance often includes:

- Peer review or code review
- Security review for sensitive changes
- Approval gates in CI/CD pipelines
- Separation of duties between developer, approver, and deployer where required

**SDLC integration and CI/CD controls**

Change management should integrate with the SDLC (Software Development Life Cycle), not sit outside it. In modern environments, many changes are software-defined and deployed through pipelines.

Common controls include:

- **Code review**
  - Another person reviews changes before merge.
  - Helps catch logic errors, insecure patterns, and policy violations.
- **CI/CD gates**
  - Automated tests, linting, dependency checks, and security scans block unsafe releases.
  - Deployment may require manual approval for production.
- **Release approvals**
  - Production deployment may require sign-off from operations, security, or product owners.
- **Infrastructure as code**
  - Infrastructure changes are reviewed and versioned like application code.
- **Policy as code**
  - Security and compliance rules are enforced automatically in pipelines or cloud control planes.

Example CI/CD gate logic:

```yaml
stages:
  - test
  - security_scan
  - approve
  - deploy

test:
  script:
    - pytest -q

security_scan:
  script:
    - trivy fs .
    - checkov -d .

deploy_prod:
  when: manual
  script:
    - ./deploy.sh prod
```

This model reduces human error and improves repeatability, but it does not eliminate the need for governance. The pipeline itself becomes part of the controlled environment.

**Cloud and DevOps configuration management**

Cloud and DevOps environments make configuration management more dynamic. Resources are created and destroyed frequently, so manual tracking is not enough.

Important cloud/DevOps patterns:

- **Drift remediation**
  - Detect when the live environment differs from the approved template or baseline.
  - Reconcile drift by reapplying the desired state.
- **Immutable infrastructure**
  - Replace servers or containers instead of modifying them in place.
  - Reduces configuration drift and simplifies rollback.
- **Policy-as-code enforcement**
  - Enforce rules such as encryption, tagging, logging, and network restrictions automatically.
- **Desired state management**
  - Tools continuously compare actual state to intended state and correct deviations.

Examples:

```bash
terraform plan
terraform apply
```

```bash
kubectl diff -f deployment.yaml
kubectl apply -f deployment.yaml
```

```bash
ansible-playbook site.yml --check
```

In cloud environments, configuration management often depends on:

- Templates such as Terraform, CloudFormation, or ARM/Bicep
- Orchestration tools such as Ansible, Puppet, or Chef
- Container manifests and Helm charts
- Cloud-native policy engines and guardrails

A common exam scenario is a production server that was manually patched and now differs from the approved image. The correct response is usually to restore the approved baseline or redeploy from the controlled image, not to keep ad hoc manual changes.

**Configuration management versus asset management**

These terms are related but distinct.

| Topic | Primary focus | Typical question angle |
|---|---|---|
| Configuration management | Approved state, baselines, versions, dependencies, drift | “What is the correct baseline?” |
| Asset management | Ownership, inventory, lifecycle, value, disposal | “What assets do we have and who owns them?” |

Configuration management is about how a system is built and controlled. Asset management is about what exists and how it is tracked across its lifecycle. A laptop can be an asset; its installed OS version, patch level, and security settings are configuration data.

**Metrics and reporting**

Governance requires measurement. Metrics help you determine whether change control is effective or merely bureaucratic.

Useful change management metrics include:

- **Change success rate**
  - Percentage of changes implemented without incident or rollback.
- **Failed change rate**
  - Percentage of changes that caused outages, defects, or emergency remediation.
- **Emergency change frequency**
  - High frequency may indicate weak planning or unstable systems.
- **Mean time to restore (MTTR)**
  - How quickly service is restored after a failed change.
- **Audit findings**
  - Number and severity of findings related to unauthorized changes, missing approvals, or incomplete documentation.
- **Change lead time**
  - Time from request to implementation.
- **Rollback rate**
  - How often changes require backout.

These metrics support trend analysis and governance reporting. For example, a rising emergency change rate may justify more automation, better testing, or stricter release planning.

**Practical examples**

A firewall rule change should follow the full process:

1. Request the rule with business justification.
2. Analyze impact on exposed services, logging, and segmentation.
3. Obtain approval from the change authority and security owner.
4. Test in a lab or staging environment if possible.
5. Implement during a maintenance window.
6. Document the final rule, ticket number, and effective date.
7. Review logs and connectivity after deployment.

Example firewall change documentation:

```text
Change ID: CHG-2026-0412
System: Perimeter firewall
Request: Allow outbound HTTPS to payment-gateway.example
Risk: Medium
Test: Verified in staging with synthetic transaction
Rollback: Remove rule 1042 and restore previous policy bundle
Post-check: Confirmed payment API connectivity and SIEM logging
```

A cloud drift remediation example:

```bash
terraform plan -detailed-exitcode
```

- Exit code `0` means no drift.
- Exit code `2` means changes are needed.
- Exit code `1` indicates an error.

If drift is detected, you compare the live state to the approved code, determine whether the drift is authorized, and either reconcile it or update the baseline through change control.

A database schema change example:

- Additive changes, such as adding a nullable column, are usually easier to roll back.
- Destructive changes, such as dropping a column, may require backup restore or forward-fix logic.
- You should validate application compatibility before production deployment.

**Exam tips and common traps**

- If the question emphasizes approval, testing, and documentation, it is likely about **change management**.
- If the question emphasizes baseline, drift, and version history, it is likely about **configuration management**.
- If the question asks for the four classic functions, answer with **identification, control, status accounting, and verification/audit**.
- If the question mentions dependencies, think about **impact analysis** and **rollback planning**.
- If the question involves emergency fixes, remember that speed does not remove accountability.
- If the question is about production code deployment, look for **CI/CD gates**, code review, and release approvals.
- If the question is about inventory and ownership rather than system state, it is probably **asset management**.

**What to remember for CISSP**

The exam expects you to prioritize governance, repeatability, and risk reduction over convenience. Controlled change is not anti-agility; it is what makes agility safe at scale. The best answer usually preserves accountability, validates the result, and protects the baseline.
