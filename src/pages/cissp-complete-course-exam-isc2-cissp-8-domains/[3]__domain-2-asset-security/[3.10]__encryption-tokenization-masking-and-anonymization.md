## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Overview**

These four controls solve different problems in data protection, and CISSP questions often test whether you can choose the right one for the right risk. The key is to separate *confidentiality*, *privacy*, *operational usability*, and *regulatory scope*.

- **Encryption** protects data by transforming it with a key so only authorized parties can recover the original.
- **Tokenization** replaces sensitive values with surrogate tokens and keeps the original in a protected vault or mapping system.
- **Masking** hides part or all of a value for display or processing contexts where full exposure is unnecessary.
- **Anonymization** removes or irreversibly alters identifiers so the data is no longer attributable to a person.

A useful exam mindset is to ask:
- Is the data still needed in original form?
- Must it be reversible?
- Is the goal to protect storage, transmission, or presentation?
- Does a regulation require the data to remain personal data, or to be removed from scope?

**Encryption**

Encryption is the primary control for protecting data confidentiality at rest, in transit, and sometimes in use. It uses an algorithm and a key to convert plaintext into ciphertext. If the key is protected, the data remains usable only to authorized parties.

Common categories:
- **Symmetric encryption** uses the same key, or related keys, for encryption and decryption.
- **Asymmetric encryption** uses a public/private key pair.
- **Authenticated encryption** provides confidentiality and integrity together, which is usually preferred for modern designs.

Typical use cases:
- Full-disk encryption for laptops and servers.
- Database encryption for stored records.
- TLS for network sessions.
- File encryption for backups and exports.

Practical examples:
```bash
# Symmetric file encryption with OpenSSL using PBKDF2 for stronger password-based key derivation
openssl enc -aes-256-cbc -salt -pbkdf2 -iter 100000 -in report.txt -out report.enc

# Decrypt the file
openssl enc -d -aes-256-cbc -pbkdf2 -iter 100000 -in report.enc -out report.txt
```

```bash
# Generate a private key and self-signed certificate for TLS testing
openssl req -x509 -newkey rsa:3072 -keyout server.key -out server.crt -days 365 -nodes
```

```nginx
# Example TLS server block
server {
    listen 443 ssl;
    server_name example.com;

    ssl_certificate     /etc/ssl/certs/server.crt;
    ssl_certificate_key /etc/ssl/private/server.key;

    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_prefer_server_ciphers on;
}
```

Important design points:
- **Key management** matters as much as the algorithm.
- **Rotation** limits exposure if a key is compromised.
- **Separation of duties** reduces insider risk.
- **Strong randomness** is required for keys, IVs, and nonces.
- **Integrity** is essential; encryption alone does not guarantee tamper detection.

> Use authenticated encryption modes where possible. If you use a mode that does not provide integrity, pair it with a secure message authentication mechanism.

**Tokenization**

Tokenization replaces a sensitive value with a non-sensitive surrogate token. The original value is stored separately, often in a token vault, and the token is mapped back only by authorized systems.

This is common when the business process needs the original value only in a limited trust zone. For example, a payment processor may tokenize a card number so downstream systems can reference the token without handling the actual PAN (Primary Account Number).

Why organizations use it:
- Reduces the number of systems that store real sensitive data.
- Limits the blast radius of a breach.
- Helps with PCI DSS scope reduction when implemented correctly.
- Supports workflows that need a stable reference without exposing the original value.

Practical example:
- A checkout service receives `4111 1111 1111 1111`.
- The tokenization service returns `tok_8f3a91c2`.
- Internal applications store and transmit the token.
- Only the vault or payment service can detokenize when needed.

Implementation patterns:
- **Vault-based tokenization** stores the mapping centrally.
- **Vaultless tokenization** derives tokens algorithmically, but still preserves a controlled mapping or reversible process.
- **Format-preserving tokenization** keeps the token similar in structure to the original, which can help legacy systems.

Operational considerations:
- Tokenization is not a substitute for access control.
- The token vault becomes a high-value target and must be hardened.
- Tokens may still be sensitive if they can be used to retrieve the original value.
- Tokenization is often paired with encryption for the vault and transport.

**Masking**

Masking hides part of a value so users can work with it without seeing the full sensitive content. It is often used in user interfaces, logs, reports, and support tools.

Examples:
- `4111 11XX XXXX 1111`
- `j***.d***@example.com`
- `XXX-XX-1234`

Types of masking:
- **Static masking** creates a masked copy of data, often for test or non-production use.
- **Dynamic masking** changes what a user sees at query time based on role or policy.
- **Partial masking** reveals only selected characters.
- **Full masking** replaces the entire value with a placeholder.

Where masking fits:
- Customer service screens that need limited visibility.
- Analytics dashboards that do not require full identifiers.
- Test environments that should not contain real production values.

Limitations:
- Masking is a presentation or access-control aid, not a strong security boundary by itself.
- If the underlying data remains accessible through another path, masking does not eliminate exposure.
- Masked data can still be sensitive if enough context remains to re-identify a person.

Example database feature:
```sql
SELECT
    customer_id,
    CONCAT(LEFT(card_number, 4), ' XXXX XXXX ', RIGHT(card_number, 4)) AS masked_card
FROM payments;
```

Example application logic:
```python
def mask_email(email: str) -> str:
    local, domain = email.split("@", 1)
    if len(local) <= 2:
        masked_local = "*" * len(local)
    else:
        masked_local = local[0] + "*" * (len(local) - 2) + local[-1]
    return f"{masked_local}@{domain}"
```

**Anonymization**

Anonymization aims to remove personal identifiers so the data can no longer be linked to an individual. In privacy and legal contexts, truly anonymized data is intended to fall outside personal-data treatment because re-identification is not reasonably possible.

This is different from pseudonymization. If a dataset can be re-linked to a person through a key, lookup table, or controlled process, it is pseudonymized, not anonymized.

Common anonymization techniques:
- Removing direct identifiers such as name, email, phone number, and account number.
- Generalizing values, such as replacing exact age with age bands.
- Suppressing rare attributes that could enable re-identification.
- Aggregating records so individuals are not distinguishable.
- Adding noise or using differential privacy techniques in advanced analytics contexts.

Important exam distinction:
- **Anonymization** is intended to be irreversible.
- **Pseudonymization** is reversible under controlled conditions and is still often treated as personal data under regulations such as GDPR.
- A pseudonymized dataset may reduce risk, but it usually does not remove legal obligations.

Practical example:
- Original: `Jane Doe, 14 Maple St, age 43, diagnosis X`
- Pseudonymized: `Patient 10482, 14 Maple St, age 43, diagnosis X`
- Anonymized: `Age 40-49, diagnosis X, region Northeast`

**Pseudonymization vs anonymization**

This distinction is frequently tested because it affects privacy scope, breach impact, and compliance obligations.

| Control | Reversible | Personal data status | Typical use |
|---|---:|---|---|
| Pseudonymization | Yes, under controlled conditions | Usually still personal data | Analytics, research, controlled processing |
| Anonymization | No, intended to be irreversible | Usually not personal data if truly irreversible | Public datasets, statistical reporting |

Key implications:
- Pseudonymization lowers exposure but does not eliminate identity risk.
- Anonymization requires careful design to prevent re-identification through linkage attacks.
- Small datasets, rare attributes, and external data sources can defeat weak anonymization.

**Hashing, salting, and peppering**

Hashing is related to data protection, but it serves a different purpose than encryption. A hash is a one-way transformation used for integrity, fingerprinting, and password storage.

Use hashing when:
- You need to verify data integrity.
- You need to store passwords without the ability to recover them.
- You need to compare values without storing the original.

Use encryption when:
- You need to recover the original data later.

Password storage best practice:
- Use a slow password hash such as `bcrypt`, `scrypt`, `Argon2`, or PBKDF2.
- Add a unique **salt** to each password to defeat rainbow tables and precomputed attacks.
- Consider a server-side **pepper** stored separately from the database to add another secret layer.

Example:
```text
password_hash = Argon2id(password + salt + pepper)
```

Practical guidance:
- Do not use fast general-purpose hashes like SHA-256 alone for password storage.
- Salts are public and unique per password.
- Peppers are secret and should be protected like a key.

**Data classification and regulatory drivers**

The right control depends on the data type, business need, and legal environment. CISSP questions often expect you to choose based on classification and compliance requirements, not just technical strength.

Common drivers:
- **PCI DSS** for payment card data
- **HIPAA** for protected health information
- **GDPR** for personal data and privacy processing
- **Industry or internal policy** for confidential business data

Typical mapping:
- **Encryption** for data at rest, in transit, and backups when the data must remain usable.
- **Tokenization** for payment data and systems where reducing scope is valuable.
- **Masking** for operational views, support tools, and non-production use.
- **Anonymization** for analytics, research, and data sharing where identity is not needed.

Decision cues:
- If the data must be restored exactly, use encryption or tokenization.
- If the user only needs partial visibility, use masking.
- If the data should no longer identify a person, use anonymization.
- If you only need to verify a secret or password, use hashing.

**Key management and envelope encryption**

In real systems, encryption is usually implemented with a hierarchy of keys. This is where envelope encryption becomes important.

How it works:
- A data encryption key (DEK) encrypts the data.
- A key encryption key (KEK) encrypts the DEK.
- The KEK is often stored in a key management service (KMS) or hardware security module (HSM).

Benefits:
- Limits the amount of data exposed if a single key is compromised.
- Simplifies rotation because you can rewrap DEKs without re-encrypting all data.
- Supports centralized policy, auditing, and access control.

Example workflow:
1. Generate a random DEK.
2. Encrypt the file or database field with the DEK.
3. Encrypt the DEK with the KEK.
4. Store the encrypted data and encrypted DEK together.
5. Decrypt the DEK only when needed.

Practical cloud pattern:
- Application requests a data key from KMS.
- KMS returns a plaintext DEK and an encrypted copy.
- Application uses the plaintext DEK briefly in memory.
- Application stores only the encrypted DEK with the ciphertext.

This pattern is common in cloud storage, database encryption, and secrets management.

**Deployment examples**

Database field-level encryption:
```sql
-- Conceptual example: store encrypted SSN in a separate column
CREATE TABLE employees (
    employee_id INT PRIMARY KEY,
    ssn_ciphertext VARBINARY(512) NOT NULL,
    ssn_key_id VARCHAR(64) NOT NULL
);
```

Application-side encryption flow:
```python
from cryptography.fernet import Fernet

key = Fernet.generate_key()
cipher = Fernet(key)

ciphertext = cipher.encrypt(b"Sensitive record")
plaintext = cipher.decrypt(ciphertext)
```

TLS configuration checklist:
- Use TLS 1.2 or TLS 1.3.
- Disable weak protocols and ciphers.
- Use certificates from a trusted CA for production.
- Protect private keys with file permissions and hardware-backed storage where possible.
- Enable certificate validation on clients.

Example client verification:
```bash
curl --tlsv1.2 --cacert ca.pem https://example.com/api/health
```

**How to choose the right control**

Use this decision logic:
1. If the data must remain recoverable, choose **encryption**.
2. If only a substitute identifier is needed, choose **tokenization**.
3. If the user should see only part of the value, choose **masking**.
4. If the data should no longer identify a person, choose **anonymization**.
5. If you only need to protect stored secrets for verification, choose **hashing** with salt and, where appropriate, pepper.

A practical example:
- A payment gateway stores card numbers as tokens.
- The vault encrypts the original PAN.
- Support staff see masked card numbers.
- Analytics teams receive anonymized transaction data.
- Passwords are stored as salted hashes, not encrypted values.

**Common exam traps**

- Confusing tokenization with encryption: tokens are surrogates, not ciphertext.
- Treating masking as a full security control: it reduces exposure but does not replace access control.
- Assuming pseudonymized data is anonymous: it often remains personal data.
- Using encryption where hashing is required: passwords should generally be hashed, not encrypted.
- Ignoring key management: weak key handling undermines strong algorithms.
- Forgetting regulatory context: PCI DSS, HIPAA, and GDPR can drive the control choice.

**Quick comparison**

| Control | Reversible | Primary goal | Best fit |
|---|---:|---|---|
| Encryption | Yes | Confidentiality | Data at rest/in transit that must be recovered |
| Tokenization | Yes, via vault or mapping | Reduce exposure and scope | Payment data, controlled workflows |
| Masking | Usually no meaningful recovery from the view | Limit visibility | UI, logs, support, test data |
| Anonymization | No, intended to be irreversible | Remove identity | Analytics, sharing, research |
| Hashing | No | Verification/integrity | Password storage, fingerprints |

**Practical study takeaway**

For CISSP, focus on the business objective first, then map it to the control:
- Protect recoverable data with encryption.
- Reduce system scope with tokenization.
- Limit what users can see with masking.
- Remove identity with anonymization.
- Store secrets for verification with salted hashing.
