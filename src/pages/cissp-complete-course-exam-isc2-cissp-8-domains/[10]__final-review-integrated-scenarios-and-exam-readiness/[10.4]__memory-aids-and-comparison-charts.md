## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Why these memory aids matter**

CISSP questions often test distinctions between similar terms, not isolated definitions. The fastest way to improve accuracy is to group concepts by purpose, lifecycle, trust boundary, and control type. These notes focus on the comparisons that commonly appear in exam scenarios and on mnemonics that help you recall the right answer under time pressure.

A useful study pattern is to ask:
- What problem is being solved?
- Is the control preventive, detective, corrective, or compensating?
- Does the item protect confidentiality, integrity, availability, or all three?
- Is the question about policy, process, technology, or people?

**Core mnemonic families**

Many CISSP mnemonics are only useful if you know what they map to. Use them as recall triggers, not as substitutes for understanding.

- **CIA triad**: Confidentiality, Integrity, Availability.
- **AAA**: Authentication, Authorization, Accounting.
- **PDR**: Prevent, Detect, Respond.
- **DAD**: Disclosure, Alteration, Denial.
- **RACI**: Responsible, Accountable, Consulted, Informed.
- **PDCA**: Plan, Do, Check, Act.
- **SDLC**: Software Development Life Cycle.
- **IR**: Incident Response.
- **BCP/DRP**: Business Continuity Plan / Disaster Recovery Plan.

A practical way to remember control intent:
- **Preventive** controls stop an event.
- **Detective** controls reveal an event.
- **Corrective** controls fix or reduce impact after an event.
- **Deterrent** controls discourage an event.
- **Compensating** controls provide an alternate safeguard when the primary control is not feasible.

**Security governance and lifecycle mnemonics**

Governance questions often hide process order. If you remember the lifecycle, you can eliminate distractors quickly.

- **Risk management flow**:
  - Identify assets and threats.
  - Assess likelihood and impact.
  - Treat risk by avoid, mitigate, transfer, or accept.
  - Monitor and review continuously.
- **Incident response flow**:
  - Preparation.
  - Detection and analysis.
  - Containment.
  - Eradication.
  - Recovery.
  - Lessons learned.
- **SDLC security flow**:
  - Requirements.
  - Design.
  - Implementation.
  - Testing.
  - Deployment.
  - Maintenance.

A simple memory aid:
- **Prepare, Detect, Contain, Eradicate, Recover, Learn** for incident response.
- **Plan, Build, Test, Release, Maintain** for development thinking.
- **Identify, Protect, Detect, Respond, Recover** for the NIST Cybersecurity Framework style of thinking.

**Access control comparison chart**

Access control is one of the most tested comparison areas because many models sound similar but differ in who makes the decision and what data drives it.

| Model | Decision basis | Typical use | Exam clue |
|---|---|---|---|
| **DAC** | Owner decides | File sharing, discretionary environments | “Owner grants access” |
| **MAC** | Central policy and labels | Military, high-assurance systems | “Labels, clearance, classification” |
| **RBAC** | Job role | Enterprises, least privilege by function | “Role changes access” |
| **ABAC** | Attributes and policy rules | Dynamic, context-aware access | “User, resource, action, environment” |
| **Rule-based access control** | If-then rules | Network devices, firewalls, conditional access | “Time of day” or “source IP” rules |
| **Risk-based access control** | Calculated risk score | Adaptive authentication, zero trust style decisions | “Step-up auth when risk is high” |

Key distinctions:
- **ACLs** are lists attached to objects that specify who can do what.
- **Capabilities** are tokens or references held by subjects that grant access to objects.
- **Permissions** are the actual allowed actions, such as read, write, or execute.
- **Privileges** are elevated rights, often administrative, and should be tightly controlled.

A quick mental model:
- **ACL** answers: “Who can access this object?”
- **Capability** answers: “What can this subject access?”
- **Permission** answers: “What action is allowed?”
- **Privilege** answers: “What powerful right does the subject have?”

**Access control implementation examples**

ABAC is often easier to remember when you see a policy example. The decision uses multiple attributes, not just a role.

```json
{
  "subject": {
    "department": "finance",
    "clearance": "secret",
    "mfa": true
  },
  "resource": {
    "type": "payroll-record",
    "classification": "secret"
  },
  "action": "read",
  "environment": {
    "device_compliant": true,
    "location": "corporate-network",
    "time": "business-hours"
  },
  "decision": "allow"
}
```

A rule-based example on a firewall or conditional access system might look like this:

```bash
# Allow SSH only from admin subnet during business hours
iptables -A INPUT -p tcp --dport 22 -s 10.10.10.0/24 -m time --timestart 08:00 --timestop 18:00 -j ACCEPT
iptables -A INPUT -p tcp --dport 22 -j DROP
```

Use these examples to remember that:
- **RBAC** is simpler to administer at scale.
- **ABAC** is more flexible for dynamic environments.
- **Rule-based** decisions are explicit and deterministic.
- **Risk-based** decisions adapt to context and threat signals.

**Identity, authentication, and federation**

Identity questions often compare how users prove who they are and how that identity is reused across systems.

- **Authentication** proves identity.
- **Authorization** grants access after identity is known.
- **Accounting** records actions for audit and traceability.

Common identity patterns:
- **Single sign-on (SSO)** lets a user authenticate once and access multiple services.
- **Federation** extends identity across organizational boundaries using trust relationships.
- **Identity provider (IdP)** authenticates the user and issues assertions or tokens.
- **Service provider (SP)** relies on the IdP’s assertion to grant access.

Comparison points:
- **SSO** is a user experience and session model.
- **Federation** is a trust arrangement between domains.
- **IdP** is the authority that vouches for identity.
- **SP** consumes the identity assertion.

A common exam trap is confusing SSO with federation:
- SSO can exist inside one organization.
- Federation usually crosses organizational or domain boundaries.
- An IdP may support both SSO and federation.

Typical federation flow:
1. You access the service provider.
2. The service redirects you to the identity provider.
3. You authenticate at the IdP.
4. The IdP issues a signed assertion or token.
5. The service provider validates the token and grants access.

**Authentication factor comparisons**

Remember the factor categories:
- **Something you know**: password, PIN, passphrase.
- **Something you have**: smart card, hardware token, phone-based authenticator.
- **Something you are**: fingerprint, iris, facial geometry.
- **Something you do**: keystroke dynamics, gait.
- **Somewhere you are**: geolocation, network location.
- **Something you exhibit**: behavioral biometrics, often grouped with “something you do.”

Strength increases when you combine factors:
- **MFA** means multi-factor authentication.
- **2FA** means two different factor types.
- A password plus another password is not MFA.

**Cryptography comparison chart**

Cryptography questions often ask you to choose the right algorithm family for speed, interoperability, key exchange, or integrity.

| Category | Examples | Strengths | Common exam distinction |
|---|---|---|---|
| **Symmetric encryption** | AES, 3DES, Blowfish, ChaCha20 | Fast, good for bulk data | Same key encrypts and decrypts |
| **Asymmetric encryption** | RSA, ECC | Key exchange, signatures, identity | Slower than symmetric |
| **Hashing** | SHA-2, SHA-3, MD5, SHA-1 | Integrity, password storage with salt | One-way, no decryption |
| **Message authentication** | HMAC, CMAC | Integrity plus authenticity | Uses a secret key |
| **Digital signatures** | RSA, ECDSA, EdDSA | Integrity, nonrepudiation | Signed with private key, verified with public key |

High-yield comparisons:
- **AES** is the modern symmetric standard and is generally preferred over 3DES.
- **3DES** is legacy and slower; it is largely deprecated in modern designs.
- **Blowfish** is older and can be acceptable in some contexts, but AES is usually the better exam answer for new systems.
- **ChaCha20** is a modern stream cipher often used where software efficiency matters, especially on devices without AES acceleration.

Hash algorithm distinctions:
- **SHA-2** remains widely used and includes SHA-256 and SHA-512.
- **SHA-3** is a different design family and is not a drop-in replacement in every protocol.
- **MD5** and **SHA-1** are deprecated for collision resistance and should not be used for security-sensitive integrity or signatures.

> If the question asks for password storage, do not choose a plain hash alone. Prefer a slow password hashing scheme such as bcrypt, scrypt, Argon2, or PBKDF2 with salt and work factor.

**Key exchange and public key transport**

Key exchange is a frequent comparison topic because the exam tests whether you understand how shared secrets are established.

| Method | How it works | Best use | Exam clue |
|---|---|---|---|
| **Diffie-Hellman (DH)** | Two parties derive a shared secret over an insecure channel | Key agreement | No authentication by itself |
| **Elliptic Curve Diffie-Hellman (ECDH)** | DH using elliptic curves | Efficient key agreement | Smaller keys, better performance |
| **RSA key transport** | One party encrypts a session key with the recipient’s public key | Legacy key exchange | Not true key agreement |

Important distinctions:
- **DH** and **ECDH** are key agreement methods.
- **RSA key transport** sends a secret to the recipient using public key encryption.
- **Perfect forward secrecy** is commonly associated with ephemeral DH variants, not static RSA key transport.

**PKI and certificate lifecycle**

Public Key Infrastructure (PKI) questions often focus on trust, issuance, and revocation.

- **Certificate Authority (CA)** issues certificates.
- **Registration Authority (RA)** validates identity before issuance.
- **Certificate Revocation List (CRL)** is a published list of revoked certificates.
- **Online Certificate Status Protocol (OCSP)** checks certificate status in near real time.
- **OCSP stapling** lets the server present a signed status response to reduce client lookups.

Revocation workflow example:
1. A certificate is compromised or no longer valid.
2. The CA marks it revoked.
3. The revocation status is published through CRL or OCSP.
4. Clients check status before trusting the certificate.

Practical OpenSSL checks:

```bash
openssl x509 -in server.crt -text -noout
openssl verify -CAfile ca-chain.pem server.crt
openssl ocsp -issuer ca.crt -cert server.crt -url http://ocsp.example.com
```

Remember:
- **CRL** is batch-oriented and can be stale.
- **OCSP** is more current but depends on responder availability.
- **Stapling** improves performance and privacy by reducing client queries.

**Network security device comparisons**

Network controls are often tested by function, not by product name.

| Control | Primary purpose | Exam distinction |
|---|---|---|
| **Firewall** | Filter traffic by policy | Stateful or stateless packet control |
| **IDS** | Detect suspicious activity | Alerts only |
| **IPS** | Detect and block suspicious activity | Inline prevention |
| **NAC** | Enforce endpoint access posture | Checks device compliance before access |
| **DLP** | Prevent sensitive data leakage | Content-aware inspection |
| **UTM** | Bundle multiple security functions | Firewall, IPS, AV, web filtering in one platform |
| **Honeypot** | Decoy system to attract attackers | Single decoy target |
| **Honeynet** | Network of decoys | More realistic attacker observation |

Useful distinctions:
- **IDS** is passive; **IPS** is active.
- **NAC** controls who and what can connect.
- **DLP** focuses on data leaving the organization.
- **UTM** is a convenience and consolidation concept, not a single security function.
- **Honeypots** and **honeynets** are deception tools used for detection and research.

A simple topology example:

```text
Internet -> Firewall -> IPS -> Internal Network
                         |
                         +-> Honeypot VLAN
```

**Logging, monitoring, and analytics**

Logging questions often ask where logs come from and how they are used.

Common log sources:
- **Operating system logs** record authentication, process, and system events.
- **Application logs** record app-specific actions and errors.
- **Security logs** record access attempts, policy events, and audit trails.
- **Network device logs** record routing, firewall, VPN, and IDS/IPS events.
- **Cloud logs** record control plane and workload activity.

Comparison of monitoring platforms:
- **SIEM** collects, normalizes, correlates, and alerts on log data.
- **SOAR** automates response workflows and playbooks.
- **UEBA** uses behavioral baselines to detect anomalies in users and entities.

Relationship summary:
- **SIEM** tells you what happened.
- **UEBA** helps identify unusual behavior.
- **SOAR** helps you respond faster and more consistently.

Example log pipeline:

```text
Endpoints / Servers / Firewalls / Cloud APIs
                -> Log Collector
                -> SIEM
                -> Alerting / Dashboards
                -> SOAR Playbooks
```

Practical command examples:
```bash
journalctl -u ssh --since "1 hour ago"
tail -f /var/log/auth.log
grep -i "failed password" /var/log/secure
```

**Backup, archive, snapshot, and replication**

These terms are often confused, but they serve different goals.

| Term | Purpose | Retention | Key difference |
|---|---|---|---|
| **Backup** | Recover data after loss | Usually time-based retention | Point-in-time copy for restore |
| **Archive** | Long-term retention and compliance | Long retention, often immutable | Kept for legal or historical reasons |
| **Snapshot** | Fast point-in-time state capture | Short-term, storage-system dependent | Often not a full independent backup |
| **Replication** | Keep another copy in sync | Near real time or scheduled | Supports availability, not just recovery |

Exam traps:
- A **snapshot** is not always a backup because it may depend on the same storage system.
- **Replication** improves availability but can replicate corruption or ransomware quickly.
- **Archive** is for retention, not rapid restore.
- **Backup** is for recovery from deletion, corruption, or disaster.

A practical backup example:

```bash
# Create a compressed backup of a directory
tar -czf /backups/home-$(date +%F).tar.gz /home

# Verify archive contents
tar -tzf /backups/home-$(date +%F).tar.gz | head
```

**Availability and resilience comparisons**

Availability questions often compare redundancy patterns and recovery objectives.

- **RTO**: Recovery Time Objective, how long you can be down.
- **RPO**: Recovery Point Objective, how much data loss you can tolerate.
- **MTBF**: Mean Time Between Failures, reliability measure.
- **MTTR**: Mean Time To Repair or Recover, restoration speed.

Common resilience patterns:
- **Redundancy** adds alternate components.
- **Failover** switches to a standby system.
- **Load balancing** distributes traffic across multiple systems.
- **Clustering** groups systems to act as one service.
- **Hot site** is ready quickly.
- **Warm site** needs some setup.
- **Cold site** is least ready and cheapest.

Quick memory aid:
- **Hot** = fastest recovery, highest cost.
- **Warm** = middle ground.
- **Cold** = slowest recovery, lowest cost.

**Physical security and environmental controls**

Physical controls are often easier to remember when grouped by what they stop.

- **Mantrap** prevents tailgating by allowing one person or vehicle at a time.
- **Badge reader** controls entry based on credentials.
- **CCTV** provides surveillance and evidence.
- **Motion sensors** detect movement.
- **HVAC** maintains temperature and humidity.
- **Fire suppression** protects assets from fire damage.

A useful distinction:
- **Deterrence** discourages intrusion.
- **Detection** reveals intrusion.
- **Delay** slows intrusion.
- **Response** contains or stops intrusion.

**Data states and protection methods**

Data protection questions often compare where data exists and which control applies.

- **Data at rest**: stored on disk, database, backup media.
- **Data in transit**: moving across a network.
- **Data in use**: actively processed in memory or CPU.

Typical controls:
- **At rest**: full-disk encryption, database encryption, tokenization.
- **In transit**: TLS, IPsec, VPNs.
- **In use**: memory protection, secure enclaves, least privilege, application hardening.

Remember:
- Encryption alone does not solve access control.
- Tokenization replaces sensitive values with non-sensitive substitutes.
- Masking hides data for display or testing, but may not protect the underlying record.

**Threat and attack mnemonics**

Attack questions often use similar wording for different outcomes.

- **Spoofing**: pretending to be something else.
- **Tampering**: unauthorized modification.
- **Repudiation**: denying an action without proof.
- **Information disclosure**: exposing data.
- **Denial of service**: making a service unavailable.
- **Elevation of privilege**: gaining higher rights.

This is the **STRIDE** model:
- Spoofing
- Tampering
- Repudiation
- Information disclosure
- Denial of service
- Elevation of privilege

A useful pairing:
- **Phishing** targets people.
- **Spoofing** targets trust.
- **Tampering** targets integrity.
- **DoS** targets availability.

**Malware and persistence comparisons**

Malware terms are often tested by behavior.

- **Virus** attaches to a host file and spreads when executed.
- **Worm** self-replicates across networks.
- **Trojan** disguises itself as legitimate software.
- **Ransomware** encrypts or blocks access for extortion.
- **Rootkit** hides malicious presence and preserves access.
- **Spyware** collects information covertly.
- **Botnet** is a group of compromised systems under control.

Memory aid:
- **Virus** needs a host.
- **Worm** travels on its own.
- **Trojan** tricks the user.
- **Rootkit** hides the attacker.

**Cloud and shared responsibility comparisons**

Cloud questions often test who is responsible for what.

- **IaaS**: Infrastructure as a Service.
- **PaaS**: Platform as a Service.
- **SaaS**: Software as a Service.

Responsibility trend:
- As you move from **IaaS** to **SaaS**, the provider manages more.
- You still own data classification, identity governance, and access decisions.

Common exam clue:
- If the question mentions patching the guest OS, that is usually your responsibility in **IaaS**.
- If the question mentions application configuration in **SaaS**, you usually control settings, not the underlying platform.

**Data classification and handling**

Classification questions are about business impact and handling requirements.

- **Public**: no harm if disclosed.
- **Internal**: limited business use.
- **Confidential**: sensitive business data.
- **Restricted/Secret/Highly confidential**: highest protection, depending on policy.

Handling principles:
- Label data clearly.
- Apply least privilege.
- Encrypt sensitive data where appropriate.
- Dispose of media securely.
- Reassess classification when context changes.

**Exam-oriented compare and contrast reminders**

Use these short contrasts when you are eliminating answer choices.

- **Policy** tells you what must happen.
- **Standard** tells you the mandatory specifics.
- **Procedure** tells you how to do it.
- **Guideline** suggests a recommended approach.
- **Baseline** is the minimum acceptable configuration.

- **Authentication** is identity proof.
- **Authorization** is access decision.
- **Accounting** is logging and traceability.

- **Hashing** is one-way.
- **Encryption** is reversible with a key.
- **Encoding** is not security.

- **Symmetric** is faster for bulk data.
- **Asymmetric** is better for key exchange and signatures.

- **Backup** restores data.
- **Replication** improves availability.
- **Archive** preserves records.
- **Snapshot** captures state quickly.

- **SIEM** correlates logs.
- **SOAR** automates response.
- **UEBA** detects anomalies.

**Practical study workflow**

1. Read the question and identify the domain.
2. Mark whether it asks for prevention, detection, recovery, or governance.
3. Compare the answer choices by purpose, not by buzzword.
4. Eliminate options that solve a different problem.
5. Choose the control that best matches the scenario and the least privilege principle.

A compact review habit:
- Build flashcards for pairs like **AES vs 3DES**, **DH vs ECDH vs RSA**, **ACL vs capability**, **SSO vs federation**, **backup vs snapshot vs replication**, and **SIEM vs SOAR vs UEBA**.
- Rehearse process mnemonics for incident response, risk management, and SDLC.
- Practice with scenario questions so the comparison becomes automatic.

**High-yield one-line memory anchors**

- **AES** for modern symmetric encryption.
- **SHA-2/SHA-3** for secure hashing; avoid **MD5/SHA-1** for trust.
- **DH/ECDH** for key agreement; **RSA** for legacy key transport.
- **RBAC** for job-based access; **ABAC** for attribute-driven access.
- **SSO** reduces logins; **federation** extends trust across domains.
- **NAC** checks device posture; **DLP** protects data leaving the organization.
- **SIEM** correlates; **SOAR** automates; **UEBA** spots anomalies.
- **Backup** restores; **archive** retains; **snapshot** captures; **replication** mirrors.
