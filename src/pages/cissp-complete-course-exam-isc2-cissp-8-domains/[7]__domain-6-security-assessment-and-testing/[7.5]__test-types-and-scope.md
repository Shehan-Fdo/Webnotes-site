## **Course:** CISSP Complete Course (Exam ISC2 CISSP – 8 Domains)

**Test types and scope overview**

Security testing is not one activity; it is a set of methods used at different points in the lifecycle to find weaknesses, validate controls, and measure risk. In CISSP exam language, you must distinguish between **vulnerability assessment**, **penetration testing**, and broader **security testing** because they answer different questions.

- **Vulnerability assessment** identifies and prioritizes weaknesses.
- **Penetration testing** attempts to exploit weaknesses to prove impact.
- **Security testing** is the umbrella term that can include both, plus code review, fuzzing, and control validation.

A useful mental model is:
- *What is weak?* → vulnerability assessment
- *What can be exploited?* → penetration testing
- *What is the overall control posture?* → security testing

**Core distinctions for CISSP**

A **vulnerability assessment** is usually broader and less intrusive. It uses scanners, configuration checks, and manual review to identify known issues, then ranks them by severity and exposure. It does not normally attempt to exploit the weakness.

A **penetration test** is goal-driven and adversarial. It validates whether an attacker can chain weaknesses into a real compromise. It is narrower in scope than a vulnerability assessment but deeper in execution.

A **security test** can refer to any test that evaluates security properties, including:
- Authentication and authorization checks
- Input validation testing
- Secure configuration validation
- Application logic testing
- Code-level review
- Runtime behavior analysis

For exam questions, read the wording carefully:
- If the goal is to discover as many weaknesses as possible, think vulnerability assessment.
- If the goal is to demonstrate exploitability or business impact, think penetration testing.
- If the question asks about verifying a specific control or security property, think security testing more generally.

**Common security test types**

Security testing spans multiple techniques, each useful at a different stage of development or operations.

- **Static analysis** examines source code, bytecode, or binaries without executing them.
  - Use it early in development to catch insecure patterns before deployment.
  - It is also called SAST, or static application security testing.
- **Dynamic analysis** evaluates a running application or system.
  - Use it in test, staging, or pre-production environments to observe runtime behavior.
  - It is also called DAST, or dynamic application security testing.
- **Code review** is a manual or assisted inspection of code for security defects.
  - Use it for logic flaws, insecure assumptions, and risky design choices.
  - It is especially valuable when automated tools miss context-sensitive issues.
- **Fuzz testing** sends malformed, unexpected, or random inputs to discover crashes, hangs, memory corruption, or parsing flaws.
  - Use it for parsers, file handlers, network services, APIs, and protocol implementations.
  - It is especially useful when input handling is complex or attack surface is large.
- **Regression testing** confirms that a fix did not break existing security or functionality.
  - Use it after remediation, patching, or configuration changes.
- **Configuration testing** checks whether systems match secure baselines.
  - Use it for operating systems, cloud services, network devices, and containers.
- **Physical security testing** validates access controls, surveillance, alarms, and human procedures.
  - Use it carefully to avoid harm to people, operations, or critical infrastructure.

**Lifecycle-based selection**

Choose the test type based on where the system is in its lifecycle.

- **Development**
  - Use static analysis, code review, and unit-level security tests.
  - Catch defects early when fixes are cheapest.
  - Integrate checks into the CI/CD pipeline where possible.
- **Pre-production / staging**
  - Use dynamic analysis, fuzzing, configuration validation, and controlled penetration testing.
  - Verify how the system behaves in an environment close to production.
- **Production**
  - Use low-impact validation, monitoring, and carefully scoped testing.
  - Avoid disruptive tests unless explicitly authorized and operationally safe.
- **Post-incident**
  - Use forensic review, targeted validation, and regression testing.
  - Confirm whether the root cause is fixed and whether similar weaknesses remain.

A common exam trap is choosing a highly disruptive test in production when a safer method would satisfy the objective. If the question emphasizes uptime, safety, or business continuity, prefer less intrusive testing first.

**Rules of engagement**

Rules of engagement define how testing is authorized and controlled. They are essential for both legal protection and operational safety.

A strong rules-of-engagement document should include:
- **Scope boundaries**
  - In-scope systems, applications, accounts, facilities, and time windows
  - Explicit exclusions such as third-party systems or safety-critical assets
- **Authorization**
  - Written approval from the appropriate owner
  - Named testers and points of contact
- **Test objectives**
  - What you are trying to prove or measure
  - Success criteria and stop conditions
- **Data handling**
  - How you collect, store, encrypt, transfer, and destroy evidence
  - Whether sensitive data may be copied, masked, or retained
- **Evidence retention**
  - Retention period for screenshots, logs, packet captures, exploit artifacts, and notes
  - Chain-of-custody requirements if evidence may support legal or disciplinary action
- **Notification requirements**
  - Who must be informed before, during, and after testing
  - Whether operations, SOC, help desk, legal, HR, or third parties need notice
- **Escalation paths**
  - Stop contacts for unexpected outages or safety issues
  - Escalation contacts for confirmed compromise or critical findings
- **Timing and maintenance windows**
  - Approved test periods and blackout dates
  - Coordination with change management
- **Allowed techniques**
  - Social engineering, phishing, wireless testing, physical entry attempts, exploit use, or denial-of-service simulation
- **Prohibited actions**
  - Actions that could damage systems, violate law, or endanger people

> If the rules of engagement are unclear, you do not improvise. You stop and obtain clarification before continuing.

**Safety considerations for physical testing**

Physical security testing can create real-world risk, so safety is part of scope management.

- Avoid actions that could injure people, trigger panic, or block emergency exits.
- Do not interfere with life safety systems, medical devices, industrial controls, or critical operations unless the test is explicitly designed and approved for that environment.
- Coordinate with facility management and security staff when testing badges, locks, tailgating, cameras, or visitor procedures.
- Use non-destructive methods first, such as observation, policy review, and controlled walkthroughs.
- If a test could cause operational disruption, define a rollback or abort procedure before starting.

For CISSP, the key idea is that authorization does not remove the duty to minimize harm. A valid test still needs safe execution.

**Practical examples of choosing the right test**

Use the scenario to select the most appropriate method.

1. A development team wants to catch insecure coding patterns before release.
   - Choose **static analysis** and **code review**.
   - Why: the code is not yet deployed, and early detection reduces cost.
2. A web app is in staging, and you want to see whether input validation fails at runtime.
   - Choose **dynamic analysis** and targeted **fuzz testing**.
   - Why: you need live behavior, not just source inspection.
3. Management asks whether a known SQL injection flaw can actually lead to data access.
   - Choose **penetration testing**.
   - Why: the goal is exploitability and impact, not just discovery.
4. Operations wants to verify that a firewall rule set matches the approved baseline.
   - Choose **configuration testing**.
   - Why: the question is control compliance.
5. After patching a parser crash, you want to confirm the fix and ensure no regressions.
   - Choose **regression testing** and possibly **fuzz testing** again.
   - Why: you need to validate the fix and re-check edge cases.
6. A company wants to test badge controls at a branch office without disrupting business.
   - Choose a carefully scoped **physical security test** during a maintenance window.
   - Why: safety and continuity matter as much as access control.

**Test depth, impact, and frequency**

The right frequency depends on risk, change rate, and environment criticality.

- **Continuous or frequent**
  - Static analysis in CI/CD
  - Dependency scanning
  - Configuration compliance checks
- **Per release**
  - Code review for security-sensitive changes
  - Dynamic analysis in staging
  - Targeted fuzzing for high-risk parsers or interfaces
- **Periodic**
  - Penetration testing for major applications or significant changes
  - Physical security assessments
  - Full vulnerability assessments
- **Event-driven**
  - After major architecture changes
  - After a significant incident
  - After patching a critical flaw
  - After mergers, acquisitions, or cloud migrations

A good exam answer often ties frequency to change. Systems that change often need more frequent validation; stable systems may need periodic reassessment, especially when risk is high.

**How the test types relate**

These terms overlap, but they are not interchangeable.

| Term | Primary goal | Typical depth | Common timing |
|---|---|---:|---|
| Vulnerability assessment | Find and rank weaknesses | Broad | Periodic or after change |
| Penetration testing | Prove exploitability and impact | Deep but targeted | Periodic, major change, or incident-driven |
| Security testing | Validate security properties and controls | Varies | Throughout lifecycle |
| Static analysis | Find issues without execution | Code-focused | Development and CI/CD |
| Dynamic analysis | Observe runtime behavior | Runtime-focused | Staging or pre-production |
| Fuzz testing | Break parsers and input handling | Input-focused | Development, staging, or pre-release |
| Code review | Identify logic and design flaws | Human/contextual | Development and release review |

**Practical CLI examples**

Security testing often uses standard tooling, but the tool is less important than the objective and authorization.

```bash
# Basic host discovery and service enumeration
nmap -sS -sV -O 192.0.2.10

# Web application directory and parameter discovery
ffuf -u https://app.example.com/FUZZ -w /usr/share/wordlists/dirb/common.txt

# TLS configuration review
testssl.sh https://app.example.com

# Simple file integrity check for baseline comparison
sha256sum /etc/ssh/sshd_config
```

For code and dependency review, you may see commands like:
```bash
# Static analysis example
semgrep --config=auto src/

# Dependency vulnerability scan example
npm audit

# Python dependency check example
pip-audit
```

These tools support security testing, but they do not replace judgment. A scanner can identify candidates; a human decides whether the finding is exploitable, relevant, and in scope.

**Exam-focused decision cues**

Look for these phrases in questions:
- **“Find as many weaknesses as possible”** → vulnerability assessment
- **“Demonstrate exploitability”** → penetration testing
- **“Verify secure coding”** → code review or static analysis
- **“Test the running application”** → dynamic analysis
- **“Send malformed inputs”** → fuzz testing
- **“Confirm the fix did not break anything”** → regression testing
- **“Check baseline compliance”** → configuration testing
- **“Avoid disruption to operations”** → choose lower-impact methods first

**Common pitfalls**

- Treating vulnerability assessment and penetration testing as the same activity.
- Choosing a destructive test when a non-destructive one answers the question.
- Ignoring lifecycle stage and testing too late.
- Forgetting that physical tests must protect people and critical operations.
- Overlooking evidence handling and retention requirements.
- Assuming authorization alone is enough without clear escalation and notification paths.

**Key takeaways**

Security testing is a broad discipline, but CISSP expects you to match the method to the objective, lifecycle stage, and risk tolerance. The best answer is usually the one that is:
- Authorized
- Scoped
- Safe
- Appropriate to the system’s stage
- Aligned with the question’s goal

If you remember only one thing, remember this: vulnerability assessment finds weaknesses, penetration testing proves impact, and security testing is the broader umbrella that includes both plus code, runtime, and input-focused techniques.
