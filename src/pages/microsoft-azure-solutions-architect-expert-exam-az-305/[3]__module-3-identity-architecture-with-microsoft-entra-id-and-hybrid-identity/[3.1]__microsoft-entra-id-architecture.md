## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Microsoft Entra ID architecture overview**

Microsoft Entra ID is Microsoft’s cloud identity and access management service for workforce identities. In architecture terms, you use it to control who can authenticate, what they can access, and under which conditions they can do so. For AZ-305, focus on how Entra ID fits into Azure landing zones, application access, hybrid identity, governance, and security controls.

A useful mental model is:

- **Tenant**: the security and policy boundary for identities and directory objects.
- **Identity objects**: users, groups, devices, service principals, managed identities, and related configuration.
- **Authentication and authorization**: how tokens are issued, validated, and constrained by policy.
- **Governance and protection**: how you manage lifecycle, access, risk, and compliance.

**Tenant architecture and boundaries**

An Entra ID tenant is a dedicated instance of the directory service. It is the primary boundary for identity data, policies, and administrative control. In Azure architecture, the tenant is separate from subscriptions and management groups, even though they are often linked operationally.

Key relationships:

- **Tenant** is the identity plane boundary.
- **Subscription** is the billing and resource deployment boundary.
- **Management group** is the governance hierarchy for subscriptions.
- A single tenant can own many subscriptions.
- A subscription belongs to exactly one tenant at a time.
- Management groups exist within a tenant and organize subscriptions for policy and access control.

Design implications:

- Use one tenant for a single enterprise identity boundary when possible.
- Use multiple tenants when you need hard separation for legal, regulatory, merger, or business autonomy reasons.
- Keep in mind that cross-tenant collaboration adds complexity for access, governance, and lifecycle management.

Tenant concepts that matter in design:

- **Home tenant**: where a user or object originates.
- **Resource tenant**: where an app, resource, or collaboration target exists.
- **Directory partitioning**: Entra ID stores different object types and policy data in the tenant directory; you do not manage it like a traditional LDAP tree.
- **Global admin scope**: administrative rights are tenant-scoped, not subscription-scoped.

> A subscription does not contain the tenant. The tenant is the identity authority that can govern one or more subscriptions.

**Identity object model and lifecycle**

Entra ID uses directory objects to represent identities and access relationships. Understanding object lifecycle is essential for hybrid and cloud-only designs.

**Users**

- Cloud-only users are created directly in Entra ID.
- Synchronized users originate in on-premises Active Directory Domain Services (AD DS) and are projected into Entra ID by synchronization tooling.
- Guest users are external identities invited into the tenant for collaboration.
- User lifecycle can be managed manually, by synchronization, or through automated provisioning.

**Groups**

- Security groups control access to resources and applications.
- Microsoft 365 groups support collaboration workloads.
- Dynamic groups evaluate rules and update membership automatically.
- Groups can be cloud-only or synchronized from on-premises AD DS, depending on the scenario.

**Devices**

- Devices can be registered, joined, or hybrid joined.
- Device objects support Conditional Access, compliance evaluation, and device-based access decisions.
- Device state matters for browser sessions, single sign-on, and endpoint trust.

**Service principals**

- An app registration defines the application identity and configuration in the home tenant.
- A **service principal** is the tenant-local instance of that application identity in a specific tenant where the app is used.
- Service principals are separate directory objects created in each tenant that uses the app.
- They hold permissions, assignment state, and tenant-specific settings.

**Managed identities**

- Managed identities are service principals managed by Azure.
- Use them to avoid storing credentials in code or configuration.
- System-assigned identities are tied to a single Azure resource.
- User-assigned identities are standalone resources that can be attached to multiple supported resources.

Lifecycle patterns:

- Cloud-only objects are created and deleted in Entra ID directly.
- Synchronized objects are mastered on-premises and flow to the cloud.
- Some attributes are source-of-authority controlled on-premises, while others are cloud-managed.
- Deletion behavior depends on the object type and sync method; synchronized objects may be soft-deleted or re-created based on source changes.

**Authentication architecture**

Entra ID authentication is based on modern token issuance and session management. The service authenticates the user or workload, then issues tokens that apps and APIs validate.

Core concepts:

- **Primary authentication** verifies the initial sign-in.
- **Multi-factor authentication (MFA)** adds a second factor or stronger method.
- **Tokens** carry claims about identity, device, app, and policy context.
- **Sessions** reduce repeated prompts while preserving security controls.

Important token and session artifacts:

- **Primary Refresh Token (PRT)**: a device-bound token used on Windows and some joined devices to enable seamless single sign-on.
- **Access token**: presented to APIs and resources.
- **ID token**: used by client apps to understand the authenticated user.
- **Refresh token**: used to obtain new access tokens without re-prompting the user.
- **Session cookie**: browser-based state that can persist sign-in context.

How sessions behave:

- Browser sessions often rely on cookies plus token refresh behind the scenes.
- Native apps use refresh tokens and brokered authentication where supported.
- Device sign-in state can enable SSO across Microsoft apps and many integrated apps.
- Conditional Access and sign-in frequency can force reauthentication even when a session exists.

Practical implications:

- Use PRT-aware device join models to improve user experience and reduce password prompts.
- Use sign-in frequency and persistent browser session controls carefully; they affect usability and security.
- Understand that token lifetime is not the only control; policy can interrupt or re-evaluate sessions.

**Modern app platform architecture**

App registration design is a common AZ-305 topic because it determines how applications authenticate and what permissions they receive.

App types and patterns:

- **Web app**: server-side app with a confidential client and redirect URI.
- **Single-page application (SPA)**: browser-based app using authorization code flow with PKCE (Proof Key for Code Exchange).
- **Mobile app**: public client using brokered or system browser sign-in.
- **Daemon/service app**: non-interactive workload using client credentials flow.
- **API**: resource that exposes scopes and app roles.

Key design elements:

- **Redirect URIs** must match the app type and platform.
- **Scopes** represent delegated permissions on behalf of a signed-in user.
- **App roles** represent application permissions, often used for daemon apps or role-based authorization.
- **Certificates and secrets** authenticate confidential clients.
- **Federated credentials** can replace secrets for some workload identity scenarios.

Permission model:

- **Delegated permissions**: app acts as the signed-in user.
- **Application permissions**: app acts as itself without a user.
- Admin consent is often required for higher-privilege permissions.
- User consent is frequently restricted or disabled in enterprise tenants.

Example app registration with Microsoft Graph permissions:

```bash
az ad app create \
  --display-name "orders-api"
```

```bash
az ad app permission add \
  --id <appId> \
  --api 00000003-0000-0000-c000-000000000000 \
  --api-permissions e1fe6dd8-ba31-4d61-89e7-88639da4683d=Scope
```

```bash
az ad app permission admin-consent --id <appId>
```

Notes:

- `az ad` commands are increasingly legacy and many identity workflows are moving toward Microsoft Graph, `az rest`, and Microsoft Graph PowerShell.
- `az ad sp create-for-rbac` creates a service principal and assigns an Azure RBAC role; it does **not** assign an Entra ID directory role.

**Authentication methods and passwordless design**

Modern identity architecture should reduce password dependence. Passwordless methods improve phishing resistance and user experience.

Common methods:

- **FIDO2 security keys**: phishing-resistant hardware-backed authentication.
- **Windows Hello for Business**: device-bound strong authentication using PIN or biometrics backed by keys.
- **Microsoft Authenticator**: supports push, number matching, and passwordless sign-in in supported scenarios.
- **Temporary Access Pass (TAP)**: time-limited bootstrap credential for onboarding and recovery.

Design guidance:

- Prefer phishing-resistant methods for privileged users.
- Use TAP for enrollment, recovery, and device bootstrap.
- Require MFA registration for users who need access to sensitive apps.
- Use authentication strengths to require specific methods rather than generic MFA.

Authentication strengths:

- Let you require a specific class of authentication method.
- Are useful when you want to enforce phishing-resistant authentication for admins or high-value apps.
- Work well with Conditional Access to separate low-risk and high-risk access paths.

**Conditional Access architecture**

Conditional Access is the policy engine that enforces access decisions based on identity, device, location, risk, and app context. It is central to Entra ID security architecture.

Core signals:

- **User or group**
- **Cloud app or action**
- **Device platform**
- **Device state or compliance**
- **Named location**
- **Sign-in risk**
- **User risk**
- **Client app type**
- **Authentication strength**

Common controls:

- Require MFA.
- Require compliant device.
- Require hybrid joined device.
- Block legacy authentication.
- Require approved client app or app protection policy.
- Require authentication strength.
- Control session behavior.

Important design features:

- **Named locations** define trusted or blocked IP ranges and countries/regions.
- **Report-only mode** lets you validate policy impact before enforcement.
- **Session controls** can limit browser persistence, sign-in frequency, or app-enforced restrictions.
- **Break-glass accounts** should be excluded from most policies and monitored closely.

Break-glass design:

- Create at least two emergency access accounts.
- Use strong, unique credentials.
- Exclude them from Conditional Access only when necessary.
- Monitor sign-ins and alert on any use.
- Store credentials securely and test recovery procedures.

Example Conditional Access design pattern:

1. Target all users except emergency access accounts.
2. Include Microsoft 365 and Azure management apps.
3. Require MFA for all cloud apps.
4. Require compliant device for privileged admin portals.
5. Block legacy authentication.
6. Start in report-only mode.
7. Review sign-in logs and policy impact.
8. Move to enforcement after validation.

Example policy logic:

```text
If user is in "All Users"
AND cloud app is "Azure portal"
AND device is not compliant
THEN block access
```

**Identity protection and risk-based access**

Identity Protection uses risk detections to identify suspicious users and sign-ins. It helps you automate response to likely compromise.

Risk concepts:

- **Sign-in risk**: likelihood that a sign-in is malicious.
- **User risk**: likelihood that the account is compromised.
- **Risk detections**: signals such as atypical travel, unfamiliar sign-in properties, leaked credentials, or impossible travel patterns.

Typical remediation actions:

- Require MFA.
- Require password change.
- Block access.
- Force reauthentication.
- Trigger investigation workflows.

Architecture guidance:

- Use risk-based policies for high-value users and apps.
- Combine risk with Conditional Access for layered enforcement.
- Do not rely on risk alone for privileged access; pair it with phishing-resistant methods.

**Hybrid identity architecture**

Hybrid identity connects on-premises AD DS with Entra ID. The design choice depends on your source of authority, app compatibility, and operational model.

Main synchronization options:

- **Microsoft Entra Connect Sync**
- **Microsoft Entra Cloud Sync**

**Entra Connect Sync**

- Traditional synchronization engine.
- Supports more advanced hybrid scenarios.
- Commonly used when you need broader feature coverage.
- Often chosen for complex environments with password hash sync, pass-through authentication, federation integration, and advanced writeback needs.

**Cloud Sync**

- Lightweight agent-based synchronization.
- Easier to deploy and operate.
- Good for simpler synchronization scenarios.
- Not a full replacement for every advanced feature in Connect Sync.

Comparison guidance:

| Capability | Entra Connect Sync | Cloud Sync |
|---|---:|---:|
| Deployment complexity | Higher | Lower |
| Feature breadth | Broader | More limited |
| Operational overhead | Higher | Lower |
| Best fit | Complex hybrid estates | Simpler sync requirements |

Hybrid authentication options:

- **Password hash synchronization (PHS)**: syncs password hashes to Entra ID for cloud authentication.
- **Pass-through authentication (PTA)**: validates passwords against on-premises AD DS through agents.
- **Federation**: uses a federation service such as AD FS when you need specific claims or legacy requirements.

Federation in this context means Entra ID federates with an external identity provider or on-premises federation service for user authentication. It is not a generic label for any external IdP scenario.

Hybrid design considerations:

- Prefer PHS for simplicity and resilience when possible.
- Use PTA when you need on-premises password validation without federation.
- Use federation only when you have a clear requirement.
- Plan for high availability of sync and authentication components.

Writeback features:

- **Password writeback**: lets users reset passwords in the cloud and write them back to on-premises AD DS.
- **Group writeback**: can project certain cloud groups back to AD DS in supported scenarios.
- **Device writeback**: primarily supports on-premises AD DS and AD FS scenarios that require device objects.

> Device writeback is not a general-purpose visibility feature for all dependent systems. Design it only when a supported on-premises scenario requires device objects.

**Microsoft Entra Domain Services**

Microsoft Entra Domain Services provides managed domain capabilities such as LDAP, Kerberos, and NTLM for legacy applications that cannot use modern authentication directly.

Use it when:

- You need domain join or LDAP for legacy apps.
- You want managed domain services without running domain controllers.
- You need compatibility with older authentication protocols.

Do not use it as a replacement for modern Entra ID authentication. It is a compatibility bridge, not the primary identity platform.

**B2B collaboration and cross-tenant access**

B2B (business-to-business) collaboration lets external users access resources in your tenant. Cross-tenant access settings control trust and collaboration behavior between tenants.

Key concepts:

- **Guest user**: external identity represented in your tenant.
- **Cross-tenant access policy**: controls inbound and outbound trust, MFA trust, device trust, and collaboration settings.
- **Tenant restrictions**: can limit where users can authenticate from managed devices or browsers.
- **Direct connect**: used in some collaboration scenarios to improve cross-tenant experiences.

Design considerations:

- Use cross-tenant access settings to define trust explicitly.
- Decide whether to trust MFA claims from partner tenants.
- Decide whether to trust compliant device claims from partner tenants.
- Restrict guest access to the minimum required resources.

Practical example:

- A partner tenant authenticates its user.
- Your tenant evaluates cross-tenant trust settings.
- If you trust the partner’s MFA claim, you may not require a second MFA prompt.
- If you do not trust the claim, your Conditional Access policy can require your own MFA.

**Governance architecture**

Governance controls who gets access, how long they keep it, and how access is reviewed.

Core services:

- **Entitlement management**
- **Access packages**
- **Access reviews**
- **Privileged Identity Management (PIM)**

Access packages:

- Bundle resources such as groups, apps, and SharePoint sites.
- Support request, approval, and expiration workflows.
- Can include assignment policies for users, guests, or connected organizations.
- Help standardize onboarding and project-based access.

Access package design elements:

- Resource roles define what access is granted to each resource.
- Approval workflows can require one or more approvers.
- Expiration settings limit access duration.
- Lifecycle policies can trigger removal or review.

Access reviews:

- Revalidate access to groups, apps, and guest users.
- Can be recurring or one-time.
- Can use self-review, manager review, or delegated reviewers.
- Help reduce privilege creep and stale guest access.

PIM:

- Manages just-in-time elevation for privileged roles and groups.
- Supports approval, activation duration, and justification.
- Reduces standing privilege.
- Works best with break-glass accounts excluded and monitored separately.

**Security architecture and privileged access**

Identity security architecture should assume compromise and minimize blast radius.

Best practices:

- Use least privilege.
- Separate admin accounts from user accounts.
- Require phishing-resistant MFA for privileged roles.
- Use PIM for role activation.
- Restrict legacy authentication.
- Monitor risky sign-ins and anomalous behavior.
- Use Conditional Access to protect admin portals and sensitive apps.

Privileged account design:

- Create dedicated admin accounts.
- Avoid daily productivity use on privileged accounts.
- Use separate devices or hardened admin workstations for sensitive administration.
- Limit global administrator assignments.

**Monitoring, logging, and troubleshooting**

Entra ID logs are essential for troubleshooting sign-in failures, provisioning issues, and policy behavior.

Main log types:

- **Sign-in logs**: authentication events, token issuance, Conditional Access results, client details.
- **Audit logs**: directory changes, policy changes, app changes, group changes.
- **Provisioning logs**: lifecycle and synchronization events for SCIM and app provisioning.

Operational guidance:

- Configure diagnostic settings to send logs to Log Analytics, Event Hub, or storage.
- Decide retention based on compliance and investigation needs.
- Keep longer retention for audit and security investigations.
- Use centralized monitoring for correlation with endpoint, network, and application telemetry.

Troubleshooting approach:

1. Check the sign-in log for the failure reason.
2. Review Conditional Access results and applied policies.
3. Validate user risk and sign-in risk status.
4. Confirm app registration, redirect URI, and consent state.
5. Check provisioning logs for SCIM or sync failures.
6. Correlate with device compliance and network location.

Common failure patterns:

- Missing admin consent for required permissions.
- Redirect URI mismatch.
- Conditional Access blocking legacy clients.
- Device not compliant or not hybrid joined.
- Guest user not allowed by cross-tenant policy.
- Sync attribute conflict or source-of-authority mismatch.

**Practical deployment examples**

**1. Secure admin access with Conditional Access**

```text
Policy: Require phishing-resistant MFA for Azure management
Assignments:
  Users: All users except emergency access accounts
  Cloud apps: Azure portal, Azure PowerShell, Azure CLI, Microsoft Entra admin center
  Conditions: Any location, any device
Controls:
  Grant access only if authentication strength = phishing-resistant MFA
Session:
  Sign-in frequency = 8 hours
  Persistent browser session = disabled
Mode:
  Report-only first, then On
```

**2. Register a web API and expose a scope**

```bash
az ad app create --display-name "inventory-api"
```

```bash
az ad app update \
  --id <appId> \
  --identifier-uris api://inventory-api
```

```bash
az ad app permission add \
  --id <clientAppId> \
  --api <apiAppId> \
  --api-permissions <scopeId>=Scope
```

Conceptually:

- The API exposes a scope such as `api://inventory-api/read`.
- A client app requests that scope.
- The user or admin consents to the permission.
- The access token contains the granted scope claim.

**3. Configure cross-tenant collaboration**

```text
Partner tenant: Contoso
Your tenant: Fabrikam

Inbound trust:
- Trust MFA from Contoso: Yes
- Trust compliant device from Contoso: No

Outbound trust:
- Allow your users to access Contoso resources: Yes

Conditional Access:
- Require MFA for guests unless trusted MFA is present
- Restrict access to approved apps only
```

**4. Hybrid identity selection**

- Choose **Cloud Sync** when you want simpler deployment and your requirements fit the supported feature set.
- Choose **Entra Connect Sync** when you need broader hybrid capabilities or advanced writeback and federation-related scenarios.
- Choose **PHS** when you want the simplest resilient sign-in model.
- Choose **PTA** when you need on-premises password validation without federation.
- Choose **federation** only for specific legacy or claims-based requirements.

**Exam-focused distinctions**

- A **tenant** is not the same as a subscription.
- A **service principal** is not the app registration; it is the tenant-local instance of the app.
- **Enterprise applications** represent the service principal and its tenant-specific settings in the portal.
- **User consent** is not a reliable assumption; many tenants restrict it.
- **Password-based sign-in** is not a modern app authentication pattern.
- **Federation** in Entra ID architecture usually means federation with an external IdP or on-premises federation service for user authentication.
- **Azure RBAC** role assignment is different from Entra ID directory role assignment.
- **Cloud Sync** and **Connect Sync** are not interchangeable; choose based on feature needs.
- **Conditional Access** should be designed with report-only testing and emergency access accounts.
- **Identity Protection** complements, but does not replace, strong authentication and Conditional Access.

**Design checklist**

- Define the tenant boundary and whether a single-tenant or multi-tenant model fits the organization.
- Choose the source of authority for identities and attributes.
- Select the right hybrid sync method and authentication model.
- Register apps with the correct platform type, redirect URIs, and permission model.
- Enforce MFA and authentication strengths for sensitive access.
- Use Conditional Access with report-only validation.
- Protect privileged roles with PIM and break-glass accounts.
- Plan logging, retention, and troubleshooting workflows.
- Apply governance with access packages and access reviews.
- Use cross-tenant access settings for external collaboration.
