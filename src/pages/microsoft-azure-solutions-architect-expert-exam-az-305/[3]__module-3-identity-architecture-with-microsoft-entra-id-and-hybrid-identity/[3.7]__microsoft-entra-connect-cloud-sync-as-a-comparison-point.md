## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Overview**

Microsoft Entra Connect Cloud Sync is a lightweight identity synchronization option for provisioning on-premises Active Directory identities into Microsoft Entra ID. In AZ-305, you usually compare it with Microsoft Entra Connect to choose the right identity architecture for hybrid environments. The key exam idea is that Cloud Sync is simpler to deploy and operate, but it does not replace every capability of Entra Connect.

Cloud Sync is best understood as a provisioning service with an agent-based architecture. It helps you create and maintain user objects in Entra ID from on-premises directories, which then participate in Entra ID authentication, Conditional Access, and other cloud policy decisions. It is not a federation product, and it is not a full parity replacement for every synchronization, writeback, or hybrid identity scenario.

**What Cloud Sync is for**

Cloud Sync is designed for organizations that want a simpler, cloud-managed synchronization model. You install a lightweight provisioning agent on-premises, configure synchronization in the cloud, and let the service handle the orchestration.

Typical reasons to choose it include:

- **Operational simplicity** when you want less infrastructure to manage than a full sync server.
- **Cloud-first identity provisioning** when your main goal is to get users into Entra ID for Microsoft 365 and SaaS access.
- **Reduced server footprint** because you do not need a dedicated synchronization server in the same way as classic Entra Connect.
- **Centralized configuration** through the Entra admin experience rather than extensive local server management.

Cloud Sync is especially attractive when your identity needs are straightforward: synchronize users and selected groups from one or more supported on-premises AD DS environments into Entra ID, then use those identities for cloud access and policy enforcement.

**How it differs from Microsoft Entra Connect**

Microsoft Entra Connect is the broader hybrid identity tool. It supports more advanced scenarios, including federation-related hybrid identity patterns, more complex synchronization topologies, and a wider set of writeback and hybrid features. Cloud Sync is narrower in scope and should be treated as a provisioning-focused alternative, not a universal replacement.

| Capability | Microsoft Entra Connect Cloud Sync | Microsoft Entra Connect | Why it matters in AZ-305 |
|---|---|---|---|
| Primary purpose | Provision identities to Entra ID | Full hybrid identity synchronization and related features | Helps you choose the right tool for the required scope |
| Deployment model | Lightweight agent-based service | Dedicated sync server with sync engine | Impacts operational overhead and HA design |
| User sync | Supported | Supported | Core use case for both |
| Group sync | Limited support for selected scenarios | Broader support | Avoid assuming full group parity |
| Password hash sync | Supported in supported scenarios | Supported | Important for cloud authentication design |
| Pass-through authentication | Not the main design point | Supported | Matters when you need on-prem authentication path |
| Federation integration | Not provided | Can participate in federation-related hybrid designs | Cloud Sync is not a federation solution |
| Writeback features | More limited | Broader set of writeback capabilities | Important for hybrid device and password scenarios |
| Multi-forest support | Supported only in limited, supported topologies | More mature and flexible | Do not assume all multi-forest designs fit Cloud Sync |
| Complexity | Lower | Higher | Tradeoff between simplicity and feature depth |

> Cloud Sync is not a drop-in replacement for every Entra Connect deployment. If you need federation, advanced writeback, or complex synchronization topologies, Entra Connect is usually the safer fit.

**Architecture and data flow**

Cloud Sync uses an on-premises provisioning agent that communicates outbound to Microsoft Entra ID. This reduces inbound firewall exposure and simplifies network design. The service is cloud-managed, so configuration and orchestration happen in Entra rather than on a local sync server.

A typical flow looks like this:

1. You install the Cloud Sync provisioning agent on a domain-joined server.
2. The agent registers with your Entra tenant.
3. You define synchronization scope in the cloud.
4. The service reads objects from Active Directory Domain Services (AD DS).
5. The service provisions or updates matching objects in Entra ID.

```text
On-prem AD DS
   |
   |  outbound HTTPS
   v
Cloud Sync agent  --->  Microsoft Entra ID
                         |
                         +--> Microsoft 365
                         +--> SaaS apps
                         +--> Conditional Access / MFA policies
```

The important distinction is that Cloud Sync provisions identities into Entra ID; it does not itself perform access control. However, once identities exist in Entra ID, they are used by authentication flows, authorization decisions, and policy evaluation in the cloud.

**Identity matching and source anchor concepts**

In hybrid identity, the service must know which on-premises object corresponds to which cloud object. This is where identity matching and source anchor behavior matter.

- **Source anchor** is the immutable identifier used to correlate the on-premises object with the cloud object.
- In AD-based hybrid identity, this is commonly derived from a stable attribute such as `ms-DS-ConsistencyGuid` or another configured anchor strategy, depending on the tool and deployment pattern.
- The goal is to prevent duplicate cloud objects when attributes change, especially UPN changes, renames, or moves between OUs.

Practical implications:

- If you change a user’s `userPrincipalName`, the object should still match the same cloud identity if the source anchor remains stable.
- If you accidentally change the anchor strategy or scope incorrectly, you can create duplicate or orphaned objects.
- Matching is critical during migrations from one sync method to another, such as moving from Entra Connect to Cloud Sync.

Example PowerShell checks on-premises:

```powershell
Get-ADUser -Identity jdoe -Properties userPrincipalName,mail,msDS-ConsistencyGuid |
Select-Object SamAccountName,UserPrincipalName,Mail,msDS-ConsistencyGuid
```

Example Azure CLI check for the cloud object:

```bash
az login
az ad user show --id jdoe@contoso.com
```

**Attribute flow and mapping**

Cloud Sync uses attribute mappings to determine which on-premises values populate Entra ID attributes. In exam scenarios, focus on the business effect of attribute flow rather than memorizing every mapping detail.

Commonly relevant attributes include:

- **`userPrincipalName`** for sign-in identity.
- **`mail`** and related mail attributes for email-enabled scenarios.
- **`displayName`** for user presentation.
- **`proxyAddresses`** for mail routing and coexistence.
- **`accountEnabled`** or equivalent status handling for account lifecycle.

Why this matters:

- Incorrect attribute flow can break sign-in, email routing, or app assignment.
- UPN alignment is especially important when users sign in with a cloud-friendly domain.
- Attribute normalization can be necessary before synchronization if on-premises values do not meet cloud requirements.

A practical example is a migration where on-premises users sign in as `jdoe@corp.local`, but you want cloud sign-in as `jdoe@contoso.com`. You typically update the UPN suffix in AD DS, verify DNS and verified domain readiness in Entra ID, and then let synchronization update the cloud identity.

```powershell
Set-ADUser jdoe -UserPrincipalName jdoe@contoso.com
```

**Deployment model and agent considerations**

Cloud Sync uses one or more agents installed on-premises. Multiple agents can improve resilience, but you should not think of them as a simple active-active sync farm with guaranteed immediate failover. Health, registration, service scheduling, and scope all affect how processing continues if one agent becomes unavailable.

Deployment guidance:

- Place agents on reliable Windows Server hosts with appropriate domain connectivity.
- Use more than one agent when you need resilience.
- Monitor agent health and registration status.
- Keep the agent footprint small and operationally simple.

Practical considerations:

- The agent needs outbound connectivity to Microsoft services.
- You should avoid overloading the host with unrelated workloads.
- If you use multiple agents, validate that each is healthy and correctly registered before relying on redundancy.

Example service checks:

```powershell
Get-Service | Where-Object {$_.DisplayName -like "*Entra*"}
```

Example event log inspection with modern PowerShell:

```powershell
Get-WinEvent -LogName "Application" -MaxEvents 50 |
Where-Object {$_.ProviderName -like "*Entra*"} |
Select-Object TimeCreated, Id, LevelDisplayName, Message
```

**When Cloud Sync is a good fit**

Cloud Sync is a strong choice when your requirements are mostly about identity provisioning into Entra ID and you want a simpler operational model.

Use it when:

- You need to synchronize users from AD DS into Entra ID.
- You want cloud-managed configuration and less server maintenance.
- You do not need federation.
- You do not need the full breadth of Entra Connect writeback and advanced hybrid features.
- Your topology is within the supported Cloud Sync design limits.

Example architecture:

- One on-premises AD forest.
- A small number of OUs scoped for sync.
- Cloud authentication and Conditional Access in Entra ID.
- Microsoft 365 and SaaS access as the primary target.

**When Entra Connect is the better choice**

Choose Microsoft Entra Connect when the design requires more than basic provisioning.

Common reasons include:

- **Federation-related requirements** where you need a federation solution or hybrid integration pattern.
- **Advanced writeback** needs such as password writeback or device-related hybrid scenarios, depending on the exact feature set required.
- **Complex multi-forest** environments with more demanding topology and matching requirements.
- **Broader hybrid identity** use cases where you need the mature feature set of Entra Connect.

A useful exam heuristic:

- If the question emphasizes simplicity and cloud-managed provisioning, think Cloud Sync.
- If the question emphasizes federation, advanced writeback, or complex hybrid identity, think Entra Connect.

**Migration and coexistence scenarios**

A common real-world scenario is moving from Entra Connect to Cloud Sync to reduce operational overhead. This is not just a “flip the switch” change. You need to plan object continuity, scope, and feature parity carefully.

Migration checklist:

1. Inventory current Entra Connect features in use.
2. Identify any features that Cloud Sync does not support for your scenario.
3. Confirm source anchor continuity and object matching strategy.
4. Reduce scope to a pilot OU or pilot user set.
5. Validate sign-in, group membership behavior, and app access.
6. Expand gradually if the pilot succeeds.

Important coexistence rule:

> Do not let two synchronization systems manage the same objects without a clear design. Overlapping scope can create duplicate objects, attribute conflicts, or unexpected lifecycle behavior.

A practical coexistence pattern is to pilot Cloud Sync for a subset of users in a separate OU while keeping the rest of the organization on Entra Connect. This can help you validate operational behavior before a broader migration.

**Troubleshooting focus**

When Cloud Sync has issues, troubleshoot from the agent outward:

- Confirm the agent service is running.
- Confirm the agent is registered to the correct tenant.
- Check outbound connectivity and proxy settings.
- Review synchronization scope and OU filtering.
- Validate source attributes and matching behavior.
- Inspect Entra provisioning logs and agent event logs.

Useful checks:

```powershell
Get-Service | Where-Object {$_.Name -match "Provision|Sync|Entra"}
```

```powershell
Get-WinEvent -LogName "Application" -MaxEvents 100 |
Where-Object {$_.Message -match "provision|sync|error"} |
Select-Object TimeCreated, Id, LevelDisplayName, Message
```

If objects are not appearing in Entra ID, verify:

- The object is in scope.
- The object is not filtered out by OU or attribute rules.
- The UPN and required attributes are valid.
- The source anchor has not changed unexpectedly.
- The agent can reach Microsoft endpoints.

**Exam traps and wording to watch**

AZ-305 questions often use subtle wording to test whether you understand the difference between provisioning and full hybrid identity.

Watch for these traps:

- **“Cloud Sync supports federation”** — it does not. Federation is a separate identity pattern.
- **“Cloud Sync is a full replacement for Entra Connect”** — not always true.
- **“Cloud Sync provides full group synchronization parity”** — too broad and potentially incorrect.
- **“Multiple agents guarantee immediate failover”** — too strong; redundancy exists, but behavior depends on health and scheduling.
- **“Cloud Sync is only for authentication”** — misleading. It provisions identities that are then used by Entra ID authentication and policy.
- **“Any multi-forest design works”** — too broad; supported topologies are more limited than Entra Connect.

**Practical decision guide**

Use this mental model when answering design questions:

- Choose **Cloud Sync** when you want:
  - simpler deployment,
  - cloud-managed provisioning,
  - basic hybrid identity into Entra ID,
  - reduced server footprint.

- Choose **Entra Connect** when you need:
  - federation-related hybrid identity,
  - broader writeback,
  - more advanced synchronization behavior,
  - more complex forest and topology support.

**Quick comparison summary**

- **Cloud Sync**
  - Lightweight, agent-based provisioning.
  - Best for simpler identity sync to Entra ID.
  - Limited compared with Entra Connect in advanced hybrid scenarios.

- **Entra Connect**
  - Full-featured hybrid identity synchronization.
  - Better for complex or feature-rich environments.
  - More operational overhead.

**Key takeaways**

Cloud Sync is a provisioning-focused identity synchronization option for Microsoft Entra ID. It simplifies deployment and management, but it does not replace Entra Connect in every scenario. For AZ-305, the most important distinction is scope: Cloud Sync is for simpler cloud identity provisioning, while Entra Connect remains the better fit for advanced hybrid identity, federation-related designs, and broader feature requirements.
