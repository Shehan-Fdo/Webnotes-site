## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Managed identities overview**

Managed identities let Azure-hosted workloads authenticate to Azure services without you storing credentials in code, config files, or secret stores. Azure manages the underlying identity lifecycle, while your workload requests tokens from the Azure Instance Metadata Service (<abbr title="Instance Metadata Service">IMDS</abbr>) or a platform-specific identity endpoint.

Use managed identities when the workload runs in Azure and needs to access Azure resources such as Key Vault, Storage, SQL, Service Bus, Event Hubs, or Microsoft Graph where supported by the target service and permission model.

- **Secretless authentication** reduces operational risk from leaked connection strings and rotated passwords.
- **Azure-managed lifecycle** means you do not create or rotate app secrets for the identity itself.
- **Token-based access** uses Microsoft Entra ID access tokens, not shared keys.
- **Resource-scoped authorization** still requires permissions on the target service.

**Managed identity types**

Azure provides two main managed identity types:

- **System-assigned** identity
  - Tied to a single Azure resource.
  - Created when you enable it on the resource.
  - Deleted automatically when the resource is deleted.
  - Best when the identity should not be reused elsewhere.

- **User-assigned** identity
  - A standalone Azure resource.
  - Can be attached to multiple Azure resources.
  - Survives deletion of any one workload resource.
  - Best when multiple workloads need the same identity or you want independent lifecycle management.

A practical rule: choose system-assigned for simple one-to-one scenarios, and user-assigned when you need reuse, portability, or centralized governance.

**How managed identities differ from app registrations and service principals**

Managed identities are a specialized form of Microsoft Entra identity for Azure resources. In practice, they differ from app registrations and manually created service principals in important ways:

- **App registration**
  - You define an application in Microsoft Entra ID.
  - You usually manage credentials such as client secrets or certificates.
  - Use this for external apps, multi-cloud apps, or scenarios where the workload is not running in Azure.

- **Service principal**
  - The tenant-local identity used by an application or workload.
  - Often created from an app registration.
  - You manage its credentials unless another platform manages them.

- **Managed identity**
  - Azure creates and manages the service principal-like identity for you.
  - You do not handle client secrets or certificates.
  - Use this for Azure-hosted workloads that need Azure resource access.

> Managed identities are only for Azure-supported hosting environments and supported resource types. They are not a general replacement for every external application identity scenario.

**Identity selection and lifecycle**

System-assigned and user-assigned identities behave differently in deployment and operations:

- **System-assigned**
  - Identity lifecycle is coupled to the resource lifecycle.
  - Recreating the resource creates a new identity.
  - Good for single-purpose workloads with simple access needs.

- **User-assigned**
  - Identity lifecycle is independent of the workload.
  - You can assign the same identity to multiple VMs, App Services, or other supported resources.
  - Good for shared access patterns and blue/green deployments.

Example reuse scenario:

- A pair of virtual machines in an active/passive cluster can share one user-assigned identity to read the same Key Vault secrets.
- A staging slot and production slot can use different user-assigned identities to separate permissions.
- Multiple automation runbooks can use one identity to access the same storage account or queue.

**Authentication flow**

The workload does not store a password. Instead, it requests a token from the local identity endpoint and presents that token to the target service.

Typical flow:

1. Your code requests a token for a specific audience or resource, such as `https://vault.azure.net` or `https://storage.azure.com/`.
2. The Azure platform validates the workload identity.
3. Azure returns an access token.
4. Your code sends the token to the target service in the `Authorization: Bearer` header.
5. The target service checks the token and evaluates authorization through Azure RBAC or service-specific authorization.

The exact endpoint and request shape vary by hosting service and SDK. For example, VM-based workloads commonly use IMDS, while App Service, Azure Functions, and other platforms expose their own managed identity endpoint abstractions.

**Common Azure hosting scenarios**

Managed identities are widely used across Azure services:

- **Virtual Machines and Virtual Machine Scale Sets**
  - Use IMDS to obtain tokens.
  - Common for custom apps, agents, and background services.

- **App Service and Azure Functions**
  - Use the platform identity endpoint.
  - Common for web apps, APIs, and serverless functions.

- **Azure Container Apps and some containerized workloads**
  - Use platform identity integration where supported.
  - Prefer managed identity for Azure resource access, not embedded secrets.

- **Azure Kubernetes Service**
  - Distinguish managed identities for the cluster/control plane and node components from pod authentication patterns.
  - For pod-to-Azure access, workload identity federation is often the preferred model rather than classic managed identity.
  - Use the right pattern for the workload boundary you are securing.

**Managed identity vs workload identity federation**

Do not conflate managed identities with workload identity federation.

- **Managed identity**
  - Best for Azure-hosted resources that can use a platform-managed identity endpoint.
  - The platform brokers token acquisition.

- **Workload identity federation**
  - Best for Kubernetes pods or external workloads that need to authenticate to Entra ID without secrets.
  - Uses federated credentials and OIDC (OpenID Connect) trust.
  - Common in AKS when you want pod-level identity without node-level identity sharing.

For exam purposes, remember:

- Managed identity is the Azure-native identity for supported Azure resources.
- Workload identity federation is the modern secretless pattern for Kubernetes and other federated workloads.

**Authorization models for target services**

Authentication proves identity; authorization grants access. Managed identity alone does not grant permissions.

**Azure RBAC**

Azure Role-Based Access Control (<abbr title="Role-Based Access Control">RBAC</abbr>) is the recommended authorization model for many Azure services, including Key Vault when supported in your environment.

- Assign roles to the managed identity at the appropriate scope.
- Prefer least privilege and narrow scopes.
- Use built-in roles when possible, such as:
  - `Key Vault Secrets User`
  - `Storage Blob Data Reader`
  - `Storage Blob Data Contributor`
  - `Contributor` only when broader management access is truly required

Example role assignment:

```bash
az role assignment create \
  --assignee-object-id <principal-id> \
  --assignee-principal-type ServicePrincipal \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.KeyVault/vaults/<kv-name>
```

**Key Vault access policies vs Azure RBAC**

Key Vault historically supported access policies, but Azure RBAC is the preferred model for many modern deployments.

- **Access policies**
  - Vault-specific permission model.
  - Useful in older deployments or where RBAC is not enabled.
  - Managed separately from Azure role assignments.

- **Azure RBAC**
  - Centralized authorization model.
  - Better alignment with Azure governance and management groups.
  - Preferred when supported and especially for new designs.

> Be careful not to mix the two models without understanding the vault configuration. A vault typically uses either access policies or Azure RBAC for data-plane authorization, not both as the primary model.

**Storage access example**

For Azure Storage, managed identities commonly access Blob or Queue data through Azure RBAC.

Example with Azure CLI:

```bash
az role assignment create \
  --assignee-object-id <principal-id> \
  --assignee-principal-type ServicePrincipal \
  --role "Storage Blob Data Contributor" \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<storage-name>
```

In code, you typically use a token credential from the Azure Identity library rather than a connection string with an account key.

```csharp
using Azure.Identity;
using Azure.Storage.Blobs;

var client = new BlobServiceClient(
    new Uri("https://<storage-name>.blob.core.windows.net"),
    new DefaultAzureCredential());

var containers = client.GetBlobContainers();
```

**SQL access example**

For Azure SQL Database, managed identity works with Microsoft Entra authentication.

Typical approach:

- Enable Microsoft Entra admin on the SQL logical server.
- Create a contained database user for the managed identity.
- Grant database roles such as `db_datareader` or `db_datawriter` as needed.
- Use an Entra-compatible client library or connection string settings.

Example T-SQL:

```sql
CREATE USER [my-managed-identity] FROM EXTERNAL PROVIDER;
ALTER ROLE db_datareader ADD MEMBER [my-managed-identity];
```

Example connection string pattern:

```text
Server=tcp:<server>.database.windows.net,1433;
Database=<db-name>;
Encrypt=True;
TrustServerCertificate=False;
Authentication=Active Directory Managed Identity;
```

In application code, use an Azure Identity credential and a SQL client that supports Entra authentication.

```csharp
using Azure.Identity;
using Microsoft.Data.SqlClient;

var connection = new SqlConnection(
    "Server=tcp:<server>.database.windows.net,1433;Database=<db-name>;Encrypt=True;Authentication=Active Directory Managed Identity;");

await connection.OpenAsync();
```

**Key Vault access example**

A common pattern is a VM or App Service reading a secret from Key Vault.

Azure CLI example for a user-assigned identity and RBAC:

```bash
az identity create -g <rg> -n <uami-name>

az role assignment create \
  --assignee-object-id <principal-id> \
  --assignee-principal-type ServicePrincipal \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.KeyVault/vaults/<kv-name>
```

App code example:

```csharp
using Azure.Identity;
using Azure.Security.KeyVault.Secrets;

var client = new SecretClient(
    new Uri("https://<kv-name>.vault.azure.net/"),
    new DefaultAzureCredential());

KeyVaultSecret secret = await client.GetSecretAsync("DbPassword");
Console.WriteLine(secret.Value);
```

**Portal, ARM, Bicep, and PowerShell configuration**

You should know both portal and infrastructure-as-code paths.

- **Azure portal**
  - Enable identity on the resource under the Identity blade.
  - For user-assigned identities, attach the identity to the resource.
  - Assign roles on the target resource or configure Key Vault authorization.

- **ARM template**
  - Use the `identity` property on supported resources.
  - For user-assigned identities, reference the identity resource ID.

- **Bicep**
  - Declarative and concise for identity-enabled resources.
  - Good for repeatable deployments and exam scenarios.

Example Bicep for a system-assigned identity:

```bicep
resource vm 'Microsoft.Compute/virtualMachines@2024-03-01' = {
  name: 'vm1'
  location: resourceGroup().location
  identity: {
    type: 'SystemAssigned'
  }
  properties: {
    // omitted for brevity
  }
}
```

Example Bicep for a user-assigned identity attached to a web app:

```bicep
resource uami 'Microsoft.ManagedIdentity/userAssignedIdentities@2023-01-31' = {
  name: 'uami-shared'
  location: resourceGroup().location
}

resource app 'Microsoft.Web/sites@2023-12-01' = {
  name: 'app1'
  location: resourceGroup().location
  identity: {
    type: 'UserAssigned'
    userAssignedIdentities: {
      '${uami.id}': {}
    }
  }
  properties: {
    serverFarmId: appServicePlan.id
  }
}
```

PowerShell example:

```powershell
Connect-AzAccount

$vm = Get-AzVM -ResourceGroupName "rg1" -Name "vm1"
$vm.Identity = New-Object Microsoft.Azure.Management.Compute.Models.VirtualMachineIdentity
$vm.Identity.Type = "SystemAssigned"
Update-AzVM -ResourceGroupName "rg1" -VM $vm
```

**CLI examples for identity management**

Create a user-assigned identity:

```bash
az identity create \
  --resource-group rg1 \
  --name uami-shared
```

Show identity details:

```bash
az identity show \
  --resource-group rg1 \
  --name uami-shared
```

Assign a user-assigned identity to a VM:

```bash
az vm identity assign \
  --resource-group rg1 \
  --name vm1 \
  --identities /subscriptions/<sub-id>/resourceGroups/rg1/providers/Microsoft.ManagedIdentity/userAssignedIdentities/uami-shared
```

Enable system-assigned identity on an App Service:

```bash
az webapp identity assign \
  --resource-group rg1 \
  --name app1
```

List role assignments for the identity:

```bash
az role assignment list \
  --assignee-object-id <principal-id> \
  --all
```

**Using the identity in code**

Most modern Azure SDKs support `DefaultAzureCredential`, which tries several credential sources and uses managed identity automatically when running in Azure.

```csharp
using Azure.Identity;

var credential = new DefaultAzureCredential();
```

This is useful because:

- It works locally with developer sign-in.
- It works in Azure with managed identity.
- It reduces environment-specific code.

If you need to force a specific user-assigned identity, use a credential that accepts the client ID.

```csharp
using Azure.Identity;

var credential = new ManagedIdentityCredential(clientId: "<user-assigned-client-id>");
```

Remember the distinction:

- **Client ID** is the application ID for the user-assigned identity.
- **Principal ID** is the service principal object ID used in role assignments and directory operations.

**Practical topology patterns**

Common architecture patterns include:

- **Web app to Key Vault**
  - App Service uses system-assigned identity.
  - Key Vault uses Azure RBAC.
  - App reads secrets at runtime.

- **VM to Storage**
  - VM uses system-assigned identity.
  - Storage account grants `Storage Blob Data Reader`.
  - App downloads blobs without account keys.

- **Shared identity across multiple resources**
  - One user-assigned identity is attached to several VMs.
  - All VMs access the same database or vault with the same permissions.
  - Useful for clustered services and standardized automation.

- **Multi-tier application**
  - Front-end and back-end use different identities.
  - Each tier gets only the permissions it needs.
  - Improves blast-radius control.

**Limitations and constraints**

Managed identities are powerful, but they have boundaries:

- They are intended for **Azure-hosted workloads**, not arbitrary external applications.
- They depend on **supported Azure resource types** and platform integration.
- They are generally **tenant-bound** to the Microsoft Entra tenant associated with the Azure subscription and resource.
- They do not eliminate the need for **authorization** on the target service.
- They may require **propagation time** after creation or role assignment before access works everywhere.
- They can fail if the hosting platform’s identity endpoint is unavailable or if the workload requests the wrong resource/audience.

**Troubleshooting and common failures**

When managed identity access fails, check the following:

- **Wrong audience/resource**
  - The token must target the correct resource URI.
  - Example: Key Vault uses `https://vault.azure.net`, Storage uses `https://storage.azure.com/`.

- **Missing role assignment**
  - The identity may authenticate successfully but still receive `403 Forbidden`.
  - Verify the role scope and role definition.

- **Propagation delay**
  - New identities and role assignments can take time to become effective.
  - Retry after a short delay before assuming configuration is wrong.

- **Identity not enabled on the resource**
  - The workload must have system-assigned or user-assigned identity attached.

- **Wrong identity selected**
  - If multiple user-assigned identities are attached, your code may use the wrong one unless you specify the client ID.

- **Endpoint availability**
  - On VMs, IMDS is only available from the Azure-hosted environment.
  - Local development cannot use IMDS directly; use developer credentials locally.

- **Key Vault authorization mode mismatch**
  - If the vault uses Azure RBAC, access policies will not grant data-plane access.
  - If the vault uses access policies, RBAC role assignment alone may not be enough.

**Exam-focused quick reference**

- **Managed identity**
  - Azure-managed identity for Azure-hosted workloads.
  - No client secret or certificate to manage.

- **System-assigned**
  - One resource, one identity, lifecycle tied to the resource.

- **User-assigned**
  - Reusable identity, independent lifecycle, attach to multiple resources.

- **Principal ID**
  - The service principal object ID used for role assignments.

- **Client ID**
  - The application ID used by code when selecting a specific user-assigned identity.

- **Best authorization model**
  - Prefer Azure RBAC for supported services.
  - Use Key Vault access policies only when required by the vault’s configuration or legacy design.

- **Best Kubernetes pattern**
  - Prefer workload identity federation for pod authentication.
  - Do not assume managed identity is the primary AKS pod identity model.

**When to choose managed identity**

Use managed identity when:

- Your workload runs in Azure.
- You want to avoid secrets and connection strings.
- The target service supports Entra-based authentication.
- You want centralized governance through Azure RBAC.

Do not use managed identity when:

- The app runs outside Azure and cannot use Azure-hosted identity endpoints.
- The target service requires a credential type not supported by managed identity.
- You need a cross-cloud or external identity federation pattern instead.
