## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Group-based access design overview**

Group-based access design is the practice of assigning permissions to groups instead of individual users. In Azure, this reduces administrative overhead, improves consistency, and makes access reviews and offboarding much easier. You design groups differently depending on whether you are controlling access to Azure resources, Microsoft 365 workloads, or privileged directory operations.

The core idea is simple: users change often, resources and roles should not. If you assign access to a group, you can add or remove users without rewriting role assignments across subscriptions, resource groups, or applications.

- **Least privilege** is easier to enforce when access is centralized in groups.
- **Scalability** improves because one role assignment can cover many users.
- **Auditability** improves because group membership changes are easier to review than scattered direct assignments.
- **Separation of duties** is easier when you split admin, operator, and reader groups.

**Choose the right group type**

Azure and Microsoft Entra ID support different group types, and the choice matters.

- **Security groups** are the standard choice for Azure role-based access control (RBAC) assignments.
- **Microsoft 365 groups** are designed for collaboration services such as Teams, Outlook, and SharePoint, not as the typical principal for Azure RBAC.
- **Dynamic groups** automatically add or remove members based on user or device attributes.
- **Assigned groups** use manual membership and are best when access changes are controlled by process.
- **Role-assignable groups** can be assigned Microsoft Entra roles and are useful for privileged directory administration.

A practical rule is to use security groups for Azure resource access, Microsoft 365 groups for collaboration, and dynamic groups when membership should follow identity attributes such as department, location, or job title.

| Group type | Typical use | Good for Azure RBAC? | Notes |
|---|---|---:|---|
| Security group | Resource access and app authorization | Yes | Standard choice for subscriptions, resource groups, and management groups |
| Microsoft 365 group | Collaboration and shared content | Usually no | Not the usual authorization principal for Azure RBAC |
| Dynamic security group | Attribute-driven access | Yes | Useful when membership should be automated |
| Role-assignable group | Privileged Entra role assignment | Sometimes | Use for directory roles, not general collaboration |
| Mail-enabled security group | Email plus access control | Sometimes | Can be used for access, but keep governance clear |

**How Azure RBAC uses groups**

Azure RBAC authorizes access to Azure resources by assigning roles at a scope. The scope can be management group, subscription, resource group, or resource. When you assign a role to a group, every current member inherits that access.

This is the preferred pattern for most enterprise designs because it separates identity management from resource authorization.

- Assign the role to the **group**, not the user, when access is shared by a team.
- Use the narrowest **scope** that satisfies the requirement.
- Prefer built-in roles before creating custom roles.
- Keep direct user assignments rare and time-bound.

Common examples:

- A platform team gets **Contributor** on a resource group that contains shared infrastructure.
- A security operations team gets **Reader** on subscriptions for visibility.
- An application support group gets a custom role with only the actions needed to restart or read specific resources.
- A break-glass admin account is excluded from normal group-based workflows and protected separately.

**Designing group structure**

A good group design reflects how the organization works. You usually model access by function, environment, and sometimes application.

- **Functional groups** map to job roles such as `rg-app1-dev-contributors` or `sub-prod-readers`.
- **Environment groups** separate dev, test, and production access.
- **Application groups** isolate access for a single workload or service team.
- **Privileged groups** hold elevated permissions and should have stricter controls.

A common naming pattern is:

```text
<scope>-<workload>-<environment>-<role>
```

Examples:

```text
sub-payments-prod-readers
rg-web-dev-contributors
app-erp-prod-operators
```

This naming style helps you understand intent quickly during audits and incident response.

**When to use security groups vs Microsoft 365 groups vs dynamic groups**

Use the group type that matches the control plane you are managing.

- Use **security groups** when the main goal is authorization.
- Use **Microsoft 365 groups** when the main goal is collaboration and shared content.
- Use **dynamic security groups** when membership should be driven by identity attributes and you want to reduce manual updates.
- Use **role-assignable groups** when you need a group to receive Entra directory roles.

Real-world decision examples:

- If all finance analysts should access a reporting app and Azure SQL database, use a security group for the app and Azure RBAC.
- If a project team needs a shared mailbox, Teams channel, and SharePoint site, use a Microsoft 365 group.
- If all users in the `Department = Sales` attribute should get a SaaS app assignment, use a dynamic security group.
- If a small identity admin team needs the ability to manage users in Entra ID, use a role-assignable group with a directory role assignment.

> Microsoft 365 groups are collaboration-first objects. For Azure RBAC, security groups are the normal and safest choice.

**Azure RBAC assignment workflow**

A typical workflow is:

1. Create or identify the security group.
2. Add the right users, service accounts, or managed identities to the group.
3. Assign the Azure role to the group at the correct scope.
4. Verify access with a test user or by checking effective permissions.
5. Review membership and role assignments regularly.

Example with Azure CLI:

```bash
# Create a security group
az ad group create \
  --display-name "rg-web-dev-contributors" \
  --mail-nickname "rg-web-dev-contributors"

# Get the group object ID
az ad group show \
  --group "rg-web-dev-contributors" \
  --query id -o tsv

# Assign Contributor at a resource group scope to the group object ID
az role assignment create \
  --assignee-object-id <group-object-id> \
  --assignee-principal-type Group \
  --role Contributor \
  --scope /subscriptions/<subscription-id>/resourceGroups/rg-web-dev
```

The object ID form is the clearest for groups because it avoids ambiguity. In exam scenarios, prefer the explicit group object ID and principal type when describing role assignment.

**Scope design and inheritance**

Azure RBAC scopes inherit downward. A role assigned at a management group applies to child subscriptions, resource groups, and resources unless a more specific assignment changes the effective access.

- **Management group scope** is best for broad platform governance.
- **Subscription scope** is common for shared operations teams.
- **Resource group scope** is the most common scope for application teams.
- **Resource scope** is best for tightly limited access to one resource.

Use the highest practical scope only when the same access is truly needed everywhere below it. Otherwise, keep scope narrow to reduce blast radius.

Example decision:

- A central cloud operations team may need Reader across all subscriptions, so management group scope is appropriate.
- An app team managing one workload should usually get Contributor only on that workload’s resource group.
- A database administrator may need access only to one Azure SQL server or database, not the whole subscription.

**Built-in roles and custom roles**

Built-in roles cover most needs, but custom roles are useful when built-in roles are too broad.

- **Reader** is for visibility without change rights.
- **Contributor** can manage resources but cannot grant access.
- **Owner** can manage resources and access, so it is highly privileged.
- **User Access Administrator** can manage role assignments and is sensitive.
- **Custom roles** let you allow only the required actions.

A custom role is useful when a team needs a narrow operational set, such as reading a VM, restarting it, or managing tags, without full Contributor rights.

Example custom role definition:

```json
{
  "Name": "VM Operator Limited",
  "IsCustom": true,
  "Description": "Can read and restart virtual machines, but cannot change access.",
  "Actions": [
    "Microsoft.Compute/virtualMachines/read",
    "Microsoft.Compute/virtualMachines/restart/action",
    "Microsoft.Resources/subscriptions/resourceGroups/read"
  ],
  "NotActions": [
    "Microsoft.Authorization/*/write",
    "Microsoft.Authorization/*/delete"
  ],
  "AssignableScopes": [
    "/subscriptions/<subscription-id>"
  ]
}
```

This matters because it lets you align permissions with a real operational task instead of granting broad Contributor access.

**Privileged access and PIM**

<abbr title="Privileged Identity Management">PIM</abbr> helps you reduce standing privilege by making access eligible instead of permanently active. In Azure and Microsoft Entra ID, PIM can manage eligible assignments for directory roles and Azure resource roles. PIM for Groups manages eligible or active membership and ownership for privileged groups.

Use PIM when access is sensitive, infrequent, or should require approval and justification.

- Use **eligible** assignments for admins who should activate access only when needed.
- Use **time-bound** activation to limit exposure.
- Use **approval** for high-risk roles.
- Use **justification** and **MFA** (multifactor authentication) for stronger control.

A common pattern is:

- Put privileged users in a role-assignable group.
- Make the group eligible for a directory role through PIM.
- Require activation before the role becomes active.
- Review activations and membership regularly.

> Do not treat PIM for groups as the same thing as PIM for Azure resource roles. They solve related but different problems.

**Dynamic group design**

Dynamic groups reduce manual membership management by evaluating rules against user or device attributes. They are useful when access should follow HR or directory data.

Example dynamic membership rule:

```text
(user.department -eq "Finance") and (user.accountEnabled -eq true)
```

Use dynamic groups when:

- Membership changes frequently.
- Access should follow authoritative identity attributes.
- You want to reduce help desk work for joiners, movers, and leavers.

Avoid dynamic groups when:

- Membership needs manual approval.
- The rule is too complex or unstable.
- The access is highly sensitive and should be reviewed individually.

Dynamic groups are especially useful for SaaS app assignments, but they can also support Azure access patterns when the membership logic is stable and well governed.

**Practical access design scenarios**

A few end-to-end examples make the design choices clearer.

- **Scenario 1: Application team access to a dev subscription**
  - Create a security group for the app team.
  - Assign Contributor at the resource group scope.
  - Keep production in a separate group with stricter membership controls.
  - Use PIM if the production role is privileged.

- **Scenario 2: Central operations visibility**
  - Create a Reader group for the operations team.
  - Assign Reader at the management group or subscription scope.
  - Use the same group across multiple subscriptions if the visibility requirement is consistent.

- **Scenario 3: Privileged identity administration**
  - Use a role-assignable group for identity admins.
  - Protect membership with PIM.
  - Require approval and MFA for activation.
  - Review access frequently.

- **Scenario 4: Department-driven SaaS access**
  - Use a dynamic security group based on department.
  - Assign the group to the application.
  - Let HR attribute changes drive access automatically.

**Troubleshooting access issues**

When a user cannot access a resource, check the full authorization path before changing roles.

1. Confirm the user is in the expected group.
2. Confirm the group has the role assignment.
3. Confirm the role assignment is at the correct scope.
4. Confirm the user is using the right tenant and subscription.
5. Confirm the role is active if PIM is involved.
6. Check whether the resource supports the action you expect.

Useful CLI checks:

```bash
# Show group membership
az ad group member list --group "rg-web-dev-contributors" -o table

# Show role assignments for a principal
az role assignment list \
  --assignee <group-object-id> \
  --all -o table

# Show role assignments at a scope
az role assignment list \
  --scope /subscriptions/<subscription-id>/resourceGroups/rg-web-dev \
  -o table
```

If access still fails, verify whether the user is looking at the correct resource scope or whether the role simply does not include the required action. Governance tools such as Azure Policy can block resource creation or modification, but they are not a substitute for RBAC and should not be confused with access assignment.

**Governance and lifecycle**

Group-based access works best when paired with lifecycle controls.

- **Access reviews** help you remove stale membership.
- **Joiner-mover-leaver** processes keep membership aligned with employment status.
- **Naming standards** make audits faster.
- **Ownership** should be assigned so every group has a responsible manager.
- **Expiration** is useful for temporary project access.

A mature design treats groups as managed assets, not just containers for users.

**Exam focus points**

- Use **security groups** for Azure RBAC by default.
- Use **Microsoft 365 groups** for collaboration, not as the standard Azure authorization principal.
- Use **dynamic groups** when membership should be attribute-driven.
- Scope roles as narrowly as possible.
- Prefer **built-in roles** unless a custom role is needed.
- Use **PIM** to reduce standing privilege.
- Use explicit group object IDs in CLI role assignment examples.
- Distinguish between **directory roles**, **Azure resource roles**, and **group membership**.

**Quick reference**

- Azure RBAC answers the question: “Who can do what on which Azure resource?”
- Groups answer the question: “How do I manage many users as one authorization unit?”
- PIM answers the question: “How do I make privileged access temporary and controlled?”
- Dynamic groups answer the question: “How do I automate membership based on identity data?”
