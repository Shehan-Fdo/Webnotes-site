## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Identity source of authority**

The source of authority is the system that owns a given identity object and is responsible for creating, updating, and deleting it. In Azure identity architecture, you decide whether the authoritative source is on-premises Active Directory Domain Services (AD DS), Microsoft Entra ID, an HR system, or an application-specific directory.

This decision affects provisioning, deprovisioning, attribute flow, governance, and recovery. If you get the source of authority wrong, you create duplicate accounts, stale access, and inconsistent lifecycle behavior across apps and groups.

- **Authoritative source** determines where changes should be made first.
- **Downstream systems** consume replicated or synchronized identity data.
- **Lifecycle automation** should follow business events such as hire, transfer, and termination.

**Core lifecycle model**

A practical identity lifecycle usually includes joiner, mover, and leaver events:

- **Joiner**: create the account, assign baseline access, and establish authentication methods.
- **Mover**: update department, manager, role, group membership, and app access.
- **Leaver**: disable access, revoke sessions, remove licenses, and eventually delete or archive the account.

In Microsoft Entra ID, lifecycle design often combines manual administration, group-based assignment, entitlement management, and automated provisioning from HR or other authoritative systems.

**Microsoft Entra ID user lifecycle administration**

Microsoft Entra ID supports several user lifecycle capabilities that matter in design and exam scenarios:

- **Self-service sign-up** lets users create accounts for certain apps or tenant experiences without help desk intervention.
- **Self-service password reset (SSPR)** allows users to reset passwords after verifying identity through registered methods.
- **Account recovery** covers restoring access when a user loses a password, device, or authentication method, often through alternate methods or admin-assisted recovery.
- **Self-service password change** is different from reset; the user knows the current password and changes it proactively.
- **User provisioning** can be automated from HR or other systems using Microsoft Entra provisioning or SCIM (System for Cross-domain Identity Management).

Design these features together. For example, if you enable SSPR but do not require strong authentication method registration, you weaken recovery assurance. If you automate provisioning but leave manual license assignment, you create drift.

**Identity source of authority by object type**

Different object types can have different authorities. That is normal and often desirable.

- **User accounts**
  - Cloud-only users are authoritative in Microsoft Entra ID.
  - Hybrid users are often authoritative in on-premises AD DS and synchronized to Entra ID.
  - External users are usually authoritative in the external tenant, with your tenant controlling access policy.
- **Groups**
  - Cloud-managed groups are authoritative in Entra ID.
  - Synced groups are authoritative in AD DS and should not be edited in the cloud for membership.
  - Dynamic groups are authoritative in Entra ID rules, not manual membership.
- **Devices**
  - Device identity may be authoritative in Intune, Entra ID join, or hybrid join depending on management model.
  - For hybrid joined devices, AD DS and Entra ID both matter, but the join state and registration source determine lifecycle actions.
- **Service principals and app registrations**
  - App registrations are authoritative in the tenant where the application is registered.
  - Service principals are tenant-local instances of an app and are managed in the consuming tenant.
  - Secrets, certificates, and permissions require explicit lifecycle ownership because they are not user-managed objects.

> If an object is synchronized, do not treat the cloud copy as the source of truth for mutable attributes that originate on-premises.

**Hybrid identity design choices**

Hybrid identity is common when you need on-premises authentication, legacy app support, or staged migration. The main design choice is how identity data and authentication are synchronized or validated.

- **Microsoft Entra Connect Sync**
  - Traditional synchronization engine for users, groups, and contacts.
  - Supports password hash synchronization, pass-through authentication, and federation scenarios.
  - Common when you need richer synchronization customization or existing deployments.
- **Microsoft Entra Cloud Sync**
  - Lightweight cloud-managed provisioning agent model.
  - Often simpler to deploy and operate for many synchronization scenarios.
  - Good fit when you want reduced infrastructure overhead and cloud-first management.
- **Password hash synchronization (PHS)**
  - Synchronizes a hash of the password to Entra ID.
  - Enables cloud authentication and supports SSPR writeback scenarios in many designs.
  - Often the simplest and most resilient sign-in method.
- **Pass-through authentication (PTA)**
  - Validates passwords against on-premises AD DS through an agent.
  - Keeps authentication decisions on-premises while still using Entra ID for access.
  - Useful when you need immediate password validation against AD DS.
- **Federation**
  - Redirects authentication to a federation service such as AD FS.
  - Used when you need advanced on-premises authentication policies or legacy dependencies.
  - Adds operational complexity and should be justified by requirements.

A common architecture decision is to prefer PHS unless a requirement specifically demands PTA or federation. PHS is usually simpler to operate and more resilient because sign-in does not depend on live on-premises authentication infrastructure.

**Practical hybrid example**

```text
HR system -> Entra provisioning -> Entra ID user
AD DS -> Entra Connect Sync -> Entra ID hybrid user
Entra ID -> SaaS apps via SSO
```

In this model:

- HR is authoritative for employment status and core attributes.
- AD DS is authoritative for legacy directory attributes.
- Entra ID is authoritative for cloud app access and governance.
- SaaS apps consume identity through single sign-on (SSO) and provisioning.

**Provisioning from HR systems**

HR-driven provisioning is a common enterprise pattern because HR usually owns the employee record. The HR system becomes the trigger for account creation, updates, and termination.

Typical flow:

1. HR creates or updates the worker record.
2. A provisioning engine reads the event.
3. The engine creates or updates the Entra user.
4. The engine assigns groups, licenses, and app access.
5. The engine disables or deletes the account when employment ends.

Example SCIM-style provisioning payload:

```json
{
  "userName": "alex.wilson@contoso.com",
  "name": {
    "givenName": "Alex",
    "familyName": "Wilson"
  },
  "active": true,
  "emails": [
    {
      "value": "alex.wilson@contoso.com",
      "primary": true
    }
  ],
  "urn:ietf:params:scim:schemas:extension:enterprise:2.0:User": {
    "department": "Finance",
    "manager": "maria.chen@contoso.com"
  }
}
```

Use HR as the source of authority for employment status, but not necessarily for every access decision. Access governance should still be policy-driven through groups, access packages, and reviews.

**Group lifecycle and source of authority**

Group lifecycle is a major exam topic because group ownership determines access governance.

- **Cloud-managed groups**
  - Created and maintained in Entra ID.
  - Best for cloud app access, dynamic membership, and entitlement management.
  - Can be governed with access reviews and privileged identity management where applicable.
- **Synced groups from AD DS**
  - Membership is mastered on-premises.
  - Cloud changes are overwritten by synchronization.
  - Useful when legacy processes or on-premises admin teams own membership.
- **Dynamic groups**
  - Membership is calculated from user or device attributes.
  - Good for role-based access, department-based access, or device compliance targeting.
  - Reduce manual administration but require clean attribute data.

Design implications:

- If a group is synced from AD DS, do not use Entra ID to manage membership.
- If a group is dynamic, ensure the rule reflects business intent and does not accidentally overgrant access.
- If a group is cloud-managed, define an owner and review cadence to prevent privilege creep.

Example dynamic group rule:

```text
(user.department -eq "Finance") and (user.accountEnabled -eq true)
```

Example PowerShell for cloud group membership management:

```powershell
Connect-MgGraph -Scopes "Group.ReadWrite.All"
New-MgGroup -DisplayName "Finance-App-Users" -MailEnabled:$false -SecurityEnabled:$true -MailNickname "finance-app-users"
Add-MgGroupMemberByRef -GroupId $groupId -DirectoryObjectId $userId
```

**Access governance with entitlement management**

Microsoft Entra entitlement management helps you package access into access packages. This is useful when users need a controlled request-and-approval path for groups, apps, and SharePoint sites.

Use access packages when:

- Access should be requestable by users or sponsors.
- Approval is required before access is granted.
- Access should expire automatically.
- You need periodic revalidation of access.

Example access package policy design:

```text
Access package: Finance Contractor Access
Resources:
- Security group: Finance-App-Users
- Enterprise app: SAP Finance
- SharePoint site: Finance Docs

Policy:
- Requestors: external users from approved partner domains
- Approval: manager + resource owner
- Duration: 30 days
- Expiration: automatic
- Reviews: every 30 days
```

This pattern is stronger than manually adding users to groups because it creates an auditable lifecycle with expiration and review.

**Access reviews**

Access reviews help you validate whether users still need access. They are especially important for guests, privileged roles, and high-risk groups.

Good review targets include:

- Privileged role assignments
- Guest users
- Sensitive application groups
- Access packages with business-critical resources

Example scheduling pattern:

```text
Review name: Quarterly Finance Access Review
Scope: Finance-App-Users group
Frequency: Every 90 days
Reviewers: resource owner and manager
Auto-apply results: remove access for non-responders
```

Use shorter review cycles for privileged access and longer cycles for low-risk access. If the business cannot support frequent reviews, reduce the scope or automate the decision criteria.

**Privileged Identity Management**

Privileged Identity Management (PIM) provides just-in-time access for privileged roles and groups. It reduces standing privilege by requiring activation when needed.

Common PIM settings:

- **Eligible assignment** means the user can activate the role but does not hold it permanently.
- **Activation duration** limits how long the role stays active.
- **MFA** (multifactor authentication) can be required at activation.
- **Approval** can be required for sensitive roles.
- **Justification** and **ticket number** can be required for auditability.

Example role activation policy:

```text
Role: Global Administrator
Assignment: Eligible
Activation duration: 1 hour
Require MFA: Yes
Require approval: Yes
Require justification: Yes
Require ticket number: Yes
```

Use PIM for administrative roles, privileged groups, and sometimes application roles where just-in-time elevation is appropriate.

**External identities and B2B lifecycle**

B2B (business-to-business) collaboration introduces external users whose identity is managed in their home tenant. Your tenant controls access to your resources, not the external user’s primary identity.

Lifecycle considerations:

- **Invitation**
  - You invite the guest or allow self-service redemption where supported.
  - The guest account is created in your tenant as a representation of the external identity.
- **Redemption**
  - The guest accepts the invitation and proves control of the external identity.
  - Access is then governed by your tenant policies.
- **Cross-tenant access settings**
  - Control inbound and outbound collaboration trust.
  - Define whether you trust MFA, compliant device claims, and other signals from the partner tenant.
- **Governance**
  - Use access reviews, expiration, and conditional access.
  - Remove guest access when the business relationship ends.

Example cross-tenant access design:

```text
Partner tenant: Contoso Suppliers
Inbound trust:
- Trust MFA claims: Yes
- Trust compliant device claims: No
Outbound access:
- Allow collaboration: Yes
- Allow B2B direct connect: Only for Teams
```

For external identities, the source of authority for the person remains the home tenant. Your tenant is authoritative only for the guest object, access policy, and lifecycle of access to your resources.

**Account disable vs delete**

Disabling and deleting are not the same.

- **Disable**
  - Stops sign-in immediately.
  - Preserves the object for investigation, mailbox handling, license cleanup, and recovery planning.
  - Often the first step in leaver processing.
- **Delete**
  - Removes the object from the directory.
  - May trigger soft-delete behavior depending on the object type and service.
  - Should happen only after retention, legal, and operational requirements are satisfied.

Operationally, disable first, then remove access, then decide whether to delete after the retention window. This sequence reduces risk and preserves auditability.

Important effects:

- Licenses are released when the user is removed or unlicensed.
- App assignments may remain in app systems unless those systems also receive deprovisioning events.
- Mailboxes, OneDrive, and other workloads may have separate retention and recovery behavior.
- Soft-deleted objects can often be restored within a limited period, but exact behavior depends on the object and service.

> Deleting a user in Entra ID does not guarantee immediate deletion in every downstream SaaS app. Always design explicit deprovisioning.

**Soft-delete and restore behavior**

Soft-delete is a safety mechanism that allows recovery after accidental deletion. The exact retention window varies by service and object type.

Use restore when:

- The deletion was accidental.
- The user still needs access and the retention window has not expired.
- You need to recover licenses, group membership, or object references.

Example Microsoft Graph restore pattern:

```http
POST https://graph.microsoft.com/v1.0/directory/deletedItems/{id}/restore
```

For operational design, document:

- Which objects support restore.
- How long deleted objects remain recoverable.
- Which downstream systems need manual reconciliation after restore.

**Device identity source of authority**

Device identity is often overlooked, but it matters in access decisions and compliance.

- **Entra joined devices**
  - Cloud-managed identity.
  - Common for modern managed endpoints.
  - Best aligned with Intune and conditional access.
- **Hybrid joined devices**
  - Joined to AD DS and registered in Entra ID.
  - Useful when you still depend on on-premises domain services.
- **Registered devices**
  - Often personal or bring-your-own-device scenarios.
  - Identity is lighter weight and usually tied to user sign-in rather than full domain join.

Design choice:

- Use Entra join for cloud-first managed endpoints.
- Use hybrid join when legacy domain dependencies remain.
- Use device compliance and conditional access to govern access rather than relying only on device join state.

**Service principals and app registrations**

Applications also have a lifecycle and source of authority.

- **App registration**
  - Defines the application identity in the home tenant.
  - Owns client ID, redirect URIs, credentials, and permissions.
- **Service principal**
  - Represents the app in a tenant where it is used.
  - Owns local assignments, consent state, and access to resources.

Lifecycle tasks include:

- Rotating secrets and certificates.
- Removing unused app permissions.
- Reviewing consent grants.
- Deleting stale service principals and app registrations.

Example Azure CLI commands:

```bash
az ad app create --display-name "Finance API"
az ad sp create --id <appId>
az ad app credential reset --id <appId> --append --years 1
```

Treat app credentials like privileged secrets. They need ownership, rotation, and expiration just like user access.

**Practical deprovisioning example**

A good leaver workflow combines identity, access, and workload actions:

1. Disable the user in Entra ID.
2. Revoke refresh tokens and active sessions.
3. Remove group memberships and access package assignments.
4. Unassign licenses.
5. Trigger downstream app deprovisioning through SCIM or app-specific automation.
6. Archive or transfer data according to retention policy.
7. Delete the account only after the retention period.

Example Graph-based actions:

```http
POST https://graph.microsoft.com/v1.0/users/{id}/revokeSignInSessions
PATCH https://graph.microsoft.com/v1.0/users/{id}
{
  "accountEnabled": false
}
```

**Design patterns and exam guidance**

Use these patterns when deciding source of authority:

- **HR is authoritative for employment status**
  - Best for create, move, and terminate events.
- **AD DS is authoritative for legacy directory attributes**
  - Best when on-premises apps still depend on domain objects.
- **Entra ID is authoritative for cloud access**
  - Best for cloud apps, conditional access, and governance.
- **Application is authoritative for app-specific attributes**
  - Best when the app owns its own profile or entitlement model.

Common mistakes to avoid:

- Managing synced group membership in the cloud.
- Using manual user administration where HR-driven provisioning is required.
- Deleting users before downstream systems are deprovisioned.
- Treating guest users as if your tenant owns their primary identity.
- Ignoring device and app identity lifecycles in broader architecture decisions.

**Quick comparison**

| Object type | Typical source of authority | Lifecycle control |
|---|---|---|
| User | HR, AD DS, or Entra ID | Provision, update, disable, delete |
| Group | AD DS, Entra ID, or dynamic rule | Membership and ownership |
| Guest user | External home tenant | Access in your tenant |
| Device | Intune, AD DS, or Entra join state | Join, compliance, retire |
| App registration | Tenant where app is registered | Credentials, permissions, deletion |
| Service principal | Consuming tenant | Assignment, consent, access |

**Key takeaways**

- Choose one authoritative source per attribute set and avoid conflicting ownership.
- Prefer automation for joiner, mover, and leaver events.
- Use PHS, PTA, or federation based on operational and security requirements, not habit.
- Govern groups, guests, privileged roles, and apps with reviews, expiration, and just-in-time access.
- Disable first, then deprovision, then delete after retention requirements are met.
