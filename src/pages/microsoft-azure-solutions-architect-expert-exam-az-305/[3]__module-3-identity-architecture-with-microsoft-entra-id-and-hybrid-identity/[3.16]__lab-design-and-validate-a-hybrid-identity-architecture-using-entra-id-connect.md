## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Hybrid identity architecture overview**

Hybrid identity connects on-premises Active Directory Domain Services (AD DS) with Microsoft Entra ID so users can authenticate and access cloud and on-premises resources with a consistent identity. In an AZ-305 design scenario, you choose the right authentication model, sync scope, and writeback features based on security, resilience, and operational requirements.

A typical design includes:
- **On-premises AD DS** as the authoritative source for user and group objects.
- **Microsoft Entra ID Connect Sync** or **Microsoft Entra Cloud Sync** to provision identities into Entra ID.
- One or more sign-in methods:
  - **Password Hash Synchronization (PHS)**
  - **Pass-through Authentication (PTA)**
  - **Federation** with AD FS or another identity provider when a specific technical requirement cannot be met by cloud authentication
- Optional features such as **Seamless Single Sign-On (Seamless SSO)**, **password writeback**, **group writeback**, and **device writeback** depending on the sync platform and design.

A simple flow looks like this:

```text
On-prem AD DS
   |  (sync)
   v
Entra ID Connect / Cloud Sync
   |  (provision users, groups, attributes)
   v
Microsoft Entra ID
   |  (sign-in, conditional access, app access)
   v
Cloud apps / Microsoft 365 / SaaS
```

**Design goals and decision points**

You usually design hybrid identity around four questions:
- **How do users authenticate?**
- **How do identities get provisioned and updated?**
- **Which attributes must remain authoritative on-premises?**
- **Which recovery and migration scenarios must be supported?**

The answer determines whether you use PHS, PTA, or federation, and whether you need writeback or advanced sync features.

**Authentication model selection**

PHS is often the simplest and most resilient cloud authentication option. PTA keeps password validation on-premises while still using Entra ID for the user experience. Federation is more complex and is typically reserved for cases where a specific technical or policy requirement cannot be met by PHS or PTA.

| Option | Where authentication happens | Strengths | Tradeoffs |
|---|---|---|---|
| PHS | Entra ID validates a hash of the password | Simple, resilient, low operational overhead | Password validation is in the cloud |
| PTA | On-premises agent validates the password | Keeps password validation on-premises | Depends on agent availability and network path |
| Federation | External identity provider issues tokens | Supports specialized requirements | Highest complexity and operational burden |

> Federation is not required for compliance by default. Use it only when a specific requirement cannot be satisfied by PHS or PTA.

**Entra ID Connect Sync vs Entra Cloud Sync**

Choose the sync platform based on feature needs and operational model. Entra ID Connect Sync is the more feature-rich option and supports advanced hybrid scenarios. Entra Cloud Sync is lighter weight and easier to deploy for many organizations, but it does not replace every Connect Sync capability.

| Capability | Entra ID Connect Sync | Entra Cloud Sync |
|---|---:|---:|
| User and group synchronization | Yes | Yes |
| Password hash synchronization | Yes | Yes, for supported scenarios |
| Pass-through Authentication | Yes | No |
| Seamless SSO | Yes | No native support |
| Password writeback | Yes | No |
| Group writeback | Yes | No |
| Device writeback | Yes | No |
| Advanced hybrid feature set | Yes | Limited |

Use this distinction carefully in exam scenarios:
- **Cloud Sync can support password hash synchronization** in supported scenarios.
- **Cloud Sync does not provide writeback features** such as password writeback, group writeback, or device writeback.
- **Seamless SSO is a feature of Entra ID Connect Sync**, not Cloud Sync.

**Installation prerequisites and platform considerations**

Before installing Entra ID Connect Sync, validate the host and directory prerequisites. Many lab failures come from missing platform dependencies rather than sync logic.

Common checks include:
- Supported Windows Server version for the sync server.
- Required .NET Framework version installed.
- TLS 1.2 enabled and available for secure communication.
- Domain-joined server with line-of-sight to domain controllers and Entra endpoints.
- Appropriate permissions for the installation account and sync account.
- SQL database choice:
  - **Built-in SQL Server Express** for smaller deployments and labs.
  - **Existing SQL Server** for larger environments or when you need central database management.

Practical guidance:
- Use **SQL Server Express** when the directory size and projected object count fit the supported limits and you want a simpler lab or small deployment.
- Use **full SQL Server** when you need higher scale, existing backup/HA processes, or operational standardization.
- Confirm outbound connectivity to Microsoft Entra endpoints through firewalls and proxies.
- Verify that the server can resolve internal AD DS and external DNS names correctly.

Example validation commands:

```powershell
# Check .NET Framework release value
Get-ItemProperty 'HKLM:\SOFTWARE\Microsoft\NET Framework Setup\NDP\v4\Full' |
  Select-Object Release

# Check TLS 1.2-related SCHANNEL settings
Get-ChildItem 'HKLM:\SYSTEM\CurrentControlSet\Control\SecurityProviders\SCHANNEL\Protocols'

# Confirm domain membership and OS details
Get-CimInstance Win32_ComputerSystem | Select-Object Domain, PartOfDomain
Get-ComputerInfo | Select-Object WindowsProductName, WindowsVersion
```

**Express settings vs custom installation**

Express settings are useful when you want a fast deployment with common defaults. Custom installation is required when you need to control sign-in method, filtering, writeback, staging mode, or advanced topology.

Use **Express settings** when:
- You want a quick lab deployment.
- You are using the default synchronization and sign-in pattern.
- You do not need advanced filtering or special topology choices.

Use **Custom installation** when:
- You need to choose between PHS, PTA, or federation.
- You need to enable Seamless SSO.
- You need OU-based or attribute-based filtering.
- You need staging mode or a secondary sync server.
- You need writeback features or custom connector settings.

**Source anchor and immutable ID**

The **source anchor** is the immutable identifier used to tie an on-premises object to its cloud counterpart. In modern designs, this is commonly based on the on-premises objectGUID or another stable immutable value. The cloud uses this value to maintain object continuity across sync cycles.

Why it matters:
- It prevents duplicate cloud objects when the same on-premises user is re-synced.
- It supports **hard match** during recovery or migration.
- It helps preserve object linkage after reinstallation or topology changes.
- It is critical when you restore, rehome, or re-link identities.

Key scenarios:
- If you delete and recreate an on-premises user, the new object has a different source anchor.
- If you want to reconnect an existing cloud user to a restored on-premises object, you may need a **hard match** using the immutable ID.
- If the source anchor changes unexpectedly, you can create duplicate cloud accounts or break synchronization continuity.

Soft match vs hard match:
- **Soft match** usually relies on matching attributes such as **UPN** and, for mail-enabled objects, **proxyAddresses/SMTP** values.
- **Hard match** uses the immutable ID/source anchor to explicitly link the on-premises object to the existing cloud object.

Example PowerShell pattern for inspecting immutable ID values:

```powershell
# On-premises objectGUID to base64 for immutableId comparison
$guid = (Get-ADUser -Identity 'alice' -Properties objectGUID).objectGUID
[Convert]::ToBase64String($guid.ToByteArray())
```

**Authentication methods in practice**

**Password Hash Synchronization (PHS)**

PHS synchronizes a hash of the on-premises password to Entra ID. Entra ID validates the sign-in directly, which reduces dependency on on-premises infrastructure during authentication.

Why use it:
- Simplest operational model.
- Strong resilience if the on-premises environment is unavailable.
- Works well with Conditional Access and modern authentication.

Validation points:
- Confirm the user can sign in to Entra ID with the synced account.
- Check Entra sign-in logs for successful cloud authentication.
- Verify password changes on-premises are reflected in the cloud after sync.

**Pass-through Authentication (PTA)**

PTA validates the password against on-premises AD DS through an authentication agent. The user still signs in through Entra ID, but the password check is performed on-premises.

Why use it:
- You want password validation to remain on-premises.
- You need a simpler alternative to federation without storing password hashes in the cloud.

Validation points:
- Confirm at least one PTA agent is healthy.
- Test sign-in while monitoring agent status.
- Verify that the on-premises path to domain controllers is available.

**Seamless Single Sign-On (Seamless SSO)**

Seamless SSO reduces prompts for domain-joined devices on the corporate network. It uses Kerberos to obtain a ticket from the on-premises domain and then transparently signs the user in to Microsoft Entra ID.

Important conditions:
- The device must be domain-joined or hybrid joined, depending on the scenario.
- The user must be on the corporate network or connected through a network path that can reach the domain controller for Kerberos.
- Browser and client behavior must support integrated Windows authentication.

Validation steps:
1. Sign in to a domain-joined device with a domain account.
2. Open a browser and navigate to a Microsoft 365 or Entra-protected app.
3. Confirm that the user is signed in without an interactive password prompt.
4. Check the browser and sign-in logs for evidence of integrated authentication.

Common validation clues:
- The browser may show a silent redirect rather than a credential prompt.
- Entra sign-in logs may show the sign-in as coming from a federated or integrated flow depending on the configuration path.
- If Seamless SSO fails, check Kerberos, browser zone settings, and domain connectivity.

> Seamless SSO is not a general Kerberos sign-in to Entra ID. Kerberos is used on-premises to obtain the ticket that enables transparent sign-in.

**Federation scenarios and migration considerations**

Federation uses an external identity provider, such as AD FS, to issue tokens for Entra ID. It adds complexity but may be necessary when a specific technical or policy requirement cannot be met by cloud authentication.

Use federation when:
- You have an application or policy requirement that depends on custom claims issuance.
- You need advanced sign-in behavior that cannot be replicated with PHS or PTA.
- You are in a transition state and must preserve existing federation dependencies temporarily.

Do not use federation just because it seems more secure or more compliant. In many environments, PHS with Conditional Access provides a simpler and more resilient design.

When migrating from federation to PHS or PTA, validate:
- Application dependencies on AD FS claims rules.
- Sign-in policies and Conditional Access behavior.
- Device and browser sign-in flows.
- Legacy protocols or apps that may still depend on federation endpoints.
- User experience for MFA, SSO, and app access.

**Writeback features**

Writeback lets cloud-originated changes flow back to on-premises AD DS. This is important when you need hybrid management of passwords, groups, or devices.

Common writeback features:
- **Password writeback**: lets users reset passwords in the cloud and have them written back to AD DS.
- **Group writeback**: writes Microsoft 365 groups back to AD DS as distribution or mail-enabled objects, depending on the scenario.
- **Device writeback**: writes registered devices back to AD DS for certain hybrid scenarios.

Operational notes:
- These features are generally associated with **Entra ID Connect Sync**, not Cloud Sync.
- Password writeback requires the correct configuration in the sync wizard and the necessary permissions.
- Group and device writeback require additional planning for object placement and permissions.

Validation points after enabling password writeback:
- Perform a password reset in the Entra admin center or self-service password reset flow.
- Confirm the password change is accepted and replicated to AD DS.
- Test sign-in to an on-premises resource with the new password.
- Review sync and event logs for writeback success or failure.

Example validation command for a lab user:

```powershell
# Create a test user, then set a password before enabling the account
New-ADUser -Name "Lab User" -SamAccountName labuser `
  -UserPrincipalName labuser@contoso.com `
  -Path "OU=LabUsers,DC=contoso,DC=com" `
  -AccountPassword (Read-Host -AsSecureString "Enter password") `
  -Enabled $true
```

> In many environments, you must set a password before enabling a new AD user. If the domain policy requires a password, creating an enabled account without one can fail.

**Staging mode and high availability**

A staging server is a secondary Entra ID Connect Sync server that imports and synchronizes data but does not export changes to Entra ID. It is used for failover, testing, and configuration validation.

What staging mode does:
- Imports from AD DS and Entra ID.
- Processes synchronization rules.
- Maintains a ready-to-promote configuration.

What staging mode does not do:
- It does not export changes to Entra ID.
- It does not perform password writeback.
- Because exports are disabled, it also does not perform password hash synchronization to Entra ID.

Why it matters:
- You can validate configuration changes safely.
- You can keep a warm standby server ready for failover.
- You can test upgrades or rule changes before promoting them.

Validation approach:
1. Install a second sync server.
2. Enable staging mode on the secondary server.
3. Confirm it imports and processes objects without exporting.
4. Promote the staging server only when you are ready to take over synchronization.

**Synchronization scope and filtering**

Filtering controls which objects are synchronized. Good filtering reduces clutter, limits cloud object sprawl, and supports least-privilege design.

Common filtering methods:
- **Domain filtering**
- **OU filtering**
- **Attribute-based filtering**
- **Group-based filtering** for targeted lab scenarios, where supported

Design guidance:
- Use OU filtering when your directory structure cleanly separates sync scope.
- Use attribute-based filtering when you need more precise control.
- Avoid overly broad sync scope in labs unless the exercise requires it.

Practical example:
- Sync only `OU=Users,OU=Corp,DC=contoso,DC=com`.
- Exclude service accounts and test OUs unless they are part of the scenario.

**Validation in the Entra admin center**

The Entra admin center provides evidence that sync and sign-in are working. In exam scenarios, you should know what to inspect.

Useful validation areas:
- **Users**: confirm the object exists and has the expected source.
- **Sign-in logs**: inspect authentication method, status, client app, and conditional access result.
- **Audit logs**: confirm password reset, group changes, or directory updates.
- **Identity > Hybrid management**: review sync health and connector status where available.

Fields to inspect in sign-in logs:
- **Status**: success or failure.
- **Authentication requirement**: MFA, single-factor, or passwordless.
- **Authentication method**: PHS, PTA, or federated flow.
- **Conditional Access**: policy applied, grant result, or failure reason.
- **Device info**: joined state, compliance, and browser/client details.

**Troubleshooting workflow**

A disciplined troubleshooting flow helps you isolate whether the issue is in AD DS, sync, authentication, or the cloud service.

1. **Confirm object presence**
   - Check whether the user or group exists in AD DS.
   - Verify the object is in scope for synchronization.
   - Confirm the source anchor has not changed unexpectedly.

2. **Check sync health**
   - Review the sync service manager or synchronization service.
   - Look for import, synchronization, or export errors.
   - Inspect connector status and recent run history.

3. **Inspect connector space and metaverse**
   - Use the synchronization service manager to compare the connector space object with the metaverse object.
   - Confirm joins, projections, and attribute flows are correct.
   - Look for disconnector states or unexpected joins.

4. **Review rule conflicts**
   - Check whether custom sync rules override built-in rules.
   - Validate precedence and scoping filters.
   - Confirm attribute flows are not being blocked.

5. **Validate sign-in path**
   - For PHS, confirm the cloud sign-in succeeds and the password hash is current.
   - For PTA, confirm the agent is healthy and can reach AD DS.
   - For Seamless SSO, confirm Kerberos and browser conditions are met.
   - For federation, verify token issuance and claims.

Common error patterns:
- **Export errors** often indicate permission, duplicate attribute, or object conflict issues.
- **Import errors** often indicate source directory access or schema-related problems.
- **Join errors** often indicate matching or source anchor issues.
- **Authentication failures** often indicate method-specific configuration or network problems.

**Practical lab validation checklist**

Use this checklist to prove the architecture works end to end:

1. Confirm the sync server meets OS, .NET, TLS, and connectivity prerequisites.
2. Install Entra ID Connect Sync or Cloud Sync according to the scenario.
3. Scope the correct OUs or attributes for synchronization.
4. Enable the required sign-in method:
   - PHS for cloud validation
   - PTA for on-premises validation
   - Seamless SSO if the scenario requires silent sign-in
5. Verify the user appears in Entra ID.
6. Test sign-in from a domain-joined device.
7. Review Entra sign-in logs for the expected authentication method.
8. If writeback is enabled, test password reset and confirm on-premises effect.
9. If staging mode is used, confirm the server imports but does not export.
10. If migration is involved, validate soft match or hard match behavior before cutover.

**Common lab pitfalls**

- **Missing password on a new AD user** can prevent a realistic sign-in test.
- **Incorrect source anchor handling** can create duplicate cloud objects.
- **Assuming Cloud Sync supports every Connect Sync feature** can lead to failed design choices.
- **Ignoring browser and device requirements** can make Seamless SSO appear broken.
- **Skipping sign-in logs** can hide the real authentication method or policy failure.
- **Overlooking AD FS dependencies** can break migration from federation to cloud authentication.

**Exam-focused design takeaways**

- Choose **PHS** for simplicity and resilience unless a specific requirement forces another method.
- Choose **PTA** when password validation must remain on-premises.
- Use **federation** only when PHS or PTA cannot satisfy a concrete requirement.
- Use **Entra ID Connect Sync** when you need advanced hybrid features such as Seamless SSO or writeback.
- Use **Cloud Sync** when you want a lighter sync model and do not need the full Connect Sync feature set.
- Treat **source anchor/immutable ID** as the key to object continuity.
- Validate with **sign-in logs, sync logs, and object state** rather than assuming configuration success.
