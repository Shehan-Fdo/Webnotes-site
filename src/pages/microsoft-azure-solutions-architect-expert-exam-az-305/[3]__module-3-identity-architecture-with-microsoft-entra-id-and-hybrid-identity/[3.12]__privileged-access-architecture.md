## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Privileged access architecture overview**

Privileged access architecture is the set of controls you use to protect identities that can change security posture, data access, and platform configuration. In Azure, this usually spans Microsoft Entra ID, Azure RBAC, Privileged Identity Management (PIM), Conditional Access, monitoring, and emergency access design.

The goal is to reduce standing privilege, make elevation time-bound and auditable, and ensure that administrative access is both tightly controlled and recoverable. For AZ-305, you should think in terms of identity, scope, approval, device trust, and operational resilience.

- **Standing access** is persistent privilege that should be minimized.
- **Just-in-time (JIT)** access is activated only when needed and for a limited duration.
- **Least privilege** means granting only the permissions required for the task.

**Core design principles**

A strong privileged access design uses multiple layers rather than a single control. You typically combine role-based access control, conditional access, privileged role activation, and logging so that compromise of one control does not automatically expose the environment.

- Separate day-to-day user accounts from administrative accounts when practical.
- Use role assignments at the narrowest effective scope.
- Prefer time-bound activation over permanent assignment.
- Require strong authentication and device posture for admin actions.
- Keep emergency access available even if normal identity controls fail.

**Microsoft Entra ID privileged access model**

Microsoft Entra ID is the identity control plane for Azure administration. Privileged access in Entra includes directory roles, administrative units, PIM, access reviews, entitlement management, and access packages for onboarding workflows.

PIM is the primary mechanism for reducing standing privilege. It lets you make users eligible for a role and require activation before they can use it. Activation can require MFA, justification, approval, ticket information, and time limits.

- **Eligible assignment** means the user can activate the role when needed.
- **Active assignment** means the role is currently usable.
- **Access review** helps you periodically validate whether access is still needed.
- **Entitlement management** can package access requests and approvals for onboarding or recurring access workflows.

**Privileged Identity Management in practice**

PIM is used for both Entra roles and Azure resource roles. For Entra roles, it protects directory-level administration such as User Administrator, Privileged Role Administrator, or Global Administrator. For Azure resources, it can govern Azure RBAC role assignments such as Owner or User Access Administrator.

A typical privileged workflow is:

1. Assign the user as eligible for the role.
2. Require activation with MFA and justification.
3. Limit activation duration.
4. Send alerts for risky or unusual activations.
5. Review access periodically.

Example activation policy concepts:

```text
Activation requires:
- Multi-factor authentication
- Justification
- Approval for high-impact roles
- Maximum activation duration: 1 hour
- Ticket number for change tracking
```

**Built-in roles versus custom roles**

Built-in roles are convenient and should be your default choice when they already match the task. Use custom roles when built-in roles are too broad or include permissions you do not want to grant.

Built-in roles are appropriate when:
- The role is well understood and widely used.
- The permission set matches the operational need.
- You want simpler administration and easier troubleshooting.

Custom roles are appropriate when:
- You need a narrow permission set for a specific team or automation.
- You want to separate read, write, and delete operations.
- You need to avoid broad permissions such as full resource management.

A least-privilege custom role often starts with a built-in role as a reference, then removes unnecessary actions. For example, a support team might need to restart virtual machines and read diagnostics, but not delete resources or modify networking.

Example custom role definition pattern:

```json
{
  "Name": "VM Operator Limited",
  "IsCustom": true,
  "Description": "Can start, stop, and restart virtual machines, and read diagnostics.",
  "Actions": [
    "Microsoft.Compute/virtualMachines/read",
    "Microsoft.Compute/virtualMachines/start/action",
    "Microsoft.Compute/virtualMachines/restart/action",
    "Microsoft.Compute/virtualMachines/deallocate/action",
    "Microsoft.Insights/diagnosticSettings/read"
  ],
  "NotActions": [
    "Microsoft.Compute/virtualMachines/delete"
  ],
  "AssignableScopes": [
    "/subscriptions/00000000-0000-0000-0000-000000000000"
  ]
}
```

**Scope design and inheritance**

Azure RBAC scope is hierarchical. A role assignment at a management group can flow down to subscriptions, resource groups, and resources beneath it. This is powerful, but it can also create unintended privilege if you assign too high in the hierarchy.

Use the narrowest scope that still supports the operational need:
- **Management group** for platform-wide governance or central operations.
- **Subscription** for environment-level administration.
- **Resource group** for team-owned workloads.
- **Resource** for highly specific operations.

Be careful with inheritance:
- A role assigned at a management group can affect many subscriptions.
- A role assigned at a subscription can still be too broad for a single workload team.
- A resource group assignment is often the best balance for application support teams.

**Administrative units for delegated administration**

Administrative units let you scope certain Entra role administration to a subset of users or groups. This is useful when regional, departmental, or subsidiary administrators should manage only their own population.

Use administrative units when:
- You need delegated user administration without tenant-wide rights.
- You want help desk or HR admins to manage only a subset of identities.
- You want to reduce the blast radius of directory roles.

Administrative units are especially relevant for roles such as:
- User Administrator
- Helpdesk Administrator
- Groups Administrator

They do not replace Azure RBAC for resource management. They are for directory-scoped administration, not subscription or resource permissions.

**Conditional Access for privileged users**

Conditional Access enforces access conditions before authentication succeeds or before a token is issued. For privileged access, it is a critical control because it can require stronger authentication and restrict where and how admins sign in.

Common privileged-user policies include:
- Require MFA for all administrative roles.
- Require compliant or hybrid-joined devices for admin portals.
- Block legacy authentication.
- Restrict access from high-risk sign-in locations.
- Require phishing-resistant authentication for the most sensitive roles where possible.

For privileged access, avoid weak patterns that allow either/or conditions to satisfy access too easily. A stronger posture usually requires multiple controls together, such as compliant device plus MFA, or trusted location plus strong authentication, depending on the risk model.

Example policy intent:
- Admins must use a compliant device.
- Admins must complete MFA.
- Admins must not use legacy protocols.
- Admins may be restricted to named locations for sensitive operations.

> Do not exclude emergency accounts from all monitoring. Exclude them from normal Conditional Access enforcement only when necessary, but keep logging, alerting, and periodic testing in place.

**Break-glass and emergency access accounts**

Break-glass accounts are emergency access identities used when normal administrative access is unavailable. They are a resilience control, not a convenience account.

Design them carefully:
- Use at least two emergency accounts.
- Keep them cloud-only and highly protected.
- Exclude them from Conditional Access policies that could lock out all admins.
- Do not exclude them from logging, alerting, or security monitoring.
- Store credentials securely and rotate them on a defined schedule.
- Test them regularly with a documented recovery runbook.

Practical guidance:
- Use long, unique passwords or stronger authentication methods where supported by your operational model.
- Monitor sign-ins and alert on any use.
- Review whether the accounts are still needed after major identity changes.
- Test recovery procedures on a schedule, such as quarterly or after major identity changes.

A simple operational checklist:

1. Create two emergency accounts.
2. Exclude only those accounts from normal Conditional Access.
3. Enable alerting for any sign-in or role activation.
4. Store credentials in a secure vault or offline sealed process.
5. Test sign-in and recovery procedures regularly.
6. Document who can authorize use and how incidents are escalated.

**Azure RBAC and privileged resource access**

Azure RBAC controls access to Azure resources. Privileged resource roles include Owner, Contributor, User Access Administrator, and many service-specific roles. The most sensitive of these is often Owner because it can manage access and resources.

Use RBAC carefully:
- Prefer Contributor or a narrower custom role over Owner.
- Use User Access Administrator only when a user must manage role assignments.
- Assign roles to groups when possible to simplify lifecycle management.
- Use PIM for eligible role activation on sensitive assignments.

A common pattern is:
- Users are members of a privileged access group.
- The group is eligible for a role via PIM.
- The role is assigned at the smallest practical scope.

This separates identity lifecycle from resource authorization and makes reviews easier.

**Privileged access groups and role-assignable groups**

Role-assignable groups can be assigned Microsoft Entra directory roles. This is useful when you want a group to hold a directory role rather than assigning it directly to many users.

Important distinctions:
- A role-assignable group is an Entra feature for directory roles.
- PIM for groups governs membership or ownership activation for groups.
- Azure RBAC group assignments are separate from Entra directory role assignments.

Use role-assignable groups when:
- You want a stable administrative identity for a team.
- You want to manage membership rather than individual role assignments.
- You need cleaner auditing and easier offboarding.

Use PIM for groups when:
- Membership itself should be time-bound.
- You want users to activate membership before they gain downstream access.
- You want approvals and justification for group membership changes.

**Managed identities for privileged automation**

Managed identities are preferred for Azure automation because they remove the need to store credentials in code or configuration. They are not human identities, but they often hold powerful permissions and therefore need privileged access design.

Use managed identities when:
- An Azure resource needs to call Azure APIs securely.
- You want to avoid secrets, certificates, or client secrets.
- You need identity-based access for automation, functions, runbooks, or apps.

Choose the identity type carefully:
- **System-assigned managed identity** is tied to one resource and is deleted with it.
- **User-assigned managed identity** can be shared across multiple resources and managed independently.

Scope RBAC narrowly:
- Assign only the permissions the workload needs.
- Prefer resource group or resource scope over subscription scope.
- Use separate identities for separate workloads when blast radius matters.

Example Azure CLI pattern for assigning a role to a managed identity:

```bash
az role assignment create \
  --assignee-object-id <managed-identity-principal-id> \
  --assignee-principal-type ServicePrincipal \
  --role "Reader" \
  --scope /subscriptions/<subscription-id>/resourceGroups/<rg-name>
```

For write operations, use a narrower custom role if Reader is insufficient. Avoid granting Owner to automation unless there is a strong, documented reason.

**Service principals and app governance**

Service principals represent applications and automation identities in Entra ID. They can be powerful and are often overlooked in privileged access reviews.

Govern service principals by:
- Using certificates or managed identities instead of client secrets where possible.
- Limiting API permissions and Azure RBAC scope.
- Reviewing app role assignments and directory permissions.
- Rotating secrets and certificates on a schedule.
- Monitoring for unused or overprivileged applications.

A practical governance pattern is:
- Use managed identity for Azure-hosted workloads.
- Use service principals only when the workload cannot use managed identity.
- Store secrets in Azure Key Vault if a secret is unavoidable.
- Review app permissions and ownership regularly.

**Azure Lighthouse and delegated administration**

Azure Lighthouse enables cross-tenant delegated resource management. It is commonly used by managed service providers or central IT teams that administer customer or subsidiary subscriptions without switching tenants.

Key concepts:
- **Delegation registration definition** describes the delegated access.
- **Managed service provider** or managing tenant performs the operations.
- **Customer tenant** retains ownership of the resources.

Use Lighthouse when:
- You need secure cross-tenant administration.
- You want centralized operations across multiple tenants.
- You need delegated access without sharing customer credentials.

Practical considerations:
- Delegate only the roles required for the task.
- Scope delegation to the smallest practical subscription or resource group.
- Review who can manage the delegation definitions.
- Understand whether access is active or eligible in your operational model; this depends on the specific delegated access design and tenant configuration.

Lighthouse does not replace Entra PIM or Azure RBAC. It complements them by enabling delegated administration across tenants, but the exact relationship with PIM depends on how the delegated access is implemented and governed.

**Entitlement management and access packages**

Entitlement management helps you package access into requestable bundles. This is useful for privileged onboarding workflows where users need a defined set of roles, group memberships, and approvals.

Use access packages when:
- A user needs multiple related permissions at once.
- You want a request-and-approval workflow.
- You need time-bound access with periodic revalidation.

A privileged onboarding package might include:
- Membership in a privileged access group.
- Eligibility for a directory role via PIM.
- Access to a support portal or ticketing group.
- Expiration and access review requirements.

This is especially useful when access is temporary, project-based, or tied to a change window.

**Monitoring, detection, and audit**

Privileged access must be monitored continuously. The goal is to detect misuse, unusual activation, and policy drift before they become incidents.

Monitor:
- PIM activation events and role assignment changes.
- Conditional Access failures and bypass attempts.
- Sign-ins for privileged accounts.
- Changes to emergency access accounts.
- Azure Activity Log events for subscription and resource changes.
- Entra audit logs for directory changes.

Integrate with security tooling:
- Microsoft Defender for Cloud can surface security recommendations and posture issues related to Azure resources.
- Microsoft Entra ID Protection can help identify risky users and risky sign-ins where available in your licensing and configuration.
- Microsoft Sentinel or another SIEM can correlate identity and resource events.

Useful alert examples:
- Global Administrator role activated outside business hours.
- Emergency account sign-in from a new location.
- Owner role assigned at subscription scope.
- Conditional Access policy disabled.
- Managed identity granted a broader role than expected.

**Portal-based configuration example**

A common exam scenario is to secure an admin role with PIM and Conditional Access while preserving emergency access.

1. In Microsoft Entra admin center, open **Privileged Identity Management**.
2. Select the target role, such as **User Administrator** or **Global Administrator**.
3. Assign the user as **Eligible** rather than Active.
4. Configure activation requirements:
   - Require MFA
   - Require justification
   - Set activation duration
   - Require approval for sensitive roles
5. Create a Conditional Access policy for admin roles:
   - Target administrative roles
   - Require compliant device
   - Require MFA
   - Block legacy authentication
6. Create two emergency accounts.
7. Exclude only those emergency accounts from the Conditional Access policy.
8. Keep audit logging and alerting enabled for all accounts, including emergency accounts.
9. Test activation and emergency sign-in using a documented runbook.

**CLI examples for common tasks**

Azure CLI is useful for repeatable RBAC operations, but some identity and PIM tasks are easier in the portal or Microsoft Graph. Use the right tool for the job.

Assign a role at resource group scope:

```bash
az role assignment create \
  --assignee user@contoso.com \
  --role "Contributor" \
  --scope /subscriptions/<subscription-id>/resourceGroups/<rg-name>
```

List role assignments at a scope:

```bash
az role assignment list \
  --scope /subscriptions/<subscription-id>/resourceGroups/<rg-name> \
  --output table
```

Create a custom role from a JSON definition:

```bash
az role definition create --role-definition vm-operator-limited.json
```

Assign a custom role:

```bash
az role assignment create \
  --assignee user@contoso.com \
  --role "VM Operator Limited" \
  --scope /subscriptions/<subscription-id>/resourceGroups/<rg-name>
```

Create an Azure Policy assignment with the correct parameter name:

```bash
az policy assignment create \
  --name deny-public-ip \
  --scope /subscriptions/<subscription-id> \
  --policy-definition /providers/Microsoft.Authorization/policyDefinitions/<policy-definition-id>
```

**Exam-focused comparisons**

| Area | Best use | Common pitfall |
|---|---|---|
| PIM for Entra roles | Time-bound directory administration | Leaving Global Administrator permanently active |
| PIM for Azure RBAC | Just-in-time resource access | Assigning Owner broadly at subscription scope |
| Administrative units | Delegated directory administration | Assuming they control Azure resource permissions |
| Role-assignable groups | Group-based directory role assignment | Confusing them with Azure RBAC group assignments |
| Managed identities | Secretless workload access | Granting overly broad subscription permissions |
| Lighthouse | Cross-tenant delegated resource management | Treating it as a replacement for identity governance |

**Common architecture patterns**

A secure enterprise pattern often looks like this:
- Users authenticate with strong MFA.
- Admins use separate privileged accounts.
- Privileged roles are eligible in PIM.
- Conditional Access requires compliant devices for admin portals.
- Emergency accounts are excluded from normal access policies but monitored.
- Managed identities handle automation.
- Access reviews and entitlement management govern lifecycle.

A delegated operations pattern often looks like this:
- Central IT uses Azure Lighthouse for cross-tenant resource administration.
- Local admins are limited by administrative units for directory tasks.
- Subscription-level RBAC is granted through groups, not individuals.
- High-impact roles are activated only when needed.

**What to remember for AZ-305**

AZ-305 questions usually test design tradeoffs rather than memorized commands. Focus on why a control exists and how it reduces risk.

- Use PIM to eliminate standing privilege.
- Use Conditional Access to enforce strong sign-in conditions.
- Use administrative units to scope directory administration.
- Use managed identities for secretless automation.
- Use custom roles when built-in roles are too broad.
- Use break-glass accounts for resilience, not routine administration.
- Use monitoring and access reviews to keep the model trustworthy over time.
