## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Overview**

Microsoft Entra hybrid sign-in design usually centers on three approaches: **password hash synchronization (PHS)**, **pass-through authentication (PTA)**, and **federation**. The right choice depends on where authentication happens, how much trust you place in cloud versus on-premises components, and what availability or compliance requirements you must satisfy.

For exam scenarios, remember the default guidance: **Microsoft generally recommends PHS for most organizations unless a specific requirement prevents it**. PHS is the simplest to operate, has the smallest on-premises attack surface, and gives the best resilience when on-premises identity infrastructure is unavailable. PTA and federation exist for cases where you need on-premises validation or more advanced sign-in control.

**Core comparison**

- **PHS** copies a password-derived hash to Microsoft Entra ID and authenticates in the cloud.
- **PTA** sends the sign-in request to an on-premises agent that validates the password against Active Directory Domain Services (AD DS).
- **Federation** redirects authentication to a federation service such as Active Directory Federation Services (AD FS) or a third-party identity provider.

The main tradeoff is where the trust boundary sits:
- With **PHS**, Microsoft Entra ID becomes the primary authentication authority for the cloud sign-in.
- With **PTA**, the on-premises directory remains the password verifier.
- With **federation**, the federation service becomes the sign-in authority and can enforce custom policies before tokens are issued.

**Security and trust tradeoffs**

PHS reduces password replay risk because Microsoft Entra ID does not store the user’s plaintext password. Instead, it stores a hash of a hash that is used only for verification. The cloud can authenticate users even if the on-premises environment is offline, which lowers operational dependency on local infrastructure.

PTA keeps the password verification process on-premises, which can be attractive when policy requires that passwords are never validated in the cloud. However, the password still traverses the network path to the PTA agent in a protected form during the sign-in flow, so you must trust the agent, the network path, and the on-premises AD DS environment.

Federation creates the largest trust boundary and the most moving parts. It can support advanced claims issuance, step-up logic, and legacy integration, but it also increases attack surface because you must secure the federation servers, proxy components, certificates, and token issuance pipeline. If the federation service is compromised, the impact can be broad because it sits directly in the authentication path.

- **PHS**
  - Lowest operational complexity.
  - Lowest on-premises attack surface.
  - Best cloud resilience.
  - Password replay risk is reduced because the cloud uses a synchronized hash, not the original password.
- **PTA**
  - Password validation stays on-premises.
  - More dependency on local agents and network connectivity.
  - Better fit when policy forbids password verification in the cloud.
  - Sign-in availability depends on agent health and redundancy.
- **Federation**
  - Highest flexibility for claims and custom rules.
  - Highest infrastructure and certificate management overhead.
  - Broadest attack surface.
  - Often used only when a specific requirement demands it.

> If a question asks for the default choice in a modern hybrid design, choose **PHS** unless the scenario explicitly requires on-premises authentication, custom claims, or a legacy federation dependency.

**Password hash synchronization**

PHS synchronizes a password hash from on-premises AD DS to Microsoft Entra ID through Microsoft Entra Connect or Microsoft Entra Cloud Sync. The synchronized value is not the user’s actual password and cannot be reversed into it. During sign-in, Microsoft Entra ID validates the password against the stored hash and issues the token directly.

This model is usually the best fit for:
- Most new hybrid deployments.
- Organizations that want the simplest sign-in architecture.
- Disaster recovery designs that must survive on-premises outages.
- Environments that want to reduce dependence on federation infrastructure.

**Why PHS is preferred**

PHS is preferred because it:
- Removes the need for a live authentication hop to on-premises infrastructure.
- Simplifies high availability because Microsoft Entra ID handles authentication.
- Supports features such as **Seamless SSO** and **password writeback** in hybrid scenarios.
- Reduces the number of components that can fail during sign-in.

A practical sign-in flow looks like this:
1. The user enters credentials on the Microsoft Entra sign-in page.
2. Microsoft Entra ID checks the synchronized password hash.
3. If the password is valid, Microsoft Entra ID issues the token.
4. Optional features such as Conditional Access then evaluate the session.

**PHS configuration and verification**

In Microsoft Entra Connect, you typically enable PHS during initial setup or by changing the sign-in method later.

Common setup path:
1. Open **Microsoft Entra Connect** on the sync server.
2. Select **Configure**.
3. Choose **Change user sign-in**.
4. Select **Password Hash Synchronization**.
5. Complete the wizard and verify synchronization.

In the Microsoft Entra admin center, you can also review the sign-in method and related hybrid identity settings:
- Go to **Microsoft Entra admin center**.
- Open **Microsoft Entra ID**.
- Navigate to **Microsoft Entra Connect** or **Hybrid management** areas depending on portal layout.
- Verify the configured sign-in method and synchronization status.

Useful validation commands and checks:
```powershell
# Check Entra Connect sync status on the server
Get-ADSyncScheduler

# Force a delta sync
Start-ADSyncSyncCycle -PolicyType Delta

# Force a full sync if required
Start-ADSyncSyncCycle -PolicyType Initial
```

If password sync is delayed, users may temporarily authenticate with an old password in the cloud until the next sync cycle completes. This is an exam-relevant edge case: password changes are not always instantaneous across all systems, so you should expect a short propagation delay.

**PHS high availability and outage behavior**

PHS is highly resilient because Microsoft Entra ID performs the authentication. If the on-premises sync server fails, existing synchronized hashes remain usable for sign-in until the next password change or sync update is needed. If Microsoft Entra experiences a service issue, cloud sign-in is affected regardless of the on-premises state.

Design guidance:
- Use more than one Microsoft Entra Connect server only for staging and recovery, not active-active sync.
- Protect the sync server because it has privileged access to directory data.
- Monitor sync health and password sync status.
- Plan for recovery of the sync engine, not for authentication continuity, because authentication itself is cloud-hosted.

**Pass-through authentication**

PTA validates the user’s password against on-premises AD DS in real time. Microsoft Entra ID receives the sign-in request, then forwards it to a PTA agent installed on a domain-joined server. The agent decrypts the request in a secure channel and checks the password against AD DS. If validation succeeds, the agent returns the result to Microsoft Entra ID, which issues the token.

PTA is useful when:
- You want cloud sign-in but must keep password validation on-premises.
- You need to avoid storing password hashes in the cloud.
- You want a simpler alternative to federation while preserving on-premises authentication.

**PTA configuration and agent flow**

Typical setup path:
1. Open **Microsoft Entra Connect**.
2. Select **Configure**.
3. Choose **Change user sign-in**.
4. Select **Pass-through Authentication**.
5. Enable the feature and sign in with a Microsoft Entra hybrid identity administrator account.
6. Register the PTA agent with the tenant.
7. Install additional agents for redundancy.

Agent registration flow:
- The agent is installed on a Windows Server joined to the domain.
- During setup, you sign in to Microsoft Entra ID to register the agent.
- The agent establishes outbound-only connectivity to Microsoft Entra, so no inbound firewall ports are required for the authentication path.
- Multiple agents can be deployed for load distribution and failover.

Verification points:
- In the Microsoft Entra admin center, check the **Pass-through authentication** status and agent health.
- Confirm the agent appears as **active** and healthy.
- Review sign-in logs for PTA-related failures or agent timeouts.

Example PowerShell checks on the server:
```powershell
# Verify the PTA agent service
Get-Service "Microsoft Azure AD Connect Authentication Agent"

# Review recent service status
Get-WinEvent -LogName Application -MaxEvents 20 |
  Where-Object {$_.ProviderName -like "*Authentication Agent*"}
```

**PTA security and availability tradeoffs**

PTA reduces cloud credential exposure because the password is not validated in Microsoft Entra ID. However, it increases dependence on on-premises infrastructure and the PTA agent tier. The attack surface is smaller than federation but larger than PHS because the authentication path still depends on local servers.

Availability considerations:
- If one PTA agent fails, another agent can take over.
- If all PTA agents or the on-premises AD DS environment are unavailable, users cannot authenticate with PTA.
- If Microsoft Entra is unavailable, sign-in is also affected because the cloud front end still brokers the request.
- Redundancy should include multiple agents on separate servers and resilient network connectivity.

A practical design pattern is to deploy at least two PTA agents in different fault domains. This protects against a single server failure, but it does not remove the dependency on the on-premises directory or the cloud sign-in service.

**Federation**

Federation delegates authentication to a separate identity provider. The classic Microsoft example is **AD FS**, but exam questions may also reference third-party federation services or modern federation patterns that use another security token service (STS). The key idea is that Microsoft Entra redirects the user to the federation authority, which authenticates the user and issues a token or assertion that Microsoft Entra trusts.

Federation is appropriate when you need:
- Custom claims issuance.
- Complex sign-in rules.
- Legacy application compatibility.
- Existing investment in a federation platform.
- Integration with a non-Microsoft identity provider.

**Federation alternatives**

Although AD FS is the most familiar Microsoft-native option, federation can also involve:
- Third-party identity providers that support SAML, WS-Federation, or OpenID Connect.
- Cloud identity platforms acting as the primary STS.
- Modern federation patterns where the external IdP handles authentication and Microsoft Entra consumes the resulting assertion.

For architecture questions, focus on the pattern rather than the vendor:
- Microsoft Entra redirects the user.
- The external IdP authenticates the user.
- The external IdP returns a signed assertion or token.
- Microsoft Entra trusts that assertion and completes sign-in.

**Federation configuration and verification**

In Microsoft Entra Connect, federation is typically configured through the sign-in method wizard when you choose federation with AD FS or another supported path.

Common setup path:
1. Open **Microsoft Entra Connect**.
2. Select **Configure**.
3. Choose **Change user sign-in**.
4. Select **Federation with AD FS** or the relevant federation option.
5. Provide federation service details and certificates.
6. Complete trust configuration and verify token issuance.

In the Microsoft Entra admin center, verify:
- The domain is marked as federated.
- The federation metadata and certificates are current.
- Sign-in logs show redirects to the federation endpoint.

Useful AD FS-side checks:
```powershell
# View relying party trust details
Get-AdfsRelyingPartyTrust

# Check certificate expiration
Get-AdfsCertificate -CertificateType Token-Signing, Token-Decrypting
```

**Federation security and availability tradeoffs**

Federation has the widest attack surface because it introduces federation servers, proxies, certificates, and trust relationships. It also creates the most complex failure modes. If the federation service is down, users cannot sign in unless you have a carefully designed fallback or alternate path.

Compared with PHS and PTA:
- Federation exposes more infrastructure to harden and monitor.
- Federation can support richer policy control.
- Federation often requires more operational expertise.
- Federation can be harder to recover during outages because multiple components must be healthy.

**User experience differences**

The user experience varies noticeably across the three methods.

With **PHS**:
- Users usually sign in directly on the Microsoft Entra page.
- Internal users may get a seamless experience if **Seamless SSO** is enabled.
- External users typically see the standard cloud sign-in prompt.
- Hybrid-joined devices can often authenticate with fewer prompts, especially when device state and SSO are configured correctly.

With **PTA**:
- Users still sign in through Microsoft Entra, but the password is validated on-premises.
- The prompt usually looks similar to PHS from the user’s perspective.
- The difference is mostly invisible unless there is a delay or failure.
- If the PTA path is unhealthy, users may see sign-in failures or retries.

With **federation**:
- Users are often redirected to the federation service sign-in page.
- Internal users may get integrated authentication if the federation service supports it.
- External users usually see a redirect and then a separate credential prompt.
- The experience can be more complex, especially with MFA, claims rules, or custom branding.

**Seamless SSO**

**Seamless SSO** lets domain-joined devices automatically sign in to Microsoft Entra using the user’s existing Windows session. It is commonly paired with PHS, though it can also be used in some hybrid sign-in designs to reduce prompts.

Important points:
- It improves internal user experience.
- It does not replace authentication; it reduces repeated prompts.
- It is especially useful for corporate network or VPN-connected devices.
- It depends on proper browser and device configuration.

Example browser behavior:
- On a domain-joined device inside the corporate network, the user may not see a password prompt.
- On an external device, the user usually sees the normal sign-in page.
- On a hybrid-joined device, the experience can be even smoother if device registration and SSO are healthy.

**Licensing and prerequisites**

Some hybrid identity features depend on licensing or specific prerequisites. Exact licensing can vary by feature and tenant agreement, so verify current Microsoft licensing guidance before implementation.

Common prerequisites to remember:
- **Microsoft Entra Connect** or **Microsoft Entra Cloud Sync** for synchronization.
- Domain-joined or hybrid-joined devices for Seamless SSO scenarios.
- Proper network connectivity from sync servers and PTA agents to Microsoft Entra endpoints.
- Administrative permissions to configure sign-in methods and domain federation.

Feature-specific notes:
- **Password writeback** is commonly used with self-service password reset (SSPR) in hybrid environments and requires the appropriate identity protection or premium licensing.
- **SSPR** in hybrid environments often requires password writeback so cloud password resets can update on-premises AD DS.
- **Seamless SSO** requires configuration in the sync tool and browser support on client devices.

**Password writeback and SSPR**

Password writeback lets a password reset in Microsoft Entra flow back to on-premises AD DS. This is important in hybrid environments because users expect one password to work everywhere.

Why it matters:
- It reduces help desk calls.
- It keeps cloud and on-premises passwords aligned.
- It supports self-service password reset workflows.

Example operational flow:
1. The user resets the password in Microsoft Entra.
2. Microsoft Entra validates the reset policy.
3. The new password is written back to on-premises AD DS.
4. The user can sign in to both cloud and on-premises resources with the new password.

If writeback is not enabled, a cloud password reset may not update on-premises credentials, which can create sign-in confusion.

**Troubleshooting and sign-in logs**

Sign-in logs are essential for diagnosing all three methods. They show whether the sign-in was handled by cloud validation, PTA, or federation, and they help you identify where the failure occurred.

What to look for:
- **PHS**: cloud authentication failures, invalid password, or sync-related delays.
- **PTA**: agent timeouts, agent unavailable, or on-premises AD DS validation failures.
- **Federation**: redirect failures, token issuance errors, certificate problems, or claims rule issues.

Practical troubleshooting approach:
1. Check Microsoft Entra sign-in logs for the authentication method and error code.
2. Verify the sync or agent health on the on-premises server.
3. Confirm network connectivity and certificate validity.
4. Test with a known-good account and compare behavior across methods.

Useful commands and checks:
```powershell
# Check Entra Connect sync health
Get-ADSyncScheduler

# Test DNS and connectivity to Microsoft Entra endpoints
Test-NetConnection login.microsoftonline.com -Port 443

# Review AD FS service health if federated
Get-Service adfssrv
```

**Exam-oriented edge cases**

- If password sync is delayed, users may authenticate with the old password until the next sync completes.
- If a user changes a password on-premises and PHS is enabled, the new hash must sync before cloud sign-in reflects the change.
- If PTA agents are down, sign-in fails even though the password is still in AD DS.
- If the federation service is down, sign-in fails unless an alternate path exists.
- If Microsoft Entra is unavailable, PHS and PTA both fail because the cloud front end is part of the sign-in path.
- If on-premises AD DS is unavailable, PHS can still authenticate users, but PTA and federation that depend on on-premises validation may fail.
- Account lockout behavior depends on the method and the source of validation; sign-in logs and on-premises security logs help determine where the lockout occurred.

**How to choose the right method**

Use this decision logic in architecture questions:
- Choose **PHS** when you want the simplest, most resilient, and Microsoft-recommended default option.
- Choose **PTA** when you must validate passwords on-premises but do not need federation features.
- Choose **federation** when you need custom claims, legacy integration, or a specific external identity provider pattern.

A concise rule of thumb:
- **PHS** for most modern hybrid deployments.
- **PTA** for on-premises password validation without federation complexity.
- **Federation** only when a requirement explicitly demands it.

**Quick comparison**

| Feature | PHS | PTA | Federation |
|---|---|---|---|
| Password verification location | Microsoft Entra ID | On-premises AD DS | Federation service / IdP |
| On-premises dependency | Low | Medium | High |
| Attack surface | Lowest | Moderate | Highest |
| Password replay risk | Lower | Lower than federation, but password still traverses auth path | Depends on federation design; broader token/claims surface |
| Availability during on-prem outage | Best | Poor | Poor unless special fallback exists |
| User prompts | Usually minimal | Similar to PHS | Often redirect-based |
| Best use case | Default modern hybrid | On-prem validation requirement | Custom claims or legacy federation |

**Key takeaways**

- Microsoft generally prefers **PHS** for most scenarios.
- **PTA** keeps password validation on-premises but adds agent and network dependency.
- **Federation** offers the most flexibility and the most complexity.
- High availability planning differs: PHS relies on Microsoft Entra, PTA relies on multiple agents and AD DS, and federation relies on redundant federation infrastructure.
- Troubleshoot with **sign-in logs**, sync health, agent health, and federation service status.
