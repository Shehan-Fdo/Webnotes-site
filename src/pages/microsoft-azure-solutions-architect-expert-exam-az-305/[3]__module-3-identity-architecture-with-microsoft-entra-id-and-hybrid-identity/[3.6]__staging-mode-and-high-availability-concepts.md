## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Staging mode in Azure App Service**

Staging mode usually means using an App Service deployment slot to validate a new build before it serves production traffic. A slot is a live app endpoint with its own hostname, configuration surface, and deployment history. You use it to reduce release risk by testing in an environment that is close to production.

Slots are most useful when you want:
- **Zero-downtime releases** by swapping validated code into production.
- **Production-like testing** with real app settings, connection strings, and scale characteristics.
- **Rollback speed** by swapping back to the previous slot if a release fails.

A common pattern is `production` plus `staging`, where you deploy to staging, warm it up, validate it, and then swap it into production.

**Deployment slot behavior**

Slots are not identical copies in every respect. Some settings move with the content during a swap, while others stay with the slot. This distinction is critical for exam scenarios and real deployments.

- **Swapped settings** typically include:
  - App content
  - Non-sticky app settings
  - Non-sticky connection strings
  - Handler mappings and some platform settings, depending on the app type
- **Sticky settings** stay with the slot:
  - Settings marked as slot-specific
  - Connection strings marked as slot-specific
  - Custom domain bindings
  - TLS/SSL certificates in many scenarios
  - Certain managed identity and networking-related settings, depending on feature support

You mark a setting as slot-specific so it does not move during a swap. This is important for values such as:
- Test database connection strings
- Feature flags for pre-production validation
- Diagnostic endpoints
- Temporary credentials used only in staging

In the Azure portal, you can mark app settings and connection strings as deployment slot settings. In infrastructure as code, you usually model this by keeping environment-specific values separate and injecting them per slot.

**Warm-up and readiness before swap**

A swap is not just a file move. Azure prepares the target slot, warms it up, and then routes traffic when the app is ready. Warm-up matters because many apps need time to initialize caches, compile views, establish database connections, or load secrets.

Key readiness concepts:
- **Warm-up path**: a URL path Azure can call to verify the app is ready.
- **Health checks**: application endpoints that confirm the app is healthy before traffic is shifted.
- **ARR affinity**: Application Request Routing cookie-based session affinity can affect how users are pinned to instances during validation and after swap.

For apps that need explicit warm-up behavior, configure a warm-up endpoint that returns success only when the app can safely receive traffic. A typical pattern is a lightweight endpoint such as `/health` or `/ready`.

```json
{
  "siteConfig": {
    "healthCheckPath": "/health",
    "alwaysOn": true
  }
}
```

Practical considerations:
- Use `alwaysOn` for production-like slots so the app stays warm.
- Avoid long startup work in the request path.
- Ensure the warm-up endpoint does not depend on user session state.
- Validate that the slot can reach downstream dependencies before swap.

> If your app uses background jobs, a swap can cause duplicate processing if both slots run the same scheduled work. Disable or isolate job execution in staging.

**Slot swap workflow**

A safe release flow usually follows this sequence:

1. Deploy the new build to the staging slot.
2. Run smoke tests against the staging hostname.
3. Warm up the app and verify dependencies.
4. Swap staging into production.
5. Monitor logs, metrics, and error rates.
6. Swap back if rollback is needed.

Azure CLI example for creating a slot and cloning configuration from production:

```bash
az webapp deployment slot create \
  --resource-group rg-apps \
  --name mywebapp \
  --slot staging \
  --configuration-source mywebapp
```

Swap staging into production:

```bash
az webapp deployment slot swap \
  --resource-group rg-apps \
  --name mywebapp \
  --slot staging
```

If you want to swap between two non-production slots, specify the source slot and the target slot explicitly according to the supported command form for your CLI version and scenario. For exam purposes, remember that production is often the implicit target when you swap from a staging slot.

Swap with preview is useful when you want to validate the target configuration before the final traffic move. It is not universal for every setting or app behavior, so do not assume every dependency can be previewed safely.

**Slot limitations and operational risks**

Slots are powerful, but they are not a cure-all.

- **Local cache** can complicate swaps because cached content may not match the newly swapped code.
- **Session affinity** can hide issues during testing because users may stay pinned to a specific instance.
- **Background jobs** may run in both slots unless you explicitly disable them in staging.
- **Database migrations** can break rollback if the schema change is not backward compatible.
- **Static content** and CDN behavior may lag behind the swap if caching layers are involved.

A good release strategy keeps schema changes backward compatible so the old slot can still run after a rollback. That is especially important when you use blue-green style deployments.

**High availability fundamentals**

High availability (HA) is about keeping a service available during component failures. In Azure architecture, HA usually means removing single points of failure and designing for automatic recovery within a region or across regions.

Core HA building blocks:
- **Redundancy**: duplicate critical components.
- **Fault domains**: separate hardware failure boundaries.
- **Update domains**: separate maintenance boundaries.
- **Health probes**: detect unhealthy instances and route around them.
- **Autoscale**: add capacity when demand increases or when instances fail.
- **Zone redundancy**: spread resources across availability zones where supported.

HA is not the same as disaster recovery (DR). HA focuses on minimizing downtime from localized failures. DR focuses on restoring service after a major outage, often in another region.

**Availability zones and zone-redundant design**

Availability zones are physically separate datacenters within an Azure region. They have independent power, cooling, and networking. Use them when you need resilience against datacenter-level failures.

Important distinctions:
- **Availability zone**: the physical isolation boundary.
- **Zone-redundant service**: the service itself spans zones.
- **Zone-redundant deployment**: you place your own resources across zones.

Examples:
- A zone-redundant Azure SQL Database deployment uses platform-managed replication across zones.
- A VM-based application can be deployed across multiple zones with separate VMs, load balancing, and replicated data.
- Some services support zone redundancy natively; others require you to architect it yourself.

> Do not assume every Azure service supports zones. Check whether the service, SKU, and region support zone redundancy before designing around it.

**App Service high availability**

Azure App Service provides built-in platform availability, but your design still matters. A single instance is not enough for production HA because it creates a scale and failure bottleneck.

Use these patterns:
- **Multiple instances** to survive instance failure.
- **Autoscale** to handle load spikes and instance loss.
- **Zone redundancy** where supported by the App Service plan and region.
- **Health checks** to remove unhealthy instances from rotation.
- **Deployment slots** to reduce release-related downtime.

Practical guidance:
- Run at least two instances for production workloads that need higher availability.
- Use autoscale rules based on CPU, memory, HTTP queue length, or custom metrics.
- Keep startup time low so replacement instances become ready quickly.
- Use health check endpoints that verify critical dependencies.

Example autoscale rule concept:
- Scale out when CPU exceeds 70% for 10 minutes.
- Scale in when CPU drops below 40% for 10 minutes.
- Keep a minimum of 2 instances.

**Azure Load Balancer health probes**

Azure Load Balancer uses health probes to determine whether a backend instance is healthy. For Standard Load Balancer, probes can be TCP, HTTP, or HTTPS depending on the configuration and scenario. The probe checks whether the backend can accept traffic and helps the load balancer route around failures.

Typical use cases:
- **TCP probe** for port-level reachability.
- **HTTP/HTTPS probe** for application-level health.
- **Failover** between backend instances when one becomes unhealthy.

Example CLI for a Standard Load Balancer probe:

```bash
az network lb probe create \
  --resource-group rg-network \
  --lb-name lb-prod \
  --name probe-http \
  --protocol Http \
  --port 80 \
  --path /health
```

Use a probe path that returns a fast, deterministic success response. Avoid endpoints that depend on slow downstream calls unless you intentionally want dependency failure to remove the instance from service.

**Application Gateway health probes**

Application Gateway performs layer 7 health probing and is often used for web applications that need path-based routing, TLS termination, and WAF (Web Application Firewall) integration. Probes should use a valid host header or hostname that the application expects.

A probe is not just a ping. It checks whether the backend responds correctly to HTTP or HTTPS requests.

Example CLI for a probe:

```bash
az network application-gateway probe create \
  --resource-group rg-network \
  --gateway-name appgw-prod \
  --name probe-web \
  --protocol Http \
  --host www.contoso.com \
  --path /health \
  --interval 30 \
  --timeout 30 \
  --threshold 3
```

Use a hostname that matches the backend application’s expected host header. Do not use an IP address when the app requires a hostname-based virtual host.

**Traffic Manager and Front Door failover**

Azure Traffic Manager and Azure Front Door both help with global routing, but they solve different problems.

- **Traffic Manager** is DNS-based routing.
- **Front Door** is edge-based HTTP/HTTPS routing with global acceleration and health-based failover.

Traffic Manager routing methods:
- **Priority**: active-passive failover. Primary endpoint receives traffic unless it fails health checks.
- **Weighted**: active-active or gradual traffic shifting. Traffic is distributed by weight.
- **Performance**: routes users to the lowest-latency healthy endpoint.
- **Geographic**: routes based on user location and policy.

Front Door behavior:
- Uses health probes to detect unhealthy origins.
- Can fail over to a secondary origin group automatically.
- Supports active-active patterns with traffic splitting.
- Works well for global web apps that need low latency and fast failover.

Use cases:
- **Priority routing** for regional failover.
- **Weighted routing** for blue-green or canary releases.
- **Performance routing** for globally distributed user bases.

**Active-active and active-passive architectures**

Active-active means multiple regions or instances serve traffic at the same time. Active-passive means one site serves traffic while another waits as standby.

Active-active benefits:
- Better utilization of resources
- Lower failover time
- Higher resilience to regional issues

Active-active tradeoffs:
- More complex data synchronization
- Higher cost
- Harder session management

Active-passive benefits:
- Simpler operational model
- Easier data consistency
- Lower steady-state cost

Active-passive tradeoffs:
- Standby capacity may sit idle
- Failover can take longer
- You must test failover regularly

Data implications matter:
- Stateless web tiers are easiest to make active-active.
- Stateful systems need replication, conflict handling, or a single write region.
- Databases often use geo-replication, failover groups, or application-level replication.

Example pattern:
- Web tier: active-active across two regions.
- Data tier: primary write region with read replicas or failover group.
- Cache: regional cache with graceful rebuild on failover.
- Storage: geo-redundant storage for durability and regional recovery.

**Azure SQL Database availability and failover**

Azure SQL Database includes built-in high availability as part of the service. You do not build it the same way you would build HA for a VM-hosted database. The platform maintains redundancy for the database service and handles many failure scenarios automatically.

For broader resilience:
- Use **zone redundancy** where supported.
- Use **failover groups** for cross-region disaster recovery.
- Use **active geo-replication** when you need readable secondary databases or custom failover control.

Important distinction:
- Built-in HA protects against local infrastructure failures.
- Geo-replication and failover groups protect against regional outages.

Example failover group concept:
- Primary database in `eastus`
- Secondary database in `westus`
- Application connection string points to the failover group listener
- Failover switches the listener to the secondary region

**Virtual machines and SLA considerations**

A single VM can be highly available only under specific conditions, and the SLA depends on the deployment model and redundancy choices. Do not assume every single VM has the same availability guarantee.

General guidance:
- A lone VM is a single point of failure unless you add redundancy.
- Availability sets improve resilience against host and maintenance failures.
- Availability zones improve resilience against datacenter failures.
- Multiple VMs behind a load balancer improve service continuity.

SLA thinking:
- More redundancy usually improves the service-level agreement.
- The exact SLA depends on the service, SKU, and deployment pattern.
- For exam questions, compare architectures by their failure domains rather than by oversimplified “single VM vs set vs zone” statements.

**Storage redundancy and availability**

Storage design affects both HA and DR.

Common redundancy options:
- **LRS**: Locally redundant storage, copies data within one datacenter.
- **ZRS**: Zone-redundant storage, copies data across zones in a region.
- **GRS**: Geo-redundant storage, copies data to a paired region.
- **GZRS**: Geo-zone-redundant storage, combines zone and geo redundancy.

Use cases:
- **LRS** for lower cost and noncritical workloads.
- **ZRS** for regional zone resilience.
- **GRS/GZRS** for regional disaster recovery.

Choose storage redundancy based on:
- Recovery time objective (RTO)
- Recovery point objective (RPO)
- Cost tolerance
- Application write patterns

**Backup and disaster recovery services**

Azure Backup and Azure Site Recovery are often part of a complete resilience design.

- **Azure Backup** protects against accidental deletion, corruption, and ransomware recovery scenarios.
- **Azure Site Recovery** replicates workloads to another region for failover and DR.
- **Failover groups** provide managed database failover for supported Azure SQL scenarios.

Use them together when the architecture needs both operational recovery and regional recovery. For example:
- App Service with deployment slots for release safety
- Azure SQL failover groups for database DR
- Azure Backup for long-term retention
- Azure Site Recovery for VM-based workloads

**SLA, RTO, and RPO**

These terms often appear together in AZ-305 questions.

- **SLA**: Service Level Agreement, the availability commitment.
- **RTO**: Recovery Time Objective, how long recovery can take.
- **RPO**: Recovery Point Objective, how much data loss is acceptable.

How redundancy affects them:
- More instances and zones usually improve SLA.
- Faster failover reduces RTO.
- More frequent replication reduces RPO.
- Cross-region designs usually improve DR but may increase complexity and cost.

A practical way to think about it:
- If you need near-zero downtime, use active-active or zone-redundant designs.
- If you can tolerate a short outage, active-passive with automated failover may be enough.
- If you can tolerate data loss, asynchronous replication may be acceptable.
- If you cannot tolerate data loss, use synchronous or near-synchronous replication where supported.

**End-to-end release pattern example**

A common production pattern for a web app looks like this:

1. Deploy code to the staging slot.
2. Run integration tests against the staging hostname.
3. Verify app settings, connection strings, and feature flags.
4. Confirm health check and warm-up success.
5. Swap staging into production.
6. Monitor telemetry and error rates.
7. Roll back by swapping back if needed.

Example app settings strategy:
- `APP_ENV=Production` in production slot
- `APP_ENV=Staging` in staging slot
- `ConnectionStrings:MainDb` marked as slot-specific
- `FeatureFlags:NewCheckout=true` only in staging

Example Azure CLI to set a slot-specific app setting:

```bash
az webapp config appsettings set \
  --resource-group rg-apps \
  --name mywebapp \
  --slot staging \
  --settings FeatureFlags__NewCheckout=true
```

Example of a health endpoint in ASP.NET Core:

```csharp
app.MapGet("/health", () => Results.Ok(new { status = "healthy" }));
```

**Exam-focused design guidance**

When you evaluate a scenario, ask:
- Is the goal release safety, regional resilience, or both?
- Is the workload stateless or stateful?
- Can the service support zones natively?
- Do you need active-active or active-passive failover?
- What are the RTO and RPO targets?
- Which settings must remain slot-specific?
- Can the app warm up safely before traffic shifts?

Good AZ-305 answers usually align the service choice with the failure domain:
- Use deployment slots for release management.
- Use multiple instances and autoscale for instance-level resilience.
- Use availability zones for datacenter-level resilience.
- Use Traffic Manager or Front Door for regional routing.
- Use Backup, Site Recovery, and failover groups for DR.

**Common pitfalls**

- Treating a deployment slot as a separate environment with no shared dependencies.
- Swapping an app before validating warm-up and health checks.
- Forgetting to mark sensitive or environment-specific settings as slot-specific.
- Using session state that breaks when traffic shifts.
- Assuming every Azure service supports availability zones.
- Confusing HA with DR.
- Designing active-active without a data synchronization plan.
- Ignoring the impact of caching, background jobs, and schema changes on rollback.

**Quick comparison**

| Concept | Primary goal | Typical scope | Example |
|---|---|---|---|
| Deployment slot | Safe release | App Service | Staging to production swap |
| Availability set | Host failure resilience | Single region | Multiple VMs across fault domains |
| Availability zone | Datacenter resilience | Single region | VMs in zones 1, 2, and 3 |
| Traffic Manager | DNS failover | Global | Priority routing to secondary region |
| Front Door | Global HTTP failover | Global edge | Active-active web routing |
| Azure Backup | Recovery from data loss | Workload/data | Restore deleted VM or files |
| Site Recovery | Disaster recovery | Cross-region | Fail over VMs to another region |

**Key takeaways**

- Deployment slots reduce release risk, but you must manage sticky settings, warm-up, and rollback behavior carefully.
- High availability is about surviving failures with minimal downtime, while disaster recovery is about restoring service after larger outages.
- Azure design choices should match the failure domain: instance, host, zone, region, or global edge.
- For AZ-305, always connect the service feature to the business requirement, especially RTO, RPO, and SLA.
