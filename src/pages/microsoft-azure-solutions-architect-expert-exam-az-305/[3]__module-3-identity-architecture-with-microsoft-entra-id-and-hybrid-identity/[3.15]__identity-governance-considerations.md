## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Identity governance in Azure**

Identity governance is the set of controls you use to manage *who* can get access, *how* they get it, *when* they use it, and *how long* that access remains valid. In Azure and Microsoft Entra, this usually spans privileged access, access lifecycle, access reviews, entitlement management, and policy enforcement.

For architecture decisions, focus on three outcomes:

- **Least privilege**: users and workloads receive only the access they need.
- **Just-in-time access**: elevated access is activated only when required.
- **Continuous validation**: access is reviewed and removed when no longer justified.

Identity governance does not replace network security, application authorization, or data protection. It complements them by reducing standing access and improving auditability.

**Core building blocks**

Microsoft Entra provides several governance capabilities that often work together:

- **Microsoft Entra ID PIM**: Privileged Identity Management for eligible role assignment and time-bound activation.
- **Access reviews**: periodic checks to confirm whether users still need access.
- **Entitlement management**: access packages and lifecycle workflows for requesting and approving access.
- **Conditional Access**: policy enforcement based on user, device, location, risk, and app context.
- **Administrative units**: scoped administration for subsets of users and groups.
- **Privileged access groups**: time-bound or eligible membership for groups that grant elevated permissions.

A useful design pattern is to combine these controls rather than rely on one feature alone. For example, you can make a user eligible for a privileged role, require MFA (multi-factor authentication) at activation, and then review that eligibility on a schedule.

**Privileged Identity Management**

PIM helps you reduce standing privilege by separating *eligibility* from *activation*.

- **Eligible assignment**: the user can activate the role when needed.
- **Active assignment**: the user has immediate access without activation.
- **Activation controls**: you can require justification, MFA, approval, ticket information, or a maximum activation duration.

PIM is commonly used for:

- Azure resource roles such as Owner, Contributor, and User Access Administrator.
- Microsoft Entra directory roles such as Global Administrator or Privileged Role Administrator.
- Privileged access groups, where group membership can be eligible and time-bound.

PIM is not a replacement for all privileged access controls. It governs privileged role and group activation, but you still need other controls for application permissions, workload identities, break-glass accounts, and resource-specific authorization patterns.

> PIM controls activation and eligibility. It does not automatically govern every privileged path in the tenant.

**Typical PIM design considerations**

- Use **eligible** assignments for administrators who do not need permanent elevation.
- Keep a small number of **permanent** assignments for break-glass or operational continuity, and protect them heavily.
- Require **MFA** for activation when the role is sensitive.
- Use **approval workflows** for high-impact roles.
- Set **activation duration** to the shortest practical window.
- Log and monitor activation events for audit and incident response.

**Example: Azure CLI for role assignment**

Azure CLI can create a standard Azure RBAC role assignment, but that is not the same as a PIM eligible assignment. Use it when you need a permanent assignment or when you are automating RBAC outside PIM.

```bash
az role assignment create \
  --assignee user@contoso.com \
  --role Reader \
  --scope /subscriptions/<subscription-id>
```

For PIM, configure eligibility and activation through Microsoft Entra PIM or the Microsoft Graph API where supported. Do not assume `az role assignment create` creates an eligible assignment.

**Access reviews**

Access reviews help you validate whether users, guests, groups, and privileged assignments still make sense. They are especially useful for reducing access drift over time.

Use access reviews when you need to:

- Reconfirm guest access to applications or groups.
- Validate membership in privileged groups.
- Review role assignments for users who may no longer need access.
- Support compliance evidence for periodic access recertification.

Common review inputs include:

- **Users**
- **Groups**
- **Applications**
- **Privileged role assignments**
- **Access package assignments**

Review outcomes are typically decisions such as approved, denied, or not responded. Recommendations may be shown to reviewers, but they are not usually the final outcome state.

**Access review design tips**

- Choose a review frequency that matches the sensitivity of the access.
- Use **group owners** or **application owners** as reviewers when they understand the business need.
- Use **self-review** for access that the user can justify directly.
- Enable **auto-apply results** when you want removals to happen automatically after the review ends.
- Keep the scope narrow enough that reviewers can make informed decisions.

**Example: access review workflow**

1. Create a review for a group or app assignment.
2. Assign reviewers and set the review period.
3. Notify reviewers and collect decisions.
4. Apply results automatically or manually.
5. Record the outcome for audit and compliance.

**Entitlement management**

Entitlement management provides a request-and-approval model for access packages. It is useful when access must be granted through a controlled business process rather than direct assignment.

An access package can include:

- Group membership
- Application access
- SharePoint or Teams-related access where supported
- Connected resources and approval workflows

This is valuable for:

- Employee onboarding
- Contractor access
- Partner access
- Temporary project access
- Access that must expire automatically

Entitlement management is often a better fit than manual group assignment when you need:

- A formal request workflow
- Approval by business owners
- Expiration dates
- Periodic revalidation
- Separation of duties

**Practical example**

A contractor requests access to a project application and a security group. The request is approved by the project owner, access is granted for 90 days, and an access review runs before renewal.

**Conditional Access and governance**

Conditional Access is not identity governance by itself, but it is a critical enforcement layer. It helps you decide whether a sign-in or sensitive action should be allowed based on conditions such as:

- User or group membership
- Device compliance
- Location
- Sign-in risk
- User risk
- Application
- Client app type

For privileged access, Conditional Access can require stronger authentication or device compliance before a user can sign in to admin portals or perform sensitive actions. Whether a PIM activation succeeds depends on the specific PIM settings and Conditional Access policies in place.

Common patterns include:

- Requiring **MFA** for all privileged role activations.
- Requiring a **compliant device** for admin portal access.
- Blocking legacy authentication for privileged accounts.
- Using sign-in risk policies to force stronger verification.

**Example: conceptual Conditional Access policy**

```text
If user is in privileged admin group
AND cloud app is Microsoft Azure Management
THEN require MFA
AND require compliant device
```

This kind of policy helps reduce the risk of stolen credentials being used to access administrative surfaces.

**Administrative units**

Administrative units let you scope administrative permissions to a subset of users or groups. This is useful in large organizations where local administrators should manage only a region, department, or business unit.

Use administrative units when you need:

- Delegated help desk administration
- Regional user management
- Scoped group or user administration
- Reduced blast radius for delegated roles

Administrative units are about *scope of administration*, not a full governance workflow. They can complement access reviews and PIM, but they are not a substitute for them.

**Example use case**

A global organization creates administrative units for North America, Europe, and APAC. Regional support staff can reset passwords and update user attributes only within their assigned unit.

**Workload identities and service principals**

Governance also applies to non-human identities such as service principals and managed identities. These identities often have powerful permissions and are easy to overlook.

Key points:

- **Managed identities** are preferred for Azure-hosted workloads because Azure manages the credentials.
- **Service principals** are still valid and common for app registrations, automation, and cross-tenant scenarios.
- You should review and minimize permissions for both types.
- Use access reviews or periodic audits where applicable to detect stale app permissions and unused credentials.

A managed identity is represented in Entra as a service principal object. When assigning Azure RBAC roles to a managed identity, use the managed identity’s service principal object ID, not the resource ID.

**Example: assign a Key Vault role to a managed identity**

```bash
az role assignment create \
  --assignee-object-id <managed-identity-service-principal-object-id> \
  --assignee-principal-type ServicePrincipal \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<subscription-id>/resourceGroups/<rg-name>/providers/Microsoft.KeyVault/vaults/<vault-name>
```

If you use the wrong identifier, the assignment may fail or target the wrong principal. This is a common implementation mistake.

**Break-glass and emergency access**

Every tenant should have a small number of emergency access accounts that are excluded from normal conditional access and PIM workflows, but protected with strong operational controls.

Recommended characteristics:

- Very limited number of accounts
- Strong, unique credentials
- Stored securely and monitored
- No routine use
- Tested recovery process
- Alerting on sign-in and role activation

These accounts are for tenant recovery when normal identity controls are unavailable. They should not be used for day-to-day administration.

**Governance architecture patterns**

A practical governance design often looks like this:

- **Baseline access**: users receive standard roles and group memberships through onboarding workflows.
- **Privileged elevation**: admins activate roles through PIM when needed.
- **Periodic validation**: access reviews confirm continued need.
- **Controlled requests**: entitlement management handles temporary or business-approved access.
- **Enforcement**: Conditional Access protects sign-in and admin actions.
- **Scoped delegation**: administrative units limit who can manage which users or groups.

This layered approach is stronger than any single feature because it addresses assignment, activation, review, and enforcement separately.

**Licensing and planning considerations**

Licensing for identity governance is nuanced and should be checked against the current Microsoft licensing terms for your tenant.

General planning guidance:

- **Conditional Access** is commonly associated with Microsoft Entra ID P1.
- **PIM** and advanced governance capabilities are typically associated with Microsoft Entra ID P2 or governance bundles.
- **Access reviews** and entitlement management may be available through Entra governance licensing options, depending on the exact feature and tenant licensing.
- Do not assume one license covers every governance scenario.

For exam and architecture purposes, focus on matching the feature to the control objective rather than memorizing a simplistic license matrix.

**Operational monitoring**

Identity governance is only effective if you monitor it.

Track:

- PIM activation events
- Role assignment changes
- Access review completion and removals
- Access package requests and approvals
- Conditional Access failures for privileged users
- Sign-ins by emergency access accounts

Useful investigation sources include:

- Microsoft Entra audit logs
- Microsoft Entra sign-in logs
- Azure Activity Log for resource-level changes
- Microsoft Sentinel or another SIEM for correlation

**Example: Azure CLI for auditing role assignments**

```bash
az role assignment list \
  --scope /subscriptions/<subscription-id> \
  --output table
```

Use this to compare permanent assignments against what should be eligible through PIM.

**Common exam considerations**

- Choose **PIM** when the requirement is just-in-time privileged access.
- Choose **access reviews** when the requirement is periodic recertification.
- Choose **entitlement management** when access must follow a request/approval lifecycle.
- Choose **Conditional Access** when the requirement is to enforce sign-in or session conditions.
- Choose **administrative units** when delegated administration must be scoped.
- Choose **managed identities** for Azure-hosted workloads whenever possible, but use service principals when app registration or cross-tenant automation is the better fit.

**Quick comparison**

| Capability | Primary purpose | Best used for |
|---|---|---|
| PIM | Time-bound privileged activation | Admin roles and privileged groups |
| Access reviews | Periodic access validation | Recertifying users, guests, groups, and assignments |
| Entitlement management | Request and approval workflow | Temporary or business-approved access |
| Conditional Access | Sign-in and session enforcement | MFA, device compliance, risk-based controls |
| Administrative units | Scoped delegation | Regional or departmental administration |

**Practical architecture guidance**

When designing identity governance for an Azure solution, ask these questions:

- Who needs access, and is it permanent or temporary?
- Can the access be made eligible instead of active?
- Who should approve the access request?
- How often should the access be reviewed?
- What conditions should be required at sign-in or activation?
- Which identities are human, and which are workloads?
- What is the recovery plan for emergency access?

If you can answer those questions clearly, you can usually map the requirement to the right Microsoft Entra control.
