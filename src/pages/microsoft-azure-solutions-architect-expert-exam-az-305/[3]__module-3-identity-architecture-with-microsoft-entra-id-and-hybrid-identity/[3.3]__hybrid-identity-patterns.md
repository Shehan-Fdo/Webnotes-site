## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Hybrid identity overview**

Hybrid identity connects on-premises identity systems, typically Active Directory Domain Services (AD DS), with Microsoft Entra ID so users can access cloud and on-premises resources with a consistent identity. For AZ-305, you need to choose the right pattern based on security, user experience, operational complexity, and resilience.

The main design choices are:

- **Password Hash Synchronization (PHS)** for simplest cloud authentication with strong resilience.
- **Pass-through Authentication (PTA)** when you want on-premises password validation without full federation.
- **Federation** with Active Directory Federation Services (AD FS) when you need advanced claims or legacy requirements.
- **Device identity** patterns for Microsoft Entra join, hybrid Microsoft Entra join, and Microsoft Entra registered devices.
- **Passwordless** methods such as Windows Hello for Business (WHfB) and FIDO2 security keys.
- **Governance and monitoring** to keep hybrid identity supportable at scale.

A good architecture usually prefers the simplest pattern that satisfies business and compliance requirements. In many modern designs, that means PHS plus Conditional Access rather than federation.

**Microsoft Entra Connect Sync architecture**

Microsoft Entra Connect Sync is the core synchronization engine for most hybrid identity deployments. It imports objects from AD DS, applies filtering and transformations, and exports selected attributes to Microsoft Entra ID. The design matters because identity data shape affects sign-in, device trust, group-based access, and lifecycle automation.

Key architecture concepts:

- **Source anchor** is the immutable link between the on-premises object and the cloud object.
  - In modern deployments, this is typically `ms-DS-ConsistencyGuid`.
  - Older deployments may use `objectGUID`.
  - You should keep the source anchor stable because it prevents duplicate cloud identities during moves or restores.
- **Immutable ID** is the cloud-side representation of the source anchor.
  - It is critical for federation and for matching existing cloud accounts during migration.
  - If the source anchor changes, user matching can break and create duplicate accounts.
- **UPN (User Principal Name)** should be routable and match the user’s sign-in name where possible.
  - Example: `alice@contoso.com` is preferable to `alice@contoso.local`.
  - If the UPN suffix is not verified in Microsoft Entra ID, users may need alternate sign-in names or experience sign-in confusion.
- **Attribute flow** determines what is authoritative in the cloud.
  - Some attributes remain mastered on-premises, such as `displayName`, `proxyAddresses`, and group membership in many designs.
  - Others can be cloud-managed depending on the feature and writeback configuration.
- **Join rules** determine how users, groups, and contacts are matched and projected.
  - Correct matching avoids duplicate objects and preserves access assignments.

A practical design decision is whether the cloud or on-premises directory is the source of authority for each attribute. For example, if Exchange is hybrid, `proxyAddresses` and mail-related attributes often remain on-premises authoritative, which affects how you plan writeback and lifecycle operations.

**Synchronization modes and staging mode**

Microsoft Entra Connect Sync supports a production server and a staging server. Staging mode is a key high-availability and change-control feature.

- **Production server**
  - Performs import, synchronization, and export.
  - Writes changes to Microsoft Entra ID.
- **Staging mode server**
  - Imports and synchronizes data.
  - Does **not** export changes to Microsoft Entra ID.
  - Is used for disaster recovery, validation, and configuration testing.

Staging mode is useful when you want:

- A warm standby for failover.
- A safe place to test filtering, attribute flows, and connector changes.
- A way to compare sync results before promoting changes.

> Staging mode does not eliminate the need for operational monitoring. If the active sync server fails, you must manually promote the staging server to production.

Typical failover approach:

1. Disable staging mode on the standby server.
2. Verify connector configuration and credentials.
3. Confirm export readiness.
4. Re-enable staging mode on the failed server only after recovery.

**Password Hash Synchronization**

PHS copies a hash of the on-premises password to Microsoft Entra ID. During sign-in, Microsoft Entra ID validates the password hash in the cloud. This is the simplest and most resilient hybrid authentication pattern.

Why it is often preferred:

- Users can sign in even if on-premises AD DS is unavailable.
- It supports Microsoft Entra ID Protection and many Conditional Access scenarios.
- It reduces dependency on on-premises authentication infrastructure.
- It is easier to operate than federation.

Important design points:

- PHS is not the same as storing the cleartext password in the cloud.
- It supports seamless sign-in when combined with domain-joined or hybrid-joined devices in some scenarios.
- It is often the best default choice unless a specific requirement forces PTA or federation.

**Pass-through Authentication**

PTA validates passwords against on-premises AD DS through lightweight agents installed on-premises. Microsoft Entra ID sends the authentication request to an available PTA agent, which then validates against AD DS.

Use PTA when:

- You need on-premises password validation.
- You want to avoid federation infrastructure.
- You have a requirement to keep password verification on-premises.

Operational considerations:

- PTA agents should be installed on multiple servers for redundancy.
- Agents need outbound connectivity to Microsoft Entra ID.
- Authentication depends on agent availability and network connectivity.
- If all agents are unavailable, sign-in fails unless another authentication method is configured.

PTA agent placement and network requirements matter:

- Place agents on servers with reliable outbound internet access.
- Avoid single points of failure by deploying at least two agents.
- Ensure firewall rules allow outbound HTTPS connectivity to Microsoft Entra endpoints.
- Monitor agent health and latency.

**Federation with AD FS**

Federation uses AD FS or another security token service to authenticate users and issue claims. It is the most flexible but also the most operationally complex hybrid pattern.

Use federation when you need:

- Advanced claims issuance or transformation.
- Legacy application compatibility.
- Smart card or specialized authentication requirements.
- Complex sign-in policies that cannot be met with cloud authentication alone.

Federation is less common in modern greenfield designs because cloud authentication plus Conditional Access often replaces it. However, it remains relevant for specific enterprise requirements.

Core federation concepts:

- **Claims transformation** maps incoming identity data to token claims.
- **Relying party trust** defines the application that consumes the token.
- **Sign-in fallback behavior** depends on the design:
  - If AD FS is unavailable, users may be unable to sign in unless a fallback path exists.
  - Some organizations maintain emergency cloud authentication accounts to reduce outage impact.
- **External publishing** usually requires Web Application Proxy or another reverse proxy pattern.
  - This adds exposure and operational overhead.
  - It must be designed carefully for high availability and certificate management.

Example claims rule pattern:

```text
c:[Type == "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/upn"]
 => issue(Type = "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/nameidentifier", 
          Value = c.Value);
```

This kind of rule is used to transform a UPN into a token identifier for a relying party. In practice, you design claims rules to satisfy application expectations while minimizing custom logic.

**Migration patterns from federation to cloud authentication**

A common AZ-305 scenario is moving from AD FS to PHS or PTA. The goal is usually to reduce complexity while preserving user sign-in continuity.

Typical migration approach:

1. Verify that user UPNs are routable and match verified domains.
2. Ensure Microsoft Entra Connect Sync is healthy and source anchors are stable.
3. Enable PHS or PTA alongside federation.
4. Test sign-in for pilot users.
5. Update domain authentication settings in Microsoft Entra ID.
6. Monitor sign-in logs and help desk tickets.
7. Decommission federation only after validation.

Design notes:

- PHS is usually the easiest target state.
- PTA can be a transitional step if the organization wants on-premises password validation during migration.
- Keep emergency access accounts that bypass federation dependencies.
- Validate application dependencies before removing AD FS, especially for claims-based apps.

**Device identity patterns**

Device identity determines how endpoints are represented in Microsoft Entra ID and how they participate in Conditional Access, device compliance, and passwordless authentication.

The three main patterns are:

- **Microsoft Entra joined**
  - The device is joined directly to Microsoft Entra ID.
  - Best for cloud-first or remote-first organizations.
  - Works well with Intune and modern management.
- **Hybrid Microsoft Entra joined**
  - The device is joined to AD DS and registered with Microsoft Entra ID.
  - Best for organizations with existing domain-joined Windows fleets.
  - Preserves on-premises domain capabilities while enabling cloud access controls.
- **Microsoft Entra registered**
  - The device is registered, not fully joined.
  - Common for personal devices or bring-your-own-device (BYOD) scenarios.
  - Provides a lighter trust relationship than join.

Use cases and effects:

- **Entra joined**
  - Strong fit for zero-trust and cloud-native management.
  - Supports Conditional Access based on device state.
  - Simplifies WHfB deployment.
- **Hybrid joined**
  - Useful when legacy line-of-business apps still require domain join.
  - Supports device-based Conditional Access and WHfB in hybrid environments.
  - Requires line-of-sight to AD DS during join and often during policy processing.
- **Registered**
  - Good for personal devices accessing work apps.
  - Often combined with app protection policies rather than full device compliance.
  - Does not provide the same level of trust as joined devices.

Prerequisites differ by pattern:

- **Entra joined**
  - Device can reach Microsoft Entra ID.
  - User has appropriate licensing and tenant settings.
  - Device management strategy is defined, often with Intune.
- **Hybrid joined**
  - AD DS domain membership.
  - Microsoft Entra Connect configured for device writeback or hybrid join registration.
  - Network connectivity to domain controllers during join.
- **Registered**
  - User signs in with a work account.
  - Device registration is allowed in tenant settings.
  - Often used on unmanaged or personal endpoints.

Conditional Access implications:

- Require compliant device for managed endpoints.
- Require hybrid joined device for legacy corporate desktops.
- Use app protection policies for registered personal devices.
- Combine device filters with user risk and sign-in risk for stronger controls.

**Windows Hello for Business and passwordless**

Windows Hello for Business is a passwordless authentication method that uses a user gesture such as PIN or biometrics to unlock a device-bound key pair. It reduces password exposure and improves user experience.

Deployment models in hybrid environments:

- **Key trust**
  - Uses public key infrastructure without requiring user certificates for sign-in.
  - Common in modern hybrid deployments.
  - Simpler than certificate trust.
- **Certificate trust**
  - Uses certificates issued to the user or device.
  - Useful when existing PKI and smart card-like requirements exist.
  - More operational overhead.
- **Cloud trust**
  - Uses Microsoft Entra ID and on-premises trust integration.
  - Designed to simplify hybrid WHfB deployment.
  - Often the preferred modern option when supported.

When to use each:

- Choose **cloud trust** when you want the simplest modern hybrid passwordless design and your environment supports it.
- Choose **key trust** when you need hybrid support without user certificates and your infrastructure aligns with key-based trust.
- Choose **certificate trust** when you already have certificate-based authentication dependencies or smart card requirements.

Practical design considerations:

- WHfB works best on Entra joined or hybrid joined devices.
- It pairs well with Conditional Access requiring MFA (multifactor authentication) or phishing-resistant authentication.
- It reduces help desk calls for password resets.
- It depends on device trust, so device registration health is critical.

Example policy intent:

```text
Conditional Access:
- Users: All users
- Cloud apps: Office 365
- Grant: Require multifactor authentication
- Session: Sign-in frequency = 12 hours
- Exclude: Break-glass accounts
```

This kind of policy is often paired with WHfB to encourage passwordless sign-in while preserving emergency access.

**Self-service password reset and writeback**

Self-service password reset (SSPR) lets users reset their passwords without help desk intervention. In hybrid environments, password writeback is important because users often need the reset to flow back to on-premises AD DS.

Why writeback matters:

- Users authenticate to cloud apps with the same identity as on-premises resources.
- A cloud-only reset would otherwise leave the on-premises password unchanged.
- Writeback keeps the password consistent across environments.

Typical configuration considerations:

- Enable SSPR for the target user group.
- Configure authentication methods such as Microsoft Authenticator, email, or phone.
- Enable password writeback in Microsoft Entra Connect Sync.
- Test reset and change flows from both cloud and on-premises contexts.

Example operational flow:

1. User initiates password reset in Microsoft Entra ID.
2. User completes verification.
3. Microsoft Entra ID sends the new password to the sync service.
4. The sync service writes the password back to AD DS.
5. The user can sign in to both cloud and on-premises resources with the new password.

**Conditional Access design**

Conditional Access is the policy engine that turns identity signals into access decisions. In hybrid identity, it is the main control plane for enforcing MFA, device compliance, location restrictions, and risk-based access.

Common design patterns:

- Require MFA for all users except emergency accounts.
- Require compliant or hybrid joined devices for sensitive apps.
- Block legacy authentication protocols.
- Require phishing-resistant authentication for privileged roles.
- Use sign-in risk and user risk policies where licensed.

Example policy scenarios:

- Finance users:
  - Require compliant device.
  - Require MFA.
  - Block access from unmanaged devices.
- Administrators:
  - Require phishing-resistant MFA.
  - Require approved device.
  - Use privileged identity management for just-in-time elevation.
- Contractors:
  - Allow access only from registered devices.
  - Restrict to browser sessions.
  - Apply session controls.

A simple policy example in pseudocode:

```text
If user is in "All Users"
and cloud app is "Azure Portal"
and device is not compliant
then block access
```

**Governance and privileged access**

Hybrid identity is not only about sign-in. It also supports governance, entitlement, and privileged access workflows.

Important integrations:

- **Privileged Identity Management (PIM)**
  - Provides just-in-time activation for privileged roles.
  - Reduces standing admin access.
  - Works best with strong authentication and device trust.
- **Access packages**
  - Bundle groups, apps, and SharePoint resources into governed requests.
  - Useful for onboarding contractors and project teams.
- **Group-based access**
  - Simplifies assignment to apps, roles, and policies.
  - Supports lifecycle automation when paired with dynamic groups or identity governance.

Design implications:

- Use PIM for admin roles rather than permanent assignment.
- Use access packages for temporary access with approval workflows.
- Use dynamic groups to drive Conditional Access or app assignment where appropriate.
- Keep privileged accounts separate from daily user accounts.

**Microsoft Entra Connect Health and monitoring**

Operational monitoring is essential for sync, PTA, and AD FS. Microsoft Entra Connect Health provides visibility into these components and helps you detect failures before users are impacted.

What to monitor:

- **Sync health**
  - Export/import errors.
  - Connector failures.
  - Staging server readiness.
  - Attribute conflicts and duplicate objects.
- **PTA health**
  - Agent availability.
  - Authentication latency.
  - Failure rates.
  - Connectivity to Microsoft Entra ID.
- **AD FS health**
  - Token issuance failures.
  - Proxy availability.
  - Certificate expiration.
  - Relying party trust issues.

Alerting scenarios:

- Sync export stops after a credential change.
- PTA agents go offline due to patching or firewall changes.
- AD FS certificate is near expiration.
- Token issuance failures spike after a claims rule change.

Practical monitoring commands and checks:

```powershell
# Check Microsoft Entra Connect Sync scheduler
Get-ADSyncScheduler

# Force a delta sync
Start-ADSyncSyncCycle -PolicyType Delta

# Check sync service status
Get-Service ADSync

# Inspect PTA agent service
Get-Service "Azure AD Connect Authentication Agent"
```

For AD FS, you also validate service and certificate health:

```powershell
Get-Service adfssrv
Get-AdfsProperties
Get-AdfsSslCertificate
```

**Troubleshooting device registration**

Device registration issues often show up as failed hybrid join, missing device state in Microsoft Entra ID, or Conditional Access failures.

Useful checks:

```cmd
dsregcmd /status
```

Key fields to review:

- `AzureAdJoined`
- `DomainJoined`
- `DeviceId`
- `TenantId`
- `SSO State`
- `NgcSet`

Interpretation examples:

- `AzureAdJoined : YES` indicates Microsoft Entra joined.
- `DomainJoined : YES` and `AzureAdJoined : YES` often indicates hybrid join.
- `AzureAdRegistered : YES` indicates a registered device.

Common causes of failure:

- Device cannot reach Microsoft Entra registration endpoints.
- SCP (service connection point) is missing or misconfigured.
- Time skew breaks authentication.
- User or device registration is disabled in tenant settings.
- The device is not in the correct OU or scope for hybrid join.

**Real-world design comparisons**

| Pattern | Best for | Strengths | Tradeoffs |
|---|---|---|---|
| PHS | Most modern hybrid deployments | Simple, resilient, low ops | Password validation occurs in cloud |
| PTA | On-prem password validation without federation | No password hash in cloud for validation path | Depends on agent availability |
| Federation | Advanced claims and legacy requirements | Maximum flexibility | Highest complexity and outage risk |
| Entra joined | Cloud-first endpoints | Strong cloud integration | May not fit legacy domain dependencies |
| Hybrid joined | Existing domain-joined fleets | Preserves domain capabilities | More complex device lifecycle |
| Registered | BYOD and light trust | Easy onboarding | Less control than joined devices |

**Exam-relevant outage and fallback behavior**

You should always think about what happens when a dependency fails.

- With **PHS**, cloud sign-in can continue even if AD DS is unavailable.
- With **PTA**, sign-in fails if agents or connectivity are unavailable.
- With **AD FS**, sign-in can fail if federation services, proxies, or certificates fail.
- With **hybrid join**, device-based access may fail if registration or policy evaluation is broken.
- With **WHfB**, users may be blocked if device trust or key provisioning fails.

This is why many designs prefer PHS plus Conditional Access and reserve federation for special cases.

**Practical configuration examples**

Enable PHS and password writeback in a hybrid environment:

```powershell
# Example: verify sync scheduler
Get-ADSyncScheduler

# Example: trigger a delta sync after configuration changes
Start-ADSyncSyncCycle -PolicyType Delta
```

A typical tenant-side configuration intent:

```text
Microsoft Entra Connect:
- Enable Password Hash Synchronization
- Enable Password writeback
- Enable device writeback only if required by downstream design
- Use staging mode server for failover
```

Troubleshoot hybrid join on a client:

```cmd
dsregcmd /status
gpupdate /force
```

If the device is not registering, check:

- Domain join status.
- SCP configuration.
- Network access to Microsoft Entra endpoints.
- User sign-in context.
- Event logs for device registration.

**Design guidance for AZ-305**

When you design hybrid identity, start with business requirements and then choose the least complex pattern that satisfies them.

Use these heuristics:

- Prefer **PHS** unless a requirement explicitly needs PTA or federation.
- Use **PTA** when password validation must remain on-premises but federation is unnecessary.
- Use **federation** only for advanced claims, legacy apps, or strict authentication requirements.
- Use **hybrid join** when you have a large domain-joined Windows estate and need device-based Conditional Access.
- Use **Entra join** for cloud-first managed endpoints.
- Use **registered devices** for BYOD or limited-trust scenarios.
- Use **WHfB cloud trust** where supported to simplify passwordless rollout.
- Use **Connect Health** and alerting from day one, not after an outage.

A strong AZ-305 answer usually explains not just the technology, but the operational impact, failure mode, and migration path. That is the difference between a working identity design and a supportable enterprise architecture.
