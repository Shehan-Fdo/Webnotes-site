## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Overview**

Microsoft Entra ID Connect synchronization is the core hybrid identity bridge between on-premises Active Directory Domain Services (AD DS) and Microsoft Entra ID. It synchronizes identities, attributes, and selected writeback objects so users can authenticate and access cloud resources with a consistent identity model.

At a design level, you choose Microsoft Entra Connect Sync when you need:
- Rich synchronization logic across one or more forests
- Advanced filtering and transformation rules
- Password hash sync, pass-through authentication, or federation integration
- Writeback features such as password, device, or group writeback
- A mature on-premises sync engine with staging and disaster recovery options

<details>
<summary>High-level architecture view</summary>

```text
On-prem AD DS forests
   |   |   |
   |   |   +--> Connector(s) --> Connector Space --> Metaverse --> Export --> Microsoft Entra ID
   |   |
   |   +------> Import / Sync / Export cycles
   |
Microsoft Entra Connect Sync server
   |-- Sync engine
   |-- SQL Server LocalDB or full SQL Server
   |-- Authentication agents / SSO components
   |-- Scheduler
   |-- Writeback connectors
```
</details>

**Core components**

Microsoft Entra Connect Sync is not just a single installer; it is a set of components that work together to import, normalize, and export identity data.

- **Connector**: A connection to a directory source such as AD DS or Microsoft Entra ID.
- **Connector space**: Staging area that holds objects as they exist in a specific connected directory.
- **Metaverse**: The central logical identity store where joined objects are represented as a single identity.
- **Synchronization rules**: Policies that control import, join, projection, attribute flow, and export behavior.
- **Scheduler**: The built-in engine that runs sync cycles on a cadence.
- **Writeback components**: Optional features that send selected cloud changes back to on-premises AD DS.

The architecture matters for exam questions because many issues are not “cloud” problems or “AD” problems; they are sync-engine problems caused by joins, precedence, filtering, or export conflicts.

**Server prerequisites and deployment dependencies**

A Microsoft Entra Connect Sync server has specific platform and dependency requirements. In design questions, validate the server OS, directory permissions, network access, and service account model before you choose the sync topology.

- **Supported Windows Server versions**: Use a supported Windows Server release for the version of Entra Connect you deploy. In practice, this is typically a modern supported server version such as Windows Server 2016, 2019, or 2022, depending on the product release and support matrix.
- **Domain membership**: The server is usually joined to the on-premises domain so it can reach domain controllers and read directory data.
- **Permissions**:
  - AD DS read permissions for import
  - Elevated permissions for features such as password writeback, device writeback, or group writeback
  - Microsoft Entra permissions for tenant configuration and synchronization
- **Service accounts**:
  - The sync engine uses service accounts for directory connectors.
  - In many deployments, the wizard creates or configures the required accounts automatically.
  - For least privilege, use the minimum permissions required for the selected features.
- **SQL dependency**:
  - The sync engine stores configuration and metaverse data in SQL.
  - Small deployments often use SQL Server LocalDB.
  - Larger or more resilient deployments may use a full SQL Server instance.
- **Network and proxy**:
  - Outbound HTTPS access to Microsoft Entra endpoints is required.
  - If a proxy is used, the sync server and any authentication agents must be able to reach required cloud endpoints through it.
  - TLS inspection or restrictive firewalls can break authentication or export operations if they interfere with certificate validation or long-lived connections.

> If the sync server cannot reach Microsoft Entra endpoints reliably, you may see import/export failures, authentication agent failures, or stale synchronization cycles even when AD DS connectivity is healthy.

**Sync engine internals**

The sync engine is the most exam-relevant part of the architecture. It processes objects in stages rather than moving them directly from AD DS to Microsoft Entra ID.

- **Import**:
  - Reads objects and attributes from a connected directory into the connector space.
  - Can be full or delta.
- **Synchronization**:
  - Evaluates join and projection rules.
  - Moves data from connector space into the metaverse.
  - Applies attribute flows and precedence.
- **Export**:
  - Writes changes from the metaverse or connector space to the target system, such as Microsoft Entra ID or AD DS for writeback.

The distinction between the **connector space** and the **metaverse** is critical:
- The connector space is source-specific and reflects what each connector sees.
- The metaverse is identity-centric and represents the unified person or object across sources.
- A single user may exist in multiple connector spaces but map to one metaverse object.

**Join, projection, and precedence**

The sync engine decides whether an incoming object joins an existing metaverse object or creates a new one.

- **Join**:
  - Links an imported object to an existing metaverse object.
  - Used when the object is recognized as the same identity.
- **Projection**:
  - Creates a new metaverse object when no suitable match exists.
- **Precedence**:
  - Determines which rule or source wins when multiple flows target the same attribute.
  - Higher-precedence rules override lower-precedence rules.

Practical implications:
- If join rules are too loose, unrelated objects can merge incorrectly.
- If join rules are too strict, duplicate identities can appear in the metaverse.
- If attribute flow precedence is misconfigured, cloud values may overwrite authoritative on-premises values or vice versa.

A common exam scenario is a user with multiple source objects across forests:
- One forest contains the account object.
- Another forest contains the resource mailbox or application-linked attributes.
- The sync engine must join them into one metaverse identity using consistent matching logic.

**Synchronization rule precedence and conflict resolution**

Synchronization rules determine how attributes move and which source wins.

- **Inbound rules** bring data from connectors into the metaverse.
- **Outbound rules** send data from the metaverse to a target connector.
- **Custom rules** can override or supplement built-in rules.
- **Precedence order** matters when multiple rules target the same attribute.

Conflict resolution typically follows these principles:
- The rule with the highest precedence wins for a given attribute flow.
- A more specific rule can override a generic built-in rule.
- If two sources provide the same attribute, the configured precedence and scoping filters determine the final value.

Use the **Synchronization Rules Editor** when you need to inspect or adjust rule behavior. This is often safer than editing low-level configuration directly because it shows the effective rule set and precedence.

**Typical sync cycle behavior**

The scheduler runs synchronization in cycles. The default cadence is commonly every 30 minutes for delta synchronization, though this can be customized.

- **Delta sync**:
  - Processes only changes since the last cycle.
  - Used for normal operations.
- **Full sync**:
  - Re-evaluates all objects and rules.
  - Used after major rule changes, filtering changes, or schema changes.
- **Initial sync**:
  - Often used during first deployment or after a major reconfiguration.
  - Rebuilds object relationships and attribute flows more comprehensively.

Example scheduler commands:

```powershell
# View scheduler state
Get-ADSyncScheduler

# Trigger a delta synchronization cycle
Start-ADSyncSyncCycle -PolicyType Delta

# Trigger a full synchronization cycle
Start-ADSyncSyncCycle -PolicyType Initial

# Temporarily disable the scheduler for maintenance
Set-ADSyncScheduler -SyncCycleEnabled $false

# Re-enable the scheduler
Set-ADSyncScheduler -SyncCycleEnabled $true
```

Use maintenance windows carefully:
- Disable the scheduler before large rule changes or bulk directory cleanup.
- Re-enable it after validation.
- Remember that delta sync may not fully reflect structural changes until you run an initial or full cycle.

**Authentication method architecture**

Microsoft Entra Connect Sync often appears in hybrid sign-in designs, but it is not itself the authentication method. It supports or coordinates with several sign-in models.

- **Password Hash Synchronization (PHS)**:
  - Hashes of on-premises passwords are synchronized to Microsoft Entra ID.
  - Microsoft Entra ID authenticates the user directly.
  - This is usually the simplest and most resilient hybrid sign-in model.
- **Pass-through Authentication (PTA)**:
  - Authentication requests are validated against on-premises AD DS through PTA agents.
  - Passwords are not stored in Microsoft Entra ID in the same way as PHS.
  - Requires agent availability and outbound connectivity.
- **Federation**:
  - Authentication is delegated to a federation service such as AD FS.
  - More complex and usually used when specific claims or legacy requirements exist.
- **Seamless Single Sign-On (Seamless SSO)**:
  - Provides transparent sign-in on domain-joined devices.
  - Uses a computer account and Kerberos-based flow to reduce prompts.
  - Commonly paired with PHS or PTA.

**PHS with Seamless SSO example**

This is a common modern design because it balances simplicity and user experience.

```text
AD DS password change
   -> Entra Connect syncs password hash
   -> User signs in to Microsoft Entra ID
   -> Seamless SSO provides silent sign-in on domain-joined device
```

Why this design is popular:
- High availability is strong because Microsoft Entra ID can authenticate even if the on-premises network is unavailable.
- Operational overhead is lower than federation.
- It is often the preferred baseline unless a specific requirement forces PTA or federation.

**PTA agent architecture**

PTA uses lightweight agents installed on one or more on-premises servers.

- Agents listen for authentication requests from Microsoft Entra ID.
- Requests are encrypted and relayed through outbound connections.
- Multiple agents provide redundancy.
- The agents do not require inbound firewall openings from the internet.

Design considerations:
- Deploy more than one PTA agent for resilience.
- Place agents on reliable servers with stable network access to domain controllers.
- Monitor agent health and connectivity.
- Use PTA when you need on-premises password validation without federation infrastructure.

**Staging mode and high availability / disaster recovery**

Staging mode is the primary built-in high availability pattern for Microsoft Entra Connect Sync.

- A **staging server** imports and synchronizes data but does not export changes to Microsoft Entra ID.
- It maintains a warm copy of configuration and sync state.
- You can promote it to active mode if the primary server fails.

This is useful for:
- Planned maintenance
- Disaster recovery
- Configuration testing before production cutover

Recovery expectations:
- The sync configuration is not “magically cloud-hosted”; you should back up the server and document the configuration.
- If the active server fails, you may need to reinstall Entra Connect on a replacement server and restore or reconfigure settings.
- Preserve:
  - Sync rules and customizations
  - Connector configuration
  - Service account details
  - Scheduler settings
  - Staging mode state
- Validate that the replacement server can reach the same domain controllers and cloud endpoints.

> Staging mode protects against sync-server failure, but it does not replace backups. Treat the sync server as a critical identity infrastructure component.

**Filtering and scoping**

Filtering controls which objects are synchronized. It is one of the most common design and troubleshooting areas.

Common filtering methods:
- **Domain filtering**
- **OU filtering**
- **Attribute-based filtering**
- **Group-based filtering**

Guidance:
- Use one primary filtering strategy when possible.
- Avoid combining multiple filtering methods unless you understand the precedence and operational impact.
- Attribute-based filtering is flexible but can be harder to troubleshoot.
- OU filtering is often easier to explain and maintain.

Risks of combining methods:
- An object may be excluded by one filter and unexpectedly included by another during rule changes.
- Different filters can create inconsistent behavior between initial and delta sync cycles.
- A change in OU placement may not behave the same as an attribute change if both are used.

Filtering and sync cycle behavior:
- During **initial sync**, the engine evaluates the full object set and can remove or project objects based on current filters.
- During **delta sync**, only changed objects are processed, so filter changes may not fully apply until a full or initial cycle runs.
- If you tighten filtering, objects may remain in the metaverse until the next full evaluation and export cleanup.

**Writeback features**

Writeback extends the hybrid model by sending selected cloud-side changes back to AD DS.

- **Password writeback**:
  - Allows users to reset passwords in Microsoft Entra ID and have them written back to on-premises AD DS.
  - Requires the appropriate licensing and configuration.
  - Useful for self-service password reset in hybrid environments.
- **Device writeback**:
  - Writes registered devices back to AD DS.
  - Used in some hybrid device and access control scenarios.
  - Requires specific prerequisites and is not universally needed.
- **Group writeback**:
  - Writes Microsoft 365 groups back to AD DS as universal security groups in supported scenarios.
  - Often used for on-premises applications that need to consume cloud-managed group membership.

Important limitations and prerequisites:
- Not every tenant or feature combination supports every writeback type.
- Device writeback and group writeback have version, licensing, and directory prerequisites.
- Writeback requires careful permissions and schema readiness.
- You should only enable writeback when there is a real downstream on-premises dependency.

**Multi-forest design patterns**

Microsoft Entra Connect Sync is often used in multi-forest environments.

Common patterns:
- **Account forest + resource forest**
  - User accounts live in one forest.
  - Mailboxes, groups, or resources live in another.
  - The sync engine joins related objects into one metaverse identity.
- **Multiple account forests**
  - Requires careful matching rules to avoid duplicate cloud identities.
- **Merged or staged forest migrations**
  - Sync rules and filtering must support coexistence during migration.

Example topology:

```text
Forest A: user accounts
Forest B: Exchange resources
Forest C: legacy apps

        \        |        /
         \       |       /
          Microsoft Entra Connect Sync
                 |
          Microsoft Entra ID
```

Key design tradeoffs:
- Use authoritative source rules to avoid attribute collisions.
- Ensure immutable identifiers remain stable during migrations.
- Plan for soft match or hard match behavior when consolidating identities.

**Cloud Sync versus Connect Sync**

Cloud Sync is a lighter-weight alternative, but it does not replace Microsoft Entra Connect Sync in every scenario.

Choose **Cloud Sync** when:
- You want simpler deployment and management
- Your requirements fit supported synchronization scenarios
- You do not need advanced features that only Connect Sync provides

Choose **Connect Sync** when:
- You need advanced writeback capabilities
- You need complex multi-forest joins or custom sync rules
- You need features not available in Cloud Sync
- You need mature staging-mode-based DR patterns

Feature gaps and unsupported scenarios are important:
- Cloud Sync does not provide the same depth of customization as Connect Sync.
- Some advanced writeback and transformation scenarios are not supported.
- Complex federation-adjacent or legacy hybrid designs may still require Connect Sync.

**Monitoring and troubleshooting**

Use multiple tools together; no single tool shows the full picture.

- **Synchronization Service Manager**:
  - Inspect connectors, runs, and object flow.
  - Review import, sync, and export steps.
- **Synchronization Rules Editor**:
  - Inspect rule precedence and attribute flow logic.
- **IdFix**:
  - Detect and remediate directory attribute issues such as invalid characters or duplicate proxy addresses before synchronization.
- **Event Viewer**:
  - Review application and service logs for sync engine and agent errors.
- **Microsoft Entra admin center**:
  - Check sync status, provisioning health, and sign-in related issues.
- **PowerShell**:
  - Query scheduler state and trigger sync cycles.

Common error categories:
- **Import errors**:
  - Source directory read issues
  - Schema or attribute problems
- **Synchronization errors**:
  - Join conflicts
  - Duplicate attribute values
  - Rule conflicts
- **Export errors**:
  - Duplicate proxyAddresses
  - Immutable ID mismatches
  - Permission failures
  - Object already exists in target

Practical troubleshooting example:
- If a user does not appear in Microsoft Entra ID, check whether the object is filtered out in the connector space.
- If the object exists but has wrong attributes, inspect rule precedence and attribute flow.
- If export fails with duplicate attribute errors, use IdFix or correct the source attribute conflict before rerunning sync.

**Common configuration scenarios**

1. **Password hash sync with Seamless SSO**
   - Use when you want the simplest resilient hybrid sign-in.
   - Enable PHS and Seamless SSO during setup.
   - Validate domain-joined device behavior and browser sign-in experience.

2. **Pass-through authentication with a staging server**
   - Use when you need on-premises password validation.
   - Deploy at least two PTA agents.
   - Keep a staging Entra Connect server ready for failover.
   - Monitor agent health and outbound connectivity.

3. **Multi-forest account/resource configuration**
   - Use when user accounts and resources are split across forests.
   - Define authoritative source forests.
   - Configure join rules carefully to avoid duplicate metaverse objects.
   - Test with a pilot OU before broad rollout.

Example PowerShell and operational checks:

```powershell
# Check current sync scheduler settings
Get-ADSyncScheduler | Format-List *

# Force a delta sync after a controlled attribute change
Start-ADSyncSyncCycle -PolicyType Delta

# Temporarily stop sync during maintenance
Set-ADSyncScheduler -SyncCycleEnabled $false
```

**Design and exam decision points**

When answering AZ-305 questions, focus on the business and operational requirement behind the architecture.

- Choose **Connect Sync** for advanced hybrid identity needs, complex topologies, or writeback-heavy designs.
- Choose **Cloud Sync** for simpler, lighter deployments when feature requirements fit.
- Choose **PHS** when you want the most resilient and simplest sign-in model.
- Choose **PTA** when passwords must be validated on-premises without federation.
- Choose **Seamless SSO** when you want reduced prompts for domain-joined users.
- Choose **staging mode** when you need failover, testing, or maintenance protection.

A good design answer explains not only what to deploy, but why the chosen sync architecture satisfies availability, security, and operational requirements.
