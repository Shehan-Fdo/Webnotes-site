## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Identity model overview**

In Microsoft Entra ID, an application is defined once and then represented in each tenant where it is used. The design-level distinction matters:

- **App registration**: the application definition in the home tenant.
- **Service principal**: the tenant-local identity instance that represents that app in a specific tenant.
- **Managed identity**: a special type of service principal managed by Azure for supported resources.

This separation helps you design for authentication, authorization, tenant boundaries, and lifecycle management. The app registration describes *what the app is*; the service principal describes *how that app is recognized and authorized in a tenant*.

A common design mistake is to treat these as interchangeable. They are related, but they serve different purposes. The app registration is where you define redirect URIs, exposed APIs, app roles, and credential configuration. The service principal is what receives permissions, appears in tenant directories, and is used for access control decisions in that tenant.

**Core concepts**

<dl>
<dt><strong>Application object</strong></dt>
<dd>The global definition of an app in its home tenant. It stores metadata such as display name, redirect URIs, exposed scopes, and credential references.</dd>
<dt><strong>Service principal</strong></dt>
<dd>The local identity object created in a tenant for an application. It is the security principal that can be assigned roles and granted access in that tenant.</dd>
<dt><strong>Enterprise application</strong></dt>
<dd>The portal view of the service principal and its related configuration in a tenant.</dd>
<dt><strong>Managed identity</strong></dt>
<dd>An Azure-managed service principal used by supported Azure resources to authenticate without you managing secrets or certificates.</dd>
</dl>

**How the objects relate**

When you register an app, Microsoft Entra ID creates the application object in the home tenant. If the app is used in another tenant, that tenant gets its own service principal instance. This is why permissions and consent are tenant-specific.

A useful mental model is:

- **Application object** = blueprint
- **Service principal** = instantiated identity in a tenant
- **Credential** = proof the app presents when authenticating
- **Permission grant** = what the tenant allows that identity to do

For single-tenant apps, the application object and service principal are often managed together in the same tenant. For multi-tenant apps, each customer tenant has its own service principal, which allows that tenant to control consent and access independently.

**When to use each identity pattern**

Design choice depends on where the workload runs, who owns it, and how it authenticates.

- Use an **app registration** when you need to define an application that will authenticate users or services, expose an API, or request delegated/application permissions.
- Use a **service principal** when you need a tenant-local identity that can be assigned Azure RBAC roles, granted API permissions, or used by automation in that tenant.
- Use a **managed identity** when the workload runs on a supported Azure resource and you want Azure to manage the credential lifecycle.
- Use **workload identity federation** when an external workload, such as CI/CD, should authenticate without storing a client secret or certificate.

> A service principal is not “the app” itself; it is the identity object that represents the app in a tenant. For external workloads, federation is often preferable to long-lived secrets.

**Authentication and authorization flow**

Authentication answers “who are you?” Authorization answers “what can you do?”

Typical service-to-service flow:

1. The workload authenticates to Microsoft Entra ID using a credential or federated assertion.
2. Entra ID issues an access token for a resource.
3. The resource validates the token and checks claims such as `aud`, `iss`, `tid`, `appid`, and roles/scopes.
4. The resource authorizes the request based on app roles, delegated scopes, or Azure RBAC.

For Azure resource access, the service principal is often granted an Azure role assignment at a subscription, resource group, or resource scope. For Microsoft Graph or custom APIs, permissions are granted through API permissions, app roles, or consent.

**Credential options**

Credentials are attached to the application object and used by the service principal to authenticate.

- **Client secret**
  - Simple to configure.
  - Easier to leak and harder to govern at scale.
  - Best avoided for long-lived production automation when better options exist.
- **Certificate**
  - Stronger operational posture than a secret because the private key is harder to exfiltrate and can be stored in a secure key store.
  - Useful for server-side apps and automation where certificate lifecycle can be managed.
- **Federated credential**
  - No stored secret in the identity platform.
  - Common for GitHub Actions, Azure DevOps, and other external identity providers that can issue trusted OIDC (OpenID Connect) assertions.
  - Reduces secret rotation burden.

A practical design rule is to prefer the least operationally risky credential that fits the platform. If the workload can use federation, that is often the cleanest option. If it cannot, certificates are generally preferable to secrets for production use.

**Example: app registration and service principal lifecycle**

A typical lifecycle looks like this:

1. Register the app in the home tenant.
2. Add redirect URIs, scopes, or app roles.
3. Create a credential or federated identity configuration.
4. Grant permissions and consent.
5. Deploy the workload and authenticate using the app identity.
6. Rotate or replace credentials as needed.
7. Remove the service principal and app registration when the workload is retired.

This lifecycle matters because deleting only the app registration does not always remove all tenant-local access patterns immediately in the way you expect. You should plan cleanup for both the application object and any service principals, role assignments, and consent grants.

**Practical CLI examples**

Microsoft Graph-based tooling is the current direction for identity management. Use Azure CLI carefully and prefer Graph-oriented commands or portal workflows where appropriate.

Create an app registration with Azure CLI:

```bash
az ad app create \
  --display-name "orders-api"
```

Create a service principal for that app:

```bash
az ad sp create --id <appId>
```

Assign an Azure role to the service principal:

```bash
az role assignment create \
  --assignee <servicePrincipalObjectId> \
  --role "Reader" \
  --scope /subscriptions/<subscriptionId>/resourceGroups/<rgName>
```

Create a federated credential for GitHub Actions using Microsoft Graph PowerShell:

```powershell
Connect-MgGraph -Scopes "Application.ReadWrite.All"

New-MgApplicationFederatedIdentityCredential `
  -ApplicationId <appObjectId> `
  -BodyParameter @{
    name = "github-main"
    issuer = "https://token.actions.githubusercontent.com"
    subject = "repo:contoso/orders-api:ref:refs/heads/main"
    audiences = @("api://AzureADTokenExchange")
  }
```

Example GitHub Actions login using OIDC:

```yaml
permissions:
  id-token: write
  contents: read

steps:
  - uses: azure/login@v2
    with:
      client-id: ${{ secrets.AZURE_CLIENT_ID }}
      tenant-id: ${{ secrets.AZURE_TENANT_ID }}
      subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}
```

This pattern avoids storing a client secret in the pipeline. The workload exchanges the GitHub-issued token for an Entra access token through federation.

**App roles, scopes, and consent**

Permissions are not all the same.

- **Delegated permissions** apply when a user is present and the app acts on behalf of that user.
- **Application permissions** apply when the app acts as itself without a signed-in user.
- **App roles** are permissions exposed by an API and assigned to users, groups, or service principals.
- **Scopes** are delegated permissions exposed by an API for user-consent scenarios.

Consent is the mechanism that records approval for permissions. In enterprise environments, admin consent is often required for application permissions or high-impact delegated permissions. Design your app so that permissions are narrow and understandable, because broad permissions increase blast radius.

**Azure RBAC versus Entra permissions**

These are often confused, but they solve different problems.

| Area | Controls access to | Typical assignment target |
|---|---|---|
| Azure RBAC | Azure resources such as subscriptions, resource groups, and resources | User, group, service principal, managed identity |
| Microsoft Entra permissions | Directory objects and APIs such as Microsoft Graph | Application, service principal, user, admin consent |

Use Azure RBAC when the workload needs to manage Azure resources. Use Entra permissions when the workload needs to read or manage directory data or call APIs protected by Entra ID.

**Design considerations for Azure architectures**

When you design an Azure solution, ask these questions:

- Where does the workload run?
- Does the platform support managed identity?
- Is the workload external to Azure?
- Is the identity single-tenant or multi-tenant?
- Does the app need user delegation or app-only access?
- What is the credential rotation strategy?
- What permissions are truly required?

These questions help you choose between managed identity, app registration plus service principal, or federated identity.

**Common scenarios**

- **Azure-hosted app accessing Key Vault or Storage**
  - Prefer managed identity when supported.
  - Grant the identity the minimum Azure RBAC role or data-plane permission required.
- **CI/CD pipeline deploying to Azure**
  - Prefer workload identity federation where supported.
  - Avoid long-lived secrets in pipeline variables.
- **Daemon or background service outside Azure**
  - Use an app registration with a service principal.
  - Prefer certificate or federation over a client secret when possible.
- **Multi-tenant SaaS API**
  - Use a single app registration in the publisher tenant.
  - Each customer tenant gets its own service principal after consent.
  - Design for tenant isolation and consent-driven onboarding.

**Security and governance**

Identity design is part of your security boundary.

- **Least privilege** reduces the impact of compromise.
- **Credential rotation** limits exposure time.
- **Conditional Access** can apply to service principals in some scenarios through workload identity controls and related governance patterns.
- **Privileged Identity Management (PIM)** can help with just-in-time elevation for human administrators, though it is not a substitute for workload credential hygiene.
- **Monitoring** should include sign-in logs, audit logs, role assignments, and consent grants.

A secure design also separates environments. Use different app registrations or at least different credentials and permissions for dev, test, and production. Reusing one identity across environments increases the chance of accidental overreach.

**Troubleshooting signals**

When authentication fails, inspect the failure mode before changing the design.

- `invalid_client`
  - The client ID, secret, certificate, or federated credential is wrong or expired.
- `unauthorized_client`
  - The app is not allowed to use the requested flow.
- `invalid_grant`
  - The assertion, consent, or token exchange failed.
- `insufficient privileges`
  - The service principal lacks the required API permission or Azure role.
- `AADSTS` errors
  - The error code often points to tenant, consent, redirect URI, or credential issues.

Useful checks:

```bash
az account show
az ad sp show --id <appId>
az role assignment list --assignee <servicePrincipalObjectId> -o table
```

For Microsoft Graph, inspect the application and service principal objects separately when troubleshooting. A permission may exist on the app registration but not be consented in the tenant, or a role assignment may target the wrong object ID.

**Design anti-patterns**

- Storing client secrets in source control or plain pipeline variables.
- Using one identity for all environments.
- Granting broad directory permissions when a narrow API permission is enough.
- Assuming an app registration automatically has access in every tenant.
- Treating managed identity as universally available for every Azure service.
- Using a secret when federation is available and supported by the platform.
- Confusing Azure RBAC with Microsoft Entra API permissions.

**Quick comparison**

| Capability | App registration | Service principal | Managed identity |
|---|---|---|---|
| Defines the app | Yes | No | No |
| Exists in a tenant | Home tenant | Yes | Yes, as Azure-managed identity |
| Can receive role assignments | No | Yes | Yes |
| Can expose API scopes/roles | Yes | No | No |
| Credential managed by Azure | No | No | Yes |
| Best for external automation | Sometimes | Yes | No |
| Best for Azure-hosted workloads | Sometimes | Sometimes | Often |

**Key takeaways**

- The **app registration** is the application definition.
- The **service principal** is the tenant-local identity used for access control.
- The **managed identity** is an Azure-managed service principal for supported resources.
- Prefer **federation** or managed identity over long-lived secrets when possible.
- Use **Azure RBAC** for Azure resources and **Entra permissions** for directory/API access.
- Design for least privilege, environment separation, and credential lifecycle management.
