## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Conditional Access design goals**

Microsoft Entra Conditional Access is the policy engine you use to enforce access decisions after authentication signals are evaluated. In practice, you design policies to answer four questions:

- Who is trying to sign in?
- What app or resource are they accessing?
- From what risk, device, location, or client context?
- What control should be enforced before access is granted?

A strong design separates broad guardrails from targeted exceptions. You usually combine Conditional Access with identity protection, device compliance, app protection, and privileged access workflows rather than trying to solve every requirement in one policy.

**Core policy model**

A Conditional Access policy has two parts:

- **Assignments** define the scope:
  - Users and groups
  - Cloud apps or actions
  - Conditions such as sign-in risk, user risk, device platform, client app, location, device state, and authentication context
- **Access controls** define the response:
  - Grant controls such as require MFA, require authentication strength, require compliant device, require hybrid Azure AD joined device, require approved client app, require app protection policy, require password change, or require terms of use
  - Session controls such as sign-in frequency, persistent browser session, app enforced restrictions, and Defender for Cloud Apps controls

Design the policy so the assignment is as narrow as possible and the control is as strong as the business scenario requires.

**Policy hierarchy and evaluation behavior**

Conditional Access is evaluated after primary authentication but before the token is issued for the target resource. Multiple policies can apply to the same sign-in.

- If any applicable policy blocks access, access is blocked.
- If policies require different grants, the user must satisfy the combined requirements.
- Session controls are applied only when the client and resource support them.
- Some controls are not meaningful for every app type, especially legacy protocols and non-interactive flows.

Operationally, you should think in layers:

1. Baseline policies for all users
2. Stronger policies for privileged users and sensitive apps
3. Exception handling for break-glass accounts and service scenarios
4. Session and device controls for data protection

> Emergency access accounts should be excluded from normal Conditional Access policies, but only with strict operational controls. Keep them monitored, cloud-only, protected by long random passwords or hardware keys, and tested regularly. Do not use them for day-to-day administration.

**Common design patterns**

**1. Baseline secure access**

Use a baseline policy to require MFA for all interactive users and block legacy authentication.

- Scope:
  - All users
  - All cloud apps
  - Exclude emergency access accounts
- Controls:
  - Require multifactor authentication
  - Block legacy authentication clients

This pattern reduces password-only access and closes common attack paths such as IMAP, POP, SMTP AUTH, and older Office protocols.

Example intent:

```text
If user is interactive and not an emergency account, then require MFA for modern auth and block legacy auth.
```

**2. Privileged access hardening**

Privileged roles need stronger controls than standard users.

- Scope:
  - Directory roles such as Global Administrator, Privileged Role Administrator, Security Administrator
  - Administrative portals and management apps
- Controls:
  - Require phishing-resistant MFA or authentication strength
  - Require compliant or hybrid joined device
  - Require sign-in risk to be low
  - Use shorter sign-in frequency

This pattern limits token theft and reduces the chance that a compromised password can be used for admin access.

**3. High-risk sign-in response**

Use risk-based policies when Microsoft Entra ID Protection is available.

- Sign-in risk indicates the likelihood that the current sign-in is malicious.
- User risk indicates the likelihood that the account itself is compromised.

Typical pattern:

- Low risk: allow with MFA
- Medium risk: require MFA or stronger authentication
- High risk: block access or require password change for the user

For exam scenarios, remember that risk-based policies are best for adaptive enforcement, not for replacing baseline MFA.

**4. Device-trusted access**

Use device-based controls when the business requires managed endpoints.

- Require compliant device for corporate data access
- Require hybrid Azure AD joined device for legacy on-premises managed fleets
- Use device filters when you need to target a subset of devices by attributes rather than by compliance state

Choose the control based on the trust signal you actually need:

- **Compliant device** means the device is managed and meets Intune compliance policy.
- **Hybrid Azure AD joined** means the device is joined to on-premises AD and registered in Entra ID.
- **Device filters** let you target devices by properties such as operating system, trust type, or ownership.

Use device filters when compliance is not the right signal. For example, you may want to target only corporate Windows devices, or exclude kiosk devices, without requiring full compliance evaluation.

**5. App protection for mobile and BYOD**

For bring-your-own-device scenarios, app protection is often better than full device compliance.

- Scope mobile users accessing corporate data from personal devices
- Require approved client app
- Require app protection policy
- Optionally require MFA

This pattern protects data inside the app without forcing device enrollment.

**6. Terms of use and compliance acknowledgment**

Conditional Access can require users to accept terms of use before access is granted.

Use this when you need:

- Legal acknowledgment
- Security policy acceptance
- Compliance attestation
- Region-specific or regulated access confirmation

This is not a technical security control by itself, but it is a valid access pattern for governance and audit requirements. It is often paired with MFA or device compliance.

**Authentication strength selection**

Authentication strength is more precise than a generic MFA requirement. It lets you require a specific class of authentication method.

Use the following guidance:

- **Standard MFA**:
  - Good for general workforce access
  - Suitable when you need stronger-than-password authentication but do not require the highest phishing resistance
- **Passwordless authentication**:
  - Good for reducing password fatigue and improving user experience
  - Examples include Microsoft Authenticator passwordless phone sign-in and Windows Hello for Business
  - Useful for standard users and frequent sign-ins
- **Phishing-resistant MFA**:
  - Best for privileged users, sensitive apps, and high-risk operations
  - Examples include FIDO2 security keys and certificate-based authentication
  - Use when the threat model includes adversary-in-the-middle attacks and token replay

A practical mapping:

- Finance app with moderate sensitivity: standard MFA or passwordless
- Admin portal: phishing-resistant MFA
- External partner access to low-risk collaboration: MFA plus guest restrictions
- High-value transaction approval: phishing-resistant MFA plus compliant device

> Do not assume all MFA methods are equal. A push notification is not the same as phishing-resistant authentication. For exam design questions, choose the strongest method that matches the risk and user experience requirements.

**Authentication context and custom controls**

Authentication context lets you trigger stronger controls only when a user reaches a sensitive action or app path. This is useful when you do not want to force the same control everywhere.

Common use cases:

- Require stronger authentication before accessing payroll, HR, or finance data
- Step up authentication before approving payments or changing security settings
- Apply a separate policy to a sensitive SharePoint site or application action

Authentication context fits into a layered design:

- Baseline policy covers all access
- Sensitive app or action policy adds step-up requirements
- Custom controls or external workflows handle specialized approval or compliance steps

Custom controls are used when a built-in control is not enough and you need to integrate an external system or workflow. In design terms, they are best for niche requirements such as third-party authentication or specialized compliance gates. Use them sparingly because they add operational complexity.

**Terms of use and compliance patterns**

Terms of use are often paired with access to regulated data or external collaboration.

Typical patterns:

- Internal users must accept acceptable use terms before accessing a sensitive app
- Guests must accept collaboration terms before entering a shared tenant resource
- Contractors must acknowledge security and privacy requirements before accessing project data

This pattern is useful when the requirement is governance, not just authentication. It does not replace MFA, device compliance, or risk-based controls.

**Guest, B2B, and cross-tenant access design**

Guest access needs explicit design because external identities behave differently from internal users.

Use these controls:

- Cross-tenant access settings to define trust with partner organizations
- Guest user restrictions to limit what guests can do
- Separate policies for guests versus members
- App-specific policies for collaboration workloads

Cross-tenant access trust settings let you decide whether to trust the partner tenant’s MFA, compliant device, or hybrid joined device claims. This is useful when you collaborate with another organization and want to avoid re-prompting users unnecessarily.

Example intent:

- Trust partner MFA for low-risk collaboration
- Do not trust partner device compliance for highly sensitive apps
- Require your own MFA or phishing-resistant authentication for admin or finance apps

Guest scenario examples:

- External project guest accessing Teams and SharePoint:
  - Allow with MFA
  - Require terms of use
  - Limit download on unmanaged devices
- Partner admin accessing a shared management portal:
  - Require phishing-resistant MFA
  - Require compliant device if the partner tenant can assert it and you trust that claim
  - Otherwise require your own managed device or block access

**Built-in exclusions and emergency access**

You need operational exclusions, but they must be tightly controlled.

Recommended exclusions:

- Emergency access accounts
- Service principals and managed identities, when the policy targets user sign-ins
- Break-glass accounts used only for tenant recovery

Why these matter:

- Emergency access accounts must remain available if Conditional Access, MFA, or identity protection systems fail.
- Service principals and managed identities are not standard interactive users, so user-based Conditional Access does not apply to them in the same way.
- Excluding them prevents accidental lockout and avoids applying user-centric controls to workload identities.

Operational guidance:

- Keep emergency accounts cloud-only and highly protected
- Store credentials securely and rotate them
- Monitor sign-ins and alert on any use
- Test them periodically in a controlled way
- Exclude them from policies only when necessary, and document the reason

**Device state, compliance, and filters**

Device-related conditions are often confused in exam questions.

Use this mental model:

- **Compliance** is a management outcome from Intune or another MDM solution.
- **Join state** indicates how the device is registered or joined.
- **Device filters** target devices by attributes.
- **Device state conditions** are useful when you need to distinguish managed from unmanaged or specific device categories.

Choose the right signal:

- Use compliant device when you need a strong management posture.
- Use hybrid Azure AD joined when you need to support domain-joined corporate endpoints.
- Use device filters when you need precision targeting, such as excluding shared devices or targeting only Windows 11 corporate devices.
- Avoid overusing join state when compliance is the real business requirement.

Example device filter intent:

```text
(device.deviceOwnership -eq "Company") and (device.operatingSystem -eq "Windows")
```

This is useful when you want to target corporate Windows devices without requiring every scenario to be fully compliant.

**Named locations and country-based conditions**

Named locations are helpful, but they are not proof of trust.

Use them for:

- Office IP ranges
- Known partner egress ranges
- Country-based access restrictions

Be careful with limitations:

- VPNs can make users appear to come from a trusted location even when they are remote.
- NAT can collapse many users behind one public IP.
- Mobile carriers and cloud proxies can shift apparent geography.
- Country-based conditions are coarse and can produce false trust assumptions.

Design implication:

- Use named locations as one signal, not the only signal.
- Combine them with MFA, device compliance, or risk-based controls.
- Do not treat “inside corporate network” as equivalent to “safe.”

**Session controls**

Session controls shape what happens after access is granted. They are not a substitute for strong authentication.

Key controls:

- **Sign-in frequency** forces reauthentication after a set interval
- **Persistent browser session** controls whether browser sessions survive closing the browser
- **App enforced restrictions** can limit download, copy, and print in supported apps
- **Defender for Cloud Apps session controls** can inspect and govern web sessions

Important limitations:

- Not all client types honor all session controls.
- Mobile apps, desktop apps, and browser sessions behave differently.
- Some controls require supported apps or brokers.
- Session controls do not fully protect data once it leaves the managed app unless paired with app protection or endpoint controls.

Practical guidance:

- Use shorter sign-in frequency for privileged users and sensitive apps.
- Use persistent browser session carefully on shared devices.
- Use app enforced restrictions for unmanaged browser access to sensitive content.
- Do not rely on session controls alone for data exfiltration prevention.

**Legacy authentication blocking**

Blocking legacy authentication is one of the highest-value Conditional Access patterns.

Legacy protocols often cannot satisfy modern MFA and are common attack targets.

Typical block pattern:

- Scope all users except emergency accounts
- Target all cloud apps
- Block client apps that use legacy authentication

Examples of legacy protocols to watch for:

- IMAP
- POP
- SMTP AUTH
- Older Office protocols
- Basic authentication flows

If a business process still depends on legacy auth, treat it as a migration issue, not a permanent exception.

**Practical policy examples**

**1. Baseline MFA and legacy auth block**

```json
{
  "displayName": "Baseline - Require MFA and block legacy auth",
  "state": "enabled",
  "conditions": {
    "users": {
      "includeUsers": ["All"],
      "excludeUsers": ["<break-glass-account-id>"]
    },
    "applications": {
      "includeApplications": ["All"]
    },
    "clientAppTypes": ["all"],
    "signInRiskLevels": [],
    "userRiskLevels": []
  },
  "grantControls": {
    "operator": "AND",
    "builtInControls": ["mfa"]
  }
}
```

In a real deployment, you would add a separate policy to block legacy authentication client types explicitly.

**2. Risk-based policy**

```json
{
  "displayName": "Risk - Medium sign-in risk requires MFA",
  "state": "enabled",
  "conditions": {
    "users": {
      "includeUsers": ["All"]
    },
    "applications": {
      "includeApplications": ["All"]
    },
    "signInRiskLevels": ["medium", "high"]
  },
  "grantControls": {
    "operator": "AND",
    "builtInControls": ["mfa"]
  }
}
```

A stronger variant for privileged apps is to require phishing-resistant authentication instead of generic MFA.

**3. B2B collaboration policy**

```json
{
  "displayName": "Guest - Teams and SharePoint collaboration",
  "state": "enabled",
  "conditions": {
    "users": {
      "includeUsers": ["GuestsOrExternalUsers"]
    },
    "applications": {
      "includeApplications": ["Microsoft Teams", "Office 365 SharePoint Online"]
    }
  },
  "grantControls": {
    "operator": "AND",
    "builtInControls": ["mfa", "termsOfUse"]
  },
  "sessionControls": {
    "signInFrequency": "12h"
  }
}
```

This pattern supports collaboration while keeping governance visible.

**4. Privileged admin policy with authentication strength**

```json
{
  "displayName": "Admin - Phishing-resistant MFA on compliant device",
  "state": "enabled",
  "conditions": {
    "users": {
      "includeRoles": ["Global Administrator", "Privileged Role Administrator"]
    },
    "applications": {
      "includeApplications": ["Microsoft Azure Management"]
    }
  },
  "grantControls": {
    "operator": "AND",
    "authenticationStrength": "phishing-resistant",
    "builtInControls": ["compliantDevice"]
  }
}
```

This is a strong exam pattern for high-value administrative access.

**5. Unmanaged device browser restriction**

```json
{
  "displayName": "Unmanaged device - web only with session restrictions",
  "state": "enabled",
  "conditions": {
    "users": {
      "includeUsers": ["All"]
    },
    "applications": {
      "includeApplications": ["SharePoint Online", "Exchange Online"]
    },
    "devicePlatforms": ["windows", "macOS", "iOS", "android"]
  },
  "grantControls": {
    "operator": "AND",
    "builtInControls": ["mfa"]
  },
  "sessionControls": {
    "appEnforcedRestrictions": true
  }
}
```

This pattern is useful when you want access from unmanaged devices but need to reduce download risk.

**PowerShell and Graph examples**

You often manage Conditional Access with Microsoft Graph PowerShell. The exact cmdlets and properties can vary by module version, but the workflow is consistent.

List policies:

```powershell
Connect-MgGraph -Scopes "Policy.Read.All","Policy.ReadWrite.ConditionalAccess"
Get-MgIdentityConditionalAccessPolicy
```

Create a policy skeleton:

```powershell
$params = @{
  displayName = "Baseline - Require MFA"
  state       = "enabled"
  conditions  = @{
    users = @{
      includeUsers = @("All")
      excludeUsers = @("break-glass-object-id")
    }
    applications = @{
      includeApplications = @("All")
    }
  }
  grantControls = @{
    operator = "OR"
    builtInControls = @("mfa")
  }
}

New-MgIdentityConditionalAccessPolicy -BodyParameter $params
```

Update a policy to add a session control:

```powershell
Update-MgIdentityConditionalAccessPolicy -ConditionalAccessPolicyId $policyId -BodyParameter @{
  sessionControls = @{
    signInFrequency = @{
      value = 8
      type  = "hours"
    }
  }
}
```

For exam purposes, know that Graph is the preferred automation path for modern identity configuration, while older AzureAD and MSOnline modules are legacy.

**Designing for client type differences**

Different clients support different controls.

- Browser sessions support the widest range of session controls.
- Mobile and desktop apps may support MFA and device compliance but not every browser-style session restriction.
- Legacy protocols often bypass modern controls entirely, which is why blocking them is important.
- Service-to-service and workload identities follow different authorization patterns than interactive users.

When a question asks which control to use, first identify the client type. The right answer often depends on whether the access is browser-based, native app-based, or non-interactive.

**Exam decision patterns**

Use these shortcuts when evaluating scenarios:

- If the requirement is “everyone must sign in securely,” choose baseline MFA plus legacy auth block.
- If the requirement is “admins need stronger protection,” choose phishing-resistant MFA and device compliance.
- If the requirement is “users on personal devices may access web apps but not download data,” choose app enforced restrictions or app protection.
- If the requirement is “external partners should use their own MFA,” consider cross-tenant trust settings.
- If the requirement is “access only from managed devices,” choose compliant device or hybrid joined device based on the management model.
- If the requirement is “sensitive action needs extra verification,” use authentication context.
- If the requirement is “users must accept policy before access,” use terms of use.
- If the requirement is “block risky sign-ins,” use sign-in risk conditions and a block or step-up policy.

**Common pitfalls**

- Treating MFA as a single uniform control instead of selecting the right authentication strength
- Using named locations as a trust boundary without considering VPN or NAT
- Forgetting to exclude emergency access accounts from broad policies
- Applying user Conditional Access concepts to service principals or managed identities
- Relying on session controls to replace device or app protection
- Using compliance when the requirement is really just device targeting
- Overlooking guest and cross-tenant trust settings in B2B scenarios
- Ignoring terms of use when the business requirement is legal or compliance acknowledgment

**Quick reference**

- **MFA**: good baseline for most users
- **Passwordless**: best for usability and reduced password risk
- **Phishing-resistant MFA**: best for admins and sensitive actions
- **Compliant device**: best for managed endpoints
- **Authentication context**: best for step-up access to sensitive actions
- **Terms of use**: best for governance and acknowledgment
- **Sign-in frequency**: best for reauthentication cadence
- **App enforced restrictions**: best for browser-based data protection
- **Cross-tenant trust**: best for partner collaboration efficiency

**Summary**

Conditional Access design is about combining identity signals, device posture, app sensitivity, and session governance into a policy set that matches business risk. The strongest designs are layered, explicit about exceptions, and realistic about client support. For AZ-305, focus on choosing the right control for the scenario rather than memorizing a single “best” policy.
