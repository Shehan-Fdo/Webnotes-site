## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**MFA and authentication strength overview**

<abbr title="Multi-Factor Authentication">MFA</abbr> requires two or more verification factors from different categories: something you know, something you have, and something you are. In Microsoft Entra, the practical exam focus is not just whether MFA is enabled, but which methods are allowed and how strong those methods are for a given access decision.

Authentication strength is a Conditional Access concept that lets you require a specific set of authentication methods rather than a generic MFA claim. This matters because not all MFA methods provide the same assurance. For example, a password plus SMS code is different from a passwordless sign-in using a FIDO2 security key, Windows Hello for Business, or a passkey with phishing-resistant properties.

**Core factor categories**

- **Knowledge factor**: something you know, such as a password or PIN.
- **Possession factor**: something you have, such as an authenticator app, FIDO2 security key, or device-bound credential.
- **Inherence factor**: something you are, such as biometrics.
- **Phone-based methods**: SMS and voice are possession-like in practice, but they are weaker and more vulnerable to interception, SIM swap, and social engineering than stronger possession methods.

A common exam trap is assuming every MFA method is equally strong. In reality, Microsoft Entra treats methods differently when you build Conditional Access policies, especially if you want phishing resistance or device-bound assurance.

**Why authentication strength exists**

Authentication strength helps you express policy intent more precisely than a simple “require MFA” rule. You use it when you want to say things like:

- Require any MFA for low-risk internal apps.
- Require phishing-resistant MFA for privileged roles.
- Require passwordless or hardware-backed methods for sensitive data access.
- Allow different method sets for different user populations or applications.

This is useful because a generic MFA requirement can be satisfied by weaker methods that may not be appropriate for high-value targets. Authentication strength lets you align the sign-in method with the risk of the resource.

**How Conditional Access uses it**

Conditional Access evaluates signals such as user, group, device state, location, application, risk, and session controls. Authentication strength is one of the grant controls you can apply.

Typical flow:

1. A user attempts to access an app.
2. Conditional Access evaluates the policy conditions.
3. The policy checks whether the user has satisfied the required authentication strength.
4. If not, Entra prompts for an allowed method or blocks access.

A policy can require:

- **MFA** in general, which allows any registered MFA method that satisfies the policy.
- **Authentication strength**, which restricts the user to a defined set of methods.
- **Combinations of controls**, such as compliant device plus authentication strength.

> Conditional Access and authentication strengths are licensed features. In practice, you need the appropriate Microsoft Entra ID P1 or P2 licensing for Conditional Access enforcement, and authentication strengths are used within that Conditional Access framework.

**Method strength and practical interpretation**

Not every method should be treated the same in design or exam scenarios.

- **Stronger, phishing-resistant methods**
  - FIDO2 security keys
  - Windows Hello for Business
  - Certificate-based authentication in supported scenarios
  - Passkeys when implemented in a phishing-resistant, device-bound way and supported by the platform and registration flow

- **Common MFA methods with lower assurance**
  - Microsoft Authenticator push approval
  - One-time passcodes from an authenticator app
  - SMS codes
  - Voice calls

The key distinction is whether the method is resistant to phishing and replay. Push-based MFA can still be abused by MFA fatigue attacks if users approve prompts blindly. SMS and voice are also weaker because the secret can be intercepted or redirected.

**Passkeys and passwordless sign-in**

Passkeys are a modern credential type that can support passwordless authentication. In Microsoft Entra, the exact assurance depends on the passkey type, device binding, and platform support.

Use careful wording:

- A passkey can be **passwordless**.
- A passkey can be **phishing-resistant** when it is device-bound and used in a supported flow.
- Not every passkey deployment has the same assurance level.

This distinction matters because exam questions may describe a passkey-like experience without guaranteeing phishing resistance. Always check whether the scenario includes device binding, platform support, and a strong registration flow.

**Authentication strength examples**

Authentication strengths are conceptually defined as allowed method sets. The exact Microsoft Graph schema can change, so treat the following as a policy design example rather than a literal API payload.

```json
{
  "displayName": "Phishing-resistant MFA",
  "description": "Require strong, phishing-resistant methods",
  "allowedCombinations": [
    "fido2",
    "windowsHelloForBusiness",
    "passkeyDeviceBound"
  ]
}
```

Use this only as a mental model. In real Microsoft Graph or portal configuration, the property names and identifiers must match the current API or admin UI. For implementation, verify the current Microsoft Graph authentication strength policy model before automating.

**Common method groupings**

| Method type | Typical assurance | Notes |
|---|---:|---|
| SMS / voice | Lower | Vulnerable to interception and social engineering |
| Authenticator push | Medium | Better than SMS, but not phishing-resistant by itself |
| TOTP app code | Medium | Better than SMS, still replayable if phished |
| FIDO2 security key | High | Strong phishing resistance |
| Windows Hello for Business | High | Device-bound and strong for enterprise use |
| Passkey | Varies | Depends on platform, binding, and registration flow |

**When to use each strength**

- **Any MFA**
  - Use for broad access where you want to improve baseline security.
  - Suitable for many standard business apps.
  - Not ideal for privileged or highly sensitive access by itself.

- **Phishing-resistant MFA**
  - Use for admin roles, finance systems, identity operations, and sensitive data.
  - Prefer when the threat model includes credential phishing or adversary-in-the-middle attacks.
  - Often paired with device compliance or trusted device requirements.

- **Passwordless**
  - Use to reduce password dependence and improve user experience.
  - Best when you want to remove password entry from the sign-in flow.
  - Still evaluate whether the method is strong enough for the target resource.

**Design patterns**

A secure enterprise design often uses different strengths for different tiers:

- **Tier 0 / privileged access**
  - Require phishing-resistant authentication strength.
  - Add compliant device or hybrid joined device requirements.
  - Limit access to admin workstations or privileged access workstations.

- **Business applications**
  - Require MFA for all users.
  - Allow authenticator app or passwordless methods.
  - Add sign-in risk controls if available.

- **External or high-risk access**
  - Require stronger methods and tighter session controls.
  - Consider blocking legacy authentication entirely.

This tiered approach reduces friction for ordinary work while protecting the most sensitive assets with stronger controls.

**Conditional Access policy example**

```text
Policy name: Require phishing-resistant MFA for admins
Assignments:
  Users: Directory roles -> Global Administrator, Privileged Role Administrator
  Cloud apps: All cloud apps
Conditions:
  Client apps: Browser and mobile/desktop
Grant:
  Require authentication strength: Phishing-resistant MFA
  Require device to be marked as compliant
Session:
  Sign-in frequency: 8 hours
```

This pattern is common in real deployments because it combines identity assurance with device assurance. Authentication strength alone does not guarantee the endpoint is healthy.

**Practical CLI and automation examples**

You typically manage Conditional Access and authentication strengths through the Microsoft Entra admin center or Microsoft Graph. For automation, you may inspect policies with Graph PowerShell.

```powershell
Connect-MgGraph -Scopes "Policy.Read.All","Policy.ReadWrite.ConditionalAccess"

Get-MgIdentityConditionalAccessPolicy | Select-Object DisplayName, State
```

To inspect authentication strength-related objects, use the current Graph endpoints and schema documented by Microsoft. The exact cmdlets and resource names can vary by module version, so validate against your tenant and module release.

If you are testing sign-in behavior, use the Entra sign-in logs:

```text
Microsoft Entra admin center
  -> Identity
  -> Monitoring & health
  -> Sign-in logs
```

Look for:

- Applied Conditional Access policies
- Authentication method used
- MFA requirement satisfied
- Failure reason or challenge result

**Troubleshooting sign-in issues**

When a user cannot satisfy an authentication strength policy, check the following:

- The user has a registered method that matches the required strength.
- The method is enabled in the tenant authentication methods policy.
- The Conditional Access policy targets the correct users and apps.
- The device or browser supports the required method.
- The user is not blocked by another policy, such as device compliance or legacy auth restrictions.

Common failure patterns:

- The user only has SMS registered, but the policy requires phishing-resistant MFA.
- The user has a passkey, but the platform or registration flow does not meet the required assurance.
- The user is excluded from the policy but still blocked by another overlapping policy.
- The app uses legacy authentication and cannot satisfy modern Conditional Access requirements.

**Authentication methods policy relationship**

Authentication strengths do not replace the authentication methods policy. The methods policy controls which methods users can register and use. Authentication strength controls which of those methods are acceptable for a specific access decision.

Think of it this way:

- **Authentication methods policy** = what users may enroll and use.
- **Authentication strength** = what the policy requires at sign-in.

If a method is disabled in the tenant, users cannot use it even if a Conditional Access policy would otherwise allow it.

**Exam-focused distinctions**

- **MFA** is a broad requirement.
- **Authentication strength** is a method-specific requirement.
- **Passwordless** is a sign-in experience, not automatically a guarantee of phishing resistance.
- **Phishing-resistant** means the method resists credential replay and phishing attacks.
- **SMS and voice** should not be treated as equivalent to stronger possession factors in high-assurance scenarios.

**Quick decision guide**

- Use **MFA** when you need baseline additional verification.
- Use **authentication strength** when you need to specify which methods are acceptable.
- Use **phishing-resistant MFA** for privileged or sensitive access.
- Use **passwordless** when you want to reduce password use, but still validate the method’s assurance level.

**Key takeaways**

- Authentication strength is a Conditional Access control for specifying acceptable authentication methods.
- Not all MFA methods are equally strong.
- Strong methods are typically device-bound or phishing-resistant.
- Passkeys can be strong, but their assurance depends on implementation details.
- Licensing and policy design matter because authentication strength is enforced through Conditional Access.
