## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Tenant fundamentals**

An <abbr title="Microsoft Entra tenant">tenant</abbr> is the top-level identity boundary for Microsoft Entra ID, directory objects, policies, and many governance controls. In Azure architecture, tenant design affects identity, collaboration, security, compliance, and operational separation more than it affects raw compute or networking.

A tenant is not just a login container. It influences:

- Identity lifecycle and authentication boundaries
- Directory-wide policies and governance
- External collaboration patterns
- Subscription ownership and billing relationships
- Branding, domains, and long-term organizational structure

For AZ-305, the key question is usually not “How do I create a tenant?” but “Should this capability live in the same tenant, a separate tenant, or a delegated model?”

**When a single tenant is usually the right choice**

Use one tenant when you want centralized identity and governance with minimal duplication. This is common when business units share the same security posture, compliance requirements, and collaboration model.

A single tenant works well when:

- Users need to collaborate frequently across the organization
- You want one set of identity policies, access reviews, and privileged controls
- Central IT manages apps, groups, and lifecycle processes
- You want simpler administration and fewer cross-tenant trust issues

A single tenant reduces complexity, but it also increases the blast radius of misconfiguration. Strong governance becomes more important as the tenant grows.

**When separate tenants are justified**

Separate tenants are appropriate when the organization needs hard boundaries that a tenant cannot provide through role assignment alone. This is a structural decision, not just an administrative convenience.

Common reasons include:

- Distinct legal entities with separate administration
- Regulatory or data residency requirements that demand isolation
- Mergers, acquisitions, or divestitures
- Strong separation between production and nonproduction identity planes
- Different identity owners, branding, or lifecycle processes
- Need to isolate external collaboration from internal identity governance

A separate tenant is often the right answer when you need independent control planes, not just delegated access.

> A tenant boundary is stronger than administrative delegation. If you need separate identity policy, separate object ownership, or separate external collaboration posture, a separate tenant may be required.

**Tenant naming, branding, and domain strategy**

Tenant naming decisions are long-lived. You can change some display values later, but you should not assume the tenant identity can be freely rebranded after adoption.

Important considerations:

- The initial tenant name and default domain often become part of early user experience
- Custom domains should be planned early for sign-in, email, and app branding
- Tenant rename capabilities are limited and not a general design strategy
- Domain changes can affect user principal names, application registrations, and email routing

Plan the tenant around the organization’s durable identity, not a temporary project name. If you expect a future merger, acquisition, or brand change, design the domain strategy up front.

**Tenant geography and data residency**

Microsoft Entra tenant creation includes an initial country or region choice, but this is not the same as selecting an Azure resource region. The choice influences Microsoft-managed service behavior and residency expectations, yet the exact data residency model depends on the service and Microsoft’s operational design.

Use this as a planning input for:

- Regulatory alignment
- Data residency expectations
- Tenant lifecycle and compliance documentation

Do not treat tenant geography as a simple “move data here” switch. For exam purposes, distinguish between:

- **Tenant geography**: a directory-level residency and service behavior consideration
- **Azure region**: where Azure resources are deployed
- **Data residency requirements**: a compliance outcome that may involve multiple services and controls

**Tenant-wide governance settings**

Tenant design includes the default governance posture. These settings often determine whether users can self-service access, invite guests, or register applications.

Common tenant-level controls include:

- User consent settings for enterprise applications
- Admin consent workflow for app permissions
- External collaboration and guest invitation restrictions
- Guest access permissions in Microsoft 365 and Entra
- Security defaults or Conditional Access policies
- Self-service app registration and group creation controls

These settings are often tested because they shape the attack surface. A permissive tenant can accelerate adoption, but it also increases risk from consent phishing, unmanaged guests, and shadow IT.

**Consent and app governance**

Application consent is a major tenant design decision. If users can grant broad permissions to apps, the tenant becomes vulnerable to malicious or overprivileged applications.

Typical patterns:

- Disable user consent and require admin approval for sensitive permissions
- Use an admin consent workflow for controlled approvals
- Restrict verified publishers or low-impact permissions if your policy allows it
- Review enterprise applications regularly

Example policy posture:

```text
User consent: Disabled
Admin consent workflow: Enabled
Users can register applications: No
Users can create security groups: No
```

This is not a universal best practice, but it is a common enterprise baseline when governance matters more than self-service.

**External collaboration options**

External collaboration is not one feature; it is a set of related capabilities with different trust and lifecycle models.

Key options include:

- **B2B guest access**: Invite external users as guests in your tenant for collaboration and access to resources
- **Cross-tenant access settings**: Define inbound and outbound trust, MFA (multifactor authentication), device claims, and collaboration rules between tenants
- **Cross-tenant synchronization**: Automate lifecycle and identity provisioning between tenants for selected users
- **B2B direct connect**: Enable seamless collaboration in supported apps and scenarios without creating a traditional guest object in the same way as guest access

Use the right model for the job:

- Choose guest access when you need broad, familiar collaboration with external identities
- Choose cross-tenant synchronization when you need managed lifecycle and consistent identity representation across tenants
- Choose direct connect only when the workload supports it and the scenario fits its limited scope

> B2B direct connect is not a general replacement for guest access. It applies only to supported apps and scenarios.

**Cross-tenant access settings**

Cross-tenant access settings let you define trust and collaboration rules with specific partner tenants. This is more precise than a blanket allow/deny model.

You can control:

- Inbound access from external tenants
- Outbound access to external tenants
- Trust for MFA claims
- Trust for compliant device claims
- Guest and direct connect collaboration behavior

Example conceptual configuration:

```powershell
# Conceptual example: inspect cross-tenant access settings
Get-MgPolicyCrossTenantAccessPolicy
Get-MgPolicyCrossTenantAccessPolicyPartner -CrossTenantAccessPolicyPartnerTenantId "00000000-0000-0000-0000-000000000000"
```

Use this model when you need to collaborate with specific partners while keeping a strong default-deny posture for everyone else.

**Guest lifecycle and external identity management**

Guest access is easy to start and easy to forget. Tenant design should include lifecycle controls so external users do not accumulate indefinitely.

Good practices include:

- Expiration or periodic review of guest access
- Access reviews for guest users and groups
- Entitlement management for package-based access requests
- Conditional Access for external users
- Clear ownership for guest sponsorship and removal

Microsoft Entra External ID is the broader identity capability set for external users and customers. In exam scenarios, distinguish between:

- **B2B collaboration**: external partner access to your tenant resources
- **B2C-style customer identity**: customer-facing sign-in patterns, where applicable
- **Cross-tenant synchronization**: managed identity lifecycle across organizations

**Administrative units**

Administrative units let you scope certain administrative roles to a subset of users, groups, or devices within a tenant. They are useful for delegated administration without creating a separate tenant.

Use administrative units when:

- Regional IT teams manage local users
- Help desk staff should only manage a subset of objects
- You want delegated control with tenant-wide policy consistency

They do not replace a separate tenant when you need:

- Separate identity policy boundaries
- Separate branding or domain ownership
- Separate external collaboration posture
- Full isolation of directory-wide settings

Administrative units are limited in scope. They do not segment the entire tenant, and not every role or object type is supported.

Example PowerShell pattern:

```powershell
Connect-MgGraph -Scopes "AdministrativeUnit.ReadWrite.All","RoleManagement.ReadWrite.Directory"

# Create an administrative unit
New-MgDirectoryAdministrativeUnit -DisplayName "EMEA Help Desk"

# Add a user to the administrative unit
New-MgDirectoryAdministrativeUnitMemberByRef `
  -AdministrativeUnitId "11111111-1111-1111-1111-111111111111" `
  -BodyParameter @{
    "@odata.id" = "https://graph.microsoft.com/v1.0/users/22222222-2222-2222-2222-222222222222"
  }
```

**Identity governance features that influence tenant design**

Tenant design should account for governance features that are easier to operate in a unified tenant than across multiple tenants.

Important capabilities include:

- **Privileged Identity Management (PIM)** for just-in-time privileged role activation
- **Access reviews** for users, groups, apps, and guests
- **Entitlement management** for access packages and request workflows
- **Identity Protection** for risk-based detection and remediation
- **Conditional Access** for policy enforcement at sign-in

These features are often easier to standardize in one tenant. If you split tenants, you may need to duplicate policies, review processes, and reporting.

**Hybrid identity considerations**

Hybrid identity affects tenant design, but it does not automatically require multiple tenants. Most organizations use one tenant with synchronized identities from on-premises directories.

Common hybrid patterns:

- **Password hash synchronization (PHS)**: synchronizes a hash of the password hash to Entra ID for cloud authentication
- **Pass-through authentication (PTA)**: validates credentials against on-premises agents
- **Federation**: redirects authentication to a federation service such as AD FS (Active Directory Federation Services) or another supported identity provider

Choose based on:

- Authentication requirements
- High availability and operational complexity
- Legacy application dependencies
- Compliance or sign-in control needs

Hybrid identity usually influences authentication architecture, not tenant count. A separate tenant is only needed if the business boundary or governance model requires it.

**Subscription, management group, and tenant relationships**

Azure subscriptions are associated with a single tenant at a time. Tenant design must account for how subscriptions are organized, delegated, and moved.

Key points:

- Management groups exist within a tenant hierarchy
- Policies and RBAC (role-based access control) are assigned within the tenant context
- Subscriptions can be moved between tenants, but the move is disruptive and requires planning
- Moving a subscription affects RBAC assignments, managed identities, Key Vault references, and some service integrations

Before moving a subscription, verify:

- Identity and access assignments can be recreated
- Policies and management group placement can be re-established
- Billing and support implications are understood
- Resource providers and dependent services support the move

Example CLI to inspect the current context:

```bash
az account show --output table
az account list --output table
```

`az account list` shows subscriptions available to the signed-in identity, which may span multiple tenants. It is not limited to a single tenant unless your context and access are constrained.

**Tenant migration and domain transfer**

Tenant migration is more than copying users. It usually involves identity, apps, devices, mail, and domain dependencies.

Typical migration concerns:

- Recreating users, groups, and app registrations
- Reassigning licenses and access policies
- Migrating devices and Intune enrollment
- Reconfiguring federation or synchronization
- Moving custom domains only after removing dependencies in the source tenant

Domain transfer is often constrained by objects that still reference the domain. You usually need to:

1. Remove or update user principal names, proxy addresses, and app references
2. Ensure no mailboxes, groups, or service principals still depend on the domain
3. Remove the domain from the source tenant
4. Add and verify the domain in the target tenant

This is rarely a simple cutover. Plan for coexistence, staged migration, and validation.

**When a separate tenant is required versus when it is not**

Use this decision logic:

- Choose a separate tenant when you need hard isolation of identity, policy, or collaboration
- Choose administrative units when you need delegated administration inside one tenant
- Choose cross-tenant collaboration when you need external access without merging identity boundaries
- Choose subscription separation when you only need workload isolation, billing separation, or policy segmentation at the Azure resource layer

A separate tenant is usually unnecessary if the requirement is only:

- Different Azure subscriptions
- Different resource groups
- Different management groups
- Different admin teams with scoped permissions
- Different application ownership inside the same organization

**Practical design examples**

A centralized enterprise might use:

- One tenant for workforce identity
- Multiple management groups for platform, landing zones, and workloads
- Separate subscriptions for production, nonproduction, and shared services
- PIM, access reviews, and Conditional Access at the tenant level
- Administrative units for regional help desk delegation

A merger scenario might use:

- Temporary cross-tenant collaboration
- Cross-tenant synchronization for selected users
- Separate tenants during transition
- A long-term consolidation plan if governance and business ownership align

A managed service provider scenario might use:

- Customer tenants for identity boundaries
- Azure Lighthouse for delegated Azure resource management
- Cross-tenant access only where identity collaboration is needed
- No assumption that Lighthouse replaces identity administration

**Quick comparison**

| Requirement | Better fit | Why |
|---|---|---|
| Scoped help desk administration | Administrative units | Delegates limited object management inside one tenant |
| External partner collaboration | B2B guest access or cross-tenant access | Preserves tenant boundaries while enabling access |
| Managed Azure resource operations across tenants | Azure Lighthouse | Delegates Azure resource management, not identity control |
| Separate identity governance and branding | Separate tenant | Provides stronger isolation than role scoping |
| Regional workload isolation only | Separate subscriptions or management groups | Keeps identity centralized |

**Exam tips**

- Treat tenant design as an identity and governance decision first.
- Do not confuse Azure regions with tenant geography.
- Do not use administrative units as a substitute for tenant isolation.
- Do not assume guest access, cross-tenant synchronization, and direct connect are interchangeable.
- Remember that subscription moves between tenants are possible but operationally significant.
- Use governance features such as PIM, access reviews, and entitlement management as part of the design discussion.

**Common pitfalls**

- Assuming one tenant is always enough for every organization
- Assuming multiple tenants automatically improve security
- Overstating the scope of administrative units
- Treating Azure Lighthouse as a universal cross-tenant solution
- Ignoring consent and external collaboration settings
- Forgetting that domain and app dependencies complicate tenant migration

**Summary**

Tenant design is about choosing the right identity boundary for the business. Most organizations benefit from a single well-governed tenant, but separate tenants are justified when isolation, compliance, or organizational boundaries demand it. Use administrative units, cross-tenant access, and governance controls to reduce the need for extra tenants, and reserve tenant separation for cases where the boundary itself is the requirement.
