## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Why naming and tagging matter**

Naming and tagging are governance controls, not just documentation habits. In Azure, they help you identify resources quickly, automate operations, enforce standards, allocate costs, and reduce ambiguity across subscriptions, management groups, and resource groups.

A strong standard gives you:

- **Consistency** across teams and environments.
- **Searchability** in the portal, CLI, and reporting tools.
- **Automation** support for scripts, policy, and IaC.
- **Cost visibility** through chargeback and showback.
- **Operational clarity** during incident response and lifecycle management.

**Core design principles**

A practical enterprise standard usually balances human readability with machine parsing. You want names that are short enough to satisfy service limits, but descriptive enough to identify the workload without opening the portal.

Common design goals:

- Use a predictable pattern such as `<org>-<app>-<env>-<region>-<instance>`.
- Keep names short enough for the strictest service in the design.
- Use lowercase where service rules require it.
- Avoid special characters unless the specific service allows them.
- Standardize abbreviations for environment, region, and workload type.
- Reserve room for future growth, such as additional instances or regions.

A good naming convention is not universal. Azure resource name rules vary by service, so the standard must be based on the most restrictive resources in your architecture.

**Azure naming constraints you must remember**

Azure resource naming rules are service-specific. Some resources allow almost any display name, while others have strict character and length limits.

Key examples:

- **Storage accounts** must be 3-24 characters, lowercase letters and numbers only, and globally unique.
- **Key Vault** names use lowercase letters, numbers, and hyphens are not allowed in the vault name.
- **Virtual network** names are generally flexible as Azure resource names, but related objects such as subnets have their own rules.
- **Public IP** resources have service-specific naming rules, but the public IP address itself is what must be unique when assigned.
- **Resource groups** allow a broader set of characters than many child resources, but you should still keep them simple and consistent.
- **Tags** are metadata, not resource names, and they follow their own constraints.

> Do not assume one naming rule applies to all Azure resources. Always verify the service-specific limits before standardizing a pattern.

**Recommended naming strategy**

A naming standard should encode only the information you truly need. Overly long names become hard to read and may fail service limits.

A common enterprise pattern is:

- `<workload>-<resource>-<env>-<region>-<instance>`

Examples:

- `pay-api-prod-weu-01`
- `hr-sql-dev-eus-01`
- `fin-vm-test-uks-02`

Use abbreviations that are documented and stable:

- `prod`, `test`, `dev`, `uat`
- `weu`, `eus`, `uks`
- `vm`, `sql`, `vnet`, `nsg`, `kv`

For resources with strict limits, shorten the pattern further:

- Storage account: `payprodweu01`
- Key Vault: `payprodweu01kv`
- Public IP: `pay-api-prod-weu-01-pip` only if the service rules allow it

When a service has a global uniqueness requirement, include a suffix or random component if needed. For example, storage accounts often need a unique string that still fits the 24-character limit.

**Naming by resource type**

Different resource types benefit from different naming emphasis.

- **Resource groups**
  - Use names that describe the workload and environment.
  - Example: `rg-payments-prod-weu`
  - Keep them stable because they often outlive individual resources.

- **Virtual networks**
  - Use names that identify the workload and region.
  - Example: `vnet-pay-prod-weu`
  - Subnets should be named for function, such as `snet-app`, `snet-data`, or `snet-private-endpoints`.

- **Network security groups**
  - Use names that show scope and purpose.
  - Example: `nsg-pay-app-prod-weu`

- **Key Vault**
  - Keep the name short and compliant.
  - Example: `kvpayprodweu01`
  - Avoid hyphens in the vault name.

- **Storage accounts**
  - Use a compact, lowercase, alphanumeric name.
  - Example: `stpayprodweu01`
  - Plan for uniqueness and length limits early.

- **Public IP addresses**
  - Use a readable resource name, but remember the IP address assignment and SKU matter operationally.
  - Example: `pip-pay-prod-weu-01`

- **App Service plans and apps**
  - Distinguish the hosting plan from the application.
  - Example: `asp-pay-prod-weu` and `app-pay-api-prod-weu`

**Tagging fundamentals**

Tags are name-value pairs attached to Azure resources and resource groups. They help you classify resources without encoding every detail into the resource name.

Typical tag keys include:

- `Environment`
- `Application`
- `Owner`
- `CostCenter`
- `BusinessUnit`
- `DataClassification`
- `Criticality`
- `ManagedBy`

Tags are especially useful when:

- Multiple teams share subscriptions.
- You need cost allocation by department or application.
- You want to automate cleanup of temporary resources.
- You need to identify support owners during incidents.
- You want policy-driven governance at scale.

Azure tag key matching is case-insensitive in practice, so standardize the exact spelling and casing in your policy and documentation. Tag values are consumed by tools and queries in different ways, so keep them consistent and documented.

**Tag design guidance**

A useful tag standard is simple, mandatory for key resources, and aligned to reporting needs.

Recommended practices:

- Use a small, controlled set of tag keys.
- Define allowed values for environment and business unit tags.
- Keep values short and consistent.
- Use tags for metadata, not for secrets or sensitive data.
- Apply tags at deployment time whenever possible.
- Use inheritance or automation to reduce manual drift.

Examples of useful tag values:

- `Environment=Prod`
- `Application=Payments`
- `Owner=CloudPlatform`
- `CostCenter=10420`
- `DataClassification=Confidential`

Avoid these anti-patterns:

- Free-form values that vary by person.
- Duplicate keys such as `env`, `Env`, and `environment`.
- Encoding operational state in tags that should be derived elsewhere.
- Storing PII (personally identifiable information) or secrets in tags.

**Naming versus tagging**

Naming and tagging solve different problems.

| Aspect | Naming | Tagging |
|---|---|---|
| Primary purpose | Identify the resource | Classify the resource |
| Visible in ARM ID | Yes | No |
| Good for automation | Yes | Yes |
| Good for cost reporting | Limited | Strong |
| Subject to service limits | Yes | Tag limits apply separately |
| Best for | Resource identity | Metadata and governance |

Use names for identity and tags for context. Do not try to force every governance attribute into the resource name.

**Governance hierarchy and where standards apply**

Azure governance works best when naming and tagging are defined at the right scope.

- **Management groups**
  - Use them to apply standards across multiple subscriptions.
  - Good for policy assignment and inheritance.
- **Subscriptions**
  - Use them to separate billing, environments, or business units.
  - Helpful for coarse-grained governance boundaries.
- **Resource groups**
  - Use them to group resources with the same lifecycle.
  - Good place to apply shared tags.
- **Resources**
  - Use them for resource-specific tags and names.

A common enterprise model is:

- Management group for the organization.
- Child management groups for platform and application teams.
- Subscriptions per environment or business unit.
- Resource groups per workload and environment.
- Resource-level tags for ownership and classification.

**Policy-driven enforcement**

Azure Policy helps you enforce naming and tagging standards at scale. It can audit noncompliance, deny nonconforming deployments, or modify resources to add missing metadata.

Common policy effects for governance:

- **Audit**: report noncompliance without blocking deployment.
- **Deny**: block resources that violate the rule.
- **Modify**: add or update tags or properties during deployment.
- **DeployIfNotExists**: deploy supporting resources or remediate missing configuration.

For tags, `modify` is often the right choice when you want to add missing tags or normalize values during deployment. `deny` is appropriate when a required tag is missing and you want strict enforcement. `audit` is useful during rollout so teams can fix issues before you block them.

Example policy idea for required tags:

```json
{
  "mode": "Indexed",
  "parameters": {
    "tagName": {
      "type": "String",
      "metadata": {
        "displayName": "Tag name",
        "description": "The tag that must exist on resources."
      }
    }
  },
  "policyRule": {
    "if": {
      "field": "[concat('tags[', parameters('tagName'), ']')]",
      "exists": "false"
    },
    "then": {
      "effect": "deny"
    }
  }
}
```

A more advanced approach is to use `modify` to add a default tag value when the tag is missing. That is better than trying to use `append` for remediation of existing resources.

**Practical Azure Policy example**

A common enterprise pattern is to require `Environment` and `CostCenter` tags on all resource groups, then inherit or propagate them to resources through deployment standards.

Example assignment strategy:

1. Assign a policy at the management group level.
2. Audit existing subscriptions first.
3. Remediate or update templates and pipelines.
4. Switch from audit to deny after adoption stabilizes.

This staged rollout reduces friction and avoids breaking legacy deployments immediately.

**Bicep and ARM template considerations**

Infrastructure as Code (IaC) should embed naming and tagging standards so deployments are repeatable.

Example Bicep pattern for a resource group deployment:

```bicep
param workload string
param environment string
param region string = resourceGroup().location
param costCenter string

var rgName = 'rg-${workload}-${environment}-${region}'

resource storage 'Microsoft.Storage/storageAccounts@2023-01-01' = {
  name: toLower('st${workload}${environment}${region}01')
  location: region
  sku: {
    name: 'Standard_LRS'
  }
  kind: 'StorageV2'
  tags: {
    Environment: environment
    Workload: workload
    CostCenter: costCenter
  }
}
```

For strict-name resources, validate inputs before deployment. For example, storage account names must remain lowercase and within length limits. If you generate names dynamically, ensure the final string is compliant before deployment.

A safer pattern is to centralize name generation in pipeline logic or a shared module so every team uses the same rules.

**Key Vault deployment note**

When you use Azure RBAC for Key Vault authorization, do not mix that model with access-policy-based authorization in the same design. Choose one authorization approach and align your deployment template accordingly.

- Use **RBAC** when you want centralized Azure role assignments and consistent governance.
- Use **access policies** when you need the older Key Vault permission model.
- Keep the authorization model explicit in your standard so teams do not combine both unintentionally.

**CLI examples for naming and tagging**

Azure CLI is useful for validating standards and applying tags.

Create a resource group with tags:

```bash
az group create \
  --name rg-payments-prod-weu \
  --location westeurope \
  --tags Environment=Prod Application=Payments CostCenter=10420 Owner=CloudPlatform
```

Add or update tags on an existing resource group:

```bash
az group update \
  --name rg-payments-prod-weu \
  --set tags.Environment=Prod tags.Application=Payments tags.CostCenter=10420
```

List resources with a specific tag:

```bash
az resource list --tag Environment=Prod --output table
```

Show tags on a resource:

```bash
az resource show \
  --ids /subscriptions/<subId>/resourceGroups/rg-payments-prod-weu/providers/Microsoft.Storage/storageAccounts/stpayprodweu01 \
  --query tags
```

PowerShell can also help with governance reporting, but cmdlet parameters vary by module version. A reliable pattern is to retrieve resources and filter by tags explicitly when needed.

```powershell
Get-AzResource | Where-Object { $_.Tags['Environment'] -eq 'Prod' } | Select-Object Name, ResourceType, ResourceGroupName
```

**Tagging with Azure Resource Manager templates**

ARM templates can apply tags during deployment. This is the best time to enforce standards because the resource is created correctly from the start.

Example:

```json
{
  "$schema": "https://schema.management.azure.com/schemas/2019-04-01/deploymentTemplate.json#",
  "contentVersion": "1.0.0.0",
  "parameters": {
    "workload": { "type": "string" },
    "environment": { "type": "string" },
    "costCenter": { "type": "string" }
  },
  "resources": [
    {
      "type": "Microsoft.Network/virtualNetworks",
      "apiVersion": "2023-11-01",
      "name": "[format('vnet-{0}-{1}-weu', parameters('workload'), parameters('environment'))]",
      "location": "westeurope",
      "tags": {
        "Environment": "[parameters('environment')]",
        "Workload": "[parameters('workload')]",
        "CostCenter": "[parameters('costCenter')]"
      },
      "properties": {
        "addressSpace": {
          "addressPrefixes": [
            "10.10.0.0/16"
          ]
        }
      }
    }
  ]
}
```

Use template functions carefully when generating names for strict resources. Validate length, allowed characters, and uniqueness requirements before deployment.

**Real-world enterprise pattern**

A mature Azure landing zone often combines naming, tagging, policy, and management groups into one governance model.

Example design:

- Management group `mg-platform`
  - Policy assignment: require `Environment`, `Owner`, and `CostCenter`
  - Policy assignment: deny resources without approved naming pattern
- Subscription `sub-payments-prod`
  - Resource group `rg-payments-prod-weu`
  - Tags inherited or applied by pipeline
- Workload resources
  - `vnet-pay-prod-weu`
  - `nsg-pay-app-prod-weu`
  - `stpayprodweu01`
  - `kvpayprodweu01`

Operational outcome:

- Support teams can identify the workload from the name.
- Finance can report costs by `CostCenter`.
- Security can filter by `DataClassification`.
- Automation can target resources by `Environment` and `Application`.

**Common exam pitfalls**

Watch for these mistakes in AZ-305 scenarios:

- Assuming all Azure resources share the same naming rules.
- Using names that are too long for the strictest resource in the design.
- Treating tags as a substitute for a good naming convention.
- Relying on manual tagging instead of policy or IaC.
- Mixing Key Vault authorization models.
- Using policy effects incorrectly for tag remediation.
- Forgetting that governance should be applied at the right scope, usually management group or subscription.

**How to choose a standard**

When designing a standard, ask these questions:

- What resource types are most restrictive?
- Which attributes must be visible in the name?
- Which attributes belong in tags instead?
- Which tags are mandatory for reporting and governance?
- Where should policy be assigned for inheritance?
- How will teams deploy resources consistently?

A good standard is simple enough to follow, strict enough to govern, and flexible enough to support future workloads.

**Quick reference**

- Use names for identity and tags for metadata.
- Keep names short, consistent, and service-compliant.
- Use tags for ownership, cost, environment, and classification.
- Enforce standards with Azure Policy and IaC.
- Roll out governance in stages to avoid disruption.
- Validate service-specific naming rules before finalizing patterns.
