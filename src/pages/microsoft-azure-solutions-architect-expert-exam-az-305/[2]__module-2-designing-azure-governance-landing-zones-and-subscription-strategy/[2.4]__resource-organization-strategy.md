## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Resource organization strategy overview**

Azure resource organization is the hierarchy you use to control access, apply governance, separate billing and ownership, and support operational scale. The core scopes are the **management group**, **subscription**, **resource group**, and **resource**. You choose the right level based on how broadly a rule should apply and how independently a workload must be managed.

A practical design usually follows a landing zone pattern:
- **Management groups** for enterprise-wide governance and policy inheritance.
- **Subscriptions** for billing boundaries, quota boundaries, and workload isolation.
- **Resource groups** for lifecycle and deployment grouping.
- **Resources** for the actual services.

This hierarchy matters because Azure policy, role-based access control (RBAC), and many governance decisions inherit downward. If you place a control too low, you repeat work. If you place it too high, you may overconstrain workloads.

**Azure tenant, Microsoft Entra ID, and subscription placement**

An Azure tenant is associated with a single Microsoft Entra ID directory. The tenant is the security and identity boundary for the organization’s Azure environment. Subscriptions live inside a tenant, and a subscription can be moved between tenants only with deliberate administrative action and planning.

Key implications:
- **Tenant boundary** affects identity, authentication, and governance.
- **Subscription placement** determines which directory users, groups, and service principals can be assigned access.
- **Cross-tenant moves** are not routine design choices; they are migration events with identity and governance impact.

For AZ-305, treat the tenant as the top-level identity boundary and the subscription as the primary unit for operational and billing separation within that tenant.

**Management groups for enterprise governance**

Management groups let you organize subscriptions into a hierarchy and apply governance at scale. Use them when you need consistent policy, RBAC, and compliance controls across multiple subscriptions.

Typical uses:
- Apply a policy baseline to all production subscriptions.
- Delegate platform administration to a central team.
- Separate platform, landing zone, and application subscriptions.
- Inherit controls to child management groups and subscriptions.

A common hierarchy looks like this:

```text
Tenant root group
├── Platform
│   ├── Identity
│   ├── Connectivity
│   └── Management
├── LandingZones
│   ├── Prod
│   └── NonProd
└── Sandbox
```

Design guidance:
- Use **management groups** when the same governance should apply to many subscriptions.
- Keep the hierarchy simple enough to understand and audit.
- Use naming that reflects function, not project trivia.

Example naming patterns:
- `mg-root`
- `mg-platform`
- `mg-platform-connectivity`
- `mg-lz-prod`
- `mg-lz-nonprod`

Management group limits and hierarchy depth are practical design constraints. Azure supports large hierarchies, but you should still avoid unnecessary nesting because it complicates policy evaluation, access reviews, and troubleshooting.

> Management group scope is ideal for governance, but not every control belongs there. Put only the controls that truly need broad inheritance.

**Subscription strategy**

A subscription is a strong boundary for:
- **Billing**
- **Quota and service limits**
- **Operational ownership**
- **Blast-radius reduction**
- **Environment separation**

Use separate subscriptions when you need one or more of the following:
- Distinct billing or chargeback.
- Different teams with independent operational control.
- Production isolation from nonproduction.
- Separate compliance or regulatory requirements.
- Different quota needs for a workload.

Subscription design is often constrained by practical limits:
- You must manage subscription sprawl.
- Some organizations hit internal governance limits before Azure platform limits.
- Large estates need a naming and ownership model to keep subscriptions discoverable.

Common subscription naming patterns:
- `sub-platform-prod`
- `sub-platform-nonprod`
- `sub-app-payments-prod`
- `sub-app-payments-dev`
- `sub-shared-services`

A good subscription name usually includes:
- Environment
- Business unit or workload
- Ownership or function
- Optional region or compliance tag if needed

Example:
- `sub-finance-prod-eus2`
- `sub-shared-networking-prod`
- `sub-devops-sandbox`

**When to choose a subscription instead of a resource group**

Choose a subscription when the boundary must be stronger than a resource group can provide:
- You need separate billing.
- You need separate Azure quotas.
- You need independent policy or RBAC administration.
- You need to isolate production from nonproduction.
- You need to separate platform services from application workloads.

Choose a resource group when the resources share a lifecycle and are managed together inside one subscription.

**Resource groups for lifecycle and deployment**

A resource group is a logical container for resources that share a lifecycle. It is the most common unit for deployment, deletion, and operational grouping.

Use resource groups to:
- Deploy an application stack together.
- Group resources that are created and removed together.
- Delegate access to a team for a specific workload.
- Simplify monitoring, tagging, and cost analysis.

Important behavior:
- A resource group’s **location** is metadata for the group itself.
- The resource group location is not a hard geographic boundary for all contained resources.
- Most resources can be placed in a resource group regardless of the group’s location.
- Some resource types have their own regional requirements, and those are resource-specific.

Example resource group naming patterns:
- `rg-payments-prod-eus2`
- `rg-webapp-dev-weu`
- `rg-shared-monitoring-prod`

A common convention is:
- `rg-<workload>-<environment>-<region>`
- `sub-<platform|app|shared>-<environment>`
- `mg-<scope>`

This makes it easier to identify ownership, environment, and region at a glance.

**Resource group lifecycle implications**

Deleting a resource group deletes all resources in it. This is one of the most important operational facts in Azure organization design.

Practical implications:
- Use resource groups only for resources that should be retired together.
- Do not mix long-lived shared services with short-lived test resources unless that is intentional.
- Be careful with automation that deletes and recreates groups.

Example:
- A web app, App Service plan, Key Vault, and monitoring resources may belong in one group if they are deployed and retired together.
- A shared virtual network or central firewall usually belongs in a separate platform group or subscription because it has a different lifecycle.

**Resource organization for shared services and hub-spoke**

Shared services are often better placed in a dedicated platform subscription or platform resource group rather than inside each application subscription.

Common shared services:
- Hub virtual network
- Azure Firewall
- VPN gateway or ExpressRoute gateway
- DNS services
- Bastion
- Central monitoring and logging
- Identity-related platform components

In a hub-spoke topology:
- Put shared connectivity in a **hub** subscription or platform resource group.
- Put application workloads in **spoke** subscriptions.
- Use management groups to apply baseline policy across both.

This separation helps you:
- Centralize network control.
- Reduce duplication.
- Keep platform ownership distinct from workload ownership.
- Apply different RBAC models to platform and application teams.

**Governance: RBAC and Azure Policy**

RBAC controls who can do what. Azure Policy controls what can be deployed and how resources must be configured. Both can be assigned at different scopes.

Scope choices:
- **Management group**: broad governance across many subscriptions.
- **Subscription**: governance for one environment or business unit.
- **Resource group**: workload-specific delegation.
- **Resource**: fine-grained exceptions or special cases.

Example policy assignment at management group scope:

```bash
az policy assignment create \
  --name require-tags \
  --display-name "Require cost center and owner tags" \
  --scope /providers/Microsoft.Management/managementGroups/mg-platform \
  --policy /providers/Microsoft.Authorization/policyDefinitions/require-tags \
  --params '{
    "tagName1": { "value": "CostCenter" },
    "tagName2": { "value": "Owner" }
  }'
```

Example RBAC assignment at subscription scope:

```bash
az role assignment create \
  --assignee-object-id 00000000-0000-0000-0000-000000000000 \
  --role "Contributor" \
  --scope /subscriptions/11111111-1111-1111-1111-111111111111
```

Example RBAC assignment at resource group scope:

```bash
az role assignment create \
  --assignee-object-id 00000000-0000-0000-0000-000000000000 \
  --role "Reader" \
  --scope /subscriptions/11111111-1111-1111-1111-111111111111/resourceGroups/rg-webapp-prod-eus2
```

Design guidance:
- Assign broad governance at higher scopes.
- Assign operational permissions at the lowest scope that still fits the team’s responsibility.
- Use groups, not individual users, for most assignments.

**Azure CLI examples for organization tasks**

Use CLI examples carefully because command support can vary by Azure CLI version and installed extensions. Verify syntax in your environment.

List management groups:

```bash
az account management-group list
```

Show a management group:

```bash
az account management-group show --name mg-platform
```

Create a resource group:

```bash
az group create \
  --name rg-payments-prod-eus2 \
  --location eastus2
```

Move a resource to another resource group:

```bash
az resource move \
  --destination-group rg-shared-prod-eus2 \
  --ids /subscriptions/11111111-1111-1111-1111-111111111111/resourceGroups/rg-old/providers/Microsoft.Storage/storageAccounts/stpay01
```

Move a subscription under a management group is a governance operation and may require tenant-level permissions and the correct CLI support for your version. In design notes and exam prep, focus on the concept: subscriptions are organized under management groups for inheritance.

**Bicep and deployment scope**

Bicep supports different deployment scopes. Resource organization affects where you deploy and what you can reference.

Resource group scope example:

```bicep
targetScope = 'resourceGroup'

param location string = resourceGroup().location
param name string

resource storage 'Microsoft.Storage/storageAccounts@2023-01-01' = {
  name: name
  location: location
  sku: {
    name: 'Standard_LRS'
  }
  kind: 'StorageV2'
}
```

Management group scope example:

```bicep
targetScope = 'managementGroup'

param policyName string = 'require-tags'

resource policyAssignment 'Microsoft.Authorization/policyAssignments@2022-06-01' = {
  name: policyName
  properties: {
    displayName: 'Require tags at scale'
    policyDefinitionId: '/providers/Microsoft.Authorization/policyDefinitions/require-tags'
  }
}
```

Use the correct scope for the control you are deploying:
- Resource group scope for workload resources.
- Subscription scope for subscription-level governance or shared services.
- Management group scope for inherited policy and RBAC.

**Resource movement and migration constraints**

Moving resources is not universal. Support depends on the resource type, dependencies, and target scope.

General rules:
- Many resources can move between resource groups in the same subscription.
- Some resources can move across subscriptions if both subscriptions are in the same tenant.
- Some resources cannot be moved at all.
- Dependent resources often must move together.

Examples of common move considerations:
- Virtual machines, disks, and network interfaces often require coordinated moves.
- Public IPs, load balancers, and virtual networks may have move constraints depending on dependencies.
- Some platform resources, marketplace resources, and certain managed services may not support moves.

Design guidance:
- Verify move support before you design a migration around it.
- Treat cross-subscription moves as a planned operation, not a default assumption.
- Expect validation failures if dependencies are incomplete.

**Naming and tagging strategy**

Naming and tagging are essential for discoverability, chargeback, and automation.

Recommended tags:
- `CostCenter`
- `Owner`
- `Environment`
- `Application`
- `DataClassification`
- `BusinessUnit`
- `Criticality`

Example tagging strategy:
- Apply mandatory tags at the management group or subscription level with Azure Policy.
- Use resource group tags for workload context.
- Use resource tags for resource-specific metadata.

Example resource group creation with tags:

```bash
az group create \
  --name rg-payments-prod-eus2 \
  --location eastus2 \
  --tags CostCenter=FIN-102 Owner=payments-team Environment=prod Application=payments
```

Good naming conventions should be:
- Consistent
- Short enough to read
- Machine-friendly
- Stable over time

Avoid names that encode temporary project details unless the resource is truly temporary.

**Choosing the right scope in AZ-305 scenarios**

Use this decision logic in design questions:
- If the requirement is enterprise-wide governance, choose **management groups**.
- If the requirement is billing, quota, or workload isolation, choose **subscriptions**.
- If the requirement is lifecycle grouping or deployment packaging, choose **resource groups**.
- If the requirement is a single service instance, use the **resource** itself.

Common exam-style scenarios:
- A company wants one policy baseline for all business units: use management groups.
- A product team needs separate billing for production and development: use separate subscriptions.
- A web application stack must be deployed and removed together: use one resource group.
- A shared network hub must serve multiple applications: place it in a platform subscription or dedicated resource group.
- A regulated workload needs stronger isolation from other teams: use a separate subscription and tighter management group controls.

**Compliance, billing, and operational ownership tradeoffs**

Resource organization is always a tradeoff between control and flexibility.

Management groups:
- Best for compliance inheritance.
- Not a billing boundary.
- Not a direct workload boundary.

Subscriptions:
- Best for billing and quota separation.
- Good for ownership boundaries.
- More overhead than resource groups.

Resource groups:
- Best for lifecycle and deployment grouping.
- Easy to delegate.
- Too narrow for enterprise-wide governance.

A balanced design often uses all three:
- Management groups for policy and structure.
- Subscriptions for environment and ownership separation.
- Resource groups for application lifecycle.

**Blueprints and landing zone governance**

Azure Blueprints was historically used to package governance artifacts such as policy, RBAC, and ARM templates. In modern Azure design discussions, landing zones and policy-driven governance are more common, and Blueprints is generally treated as legacy or de-emphasized.

For exam prep:
- Know that governance can be standardized.
- Prefer management groups, Azure Policy, RBAC, and infrastructure as code.
- Treat Blueprints as a legacy concept unless a question explicitly references it.

**Practical design checklist**

1. Identify the governance boundary.
2. Decide whether billing or quota isolation is required.
3. Separate platform services from application workloads.
4. Define naming conventions for management groups, subscriptions, and resource groups.
5. Apply tags consistently.
6. Use policy at the highest practical scope.
7. Delegate RBAC at the lowest practical scope.
8. Verify move support before planning migrations.
9. Keep shared services centralized.
10. Review lifecycle implications before placing resources in the same resource group.

**Quick comparison**

| Scope | Best for | Common use | Key caution |
|---|---|---|---|
| Management group | Governance inheritance | Policy and RBAC across subscriptions | Not a billing boundary |
| Subscription | Billing and isolation | Environment or workload separation | Can become fragmented if overused |
| Resource group | Lifecycle grouping | Deploy and delete together | Deleting the group deletes all resources |
| Resource | Single service control | Fine-grained configuration | Hard to manage at scale if overused |

**Exam takeaways**

- Use **management groups** for inherited governance across subscriptions.
- Use **subscriptions** for billing, quota, and strong operational boundaries.
- Use **resource groups** for lifecycle and deployment grouping.
- Remember that a resource group’s location is metadata, not a hard location boundary for all contained resources.
- Design around tenant boundaries, especially when identity and governance must remain separate.
- Use naming and tagging to make the hierarchy understandable and automatable.
- Verify resource move support instead of assuming every resource can be relocated.
