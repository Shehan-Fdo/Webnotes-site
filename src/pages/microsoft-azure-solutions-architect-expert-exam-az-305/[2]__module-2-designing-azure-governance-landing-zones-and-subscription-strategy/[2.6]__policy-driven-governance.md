## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Policy-driven governance overview**

Policy-driven governance is how you enforce organizational standards across Azure resources without relying on manual review. In practice, you use **Azure Policy** to define rules, assign them at the right scope, and measure compliance continuously. This is a core AZ-305 design topic because it connects architecture decisions to operational control, security, and cost management.

Policy-driven governance works best when you treat it as a layered control plane:

- **Management groups** define governance boundaries for large organizations.
- **Policy assignments** enforce rules at subscription, resource group, or management group scope.
- **Initiatives** bundle related policies for a framework or standard.
- **Exemptions** handle approved exceptions without weakening the baseline.
- **Remediation** brings existing resources back into compliance when the policy supports it.

A good governance design balances enforcement with flexibility. You want to prevent drift, but you also need a controlled path for exceptions, legacy workloads, and phased adoption.

**Azure Policy fundamentals**

Azure Policy evaluates Azure resources against rules that you define in JSON. A policy definition contains a condition and an effect. The condition describes what to check, and the effect determines what happens when the condition matches.

Common policy effects include:

- **deny**: blocks noncompliant create or update requests.
- **audit**: allows the request but marks the resource noncompliant.
- **append**: adds missing request properties during create or update.
- **modify**: changes request properties during create or update and can also remediate existing resources when configured correctly.
- **deployIfNotExists**: deploys a related resource or configuration when the target resource does not meet the rule.
- **auditIfNotExists**: audits when a related resource or configuration is missing.

Use the effect based on intent:

- Use **deny** when the rule is mandatory and should stop bad deployments.
- Use **audit** when you need visibility before enforcement.
- Use **modify** when you want Azure Policy to correct supported properties.
- Use **deployIfNotExists** when compliance requires a companion resource or configuration, such as diagnostics settings or a managed identity-linked deployment.
- Use **append** only for request-time property injection; it does not remediate existing resources.

> Azure Policy can only govern what Azure exposes through supported aliases and resource provider behavior. If a property has no alias or the provider does not support the scenario, policy may not be able to enforce it.

**Scopes, inheritance, and evaluation**

Azure governance is hierarchical. Policies assigned at a higher scope flow down to child scopes unless excluded.

Typical scopes:

- **Management group**
- **Subscription**
- **Resource group**
- **Resource**

Use higher scopes for enterprise standards and lower scopes for workload-specific exceptions or special cases. Management group assignments are especially useful for landing zones because they create a consistent baseline across many subscriptions.

Policy evaluation happens in several ways:

- **On create and update**: Azure evaluates the request before or during deployment.
- **Periodic compliance scans**: Azure re-evaluates existing resources on a schedule.
- **Manual compliance scans**: you can trigger a scan after changes or remediation.
- **Resource changes outside ARM**: changes made through portal, CLI, PowerShell, SDKs, or templates are still evaluated.

A useful mental model is:

1. You define the rule.
2. You assign it at the right scope.
3. Azure evaluates new and changed resources.
4. Azure records compliance state.
5. You remediate or exempt as needed.

**Policy definition structure**

A policy definition is usually composed of:

- **mode**: controls which resource types are evaluated.
- **parameters**: makes the policy reusable.
- **policyRule**: contains `if` and `then`.
- **metadata**: stores versioning and category information.

Example of a simple audit policy for allowed locations:

```json
{
  "mode": "All",
  "parameters": {
    "listOfAllowedLocations": {
      "type": "Array",
      "metadata": {
        "displayName": "Allowed locations",
        "description": "The list of locations that can be specified when deploying resources."
      }
    }
  },
  "policyRule": {
    "if": {
      "not": {
        "field": "location",
        "in": "[parameters('listOfAllowedLocations')]"
      }
    },
    "then": {
      "effect": "audit"
    }
  }
}
```

This pattern is common for AZ-305 because it shows how you separate the rule from the values. The same definition can be reused across environments by changing the parameter at assignment time.

**Modes and resource coverage**

The `mode` setting determines which resource types are evaluated.

- **All** evaluates all resource types that Azure Policy can inspect.
- **Indexed** evaluates resource types that support tags and location and excludes resource types that do not support those properties.

Use `Indexed` when you are targeting tag or location governance and want to avoid evaluating unsupported resource types. Use `All` when you need broader coverage, especially for nested properties or resource types beyond simple tag/location scenarios.

**Parameters and parameterization patterns**

Parameters make policies reusable and easier to manage at scale. You should parameterize values that vary by environment, business unit, or region.

Common parameter patterns:

- **Allowed values** for controlled inputs such as regions, SKUs, or tag names.
- **StrongType** for portal-friendly selection of locations or resource types.
- **Default values** for optional settings.
- **Object parameters** for structured configuration in advanced policies.

Example parameterized policy for required tags:

```json
{
  "mode": "Indexed",
  "parameters": {
    "tagName": {
      "type": "String",
      "metadata": {
        "displayName": "Tag name",
        "description": "Name of the tag to require."
      }
    }
  },
  "policyRule": {
    "if": {
      "field": "[concat('tags[', parameters('tagName'), ']')]",
      "exists": "false"
    },
    "then": {
      "effect": "deny"
    }
  }
}
```

The example above illustrates the intent, but in real Azure Policy definitions you must use valid aliases and supported expressions for the resource type you target. In practice, tag policies are often implemented with built-in definitions or carefully tested custom definitions because alias support varies by resource provider.

A better exam takeaway is this:

- Use parameters to avoid hardcoding environment-specific values.
- Pass parameter values at assignment time.
- Keep policy definitions generic and reusable.
- Validate alias support before relying on a custom rule.

**Initiatives and compliance frameworks**

An **initiative** is a policy set definition that groups multiple policy definitions under one assignment. Use initiatives when you want to enforce a broader standard such as:

- security baseline
- regulatory framework
- landing zone baseline
- cost governance baseline
- operational readiness baseline

Initiatives are especially useful when a compliance framework maps to many controls. Instead of assigning dozens of policies individually, you assign one initiative and track compliance at both the initiative and policy levels.

Why initiatives matter:

- They simplify assignment management.
- They provide a single compliance view for a standard.
- They let you pass parameters to child policies consistently.
- They support versioned governance baselines.

Example initiative use case:

- One initiative enforces:
  - allowed locations
  - required tags
  - diagnostic settings
  - approved SKUs
  - resource type restrictions

At the initiative level, you see overall compliance. At the child policy level, you identify which control failed.

**Passing parameters from initiatives to child policies**

Initiatives can map initiative parameters to child policy parameters. This is important when multiple policies need the same business value, such as a tag name or allowed region list.

Conceptually:

- The initiative defines a shared parameter.
- Each child policy references that shared parameter.
- The assignment supplies the value once.

Example pattern:

```json
{
  "properties": {
    "displayName": "Enterprise Governance Baseline",
    "policyDefinitions": [
      {
        "policyDefinitionId": "/providers/Microsoft.Authorization/policyDefinitions/allowed-locations",
        "parameters": {
          "listOfAllowedLocations": {
            "value": "[parameters('allowedLocations')]"
          }
        }
      },
      {
        "policyDefinitionId": "/providers/Microsoft.Authorization/policyDefinitions/require-tag",
        "parameters": {
          "tagName": {
            "value": "[parameters('requiredTagName')]"
          }
        }
      }
    ],
    "parameters": {
      "allowedLocations": {
        "type": "Array"
      },
      "requiredTagName": {
        "type": "String"
      }
    }
  }
}
```

This pattern is common in enterprise governance because it reduces drift between related policies.

**Assignments, exclusions, and exemptions**

A policy assignment applies a definition or initiative to a scope. You can narrow the effect with exclusions or formal exemptions.

Use these concepts carefully:

- **Exclusions / not-scopes**: remove a child scope from evaluation at assignment time.
- **Exemptions**: formally document an approved exception while preserving visibility and governance reporting.

Prefer **exemptions** when you need:

- time-bound exceptions
- business justification
- approval tracking
- auditability
- reporting without changing the baseline assignment

Use **not-scopes** when you need a structural exclusion, such as a management group or subscription that is intentionally outside the policy boundary.

A practical rule:

- Use **exemptions** for exceptions to policy.
- Use **not-scopes** for scopes that should never have the policy applied.

**Assignment identity and permissions**

Some policy effects require Azure to act on your behalf. In those cases, the assignment needs a managed identity and the identity needs permissions.

This is critical for:

- **modify**
- **deployIfNotExists**

Why identity matters:

- Azure Policy must be able to write changes or deploy resources.
- The managed identity performs the remediation or deployment.
- The identity needs the right role assignments at the correct scope.

Common permission requirements:

- **Contributor** or a narrower custom role for deployment actions
- **Resource Policy Contributor** for policy management tasks
- Specific resource roles for the target resource type when remediation is needed

For `deployIfNotExists`, the policy definition typically needs:

- `then.details.roleDefinitionIds`
- a valid deployment template
- `existenceCondition`
- `existenceScope` or `resourceGroupName` when appropriate

For `modify`, the policy definition typically needs:

- `then.details.roleDefinitionIds`
- `then.details.conflictEffect`
- valid operations that target supported aliases

> If the assignment identity does not have the required permissions, remediation fails even if the policy is correctly defined.

**Remediation and compliance correction**

Compliance reporting tells you what is wrong. Remediation fixes what is already deployed, when the policy supports it.

Use remediation when:

- a policy was assigned after resources already existed
- you changed a policy from audit to enforce
- you need to bring legacy resources into compliance
- the effect supports remediation, such as `modify` or `deployIfNotExists`

Important distinctions:

- **append** affects only create/update requests.
- **deny** prevents future noncompliance but does not fix existing resources.
- **modify** can correct supported properties and remediate existing resources when configured with identity and permissions.
- **deployIfNotExists** can deploy missing related resources or settings.

Example remediation command pattern:

```bash
az policy remediation create \
  --name remediate-allowed-locations \
  --policy-assignment "/subscriptions/<subscription-id>/providers/Microsoft.Authorization/policyAssignments/<assignment-name>" \
  --resource-discovery-mode ReEvaluateCompliance
```

The exact scope and parameters depend on the assignment scope and remediation target. In practice, you often reference the policy assignment resource ID and let Azure discover noncompliant resources within scope.

**Common governance scenarios for AZ-305**

Azure architecture questions often focus on practical governance controls rather than policy syntax alone.

Typical scenarios include:

- **Allowed locations**
  - Prevent deployments in unsupported regions.
  - Use this for data residency and operational consistency.
- **SKU restrictions**
  - Block expensive or unsupported SKUs.
  - Useful for cost control and standardization.
- **Resource type restrictions**
  - Allow only approved services.
  - Useful in regulated environments or landing zones.
- **Required tags**
  - Enforce ownership, cost center, environment, or application metadata.
  - Useful for chargeback and lifecycle management.
- **Diagnostic settings**
  - Ensure logs and metrics are sent to Log Analytics, Event Hub, or storage.
  - Useful for security and operations.
- **Private endpoint or public access restrictions**
  - Reduce exposure by enforcing network design standards.
- **TLS or secure transfer settings**
  - Enforce secure transport where supported.

Example allowed locations policy pattern:

```json
{
  "mode": "All",
  "parameters": {
    "allowedLocations": {
      "type": "Array"
    }
  },
  "policyRule": {
    "if": {
      "not": {
        "field": "location",
        "in": "[parameters('allowedLocations')]"
      }
    },
    "then": {
      "effect": "deny"
    }
  }
}
```

Example SKU restriction pattern:

```json
{
  "mode": "All",
  "parameters": {
    "allowedSkus": {
      "type": "Array"
    }
  },
  "policyRule": {
    "if": {
      "field": "sku.name",
      "notIn": "[parameters('allowedSkus')]"
    },
    "then": {
      "effect": "deny"
    }
  }
}
```

Example resource type restriction pattern:

```json
{
  "mode": "All",
  "policyRule": {
    "if": {
      "field": "type",
      "notIn": [
        "Microsoft.Storage/storageAccounts",
        "Microsoft.KeyVault/vaults",
        "Microsoft.Compute/virtualMachines"
      ]
    },
    "then": {
      "effect": "deny"
    }
  }
}
```

These examples show the design pattern, but in real implementations you should validate the exact aliases and resource provider support for the target resource types.

**Built-in policies versus custom policies**

Use built-in policies whenever they meet the requirement. Built-ins are easier to assign, maintain, and audit.

Choose custom policies when you need:

- organization-specific rules
- unsupported combinations of conditions
- custom parameterization
- special business logic
- a control not covered by built-ins

A practical governance strategy is:

- Start with built-in policies for common controls.
- Add custom policies only for gaps.
- Group both into initiatives for consistent assignment.

**Policy limitations and unsupported scenarios**

Azure Policy is powerful, but it is not universal.

Be aware of these limitations:

- Not every resource property has a usable alias.
- Some resource providers expose limited policy support.
- Some settings are only enforceable at deployment time, not after creation.
- Some legacy resources may not be fully remediable.
- Cross-resource or cross-subscription logic is limited compared with custom automation.
- Policy is not a replacement for application-level validation.

If a requirement depends on unsupported aliases or provider behavior, you may need to combine policy with:

- Azure Resource Manager templates
- Bicep
- Azure Automation
- Azure Functions
- CI/CD pipeline checks
- custom scripts

**Guest Configuration and machine configuration**

For broader governance, Azure also supports guest-level configuration for machines. This is useful when you need to validate settings inside the operating system rather than only at the Azure resource layer.

Use this when you need to check things like:

- OS configuration
- installed software
- local security settings
- registry or file-based compliance

This is more specialized than standard Azure Policy and is typically used for deeper compliance scenarios. For AZ-305, know that Azure governance can extend beyond ARM resource properties when machine-level compliance is required.

**Azure Blueprints and modern governance design**

Azure Blueprints was historically used to package governance artifacts such as policy assignments, role assignments, and templates. It is now retired for new use, so modern designs should not depend on it.

Current governance designs typically use:

- **management groups**
- **Azure Policy**
- **initiatives**
- **RBAC**
- **IaC** such as Bicep or Terraform
- **exemptions**
- **automation for remediation**

This matters for AZ-305 because you should recommend current, supportable patterns rather than legacy packaging approaches.

**CLI and operational commands**

You should know the common Azure CLI commands used to manage policy governance.

List policy definitions:

```bash
az policy definition list --query "[].{name:name, displayName:displayName}" -o table
```

Create a custom policy definition:

```bash
az policy definition create \
  --name allowed-locations \
  --display-name "Allowed locations" \
  --rules allowed-locations.json \
  --params allowed-locations-params.json \
  --mode All
```

Assign a policy at subscription scope:

```bash
az policy assignment create \
  --name enforce-allowed-locations \
  --display-name "Enforce allowed locations" \
  --policy allowed-locations \
  --scope /subscriptions/<subscription-id> \
  --params '{"allowedLocations":{"value":["eastus","westus2"]}}'
```

Create an initiative assignment:

```bash
az policy set-definition create \
  --name enterprise-baseline \
  --display-name "Enterprise baseline" \
  --definitions baseline-set.json
```

Trigger a compliance scan:

```bash
az policy state trigger-scan --subscription <subscription-id>
```

List noncompliant resources:

```bash
az policy state list \
  --filter "ComplianceState eq 'NonCompliant'" \
  -o table
```

Create an exemption:

```bash
az policy exemption create \
  --name temporary-exception \
  --scope /subscriptions/<subscription-id>/resourceGroups/<rg-name> \
  --policy-assignment /subscriptions/<subscription-id>/providers/Microsoft.Authorization/policyAssignments/<assignment-name> \
  --exemption-category Waiver
```

These commands are useful for operational familiarity, but always verify the exact scope and resource IDs in your environment.

**Design guidance for AZ-305 scenarios**

When you design policy-driven governance, think in terms of control objectives.

Use this decision logic:

1. Identify the business requirement.
2. Decide whether the control should prevent, detect, or correct.
3. Choose the right effect.
4. Assign at the right scope.
5. Parameterize for reuse.
6. Add exemptions for approved exceptions.
7. Use initiatives for grouped standards.
8. Plan remediation for existing resources.

A few practical recommendations:

- Use management groups for enterprise-wide baselines.
- Use initiatives for regulatory or framework-based controls.
- Use deny sparingly for high-confidence rules.
- Use audit first when introducing new governance.
- Use modify and deployIfNotExists only when you can support the identity and permissions model.
- Use exemptions instead of weakening the policy definition.
- Document ownership for every policy and initiative.

**Exam-focused distinctions**

Know the difference between these commonly confused concepts:

- **deny vs audit**
  - deny blocks deployment
  - audit records noncompliance without blocking
- **append vs modify**
  - append adds request properties
  - modify changes request properties and can remediate when configured
- **not-scope vs exemption**
  - not-scope excludes a scope from assignment
  - exemption records an approved exception
- **policy vs initiative**
  - policy is one rule
  - initiative is a grouped set of rules
- **compliance scan vs remediation**
  - scan reports state
  - remediation fixes supported noncompliance

**Practical governance topology**

A common enterprise topology looks like this:

- **Tenant root management group**
  - platform management group
  - landing zones management group
  - sandbox management group
- **Platform subscriptions**
  - identity
  - connectivity
  - management
- **Workload subscriptions**
  - production
  - nonproduction
  - regulated workloads

Policy assignments are often layered:

- root management group: global baseline
- landing zones: standard workload controls
- subscription: workload-specific controls
- resource group: exception handling or special cases

This structure helps you enforce consistency while preserving workload autonomy.

**Key takeaways**

Azure Policy is the primary tool for policy-driven governance in Azure. For AZ-305, focus on how you design governance, not just how you write JSON.

Remember these essentials:

- Use **initiatives** for grouped standards and compliance frameworks.
- Use **parameters** to make policies reusable across environments.
- Use **exemptions** for approved exceptions and auditability.
- Use **managed identities** for remediation-capable effects.
- Use **remediation** to correct existing resources when supported.
- Use **management groups** to scale governance across subscriptions.
- Validate alias and provider support before relying on a custom policy.
