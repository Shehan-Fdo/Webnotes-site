## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Azure landing zone subscription model**

In the Cloud Adoption Framework (CAF) and Azure Landing Zone (ALZ) reference architecture, you separate responsibilities into subscriptions so that governance, billing, access control, and operational ownership stay clear. The most important distinction is between **platform subscriptions** and **workload subscriptions**.

- **Platform subscriptions** host shared capabilities used by many workloads.
- **Workload subscriptions** host application-specific resources owned by a product or application team.
- **Subscription vending** is the controlled process for creating new workload subscriptions with standard guardrails.
- **Shared services** often include identity, management, security tooling, and common platform services.
- **Connectivity** usually includes hub networking, firewalls, DNS forwarding, and on-premises connectivity.

This separation is not just organizational. It reduces blast radius, supports delegated administration, and lets you apply different policy, RBAC, and network controls depending on the resource type.

**Platform subscriptions vs workload subscriptions**

A common ALZ pattern uses a small number of centrally managed platform subscriptions and many delegated workload subscriptions. The platform layer typically includes:

- **Connectivity subscription**
  - Virtual WAN or hub virtual network
  - Azure Firewall
  - VPN and ExpressRoute gateways
  - Route tables and shared network appliances
  - DNS forwarding and private name resolution components
- **Identity subscription**
  - Identity-related infrastructure that must be centrally controlled
  - Domain controllers if you still run them in Azure
  - Identity synchronization components such as Microsoft Entra Connect or Cloud Sync agents when required
  - Supporting services for authentication and directory integration
- **Management subscription**
  - Log Analytics workspaces
  - Microsoft Sentinel
  - Automation, update management, monitoring, and backup tooling
  - Central policy and compliance reporting

Workload subscriptions usually contain:

- Application resource groups
- App service plans, AKS clusters, databases, storage accounts, and app networking
- Team-owned identities and managed identities for the application
- Delegated RBAC for application operators and developers

The key design question is whether a service is shared across many workloads or tightly coupled to one workload. If it is shared and needs consistent governance, place it in a platform subscription. If it is application-specific, place it in the workload subscription.

**Subscription vending and governance**

Subscription vending is the automated or semi-automated process that creates a new subscription with the right baseline controls. In enterprise landing zones, you do not hand-create subscriptions ad hoc. Instead, you provision them through a governed workflow that applies:

- Management group placement
- Policy assignments
- RBAC role assignments
- Budget and tagging standards
- Network connectivity defaults
- Diagnostic settings and logging destinations

A typical vending workflow:

1. A team requests a new subscription for a workload.
2. An automation pipeline creates or assigns the subscription.
3. The subscription is placed under the correct management group.
4. Baseline Azure Policy initiatives are assigned.
5. Standard role assignments are applied to the workload team.
6. Network and DNS integration are configured.
7. Cost management and monitoring are enabled.

This model helps you enforce guardrails without blocking delivery. It also makes it easier to prove compliance because every workload subscription starts from the same baseline.

> Use subscription vending when you need repeatable governance at scale. Avoid manual subscription creation for enterprise workloads unless you have a very small environment.

**Governance boundaries and ownership**

A strong landing zone design clearly separates what is centrally managed from what is delegated.

- Centrally managed resources usually include:
  - Hub networking
  - Firewalls and routing policy
  - Shared DNS infrastructure
  - Central logging and security monitoring
  - Policy definitions and assignments
  - ExpressRoute circuits and shared connectivity services
- Delegated resources usually include:
  - Application VNets inside workload subscriptions
  - App-specific NSGs, route tables, and private endpoints
  - Application identities and secrets
  - Data services owned by the application team
  - Resource group-level deployment permissions

This boundary affects **RBAC** and **policy** design:

- Give platform teams subscription-level or resource-group-level control over shared services.
- Give workload teams contributor access only within their own subscriptions or resource groups.
- Use Azure Policy to prevent workload teams from bypassing required controls such as region restrictions, private endpoint requirements, or diagnostic settings.
- Avoid giving workload teams direct control over shared firewalls, hub route tables, or central DNS zones unless they are explicitly delegated operators.

**Connectivity subscription design**

The connectivity subscription is the network backbone of the landing zone. It connects Azure workloads to each other, to shared services, and to on-premises networks. The design depends on scale and topology.

Common responsibilities include:

- Hub-and-spoke networking
- Azure Virtual WAN deployment
- Azure Firewall or third-party network virtual appliances
- VPN and ExpressRoute connectivity
- Shared DNS resolution
- Network segmentation and routing control

For smaller environments, a classic **hub-and-spoke** model is often sufficient. For larger or globally distributed environments, **Azure Virtual WAN** can simplify routing and branch connectivity.

**Classic hub-and-spoke architecture**

In a classic design, the connectivity subscription contains a central hub virtual network. Workload subscriptions host spoke VNets that peer to the hub.

Typical hub components:

- Azure Firewall
- VPN gateway or ExpressRoute gateway
- Bastion host
- DNS forwarders or Azure DNS Private Resolver
- Shared management jump services if required

Typical spoke components:

- Application VNets
- Private endpoints
- App-specific subnets
- User-defined routes to force traffic through the hub

A common peering pattern is:

```bash
az network vnet peering create \
  --resource-group rg-connectivity \
  --vnet-name vnet-hub \
  --name hub-to-spoke1 \
  --remote-vnet /subscriptions/<sub-id>/resourceGroups/rg-app1/providers/Microsoft.Network/virtualNetworks/vnet-app1 \
  --allow-vnet-access \
  --allow-forwarded-traffic \
  --allow-gateway-transit

az network vnet peering create \
  --resource-group rg-app1 \
  --vnet-name vnet-app1 \
  --name spoke1-to-hub \
  --remote-vnet /subscriptions/<hub-sub-id>/resourceGroups/rg-connectivity/providers/Microsoft.Network/virtualNetworks/vnet-hub \
  --allow-vnet-access \
  --allow-forwarded-traffic \
  --use-remote-gateways
```

Important peering settings:

- `--allow-vnet-access` enables direct VNet-to-VNet connectivity.
- `--allow-forwarded-traffic` allows traffic forwarded by appliances such as Azure Firewall.
- `--allow-gateway-transit` lets the hub share its gateway with spokes.
- `--use-remote-gateways` lets the spoke use the hub gateway.

Use gateway transit when you want one shared VPN or ExpressRoute gateway in the hub instead of deploying gateways in every spoke.

**Azure Virtual WAN in enterprise connectivity**

Azure Virtual WAN is a managed networking service that simplifies large-scale branch, user, and VNet connectivity. It is often preferred when you need many sites, many regions, or centralized security with less operational overhead than classic hub-and-spoke.

Virtual WAN concepts:

- **Virtual WAN** is the global container.
- **Virtual hub** is the regional managed hub.
- **Secured virtual hub** adds integrated security, typically with Azure Firewall.
- **Hub routing** controls traffic between branches, VNets, and the internet.
- **Connections** attach VNets, VPN sites, ExpressRoute circuits, and remote users.

Use Virtual WAN when:

- You have many branches or remote sites.
- You need multi-region hub connectivity with consistent routing.
- You want simplified branch-to-branch and branch-to-Azure connectivity.
- You want a managed hub model instead of operating your own hub VNet.

Use classic hub-and-spoke when:

- You need maximum control over routing and appliances.
- Your environment is smaller or simpler.
- You already have a mature hub VNet design and do not need Virtual WAN features.

A secured virtual hub often replaces the classic hub firewall pattern. Traffic can be routed through the hub firewall using routing intent or hub route tables, depending on the design.

Practical design points:

- Place Virtual WAN resources in the connectivity subscription.
- Use one or more regional hubs for latency and resiliency.
- Attach workload VNets to the nearest hub when possible.
- Use hub route tables to control propagation and segmentation.
- Use secured hubs when you want centralized inspection without deploying separate hub firewalls.

**Routing, UDRs, and propagation**

Routing is one of the main reasons to centralize connectivity. You need to control how traffic reaches the internet, on-premises networks, and shared services.

Key mechanisms:

- **User-defined routes (UDRs)** force subnet traffic to a next hop such as Azure Firewall.
- **Route propagation** determines whether routes learned from gateways or hubs are automatically available.
- **System routes** are Azure-managed default routes.
- **BGP** (Border Gateway Protocol) distributes routes from VPN/ExpressRoute gateways.

Example UDR to send all outbound traffic to Azure Firewall:

```bash
az network route-table create \
  --resource-group rg-connectivity \
  --name rt-spoke-egress

az network route-table route create \
  --resource-group rg-connectivity \
  --route-table-name rt-spoke-egress \
  --name default-to-firewall \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.0.0.4
```

Associate the route table to a spoke subnet:

```bash
az network vnet subnet update \
  --resource-group rg-app1 \
  --vnet-name vnet-app1 \
  --name snet-app \
  --route-table rt-spoke-egress
```

When designing route propagation:

- Disable propagation where you need strict segmentation.
- Allow propagation where shared routes should reach spokes automatically.
- Verify that forced tunneling does not break private endpoint access or platform dependencies.
- Test asymmetric routing carefully when firewalls and gateways are involved.

**Azure Firewall and policy association**

Azure Firewall is commonly deployed in the connectivity subscription because it is a shared control point for egress, ingress, and east-west inspection.

Use it for:

- Central outbound filtering
- DNAT for inbound publishing
- Network and application rule enforcement
- Threat intelligence filtering
- Logging and policy-based governance

A firewall policy lets you separate rule management from the firewall resource itself. This is useful when different teams manage different rule sets.

Example policy association:

```bash
az network firewall policy create \
  --resource-group rg-connectivity \
  --name fwpol-central

az network firewall create \
  --resource-group rg-connectivity \
  --name azfw-hub \
  --location eastus \
  --firewall-policy fwpol-central
```

Design guidance:

- Keep the firewall in the connectivity subscription.
- Use firewall policy for centralized rule governance.
- Delegate rule collection management only if your operating model supports it.
- Log all firewall traffic to a central workspace in the management subscription.

**DNS architecture and Azure DNS Private Resolver**

DNS is a critical shared service in landing zones because private endpoints, hybrid name resolution, and cross-network service discovery all depend on it.

You typically choose between:

- **Custom DNS servers** such as Windows Server DNS or BIND
- **Azure DNS Private Resolver** for managed inbound and outbound resolution
- A hybrid model where custom DNS remains authoritative for some zones

Use custom DNS when:

- You already have enterprise DNS infrastructure.
- You need advanced conditional forwarding or legacy integrations.
- You require tight control over DNS server behavior.

Use Azure DNS Private Resolver when:

- You want managed DNS forwarding without running VM-based DNS servers.
- You need private DNS resolution between on-premises and Azure.
- You want inbound and outbound endpoints for hybrid name resolution.

Azure DNS Private Resolver architecture:

- **Inbound endpoint** receives DNS queries from on-premises or other networks.
- **Outbound endpoint** forwards Azure-originated queries to on-premises DNS or other resolvers.
- **DNS forwarding ruleset** maps domains to target DNS servers.
- **Private DNS zones** resolve Azure private endpoint names.

A common pattern is:

- Deploy the resolver in the connectivity subscription.
- Place inbound and outbound endpoints in dedicated subnets.
- Link private DNS zones to workload VNets as needed.
- Use forwarding rules for on-premises zones such as `corp.contoso.com`.

Example conceptual setup:

```bash
az network dns-resolver create \
  --resource-group rg-connectivity \
  --name dnsr-central \
  --location eastus

az network dns-resolver inbound-endpoint create \
  --resource-group rg-connectivity \
  --dns-resolver-name dnsr-central \
  --name inbound1 \
  --subnet /subscriptions/<sub-id>/resourceGroups/rg-connectivity/providers/Microsoft.Network/virtualNetworks/vnet-hub/subnets/snet-dns-in

az network dns-resolver outbound-endpoint create \
  --resource-group rg-connectivity \
  --dns-resolver-name dnsr-central \
  --name outbound1 \
  --subnet /subscriptions/<sub-id>/resourceGroups/rg-connectivity/providers/Microsoft.Network/virtualNetworks/vnet-hub/subnets/snet-dns-out
```

Practical DNS guidance:

- Use private DNS zones for Azure private endpoints.
- Link zones to the VNets that need resolution.
- Forward on-premises zones to enterprise DNS or resolver targets.
- Avoid placing DNS servers in every workload subscription unless the workload has a special requirement.
- Keep DNS forwarding rules and resolver endpoints under central governance.

**Identity-related platform services**

Identity is often treated as a platform concern because it supports many workloads and must be protected carefully. In Azure landing zones, identity-related infrastructure may include:

- Domain controllers for legacy Active Directory Domain Services
- Microsoft Entra Connect synchronization servers
- Microsoft Entra Cloud Sync agents
- Federation or authentication appliances if required
- Supporting DNS and management components

Placement guidance:

- Put identity infrastructure in a centrally managed platform subscription or a dedicated identity subscription.
- Keep it isolated from workload subscriptions.
- Use separate subnets and strict NSGs.
- Protect it with backup, monitoring, and patching controls.
- Avoid mixing identity infrastructure with application workloads.

If you still run domain controllers in Azure, they usually belong in the shared services or identity platform because many workloads may depend on them. If you use Microsoft Entra ID only, you may not need these servers at all, but you still need to plan for identity-related network dependencies such as DNS and secure admin access.

**Management and monitoring subscription**

The management subscription is another platform subscription that often sits alongside connectivity and identity. It is not the focus of networking, but it matters because shared services generate logs and telemetry that must be centrally collected.

Typical services:

- Log Analytics workspaces
- Microsoft Sentinel
- Azure Monitor
- Automation accounts
- Update management
- Backup vaults

Why it matters for connectivity:

- Firewall logs, route diagnostics, and DNS logs should flow to central monitoring.
- Shared services need consistent alerting and incident response.
- Centralized telemetry supports compliance and troubleshooting.

**Subscription limits and why they matter**

Azure subscriptions have practical limits and quotas that influence architecture. Even when a limit is not immediately reached, it can shape how you split platform services.

Examples of why you split subscriptions:

- You want to avoid hitting resource or policy scale limits in a single subscription.
- You need separate billing and chargeback for platform and workloads.
- You want different access boundaries for network, identity, and app teams.
- You need to isolate high-change workloads from stable shared services.
- You want to reduce the operational impact of accidental changes.

Common design implications:

- Put connectivity in its own subscription so route tables, gateways, and firewalls do not compete with application resources.
- Put shared services in separate subscriptions when they have different lifecycle or ownership.
- Keep workload subscriptions smaller and more uniform for easier governance.
- Use management groups to apply policy consistently across many subscriptions.

**Exam-focused design choices**

For AZ-305, you should recognize the tradeoffs between centralization and delegation.

Choose centralization when:

- The service is shared by many workloads.
- The service requires strict security or compliance controls.
- The service is expensive or operationally complex.
- The service must be highly available and consistently monitored.

Choose delegation when:

- The service is specific to one workload.
- The workload team needs autonomy for rapid change.
- The resource does not create broad blast radius.
- The team can safely manage it within policy guardrails.

Typical centrally managed resources:

- Hub VNets or Virtual WAN hubs
- Firewalls and route control
- DNS resolvers and private DNS zones
- ExpressRoute and VPN gateways
- Central logging and security tooling
- Identity infrastructure

Typical delegated resources:

- App VNets and subnets
- Private endpoints for the workload
- App-specific NSGs and UDRs
- Managed identities and app secrets
- Data services owned by the application team

**Practical topology example**

A common enterprise landing zone layout looks like this:

- **Management group**
  - Platform
    - Connectivity subscription
    - Identity subscription
    - Management subscription
  - Landing zones
    - Workload subscription A
    - Workload subscription B
    - Workload subscription C

Traffic flow example:

- Workload subnet sends internet-bound traffic to Azure Firewall.
- Workload subnet resolves private endpoints through Azure DNS Private Resolver.
- Workload subnet reaches on-premises through ExpressRoute or VPN in the hub.
- Central monitoring collects logs from all subscriptions.

**Troubleshooting and validation**

When validating a shared services and connectivity design, check:

- Whether workload VNets can reach shared services through the intended path.
- Whether DNS resolves private endpoints correctly.
- Whether UDRs and gateway transit produce the expected next hop.
- Whether firewall policy allows required traffic without overexposing the network.
- Whether RBAC prevents workload teams from changing central controls.

Useful validation commands:

```bash
az network nic show-effective-route-table \
  --resource-group rg-app1 \
  --name nic-app1

az network nic show-effective-nsg \
  --resource-group rg-app1 \
  --name nic-app1

az network watcher test-connectivity \
  --source-resource /subscriptions/<sub-id>/resourceGroups/rg-app1/providers/Microsoft.Network/networkInterfaces/nic-app1 \
  --dest-address 10.0.0.4 \
  --protocol TCP \
  --dest-port 53
```

These checks help you confirm whether routing, NSGs, and DNS are aligned with the landing zone design.

**Key takeaways**

- Platform subscriptions host shared, centrally governed services.
- Workload subscriptions host application-specific resources with delegated ownership.
- Subscription vending standardizes new workload onboarding and policy enforcement.
- Connectivity subscriptions usually contain hub networking, firewalls, gateways, and DNS services.
- Azure Virtual WAN is a strong choice for large, distributed enterprise connectivity.
- Azure DNS Private Resolver reduces the need for VM-based DNS forwarding infrastructure.
- Identity infrastructure belongs in a centrally managed platform boundary when it serves multiple workloads.
- Subscription limits, governance, and operational ownership all influence how you split services across subscriptions.
