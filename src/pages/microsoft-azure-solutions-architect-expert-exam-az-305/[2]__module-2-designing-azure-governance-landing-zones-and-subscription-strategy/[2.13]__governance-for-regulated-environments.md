## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Governance model for regulated environments**

Regulated environments require you to prove control, not just implement control. In Azure, that means designing for policy enforcement, identity governance, network isolation, data residency, logging, and auditability across the full landing zone.

A strong governance design usually combines:

- **Management groups** for enterprise-wide policy inheritance
- **Subscriptions** for workload and environment separation
- **Resource groups** for lifecycle and delegation boundaries
- **Azure Policy** for preventive and detective control
- **Role-based access control (RBAC)** for least privilege
- **Locks** for change control on critical resources
- **Centralized logging** for audit evidence and incident response

In exam scenarios, look for requirements such as financial services, healthcare, public sector, export controls, or internal audit. These usually imply stricter controls around data location, privileged access, encryption, retention, and network exposure.

**Landing zones and scope hierarchy**

Azure governance works best when you align control scope to the Azure hierarchy. Management groups let you apply standards once and inherit them downward, which is essential for enterprise-scale landing zones.

Typical hierarchy:

- **Tenant root group**
  - **Platform management group**
    - Identity subscription
    - Connectivity subscription
    - Management subscription
  - **Landing zone management group**
    - Production subscriptions
    - Nonproduction subscriptions
  - **Sandbox management group**
    - Experimental subscriptions

This structure helps you separate shared services from workloads while keeping policy inheritance predictable. It also supports different control levels for regulated and nonregulated workloads.

Key design principles:

- Use **management groups** for global guardrails.
- Use **subscriptions** to isolate billing, quotas, and blast radius.
- Use **resource groups** for deployment and lifecycle management.
- Avoid placing all workloads in one subscription when compliance boundaries differ.

For enterprise-scale landing zones, governance often starts at the tenant root and management group layer, then flows down into subscription templates, policy assignments, and role assignments.

**Azure Policy for compliance and prevention**

Azure Policy is the primary control plane for enforcing standards. It evaluates resources against rules and can deny, audit, append, modify, or deploy supporting resources.

Common policy effects:

- **Deny** blocks noncompliant deployments.
- **Audit** records noncompliance without blocking.
- **Append** adds fields to requests where supported.
- **Modify** changes request properties before deployment.
- **DeployIfNotExists** deploys a related resource if missing.

Use Azure Policy when you need to:

- Restrict allowed regions
- Enforce tagging
- Require private endpoints
- Require encryption settings
- Block public IPs on sensitive workloads
- Enforce diagnostic settings
- Deploy baseline resources such as monitoring agents or log settings

Example policy assignment at management group scope:

```bash
az policy assignment create \
  --name "deny-nonapproved-regions" \
  --display-name "Deny nonapproved regions" \
  --scope "/providers/Microsoft.Management/managementGroups/contoso-root" \
  --policy "/providers/Microsoft.Authorization/policyDefinitions/allowed-locations" \
  --params '{
    "listOfAllowedLocations": {
      "value": ["eastus", "eastus2", "centralus"]
    }
  }'
```

For initiatives, use a policy set definition. The assignment references the initiative, not an individual policy definition.

```bash
az policy assignment create \
  --name "regulated-baseline" \
  --display-name "Regulated baseline" \
  --scope "/providers/Microsoft.Management/managementGroups/contoso-root" \
  --policy-set-definition "regulated-baseline-initiative"
```

Policy design notes:

- Use **initiatives** to group related policies into a single control package.
- Prefer **audit** first when introducing controls to avoid breaking existing workloads.
- Use **deny** for hard requirements such as region restrictions or public network exposure.
- Use **modify** and **DeployIfNotExists** when you need automatic remediation.

> For `DeployIfNotExists` and `Modify`, the assignment usually needs a managed identity with sufficient permissions to create or update the target resource. Without that identity and role assignment, remediation fails.

**Policy remediation and managed identity**

Remediation is important in regulated environments because compliance is not only about future deployments. You also need to bring existing resources into alignment.

Typical remediation workflow:

1. Assign the policy or initiative at the correct scope.
2. Enable a managed identity on the assignment if the effect requires it.
3. Grant the identity the minimum role needed for remediation.
4. Create a remediation task for existing noncompliant resources.
5. Review compliance results and exceptions.

Example remediation-related role assignment:

```bash
az role assignment create \
  --assignee-object-id <managed-identity-object-id> \
  --role "Contributor" \
  --scope "/subscriptions/<subscription-id>/resourceGroups/<rg-name>"
```

Use a narrower role when possible. For example, a custom role that only allows diagnostic settings or tag updates is better than broad Contributor access.

Common exam scenario:

- You need all storage accounts to have diagnostic settings enabled.
- The policy uses `DeployIfNotExists`.
- The assignment identity must be able to create diagnostic settings on the target resource.
- You then run remediation to apply the setting to existing accounts.

**Azure Policy versus resource locks**

Policy and locks solve different problems.

| Control | Primary purpose | Best use case | Limitation |
|---|---|---|---|
| Azure Policy | Enforce configuration and compliance | Prevent insecure or nonstandard deployments | Does not stop all changes after deployment |
| Resource lock | Prevent deletion or modification | Protect critical resources from accidental change | Does not validate configuration compliance |

Use **CanNotDelete** locks for critical shared resources such as key vaults, log analytics workspaces, or hub networking components. Use **ReadOnly** locks sparingly because they can interfere with operations and automation.

In regulated environments:

- Use **policy** to enforce standards.
- Use **locks** to protect change-controlled assets.
- Use both when the resource is both sensitive and operationally critical.

Example scenario:

- A production Key Vault should not be deleted.
- A lock prevents accidental deletion.
- Policy enforces soft delete, purge protection, private endpoint use, and logging.

**Identity governance and privileged access**

Identity is a major control point in regulated environments. Azure uses Microsoft Entra ID for authentication and authorization, and governance often extends beyond Azure RBAC into privileged access workflows.

Important controls:

- **Privileged Identity Management (PIM)** for just-in-time elevation
- **Conditional Access** for device, location, and risk-based access
- **Administrative units** for delegated administration boundaries
- **Cross-tenant access settings** for B2B collaboration control
- **Tenant restrictions** to limit access to approved tenants or apps

Use PIM when administrators should not have standing privileged access. This is common for subscription owners, security admins, and key vault administrators.

Example governance pattern:

- Users have no permanent Owner role.
- They activate Owner through PIM for a limited time.
- Activation requires approval and MFA.
- Activity is logged for audit.

Administrative units are useful when you need delegated administration for a subset of users or groups without granting tenant-wide permissions. They are especially relevant in large regulated organizations with regional or business-unit separation.

**Network governance for regulated workloads**

Network design is a core part of compliance because it controls exposure, inspection, and segmentation. Regulated workloads usually require private connectivity, restricted egress, and centralized inspection.

Common controls:

- **Azure Firewall** for centralized outbound and inbound inspection
- **Network security groups (NSGs)** for subnet and NIC-level filtering
- **Private Link** for private access to PaaS services
- **Private DNS** for name resolution of private endpoints
- **Service endpoints** for limited service access over the Azure backbone
- **User-defined routes (UDRs)** for forced tunneling through inspection points

A common regulated topology uses a hub-and-spoke design:

- Hub contains firewall, DNS, and shared connectivity.
- Spokes host application workloads.
- Spokes have no direct internet exposure.
- Private endpoints connect workloads to PaaS services.
- DNS resolves service names to private IP addresses.

Practical guidance:

- Prefer **Private Link** when you need the strongest isolation for PaaS access.
- Use **NSGs** to restrict east-west traffic and subnet exposure.
- Use **Azure Firewall** for centralized egress control and logging.
- Use **private DNS zones** so private endpoints resolve correctly from workloads.
- Use **service endpoints** only when private endpoints are not required and the service supports the scenario.

Example NSG rule creation:

```bash
az network nsg rule create \
  --resource-group rg-network \
  --nsg-name nsg-app \
  --name AllowAppToSql \
  --priority 200 \
  --direction Outbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes VirtualNetwork \
  --source-port-ranges "*" \
  --destination-address-prefixes Sql \
  --destination-port-ranges 1433
```

Example private endpoint concept:

```bash
az network private-endpoint create \
  --name pe-kv-prod \
  --resource-group rg-network \
  --vnet-name vnet-hub \
  --subnet subnet-private-endpoints \
  --private-connection-resource-id <key-vault-resource-id> \
  --group-ids vault \
  --connection-name kv-privatelink
```

**Data residency and sovereignty**

Data residency means data stays in approved geographic boundaries. Sovereignty adds legal and operational constraints around who can access data, where it is processed, and how it crosses borders.

Design considerations:

- Choose Azure regions that satisfy residency requirements.
- Avoid unnecessary geo-replication across jurisdictions.
- Understand service-specific replication behavior.
- Control backup and disaster recovery region selection.
- Review logging and telemetry destinations, because logs can also contain regulated data.

Important exam point: not all Azure services replicate data the same way. Some services store metadata or backups in paired regions, and some features may not be available in every region. You must validate the service’s data handling model before deployment.

Practical controls:

- Restrict allowed locations with Azure Policy.
- Use region-specific resource groups and subscriptions.
- Keep backups in approved regions only.
- Review cross-border access for support, operations, and monitoring.
- Use customer-managed keys when required by policy or contract.

If a workload must remain within a country or legal boundary, design for:

- Single-region deployment where possible
- Region-paired disaster recovery only if the pair is compliant
- Localized logging and backup retention
- Strict access controls for operators outside the boundary

**Key Vault governance**

Azure Key Vault is central to regulated environments because it protects secrets, keys, and certificates. Governance must cover both access and lifecycle.

Key controls:

- **Soft delete** protects against accidental deletion.
- **Purge protection** prevents permanent deletion during retention.
- **RBAC** or access policies control who can manage vault contents.
- **Key rotation** reduces exposure from long-lived keys.
- **Private endpoint** reduces public exposure.
- **Diagnostic logging** supports audit and incident response.

Access model considerations:

- Use **Azure RBAC** when you want centralized role management and consistent governance.
- Use **access policies** when you need vault-specific permissions and the environment still relies on that model.
- Avoid mixing models without a clear operational standard, because it complicates troubleshooting and audit evidence.

Customer-managed keys (CMK) are often required for regulated data. With CMK:

- You control the encryption key lifecycle.
- You can rotate keys on a schedule.
- You must plan for key availability because service access can depend on the vault.
- You should monitor key expiration and access failures.

Example governance checklist for Key Vault:

- Enable soft delete.
- Enable purge protection.
- Restrict public network access.
- Use private endpoint.
- Turn on diagnostic logs.
- Assign least-privilege roles.
- Document key rotation and recovery procedures.

**Logging, monitoring, and audit evidence**

Regulated environments need centralized, tamper-resistant evidence. Azure Monitor, Log Analytics, and Microsoft Sentinel are common building blocks.

Use logging to answer:

- Who changed what?
- When did the change happen?
- Was the change approved?
- Did the control detect or block the change?
- Was the workload compliant at the time of audit?

Typical evidence sources:

- **Azure Activity Log** for subscription-level control plane events
- **Resource logs** for service-specific operational events
- **Azure Policy compliance data** for control status
- **Microsoft Defender for Cloud** for security posture and recommendations
- **Microsoft Sentinel** for SIEM (Security Information and Event Management) correlation and incident response

Example diagnostic settings to Log Analytics:

```bash
az monitor diagnostic-settings create \
  --name "send-to-law" \
  --resource <resource-id> \
  --workspace <log-analytics-workspace-id> \
  --logs '[{"category":"AuditEvent","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

> Diagnostic categories vary by resource provider. Always verify the available categories for the specific service before writing automation or exam answers.

For compliance reporting, auditors often want:

- Policy compliance summaries
- Change history from Activity Log
- Security alerts and recommendations
- Evidence of retention settings
- Proof of access reviews and privileged activations

Microsoft Sentinel is useful when you need centralized correlation across Azure, on-premises, and third-party systems. It helps connect policy violations, sign-in anomalies, firewall events, and workload alerts into a single investigation workflow.

**Compliance reporting workflow**

A practical audit workflow usually combines multiple sources rather than relying on one report.

1. Review Azure Policy compliance for preventive and detective control status.
2. Pull Azure Activity Log entries for change evidence.
3. Check Log Analytics for resource logs and retention settings.
4. Review Defender for Cloud recommendations and secure score trends.
5. Correlate incidents and alerts in Microsoft Sentinel if centralized monitoring is required.
6. Export evidence for auditors with timestamps, scope, and ownership context.

This workflow matters because auditors usually want both control design and proof of operation. A policy assignment alone is not enough; you need evidence that the control is active, inherited, and monitored.

**Resource governance and change control**

Resource governance in regulated environments often includes deployment standards, naming, tagging, and approval workflows.

Common controls:

- Mandatory tags such as `CostCenter`, `DataClassification`, `Owner`, and `Environment`
- Standardized naming conventions
- Infrastructure as Code (IaC) with review gates
- Blue/green or controlled release patterns for production changes
- Change windows and approval records

Azure Policy can enforce required tags:

```bash
az policy assignment create \
  --name "require-tags" \
  --scope "/subscriptions/<subscription-id>" \
  --policy "/providers/Microsoft.Authorization/policyDefinitions/require-tag" \
  --params '{
    "tagName": { "value": "DataClassification" }
  }'
```

Use tags to support:

- Cost allocation
- Data classification
- Ownership
- Retention requirements
- Incident response routing

**Blueprints, templates, and governance-as-code**

Azure Blueprints has been positioned as a legacy or maintenance-oriented governance approach over time, so for new designs you should favor policy, ARM templates, Bicep, Terraform, and landing zone automation patterns. The exam may still reference Blueprints conceptually, but governance-as-code is the more current design direction.

Recommended approach:

- Use **Bicep** or **Terraform** for repeatable deployments.
- Use **Azure Policy** for guardrails.
- Use **management groups** for inheritance.
- Use **PIM** and RBAC for access governance.
- Use **CI/CD** approvals for controlled promotion.

Example Bicep snippet for a policy assignment pattern:

```bicep
resource policyAssignment 'Microsoft.Authorization/policyAssignments@2022-06-01' = {
  name: 'deny-public-ip'
  scope: managementGroup('contoso-root')
  properties: {
    displayName: 'Deny public IPs'
    policyDefinitionId: '/providers/Microsoft.Authorization/policyDefinitions/<policy-definition-id>'
  }
}
```

**Tenant-level governance considerations**

Some regulated requirements start above the subscription level. The tenant is where identity, external collaboration, and global access boundaries are controlled.

Consider:

- Restricting guest access and external collaboration
- Using cross-tenant access settings for partner organizations
- Limiting app consent and enterprise application exposure
- Applying administrative units for delegated administration
- Controlling tenant-wide security defaults or Conditional Access policies

These controls matter when auditors ask how you prevent unauthorized identity sprawl or cross-border administrative access.

**Common exam scenarios**

- You need to prevent deployments outside approved regions.
  - Use Azure Policy with an allowed locations definition.
- You need to stop accidental deletion of a production Key Vault.
  - Use a resource lock and policy for vault configuration.
- You need to ensure all resources send logs to a central workspace.
  - Use `DeployIfNotExists` policy with a managed identity.
- You need just-in-time admin access for subscription owners.
  - Use PIM.
- You need private access to PaaS services from a regulated app.
  - Use Private Link, private DNS, and NSGs.
- You need audit evidence for a quarterly review.
  - Use Policy compliance, Activity Log, Log Analytics, and Sentinel.
- You need to keep data within a legal boundary.
  - Use region restrictions, approved backup locations, and service-specific residency validation.

**Practical design checklist**

- Define management group hierarchy before subscription deployment.
- Assign baseline policies at the highest practical scope.
- Use initiatives for grouped controls.
- Separate platform, shared services, and workload subscriptions.
- Enforce tags, regions, and network restrictions.
- Protect critical resources with locks.
- Use PIM for privileged roles.
- Centralize logs and retain them according to policy.
- Validate Key Vault hardening and key lifecycle controls.
- Design private connectivity for regulated PaaS access.
- Document audit evidence sources and retention.

**Key takeaways**

Regulated Azure environments are built on layered governance. Policy enforces standards, RBAC controls access, locks protect critical resources, network controls reduce exposure, and centralized logging provides audit evidence. The best designs use management groups and landing zones to apply these controls consistently across subscriptions while still allowing workload-specific exceptions where justified and documented.
