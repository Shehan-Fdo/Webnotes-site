## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Environment segmentation overview**

Environment segmentation means separating **development**, **test**, and **production** workloads so each environment can have different security, governance, cost, and reliability controls. In Azure, this is usually more than just naming resources differently. You typically isolate environments with a combination of management groups, subscriptions, resource groups, identity boundaries, network boundaries, and deployment pipelines.

The main goal is to reduce blast radius. A bad deployment, misconfigured policy, or leaked secret in dev should not affect prod. At the same time, segmentation should still support efficient delivery, shared governance, and repeatable deployments.

- **Dev** is optimized for speed, experimentation, and low cost.
- **Test** is optimized for validation, integration, and controlled release checks.
- **Prod** is optimized for availability, security, compliance, and recoverability.

**Why segmentation matters**

Segmentation helps you apply different controls where they matter most. Production often needs stricter access, longer retention, stronger backup policies, and tighter change control. Dev and test can be more permissive, but they should still follow baseline governance so they do not become a security gap.

Common benefits include:

- Reduced blast radius for outages and mistakes
- Clear separation of duties between teams
- Different cost controls per environment
- Easier compliance and audit reporting
- Safer release promotion from lower environments to prod

**Recommended Azure hierarchy**

A common enterprise pattern is to use an **Azure landing zone** approach with **enterprise-scale** governance. This usually means organizing the tenant with management groups above subscriptions, then applying policy and RBAC at the right scope.

A typical hierarchy looks like this:

```text
Tenant
└── Management Group: Platform
    ├── Subscription: Connectivity
    ├── Subscription: Identity
    └── Subscription: Management
└── Management Group: Landing Zones
    ├── Management Group: Non-Production
    │   ├── Subscription: App-Dev
    │   └── Subscription: App-Test
    └── Management Group: Production
        └── Subscription: App-Prod
```

This pattern gives you:

- **Management group** inheritance for policy and access
- Separate **subscriptions** for environment-level isolation
- Shared platform services without mixing app workloads
- A clean path to scale multiple applications and teams

> Use subscriptions as the main boundary when you need strong isolation for billing, policy, access, and quotas. Use resource groups for lifecycle grouping inside an environment, not as the primary security boundary.

**Management groups, subscriptions, and resource groups**

Each scope has a different purpose:

| Scope | Best use | What it isolates well | What it does not isolate well |
|---|---|---|---|
| Management group | Governance and inheritance | Policy, RBAC inheritance, compliance | Direct workload separation |
| Subscription | Environment or business boundary | Quotas, billing, access, policy scope | Shared tenant-level identity issues |
| Resource group | Lifecycle grouping | Deployment and deletion grouping | Strong security boundary |

A practical design is:

- Put platform services in dedicated subscriptions.
- Put dev, test, and prod workloads in separate subscriptions when possible.
- Use resource groups to group app components within each environment.
- Avoid mixing prod and non-prod resources in the same resource group.

**Identity separation**

Identity should be separated by environment as much as possible. This reduces the chance that a dev credential or managed identity can reach prod data.

Recommended practices:

- Use separate **service principals** or workload identities per environment.
- Use distinct **managed identities** for dev, test, and prod resources.
- Assign least-privilege RBAC roles at the narrowest scope possible.
- Avoid reusing the same app registration across all environments unless there is a strong reason and compensating controls.
- Keep admin access to prod tightly controlled with privileged access workflows.

For example, an app might use:

- `app-dev-mi` for development resources
- `app-test-mi` for test resources
- `app-prod-mi` for production resources

This prevents a lower environment identity from being used to access prod Key Vault or prod databases.

**Secrets and configuration separation**

Secrets should never be shared across environments. Each environment should have its own secret store, its own rotation schedule, and its own access policy.

Recommended pattern:

- Separate **Key Vaults** per environment, such as `kv-app-dev`, `kv-app-test`, and `kv-app-prod`
- Store environment-specific connection strings, certificates, and API keys in the matching vault
- Use managed identities to retrieve secrets instead of embedding credentials in code
- Rotate secrets independently based on environment risk and change frequency

Example Key Vault access pattern:

```bash
az keyvault create \
  --name kv-app-prod \
  --resource-group rg-app-prod \
  --location eastus

az keyvault set-policy \
  --name kv-app-prod \
  --object-id <prod-managed-identity-object-id> \
  --secret-permissions get list
```

A good rule is:

- Dev secrets can change often and may be short-lived.
- Test secrets should mirror prod structure but not reuse prod values.
- Prod secrets should be tightly controlled, audited, and rotated on a schedule.

**Environment-specific application settings**

Application configuration should vary by environment without changing application code. Azure services support this through app settings, connection strings, and deployment-time configuration.

For **Azure App Service**, use slot settings and app settings:

```bash
az webapp config appsettings set \
  --resource-group rg-app-prod \
  --name app-prod-web \
  --settings \
    ASPNETCORE_ENVIRONMENT=Production \
    ConnectionStrings__MainDb="Server=tcp:prod-sql.database.windows.net,1433;Database=appdb;..."

az webapp config connection-string set \
  --resource-group rg-app-prod \
  --name app-prod-web \
  --settings MainDb="Server=tcp:prod-sql.database.windows.net,1433;Database=appdb;..." \
  --connection-string-type SQLAzure
```

For **deployment slots**, mark settings as slot-specific when they should not move during swap:

```bash
az webapp deployment slot create \
  --resource-group rg-app-prod \
  --name app-prod-web \
  --slot staging
```

Use slots for safe promotion:

- Deploy to `staging`
- Run smoke tests
- Swap into production when validation passes

Slot settings are useful for:

- Connection strings
- Feature flags
- Diagnostic endpoints
- Environment-specific secrets

For **Azure Functions**, use application settings per function app and separate function apps per environment when isolation matters. For **AKS**, use separate namespaces only for light separation; use separate clusters or node pools when you need stronger isolation. Store environment config in ConfigMaps and Secrets, but keep prod secrets in a separate vault-backed process.

Example Kubernetes-style separation:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
  namespace: prod
data:
  ASPNETCORE_ENVIRONMENT: "Production"
  ApiBaseUrl: "https://api.contoso.com"
```

**Network segmentation**

Network boundaries should reflect environment boundaries. Production often needs stricter inbound and outbound controls than dev or test.

Common patterns:

- Separate VNets per environment
- Separate subnets for app, data, and management tiers
- Use private endpoints for PaaS services in prod
- Restrict peering so dev cannot freely reach prod
- Use Azure Firewall or network virtual appliances for controlled egress

A typical topology is:

- Dev VNet with relaxed outbound access for developer productivity
- Test VNet with controlled access to shared test dependencies
- Prod VNet with private access only and tightly filtered egress

If you must share services, prefer read-only or sanitized dependencies rather than direct prod data access from lower environments.

**Policy and RBAC inheritance**

Azure Policy and RBAC inheritance are powerful tools for enforcing segmentation. Apply broad governance at management group scope, then refine at subscription or resource group scope.

Example policy inheritance model:

- `Platform` management group
  - Enforce allowed regions
  - Require diagnostic settings
  - Deny public IPs for certain workloads
- `Non-Production` management group
  - Allow more permissive SKUs
  - Permit sandbox resource creation
- `Production` management group
  - Deny unsupported regions
  - Require private endpoints
  - Enforce tagging and backup policies

Example RBAC pattern:

- Grant platform team `Owner` or `Contributor` only on platform subscriptions
- Grant app team `Contributor` on dev and test subscriptions
- Grant app team limited `Contributor` or custom roles on prod
- Grant security and operations teams read access across all environments

A simple inheritance example:

```text
Management Group: Production
└── Subscription: App-Prod
    ├── Policy: Deny public storage access
    ├── Policy: Require backup
    └── RBAC: App Team = Reader, Release Pipeline = Contributor on specific RG
```

This means the subscription inherits the production guardrails automatically, while the app team still gets only the access it needs.

**Deployment strategy and promotion**

A strong segmentation model supports progressive delivery. You should promote the same artifact through dev, test, and prod rather than rebuilding differently for each environment.

Recommended flow:

1. Build once.
2. Deploy to dev.
3. Validate in test.
4. Promote the same artifact to prod.
5. Use approvals and checks before production release.

This reduces configuration drift and makes failures easier to diagnose.

Useful release patterns include:

- **Blue-green** deployment for near-zero-downtime cutover
- **Canary** deployment for gradual traffic shifting
- **Deployment slots** for App Service release validation
- Separate pipelines or stages for each environment with approvals

Example Azure DevOps or GitHub Actions concept:

```yaml
stages:
  - dev
  - test
  - prod
```

Each stage should inject environment-specific variables rather than changing the application package.

**Backup, DR, and recovery isolation**

Backup and disaster recovery should also be segmented. Production usually has stricter **RPO** (Recovery Point Objective) and **RTO** (Recovery Time Objective) requirements than dev or test.

Practical guidance:

- Use separate backup policies for prod and non-prod
- Keep prod backups in the correct region or paired region based on recovery design
- Avoid restoring prod backups into dev without sanitization
- Test restore procedures in non-prod regularly
- Protect backup vault access with separate RBAC and monitoring

If dev and test are disposable, you may use shorter retention and simpler recovery. Production should have documented restore steps, immutable or protected backups where appropriate, and clear ownership.

**Cost management and quotas**

Segmentation also helps with cost control. Dev and test should not consume prod budgets or quotas unintentionally.

Use:

- Separate subscriptions for clear chargeback or showback
- Budgets and alerts per subscription
- Different VM sizes or autoscale limits per environment
- Resource locks or policy controls in prod to prevent accidental deletion

Example budget approach:

- Dev: low monthly budget, aggressive auto-shutdown
- Test: moderate budget, scheduled usage windows
- Prod: higher budget, alerting focused on anomalies rather than shutdown

**Practical environment matrix**

| Area | Dev | Test | Prod |
|---|---|---|---|
| Access | Broad for developers | Controlled for QA and release teams | Restricted and audited |
| Secrets | Non-prod only | Test-specific | Prod-only |
| Network | Flexible | Moderate controls | Strict private access |
| Backup | Minimal or disposable | Limited retention | Full policy and restore testing |
| Deployment | Frequent | Release candidate validation | Approved promotion |
| Monitoring | Debug-friendly | Validation-focused | SLO and incident-focused |

**Common design mistakes**

Avoid these patterns:

- Sharing the same subscription for dev and prod
- Reusing prod secrets in lower environments
- Granting broad prod access to CI/CD identities
- Using resource groups as the only isolation boundary
- Allowing dev to connect directly to prod databases
- Skipping backup and restore testing for prod
- Deploying different codebases to each environment instead of one promoted artifact

**When to use shared services**

Some services can be shared across environments, but only with care. Shared platform services such as identity, logging, or connectivity can reduce duplication. However, shared services must not become a hidden dependency that breaks isolation.

Good candidates for shared services:

- Central logging and monitoring
- Shared DNS or hub networking
- Identity platform components
- Image registries or artifact feeds

Use caution with:

- Shared databases
- Shared Key Vaults
- Shared storage accounts for app data
- Shared admin credentials

If a shared service stores environment-specific data, separate it by subscription, resource group, or at least by strict access control and naming conventions.

**Example naming and tagging**

Consistent naming makes segmentation easier to operate and audit.

Example naming pattern:

```text
rg-app-dev-eastus
rg-app-test-eastus
rg-app-prod-eastus
kv-app-dev
kv-app-test
kv-app-prod
```

Useful tags:

- `Environment=Dev|Test|Prod`
- `Owner=TeamName`
- `CostCenter=1234`
- `DataClassification=Public|Internal|Confidential`
- `Criticality=Low|Medium|High`

Tags help with reporting, policy, and automation, but they do not replace real isolation.

**Exam focus**

For AZ-305, focus on the design tradeoffs:

- Use **subscriptions** for strong environment separation.
- Use **management groups** for inherited governance.
- Use **resource groups** for lifecycle grouping, not security isolation.
- Separate **identities**, **secrets**, and **network paths** per environment.
- Use **deployment slots** and progressive delivery for safer production releases.
- Align **backup/DR** with production recovery requirements.
- Apply **policy** and **RBAC** at the right scope to enforce boundaries.

A strong answer usually combines governance, identity, networking, and release strategy rather than relying on a single control.
