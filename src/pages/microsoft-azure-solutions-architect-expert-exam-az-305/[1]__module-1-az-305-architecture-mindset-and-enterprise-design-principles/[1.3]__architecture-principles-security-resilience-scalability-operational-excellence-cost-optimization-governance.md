## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Architecture principles overview**

AZ-305 expects you to design solutions that balance business requirements across six pillars: security, resilience, scalability, operational excellence, cost optimization, and governance. In practice, you rarely optimize one pillar in isolation. A strong design makes tradeoffs explicit, documents assumptions, and chooses Azure services that fit the workload’s risk, performance, and compliance profile.

A useful exam mindset is to ask:
- What must be protected?
- What can fail, and how fast must it recover?
- What load pattern must the system absorb?
- How will you operate, observe, and change it safely?
- What is the minimum cost that still meets the SLO?
- What controls prove compliance and prevent drift?

**Security**

Security design in Azure starts with identity, then extends to network, data, and workload protection. Use a zero trust approach: verify explicitly, use least privilege, and assume breach.

**Identity and access**

Microsoft Entra ID is the identity control plane for Azure. Use it for authentication, authorization, and conditional access decisions.

- **Microsoft Entra Conditional Access** enforces access rules based on user, device, location, risk, and application.
- **Multi-Factor Authentication (MFA)** adds a second verification factor and remains a core baseline control.
- **Passwordless authentication** reduces phishing risk and improves user experience.
- **Privileged Identity Management (PIM)** provides just-in-time elevation for administrative roles.
- **Managed identities** let Azure resources authenticate to other services without storing secrets.

Common exam guidance:
- Use Conditional Access to require MFA for sensitive apps or risky sign-ins.
- Use PIM for privileged roles instead of permanent assignment.
- Use managed identities for app-to-service access when the workload runs in Azure.
- Use service principals only when a managed identity is not possible.

**Network security**

Segment networks to reduce blast radius and control east-west and north-south traffic.

- Use **hub-spoke** for centralized shared services such as firewall, DNS, and Bastion.
- Use **virtual network peering** for low-latency private connectivity between VNets.
- Use **Network Security Groups (NSGs)** for subnet and NIC-level filtering.
- Use **Azure Firewall** for centralized stateful inspection and egress control.
- Use **DDoS Protection Standard** for public IPs that need enhanced mitigation and telemetry.
- Use **Private Link** and private endpoints to expose PaaS services privately.
- Use **service endpoints** when you want simpler regional VNet-to-service access and the service model fits that design; Private Link is generally preferred when you need stronger isolation and private addressability.

> Do not assume every internet-facing workload needs the same controls. A public API behind Front Door and WAF has different exposure than an internal line-of-business app reachable only through private endpoints.

**Web application protection**

Use a **Web Application Firewall (WAF)** to inspect HTTP/S traffic and block common attacks such as SQL injection and cross-site scripting.

- Put WAF in front of Application Gateway or Azure Front Door when you need L7 protection.
- Tune WAF rules to reduce false positives for APIs, file uploads, and legacy apps.
- Use custom rules for IP allowlists, geo restrictions, or header-based controls.
- Monitor WAF logs before enabling aggressive blocking in production.

**Data protection and secrets**

Protect data at rest, in transit, and in use where possible.

- Use TLS for all service-to-service traffic.
- Use customer-managed keys when compliance or key ownership requires it.
- Use **Azure Key Vault** for secrets, keys, and certificates.
- Prefer **Azure RBAC** for Key Vault authorization in new designs when it aligns with your governance model.
- Use access policies only when you need the legacy model or a specific compatibility requirement.
- Enable **soft delete** and **purge protection** to prevent accidental or malicious key vault deletion.
- Consider **Managed HSM** when you need hardware-backed key control with stricter separation and compliance requirements.

Practical Key Vault example:
```bash
az keyvault create \
  --name kv-prod-001 \
  --resource-group rg-platform \
  --location eastus \
  --enable-rbac-authorization true \
  --enable-purge-protection true
```

**Security monitoring**

Security controls are incomplete without detection and response.

- Send logs to Log Analytics, Microsoft Sentinel, or a SIEM/SOAR platform.
- Use Microsoft Defender for Cloud for posture management and workload threat protection.
- Review identity risk, sign-in logs, and privileged activity regularly.
- Alert on key events such as public IP creation, NSG rule changes, and Key Vault access anomalies.

**Resilience**

Resilience is the ability to continue operating during failures and to recover quickly when failures occur. Design for component failure, zone failure, region failure, and dependency failure.

**Availability patterns**

Choose the right redundancy model for the service and the business objective.

- **Availability sets** protect against hardware and rack-level failures within a datacenter.
- **Availability zones** protect against datacenter-level failures within a region.
- **Zone-redundant services** spread replicas across zones when the service supports it.
- **Region pairs** are Microsoft’s guidance construct for platform resiliency, planned updates, and data residency considerations; use them as a design input, but base disaster recovery on workload requirements and service capabilities.

Typical guidance:
- Use availability sets for legacy or non-zone-aware IaaS workloads that still need intra-region redundancy.
- Use availability zones for higher availability when the service supports zone deployment.
- Use paired regions for cross-region recovery planning when you need geographic separation and Microsoft-supported regional pairing benefits.
- Use active-active across regions only when the application and data layer can support it.

**State, consistency, and quorum**

Resilient design depends on how the application handles state.

- Stateless tiers are easier to scale and recover.
- Stateful systems need replication, failover, and consistency planning.
- Distributed systems may require quorum to avoid split-brain conditions.
- Strong consistency can reduce availability during partitions.
- Eventual consistency can improve availability but requires application tolerance for replication lag.

When designing databases or clustered services, ask:
- What happens if one replica is unavailable?
- Can the system continue with a minority of nodes?
- Is read-after-write consistency required?
- How is failover detected and promoted?

**Backup and restore**

Backup is not the same as high availability. HA reduces downtime; backup protects against data loss, corruption, and ransomware.

- Use **Azure Backup** for supported IaaS workloads such as Azure VMs and Azure Files, and for specific supported PaaS scenarios where the service offers backup integration.
- Use native backup and restore features for services like Azure SQL Database, Azure Database for PostgreSQL, and Cosmos DB where applicable.
- Use point-in-time restore when you need recovery from logical corruption.
- Test restore procedures regularly; a backup that has never been restored is only an assumption.

Service-specific examples:
- **Azure VMs**: use Recovery Services vault backups and consider zone-aware VM placement.
- **Azure Files**: use share snapshots and Azure Backup for file share protection.
- **Azure SQL**: use automated backups, long-term retention, and geo-replication where needed.
- **Cosmos DB**: choose backup mode based on recovery objectives; understand that restore behavior differs from traditional database backup.
- **AKS**: protect cluster state, persistent volumes, and application data separately; do not assume the cluster control plane backup covers everything.

**Disaster recovery**

Disaster recovery design should align with RTO and RPO.

- **RTO (Recovery Time Objective)** is how long recovery can take.
- **RPO (Recovery Point Objective)** is how much data loss is acceptable.

Common patterns:
- Active-passive for simpler failover and lower cost.
- Active-active for lower RTO and higher complexity.
- Pilot light for minimal standby cost with slower recovery.
- Warm standby for faster recovery with moderate cost.
- Backup/restore for the lowest cost and longest recovery time.

Practical DR decision:
- Use active-active for customer-facing systems with strict uptime requirements and globally distributed users.
- Use pilot light for internal systems that can tolerate a longer recovery window.
- Use backup/restore for low-criticality workloads or when the application is not easily replicated.

**Scalability**

Scalability is the ability to handle growth in users, transactions, data, and geographic reach without redesigning the system.

**Scale up vs scale out**

- **Scale up** increases the size of a single instance.
- **Scale out** adds more instances.

Use scale up when:
- The workload is constrained by single-node state or licensing.
- The service has limited horizontal scaling support.
- You need a quick capacity increase with minimal application change.

Use scale out when:
- The workload is stateless or partitionable.
- You need elasticity for variable demand.
- You want better fault tolerance through multiple instances.

**Azure-specific scaling patterns**

- Use **Azure App Service autoscale** on the App Service plan, not the web app itself.
- Use **Virtual Machine Scale Sets (VMSS)** for uniform VM fleets.
- Use **AKS cluster autoscaler** and pod autoscaling for container workloads.
- Use **Azure SQL elastic pools** when many databases have variable usage patterns.
- Use **Cosmos DB** throughput scaling and partitioning for globally distributed, high-throughput workloads.
- Use **Azure Cache for Redis** to offload hot reads and reduce database pressure.
- Use **Azure Front Door** or **Azure CDN** for edge caching and global content acceleration.

Example App Service plan autoscale concept:
```bash
az monitor autoscale create \
  --resource-group rg-web \
  --resource /subscriptions/<subId>/resourceGroups/rg-web/providers/Microsoft.Web/serverfarms/asp-prod \
  --name asp-prod-autoscale \
  --min-count 2 \
  --max-count 10 \
  --count 2
```

Example scale rule concept:
```bash
az monitor autoscale rule create \
  --resource-group rg-web \
  --autoscale-name asp-prod-autoscale \
  --condition "Percentage CPU > 70 avg 5m" \
  --scale out 1
```

> Autoscale rules vary by resource type and CLI version. In exam scenarios, focus on the principle: define rules on the scalable resource, use metrics that reflect real saturation, and test thresholds under load.

**Caching and content delivery**

Caching reduces latency and backend load.

- Use Redis for session state, reference data, and expensive query results.
- Use Front Door or CDN for static assets and globally distributed content.
- Use application-level caching when the data is highly specific to the workload.
- Use HTTP cache headers to control browser and edge caching behavior.

Choose caching when:
- Reads greatly outnumber writes.
- Data can tolerate short staleness.
- Backend latency or cost is a bottleneck.

**Asynchronous messaging**

Use asynchronous patterns to decouple producers and consumers.

- Use **Azure Service Bus** for reliable enterprise messaging, commands, and workflows.
- Use **Azure Queue Storage** for simple, low-cost queueing.
- Use **Event Grid** for event routing and reactive architectures.
- Use **Event Hubs** for telemetry and high-throughput streaming.

Why it matters:
- It smooths spikes.
- It improves resilience by buffering work.
- It allows independent scaling of components.

**Operational excellence**

Operational excellence means you can deploy, observe, troubleshoot, and change the system safely and repeatedly.

**Infrastructure as Code**

Use code to define infrastructure so environments are repeatable and reviewable.

- Use **Bicep**, **ARM templates**, or **Terraform** for declarative provisioning.
- Store templates in source control.
- Use pull requests and approvals for changes.
- Parameterize environment-specific values.
- Validate templates before deployment.

Example Bicep snippet:
```bicep
param location string = resourceGroup().location
param appName string

resource plan 'Microsoft.Web/serverfarms@2023-12-01' = {
  name: '${appName}-plan'
  location: location
  sku: {
    name: 'P1v3'
    tier: 'PremiumV3'
  }
}
```

**Configuration management and release governance**

Infrastructure is only part of the system; configuration drift can break production.

- Keep app settings, feature flags, and secrets separate from code.
- Use Azure App Configuration for centralized configuration and feature management.
- Use Key Vault references for secret injection where supported.
- Use deployment slots for safe web app releases.
- Use blue-green or canary deployment patterns to reduce release risk.
- Use change approvals for production infrastructure and policy changes.

**Observability**

Observability combines logs, metrics, and traces to explain system behavior.

- Use **Azure Monitor** as the central monitoring platform.
- Use **Log Analytics** for queryable logs.
- Use **Application Insights** for application telemetry and distributed tracing.
- Use **workbooks** and dashboards for operational views.
- Use alerts based on symptoms and service-level indicators, not only on infrastructure metrics.

SRE-style concepts:
- **SLI (Service Level Indicator)** is a measured signal such as latency or error rate.
- **SLO (Service Level Objective)** is the target for that signal.
- **Error budget** is the allowed unreliability before you slow feature releases.

Practical monitoring design:
- Collect platform logs from subscriptions, VNets, firewalls, and PaaS services.
- Centralize logs in a shared monitoring subscription or workspace strategy.
- Retain logs according to compliance and incident response needs.
- Correlate application telemetry with infrastructure events.

**Cost optimization**

Cost optimization means meeting requirements with the least waste, not simply choosing the cheapest service.

**Cost controls**

- Use **Azure Cost Management** for cost analysis, budgets, and alerts.
- Set budgets at management group, subscription, or resource group scope.
- Use alerts for forecasted overspend, not only actual overspend.
- Tag resources for chargeback and showback.
- Review idle resources, orphaned disks, unattached public IPs, and stale snapshots.

Example budget creation:
```bash
az consumption budget create \
  --budget-name prod-budget \
  --amount 5000 \
  --time-grain Monthly \
  --category Cost \
  --start-date 2026-04-01 \
  --end-date 2027-03-31
```

**Lifecycle automation**

- Schedule shutdown for non-production VMs.
- Use autoscale to match capacity to demand.
- Delete temporary environments after testing.
- Use policy or automation to remove unused resources.
- Use reserved capacity or savings plans when usage is predictable.

**Storage and data transfer costs**

Storage cost is not just about capacity.

- Use hot, cool, and archive tiers based on access frequency.
- Use lifecycle management to move blobs between tiers.
- Consider transaction and retrieval costs, not only storage price.
- Minimize unnecessary cross-region data transfer.
- Place compute near data when possible to reduce egress charges.

Workload tradeoffs:
- Use premium storage when latency and IOPS matter more than cost.
- Use standard storage when throughput needs are moderate.
- Use caching to reduce expensive database reads.
- Use CDN to reduce origin bandwidth and improve global performance.

**Service selection and right-sizing**

Choose the smallest service tier that meets requirements with headroom.

- Prefer PaaS when it reduces operational overhead.
- Prefer serverless when traffic is spiky and event-driven.
- Prefer IaaS when you need OS-level control or unsupported software.
- Right-size VM SKUs, database tiers, and storage performance tiers regularly.

**Governance**

Governance ensures that architecture decisions are enforceable, auditable, and aligned with enterprise standards.

**Azure landing zones**

An Azure landing zone provides a scalable foundation for subscriptions, identity, networking, policy, and management.

- Use management groups to organize subscriptions by platform, landing zone, and environment.
- Separate platform services from application subscriptions.
- Use subscription vending to create new subscriptions through a controlled process.
- Standardize naming, tagging, and region selection.
- Apply baseline policies before workloads are deployed.

Example management group pattern:
- Root management group
  - Platform
    - Identity
    - Connectivity
    - Management
  - LandingZones
    - Corp
    - Online
  - Sandbox

**Azure RBAC vs Azure Policy**

These are different controls and exam questions often test the distinction.

- **Azure RBAC** controls who can do what on a resource.
- **Azure Policy** controls what can be deployed and how resources must be configured.

Use RBAC for:
- Access assignment
- Role-based administration
- Least privilege

Use Policy for:
- Denying public IPs in restricted subscriptions
- Enforcing tags
- Requiring diagnostic settings
- Restricting allowed SKUs or regions
- Auditing compliance

Example policy assignment concept:
```bash
az policy assignment create \
  --name deny-public-ip \
  --scope /subscriptions/<subId> \
  --policy /providers/Microsoft.Authorization/policyDefinitions/<policyDefinitionId>
```

**Naming, tagging, and compliance**

- Use consistent naming conventions for automation and inventory.
- Tag resources with owner, environment, cost center, application, and data classification.
- Use policy to require or append tags.
- Use compliance reports to track drift and remediation status.
- Use resource locks sparingly for critical resources that should not be deleted or modified accidentally.

**Enterprise controls**

- Use management groups for policy inheritance.
- Use Defender for Cloud regulatory compliance dashboards for posture tracking.
- Use Azure Monitor and activity logs for audit trails.
- Use Microsoft Purview when you need broader data governance and cataloging.

**Scenario-based decision examples**

**Example 1: Secure external web app**
- Requirement: Public web app, sensitive data, global users, low latency.
- Best fit:
  - Azure Front Door with WAF for global entry and edge protection
  - App Service or AKS behind private endpoints where possible
  - Key Vault for secrets
  - Conditional Access for admin access
- Why: You need L7 protection, global routing, and strong identity controls.

**Example 2: Internal line-of-business app**
- Requirement: Private access only, moderate traffic, simple operations.
- Best fit:
  - Hub-spoke network
  - Private endpoints to PaaS
  - Azure Firewall for egress control
  - Azure Monitor centralized logging
- Why: Private connectivity and centralized governance matter more than global edge delivery.

**Example 3: Spiky transactional workload**
- Requirement: Unpredictable demand, many small requests, cost sensitivity.
- Best fit:
  - Queue-based decoupling with Service Bus
  - Autoscale on compute
  - Redis caching for hot data
  - Budgets and alerts for cost control
- Why: Asynchronous buffering and caching reduce peak pressure and avoid overprovisioning.

**Example 4: Regulated enterprise platform**
- Requirement: Strong compliance, auditability, and separation of duties.
- Best fit:
  - Management group hierarchy
  - Azure Policy for guardrails
  - RBAC with PIM
  - Key Vault with RBAC, soft delete, and purge protection
  - Central logging and retention
- Why: Governance and evidence are as important as technical functionality.

**Practical design checklist**

1. Define business requirements, RTO, RPO, and compliance constraints.
2. Classify data and identify trust boundaries.
3. Choose identity controls first, then network and data controls.
4. Select redundancy based on service capability and failure domain.
5. Decide whether the workload should scale up, scale out, or both.
6. Add observability before production launch.
7. Put cost budgets, tags, and lifecycle automation in place.
8. Enforce standards with management groups, RBAC, and Azure Policy.
9. Test failover, restore, and rollback procedures.
10. Review the design against real operational ownership.

**Common exam traps**

- Confusing **RBAC** with **Policy**.
- Assuming backup equals high availability.
- Using private endpoints without planning DNS.
- Choosing active-active without confirming data consistency and application support.
- Treating service endpoints and private endpoints as interchangeable.
- Designing autoscale on the wrong resource, such as the app instead of the plan.
- Assuming all PaaS services have the same backup and restore capabilities.
- Ignoring logging, alerting, and operational ownership until after deployment.

**Key takeaways**

A strong AZ-305 design is secure by default, resilient to failure, scalable under load, observable in production, cost-aware over time, and governed through repeatable controls. The best answer is usually the one that matches the workload’s requirements with the least operational risk and the clearest control plane.
