## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Enterprise design patterns in Azure**

Enterprise Azure design is about creating a platform that is governable, secure, resilient, observable, and scalable without becoming operationally fragile. For AZ-305, you are expected to choose patterns that fit business, compliance, and workload requirements rather than defaulting to a single “best” service.

A strong design usually separates concerns across:
- **Management and governance** for policy, access, and subscription structure
- **Network architecture** for segmentation, ingress, and private connectivity
- **Identity and security** for least privilege and conditional access
- **Application and integration** for decoupled, event-driven systems
- **Data and analytics** for operational and analytical workloads
- **Resilience and operations** for backup, failover, and monitoring

**Azure governance and management group design**

Management groups provide the top-down governance layer above subscriptions. In enterprise landing zones, you typically use them to apply policy and access inheritance consistently across many subscriptions.

A common pattern is:
- **Tenant root group** as the top-most governance boundary
- **Platform management groups** for shared services such as identity, connectivity, and management
- **Landing zone management groups** for application subscriptions
- **Sandbox or innovation management groups** with looser controls
- **Decommissioned or quarantine groups** for controlled retirement

The tenant root group should be treated as a guardrail layer, not a place for broad day-to-day administration. Use it for the most universal controls, such as:
- Denying creation of unsupported regions
- Enforcing tagging standards
- Restricting public IP creation where appropriate
- Preventing resource types that violate platform standards

Subscription placement strategy matters because subscriptions are both a billing boundary and a governance boundary. Place subscriptions according to operational ownership and blast radius:
- **Platform subscriptions** for shared network, identity, and monitoring services
- **Application subscriptions** for business workloads with aligned ownership
- **Environment separation** for dev, test, and prod when isolation is required
- **Regulated workloads** in dedicated subscriptions when compliance or access boundaries demand it

Inheritance and denies are typically structured so that:
- Broad policies apply high in the hierarchy
- Exceptions are rare and explicit
- Deny assignments or policy effects protect shared services from accidental drift
- Workload subscriptions inherit baseline controls, then add workload-specific policies

In enterprise landing zones, you often combine:
- **Azure Policy** for compliance and configuration enforcement
- **RBAC** for access control
- **Management groups** for inheritance
- **Blueprint-like standardization** through templates, Bicep, or Terraform

Example policy assignment at a management group scope:
```bash
az policy assignment create \
  --name "deny-public-ip" \
  --display-name "Deny Public IPs" \
  --scope "/providers/Microsoft.Management/managementGroups/contoso-platform" \
  --policy "/providers/Microsoft.Authorization/policyDefinitions/deny-public-ip"
```

Example RBAC assignment at subscription scope:
```bash
az role assignment create \
  --assignee user@contoso.com \
  --role "Reader" \
  --scope "/subscriptions/00000000-0000-0000-0000-000000000000"
```

**Resource group boundaries and naming conventions**

Resource groups are lifecycle and deployment boundaries, not security boundaries. Use them to group resources that share the same:
- Deployment cadence
- Ownership team
- Lifecycle and deletion pattern
- Monitoring and cost center
- Region or availability requirement

A practical enterprise pattern is to align one resource group to one workload component or tier when that component has a distinct lifecycle. For example:
- `rg-app-prod-weu`
- `rg-app-prod-weu-data`
- `rg-app-prod-weu-net`

This makes it easier to:
- Apply tags consistently
- Delete or redeploy a component safely
- Delegate operational ownership
- Track cost and compliance

Naming conventions should be predictable and machine-friendly. A common structure is:
`<org>-<app>-<env>-<region>-<resource>`

Example:
- `contoso-payments-prod-weu-app`
- `contoso-payments-prod-weu-kv`
- `contoso-shared-nonprod-eus-net`

Good naming conventions help with:
- Automation
- Inventory
- Cost management
- Incident response
- Policy targeting

Tags complement naming by carrying metadata that should not be encoded in the resource name:
- `CostCenter`
- `Owner`
- `Application`
- `Environment`
- `DataClassification`
- `Criticality`

> Resource groups are not a substitute for governance. Use them for operational grouping, while management groups, policy, and RBAC enforce enterprise standards.

**Identity and access design**

Identity is the control plane for enterprise Azure. Microsoft Entra ID is the identity provider, and Microsoft Entra Conditional Access is the policy engine that enforces access conditions based on user, device, location, risk, and application.

Key enterprise identity patterns include:
- **Least privilege** through role-based access control
- **Privileged Identity Management (PIM)** for just-in-time elevation
- **Conditional Access** for MFA, compliant devices, and sign-in risk controls
- **Separate admin accounts** for privileged operations
- **Managed identities** for workload-to-workload authentication

Use managed identities whenever an Azure service needs to access another Azure resource. This removes secrets from code and reduces operational overhead.

Example: grant a managed identity access to Key Vault secrets:
```bash
az role assignment create \
  --assignee-object-id <managed-identity-object-id> \
  --role "Key Vault Secrets User" \
  --scope "/subscriptions/<sub-id>/resourceGroups/rg-app-prod-weu-kv/providers/Microsoft.KeyVault/vaults/kv-contoso-prod"
```

For enterprise admin access:
- Use PIM for eligible roles
- Require MFA for privileged roles
- Restrict admin access to compliant devices
- Use break-glass accounts with strong monitoring and limited use

**Security architecture and Zero Trust**

Azure enterprise security design usually follows the Zero Trust model:
- **Verify explicitly**
- **Use least privilege**
- **Assume breach**

Network segmentation is central to Zero Trust. Segment by:
- Environment: dev, test, prod
- Tier: web, app, data
- Sensitivity: regulated, internal, public
- Trust boundary: shared services vs workload-specific

A common segmentation pattern is:
- Internet-facing edge in a dedicated ingress layer
- Web tier in a restricted subnet
- App tier with no direct internet exposure
- Data tier isolated behind private endpoints
- Shared services in a platform subscription

Use network security groups (NSGs), Azure Firewall, route tables, and private endpoints to reduce lateral movement. Prefer private access to PaaS services when possible.

Key management patterns:
- **Azure Key Vault** for secrets, keys, and certificates
- **Managed HSM** for high-assurance key protection and dedicated hardware-backed key control
- **Customer-managed keys (CMK)** when compliance requires customer control of encryption keys

Use Key Vault when you need:
- Centralized secret storage
- Certificate lifecycle management
- Application secret retrieval

Use Managed HSM when you need:
- Stronger isolation for cryptographic keys
- Dedicated HSM-backed key protection
- More stringent compliance requirements

Example Key Vault secret retrieval from CLI:
```bash
az keyvault secret show \
  --vault-name kv-contoso-prod \
  --name SqlAdminPassword
```

Example private endpoint DNS pattern:
```bash
az network private-dns zone create \
  --resource-group rg-app-prod-weu-net \
  --name privatelink.vaultcore.azure.net

az network private-dns link vnet create \
  --resource-group rg-app-prod-weu-net \
  --zone-name privatelink.vaultcore.azure.net \
  --name link-weu-hub \
  --virtual-network vnet-hub-weu \
  --registration-enabled false
```

**Networking and ingress patterns**

Enterprise networking in Azure usually combines hub-and-spoke or virtual WAN with centralized inspection and controlled ingress. The design choice depends on scale, geography, and operational model.

Common ingress services:
- **Azure Front Door** for global HTTP(S) entry, edge acceleration, and web application delivery
- **Application Gateway with WAF** for regional Layer 7 ingress, TLS termination, and path-based routing
- **Traffic Manager** for DNS-based global traffic distribution across endpoints

Decision criteria:
- Use **Front Door** when you need global anycast entry, fast failover, and edge-based web delivery.
- Use **Application Gateway** when you need regional ingress, internal or external Layer 7 routing, and WAF close to the workload.
- Use **Traffic Manager** when you need DNS-level routing for non-HTTP or simple endpoint distribution, and you can tolerate DNS-based failover behavior.

A common enterprise pattern is:
- Front Door at the global edge
- Application Gateway in each region for regional web ingress
- Private backends behind Application Gateway
- Azure Firewall for centralized egress and inspection

Example topology:
- Internet users -> Front Door -> regional Application Gateway/WAF -> app services or AKS
- Internal users -> VPN/ExpressRoute -> hub -> Application Gateway or internal load balancer -> workloads

For private connectivity:
- Use **ExpressRoute** for private, predictable connectivity to Azure
- Use **VPN Gateway** for lower-cost or backup connectivity
- Use **Private Link** for PaaS access without public exposure

Example private DNS zone for SQL Private Link:
```bash
az network private-dns zone create \
  --resource-group rg-platform-dns \
  --name privatelink.database.windows.net
```

**Application architecture and integration**

Enterprise applications often work best when they are loosely coupled. Event-driven architecture reduces direct dependencies and improves resilience.

Core patterns:
- **Pub/sub** for one-to-many event distribution
- **Queue-based load leveling** for smoothing bursts
- **Choreography** for decentralized event reactions
- **Orchestration** for centrally coordinated workflows

Use choreography when:
- Services can react independently
- Business flow is simple or naturally distributed
- You want to reduce central workflow coupling

Use orchestration when:
- Steps must happen in a controlled sequence
- Compensation logic is important
- Human approval or branching is required

Azure services commonly used:
- **Azure Service Bus** for enterprise messaging, queues, and topics
- **Azure Event Grid** for event routing and reactive integrations
- **Azure Functions** for lightweight event handlers and serverless processing
- **Logic Apps** for workflow orchestration, connectors, and integration with SaaS systems

Practical guidance:
- Use Service Bus when you need ordered processing, transactions, duplicate detection, or durable enterprise messaging.
- Use Event Grid when you need event notification and fan-out with low operational overhead.
- Use Functions for code-first event processing.
- Use Logic Apps for connector-heavy workflows, approvals, and business process integration.

Example Service Bus topic subscription pattern:
```bash
az servicebus topic create \
  --resource-group rg-integration-prod-weu \
  --namespace-name sb-contoso-prod \
  --name orders

az servicebus topic subscription create \
  --resource-group rg-integration-prod-weu \
  --namespace-name sb-contoso-prod \
  --topic-name orders \
  --name billing
```

**Data architecture and analytics patterns**

Enterprise data design usually separates operational data from analytical data. Do not force reporting workloads onto transactional systems when a lake or warehouse pattern is more appropriate.

Common patterns:
- **Data lake** for raw, curated, and enriched data at scale
- **Lakehouse or unified analytics** for combining data engineering and analytics
- **Data warehouse** for structured, governed analytical queries
- **Operational data store** for near-real-time reporting from transactional sources

A typical enterprise analytics flow is:
1. Ingest data from operational systems
2. Land raw data in a data lake
3. Transform and curate data
4. Serve analytics through a warehouse or semantic layer
5. Expose dashboards and governed access

Azure services often used:
- **Azure Data Lake Storage Gen2** for scalable storage
- **Azure Synapse Analytics** for SQL and Spark-based analytics
- **Microsoft Fabric** for integrated analytics experiences where appropriate
- **Azure Data Factory** for ETL (extract, transform, load) and ELT (extract, load, transform)
- **Event Hubs** for streaming ingestion

Use ETL when:
- You need heavy transformation before loading
- Source systems are inconsistent
- Governance requires curated data before consumption

Use ELT when:
- The target platform can handle transformation efficiently
- You want to land raw data quickly
- You need flexible downstream modeling

Example Data Factory pipeline concept:
- Copy from on-premises SQL Server to ADLS Gen2
- Transform with Mapping Data Flows or Spark
- Load curated data into Synapse or Fabric warehouse

**Resilience, backup, and disaster recovery**

Resilience design should match the service’s native capabilities. Not every Azure service supports the same failover model, so you must align the architecture with what the service can actually do.

Common patterns:
- **Active-active** for low latency and high availability across multiple regions
- **Active-passive** for simpler failover and lower cost
- **Pilot light** for minimal standby capacity
- **Warm standby** for faster recovery with partial pre-provisioning

Choose active-active when:
- The workload can handle multi-region writes or stateless scaling
- You need very high availability
- You can manage data consistency and routing complexity

Choose active-passive when:
- The workload is stateful or harder to split across regions
- You want simpler operational behavior
- Cost is a concern and some failover delay is acceptable

Align failover design with service capabilities:
- Some PaaS services support built-in geo-replication or zone redundancy
- Some require application-level replication and routing
- Some need paired-region recovery planning and manual failover steps

Backup and recovery patterns:
- Use **Recovery Services Vault** for Azure VM backup and many traditional backup scenarios
- Use **Backup vault** for newer backup scenarios and certain data protection workloads
- Define retention by business and regulatory requirements
- Test restore procedures regularly, not just backup success

Example backup concept:
- Production VMs back up daily to a Recovery Services Vault
- Critical databases use native backup plus long-term retention
- Application configuration is stored in source control and redeployed via IaC

Example failover routing with Traffic Manager:
```bash
az network traffic-manager profile create \
  --resource-group rg-network-prod \
  --name tm-contoso-orders \
  --routing-method Priority \
  --unique-dns-name contoso-orders

az network traffic-manager endpoint create \
  --resource-group rg-network-prod \
  --profile-name tm-contoso-orders \
  --name eastus-endpoint \
  --type azureEndpoints \
  --target-resource-id /subscriptions/<sub-id>/resourceGroups/rg-app-eastus/providers/Microsoft.Web/sites/app-eastus \
  --priority 1
```

**Observability and operational design**

Observability should be centralized so operations teams can detect issues, trace dependencies, and retain evidence for compliance. Azure Monitor is the core platform for metrics, logs, alerts, and visualization.

Key components:
- **Azure Monitor metrics** for near-real-time platform signals
- **Log Analytics workspaces** for queryable logs
- **Application Insights** for application performance monitoring and distributed tracing
- **Azure Monitor Workbooks** for interactive dashboards and operational views
- **Alerts and action groups** for notification and automation

Distributed tracing is essential for microservices and event-driven systems. Application Insights can correlate requests, dependencies, exceptions, and custom events across services.

Centralized log retention strategy should define:
- Hot retention for active troubleshooting
- Archive retention for compliance and forensic needs
- Workspace separation by environment or regulatory boundary when required
- Export to storage or SIEM if long-term retention or security analytics is needed

Example diagnostic settings to send logs to Log Analytics:
```bash
az monitor diagnostic-settings create \
  --name send-to-law \
  --resource /subscriptions/<sub-id>/resourceGroups/rg-app-prod-weu/providers/Microsoft.Web/sites/app-prod \
  --workspace /subscriptions/<sub-id>/resourceGroups/rg-monitoring/providers/Microsoft.OperationalInsights/workspaces/law-contoso-prod \
  --logs '[{"category":"AppServiceHTTPLogs","enabled":true},{"category":"AppServiceConsoleLogs","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

Workbook use cases:
- Service health overview
- Dependency latency trends
- Failed request drill-down
- Capacity and saturation tracking
- Security and audit reporting

**Enterprise policy, RBAC, and private access examples**

Policy and RBAC should work together. Policy enforces what can exist; RBAC controls who can manage it.

Example policy assignment to restrict locations:
```bash
az policy assignment create \
  --name "allowed-locations" \
  --display-name "Allowed Locations" \
  --scope "/providers/Microsoft.Management/managementGroups/contoso-root" \
  --policy "/providers/Microsoft.Authorization/policyDefinitions/allowed-locations" \
  --params '{"listOfAllowedLocations":{"value":["eastus","westus2","westeurope"]}}'
```

Example custom role assignment concept:
```json
{
  "Name": "App Operator",
  "IsCustom": true,
  "Description": "Can manage application resources but not networking or identity",
  "Actions": [
    "Microsoft.Web/sites/*",
    "Microsoft.Insights/alertRules/*"
  ],
  "NotActions": [
    "Microsoft.Authorization/*",
    "Microsoft.Network/*",
    "Microsoft.ManagedIdentity/*"
  ],
  "AssignableScopes": [
    "/subscriptions/<sub-id>"
  ]
}
```

Example private DNS and private endpoint flow:
1. Create the private endpoint for the PaaS resource.
2. Link the appropriate private DNS zone to the VNet.
3. Ensure clients resolve the service FQDN to the private IP.
4. Disable or restrict public network access where supported.

**How to choose the right pattern**

Use the workload requirements to drive the design:
- Choose **management group hierarchy** for enterprise-scale governance and inheritance.
- Choose **resource groups** for lifecycle and deployment boundaries.
- Choose **Front Door** for global web entry and **Application Gateway** for regional WAF-protected ingress.
- Choose **Service Bus** for durable enterprise messaging and **Event Grid** for event fan-out.
- Choose **Logic Apps** for workflow integration and **Functions** for code-driven event handling.
- Choose **Key Vault** or **Managed HSM** based on key protection requirements.
- Choose **active-active** only when the application and data layer can support it.
- Choose **Azure Monitor + Application Insights** for centralized observability and tracing.

A good AZ-305 answer explains not just the service, but the tradeoff:
- Cost versus resilience
- Simplicity versus control
- Global versus regional scope
- Public versus private access
- Centralized versus distributed operations

**Exam-oriented design checklist**

Before you finalize a design, verify:
- Management groups reflect tenant, platform, landing zone, and sandbox boundaries
- Tenant root group contains only universal guardrails
- Subscriptions are placed by ownership, environment, and blast radius
- Resource groups align to lifecycle and deployment units
- Naming and tagging support automation and governance
- Identity uses Conditional Access, PIM, and managed identities
- Network segmentation supports Zero Trust and private access
- Ingress uses the right combination of Front Door, Application Gateway, and Traffic Manager
- Key management uses Key Vault or Managed HSM appropriately
- Integration uses the right mix of queues, topics, events, Functions, and Logic Apps
- Data architecture separates operational and analytical workloads
- Resilience includes tested backup, restore, and failover
- Observability includes logs, metrics, tracing, workbooks, and retention
