## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Overview**

A **reference architecture** is a conceptual or logical blueprint that shows how components should fit together to satisfy common requirements such as scalability, resiliency, security, and governance. It is intentionally technology-aware but still abstract enough to be reused across many organizations.

An **implementation architecture** is the concrete, environment-specific design you actually deploy. It includes exact Azure services, SKUs, regions, network ranges, identity settings, policies, naming standards, and operational choices.

For AZ-305, you need to understand both because the exam often tests whether you can translate business and technical requirements into a practical Azure design. The reference architecture tells you *what good looks like*; the implementation architecture tells you *how you will build it here*.

**Core distinction**

- **Reference architecture** focuses on patterns, principles, and reusable guidance.
- **Implementation architecture** focuses on specific decisions, configurations, and deployment details.
- **Reference architecture** is often vendor-neutral or broadly cloud-oriented, even when Azure examples are used.
- **Implementation architecture** is tied to a particular subscription, tenant, region, compliance boundary, and operational model.
- **Reference architecture** helps you compare options and communicate intent.
- **Implementation architecture** helps engineers deploy, validate, and operate the solution.

**Why the distinction matters**

In real projects, teams often start with a reference architecture and then refine it into an implementation architecture. That transition is where most design risk appears:

- A reference design may assume multi-region failover, but the implementation may only support one region because of budget or data residency.
- A reference design may show a hub-and-spoke network, but the implementation must account for existing IP ranges, firewall rules, and on-premises routing.
- A reference design may recommend managed identity, but the implementation must define which user-assigned identities exist and where they are assigned.

This is why architects must validate assumptions early. A design that looks correct on paper can fail if it does not match operational constraints.

**Reference architecture characteristics**

Reference architectures usually include:

- **Common patterns** such as hub-and-spoke networking, landing zones, microservices, or event-driven integration.
- **Logical components** such as web tier, application tier, data tier, identity, and monitoring.
- **Design principles** such as least privilege, high availability, and separation of duties.
- **Decision guidance** explaining when to choose one pattern over another.
- **Trade-offs** such as cost versus resilience or simplicity versus flexibility.

They usually avoid:

- Exact resource names
- Final IP address ranges
- Subscription IDs
- Region-specific deployment details
- Organization-specific policy assignments

A reference architecture is useful when you need to answer questions like:

- What is the recommended pattern for secure internet-facing workloads?
- How should I structure a platform landing zone?
- What components are typically needed for a resilient data platform?

**Implementation architecture characteristics**

Implementation architecture includes the details required to build and operate the solution:

- **Azure services** selected for the workload, such as Azure App Service, Azure Kubernetes Service (AKS), Azure SQL Database, or Azure Virtual Machines.
- **Sizing and SKUs** such as Standard vs. Premium tiers.
- **Network design** including VNets, subnets, route tables, private endpoints, and DNS.
- **Identity design** including Microsoft Entra ID groups, managed identities, and role assignments.
- **Security controls** such as Azure Policy, network security groups (NSGs), Key Vault, and Defender for Cloud settings.
- **Operational controls** such as monitoring alerts, backup schedules, and disaster recovery runbooks.

Implementation architecture answers questions like:

- Which Azure region do you deploy to?
- What subnet CIDR blocks do you use?
- Which service tier meets the performance target?
- How do you connect private services without exposing public endpoints?

**How they relate**

A practical design process often follows this flow:

1. Start with a reference architecture to establish the pattern.
2. Validate requirements, constraints, and nonfunctional needs.
3. Convert the pattern into an implementation architecture.
4. Review the implementation for cost, security, operability, and compliance.
5. Iterate as constraints or requirements change.

The reference architecture is not a final deployment plan. It is a starting point for design reasoning. The implementation architecture is the final, actionable design that can be deployed through Infrastructure as Code (IaC), manual configuration, or a mix of both.

**Common Azure examples**

**1. Hub-and-spoke networking**

A reference architecture may show:

- A central hub for shared services
- Spoke VNets for workloads
- Centralized firewalling and DNS
- On-premises connectivity through VPN or ExpressRoute

An implementation architecture must define:

- Hub VNet address space
- Spoke VNet address spaces
- Peering settings
- Route propagation behavior
- Firewall SKU and policy
- DNS forwarders or Azure DNS Private Resolver
- Connectivity to on-premises networks

Example implementation snippet:

```bicep
resource hubVnet 'Microsoft.Network/virtualNetworks@2023-11-01' = {
  name: 'vnet-hub-eastus'
  location: resourceGroup().location
  properties: {
    addressSpace: {
      addressPrefixes: [
        '10.0.0.0/16'
      ]
    }
    subnets: [
      {
        name: 'AzureFirewallSubnet'
        properties: {
          addressPrefix: '10.0.1.0/24'
        }
      }
      {
        name: 'GatewaySubnet'
        properties: {
          addressPrefix: '10.0.2.0/24'
        }
      }
    ]
  }
}
```

**2. Web application architecture**

A reference architecture may show:

- Front door or application gateway
- Web tier
- API tier
- Data tier
- Centralized logging and monitoring

An implementation architecture must define:

- Whether you use Azure Front Door, Application Gateway, or both
- Whether the app runs on App Service, AKS, or VMs
- TLS certificate handling
- Private access to backend services
- Autoscale rules
- Backup and restore strategy

Example CLI validation:

```bash
az webapp show \
  --name contoso-web-prod \
  --resource-group rg-app-prod \
  --query "{state:state, defaultHostName:defaultHostName, httpsOnly:httpsOnly}"
```

**3. Data platform architecture**

A reference architecture may show:

- Ingestion
- Storage
- Processing
- Serving
- Governance

An implementation architecture must define:

- Storage account redundancy
- Data encryption settings
- Private endpoint usage
- Access control model
- Retention and lifecycle policies
- Region pairing and failover approach

Example storage configuration idea:

```bash
az storage account create \
  --name contosodata01 \
  --resource-group rg-data-prod \
  --location eastus \
  --sku Standard_ZRS \
  --kind StorageV2 \
  --https-only true
```

**Decision factors that move you from reference to implementation**

When you convert a reference architecture into an implementation architecture, you must account for:

- **Business requirements** such as uptime, latency, and growth.
- **Security requirements** such as zero trust, segmentation, and encryption.
- **Compliance requirements** such as data residency, auditability, and retention.
- **Operational requirements** such as support model, patching, and monitoring.
- **Cost constraints** such as budget caps and reserved capacity.
- **Existing environment constraints** such as legacy DNS, IP overlap, or identity integration.

These factors often force changes to the reference pattern. For example, a globally distributed reference design may become a single-region active-passive deployment if the business cannot justify the cost of active-active replication.

**Typical artifacts**

A reference architecture is often documented with:

- High-level diagrams
- Pattern descriptions
- Design principles
- Service recommendations
- Trade-off analysis

An implementation architecture is often documented with:

- Detailed network diagrams
- Resource inventories
- Naming and tagging standards
- IaC templates
- Security and operations runbooks
- Deployment sequence and rollback plan

**How to read architecture diagrams**

When you review a diagram, ask:

- Is this showing a pattern or a deployable design?
- Are the components logical or physical?
- Are service names generic or specific?
- Are IPs, regions, and SKUs defined?
- Does the diagram include operational dependencies such as monitoring, backup, and identity?

If the diagram omits those details, it is probably a reference architecture. If it includes them, it is moving toward implementation architecture.

**Exam perspective for AZ-305**

AZ-305 expects you to choose designs that satisfy requirements, not just identify attractive patterns. You should be able to:

- Recognize when a reference architecture is appropriate for discussion or planning.
- Translate a pattern into Azure services and settings.
- Identify missing implementation details that could break a design.
- Compare alternatives based on resiliency, security, manageability, and cost.
- Adjust a design when constraints such as region availability or compliance appear.

Common exam traps include:

- Selecting a pattern without checking whether it can be implemented within the stated constraints.
- Assuming a reference architecture automatically satisfies security or governance requirements.
- Ignoring operational details such as monitoring, backup, and identity.
- Overdesigning for global scale when the requirement only needs regional resilience.

**Practical translation checklist**

Use this checklist when turning a reference architecture into an implementation architecture:

1. Identify the workload type and business goals.
2. Confirm nonfunctional requirements such as availability, RTO, and RPO.
3. Select Azure services that fit the pattern and constraints.
4. Define network topology, address space, and routing.
5. Define identity, access, and secret management.
6. Add security controls and policy enforcement.
7. Add observability, backup, and recovery.
8. Validate cost and operational ownership.
9. Document deployment steps and rollback options.

**Example: from pattern to deployment**

A reference architecture may say: “Use private access for all PaaS services.”

The implementation architecture must specify:

- Which services use private endpoints
- Which VNets host the private endpoints
- How DNS resolves private names
- Which subnets allow endpoint traffic
- How administrators access the services
- How you test name resolution and connectivity

Example DNS validation:

```bash
nslookup contosodb.privatelink.database.windows.net
```

Example private endpoint creation:

```bash
az network private-endpoint create \
  --name pe-sql-prod \
  --resource-group rg-network-prod \
  --vnet-name vnet-hub-eastus \
  --subnet subnet-private-endpoints \
  --private-connection-resource-id /subscriptions/<subId>/resourceGroups/rg-data-prod/providers/Microsoft.Sql/servers/contosodb \
  --group-ids sqlServer \
  --connection-name pe-sql-prod-conn
```

**Best practices**

- Use a reference architecture to align stakeholders on the target pattern.
- Convert it into an implementation architecture only after validating constraints.
- Keep the implementation architecture versioned and reviewable.
- Use IaC to reduce drift between design and deployment.
- Revisit the implementation when requirements, services, or compliance rules change.
- Document assumptions explicitly so the team can challenge them early.

**Common mistakes**

- Treating a reference diagram as if it were deployment-ready.
- Leaving out identity, DNS, and routing details.
- Choosing services before confirming workload requirements.
- Ignoring operational ownership after deployment.
- Failing to account for shared services and platform dependencies.

**Quick comparison**

| Aspect | Reference architecture | Implementation architecture |
|---|---|---|
| Purpose | Guide design and standardize patterns | Define the deployable solution |
| Level of detail | High to medium | High and specific |
| Scope | Reusable across environments | Specific to one environment |
| Includes | Logical components, principles, trade-offs | SKUs, IPs, regions, policies, IaC |
| Audience | Architects, stakeholders, reviewers | Engineers, operators, implementers |

**Key takeaway**

Use the reference architecture to decide *what pattern fits the problem*. Use the implementation architecture to decide *exactly how you will build it in Azure*. Strong AZ-305 answers show that you can move from abstract guidance to concrete, constraint-aware design.
