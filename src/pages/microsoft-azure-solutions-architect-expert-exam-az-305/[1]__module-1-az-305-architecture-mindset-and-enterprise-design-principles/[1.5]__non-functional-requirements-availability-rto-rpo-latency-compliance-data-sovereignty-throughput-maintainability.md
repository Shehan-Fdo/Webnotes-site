## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**What non-functional requirements mean in Azure design**

Non-functional requirements describe *how well* a solution must work, not just *what* it must do. In AZ-305, you translate business goals into architecture choices that balance resilience, performance, governance, and operability.

Common design tensions include:

- Higher availability often increases cost and operational complexity.
- Lower latency can conflict with strict data residency or centralized governance.
- Faster recovery can require replication that increases write latency.
- Strong compliance controls can slow delivery unless you standardize landing zones and policy.

**Availability**

Availability is the ability of a service to remain usable when components fail. In Azure design, you usually think in layers:

- **Platform availability**: the Azure service itself, including its SLA and redundancy options.
- **Application availability**: how your app handles instance loss, zone loss, or regional loss.
- **Dependency availability**: databases, queues, identity, DNS, and external APIs.

Azure SLAs matter in exam scenarios because they are tied to service tiers and deployment choices. A single instance often has no meaningful SLA for high availability, while multiple instances across zones or regions can improve the composite availability target. Always check whether the chosen SKU supports the required redundancy.

Key patterns:

- **Availability Zones** protect against datacenter failure within a region when the service and region support zone redundancy.
- **Availability Sets** are for VM-based workloads when zones are unavailable or when you need fault and update domain separation inside a region.
- **Region pairs** help with regional disaster recovery, but they are not a substitute for zone redundancy.
- **Load balancing** across healthy instances prevents a single node failure from taking down the service.

> Zone support varies by service, region, and SKU. Do not assume every Azure service can be made zone-redundant.

Practical design guidance:

- Use zones for production workloads that need higher uptime and the service supports them.
- Use availability sets for VM-only workloads that cannot use zones.
- Use managed services with built-in redundancy when possible to reduce operational burden.
- Design health probes and graceful degradation so the app can continue with partial dependency loss.

Example: zone-redundant App Service deployment behind Front Door

```text
Users -> Azure Front Door -> App Service in Region A (zone-redundant where supported)
                               -> App Service in Region B (warm standby or active-active)
```

This pattern improves availability and can also improve latency if Front Door routes users to the nearest healthy origin. It does not automatically solve database locality; the data tier still needs its own design.

**RTO and RPO**

<abbr title="Recovery Time Objective">RTO</abbr> is the maximum acceptable time to restore service after an outage.  
<abbr title="Recovery Point Objective">RPO</abbr> is the maximum acceptable data loss measured in time.

These are not abstract terms in AZ-305; they drive the choice between backup-based recovery and replication-based failover.

Typical mapping:

- **Long RTO, higher RPO tolerance**
  - Use backups and restore.
  - Lower cost, simpler operations.
  - Suitable for noncritical workloads.
- **Short RTO, low RPO**
  - Use replication, failover groups, or active-passive architectures.
  - Higher cost and more operational complexity.
- **Near-zero RTO and near-zero RPO**
  - Consider synchronous replication or carefully designed active-active systems.
  - Requires conflict handling, careful data partitioning, and often higher cost.

Recovery options and tradeoffs:

| Option | Best for | RTO | RPO | Notes |
|---|---|---:|---:|---|
| Backup restore | Cost-sensitive recovery | Higher | Higher | Restore time depends on data size and backup location |
| Geo-restore | Regional disaster recovery for some services | Higher | Higher | Uses geo-redundant backups; slower than replication failover |
| Failover groups | Azure SQL Database / Managed Instance | Lower | Low | Good for automatic or planned failover across regions |
| Replication / read replicas | Some data services | Lower | Low | Useful for read scaling and DR, but service-specific |
| Active-active | Critical global apps | Lowest | Lowest | Complex conflict resolution and data consistency design |

Important Azure examples:

- **Azure SQL Database failover groups** provide a strong DR pattern for low RTO and low RPO.
- **Azure Backup** provides restore points and backup retention, but point-in-time restore is workload-specific and not universal across all protected services.
- **Storage account geo-redundancy** improves durability and recovery options, but failover behavior depends on the service and configuration.
- **VM backups** help recover entire machines, but restore time is usually longer than service-native replication.

A practical decision rule:

1. If the workload can tolerate hours of downtime, use backup and restore.
2. If the workload needs minutes of downtime, use replication or failover groups.
3. If the workload needs seconds of downtime, design for multi-instance failover and automated traffic switching.
4. If the workload needs near-zero data loss, validate whether synchronous replication is available and whether the application can tolerate the write latency.

**Latency**

Latency is the time it takes for a request to travel and be processed. In Azure, latency is affected by more than geography.

Major factors:

- **User-to-edge distance**: global entry services can reduce the time to reach the nearest Azure edge.
- **Edge-to-origin distance**: routing to a closer region can reduce end-to-end latency.
- **App-to-database distance**: placing compute and data in different regions increases transaction latency.
- **Network path**: ExpressRoute, VPN, peering, and private endpoints all influence routing and performance.
- **Service behavior**: some services add extra hops or control-plane dependencies.

Design implications:

- **Azure Front Door** can reduce latency for global users by routing them to the nearest healthy origin and terminating traffic at the edge.
- **Azure Traffic Manager** is DNS-based and helps with global failover and routing, but it does not provide the same edge acceleration as Front Door.
- **ExpressRoute** usually provides more predictable latency and less internet variability than site-to-site VPN.
- **Private Endpoints** improve security and routing control, but they can add complexity and sometimes extra network hops depending on topology.
- **VNet peering** is generally lower latency than traversing the public internet and is often used for regional application tiers.

A common exam trap is assuming a global entry service solves all latency problems. It can improve user-to-app latency, but if your app still talks to a distant database, transaction latency remains high. The best design often places the app and its data in the same region or uses a regional pair with a clear failover strategy.

Example: latency-aware topology

```text
Users in Europe -> Front Door -> App in West Europe -> Azure SQL in West Europe
Users in Asia   -> Front Door -> App in Southeast Asia -> Azure SQL replica in same region
```

This design reduces both edge latency and app-to-database latency. If compliance requires data to stay in a specific geography, you must validate that both the app and data placement satisfy that requirement.

**Compliance**

Compliance means meeting legal, regulatory, and organizational requirements. In Azure, compliance is not just a checklist; it is an architecture constraint.

Core governance tools:

- **Management groups** organize subscriptions and apply policy at scale.
- **Azure Policy** enforces or audits configuration rules.
- **Initiatives** group related policies into a single assignment.
- **Defender for Cloud** helps assess security posture and compliance gaps.
- **RBAC** (Role-Based Access Control) limits who can do what.
- **Resource locks** reduce accidental deletion or modification.

Azure landing zones are important in AZ-305 because they provide a repeatable governance foundation:

- Separate platform and application subscriptions.
- Apply policy inheritance from management groups.
- Standardize identity, networking, logging, and security baselines.
- Use environment separation for dev, test, and prod.

Example Azure Policy initiative assignment:

```bash
az policy assignment create \
  --name "baseline-governance" \
  --display-name "Baseline Governance" \
  --scope "/subscriptions/<subscription-id>" \
  --policy-set-definition "/providers/Microsoft.Authorization/policySetDefinitions/<initiative-name>"
```

Why this matters:

- You can enforce allowed regions, required tags, private endpoint usage, or diagnostic settings.
- You reduce drift across teams and subscriptions.
- You make compliance auditable instead of relying on manual review.

Useful compliance patterns:

- Use **deny** policies for hard requirements such as disallowed regions.
- Use **audit** policies for gradual adoption and reporting.
- Use **deployIfNotExists** for automatic remediation, such as enabling diagnostics.
- Use **Defender for Cloud** to surface secure score and regulatory compliance views.

**Data sovereignty**

Data sovereignty is about where data is stored, processed, and backed up, and which legal jurisdiction applies. This is especially important when the exam scenario mentions country, region, residency, or regulatory boundary.

Important nuances:

- Sovereignty must be validated per **service**, per **data type**, and per **replica/backup/log** location.
- A service may store primary data in one region while logs, backups, or telemetry are handled differently.
- Global services may process metadata outside the primary region depending on the service design.
- Region pairs are useful for disaster recovery, but they do not automatically guarantee a strict legal boundary for every service.

Practical guidance:

- Verify where the service stores:
  - primary data
  - backups
  - diagnostic logs
  - telemetry
  - replicas
- Check whether the service supports:
  - regional deployment
  - zone redundancy
  - customer-managed keys
  - private networking
- Prefer services with explicit residency controls when sovereignty is a hard requirement.

Examples:

- **Azure SQL** can be deployed in a specific region, but backup and failover behavior must be checked for residency implications.
- **Azure Storage** offers region-specific redundancy options, but geo-replication crosses regions by design.
- **Microsoft Entra ID** and some global control-plane services are not designed as region-pinned data stores in the same way as application data services.
- **Azure Monitor** and logging services may have retention and region considerations that need validation.

> For exam answers, do not assume “same geography” automatically equals “same legal boundary.” Validate the service behavior.

**Throughput**

Throughput is the amount of work a system can process over time. In Azure, throughput is often constrained by service limits, partitioning, network bandwidth, and downstream dependencies.

Key throughput concepts:

- **Throttling** occurs when a service rejects or delays requests because you exceed its limits.
- **Rate limiting** protects services from overload and can be explicit or implicit.
- **Partitioning** spreads load across multiple partitions or shards.
- **Hot partitions** happen when one partition key receives disproportionate traffic.
- **Backpressure** is a signal that consumers should slow down.

Service-specific considerations:

- **Azure Cosmos DB**
  - Throughput is measured in request units per second (<abbr title="Request Units per second">RU/s</abbr>).
  - Partition key design is critical for scale and avoiding hot partitions.
  - Poor partitioning can cause throttling even when total provisioned throughput looks sufficient.
- **Azure Service Bus**
  - Throughput depends on tier, entity type, message size, and operations per second.
  - Premium tier provides more predictable performance and isolation.
- **Azure Event Hubs**
  - Throughput depends on throughput units or capacity units, partition count, and ingress/egress patterns.
  - Good for telemetry and streaming workloads with high ingest rates.
- **App Service**
  - Throughput depends on instance count, SKU, CPU, memory, and app efficiency.
  - Vertical scaling increases per-instance capacity; horizontal scaling increases total capacity.

Example Cosmos DB partitioning idea:

```json
{
  "id": "order-12345",
  "tenantId": "contoso",
  "region": "westeurope",
  "status": "pending"
}
```

If `tenantId` is a good partition key for your workload, it can distribute load evenly. If one tenant dominates traffic, you may need a different key strategy or a synthetic partition key.

Scaling guidance:

- Use **horizontal scaling** when the service supports it and the workload is stateless or partitionable.
- Use **vertical scaling** when you need more resources per instance and the service tier supports it.
- Use **caching** to reduce repeated reads and protect downstream systems.
- Use **async processing** with queues or events to absorb bursts.

**Maintainability**

Maintainability is how easily you can operate, patch, change, and recover the solution over time. In AZ-305, a maintainable design reduces risk and makes future changes predictable.

Core practices:

- **Infrastructure as Code** (IaC) with Bicep, ARM templates, or Terraform.
- **Versioning** for application and infrastructure changes.
- **Rollback strategy** for failed deployments.
- **Environment separation** for dev, test, staging, and production.
- **Patch management** for VMs, containers, and platform dependencies.
- **Observability** through logs, metrics, traces, and alerts.

Operational patterns:

- Use blue-green or canary deployments to reduce release risk.
- Automate configuration drift detection.
- Keep secrets in Key Vault rather than in code or config files.
- Standardize naming, tagging, and resource organization.
- Use managed services where possible to reduce patching burden.

Example Bicep snippet for repeatable deployment:

```bicep
param location string = resourceGroup().location
param appName string

resource plan 'Microsoft.Web/serverfarms@2023-12-01' = {
  name: '${appName}-plan'
  location: location
  sku: {
    name: 'P1v3'
    tier: 'PremiumV3'
    capacity: 2
  }
}

resource app 'Microsoft.Web/sites@2023-12-01' = {
  name: appName
  location: location
  properties: {
    serverFarmId: plan.id
    httpsOnly: true
  }
}
```

This kind of template improves maintainability because you can recreate, review, and version the environment consistently. Note that changing the SKU here is a deployment-time capacity decision, not the same thing as autoscaling rules.

**Decision tradeoffs you should recognize**

AZ-305 questions often ask you to choose between competing requirements. The right answer depends on which requirement is primary.

Common tradeoffs:

- **Availability vs cost**
  - Multi-zone and multi-region designs cost more.
  - Use the simplest design that meets the SLA and business impact.
- **Latency vs sovereignty**
  - Keeping data in one jurisdiction can increase latency for distant users.
  - Use regional deployments and edge routing carefully.
- **RTO/RPO vs complexity**
  - Lower recovery targets usually require replication and automation.
  - Backup-only designs are simpler but slower to recover.
- **Throughput vs consistency**
  - Higher throughput often means partitioning, eventual consistency, or asynchronous processing.
- **Maintainability vs customization**
  - Managed services reduce operational burden but may limit low-level control.

**Practical Azure design examples**

1. **Global web app with low latency and high availability**
   - Use Azure Front Door for global routing.
   - Deploy app instances in multiple regions.
   - Keep each app instance close to its regional data store.
   - Use health probes and automated failover.

2. **Regulated workload with residency constraints**
   - Use management groups and Azure Policy to restrict regions.
   - Deploy resources only in approved regions.
   - Validate backup, logging, and telemetry residency.
   - Use Defender for Cloud and RBAC for governance.

3. **Mission-critical database with low RTO/RPO**
   - Use Azure SQL failover groups or equivalent service-native replication.
   - Automate failover testing.
   - Keep application connection strings failover-aware.
   - Document recovery runbooks and test them regularly.

4. **Burst-heavy event ingestion system**
   - Use Event Hubs or Service Bus to buffer spikes.
   - Scale consumers horizontally.
   - Monitor throttling and partition hot spots.
   - Use autoscale rules based on queue depth or CPU where appropriate.

**CLI and configuration examples**

Azure Policy initiative assignment:

```bash
az policy assignment create \
  --name "allowed-regions" \
  --display-name "Allowed Regions" \
  --scope "/subscriptions/<subscription-id>" \
  --policy-set-definition "/providers/Microsoft.Authorization/policySetDefinitions/<initiative-id>"
```

App Service autoscale profile concept:

```json
{
  "name": "defaultProfile",
  "capacity": {
    "minimum": "2",
    "maximum": "10",
    "default": "2"
  },
  "rules": [
    {
      "metricTrigger": {
        "metricName": "CpuPercentage",
        "metricResourceUri": "/subscriptions/<sub>/resourceGroups/<rg>/providers/Microsoft.Web/serverfarms/<plan>",
        "timeGrain": "PT1M",
        "statistic": "Average",
        "timeWindow": "PT5M",
        "timeAggregation": "Average",
        "operator": "GreaterThan",
        "threshold": 70
      },
      "scaleAction": {
        "direction": "Increase",
        "type": "ChangeCount",
        "value": "1",
        "cooldown": "PT5M"
      }
    }
  ]
}
```

This is the correct autoscale pattern conceptually: you define a profile, metric trigger, and scale action. Autoscale is not the same as changing the SKU.

SQL failover group concept:

```bash
az sql failover-group create \
  --name "fg-prod" \
  --resource-group "rg-prod" \
  --server "sql-primary" \
  --partner-server "sql-secondary" \
  --failover-policy Automatic \
  --grace-period 60
```

This pattern is useful when you need low RTO and low RPO for Azure SQL workloads.

**How to answer AZ-305 scenario questions**

When you read a scenario, identify the dominant requirement first:

1. Is the priority uptime, recovery, latency, compliance, throughput, or maintainability?
2. Is the workload stateful or stateless?
3. Is the data tier the real bottleneck?
4. Are there residency or regulatory constraints?
5. Does the service tier support the required redundancy or scaling model?
6. Is the business asking for prevention, detection, or recovery?

Then choose the simplest Azure-native design that satisfies the requirement with the least operational risk.

**Quick memory anchors**

- **Availability**: zones, sets, multi-region, SLAs.
- **RTO/RPO**: backup for slower recovery, replication for faster recovery.
- **Latency**: edge routing, regional proximity, network path, data locality.
- **Compliance**: policy, management groups, RBAC, Defender for Cloud.
- **Data sovereignty**: validate residency for data, backups, logs, and telemetry.
- **Throughput**: partitioning, throttling, autoscale, service limits.
- **Maintainability**: IaC, versioning, rollback, patching, environment separation.
