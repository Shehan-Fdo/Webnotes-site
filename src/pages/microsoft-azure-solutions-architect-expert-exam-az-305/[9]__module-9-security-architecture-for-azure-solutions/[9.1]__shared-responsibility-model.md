## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Shared responsibility model overview**

The shared responsibility model defines which security, compliance, and operational tasks are handled by Microsoft and which remain with you. In Azure, the split changes by service model: the more managed the service, the more Microsoft owns, but you still retain responsibility for data, identities, configurations, and access.

For AZ-305, questions often test whether you understand the boundary between:
- **Microsoft compliance of the cloud**: the platform, datacenters, and managed services that Microsoft operates.
- **Customer compliance in the cloud**: your data, identities, configurations, and how you use Azure services.

A useful exam mindset is: Microsoft secures the underlying cloud; you secure what you put in it and how you configure it.

**Core responsibility split**

At a high level, Microsoft is responsible for the physical infrastructure and the foundational cloud platform, while you are responsible for your workloads, data, and governance choices. The exact split depends on whether you use IaaS, PaaS, or SaaS.

- **Microsoft responsibilities**
  - Physical datacenters, hardware, networking, and host OS for managed services.
  - Platform availability, resiliency of Azure services, and patching of Microsoft-managed components.
  - Baseline security controls for the cloud platform.
  - Compliance attestations for Azure services and infrastructure.

- **Customer responsibilities**
  - Data classification, protection, and retention.
  - Identity and access management, including least privilege and privileged access.
  - Secure configuration of services, network exposure, and encryption choices where customer-managed.
  - Application code, guest OS hardening for IaaS, and workload-level compliance.

> AZ-305 commonly asks you to identify who is responsible for a control in a scenario. If the control is about the Azure platform itself, it is usually Microsoft. If it is about your tenant, subscription, resource configuration, or data, it is usually you.

**Responsibility by service model**

The service model determines how much operational burden shifts to Microsoft.

| Layer / control area | IaaS | PaaS | SaaS |
|---|---:|---:|---:|
| Physical datacenter | Microsoft | Microsoft | Microsoft |
| Network fabric | Microsoft | Microsoft | Microsoft |
| Host OS / hypervisor | Microsoft | Microsoft | Microsoft |
| Guest OS patching | Customer | Microsoft | Microsoft |
| Runtime / middleware | Customer | Microsoft | Microsoft |
| Application code | Customer | Customer | Microsoft |
| Data | Customer | Customer | Customer |
| Identity and access | Customer | Customer | Customer |
| Configuration / governance | Customer | Shared | Shared |
| Backup / recovery design | Customer | Shared | Shared |

This table is a revision shortcut, but remember that “shared” does not mean equal. Microsoft may provide the feature, while you decide policy, scope, and access.

**Tenant-level vs subscription/resource-level responsibilities**

Azure exam scenarios often separate responsibilities by scope.

- **Tenant-level responsibilities**
  - Microsoft Entra ID tenant configuration, such as identity governance, conditional access, and external collaboration settings.
  - Directory-wide security settings, privileged role assignments, and tenant trust boundaries.
  - Cross-subscription policies and management group structure.

- **Subscription/resource-level responsibilities**
  - Resource deployment, tagging, locks, and access control at the subscription or resource group level.
  - Azure Policy assignments, resource-specific RBAC (role-based access control), and network security group rules.
  - Service configuration such as storage account public access, Key Vault access policies, or AKS cluster settings.

A common exam trap is confusing tenant-wide identity controls with subscription-scoped resource controls. For example, Conditional Access is a tenant-level identity control, while a storage account firewall rule is a resource-level control.

**Compliance responsibility: cloud vs in the cloud**

Microsoft provides compliance of the cloud, meaning it operates Azure in a compliant manner and maintains certifications, attestations, and controls for the platform. You are responsible for compliance in the cloud, meaning your workloads, data handling, and configurations must meet your regulatory obligations.

This distinction is frequently tested in AZ-305 because many organizations assume that using Azure automatically makes them compliant. It does not.

- **Microsoft compliance of the cloud**
  - Datacenter physical security.
  - Platform certifications and audit reports.
  - Service-level controls for Azure-managed infrastructure.
  - Secure operation of the cloud platform.

- **Customer compliance in the cloud**
  - Data residency and sovereignty decisions.
  - Encryption key management choices.
  - Access reviews, logging, and retention policies.
  - Application design, secure development, and workload-specific controls.

For example, if you store regulated data in Azure Storage, Microsoft secures the service infrastructure, but you must classify the data, restrict access, configure encryption, and define retention.

**How Azure security services fit into the model**

Azure security services are often misunderstood because they support compliance and security, but they do not replace your responsibility.

- **Microsoft Defender for Cloud**
  - Provides security posture management and workload protection recommendations.
  - Helps you identify misconfigurations, missing controls, and threats.
  - Does not make you compliant by itself; you still must remediate findings and govern the environment.
  - Can assess resources across subscriptions and recommend secure configurations.

- **Azure Policy**
  - Enforces or audits organizational rules on Azure resources.
  - Helps you prevent noncompliant deployments, such as disallowing public IPs or requiring tags.
  - Supports customer governance, not Microsoft platform responsibility.
  - Is commonly used to implement compliance guardrails at management group, subscription, or resource group scope.

- **Microsoft Defender for Cloud and Azure Policy together**
  - Defender for Cloud identifies risk and recommends actions.
  - Azure Policy can prevent or audit drift from required standards.
  - In exam scenarios, Policy is the control mechanism; Defender for Cloud is the visibility and recommendation layer.

- **Microsoft Purview**
  - Supports data governance, classification, and compliance management.
  - Helps you discover and label data, but you still own the policy decisions and enforcement.

**Practical examples by Azure service**

Different services shift different responsibilities.

- **Azure Virtual Machines**
  - Microsoft manages the host, hypervisor, and physical infrastructure.
  - You manage the guest OS, patches, antivirus, endpoint protection, and applications.
  - You also manage disk encryption choices, network exposure, and backup strategy.

- **Azure App Service**
  - Microsoft manages the OS, runtime platform, and underlying infrastructure.
  - You manage application code, secrets, identity integration, and app configuration.
  - You still need to secure access, configure TLS, and control deployment pipelines.

- **Azure SQL Database**
  - Microsoft manages the database engine, patching, backups, and high availability platform.
  - You manage database schema, data access, authentication model, and query-level security.
  - You are still responsible for data classification, retention, and access governance.

- **Azure Storage**
  - Microsoft manages the storage service infrastructure and durability.
  - You manage access keys, shared access signatures, network rules, encryption settings, and data lifecycle.
  - If you expose a storage account publicly, that is a customer configuration choice.

- **Azure Kubernetes Service (AKS)**
  - Microsoft manages the control plane in the managed service model.
  - You manage worker node configuration, container images, Kubernetes manifests, network policies, secrets, and workload security.
  - You are responsible for cluster access, namespace design, and admission controls where used.

- **Azure Virtual Desktop**
  - Microsoft provides the service control plane and brokering capabilities.
  - You manage session host images, user profiles, identity, access, and endpoint hardening.
  - You also manage app publishing, FSLogix profile storage, and data protection.

- **Microsoft 365**
  - Microsoft operates the SaaS platform and underlying service infrastructure.
  - You manage tenant configuration, identity, access policies, data governance, and user behavior controls.
  - You remain responsible for compliance decisions, retention, and information protection settings.

These examples matter because AZ-305 often asks you to choose the right service or control based on who owns the operational burden.

**Backup and recovery responsibilities**

Backup and recovery are shared differently depending on the service.

- **IaaS**
  - You usually design and manage backups for VMs, disks, and application data.
  - Azure Backup can automate protection, but you still define policy, retention, and restore objectives.
  - Microsoft does not automatically protect your guest OS or application state unless you configure it.

- **PaaS**
  - Microsoft often provides built-in backup, point-in-time restore, or geo-redundancy features.
  - You choose retention, redundancy, and recovery objectives where configurable.
  - You remain responsible for validating restore procedures and ensuring the data you need is actually protected.

- **SaaS**
  - Microsoft typically manages service availability and platform recovery.
  - You are still responsible for data retention, export, legal hold, and tenant-level recovery planning.
  - For Microsoft 365, for example, Microsoft provides service resiliency, but you must manage retention policies, eDiscovery, and data governance.

Concrete examples:
- **Azure SQL Database**
  - Microsoft manages automated backups and point-in-time restore capabilities.
  - You decide retention settings and whether to use long-term retention.
- **Azure Storage**
  - Microsoft provides redundancy options such as locally redundant storage and geo-redundant storage.
  - You choose the redundancy model and configure soft delete or versioning where appropriate.
- **Microsoft 365**
  - Microsoft manages service continuity and platform resilience.
  - You manage retention, legal hold, and recovery of user content through tenant features and governance processes.

> Do not assume “Microsoft backs it up” means “you can ignore recovery design.” AZ-305 scenarios often test whether you understand that recovery objectives are still a customer decision.

**Identity, access, and governance**

Identity is one of the clearest customer responsibilities in Azure.

- **Microsoft Entra ID**
  - Microsoft operates the identity platform.
  - You configure users, groups, roles, Conditional Access, and identity governance.
  - You decide how strong authentication should be and who can access what.

- **Azure RBAC**
  - You assign permissions to subscriptions, resource groups, and resources.
  - You should use least privilege and separate duties where possible.
  - Built-in roles are Microsoft-defined, but role assignment is your responsibility.

- **Management groups and Azure Policy**
  - You use management groups to apply governance across subscriptions.
  - You use Azure Policy to enforce standards such as allowed locations, required tags, or approved SKUs.
  - These are customer governance controls, not Microsoft platform controls.

A practical governance example:

```bash
az policy assignment create \
  --name require-tags \
  --display-name "Require costCenter and owner tags" \
  --scope /subscriptions/<subscription-id> \
  --policy /providers/Microsoft.Authorization/policyDefinitions/<policy-definition-id>
```

This kind of control helps you meet internal compliance requirements, but it does not transfer responsibility away from you.

**Security controls and where they belong**

A good way to reason about the model is to map controls to ownership.

- **Microsoft-owned controls**
  - Datacenter physical security.
  - Host patching for managed services.
  - Service availability and platform resilience.
  - Baseline security of Azure-managed components.

- **Customer-owned controls**
  - Network segmentation, firewall rules, and private endpoints where configured.
  - Encryption key management when using customer-managed keys.
  - Secrets management and application authentication.
  - Logging, alerting, and incident response for your workloads.

- **Shared controls**
  - Encryption at rest and in transit, where Microsoft provides the feature and you configure the policy.
  - Backup and restore, where Microsoft may provide the mechanism but you define the objectives.
  - Monitoring, where Microsoft supplies telemetry and you decide what to alert on and how to respond.

**Common exam scenarios**

AZ-305 often frames questions around “who is responsible” or “which service reduces operational overhead.”

- If the question asks about patching the guest OS of a VM, the answer is customer responsibility.
- If the question asks about patching the Azure SQL Database engine, the answer is Microsoft responsibility.
- If the question asks about restricting who can create resources in a subscription, the answer is customer governance using RBAC or Azure Policy.
- If the question asks about physical security of the datacenter, the answer is Microsoft.
- If the question asks about protecting data in Microsoft 365, the answer is shared, with the customer responsible for tenant configuration, retention, and access controls.

A useful rule: if you can change it in your tenant, subscription, or resource, you usually own the responsibility for governing it.

| Scenario | Microsoft | Customer |
|---|---|---|
| Datacenter physical security | Yes | No |
| Hypervisor and host patching | Yes | No |

**Practical design guidance**

When you design an Azure solution, use the shared responsibility model to decide which controls you must add.

- Choose **PaaS** when you want Microsoft to take over more platform operations such as patching and availability.
- Choose **IaaS** when you need maximum control over the OS and runtime, but accept more operational responsibility.
- Use **SaaS** when you want the least operational overhead, but remember that governance, identity, and data protection still remain with you.
- Apply **Azure Policy** and **Defender for Cloud** to reduce drift and improve visibility.
- Document ownership clearly for security, backup, identity, and compliance tasks so teams do not assume Microsoft handles them.

**Example architecture thinking**

If you deploy a web app on App Service with Azure SQL Database and Microsoft Entra ID:
- Microsoft manages the App Service platform and SQL engine.
- You manage the app code, secrets, identity integration, access policies, and data governance.
- You use Defender for Cloud to identify risks and Azure Policy to enforce standards.
- You still define backup retention, access reviews, and compliance controls.

If you deploy the same app on VMs:
- Microsoft manages the physical infrastructure and hypervisor.
- You manage the OS, middleware, patching, app runtime, and most of the security stack.
- Your operational burden is higher, but so is your control.

**Key exam takeaways**

- **Microsoft compliance of the cloud** covers the Azure platform and infrastructure.
- **Customer compliance in the cloud** covers your data, identities, configurations, and workload controls.
- **Tenant-level** controls usually involve identity and directory governance.
- **Subscription/resource-level** controls usually involve deployment, access, and policy enforcement.
- **Defender for Cloud** helps you assess and improve security posture.
- **Azure Policy** helps you enforce governance and compliance rules.
- The more managed the service, the more Microsoft owns, but you never lose responsibility for your data and access.
