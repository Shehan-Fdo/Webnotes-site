## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Zero Trust overview**

Zero Trust is a security design model built on the idea that no request is trusted by default, even if it originates inside the network perimeter. In Azure architecture, you apply this model by continuously verifying identity, device health, location, risk, and resource sensitivity before granting access.

The core mindset is to reduce implicit trust and replace it with explicit, policy-driven decisions. For AZ-305, expect design questions that ask you to combine identity, device compliance, network segmentation, data protection, and monitoring into a layered architecture.

- **Verify explicitly**: evaluate every access request using identity, device, location, risk, and resource context.
- **Least privilege**: grant only the minimum access needed, and prefer just-in-time and just-enough access.
- **Assume breach**: design as if attackers already have some foothold and limit blast radius.

**Core Zero Trust pillars**

Azure Zero Trust designs usually map to three broad control areas: identity, devices, and resources. In practice, you implement them through Microsoft Entra ID, Conditional Access, Microsoft Intune, network controls, data controls, and security monitoring.

<dl>
<dt>Identity</dt>
<dd>Authenticate users and workloads, then authorize access based on policy, risk, and context.</dd>
<dt>Device</dt>
<dd>Require managed, compliant, and trusted endpoints for sensitive access.</dd>
<dt>Network and resources</dt>
<dd>Segment traffic, restrict exposure, and protect data and applications at each layer.</dd>
</dl>

A good exam answer usually shows that you do not rely on a single control. For example, you do not use VPN access alone to prove trust; you combine identity assurance, device compliance, and resource-level restrictions.

**Identity-first access control**

Identity is the primary control plane in Zero Trust. In Azure, Microsoft Entra ID provides authentication, conditional access, multifactor authentication (MFA), risk-based policies, and privileged identity controls.

Use Conditional Access to enforce access decisions dynamically. Typical signals include user risk, sign-in risk, device compliance, application sensitivity, and network location. This is more effective than static allow lists because it adapts to changing conditions.

- **MFA**: require a second factor for sensitive apps, admin roles, or risky sign-ins.
- **Conditional Access**: block, require MFA, require compliant device, or require approved client app.
- **Privileged Identity Management (PIM)**: make admin access eligible and time-bound instead of permanently active.
- **Identity Protection**: detect risky users and risky sign-ins and trigger remediation.

A common pattern is to require MFA for all users, then add stronger controls for privileged roles and high-value applications. For example, finance apps may require a compliant device and a trusted location, while admin portals may require phishing-resistant authentication and PIM activation.

```bash
# Example: inspect a user's sign-in risk and MFA status with Microsoft Graph PowerShell
Connect-MgGraph -Scopes "User.Read.All","IdentityRiskEvent.Read.All","Policy.Read.All"
Get-MgUser -UserId user@contoso.com | Select-Object DisplayName,UserPrincipalName,AccountEnabled
```

**Device trust and endpoint management**

Device trust is a major part of Zero Trust because a valid identity alone does not prove the endpoint is safe. Microsoft Intune is the primary endpoint management service used to enroll devices, apply configuration profiles, deploy compliance policies, and report device health.

Conditional Access can require a device to be marked compliant before granting access. Compliance typically checks settings such as encryption, password or PIN requirements, OS version, jailbreak/root status, and security baselines. This is especially important for SaaS apps, admin portals, and sensitive internal applications.

Device join state matters in architecture decisions:

| Join state | Typical use | Zero Trust relevance |
|---|---|---|
| **Microsoft Entra joined** | Cloud-native corporate devices | Strong fit for modern management and Conditional Access |
| **Hybrid Microsoft Entra joined** | Domain-joined devices that also register with Entra ID | Useful during migration from on-premises AD |
| **Microsoft Entra registered** | Personal or BYOD devices | Lower trust; often paired with app protection policies |

Use **Microsoft Entra joined** for cloud-first managed endpoints when possible. Use **hybrid join** when you still depend on on-premises domain services or legacy management. Use **registered** for bring-your-own-device scenarios where you want app-level protection rather than full device control.

Intune app protection policies are useful when you cannot fully manage the device. They can restrict copy/paste, save-as, and data transfer between managed and unmanaged apps. This is a practical way to support mobile and BYOD access without granting full device trust.

> Do not confuse device registration with device compliance. A registered device can exist in Entra ID without meeting the compliance requirements needed by Conditional Access.

Example Conditional Access logic:

1. Require MFA for all cloud apps.
2. Require compliant device for SharePoint and admin portals.
3. Block legacy authentication.
4. Require phishing-resistant authentication for privileged roles.

**Network segmentation and east-west traffic control**

Zero Trust is not only about internet-facing access. It also limits lateral movement inside Azure. East-west traffic control reduces the chance that a compromised workload can reach other workloads.

Use network segmentation to isolate tiers, environments, and trust zones. Common tools include network security groups (NSGs), Azure Firewall, application security groups (ASGs), user-defined routes, and private connectivity patterns.

- **NSGs**: filter traffic at subnet or NIC level using 5-tuple rules.
- **ASGs**: group VMs by application role so rules stay readable and scalable.
- **Service tags**: simplify rules for Microsoft services or broad Azure ranges.
- **Azure Firewall**: centralize stateful filtering, FQDN filtering, threat intelligence, and logging.
- **Private endpoints**: remove public exposure for PaaS services.

ASGs are especially useful in multi-tier apps. Instead of writing rules for individual IP addresses, you can allow traffic from `asg-web` to `asg-app` on port 443 and from `asg-app` to `asg-data` on port 1433.

```bash
# Example: create an NSG rule that allows web tier to app tier
az network nsg rule create \
  --resource-group rg-app \
  --nsg-name nsg-app \
  --name Allow-Web-To-App \
  --priority 200 \
  --source-address-prefixes VirtualNetwork \
  --source-port-ranges '*' \
  --destination-address-prefixes '*' \
  --destination-port-ranges 443 \
  --access Allow \
  --protocol Tcp
```

For larger environments, use a hub-spoke topology or Azure Virtual WAN to centralize shared services and inspection. The hub typically hosts Azure Firewall, DNS, VPN/ExpressRoute gateways, and security tooling. Spokes host application workloads and are peered to the hub with controlled routing.

- **Hub-spoke**: best when you want centralized inspection and simpler governance.
- **Virtual WAN**: useful for large-scale branch, remote user, and global connectivity scenarios.
- **Micro-segmentation**: use NSGs, ASGs, and route controls to isolate workloads within a spoke.

A strong design often combines private endpoints with forced tunneling through Azure Firewall. That way, traffic to PaaS services stays private, while outbound internet access is inspected and logged.

**Data protection and private access**

Zero Trust treats data as a high-value asset that must be protected at rest, in transit, and in use. In Azure, this means using encryption, access controls, classification, and network isolation together.

Most Azure storage services encrypt data at rest by default using Microsoft-managed keys. For higher control, use customer-managed keys (CMK) in Azure Key Vault or Managed HSM. This is common for regulated workloads or when you need key rotation and separation of duties.

Practical Azure data controls include:

- **Storage public access**: disable public blob access when not needed.
- **Public network access**: disable public endpoints for storage, SQL, and other PaaS services when private access is available.
- **Private endpoints**: expose services through private IPs in your VNet.
- **DNS integration**: configure private DNS zones so clients resolve the private endpoint correctly.
- **Database firewall rules**: avoid broad IP allow lists when private connectivity is possible.

Private endpoints change both access control and name resolution. If you create a private endpoint for Azure SQL Database, clients in the VNet should resolve the service name to the private IP through a private DNS zone such as `privatelink.database.windows.net`. Without DNS integration, clients may still resolve the public endpoint and fail or bypass the intended path.

```bash
# Example: disable public network access on a storage account
az storage account update \
  --name mystorageacct \
  --resource-group rg-data \
  --public-network-access Disabled
```

For SQL, prefer private access and least-privilege database roles. For storage, combine RBAC, shared access signatures when appropriate, and network restrictions. For highly sensitive data, add Microsoft Purview for classification, labeling, and governance, and use DLP (data loss prevention) policies to reduce exfiltration risk.

**Assume breach and detection/response**

Assume breach means you design for containment, detection, and rapid response rather than perfect prevention. This is where monitoring and security operations become part of the architecture, not an afterthought.

Microsoft Sentinel is the cloud-native SIEM (security information and event management) and SOAR (security orchestration, automation, and response) platform in Azure. Microsoft Defender XDR integrates signals across endpoints, identities, email, and cloud apps to help correlate attacks and prioritize incidents.

Use these capabilities together:

- **Defender XDR**: detect threats across identities, endpoints, and cloud workloads.
- **Microsoft Sentinel**: centralize logs, analytics, hunting, and automated response.
- **Alerting**: trigger notifications and playbooks for suspicious activity.
- **Incident response**: isolate devices, disable accounts, revoke sessions, and investigate logs.

A practical workflow is:

1. Defender detects a suspicious sign-in or malware event.
2. Sentinel correlates the event with identity and network telemetry.
3. An alert triggers an incident and a response playbook.
4. The playbook disables the account, revokes tokens, and isolates the endpoint if needed.
5. Analysts review logs and close the incident after remediation.

This is especially important for privileged accounts and internet-facing workloads. If an attacker compromises one workload, your goal is to detect lateral movement quickly and contain it before it reaches sensitive data.

**Governance, policy inheritance, and landing zones**

Zero Trust at scale depends on governance. Azure landing zones provide a structured way to organize subscriptions, management groups, policies, and shared services so security controls apply consistently.

Management groups are the main inheritance boundary for Azure Policy and role assignments. Place subscriptions into a hierarchy that reflects environment, business unit, or platform ownership. Then assign policies and initiatives at the management group level so child subscriptions inherit the baseline.

An **initiative** is a collection of Azure Policy definitions packaged together for a common goal, such as a secure landing zone baseline. This is useful when you want to enforce multiple controls at once, such as allowed regions, diagnostic settings, private endpoint requirements, and tagging standards.

- **Management group design**: use it to separate platform, landing zone, and application subscriptions.
- **Policy inheritance**: assign baseline policies high in the hierarchy to reduce drift.
- **Initiative assignments**: bundle related controls for repeatable governance.
- **Exemptions**: use sparingly and document them for exceptions.

A common landing zone pattern is:

- Root management group
  - Platform management group
    - Connectivity subscription
    - Identity subscription
    - Management subscription
  - Landing zones management group
    - Prod subscriptions
    - Non-prod subscriptions

This structure helps you apply different policies to platform services and application workloads while keeping a consistent security baseline.

```json
{
  "properties": {
    "displayName": "Secure Landing Zone Baseline",
    "policyType": "Custom",
    "mode": "All",
    "parameters": {},
    "policyRule": {
      "if": {
        "field": "type",
        "equals": "Microsoft.Storage/storageAccounts"
      },
      "then": {
        "effect": "deny"
      }
    }
  }
}
```

**Practical design patterns**

Zero Trust is easiest to remember when you think in patterns rather than isolated products. The right pattern depends on the workload and risk level.

- **SaaS access**: Entra ID, MFA, Conditional Access, Intune compliance, and session controls.
- **Internal app access**: private endpoints, app segmentation, and identity-aware access.
- **Admin access**: PIM, phishing-resistant MFA, compliant device, and logging.
- **Data access**: RBAC, private networking, encryption, and classification.
- **Operations**: Sentinel, Defender XDR, alerting, and incident workflows.

For example, a secure Azure SQL deployment might use a private endpoint, public network access disabled, Entra authentication, least-privilege database roles, diagnostic logs to Sentinel, and policy enforcement to prevent public exposure. That is a much stronger Zero Trust design than simply placing the database behind a firewall rule.

**Exam guidance**

AZ-305 questions often ask you to choose the best combination of controls rather than the single strongest control. Look for clues about compliance, device ownership, legacy dependencies, and operational scale.

- If the scenario mentions BYOD, prefer app protection policies and Conditional Access over full device management.
- If the scenario mentions corporate laptops, prefer Intune compliance and Entra joined devices.
- If the scenario mentions lateral movement, emphasize segmentation, private endpoints, and centralized inspection.
- If the scenario mentions incident response, include Sentinel, Defender XDR, and automated remediation.
- If the scenario mentions governance drift, use management groups, policy inheritance, and initiatives.

A strong answer usually shows layered defense:
identity controls at the front door, device trust at the endpoint, segmentation inside the network, private access to data, and continuous monitoring for breach detection.
