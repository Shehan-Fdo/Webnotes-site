## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Identity model overview**

Workload identity is the pattern of giving an application, service, or automation process its own identity so it can authenticate to Azure and other services without storing long-lived secrets. In Azure architecture, the two most important options are **managed identities** and **workload identities**. They solve the same core problem—secure non-human authentication—but they apply in different hosting environments.

A good design choice depends on where the workload runs:

- Use **managed identity** when the workload runs on an Azure resource that supports it.
- Use **workload identity** when the workload runs outside Azure or in Kubernetes and needs federated sign-in.
- Avoid client secrets and certificates unless you have a specific constraint that prevents identity federation.

**Why this matters in architecture**

Secret-based authentication creates operational risk:
- Secrets expire and require rotation.
- Secrets are often copied into configuration files, pipelines, or container images.
- Secrets increase blast radius if leaked.
- Secrets make least-privilege enforcement harder to maintain.

Identity-based authentication reduces that risk by letting Azure issue tokens at runtime. The workload requests a token, Azure validates the workload’s identity, and the service authorizes access based on role assignments or application permissions.

**Managed identities**

A **managed identity** is an identity in Microsoft Entra ID that is automatically managed by Azure. You do not create credentials, store secrets, or rotate keys manually. Azure handles the lifecycle of the identity.

There are two types:

- **System-assigned managed identity**
  - Tied to a single Azure resource.
  - Created and deleted with the resource.
  - Best when the identity should not outlive the workload.
- **User-assigned managed identity**
  - Created as a standalone Azure resource.
  - Can be attached to one or more Azure resources.
  - Best when multiple workloads need the same identity or when you want identity lifecycle independent of the host.

**Common managed identity use cases**

Managed identities are the default choice for Azure-hosted workloads that need to access other Azure services.

Typical examples include:
- An Azure App Service reading secrets from Azure Key Vault.
- An Azure Function writing messages to Azure Service Bus.
- An Azure Virtual Machine accessing Azure Storage or Azure SQL Database.
- An Azure Container App pulling data from Cosmos DB.
- An Azure Automation runbook authenticating to Azure Resource Manager.

**Why managed identity is preferred in Azure**

Managed identity is attractive because it:
- Eliminates embedded credentials.
- Uses Azure-native token acquisition.
- Integrates cleanly with Azure role-based access control (RBAC).
- Supports least privilege at the resource level.
- Simplifies rotation because there is no secret to rotate.

A managed identity is especially useful when:
- The workload is fully hosted in Azure.
- You want minimal operational overhead.
- You need access to Azure services that support Microsoft Entra authentication.
- You want to avoid certificate management in app code or deployment pipelines.

**Managed identity access flow**

A typical flow looks like this:

1. You enable a managed identity on the Azure resource.
2. You grant that identity access to a target resource, such as Key Vault or Storage.
3. The workload requests an access token from the Azure Instance Metadata Service (IMDS) or platform endpoint.
4. Azure issues a token for the target resource.
5. The workload uses the token to call the service.

Example: Azure VM accessing Key Vault with a system-assigned identity

```bash
# Enable system-assigned managed identity on a VM
az vm identity assign \
  --resource-group rg-app \
  --name vm-app01

# Get the principal ID
az vm show \
  --resource-group rg-app \
  --name vm-app01 \
  --query identity.principalId \
  --output tsv

# Grant Key Vault secrets access using RBAC
az role assignment create \
  --assignee <principal-id> \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<sub-id>/resourceGroups/rg-sec/providers/Microsoft.KeyVault/vaults/kv-prod
```

Example: token request from inside an Azure VM

```bash
curl -H Metadata:true \
  "http://169.254.169.254/metadata/identity/oauth2/token?api-version=2018-02-01&resource=https://vault.azure.net"
```

The response contains an access token that the application can use against Key Vault.

**Managed identity and authorization**

Authentication proves who the workload is. Authorization determines what it can do.

For Azure services, you usually combine managed identity with one of these:
- **Azure RBAC** for data-plane or management-plane access where supported.
- **Access policies** in some legacy Key Vault designs, though RBAC is generally preferred for new deployments.
- **Application permissions** when the target is a Microsoft Entra-protected API.

Use the least privileged role possible:
- `Reader` for read-only management-plane access.
- `Storage Blob Data Reader` for blob read access.
- `Storage Blob Data Contributor` for read/write blob access.
- `Key Vault Secrets User` for secret retrieval.
- `Key Vault Crypto User` for cryptographic operations.

> Managed identity is not a permission by itself. It is an identity that you must still authorize against each target resource.

**User-assigned vs system-assigned managed identity**

| Aspect | System-assigned | User-assigned |
|---|---|---|
| Lifecycle | Bound to one resource | Independent resource |
| Reuse | No | Yes |
| Best for | Single workload | Shared identity across workloads |
| Cleanup | Automatic with host | Manual |
| Portability | Low | Higher |

Choose **system-assigned** when the identity should be tightly coupled to one resource. Choose **user-assigned** when you need:
- Multiple apps to share the same identity.
- Blue/green deployments with identity continuity.
- A stable identity during host replacement.
- Easier separation of identity lifecycle from compute lifecycle.

**Managed identity limitations and design considerations**

Managed identities are powerful, but they are not universal:
- They only work where Azure provides platform support.
- They are not ideal for workloads running outside Azure.
- They do not replace authorization design.
- They do not eliminate the need to secure network paths, private endpoints, or firewall rules.

Practical considerations:
- Some services support managed identity only for specific operations.
- Cross-tenant scenarios may require additional planning.
- You still need to configure network access to the target service.
- Token acquisition depends on platform availability and correct resource URI.

**Workload identity**

**Workload identity** usually refers to federated identity for applications and automation that run outside a managed Azure host, especially in Kubernetes. Instead of storing a secret, the workload exchanges an external token for an Azure access token through Microsoft Entra federation.

The most common Azure implementation is **Microsoft Entra Workload ID** for Kubernetes. It uses:
- An **OpenID Connect (OIDC)** issuer.
- A federated identity credential in Microsoft Entra ID.
- A Kubernetes service account token or external OIDC token.
- Token exchange with Azure without a client secret.

This is especially useful when you run:
- Azure Kubernetes Service (AKS) workloads.
- GitHub Actions or other CI/CD systems with OIDC federation.
- Containerized apps in non-Azure Kubernetes clusters.
- Hybrid workloads that need Azure access without secret storage.

**Why workload identity exists**

Traditional service principal authentication often uses:
- Client secrets, which expire and are easy to leak.
- Certificates, which are safer than secrets but still require lifecycle management.

Workload identity improves this by:
- Removing long-lived secrets.
- Using short-lived federated tokens.
- Aligning with modern zero-trust authentication patterns.
- Reducing operational overhead in pipelines and clusters.

**Workload identity in Kubernetes**

In AKS, workload identity lets a pod authenticate as a Microsoft Entra application or user-assigned managed identity through federation.

High-level flow:
1. You enable OIDC issuer and workload identity on the cluster.
2. You create a federated identity credential in Microsoft Entra ID.
3. You annotate the Kubernetes service account.
4. The pod uses the service account token to request an Azure token.
5. Azure validates the federation and issues an access token.

Example AKS-related setup commands:

```bash
# Enable OIDC issuer and workload identity on an AKS cluster
az aks update \
  --resource-group rg-aks \
  --name aks-prod \
  --enable-oidc-issuer \
  --enable-workload-identity

# Get the OIDC issuer URL
az aks show \
  --resource-group rg-aks \
  --name aks-prod \
  --query oidcIssuerProfile.issuerUrl \
  --output tsv
```

Example federated identity credential concept:

```json
{
  "name": "aks-federation",
  "issuer": "https://oidc.prod-aks.azure.com/<tenant>/<cluster-id>/",
  "subject": "system:serviceaccount:app-ns:app-sa",
  "audiences": [
    "api://AzureADTokenExchange"
  ]
}
```

Example Kubernetes service account annotation:

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: app-sa
  namespace: app-ns
  annotations:
    azure.workload.identity/client-id: "<client-id>"
```

**When to use workload identity**

Use workload identity when:
- The workload runs in Kubernetes and needs Azure access.
- You want to avoid storing secrets in Kubernetes secrets.
- You need CI/CD federation from an external identity provider.
- You want short-lived, auditable authentication.

It is a strong fit for:
- Microservices in AKS accessing Key Vault, Storage, or Service Bus.
- Build pipelines deploying to Azure without service principal secrets.
- Multi-environment deployments where identity should be tied to workload context.

**Managed identity vs workload identity**

| Scenario | Best choice | Reason |
|---|---|---|
| Azure VM accessing Storage | Managed identity | Native Azure host support |
| App Service calling Key Vault | Managed identity | Built-in platform identity |
| AKS pod accessing Azure SQL | Workload identity | Kubernetes federation |
| GitHub Actions deploying to Azure | Workload identity | Secretless CI/CD |
| Multiple Azure apps sharing one identity | User-assigned managed identity | Reusable Azure-native identity |
| Non-Azure container platform accessing Azure | Workload identity | No Azure host dependency |

**Practical architecture patterns**

A common secure architecture uses identity plus network controls:
- App Service uses a managed identity to read secrets from Key Vault.
- Key Vault is reachable only through a private endpoint.
- The app is integrated with a virtual network.
- RBAC grants only secret read access.

Another pattern:
- AKS pods use workload identity to access Storage and Service Bus.
- The cluster uses OIDC federation.
- No Kubernetes secret stores Azure credentials.
- Network policies and private endpoints restrict service access.

Example: App Service to Key Vault with managed identity

```bash
az webapp identity assign \
  --resource-group rg-web \
  --name app-web01

az role assignment create \
  --assignee <principal-id> \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<sub-id>/resourceGroups/rg-sec/providers/Microsoft.KeyVault/vaults/kv-prod
```

Example: Azure CLI login with a user-assigned managed identity on a VM

```bash
az login --identity --username <client-id-or-resource-id>
az account show
```

**Security best practices**

- Use **least privilege** for every identity.
- Prefer **user-assigned managed identity** when identity continuity matters.
- Prefer **workload identity** over client secrets in Kubernetes and CI/CD.
- Restrict access with **private endpoints** and firewall rules.
- Monitor sign-in logs and resource access logs.
- Separate identities by application, environment, and function.
- Rotate away from secrets and certificates where federation is available.

**Common exam cues**

Look for these clues in AZ-305 scenarios:
- “Azure-hosted app needs access to Key Vault without storing credentials” → managed identity.
- “AKS pod needs to access Azure Storage securely” → workload identity.
- “Multiple apps need the same identity” → user-assigned managed identity.
- “Pipeline must authenticate to Azure without a secret” → workload identity or federated credential.
- “Identity should be deleted with the resource” → system-assigned managed identity.

**Troubleshooting checklist**

If authentication fails, verify:
- The identity is enabled on the resource.
- The target resource has the correct role assignment.
- The token audience/resource URI is correct.
- The workload is using the right identity when multiple identities exist.
- Network access to the target service is allowed.
- The federated credential subject, issuer, and audience match exactly.

Useful checks:
```bash
# Show identity details on an Azure resource
az vm show -g rg-app -n vm-app01 --query identity

# List role assignments for a principal
az role assignment list --assignee <principal-id> --output table

# Inspect AKS OIDC issuer
az aks show -g rg-aks -n aks-prod --query oidcIssuerProfile
```

**Decision guidance**

Use this mental model:
- If the workload runs on an Azure resource that supports platform identity, choose **managed identity**.
- If the workload runs in Kubernetes or external automation and can federate an external token, choose **workload identity**.
- If you need a reusable Azure identity across multiple hosts, choose **user-assigned managed identity**.
- If you need the simplest secure default for Azure-native apps, start with **system-assigned managed identity**.

**Key takeaways**

- **Managed identity** is Azure-native, secretless authentication for supported Azure resources.
- **Workload identity** extends secretless authentication to Kubernetes and external federated workloads.
- Authentication and authorization are separate; you still need RBAC or equivalent permissions.
- Prefer short-lived tokens and federation over secrets and certificates.
- The right choice depends on hosting model, lifecycle needs, and operational simplicity.
