## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Overview**

Azure private access patterns let you reach Azure services and your own services without exposing traffic to the public internet. In exam scenarios, you usually choose between three broad approaches:

- **Private Endpoint** for consuming a PaaS service privately through a private IP in your VNet.
- **Private Link Service** for publishing your own service privately to other VNets or tenants.
- **DNS and routing design** to make name resolution and traffic flow work consistently across Azure, on-premises, and multiple VNets.

The key design goal is to reduce public exposure while preserving predictable connectivity, identity, and operational control.

**Core concepts**

A **private endpoint** is a network interface in your subnet that maps to a specific Azure service instance. Clients connect to the private IP, and Azure privately carries the traffic to the service. This is the standard pattern for services such as Azure Storage, Azure SQL Database, Key Vault, Azure App Configuration, and many others.

A **Private Link Service** is the publishing side of the same model. You place a service behind an Azure Standard Load Balancer, expose it through a Private Link Service, and consumers create private endpoints to reach it. This is the pattern for private access to your own applications or APIs.

A **private DNS zone** is usually required so clients resolve the service name to the private endpoint IP instead of the public endpoint. Without DNS alignment, the network path may be private but the client still resolves the public address.

**Private Endpoint architecture**

A private endpoint is deployed into a subnet in a virtual network. The endpoint gets a private IP from that subnet and represents a connection to a specific resource instance, not to the whole service family.

Important design points:

- **Subnet planning** matters because each private endpoint consumes an IP address.
- **Service association** is resource-specific; one storage account or one SQL server can have multiple private endpoints for different subresources.
- **Public access** can remain enabled or be disabled depending on your security posture.
- **Approval workflow** controls whether the endpoint can be used.

Private endpoints are often used when you want:

- Private access from application subnets to PaaS services.
- Access from on-premises networks through VPN or ExpressRoute.
- Segmented access across subscriptions, tenants, or business units.
- A controlled migration path where public access is removed after validation.

**Connection states and lifecycle**

Private endpoint connections have a lifecycle that matters in design and troubleshooting. Common states include:

- **Pending**: the request exists, but the service owner has not approved it yet.
- **Approved**: traffic can flow if DNS and routing are correct.
- **Rejected**: the connection is denied and cannot be used.
- **Disconnected**: the link is no longer active, often after deletion or service-side removal.

In practice, a private endpoint can exist but still not work if the connection is pending or rejected. Always verify both the network object and the service-side connection state.

**Private Link Service for your own applications**

Use **Private Link Service** when you need to expose a service you control privately to consumers. This is common for shared internal APIs, line-of-business platforms, or partner access.

Typical pattern:

- Your service runs behind an Azure Standard Load Balancer.
- You publish it through a Private Link Service.
- Consumers create private endpoints in their VNets.
- Consumers resolve the service name to a private IP through DNS.

This pattern is useful when you want:

- Private access across tenants without opening inbound public endpoints.
- Centralized service hosting with controlled consumer onboarding.
- A stable private interface for internal or partner applications.

> Private Link Service is not the same as a private endpoint. The service side publishes access; the consumer side creates the endpoint.

**DNS design for private access**

DNS is usually the hardest part of private access patterns. The network path may be correct, but if name resolution points to the public endpoint, the application still fails or bypasses the intended path.

The most common design is **split-horizon DNS**:

- Internal clients resolve the service name to the private endpoint IP.
- External clients resolve the same name to the public endpoint, if public access is still enabled.
- The private DNS zone is linked to the VNets that need private resolution.

For Azure PaaS services, you often use a private DNS zone such as:

- `privatelink.blob.core.windows.net`
- `privatelink.database.windows.net`
- `privatelink.vaultcore.azure.net`
- `privatelink.azurewebsites.net`

The exact zone depends on the service. The private endpoint usually creates or uses a DNS record that maps the service name to the private IP.

**DNS patterns in hybrid and multi-VNet designs**

For enterprise environments, you often need more than a single private DNS zone link.

Common patterns:

- **Single VNet**: link the private DNS zone to the application VNet.
- **Hub-and-spoke**: link the zone to the hub or to each spoke that needs resolution.
- **On-premises access**: forward queries from on-premises DNS to Azure DNS Private Resolver or to custom DNS servers in Azure.
- **Cross-subscription or cross-tenant**: ensure the DNS zone is reachable from the consumer network even if the endpoint and service live elsewhere.

A common split-horizon design is:

1. On-premises DNS forwards `privatelink.<service-zone>` queries to Azure.
2. Azure DNS Private Resolver receives the query through an inbound endpoint.
3. The resolver answers from the private DNS zone linked to the relevant VNets.

If you use custom DNS servers, they must be able to resolve the private zone and forward appropriately. Azure-provided DNS works well inside a VNet, but hybrid scenarios often need explicit forwarding.

**Azure DNS Private Resolver**

Azure DNS Private Resolver helps you resolve names between on-premises and Azure without deploying your own DNS VMs. It supports:

- **Inbound endpoints** for receiving DNS queries from on-premises or other networks.
- **Outbound endpoints** for forwarding Azure-originated queries to external DNS servers.
- **DNS forwarding rulesets** to control which domains are forwarded and where.

A practical hybrid design often looks like this:

```text
On-prem DNS
   |
   | conditional forwarder for privatelink.database.windows.net
   v
Azure DNS Private Resolver inbound endpoint
   |
   v
Private DNS zone linked to spoke VNet
```

Example deployment flow with Azure CLI:

```bash
# Create a DNS private resolver
az network dns-resolver create \
  --resource-group rg-network \
  --name dnsr-hub \
  --location eastus \
  --virtual-network vnet-hub

# Create an inbound endpoint
az network dns-resolver inbound-endpoint create \
  --resource-group rg-network \
  --dns-resolver-name dnsr-hub \
  --name inbound01 \
  --location eastus \
  --subnet /subscriptions/<subId>/resourceGroups/rg-network/providers/Microsoft.Network/virtualNetworks/vnet-hub/subnets/snet-dns-inbound

# Create an outbound endpoint
az network dns-resolver outbound-endpoint create \
  --resource-group rg-network \
  --dns-resolver-name dnsr-hub \
  --name outbound01 \
  --location eastus \
  --subnet /subscriptions/<subId>/resourceGroups/rg-network/providers/Microsoft.Network/virtualNetworks/vnet-hub/subnets/snet-dns-outbound

# Create a forwarding ruleset
az network dns-resolver forwarding-ruleset create \
  --resource-group rg-network \
  --dns-resolver-name dnsr-hub \
  --name frs-hub \
  --location eastus

# Link the ruleset to the outbound endpoint
az network dns-resolver forwarding-ruleset vnet-link create \
  --resource-group rg-network \
  --dns-resolver-name dnsr-hub \
  --forwarding-ruleset-name frs-hub \
  --name link-spoke1 \
  --virtual-network /subscriptions/<subId>/resourceGroups/rg-network/providers/Microsoft.Network/virtualNetworks/vnet-spoke1

# Add a forwarding rule
az network dns-resolver forwarding-rule create \
  --resource-group rg-network \
  --dns-resolver-name dnsr-hub \
  --forwarding-ruleset-name frs-hub \
  --name rule-onprem \
  --domain-name corp.contoso.com. \
  --target-dns-servers 10.10.0.4 10.10.0.5
```

Use Azure DNS Private Resolver when:

- You need managed DNS forwarding in hybrid environments.
- You want to avoid running custom DNS VMs.
- You need clear inbound and outbound DNS paths for private endpoints and internal zones.

Use custom DNS servers when:

- You already operate enterprise DNS infrastructure.
- You need advanced policies or integrations not covered by the managed resolver.
- You want a single authoritative DNS layer for many internal zones.

**Private DNS zone linking and record management**

Private DNS zones must be linked to the VNets that need resolution. A zone link does not move traffic; it only enables name resolution from that VNet.

Typical steps:

1. Create the private DNS zone.
2. Link the zone to the VNet.
3. Create or auto-create the A record for the private endpoint.
4. Verify client resolution from the target subnet.

Example:

```bash
az network private-dns zone create \
  --resource-group rg-network \
  --name privatelink.blob.core.windows.net

az network private-dns link vnet create \
  --resource-group rg-network \
  --zone-name privatelink.blob.core.windows.net \
  --name link-spoke1 \
  --virtual-network /subscriptions/<subId>/resourceGroups/rg-network/providers/Microsoft.Network/virtualNetworks/vnet-spoke1 \
  --registration-enabled false
```

If you manage records manually, ensure the record points to the private endpoint IP. In many cases, Azure can create the record automatically when you associate the private DNS zone group with the endpoint.

**Private DNS zone groups**

A private DNS zone group associates one private endpoint with one or more private DNS zones. This is the mechanism that keeps DNS aligned with the endpoint.

Conceptually:

- The private endpoint is the network object.
- The zone group is the DNS association.
- The private DNS zone contains the record.

This is especially important when a service uses multiple subresources or when you need more than one DNS zone for the same endpoint.

**Subnet requirements and IP planning**

Private endpoints consume IP addresses from the subnet where they are deployed. This is easy to overlook in small subnets or dense hub designs.

Plan for:

- Enough free IPs for current and future private endpoints.
- Separate subnets for different workloads if you need cleaner governance.
- Avoiding address exhaustion in shared hub subnets.

Practical guidance:

- Use a dedicated subnet for private endpoints when possible.
- Reserve capacity for growth, especially in landing zones with many PaaS services.
- Check subnet size before large-scale rollout.

**Network security and traffic behavior**

Private endpoint traffic behaves differently from VM-to-VM traffic, so you need to understand where security controls apply.

Key points:

- **Consumer subnet controls** still matter. NSGs and route tables on the client side can affect whether the client reaches the endpoint.
- **Private endpoint subnet policies** are handled differently from normal VM subnets.
- **Traffic to the private endpoint** is not protected by a VM NIC-style NSG on the endpoint itself in the same way as a regular workload NIC.
- **Service-side controls** such as firewall rules and public network access settings still matter for the target service.

For exam answers, avoid oversimplifying this as “NSGs do not matter.” Instead, think in terms of where the traffic originates, where the endpoint lives, and which subnet policies are supported.

**Routing and forced tunneling**

Private endpoint traffic should usually stay on the private network path. However, user-defined routes (UDRs) and forced tunneling can create unexpected behavior if they send traffic to an appliance or default route that does not preserve the private path.

Troubleshooting checklist:

- Verify the client subnet route table.
- Confirm the effective route to the private endpoint IP.
- Ensure the next hop does not break local VNet delivery.
- Check whether a firewall or NVA is intercepting traffic unnecessarily.

Example validation commands:

```bash
az network nic show-effective-route-table \
  --resource-group rg-network \
  --name nic-app01

az network nic show-effective-nsg \
  --resource-group rg-network \
  --name nic-app01
```

If traffic to the private endpoint fails, compare:

- DNS resolution result
- Effective routes
- NSG rules on the client subnet
- Service-side access settings

**Public network access versus firewall rules**

Many Azure PaaS services let you choose between disabling public access entirely or keeping public access enabled with firewall restrictions.

Use **disable public network access** when:

- You want the strongest private-only posture.
- You are confident all consumers can use private endpoints.
- You want to reduce the risk of accidental public exposure.

Use **public access with firewall rules** when:

- You need a transition period during migration.
- Some trusted public or partner access must remain.
- You are not ready to move all consumers to private connectivity.

Exam nuance:

- Disabling public access does not replace DNS or private endpoint configuration.
- Firewall rules are not a substitute for private access if the design requires private-only connectivity.
- Some services have service-specific behavior, so always verify the exact resource type.

**Cross-subscription and cross-tenant scenarios**

Private access patterns often cross administrative boundaries.

Common enterprise cases:

- A shared network subscription hosts the private endpoint.
- A platform subscription hosts the PaaS service.
- A separate application subscription consumes the service.
- A partner tenant consumes a service published through Private Link Service.

Design considerations:

- Approval workflow may require coordination between owners.
- DNS zone ownership may differ from endpoint ownership.
- RBAC (role-based access control) must allow endpoint creation, zone linking, and service approval.
- Billing and governance may be split across subscriptions.

Cross-tenant access is especially useful for partner or merger scenarios, but it requires careful identity, approval, and DNS planning.

**Service-specific private access patterns**

Some Azure services have special considerations beyond the basic private endpoint model.

Examples:

- **Azure Storage**: separate subresources such as blob, file, queue, and table may need distinct private endpoint planning.
- **Azure SQL**: private endpoint plus DNS zone alignment is critical for client libraries.
- **Key Vault**: private access is common for secrets and certificates, and public access is often disabled after validation.
- **Azure Monitor / Log Analytics**: private access may involve multiple related endpoints and careful DNS planning.
- **Azure Synapse**: workspace connectivity can involve multiple endpoints and managed private links.
- **Azure OpenAI**: private access patterns depend on the service and region availability, so verify current service support before designing.

When a service has multiple control-plane or data-plane endpoints, map each one explicitly. Do not assume one private endpoint covers every function.

**Practical design examples**

A common hub-and-spoke pattern:

- Hub VNet hosts DNS infrastructure and shared services.
- Spoke VNets host applications.
- Private endpoints live in a dedicated subnet in the spoke or hub, depending on governance.
- Private DNS zones are linked to all VNets that need resolution.
- On-premises DNS forwards private zone queries to Azure DNS Private Resolver.

A private service publishing pattern:

- Internal API runs behind Standard Load Balancer.
- Private Link Service exposes the API privately.
- Consumer VNets create private endpoints.
- DNS maps the service name to the consumer-side private IP.

A migration pattern:

1. Deploy private endpoint.
2. Link private DNS zone.
3. Validate application connectivity.
4. Restrict public access with firewall rules.
5. Disable public network access after cutover.

**CLI examples for private endpoint workflows**

Create a private endpoint for a storage account:

```bash
az network private-endpoint create \
  --resource-group rg-app \
  --name pe-storage01 \
  --vnet-name vnet-app \
  --subnet snet-private-endpoints \
  --private-connection-resource-id /subscriptions/<subId>/resourceGroups/rg-data/providers/Microsoft.Storage/storageAccounts/stacct01 \
  --group-id blob \
  --connection-name pe-storage01-conn
```

Associate a private DNS zone group:

```bash
az network private-endpoint dns-zone-group create \
  --resource-group rg-app \
  --endpoint-name pe-storage01 \
  --name default \
  --private-dns-zone privatelink.blob.core.windows.net \
  --zone-name blob-zone
```

> CLI parameters can vary by Azure CLI version and resource type. Always verify the current command help before using it in production or exam labs.

Validate the private endpoint connection:

```bash
az network private-endpoint-connection list \
  --resource-group rg-data \
  --name stacct01 \
  --type Microsoft.Storage/storageAccounts
```

**Troubleshooting checklist**

When private access fails, check in this order:

1. Confirm the private endpoint connection is approved.
2. Confirm the client resolves the private FQDN to the private IP.
3. Confirm the private DNS zone is linked to the correct VNet.
4. Confirm the client subnet route table allows local delivery.
5. Confirm NSGs and firewalls do not block the client path.
6. Confirm the service has public access disabled only after private access is validated.
7. Confirm the correct subresource or zone is used for the service.

Common symptoms:

- **Name resolves to public IP**: DNS zone link or forwarding is missing.
- **Connection times out**: route, NSG, or firewall issue.
- **Pending approval**: service owner has not accepted the connection.
- **Works in one VNet but not another**: private DNS zone is not linked everywhere it needs to be.

**Exam decision guidance**

Choose **Private Endpoint** when you need private consumption of a managed Azure service.

Choose **Private Link Service** when you need to publish your own service privately.

Choose **Azure DNS Private Resolver** when you need managed hybrid DNS forwarding and resolution.

Choose **custom DNS** when enterprise DNS policy or existing infrastructure requires it.

Choose **public access with firewall rules** only when private-only access is not yet required or not yet feasible.

The best exam answer usually combines the right network path, the right DNS design, and the right access control model. Private access is not just a subnet feature; it is a full design pattern that spans identity, DNS, routing, and service configuration.
