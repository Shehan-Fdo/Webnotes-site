## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Identity-centric security overview**

Identity-centric security means you design access control around who or what is requesting access, then layer policy, privilege, and governance around that identity. In Azure, this usually centers on Microsoft Entra ID, Azure RBAC (Role-Based Access Control), Conditional Access, Privileged Identity Management, and governance features such as access reviews and entitlement management.

The architectural goal is to reduce implicit trust. Instead of assuming that a user, device, network, or workload is safe because it is inside a perimeter, you continuously evaluate identity signals and grant only the minimum access needed for the shortest practical time.

- **Identity** becomes the primary security boundary for users, apps, and workloads.
- **Least privilege** limits blast radius when credentials or sessions are compromised.
- **Continuous evaluation** helps you adapt access decisions to risk, device state, and context.

**Azure tenant, subscription, and governance boundaries**

A Microsoft Entra tenant is the identity boundary for users, groups, applications, and policies. Azure subscriptions are the billing and resource deployment boundaries. Management groups sit above subscriptions and help you apply governance at scale. These boundaries matter because identity design and resource governance are related but not identical.

Use this mental model:

- **Tenant**: identity plane for authentication, authorization, and directory objects.
- **Subscription**: resource plane for Azure services and billing.
- **Management group**: policy and RBAC inheritance across subscriptions.
- **Resource group**: lifecycle boundary for related resources.

A common architecture pattern is to centralize identity in one tenant, then organize subscriptions by environment, business unit, or workload. Management groups let you apply Azure Policy and RBAC consistently across those subscriptions.

```text
Tenant
└── Management group
    ├── Subscription - Prod
    │   └── Resource groups
    └── Subscription - NonProd
        └── Resource groups
```

When you design governance, remember that identity boundaries and resource boundaries do not always align perfectly. For example, a single tenant can manage multiple subscriptions, but cross-tenant collaboration requires explicit guest access, cross-tenant access settings, or application federation.

**Microsoft Entra ID core concepts**

Microsoft Entra ID is the cloud identity provider for Azure and many Microsoft 365 and SaaS scenarios. It authenticates users, issues tokens, and stores directory objects such as users, groups, service principals, and managed identities.

Key directory objects:

- **User**: human identity, usually synced from on-premises or cloud-only.
- **Group**: used for access assignment, policy targeting, and app assignment.
- **Application registration**: the app definition in the home tenant.
- **Service principal**: the app instance in a tenant, used for access and consent.
- **Managed identity**: an Azure-managed service principal for Azure resources.
- **Device**: used in Conditional Access and compliance decisions.

A useful distinction for exam scenarios:

- **Application registration** defines the app’s identity, credentials, redirect URIs, and permissions.
- **Enterprise application** represents the service principal and its tenant-specific configuration, such as assignment and consent.

This distinction matters when you troubleshoot authentication or authorization. If an app is registered in one tenant but used in another, the consuming tenant typically sees an enterprise application object, not the original registration.

**Authentication, authorization, and modern protocols**

Authentication answers “who are you?” Authorization answers “what can you do?” In modern Azure designs, you should prefer modern authentication protocols such as OAuth 2.0, OpenID Connect (OIDC), and SAML for federation scenarios.

- **OAuth 2.0**: delegated and application permissions for API access.
- **OpenID Connect**: authentication layer on top of OAuth 2.0 for sign-in.
- **SAML**: common for enterprise single sign-on with older SaaS integrations.
- **Legacy authentication**: older protocols such as basic auth that do not support modern controls well.

Modern authentication matters because Conditional Access, MFA (Multi-Factor Authentication), token protection features, and risk-based controls work best with modern protocols. Legacy authentication is harder to secure and should be blocked wherever possible.

> Block legacy authentication unless you have a documented exception and a migration plan. Legacy protocols often bypass stronger controls and increase account compromise risk.

A practical design approach is:

- Use OIDC/OAuth 2.0 for custom apps and APIs.
- Use SAML for SaaS apps that require it.
- Disable legacy authentication at the tenant level when possible.
- Monitor sign-ins for protocol usage before enforcing a block.

**Application registration vs enterprise application**

This distinction is frequently tested because it affects app design, consent, and troubleshooting.

| Concept | Purpose | Typical location | Common use |
|---|---|---|---|
| Application registration | Defines the app identity and configuration | Home tenant | App development, credentials, redirect URIs, API permissions |
| Enterprise application | Tenant-specific service principal | Consuming tenant | Assignment, SSO configuration, consent, access control |

Practical implications:

- You register an app when you build or own it.
- You configure the enterprise application when you want to control how users in a tenant access it.
- You often troubleshoot sign-in issues in the enterprise application blade, not only in the registration.

Example app registration command:

```bash
az ad app create \
  --display-name "contoso-portal" \
  --sign-in-audience AzureADMyOrg
```

Example service principal creation for app access:

```bash
az ad sp create --id <appId>
```

**Permissions, scopes, and consent**

Azure and Entra app authorization relies on permissions and consent. For APIs, permissions are typically expressed as delegated permissions or application permissions.

- **Delegated permissions**: the app acts on behalf of a signed-in user.
- **Application permissions**: the app acts as itself without a user present.
- **Scopes**: delegated permissions requested by a client app.
- **App roles**: role-based assignments exposed by an app or API.

Consent determines whether an app can use those permissions.

- **User consent** allows users to approve low-risk permissions for apps, if enabled.
- **Admin consent** is required for high-privilege permissions or when user consent is disabled.
- **Tenant-wide consent policies** help reduce shadow IT and risky app approvals.

For architecture, prefer tight consent governance:

- Restrict user consent or use consent settings that limit risky permissions.
- Require admin consent for sensitive APIs.
- Review high-privilege app permissions regularly.
- Use publisher verification and app governance where available.

Example Microsoft Graph permission grant pattern:

```http
POST https://graph.microsoft.com/v1.0/servicePrincipals/{id}/appRoleAssignedTo
Content-Type: application/json

{
  "principalId": "11111111-1111-1111-1111-111111111111",
  "resourceId": "22222222-2222-2222-2222-222222222222",
  "appRoleId": "33333333-3333-3333-3333-333333333333"
}
```

**Conditional Access design**

Conditional Access is the policy engine that evaluates signals before granting access. It is not a firewall replacement; it is an identity-driven decision layer that can require stronger authentication, block access, or enforce session controls based on conditions.

Common signals include:

- User or group membership
- Application being accessed
- Device compliance or hybrid join state
- Location or named network
- Client app type
- Sign-in risk
- User risk
- Platform
- Authentication strength

A typical policy structure is:

1. Select users or workload identities.
2. Select cloud apps or actions.
3. Define conditions.
4. Choose grant or session controls.
5. Exclude break-glass accounts and test carefully.

Example policy intent:

- Require MFA for all users accessing Azure management.
- Block legacy authentication.
- Require compliant devices for sensitive SaaS apps.
- Require phishing-resistant authentication for privileged roles.

Example conceptual policy using Microsoft Graph PowerShell:

```powershell
Connect-MgGraph -Scopes "Policy.ReadWrite.ConditionalAccess"

$policy = @{
  displayName = "Require MFA for Azure portal"
  state       = "enabled"
  conditions  = @{
    users = @{
      includeUsers = @("All")
      excludeUsers = @("breakglass1@contoso.com","breakglass2@contoso.com")
    }
    applications = @{
      includeApplications = @("797f4846-ba00-4fd7-ba43-dac1f8f63013") # Azure portal
    }
  }
  grantControls = @{
    operator = "OR"
    builtInControls = @("mfa")
  }
}

New-MgIdentityConditionalAccessPolicy -BodyParameter $policy
```

Session controls are useful, but they are not universal across all apps. For example, sign-in frequency can force reauthentication, and app-enforced restrictions apply only in supported scenarios. Use them where the app and protocol support them.

**Conditional Access vs RBAC vs PIM**

These controls solve different problems:

- **Conditional Access** decides whether a sign-in or token issuance should be allowed.
- **Azure RBAC** decides what a principal can do on Azure resources after access is granted.
- **Privileged Identity Management (PIM)** controls just-in-time elevation for privileged roles.

A good exam rule:

- Use **Conditional Access** for access conditions and authentication requirements.
- Use **RBAC** for resource authorization.
- Use **PIM** for time-bound privileged role activation.

Scenario examples:

- A user must use MFA to sign in to the Azure portal: use Conditional Access.
- A user can read storage accounts but not modify them: use RBAC.
- A user can activate the Owner role for 1 hour after approval: use PIM.

**Privileged Identity Management**

PIM reduces standing privilege by making privileged access eligible rather than permanently active. This is one of the most important identity-centric controls for Azure architecture.

PIM supports:

- Eligible assignments for roles
- Just-in-time activation
- Approval workflows
- Time-bound access
- MFA on activation
- Justification and ticketing requirements
- Access reviews for privileged roles

A strong privileged access model usually includes:

- Separate admin accounts for privileged work.
- Eligible rather than permanent role assignments.
- MFA and authentication strength requirements for activation.
- Approval for high-impact roles such as Global Administrator or Owner.
- Alerts for unusual activations.

Example PIM design:

- Global Administrator: 2 eligible accounts, no permanent assignment.
- Subscription Owner: eligible for platform engineers, 1-hour activation.
- Security Reader: permanent assignment for SOC analysts if needed.

Example Azure CLI role assignment for a privileged role should use object IDs for determinism:

```bash
az role assignment create \
  --assignee-object-id 11111111-1111-1111-1111-111111111111 \
  --assignee-principal-type User \
  --role "Reader" \
  --scope /subscriptions/22222222-2222-2222-2222-222222222222
```

> For reproducible automation, prefer `--assignee-object-id` over resolving a UPN at runtime.

**Authentication strengths and phishing-resistant access**

Authentication strengths let you require specific authentication methods rather than generic MFA. This is useful when you want stronger assurance for sensitive actions or privileged roles.

Common use cases:

- Require phishing-resistant methods for administrators.
- Require passwordless sign-in for high-risk apps.
- Allow only FIDO2 security keys or certificate-based authentication for specific groups.

Use authentication strengths inside Conditional Access, not as a standalone control. For example, you can require a phishing-resistant strength for privileged role activation or for access to the Azure portal.

Example policy intent:

- Admins accessing Azure management must use phishing-resistant MFA.
- Standard users can use regular MFA for low-risk apps.
- Break-glass accounts are excluded from Conditional Access but monitored heavily.

**Identity Protection**

Microsoft Entra ID Protection uses risk signals to detect potentially compromised identities and risky sign-ins. It is especially useful for automated response to suspicious activity.

Two important policy types:

- **User risk policy**: responds when a user account is judged risky overall.
- **Sign-in risk policy**: responds to a specific sign-in attempt that appears risky.

Operational distinction:

- User risk is about the account state and may require password reset or stronger remediation.
- Sign-in risk is about the current authentication attempt and often triggers MFA or block decisions.

Typical response patterns:

- Low sign-in risk: require MFA.
- Medium sign-in risk: require MFA or block depending on sensitivity.
- High user risk: force password change and investigate.

Example architecture guidance:

- Use sign-in risk policies to stop suspicious access in real time.
- Use user risk policies to remediate compromised accounts.
- Combine with Conditional Access for app-specific enforcement.

**Break-glass account design**

Break-glass accounts are emergency access accounts used when normal identity controls fail. They are not a substitute for good administration; they are a resilience measure.

Recommended design:

- Create at least two cloud-only emergency accounts.
- Exclude them from Conditional Access policies.
- Use very strong, unique credentials stored securely.
- Do not assign them daily-use licenses unless required.
- Monitor them continuously with alerts and sign-in logs.
- Test them on a scheduled basis.

Operational guidance:

- Review break-glass account usage monthly or after any incident.
- Store credentials in a secure vault with restricted access.
- Alert on any sign-in, password change, or role assignment.
- Keep the accounts out of normal admin workflows.

> If a break-glass account is used, treat it as a security event and investigate immediately.

**Privileged access strategy**

A mature privileged access strategy combines multiple controls rather than relying on one feature.

Recommended layers:

- Separate admin and user accounts.
- PIM for just-in-time elevation.
- Conditional Access for MFA, device, and location requirements.
- RBAC for resource permissions.
- Access reviews for periodic validation.
- Logging and alerting for all privileged actions.

Scenario guidance:

- Use **PIM** when you want to reduce standing privilege.
- Use **Conditional Access** when you want to enforce stronger sign-in requirements.
- Use **RBAC** when you want to scope resource actions.
- Use **access reviews** when you want to validate whether access is still needed.

**Access reviews and entitlement management**

Access reviews help you periodically validate whether users, guests, and privileged role holders still need access. Entitlement management helps you package access into access packages with approval, expiration, and lifecycle workflows.

Use access reviews for:

- Guest users
- Group memberships
- Application assignments
- Privileged role assignments

Use entitlement management for:

- Self-service access requests
- Approval workflows
- Time-bound access packages
- External collaboration governance

Example scenario:

- A contractor needs access to a project app and SharePoint site for 30 days.
- You create an access package with group membership, app assignment, and expiration.
- An approver validates the request.
- Access is automatically removed at expiration.

This is often better than manually assigning groups because it creates an auditable lifecycle.

**Managed identities**

Managed identities provide secretless authentication for Azure resources. Azure creates and manages the identity, so you do not store client secrets in code or configuration.

Types:

- **System-assigned managed identity**: tied to one Azure resource and deleted with it.
- **User-assigned managed identity**: standalone identity that can be attached to multiple resources.

Use managed identities for:

- Azure Functions accessing Key Vault
- App Service accessing Storage
- Virtual machines calling Microsoft Graph or Azure services
- Automation accounts and container workloads

Example enabling a system-assigned identity:

```bash
az webapp identity assign \
  --resource-group rg-app \
  --name contoso-webapp
```

Example granting Key Vault access with Azure RBAC:

```bash
az role assignment create \
  --assignee-object-id <managed-identity-object-id> \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<subId>/resourceGroups/rg-sec/providers/Microsoft.KeyVault/vaults/kv-contoso
```

**Key Vault permission models**

Azure Key Vault supports two main data-plane authorization models:

- **Access policy model**: legacy, vault-specific permissions.
- **Azure RBAC model**: uses Azure role assignments for data-plane access.

Important nuance:

- Azure RBAC controls data-plane access when that model is enabled.
- Some Key Vault operations still require management-plane permissions, such as creating or configuring the vault resource itself.
- Do not assume the data-plane model covers every administrative action.

Choose Azure RBAC when you want centralized governance and consistent role management. Use access policies only when you need compatibility with older designs or specific operational constraints.

Example RBAC roles:

- Key Vault Secrets User
- Key Vault Secrets Officer
- Key Vault Crypto User
- Key Vault Administrator

Example access policy command:

```bash
az keyvault set-policy \
  --name kv-contoso \
  --object-id <object-id> \
  --secret-permissions get list
```

**Workload identity federation**

Workload identity federation is a secretless pattern for external workloads such as GitHub Actions, Kubernetes, or other OIDC-capable systems. Instead of storing a client secret, you trust an external identity provider token and exchange it for an Entra token.

Use this when:

- You want to avoid long-lived secrets.
- You run CI/CD in GitHub Actions or another OIDC provider.
- You deploy to Azure from Kubernetes or other federated environments.

Benefits:

- No secret rotation burden
- Reduced credential leakage risk
- Better alignment with zero trust principles

Example GitHub Actions concept:

- GitHub issues an OIDC token for the workflow.
- Entra validates the federated credential.
- The workflow gets Azure access without a stored secret.

Example federated credential creation concept:

```bash
az ad app federated-credential create \
  --id <app-object-id> \
  --parameters '{
    "name": "github-main",
    "issuer": "https://token.actions.githubusercontent.com",
    "subject": "repo:contoso/app:ref:refs/heads/main",
    "audiences": ["api://AzureADTokenExchange"]
  }'
```

**Hybrid identity integration**

Hybrid identity connects on-premises directories to Entra ID so users can access cloud resources with a consistent identity. The main options are Entra Connect and cloud sync.

- **Microsoft Entra Connect**: traditional synchronization and federation toolset.
- **Cloud sync**: lighter-weight synchronization option for many scenarios.
- **Password hash synchronization**: syncs password hashes for cloud authentication.
- **Pass-through authentication**: validates passwords against on-premises agents.
- **Federation**: redirects authentication to an on-premises identity provider.

Design considerations:

- Use synchronization to create a common identity for hybrid users.
- Decide whether you need password hash sync, pass-through auth, or federation based on resilience and operational complexity.
- Plan for device identity and compliance if you want Conditional Access to evaluate hybrid devices.

A common architecture choice is password hash synchronization with Seamless SSO for simplicity and resilience. Federation is usually reserved for specific requirements.

**Device identity and compliance**

Device state is often part of access decisions. Entra can evaluate whether a device is compliant, hybrid joined, or managed by Intune.

Use device-based controls when:

- You need to restrict access to managed endpoints.
- You want to reduce risk from unmanaged or jailbroken devices.
- You need to enforce encryption, OS version, or compliance posture.

Typical policy examples:

- Require compliant device for finance apps.
- Allow browser access from unmanaged devices but block downloads.
- Require hybrid joined devices for internal admin portals.

**Azure RBAC and scope design**

Azure RBAC controls access to Azure resources at different scopes:

- Management group
- Subscription
- Resource group
- Resource

Role assignments inherit downward unless overridden by scope design. This makes scope selection critical.

Common built-in roles:

- Owner
- Contributor
- Reader
- User Access Administrator
- Security Reader
- Security Admin
- Key Vault roles
- Storage roles

Design guidance:

- Assign roles to groups, not individual users, when possible.
- Use the narrowest scope that satisfies the requirement.
- Avoid broad roles like Owner unless absolutely necessary.
- Separate platform administration from workload administration.

Example role assignment at resource group scope:

```bash
az role assignment create \
  --assignee-object-id <group-object-id> \
  --role "Contributor" \
  --scope /subscriptions/<subId>/resourceGroups/rg-app
```

**Governance controls and lifecycle management**

Identity-centric security is not only about sign-in. It also includes lifecycle governance for users, guests, apps, and privileged access.

Use these controls together:

- **Access reviews** to validate ongoing need.
- **Entitlement management** to package and expire access.
- **PIM** to reduce standing privilege.
- **Azure Policy** to enforce resource governance.
- **RBAC** to authorize resource actions.
- **Conditional Access** to enforce sign-in requirements.

A practical lifecycle pattern:

1. A user requests access through an access package.
2. An approver validates the request.
3. The user receives group membership and app access.
4. Conditional Access enforces MFA and device requirements.
5. PIM controls any privileged elevation.
6. Access review later confirms whether access should continue.

**Scenario-based decision guide**

Use the following patterns to choose the right control:

- **Need to block risky sign-ins**: Conditional Access with Identity Protection.
- **Need to limit Azure resource actions**: Azure RBAC.
- **Need to reduce standing admin privilege**: PIM.
- **Need to validate continued access**: Access reviews.
- **Need to manage external collaboration**: Entitlement management and guest governance.
- **Need secretless app-to-Azure auth**: Managed identity or workload identity federation.
- **Need to secure admin sign-in**: Conditional Access with authentication strengths and privileged role policies.

Examples:

- A developer needs temporary Contributor access to a subscription: use PIM with eligible assignment.
- A contractor needs access to one app for 30 days: use entitlement management.
- A service running in Azure needs to read secrets: use managed identity.
- A GitHub Actions pipeline needs to deploy to Azure: use workload identity federation.
- A user signs in from a risky location: use Conditional Access plus Identity Protection.

**Practical architecture checklist**

- Centralize identity in Microsoft Entra ID.
- Use modern authentication protocols and block legacy auth.
- Separate admin and user accounts.
- Require MFA for all users and stronger authentication for privileged roles.
- Use PIM for privileged access.
- Use RBAC with group-based assignments and least privilege.
- Use managed identities or workload identity federation instead of secrets.
- Govern guest and app access with reviews and entitlement management.
- Design break-glass accounts, monitoring, and testing before an incident occurs.
- Integrate hybrid identity carefully and choose sync or federation based on business needs.

**Common exam pitfalls**

- Confusing application registration with enterprise application.
- Treating Conditional Access as a replacement for RBAC.
- Assuming PIM alone protects all privileged scenarios without role design and CA.
- Forgetting that Key Vault has separate management-plane and data-plane considerations.
- Using user consent too broadly for enterprise apps.
- Overlooking workload identity federation when discussing secretless authentication.
- Ignoring tenant, subscription, and management group boundaries in governance design.

**Quick reference**

- **Entra ID**: identity provider and directory.
- **Conditional Access**: sign-in policy engine.
- **RBAC**: resource authorization.
- **PIM**: just-in-time privileged access.
- **Identity Protection**: risk-based detection and response.
- **Managed identity**: Azure-managed workload identity.
- **Workload identity federation**: secretless external workload authentication.
- **Access reviews**: periodic access validation.
- **Entitlement management**: governed access packages.
- **Key Vault RBAC**: data-plane authorization through Azure roles.
