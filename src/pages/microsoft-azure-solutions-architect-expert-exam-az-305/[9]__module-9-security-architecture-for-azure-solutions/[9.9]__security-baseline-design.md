## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Security baseline design overview**

A security baseline defines the minimum set of controls you apply consistently across Azure tenants, management groups, subscriptions, and workloads. In AZ-305, you are expected to design a baseline that is secure by default, scalable, and enforceable through governance rather than manual review.

A strong baseline usually combines:
- **Identity controls** for authentication, authorization, and privileged access.
- **Network controls** for segmentation, ingress/egress filtering, and private access.
- **Data controls** for encryption, key management, and secret protection.
- **Monitoring controls** for detection, retention, and incident response.
- **Recovery controls** for backup, immutability, and disaster recovery.

The key design principle is consistency. You do not want each team inventing its own security posture. Instead, you define guardrails at the management group and subscription layers, then allow workload teams to add stricter controls where needed.

**Azure landing zone governance pattern**

A landing zone security baseline starts with hierarchy design. You typically use management groups to separate platform services from workload subscriptions so that inherited controls match the risk profile of each layer.

A common structure is:
- **Tenant root group** for global governance and policy inheritance.
- **Platform management groups** for shared services such as identity, connectivity, and management.
- **Landing zone management groups** for application subscriptions.
- **Sandbox or innovation groups** for lower-trust experimentation.

This separation matters because platform subscriptions often host shared security services, while workload subscriptions host application data and compute. If you place both in the same management group, you make it harder to apply different policy sets, RBAC boundaries, and logging requirements.

Typical placement strategy:
- Put **identity-adjacent services** and shared networking in platform subscriptions.
- Put **application workloads** in separate subscriptions by environment or business unit.
- Use **management groups** to inherit policy, not to host workloads directly.
- Reserve a **break-glass** or emergency access model at the tenant level, not inside a workload subscription.

A practical hierarchy example:

```text
Tenant Root
├─ Platform
│  ├─ Identity
│  ├─ Connectivity
│  └─ Management
├─ LandingZones
│  ├─ Prod
│  │  ├─ App1-Subscription
│  │  └─ App2-Subscription
│  └─ NonProd
│     ├─ App1-Dev
│     └─ App2-Test
└─ Sandbox
```

This design lets you apply stricter policies to production while keeping nonproduction flexible enough for delivery teams.

**Microsoft Entra ID tenant baseline**

The identity plane is the first place to establish a baseline because most Azure compromise paths begin with identity abuse. Microsoft Entra ID should be configured with strong authentication, limited privilege, and explicit governance for external access.

Core tenant baseline controls include:
- **Conditional Access** for enforcing MFA, device compliance, location rules, and session controls.
- **Privileged Identity Management (PIM)** for just-in-time elevation of admin roles.
- **Break-glass accounts** for emergency access when normal controls fail.
- **Tenant restrictions** to limit which external tenants users can access from managed devices or networks.
- **Admin consent workflow** to prevent uncontrolled application permissions.
- **External collaboration settings** to control guest access and cross-tenant trust.
- **Authentication methods policy** to restrict weak methods and prefer phishing-resistant options where appropriate.

Break-glass accounts deserve special handling:
- Keep them highly protected, monitored, and excluded from normal Conditional Access policies that could lock you out.
- Use long, unique passwords and store them securely.
- Limit their use to emergency scenarios only.
- Alert on any sign-in or role activation involving these accounts.

> Do not design break-glass accounts as ordinary privileged users with normal MFA enforcement. Their purpose is emergency recovery when the primary identity controls are unavailable.

For enterprise baselines, security defaults are usually not the design target because they are not customizable enough for mature governance. In designed enterprise environments, Conditional Access is the preferred control plane.

A practical Conditional Access baseline often includes:
- Require MFA for all users except emergency accounts.
- Require stronger controls for privileged roles.
- Block legacy authentication.
- Require compliant or hybrid-joined devices for sensitive apps.
- Use sign-in risk and user risk policies where licensed and appropriate.

Example policy intent:
- Users: all users
- Cloud apps: all apps
- Grant: require MFA
- Exclude: break-glass accounts
- Session: sign-in frequency for sensitive apps

**Identity governance and external access**

A secure baseline also controls how identities are created and trusted. External collaboration is useful, but it must be bounded.

Design considerations:
- Allow guest access only where business need exists.
- Restrict guest permissions to the minimum required.
- Review guest accounts and app assignments regularly.
- Use cross-tenant access settings to define inbound and outbound trust.
- Require approval for admin consent to enterprise applications.

Tenant restrictions are especially useful in managed environments because they reduce the chance that users authenticate to unauthorized tenants from corporate devices or networks.

**Network security baseline**

Network baseline design is about reducing exposure, segmenting trust zones, and forcing sensitive traffic through controlled paths. In Azure, you usually combine subnet design, network security groups, application-layer protection, and private connectivity.

Key components:
- **Network Security Groups (NSGs)** filter traffic at subnet or NIC level using 5-tuple rules.
- **Application Security Groups (ASGs)** simplify NSG rules by grouping VMs by function rather than IP address.
- **Azure DDoS Protection** helps protect public endpoints from volumetric attacks.
- **Application Gateway with Web Application Firewall (WAF)** protects HTTP/S workloads at layer 7.
- **Azure Firewall** centralizes network and application filtering for egress and east-west control.
- **Private Endpoints** expose PaaS services privately through Azure Private Link.
- **User Defined Routes (UDRs)** force traffic through inspection or hub networking.

Use NSGs when:
- You need subnet or NIC-level allow/deny control.
- You want simple segmentation between tiers.
- You need to block lateral movement inside a virtual network.

Use ASGs when:
- You want to avoid hardcoding IP addresses in NSG rules.
- You have dynamic VM scale sets or changing IPs.
- You want rules based on workload role, such as `web`, `app`, and `db`.

Use WAF when:
- You expose web applications to the internet.
- You need protection against common web attacks such as SQL injection and cross-site scripting.
- You want centralized HTTP/S inspection and policy tuning.

Use DDoS Protection when:
- You host public IPs that must remain available under attack.
- You need enhanced telemetry and mitigation for critical internet-facing services.

A common secure reference pattern is:
- Internet traffic enters through **Application Gateway WAF** or a front door service.
- Traffic is forwarded to private application subnets.
- Backend services are reachable only through **Private Endpoints** or internal load balancers.
- Egress is controlled through **Azure Firewall**.
- NSGs enforce subnet-level segmentation.

Example NSG rule intent:
- Allow `443` from WAF subnet to app subnet.
- Allow app subnet to database subnet on `1433` or `3306` only if required.
- Deny all other inbound traffic.
- Restrict outbound internet access from workload subnets.

Example CLI for an NSG rule:

```bash
az network nsg rule create \
  --resource-group rg-app-prod \
  --nsg-name nsg-app \
  --name Allow-HTTPS-From-WAF \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes 10.10.1.0/24 \
  --source-port-ranges '*' \
  --destination-address-prefixes 10.10.2.0/24 \
  --destination-port-ranges 443
```

**Hub-and-spoke and segmentation**

For enterprise baselines, hub-and-spoke remains a common pattern because it centralizes shared services and inspection.

Typical hub responsibilities:
- Shared DNS
- Azure Firewall
- Bastion or secure admin access
- DDoS-enabled public ingress
- Private DNS zones
- Network monitoring

Typical spoke responsibilities:
- Application subnets
- Data subnets
- Private endpoints
- Workload-specific NSGs and route tables

This pattern supports:
- Centralized inspection and logging.
- Reduced duplication of security services.
- Clear separation between platform and workload ownership.

Be careful not to over-centralize. If every packet hairpins through the hub without design discipline, you can create latency and operational bottlenecks. Use the hub for shared controls, not as a universal choke point for every scenario.

**Azure Key Vault baseline**

Key Vault is a core security control because it stores secrets, keys, and certificates. A secure baseline must protect both the vault and the lifecycle of the objects inside it.

Critical baseline settings:
- Enable **soft delete** so deleted objects can be recovered during accidental or malicious deletion.
- Enable **purge protection** so deleted items cannot be permanently removed before retention expires.
- Prefer **Azure RBAC** for authorization when you want centralized role-based control and consistent governance.
- Use **access policies** only when you need the legacy model or specific operational constraints.
- Restrict network access with **private endpoints** and disable public network access where feasible.
- Log access and administrative operations to a central monitoring workspace.

RBAC vs access policies:
- **Azure RBAC** is better for enterprise governance and role inheritance.
- **Access policies** are simpler in small deployments but can become harder to manage at scale.
- Do not mix models casually without a clear operating standard.

Lifecycle management matters:
- Rotate secrets and keys on a defined schedule.
- Use versioned secrets and certificates.
- Remove unused secrets and expired certificates.
- Monitor for near-expiry events.
- Use managed identities instead of storing credentials whenever possible.

Example Key Vault hardening commands:

```bash
az keyvault create \
  --name kv-prod-core \
  --resource-group rg-security-prod \
  --location eastus \
  --enable-rbac-authorization true \
  --enable-purge-protection true \
  --enable-soft-delete true
```

Private endpoint example:

```bash
az network private-endpoint create \
  --name pe-kv-prod-core \
  --resource-group rg-security-prod \
  --vnet-name vnet-hub \
  --subnet snet-private-endpoints \
  --private-connection-resource-id /subscriptions/<subId>/resourceGroups/rg-security-prod/providers/Microsoft.KeyVault/vaults/kv-prod-core \
  --group-id vault \
  --connection-name kv-prod-core-conn
```

**Data protection baseline**

Data protection starts with encryption, but a mature baseline also covers key ownership, access boundaries, and data classification.

Baseline design choices:
- Use platform-managed encryption by default for most services.
- Use customer-managed keys (CMK) when regulatory, contractual, or risk requirements demand customer control.
- Store CMKs in Key Vault or Managed HSM when stronger key isolation is required.
- Classify data by sensitivity and apply controls accordingly.
- Restrict export, sharing, and public exposure of sensitive data.

Common patterns:
- Storage accounts with secure transfer required and private access.
- SQL databases with transparent data encryption and private endpoints.
- Backup vaults protected by RBAC and immutability.
- Managed identities for applications accessing data services.

For highly sensitive workloads, consider:
- Double encryption where supported and justified.
- Separate subscriptions or resource groups for regulated data.
- Dedicated key management and stricter logging.

**Monitoring, logging, and tamper resistance**

A security baseline is incomplete without detection and evidence. Monitoring should be centralized, retained long enough for investigation, and protected from tampering.

Design goals:
- Centralize logs in a dedicated Log Analytics workspace or SIEM platform.
- Separate operational logs from workload owners when needed.
- Retain logs according to compliance and incident response requirements.
- Alert on identity, network, and data-plane anomalies.
- Protect the logging pipeline itself from deletion or unauthorized changes.

Important log sources:
- Entra ID sign-in and audit logs
- Azure Activity Log
- NSG flow logs where used
- Firewall logs
- Key Vault diagnostic logs
- PaaS service diagnostics
- Defender for Cloud alerts

Retention strategy:
- Keep short-term hot data for rapid investigation.
- Archive longer-term logs for compliance and forensics.
- Use immutable or restricted storage for critical audit records where supported.

Secure the logging pipeline by:
- Restricting who can delete workspaces and diagnostic settings.
- Using separate subscriptions for security operations where appropriate.
- Applying locks or policy controls to critical logging resources.
- Monitoring changes to diagnostic settings and alert rules.

Example diagnostic setting pattern:

```bash
az monitor diagnostic-settings create \
  --name send-to-law \
  --resource /subscriptions/<subId>/resourceGroups/rg-app-prod/providers/Microsoft.KeyVault/vaults/kv-prod-core \
  --workspace /subscriptions/<subId>/resourceGroups/rg-monitoring/providers/Microsoft.OperationalInsights/workspaces/law-secops \
  --logs '[{"category":"AuditEvent","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

**Defender for Cloud and posture management**

Microsoft Defender for Cloud helps you assess baseline drift and identify missing controls. It is useful for continuous posture management, but it does not replace architecture decisions.

Use it to:
- Assess secure score and recommendations.
- Detect misconfigurations in compute, storage, databases, and containers.
- Enable relevant plans for the services you actually use.
- Prioritize remediation based on exposure and risk.

Be careful with plan naming and service coverage because Defender offerings are service-specific and can change over time. In exam scenarios, focus on the design intent:
- Enable the plan for the workload type.
- Use recommendations to drive remediation.
- Integrate alerts with your SOC or incident response process.

**Recovery and resilience baseline**

Security baseline design includes recovery because availability and recoverability are part of security. Ransomware, deletion, and corruption are security events as much as operational failures.

Core recovery controls:
- **Azure Backup** for supported IaaS and some PaaS workloads.
- **Immutable backup** or soft-delete features where available.
- **Cross-region disaster recovery** for critical services.
- **Recovery Services vault** hardening with RBAC and network restrictions.
- **Backup vault** or service-specific backup controls for newer workloads.

Design principles:
- Back up critical data separately from the primary subscription when possible.
- Protect backup administrators with PIM and least privilege.
- Test restore procedures regularly.
- Use geo-redundant or zone-redundant options where business continuity requires them.
- Keep recovery credentials and keys available during a regional outage.

Service examples:
- **Azure VMs**: Azure Backup with vault soft delete and immutability features where supported.
- **Azure SQL**: automated backups plus long-term retention and geo-replication for critical databases.
- **Storage**: versioning, soft delete, change feed, and replication strategy.
- **AKS**: backup of cluster state and persistent volumes using supported tooling and separate identity controls.

A secure recovery pattern often includes:
- Primary region workload
- Secondary region standby or active-active deployment
- Backup vault in a separate subscription
- Restore testing in an isolated subscription or resource group

**Container and AKS baseline**

Containers and Kubernetes require their own baseline because cluster compromise can expose many workloads at once. The goal is to reduce cluster privilege, control image provenance, and limit network movement.

AKS baseline controls:
- Use **managed identities** instead of service principals where possible.
- Enable **Azure AD integration** for cluster access control.
- Restrict cluster admin access and use RBAC.
- Use **private clusters** when public API exposure is unnecessary.
- Apply **network policies** to control pod-to-pod traffic.
- Use **pod security** controls and admission policies to prevent privileged workloads.
- Scan images for vulnerabilities before deployment.
- Pull images from trusted registries such as Azure Container Registry with content trust or scanning controls where available.

Workload design considerations:
- Separate system and user node pools.
- Use taints and tolerations to isolate sensitive workloads.
- Restrict outbound internet access from pods.
- Mount secrets from Key Vault through managed identity rather than embedding them in manifests.
- Rotate cluster credentials and review role assignments regularly.

Example AKS hardening intent:
- Private API server
- Azure AD RBAC
- Azure Policy for Kubernetes
- Network policy enabled
- Managed identity for node and workload access
- Defender for Containers enabled where used

Example deployment snippet using managed identity and Key Vault access pattern:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app
spec:
  serviceAccountName: workload-identity-sa
  containers:
    - name: app
      image: myregistry.azurecr.io/app:1.0.0
      env:
        - name: KEY_VAULT_NAME
          value: kv-prod-core
```

**Governance with Azure Policy**

Azure Policy is the main enforcement mechanism for a baseline because it can deny, audit, or deploy settings at scale.

Use policy to:
- Deny public IP creation where not allowed.
- Require private endpoints for sensitive PaaS services.
- Enforce diagnostic settings.
- Require tags for ownership and data classification.
- Restrict allowed regions.
- Require secure transfer and encryption settings.
- Audit or deny weak configurations such as open storage accounts.

Policy design tips:
- Assign policies at the highest practical management group scope.
- Use initiatives to bundle related controls.
- Separate audit-only policies from deny policies during rollout.
- Test in nonproduction before enforcing in production.
- Use exemptions sparingly and with expiry.

Example policy assignment pattern:

```bash
az policy assignment create \
  --name deny-public-ip \
  --scope /providers/Microsoft.Management/managementGroups/mg-landingzones \
  --policy /providers/Microsoft.Authorization/policyDefinitions/<policy-definition-id>
```

If you use an initiative, ensure the policy set definition exists and use the correct full resource ID or name for the target scope.

**Scenario-based design choices**

A good exam answer shows how you combine controls rather than naming them in isolation.

Scenario 1: Internet-facing web app
- Place the app behind Application Gateway WAF.
- Use private backend subnets.
- Restrict database access with NSGs and private endpoints.
- Store secrets in Key Vault with purge protection and private access.
- Send logs to a central workspace.
- Enable DDoS Protection if the public endpoint is business critical.

Scenario 2: Regulated internal line-of-business app
- Use a dedicated subscription under a workload management group.
- Require compliant devices and MFA through Conditional Access.
- Use private endpoints for data services.
- Disable public network access where possible.
- Enforce Azure Policy for diagnostics and encryption.
- Protect backups with immutability and separate administrative roles.

Scenario 3: Shared platform services
- Put shared networking, monitoring, and identity-adjacent services in platform subscriptions.
- Apply stricter RBAC and PIM to platform operators.
- Centralize firewall, DNS, and logging.
- Keep workload teams out of platform subscriptions except through controlled roles.

Scenario 4: Kubernetes-based microservices
- Use a private AKS cluster.
- Enforce Azure AD RBAC and managed identity.
- Apply network policies between namespaces.
- Scan images before deployment.
- Store secrets in Key Vault and inject them at runtime.
- Monitor cluster audit logs and container security alerts centrally.

**Operational checklist**

1. Define the management group hierarchy and subscription placement model.
2. Establish Entra ID controls, including Conditional Access, PIM, and break-glass accounts.
3. Design network segmentation with NSGs, ASGs, WAF, firewall, and private endpoints.
4. Harden Key Vault with soft delete, purge protection, RBAC, and private access.
5. Enforce data protection with encryption, CMK where needed, and managed identities.
6. Centralize logs, alerts, and retention in a secure monitoring architecture.
7. Enable posture management and policy-based enforcement.
8. Design backup and disaster recovery with immutability and restore testing.
9. Apply container and AKS controls for identity, network, and image security.
10. Review and test the baseline regularly for drift and exceptions.

**Exam focus reminders**

- **Management groups** are for inheritance and governance, not workload hosting.
- **Conditional Access** is the enterprise baseline control; security defaults are not the usual design choice for mature tenants.
- **Break-glass accounts** are emergency access accounts and should not be trapped by normal access controls.
- **Key Vault** should usually have soft delete, purge protection, RBAC, and private access.
- **WAF** protects web apps; **NSGs** and **ASGs** handle network segmentation; **DDoS Protection** addresses volumetric attack resilience.
- **Monitoring** must include retention and protection of the logging pipeline itself.
- **Recovery** is part of security because it protects against destructive attacks and accidental deletion.
- **AKS** needs identity, network, and supply-chain controls, not just cluster creation.
