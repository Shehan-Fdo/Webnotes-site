## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Segmentation fundamentals**

Network segmentation divides a larger network into smaller trust zones so you can reduce blast radius, apply different controls, and simplify policy enforcement. In Azure, segmentation is usually implemented with a combination of virtual networks, subnets, network security groups (NSGs), route tables, Azure Firewall, private endpoints, and identity-aware controls.

Micro-segmentation is a finer-grained version of segmentation. Instead of treating an entire subnet as one trust zone, you restrict traffic between specific workloads, pods, or application tiers. The goal is to allow only the minimum required flows and to make lateral movement harder if one workload is compromised.

- **Segmentation** is the broad separation of environments, tiers, or business units.
- **Micro-segmentation** is workload-level restriction inside a segment.
- **Defense in depth** matters because no single control fully isolates traffic.
- **Least privilege** applies to network paths, not just identities and permissions.

A practical Azure design often combines:
- Separate subscriptions or management groups for governance boundaries.
- Separate virtual networks for major environments such as prod, non-prod, and shared services.
- Separate subnets for tiers such as web, app, data, management, and private endpoints.
- NSGs for packet filtering.
- Azure Firewall or third-party firewalls for centralized inspection and egress control.
- Private Link for private access to PaaS services.
- Application-layer controls such as authentication, authorization, and mTLS.

**Why segmentation matters in Azure**

Segmentation helps you control east-west traffic inside the cloud and north-south traffic entering or leaving it. It is especially important when:
- You host multiple applications with different trust levels.
- You need to isolate regulated workloads.
- You want to reduce the impact of credential theft or malware.
- You must meet compliance requirements for separation of duties or data boundaries.
- You need predictable routing and inspection for shared services.

Azure is a software-defined environment, so segmentation is not only about IP ranges. You also design around identity, routing, service exposure, and policy enforcement points.

**Core Azure segmentation building blocks**

**Virtual networks and subnets**

A virtual network (VNet) is the primary isolation boundary in Azure networking. Subnets divide the VNet into smaller address spaces and are the usual place to attach NSGs, route tables, NAT Gateway, service endpoints, and private endpoint policies.

Use subnets to separate:
- Application tiers
- Shared services
- Private endpoints
- Azure Bastion
- Azure Firewall
- VPN or ExpressRoute gateways

> Some Azure platform services require dedicated subnets. For example, gateways, Azure Firewall, and Bastion each have specific subnet requirements. Private endpoints are commonly placed in a dedicated subnet to simplify policy and DNS design.

A common address plan looks like this:

```text
10.10.0.0/16  VNet
├─ 10.10.1.0/24  web-subnet
├─ 10.10.2.0/24  app-subnet
├─ 10.10.3.0/24  data-subnet
├─ 10.10.4.0/24  shared-services-subnet
├─ 10.10.5.0/24  AzureFirewallSubnet
├─ 10.10.6.0/24  AzureBastionSubnet
└─ 10.10.7.0/24  private-endpoints-subnet
```

This layout makes policy easier to reason about and reduces accidental overlap between platform and workload traffic.

**Network Security Groups**

An NSG is a stateful packet filter that allows or denies traffic based on source, destination, protocol, and port. You can associate an NSG with a subnet, a NIC, or both.

Important behavior:
- NSGs are evaluated on the resource where they are attached.
- Subnet-level NSGs filter traffic before it reaches the NIC.
- NIC-level NSGs add another filter at the workload interface.
- If both exist, traffic must be allowed by both.
- NSGs are stateful, so return traffic for an allowed flow is automatically permitted.

NSG rule evaluation follows priority order:
- Lower priority numbers are processed first.
- The first matching rule wins.
- Default rules exist and cannot be removed.
- If no custom rule matches, the default deny rules apply.

Common default rules include:
- Allow VNet inbound and outbound
- Allow Azure Load Balancer inbound
- Deny all inbound
- Deny all outbound

A simple inbound rule example for app-tier access:

```bash
az network nsg rule create \
  --resource-group rg-network \
  --nsg-name nsg-app \
  --name allow-web-to-app-443 \
  --priority 200 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes 10.10.1.0/24 \
  --source-port-ranges '*' \
  --destination-address-prefixes 10.10.2.0/24 \
  --destination-port-ranges 443
```

Use this pattern when the source and destination prefixes match the actual traffic path between the web and app tiers. In practice, the rule is enforced where the NSG is associated, not as a global policy across the whole network.

**Application Security Groups**

Application Security Groups (ASGs) let you group NICs by application role instead of by IP address. They are useful when workloads scale dynamically and IP-based rules become hard to maintain.

Typical use:
- `asg-web`
- `asg-app`
- `asg-db`

Example:

```bash
az network nsg rule create \
  --resource-group rg-network \
  --nsg-name nsg-app \
  --name allow-web-to-app \
  --priority 210 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-asgs asg-web \
  --destination-asgs asg-app \
  --destination-port-ranges 443
```

ASGs are useful for role-based segmentation, but they are not a replacement for subnet design. They apply to NIC-level membership and work with NSGs; they do not segment arbitrary Azure resources.

**Route tables and traffic steering**

User-defined routes (UDRs) let you override system routes and steer traffic through a firewall, virtual appliance, or hub. This is essential when you want east-west traffic inspection or forced egress control.

Example default route to a firewall:

```bash
az network route-table route create \
  --resource-group rg-network \
  --route-table-name rt-spoke \
  --name default-to-firewall \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.10.5.4
```

Use UDRs carefully:
- A default route is not the whole routing story.
- System routes, peering behavior, service endpoints, and private endpoints also affect effective paths.
- You must validate effective routes on the NIC to confirm the actual path.

**Azure Firewall and centralized inspection**

Azure Firewall provides centralized filtering and logging for traffic that you intentionally route through it. It is commonly used in hub-and-spoke designs for:
- Egress control
- Inbound DNAT
- Network and application rule filtering
- Threat intelligence-based filtering
- TLS inspection in Azure Firewall Premium, where supported and trusted

Azure Firewall does not inspect east-west traffic automatically. You must design the routing so that traffic flows through the firewall, usually with UDRs or hub transit patterns.

Typical uses:
- Spoke-to-internet egress
- Spoke-to-spoke inspection through a hub
- Shared services access control
- Controlled access to PaaS endpoints via private connectivity

Azure Firewall Premium can inspect encrypted traffic with TLS inspection, but only in supported scenarios where certificate trust and policy design allow it. Do not assume every encrypted flow can be decrypted.

**Private Link and private endpoints**

Private Link exposes a service over a private IP in your VNet through a private endpoint. This is a major segmentation tool because it removes public exposure for supported PaaS services.

Benefits:
- Traffic stays on private IP space
- You can disable public network access on many services
- You can place access behind NSGs, routing, and DNS controls
- You reduce exposure to internet-originated attacks

Private DNS is usually required so clients resolve the service name to the private endpoint IP. The exact zone depends on the service.

Examples:
- Azure SQL Database: `privatelink.database.windows.net`
- Azure Key Vault: `privatelink.vaultcore.azure.net`
- Azure Storage: `privatelink.blob.core.windows.net` and related subresource zones
- Azure App Service: service-specific private DNS zones depending on the app type

A private endpoint example:

```bash
az network private-endpoint create \
  --name pe-sql-prod \
  --resource-group rg-app \
  --vnet-name vnet-prod \
  --subnet private-endpoints-subnet \
  --private-connection-resource-id /subscriptions/<subId>/resourceGroups/rg-data/providers/Microsoft.Sql/servers/sqlprod01 \
  --group-id sqlServer \
  --connection-name pe-sql-prod-conn
```

Private Link is not the same as a service endpoint:
- Private Link gives a private IP and stronger isolation.
- Service endpoints extend VNet identity to the service side but do not create a private IP.
- Service endpoints are simpler, but they are less isolated than Private Link.

**Service endpoints**

Service endpoints let Azure PaaS services recognize traffic as coming from a specific VNet or subnet context. The service still uses its public endpoint, but access can be restricted to selected VNets/subnets on the service side.

Use service endpoints when:
- You want simpler configuration than Private Link.
- The service supports service endpoints and you do not need private IP exposure.
- You are comfortable with public endpoint semantics plus VNet-based access control.

Use Private Link when:
- You want private IP-based access.
- You want to minimize public exposure.
- You need stronger segmentation for sensitive services.

**Micro-segmentation patterns**

Micro-segmentation is about reducing trust between workloads that share infrastructure. In Azure, you usually implement it with a combination of NSGs, ASGs, private endpoints, identity, and application controls.

Common patterns:
- Web tier can reach app tier on only required ports.
- App tier can reach database tier only on database ports.
- Management access is restricted to jump hosts or Bastion.
- Shared services are isolated from workload subnets.
- Private endpoints are reachable only from approved subnets.

Example policy intent:
- Web subnet: allow inbound 443 from the internet through a load balancer or application gateway.
- App subnet: allow inbound 443 only from the web subnet.
- Data subnet: allow inbound 1433 only from the app subnet.
- Management subnet: allow RDP or SSH only from Bastion or a secure admin network.

A layered design often looks like this:
- Edge protection with DDoS Protection and WAF.
- Perimeter inspection with Azure Firewall.
- Subnet and NIC filtering with NSGs.
- Workload grouping with ASGs.
- Service isolation with Private Link.
- Identity-based access with Entra ID and managed identities.

**Application-layer segmentation**

Network controls are necessary, but they are not sufficient. Application-layer segmentation reduces the impact of a compromised network path.

Useful controls:
- **Authentication and authorization** between services
- **Managed identities** for Azure-to-Azure service access
- **mTLS** for service-to-service encryption and mutual trust
- **API gateways** to centralize policy, throttling, and validation
- **Zero Trust** principles that verify explicitly and assume breach

Examples:
- A front-end calls an API through Azure API Management instead of directly reaching backend services.
- Services authenticate with managed identities instead of shared secrets.
- Internal APIs require JWT validation and role-based authorization.
- Service mesh or sidecar-based mTLS can protect pod-to-pod traffic in Kubernetes.

These controls complement network segmentation because a network allow rule does not prove the caller is trusted.

**Kubernetes and AKS segmentation**

Azure Kubernetes Service (AKS) adds another segmentation layer. You can segment by namespace, node pool, network policy, and service exposure.

Key ideas:
- Separate workloads into namespaces by application or environment.
- Use dedicated node pools for sensitive workloads.
- Apply Kubernetes NetworkPolicy to restrict pod-to-pod traffic.
- Expose services through internal load balancers or ingress controllers.
- Use private clusters when you want to reduce control-plane exposure.

Network model matters:
- **Azure CNI** assigns pod IPs from the VNet, which simplifies routing and integration but consumes VNet address space.
- **Overlay networking** reduces VNet IP consumption by using an overlay for pod addresses, but you must understand how routing and policy enforcement behave in your chosen CNI implementation.

Practical considerations:
- NetworkPolicy behavior depends on the CNI plugin and its enforcement capabilities.
- A podSelector-only rule may not be enough if you need cross-namespace access.
- Namespace scoping, default deny policies, and ingress/egress rules must be designed together.

Example NetworkPolicy:

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-app-from-web
  namespace: app
spec:
  podSelector:
    matchLabels:
      app: api
  policyTypes:
  - Ingress
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          name: web
      podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080
```

This example allows traffic from labeled pods in the `web` namespace to labeled pods in the `app` namespace on port `8080`.

**Hub-and-spoke, virtual WAN, and segmentation at scale**

For larger Azure estates, topology choice affects how segmentation is enforced.

**Hub-and-spoke**
- Best when you want centralized inspection and shared services.
- The hub hosts firewalls, gateways, DNS, and Bastion.
- Spokes host application workloads.
- UDRs and peering control traffic paths.

Use hub-and-spoke when:
- You need strong central governance.
- You want a clear inspection point.
- Your environment is regional or moderately scaled.

**Virtual WAN**
- Best when you need large-scale branch, remote user, and global connectivity.
- It simplifies connectivity and routing across many sites and regions.
- It can integrate security and routing services in a managed framework.

Use virtual WAN when:
- You have many branches or remote sites.
- You need global transit and simplified operations.
- You want a managed connectivity backbone rather than hand-built peering.

A common exam distinction:
- Hub-and-spoke gives you more explicit control.
- Virtual WAN gives you more managed scale and operational simplicity.

**Subnet design best practices**

Good subnet design makes segmentation easier to operate and audit.

Recommended practices:
- Use dedicated subnets for platform services such as gateways, firewalls, and Bastion.
- Separate workload tiers into different subnets.
- Reserve a subnet for private endpoints when possible.
- Avoid mixing unrelated trust levels in the same subnet.
- Plan address space for growth and peering.
- Keep subnet sizes large enough for scaling and platform requirements.

Why this matters:
- NSGs are easier to reason about when each subnet has one purpose.
- Route tables are simpler when traffic patterns are predictable.
- Private endpoint DNS and policy are easier to manage.
- Platform services often have subnet-specific constraints.

**DDoS Protection and perimeter resilience**

Azure DDoS Protection is an adjacent control that protects public-facing entry points from volumetric and protocol attacks. It does not segment internal traffic, but it is relevant because segmentation often creates a smaller set of exposed ingress points that need stronger protection.

Use DDoS Protection when:
- You expose public IPs for load balancers, gateways, or other services.
- You want telemetry and mitigation for attack traffic.
- You need protection for critical internet-facing workloads.

DDoS Protection complements segmentation by protecting the edges of your trust zones.

**Monitoring and validation**

Segmentation is only useful if you can verify it. Azure provides telemetry to confirm that traffic is being allowed, denied, or routed as intended.

Useful tools and logs:
- **NSG flow logs** for allowed and denied traffic patterns
- **Azure Firewall logs** for network, application, and threat intelligence events
- **Azure Monitor** and Log Analytics for centralized analysis
- **Network Watcher** for effective routes, connection troubleshooting, and packet capture
- **NSG effective security rules** to confirm applied policy
- **Connection troubleshoot** to validate reachability

Example validation commands:

```bash
az network nic show-effective-route-table \
  --resource-group rg-app \
  --nic-name nic-app01

az network nic list-effective-nsg \
  --resource-group rg-app \
  --nic-name nic-app01
```

Use logs to answer questions such as:
- Which flows are being denied?
- Is traffic bypassing the firewall?
- Are private endpoints resolving correctly?
- Is a subnet-level NSG blocking traffic before the NIC-level NSG?
- Are unexpected east-west connections occurring?

**Design tradeoffs and exam guidance**

When choosing a segmentation approach, balance security, operability, and cost.

Choose simpler controls when:
- The workload is low risk.
- The environment is small.
- Operational overhead must stay low.

Choose stronger segmentation when:
- You host sensitive data.
- You need strict tenant or workload isolation.
- You must reduce lateral movement risk.
- You need auditable enforcement points.

Tradeoffs to remember:
- NSGs are lightweight and fast, but they are not deep inspection.
- Azure Firewall centralizes control, but it adds cost and routing complexity.
- Private Link improves isolation, but it requires DNS and service-specific setup.
- ASGs simplify rule management, but they do not replace subnet architecture.
- Kubernetes NetworkPolicy helps inside clusters, but it depends on the CNI and policy engine.

**Common scenario patterns**

**Web-app-data tiering**
- Web subnet receives internet traffic through WAF or load balancer.
- App subnet accepts only web-tier traffic.
- Data subnet accepts only app-tier traffic.
- Private endpoints are used for PaaS databases and storage.

**Shared services hub**
- DNS, Bastion, firewall, and gateway live in the hub.
- Spokes host applications.
- UDRs force inspection through the hub.
- Private DNS zones are linked to spokes as needed.

**Regulated workload isolation**
- Dedicated subscription and VNet.
- Separate subnets for app, data, and private endpoints.
- Private Link for PaaS.
- Azure Firewall for egress control.
- DDoS Protection for public ingress.
- Logging and alerting for all deny events.

**Kubernetes micro-segmentation**
- Separate namespaces per app or environment.
- Default deny policies.
- NetworkPolicy for pod-to-pod restrictions.
- Internal ingress only.
- Managed identities and mTLS for service trust.

**Quick comparison**

| Control | Primary purpose | Strengths | Limitations |
|---|---|---|---|
| NSG | Packet filtering | Simple, fast, subnet/NIC enforcement | No deep inspection |
| ASG | Role-based grouping | Easier rule management | NIC-level scope only |
| UDR | Traffic steering | Forces inspection paths | Requires careful route design |
| Azure Firewall | Centralized filtering | Logging, inspection, egress control | Cost and routing complexity |
| Private Link | Private service access | Strong isolation, private IP | DNS and service setup required |
| Service endpoint | VNet-aware service access | Simple for supported services | No private IP, less isolation |
| NetworkPolicy | Pod-level control | Fine-grained Kubernetes segmentation | Depends on CNI and policy engine |

**Practical checklist**

1. Define trust zones by environment, tier, and sensitivity.
2. Allocate address space with room for growth.
3. Place platform services in dedicated subnets.
4. Apply NSGs at subnet and NIC boundaries where appropriate.
5. Use ASGs to reduce IP-based rule sprawl.
6. Route sensitive traffic through inspection points when needed.
7. Prefer Private Link for sensitive PaaS access.
8. Add application-layer authentication and authorization.
9. Use Kubernetes NetworkPolicy for pod-level restrictions.
10. Enable logs and validate effective routes and rules.

**Key exam takeaways**

- Segmentation is about reducing trust and blast radius, not just splitting IP ranges.
- Micro-segmentation adds workload-level controls inside a broader network design.
- NSGs enforce stateful packet rules at subnet and NIC scope.
- ASGs simplify NSG management for workload roles.
- Azure Firewall provides centralized inspection only when traffic is routed through it.
- Private Link is stronger than service endpoints for isolation.
- Kubernetes segmentation depends on namespaces, policies, and the CNI implementation.
- DDoS Protection and logging are important supporting controls for segmented architectures.
