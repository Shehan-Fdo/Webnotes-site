## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Azure key management overview**

Azure key management centers on protecting cryptographic material and application secrets while keeping operational access auditable and tightly controlled. In practice, you usually combine **Azure Key Vault** for general-purpose secret, key, and certificate storage with **Azure Managed HSM** for higher-assurance, hardware-backed key protection and stricter administrative separation.

A good architecture answers four questions:

- Where do you store secrets, keys, and certificates?
- How do applications authenticate without embedding credentials?
- How do you recover from accidental deletion or rotation mistakes?
- How do you monitor access, expiration, and suspicious activity?

For AZ-305, focus on the design tradeoffs: security boundary, operational complexity, recovery, and integration with Azure services.

**Core object types in Key Vault**

Azure Key Vault supports three first-class object types:

- **Secrets** store arbitrary sensitive values such as connection strings, API tokens, passwords, and certificates in PFX/PEM form when managed as secrets.
- **Keys** store cryptographic keys used for encryption, signing, and wrapping/unwrapping operations.
- **Certificates** are first-class objects that combine certificate metadata with a key and a secret representation, enabling lifecycle management and integration with certificate authorities.

A certificate object is not just a file upload. Key Vault tracks the certificate policy, issuer, key material, and secret material together so you can automate renewal and retrieval.

Typical use cases:

- Store database passwords as secrets.
- Use keys for envelope encryption or application signing.
- Manage TLS certificates for apps, gateways, and internal services.

**Key Vault versus Managed HSM**

Use **Key Vault** when you need broad Azure integration, secret storage, certificate management, and standard key protection. Use **Managed HSM** when you need dedicated HSM-backed key storage with stronger isolation and compliance-oriented controls.

| Capability | Azure Key Vault | Azure Managed HSM |
|---|---|---|
| Secrets | Yes | No |
| Certificates | Yes | No |
| Keys | Yes | Yes |
| HSM-backed protection | Optional for some key operations | Yes, dedicated HSM cluster |
| Typical use | App secrets, certs, keys | High-assurance cryptographic key custody |
| Azure service integration | Broad | More specialized |

Managed HSM is not a drop-in replacement for Key Vault. It is primarily for key operations, not secret or certificate management. Plan migration carefully if you move from Key Vault keys to Managed HSM, because application code, permissions, and operational workflows may change.

**Authentication and authorization**

Azure supports two main authorization models for Key Vault:

- **Access policies**
- **Azure role-based access control (Azure RBAC)**

Access policies are the older, vault-scoped model. Azure RBAC is the modern model and is generally preferred for new deployments because it aligns with Azure governance, centralizes permissions, and scales better across subscriptions and management groups.

Choose access policies when:

- You are maintaining an existing vault that already uses them.
- You need a simpler, vault-local permission model for a limited scenario.
- You are constrained by legacy application or operational patterns.

Choose Azure RBAC when:

- You want consistent governance across Azure resources.
- You need separation of duties and centralized role assignment.
- You are designing new environments and want a future-friendly model.

Common built-in roles include:

- **Key Vault Secrets User**
- **Key Vault Secrets Officer**
- **Key Vault Crypto User**
- **Key Vault Crypto Officer**
- **Key Vault Administrator**

For managed identities, assign the role to the identity’s principal at the vault scope or a narrower scope if appropriate. Be careful with role assignment syntax and identity type; the exact CLI form can vary by identity and environment.

Example role assignment with Azure CLI:

```bash
az role assignment create \
  --assignee <managed-identity-principal-id> \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg-name>/providers/Microsoft.KeyVault/vaults/<vault-name>
```

If you are using a user-assigned managed identity, confirm the principal ID and the target scope before assigning permissions. For system-assigned identities, the principal exists only after the resource is created and identity is enabled.

**Secretless application patterns**

The best design is often to avoid storing credentials in application code or configuration files at all. Instead, use **managed identities** so the application authenticates to Key Vault without a password or client secret.

Common patterns:

- App Service or Azure Functions uses a managed identity to read secrets from Key Vault.
- Azure Virtual Machines use managed identity plus Key Vault SDKs or cloud-init/bootstrap scripts.
- Azure Kubernetes Service (AKS) workloads use workload identity or managed identity integration to retrieve secrets at runtime.
- Azure services that support native Key Vault references resolve secrets without the app directly calling the vault.

App Service Key Vault references are useful for app settings, but they are not magic by themselves. The app or service must have:

- A managed identity enabled
- Permission to read the secret
- Network access to the vault if private networking is used

Example app setting reference:

```text
@Microsoft.KeyVault(SecretUri=https://myvault.vault.azure.net/secrets/db-password/)
```

Use this pattern when you want App Service to resolve the secret value at runtime. It is especially useful for connection strings and configuration values that should not be stored in plain text.

> If the vault is private-only, make sure the app can reach it through the correct private networking path and DNS resolution. A valid reference still fails if the identity, network, or DNS configuration is incomplete.

**Versioning and application design**

Key Vault objects are versioned. Every update to a secret, key, or certificate creates a new version, and the latest version becomes the default when you reference the object without a version.

Design implications:

- Use the unversioned URI when you want applications to automatically pick up the latest approved value.
- Use a version-specific URI when you need deterministic behavior, controlled rollout, or rollback safety.
- Test rotation carefully so you do not break long-lived processes that cache old values.

Examples:

- Latest version reference:
  - `https://myvault.vault.azure.net/secrets/db-password/`
- Specific version reference:
  - `https://myvault.vault.azure.net/secrets/db-password/3f2c1d...`

Practical guidance:

- Use the latest version for stateless apps that can tolerate rotation.
- Use a pinned version for critical workloads during staged migration.
- Plan for cache invalidation in clients that store secrets in memory.

**Secret rotation and lifecycle management**

Rotation is a core design concern, not an afterthought. The goal is to reduce exposure time if a secret is compromised and to avoid manual updates across many applications.

Common rotation approaches:

- Manual rotation with a controlled deployment window
- Automated rotation using Azure automation, Functions, Logic Apps, or external orchestration
- Native certificate auto-renewal through issuer integration

A practical rotation workflow often looks like this:

1. Create a new secret or certificate version.
2. Update dependent applications or configuration references.
3. Validate the new value in production.
4. Retire the old version after a safe overlap period.

For keys, rotation may require coordination with encryption consumers because some data may have been encrypted with older key versions. Keep old key versions available until all protected data has been rewrapped or re-encrypted.

**Certificates in Key Vault**

Certificates are useful when you need lifecycle management for TLS/SSL or client authentication certificates. Key Vault can integrate with certificate issuers so that renewal can be automated when supported by the issuer and policy.

Important certificate concepts:

- A certificate object includes certificate metadata and is associated with a key and a secret.
- The private key may be exportable or non-exportable depending on how the certificate is created and the policy used.
- Auto-renewal depends on issuer support, policy configuration, and the certificate lifecycle settings.

Typical scenarios:

- TLS certificates for App Service, Application Gateway, or internal services
- Client certificates for mutual TLS or service authentication
- Certificates for signing or secure communication where lifecycle automation matters

Example certificate policy conceptually:

```json
{
  "issuerParameters": {
    "name": "Unknown"
  },
  "keyProperties": {
    "exportable": false,
    "keyType": "RSA",
    "keySize": 2048,
    "reuseKey": false
  },
  "secretProperties": {
    "contentType": "application/x-pkcs12"
  }
}
```

Use non-exportable keys when you want stronger protection and do not need to move the private key outside Key Vault. Use exportable material only when a downstream system requires it and your security model allows it.

**Backup, restore, and recovery**

Key Vault supports backup and restore operations for secrets, keys, and certificates. This is important for operational recovery, migration, and disaster recovery planning.

Use backup and restore when:

- You need to move objects between vaults.
- You want a recovery path beyond soft delete.
- You are preparing for vault replacement or environment migration.

Example CLI patterns:

```bash
az keyvault secret backup --vault-name myvault --name db-password --file db-password.bak
az keyvault secret restore --vault-name myvault --file db-password.bak
```

Similar commands exist for keys and certificates. Treat backups as sensitive artifacts because they contain protected material or recoverable references to it.

**Soft delete and purge protection**

**Soft delete** protects against accidental deletion by retaining deleted vault objects for a retention period. **Purge protection** prevents permanent deletion until the retention period expires.

Why this matters:

- It reduces the risk of accidental or malicious deletion.
- It helps meet recovery and compliance requirements.
- It prevents immediate purge even if an attacker gains elevated access.

Operational guidance:

- Enable soft delete and purge protection for production vaults.
- Treat purge protection as a safeguard for high-value environments.
- Verify retention behavior before relying on deletion as a cleanup mechanism.

> Once purge protection is enabled, you cannot immediately remove protected items. Plan your lifecycle and naming strategy accordingly.

**Networking and private access**

For modern designs, prefer **private endpoints** for Key Vault access. This keeps traffic on the Microsoft backbone and exposes the vault through a private IP in your virtual network.

Key networking considerations:

- Private endpoints are the preferred approach for strong isolation.
- Private DNS is critical so clients resolve the vault name to the private IP.
- Public network access can be disabled when all consumers use private connectivity.
- Service endpoints are generally less preferred for new designs.

A common private endpoint pattern includes:

- A vault with public access disabled
- A private endpoint in a spoke VNet
- A private DNS zone linked to the VNet
- Workloads in the same VNet or peered VNets resolving the vault name privately

Troubleshooting order for private access:

1. Verify DNS resolution for `myvault.vault.azure.net`.
2. Confirm the private DNS zone and VNet link.
3. Check the private endpoint connection state.
4. Confirm identity permissions.
5. Confirm public access settings if the workload is expected to use the public endpoint.

Example DNS check:

```bash
nslookup myvault.vault.azure.net
```

If the name resolves to a public IP when you expect private access, fix DNS before chasing authorization errors.

**Throttling, limits, and retry behavior**

Key Vault enforces service limits and can throttle excessive requests. This matters in high-scale applications, startup storms, and poorly cached secret retrieval patterns.

Common causes of throttling:

- Repeated secret fetches on every request
- Many instances starting simultaneously
- Aggressive polling for key or certificate values
- Lack of client-side caching or backoff

Design guidance:

- Cache secrets in memory for a reasonable period when the security model allows it.
- Avoid calling Key Vault on every transaction.
- Use exponential backoff with jitter for retries.
- Batch or prefetch where appropriate.

Example retry pattern in pseudocode:

```python
for attempt in range(5):
    try:
        value = get_secret_from_key_vault()
        break
    except ThrottlingError:
        sleep(base_delay * (2 ** attempt) + random_jitter())
```

Do not treat Key Vault as a low-latency configuration database. It is a secure control plane service, so design for occasional latency and transient failures.

**Monitoring, alerting, and security posture**

Monitoring should cover both availability and security events. Use Azure Monitor diagnostic settings to send Key Vault logs to Log Analytics, Event Hub, or Storage for analysis and retention.

Track:

- Successful and failed access attempts
- Authorization failures
- Secret, key, and certificate operations
- Expiration events and near-expiry conditions
- Private endpoint and network-related failures

Alert on:

- Repeated failed access from unexpected identities
- Sudden spikes in secret reads
- Certificate expiration windows
- Key rotation failures
- Purge or delete operations in sensitive vaults

Microsoft Defender for Cloud can help surface security recommendations and suspicious activity patterns. Use it as part of a broader security posture strategy, not as a replacement for logging and alerting.

**Operational examples**

Retrieve a secret with Azure CLI:

```bash
az keyvault secret show \
  --vault-name myvault \
  --name db-password \
  --query value -o tsv
```

Set a secret:

```bash
az keyvault secret set \
  --vault-name myvault \
  --name db-password \
  --value 'P@ssw0rd!'
```

Create a key:

```bash
az keyvault key create \
  --vault-name myvault \
  --name app-signing-key \
  --kty RSA \
  --size 2048
```

Create a certificate request conceptually:

```bash
az keyvault certificate create \
  --vault-name myvault \
  --name tls-cert \
  --policy @policy.json
```

These commands are useful for understanding the operational model, but in production you should automate them through infrastructure as code, CI/CD, or controlled administrative workflows.

**Design patterns and exam-ready guidance**

Use these patterns when designing secure Azure solutions:

- Store application secrets in Key Vault, not in code, config files, or pipeline variables.
- Prefer managed identities over client secrets for Azure-to-Azure authentication.
- Use Azure RBAC for new vaults unless a legacy access policy model is required.
- Enable soft delete and purge protection in production.
- Prefer private endpoints and private DNS for restricted access.
- Use versioned references when you need deterministic behavior.
- Use unversioned references when you want controlled automatic rotation.
- Use Managed HSM for high-assurance key custody, not for secrets or certificates.
- Monitor access, expiration, and deletion events continuously.

Common mistakes to avoid:

- Storing secrets in plain text app settings without Key Vault integration.
- Assuming a certificate is just a file and ignoring its lifecycle policy.
- Fetching secrets on every request instead of caching responsibly.
- Forgetting DNS when private endpoints are enabled.
- Treating purge protection as optional in production.
- Using Key Vault as a general-purpose data store.

**Quick comparison of access models**

| Topic | Access policies | Azure RBAC |
|---|---|---|
| Scope | Vault-local | Azure resource hierarchy |
| Governance | More fragmented | Centralized |
| New deployments | Less preferred | Preferred |
| Migration path | Legacy support | Strategic direction |

**Troubleshooting checklist**

When a workload cannot access Key Vault, check the problem in this order:

1. Verify DNS resolution and private endpoint routing if private access is used.
2. Confirm the managed identity is enabled and the correct principal is targeted.
3. Check authorization: RBAC role assignment or access policy.
4. Confirm the secret/key/certificate URI is correct, including versioning if used.
5. Review network restrictions, firewall settings, and public access configuration.
6. Check throttling, transient failures, and client retry logic.
7. Validate that the object still exists and was not soft-deleted or purged.

**Exam focus summary**

For AZ-305, you should be able to choose the right service and design pattern for the workload:

- **Key Vault** for secrets, keys, and certificates with broad Azure integration.
- **Managed HSM** for dedicated HSM-backed key protection.
- **Managed identities** for secretless authentication.
- **Azure RBAC** for modern authorization governance.
- **Private endpoints** for strong network isolation.
- **Soft delete and purge protection** for recovery and resilience.
- **Rotation, versioning, backup, and monitoring** for operational readiness.
