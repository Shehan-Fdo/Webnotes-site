## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Secure administration goals**

Secure administration patterns reduce the attack surface of privileged access by separating admin workflows from day-to-day user activity. In Azure, the main objective is to make administrative access deliberate, traceable, time-bound, and resistant to credential theft.

The most important design principle is to avoid permanent broad privilege. Instead, you combine identity controls, network isolation, privileged access workflows, and hardened management endpoints so that an attacker cannot easily pivot from a compromised workstation into critical resources.

- **Least privilege** limits each admin account to only the permissions required for the task.
- **Separation of duties** keeps routine user access distinct from privileged operations.
- **Just-in-time** access reduces the time window in which elevated permissions exist.
- **Privileged access workstations** isolate sensitive admin activity from general browsing and email.

**Identity-first administration**

Azure administration should start with Microsoft Entra ID, formerly Azure Active Directory (Azure AD), because identity is the control plane for most management actions. You should prefer role-based access control (RBAC) assignments over shared credentials and use group-based role assignment where possible to simplify governance.

A secure pattern is to create dedicated admin accounts for privileged work and keep them separate from normal user accounts. This reduces the chance that phishing, malware, or browser-based attacks on a daily-use account can directly expose high-impact permissions.

- Use **separate admin accounts** for privileged tasks.
- Assign roles to **groups**, not individuals, when you need repeatable governance.
- Prefer **built-in roles** before creating custom roles.
- Review **role assignments** regularly to remove stale access.

Common Azure RBAC examples include:
- **Owner**: full access including role assignment management.
- **Contributor**: can manage resources but cannot grant access.
- **Reader**: can view resources without changing them.
- **User Access Administrator**: can manage access assignments.

> Avoid using Owner broadly. It is often more privilege than an operator needs and increases blast radius if the account is compromised.

**Privileged Identity Management**

Microsoft Entra Privileged Identity Management (PIM) is a core secure administration pattern because it makes privileged access eligible rather than permanently active. With PIM, users activate roles only when needed, often with approval, justification, MFA (multifactor authentication), and time limits.

PIM is especially useful for:
- Azure subscription and resource roles
- Microsoft Entra directory roles
- Privileged access to groups
- Time-bound elevation for operations and incident response

A practical workflow is:
1. Make the user **eligible** for a role instead of permanently active.
2. Require **MFA** and justification at activation.
3. Set a short activation duration.
4. Log and review activations for anomalies.

Example Azure CLI pattern for role assignment:
```bash
az role assignment create \
  --assignee user@contoso.com \
  --role "Contributor" \
  --scope /subscriptions/<subscription-id>/resourceGroups/<rg-name>
```

Example of a custom role definition skeleton:
```json
{
  "Name": "App Support Operator",
  "IsCustom": true,
  "Description": "Can manage app services but not access control",
  "Actions": [
    "Microsoft.Web/sites/read",
    "Microsoft.Web/sites/start/action",
    "Microsoft.Web/sites/stop/action",
    "Microsoft.Web/sites/restart/action"
  ],
  "NotActions": [
    "Microsoft.Authorization/*/write"
  ],
  "AssignableScopes": [
    "/subscriptions/<subscription-id>"
  ]
}
```

**Privileged access workstations**

A privileged access workstation (PAW) is a hardened device used only for administrative tasks. The purpose is to reduce exposure to web browsing, email, document macros, and other common malware entry points that can steal tokens or credentials.

A PAW pattern typically includes:
- Dedicated hardware or a dedicated virtual desktop
- No general-purpose browsing or email
- Strong device compliance and endpoint protection
- Conditional Access policies that allow admin sign-in only from trusted devices

You can also use a tiered admin model:
- **Tier 0**: identity systems, directory services, security tooling
- **Tier 1**: servers and platform services
- **Tier 2**: user endpoints and standard apps

This model helps you prevent lower-trust administration from reaching higher-trust systems.

**Just-in-time and just-enough access**

Just-in-time (JIT) access means you grant elevated permissions only for a short period. Just-enough administration means you scope those permissions narrowly to the task. Together, they reduce standing privilege and limit what an attacker can do if access is abused.

Use JIT for:
- Virtual machine administration
- Break-glass support scenarios
- Temporary operational changes
- Emergency troubleshooting

Use just-enough access by:
- Limiting scope to a resource group or single resource
- Using custom roles with only required actions
- Avoiding subscription-wide permissions unless necessary

A common operational pattern is to combine PIM with a custom role and a short activation window. This gives you traceability and minimizes the duration of elevated access.

**Network isolation for administration**

Administrative endpoints should not be exposed broadly to the internet. Secure administration patterns often place management access behind private connectivity, jump hosts, or controlled ingress paths.

Common approaches include:
- **Azure Bastion** for browser-based RDP (Remote Desktop Protocol) and SSH (Secure Shell) without public IP exposure on virtual machines
- **VPN** or ExpressRoute for private access to management networks
- **Jump servers** in a dedicated management subnet
- **Private endpoints** for platform services that support them

Azure Bastion is useful when you want to manage VMs without opening inbound ports like 3389 or 22 to the public internet. It reduces attack surface while preserving operational convenience.

Example network security group (NSG) rule strategy:
- Deny inbound RDP/SSH from Internet
- Allow RDP/SSH only from a management subnet or Bastion
- Restrict admin ports to known source IP ranges when private access is not possible

Example Azure CLI to create a Bastion host is environment-specific, but the design intent is consistent: place Bastion in a dedicated subnet named `AzureBastionSubnet` and keep the target VM without a public IP.

**Management plane protection**

The management plane is the set of APIs and portals used to configure Azure resources. Protecting it is as important as protecting the data plane because an attacker with management access can often change firewall rules, rotate secrets, or disable logging.

Key controls include:
- Enforce **MFA** for all privileged users.
- Use **Conditional Access** to require compliant devices, trusted locations, or approved client apps.
- Restrict administrative access to specific networks.
- Monitor sign-ins and role activations.
- Use **resource locks** where accidental deletion would be costly.

Resource locks are not a substitute for RBAC, but they are useful guardrails:
- **CanNotDelete** prevents deletion while allowing changes.
- **ReadOnly** blocks modifications and deletions.

Example CLI:
```bash
az lock create \
  --name "protect-prod-rg" \
  --lock-type CanNotDelete \
  --resource-group prod-rg
```

**Break-glass accounts**

Break-glass accounts are emergency access accounts used when normal identity controls fail, such as during Conditional Access misconfiguration or identity provider outages. They should be rare, tightly controlled, and heavily monitored.

Good break-glass practices:
- Keep at least two emergency accounts.
- Exclude them from normal Conditional Access policies only when necessary.
- Use strong, unique passwords stored securely.
- Monitor sign-ins and alert on any use.
- Test them periodically to ensure they still work.

> Break-glass accounts are for emergencies only. If they become routine admin accounts, they defeat the purpose of secure administration.

**Logging, monitoring, and auditability**

Secure administration is incomplete without visibility. You need to know who elevated, what they changed, from where they connected, and whether the activity matched expected behavior.

Important telemetry sources:
- **Azure Activity Log** for subscription-level control plane operations
- **Microsoft Entra sign-in logs** for authentication events
- **Audit logs** for directory changes and role assignments
- **PIM audit history** for activations and approvals
- **Defender for Cloud** recommendations and alerts

Useful monitoring patterns:
- Alert on role assignment changes.
- Alert on PIM activations outside business hours.
- Alert on sign-ins from unfamiliar geographies.
- Correlate admin actions with ticket numbers or change records.

Example Kusto Query Language (KQL) for suspicious role assignments:
```kusto
AzureActivity
| where OperationNameValue has "Microsoft.Authorization/roleAssignments/write"
| project TimeGenerated, Caller, ActivityStatusValue, ResourceGroup, SubscriptionId, Properties
| order by TimeGenerated desc
```

**Administrative access design patterns**

A secure Azure administration design usually combines several controls rather than relying on one feature. The right pattern depends on the workload sensitivity, operational model, and compliance requirements.

| Pattern | Best for | Benefit | Tradeoff |
|---|---|---|---|
| Separate admin accounts | General privileged operations | Reduces exposure from daily-use accounts | More accounts to manage |
| PIM eligible roles | Most enterprise admin scenarios | Removes standing privilege | Requires activation workflow |
| PAW | High-trust administration | Strong isolation from malware | Extra device management |
| Bastion or jump host | VM administration | No public management ports | Additional infrastructure |
| Custom RBAC roles | Narrow operational tasks | Least privilege at scale | Requires careful role design |
| Resource locks | Critical resources | Prevents accidental deletion | Can slow emergency changes |

**Practical implementation example**

A secure operations model for a production subscription often looks like this:
- Admins use separate privileged accounts.
- Those accounts are eligible for roles through PIM.
- Admins connect from PAWs only.
- VMs are managed through Bastion or private connectivity.
- Critical resource groups have `CanNotDelete` locks.
- Activity logs and sign-in logs feed a SIEM (Security Information and Event Management) platform.

Example workflow for a production change:
1. You sign in from a PAW using your admin account.
2. You activate the required PIM role with MFA and justification.
3. You connect to the target VM through Bastion.
4. You make the change using the minimum required scope.
5. You verify logs and remove any temporary access if applicable.

Example Azure CLI to inspect role assignments:
```bash
az role assignment list \
  --assignee user@contoso.com \
  --all \
  --output table
```

Example to view activity log events:
```bash
az monitor activity-log list \
  --resource-group prod-rg \
  --offset 7d \
  --output table
```

**Common pitfalls**

- Using shared admin accounts makes attribution difficult and increases compromise risk.
- Granting Owner at subscription scope when a resource-group scope is sufficient expands blast radius.
- Leaving management ports open to the internet exposes VMs to brute force and scanning.
- Treating PIM as optional reduces its value; the pattern works best when activation is required for privileged actions.
- Forgetting to monitor break-glass usage can hide emergency access abuse.
- Relying on network controls alone is insufficient if identity is compromised.

**Exam focus**

For AZ-305, focus on choosing the right secure administration pattern for the scenario:
- Use **PIM** when the requirement is time-bound privileged access.
- Use **PAW** when the scenario emphasizes hardened admin endpoints.
- Use **Bastion** when you need secure VM access without public IPs.
- Use **custom RBAC** when built-in roles are too broad.
- Use **resource locks** when accidental deletion must be prevented.
- Use **Conditional Access** and **MFA** when identity hardening is required.

The best answer is usually the one that reduces standing privilege, narrows scope, and improves auditability while still allowing operations to continue.
