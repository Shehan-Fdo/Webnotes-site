## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Security tradeoffs in distributed architectures**

Distributed systems improve resilience, scale, and user experience, but every added boundary, hop, and integration point expands the attack surface. In Azure, you usually trade between stronger isolation and higher operational complexity, or between simpler operations and broader exposure. Exam questions often test whether you can choose the right control for the right risk, not whether you can maximize every control at once.

A good architecture balances:
- **Identity** over shared secrets where possible
- **Private connectivity** where data sensitivity justifies it
- **Least privilege** without making operations brittle
- **Observability** without over-collecting sensitive data
- **Availability** without weakening security controls

**Core security tradeoff model**

A distributed architecture usually forces you to choose among three goals:
- **Security**
  - Reduce exposure by limiting network paths, credentials, and permissions.
- **Operability**
  - Keep deployments, troubleshooting, and rotation manageable.
- **Cost and latency**
  - Add controls only where the risk justifies the overhead.

A common exam pattern is a workload that needs secure access to PaaS services, cross-subnet communication, or multi-tier application traffic. The best answer is often not “turn on every feature,” but “use the least complex control that meets the requirement.”

**Identity versus secrets**

Using Microsoft Entra ID identities and managed identities reduces secret sprawl, but it changes how applications authenticate and how you troubleshoot access.

- **Managed identity**
  - Removes the need to store credentials in code or configuration.
  - Works well for Azure-to-Azure access such as App Service to Key Vault or Function App to Storage.
  - Can complicate local development, cross-tenant access, and hybrid scenarios.
- **Service principal with secret or certificate**
  - Easier to use in some automation and non-Azure environments.
  - Requires secret lifecycle management, rotation, and secure storage.
  - Increases the chance of credential leakage if stored poorly.
- **Shared access signature (SAS)**
  - Useful for scoped, time-limited access to storage resources.
  - Easier to distribute than full credentials.
  - Still a bearer token, so leakage is a real risk.

A practical rule:
- Prefer **managed identity** for Azure-hosted workloads.
- Use **certificates** over client secrets when you must use an application identity outside managed identity.
- Use **SAS** only for narrowly scoped, time-bound storage access.

Example: assigning a managed identity to an App Service and granting Key Vault access with Azure RBAC:

```bash
az webapp identity assign \
  --resource-group rg-app \
  --name app-prod

principalId=$(az webapp identity show \
  --resource-group rg-app \
  --name app-prod \
  --query principalId -o tsv)

az role assignment create \
  --assignee-object-id "$principalId" \
  --assignee-principal-type ServicePrincipal \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<subId>/resourceGroups/rg-sec/providers/Microsoft.KeyVault/vaults/kv-prod
```

**Network isolation versus accessibility**

Network controls reduce exposure, but they can also make services harder to reach, harder to troubleshoot, and more dependent on DNS and routing correctness.

- **Public endpoint**
  - Simplest to operate and easiest to integrate.
  - Exposes the service to the internet, even if access is restricted by firewall rules.
  - Often acceptable when combined with strong authentication, TLS, and IP restrictions.
- **Service endpoint**
  - Extends a virtual network identity to a PaaS service over the Azure backbone.
  - The service is still reached through its public endpoint, but access is restricted by service-side virtual network rules.
  - Does not provide a private IP or private DNS.
  - Useful when you want simpler setup than private endpoints and can tolerate public endpoint exposure.
- **Private endpoint**
  - Maps a service to a private IP in your virtual network.
  - Stronger isolation because traffic stays on private addressing and name resolution can be private.
  - Requires DNS planning and usually more operational overhead.
  - Best for sensitive workloads or when you want to remove public network exposure.

> Service endpoints are not the same as private endpoints. They improve network access control, but they do not eliminate the public endpoint model.

A practical comparison:

| Control | Exposure model | DNS impact | Operational complexity | Typical use |
|---|---|---:|---:|---|
| Public endpoint + firewall | Public endpoint remains reachable | Minimal | Low | Simple internet-facing or controlled access |
| Service endpoint | Public endpoint with VNet-based allow rules | Minimal | Medium | Azure-only access without private IPs |
| Private endpoint | Private IP in VNet | High | Higher | Sensitive data and strict isolation |

**Private endpoint implementation pattern**

Private endpoints usually require more than just endpoint creation. You also need DNS resolution that points clients to the private IP.

Typical flow:
1. Create the private endpoint.
2. Configure or link the private DNS zone.
3. Ensure the client VNet can resolve the service name to the private IP.
4. Validate routing and firewall rules.

Example pattern for a private endpoint and DNS zone group:

```bash
az network private-endpoint create \
  --name pe-kv-prod \
  --resource-group rg-net \
  --vnet-name vnet-app \
  --subnet snet-private-endpoints \
  --private-connection-resource-id /subscriptions/<subId>/resourceGroups/rg-sec/providers/Microsoft.KeyVault/vaults/kv-prod \
  --group-ids vault \
  --connection-name kv-prod-conn

az network private-dns zone create \
  --resource-group rg-net \
  --name privatelink.vaultcore.azure.net

az network private-dns link vnet create \
  --resource-group rg-net \
  --zone-name privatelink.vaultcore.azure.net \
  --name link-vnet-app \
  --virtual-network vnet-app \
  --registration-enabled false
```

For exam purposes, remember:
- Endpoint creation alone is not enough for functional name resolution.
- DNS is often the hidden failure point.
- Private endpoints are strongest when paired with public network access disabled on the service, if the service supports it.

**Application Gateway, Front Door, and source IP tradeoffs**

Layer 7 (application layer) services improve routing, TLS termination, and web protection, but they can complicate backend source IP expectations.

- **Application Gateway**
  - Often terminates TLS and forwards traffic to backends.
  - Backend systems may see the gateway as the source, not the original client.
  - Client IP preservation depends on headers and architecture, not on a simple subnet rule.
- **Azure Front Door**
  - Acts as a global entry point and reverse proxy.
  - Backends usually see Front Door-related headers and source addresses, not the end user directly.
- **Network Security Group (NSG) rules**
  - Should be based on the actual traffic path and service behavior.
  - Do not assume the client IP is always visible at the backend.

A safer design approach:
- Allow traffic from the gateway or service tag when the platform supports it.
- Validate whether the backend needs the original client IP in headers.
- Use application-layer logging for client identity instead of relying on source IP alone.

Example NSG rule allowing traffic from an Azure service tag rather than a guessed client subnet:

```bash
az network nsg rule create \
  --resource-group rg-net \
  --nsg-name nsg-backend \
  --name AllowAppGateway \
  --priority 200 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes ApplicationGateway \
  --source-port-ranges '*' \
  --destination-address-prefixes '*' \
  --destination-port-ranges 443
```

**Secrets, keys, and certificates**

Secret management is a major tradeoff area because distributed systems often need many credentials, but every stored secret increases risk.

- **Azure Key Vault**
  - Centralizes secrets, keys, and certificates.
  - Supports access control through Azure RBAC or vault access policies, depending on configuration.
  - Reduces secret duplication across applications.
- **Customer-managed keys (CMK)**
  - Give you more control over encryption key lifecycle.
  - Increase operational responsibility for rotation, permissions, and availability.
  - Can create dependency chains if the key vault is unavailable.
- **Platform-managed keys**
  - Simpler and lower operational burden.
  - Less control over key lifecycle and sometimes less compliance flexibility.

Tradeoff summary:
- Use **CMK** when compliance, regulatory control, or explicit key ownership matters.
- Use platform-managed encryption when operational simplicity matters more than key control.
- Keep the key vault highly available and monitor access carefully if workloads depend on it.

Example: retrieving a secret from Key Vault with managed identity from Azure CLI:

```bash
az login
az account set --subscription <subId>

az keyvault secret show \
  --vault-name kv-prod \
  --name DbPassword \
  --query value -o tsv
```

**Encryption at rest and in transit**

Encryption is not optional in distributed architectures, but the tradeoff is usually between stronger guarantees and more configuration.

- **In transit**
  - Use TLS 1.2 or later for service-to-service and client-to-service communication.
  - Prefer end-to-end encryption when intermediaries do not need to inspect payloads.
  - Termination at a gateway can simplify certificate management but shifts trust to the gateway.
- **At rest**
  - Use service encryption by default.
  - Add CMK when you need customer control or compliance evidence.
  - Understand that encryption does not replace access control; it complements it.

Practical guidance:
- Do not rely on encryption alone to protect data from unauthorized access.
- Combine encryption with identity, network restrictions, and logging.
- Rotate certificates and keys on a schedule that matches your risk tolerance and operational capacity.

**Monitoring, logging, and privacy**

Security monitoring improves detection and response, but it can also collect sensitive data and increase storage and retention obligations.

- **Diagnostic logs**
  - Help you investigate access, failures, and suspicious activity.
  - May contain identifiers, request metadata, or payload-adjacent information.
- **Metrics**
  - Useful for alerting and capacity planning.
  - Usually lower risk than verbose logs.
- **Centralized logging**
  - Simplifies correlation across services.
  - Can create a high-value target if log access is too broad.

Use Azure Monitor and Log Analytics with least privilege:
- Restrict who can query logs.
- Set retention based on compliance and incident response needs.
- Avoid logging secrets, tokens, or full personal data unless required.

Example enabling diagnostic settings to a Log Analytics workspace:

```bash
az monitor diagnostic-settings create \
  --name diag-app \
  --resource /subscriptions/<subId>/resourceGroups/rg-app/providers/Microsoft.Web/sites/app-prod \
  --workspace /subscriptions/<subId>/resourceGroups/rg-monitor/providers/Microsoft.OperationalInsights/workspaces/law-prod \
  --logs '[{"category":"AppServiceHTTPLogs","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

The exact log categories vary by resource type, so validate the supported categories before scripting a deployment.

**Perimeter controls versus zero trust**

Traditional perimeter thinking assumes that once traffic is inside the network, it is trusted. Distributed architectures work better with a **zero trust** mindset:
- Verify explicitly.
- Use least privilege.
- Assume breach.

This changes design choices:
- Authenticate every request, not just the first hop.
- Use conditional access and identity-based authorization.
- Segment workloads by function and sensitivity.
- Treat internal networks as untrusted unless proven otherwise.

A zero trust design often uses:
- Private endpoints for sensitive PaaS access
- Managed identities for workload authentication
- NSGs and route tables for segmentation
- Application-layer authorization for user and service access
- Centralized logging and alerting for detection

**Availability versus security**

Security controls can reduce availability if they create dependencies or single points of failure.

Examples:
- A workload that depends on Key Vault for startup may fail if the vault is unreachable.
- A strict firewall rule can block legitimate failover traffic.
- A private endpoint design can break if DNS is misconfigured during a regional failover.
- Overly aggressive conditional access can block automation or disaster recovery.

Mitigations:
- Cache non-sensitive configuration where appropriate.
- Test failover paths with security controls enabled.
- Document break-glass access for emergencies.
- Use redundant monitoring and alerting paths.

**Common exam decision cues**

When you see a scenario, look for the real constraint:
- **“No public exposure”**
  - Prefer private endpoints and disable public network access where supported.
- **“Simplest secure access from Azure workloads”**
  - Prefer managed identity plus RBAC.
- **“Need to restrict by VNet but keep setup simple”**
  - Consider service endpoints.
- **“Need to share limited access externally”**
  - Consider SAS or time-bound access tokens.
- **“Need strong auditability and centralized control”**
  - Use Key Vault, Azure Monitor, and role-based access control.
- **“Need low latency and high availability”**
  - Avoid unnecessary security hops unless the risk justifies them.

**Practical design patterns**

**Pattern 1: App Service to Storage**
- Use managed identity for authentication.
- Use private endpoint if storage must not be publicly reachable.
- Use RBAC for authorization.
- Add diagnostic logs for access monitoring.

**Pattern 2: Multi-tier web app**
- Put the web tier behind Application Gateway or Front Door.
- Keep the app tier private.
- Use private endpoints for database and Key Vault access.
- Restrict backend access with NSGs and service tags.
- Validate DNS for all private services.

**Pattern 3: Hybrid integration**
- Use VPN or ExpressRoute for private connectivity.
- Use certificates or federated identity where managed identity is not available.
- Plan for local development and on-premises DNS resolution.
- Monitor both cloud and on-premises trust boundaries.

**Troubleshooting checklist**

When security controls break a distributed app, check in this order:
1. Identity assignment and role assignment
2. DNS resolution for private services
3. NSG and route table rules
4. Service firewall or public network access settings
5. TLS and certificate trust
6. Diagnostic logs and platform metrics

Useful commands:
```bash
az role assignment list --assignee <principalId> -o table
az network private-endpoint list -g rg-net -o table
az network private-dns zone list -g rg-net -o table
az network nsg rule list -g rg-net --nsg-name nsg-backend -o table
```

**Key takeaways**

- Stronger security usually increases operational complexity.
- Managed identity reduces secret management but is not always the easiest option in every environment.
- Service endpoints improve network control but do not replace private endpoints.
- Private endpoints require DNS planning and careful validation.
- Logging and monitoring are essential, but they must be scoped to avoid leaking sensitive data.
- The best AZ-305 answer is usually the one that meets the requirement with the least unnecessary complexity.
