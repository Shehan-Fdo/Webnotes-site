## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Security monitoring architecture overview**

A strong Azure security monitoring design combines telemetry collection, alerting, investigation, and response. The goal is not just to detect issues, but to route the right signals to the right people with enough context to act quickly.

At a high level, you design for:

- **Collection** of logs, metrics, and security events from Azure, hybrid, and SaaS sources.
- **Correlation** across signals to reduce noise and improve fidelity.
- **Detection** using rules, thresholds, anomaly logic, and threat intelligence.
- **Response** through incidents, automation, and escalation paths.
- **Retention** that supports investigations, compliance, and cost control.

A practical architecture usually includes:

- **Azure Monitor** for platform telemetry, metrics, logs, alerts, and dashboards.
- **Log Analytics workspaces** for queryable log storage and KQL (Kusto Query Language) analysis.
- **Microsoft Sentinel** as the SIEM (Security Information and Event Management) and SOAR (Security Orchestration, Automation, and Response) layer.
- **Microsoft Defender for Cloud** for posture management and workload protection signals.
- **Automation** through action groups, Logic Apps, runbooks, and workflows.

**Core telemetry sources**

Azure monitoring works best when you treat each signal type differently. Metrics are near real-time and efficient for threshold-based alerting. Logs are richer and better for investigation and correlation. Activity data helps you understand control-plane changes, while security products add threat context.

Common sources include:

- **Azure Activity Log** for subscription-level control-plane operations such as create, update, delete, policy assignment, and RBAC changes.
- **Resource logs** for service-specific diagnostics, such as Key Vault audit events, App Service logs, or firewall logs.
- **Platform metrics** for performance and availability signals.
- **Guest OS logs** from Windows Event Logs and Linux syslog.
- **Identity logs** from Microsoft Entra ID sign-in and audit logs.
- **Security product alerts** from Defender for Cloud, Defender XDR, and third-party tools.
- **Network telemetry** from NSG flow logs, firewall logs, and load balancer diagnostics.

> Activity Log is useful for control-plane visibility, but it is not a complete identity or authorization audit trail. For access investigations, you often need Entra audit logs, sign-in logs, and workload-specific logs as well.

**Azure Monitor building blocks**

Azure Monitor is the foundation for observability in Azure. It collects and analyzes telemetry, then drives alerting and visualization.

Key components:

- **Metrics**
  - Numeric time-series data.
  - Best for fast alerting on CPU, latency, availability, queue depth, and similar signals.
  - Supports near real-time evaluation.
- **Logs**
  - Structured or semi-structured records stored in Log Analytics.
  - Best for deep investigation, correlation, and long-term analysis.
  - Queried with KQL.
- **Alerts**
  - Trigger when a condition is met.
  - Can target metrics, logs, activity events, service health, and resource health.
- **Action groups**
  - Define what happens when an alert fires.
  - Common actions include email, SMS, push notifications, voice, webhook, Azure Function, Logic App, Automation Runbook, and ITSM integration through workflow tooling.
- **Workbooks**
  - Interactive dashboards for operational and security analysis.
- **Application Insights**
  - Application performance monitoring and distributed tracing for apps.

**Alert types you should know**

Azure alerting is broader than just metric and log alerts. In exam scenarios, choose the alert type that matches the signal and response need.

- **Metric alerts**
  - Use for numeric thresholds and fast detection.
  - Good for CPU, memory, disk, request count, and latency.
- **Log alerts**
  - Use KQL to detect patterns in logs.
  - Good for security detections, correlation, and complex conditions.
- **Activity Log alerts**
  - Detect subscription-level operations such as policy changes, role assignments, resource deletions, and service health events.
- **Service Health alerts**
  - Notify you about Azure service incidents, planned maintenance, and advisories.
- **Resource Health alerts**
  - Track the health state of a specific resource.
- **Smart detection / anomaly-based alerts**
  - Useful when baseline behavior matters more than fixed thresholds.
  - Common in application monitoring and some security analytics scenarios.
- **Alert processing rules**
  - Suppress, route, or modify alerts during maintenance windows or known noisy periods.

**Practical alert design**

Good alert design balances sensitivity and noise. If you alert on every deviation, operators ignore the system. If you tune too loosely, you miss incidents.

Use these principles:

- **Set thresholds from baselines**, not guesses.
- **Prefer dynamic thresholds** when workloads have seasonal or variable behavior.
- **Group related signals** into a single incident when possible.
- **Suppress known maintenance windows** with alert processing rules.
- **Deduplicate repeated alerts** to avoid paging storms.
- **Separate detection from notification** so you can change routing without rewriting rules.

Example metric alert CLI pattern:

```bash
az monitor metrics alert create \
  --name cpu-high-alert \
  --resource-group rg-monitoring \
  --scopes /subscriptions/<subId>/resourceGroups/rg-app/providers/Microsoft.Compute/virtualMachines/vm01 \
  --condition "avg Percentage CPU > 80" \
  --description "CPU above 80 percent for the VM" \
  --window-size 5m \
  --evaluation-frequency 1m \
  --severity 2 \
  --action /subscriptions/<subId>/resourceGroups/rg-monitoring/providers/microsoft.insights/actionGroups/ag-ops
```

Use the Azure CLI help for your version before automating alert creation, because parameter requirements can vary by resource type and CLI release.

**Log Analytics workspace design**

Workspace design is a major architecture decision because it affects cost, access control, query scope, retention, and operational separation.

Important design considerations:

- **Centralized workspace**
  - Easier for SOC (Security Operations Center) analysts to query across subscriptions and workloads.
  - Simplifies Sentinel deployment and cross-domain correlation.
  - Can create higher ingestion cost concentration and broader access scope.
- **Segmented workspaces**
  - Useful when business units, regions, or compliance boundaries require separation.
  - Helps isolate access and retention policies.
  - Makes cross-workspace investigation more complex.
- **Hybrid approach**
  - Common in enterprises.
  - Use a central security workspace for security telemetry and separate operational workspaces for app teams.

Design factors to evaluate:

- **Data residency**
  - Keep logs in the required region for regulatory or contractual reasons.
- **RBAC (Role-Based Access Control)**
  - Limit who can query sensitive logs.
  - Separate SOC read access from workspace administration.
- **Cross-workspace querying**
  - Useful when you must investigate across multiple business units.
  - Adds complexity and can affect query performance.
- **Retention**
  - Match retention to investigation and compliance needs.
- **Cost**
  - Ingestion and retention costs can dominate at scale.

A common enterprise pattern is:

- One or more **security workspaces** for Sentinel and central investigations.
- Separate **application workspaces** for engineering teams.
- Restricted access to sensitive tables such as identity, endpoint, and firewall logs.

**Retention and compliance strategy**

Retention is not just a storage setting. It is part of your investigation and compliance design.

Think in layers:

- **Hot retention**
  - Fast query access in Log Analytics.
  - Best for active investigations and recent operations.
- **Archive retention**
  - Lower-cost storage for long-term retention.
  - Useful when you need to keep data for compliance but query it less often.
- **Immutable storage**
  - Protects logs from tampering or deletion when compliance requires stronger evidence preservation.
  - Often implemented with storage account immutability or related controls, depending on the data path.

Practical guidance:

- Keep high-value security logs hot long enough to cover typical detection and response windows.
- Archive older logs that are still required for audit or legal hold.
- Align retention with incident response, audit, and regulatory requirements.
- Verify that your chosen log source supports the retention model you want.

> Do not assume every log source supports the same retention or immutability options. Validate the ingestion path, workspace settings, and downstream export design.

**Azure Monitor Agent and data collection rules**

The Azure Monitor Agent (AMA) is the modern agent for collecting guest-level telemetry from Windows and Linux machines. It replaces the legacy Log Analytics agent in most new designs.

Why AMA matters:

- It uses **Data Collection Rules (DCRs)** to define what to collect and where to send it.
- It supports more flexible, centralized configuration.
- It is the preferred path for new deployments and migration planning.

DCRs let you specify:

- Which machines or scale sets receive the configuration.
- Which logs or performance counters to collect.
- Which destinations receive the data, such as Log Analytics or Azure Monitor Metrics.
- Filtering and transformation options, depending on the data source and feature support.

Example conceptual DCR flow:

```text
VM / Arc-enabled server
        |
        v
Azure Monitor Agent
        |
        v
Data Collection Rule
   |            |
   v            v
Log Analytics   Azure Monitor Metrics
```

Practical design notes:

- Use DCRs to standardize collection across fleets.
- Group similar servers under the same rule to reduce configuration drift.
- Plan migration from the legacy agent carefully if you still depend on older collection methods.
- Validate that the logs you need are supported by AMA and the relevant DCR data source.

**Example DCR JSON skeleton**

This is a simplified shape to illustrate the concept, not a complete deployment template:

```json
{
  "properties": {
    "dataSources": {
      "performanceCounters": [
        {
          "name": "cpuCounters",
          "streams": [ "Microsoft-Perf" ],
          "samplingFrequencyInSeconds": 60,
          "counterSpecifiers": [ "\\Processor(_Total)\\% Processor Time" ]
        }
      ]
    },
    "destinations": {
      "logAnalytics": [
        {
          "name": "la1",
          "workspaceResourceId": "/subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.OperationalInsights/workspaces/<workspace>"
        }
      ]
    }
  }
}
```

**Microsoft Sentinel architecture**

Microsoft Sentinel is the cloud-native SIEM and SOAR platform on Azure. It sits on top of Log Analytics and adds security analytics, incident management, hunting, and automation.

Sentinel is used for:

- **Ingestion** of security-relevant data from Microsoft and third-party sources.
- **Analytics** using scheduled rules, near-real-time rules, and anomaly detection.
- **Hunting** with KQL queries and bookmarks.
- **Incident management** with triage, grouping, and assignment.
- **Automation** through playbooks and workflows.
- **Threat intelligence** correlation.

Common data sources:

- Microsoft Entra ID sign-in and audit logs
- Microsoft Defender for Cloud alerts
- Microsoft Defender XDR signals
- Azure Activity Log
- Azure Firewall, NSG, and WAF logs
- Endpoint and server telemetry
- SaaS and third-party security products via connectors

Be precise about connectors:

- Some Microsoft sources connect natively.
- Some Azure resource logs arrive through diagnostic settings.
- Some third-party sources use CEF (Common Event Format), Syslog, API connectors, or Event Hub pipelines.
- “Azure resources” is not a single universal connector; ingestion depends on the specific service and diagnostic path.

**Sentinel incident lifecycle**

Sentinel helps you move from raw alerts to managed incidents.

Typical flow:

1. Data lands in Log Analytics or Sentinel-connected tables.
2. Analytics rules evaluate the data.
3. Matching rules generate alerts.
4. Related alerts may be grouped into incidents, depending on rule and configuration.
5. Analysts triage, investigate, and assign the incident.
6. Automation can enrich, notify, or remediate.
7. The incident is closed after resolution.

Important nuance:

- Not every alert source behaves identically.
- Some detections create incidents automatically.
- Some connector-generated alerts may require different handling or manual triage.
- Incident grouping settings matter for reducing duplicate work.

**KQL examples for security investigations**

KQL is central to Azure security monitoring. You use it to detect patterns, investigate timelines, and build analytics rules.

Example: recent failed sign-ins, with a cautious filter:

```kusto
SigninLogs
| where TimeGenerated > ago(24h)
| where ResultType != 0
| summarize FailedAttempts = count() by UserPrincipalName, IPAddress, AppDisplayName
| order by FailedAttempts desc
```

This is a useful starting point, but it is only a heuristic. In production, you usually refine by specific failure codes, sign-in type, location, and user risk to reduce false positives.

Example: suspicious delete activity from the Activity Log:

```kusto
AzureActivity
| where TimeGenerated > ago(24h)
| where OperationNameValue has_any ("delete", "Delete", "DELETE")
| project TimeGenerated, Caller, ResourceGroup, ResourceProviderValue, OperationNameValue, ActivityStatusValue, CorrelationId
| order by TimeGenerated desc
```

This is a sample pattern, not a complete deletion detector. Provider-specific delete operations may use different names, so you should tune the query to the resource types you care about.

Example: identify repeated key vault access failures:

```kusto
AzureDiagnostics
| where TimeGenerated > ago(6h)
| where ResourceProvider == "MICROSOFT.KEYVAULT"
| where ResultType != "Success"
| summarize Failures = count() by CallerIPAddress, OperationName, bin(TimeGenerated, 15m)
| where Failures > 10
```

**Alert tuning and noise reduction**

Noise reduction is one of the most important design tasks in security monitoring. A technically correct alert that fires too often becomes operationally useless.

Use these techniques:

- **Threshold calibration**
  - Start with historical baselines.
  - Adjust for business hours, seasonality, and workload type.
- **Suppression**
  - Silence alerts during approved maintenance or change windows.
- **Deduplication**
  - Collapse repeated alerts from the same root cause.
- **Grouping**
  - Combine related alerts into one incident when they share context.
- **Enrichment**
  - Add asset, owner, and threat intelligence context before paging.
- **Severity mapping**
  - Reserve high severity for high-confidence, high-impact detections.
- **Scope control**
  - Avoid applying the same rule to every resource if only a subset matters.

A practical tuning workflow:

1. Run the rule in a low-impact mode first.
2. Measure alert volume and false positives.
3. Adjust thresholds, filters, and grouping.
4. Add suppression for known benign patterns.
5. Promote to incident generation when confidence is acceptable.

**Automation and response**

Alerting is only useful if it leads to action. Azure provides several response mechanisms.

Common response options:

- **Action groups**
  - Notify people or systems when an alert fires.
- **Logic Apps**
  - Orchestrate enrichment, ticket creation, and remediation.
- **Azure Automation runbooks**
  - Execute scripted remediation or maintenance tasks.
- **Functions**
  - Run lightweight custom code.
- **Webhooks**
  - Integrate with external systems.
- **ITSM workflows**
  - Usually implemented through connectors or orchestration, not as a universal native alert action.

Use automation carefully:

- Automate low-risk, repeatable tasks first.
- Keep human approval for destructive actions.
- Log every automated response for auditability.
- Test playbooks in nonproduction before enabling them broadly.

**Microsoft Defender for Cloud integration**

Microsoft Defender for Cloud complements Azure Monitor and Sentinel by adding security posture management and workload protection.

What it contributes:

- **Secure score** and recommendations for posture improvement.
- **Regulatory compliance** dashboards and assessments.
- **Defender plans** for servers, databases, containers, storage, and more.
- **Security alerts** that can be exported to Sentinel.
- **Vulnerability and threat findings** that help prioritize remediation.

Why it matters in architecture:

- It bridges prevention and detection.
- It helps you see misconfigurations before they become incidents.
- It provides a compliance view that supports governance and audit requirements.

Typical integration pattern:

- Defender for Cloud detects a threat or misconfiguration.
- The alert is forwarded to Sentinel.
- Sentinel correlates it with identity, network, and activity data.
- Analysts investigate and trigger response workflows.

**Role-based access control and separation of duties**

Monitoring platforms contain sensitive operational and security data. Access design matters as much as telemetry design.

Recommended role separation:

- **SOC analysts**
  - Read incidents, run queries, and manage investigations.
  - Usually do not need workspace administration rights.
- **Monitoring administrators**
  - Configure data collection, alerts, and integrations.
  - Should not necessarily have broad access to all security data.
- **Resource owners**
  - View alerts and logs for their own applications or subscriptions.
  - Should have limited scope and least privilege.
- **Security engineers**
  - Build analytics rules, automation, and hunting queries.
  - May need broader read access, but not full tenant administration.

Design principles:

- Use **least privilege** everywhere.
- Separate **query access** from **configuration access**.
- Restrict access to sensitive tables and workspaces.
- Review role assignments regularly.
- Use privileged access workflows for elevated operations when possible.

**Enterprise-scale design decisions**

AZ-305 scenarios often ask you to choose between centralized and distributed designs. The right answer depends on governance, scale, and operational model.

Centralize when:

- You need a single SOC view across many subscriptions.
- You want consistent detection logic and incident handling.
- You need cross-tenant or cross-subscription correlation.
- Compliance allows central storage.

Segment when:

- Data residency or legal boundaries require separation.
- Business units operate independently.
- Different teams need different retention or access policies.
- Ingestion volume is so high that a single workspace becomes hard to manage.

A common compromise:

- Centralize security telemetry and Sentinel.
- Segment operational logs by application or region.
- Use standardized naming, tagging, and routing so investigations remain manageable.

**Service Health and Resource Health**

Availability monitoring is part of security monitoring because outages, platform incidents, and degraded services can affect detection and response.

Use:

- **Service Health alerts**
  - For Azure platform incidents, planned maintenance, and advisories.
- **Resource Health alerts**
  - For the health state of a specific resource.

Why they matter:

- They help distinguish platform issues from application issues.
- They support incident prioritization.
- They reduce time spent investigating symptoms caused by Azure service disruptions.

**Practical architecture pattern**

A mature Azure security monitoring architecture often looks like this:

- Azure resources emit diagnostic logs and metrics.
- VMs and servers use AMA with DCRs.
- Identity logs flow from Entra ID.
- Defender for Cloud sends security alerts and recommendations.
- Sentinel ingests and correlates the data.
- Analytics rules generate incidents.
- Action groups and playbooks notify, enrich, and remediate.
- Retention and archive policies preserve evidence for compliance.

Example topology:

```text
Azure resources -> Diagnostic settings -> Log Analytics -> Sentinel
VMs/Servers     -> AMA + DCR          -> Log Analytics -> Sentinel
Entra ID        -> Native connector    -> Sentinel
Defender for Cloud -> Security alerts  -> Sentinel
Sentinel -> Analytics rules -> Incidents -> Playbooks / Action groups
```

**Exam-oriented decision cues**

When you see a scenario, map the requirement to the right service:

- Need fast threshold detection on a VM metric -> **Metric alert**
- Need to detect a suspicious pattern in logs -> **Log alert**
- Need to know when Azure changes a resource or policy -> **Activity Log alert**
- Need security correlation and incident management -> **Microsoft Sentinel**
- Need posture and compliance visibility -> **Defender for Cloud**
- Need guest OS log collection at scale -> **AMA + DCR**
- Need to reduce alert noise during maintenance -> **Alert processing rules**
- Need to preserve logs for audit -> **Retention and archive strategy**

**Common pitfalls**

- Using metrics for problems that require log correlation.
- Centralizing everything without considering access boundaries or cost.
- Treating Activity Log as a complete security audit source.
- Over-alerting on noisy thresholds without tuning.
- Ignoring DCR and AMA migration planning.
- Assuming every resource supports the same diagnostic settings or log categories.
- Sending alerts to too many teams without a clear ownership model.
- Forgetting service health and resource health in availability monitoring.

**Quick reference**

- **Azure Monitor**: platform for metrics, logs, alerts, and dashboards.
- **Log Analytics**: queryable log store for KQL analysis.
- **AMA**: modern agent for guest telemetry collection.
- **DCR**: defines what AMA collects and where it sends data.
- **Sentinel**: SIEM/SOAR for detection, investigation, and response.
- **Defender for Cloud**: posture management and workload protection.
- **Action groups**: notification and automation routing.
- **Alert processing rules**: suppression and routing control.
- **Service Health / Resource Health**: availability and platform status monitoring.
