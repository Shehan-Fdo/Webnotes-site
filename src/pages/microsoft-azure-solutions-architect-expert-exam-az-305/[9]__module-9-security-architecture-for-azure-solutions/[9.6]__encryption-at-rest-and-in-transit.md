## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Encryption goals in Azure**

Encryption protects data in two main states:

- **At rest**: data stored on disks, in databases, object storage, backups, and logs.
- **In transit**: data moving between clients, services, and networks.

For AZ-305, you need to choose the right control for the workload, compliance requirement, and operational model. The key design question is not just “is it encrypted?” but “who controls the keys, where is encryption enforced, and what layer is responsible?”

A practical Azure design often combines multiple controls:

- Platform-managed encryption for baseline protection.
- Customer-managed keys for key ownership and rotation control.
- TLS for transport security.
- Private connectivity to reduce public exposure.
- Guest-level encryption only when the workload specifically requires it.

**Core encryption models**

Azure commonly uses these encryption patterns:

- **Platform-managed keys (PMK)**: Microsoft manages the keys and the service handles encryption automatically.
- **Customer-managed keys (CMK)**: you store and control the key in Azure Key Vault or Azure Key Vault Managed HSM.
- **Double encryption**: some services support an additional layer on top of platform encryption.
- **Guest encryption**: the operating system inside the VM encrypts data, usually with BitLocker or DM-Crypt.

A useful rule for exam scenarios:

- Use **PMK** when you want the simplest secure default.
- Use **CMK** when compliance, key ownership, or revocation requirements demand it.
- Use **guest encryption** only when you need encryption inside the guest OS boundary.
- Use **TLS** and **private endpoints** for data in motion.

**At-rest encryption by service**

Azure services differ in how they implement encryption at rest. Do not assume every service supports CMK in the same way.

| Service area | Default at-rest protection | CMK support notes |
|---|---|---|
| Azure Storage | Storage Service Encryption (SSE) by default | CMK is supported for many storage scenarios, but support varies by account kind and service. Blob and File are the most common exam-relevant examples. Queue and Table support is more limited and not equivalent across all scenarios. |
| Azure SQL Database / SQL Managed Instance | Transparent Data Encryption (TDE) by default | CMK is supported through a TDE protector stored in Key Vault. |
| Azure Disk Storage | Server-side encryption by default | CMK is supported for managed disks and snapshots. |
| Azure Backup | Backup data is encrypted at rest by Azure Backup | Backup encryption is managed by Azure Backup; source workload encryption does not automatically define backup encryption. |
| Azure App Service / Functions / Container Apps | Platform encryption for underlying storage and service data | CMK support is service-specific and often applies to storage, secrets, or dependent resources rather than every data path. |

> Do not generalize CMK support across all Azure services. In exam questions, check whether the service supports CMK for the specific data type and account type involved.

**Azure Storage encryption**

Azure Storage encrypts data at rest by default with **Storage Service Encryption (SSE)**. This applies to the storage service layer, not to your application logic.

Common design points:

- SSE is enabled automatically and requires no application changes.
- CMK can be used for some storage scenarios when you need key control.
- Support is service-specific:
  - Blob and File are the most common CMK scenarios.
  - Queue and Table support is more limited and should not be assumed as a default exam answer.
- Encryption scope can help isolate encryption boundaries within a storage account for certain workloads.

Example CLI for a storage account with CMK:

```bash
az storage account create \
  --name mystorageacct01 \
  --resource-group rg-secure \
  --location eastus \
  --sku Standard_LRS \
  --kind StorageV2 \
  --encryption-key-source Microsoft.Keyvault \
  --encryption-key-vault /subscriptions/<subId>/resourceGroups/rg-secure/providers/Microsoft.KeyVault/vaults/kv-secure \
  --encryption-key-name storage-key \
  --encryption-key-version <key-version>
```

Practical notes:

- The key must exist in Key Vault and be accessible to the storage service identity.
- Key rotation is operationally important; plan for versioning and re-encryption behavior.
- If the workload needs strict separation, consider separate storage accounts or encryption scopes.

**Azure SQL encryption**

Azure SQL uses **Transparent Data Encryption (TDE)** to encrypt database files at rest. TDE protects data files, log files, and backups associated with the database engine.

Important distinctions:

- TDE is about data at rest in the database engine.
- **Always Encrypted** protects sensitive columns from the database engine itself.
- **TLS** protects client-to-database traffic in transit.
- CMK for TDE is configured at the logical server level by setting a TDE protector from Key Vault.

Example flow for TDE with CMK:

1. Create or use a Key Vault key.
2. Enable a managed identity on the Azure SQL logical server.
3. Grant the identity access to the key in Key Vault.
4. Set the TDE protector to the key URI with a valid key version.

Example CLI:

```bash
az sql server update \
  --name sqlsrv01 \
  --resource-group rg-secure \
  --assign-identity

az keyvault set-policy \
  --name kv-secure \
  --object-id <sql-server-managed-identity-object-id> \
  --key-permissions get wrapKey unwrapKey

az sql server tde-key set \
  --server sqlsrv01 \
  --resource-group rg-secure \
  --kid https://kv-secure.vault.azure.net/keys/sql-tde-key/<key-version>
```

Key exam details:

- The SQL server must have a managed identity.
- The Key Vault key URI must include a valid key version.
- The identity needs permission to use the key for TDE operations.
- This is not the same as encrypting a single database with a generic “CMK switch.”

**Always Encrypted vs TDE**

These two are often confused.

| Feature | Protects from | Encrypts where | Typical use |
|---|---|---|---|
| TDE | Offline data theft from storage media | Database engine storage layer | Default at-rest protection for SQL databases |
| Always Encrypted | Database administrators and the database engine itself | Client-side before data reaches SQL | Highly sensitive columns such as SSNs or credit card numbers |

Use **Always Encrypted** when the database service should not see plaintext for selected columns. Use **TDE** when you need broad database-at-rest protection.

**Azure Disk encryption choices**

Azure offers multiple disk protection options, and the right one depends on whether you need guest-level encryption or platform-level encryption.

- **Server-side encryption for managed disks**:
  - Default for Azure managed disks.
  - Usually the preferred baseline.
  - Can use Microsoft-managed keys or CMK.
- **Azure Disk Encryption (ADE)**:
  - Guest-level encryption inside the VM.
  - Uses **BitLocker** for Windows and **DM-Crypt** for Linux.
  - Best when policy requires encryption from inside the guest OS boundary.
  - Not the default choice for most modern designs.

> ADE is a specialized option. For many architectures, server-side encryption with CMK on managed disks is the more common and operationally simpler answer.

Example conceptual comparison:

```text
Application -> OS -> Disk encryption inside guest (ADE)
Application -> Azure managed disk -> Server-side encryption at storage layer
```

Use cases:

- Choose **server-side encryption** for standard VM disk protection.
- Choose **ADE** when compliance requires guest-managed encryption or OS-level control.
- Use **CMK** when key ownership and revocation matter.

**Key management with Azure Key Vault and Managed HSM**

Azure Key Vault is the central service for storing and managing cryptographic keys, secrets, and certificates.

Key concepts:

- **Keys** are used for encryption operations such as CMK.
- **Secrets** store values like connection strings or certificate private keys.
- **Certificates** are managed objects that can include a certificate, key, and secret.

For high-assurance scenarios, **Azure Key Vault Managed HSM** provides dedicated hardware security module capabilities.

Design guidance:

- Use **Key Vault** for most CMK scenarios.
- Use **Managed HSM** when you need stronger isolation or HSM-backed key operations.
- Use soft delete and purge protection to reduce accidental key loss.
- Plan for key rotation and versioning.

Example key creation:

```bash
az keyvault key create \
  --vault-name kv-secure \
  --name app-cmk-key \
  --kty RSA \
  --size 2048
```

**RBAC and access policies**

Key Vault can be configured with either access policies or Azure role-based access control (RBAC), depending on the vault configuration.

In RBAC mode:

- Assign roles at the **vault scope** or a narrower scope if supported.
- The managed identity must have the correct data-plane permissions for the operation.
- A role like **Key Vault Crypto User** may be sufficient for some encryption operations, but not all service scenarios use the same permission set.

In access policy mode:

- Grant the identity the required key permissions such as `get`, `wrapKey`, and `unwrapKey`.
- Some services also need `list` or additional permissions depending on how they resolve key versions.

Example RBAC assignment:

```bash
az role assignment create \
  --assignee-object-id <managed-identity-object-id> \
  --assignee-principal-type ServicePrincipal \
  --role "Key Vault Crypto User" \
  --scope /subscriptions/<subId>/resourceGroups/rg-secure/providers/Microsoft.KeyVault/vaults/kv-secure
```

Practical caution:

- The exact role and permissions depend on the Azure service using the key.
- If a service cannot access the key, check both the vault authorization model and the identity’s permissions.
- Key Vault firewall and private endpoint settings can also block access.

**Encryption in transit**

Encryption in transit usually means **TLS** (Transport Layer Security), which protects data as it moves between endpoints.

Common Azure patterns:

- HTTPS for web apps and APIs.
- TLS for database connections.
- SMB encryption for Azure Files when supported by the client and configuration.
- Service-to-service TLS for PaaS integrations.

Design goals:

- Prevent eavesdropping.
- Prevent tampering.
- Authenticate the server, and sometimes the client.

Example HTTPS enforcement for App Service:

```bash
az webapp update \
  --name app-secure01 \
  --resource-group rg-secure \
  --https-only true
```

For application code, prefer modern TLS settings:

```csharp
ServicePointManager.SecurityProtocol = SecurityProtocolType.Tls12;
```

Better yet, use the platform defaults and modern SDKs rather than hard-coding protocol versions unless required by legacy compatibility.

**Private Endpoint and Private Link**

**Private Endpoint** gives a private IP address in your virtual network for a PaaS service. It reduces public exposure and keeps traffic off public endpoints, but the service still runs in Microsoft’s cloud and traffic traverses Microsoft’s backbone to reach the service.

Use Private Endpoint when you want:

- No public IP exposure for the service.
- Access from private networks only.
- Better control over routing and DNS.
- Stronger segmentation for regulated workloads.

Example pattern:

```text
Client VM -> Private IP in VNet -> Private Endpoint -> Azure PaaS service
```

Important details:

- You still need DNS configuration so the service name resolves to the private IP.
- Private Endpoint does not mean the service is deployed inside your subnet.
- Network Security Groups (NSGs) and firewall rules still matter for the client side.

**Service-specific in-transit examples**

Different services use different transport patterns.

- **Azure Storage**:
  - Use HTTPS for REST access.
  - Use private endpoints for blob/file access when public exposure is not desired.
- **Azure SQL**:
  - Use TLS connections from clients.
  - Enforce minimum TLS versions where possible.
- **App Service / Functions**:
  - Use HTTPS-only.
  - Bind custom domains with certificates.
  - Consider private endpoints for inbound access.
- **Container Apps**:
  - Use ingress with TLS termination or end-to-end TLS depending on the architecture.
  - Use private ingress for internal-only services when appropriate.

**Certificates and secret identifiers**

Certificates in Azure can be used for TLS termination, client authentication, and application identity.

Important distinction:

- The certificate **secret identifier** points to the certificate secret.
- The certificate **key identifier** points to the public key material used for cryptographic operations.

Example:

```bash
az keyvault certificate show \
  --vault-name kv-secure \
  --name tls-cert \
  --query sid -o tsv
```

Use cases:

- App Service custom TLS bindings often need the certificate secret URI.
- CMK scenarios usually need the key URI, not the certificate secret ID.

**Backup encryption**

Azure Backup encrypts backup data at rest using its own encryption model. Do not assume the source workload’s encryption settings automatically carry over to backup storage.

Key points:

- Backup encryption is managed by Azure Backup.
- Source disk encryption, SQL TDE, or storage CMK does not automatically define backup encryption behavior.
- Recovery points are protected independently of the source workload’s encryption mechanism.

This matters in design questions because backup security is a separate control plane concern from workload encryption.

**Decision guide: which control should you choose?**

| Requirement | Best fit | Why |
|---|---|---|
| Basic protection with minimal operations | Platform-managed encryption | Default, simple, low overhead |
| Key ownership and rotation control | CMK | You control the key lifecycle |
| Database engine at-rest protection | TDE | Native SQL protection for files and backups |
| Column-level secrecy from DB admins | Always Encrypted | Client-side encryption before SQL sees plaintext |
| OS-level disk encryption requirement | Azure Disk Encryption | Guest-level encryption boundary |
| Secure transport between client and service | TLS | Standard in-transit protection |
| Private access to PaaS | Private Endpoint | Private IP connectivity and no public exposure |

**End-to-end architecture examples**

A good AZ-305 answer often combines multiple controls.

Scenario 1: Regulated web app with SQL backend

- App Service uses HTTPS-only.
- App Service integrates with Key Vault for secrets.
- Azure SQL uses TDE with CMK.
- Storage account uses SSE and CMK for blob data.
- Private Endpoints protect SQL, Storage, and Key Vault access.
- DNS is configured for private name resolution.

Why this works:

- TLS protects user traffic.
- Private endpoints reduce public exposure.
- CMK satisfies key ownership requirements.
- TDE protects database files at rest.

Scenario 2: Sensitive analytics workload

- Data lands in Azure Storage with SSE.
- Selected blobs use encryption scopes or CMK.
- Processing occurs in a private network.
- Access to Key Vault is restricted through private connectivity.
- If analysts must not see plaintext columns, use Always Encrypted or application-layer encryption.

Why this works:

- Storage encryption protects the data lake.
- Network isolation reduces attack surface.
- Column-level controls protect highly sensitive fields.

Scenario 3: VM-based legacy application

- Managed disks use server-side encryption.
- ADE is added only if the compliance requirement explicitly demands guest-level encryption.
- TLS secures app-to-database traffic.
- Key Vault stores certificates and keys.
- Private Endpoint protects dependent PaaS services.

Why this works:

- You avoid unnecessary guest-level complexity unless required.
- You still meet transport and at-rest requirements.

**Common exam traps**

- Assuming every Azure service supports CMK the same way.
- Confusing TDE with Always Encrypted.
- Treating Azure Disk Encryption as the default modern choice.
- Assuming backup encryption automatically matches source workload encryption.
- Thinking Private Endpoint keeps traffic entirely inside your VNet; it provides private IP connectivity, not local-only routing.
- Using a certificate secret URI where a key URI is required, or vice versa.
- Forgetting that Key Vault access depends on both authorization model and network reachability.

**Quick CLI reference**

```bash
# Enforce HTTPS on App Service
az webapp update --name app-secure01 --resource-group rg-secure --https-only true

# Create a Key Vault key
az keyvault key create --vault-name kv-secure --name app-cmk-key --kty RSA --size 2048

# Assign a role in RBAC mode
az role assignment create \
  --assignee-object-id <object-id> \
  --role "Key Vault Crypto User" \
  --scope /subscriptions/<subId>/resourceGroups/rg-secure/providers/Microsoft.KeyVault/vaults/kv-secure

# Set SQL TDE protector
az sql server tde-key set \
  --server sqlsrv01 \
  --resource-group rg-secure \
  --kid https://kv-secure.vault.azure.net/keys/sql-tde-key/<key-version>
```

**Study checklist**

- Know the difference between **at rest** and **in transit** encryption.
- Know when to choose **PMK**, **CMK**, **TDE**, **Always Encrypted**, **ADE**, and **TLS**.
- Understand service-specific CMK support, especially for Storage and SQL.
- Remember that **Private Endpoint** provides private IP connectivity and avoids public internet exposure.
- Be able to explain Key Vault permissions, identity requirements, and key versioning.
- Recognize that backup encryption is a separate design concern.
