## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Application Insights in the Azure Monitor architecture**

Application Insights is the application performance monitoring and observability layer within **Azure Monitor**. In design terms, you use it to collect telemetry from application code and runtime components, then correlate that telemetry with platform signals from Azure resources.

Azure Monitor is the broader observability platform and typically includes:

- **Metrics** for near-real-time numeric signals and alerting.
- **Logs** for queryable event, trace, and diagnostic data in Log Analytics workspaces.
- **Traces** for request flow and dependency correlation across services.
- **Alerts** and action groups for operational response.
- **Workbooks** and dashboards for visualization and analysis.

Application Insights primarily focuses on application telemetry such as:

- Requests and response times
- Dependencies to databases, queues, and APIs
- Exceptions and failures
- Traces and custom events
- User and session activity
- Availability tests and synthetic monitoring

A useful exam framing is that Application Insights is not a replacement for Azure Monitor, but a specialized telemetry source and analysis experience inside it. You often combine it with platform metrics, diagnostic settings, and Log Analytics to get a complete operational view.

**When to choose Application Insights**

Use Application Insights when you need application-level visibility that platform metrics alone cannot provide.

- Choose it for code-level performance analysis and dependency correlation.
- Choose it for distributed tracing across microservices.
- Choose it for user journey and business transaction telemetry.
- Choose it when you need exception details, request sampling, and custom instrumentation.
- Choose it when you want availability tests and application-centric alerts.

Use other Azure monitoring options when the requirement is different:

- **Platform metrics** for infrastructure health, autoscale signals, and low-latency numeric alerting.
- **Diagnostic settings** for resource logs from Azure services such as App Service, Key Vault, or Azure SQL.
- **Log Analytics-only logging** when you mainly need centralized log retention, KQL (Kusto Query Language) analysis, or cross-resource operational reporting without app SDK instrumentation.
- **Azure Monitor metrics** when the signal is already emitted by the platform and does not require code changes.

> If the scenario asks for application code insight, dependency correlation, or business transaction tracing, Application Insights is usually the right answer. If it asks for resource-level diagnostics or platform health, start with metrics or diagnostic settings.

**Telemetry model and data types**

Application Insights telemetry is designed around correlated events that describe what the application did and how it behaved.

- **Requests** represent incoming operations such as HTTP calls or message handling.
- **Dependencies** represent outbound calls such as SQL queries, REST APIs, Service Bus operations, or storage access.
- **Exceptions** capture failures and stack traces.
- **Traces** capture diagnostic messages and structured logging.
- **Events** capture custom business or application events.
- **Metrics** capture numeric measurements, often aggregated.
- **Page views** and browser telemetry support client-side monitoring for web apps.

The important design idea is correlation. A single user action can produce a request, several dependencies, traces, and an exception, all tied together by a shared operation context. This is what makes Application Insights valuable for root-cause analysis.

**Custom dimensions, properties, and business telemetry**

Custom telemetry is critical in solution design because it lets you model business context, not just technical behavior.

- Use **custom properties** to attach contextual data such as tenant ID, order ID, region, SKU, or feature flag.
- Use **custom dimensions** to enrich telemetry with structured fields that you can query in KQL.
- Use **custom events** to record business milestones such as `CheckoutStarted`, `PaymentAuthorized`, or `SubscriptionUpgraded`.
- Use **custom metrics** for domain-specific measurements such as cart size, queue depth, or conversion latency.

This matters in exam scenarios because technical telemetry alone may not answer business questions. For example, a request failure rate may be acceptable overall, but custom dimensions can reveal that only one tenant, region, or SKU is affected.

Example in .NET:

```csharp
using Microsoft.ApplicationInsights;
using Microsoft.ApplicationInsights.DataContracts;

var telemetryClient = new TelemetryClient();

var props = new Dictionary<string, string>
{
    ["TenantId"] = tenantId,
    ["OrderId"] = orderId,
    ["Region"] = "eastus",
    ["FeatureFlag"] = "new-checkout"
};

telemetryClient.TrackEvent("CheckoutStarted", props);

telemetryClient.TrackMetric("CartItemCount", cart.Items.Count, props);

var requestTelemetry = new RequestTelemetry
{
    Name = "POST /checkout",
    Success = true,
    ResponseCode = "200"
};
requestTelemetry.Properties["TenantId"] = tenantId;
telemetryClient.TrackRequest(requestTelemetry);
```

In KQL, you can query these fields for targeted analysis:

```kusto
customEvents
| where name == "CheckoutStarted"
| extend TenantId = tostring(customDimensions.TenantId)
| summarize count() by TenantId
| order by count_ desc
```

**Distributed tracing and correlation**

Modern Azure architectures often span multiple services, so correlation is a major design consideration. Application Insights supports distributed tracing so you can follow a request across service boundaries.

Key concepts:

- **Operation ID** ties related telemetry together within a logical transaction.
- **Parent-child relationships** show which dependency was called by which request.
- **Correlation headers** propagate context between services.
- **W3C Trace Context** is the current standard for cross-service trace propagation.

In exam terms, you should recognize that W3C trace context helps maintain end-to-end observability across heterogeneous services and languages. It uses headers such as `traceparent` and `tracestate` to carry trace identity across HTTP calls and other supported transports.

Practical implications:

- Ensure all services in the call chain use compatible instrumentation.
- Prefer SDKs and agents that support W3C propagation.
- Avoid breaking correlation by manually creating outbound calls without passing trace context.
- Use correlation to diagnose latency spikes in downstream dependencies, not just the front-end request.

Example of an outbound HTTP call with trace propagation in .NET is usually handled automatically by the SDK and `HttpClient` instrumentation when configured correctly. In design reviews, the key point is not the code syntax but whether the architecture preserves trace continuity across services.

**Azure Monitor integration and data flow**

A common architecture is:

1. Application code emits telemetry through the Application Insights SDK or OpenTelemetry exporter.
2. Telemetry is ingested by Azure Monitor.
3. Data is stored and analyzed in an Application Insights resource and, in workspace-based mode, in a Log Analytics workspace.
4. Alerts, workbooks, and dashboards consume the data.
5. Operational teams use KQL to investigate incidents and trends.

This is why Application Insights should be discussed as part of the broader Azure Monitor architecture, not as a standalone product. The design question is often about where telemetry lands, who can query it, and how it is used for alerting and governance.

**Workspace-based vs classic deployment**

Workspace-based Application Insights is the modern model and is generally preferred for new designs.

| Aspect | Workspace-based | Classic |
|---|---|---|
| Data storage | Log Analytics workspace | Application Insights resource |
| Querying | Unified KQL across Azure Monitor | More isolated experience |
| Governance | Better RBAC and centralization | Less flexible |
| Integration | Stronger with Azure Monitor | More limited |
| Recommendation | Preferred for new deployments | Legacy only |

Why this matters:

- Workspace-based mode supports centralized governance and cross-resource analysis.
- It simplifies enterprise monitoring by aligning telemetry with Log Analytics.
- It helps when multiple teams need shared operational visibility.
- It supports better lifecycle management and retention policies.

**Region selection, data residency, and workspace placement**

Data residency and latency are important design decisions in regulated or global environments.

Consider:

- **Region alignment** between the application, Application Insights resource, and Log Analytics workspace.
- **Compliance requirements** that may restrict telemetry storage to a specific geography.
- **Latency** between the app and the ingestion endpoint.
- **Operational ownership** if a central platform team manages monitoring for many workloads.

Design guidance:

- Place telemetry resources in the same region or geography as the workload when compliance or latency matters.
- Use a regional workspace when data residency requirements are strict.
- Avoid sending telemetry across distant regions unless centralization is more important than latency.
- For multi-region applications, decide whether each region has its own monitoring stack or whether a central workspace aggregates telemetry.

A common enterprise pattern is:

- Regional application instances send telemetry to regional ingestion endpoints.
- Telemetry is stored in region-appropriate workspaces.
- A central operations team uses cross-workspace queries and workbooks for global visibility.

This balances compliance, performance, and operational consistency.

**Governance and RBAC**

In enterprise environments, telemetry access is a governance issue, not just a technical one. Application Insights and Log Analytics often contain sensitive operational data, user identifiers, and business context.

Use **role-based access control** (RBAC) to separate duties:

- Grant developers access to their application telemetry only.
- Grant operations teams read access to shared workspaces.
- Restrict workspace administration to platform or security teams.
- Use least privilege for alert management and query access.

Common governance considerations:

- Separate workspaces by environment, business unit, or compliance boundary when needed.
- Use Azure resource locks and policy to prevent accidental deletion or misconfiguration.
- Control who can create diagnostic settings, alerts, and data export rules.
- Review whether telemetry contains personal data before broadening access.

Typical roles and intent:

- **Monitoring Reader** for read-only operational access.
- **Log Analytics Reader** for KQL query access.
- **Monitoring Contributor** for creating alerts and workbooks.
- **Contributor** or more specific custom roles for resource management.

In multi-team environments, a shared workspace can simplify operations, but it can also create noisy access boundaries. If teams need isolation, separate workspaces or resource groups may be better than one large shared monitoring boundary.

**Alerting and action design**

Application Insights supports alerting on telemetry patterns such as failed requests, dependency failures, availability test failures, and custom metrics. In Azure Monitor, alerts should be tied to actionable response paths.

Design principles:

- Alert on symptoms that matter to users, not every technical anomaly.
- Use thresholds that reflect service-level objectives.
- Route alerts to the right team through action groups.
- Avoid excessive alert noise by tuning severity and suppression.

Example alert scenario:

- Trigger an alert when failed requests exceed 5% over 10 minutes.
- Send the alert to an action group that posts to Teams, creates a ticket, and pages on-call staff.
- Include a link to the relevant workbook or KQL query for investigation.

Portal-based creation is common in exams and real operations:

1. Open the Application Insights resource.
2. Select **Alerts** and create a new alert rule.
3. Choose a signal such as failed requests or a custom metric.
4. Define the threshold, aggregation, and evaluation frequency.
5. Attach an action group.
6. Add an alert description with investigation guidance.

Example ARM/Bicep-style alert rule for a metric signal:

```bicep
param actionGroupId string
param appInsightsName string
param location string = resourceGroup().location

resource appInsights 'Microsoft.Insights/components@2020-02-02' existing = {
  name: appInsightsName
}

resource alertRule 'Microsoft.Insights/metricAlerts@2018-03-01' = {
  name: 'high-failure-rate'
  location: 'global'
  properties: {
    description: 'Alert when failed requests are elevated'
    severity: 2
    enabled: true
    scopes: [
      appInsights.id
    ]
    evaluationFrequency: 'PT5M'
    windowSize: 'PT10M'
    criteria: {
      allOf: [
        {
          criterionType: 'StaticThresholdCriterion'
          name: 'FailedRequests'
          metricName: 'requests/failed'
          metricNamespace: 'microsoft.insights/components'
          operator: 'GreaterThan'
          threshold: 5
          timeAggregation: 'Average'
        }
      ]
    }
    actions: {
      actionGroups: [
        {
          actionGroupId: actionGroupId
        }
      ]
    }
  }
}
```

Terraform example for a simple metric alert:

```hcl
resource "azurerm_monitor_metric_alert" "failed_requests" {
  name                = "high-failure-rate"
  resource_group_name  = var.resource_group_name
  scopes              = [azurerm_application_insights.app.id]
  description         = "Alert when failed requests are elevated"
  severity            = 2
  frequency           = "PT5M"
  window_size         = "PT10M"

  criteria {
    metric_namespace = "microsoft.insights/components"
    metric_name      = "requests/failed"
    aggregation      = "Average"
    operator         = "GreaterThan"
    threshold        = 5
  }

  action {
    action_group_id = azurerm_monitor_action_group.ops.id
  }
}
```

**Practical configuration and instrumentation**

You can instrument applications in several ways depending on platform and control level.

- Use the **Application Insights SDK** for direct application telemetry.
- Use **OpenTelemetry** when you want vendor-neutral instrumentation and future portability.
- Use auto-instrumentation where supported to reduce code changes.
- Use diagnostic settings for Azure platform resources that emit logs outside the app code.

Example .NET setup with Application Insights:

```csharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddApplicationInsightsTelemetry(options =>
{
    options.ConnectionString = builder.Configuration["APPLICATIONINSIGHTS_CONNECTION_STRING"];
});

var app = builder.Build();

app.MapGet("/health", () => Results.Ok("healthy"));

app.Run();
```

Example environment variable configuration:

```bash
export APPLICATIONINSIGHTS_CONNECTION_STRING="InstrumentationKey=...;IngestionEndpoint=https://..."
```

Example KQL for request and dependency analysis:

```kusto
requests
| where timestamp > ago(24h)
| summarize avgDuration = avg(duration), failures = countif(success == false) by name
| order by failures desc
```

```kusto
dependencies
| where timestamp > ago(24h)
| summarize avgDuration = avg(duration), failures = countif(success == false) by target
| order by failures desc
```

**OpenTelemetry and modern observability design**

OpenTelemetry is increasingly relevant in Azure designs because it standardizes traces, metrics, and logs instrumentation. Application Insights can act as a backend for OpenTelemetry data in many scenarios.

Why this matters:

- It reduces vendor lock-in at the instrumentation layer.
- It supports consistent telemetry across languages and platforms.
- It aligns well with distributed tracing and W3C trace context.
- It helps when you need to instrument both cloud-native and hybrid workloads.

In exam scenarios, the key design point is that OpenTelemetry does not replace Azure Monitor; it changes how telemetry is produced and exported. You still need to decide where the data lands, how it is governed, and how it is queried.

**End-to-end design scenario**

Consider an e-commerce platform with a web app, API layer, and payment service.

Design choices:

- Use Application Insights for each application component to capture requests, dependencies, exceptions, and custom events.
- Use workspace-based Application Insights so telemetry lands in a Log Analytics workspace governed by the platform team.
- Keep the workspace in the same region as the primary production deployment for compliance and latency.
- Use custom properties such as `TenantId`, `OrderId`, and `PaymentProvider`.
- Enable W3C trace context so a checkout request can be traced from the web front end through the API to the payment service.
- Create alerts for elevated failed checkout requests and payment dependency failures.
- Use RBAC so developers can query only their application telemetry while operations can query the shared workspace.
- Use platform metrics and diagnostic settings alongside Application Insights for App Service, Key Vault, and SQL Database health.

This design gives you:

- Application-level visibility
- Cross-service correlation
- Compliance-aware storage
- Controlled access
- Actionable alerting

**Exam decision guide**

Use this quick mental model in AZ-305 questions:

- If the problem is about user experience, request latency, exceptions, or dependency failures, think **Application Insights**.
- If the problem is about VM, App Service, or resource health at the platform layer, think **Azure Monitor metrics**.
- If the problem is about Azure service logs, think **diagnostic settings** to Log Analytics.
- If the problem is about centralized querying and retention, think **Log Analytics workspace**.
- If the problem is about end-to-end service correlation, think **distributed tracing** and **W3C trace context**.
- If the problem is about governance, think **RBAC**, workspace boundaries, and data residency.

**Key takeaways**

- Application Insights is the application telemetry component within Azure Monitor.
- It is best for code-level observability, dependency correlation, and business transaction analysis.
- Workspace-based deployment is the preferred modern design.
- Region selection and workspace placement matter for compliance and latency.
- RBAC and governance are essential in multi-team environments.
- Custom dimensions and events make telemetry useful for business and solution design scenarios.
- W3C trace context is the standard way to preserve correlation across services.
- Alerts should be actionable and tied to operational response.
