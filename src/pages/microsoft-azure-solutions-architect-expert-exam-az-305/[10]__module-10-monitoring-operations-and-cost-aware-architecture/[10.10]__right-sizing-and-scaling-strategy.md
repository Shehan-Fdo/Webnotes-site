## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Right-sizing and scaling overview**

Right-sizing means matching a workload’s resources to its actual demand so you avoid paying for unused capacity while still meeting performance and availability targets. Scaling is the mechanism you use when demand changes over time. In Azure architecture, you usually combine both: first choose an appropriate baseline size, then decide how the workload should grow or shrink.

The key design question is not “How do I make it bigger?” but “What should scale, when should it scale, and what limits protect the system from over- or under-provisioning?” That decision depends on workload pattern, statefulness, latency sensitivity, and operational complexity.

- **Right-sizing** reduces waste by aligning compute, storage, and network capacity with observed usage.
- **Vertical scaling** increases the size of a single instance or service tier.
- **Horizontal scaling** adds more instances or partitions to distribute load.
- **Autoscaling** adjusts capacity automatically based on metrics or schedules.
- **Elasticity** is the ability to scale up and down quickly as demand changes.

**Core design principles**

Start with the workload’s service-level objectives (SLOs) and service-level agreements (SLAs). If you do not know the latency, throughput, and availability targets, you cannot judge whether a resource is oversized or undersized.

Use these principles when designing a scaling strategy:

- **Measure first** using Azure Monitor, Log Analytics, and service-native metrics.
- **Prefer managed scaling** where the platform can add capacity without custom orchestration.
- **Keep state external** so instances can be replaced or multiplied safely.
- **Design for failure** because scaling events often expose hidden dependencies.
- **Set guardrails** with minimum and maximum instance counts, budgets, and alerts.

A practical right-sizing cycle looks like this:

1. Collect baseline metrics for CPU, memory, disk I/O, network, request rate, and latency.
2. Identify the bottleneck rather than assuming CPU is the only constraint.
3. Resize or re-tier the resource if the current SKU is consistently too large or too small.
4. Add autoscale rules or schedules for predictable variation.
5. Reassess after deployment because application changes can shift the bottleneck.

**Vertical scaling versus horizontal scaling**

Vertical scaling, also called scale up or scale down, changes the size of a single resource. Horizontal scaling, also called scale out or scale in, changes the number of resources. Both are valid, but they solve different problems.

| Approach | Best for | Strengths | Tradeoffs |
|---|---|---|---|
| Vertical scaling | Simple workloads, legacy apps, stateful services | Easy to implement, minimal app changes | Limited by SKU ceiling, possible downtime during resize |
| Horizontal scaling | Stateless web tiers, APIs, distributed processing | Better resilience, elastic capacity, fault isolation | Requires load balancing and stateless design |
| Hybrid scaling | Mixed workloads | Balances simplicity and elasticity | More design and operational complexity |

Use vertical scaling when:

- The application is not designed for multiple instances.
- The workload needs more memory, CPU, or IOPS on a single node.
- You need a quick improvement with minimal code changes.

Use horizontal scaling when:

- The workload is stateless or can externalize session state.
- Demand varies significantly across the day or season.
- You need higher availability through instance redundancy.

> Vertical scaling can be the fastest fix, but it is not always the best long-term design. If the workload keeps hitting the same ceiling, scale out or redesign the bottleneck.

**Azure Monitor and the signals you should watch**

Right-sizing depends on telemetry. Azure Monitor provides platform metrics, logs, alerts, and autoscale signals. You should not rely on a single metric because different services bottleneck differently.

Common signals include:

- **CPU percentage** for compute-bound workloads.
- **Memory pressure** for containers, virtual machines, and app services where memory is exposed.
- **Disk latency** and queue depth for storage-bound systems.
- **Request count** and **response time** for web workloads.
- **DTU/vCore consumption** for Azure SQL Database.
- **RU/s consumption** for Azure Cosmos DB.
- **Queue length** for message-driven systems.
- **Connection count** for services with session or socket pressure.

Useful Azure CLI examples:

```bash
# View recent metrics for a VM
az monitor metrics list \
  --resource /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Compute/virtualMachines/<vmName> \
  --metric Percentage CPU \
  --interval PT1H

# Create a metric alert for high CPU
az monitor metrics alert create \
  --name vm-high-cpu \
  --resource-group <rg> \
  --scopes /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Compute/virtualMachines/<vmName> \
  --condition "avg Percentage CPU > 80" \
  --description "Alert when CPU stays high"
```

Use alerts to detect sustained pressure, not short spikes. A workload that briefly reaches 90% CPU may be healthy, while one that sits at 70% with rising latency may already need attention.

**Autoscaling patterns**

Autoscaling is most effective when the workload can add capacity without manual intervention. Azure supports several common patterns:

- **Metric-based autoscale** reacts to resource utilization such as CPU, queue length, or request count.
- **Schedule-based autoscale** changes capacity at known times, such as business hours.
- **Predictive or demand-aware scaling** is available in some services and can reduce reaction lag.
- **Event-driven scaling** is common in serverless and message-processing designs.

A good autoscale policy includes:

- A clear metric that correlates with user experience.
- A cooldown period to avoid oscillation.
- Minimum and maximum bounds.
- Separate scale-out and scale-in thresholds.
- A fallback plan if the service reaches its maximum capacity.

Example autoscale profile for a VM Scale Set:

```json
{
  "name": "weekday-profile",
  "capacity": {
    "minimum": "2",
    "maximum": "10",
    "default": "2"
  },
  "rules": [
    {
      "metricTrigger": {
        "metricName": "Percentage CPU",
        "metricResourceUri": "/subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Compute/virtualMachineScaleSets/<vmssName>",
        "timeGrain": "PT1M",
        "statistic": "Average",
        "timeWindow": "PT5M",
        "timeAggregation": "Average",
        "operator": "GreaterThan",
        "threshold": 70
      },
      "scaleAction": {
        "direction": "Increase",
        "type": "ChangeCount",
        "value": "1",
        "cooldown": "PT5M"
      }
    }
  ]
}
```

**Compute services and scaling choices**

Different Azure compute services scale in different ways. You need to know the service model before choosing a strategy.

- **Virtual Machines** scale vertically by resizing the VM size.
- **Virtual Machine Scale Sets** scale horizontally by adding or removing instances.
- **Azure App Service** scales by changing the plan size or instance count.
- **Azure Kubernetes Service (AKS)** scales pods and nodes separately.
- **Azure Functions** scales automatically based on triggers and hosting plan.
- **Azure Container Apps** scales replicas based on HTTP traffic, events, or custom metrics.

For App Service, distinguish between:
- **Scale up**: move to a larger App Service plan SKU.
- **Scale out**: increase the number of instances in the plan.
- **Autoscale**: let Azure adjust instance count based on rules or schedules.

Example App Service plan scaling with Azure CLI:

```bash
# Scale up by changing the plan SKU
az appservice plan update \
  --name asp-prod \
  --resource-group rg-app \
  --sku P1v3

# Scale out by changing the instance count
az appservice plan update \
  --name asp-prod \
  --resource-group rg-app \
  --number-of-workers 3
```

Use scale up when the app needs more CPU, memory, or faster disks per instance. Use scale out when the app can run on multiple instances and you want better throughput or availability.

**Important scaling caveat**

> Scaling out App Service does not make a stateful application automatically safe. If the app stores session state in memory, you must externalize it to a shared store such as Azure Cache for Redis, SQL Database, or a durable session mechanism.

**Azure Kubernetes Service scaling**

AKS has two separate scaling layers:

- **Pod scaling** changes the number of application replicas.
- **Node scaling** changes the size or count of cluster nodes.

Common tools include:

- **Horizontal Pod Autoscaler (HPA)** for pod-level scaling.
- **Cluster Autoscaler** for node pool scaling.
- **Vertical Pod Autoscaler (VPA)** for recommended or automatic resource requests in some scenarios.

Example HPA manifest:

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: web-api-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web-api
  minReplicas: 3
  maxReplicas: 12
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
```

AKS scaling works best when your application is stateless and your container images start quickly. If startup time is long, autoscaling may lag behind demand spikes.

**Data platform scaling**

Data services require special care because scaling can affect consistency, latency, and cost.

**Azure SQL Database**

Azure SQL Database supports several scale-related options:

- **Scale up/down** by changing service tier, compute size, or hardware generation.
- **Scale out for reads** using read replicas in supported tiers and features.
- **Hyperscale** offers different scale characteristics, including rapid storage growth and read scale-out capabilities.

Use Azure SQL Database when you want managed relational scaling without managing the underlying OS. Be careful not to assume that a single database behaves like a horizontally sharded system. A single database can scale vertically, and some tiers support read scale-out, but write scaling is still constrained by the service model.

Example CLI pattern for changing compute:

```bash
az sql db update \
  --resource-group rg-data \
  --server sql-prod-01 \
  --name appdb \
  --service-objective GP_Gen5_4
```

Choose the service objective based on current region availability and workload needs. Avoid hard-coding an older hardware generation in design notes unless the scenario explicitly requires it.

**Azure Cosmos DB**

Cosmos DB scales through throughput and partitioning.

- **Manual throughput** sets a fixed RU/s level.
- **Autoscale throughput** adjusts RU/s within a defined range.
- **Partitioning** distributes data and request load across logical partitions.

Example autoscale throughput creation for a SQL API container:

```bash
az cosmosdb sql container create \
  --account-name cosmos-prod \
  --database-name appdb \
  --name orders \
  --partition-key-path "/tenantId" \
  --max-throughput 4000
```

Use autoscale when traffic is bursty and you want the service to absorb spikes without constant manual tuning. Design the partition key carefully because poor partitioning can create hot partitions that limit scale even when total throughput is high.

**Storage scaling**

Azure Storage scales differently from compute. You usually do not “scale up” a storage account in the same way you resize a VM. Instead, you design for throughput, redundancy, and access tier.

Key considerations:

- **Blob Storage** scales by account design, request patterns, and tier selection.
- **Lifecycle management** moves data between hot, cool, and archive tiers based on age or access.
- **Premium storage** can improve latency for performance-sensitive workloads.
- **Redundancy options** such as LRS, ZRS, GRS, and GZRS affect resilience and cost.

Example lifecycle management policy:

```json
{
  "rules": [
    {
      "enabled": true,
      "name": "move-to-cool",
      "type": "Lifecycle",
      "definition": {
        "filters": {
          "blobTypes": ["blockBlob"],
          "prefixMatch": ["logs/"]
        },
        "actions": {
          "baseBlob": {
            "tierToCool": {
              "daysAfterModificationGreaterThan": 30
            }
          }
        }
      }
    }
  ]
}
```

Use lifecycle policies carefully. Archive tiering is not appropriate for every blob type or access pattern, and retrieval latency can be significant. Validate the policy against the workload’s restore expectations before applying it broadly.

**Network and edge considerations**

Scaling is not only about compute. Network bottlenecks often appear first when traffic grows.

Watch for:

- **SNAT port exhaustion** in outbound-heavy applications.
- **Load balancer limits** and backend health.
- **Firewall throughput** and inspection latency.
- **DNS caching behavior** during failover or scale events.
- **Bandwidth ceilings** on VM sizes and NIC configurations.

If outbound connections are the bottleneck, adding more app instances may worsen the problem unless you also redesign connection reuse, NAT, or egress architecture.

**Designing for scale in the application layer**

The application must cooperate with the platform. A scalable Azure architecture usually follows these patterns:

- Keep web tiers **stateless**.
- Store session state in a shared service.
- Use queues to absorb bursts and decouple producers from consumers.
- Make background jobs idempotent so retries do not corrupt data.
- Cache expensive reads to reduce backend pressure.
- Use partitioning or sharding when a single data store becomes the bottleneck.

A common topology is:

- Front end on App Service or AKS
- Shared cache in Azure Cache for Redis
- Queue in Azure Service Bus or Storage Queues
- Worker tier that scales independently
- Database tier with carefully chosen throughput and partitioning

This pattern lets you scale each tier according to its own bottleneck instead of scaling the entire system uniformly.

**Cost and governance**

Right-sizing is also a financial control. Oversized resources waste budget, while undersized resources create outages and emergency changes.

Use these controls:

- **Azure Advisor** recommendations for underutilized resources.
- **Budgets** in Cost Management to detect runaway spend.
- **Tags** to attribute cost to applications and environments.
- **Policy** to enforce approved SKUs or regions.
- **Reservations** or savings plans when workload demand is stable enough to justify commitment.

A good governance model prevents teams from scaling up indefinitely without review. It also helps you standardize approved sizes for dev, test, and production.

**Common exam scenarios**

Expect questions that ask you to choose the best scaling approach for a workload pattern.

- If the workload is a bursty web app with no session affinity requirement, choose **scale out** and autoscale.
- If the workload is a legacy app that cannot run multiple instances, choose **scale up**.
- If the workload is queue-driven, scale the worker tier based on queue depth.
- If the workload is database-bound, consider indexing, partitioning, read replicas, or a higher service tier before adding more app instances.
- If the workload has predictable business-hour peaks, use a **schedule-based** profile.
- If the workload is cost-sensitive and lightly used, right-size first before enabling aggressive autoscale.

**Troubleshooting scaling problems**

When scaling does not improve performance, the bottleneck is often elsewhere.

Check for:

- **Stateful dependencies** that prevent horizontal scale.
- **Connection pool exhaustion** in the app or database.
- **Cold start latency** in serverless or container platforms.
- **Hot partitions** in Cosmos DB or storage.
- **Thread starvation** or lock contention in the application.
- **Downstream throttling** from APIs, databases, or third-party services.

A useful troubleshooting sequence is:

1. Confirm the workload is actually hitting the expected metric.
2. Verify that scaling occurred and the new capacity is healthy.
3. Check whether latency moved to another tier.
4. Inspect logs for throttling, retries, or connection failures.
5. Revisit the architecture if the same bottleneck returns after each scale event.

**Practical decision guide**

Use this quick mental model:

- **Need more power per instance?** Scale up.
- **Need more throughput or resilience?** Scale out.
- **Need automatic adjustment?** Add autoscale.
- **Need lower cost at low usage?** Right-size and set minimums carefully.
- **Need to handle spikes without redesign?** Use managed autoscaling where available.
- **Need to scale a database?** Tune the data tier first, then consider read scale-out, partitioning, or a different service model.

**Key takeaways**

Right-sizing and scaling are not separate topics; they are part of the same capacity strategy. You start with telemetry, choose the smallest viable baseline, and then add the right scaling mechanism for the workload’s behavior. The best Azure design scales the tier that is actually constrained, keeps state out of the scaling path, and uses platform-native autoscaling where possible.
