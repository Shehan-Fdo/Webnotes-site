## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Alerting design overview**

Azure alerting design is the practice of turning telemetry into actionable operations. You choose the right signal, define a threshold or query, route notifications to the right responders, and suppress noise without hiding real incidents.

A good design balances:
- **Speed** of detection for customer-impacting issues
- **Precision** to avoid false positives
- **Coverage** across infrastructure, platform, and application layers
- **Governance** so large environments remain manageable

Azure Monitor is the core service for metrics, logs, alerts, dashboards, and action routing. In practice, alerting design is not just about creating rules; it is about building an operating model for incident detection and response.

**Core alert types and when to use them**

Azure Monitor supports several alert families. The right choice depends on the signal source and the operational question you want answered.

| Alert type | Best for | Typical signal | Strengths | Tradeoffs |
|---|---|---|---|---|
| Metric alert | Fast detection of numeric resource behavior | CPU, latency, queue depth, availability, custom metrics | Low latency, near real-time, simple thresholds | Limited context, can be noisy if thresholds are poorly tuned |
| Log alert | Complex conditions, trends, and correlation | KQL (Kusto Query Language) queries over Log Analytics | Flexible, rich context, cross-resource analysis | Higher latency, query cost, more tuning required |
| Activity log alert | Control-plane events | Create/update/delete, policy changes, service events | Good for governance and change detection | Not for performance telemetry |
| Service health alert | Azure service incidents and advisories | Planned maintenance, incidents, advisories | Useful for platform dependency awareness | Only covers Azure service health signals |
| Resource health alert | Health state of a specific resource | Available, degraded, unavailable | Good for resource-specific outage detection | Limited to resource health model |
| Prometheus alert | Kubernetes and cloud-native workloads | PromQL on managed Prometheus metrics | Native for container observability | Requires Prometheus-based monitoring setup |

Use metric alerts when you need quick detection on a single resource or metric. Use log alerts when you need richer logic, joins, or aggregation. Use activity log alerts for governance and change tracking. Use service health and resource health alerts to detect Azure-side issues that may not appear in your application telemetry.

**Signal types and alert rule criteria**

Azure alert rules are built around a signal source and a condition. The signal type determines what data the rule evaluates.

- **Metrics** are time-series values collected at short intervals.
- **Logs** are queryable records stored in Log Analytics.
- **Activity logs** capture subscription-level control-plane events.
- **Service health** and **resource health** expose platform and resource status.
- **Prometheus metrics** support Kubernetes and container scenarios through managed Prometheus.

Common metric alert criteria patterns include:
- Threshold on a single metric, such as `Percentage CPU > 80`
- Multiple conditions on the same resource, such as CPU and memory pressure
- Dynamic thresholds that learn normal behavior and flag anomalies
- Split by dimension, such as per instance, per API name, or per disk

Common log alert patterns include:
- Count of matching events over a time window
- Aggregation of numeric values from query results
- Detection of missing data or heartbeat loss
- Correlation across tables, such as failures after a deployment event

> Azure alert rules are evaluated differently depending on the signal. Do not assume a log query can be copied into a metric-style threshold rule without adjusting the evaluation model.

**Metric alerts**

Metric alerts are usually the first choice for operational thresholds because they are fast and inexpensive to evaluate. They work well for infrastructure and platform metrics that already exist in Azure Monitor.

Typical uses:
- VM CPU, disk, and network saturation
- App Service response time and HTTP error rate
- Storage account availability or transaction failures
- Service Bus queue length or dead-letter growth
- AKS node or pod-level custom metrics when exposed through monitoring

Design considerations:
- Choose the right aggregation, such as `Average`, `Maximum`, or `Total`
- Match the evaluation window to the behavior you want to catch
- Use dimensions to avoid hiding a single bad instance behind fleet averages
- Prefer dynamic thresholds when workload patterns are seasonal or bursty

Example metric alert concept in Azure CLI:
```bash
az monitor metrics alert create \
  --name "vm-high-cpu" \
  --resource-group "rg-ops" \
  --scopes "/subscriptions/<subId>/resourceGroups/rg-app/providers/Microsoft.Compute/virtualMachines/vm01" \
  --condition "avg Percentage CPU > 80" \
  --description "Alert when VM CPU stays above 80 percent" \
  --severity 2 \
  --window-size 5m \
  --evaluation-frequency 1m \
  --action "/subscriptions/<subId>/resourceGroups/rg-ops/providers/microsoft.insights/actionGroups/ag-ops"
```

This example shows the pattern, but real metric names, namespaces, and dimensions are resource-specific. Always verify the exact metric name in the Azure portal or with `az monitor metrics list-definitions`.

**Log alerts and KQL**

Log alerts use Kusto Query Language (KQL) against Log Analytics or other supported log sources. They are ideal when the alert condition depends on context, grouping, or multiple tables.

Use log alerts when you need:
- Failure counts over time
- Correlation between events
- Custom business logic
- Detection of missing telemetry
- Cross-resource or cross-subscription analysis

A log alert query should usually return a shape that the alert engine can evaluate, often a numeric result or a summarized table. Keep the query aligned with the alert rule’s evaluation mode.

Example KQL for failed sign-ins:
```kusto
SigninLogs
| where TimeGenerated > ago(15m)
| where ResultType != 0
| summarize FailedSignIns = count() by bin(TimeGenerated, 5m)
```

Example KQL for heartbeat loss:
```kusto
Heartbeat
| where TimeGenerated > ago(10m)
| summarize LastSeen = max(TimeGenerated) by Computer
| where LastSeen < ago(5m)
```

Practical guidance:
- Keep queries efficient; alert queries run repeatedly
- Use `summarize` and `bin()` to shape time-series results
- Avoid overly complex joins unless the signal truly needs them
- Test the query in Logs before converting it into an alert rule

**Activity log alerts**

Activity log alerts detect control-plane events in Azure. They are useful for governance, security, and change management.

Common scenarios:
- Resource group deletion attempts
- Role assignment changes
- Policy assignment or exemption changes
- Service principal or subscription-level administrative actions
- Azure service events that affect subscriptions

Example use case:
- Alert when a network security group rule is deleted
- Alert when a key vault access policy changes
- Alert when a subscription owner is added

Activity log alerts are especially valuable because they detect changes that may not appear in application logs. They help you monitor who changed what and when.

**Service health and resource health**

Service health alerts notify you about Azure platform incidents, planned maintenance, and advisories that may affect your resources. Resource health alerts focus on the health state of a specific Azure resource.

Use service health alerts when:
- You depend on a regional Azure service
- You need early warning for platform incidents
- You want to correlate outages with Azure advisories

Use resource health alerts when:
- You need to know whether a specific VM, database, or app is unavailable
- You want a resource-centric outage signal
- You need a simple health-state trigger for operations

These alerts are important in architecture design because not every outage is caused by your application. Sometimes the platform is the issue, and these signals help you separate Azure-side incidents from app-side failures.

**Prometheus and Grafana integration**

For Kubernetes and cloud-native environments, Azure Monitor managed Prometheus provides a native metrics path for Prometheus-style monitoring. You can alert on PromQL queries and visualize data in Grafana.

Typical design pattern:
- Collect cluster and workload metrics with managed Prometheus
- Create Prometheus alert rules for container-specific conditions
- Use Grafana dashboards for visualization and drill-down
- Route critical alerts through Azure Monitor action groups or integrated notification paths

Example PromQL alert concept:
```promql
sum(rate(container_cpu_usage_seconds_total[5m])) by (namespace) > 2
```

Use this pattern when you need Kubernetes-native observability and alerting semantics. It is especially useful for AKS environments where application teams already understand Prometheus metrics and alert expressions.

**Action groups and notification design**

Action groups define what happens when an alert fires. They are the notification and automation targets for Azure alerts.

Common action group targets:
- Email
- SMS
- Push notifications in the Azure mobile app
- Voice call
- Webhook
- Azure Function
- Logic App
- Automation Runbook
- ITSM connector
- Event Hub

Design principles:
- Use separate action groups for different audiences, such as operations, security, and application owners
- Keep critical paths simple and highly available
- Use automation targets only when the downstream process is idempotent and secure
- Prefer standardized naming so responders can identify the purpose quickly

Example action group creation pattern:
```bash
az monitor action-group create \
  --name "ag-ops" \
  --resource-group "rg-ops" \
  --short-name "ops" \
  --action email ops-team ops@example.com \
  --action webhook incident-hook https://example.contoso.com/webhooks/azure-alerts
```

Webhook and automation security considerations:
- Use HTTPS endpoints only
- Validate the incoming payload and source
- Protect endpoints with authentication where possible
- Rotate secrets and keys regularly
- Treat alert webhooks as operational integrations, not public APIs

> Alert processing rules do not replace action groups. Action groups define notification targets; processing rules are mainly for suppression and filtering.

**Suppression, maintenance, and alert noise control**

Noise control is a major part of alerting design. If you do not suppress expected alerts, responders stop trusting the system.

Primary suppression mechanisms:
- **Alert processing rules** to suppress or filter alerts based on scope, time, or conditions
- **Maintenance configurations** to suppress alerts during planned maintenance windows
- **Rule tuning** to adjust thresholds, evaluation frequency, and dimensions
- **Alert grouping** to reduce duplicate notifications from related signals

Use maintenance configurations when:
- You know a planned change will generate expected alerts
- You want a time-bound suppression tied to maintenance activity

Use alert processing rules when:
- You need broader suppression or filtering logic
- You want to mute alerts for a scope, severity, or time window
- You need operational control without changing the alert rule itself

Important distinction:
- Maintenance is usually tied to planned work.
- Processing rules are broader operational filters and suppressors.

**Severity design**

Azure Monitor severity uses a numeric scale from `0` to `4`, where lower numbers indicate higher severity.

A practical mapping often looks like:
- `0` critical
- `1` error
- `2` warning
- `3` informational
- `4` verbose

This mapping is an organizational convention, not a hard semantic rule enforced by Azure. Use a consistent internal standard so responders know what each severity means.

Design tips:
- Reserve `0` and `1` for customer-impacting or urgent issues
- Use `2` for actionable warnings that need attention
- Use `3` and `4` for informational signals, trend monitoring, or low-priority events
- Align severity with paging policy, not just technical importance

**Alert rule configuration details**

When you design an alert rule, pay attention to the following configuration elements:

- **Scope**: the resource, resource group, subscription, or workspace being monitored
- **Signal type**: metric, log, activity log, service health, resource health, or Prometheus
- **Condition**: threshold, query result, health state, or anomaly detection
- **Evaluation frequency**: how often Azure checks the condition
- **Window size**: how much data is included in each evaluation
- **Severity**: the operational priority
- **Action groups**: who gets notified or what automation runs
- **Description**: the human-readable explanation for responders
- **Dimensions**: how the signal is partitioned
- **Suppression**: whether maintenance or processing rules apply

A well-designed rule answers:
- What is failing?
- How do you know?
- Who should respond?
- How quickly should they know?
- When should the alert be muted?

**Portal, ARM, and Bicep deployment considerations**

In enterprise environments, alert rules should be deployed as code rather than created manually in the portal.

Why use infrastructure as code:
- Repeatable deployments across environments
- Version control and peer review
- Easier drift detection
- Standardized naming and tagging
- Faster recovery and environment cloning

Common deployment options:
- Azure portal for ad hoc or exploratory setup
- Azure CLI for scripting and automation
- ARM templates for declarative deployment
- Bicep for readable infrastructure as code

Example Bicep pattern for an action group:
```bicep
resource actionGroup 'Microsoft.Insights/actionGroups@2023-01-01' = {
  name: 'ag-ops'
  location: 'global'
  properties: {
    groupShortName: 'ops'
    enabled: true
    emailReceivers: [
      {
        name: 'ops-email'
        emailAddress: 'ops@example.com'
        useCommonAlertSchema: true
      }
    ]
  }
}
```

Example Bicep pattern for a metric alert:
```bicep
resource metricAlert 'Microsoft.Insights/metricAlerts@2018-03-01' = {
  name: 'vm-high-cpu'
  location: 'global'
  properties: {
    description: 'CPU above threshold'
    severity: 2
    enabled: true
    scopes: [
      resourceId('Microsoft.Compute/virtualMachines', 'vm01')
    ]
    evaluationFrequency: 'PT1M'
    windowSize: 'PT5M'
    criteria: {
      'odata.type': 'Microsoft.Azure.Monitor.SingleResourceMultipleMetricCriteria'
      allOf: [
        {
          name: 'cpuHigh'
          metricName: 'Percentage CPU'
          metricNamespace: 'Microsoft.Compute/virtualMachines'
          operator: 'GreaterThan'
          threshold: 80
          timeAggregation: 'Average'
        }
      ]
    }
    autoMitigate: true
    actions: [
      {
        actionGroupId: actionGroup.id
      }
    ]
  }
}
```

Use the exact resource type and API version that matches your deployment standard. Alert resource schemas evolve, so validate templates against current Azure documentation and your CI pipeline.

**Governance, naming, and scale**

At enterprise scale, alerting design becomes a governance problem.

Recommended practices:
- Use a naming convention that encodes workload, signal type, and severity
- Tag alert resources with owner, environment, and application metadata
- Standardize action groups by team or service
- Maintain a catalog of approved alert patterns
- Review alert volume and false-positive rates regularly

Useful naming pattern:
- `ag-<team>-<env>`
- `al-<workload>-<signal>-<severity>`
- `ar-<platform>-<resource>-<purpose>`

Governance concerns:
- Too many unique alert rules create maintenance overhead
- Inconsistent severities make paging unreliable
- Duplicate rules across subscriptions increase noise
- Unowned alerts become stale and ignored

**Limits, quotas, and operational constraints**

Large environments must account for service limits and operational overhead. Azure Monitor has quotas and limits for alert rules, action groups, and related resources, and these can vary by alert type and region.

Design implications:
- Do not assume unlimited alert rule creation
- Consolidate where appropriate, but avoid over-aggregation that hides failures
- Use shared action groups carefully to avoid blast radius
- Plan for subscription and resource group organization that supports scale

Because limits can change, verify current quotas before large rollouts. In exam scenarios, the key idea is that alerting design must consider scale, not just functionality.

**Testing and validation**

Alerting is only useful if it works end to end. Validate both the rule logic and the notification path.

Practical validation methods:
1. Trigger a controlled condition, such as a test VM CPU spike or a synthetic failure.
2. Run the query or metric condition in the portal to confirm it matches expected data.
3. Verify the alert fires with the intended severity and scope.
4. Confirm the action group receives the notification.
5. Check downstream automation, ticket creation, or webhook processing.
6. Clear the condition and confirm resolution behavior.

Additional validation ideas:
- Use synthetic transactions for application availability
- Simulate log events in a nonproduction workspace
- Test maintenance suppression during planned changes
- Verify that on-call routing and escalation policies receive the alert

**Decision matrix for exam scenarios**

Choose the alert type based on the operational question:

- If you need fast detection of a numeric resource condition, use a **metric alert**
- If you need correlation or custom logic, use a **log alert**
- If you need to detect Azure control-plane changes, use an **activity log alert**
- If you need to know about Azure platform incidents, use a **service health alert**
- If you need resource-specific availability state, use a **resource health alert**
- If you run Kubernetes with Prometheus metrics, use a **Prometheus alert**

A simple rule of thumb:
- Metrics for speed
- Logs for logic
- Activity logs for governance
- Health signals for platform awareness
- Prometheus for cloud-native workloads

**Common design mistakes**

- Alerting on averages when one bad instance matters
- Using too-short windows that create flapping
- Using too-long windows that delay response
- Sending every alert to the same action group
- Failing to suppress planned maintenance
- Ignoring resource health and service health signals
- Building alerts manually without source control
- Treating severity as a universal truth instead of an operational policy
- Overusing log alerts for conditions that metrics can detect faster

**Exam-oriented takeaways**

Azure alerting design is about selecting the right signal, tuning the rule, and routing the response. The best architecture uses multiple alert types together, with clear ownership and suppression controls.

Remember these high-value points:
- Metric alerts are best for fast threshold detection.
- Log alerts are best for flexible, query-driven conditions.
- Activity log alerts are best for control-plane governance.
- Service health and resource health alerts help you distinguish Azure-side issues from application issues.
- Action groups define notification and automation targets.
- Alert processing rules and maintenance configurations reduce noise, but they do not replace good rule design.
- Infrastructure as code is the preferred way to manage alerting at scale.
