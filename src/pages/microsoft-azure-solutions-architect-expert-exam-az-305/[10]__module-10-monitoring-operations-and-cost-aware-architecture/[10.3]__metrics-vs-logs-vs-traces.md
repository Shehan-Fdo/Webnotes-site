## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Observability model**

Azure observability usually combines three signal types:

- **Metrics** tell you *how much* and *how often* something happens.
- **Logs** tell you *what happened* and preserve detailed event context.
- **Traces** tell you *where time is spent* across a request path.

Use them together rather than treating them as competing tools. Metrics are best for fast health checks and alerting, logs are best for investigation and auditing, and traces are best for distributed request analysis.

A practical way to remember the difference is:

- Metrics are aggregated numeric time series.
- Logs are discrete records with rich fields.
- Traces are correlated spans that follow a request through components.

**Metrics**

Metrics are lightweight numeric measurements collected at regular intervals. In Azure, they are commonly used for near-real-time monitoring, dashboards, and alert rules.

Typical metric characteristics:

- **Low cardinality** compared with logs, which makes them efficient for charting and alerting.
- **Time-series** format, so you can trend CPU, latency, queue depth, or request rate.
- **Fast evaluation** for operational alerts because they are already aggregated.

Use metrics when you need to answer questions like:

- Is the service healthy right now?
- Is latency increasing over the last 15 minutes?
- Are we approaching a capacity limit?
- Did error rate spike after a deployment?

Common Azure metric examples include:

- CPU percentage
- Memory usage, where available
- HTTP server errors
- Request count
- Queue length
- Disk I/O
- Network throughput

In Azure Monitor, metrics are often the first signal you use for alerting because they are efficient and easy to visualize. However, they usually do not contain enough detail to explain *why* a problem happened.

> <mark>Do not assume every Azure resource exposes the same metric names.</mark> Metric namespaces and available counters are resource-provider specific.

**Logs**

Logs are detailed records of events, requests, exceptions, and platform activity. They are more verbose than metrics and are designed for investigation, auditing, and forensic analysis.

Use logs when you need to answer questions like:

- Which request failed?
- What exception occurred?
- Which user or dependency caused the issue?
- What changed before the incident?
- Which resource generated the warning?

Azure log sources commonly include:

- **Azure Activity Log** for subscription-level control-plane events such as create, update, delete, and role assignment actions.
- **Resource logs** for service-specific diagnostic data.
- **Application logs** for app-level events, exceptions, and traces.
- **Security logs** for access and threat investigation, depending on the service.

Logs are usually stored in a Log Analytics workspace for querying with Kusto Query Language (KQL). In workspace-based Application Insights, application telemetry is also available in Log Analytics tables, which makes cross-resource analysis much easier.

Example KQL for workspace-based Application Insights:

```kusto
requests
| where timestamp > ago(1h)
| where success == false
| project timestamp, name, resultCode, duration, operation_Id
| order by timestamp desc
```

Another example for exceptions:

```kusto
exceptions
| where timestamp > ago(24h)
| project timestamp, type, outerMessage, operation_Id, cloud_RoleName
| order by timestamp desc
```

And a dependency-focused query:

```kusto
dependencies
| where timestamp > ago(1h)
| summarize count(), avg(duration) by target, type
| order by count_ desc
```

These tables are typical of the workspace-based schema. If you are working with older classic Application Insights, the storage and query model can differ.

**Traces**

Traces are correlated records that show the path of a request across services. They are most useful in distributed systems where one user action may touch a web app, API, queue, database, and downstream service.

A trace usually contains:

- A **trace ID** that ties related spans together.
- A **span** for each operation in the request path.
- Timing information for each span.
- Parent-child relationships that show call hierarchy.

Use traces when you need to answer questions like:

- Where is the latency coming from?
- Which downstream dependency is slow?
- Did the request fail in the frontend, API, or database layer?
- How did the request move through microservices?

In Azure, traces are commonly collected through OpenTelemetry and exported to Azure Monitor using the Azure Monitor OpenTelemetry Distro or related instrumentation. Application Insights is part of Azure Monitor in the modern model, so think of it as the telemetry experience rather than a separate backend.

A simple trace flow might look like this:

```text
Client -> Web App -> API -> Database
          |          |      |
          span       span   span
```

If the API span is slow, you can inspect its child spans to see whether the delay comes from a database call, an external HTTP dependency, or internal processing.

**Side-by-side comparison**

| Signal | Best for | Data shape | Typical retention use | Strength | Limitation |
|---|---|---|---|---|---|
| Metrics | Alerting and trending | Numeric time series | Operational dashboards and threshold alerts | Fast and efficient | Limited detail |
| Logs | Investigation and auditing | Event records | Deep troubleshooting and compliance | Rich context | Higher volume and query cost |
| Traces | Distributed request analysis | Correlated spans | End-to-end latency analysis | Shows request flow | Requires instrumentation |

Metrics are usually the first line of defense, logs are the evidence trail, and traces are the path map.

**How they work together**

A strong troubleshooting workflow often starts broad and ends narrow:

1. Check a metric dashboard for the symptom.
2. Use logs to identify the failing component or error pattern.
3. Use traces to follow the request path and isolate the slow or failing dependency.

For example, if request latency increases:

- Metrics show the latency spike.
- Logs show increased timeout exceptions.
- Traces reveal that the API is waiting on a downstream database call.

This layered approach reduces mean time to resolution because each signal answers a different question.

**Azure Monitor and Application Insights**

Azure Monitor is the central observability platform in Azure. It collects metrics, logs, and traces from Azure resources, virtual machines, containers, and applications.

Application Insights is the application monitoring capability within Azure Monitor. In modern setups, especially workspace-based Application Insights, telemetry lands in Log Analytics and can be queried with KQL alongside other Azure data.

Common telemetry mapping:

- **Metrics** for availability, request rate, and performance trends.
- **Logs** for requests, exceptions, dependencies, and custom events.
- **Traces** for distributed request correlation and dependency timing.

A typical application monitoring setup might include:

- Application code instrumented with OpenTelemetry.
- Azure Monitor ingestion for telemetry.
- Log Analytics workspace for querying and retention.
- Metric alerts for fast detection.
- Log alerts for pattern-based detection.
- Workbooks or dashboards for visualization.

**Practical Azure CLI examples**

You often use Azure CLI to inspect monitoring configuration and resource telemetry settings.

List diagnostic settings on a resource:

```bash
az monitor diagnostic-settings list \
  --resource /subscriptions/<subId>/resourceGroups/<rg>/providers/<provider>/<resourceType>/<resourceName>
```

Create a diagnostic setting by selecting categories that the resource actually exposes:

```bash
az monitor diagnostic-settings create \
  --name send-to-law \
  --resource /subscriptions/<subId>/resourceGroups/<rg>/providers/<provider>/<resourceType>/<resourceName> \
  --workspace /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.OperationalInsights/workspaces/<workspaceName> \
  --logs '[{"category":"<log-category>","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

> <mark>Always verify the available diagnostic categories for the specific resource.</mark> Categories vary by Azure service and resource type.

List metric definitions for a resource:

```bash
az monitor metrics list-definitions \
  --resource /subscriptions/<subId>/resourceGroups/<rg>/providers/<provider>/<resourceType>/<resourceName>
```

Query a metric once you know the exact metric name:

```bash
az monitor metrics list \
  --resource /subscriptions/<subId>/resourceGroups/<rg>/providers/<provider>/<resourceType>/<resourceName> \
  --metric "<metric-name>" \
  --interval PT1M \
  --aggregation Average
```

Because metric names differ by service, check the definitions first instead of assuming a name such as `Requests` exists for every resource.

**KQL examples for investigation**

Use Kusto Query Language to correlate signals in Log Analytics.

Find failed requests in the last hour:

```kusto
requests
| where timestamp > ago(1h)
| where success == false
| summarize failures = count() by resultCode, name
| order by failures desc
```

Find slow dependencies:

```kusto
dependencies
| where timestamp > ago(1h)
| summarize avgDuration = avg(duration), p95Duration = percentile(duration, 95) by target
| order by p95Duration desc
```

Correlate a request with its dependencies using `operation_Id`:

```kusto
requests
| where timestamp > ago(1h)
| where duration > 2s
| project timestamp, operation_Id, name, duration, resultCode
| join kind=inner (
    dependencies
    | project operation_Id, dependencyName = name, target, duration, success
) on operation_Id
| order by timestamp desc
```

These queries are most useful when telemetry is consistently instrumented and correlated across services.

**When to choose each signal**

Choose **metrics** when:

- You need fast alerting.
- You want a dashboard with trends.
- You care about thresholds and SLOs.
- You need low-overhead monitoring.

Choose **logs** when:

- You need detailed context.
- You are investigating a failure.
- You need auditability or event history.
- You want to search by fields, text, or patterns.

Choose **traces** when:

- You have microservices or distributed transactions.
- You need end-to-end latency breakdown.
- You want to identify the slow hop in a call chain.
- You need request correlation across components.

**Common design patterns**

A well-designed Azure observability strategy often uses all three:

- **Metrics for detection**
  - Alert on CPU, latency, error rate, or queue depth.
- **Logs for diagnosis**
  - Capture exceptions, request details, and platform events.
- **Traces for root cause**
  - Follow the request path through services and dependencies.

This pattern helps you avoid overusing logs for everything, which can increase cost and make alerting noisy.

**Operational guidance**

Keep these practical points in mind:

- Instrument applications consistently so traces and logs share correlation IDs.
- Use metric alerts for symptoms that need immediate response.
- Use log queries for deeper analysis after an alert fires.
- Retain only the telemetry you need for the required period.
- Sample high-volume traces carefully to control cost while preserving useful coverage.

A simple architecture for an app might be:

```text
App Service / AKS / VM
   -> metrics to Azure Monitor
   -> logs to Log Analytics
   -> traces via OpenTelemetry to Azure Monitor
   -> alerts and workbooks for operations
```

**Exam focus**

For AZ-305, you should be able to distinguish the signals by purpose:

- **Metrics** are for quantitative health and alerting.
- **Logs** are for detailed event analysis and auditing.
- **Traces** are for distributed request flow and latency breakdown.

You should also recognize that:

- Azure Monitor is the central platform.
- Application Insights is the application telemetry experience within Azure Monitor.
- Workspace-based telemetry enables KQL queries across application and platform data.
- Diagnostic settings and metric names are resource-specific, so you verify them before configuring alerts or queries.

**Quick memory aid**

- Metrics answer: *Is it healthy?*
- Logs answer: *What happened?*
- Traces answer: *Where did it slow down?*
