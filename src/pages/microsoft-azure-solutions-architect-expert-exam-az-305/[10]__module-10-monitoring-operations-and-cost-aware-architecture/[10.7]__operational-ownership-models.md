## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Operational ownership model overview**

Operational ownership defines who is responsible for each layer of a solution: physical infrastructure, host OS, runtime, application code, data, identity, networking, security controls, backup, monitoring, and incident response. In Azure architecture decisions, this model helps you choose the right service type and avoid gaps where everyone assumes someone else is handling a control.

A useful exam mindset is to ask: “What do I still own after I choose this service?” The answer changes significantly between **IaaS** (Infrastructure as a Service), **PaaS** (Platform as a Service), **SaaS** (Software as a Service), and hybrid/on-premises designs.

**Shared responsibility by service model**

The shared responsibility model is not a single fixed table; it varies by service and by feature. Still, the broad pattern is consistent:

| Responsibility area | IaaS | PaaS | SaaS |
|---|---|---|---|
| Physical datacenter, power, cooling | Microsoft | Microsoft | Microsoft |
| Host hardware and virtualization | Microsoft | Microsoft | Microsoft |
| Guest OS patching | Customer | Microsoft | Microsoft |
| Runtime/platform patching | Customer | Microsoft | Microsoft |
| Application code | Customer | Customer | Microsoft |
| Data classification and access | Customer | Customer | Customer |
| Identity and access for users | Customer | Customer | Customer |
| Service-side authorization model | Customer | Customer | Microsoft |
| Network controls and segmentation | Customer | Shared | Mostly Microsoft, with customer configuration options |
| Backup/restore design | Customer | Shared | Shared, service-dependent |
| Monitoring and alerting | Customer | Shared | Shared |
| Compliance evidence for app/data | Customer | Shared | Shared |

This matrix is intentionally high level. For exam questions, the key is to identify the boundary: Azure always owns the underlying cloud platform, but your ownership shrinks as you move from IaaS to PaaS to SaaS.

**IaaS ownership boundaries**

With IaaS, you manage the operating system and everything above it. Azure provides the virtual machine, storage, and networking primitives, but you are responsible for hardening, patching, and operating the guest workload.

Common IaaS examples include:
- **Azure Virtual Machines**
- **Azure Virtual Machine Scale Sets**
- Self-managed databases or application servers running on VMs
- Custom network appliances deployed as VMs

You typically own:
- **Guest OS** installation, patching, and configuration
- **Middleware/runtime** such as IIS, Apache, NGINX, Java, .NET runtime, or database engine
- **Application code** and deployment pipelines
- **Data protection** inside the guest, including encryption choices and key handling
- **Backup** strategy for the VM and application-consistent recovery
- **Security hardening** such as endpoint protection, local firewall rules, and vulnerability remediation
- **Monitoring agents** and log collection inside the guest

Azure typically owns:
- Datacenter facilities
- Physical servers, storage, and networking
- Hypervisor and virtualization layer
- Basic platform availability for the VM service

A practical IaaS example:

```bash
az vm create \
  --resource-group rg-prod \
  --name vm-app01 \
  --image Ubuntu2204 \
  --admin-username azureadmin \
  --generate-ssh-keys
```

After deployment, you still need to:
- Patch the OS
- Configure SSH hardening
- Install and maintain the application stack
- Set up backup and monitoring
- Define network security groups and access rules

**PaaS ownership boundaries**

With PaaS, Azure manages the operating system, runtime, and much of the platform lifecycle. You focus on your application, data, and service configuration. This is why PaaS reduces operational overhead and often improves patching consistency.

Common PaaS examples include:
- **Azure App Service**
- **Azure SQL Database**
- **Azure SQL Managed Instance**
- **Azure Functions**
- **Azure Container Apps** in a managed platform model
- **Azure Kubernetes Service (AKS)** as a managed control plane service, though you still own cluster design and workload operations more than with classic PaaS

You typically own:
- **Application code**
- **Data schema and data access**
- **Configuration** such as app settings, connection strings, and feature flags
- **Identity integration** for users and service principals
- **Network exposure** such as private endpoints, access restrictions, and firewall rules where supported
- **Backup/restore settings** when the service exposes them
- **Monitoring configuration** such as diagnostic settings, alerts, and log routing

Azure typically owns:
- OS patching
- Runtime patching
- Platform scaling mechanics
- Service availability of the managed platform
- Many built-in security and resilience controls

> Do not assume every PaaS service has the same backup, restore, or point-in-time recovery behavior. These capabilities are service-specific and may require explicit configuration.

A practical App Service configuration example:

```bash
az webapp config appsettings set \
  --resource-group rg-prod \
  --name app-prod-01 \
  --settings MySetting=Production ApiBaseUrl=https://api.contoso.com
```

A practical SQL Database backup and geo-redundancy example:

```bash
az sql db create \
  --resource-group rg-prod \
  --server sql-prod-01 \
  --name appdb \
  --service-objective GP_Gen5_2 \
  --backup-storage-redundancy Geo
```

Use PaaS when you want:
- Faster delivery with less patching overhead
- Managed scaling and platform maintenance
- Stronger standardization for common application patterns
- Reduced operational burden for small platform teams

**SaaS ownership boundaries**

With SaaS, Azure or the vendor operates the application itself. You configure the tenant, users, policies, and data governance, but you do not manage the application code or platform internals.

Common SaaS examples in the Microsoft ecosystem include:
- **Microsoft 365**
- **Dynamics 365**
- **Power BI**
- **Azure DevOps Services** as a SaaS-adjacent managed service
- Many third-party applications delivered through Azure Marketplace as SaaS offerings

You typically own:
- **User identity lifecycle** in your tenant
- **Access control** through roles, groups, and conditional access
- **Data governance** such as retention, classification, and sharing policies
- **Tenant configuration** and service settings
- **Endpoint/device posture** when access depends on managed devices

The vendor typically owns:
- Application code and service updates
- Platform patching
- Service-side authorization logic
- Availability of the SaaS application
- Core incident response for the service

A common exam trap is to overstate customer ownership in SaaS. You usually manage who can access the service and what they can do in your tenant, but the vendor still controls the application’s internal authorization boundaries and service identity model.

**Identity, encryption, and key management**

Ownership of security controls changes by service model, and exam questions often test this boundary.

- **Identity**
  - In IaaS, you manage local accounts, domain integration, and host access.
  - In PaaS, you usually manage application identities, managed identities, and tenant access.
  - In SaaS, you manage tenant users, groups, and access policies, while the vendor manages the application service identity model.

- **Encryption**
  - In IaaS, you often manage disk encryption, certificate deployment, and key storage.
  - In PaaS, Azure may provide encryption at rest by default, but you may still choose customer-managed keys for some services.
  - In SaaS, encryption is mostly vendor-managed, though you may control tenant-level settings or key options where supported.

- **Key management**
  - Use **Azure Key Vault** when you need centralized secret, key, and certificate management.
  - In IaaS, you often integrate Key Vault with VMs or custom apps.
  - In PaaS, you commonly use managed identities to access Key Vault without storing secrets in code.
  - In SaaS, key management options are limited and service-specific.

Example of a managed identity accessing Key Vault from an App Service:

```bash
az webapp identity assign \
  --resource-group rg-prod \
  --name app-prod-01

az keyvault set-policy \
  --name kv-prod-01 \
  --object-id <principal-object-id> \
  --secret-permissions get list
```

**Backup and disaster recovery**

Backup and disaster recovery (DR) responsibilities depend on the service and the recovery objective.

- In **IaaS**, you design the full backup and DR strategy:
  - VM backups
  - Application-consistent snapshots
  - Database backups
  - Cross-region replication
  - Recovery testing

- In **PaaS**, Azure may provide built-in backup or restore features, but you still need to:
  - Enable them
  - Validate retention
  - Understand restore scope
  - Test recovery procedures
  - Confirm region and service limitations

- In **SaaS**, the vendor usually provides service-level resilience, but you may still need:
  - Data export
  - Retention policies
  - Legal hold or eDiscovery features
  - Tenant recovery planning

A practical VM backup example:

```bash
az backup vault create \
  --resource-group rg-prod \
  --name vault-prod-01 \
  --location eastus

az backup protection enable-for-vm \
  --resource-group rg-prod \
  --vault-name vault-prod-01 \
  --vm vm-app01 \
  --policy-name DefaultPolicy
```

A practical geo-redundancy choice for storage-backed workloads:

```bash
az storage account create \
  --name stprod01 \
  --resource-group rg-prod \
  --location eastus \
  --sku Standard_GRS
```

Use geo-redundancy when business continuity requires regional resilience, but confirm the service’s actual failover and restore behavior before assuming automatic application recovery.

**Monitoring, logging, and incident response**

Monitoring ownership also shifts with the service model.

- In **IaaS**, you install and manage most monitoring agents and logs.
- In **PaaS**, you usually enable diagnostic settings, platform metrics, and application telemetry.
- In **SaaS**, you rely on vendor-provided audit logs, usage reports, and tenant-level monitoring.

A practical Azure Monitor diagnostic configuration example:

```bash
az monitor diagnostic-settings create \
  --name send-to-law \
  --resource /subscriptions/<sub-id>/resourceGroups/rg-prod/providers/Microsoft.Web/sites/app-prod-01 \
  --workspace /subscriptions/<sub-id>/resourceGroups/rg-monitor/providers/Microsoft.OperationalInsights/workspaces/law-prod \
  --logs '[{"category":"AppServiceHTTPLogs","enabled":true},{"category":"AppServiceConsoleLogs","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

Operationally, ownership affects:
- **Alert tuning**: who creates and maintains thresholds
- **Log retention**: who pays and who configures it
- **Incident triage**: who checks app logs first
- **Escalation**: whether you open a Microsoft support case or investigate your own code and configuration first
- **SLA interpretation**: whether the issue is platform availability, application failure, or a customer configuration problem

**Support boundaries and SLAs**

Support boundaries follow ownership boundaries. If Microsoft owns the platform layer, Microsoft support can investigate platform incidents. If you own the guest OS or application, you must troubleshoot those layers before escalating.

A practical way to think about support:
- If the issue is in the **Azure service control plane**, Microsoft support is relevant.
- If the issue is in your **code, configuration, or guest OS**, you own the first response.
- If the issue spans both, you need evidence from logs, metrics, and configuration to separate platform from customer responsibility.

SLA (Service Level Agreement) and SLO (Service Level Objective) interpretation also depends on ownership:
- A platform SLA does not guarantee your application is healthy.
- A highly available service can still fail due to bad configuration, expired certificates, or broken deployment logic.
- For composite solutions, the end-to-end availability is only as strong as the weakest owned layer.

**Hybrid and on-premises ownership**

Hybrid solutions split responsibility across Azure, on-premises infrastructure, and sometimes third-party services. This matters in migration and modernization scenarios.

Typical hybrid ownership patterns:
- On-premises datacenter teams own physical servers, network gear, and local storage.
- You may own identity synchronization, connectivity, and application dependencies across both environments.
- Azure owns the cloud platform components you deploy there.
- Shared responsibility becomes more complex because failure domains cross boundaries.

Common hybrid examples:
- **Azure Arc**-enabled servers and Kubernetes clusters
- **VPN** or **ExpressRoute** connectivity to on-premises networks
- **Azure Stack HCI** or other edge/hybrid platforms
- Lift-and-shift workloads where some tiers remain on-premises

A hybrid design often requires you to define:
- Which team patches which OS instances
- Where logs are collected
- Which side owns DNS, certificates, and firewall rules
- How identity is federated
- How backup and DR work across sites

**Ownership by control area**

| Control area | IaaS | PaaS | SaaS |
|---|---|---|---|
| Compute | Customer manages VM workload | Microsoft manages platform compute | Microsoft manages application compute |
| OS patching | Customer | Microsoft | Microsoft |
| Application deployment | Customer | Customer | Microsoft |
| Data protection | Customer | Shared | Shared |
| Network security | Customer | Shared | Mostly Microsoft |
| Identity for users | Customer | Customer | Customer |
| Service identity | Customer | Shared | Microsoft |
| Certificates | Customer | Shared | Shared |
| Monitoring | Customer | Shared | Shared |
| Compliance evidence | Customer | Shared | Shared |

This matrix is useful for exam questions that ask who owns a control, who configures it, or who is responsible when something fails.

**Practical decision guidance**

Choose the service model based on how much operational ownership you want to keep.

- Choose **IaaS** when:
  - You need full OS and software control
  - You have legacy dependencies
  - You need custom agents, drivers, or unsupported software
  - You are migrating quickly and will modernize later

- Choose **PaaS** when:
  - You want to reduce patching and platform maintenance
  - You can adapt the application to managed service constraints
  - You want built-in scaling and easier operations
  - You need faster delivery with fewer infrastructure tasks

- Choose **SaaS** when:
  - The business process fits a standard product
  - You want minimal operational overhead
  - You can accept vendor-defined features and release cadence
  - You mainly need tenant configuration and governance

**Exam cues and common traps**

- **Azure manages the platform, not your app**: true for PaaS, false for SaaS.
- **Customer manages everything in IaaS**: mostly true above the hypervisor, but Azure still owns the physical layer.
- **Backups are automatic everywhere in PaaS**: false; service behavior varies.
- **Identity is fully Microsoft-managed in SaaS**: false; you still manage tenant users and access.
- **Monitoring is “built in” so you do not need to configure it**: false; you often need diagnostic settings, alerts, and retention.
- **Managed service means no operational responsibility**: false; it usually means reduced, not eliminated, responsibility.

**Quick reference summary**

- **IaaS**: you own the OS, runtime, app, data, and most security/ops tasks.
- **PaaS**: Azure owns the platform; you own the app, data, and configuration.
- **SaaS**: the vendor owns the app and platform; you own tenant governance, access, and data policies.
- **Hybrid**: ownership is split across environments, so document boundaries explicitly.
- **Security, backup, monitoring, and support** always need a clear ownership map before deployment.
