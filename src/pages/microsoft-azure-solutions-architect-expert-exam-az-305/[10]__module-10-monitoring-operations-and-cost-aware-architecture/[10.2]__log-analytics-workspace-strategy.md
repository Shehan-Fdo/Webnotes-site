## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Workspace strategy overview**

A Log Analytics workspace is the core Azure Monitor data store for logs, queries, alerts, and many monitoring workflows. Your strategy should balance **security boundaries**, **cost**, **regional placement**, **operational simplicity**, and **query/access patterns** rather than treating the workspace as a generic bucket for all telemetry.

In Azure architecture design, the workspace is often the decision point for:
- Where logs are stored and queried.
- Who can access which data.
- How much ingestion you pay for.
- Whether monitoring is centralized or split by environment, business unit, or compliance boundary.

A good design starts with the question: do you need one workspace for simplicity, or multiple workspaces for isolation and governance?

**Core design goals**

Use the workspace as a deliberate boundary, not an afterthought. The main goals are:
- **Isolation** for production, nonproduction, regulated, or tenant-separated environments.
- **Cost control** through ingestion filtering, table plans, retention, and commitment tiers.
- **Operational clarity** so teams know where to query, alert, and troubleshoot.
- **Data residency** alignment with regional and compliance requirements.
- **Scalable administration** for RBAC, private access, and lifecycle management.

A single workspace can simplify queries and alerting, but it can also create access sprawl and make governance harder. Multiple workspaces improve separation, but they increase management overhead and cross-workspace query complexity.

**Workspace placement and region strategy**

Workspace region matters because it affects latency, data residency, and operational design. In general, place the workspace in the same region as the resources that generate the majority of telemetry when possible. This reduces ingestion latency and keeps data in the intended geography.

Important placement considerations:
- **Region alignment** with monitored resources is usually preferred for latency and residency.
- **Compliance requirements** may require logs to remain in a specific geography.
- **Cross-region monitoring** is possible, but it can complicate design and may introduce latency or governance concerns.
- **Resource location and workspace location do not always need to match exactly**, but you should validate any service-specific constraints before standardizing.

For landing zone designs, common patterns are:
- One workspace per region per environment.
- One workspace per environment in a compliance-approved region.
- One central workspace for a small organization with limited separation needs.

> Do not assume a workspace is a dynamic routing engine. If you need different storage or access behavior, you usually create separate workspaces, separate data collection rules, or separate associations.

**Centralized vs decentralized monitoring**

The workspace model should match your operating model.

| Pattern | Strengths | Tradeoffs | Best fit |
|---|---|---|---|
| Centralized | Easier cross-resource search, simpler alerting, fewer workspaces | Broader access scope, larger blast radius, harder compliance separation | Small-to-medium orgs, shared operations teams |
| Decentralized | Stronger isolation, clearer ownership, easier compliance boundaries | More administration, more cross-workspace queries, duplicated setup | Large enterprises, regulated workloads, multi-team environments |
| Hybrid | Central visibility with selective isolation | Requires clear standards and governance | Azure landing zones with mixed requirements |

A hybrid model is common in enterprise Azure landing zones:
- Shared platform logs may go to a central workspace.
- Application teams may use separate workspaces for production workloads.
- Regulated workloads may require dedicated workspaces and restricted access.

**Ingestion architecture: agents, diagnostic settings, and DCRs**

Azure Monitor ingestion is not one mechanism. You need to understand how different sources reach the workspace.

**Azure Monitor Agent and Data Collection Rules**

The **Azure Monitor Agent (AMA)** is the modern agent for collecting guest-level logs and performance data from Azure VMs, Arc-enabled servers, and some other supported machines. It uses **Data Collection Rules (DCRs)** to define:
- What data to collect.
- Which streams to collect.
- Which destinations to send data to.
- Which machines or scopes the rule applies to.

Typical DCR concepts:
- A DCR defines collection logic.
- A **data collection endpoint** may be used in some scenarios to receive data.
- Associations bind the DCR to resources or scopes.
- DCRs are not a general-purpose router that automatically chooses a workspace based on environment or sensitivity.

Example ARM-style shape for a DCR targeting a Log Analytics workspace:

```json
{
  "type": "Microsoft.Insights/dataCollectionRules",
  "apiVersion": "2023-03-11",
  "name": "dcr-vm-logs-prod",
  "location": "eastus",
  "properties": {
    "destinations": {
      "logAnalytics": [
        {
          "name": "la-prod",
          "workspaceResourceId": "/subscriptions/00000000-0000-0000-0000-000000000000/resourceGroups/rg-monitoring/providers/Microsoft.OperationalInsights/workspaces/law-prod"
        }
      ]
    },
    "dataSources": {
      "performanceCounters": [
        {
          "name": "perf",
          "streams": [ "Microsoft-Perf" ],
          "samplingFrequencyInSeconds": 60,
          "counterSpecifiers": [ "\\Processor(_Total)\\% Processor Time" ]
        }
      ]
    },
    "dataFlows": [
      {
        "streams": [ "Microsoft-Perf" ],
        "destinations": [ "la-prod" ]
      }
    ]
  }
}
```

**Diagnostic settings**

Use **diagnostic settings** for many Azure platform resources such as Key Vault, App Service, Azure SQL, Storage, and networking services. Diagnostic settings typically send platform logs and metrics to:
- A Log Analytics workspace.
- An Event Hub.
- A Storage account.
- A partner solution.

Practical implications:
- Diagnostic settings are the standard path for many **resource logs**.
- They are configured per resource or resource scope.
- They are often the simplest way to centralize platform telemetry.

Example CLI pattern:

```bash
az monitor diagnostic-settings create \
  --name send-to-law \
  --resource /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.KeyVault/vaults/<kvName> \
  --workspace /subscriptions/<subId>/resourceGroups/<monitorRg>/providers/Microsoft.OperationalInsights/workspaces/<workspaceName> \
  --logs '[{"category":"AuditEvent","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

**Choosing between AMA and diagnostic settings**
- Use **AMA + DCRs** for guest OS telemetry, custom text logs, and flexible machine-level collection.
- Use **diagnostic settings** for Azure resource platform logs and metrics.
- Use both when you need host telemetry and platform telemetry in the same monitoring design.

**Workspace access control and RBAC**

Access design is one of the most important workspace decisions. Azure uses **role-based access control (RBAC)** to control who can read, query, configure, or manage monitoring data.

Two common access models are:
- **Workspace-centric RBAC**: grant permissions directly on the workspace.
- **Resource-centric RBAC**: grant access through the monitored resource scope.

Resource-centric access is useful when:
- A team should only see logs for its own application resources.
- You want to avoid giving broad access to the entire workspace.
- Multiple teams share a workspace but must not see each other’s data.

Workspace-centric access is useful when:
- A central operations team manages all monitoring.
- Analysts need broad query access across many resources.
- The workspace is already a deliberate shared boundary.

Example scenario:
- A platform team gets `Log Analytics Reader` on the central workspace.
- An application team gets access only to its application resource group and uses resource-centric queries for its own resources.
- A security team gets broader access to a dedicated security workspace that contains sensitive logs.

Common roles and uses:
- **Log Analytics Reader**: query and view data.
- **Log Analytics Contributor**: manage workspace settings and content.
- **Monitoring Reader**: read monitoring data across resources.
- **Monitoring Contributor**: manage monitoring configuration.

> Use the least privilege model. If a team only needs dashboards and queries for one application, do not grant full workspace-wide write access.

**Private access and network isolation**

For private connectivity, Azure Monitor uses **Azure Monitor Private Link Scope (AMPLS)** to extend private access to monitoring resources. This is different from simply enabling a private endpoint on a workspace.

Key points:
- **AMPLS** groups Azure Monitor resources that should be reachable privately.
- It helps keep ingestion and query traffic off the public internet where supported.
- It is commonly used in locked-down enterprise networks.
- Private access design may require DNS planning and careful validation of dependent services.

Design considerations:
- Private access is not just a workspace setting; it is part of a broader Azure Monitor private connectivity model.
- You may need private DNS zones and network rules for clients and agents.
- Some monitoring features may still require public endpoints or additional configuration depending on the service and region.

**Cost model and ingestion strategy**

Workspace strategy is strongly influenced by cost. Azure Monitor pricing is driven mainly by ingestion, retention, and advanced features rather than by the mere existence of a workspace.

Important cost levers:
- **Ingestion volume**: the biggest cost driver in many designs.
- **Retention**: longer retention increases cost.
- **Commitment tiers**: can reduce cost when ingestion is predictable.
- **Archive and search**: cheaper long-term storage patterns for older data.
- **Table plans**: different table types can affect queryability and cost.

Practical cost controls:
- Filter noisy logs at the source.
- Collect only required counters and event channels.
- Use separate workspaces for high-volume telemetry if governance or cost allocation requires it.
- Review retention by table, not only at the workspace level.
- Use commitment tiers when ingestion is stable enough to justify them.

Example CLI to inspect workspace properties:

```bash
az monitor log-analytics workspace show \
  --resource-group rg-monitoring \
  --workspace-name law-prod \
  --query "{location:location, sku:sku.name, retentionInDays:retentionInDays, publicNetworkAccessForIngestion:publicNetworkAccessForIngestion, publicNetworkAccessForQuery:publicNetworkAccessForQuery}"
```

**Retention, archive, and table plans**

Retention is not just a single number. You should think in terms of how data is stored, queried, and aged out.

Key concepts:
- **Hot data** is immediately queryable and used for active operations.
- **Archive** is cheaper for long-term retention but is not the same as normal interactive query storage.
- **Search and restore** patterns may be needed for older data.
- **Table plans** influence how data is billed and queried.

Common table plan concepts:
- **Analytics tables**: optimized for interactive querying and alerting.
- **Basic tables**: lower-cost storage for data you query less often and do not need for all advanced features.
- **Archive**: long-term retention for compliance or historical needs.

Design implications:
- Put high-value operational data in analytics tables.
- Use basic or archive options for lower-value, high-volume logs when supported.
- Keep security and compliance requirements in mind before moving data to lower-cost tiers.

> Table behavior affects what you can do with the data later. Lower cost often means fewer query and alerting capabilities.

**Workspace sizing, caps, and operational limits**

Workspace design should account for service limits and operational constraints, but avoid inventing arbitrary size thresholds. Azure does not give a simple universal “too large” workspace cutoff for all scenarios.

Instead, evaluate:
- Ingestion volume and daily spikes.
- Retention requirements.
- Query concurrency and operational usage.
- Access boundaries and compliance needs.
- Administrative overhead.

Useful design signals for splitting a workspace:
- Different teams need separate access control.
- Different environments require separate retention or compliance handling.
- A single workspace becomes hard to govern or troubleshoot.
- Cost allocation needs to be separated by business unit or workload.

Useful signals for consolidation:
- Multiple workspaces duplicate the same alerting and dashboard logic.
- Teams need frequent cross-workspace queries.
- The organization wants a simpler operational model.

**Lifecycle management: migration, consolidation, and splitting**

Workspace strategy changes over time. You may need to migrate, consolidate, or split workspaces as the organization grows.

Common lifecycle scenarios:
- **Consolidation**: merge multiple small workspaces into a central workspace to simplify operations.
- **Splitting**: separate a shared workspace into production and nonproduction or into business-unit-specific workspaces.
- **Migration**: move from legacy agents to AMA and DCR-based collection.
- **Replatforming**: change retention, table plans, or private access posture.

Practical migration considerations:
- Recreate diagnostic settings and DCR associations before cutover.
- Validate query parity and alert rules after moving data sources.
- Expect historical data to remain in the old workspace unless you export or reingest it.
- Plan for dashboards, workbooks, and alerts that reference old workspace IDs.

A simple migration checklist:
1. Inventory current data sources, alerts, workbooks, and queries.
2. Define the target workspace model and access boundaries.
3. Recreate collection rules and diagnostic settings.
4. Validate ingestion and query results.
5. Update automation, dashboards, and alert scopes.
6. Decommission old workspaces only after retention and compliance requirements are satisfied.

**Practical query examples**

Cross-workspace queries are useful in centralized operations, but they should be used intentionally.

Example pattern:

```kusto
union
  workspace("law-prod").Heartbeat,
  workspace("law-nonprod").Heartbeat
| summarize Count = count() by Computer, _ResourceId
| order by Count desc
```

Notes:
- Use tables that are actually present in the target workspaces.
- Not every workspace contains the same tables.
- Cross-workspace queries can become expensive and harder to maintain.

A safer operational pattern is to standardize on a small set of tables and naming conventions across workspaces so queries remain portable.

**Naming and governance conventions**

Consistent naming reduces confusion and helps automation.

Recommended conventions:
- Include environment in the workspace name, such as `law-prod-eus`.
- Include region when you operate multiple regions.
- Use a dedicated resource group for monitoring resources.
- Keep naming aligned with landing zone standards.

Governance practices:
- Tag workspaces with owner, environment, cost center, and data classification.
- Restrict who can create new workspaces.
- Define when a team must use a shared workspace versus a dedicated one.
- Review retention and ingestion settings regularly.

**Exam-oriented decision guidance**

When you see a workspace design question, ask:
- Do you need **isolation** or **simplicity**?
- Is the data subject to **regional** or **compliance** constraints?
- Will teams need **broad query access** or only scoped access?
- Is the ingestion pattern stable enough for **commitment tiers**?
- Do you need **private connectivity** for monitoring traffic?
- Are you collecting **platform logs**, **guest logs**, or both?

Typical recommendations:
- Use a **centralized workspace** for shared operations and simpler management.
- Use **multiple workspaces** when access boundaries, compliance, or business ownership require separation.
- Use **AMA + DCRs** for guest telemetry and **diagnostic settings** for Azure resource logs.
- Use **resource-centric RBAC** when teams should only see their own resources.
- Use **workspace-centric RBAC** when a central team owns the monitoring platform.

**Quick reference**

- **AMA**: modern agent for guest telemetry.
- **DCR**: defines what to collect and where to send it.
- **Diagnostic settings**: send Azure resource logs and metrics to destinations.
- **AMPLS**: Azure Monitor Private Link Scope for private monitoring access.
- **Analytics table**: interactive, higher-cost, feature-rich.
- **Basic table**: lower-cost, limited feature set.
- **Archive**: long-term retention for older data.

**Common pitfalls**

- Treating one workspace as a universal routing layer.
- Using a single workspace without considering access boundaries.
- Ignoring region and residency requirements.
- Sending all logs to the workspace without filtering noise.
- Forgetting to update alerts and workbooks during migration.
- Assuming all tables are equally queryable or equally priced.
- Designing private access without planning DNS and network dependencies.

**Summary**

A strong Log Analytics workspace strategy is a governance and economics decision as much as a technical one. Choose the number of workspaces based on access, compliance, region, and operational ownership. Use the right ingestion path for each telemetry source, control cost with retention and table planning, and design for lifecycle changes over time.
