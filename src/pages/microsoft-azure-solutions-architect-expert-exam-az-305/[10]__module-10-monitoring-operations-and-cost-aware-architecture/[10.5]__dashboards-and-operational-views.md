## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Operational views in Azure**

Operational views help you observe the health, performance, and availability of your platform. In Azure, you usually combine several tools rather than rely on one pane of glass. The main choices are Azure portal dashboards, Azure Monitor Workbooks, Azure Managed Grafana, Azure Monitor Logs, Application Insights, and health services such as Azure Service Health and Resource Health.

A good AZ-305 answer explains not just *what* each tool does, but *where* it fits:

- **Dashboards** give a lightweight, shareable portal landing page.
- **Workbooks** provide interactive, parameterized analysis and reporting.
- **Grafana** is best when you need rich time-series visualization and an existing Grafana operating model.
- **Logs and Application Insights** provide the underlying telemetry and query layer.
- **Service Health** and **Resource Health** explain platform and resource incidents that are not always visible in your own telemetry.

**Reference architecture for operational visibility**

```text
Users / Operators
      |
      v
+-------------------+      +-------------------+
| Azure Portal      |      | Azure Managed     |
| Dashboard         |      | Grafana           |
+-------------------+      +-------------------+
      |                           |
      | tiles / links             | dashboards / panels
      v                           v
+-------------------+      +-------------------+
| Workbooks         |<---->| Azure Monitor     |
| (interactive)     |      | Logs / Metrics    |
+-------------------+      +-------------------+
      |                           |
      v                           v
+-------------------+      +-------------------+
| Application       |      | Resource Health   |
| Insights          |      | Service Health    |
+-------------------+      +-------------------+
```

**Azure portal dashboards**

Azure portal dashboards are curated collections of tiles that you pin from portal experiences. They are useful for a shared operational landing page, especially for NOC-style monitoring, executive summaries, or a team’s daily start page.

Use dashboards when you want:

- A simple, low-friction overview.
- Quick access to common resources, metrics, and links.
- A portal-native experience for broad audiences.
- A shareable view that is easy to consume without learning a query language.

Dashboards are not the best choice for deep analysis. They are static compared with Workbooks, and they are not a replacement for a log explorer or a metrics investigation workflow.

Important distinctions:

- You can pin **metric charts**, **workbook visuals**, and some portal blades or links.
- Dashboards are managed in the Azure portal and can be shared through Azure RBAC.
- Dashboards are best for curated summaries, not ad hoc investigation.
- Portal dashboards do not act as a general-purpose log viewer tile; logs are usually explored in Logs or surfaced through workbook visuals.

**When to choose a portal dashboard**

- You need a team landing page with a few critical KPIs.
- You want to expose a curated operational summary to service owners.
- You need a lightweight view for managers or on-call staff.
- You want to combine links, charts, and status tiles in one place.

**Azure Monitor Workbooks**

Workbooks are interactive reports built on Azure Monitor data sources. They are the strongest choice when you need parameterized, drillable, and reusable operational analysis. Workbooks can query logs, metrics, Application Insights, Resource Graph, and other supported sources.

Use Workbooks when you need:

- Interactive filtering with parameters such as subscription, resource group, region, or time range.
- Drill-down from summary to detail.
- A repeatable investigation or reporting template.
- A richer operational report than a dashboard tile can provide.

Workbooks are especially useful for:

- Incident triage.
- Service health summaries.
- Capacity and performance analysis.
- Application and infrastructure correlation.
- Security or compliance reporting where query logic matters.

**Workbook creation and sharing**

Workbooks are created and managed in Azure Monitor or from the resource context in the portal. You can save them, share them, and export them as JSON for reuse. In practice, teams often store workbook definitions in source control and deploy them as part of infrastructure-as-code or operational content management.

Common sharing and access patterns:

- Viewers need permission to the underlying data source, such as a Log Analytics workspace or Application Insights resource.
- Editors need rights to modify the workbook resource and often the data source.
- Workbooks can be shared by link or saved in a shared location, depending on the deployment model.
- Access is governed by Azure RBAC and the permissions on the queried resources.

**Workbook example with parameters**

```json
{
  "version": "Notebook/1.0",
  "items": [
    {
      "type": 9,
      "content": {
        "version": "KqlItem/1.0",
        "query": "AzureActivity | where TimeGenerated > ago(24h) | summarize Count=count() by ActivityStatusValue",
        "size": 0,
        "title": "Activity status summary",
        "queryType": 0,
        "resourceType": "microsoft.operationalinsights/workspaces"
      }
    }
  ]
}
```

This example is intentionally simple. In real workbooks, you usually add parameters, multiple sections, and drill-through links. The exact schema varies by workbook version and visual type.

**Azure Monitor Logs, Log Analytics, and Application Insights**

Azure Monitor Logs is the query experience over data stored in Log Analytics workspaces. Kusto Query Language (KQL) is the query language used to analyze that data. A Log Analytics workspace is the storage and query boundary for many Azure Monitor logs.

Application Insights is the application performance monitoring service. In modern deployments, it is often workspace-based, which means telemetry is stored in a Log Analytics workspace and queried with KQL. This matters because the schema depends on whether you are using:

- **Classic Application Insights**: older standalone schema and tables.
- **Workspace-based Application Insights**: telemetry lands in Log Analytics tables and integrates more directly with Azure Monitor Logs.

Operationally, this means:

- Use Application Insights for app-centric telemetry such as requests, dependencies, exceptions, and availability tests.
- Use Log Analytics for broader platform, infrastructure, and cross-resource telemetry.
- Use Workbooks to combine both app and infrastructure views in one report when the data is available in the same workspace or query scope.

**KQL example for workspace-based app telemetry**

```kusto
requests
| where timestamp > ago(24h)
| summarize Requests=count(), AvgDurationMs=avg(duration) by bin(timestamp, 1h)
| order by timestamp asc
```

This pattern is common in workspace-based Application Insights, but table names and columns can differ depending on the schema and ingestion path. Always validate the exact table set in your workspace.

**Azure Managed Grafana**

Azure Managed Grafana is a managed Grafana service for teams that already use Grafana-style dashboards and time-series visualization. It is useful when you want Grafana’s visualization model without operating the Grafana server yourself.

Use Azure Managed Grafana when:

- Your operations team already knows Grafana.
- You need rich time-series dashboards and alerting workflows.
- You want to integrate Azure data sources with Grafana panels.
- You prefer a managed service over self-hosting.

Be precise about capability:

- Azure Managed Grafana supports a curated set of plugins and data sources.
- It does not imply arbitrary plugin parity with self-hosted Grafana.
- Authentication and authorization are integrated with Azure identity and workspace access patterns.

**Authentication and authorization for Grafana**

Typical enterprise setup:

- Use Microsoft Entra ID for user authentication.
- Assign users or groups access to the Grafana workspace.
- Grant Grafana data source permissions to the underlying Azure resources.
- Use managed identity or service principals where supported for data access.

A common design is to let operators authenticate with Entra ID, then authorize them through Azure RBAC and Grafana workspace roles. This reduces credential sprawl and keeps access aligned with Azure governance.

**When to choose Grafana**

- You need advanced time-series dashboards for SRE or platform teams.
- You already standardize on Grafana across clouds.
- You want a unified visualization layer over Azure Monitor and other supported sources.
- You need panel layouts and alerting patterns that match existing Grafana operations.

**Azure Monitor metrics versus logs**

Metrics and logs serve different operational needs.

| Aspect | Metrics | Logs |
|---|---|---|
| Granularity | Aggregated, near real time | Detailed, event-level |
| Retention | Often shorter and cheaper for high-frequency data | Longer and more flexible, but query cost matters |
| Best for | Alerting on thresholds, health, performance trends | Investigation, correlation, root cause analysis |
| Query style | Metric explorer, alerts, dashboards | KQL in Logs and Workbooks |

Use metrics when you need fast alerting on numeric signals such as CPU, request count, or latency. Use logs when you need to explain *why* something happened.

**Health views: Service Health and Resource Health**

Azure Service Health and Resource Health are essential operational views because they explain issues that may not appear in your application telemetry.

- **Azure Service Health** reports Azure platform incidents, planned maintenance, and health advisories that affect your subscriptions and regions.
- **Resource Health** shows the health state of a specific Azure resource and helps distinguish between platform issues and customer-side issues.

Use these views when:

- An application is unavailable but your own logs do not show an obvious failure.
- You need to determine whether Azure had a regional incident.
- You want to correlate an outage with a platform event or maintenance window.
- You need a support-ready timeline for an incident.

**Operational workflow with health views**

1. Check **Resource Health** for the affected VM, App Service, database, or other resource.
2. Check **Service Health** for regional incidents, advisories, or maintenance.
3. Correlate with **Logs** and **Application Insights** to identify customer-side symptoms.
4. Use a **Workbook** to summarize the incident timeline and impacted resources.
5. Pin the most relevant summary to a **dashboard** for the operations team.

**Alerts in operational views**

Alerts are part of the operational workflow, not a separate island. You typically use alerts to detect issues, then use dashboards, workbooks, and Grafana to investigate and communicate.

Common patterns:

- Pin an alert summary tile to a dashboard for at-a-glance status.
- Use a workbook to show active alerts, affected resources, and recent changes.
- Link alert rules to action groups for email, SMS, push, webhook, or ITSM integration.
- Use alert rule management views to review thresholds, scopes, and suppression settings.

**Action groups**

Action groups define what happens when an alert fires. They are central to operational response.

Typical actions include:

- Email notifications.
- SMS or voice calls.
- Azure app push notifications.
- Webhooks.
- Logic Apps or automation runbooks.
- IT service management integrations.

A practical design is to route critical alerts to an on-call action group and lower-severity alerts to a ticketing or reporting workflow.

**Example alert rule pattern**

```bash
az monitor metrics alert create \
  --name "vm-cpu-high" \
  --resource-group rg-ops \
  --scopes /subscriptions/<subId>/resourceGroups/rg-ops/providers/Microsoft.Compute/virtualMachines/vm01 \
  --condition "avg Percentage CPU > 80" \
  --window-size 5m \
  --evaluation-frequency 1m \
  --action /subscriptions/<subId>/resourceGroups/rg-ops/providers/microsoft.insights/actionGroups/ag-oncall
```

This is a representative pattern for metric alerts. Exact syntax and supported conditions vary by resource type and alert type.

**KQL alert example for logs**

```kusto
AzureActivity
| where TimeGenerated > ago(15m)
| where ActivityStatusValue in ("Failed", "Started", "Accepted")
| summarize Count=count() by OperationNameValue, ActivityStatusValue
| where Count > 0
```

This is only a starting point. In practice, you scope the query to the resource, subscription, or operation you care about, and you validate the exact schema in your workspace. For failure detection, the field names and values can vary by table and ingestion path.

**Dashboards versus Workbooks versus Grafana**

| Capability | Azure portal dashboard | Azure Monitor Workbook | Azure Managed Grafana |
|---|---|---|---|
| Primary purpose | Curated overview | Interactive analysis | Rich visualization and time-series ops |
| Best for | Landing page, summary | Investigation, reporting | SRE-style monitoring |
| Data interaction | Limited | Strong | Strong |
| Parameters | Limited | Strong | Moderate to strong |
| Sharing | Portal sharing and RBAC | Share/export/import, RBAC | Workspace and Entra ID based |
| Query support | Indirect via pinned visuals | KQL, metrics, Resource Graph, more | Data source dependent |
| Operational depth | Low to medium | High | High |

**Exam decision guidance**

Choose the tool that matches the scenario:

- If the requirement is a **simple team homepage**, choose a portal dashboard.
- If the requirement is **interactive filtering and drill-down**, choose a workbook.
- If the requirement is **existing Grafana skills, time-series panels, or cross-platform visualization**, choose Azure Managed Grafana.
- If the requirement is **platform incident context**, include Service Health and Resource Health.
- If the requirement is **application telemetry analysis**, use Application Insights and Logs, then surface the result in a workbook or dashboard.

**Scenario examples for AZ-305**

- A global enterprise wants a read-only landing page for executives showing service status, top alerts, and a few KPIs. Use a **portal dashboard** with pinned charts and links.
- An operations team needs to investigate latency spikes by region, subscription, and deployment slot. Use a **Workbook** with parameters and KQL queries.
- A platform team already uses Grafana for Kubernetes and VM observability and wants Azure data in the same visual language. Use **Azure Managed Grafana** with approved Azure data sources.
- A SaaS application has intermittent outages, but app logs show no obvious errors. Check **Resource Health** and **Service Health** before assuming the issue is in the application.
- A support engineer needs a repeatable incident report that combines failed deployments, activity log events, and app exceptions. Build a **Workbook** and optionally pin summary visuals to a dashboard.

**Access control and governance**

Operational views are only useful if the right people can see them.

Common Azure RBAC considerations:

- Viewers need read access to the dashboard or workbook and the underlying data.
- Editors need permissions to modify the artifact and often to query the source.
- Workspace access should be scoped to the minimum required subscription, resource group, or workspace.
- Separate operational read access from administrative write access where possible.

Practical role patterns:

- Use **Reader** for broad dashboard consumption when the data source permissions are already granted.
- Use **Monitoring Reader** for users who need to view monitoring data without changing resources.
- Use **Monitoring Contributor** for users who create or edit monitoring artifacts.
- Use resource-specific roles where a service team owns only one workspace or application.

**Operational design patterns**

A strong Azure operations design usually layers the tools:

- **Alerts** detect the issue.
- **Logs and Application Insights** explain the issue.
- **Workbooks** summarize and correlate the issue.
- **Dashboards** broadcast the status.
- **Grafana** provides a specialized visualization layer when needed.
- **Service Health** and **Resource Health** confirm whether Azure itself is involved.

This layered approach is more resilient than trying to force one tool to do everything.

**Practical implementation tips**

- Keep dashboards small and focused; too many tiles reduce usefulness.
- Use workbook parameters to avoid duplicating reports for each subscription or environment.
- Standardize alert naming so dashboard tiles and workbook sections are easy to interpret.
- Validate table schemas before writing KQL for alerts or reports.
- Use managed identity or Entra ID-based access where possible instead of shared secrets.
- Store workbook and dashboard definitions in source control when you need repeatable deployment.

**Common pitfalls**

- Treating dashboards as a replacement for log analysis.
- Assuming Application Insights always uses the same schema; workspace-based and classic deployments differ.
- Using Grafana as if it supports every self-hosted plugin; managed service support is curated.
- Forgetting to grant access to the underlying workspace or resource, which makes the dashboard appear broken.
- Ignoring Service Health and Resource Health during incident triage.
- Building alert queries without validating the exact table and field names in the target workspace.

**Quick comparison of operational use cases**

- **Dashboard**: “Show me the current state.”
- **Workbook**: “Let me investigate and explain the state.”
- **Grafana**: “Let me visualize time-series data in a familiar ops tool.”
- **Logs/Application Insights**: “Let me query the raw telemetry.”
- **Service Health/Resource Health**: “Let me determine whether Azure is part of the problem.”

**Minimal workbook-style KQL example**

```kusto
AzureActivity
| where TimeGenerated > ago(24h)
| summarize Events=count() by bin(TimeGenerated, 1h), Category
| order by TimeGenerated asc
```

This kind of query is useful for trend summaries in a workbook. For production use, you usually add filters for subscription, resource group, or operation name.

**Minimal dashboard tile strategy**

- Pin a CPU or request-rate metric chart for fast status.
- Pin a workbook visual for incident context.
- Add a link tile to Logs for deeper investigation.
- Add a link tile to Service Health for platform status.

**What AZ-305 expects you to know**

You should be able to explain:

- The difference between dashboards, Workbooks, and Grafana.
- How Azure Monitor Logs, Log Analytics, and Application Insights relate.
- Why Service Health and Resource Health matter in operations.
- How alerts and action groups fit into the operational workflow.
- How access control affects who can view or edit operational views.
- Which tool to choose for a given enterprise scenario.
