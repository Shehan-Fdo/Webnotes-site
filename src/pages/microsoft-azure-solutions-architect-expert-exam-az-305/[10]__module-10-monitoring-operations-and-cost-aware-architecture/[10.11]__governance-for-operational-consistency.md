## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Governance for operational consistency**

Operational consistency means you design Azure so that workloads are deployed, secured, monitored, and changed in a predictable way across subscriptions, resource groups, and regions. In practice, this is a governance problem: you use management groups, Azure Policy, RBAC (role-based access control), locks, monitoring standards, and infrastructure as code to reduce drift and enforce baseline behavior.

A strong design balances control and agility. You want platform teams to define guardrails, while application teams can still deploy within approved boundaries. For AZ-305, focus on how governance choices affect scale, compliance, and day-to-day operations.

**Core governance model**

Azure governance is usually layered from the top down:

- **Management groups** organize subscriptions and apply inherited governance at scale.
- **Subscriptions** provide billing and administrative boundaries.
- **Resource groups** group lifecycle-related resources for deployment and management.
- **Resources** inherit many controls from higher scopes, but not all settings behave the same way.

This hierarchy matters because many governance controls are scope-aware. A policy assigned at a management group can affect all child subscriptions, but a lock on a resource group does not automatically protect unrelated resources elsewhere. RBAC assignments also inherit downward, which is useful for delegation but can create over-permission if you assign too broadly.

**Management group strategy**

Management groups are the preferred place to apply enterprise-wide governance. They let you separate platform and application concerns and keep policy and RBAC consistent across many subscriptions.

Common landing zone patterns include:

- **Platform subscriptions** for shared services such as identity, connectivity, and management.
- **Connectivity subscription** for hub networking, firewalls, DNS, and shared routing.
- **Identity subscription** for directory-related supporting services, if needed.
- **Management subscription** for monitoring, automation, and security tooling.
- **Landing zone subscriptions** for application teams and workloads.

This separation helps you apply different controls to different layers of the environment. For example, platform subscriptions often have stricter change control, while workload subscriptions can be more self-service within policy guardrails.

> Management groups are not just folders. They are a governance boundary that can carry policy and RBAC inheritance to all child subscriptions.

**Scope and inheritance nuances**

Use the right scope for the control you want to enforce:

- **Management group scope**
  - Best for enterprise standards, allowed regions, tagging rules, and baseline security controls.
  - Useful when you want one policy assignment to cover many subscriptions.
- **Subscription scope**
  - Best for subscription-specific exceptions, billing boundaries, or workload-specific guardrails.
  - Useful when different teams own different subscriptions.
- **Resource group scope**
  - Best for lifecycle-specific controls, such as protecting a production app group from accidental deletion.
- **Resource scope**
  - Best for a single resource exception or a targeted lock, policy exemption, or RBAC assignment.

Inheritance is powerful, but not absolute:

- RBAC assignments inherit from parent scopes unless denied by lack of permission or constrained by a custom role.
- Policy assignments inherit downward, but exemptions can be used to document approved exceptions.
- Locks apply only to the scope where they are placed and below that scope, depending on lock type and resource behavior.
- Some resource providers or child resources may not behave exactly like parent scope assumptions, so validate critical controls in design reviews.

**Azure Policy for guardrails**

Azure Policy enforces organizational rules and evaluates compliance. It is the main tool for keeping resources consistent at scale.

Policy concepts to know:

- **Policy definition**: a single rule, such as “only allow approved regions.”
- **Initiative** or **policy set definition**: a collection of related policy definitions.
- **Assignment**: applies a policy or initiative to a scope.
- **Parameters**: make a policy reusable, such as passing allowed locations.
- **Effect**: determines what happens when a rule matches or fails.

Common effects include:

- **deny**: blocks noncompliant requests.
- **audit**: allows the request but records noncompliance.
- **append**: adds supported request properties when the alias exists.
- **modify**: changes request content or adds tags/fields when supported.
- **deployIfNotExists**: deploys a related resource if it is missing.
- **auditIfNotExists**: audits when a related resource is missing.
- **denyAction**: blocks specific actions on resources, useful for certain operational controls.

Use `append` when you want to add a property to a request without changing existing values. Use `modify` when you need to update or add supported fields more flexibly, such as tags or location-related metadata. They are different tools, not interchangeable.

A simple allowed-locations policy pattern looks like this:

```json
{
  "mode": "All",
  "parameters": {
    "listOfAllowedLocations": {
      "type": "Array",
      "metadata": {
        "displayName": "Allowed locations"
      }
    }
  },
  "policyRule": {
    "if": {
      "not": {
        "field": "location",
        "in": "[parameters('listOfAllowedLocations')]"
      }
    },
    "then": {
      "effect": "deny"
    }
  }
}
```

A tag enforcement pattern often uses `modify` or `append` depending on the requirement:

```json
{
  "mode": "Indexed",
  "policyRule": {
    "if": {
      "field": "tags['CostCenter']",
      "exists": "false"
    },
    "then": {
      "effect": "modify",
      "details": {
        "operations": [
          {
            "operation": "addOrReplace",
            "field": "tags['CostCenter']",
            "value": "0000"
          }
        ]
      }
    }
  }
}
```

**DeployIfNotExists and auditIfNotExists**

These effects are important for operational consistency because they let you enforce supporting configuration, not just block bad requests.

Typical use cases:

- Deploy diagnostic settings when a resource is created.
- Ensure a Log Analytics workspace is linked.
- Ensure encryption, backup, or monitoring dependencies exist.

A valid `deployIfNotExists` policy usually needs more than a simple existence check. It commonly includes:

- `roleDefinitionIds` so the policy assignment identity can deploy resources.
- A `deployment` template that creates the missing resource.
- An `existenceCondition` that checks the exact expected configuration.
- A clear `name` or resource identifier for the related resource.

Example pattern for diagnostic settings governance:

```json
{
  "mode": "Indexed",
  "policyRule": {
    "if": {
      "field": "type",
      "equals": "Microsoft.KeyVault/vaults"
    },
    "then": {
      "effect": "deployIfNotExists",
      "details": {
        "type": "Microsoft.Insights/diagnosticSettings",
        "existenceCondition": {
          "allOf": [
            {
              "field": "Microsoft.Insights/diagnosticSettings/workspaceId",
              "exists": "true"
            }
          ]
        },
        "roleDefinitionIds": [
          "/providers/Microsoft.Authorization/roleDefinitions/749f88d5-cbae-40b8-bcfc-e573ddc772fa"
        ],
        "deployment": {
          "properties": {
            "mode": "incremental",
            "template": {
              "$schema": "https://schema.management.azure.com/schemas/2019-04-01/deploymentTemplate.json#",
              "contentVersion": "1.0.0.0",
              "resources": []
            }
          }
        }
      }
    }
  }
}
```

This is only a pattern. In real designs, the existence condition should verify the correct diagnostic setting name, categories, and destination, not just that some workspace ID exists.

**Policy assignments, exemptions, and compliance**

Policy assignments are where governance becomes operational. When you assign a policy, you can also configure:

- **Parameters** to tailor the rule to a scope.
- **Non-compliance messages** to explain why a request or resource is noncompliant.
- **Managed identity** for remediation when the policy needs to deploy or modify resources.
- **Exemptions** for approved exceptions with expiration or justification.

Exemptions are important because they let you preserve governance without breaking legitimate business needs. Use them sparingly and document the reason, owner, and review date.

Compliance reporting is a major part of policy operations:

- **Compliant** means the resource or scope meets the policy rule.
- **Non-compliant** means the resource violates the rule or has not yet been remediated.
- **Conflict** can occur when policies overlap or one policy contradicts another.
- **Not started** or **unknown** may appear before evaluation completes.

The compliance dashboard helps you answer:

- Which subscriptions are drifting from baseline?
- Which policies are causing the most violations?
- Which resources need remediation?
- Which exemptions are still valid?

For exam scenarios, remember that policy compliance is not the same as security incident response. Policy tells you whether a resource matches the rule; it does not automatically mean the workload is safe.

**Remediation operations**

Remediation is the process of bringing existing resources into compliance. It is commonly used with `deployIfNotExists` and `modify`.

Operational points:

- The policy assignment identity needs the right permissions.
- The remediation task runs at a defined scope.
- You often remediate after assigning a policy to existing resources.
- Some resources may require manual intervention if the policy cannot safely modify them.

A typical Azure CLI flow is to assign a policy, then create a remediation task at the correct scope. The exact command syntax can vary by CLI version and policy type, so verify the current Azure CLI help before execution.

```bash
az policy assignment list --scope /subscriptions/<subscription-id>
az policy state list --resource /subscriptions/<subscription-id>/resourceGroups/<rg-name>
az policy remediation list --scope /subscriptions/<subscription-id>
```

**RBAC for operational consistency**

RBAC controls who can do what. It complements policy by controlling access to governance operations, remediation, monitoring, and platform administration.

Key RBAC concepts:

- **Built-in roles** cover common tasks and are preferred when they fit the need.
- **Custom roles** are used when built-in roles are too broad or too narrow.
- **Least privilege** means granting only the actions required.
- **Scope** determines where the role applies.

Useful built-in role examples:

- **Owner**: full access, including role assignment; use sparingly.
- **Contributor**: can manage resources but cannot grant access.
- **Reader**: can view resources but not change them.
- **Policy Contributor**: can manage policy definitions and assignments.
- **Resource Policy Contributor**: can manage policy resources in some scenarios.
- **Monitoring Contributor**: can manage monitoring settings and alerts.
- **Log Analytics Contributor**: can manage workspaces and related settings.
- **User Access Administrator**: can manage RBAC assignments.

For governance operations, prefer narrow assignments:

- Give policy authors **Policy Contributor** at the management group or subscription scope where they manage policy.
- Give remediation identities only the permissions needed to deploy the specific resource type.
- Give monitoring teams **Monitoring Contributor** or a custom role that allows alert and action group management without full resource control.
- Avoid using **Owner** for automation unless there is no practical alternative.

A custom role example for a remediation identity might include only the actions needed to create diagnostic settings and read target resources:

```json
{
  "Name": "Diagnostic Settings Remediator",
  "IsCustom": true,
  "Description": "Can read resources and create diagnostic settings.",
  "Actions": [
    "Microsoft.Insights/diagnosticSettings/*",
    "Microsoft.Resources/subscriptions/resourceGroups/read",
    "Microsoft.Resources/subscriptions/resourceGroups/resources/read"
  ],
  "NotActions": [],
  "AssignableScopes": [
    "/subscriptions/<subscription-id>"
  ]
}
```

**Resource locks**

Resource locks prevent accidental deletion or modification. They are a simple but effective governance control for operational consistency.

Lock types:

- **CanNotDelete**: users can read and modify, but not delete.
- **ReadOnly**: users can read, but cannot modify or delete.

Use locks for:

- Shared networking components.
- Key vaults or critical platform resources.
- Production resource groups that should not be deleted accidentally.
- Resources that are expensive or difficult to recreate.

Be careful with locks because they can block legitimate operations, including deployments, updates, and remediation. Always test lock placement in a nonproduction environment first.

**Azure Monitor governance**

Monitoring is part of governance because it standardizes how you detect and respond to operational issues. A consistent monitoring design reduces alert noise and improves incident response.

Core components:

- **Metrics** for near-real-time numeric signals.
- **Logs** for detailed event and diagnostic data.
- **Log Analytics workspace** as the central log store and query engine.
- **Alert rules** for proactive notification or automation.
- **Action groups** for routing alerts to email, SMS, webhook, ITSM, or automation.
- **Diagnostic settings** for sending resource logs and metrics to destinations.

Governance decisions to make:

- Which resources must send diagnostics to the central workspace?
- Which logs are retained locally versus centrally?
- Which teams own alert rules and action groups?
- Which alerts are standardized across all subscriptions?

A practical monitoring standard often includes:

- One or more central Log Analytics workspaces per region or environment.
- Shared action groups for production incidents.
- Naming conventions for alerts and action groups.
- Tagging for ownership and cost center.
- Policy enforcement for diagnostic settings on critical resource types.

Example diagnostic settings CLI pattern, with the important caveat that categories and destinations vary by resource:

```bash
az monitor diagnostic-settings create \
  --name diag-to-law \
  --resource /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.KeyVault/vaults/<kv-name> \
  --workspace /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.OperationalInsights/workspaces/<law-name> \
  --logs '[{"category":"AuditEvent","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

Use the current CLI help for the target resource because diagnostic categories differ by service and some resources require additional parameters.

**Alert processing and action groups**

Alert processing rules help you suppress, route, or modify alerts so operations teams receive useful signals instead of noise.

Use them to:

- Suppress alerts during planned maintenance.
- Route alerts by environment or severity.
- Reduce duplicate notifications.
- Apply consistent handling for recurring operational windows.

Action group design should be standardized:

- Use shared action groups for critical production alerts.
- Separate action groups by severity or team ownership.
- Include automation hooks only when the runbook is safe and idempotent.
- Keep contact methods current and tested.

**Azure Resource Graph for governance queries**

Azure Resource Graph helps you query resources across subscriptions quickly. It is useful for governance reporting, drift detection, and inventory checks.

Example: find resources missing an `Owner` tag safely.

```kusto
Resources
| where isnull(tags) or isnull(tags['Owner']) or tostring(tags['Owner']) == ''
| project name, type, resourceGroup, subscriptionId, tags
```

Example: find resources outside approved regions.

```kusto
Resources
| where location !in~ ('eastus', 'eastus2', 'centralus')
| project name, type, location, subscriptionId
```

Use Resource Graph when you need cross-subscription visibility without querying each subscription separately.

**Azure Arc and hybrid governance**

Azure Arc extends governance patterns to servers, Kubernetes, and some data services outside native Azure. This matters when operational consistency must span on-premises, edge, and multi-cloud environments.

Use Arc when you need:

- Central policy and inventory for non-Azure servers.
- Consistent tagging and compliance reporting across hybrid estates.
- Monitoring and configuration management for distributed workloads.
- A single governance model for both Azure and connected resources.

Arc does not make non-Azure resources identical to native Azure resources, but it helps you apply similar operational standards.

**Azure Blueprints and current guidance**

Azure Blueprints has been deprecated for new designs and is not recommended for new governance architectures. It may still exist in some environments, but for exam purposes you should prefer management groups, Azure Policy, RBAC, templates, and landing zone patterns.

Use this as the design takeaway:

- Prefer **management groups + policy + RBAC + IaC** for new implementations.
- Avoid introducing Blueprints into new architectures unless you are maintaining an existing environment that already depends on them.

**Infrastructure as code for consistency**

Infrastructure as code (IaC) is essential for repeatable governance. It ensures that the same baseline is deployed every time.

Common tools:

- **Bicep** for native Azure declarative deployments.
- **ARM templates** for JSON-based deployments.
- **Terraform** when your organization standardizes on multi-cloud IaC.

A Bicep file that creates a resource group must target the subscription scope, not the resource group scope:

```bicep
targetScope = 'subscription'

param rgName string
param location string = 'eastus'

resource rg 'Microsoft.Resources/resourceGroups@2022-09-01' = {
  name: rgName
  location: location
}
```

This is important because resource groups are subscription-level resources. If you try to model them as ordinary resource-group-scoped resources, the deployment will be invalid.

Use IaC to standardize:

- Resource group naming.
- Tags.
- Diagnostic settings.
- Locks.
- Role assignments.
- Policy assignments where appropriate.

**End-to-end governance design pattern**

A practical enterprise design often combines several controls:

1. Create management groups for platform and landing zone subscriptions.
2. Assign baseline policies at the management group level.
3. Use RBAC to delegate policy management and workload administration separately.
4. Apply resource locks to critical shared services.
5. Enforce diagnostic settings and tag standards with policy.
6. Centralize logs in Log Analytics and standardize action groups.
7. Use IaC to deploy repeatable environments.
8. Use Azure Resource Graph and policy compliance reports to detect drift.
9. Extend the same model to hybrid resources with Azure Arc where needed.

This layered approach gives you operational consistency without making every team use the same permissions or deployment path.

**Exam-focused design tradeoffs**

When choosing a governance approach, ask:

- Do you need prevention or detection?
  - Use **deny** for prevention.
  - Use **audit** or monitoring for detection.
- Do you need one-time enforcement or ongoing compliance?
  - Use policy for ongoing enforcement.
  - Use IaC for repeatable deployment.
- Do you need broad standards or local exceptions?
  - Use management group assignments for standards.
  - Use exemptions for approved exceptions.
- Do you need to protect against deletion?
  - Use locks.
- Do you need to control who can change governance?
  - Use RBAC with least privilege.

**Common pitfalls**

- Assigning policy at too low a scope and then duplicating it across subscriptions.
- Using overly broad RBAC roles for automation.
- Relying on tags alone without policy enforcement.
- Forgetting remediation permissions for `deployIfNotExists` or `modify`.
- Treating monitoring as an afterthought instead of a governance control.
- Using locks so aggressively that they block legitimate operations.
- Designing diagnostic settings without considering destination, categories, and retention.
- Assuming compliance equals security.

**Quick reference**

| Control | Primary purpose | Best scope |
|---|---|---|
| Management groups | Inheritance and enterprise structure | Tenant hierarchy |
| Azure Policy | Enforce standards and compliance | MG, subscription, RG |
| RBAC | Control who can manage what | Any scope |
| Resource locks | Prevent deletion or modification | RG, resource |
| Azure Monitor | Detect and respond to issues | Subscription, RG, resource |
| IaC | Repeatable deployment | Source-controlled pipeline |
| Azure Arc | Extend governance to hybrid | Connected machines/clusters |

**Practical CLI examples**

List management groups:

```bash
az account management-group list
```

Assign a policy at a management group scope:

```bash
az policy assignment create \
  --name allowed-locations \
  --display-name "Allowed locations" \
  --scope /providers/Microsoft.Management/managementGroups/<mg-id> \
  --policy <policy-definition-id> \
  --params '{"listOfAllowedLocations":{"value":["eastus","eastus2"]}}'
```

> Verify the current CLI syntax for your policy type and Azure CLI version before running assignments in production.

Create a resource lock:

```bash
az lock create \
  --name protect-prod-rg \
  --lock-type CanNotDelete \
  --resource-group <rg-name>
```

Query policy compliance:

```bash
az policy state summarize --scope /subscriptions/<subscription-id>
```

Query resources with Azure Resource Graph:

```bash
az graph query -q "Resources | where location =~ 'eastus' | project name, type, resourceGroup"
```

**Key takeaways**

- Use **management groups** to apply governance consistently across many subscriptions.
- Use **Azure Policy** to enforce standards, detect drift, and remediate supported settings.
- Use **RBAC** to keep governance administration least-privileged.
- Use **resource locks** to protect critical resources from accidental change.
- Use **Azure Monitor** and standardized action groups to make operations predictable.
- Use **IaC** to deploy the same baseline every time.
- Extend governance to hybrid environments with **Azure Arc** when needed.
