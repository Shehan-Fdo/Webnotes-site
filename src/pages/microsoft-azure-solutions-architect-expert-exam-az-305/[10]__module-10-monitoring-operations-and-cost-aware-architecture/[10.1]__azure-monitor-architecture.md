## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Azure Monitor overview**

Azure Monitor is Azure’s unified observability platform. It collects telemetry from Azure resources, guest operating systems, applications, and custom sources, then supports analysis, alerting, visualization, and automation. For architecture decisions, the key question is not just “how do I monitor?” but “which telemetry type should I use, where does it land, how long is it retained, and how do I secure access to it?”

Azure Monitor is best understood as a set of services and data paths rather than a single database. The main architectural building blocks are:

- **Platform metrics** for near-real-time numeric measurements.
- **Platform logs** for event and diagnostic records from Azure services.
- **Activity logs** for subscription-level control-plane events.
- **Resource logs** for service-specific operational data.
- **Guest and application telemetry** from agents, SDKs, and OpenTelemetry-based instrumentation.

**Core telemetry types and where they go**

Azure Monitor uses different ingestion and storage paths depending on the telemetry source and purpose. This distinction matters for design, cost, retention, and query performance.

| Telemetry type | Typical source | Primary storage/query path | Common use |
|---|---|---|---|
| Platform metrics | Azure resources | Azure Monitor metrics store | Fast charts, alerting, autoscale |
| Activity log | Azure subscription | Azure Activity Log, export to Log Analytics/Event Hub/Storage | Governance, auditing, incident review |
| Resource logs | Azure services via diagnostic settings | Log Analytics, Event Hub, Storage, partner tools | Troubleshooting, compliance, SIEM integration |
| Guest telemetry | VM agents, AMA, custom logs | Log Analytics workspace | OS-level monitoring, log analytics |
| Application telemetry | App Insights SDK/OpenTelemetry | Application Insights backend and/or Log Analytics workspace | Request tracing, dependencies, exceptions |

Platform metrics are optimized for low-latency numeric time series. They are stored in the Azure Monitor metrics store, not in a user-managed database. Use them when you need fast alerting, dashboards, and short-interval trend analysis.

Logs are better for rich, structured, or semi-structured records. They typically land in a Log Analytics workspace, where you query them with Kusto Query Language (KQL). Logs are the right choice when you need correlation, filtering, joins, and longer investigative workflows.

> Azure Monitor does not automatically centralize every telemetry type into one place. You design the flow: source, collection method, destination, retention, and access model.

**Architecture components**

Azure Monitor architecture usually includes these layers:

- **Data sources**
  - Azure platform resources such as VMs, App Service, SQL, Storage, Key Vault, and networking services.
  - Guest operating systems and applications running on VMs or containers.
  - Custom applications instrumented with SDKs or OpenTelemetry.
  - Subscription and tenant control-plane events.

- **Collection and ingestion**
  - **Diagnostic settings** for many Azure PaaS services to export logs and metrics.
  - **Azure Monitor Agent (AMA)** for guest OS logs, performance counters, and custom text logs.
  - **Data Collection Rules (DCRs)** to define what AMA collects and where it sends data.
  - **Application Insights** instrumentation for app-level telemetry.
  - **Azure Activity Log** export for subscription events.

- **Storage and analysis**
  - **Azure Monitor metrics store** for platform metrics.
  - **Log Analytics workspace** for logs and query-based analysis.
  - **Application Insights** for application telemetry, often workspace-based in modern deployments.
  - **Storage accounts** for archive and long-term retention scenarios.
  - **Event Hubs** for streaming to external SIEM or analytics platforms.

- **Action and automation**
  - **Alert rules** for metric, log, and activity log conditions.
  - **Action groups** for notifications and automation.
  - **Alert processing rules** for suppression, routing, and maintenance-window handling.
  - **Automation runbooks, Logic Apps, Functions, and webhooks** for remediation.

**Data collection patterns**

Azure Monitor uses different collection patterns depending on the workload.

- **PaaS diagnostics**
  - You usually configure **diagnostic settings** on the resource.
  - You choose categories such as audit, request, performance, or service-specific logs.
  - You send data to one or more destinations: Log Analytics, Event Hub, or Storage.
  - This is the standard pattern for Azure service logs and many platform metrics exports.

- **IaaS and guest OS monitoring**
  - You install **Azure Monitor Agent** on Windows or Linux VMs, Arc-enabled servers, and some supported environments.
  - You attach a **DCR** to define data sources and destinations.
  - You use this for performance counters, syslog, Windows Event Logs, and custom text logs.

- **Application telemetry**
  - You instrument code with Application Insights SDKs or OpenTelemetry.
  - You capture requests, dependencies, exceptions, traces, and custom events.
  - You use this when you need distributed tracing and application-level diagnostics.

- **Control-plane auditing**
  - You use the **Activity Log** for subscription-level operations such as create, update, delete, and policy events.
  - You often export it to a Log Analytics workspace for correlation with resource logs.

**Azure Monitor Agent and Data Collection Rules**

AMA is the modern agent for Azure monitoring on machines. It replaces older agent patterns in many scenarios and gives you a more flexible, centralized configuration model.

A DCR defines:

- **dataSources**: what to collect, such as performance counters or event logs.
- **destinations**: where to send collected data.
- **dataFlows**: how collected streams map to destinations.

A simplified example helps illustrate the shape, but real DCRs must match the supported schema for the resource type and stream names.

```json
{
  "location": "eastus",
  "properties": {
    "dataSources": {
      "performanceCounters": [
        {
          "name": "perfCounters",
          "streams": [ "Microsoft-Perf" ],
          "samplingFrequencyInSeconds": 60,
          "counterSpecifiers": [
            "\\Processor(_Total)\\% Processor Time",
            "\\Memory\\Available MBytes"
          ]
        }
      ]
    },
    "destinations": {
      "logAnalytics": [
        {
          "name": "la-workspace",
          "workspaceResourceId": "/subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.OperationalInsights/workspaces/<workspace>"
        }
      ]
    },
    "dataFlows": [
      {
        "streams": [ "Microsoft-Perf" ],
        "destinations": [ "la-workspace" ]
      }
    ]
  }
}
```

Use DCRs when you want centralized, repeatable collection across many machines. They are especially useful in enterprise designs because you can standardize what gets collected and where it goes.

**Practical CLI examples**

The exact commands vary by resource type and Azure CLI version, so treat these as patterns and verify against the target resource.

Create a Log Analytics workspace:

```bash
az monitor log-analytics workspace create \
  --resource-group rg-monitoring \
  --workspace-name law-prod \
  --location eastus
```

Create a diagnostic setting for a resource:

```bash
az monitor diagnostic-settings create \
  --name send-to-law \
  --resource /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<account> \
  --workspace /subscriptions/<sub-id>/resourceGroups/rg-monitoring/providers/Microsoft.OperationalInsights/workspaces/law-prod \
  --logs '[{"category":"StorageRead","enabled":true},{"category":"StorageWrite","enabled":true}]' \
  --metrics '[{"category":"Transaction","enabled":true}]'
```

> Diagnostic categories and supported destinations are resource-specific. Always check the target resource’s available categories before automating.

Create an action group:

```bash
az monitor action-group create \
  --resource-group rg-monitoring \
  --name ag-ops \
  --short-name ops \
  --action email ops-team ops@example.com
```

Create a metric alert:

```bash
az monitor metrics alert create \
  --name cpu-high \
  --resource-group rg-monitoring \
  --scopes /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.Compute/virtualMachines/vm01 \
  --condition "avg Percentage CPU > 80" \
  --window-size 5m \
  --evaluation-frequency 1m \
  --severity 2 \
  --action ag-ops
```

Create a log alert with KQL:

```bash
az monitor scheduled-query create \
  --name failed-requests \
  --resource-group rg-monitoring \
  --scopes /subscriptions/<sub-id>/resourceGroups/rg-monitoring/providers/Microsoft.OperationalInsights/workspaces/law-prod \
  --condition "count 'requests | where success == false' > 10" \
  --description "More than 10 failed requests in 5 minutes" \
  --severity 2
```

**Log Analytics and KQL**

Log Analytics is the main query and analysis surface for Azure Monitor logs. It uses KQL, which is designed for fast filtering, aggregation, joins, and time-series analysis.

A simple query might look like this:

```kusto
AzureActivity
| where TimeGenerated > ago(24h)
| where ActivityStatusValue == "Failed"
| summarize Failures = count() by Caller, OperationNameValue
| order by Failures desc
```

Use KQL when you need:

- Correlation across multiple sources.
- Root-cause analysis.
- Custom dashboards and workbooks.
- Scheduled log alerts.

Use metrics when you need:

- Fast alerting on numeric values.
- Near-real-time charts.
- Autoscale triggers.
- Low-latency threshold evaluation.

**Application Insights and distributed tracing**

Application Insights is the application performance monitoring component of Azure Monitor. In modern designs, it is commonly workspace-based, which improves integration with Log Analytics and central governance.

It captures:

- **Requests** and response times.
- **Dependencies** such as SQL, HTTP, and storage calls.
- **Exceptions** and failures.
- **Traces** and custom events.
- **Distributed traces** across services.

OpenTelemetry is increasingly important because it standardizes instrumentation across languages and platforms. Use it when you want vendor-neutral telemetry collection and easier portability.

A minimal .NET example:

```csharp
builder.Services.AddApplicationInsightsTelemetry();
```

A more modern OpenTelemetry-style approach may export to Azure Monitor through supported exporters, depending on the runtime and package set.

**Alerting architecture**

Azure Monitor alerting is not one thing; it is a family of alert types with different evaluation models.

- **Metric alerts**
  - Evaluate numeric metrics quickly.
  - Best for CPU, memory, queue depth, latency, and availability signals.
  - Support dynamic thresholds in some scenarios.

- **Log alerts**
  - Evaluate KQL queries on log data.
  - Best for complex conditions, correlation, and custom logic.
  - Usually slower than metric alerts because they query log data.

- **Activity log alerts**
  - Trigger on subscription-level events.
  - Useful for governance, policy changes, and resource lifecycle events.

- **Smart detection and anomaly-style capabilities**
  - Some Azure services and Application Insights features can detect unusual patterns automatically.
  - These are useful for surfacing regressions, but they do not replace explicit alert design.

Alert design should include:

- **Severity**
  - Use severity to communicate urgency and routing priority.
  - Keep severity consistent across teams.

- **Scope**
  - Scope alerts to the smallest practical resource set.
  - Avoid broad scopes that create noisy incidents.

- **Frequency and window**
  - Short windows catch fast failures.
  - Longer windows reduce noise for slow-moving trends.

- **Action groups**
  - Use them to centralize notification and remediation actions.
  - Reuse them across many alerts.

- **Alert processing rules**
  - Use them for suppression during maintenance, routing changes, or temporary exclusions.
  - Prefer this current terminology over older “action rule” wording.

**Example alerting design**

A practical enterprise pattern is:

- Metric alerts on VM CPU, disk, and availability.
- Log alerts on application exceptions and failed requests.
- Activity log alerts on resource deletion, policy changes, and role assignment changes.
- Action groups that notify email, SMS, ITSM, and webhook endpoints.
- Alert processing rules that suppress alerts during planned maintenance windows.

**Private access and network isolation**

Azure Monitor private access is component-specific. Do not assume every Azure Monitor endpoint is privately reachable in every scenario.

The main concept to know is **Azure Monitor Private Link Scope (AMPLS)**. AMPLS lets you group supported Azure Monitor resources and expose them through private endpoints so that ingestion and query traffic can stay on private network paths where supported.

Common design points:

- Use AMPLS when you need private access to supported Azure Monitor components.
- Use private endpoints in the monitoring VNet or hub network.
- Plan DNS carefully so clients resolve the private endpoint correctly.
- Verify which services and paths are supported before assuming full private coverage.

> Private Link support is not uniform across all Azure Monitor features. Validate the exact component, region, and traffic path before designing a fully private monitoring architecture.

**Retention, limits, and cost considerations**

Monitoring architecture is strongly shaped by retention and service limits.

- **Metrics**
  - Retention is typically shorter than logs.
  - Best for recent operational analysis and alerting.
  - High-cardinality dimensions can increase complexity and cost.

- **Logs**
  - Retention is configurable in Log Analytics.
  - Longer retention increases cost.
  - Archive and export patterns help with compliance and long-term storage.

- **Activity log**
  - Retention in the platform is limited.
  - Export it if you need longer retention or centralized auditing.

- **Application telemetry**
  - Sampling is often necessary at scale.
  - Use sampling to control ingestion volume without losing signal quality.

Design decisions should account for:

- Ingestion volume.
- Query frequency.
- Retention requirements.
- Regulatory obligations.
- Cross-region or cross-subscription centralization.
- Cost of high-cardinality telemetry and verbose logs.

**Authentication and authorization**

Monitoring access should follow least privilege.

- Use **Microsoft Entra ID** for authentication to Azure Monitor, Log Analytics, and related services.
- Use **Azure RBAC** to control who can read logs, manage alerts, or configure diagnostic settings.
- Use **managed identities** for automation instead of storing secrets in scripts or pipelines.
- Grant only the permissions needed for the task, such as reading logs, writing diagnostic settings, or managing alert rules.

Typical patterns include:

- A deployment pipeline uses a managed identity to configure diagnostic settings.
- An automation runbook uses a managed identity to query logs and trigger remediation.
- A SOC analyst gets read access to a workspace but not permission to change alert rules.

**End-to-end enterprise reference architecture**

A common enterprise monitoring design looks like this:

1. Azure platform resources emit metrics and logs.
2. PaaS services use diagnostic settings to send logs to a central Log Analytics workspace and selected logs to Event Hubs for SIEM integration.
3. VMs and Arc-enabled servers use AMA with DCRs to collect performance counters, Windows Event Logs, and syslog.
4. Applications use Application Insights and OpenTelemetry for request tracing and dependency tracking.
5. Activity logs are exported to the same workspace for governance correlation.
6. Metric alerts handle fast operational thresholds.
7. Log alerts handle complex application and security conditions.
8. Action groups notify operations and trigger automation.
9. Alert processing rules suppress noise during maintenance.
10. AMPLS and private endpoints are used where supported to reduce public exposure.

This design gives you:

- Centralized visibility.
- Separate paths for fast metrics and rich logs.
- Better security through RBAC and managed identities.
- Flexible routing to SIEM, storage, and automation systems.

**Design tradeoffs to remember**

- Use **metrics** for speed and alerting; use **logs** for depth and investigation.
- Use **diagnostic settings** for Azure PaaS; use **AMA + DCR** for guest and server telemetry.
- Use **Application Insights** for app behavior; do not rely on platform logs alone for application troubleshooting.
- Use **alert processing rules** for suppression and routing; do not confuse them with older terminology.
- Use **private access** only where the specific Azure Monitor component supports it.
- Use **managed identities** for automation and configuration tasks whenever possible.

**Exam focus summary**

For AZ-305, be ready to explain:

- The difference between platform metrics, platform logs, activity logs, resource logs, guest telemetry, and application telemetry.
- Which storage or query path each telemetry type uses.
- When to choose diagnostic settings versus AMA and DCRs.
- How Log Analytics, Application Insights, and the metrics store fit together.
- How alert rules, action groups, and alert processing rules work together.
- Where private access is supported and why AMPLS matters.
- How RBAC and managed identities shape secure monitoring designs.
- How retention, cost, and service limits influence architecture choices.
