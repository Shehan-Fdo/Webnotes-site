## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Virtual WAN overview**

Azure Virtual WAN is a managed networking service for large-scale branch, remote user, and cloud connectivity. It centralizes connectivity through one or more Microsoft-managed virtual hubs and is designed to reduce the operational burden of building and maintaining a traditional transit network.

Use Virtual WAN when you need:

- Many branch sites or remote users connected to Azure
- Transitive routing between branches, VNets, and other connectivity types
- Centralized security and routing policy
- Multi-region connectivity with simplified operations

Virtual WAN is not just a “VNet hub.” In exam scenarios, treat it as a managed transit architecture that can connect:

- **VNets** through VNet connections
- **Branch sites** through site-to-site VPN
- **Point-to-site users** through P2S VPN
- **ExpressRoute circuits** through ExpressRoute gateways

**Core architecture components**

A Virtual WAN deployment usually includes a few distinct layers:

- **Virtual WAN**: The global resource that groups hubs and connectivity policy.
- **Virtual hub**: A Microsoft-managed routing and connectivity hub deployed in a region.
- **Hub connections**: Attachments from VNets, VPN sites, users, or ExpressRoute circuits into the hub.
- **Gateways**: Managed VPN and ExpressRoute gateways that terminate external connectivity.
- **Routing tables**: Control which connected networks learn which routes and where traffic is forwarded.
- **Security layer**: Often Azure Firewall in a secured virtual hub, or a third-party NVA if the design requires it.

A useful mental model is that the hub is the transit point, while the connections are the spokes. In Virtual WAN, spokes are broader than VNets alone.

**Virtual hub behavior**

A virtual hub is a regional routing node managed by Azure. It provides transitive connectivity between attached networks and can host gateways and security services.

Important hub behaviors:

- Traffic between attached networks is routed through the hub rather than directly between spokes.
- The hub can learn routes from connected VNets, VPN sites, ExpressRoute, and P2S users.
- Multiple hubs can exist in different regions under the same Virtual WAN.
- Hubs can exchange routes to support inter-hub transit, depending on design and enabled features.

For AZ-305, remember that the hub is not a customer-managed virtual appliance. You do not configure it like a custom router VM. Instead, you influence behavior through connections, route tables, propagation, and security settings.

**Connectivity types and when to use them**

Virtual WAN supports multiple connectivity patterns, and the right choice depends on the source network and latency, bandwidth, and operational requirements.

- **Site-to-site VPN**
  - Best for branch offices, temporary sites, or lower-cost connectivity.
  - Uses IPsec/IKE tunnels over the internet.
  - Good for rapid deployment and broad compatibility.
- **ExpressRoute**
  - Best for private, predictable, high-throughput enterprise connectivity.
  - Preferred for production workloads with strict latency, reliability, or compliance requirements.
  - Can coexist with VPN in the same Virtual WAN for resilience or phased migration.
- **Point-to-site VPN**
  - Best for individual users, developers, or administrators.
  - Useful for remote access and secure admin connectivity.
- **VNet connections**
  - Best for Azure workloads that need transit connectivity to branches, users, or other VNets.
  - Common for shared services, application tiers, and landing zones.

A common exam pattern is to ask whether you should use VPN, ExpressRoute, or both. The answer depends on business requirements:

- Choose **VPN** when cost and speed of deployment matter most.
- Choose **ExpressRoute** when private connectivity and predictable performance matter most.
- Choose **both** when you need resilience, migration flexibility, or mixed branch capabilities.

**Basic versus Standard Virtual WAN**

Virtual WAN has service tiers that affect available features. The exact feature set can evolve, so validate current service capabilities during implementation, but the architectural distinction remains important.

| Tier | Typical use | Design implications |
|---|---|---|
| Basic | Simple connectivity scenarios | Limited advanced routing and security capabilities; not ideal for complex enterprise transit designs |
| Standard | Enterprise hub-and-spoke and secured hub scenarios | Supports richer routing, security, and multi-region designs |

For AZ-305, assume **Standard** when the scenario includes:

- Multiple VNets with route isolation
- Secured hub inspection
- Branch-to-branch transit
- Multi-region transit routing
- Advanced route control

Do not over-assume that Basic is sufficient for enterprise hub-and-spoke designs.

**Routing model**

Routing is the most tested concept in Virtual WAN. The hub learns routes from attached connections and uses route tables to decide where traffic goes.

Key routing ideas:

- **Default route table**: The baseline table used by many connections unless you override it.
- **Custom route tables**: Used to isolate groups of spokes or control propagation.
- **Propagation**: Determines which routes a connection advertises into a route table.
- **Association**: Determines which route table a connection uses for forwarding decisions.

This separation matters:

- Propagation controls what routes are visible.
- Association controls what routes are used.

That distinction is often the difference between a working design and a confusing one.

**Practical route-table scenarios**

Use route tables to build segmentation inside the hub:

- **Shared services table**
  - Connects application spokes to shared services such as DNS, identity, or management.
- **Prod table**
  - Keeps production VNets isolated from non-production networks.
- **Branch table**
  - Allows branch sites to reach only approved Azure resources.
- **Default table**
  - Useful for broad transit, but often too permissive for segmented enterprise designs.

A common pattern is:

- Associate a spoke VNet with a custom route table.
- Propagate only the routes that spoke should learn.
- Use a separate table for shared services so only approved networks can reach them.

Example conceptual design:

- `vnet-app1` learns routes to `vnet-shared` and branch prefixes.
- `vnet-dev` learns only internet egress and a limited set of management prefixes.
- Branch sites learn application prefixes but not other branch prefixes unless branch-to-branch transit is intended.

**Hub-to-hub and branch-to-branch transit**

Virtual WAN is valuable because it supports transitive routing. This means traffic can move between attached networks through the managed hub instead of requiring point-to-point tunnels.

Common transit paths:

- **Branch to VNet**
- **Branch to branch**
- **VNet to VNet**
- **P2S user to branch**
- **P2S user to VNet**
- **Inter-hub transit** between regions

This is important for exam questions because the hub is not just a termination point; it is a transit router.

Example flow:

- A branch site connects to Hub A.
- A workload VNet connects to Hub A.
- Another branch site connects to Hub B in a different region.
- If inter-hub transit is enabled and routes are propagated, traffic can traverse Hub A to Hub B without a separate customer-managed transit appliance.

Be careful with assumptions:

- Transit depends on route propagation and association.
- Security policies can alter the path.
- Not every attachment should automatically learn every route.

**Secured virtual hub and Azure Firewall**

A secured virtual hub inserts Azure Firewall into the hub so traffic can be inspected centrally. This is the native security pattern most often associated with Virtual WAN.

Use a secured hub when you need:

- Centralized inspection of internet-bound traffic
- Inspection of private traffic between spokes or toward on-premises
- Simplified firewall operations compared with deploying and scaling custom NVAs

Important nuance:

- Azure Firewall is a major native security option, but not the only possible security pattern in Virtual WAN.
- Third-party NVAs and other routing/security designs can still be used depending on requirements.

**Route intent and traffic steering**

Route intent is used in secured hubs to steer selected traffic through Azure Firewall. It is not a generic “send everything anywhere” switch.

Typical intent categories include:

- **Internet traffic**
- **Private traffic**

What this means in practice:

- Internet-bound traffic from connected networks can be forced through the firewall for inspection before egress.
- Private traffic between spokes or toward on-premises can also be steered through the firewall when the design requires inspection.
- Route tables and propagation still matter; route intent does not replace routing design.

A good exam answer should mention both:

- The firewall is inserted into the path.
- Route tables and propagation determine which networks actually use that path.

**Forced tunneling and inspection path**

Forced tunneling in a secured hub means traffic is intentionally routed through a security device rather than taking a direct path.

Typical reasons to use it:

- Centralized egress control
- Threat inspection
- Logging and compliance
- Consistent policy enforcement across many spokes

Design considerations:

- Ensure the firewall has a valid next hop and egress path.
- Confirm route propagation so spokes learn the intended default or private routes.
- Validate asymmetric routing risks, especially when mixing custom routes and managed hub routing.

**On-premises connectivity design choices**

Virtual WAN commonly connects Azure to on-premises networks. The two main enterprise choices are VPN and ExpressRoute, and they can coexist.

Use **VPN** when:

- You need quick deployment
- Bandwidth needs are moderate
- Cost is a primary concern
- You need backup connectivity alongside ExpressRoute

Use **ExpressRoute** when:

- You need private connectivity
- You need higher throughput and more predictable latency
- You have enterprise WAN integration requirements
- You are connecting critical production workloads

Coexistence patterns:

- ExpressRoute for primary private connectivity
- VPN for backup or branch sites that do not justify ExpressRoute
- Both attached to the same Virtual WAN for unified routing and policy

**VNet connectivity design**

VNet connections are how Azure workloads join the Virtual WAN transit fabric. In practice, this is how you connect landing zones, shared services, and application spokes.

Design guidance:

- Use separate VNets for distinct trust zones or application tiers.
- Connect each VNet to the hub that best matches its region and traffic pattern.
- Use route tables to prevent unnecessary east-west access.
- Avoid assuming all spokes should see all other spokes.

A common enterprise pattern is:

- Shared services VNet in one region
- Application VNets in multiple regions
- Branch sites and remote users connected to the same WAN
- Security inspection centralized in a secured hub

**Example architecture scenario**

A company has:

- Two Azure regions
- Three branch offices
- Remote developers
- Shared DNS and identity services
- A requirement to inspect internet traffic centrally

Recommended design:

- Deploy a Virtual WAN with two virtual hubs, one per region.
- Connect branch offices to the nearest hub using site-to-site VPN.
- Connect remote developers using point-to-site VPN.
- Connect shared services and application VNets to the appropriate regional hub.
- Enable a secured hub with Azure Firewall for internet inspection.
- Use custom route tables to isolate production and non-production VNets.
- Propagate only approved prefixes between branches and Azure workloads.

Why this works:

- It reduces operational complexity.
- It supports transitive routing.
- It centralizes security.
- It scales better than many manual point-to-point tunnels.

**Example CLI workflow**

Azure CLI command names and parameters can change, so validate against the current CLI reference before deployment. The example below shows the general workflow and the kinds of resources you create.

```bash
# Create a resource group
az group create \
  --name rg-networking \
  --location eastus

# Create a Virtual WAN
az network vwan create \
  --resource-group rg-networking \
  --name vwan-prod \
  --location eastus \
  --type Standard

# Create a virtual hub
az network vhub create \
  --resource-group rg-networking \
  --name hub-eastus \
  --vwan vwan-prod \
  --address-prefix 10.100.0.0/24 \
  --location eastus
```

For branch connectivity, you typically define a VPN site and then connect it to the hub through the appropriate gateway workflow.

```bash
# Conceptual example only: validate current CLI syntax before use
az network vpn-site create \
  --resource-group rg-networking \
  --name branch01 \
  --location eastus \
  --ip-address 203.0.113.10 \
  --address-prefixes 10.10.0.0/16
```

For VNet attachment, you connect the spoke VNet to the hub and control routing with route tables and propagation settings.

```bash
# Conceptual example only: validate current CLI syntax before use
az network vhub connection create \
  --resource-group rg-networking \
  --name conn-app01 \
  --vhub-name hub-eastus \
  --remote-vnet /subscriptions/<sub-id>/resourceGroups/rg-app/providers/Microsoft.Network/virtualNetworks/vnet-app01
```

Use CLI examples to understand the resource flow, but always confirm the exact command shape in the current Azure CLI version.

**Gateway roles in the hub**

Virtual WAN hubs can host different gateway types, and the distinction matters.

- **VPN gateway**
  - Terminates IPsec/IKE tunnels from branch sites or other VPN devices.
  - Used for site-to-site connectivity.
- **ExpressRoute gateway**
  - Terminates ExpressRoute circuits.
  - Used for private enterprise connectivity.
- **Point-to-site gateway**
  - Supports remote user access.
  - Often integrated into the hub’s managed connectivity model.

Exam focus:

- Know which gateway type matches which connectivity source.
- Know that the hub can host more than one connectivity type.
- Know that routing is shared through the hub, not through separate isolated appliances.

**Monitoring and troubleshooting**

When a Virtual WAN design fails, the issue is often routing, propagation, or gateway connectivity rather than the hub itself.

Useful checks:

- **Connection status**
  - Verify that the VNet, VPN site, or ExpressRoute circuit is connected.
- **Effective routes**
  - Confirm the routes learned by a spoke or gateway.
- **Route table association**
  - Check whether the connection is using the intended table.
- **Route propagation**
  - Confirm that the expected prefixes are being advertised.
- **Firewall logs**
  - Inspect traffic denied or allowed by Azure Firewall.
- **Diagnostic settings**
  - Send hub, gateway, and firewall logs to Log Analytics or another sink.

Useful Azure tools and commands:

```bash
# View a VNet's effective routes from the NIC perspective
az network nic show-effective-route-table \
  --resource-group rg-app \
  --name nic-app01

# Check connection state
az network vhub connection show \
  --resource-group rg-networking \
  --vhub-name hub-eastus \
  --name conn-app01
```

In the portal, also review:

- Virtual hub route tables
- Hub connections
- VPN gateway connection health
- ExpressRoute circuit status
- Azure Firewall logs and metrics

**Common design mistakes**

- Treating Virtual WAN like a simple VNet hub only
- Assuming all spokes should learn all routes
- Using Basic when the scenario needs advanced routing or security
- Forgetting that branch sites, P2S users, and ExpressRoute are first-class attachments
- Assuming route intent alone guarantees traffic inspection without route-table design
- Overlooking inter-hub transit requirements in multi-region designs
- Mixing custom routes and managed routes without checking for asymmetric paths

**When Virtual WAN is not the best choice**

Virtual WAN is powerful, but it is not always the right answer.

Consider alternatives when:

- You have only a few VNets and no branch connectivity needs
- You need very custom routing behavior that is easier to implement with NVAs or traditional hub-and-spoke
- You want full control over every routing hop and appliance
- Your design is small enough that the operational overhead of Virtual WAN is unnecessary

In smaller environments, a traditional hub VNet with Azure Firewall or an NVA may be simpler. Virtual WAN shines when scale, transitive routing, and managed operations matter.

**Exam-style decision guidance**

When you see a scenario, map requirements to architecture choices:

- If the problem is many branches and remote users, think **Virtual WAN**.
- If the problem is centralized inspection, think **secured hub** and **Azure Firewall**.
- If the problem is private enterprise connectivity, think **ExpressRoute**.
- If the problem is branch-to-branch or VNet-to-branch transit, think **hub routing and propagation**.
- If the problem is segmentation, think **custom route tables** and controlled propagation.

A strong AZ-305 answer explains not only what to deploy, but why the routing and security model fits the business requirement.

**Key takeaways**

- Virtual WAN is a managed transit architecture, not just a VNet hub.
- The hub routes traffic between branches, users, VNets, and ExpressRoute.
- Route tables, association, and propagation are essential to correct design.
- Secured hubs use Azure Firewall for centralized inspection, but routing still determines the path.
- ExpressRoute and VPN can coexist in the same WAN for resilience and flexibility.
- Standard tier is usually the safer choice for enterprise hub-and-spoke scenarios.
