## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Overview**

Hub-and-spoke is a common Azure network topology for enterprise environments. You place shared services and centralized controls in a **hub** virtual network, then connect workload-specific **spoke** virtual networks to it. This design helps you standardize connectivity, security inspection, DNS, and shared platform services while keeping application networks isolated.

The pattern is especially useful when you need:
- Centralized internet egress control
- Shared on-premises connectivity
- Consistent network security policy
- Segmentation between applications, environments, or business units
- A scalable landing zone model for many subscriptions and teams

A hub-and-spoke design is not a single Azure feature. It is an architecture pattern built from virtual networks, peering, routing, network security controls, and often Azure Firewall, VPN Gateway, or ExpressRoute.

**Core architecture**

The **hub** is the shared-services network. It commonly contains:
- Azure Firewall or a third-party network virtual appliance
- VPN Gateway or ExpressRoute gateway
- DNS forwarders or Azure DNS Private Resolver
- Bastion host for secure admin access
- Shared monitoring or management services

The **spokes** are application or workload networks. They usually contain:
- App tiers and data tiers
- Private endpoints for PaaS services
- Workload-specific network security groups
- Application gateways or internal load balancers when needed

A typical topology looks like this:

```text
On-premises
    |
[VPN/ExpressRoute]
    |
   Hub VNet
  /   |    \
Spoke Spoke Spoke
App1  App2  App3
```

**Why you use it**

Hub-and-spoke gives you a balance between central control and workload autonomy.

- **Centralization** reduces duplicated infrastructure and simplifies governance.
- **Isolation** limits lateral movement between workloads.
- **Scalability** lets you add spokes without redesigning the core network.
- **Operational consistency** makes it easier to apply standard routing, logging, and security controls.
- **Cost control** can improve because shared services are deployed once instead of per workload.

This pattern is often preferred over a flat network when you have multiple applications, multiple subscriptions, or a need to separate production from nonproduction environments.

**Hub responsibilities**

The hub usually acts as the network control plane for shared connectivity.

Common hub functions include:
- **Transit connectivity** to on-premises networks through VPN or ExpressRoute
- **Centralized inspection** using Azure Firewall or an NVA
- **Name resolution** for private DNS zones and hybrid DNS
- **Administrative access** through Azure Bastion or jump hosts
- **Shared ingress/egress** patterns for controlled traffic flow

A hub should be designed carefully because it becomes a critical dependency. If the hub is unavailable, many spokes may lose connectivity to on-premises systems, DNS resolution, or internet egress.

**Spoke responsibilities**

Spokes host application workloads and should remain as independent as possible.

Good spoke design usually includes:
- Dedicated address space per spoke
- Subnets aligned to application tiers or functions
- NSGs to enforce subnet-level filtering
- Route tables to direct traffic through the hub when required
- Private endpoints for Azure PaaS access without public exposure

Keep spokes focused on workload needs. Avoid placing shared services in spokes unless they are truly workload-specific.

**Peering and traffic flow**

Azure Virtual Network peering connects hub and spoke networks with low-latency private connectivity. Peering is not transitive by default, so traffic does not automatically pass from one spoke to another through the hub unless you configure routing and allow it.

Important peering behaviors:
- **Low latency** and high bandwidth between peered VNets
- **No gateway transit by default**
- **No transitive routing by default**
- **No overlapping IP address spaces**
- **Traffic remains on Microsoft’s backbone network**

In many designs, you enable gateway transit on the hub peering and use spoke peering to allow spokes to use the hub gateway.

Example peering configuration concept:
- Hub VNet has a VPN Gateway or ExpressRoute gateway
- Hub peering allows gateway transit
- Spoke peering uses remote gateways

Azure CLI example:

```bash
az network vnet peering create \
  --resource-group rg-network-hub \
  --vnet-name vnet-hub \
  --name hub-to-spoke1 \
  --remote-vnet /subscriptions/<sub-id>/resourceGroups/rg-network-spoke1/providers/Microsoft.Network/virtualNetworks/vnet-spoke1 \
  --allow-vnet-access \
  --allow-forwarded-traffic \
  --allow-gateway-transit

az network vnet peering create \
  --resource-group rg-network-spoke1 \
  --vnet-name vnet-spoke1 \
  --name spoke1-to-hub \
  --remote-vnet /subscriptions/<sub-id>/resourceGroups/rg-network-hub/providers/Microsoft.Network/virtualNetworks/vnet-hub \
  --allow-vnet-access \
  --allow-forwarded-traffic \
  --use-remote-gateways
```

**Routing design**

Routing is what makes hub-and-spoke work as a controlled transit architecture. You often use **user-defined routes** (UDRs) to force traffic through the hub for inspection or egress control.

Common routing goals:
- Send internet-bound traffic from spokes to Azure Firewall in the hub
- Send on-premises-bound traffic to the hub gateway
- Prevent direct spoke-to-internet paths when policy requires inspection
- Control east-west traffic between spokes

Example route table for a spoke subnet:

```bash
az network route-table create \
  --resource-group rg-network-spoke1 \
  --name rt-spoke1

az network route-table route create \
  --resource-group rg-network-spoke1 \
  --route-table-name rt-spoke1 \
  --name default-to-firewall \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.0.0.4
```

Typical route patterns:
- `0.0.0.0/0` to Azure Firewall for forced tunneling or controlled egress
- On-premises prefixes to the hub gateway
- Specific spoke prefixes to a firewall or NVA for inspection

> Be careful with routing loops. If you force traffic to a firewall, ensure return paths are symmetric and that the firewall can reach the destination networks.

**Security controls**

Hub-and-spoke is often chosen because it supports layered security.

Common controls include:
- **Network Security Groups (NSGs)** for subnet and NIC filtering
- **Azure Firewall** for centralized stateful filtering and application rules
- **Web Application Firewall (WAF)** for HTTP/S protection at the edge
- **Private Link** to keep PaaS traffic off the public internet
- **DDoS Protection Standard** for critical public-facing VNets
- **Just-in-time access** and Bastion for administrative access

A practical security model is:
- Use NSGs in spokes to enforce least privilege
- Use Azure Firewall in the hub for outbound and east-west inspection
- Use private endpoints for storage, SQL, Key Vault, and other PaaS services
- Avoid public IPs on workload VMs unless absolutely necessary

Example NSG rule concept:

```bash
az network nsg rule create \
  --resource-group rg-network-spoke1 \
  --nsg-name nsg-app \
  --name Allow-App-To-Sql \
  --priority 200 \
  --direction Outbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes 10.1.1.0/24 \
  --source-port-ranges '*' \
  --destination-address-prefixes 10.2.2.10 \
  --destination-port-ranges 1433
```

**DNS in hub-and-spoke**

DNS is a frequent source of issues in hub-and-spoke environments. If spokes use private endpoints or hybrid connectivity, you need a deliberate DNS strategy.

Common options:
- **Azure DNS Private Resolver** for managed inbound and outbound DNS forwarding
- **Custom DNS servers** in the hub for conditional forwarding
- **Azure-provided DNS** for simple, isolated workloads

Use DNS forwarding when:
- Spokes must resolve on-premises names
- On-premises systems must resolve Azure private endpoint names
- You need consistent name resolution across many spokes

Example conditional forwarder concept on a DNS server:

```powershell
Add-DnsServerConditionalForwarderZone `
  -Name "privatelink.database.windows.net" `
  -MasterServers 10.0.0.10,10.0.0.11 `
  -ReplicationScope Forest
```

If you use private endpoints, remember that name resolution must point to the private IP address, not the public service endpoint.

**Connectivity to on-premises**

The hub is usually the termination point for hybrid connectivity.

Common options:
- **Site-to-site VPN** for encrypted connectivity over the internet
- **ExpressRoute** for private, predictable connectivity
- **Point-to-site VPN** for individual admin or developer access

Design considerations:
- Place the gateway in the hub, not in each spoke
- Use gateway transit so spokes can reach on-premises through the hub
- Plan address spaces to avoid overlap with on-premises networks
- Consider active-active gateways for higher availability

ExpressRoute is preferred when you need:
- Higher bandwidth
- Lower jitter
- More predictable latency
- Private connectivity that does not traverse the public internet

**Shared services in the hub**

The hub often hosts services that many spokes consume.

Examples:
- DNS forwarders or Azure DNS Private Resolver
- Azure Bastion
- Azure Firewall
- VPN/ExpressRoute gateway
- Shared jump hosts
- Monitoring collectors or management tooling

Keep shared services minimal and intentional. The hub should not become a dumping ground for unrelated workloads.

**Address space planning**

Address planning is critical because peering requires non-overlapping IP ranges.

Good practices:
- Allocate a large hub range for shared services and future growth
- Assign each spoke its own CIDR block
- Reserve space for future spokes and regional expansion
- Align address ranges with environment or business unit boundaries

Example:
- Hub: `10.0.0.0/16`
- Spoke 1: `10.1.0.0/16`
- Spoke 2: `10.2.0.0/16`
- Spoke 3: `10.3.0.0/16`

Avoid small, fragmented ranges if you expect growth. Renumbering later is expensive and disruptive.

**Subscription and management group alignment**

In enterprise Azure, hub-and-spoke often maps to governance boundaries.

Common patterns:
- Hub in a shared networking subscription
- Spokes in application subscriptions
- Policy assigned at management group level
- Central security and monitoring teams controlling the hub

This separation supports:
- Clear ownership
- Delegated administration
- Consistent policy enforcement
- Easier chargeback or showback

Azure Policy can help enforce:
- Required NSGs
- Denied public IPs
- Required private endpoints
- Approved regions
- Mandatory diagnostic settings

**When to use Azure Firewall vs NVA**

Azure Firewall is often the default centralized inspection service because it is managed and integrates well with Azure networking.

Use **Azure Firewall** when you want:
- Managed service with built-in scaling
- Network and application rule support
- Threat intelligence filtering
- Easier operational overhead

Use a **network virtual appliance** when you need:
- Vendor-specific features
- Advanced inspection or custom routing behavior
- Existing enterprise security tooling already standardized on a third-party platform

Tradeoff summary:
- Azure Firewall is simpler to operate
- NVAs may offer deeper feature sets but require more management

**Common design patterns**

1. **Centralized egress**
   - Spokes send outbound traffic to the hub firewall
   - Firewall applies allow/deny and logging
   - Useful for compliance and internet control

2. **Hybrid transit**
   - Hub contains VPN or ExpressRoute gateway
   - Spokes use gateway transit
   - Useful for enterprise apps that depend on on-premises systems

3. **Private PaaS access**
   - Spokes use private endpoints for Azure services
   - DNS resolves to private IPs
   - Useful for reducing public exposure

4. **Shared services hub**
   - Hub hosts DNS, Bastion, firewall, and gateway
   - Spokes remain workload-focused
   - Useful for landing zone architectures

**Operational considerations**

Hub-and-spoke is not just about deployment. You also need to operate it well.

Monitor:
- Firewall logs and rule hits
- Gateway health and tunnel status
- Effective routes on NICs
- DNS resolution failures
- NSG flow logs if enabled
- Private endpoint connectivity

Troubleshooting checklist:
- Verify peering is enabled in both directions
- Confirm address spaces do not overlap
- Check UDRs for unintended next hops
- Validate DNS resolution for private endpoints
- Inspect effective routes and security rules on the VM NIC
- Confirm gateway transit settings if spokes need on-premises access

Useful CLI commands:

```bash
az network nic show-effective-route-table \
  --resource-group rg-app \
  --name vmnic01

az network nic list-effective-nsg \
  --resource-group rg-app \
  --name vmnic01

az network vnet peering list \
  --resource-group rg-network-hub \
  --vnet-name vnet-hub
```

**Limitations and pitfalls**

- **Non-transitive peering** means you cannot assume spoke-to-spoke connectivity through the hub without explicit design.
- **Overlapping IP ranges** break peering and hybrid routing.
- **Asymmetric routing** can cause firewall or NVA session failures.
- **DNS misconfiguration** often looks like a network outage even when routing is correct.
- **Hub dependency** can create a blast radius if the hub is overloaded or misconfigured.
- **Too many shared services** in the hub can reduce the clarity of the architecture.

> Do not use the hub as a general-purpose application subnet. Keep it focused on shared network services and control points.

**Practical example topology**

A common enterprise deployment looks like this:
- Hub VNet in a networking subscription
- Azure Firewall in a dedicated subnet
- VPN Gateway or ExpressRoute gateway in the hub
- Azure Bastion for admin access
- Spoke VNets in application subscriptions
- Private endpoints for storage and databases in each spoke
- Azure Policy enforcing no public IPs on workload VMs

This model supports secure, repeatable landing zones and scales well as new applications are added.

**Summary**

Hub-and-spoke is a foundational Azure architecture for enterprise networking. It gives you centralized control over connectivity and security while preserving workload isolation in spokes. Use it when you need scalable segmentation, hybrid connectivity, and consistent governance across many applications or subscriptions.
