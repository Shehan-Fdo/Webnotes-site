## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Lab overview**

Azure Virtual WAN is a managed networking service for large-scale branch, remote user, and virtual network connectivity. In a multinational enterprise, you use it to centralize connectivity across regions while keeping routing, segmentation, and security manageable.

A typical lab topology includes:

- A **Virtual WAN** resource as the global container.
- One or more **virtual hubs** in strategic Azure regions.
- **VPN sites** for branch offices and on-premises locations.
- **Virtual network connections** for spoke VNets.
- Optional **ExpressRoute gateways** for private enterprise connectivity.
- Optional **secured virtual hubs** with **Azure Firewall** for inspection and egress control.
- Optional **Point-to-Site (P2S) VPN** for remote users.

A simple mental model is:

```text
Branch sites ---- VPN ----\
                           \
Spoke VNets ---- VNet ------> Virtual Hub ---- Routing / Security ---- Other sites
                           /
Remote users --- P2S -----/
```

**What Virtual WAN solves**

Virtual WAN reduces the operational burden of building many point-to-point tunnels and custom route meshes. Instead of manually stitching every branch to every VNet, you use the hub as a managed transit point.

Use it when you need:

- Many branches or regions.
- Centralized routing policy.
- Hub-and-spoke connectivity at scale.
- Integrated branch, user, and private WAN connectivity.
- Optional security inspection in the hub.

**Prerequisites and planning**

Before you build the lab, confirm the following:

- The required Azure resource providers are registered, especially `Microsoft.Network`.
- You have permissions to create networking resources in the target subscription.
- The selected Azure regions support Virtual WAN hub deployment.
- Address spaces are planned so they do not overlap:
  - Hub address space must not overlap with connected VNets or branch prefixes.
  - Connected VNets must not overlap with each other.
  - On-premises and branch prefixes must not overlap with Azure address spaces.
- You know which traffic should be:
  - Routed directly between spokes.
  - Sent through a firewall.
  - Allowed to exit to the internet locally or centrally.
- You have a naming convention for hubs, sites, connections, and route tables.

A practical naming pattern is:

- `vwan-global`
- `vhub-emea`
- `vhub-us`
- `vpnsite-london`
- `vpnsite-singapore`
- `vnet-spoke-finance`
- `vnet-spoke-hr`
- `rt-default`
- `rt-segment-finance`
- `azfw-vhub-emea`

**Core resource relationships**

Virtual WAN has a few resource types that are easy to confuse:

- **Virtual WAN**: the global container for hubs and connectivity policy.
- **Virtual hub**: the regional transit point.
- **VPN site**: a branch or on-premises location definition used for site-to-site VPN.
- **VPN gateway**: the managed gateway inside the hub that terminates VPN connections.
- **ExpressRoute gateway**: the managed gateway inside the hub that terminates ExpressRoute connectivity.
- **VNet connection**: the attachment between a spoke VNet and a virtual hub.
- **Route table**: the mechanism used to control propagation and segmentation.
- **Secured virtual hub**: a hub with integrated security routing through Azure Firewall.

**Deployment sequence**

A clean lab sequence helps you avoid dependency issues.

1. Create the Virtual WAN.
2. Create the virtual hubs in the required regions.
3. Create VPN sites for branch locations.
4. Create VPN site-to-site connections.
5. Connect spoke VNets to the hubs.
6. Configure route tables and propagation.
7. Add security with Azure Firewall if required.
8. Add P2S VPN if remote user access is part of the design.
9. Validate routing, reachability, and failover.

**Create the Virtual WAN**

You can create the WAN in the portal or with Azure CLI. The WAN is usually created first because hubs attach to it.

```bash
az network vwan create \
  --resource-group rg-networking \
  --name vwan-global \
  --location eastus \
  --type Standard
```

Notes:

- Use **Standard** for enterprise scenarios that need hub routing, branch connectivity, and advanced features.
- Use a consistent region strategy for hubs close to major user and branch populations.
- The WAN itself is global, but hubs are regional.

**Create virtual hubs**

Create a hub in each region where you need transit.

```bash
az network vhub create \
  --resource-group rg-networking \
  --name vhub-emea \
  --address-prefix 10.100.0.0/23 \
  --location westeurope \
  --vwan vwan-global
```

Important planning points:

- The hub address prefix must be unique and non-overlapping.
- Choose a prefix large enough for hub infrastructure and future growth.
- Keep hub prefixes separate from all branch and VNet prefixes.

**Create branch VPN sites**

A VPN site represents the branch-side endpoint and its advertised prefixes. In practice, you define the branch public IP and the local address spaces that should be reachable through the site.

Example CLI pattern:

```bash
az network vpn-site create \
  --resource-group rg-networking \
  --name vpnsite-london \
  --location westeurope \
  --virtual-wan vwan-global \
  --ip-address 203.0.113.10 \
  --address-prefixes 10.10.0.0/16 10.11.0.0/16
```

Key points:

- The branch public IP is the VPN device endpoint.
- The address prefixes are the branch networks behind that device.
- The exact CLI parameters can vary by Azure CLI version; verify with `az network vpn-site -h` if needed.
- In the portal, you typically enter the site name, device IP, and local prefixes.

**Create VPN connections**

After the site exists, connect it to the hub through the hub VPN gateway.

```bash
az network vpn-gateway connection create \
  --resource-group rg-networking \
  --gateway-name vhub-emea-vpnGateway \
  --name conn-london-emea \
  --remote-vpn-site vpnsite-london \
  --enable-bgp true
```

Use this when:

- You need branch-to-Azure connectivity over IPsec/IKE.
- You want BGP (Border Gateway Protocol) for dynamic route exchange.
- You need multiple branches connected to one or more hubs.

> The exact gateway name is the hub’s VPN gateway resource, not a standalone classic VPN gateway. In Virtual WAN, the gateway is managed as part of the hub.

**Connect VNets to the hub**

Spoke VNets connect to the hub through a VNet connection. This is the core of hub-and-spoke transit.

```bash
az network vhub connection create \
  --resource-group rg-networking \
  --name conn-finance-emea \
  --vhub-name vhub-emea \
  --remote-vnet /subscriptions/<subId>/resourceGroups/rg-app/providers/Microsoft.Network/virtualNetworks/vnet-spoke-finance \
  --internet-security false
```

Practical notes:

- The VNet must not overlap with the hub or any other connected network.
- The connection enables managed transit through the hub.
- If you enable internet security on the connection, traffic can be steered through the secured hub path when routing is configured accordingly.

**Route tables and segmentation**

Route tables are how you control which routes are learned and which networks can talk to each other. Virtual WAN provides managed transitive routing in common hub-and-spoke scenarios, but route tables define segmentation and exceptions.

Use route tables to:

- Separate finance, HR, and shared services.
- Prevent unnecessary spoke-to-spoke access.
- Control which prefixes are propagated to which connections.
- Direct traffic to a firewall or specific next hop.

A common pattern is:

- `rt-default` for general propagation.
- `rt-finance` for finance-only routes.
- `rt-hr` for HR-only routes.
- `rt-security` for routes that must traverse Azure Firewall.

Example conceptual association:

| Connection | Associated route table | Propagates to |
|---|---|---|
| Branch London | `rt-default` | Shared services, approved spokes |
| Finance VNet | `rt-finance` | Finance peers, shared services |
| HR VNet | `rt-hr` | HR peers, shared services |
| Shared services VNet | `rt-default` | All approved segments |

A practical segmentation example:

- Finance VNet can reach shared services and branch London.
- HR VNet can reach shared services but not finance.
- Branch London can reach both finance and HR only if policy allows it.
- Internet-bound traffic from branches is only forced through the firewall if you explicitly configure routing for that path.

**Worked routing example**

Assume these prefixes:

- London branch: `10.10.0.0/16`
- Singapore branch: `10.20.0.0/16`
- Finance VNet: `10.30.0.0/16`
- HR VNet: `10.40.0.0/16`
- Shared services VNet: `10.50.0.0/16`

A sensible design is:

- Associate London branch and shared services with `rt-default`.
- Associate Finance VNet with `rt-finance`.
- Associate HR VNet with `rt-hr`.
- Propagate finance routes to shared services and approved branch tables.
- Propagate HR routes to shared services and approved branch tables.
- Do not propagate finance routes to HR and vice versa.

This gives you managed transitive routing where needed, while still enforcing segmentation.

**Secured virtual hub and Azure Firewall**

A secured virtual hub is a hub configured with security routing through Azure Firewall. You do not treat this as a separate ad hoc firewall deployment flow; instead, you enable hub security and attach the firewall as part of the secured hub design.

Use it when you need:

- Central inspection of east-west traffic.
- Central internet egress control.
- Threat filtering and logging.
- Policy-based control across branches and spokes.

Typical design goals:

- Branch internet traffic is inspected only when you explicitly route it through the secured hub.
- Spoke-to-spoke traffic can be forced through the firewall if the route tables and security configuration send it there.
- Not every packet automatically traverses the firewall; routing determines the path.

Example conceptual flow:

```text
Branch -> Hub -> Azure Firewall -> Internet
Spoke A -> Hub -> Azure Firewall -> Spoke B
Spoke -> Hub -> Direct transit only when allowed by route design
```

A practical Azure Firewall deployment in a secured hub usually involves:

- Enabling security on the hub.
- Deploying or associating Azure Firewall in the hub security configuration.
- Creating route tables or routing intent so traffic uses the firewall path.
- Defining firewall rules for allowed destinations and ports.

Example firewall policy rule collection snippet:

```json
{
  "ruleCollectionGroups": [
    {
      "name": "rcg-allow-web",
      "priority": 100,
      "ruleCollections": [
        {
          "name": "allow-http-https",
          "priority": 100,
          "ruleCollectionType": "FirewallPolicyFilterRuleCollection",
          "action": { "type": "Allow" },
          "rules": [
            {
              "name": "allow-web",
              "ruleType": "NetworkRule",
              "ipProtocols": ["TCP"],
              "sourceAddresses": ["10.10.0.0/16", "10.30.0.0/16"],
              "destinationAddresses": ["0.0.0.0/0"],
              "destinationPorts": ["80", "443"]
            }
          ]
        }
      ]
    }
  ]
}
```

**Point-to-Site VPN**

P2S VPN gives individual users secure access into the Virtual WAN environment. It is useful for administrators, contractors, and remote employees.

Typical configuration elements:

- An address pool for clients, such as `172.16.100.0/24`.
- An authentication method:
  - Microsoft Entra ID
  - Certificate-based authentication
  - RADIUS, where supported
- A client profile that users download and install.

Example conceptual setup:

```bash
az network vhub p2svpn-gateway create \
  --resource-group rg-networking \
  --vhub-name vhub-emea \
  --name p2s-emea \
  --address-prefixes 172.16.100.0/24 \
  --vpn-client-protocol OpenVPN
```

Validation ideas:

- Download the client profile from the portal.
- Connect from a test workstation.
- Confirm the client receives an address from the P2S pool.
- Test access to a spoke VNet and shared services.
- Verify that DNS resolution works if you configured custom DNS.

**ExpressRoute in Virtual WAN**

ExpressRoute provides private connectivity from on-premises to Azure without traversing the public internet. In Virtual WAN, the ExpressRoute circuit connects through the hub’s ExpressRoute gateway.

Important distinctions:

- The **ExpressRoute circuit** is the provider-backed private connection.
- The **ExpressRoute gateway** is the managed gateway in the virtual hub.
- The **hub** is the transit point where the circuit is attached for routing.

Practical flow:

1. Create or obtain the ExpressRoute circuit.
2. Create the ExpressRoute gateway in the virtual hub if it is not already present.
3. Link the circuit to the hub gateway.
4. Verify route exchange and reachability.

Conceptual CLI pattern:

```bash
az network express-route gateway create \
  --resource-group rg-networking \
  --name er-gw-emea \
  --virtual-hub vhub-emea \
  --location westeurope
```

Validation focus:

- Confirm on-premises prefixes are learned in the hub.
- Confirm Azure prefixes are advertised back to on-premises.
- Verify private peering routes are present on the circuit side.
- Test connectivity from on-premises to a spoke VNet and to shared services.

**End-to-end lab build example**

A practical multinational enterprise lab can be built in this order:

1. Create `vwan-global` in `eastus`.
2. Create `vhub-emea` in `westeurope` and `vhub-us` in `eastus2`.
3. Create branch sites:
   - `vpnsite-london` with `10.10.0.0/16`
   - `vpnsite-singapore` with `10.20.0.0/16`
4. Create spoke VNets:
   - `vnet-spoke-finance` with `10.30.0.0/16`
   - `vnet-spoke-hr` with `10.40.0.0/16`
   - `vnet-shared-services` with `10.50.0.0/16`
5. Connect the VNets to the nearest hub.
6. Create route tables:
   - `rt-default`
   - `rt-finance`
   - `rt-hr`
   - `rt-security`
7. Associate finance and HR spokes with their segment tables.
8. Propagate shared services routes to both segments.
9. Enable hub security and attach Azure Firewall.
10. Configure routing so internet-bound and restricted east-west traffic uses the firewall.
11. Add P2S VPN for remote admins.
12. Validate branch, spoke, and remote-user access.

**Validation checklist**

Use validation to prove the design works, not just that resources exist.

- Confirm hub and connection provisioning states are successful.
- Check that branch prefixes appear in the hub route tables.
- Check that spoke prefixes are propagated where expected.
- Test branch-to-spoke connectivity.
- Test spoke-to-spoke connectivity only where allowed.
- Test internet egress behavior from a secured path.
- Test P2S user access to approved resources.
- Test ExpressRoute reachability if included.
- Confirm blocked traffic is actually blocked.

Useful commands and checks:

```bash
az network vhub show \
  --resource-group rg-networking \
  --name vhub-emea
```

```bash
az network vpn-site show \
  --resource-group rg-networking \
  --name vpnsite-london
```

```bash
az network vnet show \
  --resource-group rg-app \
  --name vnet-spoke-finance
```

For route validation, use Network Watcher and NIC effective routes rather than assuming a VNet-level route command exists.

Examples:

```bash
az network nic show-effective-route-table \
  --resource-group rg-app \
  --name nic-finance-vm01
```

```bash
az network watcher test-connectivity \
  --source-resource /subscriptions/<subId>/resourceGroups/rg-app/providers/Microsoft.Compute/virtualMachines/vm-finance-01 \
  --dest-address 10.50.1.10 \
  --protocol TCP \
  --dest-port 443
```

**Troubleshooting symptoms**

Common issues usually point to routing or address planning problems.

- If a branch cannot reach a spoke:
  - Check VPN site connection status.
  - Check route propagation to the correct table.
  - Confirm no overlapping prefixes exist.
- If spoke-to-spoke traffic bypasses the firewall:
  - Check route table associations.
  - Check whether the firewall path is explicitly configured.
  - Check routing intent or security configuration.
- If P2S clients connect but cannot reach resources:
  - Check client address pool overlap.
  - Check DNS settings.
  - Check propagated routes to the P2S gateway.
- If ExpressRoute routes are missing:
  - Check circuit status and peering.
  - Check gateway association in the hub.
  - Check advertised prefixes and route propagation.

**Basic vs Standard Virtual WAN**

| Capability | Basic | Standard |
|---|---|---|
| Hub-based enterprise routing | Limited | Yes |
| Branch VPN connectivity | Not the primary enterprise model | Yes |
| VNet connectivity | Not the main enterprise design | Yes |
| Secured virtual hub / Azure Firewall integration | No | Yes |
| ExpressRoute integration | Not the enterprise target | Yes |
| P2S VPN at scale | Limited | Yes |
| Segmentation and route control | Limited | Yes |

Use **Standard** for the lab and for most AZ-305 enterprise scenarios. Basic is not the right choice when you need full hub routing, security, and multi-site enterprise connectivity.

**Exam and design takeaways**

- Virtual WAN is a managed global transit service, not just another VPN gateway.
- The hub is the regional control point for routing, VPN, ExpressRoute, and optional security.
- Route tables control segmentation and propagation; they do not replace the managed transitive routing model.
- A secured virtual hub uses Azure Firewall as part of the hub security design.
- Internet egress through the firewall happens only when you explicitly route traffic that way.
- Address planning is critical because overlap breaks connectivity and route learning.
- Validation should prove actual traffic paths, not just resource deployment.

**Quick reference checklist**

- Create the WAN first.
- Use Standard for enterprise labs.
- Keep hub prefixes unique.
- Avoid all address overlap.
- Attach branches through VPN sites and hub gateways.
- Attach VNets through VNet connections.
- Use route tables for segmentation.
- Enable hub security when you need inspection.
- Validate with effective routes and connectivity tests.
