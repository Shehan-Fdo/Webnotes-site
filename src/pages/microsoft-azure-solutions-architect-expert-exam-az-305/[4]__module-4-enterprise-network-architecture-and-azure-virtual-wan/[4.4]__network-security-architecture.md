## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Network security architecture overview**

Azure network security is about controlling who can reach what, over which path, and with what level of inspection. In AZ-305, you are expected to choose controls that fit the workload, not just name the most restrictive service. Good designs combine segmentation, private access, perimeter inspection, and monitoring.

A practical architecture usually layers controls:

- **Subnet and NIC filtering** with Network Security Groups (NSGs)
- **Central inspection** with Azure Firewall or a third-party NVA (network virtual appliance)
- **Application-layer protection** with Web Application Firewall (WAF)
- **Private access** with Private Link and private endpoints
- **Edge protection** with DDoS Protection
- **Secure admin access** with Azure Bastion
- **Logging and alerting** with Azure Monitor, Log Analytics, and Defender for Cloud

The exam often tests whether you understand which control belongs at which layer. For example, NSGs are ideal for east-west and subnet-level filtering, while WAF protects HTTP/S applications from web attacks. Azure Firewall is a centralized stateful firewall and policy engine, but it does not automatically inspect all traffic unless routing is designed to send traffic through it.

**Core design principles**

- **Least privilege**: allow only required ports, protocols, sources, and destinations.
- **Segmentation**: isolate tiers, environments, and trust zones.
- **Private access first**: prefer private endpoints and private DNS over public exposure.
- **Centralized inspection**: inspect shared egress and ingress paths in hub-and-spoke designs.
- **Defense in depth**: combine controls so one failure does not expose the workload.
- **Observability**: log decisions, flows, and denied traffic for investigation and tuning.

**Common Azure network security controls**

| Control | Primary purpose | Best use case | Key limitation |
|---|---|---|---|
| NSG | L3/L4 filtering | Subnet and NIC segmentation | No application awareness |
| Azure Firewall | Centralized stateful filtering and egress control | Hub-and-spoke inspection, FQDN filtering, threat intelligence | Requires routing design |
| WAF | HTTP/S attack protection | Internet-facing web apps | Only protects web traffic |
| Private Link | Private access to PaaS and services | Remove public exposure | Requires DNS planning |
| DDoS Protection | Mitigate volumetric attacks | Public IP workloads | Does not replace app or firewall controls |
| Azure Bastion | Secure admin access | RDP/SSH without public IPs | Not a general-purpose proxy |

**Network segmentation with VNets, subnets, and NSGs**

Virtual networks (VNets) define the address space and trust boundary. Subnets let you separate tiers such as web, app, data, management, and shared services. NSGs apply allow/deny rules at the subnet or NIC level and are the first line of defense for internal segmentation.

NSGs are stateful. If you allow outbound traffic from a VM, return traffic is automatically allowed. This matters when designing rules for application tiers because you usually only need to define the initiating direction.

Typical NSG design patterns:

- **Web subnet**: allow inbound from load balancer or application gateway, deny direct admin access, allow outbound to app tier.
- **App subnet**: allow inbound only from web subnet, allow outbound to data tier and required platform services.
- **Data subnet**: allow inbound only from app subnet, restrict outbound tightly.
- **Management subnet**: allow admin access only from trusted source ranges or via Bastion.

Example NSG rule creation:

```bash
az network nsg create \
  --resource-group rg-network \
  --name nsg-app

az network nsg rule create \
  --resource-group rg-network \
  --nsg-name nsg-app \
  --name Allow-Web-To-App-443 \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes 10.10.1.0/24 \
  --source-port-ranges '*' \
  --destination-address-prefixes 10.10.2.0/24 \
  --destination-port-ranges 443
```

Important NSG behaviors:

- **Service tags** simplify rules for Azure services such as `VirtualNetwork`, `AzureLoadBalancer`, and `Internet`.
- **Application security groups (ASGs)** let you group VMs logically and reference the group in rules.
- **Default rules** exist, so you should understand what is already allowed before adding custom rules.
- **UDRs** (user-defined routes) and NSGs solve different problems: routing versus filtering.

**When to use NSGs**

- You need subnet or NIC-level filtering.
- You want to reduce lateral movement between tiers.
- You need simple, low-cost enforcement close to workloads.
- You want to complement, not replace, centralized firewalling.

**When NSGs are not enough**

- You need FQDN-based egress control.
- You need centralized logging and policy across many spokes.
- You need threat intelligence or TLS inspection capabilities.
- You need application-layer protection for HTTP/S.

**Azure Firewall for centralized inspection**

Azure Firewall is a managed, stateful firewall service used for centralized control of inbound, outbound, and east-west traffic. It is commonly deployed in a hub network and used to inspect traffic from spokes, on-premises networks, and internet-bound workloads, but only if you design routing so traffic actually traverses the firewall.

Azure Firewall is typically used for:

- **Egress control** with FQDN and application rules
- **Network filtering** with source/destination IP, ports, and protocols
- **Threat intelligence** alerts or deny actions
- **DNAT** (destination network address translation) for selected inbound scenarios
- **Central logging** and policy management

Azure Firewall SKU considerations:

- **Basic**: suitable for simpler scenarios and smaller environments.
- **Standard**: adds more advanced filtering and threat intelligence features.
- **Premium**: adds advanced capabilities such as TLS inspection and IDPS (intrusion detection and prevention system) features, plus stronger application-layer inspection options.

Be careful not to overstate Premium capabilities. It supports URL filtering and enhanced inspection, but it is not a full replacement for a dedicated secure web gateway or a complete web content categorization platform in every scenario.

Typical hub-and-spoke routing pattern:

- Spoke subnets use UDRs to send internet-bound traffic to the firewall.
- Spoke-to-spoke traffic can also be forced through the hub if required.
- On-premises routes are propagated or explicitly defined so traffic can be inspected centrally.

Example route table association:

```bash
az network route-table create \
  --resource-group rg-network \
  --name rt-spoke

az network route-table route create \
  --resource-group rg-network \
  --route-table-name rt-spoke \
  --name DefaultToFirewall \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.0.0.4
```

> Azure Firewall does not inspect traffic that bypasses it. If you do not associate the right UDRs, spoke traffic may go directly to the internet or other networks.

**Firewall policy and rule types**

Azure Firewall Policy centralizes rule management and supports reuse across firewalls. Rule collection groups organize rules by priority and type.

Rule types:

- **DNAT rules**: translate public IP/port to private destination.
- **Network rules**: filter by IP, port, and protocol.
- **Application rules**: filter HTTP/S and other supported application traffic by FQDN.

Practical rule design guidance:

- Use **network rules** for non-HTTP protocols such as SQL, SSH, or custom TCP/UDP services.
- Use **application rules** for outbound web access by FQDN.
- Use **DNAT** sparingly and only for services that must be reachable from the internet.

Example conceptual policy structure:

```text
Firewall Policy
  ├─ Rule Collection Group: Base-Egress
  │    ├─ Application Rule Collection: Allow-Microsoft-Updates
  │    └─ Network Rule Collection: Allow-DNS-To-Approved-Resolvers
  └─ Rule Collection Group: Inbound-Publishing
       └─ DNAT Rule Collection: Publish-Web-App
```

**Azure Firewall and DNS**

DNS design is critical in firewall architectures because many controls depend on name resolution.

Good patterns:

- Use your organization’s approved DNS servers or Azure DNS Private Resolver.
- Use DNS forwarding rulesets when you need conditional forwarding to on-premises or other DNS servers.
- Ensure workloads can resolve private endpoint names to private IP addresses.

Avoid assuming that a firewall rule to a platform IP solves DNS design. Azure DNS Private Resolver uses an **inbound endpoint** to receive queries from your network and an **outbound endpoint** with a forwarding ruleset to send queries to configured DNS servers. The outbound endpoint does not forward queries by itself without the ruleset.

Example DNS Private Resolver concept:

```text
Spoke VM -> DNS server or resolver -> DNS Private Resolver inbound endpoint
DNS Private Resolver outbound endpoint + forwarding ruleset -> target DNS servers
```

**Web Application Firewall and Application Gateway**

WAF protects web applications from common attacks such as SQL injection, cross-site scripting, and protocol anomalies. In Azure, WAF is commonly deployed with Application Gateway or Front Door.

Use WAF when:

- You expose HTTP/S applications to the internet.
- You need Layer 7 inspection and managed rule sets.
- You want to protect against OWASP Top 10-style threats.

Choose the right front door pattern:

- **Application Gateway WAF**: regional, VNet-integrated, good for internal or internet-facing regional apps.
- **Front Door WAF**: global edge entry point, useful for multi-region and global applications.

WAF is not a substitute for NSGs or Azure Firewall:

- WAF protects web traffic only.
- It does not replace network segmentation.
- It does not inspect arbitrary TCP/UDP application traffic.

**Private Link and private endpoints**

Private Link lets you access Azure PaaS services and some partner services over a private IP in your VNet. A private endpoint is the network interface that receives the private IP address.

Why it matters:

- You can disable public network access for many services.
- Traffic stays on the Microsoft backbone rather than traversing the public internet.
- You reduce exposure and simplify compliance boundaries.

Important design details:

- **Private DNS zones** are usually required so the service name resolves to the private endpoint IP.
- **Private endpoint network policies** on the subnet are typically disabled because private endpoints need special handling.
- **Approval workflow** may be required when connecting to services owned by another subscription or tenant.
- **Service-specific limitations** exist; not every feature of a public endpoint is always available through Private Link.

Example: Storage account private endpoint with DNS zone group

```bash
az network private-endpoint create \
  --name pe-storage \
  --resource-group rg-app \
  --vnet-name vnet-app \
  --subnet subnet-private-endpoints \
  --private-connection-resource-id /subscriptions/<subId>/resourceGroups/rg-data/providers/Microsoft.Storage/storageAccounts/stacct01 \
  --group-id blob \
  --connection-name pe-storage-conn

az network private-dns zone create \
  --resource-group rg-network \
  --name privatelink.blob.core.windows.net

az network private-endpoint dns-zone-group create \
  --resource-group rg-app \
  --endpoint-name pe-storage \
  --name default \
  --private-dns-zone privatelink.blob.core.windows.net \
  --zone-name privatelink.blob.core.windows.net
```

Private endpoint troubleshooting considerations:

- Confirm DNS resolves to the private IP, not the public endpoint.
- Check whether the private endpoint connection is approved.
- Verify subnet policies and route tables do not interfere with expected traffic.
- Confirm the target service supports the specific subresource you are trying to expose.

**Azure DDoS Protection**

DDoS Protection helps defend public IP resources against volumetric and protocol-based attacks. It is especially relevant for internet-facing workloads that use public IPs, such as Application Gateway, Azure Firewall public IPs, or public load balancers.

Two tiers matter for exam design:

- **Basic**: enabled by default for all Azure public IPs.
- **Standard**: adds enhanced mitigation, telemetry, and cost protection for protected resources.

Use DDoS Protection Standard when:

- You host critical internet-facing services.
- You need attack telemetry and mitigation reporting.
- You want stronger protection for public endpoints in a shared architecture.

DDoS Protection does not replace:

- WAF for HTTP/S application attacks
- Azure Firewall for egress and network policy
- NSGs for subnet-level segmentation

**Azure Bastion for secure administration**

Azure Bastion provides secure RDP (Remote Desktop Protocol) and SSH (Secure Shell) access to VMs without exposing public IPs on the VMs. It is a common exam answer for secure management access in Azure.

Why Bastion is useful:

- You avoid opening inbound RDP/SSH to the internet.
- You reduce attack surface on management ports.
- You can manage private VMs through the Azure portal or supported native client workflows.

Typical pattern:

- Deploy Bastion in a dedicated subnet named `AzureBastionSubnet`.
- Remove public IPs from management VMs.
- Restrict admin access with NSGs and identity-based controls.

Example management posture:

```text
Admin workstation -> Azure portal -> Bastion -> Private VM
```

Bastion is not a replacement for:

- Just-in-time VM access controls
- Strong identity and MFA
- NSG restrictions
- Privileged access management

**Hybrid connectivity and inspection paths**

Hybrid designs often connect on-premises networks to Azure using VPN Gateway or ExpressRoute. Security architecture must account for where traffic is inspected and whether it bypasses certain controls.

Key points:

- **VPN Gateway** traffic can be routed through Azure Firewall if UDRs and routing are configured appropriately.
- **ExpressRoute** provides private connectivity, but traffic still needs explicit routing if you want inspection in Azure.
- **ExpressRoute FastPath** can improve performance by sending some traffic directly to the virtual network, which may bypass the gateway data path. That can affect where inspection occurs and should be considered carefully in security designs.

Design implications:

- If you require centralized inspection, validate the exact path for each traffic flow.
- Do not assume all hybrid traffic automatically passes through the firewall.
- Document which flows are inspected, which are exempt, and why.

**Monitoring, logging, and alerting**

Security architecture is incomplete without telemetry. Azure provides multiple layers of monitoring for network security.

Useful sources:

- **Azure Firewall logs**: application, network, DNAT, and threat intelligence events.
- **NSG flow logs**: traffic flow visibility at the NSG level.
- **Application Gateway WAF logs**: web attack and rule match details.
- **DDoS Protection telemetry**: mitigation events and attack metrics.
- **Private Endpoint and DNS logs**: useful for name resolution and connectivity troubleshooting.
- **Defender for Cloud**: security recommendations, alerts, and posture management.

A common monitoring pattern is to send logs to Log Analytics and query them with KQL (Kusto Query Language).

Example Azure Firewall query:

```kusto
AzureDiagnostics
| where Category in ("AzureFirewallApplicationRule", "AzureFirewallNetworkRule", "AzureFirewallDnsProxy")
| summarize Count = count() by Category, bin(TimeGenerated, 1h)
| order by TimeGenerated desc
```

Example NSG flow log analysis concept:

```kusto
AzureNetworkAnalytics_CL
| where FlowDirection_s == "I"
| summarize Count = count() by DestPort_d, Action_s
| order by Count desc
```

Defender for Cloud integration helps you:

- Identify exposed resources
- Detect missing protections
- Prioritize remediation
- Track secure score improvements

**Practical decision matrix**

Use the following guidance when choosing controls:

| Scenario | Best-fit control | Why |
|---|---|---|
| Block traffic between app and data subnets | NSG | Simple, local segmentation |
| Force all spoke egress through a central point | Azure Firewall + UDRs | Centralized inspection and logging |
| Protect a public web app from OWASP attacks | WAF | Layer 7 web protection |
| Remove public exposure from a PaaS service | Private Link | Private connectivity |
| Protect public IP workloads from DDoS | DDoS Protection Standard | Attack mitigation and telemetry |
| Administer VMs without public IPs | Azure Bastion | Secure remote access |
| Inspect hybrid traffic centrally | Azure Firewall + routing design | Consistent policy enforcement |

**Scenario examples**

1. **Internet-facing web app**
   - Use Application Gateway with WAF.
   - Place the app tier in private subnets.
   - Use NSGs to restrict east-west traffic.
   - Use Private Link for backend PaaS dependencies where possible.
   - Add DDoS Protection Standard if the public endpoint is business critical.

2. **Hub-and-spoke enterprise**
   - Deploy Azure Firewall in the hub.
   - Force spoke egress through UDRs.
   - Use NSGs in spokes for tier isolation.
   - Use Bastion for admin access.
   - Centralize logs in Log Analytics.

3. **Private PaaS access**
   - Create private endpoints for storage, SQL, or Key Vault.
   - Configure private DNS zones and zone groups.
   - Disable public network access if supported and required.
   - Validate name resolution from each spoke.

4. **Hybrid line-of-business app**
   - Connect on-premises with VPN or ExpressRoute.
   - Decide whether traffic must be inspected in Azure.
   - Use Azure Firewall for shared egress and policy enforcement.
   - Consider FastPath implications before assuming all traffic is inspected.

**CLI and configuration examples**

Create a Bastion host subnet and Bastion resource:

```bash
az network vnet subnet create \
  --resource-group rg-network \
  --vnet-name vnet-hub \
  --name AzureBastionSubnet \
  --address-prefixes 10.0.10.0/26

az network bastion create \
  --name bastion-hub \
  --resource-group rg-network \
  --vnet-name vnet-hub \
  --public-ip-address pip-bastion
```

Create a private DNS zone link:

```bash
az network private-dns link vnet create \
  --resource-group rg-network \
  --zone-name privatelink.database.windows.net \
  --name link-spoke1 \
  --virtual-network vnet-spoke1 \
  --registration-enabled false
```

Associate an NSG to a subnet:

```bash
az network vnet subnet update \
  --resource-group rg-network \
  --vnet-name vnet-spoke1 \
  --name subnet-app \
  --network-security-group nsg-app
```

**Exam focus reminders**

- **NSG** = subnet/NIC filtering, not centralized inspection.
- **Azure Firewall** = centralized stateful filtering, but only if routing sends traffic through it.
- **WAF** = web application protection, not general network security.
- **Private Link** = private access plus DNS design.
- **DDoS Protection** = public endpoint attack mitigation, not app-layer security.
- **Bastion** = secure admin access without public IPs.
- **Monitoring** = logs, flow visibility, and security posture are part of the architecture, not an afterthought.

**Common pitfalls**

- Assuming Azure Firewall automatically sees all spoke traffic.
- Using NSGs as a substitute for web application protection.
- Forgetting private DNS when deploying private endpoints.
- Exposing RDP/SSH publicly instead of using Bastion.
- Treating DDoS Protection as a replacement for WAF or firewall policy.
- Ignoring ExpressRoute FastPath when designing inspection paths.
- Designing egress control without logging and alerting.

**Summary**

Azure network security architecture is a layered design problem. You choose the right control for the right layer: NSGs for segmentation, Azure Firewall for centralized policy, WAF for web threats, Private Link for private service access, DDoS Protection for public endpoint resilience, and Bastion for secure administration. The strongest AZ-305 answers explain not just what to deploy, but how traffic flows, where inspection occurs, and how you will monitor and validate the design.
