## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Private Link overview**

Azure Private Link gives you private access to Azure PaaS services, Microsoft partner services, and customer-owned services over a private endpoint in your virtual network. The key design goal is to keep traffic off the public internet while preserving service reachability from workloads that live in a VNet, on-premises network, or connected environment.

Use Private Link when you want to:

- Expose a service privately without assigning a public IP to the consumer path.
- Reduce data exfiltration risk by limiting access to specific private endpoints.
- Simplify network security by using private IP-based access patterns.
- Support cross-subscription or cross-tenant service consumption with explicit approval.

Private Link is broader than “private access to Azure services.” In exam scenarios, it can connect to:

- Azure PaaS services such as Storage, SQL Database, Key Vault, Azure App Service, Azure Cosmos DB, and many others.
- Microsoft partner services that publish Private Link-enabled offerings.
- Customer-owned services exposed through a Private Link Service.

**Core components**

Private Link solutions usually involve three related objects:

- **Private endpoint**: A network interface in your VNet with a private IP address that maps to a specific service resource and subresource.
- **Private Link service**: A service-provider-side resource that exposes a service privately through a Standard Load Balancer frontend.
- **Private DNS zone**: A DNS zone that resolves the public service name to the private endpoint IP through CNAME and A record behavior.

A private endpoint is created in the consumer network. The service may live in another subscription, another tenant, or even another organization. Approval is often required unless the service owner has configured auto-approval for the consumer’s subscription or tenant.

**Private endpoint behavior and IP allocation**

A private endpoint is implemented as a dedicated network interface in the subnet you choose. Azure assigns a private IP address from that subnet to the endpoint NIC. In practice, you can let Azure allocate the IP dynamically, or you can request a specific static private IP during creation if the service and deployment method support it.

Important design points:

- The private endpoint NIC is not a general-purpose VM NIC.
- The endpoint consumes one IP address from the subnet.
- The subnet should be sized with room for growth and operational overhead.
- The private endpoint maps to one service resource and one or more subresources, depending on the service.

For example, Azure Storage uses different subresources for different data planes:

- `blob`
- `file`
- `queue`
- `table`
- `web`
- `dfs`

A single storage account can have multiple private endpoints if you need separate access paths for different subresources, but each private endpoint connection is still scoped to the selected subresource(s). You do not use one private endpoint to privately expose unrelated services.

> Private endpoints are not protected the same way as VM NICs by NSGs and UDRs on the endpoint NIC itself. However, subnet-level network controls, routing, and service-specific policies can still affect traffic patterns in the surrounding network design.

**Supported services and common exam scenarios**

AZ-305 questions often focus on which services commonly use Private Link and how you secure them.

Typical services include:

- **Azure Storage**: Blob, file, queue, table, and Data Lake Storage Gen2 endpoints.
- **Azure SQL Database / SQL Managed Instance**: Private access to database endpoints and management scenarios.
- **Azure Key Vault**: Private access for secrets, keys, and certificates.
- **Azure App Service**: Private Endpoint for inbound access to web apps and APIs.
- **Azure Cosmos DB**: Private access to the database account.
- **Azure Container Registry**: Private access to image registry endpoints.
- **Azure Event Hubs** and **Azure Service Bus**: Private access to messaging endpoints.
- **Azure Synapse** and related analytics services: Private connectivity for workspace and data access patterns.

Notable limitations and design considerations:

- Not every Azure service supports Private Link in every feature area.
- Some services require separate private endpoints for different subresources or planes.
- Regional availability can vary by service and feature.
- Quotas apply to private endpoints, private DNS zones, and DNS zone links.
- A single private endpoint does not generally replace all network design needs for a service; you still need identity, authorization, and routing design.

**Public network access and secure design**

For many supported services, you can disable public network access after Private Link is in place. This is often the preferred secure design because it prevents fallback to public endpoints.

Use this pattern when:

- The service should only be reachable from approved private networks.
- You want to enforce private-only access for compliance.
- You are designing a hub-and-spoke or landing zone architecture with centralized network controls.

Keep in mind:

- Some services allow public access to remain enabled while still using Private Link.
- Disabling public access is a service-side setting, not a Private Link setting.
- You should verify whether the service supports private-only operation for the exact feature you use.

Example Azure CLI pattern for a service that supports public network access control:

```bash
az storage account update \
  --name mystorageacct \
  --resource-group rg-data \
  --public-network-access Disabled
```

For Key Vault, the equivalent design is to disable public network access and rely on private endpoints plus RBAC or access policies:

```bash
az keyvault update \
  --name kv-prod-01 \
  --resource-group rg-security \
  --public-network-access Disabled
```

**DNS resolution model**

Private Link depends on DNS. The client usually continues to query the normal service name, and Azure DNS resolution redirects that name through a CNAME chain to the private endpoint target.

For Azure Storage, a client queries:

- `mystorage.blob.core.windows.net`

Azure DNS then resolves the public name to a `privatelink` zone entry such as:

- `mystorage.privatelink.blob.core.windows.net`

The private DNS zone typically contains an A record that points to the private endpoint IP. The client still uses the original service name; the DNS chain makes the private path work.

Common private DNS zone names include:

| Service | Common private DNS zone |
|---|---|
| Storage Blob | `privatelink.blob.core.windows.net` |
| Storage File | `privatelink.file.core.windows.net` |
| Storage Queue | `privatelink.queue.core.windows.net` |
| Storage Table | `privatelink.table.core.windows.net` |
| Key Vault | `privatelink.vaultcore.azure.net` |
| SQL Database | `privatelink.database.windows.net` |
| App Service / Functions | `privatelink.azurewebsites.net` |
| Cosmos DB | `privatelink.documents.azure.com` |
| Event Hubs | `privatelink.servicebus.windows.net` for Service Bus, and service-specific zones for Event Hubs deployments |
| ACR | `privatelink.azurecr.io` |

> DNS zone names are service-specific. Do not assume a wildcard `privatelink.*` forwarding rule is sufficient for on-premises DNS.

A typical Azure DNS setup includes:

1. Create the private DNS zone for the service.
2. Link the zone to the VNet where clients live.
3. Create the private endpoint with DNS integration enabled.
4. Confirm the A record appears in the zone.

Example CLI for Storage Blob:

```bash
az network private-dns zone create \
  --resource-group rg-network \
  --name privatelink.blob.core.windows.net

az network private-dns link vnet create \
  --resource-group rg-network \
  --zone-name privatelink.blob.core.windows.net \
  --name link-spoke1 \
  --virtual-network vnet-spoke1 \
  --registration-enabled false
```

If you create the private endpoint with DNS zone group integration, Azure can automatically create the record set in the private DNS zone.

**End-to-end deployment example**

A common exam scenario is a storage account that must be reachable only from an application subnet and from on-premises users through VPN or ExpressRoute.

A practical sequence looks like this:

1. Create or identify the storage account.
2. Disable public network access on the storage account.
3. Create a private DNS zone for the storage subresource you need.
4. Link the zone to the application VNet.
5. Create the private endpoint in the application subnet.
6. Approve the private endpoint connection if the service owner must approve it.
7. Validate DNS resolution and connectivity from the client subnet and from on-premises DNS.

Example CLI for a blob private endpoint:

```bash
az network private-endpoint create \
  --name pe-storage-blob \
  --resource-group rg-network \
  --vnet-name vnet-spoke1 \
  --subnet subnet-app \
  --private-connection-resource-id /subscriptions/<sub-id>/resourceGroups/rg-data/providers/Microsoft.Storage/storageAccounts/mystorageacct \
  --group-ids blob \
  --connection-name pe-storage-blob-conn
```

Example DNS zone group association:

```bash
az network private-endpoint dns-zone-group create \
  --resource-group rg-network \
  --endpoint-name pe-storage-blob \
  --name default \
  --private-dns-zone privatelink.blob.core.windows.net \
  --zone-name privatelink.blob.core.windows.net
```

Validation commands:

```bash
nslookup mystorageacct.blob.core.windows.net
```

```powershell
Test-NetConnection mystorageacct.blob.core.windows.net -Port 443
```

You should see the name resolve to the private endpoint IP, and TCP 443 should succeed from an authorized network path.

**Approval and cross-tenant scenarios**

Private Link often crosses subscription or tenant boundaries. This is common when:

- A central platform team hosts a service for multiple business units.
- A vendor exposes a service to a customer tenant.
- A shared services subscription publishes a private service to spoke subscriptions.

Approval behavior depends on the service and the provider’s configuration:

- Some services support auto-approval for specific subscriptions or tenants.
- Others require manual approval of each private endpoint connection.
- The service owner usually reviews the connection request in the portal or with CLI/PowerShell.

A typical approval workflow is:

1. Consumer creates the private endpoint.
2. Service owner sees a pending private endpoint connection.
3. Service owner approves or rejects the request.
4. DNS and connectivity become effective after approval.

Example CLI pattern to inspect and approve a pending connection:

```bash
az network private-endpoint-connection list \
  --id /subscriptions/<provider-sub>/resourceGroups/<rg>/providers/Microsoft.<Service>/<resourceType>/<resourceName>
```

```bash
az network private-endpoint-connection approve \
  --id /subscriptions/<provider-sub>/resourceGroups/<rg>/providers/Microsoft.<Service>/<resourceType>/<resourceName>/privateEndpointConnections/<connection-name> \
  --description "Approved for private access"
```

The exact resource provider path varies by service, so always use the service-specific resource ID.

**Private Link Service for customer-owned services**

When you publish your own service privately, you use a Private Link Service behind a Standard Load Balancer. Consumers create private endpoints to your service, and you approve those connections.

Design points:

- The service is fronted by a Standard Load Balancer.
- The load balancer forwards traffic to your backend service.
- The Private Link Service exposes the load balancer frontend to consumers.
- You control which consumer requests are approved.

This pattern is useful for:

- Internal APIs shared across tenants.
- Managed service offerings.
- Partner integrations that must avoid public exposure.

A simplified Bicep sketch for the provider side looks like this:

```bicep
resource lb 'Microsoft.Network/loadBalancers@2023-11-01' = {
  name: 'lb-private'
  location: resourceGroup().location
  sku: {
    name: 'Standard'
  }
  properties: {
    frontendIPConfigurations: [
      {
        name: 'fe'
        properties: {
          privateIPAllocationMethod: 'Dynamic'
          subnet: {
            id: subnetId
          }
        }
      }
    ]
  }
}

resource pls 'Microsoft.Network/privateLinkServices@2023-11-01' = {
  name: 'pls-api'
  location: resourceGroup().location
  properties: {
    loadBalancerFrontendIpConfigurations: [
      {
        id: lb.properties.frontendIPConfigurations[0].id
      }
    ]
    ipConfigurations: []
    visibility: {
      subscriptions: [
        subscription().subscriptionId
      ]
    }
    autoApproval: {
      subscriptions: [
        subscription().subscriptionId
      ]
    }
  }
}
```

The exact schema can vary by API version and deployment pattern, so validate against the current resource provider documentation when implementing.

**Network security and routing considerations**

Private Link changes how traffic reaches the service, but it does not remove the need for network design.

Consider:

- **NSGs**: Network Security Groups still matter for the subnet where the private endpoint lives and for client subnets.
- **UDRs**: User Defined Routes can influence how traffic reaches the private endpoint and whether traffic hairpins through appliances.
- **Firewall and NVA**: Network Virtual Appliances may inspect or control traffic paths in hub-and-spoke designs.
- **Service-side authorization**: Identity and access control still determine what the caller can do after connectivity is established.

Practical guidance:

- Place private endpoints in dedicated subnets when you want clearer policy boundaries.
- Avoid overlapping route policies that break return traffic.
- Test from the exact client subnet that production workloads use.
- Confirm that DNS resolution and TCP connectivity both work before troubleshooting application-layer issues.

**Hybrid DNS design**

For on-premises clients, you usually configure conditional forwarding for the specific Azure private DNS zones you use.

Example:

- Forward `privatelink.blob.core.windows.net` to Azure DNS or to a DNS forwarder in Azure.
- Forward `privatelink.vaultcore.azure.net` for Key Vault.
- Forward `privatelink.database.windows.net` for SQL.

Do not rely on a generic `privatelink.*` rule unless your DNS platform explicitly supports that pattern and you have validated it. In many environments, you configure each zone explicitly.

A common hybrid pattern is:

- On-premises DNS server forwards Azure private zones to a DNS forwarder in the hub VNet.
- The hub VNet has the private DNS zone linked.
- Clients on-premises resolve the public service name and receive the private endpoint IP through the forwarding chain.

**Troubleshooting checklist**

When Private Link fails, isolate the problem in layers.

1. Verify the private endpoint connection state in Azure.
2. Confirm the private DNS zone contains the expected A record.
3. Check that the VNet link exists for the zone.
4. Validate name resolution from the client.
5. Test TCP connectivity to the service port.
6. Review service-side public network access settings.
7. Check NSGs, routes, and firewall rules in the client path.

Useful commands:

```bash
az network private-endpoint show \
  --name pe-storage-blob \
  --resource-group rg-network \
  --query "privateLinkServiceConnections"
```

```bash
az network private-endpoint-connection list \
  --id /subscriptions/<sub-id>/resourceGroups/rg-data/providers/Microsoft.Storage/storageAccounts/mystorageacct \
  --query "[].{name:name,status:privateLinkServiceConnectionState.status,description:privateLinkServiceConnectionState.description}"
```

```bash
az network private-dns record-set a list \
  --resource-group rg-network \
  --zone-name privatelink.blob.core.windows.net
```

```powershell
Resolve-DnsName mystorageacct.blob.core.windows.net
```

If DNS resolves to the public IP, the private DNS zone or forwarding path is incomplete. If DNS resolves correctly but the app still fails, inspect routing, firewall, and service authorization.

**Exam-focused distinctions**

Private Link is often confused with service endpoints. The difference matters in AZ-305 design questions.

| Capability | Private Link | Service Endpoints |
|---|---|---|
| Access path | Private IP via private endpoint | Service remains reachable through Azure backbone from subnet |
| DNS dependency | Yes | Usually no special private DNS requirement |
| Service exposure | Private endpoint to specific resource | Service still uses public service endpoint with subnet restrictions |
| Cross-tenant support | Yes, with approval | Limited and service-specific |
| Granularity | Resource/subresource level | Service and subnet level |
| Typical secure posture | Public access can often be disabled | Public access may still exist depending on service design |

Use Private Link when you need the strongest private access pattern and explicit resource-level exposure. Use service endpoints when you want simpler subnet-based access to supported PaaS services and the service design fits that model.

**Design patterns to remember**

- Use a hub-and-spoke network with centralized DNS when many spokes need private access.
- Use separate private DNS zones per service family.
- Use dedicated subnets for private endpoints when governance or policy clarity matters.
- Disable public network access on the service when the business requirement is private-only access.
- Plan for approval workflows in cross-tenant and shared-service scenarios.
- Validate both DNS and network paths before declaring the deployment complete.

**Quick reference**

- Private endpoint = consumer-side private IP in your VNet.
- Private Link service = provider-side private exposure through Standard Load Balancer.
- Private DNS zone = name resolution glue between public service names and private IPs.
- Approval = service-owner control for cross-boundary access.
- Public access disablement = service-side hardening step, often recommended after validation.
