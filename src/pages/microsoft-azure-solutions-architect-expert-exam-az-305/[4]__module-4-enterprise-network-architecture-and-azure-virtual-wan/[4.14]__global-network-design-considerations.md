## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Global network design goals**

Global network design in Azure is about balancing reach, latency, security, resiliency, and operational simplicity. You usually design for multiple traffic classes at once:

- User-to-application traffic over the internet
- Branch-to-Azure and on-premises-to-Azure hybrid traffic
- Azure-to-Azure east-west traffic across regions or subscriptions
- Administrative access for operators and support teams
- Outbound internet access from workloads with predictable control

The best design is rarely a single product. You combine global entry points, regional application delivery, private connectivity, routing control, and segmentation so each traffic path uses the right service.

**Core design principles**

A strong Azure network architecture starts with clear boundaries and predictable traffic flow. You should decide early where traffic enters, where it is inspected, and where it exits.

- **Least privilege routing** limits who can reach what and reduces blast radius.
- **Segmentation** separates environments, business units, and trust zones.
- **Centralized inspection** improves governance when security teams need shared control.
- **Regional affinity** keeps latency-sensitive traffic close to the workload.
- **Resiliency by design** uses redundant paths, zones, and regions rather than reactive failover.

A common exam theme is choosing between centralized and distributed network control. Centralized designs simplify policy and inspection, while distributed designs reduce latency and avoid bottlenecks. In practice, many enterprises use a hybrid model: centralized ingress, shared security services, and regional spokes for application workloads.

**Azure network building blocks**

Azure global network design often uses these services together:

- **Virtual network (VNet)** for isolated IP space and subnet design
- **Virtual network peering** for private connectivity between VNets
- **VPN Gateway** for encrypted site-to-site or point-to-site connectivity
- **ExpressRoute** for private dedicated connectivity to Microsoft
- **Azure Virtual WAN** for large-scale branch, hub, and transit scenarios
- **Azure Firewall** for centralized network and application filtering
- **Azure Front Door** for global HTTP(S) entry and acceleration
- **Application Gateway** for regional Layer 7 delivery and web application firewall
- **NAT Gateway** for scalable outbound SNAT and predictable egress
- **Azure Bastion** for secure administrative access without public IPs
- **Private Link / Private Endpoints** for private access to PaaS services
- **Traffic Manager** for DNS-based global traffic distribution

Each service solves a different problem. The exam often tests whether you can place the right service at the right layer.

**Hub-and-spoke and segmentation strategy**

Hub-and-spoke remains a common enterprise pattern because it gives you a central place for shared services and security. The hub usually contains connectivity, inspection, DNS, and admin access. Spokes host application workloads.

Typical hub responsibilities:

- Shared ingress and egress
- Firewall and route control
- DNS forwarding and name resolution
- Bastion or jump access
- Connectivity to on-premises networks

Typical spoke responsibilities:

- Application subnets
- Data tier subnets
- Private endpoints
- Workload-specific NSGs and UDRs

For enterprise segmentation, you often separate by:

- **Environment**: production, nonproduction, sandbox
- **Business unit**: finance, HR, engineering
- **Trust level**: internet-facing, internal-only, regulated
- **Subscription**: platform, landing zone, application, shared services
- **Management group**: policy and governance boundaries

This separation matters because Azure Policy, RBAC, and network controls are easier to enforce when the hierarchy matches the organization. A common pattern is to place shared network services in a platform subscription and application workloads in separate landing zone subscriptions.

> Use management groups and subscriptions to enforce network guardrails early. Retrofitting segmentation later is harder because peering, DNS, and route dependencies spread quickly.

**Virtual network peering and transit**

VNet peering provides private, low-latency connectivity between VNets. It is not transitive by default, so if VNet A peers with VNet B and VNet B peers with VNet C, A does not automatically reach C through B.

Important peering behaviors:

- Traffic stays on the Microsoft backbone
- No gateway transit unless explicitly configured
- No transitive routing by default
- Useful for regional or cross-region private connectivity

Gateway transit lets a spoke use the hub’s VPN or ExpressRoute gateway. This is common in hub-and-spoke designs where you want one shared connectivity edge.

Practical considerations:

- Use peering for low-latency private connectivity between workloads.
- Use gateway transit when spokes need shared on-premises access.
- Avoid excessive mesh peering because it becomes difficult to manage at scale.

**Azure Virtual WAN design**

Azure Virtual WAN is a managed networking service for large-scale branch connectivity, hub transit, and centralized routing. It is often preferred when you have many branches, multiple regions, or a need for simplified global transit.

Virtual WAN strengths:

- Managed hubs reduce operational overhead
- Branch connectivity scales better than hand-built hub-and-spoke
- Hub-to-hub transit supports multi-region connectivity
- Branch-to-branch connectivity can be enabled through the virtual WAN fabric
- Centralized routing and security are easier to standardize

Use Virtual WAN when:

- You have many sites or VPN connections
- You need hub-to-hub transit across regions
- You want a managed global transit architecture
- You want to simplify branch onboarding and routing policy

A **secured virtual hub** adds integrated security, typically with Azure Firewall, so traffic can be inspected centrally. In exam scenarios, this is useful when you need centralized control over internet-bound and private traffic through the hub security stack. It is not a general-purpose routing policy engine; it is a managed way to steer traffic through the secured hub and firewall.

**Virtual WAN routing details**

Virtual WAN routing is often tested in architecture questions because it changes how traffic moves between branches, VNets, and regions.

Key routing behaviors:

- **Hub-to-hub transit** allows traffic to move between hubs in different regions.
- **Branch-to-branch connectivity** enables branch sites to communicate through the WAN fabric.
- **VNet-to-VNet transit** can occur through the virtual hub rather than direct peering.
- **Route propagation** is managed centrally, reducing manual route tables.

Use Virtual WAN when you need a global transit backbone rather than a collection of isolated hubs. It is especially useful when multiple regions must share branch connectivity and security policy.

**VPN Gateway design**

VPN Gateway is the standard Azure option for encrypted IPsec/IKE connectivity over the internet. It is common for smaller branch deployments, backup connectivity, or environments that do not require ExpressRoute.

Design choices to know:

- **Route-based VPN** is the modern default and supports dynamic routing.
- **Policy-based VPN** is limited and generally used only for legacy compatibility.
- **BGP** over VPN enables dynamic route exchange and better failover.
- **Active-active VPN gateways** improve availability by using two gateway instances.
- **Multiple tunnels** can provide redundancy to different on-premises devices or ISPs.

Typical design guidance:

- Prefer route-based VPN for new deployments.
- Use BGP when you need automatic route exchange and failover.
- Use active-active mode for higher resiliency.
- Use dual tunnels to separate devices or paths when possible.

Example Azure CLI pattern for a VPN gateway deployment:

```bash
az network public-ip create \
  --resource-group rg-network \
  --name pip-vng \
  --sku Standard

az network vnet-gateway create \
  --resource-group rg-network \
  --name vng-hub \
  --vnet hub-vnet \
  --public-ip-address pip-vng \
  --gateway-type Vpn \
  --vpn-type RouteBased \
  --sku VpnGw2 \
  --no-wait
```

This is only a skeleton. In real deployments, the gateway subnet must already exist, and you often add BGP settings, active-active configuration, and local network gateways.

**ExpressRoute design**

ExpressRoute provides private connectivity to Microsoft through a connectivity provider. It is used when you need predictable latency, higher bandwidth, and private connectivity that does not traverse the public internet.

Important design options:

- **Dual circuits** improve resiliency and maintenance flexibility.
- **Provider diversity** reduces dependency on a single carrier or peering location.
- **Active-active gateways** support higher availability at the Azure edge.
- **FastPath** can improve data path performance by bypassing some gateway processing for supported scenarios.
- **Global reach** can connect on-premises sites through Microsoft’s backbone in some designs, depending on provider and service availability.

Design considerations:

- Use ExpressRoute for mission-critical hybrid connectivity.
- Plan for redundant circuits in different peering locations when possible.
- Validate bandwidth, SLA, and provider support before committing.
- Use BGP to exchange routes dynamically with on-premises routers.

A common exam distinction is that ExpressRoute is private hybrid connectivity, while internet ingress for users is usually handled by Front Door or Application Gateway. They solve different problems and are not interchangeable.

**ExpressRoute resiliency patterns**

For high availability, you usually design beyond a single circuit or single gateway.

- Use two circuits where business continuity requires it.
- Place gateways in active-active mode when supported by the design.
- Use separate physical paths or providers if the business requires carrier diversity.
- Validate failover behavior with BGP and route preference testing.

Example conceptual topology:

- On-premises edge router A connects to ExpressRoute circuit 1
- On-premises edge router B connects to ExpressRoute circuit 2
- Azure uses an active-active gateway pair
- BGP advertises routes on both paths

This reduces the chance that a single device, link, or provider outage breaks connectivity.

**Global ingress: Azure Front Door vs Application Gateway**

Azure Front Door and Application Gateway both deliver HTTP(S) traffic, but they solve different problems.

| Capability | Azure Front Door | Application Gateway |
|---|---|---|
| Scope | Global | Regional |
| Primary use | Internet-facing global entry, acceleration, failover | Regional Layer 7 load balancing and WAF |
| Edge presence | Microsoft edge network | Deployed in a VNet |
| Best for | Multi-region apps, global routing, CDN-like acceleration | Internal apps, regional web apps, private backends |
| TLS termination | Yes | Yes |
| WAF | Yes | Yes |
| Private backend access | Limited by design pattern | Strong fit |

Use **Front Door** when:

- You need global anycast entry and low-latency routing
- You want to route users to the nearest healthy region
- You need global failover across regions
- You want internet-facing HTTP(S) acceleration

Use **Application Gateway** when:

- You need regional Layer 7 routing inside a VNet
- You want to publish internal web apps
- You need WAF close to the workload
- You need path-based routing to regional backends

A common architecture is Front Door in front of regional Application Gateways. Front Door handles global entry and failover, while Application Gateway handles regional routing and private backend delivery.

**Application Gateway design details**

Application Gateway is a regional Layer 7 load balancer with optional Web Application Firewall (WAF). It is often used for internal applications, ingress to AKS, or regional web publishing.

Key features:

- Host-based and path-based routing
- TLS termination and end-to-end TLS
- WAF policy integration
- Internal or internet-facing deployment
- Integration with private backends in a VNet

Practical use cases:

- Internal line-of-business apps
- Regional app ingress behind Front Door
- Reverse proxy for private services
- WAF enforcement for web applications

Example ARM/Bicep-style concept for a listener and backend pool is often too long for notes, but the important design point is that Application Gateway lives in a dedicated subnet and requires careful subnet sizing.

**Traffic Manager vs Front Door**

Traffic Manager is DNS-based global traffic distribution. Front Door is an application-layer global entry service.

Choose **Traffic Manager** when:

- You need DNS-based failover or priority routing
- You have non-HTTP workloads or simple endpoint selection
- You can tolerate DNS caching behavior
- You want a lightweight global routing layer

Choose **Front Door** when:

- You need HTTP(S) acceleration and edge termination
- You want health-based routing at the application layer
- You need WAF and global anycast entry
- You want better user experience for web traffic

Traffic Manager does not proxy traffic. It returns DNS answers, so client caching and TTL values affect failover timing.

**Outbound internet design and NAT Gateway**

NAT Gateway provides scalable outbound SNAT (source network address translation) for subnet egress. It is a strong choice when you need predictable outbound IPs and high SNAT port capacity.

Use NAT Gateway when:

- Workloads need stable outbound public IPs
- You want to avoid SNAT exhaustion on shared load balancers
- You need simple outbound internet access for many instances
- You want to decouple outbound egress from inbound load balancing

Design benefits:

- Predictable outbound IP addresses
- High scale for many concurrent connections
- Works well for container platforms and VM subnets
- Simplifies allow-listing with third parties

Example CLI pattern:

```bash
az network public-ip create \
  --resource-group rg-network \
  --name pip-nat \
  --sku Standard \
  --allocation-method Static

az network nat gateway create \
  --resource-group rg-network \
  --name nat-egress \
  --public-ip-addresses pip-nat \
  --idle-timeout 10

az network vnet subnet update \
  --resource-group rg-network \
  --vnet-name spoke-vnet \
  --name app-subnet \
  --nat-gateway nat-egress
```

NAT Gateway is for outbound internet access. It does not provide inbound publishing or application-layer inspection.

**Azure Firewall and centralized inspection**

Azure Firewall is a managed, stateful firewall service used for centralized network filtering and logging. It is commonly placed in the hub or secured virtual hub.

Use it for:

- Network and application rule enforcement
- Centralized egress control
- Threat intelligence filtering
- Logging and policy visibility
- East-west inspection in hub-and-spoke or Virtual WAN

Design considerations:

- Use firewall policy for centralized rule management.
- Route traffic through user-defined routes when you need inspection.
- Plan for DNS proxy if you need name-based filtering.
- Consider Premium features when you need TLS inspection or IDPS, depending on requirements.

A typical pattern is to force spoke subnets to send internet-bound traffic to the firewall, then allow the firewall to perform SNAT and policy enforcement.

**Azure Bastion for secure administration**

Azure Bastion provides secure RDP (Remote Desktop Protocol) and SSH (Secure Shell) access to VMs without exposing public IPs on the workload. It is a common requirement in secure hub-and-spoke designs.

Why it matters:

- Removes the need for public IPs on admin VMs
- Reduces attack surface
- Supports browser-based access from the Azure portal
- Fits well with centralized admin access patterns

Design considerations:

- Place Bastion in a dedicated subnet named `AzureBastionSubnet`
- Use it for administrative access to hub and spoke VMs
- Combine it with NSGs and Just-In-Time access where appropriate
- Ensure address space sizing supports the deployment model

**Private endpoints and DNS design**

Private Endpoints let you access Azure PaaS services over private IP addresses in your VNet. This is a major design choice for reducing public exposure.

Common services include:

- Storage accounts
- Azure SQL
- Key Vault
- App Service
- Azure Container Registry

DNS is the hard part. Private Endpoint design often fails when name resolution is not planned carefully.

Important DNS patterns:

- Link the private DNS zone to every VNet that needs resolution.
- Use conditional forwarders or Azure DNS Private Resolver when on-premises systems must resolve private names.
- Plan split-horizon DNS so internal and external clients resolve the correct endpoint.
- Avoid overlapping private zones that create ambiguous resolution.

Example private DNS zone pattern:

```bash
az network private-dns zone create \
  --resource-group rg-network \
  --name privatelink.database.windows.net

az network private-dns link vnet create \
  --resource-group rg-network \
  --zone-name privatelink.database.windows.net \
  --name link-spoke1 \
  --virtual-network spoke-vnet
```

For hybrid environments, you often need DNS forwarding from on-premises to Azure and back. Azure DNS Private Resolver can simplify this by providing inbound and outbound endpoints for DNS forwarding without custom VM-based DNS servers.

**DNS split-horizon and hybrid name resolution**

Split-horizon DNS means the same name resolves differently depending on where the query originates. This is common with private endpoints and internal services.

Design challenges:

- On-premises clients may need to resolve Azure private endpoint names.
- Azure workloads may need to resolve on-premises internal zones.
- Public and private records may share the same service name.

Best practices:

- Keep authoritative zones clear and documented.
- Use conditional forwarding rather than duplicating records manually.
- Test resolution from on-premises, hub, and spoke networks.
- Be careful with recursive loops between DNS servers.

If you use custom DNS servers, make sure they can forward to Azure or to the correct private resolver path. Otherwise, private endpoint connectivity may work at the network layer but fail at name resolution.

**Hybrid connectivity and route exchange**

Hybrid connectivity usually combines VPN or ExpressRoute with routing protocols such as BGP. Azure learns on-premises routes through the gateway, and on-premises learns Azure prefixes through the same path.

Important distinction:

- Azure Route Server is mainly for exchanging routes between an NVA (network virtual appliance) and Azure inside the virtual network.
- It is not the primary mechanism for direct Azure-to-on-premises route exchange.
- On-premises route exchange still happens through VPN or ExpressRoute with BGP.

Use Azure Route Server when:

- You have an NVA that needs dynamic route exchange with Azure
- You want to avoid manual route table updates for the appliance
- You need the NVA to learn Azure routes and advertise appliance routes within Azure

This is especially useful in hub designs with third-party firewalls or SD-WAN appliances.

**Network virtual appliances and routing control**

NVAs are third-party or custom appliances used for firewalling, routing, SD-WAN, or inspection. They are common when Azure-native services do not meet a specific enterprise requirement.

Use NVAs when:

- You need vendor-specific security features
- You require advanced routing or inspection logic
- You are migrating an existing network security stack
- You need integration with a broader WAN or SD-WAN strategy

Trade-offs:

- More operational complexity
- More patching and lifecycle management
- Potential scaling and availability concerns
- Need for route management and health monitoring

Azure Route Server can reduce some routing complexity for NVAs, but it does not remove the need to design for failover, health, and asymmetric routing.

**Traffic flow examples**

A good architecture separates traffic by purpose.

1. **Internet user traffic**
   - User connects to Azure Front Door.
   - Front Door routes to the nearest healthy region.
   - Regional traffic may go to Application Gateway.
   - Application Gateway forwards to private backends.

2. **On-premises private traffic**
   - Branch or datacenter connects through ExpressRoute or VPN.
   - Azure gateway advertises routes via BGP.
   - Traffic reaches hub services or spoke workloads.
   - Optional firewall inspection occurs through UDRs or secured hub routing.

3. **Outbound workload traffic**
   - Workload subnet sends internet-bound traffic to NAT Gateway or Azure Firewall.
   - Firewall applies policy and logs traffic if centralized inspection is required.
   - NAT Gateway provides stable egress IPs when inspection is not needed.

4. **Administrative access**
   - Operator uses Azure Bastion or a secured jump host.
   - No public IP is required on the target VM.
   - NSGs restrict direct inbound access.

This separation helps you choose the right service for each path instead of forcing one product to do everything.

**End-to-end reference architecture**

A common enterprise design looks like this:

- Global users enter through Azure Front Door.
- Front Door routes to regional Application Gateways.
- Application Gateways send traffic to private app subnets in spokes.
- Spokes connect to a hub through VNet peering.
- The hub contains Azure Firewall, Bastion, DNS services, and connectivity gateways.
- On-premises sites connect through ExpressRoute or VPN.
- Outbound internet traffic from workloads uses NAT Gateway or firewall egress.
- Private PaaS access uses Private Endpoints and private DNS zones.

This pattern supports global ingress, regional delivery, secure admin access, and hybrid connectivity without exposing workloads unnecessarily.

**Practical CLI examples**

Create a NAT Gateway and associate it with a subnet:

```bash
az network public-ip create \
  --resource-group rg-network \
  --name pip-egress \
  --sku Standard \
  --allocation-method Static

az network nat gateway create \
  --resource-group rg-network \
  --name nat-spoke \
  --public-ip-addresses pip-egress

az network vnet subnet update \
  --resource-group rg-network \
  --vnet-name spoke-vnet \
  --name app-subnet \
  --nat-gateway nat-spoke
```

Create a private DNS zone and link it to a VNet:

```bash
az network private-dns zone create \
  --resource-group rg-network \
  --name privatelink.blob.core.windows.net

az network private-dns link vnet create \
  --resource-group rg-network \
  --zone-name privatelink.blob.core.windows.net \
  --name link-hub \
  --virtual-network hub-vnet \
  --registration-enabled false
```

Create a route table entry to force traffic to a firewall:

```bash
az network route-table create \
  --resource-group rg-network \
  --name rt-spoke

az network route-table route create \
  --resource-group rg-network \
  --route-table-name rt-spoke \
  --name default-to-firewall \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.0.0.4
```

These examples are intentionally simple. In production, you also validate subnet sizing, NSGs, DNS, and failover behavior.

**Common exam decision points**

When you see a scenario, ask these questions:

- Is the traffic global HTTP(S), regional HTTP(S), or non-HTTP?
- Does the workload need private or public ingress?
- Is the requirement for outbound egress, inbound publishing, or both?
- Do you need centralized inspection or local autonomy?
- Is the connectivity path internet-based VPN or private ExpressRoute?
- Do you need many branches and transit hubs, or just a few VNets?
- Is DNS resolution part of the problem?

Typical choices:

- **Front Door** for global web entry
- **Application Gateway** for regional web delivery
- **Traffic Manager** for DNS-based failover
- **NAT Gateway** for scalable outbound SNAT
- **Azure Bastion** for secure VM administration
- **ExpressRoute** for private hybrid connectivity
- **VPN Gateway** for encrypted internet-based hybrid connectivity
- **Virtual WAN** for large-scale transit and branch connectivity
- **Private Endpoints** for private PaaS access

**Troubleshooting and validation**

Network designs fail most often because of routing, DNS, or asymmetric paths.

Check these areas first:

- Effective routes on the subnet
- NSG rules on source and destination subnets
- DNS resolution from the client location
- Firewall logs and UDR next hops
- BGP route propagation on gateways
- Private endpoint approval and zone links

Useful validation commands:

```bash
az network nic show-effective-route-table \
  --resource-group rg-app \
  --name vmnic01

az network nic show-effective-nsg \
  --resource-group rg-app \
  --name vmnic01
```

For packet-level or path validation, use tools like `traceroute`, `tcping`, `Test-NetConnection`, or Azure Network Watcher connection troubleshoot, depending on the scenario.

**Quick comparison summary**

| Need | Best fit |
|---|---|
| Global HTTP(S) entry and acceleration | Azure Front Door |
| Regional web ingress and WAF | Application Gateway |
| DNS-based global failover | Traffic Manager |
| Scalable outbound internet SNAT | NAT Gateway |
| Secure VM admin access | Azure Bastion |
| Private hybrid connectivity | ExpressRoute |
| Encrypted internet-based hybrid connectivity | VPN Gateway |
| Large-scale branch transit | Azure Virtual WAN |
| Private access to PaaS | Private Endpoint |

**Exam-focused takeaways**

- Choose services by traffic type, not by brand familiarity.
- Use Front Door for global user ingress and Application Gateway for regional delivery.
- Use NAT Gateway for predictable outbound egress, not for inbound publishing.
- Use ExpressRoute for private hybrid connectivity and VPN for encrypted internet-based connectivity.
- Use Virtual WAN when scale, transit, and branch onboarding matter more than hand-built routing.
- Plan DNS early for private endpoints and hybrid name resolution.
- Use Bastion to remove public IP exposure from administrative access.
- Treat segmentation as a governance requirement, not just a network design preference.
