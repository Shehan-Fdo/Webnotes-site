## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Overview**

Secure internet egress is the set of controls you use to govern how workloads leave Azure and reach public endpoints. The goal is to reduce exposure, enforce policy, preserve auditability, and keep traffic paths predictable.

In Azure, egress design usually combines:
- **Routing** to force traffic through a control point.
- **Filtering** to allow only approved destinations and protocols.
- **DNS control** to prevent bypass via direct IP use or unmanaged name resolution.
- **Inspection** to detect threats in outbound web traffic when required.

A good design starts with the question: do you need simple outbound allowlisting, full inspection, or both? The answer determines whether you use Azure Firewall, a third-party NVA (network virtual appliance), proxy services, or a layered approach.

**Core design goals**

Secure egress patterns usually aim to:
- Centralize outbound control for multiple spokes, VNets, or subscriptions.
- Reduce direct public IP exposure from workloads.
- Log who accessed what, when, and through which rule.
- Support compliance requirements such as URL filtering, TLS (Transport Layer Security) inspection, or data exfiltration prevention.
- Keep private-to-private traffic from being treated like internet-bound traffic.

A practical architecture often uses a hub-and-spoke network with a shared egress control plane in the hub.

**Common egress patterns**

**1. Azure Firewall as centralized egress control**

Azure Firewall is the most common native option for centralized outbound filtering. It is a managed, stateful firewall that supports:
- **Network rules** for L3/L4 filtering by source, destination, port, and protocol.
- **Application rules** for outbound HTTP/HTTPS and, in supported scenarios, MSSQL traffic control by FQDN.
- **DNAT** for inbound publishing when needed.
- **Threat intelligence** filtering and logging.
- **TLS inspection** in Premium for deeper inspection of encrypted web traffic.

Use Azure Firewall when you want:
- Central policy enforcement across many workloads.
- FQDN-based outbound control for web traffic.
- Native Azure integration and simplified operations.
- Logging in Azure Monitor and Microsoft Sentinel.

A typical hub-and-spoke flow is:
- Spoke subnet default route points to Azure Firewall private IP.
- Firewall allows only approved outbound destinations.
- Internet-bound traffic exits through the firewall public IP.
- Private traffic to on-premises or other VNets follows separate routes and should not be forced through internet egress controls unless intentionally designed.

Example user-defined route (UDR) for a spoke subnet:
```bash
az network route-table route create \
  --resource-group rg-network \
  --route-table-name rt-spoke \
  --name default-to-firewall \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.0.0.4
```

**2. Forced tunneling**

Forced tunneling sends all internet-bound traffic to a central security device or on-premises edge instead of allowing direct internet breakout from Azure workloads.

Use forced tunneling when:
- You need all outbound traffic inspected by a corporate security stack.
- You already have a perimeter firewall or secure web gateway on-premises.
- You want a single egress policy for cloud and on-premises workloads.

Key points:
- Azure Firewall can support forced tunneling scenarios.
- You usually advertise a default route from on-premises via VPN or ExpressRoute.
- You must design carefully to avoid asymmetric routing and to preserve return path consistency.
- Private endpoints and private service access should remain on private paths, not be hairpinned to the internet.

> Forced tunneling is not the same as “send everything through the internet.” It means “send internet-bound traffic to a controlled next hop,” often a private appliance or on-premises edge.

**3. Secure web proxy / explicit proxy**

A proxy pattern is useful when you need application-aware outbound control, user identity integration, or detailed URL categorization.

Typical use cases:
- Browser traffic from managed endpoints.
- SaaS access control.
- URL allow/deny policies.
- Authentication-based policy enforcement.

Proxy patterns can be:
- **Explicit proxy**: clients are configured to use the proxy.
- **Transparent proxy**: traffic is redirected without client configuration, usually via network controls.

In Azure, proxying is often combined with firewalling:
- Firewall handles broad network policy.
- Proxy handles web-specific policy and user context.

**4. Third-party NVA egress**

A third-party firewall or secure web gateway may be preferred when you need:
- Existing vendor policy and operational tooling.
- Advanced application control.
- Integrated SD-WAN or branch connectivity.
- Specific compliance or inspection features not available natively.

Trade-offs:
- More operational overhead.
- Licensing and scaling complexity.
- Potentially more complex routing and HA design.

**5. Direct internet breakout with controls at the workload edge**

Some workloads can use direct outbound access with NSGs (network security groups), service endpoints, or application-level controls. This is usually the least centralized pattern.

Use only when:
- The workload is low risk.
- You have strong host-based controls.
- Centralized egress is unnecessary or impractical.

This pattern is generally weaker for enterprise governance because it is harder to audit and enforce consistently.

**Azure Firewall capabilities and limits**

Azure Firewall is often the exam-relevant answer for secure egress, but you need to know what it does and does not do.

**Network rules**
- Filter by IP address, port, and protocol.
- Suitable for non-HTTP traffic.
- Common for DNS to approved resolvers, NTP, SMTP relay, or custom application ports.
- Do not confuse network rules with application rules.

**Application rules**
- Designed for outbound web traffic.
- Match on FQDNs and URLs for HTTP/HTTPS scenarios.
- Useful for allowlisting domains such as `login.microsoftonline.com` or `*.microsoft.com`.
- For encrypted traffic, the firewall can often match on SNI (Server Name Indication) or host headers without decrypting content.
- TLS inspection is needed when you want to inspect the encrypted payload itself, not merely the destination name.

**Premium features**
- **TLS inspection** for deeper inspection of HTTPS traffic.
- **IDPS** (intrusion detection and prevention system).
- **URL filtering** for more granular web control.
- **Web categories** for category-based allow/deny policies.

Important nuance:
- URL filtering is not universally dependent on TLS inspection for all matching. Some filtering can work with host/SNI-based information.
- TLS inspection becomes relevant when you need visibility into encrypted content beyond name-based matching.

**MSSQL note**
- Azure Firewall can support MSSQL-related filtering in specific network scenarios, but do not treat MSSQL as a general application-rule FQDN use case.
- For exam purposes, remember: application rules are primarily for HTTP/HTTPS; non-web protocols are generally handled with network rules or other controls.

**DNS control in secure egress**

DNS is a critical part of egress security because users and workloads can bypass policy if they resolve names through unmanaged resolvers or connect directly to IP addresses.

**Why DNS matters**
- Firewall policies often rely on FQDNs.
- Malicious or unmanaged DNS can bypass intended controls.
- Split-horizon DNS is common in hybrid environments.
- Private endpoints require correct name resolution to private IPs.

**Common DNS patterns**
- **Azure-provided DNS** for basic VNet name resolution.
- **Custom DNS servers** for enterprise control, filtering, or hybrid resolution.
- **Azure DNS Private Resolver** for managed DNS forwarding between Azure and on-premises.

**Azure DNS Private Resolver**
Azure DNS Private Resolver is a managed DNS forwarding service. It does not host private DNS zones itself. Private DNS zones are hosted in Azure Private DNS; the resolver helps forward queries between Azure, on-premises, and other DNS systems.

Use it when you need:
- Inbound endpoints to receive DNS queries from on-premises.
- Outbound endpoints to forward Azure queries to on-premises DNS.
- Reduced need to run your own DNS forwarder VMs.

Example topology:
- Workloads in Azure query the resolver.
- The resolver forwards selected queries to on-premises DNS.
- Private DNS zones remain in Azure Private DNS and resolve private endpoints to private IPs.

**Practical DNS guidance**
- Ensure workloads use approved resolvers.
- Avoid hardcoding public DNS servers unless explicitly allowed.
- Validate private endpoint resolution with `nslookup` or `Resolve-DnsName`.
- Keep firewall rules aligned with your DNS architecture.

Example checks:
```powershell
Resolve-DnsName www.microsoft.com
Resolve-DnsName myapp.privatelink.database.windows.net
```

**Routing patterns**

Routing determines whether traffic reaches the firewall or bypasses it.

**User-defined routes**
- Set `0.0.0.0/0` to the firewall private IP for internet-bound traffic.
- Add more specific routes for on-premises prefixes, private endpoints, or shared services.
- Use longest-prefix match to keep private traffic on private paths.

Example route table entries:
```bash
az network route-table route create \
  --resource-group rg-network \
  --route-table-name rt-spoke \
  --name default-internet \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.0.0.4

az network route-table route create \
  --resource-group rg-network \
  --route-table-name rt-spoke \
  --name onprem-prefix \
  --address-prefix 10.50.0.0/16 \
  --next-hop-type VirtualNetworkGateway
```

**System routes and service endpoints**
- System routes handle default Azure behavior.
- Service endpoints keep traffic to Azure PaaS services on the Microsoft backbone, but they are not a general internet egress control.
- Private endpoints are usually preferred for private access to PaaS because they remove public exposure.

**SNAT behavior**
Azure Firewall performs source network address translation (SNAT) for outbound traffic to public destinations so return traffic can come back through the firewall. For private destinations, SNAT is generally not used by default.

Design implications:
- Public egress needs firewall public IP capacity and SNAT planning.
- Private traffic should be routed intentionally and not assumed to behave like internet traffic.
- Forced tunneling and hybrid paths can change the effective return path, so test carefully.

**Logging and monitoring**

You need logs to prove policy enforcement and troubleshoot blocked traffic.

Useful telemetry includes:
- Firewall application and network rule logs.
- Threat intelligence alerts.
- DNS query logs if you use custom DNS or resolver logging.
- NSG flow logs for subnet-level visibility.
- Azure Monitor metrics for throughput, CPU, and connection counts.

Example KQL pattern for firewall logs:
```kusto
AzureDiagnostics
| where ResourceType == "AZUREFIREWALLS"
| where Category in ("AzureFirewallApplicationRule", "AzureFirewallNetworkRule")
| project TimeGenerated, msg_s, Category, Resource, OperationName
| order by TimeGenerated desc
```

> Log table names and schemas vary by diagnostic settings, resource-specific tables, and Azure Firewall feature set. Validate the exact table in your workspace before building alerts or dashboards.

Common questions to answer from logs:
- Which destination FQDN was allowed or denied?
- Which source subnet generated the traffic?
- Was the block caused by rule order, threat intelligence, or DNS failure?
- Did traffic bypass the firewall due to routing?

**Security controls to combine with egress**

A secure egress design is stronger when layered.

**Network Security Groups**
- Restrict east-west and subnet-level traffic.
- Reduce the blast radius if a workload is compromised.
- Do not replace centralized outbound inspection.

**Private endpoints**
- Keep PaaS access private.
- Reduce the need for public service exposure.
- Pair with DNS control so names resolve to private IPs.

**Microsoft Defender for Cloud**
- Helps identify risky configurations.
- Can surface recommendations for public exposure and network hardening.

**Web filtering and threat protection**
- Use category-based controls for broad policy.
- Use threat intelligence to block known malicious destinations.
- Consider TLS inspection only when the business need justifies the operational and privacy impact.

**Design patterns by scenario**

**Enterprise shared services**
- Hub-and-spoke network.
- Azure Firewall in the hub.
- Central DNS forwarding.
- Private endpoints for PaaS.
- Logging to a central SIEM.

**Regulated workloads**
- Forced tunneling to a corporate security stack.
- Strict allowlists.
- TLS inspection where policy requires content visibility.
- Separate subscriptions or management groups for policy isolation.

**Developer or low-risk workloads**
- Simpler outbound allowlists.
- Limited direct internet access.
- NSG and Azure Firewall policy for critical destinations only.
- Strong monitoring and alerting.

**Hybrid enterprise**
- On-premises DNS and security services integrated with Azure DNS Private Resolver.
- ExpressRoute or VPN for private connectivity.
- Clear separation between private enterprise traffic and public internet egress.

**Implementation checklist**

1. Define which destinations must be reachable.
2. Decide whether you need FQDN allowlisting, URL filtering, or full TLS inspection.
3. Place a centralized firewall or proxy in the egress path.
4. Add UDRs so spoke subnets send internet-bound traffic to the control point.
5. Configure DNS so workloads use approved resolvers.
6. Prefer private endpoints for PaaS access.
7. Enable logging and validate rule hits.
8. Test failover, asymmetric routing, and SNAT behavior.
9. Review exceptions for SaaS, update services, and identity endpoints.
10. Reassess periodically as application dependencies change.

**Troubleshooting tips**

- If traffic bypasses the firewall, check route tables and effective routes.
- If FQDN rules fail, confirm DNS resolution and that the application uses the expected hostname.
- If HTTPS filtering is incomplete, determine whether you need TLS inspection or only host-based matching.
- If private endpoints fail, verify private DNS zone links and resolver forwarding.
- If outbound connections fail after enabling forced tunneling, inspect return path symmetry and NAT behavior.

**Exam focus**

For AZ-305, focus on choosing the right pattern rather than memorizing every feature:
- Use **Azure Firewall** for centralized outbound control.
- Use **forced tunneling** when all internet-bound traffic must pass through a controlled edge.
- Use **Azure DNS Private Resolver** for managed DNS forwarding, not for hosting private zones.
- Use **private endpoints** to avoid public exposure to PaaS.
- Use **TLS inspection** only when encrypted payload inspection is required.
- Use **network rules** for non-web protocols and **application rules** for HTTP/HTTPS.

**Quick comparison**

| Pattern | Best for | Strengths | Trade-offs |
|---|---|---|---|
| Azure Firewall | Centralized egress control | Native integration, logging, policy | Requires routing design |
| Forced tunneling | Corporate-controlled outbound | Strong governance | More complex routing |
| Proxy | Web policy and user context | URL control, identity integration | Client/config overhead |
| Third-party NVA | Vendor-specific features | Advanced controls | Cost and operations |
| Direct breakout | Simple workloads | Low latency, simple setup | Weak governance |

**Key takeaways**

Secure internet egress is about controlling both the path and the policy of outbound traffic. The strongest designs combine routing, DNS, and filtering so workloads cannot bypass controls accidentally or intentionally.

If you remember only a few points:
- Route internet-bound traffic through a centralized control point.
- Use DNS intentionally so FQDN-based rules remain effective.
- Prefer private access for Azure PaaS where possible.
- Use inspection features only to the extent required by the business and compliance model.
- Validate the full path, not just the firewall rule.
