## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Virtual network segmentation overview**

Virtual network segmentation is the practice of dividing Azure network space into smaller trust zones so you can control east-west traffic, reduce blast radius, and apply different security and routing policies to different workloads. In Azure, segmentation is usually implemented with a combination of subnets, network security groups (NSGs), user-defined routes (UDRs), peering, Azure Firewall, and sometimes Azure Virtual Network Manager (AVNM).

The goal is not just isolation. You also want predictable traffic flow, clear inspection points, and a design that still supports platform services, hybrid connectivity, and future growth.

- **Isolation** separates workloads by trust level, environment, or application tier.
- **Control** comes from explicit allow/deny rules and route design.
- **Inspection** is often centralized through a hub firewall or network virtual appliance.
- **Scalability** depends on subnet planning, peering strategy, and governance.

**Core segmentation models**

The most common Azure design is hub-and-spoke. A central hub hosts shared services such as Azure Firewall, VPN/ExpressRoute gateways, DNS, and sometimes Bastion. Spoke virtual networks host application workloads and are segmented by subnet, NSG, and route policy.

A flat network is simpler but usually weaker from a security and governance perspective. It can work for small environments, but it becomes difficult to manage as the number of applications, teams, and compliance boundaries grows.

- **Hub-and-spoke** centralizes shared services and inspection.
- **Mesh** is useful when many VNets need direct connectivity, but it is harder to govern.
- **Virtual network manager** can help enforce segmentation at scale across many VNets.

**Subnet design and planning**

Subnet design is the foundation of segmentation. You should plan subnets around workload tiers, platform requirements, and future expansion. Azure reserves some subnet names and sizes for specific services, and some services require dedicated subnets.

Common planning considerations include:

- **Address space** should leave room for growth and route summarization.
- **Subnet size** should account for scale units, private endpoints, and service requirements.
- **Reserved subnets** may be required for services such as Azure Bastion, Azure Firewall, Application Gateway, or delegated services.
- **Delegated subnets** are often required for PaaS services that deploy into your VNet.

A practical design usually separates:

- Web or ingress tier
- Application tier
- Data tier
- Shared services
- Private endpoint subnet
- Management or jump access subnet
- Azure platform service subnets

> Subnet sizing matters because resizing later can be disruptive. If you expect private endpoints, scale-out instances, or service delegation, allocate more space than the minimum.

**Example hub-and-spoke segmentation layout**

A realistic segmentation design often looks like this:

- **Hub VNet**
  - Azure Firewall subnet
  - Gateway subnet for VPN/ExpressRoute
  - Bastion subnet
  - Shared DNS or management services
- **Spoke VNet 1**
  - Web subnet
  - App subnet
  - Data subnet
- **Spoke VNet 2**
  - Separate workload or environment
- **Private endpoint subnet**
  - Private Link endpoints for PaaS access

Traffic flow is typically:

1. Spoke-to-spoke traffic is blocked by default unless explicitly routed or allowed.
2. Spoke-to-internet traffic is forced through Azure Firewall.
3. Spoke-to-on-premises traffic uses the hub gateway or firewall path.
4. PaaS access is restricted with private endpoints and service endpoint policies where appropriate.

**Network security groups and application security groups**

NSGs filter traffic at the subnet or NIC level using 5-tuple rules. They are the primary tool for micro-segmentation inside a VNet. You usually apply NSGs to subnets rather than individual NICs unless you need very specific host-level control.

Application Security Groups (ASGs) simplify NSG management by letting you group VMs by function instead of hard-coding IP addresses. This is especially useful for tier-based segmentation.

- **NSGs** control allowed and denied traffic.
- **ASGs** make rules easier to maintain as IPs change.
- **Service tags** reduce rule complexity for Azure platform destinations.
- **Priority** determines which NSG rule is evaluated first.

Example NSG rule using ASGs:

```bash
az network nsg rule create \
  --resource-group rg-network \
  --nsg-name nsg-app \
  --name Allow-Web-To-App \
  --priority 200 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes ApplicationGateway \
  --source-port-ranges '*' \
  --destination-address-prefixes AppTier \
  --destination-port-ranges 443
```

In practice, you often combine ASGs with subnet-level NSGs:

- Web subnet allows inbound from the load balancer or gateway.
- App subnet allows inbound only from the web tier.
- Data subnet allows inbound only from the app tier.
- Management access is restricted to approved admin sources.

**User-defined routes and traffic steering**

UDRs let you override system routes for a subnet. Azure route selection works by choosing the most specific prefix first. If prefixes are equal, a user-defined route takes precedence over a system route.

This matters because segmentation often depends on forcing traffic through a firewall or virtual appliance. You should be careful not to create overly broad routes that blackhole traffic.

Example route table for a spoke subnet:

```bash
az network route-table create \
  --resource-group rg-network \
  --name rt-spoke-app

az network route-table route create \
  --resource-group rg-network \
  --route-table-name rt-spoke-app \
  --name DefaultToFirewall \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.0.0.4
```

A real segmentation design usually needs more than a single default route. You often add specific routes for:

- On-premises prefixes
- Hub VNet prefixes
- Other spoke prefixes that must be reachable
- Azure platform exceptions where required
- Private endpoint or service-specific paths

> A default route to a firewall does not automatically solve all segmentation. If you do not account for local VNet, gateway, and platform traffic, you can break DNS, private endpoints, or hybrid connectivity.

**Azure Firewall and centralized inspection**

Azure Firewall is commonly used as the central inspection point in a hub-and-spoke design. It provides stateful filtering, application rules, network rules, and threat intelligence-based filtering. It is a common choice when you need centralized egress control and east-west inspection between spokes.

Azure Firewall Policy is the modern management model. Policy-based deployments use rule collection groups in the policy, not directly on the firewall resource as a separate older model. Classic deployments used firewall-level rule collections.

Typical uses include:

- Internet egress filtering
- FQDN-based application control
- East-west inspection between spokes
- DNAT for controlled inbound publishing
- Logging and policy governance

Example Azure Firewall policy rule collection group snippet:

```json
{
  "name": "rcg-app-egress",
  "priority": 200,
  "ruleCollections": [
    {
      "name": "app-web-allow",
      "ruleCollectionType": "FirewallPolicyFilterRuleCollection",
      "action": { "type": "Allow" },
      "rules": [
        {
          "name": "allow-microsoft-update",
          "ruleType": "ApplicationRule",
          "sourceAddresses": ["10.1.1.0/24"],
          "protocols": [{ "protocolType": "Https", "port": 443 }],
          "targetFqdns": ["*.windowsupdate.com"]
        }
      ]
    }
  ]
}
```

**Peering settings and segmentation behavior**

VNet peering connects VNets privately over the Azure backbone. Peering is not transitive by default, so spoke A does not automatically reach spoke B through the hub unless you design for it.

Important peering options include:

- **Allow virtual network access** enables basic connectivity between peered VNets.
- **Allow forwarded traffic** is needed when traffic is forwarded by a firewall or NVA.
- **Allow gateway transit** lets a peered VNet use the hub gateway.
- **Use remote gateways** lets a spoke use the hub’s gateway instead of deploying its own.

These settings are often tested in exam scenarios because they affect hybrid and segmented topologies.

A common hub-spoke pattern is:

- Hub peering: `Allow gateway transit`
- Spoke peering: `Use remote gateways`
- Both directions: `Allow forwarded traffic` if firewall/NVA forwarding is used

Example conceptual topology:

```text
Spoke A --> Hub Firewall --> Hub Gateway --> On-premises
   |             |
   |             +--> Spoke B only if explicitly allowed and routed
   +--> Internet egress forced through firewall
```

**Gateway transit and hybrid connectivity**

Gateway transit lets spokes use a shared VPN or ExpressRoute gateway in the hub. This is useful for reducing cost and centralizing hybrid connectivity. It does not provide general transitive routing for all traffic.

That distinction matters:

- **Gateway transit** applies to gateway traffic.
- **Spoke-to-spoke transit** still requires explicit routing and security design.
- **Forced tunneling** may send internet-bound traffic on-premises or to a firewall, depending on your architecture.

When you combine segmentation with on-premises connectivity, verify:

- Which prefixes are advertised or learned
- Whether the hub gateway is the default path
- Whether firewall inspection occurs before or after the gateway
- Whether return traffic follows the same path

**Azure Virtual Network Manager**

Azure Virtual Network Manager helps you manage segmentation across many VNets at scale. It is especially useful when you need consistent policy across multiple subscriptions or regions.

Key AVNM capabilities include:

- **Connectivity configurations** to define which VNets can talk to each other
- **Security admin rules** to enforce high-priority allow or deny behavior centrally
- **Network groups** to target sets of VNets by membership or criteria

AVNM is valuable when manual peering and per-VNet rule management becomes too complex. It does not replace all subnet-level controls, but it can provide governance and consistency at the network fabric level.

**Service endpoints, private endpoints, and service endpoint policies**

Segmentation is not only about VM-to-VM traffic. You also need to control access to PaaS services.

Service endpoints extend VNet identity to supported Azure services, but traffic still uses the public endpoint of the service. Private endpoints are usually preferred when you want private IP access and stronger isolation.

Service endpoint policies restrict which Azure service resources a subnet can reach through service endpoints. They are useful when you want to limit access to specific storage accounts or other supported services.

Use cases:

- Restrict a subnet to only approved storage accounts
- Prevent broad access to all instances of a PaaS service
- Combine with NSGs and UDRs for layered control

**Delegated subnets and service constraints**

Some Azure services require subnet delegation so the service can manage resources in that subnet. Delegation does not automatically block NSGs or UDRs, but the exact restrictions depend on the service.

You should check service-specific requirements for:

- Required subnet size
- Whether NSGs are supported
- Whether UDRs are supported
- Whether private endpoints are allowed in the same subnet
- Whether the subnet must be dedicated

Examples of services that commonly use delegation include:

- Azure App Service Environment
- Azure Container Instances
- Azure SQL Managed Instance
- Azure Databricks
- Azure Kubernetes Service in some networking modes

> Do not assume all delegated subnets behave the same. Always verify the service documentation before designing segmentation around delegation.

**Azure Bastion in segmented networks**

Azure Bastion provides secure RDP and SSH access to VMs without exposing public IPs. It is commonly placed in the hub to support administrative access to spoke workloads.

Bastion can support browser-based access through the Azure portal, and native client support is available only in certain Bastion configurations and scenarios.

Typical benefits:

- No public IP on the VM
- Reduced inbound attack surface
- Centralized admin access
- Better fit for locked-down segmented environments

Bastion requires its own dedicated subnet named `AzureBastionSubnet`.

**Practical hub-and-spoke example**

A secure segmented design for a three-tier application might look like this:

- **Hub VNet**
  - `AzureFirewallSubnet` with Azure Firewall
  - `GatewaySubnet` with VPN/ExpressRoute gateway
  - `AzureBastionSubnet` with Bastion
- **Spoke VNet**
  - `snet-web` with web VMs or app gateway backend
  - `snet-app` with application VMs
  - `snet-data` with database tier or private endpoints
- **Policies**
  - NSG on `snet-web` allows inbound only from approved ingress sources
  - NSG on `snet-app` allows inbound only from `snet-web`
  - NSG on `snet-data` allows inbound only from `snet-app`
  - UDR on spoke subnets sends `0.0.0.0/0` to Azure Firewall
  - Specific routes preserve access to on-premises and hub prefixes
  - Peering uses `Allow forwarded traffic`
  - Spokes use `Use remote gateways` if they need shared hybrid access
  - Service endpoint policies or private endpoints restrict PaaS access

A simplified route and security flow:

```text
Internet
  |
  v
Azure Firewall <--- UDR from spoke subnets
  |
  +--> Allowed outbound to approved FQDNs
  |
  +--> On-premises via hub gateway
  |
  +--> Spoke-to-spoke only if routes and NSGs permit
```

**Design best practices**

- **Start with trust boundaries** such as environment, application, and data sensitivity.
- **Use ASGs** to reduce NSG rule sprawl.
- **Prefer private endpoints** for sensitive PaaS access.
- **Centralize inspection** when compliance or egress control matters.
- **Plan routes carefully** to avoid accidental blackholing.
- **Reserve subnet space** for Azure services and future growth.
- **Document peering intent** so gateway transit and forwarded traffic settings are consistent.
- **Use AVNM** when manual segmentation becomes operationally expensive.

**Common exam pitfalls**

- Assuming VNet peering is transitive by default.
- Forgetting that route specificity wins before route type precedence.
- Using one broad UDR and unintentionally breaking local or hybrid traffic.
- Treating gateway transit as general spoke-to-spoke routing.
- Assuming all delegated subnets have the same NSG and UDR behavior.
- Confusing service endpoints with private endpoints.
- Forgetting `Allow forwarded traffic` when a firewall or NVA forwards packets.
- Placing Azure Bastion in a regular subnet instead of `AzureBastionSubnet`.

**Quick comparison**

| Feature | Primary purpose | Typical use in segmentation |
|---|---|---|
| NSG | Packet filtering | Tier isolation and subnet access control |
| ASG | Grouping endpoints | Simplify tier-based NSG rules |
| UDR | Route steering | Force traffic through firewall or gateway |
| VNet peering | Private VNet connectivity | Hub-spoke and shared services |
| Azure Firewall | Central inspection | Egress control and east-west filtering |
| AVNM | Central network governance | Large-scale segmentation policy |
| Service endpoint policy | PaaS access restriction | Limit service endpoint destinations |
| Private endpoint | Private PaaS access | Stronger isolation for data services |

**Key takeaways**

Virtual network segmentation in Azure is a layered design problem. You combine address planning, subnet boundaries, NSGs, ASGs, UDRs, peering settings, and centralized inspection to create secure and maintainable traffic flows. For AZ-305, focus on how these controls work together, especially in hub-and-spoke designs with hybrid connectivity and PaaS access.
