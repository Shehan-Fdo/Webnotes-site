## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Overview**

Network Security Groups (**NSGs**) and user-defined routes (**UDRs**) are the core tools you use to control traffic flow in Azure virtual networks. In design questions, you are usually balancing three goals:

- **Security**: allow only the traffic that the workload needs.
- **Predictability**: make routing behavior explicit instead of relying on defaults.
- **Operability**: keep the design easy to troubleshoot and scale.

NSGs filter traffic. Route tables influence where traffic goes. They solve different problems, and Azure evaluates them differently. A strong design uses both together, plus peering, service endpoints, private endpoints, and sometimes Azure Firewall or another network virtual appliance (**NVA**).

**Core concepts**

An NSG is a stateful packet filter applied at the subnet or network interface card (**NIC**) level. A route table is associated to a subnet and controls next-hop selection for traffic leaving that subnet. You do not attach a route table to a NIC, which matters when you troubleshoot asymmetric routing or subnet-wide policy.

- **NSG**: permits or denies traffic based on 5-tuple-like matching fields such as source, destination, protocol, and port.
- **Route table**: selects the next hop for packets based on destination prefix.
- **Stateful behavior**: if you allow outbound traffic, return traffic is automatically allowed for that flow.
- **Subnet scope**: route tables apply to subnets, not individual NICs.
- **Policy layering**: NSGs and routes are evaluated independently; a packet can be allowed by one and blocked by the other.

**NSG rule model**

NSG rules are evaluated by priority, from lowest number to highest precedence. The first matching rule wins. If no custom rule matches, Azure applies default rules.

Common matching fields include:

- **Source and destination address prefixes**
  - CIDR blocks such as `10.10.1.0/24`
  - Service tags such as `Internet`, `VirtualNetwork`, `AzureLoadBalancer`, `Storage`, or `Sql`
- **Source and destination ports**
  - Single ports like `443`
  - Ranges like `10000-20000`
- **Protocol**
  - `TCP`, `UDP`, or `*`
- **Direction**
  - `Inbound` or `Outbound`
- **Access**
  - `Allow` or `Deny`
- **Priority**
  - Integer from `100` to `4096` for custom rules, where lower is evaluated first

Application security groups (**ASGs**) let you group NICs logically and reference the group in NSG rules. This is useful when you want tier-based rules without hardcoding IP addresses.

**Default NSG rules**

Azure creates default rules that you should know for exam scenarios:

- **AllowVNetInBound**
  - Allows traffic within the virtual network
- **AllowAzureLoadBalancerInBound**
  - Allows traffic from the Azure Load Balancer service tag
- **DenyAllInBound**
  - Blocks everything else inbound
- **AllowVNetOutBound**
  - Allows outbound within the virtual network
- **AllowInternetOutBound**
  - Allows outbound to the Internet service tag
- **DenyAllOutBound**
  - Blocks everything else outbound

These defaults matter because many workloads depend on them. For example, a public load balancer health probe often arrives from the `AzureLoadBalancer` service tag, so if you override inbound rules too aggressively, the backend may stop receiving probes.

> Be careful with the `Internet` service tag in exam scenarios. It represents public internet sources, not traffic from an Azure Load Balancer or a private front-end path.

**NSG design patterns**

A good NSG design is usually tier-based and least-privilege. You often create separate NSGs for web, app, and data subnets, then allow only the required flows between them.

Typical patterns:

- **Web subnet**
  - Allow inbound `TCP 80/443` from the appropriate front-end source
  - Allow inbound health probes from `AzureLoadBalancer` when behind a load balancer
  - Deny inbound management ports such as `22`, `3389`, or `5985` from broad sources
- **App subnet**
  - Allow inbound only from the web tier on the application port
  - Deny direct internet access
- **Data subnet**
  - Allow inbound only from the app tier on database ports such as `1433`, `3306`, or `5432`
  - Restrict administrative access to a jump host or private management path

Using ASGs simplifies these rules because you can target `asg-web`, `asg-app`, and `asg-data` instead of maintaining IP lists.

**Example: tiered NSG rules with ASGs**

```bash
az network nsg rule create \
  --resource-group rg-app \
  --nsg-name nsg-web \
  --name Allow-HTTPS-From-FrontEnd \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes AzureLoadBalancer \
  --source-port-ranges '*' \
  --destination-address-prefixes '*' \
  --destination-port-ranges 443

az network nsg rule create \
  --resource-group rg-app \
  --nsg-name nsg-app \
  --name Allow-App-From-Web-ASG \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes '*' \
  --source-port-ranges '*' \
  --destination-address-prefixes '*' \
  --destination-port-ranges 8443 \
  --source-asgs asg-web

az network nsg rule create \
  --resource-group rg-app \
  --nsg-name nsg-data \
  --name Allow-SQL-From-App-ASG \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes '*' \
  --source-port-ranges '*' \
  --destination-address-prefixes '*' \
  --destination-port-ranges 1433 \
  --source-asgs asg-app
```

**Practical NSG considerations**

- **Stateful inspection** means you usually do not need a separate outbound rule for return traffic.
- **Subnet NSGs** are often easier to manage than NIC-level NSGs for multi-VM tiers.
- **NIC NSGs** can be useful for exceptions on a single VM, but they increase operational complexity.
- **Rule specificity** matters when you need to block a narrow set of sources or ports without affecting the whole subnet.
- **Service tags** reduce maintenance and are preferred over static public IPs when the source is an Azure service.

**Route table model**

A route table contains UDRs that tell Azure where to send traffic for a destination prefix. Azure uses the most specific matching route first. If multiple routes have the same prefix length, Azure then uses route source preference.

General precedence is:

1. **Longest prefix match**
2. **Route source preference** when prefix length is equal
3. **System routes** and propagation behavior depending on configuration

A UDR does not override a more specific route from another source. For example, a `/24` BGP route wins over a `/16` UDR because the `/24` is more specific.

**Route source preference**

When prefixes are equal, Azure generally prefers routes in this order:

- **User-defined route**
- **BGP route**
- **System route**

This matters in hub-and-spoke and hybrid designs. If you advertise a prefix from on-premises via Border Gateway Protocol (**BGP**) and also create a UDR with the same prefix, the UDR can win if the prefix length is equal. But if the BGP route is more specific, the more specific route wins.

**Route table scope and limits**

- A route table is associated to a **subnet**, not a NIC.
- Every VM in the subnet inherits the subnet route table.
- You can associate one route table per subnet.
- Route propagation from BGP can be disabled on a route table when you want to prevent learned routes from affecting that subnet.
- Route tables are useful for forcing traffic through an NVA, Azure Firewall, or a gateway path.

**Common next hops**

- **Virtual appliance**
  - Sends traffic to an NVA or firewall IP
- **Virtual network gateway**
  - Sends traffic to a VPN or ExpressRoute gateway in the same virtual network
- **Internet**
  - Sends traffic to the public internet
- **Virtual network**
  - Uses the local VNet route
- **None**
  - Drops traffic

> `VirtualNetworkGateway` is only valid when the subnet can actually reach a gateway in the same VNet. Do not use it as a generic “on-premises” next hop unless the gateway exists in that design.

**Example: force spoke-to-on-prem traffic through a firewall**

```bash
az network route-table create \
  --resource-group rg-network \
  --name rt-spoke

az network route-table route create \
  --resource-group rg-network \
  --route-table-name rt-spoke \
  --name DefaultToFirewall \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.0.0.4

az network vnet subnet update \
  --resource-group rg-network \
  --vnet-name vnet-spoke \
  --name app-subnet \
  --route-table rt-spoke
```

This pattern is common when you want all outbound traffic from a spoke subnet to traverse Azure Firewall or another inspection device.

**Hub-and-spoke routing**

Hub-and-spoke is a common Azure design. The hub hosts shared services such as firewall, DNS, VPN/ExpressRoute gateway, and jump access. Spokes host workloads.

Important peering behaviors:

- **Peering is non-transitive by default**
  - If VNet A is peered to VNet B, and VNet B is peered to VNet C, A does not automatically route to C through B.
- **Gateway transit**
  - The hub can share its gateway with spokes when peering is configured correctly.
- **Use remote gateway**
  - A spoke can use the hub’s gateway if the hub peering allows gateway transit and the spoke peering is configured to use the remote gateway.
- **One remote gateway per VNet**
  - A VNet can use only one remote gateway at a time.
- **Allow forwarded traffic**
  - Required when traffic is forwarded by an NVA or firewall across peering.

A common hub-and-spoke flow is:

```text
Spoke VM -> Spoke subnet UDR -> Firewall in hub -> Hub gateway or internet
```

Or for hybrid access:

```text
Spoke VM -> Hub peering -> Hub gateway -> On-premises
```

**Peering configuration implications**

- **Allow virtual network access**
  - Enables direct connectivity between peered VNets
- **Allow forwarded traffic**
  - Needed when an NVA forwards packets between peered VNets
- **Allow gateway transit**
  - Set on the hub peering when the hub provides gateway access
- **Use remote gateways**
  - Set on the spoke peering when the spoke consumes the hub gateway

**Example: hub gateway transit design**

1. Configure the hub peering to allow gateway transit.
2. Configure the spoke peering to use the remote gateway.
3. Ensure the spoke does not already use another remote gateway.
4. Add UDRs in the spoke if you need to force traffic through a firewall before reaching the gateway.

**Service endpoints and private endpoints**

Service endpoints and private endpoints both improve access to PaaS services, but they work differently.

**Service endpoints**

A service endpoint extends the subnet’s identity and routing over the Azure backbone to a supported Azure service. The service then authorizes access based on the subnet or VNet rule.

Use service endpoints when:

- You want to keep traffic on the Azure backbone.
- You want to restrict a PaaS service to specific subnets.
- You do not need a private IP address for the service.

Key points:

- The service still authorizes based on the subnet/VNet rule.
- DNS usually still resolves to the public service endpoint address.
- NSGs on the subnet can still matter for traffic leaving the subnet.
- Service endpoints are simpler than private endpoints, but they expose the service through its public endpoint with network restrictions.

**Private endpoints**

A private endpoint gives a private IP address from your VNet to a PaaS service. Traffic stays private and uses the private IP.

Use private endpoints when:

- You want private IP-based access to PaaS.
- You want to disable public network access.
- You need tighter isolation and private DNS integration.

Important design points:

- Private endpoints usually require **Private DNS zones** so clients resolve the service name to the private IP.
- Microsoft guidance commonly recommends disabling private endpoint network policies on the subnet hosting the private endpoint.
- NSG behavior on private endpoint subnets is limited and should not be treated as the primary access-control mechanism.
- Access control is typically enforced by the private endpoint connection approval, the service’s access settings, and DNS design.

**Private DNS example**

```bash
az network private-dns zone create \
  --resource-group rg-network \
  --name privatelink.database.windows.net

az network private-dns link vnet create \
  --resource-group rg-network \
  --zone-name privatelink.database.windows.net \
  --name link-spoke \
  --virtual-network vnet-spoke \
  --registration-enabled false
```

Without DNS, clients may still try to reach the public endpoint and fail or bypass the intended private path.

**Route tables versus NSGs**

| Capability | NSG | Route table |
|---|---|---|
| Primary purpose | Allow/deny traffic | Choose next hop |
| Scope | Subnet or NIC | Subnet only |
| Stateful | Yes | No |
| Uses ports | Yes | No |
| Uses prefixes | Yes | Yes |
| Affects return traffic | Stateful return allowed | No |
| Typical use | Security segmentation | Traffic steering |

A useful mental model is:

- Use **NSGs** to decide whether traffic is permitted.
- Use **UDRs** to decide where permitted traffic goes.

**Route precedence and troubleshooting**

Azure route selection is usually the source of confusion in exam questions. The safest rule is:

- First, find the **longest matching prefix**.
- Then, if prefixes are equal, consider **route source preference**.
- Then, check whether route propagation is enabled or disabled on the subnet’s route table.

Examples:

- A `/24` route to an NVA beats a `/16` system route.
- A BGP route for `10.1.2.0/24` beats a UDR for `10.1.0.0/16`.
- If you disable BGP propagation on a route table, learned routes do not appear in that subnet’s effective routes.

**Troubleshooting tools**

Use these tools to determine whether a problem is caused by routing or filtering:

- **Effective routes**
  - Shows the routes Azure actually applies to a NIC
- **Effective security rules**
  - Shows the NSG rules applied to a NIC
- **IP flow verify**
  - Tests whether a specific flow is allowed or denied
- **Connection troubleshoot**
  - Helps validate end-to-end connectivity
- **Network Watcher packet capture**
  - Useful for deeper packet-level analysis
- **NSG flow logs**
  - Useful for historical traffic analysis and deny/allow patterns

**Example: inspect effective routes**

```bash
az network nic show-effective-route-table \
  --resource-group rg-app \
  --name vm1-nic
```

**Example: inspect effective NSG rules**

```bash
az network nic list-effective-nsg \
  --resource-group rg-app \
  --name vm1-nic
```

**Example: test a flow**

```bash
az network watcher test-ip-flow \
  --resource-group rg-app \
  --vm vm1 \
  --direction Inbound \
  --protocol TCP \
  --local 10.1.1.4:3389 \
  --remote 203.0.113.10:50000
```

Use the output to determine whether the deny comes from an NSG rule, a missing route, or an unexpected next hop.

**Common exam scenarios**

- **Block management ports from the internet**
  - Deny inbound `22`, `3389`, and `5985-5986` from broad sources.
  - Prefer Bastion, VPN, or jump host access instead of public exposure.
- **Allow only web to app to data**
  - Use separate subnets and NSGs.
  - Allow only the required ports between tiers.
  - Deny all other inter-tier traffic.
- **Force inspection through a firewall**
  - Use UDRs to send `0.0.0.0/0` or specific prefixes to a virtual appliance.
  - Ensure peering allows forwarded traffic.
- **Use a hub gateway from spokes**
  - Enable gateway transit on the hub peering.
  - Use remote gateway on the spoke peering.
  - Remember that a spoke can use only one remote gateway.
- **Reach PaaS privately**
  - Prefer private endpoints for private IP access.
  - Add Private DNS zones.
  - Do not rely on NSGs alone for private endpoint access control.
- **Restrict PaaS to selected subnets**
  - Service endpoints can be enough when private IP is not required.
  - Use service endpoint policies or service-side firewall rules where supported.

**Practical design guidance**

- Use **NSGs** for segmentation and least privilege.
- Use **ASGs** to reduce rule sprawl in tiered applications.
- Use **UDRs** when you need deterministic next-hop control.
- Use **service tags** instead of static public IPs for Azure services.
- Use **private endpoints** when you need private IP access and stronger isolation.
- Use **service endpoints** when you want simpler network restriction without private IPs.
- Use **Azure Firewall** or an NVA when you need centralized inspection, logging, or egress control.
- Validate the design with effective routes and effective security rules before deployment.

**Quick decision guide**

- If you need to **allow or deny** traffic, use an **NSG**.
- If you need to **steer** traffic, use a **route table**.
- If you need to **group workloads by role**, use an **ASG**.
- If you need to **reach a PaaS service privately**, use a **private endpoint**.
- If you need to **restrict PaaS access to a subnet over the Azure backbone**, use a **service endpoint**.
- If you need **central inspection**, route through a firewall or NVA.

**Troubleshooting checklist**

1. Check whether the subnet has the expected route table.
2. Review effective routes on the NIC.
3. Confirm the longest-prefix route that matches the destination.
4. Check whether BGP propagation is disabled.
5. Review effective NSG rules on the NIC or subnet.
6. Use IP flow verify to test the exact source, destination, port, and protocol.
7. Confirm DNS resolution for private endpoint scenarios.
8. Verify peering settings such as forwarded traffic, gateway transit, and remote gateway usage.

**Common pitfalls**

- Assuming a UDR overrides every other route.
- Forgetting that route tables apply to subnets, not NICs.
- Using `Internet` when the traffic actually comes from an Azure service or load balancer.
- Treating NSGs on private endpoint subnets as the main control plane.
- Forgetting Private DNS when using private endpoints.
- Assuming peering is transitive.
- Forgetting `Allow forwarded traffic` when an NVA sits between peered VNets.
- Overusing public IPs and management ports instead of private access paths.

**Mental model for AZ-305**

Think in layers:

- **Identity and access** decide who can connect.
- **DNS** decides what name resolves to.
- **Routing** decides where packets go.
- **NSGs** decide whether packets are allowed.
- **Service controls** decide whether the Azure service accepts the connection.

When you design with that order in mind, Azure network questions become much easier to reason about.
