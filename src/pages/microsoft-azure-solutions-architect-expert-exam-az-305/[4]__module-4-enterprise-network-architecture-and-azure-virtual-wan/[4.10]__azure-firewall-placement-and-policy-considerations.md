## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Azure Firewall overview**

Azure Firewall is a managed, stateful, cloud-native network security service that helps you control traffic between Azure networks, the internet, and on-premises environments. In architecture design, you place it where you need centralized inspection, segmentation, and policy enforcement rather than at every workload subnet.

Use Azure Firewall when you need:

- Centralized egress control from spoke networks
- Inbound publishing with DNAT (Destination Network Address Translation)
- East-west inspection between spokes or between on-premises and Azure
- FQDN (Fully Qualified Domain Name)-based application filtering
- Threat intelligence and logging for network flows

Azure Firewall is not a WAF (Web Application Firewall). It complements WAF by controlling network and application traffic, but it does not replace HTTP request protection at the application layer.

**SKU selection and when to use each**

Azure Firewall comes in three main SKUs: Basic, Standard, and Premium. Choose the SKU based on security depth, scale, and compliance needs.

| SKU | Best fit | Key capabilities | Notes |
|---|---|---|---|
| Basic | Small environments, cost-sensitive designs | Core L3/L4 filtering, DNAT, SNAT, logging | Fewer advanced features |
| Standard | General enterprise hub-and-spoke | Network rules, application rules, threat intelligence, DNS proxy | Common default choice |
| Premium | High-security and regulated workloads | TLS (Transport Layer Security) inspection, IDPS (Intrusion Detection and Prevention System), URL filtering, advanced threat protection | Use when you need deeper inspection |

Important design guidance:

- **Basic** is suitable when you mainly need simple centralized filtering and lower cost.
- **Standard** is the common enterprise choice for hub-and-spoke egress control and segmentation.
- **Premium** is the right choice when you need TLS inspection or stronger threat detection, but remember that it is still not a WAF.

> Azure Firewall Premium can inspect TLS traffic, but that is different from application-layer request protection. If you need OWASP-style web request filtering, place a WAF in front of the app.

**Placement patterns**

Azure Firewall placement depends on your network topology and the traffic you want to control.

Common patterns include:

- **Hub-and-spoke**
  - Place Azure Firewall in a shared services hub VNet.
  - Route spoke traffic through the hub for centralized inspection.
  - Use user-defined routes (UDRs) to force spoke egress through the firewall.
- **Virtual WAN secured hub**
  - Deploy Azure Firewall in a secured virtual hub.
  - Let Virtual WAN manage routing between branches, spokes, and internet egress.
  - Use this when you want global transit and simplified branch connectivity.
- **Perimeter or DMZ-style design**
  - Use Azure Firewall as the controlled ingress/egress point for published services.
  - Combine with Application Gateway WAF when you need HTTP/S request filtering.
- **Hybrid transit design**
  - Place the firewall in the transit path between on-premises and Azure spokes.
  - Ensure return traffic follows the same path to avoid asymmetric routing.

A practical rule: place Azure Firewall where you can control the most traffic with the fewest route exceptions. Centralized placement reduces policy sprawl and improves logging consistency.

**Hub-and-spoke design considerations**

In a hub-and-spoke architecture, the hub usually contains shared services such as Azure Firewall, VPN Gateway, ExpressRoute Gateway, DNS, and monitoring. Spokes host application workloads.

Key design points:

- **Egress control**
  - Force spoke outbound traffic to the firewall using UDRs.
  - Use application rules for FQDN-based outbound access and network rules for IP/port-based traffic.
- **East-west control**
  - Route spoke-to-spoke traffic through the firewall if you need inspection.
  - Do not assume VNet peering alone provides security boundaries.
- **Shared services**
  - Keep the firewall in the hub so all spokes can consume the same policy.
- **Route propagation**
  - Be deliberate about route tables on spoke subnets and gateway subnets.
  - Avoid accidental bypass paths through peering or gateway routes.

Example spoke subnet route table for forced egress:

```bash
az network route-table route create \
  --resource-group rg-network \
  --route-table-name rt-spoke-app \
  --name default-to-firewall \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.0.0.4
```

This sends default internet-bound traffic to the firewall private IP. You typically associate this route table with workload subnets, not the firewall subnet itself.

**Virtual WAN secured hub considerations**

Virtual WAN changes the routing model. Instead of manually stitching many VNets and branches together, you use a secured virtual hub with Azure Firewall integrated into the hub.

Use this model when you need:

- Many branch sites or VPN connections
- Simplified transitive routing
- Centralized security for global connectivity
- Easier scale-out than a manually managed hub-and-spoke mesh

Practical design notes:

- The firewall is deployed in the secured hub, not in each spoke.
- Branch-to-branch, branch-to-spoke, and spoke-to-spoke traffic can be steered through the hub.
- Routing intent is managed through Virtual WAN routing tables and associations.
- You still need to validate return paths for hybrid flows.

A simplified conceptual flow:

```text
Branch/VPN/ER -> Virtual WAN Hub -> Azure Firewall -> Spoke VNet
Spoke VNet -> Azure Firewall -> Internet
Spoke A -> Azure Firewall -> Spoke B
```

If you use Virtual WAN, design the routing tables so that the intended traffic classes are associated with the correct hub route table. Misplaced associations can create bypasses or black holes.

**Forced tunneling and internet egress**

Forced tunneling means you direct internet-bound traffic away from the default Azure internet path and through a controlled next hop, often on-premises or a firewall. In Azure Firewall designs, this is a common requirement for compliance and inspection.

Two common forced tunneling models exist:

- **Firewall as the internet egress point**
  - Spokes send `0.0.0.0/0` to Azure Firewall.
  - Azure Firewall uses its public IPs for SNAT (Source Network Address Translation) on outbound internet traffic.
- **Firewall behind an upstream network device**
  - Azure Firewall sends internet-bound traffic to on-premises or another security appliance.
  - This is less common and requires careful route design.

For Azure Firewall forced tunneling, the management subnet is important:

- The firewall still needs platform connectivity for management and updates.
- In forced tunneling scenarios, Azure uses a dedicated management subnet to maintain control-plane connectivity.
- You must not place workload traffic in the management subnet.
- The management subnet is not a general-purpose transit subnet.

Design implications:

- Ensure the firewall subnet and management subnet are sized correctly.
- Do not block required platform traffic with overly restrictive routes or NSGs (Network Security Groups).
- Validate that outbound dependencies for the firewall itself are allowed in your design.

> If you force all traffic through a firewall without planning its own management connectivity, you can break updates, telemetry, or service operation.

**SNAT, public IPs, and outbound design**

Azure Firewall performs SNAT for outbound connections unless the traffic is exempted by private routing scenarios. This matters because the source IP seen by the destination may be the firewall public IP rather than the workload IP.

Design considerations:

- **Public IP count**
  - More public IPs increase available SNAT ports and reduce port exhaustion risk.
  - Use multiple public IPs for high-scale outbound workloads.
- **Outbound allowlists**
  - External services may need to allow the firewall public IPs, not the workload private IPs.
- **Troubleshooting**
  - If a destination sees the firewall IP, that is expected for internet egress.
  - If you need stable egress identity, use dedicated firewall public IPs.

Example outbound rule concept:

```text
Spoke subnet -> UDR default route -> Azure Firewall private IP
Azure Firewall -> SNAT -> Firewall public IP -> Internet destination
```

For high-volume outbound applications, monitor SNAT port usage and scale public IPs or redesign egress patterns if needed.

**Rule types and when to use them**

Azure Firewall uses three main rule categories:

- **NAT rules**
  - Translate inbound public traffic to private backend addresses.
  - Use for publishing services from the internet or from another routed network.
- **Network rules**
  - Control traffic by source, destination, protocol, and port.
  - Use for IP-based allow/deny scenarios, including non-HTTP protocols.
- **Application rules**
  - Control outbound HTTP, HTTPS, and MSSQL (Microsoft SQL Server) traffic by FQDN or FQDN tags.
  - Use when you want domain-based control rather than raw IP control.

Practical guidance:

- Use **application rules** for web destinations and SaaS endpoints when FQDN-based control is enough.
- Use **network rules** for protocols that do not fit application filtering, such as custom TCP/UDP services.
- Use **NAT rules** for inbound publishing when you need a firewall-managed entry point.

Example application rule collection:

```json
{
  "name": "app-allow-web",
  "priority": 200,
  "ruleCollectionType": "FirewallPolicyFilterRuleCollection",
  "action": {
    "type": "Allow"
  },
  "rules": [
    {
      "name": "allow-microsoft-update",
      "sourceAddresses": [
        "10.10.1.0/24"
      ],
      "protocols": [
        {
          "protocolType": "Https",
          "port": 443
        }
      ],
      "targetFqdns": [
        "*.windowsupdate.com",
        "*.update.microsoft.com"
      ]
    }
  ]
}
```

Example network rule collection:

```json
{
  "name": "net-allow-dns",
  "priority": 100,
  "ruleCollectionType": "FirewallPolicyFilterRuleCollection",
  "action": {
    "type": "Allow"
  },
  "rules": [
    {
      "name": "allow-dns-to-resolver",
      "sourceAddresses": [
        "10.10.1.0/24"
      ],
      "destinationAddresses": [
        "10.0.0.10"
      ],
      "destinationPorts": [
        "53"
      ],
      "ipProtocols": [
        "UDP"
      ]
    }
  ]
}
```

Example NAT rule collection concept:

```json
{
  "name": "nat-publish-web",
  "priority": 100,
  "ruleCollectionType": "FirewallPolicyNatRuleCollection",
  "action": {
    "type": "Dnat"
  },
  "rules": [
    {
      "name": "publish-app1",
      "sourceAddresses": [
        "*"
      ],
      "destinationAddresses": [
        "20.40.50.60"
      ],
      "destinationPorts": [
        "443"
      ],
      "translatedAddress": "10.10.2.4",
      "translatedPort": "443",
      "ipProtocols": [
        "TCP"
      ]
    }
  ]
}
```

The exact schema can vary by API version, but the important design point is that NAT, network, and application collections are distinct and should not be modeled as the same type.

**Rule collection group priority and evaluation**

Azure Firewall Policy uses rule collection groups to organize rules at a higher level. Within and across groups, priority determines evaluation order.

Design principles:

- Lower numeric priority values are evaluated first.
- Keep NAT, network, and application rules logically separated.
- Use consistent naming and priority bands for operational clarity.
- Avoid overlapping rules that make troubleshooting ambiguous.

A common priority strategy:

| Priority band | Use |
|---|---|
| 100-199 | Critical NAT publishing rules |
| 200-399 | Shared network allow rules |
| 400-599 | Application allow rules |
| 600+ | Exception or lower-trust rules |

Example conceptual ordering:

1. NAT rule collection group for inbound publishing
2. Network rule collection group for DNS, NTP (Network Time Protocol), and private service access
3. Application rule collection group for outbound web access
4. Deny or exception groups as needed

Remember that rule evaluation is not just about one list. You need to understand the order of rule collection groups, the order of collections inside a group, and the order of rules inside a collection.

**Policy inheritance and scope**

Azure Firewall Policy supports hierarchical design. A parent policy can provide baseline controls, and child policies can add more specific rules.

Use inheritance when you want:

- A corporate baseline for all regions or business units
- Local exceptions for a specific environment
- Separation of central security governance from application team needs

Important scope behaviors:

- Parent policy rules are inherited by child policies.
- Child policies can add rules, but they should not be used to weaken the central security baseline.
- Local firewall rules may exist, but policy-based management is preferred for consistency.
- Design your governance model so that local exceptions are limited and auditable.

Practical guidance:

- Put global deny and mandatory allow rules in the parent policy.
- Put workload-specific allows in child policies.
- Use naming conventions that identify the owning team and environment.
- Review whether local rules are allowed in your governance model; many enterprises restrict them to avoid drift.

**Service tags, FQDN tags, and application control**

Azure Firewall supports service tags and FQDN tags to simplify rule management.

Use them when:

- You want to avoid maintaining long IP allowlists.
- You need to allow Microsoft service endpoints that change over time.
- You want a simpler rule set for common platform dependencies.

Examples:

- **Service tags**
  - Useful in network rules for Azure platform services.
  - Reduce maintenance compared with static IP ranges.
- **FQDN tags**
  - Useful in application rules for common Microsoft service categories.
  - Help with Microsoft update and platform access scenarios.

Example application rule using an FQDN tag:

```json
{
  "name": "allow-windows-update",
  "sourceAddresses": [
    "10.10.1.0/24"
  ],
  "protocols": [
    {
      "protocolType": "Https",
      "port": 443
    }
  ],
  "fqdnTags": [
    "WindowsUpdate"
  ]
}
```

Use tags carefully:

- They simplify operations, but they can broaden access more than a tightly scoped FQDN list.
- Validate that the tag matches your compliance requirements.

**Asymmetric routing and return-path issues**

Asymmetric routing occurs when traffic takes one path outbound and a different path inbound. Firewalls are stateful, so asymmetric paths can break sessions or cause intermittent failures.

Common causes:

- Spoke-to-spoke traffic bypassing the firewall on return
- On-premises return traffic not following the same transit path
- Overlapping UDRs and propagated routes
- Direct peering paths that bypass the intended inspection point

How to avoid it:

- Ensure both directions of a flow traverse the firewall when stateful inspection is required.
- Review route tables on all participating subnets.
- Validate gateway route propagation and peering settings.
- Test hybrid flows, not just internet egress.

Example troubleshooting mindset:

- If a connection works one way but fails on return, inspect the effective routes.
- If logs show the outbound flow but not the return, the return path may be bypassing the firewall.
- For hybrid designs, confirm that on-premises routers know how to return traffic to the firewall path.

**Availability, resiliency, and zone design**

Azure Firewall is a regional service, so design for regional resiliency and zone support where available.

Consider:

- **Availability zones**
  - Use zone-redundant deployment when the region supports it and your SLA or resilience goals require it.
  - Zone redundancy helps protect against datacenter-level failures.
- **Regional placement**
  - Deploy the firewall in the same region as the hub or secured hub to minimize latency and routing complexity.
- **High availability**
  - Azure Firewall is managed and scales automatically, but your architecture still needs resilient routing and redundant connectivity.
- **Gateway dependencies**
  - If the firewall is part of a hybrid transit design, ensure VPN or ExpressRoute redundancy matches the firewall design.

Design tip: a highly available firewall is not enough if the hub, gateway, or route tables create a single point of failure.

**Logging, monitoring, and Microsoft Defender for Cloud**

Operational visibility is essential for firewall governance and incident response.

Use:

- **Azure Monitor**
  - Centralize metrics and logs.
  - Build alerts for deny spikes, SNAT exhaustion, or unexpected rule hits.
- **Diagnostic settings**
  - Send firewall logs to Log Analytics, Event Hub, or Storage.
  - Keep logs long enough for investigations and compliance.
- **Microsoft Defender for Cloud**
  - Use it to improve security posture and detect misconfigurations.
  - Combine firewall telemetry with broader cloud security recommendations.

Useful log categories often include:

- Application rule logs
- Network rule logs
- NAT rule logs
- Threat intelligence logs
- DNS proxy logs, if enabled

Example diagnostic configuration command:

```bash
az monitor diagnostic-settings create \
  --name fw-diag \
  --resource /subscriptions/<subId>/resourceGroups/rg-network/providers/Microsoft.Network/azureFirewalls/fw-hub \
  --workspace /subscriptions/<subId>/resourceGroups/rg-monitor/providers/Microsoft.OperationalInsights/workspaces/law-sec \
  --logs '[{"category":"AzureFirewallApplicationRule","enabled":true},{"category":"AzureFirewallNetworkRule","enabled":true},{"category":"AzureFirewallNatRule","enabled":true}]'
```

For KQL (Kusto Query Language), prefer queries that match the actual table or resource-specific schema in your workspace rather than relying on a single hard-coded resource type filter.

**Practical CLI examples**

Create a firewall policy:

```bash
az network firewall policy create \
  --resource-group rg-network \
  --name fwpol-hub \
  --location eastus
```

Create a firewall and associate the policy, if supported by your CLI version and deployment model:

```bash
az network firewall create \
  --resource-group rg-network \
  --name fw-hub \
  --location eastus \
  --firewall-policy fwpol-hub
```

Create a rule collection group:

```bash
az network firewall policy rule-collection-group create \
  --resource-group rg-network \
  --policy-name fwpol-hub \
  --name rcg-hub-egress \
  --priority 200
```

Add an application rule collection:

```bash
az network firewall policy rule-collection-group collection add-filter-collection \
  --resource-group rg-network \
  --policy-name fwpol-hub \
  --rcg-name rcg-hub-egress \
  --name app-allow-web \
  --priority 300 \
  --action Allow
```

These commands vary by Azure CLI version, so verify the current syntax in your environment before using them in automation.

**Exam-focused design checklist**

When you see an AZ-305 scenario, evaluate these points:

- Which SKU matches the security and compliance requirement?
- Is the firewall in a hub-and-spoke, secured virtual hub, or perimeter design?
- Do you need forced tunneling, and how will the management subnet stay reachable?
- Are routes symmetric for hybrid and spoke-to-spoke traffic?
- Do you need NAT, network, or application rules?
- Should you use service tags or FQDN tags to reduce maintenance?
- Is policy inheritance required across regions or business units?
- Are availability zones and regional resiliency addressed?
- Are logs and Defender for Cloud integrated into the design?
- Will SNAT and public IP usage affect outbound allowlists or scale?

**Common mistakes to avoid**

- Treating Azure Firewall as a replacement for WAF
- Using the wrong SKU for TLS inspection or advanced threat needs
- Forgetting to route spoke traffic through the firewall
- Ignoring return-path symmetry in hybrid designs
- Overusing local rules instead of policy governance
- Assuming one public IP is enough for high-scale outbound traffic
- Mixing NAT, network, and application rule concepts in the same collection
- Designing forced tunneling without accounting for firewall management connectivity

**Quick mental model**

Think of Azure Firewall as the centralized traffic control point for Azure network security:

- **Placement** decides what traffic you can inspect.
- **Policy** decides what traffic you allow or deny.
- **Routing** decides whether traffic reaches the firewall at all.
- **Monitoring** tells you whether the design works in practice.

If any one of those four is wrong, the architecture can fail even if the firewall itself is configured correctly.
