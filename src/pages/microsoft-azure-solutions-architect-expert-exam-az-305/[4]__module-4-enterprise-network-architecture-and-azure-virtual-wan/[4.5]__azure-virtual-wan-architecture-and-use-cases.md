## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Azure Virtual WAN overview**

Azure Virtual WAN is a managed networking service for large-scale branch, remote user, and multi-region connectivity. It reduces the operational burden of building and maintaining a traditional hub-and-spoke network by centralizing transit, routing, and optional security in Microsoft-managed hubs.

Use Virtual WAN when you need:

- Many branches or sites connected through a common transit layer
- Remote user VPN access at scale
- Multi-region connectivity with centralized routing
- Integrated security inspection through Azure Firewall in a secured hub
- Simplified operations compared with manually peered hub-and-spoke topologies

Virtual WAN is not just a single gateway. It is an architecture built from a **virtual WAN** resource, one or more **virtual hubs**, and attached connectivity resources such as VPN, ExpressRoute, and point-to-site gateways.

**Core resource model**

The main resource types are easy to confuse, so separate them clearly:

- **Virtual WAN**: The top-level container and policy/management plane for the overall network.
- **Virtual hub**: A Microsoft-managed transit hub deployed in a region and associated with a virtual WAN.
- **Secured virtual hub**: A virtual hub with integrated security, typically Azure Firewall, and routing intent configured for inspection.
- **Hub connections**: Attachments from VNets, VPN sites, ExpressRoute circuits, and users into the hub.

A virtual WAN can contain multiple hubs across regions. Each hub is regional, and the design often uses more than one hub for resilience and locality.

**Hub types and how they differ**

A common exam trap is mixing up the WAN resource with the hub and the secured hub.

- **Virtual WAN** is the parent resource.
- **Virtual hub** provides transit and routing.
- **Secured virtual hub** adds centralized security services and traffic steering.

In practice, you choose the hub type based on whether you need only transit or transit plus inspection. A secured hub is appropriate when you want to centralize internet-bound and private traffic through Azure Firewall for policy enforcement and logging.

> A secured virtual hub is not a generic policy engine for arbitrary traffic steering. Routing intent is used to direct supported traffic flows, especially internet-bound and private traffic, through Azure Firewall in the hub.

**Connectivity patterns**

Virtual WAN supports several major connectivity patterns:

- **Branch-to-Azure**: On-premises sites connect through VPN or ExpressRoute into the hub.
- **User-to-Azure**: Remote users connect with point-to-site VPN.
- **VNet-to-VNet transit**: Connected VNets can communicate through hub routing when route association and propagation allow it.
- **Branch-to-branch**: Branch sites can communicate through the hub when branch-to-branch behavior is enabled and routing supports it.
- **Multi-region transit**: Hubs in different regions can exchange routes and support global connectivity in Standard Virtual WAN designs.

The key design idea is that the hub acts as a transit point. You do not need full mesh peering between every VNet or every branch.

**Virtual WAN SKUs and feature scope**

Virtual WAN has feature tiers that affect what you can deploy. For exam purposes, focus on the practical distinction:

- **Basic**: Limited feature set for simpler scenarios.
- **Standard**: Required for advanced transit, multi-region, and security-centric designs.

Do not memorize an oversimplified capability table. Instead, remember that Standard is the tier used for most enterprise architecture questions because it supports the richer hub, routing, and security features needed for production transit designs.

**When to choose Virtual WAN**

Choose Virtual WAN when:

- You have many branches or remote users and want centralized management.
- You need a managed global transit architecture.
- You want to reduce the number of custom route tables, peering relationships, and gateway deployments.
- You plan to use Azure Firewall in a centralized hub.
- You need a design that scales across regions with less operational overhead.

Choose a traditional hub-and-spoke design when:

- You have a smaller environment and want full control over routing and appliances.
- You already rely on custom NVAs and bespoke routing logic.
- You need very specific traffic inspection or segmentation patterns that are easier to express with UDRs and peering.

**Virtual WAN versus traditional hub-and-spoke**

Virtual WAN and traditional hub-and-spoke solve similar problems, but they optimize for different tradeoffs.

| Aspect | Virtual WAN | Traditional hub-and-spoke |
|---|---|---|
| Operations | Managed transit and routing | More manual design and maintenance |
| Scale | Better for many branches/users/regions | Better for smaller or highly customized environments |
| Routing | Hub-based transit with propagation/association | UDRs, peering, and gateway design are manually controlled |
| Security | Secured hub with Azure Firewall integration | Azure Firewall or NVA placed in a custom hub |
| Flexibility | Strong for standardized enterprise patterns | Strong for bespoke network control |

Use Virtual WAN when the architecture benefits from managed transit. Use traditional hub-and-spoke when you need granular control and are willing to manage the complexity.

**Branch connectivity and VPN concepts**

Branch connectivity in Virtual WAN is built around VPN sites and site links.

- A **VPN site** represents an on-premises location.
- A **site link** represents a physical or logical connection from that site to Azure.
- A branch connection is created from the site into the virtual hub.
- Multiple links can support redundancy and path selection.

For architecture questions, remember that branch connectivity is not just “turn on routing.” It depends on the site definition, hub connection, route propagation, and whether branch-to-branch communication is enabled.

A typical branch design includes:

- One or more VPN sites per location
- One or more site links per site for redundancy
- A connection from each site to the virtual hub
- Route propagation to the appropriate hub route table
- Optional branch-to-branch connectivity if sites must talk to each other

**VPN authentication and remote user access**

Point-to-site (P2S) VPN in Virtual WAN supports common authentication models used in enterprise designs:

- **Certificate-based authentication** for smaller or controlled deployments
- **RADIUS** for integration with existing authentication infrastructure
- **Microsoft Entra ID** authentication for modern identity-based access

Use P2S when you need remote user access without deploying a full branch site. In design questions, P2S is often the right answer for contractors, mobile users, or temporary access scenarios.

A typical P2S decision depends on:

- Identity integration requirements
- Client platform support
- Certificate lifecycle management
- Whether you need MFA through your identity provider

**ExpressRoute integration**

Virtual WAN integrates with ExpressRoute for private connectivity from on-premises networks into Azure.

Important concepts:

- An **ExpressRoute circuit** provides the private WAN connection.
- The circuit is connected to the virtual hub through an **ExpressRoute gateway** in the hub.
- The hub becomes the transit point for private connectivity into connected VNets and, where configured, other hubs.

Design considerations:

- Use ExpressRoute when you need private, predictable connectivity with enterprise WAN integration.
- Use Virtual WAN when you want to manage multiple ExpressRoute and VPN branches through a common transit architecture.
- Plan for route propagation carefully so that private prefixes are advertised only where needed.
- Consider bandwidth, redundancy, and regional placement of the hub relative to the circuit provider and workloads.

A common flow is:

1. On-premises network connects to the ExpressRoute circuit.
2. The circuit is associated with the virtual hub through the ExpressRoute gateway.
3. The hub learns on-premises routes.
4. Connected VNets and other allowed networks receive those routes through hub propagation.

**Routing model and route tables**

Routing in Virtual WAN is based on hub route tables, associations, and propagation.

- **Association** determines which route table a connection uses for forwarding decisions.
- **Propagation** determines which routes a connection advertises into a route table.
- **Hub route tables** control how traffic is exchanged between connected resources.

This model is central to exam questions because it determines whether traffic can flow between branches, VNets, and on-premises networks.

Key points:

- Route propagation is not the same as automatic full mesh connectivity.
- Traffic only flows where the route tables and associations allow it.
- Segmentation is achieved by using separate route tables and selective propagation.
- Default routing behavior may differ depending on the connection type and hub configuration.

A practical mental model is:

- If a connection learns a route, it can send traffic toward that destination.
- If a connection is associated with the wrong route table, traffic may not follow the intended path.
- If propagation is missing, the destination may never be learned.

**Branch-to-branch behavior**

Branch-to-branch connectivity allows branch sites connected to the same virtual hub to communicate through the hub.

Use it when:

- Branches need to exchange traffic directly through Azure
- You want centralized transit instead of backhauling through a datacenter
- You need site-to-site communication without building a mesh of VPN tunnels

Be precise: branch-to-branch behavior depends on the hub routing model, connection settings, and route table configuration. It is not simply a generic “enable propagation” switch.

**VNet connectivity and transitive routing**

VNets connect to the virtual hub through hub connections. The hub can then provide transitive routing between connected VNets and other attached networks when routing is configured appropriately.

Important nuance:

- Transitive routing is a design goal of Virtual WAN.
- It is not automatic in every case.
- Route association and propagation still matter.
- Security and segmentation requirements may intentionally prevent some transitive paths.

Use Virtual WAN when you want to avoid building many direct VNet peerings. The hub becomes the transit layer, but you still control which networks can reach each other.

**Hub-to-hub connectivity**

Hub-to-hub connectivity is a Standard Virtual WAN capability used in multi-region designs. It allows hubs in different regions to exchange routes and support global transit.

Use it when:

- You have workloads in multiple Azure regions
- You need regional resilience and global reach
- You want branch or VNet traffic to reach resources in another region through the WAN fabric

Design considerations:

- Place hubs near the workloads and users they serve.
- Use hub-to-hub connectivity for regional failover and global transit.
- Validate route propagation so that only intended prefixes are shared.
- Consider latency and data transfer costs when traffic crosses regions.

**Secured virtual hub and routing intent**

A secured virtual hub combines transit with centralized inspection, typically using Azure Firewall.

**Routing intent** is used to steer supported traffic classes through the firewall:

- Internet-bound traffic
- Private traffic that should be inspected centrally

This is different from custom route tables or user-defined routes (UDRs):

- **Routing intent** is a hub-level feature for supported traffic steering.
- **Custom route tables** control hub route propagation and association.
- **UDRs** are subnet-level routes used in VNets.

Use routing intent when you want a managed way to send traffic through Azure Firewall in a secured hub. Use UDRs when you need subnet-specific control inside a VNet.

A common secured hub pattern is:

- Branches and VNets connect to the hub
- The hub learns routes from those connections
- Routing intent sends selected traffic to Azure Firewall
- Firewall policies inspect and log traffic
- Allowed traffic continues to its destination

> Routing intent is not a replacement for all routing design. It complements hub route tables and does not eliminate the need to plan propagation and segmentation.

**Practical end-to-end design example**

Scenario: You have two branches, one remote user population, and two VNets in different regions. You want centralized inspection and branch-to-branch communication.

A workable design is:

1. Deploy a Standard Virtual WAN.
2. Create two virtual hubs, one per region.
3. Enable secured hub functionality where inspection is required.
4. Deploy Azure Firewall in the secured hub.
5. Create VPN sites for each branch.
6. Add site links for each branch connection.
7. Connect the branches to the nearest hub.
8. Connect VNets to the appropriate regional hub.
9. Configure route table association and propagation so branches learn VNet prefixes and VNets learn branch prefixes.
10. Enable branch-to-branch connectivity if site-to-site communication is required.
11. Configure routing intent so internet-bound and private traffic that must be inspected is sent through Azure Firewall.
12. Validate routes from each connection and test failover between hubs if multi-region resilience is required.

This pattern gives you centralized transit, optional inspection, and a cleaner operational model than a fully manual mesh.

**CLI examples**

Use the Azure CLI to create and connect Virtual WAN resources. Exact parameters vary by scenario, so validate the current command help before deployment.

Create a virtual WAN:

```bash
az network vwan create \
  --resource-group rg-network \
  --name vwan-prod \
  --location eastus \
  --type Standard
```

Create a virtual hub:

```bash
az network vhub create \
  --resource-group rg-network \
  --name hub-eastus \
  --vwan vwan-prod \
  --address-prefix 10.100.0.0/24 \
  --location eastus
```

Create a VPN site with one public IP address:

```bash
az network vpn-site create \
  --resource-group rg-network \
  --name branch-seattle \
  --location eastus \
  --virtual-wan vwan-prod \
  --ip-addresses 203.0.113.10 \
  --address-prefixes 10.10.0.0/16
```

Create a VPN site link:

```bash
az network vpn-site link create \
  --resource-group rg-network \
  --vpn-site-name branch-seattle \
  --name link1 \
  --link-speed 100 \
  --ip-address 203.0.113.10
```

> CLI syntax can vary by Azure CLI version and resource type. For exam prep, focus on the resource relationships and required inputs rather than memorizing every flag.

**Example configuration snippets**

A VPN device configuration often includes IKE/IPsec parameters, BGP, and tunnel endpoints. The exact vendor syntax varies, but the architecture usually maps to:

```text
tunnel-group <azure-peer-ip> type ipsec-l2l
tunnel-group <azure-peer-ip> ipsec-attributes
 ikev2 remote-authentication pre-shared-key <shared-key>
 ikev2 local-authentication pre-shared-key <shared-key>
```

For route control inside a VNet, you may still use UDRs when you need subnet-specific behavior:

```json
{
  "name": "route-to-firewall",
  "properties": {
    "addressPrefix": "0.0.0.0/0",
    "nextHopType": "VirtualAppliance",
    "nextHopIpAddress": "10.0.1.4"
  }
}
```

Use UDRs carefully in Virtual WAN designs because they can override or complicate hub-based routing intent.

**Design tradeoffs and exam-relevant limitations**

Virtual WAN is powerful, but it is not always the best answer.

Consider these constraints:

- Feature availability depends on the SKU and region.
- Not every region or feature combination is available everywhere.
- Some designs require Standard Virtual WAN for hub-to-hub, secured hub, or advanced routing.
- Large-scale designs still need route planning to avoid unintended propagation.
- Service limits exist for hubs, connections, routes, and gateways, so very large deployments need capacity planning.

Architectural tradeoffs:

- Virtual WAN simplifies operations but reduces some low-level control.
- Traditional hub-and-spoke gives you more control but increases management overhead.
- Azure Firewall in a secured hub centralizes inspection but may add latency and cost.
- ExpressRoute improves private connectivity but requires circuit planning and provider coordination.
- P2S is flexible for users but depends on identity and client management.

**Common AZ-305 decision cues**

Look for these phrases in exam questions:

- “Many branches” or “global branch connectivity” → Virtual WAN
- “Remote users at scale” → P2S in Virtual WAN
- “Centralized inspection” → Secured virtual hub with Azure Firewall
- “Multi-region transit” → Standard Virtual WAN with hub-to-hub connectivity
- “Private enterprise WAN integration” → ExpressRoute with Virtual WAN
- “Need minimal operational overhead” → Virtual WAN
- “Need highly customized routing” → Traditional hub-and-spoke may fit better

**Troubleshooting and validation**

When a Virtual WAN design does not work, check the following:

- The correct SKU is deployed for the required feature.
- The hub exists in the expected region.
- The connection is associated with the intended route table.
- Route propagation is enabled where needed.
- Branch-to-branch is enabled if site-to-site traffic is expected.
- The VPN site and site links are defined correctly.
- ExpressRoute circuit association is complete.
- Firewall policy and routing intent do not block the path.
- Effective routes on the VNet or connection match the intended design.

Useful validation commands include:

```bash
az network vhub route-table list \
  --resource-group rg-network \
  --hub-name hub-eastus
```

```bash
az network vnet show-effective-route-table \
  --resource-group rg-app \
  --name vnet-app
```

**Mental model for architecture questions**

Think of Virtual WAN as a managed transit fabric:

- The **WAN** is the container.
- The **hub** is the regional transit point.
- The **connections** are the spokes and external networks.
- The **route tables** decide who learns what.
- The **secured hub** adds inspection.
- The **routing intent** steers supported traffic through the firewall.

If you remember that model, you can reason through most AZ-305 scenarios without memorizing every feature detail.
