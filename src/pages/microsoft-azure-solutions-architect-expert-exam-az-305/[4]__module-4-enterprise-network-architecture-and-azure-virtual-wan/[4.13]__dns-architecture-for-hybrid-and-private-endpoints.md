## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**DNS architecture goals in Azure**

DNS in Azure is not just about resolving names. In hybrid and private endpoint designs, you use DNS to control *where* traffic goes, *which* IP address clients see, and *whether* a name resolves to a public or private destination. For AZ-305, the key is understanding the resolution path, not just the record type.

Azure DNS architecture usually combines:

- **Azure-provided DNS** for default resolution inside VNets
- **Custom DNS servers** for enterprise control and hybrid forwarding
- **Azure DNS Private Zones** for private name resolution
- **Azure DNS Private Resolver** for managed inbound and outbound forwarding
- **Private endpoints** for private access to PaaS services

The design choice depends on whether clients are:

- In Azure
- On-premises
- In another cloud
- Using centralized or distributed DNS

**Azure-provided DNS and custom DNS**

Azure-provided DNS is the default resolver behavior for resources in a virtual network (VNet) when you do not specify custom DNS servers. It resolves:

- Hostnames of Azure VMs in the same VNet
- Names from linked Azure Private DNS zones
- Public internet names through Azure’s recursive infrastructure

When you configure custom DNS servers on a VNet, Azure-provided DNS is bypassed for clients using that VNet’s DHCP-provided DNS settings. That means:

- Azure VMs use the DNS server IPs handed out by the VNet
- The custom DNS server becomes the first resolver hop
- Your DNS server must forward or resolve Azure-specific names correctly

This is a common exam trap: if you set custom DNS on a VNet, Azure does not “mix in” Azure-provided DNS automatically for the VM. Your DNS server must be able to resolve both internal and external names, often by forwarding to Azure DNS or another recursive resolver.

**Typical resolution behavior**

- With Azure-provided DNS:
  - VM queries Azure’s built-in resolver
  - Azure resolves linked private zones and public names
- With custom DNS:
  - VM queries your DNS server
  - Your DNS server may forward Azure-specific queries to Azure DNS Private Resolver or another upstream resolver
- With on-premises clients:
  - On-prem DNS usually forwards Azure private names to Azure via conditional forwarders

**Azure DNS Private Zones**

Azure Private DNS zones provide name resolution for private resources inside Azure. They are linked to VNets so that resources in those VNets can resolve records in the zone.

Common uses:

- Private endpoint name resolution
- VM internal name resolution across VNets
- Centralized DNS for Azure workloads

A private zone is not reachable from the public internet. Resolution works only when the querying client is in a linked VNet or reaches the zone through a DNS forwarding path.

Common service zones include:

| Service | Common private DNS zone |
|---|---|
| Azure Storage blob | `privatelink.blob.core.windows.net` |
| Azure Storage file | `privatelink.file.core.windows.net` |
| Azure SQL Database | `privatelink.database.windows.net` |
| Azure Key Vault | `privatelink.vaultcore.azure.net` |
| Azure App Service | `privatelink.azurewebsites.net` |
| Azure Cosmos DB | `privatelink.documents.azure.com` |

Some services require multiple zones or subresource-specific zones. Always match the zone to the service and subresource you are using. Do not assume one generic private zone works for every PaaS service.

**Private endpoint DNS flow**

A private endpoint gives a PaaS service a private IP address in your VNet. DNS must direct clients to that private IP instead of the public service endpoint.

A common pattern is:

1. Client queries the public service FQDN, such as `mystorage.blob.core.windows.net`
2. The public DNS response includes a CNAME to the `privatelink` name
3. The `privatelink` name resolves to the private endpoint IP through a linked private DNS zone
4. The client connects to the private IP over the VNet or hybrid path

This is common, but not universal for every service. The exact DNS chain varies by service and subresource, so verify the service documentation when designing or troubleshooting.

> Do not present the `privatelink` name as the normal client query target unless you have a specific reason. In most deployments, clients use the public FQDN and DNS redirects them to the private name behind the scenes.

**Private endpoint deployment and DNS automation**

When you create a private endpoint, you usually also need DNS integration. In practice, that means:

- Creating or using the correct private DNS zone
- Linking the zone to the VNets that need resolution
- Creating a private DNS zone group on the private endpoint when supported

Zone groups are the standard automation mechanism for many private endpoint scenarios. Manual record creation is usually a fallback when automation is unavailable or when you need custom control.

Example CLI pattern:

```bash
az network private-endpoint create \
  --name pe-mystorage \
  --resource-group rg-network \
  --vnet-name vnet-app \
  --subnet pe-subnet \
  --private-connection-resource-id /subscriptions/<subId>/resourceGroups/rg-app/providers/Microsoft.Storage/storageAccounts/mystorage \
  --group-ids blob \
  --connection-name pe-mystorage-conn
```

Private DNS zone group example:

```bash
az network private-endpoint dns-zone-group create \
  --resource-group rg-network \
  --endpoint-name pe-mystorage \
  --name default \
  --private-dns-zone privatelink.blob.core.windows.net \
  --zone-name privatelink.blob.core.windows.net
```

CLI syntax can vary by Azure CLI version and resource type, so verify the current command reference before using it in production.

**Azure DNS Private Resolver**

Azure DNS Private Resolver is a managed service for hybrid DNS forwarding. It helps you move DNS queries between on-premises networks and Azure without deploying your own DNS VMs.

It has two main endpoint types:

- **Inbound endpoint**: receives DNS queries from on-premises or other networks and resolves them using Azure DNS capabilities and linked private zones
- **Outbound endpoint**: sends DNS queries from Azure to external DNS servers through forwarding rules

Think of it as a managed forwarding bridge, not a replacement for every DNS server role.

**How the inbound endpoint works**

The inbound endpoint listens on private IP addresses in a delegated subnet. On-premises DNS servers can forward queries for Azure private names to those IPs.

Important nuance:

- The inbound endpoint is a listener and resolution entry point
- It is not a general-purpose recursive resolver for arbitrary internet DNS in the way a traditional DNS server might be
- It resolves names based on linked private zones and Azure DNS resolution behavior

**How the outbound endpoint works**

The outbound endpoint lets Azure workloads reach on-premises DNS servers or other DNS infrastructure through forwarding rulesets.

Typical use cases:

- Azure VMs need to resolve on-premises zones
- Azure workloads need split-horizon DNS
- You want centralized forwarding rules for multiple VNets

**Forwarding rulesets**

A forwarding ruleset defines which DNS suffixes are forwarded and to which target DNS servers. You associate the ruleset with VNets that should use it.

Example conceptual configuration:

```json
{
  "rules": [
    {
      "domainName": "corp.contoso.com.",
      "targetDnsServers": [
        { "ipAddress": "10.10.0.4", "port": 53 },
        { "ipAddress": "10.10.0.5", "port": 53 }
      ]
    }
  ]
}
```

**Subnet and deployment constraints**

Azure DNS Private Resolver requires dedicated subnets for its endpoints. In exam scenarios, remember these constraints:

- Use a dedicated subnet for the inbound endpoint
- Use a dedicated subnet for the outbound endpoint
- Do not place other resources in those subnets
- Size the subnets to allow for current and future IP needs
- Delegate the subnet to the private resolver service as required

A common design mistake is trying to reuse an application subnet. That is not the intended pattern.

**Hybrid resolution path example**

A typical hybrid flow looks like this:

1. An on-premises client queries `mystorage.blob.core.windows.net`
2. On-premises DNS has a conditional forwarder for `privatelink.blob.core.windows.net`
3. The query is forwarded to the Azure DNS Private Resolver inbound endpoint IPs
4. The inbound endpoint resolves the name using the linked private DNS zone
5. The response returns the private endpoint IP
6. The client connects over VPN or ExpressRoute to the private IP

If you use a custom DNS server on-premises, that server usually holds the conditional forwarder. The forwarder target is the inbound endpoint IP address, not the private endpoint IP itself.

**Azure-provided DNS vs custom DNS in practice**

The choice affects both Azure and hybrid clients.

- **Azure-provided DNS**
  - Best for simple Azure-only environments
  - Easy to use with linked private DNS zones
  - Limited control over split-horizon and enterprise forwarding
- **Custom DNS**
  - Best for enterprise naming standards and hybrid integration
  - Requires you to manage forwarding and recursion carefully
  - Can become a single point of failure if not designed redundantly

A common pattern is to use custom DNS servers in Azure that forward:

- Azure private zone queries to Azure DNS Private Resolver or Azure-provided resolution paths
- On-premises zone queries to on-prem DNS
- Public queries to a recursive resolver

**Hub-and-spoke DNS design**

Hub-and-spoke is the most common enterprise pattern.

In a centralized design:

- DNS infrastructure lives in the hub VNet
- Private DNS zones are linked to the hub and spoke VNets that need resolution
- Azure DNS Private Resolver endpoints are placed in the hub
- Spokes use custom DNS servers or forwarding paths through the hub

Benefits:

- Centralized control
- Easier governance
- Fewer duplicated DNS components

Trade-offs:

- More dependency on hub connectivity
- More planning for routing and forwarding
- Potential blast radius if the hub DNS path fails

In a distributed design:

- Each spoke may have its own DNS integration
- Private DNS zones may be linked to multiple VNets directly
- Useful for smaller environments or isolated workloads

Benefits:

- Simpler local autonomy
- Reduced dependency on a central hub

Trade-offs:

- More operational overhead
- Harder to standardize
- More zone links and forwarding rules to manage

**Where to place private DNS zones**

A practical rule is:

- Place private DNS zones in a central network subscription or shared services subscription when governance is centralized
- Link the zones to every VNet that needs to resolve them
- Use resolver forwarding to bridge on-premises and Azure

This is especially useful when multiple spokes need the same private endpoint names.

**Troubleshooting DNS resolution**

Start by identifying the client, the resolver, and the expected answer.

Useful commands:

```bash
nslookup mystorage.blob.core.windows.net
nslookup mystorage.privatelink.blob.core.windows.net
Resolve-DnsName mystorage.blob.core.windows.net
```

For Windows clients, `Resolve-DnsName` is often more informative than `nslookup` because it shows the full chain and record types.

For Linux clients:

```bash
dig mystorage.blob.core.windows.net
dig @10.10.0.4 corp.contoso.com
```

Common failure cases:

- Missing VNet link to the private DNS zone
- Private endpoint created but no zone group or record exists
- Wrong private DNS zone name for the service
- Custom DNS server not forwarding the private zone suffix
- Conditional forwarder points to the wrong inbound endpoint IP
- Outbound endpoint or forwarding ruleset not associated with the VNet
- DNS record conflict after recreating or moving a private endpoint
- Stale cached records after a private endpoint IP change

If a private endpoint changes, DNS propagation and client caching can delay the new answer. Flush caches where appropriate:

```bash
ipconfig /flushdns
```

**Exam-relevant design scenarios**

- If Azure VMs must resolve on-premises names, use custom DNS or Azure DNS Private Resolver outbound forwarding
- If on-premises clients must resolve Azure private endpoint names, use a conditional forwarder to the inbound endpoint
- If you need centralized control across many VNets, use a hub-and-spoke DNS design with shared private zones
- If you need minimal infrastructure, Azure-provided DNS plus linked private zones may be enough
- If you need split-horizon DNS and enterprise suffix forwarding, custom DNS plus Private Resolver is usually the cleanest design

**Decision guide**

| Scenario | Preferred pattern |
|---|---|
| Azure-only, simple private endpoints | Azure-provided DNS + private DNS zones |
| Hybrid with on-prem clients | Private Resolver inbound endpoint + conditional forwarders |
| Azure workloads need on-prem resolution | Private Resolver outbound endpoint + forwarding ruleset |
| Centralized enterprise DNS | Hub-and-spoke with shared zones and resolver endpoints |
| Small isolated workload | Local DNS integration with direct zone links |

**Practical configuration example**

A common enterprise setup uses:

- On-prem DNS server: `10.10.0.4`
- Azure DNS Private Resolver inbound endpoint IPs: `10.20.1.4`, `10.20.1.5`
- Private DNS zone: `privatelink.blob.core.windows.net`
- Hub VNet linked to the zone
- Spoke VNets linked to the zone if they need direct resolution

On-prem conditional forwarder example:

```powershell
Add-DnsServerConditionalForwarderZone `
  -Name "privatelink.blob.core.windows.net" `
  -MasterServers 10.20.1.4,10.20.1.5 `
  -ReplicationScope "Forest"
```

This sends queries for the private endpoint suffix to Azure, where the inbound endpoint can answer using the linked private zone.

**Key takeaways**

- Azure DNS architecture is about controlling resolution paths, not just creating records
- Azure-provided DNS is simple, but custom DNS is often required in enterprise and hybrid designs
- Private DNS zones are the core mechanism for private endpoint name resolution
- Azure DNS Private Resolver bridges on-premises and Azure DNS without self-managed DNS VMs
- Hub-and-spoke designs centralize DNS control, while distributed designs reduce dependency on a central hub
- For private endpoints, always verify the service-specific zone name and the full resolution chain
