## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Routing design overview**

Azure routing design is about controlling how packets move between subnets, virtual networks, on-premises networks, and security appliances. For AZ-305, you need to design for reachability, segmentation, inspection, resiliency, and operational simplicity at the same time.

Azure evaluates routes using the **effective route table** on each NIC. The effective result is built from multiple sources, including system routes, user-defined routes (UDRs), and dynamically learned routes such as Border Gateway Protocol (BGP) routes. The exact outcome depends on prefix specificity, route type, propagation settings, and the networking feature in use.

- **System routes** provide Azure’s default connectivity behavior.
- **UDRs** let you steer traffic to a virtual appliance, gateway, or other next hop.
- **BGP** adds dynamic routes from hybrid connectivity components.
- **Peering, service endpoints, and private endpoints** all influence routing and reachability in different ways.

**Core route types and next hop options**

Azure route tables can contain routes with these common next hop types:

- **Virtual network**: traffic stays inside the VNet address space.
- **Internet**: traffic is sent to the public internet.
- **Virtual network gateway**: traffic is sent to a VPN gateway or ExpressRoute gateway.
- **Virtual appliance**: traffic is sent to an NVA, such as Azure Firewall or a third-party appliance.
- **None**: traffic is dropped.
- **VNet peering-related paths**: peered VNets are reachable through peering system routes, not by manually creating a special peering next hop in a UDR.

A route table can also be associated with a subnet and can optionally disable BGP route propagation. That choice matters because it changes which learned routes appear in the effective route table.

**How Azure chooses a route**

Azure does not use a single simplistic rule for every case. In practice, route selection is driven by:

- **Longest prefix match first**
  - A `/24` beats a `/16` for the same destination.
- **Route source and type**
  - System routes, UDRs, and learned routes can all compete when prefix length is equal.
- **Feature-specific behavior**
  - Peering, gateway propagation, service endpoints, and private endpoints can add routes or alter reachability.

> Do not assume a universal “UDR always beats BGP always beats system” rule in every scenario. Azure routing behavior is more nuanced, especially when multiple route sources and platform features overlap.

**Effective routes and route association**

The effective route table is the practical truth for a NIC. It reflects what Azure will actually use after combining:

- Subnet association to a route table
- System routes for the VNet and Azure platform
- Peering routes
- BGP-learned routes, if propagation is enabled
- UDRs on the subnet
- Special routes for service endpoints and private endpoints

If a subnet has no route table, Azure still applies system routes and any propagated routes that are relevant. If a subnet does have a route table, the UDRs in that table influence traffic for resources in that subnet. If BGP propagation is disabled on the route table, learned routes from gateways do not appear in the effective route table for that subnet.

**Common Azure route behaviors**

- **VNet-local traffic**
  - Traffic between subnets in the same VNet uses the VNet-local system route.
  - This covers subnet-to-subnet traffic inside the same virtual network, not just traffic within one subnet.
- **Peered VNet traffic**
  - Azure adds routes for peered address spaces so VNets can communicate through peering when allowed.
  - Peering is non-transitive by default.
- **Gateway-learned traffic**
  - VPN Gateway and ExpressRoute Gateway can advertise routes into Azure through BGP.
- **Forced tunneling**
  - You can send internet-bound traffic to an NVA or on-premises network by advertising a default route or using a UDR for `0.0.0.0/0`.
  - Whether that route is used depends on propagation, prefix match, and the effective route table.
- **Blackholing**
  - A `None` next hop can intentionally drop traffic for a prefix, often for segmentation or containment.

**Practical route table example**

```bash
az network route-table create \
  --resource-group rg-network \
  --name rt-spoke-apps

az network route-table route create \
  --resource-group rg-network \
  --route-table-name rt-spoke-apps \
  --name default-to-firewall \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.0.1.4

az network vnet subnet update \
  --resource-group rg-network \
  --vnet-name vnet-spoke-01 \
  --name app-subnet \
  --route-table rt-spoke-apps
```

This pattern is common in hub-and-spoke designs where you want all outbound traffic from a spoke subnet inspected by a firewall in the hub.

**Hub-and-spoke routing design**

Hub-and-spoke is the most common Azure enterprise topology. The hub hosts shared services such as:

- Azure Firewall or another NVA
- VPN Gateway or ExpressRoute Gateway
- DNS forwarders
- Bastion
- Shared monitoring and security tooling

Spokes host workloads and usually do not talk directly to the internet or to each other without inspection.

Key design points:

- **Centralize inspection**
  - Use UDRs in spokes to send traffic to the hub firewall.
- **Control east-west traffic**
  - Use peering and UDRs to decide whether spoke-to-spoke traffic is allowed directly or must traverse the hub.
- **Keep routing simple**
  - Use summarized address spaces where possible to reduce route count and operational complexity.
- **Plan for transitivity**
  - VNet peering is not transitive, so spoke-to-spoke routing through the hub requires explicit design.

A common spoke UDR pattern is:

```text
0.0.0.0/0       -> VirtualAppliance (Azure Firewall)
10.0.0.0/16     -> VirtualNetworkGateway or VirtualAppliance, depending on design
10.1.0.0/16     -> VirtualAppliance
```

Use specific prefixes for exceptions rather than relying on broad defaults alone.

**Forced tunneling and internet egress**

Forced tunneling sends outbound traffic away from Azure’s default internet path. You typically use it when:

- Security policy requires all egress inspection
- Compliance requires centralized logging
- You want on-premises internet breakout
- You need a controlled proxy or firewall path

Design considerations:

- A default route from on-premises can influence Azure traffic if it is propagated and selected.
- A UDR for `0.0.0.0/0` is often used to steer traffic to an NVA.
- Exceptions may be needed for Azure platform dependencies, management endpoints, or private service access.
- Do not assume a UDR automatically overrides every learned route in every context; verify the effective route table.

**Azure Firewall and routing**

Azure Firewall is a stateful network virtual appliance used for centralized inspection. In routing designs, it commonly serves as the next hop for:

- Internet egress
- Spoke-to-spoke inspection
- On-premises traffic inspection
- Private endpoint or service access control, when combined with DNS and subnet design

Important design points:

- **Stateful inspection**
  - Return traffic must be consistent with the session and NAT/routing design so the firewall can track the flow.
- **SNAT behavior**
  - Source network address translation can simplify return routing for outbound flows.
- **Asymmetric routing**
  - If one direction bypasses the firewall, sessions can fail or behave unpredictably.
- **Policy and route alignment**
  - Firewall rules alone are not enough; the route table must send traffic through the firewall.

Example spoke-to-hub routing pattern:

```text
Spoke subnet UDR:
- 0.0.0.0/0 -> 10.0.1.4 (Azure Firewall)
- 10.2.0.0/16 -> 10.0.1.4 (other spoke prefixes)
- 10.255.0.0/16 -> VirtualNetwork (local services, if needed)
```

**VNet peering and gateway transit**

VNet peering connects VNets privately over the Microsoft backbone. It is fast and low-latency, but it is not transitive by default.

Key options:

- **Allow virtual network access**
  - Enables traffic between peered VNets.
- **Allow forwarded traffic**
  - Allows traffic that has already been forwarded by an NVA or gateway.
- **Allow gateway transit**
  - Lets the hub VNet share its gateway with peered spokes.
- **Use remote gateways**
  - Lets the spoke use the hub’s gateway for on-premises connectivity.

A realistic gateway transit design requires both sides to be configured correctly and the hub to already have a gateway.

```bash
# Hub peering: allow gateway transit
az network vnet peering create \
  --resource-group rg-network \
  --vnet-name vnet-hub \
  --name hub-to-spoke01 \
  --remote-vnet vnet-spoke-01 \
  --allow-vnet-access \
  --allow-gateway-transit

# Spoke peering: use remote gateway
az network vnet peering create \
  --resource-group rg-network \
  --vnet-name vnet-spoke-01 \
  --name spoke01-to-hub \
  --remote-vnet vnet-hub \
  --allow-vnet-access \
  --use-remote-gateways
```

**BGP, VPN Gateway, ExpressRoute, and Virtual WAN**

BGP is the protocol Azure uses to exchange routes dynamically with gateways and routing components.

Use BGP when you need:

- Dynamic route exchange
- Failover between paths
- Reduced manual UDR maintenance
- Hybrid connectivity with changing prefixes

**VPN Gateway**
- Learns and advertises routes over site-to-site VPN connections.
- Supports active-active designs for resilience.
- Can coexist with ExpressRoute in some architectures for failover or branch connectivity.

**ExpressRoute**
- Provides private connectivity to Microsoft networks.
- Uses private peering route exchange for private workloads.
- Can advertise on-premises prefixes into Azure and learn Azure prefixes back, subject to configuration and route limits.
- Route filters are relevant for Microsoft peering scenarios, where you control which Microsoft services are advertised.

**Virtual WAN**
- Uses a virtual hub routing model.
- Routes are propagated through hub routing and connected components such as VPN and ExpressRoute gateways.
- It is not accurate to think of “Virtual WAN” itself as a BGP route source in the same way a gateway is; the hub and its gateways are the routing participants.

**Route Server**

Azure Route Server helps integrate NVAs and dynamic routing into Azure without manually maintaining large numbers of UDRs.

Use it when:

- You need dynamic route exchange between an NVA and Azure
- You want the NVA to learn Azure routes and advertise routes back
- You are building advanced hub routing or branch connectivity patterns

Typical use cases:

- Third-party SD-WAN integration
- NVA-based transit routing
- Dynamic failover between appliances

Route Server is especially useful when static UDRs become hard to manage at scale.

**Route summarization and supernetting**

Summarization reduces route table size and operational complexity. It is important in large hub-and-spoke or branch environments.

Benefits:

- Fewer routes to manage
- Smaller effective route tables
- Easier troubleshooting
- Better scalability for many spokes or branches

Tradeoffs:

- Less granular control
- Harder exception handling
- Risk of over-broad routing if summaries are too large

Design guidance:

- Summarize contiguous address spaces where possible.
- Keep exception prefixes explicit.
- Avoid overlapping address plans that force many specific routes.
- Plan address space before deployment; retrofitting summaries is difficult.

**Service endpoints versus private endpoints**

These are often confused, but they solve different problems.

**Service endpoints**
- Extend VNet identity to a supported PaaS service.
- Traffic still uses the Azure backbone.
- The destination remains the service’s public endpoint.
- They do not create a private IP in your VNet.
- They are useful when you want to restrict service access to selected VNets without changing DNS to a private address.

**Private endpoints**
- Create a private IP in your VNet for the service.
- Require DNS to resolve the service name to the private IP.
- Are usually the preferred option for private access to PaaS services.
- Can fail if DNS resolves the public name instead of the private endpoint address.

Private endpoint design usually needs:

- Private DNS zone integration
- Correct VNet links
- Careful NSG and UDR review
- Awareness of peering and DNS forwarding paths

> Forced tunneling alone does not inherently break private endpoint access. Private endpoint issues are more often caused by DNS resolution, NSGs, UDRs, or peering/DNS forwarding misconfiguration.

Example private DNS zone setup:

```bash
az network private-dns zone create \
  --resource-group rg-network \
  --name privatelink.blob.core.windows.net

az network private-dns link vnet create \
  --resource-group rg-network \
  --zone-name privatelink.blob.core.windows.net \
  --name link-spoke01 \
  --virtual-network vnet-spoke-01 \
  --registration-enabled false
```

**DNS design for routing**

Routing and DNS are tightly coupled in Azure, especially for private endpoints and hybrid name resolution.

Design goals:

- Resolve private endpoint names to private IPs
- Forward on-premises queries to Azure when needed
- Avoid split-brain DNS mistakes
- Keep hub DNS services reachable from spokes and on-premises networks

Common patterns:

- Azure-provided DNS for simple environments
- Custom DNS servers in the hub for enterprise environments
- Conditional forwarders for private zones
- Azure DNS Private Resolver for managed inbound and outbound resolution

Practical DNS checks:

```bash
nslookup myaccount.blob.core.windows.net
nslookup myaccount.privatelink.blob.core.windows.net
```

If the public name resolves instead of the private IP, the route table may look correct while the application still fails.

**Common exam scenarios**

1. **Spoke-to-spoke traffic through a firewall**
   - Use hub-and-spoke peering.
   - Add UDRs in spokes for remote spoke prefixes to the firewall.
   - Ensure peering allows forwarded traffic.
   - Verify firewall rules and return path symmetry.

2. **Branch-to-branch routing in Virtual WAN**
   - Use hub routing and branch connectivity through the virtual hub.
   - Confirm propagation and routing intent.
   - Avoid assuming direct branch transitivity without the hub design.

3. **Forced tunneling with internet exceptions**
   - Send `0.0.0.0/0` to the firewall or on-premises gateway.
   - Add explicit routes for Azure management, private endpoints, or required SaaS destinations if needed.
   - Validate with effective routes and next hop tests.

4. **Private endpoint access from on-premises**
   - Configure DNS so on-premises clients resolve the private IP.
   - Ensure routing reaches the VNet where the private endpoint lives.
   - Check NSGs, firewall rules, and DNS forwarding.

5. **ExpressRoute and VPN coexistence**
   - Use one path as primary and the other as backup, or use both for different traffic classes.
   - Understand which prefixes are advertised over each path.
   - Verify route preference and failover behavior with effective routes.

**Troubleshooting routing issues**

Start with the effective route table, then validate DNS, NSGs, and appliance logs.

Useful tools:

- **Network Watcher next hop**
  - Shows which next hop Azure will use for a destination.
- **Network Watcher connection troubleshoot**
  - Tests reachability between source and destination.
- **Effective route table**
  - Shows the actual routes on a NIC.
- **Packet capture**
  - Helps confirm whether packets leave or return as expected.
- **Firewall logs**
  - Useful for verifying whether traffic reaches the NVA.

CLI examples:

```bash
az network nic show-effective-route-table \
  --resource-group rg-network \
  --name nic-app01
```

Depending on Azure CLI version and context, you may need to inspect the NIC in the portal or use Network Watcher workflows if the command behavior differs.

Next hop test example:

```bash
az network watcher show-next-hop \
  --resource-group rg-network \
  --vm vm-app01 \
  --source-ip 10.1.1.4 \
  --dest-ip 10.2.1.4 \
  --location eastus
```

Troubleshooting checklist:

- Confirm the destination prefix is not shadowed by a more specific route.
- Check whether BGP propagation is enabled on the route table.
- Verify peering settings on both VNets.
- Confirm the firewall or NVA has a return path.
- Validate DNS for private endpoints.
- Check NSGs on both source and destination subnets.
- Use packet capture if the packet leaves the source but never returns.

**Route scale and operational limits**

Large environments can hit operational complexity before they hit hard platform limits.

Watch for:

- Too many UDRs across many spokes
- Overlapping address spaces
- Excessive route propagation from hybrid links
- Long troubleshooting cycles caused by unclear ownership

Design for scale by:

- Using address summarization
- Standardizing spoke CIDRs
- Centralizing shared services
- Minimizing exception routes
- Automating route table deployment with IaC

Example Bicep snippet for a route table:

```bicep
resource rt 'Microsoft.Network/routeTables@2023-09-01' = {
  name: 'rt-spoke-apps'
  location: resourceGroup().location
  properties: {
    disableBgpRoutePropagation: false
    routes: [
      {
        name: 'default-to-fw'
        properties: {
          addressPrefix: '0.0.0.0/0'
          nextHopType: 'VirtualAppliance'
          nextHopIpAddress: '10.0.1.4'
        }
      }
    ]
  }
}
```

**Design principles to remember**

- Prefer the simplest route design that meets security and connectivity requirements.
- Use UDRs for explicit control, not as a substitute for poor address planning.
- Verify the effective route table instead of assuming the intended design is active.
- Treat DNS as part of routing design for private endpoints and hybrid name resolution.
- Use dynamic routing where scale and failover justify it.
- Keep peering, gateway transit, and firewall routing aligned so traffic does not become asymmetric.

**Quick comparison**

| Feature | What it does | Key routing impact |
|---|---|---|
| UDR | Overrides default behavior for a subnet | Steers traffic to firewall, gateway, or drop |
| BGP | Learns routes dynamically | Adds hybrid prefixes to effective routes |
| VNet peering | Connects VNets privately | Adds peering routes; not transitive by default |
| Service endpoint | Extends VNet identity to PaaS | Keeps public service endpoint, no private IP |
| Private endpoint | Gives service a private IP | Requires DNS and private routing design |
| Route Server | Exchanges routes with NVAs | Reduces static route management |

**Exam focus summary**

For AZ-305, be ready to choose the right routing pattern for the scenario:

- Use **hub-and-spoke** for centralized security and shared services.
- Use **gateway transit** when spokes need shared hybrid connectivity.
- Use **Azure Firewall** or an NVA when inspection and egress control are required.
- Use **Route Server** when dynamic NVA routing is needed.
- Use **private endpoints** for private PaaS access and pair them with DNS design.
- Use **service endpoints** when you need simpler PaaS restriction without private IPs.
- Use **ExpressRoute or VPN Gateway** with BGP when hybrid route exchange and failover matter.
- Use **summarization** to keep large designs manageable.
