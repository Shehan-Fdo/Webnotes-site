## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Hybrid connectivity overview**

Site-to-site (S2S) VPN and ExpressRoute are the two core Azure patterns for connecting on-premises networks to Azure. In AZ-305, you need to know not only what each service does, but how they behave together in a resilient architecture.

- **S2S VPN** uses IPsec/IKE tunnels over the public internet.
- **ExpressRoute** uses private connectivity through a connectivity provider or direct peering model.
- A common design uses ExpressRoute as the primary path and VPN as a backup path.
- Route selection, gateway SKU limits, and BGP behavior determine whether failover works as intended.

A practical hybrid design usually starts with a hub-and-spoke network. The hub contains the connectivity gateway and often a firewall or NVA (network virtual appliance). Spokes host workloads and use UDRs (user-defined routes) to control traffic flow.

> **Important:** ExpressRoute and VPN coexistence is supported only in specific gateway and topology configurations. Do not assume every hub VNet can use both paths simultaneously without checking SKU and design constraints.

**Core service differences**

ExpressRoute is designed for predictable latency, higher throughput, and private connectivity. VPN is simpler to deploy and is often used for smaller environments, temporary connectivity, or failover.

| Capability | Site-to-site VPN | ExpressRoute |
|---|---|---|
| Transport | Internet | Private circuit/provider path |
| Encryption | IPsec/IKE tunnel | Not encrypted end-to-end by default |
| Typical use | Backup, small sites, quick setup | Primary enterprise connectivity |
| Routing | Static or BGP | BGP required for most enterprise designs |
| Performance | Lower and variable | Higher and more predictable |

ExpressRoute is not a generic encryption service. If you need encryption over the circuit, you typically add application-layer encryption or a separate VPN/overlay solution. Do not describe Azure as providing a built-in overlay encryption layer for ExpressRoute.

**ExpressRoute design concepts**

ExpressRoute has several operational concepts that matter in architecture questions.

- **Private peering** carries traffic to private IPs in Azure VNets.
- **Microsoft peering** carries traffic to Microsoft SaaS and PaaS endpoints that support it.
- **Global Reach** connects two on-premises sites through the Microsoft backbone using ExpressRoute circuits.
- **FastPath** can bypass the gateway data path for supported scenarios to improve throughput and reduce latency.
- **BGP** (Border Gateway Protocol) is the routing protocol used to exchange routes with the circuit.

Private peering is the main path for VNet connectivity. Microsoft peering is used when you need access to supported Microsoft services over the circuit. In exam scenarios, distinguish these carefully because they solve different problems.

**ExpressRoute connectivity workflow**

A typical ExpressRoute deployment follows a sequence. The exact portal labels vary, but the architecture flow is consistent.

1. Create or obtain an ExpressRoute circuit from a connectivity provider or ExpressRoute Direct.
2. Configure peering on the circuit, including BGP settings and VLAN details where required.
3. Create a virtual network gateway of type `ExpressRoute` in the hub VNet.
4. Authorize the circuit to connect to the VNet gateway.
5. Link the circuit to the gateway connection.
6. Verify route propagation into the VNet and spokes.

A simplified example of the relationship looks like this:

```text
On-premises router
   |
   |  BGP over provider circuit
   v
ExpressRoute circuit ---- authorization ----> VNet gateway (hub)
                                                |
                                                +--> spoke VNets
```

A useful mental model is that the circuit and the gateway are separate resources. The circuit provides the private transport, while the gateway terminates the Azure side of the connection.

**VPN gateway design concepts**

VPN gateways are still important in AZ-305 because they provide backup connectivity and are often used with ExpressRoute in resilient designs.

- **Route-based VPN** is the standard choice for Azure-to-on-premises connectivity.
- **Policy-based VPN** is legacy and generally not preferred for modern designs.
- **BGP support** improves route exchange and failover behavior.
- **Active-active** gateways provide two instances and can improve availability.
- **IKEv1/IKEv2** compatibility matters when matching on-premises devices.

For enterprise designs, route-based VPN with BGP is usually the safest recommendation. It supports dynamic route exchange and makes failover between ExpressRoute and VPN more predictable.

Example Azure CLI for a route-based VPN gateway:

```bash
az network vnet-gateway create \
  --resource-group rg-hub \
  --name vng-hub \
  --public-ip-addresses pip-vng-1 pip-vng-2 \
  --vnet hub-vnet \
  --gateway-type Vpn \
  --vpn-type RouteBased \
  --sku VpnGw2AZ \
  --enable-bgp true
```

This example shows the key design choices:
- `RouteBased` for modern interoperability
- `VpnGw2AZ` for zone-redundant availability where supported
- `--enable-bgp true` for dynamic routing

**Coexistence patterns and limitations**

ExpressRoute and VPN can coexist, but the supported pattern depends on the gateway SKU and configuration. In exam terms, the important point is that coexistence is not automatic.

Common patterns include:
- **Primary ExpressRoute, backup VPN**
- **Separate gateways in different VNets**
- **Hub VNet with a gateway that supports both paths**
- **VPN-only branch sites that still reach Azure through the hub**

What you should remember:
- Not every gateway SKU supports both ExpressRoute and VPN on the same gateway.
- Some designs use separate gateways or separate hubs to meet availability or scale requirements.
- Gateway transit through peering can extend connectivity to spokes, but it does not remove gateway SKU constraints.

A hub-and-spoke example:

```text
                On-premises
               /           \
              /             \
      ExpressRoute         VPN backup
            |                 |
            +------ Hub VNet --+
                     |
          -------------------------
          |           |           |
       Spoke A     Spoke B     Spoke C
```

In this design, the hub is the routing and security anchor. Spokes typically do not host their own gateways unless there is a special requirement.

**Routing precedence and path selection**

When both ExpressRoute and VPN advertise the same prefix, Azure route selection depends on route specificity and learned routes. In general, the most specific prefix wins. If prefixes are equal, Azure prefers the route with the better path according to its routing rules and learned source.

Practical implications:
- Advertise consistent prefixes from on-premises to avoid asymmetric routing.
- Use more specific prefixes only when you intentionally want traffic to prefer one path.
- Validate BGP advertisements on both ExpressRoute and VPN so failover does not create black holes.

A common exam scenario is:
- ExpressRoute advertises `10.10.0.0/16`
- VPN advertises `10.10.0.0/16`
- Azure may prefer one learned path based on routing precedence, but you should not rely on ambiguity for design

Better design:
- Keep the same prefix set on both paths for failover
- Use route filtering or UDRs only when you need explicit steering
- Avoid overlapping advertisements from multiple on-premises devices unless you understand the resulting path selection

**Hub-and-spoke routing with UDRs**

UDRs are essential when you want traffic inspection or deterministic routing through a firewall/NVA.

Common patterns:
- Force spoke-to-on-premises traffic through an Azure Firewall in the hub.
- Disable route propagation on a subnet when you want to override learned routes.
- Use UDRs to send default traffic to a firewall while preserving specific routes to ExpressRoute or VPN.

Example route table for a spoke subnet:

```bash
az network route-table create \
  --resource-group rg-hub \
  --name rt-spoke-a

az network route-table route create \
  --resource-group rg-hub \
  --route-table-name rt-spoke-a \
  --name default-to-firewall \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.0.1.4
```

If you attach this route table to a spoke subnet, all internet-bound traffic goes to the firewall. You can then add more specific routes for on-premises prefixes if needed.

When to disable propagation:
- You want the subnet to ignore learned routes from the gateway.
- You need strict inspection through an NVA.
- You are preventing route conflicts in complex hybrid topologies.

**Failover design: ExpressRoute to VPN**

A resilient architecture often uses ExpressRoute as the primary path and VPN as the backup. The goal is to keep the same Azure and on-premises prefixes reachable through both paths, then let routing converge when the primary path fails.

Typical failover behavior:
- ExpressRoute is preferred while healthy.
- If the circuit or peering fails, BGP withdraws routes.
- VPN routes become active and traffic shifts to the tunnel.
- Convergence time depends on BGP timers, gateway health, and on-premises device behavior.

A practical validation approach:
1. Confirm both paths are up and advertising the same prefixes.
2. Verify effective routes on a VM in a spoke subnet.
3. Simulate ExpressRoute failure by disabling peering or disconnecting the circuit.
4. Watch route withdrawal and confirm VPN becomes the active path.
5. Test application connectivity, not just ping.

Useful Azure-side checks:

```bash
az network nic show-effective-route-table \
  --resource-group rg-app \
  --name nic-app01
```

This output helps you confirm which next hop Azure is using for on-premises prefixes.

On the on-premises side, inspect BGP neighbors and route tables on your edge device. The exact commands depend on the vendor, but you want to confirm:
- ExpressRoute routes are withdrawn
- VPN routes are learned
- The preferred next hop changes as expected

**VPN and ExpressRoute coexistence with BGP**

BGP is the key to clean coexistence. Without BGP, failover is harder to manage and route symmetry is less reliable.

Design guidance:
- Enable BGP on the VPN gateway when you want dynamic route exchange.
- Use the same on-premises prefixes on both paths where possible.
- Keep ASN (Autonomous System Number) planning consistent across devices.
- Avoid route summarization that hides important failover paths.

BGP also matters for active-active VPN gateways. Each instance can establish its own tunnel, which improves resilience and can help with throughput.

**Active-active VPN gateway considerations**

Active-active VPN gateways are useful when you want higher availability and better resilience.

Key points:
- Two gateway instances each have their own public IP.
- On-premises devices should build tunnels to both IPs.
- BGP can advertise routes over both tunnels.
- Failover is faster when both tunnels are already established.

Example topology:

```text
On-prem router
   | \ 
   |  \__ Tunnel 1 --> VPN GW instance 1
   |
   \_____ Tunnel 2 --> VPN GW instance 2
```

Use active-active when:
- You need better availability than a single gateway instance.
- Your on-premises device supports multiple tunnels.
- You want smoother failover during maintenance or outages.

**Throughput, scale, and sizing**

Gateway sizing is a common AZ-305 concern because architecture decisions must match workload demand.

Consider:
- **Gateway SKU** throughput limits
- Number of tunnels supported
- Number of routes that can be learned or advertised
- Aggregate bandwidth across spokes and on-premises sites
- Whether FastPath is available for the chosen design

General sizing guidance:
- Use ExpressRoute for higher, more predictable throughput.
- Use VPN for smaller sites or as a backup path.
- Do not assume a single gateway can support unlimited spokes or routes.
- Validate against current Azure limits before final design.

> **Important:** Gateway throughput, tunnel counts, and route scale limits vary by SKU and can change over time. Always verify the current service limits before committing to a production design.

**ExpressRoute FastPath**

FastPath is a performance feature that can improve data path efficiency for supported scenarios. It reduces gateway involvement for certain traffic flows and can lower latency.

Use FastPath when:
- You need better throughput for supported VNet-to-on-premises traffic.
- Your design uses supported gateway and circuit combinations.
- You want to reduce bottlenecks at the gateway layer.

Do not assume FastPath is universal. It depends on the gateway type, circuit configuration, and supported traffic patterns.

**ExpressRoute Global Reach**

Global Reach connects two on-premises sites through Microsoft’s backbone using ExpressRoute circuits. It is useful when you want private site-to-site connectivity between branches without hairpinning through Azure workloads.

Use cases:
- Branch-to-branch connectivity over the Microsoft backbone
- Consolidating WAN paths
- Reducing dependence on public internet transit between sites

It is not a replacement for all WAN design. It is a specific interconnect feature that complements, rather than replaces, standard ExpressRoute-to-Azure connectivity.

**Practical end-to-end scenario**

A retail company has:
- A primary datacenter connected to Azure through ExpressRoute
- A backup IPsec VPN from the same datacenter
- A hub VNet with Azure Firewall
- Multiple spoke VNets for apps and data

Design choices:
- ExpressRoute is the primary path for application traffic.
- VPN is configured as a backup with BGP.
- Spoke subnets use UDRs to send outbound traffic to Azure Firewall.
- The hub gateway advertises on-premises prefixes to spokes.
- Route propagation is controlled so the firewall remains the inspection point.

Why this works:
- ExpressRoute gives stable performance for production traffic.
- VPN provides continuity if the circuit fails.
- UDRs preserve security inspection.
- BGP keeps route convergence manageable.

**Troubleshooting checklist**

When connectivity fails, isolate the problem by layer.

- **Physical/provider layer**
  - Check circuit status with the provider.
  - Confirm the ExpressRoute circuit is provisioned and peering is up.
- **Routing layer**
  - Verify BGP neighbors are established.
  - Confirm prefixes are advertised and received.
  - Check for overlapping or missing routes.
- **Azure gateway layer**
  - Validate gateway SKU and configuration.
  - Confirm the gateway is attached to the correct VNet.
- **Subnet layer**
  - Review UDRs and route propagation settings.
  - Check whether a firewall/NVA is dropping traffic.
- **Application layer**
  - Test TCP/UDP ports, not just ICMP.
  - Confirm DNS resolution and service endpoints.

Useful commands:

```bash
az network vnet-gateway show \
  --resource-group rg-hub \
  --name vng-hub
```

```bash
az network nic show-effective-route-table \
  --resource-group rg-app \
  --name nic-app01
```

These commands help you confirm gateway state and the routes a VM actually sees.

**Configuration examples**

A minimal ExpressRoute gateway connection workflow often includes authorization and connection creation. The exact command set can vary by Azure CLI version, so treat the following as a pattern and verify against current CLI help.

```bash
az network express-route authorization create \
  --resource-group rg-er \
  --circuit-name er-circuit \
  --name auth-hub

az network express-route peering create \
  --resource-group rg-er \
  --circuit-name er-circuit \
  --peering-type AzurePrivatePeering \
  --peer-asn 65010 \
  --vlan-id 200 \
  --primary-peer-subnet 192.168.10.0/30 \
  --secondary-peer-subnet 192.168.10.4/30
```

These steps represent:
- Creating authorization for the VNet gateway to use the circuit
- Configuring private peering with BGP details
- Establishing the Azure side of the route exchange

**Exam-focused takeaways**

- ExpressRoute is the preferred primary path for enterprise hybrid connectivity.
- VPN is the most common backup path and is often paired with ExpressRoute.
- Coexistence depends on supported gateway SKUs and topology, not just intent.
- BGP is central to route exchange, failover, and path selection.
- UDRs and route propagation control are essential in hub-and-spoke designs.
- FastPath, Global Reach, and peering types are important differentiators.
- Validate failover with real route changes and application tests, not only tunnel status.

**Quick mental model**

Think of the design in three layers:
- **Transport**: internet VPN or private ExpressRoute circuit
- **Routing**: BGP, UDRs, and route propagation
- **Security**: firewall/NVA inspection and encryption requirements

If you get these three layers right, the hybrid design is usually correct for AZ-305.
