## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Branch connectivity overview**

Branch connectivity covers how on-premises sites, remote users, and partner networks reach Azure workloads securely and predictably. In AZ-305, you are expected to choose the right connectivity pattern based on latency, bandwidth, resiliency, operational complexity, and routing requirements.

The main design choices are:

- **Site-to-site VPN** for encrypted connectivity over the public internet.
- **Point-to-site VPN** for individual users or small groups connecting from client devices.
- **ExpressRoute** for private, provider-backed connectivity with predictable performance.
- **Virtual WAN** for large-scale branch and hub connectivity with centralized routing and security options.
- **Routing and DNS** design to make hybrid name resolution and traffic flow work correctly.

A good design starts with the business requirement, not the technology. For example, a small branch may only need a VPN tunnel, while a regulated enterprise may require ExpressRoute with redundant circuits and controlled route propagation.

**Connectivity patterns and when to use them**

Use the simplest option that satisfies the requirement, then add resiliency and scale only when needed.

- **Site-to-site VPN**
  - Best for lower-cost branch connectivity.
  - Uses IPsec/IKE tunnels over the internet.
  - Suitable when bandwidth and latency are acceptable and you can tolerate internet variability.
- **Point-to-site VPN**
  - Best for remote workers, contractors, and admin access.
  - Connects a client device directly to Azure.
  - Useful when users do not have a branch router or when you need per-user access control.
- **ExpressRoute**
  - Best for private connectivity with higher reliability and more consistent latency.
  - Uses a connectivity provider or direct cross-connect model.
  - Common for enterprise branch aggregation, data center interconnect, and regulated workloads.
- **Virtual WAN**
  - Best for many branches, distributed sites, or centralized hub-and-spoke operations at scale.
  - Provides managed hubs, routing, and optional security insertion.
  - Useful when you want to reduce custom hub management across regions.

**Site-to-site VPN fundamentals**

A site-to-site VPN connects an on-premises VPN device to an Azure VPN gateway. You typically deploy the gateway into a dedicated `GatewaySubnet` in the virtual network. The gateway terminates the tunnel and exchanges routes with the on-premises device.

Key concepts:

- **IPsec/IKE**
  - The tunnel is encrypted using Internet Protocol Security and Internet Key Exchange.
  - Match parameters on both sides, including IKE version, encryption, integrity, and Diffie-Hellman group.
- **GatewaySubnet**
  - Required for Azure VPN gateways and ExpressRoute virtual network gateways.
  - Keep it dedicated; do not place workloads in it.
- **Gateway SKU**
  - Determines throughput, features, and scale.
  - Higher SKUs generally support more tunnels and better performance.
- **Active-active**
  - Uses two gateway instances for higher availability.
  - Preferred when you need resiliency against instance failure.
- **BGP**
  - Border Gateway Protocol can automate route exchange.
  - Useful when you have multiple prefixes or want dynamic failover.

A typical branch VPN design includes a hub VNet with the gateway, spoke VNets for workloads, and user-defined routes where needed to control traffic flow.

**Point-to-site VPN for remote users**

Point-to-site VPN is the right choice when individual users need secure access from laptops or mobile devices. It is not a branch-to-Azure replacement; it complements site-to-site connectivity.

Common use cases:

- Admin access to Azure resources from unmanaged networks.
- Temporary access for contractors or support engineers.
- Remote work scenarios where a full branch tunnel is unnecessary.

Important design points:

- **Authentication**
  - Use Microsoft Entra ID, certificate-based auth, or RADIUS depending on the client and security model.
- **Client address pool**
  - Allocate a non-overlapping IP range for VPN clients.
  - Ensure it does not overlap with on-premises or Azure address spaces.
- **Split tunneling**
  - Send only Azure or corporate traffic through the VPN when appropriate.
  - Reduces bandwidth use and improves user experience.
- **DNS**
  - Push internal DNS settings so users can resolve private names.

Example Azure CLI pattern for a point-to-site configuration:

```bash
az network vnet-gateway update \
  --resource-group rg-network \
  --name vnet-gw-hub \
  --client-protocol IkeV2 \
  --vpn-client-address-pool 172.16.201.0/24
```

> Verify the exact CLI parameters for your gateway type and authentication model, because point-to-site settings vary by SKU and client configuration.

**ExpressRoute design**

ExpressRoute provides private connectivity between your network and Microsoft’s network. Traffic does not traverse the public internet, which makes it attractive for predictable performance and enterprise governance.

Core ExpressRoute concepts:

- **Circuit**
  - The service you purchase from a connectivity provider or via ExpressRoute Direct.
  - Represents the private connection into Microsoft’s edge.
- **Peering**
  - Microsoft peering and private peering are legacy concepts in many designs; modern Azure guidance focuses on private connectivity to VNets through the circuit and gateway.
- **Virtual network gateway**
  - The Azure endpoint that connects VNets to the ExpressRoute circuit.
  - Uses `GatewaySubnet`, just like VPN gateways.
- **Connection resource**
  - The circuit is associated with the gateway through a connection object, not by the gateway alone.
- **Provider-managed vs direct**
  - Provider-managed ExpressRoute uses a carrier or exchange provider.
  - ExpressRoute Direct gives you dedicated physical ports into Microsoft’s edge for higher control and capacity.

Practical design guidance:

- Use ExpressRoute when you need consistent latency, higher bandwidth, or private connectivity requirements.
- Use VPN as a backup path in many designs.
- Use redundant circuits and diverse peering locations when availability is critical.
- Plan IP addressing carefully to avoid overlap with on-premises networks.

Example ExpressRoute gateway creation pattern:

```bash
az network vnet-gateway create \
  --resource-group rg-network \
  --name er-gw-hub \
  --location eastus \
  --vnet hub-vnet \
  --gateway-type ExpressRoute \
  --sku ErGw2AZ \
  --public-ip-addresses er-gw-pip-1 er-gw-pip-2
```

After the gateway exists, you associate it with the ExpressRoute circuit using the appropriate connection resource or circuit authorization workflow.

Example circuit authorization retrieval:

```bash
az network express-route auth create \
  --resource-group rg-network \
  --circuit-name er-circuit-prod \
  --name auth-hub
```

**ExpressRoute FastPath and Global Reach**

These features are often tested conceptually because they affect architecture decisions.

- **FastPath**
  - Bypasses some gateway processing for data-plane traffic.
  - Can improve throughput and reduce latency.
  - Not all gateway or topology combinations support it.
  - Use it when supported and when the design benefits from lower gateway overhead.
- **Global Reach**
  - Connects two on-premises networks through Microsoft’s backbone using ExpressRoute.
  - Useful for branch-to-branch or data center-to-data center connectivity when both sides already use ExpressRoute.
  - It is not a general-purpose transit feature for arbitrary networks.

Design implication:

- Do not assume ExpressRoute automatically creates full mesh connectivity between all connected sites.
- Validate whether the traffic path is between on-premises networks, VNets, or both, and whether the required feature is supported in that topology.

**Gateway SKUs, redundancy, and limits**

Gateway selection affects scale, availability, and feature support.

Common considerations:

- **SKU family**
  - VPN and ExpressRoute gateways use different SKU families.
  - Higher SKUs generally support more throughput and more connections.
- **Zone-redundant gateways**
  - Use zone-redundant options in supported regions for better resiliency.
  - Prefer them for production hub gateways when regional zones are available.
- **Active-active gateways**
  - Improve availability and can help with maintenance events.
  - Often used in enterprise hub designs.
- **Throughput and tunnel count**
  - Gateway SKUs have practical limits on aggregate throughput and number of tunnels.
  - Do not design a large branch aggregation solution on a small SKU.
- **GatewaySubnet sizing**
  - Size the subnet generously enough for the gateway type and future growth.
  - Avoid using a minimal subnet if you expect to add redundancy or change SKUs later.

A common exam trap is assuming the smallest gateway SKU is fine because the first branch is small. In real designs, growth, failover, and route scale often justify a larger SKU.

**Virtual WAN architecture**

Virtual WAN is a managed networking service for large-scale branch connectivity, hub routing, and optional security insertion. It reduces the need to build and operate many custom hub gateways.

Main components:

- **Virtual WAN**
  - The top-level resource that contains hubs and connectivity policy.
- **Virtual hub**
  - A Microsoft-managed hub in a region.
  - Can host VPN, ExpressRoute, and routing services.
- **VPN gateway**
  - Connects branches or remote sites to the virtual hub.
- **ExpressRoute gateway**
  - Connects ExpressRoute circuits to the virtual hub.
- **Secured hub**
  - A hub with integrated security services or routing to a firewall appliance.
- **Connections**
  - Branches, VNets, and circuits attach to the hub through managed connections.

Virtual WAN is useful when:

- You have many branches and want centralized management.
- You need multi-region hub connectivity without building each hub manually.
- You want managed routing between branches, VNets, and security services.

Be careful with the term “transitive connectivity.” Virtual WAN supports routing within its architecture, but traffic flow still depends on hub association, routing intent, and security policy. It does not magically make every connected network transit to every other network in all cases.

Example Virtual WAN creation pattern:

```bash
az network vwan create \
  --resource-group rg-network \
  --name vwan-global \
  --location eastus \
  --type Standard
```

Example virtual hub creation:

```bash
az network vhub create \
  --resource-group rg-network \
  --name hub-eastus \
  --vwan vwan-global \
  --address-prefix 10.100.0.0/23 \
  --location eastus
```

**Branch-to-branch routing in Virtual WAN**

Branch-to-branch routing is a common reason to choose Virtual WAN. It can simplify connectivity between many sites, but you still need to understand the routing model.

Key points:

- Branches connected to the same hub can often reach each other through the hub routing fabric.
- Routing behavior depends on the connection type and hub configuration.
- Security policies or route tables can restrict east-west traffic.
- If you need strict segmentation, use separate route tables or secured hubs.

Use Virtual WAN when you want managed branch interconnect, but do not rely on it as a substitute for explicit segmentation design.

**Routing design: UDRs, propagation, and system routes**

Routing is one of the most important exam areas because connectivity often fails due to route selection, not tunnel setup.

Important routing concepts:

- **User-defined routes (UDRs)**
  - Custom routes you assign to subnets.
  - Used to force traffic through an NVA, firewall, or specific next hop.
- **Route propagation**
  - Gateway routes can propagate into VNets and subnets.
  - You can disable propagation in some designs to prevent unwanted path selection.
- **System routes**
  - Azure automatically creates routes for local VNet, peered VNets, and default internet paths.
  - These routes interact with UDRs and gateway-learned routes.
- **Longest prefix match**
  - Azure chooses the most specific route.
  - A more specific UDR can override a broader propagated route.

Practical design guidance:

- Use UDRs when you need deterministic inspection or forced tunneling.
- Disable propagation only when you understand the impact on return paths and branch reachability.
- Check for overlapping prefixes between on-premises, Azure, and VPN client pools.
- Remember that route asymmetry can break stateful firewalls and cause intermittent failures.

Example route table association:

```bash
az network route-table create \
  --resource-group rg-network \
  --name rt-spoke-app

az network route-table route create \
  --resource-group rg-network \
  --route-table-name rt-spoke-app \
  --name default-to-fw \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.10.1.4
```

**Routing and BGP**

BGP is the standard way to exchange routes dynamically between Azure gateways and on-premises devices.

Why use BGP:

- Reduces manual route maintenance.
- Supports failover when a path becomes unavailable.
- Helps with multi-site and multi-prefix environments.

Design notes:

- Use BGP when your on-premises edge device supports it.
- Advertise only the prefixes you intend to expose.
- Avoid route overlap between branch sites unless you intentionally summarize or translate.
- Validate ASN (Autonomous System Number) planning across your network.

Example VPN gateway BGP configuration pattern:

```bash
az network vnet-gateway update \
  --resource-group rg-network \
  --name vnet-gw-hub \
  --asn 65010 \
  --enable-bgp true
```

Example local network gateway with BGP settings:

```bash
az network local-gateway create \
  --resource-group rg-network \
  --name lng-branch1 \
  --gateway-ip-address 203.0.113.10 \
  --local-address-prefixes 10.20.0.0/16 \
  --asn 65020 \
  --bgp-peering-address 10.20.255.254
```

**DNS for hybrid and branch connectivity**

DNS is often the hidden dependency in branch connectivity. Network paths may be correct, but applications still fail if names do not resolve.

Key DNS patterns:

- **Conditional forwarding**
  - Forward specific zones to Azure or on-premises DNS servers.
  - Useful for split-brain or hybrid name resolution.
- **Private endpoint resolution**
  - Private endpoints use private IPs and require matching DNS records.
  - Use private DNS zones or custom DNS forwarding to resolve service names correctly.
- **Azure DNS Private Resolver**
  - Provides inbound and outbound endpoints for hybrid DNS forwarding.
  - Helps integrate on-premises DNS with Azure private name resolution without deploying custom DNS VMs.
- **On-premises integration**
  - Ensure branch DNS servers can resolve Azure private zones.
  - Ensure Azure workloads can resolve on-premises internal zones when needed.

Practical example: on-premises DNS server forwarding Azure private zones to Azure DNS Private Resolver outbound rules.

```text
Zone: privatelink.database.windows.net
Forward to: 10.100.2.4
```

Example Azure DNS Private Resolver deployment pattern:

```bash
az network dns-resolver create \
  --resource-group rg-network \
  --name dnsr-hub \
  --location eastus \
  --virtual-network hub-vnet
```

Example inbound endpoint:

```bash
az network dns-resolver inbound-endpoint create \
  --resource-group rg-network \
  --dns-resolver-name dnsr-hub \
  --name inbound1 \
  --subnet /subscriptions/<sub>/resourceGroups/rg-network/providers/Microsoft.Network/virtualNetworks/hub-vnet/subnets/dns-inbound
```

**Private endpoint DNS resolution**

Private endpoints are common in branch-connected architectures because they keep PaaS traffic off the public internet.

What you need to remember:

- Private endpoints map a service to a private IP in your VNet.
- DNS must resolve the service name to the private IP.
- If DNS is wrong, the service may still be reachable publicly, which defeats the design intent.
- Use private DNS zones and forwarding rules to keep resolution consistent across branches and Azure.

Common pattern:

- Create the private endpoint.
- Link the private DNS zone to the VNet.
- Forward the zone from on-premises DNS to Azure DNS Private Resolver or a DNS server that knows the zone.

**Security and segmentation**

Branch connectivity is not only about reachability. It also defines trust boundaries.

Design considerations:

- **Segmentation**
  - Separate branches, users, and workloads by route tables, VNets, or hubs.
- **Inspection**
  - Force traffic through Azure Firewall or an NVA when policy requires inspection.
- **Least privilege**
  - Advertise only required prefixes.
  - Restrict branch access to only the subnets and services needed.
- **Shared services**
  - Place DNS, firewall, and management services in a controlled hub.

A common enterprise pattern is hub-and-spoke with a firewall in the hub and spokes for applications. Branches connect to the hub, and UDRs steer traffic through inspection before reaching spokes or the internet.

**Monitoring, validation, and troubleshooting**

Connectivity design is incomplete without validation. In Azure, you should verify both control plane and data plane behavior.

Useful tools and signals:

- **Connection Monitor**
  - Tests reachability and latency between endpoints.
  - Useful for branch-to-Azure and branch-to-branch validation.
- **Network Watcher**
  - Provides packet capture, next hop, effective routes, and connection troubleshooting.
- **Gateway diagnostics**
  - VPN and ExpressRoute gateways emit logs and metrics that help identify tunnel or route issues.
- **Azure Monitor**
  - Use metrics and logs to watch tunnel status, throughput, and failures.
- **On-premises device logs**
  - Check IKE negotiation, BGP session state, and interface health on the branch edge device.

Example connection monitor creation pattern:

```bash
az network watcher connection-monitor create \
  --resource-group rg-network \
  --name cm-branch-to-app \
  --location eastus
```

Example effective route inspection:

```bash
az network nic show-effective-route-table \
  --resource-group rg-app \
  --name appvm-nic
```

Troubleshooting checklist:

1. Confirm IP ranges do not overlap.
2. Verify the tunnel or circuit is up.
3. Check BGP neighbor state and advertised prefixes.
4. Inspect effective routes on the VM or subnet.
5. Validate DNS resolution for the target name.
6. Test from both directions to detect asymmetric routing.
7. Review firewall or NVA logs for dropped traffic.

> If the tunnel is up but traffic fails, the most common causes are route mismatch, DNS failure, or security policy blocking the path.

**Design comparisons for exam scenarios**

| Requirement | Best fit | Why |
|---|---|---|
| Small branch, low cost | Site-to-site VPN | Simple and inexpensive |
| Remote employee access | Point-to-site VPN | Per-user access without branch hardware |
| Predictable private enterprise connectivity | ExpressRoute | Private, provider-backed path |
| Many branches and centralized routing | Virtual WAN | Managed scale and hub routing |
| Dynamic route exchange | BGP | Reduces manual route management |
| Private PaaS access from branches | Private endpoints + DNS integration | Keeps service traffic private |

**Common exam pitfalls**

- Assuming a VPN gateway alone connects an ExpressRoute circuit.
- Forgetting that both VPN and ExpressRoute gateways use `GatewaySubnet`.
- Treating Virtual WAN as automatic full-mesh transit without considering routing intent and policy.
- Using UDRs without checking return path symmetry.
- Ignoring DNS when private endpoints are involved.
- Choosing ExpressRoute when the requirement only needs a simple remote-user VPN.
- Assuming Route Server replaces all custom routing or firewall design.
- Overlooking gateway SKU and zone-redundancy requirements for production resiliency.

**Practical design examples**

A small retail branch may use:

- Site-to-site VPN to a hub VNet.
- BGP for route exchange.
- Azure Firewall in the hub.
- Private DNS forwarding for internal apps.

A global enterprise may use:

- ExpressRoute for primary branch and data center connectivity.
- VPN as backup.
- Zone-redundant gateway SKUs.
- Azure DNS Private Resolver for hybrid name resolution.
- Connection Monitor for continuous validation.

A distributed organization with many sites may use:

- Virtual WAN with multiple hubs.
- Branch connections into regional hubs.
- Secured hub routing for inspection.
- Separate route tables for segmentation.

**Key takeaways**

Branch connectivity design in Azure is a balance of performance, cost, security, and operational simplicity. The right answer depends on whether you need remote user access, branch-to-Azure connectivity, private enterprise networking, or large-scale managed routing.

Focus on these exam priorities:

- Choose the right connectivity model for the business need.
- Understand gateway placement, SKU selection, and redundancy.
- Design routing intentionally with UDRs, BGP, and propagation in mind.
- Treat DNS as part of connectivity, especially for private endpoints.
- Validate with monitoring and troubleshooting tools, not assumptions.
