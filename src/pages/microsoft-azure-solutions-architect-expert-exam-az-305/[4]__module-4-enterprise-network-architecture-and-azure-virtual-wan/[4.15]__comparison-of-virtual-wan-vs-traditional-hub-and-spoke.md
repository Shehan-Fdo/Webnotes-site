## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Overview**

Azure networking commonly uses either a **traditional hub-and-spoke** design or **Azure Virtual WAN**. Both patterns centralize connectivity, security, and routing, but they solve different problems.

Traditional hub-and-spoke gives you direct control over virtual network peering, route tables, and network virtual appliances (NVAs). Virtual WAN is a managed networking service that abstracts much of that complexity and is designed for large-scale branch, remote user, and multi-region connectivity.

The right choice depends on scale, operational model, routing complexity, and how much control you need over the network path.

**Core design goals**

- **Traditional hub-and-spoke** is best when you want explicit control over routing and security.
- **Virtual WAN** is best when you want a managed global transit architecture with simpler branch and multi-site connectivity.
- Both can support shared services, inspection, and connectivity to on-premises networks.
- Neither pattern is automatically “better” in all cases; the tradeoff is control versus managed simplicity.

**Traditional hub-and-spoke architecture**

In a traditional hub-and-spoke model, you deploy one or more hub virtual networks and connect spoke VNets to them using VNet peering. The hub usually hosts shared services such as Azure Firewall, VPN gateways, Bastion, DNS forwarders, or other NVAs.

This model is highly flexible, but you must design routing carefully. VNet peering is not transitive by default, so traffic does not automatically flow from one spoke to another or from one hub to another without explicit design.

- The **hub** is the central routing and security point.
- **Spokes** host application workloads and usually do not expose shared infrastructure.
- **User-defined routes (UDRs)** often steer traffic to an NVA or firewall for inspection.
- **Gateway transit** can allow spokes to use a hub gateway for on-premises connectivity when configured correctly.

A common topology looks like this:

```text
On-premises
   |
VPN/ExpressRoute
   |
Hub VNet
 |   |   \
Spoke Spoke Spoke
```

**Why architects choose traditional hub-and-spoke**

Traditional hub-and-spoke is often chosen when you need precise control over packet flow, custom routing, or specialized appliances.

- You can place an Azure Firewall or third-party NVA in the hub and force traffic through it.
- You can design separate hubs for different environments, such as production and nonproduction.
- You can tune routing for asymmetric or inspection-heavy traffic patterns.
- You can keep the architecture simple for small or medium environments.

Cost is not automatically lower, but it can be more economical for smaller deployments because you avoid the additional managed service components that Virtual WAN often requires. However, once you add multiple regions, many branches, or complex connectivity, operational overhead can rise quickly.

**Routing in traditional hub-and-spoke**

Routing is the main design challenge. You typically combine peering, gateway transit, and UDRs to control traffic paths.

Common patterns include:

- **Spoke-to-internet inspection**
  - Route `0.0.0.0/0` to Azure Firewall or an NVA in the hub.
- **Spoke-to-on-premises**
  - Use gateway transit through the hub VPN or ExpressRoute gateway.
- **Spoke-to-spoke**
  - Use peering plus routing through the hub if you need inspection or segmentation.

Example UDR for a spoke subnet that sends traffic to a firewall:

```bash
az network route-table route create \
  --resource-group rg-network \
  --route-table-name rt-spoke-app \
  --name default-to-firewall \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.0.0.4
```

A spoke subnet route to a **virtual appliance** is the common pattern when you want forced tunneling or centralized inspection. Using `VirtualNetworkGateway` as the next hop is not the usual spoke subnet pattern for inspection traffic.

**Limitations of traditional hub-and-spoke**

- **Transitive routing** is not automatic.
- **Inter-hub connectivity** requires explicit design and is easy to get wrong.
- **Operational overhead** increases as the number of spokes, regions, and policies grows.
- **Manual configuration** can become repetitive across many VNets.
- **Scaling branch connectivity** across many sites is more complex than in Virtual WAN.

**Azure Virtual WAN architecture**

Azure Virtual WAN is a managed networking service built around a Microsoft-managed transit architecture. It provides a central **Virtual WAN** resource, one or more **virtual hubs**, and connectivity options for branches, users, and VNets.

Virtual WAN is designed to simplify:

- Branch connectivity
- Site-to-site VPN
- Point-to-site VPN
- ExpressRoute integration
- Inter-region connectivity
- Centralized routing and security

A virtual hub is a Microsoft-managed virtual network that can host gateways and route traffic between connected resources.

```text
Branch sites / users / ExpressRoute
              |
         Virtual WAN
              |
        Virtual Hub(s)
          /    |     \
       VNet  VNet   Security
```

**Why architects choose Virtual WAN**

Virtual WAN is attractive when you need a managed, scalable transit backbone.

- It reduces the amount of custom routing you manage.
- It simplifies branch onboarding across many sites.
- It supports multi-region connectivity with a more standardized model.
- It can centralize routing intent and security integration.
- It is useful when you want a consistent global network fabric rather than hand-built peering meshes.

Virtual WAN is not always cheaper. It often introduces additional service and gateway costs, and for small environments a traditional hub-and-spoke design may be less expensive and easier to justify.

**Virtual WAN connectivity model**

Virtual WAN uses hubs and connections rather than the classic peering-centric model.

Typical components:

- **Virtual WAN**: the top-level container for the network.
- **Virtual hub**: the regional transit point.
- **VPN gateway**: enables site-to-site and point-to-site connectivity.
- **ExpressRoute gateway**: connects private circuits.
- **VNet connections**: attach VNets to the hub.
- **VPN site**: represents a branch location.
- **Site link**: defines the branch connection details.

Creating a VPN site alone does not connect a branch to Azure. You still need the virtual hub gateway and the site connection resources.

Example: create a VPN site definition

```bash
az network vpn-site create \
  --resource-group rg-network \
  --name branch-london \
  --location uksouth \
  --virtual-wan vwan-global \
  --address-prefixes 10.20.0.0/16 \
  --device-model "Contoso Edge 1000" \
  --device-vendor "Contoso"
```

This creates the site object, but branch connectivity still requires:

1. A virtual hub in the target region.
2. A VPN gateway in that hub.
3. One or more site links.
4. A connection between the site and the hub gateway.

**Routing in Virtual WAN**

Virtual WAN uses hub route tables and routing intent to simplify traffic steering. The service can propagate routes between connected VNets, branches, and gateways according to the hub configuration.

Important routing concepts:

- **Route propagation** controls which routes are learned by which connections.
- **Hub route tables** segment traffic and can isolate groups of spokes or branches.
- **Routing intent** can direct traffic to security services such as Azure Firewall in supported scenarios.
- **Branch-to-branch** and **branch-to-VNet** connectivity are handled through the managed hub fabric.

A conceptual routing intent example:

```text
Traffic type: Internet
Next hop: Azure Firewall

Traffic type: Private
Next hop: Azure Firewall
```

This is a conceptual model only. Actual configuration is done through Azure resources, portal settings, ARM templates, Bicep, or CLI commands that match the Virtual WAN resource schema.

**Traditional hub-and-spoke vs. Virtual WAN**

| Area | Traditional hub-and-spoke | Virtual WAN |
|---|---|---|
| Control | High control over routes and appliances | More managed, less granular |
| Setup effort | More manual | Faster for standardized deployments |
| Branch connectivity | Possible, but more design work | Built for branch scale |
| Multi-region transit | Requires careful design | Native global transit model |
| Routing complexity | You manage peering, UDRs, gateways | Centralized hub routing model |
| Security insertion | Flexible with NVAs and Azure Firewall | Supported, but within Virtual WAN patterns |
| Cost profile | Often lower for small/simple environments | Can be higher due to managed components |
| Operational overhead | Higher as scale grows | Lower for many sites and regions |

**When to use traditional hub-and-spoke**

Use traditional hub-and-spoke when:

- You need fine-grained control over routing and inspection.
- Your environment is small to medium sized.
- You already have a mature network team and established patterns.
- You rely on custom NVAs or specialized routing behavior.
- You want to minimize service complexity and keep the design explicit.

Example use cases:

- A single-region enterprise with a few application spokes.
- A regulated environment that requires strict traffic inspection paths.
- A landing zone with custom firewall policy and bespoke routing.

**When to use Virtual WAN**

Use Virtual WAN when:

- You have many branch sites or remote users.
- You need multi-region transit with less manual configuration.
- You want a managed backbone for VPN and ExpressRoute connectivity.
- You prefer standardized routing and centralized operations.
- You expect the network to grow and want to reduce future redesign work.

Example use cases:

- A global company onboarding dozens or hundreds of branches.
- A distributed organization with multiple Azure regions and shared connectivity requirements.
- A network team that wants to reduce the operational burden of maintaining many peering relationships.

**Security considerations**

Both models can support centralized security, but the implementation differs.

Traditional hub-and-spoke:

- Place Azure Firewall or an NVA in the hub.
- Use UDRs to force traffic through inspection points.
- Apply NSGs (Network Security Groups) at subnet level for workload segmentation.
- Use separate hubs if you need strong environment isolation.

Virtual WAN:

- Use hub-integrated security services where supported.
- Define routing intent or hub route tables to steer traffic.
- Centralize branch and VNet traffic through the virtual hub.
- Validate how security policies interact with propagated routes.

> Be careful with asymmetric routing. If one path bypasses the firewall and the return path does not, stateful inspection can fail.

**Operational differences**

Traditional hub-and-spoke usually requires more hands-on management:

- You maintain peering relationships.
- You manage route tables and propagation.
- You monitor gateway and firewall placement.
- You update each hub or spoke as the topology changes.

Virtual WAN reduces repetitive work:

- You connect resources to a managed hub.
- You use standardized connectivity patterns.
- You scale branch onboarding more consistently.
- You centralize many routing decisions in the hub fabric.

**Practical decision guidance**

Choose based on the problem you are solving:

- If you need **maximum control**, choose traditional hub-and-spoke.
- If you need **managed scale**, choose Virtual WAN.
- If you have **few networks and simple routing**, traditional hub-and-spoke is often sufficient.
- If you have **many branches, regions, or users**, Virtual WAN usually becomes more compelling.

A useful exam mindset is to ask:

1. How many sites, VNets, and regions are involved?
2. Do you need custom routing or standardized transit?
3. Is branch connectivity a primary requirement?
4. Do you want to manage peering and UDRs yourself?
5. Is the cost of managed simplicity justified by the scale?

**Common exam traps**

- Assuming Virtual WAN is always cheaper.
- Assuming VNet peering is transitive by default.
- Assuming a VPN site definition alone creates branch connectivity.
- Assuming a spoke subnet should route to a virtual network gateway for inspection traffic.
- Assuming inter-hub routing is automatic in a traditional design.

**Quick summary**

- **Traditional hub-and-spoke** gives you explicit control and is often best for smaller or highly customized environments.
- **Virtual WAN** gives you managed global transit and is best for larger, distributed, or branch-heavy environments.
- Cost, complexity, and scale determine the right choice more than any single feature.
- For AZ-305, focus on routing behavior, transitivity limits, branch onboarding, and operational tradeoffs.
