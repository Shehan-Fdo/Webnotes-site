## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**CI/CD integration overview**

Bicep is an <abbr title="Infrastructure as Code">IaC</abbr> language for Azure that compiles to Azure Resource Manager (<abbr title="Azure Resource Manager">ARM</abbr>) JSON at build time, but Azure deployment tooling can often deploy Bicep files directly. In CI/CD, the goal is not just to “run a deployment,” but to make infrastructure changes repeatable, reviewable, and safe across environments.

A good pipeline usually separates these concerns:

- **Build**: lint, format, and compile Bicep.
- **Validate**: check syntax, dependencies, and deployment feasibility.
- **Preview**: use `what-if` to estimate changes.
- **Deploy**: apply changes to a target scope with controlled approvals.
- **Promote**: reuse the same artifact across environments.

This separation matters because infrastructure changes can fail for reasons that application builds do not, such as permission gaps, name collisions, policy restrictions, or resource provider limitations.

**Core Bicep pipeline principles**

Bicep fits well into CI/CD because it is declarative. You describe the desired state, and the platform reconciles the target environment toward that state. That means your pipeline should focus on verifying intent and controlling rollout, not on imperative step-by-step provisioning logic.

Key principles:

- **Immutable artifacts**: build once, deploy many times.
- **Environment parity**: keep dev, test, and prod as similar as possible.
- **Parameterization**: vary only values that must differ, such as SKU, region, or tags.
- **Least privilege**: use scoped identities with only the permissions needed for each stage.
- **Traceability**: store templates, parameter files, and deployment logs as pipeline outputs.

Bicep also supports modules, including local modules and registry-based modules. In CI/CD, this means you can compose reusable infrastructure components and version them independently. A pipeline should pin module versions where possible to avoid unexpected drift from upstream changes.

**Typical pipeline stages**

A practical pipeline for Bicep often includes these stages:

1. **Source checkout**
   - Pull the repository containing `main.bicep`, modules, and parameter files.
2. **Static checks**
   - Run formatting and linting.
   - Catch style issues and common authoring mistakes early.
3. **Build**
   - Compile Bicep to ARM JSON when you need a deployable artifact or a reviewable output.
4. **Validation**
   - Use Azure deployment validation to check whether the template can be deployed.
5. **Preview**
   - Run `what-if` to inspect the expected delta.
6. **Approval**
   - Require manual approval for higher environments.
7. **Deployment**
   - Deploy to the target scope using the approved artifact.
8. **Post-deployment verification**
   - Confirm resource state, outputs, and health checks.

**Bicep authoring checks**

Before deployment, validate the Bicep source itself. Common commands include:

```bash
az bicep version
az bicep build --file main.bicep
az bicep lint --file main.bicep
az deployment group validate \
  --resource-group rg-app-dev \
  --template-file main.bicep \
  --parameters dev.bicepparam
```

Useful checks:

- `az bicep build` compiles Bicep to ARM JSON.
- `az bicep lint` identifies style and correctness issues.
- `az deployment group validate` checks whether a deployment is likely to succeed at the target scope.
- `az deployment group what-if` previews the expected changes.

> `what-if` is a best-effort preview, not a perfect diff. It can miss some provider-specific changes, show incomplete details, or behave differently depending on resource provider support and API behavior.

**Deployment scopes**

Bicep deployments can target different Azure scopes. Your pipeline must match the scope to the resource type and deployment intent.

- **Resource group scope**
  - Use for most application and platform resources.
  - Common command: `az deployment group create`.
- **Subscription scope**
  - Use for resource groups, policy assignments, role assignments, and subscription-level resources.
  - Common command: `az deployment sub create`.
- **Management group scope**
  - Use for governance at scale, such as policy definitions and assignments.
  - Common command: `az deployment mg create`.
- **Tenant scope**
  - Use for tenant-wide resources and some governance operations.
  - Common command: `az deployment tenant create`.

Example resource group deployment:

```bash
az deployment group create \
  --resource-group rg-app-prod \
  --template-file main.bicep \
  --parameters prod.bicepparam
```

**Artifact strategy**

A reliable CI/CD design usually builds once and deploys the same artifact through all environments. This reduces the risk that dev and prod are built from different source states.

Common artifact patterns:

- **Source artifact**
  - Store the Bicep source, modules, and parameter files.
- **Compiled artifact**
  - Store the generated ARM JSON from `az bicep build`.
- **Deployment package**
  - Store the exact files needed by the deploy stage, including parameter files if they are environment-specific.

If you publish a directory in a pipeline, remember that the artifact contents are what matter, not just the directory name. The download path in a later stage depends on how the artifact was published and extracted.

Example build step:

```bash
mkdir -p $(Build.ArtifactStagingDirectory)/infra
az bicep build \
  --file infra/main.bicep \
  --outdir $(Build.ArtifactStagingDirectory)/infra
cp infra/*.bicepparam $(Build.ArtifactStagingDirectory)/infra/
```

Example Azure DevOps publish step:

```yaml
- task: PublishPipelineArtifact@1
  inputs:
    targetPath: '$(Build.ArtifactStagingDirectory)/infra'
    artifact: 'infra-drop'
```

**Azure DevOps integration**

Azure DevOps works well for Bicep because it supports multi-stage YAML pipelines, approvals, variable groups, and service connections.

A typical flow:

- Checkout repository.
- Install or use the Azure CLI.
- Install the Bicep CLI if needed.
- Build and validate the template.
- Publish the compiled output and parameter files.
- Download the artifact in a deployment stage.
- Run `what-if`.
- Deploy after approval.

Example pipeline skeleton:

```yaml
trigger:
- main

stages:
- stage: Build
  jobs:
  - job: BuildBicep
    pool:
      vmImage: ubuntu-latest
    steps:
    - checkout: self
    - task: AzureCLI@2
      inputs:
        azureSubscription: 'sc-az-dev'
        scriptType: bash
        scriptLocation: inlineScript
        inlineScript: |
          az bicep version
          az bicep build --file infra/main.bicep --outdir $(Build.ArtifactStagingDirectory)/infra
          cp infra/*.bicepparam $(Build.ArtifactStagingDirectory)/infra/
    - publish: $(Build.ArtifactStagingDirectory)/infra
      artifact: infra-drop

- stage: Deploy_Dev
  dependsOn: Build
  jobs:
  - deployment: DeployDev
    environment: dev
    pool:
      vmImage: ubuntu-latest
    strategy:
      runOnce:
        deploy:
          steps:
          - checkout: self
          - download: current
            artifact: infra-drop
          - task: AzureCLI@2
            inputs:
              azureSubscription: 'sc-az-dev'
              scriptType: bash
              scriptLocation: inlineScript
              inlineScript: |
                az deployment group validate \
                  --resource-group rg-app-dev \
                  --template-file $(Pipeline.Workspace)/infra-drop/main.bicep \
                  --parameters $(Pipeline.Workspace)/infra-drop/dev.bicepparam
                az deployment group what-if \
                  --resource-group rg-app-dev \
                  --template-file $(Pipeline.Workspace)/infra-drop/main.bicep \
                  --parameters $(Pipeline.Workspace)/infra-drop/dev.bicepparam
                az deployment group create \
                  --resource-group rg-app-dev \
                  --template-file $(Pipeline.Workspace)/infra-drop/main.bicep \
                  --parameters $(Pipeline.Workspace)/infra-drop/dev.bicepparam
```

Important Azure DevOps considerations:

- Use a service connection with the correct scope and permissions.
- Keep parameter files with the artifact if the deploy stage needs them.
- Use environment approvals for production.
- Prefer deployment jobs for environment tracking and approvals.
- Store secrets in variable groups or Azure Key Vault, not in parameter files.

**GitHub Actions integration**

GitHub Actions can also run Bicep pipelines effectively, especially when paired with OpenID Connect (<abbr title="OpenID Connect">OIDC</abbr>) for federated authentication.

A working workflow usually needs:

- Azure CLI available on the runner, or installed explicitly.
- Bicep CLI available, or installed through `az bicep install`.
- Source checkout in each job that needs repository files.
- Artifact upload and download if build and deploy happen in separate jobs.

Example workflow:

```yaml
name: deploy-bicep

on:
  push:
    branches: [ main ]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install Azure CLI
        run: |
          curl -sL https://aka.ms/InstallAzureCLIDeb | sudo bash

      - name: Install Bicep
        run: az bicep install

      - name: Build Bicep
        run: |
          mkdir -p out
          az bicep build --file infra/main.bicep --outdir out
          cp infra/*.bicepparam out/

      - uses: actions/upload-artifact@v4
        with:
          name: infra-drop
          path: out/

  deploy-dev:
    needs: build
    runs-on: ubuntu-latest
    permissions:
      id-token: write
      contents: read
    steps:
      - uses: actions/checkout@v4

      - uses: actions/download-artifact@v4
        with:
          name: infra-drop
          path: out

      - name: Azure login
        uses: azure/login@v2
        with:
          client-id: ${{ secrets.AZURE_CLIENT_ID }}
          tenant-id: ${{ secrets.AZURE_TENANT_ID }}
          subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}

      - name: Validate and deploy
        run: |
          az deployment group validate \
            --resource-group rg-app-dev \
            --template-file out/main.bicep \
            --parameters out/dev.bicepparam
          az deployment group what-if \
            --resource-group rg-app-dev \
            --template-file out/main.bicep \
            --parameters out/dev.bicepparam
          az deployment group create \
            --resource-group rg-app-dev \
            --template-file out/main.bicep \
            --parameters out/dev.bicepparam
```

Key GitHub Actions points:

- Install Azure CLI explicitly if you need a predictable environment.
- Use `azure/login` for federated authentication instead of storing long-lived secrets when possible.
- Download both the compiled template and any parameter files needed by the deploy job.
- Keep build and deploy jobs separate so you can promote the same artifact.

**Parameterization and environment promotion**

Parameter files let you keep the template stable while varying environment-specific values. This is especially useful for region, SKU, naming, tags, and feature flags.

Example parameter file:

```bicep
using './main.bicep'

param location = 'eastus'
param environmentName = 'dev'
param tags = {
  workload: 'orders'
  owner: 'platform'
}
```

Promotion strategy:

- Use one template for all environments.
- Use separate parameter files for dev, test, and prod.
- Keep secrets out of parameter files when possible.
- Promote the same compiled artifact through the pipeline.

This approach reduces drift and makes reviews easier because the infrastructure logic stays constant while only the inputs change.

**Validation, preview, and deployment commands**

These commands are commonly used in CI/CD:

```bash
az bicep build --file main.bicep
az deployment group validate \
  --resource-group rg-app-dev \
  --template-file main.bicep \
  --parameters dev.bicepparam
az deployment group what-if \
  --resource-group rg-app-dev \
  --template-file main.bicep \
  --parameters dev.bicepparam
az deployment group create \
  --resource-group rg-app-dev \
  --template-file main.bicep \
  --parameters dev.bicepparam
```

When to use each:

- **Build**: when you want a compiled artifact or to catch syntax issues early.
- **Validate**: when you want a preflight check before preview or deployment.
- **What-if**: when you want a change preview for review or approval.
- **Create**: when you are ready to apply changes.

**Modules and registries in pipelines**

Bicep modules improve reuse and consistency. In pipelines, modules can come from:

- Local files in the repository.
- Public or private Azure Container Registry (<abbr title="Azure Container Registry">ACR</abbr>) module registries.
- Template specs in some legacy or specialized workflows.

Practical guidance:

- Pin module versions to avoid surprise changes.
- Restore or build modules during the pipeline if the environment requires it.
- Ensure the pipeline identity can read private registries.
- Treat module updates like code changes and review them carefully.

Example module reference:

```bicep
module storage './modules/storage-account.bicep' = {
  name: 'storageDeploy'
  params: {
    location: location
    name: storageName
  }
}
```

**Security and secrets**

Infrastructure pipelines often need credentials, connection strings, or keys, but you should minimize secret exposure.

Best practices:

- Use managed identities or federated identity where possible.
- Store secrets in Azure Key Vault and reference them at runtime.
- Avoid hardcoding secrets in Bicep or parameter files.
- Restrict pipeline permissions to the target scope only.
- Separate build permissions from deploy permissions.

If a deployment needs a secret value, prefer passing it from a secure variable group or secret store rather than committing it to source control.

**Common pitfalls**

- Missing parameter files in the deploy job.
- Assuming the runner already has Azure CLI or Bicep installed.
- Deploying from a different artifact than the one validated.
- Mixing source files and compiled output without clear versioning.
- Using overly broad service connection permissions.
- Treating `what-if` as an exact guarantee instead of a preview.
- Forgetting that some changes are provider-specific and may not appear clearly in previews.

**Exam-focused takeaways**

For AZ-305, focus on the design choices behind the pipeline rather than memorizing one tool syntax.

You should be able to explain:

- Why Bicep fits CI/CD better than manual portal changes.
- How to separate build, validation, preview, and deployment stages.
- Why artifact immutability improves reliability.
- When to use resource group, subscription, management group, or tenant scope.
- How parameter files support environment promotion.
- Why `what-if` is useful but not authoritative.
- How to secure pipeline identities and secrets.
