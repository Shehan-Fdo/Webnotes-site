## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Compute selection mindset**

Azure compute choices are usually driven by how much control you need, how much platform management you want to offload, and what kind of networking and deployment model the workload requires. In AZ-305 scenarios, the right answer is rarely “the newest service”; it is the service that best matches operational responsibility, isolation, scaling, and application architecture.

A useful mental model is:

- **VMs** give you the most control and the most responsibility.
- **App Service** gives you managed web hosting with minimal infrastructure work.
- **Containers** give you packaging consistency and deployment flexibility, with varying levels of orchestration depending on the service.

When you compare options, think about:

- OS control and patching
- Application runtime support
- Network isolation and private access
- Scaling model
- Stateful versus stateless design
- Operational overhead
- Migration effort from existing systems

**Virtual machines: when infrastructure control matters**

Azure Virtual Machines (VMs) are Infrastructure as a Service (IaaS). You manage the guest operating system, installed software, patching strategy, and most of the application stack. Azure manages the physical host, storage platform, and underlying datacenter infrastructure.

Use VMs when you need:

- Full OS-level control
- Custom software or agents that a PaaS service cannot support
- Legacy applications with strict installation requirements
- Specialized networking or security tooling
- Lift-and-shift migration with minimal code changes
- Non-web workloads such as domain controllers, file servers, or custom middleware

VMs are often the safest choice for compatibility, but they increase operational burden. You must plan for:

- **Patching** of the guest OS and installed software
- **Backup** and restore of disks and application data
- **Monitoring** of OS health, logs, and performance counters
- **High availability** using availability zones, availability sets, or clustered designs
- **Disaster recovery** using replication, backup, or site recovery patterns

A VM is not inherently “stateful” or “stateless”; it can host either. The important point is that local disk state is not a durable design target unless you intentionally architect for it. For durable state, use managed disks, shared storage, or external data services.

**VM operational responsibility model**

| Area | Azure manages | You manage |
|---|---|---|
| Physical hardware | Yes | No |
| Hypervisor | Yes | No |
| Guest OS | No | Yes |
| Patching | No | Yes |
| Application runtime | No | Yes |
| Data protection | Platform options exist | Design and configure |
| Scaling | Infrastructure available | You design and automate |

This responsibility split is often the deciding factor in architecture questions. If the workload needs a custom agent, kernel-level access, or a legacy installer, a VM is usually the correct answer even if it is not the simplest operationally.

**VM examples and CLI patterns**

Create a Linux VM for a custom API host:

```bash
az vm create \
  --resource-group rg-app \
  --name vm-api-01 \
  --image Ubuntu2204 \
  --admin-username azureadmin \
  --generate-ssh-keys
```

Create a Windows VM for a legacy .NET Framework application:

```bash
az vm create \
  --resource-group rg-legacy \
  --name vm-legacy-01 \
  --image Win2022Datacenter \
  --admin-username azureadmin \
  --admin-password 'Use-A-Strong-Password-Here!'
```

> Use `--generate-ssh-keys` for Linux SSH access, not for Windows password-based administration.

**App Service: managed web hosting for application code**

Azure App Service is a Platform as a Service (PaaS) offering for web apps, REST APIs, and mobile back ends. It is designed to host application code without requiring you to manage the underlying operating system or web server.

Choose App Service when you want:

- Fast deployment of web applications and APIs
- Managed patching of the platform
- Built-in deployment slots for supported plans and scenarios
- Easy integration with identity, certificates, and application settings
- Autoscale based on metrics or schedules
- A simpler operational model than VMs

App Service is best for application instances that are designed to be **stateless at the instance level**. The application can still use external state stores such as Azure SQL Database, Azure Cosmos DB, Azure Cache for Redis, or Blob Storage. The key idea is that local instance state should not be required for correctness.

Common App Service use cases:

- Public web apps
- Internal line-of-business web apps
- APIs that do not need OS-level control
- Modernized web apps that can run on supported runtimes
- Web front ends that rely on external databases and storage

App Service supports multiple runtime stacks, including .NET, Java, Node.js, Python, PHP, and custom containers. For legacy .NET Framework web apps, App Service can be a strong fit if the app is compatible with the Windows-based hosting model and does not require unsupported machine-level dependencies.

**App Service plan and isolation considerations**

App Service capabilities depend on the plan tier. In practice, the tier affects:

- Scale-out limits
- Availability of deployment slots
- VNet integration options
- Custom domains and TLS features
- Backup and restore options
- SLA and performance characteristics

For private, isolated hosting, consider:

- **App Service Environment (ASE)** for dedicated, isolated hosting in a virtual network
- VNet integration for outbound connectivity from the app to private resources
- Private Endpoint for private inbound access where supported by the chosen App Service configuration

ASE is relevant when you need stronger network isolation, internal-only exposure, or dedicated hosting characteristics that go beyond standard multi-tenant App Service. It is more expensive and operationally heavier than standard App Service, so use it when isolation requirements justify it.

**App Service deployment example**

Deploy a web app with a runtime stack:

```bash
az webapp create \
  --resource-group rg-web \
  --plan asp-web \
  --name mywebapp12345 \
  --runtime "DOTNET:8"
```

The exact runtime string varies by CLI version and supported stack. Validate available runtimes before deployment:

```bash
az webapp list-runtimes --os linux
```

**Containers: packaging consistency and deployment flexibility**

Containers package an application and its dependencies into an image. They help you move the same artifact across development, test, and production with fewer environment differences. In Azure, container hosting spans multiple services, and the right choice depends on how much orchestration and platform management you need.

Core container options to compare:

- **Azure Container Instances (ACI)** for simple, fast, single-container or small group deployments
- **Azure App Service for Containers** for web apps or APIs packaged as containers with managed web hosting
- **Azure Container Apps** for microservices, event-driven workloads, and serverless-style container hosting
- **Azure Kubernetes Service (AKS)** for advanced orchestration, complex microservices, and platform control

**Container service selection at a glance**

| Service | Best fit | Operational model | Orchestration depth |
|---|---|---|---|
| ACI | Simple jobs, burst workloads, quick container runs | Minimal management | None to very limited |
| App Service for Containers | Web apps/APIs in containers | Managed web hosting | Platform-managed app hosting |
| Container Apps | Microservices, event-driven APIs, scale-to-zero | Managed container platform | Built-in app-level orchestration |
| AKS | Large-scale orchestration, custom scheduling, advanced networking | Highest control among managed container platforms | Full Kubernetes |

**When to prefer Azure Container Apps**

Azure Container Apps is often the best middle ground when you want containerized workloads without managing Kubernetes directly. It is a strong fit for:

- Microservices
- Event-driven processing
- HTTP APIs with autoscaling
- Background workers
- Workloads that benefit from scale-to-zero
- Internal services that need simple service-to-service communication

Prefer Container Apps over ACI when you need:

- Multiple revisions
- Autoscaling beyond simple manual start/stop
- Dapr integration or service-to-service patterns
- More production-ready app hosting than a one-off container run

Prefer Container Apps over App Service for Containers when:

- You want container-native scaling behavior
- You are building microservices rather than a traditional web app
- You want a more flexible container platform without full Kubernetes overhead

Prefer AKS over Container Apps when:

- You need Kubernetes APIs and ecosystem tooling
- You require custom scheduling, operators, or advanced cluster control
- You have a platform team ready to manage cluster complexity
- You need deep control over networking, storage, and workload placement

**Stateful versus stateless in containerized designs**

Containers are often treated as stateless, but that is a design preference, not a hard rule. The container filesystem is ephemeral by default, so persistent data should usually live outside the container.

Use external storage for durable state:

- Azure Files for shared file access
- Azure Disks for attached persistent volumes in orchestrated environments
- Azure Blob Storage for object data
- Managed databases for application state

Stateful containers are possible, especially in AKS with persistent volumes and StatefulSets, but they require deliberate storage and failover design. For exam scenarios, ask whether the workload needs durable local identity, stable storage, or ordered startup. If yes, a container platform may still work, but the storage and orchestration design becomes more important.

**Container examples**

Run a simple container in ACI:

```bash
az container create \
  --resource-group rg-containers \
  --name aci-api \
  --image mcr.microsoft.com/azuredocs/aci-helloworld \
  --dns-name-label aci-api-demo \
  --ports 80
```

Deploy a containerized web app to App Service:

```bash
az webapp create \
  --resource-group rg-web \
  --plan asp-containers \
  --name mycontainerwebapp \
  --deployment-container-image-name myregistry.azurecr.io/myapi:1.0
```

Create a Container Apps environment and app:

```bash
az containerapp env create \
  --resource-group rg-ca \
  --name cae-prod \
  --location eastus

az containerapp create \
  --resource-group rg-ca \
  --name ca-api \
  --environment cae-prod \
  --image mcr.microsoft.com/azuredocs/containerapps-helloworld:latest \
  --target-port 80 \
  --ingress external
```

**Networking and private access**

Networking often decides the answer in AZ-305 more than compute features do. The same application can be hosted on different compute services, but the network exposure model changes the design.

Key distinctions:

- **VMs** can be placed in a subnet and exposed through public IPs, load balancers, application gateways, or private-only designs.
- **App Service** supports outbound VNet integration and, in supported configurations, private inbound access through Private Endpoint or ASE.
- **Containers** vary by service:
  - ACI can be deployed into a virtual network in some scenarios.
  - App Service for Containers follows App Service networking patterns.
  - Container Apps supports environment-level networking and internal ingress options.
  - AKS offers the most flexible network control.

For internal-only applications:

- Use VMs when you need full network control and custom appliances.
- Use App Service with private access features when you want managed web hosting and the app fits PaaS constraints.
- Use Container Apps or AKS when the app is container-native and private service-to-service communication matters.

For identity integration:

- Prefer managed identities where supported.
- Use Key Vault for secrets instead of embedding credentials in images or config files.
- Use private endpoints for data services when the app must avoid public internet paths.

**Decision examples for common AZ-305 scenarios**

1. A legacy .NET Framework intranet app requires IIS, custom COM components, and a third-party agent.
   - Choose a **VM**.
   - Reason: OS-level control and legacy compatibility matter more than platform convenience.

2. A Linux-based REST API is already containerized and needs predictable deployments with minimal ops overhead.
   - Choose **App Service for Containers** or **Container Apps**.
   - Reason: You want managed hosting without VM administration.
   - If the API is microservice-oriented or needs scale-to-zero, prefer **Container Apps**.

3. A public web app needs rapid deployment, autoscale, and easy slot-based releases.
   - Choose **App Service**.
   - Reason: It is optimized for web hosting and deployment workflows.

4. An internal-only service must run in a private network with strict isolation and dedicated hosting.
   - Choose **ASE**, **AKS**, or a **VM-based** design depending on operational maturity.
   - Reason: Private access and isolation are primary requirements.

5. A batch job needs to run a container for a short period and then exit.
   - Choose **ACI** or a job-oriented container pattern.
   - Reason: You do not need a full web hosting platform.

**Operational comparison: patching, backup, monitoring, and DR**

The more control you take, the more operational work you own.

- **VMs**
  - You patch the guest OS and installed software.
  - You design backup and restore for disks and application data.
  - You monitor OS metrics, logs, and application health.
  - You plan disaster recovery at the VM, application, and data layers.

- **App Service**
  - Azure patches the platform.
  - You focus on application code, configuration, and data protection.
  - Backups are simpler for supported plans, but the app should still rely on external durable data stores.
  - Disaster recovery often uses deployment automation, traffic routing, and data-layer replication.

- **Containers**
  - You patch the base image and rebuild/redeploy.
  - You externalize state and secrets.
  - Monitoring includes container logs, health probes, and platform metrics.
  - DR usually depends on image registry resilience, redeployment automation, and data replication.

A practical rule: if the workload team wants to spend time on application features rather than host maintenance, prefer PaaS or managed containers. If the workload requires host customization or legacy compatibility, accept the VM overhead.

**Quick selection guide**

- Choose **VMs** when you need maximum control, legacy compatibility, or custom host-level software.
- Choose **App Service** when you are hosting a web app or API and want the simplest managed platform.
- Choose **App Service for Containers** when you want container packaging but still want web-app-style hosting.
- Choose **Container Apps** when you want managed container hosting for microservices, APIs, or event-driven workloads.
- Choose **AKS** when you need full Kubernetes control and are prepared to manage that complexity.
- Choose **ACI** for simple, short-lived, or burst container workloads.

**Exam-focused reminders**

- Do not assume “stateless” means “no data”; it usually means no reliance on local instance state.
- Do not assume App Service replaces every container platform; it is best for web hosting, not all container orchestration needs.
- Do not assume VMs are obsolete; they remain the correct answer for compatibility, isolation, and custom control.
- Do not overstate private networking features; verify whether the service supports inbound private access, outbound VNet integration, or both.
- Do not confuse container packaging with orchestration; a container image is not the same as a managed platform.

**Practical architecture patterns**

- Use **App Service + Azure SQL Database** for a standard web app with minimal ops.
- Use **Container Apps + Azure Container Registry + Key Vault** for a modern API platform.
- Use **VMs + load balancer + managed disks** for a legacy application that cannot be refactored.
- Use **AKS + persistent volumes + ingress controller** for a complex microservices platform.
- Use **ASE or private networking** when the app must remain isolated from the public internet.

**Common pitfalls**

- Treating App Service as if it requires no scaling design.
- Assuming containers automatically solve state management.
- Choosing AKS when the workload does not justify Kubernetes complexity.
- Choosing VMs when a managed PaaS option clearly fits the app.
- Ignoring network isolation requirements until late in the design.
