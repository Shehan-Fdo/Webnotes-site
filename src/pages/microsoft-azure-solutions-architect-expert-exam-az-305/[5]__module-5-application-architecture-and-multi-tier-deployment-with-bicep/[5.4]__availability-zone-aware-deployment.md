## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Availability zones: the core idea**

Availability zones are physically separate locations within an Azure region. Each zone has independent power, cooling, and networking, so a failure in one zone does not automatically take down the others. You use zone-aware design to reduce the blast radius of datacenter-level failures while staying within a single region for lower latency and simpler data residency.

A zone-aware architecture is not the same as a multi-region architecture. Zones protect against localized infrastructure failure inside a region; regions protect against broader regional outages. In practice, you often combine both.

- **Zone**: an isolated datacenter group within a region.
- **Region**: a geographic area that contains one or more datacenters and may contain multiple zones.
- **Zone-redundant**: a service or deployment model that spans multiple zones.
- **Zonal**: pinned to a specific zone.

**When to use availability zones**

Use zones when you need higher availability than a single datacenter can provide, but you do not want the complexity of active-active multi-region design. Zones are a strong fit for production workloads that can tolerate some cross-zone latency and replication overhead.

Typical use cases include:
- Web applications with stateless front ends and replicated back ends.
- Databases that support synchronous or zone-aware replication.
- Shared services such as load balancers, gateways, and storage accounts configured for zone resilience.
- Mission-critical workloads where a single-zone outage must not cause a full service outage.

> Zone-aware design improves resilience, but it does not remove the need for backups, disaster recovery, or application-level fault handling.

**Deployment models you should recognize**

Azure services usually expose one of three patterns:

| Pattern | Meaning | Example use |
|---|---|---|
| Zonal | You choose a specific zone | A VM in zone 1 |
| Zone-redundant | Azure spreads instances across zones | A zone-redundant storage account |
| Regional | The service is regional and abstracts zones | Some PaaS services with built-in resilience |

For exam scenarios, identify whether the service is:
- **Pinned** to a zone and therefore vulnerable to that zone’s outage.
- **Spread** across zones and therefore more resilient.
- **Regional only**, where zone behavior is hidden or limited by the service model.

**Compute design patterns**

Virtual Machines (VMs) are the most common example of zonal deployment. You place individual VMs in specific zones and then distribute instances across zones for high availability. A single VM in one zone is not zone resilient; you need at least two instances across different zones, usually behind a load balancer or application gateway.

A common pattern is:
- Deploy two or more VM instances across zones 1, 2, and 3.
- Place them in an availability set only when zones are not available in the region or when you need intra-datacenter fault domain protection instead of zone protection.
- Use a load balancer or Application Gateway to distribute traffic.

Example Azure CLI for zonal VM placement:
```bash
az vm create \
  --resource-group rg-app \
  --name vm-app-01 \
  --image Ubuntu2204 \
  --admin-username azureuser \
  --generate-ssh-keys \
  --zone 1
```

If you need multiple instances, create one VM per zone or use a Virtual Machine Scale Set (VMSS) with zone distribution. VMSS is often the better choice for stateless workloads because it simplifies scaling and replacement.

Example VMSS across zones:
```bash
az vmss create \
  --resource-group rg-app \
  --name vmss-web \
  --image Ubuntu2204 \
  --instance-count 3 \
  --zones 1 2 3 \
  --upgrade-policy-mode automatic
```

**Important design point**

- **Availability sets** protect against hardware and rack failures inside a datacenter.
- **Availability zones** protect against datacenter-level failures inside a region.
- Do not treat them as interchangeable.

**Load balancing and traffic distribution**

A zone-aware front end usually needs a regional entry point that can send traffic to healthy instances in multiple zones. Azure Load Balancer and Application Gateway are common choices for regional traffic distribution.

For Standard Load Balancer:
- The load balancer is regional.
- Frontend IP configurations can be zonal or zone-redundant depending on the SKU and configuration.
- Backend pool members are the zonal resources, such as VMs or VMSS instances.
- Health probes determine which instances receive traffic.

For Application Gateway:
- You can deploy it in a zone-aware configuration in supported regions and SKUs.
- It is useful for Layer 7 routing, TLS termination, and web application firewall integration.
- It is not a replacement for cross-region disaster recovery.

Example topology:
- Zone 1: VM or VMSS instance
- Zone 2: VM or VMSS instance
- Zone 3: VM or VMSS instance
- Regional load balancer or Application Gateway in front

**Storage and data layer considerations**

Storage design is often the deciding factor in zone-aware architecture. You need to know whether the service replicates across zones automatically or whether you must design replication yourself.

Common patterns:
- **Zone-redundant storage** for object and file data where supported.
- **Geo-redundant storage** for cross-region durability.
- **Application-managed replication** for databases and stateful services.

For Azure Storage, redundancy options typically include:
- Locally redundant storage (LRS): copies within one datacenter.
- Zone-redundant storage (ZRS): copies across zones in a region.
- Geo-redundant variants: copies to a paired region as well.

Use ZRS when you want regional durability across zones without moving data to another region. Use geo-redundant options when you also need regional disaster recovery.

Example storage account creation with ZRS:
```bash
az storage account create \
  --name stappzone01 \
  --resource-group rg-app \
  --location eastus \
  --sku Standard_ZRS \
  --kind StorageV2
```

For managed disks, zone placement matters:
- A zonal VM typically uses disks in the same zone.
- If the VM fails over to another zone, you usually need a recovery process that recreates the VM and reattaches data disks as supported by the design.
- Do not assume a disk automatically moves across zones with the VM.

**Database and stateful service design**

Stateful services require careful attention because the application can be zone-aware while the data layer is not.

General guidance:
- Prefer services with built-in zone redundancy when available.
- Use replication, failover groups, or clustering when the service does not automatically span zones.
- Test failover behavior, not just deployment success.

Examples:
- Azure SQL Database can support zone redundancy in certain service tiers and regions, but not every deployment model has the same capability.
- Azure SQL Managed Instance zone support depends on the service tier, region, and configuration.
- Azure Database for PostgreSQL and MySQL offerings may support zone-redundant high availability in specific SKUs and regions.
- Cosmos DB is designed for high availability and can be deployed with multi-region capabilities, which is a different resilience model from single-region zone awareness.

For SQL connectivity, use the client library or driver’s supported authentication mechanism. Managed identity is typically handled through the SDK, access token, or driver-specific authentication settings rather than a generic connection string keyword.

Example .NET connection pattern using an access token:
```csharp
using Azure.Core;
using Azure.Identity;
using Microsoft.Data.SqlClient;

var credential = new DefaultAzureCredential();
var token = await credential.GetTokenAsync(
    new TokenRequestContext(new[] { "https://database.windows.net/.default" }));

var connection = new SqlConnection("Server=tcp:myserver.database.windows.net,1433;Database=mydb;Encrypt=True;TrustServerCertificate=False;");
connection.AccessToken = token.Token;
await connection.OpenAsync();
```

**Platform services and zone behavior**

Many Azure PaaS services abstract zone placement, but their resilience characteristics still matter.

Examples of zone-aware or zone-capable services:
- App Service with zone redundancy in supported plans and regions.
- Azure Kubernetes Service (AKS) with node pools spread across zones.
- Azure SQL offerings with zone-redundant options in supported configurations.
- Azure Cache for Redis and messaging services with zone-aware tiers or replicas where supported.

For AKS, zone-aware design usually means:
- Multiple node pools or nodes across zones.
- Pod disruption budgets to prevent too many replicas from being evicted at once.
- Storage classes that match the zone behavior of the workload.

Example AKS cluster with zones:
```bash
az aks create \
  --resource-group rg-app \
  --name aks-prod \
  --node-count 3 \
  --zones 1 2 3 \
  --enable-managed-identity \
  --generate-ssh-keys
```

**Traffic routing across zones and regions**

Use the right routing layer for the right failure domain.

- **Within a region across zones**: use regional load balancing, Application Gateway, or service-native zone redundancy.
- **Across regions**: use Azure Front Door, Traffic Manager, or application-level failover.

Azure Front Door is a global Layer 7 service. It routes user traffic to healthy origins across regions and can improve availability during regional outages. It does not provide intra-region zone failover in the same way a regional load balancer or Application Gateway does.

Azure Traffic Manager is DNS-based and can direct clients to different endpoints, but DNS caching can slow failover. It is useful for regional failover, not for fast in-region zone balancing.

**Designing for failure**

A good zone-aware design assumes one zone can fail at any time. Your application should continue operating with reduced capacity.

Design principles:
- Keep front ends stateless when possible.
- Replicate state across zones.
- Use health probes and automatic instance replacement.
- Avoid single-zone dependencies such as a lone VM, a single-zone disk, or a zonal gateway without redundancy.
- Test failover under load.

A practical pattern for a web app:
1. Deploy three app instances across zones 1, 2, and 3.
2. Put a regional load balancer or Application Gateway in front.
3. Store session state in a distributed cache or external state store.
4. Use ZRS or replicated storage for shared files.
5. Place the database in a zone-aware or highly available configuration.
6. Validate that the app still serves traffic when one zone is unavailable.

resource nic 'Microsoft.Network/networkInterfaces@2023-09-01' = {

      {

            id: subnetId

        }
      }

  }

resource vm 'Microsoft.Compute/virtualMachines@2023-09-01' = {
  name: 'vm-app-01'
  location: location
  zones: [
    zone
  ]
  properties: {
    hardwareProfile: {
      vmSize: 'Standard_D2s_v5'
    }
    networkProfile: {
      networkInterfaces: [
        {
          id: nic.id

**Exam cues and decision hints**

When you see a scenario, ask:
- Does the workload need protection from a datacenter failure or a regional failure?
- Is the service zonal, zone-redundant, or regional?
- Can the application tolerate cross-zone latency?
- Is the data layer replicated across zones?
- Is failover automatic, manual, or DNS-based?

Common exam traps:
- Assuming a single VM in one zone is highly available.
- Confusing availability sets with availability zones.
- Treating Front Door as an intra-region zone failover tool.
- Assuming every PaaS service supports zone redundancy in every tier and region.
- Forgetting that stateful components often determine the real resilience level.

**Operational validation**

You should verify zone placement and resilience after deployment.

Useful commands:
```bash
az vm show -g rg-app -n vm-app-01 --query zones
az vmss list-instances -g rg-app -n vmss-web --query "[].{name:name,zone:zones}"
az network public-ip show -g rg-app -n pip-app --query "{sku:sku.name,zones:zones}"
```

You can also inspect resources in the portal:
- Check the zone field on compute resources.
- Confirm frontend IP configuration behavior on load balancers.
- Review redundancy settings on storage and database services.
- Test failover by stopping or deallocating a zonal instance and observing traffic shift.

**Practical architecture summary**

A strong zone-aware Azure solution usually includes:
- Multiple compute instances spread across zones.
- A regional traffic manager such as Load Balancer or Application Gateway.
- Zone-redundant storage or replicated data services.
- Health probes and automated recovery.
- A separate regional disaster recovery plan.

**What to remember for AZ-305**

- **Zones** protect against datacenter failure inside a region.
- **Zone-redundant** services spread risk across zones.
- **Zonal** resources need explicit distribution to achieve resilience.
- **State** is the hardest part of zone-aware design.
- **Global routing** services complement zones but do not replace them.
