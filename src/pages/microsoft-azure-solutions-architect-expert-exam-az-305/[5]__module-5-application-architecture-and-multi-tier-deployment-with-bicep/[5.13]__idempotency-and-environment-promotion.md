## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Idempotency in Azure architecture**

Idempotency means you can run the same operation multiple times and end up with the same intended result. In cloud architecture, this matters because deployments, configuration changes, and automation often rerun due to retries, pipeline re-execution, or drift correction.

For AZ-305, separate the idea of idempotent **infrastructure provisioning** from idempotent **application and data-plane operations**:

- **Infrastructure idempotency** means your template or IaC tool converges resources to a desired state.
- **Application/data-plane idempotency** means repeated app actions do not create duplicate business effects.
- These are related but not identical; a deployment can be idempotent for infrastructure while still causing non-idempotent data changes.

Common Azure examples include:

- Reapplying an ARM template, Bicep file, or Terraform plan to the same scope.
- Re-running a pipeline stage that updates App Service settings without duplicating resources.
- Retrying a message handler that must avoid double-charging, double-inserting, or double-sending notifications.

**Why idempotency matters**

Idempotency reduces operational risk in automated delivery:

- You can safely retry failed deployments.
- You can recover from partial failures without manual cleanup.
- You can promote the same artifact through environments with controlled differences.
- You can detect and correct configuration drift more reliably.

> A deployment process is only as safe as its least idempotent step. If one script inserts data blindly or appends configuration repeatedly, the whole pipeline becomes fragile.

**Infrastructure-layer idempotency**

Infrastructure as Code (IaC) tools are designed to converge state, not just execute commands. In Azure, this is usually achieved with ARM templates, Bicep, Terraform, or deployment scripts that check current state before changing it.

Typical patterns:

- Use declarative templates instead of imperative scripts when possible.
- Prefer resource definitions that describe the desired end state.
- Avoid random names unless uniqueness is required.
- Use `what-if` or plan-style previews before applying changes.

Example Bicep deployment:

```bicep
param location string = resourceGroup().location
param appName string
param skuName string = 'P1v3'

resource plan 'Microsoft.Web/serverfarms@2023-12-01' = {
  name: '${appName}-plan'
  location: location
  sku: {
    name: skuName
    tier: 'PremiumV3'
  }
}

resource app 'Microsoft.Web/sites@2023-12-01' = {
  name: appName
  location: location
  properties: {
    serverFarmId: plan.id
    httpsOnly: true
  }
}
```

Useful CLI commands:

```bash
az deployment group what-if \
  --resource-group rg-app-dev \
  --template-file main.bicep \
  --parameters appName=contoso-web location=eastus

az deployment group create \
  --resource-group rg-app-dev \
  --template-file main.bicep \
  --parameters appName=contoso-web location=eastus
```

**Application and data-plane idempotency**

Application operations often need explicit design because they are not automatically idempotent. A retry can create duplicate rows, duplicate events, or repeated side effects unless you design for safe repetition.

Use these techniques:

- Include a client-generated request ID or idempotency key.
- Make writes conditional on current state.
- Use upserts where appropriate.
- Design message consumers to tolerate duplicate delivery.
- Store processed message IDs when exactly-once behavior is not guaranteed.

Examples:

- `PUT /orders/{orderId}` is often more idempotent than `POST /orders`.
- A payment API may accept an idempotency key to prevent duplicate charges.
- A queue-triggered function should ignore messages already processed.

Example pseudo-logic for a handler:

```python
def handle_event(event):
    if already_processed(event.id):
        return "duplicate ignored"

    apply_business_change(event)
    mark_processed(event.id)
```

**Environment promotion overview**

Environment promotion is the controlled movement of the same application artifact through environments such as dev, test, staging, and production. The goal is to keep the artifact stable while changing only the environment-specific inputs.

Promotion should separate:

- **Application configuration management**
- **Infrastructure provisioning**
- **Database/schema changes**
- **Release orchestration and approvals**

This separation helps you avoid rebuilding the app for each environment and reduces configuration drift.

**Promoting application configuration separately from infrastructure**

Application settings are often promoted independently from the infrastructure that hosts them. In Azure, this includes:

- App settings
- Connection strings
- Feature flags
- Environment-specific variables
- Secrets referenced from Azure Key Vault

The infrastructure may be identical across environments, while the configuration values differ. For example, the same App Service or Container App can run in dev and prod with different settings injected at deployment time.

Common approaches:

- Use environment-specific parameter files.
- Use variable groups in Azure DevOps.
- Use GitHub Actions environment variables or secrets.
- Reference Key Vault for secrets instead of hardcoding them.
- Keep the number of environment differences as small as possible.

Example App Service settings update:

```bash
az webapp config appsettings set \
  --resource-group rg-app-prod \
  --name contoso-web-prod \
  --settings \
    ASPNETCORE_ENVIRONMENT=Production \
    FeatureFlags__NewCheckout=false \
    ApiBaseUrl=https://api.contoso.com
```

Example connection string update:

```bash
az webapp config connection-string set \
  --resource-group rg-app-prod \
  --name contoso-web-prod \
  --connection-string-type SQLAzure \
  --settings DefaultConnection="Server=tcp:contoso.database.windows.net,1433;Database=appdb;User ID=appuser;Password=***"
```

**Parameterization strategies for promotion**

Parameterization keeps the deployment artifact constant while allowing controlled variation by environment. This is a core exam concept because it supports repeatability and reduces configuration drift.

Common strategies:

- **Parameter files**: Separate files for dev, test, and prod values.
- **Variable groups**: Centralized pipeline variables in Azure DevOps.
- **Template specs**: Versioned, reusable Azure Resource Manager templates stored in Azure.
- **Pipeline variables and secrets**: Scoped to stages or environments.
- **Key Vault references**: Resolve secrets at runtime without storing them in the pipeline.

Example ARM parameter file:

```json
{
  "$schema": "https://schema.management.azure.com/schemas/2019-04-01/deploymentParameters.json#",
  "contentVersion": "1.0.0.0",
  "parameters": {
    "appName": { "value": "contoso-web-prod" },
    "skuName": { "value": "P1v3" },
    "featureFlagNewCheckout": { "value": false }
  }
}
```

Example Bicep parameterization:

```bicep
param appName string
param skuName string = 'B1'
param featureFlagNewCheckout bool = false
param sqlServerName string
param sqlDatabaseName string
```

Example Azure DevOps variable group usage:

```yaml
variables:
- group: contoso-prod-vars

steps:
- script: |
    echo "Deploying $(appName) to $(environmentName)"
```

Template specs are useful when you want a centrally managed, versioned template that multiple teams or pipelines can consume consistently. They help standardize deployments across subscriptions and resource groups while still allowing parameter overrides.

Keep differences minimal and controlled:

- Use the same template across environments.
- Override only values that must differ.
- Avoid branching logic that makes prod structurally different from dev.
- Prefer feature flags over separate code branches when possible.

**Feature flags and controlled rollout**

Feature flags let you deploy code before enabling functionality. This supports promotion because you can move the same artifact through environments and enable features gradually.

Benefits:

- Decouple deployment from release.
- Reduce risk by hiding incomplete or risky features.
- Enable testing in production-like environments without exposing all users.

Typical pattern:

- Deploy code with the feature disabled.
- Validate in lower environments.
- Enable the flag for a small audience or internal users.
- Expand gradually after verification.

Example configuration:

```json
{
  "FeatureManagement": {
    "NewCheckout": false,
    "UseNewSearchIndex": true
  }
}
```

**Database and schema promotion**

Database promotion is often the hardest part of environment promotion because schema and data changes can break compatibility. Treat database changes as part of the release pipeline, not as an afterthought.

Safe database promotion practices:

- Use backward-compatible schema changes first.
- Add columns before removing old ones.
- Deploy code that can read both old and new schema during transition.
- Use migrations that are repeatable and versioned.
- Seed only the minimum required reference data.

Examples of backward-compatible changes:

- Add a nullable column before making it required.
- Add a new table and write to both old and new structures temporarily.
- Expand an enum or lookup table before switching application logic.
- Avoid dropping columns until all dependent code is retired.

Example migration flow:

1. Add `NewStatus` column as nullable.
2. Deploy application code that writes both `Status` and `NewStatus`.
3. Backfill existing rows.
4. Switch reads to `NewStatus`.
5. Remove legacy column in a later release.

Example SQL migration:

```sql
ALTER TABLE dbo.Orders
ADD NewStatus nvarchar(50) NULL;
GO

UPDATE dbo.Orders
SET NewStatus = Status
WHERE NewStatus IS NULL;
GO
```

Data seeding guidance:

- Seed reference data that the app requires to start.
- Make seed scripts idempotent by checking for existing rows.
- Avoid seeding transactional data in production unless explicitly required.
- Keep seed data versioned with the application.

Example idempotent seed pattern:

```sql
IF NOT EXISTS (SELECT 1 FROM dbo.FeatureCatalog WHERE FeatureName = 'NewCheckout')
BEGIN
    INSERT INTO dbo.FeatureCatalog (FeatureName, IsEnabled)
    VALUES ('NewCheckout', 0);
END
```

**Promotion models: blue-green, canary, and deployment rings**

Promotion is not only about moving from one environment to another. It also includes how traffic is shifted after deployment.

Common models:

- **Blue-green**: Two identical environments; switch traffic from blue to green after validation.
- **Canary**: Send a small percentage of traffic to the new version first.
- **Deployment rings**: Roll out to internal users, then pilot users, then broader production rings.
- **Phased rollout**: Increase exposure in stages based on health and telemetry.

Deployment rings are especially relevant in enterprise Azure environments because they support controlled adoption across business units or user cohorts.

Example ring strategy:

- Ring 0: engineering and test users
- Ring 1: pilot business users
- Ring 2: regional production users
- Ring 3: full production

This model works well when you need governance, validation, and rollback checkpoints between stages.

**Full promotion pipeline example**

A strong exam answer should describe a pipeline where the same artifact moves through environments while only parameters and approvals change.

Example flow:

1. Build the application once and publish an immutable artifact.
2. Deploy to dev using dev parameters and nonproduction secrets.
3. Run automated tests and smoke checks.
4. Promote the same artifact to test with test-specific variable group values.
5. Apply database migrations in a backward-compatible way.
6. Approve staging or preproduction deployment.
7. Deploy to staging with production-like settings and feature flags off.
8. Validate telemetry, performance, and integration points.
9. Approve production deployment.
10. Deploy the same artifact to prod with prod parameters, Key Vault references, and controlled rollout.

Example Azure DevOps-style YAML sketch:

```yaml
stages:
- stage: Build
  jobs:
  - job: BuildApp
    steps:
    - script: dotnet publish -c Release -o आउट
```

```yaml
- stage: Dev
  dependsOn: Build
  variables:
  - group: contoso-dev-vars
  jobs:
  - deployment: DeployDev
    environment: dev
    strategy:
      runOnce:
        deploy:
          steps:
          - script: az deployment group create --resource-group rg-dev --template-file main.bicep --parameters @dev.parameters.json
```

```yaml
- stage: Prod
  dependsOn: Staging
  variables:
  - group: contoso-prod-vars
  jobs:
  - deployment: DeployProd
    environment: prod
    strategy:
      runOnce:
        deploy:
          steps:
          - script: az deployment group create --resource-group rg-prod --template-file main.bicep --parameters @prod.parameters.json
```

In a real pipeline, you would also add approvals, checks, and health gates before production.

**Approvals, gates, and governance**

Promotion in Azure enterprise environments often requires human approval or policy-based checks. These controls reduce risk when moving from lower environments to production.

Useful controls:

- Manual approvals for production stages.
- Azure Policy compliance checks.
- Health probes and smoke tests.
- Change windows and release calendars.
- Service connection and environment permissions.

Approvals are not a substitute for idempotency; they complement it. If a deployment is safe to rerun, approvals become less risky because retries do not create inconsistent state.

**Exam distinctions to remember**

- **Idempotent deployment** means rerunning the deployment does not create duplicate infrastructure or unintended configuration changes.
- **Idempotent application behavior** means repeated requests or retries do not duplicate business actions.
- **Promotion** means moving the same artifact through environments with controlled parameters.
- **Release** means making functionality available to users, which may happen separately from deployment.
- **Schema migration** must be backward compatible when old and new versions may coexist during rollout.

**Practical Azure commands and patterns**

Inspect current state before changing it:

```bash
az resource show \
  --ids /subscriptions/<subId>/resourceGroups/rg-app-prod/providers/Microsoft.Web/sites/contoso-web-prod
```

Deploy with a parameter file:

```bash
az deployment group create \
  --resource-group rg-app-test \
  --template-file main.bicep \
  --parameters @test.parameters.json
```

Use Key Vault references in App Service settings:

```bash
az webapp config appsettings set \
  --resource-group rg-app-prod \
  --name contoso-web-prod \
  --settings DbPassword='@Microsoft.KeyVault(SecretUri=https://contoso-kv.vault.azure.net/secrets/DbPassword/...)'
```

Use deployment slots for safer promotion when supported:

- Deploy to a staging slot first.
- Warm up and validate the slot.
- Swap slots to promote the same build into production.
- Keep slot settings marked as sticky when they must not move.

**Common pitfalls**

- Rebuilding the app separately for each environment instead of promoting one artifact.
- Hardcoding environment-specific values into code.
- Dropping or renaming database columns too early.
- Treating infrastructure idempotency as sufficient for application safety.
- Using feature flags without a clear enablement and rollback plan.
- Allowing dev, test, and prod to drift into different topologies without justification.

**What to emphasize in AZ-305 answers**

When asked about idempotency or environment promotion, focus on:

- Repeatable deployments with declarative IaC.
- A single immutable artifact promoted across environments.
- Parameterization through files, variable groups, and template specs.
- Separate handling of app settings, connection strings, feature flags, and secrets.
- Backward-compatible database migrations and idempotent seed data.
- Phased rollout models such as blue-green, canary, and deployment rings.
- Approvals and health gates that support controlled production release.
