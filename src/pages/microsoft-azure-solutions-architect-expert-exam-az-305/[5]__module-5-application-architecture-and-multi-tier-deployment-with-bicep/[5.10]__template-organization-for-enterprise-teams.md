## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Enterprise template organization**

Enterprise teams use templates to make infrastructure repeatable, reviewable, and governable. In Azure, the main goal is not just to deploy resources faster, but to ensure every deployment follows the same standards for naming, tagging, security, and policy compliance.

A good template organization strategy separates concerns:

- **Platform teams** define shared foundations such as networking, identity, logging, and policy.
- **Application teams** consume approved modules and focus on app-specific resources.
- **Governance teams** enforce guardrails through Azure Policy, role-based access control (RBAC), and deployment controls.

This separation reduces drift, improves auditability, and makes it easier to scale across subscriptions, management groups, and regions.

**Why template organization matters**

Poorly organized templates create duplication, inconsistent parameters, and hidden dependencies. In enterprise environments, that leads to:

- Slower reviews and approvals
- Inconsistent security settings
- Hard-to-maintain parameter files
- Repeated logic across teams
- Higher risk of deployment failures

Well-organized templates help you:

- Reuse common infrastructure patterns
- Standardize resource naming and tagging
- Separate reusable modules from environment-specific values
- Support CI/CD pipelines with predictable inputs
- Apply policy and compliance checks before deployment

**Common Azure template approaches**

Azure teams typically organize infrastructure as code using one or more of these approaches:

- **ARM templates** for native Azure Resource Manager deployments
- **Bicep** as a simpler, more maintainable abstraction over ARM
- **Template Specs** to publish versioned, reusable templates in Azure
- **Terraform** when the organization standardizes on multi-cloud or provider-agnostic workflows

For AZ-305, focus on how these approaches support enterprise reuse and governance rather than memorizing syntax alone.

**Recommended enterprise structure**

A practical repository structure separates reusable building blocks from deployment entry points.

```text
repo/
├── modules/
│   ├── network/
│   ├── compute/
│   ├── storage/
│   ├── identity/
│   └── monitoring/
├── environments/
│   ├── dev/
│   ├── test/
│   ├── prod/
│   └── shared/
├── policies/
├── pipelines/
├── parameter-files/
└── docs/
```

This layout supports:

- **Modules** for reusable components
- **Environment folders** for environment-specific parameters
- **Policies** for governance artifacts
- **Pipelines** for automated validation and deployment
- **Docs** for usage, ownership, and change control

A module-first design is usually better than copying full templates for each workload. You define a network module once, then call it from multiple application deployments with different parameters.

**Bicep organization patterns**

Bicep is often the best fit for Azure-first enterprises because it supports modularization, readable syntax, and native Azure deployment workflows.

Typical Bicep organization includes:

- A main entry file such as `main.bicep`
- Reusable modules such as `modules/vnet.bicep` or `modules/keyvault.bicep`
- Parameter files for environment-specific values
- Output values for downstream automation

Example module usage:

```bicep
module network 'modules/network.bicep' = {
  name: 'networkDeployment'
  params: {
    vnetName: 'vnet-prod-eastus'
    addressPrefix: '10.10.0.0/16'
    subnetPrefix: '10.10.1.0/24'
  }
}
```

Key Bicep benefits in enterprise teams:

- **Modules** reduce duplication and centralize best practices
- **Parameters** allow the same template to serve multiple environments
- **Outputs** let one deployment feed another
- **Scopes** support resource group, subscription, management group, and tenant deployments

Use Bicep when you want Azure-native governance and simpler authoring than raw JSON ARM templates.

**ARM template organization**

ARM templates remain important because many enterprises still maintain existing JSON-based deployments. A well-structured ARM solution usually includes:

- A root template that orchestrates deployment
- Linked or nested templates for reusable components
- Parameter files for each environment
- Consistent naming and tagging conventions

Example ARM deployment command:

```bash
az deployment group create \
  --resource-group rg-app-prod \
  --template-file main.json \
  --parameters @prod.parameters.json
```

ARM templates are verbose, so organization matters more than ever. Keep templates small and focused, and avoid putting unrelated resources into one large file unless they are tightly coupled.

**Template Specs for centralized reuse**

Template Specs let you store versioned templates in Azure and deploy them like managed artifacts. They are useful when multiple teams need the same approved template but should not copy the source into every repository.

Use Template Specs when you need:

- Central publishing of approved templates
- Version control for shared infrastructure patterns
- Controlled consumption across subscriptions and teams
- A single source of truth for common deployments

A typical enterprise pattern is:

- Platform team publishes a Template Spec
- Application teams deploy from that spec
- Governance team approves new versions before release

This model works well for shared services such as hub networking, logging workspaces, or baseline security resources.

**Governance and guardrails**

Template organization should align with governance. Templates are not just deployment artifacts; they are enforcement points for standards.

Important governance practices include:

- **Tagging** for cost center, owner, environment, and data classification
- **Naming standards** for predictable resource identification
- **Azure Policy** to deny or audit noncompliant resources
- **RBAC** to limit who can deploy or modify templates
- **Locks** for critical shared resources
- **Managed identities** for pipeline authentication instead of stored secrets

Example policy-aligned tagging parameters:

```bicep
param environment string
param owner string
param costCenter string

resource rg 'Microsoft.Resources/resourceGroups@2021-04-01' = {
  name: 'rg-app-${environment}'
  location: resourceGroup().location
  tags: {
    Environment: environment
    Owner: owner
    CostCenter: costCenter
  }
}
```

When you standardize tags in templates, you make chargeback, reporting, and incident ownership much easier.

**Parameter management**

Enterprise teams should avoid hardcoding values in templates. Instead, separate reusable logic from environment-specific data.

Best practices:

- Use parameter files for dev, test, and prod
- Keep secrets out of parameter files
- Store secrets in Azure Key Vault and reference them securely
- Validate parameter values in pipelines before deployment
- Use descriptive parameter names that match business meaning

Example parameter file:

```json
{
  "$schema": "https://schema.management.azure.com/schemas/2019-04-01/deploymentParameters.json#",
  "contentVersion": "1.0.0.0",
  "parameters": {
    "environment": { "value": "prod" },
    "owner": { "value": "platform-team" },
    "costCenter": { "value": "CC1024" }
  }
}
```

For secrets, prefer Key Vault references or pipeline secret variables rather than plain text values in source control.

**CI/CD integration**

Templates become much more valuable when integrated into continuous integration and continuous delivery (CI/CD) pipelines. The pipeline should validate, test, and deploy templates in stages.

A common pipeline flow is:

1. Lint or validate the template
2. Run what-if or preview deployment
3. Approve changes
4. Deploy to a nonproduction environment
5. Promote to production after verification

Example Azure CLI validation:

```bash
az deployment group validate \
  --resource-group rg-app-dev \
  --template-file main.bicep \
  --parameters @dev.parameters.json
```

Example what-if preview:

```bash
az deployment group what-if \
  --resource-group rg-app-prod \
  --template-file main.bicep \
  --parameters @prod.parameters.json
```

What-if is especially useful in enterprise teams because it helps reviewers understand the impact before resources change.

**Module design principles**

Good modules are small, predictable, and reusable. They should represent a single responsibility, such as:

- Virtual network creation
- Key Vault provisioning
- App Service deployment
- Log Analytics workspace setup
- Role assignment configuration

Design modules so they:

- Expose only necessary parameters
- Return useful outputs
- Avoid hidden dependencies
- Use secure defaults
- Support multiple environments without code duplication

A module should not try to solve every scenario. If a module becomes too large, split it into smaller components.

**Versioning and change control**

Enterprise template organization depends on controlled change. If teams consume shared templates, versioning becomes critical.

Recommended practices:

- Use semantic versioning for shared modules and Template Specs
- Publish breaking changes as new major versions
- Keep backward compatibility when possible
- Document deprecations and migration paths
- Require pull request review for template changes

Versioning helps you avoid breaking dozens of workloads when a shared module changes.

**Security considerations**

Templates can introduce risk if they are not reviewed carefully. Common security issues include:

- Overly permissive RBAC assignments
- Public network exposure by default
- Secrets embedded in parameters
- Missing diagnostic settings
- Inconsistent encryption settings

Mitigations:

- Use least privilege for deployment identities
- Enforce private endpoints where required
- Integrate secret retrieval from Key Vault
- Add diagnostic settings to critical resources
- Apply Azure Policy to block insecure configurations

> Never store long-lived secrets in source control or plain-text parameter files. Use managed identities, Key Vault, or secure pipeline variables instead.

**Practical deployment example**

A simple enterprise deployment pattern might look like this:

```bash
az group create \
  --name rg-app-prod \
  --location eastus

az deployment group create \
  --resource-group rg-app-prod \
  --template-file main.bicep \
  --parameters @prod.parameters.json
```

This pattern works well when:

- The resource group already exists or is created as part of a higher-level process
- The template is environment-aware through parameters
- The pipeline controls approvals and deployment timing

**When to use management group or subscription scope**

Not all templates should deploy at the resource group level. Enterprise foundation work often belongs at higher scopes.

Use these scopes when appropriate:

- **Management group scope** for policy assignments, role assignments, and governance baselines
- **Subscription scope** for shared services, resource groups, and subscription-level configuration
- **Resource group scope** for application workloads and tightly related resources
- **Tenant scope** for directory-wide or cross-tenant governance scenarios, where supported

Choosing the right scope helps you separate platform governance from workload deployment.

**Operational documentation**

Template repositories should include documentation that explains:

- What the template deploys
- Which parameters are required
- Which environments are supported
- Who owns the module
- How to test and validate changes
- How to roll back or redeploy safely

This documentation reduces support burden and helps new teams adopt the standard patterns quickly.

**Exam-focused takeaways**

For AZ-305, remember these design principles:

- Use reusable modules to reduce duplication
- Separate environment values from deployment logic
- Apply governance through policy, RBAC, and tagging
- Prefer versioned shared artifacts for enterprise reuse
- Validate templates before deployment with pipeline checks
- Choose the correct deployment scope for the control plane task

**Quick comparison**

| Approach | Best for | Strengths | Tradeoffs |
|---|---|---|---|
| ARM templates | Native Azure deployments | Mature, supported, expressive | Verbose JSON, harder to maintain |
| Bicep | Azure-first infrastructure as code | Readable, modular, simpler authoring | Azure-focused, requires Bicep tooling |
| Template Specs | Centralized shared templates | Versioned, reusable, governed | Less flexible than source-based repos |
| Terraform | Multi-cloud or standardized IaC | Broad ecosystem, reusable modules | Different workflow from native Azure tools |

**Practical checklist for enterprise template organization**

- Define a standard repository structure
- Create reusable modules for common services
- Keep environment-specific values in parameter files
- Enforce naming, tagging, and policy requirements
- Validate templates in CI/CD before deployment
- Use secure secret handling with Key Vault or managed identities
- Version shared modules and publish approved releases
- Document ownership, scope, and rollback guidance

**Common mistakes to avoid**

- Copying entire templates for each environment
- Hardcoding names, regions, or secrets
- Mixing governance and workload logic in one file
- Skipping validation before production deployment
- Allowing teams to bypass approved modules
- Using one giant template for unrelated resources

A disciplined template organization strategy gives enterprise teams consistency without sacrificing flexibility. The best designs are modular, versioned, and governed, with clear boundaries between platform standards and application-specific deployment logic.
