## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Deployment governance and repeatability**

Deployment governance is the set of controls that keeps Azure environments compliant, secure, and consistent. Repeatability is the ability to deploy the same infrastructure and configuration reliably across environments such as dev, test, and prod. In practice, you combine policy, access control, templates, versioning, and automation so that deployments are predictable and auditable.

For AZ-305, focus on the relationship between:
- **Governance**: what is allowed, required, or denied.
- **Repeatability**: how you deploy the same thing the same way every time.
- **Lifecycle control**: how you track, update, and retire resources safely.

**Core governance goals**

A good Azure governance model reduces drift, limits risk, and makes deployments easier to review. You usually want to standardize:
- Naming conventions
- Tagging
- Region selection
- Allowed SKUs and services
- Network boundaries
- Identity and access patterns
- Resource lifecycle and cleanup

Governance should support delivery, not block it. The best designs make the secure path the easiest path.

**Primary Azure governance tools**

| Tool | Main purpose | Typical scope | Best use |
|---|---|---:|---|
| **Management groups** | Organize subscriptions for consistent control | Tenant hierarchy | Apply policy and RBAC across many subscriptions |
| **Azure Policy** | Enforce rules and evaluate compliance | Management group, subscription, resource group, resource | Require tags, restrict locations, audit settings |
| **RBAC** | Control who can do what | Management group, subscription, resource group, resource | Separate duties and limit deployment rights |
| **Resource locks** | Prevent accidental delete or change | Resource group, resource | Protect critical resources from accidental operations |
| **ARM templates / Bicep** | Declarative infrastructure as code | Deployment scope | Repeatable, versioned deployments |
| **Template Specs** | Store and share versioned templates | Azure resource | Reuse approved templates centrally |
| **Deployment Stacks** | Track and manage deployed resources as a unit | Deployment scope | Safer lifecycle management and cleanup |
| **CI/CD pipelines** | Automate validation and deployment | Repo and pipeline scope | Consistent promotion across environments |

**Management groups and hierarchy design**

Management groups let you apply governance above the subscription level. This is useful when you have many subscriptions or separate landing zones for different business units.

A common hierarchy looks like this:
- Tenant root group
  - Platform
    - Identity
    - Connectivity
    - Management
  - LandingZones
    - Corp
    - Online
  - Sandbox

This structure helps you:
- Apply baseline policy at the right level
- Delegate subscription ownership without losing central control
- Separate platform services from application workloads

Use management groups when you need consistent governance across multiple subscriptions. Use subscriptions when you need billing, quota, or administrative boundaries.

**Azure Policy fundamentals**

Azure Policy evaluates Azure resources against rules. It can:
- **Audit** noncompliant resources
- **Deny** noncompliant requests
- **Append** or **modify** request properties
- **DeployIfNotExists** related resources or configurations when a matching resource is missing

> Azure Policy does not generally deploy arbitrary missing configurations. Only the `DeployIfNotExists` effect deploys a related resource, and it does so based on a defined condition and deployment template.

Policy is ideal for enforcing standards such as:
- Allowed regions
- Required tags
- Approved VM sizes
- HTTPS-only settings
- Diagnostic settings
- Private endpoint requirements
- Storage account security settings

A policy definition describes the rule. An initiative, also called a policy set definition, groups multiple policies into one assignment. This is useful when you want a baseline such as “production security controls” or “platform landing zone controls.”

**Common policy effects**

- **Audit**: reports noncompliance without blocking deployment
- **Deny**: blocks noncompliant requests
- **Append**: adds a field if it is missing
- **Modify**: changes request properties before creation or update
- **DeployIfNotExists**: deploys a related resource when the condition is met and the target resource is missing

Use `Audit` first when introducing a new control. Move to `Deny` only after you understand the impact on existing workloads.

**Example policy assignment**

A policy assignment should reference the correct scope and the correct policy or initiative ID. In Azure CLI, the assignment target must match the resource hierarchy you intend to govern.

```bash
az policy assignment create \
  --name "require-tags-prod" \
  --scope "/subscriptions/00000000-0000-0000-0000-000000000000/resourceGroups/prod-rg" \
  --policy "/providers/Microsoft.Authorization/policyDefinitions/require-tags" \
  --params '{
    "tagName": { "value": "Environment" },
    "tagValue": { "value": "Prod" }
  }'
```

For an initiative at management group scope, use the initiative ID that exists at that scope:

```bash
az policy assignment create \
  --name "platform-baseline" \
  --scope "/providers/Microsoft.Management/managementGroups/contoso-platform" \
  --policy-set-definition "/providers/Microsoft.Management/managementGroups/contoso-platform/providers/Microsoft.Authorization/policySetDefinitions/platform-baseline"
```

**RBAC and least privilege**

Role-Based Access Control (RBAC) determines who can create, modify, or delete resources. It is separate from Azure Policy:
- **RBAC** answers “who can do it?”
- **Policy** answers “is it allowed?”

Use built-in roles where possible:
- **Owner**: full access including role assignment
- **Contributor**: manage resources but not access control
- **Reader**: view-only access
- **User Access Administrator**: manage role assignments
- Service-specific roles such as **Virtual Machine Contributor** or **Network Contributor**

Design principles:
- Grant the minimum role needed
- Scope access as narrowly as practical
- Use groups instead of individual users
- Separate platform operators from application deployers
- Use privileged access workflows for elevated tasks when available

A common pattern is:
- Platform team gets subscription or management group-level control
- App team gets resource group-level Contributor access
- Security team gets Reader plus policy and compliance visibility
- CI/CD service principal or managed identity gets only deployment permissions

**Resource locks**

Resource locks help prevent accidental deletion or modification. They are useful for critical shared resources, but they are not a blanket requirement for all production resources.

Two lock types matter:
- **CanNotDelete**: allows reads and updates but blocks deletion
- **ReadOnly**: blocks changes and deletions

Use locks carefully because they can interfere with automation, scaling, patching, and incident response. Apply them to:
- Shared networking components
- Key vaults or critical platform resources when justified
- Resources that should not be removed casually

Avoid overusing locks on resources that need frequent change. Governance should not create operational dead ends.

**Infrastructure as code for repeatability**

Infrastructure as code (IaC) is the foundation of repeatable deployment. In Azure, the main declarative options are:
- **ARM templates**
- **Bicep**
- Terraform, when your organization standardizes on it

Bicep is a domain-specific language that compiles to ARM JSON. It is often easier to read and maintain than raw JSON templates.

Benefits of IaC:
- Version control
- Peer review
- Reproducible environments
- Parameterized deployments
- Faster recovery and rebuilds
- Easier drift detection

A simple Bicep example:

```bicep
param location string = resourceGroup().location
param storageAccountName string

resource stg 'Microsoft.Storage/storageAccounts@2023-01-01' = {
  name: storageAccountName
  location: location
  sku: {
    name: 'Standard_LRS'
  }
  kind: 'StorageV2'
  properties: {
    allowBlobPublicAccess: false
    minimumTlsVersion: 'TLS1_2'
  }
}
```

Deploy it with Azure CLI:

```bash
az deployment group create \
  --resource-group rg-app-prod \
  --template-file main.bicep \
  --parameters storageAccountName=stgprod001
```

**Parameters, variables, and environment promotion**

Repeatability depends on separating code from environment-specific values. Use:
- **Parameters** for values that change per environment
- **Variables** for reusable expressions inside the template
- **Parameter files** for environment-specific settings

A practical pattern is:
- `main.bicep` contains the resource definitions
- `dev.bicepparam` contains dev values
- `prod.bicepparam` contains prod values

Example parameter file:

```bicep
using './main.bicep'

param storageAccountName = 'stgprod001'
```

This lets you promote the same template through environments while changing only approved inputs.

**Template Specs**

Template Specs store versioned templates in Azure as a resource. They are useful when you want a central, approved template catalog that teams can reuse.

Use Template Specs when:
- You want to publish a standard template for multiple teams
- You need versioning and controlled reuse
- You want to avoid copy-pasting template files across repositories

A safe workflow is:
1. Build or validate the template locally.
2. Publish the template as a Template Spec.
3. Reference the Template Spec in deployments.

If you use Bicep, remember that CLI support can vary by version and workflow. When in doubt, compile Bicep to JSON first for maximum compatibility.

```bash
az bicep build --file main.bicep --outfile main.json
az ts create \
  --name app-baseline \
  --version 1.0.0 \
  --resource-group rg-shared-platform \
  --location eastus \
  --template-file main.json
```

**Deployment Stacks**

Deployment Stacks are an Azure feature for managing a set of resources as a unit. They help with lifecycle tracking and cleanup by associating resources with the stack that created them.

Use them when you need:
- Better control over resource ownership
- Safer updates to a group of resources
- Cleanup of resources that are no longer part of the desired state

Do not treat Deployment Stacks as a universal replacement for all lifecycle management. They are a specific feature and may not fit every organization or deployment model.

**CI/CD and release governance**

Continuous integration and continuous delivery/deployment pipelines enforce repeatability by automating validation and rollout. A good pipeline usually includes:
- Linting and static validation
- What-if or preview checks
- Policy compliance checks
- Approval gates for production
- Deployment to lower environments first
- Rollback or redeploy strategy

Example GitHub Actions workflow using federated identity with Azure login:

```yaml
name: deploy-infra

on:
  push:
    branches: [ main ]

jobs:
  deploy:
    runs-on: ubuntu-latest
    permissions:
      id-token: write
      contents: read
    steps:
      - uses: actions/checkout@v4

      - name: Azure login
        uses: azure/login@v2
        with:
          client-id: ${{ secrets.AZURE_CLIENT_ID }}
          tenant-id: ${{ secrets.AZURE_TENANT_ID }}
          subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}

      - name: Deploy Bicep
        run: |
          az deployment group create \
            --resource-group rg-app-dev \
            --template-file main.bicep \
            --parameters storageAccountName=stgdev001
```

This pattern assumes you have configured federated credentials for the app registration or managed identity. Without that setup, the login step will not authenticate successfully.

**What-if and validation**

Before deploying, validate the template and preview the changes. This reduces surprises and supports change control.

Useful commands:
```bash
az bicep build --file main.bicep
az deployment group validate \
  --resource-group rg-app-dev \
  --template-file main.bicep \
  --parameters storageAccountName=stgdev001

az deployment group what-if \
  --resource-group rg-app-dev \
  --template-file main.bicep \
  --parameters storageAccountName=stgdev001
```

Use `what-if` to see:
- Resources to be created
- Resources to be modified
- Resources to be deleted

This is especially valuable in production where accidental replacement or deletion is costly.

**Blueprints and current guidance**

Azure Blueprints was used to package policy, role assignments, ARM templates, and resource groups into a reusable blueprint. For new deployments, treat Blueprints as retired for most practical purposes and prefer current governance patterns such as:
- Management groups
- Azure Policy initiatives
- RBAC
- IaC
- Template Specs
- Deployment Stacks where appropriate

If you encounter Blueprints in older environments, understand them for maintenance and migration, but do not choose them as the first option for new designs.

**Real-world governance scenarios**

A governance design changes by environment.

**Scenario 1: Dev environment**
- Allow broader contributor access for developers
- Use `Audit` policies first
- Keep locks minimal
- Allow more frequent template changes
- Use lower-cost SKUs and ephemeral resources

**Scenario 2: Test environment**
- Enforce naming and tagging
- Require diagnostic settings
- Restrict regions to match production
- Use approval gates for shared test infrastructure
- Keep templates aligned with production patterns

**Scenario 3: Production environment**
- Use tighter RBAC and separation of duties
- Apply `Deny` for disallowed regions or insecure settings
- Require private networking where needed
- Use locks only for critical shared resources
- Enforce change control and deployment approvals

A practical example:
- Dev team deploys a web app with open experimentation and `Audit` policies.
- Test environment uses the same Bicep template but different parameters and stricter policy.
- Prod uses the same template again, but deployment is gated, policy is enforced, and access is limited to a release pipeline and a small operations group.

This is the core of repeatability: same template, controlled inputs, different governance posture.

**Common exam distinctions**

- **Policy vs RBAC**: policy governs resource compliance; RBAC governs permissions.
- **Template Specs vs Bicep files**: Bicep files are source code in your repo; Template Specs are centrally stored Azure resources for reuse.
- **Locks vs policy**: locks prevent accidental operations; policy enforces standards.
- **Audit vs Deny**: audit reports; deny blocks.
- **DeployIfNotExists vs Modify**: deploys a related resource versus changes the request itself.
- **Management groups vs subscriptions**: management groups organize governance; subscriptions provide administrative and billing boundaries.

**Practical design checklist**

1. Define the management group hierarchy.
2. Assign baseline policies at the highest appropriate scope.
3. Use RBAC groups for human and pipeline access.
4. Standardize on Bicep or another IaC tool.
5. Parameterize environment-specific values.
6. Validate and preview deployments before release.
7. Use locks sparingly and intentionally.
8. Track reusable templates with versioning.
9. Automate promotion through CI/CD.
10. Review compliance continuously.

**Quick recall**

- **Governance** controls what is allowed.
- **Repeatability** ensures the same deployment can be recreated reliably.
- **Policy** enforces standards.
- **RBAC** controls access.
- **IaC** makes deployments consistent.
- **Pipelines** automate promotion and validation.
- **Locks** protect critical resources from accidental change.
- **Management groups** scale governance across subscriptions.
