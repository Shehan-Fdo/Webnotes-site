## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Bicep overview for architects**

Bicep is a domain-specific language (DSL) for deploying Azure resources declaratively. It compiles to Azure Resource Manager (ARM) JSON, but it is easier to read, modularize, and review than raw JSON templates. For architects, the value is not just syntax reduction; it is standardization, repeatability, and safer reuse across environments.

Bicep is especially useful when you need:
- Consistent infrastructure patterns across subscriptions and resource groups
- Parameterized deployments for dev/test/prod
- Reusable building blocks for landing zones, application stacks, and platform services
- Clear dependency management between resources and modules

A practical mental model is:
- **Parameters** define what varies.
- **Resources** define what gets deployed.
- **Outputs** expose what downstream automation needs.
- **Modules** package reusable units.
- **Scopes** control where deployment happens.

**Core Bicep file structure**

A Bicep file typically contains declarations in this order:
- `targetScope`
- `param`
- `var`
- `resource`
- `module`
- `output`

This order is not always mandatory, but it improves readability and helps teams standardize templates.

```bicep
targetScope = 'resourceGroup'

param location string = resourceGroup().location
param appName string
param tags object = {}

var storageName = toLower('${appName}stg')

resource stg 'Microsoft.Storage/storageAccounts@2023-01-01' = {
  name: storageName
  location: location
  tags: tags
  sku: {
    name: 'Standard_LRS'
  }
  kind: 'StorageV2'
}

output storageAccountName string = stg.name
output storageAccountId string = stg.id
```

**Parameters**

Parameters make templates reusable and environment-aware. Use them for values that change between deployments, such as names, locations, SKUs, tags, CIDR ranges, and feature flags.

Common parameter types:
- `string`
- `int`
- `bool`
- `array`
- `object`
- `secureString`
- `secureObject`

Use `secureString` and `secureObject` for secrets such as passwords, connection strings, and certificates. These values are masked in deployment history, but you should still avoid passing secrets unless necessary.

```bicep
param adminUsername string
@secure()
param adminPassword string
param allowedIps array = [
  '203.0.113.10'
  '203.0.113.11'
]
param tags object = {
  workload: 'web'
  env: 'prod'
}
```

Useful parameter decorators:
- `@minLength()`, `@maxLength()`
- `@minValue()`, `@maxValue()`
- `@allowed()`
- `@description()`
- `@metadata()`
- `@secure()`

Example with validation:

```bicep
@description('Deployment environment name')
@allowed([
  'dev'
  'test'
  'prod'
])
param environment string

@minValue(1)
@maxValue(10)
param instanceCount int = 2
```

Validation matters in architecture work because it prevents invalid combinations from reaching Azure. For example, you can restrict a SKU to approved values or ensure a subnet prefix is within an expected range.

**Variables**

Variables are for derived values and repeated expressions. Use them to reduce duplication and make templates easier to maintain.

Typical uses:
- Naming conventions
- Resource IDs
- Computed strings
- Conditional expressions
- Repeated tag objects

```bicep
param appName string
param environment string

var namePrefix = toLower('${appName}-${environment}')
var commonTags = {
  app: appName
  env: environment
}
```

Use variables when the value is derived inside the template. Use parameters when the value should be supplied by the caller.

**Resources**

Resources declare Azure objects to create or update. Each resource declaration includes a type, API version, name, and properties.

```bicep
resource rg 'Microsoft.Resources/resourceGroups@2024-03-01' = {
  name: 'rg-app-prod'
  location: 'eastus'
}
```

Architectural considerations:
- Choose stable naming conventions early.
- Prefer explicit API versions for predictable behavior.
- Use tags for ownership, cost allocation, and lifecycle management.
- Keep resource declarations small and composable when possible.

Bicep supports symbolic names, which make dependencies easier to read:
- `stg.id`
- `stg.name`
- `stg.properties`

**Modules**

Modules let you split large deployments into reusable units. A module can represent a network, identity, storage, application tier, or policy assignment pattern.

Use modules when:
- A deployment is too large for one file
- You want to reuse a pattern across teams or subscriptions
- You need a clean boundary between platform and workload concerns
- You want to version infrastructure components independently

Example module call:

```bicep
module storage './modules/storage-account.bicep' = {
  name: 'storageDeployment'
  params: {
    location: location
    storageName: storageName
  }
}
```

A module is still deployed through ARM, but Bicep gives you a cleaner authoring experience and better composition.

**Module design guidance**

Good modules are:
- Small enough to understand quickly
- Focused on one responsibility
- Parameterized for environment differences
- Opinionated about safe defaults
- Versioned when shared across teams

Avoid modules that:
- Do too many unrelated things
- Hard-code environment-specific values
- Expose every internal detail as a parameter
- Hide critical dependencies or naming rules

**Reusable templates vs modules**

Bicep modules and reusable templates solve related but different problems.

| Mechanism | Best for | Strengths | Tradeoffs |
|---|---|---|---|
| Local module file | Team reuse within a repo | Simple authoring, easy refactoring | Versioning is repo-based |
| Bicep registry module | Shared module distribution | Centralized versioning, discoverability | Requires registry setup and governance |
| Template spec | Versioned ARM template distribution | Azure-native artifact, RBAC-controlled access | Less ergonomic than Bicep modules for authoring |
| Linked template pattern | Legacy or cross-file ARM reuse | Familiar in older ARM estates | More complex than Bicep modules |

Use a **Bicep registry** when you want to publish reusable modules as versioned artifacts for multiple teams. Use a **template spec** when you need an Azure resource that stores a versioned template and can be deployed with controlled access. They are not identical, and the choice depends on whether you are standardizing module distribution or template distribution.

Example registry-style reference syntax varies by registry and setup, so validate your organization’s naming and publishing conventions before using it in production.

**Parameters passed to modules**

Modules accept parameters just like top-level templates. This is how you keep modules generic and environment-specific at the same time.

```bicep
module network './modules/network.bicep' = {
  name: 'networkDeployment'
  params: {
    vnetName: 'vnet-prod'
    addressPrefix: '10.10.0.0/16'
    subnetPrefix: '10.10.1.0/24'
  }
}
```

Common mistakes:
- Passing a string where an array is expected
- Forgetting to update a module parameter after renaming a variable
- Using a parameter default that conflicts with a decorator constraint
- Passing a secret as plain text instead of `@secure()`

**Outputs**

Outputs expose values from a deployment for downstream use. They are useful for chaining deployments, feeding CI/CD pipelines, and returning resource identifiers.

Good output candidates:
- Resource IDs
- Names
- FQDNs (fully qualified domain names)
- Connection strings only when absolutely necessary and handled securely
- Managed identity principal IDs

```bicep
output storageAccountName string = stg.name
output storageAccountId string = stg.id
```

Be careful with outputs:
- Do not output secrets unless there is a strong reason and secure handling is in place
- Prefer IDs and names over deeply nested properties when the property shape may vary by API version
- Use outputs to support automation, not to expose unnecessary internal details

A common troubleshooting pattern is a deployment that succeeds, but the pipeline fails because it expects an output name that was renamed. Keep output contracts stable.

**Scopes**

Scope determines where a deployment is applied. Bicep supports multiple deployment scopes:
- Resource group
- Subscription
- Management group
- Tenant

Set the file scope with `targetScope`:

```bicep
targetScope = 'subscription'
```

Then deploy resources appropriate to that scope. Some resource types are only valid at certain scopes.

Scope examples:
- Resource group scope for application resources
- Subscription scope for creating resource groups, role assignments, and policy assignments
- Management group scope for governance artifacts
- Tenant scope for tenant-level operations such as some policy or identity-related deployments

**Cross-scope deployment**

A single Bicep file can deploy resources at different scopes by using explicit `scope:` on resource or module declarations, as long as the target resource type supports that scope.

```bicep
targetScope = 'subscription'

param rgName string
param rgLocation string = 'eastus'

resource appRg 'Microsoft.Resources/resourceGroups@2024-03-01' = {
  name: rgName
  location: rgLocation
}

module app './modules/app.bicep' = {
  name: 'appModule'
  scope: resourceGroup(rgName)
  params: {
    location: rgLocation
  }
}
```

Important nuance:
- The target resource group must exist when the module scope is resolved.
- If you create the resource group in the same deployment, the deployment order must allow the scope to be resolved correctly.
- `dependsOn` helps with ordering, but scope resolution and resource-type support still matter.

Common scope mismatch errors:
- Deploying a resource-group-scoped resource from a subscription-scoped file without setting the correct `scope`
- Trying to create a resource type at a scope it does not support
- Referencing a resource group name that does not exist yet when the module needs it
- Using a management-group deployment for a resource that only exists at resource-group scope

**Scope selection guidance**

| Scope | Typical use cases | Common examples |
|---|---|---|
| Resource group | Workload resources | VMs, storage accounts, App Service, VNets |
| Subscription | Governance and platform setup | Resource groups, role assignments, policy assignments |
| Management group | Enterprise governance | Policy definitions, initiatives, access control patterns |
| Tenant | Directory-wide operations | Tenant-level governance and some identity-related artifacts |

Choose the narrowest scope that satisfies the deployment. This reduces blast radius and makes permissions easier to reason about.

**Dependencies**

Bicep infers many dependencies automatically when one resource references another. You usually do not need to write `dependsOn` unless the dependency is not obvious from the expression graph.

```bicep
resource plan 'Microsoft.Web/serverfarms@2023-12-01' = {
  name: 'asp-prod'
  location: location
  sku: {
    name: 'P1v3'
  }
}

resource app 'Microsoft.Web/sites@2023-12-01' = {
  name: 'app-prod'
  location: location
  properties: {
    serverFarmId: plan.id
  }
}
```

Use `dependsOn` sparingly:
- When a resource must wait for another even without a direct property reference
- When a module or nested deployment has an ordering requirement
- When you need to make intent explicit for maintainers

**CLI workflow**

Use Azure CLI to validate and deploy Bicep files.

```bash
az bicep version
az bicep build --file main.bicep
az deployment group validate \
  --resource-group rg-app-prod \
  --template-file main.bicep \
  --parameters appName=web01 environment=prod
az deployment group create \
  --resource-group rg-app-prod \
  --template-file main.bicep \
  --parameters appName=web01 environment=prod
```

For subscription scope:

```bash
az deployment sub validate \
  --location eastus \
  --template-file main.bicep \
  --parameters rgName=rg-app-prod rgLocation=eastus
```

Useful commands:
- `az bicep build` to compile Bicep to ARM JSON
- `az deployment group validate` to catch template issues before deployment
- `az deployment group what-if` to preview changes
- `az deployment sub create` for subscription-level deployments

**Troubleshooting and failure scenarios**

Common issues in real deployments:
- Parameter type mismatch: a string is passed where an array or object is expected
- Invalid location: the chosen region does not support the resource SKU
- Name collision: globally unique names such as storage accounts already exist
- Scope error: the deployment target does not match the resource type
- Missing permissions: the identity running the deployment lacks `Owner`, `Contributor`, or a more specific role
- Output mismatch: downstream pipeline expects an output that no longer exists

A practical troubleshooting sequence:
1. Run `az bicep build` to catch syntax issues.
2. Run `validate` or `what-if` to inspect deployment behavior.
3. Check scope and permissions for the deployment identity.
4. Confirm resource naming and regional availability.
5. Review module parameter values and defaults.

> If a module deployment fails with a scope-related error, verify both the `targetScope` of the parent file and the `scope:` assigned to the module. A valid module path does not guarantee a valid deployment scope.

**Architectural patterns**

Bicep fits well into layered architecture:
- **Foundation layer** for resource groups, policy, RBAC (role-based access control), and networking
- **Platform layer** for shared services such as monitoring, identity integration, and key management
- **Workload layer** for application-specific resources

This separation helps you:
- Delegate ownership cleanly
- Reuse platform modules across workloads
- Apply different deployment permissions by layer
- Reduce accidental coupling between teams

Example pattern:
- Subscription-scoped file creates resource groups and assigns policies
- Resource-group-scoped workload file deploys app resources
- Shared modules publish standard network and identity components

**When to use Bicep in AZ-305 scenarios**

Use Bicep when the scenario asks for:
- Repeatable infrastructure deployment
- Standardized environment provisioning
- Modular design for multiple applications
- Governance-friendly infrastructure as code
- Clear separation of concerns between platform and workload

Bicep is especially relevant when you need to explain:
- How to structure reusable templates
- How to parameterize deployments for multiple environments
- How to manage scope across subscriptions and resource groups
- How to expose outputs for automation and integration

**Exam-focused reminders**

- **Parameters** are for external inputs; variables are for internal calculations.
- **Outputs** should return stable, useful deployment data.
- **Modules** improve reuse, maintainability, and team boundaries.
- **Scopes** must match both the deployment target and the resource type.
- **Template specs** and **Bicep registries** are related but not interchangeable.
- Cross-scope deployment is possible, but the target scope must be valid and resolvable.

**Quick reference**

```bicep
targetScope = 'resourceGroup'

param location string = resourceGroup().location
param namePrefix string

var storageName = toLower('${namePrefix}stg')

resource stg 'Microsoft.Storage/storageAccounts@2023-01-01' = {
  name: storageName
  location: location
  sku: {
    name: 'Standard_LRS'
  }
  kind: 'StorageV2'
}

output storageId string = stg.id
```

This pattern shows the core architecture workflow:
- Accept inputs with parameters
- Derive internal values with variables
- Deploy resources declaratively
- Return outputs for downstream automation
