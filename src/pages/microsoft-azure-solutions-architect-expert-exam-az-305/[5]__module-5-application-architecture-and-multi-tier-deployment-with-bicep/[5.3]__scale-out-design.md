## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Scale-out design fundamentals**

Scale-out increases capacity by adding more instances, partitions, or replicas instead of making a single node larger. In Azure architecture, you use scale-out to improve throughput, absorb spikes, and reduce the blast radius of failures. The key design question is not only *how* to scale, but *what* to scale: compute, messaging, storage, or data.

A good scale-out design separates stateless work from stateful data. Stateless tiers are easier to replicate horizontally, while stateful tiers usually need partitioning, sharding, caching, or read replicas to avoid becoming the bottleneck.

- **Horizontal scaling** adds more instances of the same component.
- **Vertical scaling** increases the size of one instance; it is simpler but has hard limits.
- **Elastic scaling** adjusts capacity automatically based on demand.
- **Partitioning** splits data or workload into independent slices.
- **Replication** copies data to improve read scale, availability, or locality.

**When to scale out vs scale up**

Use scale-out when demand is variable, throughput is the main constraint, or you need high availability through redundancy. Use scale-up when the workload is tightly coupled to a single node, the service has no horizontal option, or the operational complexity of distributed design is not justified.

| Choice | Best for | Tradeoff |
|---|---|---|
| Scale up | Simple workloads, short-term relief | Hard limits, larger failure domain |
| Scale out | Web apps, APIs, workers, event processing | More coordination, state management |
| Partition | Large datasets, high write throughput | Routing complexity, cross-partition queries |
| Replication | Read-heavy workloads, DR, geo-read access | Replication lag, conflict handling |

> <mark>Exam tip:</mark> If the question asks how to handle unpredictable spikes with minimal manual intervention, prefer autoscaling and stateless scale-out before redesigning the database.

**Azure App Service and web tier scale-out**

App Service supports instance-based scale-out for web apps and APIs. You can scale manually, schedule rules, or use autoscale with metrics such as CPU percentage, memory, HTTP queue length, or custom metrics. App Service is a common answer when the application is already stateless or can externalize session state.

For example, you can configure autoscale with Azure CLI:

```bash
az monitor autoscale create \
  --resource-group rg-app \
  --resource /subscriptions/<subId>/resourceGroups/rg-app/providers/Microsoft.Web/serverfarms/appservice-plan \
  --resource-type Microsoft.Web/serverfarms \
  --name appservice-autoscale \
  --min-count 2 --max-count 10 --count 2
```

A typical rule-based profile uses CPU and request queue depth:

```json
{
  "profiles": [
    {
      "name": "default",
      "capacity": { "minimum": "2", "maximum": "10", "default": "2" },
      "rules": [
        {
          "metricTrigger": {
            "metricName": "CpuPercentage",
            "operator": "GreaterThan",
            "threshold": 70,
            "timeGrain": "PT1M",
            "timeWindow": "PT5M",
            "statistic": "Average"
          },
          "scaleAction": { "direction": "Increase", "type": "ChangeCount", "value": "1", "cooldown": "PT5M" }
        }
      ]
    }
  ]
}
```

- **Stateless design** is essential for safe web tier scale-out.
- **Session state** should move to Redis Cache, SQL, or another shared store if sticky sessions are not sufficient.
- **Custom metrics** are useful when CPU does not reflect user experience, such as queue depth or request latency.
- **Scale limits** depend on the App Service plan, region capacity, and SKU.

**Azure Functions scale-out**

Azure Functions is designed for event-driven scale-out. In the Consumption plan, the platform automatically adds instances based on trigger load, event backlog, and execution demand. This is a strong fit for bursty workloads, background processing, and integration tasks where you do not want to manage servers.

Important scale-out behaviors include:

- **Consumption plan** scales out automatically and can scale to zero when idle.
- **Event-driven triggers** such as Queue Storage, Service Bus, Event Hubs, and HTTP can drive scaling.
- **Cold start** can affect latency, especially for infrequent invocations.
- **Execution limits** and plan constraints matter for long-running or memory-intensive jobs.
- **Premium plan** reduces cold start and supports more predictable scaling for latency-sensitive workloads.

For AZ-305 scenarios, choose Functions when the workload is naturally event-based and you want automatic scale-out without managing worker pools. Avoid it when you need very long-running processes, strict always-on latency, or heavy local state.

- **Queue-triggered functions** are ideal for burst smoothing.
- **Service Bus triggers** support reliable message processing with competing consumers.
- **Event Hubs triggers** work well for high-throughput telemetry ingestion.
- **HTTP-triggered functions** can scale, but upstream throttling and downstream dependencies still limit end-to-end throughput.

**AKS scale-out design**

Azure Kubernetes Service (AKS) scales at two levels: pods and nodes. You use the Horizontal Pod Autoscaler (HPA) to add or remove pods based on metrics, and the Cluster Autoscaler to add or remove nodes when the scheduler cannot place pods.

Choose pod scaling when the application is stateless and the bottleneck is application throughput. Choose node scaling when the cluster lacks compute capacity for the desired pod count. In practice, you often need both.

- **HPA** reacts to CPU, memory, or custom metrics.
- **Cluster Autoscaler** reacts to unschedulable pods and node pool pressure.
- **Node pools** let you isolate workloads with different hardware, taints, or scaling profiles.
- **DaemonSets** and system pods consume node resources, so node scaling is not equal to pod scaling.

Example HPA manifest:

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api
  minReplicas: 3
  maxReplicas: 20
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
```

Example deployment with resource requests, which HPA and scheduling both depend on:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api
spec:
  replicas: 3
  selector:
    matchLabels:
      app: api
  template:
    metadata:
      labels:
        app: api
    spec:
      containers:
        - name: api
          image: myregistry.azurecr.io/api:v1
          resources:
            requests:
              cpu: "250m"
              memory: "512Mi"
            limits:
              cpu: "1"
              memory: "1Gi"
```

- **Pod scaling** is faster and cheaper when spare node capacity exists.
- **Node scaling** is required when pods cannot be scheduled.
- **Custom metrics** are useful for queue length, request latency, or business KPIs.
- **Scale-out limits** include regional VM capacity, node pool quotas, and IP address exhaustion in the cluster network design.

> <mark>Exam tip:</mark> If the question asks whether to scale pods or nodes, first ask whether the scheduler can place more pods. If not, you need node scale-out.

**VM Scale Sets and custom autoscale**

Virtual Machine Scale Sets (VMSS) are the classic Azure compute scale-out primitive for IaaS workloads. They are useful when you need full OS control, legacy software, or custom agents. VMSS supports autoscale rules based on platform metrics and custom metrics.

A common pattern is to scale a worker farm behind a load balancer or to run batch processors that pull from a queue. You can scale on CPU, memory via guest metrics, queue depth, or custom telemetry.

Example autoscale rule concept for a VMSS worker farm:

```bash
az monitor autoscale create \
  --resource-group rg-workers \
  --resource /subscriptions/<subId>/resourceGroups/rg-workers/providers/Microsoft.Compute/virtualMachineScaleSets/workers \
  --resource-type Microsoft.Compute/virtualMachineScaleSets \
  --name workers-autoscale \
  --min-count 2 --max-count 20 --count 2
```

- **VMSS Flexible orchestration** is often preferred for mixed or more complex topologies.
- **Backend pool limits** on load balancers and application gateways can become a bottleneck.
- **Image boot time** affects scale-out responsiveness.
- **Stateful workloads** on VMSS are harder to scale safely than stateless workers.

**Messaging scale-out patterns**

Messaging decouples producers from consumers and is one of the most important scale-out tools in Azure. It lets you absorb bursts, smooth load, and scale processors independently from front-end traffic.

**Service Bus scale-out**

Azure Service Bus is suited to reliable enterprise messaging, command processing, and workflows that need ordered or transactional handling. The main scale-out pattern is the **competing consumers** model, where multiple workers read from the same queue or subscription and process messages in parallel.

Key design points:

- **Competing consumers** increase throughput by adding more processors.
- **Sessions** preserve ordered processing for related messages, but they reduce parallelism because a session is locked to one consumer at a time.
- **Partitioning** improves throughput and availability by distributing messages across partitions, but ordering is only guaranteed within a partition, not across the whole entity.
- **Prefetch** can improve throughput, but too much prefetch can increase lock loss or uneven load.
- **Message lock duration** and **auto-renewal** matter when processing time varies.

Safe scaling guidance:

- Increase consumer count gradually to avoid overwhelming downstream systems.
- Use idempotent handlers because retries and duplicate deliveries can happen.
- Tune `MaxConcurrentCalls` and message lock settings to match processing time.
- Use dead-letter queues for poison messages that repeatedly fail.

Example .NET worker configuration:

```csharp
var client = new ServiceBusClient(connectionString);
var processor = client.CreateProcessor("orders", new ServiceBusProcessorOptions
{
    MaxConcurrentCalls = 8,
    AutoCompleteMessages = false,
    PrefetchCount = 32
});

processor.ProcessMessageAsync += async args =>
{
    // process idempotently
    await HandleOrderAsync(args.Message);
    await args.CompleteMessageAsync(args.Message);
};

processor.ProcessErrorAsync += ErrorHandlerAsync;
await processor.StartProcessingAsync();
```

- **Sessions** are best when message affinity or ordering is required.
- **Partitioning** is best when throughput matters more than strict global ordering.
- **Dead-lettering** protects the system from poison messages.
- **Scale limits** include namespace tier, messaging unit capacity, and downstream database throughput.

**Event Hubs scale-out**

Event Hubs is designed for high-throughput telemetry and streaming ingestion. Its scale-out model depends on **partitions**, **throughput units** or **capacity units**, and **consumer groups**.

Important concepts:

- **Partitions** are the unit of parallelism for producers and consumers.
- **Consumer groups** let multiple independent applications read the same stream without interfering with each other.
- **Partition assignment** determines which consumer instance reads which partition.
- **Checkpointing** tracks progress so consumers can resume after failure.
- **Throughput units** or **capacity units** define service-side ingestion and egress capacity, depending on the tier.

Design implications:

- More partitions can increase parallelism, but partition count is fixed or constrained by the namespace and should be chosen carefully.
- A consumer group should usually map to one logical application or processing pipeline.
- If you scale consumers beyond the number of partitions, some consumers will sit idle.
- If you have too few partitions, one hot partition can limit throughput.

Example consumer pattern:

```csharp
var client = new EventHubConsumerClient(
    EventHubConsumerClient.DefaultConsumerGroupName,
    connectionString,
    eventHubName);

await foreach (PartitionEvent partitionEvent in client.ReadEventsAsync())
{
    // process event
    await CheckpointAsync(partitionEvent);
}
```

- **Partition key selection** affects load distribution and ordering.
- **Hot partitions** are a common scale-out failure mode.
- **Consumer lag** is the main signal that you need more processing capacity or more partitions in a new design.
- **Scale-out limits** include partition count, ingress quotas, and downstream sink throughput.

**Storage Queue scale-out**

Azure Queue Storage is a simple, low-cost queue for worker farms and burst smoothing. It scales well for basic asynchronous processing, but it has fewer features than Service Bus.

Use the **competing consumers** pattern with multiple workers polling the queue. Each worker retrieves messages, processes them, and deletes them on success. If a worker fails, the message becomes visible again after the visibility timeout expires.

Key considerations:

- **Visibility timeout** must be long enough for normal processing, but not so long that failed work is delayed unnecessarily.
- **Poison messages** should be tracked after repeated dequeue attempts and moved aside for manual inspection or a dead-letter-like process you implement yourself.
- **Idempotency** is essential because messages can be processed more than once.
- **Backoff** helps avoid hot polling when the queue is empty or downstream systems are overloaded.

Example worker loop:

```python
from azure.storage.queue import QueueClient
import time

queue = QueueClient.from_connection_string(conn_str, "jobs")

while True:
    messages = queue.receive_messages(messages_per_page=16, visibility_timeout=60)
    for msg in messages:
        try:
            process_job(msg.content)
            queue.delete_message(msg)
        except Exception:
            # let visibility timeout expire for retry
            pass
    time.sleep(1)
```

- **Visibility timeout** is the main control for retry timing.
- **Poison handling** prevents infinite retry loops.
- **Worker farms** scale horizontally, but each worker must be safe to duplicate.
- **Scale limits** are usually downstream compute, storage account throughput, and queue polling efficiency rather than the queue itself.

**Data scale-out: partitioning, replication, and caching**

Data is often the real bottleneck in scale-out designs. You usually choose among partitioning, replication, and caching based on whether the problem is write throughput, read throughput, latency, or availability.

- **Partitioning** splits data across shards or logical partitions to distribute writes and storage.
- **Replication** copies data to improve read scale, resilience, or geo proximity.
- **Caching** reduces repeated reads and offloads hot data from the primary store.

Use partitioning when writes are the bottleneck and the data can be split by tenant, region, customer, or time. Use replication when reads dominate and the data can tolerate lag or read-only copies. Use caching when the same data is read repeatedly and freshness requirements allow short staleness.

| Pattern | Best use | Risk |
|---|---|---|
| Partitioning | High write throughput, large datasets | Cross-partition queries, rebalancing |
| Replication | Read scale, DR, geo-read access | Lag, conflict resolution |
| Caching | Hot reads, latency reduction | Stale data, cache invalidation |

**Azure SQL Database scale-out**

Azure SQL Database supports scale-out through read replicas, geo-replication, and sharding patterns. For read-heavy workloads, read scale-out can offload reporting or query traffic from the primary. For write-heavy or very large datasets, sharding distributes data across multiple databases.

Relevant concepts:

- **Read replicas** help with read-only workloads and reporting.
- **Active geo-replication** supports disaster recovery and read access in secondary regions.
- **Sharding** splits tenants or data ranges across multiple databases.
- **Elastic Database tools** concepts help manage shard maps and routing in multi-database designs.

Practical guidance:

- Use a shard key that minimizes cross-shard transactions.
- Keep shard sizes balanced to avoid hot shards.
- Route requests through a shard map manager or application-level routing layer.
- Expect operational complexity for schema changes, cross-shard queries, and rebalancing.

> <mark>Exam tip:</mark> If the question asks how to remove a database bottleneck for a multi-tenant SaaS app, sharding is often the scale-out answer; replicas help reads, not write throughput.

**Multi-region scale-out and consistency**

Active/active designs improve availability and latency, but they introduce consistency and conflict challenges. When multiple regions can accept writes, you need a clear conflict resolution strategy.

Common approaches:

- **Single-writer per entity** avoids conflicts by routing each logical record to one region.
- **Last write wins** is simple but can lose updates if clocks or arrival order differ.
- **Version vectors or ETags** help detect conflicts before overwriting data.
- **Application-level merge** is best when business rules can reconcile changes safely.
- **Event sourcing** can help by treating writes as immutable events and rebuilding state.

Design considerations:

- Keep write locality aligned with user or tenant affinity when possible.
- Use Front Door or Traffic Manager to route users to the nearest healthy region.
- Replicate data asynchronously unless the business truly requires synchronous multi-region writes.
- Test failover and conflict scenarios explicitly; they are often the hardest part of active/active systems.

**Front Door vs Traffic Manager**

Azure Front Door and Azure Traffic Manager both help with global routing, but they solve different problems.

| Feature | Front Door | Traffic Manager |
|---|---|---|
| Routing layer | Application layer | DNS layer |
| TLS termination | Yes | No |
| Failover speed | Faster, because routing is proxy-based | Slower, because DNS caching affects cutover |
| Path-based routing | Yes | No |
| WAF integration | Yes | No |
| Best for | Web apps, APIs, global HTTP(S) routing | Simple DNS-based failover |

Use Front Door when you need global HTTP(S) load balancing, TLS termination, path-based routing, or web application firewall integration. Use Traffic Manager when you need DNS-based failover for endpoints that are not HTTP-aware or when simple geographic routing is sufficient.

- **Front Door** is better for application-layer control and faster failover.
- **Traffic Manager** is simpler but depends on DNS TTL and client caching.
- **Neither** replaces application-level data consistency design.

**Autoscale design for non-App Service resources**

Autoscale is not limited to web apps. For AZ-305, you should know how to apply it to VMSS, AKS, and custom metric-driven worker systems.

A practical pattern is to scale workers based on queue depth per instance:

```text
desired_instances = ceil(queue_length / target_messages_per_instance)
```

Example logic for a custom metric pipeline:

- Publish queue depth to Azure Monitor.
- Create an autoscale rule on the custom metric.
- Set a cooldown to prevent oscillation.
- Cap maximum instances to protect downstream dependencies.

Common autoscale signals:

- **CPU** for compute-bound workloads.
- **Memory** for memory-bound services.
- **Queue length** for asynchronous worker farms.
- **Request latency** for user-facing services.
- **Custom business metrics** for domain-specific scaling.

**Scale-out limits and tradeoffs**

Every Azure service has practical limits. The exam often tests whether you recognize that scale-out is constrained by more than just instance count.

- **Regional capacity** can block VM or node expansion even when your configuration is correct.
- **Backend pool limits** can constrain load-balanced VMSS or service topologies.
- **Database replica limits** affect read scale and failover topology.
- **Partition count** limits parallelism in Event Hubs and can create hot partitions.
- **Namespace or tier quotas** limit messaging throughput and entity counts.
- **Network limits** such as IP exhaustion, SNAT port pressure, or load balancer rules can become hidden bottlenecks.
- **Downstream dependencies** often fail before the front-end does, so scale-out must be end-to-end.

**Exam-oriented design heuristics**

Use these rules of thumb when choosing a scale-out approach:

- Prefer **stateless compute** first, then move state to shared services.
- Use **queues** to absorb bursts and decouple producers from consumers.
- Use **pods** for application elasticity and **nodes** for cluster capacity.
- Use **sessions** when ordering matters; use **partitioning** when throughput matters more.
- Use **replication** for read scale and availability; use **sharding** for write scale.
- Use **caching** to remove repeated reads before scaling the database.
- Use **Front Door** for application-layer global routing; use **Traffic Manager** for DNS-based failover.
- Use **idempotent processing** everywhere messages or events can be retried.

**Practical architecture examples**

A common web workload might look like this:

```text
Users -> Front Door -> App Service / AKS -> Redis Cache -> Azure SQL (read replicas)
                               |
                               +-> Service Bus -> worker pool -> storage / database
```

A telemetry pipeline might look like this:

```text
Devices -> Event Hubs -> stream processors -> storage / analytics
                      -> consumer group for alerting
                      -> consumer group for archival
```

A batch processing design might look like this:

```text
API -> Storage Queue -> VMSS workers -> Azure SQL shard set
```

These patterns work because they separate request handling from processing, isolate hot paths, and let each tier scale independently.

**Key takeaways**

Scale-out is a system design problem, not just a compute setting. The best Azure solutions combine autoscaling, decoupling, partitioning, and careful data design so that no single tier becomes the bottleneck.

- **Compute** scales best when it is stateless.
- **Messaging** smooths bursts and enables safe worker scale-out.
- **Data** usually needs partitioning, replication, or caching to scale effectively.
- **Global routing** must be chosen based on failover speed and protocol needs.
- **Multi-region writes** require explicit conflict handling, not just replication.
