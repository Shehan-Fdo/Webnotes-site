## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**What ingress and load balancing solve**

Ingress is the controlled entry point for client traffic into your application. Load balancing distributes requests across multiple backends so you improve availability, scale, and fault tolerance. In Azure, the right choice depends on whether you need Layer 4 or Layer 7 behavior, global or regional distribution, public or private exposure, and whether you need TLS termination, path-based routing, or Web Application Firewall (WAF).

A useful exam mindset is to separate the problem into two questions:

- Where does the traffic enter?
- How is the traffic distributed after it enters?

That distinction helps you choose between Azure Load Balancer, Application Gateway, Front Door, Traffic Manager, and DNS-based patterns.

**Core decision model**

Use this quick mental flow when designing ingress:

1. If you need global anycast entry with edge acceleration and HTTP(S) routing, consider **Azure Front Door**.
2. If you need regional Layer 7 routing, TLS termination, and WAF for web apps, consider **Application Gateway**.
3. If you need Layer 4 distribution for TCP or UDP workloads, consider **Azure Load Balancer**.
4. If you need DNS-based failover or latency-based endpoint selection without proxying traffic, consider **Traffic Manager**.
5. If you only need name resolution to direct clients to a service endpoint, consider **Azure DNS** or a custom DNS pattern.

**Azure Load Balancer**

Azure Load Balancer is a Layer 4 load balancer. It distributes TCP and UDP traffic based on five-tuple hashing and does not inspect HTTP headers, cookies, or paths. It is the right tool when you need high throughput, low latency, and simple packet forwarding.

Common use cases include:

- Internal line-of-business services
- Stateful clustered workloads
- SSH, RDP, SQL, and custom TCP/UDP services
- High availability for virtual machines in a backend pool

There are two main deployment scopes:

- **Public Load Balancer** exposes a public frontend IP.
- **Internal Load Balancer** uses a private frontend IP for east-west or internal-only access.

Key concepts:

- **Frontend IP configuration** defines where clients connect.
- **Backend pool** contains the virtual machines, scale sets, or IP-based backends that receive traffic.
- **Health probe** checks backend availability.
- **Load-balancing rule** maps frontend ports to backend ports.
- **Inbound NAT rule** forwards a specific frontend port to a specific backend instance.

A standard rule example looks like this:

```bash
az network lb create \
  --resource-group rg-network \
  --name lb-app \
  --sku Standard \
  --frontend-ip-name fe-public \
  --backend-pool-name be-app \
  --public-ip-address pip-lb

az network lb probe create \
  --resource-group rg-network \
  --lb-name lb-app \
  --name probe-tcp-80 \
  --protocol tcp \
  --port 80

az network lb rule create \
  --resource-group rg-network \
  --lb-name lb-app \
  --name rule-http \
  --protocol tcp \
  --frontend-port 80 \
  --backend-port 80 \
  --frontend-ip-name fe-public \
  --backend-pool-name be-app \
  --probe-name probe-tcp-80
```

Important behavior:

- **Standard SKU** is the modern choice for production and supports zone-redundant frontends and secure defaults.
- **Basic SKU** is legacy and lacks many production features.
- **Floating IP** changes how frontend and backend ports are mapped and is used for specific clustered or HA scenarios, not as a general-purpose shortcut.
- Health probes determine whether a backend is eligible for traffic; a backend can be running but still removed from rotation if the probe fails.

> Standard Load Balancer is the default exam answer for high-performance Layer 4 distribution. If the question mentions HTTP routing, path-based rules, or TLS offload, you usually need a Layer 7 service instead.

**Application Gateway**

Application Gateway is a regional Layer 7 load balancer for HTTP, HTTPS, and WebSocket traffic. It understands application-layer constructs such as host headers, paths, cookies, and TLS certificates. It is commonly used for web applications that need routing logic and WAF protection.

Use it when you need:

- Path-based routing such as `/api` and `/images`
- Host-based routing for multi-site hosting
- TLS termination or end-to-end TLS
- WAF for OWASP-style protection
- Session affinity when sticky behavior is required
- Private or public frontend exposure depending on design

Core building blocks:

- **Frontend IP configuration**: public or private frontend address
- **Frontend port**: listener port such as 80 or 443
- **Listener**: accepts client traffic
- **Backend pool**: target servers or app services
- **HTTP settings**: backend port, protocol, host header behavior, cookie-based affinity, timeout
- **Health probe**: backend health check
- **Routing rule**: connects listener to backend pool
- **URL path map**: routes different URL paths to different backend pools
- **Rewrite rule set**: modifies headers or URLs when needed

A practical architecture is:

- Internet client
- Application Gateway with WAF
- Backend pool of App Service, AKS ingress, or VMs
- Optional private backend connectivity through VNet integration or private endpoints where supported by the backend service

Example CLI pattern for a simple listener and rule workflow:

```bash
az network application-gateway create \
  --resource-group rg-app \
  --name agw-web \
  --location eastus \
  --sku Standard_v2 \
  --capacity 2 \
  --vnet-name vnet-app \
  --subnet appgw-subnet \
  --public-ip-address pip-agw
```

After creation, you typically add listeners, backend pools, HTTP settings, probes, and rules. The exact names must match the resources you create first.

Example of a path-based routing concept:

```bash
az network application-gateway url-path-map create \
  --resource-group rg-app \
  --gateway-name agw-web \
  --name pathmap-web \
  --default-backend-pool be-web \
  --default-http-settings http-web

az network application-gateway url-path-map rule create \
  --resource-group rg-app \
  --gateway-name agw-web \
  --path-map-name pathmap-web \
  --name rule-api \
  --paths /api/* \
  --backend-pool be-api \
  --http-settings http-api
```

Practical cautions:

- You must create the referenced backend pools, HTTP settings, listeners, and certificates before binding rules.
- HTTPS listeners require a certificate upload or Key Vault integration.
- Application Gateway is regional, so it is not a global traffic distribution service.
- WAF protects HTTP/S traffic; it does not inspect arbitrary TCP or UDP flows.

**Azure Front Door**

Azure Front Door is a global Layer 7 entry point that uses Microsoft’s edge network to route HTTP(S) traffic to the best available origin. It is designed for internet-facing applications that need global reach, fast failover, and edge-based acceleration.

Use Front Door when you need:

- Global anycast ingress
- Fast failover across regions
- HTTP(S) routing at the edge
- WAF at the edge
- URL-based routing and host-based routing
- Optional Private Link to supported origins in Premium scenarios

Front Door is often the best fit for:

- Multi-region web apps
- Global SaaS platforms
- Static and dynamic web front ends
- Architectures that need low-latency user entry worldwide

Important concepts:

- **Endpoint**: the public entry hostname
- **Origin group**: a set of backends with health and load-balancing settings
- **Origin**: a backend service such as App Service, Storage static website, or Application Gateway
- **Route**: binds frontend domains and paths to an origin group
- **Rule set**: applies header, redirect, or rewrite logic
- **WAF policy**: protects the edge entry point

A realistic CLI sequence is usually multi-step. You create the profile, endpoint, origin group, origin, and route:

```bash
az afd profile create \
  --resource-group rg-edge \
  --profile-name afd-global \
  --sku Standard_AzureFrontDoor

az afd endpoint create \
  --resource-group rg-edge \
  --profile-name afd-global \
  --endpoint-name app-edge

az afd origin-group create \
  --resource-group rg-edge \
  --profile-name afd-global \
  --origin-group-name og-web

az afd origin create \
  --resource-group rg-edge \
  --profile-name afd-global \
  --origin-group-name og-web \
  --origin-name origin-app \
  --host-name app1.contoso.com \
  --http-port 80 \
  --https-port 443

az afd route create \
  --resource-group rg-edge \
  --profile-name afd-global \
  --endpoint-name app-edge \
  --route-name route-web \
  --origin-group og-web \
  --supported-protocols Http Https \
  --patterns-to-match "/*" \
  --forwarding-protocol MatchRequest
```

Why this matters:

- The route is what actually connects client requests to the origin group.
- Without a route, the endpoint exists but does not serve application traffic.
- Health probes and load-balancing settings are part of the origin group design, not just the origin object.

Front Door Standard and Premium are both modern choices. Premium adds more advanced private connectivity and security capabilities for supported scenarios, while Standard covers many public web delivery needs. Do not assume Premium is required for every WAF or routing scenario.

> Front Door Private Link is not a generic tunnel to any private IP. It works with supported Azure origins and correctly configured Private Link-enabled backends.

**Traffic Manager**

Traffic Manager is a DNS-based traffic routing service. It does not proxy traffic. Instead, it answers DNS queries with the best endpoint based on the selected routing method and endpoint health.

Use Traffic Manager when you need:

- DNS-level failover
- Latency-based endpoint selection
- Geographic routing
- Priority-based active/passive designs
- Weighted distribution across endpoints

Because it works at DNS, it is useful for:

- Multi-region failover where clients can tolerate DNS caching
- Directing users to the nearest region
- Blue/green or canary patterns at the DNS layer

Routing methods:

- **Priority**: active/passive failover
- **Weighted**: split traffic by configured weights
- **Performance**: route to the lowest-latency endpoint
- **Geographic**: route based on client location rules
- **Subnet**: route based on source IP ranges
- **MultiValue**: return multiple healthy endpoints
- **Nested**: combine profiles for complex topologies

Example profile creation:

```bash
az network traffic-manager profile create \
  --resource-group rg-dns \
  --name tm-global \
  --routing-method Performance \
  --unique-dns-name contoso-global \
  --ttl 30 \
  --monitor-protocol HTTPS \
  --monitor-port 443 \
  --monitor-path /health
```

Then add endpoints appropriate to your CLI version and routing method. Endpoint types and parameter names can vary slightly by Azure CLI release, so verify with `az network traffic-manager endpoint create -h` before copying commands into production.

Design notes:

- Traffic Manager cannot terminate TLS.
- Traffic Manager cannot inspect URLs or headers.
- DNS caching means failover is not instantaneous.
- It is often paired with Front Door or Application Gateway rather than replacing them.

**Choosing the right service**

| Requirement | Best fit | Why |
|---|---|---|
| TCP/UDP distribution | Azure Load Balancer | Layer 4, high throughput, low latency |
| HTTP path routing in one region | Application Gateway | Layer 7 routing and WAF |
| Global web ingress | Azure Front Door | Edge-based global HTTP(S) entry |
| DNS failover | Traffic Manager | DNS-level routing without proxying |
| Internal-only service exposure | Internal Load Balancer or private Application Gateway frontend | Private address space access |
| Edge WAF for web apps | Front Door or Application Gateway WAF | HTTP/S inspection and policy enforcement |

A simple decision rule:

- Choose **Load Balancer** for non-HTTP traffic or when you need the simplest packet distribution.
- Choose **Application Gateway** for regional web ingress with routing logic.
- Choose **Front Door** for global web ingress and edge acceleration.
- Choose **Traffic Manager** when DNS-based steering is enough.

**Health probes and availability**

Health probes are critical because they determine whether traffic is sent to a backend. A backend can be reachable at the network layer but still fail application health checks.

Best practices:

- Probe the actual application path when possible.
- Return a fast, deterministic success code such as `200 OK`.
- Avoid heavy dependencies in the health endpoint.
- Keep probe timeouts and intervals aligned with your failover tolerance.
- Make sure the probe protocol matches the backend listener.

Examples:

- For Load Balancer, probe a TCP port or a simple HTTP endpoint if supported by the design.
- For Application Gateway, use an HTTP probe path like `/health`.
- For Front Door, configure origin health probes so unhealthy origins are removed from rotation.
- For Traffic Manager, configure the monitor protocol, port, and path so endpoint health reflects real service availability.

**TLS, certificates, and termination**

TLS termination means the ingress service decrypts client traffic before forwarding it to the backend. This simplifies certificate management and enables routing features that require HTTP visibility.

Common patterns:

- **End-to-end TLS**: client to ingress and ingress to backend both use TLS.
- **TLS termination at ingress**: ingress decrypts, backend receives HTTP.
- **Re-encryption**: ingress terminates and then re-establishes TLS to the backend.

When to use each:

- Use termination at ingress when backend simplicity matters and internal network trust is sufficient.
- Use end-to-end TLS when compliance or zero-trust requirements demand encryption throughout.
- Use re-encryption when you want Layer 7 features at the edge but still need encrypted backend transport.

**Private ingress patterns**

Private ingress means clients inside a network reach services without exposing them publicly. This is common for internal apps, admin portals, and service-to-service traffic.

Patterns include:

- Internal Load Balancer for private TCP/UDP distribution
- Application Gateway with a private frontend IP for private web ingress where supported
- Private Link to supported PaaS origins
- Private DNS to resolve private endpoints correctly

Practical example:

- An internal HR portal sits behind an internal Application Gateway.
- The gateway routes `/api` to an internal API and `/static` to a storage-backed origin.
- Users connect only from the corporate network or through VPN/ExpressRoute.

**Real-world architecture examples**

**1. Global e-commerce web app**

- Front Door provides the global entry point.
- WAF blocks malicious requests at the edge.
- Origin groups point to regional Application Gateways or App Services.
- Each region has its own backend pool and health probe.
- Traffic Manager is not necessary unless you need DNS-level control for a separate use case.

Why this works:

- Users connect to the nearest edge.
- Regional outages fail over quickly.
- You keep application routing flexible without exposing regional origins directly.

**2. Internal line-of-business API**

- Internal Load Balancer fronts a set of API VMs or VM scale sets.
- Health probes remove unhealthy instances.
- Private DNS resolves the service name to the internal frontend IP.
- No public exposure is required.

Why this works:

- The service stays inside the virtual network.
- Layer 4 distribution is enough for API traffic that does not need path routing.

**3. Multi-region active/passive web app**

- Traffic Manager uses Priority routing.
- Primary region hosts Application Gateway and app backends.
- Secondary region stays warm and ready.
- DNS TTL is kept low to reduce failover delay.

Why this works:

- You get simple failover without global proxying.
- This is useful when clients can tolerate DNS caching delays.

**Common exam traps**

- Confusing **Traffic Manager** with a proxy-based load balancer.
- Choosing **Application Gateway** for UDP traffic.
- Choosing **Load Balancer** when the question requires path-based routing or WAF.
- Assuming **Front Door** is only for static websites.
- Treating **DNS failover** as instant.
- Assuming private connectivity means any private IP can be reached through Front Door Private Link.
- Forgetting that health probes and routes must be configured before traffic flows.

**CLI and configuration reminders**

When you work with Azure ingress services, resource names must line up exactly:

- Backend pools must exist before rules reference them.
- HTTP settings must match the backend protocol and port.
- Listeners must reference the correct frontend IP and port.
- Certificates must be uploaded or referenced before HTTPS listeners can work.
- Routes in Front Door must bind the endpoint to an origin group.
- Traffic Manager endpoints must match the CLI version and routing method.

A minimal Application Gateway backend HTTP setting example:

```bash
az network application-gateway http-settings create \
  --resource-group rg-app \
  --gateway-name agw-web \
  --name http-web \
  --port 80 \
  --protocol Http \
  --cookie-based-affinity Disabled \
  --timeout 30
```

A minimal Load Balancer probe example:

```bash
az network lb probe create \
  --resource-group rg-network \
  --lb-name lb-app \
  --name probe-http \
  --protocol Http \
  --port 80 \
  --path /health
```

**Summary for AZ-305**

For architecture questions, map the requirement to the service layer first:

- **Layer 4, TCP/UDP, simple and fast**: Azure Load Balancer
- **Regional Layer 7, WAF, path routing**: Application Gateway
- **Global Layer 7, edge routing, acceleration**: Azure Front Door
- **DNS-based steering and failover**: Traffic Manager

If you remember only one thing, remember this: choose the service based on protocol, scope, and routing intelligence, not just on whether it “balances traffic.”
