## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Overview**

In Azure architecture, the difference between **stateless** and **stateful** components drives decisions about scaling, failover, session handling, and data placement. A stateless component does not depend on local runtime memory or disk to preserve user-specific context between requests. A stateful component retains information across requests, either in memory, on disk, or in an external store that the component depends on for continuity.

For AZ-305, you should be able to choose the right pattern for web apps, APIs, background workers, messaging systems, and data services. The key design question is simple: if a node disappears, can another node continue the work without losing important context?

**Core definitions**

<dl>
<dt><strong>Stateless</strong></dt>
<dd>Each request is independent. Any instance can handle any request because the instance does not need prior request history stored locally.</dd>
<dt><strong>Stateful</strong></dt>
<dd>The component depends on retained context. The same instance, session, or storage location may be required to continue processing correctly.</dd>
<dt><strong>State</strong></dt>
<dd>Information that must persist across requests, such as session data, workflow progress, cache contents, locks, offsets, or transaction context.</dd>
</dl>

**Why this matters in Azure**

Stateless designs usually scale out more easily because Azure can add or remove instances without needing to preserve local context. Stateful designs often need extra coordination, replication, or persistence to avoid data loss and inconsistent behavior.

- **Scale-out** is simpler for stateless services because any instance can serve the next request.
- **Failover** is easier when another instance can take over without session affinity.
- **Maintenance** is safer when instances can be recycled without user impact.
- **Resilience** improves when the system does not depend on a single node’s memory or disk.

**Stateless components**

Stateless components process each request independently. They may still use external storage, but they do not rely on local instance state to complete the next request. Common examples include web front ends, API gateways, and many microservice endpoints.

Typical Azure services and patterns that fit stateless design:

- **Azure App Service** web apps when session data is externalized.
- **Azure Functions** for event-driven workloads, especially when each invocation is independent.
- **Azure Container Apps** or containers behind a load balancer when the app does not store session data locally.
- **Azure API Management** as a request-routing and policy layer.
- **Worker services** that pull discrete jobs from a queue and do not need prior job context on the same node.

Stateless services are a strong fit when you want horizontal scaling and simple recovery. If an instance fails, the load balancer can send the next request to another healthy instance with little or no disruption.

**Characteristics of stateless design**

- Requests are self-contained.
- Any instance can handle any request.
- Local memory is disposable.
- Horizontal scaling is straightforward.
- Instance replacement is low risk.
- Session data is stored externally if needed.

**Practical example: stateless API**

A REST API that reads and writes customer records from Azure SQL Database is stateless if it does not keep user session data in memory between calls.

```bash
# Example: multiple requests can hit different instances
curl https://api.contoso.com/customers/1001
curl https://api.contoso.com/customers/1002
```

If the API needs user preferences or shopping cart data, store that data in a shared system such as Azure Cache for Redis, Azure SQL Database, Cosmos DB, or a distributed session store rather than in the app instance.

**Stateful components**

Stateful components retain context that affects future processing. The state may be local, replicated, or stored in a durable backend, but the component depends on that state being available and consistent.

Common stateful examples include:

- **Databases** such as Azure SQL Database, Azure Database for PostgreSQL, and Azure Cosmos DB.
- **Message brokers** and queues that preserve message order or delivery state.
- **Session-dependent applications** that store user context in memory.
- **Workflow engines** that track long-running process progress.
- **Distributed systems** that maintain leader election, checkpoints, or partition ownership.

Stateful components are not inherently bad. They are essential for persistence, transactions, ordering, and long-running business processes. The design challenge is to isolate state where it belongs and avoid coupling it to ephemeral compute instances.

**Characteristics of stateful design**

- Requests may depend on prior context.
- Instance identity can matter.
- Local storage may contain important data.
- Failover may require replication or recovery.
- Scaling often needs coordination.
- Data consistency becomes a primary concern.

**Practical example: stateful session handling**

If a shopping cart is stored only in server memory, the user may lose the cart when the app scales out or the instance restarts.

```csharp
// Example of stateful in-memory session data
public class CartController : Controller
{
    public IActionResult AddItem(string sku)
    {
        HttpContext.Session.SetString("LastSku", sku);
        return Ok();
    }
}
```

This approach is fragile unless the session is backed by a distributed store. In Azure, prefer external session storage or redesign the app so the client or database holds the durable state.

**Stateless vs. stateful comparison**

| Aspect | Stateless | Stateful |
|---|---|---|
| Request dependency | Independent | Depends on prior context |
| Scaling | Easier to scale out | Often requires coordination |
| Failover | Simpler | More complex |
| Local memory reliance | Low | Often high |
| Recovery after restart | Usually easy | May require state restoration |
| Typical examples | APIs, web front ends, functions | Databases, sessions, workflows |

**Azure design implications**

When you design for Azure, separate compute from state whenever possible. Use stateless compute for request handling and durable services for persistence.

- Put **business data** in durable storage, not in VM memory.
- Put **session state** in a shared cache or database if the app must remember users.
- Put **queue state** in managed messaging services rather than custom in-memory queues.
- Put **workflow checkpoints** in durable orchestration services or storage.
- Use **load balancers** and autoscaling with stateless tiers to improve elasticity.

This separation makes it easier to replace instances, patch systems, and recover from failures without losing important information.

**Common Azure patterns**

**1. Web tier + shared state store**

A web app runs statelessly and stores user session data in Azure Cache for Redis or a database.

```text
Users -> Azure Front Door / Application Gateway -> App Service instances
                                         |
                                         +-> Azure Cache for Redis
                                         +-> Azure SQL Database
```

Use this pattern when you need fast scale-out and predictable failover.

**2. Queue-based worker pattern**

A front-end service accepts requests and places work items on Azure Service Bus or Azure Storage Queues. Worker instances process messages independently.

```text
Client -> API -> Queue -> Worker 1 / Worker 2 / Worker 3
```

This pattern is naturally stateless at the worker layer because each message is a self-contained unit of work.

**3. Durable workflow pattern**

Long-running processes, such as approvals or order fulfillment, keep state in a workflow engine or durable store rather than in memory.

```text
Request -> Orchestrator -> Durable state store -> Activities
```

Use this when a process spans minutes, hours, or days and must survive restarts.

**Session state and affinity**

Session state is a common reason applications become stateful. If a user must return to the same server to continue a session, the system depends on **session affinity** or sticky sessions. That can work, but it reduces flexibility and resilience.

> Avoid sticky sessions unless you truly need them. They can create uneven load distribution and make failover less predictable.

Better options include:

- Store session data in a distributed cache.
- Use tokens such as JWTs (JSON Web Tokens) for client-held identity context.
- Design APIs so each request contains all required context.
- Persist workflow state in a durable backend.

**When to use sticky sessions**

- Legacy applications cannot be refactored quickly.
- Session data is small and short-lived.
- You accept reduced resilience in exchange for compatibility.

**When to avoid sticky sessions**

- You need rapid autoscaling.
- You expect frequent instance recycling.
- You want active-active load balancing across many nodes.
- You need predictable failover behavior.

**Caching is not the same as state**

A cache may hold data temporarily, but cache contents are usually disposable. That means a cache can support a stateless application without making the application itself stateful.

- **Cache as optimization**: acceptable if the system can rebuild the data.
- **Cache as source of truth**: risky unless the cache is durable and replicated.
- **In-memory cache**: fast, but lost on restart.
- **Distributed cache**: better for shared access and resilience.

**Example: Redis-backed session**

```json
{
  "SessionId": "abc123",
  "UserId": "42",
  "CartItems": [
    { "Sku": "P100", "Qty": 2 },
    { "Sku": "P200", "Qty": 1 }
  ]
}
```

Store this in Azure Cache for Redis or a database if multiple app instances must read it.

**Stateful services in Azure**

Some Azure services are inherently stateful because persistence is their purpose.

- **Azure SQL Database** stores relational data with transactions.
- **Azure Cosmos DB** stores distributed NoSQL data with partitioning and replication.
- **Azure Storage** persists blobs, files, queues, and tables.
- **Azure Service Bus** preserves messages and supports durable messaging semantics.
- **Azure Cache for Redis** can be used as a shared state store, though its role is often transient or performance-oriented.

For AZ-305, recognize that stateful services are not a problem to avoid; they are the foundation for durable application design. The goal is to keep compute stateless where practical and let managed services handle persistence.

**Design trade-offs**

Stateless and stateful components each have trade-offs.

- **Stateless advantages**
  - Easier autoscaling
  - Simpler failover
  - Better load balancing
  - Easier blue-green deployments
  - Lower coupling to instance identity

- **Stateless disadvantages**
  - Requires external storage for shared context
  - Can increase network calls to retrieve state
  - May add latency if too much state is externalized

- **Stateful advantages**
  - Efficient access to local context
  - Natural fit for transactions and workflows
  - Useful for ordered processing and durable persistence

- **Stateful disadvantages**
  - Harder to scale horizontally
  - More complex recovery and replication
  - Greater risk of data loss if state is local only
  - More operational complexity

**Real-world architecture guidance**

Use stateless compute when:

- You expect frequent scaling events.
- You want simple instance replacement.
- You can store all durable data externally.
- You are building APIs, web front ends, or event handlers.

Use stateful components when:

- You need durable records, transactions, or ordering.
- You are implementing sessions, workflows, or coordination.
- The service’s purpose is to preserve data.

A good Azure architecture usually combines both: stateless application tiers and stateful managed services.

**Troubleshooting signals**

- Users lose sessions after deployment: session state is probably stored locally.
- Scaling out causes inconsistent behavior: the app likely depends on instance affinity.
- Failover drops in-flight work: the workflow may not be checkpointing state.
- One node becomes overloaded: sticky sessions or local state may be pinning traffic.
- Restarts cause data loss: the component is treating ephemeral memory as durable storage.

**Exam focus**

For AZ-305, expect questions that ask you to choose between:

- local memory versus distributed cache
- sticky sessions versus token-based or shared-state designs
- stateless web tiers versus stateful backends
- queue-based processing versus synchronous in-memory workflows
- managed services versus custom state handling on VMs

The best answer usually favors managed durability for state and statelessness for compute.

**Quick decision guide**

1. Ask whether the component must remember anything between requests.
2. If yes, decide whether that state is durable business data, temporary session data, or transient cache data.
3. Store durable data in a managed stateful service.
4. Keep the compute tier stateless whenever possible.
5. Use affinity only when you cannot remove the state dependency.

**Summary**

Stateless components are easier to scale, replace, and recover because they do not depend on local runtime state. Stateful components preserve context and are essential for persistence, transactions, and workflows, but they add complexity and reduce elasticity. In Azure architecture, the preferred pattern is usually stateless compute with state moved into managed, durable services.
