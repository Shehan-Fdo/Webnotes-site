## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Multi-tier application design in Azure**

A multi-tier web application separates presentation, application, and data responsibilities so you can scale, secure, and operate each layer independently. In Azure, the design usually combines network segmentation, tier-appropriate load balancing, private connectivity, managed identity, and observability.

The main architectural goal is to reduce blast radius. You isolate tiers with virtual networks, subnets, network security groups, and user-defined routes, then expose only the minimum required ingress and egress paths.

**Core reference architecture**

A common Azure pattern uses:

- **Edge delivery** with Azure Front Door for global entry, TLS termination, and web application firewall capabilities at the edge.
- **Web tier** in one subnet or scale set, often behind Application Gateway when you need regional Layer 7 routing.
- **Application tier** in a separate subnet, reachable only from the web tier or internal API consumers.
- **Data tier** in a private subnet or as a managed PaaS service with private endpoints.
- **Shared services** such as Azure Cache for Redis, Key Vault, and monitoring resources.

A practical flow looks like this:

```text
Internet
  -> Azure Front Door
  -> Application Gateway (regional, optional)
  -> Web tier (App Service, AKS ingress, or VM scale set)
  -> App tier (internal API, service layer)
  -> Data tier (Azure SQL, Cosmos DB, Storage)
```

Use Front Door when you need global anycast entry, fast failover, and edge routing across regions. Use Application Gateway when you need regional Layer 7 routing, path-based routing, cookie-based affinity, or integration with a virtual network. Use Azure Load Balancer when you need Layer 4 distribution for non-HTTP traffic or internal east-west traffic.

**Network segmentation and traffic control**

Network segmentation is essential for multi-tier apps. You typically place each tier in its own subnet and control traffic with NSGs and route tables.

- **Virtual network (VNet)** provides the boundary for private IP addressing and routing.
- **Subnet** boundaries separate tiers and simplify policy enforcement.
- **Network Security Group (NSG)** rules allow or deny traffic by source, destination, port, and protocol.
- **User-defined route (UDR)** tables force traffic through firewalls, NVA appliances, or inspection points.

A typical segmentation model:

| Tier | Example subnet | Allowed inbound | Allowed outbound |
|---|---|---|---|
| Edge / ingress | `snet-ingress` | From Front Door or Internet to gateway | To web tier only |
| Web tier | `snet-web` | From gateway or Front Door origin | To app tier, monitoring, Key Vault |
| App tier | `snet-app` | From web tier or internal API gateway | To data tier, cache, identity endpoints |
| Data tier | `snet-data` | From app tier only | Minimal, often restricted to platform dependencies |

Use NSGs to enforce east-west restrictions between tiers. For example, allow the web tier to reach the app tier on `443`, but deny direct access from the internet to the app subnet.

```bash
az network nsg rule create \
  --resource-group rg-app \
  --nsg-name nsg-app \
  --name Allow-Web-To-App-443 \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes 10.10.1.0/24 \
  --source-port-ranges '*' \
  --destination-address-prefixes 10.10.2.0/24 \
  --destination-port-ranges 443
```

Use UDRs when you need centralized inspection or forced tunneling. For example, route all outbound traffic from app and data subnets to Azure Firewall.

```bash
az network route-table route create \
  --resource-group rg-app \
  --route-table-name rt-app \
  --name DefaultToFirewall \
  --address-prefix 0.0.0.0/0 \
  --next-hop-type VirtualAppliance \
  --next-hop-ip-address 10.10.100.4
```

> Avoid flattening all tiers into one subnet. It makes policy harder to reason about and increases the chance of lateral movement if one tier is compromised.

**Load balancing choices between tiers**

Choose the load-balancing service based on protocol, scope, and routing needs.

- **Azure Front Door**
  - Global Layer 7 entry point.
  - Best for multi-region web apps, failover, and edge acceleration.
  - Commonly placed in front of regional ingress.
- **Application Gateway**
  - Regional Layer 7 load balancer.
  - Best for path-based routing, TLS offload, WAF, and private backend pools.
  - Commonly placed inside a VNet before web or API tiers.
- **Azure Load Balancer**
  - Layer 4 load balancer.
  - Best for TCP/UDP traffic, internal service distribution, and non-HTTP workloads.
  - Commonly used between internal tiers or for VM-based services.

A common placement pattern is:

- Front Door at the global edge.
- Application Gateway in each region for HTTP(S) routing into the VNet.
- Internal Load Balancer for app-to-app or app-to-service traffic that should remain private.

For example, if the web tier is an App Service and the app tier is a VM scale set, Front Door can route to Application Gateway, which then forwards to the internal app tier. If the app tier exposes a private API, an internal load balancer or private ingress endpoint keeps it off the public internet.

**Service-to-service communication patterns**

Multi-tier systems often need more than direct HTTP calls.

- **Managed identity** lets a workload authenticate to Azure services without secrets.
- **API Management (APIM)** provides a controlled internal or external API facade, policy enforcement, and versioning.
- **Internal load balancers** expose private services to other tiers without public IPs.
- **Private ingress** keeps internal APIs reachable only from the VNet or peered networks.

Use APIM when you need:
- API versioning and deprecation control.
- Centralized authentication, throttling, and transformation.
- A stable contract between web and app tiers.

Use an internal load balancer when:
- You need simple private TCP/HTTP distribution.
- You do not need API gateway features.
- You want to keep the service reachable only from private networks.

Use private ingress when:
- The service should never be internet-facing.
- You need private endpoints or internal ingress for AKS, App Service Environment, or internal Application Gateway patterns.

A simple internal API pattern:

```text
Web tier -> APIM internal mode -> App tier service
```

Or, for direct private service access:

```text
Web tier -> Internal Load Balancer -> App tier VMs
```

**State management and scaling**

State is a major design concern in scaled-out web tiers. The safest default is to keep web and app tiers stateless.

- **Stateless design** allows any instance to handle any request.
- **Distributed session state** stores session data in a shared external store such as Redis or SQL.
- **Sticky sessions** bind a client to a specific backend instance.

Sticky sessions can simplify legacy apps, but they reduce resilience and make scaling less predictable. Use them only when the application cannot be refactored quickly.

> In-memory session state is an anti-pattern for horizontally scaled tiers because instance restarts, autoscale events, and zone failures can lose user state.

Prefer these patterns:

- Store session data in Azure Cache for Redis when you need low-latency shared session state.
- Store durable user state in a database rather than in process memory.
- Use tokens or claims-based auth so the app does not depend on server affinity.
- Externalize temporary files to Blob Storage or a shared file service when multiple instances need access.

A Redis-backed session example in ASP.NET Core:

```csharp
builder.Services.AddStackExchangeRedisCache(options =>
{
    options.Configuration = "myredis.redis.cache.windows.net:6380,password=***,ssl=True";
    options.InstanceName = "webapp:";
});

builder.Services.AddSession(options =>
{
    options.Cookie.Name = ".MyApp.Session";
    options.IdleTimeout = TimeSpan.FromMinutes(20);
});
```

**Data tier availability and disaster recovery**

The data tier often determines the real resilience of the whole application. Design for both high availability and disaster recovery.

- **Zone-redundant deployments** protect against a datacenter failure within a region.
- **Geo-replication** copies data to another region for disaster recovery.
- **Failover groups** provide managed failover for Azure SQL Database and Azure SQL Managed Instance.
- **Backup and restore** remain essential even when replication exists, because replication does not protect against logical corruption or accidental deletion.

For Azure SQL, common options include:

- Zone redundancy for local resilience.
- Active geo-replication for readable secondary databases.
- Auto-failover groups for managed regional failover.
- Point-in-time restore for recovery from user error.

For Cosmos DB, use:
- Multi-region writes when the application needs active-active writes.
- Read regions for low-latency global reads.
- Automatic failover for regional outage handling.

For storage-backed data, consider:
- GRS or RA-GRS for geo-redundant storage.
- Immutable storage or soft delete where appropriate.
- Regular restore testing, not just backup creation.

A practical rule:
- Use replication for availability.
- Use backups for recovery from corruption, deletion, or bad deployments.

**Example end-to-end architecture**

A realistic exam-style architecture for a multi-tier web app might look like this:

- **Global entry:** Azure Front Door with WAF.
- **Regional ingress:** Application Gateway in each region.
- **Web tier:** App Service or VM scale set in `snet-web`.
- **App tier:** Internal API on AKS or VM scale set in `snet-app`.
- **API facade:** API Management in internal mode for service-to-service calls.
- **Data tier:** Azure SQL Database with failover group and private endpoint.
- **Cache:** Azure Cache for Redis for session and response caching.
- **Secrets:** Key Vault with private endpoint and managed identity access.
- **Monitoring:** Application Insights and Log Analytics.

Traffic flow:

```text
User
 -> Front Door
 -> Application Gateway
 -> Web tier
 -> APIM internal
 -> App tier
 -> Azure SQL / Redis / Storage
```

This design gives you:
- Global failover at the edge.
- Regional isolation and private backends.
- Controlled east-west traffic.
- A clear path for scaling each tier independently.

**Observability and distributed tracing**

Observability should span all tiers so you can trace a request end to end.

- **Application Insights** captures requests, dependencies, exceptions, and custom events.
- **Distributed tracing** correlates activity across web, app, and data tiers.
- **Alerts** notify you when latency, failures, or saturation exceed thresholds.
- **Log Analytics** centralizes logs for querying and correlation.

For .NET applications, enable Application Insights and dependency tracking:

```csharp
builder.Services.AddApplicationInsightsTelemetry(options =>
{
    options.ConnectionString = builder.Configuration["APPLICATIONINSIGHTS_CONNECTION_STRING"];
});
```

For OpenTelemetry-based tracing, export to Azure Monitor:

```csharp
using OpenTelemetry.Trace;

builder.Services.AddOpenTelemetry()
    .WithTracing(tracing =>
    {
        tracing
            .AddAspNetCoreInstrumentation()
            .AddHttpClientInstrumentation()
            .AddSqlClientInstrumentation()
            .AddAzureMonitorTraceExporter();
    });
```

Useful alert examples:
- Request failure rate above a threshold.
- P95 latency above target for the web tier.
- SQL DTU or vCore saturation.
- Redis server load or memory pressure.
- Backend health probe failures on Application Gateway.

A simple alerting strategy:
1. Alert on user-facing symptoms first, such as failed requests and latency.
2. Alert on resource saturation next, such as CPU, memory, and connection limits.
3. Alert on dependency health last, such as database failover or cache unavailability.

**Security considerations**

Security in multi-tier apps depends on layered controls.

- Use **private endpoints** for PaaS services when possible.
- Use **managed identity** instead of stored credentials.
- Store secrets in **Key Vault** and rotate them.
- Restrict inbound access with NSGs, WAF, and private ingress.
- Restrict outbound access with UDRs and firewall policies.

A secure pattern is to deny public network access to the database and expose it only through a private endpoint. The app tier then connects over the VNet, not the public internet.

**Practical Azure CLI examples**

Create a VNet and subnets for separate tiers:

```bash
az network vnet create \
  --resource-group rg-app \
  --name vnet-app \
  --address-prefix 10.10.0.0/16 \
  --subnet-name snet-web \
  --subnet-prefix 10.10.1.0/24

az network vnet subnet create \
  --resource-group rg-app \
  --vnet-name vnet-app \
  --name snet-app \
  --address-prefixes 10.10.2.0/24

az network vnet subnet create \
  --resource-group rg-app \
  --vnet-name vnet-app \
  --name snet-data \
  --address-prefixes 10.10.3.0/24
```

Create a private endpoint for Azure SQL:

```bash
az network private-endpoint create \
  --resource-group rg-app \
  --name pe-sql \
  --vnet-name vnet-app \
  --subnet snet-data \
  --private-connection-resource-id /subscriptions/.../resourceGroups/rg-data/providers/Microsoft.Sql/servers/sqlprod \
  --group-id sqlServer \
  --connection-name pe-sql-conn
```

Enable a managed identity on an app service:

```bash
az webapp identity assign \
  --resource-group rg-app \
  --name web-prod
```

Grant the identity access to Key Vault:

```bash
az keyvault set-policy \
  --name kv-prod \
  --object-id <principal-id> \
  --secret-permissions get list
```

**Design tradeoffs to remember**

- **Front Door vs Application Gateway**
  - Front Door is global and edge-oriented.
  - Application Gateway is regional and VNet-integrated.
- **Load balancer vs API Management**
  - Load Balancer is simple transport distribution.
  - APIM adds policy, auth, versioning, and governance.
- **Sticky sessions vs stateless design**
  - Sticky sessions can help legacy apps.
  - Stateless design scales and fails over better.
- **Replication vs backup**
  - Replication improves availability.
  - Backup protects against logical mistakes and corruption.

**Exam-focused guidance**

When you evaluate a scenario, ask:
- Which tier must be public, and which tiers must remain private?
- Do you need global routing, regional routing, or both?
- Is the workload HTTP-based, TCP-based, or mixed?
- Can the app be stateless, or does it require shared session state?
- What is the recovery objective for the data tier?
- How will you trace a request across tiers during an incident?

A strong AZ-305 answer usually combines:
- Network segmentation with NSGs and UDRs.
- The right load balancer at the right layer.
- Private service-to-service communication.
- Resilient data design with failover and backups.
- End-to-end monitoring and tracing.
