## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Deployment automation architecture overview**

Deployment automation architecture defines how you move infrastructure and application changes from source control into Azure in a repeatable, auditable, and secure way. For AZ-305, focus on the architecture decisions: which tool owns what, where approvals happen, how secrets are protected, and how you separate platform provisioning from application delivery.

A strong design usually combines:
- **Infrastructure as Code (IaC)** for repeatable environment creation.
- **Pipeline orchestration** for build, test, and release flow.
- **Governance controls** for policy, identity, and approvals.
- **Release strategies** for safe rollout and rollback.

The goal is not just automation speed. It is controlled change with traceability, least privilege, and predictable recovery.

**Core design principles**

Good deployment automation starts with idempotent templates and clear ownership boundaries. You want the same input to produce the same result, whether you deploy once or a hundred times.

- **Idempotency** means repeated deployments converge on the desired state.
- **Separation of concerns** keeps platform, shared services, and application pipelines independent.
- **Environment parity** reduces drift between dev, test, and production.
- **Traceability** links every deployment to a commit, build, and approver.
- **Least privilege** limits what the pipeline identity can change.

Use these principles to decide whether a change belongs in a platform pipeline, an application pipeline, or a manual change process with controlled automation.

**Common Azure deployment toolchain**

Azure supports several automation layers, and the right choice depends on the scope of change.

- **Bicep** is the preferred Azure-native IaC language for ARM deployments.
- **ARM templates** remain relevant for existing estates and generated deployments.
- **Terraform** is common when you need multi-cloud or a mature provider ecosystem.
- **Azure DevOps** and **GitHub Actions** orchestrate builds and releases.
- **Azure CLI** and **PowerShell** are useful for scripts, validation, and operational tasks.

A practical architecture often uses Bicep for Azure resources, then a pipeline runner to validate, deploy, and gate the change.

**Bicep and ARM deployment patterns**

Bicep compiles to Azure Resource Manager (ARM) JSON, so it gives you a cleaner authoring experience while still using ARM deployment semantics. Use modules to split reusable units, parameters to vary environments, and outputs to connect dependent resources.

Example reusable module pattern:

```bicep
// main.bicep
param location string = resourceGroup().location
param appName string
param planName string

module appServicePlan './modules/appserviceplan.bicep' = {
  name: 'appServicePlanDeploy'
  params: {
    location: location
    planName: planName
    skuName: 'P1v3'
  }
}

module webApp './modules/webapp.bicep' = {
  name: 'webAppDeploy'
  params: {
    location: location
    appName: appName
    appServicePlanId: appServicePlan.outputs.id
  }
}
```

```bicep
// modules/appserviceplan.bicep
param location string
param planName string
param skuName string = 'P1v3'

resource asp 'Microsoft.Web/serverfarms@2023-12-01' = {
  name: planName
  location: location
  sku: {
    name: skuName
    tier: 'PremiumV3'
  }
}

output id string = asp.id
```

```bicep
// modules/webapp.bicep
param location string
param appName string
param appServicePlanId string

resource web 'Microsoft.Web/sites@2023-12-01' = {
  name: appName
  location: location
  properties: {
    serverFarmId: appServicePlanId
  }
}
```

Key points:
- **Modules** should be reusable and parameterized, not hardcoded to one environment.
- **Outputs** let one module feed another without manual wiring.
- **Parameters** should capture environment-specific values such as names, SKUs, and locations.
- **What-if** is useful to preview changes before deployment.

Useful commands:

```bash
az bicep build --file infra/main.bicep
az deployment group what-if \
  --resource-group rg-app-prod \
  --template-file infra/main.bicep \
  --parameters appName=web-prod planName=asp-prod
az deployment group create \
  --resource-group rg-app-prod \
  --template-file infra/main.bicep \
  --parameters appName=web-prod planName=asp-prod
```

> `az bicep build` compiles Bicep to ARM JSON. It is not a full validation substitute for `what-if` or an actual deployment.

**Azure DevOps multi-stage YAML**

Azure DevOps multi-stage YAML is a common enterprise choice when you need integrated repos, pipelines, approvals, environments, and artifact management in one system. The core concepts are:

- **Stages** group major phases such as build, test, and deploy.
- **Jobs** run within a stage and can be agent-based or server-based.
- **Deployment jobs** deploy to an Azure DevOps **environment** and support deployment history.
- **Environments** represent deployment targets and are where approvals and checks are enforced.
- **Variable groups** centralize shared variables and can link to Azure Key Vault for secrets.

A typical flow is:
1. Build and test the application.
2. Publish an artifact.
3. Deploy to a dev environment.
4. Promote to test or staging after checks.
5. Deploy to production after environment approvals and policy checks.

Example YAML skeleton:

```yaml
trigger:
- main

variables:
- group: app-shared-vars

stages:
- stage: Build
  jobs:
  - job: BuildJob
    pool:
      vmImage: ubuntu-latest
    steps:
    - script: npm ci
    - script: npm test
    - publish: $(Build.SourcesDirectory)
      artifact: drop

- stage: Deploy_Dev
  dependsOn: Build
  jobs:
  - deployment: DeployWeb
    environment: dev-web
    strategy:
      runOnce:
        deploy:
          steps:
          - download: current
            artifact: drop
          - script: echo "Deploy to dev"
```

Important architecture notes:
- **Approvals and checks** belong on the environment, not as a `preDeploy` script step.
- **Deployment jobs** tie the release record to an environment.
- **Variable groups** are best for shared non-secret values and secret references.
- **Key Vault integration** is preferable for secrets over plain pipeline variables.

**Azure DevOps security and governance**

Pipeline security is part of deployment architecture, not an afterthought. Protect the source, the build, the artifact, and the release path.

- Use **branch policies** to require pull requests, reviewers, and successful builds before merge.
- Enable **code scanning** and dependency scanning where available.
- Use **secret scanning** to detect credentials in source control.
- Sign or attest artifacts when your supply-chain requirements demand integrity verification.
- Restrict service connection usage to approved pipelines and scopes.
- Store secrets in **Azure Key Vault** and reference them from pipelines rather than hardcoding them.

For Azure service connections, the common authentication models are:
- **Service principal** credentials.
- **Workload identity federation** for secretless authentication.

Managed identity is typically used by Azure resources at runtime, not as the usual mechanism for Azure DevOps service connections themselves.

Example Azure CLI service principal login for automation:

```bash
az login --service-principal \
  --username "$APP_ID" \
  --password "$CLIENT_SECRET" \
  --tenant "$TENANT_ID"
```

Example federated credential concept:
- Azure DevOps pipeline identity exchanges a signed token for Azure access.
- No long-lived client secret is stored in the pipeline.

**GitHub Actions deployment automation**

GitHub Actions is often the preferred choice when source control and automation live in GitHub. It is especially useful for teams that want close integration with pull requests, environments, and reusable workflows.

Use GitHub Actions when you need:
- Native repository-centric automation.
- Environment protection rules and required reviewers.
- Reusable workflows across multiple repositories.
- OIDC (OpenID Connect) federation to Azure without storing secrets.

Example deployment workflow:

```yaml
name: deploy-web
on:
  push:
    branches: [ main ]

permissions:
  id-token: write
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4

      - name: Azure login
        uses: azure/login@v2
        with:
          client-id: ${{ secrets.AZURE_CLIENT_ID }}
          tenant-id: ${{ secrets.AZURE_TENANT_ID }}
          subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}

      - name: Deploy Bicep
        run: |
          az deployment group create \
            --resource-group rg-app-prod \
            --template-file infra/main.bicep \
            --parameters appName=web-prod planName=asp-prod
```

Use GitHub Actions when:
- Your organization standardizes on GitHub.
- You want simple repo-to-deploy workflows.
- You need environment protection rules and OIDC-based Azure auth.

Use Azure DevOps when:
- You need deeper enterprise release management.
- You rely on Azure Boards, classic enterprise controls, or existing DevOps estates.
- You want integrated variable groups, environments, and deployment jobs in one platform.

**Release strategies**

Release strategy is how you reduce risk while moving changes through environments. The right strategy depends on blast radius, rollback complexity, and user tolerance for partial exposure.

Common patterns:
- **Blue/green**: deploy to a parallel environment, then switch traffic.
- **Canary**: send a small percentage of traffic to the new version first.
- **Ring-based**: release to internal users, then pilot users, then broad production.
- **Phased rollout**: deploy in waves across regions, subscriptions, or business units.
- **Feature flags**: deploy code disabled by default, then enable behavior gradually.

When to use each:
- **Blue/green** for fast cutover and simple rollback.
- **Canary** for telemetry-driven validation under real traffic.
- **Ring-based** for enterprise user segmentation and controlled adoption.
- **Feature flags** when you need decoupled deployment and release.
- **Phased rollout** when geography or scale requires staged infrastructure changes.

Azure examples:
- App Service deployment slots support blue/green-style swaps.
- Azure Front Door or Application Gateway can shift traffic gradually.
- Feature flags can be managed in application code or a feature management service.

> Deployment slots are not universal across all Azure services or hosting plans. For example, Azure Functions slot support depends on the hosting plan and platform capabilities.

**Approvals, checks, and policy integration**

Approvals and checks should be enforced at the environment or resource boundary, not inside the deployment script. This keeps governance separate from application logic.

Typical enterprise release flow:
1. A pull request merges after branch policy validation.
2. The build pipeline produces a signed or traceable artifact.
3. The release pipeline deploys to a test environment.
4. Environment checks verify business approval, security scan status, or change window.
5. Production deployment proceeds only after approval and policy compliance.

Checks can include:
- Manual approval by a release manager.
- Azure Policy compliance gate.
- Business hours or maintenance window checks.
- Query-based checks against monitoring or incident systems.
- Service connection or environment restrictions.

Azure Policy helps prevent or detect noncompliant resources, and remediation can correct some existing resources depending on the policy effect and resource provider support. It is not a universal fix for every drift scenario.

Example policy assignment flow:

```bash
az policy assignment create \
  --name deny-public-ip \
  --scope /subscriptions/<sub-id>/resourceGroups/rg-prod \
  --policy /providers/Microsoft.Authorization/policyDefinitions/<policy-definition-id>
```

For initiatives, use the full initiative resource ID when assigning a policy set definition.

**Deployment targets and orchestration**

Different Azure services need different deployment approaches. Architecture decisions should reflect the service model, not force one pipeline pattern everywhere.

- **Azure App Service**: deploy code, container images, or infrastructure separately; use slots for safe swaps where supported.
- **Azure Functions**: deploy code with slot support only on supported plans; use package-based deployment and configuration management.
- **Azure Container Apps**: deploy container revisions and use traffic splitting for progressive delivery.
- **Azure API Management**: deploy APIs, policies, products, and revisions carefully because changes can affect consumers immediately.
- **Azure SQL**: use schema migration tools, DACPACs, or migration scripts with prechecks and rollback planning.

Practical patterns:
- Deploy infrastructure first, then application code.
- Use separate pipelines for schema changes and app releases when rollback coupling is risky.
- Treat API changes as versioned contracts.
- Validate container images before promotion.

Example Azure Container Apps deployment:

```bash
az containerapp update \
  --name app-api \
  --resource-group rg-platform \
  --image myregistry.azurecr.io/app-api:1.4.2
```

Example Azure SQL migration pattern:
- Run schema migration in a controlled step.
- Back up or snapshot before destructive changes.
- Deploy backward-compatible changes first.
- Remove old columns or procedures only after the app no longer depends on them.

**Platform pipelines vs application pipelines**

Enterprise architecture usually separates shared platform automation from workload automation.

Platform pipelines manage:
- Landing zones.
- Networking.
- Identity and access.
- Policy assignments.
- Shared monitoring and logging.
- Key Vault, hub-spoke, and connectivity components.

Application pipelines manage:
- App code.
- App-specific infrastructure.
- Configuration.
- Database migrations.
- Release promotion.

This separation matters because:
- Platform changes have broader blast radius.
- Platform owners and app teams often differ.
- Approval paths and change windows may differ.
- Reuse improves consistency across workloads.

A common ownership model is:
- Central platform team owns baseline subscriptions, policies, and shared services.
- Product teams own application resource groups and release pipelines.
- Security and compliance teams define guardrails and checks.

**Pipeline validation and testing**

Validation should happen at multiple layers, not only during deployment.

- **Linting** catches style and structural issues early.
- **Template build** catches syntax problems in Bicep or ARM generation.
- **What-if** previews resource changes.
- **Unit tests** validate application logic.
- **Integration tests** verify service interactions.
- **Smoke tests** confirm the deployed service is healthy.
- **Post-deployment checks** confirm telemetry and user-facing behavior.

Example validation sequence:

```bash
az bicep build --file infra/main.bicep
az deployment group what-if \
  --resource-group rg-test \
  --template-file infra/main.bicep
```

Then deploy only if the preview matches expectations.

**Secrets, identities, and access**

Use identity-based access wherever possible. The pipeline identity should have only the permissions required for the target scope.

Recommended practices:
- Use **service principals** or **workload identity federation** for pipeline-to-Azure authentication.
- Use **Key Vault** for secrets, certificates, and connection strings.
- Prefer **managed identity** for Azure-hosted workloads at runtime.
- Scope role assignments to resource groups or specific resources when possible.
- Rotate credentials if you must use secrets.

Example role assignment:

```bash
az role assignment create \
  --assignee-object-id <object-id> \
  --role "Contributor" \
  --scope /subscriptions/<sub-id>/resourceGroups/rg-app-prod
```

Use narrower roles when possible, such as custom roles or service-specific roles.

**Infrastructure deployment options compared**

| Option | Best for | Strengths | Tradeoffs |
|---|---|---|---|
| Bicep/ARM | Azure-native infrastructure | Declarative, repeatable, policy-friendly | Azure-focused |
| Terraform | Multi-cloud or shared IaC standard | Broad ecosystem, mature state model | State management complexity |
| Azure CLI/PowerShell scripts | Operational tasks and simple automation | Flexible, easy to start | Harder to govern at scale |
| Azure DevOps YAML | Enterprise orchestration | Approvals, environments, artifacts | Platform-specific learning curve |
| GitHub Actions | Repo-centric automation | OIDC, reusable workflows, strong GitHub integration | May need extra enterprise governance design |

Choose the simplest tool that satisfies governance, repeatability, and team ownership requirements.

**Exam-oriented architecture guidance**

For AZ-305, expect scenario questions that ask you to choose the right automation model for enterprise constraints.

Look for these signals:
- Need for **auditable approvals**: use environment checks or protected environments.
- Need for **repeatable Azure resource deployment**: use Bicep or ARM.
- Need for **multi-cloud or cross-platform IaC**: consider Terraform.
- Need for **application release orchestration**: use Azure DevOps or GitHub Actions.
- Need for **secretless authentication**: use OIDC/workload identity federation where supported.
- Need for **safe production rollout**: use slots, canary, rings, or feature flags.
- Need for **governance enforcement**: combine branch policies, policy assignments, and environment checks.

A good answer usually separates:
- Source control controls.
- Build validation.
- Artifact creation.
- Deployment orchestration.
- Environment governance.
- Runtime monitoring and rollback.

**Quick reference**

- **Use Bicep** for Azure infrastructure definitions.
- **Use Azure DevOps** when you need enterprise multi-stage YAML, environments, approvals, and deployment history.
- **Use GitHub Actions** when your delivery model is GitHub-centric and OIDC is a fit.
- **Use policy and branch controls** to prevent bad changes from reaching production.
- **Use release strategies** like blue/green, canary, rings, and feature flags to reduce risk.
- **Use separate pipelines** for platform and application workloads to match ownership and blast radius.
