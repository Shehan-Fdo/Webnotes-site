## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Overview**

Secrets and configuration management in Azure is about separating sensitive values from application code, controlling access with identity, and keeping settings consistent across environments. The main goal is to reduce secret sprawl, avoid hard-coded credentials, and support secure rotation and auditing.

In practice, you usually combine:
- **Azure Key Vault** for secrets, keys, and certificates.
- **Managed identities** for passwordless access from Azure services.
- **Azure App Configuration** for non-secret application settings and feature flags.
- **Infrastructure as Code (IaC)** and CI/CD pipelines for repeatable deployment.

A good design answers three questions:
- Where do secrets live?
- How do workloads authenticate to retrieve them?
- How do you rotate and audit them without downtime?

**Core design principles**

Use the least privileged access model and keep secrets out of source control, build logs, and deployment artifacts. Prefer identity-based access over embedded credentials whenever the platform supports it.

- **Separate concerns** by storing secrets, certificates, and configuration in the right service.
- **Minimize exposure** by retrieving values at runtime instead of baking them into images or packages.
- **Plan rotation** so applications can tolerate secret changes.
- **Audit access** with logs and alerts for retrieval, modification, and deletion events.

A common anti-pattern is placing connection strings in app code, container images, or plain-text pipeline variables. That makes rotation difficult and increases blast radius if a repository or artifact is exposed.

**Azure Key Vault**

Azure Key Vault is the primary Azure service for protecting sensitive data. It stores:
- **Secrets** such as API keys, passwords, and connection strings.
- **Keys** for cryptographic operations and customer-managed encryption scenarios.
- **Certificates** for TLS/SSL and application authentication.

Key Vault supports two broad authorization approaches for data-plane access:
- **Azure role-based access control (RBAC)** using Azure roles such as Key Vault Secrets User or Key Vault Administrator.
- **Access policies** using vault-specific permissions.

Use one model consistently for the vault’s data-plane access design, while remembering that the broader Azure resource model still includes management-plane permissions and legacy scenarios.

Important operational points:
- New Key Vaults have **soft delete** and **purge protection** concepts that help prevent accidental loss.
- Soft delete allows recovery after deletion within the retention window.
- Purge protection prevents permanent deletion until the retention period expires.
- Network access can be restricted with **firewalls**, **private endpoints**, and selected trusted services where appropriate.

> Disabling public network access alone is not enough if you still need workloads to reach the vault. You must design private connectivity and DNS resolution for private endpoint access.

**Key Vault access patterns**

The safest pattern is to let the workload authenticate with a managed identity and retrieve only the specific secret it needs.

Typical flow:
1. The app or service uses its managed identity to request an access token from Microsoft Entra ID.
2. Key Vault validates the token and authorization.
3. The app reads the secret at runtime.

Example Azure CLI commands:
```bash
# Create a Key Vault
az keyvault create \
  --name kv-prod-001 \
  --resource-group rg-platform-prod \
  --location eastus

# Enable a system-assigned managed identity on an App Service
az webapp identity assign \
  --name app-prod-001 \
  --resource-group rg-app-prod

# Grant the app access to read secrets using RBAC
az role assignment create \
  --assignee-object-id <principal-object-id> \
  --role "Key Vault Secrets User" \
  --scope $(az keyvault show --name kv-prod-001 --query id -o tsv)
```

If you use access policies instead of RBAC, you grant explicit secret permissions:
```bash
az keyvault set-policy \
  --name kv-prod-001 \
  --object-id <principal-object-id> \
  --secret-permissions get list
```

Use RBAC when you want centralized Azure authorization management and consistent governance. Use access policies only when you need compatibility with an existing design or legacy pattern.

**Network security for Key Vault**

Key Vault should not be treated as a public, open dependency in sensitive environments. Network controls reduce exposure if credentials are compromised.

Common options:
- **Public endpoint with firewall rules** for limited scenarios.
- **Private endpoint** for private IP access from a virtual network.
- **Private DNS zone** to resolve the vault name to the private endpoint IP.
- **Service endpoints** in some older designs, though private endpoints are generally preferred for stronger isolation.

Example private endpoint topology:
- App Service or VM in a virtual network
- Private endpoint for `kv-prod-001.vault.azure.net`
- Private DNS zone `privatelink.vaultcore.azure.net`
- DNS link to the workload virtual network

A minimal private endpoint creation example:
```bash
az network private-endpoint create \
  --name pe-kv-prod-001 \
  --resource-group rg-platform-prod \
  --vnet-name vnet-prod \
  --subnet snet-private-endpoints \
  --private-connection-resource-id $(az keyvault show --name kv-prod-001 --query id -o tsv) \
  --group-ids vault \
  --connection-name kv-prod-001-conn
```

Then create and link the private DNS zone:
```bash
az network private-dns zone create \
  --resource-group rg-platform-prod \
  --name privatelink.vaultcore.azure.net

az network private-dns link vnet create \
  --resource-group rg-platform-prod \
  --zone-name privatelink.vaultcore.azure.net \
  --name link-prod-vnet \
  --virtual-network vnet-prod \
  --registration-enabled false
```

**Secret rotation and lifecycle**

Rotation is a design requirement, not an afterthought. A secret management solution is only useful if applications can survive secret changes.

Good rotation practices:
- Use versioned secrets in Key Vault.
- Keep the application reading the latest version when possible.
- Support overlapping validity windows during rotation.
- Rotate credentials for external systems on a schedule.
- Remove old versions only after confirming all consumers have switched.

Example secret versioning:
```bash
az keyvault secret set \
  --vault-name kv-prod-001 \
  --name SqlAdminPassword \
  --value 'P@ssw0rd-2026!'

az keyvault secret show \
  --vault-name kv-prod-001 \
  --name SqlAdminPassword
```

Practical rotation pattern:
- Create a new secret version.
- Update the downstream system to accept the new credential.
- Confirm the application can authenticate with the new value.
- Retire the old credential after validation.

For certificates, consider:
- Automatic renewal where supported.
- Export restrictions if the private key must remain non-exportable.
- Integration with services that can consume certificates directly from Key Vault.

**Azure App Configuration**

Azure App Configuration is designed for application settings, feature flags, and centralized configuration, not for storing secrets. It helps you manage non-sensitive values across environments and deployments.

Use it for:
- Feature flags
- Environment-specific settings
- Application tuning values
- Dynamic configuration refresh

Use Key Vault for:
- Passwords
- API keys
- Certificates
- Connection strings that are sensitive

A common pattern is to store a Key Vault reference or secret URI in App Configuration, then let the application resolve the secret through Key Vault using managed identity.

Example App Configuration key-value retrieval:
```bash
az appconfig kv set \
  --name appcfg-prod-001 \
  --key "App:Theme" \
  --value "dark"

az appconfig kv set \
  --name appcfg-prod-001 \
  --key "FeatureFlags:NewCheckout" \
  --value "true"
```

Feature flags are useful when you need:
- Gradual rollout
- A/B testing
- Emergency disablement
- Environment-specific enablement

**App Service and Key Vault references**

Azure App Service supports Key Vault references so you can reference a secret without embedding the value in app settings. The platform resolves the reference and caches it, then refreshes it periodically.

Example app setting:
```text
@Microsoft.KeyVault(SecretUri=https://kv-prod-001.vault.azure.net/secrets/SqlAdminPassword/)
```

Use this pattern when:
- You want to keep the secret out of the app setting value.
- The app can tolerate cached refresh behavior.
- You want a simple integration without custom secret-fetching code.

Be careful with expectations:
- The secret is not guaranteed to update instantly after rotation.
- Refresh timing is platform-managed and can vary.
- You should design for eventual refresh and avoid assuming startup-only resolution.

**Azure App Configuration with Key Vault references**

App Configuration can store references to Key Vault secrets, which lets you centralize app settings while keeping sensitive values in Key Vault.

Typical flow:
1. Store a Key Vault reference in App Configuration.
2. Grant the app managed identity access to App Configuration and Key Vault.
3. The app reads configuration from App Configuration.
4. The app resolves the secret from Key Vault when needed.

This is useful when you want:
- One configuration plane for many apps
- Separation of secret and non-secret data
- Centralized feature flag management

**Managed identities**

Managed identities remove the need to store service credentials in code or configuration. Azure manages the identity lifecycle and token issuance.

Types:
- **System-assigned managed identity**: tied to one Azure resource and deleted with it.
- **User-assigned managed identity**: standalone identity that can be shared across resources.

Use system-assigned identities when:
- The identity should live and die with the resource.
- You want simpler lifecycle management.

Use user-assigned identities when:
- Multiple resources need the same identity.
- You need stable identity reuse across redeployments.
- You want to decouple identity lifecycle from the workload.

Example enabling a user-assigned identity on a web app:
```bash
az identity create \
  --name id-app-prod \
  --resource-group rg-platform-prod

az webapp identity assign \
  --name app-prod-001 \
  --resource-group rg-app-prod \
  --identities $(az identity show --name id-app-prod --resource-group rg-platform-prod --query id -o tsv)
```

**Configuration in application code**

Your application should load configuration from the platform rather than from hard-coded values. The exact implementation depends on the runtime.

Example .NET pattern using Azure App Configuration and Key Vault:
```csharp
using Azure.Identity;

var builder = WebApplication.CreateBuilder(args);

builder.Configuration.AddAzureAppConfiguration(options =>
{
    options.Connect(new Uri("https://appcfg-prod-001.azconfig.io"), new DefaultAzureCredential())
           .ConfigureKeyVault(kv => kv.SetCredential(new DefaultAzureCredential()));
});

var app = builder.Build();
app.MapGet("/", () => "OK");
app.Run();
```

Example environment variable pattern for local development:
```bash
export AZURE_TENANT_ID="<tenant-id>"
export AZURE_CLIENT_ID="<client-id>"
export AZURE_CLIENT_SECRET="<client-secret>"
```

Use `DefaultAzureCredential` when you want the SDK to try multiple identity sources in a predictable order, such as managed identity in Azure and developer credentials locally.

**Infrastructure as Code and deployment**

IaC helps you define secret and configuration resources consistently. For Azure, common tools include Bicep, ARM templates, Terraform, and Azure CLI in pipelines.

A Bicep example for a Key Vault with private access and RBAC-oriented design:
```bicep
param location string = resourceGroup().location
param vaultName string

resource kv 'Microsoft.KeyVault/vaults@2023-07-01' = {
  name: vaultName
  location: location
  properties: {
    tenantId: tenant().tenantId
    sku: {
      family: 'A'
      name: 'standard'
    }
    enableRbacAuthorization: true
    publicNetworkAccess: 'Disabled'
    networkAcls: {
      defaultAction: 'Deny'
      bypass: 'AzureServices'
      ipRules: []
      virtualNetworkRules: []
    }
    accessPolicies: []
  }
}
```

Use this pattern only when you also provide private endpoint connectivity and DNS resolution for the workloads that must reach the vault.

A Bicep example for App Configuration:
```bicep
resource appcfg 'Microsoft.AppConfiguration/configurationStores@2023-03-01' = {
  name: 'appcfg-prod-001'
  location: location
  sku: {
    name: 'standard'
  }
}
```

Deployment pipeline guidance:
- Inject secrets from secure stores, not plain pipeline variables.
- Prefer federated identity or managed identity for pipeline authentication where possible.
- Mask sensitive values in logs.
- Avoid echoing secret values in scripts.

Example Azure DevOps or shell-style secret retrieval:
```bash
SECRET_VALUE=$(az keyvault secret show \
  --vault-name kv-prod-001 \
  --name SqlAdminPassword \
  --query value -o tsv)

echo "Secret retrieved without printing the value"
```

**Local development**

Local development should mirror production patterns as closely as possible without exposing production secrets.

Recommended approach:
- Use developer identity with `az login`.
- Use a separate dev Key Vault and App Configuration instance.
- Store local-only settings in user secrets or environment variables.
- Never copy production secrets to a laptop unless absolutely required and approved.

Example .NET user secrets:
```bash
dotnet user-secrets init
dotnet user-secrets set "ConnectionStrings:SqlDb" "Server=localhost;Database=appdb;Trusted_Connection=True;"
```

This is appropriate for local-only values, but not for shared team secrets or production credentials.

**Governance, monitoring, and compliance**

Secret management is also a governance problem. You need visibility into who accessed what and when.

Monitor:
- Key Vault diagnostic logs
- Secret get/list/set/delete operations
- Private endpoint health
- Failed authentication attempts
- Configuration changes in App Configuration

Useful controls:
- **Azure Policy** to require private endpoints, purge protection, or diagnostic settings.
- **Microsoft Defender for Cloud** recommendations for exposed vaults and weak configurations.
- **Role reviews** to remove stale access.
- **Resource locks** for critical vaults where appropriate.

Example policy goals:
- Deny Key Vaults without purge protection.
- Audit vaults with public network access enabled.
- Require diagnostic settings to a Log Analytics workspace.

**Common exam scenarios**

When you see a scenario, map the requirement to the right service:
- Store a password or API key: use **Key Vault**.
- Store feature flags or app settings: use **App Configuration**.
- Avoid credentials in code: use **managed identity**.
- Restrict access to a vault from a subnet: use **private endpoint** and DNS.
- Rotate secrets without redeploying code: use **versioned secrets** and runtime retrieval.
- Centralize non-secret settings across many apps: use **App Configuration**.
- Secure certificate distribution: use **Key Vault certificates**.

**Troubleshooting checklist**

If an app cannot read a secret:
- Confirm the identity is enabled on the workload.
- Confirm the identity has the correct Key Vault role or access policy.
- Confirm network access to the vault is allowed.
- Confirm private DNS resolves the vault name correctly.
- Confirm the secret name and version are correct.
- Check whether the app is using a cached reference or stale configuration.

If a secret rotation appears not to work:
- Verify the new version exists in Key Vault.
- Confirm the app refresh mechanism is active.
- Check whether the downstream service still accepts the old credential.
- Review logs for authorization or network failures.

**Quick comparison**

| Capability | Key Vault | App Configuration |
|---|---|---|
| Secrets | Yes | No |
| Feature flags | No | Yes |
| Certificates | Yes | No |
| Non-secret app settings | Limited use | Yes |
| Rotation support | Secret versioning | Refresh and change management |
| Typical access model | Managed identity + RBAC/access policy | Managed identity + RBAC |

**Exam takeaways**

- Use **Key Vault** for sensitive data and cryptographic material.
- Use **App Configuration** for non-secret settings and feature flags.
- Prefer **managed identities** over stored credentials.
- Design for **private connectivity** when the vault must not be publicly reachable.
- Plan for **rotation**, **auditing**, and **eventual refresh** behavior.
- Keep secret retrieval at runtime and avoid hard-coding values into code or images.
