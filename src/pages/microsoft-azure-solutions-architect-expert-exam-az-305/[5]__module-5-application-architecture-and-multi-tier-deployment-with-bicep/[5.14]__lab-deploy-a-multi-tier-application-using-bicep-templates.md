## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Lab overview**

A multi-tier application separates concerns into presentation, application, and data layers so you can scale, secure, and operate each tier independently. In Azure, Bicep gives you a declarative way to define the full environment: virtual network, network security groups (NSGs), compute, database, private connectivity, identity, and monitoring.

A realistic lab should model a business app such as an internal order-processing system:

- **Web tier** serves the user interface.
- **App tier** hosts business logic and APIs.
- **Data tier** stores relational data in Azure SQL Database.
- **Shared services** provide secrets, logs, and deployment automation.

A good exam-ready design emphasizes least privilege, private access, and repeatable deployment. You should be able to explain not only what each resource does, but also why it belongs in that tier.

**Reference architecture**

```text
Internet
   |
[Application Gateway / Public entry point]
   |
[Web tier subnet] -----> [App tier subnet] -----> [SQL private endpoint]
   |                           |                          |
   |                           |                          +--> Private DNS zone
   |                           +--> Key Vault / Managed Identity
   +--> NSG rules              +--> NSG rules             +--> NSG rules
```

**Core design goals**

- **Repeatability**: deploy the same environment from source control.
- **Isolation**: keep tiers in separate subnets and restrict traffic.
- **Private access**: use private endpoints where possible.
- **Identity-first access**: prefer managed identity over stored credentials.
- **Observability**: send logs and metrics to Log Analytics.

**Bicep project structure**

A maintainable lab usually splits infrastructure into modules.

```text
main.bicep
modules/
  network.bicep
  nsg.bicep
  keyvault.bicep
  sql.bicep
  appvm.bicep
  private-endpoint.bicep
  monitoring.bicep
  role-assignments.bicep
```

This structure helps you test each layer independently and reuse modules in other labs.

**Bicep fundamentals you should know**

Bicep is a domain-specific language for Azure Resource Manager (ARM) templates. It compiles to JSON ARM templates, but it is easier to read and maintain.

- **Parameters** let you vary names, sizes, and locations.
- **Variables** reduce repetition and build derived values.
- **Resources** declare Azure objects.
- **Modules** compose reusable templates.
- **Outputs** expose values for later steps or pipelines.

A minimal Bicep file looks like this:

```bicep
param location string = resourceGroup().location
param appName string

resource storage 'Microsoft.Storage/storageAccounts@2023-01-01' = {
  name: toLower('${appName}st')
  location: location
  sku: {
    name: 'Standard_LRS'
  }
  kind: 'StorageV2'
}

output storageName string = storage.name
```

**Deployment workflow**

1. Create or select a resource group.
2. Validate the template.
3. What-if the deployment to inspect changes.
4. Deploy the template.
5. Verify networking, identity, and application access.
6. Attach diagnostics and confirm logs arrive in Log Analytics.

Use validation as a helpful check, but not as a guarantee that every runtime dependency will succeed.

```bash
az group create -n rg-multitier-lab -l eastus

az deployment group validate \
  --resource-group rg-multitier-lab \
  --template-file main.bicep \
  --parameters @main.parameters.json

az deployment group what-if \
  --resource-group rg-multitier-lab \
  --template-file main.bicep \
  --parameters @main.parameters.json

az deployment group create \
  --resource-group rg-multitier-lab \
  --template-file main.bicep \
  --parameters @main.parameters.json
```

**Networking layer**

The network is the foundation of the lab. Separate subnets let you apply different NSG rules and private endpoint policies.

Typical subnet layout:

- **`snet-web`** for the web VM or web service.
- **`snet-app`** for the application VM.
- **`snet-pe`** for private endpoints.
- **`snet-bastion`** if you use Azure Bastion for secure administration.

A simple virtual network module:

```bicep
param location string
param vnetName string = 'vnet-multitier'
param addressPrefix string = '10.10.0.0/16'

resource vnet 'Microsoft.Network/virtualNetworks@2023-11-01' = {
  name: vnetName
  location: location
  properties: {
    addressSpace: {
      addressPrefixes: [
        addressPrefix
      ]
    }
    subnets: [
      {
        name: 'snet-web'
        properties: {
          addressPrefix: '10.10.1.0/24'
        }
      }
      {
        name: 'snet-app'
        properties: {
          addressPrefix: '10.10.2.0/24'
        }
      }
      {
        name: 'snet-pe'
        properties: {
          addressPrefix: '10.10.3.0/24'
          privateEndpointNetworkPolicies: 'Disabled'
        }
      }
    ]
  }
}
```

The `privateEndpointNetworkPolicies` setting must be disabled on the subnet that hosts private endpoints.

**NSG rules**

NSGs control traffic between tiers and from the internet. In a lab, you usually allow only the ports required for the app.

Example NSG for the web subnet:

```bicep
param location string
param nsgName string = 'nsg-web'

resource nsg 'Microsoft.Network/networkSecurityGroups@2023-11-01' = {
  name: nsgName
  location: location
  properties: {
    securityRules: [
      {
        name: 'Allow-HTTP-In'
        properties: {
          priority: 100
          direction: 'Inbound'
          access: 'Allow'
          protocol: 'Tcp'
          sourcePortRange: '*'
          destinationPortRange: '80'
          sourceAddressPrefix: 'Internet'
          destinationAddressPrefix: '*'
        }
      }
      {
        name: 'Allow-HTTPS-In'
        properties: {
          priority: 110
          direction: 'Inbound'
          access: 'Allow'
          protocol: 'Tcp'
          sourcePortRange: '*'
          destinationPortRange: '443'
          sourceAddressPrefix: 'Internet'
          destinationAddressPrefix: '*'
        }
      }
      {
        name: 'Deny-All-In'
        properties: {
          priority: 4096
          direction: 'Inbound'
          access: 'Deny'
          protocol: '*'
          sourcePortRange: '*'
          destinationPortRange: '*'
          sourceAddressPrefix: '*'
          destinationAddressPrefix: '*'
        }
      }
    ]
  }
}
```

For the app subnet, allow only traffic from the web subnet and deny everything else.

```bicep
resource appNsg 'Microsoft.Network/networkSecurityGroups@2023-11-01' = {
  name: 'nsg-app'
  location: location
  properties: {
    securityRules: [
      {
        name: 'Allow-Web-To-App'
        properties: {
          priority: 100
          direction: 'Inbound'
          access: 'Allow'
          protocol: 'Tcp'
          sourcePortRange: '*'
          destinationPortRange: '8080'
          sourceAddressPrefix: '10.10.1.0/24'
          destinationAddressPrefix: '10.10.2.0/24'
        }
      }
      {
        name: 'Deny-All-In'
        properties: {
          priority: 4096
          direction: 'Inbound'
          access: 'Deny'
          protocol: '*'
          sourcePortRange: '*'
          destinationPortRange: '*'
          sourceAddressPrefix: '*'
          destinationAddressPrefix: '*'
        }
      }
    ]
  }
}
```

> Be careful not to block platform traffic needed for VM provisioning, extension installation, or private endpoint resolution.

**Compute tier**

For a lab, the web and app tiers are often deployed as Linux or Windows VMs with custom script extensions, or as App Service and VM combinations. If the topic specifically uses VMs, you should show how to provision them and then configure the application after deployment.

A VM module should include:

- NICs attached to the correct subnet.
- NSG association at the subnet or NIC level.
- OS disk and image selection.
- Admin credentials or SSH keys.
- Extensions for post-deployment configuration.

Example Linux VM module:

```bicep
param location string
param vmName string
param subnetId string
param adminUsername string
@secure()
param sshPublicKey string

resource nic 'Microsoft.Network/networkInterfaces@2023-11-01' = {
  name: '${vmName}-nic'
  location: location
  properties: {
    ipConfigurations: [
      {
        name: 'ipconfig1'
        properties: {
          subnet: {
            id: subnetId
          }
          privateIPAllocationMethod: 'Dynamic'
        }
      }
    ]
  }
}

resource vm 'Microsoft.Compute/virtualMachines@2023-09-01' = {
  name: vmName
  location: location
  properties: {
    hardwareProfile: {
      vmSize: 'Standard_B2s'
    }
    osProfile: {
      computerName: vmName
      adminUsername: adminUsername
      linuxConfiguration: {
        disablePasswordAuthentication: true
        ssh: {
          publicKeys: [
            {
              path: '/home/${adminUsername}/.ssh/authorized_keys'
              keyData: sshPublicKey
            }
          ]
        }
      }
    }
    storageProfile: {
      imageReference: {
        publisher: 'Canonical'
        offer: '0001-com-ubuntu-server-jammy'
        sku: '22_04-lts'
        version: 'latest'
      }
      osDisk: {
        createOption: 'FromImage'
      }
    }
    networkProfile: {
      networkInterfaces: [
        {
          id: nic.id
        }
      ]
    }
  }
}
```

**Post-deployment application setup**

A VM is not a complete application until you install and configure the app code. Use a VM extension or a deployment script to bootstrap the software.

Example Custom Script Extension for a Linux app server:

```bicep
resource appInstall 'Microsoft.Compute/virtualMachines/extensions@2023-09-01' = {
  name: '${vm.name}/customScript'
  location: location
  properties: {
    publisher: 'Microsoft.Azure.Extensions'
    type: 'CustomScript'
    typeHandlerVersion: '2.1'
    settings: {
      fileUris: [
        'https://raw.githubusercontent.com/example-org/multitier-lab/main/install-app.sh'
      ]
      commandToExecute: 'bash install-app.sh'
    }
  }
}
```

A script like `install-app.sh` might install NGINX, copy application files, and configure the service:

```bash
#!/usr/bin/env bash
set -euo pipefail

apt-get update
apt-get install -y nginx
systemctl enable nginx
systemctl start nginx
```

If the app tier exposes an API, you can configure it to listen on a private port such as `8080` and have the web tier proxy requests to it.

**Data tier with Azure SQL Database**

Azure SQL Database is a common choice for the data tier because it is managed, highly available, and supports private connectivity. Choose a service objective that matches the lab scenario. Do not pair a small SKU with an oversized database limit.

Example SQL server and database:

```bicep
param location string
param sqlServerName string
param sqlAdminLogin string
@secure()
param sqlAdminPassword string

resource sqlServer 'Microsoft.Sql/servers@2023-08-01-preview' = {
  name: sqlServerName
  location: location
  properties: {
    administratorLogin: sqlAdminLogin
    administratorLoginPassword: sqlAdminPassword
    version: '12.0'
    publicNetworkAccess: 'Disabled'
  }
}

resource sqlDb 'Microsoft.Sql/servers/databases@2023-08-01-preview' = {
  parent: sqlServer
  name: 'appdb'
  location: location
  sku: {
    name: 'S0'
    tier: 'Standard'
    capacity: 10
  }
  properties: {
    maxSizeBytes: 5368709120
    zoneRedundant: false
  }
}
```

For a lab, keep the database size aligned with the selected service objective. If you need a larger database, choose a more appropriate tier and SKU.

**Private endpoint and private DNS**

Private endpoints let Azure PaaS services appear as private IPs inside your virtual network. This is a key exam concept because it reduces exposure and supports zero-trust design.

For Azure SQL Database, you typically create:

- A private endpoint in the `snet-pe` subnet.
- A private DNS zone such as `privatelink.database.windows.net`.
- A virtual network link to the DNS zone.

Example:

```bicep
param location string
param peName string = 'pe-sql'
param subnetId string
param sqlServerId string
param dnsZoneName string = 'privatelink.database.windows.net'
param vnetId string

resource dnsZone 'Microsoft.Network/privateDnsZones@2020-06-01' = {
  name: dnsZoneName
  location: 'global'
}

resource dnsLink 'Microsoft.Network/privateDnsZones/virtualNetworkLinks@2020-06-01' = {
  parent: dnsZone
  name: 'link-multitier-vnet'
  location: 'global'
  properties: {
    virtualNetwork: {
      id: vnetId
    }
    registrationEnabled: false
  }
}

resource privateEndpoint 'Microsoft.Network/privateEndpoints@2023-11-01' = {
  name: peName
  location: location
  properties: {
    subnet: {
      id: subnetId
    }
    privateLinkServiceConnections: [
      {
        name: 'sqlConnection'
        properties: {
          privateLinkServiceId: sqlServerId
          groupIds: [
            'sqlServer'
          ]
        }
      }
    ]
  }
}

resource dnsZoneGroup 'Microsoft.Network/privateEndpoints/privateDnsZoneGroups@2023-11-01' = {
  parent: privateEndpoint
  name: 'default'
  properties: {
    privateDnsZoneConfigs: [
      {
        name: 'sqlZone'
        properties: {
          privateDnsZoneId: dnsZone.id
        }
      }
    ]
  }
}
```

**Identity and access**

Managed identity is the preferred way for the app tier to access Azure resources without storing secrets in code or configuration. Use a system-assigned identity for a single VM or a user-assigned identity if multiple resources share the same identity.

Example VM with identity:

```bicep
resource appVm 'Microsoft.Compute/virtualMachines@2023-09-01' = {
  name: 'mtapp-appvm'
  location: location
  identity: {
    type: 'SystemAssigned'
  }
  properties: {
    // omitted for brevity
  }
}
```

To allow the VM to read secrets from Key Vault, assign a role at the vault scope.

```bicep
param principalId string
param keyVaultName string

resource kv 'Microsoft.KeyVault/vaults@2023-07-01' existing = {
  name: keyVaultName
}

resource kvSecretsUser 'Microsoft.Authorization/roleAssignments@2022-04-01' = {
  name: guid(kv.id, principalId, 'KeyVaultSecretsUser')
  scope: kv
  properties: {
    principalId: principalId
    roleDefinitionId: subscriptionResourceId(
      'Microsoft.Authorization/roleDefinitions',
      '4633458b-17de-408a-b874-0445c86b69e6'
    )
    principalType: 'ServicePrincipal'
  }
}
```

If you use Key Vault in RBAC mode, do not rely on access policies. In that case, role assignments control access.

For SQL access, you can use Microsoft Entra ID authentication with a managed identity. The exact setup depends on the SQL configuration and whether you create an Entra admin.

**Key Vault**

Key Vault stores secrets such as connection strings, API keys, or certificates. In a secure design, the app retrieves secrets at runtime rather than embedding them in templates.

Example Key Vault with RBAC:

```bicep
param location string
param keyVaultName string
param tenantId string

resource kv 'Microsoft.KeyVault/vaults@2023-07-01' = {
  name: keyVaultName
  location: location
  properties: {
    tenantId: tenantId
    sku: {
      family: 'A'
      name: 'standard'
    }
    enableRbacAuthorization: true
    publicNetworkAccess: 'Disabled'
    networkAcls: {
      defaultAction: 'Deny'
      bypass: 'AzureServices'
    }
  }
}
```

A secret can be deployed after the vault exists:

```bicep
resource sqlConnSecret 'Microsoft.KeyVault/vaults/secrets@2023-07-01' = {
  parent: kv
  name: 'SqlConnectionString'
  properties: {
    value: 'Server=tcp:myserver.database.windows.net,1433;Database=appdb;Authentication=Active Directory Managed Identity;'
  }
}
```

In practice, you often populate secrets from a deployment script or pipeline variable rather than hardcoding values in the template.

**Deployment scripts**

Deployment scripts are useful when you need to run Azure CLI or PowerShell during deployment, such as creating a SQL login, seeding data, or configuring a resource that Bicep cannot express directly.

A valid deployment script resource needs the required execution settings, identity, and storage configuration. A minimal Azure CLI example:

```bicep
param location string
param scriptName string = 'seed-sql'
param storageAccountName string
param userAssignedIdentityId string

resource script 'Microsoft.Resources/deploymentScripts@2023-08-01' = {
  name: scriptName
  location: location
  kind: 'AzureCLI'
  identity: {
    type: 'UserAssigned'
    userAssignedIdentities: {
      '${userAssignedIdentityId}': {}
    }
  }
  properties: {
    azCliVersion: '2.57.0'
    retentionInterval: 'P1D'
    cleanupPreference: 'OnSuccess'
    forceUpdateTag: utcNow()
    storageAccountSettings: {
      storageAccountName: storageAccountName
      storageAccountKey: listKeys(resourceId('Microsoft.Storage/storageAccounts', storageAccountName), '2023-01-01').keys[0].value
    }
    scriptContent: '''
      echo "Seeding database..."
      az account show
    '''
  }
}
```

Use deployment scripts when you need controlled imperative steps, but keep the main infrastructure declarative.

**Monitoring and diagnostics**

A Log Analytics workspace centralizes logs and metrics from the lab. Attach diagnostic settings to the resources you care about, such as the VM, SQL server, Key Vault, and network resources.

Example workspace:

```bicep
resource law 'Microsoft.OperationalInsights/workspaces@2023-09-01' = {
  name: 'law-multitier'
  location: location
  properties: {
    sku: {
      name: 'PerGB2018'
    }
    retentionInDays: 30
  }
}
```

Example diagnostic settings for SQL:

```bicep
resource sqlDiag 'Microsoft.Insights/diagnosticSettings@2021-05-01-preview' = {
  name: 'send-to-law'
  scope: sqlServer
  properties: {
    workspaceId: law.id
    logs: [
      {
        category: 'SQLSecurityAuditEvents'
        enabled: true
      }
      {
        category: 'AutomaticTuning'
        enabled: true
      }
    ]
    metrics: [
      {
        category: 'AllMetrics'
        enabled: true
      }
    ]
  }
}
```

You can apply similar diagnostic settings to Key Vault and virtual machines. This is important for troubleshooting and for demonstrating operational readiness.

**End-to-end wiring**

A complete lab should connect the tiers rather than deploy them in isolation.

- The **web tier** receives public traffic or traffic from a front door component.
- The **app tier** accepts only web-tier traffic on a private port.
- The **app tier** uses managed identity to read secrets from Key Vault.
- The **app tier** connects to SQL through a private endpoint and private DNS.
- The **monitoring tier** collects logs from all major resources.

A practical app configuration example on Linux might use environment variables:

```bash
export APP_PORT=8080
export SQL_SERVER=myserver.database.windows.net
export KEYVAULT_URI=https://kv-multitier.vault.azure.net/
```

If you use App Service instead of a VM for the web tier, you can store connection details in application settings and reference Key Vault secrets directly. For example:

```json
{
  "ConnectionStrings__AppDb": "@Microsoft.KeyVault(SecretUri=https://kv-multitier.vault.azure.net/secrets/SqlConnectionString/)"
}
```

**Verification commands**

After deployment, verify each layer separately.

- Check resource creation:
  ```bash
  az resource list -g rg-multitier-lab -o table
  ```
- Check VM private IPs:
  ```bash
  az vm list-ip-addresses -g rg-multitier-lab -n mtapp-appvm -o jsonc
  ```
- Check private endpoint connections:
  ```bash
  az network private-endpoint-connection list \
    --id $(az sql server show -g rg-multitier-lab -n myserver --query id -o tsv) \
    -o table
  ```
- Check NSG rules:
  ```bash
  az network nsg rule list -g rg-multitier-lab --nsg-name nsg-app -o table
  ```
- Check diagnostics:
  ```bash
  az monitor diagnostic-settings list --resource $(az sql server show -g rg-multitier-lab -n myserver --query id -o tsv) -o jsonc
  ```

**Troubleshooting**

Common issues in this lab usually come from networking or identity.

- **Private DNS not resolving**: confirm the DNS zone link exists and the private endpoint has a zone group.
- **VM cannot reach SQL**: verify NSG rules, private endpoint approval, and SQL public access settings.
- **Key Vault access denied**: confirm the managed identity exists and has the correct RBAC role.
- **Deployment script fails**: check identity permissions, storage settings, and CLI version compatibility.
- **App install fails**: inspect extension logs on the VM and confirm outbound internet access if the script downloads packages.

> If you disable public network access on SQL or Key Vault, you must provide a working private path and DNS resolution before the app can use those services.

**Exam takeaways**

- **Bicep modules** help you organize complex Azure solutions.
- **NSGs** enforce tier-to-tier traffic boundaries.
- **Private endpoints** keep PaaS traffic off the public internet.
- **Managed identity** removes the need for embedded secrets.
- **Diagnostic settings** are essential for operational visibility.
- **Validation and what-if** reduce deployment risk, but you still need runtime verification.

**Suggested lab completion checklist**

1. Deploy the virtual network and subnets.
2. Attach NSGs to the web and app subnets.
3. Deploy the web and app VMs.
4. Install the application on the app VM.
5. Deploy Azure SQL Database with public access disabled.
6. Create the SQL private endpoint and private DNS zone link.
7. Deploy Key Vault and grant the app identity access.
8. Configure app settings or secrets references.
9. Deploy Log Analytics and diagnostic settings.
10. Verify end-to-end connectivity and logging.
