## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Hybrid resilience design goals**

Hybrid resilience means your solution keeps working when a failure affects one or more layers: identity, network, compute, storage, or an entire site. In Azure architectures, you usually design for a specific recovery objective rather than “no downtime” in the abstract.

Key terms to anchor your design:

- **RPO** (Recovery Point Objective): how much data loss is acceptable.
- **RTO** (Recovery Time Objective): how long recovery can take.
- **Failover**: switching to a secondary system after a failure.
- **Failback**: returning to the primary system after recovery.

A good hybrid design separates failure domains so that one outage does not cascade into every dependency. For example, a branch office can lose its WAN link while Azure-hosted services remain available, or an on-premises identity component can fail while cloud authentication continues through a different method.

**Identity resilience in hybrid environments**

Identity is often the first hidden dependency in a hybrid outage. If users cannot authenticate, the application may be healthy but still unusable.

Microsoft Entra ID is the cloud control plane for identity, and hybrid authentication commonly uses one of these patterns:

- **Password hash synchronization (PHS)**: hashes of on-premises passwords are synchronized to Microsoft Entra ID. Sign-in can continue even if the on-premises directory or federation path is unavailable.
- **Pass-through authentication (PTA)**: authentication requests are validated against on-premises agents. If all agents are unavailable, sign-in for affected users can fail.
- **Federation with AD FS**: sign-in depends on the federation service path, including DNS, certificates, AD FS servers, and often Web Application Proxy (WAP). If any critical part fails, federated sign-in can fail.

Practical guidance:

- Prefer **PHS** for most resilience-focused designs because it removes dependency on the on-premises authentication path during sign-in.
- Use **PTA** when you need on-premises password validation but still want a simpler model than federation.
- Use **AD FS** only when you have a specific requirement that cannot be met with PHS or PTA, such as certain legacy or advanced federation scenarios.

> Federated sign-in availability depends on the full path, not just the AD FS service. DNS, certificates, load balancers, WAP, and network reachability are all part of the failure domain.

For Entra Connect, staging mode is a standby pattern, not an active sync mode:

- A staging server imports and synchronizes metadata internally.
- It does **not** export changes while in staging mode.
- You use it to validate configuration, test upgrades, and provide a warm standby for failover.

Example operational approach:

1. Keep one active Entra Connect server exporting changes.
2. Keep a second server in staging mode.
3. Monitor synchronization health and configuration drift.
4. If the primary fails, disable staging mode on the standby and promote it to active.

**Network resilience for hybrid connectivity**

Hybrid connectivity usually relies on VPN, ExpressRoute, or both. The right choice depends on bandwidth, latency, SLA, and failure tolerance.

**Site-to-site VPN**

Use VPN when you need lower cost and acceptable internet-based connectivity. For resilience:

- Deploy redundant VPN devices on-premises.
- Use redundant ISPs where possible.
- Prefer diverse physical paths into the datacenter.
- Configure active-active VPN gateways when supported by your design.
- Use BGP (Border Gateway Protocol) when you need dynamic route exchange and faster convergence.

Example Azure CLI for a VPN gateway with active-active support depends on SKU and region capabilities:

```bash
az network vnet-gateway create \
  --resource-group rg-hybrid \
  --name vnet-gw-eastus \
  --vnet vnet-hub \
  --public-ip-addresses pip-gw-1 pip-gw-2 \
  --gateway-type Vpn \
  --vpn-type RouteBased \
  --sku VpnGw2AZ \
  --enable-bgp true
```

**ExpressRoute**

Use ExpressRoute when you need private connectivity with predictable performance and stronger enterprise resilience. For high availability:

- Provision **dual circuits** in different peering locations or with different providers when possible.
- Use diverse paths from your datacenter to the provider edge.
- Enable BGP for route exchange and failover.
- Consider **ExpressRoute Global Reach** when you need private connectivity between on-premises sites through Microsoft’s backbone.
- Consider **FastPath** where supported to reduce data path latency by bypassing the gateway for certain traffic flows.

Example pattern for redundancy:

- Circuit A: primary provider, primary metro.
- Circuit B: secondary provider or secondary metro.
- On-premises edge routers: separate power, separate racks, separate uplinks.
- Azure hub: zone-redundant gateway where supported by SKU and region.

> Zone-redundant VPN gateways are not universally available. Support depends on the gateway SKU and Azure region.

**Virtual WAN**

Use Azure Virtual WAN when you want centralized branch connectivity, simplified routing, and managed hub-to-hub or branch-to-Azure connectivity. It is useful when many sites need consistent routing and failover behavior.

Choose Virtual WAN when:

- You have many branches.
- You want centralized policy and routing.
- You need easier branch onboarding than managing many point-to-point tunnels.

Choose ExpressRoute or VPN directly when:

- You need a very specific topology.
- You require tight control over routing or appliances.
- You have a small number of sites and want minimal abstraction.

**On-premises network resilience**

Hybrid resilience is not only an Azure design problem. Your on-premises edge is part of the failure domain.

Design for:

- **Redundant ISPs** to reduce carrier outage risk.
- **Redundant firewalls and routers** to avoid single-device failure.
- **Diverse physical paths** to reduce fiber cut impact.
- Separate power feeds, UPS, and generator coverage where possible.
- Health checks for WAN links, tunnels, and BGP neighbors.

A common exam trap is assuming cloud redundancy compensates for a single on-premises edge device. It does not. If your only firewall fails, the Azure side may still be healthy but unreachable.

**Data-layer resilience**

Data resilience is different from compute resilience. Replication keeps data available across failure domains; backup protects against corruption, deletion, and ransomware.

**Replication and high availability**

Use replication when you need low RPO and fast failover:

- **SQL Server Always On availability groups**: best when you need database-level high availability and controlled failover.
- **SQL Server failover cluster instances**: useful when shared storage and instance-level failover fit the workload.
- **Storage replication**: useful for file shares, blobs, or storage-backed workloads depending on the platform.
- **Application-level replication**: useful when the app can manage its own state or queue-based consistency.

Choose based on workload behavior:

- Use **Always On availability groups** when the application can reconnect to a secondary replica and you need database-level resilience.
- Use **backup and restore** when the workload can tolerate longer recovery times and you need protection from logical corruption.
- Use **storage replication** when the storage layer is the right abstraction and the application is not replication-aware.

Example SQL Server Always On considerations:

- Synchronous commit gives lower data loss but can increase latency.
- Asynchronous commit reduces latency but can increase RPO.
- Automatic failover requires the right quorum and configuration.
- Cross-region or cross-site designs often use asynchronous replication.

**Backup as a resilience layer**

Backup is not the same as disaster recovery (DR). Backup protects against accidental deletion, corruption, and ransomware. DR focuses on restoring service after a site or platform failure.

Use Azure Backup and hybrid backup integration when you need:

- Centralized policy management.
- Offsite copies in Azure.
- Recovery from accidental deletion or malicious changes.
- Long-term retention.

Common patterns:

- On-premises servers back up to a local appliance or backup server, then replicate backup copies to Azure.
- Azure VMs use Azure Backup for point-in-time recovery.
- Critical databases use native backup tools plus offsite retention.

Practical distinction:

- **Replication** helps you recover quickly from infrastructure failure.
- **Backup** helps you recover from data loss, corruption, or bad changes.

> Replication can faithfully copy corruption. Backup gives you a restore point before the corruption occurred.

**Azure Site Recovery for hybrid DR**

Azure Site Recovery (ASR) is the primary Azure DR service for orchestrating replication, failover, and failback for supported workloads.

Use ASR when you need:

- Replication of on-premises VMs to Azure.
- Orchestrated failover during a site outage.
- Planned failover for maintenance or migration.
- Failback after the primary site is restored.

Core ASR concepts:

- **Replication**: continuously copies changes to the recovery region.
- **Test failover**: validates recovery without impacting production.
- **Planned failover**: controlled switchover with source site available.
- **Unplanned failover**: recovery after source site loss.
- **Failback**: returns workloads to the primary site after recovery.

Typical workflow:

1. Enable replication for the protected VM or workload.
2. Define recovery point settings and network mapping.
3. Run a test failover in an isolated network.
4. Validate application health, DNS, and user access.
5. Execute planned or unplanned failover when needed.
6. Reprotect and fail back when the primary site is stable.

Example validation checklist after failover:

- Confirm VM boots and services start.
- Verify IP addressing and DNS resolution.
- Check application dependencies and connection strings.
- Validate route convergence from branch sites.
- Confirm monitoring and alerting are active in the recovery site.

**Testing failover and failback**

A resilient design is not complete until you test it. Many hybrid outages become worse because failover was never validated end to end.

Test these scenarios:

- **Planned failover**: verify controlled movement of workloads.
- **Unplanned failover**: verify recovery when the source is unavailable.
- **Failback**: verify return to the primary site without data loss beyond the accepted RPO.
- **Route convergence**: verify BGP or static route updates propagate correctly.
- **Application health**: verify the app is not only running but usable.

Practical test guidance:

- Use a nonproduction recovery network for ASR test failovers.
- Validate DNS TTL (time to live) behavior so clients do not cache stale records too long.
- Confirm load balancers and health probes point to the active site.
- Document manual steps for services that cannot fail over automatically.

**Choosing between VPN, ExpressRoute, Virtual WAN, and ASR**

These services solve different problems, so the right answer depends on the failure domain.

| Scenario | Best fit | Why |
|---|---|---|
| Small branch needs affordable connectivity | VPN | Lower cost and simpler deployment |
| Enterprise datacenter needs predictable private connectivity | ExpressRoute | Private path, better performance, stronger enterprise design |
| Many branches need centralized routing and onboarding | Virtual WAN | Managed hub model and simplified branch connectivity |
| Entire on-premises site must recover in Azure | ASR | Replication and orchestrated failover/failback |

Decision examples:

- If the issue is “users cannot reach Azure because the internet link failed,” use redundant ISPs and VPN/ExpressRoute redundancy.
- If the issue is “the datacenter is down,” use ASR for workload recovery and backup for data protection.
- If the issue is “a database was corrupted by a bad deployment,” use backup or point-in-time restore, not failover replication.
- If the issue is “a branch needs automatic path selection across multiple links,” use BGP with VPN or Virtual WAN.

**Common exam nuances**

- **PHS** usually gives better sign-in resilience than federation because it removes dependency on the on-premises authentication path.
- **PTA** can survive some agent failures if multiple agents are deployed, but it still depends on on-premises reachability.
- **AD FS** resilience depends on the whole federation stack, not just the AD FS server.
- **Backup** and **replication** are complementary, not interchangeable.
- **ASR** is for DR orchestration, not long-term backup retention.
- **ExpressRoute** improves connectivity resilience, but it does not replace application-level DR.
- **Zone-redundant gateways** are conditional on SKU and region support.
- **Failover testing** should include DNS, routing, authentication, and application dependencies, not only VM power state.

**Design checklist for hybrid resilience**

Use this checklist when evaluating a hybrid architecture:

- Identify the critical failure domains:
  - identity
  - network
  - compute
  - storage
  - application state
  - site-level outage
- Define RPO and RTO for each workload.
- Choose the right identity method:
  - PHS for highest sign-in resilience
  - PTA when on-prem validation is required
  - AD FS only when required by business constraints
- Build network redundancy:
  - dual ISPs
  - redundant firewalls and routers
  - diverse paths
  - VPN or ExpressRoute redundancy
- Protect data:
  - backup for recovery from corruption or deletion
  - replication for fast failover
- Plan DR:
  - ASR for supported workloads
  - documented failover and failback runbooks
- Test regularly:
  - planned failover
  - unplanned failover
  - failback
  - route convergence
  - application validation

**Summary**

Hybrid resilience is about matching the recovery method to the failure mode. Use identity patterns that reduce dependency on the on-premises path, design network redundancy at both Azure and datacenter edges, protect data with both backup and replication, and validate recovery with real failover tests. For AZ-305, the key is to choose the service that fits the outage scenario and to explain why that choice meets the required RPO and RTO.
