## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Core idea**

Azure gives you two common infrastructure-level resiliency patterns for virtual machines and related resources: **availability sets** and **availability zones**. Both reduce the impact of failures, but they protect against different failure domains.

- **Availability sets** protect against hardware and maintenance failures inside a single datacenter.
- **Availability zones** protect against datacenter-level failures within an Azure region.
- You choose based on the level of isolation your workload needs, the services you use, and the latency tolerance between instances.

**Availability sets**

An availability set is a logical grouping of VMs that tells Azure to spread them across separate physical infrastructure. Azure places VMs into different **fault domains** and **update domains** so that a single rack failure or planned maintenance event does not take down every instance at once.

- **Fault domain**: a group of hardware that shares a common power source and network switch.
- **Update domain**: a group of VMs that Azure updates together during planned maintenance.
- Use availability sets when you need high availability for VMs but do not need zone-level isolation.
- Availability sets are a good fit for legacy workloads, smaller deployments, and designs where zone support is unavailable.

A simple example is a two-VM web tier behind a load balancer. If one VM is rebooted for maintenance, the other can continue serving traffic.

```bash
az vm availability-set create \
  --resource-group rg-app \
  --name as-web \
  --platform-fault-domain-count 2 \
  --platform-update-domain-count 5
```

When you place VMs in an availability set, Azure automatically distributes them. You do not manually choose the fault domain or update domain for each VM.

**Availability zones**

Availability zones are physically separate locations within an Azure region. Each zone has independent power, cooling, and networking. They are designed so that a failure in one zone does not affect the others.

- Zones are typically labeled **Zone 1**, **Zone 2**, and **Zone 3** within a region.
- Use zones for higher resiliency than availability sets when the region supports them.
- Zone-aware services can be deployed across multiple zones for better uptime.
- Zone-to-zone traffic may add latency, so design for synchronous or asynchronous replication accordingly.

A common pattern is to deploy one VM in each zone and place them behind a zone-redundant load balancer or application gateway.

```bash
az vm create \
  --resource-group rg-app \
  --name vm-zone1 \
  --image Ubuntu2204 \
  --zone 1
```

> Availability zones are not available in every Azure region, and not every service supports zone deployment. Always verify regional and service support before designing for zone redundancy.

**Fault tolerance comparison**

| Capability | Availability set | Availability zone |
|---|---|---|
| Isolation scope | Hardware and maintenance within a datacenter | Entire datacenter within a region |
| Placement | Multiple fault and update domains | Separate physical zones |
| Typical use | VM resiliency inside one datacenter | Regional resiliency across datacenters |
| Failure protection | Rack or host failure, planned maintenance | Zone outage, datacenter outage |
| Latency between instances | Lower, because instances stay in one datacenter | Higher than same-datacenter placement |
| Service support | Broad for IaaS VMs | Depends on region and service |

**When to use each**

Use the simplest option that meets your availability target.

- Choose **availability sets** when:
  - You run IaaS VMs in a single datacenter footprint.
  - Your application is not zone-aware.
  - You need protection from host or rack failures and planned maintenance.
- Choose **availability zones** when:
  - You need stronger resiliency against datacenter loss.
  - Your region supports the required service in zones.
  - Your application can tolerate cross-zone latency and data replication design.

For many modern architectures, zones are the preferred choice for production workloads. Availability sets remain useful for older applications, smaller environments, or services that do not support zones.

**Design patterns**

A few common patterns help you map the concept to real deployments.

- **Single-region, multi-VM app**
  - Use an availability set for the web tier and another for the app tier.
  - Place a load balancer in front of each tier.
- **Zone-redundant app**
  - Deploy one instance per zone.
  - Use a zone-redundant load balancer or application gateway.
  - Replicate data across zones using a service that supports zone redundancy.
- **Hybrid legacy workload**
  - Keep the application in an availability set if the software assumes low-latency local communication.
  - Add backup and disaster recovery separately, because availability sets do not protect against region-wide outages.

**Practical Azure CLI examples**

Create an availability set and then deploy VMs into it.

```bash
az vm availability-set create \
  --resource-group rg-prod \
  --name as-prod \
  --platform-fault-domain-count 2 \
  --platform-update-domain-count 20
```

Create a VM in a specific zone.

```bash
az vm create \
  --resource-group rg-prod \
  --name vm-app-01 \
  --image Win2022Datacenter \
  --admin-username azureadmin \
  --generate-ssh-keys \
  --zone 2
```

Inspect VM placement and availability configuration.

```bash
az vm show \
  --resource-group rg-prod \
  --name vm-app-01 \
  --query "{zone:zones, availabilitySet:availabilitySet.id}" \
  -o json
```

**Operational considerations**

Availability design is not just about placement. You also need to think about how traffic and data move.

- Use a **load balancer** or application gateway so clients do not depend on a single VM.
- Ensure your data layer matches your compute resiliency model.
- Test failover behavior, not just deployment success.
- Monitor health probes, VM status, and application-level readiness.
- Remember that availability sets do not provide automatic disaster recovery across regions.

**Common exam distinctions**

AZ-305 questions often test whether you can identify the right resiliency primitive.

- If the question mentions **separate datacenters in a region**, think **availability zones**.
- If the question mentions **fault domains** and **update domains**, think **availability sets**.
- If the workload is a set of VMs and the goal is to survive host maintenance, availability sets are usually enough.
- If the workload must survive a datacenter outage, zones are the better answer.

**Quick decision guide**

- Need protection from host or rack failure only? Use **availability set**.
- Need protection from a datacenter outage? Use **availability zone**.
- Need both regional resiliency and global continuity? Combine zones with backup, replication, and disaster recovery planning.

**Key takeaway**

Availability sets improve VM resiliency inside a datacenter by spreading instances across fault and update domains. Availability zones go further by spreading workloads across physically separate datacenters in the same region. For AZ-305, choose the option that matches the failure scope you need to survive.
