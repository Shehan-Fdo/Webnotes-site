## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Regional resiliency overview**

Regional resiliency is the ability of a solution to continue operating, or recover quickly, when an Azure region experiences an outage, degradation, or disaster. In Azure architecture, this usually means designing across two or more regions, not just across availability zones within one region.

A good design starts with the business requirements:

- **RPO**: Recovery Point Objective, the maximum acceptable data loss measured in time.
- **RTO**: Recovery Time Objective, the maximum acceptable downtime before service restoration.
- **Data residency**: Where data is allowed to live, replicate, and be processed.
- **Service scope**: Whether the Azure service is global, regional, or zone-aware.

Use region pairs when you need a Microsoft-supported regional pairing model for disaster recovery, platform update coordination, and geography-aware resiliency planning. Use other multi-region patterns when the service or business requirement needs more control than the pair model provides.

**Azure region pairs**

Azure pairs regions within the same geography for resiliency and operational coordination. The pair is a Microsoft-defined relationship, not something you infer from distance alone.

Key exam points:

- Paired regions are in the same geography, such as the same country or geopolitical boundary where Azure defines one.
- Microsoft commonly describes paired regions as typically within about 300 miles or 500 km when applicable, but this is not a guarantee and should not be treated as the defining rule.
- Platform updates are staggered across paired regions so that one region can be prioritized while the other remains available.
- If a broad outage affects one region, Microsoft may prioritize recovery of the paired region to help restore services.
- Region pairs are used by some Azure services as a default geo-redundancy or failover target, but not every service uses them the same way.

> Do not use physical distance as the primary design rule. The important exam concept is same-geography pairing plus coordinated platform resiliency.

**Why region pairs matter**

Region pairs help you design for:

- **Operational continuity** during platform maintenance or regional incidents.
- **Geo-redundant storage** and backup placement within a geography.
- **Compliance alignment** when data must remain inside a specific geography.
- **Predictable DR planning** because Microsoft publishes the pair relationship.

They do not replace application design. You still need:

- Stateless app tiers where possible.
- Data replication strategy.
- Traffic failover logic.
- Tested recovery procedures.

**Region pairs and geography boundaries**

Azure geography boundaries matter for legal and regulatory reasons. A geography can represent a country, a sovereign cloud boundary, or another Microsoft-defined compliance boundary. When you choose a paired-region design, you should confirm that replication and failover stay within the required geography.

Use this approach when compliance matters:

- Keep primary and secondary resources in the same approved geography if regulations require it.
- Avoid assuming that a paired region automatically satisfies residency rules for every workload.
- Validate service-specific data replication behavior, because some services replicate metadata or diagnostics differently from customer data.

**Region pairs and platform-managed behaviors**

Some Azure services use paired regions behind the scenes for resiliency or geo-redundancy defaults. The exact behavior depends on the service.

Common examples include:

- **Azure Storage** geo-redundant options that replicate to the paired region.
- **Azure Backup** vault redundancy options that can use geo-redundant storage.
- **Azure Site Recovery** replication targets that are often chosen in a different region based on DR design, not necessarily only the paired region.
- **Platform update sequencing** that reduces simultaneous impact across paired regions.

Do not assume every service automatically fails over to the paired region. Many services require explicit configuration, and some require you to build the failover path yourself.

**Regional resiliency patterns**

There are two broad patterns for multi-region design: active-active and active-passive.

| Pattern | Description | Strengths | Tradeoffs |
|---|---|---|---|
| Active-active | Two regions serve traffic at the same time | Better availability, lower failover time, can absorb regional load | More complex data consistency, routing, and cost |
| Active-passive | One region serves traffic while the other waits | Simpler operations, easier data control, lower steady-state cost | Failover takes longer, passive region may be under-tested |

Choose based on business goals:

- Use **active-active** when you need high availability, global performance, or can tolerate distributed consistency design.
- Use **active-passive** when you need simpler failover, strict control over writes, or lower cost.
- Use a **warm standby** when you want a partially running secondary environment that can scale up quickly.
- Use a **cold standby** when cost matters more than recovery speed.

For AZ-305, explain the tradeoff in terms of RPO and RTO:

- Active-active usually supports lower RTO and potentially lower effective RPO, but it increases complexity.
- Active-passive is easier to reason about, but failover time and data lag are usually higher.
- The right answer depends on business criticality, not on a universal best practice.

**Azure Site Recovery**

Azure Site Recovery (ASR) is a core disaster recovery service for regional resiliency. It replicates workloads to another Azure region or to another site, then supports failover and failback operations.

ASR concepts:

- **Replication**: Continuous or near-continuous copy of VM data to the recovery region.
- **Test failover**: A non-disruptive validation of the recovery plan.
- **Planned failover**: Controlled failover during maintenance or before an expected outage.
- **Unplanned failover**: Recovery after an outage or disaster.
- **Failback**: Returning workloads to the primary site after recovery.

Use ASR when you need:

- VM-based disaster recovery.
- Recovery plans with ordered startup of application tiers.
- Validation of DR readiness without impacting production.
- A defined recovery workflow for infrastructure workloads.

Practical example:

```bash
az site-recovery fabric list --resource-group rg-dr --vault-name drVault
az site-recovery protection-container list --resource-group rg-dr --vault-name drVault --fabric-name eastus-fabric
```

ASR is especially useful when your application is not natively multi-region aware. It gives you infrastructure-level recovery, but you still need to validate application dependencies, DNS, and data consistency.

**Recovery planning with RPO and RTO**

RPO and RTO drive the architecture.

- **RPO** answers: How much data can you lose?
- **RTO** answers: How long can the service be unavailable?

Design implications:

- If RPO must be near zero, you need synchronous or near-synchronous replication, which usually increases latency and complexity.
- If RTO must be very low, you need pre-provisioned secondary capacity and automated traffic switching.
- If both are strict, you may need active-active design, database replication, and global traffic management.
- If both are moderate, active-passive with ASR or managed failover may be enough.

A simple decision rule:

- Low cost, moderate downtime tolerance: active-passive.
- High availability, low downtime tolerance: active-active.
- Infrastructure recovery focus: ASR.
- Managed PaaS failover: service-native geo-replication or failover groups.

**Azure SQL regional resiliency**

Azure SQL has multiple resiliency options, and the right choice depends on whether you use SQL Database or SQL Managed Instance.

For **Azure SQL Database**:

- **Active geo-replication** creates readable secondary databases in another region.
- **Auto-failover groups** provide managed failover for one or more databases and can expose a readable secondary endpoint.
- Use active geo-replication when you want more direct control over replication relationships and failover timing.
- Use auto-failover groups when you want a managed failover experience and a stable listener endpoint.

For **Azure SQL Managed Instance**:

- Failover groups support instance-level disaster recovery.
- Managed Instance failover groups have networking and DNS considerations that differ from SQL Database.
- Plan private connectivity, routing, and name resolution carefully before you rely on failover.

Example CLI pattern for failover groups:

```bash
az sql server create \
  --resource-group rg-sql \
  --name sql-primary-01 \
  --location eastus \
  --admin-user sqladmin \
  --admin-password 'Use-A-Strong-Password!'

az sql server create \
  --resource-group rg-sql \
  --name sql-secondary-01 \
  --location westus2 \
  --admin-user sqladmin \
  --admin-password 'Use-A-Strong-Password!'

az sql failover-group create \
  --resource-group rg-sql \
  --server sql-primary-01 \
  --name fog-app01 \
  --partner-server sql-secondary-01 \
  --failover-policy Automatic \
  --grace-period 1 \
  --add-db appdb
```

Notes:

- Create the secondary server first.
- Add the database names explicitly when configuring the failover group.
- Validate the current Azure CLI syntax for your version, because parameter names can vary slightly across releases.

**Azure Storage resiliency**

Azure Storage offers redundancy options that are central to regional resiliency.

Common redundancy models:

- **LRS**: Locally redundant storage. Stores three copies within a single datacenter in the primary region.
- **ZRS**: Zone-redundant storage. Spreads copies across availability zones in the same region.
- **GRS**: Geo-redundant storage. Replicates data asynchronously to a paired secondary region.
- **RA-GRS**: Read-access geo-redundant storage. Adds read access to the secondary copy.
- **GZRS**: Geo-zone-redundant storage. Combines zone redundancy in the primary region with geo-replication to the paired region.
- **RA-GZRS**: Read-access geo-zone-redundant storage.

Use these options like this:

- Choose **LRS** for lowest cost when regional loss is acceptable or data is easily recreated.
- Choose **ZRS** when you want protection from a zone failure within one region.
- Choose **GRS** or **RA-GRS** when you need regional disaster recovery.
- Choose **GZRS** or **RA-GZRS** when you want both zone resilience and geo-resilience.

Example storage account creation:

```bash
az storage account create \
  --name stappresil01 \
  --resource-group rg-storage \
  --location eastus \
  --sku Standard_GZRS \
  --kind StorageV2
```

Important distinctions:

- LRS protects against local hardware failure, not region loss.
- ZRS protects against zone failure, not region loss by itself.
- Geo-redundant options replicate to the paired region asynchronously, so there can be replication lag.
- Read-access options let you read from the secondary endpoint, which can help with DR validation or read-heavy scenarios.

**Traffic routing and failover**

Regional resiliency is incomplete without traffic management. You need a way to direct users to the healthy region.

Common Azure options:

- **Azure Traffic Manager**: DNS-based traffic routing across endpoints.
- **Azure Front Door**: Global HTTP/HTTPS load balancing and application acceleration.
- **Azure Load Balancer**: Regional load balancing, not a cross-region failover service by itself.

Use Traffic Manager when:

- You need DNS-based failover across regions.
- Your application can tolerate DNS propagation and caching behavior.
- You want routing methods such as priority, performance, or weighted distribution.

Use Front Door when:

- You need global anycast entry, fast failover, and layer 7 routing.
- You want WAF (Web Application Firewall) integration and path-based routing.
- You are fronting web applications or APIs.

Example Traffic Manager profile creation:

```bash
az network traffic-manager profile create \
  --resource-group rg-tm \
  --name tm-app01 \
  --routing-method Priority \
  --unique-dns-name tm-app01-az305 \
  --ttl 30
```

Example endpoint addition:

```bash
az network traffic-manager endpoint create \
  --resource-group rg-tm \
  --profile-name tm-app01 \
  --name ep-eastus \
  --type azureEndpoints \
  --target-resource-id /subscriptions/<subId>/resourceGroups/rg-app/providers/Microsoft.Web/sites/app-eastus \
  --endpoint-status Enabled \
  --priority 1
```

Practical cautions:

- DNS-based failover is not instant because clients and resolvers cache records.
- Lower TTL can help, but it does not eliminate caching.
- Health probes must be configured correctly or failover will not happen as expected.
- For web workloads, Front Door often provides a better user experience than DNS-only failover.

**App Service and regional resiliency**

Azure App Service is a regional service, so you usually design resiliency around multiple App Service instances in different regions plus a global traffic layer.

Important points:

- App Service itself is deployed in a region.
- Zone redundancy is available only for specific plans and supported regions.
- You should not assume every App Service plan supports zone redundancy.
- For regional resiliency, pair App Service with Traffic Manager or Front Door.
- Store state outside the app tier, such as in Azure SQL, Storage, or a cache with a recovery plan.

A common pattern:

- Deploy the app to two regions.
- Put Front Door or Traffic Manager in front.
- Use a replicated database or failover group.
- Keep configuration in Key Vault or App Configuration with a multi-region design.

**Azure Key Vault and multi-region design**

Azure Key Vault is region-scoped, so multi-region resiliency requires explicit design. Do not assume transparent cross-region replication of secrets, keys, or certificates.

Design options:

- Deploy separate vaults in each region.
- Replicate secrets through your deployment pipeline or automation.
- Use managed identities and access policies or RBAC consistently across regions.
- Plan for key rotation and certificate renewal in both regions.

Example deployment pattern:

```bash
az keyvault create \
  --name kv-app-eastus \
  --resource-group rg-sec \
  --location eastus

az keyvault create \
  --name kv-app-westus2 \
  --resource-group rg-sec \
  --location westus2
```

Operationally, you should:

- Sync secrets through CI/CD or automation.
- Test application startup against the secondary vault.
- Avoid hard-coding region-specific secret URIs unless your failover logic handles them.

**Global vs regional services**

Some Azure services are global, while others are regional. This distinction affects whether region pairs are relevant.

Examples:

- **Global services**: Azure Front Door, Traffic Manager, Microsoft Entra ID.
- **Regional services**: Azure SQL Database, App Service, Key Vault, Virtual Machines.
- **Hybrid or mixed**: Some services have global control planes but regional data planes.

Why this matters:

- Global services often reduce the need for you to design your own cross-region control plane.
- Regional services require explicit multi-region deployment and failover planning.
- Some global services still depend on regional backends, so you must understand the service architecture rather than assuming “global” means “no regional dependency.”

**Testing and validation**

A resiliency design is not complete until you test it.

Test activities:

1. Run a **test failover** for ASR or a non-production recovery plan.
2. Validate DNS and traffic routing behavior.
3. Confirm application startup order and dependency readiness.
4. Check data consistency after failover.
5. Perform a controlled **failback** and verify the original region can resume service.

Useful validation checks:

- Application health endpoints return success.
- Database connections resolve to the intended endpoint.
- Secrets and certificates are available in the recovery region.
- Monitoring and alerting continue to function after failover.
- Runbooks are current and executable by the operations team.

Example validation commands:

```bash
az network traffic-manager endpoint show \
  --resource-group rg-tm \
  --profile-name tm-app01 \
  --name ep-eastus
```

For ASR, validate recovery plans in a test network so you do not impact production IP ranges or routing.

**Common design mistakes**

- Treating region pairs as a substitute for application design.
- Assuming distance alone defines resiliency.
- Using DNS failover without accounting for caching and TTL.
- Replicating data but forgetting secrets, certificates, or identity dependencies.
- Designing active-active without handling write conflicts.
- Assuming every service supports the same failover model.
- Ignoring compliance boundaries when selecting paired regions.
- Not testing failover and failback before go-live.

**Exam-focused guidance**

When you answer AZ-305 questions, look for these clues:

- If the question emphasizes **minimal downtime**, consider active-active, Front Door, or managed failover.
- If it emphasizes **lower cost** and acceptable downtime, consider active-passive.
- If it emphasizes **VM recovery**, think Azure Site Recovery.
- If it emphasizes **database continuity**, think SQL failover groups or geo-replication.
- If it emphasizes **storage durability**, think GRS, RA-GRS, GZRS, or RA-GZRS.
- If it emphasizes **compliance geography**, verify that the design stays within the required Azure geography.
- If it emphasizes **platform maintenance resilience**, remember that Azure uses paired regions to stagger updates.

**Quick reference**

- **Region pair**: Microsoft-defined same-geography pairing for resiliency and operational coordination.
- **RPO**: Acceptable data loss window.
- **RTO**: Acceptable downtime window.
- **ASR**: VM and site disaster recovery service.
- **GRS/GZRS**: Storage options that replicate to a paired region.
- **Traffic Manager / Front Door**: Cross-region traffic routing options.
- **Active-active**: Both regions serve traffic.
- **Active-passive**: One region serves, one waits.

**Practical architecture example**

A common resilient web application design looks like this:

- Front Door routes users to the nearest healthy region.
- App Service runs in two regions.
- Azure SQL Database uses auto-failover groups.
- Storage uses GZRS or RA-GZRS.
- Key Vault is deployed separately in each region.
- Monitoring alerts on health probe failures and replication lag.
- Runbooks define failover, validation, and failback steps.

This pattern balances availability, recovery speed, and operational complexity. It is a strong exam answer when the scenario requires both regional resiliency and user-facing performance.
