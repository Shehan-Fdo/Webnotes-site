## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Resilience design goals**

Application resilience is the ability to keep delivering acceptable service when components fail, degrade, or become unavailable. In Azure architecture questions, you usually balance three objectives:

- **Availability**: the service stays reachable.
- **Reliability**: the service continues to function correctly under failure.
- **Recoverability**: the service can be restored within target time and data-loss limits.

Two planning metrics drive most design choices:

- **RTO** (<abbr title="Recovery Time Objective">RTO</abbr>): how long recovery can take.
- **RPO** (<abbr title="Recovery Point Objective">RPO</abbr>): how much data loss is acceptable.

A resilient design does not try to eliminate failure. It assumes failure will happen and limits blast radius, detects issues quickly, and fails over in a controlled way.

**Core resilience patterns**

Azure solutions commonly combine several patterns rather than relying on one service feature alone.

- **Redundancy**: duplicate critical components so one failure does not stop the workload.
- **Health-based routing**: send traffic only to healthy endpoints.
- **Retry with backoff**: recover from transient faults without overwhelming dependencies.
- **Circuit breaker**: stop calling a failing dependency temporarily.
- **Bulkhead isolation**: separate resources so one workload cannot exhaust all capacity.
- **Graceful degradation**: return partial functionality instead of total failure.
- **Partitioning and sharding**: split data or traffic to reduce contention and failure impact.
- **Failover**: switch to a secondary instance, region, or path when the primary fails.

These patterns work best when you apply them at multiple layers: client, application, platform, and data.

**Azure Front Door for global entry points**

<abbr title="Azure Front Door Standard and Premium">Azure Front Door</abbr> is a global layer 7 entry point for internet-facing applications. Use it when you need fast failover, global routing, TLS termination, and web application firewall capabilities at the edge.

Front Door Standard/Premium supports:

- **Origin groups**: a logical set of backends with health probes and priority/weight routing.
- **Health probes**: periodic checks that determine whether an origin is healthy.
- **Session affinity**: sticky routing when you need a user to stay on the same origin.
- **TLS termination**: HTTPS is decrypted at the edge, reducing backend overhead.
- **WAF** (<abbr title="Web Application Firewall">Web Application Firewall</abbr>): protects against common web attacks.

Use Front Door when you want active-active or active-passive routing across regions.

- **Active-active**: traffic is distributed across multiple healthy origins at the same time.
- **Active-passive**: one origin receives traffic until it fails, then another takes over.

A practical origin group example:

```json
{
  "name": "app-origin-group",
  "properties": {
    "healthProbeSettings": {
      "probePath": "/health",
      "probeProtocol": "Https",
      "probeRequestType": "GET",
      "probeIntervalInSeconds": 30
    },
    "loadBalancingSettings": {
      "sampleSize": 4,
      "successfulSamplesRequired": 3,
      "additionalLatencyInMilliseconds": 50
    },
    "origins": [
      {
        "name": "eastus-app",
        "properties": {
          "hostName": "app-eastus.contoso.com",
          "priority": 1,
          "weight": 100
        }
      },
      {
        "name": "westus-app",
        "properties": {
          "hostName": "app-westus.contoso.com",
          "priority": 2,
          "weight": 100
        }
      }
    ]
  }
}
```

Important exam nuance:

- Front Door makes routing decisions based on origin health and routing rules.
- It does **not** place your backends into zones or regions for you.
- Backend placement, replication, and zone distribution are your responsibility.

**Application Gateway for regional web traffic**

Application Gateway is a regional layer 7 load balancer for HTTP/HTTPS workloads. It is useful when you need path-based routing, TLS offload, cookie-based affinity, or integration with a regional WAF.

Use it when:

- Your users are in one region or you already have global routing elsewhere.
- You need application-aware routing inside a region.
- You want to route to multiple backend pools based on host or path.

A health probe example:

```json
{
  "name": "appgw-probe",
  "properties": {
    "protocol": "Https",
    "path": "/health",
    "interval": 30,
    "timeout": 30,
    "unhealthyThreshold": 3,
    "pickHostNameFromBackendHttpSettings": true,
    "match": {
      "statusCodes": ["200-399"]
    }
  }
}
```

Use this as a conceptual example, not a complete deployment template. In real deployments, you also define listeners, backend HTTP settings, routing rules, and the gateway resource itself.

Key points:

- **Probe path** should be lightweight and dependency-aware.
- **Host header** handling matters when the backend expects a specific hostname.
- **Cookie affinity** can help stateful web sessions, but it is not a substitute for shared session state.

**Azure Load Balancer for TCP and UDP resilience**

Azure Load Balancer is a layer 4 service for TCP and UDP traffic. It is appropriate for non-HTTP workloads, internal service tiers, and scenarios where you need high throughput with low latency.

Use it when:

- You need to balance traffic for custom protocols.
- You want internal load balancing between application tiers.
- You need a public or private frontend for virtual machines or VM scale sets.

Important design detail:

- A **zone-redundant Standard Load Balancer** frontend can be zone-redundant.
- Backend instances are not automatically spread across zones by the load balancer itself.
- You still need to design backend placement across zones using VM placement, availability zones, or VM scale set configuration.

Health probes are essential:

```bash
az network lb probe create \
  --resource-group rg-app \
  --lb-name lb-app \
  --name tcp-probe \
  --protocol tcp \
  --port 8080
```

A load balancer only routes to healthy instances. If all backends fail health checks, traffic stops rather than being sent to unhealthy nodes.

**App Service resilience features**

Azure App Service provides platform-managed web hosting, but resilience still depends on how you deploy and configure the app.

Useful resilience features include:

- **Deployment slots**: stage a new version, warm it up, then swap it into production.
- **Health check endpoint**: App Service can remove unhealthy instances from rotation.
- **Autoscale**: add instances under load to reduce saturation failures.
- **Zone redundancy**: available in some App Service plans and regions, but not universally.
- **Backup and restore**: useful for configuration and content recovery.

Deployment slots are especially important for reducing deployment risk.

1. Deploy to a staging slot.
2. Validate the app and dependencies.
3. Warm up the slot.
4. Swap into production.
5. Keep the previous version available for rollback.

Example CLI:

```bash
az webapp deployment slot create \
  --resource-group rg-app \
  --name contoso-web \
  --slot staging

az webapp deployment slot swap \
  --resource-group rg-app \
  --name contoso-web \
  --slot staging \
  --target-slot production
```

Use slots when you need near-zero-downtime releases and fast rollback. They are not a substitute for multi-region disaster recovery.

**AKS resilience considerations**

Azure Kubernetes Service (<abbr title="Azure Kubernetes Service">AKS</abbr>) resilience depends on both the cluster and the workload design.

At the cluster level:

- Use multiple node pools when you need workload isolation.
- Spread nodes across availability zones when the region supports it.
- Use cluster autoscaler and node auto-provisioning where appropriate.
- Keep system and user workloads separated when possible.

At the workload level:

- Set **liveness probes** to restart dead containers.
- Set **readiness probes** to keep unready pods out of service.
- Set **startup probes** for slow-starting containers so they are not killed too early.

Probe behavior matters:

- **Liveness** answers: should this container be restarted?
- **Readiness** answers: should this pod receive traffic?
- **Startup** answers: is the application still starting up?

Example Kubernetes manifest:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: orders-api
spec:
  replicas: 3
  selector:
    matchLabels:
      app: orders-api
  template:
    metadata:
      labels:
        app: orders-api
    spec:
      containers:
      - name: orders-api
        image: contoso/orders-api:1.0.0
        ports:
        - containerPort: 8080
        startupProbe:
          httpGet:
            path: /health/startup
            port: 8080
          failureThreshold: 30
          periodSeconds: 10
        readinessProbe:
          httpGet:
            path: /health/ready
            port: 8080
          periodSeconds: 5
        livenessProbe:
          httpGet:
            path: /health/live
            port: 8080
          periodSeconds: 10
```

Resilience caveats:

- More replicas improve availability, but only if they are spread across failure domains.
- A single-node pool or single-zone design creates a hidden single point of failure.
- Stateful workloads need storage and data-layer resilience, not just pod restarts.

**Azure SQL Database and failover groups**

Azure SQL Database resilience is usually built with geo-replication and failover groups for cross-region disaster recovery.

A failover group provides:

- A **read-write listener** for application connections.
- A **read-only listener** for read-only workloads.
- Automatic or manual failover between primary and secondary regions.

Important limitations and behaviors:

- Automatic failover policy is configurable, but it is not a substitute for testing.
- Client applications should connect through the listener endpoints, not hard-coded server names.
- During failover, connection drops are expected; the client must retry.
- Read scale-out is not a general-purpose feature for every tier and scenario; it is mainly associated with Hyperscale read replicas/read scale-out.

Example failover group creation:

```bash
az sql failover-group create \
  --resource-group rg-data \
  --server primary-sql \
  --name fog-contoso \
  --partner-server secondary-sql \
  --failover-policy Automatic \
  --grace-period 60
```

Connection string guidance:

- Use the **read-write listener** for transactional workloads.
- Use the **read-only listener** only for queries that can tolerate replica lag.
- Keep retry logic in the application because failover causes transient connection failures.

**Cosmos DB resilience and consistency**

Azure Cosmos DB is designed for global distribution, but resilience depends on region configuration and consistency choices.

Key configuration concepts:

- **Preferred regions**: the order Cosmos DB uses for reads and failover preference.
- **Failover priorities**: determine which region becomes primary during failover.
- **Consistency levels**: trade off latency, throughput, and staleness.

Common consistency levels:

- **Strong**: highest consistency, higher latency, and more constraints.
- **Bounded staleness**: bounded lag with predictable consistency.
- **Session**: common default for user-centric apps; good balance of consistency and performance.
- **Consistent prefix**: preserves write order.
- **Eventual**: lowest guarantees, highest flexibility.

Practical guidance:

- Use stronger consistency only when the business requirement justifies the latency and throughput cost.
- Use session consistency for most interactive applications.
- Design for regional failover by setting preferred regions and testing failover behavior.

Example CLI:

```bash
az cosmosdb update \
  --name contoso-cosmos \
  --resource-group rg-data \
  --locations regionName=eastus failoverPriority=0 isZoneRedundant=True \
              regionName=westus failoverPriority=1 isZoneRedundant=True
```

**Azure Storage resilience**

Azure Storage services are strongly consistent within a region for read-after-write behavior in normal operation. The main resilience concerns are around redundancy choices and geo-failover, not ordinary in-region reads.

Redundancy options:

- **LRS** (<abbr title="Locally Redundant Storage">LRS</abbr>): three copies in one datacenter or stamp.
- **ZRS** (<abbr title="Zone-Redundant Storage">ZRS</abbr>): copies across availability zones in a region.
- **GRS** (<abbr title="Geo-Redundant Storage">GRS</abbr>): copies to a paired region.
- **GZRS** (<abbr title="Geo-Zone-Redundant Storage">GZRS</abbr>): zone redundancy in the primary region plus geo-replication.

Use cases:

- **LRS**: lowest cost, acceptable when regional loss is not a concern.
- **ZRS**: better protection against zone failure.
- **GRS/GZRS**: better disaster recovery posture when regional recovery matters.

Important note:

> Geo-replication introduces replication lag. Plan for possible data staleness during failover or read access to secondary endpoints.

**Service Bus resilience and messaging tradeoffs**

Azure Service Bus helps decouple components so one failure does not cascade through the system.

Resilience features include:

- **Queues and topics**: buffer work during downstream outages.
- **Dead-letter queues**: isolate poison messages.
- **Duplicate detection**: reduce duplicate processing after retries or replay.
- **Sessions**: preserve ordering and enable session-aware processing.

Tradeoffs:

- **Duplicate detection** helps when clients may resend messages, but it requires a detection window and message IDs.
- **Sessions** preserve order within a session, but they can become a bottleneck if too much traffic is funneled into one session key.
- Session-based processing can reduce parallelism and complicate failover if a session lock is held by a failed processor.

Practical guidance:

- Use sessions only when ordering or affinity is required.
- Use duplicate detection when at-least-once delivery could create harmful duplicates.
- Keep message handlers idempotent so retries do not corrupt state.

**Retries, backoff, and circuit breakers**

Transient faults are normal in cloud systems. Your application should retry only when the operation is safe to repeat and the failure is likely temporary.

Recommended patterns:

- **Exponential backoff**: increase delay between retries.
- **Jitter**: randomize retry timing to avoid synchronized spikes.
- **Circuit breaker**: stop calling a failing dependency after repeated failures.
- **Timeouts**: fail fast instead of waiting indefinitely.

Example with Polly in .NET:

```csharp
var retryPolicy = Policy
    .Handle<HttpRequestException>()
    .Or<TaskCanceledException>()
    .WaitAndRetryAsync(
        retryCount: 3,
        sleepDurationProvider: attempt => TimeSpan.FromMilliseconds(200 * Math.Pow(2, attempt)),
        onRetry: (exception, delay, attempt, context) =>
        {
            Console.WriteLine($"Retry {attempt} after {delay.TotalMilliseconds}ms");
        });

var circuitBreaker = Policy
    .Handle<HttpRequestException>()
    .CircuitBreakerAsync(
        exceptionsAllowedBeforeBreaking: 5,
        durationOfBreak: TimeSpan.FromSeconds(30));
```

Azure SDKs also expose retry settings. Use them instead of writing custom retry logic when the SDK already handles transient faults.

Example Azure SDK-style configuration:

```csharp
var clientOptions = new BlobClientOptions
{
    Retry =
    {
        Mode = RetryMode.Exponential,
        MaxRetries = 5,
        Delay = TimeSpan.FromSeconds(0.8),
        MaxDelay = TimeSpan.FromSeconds(8)
    }
};
```

**Graceful degradation and fallback design**

A resilient app should fail partially before it fails completely.

Common fallback strategies:

- Return cached data when the primary data source is unavailable.
- Disable nonessential features such as recommendations or analytics.
- Serve a static maintenance page when the dynamic app is down.
- Reduce image quality or expensive computations under load.

Design principles:

- Prioritize core user journeys.
- Make degraded behavior explicit to users.
- Keep fallback paths simple and tested.

Example pattern:

- If the product catalog API fails, show the last cached catalog.
- If personalization fails, show generic recommendations.
- If checkout is unavailable, preserve the cart and ask the user to retry later.

**Sharding, partitioning, and blast-radius reduction**

Partitioning improves both scale and resilience by limiting the impact of a failure.

Use partitioning when:

- A single database or queue becomes a bottleneck.
- You need to isolate tenants, regions, or business units.
- You want to reduce the size of a failover domain.

Examples:

- Partition a database by customer or region.
- Use separate Service Bus topics for different workloads.
- Split storage accounts by application or environment.
- Use multiple AKS node pools for different workload classes.

Tradeoff:

- More partitions increase operational complexity.
- Cross-partition queries and transactions become harder.

**Operational resilience: runbooks and failover testing**

Resilience is not complete until you test it.

Your operational plan should include:

- **Disaster recovery runbooks**: documented steps for failover and failback.
- **Failover tests**: scheduled validation of regional and zonal recovery.
- **Dependency tests**: verify that DNS, identity, storage, and messaging recover as expected.
- **RTO/RPO validation**: measure actual recovery against business targets.

A practical runbook should answer:

1. What failed?
2. What is the blast radius?
3. Which dependency is the source of truth?
4. How do you fail over?
5. How do you validate service health?
6. How do you fail back safely?

Do not assume a design is resilient because it has redundancy. Test the full path, including client retries, DNS propagation, connection string updates, and operator procedures.

**Common exam scenarios and how to think about them**

When you see a resilience question, identify the failure domain first.

- **Single instance failure**: use replicas, health probes, and autoscale.
- **Zone failure**: use zone-redundant services and spread backends across zones.
- **Regional failure**: use multi-region deployment, Front Door, and geo-redundant data.
- **Dependency outage**: use retries, circuit breakers, queues, and graceful degradation.
- **Deployment failure**: use slots, canary releases, and rollback paths.

A good Azure architecture usually combines:

- Global traffic management at the edge.
- Regional load balancing inside the region.
- Redundant compute across zones or regions.
- Replicated data with tested failover.
- Application-level retry and fallback logic.
- Operational runbooks and regular drills.

**Quick comparison of common patterns**

| Pattern | Best for | Key limitation |
|---|---|---|
| Front Door Standard/Premium | Global HTTP(S) routing and failover | Does not manage backend placement |
| Application Gateway | Regional HTTP(S) routing | Regional scope only |
| Load Balancer | TCP/UDP and non-HTTP workloads | No application-layer awareness |
| App Service slots | Safe deployments and rollback | Not a DR solution |
| AKS probes | Pod health and traffic gating | Must be designed correctly per workload |
| SQL failover groups | Cross-region database DR | Client retry behavior still required |
| Cosmos DB multi-region | Global distribution and failover | Consistency choice affects latency and cost |
| Service Bus sessions | Ordered processing | Can reduce throughput and increase bottlenecks |

**Practical checklist**

- Define RTO and RPO before choosing services.
- Place compute across zones or regions based on the failure domain.
- Use Front Door Standard/Premium for global web entry points.
- Use Application Gateway for regional HTTP routing.
- Use Load Balancer for TCP/UDP backends.
- Add health probes at every layer that can fail independently.
- Use App Service deployment slots for controlled releases.
- Configure AKS liveness, readiness, and startup probes correctly.
- Use SQL failover groups with listener-based connection strings.
- Set Cosmos DB preferred regions and test failover.
- Choose Storage redundancy based on zone and region recovery needs.
- Use Service Bus duplicate detection and sessions only when the tradeoff is justified.
- Implement retries, circuit breakers, and timeouts in the application.
- Write and test disaster recovery runbooks regularly.
