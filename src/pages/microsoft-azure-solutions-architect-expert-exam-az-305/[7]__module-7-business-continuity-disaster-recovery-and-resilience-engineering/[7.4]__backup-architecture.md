## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Backup architecture overview**

Azure backup architecture is about more than copying data. You design for recoverability, retention, compliance, operational simplicity, and cost. The right design depends on the workload, the restore objective, the retention period, and whether the service supports native backup, snapshot-based protection, or agent-based protection.

A good architecture answers four questions:

- What are you protecting?
- How fast must you restore?
- How long must you retain data?
- What failure domains must the backup survive?

For AZ-305, you should map each workload to the correct protection pattern instead of assuming one backup service fits everything.

**Core design goals**

Backup design usually balances these goals:

- **Recovery point objective (RPO)**: how much data loss is acceptable.
- **Recovery time objective (RTO)**: how long recovery can take.
- **Retention**: how long backups must remain available.
- **Isolation**: how well backup data is protected from deletion, corruption, or ransomware.
- **Operational simplicity**: how easy it is to manage policies, monitoring, and restores.
- **Cost**: storage redundancy, retention length, and restore frequency all affect cost.

In practice, you often combine multiple mechanisms:

- Native service backup for PaaS data services.
- Azure Backup for supported IaaS and platform workloads.
- Snapshots for fast local recovery.
- Replication or geo-redundancy for disaster recovery.
- Application-level export or replication for workloads that do not have full backup support.

**Vault types and workload mapping**

Azure uses different vault types for different backup scenarios. The distinction matters on the exam because the vault determines which workloads you can protect and which features you can use.

| Capability | Recovery Services vault | Backup vault |
|---|---|---|
| Primary purpose | Classic Azure Backup and Site Recovery scenarios | Microsoft Data Protection-supported workloads |
| Common workloads | Azure VMs, Azure Files, SQL in Azure VM, SAP HANA in Azure VM, MARS/MABS/DPM scenarios | Supported modern workloads such as Azure Disk Backup and selected data protection scenarios |
| Policy model | Workload-specific backup policies | Data protection policies for supported workloads |
| Restore support | Broad for classic backup workloads | Depends on workload support in Microsoft Data Protection |
| Exam takeaway | Required for many classic Azure Backup scenarios | Not a generic replacement for Recovery Services vault |

A common design mistake is assuming the newer vault replaces the older one for all workloads. It does not. You choose the vault based on workload support.

> Use the vault that matches the workload. Do not design by vault preference alone.

**Backup Center and governance**

For enterprise environments, Azure Backup Center provides centralized visibility across vaults, subscriptions, and workloads. It helps you standardize governance and monitor protection posture.

Use it to:

- View backup inventory across multiple vaults.
- Track protected and unprotected resources.
- Monitor backup jobs and alerts.
- Apply governance at scale with Azure Policy and role-based access control (RBAC).
- Identify compliance gaps, such as missing backup policies or expired recovery points.

A typical enterprise pattern is:

- Central platform team owns backup standards.
- Application teams own workload-specific policy assignment.
- Security team reviews retention, immutability, and access controls.
- Operations team validates restore readiness and job health.

**Backup policy design**

Backup policies define when backups run and how long recovery points are retained. Policy design is workload-specific, and the limits vary by service.

Important design considerations:

- **Schedule limits**: some workloads support multiple backups per day, while others support only daily schedules.
- **Retention limits**: short-term and long-term retention options differ by workload.
- **Granularity**: some services support hourly or sub-daily recovery points; others only daily.
- **Immutable retention**: some vault settings can prevent deletion or modification for a period.
- **Cross-region options**: not every workload supports cross-region restore or geo-redundant recovery in the same way.

For example:

- Azure VM backup policies support daily backup schedules and retention settings for recovery points.
- Azure Files backup supports scheduled snapshots and retention policies for file shares.
- Azure SQL Database uses automated backups with point-in-time restore rather than a traditional user-managed backup schedule.
- Azure Cosmos DB uses continuous backup or periodic backup modes depending on the account configuration.

**Backup redundancy and data durability**

Backup storage redundancy affects how backup data survives infrastructure failures. The choice is not just about cost; it affects recovery options and compliance posture.

Common redundancy options include:

- **Locally redundant storage (LRS)**: replicates data within a single datacenter.
- **Zone-redundant storage (ZRS)**: replicates across availability zones in a region.
- **Geo-redundant storage (GRS)**: replicates to a paired region.
- **Read-access geo-redundant storage (RA-GRS)**: adds read access to the secondary region for some storage scenarios.

For backup architecture, remember:

- Redundancy improves durability, but it is not the same as a tested restore.
- Geo-redundancy does not automatically mean every workload can restore in another region.
- Cross-region restore support depends on the vault type, workload, and feature availability.

Use higher redundancy when:

- The data is business critical.
- Regional outage recovery is part of the design.
- Compliance requires off-region copies.

Use lower redundancy when:

- The workload already has another disaster recovery layer.
- Cost is a stronger constraint than regional survivability.
- The backup is for short-term operational recovery only.

**Encryption and key management**

Backup data should be encrypted at rest and protected with strong access controls. Azure encrypts backup data by default, but design questions often ask about customer-managed keys (CMK), key vault integration, and access separation.

Key points:

- Azure-managed keys are the default for many services.
- Customer-managed keys can be required for regulatory or internal security controls.
- Key rotation and key vault access must be planned so backups remain recoverable.
- Backup administrators should not automatically have full access to encryption keys.
- Use least privilege for backup operators, restore operators, and security reviewers.

A secure design often separates:

- Backup administration permissions.
- Data restore permissions.
- Key management permissions.
- Subscription or resource group ownership.

**Azure VM backup architecture**

Azure VM backup is a classic Azure Backup scenario and typically uses a Recovery Services vault. It protects the VM at the platform level and creates recovery points that can be used for full VM restore or disk restore.

Typical architecture:

- VM is registered with a Recovery Services vault.
- A backup policy defines schedule and retention.
- Azure Backup takes snapshots and transfers recovery data to the vault.
- Restores can target a new VM, replace disks, or restore files depending on the scenario.

Example CLI flow:

```bash
# Create a Recovery Services vault
az backup vault create \
  --resource-group rg-backup \
  --name rsv-prod-eastus \
  --location eastus

# Set vault redundancy
az backup vault backup-properties set \
  --resource-group rg-backup \
  --vault-name rsv-prod-eastus \
  --backup-storage-redundancy GeoRedundant

# Register a VM for backup
az backup protection enable-for-vm \
  --resource-group rg-backup \
  --vault-name rsv-prod-eastus \
  --vm myVm \
  --policy-name DailyPolicy
```

Design notes:

- Instant restore uses local snapshots to speed up recovery, but snapshot retention is configurable within service limits.
- Long-term retention is stored in the vault.
- Restore options vary by CLI version and workload, so validate commands in your environment before operational use.
- For large estates, use Azure Policy to enforce backup enablement on new VMs.

**Azure Files backup architecture**

Azure Files backup is useful for shared content, user data, and application file shares. It is not a backup for arbitrary app settings outside the file share.

Use it when:

- The application stores shared files on Azure Files.
- User profiles or application data live in a file share.
- You need point-in-time recovery for file share content.

Typical design pattern:

- Store shared application data in Azure Files.
- Protect the share with Azure Backup.
- Use snapshots and retention for operational recovery.
- Combine with access controls and soft delete where appropriate.

Example:

```bash
az backup protection enable-for-azurefileshare \
  --resource-group rg-backup \
  --vault-name rsv-prod-eastus \
  --storage-account mystorageacct \
  --azure-file-share share01 \
  --policy-name FileShareDaily
```

Important limitations:

- Azure Files backup protects the file share contents, not the entire application configuration.
- If the app stores configuration in App Service settings, Key Vault, or another service, you must protect those separately.
- Restore testing should verify file permissions, share structure, and application compatibility.

**Azure SQL Database and SQL Managed Instance**

Azure SQL Database and Azure SQL Managed Instance have built-in backup capabilities managed by the platform. You usually do not deploy a separate backup agent for these services.

Azure SQL Database:

- Uses automated backups.
- Supports point-in-time restore.
- Supports long-term retention for compliance scenarios.
- Backup retention and restore behavior are service-managed.

Azure SQL Managed Instance:

- Also uses automated backups.
- Supports point-in-time restore and long-term retention.
- Can support native restore patterns that align with managed instance architecture.

Design considerations:

- Choose the correct retention period for compliance and operational recovery.
- Validate restore time for large databases.
- Consider geo-restore or failover groups for disaster recovery, but do not confuse replication with backup.
- Use long-term retention when regulatory retention exceeds the default backup window.

Example CLI for SQL Database long-term retention policy:

```bash
az sql db ltr-policy set \
  --resource-group rg-data \
  --server sqlprod01 \
  --database appdb \
  --weekly-retention P12W \
  --monthly-retention P12M \
  --yearly-retention P5Y \
  --week-of-year 16
```

For SQL Managed Instance, design questions often focus on:

- Automated backups and restore points.
- Long-term retention.
- Geo-replication for availability versus backup for recovery.
- Recovery testing in a nonproduction environment.

**SQL Server on Azure Arc**

SQL Server enabled by Azure Arc extends Azure management to SQL Server instances running outside Azure or in hybrid environments. Backup architecture here depends on the deployment model and the protection method you choose.

Common patterns:

- Use native SQL Server backup to URL or to storage, then manage retention and copy policies.
- Use Azure Arc-enabled data services where applicable for platform-managed capabilities.
- Integrate with enterprise backup tooling when native Azure Backup support is not available for the exact topology.

Design considerations:

- Confirm whether the workload is SQL Server on a VM, SQL Server on-premises, or Azure Arc-enabled SQL Server.
- Align backup method with recovery objectives and operational ownership.
- Test restore procedures for both database-level and instance-level recovery.

**Azure Cosmos DB backup architecture**

Azure Cosmos DB supports built-in backup options that are service-managed. The exact backup model depends on the API and account configuration.

Common models:

- **Periodic backup**: backups occur on a schedule and retain recovery points for a defined period.
- **Continuous backup**: supports point-in-time restore within a configured retention window.

Design notes:

- Continuous backup is preferred when you need fine-grained restore points.
- Periodic backup may be sufficient for less demanding recovery requirements.
- Backup and restore behavior is account-specific and must be validated against the chosen API and region support.
- Do not assume a generic file-based backup process applies to Cosmos DB.

Example design guidance:

- Use continuous backup for mission-critical transactional data.
- Use periodic backup when cost or feature constraints make continuous backup unnecessary.
- Test restore into a separate account or container to validate application compatibility.

**Azure App Service backup architecture**

Azure App Service has native backup and restore capabilities on supported tiers. It is not protected like a VM by default, but it does have built-in backup options for app content and configuration elements supported by the platform.

Use App Service backup when:

- You need to protect app content and selected configuration.
- You want a simple restore path for web apps.
- The app runs on a tier that supports backup.

Important points:

- Backups are tied to App Service capabilities and tier support.
- Deployment slots are not a backup strategy by themselves.
- Source control and infrastructure-as-code help with redeployment, but they do not replace data backup.
- External dependencies such as databases, storage accounts, and Key Vault secrets must be protected separately.

A practical design often combines:

- App Service backup for app-level recovery.
- Database backup for persistent data.
- Key Vault backup and soft delete for secrets recovery.
- Infrastructure-as-code for fast redeployment.

**Azure Virtual Desktop and profile data**

Azure Virtual Desktop (AVD) backup design usually focuses on user profile containers and shared data rather than the session host itself.

Common pattern:

- Store FSLogix profile containers on Azure Files or another supported storage target.
- Protect the file share with Azure Backup.
- Rebuild session hosts from images or automation rather than backing up each host as a primary strategy.

Why this matters:

- Session hosts are often ephemeral.
- User state lives in profile containers, not on the VM disk.
- Restore testing should confirm that profiles mount correctly after recovery.

Design considerations:

- Protect profile containers with file share backup.
- Use image-based deployment for session hosts.
- Keep host configuration in automation, not manual changes.
- Validate profile restore performance and lock handling.

**Azure Disk snapshots and snapshot-based protection**

Azure Disk snapshots are point-in-time copies of managed disks. They are useful for fast operational recovery, but they are not the same as a full backup architecture.

Use snapshots when:

- You need quick recovery of a disk state.
- You want a short-term operational rollback point.
- You are protecting a workload that can be restored from disk-level copies.

Use Azure Backup when:

- You need policy-driven retention.
- You need centralized governance and reporting.
- You need application-aware protection or broader restore workflows.

Key differences:

- Snapshots are typically simpler and faster for local recovery.
- Azure Backup adds policy, retention, vaulting, and centralized management.
- Snapshots alone may not satisfy compliance or long-term retention requirements.

**Backup testing and validation**

A backup that has never been restored is only a theory. AZ-305 expects you to design for validation, not just data capture.

Your backup architecture should include:

- Scheduled restore tests.
- Nonproduction recovery environments.
- Validation of application consistency after restore.
- Documentation of restore steps and dependencies.
- Measured RTO and RPO against business requirements.

A practical restore validation runbook might include:

1. Select a recent recovery point.
2. Restore to an isolated test resource group or subscription.
3. Verify data integrity and application startup.
4. Confirm identity, networking, and secret dependencies.
5. Record restore duration and issues.
6. Update the runbook if the restore path changes.

> If you cannot restore it, you cannot claim it is protected.

**Operational runbooks and incident response**

Backup architecture should include operational procedures for common failure scenarios.

Your runbooks should cover:

- Accidental deletion of a VM, database, or file share.
- Corruption of application data.
- Ransomware or malicious deletion.
- Regional outage or zone failure.
- Expired retention or failed backup jobs.
- Key loss or access revocation.

Each runbook should specify:

- Who can initiate restore.
- Which vault or service to use.
- Which recovery point to choose.
- Where to restore the data.
- How to validate success.
- How to communicate status to stakeholders.

**Security, access control, and immutability**

Backup data is a high-value target. Protect it with layered controls.

Recommended controls:

- Use RBAC to separate backup operators from subscription owners.
- Enable soft delete where supported.
- Use immutability or vault lock features where appropriate.
- Restrict vault access with network controls when supported.
- Monitor backup deletion and policy changes with Azure Monitor and activity logs.

A secure design often includes:

- A dedicated backup subscription or resource group.
- Limited administrative access to vaults.
- Private endpoints or restricted network access where supported.
- Alerting for failed jobs, deleted recovery points, and policy drift.

**Enterprise design patterns**

Common enterprise backup patterns include:

- **Centralized backup governance**: one team defines standards, many teams consume them.
- **Workload-specific vaults**: separate vaults by environment, region, or business unit.
- **Tiered retention**: short-term operational recovery plus long-term compliance retention.
- **Hybrid protection**: Azure Backup for supported workloads and native service backup for PaaS data.
- **Restore isolation**: restores land in a quarantined subscription or resource group for validation.

A simple topology might look like this:

```text
Applications
  ├─ Azure VMs ──> Recovery Services vault
  ├─ Azure Files ─> Recovery Services vault
  ├─ Azure SQL DB ─> platform-managed backups
  ├─ Cosmos DB ───> platform-managed backups
  └─ App Service ─> App Service backup + separate data protection
```

**Practical design examples**

Example 1: line-of-business web app

- App code in App Service.
- Data in Azure SQL Database.
- Shared uploads in Azure Files.
- Secrets in Key Vault.

Backup design:

- Use App Service backup for app content where supported.
- Use SQL automated backups and long-term retention.
- Use Azure Files backup for uploads.
- Protect Key Vault with soft delete and purge protection.
- Test full application restore in a staging environment.

Example 2: virtual desktop environment

- Session hosts are rebuilt from images.
- User profiles are stored in Azure Files.
- Application data is stored in a separate file share.

Backup design:

- Protect Azure Files shares with Azure Backup.
- Use image-based deployment for hosts.
- Keep host configuration in automation.
- Validate profile restore and sign-in behavior.

Example 3: regulated database platform

- Azure SQL Managed Instance hosts production databases.
- Compliance requires seven years of retention.

Backup design:

- Use automated backups for operational recovery.
- Configure long-term retention for compliance.
- Document point-in-time restore procedures.
- Test restore into a nonproduction instance regularly.

**CLI and configuration examples**

Use CLI examples as patterns, not as universal commands. Always confirm the exact syntax for the workload and Azure CLI version.

```bash

az backup vault list --resource-group rg-backup

# Show a specific Recovery Services vault
az backup vault show --resource-group rg-backup --name rsv-prod-eastus
```

**Exam focus points**

For AZ-305, be ready to choose the right backup approach based on workload and requirements.

You should know:

- When to use a Recovery Services vault versus a Backup vault.
- Which workloads have native platform backups.
- How backup differs from disaster recovery replication.
- Why restore testing is part of the architecture.
- How retention, redundancy, and encryption affect design.
- How to protect shared file data, profile data, and database data differently.
- Why some workloads require service-native backup rather than Azure Backup.

**Common pitfalls**

- Treating geo-redundant backup storage as a guaranteed cross-region restore for every workload.
- Using the wrong vault type for the workload.
- Assuming snapshots equal a full backup strategy.
- Forgetting to protect dependent services such as databases, secrets, and file shares.
- Designing backup without a restore test plan.
- Ignoring retention limits and workload-specific policy constraints.
- Overlooking access control and immutability for backup data.

**Quick decision guide**

- Use **Recovery Services vault** for classic Azure Backup workloads such as Azure VMs and Azure Files.
- Use **Backup vault** for supported Microsoft Data Protection workloads.
- Use **platform-managed backups** for Azure SQL Database, SQL Managed Instance, and Cosmos DB where applicable.
- Use **App Service backup** for supported web app tiers.
- Use **Azure Files backup** for shared file content, not for all app configuration.
- Use **snapshots** for fast operational recovery, not as the only protection layer.
- Use **restore testing** and runbooks to prove recoverability.
