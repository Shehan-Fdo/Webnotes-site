## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Pattern overview**

Active-active and active-passive describe how you place and use redundant components across zones or regions. The choice affects availability, latency, cost, operational complexity, and how quickly you can recover from a failure.

- **Active-active** means multiple instances serve traffic at the same time.
- **Active-passive** means one instance serves traffic while another waits to take over.
- The pattern can apply at different scopes:
  - **Zone-level** within one region
  - **Region-level** across multiple Azure regions
- The right pattern depends on whether your workload can tolerate:
  - Split traffic across replicas
  - Eventual consistency
  - Failover delay
  - Higher cost for duplicate capacity

A useful exam shortcut is to separate the problem into three layers:

1. **Traffic routing**: how users reach healthy endpoints.
2. **Compute redundancy**: where app instances run.
3. **Data replication**: how state stays available after a failure.

If any one of those layers is single-writer or slow to replicate, the whole solution may be active-passive even if the web tier is active-active.

**Active-active vs. active-passive at a glance**

| Aspect | Active-active | Active-passive |
|---|---|---|
| Traffic handling | Multiple endpoints serve requests simultaneously | Primary serves requests; secondary is standby |
| Failover | Usually fast, often automatic at the routing layer | Requires promotion or DNS/routing change |
| Cost | Higher, because capacity is duplicated and used | Lower, because standby may be idle or lightly used |
| Data model | Works best with multi-writer or conflict-tolerant data | Fits single-writer systems more naturally |
| Complexity | Higher, especially for stateful apps | Lower, but recovery can be slower |
| Typical use | Stateless web tiers, globally distributed apps | Legacy apps, single-writer databases, warm standby DR |

**Zone-level and region-level patterns**

Zone-level resilience protects against datacenter or rack failures inside one Azure region. Region-level resilience protects against a full regional outage.

- **Zone-level active-active**
  - You run instances in multiple availability zones in the same region.
  - A zone failure does not take the service down if enough healthy instances remain.
  - This is common for stateless application tiers and some data services with zone redundancy.
- **Zone-level active-passive**
  - One zone carries most or all traffic.
  - Another zone is ready to take over if the primary zone fails.
  - This is less common for modern Azure designs, but it can appear when capacity or application constraints limit distribution.
- **Region-level active-active**
  - You deploy the workload in two or more regions and route traffic to all of them.
  - This improves resilience to regional outages and can reduce latency for global users.
- **Region-level active-passive**
  - One region is primary and another is standby.
  - This is often simpler for stateful systems and is common when the data layer cannot support multi-region writes.

> Active-active at the web tier does not automatically make the whole solution active-active. If the database is single-writer, the end-to-end design is still constrained by the data tier.

**Choosing the right traffic-routing service**

Azure has several routing services, and they are not interchangeable. The protocol and scope matter.

| Service | Protocols | Scope | Best fit | Notes |
|---|---|---|---|---|
| Azure Front Door | HTTP/HTTPS | Global | Web apps, APIs, content delivery, edge routing | Modern Front Door uses `az afd` commands; it is not a general network failover tool |
| Traffic Manager | DNS | Global | Multi-region endpoint selection, simple failover | DNS-based, so failover depends on TTL and client caching |
| Application Gateway | HTTP/HTTPS | Regional | Layer 7 load balancing inside a region | Good for WAF (Web Application Firewall), path-based routing, and ingress |
| Azure Load Balancer | TCP/UDP | Regional or cross-region depending on SKU/design | Layer 4 distribution for VMs and services | Standard Load Balancer is common for intra-region resilience; cross-region load balancing is a separate design choice |

Use the service that matches the traffic type:

- Choose **Front Door** when you need global HTTP/HTTPS routing, fast edge-based failover, and optional WAF.
- Choose **Traffic Manager** when DNS-based global failover is acceptable and you want simple endpoint health-based routing.
- Choose **Application Gateway** when you need regional HTTP/HTTPS ingress, TLS termination, and path-based routing.
- Choose **Load Balancer** when you need TCP/UDP distribution for VMs, containers, or internal services.

**Front Door, Traffic Manager, Application Gateway, and Load Balancer**

These services solve different problems, and exam questions often test the distinction.

- **Azure Front Door**
  - Works at the application edge.
  - Supports HTTP/HTTPS only.
  - Routes to healthy origins using health probes.
  - Can distribute traffic across origin groups and origins.
  - Is a strong choice for global active-active web architectures.
- **Traffic Manager**
  - Uses DNS to direct clients to an endpoint.
  - Can do priority, weighted, performance, geographic, and subnet routing.
  - Is useful when you need global failover without proxying traffic.
  - Failover is not instant because clients and resolvers cache DNS responses.
- **Application Gateway**
  - Is regional and sits in front of web workloads.
  - Supports TLS termination, cookie-based affinity, and WAF.
  - Is often used inside a region or as a regional ingress point behind a global router.
- **Azure Load Balancer**
  - Is Layer 4 and does not inspect HTTP.
  - Is ideal for non-HTTP workloads or when you need simple TCP/UDP distribution.
  - Standard Load Balancer supports zone redundancy and high availability within a region.
  - Cross-region load balancing is a separate capability and should not be confused with standard regional load balancing.

A common design is:

- **Front Door** for global entry
- **Application Gateway** for regional web ingress
- **Load Balancer** for internal VM or service distribution

**Compute tier design**

The compute tier is often the easiest part to make active-active.

- **Virtual Machine Scale Sets (VMSS)**
  - Scale out stateless workloads across zones or regions.
  - Pair with a load balancer or Application Gateway.
- **App Service**
  - Can run multiple instances in one region.
  - For multi-region active-active, deploy separate app instances per region and route with Front Door or Traffic Manager.
- **AKS (Azure Kubernetes Service)**
  - Can run multi-zone node pools in one region.
  - Multi-region active-active is possible, but you must design ingress, identity, and data replication carefully.
- **Functions**
  - Can be deployed in multiple regions, but state and triggers often determine whether the solution is truly active-active.

For stateless compute, active-active is usually straightforward. The hard part is state:

- Session state should be externalized to a distributed store.
- File uploads should go to replicated storage or a shared service.
- Background jobs should avoid duplicate processing across regions unless they are idempotent.

**Data tier constraints**

The data layer often determines whether the architecture can truly be active-active.

- **Azure SQL Database / Azure SQL Managed Instance**
  - These are typically single-writer systems.
  - You can use failover groups for disaster recovery, but that is usually active-passive for writes.
  - Read scale-out and read replicas can help with read traffic, but they do not make the system multi-writer.
- **Azure Cosmos DB**
  - Supports multi-region distribution and configurable consistency levels.
  - Can be used in active-active designs more naturally than single-writer databases.
  - You still need to understand conflict resolution and consistency tradeoffs.
- **Azure Storage**
  - Geo-redundant options improve durability and recovery.
  - Most storage accounts are not true multi-writer active-active across regions for arbitrary application writes.
- **Redis**
  - Often used for cache or session state.
  - It can reduce database pressure, but it is not a substitute for a durable multi-region data design.

**Cosmos DB and consistency**

Cosmos DB is a common exam example because it supports multi-region patterns better than many relational databases.

- **Strong consistency**
  - Gives the highest read consistency.
  - Can increase latency and reduce write flexibility across regions.
- **Bounded staleness**
  - Allows a limited lag between writes and reads.
  - Balances consistency and performance.
- **Session consistency**
  - Common for user-facing applications.
  - Ensures a client sees its own writes.
- **Consistent prefix**
  - Preserves write order.
- **Eventual consistency**
  - Offers the lowest latency and highest availability tradeoff, but reads may lag behind writes.

Use stronger consistency when correctness matters more than latency. Use weaker consistency when global responsiveness matters more and your application can tolerate temporary divergence.

**Azure SQL failover groups and read replicas**

Azure SQL failover groups are a common DR pattern, but they are not the same as active-active writes.

- A failover group provides a primary and secondary logical server relationship.
- Automatic failover depends on the configured grace period and health conditions.
- The secondary is typically readable, which helps with reporting or read-heavy workloads.
- Writes still go to one primary at a time.
- This makes the pattern closer to active-passive for write operations.

Practical example:

```bash
az sql failover-group create \
  --name fg-prod \
  --partner-server sql-secondary \
  --resource-group rg-data \
  --server sql-primary \
  --failover-policy Automatic \
  --grace-period 1
```

Use this when you want regional disaster recovery for a relational workload that cannot support multi-writer behavior. Do not treat it as instant active-active failover.

**Routing behavior and health probes**

Health probes are central to failover, but each service handles them differently.

- **Front Door**
  - Probes origins and routes traffic away from unhealthy endpoints.
  - Works well for HTTP/HTTPS applications.
  - Can fail over quickly at the edge, but client behavior still matters.
- **Traffic Manager**
  - Probes endpoints and updates DNS responses.
  - Failover speed depends on DNS TTL and resolver caching.
- **Application Gateway**
  - Probes backend pools and stops sending traffic to unhealthy instances.
  - Works within a region.
- **Load Balancer**
  - Uses health probes to determine backend availability.
  - Routes at Layer 4 and does not understand application paths.

A practical rule:

- Use **health probes** for service health.
- Use **replication health** for data health.
- Use **synthetic transactions** when you need to verify end-to-end readiness, not just port availability.

**RTO and RPO implications**

RTO (Recovery Time Objective) is how long recovery can take. RPO (Recovery Point Objective) is how much data loss is acceptable.

- **Active-active**
  - Usually gives lower RTO because traffic can shift quickly.
  - Can give low RPO if data is synchronously replicated or conflict-tolerant.
  - May still have brief inconsistency during failover or partition events.
- **Active-passive**
  - Usually has higher RTO because failover must promote standby resources.
  - RPO depends on replication lag.
  - Can be acceptable when some downtime or data loss is tolerable.

Important tradeoff:

- Synchronous replication lowers RPO but can increase latency and reduce throughput.
- Asynchronous replication improves performance but increases the chance of data loss during a failure.

**When active-active is a good fit**

Use active-active when the workload is:

- Mostly stateless or can externalize state
- Read-heavy or globally distributed
- Able to tolerate eventual consistency or conflict resolution
- Designed for automatic traffic shifting
- Sensitive to regional latency

Typical examples:

- Public web applications
- APIs with stateless compute
- Content delivery and edge-optimized apps
- Multi-region services with distributed data stores

**When active-passive is a better fit**

Use active-passive when the workload is:

- Single-writer or strongly consistent
- Hard to partition across regions
- Dependent on legacy session state or local file systems
- Cost-sensitive and does not need duplicate hot capacity
- Acceptable with slower recovery

Typical examples:

- Legacy line-of-business applications
- Relational systems with one primary writer
- Workloads with strict transactional ordering
- Systems where operational simplicity matters more than low RTO

**End-to-end reference architectures**

A good AZ-305 answer usually connects routing, compute, and data.

**1. Global active-active web app**

- Two regions host the same stateless app.
- Front Door routes users to the nearest healthy region.
- Each region has:
  - App Service or VMSS
  - Regional ingress if needed
  - Local cache
- Data layer uses Cosmos DB multi-region writes or another distributed data design.

Flow:

1. User hits Front Door.
2. Front Door sends traffic to the closest healthy region.
3. App processes the request locally.
4. Data is written to a multi-region store.
5. If one region fails, Front Door stops sending traffic there.

This pattern gives low RTO and can support low RPO if the data layer is designed for it.

**2. Regional active-passive DR design**

- Primary region hosts the app and database.
- Secondary region has warm standby compute.
- Azure SQL failover group replicates the database.
- Traffic Manager or Front Door routes to the primary and fails over when needed.

Flow:

1. Users connect to the primary region.
2. The app writes to the primary database.
3. Replication keeps the secondary current.
4. If the primary region fails, traffic shifts to the secondary.
5. The secondary database is promoted.
6. Traffic is restored after validation and failback planning.

This pattern usually has higher RTO than active-active, but it is simpler and often more realistic for relational workloads.

**3. Zone-redundant regional service**

- App instances run across availability zones in one region.
- Standard Load Balancer or Application Gateway distributes traffic.
- Data is zone-redundant or managed by a zone-aware service.

This is not the same as multi-region active-active, but it is often the right answer when the question asks for resilience against datacenter failure rather than regional outage.

**Practical CLI examples**

Modern Front Door management uses `az afd` commands.

```bash
az afd profile create \
  --profile-name afd-prod \
  --resource-group rg-web \
  --sku Standard_AzureFrontDoor
```

Create an endpoint:

```bash
az afd endpoint create \
  --profile-name afd-prod \
  --endpoint-name web-endpoint \
  --resource-group rg-web \
  --enabled-state Enabled
```

Create an origin group:

```bash
az afd origin-group create \
  --profile-name afd-prod \
  --origin-group-name og-web \
  --resource-group rg-web \
  --probe-request-type GET \
  --probe-protocol Https \
  --probe-interval-in-seconds 30
```

Add two origins in different regions:

```bash
az afd origin create \
  --profile-name afd-prod \
  --origin-group-name og-web \
  --origin-name eastus-origin \
  --resource-group rg-web \
  --host-name app-eastus.contoso.com \
  --origin-host-header app-eastus.contoso.com \
  --priority 1 \
  --weight 1000

az afd origin create \
  --profile-name afd-prod \
  --origin-group-name og-web \
  --origin-name westus-origin \
  --resource-group rg-web \
  --host-name app-westus.contoso.com \
  --origin-host-header app-westus.contoso.com \
  --priority 1 \
  --weight 1000
```

Equal priority and equal weight are a common active-active setup. Priority is used to prefer one origin over another when needed; weight influences distribution among eligible origins.

Traffic Manager priority routing example:

```bash
az network traffic-manager profile create \
  --name tm-prod \
  --resource-group rg-web \
  --routing-method Priority \
  --unique-dns-name tm-prod-contoso

az network traffic-manager endpoint create \
  --name eastus \
  --profile-name tm-prod \
  --resource-group rg-web \
  --type azureEndpoints \
  --target-resource-id /subscriptions/<sub>/resourceGroups/rg-web/providers/Microsoft.Web/sites/app-eastus \
  --priority 1

az network traffic-manager endpoint create \
  --name westus \
  --profile-name tm-prod \
  --resource-group rg-web \
  --type azureEndpoints \
  --target-resource-id /subscriptions/<sub>/resourceGroups/rg-web/providers/Microsoft.Web/sites/app-westus \
  --priority 2
```

This is active-passive routing because only the highest-priority healthy endpoint receives traffic.

**Failover and failback**

Failover is only half the story. You also need a controlled failback plan.

- **Failover**
  - Detect the failure.
  - Stop sending traffic to the unhealthy component.
  - Promote standby resources if needed.
  - Validate data consistency before resuming service.
- **Failback**
  - Rebuild or resync the failed region.
  - Confirm health and replication lag.
  - Shift traffic back gradually.
  - Watch for split-brain or stale data issues.

> Avoid automatic failback unless you are confident the recovered region is fully synchronized and safe to serve traffic.

**Common exam traps**

- Confusing **DNS failover** with instant failover.
- Assuming **Front Door** is a general network load balancer.
- Treating **Azure SQL failover groups** as multi-writer active-active.
- Using **Load Balancer** for global HTTP routing.
- Ignoring the data tier when describing an active-active app.
- Mixing up **zone redundancy** with **multi-region** resilience.
- Assuming all services can do both active-active and active-passive equally well.

**Decision guide**

Use this quick mental model:

- If the workload is **HTTP/HTTPS** and global, start with **Front Door**.
- If the workload is **TCP/UDP**, start with **Load Balancer**.
- If the workload needs **regional web ingress** and WAF, use **Application Gateway**.
- If the workload needs **simple global DNS failover**, use **Traffic Manager**.
- If the data layer is **single-writer**, expect **active-passive** for writes.
- If the data layer supports **multi-region writes** and conflict handling, active-active becomes realistic.

**Key takeaways**

- Active-active improves availability and can reduce RTO, but it increases design complexity.
- Active-passive is simpler and often better for single-writer data systems.
- The data tier usually decides whether the architecture is truly active-active.
- Front Door, Traffic Manager, Application Gateway, and Load Balancer solve different routing problems.
- For AZ-305, always state the scope: zone-level or region-level, and compute-only or end-to-end.
