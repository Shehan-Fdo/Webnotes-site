## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Disaster recovery design goals**

Disaster recovery (DR) is about restoring service after a regional outage, data corruption event, ransomware incident, or major dependency failure. In Azure, you usually design for a target **recovery time objective** (RTO) and **recovery point objective** (RPO), then choose services and patterns that meet those targets at acceptable cost.

- **RTO** is how long you can tolerate downtime.
- **RPO** is how much data loss you can tolerate, measured in time.
- **Active-active** designs reduce downtime but increase complexity and cost.
- **Active-passive** designs are simpler and often fit most enterprise DR scenarios.
- DR is not only about application servers; it also includes identity, DNS, networking, data, and operational runbooks.

A good Azure DR design separates:
- **Availability** within a region from **disaster recovery** across regions.
- **Service-managed failover** from **application-managed cutover**.
- **Backup** from **replication**.
- **Data durability** from **application continuity**.

**Core design principles**

Use the least complex pattern that satisfies the business objective. For many workloads, the best answer is not “replicate everything everywhere,” but “replicate the critical state, automate failover, and test regularly.”

- Keep the **primary region** and **secondary region** paired and documented.
- Automate failover steps with runbooks, templates, or recovery plans.
- Validate dependencies such as DNS, certificates, secrets, and firewall rules.
- Test failover in isolation so you do not impact production traffic.
- Plan failback as carefully as failover; returning to the primary region is often more error-prone.

**Compute DR**

Compute DR depends on the workload type. Virtual machines, containers, and platform services each have different recovery mechanisms.

For Azure virtual machines, **Azure Site Recovery (ASR)** is the primary DR service. ASR replicates VM disks to a target region and orchestrates failover and failback. It is not backup; it is replication plus orchestration.

- Use ASR when you need:
  - Cross-region VM replication.
  - Orchestrated failover with recovery plans.
  - Test failover without affecting production.
  - Failback after the primary region recovers.
- Use Azure Backup when you need:
  - Point-in-time restore for supported workloads.
  - Protection against accidental deletion, corruption, or ransomware.
  - Long-term retention and compliance.

> Azure Backup and ASR solve different problems. Backup protects data over time; ASR reduces outage duration by keeping a warm replica ready for failover.

**Azure Site Recovery prerequisites and workflow**

ASR requires a **Recovery Services vault**, source and target regions, replication policy, and network mapping. You also need to validate that the target region has enough capacity and that the target virtual network can support the recovered workload.

Typical setup flow:
1. Create a **Recovery Services vault** in a region that is not the same as the protected workload region.
2. Enable replication for the VM or VM set.
3. Choose the target region, target resource group, and target virtual network.
4. Configure the replication policy for RPO, app-consistent snapshots, and retention.
5. Create a **recovery plan** if multiple VMs must fail over in a specific order.
6. Run a **test failover** into an isolated network.
7. Document failover and failback steps, then rehearse them.

Example Azure CLI for vault creation and enabling replication:
```bash
az group create -n rg-dr -l eastus

az backup vault create \
  --resource-group rg-dr \
  --name rsv-prod-dr \
  --location eastus

az backup vault backup-properties set \
  --resource-group rg-dr \
  --vault-name rsv-prod-dr \
  --backup-storage-redundancy GeoRedundant
```

ASR is commonly configured through the portal or PowerShell rather than a single simple CLI command because replication settings are workload-specific. In practice, you define the vault, policy, mapping, and recovery plan, then enable replication per VM.

Important ASR considerations:
- **Recovery plans** let you sequence multi-tier apps, such as database first, then app tier, then web tier.
- **App-consistent snapshots** reduce data inconsistency for supported guest workloads.
- **Test failover** should use an isolated network or a dedicated test subnet.
- **Failback** often requires re-protecting the VM in the original direction after the primary region is healthy.
- **Dependency mapping** matters; a VM without its database, secrets, or private endpoint is not a complete recovery.

**ASR failover and failback details**

Failover is not just a button click. You need a runbook that covers DNS, user access, data consistency, and post-failover validation.

- **Planned failover** is used when the primary region is still reachable and you can synchronize data before switching.
- **Unplanned failover** is used during an outage when the primary region is unavailable.
- **Test failover** creates a copy of the replicated VM in an isolated network for validation.
- **Failback** returns workloads to the original region after recovery and usually requires re-protection.

Practical failover checklist:
1. Confirm the latest replication health and RPO.
2. Freeze or drain writes if the outage is planned.
3. Trigger failover in the recovery plan.
4. Validate application health, identity access, and data consistency.
5. Update DNS or traffic routing.
6. Re-protect and fail back when the primary region is stable.

**Virtual machine scale sets and images**

For VM-based workloads, DR is easier when you treat compute as disposable and keep state outside the VM.

- Use **golden images** and infrastructure as code to rebuild quickly.
- Store application state in managed data services rather than local disks.
- Use **VM Scale Sets** for stateless tiers when possible.
- Keep custom scripts, extensions, and configuration in source control.

If you must preserve VM state:
- Replicate the VM with ASR.
- Ensure boot diagnostics, extensions, and managed identities are re-created or validated in the target region.
- Verify that any attached data disks are included in the replication plan.

**AKS disaster recovery**

Azure Kubernetes Service (AKS) DR depends on whether you run active-active or active-passive.

- **Active-active**:
  - Two clusters serve traffic simultaneously.
  - Best for stateless services and high availability.
  - Requires careful data synchronization and global traffic management.
- **Active-passive**:
  - One cluster is primary; the other is warm standby.
  - Simpler for many enterprise workloads.
  - Often preferred when stateful components are hard to replicate.

AKS DR considerations:
- The **control plane** is managed by Azure, but your cluster configuration, node pools, ingress, secrets, and persistent volumes still need DR planning.
- Use GitOps or IaC to recreate cluster configuration.
- Back up Kubernetes manifests, Helm charts, and cluster add-on configuration.
- Plan for **persistent volume** recovery using snapshots, backup tools, or replicated storage.
- Validate ingress, certificates, and external dependencies in the secondary region.

Example pattern for AKS:
```yaml
apiVersion: v1
kind: Service
metadata:
  name: web
spec:
  type: LoadBalancer
  selector:
    app: web
  ports:
    - port: 80
      targetPort: 8080
```

For multi-region AKS, you often combine:
- Regional clusters.
- Azure Front Door or Traffic Manager for global routing.
- External data replication for databases and caches.
- Separate identity and secret management dependencies.

**Network DR**

Network DR is often the hidden failure point. A workload can be healthy in the secondary region but still unreachable because DNS, firewall, private endpoints, or routing were not designed for failover.

**Global traffic routing**

Use a global traffic manager to direct users to the healthy region.

- **Azure Front Door** is best for HTTP/HTTPS global load balancing, TLS termination, WAF integration, and fast failover at the application edge.
- **Traffic Manager** is DNS-based and works for many protocols, but failover depends on DNS caching and client behavior.

Choose based on protocol and routing needs:
- Use **Front Door** for web apps, APIs, and internet-facing HTTP(S) services.
- Use **Traffic Manager** when you need DNS-based routing for endpoints that are not HTTP edge services.
- Use **Load Balancer** for regional layer 4 distribution, not global DR.

Example Azure Front Door origin group concept:
```bash
az afd origin-group create \
  --resource-group rg-network \
  --profile-name afd-prod \
  --origin-group-name og-app \
  --probe-request-type GET \
  --probe-protocol Https \
  --probe-path /health
```

Example Traffic Manager profile:
```bash
az network traffic-manager profile create \
  --resource-group rg-network \
  --name tm-prod \
  --routing-method Priority \
  --unique-dns-name contoso-prod \
  --ttl 30
```

Traffic Manager endpoint example for Azure App Service:
```bash
az network traffic-manager endpoint create \
  --resource-group rg-network \
  --profile-name tm-prod \
  --name app-east \
  --type azureEndpoints \
  --target-resource-id /subscriptions/<subId>/resourceGroups/rg-app/providers/Microsoft.Web/sites/app-east \
  --priority 1
```

**DNS and private DNS failover**

DNS is critical because clients often cache records longer than you expect.

- Use low TTL values for records that may change during failover.
- Keep public DNS and private DNS failover patterns separate.
- Update CNAME or A records as part of the runbook.
- Validate that health probes do not depend on the same failing region.

For private endpoints:
- Private DNS zones must resolve to the correct private IP in the active region.
- If the service is region-specific, you may need separate private endpoints per region.
- During failover, update the zone records or use a design that allows region-specific resolution.

Example private DNS zone record update:
```bash
az network private-dns record-set a add-record \
  --resource-group rg-dns \
  --zone-name privatelink.database.windows.net \
  --record-set-name sql-prod \
  --ipv4-address 10.20.1.4
```

**Azure Firewall, NVAs, and internal load balancers**

Network security appliances and internal routing must also be redundant.

- Deploy **Azure Firewall** or an NVA in each region if the workload depends on inspection.
- Replicate route tables, UDRs (user-defined routes), and NSGs (network security groups).
- Recreate internal load balancers in the secondary region.
- Ensure backend pools and health probes are valid after failover.
- Plan for asymmetric routing if you use stateful appliances.

For private-only applications:
- Rebuild internal load balancers in the secondary region.
- Repoint private DNS to the secondary private endpoint or service IP.
- Validate that on-premises connectivity, VPN, or ExpressRoute paths still reach the secondary region.

**Identity DR**

Identity is a dependency, not just a control plane service. Microsoft Entra ID is globally distributed, but you still need resilience planning for authentication, authorization, privileged access, and emergency operations.

Key identity realities:
- You do not usually design a “regional failover” for the tenant itself the way you do for an app.
- You still must protect access to the tenant if a region, network path, or admin workstation fails.
- Conditional Access, MFA (multifactor authentication), and identity governance can become dependencies during an outage.

Important identity DR practices:
- Maintain at least two **break-glass** emergency access accounts.
- Exclude break-glass accounts from Conditional Access policies, but protect them with strong controls and monitoring.
- Store credentials securely and test access procedures.
- Use **Privileged Identity Management (PIM)** for just-in-time elevation.
- Document how administrators regain access if primary authentication methods fail.

What to plan for:
- Authentication outages caused by network or federation dependencies.
- Loss of access to admin tools because of Conditional Access misconfiguration.
- Dependency on external identity providers for federated sign-in.
- Recovery of application registrations, service principals, and managed identities.

What not to assume:
- That Entra ID is immune to all outages.
- That every authentication path is regionally independent.
- That a working app can be administered if privileged access is locked out.

Practical identity resilience guidance:
- Prefer cloud-native authentication where possible.
- Avoid single points of failure in federation and MFA registration.
- Keep emergency access procedures offline and tested.
- Use PIM approval workflows that still function during a regional incident.

**Data DR**

Data services often define the real RPO and RTO. The right pattern depends on whether the service is IaaS, PaaS, or SaaS-like.

**Azure SQL Database and SQL Managed Instance**

For Azure SQL Database, the preferred DR pattern for most scenarios is **auto-failover groups**. They provide managed replication and readable secondary endpoints for failover.

- Use **auto-failover groups** for regional DR and application connection string abstraction.
- Use **active geo-replication** when you need more direct control or specific topology needs.
- Use **failover groups** as the higher-level, generally preferred option for most exam scenarios.
- For SQL Managed Instance, use failover groups for cross-region DR.

Example failover group creation:
```bash
az sql failover-group create \
  --name fog-prod \
  --resource-group rg-sql \
  --server primary-sql \
  --partner-server secondary-sql \
  --failover-policy Automatic \
  --grace-period 60
```

Operational notes:
- The application should connect through the failover group listener, not directly to the primary server.
- Failover groups simplify connection string management.
- Test failover to validate application behavior, firewall rules, and identity permissions.
- Geo-replication is asynchronous, so some data loss is possible during an unplanned failover.

**Azure Database for PostgreSQL and MySQL**

For managed PostgreSQL and MySQL, DR options depend on the service tier and deployment model.

- Use built-in **geo-redundant backup** and restore for recovery from corruption or deletion.
- Use read replicas or cross-region replicas where supported for lower RPO and faster regional recovery.
- Validate whether the service supports zone redundancy, geo-backup restore, or replica promotion in your chosen tier.
- Treat failover as service-specific; the exact behavior differs from Azure SQL.

Exam-relevant guidance:
- For PostgreSQL, cross-region read replicas are often used for DR in supported tiers, with promotion during failover.
- For MySQL, geo-redundant backup and restore are common recovery mechanisms, and replica-based patterns may be used where supported.
- Always distinguish between backup-based recovery and replica-based failover.

**Azure Synapse Analytics**

Synapse DR depends on the component:
- **Dedicated SQL pools** need backup/restore planning and workspace-level recovery considerations.
- **Serverless SQL** is less about data persistence and more about source data availability.
- **Pipelines and artifacts** should be stored in source control and redeployed with IaC.

Practical guidance:
- Back up workspace artifacts, notebooks, linked services, and pipelines.
- Replicate source data and metadata dependencies.
- Validate managed identities, Key Vault references, and firewall rules in the secondary region.

**Cosmos DB and other globally distributed data**

For globally distributed databases, DR may be built into the service.

- Use multi-region writes only when the application can handle conflict resolution.
- Use single-write with multiple read regions when you want simpler consistency.
- Understand the consistency model because it affects failover behavior and perceived data loss.
- Test regional failover and client retry logic.

**Storage DR**

Azure Storage offers several redundancy options, but redundancy is not the same as application failover.

- **LRS** (Locally Redundant Storage) keeps copies within one datacenter.
- **ZRS** (Zone-Redundant Storage) spans availability zones in one region.
- **GRS** (Geo-Redundant Storage) replicates asynchronously to a paired region.
- **RA-GRS** (Read-Access Geo-Redundant Storage) adds read access to the secondary endpoint.

Important distinctions:
- GRS and RA-GRS improve durability and regional resilience, but they do not automatically fail over your application.
- RA-GRS lets you read from the secondary endpoint, which is useful for read-only access or validation, but it is not the same as application cutover.
- Service-managed account failover is a storage account operation, not an application orchestration feature.
- After account failover, the secondary becomes primary for the storage account, but your application still needs to handle endpoint changes and any dependent services.

Use cases:
- **LRS** for noncritical or transient data.
- **ZRS** for higher availability within a region.
- **GRS/RA-GRS** for disaster recovery and durability.
- **Immutable storage** for ransomware-resistant retention.

Example storage account redundancy:
```bash
az storage account create \
  --name stproddata01 \
  --resource-group rg-storage \
  --location eastus \
  --sku Standard_RAGRS \
  --kind StorageV2
```

**Backup, immutability, and ransomware resilience**

Backup is your last line of defense against corruption, deletion, and ransomware.

Key design choices:
- Use **soft delete** to protect against accidental deletion.
- Use **immutability** for backup vaults or blob data when compliance or ransomware resistance is required.
- Separate backup credentials and access from production admin roles.
- Keep backup retention long enough to detect and recover from delayed attacks.
- Test restore regularly; a backup you have not restored is not proven.

Backup vault design considerations:
- Use a dedicated vault or logically separated vaults for critical workloads.
- Restrict who can delete backup items or change retention.
- Enable alerts for backup failures and suspicious changes.
- Consider cross-region restore capabilities where supported.

> Immutability and soft delete reduce the risk of destructive actions, but they do not replace a tested recovery plan.

**Data service patterns by workload**

Different services need different DR patterns.

- **Azure SQL Database**:
  - Prefer auto-failover groups for regional DR.
  - Use geo-restore for backup-based recovery.
- **Azure Storage**:
  - Use GRS/RA-GRS for durability and regional resilience.
  - Use application cutover procedures for failover.
- **Azure Database for PostgreSQL/MySQL**:
  - Use geo-backup, replicas, and service-supported promotion patterns.
- **Cosmos DB**:
  - Use multi-region replication and understand consistency tradeoffs.
- **Synapse**:
  - Back up artifacts and source data; rebuild workspace dependencies.
- **VM-based databases**:
  - Use ASR for the VM and database-native backups for the data.

**Traffic routing examples**

Azure Front Door example for global web failover:
```bash
az afd endpoint create \
  --resource-group rg-network \
  --profile-name afd-prod \
  --endpoint-name ep-prod

az afd origin create \
  --resource-group rg-network \
  --profile-name afd-prod \
  --origin-group-name og-app \
  --origin-name app-east \
  --host-name app-east.azurewebsites.net \
  --priority 1 \
  --weight 100
```

Traffic Manager priority routing concept:
- Priority 1 points to the primary region.
- Priority 2 points to the secondary region.
- Health probes determine when to shift traffic.

**Practical DR architecture patterns**

Common Azure DR patterns include:
- **Web app + SQL Database**:
  - Front Door or Traffic Manager for global routing.
  - App Service in two regions.
  - SQL failover group.
  - Key Vault replicated or redeployed with secrets strategy.
- **VM-based line-of-business app**:
  - ASR for compute.
  - SQL Server backup/restore or SQL Always On where appropriate.
  - DNS failover and network appliance replication.
- **AKS microservices**:
  - Two clusters in separate regions.
  - Global traffic routing.
  - Externalized state in replicated data services.
  - GitOps for cluster rebuild.
- **Private enterprise app**:
  - Private endpoints in both regions.
  - Private DNS failover.
  - Firewall/NVA redundancy.
  - Identity and certificate recovery plan.

**Testing and operational readiness**

DR is only credible if you test it.

Test types:
- **Test failover** to validate recovery without production impact.
- **Planned failover** to rehearse maintenance or migration.
- **Game day** exercises to validate people, process, and tooling.
- **Restore tests** for backup-based recovery.

What to validate:
- Application startup order.
- DNS propagation and TTL behavior.
- Firewall and NSG rules.
- Managed identities and Key Vault access.
- Database connectivity and schema compatibility.
- Monitoring, logging, and alerting in the secondary region.

Operational artifacts to keep current:
- Architecture diagrams.
- Dependency maps.
- Runbooks.
- Contact lists.
- Access procedures for emergency accounts.
- Recovery time and recovery point targets.

**Exam decision cues**

When you see a DR scenario in AZ-305, identify the service type first.

- If it is a **VM**, think ASR plus backup.
- If it is a **global web app**, think Front Door or Traffic Manager plus regional deployment.
- If it is **Azure SQL Database**, think failover groups.
- If it is **storage**, think redundancy plus application cutover.
- If it is **identity**, think tenant resilience, emergency access, and dependency planning.
- If it is **AKS**, think multi-region cluster strategy and state recovery.

A useful mental model:
- **Replication** reduces RPO.
- **Orchestration** reduces RTO.
- **Backup** protects against logical damage.
- **Routing** moves users to the healthy endpoint.
- **Identity resilience** keeps administrators and applications able to authenticate.

**Quick reference**

| Area | Preferred DR approach | Key caveat |
|---|---|---|
| VM compute | ASR | Not a backup replacement |
| Web traffic | Front Door or Traffic Manager | DNS and probe behavior matter |
| Azure SQL Database | Auto-failover groups | App must use the listener |
| Storage | GRS/RA-GRS or service failover | App cutover still required |
| AKS | Multi-region clusters | Stateful data needs separate planning |
| Identity | Break-glass, PIM, dependency planning | Entra ID is global, but access paths are not failure-proof |
| Backup | Vault-based backup with soft delete and immutability | Must be tested with restores |

**Common pitfalls**

- Assuming backup and replication are interchangeable.
- Forgetting DNS, certificates, and secrets during failover.
- Treating RA-GRS as automatic application failover.
- Using active geo-replication when a failover group is the better managed choice.
- Ignoring private endpoints and internal load balancers in regional failover.
- Designing compute DR without data DR.
- Designing data DR without identity recovery.

**Summary**

Azure DR design is a cross-cutting architecture problem. You choose the right recovery pattern for each layer: ASR for VM replication, failover groups for managed SQL, global routing for user traffic, backup for corruption and ransomware, and identity resilience for administrative continuity. The best solutions are simple enough to operate, tested often, and aligned to explicit RTO and RPO targets.
