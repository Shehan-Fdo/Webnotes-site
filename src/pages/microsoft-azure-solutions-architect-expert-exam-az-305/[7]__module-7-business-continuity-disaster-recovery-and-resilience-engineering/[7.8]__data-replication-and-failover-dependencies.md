## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Overview**

Data replication and failover design is about more than copying data to another region. You need to understand which components replicate automatically, which require explicit configuration, and which dependencies must be recovered in a specific order. In Azure, the right design depends on the workload’s recovery objectives, consistency requirements, and tolerance for data loss.

A strong design separates:
- **Replication**: how data is copied or synchronized.
- **Failover**: how traffic and workloads move to a secondary location.
- **Dependency recovery**: which services must come back first so the application can function safely.

In exam scenarios, the key question is often not “Can this fail over?” but “What breaks if this tier fails first, and what is the safest recovery sequence?”

**Core design principles**

Design for recovery around business requirements, not around a single Azure service. Start with:
- **RPO**: Recovery Point Objective, the maximum acceptable data loss.
- **RTO**: Recovery Time Objective, the maximum acceptable downtime.
- **Consistency model**: whether the app can tolerate eventual consistency, read-only replicas, or synchronous commit.
- **Dependency order**: identity, networking, data, messaging, compute, and then user traffic.

A practical recovery plan usually includes:
- Primary and secondary region mapping.
- DNS and traffic routing strategy.
- Data replication method per tier.
- Runbooks for manual and automatic failover.
- Validation steps after cutover.

**Replication patterns by Azure service**

Different Azure services solve different parts of the problem. You should know the common patterns and their limitations.

**Azure Storage**

Azure Storage replication options include:
- **LRS**: Locally redundant storage, copies data within one datacenter.
- **ZRS**: Zone-redundant storage, copies data across availability zones in one region.
- **GRS**: Geo-redundant storage, asynchronously replicates to a paired region.
- **RA-GRS**: Read-access geo-redundant storage, adds read access to the secondary endpoint.
- **GZRS**: Geo-zone-redundant storage, combines zone redundancy in the primary region with geo-replication.
- **RA-GZRS**: Read-access geo-zone-redundant storage, adds read access to the secondary endpoint.

Important behavior:
- **ZRS** does not provide a secondary region read endpoint.
- **GRS** and **RA-GRS** replicate to a secondary region asynchronously.
- **RA-GRS** and **RA-GZRS** expose a read-only secondary endpoint.
- Account failover is a deliberate operation and is not the same as automatic zone recovery.

Practical implications:
- Use **ZRS** for zone failure resilience within a region.
- Use **GRS/RA-GRS** when you need regional disaster recovery.
- Use **RA-GRS/RA-GZRS** when read-only access to secondary data is useful for reporting or validation.

Example CLI for storage account failover:
```bash
az storage account failover \
  --account-name mystorageacct \
  --resource-group rg-data-dr
```

Notes:
- Account failover is a controlled operation that promotes the secondary region.
- After failover, the account’s replication state changes and the old primary becomes the new secondary when the service supports it.
- DNS propagation and client retry behavior matter because applications may continue to resolve cached endpoints for a short time.

Limitations and cautions:
- Not every storage account configuration supports every redundancy mode.
- Failover is not a substitute for application-level data protection.
- If your app writes directly to blobs or queues, you still need idempotent processing and retry logic.

**Azure SQL Database and Azure SQL Managed Instance**

Azure SQL supports high availability within a region and disaster recovery across regions, but the exact feature set depends on the deployment model.

For **Azure SQL Database**:
- **Active geo-replication** creates readable secondary databases in other regions.
- **Auto-failover groups** provide managed failover for one or more databases.
- Failover groups also provide a stable listener endpoint for application connection strings.

For **Azure SQL Managed Instance**:
- Failover groups are supported for disaster recovery scenarios, but availability depends on service capabilities and regional support.
- Managed Instance is not identical to Azure SQL Database in topology or feature behavior, so do not assume every database pattern transfers directly.

Key concepts:
- **Readable secondary**: useful for offloading reads and validating replication.
- **Failover group listener**: a DNS endpoint that moves to the secondary during failover.
- **Grace period**: in Azure SQL failover groups, this is measured in hours, not minutes.

Example failover group creation pattern:
```bash
az sql failover-group create \
  --name fog-app1 \
  --partner-server sql-secondary-server \
  --resource-group rg-sql-primary \
  --server sql-primary-server \
  --failover-policy Automatic \
  --grace-period 1
```

Interpretation:
- `--server` is the primary server context.
- `--partner-server` identifies the secondary server.
- `--grace-period 1` means 1 hour.

Operational guidance:
- Use failover groups when you want managed DR and a stable application endpoint.
- Use active geo-replication when you need more control over individual database replicas.
- Test failover with application connection retry logic, because existing sessions are dropped during cutover.

**Azure Cosmos DB**

Azure Cosmos DB is designed for global distribution and is a major exam topic for replication and failover.

Replication options:
- **Single-region writes** with multiple read regions.
- **Multi-region writes** for active-active write capability.
- **Automatic failover** based on configured failover priorities.
- **Manual failover** when you want to control region promotion.

Consistency levels:
- **Strong**
- **Bounded staleness**
- **Session**
- **Consistent prefix**
- **Eventual**

How to think about it:
- Stronger consistency reduces the chance of stale reads but can increase latency and reduce availability during regional issues.
- Multi-region writes improve write availability but increase the need for conflict handling and careful application design.
- Failover priority determines which region becomes primary when automatic failover occurs.

Practical design points:
- Use **session consistency** for many application workloads because it balances usability and performance.
- Use **multi-region writes** only when the application can tolerate conflict resolution and duplicate write scenarios.
- Design idempotent write operations so retries do not create duplicate business records.

Example CLI to add a region and configure failover priority:
```bash
az cosmosdb update \
  --name cosmos-app1 \
  --resource-group rg-cosmos \
  --locations regionName=eastus failoverPriority=0 isZoneRedundant=False \
              regionName=westus failoverPriority=1 isZoneRedundant=False
```

Operational cautions:
- Multi-region writes do not remove the need for conflict resolution.
- Application logic must handle transient 429 and 5xx responses with retry policies.
- If you use custom conflict resolution, test it under simulated regional loss.

**Azure Database for PostgreSQL and MySQL**

Azure database services also have replication and failover patterns that appear in design questions.

For **Azure Database for PostgreSQL**:
- Flexible Server supports high availability within a region.
- Read replicas can support read scaling and some DR patterns.
- Cross-region read replicas can help with disaster recovery planning, depending on service capabilities and region support.

For **Azure Database for MySQL**:
- Flexible Server supports high availability within a region.
- Read replicas can be used for read scaling and secondary-region planning where supported.
- Replication is typically asynchronous for cross-region scenarios, so expect some data lag.

Design implications:
- Use read replicas for reporting, offloading reads, and DR warm standby patterns.
- Do not assume a read replica is a fully automatic failover target unless the service explicitly supports that workflow.
- Validate application connection strings and DNS cutover behavior before relying on a replica in production.

**Messaging services and geo-disaster recovery**

Messaging systems are often overlooked in failover plans, but they are critical dependencies.

**Azure Service Bus**
- Supports geo-disaster recovery through alias-based pairing.
- The alias provides a stable namespace name that can be switched to the secondary namespace.
- Geo-DR replicates metadata, not the actual message payloads in all scenarios, so you must verify what is and is not protected for your workload.

Important behaviors:
- Client applications should use retry logic and be prepared for transient disconnects.
- After alias failover, existing connections break and clients must reconnect.
- Duplicate message delivery can occur, so consumers should be idempotent.

Example pattern:
```bash
az servicebus georecovery-alias set \
  --resource-group rg-messaging \
  --namespace-name sb-primary \
  --alias-name sb-alias \
  --partner-namespace sb-secondary
```

**Azure Event Hubs**
- Supports geo-disaster recovery through an alias mechanism similar in concept to Service Bus.
- Geo-DR focuses on namespace metadata and endpoint failover, not full event replay protection.
- Event retention and consumer checkpointing are still your responsibility.

Design guidance:
- Use checkpointing in consumers so they can resume after failover.
- Expect at-least-once processing and design for duplicate events.
- Keep producer retries bounded to avoid message storms during recovery.

**Networking and traffic failover**

Traffic routing is separate from data replication. You need both.

**Azure Front Door**
- Best for global HTTP/HTTPS traffic distribution.
- Provides layer 7 routing, health probes, and fast failover across regions.
- Useful when you want edge-based routing and application acceleration.

**Traffic Manager**
- DNS-based global traffic distribution.
- Works at layer 4/7 only indirectly through DNS resolution.
- Failover depends on DNS TTL and client caching behavior.

**Application Gateway**
- Regional layer 7 load balancer.
- Useful inside a region or as part of a regional active-passive design.
- It does not replace global failover services by itself.

Comparison guidance:
- Use **Front Door** when you need global HTTP failover with low-latency edge routing.
- Use **Traffic Manager** when DNS-based routing is sufficient and you can tolerate DNS failover delays.
- Use **Application Gateway** for regional ingress, WAF, and internal routing, but pair it with a global failover mechanism if you need cross-region resilience.

Practical dependency note:
- If your app depends on Application Gateway, you must recover the gateway, backend pool health, and backend DNS targets before sending traffic to the region.
- If you use Front Door in front of multiple regional gateways, Front Door can fail over traffic while the regional gateway remains the local ingress point.

**Identity and access dependencies**

Identity is often the first hidden dependency in a recovery plan.

Consider:
- **Microsoft Entra ID** availability for authentication.
- Managed identities for service-to-service access.
- Key Vault access policies or role assignments.
- Private endpoint DNS resolution for identity and secret retrieval paths.

Recovery order often looks like:
1. Identity and authorization services.
2. Network connectivity and DNS.
3. Data stores and messaging.
4. Application compute.
5. Traffic routing and user access.

Why this matters:
- If Key Vault is unreachable, apps may fail to start because secrets cannot be loaded.
- If DNS is wrong, private endpoints may resolve to stale or unreachable addresses.
- If managed identity permissions are not replicated, the app may come up but fail at runtime.

**Backup, restore, and vault choices**

Backup is not the same as replication.

**Recovery Services vault**
- Commonly used for Azure VM backup and site recovery scenarios.
- Supports backup and disaster recovery workflows for infrastructure and some workloads.
- Often associated with Azure Site Recovery for VM replication and failover orchestration.

**Backup vault**
- Used with newer Azure Backup scenarios and certain workload types.
- Focuses on backup management rather than full DR orchestration.

Design distinction:
- Use **replication** for low-RTO failover.
- Use **backup** for point-in-time recovery, ransomware recovery, and long-term retention.
- Use **site recovery** when you need orchestrated VM failover between regions.

Practical exam takeaway:
- A backup vault does not replace a Recovery Services vault in every scenario.
- A backup copy is not a live secondary that can immediately take traffic.
- Restore time can be much longer than failover time.

**Dependency mapping and recovery sequencing**

A good recovery plan maps dependencies explicitly. A common multi-tier sequence is:

1. **Identity and secrets**
   - Entra ID access
   - Key Vault
   - certificates and connection strings

2. **Network and name resolution**
   - DNS records
   - private DNS zones
   - VPN/ExpressRoute connectivity
   - firewall and NSG rules

3. **Data layer**
   - SQL failover group or geo-replica
   - Cosmos DB region promotion
   - storage account failover
   - message namespace alias cutover

4. **Application layer**
   - App Service, AKS, VMs, or container apps
   - configuration refresh
   - health checks

5. **Traffic layer**
   - Front Door, Traffic Manager, or load balancer updates
   - client routing validation

Example recovery order for a three-tier app:
- Bring up the secondary database or promote the failover group.
- Verify secrets and managed identity permissions.
- Confirm storage endpoints and message namespace availability.
- Start application compute in the secondary region.
- Shift global traffic and monitor error rates.

> If you route traffic before the data tier is ready, you can create avoidable write failures, duplicate retries, and partial outages.

**Designing for consistency during failover**

Failover is not only about availability. It is also about preserving data correctness.

Use these patterns:
- **Idempotency**: repeated requests produce the same result.
- **Transaction boundaries**: keep business transactions small and explicit.
- **Outbox pattern**: persist the business change and the message to send in one transaction.
- **Inbox pattern**: deduplicate incoming messages before processing.
- **Optimistic concurrency**: use version checks to prevent lost updates.

Common failure modes:
- **Split-brain**: two regions accept writes for the same logical entity without coordination.
- **Duplicate writes**: retries after timeout create duplicate records.
- **Stale reads**: a secondary region serves old data after failover or during replication lag.

Practical guidance:
- Use a single write region unless the workload truly needs active-active writes.
- If you use active-active, define conflict resolution before production.
- Make consumers safe to run more than once.

Example idempotent API pattern:
```http
POST /orders
Idempotency-Key: 7f2c1c2a-8f1d-4f5d-9f2d-1a2b3c4d5e6f
```

Application logic should:
- Store the idempotency key.
- Return the same result for repeated requests.
- Avoid creating duplicate downstream messages.

**Practical comparison of failover mechanisms**

| Service area | Typical failover method | Notes |
|---|---|---|
| Storage | Account failover, RA-GRS/RA-GZRS read access | Secondary read access is not universal across all redundancy modes |
| Azure SQL Database | Failover groups, active geo-replication | Listener endpoint simplifies app connection management |
| Azure SQL Managed Instance | Failover groups where supported | Validate service and region support |
| Cosmos DB | Automatic or manual failover with priorities | Consistency and multi-region writes affect design |
| Service Bus | Geo-disaster recovery alias | Metadata is paired; clients must reconnect |
| Event Hubs | Geo-DR alias | Consumer checkpoints remain your responsibility |
| Front Door | Global health probe routing | Best for HTTP/HTTPS workloads |
| Traffic Manager | DNS-based failover | DNS caching affects recovery speed |
| Application Gateway | Regional ingress failover | Pair with a global routing service for cross-region DR |

**Exam tips**

- Distinguish **replication** from **failover orchestration**.
- Remember that DNS-based failover is not instant because of caching.
- Know which services support **readable secondary** access and which do not.
- Treat messaging systems as first-class dependencies, not optional extras.
- Prefer idempotent, retry-safe application design in active-active or asynchronous replication scenarios.
- Be careful with service-specific terminology:
  - Azure SQL failover group listener
  - Cosmos DB failover priority
  - Service Bus geo-recovery alias
  - Storage account failover
  - Front Door health probes
- When a question asks for the safest recovery sequence, recover identity, network, and data before shifting traffic.

**Common pitfalls**

- Assuming every secondary is writable.
- Assuming replication means automatic failover.
- Ignoring client-side connection pooling and DNS caching.
- Forgetting that backups restore data, but do not provide immediate service continuity.
- Designing active-active writes without conflict resolution.
- Overlooking private DNS and firewall dependencies during regional recovery.

**Quick mental model**

Think of failover in layers:
- **Data**: where the truth lives.
- **Access**: how clients reach it.
- **Compute**: where the app runs.
- **Control**: how you switch traffic and permissions.

If any one layer is missing, the failover is incomplete.
