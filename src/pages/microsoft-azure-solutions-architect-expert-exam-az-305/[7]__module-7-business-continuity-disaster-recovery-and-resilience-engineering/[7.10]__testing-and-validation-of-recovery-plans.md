## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Recovery plan testing goals**

A recovery plan is only useful if it works under realistic failure conditions. In Azure disaster recovery, you test to prove that applications, data, identity dependencies, networking, and operational procedures can survive a disruption and meet business expectations.

The main objectives are to validate:

- **RTO**: Recovery Time Objective, the maximum acceptable time to restore service.
- **RPO**: Recovery Point Objective, the maximum acceptable data loss measured in time.
- **Application availability**: The app starts, responds, and supports the required user flows.
- **Data integrity**: Data is consistent after failover and after failback.
- **Operational readiness**: Teams know the runbooks, approvals, and communication steps.

A good test is not just a technical exercise. It is a business validation that answers: “Can we recover within the agreed limits, and can users actually work?”

**What you validate in a recovery plan**

A recovery plan should be tested end to end, not just at the infrastructure layer. In Azure, that usually means validating the following layers together:

- **Compute**: Virtual machines, scale sets, or app hosts start correctly in the recovery region.
- **Storage**: Replicated disks, file shares, or object data are available and current enough for the RPO.
- **Network**: Subnets, DNS, routing, firewalls, and private endpoints allow the app to function.
- **Identity and access**: Domain controllers, federation services, certificates, and app authentication dependencies are reachable.
- **Platform services**: Azure SQL, App Service, Storage, Key Vault, and other dependencies are available in the target region or have a recovery strategy.
- **Monitoring and operations**: Alerts, logs, and support procedures still work after failover.

For Azure Site Recovery (ASR), testing often focuses on protected VMs and recovery plans. For platform services, you validate the service-specific failover or geo-redundancy behavior separately.

**Business requirements and acceptance criteria**

Before you run a test, define pass/fail criteria. This is critical for exam scenarios because “the failover succeeded” is not enough if the app is unusable or the data is stale.

Typical acceptance criteria include:

- The application becomes available within the target **RTO**.
- Data loss stays within the target **RPO**.
- Required user journeys succeed, such as sign-in, search, checkout, or report generation.
- No critical data corruption occurs.
- Monitoring, alerting, and support handoff procedures work.
- Security controls remain in place, including access restrictions and secrets retrieval.

A practical way to document this is to map each business requirement to a measurable test outcome.

| Requirement | Example acceptance criterion |
|---|---|
| RTO | App is available within 30 minutes of failover start |
| RPO | No more than 15 minutes of data loss |
| Availability | Users can authenticate and complete a transaction |
| Integrity | Database checks show no corruption or missing records |
| Operations | Support team receives alerts and can access logs |

> If you cannot measure success, you cannot prove recovery readiness.

**Common Azure recovery validation scenarios**

Azure recovery validation depends on the workload type. The exact tools vary, but the validation pattern is similar: fail over, verify dependencies, test the app, and then fail back.

- **Azure VMs with Azure Site Recovery**
  - Validate VM boot order, network mapping, and application startup.
  - Confirm that recovery plans group dependent machines correctly.
  - Check that scripts or manual steps in the plan run in the right order.

- **Azure SQL**
  - Validate geo-replication, failover groups, or backup restore procedures.
  - Confirm connection strings, read/write routing, and application retry logic.
  - Verify schema and data consistency after failover.

- **Storage**
  - Validate geo-redundant storage behavior, access tier expectations, and application retry handling.
  - Confirm that the app can read required blobs, queues, or files in the recovery region.

- **App Service**
  - Validate deployment slots, traffic routing, custom domains, and dependent services.
  - Confirm that secrets, certificates, and backend APIs are reachable.

- **Cross-region architectures**
  - Validate paired-region assumptions, regional service availability, and any region-specific dependencies.
  - Confirm that the recovery region supports the required SKUs, networking features, and compliance controls.

**Testing frequency and governance**

Recovery testing should be scheduled, not ad hoc. A mature program includes regular drills and change-driven retesting.

Recommended governance practices:

- **Scheduled DR drills**: Run tests on a recurring cadence, such as quarterly or semiannually, depending on business criticality.
- **Change-driven retesting**: Retest after major changes to networking, identity, application architecture, or replication settings.
- **Stakeholder sign-off**: Capture approval from application owners, operations, security, and business stakeholders after each validation.
- **Evidence retention**: Keep test results, screenshots, logs, and job outputs for audit and compliance needs.
- **Runbook updates**: Update recovery documentation after every test to reflect lessons learned.

A test that is never repeated becomes stale quickly. Azure regions, dependencies, and application code all change over time.

**Azure Site Recovery validation workflow**

For ASR, the practical workflow is usually:

1. Prepare the recovery plan and confirm the VM groups and dependencies.
2. Run a **test failover** into an isolated network or test network.
3. Validate boot, connectivity, application behavior, and data consistency.
4. Record results against the acceptance criteria.
5. Clean up the test environment.
6. If the test exposed issues, update the plan and retest.
7. When a real recovery is needed, perform failover and later failback.

A test failover should not impact production. Use a network that prevents accidental overlap with live systems unless the design explicitly supports controlled connectivity.

**PowerShell examples for ASR validation**

PowerShell is commonly used for ASR operations because it gives you repeatable automation and job tracking. The exact cmdlet set can vary by module version, so validate syntax in your environment.

A typical pattern is:

```powershell
# Connect to Azure
Connect-AzAccount

# Select subscription
Set-AzContext -SubscriptionId "00000000-0000-0000-0000-000000000000"

# Get the vault and set context
$vault = Get-AzRecoveryServicesVault -Name "dr-vault"
Set-AzRecoveryServicesAsrVaultContext -Vault $vault

# Start a test failover for a recovery plan
$rp = Get-AzRecoveryServicesAsrRecoveryPlan -Name "web-app-rp"
$job = Start-AzRecoveryServicesAsrTestFailoverJob `
  -RecoveryPlan $rp `
  -Direction PrimaryToRecovery `
  -AzureVMNetworkId "/subscriptions/00000000-0000-0000-0000-000000000000/resourceGroups/dr-test-rg/providers/Microsoft.Network/virtualNetworks/dr-test-vnet"

$job
```

After you start the job, check status and results:

```powershell
# Get the latest job status
Get-AzRecoveryServicesAsrJob | Sort-Object StartTime -Descending | Select-Object -First 5

# Inspect a specific job
Get-AzRecoveryServicesAsrJob -Name $job.Name | Format-List *
```

If your module version uses a different parameter set for recovery plans or test failover, adapt the command to the documented syntax in your environment. The important exam concept is the workflow: select vault, set context, start test failover, validate, and review job output.

**Validating the outcome of a test failover**

Do not stop at “job succeeded.” Validate the actual workload.

Use a checklist like this:

- Confirm the VM or service is running in the recovery region.
- Verify IP addressing, DNS resolution, and routing.
- Test application ports and health endpoints.
- Sign in with a test account and complete a business transaction.
- Check logs for startup errors, missing secrets, or certificate issues.
- Compare recovered data with expected source data.
- Confirm monitoring alerts and dashboards still function.

Example validation commands:

```powershell
# Check VM power state
Get-AzVM -ResourceGroupName "dr-test-rg" -Name "web01" -Status

# Test TCP connectivity to an app endpoint
Test-NetConnection -ComputerName "web01-dr.contoso.com" -Port 443

# Query a health endpoint
Invoke-WebRequest -Uri "https://web01-dr.contoso.com/health" -UseBasicParsing
```

For database-backed apps, validate both connectivity and data correctness:

```sql
SELECT TOP 10 *
FROM dbo.Orders
ORDER BY CreatedDate DESC;
```

If the app uses queues or background jobs, confirm that workers restart cleanly and that messages are not duplicated or lost beyond the accepted RPO.

**Failback and failback testing**

Recovery validation is incomplete without failback. Failback proves that you can return workloads to the primary region safely after the incident or test.

Failback considerations:

- **Data resynchronization**: Make sure changes made in the recovery region are replicated back before cutover.
- **Downtime window**: Plan for a controlled outage if the workload cannot run active-active.
- **Order of operations**: Bring back databases, then application tiers, then traffic routing.
- **Validation after return**: Re-run the same business checks you used during failover.

Failback testing is especially important when the recovery region becomes the temporary production environment. If users create or modify data there, you must preserve those changes before returning.

**Multi-region and cross-region dependency validation**

Cross-region recovery is more than “the VM starts in another region.” You must validate region-specific dependencies and service availability.

Check the following:

- **Regional service support**: Confirm the recovery region supports the required Azure services and SKUs.
- **Paired region behavior**: Understand whether the design relies on Azure paired regions for replication or service continuity.
- **Network reachability**: Validate VPN, ExpressRoute, private DNS, and firewall rules in the recovery region.
- **Latency-sensitive dependencies**: Confirm that the app still performs acceptably when dependent services are farther away.
- **External integrations**: Test APIs, payment gateways, SMTP relays, and partner systems from the recovery region.

A common failure mode is that the compute tier recovers, but a dependency remains pinned to the primary region. For example, a private endpoint, DNS zone, or firewall rule may still point to the original environment.

**Identity and authentication dependencies**

Identity is often the hidden reason a recovery test fails. The app may start, but users cannot sign in or services cannot authenticate to each other.

Validate:

- Domain controllers or managed domain services if the workload depends on Active Directory.
- Federation services if the app uses federated authentication.
- Certificates and secrets if the app depends on Key Vault or local certificate stores.
- Service principals, managed identities, and role assignments in the recovery subscription or resource group.

Be careful not to treat cloud identity as a “recoverable server” in the same way as a VM. Instead, validate that the application can still authenticate and authorize correctly in the recovery path.

**Practical end-to-end DR drill scenario**

Imagine a three-tier order processing app:

- Web tier on Azure VMs
- API tier on Azure VMs
- SQL database with geo-replication
- Storage account for invoices
- Active Directory-based authentication for internal users

A realistic drill looks like this:

1. Announce the test window and freeze nonessential changes.
2. Confirm the recovery plan groups the web, API, and database dependencies correctly.
3. Start a test failover into an isolated recovery VNet.
4. Verify that the VMs boot and that the database endpoint resolves correctly.
5. Test sign-in, order creation, invoice generation, and report retrieval.
6. Compare order counts and timestamps against the source environment.
7. Review logs for startup delays, DNS issues, or missing secrets.
8. Record whether the app met the RTO and RPO targets.
9. Clean up the test resources.
10. Update the runbook and schedule a retest if any step failed.

This kind of drill exposes real issues, such as:

- DNS records not updated for the recovery network
- SQL connection strings still pointing to the primary endpoint
- Certificates missing from the recovery VM image
- Background jobs failing because a storage account firewall blocks access

**Troubleshooting failed recovery tests**

When a test fails, classify the issue before you retry.

Common categories:

- **Replication issue**: Data is not current enough or replication is unhealthy.
- **Network issue**: Routes, DNS, NSGs, or firewalls block traffic.
- **Dependency issue**: A backend service, identity provider, or external API is unavailable.
- **Configuration issue**: App settings, secrets, certificates, or connection strings are wrong.
- **Capacity issue**: The recovery region lacks the required SKU or quota.
- **Operational issue**: The runbook step order is wrong or incomplete.

A useful troubleshooting pattern is:

1. Identify the first failed user-visible symptom.
2. Trace backward to the dependency that caused it.
3. Confirm whether the issue is in the app, platform, or recovery plan.
4. Fix the root cause, not just the symptom.
5. Retest the affected path.

**Azure CLI and portal usage**

Azure CLI can support some recovery-related operations, but PowerShell and the Azure portal are more commonly used for ASR administration and validation workflows. Use the tool that best matches your operational model and automation standards.

Example CLI patterns for general validation:

```bash
az account set --subscription 00000000-0000-0000-0000-000000000000
az vm show -g dr-test-rg -n web01 --show-details
az network nic show -g dr-test-rg -n web01-nic
```

For exam purposes, focus on the concept that tooling is secondary to the validation process. The key is proving that the recovery plan meets business requirements.

**Exam focus points**

- **Recovery plans** should be tested regularly and after major changes.
- **Test failover** validates the plan without impacting production.
- **Acceptance criteria** must include RTO, RPO, availability, and data integrity.
- **Failback** is part of end-to-end recovery validation.
- **Dependencies** such as DNS, identity, certificates, and external services often cause failures.
- **Evidence and sign-off** matter for governance and audit readiness.
- **Cross-region validation** must account for region-specific services and paired-region assumptions.

**Quick validation checklist**

- Confirm the recovery plan order and dependencies.
- Run a test failover in an isolated network.
- Validate application startup and user workflows.
- Check data consistency and freshness.
- Review job status and logs.
- Clean up test resources.
- Document findings and obtain stakeholder sign-off.
- Retest after fixes or major changes.
