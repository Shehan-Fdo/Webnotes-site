## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Tradeoff-based decision making**

AZ-305 scenarios rarely ask for a single “best” service in isolation. They ask you to choose the right architecture under constraints such as latency, compliance, cost, operational overhead, availability, and team skill. The correct answer usually reflects a compromise: you optimize for the stated business goal while accepting a known limitation elsewhere.

A strong decision process is to identify:
- The primary objective, such as lowest latency, highest resiliency, or simplest operations.
- The nonfunctional requirements, including RPO (Recovery Point Objective), RTO (Recovery Time Objective), security, and governance.
- The deployment scope, such as single region, multi-region, internet-facing, or private-only.
- The operational model, including patching, scaling, monitoring, and failover ownership.

**How to evaluate an Azure design choice**

Use a repeatable framework when comparing services:
1. Identify the workload type and traffic pattern.
2. Determine whether the requirement is global, regional, zonal, or subnet-level.
3. Decide whether the solution must be public, private, or hybrid.
4. Compare managed-service options against self-managed options.
5. Check whether the design meets the stated RPO/RTO, security, and cost constraints.
6. Prefer the simplest service that satisfies the requirement.

A useful mental model is that Azure architecture decisions are usually about *where* to enforce control:
- At the edge for global routing, caching, and web protection.
- At the region for application hosting and regional failover.
- At the subnet or NIC for network isolation.
- At the resource or subscription scope for governance and access control.
- At the data layer for consistency, replication, and backup.

**Availability and resiliency tradeoffs**

Availability design starts with understanding the failure domain. A failure domain is the smallest component whose failure should not take down the whole workload. In Azure, the main constructs are regions, availability zones, and availability sets.

- **Availability Zones** place resources in physically separate datacenters within a region.
  - Use them when the workload needs higher resiliency against datacenter failure.
  - They are best for zone-redundant application tiers, databases, and load-balanced services.
  - They usually increase complexity and can add cross-zone data transfer costs.

- **Availability Sets** spread virtual machines across fault domains and update domains within a datacenter cluster.
  - Use them for legacy or zone-unaware VM workloads that still need better availability than a single VM.
  - They do not protect against a full zone outage.
  - They are simpler than zone-based designs but less resilient.

- **Region pairs** are Azure’s paired-region strategy for disaster recovery and platform recovery.
  - Use them when you need geo-redundancy or a secondary region for failover.
  - They help with planned platform updates and regional recovery.
  - They do not automatically make your application active-active.

> A common exam trap is to confuse high availability with disaster recovery. High availability reduces downtime within a region; disaster recovery restores service after a regional outage.

**Storage redundancy choices**

Storage redundancy is a classic AZ-305 tradeoff because the right answer depends on whether you need local durability, zone resilience, or geo-resilience.

- **LRS** (Locally Redundant Storage) keeps three copies within a single datacenter.
  - Lowest cost.
  - Suitable for noncritical data, dev/test, or workloads with other resilience layers.

- **ZRS** (Zone-Redundant Storage) synchronously replicates across availability zones in the same region.
  - Better for zone failure tolerance.
  - Useful when the application must remain available during a zone outage.

- **GRS** (Geo-Redundant Storage) replicates data asynchronously to a paired secondary region.
  - Good for disaster recovery.
  - The secondary copy is not immediately writable.

- **GZRS** (Geo-Zone-Redundant Storage) combines zone redundancy in the primary region with geo-replication to a paired secondary region.
  - Stronger resilience than GRS alone.
  - Often chosen when both zone failure and regional recovery matter.

- **RA-GRS / RA-GZRS** add read access to the secondary replica.
  - Useful for read-heavy scenarios or DR validation.
  - They do not make the secondary region writable.

Practical selection rule:
- Choose **LRS** when cost matters most and the data is not critical.
- Choose **ZRS** when you need zone resilience in one region.
- Choose **GRS** when you need geo-recovery and can tolerate asynchronous replication.
- Choose **GZRS** when you need both zone and geo resilience.

**Networking and ingress tradeoffs**

Azure networking decisions often hinge on whether traffic is global or regional, whether it is public or private, and whether you need Layer 4 or Layer 7 features.

- **Azure Front Door** is a global, Layer 7 entry point for HTTP/HTTPS traffic.
  - Use it for global load balancing, fast failover, and edge acceleration.
  - It can terminate TLS at the edge and can also use end-to-end TLS to the origin.
  - It is a strong choice for internet-facing web apps that need global routing and web application firewall capabilities.

- **Traffic Manager** is DNS-based global traffic distribution.
  - Use it when you need global failover or weighted routing without proxying traffic.
  - It works at the DNS level, so client DNS caching affects failover behavior.
  - It does not provide Layer 7 inspection or content acceleration.

- **Azure Load Balancer** is a regional Layer 4 load balancer.
  - Use it for TCP/UDP traffic, internal services, or VM scale sets.
  - It is ideal for non-HTTP workloads and low-latency packet forwarding.
  - Choose the internal SKU for private-only backends and the public SKU for internet-facing endpoints.

- **Application Gateway** is a regional Layer 7 load balancer.
  - Use it for HTTP/HTTPS workloads that need path-based routing, cookie affinity, or WAF.
  - It is often the right choice for a regional web tier behind a reverse proxy.
  - It is not a global service.

A common multi-tier pattern is:
- **Front Door** at the edge for global entry and WAF.
- **Application Gateway** in-region for internal routing and app-tier protection.
- **Load Balancer** for non-HTTP backend services or VM-based tiers.

> If the scenario requires global web acceleration and regional failover, Front Door is often the first choice. If the scenario requires private regional routing for HTTP, Application Gateway is usually the better fit.

**Private access and network isolation**

Azure offers several ways to reduce public exposure, and the distinctions matter.

- **Private Endpoint** gives a private IP address from your VNet to a supported Azure service.
  - Use it when you want traffic to stay on private IP space.
  - It is the concrete network interface used by the service consumer.

- **Private Link** is the umbrella capability that enables private access to supported services.
  - It includes private endpoints and private access patterns for platform services.
  - Use it when the requirement is “no public endpoint” or “access over private IP.”

- **Service Endpoints** extend VNet and subnet reachability to Azure PaaS services over the Azure backbone.
  - They help restrict access to a service from specific subnets.
  - They do not create a private IP for the service.
  - They are simpler than Private Link but provide less isolation.

Decision guidance:
- Use **Private Link** when you need private IP-based access, stronger isolation, or to disable public network access.
- Use **Service Endpoints** when you want simpler subnet-based access control and the service supports it.
- Use **VPN Gateway** or **ExpressRoute** when you need hybrid connectivity from on-premises networks.

**Identity and access tradeoffs**

Identity architecture in Azure is usually about choosing the least operationally expensive identity model that still meets security requirements.

- **Microsoft Entra ID** is the identity provider for users, groups, and applications.
  - Use it for authentication, single sign-on, and conditional access.
  - It is the foundation for most Azure access decisions.

- **Managed identities** remove the need to store credentials in code.
  - **System-assigned managed identity** is tied to one Azure resource and is deleted with it.
  - **User-assigned managed identity** is a standalone identity that can be shared across resources.
  - Use managed identities for app-to-Azure service access whenever possible.

- **Service principals** represent applications in Entra ID.
  - Use them when a workload cannot use managed identity, such as some external CI/CD systems or non-Azure hosts.
  - Prefer managed identities over service principals for Azure-hosted workloads.

- **App roles** and **group-based assignment** help control application authorization.
  - Use app roles when the application needs role-based authorization semantics.
  - Use groups when you want simpler administration for user access.

- **Conditional Access** enforces policy based on user, device, location, and risk.
  - Use it for human access to sensitive apps and admin portals.
  - It is not a replacement for application authorization.

Practical rule:
- Use **system-assigned managed identity** for a single Azure resource that needs access to one or more services.
- Use **user-assigned managed identity** when multiple resources share the same identity or when you need identity continuity across resource replacement.
- Use **service principals** only when managed identity is not available.

Example CLI for assigning a managed identity role to a Key Vault scope:
```bash
az role assignment create \
  --assignee-object-id <principal-object-id> \
  --assignee-principal-type ServicePrincipal \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.KeyVault/vaults/<kv-name>
```

**Governance and management tradeoffs**

Governance decisions determine how much control you apply centrally versus locally.

- **Management groups** organize subscriptions and apply policy at scale.
  - Use them when multiple subscriptions need shared governance.
  - They are the right place for enterprise-wide policy and RBAC inheritance.

- **Subscriptions** are the primary billing and access boundary.
  - Use separate subscriptions for isolation, chargeback, or environment separation.
  - They are useful for separating production from nonproduction.

- **Resource groups** are lifecycle containers.
  - Use them for resources that share the same deployment and deletion lifecycle.
  - Do not use them as a security boundary.

- **Azure Policy** enforces standards and prevents drift.
  - Use it for location restrictions, SKU restrictions, tagging requirements, and allowed resource types.
  - Use initiatives to bundle related policies.

- **Resource locks** protect critical resources from accidental deletion or modification.
  - Use `CanNotDelete` for important production resources.
  - Use `ReadOnly` sparingly because it can block legitimate operations.

- **Tags** support cost allocation, ownership, and automation.
  - Use them consistently for environment, owner, application, and cost center.

Example policy assignment pattern:
```bash
az policy assignment create \
  --name "enforce-tags" \
  --scope /subscriptions/<sub-id> \
  --policy "/providers/Microsoft.Authorization/policyDefinitions/<policy-definition-id>"
```

**Monitoring and observability tradeoffs**

Monitoring architecture should answer three questions: what happened, where did it happen, and how do you know it matters?

- **Azure Monitor** is the umbrella observability platform.
  - Use it for metrics, logs, alerts, and action groups.
  - It is the central service for platform monitoring.

- **Log Analytics** stores and queries log data with KQL (Kusto Query Language).
  - Use it for deep investigation, correlation, and retention-based analysis.
  - It is the right place for operational logs and custom queries.

- **Application Insights** focuses on application performance and distributed tracing.
  - Use it for request tracing, dependency tracking, exceptions, and performance bottlenecks.
  - It is especially useful for web apps and microservices.

- **Alerts** should match the signal type.
  - Use metric alerts for near-real-time threshold conditions.
  - Use log alerts for complex queries and correlation.
  - Use activity log alerts for subscription-level control-plane events.

Practical guidance:
- Use **Application Insights** for app code and request-level telemetry.
- Use **Log Analytics** for infrastructure and cross-resource analysis.
- Use **Azure Monitor alerts** for response automation and operational notification.

Example KQL query:
```kusto
requests
| where success == false
| summarize failures = count() by bin(timestamp, 5m), cloud_RoleName
| order by timestamp desc
```

**Data platform tradeoffs**

Data architecture choices depend on consistency, relational needs, latency, and operational ownership.

- **Azure SQL Database** is a fully managed relational database.
  - Use it for modern cloud apps that need PaaS simplicity.
  - It reduces administrative overhead compared with VM-hosted SQL Server.

- **Azure SQL Managed Instance** offers near-full SQL Server compatibility with managed service benefits.
  - Use it when you need instance-level features, migration compatibility, or minimal code changes.
  - It is often the best fit for lift-and-shift database migrations that still want PaaS management.

- **Azure Cosmos DB** is a globally distributed NoSQL database.
  - Use it for low-latency, globally distributed, schema-flexible workloads.
  - Choose the API that matches the data model and application requirements:
    - SQL API for document-oriented JSON workloads.
    - MongoDB API for MongoDB-compatible applications.
    - Cassandra API for wide-column workloads.
    - Gremlin API for graph workloads.
    - Table API for key-value style access.

- **Azure Storage** is not one service but a family of services.
  - **Blob Storage** is best for object data, backups, logs, and unstructured content.
  - **Files** is best for SMB/NFS-style shared file access.
  - **Disks** are best for VM-attached block storage.
  - Choose the service based on access pattern, not just cost.

Decision examples:
- Use **Blob Storage** for static website assets, backups, and media.
- Use **Azure Files** when multiple servers need a shared file share.
- Use **managed disks** for VM operating system and data disks.
- Use **SQL Managed Instance** when migration compatibility matters more than lowest cost.
- Use **Cosmos DB** when global distribution and elastic scale matter more than relational joins.

**Caching and content delivery tradeoffs**

Caching reduces latency and origin load, but it adds invalidation and consistency concerns.

- **Azure Cache for Redis** is an in-memory cache.
  - Use it for session state, hot data, and low-latency lookups.
  - It is best when the application can tolerate cache misses and stale data windows.

- **Content Delivery Network (CDN)** or edge caching is best for static or cacheable content.
  - Use it for images, scripts, stylesheets, and downloadable assets.
  - It reduces origin traffic and improves global performance.

- **Front Door caching** can also help with global web acceleration.
  - Use it when you want edge delivery plus global routing in one service.

Tradeoff rule:
- Use **Redis** for application data acceleration.
- Use **CDN/edge caching** for static content delivery.
- Use both when the app has dynamic data and static assets with different performance needs.

**Messaging and event-driven integration tradeoffs**

Messaging choices depend on whether you need point-to-point delivery, fan-out, ordering, or decoupling.

- **Queues** are best for one-to-one asynchronous work distribution.
  - Use them when a single consumer or worker pool should process each message once.
  - They help smooth bursts and decouple producers from consumers.

- **Topics and subscriptions** are best for one-to-many fan-out.
  - Use them when multiple downstream systems need the same event.
  - They support filtering and independent consumers.

- **Event Grid** is optimized for event notification and reactive integration.
  - Use it for lightweight event routing and push-based event delivery.
  - It is a good fit for serverless and event-driven architectures.

- **Service Bus** is better for enterprise messaging.
  - Use it when you need ordered processing, duplicate detection, sessions, transactions, or dead-letter handling.
  - It is stronger than simple queues for business-critical workflows.

Practical guidance:
- Use **queues** for background jobs and work distribution.
- Use **topics** when multiple systems must react to the same business event.
- Use **dead-letter queues** when you need to isolate poison messages for later inspection.
- Use **retry policies** carefully so transient failures are retried without creating duplicate side effects.

**Backup and disaster recovery tradeoffs**

Backup and DR are related but not identical.

- **Backup** protects against data loss, corruption, and accidental deletion.
  - Use it when you need point-in-time recovery or long-term retention.
  - It is usually slower than failover for restoring service.

- **Disaster recovery** restores service after a major outage.
  - Use it when the workload must continue in another region or site.
  - It is driven by RPO and RTO targets.

- **Azure Site Recovery (ASR)** replicates supported workloads for failover.
  - Use it primarily for supported VMs and certain physical or VMware scenarios.
  - It is not a universal DR solution for every Azure service.

Decision criteria:
- Choose **backup** when the main risk is deletion, corruption, or ransomware recovery.
- Choose **ASR** when you need orchestrated failover for supported compute workloads.
- Choose **active-active** patterns when the business requires very low RTO and can handle higher cost and complexity.

**Load balancing patterns and placement**

Load balancing is not just about distributing traffic; it is about placing the right control at the right layer.

- **Public load balancer** exposes a frontend to the internet.
  - Use it for public TCP/UDP services.
  - It is common for internet-facing VM workloads.

- **Internal load balancer** keeps traffic private inside a VNet.
  - Use it for backend tiers, line-of-business apps, and east-west traffic.
  - It is a common choice for multi-tier architectures.

- **Multi-tier ingress pattern**
  - Use **Front Door** for global entry and edge protection.
  - Use **Application Gateway** for regional HTTP routing and WAF.
  - Use **Internal Load Balancer** for backend service distribution.
  - Use **Private Link** for private access to platform services.

This layered approach is often the best answer when a scenario asks for global reach, regional control, and private backend access at the same time.

**Practical decision scenarios**

1. **Global web app with strict security and low latency**
   - Use Front Door for global routing and edge WAF.
   - Use Application Gateway only if you need regional Layer 7 routing or private backend segmentation.
   - Use managed identities for app-to-data access.
   - Use Cosmos DB or geo-replicated SQL depending on data model and consistency needs.

2. **Internal line-of-business app with private database access**
   - Use an internal load balancer or Application Gateway depending on HTTP needs.
   - Use Private Link for database and storage access.
   - Use Azure Policy to restrict public endpoints.
   - Use Log Analytics and Application Insights for monitoring.

3. **Lift-and-shift SQL Server application**
   - Use Azure SQL Managed Instance if compatibility is a priority.
   - Use availability zones or zone-redundant design where supported.
   - Use backup plus geo-replication or failover groups if the RTO/RPO require regional recovery.
   - Use a user-assigned managed identity if multiple app components need stable identity access.

4. **Event-driven order processing system**
   - Use Service Bus topics if multiple downstream systems must process the same order event.
   - Use queues if each order should be processed by one worker.
   - Use dead-letter queues for poison messages.
   - Use Application Insights to trace failures across services.

**Common exam decision cues**

Look for wording that points to a specific service:
- “Global” or “edge” often suggests Front Door or Traffic Manager.
- “Private IP to PaaS” often suggests Private Link.
- “Subnet-based access to PaaS” often suggests Service Endpoints.
- “No code changes for identity” often suggests managed identity.
- “Shared file access” often suggests Azure Files.
- “Low-latency key-value access” often suggests Redis or Cosmos DB, depending on persistence needs.
- “Fan-out to multiple consumers” often suggests topics.
- “Exactly one worker processes each job” often suggests queues.
- “Regional HTTP routing with WAF” often suggests Application Gateway.
- “TCP/UDP load balancing” often suggests Azure Load Balancer.

**How to justify an answer in AZ-305**

When you choose a service, explain the tradeoff explicitly:
- State the requirement you are optimizing for.
- State the limitation you accept.
- State why the chosen service is the simplest one that satisfies the requirement.

Example justification:
- “Use Front Door because the app needs global HTTP failover and edge acceleration. Accept that it adds a proxy layer and is not the best fit for non-HTTP traffic.”
- “Use Private Link because the database must not be reachable over a public endpoint. Accept the added DNS and private endpoint management overhead.”
- “Use SQL Managed Instance because the migration needs high SQL Server compatibility. Accept higher cost than Azure SQL Database.”

**Quick comparison reference**

| Scenario | Best fit | Why |
|---|---|---|
| Global HTTP routing | Front Door | Edge routing, acceleration, global failover |
| Regional HTTP routing with WAF | Application Gateway | Layer 7 regional control |
| TCP/UDP distribution | Azure Load Balancer | Layer 4 performance |
| Private access to PaaS | Private Link | Private IP access |
| Subnet-restricted PaaS access | Service Endpoints | Simpler network restriction |
| Shared app identity in Azure | Managed identity | No secrets to manage |
| Enterprise app messaging | Service Bus | Ordering, sessions, dead-lettering |
| Fan-out events | Event Grid or topics | Multiple consumers |
| Shared file storage | Azure Files | SMB/NFS-style access |
| Object storage | Blob Storage | Unstructured data and scale |
| Regional HA for VMs | Availability Zones | Separate datacenter failure domains |
| Legacy VM availability | Availability Set | Fault/update domain separation |

**Final exam mindset**

AZ-305 rewards architecture judgment, not memorization. The best answer is usually the one that:
- Meets the stated requirement directly.
- Minimizes operational burden.
- Uses managed services where possible.
- Avoids unnecessary complexity.
- Aligns the control plane with the failure domain.

If two answers both seem plausible, compare them by:
- Scope: global vs regional vs subnet.
- Exposure: public vs private.
- Layer: L4 vs L7.
- Resiliency: HA vs DR.
- Operations: managed vs self-managed.
- Cost: baseline vs premium features.
