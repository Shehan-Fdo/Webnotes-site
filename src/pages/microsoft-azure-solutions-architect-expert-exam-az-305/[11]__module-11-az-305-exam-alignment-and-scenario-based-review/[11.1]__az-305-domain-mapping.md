## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**AZ-305 domain mapping overview**

AZ-305 is a design-focused exam. You are not being tested on memorized commands alone; you are being tested on whether you can choose the right Azure services and architecture patterns for a business requirement. The exam outline can change over time, so treat the domains below as a practical study map rather than a fixed promise of the current Microsoft weighting.

A useful way to study is to map every requirement to one of four broad design areas:

- **Identity, governance, and monitoring**
- **Data storage and business continuity**
- **Infrastructure**
- **Application architecture**

Each area overlaps with the others in real solutions. For example, a multi-region web app design involves networking, identity, monitoring, storage, and disaster recovery decisions at the same time. The exam often rewards the option that best balances availability, security, cost, and operational simplicity.

**How to think like an AZ-305 candidate**

When you read a scenario, ask:

- What is the availability target?
- What is the recovery objective?
- What data sensitivity or compliance requirement exists?
- What network boundaries or connectivity constraints apply?
- What operational model does the organization want?
- What is the simplest design that still meets the requirement?

A strong answer usually avoids overengineering. If a managed service meets the requirement, prefer it over a custom VM-based design unless the scenario explicitly needs OS-level control.

**Official-style skill mapping by domain**

The exact Microsoft wording may change, but the exam commonly tests the following sub-areas.

| Domain | Common sub-areas | What the exam expects |
|---|---|---|
| Identity, governance, and monitoring | Identity, access, governance, logging, alerting, cost control | Choose secure access patterns, policy controls, and monitoring designs |
| Data storage and business continuity | Storage accounts, databases, backup, replication, DR | Select the right data platform and recovery strategy |
| Infrastructure | Compute, networking, hybrid connectivity, DNS, routing | Design resilient, secure, and scalable infrastructure |
| Application architecture | App hosting, integration, messaging, APIs, caching | Pick the right app platform and integration pattern |

**Identity, governance, and monitoring**

This domain covers the control plane of the environment: who can access what, how you enforce standards, and how you observe the platform. In architecture questions, governance is rarely the final goal by itself; it is the mechanism that keeps the design secure and manageable at scale.

**Identity design**

Azure identity questions usually center on Microsoft Entra ID, role-based access control, managed identities, and privileged access patterns.

- **Microsoft Entra ID** is the identity provider for users, groups, and applications.
- **Role-based access control (RBAC)** assigns permissions at management group, subscription, resource group, or resource scope.
- **Managed identities** let Azure resources authenticate without storing secrets.
- **Privileged Identity Management (PIM)** supports just-in-time elevation for sensitive roles.

Use managed identities whenever an Azure service needs to call another Azure service. For example, an App Service can access Key Vault without a client secret if you enable a system-assigned managed identity and grant it the appropriate Key Vault permissions.

```bash
az webapp identity assign \
  --resource-group rg-app \
  --name app-prod

az keyvault set-policy \
  --name kv-prod \
  --object-id <principal-id> \
  --secret-permissions get list
```

**Governance design**

Governance is about consistency and guardrails. In AZ-305 scenarios, you often need to decide whether to use:

- **Management groups** for hierarchy and inheritance
- **Azure Policy** for compliance enforcement
- **Policy initiatives** to bundle related policies
- **Tags** for cost allocation and operational metadata
- **Landing zones** for scalable subscription design

A landing zone approach is common when an organization needs multiple subscriptions with shared governance. A typical pattern is:

- Platform subscriptions for identity, connectivity, and management
- Application subscriptions for workloads
- Separate subscriptions for dev, test, and production when isolation matters

Use policy initiatives when one requirement needs multiple controls, such as enforcing allowed regions, required tags, and diagnostic settings together.

**Monitoring and observability**

Monitoring is not just “turn on logs.” The exam often asks how you centralize telemetry, route alerts, and support operations.

Key design elements include:

- **Azure Monitor** for metrics, logs, and alerting
- **Log Analytics workspaces** for centralized log storage and KQL analysis
- **Diagnostic settings** to send resource logs and metrics to a workspace, storage account, or event hub
- **Action groups** to route alerts to email, SMS, webhook, ITSM, or automation
- **Application Insights** for application performance monitoring
- **Workbook** and dashboard views for operational reporting

A common enterprise pattern is to send diagnostic logs from many subscriptions into a central Log Analytics workspace in a management subscription. This simplifies investigation and retention management.

```bash
az monitor diagnostic-settings create \
  --name send-to-law \
  --resource /subscriptions/<sub-id>/resourceGroups/rg-app/providers/Microsoft.Web/sites/app-prod \
  --workspace /subscriptions/<sub-id>/resourceGroups/rg-monitor/providers/Microsoft.OperationalInsights/workspaces/law-prod \
  --logs '[{"category":"AppServiceHTTPLogs","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

> Diagnostic settings are often required for a complete monitoring design. If you only mention alerts without log collection, the design is usually incomplete.

**Alert routing and operational response**

A good monitoring design separates signal collection from response routing.

- Use **metric alerts** for near-real-time threshold conditions.
- Use **log alerts** for query-based detection.
- Use **action groups** to notify the right team or trigger automation.
- Use **automation runbooks** or Logic Apps when the response should be automated.

For example, a production web app might trigger:
- a metric alert for CPU saturation,
- a log alert for repeated 5xx responses,
- and an action group that pages the on-call team.

**Data storage and business continuity**

This domain combines storage selection, database design, backup, replication, and disaster recovery. The exam often tests whether you understand the difference between high availability, backup, and disaster recovery.

**Storage account design**

Azure Storage is not one service; it is a family of services with different access patterns.

- **Blob Storage** is for unstructured objects, backups, and static content.
- **Azure Files** provides SMB/NFS-style file shares.
- **Queue Storage** supports simple messaging.
- **Table Storage** is a NoSQL key-value store for simple schema-less workloads.

Choose the access tier based on usage:

- **Hot** for frequent access
- **Cool** for infrequent access
- **Archive** for long-term retention with rehydration delay

Lifecycle management helps move data between tiers automatically. This is useful when compliance requires retention but access is rare.

```json
{
  "rules": [
    {
      "enabled": true,
      "name": "move-to-cool",
      "type": "Lifecycle",
      "definition": {
        "filters": {
          "blobTypes": ["blockBlob"],
          "prefixMatch": ["logs/"]
        },
        "actions": {
          "baseBlob": {
            "tierToCool": { "daysAfterModificationGreaterThan": 30 },
            "tierToArchive": { "daysAfterModificationGreaterThan": 180 }
          }
        }
      }
    }
  ]
}
```

**Encryption and key management**

Storage and database designs often require encryption at rest and sometimes customer-managed keys.

- **Microsoft-managed keys** are the default and simplest option.
- **Customer-managed keys (CMK)** give the customer control over key rotation and revocation.
- **Key Vault** or **Managed HSM** can store and manage keys depending on the security requirement.

Use CMK when the scenario mentions regulatory control, key ownership, or separation of duties. Be aware that CMK adds operational complexity and dependency on key availability.

**Database design tradeoffs**

AZ-305 often asks you to choose between relational, NoSQL, and managed platform services.

- **Azure SQL Database** fits modern relational workloads with PaaS benefits.
- **Azure SQL Managed Instance** is useful when you need near-full SQL Server compatibility and instance-level features.
- **Azure Database for PostgreSQL/MySQL** fits open-source relational workloads.
- **Azure Cosmos DB** fits globally distributed, low-latency, schema-flexible workloads.

For Cosmos DB, understand the tradeoffs:

- **Multi-region reads** improve read latency.
- **Multi-region writes** support active-active write patterns but require conflict resolution planning.
- **Consistency levels** trade latency and availability against read guarantees.

Table storage-style workloads can sometimes be migrated to Cosmos DB Table API, but they are not identical services. Cosmos DB adds global distribution, throughput provisioning, and stronger platform capabilities, while Table Storage is simpler and cheaper for basic key-value access patterns.

**Backup versus replication versus DR**

These terms are not interchangeable.

- **Backup** protects against deletion, corruption, and ransomware-like scenarios.
- **Replication** improves availability and read locality.
- **Disaster recovery (DR)** restores service after a regional or site failure.

A backup strategy should define:
- retention,
- restore testing,
- recovery point objective (RPO),
- and recovery time objective (RTO).

Azure Backup is workload-specific. It does not natively cover every Azure service in the same way, so always verify support for the exact workload. For example, some databases rely on built-in platform backup and geo-redundancy rather than Azure Backup vaults.

**Business continuity patterns**

A strong AZ-305 answer distinguishes between zone resilience and region resilience.

- **Availability Zones** protect against datacenter-level failures within a region.
- **Region pairs** support regional disaster recovery.
- **Active-active** designs serve traffic from multiple locations simultaneously.
- **Active-passive** designs keep a standby environment ready for failover.

Use zone-redundant services when the requirement is high availability within a region. Use region-redundant designs when the requirement includes regional outage tolerance.

**Service-specific DR examples**

- **Azure SQL Database**
  - Use active geo-replication for readable secondary databases.
  - Use auto-failover groups when you need managed failover across regions.
- **Azure SQL Managed Instance**
  - Use failover groups for regional DR.
- **Azure Storage**
  - Use geo-redundant storage options when cross-region durability is required.
- **Azure Cosmos DB**
  - Use multi-region distribution and choose write topology based on latency and conflict requirements.

> A common exam trap is confusing high availability with disaster recovery. Zone redundancy helps with local failures; it does not replace a regional DR plan.

**Infrastructure**

Infrastructure questions focus on compute, networking, hybrid connectivity, routing, and name resolution. This is where many scenario-based questions become design tradeoff questions.

**Compute selection**

Choose the compute platform based on control, scaling, and operational overhead.

- **Virtual Machines** give the most control and are best for legacy or specialized OS requirements.
- **Virtual Machine Scale Sets** help with horizontal scaling of VM-based workloads.
- **App Service** is a managed web hosting platform for HTTP-based applications.
- **Azure Container Apps** and **AKS** support containerized workloads with different levels of control.
- **Functions** fit event-driven, serverless workloads.

A practical rule:
- Use **App Service** when you need managed web hosting with minimal infrastructure management.
- Use **AKS** when you need Kubernetes control, complex microservices, or platform standardization.
- Use **VMs** only when the workload needs OS-level access or cannot be modernized.

**Networking fundamentals**

Networking is one of the most heavily tested areas in AZ-305. You should understand:

- **Virtual networks (VNets)** and subnets
- **Network security groups (NSGs)** for subnet or NIC filtering
- **User-defined routes (UDRs)** for custom routing
- **Azure Firewall** for centralized inspection and policy
- **Private endpoints** for private access to PaaS services
- **Service endpoints** for secure service access over the Azure backbone
- **VPN Gateway** and **ExpressRoute** for hybrid connectivity

**Hub-spoke design**

Hub-spoke is a common enterprise topology.

- The **hub** hosts shared services such as firewall, DNS, VPN/ExpressRoute gateway, and monitoring.
- The **spokes** host application workloads.
- **Peering** connects spokes to the hub and, when needed, to each other through controlled routing.

A typical hub-spoke flow looks like this:

```text
On-premises
   |
ExpressRoute / VPN
   |
[Hub VNet] -- Azure Firewall -- DNS / Bastion / Gateway
   |
   +-- [Spoke VNet: App]
   +-- [Spoke VNet: Data]
   +-- [Spoke VNet: Dev]
```

Design considerations:

- Use **UDRs** to force spoke traffic through the firewall when inspection is required.
- Use **DNS forwarding** or Azure DNS Private Resolver when private name resolution must work across spokes and on-premises.
- Avoid unnecessary transitive assumptions; VNet peering is not a full routing fabric by default.

**DNS design**

Azure DNS design is often underexplained, but it matters in private connectivity scenarios.

Common patterns include:

- **Azure DNS public zones** for internet-facing names
- **Private DNS zones** for private endpoint name resolution
- **Custom DNS servers** when integrating with on-premises DNS or advanced name resolution
- **Azure DNS Private Resolver** for managed inbound and outbound DNS forwarding

Private endpoints usually require a private DNS zone link so clients resolve the service name to the private IP address. Without DNS integration, the endpoint may exist but clients still resolve the public address and fail or bypass the intended path.

**Private endpoint design**

Private endpoints are used to expose PaaS services privately inside a VNet.

Key requirements:

- Use a **dedicated subnet** for private endpoints.
- Disable private endpoint network policies on that subnet as required.
- Plan **private DNS zone** creation and VNet linking.
- Consider split-horizon DNS if on-premises clients also need access.

A correct design usually includes:
1. The private endpoint resource.
2. The private DNS zone.
3. The VNet link to the zone.
4. DNS forwarding or resolver integration for hybrid clients.

```bash
az network private-endpoint create \
  --name pe-sql-prod \
  --resource-group rg-net \
  --vnet-name vnet-hub \
  --subnet pe-subnet \
  --private-connection-resource-id /subscriptions/<sub-id>/resourceGroups/rg-data/providers/Microsoft.Sql/servers/sql-prod \
  --group-ids sqlServer

az network private-dns zone create \
  --resource-group rg-net \
  --name privatelink.database.windows.net

az network private-dns link vnet create \
  --resource-group rg-net \
  --zone-name privatelink.database.windows.net \
  --name link-hub \
  --virtual-network vnet-hub \
  --registration-enabled false
```

**Hybrid connectivity**

Choose the connectivity method based on bandwidth, latency, and reliability.

- **Site-to-site VPN** is cost-effective and suitable for smaller or less demanding environments.
- **ExpressRoute** provides private connectivity with more predictable performance and enterprise-grade design.
- **ExpressRoute FastPath** can improve data path performance for supported scenarios.

Use ExpressRoute when the scenario mentions:
- high throughput,
- predictable latency,
- private connectivity,
- or strict enterprise networking requirements.

**Routing and network virtual appliances**

Network virtual appliances (NVAs) are third-party or custom appliances used for inspection, routing, or security functions.

Use NVAs when:
- the organization already standardizes on a vendor,
- advanced inspection is required,
- or a specific security stack is mandated.

Use Azure Firewall when:
- you want a managed, scalable firewall service,
- centralized policy is preferred,
- and the scenario does not require a specialized third-party appliance.

Route tables and UDRs are important when you need to:
- force egress through a firewall,
- control spoke-to-spoke traffic,
- or direct traffic to an NVA.

**Application architecture**

This domain focuses on how you host applications, integrate services, and design for scale and resilience.

**App hosting choices**

- **App Service** is ideal for web apps and APIs with managed scaling.
- **Azure Functions** is ideal for event-driven tasks and lightweight APIs.
- **Container Apps** is useful for microservices and event-driven containers without full Kubernetes overhead.
- **AKS** is best when you need Kubernetes control, custom networking, or platform standardization.

A common exam pattern is choosing between App Service and AKS. Prefer App Service unless the scenario explicitly needs container orchestration features, sidecars, custom scheduling, or Kubernetes ecosystem tooling.

**Integration patterns**

Azure architecture questions often involve decoupling components.

- **Service Bus** is for reliable enterprise messaging, queues, and topics.
- **Event Grid** is for event routing and reactive architectures.
- **Event Hubs** is for high-throughput telemetry and streaming ingestion.
- **Logic Apps** is for workflow orchestration and SaaS integration.

Choose based on message semantics:

- Use **Service Bus** when ordering, transactions, or durable queue semantics matter.
- Use **Event Grid** when you need event notification and fan-out.
- Use **Event Hubs** when ingesting large volumes of telemetry or streaming data.

**Caching and performance**

Use **Azure Cache for Redis** when you need:
- session state,
- low-latency lookups,
- or offloading repeated reads from a backend database.

Caching improves performance, but it adds invalidation and consistency concerns. Do not use it as a substitute for proper data modeling.

**API and frontend design**

For API-centric solutions, consider:

- **API Management** for publishing, securing, throttling, and transforming APIs
- **Front Door** for global HTTP(S) routing, acceleration, and edge protection
- **Application Gateway** for regional web traffic management and WAF integration

A common pattern is:
- Front Door for global entry,
- Application Gateway for regional ingress,
- App Service or AKS behind the gateway.

**End-to-end scenario patterns**

These patterns are useful because AZ-305 questions often combine multiple domains.

**Pattern 1: Hub-spoke enterprise landing zone**

Use this when the organization wants centralized governance and shared network services.

- Management groups define policy inheritance.
- A platform subscription hosts hub networking, firewall, and DNS.
- Application subscriptions host workloads.
- Private endpoints connect PaaS services privately.
- Diagnostic settings send logs to a central workspace.

Why this works:
- It scales across teams.
- It keeps shared services centralized.
- It supports policy-driven governance.

**Pattern 2: Multi-region web application**

Use this when the app must survive a regional outage.

- Front Door routes users to the nearest healthy region.
- Each region hosts a web tier and supporting services.
- Data uses geo-redundant or multi-region replication.
- Health probes determine failover.
- Monitoring and alerts are centralized.

Tradeoffs:
- Active-active improves availability but increases complexity.
- Active-passive is simpler but may have longer failover times.

**Pattern 3: Secure PaaS data access**

Use this when a database or storage account must not be publicly reachable.

- Deploy a private endpoint.
- Place it in a dedicated subnet.
- Link the correct private DNS zone.
- Restrict public network access.
- Use managed identity for application authentication.

This pattern is common for SQL, Storage, Key Vault, and other PaaS services.

**Pattern 4: Backup and recovery for critical workloads**

Use this when the scenario emphasizes restore capability and retention.

- Define RPO and RTO.
- Use platform-native backup where supported.
- Add geo-redundancy or replication if regional failure is in scope.
- Test restores regularly.
- Store backups in a separate security boundary when required.

**Decision criteria cheat sheet**

| Requirement | Prefer | Avoid when |
|---|---|---|
| Lowest operational overhead | PaaS services | You need OS-level control |
| Private access to PaaS | Private endpoints | Public access is acceptable and simpler |
| Regional outage tolerance | Multi-region design | Only local availability is required |
| Fast web app deployment | App Service | You need Kubernetes-specific features |
| Enterprise messaging | Service Bus | You only need simple event notification |
| Telemetry ingestion | Event Hubs | You need command-style queue semantics |
| Centralized governance | Management groups + Policy | The environment is a single small subscription |
| Hybrid private connectivity | ExpressRoute | VPN bandwidth and latency are sufficient |

**Common exam traps**

- Confusing **availability zones** with **regions**.
- Assuming **backup** equals **replication**.
- Using **VMs** when a managed service fits better.
- Forgetting **DNS** in private endpoint designs.
- Choosing **Event Grid** when durable queue semantics are required.
- Treating **Azure Backup** as universal for all workloads.
- Overlooking **diagnostic settings** and **action groups** in monitoring designs.
- Designing hub-spoke routing without considering **UDRs** and firewall inspection.

**Practical CLI examples**

These examples are intentionally small and realistic, but you should still verify parameter requirements in your environment and CLI version.

Create a resource group and a Log Analytics workspace:

```bash
az group create \
  --name rg-monitor \
  --location eastus

az monitor log-analytics workspace create \
  --resource-group rg-monitor \
  --workspace-name law-prod \
  --location eastus
```

Create a metric alert with an action group:

```bash
az monitor action-group create \
  --resource-group rg-monitor \
  --name ag-ops \
  --short-name ops \
  --action email ops-team ops@example.com

az monitor metrics alert create \
  --name cpu-high \
  --resource-group rg-monitor \
  --scopes /subscriptions/<sub-id>/resourceGroups/rg-app/providers/Microsoft.Web/sites/app-prod \
  --condition "avg Percentage CPU > 80" \
  --window-size 5m \
  --evaluation-frequency 1m \
  --action ag-ops
```

Create a private endpoint and private DNS zone link:

```bash
az network private-dns zone create \
  --resource-group rg-net \
  --name privatelink.blob.core.windows.net

az network private-dns link vnet create \
  --resource-group rg-net \
  --zone-name privatelink.blob.core.windows.net \
  --name link-app \
  --virtual-network vnet-app \
  --registration-enabled false
```

**Study strategy for AZ-305**

To prepare effectively, study by scenario rather than by service list.

1. Learn the core service families and what problem each solves.
2. Practice choosing between similar services, such as App Service vs AKS or Service Bus vs Event Grid.
3. Review tradeoffs for availability, cost, security, and operational complexity.
4. Build mental patterns for hub-spoke, private endpoints, multi-region DR, and landing zones.
5. Validate every design against the requirement, not against personal preference.

If you can explain why a design is chosen, what it protects against, and what it costs operationally, you are thinking in the way AZ-305 expects.

**Quick recall summary**

- **Identity and governance** control access, standards, and compliance.
- **Monitoring** requires logs, metrics, alerts, and action routing.
- **Storage and databases** require careful choices around access tier, replication, encryption, and recovery.
- **Infrastructure** depends on compute fit, routing, DNS, and hybrid connectivity.
- **Application architecture** is about hosting, integration, and scaling with the least complexity that meets the need.
