## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**How to approach AZ-305 scenario questions**

AZ-305 questions usually describe business, technical, security, and operational constraints at the same time. Your job is to choose the design that best satisfies the stated requirements with the least unnecessary complexity.

A strong answer usually balances:

- **Fit for purpose**: the service matches the workload and constraints.
- **Operational simplicity**: fewer moving parts when requirements allow it.
- **Security by design**: identity, network, and data protection are built in.
- **Scalability and resilience**: the design meets growth, availability, and recovery targets.
- **Cost awareness**: avoid over-engineering when the scenario does not require it.

When you read a scenario, identify the hard requirements first:

- Compliance or regulatory constraints
- RTO and RPO targets
- Identity and access requirements
- Network connectivity and isolation needs
- Data platform and integration needs
- Monitoring, logging, and governance expectations

**Decision-making pattern**

Use a simple filter:

1. Identify the workload type.
2. Identify the non-negotiable constraints.
3. Eliminate services that cannot meet those constraints.
4. Choose the simplest service that satisfies the scenario.
5. Prefer native Azure integrations when they reduce operational burden.

**Identity and access design**

Identity is often the first design decision in enterprise scenarios. Microsoft Entra ID is the central identity provider for Azure, Microsoft 365, and many SaaS integrations. In AZ-305, you often need to decide how users, applications, and administrators authenticate and authorize.

Key concepts:

- **Microsoft Entra ID** is the cloud identity platform formerly known as Azure Active Directory.
- **Managed identities** let Azure resources authenticate to other Azure services without storing credentials.
- **Service principals** represent applications or automation identities in Entra ID.
- **Conditional Access** enforces access rules based on user, device, location, risk, or app.
- **Privileged Identity Management (PIM)** provides just-in-time elevation for privileged roles.
- **Cross-tenant access** controls how users and apps interact across Entra tenants.

Use managed identities when the workload runs in Azure and needs to access Azure resources such as Key Vault, Storage, or SQL. Use service principals when the application must authenticate outside Azure or when a managed identity is not available.

> Use managed identities whenever possible because they reduce secret management overhead and lower the risk of credential leakage.

**Identity design examples**

- Use **system-assigned managed identity** for a single Azure resource lifecycle.
- Use **user-assigned managed identity** when multiple resources share the same identity or when you need identity persistence across resource replacement.
- Use **PIM** for subscription owners, security admins, and other high-impact roles.
- Use **Conditional Access** to require MFA, compliant devices, or trusted locations for sensitive apps.
- Use **cross-tenant access settings** when collaborating with partners or managing B2B access.

**Practical CLI examples**

Create a user-assigned managed identity:

```bash
az identity create \
  --name app-uami \
  --resource-group rg-app \
  --location eastus
```

Assign a role to the managed identity at a storage account scope:

```bash
az role assignment create \
  --assignee-object-id <principalId> \
  --assignee-principal-type ServicePrincipal \
  --role "Storage Blob Data Contributor" \
  --scope /subscriptions/<subId>/resourceGroups/rg-data/providers/Microsoft.Storage/storageAccounts/stappdata
```

Enable a system-assigned identity on an App Service:

```bash
az webapp identity assign \
  --name app-web-prod \
  --resource-group rg-app
```

**When to choose each identity control**

| Requirement | Best fit | Why |
|---|---|---|
| Azure workload needs to read Key Vault secrets | Managed identity | No stored credentials |
| Admins need temporary elevation | PIM | Just-in-time access |
| Access must depend on device compliance | Conditional Access | Policy-based enforcement |
| Partner access across tenants | Cross-tenant access settings | Controlled external collaboration |
| App runs outside Azure | Service principal | Works with external hosts and automation |

**Networking design choices**

Networking questions often test whether you can choose the right connectivity, segmentation, and traffic flow pattern. Start by separating three concerns:

- **Connectivity**: how networks reach Azure and each other
- **Segmentation**: how you isolate workloads
- **Traffic control**: how you inspect and route traffic

**ExpressRoute vs VPN Gateway**

Choose **ExpressRoute** when the scenario needs private, predictable connectivity with higher reliability, lower latency variability, or compliance requirements that favor private connectivity. Choose **VPN Gateway** when you need faster setup, lower cost, or a backup path.

| Requirement | ExpressRoute | VPN Gateway |
|---|---|---|
| Private connectivity to Azure | Yes | No, uses public internet |
| Predictable latency | Better | Variable |
| Fastest to deploy | No | Yes |
| Lower cost | Usually no | Usually yes |
| Hybrid production workloads | Often preferred | Often acceptable for smaller or backup scenarios |

**Peering vs hub-and-spoke**

Use **virtual network peering** for direct connectivity between VNets with low latency and simple routing. Use **hub-and-spoke** when you need centralized shared services such as firewall, DNS, Bastion, or inspection.

- **Hub-and-spoke** is common in enterprise landing zones.
- The hub hosts shared network services.
- Spokes host application workloads.
- Route control is centralized through the hub.

Use peering alone when the environment is small and does not need centralized inspection. Use hub-and-spoke when you need governance, shared security controls, or multiple spokes.

**Azure Firewall vs NSGs**

- **Network Security Groups (NSGs)** filter traffic at subnet or NIC level using 5-tuple rules.
- **Azure Firewall** provides centralized, stateful network filtering, FQDN filtering, threat intelligence, and logging.

Use NSGs for basic segmentation and least-privilege subnet control. Use Azure Firewall when you need centralized egress control, application-level filtering, or inspection across multiple spokes.

**Application Gateway vs Load Balancer**

- **Azure Load Balancer** is Layer 4 and handles TCP/UDP distribution.
- **Application Gateway** is Layer 7 and supports HTTP(S), path-based routing, TLS termination, and Web Application Firewall (WAF).

Choose Application Gateway for web apps that need URL routing, cookie affinity, or WAF. Choose Load Balancer for non-HTTP workloads, internal services, or simple high availability at Layer 4.

**Common network topologies**

- **Hub-and-spoke** for enterprise shared services and inspection
- **Mesh** for tightly coupled peer workloads, though it becomes harder to manage at scale
- **Virtual WAN** for large-scale branch, remote user, and global connectivity scenarios
- **Private endpoints** for private access to PaaS services when supported by the service

> Azure Policy can help audit or deny unsupported public exposure patterns, but it does not universally force private endpoint usage for every service.

**Storage design choices**

Storage questions often focus on durability, access tier, performance, and data lifecycle. Choose the storage type based on how the data is used, how often it is accessed, and how much resilience is required.

**Blob Storage tiers**

- **Hot**: frequent access, lower access costs, higher storage cost
- **Cool**: infrequent access, lower storage cost, higher access costs
- **Archive**: long-term retention, very low storage cost, rehydration delay

Use Hot for active application data, Cool for infrequently accessed backups or logs, and Archive for long-term retention or compliance data.

**Redundancy options**

- **LRS**: Locally redundant storage within one datacenter
- **ZRS**: Zone-redundant storage across availability zones in a region
- **GRS**: Geo-redundant storage with asynchronous replication to a paired region
- **GZRS**: Zone-redundant in the primary region plus geo-replication

Choose based on availability and disaster recovery requirements:

- Use **LRS** for dev/test or non-critical data.
- Use **ZRS** when zone failure tolerance matters within a region.
- Use **GRS** when regional disaster recovery is required.
- Use **GZRS** when you want both zone resilience and geo-redundancy.

**Azure Files vs Blob Storage vs managed disks**

| Service | Best for | Notes |
|---|---|---|
| Azure Files | Shared SMB/NFS file shares | Good for lift-and-shift and shared app content |
| Blob Storage | Object data, logs, backups, archives | Best for unstructured data |
| Managed disks | VM operating system and data disks | Best for IaaS VM storage |

Use Azure Files when multiple servers need a shared file system. Use Blob Storage for application objects, media, backups, and archival data. Use managed disks for VM-based workloads.

**Practical storage examples**

Create a storage account with zone-redundant storage where supported:

```bash
az storage account create \
  --name stproddata01 \
  --resource-group rg-data \
  --location eastus2 \
  --sku Standard_ZRS \
  --kind StorageV2
```

Upload a blob to a hot container:

```bash
az storage container create \
  --name appdata \
  --account-name stproddata01

az storage blob upload \
  --account-name stproddata01 \
  --container-name appdata \
  --name report.csv \
  --file ./report.csv
```

**Backup and disaster recovery**

AZ-305 scenarios often distinguish between backup and disaster recovery. Backup protects data from deletion, corruption, or ransomware. Disaster recovery restores service after a site or region failure.

**Azure Backup**

Use **Azure Backup** for centralized backup of Azure VMs, SQL workloads, Azure Files, and some on-premises workloads. It is the right choice when the scenario asks for:

- Point-in-time recovery
- Retention policies
- Backup immutability or soft delete features
- Centralized backup management

**Recovery Services vault vs Backup vault**

- **Recovery Services vault** is commonly used for Azure VM backup and Azure Site Recovery.
- **Backup vault** is used for newer backup scenarios and some workload types depending on service support.

**Azure Site Recovery**

Use **Azure Site Recovery (ASR)** when the requirement is failover to another region or site with low RTO and controlled replication. ASR is a disaster recovery orchestration service, not a backup service.

Choose ASR when:

- The workload must fail over quickly
- You need test failover capability
- You want orchestrated recovery plans
- The scenario specifies RTO/RPO targets that backup alone cannot meet

**RTO and RPO**

- **RTO** (Recovery Time Objective): how long the service can be down
- **RPO** (Recovery Point Objective): how much data loss is acceptable

General guidance:

- Short RTO and low RPO often require replication or ASR.
- Longer RTO and moderate RPO may be satisfied by backup.
- Very strict requirements may require active-active or active-passive designs.

**Compute selection tradeoffs**

Compute questions often ask you to choose the least complex service that still meets runtime, scaling, and operational requirements.

**App Service**

Choose **Azure App Service** for web apps and APIs when you want managed hosting, built-in scaling, deployment slots, and minimal infrastructure management. It is often the best fit for standard web workloads.

Use App Service when:

- You need rapid deployment
- You want platform-managed patching
- You need HTTPS, custom domains, and autoscale
- You do not need OS-level control

**Azure Functions**

Choose **Azure Functions** for event-driven, short-lived, or bursty workloads. Functions are ideal for background processing, scheduled jobs, queue triggers, and lightweight APIs.

Use Functions when:

- The workload is event-based
- You want pay-per-execution economics
- You do not need a continuously running server
- You can tolerate cold start considerations in some plans

A working Function App usually requires a storage account and the right hosting plan and runtime settings. In current environments, creation often needs more than just runtime and Functions version.

**Azure Container Apps vs AKS**

- **Azure Container Apps** is good for microservices, event-driven containers, and simpler container operations.
- **Azure Kubernetes Service (AKS)** is best when you need Kubernetes control, advanced orchestration, custom networking, or platform engineering patterns.

Choose Container Apps when you want managed containers without Kubernetes overhead. Choose AKS when the team needs Kubernetes APIs, custom scheduling, service mesh, or complex container governance.

**Virtual Machines**

Use **VMs** when you need full OS control, legacy software, custom agents, or lift-and-shift workloads that cannot be refactored quickly. VMs are usually the least managed option and should be chosen when platform services cannot meet the requirement.

**Decision matrix for compute**

| Scenario | Best fit |
|---|---|
| Standard enterprise web app | App Service |
| Event-driven background processing | Functions |
| Microservices with minimal ops | Container Apps |
| Kubernetes-native platform | AKS |
| Legacy app requiring OS control | VM |

**Data platform choices**

Data questions often test whether you can distinguish transactional, relational, analytical, and integration workloads.

**Azure SQL Database vs SQL Managed Instance**

- **Azure SQL Database** is a fully managed database service for modern relational apps.
- **Azure SQL Managed Instance** offers broader SQL Server compatibility and instance-level features.

Choose Azure SQL Database when the app can work with database-level features and you want a simpler managed service. Choose SQL Managed Instance when you need near-full SQL Server instance compatibility, such as SQL Agent or instance-scoped features.

> Managed Instance is not automatically the best choice for every SQL Server workload. If the app only needs a relational database and can modernize, Azure SQL Database may be the better fit.

**Synapse and analytics**

Use **Azure Synapse Analytics** when the scenario requires large-scale analytics, data warehousing, or integration of big data and SQL-based analytics. It fits scenarios where you need to ingest, transform, and analyze large volumes of data.

**Microsoft Fabric considerations**

Fabric is relevant when the scenario emphasizes unified analytics, data engineering, and business intelligence in a SaaS-style analytics platform. If the question focuses on modern analytics consolidation, Fabric may be a strong candidate, but choose it only when the scenario aligns with its managed analytics model.

**Cosmos DB**

Use **Azure Cosmos DB** for globally distributed, low-latency, schema-flexible applications. It supports transactional operations within a partition, so do not treat it as “non-transactional.” It is best when you need:

- Multi-region distribution
- Flexible data models
- Elastic scale
- Low-latency reads and writes

Choose a relational database instead when the scenario clearly requires relational joins, complex relational integrity, or traditional SQL-centric transactional design.

**Integration and messaging**

- **Service Bus** for reliable enterprise messaging, queues, and topics
- **Event Grid** for event routing and reactive architectures
- **Event Hubs** for high-throughput telemetry and streaming ingestion
- **Logic Apps** for workflow orchestration and SaaS integration

Use Service Bus when you need ordered, durable business messaging. Use Event Grid for lightweight event notification. Use Event Hubs for telemetry pipelines and streaming data. Use Logic Apps when the scenario is integration-heavy and low-code orchestration is acceptable.

**Security architecture**

Security design in AZ-305 often combines identity, secrets, encryption, and network controls.

**Key Vault**

Use **Azure Key Vault** to store secrets, keys, and certificates. It is the standard choice for secret management and key protection.

Use Key Vault when you need:

- Secret rotation
- Certificate lifecycle management
- Customer-managed keys
- Centralized secret access control

**Customer-managed keys**

Use **customer-managed keys (CMK)** when the scenario requires the customer to control encryption keys rather than relying only on Microsoft-managed keys. This is common in regulated environments.

**Zero Trust**

Zero Trust means you do not trust network location by default. Instead, you verify explicitly, use least privilege, and assume breach.

Practical Zero Trust patterns:

- Authenticate every request
- Use least privilege RBAC
- Segment networks
- Prefer private access paths where supported
- Log and monitor access continuously

**Governance and landing zone design**

Governance questions often ask how to structure subscriptions, management groups, and policy assignments.

A common enterprise hierarchy is:

1. **Tenant root group**
2. **Platform management group**
3. **Landing zones management group**
4. **Sandbox management group**
5. **Subscriptions** under each branch

Typical segmentation:

- **Platform subscriptions** for identity, connectivity, and management
- **Application subscriptions** for workloads
- **Sandbox subscriptions** for experimentation
- **Shared services subscriptions** for common enterprise services

**Policy and RBAC**

Use **Azure Policy** to enforce standards such as allowed regions, required tags, SKU restrictions, and secure configuration baselines. Use **RBAC** to control who can do what.

Examples of policy use:

- Require tags like `CostCenter` and `Owner`
- Restrict resource deployment to approved regions
- Audit public IP usage
- Deny unsupported SKUs
- Audit diagnostic settings on critical resources

Sample policy assignment:

```bash
az policy assignment create \
  --name require-tags \
  --policy "/providers/Microsoft.Authorization/policyDefinitions/require-tags" \
  --scope /subscriptions/<subId>
```

**Monitoring and observability**

Monitoring questions usually ask how to detect issues, collect logs, and support operations.

**Azure Monitor**

Use **Azure Monitor** as the central platform for metrics, logs, alerts, and dashboards. It is the default answer when the scenario asks for unified observability.

**Log Analytics**

Use **Log Analytics workspace** for centralized log collection and Kusto Query Language (KQL) analysis. It is often the destination for diagnostic logs and platform logs.

**Application Insights**

Use **Application Insights** for application performance monitoring, request tracing, dependency tracking, and exception analysis.

**Diagnostic settings**

Use diagnostic settings to send resource logs and metrics to Log Analytics, Storage, or Event Hubs. This is a common design decision for auditability and troubleshooting.

**Alerts and dashboards**

- Use **metric alerts** for near-real-time threshold monitoring.
- Use **log alerts** for query-based detection.
- Use **workbooks** or dashboards for operational views.

Example diagnostic settings pattern:

```bash
az monitor diagnostic-settings create \
  --name send-to-law \
  --resource <resourceId> \
  --workspace <logAnalyticsWorkspaceId> \
  --logs '[{"category":"AuditEvent","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

**Common scenario patterns**

**Lift-and-shift web app**

- Use App Service if the app can run on a managed platform.
- Use VM if the app needs OS-level control or cannot be refactored.
- Add Application Gateway if the app needs WAF or Layer 7 routing.
- Use managed identity for backend access to Key Vault or Storage.

**Hybrid enterprise connectivity**

- Use ExpressRoute for primary private connectivity.
- Use VPN Gateway as backup or for smaller sites.
- Use hub-and-spoke with Azure Firewall for centralized inspection.
- Use private DNS and private endpoints where supported.

**Highly available transactional app**

- Use zone-redundant deployment where possible.
- Use Azure SQL Database or SQL Managed Instance based on compatibility needs.
- Use Application Gateway for web ingress.
- Use Azure Monitor and alerts for operations.
- Use Azure Backup for data protection and ASR if regional failover is required.

**Analytics platform**

- Use Event Hubs for ingestion.
- Use Synapse or Fabric for analytics depending on the scenario.
- Use Blob Storage or ADLS Gen2 for raw data landing.
- Use Log Analytics only for operational logs, not as the primary analytics warehouse.

**Exam-style service selection reminders**

- Choose **App Service** before VMs for standard web apps.
- Choose **managed identity** before secrets in code.
- Choose **Azure Firewall** when centralized egress control matters.
- Choose **Application Gateway** for HTTP(S) routing and WAF.
- Choose **ExpressRoute** for private enterprise connectivity.
- Choose **Azure Backup** for restore needs and **ASR** for failover needs.
- Choose **Azure SQL Database** for modern relational apps and **Managed Instance** for higher SQL Server compatibility.
- Choose **Azure Monitor** for observability and **Key Vault** for secrets and keys.

**Practical design checklist**

Before selecting an answer, ask:

- Does the workload need PaaS, containers, or IaaS?
- Is identity managed by Entra ID, and can you use managed identities?
- Does the network need private connectivity, inspection, or segmentation?
- Is the data relational, analytical, object-based, or event-driven?
- Are the RTO and RPO targets met by backup, replication, or both?
- Do you need governance controls such as policy, RBAC, or PIM?
- What is the simplest service that still satisfies the requirement?

**Final exam strategy**

Scenario questions reward precision. Avoid choosing a service because it is powerful if a simpler service already meets the requirement. Read for constraints, not just keywords.

If two answers both seem plausible:

- Prefer the one that matches the stated requirement most directly.
- Prefer managed services over self-managed infrastructure.
- Prefer native Azure integrations over custom glue.
- Prefer designs that reduce operational burden without violating constraints.
