## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Capstone synthesis mindset**

Capstone questions test whether you can combine isolated Azure services into a coherent architecture that satisfies business goals. You are not just selecting a service; you are balancing availability, security, governance, performance, operations, and cost.

A strong answer usually starts by identifying the workload type and the nonfunctional requirements:

- **Availability** targets such as uptime, region tolerance, and recovery objectives
- **Security** needs such as identity, network isolation, encryption, and secrets handling
- **Scalability** requirements such as burst traffic, global reach, or seasonal demand
- **Operations** needs such as monitoring, patching, deployment, and backup
- **Cost** constraints such as reserved capacity, autoscaling, and service tier selection

The exam often rewards the option that best fits the stated requirement, not the most feature-rich option. If a requirement says “minimize operational overhead,” a managed service is often better than a self-managed VM-based design. If it says “keep data in a specific region,” global replication choices must respect data residency.

**How to decompose a scenario**

Use a repeatable reading pattern so you do not miss constraints hidden in the wording.

1. Identify the business outcome.
2. Extract hard constraints such as region, compliance, budget, and downtime tolerance.
3. Separate functional requirements from nonfunctional requirements.
4. Map each requirement to one or more Azure design decisions.
5. Eliminate answers that violate a hard constraint, even if they look technically attractive.

A practical way to think about the process is to build a mental matrix:

| Requirement type | What to look for | Typical Azure design response |
|---|---|---|
| Availability | RTO, RPO, multi-region, zone tolerance | Availability Zones, zone-redundant services, failover groups, Traffic Manager, Front Door |
| Security | Identity, secrets, network boundaries, encryption | Microsoft Entra ID, managed identities, Key Vault, private endpoints, RBAC |
| Performance | Latency, throughput, global users | CDN, Front Door, caching, read replicas, autoscale |
| Operations | Monitoring, patching, deployment, backup | Azure Monitor, Log Analytics, automation, PaaS, backup policies |
| Governance | Policy, tagging, subscriptions, landing zones | Management groups, Azure Policy, RBAC, resource locks |
| Cost | Budget, licensing, utilization | right-sizing, reservations, autoscale, PaaS tiers, storage lifecycle rules |

**Architecture decision priorities**

When multiple services could work, prioritize the one that best matches the requirement hierarchy:

- **Hard constraints first**
  - Regulatory or data residency requirements
  - Required recovery objectives
  - Network isolation requirements
  - Identity and access boundaries
- **Then operational fit**
  - Managed service over self-managed infrastructure when the goal is lower administration
  - Native integration over custom glue when the goal is simplicity
- **Then optimization**
  - Cost efficiency
  - Performance tuning
  - Secondary features such as analytics or advanced routing

> If a design violates a stated constraint, it is wrong even if it is otherwise a best practice.

**Identity and access design**

Identity is usually the first control plane decision in a capstone scenario. Use Microsoft Entra ID for workforce identity, and prefer managed identities for Azure-to-Azure authentication so you avoid storing secrets in code or configuration.

Common patterns include:

- **User access**
  - Use Microsoft Entra ID groups for role assignment
  - Apply least privilege with Azure RBAC
  - Use Privileged Identity Management for just-in-time elevation when administrative control is required
- **Application access**
  - Use system-assigned managed identity for a single resource lifecycle
  - Use user-assigned managed identity when multiple resources share the same identity or when identity persistence matters
  - Store secrets and certificates in Azure Key Vault only when a managed identity cannot eliminate the secret
- **Administrative access**
  - Use Conditional Access, MFA, and break-glass accounts for critical operations
  - Separate duties with role-based access control and management group boundaries

Example CLI for assigning a role to a managed identity:

```bash
az role assignment create \
  --assignee-object-id <managed-identity-object-id> \
  --assignee-principal-type ServicePrincipal \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg-name>/providers/Microsoft.KeyVault/vaults/<kv-name>
```

The important detail is the principal object ID for the managed identity’s service principal in Microsoft Entra ID. In practice, verify the identity and scope carefully before assigning permissions.

**Network architecture synthesis**

Network design in AZ-305 is usually about choosing the right boundary and the right traffic path. Start with whether the workload needs public exposure, private access, or both.

Key building blocks:

- **Virtual network**
  - Provides IP address space and segmentation
  - Use subnets to separate tiers or functions
  - Apply network security groups for subnet or NIC-level filtering
- **Private Link and private endpoints**
  - Expose PaaS services privately through a VNet IP
  - Reduce public internet exposure
  - Useful for storage, SQL, Key Vault, and many other Azure services
- **Azure Firewall**
  - Centralized network and application rule inspection
  - Good for egress control, segmentation, and logging
  - Use when you need hub-and-spoke inspection and controlled outbound access
- **Web Application Firewall**
  - Protects HTTP/S applications at the application layer
  - Use with Application Gateway or Front Door for web traffic protection
  - Not a replacement for general network firewalling
- **VPN Gateway and ExpressRoute**
  - VPN Gateway for encrypted connectivity over the internet
  - ExpressRoute for private, more predictable connectivity to Azure
  - Choose based on latency, bandwidth, reliability, and compliance needs

A common enterprise topology is hub-and-spoke:

- **Hub**
  - Shared services such as firewall, DNS, bastion, and connectivity
- **Spokes**
  - Application VNets isolated by workload or environment
- **Peering**
  - Connects spokes to the hub and optionally to each other through controlled routing

Example hub-and-spoke routing concept:

```text
On-premises
   |
ExpressRoute or VPN
   |
Hub VNet ---- Azure Firewall ---- Internet
   |
   +---- Spoke VNet 1 (app)
   +---- Spoke VNet 2 (data)
   +---- Spoke VNet 3 (shared services)
```

**When to use private endpoints**

Use private endpoints when the requirement says the service must not be reachable from the public internet. This is especially common for:

- Storage accounts
- Azure SQL Database
- Key Vault
- Azure App Configuration
- Azure Container Registry

Remember that private endpoints change DNS behavior. You often need private DNS zones so clients resolve the service name to the private IP.

Example private DNS zone linkage:

```bash
az network private-dns zone create \
  --resource-group <rg-name> \
  --name privatelink.database.windows.net

az network private-dns link vnet create \
  --resource-group <rg-name> \
  --zone-name privatelink.database.windows.net \
  --name sql-link \
  --virtual-network <vnet-id> \
  --registration-enabled false
```

**Application hosting choices**

Choose the hosting model that matches the operational burden and scaling requirements.

- **Azure App Service**
  - Best for web apps and APIs with minimal infrastructure management
  - Supports deployment slots, autoscale, and managed certificates
  - Good when the scenario emphasizes rapid delivery and reduced ops
- **Azure Functions**
  - Best for event-driven, short-lived, or bursty workloads
  - Good for integration tasks, scheduled jobs, and lightweight APIs
- **Azure Kubernetes Service (AKS)**
  - Best when you need container orchestration and platform control
  - More operational complexity than App Service or Functions
  - Use when portability, microservices, or custom orchestration are explicit requirements
- **Virtual machines**
  - Best when you need OS-level control, legacy software, or unsupported dependencies
  - Usually the least preferred option when the goal is lower management overhead

A useful exam heuristic:

- If the scenario says “lift and shift,” VMs may be appropriate.
- If it says “minimize management,” prefer PaaS.
- If it says “event-driven,” consider Functions.
- If it says “containerized microservices,” consider AKS or Container Apps depending on the level of orchestration required.

**Data platform synthesis**

Data design questions often combine availability, performance, and consistency. The right answer depends on whether the workload is transactional, analytical, or globally distributed.

Common choices:

- **Azure SQL Database**
  - Managed relational database for modern applications
  - Supports high availability and scaling options
  - Good for transactional workloads
- **Azure SQL Managed Instance**
  - Closer compatibility with SQL Server features
  - Good for migration scenarios that need instance-level features
- **Azure Cosmos DB**
  - Globally distributed NoSQL database with multiple APIs
  - Good for low-latency global applications and flexible schema needs
- **Azure Storage**
  - Durable object, file, queue, and table storage
  - Good for unstructured data, backups, logs, and static content
- **Azure Database for PostgreSQL/MySQL**
  - Managed open-source relational options
  - Good when the application stack already depends on those engines

For disaster recovery, use the feature that matches the service:

- **Azure SQL Database / SQL Managed Instance**
  - Failover groups for regional disaster recovery
- **Storage**
  - Redundancy options such as LRS, ZRS, GRS, and GZRS depending on the service and scenario
- **Cosmos DB**
  - Multi-region replication and failover
- **Backups**
  - Use backup retention and restore features for point-in-time recovery where supported

> Failover groups are not a universal SQL feature for every SQL offering. Use them where the service supports them, and verify the exact service tier and deployment model.

**Resiliency and disaster recovery**

Resiliency is about surviving component failure without violating the business objective. Disaster recovery is about restoring service after a larger outage.

Important concepts:

- **RTO** (Recovery Time Objective)
  - Maximum acceptable downtime
- **RPO** (Recovery Point Objective)
  - Maximum acceptable data loss
- **Availability Zones**
  - Protect against datacenter-level failure within a region
- **Multi-region design**
  - Protect against regional failure
- **Active-active**
  - Multiple regions serve traffic simultaneously
- **Active-passive**
  - One region serves traffic while another waits for failover

Use the simplest design that meets the objective:

- If the requirement is zone resilience, use zone-redundant services or zone-aware deployment.
- If the requirement is regional resilience, use paired-region or multi-region architecture.
- If the requirement is global low latency, use edge routing and distributed data where appropriate.

Traffic routing options:

- **Azure Front Door**
  - Global HTTP/S load balancing and acceleration
  - Good for web apps needing global entry and fast failover
- **Traffic Manager**
  - DNS-based traffic distribution
  - Good for non-HTTP endpoints or when DNS-level routing is sufficient
- **Application Gateway**
  - Regional layer 7 load balancing and WAF
  - Good for internal or regional web traffic control

**Monitoring and operations**

Operational excellence is a major part of capstone synthesis. A good architecture includes observability from the start.

Use:

- **Azure Monitor**
  - Central monitoring platform
- **Log Analytics workspace**
  - Query and correlate logs with KQL (Kusto Query Language)
- **Application Insights**
  - Application performance monitoring for code-level telemetry
- **Alerts**
  - Trigger on metrics, logs, and activity events
- **Action groups**
  - Route alerts to email, SMS, webhook, ITSM, or automation

Example KQL query:

```kusto
AppRequests
| where TimeGenerated > ago(1h)
| summarize count(), avg(DurationMs) by ResultCode
| order by count_ desc
```

For operational design, think about:

- **What you need to detect**
  - Availability failures
  - Performance degradation
  - Security events
  - Cost anomalies
- **Who needs to respond**
  - Operations team
  - Security team
  - Application owners
- **How quickly they need to respond**
  - Immediate paging
  - Daily review
  - Monthly governance review

**Governance and landing zones**

Capstone scenarios often include enterprise governance requirements. Use management groups, subscriptions, policy, and tagging to create a scalable control structure.

Core tools:

- **Management groups**
  - Organize subscriptions by environment, business unit, or compliance boundary
- **Azure Policy**
  - Enforce allowed locations, required tags, SKU restrictions, and private endpoint requirements
- **RBAC**
  - Control who can manage resources
- **Resource locks**
  - Prevent accidental deletion or modification
- **Tags**
  - Support chargeback, ownership, and lifecycle management

Example policy assignment pattern:

```bash
az policy assignment create \
  --name require-tags \
  --display-name "Require costCenter and owner tags" \
  --scope /subscriptions/<sub-id> \
  --policy <policy-definition-id>
```

Use policy when the scenario says:

- “Only approved regions”
- “All resources must have tags”
- “Public IPs are not allowed”
- “Storage accounts must use private endpoints”
- “Only approved SKUs may be deployed”

**Security design patterns**

Security in AZ-305 is usually layered. Do not rely on a single control.

Recommended layers:

- **Identity**
  - Entra ID, MFA, Conditional Access, RBAC
- **Network**
  - Private endpoints, NSGs, firewall, segmentation
- **Data**
  - Encryption at rest, customer-managed keys when required, Key Vault
- **Application**
  - Managed identities, secrets rotation, secure headers, WAF for web apps
- **Operations**
  - Logging, alerting, Defender for Cloud, policy compliance

Practical examples:

- Use **Key Vault** for secrets, keys, and certificates.
- Use **customer-managed keys** when the scenario requires customer control over encryption keys.
- Use **Defender for Cloud** when the scenario asks for posture management or threat protection.
- Use **WAF** for HTTP/S protection against common web attacks such as SQL injection and cross-site scripting.

**Cost optimization without breaking requirements**

Cost is important, but it should not override explicit business constraints.

Common cost levers:

- **Autoscale**
  - Match capacity to demand
- **Reserved instances / reservations**
  - Reduce cost for steady-state workloads
- **Right-sizing**
  - Avoid overprovisioning
- **Storage tiering**
  - Move infrequently accessed data to cooler tiers
- **Lifecycle management**
  - Automatically archive or delete old data
- **PaaS over IaaS**
  - Reduce patching and operational overhead

Use cost-saving options only when they do not violate availability or performance requirements. For example, a cheaper single-region design is not acceptable if the scenario requires regional disaster recovery.

**Common exam traps**

Watch for wording that changes the correct answer:

- **“Publicly accessible” vs “internet-facing”**
  - Publicly accessible may still require authentication and WAF
- **“High availability” vs “disaster recovery”**
  - HA handles local failures; DR handles larger outages
- **“Low latency globally” vs “regional resilience”**
  - These often imply different routing and data replication patterns
- **“Minimize management” vs “maximum control”**
  - These point to PaaS versus IaaS
- **“Private connectivity” vs “secure connectivity”**
  - Private connectivity is a network path requirement; secure connectivity may also include encryption, identity, and inspection

> Do not assume a service is the right answer just because it is popular. Match the service to the exact requirement language.

**Practical synthesis examples**

Scenario: A global web application needs fast failover, edge acceleration, and protection for HTTP/S traffic.

- Use **Azure Front Door** for global routing and acceleration
- Use **WAF** for application-layer protection
- Deploy the app in multiple regions
- Use health probes and backend pools for failover

Scenario: An internal line-of-business app must access Azure SQL privately and avoid public endpoints.

- Place the app in a VNet-integrated hosting model or a VM in a VNet
- Use a **private endpoint** for Azure SQL Database
- Configure private DNS resolution
- Restrict public network access on the SQL server

Scenario: A regulated workload must keep secrets out of code and allow applications to authenticate to storage.

- Use a **managed identity** for the app
- Grant the identity the minimum required RBAC role
- Store any unavoidable secrets in **Key Vault**
- Enable logging and alerting for access events

Scenario: A migration from SQL Server requires instance-level compatibility and minimal code changes.

- Use **Azure SQL Managed Instance**
- Consider failover groups if regional DR is required
- Use private connectivity and subnet delegation as needed
- Validate feature compatibility before migration

**CLI and configuration snippets**

Create a resource group and a VNet for a simple landing zone:

```bash
az group create -n rg-app-prod -l eastus

az network vnet create \
  -g rg-app-prod \
  -n vnet-app-prod \
  --address-prefix 10.10.0.0/16 \
  --subnet-name snet-app \
  --subnet-prefix 10.10.1.0/24
```

Create a Key Vault and enable RBAC authorization:

```bash
az keyvault create \
  -g rg-app-prod \
  -n kvappprod001 \
  -l eastus \
  --enable-rbac-authorization true
```

Example App Service app settings for Key Vault reference:

```json
{
  "name": "ConnectionStrings__Sql",
  "value": "@Microsoft.KeyVault(SecretUri=https://kvappprod001.vault.azure.net/secrets/sql-conn/)"
}
```

Example Bicep fragment for a private endpoint concept:

```bicep
resource pe 'Microsoft.Network/privateEndpoints@2023-09-01' = {
  name: 'pe-sql'
  location: resourceGroup().location
  properties: {
    subnet: {
      id: subnetId
    }
    privateLinkServiceConnections: [
      {
        name: 'sql-connection'
        properties: {
          privateLinkServiceId: sqlServer.id
          groupIds: [
            'sqlServer'
          ]
        }
      }
    ]
  }
}
```

**Final exam approach**

When you reach the answer choices, compare them against the scenario in this order:

1. Does it satisfy every hard requirement?
2. Does it use the least complex service that still fits?
3. Does it reduce operational burden where requested?
4. Does it preserve security and governance requirements?
5. Does it scale and recover as required?
6. Does it fit the budget and licensing assumptions?

If two answers both seem plausible, choose the one that better matches the exact wording. AZ-305 rewards precision, not generic best practices.

**Quick recall checklist**

- **Identity first**: Entra ID, RBAC, managed identities
- **Network next**: VNet, private endpoints, firewall, WAF, routing
- **Data then**: service fit, replication, backup, DR
- **Operations always**: monitoring, alerts, logs, automation
- **Governance throughout**: policy, tags, management groups
- **Cost last**: optimize only after requirements are met
