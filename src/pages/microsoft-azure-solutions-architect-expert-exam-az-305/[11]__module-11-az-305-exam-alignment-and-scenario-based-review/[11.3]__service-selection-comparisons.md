## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**How to think about service selection**

AZ-305 questions usually test whether you can map a business requirement to the right Azure service, not whether you can recite feature lists. Start by identifying the dominant constraint:

- **Latency**: choose the service that minimizes hops and places traffic closest to users or workloads.
- **Security boundary**: choose the service that gives the right isolation model, such as private access, subnet filtering, or identity-based access.
- **Operational overhead**: choose managed services when you want Azure to handle patching, scaling, or failover.
- **Protocol fit**: choose services based on whether you need HTTP(S), SMB, NFS, AMQP, MQTT, SQL, or event streaming.
- **Cost model**: choose the service whose billing aligns with steady-state, bursty, or idle workloads.

A good exam answer usually comes from eliminating services that solve the wrong problem. For example, a global HTTP application needs a web routing service, not a generic TCP load balancer; a workflow with human approvals needs orchestration, not a message queue.

**Compute and app hosting choices**

Azure offers several ways to run application code, and the right choice depends on how much control you need versus how much platform management you want.

- **Azure App Service**:
  - Use for web apps, REST APIs, and background web workloads that fit a managed PaaS model.
  - Choose when you want built-in deployment slots, custom domains, TLS, autoscale, and minimal infrastructure management.
  - Avoid when you need full OS control, custom kernel modules, or non-web protocols.

- **Azure Functions**:
  - Use for event-driven, short-lived, or bursty code that reacts to triggers such as HTTP, queue messages, timers, or Event Grid.
  - Choose when you want pay-per-execution behavior and very low operational overhead.
  - Avoid for long-running, stateful, or highly specialized workloads that need persistent local state.

- **Azure Container Apps**:
  - Use for microservices, event-driven containers, and HTTP or background workers when you want container flexibility without managing Kubernetes directly.
  - Choose when you need revision-based deployments, scale-to-zero, and KEDA-based event scaling.
  - Avoid when you need full cluster control or advanced Kubernetes-specific networking and operators.

- **Azure Kubernetes Service (AKS)**:
  - Use when you need orchestration for complex container platforms, custom scheduling, service mesh patterns, or portability across environments.
  - Choose when your team can operate Kubernetes and you need fine-grained control over networking, storage, and deployment.
  - Avoid when the workload is simple enough for App Service or Container Apps.

- **Azure Virtual Machines**:
  - Use when you need full OS control, legacy software, custom agents, unsupported dependencies, or lift-and-shift migration.
  - Choose when the application requires direct access to the guest OS, registry, drivers, or specialized middleware.
  - Avoid when a managed PaaS service can meet the requirement with less overhead.

A practical rule is to prefer the highest-level managed service that still satisfies the requirement. If the question says “legacy app,” “custom OS configuration,” or “unsupported feature,” move down the stack toward VMs. If it says “event-driven,” “serverless,” or “minimal management,” move up toward Functions or Container Apps.

**Event-driven integration: Service Bus, Event Grid, and Event Hubs**

These three services are often confused because they all move data, but they solve different integration patterns.

| Service | Best for | Delivery model | Typical use cases |
|---|---|---|---|
| Azure Service Bus | Reliable application messaging | Queue or topic with brokered delivery | Order processing, decoupling microservices, workflows, dead-letter handling |
| Azure Event Grid | Reactive event routing | Push-based event notification | Resource change notifications, lightweight fan-out, serverless triggers |
| Azure Event Hubs | Telemetry and streaming ingestion | High-throughput event stream | Logs, metrics, clickstreams, IoT ingestion, analytics pipelines |

Use **Service Bus** when you need durable command or work messages, ordering guarantees within a session, duplicate detection, scheduled delivery, or dead-letter queues for poison messages. Queues are best for one-to-one work distribution; topics and subscriptions are best for one-to-many fan-out with filtering.

Use **Event Grid** when you need to react to discrete events such as “blob created,” “resource deleted,” or “file uploaded.” It is ideal for publish-subscribe notification and serverless automation, but it is not a durable work queue and should not be used as a replacement for command processing.

Use **Event Hubs** when you need to ingest large volumes of telemetry or streaming data. It is optimized for throughput and partitioned consumption, not for business workflow semantics like retries, sessions, or per-message business transactions.

> If the requirement mentions “at least once processing,” “dead-lettering,” or “message lock,” think Service Bus. If it mentions “resource events” or “react to changes,” think Event Grid. If it mentions “millions of events per second,” “telemetry,” or “stream analytics,” think Event Hubs.

**Workflow orchestration: Functions, Logic Apps, and Container Apps**

These services can all automate work, but they differ in who owns the workflow logic and how much integration support you need.

- **Azure Functions**:
  - Best for code-first automation.
  - Use when the workflow is simple, custom, or tightly coupled to application logic.
  - Good for event handlers, lightweight transformations, and glue code.

- **Azure Logic Apps**:
  - Best for low-code orchestration and enterprise integration.
  - Use when you need connectors for SaaS and enterprise systems, approvals, branching, retries, and visual workflow design.
  - Strong choice for business process automation and integration-heavy scenarios.

- **Azure Container Apps**:
  - Best for containerized services that participate in event-driven workflows.
  - Use when the automation logic is packaged as a container and you want scale-to-zero or event-based scaling.
  - Not a replacement for a visual workflow engine.

A common exam pattern is to choose Logic Apps for integration workflows with many connectors and human steps, Functions for code-based event handlers, and Container Apps for containerized microservices that need elastic scaling.

**Networking: Front Door, Application Gateway, Load Balancer, NAT Gateway, Firewall, and NSGs**

Networking questions often hinge on whether you need global routing, regional routing, inbound protection, or outbound control.

- **Azure Front Door**:
  - Use for global HTTP(S) entry, edge-based routing, acceleration, and web application delivery.
  - Choose when you need anycast entry, global failover, path-based routing, or WAF at the edge.
  - Do not use for non-HTTP protocols.

- **Application Gateway**:
  - Use for regional HTTP(S) load balancing inside a region or across a regional deployment.
  - Choose when you need Layer 7 routing, TLS termination, cookie-based affinity, path-based routing, or WAF close to the application.
  - Prefer it when the app is private or regionally scoped and you do not need global edge routing.

- **Azure Load Balancer**:
  - Use for Layer 4 TCP/UDP distribution.
  - Choose for non-HTTP workloads, high-performance internal or public load balancing, and scenarios that need simple port-based distribution.
  - It does not inspect HTTP headers or URLs.

- **NAT Gateway**:
  - Use to provide predictable outbound internet access for subnets.
  - Choose when many VMs or instances need stable outbound public IPs without exposing inbound access.
  - It is an outbound-only service and does not filter traffic.

- **Azure Firewall**:
  - Use for centralized network security, application and network rules, FQDN filtering, threat intelligence, and logging.
  - Choose when you need controlled egress, inspection, and policy enforcement across spokes or shared networks.
  - It is not a replacement for NSGs; it complements them.

- **Network Security Groups (NSGs)**:
  - Use for subnet- or NIC-level allow/deny rules based on IP, port, and protocol.
  - Choose for distributed, lightweight traffic filtering close to the workload.
  - Use NSGs for micro-segmentation, but not for advanced inspection or centralized egress control.

A useful comparison is:

- **Front Door**: global HTTP(S) edge entry.
- **Application Gateway**: regional HTTP(S) reverse proxy and WAF.
- **Load Balancer**: Layer 4 distribution.
- **NAT Gateway**: outbound SNAT at scale.
- **Firewall**: centralized inspection and policy.
- **NSG**: local traffic filtering.

**Security-focused web architecture: WAF placement**

Web Application Firewall (WAF) can be deployed in more than one place, and the choice depends on where you want protection and what traffic path you need.

- **Front Door WAF**:
  - Use when you want protection at the Microsoft edge before traffic reaches any region.
  - Best for global internet-facing apps with distributed backends.
  - Good for reducing latency and blocking malicious traffic early.

- **Application Gateway WAF**:
  - Use when you want regional WAF protection close to the application subnet.
  - Best for private or regionally deployed web apps.
  - Useful when traffic must stay within a region or VNet-centric design.

Choose Front Door WAF for global entry and Application Gateway WAF for regional ingress. In some architectures, you use both: Front Door for global routing and Application Gateway for regional private ingress.

**Storage selection: Blob, Files, Queues, Tables, and disks**

Azure Storage is not one service but a family of services with different access patterns.

- **Blob Storage**:
  - Use for unstructured objects such as images, backups, logs, media, and data lakes.
  - Choose when you need REST access, lifecycle management, and tiering.
  - Best for object storage, not file-system semantics.

- **Azure Files**:
  - Use for shared file access over SMB, and in some configurations NFS.
  - Choose when multiple clients need a managed file share with familiar file semantics.
  - Good for lift-and-shift apps that expect a shared file share.

- **Queue Storage**:
  - Use for simple, lightweight messaging.
  - Choose when you need basic asynchronous decoupling and do not need advanced broker features.
  - Prefer Service Bus when you need sessions, transactions, or richer messaging semantics.

- **Table Storage**:
  - Use for NoSQL key-value style storage with simple schema and massive scale.
  - Choose when you need cheap, flexible, semi-structured storage and can model around partition and row keys.

- **Managed Disks**:
  - Use for VM block storage.
  - Choose when you need OS disks, data disks, high IOPS, or workload-specific disk performance.

**Blob access tiers and when to use them**

Blob access tiers help you optimize cost based on how often data is read.

| Tier | Best for | Tradeoff |
|---|---|---|
| Hot | Frequently accessed data | Higher storage cost, lower access cost |
| Cool | Infrequently accessed data | Lower storage cost, higher access cost and minimum retention considerations |
| Archive | Rarely accessed data | Lowest storage cost, offline retrieval required |

Use **Hot** for active application content, **Cool** for backups or infrequently read data, and **Archive** for long-term retention where retrieval delay is acceptable. Archive is not suitable for immediate access.

**Azure Files: Standard vs Premium, SMB vs NFS**

Azure Files selection often depends on performance and protocol requirements.

- **Standard Azure Files**:
  - Use for general-purpose shared file storage with cost efficiency.
  - Good for common SMB workloads and moderate performance needs.

- **Premium Azure Files**:
  - Use for latency-sensitive or IOPS-intensive file workloads.
  - Backed by SSD-based storage and better suited for production apps with higher performance requirements.

- **SMB**:
  - Use for Windows-centric file sharing and broad client compatibility.
  - Common choice for shared application content and user profiles.

- **NFS**:
  - Use when Linux or POSIX-style semantics are required and the Azure Files configuration supports it.
  - Best for workloads that expect NFS behavior rather than SMB.

> NFS support is not universal across every Azure Files scenario. Check the account type, region, and feature support before assuming it is available.

**Managed disks: choosing the right disk type**

Managed disks are the default choice for Azure VMs because Azure handles the storage account lifecycle.

- **Standard HDD**:
  - Use for dev/test, low-IOPS workloads, or cost-sensitive VMs.
- **Standard SSD**:
  - Use for balanced performance and cost.
- **Premium SSD**:
  - Use for production workloads that need low latency and higher IOPS.
- **Ultra Disk**:
  - Use for the most demanding workloads that need very high throughput and low latency, such as large databases or performance-sensitive transactional systems.

Choose disk type based on workload IOPS, throughput, and latency requirements. For exam scenarios, database servers and latency-sensitive apps usually point to Premium SSD or Ultra Disk, while general-purpose VMs often use Standard SSD.

**Storage account redundancy and resilience**

Redundancy choices affect durability, availability, and disaster recovery posture.

- **LRS**: locally redundant storage within one datacenter.
- **ZRS**: zone-redundant storage across availability zones in a region.
- **GRS**: geo-redundant storage with asynchronous replication to a paired region.
- **GZRS**: zone-redundant in the primary region plus geo-replication to a paired region.
- **RA-GRS / RA-GZRS**: read access to the secondary region.

Use **ZRS** when you want higher regional resilience without relying on a paired region. Use **GRS** or **GZRS** when you need disaster recovery across regions. Read-access variants help with read-only secondary access, but the secondary endpoint is not a write target.

**Identity and access: managed identities vs service principals**

Identity choice matters when Azure services need to call other Azure services.

- **Managed identities**:
  - Use when an Azure resource needs to authenticate to other Azure resources without storing credentials.
  - Best for App Service, Functions, VMs, Logic Apps, and other Azure-hosted workloads.
  - Prefer this whenever the workload runs in Azure and supports it.

- **Service principals**:
  - Use when you need an application identity that is not tied to a specific Azure resource.
  - Common for CI/CD pipelines, external automation, and cross-environment apps.
  - Requires credential management, rotation, and stronger operational discipline.

A simple rule: if the workload is in Azure and supports managed identity, use managed identity. If the workload runs outside Azure or needs a reusable app registration, use a service principal.

**Data platform selection: Azure SQL Database, Managed Instance, and SQL Server on VMs**

SQL choices are a frequent exam topic because each option offers a different balance of compatibility and management.

- **Azure SQL Database**:
  - Use for cloud-native relational applications.
  - Choose when you want a fully managed database with automatic patching, backups, and scaling.
  - Best for single-database or elastic application patterns.
  - It is not a full SQL Server instance, so some instance-level features are unavailable.

- **Azure SQL Managed Instance**:
  - Use when you need near-full SQL Server compatibility with managed operations.
  - Choose for migration scenarios that depend on instance-level features, SQL Agent, cross-database queries, or broader compatibility.
  - Good middle ground between PaaS simplicity and SQL Server feature parity.

- **SQL Server on Azure VMs**:
  - Use when you need full OS and SQL Server control, unsupported features, custom extensions, or specialized migration requirements.
  - Choose when managed services cannot satisfy the workload.
  - Highest operational overhead of the three.

**SQL deployment models: single database, elastic pool, and serverless**

Within Azure SQL Database, the deployment model matters.

- **Single database**:
  - Use when one application or tenant needs dedicated resources.
  - Best for predictable workloads or when isolation matters.

- **Elastic pool**:
  - Use when many databases have variable and unpredictable usage patterns.
  - Best for SaaS multi-tenant designs where databases do not peak at the same time.
  - Helps optimize cost by sharing compute across databases.

- **Serverless**:
  - Use when workloads are intermittent and can auto-pause during idle periods.
  - Best for dev/test, sporadic usage, or applications with long idle windows.
  - Not ideal for consistently busy production workloads.

A common exam clue is “many small databases with uneven usage,” which usually points to an elastic pool. “Bursty and idle for long periods” often points to serverless.

**NoSQL and document databases: Cosmos DB choices**

Azure Cosmos DB is a globally distributed NoSQL database platform with multiple APIs.

- **Cosmos DB for NoSQL**:
  - Use for new document-oriented applications that need global distribution, low latency, and elastic scale.
  - Good default for cloud-native document workloads.

- **Cosmos DB for MongoDB**:
  - Use when you need MongoDB compatibility for an existing application or driver ecosystem.
  - Choose for migration or compatibility requirements.

- **Other APIs**:
  - Use when the application or team already depends on a specific data model or protocol.

Choose Cosmos DB when you need global distribution, multi-region writes or reads, and predictable low-latency access. Do not choose it just because it is “NoSQL”; simpler storage or database services may be more appropriate.

**Messaging and streaming examples**

A few practical examples help anchor the service choices.

```bash
# Create a Service Bus queue
az servicebus queue create \
  --resource-group rg-app \
  --namespace-name sb-prod \
  --name orders

# Create an Event Grid subscription for a storage account event
az eventgrid event-subscription create \
  --name blob-created-sub \
  --source-resource-id /subscriptions/<subId>/resourceGroups/rg-data/providers/Microsoft.Storage/storageAccounts/stacct01 \
  --endpoint https://contoso.azurewebsites.net/api/events \
  --included-event-types Microsoft.Storage.BlobCreated

# Create an Event Hubs namespace
az eventhubs namespace create \
  --resource-group rg-telemetry \
  --name eh-prod \
  --location eastus
```

These examples show the different intent:

- Service Bus for durable work messages.
- Event Grid for event notifications.
- Event Hubs for telemetry ingestion.

**Backup and disaster recovery: Azure Backup vs Site Recovery vs native replication**

Backup and disaster recovery are not interchangeable.

- **Azure Backup**:
  - Use for point-in-time recovery of supported Azure resources, VMs, files, and some workloads.
  - Best for accidental deletion, corruption, and retention requirements.
  - It is a backup solution, not a failover orchestration platform.

- **Azure Site Recovery (ASR)**:
  - Use for disaster recovery and workload failover.
  - Best when you need orchestrated replication, test failover, and recovery plans.
  - Common for VM and on-premises-to-Azure DR.

- **Native service replication**:
  - Use when the platform itself provides replication or failover, such as geo-redundant storage or database replication features.
  - Best when the service already handles data durability and failover semantics.

A useful distinction is that backup protects against data loss, while Site Recovery protects against site outage. Native replication may support both durability and availability, but it is service-specific.

**Monitoring and observability: Azure Monitor, Log Analytics, Application Insights, and alerts**

Monitoring services overlap, so the exam often tests whether you know the role of each component.

- **Azure Monitor**:
  - The umbrella platform for collecting, analyzing, and acting on telemetry.
  - Use for metrics, logs, alerts, dashboards, and diagnostic settings.

- **Log Analytics**:
  - The log query and storage workspace used by Azure Monitor.
  - Use when you need Kusto Query Language (KQL) queries across logs from multiple resources.

- **Application Insights**:
  - Use for application performance monitoring, request tracing, dependency tracking, and exception analysis.
  - Best for code-level observability in web apps and APIs.

- **Alerts**:
  - Use to notify or automate when metrics or log conditions are met.
  - Metric alerts are good for near-real-time resource health; log alerts are good for query-based conditions.

A practical pattern is: send diagnostics to Log Analytics, instrument the app with Application Insights, and create alerts in Azure Monitor.

```bash
# Example: enable diagnostic settings to a Log Analytics workspace
az monitor diagnostic-settings create \
  --name send-to-law \
  --resource /subscriptions/<subId>/resourceGroups/rg-app/providers/Microsoft.Web/sites/app01 \
  --workspace /subscriptions/<subId>/resourceGroups/rg-monitor/providers/Microsoft.OperationalInsights/workspaces/law-prod \
  --logs '[{"category":"AppServiceHTTPLogs","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

**Private Link vs service endpoints**

These two features both restrict access to PaaS services, but they do so differently.

- **Private Link**:
  - Gives the service a private endpoint in your VNet.
  - Traffic stays on private IP space from the client perspective.
  - Requires private DNS planning so clients resolve the service name to the private endpoint.
  - Best when you want strong network isolation and private access from on-premises or peered networks.

- **Service endpoints**:
  - Extend subnet identity to a service’s public endpoint.
  - The service remains publicly reachable, but access is restricted to approved VNets/subnets.
  - Simpler than Private Link, but less isolated.

Use Private Link when you need private connectivity, tighter isolation, or when public exposure is not acceptable. Use service endpoints when you want a simpler control and the service supports it, but you do not need a private IP path.

**Practical CLI examples with caution**

Use CLI examples to reinforce concepts, but be careful not to memorize commands blindly. Azure CLI syntax can vary by service version and scope.

```bash
# Create a storage account with a common general-purpose redundancy option
az storage account create \
  --name mystorageacct01 \
  --resource-group rg-data \
  --location eastus \
  --sku Standard_LRS \
  --kind StorageV2

# Create a policy assignment from a known policy definition ID
az policy assignment create \
  --name enforce-tags \
  --scope /subscriptions/<subId>/resourceGroups/rg-app \
  --policy /providers/Microsoft.Authorization/policyDefinitions/<policyDefinitionId> \
  --params '{"tagName":{"value":"environment"},"tagValue":{"value":"prod"}}'
```

> Policy commands are scope-sensitive. In practice, you must reference a valid policy definition or initiative at the correct scope, and many policies require parameters. Do not assume a placeholder path will work unchanged.

**Common exam decision patterns**

Use these patterns to narrow the answer quickly:

- **Global web app with failover and WAF**:
  - Choose Front Door, often with WAF.
- **Regional web app with private backend**:
  - Choose Application Gateway, often with WAF.
- **Non-HTTP traffic distribution**:
  - Choose Load Balancer.
- **Outbound internet control for many VMs**:
  - Choose NAT Gateway, possibly with Azure Firewall for inspection.
- **Subnet-level traffic filtering**:
  - Choose NSGs.
- **Durable work queue**:
  - Choose Service Bus queue.
- **Fan-out event notifications**:
  - Choose Event Grid.
- **High-volume telemetry ingestion**:
  - Choose Event Hubs.
- **Low-code business workflow**:
  - Choose Logic Apps.
- **Code-first event handler**:
  - Choose Functions.
- **Containerized elastic microservice**:
  - Choose Container Apps.
- **Legacy app with OS control**:
  - Choose VM.
- **Shared file access**:
  - Choose Azure Files.
- **Object storage and tiering**:
  - Choose Blob Storage.
- **Cloud-native relational app**:
  - Choose Azure SQL Database.
- **Migration with instance-level SQL features**:
  - Choose SQL Managed Instance.
- **Full SQL Server control**:
  - Choose SQL Server on VMs.
- **Global low-latency NoSQL**:
  - Choose Cosmos DB.

**Final exam guidance**

When two services seem possible, ask which one best matches the workload’s primary contract:

- If the contract is **message delivery**, think Service Bus.
- If the contract is **event notification**, think Event Grid.
- If the contract is **stream ingestion**, think Event Hubs.
- If the contract is **workflow orchestration**, think Logic Apps.
- If the contract is **code execution on triggers**, think Functions.
- If the contract is **global web delivery**, think Front Door.
- If the contract is **regional web ingress**, think Application Gateway.
- If the contract is **private network control**, think Private Link, Azure Firewall, or NSGs depending on the layer.

The best AZ-305 answer is usually the one that satisfies the requirement with the least operational complexity while preserving the needed security, scale, and protocol support.
