## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Reference architecture mindset**

AZ-305 expects you to choose and justify architectures, not just name services. A reference architecture is a repeatable pattern that maps business requirements to Azure services, network boundaries, identity controls, data flows, and recovery objectives.

The exam often tests tradeoffs:
- **Scope**: global, regional, or zonal
- **Traffic type**: HTTP/HTTPS, TCP, UDP, or mixed
- **Security posture**: public, private, or hybrid access
- **Operational model**: centralized platform team, decentralized app teams, or shared responsibility
- **Resiliency target**: availability zones, multi-region active-active, or active-passive

A good answer explains why a pattern fits the workload and what it does not solve.

**Azure landing zones and governance architecture**

Azure landing zones are the foundation for enterprise-scale design. They provide a management group hierarchy, policy guardrails, identity boundaries, and subscription placement rules so workloads can be deployed consistently.

Common landing zone concepts:
- **Platform landing zone**: shared services such as identity, connectivity, management, security, and sometimes shared data services
- **Application landing zone**: workload-specific subscriptions for one app, product, or business unit
- **Enterprise-scale landing zone**: a broader operating model that standardizes management groups, policy, and subscription vending across the organization

Typical management group structure:
- **Tenant root group**
  - **Platform**
    - Identity
    - Connectivity
    - Management
    - Security
  - **Landing zones**
    - Corp
    - Online
  - **Sandbox**
  - **Decommissioned**

Why this matters:
- Policies inherit from management groups, so you can enforce standards before subscriptions are created.
- RBAC (role-based access control) can be delegated at the right scope without giving broad tenant access.
- Subscription vending helps you create compliant subscriptions with the right tags, policies, and network baseline.

Practical policy example:
```bash
az policy definition list --query "[?contains(displayName, 'Allowed locations')].{name:name, displayName:displayName}" -o table
az policy assignment create \
  --name "deny-nonapproved-regions" \
  --scope "/subscriptions/<subscription-id>" \
  --policy "/providers/Microsoft.Authorization/policyDefinitions/<policy-definition-id>"
```

Use policy assignments to enforce:
- Allowed regions
- Required tags
- SKU restrictions
- Private endpoint or public network rules where supported
- Diagnostic settings deployment

> Be careful with built-in policy identifiers. In practice, you often discover the exact definition ID with `az policy definition list` or `az policy definition show` rather than assuming a short path.

**Identity architecture**

Identity is the control plane for nearly every Azure reference architecture. Microsoft Entra ID (formerly Azure Active Directory) is the primary identity provider for users, groups, service principals, and managed identities.

Core design patterns:
- **Centralized identity tenant** for enterprise users and admin controls
- **Privileged Identity Management (PIM)** for just-in-time elevation
- **Conditional Access** to enforce device, location, risk, or MFA requirements
- **Identity governance** for lifecycle, access reviews, and entitlement management
- **Managed identities** for Azure resources that need to authenticate without secrets

When to use each:
- Use **managed identities** for app-to-Azure authentication whenever possible.
- Use **service principals** when automation must run outside an Azure resource context.
- Use **PIM** for privileged roles such as Owner, User Access Administrator, or security administration.
- Use **Conditional Access** to protect interactive access to portals, admin tools, and sensitive apps.

Practical guidance:
- Keep human admin access separate from workload identities.
- Prefer group-based RBAC assignment over direct user assignment.
- Use least privilege at subscription, resource group, and resource scopes.
- Protect break-glass accounts with strong monitoring and limited use.

Example managed identity flow:
```bash
az webapp identity assign \
  --name myapp \
  --resource-group rg-app

az role assignment create \
  --assignee-object-id <principal-id> \
  --assignee-principal-type ServicePrincipal \
  --role "Storage Blob Data Reader" \
  --scope "/subscriptions/<sub-id>/resourceGroups/rg-data/providers/Microsoft.Storage/storageAccounts/<storage>"
```

**Network reference architectures**

Network design is one of the most tested areas in AZ-305. The main decision is usually between hub-and-spoke, virtual WAN, or a simpler single-VNet design.

**Hub-and-spoke**
- Central hub hosts shared services such as firewall, DNS, VPN/ExpressRoute, and monitoring.
- Spokes host application workloads.
- Peering connects spokes to the hub and often blocks direct spoke-to-spoke traffic unless explicitly allowed.

Use hub-and-spoke when:
- You need centralized inspection and governance.
- Multiple workloads share common network services.
- You want clear separation between platform and application subscriptions.

**Azure Virtual WAN**
- Managed global transit service for branch, VPN, ExpressRoute, and sometimes secure internet egress.
- Useful when you have many sites, many branches, or a need for simplified global connectivity.
- Reduces operational overhead compared with building and maintaining a large custom transit network.

Use Virtual WAN when:
- You need large-scale branch connectivity.
- You want managed transit across regions.
- You prefer a service-managed routing fabric over custom hub design.

**Zone-aware and regional design**
- **Availability zones** protect against datacenter-level failures within a region.
- **Zone-redundant services** replicate across zones automatically where supported.
- **Multi-region** designs protect against regional outages and are used for higher availability or disaster recovery.

Choose:
- **Zone-redundant** when the workload must survive a zone failure but can remain in one region.
- **Multi-region** when a regional outage is unacceptable or when global latency needs active distribution.

Traffic routing patterns:
- **Azure Load Balancer** for Layer 4 TCP/UDP traffic
- **Application Gateway** for regional Layer 7 HTTP/HTTPS routing, WAF, and TLS termination
- **Azure Front Door** for global Layer 7 entry, edge acceleration, and multi-region HTTP/HTTPS routing
- **Traffic Manager** for DNS-based global failover or load distribution across endpoints

Important distinctions:
- Traffic Manager makes DNS decisions, so it does not do per-request routing.
- Front Door is Layer 7 only and is not for arbitrary TCP/UDP workloads.
- Application Gateway is regional and depends on your backend network design for private access.

Example topology choices:
- Internet-facing web app with global users: Front Door + regional app tiers
- Internal enterprise app in one region: Application Gateway + private backends
- Branch-to-VNet connectivity: Virtual WAN or VPN/ExpressRoute hub
- East-west traffic control: hub firewall plus NSGs and route tables

Private connectivity patterns:
- Use **private endpoints** to expose PaaS services over private IPs in your VNet.
- Use **private DNS zones** so name resolution points clients to the private endpoint.
- Disable public network access where the service supports it, but verify the service-specific control model.

Private endpoint example:
```bash
az network private-endpoint create \
  --name pe-storage \
  --resource-group rg-network \
  --vnet-name vnet-hub \
  --subnet subnet-private-endpoints \
  --private-connection-resource-id "/subscriptions/<sub-id>/resourceGroups/rg-data/providers/Microsoft.Storage/storageAccounts/<storage>" \
  --group-id blob \
  --connection-name pe-storage-conn
```

Private DNS zone example:
```bash
az network private-dns zone create \
  --resource-group rg-network \
  --name privatelink.blob.core.windows.net

az network private-dns link vnet create \
  --resource-group rg-network \
  --zone-name privatelink.blob.core.windows.net \
  --name link-hub \
  --virtual-network vnet-hub \
  --registration-enabled false
```

**Security architecture**

Security reference architectures combine identity, network controls, data protection, and posture management.

Key building blocks:
- **Microsoft Defender for Cloud** for secure score, recommendations, and workload protection
- **Azure Policy** for preventive and detective governance
- **Key Vault** for secrets, keys, and certificates
- **Customer-managed keys (CMK)** when you need control over encryption keys
- **Network segmentation** with NSGs, ASGs, firewalls, and private endpoints

Practical security patterns:
- Use Defender for Cloud recommendations to identify missing controls.
- Rotate secrets and certificates on a defined schedule.
- Prefer managed identities over stored credentials.
- Use CMK when compliance or data sovereignty requires customer control of encryption keys.
- Plan key rotation and recovery before you adopt CMK at scale.

Key Vault example:
```bash
az keyvault create \
  --name kv-prod-001 \
  --resource-group rg-security \
  --location eastus \
  --enable-rbac-authorization true

az keyvault secret set \
  --vault-name kv-prod-001 \
  --name SqlAdminPassword \
  --value "Use-A-Real-Secret-Store"
```

Security design considerations:
- Separate platform security controls from application security controls.
- Use RBAC for access management where supported, and avoid broad access policies.
- Audit access to secrets and keys.
- Treat certificates as lifecycle-managed assets, not static files.

> Private endpoints reduce exposure, but they do not automatically guarantee private-only access in every service. Verify the service’s public access setting and network controls together.

**Data architecture**

Data reference architectures focus on storage choice, analytics patterns, governance, and integration.

Common platform choices:
- **Azure SQL Database** for modern relational applications with PaaS benefits
- **Azure SQL Managed Instance** for high SQL Server compatibility and migration scenarios
- **Azure Cosmos DB** for globally distributed, low-latency NoSQL workloads
- **Azure Storage** for object, file, queue, and archive needs
- **Synapse Analytics** or **Microsoft Fabric**-style patterns for analytics at scale, depending on course scope and organizational direction

How to choose:
- Use **Azure SQL Database** when you want managed relational PaaS with less instance-level control.
- Use **SQL Managed Instance** when migration compatibility matters and you need many SQL Server behaviors.
- Use **Cosmos DB** when you need elastic scale, global distribution, and flexible data models.
- Use **storage-based lake patterns** when you need raw, curated, and analytical zones.

Lakehouse vs warehouse:
- **Lakehouse** fits mixed structured and semi-structured data, data science, and flexible schema evolution.
- **Warehouse** fits governed, highly structured analytics and BI workloads with strong SQL-centric consumption.
- Many enterprise designs use both, with ingestion landing in a lake and curated data published to a warehouse layer.

Data governance:
- Use **Microsoft Purview** to catalog data, classify sensitive information, and support lineage and governance.
- Define ownership for datasets, pipelines, and access policies.
- Classify data by sensitivity and business criticality before exposing it broadly.

Integration patterns:
- Batch ETL (extract, transform, load) for scheduled movement
- ELT (extract, load, transform) when the target platform handles transformation efficiently
- Event-driven integration for near-real-time workflows
- API-based integration for transactional systems and service orchestration

Example SQL compatibility decision:
- Choose Managed Instance for lift-and-shift migrations where SQL Agent, cross-database scenarios, or instance-level compatibility matter.
- Choose Azure SQL Database for new cloud-native apps that do not need instance-level features.

**Observability and operations**

Observability architecture covers logs, metrics, traces, alerting, and operational ownership.

Core components:
- **Azure Monitor** for platform-wide telemetry
- **Log Analytics workspace** for log storage and KQL (Kusto Query Language) queries
- **Application Insights** for application performance monitoring
- **Action groups** for alert routing
- **Workbooks** for visualization and operational dashboards

Workspace design guidance:
- Use a centralized workspace for platform operations when you need cross-subscription visibility.
- Use separate workspaces when data residency, isolation, or team boundaries require it.
- Standardize retention based on compliance and troubleshooting needs.

Operational model:
- Platform teams own shared infrastructure alerts, policy compliance, and baseline health.
- Application teams own app-specific telemetry, SLOs (service level objectives), and business alerts.
- Use consistent naming and tagging so alerts can be routed to the right team.

KQL example:
```kusto
requests
| where timestamp > ago(24h)
| summarize avgDuration = avg(toduration(duration)) by cloud_RoleName
| order by avgDuration desc
```

Notes on schema:
- In many Application Insights schemas, `duration` is a timespan field in the `requests` table.
- Schema can vary by table, ingestion mode, and workspace-based vs classic setups, so validate the exact columns in your environment.

Alerting pattern:
- Use metric alerts for fast, low-noise signals such as CPU, availability, or queue depth.
- Use log alerts for richer conditions such as error patterns or custom business events.
- Route critical alerts to on-call, and informational alerts to dashboards or ticketing.

**Resiliency and disaster recovery**

Resiliency architecture is about designing for failure, not avoiding it. AZ-305 expects you to distinguish high availability from disaster recovery.

Core concepts:
- **RTO** (Recovery Time Objective): how long recovery can take
- **RPO** (Recovery Point Objective): how much data loss is acceptable
- **Failover**: switching to a secondary system
- **Failback**: returning to the primary after recovery

Availability patterns:
- **Availability Sets** protect against hardware and rack-level failures in legacy designs.
- **Availability Zones** protect against datacenter failures within a region.
- **Multi-region** protects against regional outages.

Backup and DR patterns:
- Use native backup features for Azure VMs, Azure SQL, and other PaaS services where available.
- Use Azure Site Recovery for orchestrated VM replication and failover.
- Test restore and failover regularly; a backup strategy is incomplete without validation.
- Document failback steps before you need them.

Azure Site Recovery example:
```bash
az recoveryservices vault create \
  --name rsv-prod-001 \
  --resource-group rg-dr \
  --location eastus
```

After vault creation, you still need to:
1. Configure replication policy.
2. Select source and target regions.
3. Enable replication for protected items.
4. Test failover.
5. Plan failback and cleanup.

Backup design considerations:
- Protect platform services and application data separately.
- Define retention by compliance and business recovery needs.
- Verify backup immutability or soft-delete options where available.
- Include key vault recovery, DNS recovery, and identity dependencies in DR plans.

**Reference architecture examples**

A strong AZ-305 answer often combines multiple services into a coherent design.

**1. Internet-facing enterprise web app**
- Azure Front Door for global entry and HTTP/HTTPS routing
- Regional Application Gateway with WAF for regional inspection
- App Service, AKS, or VMs in zone-redundant deployment
- Private endpoints for data services
- Key Vault for secrets and certificates
- Azure Monitor and Application Insights for observability

Why this works:
- Front Door handles global traffic and edge entry.
- Application Gateway adds regional Layer 7 control.
- Private endpoints reduce exposure of backend services.

**2. Regulated internal line-of-business app**
- Hub-and-spoke network
- Central firewall and DNS in the hub
- Private endpoints for PaaS dependencies
- Entra ID with Conditional Access and PIM
- Azure Policy for compliance guardrails
- Centralized logging and alerting

Why this works:
- It keeps traffic private and centrally governed.
- It supports auditability and least privilege.
- It is easier to standardize across multiple internal apps.

**3. Global branch connectivity platform**
- Azure Virtual WAN or hub-and-spoke with ExpressRoute
- Central security inspection
- Shared DNS and routing
- Regional workload spokes
- Monitoring and policy at the platform layer

Why this works:
- It scales connectivity across many sites.
- It reduces custom routing complexity.
- It supports centralized operations.

**4. Data platform for analytics**
- Ingestion layer for batch and event data
- Storage lake for raw and curated zones
- Governance with Purview
- Warehouse or lakehouse serving layer
- Private access to data services
- Monitoring for pipeline health and data freshness

Why this works:
- It separates ingestion, governance, and consumption.
- It supports both BI and advanced analytics.
- It scales as data volume and user count grow.

**How to answer architecture questions in AZ-305**

When you evaluate a scenario, work through the design in this order:
1. Identify the workload type and traffic pattern.
2. Determine the required scope: single region, multi-zone, or multi-region.
3. Choose the identity model and access boundaries.
4. Decide on network topology and private/public exposure.
5. Select data services and governance controls.
6. Add observability, backup, and DR.
7. Validate the design against cost, operations, and compliance.

Useful decision cues:
- If the workload is HTTP/HTTPS and global, consider Front Door.
- If the workload is regional and needs WAF, consider Application Gateway.
- If the workload needs private PaaS access, use private endpoints plus private DNS.
- If the workload must survive a zone failure, use zone-redundant services or zone-aware deployment.
- If the workload must survive a regional outage, add multi-region failover and data replication.

**Common exam pitfalls**

- Confusing DNS-based global routing with per-request Layer 7 routing
- Assuming private endpoints alone are enough without checking service-level public access controls
- Treating all PaaS services as if they support the same network and encryption settings
- Overlooking identity governance and privileged access controls
- Designing DR without testing failover and failback
- Using a single monitoring workspace without considering team boundaries or retention requirements
- Choosing a data platform without defining governance and lineage

**Quick comparison table**

| Scenario | Preferred pattern | Why |
|---|---|---|
| Global HTTP app | Front Door + regional app tiers | Global entry and Layer 7 routing |
| Regional web app with WAF | Application Gateway | Regional Layer 7 control |
| Branch connectivity | Virtual WAN or hub-and-spoke | Scalable transit and centralized routing |
| Private PaaS access | Private endpoint + private DNS | Private name resolution and connectivity |
| SQL migration | SQL Managed Instance | High compatibility for lift-and-shift |
| Cloud-native relational app | Azure SQL Database | Managed PaaS with less overhead |
| Multi-zone availability | Zone-redundant design | Survive datacenter failure in a region |
| Regional disaster recovery | Multi-region replication | Survive regional outage |

**Practical checklist for design reviews**

- Confirm the business objective, SLA, and recovery targets.
- Map identity, network, data, and security dependencies.
- Verify whether the service supports zones, private endpoints, CMK, and diagnostic settings.
- Check whether the architecture needs centralized or distributed operations.
- Validate monitoring, alerting, backup, and restore paths.
- Document failover, failback, and testing cadence.
- Ensure governance is enforced through management groups, policy, and RBAC.

**Final exam takeaway**

AZ-305 reference architectures are about choosing the right combination of Azure services for a specific workload and operating model. The best answer is usually the one that balances security, resiliency, manageability, and cost while matching the traffic pattern and business requirement.
