## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Why AZ-305 questions feel tricky**

AZ-305 often tests whether you can choose the *right* Azure service for a scenario, not whether you can define the service. The traps usually come from overlapping capabilities, hidden constraints, and scope boundaries. You need to think in terms of management plane vs data plane, regional vs global services, subscription limits, and the operational impact of each design choice.

A good exam answer usually reflects these questions:

- What is the access boundary: tenant, subscription, resource group, resource, or data object?
- Is the requirement for control-plane management or data-plane access?
- Does the design need active-active, active-passive, or simple backup?
- Are there quota, SKU, region, or subscription-type constraints?
- Is the service meant for ingress, egress, private access, or global delivery?

**Identity and access control traps**

A common misconception is treating Microsoft Entra ID (formerly Azure Active Directory) and Azure RBAC as interchangeable. They are related, but they solve different problems.

- **Microsoft Entra ID** authenticates identities and issues tokens.
- **Azure RBAC** authorizes actions on Azure resources in the management plane.
- **Resource-specific data-plane permissions** authorize access to the data inside a service, such as blobs, keys, secrets, or SQL data.

Use the right layer for the right question:

- If a user needs to create or delete a VM, you use **Azure RBAC** at the subscription, resource group, or resource scope.
- If a user needs to read a blob in Storage, you may need **Storage Blob Data Reader** or another data-plane role, not just Contributor.
- If a user needs to sign in to Azure, Entra ID handles authentication, but that alone does not grant resource access.

> **Exam trap:** A user with `Contributor` can manage a storage account but still cannot read blob contents unless they also have a data-plane role or shared access mechanism.

Practical scope examples:

```bash
# Assign a role at resource group scope
az role assignment create \
  --assignee user@contoso.com \
  --role "Reader" \
  --scope /subscriptions/<sub-id>/resourceGroups/rg-app

# Assign a data-plane role for Storage
az role assignment create \
  --assignee user@contoso.com \
  --role "Storage Blob Data Reader" \
  --scope /subscriptions/<sub-id>/resourceGroups/rg-data/providers/Microsoft.Storage/storageAccounts/stacct01
```

Common identity misconceptions:

- **Managed identities** are for Azure resources to authenticate without secrets.
- **Service principals** are app identities; they are not the same as a user account.
- **Privileged Identity Management (PIM)** reduces standing privilege, but it does not replace RBAC design.
- **Conditional Access** controls sign-in conditions; it does not grant Azure resource permissions.

**Governance and policy traps**

Governance questions often hide the difference between prevention, detection, and access control. Azure Policy is for enforcing or auditing resource properties, while RBAC is for who can do what.

Typical policy effects:

- **Deny** blocks noncompliant resource creation or updates.
- **Audit** records noncompliance without blocking.
- **Append** adds fields during deployment when possible.
- **DeployIfNotExists** deploys a related resource or configuration if missing.

Example policy assignment pattern:

```json
{
  "properties": {
    "displayName": "Allowed locations",
    "policyDefinitionId": "/providers/Microsoft.Authorization/policyDefinitions/allowed-locations",
    "parameters": {
      "listOfAllowedLocations": {
        "value": ["eastus", "eastus2"]
      }
    }
  }
}
```

Scope matters:

- **Management group** scope applies across multiple subscriptions.
- **Subscription** scope applies to all resource groups in that subscription.
- **Resource group** scope is useful for workload-specific controls.
- **Resource** scope is narrow and often used for exceptions or targeted governance.

Lock types are another frequent trap:

- **CanNotDelete** prevents deletion but allows read and update.
- **ReadOnly** prevents changes and deletion, but can also block some operational actions.

> **Exam trap:** A lock does not replace backup, and backup does not replace a lock. They solve different problems.

**Service limits, quotas, and subscription constraints**

AZ-305 frequently includes hidden capacity constraints. A design may be technically correct but fail because of quota, SKU, or subscription limitations.

Watch for these exam signals:

- **vCPU quotas** can block VM deployment even when the region supports the VM size.
- **Regional capacity** may be unavailable for a specific SKU in a region.
- **SKU availability** differs by region and sometimes by zone.
- **Feature availability** can vary by subscription type, tenant configuration, or preview status.
- **Subscription-level limits** can affect public IPs, cores, ExpressRoute circuits, or other resources.

Practical examples:

- You may choose a VM family that meets performance needs, but the subscription has no remaining vCPU quota in that region.
- A zone-redundant service may be unavailable if the region does not support availability zones for that SKU.
- Some advanced networking or security features may require specific subscription types or provider registrations.

Useful CLI checks:

```bash
# Check VM usage and limits in a region
az vm list-usage --location eastus

# Check available VM sizes in a region
az vm list-sizes --location eastus

# Check subscription quotas for compute
az quota list --scope /subscriptions/<sub-id>/providers/Microsoft.Compute/locations/eastus
```

Design implication:

- Always consider whether the answer is blocked by a quota or capacity issue, especially when the question mentions scale, rapid growth, or a specific region.
- If a service is not available in the target region, the exam often expects you to choose a different region, a different SKU, or a different architecture pattern.

**Networking traps: peering, hub-spoke, gateway transit, NAT, and routing**

Networking questions often test whether you understand traffic flow, not just connectivity options.

**VNet peering** is for private connectivity between virtual networks. It is low-latency and does not require a gateway for the peered path.

- Use peering when workloads need private IP connectivity across VNets.
- Use **hub-spoke** when you want centralized shared services such as firewall, DNS, VPN, or ExpressRoute.
- Use **gateway transit** when spokes need to use the hub’s VPN or ExpressRoute gateway.
- Use **use remote gateways** on the spoke when it should route through the hub gateway.

Common routing misconception:

- Peering does not automatically make all traffic flow through the hub.
- User-defined routes (UDRs) and system routes still matter.
- If you need forced tunneling or inspection, you usually add a firewall/NVA and route tables.

Example hub-spoke pattern:

```text
On-premises
   |
VPN/ExpressRoute
   |
Hub VNet ---- Azure Firewall ---- Internet
   |
   +---- Spoke VNet A
   +---- Spoke VNet B
```

NAT Gateway is another frequent confusion point:

- **NAT Gateway** provides outbound internet connectivity for subnets with predictable source IPs.
- It does not provide inbound access.
- It is not a replacement for a firewall.
- It is often used when you need stable egress IPs for allowlists.

Example subnet association:

```bash
az network vnet subnet update \
  --resource-group rg-net \
  --vnet-name vnet-app \
  --name snet-app \
  --nat-gateway natgw-app
```

Peering and gateway traps:

- VNet peering is not transitive by default.
- Gateway transit is a specific configuration, not an automatic behavior.
- ExpressRoute and VPN gateway designs differ in SLA, bandwidth, and routing behavior.
- If the question asks for centralized inspection, peering alone is usually insufficient.

**VPN, ExpressRoute, Bastion, and private access confusion**

A common exam mistake is choosing VPN when the requirement is secure administrative access, or choosing Bastion when the requirement is private network connectivity.

- **Azure VPN Gateway** connects networks over the public internet using encrypted tunnels.
- **ExpressRoute** provides private connectivity through a connectivity provider.
- **Azure Bastion** provides browser-based RDP/SSH to VMs without exposing public IPs.
- **Private Link** gives private access to PaaS services over a private endpoint.

Use cases:

- Choose **Bastion** when administrators need to reach VMs without opening inbound RDP/SSH to the internet.
- Choose **VPN** or **ExpressRoute** when you need network-to-network connectivity.
- Choose **Private Link** when you want private access to Azure PaaS services such as Storage, Key Vault, or SQL.
- Do not confuse Bastion with a site-to-site connectivity solution.

> **Exam trap:** Bastion is for VM administration, not for connecting an on-premises network to Azure.

**Application delivery traps: Front Door, CDN, Application Gateway, and API Management**

These services overlap, so the exam often checks whether you understand where each sits in the request path.

- **Azure Front Door** is a global entry point for HTTP/HTTPS with global load balancing, acceleration, and edge routing.
- **Azure CDN** is optimized for caching and content delivery from edge locations.
- **Application Gateway** is a regional Layer 7 load balancer with web application firewall capabilities.
- **API Management** is for publishing, securing, transforming, and governing APIs.

Typical distinctions:

- Use **Front Door** for global web apps needing fast failover and edge routing.
- Use **Application Gateway** for regional web traffic, internal apps, or WAF at the regional layer.
- Use **CDN** when the main need is caching static content close to users.
- Use **API Management** when you need API policies, versioning, throttling, authentication, or developer portal features.

Scenario trap:

- If the question asks for global failover across regions for a web app, Front Door is often the better fit than Application Gateway.
- If the question asks to expose APIs with quotas and transformations, API Management is usually the right answer, not Application Gateway.

**Backup, restore, replication, and business continuity traps**

AZ-305 often tests whether you can distinguish backup from disaster recovery.

- **Azure Backup** protects data and supports restore to a point in time.
- **Azure Site Recovery (ASR)** orchestrates replication and failover for workloads.
- **Native replication** is built into some services and may be the correct DR mechanism for that service.
- **Geo-redundancy** protects against regional failure for supported services, but it is not the same as full workload failover.

Use the right tool:

- Use **Azure Backup** for long-term retention, accidental deletion recovery, and point-in-time restore.
- Use **ASR** when you need orchestrated failover of VMs or on-premises workloads to another region.
- Use **native replication** when the service provides its own DR model, such as database replication or storage redundancy.

Common traps:

- Backup does not provide low-RTO failover by itself.
- Replication does not automatically provide application consistency unless the service supports it.
- A replicated VM is not the same as a fully tested recovery plan.
- Geo-redundant storage protects the data layer, but application dependencies may still fail.

Example decision logic:

- If the requirement is “restore a deleted file from 14 days ago,” choose backup.
- If the requirement is “fail over the entire app to another region within minutes,” choose ASR or a service-native failover design.
- If the requirement is “keep a database available across zones,” choose zone redundancy or a database HA feature, not backup.

**Azure SQL disaster recovery traps**

Database questions often hinge on whether you need high availability, disaster recovery, or both.

- **Failover groups** provide managed failover for Azure SQL Database and SQL Managed Instance across regions.
- **Active geo-replication** supports readable secondary replicas and manual failover for Azure SQL Database.
- **Zone redundancy** protects against datacenter failure within a region.
- **Backup retention** is for restore, not continuous availability.

Important distinctions:

- **Failover groups** are often the best answer when you need automatic or managed regional failover with a listener endpoint.
- **Geo-replication** is useful when you need readable secondary databases or more control over failover.
- **Zone redundancy** addresses intra-region resilience, not cross-region disaster recovery.
- SQL Managed Instance and Azure SQL Database do not use the exact same DR options in every case, so read the service type carefully.

Scenario examples:

- For a mission-critical app needing automatic regional failover, a failover group is often the right choice.
- For reporting against a secondary copy while keeping the primary online, geo-replication may fit better.
- For protection against a zone outage in the same region, zone redundancy is the relevant feature.

> **Exam trap:** A backup policy does not replace a failover group, and a failover group does not replace long-term backup retention.

**Storage and data protection misconceptions**

Storage questions often mix redundancy, access control, and encryption.

- **LRS** (Locally Redundant Storage) protects against local hardware failure.
- **ZRS** (Zone-Redundant Storage) protects against zone failure in supported regions.
- **GRS** (Geo-Redundant Storage) replicates to a paired region.
- **GZRS** combines zone and geo redundancy where supported.

Common traps:

- Redundancy is not the same as backup.
- Private Endpoint does not encrypt data by itself; it changes network access path.
- SAS (Shared Access Signature) grants limited access, but it is not a substitute for RBAC in all cases.
- Storage account firewall rules and private endpoints solve different access patterns.

Example access pattern:

```bash
# Generate a user delegation SAS for a blob container
az storage container generate-sas \
  --account-name stacct01 \
  --name backups \
  --permissions rl \
  --expiry 2026-05-01T00:00Z \
  --auth-mode login
```

**Compute and availability traps**

Availability sets, availability zones, and scale sets are often confused.

- **Availability sets** protect against hardware and rack-level failures within a datacenter.
- **Availability zones** protect against datacenter-level failures within a region.
- **Virtual Machine Scale Sets** help with horizontal scaling and can be zone-aware.
- **Autoscale** adjusts instance count based on metrics or schedules.

Common misconceptions:

- Availability zones are not available in every region or for every SKU.
- A scale set does not automatically make an application highly available unless the app is designed for it.
- A single VM in a zone is not the same as a zone-redundant architecture.

Practical design note:

- If the workload is stateless and horizontally scalable, a scale set behind a load balancer is often better than a single large VM.
- If the workload is stateful, you need a data-layer HA strategy in addition to compute redundancy.

**Database and platform service traps**

Platform services often have built-in HA, but the exact behavior differs.

- Some services are zone-redundant only in certain tiers or regions.
- Some services support active-active patterns, while others support active-passive failover.
- Some services require you to choose between cost, latency, and recovery objectives.

Examples to watch:

- Azure SQL Database can use zone redundancy in supported configurations.
- Cosmos DB has multi-region replication options, but consistency level affects behavior.
- App Service may need deployment slots, traffic routing, or zone redundancy depending on the scenario.
- Azure Cache for Redis, Service Bus, and other PaaS services have their own replication and failover models.

**Security service confusion**

Security questions often ask for the right control at the right layer.

- **Microsoft Defender for Cloud** is for posture management and threat protection.
- **Key Vault** stores secrets, keys, and certificates.
- **Azure Firewall** filters and logs network traffic centrally.
- **Network Security Groups (NSGs)** filter traffic at subnet or NIC scope.

Common traps:

- Key Vault is not a general-purpose database.
- NSGs do not inspect application payloads.
- Azure Firewall is not the same as a web application firewall.
- Defender for Cloud does not replace RBAC or policy.

Example NSG rule concept:

```bash
az network nsg rule create \
  --resource-group rg-net \
  --nsg-name nsg-app \
  --name AllowHttpsIn \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol Tcp \
  --source-address-prefixes Internet \
  --source-port-ranges "*" \
  --destination-port-ranges 443
```

**Hybrid and migration traps**

Hybrid scenarios often test whether you choose the simplest workable path.

- Use **Azure Migrate** for assessment and migration planning.
- Use **ASR** for replication-based migration or DR.
- Use **VPN/ExpressRoute** for ongoing hybrid connectivity.
- Use **Private Link** when the goal is private access to PaaS, not site connectivity.

Common misconceptions:

- Migration tooling does not equal DR.
- A hybrid connection does not automatically make on-premises resources reachable from every Azure subnet.
- DNS resolution is often the hidden dependency in hybrid designs.
- If the question mentions low latency or predictable throughput, ExpressRoute may be preferred over VPN.

**Scenario-based exam patterns**

1. **You need secure admin access to a VM without public IPs.**
   - Choose Azure Bastion.
   - Do not choose VPN unless the requirement is network connectivity.

2. **You need to expose an API with throttling, policies, and versioning.**
   - Choose API Management.
   - Do not choose Application Gateway unless the main need is regional HTTP load balancing.

3. **You need global web failover and edge routing.**
   - Choose Front Door.
   - Do not choose CDN unless the main need is caching static content.

4. **You need to restore a deleted database from last week.**
   - Choose backup/restore.
   - Do not choose failover groups as a substitute for retention.

5. **You need regional disaster recovery for VMs.**
   - Choose Site Recovery.
   - Do not choose Azure Backup alone.

6. **You need private access to Storage from a VNet.**
   - Choose Private Endpoint.
   - Do not confuse this with service endpoints or firewall-only access.

**How to eliminate traps during the exam**

Use a quick mental checklist before selecting an answer:

- Identify whether the question is about identity, network path, data access, or workload recovery.
- Check whether the requirement is management plane or data plane.
- Look for words like *global*, *regional*, *zone*, *private*, *public*, *failover*, *backup*, *restore*, *quota*, or *limit*.
- Verify whether the service exists in the target region and supports the required SKU.
- Confirm whether the solution needs prevention, detection, or recovery.

A practical decision flow:

1. Determine the business requirement and recovery objective.
2. Map it to the correct Azure service layer.
3. Check for region, SKU, quota, and subscription constraints.
4. Validate whether the service solves the exact problem or only part of it.
5. Prefer the simplest architecture that meets the stated requirement.

**High-frequency misconceptions to remember**

- **Entra ID** authenticates; **RBAC** authorizes Azure management actions; data-plane roles authorize service data access.
- **Backup** is not the same as **disaster recovery**.
- **Peering** is not the same as **gateway transit** or **forced routing**.
- **Bastion** is not a site-to-site connectivity tool.
- **Front Door** is not the same as **Application Gateway**.
- **CDN** is not the same as **global load balancing**.
- **Policy** is not the same as **RBAC**.
- **Locks** do not replace backup or DR.
- **Quota and capacity** issues can invalidate an otherwise correct design.
- **Zone redundancy** is not the same as cross-region failover.

**Final exam mindset**

AZ-305 rewards precise reading. The best answer is often the one that matches the exact scope, plane, and recovery objective, even if another service sounds broadly similar. When two options seem close, compare them by:

- access model,
- traffic direction,
- recovery behavior,
- regional support,
- and operational ownership.

If you can separate those dimensions quickly, you avoid most of the common traps.
