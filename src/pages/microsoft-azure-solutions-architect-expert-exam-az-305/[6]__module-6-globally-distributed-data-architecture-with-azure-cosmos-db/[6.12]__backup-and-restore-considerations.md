## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Backup and restore design goals**

Backup and restore is about more than copying data. In Azure architecture, you design for recovery objectives, operational simplicity, compliance, and cost. The right choice depends on whether you need protection against accidental deletion, corruption, ransomware, regional outage, or full workload failure.

- **RPO** (Recovery Point Objective) defines how much data loss is acceptable.
- **RTO** (Recovery Time Objective) defines how long recovery can take.
- Backups usually protect against logical corruption and deletion.
- Replication helps with availability and site failure, but it does not replace backup.
- Snapshots are fast recovery points, but they are not always enough for long-term retention or centralized governance.

A good exam answer usually distinguishes between:
- **Backup** for recoverability and retention
- **Replication** for availability and failover
- **Snapshots** for fast local rollback
- **Archive** for long-term retention and lower cost

**Choosing between backup, snapshots, and replication**

Use the recovery objective to decide the mechanism. If you need near-zero downtime for a regional outage, replication or failover is usually the primary design. If you need to recover from accidental deletion or ransomware, backup is essential. If you need quick rollback for a disk or file share, snapshots may be enough.

| Mechanism | Best for | Strengths | Limitations |
|---|---|---|---|
| Backup | Logical corruption, deletion, compliance retention | Centralized policy, retention, restore flexibility | Restore can take time |
| Snapshot | Fast point-in-time rollback | Very fast, low overhead | Limited retention and scope |
| Replication | High availability and disaster recovery | Low RTO for failover | Replicates corruption and deletion too |
| Archive | Long-term retention | Lowest storage cost | Slow retrieval |

A common architecture pattern is to combine them:
- Use **snapshots** for short-term operational recovery.
- Use **Azure Backup** for policy-based retention and restore.
- Use **replication** such as Azure Site Recovery for workload continuity.

**Azure Backup architecture**

Azure Backup is the primary managed backup service for many Azure and hybrid workloads. It centralizes policy, retention, monitoring, and restore operations. The service uses vaults to store recovery points and manage backup metadata.

The two main vault types are:
- **Recovery Services vault** for many classic backup scenarios, including Azure VMs, Azure Files, SQL in Azure VM, and hybrid workloads.
- **Backup vault** for newer workloads and scenarios, especially where the newer data protection model applies.

Use the vault type that matches the workload and the backup feature set. In exam scenarios, the vault choice often matters as much as the backup policy itself.

**Vault security and immutability**

Vault security is a major design concern because backup data is a high-value target for ransomware. Azure Backup supports controls that reduce the risk of deletion or tampering.

Key protections include:
- **Soft delete** helps recover backup items after accidental deletion.
- **Immutability** prevents changes to backup data for supported vault and configuration combinations.
- **Multi-user authorization** can require additional approval for critical operations.
- **Role-based access control** (RBAC) limits who can manage backup and restore actions.

> Immutability is not universal across all Azure backup scenarios. Treat it as a feature that depends on the vault type, workload, and configuration rather than a blanket guarantee.

Operationally, you should also protect the vault itself:
- Restrict who can delete the vault.
- Separate backup administration from workload administration.
- Use Azure Policy to enforce backup and retention standards.
- Monitor for disabled protection, failed jobs, and unusual restore activity.

**Encryption and key management**

Azure Backup data is encrypted, but the exact model depends on the workload and vault configuration. For some scenarios, Microsoft-managed keys are used by default; for others, customer-managed keys (CMK) are supported.

Important design points:
- If you use **customer-managed keys**, key availability becomes part of your recovery dependency chain.
- If the key vault or key is unavailable, restore operations may fail or be delayed.
- You should protect key vault access, key rotation processes, and access policies or RBAC assignments.
- Test recovery after key rotation and access changes.

For exam purposes, remember that encryption is not just a compliance checkbox. It affects restore feasibility, operational ownership, and incident response.

**Azure VM backup and restore**

Azure VM backup is one of the most common scenarios. Azure Backup can protect the VM without requiring an agent inside the guest OS for the basic VM backup workflow. It captures application-consistent or crash-consistent recovery points depending on workload and configuration.

Typical restore options include:
- **Create a new VM** from a recovery point.
- **Restore disks** and attach them to an existing VM.
- **Replace existing disks** in some scenarios.
- **File recovery** for browsing and copying files from a recovery point.

Restore scope can also include:
- **Cross-subscription restore** where supported and permitted.
- **Cross-region restore** for disaster recovery scenarios where the feature is available.
- **Restore to alternate region or zone** depending on the workload, vault, and region support.

A practical restore flow often looks like this:
1. Open the vault and select the protected VM.
2. Choose a recovery point.
3. Select the restore type: VM, disk, or file recovery.
4. Provide target settings such as resource group, virtual network, or storage account.
5. Validate the restored VM before putting it into production.

Example Azure CLI pattern for VM backup policy inspection:
```bash
az backup policy list \
  --resource-group rg-backup \
  --vault-name vault-prod \
  --output table
```

Example restore-oriented CLI pattern:
```bash
az backup recoverypoint list \
  --resource-group rg-backup \
  --vault-name vault-prod \
  --container-name vmapp01 \
  --item-name vmapp01 \
  --output table
```

> CLI syntax varies by workload and Azure CLI version. In exam scenarios, focus on the restore objective and required inputs rather than memorizing one exact command shape.

**Azure VM restore design considerations**

When designing VM recovery, think beyond “can I restore it?” and ask “where will it run after restore?”

- **Same-region restore** is useful for operational recovery after deletion or corruption.
- **Cross-region restore** supports regional disaster recovery when available.
- **Cross-subscription restore** helps with tenant or subscription reorganization and isolation.
- **Restore to alternate network** is often needed to avoid IP conflicts or to test recovery safely.

You should also consider:
- Whether the VM uses managed disks.
- Whether the application needs post-restore reconfiguration.
- Whether the VM depends on domain join, certificates, or identity services.
- Whether the restored VM should be isolated for malware analysis before rejoining production.

**Azure Files backup and restore**

Azure Files can be protected with Azure Backup for centralized retention and restore. This is useful when you need point-in-time recovery for file shares and protection against accidental deletion or corruption.

Common recovery patterns:
- Restore individual files or folders.
- Restore the entire share.
- Restore to an alternate location for comparison or validation.

Azure Files backup is often preferable to relying only on storage account snapshots when you need:
- Centralized policy management
- Longer retention
- Operational reporting
- Consistent restore workflows across many shares

For short-term recovery, snapshots can still be useful because they are fast and simple. For compliance or long retention, Azure Backup is usually the better choice.

**Azure Disk backup and snapshots**

Azure managed disks support snapshot-based recovery patterns. Snapshots are useful when you need a quick restore point for a disk or when you want to clone a disk for testing.

Use snapshots when:
- You need fast, local recovery.
- The scope is a single disk or a small set of disks.
- You do not need centralized retention policy.

Use Azure Backup when:
- You need policy-driven retention.
- You need centralized monitoring and governance.
- You need protection across many VMs or disks.

A practical design often combines both:
- Take snapshots before risky changes.
- Use Azure Backup for scheduled retention and long-term recovery.

**Azure SQL backup and restore**

Azure SQL workloads have different recovery models depending on the deployment type.

For **Azure SQL Database**:
- Point-in-time restore is built in.
- Long-term retention can be configured for compliance and historical recovery.
- Geo-restore can help recover from regional outages using geo-replicated backups.
- Active geo-replication or failover groups can reduce downtime, but they are not backups.

For **Azure SQL Managed Instance**:
- You can perform point-in-time restore at the instance or database level depending on the scenario.
- Managed Instance supports native backup and restore behavior that differs from Azure SQL Database.
- Restore times and options depend on the size of the instance, backup retention, and region availability.

Important distinctions:
- **Point-in-time restore** is for logical recovery from accidental changes or corruption.
- **Geo-restore** is for regional disaster recovery.
- **Failover groups** improve availability but do not replace backup retention.
- **Long-term retention** is for compliance and historical recovery, not immediate failover.

Example SQL Server backup to URL pattern for self-managed SQL Server:
```sql
CREATE CREDENTIAL [https://mystorageaccount.blob.core.windows.net/sqlbackups]
WITH IDENTITY = 'SHARED ACCESS SIGNATURE',
SECRET = '?sv=...';

BACKUP DATABASE [Sales]
TO URL = 'https://mystorageaccount.blob.core.windows.net/sqlbackups/Sales.bak'
WITH CREDENTIAL = 'https://mystorageaccount.blob.core.windows.net/sqlbackups',
     COMPRESSION,
     STATS = 10;
```

This pattern uses a SAS-based credential for SQL Server backup to Azure Blob Storage. It is not the same as Azure Backup managing the database backup for you.

**Azure Kubernetes Service and container workloads**

AKS (Azure Kubernetes Service) backup is different from VM backup because you usually need to protect both cluster state and application data. Kubernetes manifests can be recreated, but persistent volumes and application configuration often need backup.

Protect:
- Persistent volume data
- Cluster resources such as namespaces, deployments, and secrets where appropriate
- Application-specific configuration and external dependencies

Common approaches:
- Use a Kubernetes-aware backup solution for cluster objects and persistent volumes.
- Use storage snapshots for underlying persistent volumes when supported.
- Use application-consistent backup methods for databases running in containers.

For exam scenarios, the key question is whether the workload is stateless or stateful. Stateless workloads are usually redeployed from source control and IaC (Infrastructure as Code). Stateful workloads need data protection.

**Azure App Service and PaaS backup**

Azure App Service supports backup for app content and configuration in many scenarios. This is useful when you need to recover app files, settings, or deployment artifacts.

Consider:
- Backing up app content and configuration separately from source code.
- Using deployment slots and source control as part of your recovery strategy.
- Storing secrets in Key Vault rather than in app settings when possible.

App Service backup is not a substitute for a full application recovery plan. You still need:
- Source code repository
- Infrastructure templates
- Database backups
- Secret and certificate recovery

**Hybrid and on-premises backup**

Azure Backup also supports hybrid workloads. This is relevant when the architecture includes on-premises servers, branch offices, or legacy systems.

Common options include:
- **MARS** (Microsoft Azure Recovery Services) agent for file and folder backup from Windows servers.
- **Azure Backup Server** for more advanced on-premises and workload protection.
- Integration with Recovery Services vault for centralized policy and retention.

Use hybrid backup when:
- You need cloud-based retention for on-premises data.
- You want to reduce dependence on local backup infrastructure.
- You need a single operational model across cloud and on-premises systems.

**Backup Center and centralized operations**

Backup Center provides a centralized view for backup management across subscriptions and vaults. It is useful for governance, reporting, and operational oversight.

Use it to:
- View backup posture across resources.
- Identify unprotected assets.
- Track failed jobs and restore activity.
- Standardize policy enforcement.

This is especially valuable in large enterprises where backup ownership is distributed across teams and subscriptions.

**Retention design and policy tradeoffs**

Retention design is a balance between cost, compliance, and recovery usefulness. More retention increases storage cost and management complexity, but it can be required for legal or business reasons.

Design questions to ask:
- How far back do you need to restore?
- How often do you need recovery points?
- Do you need daily, weekly, monthly, or yearly retention?
- Is the workload regulated or subject to audit?
- Can older backups be moved to lower-cost tiers if supported?

A practical retention strategy often uses:
- Short-term frequent backups for operational recovery
- Weekly or monthly retention for business continuity
- Long-term retention for compliance

Example policy thinking:
- Daily backups for 30 days
- Weekly backups for 12 weeks
- Monthly backups for 12 months
- Yearly backups for 7 years

**RPO and RTO tradeoffs**

RPO and RTO drive the architecture more than the backup product name does.

If you need a low **RPO**:
- Increase backup frequency.
- Use application-consistent backups where needed.
- Consider replication or log-based recovery for databases.

If you need a low **RTO**:
- Pre-stage recovery infrastructure.
- Use snapshots or replication for faster failover.
- Automate restore and validation steps.
- Keep network, identity, and DNS dependencies documented.

A useful exam rule:
- Backup reduces data loss.
- Replication reduces downtime.
- Automation reduces human error during recovery.

**Restore testing and validation**

Backups are only useful if restores work. Regular restore testing is a core operational requirement.

Test:
- Full VM restore
- File-level restore
- Database point-in-time restore
- Cross-region recovery if your design depends on it
- Restore after key rotation or access changes

Validation should include:
- Application startup
- Data integrity checks
- Network connectivity
- Identity and certificate dependencies
- Security controls on the restored environment

A simple restore test checklist:
1. Restore to an isolated resource group or network.
2. Verify the workload starts successfully.
3. Confirm data matches the expected recovery point.
4. Document the actual restore time.
5. Compare results against RTO and RPO targets.

**Practical restore examples**

Azure portal restore flow for a VM:
1. Open the Recovery Services vault.
2. Select the protected item.
3. Choose a recovery point.
4. Select restore type.
5. Configure target resource group, virtual network, and storage.
6. Start the restore job and monitor progress.

Azure CLI example for listing backup items:
```bash
az backup item list \
  --resource-group rg-backup \
  --vault-name vault-prod \
  --backup-management-type AzureIaasVM \
  --output table
```

Azure CLI example for file share backup policy inspection:
```bash
az backup policy list \
  --resource-group rg-backup \
  --vault-name vault-prod \
  --backup-management-type AzureStorage \
  --output table
```

For storage account restore scenarios, remember that point-in-time restore and soft delete features have prerequisites and are not universal across all blob configurations. Always verify the account kind, redundancy, and feature support before assuming a restore path.

**Common exam traps**

- Assuming replication is the same as backup.
- Assuming snapshots provide long-term retention.
- Assuming all vaults support the same immutability settings.
- Assuming every workload can be restored cross-region.
- Assuming SQL Server backup to URL uses Azure Backup semantics.
- Assuming a restored VM will work without checking identity, DNS, and network dependencies.
- Assuming backup encryption details are identical across all vault types and workloads.

**Design guidance for AZ-305 scenarios**

When you see a backup question, identify the workload first:
- VM
- SQL Database
- SQL Managed Instance
- Azure Files
- App Service
- AKS
- Hybrid server

Then determine the recovery requirement:
- Fast rollback
- Long-term retention
- Regional disaster recovery
- Compliance retention
- Ransomware resilience

Then choose the mechanism:
- **Backup** for retention and logical recovery
- **Replication** for availability
- **Snapshots** for quick local recovery
- **Automation** for repeatable restore operations

A strong architecture usually layers these controls instead of relying on one feature alone.

**Key takeaways**

- Backup and restore design is driven by RPO, RTO, compliance, and workload type.
- Recovery Services vaults and Backup vaults serve different backup models.
- Immutability, soft delete, RBAC, and monitoring improve ransomware resilience.
- VM restore options can include disk, file, cross-subscription, and cross-region recovery where supported.
- SQL, AKS, App Service, and hybrid workloads each have distinct backup patterns.
- Restore testing is mandatory if you want confidence in the design.
