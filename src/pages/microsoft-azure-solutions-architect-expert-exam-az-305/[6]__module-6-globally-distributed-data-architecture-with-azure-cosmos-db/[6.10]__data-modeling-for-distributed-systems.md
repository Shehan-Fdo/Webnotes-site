## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Distributed data modeling: the design goal**

Distributed systems force you to optimize for more than storage. You model data to control latency, throughput, consistency, cost, and failure behavior across partitions and regions. In Azure, the most common exam-relevant tradeoff is between a model that is easy to query and one that is easy to scale.

A good model reduces cross-partition work, keeps hot paths predictable, and matches the access pattern of the application. A poor model often looks normalized and elegant on paper, but it creates expensive joins, hot partitions, or multi-item writes that fail under load.

- **Access pattern first**: design around the queries and transactions you must support most often.
- **Partitioning**: choose keys that spread load and keep related data together when needed.
- **Consistency**: select the weakest model that still satisfies business correctness.
- **Operational fit**: separate transactional workloads from analytics when the query shape diverges.

**Core modeling principles**

Distributed databases reward denormalization when it reduces cross-partition reads. They also punish designs that assume a single-node relational mindset. The right model depends on whether you need point lookups, range queries, transactional updates, event history, or reporting.

Use these principles as a decision framework:

- **Model for reads and writes together**: a design that is fast to read but expensive to write can still fail at scale.
- **Keep transactions local**: if multiple items must change atomically, place them in the same partition or aggregate.
- **Avoid global coordination**: distributed locks, cross-partition transactions, and fan-out queries reduce availability and increase latency.
- **Expect evolution**: schema and access patterns change, so prefer models that can absorb new fields and new query paths.

**Relational thinking versus distributed thinking**

Relational design emphasizes normalization, foreign keys, and joins. Distributed design emphasizes locality, partition affinity, and precomputed views. You still use relational concepts, but you apply them differently.

| Concern | Relational-first approach | Distributed-first approach |
|---|---|---|
| Data shape | Normalize to reduce duplication | Denormalize to reduce joins and remote reads |
| Transactions | Multi-table ACID transactions | Keep atomic scope small and local |
| Querying | Flexible ad hoc joins | Optimize for known access patterns |
| Scaling | Vertical scale and read replicas | Horizontal partitioning and regional distribution |
| Failure mode | Centralized database bottleneck | Hot partitions, cross-partition latency, replication lag |

**Azure services and where modeling matters**

Different Azure data services impose different modeling constraints. The exam often tests whether you can match the workload to the right store and then model it correctly.

- **Azure Cosmos DB**: best when you need globally distributed, low-latency operational data with flexible schema and partition-aware design.
- **Azure SQL Database / Hyperscale**: best when you need relational semantics, strong SQL querying, and transactional integrity, with scale features that still preserve a relational model.
- **Azure Cache for Redis**: best as a cache, session store, or ephemeral acceleration layer, not as the system of record.
- **Azure Data Lake Storage / Synapse / Azure Data Explorer**: best for analytics, large-scale exploration, and time-series or log-heavy workloads that do not belong in the operational store.

> Do not force an operational database to serve as a reporting warehouse if the query patterns are fundamentally different. Offload analytics when scans, aggregations, or historical retention dominate.

**Azure Cosmos DB modeling fundamentals**

Azure Cosmos DB is the most exam-relevant distributed data store for modeling questions. It is schema-agnostic at the application level, but it is not model-agnostic: your partition key, item shape, and consistency choice determine performance and cost.

Cosmos DB stores JSON-like items in containers. The container is the unit of throughput and partitioning. Your design must account for:

- **Logical partitioning**: items with the same partition key value are grouped together.
- **Physical partitioning**: Azure distributes logical partitions across physical partitions behind the scenes.
- **Request Unit (RU)** consumption: reads, writes, and queries consume throughput.
- **Indexing**: automatic indexing helps queries, but indexing strategy still affects cost.

**Partition key design**

The partition key is the most important design decision in Cosmos DB. It determines data distribution, transaction scope, and query efficiency.

Choose a partition key that:

- Has enough cardinality to distribute load.
- Avoids hot spots from skewed traffic.
- Groups items that must be read or written together.
- Supports the dominant query patterns.

Common patterns:

- **Tenant-based key**: use when most operations are scoped to a tenant and tenant isolation matters.
- **User-based key**: use when user-centric reads and writes dominate.
- **Order-based key**: use when order header and line items must be updated together.
- **Synthetic key**: combine multiple attributes when a single natural key is too low-cardinality or too skewed.

Example synthetic key:

```json
{
  "id": "order-100045",
  "tenantId": "contoso",
  "region": "us",
  "pk": "contoso|us",
  "type": "Order",
  "status": "Pending",
  "total": 149.95
}
```

This pattern helps when `tenantId` alone is too hot or when you need to spread load by region while keeping tenant-local operations together.

**Partition key anti-patterns**

Bad partition choices are a common exam trap.

- **Low-cardinality keys**: values like `status`, `country`, or `type` create hot partitions because too many items share the same key.
- **Monotonic keys**: timestamps or sequential IDs can concentrate writes if the workload always targets the newest value.
- **Query-only keys**: choosing a key because it is convenient for one query can break write distribution.
- **Overly broad keys**: putting too much unrelated data under one partition increases contention and transaction scope.

A better approach is to align the key with the unit of work, not just the query filter.

**Bad model versus corrected model**

Bad model:

```json
{
  "id": "1",
  "status": "Open",
  "createdAt": "2026-04-26T10:00:00Z",
  "customerName": "Adele",
  "lineItems": [
    { "sku": "A1", "qty": 2 },
    { "sku": "B2", "qty": 1 }
  ]
}
```

If `status` is the partition key, all open orders may pile into the same partition.

Corrected model:

```json
{
  "id": "order-1001",
  "tenantId": "contoso",
  "pk": "contoso|order-1001",
  "status": "Open",
  "createdAt": "2026-04-26T10:00:00Z",
  "customerName": "Adele",
  "lineItems": [
    { "sku": "A1", "qty": 2 },
    { "sku": "B2", "qty": 1 }
  ]
}
```

This version improves distribution, but you would still choose the exact key based on the dominant access pattern. If order header and line items must be updated together, you may instead use a partition key that groups all items for one order.

**Hierarchical partition keys and high-cardinality workloads**

For very large or skewed workloads, hierarchical partitioning can help you preserve locality while improving distribution. In Cosmos DB, hierarchical partition keys let you combine multiple path values so that the effective partition key has more entropy.

Use hierarchical or composite-style partitioning when:

- A single natural key is too hot.
- You need tenant isolation plus regional spread.
- You have many items per tenant and need a second dimension to avoid hotspots.

Examples:

- `tenantId / userId`
- `customerId / orderId`
- `deviceId / day`

This is useful when one tenant is huge or one customer generates disproportionate traffic. The first component preserves locality; the later component improves spread.

**Very high-cardinality versus skewed workloads**

- **Very high-cardinality workloads**: choose a key that naturally spreads writes, such as `userId` or `deviceId`, if each entity is independently accessed.
- **Skewed workloads**: add a synthetic suffix or second dimension to avoid one partition absorbing most traffic.
- **Time-series workloads**: avoid pure timestamps as keys; combine time with a device, tenant, or shard component.

A common pattern is sharding a hot logical entity:

```text
pk = tenantId|customerId|shardNumber
```

You then compute `shardNumber` from a hash or modulo of a stable identifier. This spreads writes while keeping the tenant/customer boundary visible.

**Schema modeling: embed or separate**

Cosmos DB supports flexible JSON documents, but you still need to decide whether to embed related data or split it into separate items or containers.

Embed when:

- The child data is usually read with the parent.
- The child set is small or bounded.
- You need atomic updates within one aggregate.
- You want to avoid extra reads.

Separate when:

- The child collection can grow large or unbounded.
- Different parts of the data have different lifecycles.
- You need independent scaling or retention.
- The child data is queried independently at high volume.

Practical guidance:

- Use **embedded documents** for order line items, profile settings, or small nested state.
- Use **separate items in the same container** for large aggregates that still need transactional batch support.
- Use **separate containers** when the access pattern, indexing, retention, or throughput profile differs materially.

Cosmos DB item size limits and RU costs matter here. Large embedded arrays can become expensive to update because the whole document may be rewritten. If a child collection grows without bound, split it into separate items.

**Container design and when to split containers**

Use separate containers when one of these is true:

- The data has a different partition key requirement.
- The indexing policy should differ significantly.
- The retention or TTL (time to live) policy differs.
- The workload is operational in one case and analytical or archival in another.
- The security boundary differs.

Example:

- `Orders` container for current order state.
- `OrderEvents` container for append-only event history.
- `Customers` container for profile data.
- `AuditLogs` container for compliance records.

This separation helps you tune throughput and indexing independently.

**Transactional design patterns**

Distributed systems often need multi-item correctness without relying on cross-partition transactions. In Cosmos DB, transactional batch works only within a single logical partition, so you should design aggregates accordingly.

Use these patterns:

- **Aggregate boundary**: keep all items that must change atomically under one partition key.
- **Transactional batch**: use for multiple creates, updates, or deletes within one partition.
- **Idempotency**: make writes safe to retry by using stable IDs or operation tokens.
- **Outbox pattern**: write the business change and the integration event in the same transaction boundary, then publish asynchronously.
- **Inbox pattern**: deduplicate incoming messages before applying side effects.

Example outbox shape:

```json
{
  "id": "evt-9001",
  "pk": "contoso|order-1001",
  "type": "OutboxEvent",
  "eventType": "OrderCreated",
  "payload": {
    "orderId": "order-1001",
    "amount": 149.95
  },
  "published": false
}
```

A background worker reads unpublished events and sends them to Service Bus, Event Grid, or another downstream system.

**Transactional batch example**

```csharp
var batch = container.CreateTransactionalBatch(new PartitionKey("contoso|order-1001"))
    .CreateItem(new { id = "order-1001", pk = "contoso|order-1001", type = "Order", status = "Pending" })
    .CreateItem(new { id = "line-1", pk = "contoso|order-1001", type = "OrderLine", sku = "A1", qty = 2 })
    .CreateItem(new { id = "outbox-1", pk = "contoso|order-1001", type = "OutboxEvent", eventType = "OrderCreated" });

var response = await batch.ExecuteAsync();
```

This works because all items share the same partition key.

**Consistency choices and their tradeoffs**

Consistency is a business decision, not just a database setting. In Cosmos DB, consistency affects read freshness, latency, and availability behavior during regional failures.

The main models are:

- **Strong**: reads always see the latest committed write. This gives the strongest correctness but increases latency and reduces flexibility in multi-region scenarios.
- **Bounded staleness**: reads lag by a bounded number of versions or time. Useful when you can tolerate small delays but want predictable freshness.
- **Session**: a client sees its own writes and monotonic reads within a session. This is the common default for many applications.
- **Consistent prefix**: reads never see out-of-order writes, but they may lag behind the latest state.
- **Eventual**: reads may be stale, but the system can offer the lowest latency and highest flexibility.

Exam-oriented guidance:

- Use **strong** when correctness is critical and the workload can tolerate the latency and regional constraints.
- Use **session** for user-facing applications that need read-your-writes behavior without the cost of strong consistency.
- Use **bounded staleness** when you need predictable freshness for globally distributed reads.
- Use **consistent prefix** or **eventual** for telemetry, feeds, or non-critical read models.

> Strong consistency increases coordination cost and can reduce availability options in multi-region designs. Choose it only when the business requirement truly needs it.

**Consistency and failover behavior**

In multi-region deployments, consistency affects how the system behaves during failover and region outages.

- **Stronger consistency** generally means more coordination and higher write/read latency across regions.
- **Weaker consistency** can improve availability and latency, especially for geographically distributed readers.
- **Session consistency** is often a practical balance because it preserves user experience while allowing regional distribution.

For exam scenarios, ask:

- Do users need to read their own latest write immediately?
- Can the application tolerate stale reads for a short period?
- Is the workload read-heavy across multiple geographies?
- What happens if the primary region fails?

If the answer requires immediate global correctness, stronger consistency may be justified. If not, prefer a weaker model and design the application to tolerate temporary staleness.

**Conflict resolution in multi-write scenarios**

When multiple regions can write, conflicts can occur. Cosmos DB supports automatic conflict resolution and custom conflict resolution depending on the API and configuration.

Common approaches:

- **Last writer wins**: choose the winning item based on a timestamp or custom numeric property.
- **Custom conflict resolution**: define a user-defined conflict resolution procedure or path-based rule to merge or select values.
- **Application-level merge**: detect conflicts and reconcile them in the application.

Practical examples:

- For a shopping cart, you may merge quantities rather than overwrite the whole document.
- For a profile record, you may prefer the latest verified email field.
- For telemetry, you may keep the highest sequence number or latest timestamp.

A custom rule might say: keep the document with the highest `priority` value, or merge arrays by unique event ID. The exact implementation depends on the API and conflict policy, but the key idea is that you define deterministic resolution logic instead of relying on a generic overwrite.

**Query modeling and indexing**

Distributed query performance depends on whether the query can stay within a partition or must fan out across many partitions.

Prefer queries that:

- Filter by partition key.
- Use selective predicates.
- Avoid large cross-partition scans.
- Return only needed fields.

Example of a partition-friendly query:

```sql
SELECT c.id, c.status, c.total
FROM c
WHERE c.pk = "contoso|order-1001"
```

Example of a cross-partition query:

```sql
SELECT c.id, c.status
FROM c
WHERE c.status = "Open"
```

The second query may fan out if `status` is not the partition key. That can be acceptable for low-volume reporting, but it is usually a poor operational pattern.

Indexing guidance:

- Keep automatic indexing unless you have a strong reason to tune it.
- Exclude large, rarely queried paths to reduce RU cost.
- Add composite indexes when you need efficient multi-property sorting or filtering.
- Test query RU consumption with realistic data volumes.

**Modeling for analytics and reporting**

Operational stores are not ideal analytics engines. If you need dashboards, trend analysis, or large aggregations, offload the data.

Use these targets:

- **Azure Synapse Analytics** for warehouse-style reporting and SQL-based analytics.
- **Azure Data Lake Storage** for raw, curated, or historical data at scale.
- **Azure Data Explorer** for log, telemetry, and time-series exploration.

Patterns for offload:

- **Change feed** from Cosmos DB to downstream analytics.
- **Event-driven replication** into a lake or warehouse.
- **ETL/ELT** pipelines for curated reporting models.

This separation keeps the operational store fast and predictable while allowing analytics to scan large volumes without impacting user transactions.

**Redis as a supporting model, not the source of truth**

Azure Cache for Redis is useful for acceleration, but it is not a primary distributed data model for durable business state.

Use Redis for:

- Caching hot reads.
- Session state.
- Rate limiting.
- Short-lived coordination with careful design.

Be cautious with distributed locking. A simple `SET key value NX PX 30000` pattern can implement a basic lock, but correct distributed locking requires careful handling of expiry, ownership, and failure scenarios. Do not assume a cache lock is equivalent to a durable transaction.

Example:

```bash
redis-cli SET lock:order-1001 abc123 NX PX 30000
```

If you use locks, ensure you also validate ownership before release and understand the failure model.

**Azure SQL Hyperscale in distributed design**

Azure SQL Hyperscale extends relational SQL with separation of compute and storage and supports read scale-out. It is still a relational database with a single-writer model for writes, so it is not a generic distributed write-sharding system.

Use Hyperscale when:

- You need SQL Server compatibility and relational integrity.
- You need large database sizes and fast scale operations.
- You want read scale-out for reporting or read-heavy workloads.
- You still need a familiar relational schema and T-SQL.

Do not assume Hyperscale solves every distributed scaling problem. If the workload needs global write distribution, Cosmos DB or a different architecture may be a better fit.

**Practical Azure CLI examples**

Use CLI examples as references, but verify current syntax for your API and service tier.

Create a Cosmos DB SQL API account:

```bash
az cosmosdb create \
  --name contoso-cosmos \
  --resource-group rg-data \
  --locations regionName=eastus failoverPriority=0 isZoneRedundant=false \
  --default-consistency-level Session \
  --kind GlobalDocumentDB
```

Create a Cosmos DB SQL database and container:

```bash
az cosmosdb sql database create \
  --account-name contoso-cosmos \
  --resource-group rg-data \
  --name appdb

az cosmosdb sql container create \
  --account-name contoso-cosmos \
  --resource-group rg-data \
  --database-name appdb \
  --name orders \
  --partition-key-path "/pk"
```

Create an Azure SQL database with Hyperscale service objective:

```bash
az sql db create \
  --resource-group rg-data \
  --server contoso-sql \
  --name appdb \
  --edition Hyperscale \
  --compute-model Provisioned \
  --family Gen5 \
  --capacity 2
```

> CLI parameters can change across Azure CLI versions and service APIs. Validate the current command syntax before using it in automation.

**Design checklist for exam scenarios**

1. Identify the dominant access pattern.
2. Decide whether the workload is operational, analytical, or mixed.
3. Choose the data store that matches the workload.
4. Select a partition key that spreads load and preserves locality.
5. Decide what must be atomic and keep it within one partition or aggregate.
6. Choose the weakest consistency level that still meets business requirements.
7. Plan for conflict resolution if multi-region writes are possible.
8. Offload reporting and heavy analytics to a separate platform.
9. Test for hot partitions, cross-partition queries, and RU cost.
10. Revisit the model when the access pattern changes.

**Common exam traps**

- Assuming normalization is always best.
- Using a low-cardinality partition key because it is easy to query.
- Designing cross-partition transactions as if they were local.
- Treating cache as a system of record.
- Using strong consistency by default without considering latency and availability.
- Querying operational data for reporting when a lake or warehouse is more appropriate.
- Ignoring how multi-region failover changes read freshness and write behavior.

**Quick mental model**

Think in terms of aggregates, partitions, and access paths:

- **Aggregate**: the smallest unit that must be consistent together.
- **Partition**: the smallest unit that scales and distributes together.
- **Access path**: the query or command that must be fast and reliable.

If those three align, your model is usually on the right track.
