## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Decision framing**

Azure Cosmos DB is a globally distributed, multi-model database service designed for low-latency applications, elastic scale, and predictable performance at high throughput. Azure SQL is a relational database platform for structured data, strong relational querying, and transactional workloads that benefit from SQL semantics, joins, constraints, and mature reporting patterns.

The exam usually tests whether you can match the service to the workload. The key is not “which service is best,” but which service best fits the data shape, access pattern, scale, consistency needs, and operational model.

- **Cosmos DB** fits applications that need globally distributed reads and writes, flexible schema, and very high request rates.
- **Azure SQL** fits relational workloads, complex joins, reporting, and transactional systems with normalized data.
- **Other Azure data services** may be better when the workload is analytical, file-based, open-source relational, or event-driven.

**Core Cosmos DB concepts**

Cosmos DB is primarily a NoSQL database service, even though it offers multiple APIs. Those APIs influence how you model data and what client libraries or query syntax you use.

- **Core (SQL) API** is the default Cosmos DB API for JSON document workloads and SQL-like querying over documents.
- **MongoDB API** is appropriate when you want MongoDB wire protocol compatibility or an easier migration path from MongoDB applications.
- **Cassandra API** fits wide-column workloads and applications built around Cassandra-style data models.
- **Gremlin API** supports graph workloads where relationships and traversals matter more than relational joins.
- **Table API** supports key-value and wide-row style access patterns similar to Azure Table Storage, but with Cosmos DB capabilities.
- API choice matters because it affects application compatibility, query style, and migration effort.

Cosmos DB capacity is commonly expressed in **request units per second (RU/s)**. RU/s is the throughput and cost abstraction you plan around. Reads, writes, and queries consume RUs based on item size, indexing, query complexity, and partitioning.

- A simple point read is usually cheaper than a cross-partition query.
- A poorly designed partition key can create hot partitions and uneven RU consumption.
- Provisioned throughput and autoscale help you control performance and cost.

**When Cosmos DB is the better choice**

Use Cosmos DB when the application needs predictable low latency at scale and the data model is naturally document, key-value, wide-column, or graph oriented.

- **Global distribution** is a strong fit when users are spread across regions and you need low-latency access near each user.
- **Elastic scale** is useful when traffic is spiky or grows quickly.
- **Flexible schema** helps when records evolve frequently or different item types share a container.
- **High availability** is important when the application must tolerate regional failures with minimal disruption.
- **Multi-region writes** can be useful for active-active designs, though they require careful conflict handling.

Typical Cosmos DB scenarios include:

- IoT telemetry ingestion with high write volume and time-series-like access patterns.
- E-commerce catalogs where product documents vary by category and attributes change often.
- User profiles, shopping carts, session state, and personalization data.
- Event-driven applications that use the **change feed** to react to inserts and updates.
- Content management or metadata stores where schema flexibility matters more than relational joins.

> Cosmos DB is not a relational database service. It is usually a poor fit for workloads that depend on complex ad hoc joins, normalized reporting, or cross-entity relational queries.

**When Azure SQL is the better choice**

Use Azure SQL when the workload is relational, transactional, and query-heavy in a way that benefits from SQL Server semantics.

- **Normalized data** works well when entities have clear relationships and you want to enforce integrity.
- **Joins and aggregations** are natural in SQL and often simpler than denormalizing data for a document model.
- **ACID transactions** are a strong fit for business systems that need reliable multi-row consistency within the database engine.
- **Reporting and BI** scenarios often integrate more naturally with relational schemas and SQL tooling.
- **Stored procedures, views, constraints, and indexes** support mature relational application patterns.

Azure SQL deployment options matter in architecture decisions:

- **Single database** is the default choice for most isolated application databases.
- **Elastic pool** is useful when you have many databases with variable usage and want to share resources efficiently.
- **Managed instance** is appropriate when you need near-full SQL Server compatibility, instance-level features, or easier migration from on-premises SQL Server.
- **Hyperscale** is important for very large relational databases that need rapid storage growth and scale-out capabilities.

Azure SQL is often preferred over Cosmos DB when:

- You need complex joins across multiple tables.
- You need strong relational integrity and foreign keys.
- You need ad hoc reporting with flexible SQL queries.
- You are migrating a traditional line-of-business application with an existing relational schema.
- You want a familiar SQL Server ecosystem for tooling, administration, and migration.

**When other Azure data services are a better fit**

Not every workload belongs in Cosmos DB or Azure SQL. The exam often expects you to recognize adjacent services.

- **Azure Database for PostgreSQL** is a strong choice when you want open-source relational compatibility, PostgreSQL features, or application portability.
- **Azure Database for MySQL** fits MySQL-based applications and migrations.
- **Azure Synapse Analytics** is better for large-scale analytics, data warehousing, and complex analytical queries over large datasets.
- **Azure Data Lake Storage** is better for raw files, logs, and large unstructured datasets used in analytics pipelines.
- **Azure Table Storage** can be a lower-cost option for simple key-value storage with minimal query requirements, but it lacks many of Cosmos DB’s global distribution and performance features.
- **Azure Cache for Redis** is better for ultra-low-latency caching, session storage, and transient data, not as a primary system of record.

A practical rule:

- Choose **Cosmos DB** for globally distributed operational data with flexible schema and high scale.
- Choose **Azure SQL** for relational transactional systems.
- Choose **PostgreSQL/MySQL** when portability or open-source compatibility matters.
- Choose **Synapse** for analytics.
- Choose **Redis** for caching.
- Choose **Table Storage** only when the access pattern is very simple and cost sensitivity outweighs advanced capabilities.

**Data modeling and partitioning in Cosmos DB**

Partitioning is one of the most important Cosmos DB exam topics. A good partition key spreads load evenly and keeps related data together for efficient reads and writes.

- A **logical partition** groups items with the same partition key value.
- A **physical partition** is the internal storage and throughput unit managed by Cosmos DB.
- Queries and transactions are most efficient when they stay within a single logical partition.

Good partition key choices usually have:

- High cardinality.
- Even distribution of traffic.
- Alignment with the most common query pattern.
- Stability over time.

Common anti-patterns include:

- Using a low-cardinality value such as `country`, `status`, or `gender` when many items share the same value.
- Using a monotonically increasing value such as `timestamp` when it creates a hot partition.
- Choosing a partition key that does not match the most common access pattern, forcing cross-partition queries.
- Using a key that changes frequently, which makes updates expensive and awkward.

Example of a poor partition key:

```json
{
  "id": "order-1001",
  "customerId": "C123",
  "status": "Pending",
  "createdAt": "2026-04-26T10:15:00Z"
}
```

If you partition only by `status`, most orders may land in a few partitions such as `Pending` or `Shipped`, causing hot partitions and uneven RU consumption.

Better approach:

```json
{
  "id": "order-1001",
  "customerId": "C123",
  "orderDate": "2026-04-26",
  "status": "Pending"
}
```

If your access pattern is “get all orders for a customer,” `customerId` is often a better partition key because it groups related items and distributes load more evenly.

**Transactions, consistency, and query behavior**

Cosmos DB supports transactions only within a single logical partition. That means transactional batch and stored procedures are useful when multiple items share the same partition key value, but they are not general-purpose cross-partition transaction mechanisms.

- **Transactional batch** is useful for atomic operations on multiple items in the same partition.
- **Stored procedures** run within a partition and can enforce server-side logic for that partition.
- **Cross-partition transactions** are not the Cosmos DB model.
- **Consistency levels** are configurable and include strong, bounded staleness, session, consistent prefix, and eventual consistency.

Azure SQL also supports configurable isolation levels, but it remains a relational engine with ACID behavior and rich query semantics. The comparison is not “Cosmos DB has consistency and SQL does not”; rather, the services expose different consistency and transaction models.

Cosmos DB is a poor fit when you need:

- Complex joins across many entities.
- Ad hoc reporting with unpredictable query shapes.
- Cross-entity transactional workflows spanning multiple partitions.
- Heavy relational integrity enforcement at the database layer.

**Change feed and event-driven patterns**

The **change feed** is a major Cosmos DB capability for downstream processing. It lets you react to inserts and updates without polling the database.

- Use it for event-driven architectures.
- Use it to build materialized views or projections.
- Use it to feed analytics, search indexes, or integration pipelines.
- Use it when you need near-real-time processing of data changes.

A common pattern is to write operational data to Cosmos DB and use the change feed to push updates into another store optimized for search or analytics.

**Practical selection examples**

1. Choose Cosmos DB for a global shopping app that stores product catalogs, user sessions, and shopping carts with low-latency reads worldwide.
2. Choose Azure SQL for an accounting system with normalized tables, foreign keys, and complex financial reporting.
3. Choose PostgreSQL for a SaaS application that wants open-source compatibility and relational features without SQL Server-specific dependencies.
4. Choose Synapse for a data warehouse that aggregates sales history across billions of rows.
5. Choose Redis for a session cache that must respond in milliseconds and can tolerate ephemeral data loss.

**CLI and deployment examples**

When you provision Cosmos DB from the Azure CLI, specify the API kind and locations correctly. For a NoSQL account, `--kind GlobalDocumentDB` is commonly used.

```bash
az cosmosdb create \
  --name mycosmosacct \
  --resource-group rg-data \
  --kind GlobalDocumentDB \
  --locations regionName=eastus failoverPriority=0 isZoneRedundant=false \
  --default-consistency-level Session
```

Create a SQL database and container in a Cosmos DB account:

```bash
az cosmosdb sql database create \
  --account-name mycosmosacct \
  --resource-group rg-data \
  --name appdb

az cosmosdb sql container create \
  --account-name mycosmosacct \
  --resource-group rg-data \
  --database-name appdb \
  --name orders \
  --partition-key-path "/customerId" \
  --throughput 400
```

Create an Azure SQL database:

```bash
az sql server create \
  --name mysqlserver123 \
  --resource-group rg-data \
  --location eastus \
  --admin-user sqladmin \
  --admin-password 'Use-A-Strong-Password!123'

az sql db create \
  --resource-group rg-data \
  --server mysqlserver123 \
  --name appdb \
  --service-objective S0
```

These examples illustrate the architectural difference:

- Cosmos DB requires you to think about API choice, partitioning, and RU/s.
- Azure SQL requires you to think about schema, indexing, service tier, and relational design.

**Exam decision cues**

Use these cues to answer AZ-305-style questions quickly:

- If the requirement says **global distribution**, **single-digit millisecond latency**, or **elastic throughput**, consider Cosmos DB.
- If the requirement says **joins**, **foreign keys**, **normalized schema**, or **reporting**, consider Azure SQL.
- If the requirement says **open-source compatibility**, consider PostgreSQL or MySQL.
- If the requirement says **analytics at scale**, consider Synapse or a lake-based design.
- If the requirement says **simple key-value storage at low cost**, consider Table Storage or Redis depending on latency and durability needs.
- If the requirement says **event-driven downstream processing**, consider Cosmos DB change feed.

**Common exam traps**

- Do not choose Cosmos DB just because it is “modern” or “scalable.”
- Do not choose Azure SQL when the workload needs multi-region low-latency writes and flexible schema.
- Do not assume Cosmos DB is a drop-in replacement for relational databases.
- Do not assume Azure SQL cannot scale; it can, especially with elastic pools and Hyperscale, but the scaling model is different.
- Do not ignore partition key design; a bad partition key can make a well-designed Cosmos DB solution expensive and slow.

**Quick comparison summary**

| Capability | Cosmos DB | Azure SQL | PostgreSQL/MySQL | Synapse |
|---|---|---|---|---|
| Data model | Document, key-value, wide-column, graph | Relational | Relational | Analytical/warehouse |
| Global distribution | Strong fit | Limited compared with Cosmos DB | Varies by service | Not primary focus |
| Joins and ad hoc SQL | Limited compared with relational engines | Strong | Strong | Strong for analytics, not OLTP |
| Transaction scope | Single logical partition for batch/procs | Database engine transactions | Database engine transactions | Analytical patterns |
| Schema flexibility | High | Moderate to low | Moderate | Varies |
| Best use case | Operational NoSQL at scale | OLTP and relational apps | Open-source relational apps | Analytics and warehousing |

**Summary**

Choose Cosmos DB when you need globally distributed, low-latency, schema-flexible operational data with predictable throughput and a partition-aware design. Choose Azure SQL when the workload is relational, transactional, and query-rich. Choose PostgreSQL, MySQL, Synapse, Redis, or Table Storage when their specific strengths better match the workload than either Cosmos DB or Azure SQL.

The exam answer is usually driven by the access pattern, not the brand name of the database.
