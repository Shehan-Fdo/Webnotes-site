## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Global distribution overview**

Global distribution patterns help you design applications and data platforms that remain available when a region fails, while also meeting latency, compliance, and cost goals. In Azure, the main design choice is not just *how* to fail over, but *what* should fail over: traffic, application state, data, or the entire workload.

A strong design usually separates these concerns:

- **Traffic distribution** decides where users connect.
- **Application deployment** decides where code runs.
- **Data replication** decides where state lives.
- **Recovery orchestration** decides how failover happens and how quickly.

For AZ-305, you should be able to compare active-active and active-passive designs, choose the right global routing service, and explain the trade-offs in consistency, cost, and operational complexity.

**Core design goals**

Global architectures usually optimize for a mix of these goals:

- **Availability**: keep the service reachable during regional outages.
- **Performance**: reduce latency by serving users from nearby regions.
- **Resiliency**: recover from zone, region, or dependency failures.
- **Consistency**: control how quickly data changes appear across regions.
- **Cost**: avoid paying for unnecessary duplicate capacity.
- **Compliance**: keep data in approved geographies or sovereign boundaries.

A common exam trap is assuming that the fastest failover design is always the best. In practice, you balance recovery time objective (RTO), recovery point objective (RPO), operational complexity, and user experience.

**Active-active and active-passive patterns**

Active-active means multiple regions serve traffic at the same time. Active-passive means one region serves traffic while another stays warm or cold until failover.

- **Active-active**
  - Best for high availability and low latency at global scale.
  - Requires careful conflict handling if multiple regions can write the same data.
  - Usually costs more because you run more capacity all the time.
  - Works well for stateless web tiers and read-heavy workloads.

- **Active-passive**
  - Simpler to operate and easier to reason about.
  - Often cheaper because the secondary region can run at reduced capacity.
  - Failover can be slower because you must promote the standby region.
  - Works well when writes must be centralized or when the app is not designed for multi-writer conflicts.

A practical rule: use active-active for stateless front ends and read scaling, but use active-passive when the data layer or business process cannot tolerate concurrent writes.

**Traffic distribution services**

Azure provides several global traffic management options, and they are not interchangeable.

- **Azure Front Door**
  - Best for global HTTP/HTTPS applications.
  - Provides layer 7 routing, TLS termination, WAF (Web Application Firewall), caching, and acceleration.
  - Routes users to healthy origins based on latency, priority, or weighted rules.
  - Supports origin health probes and origin groups for failover.
  - Can front private origins with Premium features and private link scenarios.

- **Azure Traffic Manager**
  - DNS-based global traffic distribution.
  - Works at the DNS level, not as a proxy.
  - Supports multiple routing methods, including performance, priority, weighted, geographic, and subnet.
  - Useful for non-HTTP endpoints that can be reached by DNS-resolved addresses, but it does not proxy traffic itself.
  - Failover behavior depends on DNS caching and client resolver behavior.

- **Azure Load Balancer**
  - Regional layer 4 load balancing for TCP and UDP.
  - Not a global traffic manager by itself.
  - Commonly paired with Traffic Manager or Front Door for global designs.

- **Application Gateway**
  - Regional layer 7 load balancer for HTTP/HTTPS.
  - Often used inside a region behind a global entry point.
  - Can be combined with Traffic Manager for multi-region web apps, but Front Door is usually the simpler global option.

**Azure Front Door in depth**

Azure Front Door is the preferred global entry point for internet-facing web applications when you need fast failover, edge acceleration, and layer 7 routing.

Key concepts:

- **Origin**: a backend endpoint such as an App Service, Application Gateway, AKS ingress, or custom host.
- **Origin group**: a set of origins that Front Door monitors together.
- **Health probe**: a request Front Door sends to evaluate backend health.
- **Route**: the rule that maps incoming requests to an origin group.
- **Priority and weight**: controls failover and traffic split behavior.

Front Door failover happens at the origin group level. If the primary origin becomes unhealthy, Front Door can route to a secondary origin in the same group or to another group depending on your configuration.

Typical health evaluation behavior:

- Front Door sends probes on a schedule.
- It marks an origin unhealthy after repeated probe failures.
- It routes traffic only to healthy origins unless no healthy origin exists.
- It can use path-specific probes, which is important when the app’s root path returns `200 OK` but a critical dependency is broken.

Example origin group configuration conceptually:

```json
{
  "originGroups": [
    {
      "name": "primary-group",
      "healthProbeSettings": {
        "probePath": "/health",
        "probeProtocol": "Https",
        "probeRequestType": "GET"
      },
      "origins": [
        {
          "name": "eastus-app",
          "hostName": "app-eastus.contoso.com",
          "priority": 1,
          "weight": 100
        },
        {
          "name": "westus-app",
          "hostName": "app-westus.contoso.com",
          "priority": 2,
          "weight": 100
        }
      ]
    }
  ]
}
```

Practical guidance:

- Use a dedicated `/health` endpoint that checks critical dependencies.
- Return failure when the app cannot serve real traffic, not just when the process is alive.
- Keep health probes lightweight to avoid self-inflicted load.
- Use priority-based routing for active-passive failover.
- Use weighted routing for gradual traffic shifts and canary releases.

Front Door Standard/Premium vs Classic:

- **Standard/Premium**
  - Current recommended offerings.
  - Better alignment with modern Azure networking and security features.
  - Premium adds stronger private connectivity options and advanced security scenarios.
  - Uses origin groups, routes, and modern policy constructs.

- **Classic**
  - Older generation.
  - Feature set and management model differ from Standard/Premium.
  - Migration may require redesigning routes, domains, and security policies.
  - Do not assume a classic configuration maps one-to-one into Standard/Premium.

For exam purposes, remember that Front Door Standard/Premium is the strategic choice for new global web architectures, while Classic is mainly relevant for legacy estates and migration planning.

> Front Door and Traffic Manager are primarily for HTTP/HTTPS or DNS-based routing. They are not general-purpose solutions for arbitrary TCP/UDP application protocols.

**Azure Front Door Premium private origins**

Front Door Premium is relevant when you need global edge routing but do not want your origin publicly exposed.

Use it when:

- The backend must stay private behind a private endpoint.
- You need to reduce public attack surface.
- You want global entry with private connectivity to regional services.

Common patterns include:

- Front Door Premium to private App Service or private AKS ingress through private link.
- Front Door Premium to Application Gateway in a private network.
- Front Door as the public edge, with private regional origins behind it.

This is especially useful when security requirements demand that only the edge service is internet-facing.

**Azure Traffic Manager routing methods**

Traffic Manager is DNS-based, so it is useful when you want global name resolution rather than proxy-based traffic handling.

Routing methods:

| Method | Best for | Notes |
|---|---|---|
| Performance | Lowest-latency endpoint selection | Uses DNS resolution and network proximity, not live packet latency |
| Priority | Active-passive failover | Sends all traffic to the highest-priority healthy endpoint |
| Weighted | Gradual traffic split | Useful for testing, canary, or uneven capacity allocation |
| Geographic | Region-based routing | Directs users based on their DNS-resolved geography |
| Subnet | Client IP-based routing | Useful for custom network segmentation or internal routing scenarios |

Important DNS behavior:

- Traffic Manager does not see every client directly; it responds to recursive resolvers.
- Resolver caching can delay failover.
- Negative caching can also delay recovery when a name temporarily fails.
- Different clients may experience different failover times depending on their DNS cache TTL and resolver behavior.

This means DNS-based failover is not deterministic at the same speed for every user. For strict failover timing, a proxy-based global entry point such as Front Door is often better for web traffic.

**DNS limitations and client behavior**

DNS-based global routing is simple, but it has important limitations.

- **Caching** can keep clients on an old endpoint after failover.
- **Recursive resolvers** may serve cached answers to many users.
- **TTL** (time to live) influences how quickly changes propagate, but not every resolver honors it equally.
- **Negative caching** can delay recovery after temporary lookup failures.
- **Client libraries** may cache DNS independently of the OS or resolver.

Practical implications:

- Do not promise instant global cutover with DNS alone.
- Use low TTL values only when you accept the extra DNS query load.
- Test failover from multiple networks and resolvers.
- Prefer health-aware proxy routing when you need faster and more consistent failover for web apps.

**Non-HTTP and custom protocol scenarios**

Front Door and Traffic Manager are not the right answer for every global workload.

Use other patterns when you need:

- **TCP/UDP** global distribution for custom protocols.
- **Stateful session affinity** at layer 4.
- **Private network routing** across regions.
- **Specialized appliances** or gateways.

Common approaches include:

- Traffic Manager for DNS-based selection of regional public IPs.
- Regional Azure Load Balancer behind a global DNS name.
- VPN/ExpressRoute-based designs for private cross-region connectivity.
- Application-level sharding or broker-based routing for custom protocols.

If the workload is not HTTP/HTTPS, ask whether DNS-based selection is enough or whether you need a protocol-aware global architecture.

**Azure SQL Database disaster recovery patterns**

Azure SQL Database supports multiple geo-recovery options, but they serve different purposes.

- **Active geo-replication**
  - Creates readable secondary databases in other regions.
  - Useful for read scaling and disaster recovery.
  - You manage failover more directly.
  - Good when you need one or more secondary replicas and application-controlled failover.

- **Auto-failover groups**
  - Higher-level recommended pattern for managed failover.
  - Groups one or more databases and provides a listener endpoint.
  - Supports automatic failover and simplifies application connection management.
  - Often the preferred choice for production DR because it abstracts failover coordination.

- **Read-only routing**
  - Lets applications send read-only traffic to secondary replicas.
  - Useful for offloading reporting or analytics reads.
  - Requires the app to use the appropriate connection intent or listener behavior.

Key exam distinction:

- Use **auto-failover groups** when you want managed regional failover for a production database estate.
- Use **active geo-replication** when you need more direct control, multiple secondaries, or specific topology choices.

Failover behavior considerations:

- Failover changes the primary region.
- Connection strings should use the failover group listener when possible.
- Applications should retry transient failures.
- Write operations during failover may fail briefly and must be retried safely.

Example connection string pattern:

```text
Server=tcp:myfailovergroup.database.windows.net,1433;
Database=appdb;
User ID=appuser;
Password=***;
Encrypt=True;
TrustServerCertificate=False;
Connection Timeout=30;
```

**Azure SQL design guidance**

Choose the SQL pattern based on workload needs:

- Use **auto-failover groups** for most mission-critical OLTP systems.
- Use **active geo-replication** when you need custom failover control or multiple readable secondaries.
- Use **read scale-out** for read-heavy workloads within a region when geo-failover is not the main goal.
- Use **sharding** when a single database cannot scale or when regional data partitioning is required.

Be careful with consistency expectations:

- A secondary replica is not a separate system of record.
- Replication lag means reads from secondary can be slightly stale.
- Applications must tolerate eventual consistency for read-only replicas.

**Azure Storage geo-redundancy**

Azure Storage offers several geo-redundant options, and the differences matter.

- **LRS** (Locally Redundant Storage)
  - Replicates within a single datacenter.
  - Protects against local hardware failure, not region loss.

- **ZRS** (Zone-Redundant Storage)
  - Replicates across availability zones in a region.
  - Protects against zone failure.

- **GRS** (Geo-Redundant Storage)
  - Replicates synchronously within the primary region and asynchronously to a paired secondary region.
  - Secondary is not normally readable.

- **RA-GRS** (Read-Access Geo-Redundant Storage)
  - Same as GRS, but the secondary is readable.
  - Useful for read-only access during regional issues or for testing secondary availability.

- **GZRS** (Geo-Zone-Redundant Storage)
  - Combines zone redundancy in the primary region with geo-replication to a secondary region.
  - Stronger resilience than plain GRS because the primary region also spans zones.

- **RA-GZRS** (Read-Access Geo-Zone-Redundant Storage)
  - Same as GZRS, but the secondary is readable.
  - Useful when you want both zone resilience and read access to the secondary.

When to use read access to the secondary:

- To serve read-only workloads during a regional outage.
- To validate replication and disaster recovery readiness.
- To reduce pressure on the primary during specific read scenarios.

Important caveat:

> Storage replication protects the storage account, not the application. Your app still needs a failover plan for compute, secrets, and dependencies.

**Azure Cache for Redis in global designs**

Azure Cache for Redis is a cache, not a system of record. Its role in global architecture is to improve performance and reduce load, not to store authoritative business data.

Design considerations:

- Redis availability and replication options vary by offering and SKU.
- Do not assume every tier supports the same geo-replication behavior.
- Use Redis for ephemeral data, session state, and cache acceleration.
- Avoid using Redis as the only source of truth for critical data.

Exam guidance:

- If the workload needs durable multi-region state, choose a database or storage service designed for replication.
- If the workload needs fast local reads and can tolerate cache loss, Redis is appropriate.
- If session state must survive regional failover, verify the specific Redis tier and architecture before relying on it.

**Azure Site Recovery for IaaS workloads**

Azure Site Recovery (ASR) is a key disaster recovery pattern for virtual machines and other infrastructure-as-a-service workloads.

Use ASR when:

- You need to replicate VMs to another Azure region.
- The workload cannot be easily refactored into PaaS.
- You need orchestrated failover for servers, not just data.

ASR capabilities to remember:

- Replicates VM disks to a target region.
- Supports failover and test failover.
- Helps orchestrate recovery plans for multi-tier applications.
- Is often paired with Azure Backup for data protection.

ASR is especially relevant when the app includes legacy components, custom agents, or OS-level dependencies that are hard to rebuild from code.

**Multi-region deployment orchestration**

Global distribution is not only about traffic and data. You also need consistent deployment across regions.

Key practices:

- Use infrastructure as code (IaC) such as Bicep, ARM templates, Terraform, or similar tooling.
- Keep app settings, secrets, and identities aligned across regions.
- Store configuration in centralized services where appropriate, such as Azure App Configuration and Key Vault.
- Use deployment pipelines that stamp out the same baseline into each region.
- Version your infrastructure so failover regions are not drifted from primary regions.

Practical pattern:

```bash
az deployment group create \
  --resource-group rg-eastus \
  --template-file main.bicep \
  --parameters environment=prod region=eastus
```

Then deploy the same template to the paired region with region-specific parameters.

Useful controls:

- Use managed identities instead of hard-coded secrets.
- Replicate Key Vault secrets or use region-specific vaults with controlled synchronization.
- Keep connection strings and endpoint names abstracted through configuration.
- Validate that both regions can deploy independently.

**Consistency models and conflict handling**

Distributed systems force you to choose between stronger consistency and lower latency.

Common consistency trade-offs:

- **Strong consistency**
  - Users see the latest committed value.
  - Usually increases latency and coordination cost.
  - Harder to achieve across regions.

- **Eventual consistency**
  - Updates propagate asynchronously.
  - Lower latency and better availability.
  - Requires conflict handling and user experience design.

- **Session consistency**
  - A user sees their own writes in a session.
  - Often a practical compromise for globally distributed apps.

- **Bounded staleness**
  - Limits how far behind replicas can be.
  - Useful when you want predictable lag.

Conflict handling patterns:

- Last write wins.
- Version vectors or timestamps.
- Application-level merge logic.
- Partitioning writes by user, tenant, or region to avoid conflicts.

Exam insight:

- Active-active designs often need conflict resolution.
- Active-passive designs reduce conflict risk but increase failover dependency on promotion and synchronization.
- The more regions that can write, the more important your consistency model becomes.

**Global load balancing patterns for non-web workloads**

For workloads that are not pure web apps, you often combine services.

Examples:

- **Traffic Manager + regional Load Balancer**
  - DNS directs clients to the nearest healthy region.
  - Regional Load Balancer distributes traffic inside the region.
  - Good for TCP/UDP services exposed through public IPs.

- **Traffic Manager + Application Gateway**
  - DNS selects a region.
  - Application Gateway handles regional HTTP routing and WAF.
  - Useful when you need regional L7 control.

- **Front Door + private regional services**
  - Front Door handles global entry.
  - Private origins keep backends off the public internet.
  - Good for secure internet-facing apps with private compute.

- **ASR + DNS failover**
  - ASR recovers VMs.
  - DNS or Front Door/Traffic Manager shifts users to the recovered region.
  - Common for legacy workloads.

**Testing failover and recovery**

You should test failover before an incident, not during one.

A practical test sequence for active-passive web apps:

1. Verify health probes return healthy on the primary region.
2. Confirm the secondary origin or region is deployed and reachable.
3. Simulate a primary failure by disabling the origin or blocking the health endpoint.
4. Observe routing changes in Front Door or Traffic Manager.
5. Validate application behavior, including session handling and database connectivity.
6. Restore the primary and confirm traffic returns according to policy.
7. Update the runbook with observed timings and issues.

Example validation commands:

```bash
# Check DNS resolution
nslookup app.contoso.com

# Test HTTP health endpoint
curl -I https://app.contoso.com/health

# Inspect Front Door endpoint behavior
curl -I https://app.contoso.com
```

For SQL failover groups, test:

- Application reconnect behavior.
- Read-only routing after failover.
- Listener endpoint resolution.
- Retry logic for transient errors.

For ASR, test:

- Recovery plan execution.
- Boot order of dependent tiers.
- Network mapping.
- Post-failover application validation.

**Latency, throughput, and cost trade-offs**

Global distribution is always a trade-off.

- **Active-active**
  - Lower latency for global users.
  - Better resilience to regional outages.
  - Higher cost because you run more capacity.
  - More complex data synchronization and conflict handling.

- **Active-passive**
  - Lower cost and simpler operations.
  - Higher failover latency.
  - Potentially lower utilization of standby resources.
  - Easier data consistency management.

- **Proxy-based routing**
  - Better control and faster failover for web traffic.
  - Adds service cost and architectural dependency.
  - Usually preferred for HTTP/HTTPS.

- **DNS-based routing**
  - Simpler and often cheaper.
  - Slower and less predictable failover.
  - Better for broad endpoint selection than instant cutover.

A good design matches the pattern to the business requirement rather than defaulting to the most complex option.

**Common exam decision points**

When you see a scenario, ask these questions:

- Is the workload HTTP/HTTPS?
  - If yes, Front Door is often the best global entry point.
  - If no, consider Traffic Manager, regional load balancers, or protocol-specific patterns.

- Do you need instant or near-instant failover?
  - Prefer proxy-based routing and health probes.
  - DNS-based routing may be too slow or inconsistent.

- Do you need private backend connectivity?
  - Consider Front Door Premium with private origins.

- Is the workload stateful with concurrent writes?
  - Plan for consistency, replication lag, and conflict resolution.
  - Active-active may be difficult without application changes.

- Is the workload IaaS-based?
  - Azure Site Recovery is often the right DR pattern.

- Do you need read access to secondary storage?
  - Consider RA-GRS or RA-GZRS.

**Quick comparison**

| Scenario | Best-fit pattern |
|---|---|
| Global web app with fast failover | Azure Front Door |
| DNS-based failover for multiple endpoints | Traffic Manager |
| Regional TCP/UDP load balancing | Azure Load Balancer |
| Managed SQL regional failover | Auto-failover groups |
| Readable SQL secondary for DR | Active geo-replication or failover groups with read-only routing |
| VM replication to another region | Azure Site Recovery |
| Storage with geo-resiliency and read access | RA-GRS or RA-GZRS |

**Practical architecture examples**

A common active-active web architecture:

- Front Door routes users to the nearest healthy region.
- Each region hosts App Service or AKS.
- Azure SQL uses a failover group for DR.
- Azure Cache for Redis stores ephemeral session or cache data.
- Key Vault and App Configuration are deployed per region or synchronized centrally.
- Health probes validate the full request path.

A common active-passive enterprise architecture:

- Traffic Manager uses priority routing.
- Primary region hosts the live app.
- Secondary region stays warm with scaled-down capacity.
- SQL uses auto-failover groups.
- Storage uses GZRS or RA-GZRS where appropriate.
- ASR protects legacy VMs.

**Key takeaways**

- Use **Front Door** for global HTTP/HTTPS applications that need proxy-based routing, health-aware failover, and edge features.
- Use **Traffic Manager** when DNS-based global routing is sufficient or when you need routing methods like priority, weighted, geographic, or subnet.
- Use **auto-failover groups** as the primary Azure SQL DR pattern for managed failover.
- Use **GRS, RA-GRS, GZRS, or RA-GZRS** based on the required resilience and read-access needs.
- Use **Azure Site Recovery** for VM and IaaS disaster recovery.
- Design for **consistency**, not just availability, because multi-region writes can create conflicts.
- Test failover, DNS behavior, and runbooks regularly so your design works under real conditions.
