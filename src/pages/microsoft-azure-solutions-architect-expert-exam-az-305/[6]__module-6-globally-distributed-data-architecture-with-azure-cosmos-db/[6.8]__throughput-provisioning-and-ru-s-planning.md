## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Throughput and RU/s fundamentals**

Azure Cosmos DB uses <abbr title="Request Units per second">RU/s</abbr> to abstract the cost of database operations. You plan capacity by estimating the read, write, query, and transactional workload, then mapping that demand to provisioned throughput or autoscale throughput.

RU/s is not a fixed measure of CPU or IOPS. It is a service-specific abstraction that lets you compare operations consistently within Cosmos DB. The same operation can consume different RU charges depending on item size, indexing policy, consistency level, query shape, and partitioning.

- **RU** is the unit of request cost.
- **RU/s** is the throughput budget you reserve or allow to scale.
- **Provisioned throughput** gives predictable capacity.
- **Autoscale throughput** adjusts within a defined range based on demand.

**Provisioned vs autoscale throughput**

Cosmos DB supports manual provisioned throughput and autoscale throughput, but support depends on the API and resource type. In exam scenarios, always confirm whether throughput is configured at the container, database, or another resource scope for the chosen API.

Manual provisioned throughput is best when workload demand is stable and you want the lowest predictable cost for a known baseline. Autoscale is better when traffic is variable, bursty, or hard to predict.

| Mode | Best for | Cost behavior | Operational note |
|---|---|---|---|
| Manual provisioned | Steady workloads | Pay for fixed RU/s | Good when you know the baseline |
| Autoscale | Spiky or uncertain workloads | Pay based on max RU/s setting and usage pattern | Scales automatically within a bounded range |

Autoscale is commonly described as scaling between 10% and 100% of the configured maximum RU/s, but do not treat that as a universal floor in every context. The effective minimum depends on the resource type, provisioning model, and service constraints. For exam prep, remember the practical rule: you set a maximum, Cosmos DB scales within that envelope, and billing is tied to the highest RU/s reached in each hour, not just the average usage.

> <mark>Important:</mark> Autoscale does not mean “pay only for what you use in the moment.” You are billed against the autoscale max setting and the peak usage reached in the hour, so cost planning must consider burst patterns and sustained peaks.

**How Cosmos DB throughput is consumed**

Throughput is consumed by reads, writes, queries, stored procedures, transactional batch operations, and maintenance-related activity such as indexing. The exact RU charge depends on the operation shape.

- **Reads** are usually cheaper for small point reads on a well-partitioned container.
- **Writes** cost more when documents are larger or indexing is heavier.
- **Queries** can vary widely based on filters, joins, cross-partition scans, and result size.
- **Transactional batch** and stored procedures consume RU within a single logical partition scope.

A useful mental model is that RU cost rises when Cosmos DB must do more work:
- scan more items,
- touch more partitions,
- index more paths,
- move more bytes,
- or enforce stronger transactional guarantees.

**Partitioning and throughput planning**

Partitioning is central to RU/s planning because throughput is distributed across physical partitions, while data is grouped by a logical partition key. A good partition key spreads load evenly and avoids hot partitions.

A single logical partition has its own storage and throughput constraints. Even if the container has plenty of total RU/s, one hot logical partition can become the bottleneck if too many requests target the same partition key value.

- **Logical partition** groups items by partition key value.
- **Physical partition** is the internal storage and throughput unit managed by Cosmos DB.
- **Hot partition** occurs when one key value receives disproportionate traffic.

Design your partition key to:
- distribute writes and reads evenly,
- support common query patterns,
- avoid unbounded growth for one key value,
- and keep transactional operations within a single logical partition when needed.

Practical partition key guidance:
- Choose a key with high cardinality.
- Avoid monotonically increasing keys when they create write hotspots.
- Prefer keys that align with access patterns, not just uniqueness.
- Validate that the largest logical partition will not exceed service limits over time.

> A partition key that looks good for uniqueness can still be poor for throughput if most traffic lands on one value.

**Estimating RU/s for a workload**

Estimate throughput by breaking the workload into operation types and measuring the RU cost of each operation under realistic conditions. Then multiply by expected request rate and add headroom.

A simple planning formula is:

```text
Estimated RU/s = (reads per second × RU per read)
               + (writes per second × RU per write)
               + (queries per second × RU per query)
               + headroom
```

Use headroom for:
- traffic spikes,
- indexing overhead,
- uneven partition distribution,
- failover or regional changes,
- and growth between capacity reviews.

Practical estimation workflow:
1. Identify the top request patterns.
2. Measure RU charge for each pattern with representative data.
3. Multiply by expected peak request rate.
4. Add safety margin for growth and variability.
5. Validate with load testing before production.

**Measuring RU charge in practice**

You should not guess RU/s from theory alone. Use diagnostics and load tests to validate your estimate.

Common ways to measure:
- Azure portal metrics and capacity views.
- Query metrics and diagnostics in SDK responses.
- Load testing with representative data and request patterns.
- Application telemetry that records request charge and latency.

Example .NET SDK pattern for capturing request charge:

```csharp
var response = await container.ReadItemAsync<MyItem>(
    id: "item1",
    partitionKey: new PartitionKey("pk1"));

Console.WriteLine($"RU charge: {response.RequestCharge}");
```

Example JavaScript SDK pattern:

```javascript
const { resource, requestCharge } = await container.item("item1", "pk1").read();
console.log(`RU charge: ${requestCharge}`);
```

For queries, inspect diagnostics and request charge together. A query that looks simple can still be expensive if it scans many items or crosses partitions.

**Query design and RU efficiency**

Query shape strongly affects RU consumption. Point reads are usually the most efficient access pattern. Queries that can use the index and target a single partition are generally cheaper than cross-partition scans.

- **Point read** is the preferred pattern when you know both `id` and partition key.
- **Sargable query** can use indexes effectively.
- **Cross-partition query** often costs more because it fans out across partitions.
- **Function-based predicates** may reduce index efficiency depending on the API and indexing policy.

Example of a query that can be efficient when the indexing policy supports the predicate and the partition key is targeted:

```sql
SELECT * FROM c
WHERE c.partitionKey = @pk
  AND STARTSWITH(c.orderNumber, @prefix)
```

This is not automatically cheap just because it uses `STARTSWITH`. Efficiency depends on whether the query can be served from the index for the specific API, data shape, and indexing policy.

Practical query guidance:
- Prefer point reads over queries when possible.
- Filter on the partition key early.
- Avoid broad scans over large containers.
- Test query RU charge with production-like data.
- Review indexing policy before assuming a query is efficient.

**Indexing policy impact**

Indexing improves query performance but increases write cost and storage overhead. A broad indexing policy can make writes more expensive because Cosmos DB must maintain more index paths.

Use indexing policy tuning when:
- write throughput is high,
- many properties are never queried,
- or you need to reduce RU cost for write-heavy workloads.

Typical tradeoff:
- More indexing paths: better query flexibility, higher write RU.
- Fewer indexing paths: lower write RU, more careful query design required.

Example indexing policy concept:

```json
{
  "indexingMode": "consistent",
  "automatic": true,
  "includedPaths": [
    { "path": "/*" }
  ],
  "excludedPaths": [
    { "path": "/largeBlob/*" }
  ]
}
```

Exclude large or rarely queried fields when they do not need to be indexed.

**Autoscale planning details**

Autoscale is useful when demand changes throughout the day. You configure a maximum RU/s, and Cosmos DB scales the effective throughput within the allowed range. Billing is based on the highest RU/s reached in the hour, so a short spike can influence cost.

Plan autoscale when:
- traffic has sharp peaks,
- you need protection from sudden bursts,
- or you want to reduce manual capacity tuning.

Plan manual throughput when:
- workload is steady,
- you can predict peak demand accurately,
- and you want simpler cost control.

Autoscale planning tips:
- Size the max RU/s for the expected peak, not the average.
- Include growth headroom.
- Validate whether the workload has short spikes or sustained peaks.
- Compare monthly cost against manual throughput using realistic traffic traces.

**Capacity planning and cost reasoning**

A good RU/s estimate balances performance and cost. Underprovisioning causes throttling, while overprovisioning wastes budget.

Signs of underprovisioning:
- HTTP 429 throttling responses.
- Increased retry counts.
- Higher latency during peak periods.
- Requests waiting for available RU/s.

Signs of overprovisioning:
- Low utilization for long periods.
- Large gap between provisioned RU/s and actual demand.
- Stable latency with excess unused capacity.

Use a review cycle:
- monitor actual RU consumption,
- compare peak and average usage,
- adjust throughput after workload changes,
- and revisit partitioning if one key dominates traffic.

**Per-item and transaction limits**

RU planning is not only about total throughput. Item size, batch size, and transaction scope also matter.

Important constraints to remember:
- Large documents cost more RU to read and write.
- Transactional operations are scoped to a single logical partition.
- Batch operations can be efficient, but only within the same partition key.
- Very large or complex transactions can increase RU consumption and latency.

Practical implications:
- Split oversized documents when only part of the data changes frequently.
- Keep transactional batches small and partition-local.
- Avoid designs that require cross-partition transactions.
- Test the largest expected item size, not just the average.

**Multi-region throughput planning**

In multi-region deployments, throughput planning must account for read and write regions, failover behavior, and the chosen consistency model. More regions can improve availability and read locality, but they also change how you think about capacity and failover.

For planning:
- Size write-region capacity for the primary write workload.
- Ensure read regions can absorb local read traffic.
- Consider failover capacity so a secondary region can take over if needed.
- Validate latency and RU behavior under the selected consistency level.

Example topology considerations:
- Single write region with multiple read regions for global reads.
- Multi-write regions when the application requires active-active writes.
- Failover testing to confirm the surviving region can handle the shifted load.

A practical deployment checklist:
1. Identify the primary write region.
2. Map read traffic to the nearest region.
3. Estimate failover load if the write region fails.
4. Confirm the target region has enough RU/s headroom.
5. Test failover and observe throttling, latency, and retry behavior.

**Practical sizing examples**

If a point read costs 1 RU and you expect 2,000 reads per second, the read portion alone is about 2,000 RU/s. If writes cost 5 RU each and you expect 200 writes per second, that adds about 1,000 RU/s. Then add headroom for spikes and uneven partitioning.

```text
Reads: 2,000 × 1 RU = 2,000 RU/s
Writes: 200 × 5 RU = 1,000 RU/s
Subtotal = 3,000 RU/s
Headroom at 30% = 900 RU/s
Estimated target = 3,900 RU/s
```

This is only a starting point. Validate with real data, because RU charges can change with document size, indexing, and query shape.

**Common exam traps**

- Assuming autoscale always means a fixed 10% minimum in every case.
- Treating RU/s as a direct measure of CPU or storage.
- Ignoring partition key design when estimating throughput.
- Using average traffic instead of peak traffic for sizing.
- Forgetting that query shape and indexing policy can dominate RU cost.
- Assuming a single logical partition can absorb unlimited traffic.
- Planning multi-region capacity without failover headroom.

**How to answer design questions**

When the exam asks you to choose a throughput model, think in this order:
1. Workload pattern: steady or bursty.
2. Partitioning: can traffic spread evenly?
3. Query shape: point reads or scans?
4. Cost sensitivity: fixed budget or elastic budget?
5. Availability: single region or multi-region failover?

If the workload is predictable and steady, manual throughput is often the simplest fit. If the workload spikes or is uncertain, autoscale is usually safer. If the design has a hot partition or poor query pattern, fix the data model before increasing RU/s.

**Key takeaways**

- RU/s planning starts with workload measurement, not guesswork.
- Partition key design is as important as raw throughput.
- Autoscale helps with bursts, but billing still follows peak usage in the hour.
- Query shape and indexing policy can change RU cost dramatically.
- Multi-region designs need failover-aware capacity planning.
- Validate estimates with diagnostics, metrics, and load testing before production.
