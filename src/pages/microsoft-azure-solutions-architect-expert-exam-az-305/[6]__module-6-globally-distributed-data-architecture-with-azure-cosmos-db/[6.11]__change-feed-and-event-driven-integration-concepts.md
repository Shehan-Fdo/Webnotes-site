## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Event-driven integration overview**

Event-driven integration lets systems react to changes instead of polling for them. In Azure, this usually means one service emits an event or message, and one or more consumers process it asynchronously. This improves decoupling, elasticity, and resilience when compared with synchronous request/response designs.

A useful way to think about the main Azure options is:

- **Azure Event Grid** for reactive notification and routing of discrete events.
- **Azure Service Bus** for reliable brokered messaging and enterprise workflows.
- **Azure Event Hubs** for high-throughput event ingestion and stream processing.
- **Azure Cosmos DB change feed** for reacting to data changes in a database.

The right choice depends on whether you need notification, command delivery, durable workflow, or stream analytics.

**Core integration patterns**

Event-driven systems often use a few recurring patterns:

- **Publish/subscribe**: producers publish events, and multiple subscribers react independently.
- **Queue-based load leveling**: producers send work to a queue so consumers can scale at their own pace.
- **Event sourcing**: the system stores state changes as an ordered sequence of events.
- **CQRS** (<abbr title="Command Query Responsibility Segregation">CQRS</abbr>): commands update the write model, while queries read from a separate optimized model.
- **Outbox pattern**: the application writes business data and an integration event in the same transaction, then a worker publishes the event later.
- **Inbox pattern**: the consumer stores processed message IDs to avoid duplicate side effects.

These patterns matter in AZ-305 because architecture questions often focus on reliability, replay, ordering, and eventual consistency rather than just service names.

**Azure Event Grid**

Event Grid is a routing service for discrete notifications. It is designed for lightweight event distribution, not for durable command queues or large event streams. You use it when you want to react to something that happened, such as a blob created, a resource changed, or a custom business event.

Key concepts:

- **Event source**: the service or application that emits events.
- **Topic**: the endpoint where events are published.
- **Subscription**: the routing rule that sends matching events to a handler.
- **Event handler**: the destination that receives the event, such as an Azure Function, Logic App, webhook, or Service Bus queue/topic.
- **Dead-letter destination**: storage used when delivery repeatedly fails.
- **Retry policy**: controls how Event Grid retries delivery before giving up.
- **Time to live (TTL)**: the maximum time Event Grid keeps retrying delivery.

Event Grid supports both built-in and custom routing models:

- **System topics** are used for Azure service events, such as Storage or Resource Group events.
- **Custom topics** are used when your application publishes its own events.
- **Domains** are useful when you need to manage many related custom topics with shared access and governance.

Use Event Grid when:

- You need near-real-time notification.
- You want fan-out to multiple handlers.
- You need simple event routing with low operational overhead.
- You want to trigger serverless workflows.

Avoid Event Grid when:

- You need guaranteed ordered processing of a long stream.
- You need message locking, deferral, or sessions.
- You need a durable work queue with explicit settlement semantics.

A typical custom topic flow looks like this:

```bash
az eventgrid topic create \
  --name orders-topic \
  --resource-group rg-app \
  --location eastus

az eventgrid event-subscription create \
  --name orders-to-func \
  --source-resource-id /subscriptions/<subId>/resourceGroups/rg-app/providers/Microsoft.EventGrid/topics/orders-topic \
  --endpoint-type azurefunction \
  --endpoint /subscriptions/<subId>/resourceGroups/rg-app/providers/Microsoft.Web/sites/orders-func/functions/ProcessOrder
```

A dead-letter destination is important when delivery must be auditable. If the handler is unavailable or returns repeated failures, Event Grid can place undelivered events into a storage account container for later inspection and replay.

**Event Grid delivery behavior**

Event Grid uses push delivery to subscribers, but the exam distinction is broader than “push versus pull.” The important point is that Event Grid actively delivers notifications to handlers, while brokered messaging services store messages for consumers to receive later.

Practical implications:

- Delivery is **at least once**, so handlers must be idempotent.
- Duplicate delivery can happen, so consumers should tolerate reprocessing.
- Retry behavior helps transient failures, but it does not replace durable business logic.
- Dead-lettering is for failed delivery, not for normal queue processing.

**Event Grid topics, system topics, and domains**

Use the right publishing model:

- **Custom topic**: best when your application emits its own domain events.
- **System topic**: best when you want Azure resource events without building a custom publisher.
- **Domain**: best when you need to manage many custom topics with common access control and routing.

Architecture questions often test whether you understand that system topics are tied to Azure resources, while custom topics are application-owned. Domains help at scale when many teams or tenants publish related events.

**Azure Service Bus**

Service Bus is a brokered messaging platform for reliable enterprise communication. It is designed for commands, workflows, and decoupled application components that need durable delivery and controlled processing.

Core entities:

- **Queues** for point-to-point messaging.
- **Topics and subscriptions** for publish/subscribe fan-out.
- **Sessions** for ordered, correlated processing.
- **Dead-letter queue** (<abbr title="Dead-Letter Queue">DLQ</abbr>) for messages that cannot be processed successfully.

Service Bus is a better fit than Event Grid when you need:

- Guaranteed message storage until a consumer receives it.
- Explicit message settlement.
- Locking and controlled concurrency.
- FIFO-like processing with sessions.
- Deferral for delayed handling of specific messages.

Important Service Bus concepts:

- **Peek-lock**: the consumer receives a message, locks it, processes it, then completes or abandons it.
- **Complete**: removes the message after successful processing.
- **Abandon**: releases the lock so the message can be retried.
- **Dead-letter**: moves an unprocessable message to the DLQ.
- **Defer**: keeps the message in the queue but hides it until it is retrieved by sequence number.
- **Lock renewal**: extends the lock when processing takes longer than the default lock duration.

These features are often central in AZ-305 scenarios because they support reliable business workflows and poison-message handling.

A typical queue consumer flow in code looks like this:

```csharp
var client = new ServiceBusClient(connectionString);
var processor = client.CreateProcessor("orders", new ServiceBusProcessorOptions
{
    AutoCompleteMessages = false,
    MaxConcurrentCalls = 4
});

processor.ProcessMessageAsync += async args =>
{
    try
    {
        // Process message
        await args.CompleteMessageAsync(args.Message);
    }
    catch
    {
        await args.AbandonMessageAsync(args.Message);
    }
};

processor.ProcessErrorAsync += args =>
{
    Console.WriteLine(args.Exception.ToString());
    return Task.CompletedTask;
};

await processor.StartProcessingAsync();
```

Service Bus topics are useful when multiple downstream systems need the same business event, but each subscriber may apply different filters or processing logic. This is different from Event Grid, which is optimized for event notification and routing rather than durable workflow orchestration.

**Service Bus settlement and advanced behaviors**

Service Bus supports more control than simple event delivery:

- **Peek-lock settlement** gives you explicit success or failure handling.
- **Message deferral** lets you postpone processing until prerequisites are met.
- **Scheduled messages** let you enqueue work for future delivery.
- **Sessions** preserve ordering within a session key and support correlated workflows.
- **Duplicate detection** helps prevent repeated processing when the sender retries.

Use sessions when:

- You need ordered processing per customer, order, or conversation.
- You want to process related messages on the same consumer instance.
- You need a stateful workflow with correlation.

Use deferral when:

- A message arrives before dependent data is ready.
- You need to temporarily skip a message without losing it.

**Azure Event Hubs**

Event Hubs is a big-data ingestion service for telemetry and streaming scenarios. It is optimized for high throughput, partitioned ingestion, and replayable event streams.

Use Event Hubs when:

- You ingest large volumes of telemetry, logs, or clickstream data.
- You need multiple independent consumers to read the same stream.
- You want retention-based replay for stream processing.
- You integrate with analytics engines such as Stream Analytics, Spark, or custom processors.

Key concepts:

- **Partition**: a shard of the event stream used for scale and ordering within that partition.
- **Consumer group**: a logical view of the stream that lets multiple applications read independently.
- **Retention**: how long events remain available for replay.
- **Checkpointing**: tracking progress so consumers can resume after failure.

Event Hubs is not a queue replacement. It is a stream ingestion platform. The exam distinction is important:

- **Service Bus** is for commands and work items.
- **Event Hubs** is for telemetry and event streams.
- **Event Grid** is for event notification and routing.

A simple producer example:

```csharp
var producer = new EventHubProducerClient(connectionString, "telemetry");
using EventDataBatch batch = await producer.CreateBatchAsync();

batch.TryAdd(new EventData(Encoding.UTF8.GetBytes("{\"deviceId\":\"d1\",\"temp\":72}")));
batch.TryAdd(new EventData(Encoding.UTF8.GetBytes("{\"deviceId\":\"d2\",\"temp\":75}")));

await producer.SendAsync(batch);
```

A consumer typically uses a processor that checkpoints progress:

```csharp
var processor = new EventProcessorClient(
    blobCheckpointStore,
    EventHubConsumerClient.DefaultConsumerGroupName,
    connectionString,
    "telemetry");

processor.ProcessEventAsync += async args =>
{
    Console.WriteLine(args.Data.EventBody.ToString());
    await args.UpdateCheckpointAsync();
};

processor.ProcessErrorAsync += args =>
{
    Console.WriteLine(args.Exception.Message);
    return Task.CompletedTask;
};

await processor.StartProcessingAsync();
```

**Event Hubs versus Service Bus**

| Capability | Service Bus | Event Hubs |
|---|---|---|
| Primary use | Reliable messaging and workflows | High-throughput streaming and telemetry |
| Message model | Discrete messages | Ordered event stream |
| Replay | Limited, queue-oriented | Native via retention window |
| Ordering | Sessions for ordered groups | Ordering within a partition |
| Consumer model | Receive and settle | Read, checkpoint, and replay |
| Best fit | Commands, tasks, business transactions | Logs, metrics, IoT, analytics |

Use Service Bus when the message itself is a unit of work. Use Event Hubs when the event is part of a stream that may be processed by multiple analytics or monitoring consumers.

**Azure Cosmos DB change feed**

The Cosmos DB change feed is a persistent record of changes to items in a container. In the default mode, it exposes inserts and updates. Delete support depends on the API and change feed mode, so do not assume deletes are always present.

The change feed is valuable because it lets you react to data changes without polling the database. It is commonly used for:

- Materialized views
- Cache updates
- Search indexing
- Data replication
- Integration event publishing

Important concepts:

- **Lease container**: stores checkpoint and ownership state for distributed processing.
- **Lease prefix**: separates multiple processors or applications using the same lease container.
- **Checkpoint**: records progress so processing can resume.
- **Start from beginning**: reads historical changes from the start of the feed.
- **Start from now**: processes only new changes after the processor starts.

The change feed is especially useful for event-driven architectures because it can support:

- **Event sourcing** when the database is the source of truth and changes are the event stream.
- **CQRS** when writes in Cosmos DB update read models asynchronously.
- **Outbox-like publishing** when a worker reads changes and emits integration events.
- **Inbox processing** when downstream consumers deduplicate and store processed change IDs.

A typical Azure Functions trigger configuration needs both the monitored container and lease container settings. Exact property names vary by extension version, but the binding usually includes the source container, lease container, and checkpoint-related options.

Example function binding configuration:

```json
{
  "bindings": [
    {
      "name": "input",
      "type": "cosmosDBTrigger",
      "direction": "in",
      "databaseName": "appdb",
      "containerName": "orders",
      "leaseDatabaseName": "appdb",
      "leaseContainerName": "leases",
      "createLeaseContainerIfNotExists": true,
      "startFromBeginning": true
    }
  ]
}
```

A corresponding `host.json` tuning example:

```json
{
  "version": "2.0",
  "extensions": {
    "cosmosDB": {
      "leaseOptions": {
        "leasePrefix": "orders-feed",
        "feedPollDelay": "00:00:05"
      },
      "maxItemsPerInvocation": 100
    }
  }
}
```

Use these settings carefully:

- **Lease container** must exist or be created by the runtime.
- **Lease prefix** helps isolate multiple processors.
- **Start from beginning** is useful for backfills and initial projections.
- **Start from now** is better for new consumers that should ignore history.
- **Batch size** and polling settings affect throughput and latency.

> The exact Cosmos DB trigger schema and supported options vary by Functions extension version and programming model. Validate the binding against the version you deploy.

**Cosmos DB change feed modes**

There are two important modes to understand:

- **Standard change feed**: commonly used for inserts and updates; delete events are not universally available.
- **Full-fidelity change feed**: provides a richer history and can include deletes and other change details depending on the API and configuration.

Exam guidance:

- Do not assume the default feed is a complete audit log.
- Use full-fidelity capabilities when you need richer lifecycle tracking.
- Use the change feed for downstream projections, not as a substitute for a transactional audit store unless the mode and design explicitly support that need.

**Choosing Azure Functions, Logic Apps, or custom workers**

The consumer choice matters as much as the event source.

- **Azure Functions**: best for code-first, event-driven, stateless processing with elastic scale.
- **Logic Apps**: best for low-code orchestration, SaaS integration, and workflow automation.
- **Custom workers**: best when you need specialized libraries, long-running processing, custom hosting, or advanced control.

Use Azure Functions when:

- You need quick event handlers.
- You want native bindings to Event Grid, Service Bus, Event Hubs, and Cosmos DB.
- You want serverless scaling with minimal infrastructure.

Use Logic Apps when:

- You need connectors to SaaS systems.
- You need approval workflows or business process orchestration.
- You want visual design and low-code maintenance.

Use custom workers when:

- You need fine-grained control over threading, batching, or memory.
- You need specialized protocol handling.
- You need to run in containers or on dedicated compute.

**Reliability, idempotency, and duplicate handling**

Most Azure eventing services deliver at least once. That means duplicates are possible. Your design should assume repeated delivery and make side effects safe.

Practical techniques:

- Use a unique business key to detect duplicates.
- Store processed message IDs in an inbox table.
- Make writes idempotent with upserts or conditional updates.
- Use optimistic concurrency where appropriate.
- Keep handlers small and deterministic.

A simple idempotency pattern:

```sql
CREATE TABLE ProcessedMessages (
    MessageId nvarchar(100) NOT NULL PRIMARY KEY,
    ProcessedAt datetime2 NOT NULL DEFAULT sysutcdatetime()
);
```

Your handler checks whether the message ID already exists before applying side effects. This is especially important for Event Grid and Service Bus retry scenarios.

**Dead-lettering and poison-message handling**

Dead-lettering is a safety mechanism for messages or events that cannot be processed successfully.

Use dead-lettering when:

- Validation fails permanently.
- A message is malformed.
- A dependency is missing and cannot be recovered.
- Retry would only repeat the same failure.

Service Bus has a built-in DLQ for queue and subscription entities. Event Grid supports dead-letter destinations for undeliverable events. Cosmos DB change feed does not use a DLQ in the same way; you usually handle failures in your processor and persist failed items separately.

Operationally, you should:

- Monitor dead-letter counts.
- Inspect failure reasons.
- Reprocess after fixing the root cause.
- Avoid infinite retry loops in application code.

**Architecture selection guidance**

Choose the service based on workload shape:

- **Event Grid** when you need event notification and fan-out.
- **Service Bus** when you need reliable business messaging and workflow control.
- **Event Hubs** when you need telemetry ingestion and stream analytics.
- **Cosmos DB change feed** when the database itself is the source of change events.

A quick decision rule:

- If the payload is a business command, use Service Bus.
- If the payload is a notification that something happened, use Event Grid.
- If the payload is a high-volume telemetry event, use Event Hubs.
- If the payload is a database change, use Cosmos DB change feed.

**Common exam scenarios**

- A system must notify multiple downstream handlers when a blob is uploaded: use Event Grid with subscriptions and dead-lettering.
- A payment workflow must not lose messages and must support retries and poison handling: use Service Bus.
- A telemetry platform must ingest millions of events per second and support replay: use Event Hubs.
- A read model must update whenever Cosmos DB data changes: use the change feed with a lease container and checkpointing.
- A business process needs ordered processing per customer: use Service Bus sessions.
- A reporting pipeline needs to rebuild projections from historical changes: use Cosmos DB change feed or Event Hubs retention, depending on the source.

**Practical design tips**

- Keep event payloads small and include references when data is large.
- Version your event contracts so consumers can evolve independently.
- Treat event schemas as public contracts.
- Use correlation IDs to trace a request across services.
- Separate transport concerns from business logic.
- Prefer asynchronous processing for long-running work.
- Test failure paths, not just success paths.

**Summary**

Azure event-driven integration is about matching the right service to the right workload. Event Grid routes notifications, Service Bus handles reliable messaging, Event Hubs ingests streams, and Cosmos DB change feed reacts to database changes. The strongest AZ-305 answers explain not only what each service does, but also why it fits the workload, how it handles retries and duplicates, and how consumers maintain state safely.
