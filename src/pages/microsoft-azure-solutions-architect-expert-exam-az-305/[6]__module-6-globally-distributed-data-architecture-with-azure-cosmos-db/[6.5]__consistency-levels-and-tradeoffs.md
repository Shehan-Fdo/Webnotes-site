## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Consistency in Azure: the architecture question**

Consistency describes how quickly and predictably one part of a distributed system sees changes made by another part. In Azure design, you usually balance three forces:

- **Correctness**: do readers see the latest committed data?
- **Availability**: can the system keep serving requests during failures?
- **Latency**: how fast do reads and writes complete, especially across regions?

For AZ-305, the key is not memorizing a single “best” model. You need to choose the right consistency behavior for the workload, data access pattern, and recovery goals.

**Core consistency concepts**

In distributed systems, replicas do not always update at the same instant. That creates a tradeoff between immediate agreement and resilience.

- **Strong consistency** means a read returns the most recent committed write, as if there were one authoritative copy.
- **Eventual consistency** means replicas converge over time if no new writes occur.
- **Bounded staleness** limits how far behind a read can be, either by time or by number of operations.
- **Session consistency** gives a client a coherent view within its own session when it preserves the session token.
- **Consistent prefix** ensures reads never see out-of-order writes; they may see a prefix of the write sequence.
- **Read-your-writes** means after you write data, you can read your own update back in the same session.

A useful mental model is this: the stronger the consistency guarantee, the more coordination the system usually needs. More coordination often increases latency and can reduce flexibility during regional failures.

**Azure Cosmos DB consistency levels**

Azure Cosmos DB is the main Azure service where consistency is an explicit design choice. It supports five levels:

- **Strong**
- **Bounded staleness**
- **Session**
- **Consistent prefix**
- **Eventual**

These are not just marketing labels. They affect what a client can observe after writes, how replicas are used, and what failure modes are acceptable.

**Strong consistency**

Strong consistency provides the most intuitive behavior: once a write is acknowledged, subsequent reads return that write.

Use strong consistency when:

- You need the simplest correctness model.
- You cannot tolerate stale reads for business decisions.
- You are willing to accept higher coordination cost and stricter topology limits.

Important design implications:

- Strong consistency is the most restrictive option for multi-region write/read designs.
- It is best suited to workloads where correctness matters more than read latency.
- It is often a poor fit for globally distributed, latency-sensitive user-facing apps.

Example scenario:

- A financial ledger service may require strong consistency for balance checks and transaction state.
- A shopping catalog usually does not need it because a short delay in product metadata is acceptable.

**Bounded staleness**

Bounded staleness allows reads to lag behind writes, but only within a defined bound. The bound is expressed by time or by the number of operations.

Use bounded staleness when:

- You want predictable freshness without full strong consistency.
- You can tolerate a small, controlled delay.
- You need a stronger guarantee than session or eventual consistency.

Why it matters:

- It is a middle ground for globally distributed applications.
- It gives you a measurable freshness window, which is useful for user experience and compliance discussions.
- It still imposes topology constraints compared with weaker models.

**Session consistency**

Session consistency is the default choice for many Cosmos DB applications because it gives a strong user experience without the cost of global strong consistency.

It guarantees, for a single client session that preserves the session token:

- **Read-your-writes**
- **Monotonic reads**
- **Monotonic writes**
- **Consistent prefix**

This means a user who updates a profile and immediately reloads it sees their own change, even if other clients may not see it yet.

Use session consistency when:

- Each user or request flow should see its own updates.
- You want good performance and broad deployment flexibility.
- You can carry the session token correctly through the app tier.

> If you lose the session token or fail to reuse it, you lose the session guarantee for that client flow.

Practical example:

- A ticketing app can use session consistency so the user sees their own seat selection immediately, while other users may see the update slightly later.

**Consistent prefix**

Consistent prefix ensures that reads never observe writes out of order. If writes A, B, and C happen in sequence, a reader may see A or A+B, but not B without A.

Use it when:

- Write ordering matters more than freshness.
- You can tolerate lag, but not reordering anomalies.
- You need a weaker model than session but safer than eventual consistency.

Typical use cases:

- Event streams
- Audit trails
- Ordered workflow state

**Eventual consistency**

Eventual consistency is the weakest model. It provides the highest flexibility and usually the lowest coordination overhead, but readers may temporarily see stale data.

Use eventual consistency when:

- The application can tolerate temporary divergence.
- High availability and low latency matter more than immediate agreement.
- The data is naturally tolerant of delay, such as feeds, telemetry, or cached content.

Examples:

- Social media timelines
- Product recommendations
- Analytics dashboards
- IoT telemetry ingestion

**Choosing a Cosmos DB consistency level**

A practical decision rule:

1. Start with the business question: “What is the cost of a stale read?”
2. If the answer is “unacceptable,” consider strong consistency.
3. If the answer is “small but bounded,” consider bounded staleness.
4. If the answer is “the same user must see their own changes,” choose session.
5. If ordering matters more than freshness, choose consistent prefix.
6. If the workload is tolerant of temporary divergence, choose eventual.

A simple comparison:

| Level | Freshness | Typical use | Main tradeoff |
|---|---:|---|---|
| Strong | Highest | Critical correctness | Highest coordination cost |
| Bounded staleness | High, bounded | Predictable freshness | Topology constraints |
| Session | Per-client coherent | User-facing apps | Requires session token handling |
| Consistent prefix | Ordered | Event/workflow data | Can still be stale |
| Eventual | Lowest | Tolerant workloads | Temporary inconsistency |

**Cosmos DB client behavior and session tokens**

Session consistency depends on the client preserving the session token returned by the service. In practice, SDKs handle this for you when you reuse the same client instance and follow normal patterns.

Example with the .NET SDK:

```csharp
using Microsoft.Azure.Cosmos;

var client = new CosmosClient(
    accountEndpoint: "<endpoint>",
    authKeyOrResourceToken: "<key>",
    new CosmosClientOptions
    {
        ConsistencyLevel = ConsistencyLevel.Session
    });

Container container = client.GetContainer("appdb", "orders");

ItemResponse<Order> createResponse = await container.CreateItemAsync(
    new Order { id = "1001", status = "Created" },
    new PartitionKey("1001"));

Order order = await container.ReadItemAsync<Order>(
    id: "1001",
    partitionKey: new PartitionKey("1001"));
```

Practical guidance:

- Reuse the same `CosmosClient` instance for the app lifetime.
- Avoid creating a new client per request.
- Preserve session context if you manually manage tokens across tiers.

**Inspecting Cosmos DB consistency**

You can inspect the configured consistency level through the Azure Portal, ARM templates, Bicep, or Azure CLI. Portal and ARM are the most reliable sources for the authoritative account configuration.

Example Azure CLI query:

```bash
az cosmosdb show \
  --resource-group rg-app \
  --name cosmos-prod \
  --query consistencyPolicy
```

The exact JSON shape can vary by CLI/API version, so treat the output as a configuration check rather than a fixed schema.

Example Bicep:

```bicep
resource cosmos 'Microsoft.DocumentDB/databaseAccounts@2024-05-15' = {
  name: 'cosmos-prod'
  location: 'eastus'
  kind: 'GlobalDocumentDB'
  properties: {
    consistencyPolicy: {
      defaultConsistencyLevel: 'Session'
      maxIntervalInSeconds: 5
      maxStalenessPrefix: 100
    }
    locations: [
      {
        locationName: 'eastus'
        failoverPriority: 0
        isZoneRedundant: true
      }
      {
        locationName: 'westus2'
        failoverPriority: 1
        isZoneRedundant: true
      }
    ]
  }
}
```

Notes on the template:

- `locationName` must be a valid Azure region name.
- `maxIntervalInSeconds` and `maxStalenessPrefix` are relevant for bounded staleness.
- `isZoneRedundant` applies to regional placement, not geo-replication semantics.

**Azure Storage replication versus consistency**

Azure Storage replication options are about durability and availability, not database-style read consistency.

Common replication options:

- **LRS** (Locally Redundant Storage): three copies within one datacenter.
- **ZRS** (Zone-Redundant Storage): copies across availability zones in one region.
- **GRS** (Geo-Redundant Storage): copies to a paired secondary region.
- **GZRS** (Geo-Zone-Redundant Storage): zone redundancy in the primary region plus geo-replication.
- **RA-GRS** / **RA-GZRS**: read access to the secondary region.

Key distinction:

- These options describe where data is replicated and whether a secondary can be read.
- They do not provide the same read/write consistency semantics as a database consistency model.
- ZRS is not a geo-redundant option; it stays within one region across zones.

When to use each:

- **LRS** for lower cost and when regional durability is sufficient.
- **ZRS** for higher resilience to zone failure within a region.
- **GRS/GZRS** for disaster recovery across regions.
- **RA-GRS/RA-GZRS** when read access to the secondary region helps with availability or read scaling.

A practical example:

- A static website or backup archive may use GZRS for resilience.
- A transactional app should not treat storage replication as a substitute for application-level consistency design.

**Azure SQL and consistency**

Azure SQL Database uses transactional consistency within the primary database. That means committed transactions are visible according to the database’s isolation semantics, and reads from the primary reflect current committed state.

Important nuance:

- This is not the same as saying every read in every SQL topology is instantly current.
- Read scale-out, geo-replication, and failover replicas can introduce lag.
- Secondary replicas are useful for availability and read offload, but they may not reflect the latest committed changes immediately.

Use Azure SQL when:

- You need relational transactions, constraints, and strong relational integrity.
- You want familiar SQL semantics and ACID behavior.
- You can design around replica lag for read-only secondary scenarios.

Example architecture choice:

- Use the primary for order entry and payment writes.
- Use a read replica or geo-secondary for reporting, knowing it may lag.

**Azure SQL design tradeoff example**

If a user places an order and immediately checks the order history:

- Reading from the primary gives the most current committed view.
- Reading from a geo-replicated secondary may show a short delay.
- If the business requires immediate confirmation, route the confirmation path to the primary or a strongly consistent transactional store.

**Consistency in other Azure services**

Not every Azure service exposes a named consistency menu, but consistency still matters in design.

Examples:

- **Azure Cache for Redis**: very fast, but it is a cache; treat it as a performance layer, not the source of truth.
- **Azure Service Bus**: messaging provides ordered delivery only in specific configurations, and duplicate handling may be needed.
- **Azure Event Hubs**: optimized for ingestion and streaming, not immediate cross-consumer consistency.
- **Azure Data Lake Storage**: suitable for analytics pipelines, where eventual visibility across processing stages is often acceptable.
- **Azure App Configuration**: useful for centralized configuration, but clients should still plan for refresh intervals and cache behavior.

The exam often tests whether you understand that the service’s purpose determines the consistency expectation. A cache, queue, stream, and transactional database all behave differently.

**Architecture decision patterns**

**Pattern 1: Global user-facing app**

Use session consistency in Cosmos DB when:

- Users should see their own updates immediately.
- You need low latency across regions.
- Occasional cross-user staleness is acceptable.

Why this works:

- It balances user experience and scale.
- It avoids the complexity of strong global coordination.

**Pattern 2: Financial or inventory-critical workflow**

Use strong consistency or a relational transactional store when:

- A stale read can cause overselling, double spending, or incorrect approvals.
- The workflow depends on immediate correctness.

Why this works:

- It reduces ambiguity in business logic.
- It supports deterministic decision-making.

**Pattern 3: Content distribution or telemetry**

Use eventual consistency when:

- Data is high volume and time-sensitive.
- Temporary inconsistency is acceptable.
- Availability and throughput matter more than immediate agreement.

Why this works:

- It scales well.
- It avoids unnecessary coordination for data that is naturally tolerant of delay.

**Pattern 4: Ordered event processing**

Use consistent prefix or session consistency when:

- Event order matters.
- You need to avoid seeing writes out of sequence.
- You can tolerate some lag.

Why this works:

- It preserves logical order without forcing full strong consistency.

**Common AZ-305 exam traps**

- Confusing **replication** with **consistency**.
  - Replication describes where copies live.
  - Consistency describes what readers can observe.
- Assuming **strong consistency** is always best.
  - It is often too restrictive for globally distributed apps.
- Treating **session consistency** as automatic without token handling.
  - The guarantee depends on preserving session context.
- Assuming **ZRS** is a geo-redundant read model.
  - It is zone-redundant within one region.
- Assuming **secondary replicas** always match the primary.
  - Replicas can lag depending on the service and topology.

**Practical design checklist**

1. Identify the data type: transactional, cached, analytical, or streaming.
2. Define the user-visible impact of stale reads.
3. Decide whether order matters more than freshness.
4. Determine whether the app is single-session aware or multi-client shared.
5. Check whether the service offers explicit consistency controls.
6. Validate region and failover requirements.
7. Choose the weakest consistency that still satisfies the business requirement.

**Quick decision guide**

- Choose **strong** when correctness is non-negotiable.
- Choose **bounded staleness** when freshness must be predictable.
- Choose **session** when a user should see their own writes.
- Choose **consistent prefix** when order matters.
- Choose **eventual** when scale and latency matter most.

**Summary**

Azure consistency design is about matching the data model to the workload. Cosmos DB gives you explicit consistency levels, Azure SQL gives you transactional relational semantics with replica considerations, and storage replication options focus on durability rather than read consistency. For AZ-305, the best answer is usually the one that aligns business impact, latency tolerance, and failure behavior with the service’s actual guarantees.
