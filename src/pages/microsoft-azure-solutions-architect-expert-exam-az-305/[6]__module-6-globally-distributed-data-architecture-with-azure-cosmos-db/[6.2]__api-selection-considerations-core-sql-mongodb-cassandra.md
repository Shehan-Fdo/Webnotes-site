## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**API selection in Azure Cosmos DB**

Azure Cosmos DB offers multiple APIs so you can match an application’s data model, query style, and migration path to the right service surface. For exam purposes, the key decision is not “which API is best overall,” but which API best fits the workload, the team’s existing skills, and the amount of redesign you can tolerate.

The main APIs covered here are:

- **Core (SQL)**, the native API for the Azure Cosmos DB for NoSQL data model
- **MongoDB**, for applications that already use MongoDB drivers, query patterns, or tooling
- **Cassandra**, for wide-column workloads that use CQL (Cassandra Query Language)

A good architecture decision weighs more than syntax compatibility. You also consider indexing behavior, consistency, throughput cost, SDK ecosystem, change feed support, and how much of the source database behavior Cosmos DB actually reproduces.

**Core (SQL) API: the native NoSQL choice**

Core (SQL) is the native API for Azure Cosmos DB for NoSQL. This is the most Azure-native option for new document-oriented applications because it aligns closely with Cosmos DB features such as integrated indexing, change feed, rich SDK support, and flexible JSON document modeling.

Use Core (SQL) when you want to design around Cosmos DB rather than adapt an existing database model. It is often the best fit for greenfield applications where you control the schema, partition key, and query patterns from the start.

Typical strengths include:

- **JSON document model** with flexible schema
- **SQL-like query syntax** for familiar filtering and projection
- **Integrated indexing** on document properties by default
- **Change feed** for event-driven and reactive architectures
- **SDK ecosystem** that is tightly aligned with Cosmos DB features
- **Fine-grained partitioning** control for scale and performance

A practical example is an e-commerce catalog or order system:

- Product catalog items can be stored as documents with nested attributes.
- Orders can be partitioned by `customerId` or `orderId` depending on access patterns.
- Change feed can publish order updates to downstream fulfillment or analytics systems.

Example container design:

```json
{
  "id": "order-1001",
  "customerId": "cust-42",
  "orderDate": "2026-04-26",
  "status": "Pending",
  "items": [
    { "sku": "A100", "qty": 2, "price": 19.99 },
    { "sku": "B200", "qty": 1, "price": 49.99 }
  ]
}
```

Example query:

```sql
SELECT c.id, c.customerId, c.status
FROM c
WHERE c.customerId = "cust-42" AND c.status = "Pending"
```

Important exam points:

- **Core (SQL)** supports querying within a container and can join over elements in the same JSON document or array structures, but it does not provide relational joins across arbitrary tables or containers like a traditional RDBMS.
- **Partition key choice** strongly affects performance and cost because queries that target a single partition are cheaper than cross-partition scans.
- **Request Unit (RU)** consumption depends on item size, indexing, query shape, and partition targeting.

> If you need Cosmos-native capabilities such as change feed, integrated indexing, and the most direct Azure feature alignment, Core (SQL) is usually the first choice for new document workloads.

**MongoDB API: compatibility-first document workloads**

The MongoDB API is for applications that already use MongoDB drivers, MongoDB-style queries, or MongoDB operational patterns and want to move to Azure Cosmos DB with minimal application rewrite. It is a compatibility layer, not a full clone of every MongoDB feature.

Use MongoDB API when:

- Your application already speaks MongoDB wire protocol or uses MongoDB drivers.
- You want to reduce migration effort from an existing MongoDB deployment.
- Your team is already comfortable with MongoDB query syntax and tooling.
- You can accept that some MongoDB features or behaviors may differ by supported version.

The main advantage is migration speed. The main tradeoff is that compatibility is version-dependent and not every MongoDB feature is supported exactly as in the source platform. For exam questions, always think about whether the workload needs a **lift-and-shift** or a redesign.

Common workload examples:

- **Content management** with flexible documents and existing MongoDB application code
- **User profiles** or session data where the app already uses MongoDB drivers
- **Catalog metadata** where the team wants document storage without rewriting the data access layer

Example MongoDB-style document:

```json
{
  "_id": "user-1001",
  "name": "Ava",
  "roles": ["buyer", "premium"],
  "lastLogin": "2026-04-26T10:15:00Z"
}
```

Example query:

```javascript
db.users.find(
  { roles: "premium" },
  { name: 1, lastLogin: 1 }
)
```

Practical selection guidance:

- Choose MongoDB API when the application is already built for MongoDB and you want the least disruptive migration path.
- Choose Core (SQL) when you are building new Cosmos-native document applications and want the strongest alignment with Azure Cosmos DB features.
- Do not assume MongoDB API gives you full parity with every MongoDB server version or feature set.

**MongoDB version and compatibility considerations**

Azure Cosmos DB for MongoDB supports specific compatibility levels rather than “all MongoDB versions.” The exact supported feature set depends on the account type and the compatibility level you choose.

Key implications:

- **Feature parity varies** by compatibility level.
- Some MongoDB operators, aggregation stages, or administrative behaviors may not be available.
- Driver compatibility matters because older or newer drivers may expect features not fully supported by the chosen level.
- Migration testing is essential before production cutover.

For exam prep, remember that “MongoDB API” means compatibility with MongoDB clients and syntax, not guaranteed equivalence with a self-managed MongoDB cluster.

**Cassandra API: wide-column and time-series style workloads**

The Cassandra API is intended for applications that use the Cassandra data model and CQL. It is a strong fit for wide-column workloads, high-ingest telemetry, time-series patterns, and systems that already use Cassandra tables and partitioning strategies.

Use Cassandra API when:

- Your source application already uses Cassandra and CQL.
- Your access pattern is naturally partition-oriented.
- You need to store large volumes of events, metrics, or sensor readings.
- You want to preserve a wide-column model during migration.

Typical workload examples:

- **Telemetry ingestion** from IoT devices
- **Event sourcing** streams with partitioned event history
- **Time-series** measurements keyed by device, tenant, or service
- **Session or activity logs** with predictable partition access

Example table design:

```sql
CREATE TABLE telemetry_by_device (
  device_id text,
  reading_date date,
  reading_time time,
  temperature double,
  humidity double,
  PRIMARY KEY ((device_id), reading_date, reading_time)
) WITH CLUSTERING ORDER BY (reading_date DESC, reading_time DESC);
```

Example query pattern:

```sql
SELECT device_id, reading_date, reading_time, temperature
FROM telemetry_by_device
WHERE device_id = 'dev-1001'
  AND reading_date = '2026-04-26';
```

Important Cassandra API considerations:

- **Partition key equality** is required for efficient access in most practical designs.
- **Clustering columns** support ordered retrieval within a partition.
- Range conditions are typically constrained by the clustering key order and preceding predicates.
- Secondary index behavior and feature parity are not identical to Apache Cassandra.
- Some Cassandra behaviors, schema features, or operational patterns may not be supported or may work differently in Cosmos DB.

> Do not treat Cosmos DB for Cassandra as a perfect drop-in replacement for every Cassandra feature. Validate CQL compatibility, indexing behavior, and query patterns before migration.

**Throughput, indexing, and cost**

API choice affects how you model data, but cost is still driven by request units, query shape, and indexing behavior. The same logical workload can have very different RU consumption depending on the API and schema design.

Key cost drivers:

- **Partition targeting**: single-partition reads are cheaper than fan-out queries.
- **Item size**: larger documents or rows cost more to read and write.
- **Indexing strategy**: automatic indexing is convenient but can increase write cost.
- **Query complexity**: filters, sorts, and cross-partition scans consume more throughput.
- **Data duplication**: denormalization can reduce query cost at the expense of storage and write amplification.

Practical guidance:

- In Core (SQL), design documents to match query patterns so you avoid expensive joins or scans.
- In MongoDB API, use indexes and query shapes that minimize collection scans.
- In Cassandra API, model tables around the exact query path rather than around normalization.

Example of a cost-aware design choice:

- If your app frequently reads orders by `customerId`, make `customerId` the partition key or part of the primary access path.
- If your app frequently reads telemetry by device and day, partition by device and cluster by date/time.
- If your app frequently filters by status across all records, consider whether that query should be served by a separate container or materialized view pattern rather than a cross-partition scan.

**Consistency level considerations**

Consistency is an Azure Cosmos DB platform capability, not a reason to choose one API over another. Core (SQL), MongoDB, and Cassandra APIs all run on Cosmos DB’s distributed storage engine, so consistency choices are generally independent of the API surface.

Cosmos DB consistency levels include:

- **Strong**
- **Bounded staleness**
- **Session**
- **Consistent prefix**
- **Eventual**

What matters for architecture:

- Stronger consistency can increase latency and reduce write/read flexibility in globally distributed scenarios.
- Session consistency is common for user-facing applications because it balances correctness and performance.
- Consistency choice does not replace good data modeling; a poor partition key still causes hot partitions and expensive queries.

For exam questions, separate these concerns:

- **API choice** = client compatibility, query language, migration effort, feature alignment
- **Consistency choice** = read freshness, latency, and global distribution behavior

**Migration and redesign scenarios**

A common exam theme is whether a workload can be lifted and shifted or whether it needs redesign.

Use a **lift-and-shift** approach when:

- The source application already uses MongoDB or Cassandra.
- The query patterns are stable and well understood.
- The supported feature set is sufficient for the app.
- You can validate compatibility with limited code changes.

Expect **schema or query redesign** when:

- The source database relies on unsupported features.
- The application uses relational joins or complex ad hoc analytics.
- The current schema is normalized in a way that conflicts with Cosmos DB’s partitioned model.
- The workload depends on database-specific behaviors that do not map cleanly.

Migration examples:

- A MongoDB-based customer profile service may move with minimal code changes if it uses supported drivers and common CRUD operations.
- A Cassandra-based telemetry system may migrate well if its tables already match query patterns and partitioning best practices.
- A relational reporting workload usually needs redesign because Cosmos DB APIs are not a direct substitute for a relational warehouse.

**Workload-to-API mapping**

Use the workload shape to narrow the API choice.

| Workload | Best-fit API | Why |
|---|---|---|
| New e-commerce catalog or order service | Core (SQL) | Native NoSQL model, integrated indexing, change feed, Azure-native features |
| Existing MongoDB application | MongoDB | Lowest rewrite effort for MongoDB drivers and query syntax |
| Telemetry or sensor ingestion | Cassandra | Wide-column model, partitioned access, time-series-friendly design |
| Event sourcing stream | Core (SQL) or Cassandra | Core (SQL) for change feed and document events; Cassandra for wide-column event storage |
| User profile store with MongoDB codebase | MongoDB | Compatibility with existing application stack |
| High-volume device readings by device and time | Cassandra | Natural partition and clustering model |

**CLI and provisioning notes**

Provisioning commands vary by Azure CLI version and by whether you are creating a Cosmos DB account for NoSQL, MongoDB, or Cassandra. For exam prep, focus on the API-specific account kind and the fact that the API must match the intended client model.

Examples of the general pattern:

```bash
# Core (SQL) / NoSQL account
az cosmosdb create \
  --name my-cosmos-nosql \
  --resource-group rg-data \
  --kind GlobalDocumentDB \
  --locations regionName=eastus failoverPriority=0 isZoneRedundant=false
```

```bash
# Cassandra account
az cosmosdb create \
  --name my-cosmos-cassandra \
  --resource-group rg-data \
  --kind Cassandra \
  --locations regionName=eastus failoverPriority=0 isZoneRedundant=false
```

```bash
# MongoDB account
az cosmosdb create \
  --name my-cosmos-mongo \
  --resource-group rg-data \
  --kind MongoDB \
  --locations regionName=eastus failoverPriority=0 isZoneRedundant=false
```

> Azure CLI syntax can change across versions, and some account types have additional parameters or separate provisioning paths. Validate the current command surface before using it in production.

**Exam decision cues**

When you see a scenario, ask these questions:

1. Does the application already use a specific database API or driver?
2. Is the workload document, wide-column, or event/telemetry oriented?
3. Can you redesign the schema around partition keys and query patterns?
4. Do you need Cosmos-native features such as change feed or integrated indexing?
5. Are there compatibility constraints, version limits, or unsupported features?
6. Will the query pattern be cheap under the chosen partitioning strategy?

Quick decision heuristics:

- Choose **Core (SQL)** for new Azure-native NoSQL applications.
- Choose **MongoDB** for MongoDB compatibility and lower migration friction.
- Choose **Cassandra** for wide-column, partition-heavy, telemetry-style workloads.
- Revisit the data model if the workload depends on joins, ad hoc analytics, or unsupported database-specific features.

**Summary**

API selection in Azure Cosmos DB is primarily about matching the workload to the right client model and data model. Core (SQL) is the native NoSQL choice and is usually best for new Cosmos-native applications. MongoDB is best when compatibility and migration speed matter. Cassandra is best when the workload already fits a wide-column, partition-oriented design.

The strongest architecture decisions combine API fit with partition strategy, indexing, consistency, and cost awareness. If the workload does not fit the API cleanly, redesign the schema before forcing the platform to mimic a different database.
