## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Partition key fundamentals**

A partition key is the field Cosmos DB uses to distribute items across logical partitions. Good selection is one of the most important design decisions because it affects scalability, latency, cost, and how easily you can query data.

In Azure Cosmos DB, the partition key value is part of the item’s identity within a logical partition. For direct reads, you typically need both the item `id` and the partition key value. The `id` is unique within a logical partition, not necessarily across the entire container.

- **Logical partition** groups items that share the same partition key value.
- **Physical partition** is an internal storage and compute unit that hosts one or more logical partitions.
- **Throughput** is distributed across physical partitions, so uneven key distribution can create hot spots.
- **Point reads** are fastest when you know both `id` and partition key value.

**Why partition key choice matters**

A poor partition key can create a hot partition, increase request unit consumption, and limit effective scale. A strong key spreads traffic and storage evenly while still supporting your most common access patterns.

Choose the partition key based on how the application reads and writes data, not just on what seems unique. Uniqueness alone is not enough if the key causes skew or forces cross-partition queries.

- Favor keys with high cardinality.
- Favor keys with even access distribution.
- Favor keys that appear in your most common query filters.
- Avoid keys that are monotonically increasing if they concentrate writes.

**Core selection criteria**

A good partition key usually has these properties:

- **High cardinality** so many distinct values exist.
- **Even distribution** so no single value dominates traffic.
- **Stable value** so the key does not need to change after insert.
- **Common query alignment** so reads can target a single partition when possible.
- **Reasonable item grouping** so related items can be queried together.

A partition key that is too broad can make related data harder to query efficiently. A partition key that is too narrow can create hot partitions and storage imbalance.

**Common partition key patterns**

| Pattern | When it works well | Tradeoff |
|---|---|---|
| `tenantId` | Multi-tenant SaaS with tenant-scoped access | Large tenants can become hot |
| `userId` | User-centric workloads and profile data | Cross-user analytics become expensive |
| `orderId` | Order-centric transactional data | Poor if many items share one order |
| `deviceId` | IoT telemetry and device state | A noisy device can dominate one partition |
| `region` | Small regional datasets | Usually too low-cardinality for large scale |

Use the pattern that matches your dominant access path. If you optimize for analytics but the app mostly serves tenant-scoped transactions, you will likely pay for cross-partition queries later.

**Point reads and query behavior**

A point read is the most efficient way to retrieve a single item. In Cosmos DB SQL API, a point read requires the item `id` and the partition key value. This is efficient because the service can route the request directly to the correct logical partition.

Queries are different. If your query filter includes the partition key, Cosmos DB can often limit the work to a single partition. If not, the query may fan out across partitions, which increases latency and request units.

```bash
# Azure CLI: create a Cosmos DB SQL container with a partition key path
az cosmosdb sql container create \
  --account-name mycosmosacct \
  --database-name appdb \
  --name orders \
  --partition-key-path "/tenantId"
```

```bash
# Azure CLI: list SQL containers in a database
az cosmosdb sql container list \
  --account-name mycosmosacct \
  --database-name appdb \
  --output table
```

> Azure CLI command names and parameters can vary by CLI version and extension. Verify with `az cosmosdb -h` and `az cosmosdb sql container -h` before using in production scripts.

**Designing for query patterns**

Start with the application’s top queries and write operations. Then choose a key that lets the most frequent operations stay within a single logical partition.

- If you often read all items for one customer, use `customerId` or `tenantId`.
- If you often read one order and its line items, consider `orderId` for the order aggregate.
- If you often write time-series telemetry, avoid a pure timestamp key because it can cluster writes.
- If you need both tenant isolation and per-user access, consider a composite strategy in the data model rather than forcing one key to satisfy every query.

A common mistake is choosing a key that is unique but not useful for grouping. For example, using a random GUID for every item can distribute writes well, but it makes related items hard to query together.

**Hot partitions and skew**

A hot partition occurs when one partition key value receives disproportionate traffic. This can happen when:

- One tenant is much larger than others.
- One device produces far more telemetry.
- One key value is used for all writes.
- A timestamp or status value has low cardinality.

Symptoms include throttling, uneven latency, and high request unit consumption for a subset of requests.

Mitigation options include:

- **Better key selection** to spread traffic more evenly.
- **Synthetic keys** that add a shard component, such as `tenantId-userId`.
- **Bucketing** by time or hash when the access pattern allows it.
- **Materialized views** or separate containers for read-heavy projections.
- **Data reshaping** so one logical entity does not accumulate unbounded items under one key.

**Synthetic partition keys**

A synthetic key combines multiple attributes to improve distribution or support a specific access pattern. Use this when a single natural key is too skewed or too coarse.

Example:

```json
{
  "id": "order-1001",
  "tenantId": "contoso",
  "userId": "u-42",
  "partitionKey": "contoso|u-42",
  "status": "open"
}
```

This can help distribute data better than `tenantId` alone, but it also changes how you query. You must know or derive the synthetic value to do efficient point reads.

- Use synthetic keys when you can predict the access pattern.
- Avoid them if they make the application logic too complex.
- Keep the derivation deterministic so reads can reconstruct the key.

**Hierarchical partition keys**

Hierarchical partition keys are a Cosmos DB NoSQL feature that let you partition on multiple path components. They are useful when one dimension alone is not enough to distribute data well, especially in multi-tenant designs.

Use them when:

- You need tenant isolation plus a second dimension for scale.
- You want to reduce hot partitions within a large tenant.
- Your account and container capabilities support the feature.

Be careful not to assume they solve all skew problems automatically. You still need to understand the dominant access patterns and the distribution of values at each level.

**Choosing between candidate keys**

A practical selection process looks like this:

1. Identify the most common read and write operations.
2. List candidate fields that appear in those operations.
3. Check cardinality and distribution for each candidate.
4. Estimate whether one value could dominate traffic.
5. Prefer the key that keeps the most common operations within one partition.
6. Validate with realistic data volumes and tenant sizes.

A simple decision rule is: optimize for the query you must make fast most often, not the query you hope will be rare.

**Example container design**

For a SaaS order system, `tenantId` is often a strong starting point if most operations are tenant-scoped.

```json
{
  "id": "order-1001",
  "tenantId": "contoso",
  "customerId": "cust-77",
  "orderDate": "2026-04-26",
  "status": "open"
}
```

If one tenant becomes very large, you may need a more granular key or a second-level partitioning strategy.

For telemetry, a device-centric model may work:

```json
{
  "id": "2026-04-26T10:15:00Z",
  "deviceId": "device-123",
  "bucket": "2026-04",
  "temperature": 21.4
}
```

If one device is extremely chatty, add a bucket or shard component to reduce concentration.

**ARM and Bicep considerations**

When you define a Cosmos DB SQL container in infrastructure as code, the partition key path must match the property name in your documents exactly.

```bicep
resource container 'Microsoft.DocumentDB/databaseAccounts/sqlDatabases/containers@2024-05-01' = {
  name: '${account.name}/${databaseName}/orders'
  properties: {
    resource: {
      id: 'orders'
      partitionKey: {
        paths: [
          '/tenantId'
        ]
        kind: 'Hash'
      }
      indexingPolicy: {
        indexingMode: 'consistent'
        automatic: true
      }
    }
  }
}
```

Use the exact API version supported in your environment and verify the schema against current Azure documentation before deployment. Resource model details can vary by API version and account capabilities.

**Operational checks**

After deployment, validate that the key behaves well under load.

- Watch request unit consumption by operation type.
- Check for throttling on specific tenants or key values.
- Review query metrics for cross-partition fan-out.
- Confirm that storage growth is balanced across partitions.
- Test with realistic data skew, not just uniform sample data.

Useful CLI and query examples:

```bash
# Show account metadata
az cosmosdb show --name mycosmosacct --resource-group rg-app
```

```sql
-- SQL API query that can stay partition-scoped when tenantId is supplied
SELECT * FROM c
WHERE c.tenantId = "contoso" AND c.status = "open"
```

```sql
-- Cross-partition style query if partition key is not part of the filter
SELECT * FROM c
WHERE c.status = "open"
```

**Exam-focused takeaways**

- Pick the partition key from the dominant access pattern.
- Aim for high cardinality and even distribution.
- Avoid keys that create hot partitions or force frequent cross-partition queries.
- Remember that `id` is unique within a logical partition, not globally.
- Use point reads whenever you know both `id` and partition key value.
- Treat hierarchical and synthetic keys as design tools, not default answers.

**Common mistakes to avoid**

- Choosing a key only because it is unique.
- Using a low-cardinality field like `status` or `region` for a large container.
- Ignoring tenant skew in multi-tenant systems.
- Assuming one partition key must satisfy every query equally well.
- Designing for today’s sample data instead of future growth.
