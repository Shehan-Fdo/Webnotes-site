## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Partitioning strategy overview**

Partitioning is the practice of splitting data, workload, or traffic into smaller units so you can scale, isolate failures, reduce contention, and manage lifecycle operations more effectively. In Azure architecture, the right partitioning model depends on whether you need **write scale**, **read scale**, **tenant isolation**, **data residency**, or **operational manageability**.

A useful exam distinction is this:

- **Partitioning** improves manageability and can improve read distribution or locality.
- **Sharding** is the application-level distribution of data across multiple databases or stores to achieve horizontal write scale.
- **Replication** copies data to improve availability, read performance, or disaster recovery.

> Partitioning alone does not automatically create write scale in every Azure service. In many cases, you need sharding or a different data platform for true horizontal write throughput.

**Core design goals**

Use partitioning when you need one or more of the following:

- **Scale out** by distributing load across partitions, shards, or entities.
- **Reduce blast radius** by isolating tenants, regions, or workloads.
- **Improve performance** through partition pruning, locality, or smaller working sets.
- **Simplify operations** such as archival, purge, reindexing, or backup restore.
- **Meet compliance** requirements for residency, sovereignty, or separation of duties.

Common partitioning dimensions include:

- Tenant
- Region
- Time
- Customer segment
- Device or user ID
- Workload type
- Message type

**Partitioning versus sharding versus replication**

| Pattern | Primary purpose | Typical Azure use | Key limitation |
|---|---|---|---|
| Partitioning | Manageability and locality | SQL table partitioning, Cosmos DB partitioning, Storage partition keys | Does not always increase write throughput |
| Sharding | Horizontal write scale | Multiple Azure SQL databases, app-managed data distribution | Adds routing, rebalancing, and cross-shard query complexity |
| Replication | Availability and read scale | Azure SQL geo-replication, failover groups, Cosmos DB multi-region writes | Does not solve write bottlenecks by itself |

For AZ-305, choose the simplest pattern that satisfies the requirement. If the requirement says “faster writes across very large datasets,” think **sharding** first. If it says “archive old data efficiently” or “prune queries by date,” think **partitioning**.

**Azure SQL Database and Managed Instance partitioning**

Azure SQL supports table partitioning for large tables, but it is primarily a **manageability** feature, not a horizontal scale feature. You use it to split a large table into smaller physical units based on a partition function and partition scheme, often by date or another range key.

Typical benefits:

- **Partition elimination** reduces the amount of data scanned when queries filter on the partitioning column.
- **Partition switching** enables fast load, archive, and purge operations by moving whole partitions in or out with minimal logging.
- **Maintenance isolation** lets you rebuild indexes or manage statistics on a subset of data instead of the entire table.
- **Sliding window** designs support time-based retention, such as monthly fact tables.

Important implementation details:

- The partitioning column must be chosen carefully because it affects query patterns, index design, and maintenance.
- Partitioning does not automatically distribute a single database across multiple nodes for write scale.
- Indexes must be aligned with the partition scheme when you want efficient partition operations.
- Nonaligned indexes can block partition switching or reduce the operational benefits.
- Partitioning works best when most queries filter on the partition key or when lifecycle operations dominate.

A common pattern is a time-based fact table:

```sql
CREATE PARTITION FUNCTION pf_OrderDate (date)
AS RANGE RIGHT FOR VALUES
(
    '2025-01-01',
    '2025-02-01',
    '2025-03-01'
);
GO

CREATE PARTITION SCHEME ps_OrderDate
AS PARTITION pf_OrderDate
ALL TO ([PRIMARY]);
GO

CREATE TABLE dbo.Orders
(
    OrderId     bigint       NOT NULL,
    OrderDate   date         NOT NULL,
    CustomerId  int          NOT NULL,
    Amount      decimal(18,2) NOT NULL,
    CONSTRAINT PK_Orders
        PRIMARY KEY CLUSTERED (OrderDate, OrderId)
) ON ps_OrderDate (OrderDate);
GO
```

This example shows the pattern, but in real deployments you should verify:

- The clustered index and all required nonclustered indexes are aligned.
- The partitioning key matches the most common filter or retention boundary.
- The design supports partition switching if you need fast archival.

Operational notes:

- **Partition switching** requires compatible table structures between source and target.
- **Partition elimination** only helps when predicates are sargable and reference the partitioning column in a way the optimizer can use.
- **Statistics** and index maintenance still matter; partitioning does not remove the need for tuning.
- **Compression** can be applied per partition in many designs to reduce storage and improve I/O efficiency.

A practical sliding-window workflow is:

1. Create a new empty future partition.
2. Load new data into the newest partition.
3. Switch out the oldest partition to an archive table.
4. Truncate or move the archive table as needed.
5. Merge or split boundary values as the retention window advances.

> Partitioning is not a substitute for sharding in Azure SQL. If one database cannot handle the write rate, you usually need application-level sharding across multiple databases or another data service.

**Cosmos DB partitioning**

Azure Cosmos DB uses partitioning as a core scale mechanism. The partition key determines how items are distributed across physical partitions and how requests are routed. Good partition key selection is one of the most important design decisions in Cosmos DB.

Key concepts:

- A **logical partition** groups items that share the same partition key value.
- A **physical partition** is an internal storage and throughput unit managed by Cosmos DB.
- A **point read** is most efficient when you know both the item ID and the partition key.
- A **transactional batch** is limited to items within the same logical partition.

Partition key selection criteria:

- Choose a key with **high cardinality** so data spreads evenly.
- Avoid keys that create **hot partitions**, such as a single tenant or a single status value with extreme skew.
- Prefer keys that match your most common read and write access patterns.
- Consider whether you need **point-read efficiency** or **transactional scope** within a tenant or entity group.
- Balance even distribution against query locality.

Practical examples:

- `tenantId` works well for tenant-isolated workloads if tenant sizes are balanced.
- `userId` works well for user-centric apps with many users and frequent point reads.
- `deviceId` works well for IoT telemetry when each device generates independent traffic.
- `orderId` is usually poor if it is random and not used for grouping related operations.

Cosmos DB limits and behavior to remember:

- A logical partition has a size limit, so extremely large tenants or groups can become problematic.
- A partition key value cannot be changed in place for an existing item; to change it, you create a new item and delete the old one.
- Cross-partition queries are possible, but they cost more RU and can be slower than single-partition queries.
- Transactional batch and stored procedure scope are limited to one logical partition.
- Point reads are cheapest when the partition key is part of the request.

A typical container creation example:

```bash
az cosmosdb sql container create \
  --account-name mycosmosacct \
  --database-name appdb \
  --name orders \
  --partition-key-path "/tenantId" \
  --throughput 400
```

If you use autoscale or database-level throughput, adjust the command to match the account and database model you actually deploy. The key exam idea is that the container must be created with a partition key path that matches your access pattern.

Hierarchical partition keys are relevant for large multi-tenant or high-cardinality workloads. They let you combine multiple fields, such as tenant, region, and user, to improve distribution while preserving locality for related operations. Use them when a single partition key is not enough to avoid skew or when you need a more nuanced grouping strategy.

Example conceptual model:

- `/tenantId`
- `/tenantId`, `/region`
- `/tenantId`, `/region`, `/userId`

Use hierarchical keys when:

- One tenant can become too large for a single logical partition.
- You need to keep related data together while still spreading load.
- You want better control over hot-spotting in multi-tenant systems.

Cosmos DB design guidance:

- Use a partition key that supports your most frequent **point reads**.
- Keep transactional operations within a single logical partition when possible.
- Avoid designs that require frequent cross-partition joins or aggregations.
- Model for the query patterns you actually need, not for theoretical flexibility.

**Azure Storage partitioning**

Azure Storage scales through service-managed partitioning, but the exact behavior differs by service. The main design concern is usually the **account-level scalability target** and request distribution, not container-level isolation.

For Blob Storage:

- Use naming patterns that distribute access across prefixes when you expect very high request rates.
- Avoid concentrating all hot blobs under a single narrow prefix if your workload is extremely bursty.
- Use access tiers to optimize cost and retrieval patterns:
  - **Hot** for frequent access
  - **Cool** for infrequent access
  - **Archive** for long-term retention
- Separate storage accounts when you need distinct scalability boundaries, security boundaries, or billing boundaries.

For Table Storage:

- The **PartitionKey** groups entities for efficient retrieval and scalability.
- The **RowKey** uniquely identifies an entity within a partition and supports ordered access within that partition.
- A poor PartitionKey creates hot partitions and throttling.
- Queries that filter by PartitionKey are much more efficient than scans across many partitions.

Example Table Storage entity design:

| Field | Purpose |
|---|---|
| PartitionKey | Tenant, region, or date bucket |
| RowKey | Unique entity ID or timestamp-based suffix |
| Timestamp | Service-managed metadata |
| ETag | Concurrency control |

A practical pattern:

- `PartitionKey = tenant-001`
- `RowKey = order-20260426-0001`

This works well when you need tenant-local reads and writes. If one tenant becomes too large, you may need a finer-grained key such as tenant plus month or tenant plus region.

For Queue Storage and File shares, partitioning is mostly service-managed. Your design focus is usually on account limits, request distribution, and whether you need separate accounts for isolation or scale.

Storage design guidance:

- Use separate storage accounts when you need a hard scale boundary or independent lifecycle management.
- Do not assume containers provide throughput isolation comparable to separate accounts.
- Use blob naming and folder-like prefixes to support operational organization, but remember that blob namespaces are flat.
- Choose access tiers based on access frequency and retention, not as a partitioning substitute.

**Azure Service Bus partitioning**

Azure Service Bus partitioning is available for certain entities in the **Standard** tier. It distributes messages across multiple message brokers to improve throughput and availability characteristics for the entity.

Important exam points:

- Partitioning applies to queues and topics in the Standard tier.
- It is not a universal feature for every entity type or tier.
- Partitioned entities can improve throughput, but they do not guarantee strict global ordering across all partitions.
- If you need ordering, use **sessions** or design for ordered processing within a partition or session boundary.
- Duplicate detection and sessions have design implications that you should validate against the entity configuration.

Practical implications:

- Use partitioned queues or topics when you need higher throughput and can tolerate relaxed global ordering.
- Use sessions when you need ordered processing for a related message group.
- Be careful combining partitioning with strict ordering requirements, because partitioning can spread messages across partitions.

Example conceptual configuration:

```bash
az servicebus queue create \
  --resource-group rg-messaging \
  --namespace-name sb-prod \
  --name orders-queue \
  --enable-partitioning true
```

Design guidance:

- Use partitioning for throughput and resilience, not as a substitute for application-level ordering guarantees.
- If ordering is critical, prefer session-aware processing or a single ordered stream per key.
- Validate whether your workload needs duplicate detection, dead-letter handling, or session state before enabling partitioning.

**Multi-region partitioning and data placement**

Multi-region design is about placing data close to users, meeting residency requirements, and surviving regional failures. Partitioning here often means splitting data by geography, tenant geography, or workload geography.

Common patterns:

- **Regional affinity**: keep a user’s data in the nearest or home region.
- **Geo-partitioning**: split tenants or customers by region to satisfy residency rules.
- **Active-passive**: one primary region handles writes, another region is ready for failover.
- **Active-active**: multiple regions accept traffic, usually with conflict handling or a platform that supports multi-master writes.

Azure design guidance:

- For SQL-based systems, geo-replication and failover groups improve availability and disaster recovery, but they do not provide automatic write scale-out.
- True multi-writer scale for relational workloads usually requires sharding or a different data platform.
- For Cosmos DB, multi-region writes can support active-active patterns, but you must design for conflict resolution and consistency.
- For storage and messaging, choose regionally appropriate services and route users to the nearest healthy endpoint.

Data residency considerations:

- Keep regulated data in the required geography.
- Avoid cross-region replication if it violates compliance or sovereignty requirements.
- Use separate regional deployments when legal or latency requirements differ by geography.

Routing and failover patterns:

- Use **Azure Front Door** or **Traffic Manager** for global traffic routing.
- Use health probes to direct users to a healthy region.
- Keep application configuration region-aware so you can fail over cleanly.
- Test failover paths, not just the data replication layer.

Conflict resolution patterns for active-active systems:

- Last write wins for simple, low-risk data.
- Version vectors or timestamps for more controlled merges.
- Application-level merge logic for business-critical records.
- Event sourcing or append-only models when you need a full history of changes.

**Decision guidance for AZ-305 scenarios**

Use the following decision logic when the exam asks you to choose a partitioning approach:

1. If the problem is **large table maintenance** in Azure SQL, choose table partitioning.
2. If the problem is **write throughput across many tenants**, choose sharding or a distributed database design.
3. If the problem is **efficient point reads and tenant isolation** in Cosmos DB, choose a partition key aligned to the access pattern.
4. If the problem is **message throughput** in Service Bus and ordering is not global, consider partitioned entities.
5. If the problem is **storage scale boundaries**, use separate storage accounts rather than assuming containers isolate throughput.
6. If the problem is **regional compliance or latency**, partition by geography and route users to regional endpoints.

A few exam-style examples:

- A SaaS application with many tenants and uneven load:
  - Use Cosmos DB with a tenant-aware or hierarchical partition key if the workload fits Cosmos DB.
  - Use sharding across Azure SQL databases if relational constraints require SQL and write scale is the priority.
- A telemetry platform ingesting millions of device events:
  - Use Cosmos DB partitioned by `deviceId` or a hierarchical key.
  - Use Event Hubs for ingestion and store processed data in partitioned analytical tables.
- A financial system with monthly retention and large historical tables:
  - Use Azure SQL table partitioning by date for sliding-window management.
- A global app with regional data residency:
  - Partition data by region and route users to the nearest compliant deployment.

**Practical comparison of common choices**

| Requirement | Best fit | Why |
|---|---|---|
| Fast archival of old SQL data | SQL table partitioning | Enables partition switching and sliding windows |
| More write throughput for relational data | Sharding | Distributes writes across multiple databases |
| Efficient tenant-local reads in Cosmos DB | Cosmos partition key | Keeps related items together |
| Global read availability | Replication | Improves read locality and resilience |
| Ordered message processing | Sessions | Preserves order within a session key |
| Storage scale isolation | Separate storage accounts | Provides clearer service-boundary control |

**Common pitfalls**

- Choosing a partition key with low cardinality, such as `status`, which creates hot partitions.
- Assuming partitioning automatically increases write throughput in Azure SQL.
- Designing Cosmos DB queries that require frequent cross-partition scans.
- Using Service Bus partitioning while expecting strict global ordering.
- Relying on storage containers for throughput isolation instead of storage accounts.
- Ignoring index alignment and partition switching requirements in SQL Server-based designs.
- Forgetting that changing a Cosmos DB partition key requires rewriting the item.

**Quick configuration and design checklist**

- Identify the dominant access pattern: point read, range query, batch write, or archive.
- Decide whether you need manageability, throughput scale, or availability.
- Pick the partition key or shard key based on the hottest query path.
- Validate cardinality and skew with realistic data volumes.
- Check service-specific limits, such as logical partition size or tier restrictions.
- Confirm how the design affects indexing, ordering, and transactional scope.
- Test failover, rebalancing, and maintenance operations before production.

**Summary**

Partitioning is a design tool, not a universal scaling fix. In Azure, the right choice depends on the service and the workload:

- Use **SQL partitioning** for lifecycle management and partition elimination.
- Use **Cosmos DB partitioning** for distributed scale and efficient access patterns.
- Use **Storage partitioning patterns** to respect account limits and request distribution.
- Use **Service Bus partitioning** for higher throughput when global ordering is not required.
- Use **regional partitioning** to satisfy residency, latency, and failover goals.

For AZ-305, the key is to match the partitioning strategy to the business requirement and the service’s actual behavior.
