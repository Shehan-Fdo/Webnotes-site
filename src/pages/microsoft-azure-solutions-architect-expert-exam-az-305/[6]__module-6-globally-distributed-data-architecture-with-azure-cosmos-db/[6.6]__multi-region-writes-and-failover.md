## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Overview**

Multi-region design is about balancing availability, latency, consistency, and operational complexity. In Azure, you usually choose between active-active and active-passive patterns, then pair the data layer with routing, health probes, and application-state handling that match your recovery objectives.

For AZ-305, the key is not memorizing one “best” pattern. You need to recognize which Azure service fits the workload, what failure mode you are protecting against, and how the client reaches the surviving region.

**Core design goals**

- **Availability**: keep the application reachable when a region fails.
- **Resiliency**: recover from zonal, regional, or dependency failures.
- **Performance**: reduce latency by serving users from nearby regions.
- **Data protection**: preserve acceptable <abbr title="Recovery Point Objective">RPO</abbr> and <abbr title="Recovery Time Objective">RTO</abbr>.
- **Operational simplicity**: avoid designs that are hard to test, monitor, or support.

**Common patterns**

| Pattern | Write model | Read model | Typical use |
|---|---|---|---|
| Active-active | Multiple regions accept writes | Multiple regions serve reads | Global apps with distributed users and conflict handling |
| Active-passive | One region writes, standby region takes over | Secondary may serve reads | Most business apps with simpler data semantics |
| Hybrid | Some components active-active, others active-passive | Mixed | Complex systems with different data tiers |

**Reference architecture sketch**

```text
Users
  |
  v
Azure Front Door / Traffic Manager
  |
  +-------------------+-------------------+
  |                   |                   |
Region A             Region B             Region C
App + Data           App + Data           App + Data
  |                   |                   |
  +-------- replication / sync / conflict handling --------+
```

**Active-active vs active-passive**

Active-active improves availability and can reduce latency, but it increases design complexity. You must handle concurrent writes, conflict resolution, and data consistency across regions. Active-passive is simpler because only one region accepts writes at a time, but failover can create a brief write outage and a larger RTO.

- **Active-active** works best when the application can tolerate eventual consistency or has a strong conflict strategy.
- **Active-passive** works best when the data model is centralized or the application expects a single writer.
- **Hybrid** is common when the web tier can scale globally but the transactional data tier cannot.

**Synchronous vs asynchronous replication**

Replication choice drives recovery behavior.

- **Synchronous replication** waits for acknowledgement from another location before completing the write.
  - Lower data loss risk.
  - Higher latency.
  - Usually limited to nearby regions or specific service designs.
- **Asynchronous replication** commits locally first and ships changes later.
  - Lower write latency.
  - Better geographic flexibility.
  - Some data loss is possible during a regional outage.

For exam scenarios, ask:
- How much data loss is acceptable?
- Can the app tolerate delayed replication?
- Does the service support automatic failover or only manual promotion?
- Is the write path single-master or multi-master?

**Azure Front Door and Traffic Manager**

Routing is separate from data replication. You often use a global entry point to steer clients to a healthy region.

- **Azure Front Door** is HTTP/HTTPS aware and can do layer 7 routing, health probes, path-based routing, and fast failover.
- **Azure Traffic Manager** is DNS-based and routes clients to endpoints based on priority, performance, weighted, or geographic routing.

Front Door usually gives a better user experience for web apps because it can fail over without waiting for DNS cache expiry. Traffic Manager is useful when you need DNS-level global distribution or non-HTTP endpoints.

**Front Door health probe example**

```json
{
  "healthProbeSettings": {
    "probePath": "/health",
    "probeProtocol": "Https",
    "probeMethod": "GET",
    "intervalInSeconds": 30
  },
  "loadBalancingSettings": {
    "sampleSize": 4,
    "successfulSamplesRequired": 3,
    "additionalLatencyInMilliseconds": 50
  }
}
```

- Use a dedicated `/health` endpoint that checks critical dependencies.
- Keep the probe lightweight.
- Return unhealthy when the app cannot serve real traffic, not just when the process is alive.

**Traffic Manager profile example**

```bash
az network traffic-manager profile create \
  --name tm-global-app \
  --resource-group rg-global \
  --routing-method Priority \
  --unique-dns-name contoso-global-app \
  --ttl 30 \
  --protocol HTTPS \
  --port 443 \
  --path /health
```

- **Priority** routing supports active-passive failover.
- **Performance** routing sends users to the closest healthy endpoint.
- **Weighted** routing supports gradual traffic shifts and canary releases.
- Lower TTL helps failover, but client and resolver caching still affect timing.

**Azure SQL failover groups**

Azure SQL Database and Azure SQL Managed Instance support failover groups for geo-redundancy. A failover group provides a read-write listener endpoint and a read-only listener endpoint, which simplifies application connection strings.

The important exam distinction is that failover groups are designed for a primary and secondary relationship, not true multi-master writes.

- **Automatic failover** can occur when the primary becomes unavailable and the failover policy allows it.
- **Planned failover** is used for controlled switchover, such as maintenance or region migration.
- **Unplanned failover** happens after an outage and may involve some data loss depending on replication lag.

> The listener DNS name stays stable, but the backend target changes during failover. Your app should connect to the listener, not hardcode a region-specific server.

**SQL failover group connection strings**

```ini
Server=tcp:myfailovergroup.database.windows.net,1433;
Database=appdb;
User ID=appuser;
Password=StrongPasswordHere!;
Encrypt=True;
TrustServerCertificate=False;
Connection Timeout=30;
```

- Use the **read-write listener** for transactional operations.
- Use the **read-only listener** for reporting queries when the secondary is available.
- Design retry logic for transient errors during failover.

**Failover group behavior to remember**

- The secondary is typically read-only until failover.
- Replication is asynchronous, so some lag can exist.
- Failover changes the primary role and updates the listener target.
- Applications may see brief connection drops and must reconnect.

**Cosmos DB multi-region writes**

Azure Cosmos DB is the main Azure service for true multi-region write scenarios. It supports multiple writable regions, global distribution, and tunable consistency. This makes it a strong fit for globally distributed applications that need low-latency writes near users.

However, multi-region writes are not free. They increase cost, operational complexity, and the need for conflict resolution.

**How Cosmos DB multi-region writes work**

- You can add multiple regions to the account.
- Any configured write region can accept writes.
- The SDK routes requests to the nearest available region when configured correctly.
- If a region is unavailable, the account can continue serving traffic from other regions.

**Consistency and conflict resolution**

Multi-region writes require you to think about conflicts. If two regions update the same item concurrently, Cosmos DB must resolve the conflict.

Common conflict resolution approaches include:
- **Last writer wins**: uses a timestamp or custom property to choose the winner.
- **Custom conflict resolution**: uses a stored procedure or application-defined logic in some scenarios.
- **Manual reconciliation**: the application detects and resolves conflicts later.

Consistency level matters:
- **Strong consistency** gives the highest read guarantees but is the most restrictive and can reduce write availability and increase latency.
- **Bounded staleness** limits how stale reads can be.
- **Session**, **consistent prefix**, and **eventual** consistency provide more flexibility and better performance.

For multi-region writes, choose the weakest consistency level that still satisfies the business requirement. Strong consistency across multiple write regions is usually not the practical choice for globally distributed apps.

**Cosmos DB conflict example**

```json
{
  "id": "order-1001",
  "customerId": "C123",
  "status": "Pending",
  "lastUpdated": "2026-04-26T10:15:00Z",
  "_etag": "\"0000-0000-0000\""
}
```

If two regions update `status` at nearly the same time, the final value depends on the configured conflict policy. Your application should avoid ambiguous concurrent updates when possible.

**When multi-region writes are a poor fit**

- The workload has frequent conflicts on the same records.
- The application requires strict transactional semantics across regions.
- The team cannot support conflict testing and reconciliation.
- The cost of global replication outweighs the latency benefit.

**Regional partition scenarios**

A regional partition can isolate one region from the rest of the account. In a multi-write design:
- Healthy regions may continue accepting writes.
- Clients in the isolated region may fail over to another region if routing and SDK behavior allow it.
- Conflicts can increase after connectivity is restored if both sides accepted divergent updates.

This is why idempotent writes and conflict-aware data models matter.

**Azure App Service deployment slots**

App Service slots are not a multi-region data replication feature, but they matter in failover design because they support safe application deployment and warm standby behavior.

- Use slots for blue-green deployment within a region.
- Combine slots with multi-region deployment to reduce release risk.
- Keep configuration consistent across regions using deployment automation.

Slots help with application availability, but they do not replace regional redundancy.

**AKS multi-region considerations**

Azure Kubernetes Service (AKS) can be deployed in multiple regions, but the platform does not give you a single built-in active-active application fabric.

- Use separate clusters per region for strong isolation.
- Replicate container images through Azure Container Registry.
- Externalize state to a resilient data service.
- Use global routing in front of the clusters.

Important AKS design points:
- Keep pods stateless when possible.
- Store session state outside the cluster.
- Use readiness and liveness probes correctly.
- Plan for cluster-level and region-level failover separately.

**Application state and session management**

Failover often fails because the app is stateless in theory but stateful in practice.

- Store sessions in a shared external store such as Azure Cache for Redis, SQL, or a token-based model.
- Avoid in-memory session state for multi-region apps.
- Make write operations idempotent so retries do not create duplicate records.
- Invalidate or refresh caches after failover to avoid stale reads.

**Cache and session strategy**

```text
Client -> Front Door -> App Region A
                     -> Redis / DB
                     -> App Region B
```

- Prefer stateless APIs with token-based authentication.
- Use distributed cache for performance, not as the system of record.
- Treat cache as disposable during failover.

**Practical failover routing patterns**

1. **Priority routing**
   - Send all traffic to the primary region.
   - Fail over to secondary when health probes fail.
   - Best for active-passive designs.

2. **Performance routing**
   - Send users to the nearest healthy region.
   - Best for globally distributed read-heavy apps.
   - Requires careful data consistency planning.

3. **Weighted routing**
   - Split traffic between regions.
   - Useful for gradual migration, testing, or canary releases.
   - Not a substitute for data replication.

**Example architecture choices**

- A public web app with low write volume and strict relational consistency:
  - Use Azure SQL Database with failover groups.
  - Front it with Front Door or Traffic Manager.
  - Keep the app stateless and use retry logic.

- A global mobile app with user-generated content and local writes:
  - Use Cosmos DB with multi-region writes.
  - Use conflict resolution and session consistency.
  - Route users through Front Door.

- An internal line-of-business app with one primary region and disaster recovery:
  - Use active-passive SQL or storage replication.
  - Keep the secondary warm but not serving writes.
  - Test planned failover regularly.

**Comparison table for exam decisions**

| Requirement | Better fit | Why |
|---|---|---|
| Lowest write latency worldwide | Cosmos DB multi-region writes | Local writes in multiple regions |
| Simple relational failover | Azure SQL failover group | Managed primary/secondary model |
| Fast web failover | Azure Front Door | Layer 7 health-based routing |
| DNS-based global routing | Traffic Manager | Priority, performance, weighted routing |
| Minimal conflict handling | Active-passive | Single writer simplifies data integrity |
| Global read distribution with one writer | SQL failover group or single-write Cosmos DB | Read scale without multi-master complexity |

**Design tradeoffs to evaluate**

- **Latency vs consistency**: closer writes reduce latency but increase conflict risk.
- **Availability vs simplicity**: more regions improve resilience but add operational burden.
- **RPO vs cost**: tighter recovery objectives usually require more replication and more spend.
- **Automation vs control**: automatic failover is faster, but manual failover can reduce accidental promotion.

**Failover testing and operations**

You should test failover before production incidents force it.

1. Validate health probes and routing rules.
2. Simulate regional unavailability.
3. Confirm the app reconnects and retries safely.
4. Verify data consistency after failover.
5. Measure actual RTO and compare it with the target.

Useful operational checks:
- Monitor probe failures and endpoint health.
- Alert on replication lag.
- Track connection errors during failover windows.
- Review logs for duplicate writes or conflict resolution events.

**Common exam traps**

- Assuming multi-region write means automatic conflict-free writes.
- Confusing DNS failover with instant application failover.
- Treating a secondary database as writable before promotion.
- Ignoring session state and cache behavior during region loss.
- Choosing the most resilient service without checking data model fit.
- Overlooking the difference between planned and unplanned failover.

**Decision checklist**

- Do you need multiple regions to accept writes?
- Can the application resolve write conflicts?
- Is the data relational or globally distributed document/NoSQL data?
- Do you need layer 7 routing or DNS-based routing?
- What are the acceptable RPO and RTO values?
- Can the app survive transient disconnects and retries?
- Is session state externalized?

**Summary**

Use Azure SQL failover groups when you want managed geo-redundancy with a single writer and clear failover semantics. Use Cosmos DB multi-region writes when the workload truly needs global write distribution and can handle conflict resolution and consistency tradeoffs. Use Front Door or Traffic Manager to route clients to healthy regions, and design the application to be stateless, retry-safe, and cache-aware.
