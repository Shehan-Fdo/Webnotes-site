## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Replication and latency design overview**

Replication is the mechanism that copies data or state across locations so your solution can survive failures and serve users closer to where they are. Latency design is the discipline of placing compute, data, and network paths so the user experience stays responsive while still meeting availability and recovery objectives.

For AZ-305, you usually choose among three broad patterns:

- **Synchronous replication** for low data loss and strong consistency, usually within a region or availability zone boundary.
- **Asynchronous replication** for cross-region resilience, where some replication lag is acceptable.
- **Application-level replication** for state that the platform does not replicate for you, such as sessions, caches, or custom data stores.

A good design starts with business requirements:

- **RPO** (Recovery Point Objective): how much data loss is acceptable.
- **RTO** (Recovery Time Objective): how long recovery can take.
- **Latency budget**: how much delay users can tolerate for reads, writes, and failover.

**Core design tradeoffs**

Replication improves resilience, but it can increase write latency, operational complexity, and cost. You should design for the smallest failure domain that still meets the requirement.

- **Tighter consistency** usually means higher write latency.
- **More replicas** usually means better availability but more cost and more failover complexity.
- **Cross-region replication** improves disaster recovery but introduces lag and possible data loss during failover.
- **Active-active designs** reduce failover time but require conflict handling and careful routing.

A practical way to think about it is:

- Use **zones** for datacenter-level resilience inside a region.
- Use **regions** for disaster recovery and geographic resilience.
- Use **application patterns** when the service itself does not provide the needed replication model.

**Azure resilience building blocks**

Azure gives you several layers of protection, and they solve different problems.

- **Availability Zones** protect against datacenter failures within a region.
- **Region pairs** help with disaster recovery planning and platform recovery sequencing.
- **Geo-redundant services** replicate data to another region.
- **Traffic routing services** shift user traffic when a region becomes unavailable.
- **Backup** protects against deletion, corruption, and ransomware, but it is not the same as replication.

> Backup and replication are not interchangeable. Replication keeps a secondary copy ready for failover; backup keeps point-in-time restore data for recovery from corruption, accidental deletion, or long-term retention needs.

**Replication versus backup**

This distinction is common in design questions.

- **Replication**
  - Keeps data available in another location.
  - Supports failover or read access in some services.
  - Usually follows a near-real-time or scheduled copy model.
  - Does not protect well against logical corruption if the corruption is replicated.

- **Backup**
  - Stores restore points or snapshots.
  - Supports point-in-time recovery.
  - Is better for accidental deletion, malware, and application bugs.
  - Does not provide immediate service continuity.

A strong architecture often uses both:

- Replication for continuity.
- Backup for recovery from bad data.

**Storage replication and failover**

Azure Storage offers redundancy options that balance durability, latency, and recovery.

- **LRS** (Locally Redundant Storage)
  - Stores multiple copies within a single storage scale unit in the primary region.
  - Protects against local hardware failure.
  - Lowest cost and lowest geographic resilience.

- **ZRS** (Zone-Redundant Storage)
  - Replicates data synchronously across availability zones in the same region.
  - Improves resilience to zone failure.
  - Useful when the region supports zones and the storage type supports ZRS.

- **GRS** (Geo-Redundant Storage)
  - Replicates data asynchronously to a paired secondary region.
  - Primary region remains writable.
  - Secondary is not normally readable unless you use the read-access variant.

- **RA-GRS** (Read-Access Geo-Redundant Storage)
  - Same as GRS, but the secondary endpoint is readable.
  - Useful for read-heavy workloads or DR validation.
  - Reads from the secondary are eventually consistent.

- **GZRS / RA-GZRS**
  - Combine zone redundancy in the primary region with geo-replication to a secondary region.
  - Useful when you want both zone resilience and geo resilience.

Important behavior for geo-redundant storage:

- Replication to the secondary region is asynchronous, so the secondary can lag behind the primary.
- Customer-initiated account failover promotes the secondary to primary.
- After failover, the old primary is no longer writable until re-replication completes.
- Failover can lose the most recent writes that had not replicated yet.
- DNS and service endpoints continue to use the storage account name, but the underlying primary location changes.

A simple mental model:

```text
Primary region writes -> async replication -> secondary region
        |                                   |
        |-- normal writes                   |-- optional read access with RA-GRS
        |
        +-- customer-initiated failover --> secondary becomes new primary
```

**Storage account failover considerations**

When you design for storage failover, think beyond the storage account itself.

- **Application retry logic** must handle transient 5xx and DNS propagation delays.
- **Client-side caching** may keep stale endpoints or stale data.
- **Read-after-write expectations** can break if you read from a geo-secondary.
- **Failover is not a zero-loss operation** for geo-redundant storage because of replication lag.
- **Reversal is not immediate**; the old primary must resynchronize before it can become a healthy secondary again.

Use geo-redundant storage when:

- You need built-in storage-level DR.
- You can tolerate asynchronous replication.
- Your application can handle eventual consistency for secondary reads.

Use backup instead when:

- You need point-in-time restore.
- You need protection from logical corruption.
- You need long retention or legal hold.

**Azure SQL replication and zone design**

Azure SQL has several resilience options, and the right choice depends on the service tier and the recovery objective.

- **Zone redundancy**
  - Available in supported tiers and regions.
  - Keeps database replicas across availability zones.
  - Helps survive a zone outage without a regional failover.
  - Does not replace geo-disaster recovery.

- **Failover groups**
  - Provide managed geo-replication for Azure SQL Database and SQL Managed Instance.
  - Support automatic or manual failover between primary and secondary servers.
  - Expose a read-write listener and a read-only listener.
  - Help you route application connections without changing every database endpoint manually.

- **Active geo-replication**
  - Replicates a database to readable secondary replicas.
  - Useful for read scale and DR.
  - Requires application awareness for failover and routing.

A common connection pattern for failover groups is:

- Use the **read-write listener** for transactional writes.
- Use the **read-only listener** for reporting or read scale.
- Configure the application to reconnect on failover and to retry transient errors.

Example connection string pattern:

```text
Server=tcp:<failover-group-name>.database.windows.net,1433;
Database=<db-name>;
Authentication=Active Directory Managed Identity;
Encrypt=True;
TrustServerCertificate=False;
Connection Timeout=30;
```

Practical guidance:

- Keep the application connection string pointed at the listener, not the physical server.
- Use retry policies with exponential backoff.
- Test failover to confirm DNS, firewall, and identity behavior.
- Validate whether your tier supports zone redundancy before assuming it is available.

> Zone redundancy improves availability inside a region, but it does not make a database globally resilient. You still need geo-replication or backup for regional disaster recovery.

**Azure SQL failover group behavior**

Failover groups are a common AZ-305 topic because they combine data replication with application routing.

- The **primary listener** routes writes to the current primary.
- The **secondary listener** can serve read-only traffic when configured.
- Automatic failover can be enabled for supported configurations.
- Manual failover is useful for planned maintenance or DR testing.
- Failover group endpoints abstract the current primary location, which reduces application changes.

Design implications:

- Use failover groups when you want managed geo-failover with minimal app changes.
- Use active geo-replication when you need more control over readable secondaries.
- Use zone redundancy when the main risk is a zone outage rather than a region outage.

**Azure Cosmos DB replication and consistency**

Cosmos DB is designed for global distribution, but you still need to choose the right replication and consistency model.

Key concepts:

- **Preferred regions** define the order Cosmos DB uses for failover and write routing.
- **Failover priority** determines which region becomes primary if the current write region fails.
- **Multi-region writes** allow writes in more than one region, which reduces write latency for distributed users but introduces conflict resolution concerns.
- **Consistency levels** control the tradeoff between latency, availability, and staleness.

Cosmos DB consistency options:

- **Strong**
  - Highest consistency.
  - Highest latency and lowest write flexibility.
  - Not always suitable for globally distributed low-latency apps.

- **Bounded staleness**
  - Guarantees reads lag by a bounded amount.
  - Good compromise for many globally distributed workloads.

- **Session**
  - Common default for user-centric applications.
  - Gives read-your-writes consistency within a session token.
  - Balances latency and usability well.

- **Consistent prefix**
  - Preserves write order.
  - Allows some staleness.

- **Eventual**
  - Lowest consistency guarantee.
  - Lowest latency and highest flexibility.

Design guidance:

- Use **session consistency** for most interactive applications.
- Use **strong consistency** only when the business requirement justifies the latency cost.
- Use **multi-region writes** only when you can handle conflict resolution and understand the write semantics.
- Use **preferred regions** to keep users close to a write region and to control failover order.

Example CLI pattern for a globally distributed account is version-dependent, so validate the current Azure CLI syntax before using it in production. The design intent is what matters:

```bash
az cosmosdb create \
  --name <account-name> \
  --resource-group <rg-name> \
  --locations regionName=<region1> failoverPriority=0 isZoneRedundant=False \
  --locations regionName=<region2> failoverPriority=1 isZoneRedundant=False
```

Use this only as a conceptual reference; Cosmos DB API-specific options and zone redundancy support vary by account type and region.

**Azure Database for PostgreSQL and MySQL replication**

Managed PostgreSQL and MySQL services also appear in design scenarios, especially when you need regional resilience without managing the database engine yourself.

- **Azure Database for PostgreSQL**
  - Supports read replicas in some deployment models and regions.
  - Supports high availability options depending on the service tier and deployment type.
  - Geo-replication and failover behavior depend on whether you use flexible server, single server legacy, or a specific HA configuration.

- **Azure Database for MySQL**
  - Supports high availability and read replica patterns depending on the service offering.
  - Geo-redundant DR patterns may require application-level failover or managed HA features.
  - Region and tier support vary.

Design guidance:

- Check the exact deployment model before assuming replica or failover support.
- Use read replicas for read scaling and reporting.
- Use HA for local fault tolerance.
- Use backups and restore for corruption recovery.
- Plan application reconnection logic because database endpoints can change during failover.

**Azure Site Recovery for VM replication**

Azure Site Recovery (ASR) is the main Azure service for orchestrating VM replication and disaster recovery.

What ASR does:

- Replicates Azure VMs or on-premises VMs to a recovery region.
- Orchestrates failover and failback.
- Supports **recovery plans** to coordinate multi-tier application recovery.
- Integrates with network mapping and test failover.

What ASR does not do:

- It is not a backup product.
- It does not replace application-level data consistency planning.
- It does not automatically solve database replication for every workload.

Key ASR concepts:

- **Recovery plan**
  - Defines the order in which VMs start.
  - Can include scripts, manual actions, and app grouping.
  - Useful for multi-tier apps where the database must start before the web tier.

- **Network mapping**
  - Maps source virtual networks to recovery virtual networks.
  - Ensures recovered VMs land in the correct subnet and security boundary.

- **Test failover**
  - Runs a failover in an isolated network.
  - Lets you validate recovery without impacting production.

- **Failback**
  - Returns workloads to the primary site after recovery.
  - Requires planning for data synchronization and cutover.

A practical recovery plan often looks like this:

1. Recover the database tier first.
2. Recover shared services such as identity or messaging if needed.
3. Recover the application tier.
4. Recover the web or presentation tier.
5. Run validation scripts and health checks.

Example recovery plan design:

```text
Recovery Plan: OrderProcessing-DR
- Group 1: SQL VM / database tier
- Group 2: App service VM / API tier
- Group 3: Web front end
- Action: run post-failover script to update DNS or app settings
```

For ASR, prefer portal-driven configuration and recovery plans over memorizing CLI syntax, because provider-specific CLI parameters can vary by workload and version.

**Traffic routing and latency-aware failover**

Replication alone does not move users to the right place. You also need traffic management.

- **Azure Traffic Manager**
  - DNS-based routing.
  - Supports priority, weighted, performance, geographic, and subnet routing.
  - Useful for global failover and latency-based routing.
  - DNS caching means failover is not instant.

- **Azure Front Door**
  - Global HTTP(S) load balancing and application delivery.
  - Better for web apps and APIs that need fast failover and edge acceleration.
  - Can terminate TLS, apply WAF policies, and route by path or header.

- **Azure Load Balancer**
  - Regional Layer 4 load balancing.
  - Useful inside a region or across zone-aware deployments.
  - Not a global failover service by itself.

Traffic Manager example pattern:

```text
User DNS query -> Traffic Manager -> closest healthy endpoint
```

Front Door example pattern:

```text
User -> nearest edge POP -> healthy origin in preferred region
```

Design guidance:

- Use **Traffic Manager** when DNS-based global routing is enough.
- Use **Front Door** when you need faster failover and HTTP-aware routing.
- Use **Load Balancer** for regional traffic distribution and zone-aware backends.
- Combine global routing with regional load balancing for layered resilience.

**Latency design principles**

Latency is not just network distance. It is the sum of network hops, serialization, queuing, storage access, and application processing.

Main contributors:

- **Geographic distance**
  - Longer physical distance increases round-trip time.
  - Cross-region writes are slower than in-region writes.

- **Network path**
  - Internet routing is variable.
  - Private connectivity can reduce jitter and improve predictability.

- **Service topology**
  - More hops mean more latency.
  - Cross-zone traffic is usually lower latency than cross-region traffic.

- **Consistency model**
  - Stronger consistency often requires more coordination.
  - Coordination increases write latency.

- **Client behavior**
  - Retries, timeouts, and connection pooling affect perceived latency.

Practical design rules:

- Place compute near the data when write latency matters.
- Place read replicas near users when read latency matters.
- Avoid chatty cross-region calls in the request path.
- Use caching for read-heavy, latency-sensitive data.
- Keep synchronous dependencies inside the same region when possible.

**Synchronous versus asynchronous replication**

This comparison appears often in design questions.

| Aspect | Synchronous replication | Asynchronous replication |
|---|---|---|
| Data safety | Lower RPO, often near-zero within the failure domain | Non-zero RPO due to lag |
| Write latency | Higher | Lower |
| Failover readiness | Immediate or near-immediate | Secondary may lag behind |
| Typical use | Zone resilience, HA databases | Geo-DR, cross-region copies |
| Complexity | Higher coordination cost | Easier to scale geographically |

Use synchronous replication when:

- You need strong consistency.
- You can tolerate the latency overhead.
- The replicas are close enough to keep performance acceptable.

Use asynchronous replication when:

- You need geographic separation.
- You want lower write latency.
- Some data loss during failover is acceptable.

**Application-level replication patterns**

Not every stateful component has built-in replication that fits your needs. You often need to design the application around the platform.

- **Session state**
  - Prefer stateless web tiers when possible.
  - Store sessions in a distributed cache or database if you need persistence.
  - Avoid in-memory session state on a single VM for multi-region apps.

- **Cache replication**
  - Use cache as a performance layer, not as the system of record.
  - Rebuild cache after failover if needed.
  - Expect cache warm-up after regional recovery.

- **Message queues**
  - Use durable messaging for decoupling.
  - Replicate or rehydrate queues carefully because duplicate processing can occur.
  - Design idempotent consumers.

- **File and object data**
  - Use geo-redundant storage when supported.
  - Consider application-level copy jobs for special retention or transformation needs.

- **Stateless services**
  - Prefer horizontal scale and redeployment over stateful failover.
  - Store state in managed services that already support replication.

A resilient web architecture often looks like this:

```text
Users
  -> Front Door
    -> Region A web/API
      -> Azure SQL failover group
      -> Redis cache
      -> Storage account with GZRS
    -> Region B web/API
      -> Secondary database replica
      -> Warm standby cache
```

**Networking and latency considerations**

Network design affects both user experience and failover behavior.

- **ExpressRoute**
  - Provides private connectivity to Azure.
  - Can improve predictability and reduce internet variability.
  - Does not eliminate latency from geographic distance.

- **VPN**
  - Easier to deploy but usually less predictable than ExpressRoute.
  - Suitable for smaller or less latency-sensitive environments.

- **Cross-region traffic**
  - Adds latency and can add egress cost.
  - Should be minimized in the request path.

- **Azure Load Balancer**
  - Useful for distributing traffic across VMs or zones.
  - Supports health probes for failover within a region.

- **DNS**
  - Critical for failover routing.
  - TTL (time to live) values affect how quickly clients see changes.
  - Lower TTL can speed failover but increases DNS query volume.

Design tips:

- Keep synchronous database calls in the same region as the app tier.
- Use asynchronous messaging for cross-region workflows.
- Set realistic timeouts for remote calls.
- Test failover under real network conditions, not just in a lab.

**Designing for region pairs**

Azure region pairs matter because Microsoft uses them for platform updates and disaster recovery planning.

- Paired regions are geographically separated.
- Platform updates are typically staggered across pairs.
- Some services prefer paired-region replication targets.
- Pairing helps with recovery planning, but it is not a guarantee of automatic failover for your workload.

Use region pairs to:

- Choose a secondary region with geographic separation.
- Align DR planning with Azure’s operational model.
- Reduce correlated failure risk.

Do not assume:

- The paired region is always the best latency choice.
- Every service automatically fails over to the paired region.
- Pairing alone satisfies your RTO or RPO.

**Practical architecture scenarios**

Scenario 1: Global e-commerce site

- Use **Front Door** for global HTTP routing.
- Deploy web/API tiers in two regions.
- Use **Azure SQL failover groups** for transactional data.
- Use **GZRS** for product images and static assets.
- Use **Redis** for session and cache data.
- Use **ASR** only for any remaining VM-based legacy components.

Why this works:

- Users connect to the nearest healthy region.
- Writes stay close to the primary database.
- Static content survives regional failure.
- Legacy VMs have orchestrated recovery.

Scenario 2: Analytics dashboard with heavy reads

- Use **Cosmos DB** or read replicas where appropriate.
- Place read replicas near users.
- Use **RA-GRS** or equivalent for read-heavy object data.
- Accept eventual consistency for non-critical dashboard metrics.

Why this works:

- Read latency matters more than immediate write visibility.
- Some staleness is acceptable for analytics.

Scenario 3: Internal line-of-business app

- Use a single primary region with **zone redundancy**.
- Add **backup** for corruption recovery.
- Add **ASR** for VM-based components.
- Use a secondary region only if the business requires regional DR.

Why this works:

- It avoids unnecessary multi-region complexity.
- It still protects against zone and data loss scenarios.

**CLI and configuration examples**

Use CLI examples carefully and validate them against the current Azure CLI version before production use.

Storage account with geo-redundancy:

```bash
az storage account create \
  --name mystorageacct \
  --resource-group rg-app \
  --location eastus \
  --sku Standard_GRS \
  --kind StorageV2
```

Azure SQL failover group conceptually:

```bash
az sql failover-group create \
  --name fg-app \
  --server primary-sql \
  --partner-server secondary-sql \
  --resource-group rg-data \
  --failover-policy Automatic \
  --grace-period 60
```

Traffic Manager profile with priority routing:

```bash
az network traffic-manager profile create \
  --name tm-global \
  --resource-group rg-net \
  --routing-method Priority \
  --unique-dns-name tm-global-app
```

Add endpoints conceptually:

```bash
az network traffic-manager endpoint create \
  --name region-a \
  --profile-name tm-global \
  --resource-group rg-net \
  --type azureEndpoints \
  --target-resource-id /subscriptions/<sub>/resourceGroups/rg-app/providers/Microsoft.Web/sites/app-region-a \
  --endpoint-status Enabled \
  --priority 1
```

> Endpoint syntax can vary by CLI version and resource type. Always verify the exact command for App Service, VM, or external endpoint targets.

ASR test failover is usually best configured in the portal, but the operational flow is:

1. Map source and recovery networks.
2. Create a recovery plan.
3. Run test failover into an isolated network.
4. Validate application startup order and dependencies.
5. Clean up the test environment.
6. Schedule a real failover drill.

**Exam-focused decision cues**

When you read a scenario, look for these clues:

- **“No data loss”** usually points to synchronous replication or zone redundancy.
- **“Survive a regional outage”** usually points to geo-replication, failover groups, or ASR.
- **“Fast global web failover”** usually points to Front Door or Traffic Manager.
- **“Protect against accidental deletion”** usually points to backup, not replication.
- **“Read scale across regions”** usually points to readable replicas or RA-GRS-like patterns.
- **“Minimize app changes”** usually points to managed listeners and DNS abstraction.
- **“Low latency for users worldwide”** usually points to regional placement, caching, and edge routing.

**Common pitfalls**

- Assuming replication equals backup.
- Assuming zone redundancy protects against regional disasters.
- Assuming geo-redundant storage failover is instant and lossless.
- Assuming all services support the same redundancy options in every region.
- Assuming multi-region writes are always better than single-write-region designs.
- Assuming DNS-based failover is immediate.
- Assuming a secondary replica is always writable.

**Quick comparison summary**

| Service | Primary resilience feature | Best use case | Key limitation |
|---|---|---|---|
| Azure Storage | LRS, ZRS, GRS, RA-GRS, GZRS | Durable object/file data | Geo failover is asynchronous |
| Azure SQL | Zone redundancy, failover groups, geo-replication | Transactional databases | Tier and region support vary |
| Cosmos DB | Multi-region distribution, failover priorities | Global low-latency apps | Consistency and conflict tradeoffs |
| PostgreSQL/MySQL | HA, replicas, backup/restore | Managed relational workloads | Feature support depends on deployment model |
| Azure Site Recovery | VM replication and recovery plans | VM-based DR | Not a backup replacement |
| Traffic Manager / Front Door | Global traffic routing | User failover and latency routing | DNS or edge behavior affects failover speed |

**Study takeaway**

Designing for replication and latency is about matching the service to the failure mode. Use synchronous replication and zone redundancy when you need low RPO inside a region. Use asynchronous geo-replication when you need disaster recovery across regions. Use traffic routing to move users to healthy endpoints. Use backup to recover from bad data. The best AZ-305 answer is usually the one that balances availability, consistency, cost, and operational complexity instead of maximizing only one of them.
