## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Lab overview**

A globally distributed Azure Cosmos DB design focuses on low latency, high availability, and predictable application behavior across regions. In this lab, you typically validate the core architecture choices that matter for the AZ-305 exam: region placement, replication, consistency, partitioning, throughput, security, and failover.

Cosmos DB is a **multi-model, globally distributed** database service. For exam purposes, separate the design into two layers:

- **Account-level settings**
  - Regions, failover policy, consistency, networking, backup, and identity.
- **Container-level settings**
  - Partition key, indexing policy, throughput, TTL, and conflict behavior for supported APIs.

That distinction matters because many design decisions are made once at the account level, while performance and query behavior are usually tuned at the container level.

**Core architecture goals**

A good global Cosmos DB design balances these goals:

- **Low latency** by placing data close to users.
- **High availability** by replicating across regions.
- **Predictable cost** by sizing request units and choosing the right throughput mode.
- **Operational safety** through backup, restore, and controlled failover.
- **Secure access** using identity, network isolation, and least privilege.

**Create the account with global distribution in mind**

When you create the account, choose the API that matches the workload. For the lab, the Azure Cosmos DB for NoSQL API is the most common choice because it supports flexible JSON documents and global distribution patterns.

A typical deployment flow is:

1. Create the resource group.
2. Create the Cosmos DB account.
3. Add regions.
4. Configure failover priority.
5. Create the database and container.
6. Set partitioning, indexing, and throughput.
7. Validate client routing and failover behavior.

Example Azure CLI commands:

```bash
az group create \
  --name rg-cosmos-global-lab \
  --location eastus

az cosmosdb create \
  --name cosmosgloballab123 \
  --resource-group rg-cosmos-global-lab \
  --locations regionName=eastus failoverPriority=0 isZoneRedundant=false \
  --locations regionName=westus2 failoverPriority=1 isZoneRedundant=false \
  --default-consistency-level Session \
  --enable-multiple-write-locations false
```

The account-level consistency setting applies to all databases and containers in the account unless you use a weaker consistency override at the client level. For exam scenarios, remember that stronger consistency generally increases latency and can reduce write throughput.

**Regions, replication, and client routing**

Cosmos DB replicates data to each configured region. Reads can be served from the nearest available region, while writes go to the current write region unless you enable multi-region writes.

Client applications should prefer the closest region to reduce latency. SDKs can use preferred regions so the client routes requests intelligently.

.NET example using Microsoft Entra ID authentication with token credentials:

```csharp
using Azure.Identity;
using Microsoft.Azure.Cosmos;

var endpoint = Environment.GetEnvironmentVariable("COSMOS_ENDPOINT");
var client = new CosmosClient(endpoint, new DefaultAzureCredential());

var database = client.GetDatabase("appdb");
var container = database.GetContainer("orders");
```

JavaScript example using Microsoft Entra ID authentication:

```javascript
const { CosmosClient } = require("@azure/cosmos");
const { DefaultAzureCredential } = require("@azure/identity");

const endpoint = process.env.COSMOS_ENDPOINT;
const credential = new DefaultAzureCredential();

const client = new CosmosClient({
  endpoint,
  aadCredentials: credential
});
```

If you use preferred regions, validate that the SDK is configured to read from the nearest region first. This is especially important in active-passive designs where you want reads to stay local but writes to remain centralized.

**Throughput planning: RU/s, autoscale, and replication impact**

Throughput in Cosmos DB is measured in **request units per second** (RU/s). RU cost depends on item size, indexing, query complexity, and consistency level. Global replication also affects design because writes must be replicated to each region, which increases the operational cost of write-heavy workloads.

Key throughput choices:

- **Manual throughput**
  - You provision a fixed RU/s value.
  - Best when traffic is predictable.
- **Autoscale throughput**
  - Cosmos DB scales within a configured maximum.
  - Best for variable or spiky workloads.
- **Shared throughput database**
  - Multiple containers share the database throughput.
  - Useful for small workloads or many low-traffic containers.
- **Dedicated container throughput**
  - A single container gets its own RU/s allocation.
  - Better for critical or high-traffic workloads.

Practical guidance:

- Start with a conservative RU/s estimate and validate with load testing.
- Watch for `429 Too Many Requests` responses, which indicate throttling.
- Increase throughput or improve query efficiency if throttling persists.
- Remember that multi-region writes and replication increase the amount of work the service performs.

Example CLI for creating a database and container with throughput:

```bash
az cosmosdb sql database create \
  --account-name cosmosgloballab123 \
  --resource-group rg-cosmos-global-lab \
  --name appdb \
  --throughput 400

az cosmosdb sql container create \
  --account-name cosmosgloballab123 \
  --resource-group rg-cosmos-global-lab \
  --database-name appdb \
  --name orders \
  --partition-key-path "/tenantId" \
  --throughput 400
```

If you prefer autoscale, use the autoscale-specific parameters supported by your CLI version. The exact flags can vary, so verify with `az cosmosdb sql container create -h` before running the lab.

**Partitioning strategy**

Partitioning is one of the most important design decisions in Cosmos DB. The partition key determines how data is distributed and how efficiently queries can be served.

Good partition key characteristics:

- High cardinality.
- Even distribution of writes and reads.
- Stable value that does not change often.
- Commonly used in query filters.

Poor partition key characteristics:

- Low cardinality, such as `country` for a global app with only a few values.
- Hot partitions caused by one tenant or one device generating most traffic.
- Frequently changing values that force data movement or awkward updates.

Example document shape:

```json
{
  "id": "order-1001",
  "tenantId": "contoso",
  "customerId": "c-7781",
  "status": "Pending",
  "region": "eastus",
  "total": 149.95,
  "createdAt": "2026-04-26T12:00:00Z"
}
```

In this example, `/tenantId` is a reasonable partition key if each tenant generates a balanced workload. If one tenant is much larger than the others, you may need a more granular key or a synthetic key such as `tenantId-customerId`.

Partitioning affects:

- **Scalability** because data is spread across physical partitions.
- **Query performance** because partition-targeted queries are cheaper.
- **Cost** because poorly targeted queries consume more RU/s.
- **Availability** because a balanced partition map helps the service distribute load.

**Indexing policy and query efficiency**

Cosmos DB indexes JSON data by default. This is convenient for development, but indexing strategy has a direct impact on RU consumption and write cost.

Default behavior:

- Most properties are indexed automatically.
- Queries are easier to write because you do not need to predefine every field.
- Writes cost more because the service maintains indexes.

When to customize indexing:

- Exclude large or rarely queried paths to reduce write RU cost.
- Add composite indexes for common multi-property `ORDER BY` or filter patterns.
- Use range indexes where appropriate for numeric and string comparisons.

Example custom indexing policy:

```json
{
  "indexingMode": "consistent",
  "automatic": true,
  "includedPaths": [
    { "path": "/*" }
  ],
  "excludedPaths": [
    { "path": "/largeBlob/*" },
    { "path": "/auditTrail/*" }
  ],
  "compositeIndexes": [
    [
      { "path": "/tenantId", "order": "ascending" },
      { "path": "/createdAt", "order": "descending" }
    ]
  ]
}
```

Practical guidance:

- Keep the default policy unless you have measured a cost or performance problem.
- Exclude large subdocuments that you never query.
- Use composite indexes for repeated query patterns.
- Re-test RU consumption after any indexing change.

**Consistency levels**

Cosmos DB supports multiple consistency levels. The choice affects latency, availability, and how quickly replicas converge.

Common consistency levels:

- **Strong**
  - Highest consistency.
  - Lowest latency tolerance and least flexibility.
- **Bounded staleness**
  - Reads lag behind writes by a bounded amount.
  - Useful when you want predictable staleness.
- **Session**
  - Guarantees read-your-writes within a session.
  - Common default for application workloads.
- **Consistent prefix**
  - Reads never see out-of-order writes.
- **Eventual**
  - Highest availability and lowest coordination overhead.
  - Accepts temporary inconsistency.

For most globally distributed applications, **Session** is a practical default because it balances user experience and performance. Use stronger consistency only when the business requirement truly needs it.

**Conflict resolution and multi-write design**

If you enable multi-region writes, conflicts can occur when the same item is updated in more than one region before replication converges.

Conflict resolution options depend on the API and configuration, but the main concepts are:

- **Last writer wins**
  - Uses a user-defined numeric or timestamp property to decide the winner.
  - Do not confuse this with the system `_ts` property; the conflict path is a user-selected field.
- **Custom conflict resolution**
  - Uses a merge procedure or API-specific conflict handling model.
  - The exact implementation varies by API and feature set.

A practical last-writer-wins example uses a field such as `updatedAt`:

```json
{
  "id": "order-1001",
  "tenantId": "contoso",
  "status": "Shipped",
  "updatedAt": 1714132800
}
```

If two regions update the same document, the version with the higher `updatedAt` value wins, assuming that field is configured as the conflict resolution path.

To validate conflict behavior in a lab, you can:

1. Write the same document from two regions.
2. Force a near-simultaneous update.
3. Inspect the final document version.
4. Review conflict metadata or the conflict feed where supported.

> Conflict handling is API-specific. Verify the exact behavior for the Cosmos DB API you are using before relying on a lab assumption.

**Backup, restore, and disaster recovery**

Global distribution is not a substitute for backup. Replication protects availability, but it does not protect against accidental deletes, bad writes, or application bugs.

Important backup concepts:

- **Periodic backup**
  - Backups are taken on a schedule.
  - Restore granularity is limited by the backup interval.
- **Continuous backup**
  - Supports point-in-time restore within the retention window.
  - Better for recovery from accidental deletion or corruption.
- **Point-in-time restore**
  - Restores the account or container state to a specific time.
  - Useful for operational recovery and testing rollback scenarios.

Design guidance:

- Use continuous backup when recovery objectives are strict.
- Test restore procedures before you need them.
- Document which resources are restored together and which are recreated manually.
- Treat failover and restore as different recovery tools:
  - **Failover** handles regional outages.
  - **Restore** handles data loss or corruption.

**Security architecture**

A secure Cosmos DB solution uses identity, network controls, and least privilege.

Recommended security controls:

- **Microsoft Entra ID authentication**
  - Prefer identity-based access over account keys where possible.
- **Role-based access control**
  - Assign the minimum required data-plane permissions.
- **Private endpoints**
  - Keep traffic on the Azure backbone and avoid public exposure.
- **Disable public network access** when the application can reach the account privately.
- **Managed identities**
  - Use them for Azure-hosted applications instead of storing secrets.

Example role assignment for data-plane access:

```bash
az cosmosdb sql role assignment create \
  --account-name cosmosgloballab123 \
  --resource-group rg-cosmos-global-lab \
  --scope "/" \
  --principal-id <managed-identity-object-id> \
  --role-definition-id <cosmos-data-contributor-role-id>
```

Private endpoint creation pattern:

```bash
az network private-endpoint create \
  --name pe-cosmos-global-lab \
  --resource-group rg-cosmos-global-lab \
  --vnet-name vnet-app \
  --subnet subnet-private-endpoints \
  --private-connection-resource-id $(az cosmosdb show -g rg-cosmos-global-lab -n cosmosgloballab123 --query id -o tsv) \
  --group-ids Sql \
  --connection-name cosmos-pe-conn
```

DNS validation matters because private endpoints depend on name resolution. Confirm that the Cosmos DB account name resolves to the private IP from the application subnet.

Example checks:

```bash
nslookup cosmosgloballab123.documents.azure.com
```

If the private endpoint is configured correctly, the name should resolve through the private DNS zone associated with the VNet.

**Monitoring and diagnostics**

Monitoring helps you validate whether the design meets latency, availability, and cost goals.

Track these signals:

- **Request units consumed**
- **Throttled requests**
- **Latency by region**
- **Availability and failover events**
- **Storage growth**
- **Index utilization**
- **Authentication and authorization failures**

Use Azure Monitor metrics and diagnostic logs. The exact log table depends on how diagnostics are configured, so treat Kusto queries as environment-specific rather than universal.

Example metric-oriented Kusto query pattern:

```kusto
AzureMetrics
| where ResourceProvider == "MICROSOFT.DOCUMENTDB"
| where MetricName in ("TotalRequestUnits", "ThrottledRequests", "DataUsage")
| summarize avg(Total) by MetricName, bin(TimeGenerated, 1h)
```

If your workspace uses resource-specific tables, adjust the query accordingly. Always confirm the table names in your Log Analytics workspace before relying on a query in production.

**Failover testing**

Failover is a core part of the lab because it proves that the application can survive a regional outage.

Planned failover is the safest way to test. It lets you move the write region in a controlled manner.

Validation steps:

1. Confirm the current write region and failover priorities.
2. Ensure the application uses the Cosmos DB endpoint, not a hardcoded regional endpoint.
3. Trigger planned failover.
4. Wait for the operation to complete.
5. Reconnect the application and verify writes succeed in the new write region.
6. Confirm reads continue from the nearest available region.

CLI example for failover inspection:

```bash
az cosmosdb show \
  --name cosmosgloballab123 \
  --resource-group rg-cosmos-global-lab \
  --query "locations[].{region:name, priority:failoverPriority, isZoneRedundant:isZoneRedundant}" \
  -o table
```

Use the output to confirm the configured region order. Do not assume the current write region solely from the account metadata; verify the failover operation completed and then test application writes.

**Troubleshooting common lab issues**

Common problems usually fall into a few categories.

- **403 authorization errors**
  - Check whether the app uses the correct authentication model.
  - Verify the managed identity or service principal has the correct Cosmos DB data-plane role.
  - Confirm the account is not blocking the request through network rules.
- **429 throttling**
  - Increase RU/s or autoscale max RU/s.
  - Reduce query fan-out.
  - Improve partition key choice.
  - Add or tune indexes.
- **DNS or private endpoint failures**
  - Confirm the private DNS zone is linked to the VNet.
  - Verify the application subnet uses the correct DNS resolver.
  - Check that public network access is disabled only after private connectivity works.
- **Failover test does not behave as expected**
  - Confirm the account has multiple regions.
  - Verify the write region changed after the planned failover completed.
  - Ensure the client reconnects and does not cache stale endpoints.
- **Query performance is poor**
  - Check whether the query is partition-aware.
  - Review indexing policy.
  - Avoid cross-partition scans when possible.

**Practical lab validation checklist**

Use this checklist to confirm the solution is complete:

- The account exists in at least two regions.
- The failover priority is configured correctly.
- The application uses the Cosmos DB account endpoint.
- The client prefers the nearest read region where supported.
- The container has a sensible partition key.
- Throughput is sized appropriately for the workload.
- Indexing policy matches the query pattern.
- Security uses managed identity or another secret-minimizing approach.
- Private endpoint and DNS resolution work if network isolation is required.
- Backup and restore requirements are documented.
- Planned failover is tested and verified.

**Exam focus summary**

For AZ-305, focus on the design tradeoffs rather than memorizing one command. You should be able to explain:

- Why you choose a region pair or multiple regions.
- How consistency affects latency and availability.
- How partitioning influences scale and query cost.
- When to use manual throughput versus autoscale.
- Why indexing policy matters for RU consumption.
- How security changes when you move from keys to managed identity.
- How failover differs from restore.

A strong answer usually connects the business requirement to the Cosmos DB feature that satisfies it. For example, if the requirement is global low-latency reads with regional resilience, you combine multi-region replication, session consistency, preferred regions, and tested failover.
