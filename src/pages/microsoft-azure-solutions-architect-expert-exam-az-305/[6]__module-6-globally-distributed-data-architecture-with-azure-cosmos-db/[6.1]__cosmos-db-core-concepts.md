## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Azure Cosmos DB overview**

Azure Cosmos DB is a globally distributed, multi-model NoSQL database service designed for low-latency access, elastic scale, and tunable consistency. For AZ-305, you need to understand how its design choices affect availability, performance, cost, and data modeling.

Cosmos DB is not a single database engine in the traditional sense. It is a platform with multiple APIs, each optimized for different access patterns:

- **SQL API** for document-style JSON data and the most common exam scenarios.
- **MongoDB API** for MongoDB-compatible applications.
- **Cassandra API** for wide-column workloads.
- **Gremlin API** for graph workloads.
- **Table API** for key-value style access.

The architectural decisions you make usually revolve around four questions:

- How do you partition the data?
- What consistency level do you need?
- How do you distribute data across regions?
- How do you index and secure the data?

**Data model and container basics**

Cosmos DB stores data in containers. In the SQL API, a container is the unit of throughput, indexing, and partitioning. A container holds JSON items, and each item has an `id` plus a partition key value.

A practical mental model is:

- **Account**: the top-level Cosmos DB resource.
- **Database**: a logical grouping of containers.
- **Container**: the scale and query boundary.
- **Item**: a single JSON document or record.

Example item:

```json
{
  "id": "order-1001",
  "tenantId": "contoso",
  "customerId": "c-42",
  "status": "Pending",
  "orderDate": "2026-04-26T10:15:00Z",
  "total": 149.95
}
```

For the SQL API, the partition key is a property path such as `/tenantId`. That choice strongly affects scale, cost, and query efficiency.

**Account-level concepts: regions, write regions, and failover**

Cosmos DB accounts can span multiple Azure regions. You choose where the account is available, which region is the write region, and how failover behaves.

Key concepts:

- **Read regions** serve local reads and can reduce latency.
- **Write region** accepts writes for the account.
- **Failover priority** determines the order in which regions become the write region during failover.
- **Manual failover** lets you promote a region intentionally.
- **Automatic failover** lets Azure switch to the next region based on configured priorities.

In practice, you configure regions on the account and assign priorities. A common pattern is one primary write region and multiple read regions for global applications.

Example Azure CLI pattern:

```bash
az cosmosdb create \
  --name contoso-cosmos \
  --resource-group rg-data \
  --locations regionName=eastus failoverPriority=0 isZoneRedundant=False \
  --locations regionName=westus failoverPriority=1 isZoneRedundant=False
```

You should know that:

- **Strong consistency** is supported only when the account has a single write region.
- Multi-region writes are not supported with strong consistency.
- If you need active-active writes, you typically choose session, bounded staleness, or eventual consistency depending on the workload.

**Consistency levels**

Cosmos DB offers tunable consistency. This is one of its most important exam topics because it directly affects latency, availability, and read freshness.

The main consistency levels are:

- **Strong**: reads always see the latest committed write.
- **Bounded staleness**: reads lag by a bounded number of versions or time.
- **Session**: reads are consistent within a client session.
- **Consistent prefix**: reads never see out-of-order writes.
- **Eventual**: reads may lag but converge over time.

A useful way to think about them:

- Use **strong** when correctness is more important than latency and you can live with a single write region.
- Use **session** for most user-facing applications because it gives good performance and predictable read-your-writes behavior.
- Use **bounded staleness** when you want a controlled lag across regions.
- Use **eventual** for highly distributed, read-heavy, tolerant workloads.

Example scenario:

- An e-commerce cart service often uses **session consistency** so the user sees their own updates immediately.
- A financial ledger may require **strong consistency** if the design can tolerate a single write region.
- A global product catalog may use **eventual** or **consistent prefix** if freshness is less critical than availability and latency.

> Strong consistency is a hard architectural constraint: if you need multi-region writes, you must choose a weaker consistency model.

**Partitioning and partition key design**

Partitioning is the core scaling mechanism in Cosmos DB. The partition key determines how data is distributed and how queries are routed.

Why partitioning matters:

- It enables horizontal scale.
- It affects request unit consumption.
- It determines whether a query can be served efficiently.
- It influences hot partition risk.

Important partitioning concepts:

- **Logical partition**: all items with the same partition key value.
- **Physical partition**: an internal storage and throughput unit managed by Cosmos DB.
- **Partition key cardinality**: the number of distinct partition key values.
- **Hot partition**: a partition receiving disproportionate traffic.
- **Immutable partition key**: you cannot change the partition key of an existing item; changing the partition key usually means copying data to a new item.

The often-cited 20 GB limit applies to a logical partition in the standard partition key model. In practice, you should not design around that limit as if it were the only concern. A poor partition key can create hot partitions long before storage becomes a problem.

Design guidance:

- Choose a partition key with high cardinality.
- Choose a key that spreads writes evenly.
- Choose a key that matches your most common query filters.
- Avoid low-cardinality keys such as `country`, `status`, or `type` unless the workload is tiny or carefully sharded.
- Avoid keys that create write hotspots, such as a timestamp prefix that funnels all new writes into one value.

Common patterns:

- **Tenant-based partitioning**: `/tenantId` for SaaS applications.
- **User-based partitioning**: `/userId` for user-centric workloads.
- **Synthetic partition key**: combine fields into a derived key when no single field distributes well.

Example synthetic key:

```json
{
  "id": "order-1001",
  "tenantId": "contoso",
  "orderId": "1001",
  "partitionKey": "contoso|2026-04"
}
```

Hierarchical partition keys can help model workloads with multiple dimensions and can reduce hot partition risk by spreading data more flexibly than a single key. For exam purposes, remember that partitioning strategy should be chosen based on access patterns, not just storage limits.

**Throughput and request units**

Cosmos DB uses **request units** (RUs) as a normalized measure of cost for reads, writes, and queries. RU consumption depends on item size, indexing, query complexity, and consistency.

Practical RU guidance:

- A point read of a small item is usually much cheaper than a query.
- Writes cost more when indexing is heavier or items are larger.
- Cross-partition queries consume more RUs than single-partition queries.
- Stored procedures and transactional batch operations execute within a single logical partition.

Throughput can be provisioned in two main ways:

- **Manual throughput**: you set a fixed RU/s value.
- **Autoscale throughput**: Cosmos DB automatically scales within a range based on demand.

Autoscale concepts:

- You configure a maximum RU/s.
- Cosmos DB scales up and down automatically within the allowed range.
- Autoscale is useful for spiky or unpredictable workloads.
- The minimum effective throughput is a fraction of the max, so it is not free capacity.

Example autoscale CLI pattern:

```bash
az cosmosdb sql container throughput update \
  --account-name contoso-cosmos \
  --resource-group rg-data \
  --database-name appdb \
  --name orders \
  --max-throughput 4000
```

Exam-relevant guidance:

- Use autoscale for variable traffic and seasonal spikes.
- Use manual throughput for steady, predictable workloads.
- Place throughput at the database level when multiple containers share a workload pattern.
- Place throughput at the container level when one container needs isolated performance.

**Querying and transaction behavior**

Cosmos DB is optimized for point reads and partition-aware queries. Query design matters because it directly affects latency and RU cost.

Key query behaviors:

- A **point read** uses the item `id` and partition key value.
- A **single-partition query** targets one logical partition and is efficient.
- A **cross-partition query** scans multiple partitions and costs more.
- Queries that filter on the partition key are usually much cheaper than those that do not.

Example point read in the SQL API:

```sql
SELECT * FROM c
WHERE c.id = "order-1001" AND c.tenantId = "contoso"
```

Example cross-partition query:

```sql
SELECT c.id, c.status
FROM c
WHERE c.status = "Pending"
```

If `status` is not the partition key, this query may fan out across partitions.

Transaction behavior:

- Cosmos DB supports ACID-like transactions within a single logical partition.
- **Transactional batch** lets you group multiple operations on items that share the same partition key.
- **Stored procedures** also execute within a single logical partition in the SQL API.

Example transactional batch use case:

- Create an order.
- Create order lines.
- Update inventory reservation.

This works well if all items share the same partition key, such as `tenantId` or `orderId`.

Example batch pattern:

```csharp
TransactionalBatch batch = container.CreateTransactionalBatch(
    new PartitionKey("contoso"))
    .CreateItem(new { id = "order-1001", tenantId = "contoso", total = 149.95 })
    .CreateItem(new { id = "line-1", tenantId = "contoso", orderId = "order-1001" });

TransactionalBatchResponse response = await batch.ExecuteAsync();
```

**Indexing**

Cosmos DB indexes data automatically by default in the SQL API, which is a major difference from many relational systems. This makes queries flexible, but indexing policy still matters for cost and performance.

Core indexing concepts:

- **Automatic indexing** means new items are indexed without manual intervention.
- **Included paths** define what gets indexed.
- **Excluded paths** reduce indexing overhead for large or rarely queried fields.
- **Composite indexes** improve queries with multiple `ORDER BY` or filter patterns.
- **Indexing mode** in the SQL API is typically `consistent` or `none`; `lazy` exists historically but is not the common modern choice for exam scenarios.

Default behavior in the SQL API:

- By default, most paths are indexed.
- This supports ad hoc querying.
- It can increase write RU cost because every write updates the index.

Example indexing policy:

```json
{
  "indexingMode": "consistent",
  "automatic": true,
  "includedPaths": [
    {
      "path": "/*"
    }
  ],
  "excludedPaths": [
    {
      "path": "/largeBlob/*"
    }
  ],
  "compositeIndexes": [
    [
      { "path": "/tenantId", "order": "ascending" },
      { "path": "/orderDate", "order": "descending" }
    ]
  ]
}
```

When to use excluded paths:

- Large blob-like properties that you never query.
- Deeply nested payloads that add indexing cost without query value.
- High-write workloads where reducing index overhead matters.

When to use composite indexes:

- Queries with multiple sort keys.
- Queries that combine filters and `ORDER BY`.
- Scenarios where you need efficient ordered retrieval across a partition.

Exam-style rule of thumb:

- Use excluded paths to reduce write cost.
- Use composite indexes to optimize specific query shapes.
- Do not assume indexing is free; it is a tradeoff between query flexibility and write cost.

**Unique keys**

Unique key policies enforce uniqueness within a logical partition. They do not provide global uniqueness across the entire container.

Important points:

- You define one or more unique key paths.
- Cosmos DB enforces uniqueness for items that share the same partition key value.
- The partition key itself is not part of the unique key definition.
- If you need global uniqueness, you must design for it explicitly, often by using a dedicated partitioning strategy or a separate lookup container.

Example unique key policy:

```json
{
  "uniqueKeyPolicy": {
    "uniqueKeys": [
      {
        "paths": ["/email"]
      }
    ]
  }
}
```

This means:

- Two items with the same `email` cannot exist in the same logical partition.
- The same `email` can exist in a different partition if the partition key value differs.

Practical use cases:

- Prevent duplicate usernames within a tenant.
- Prevent duplicate order numbers within a customer partition.
- Enforce uniqueness for a business identifier scoped to one tenant.

**Security and access control**

Cosmos DB security has both management-plane and data-plane dimensions. This distinction matters for AZ-305.

Management-plane access:

- Controlled through Azure RBAC on the Cosmos DB account resource.
- Used to create accounts, configure regions, set throughput, and manage failover.
- Typical roles include owner, contributor, and Cosmos DB management roles.

Data-plane access:

- Controls who can read and write data inside the database.
- Can use keys, resource tokens, or Azure AD-based role assignments depending on API and feature support.
- Data-plane RBAC is important when you want least privilege for application identities.

Practical distinction:

- A platform engineer may need management-plane rights to configure the account.
- An application managed identity may need only data-plane rights to read and write items.

Security best practices:

- Prefer Azure AD and managed identities where supported.
- Rotate keys if you use them.
- Use private endpoints to keep traffic off the public internet.
- Restrict firewall access to trusted networks.
- Separate operational access from application access.

Example CLI to assign a role at the account scope:

```bash
az role assignment create \
  --assignee <principal-id> \
  --role "Cosmos DB Built-in Data Contributor" \
  --scope /subscriptions/<sub-id>/resourceGroups/rg-data/providers/Microsoft.DocumentDB/databaseAccounts/contoso-cosmos
```

**Backup and restore**

Cosmos DB supports periodic and continuous backup models. The choice affects restore granularity, retention, and recovery objectives.

Periodic backup:

- Backups are taken at intervals.
- Retention is limited by the configured policy.
- Restore is typically to a point in time within the backup window, but not to arbitrary seconds.
- It is usually simpler and lower cost.

Continuous backup:

- Provides point-in-time restore within the retention window.
- Better for accidental deletes or bad writes where you need finer recovery.
- Typically has more operational flexibility than periodic backup.
- Restore scope and retention depend on the service capabilities and account configuration.

Operational guidance:

- Use periodic backup when you need basic disaster recovery and can tolerate coarser restore points.
- Use continuous backup when recovery precision matters.
- Remember that backup is not a substitute for application-level safeguards.

Common exam scenario:

- A developer accidentally deletes a set of documents.
- Continuous backup is the better answer if you need to restore to just before the deletion.
- Periodic backup may be acceptable if the recovery window and granularity are sufficient.

**Analytical store and Synapse Link**

Cosmos DB analytical store is optimized for analytical queries without impacting the transactional workload. Synapse Link connects Cosmos DB to Azure Synapse Analytics for near-real-time analytics.

Why use it:

- You want to run analytical queries on operational data.
- You want to avoid ETL-heavy pipelines for every reporting use case.
- You need separation between transactional and analytical workloads.

Core boundaries:

- Analytical workloads are not a replacement for transactional queries.
- Not every API or feature combination supports the same analytical capabilities.
- You should verify support for the specific API and account configuration before designing around it.

When to choose it:

- Operational reporting on recent data.
- Near-real-time dashboards.
- Hybrid transactional and analytical processing patterns where the analytical side should not burden the OLTP path.

When not to choose it:

- If you only need simple transactional reads.
- If your workload requires complex relational joins better served by a relational warehouse.
- If the data model is not suitable for analytical projection.

**Monitoring and operations**

Operational monitoring helps you detect hot partitions, RU pressure, latency spikes, and failover issues.

Useful Azure Monitor signals include:

- Request units consumed.
- Throttled requests.
- Server-side latency.
- Data usage.
- Index utilization.
- Availability.
- Replication lag or region health indicators where applicable.

Diagnostic logs can help with:

- Control-plane operations.
- Data-plane request analysis.
- Troubleshooting throttling and authorization failures.

Alerting considerations:

- Alert on sustained throttling, not just brief spikes.
- Alert on RU consumption approaching provisioned limits.
- Alert on latency increases in critical regions.
- Alert on failover events and region health changes.

Practical troubleshooting pattern:

1. Check whether the workload is hitting a hot partition.
2. Review RU consumption and throttling metrics.
3. Inspect query patterns for cross-partition scans.
4. Validate indexing policy and partition key choice.
5. Confirm region health and failover status if latency is regional.

**Architecture patterns and exam scenarios**

A good Cosmos DB design aligns partitioning, consistency, and region strategy with the business requirement.

Common patterns:

- **Global catalog**: use multiple read regions, session or eventual consistency, and a partition key that spreads product data evenly.
- **SaaS tenant store**: use tenant-based partitioning, unique keys scoped per tenant, and autoscale for bursty tenant activity.
- **Order processing**: use a partition key that keeps order header and lines together, then use transactional batch for atomic writes.
- **IoT ingestion**: use a partition key that avoids hot partitions, often by combining device identity with a time bucket.

Example design tradeoff:

- If you choose `/tenantId` as the partition key, tenant-scoped queries are efficient.
- If one tenant becomes very large, that tenant can become a hot partition.
- If you choose a synthetic key like `tenantId|shard`, you improve distribution but make some queries more complex.

Another common exam tradeoff:

- Strong consistency gives the cleanest read semantics.
- Session consistency is usually the best balance for globally distributed apps.
- Multi-region writes improve availability and write locality, but they require a weaker consistency model.

**CLI and configuration examples**

Create a Cosmos DB account with multiple regions:

```bash
az cosmosdb create \
  --name contoso-cosmos \
  --resource-group rg-data \
  --kind GlobalDocumentDB \
  --locations regionName=eastus failoverPriority=0 \
  --locations regionName=westus failoverPriority=1
```

Create a SQL database and container with throughput:

```bash
az cosmosdb sql database create \
  --account-name contoso-cosmos \
  --resource-group rg-data \
  --name appdb

az cosmosdb sql container create \
  --account-name contoso-cosmos \
  --resource-group rg-data \
  --database-name appdb \
  --name orders \
  --partition-key-path "/tenantId" \
  --throughput 400
```

Create a container with autoscale:

```bash
az cosmosdb sql container create \
  --account-name contoso-cosmos \
  --resource-group rg-data \
  --database-name appdb \
  --name orders \
  --partition-key-path "/tenantId" \
  --max-throughput 4000
```

**Quick exam reminders**

- **Partition key** choice is one of the most important design decisions.
- **Point reads** are cheaper than queries.
- **Cross-partition queries** cost more and scale less efficiently.
- **Unique keys** are scoped to a logical partition, not the whole container.
- **Strong consistency** requires a single write region.
- **Autoscale** helps with unpredictable demand.
- **Indexing** improves query flexibility but increases write cost.
- **Backup model** choice affects restore precision and retention.
- **Data-plane RBAC** is different from management-plane RBAC.
- **Monitoring** should focus on RU, latency, throttling, and partition hotspots.
