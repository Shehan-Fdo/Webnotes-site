## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Azure Migrate overview**

Azure Migrate is the central Microsoft toolset for discovering on-premises servers, assessing readiness and cost, visualizing application dependencies, and orchestrating migration to Azure. In architecture design, you use it to reduce guesswork before a move and to choose the right target sizing, network design, and migration path.

Azure Migrate is not a single feature. It is a project-based hub that can include multiple tools, such as:

- **Server Assessment** for sizing and readiness
- **Server Migration** for replication and cutover
- **Dependency visualization** for application mapping
- Workload-specific assessment tools for databases, web apps, and virtual desktops

Use Azure Migrate when you need a structured migration plan for servers and related workloads. Use other Azure-native tools when the workload is not a server migration problem, such as database modernization or app refactoring.

**Core capabilities and when to use them**

Azure Migrate helps you answer four common architecture questions:

- What do I have?
- Is it compatible with Azure?
- What depends on what?
- How do I move it with minimal risk?

A practical migration program often uses the tools in this order:

1. Discover servers and applications.
2. Group related machines into an application boundary.
3. Assess readiness and right-size targets.
4. Plan the migration wave.
5. Replicate, test, and cut over.

**Discovery methods and prerequisites**

Discovery is the foundation for assessment and migration. Azure Migrate supports different source environments, and the discovery method depends on the platform.

- **VMware discovery** typically uses an Azure Migrate appliance that connects to vCenter Server and collects inventory and performance data.
- **Hyper-V discovery** uses an appliance deployed in the Hyper-V environment and connected to the Hyper-V host or cluster.
- **Physical server discovery** uses an appliance that can discover physical machines on the network.
- **Import-based discovery** can be used in some scenarios when you already have inventory data and want to upload it for assessment.

The appliance is usually a lightweight virtual machine or server component that you deploy in the source environment. It is then registered to an Azure Migrate project and configured to collect data.

Typical prerequisites include:

- An Azure subscription with permission to create and manage the Azure Migrate project
- Network connectivity from the appliance to Azure endpoints
- Source platform credentials for inventory collection
- Appropriate permissions in the source environment, such as read access to vCenter or Hyper-V
- Firewall rules that allow the appliance to reach the source platform and Azure services

For exam purposes, remember that discovery is not passive. You must deploy and register the appliance, then configure the source connection and data collection scope.

**Appliance registration and permissions**

The appliance must be registered to the Azure Migrate project before it can send discovery data. Registration links the appliance to the project and enables data upload.

A common workflow is:

1. Create an Azure Migrate project in the Azure portal.
2. Choose the source environment, such as VMware, Hyper-V, or physical servers.
3. Download and deploy the appliance.
4. Generate a project key or registration key in the portal.
5. Register the appliance with that key.
6. Provide source credentials and start discovery.

Permissions vary by source type, but the principle is the same: the appliance needs enough access to enumerate machines, collect configuration details, and gather performance metrics.

- **VMware**: read access to vCenter inventory is usually required.
- **Hyper-V**: access to the host or cluster inventory is required.
- **Physical servers**: network reachability and credentials for agent installation or inventory collection are required, depending on the scenario.

> Be careful not to confuse Azure Migrate project permissions with source platform permissions. Azure permissions let you create and manage the project; source permissions let the appliance discover machines.

**Network and firewall considerations**

Discovery and migration both depend on network reachability. The appliance must communicate with Azure and, in many cases, with the source environment.

Common considerations include:

- Outbound HTTPS connectivity to Azure services
- DNS resolution for Azure endpoints
- Access to vCenter, Hyper-V, or physical server management interfaces
- No blocking of appliance-to-source traffic by internal firewalls
- Proxy configuration if the environment requires internet egress through a proxy

For migration, additional connectivity is needed for replication traffic and for the target Azure environment. Plan for:

- Replication traffic from source to Azure
- Access to Azure Storage and Azure Compute endpoints
- Connectivity for test migration and cutover operations
- Post-migration access to the new VM through the target network design

**Server Assessment**

Server Assessment is the main planning tool in Azure Migrate. It estimates readiness, target sizing, and cost. It is used before migration to reduce surprises.

Assessment outputs usually include:

- Azure readiness status
- Recommended VM size
- Monthly cost estimate
- Disk and storage recommendations
- Confidence rating based on collected data
- Compatibility issues that may block or complicate migration

Assessments can be based on:

- **Performance-based sizing**, which uses collected utilization data
- **As-is sizing**, which maps source configuration more directly to Azure targets

Performance-based sizing is usually preferred when you want to right-size for actual usage rather than source peak configuration.

**Readiness analysis**

Readiness analysis tells you whether a machine can move to Azure as-is, needs changes, or is not ready. The exact checks depend on the source platform and target configuration, but the assessment commonly evaluates:

- Operating system support
- Boot type and firmware compatibility
- Disk layout and supported disk count
- CPU and memory fit for Azure VM sizes
- Network adapter and IP configuration considerations
- VM generation compatibility for Azure targets
- Special hardware or driver dependencies

A readiness result is usually expressed as one of these categories:

- **Ready for Azure**
- **Ready with conditions**
- **Not ready for Azure**
- **Readiness unknown** when data is insufficient

Examples of readiness findings by source type:

- **VMware**
  - Unsupported guest OS version
  - VM uses a configuration that does not map cleanly to a supported Azure VM size
  - Disk count or disk type requires target adjustment
  - UEFI/BIOS and VM generation mapping needs review
- **Hyper-V**
  - Generation 1 versus Generation 2 mapping must be validated
  - Unsupported OS or missing integration components can affect migration
  - Dynamic memory settings may influence sizing decisions
- **Physical servers**
  - Unsupported OS or kernel version
  - Boot configuration that requires remediation before replication
  - Hardware-dependent drivers that will not transfer cleanly to Azure

The assessment is a planning aid, not a guarantee. You still validate the target design, application behavior, and operational dependencies before cutover.

**How to interpret readiness results**

Use readiness results to decide whether you can migrate directly, need remediation, or should modernize first.

- **Ready for Azure**
  - The machine can likely be migrated with minimal changes.
  - You still validate application behavior and network access.
- **Ready with conditions**
  - The machine can move, but you must address a specific issue first.
  - Common examples include resizing, OS upgrade, or disk changes.
- **Not ready for Azure**
  - The machine has a blocker that prevents direct migration.
  - You may need to remediate the source system or choose a different migration path.
- **Unknown**
  - Discovery data is incomplete.
  - Increase the collection window or fix appliance connectivity.

A good exam habit is to separate platform compatibility from application compatibility. A VM can be technically ready while the application still fails because of hardcoded IPs, missing ports, or unsupported middleware.

**Dependency visualization**

Dependency visualization helps you understand which servers communicate with each other so you can migrate related systems together. This reduces the risk of moving a database without its app tier or missing a hidden service dependency.

Azure Migrate dependency mapping is typically enabled by installing a dependency agent on the discovered machines and connecting them to the Azure Migrate project. The agent collects process and network connection data, which Azure Migrate uses to build a dependency map.

Common components include:

- The Azure Migrate appliance for discovery
- A dependency agent on each machine you want to map
- In some scenarios, a Log Analytics workspace or connected monitoring component, depending on the implementation path and source type

The dependency map usually shows:

- Inbound and outbound connections
- Process-level relationships
- Server-to-server communication paths
- Application groupings that should be migrated together

Use dependency visualization when:

- You need to define migration waves
- You suspect hidden application dependencies
- You want to avoid partial migrations that break service chains
- You need evidence for application owner discussions

**Enabling dependency mapping**

A typical workflow is:

1. Discover the servers in Azure Migrate.
2. Select the machines you want to analyze.
3. Enable dependency mapping for those machines.
4. Install the dependency agent where required.
5. Wait for traffic collection to build a useful map.
6. Review the dependency graph and group machines into an application boundary.

The map becomes more useful after enough traffic has been observed. Short observation windows may miss infrequent connections, batch jobs, or scheduled integrations.

**Interpreting the dependency map**

When you review the map, look for:

- Core application tiers, such as web, app, and database
- Shared services, such as DNS, authentication, or file shares
- External dependencies, such as APIs or message brokers
- One-way versus two-way communication
- Unexpected connections that may indicate undocumented coupling

A dependency map is not just a diagram. It is a migration planning tool. Use it to decide:

- Which machines belong in the same wave
- Which systems need firewall changes before cutover
- Which services require coordinated downtime
- Which dependencies can be replaced with Azure-native services later

**Server Migration**

Azure Migrate: Server Migration is the migration execution tool. It supports replication, test migration, cutover, and post-migration validation. It is used after discovery and assessment, not instead of them.

The migration workflow usually includes:

1. Select the source machines in Azure Migrate.
2. Configure replication settings for the target subscription, resource group, virtual network, and VM size.
3. Start replication and wait for initial sync.
4. Run a test migration into an isolated network.
5. Validate the test VM.
6. Plan and execute cutover.
7. Clean up replication and finalize the migration.

Server Migration is for moving servers. It is not the same as disaster recovery design. If your goal is continuous replication for failover and failback, Azure Site Recovery is the more appropriate service.

**Replication setup**

Replication setup is where you define how the source machine will be protected and where it will land in Azure.

You typically configure:

- Target subscription
- Target resource group
- Target virtual network and subnet
- Target availability option, such as availability zone or availability set when applicable
- VM size and disk type
- Replication policy and recovery point settings

Example portal decisions you may need to make:

- Place production workloads in a landing zone subscription with guardrails
- Use a dedicated resource group per migration wave
- Map source subnets to target subnets to preserve application routing
- Choose zone-redundant placement for critical workloads when supported

A practical target configuration example might look like this:

```text
Target subscription: Prod-Landing-Zone
Target resource group: rg-migration-wave-1
Target virtual network: vnet-prod-apps
Target subnet: snet-app-tier
Target availability: Availability Zone 1
Target VM size: Standard_D4s_v5
```

**Test migration**

Test migration is a critical step and should not be skipped. It creates a test VM in Azure without affecting the source machine or the production cutover plan.

Use test migration to validate:

- Boot success
- Network connectivity
- DNS resolution
- Application startup
- Authentication and authorization
- Data access to dependent services

Best practice is to place the test VM in an isolated network or a non-production subnet so you can validate behavior without causing conflicts.

**Cutover and post-cutover validation**

Cutover is the point where you stop relying on the source machine and switch production traffic to the Azure VM.

A typical cutover sequence is:

1. Announce the maintenance window.
2. Stop or quiesce the source application.
3. Perform final replication synchronization.
4. Shut down or freeze the source machine as required by the migration plan.
5. Complete cutover in Azure Migrate.
6. Validate the migrated VM and application.
7. Update DNS, load balancers, or routing if needed.
8. Monitor logs, performance, and user access.

After cutover, validate:

- VM boots and stays healthy
- Application services start correctly
- Network security rules allow required traffic
- Storage and data paths work as expected
- Monitoring and backup are enabled
- Identity and access controls are correct

**Assessment versus migration versus dependency mapping**

These tools solve different problems and are often used together.

| Capability | Primary purpose | Typical output | Main limitation |
|---|---|---|---|
| Server Assessment | Plan readiness and sizing | Readiness, cost, VM size | Does not move workloads |
| Server Migration | Replicate and move servers | Test migration, cutover, migrated VM | Not a DR platform |
| Dependency visualization | Reveal application relationships | Dependency map | Needs enough traffic and agent coverage |
| Workload-specific assessment | Evaluate non-server workloads | Database or app-specific guidance | Not a general server migration tool |

Use the right tool for the right question. For example, do not use server assessment to decide database platform modernization, and do not use server migration to design an application dependency map.

**Practical portal workflow example**

A common exam-style scenario is a VMware estate migration.

1. Create an Azure Migrate project in the Azure portal.
2. Select VMware as the source environment.
3. Deploy and register the appliance.
4. Add vCenter credentials and start discovery.
5. Wait for inventory and performance data to accumulate.
6. Create a server assessment using performance-based sizing.
7. Review readiness issues and right-size recommendations.
8. Enable dependency mapping for the application servers.
9. Group the app, web, and database servers into a migration wave.
10. Configure server replication and run a test migration.
11. Validate the test VM in an isolated subnet.
12. Execute cutover during the approved window.

**CLI and PowerShell context**

Azure CLI and Azure PowerShell are useful for landing zone and target-environment checks, but they do not replace Azure Migrate discovery, assessment, or migration workflows.

Use them for supporting tasks such as:

- Creating or validating resource groups
- Checking available VM sizes in a region
- Preparing networking and identity resources
- Automating target environment setup

Examples:

```bash
az group create --name rg-migration-wave-1 --location eastus
az vm list-sizes --location eastus --output table
az network vnet create \
  --resource-group rg-migration-wave-1 \
  --name vnet-prod-apps \
  --address-prefix 10.20.0.0/16 \
  --subnet-name snet-app-tier \
  --subnet-prefix 10.20.1.0/24
```

```powershell
Get-AzVMSize -Location eastus
New-AzResourceGroup -Name rg-migration-wave-1 -Location eastus
```

These commands help you prepare the target environment, but they do not assess source machines or configure Azure Migrate projects.

**Landing zone and target design considerations**

Migration decisions should align with the target landing zone. Azure Migrate tells you what can move and what size it needs; your landing zone tells you where it should live and how it should be governed.

Consider:

- **Subscription design** for separation of production, nonproduction, and shared services
- **Resource group strategy** for lifecycle and ownership boundaries
- **Network topology** for hub-and-spoke or flat network patterns
- **Identity integration** with Microsoft Entra ID and role-based access control
- **Monitoring and backup** before production cutover
- **Governance** through policies, tags, and naming standards

Also consider Azure Hybrid Benefit when sizing Windows Server or SQL Server workloads, because licensing can materially change the cost estimate. For critical workloads, decide whether the target should use availability zones, availability sets, or another resiliency pattern based on the application’s tolerance for downtime and regional failure.

**Common exam pitfalls**

- Treating Azure Migrate as a single feature instead of a suite of tools
- Assuming discovery works without deploying and registering the appliance
- Confusing assessment with migration execution
- Using dependency mapping without enough agent coverage or observation time
- Ignoring network and firewall requirements between appliance, source, and Azure
- Assuming a readiness result guarantees application success
- Choosing Server Migration when the requirement is ongoing disaster recovery rather than migration
- Using generic Azure CLI commands as if they configure Azure Migrate itself

**Quick decision guide**

- Use **Server Assessment** when you need sizing, cost, and compatibility planning.
- Use **Dependency visualization** when you need to understand application relationships.
- Use **Server Migration** when you are ready to replicate and cut over servers.
- Use **workload-specific assessment tools** when the workload is not a general server migration problem.
- Use **Azure Site Recovery** when the requirement is disaster recovery, not a one-time migration.

**Key takeaways**

Azure Migrate is most effective when you treat it as a staged workflow: discover, assess, map dependencies, then migrate. The exam often tests whether you can choose the right tool, interpret readiness correctly, and plan a safe cutover with the right target design.

Focus on these ideas:

- Discovery requires an appliance and proper permissions.
- Assessment informs sizing and compatibility, but it does not move workloads.
- Dependency mapping helps you migrate application groups safely.
- Server Migration handles replication, test migration, and cutover.
- Landing zone design, network planning, and licensing choices affect the final architecture.
