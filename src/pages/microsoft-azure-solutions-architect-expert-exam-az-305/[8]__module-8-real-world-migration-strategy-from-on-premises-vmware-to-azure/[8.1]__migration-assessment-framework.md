## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Migration assessment framework**

A migration assessment framework helps you decide what to move, where to move it, and how much effort the move requires. In Azure architecture work, assessment comes before design: you identify dependencies, compatibility gaps, performance constraints, security requirements, and business priorities before selecting a migration path.

The framework usually answers four questions:

- What is the workload?
- Is it ready for Azure as-is, with changes, or not at all?
- Which Azure target fits best?
- What migration method minimizes risk and downtime?

A good assessment reduces surprises during cutover. It also helps you separate technical feasibility from business preference, which is critical in exam scenarios.

**Core assessment goals**

Assessment is not just inventory. It is a decision-making process that turns discovery data into a migration plan.

- **Discovery** identifies servers, applications, databases, and dependencies.
- **Readiness** checks whether the workload can run in Azure.
- **Sizing** estimates the right Azure SKU, storage, and network capacity.
- **Cost estimation** compares current spend with Azure consumption.
- **Risk analysis** highlights blockers such as unsupported OS versions, hard-coded IPs, or legacy authentication.
- **Migration planning** maps each workload to a target and method.

A practical framework often uses the common migration decision model:

- **Rehost**: move with minimal change.
- **Refactor**: make small code or configuration changes.
- **Rearchitect**: redesign parts of the solution.
- **Rebuild**: create a new solution on Azure.
- **Replace**: retire the workload and use SaaS or another service.

These choices are often called the “6 Rs” in migration planning, though exact naming varies by organization.

**Assessment inputs and outputs**

Assessment starts with facts and ends with recommendations. You should know what data you need and what decisions that data supports.

| Input | Why it matters | Typical output |
|---|---|---|
| Server inventory | Establishes scope | List of machines and owners |
| Application dependencies | Prevents broken cutovers | Dependency map |
| OS and runtime versions | Determines supportability | Readiness status |
| Performance metrics | Guides sizing | VM size or service tier |
| Database schema and features | Checks compatibility | Target database recommendation |
| Network topology | Identifies routing and firewall needs | Connectivity plan |
| Security and compliance requirements | Drives landing zone design | Control gaps and remediation tasks |

Assessment outputs should be actionable. A useful report does not just say “not ready”; it explains what must change and whether the change is worth it.

**Discovery and dependency mapping**

Discovery is the first technical step. You need to know what exists before you can assess it. In Azure migration projects, discovery often includes servers, installed software, services, ports, and communication patterns.

Dependency mapping is especially important because workloads rarely run in isolation. A web app may depend on a database, file share, identity provider, message queue, or on-premises API.

Common dependency questions include:

- Which systems does the workload call?
- Which systems call into it?
- Are dependencies synchronous or asynchronous?
- Are there hidden dependencies such as scheduled tasks, shared folders, or local certificates?
- Does the workload require low-latency access to on-premises resources?

A simple topology example:

```text
Users -> Web tier -> App tier -> SQL database
                 \-> Redis cache
App tier -> On-premises file share
App tier -> Identity provider
```

If you miss the file share or identity provider dependency, the migration may technically succeed but fail in production.

**Readiness assessment**

Readiness determines whether the workload can run in Azure with the current configuration or with manageable changes. This is where you identify blockers and classify effort.

Typical readiness checks include:

- **Operating system support**: confirm the OS version is supported by the chosen Azure target.
- **Application runtime support**: verify .NET, Java, PHP, Node.js, IIS, Apache, or other runtime compatibility.
- **Database compatibility**: check engine version, features, and unsupported objects.
- **Hardware assumptions**: look for local disks, GPU needs, or specialized drivers.
- **Identity integration**: validate domain join, Kerberos, LDAP, or certificate dependencies.
- **Network assumptions**: identify static IP dependencies, multicast, or broadcast requirements.
- **Storage behavior**: review IOPS, throughput, latency, and file system requirements.

Readiness findings usually fall into one of these categories:

- Ready as-is
- Ready with minor changes
- Ready after remediation
- Not suitable for the target

> Unsupported OS does not automatically mean “redesign the application.” It may mean you rehost to a supported VM, upgrade the OS, replace the workload, or choose a different target based on supportability and business constraints.

**Target selection framework**

Assessment should lead to a target recommendation. The target depends on workload type, compatibility, and desired level of change.

| Workload condition | Common target | Why |
|---|---|---|
| Needs full OS control | Azure Virtual Machines | Preserves server-level administration |
| Web app fits platform constraints | Azure App Service | Removes server management overhead |
| Database is compatible with managed service | Azure SQL Database or Managed Instance | Reduces operational burden |
| Needs lift-and-shift with minimal change | Azure Virtual Machines | Fastest path with least code change |
| Needs modernization and cloud-native features | PaaS or container platform | Improves agility and operations |
| Requires SaaS replacement | Microsoft or third-party SaaS | Eliminates custom maintenance |

Use the target that best matches the workload’s technical fit and business goals. Do not assume every app should move to App Service or every database should move to SQL Database. Compatibility and operational requirements decide the target.

**Common Azure migration targets**

Azure offers several common destinations, each with different tradeoffs.

- **Azure Virtual Machines**
  - Best for rehosting servers with minimal change.
  - Useful when you need OS-level control, custom software, or unsupported application patterns.
  - Often the safest first step for legacy workloads.

- **Azure App Service**
  - Best for web apps and APIs that fit supported runtimes and platform constraints.
  - Good when you want managed hosting without managing the underlying OS.
  - Suitable for many web workloads that can run directly on the platform without containerization.

- **Azure SQL Database**
  - Best for modernized relational workloads that fit a managed database model.
  - Works well when the application uses supported SQL features and can accept platform limits.

- **Azure SQL Managed Instance**
  - Best when you need broad SQL Server compatibility with managed service benefits.
  - Often a strong choice for lift-and-shift database migrations that need more compatibility than Azure SQL Database provides.

- **Azure Kubernetes Service (AKS)**
  - Best for containerized microservices and orchestrated workloads.
  - Requires more platform and operational maturity.

- **Azure Storage, Azure Files, and Azure NetApp Files**
  - Useful for file shares, object storage, and high-performance file workloads.
  - Often part of a broader migration rather than the primary target.

**Migration strategy selection**

Assessment also informs the migration method. The method should match the workload type and the acceptable downtime window.

- **Offline migration**
  - You stop the source workload, copy data, and cut over.
  - Use this when downtime is acceptable or the workload is small.

- **Online migration**
  - You replicate data or state while the source stays live.
  - Use this when downtime must be minimized.

- **Phased migration**
  - You move components in stages.
  - Use this for large systems with clear boundaries.

- **Pilot migration**
  - You migrate a small subset first.
  - Use this to validate assumptions before full rollout.

A practical assessment should recommend both the target and the migration style. For example, a legacy web app may be a rehost to Azure VMs with an offline cutover, while a database may require online replication to reduce downtime.

**Azure Migrate in the assessment framework**

Azure Migrate is the primary Azure service for discovery and assessment of servers and for planning server migrations. It helps you inventory machines, assess readiness, estimate sizing, and plan migration.

Typical uses include:

- Discovering on-premises servers
- Assessing VMware, Hyper-V, and physical servers
- Estimating Azure VM sizing
- Reviewing dependency relationships
- Planning server migration into Azure

A common workflow is:

1. Create an Azure Migrate project.
2. Deploy the Azure Migrate appliance or connector as required by the source environment.
3. Discover servers and collect performance data.
4. Run an assessment.
5. Review readiness, sizing, and cost estimates.
6. Remediate blockers.
7. Migrate using the selected migration method.

Example Azure CLI pattern for creating a project:

```bash
az group create -n rg-migration -l eastus

az migrate project create \
  --resource-group rg-migration \
  --location eastus \
  --project-name mig-project
```

Assessment data is only useful if you validate it. Discovery tools can miss transient dependencies, and performance samples may not reflect peak load if the collection window is too short.

**Database and application assessment considerations**

Database and application assessment often require specialized tools or deeper analysis than server discovery alone.

For databases, you typically evaluate:

- Engine version and edition
- Unsupported features
- Cross-database dependencies
- SQL Agent jobs, linked servers, and CLR usage
- Authentication model
- Backup and restore strategy
- Transaction throughput and latency requirements

For web and application workloads, you typically evaluate:

- Runtime compatibility
- Session state handling
- File system dependencies
- Background jobs
- Authentication and authorization flows
- External API dependencies
- Configuration management

A database may be a good candidate for Azure SQL Database, Azure SQL Managed Instance, or a VM-hosted SQL Server depending on feature usage and compatibility. A web app may fit App Service directly if the runtime and dependencies are supported; it does not need to be containerized first.

**Sizing and performance analysis**

Sizing is one of the most important outputs of assessment. You want to avoid both underprovisioning and overspending.

Key sizing inputs include:

- CPU utilization
- Memory usage
- Disk IOPS and throughput
- Network throughput
- Peak versus average demand
- Growth trends

When sizing Azure VMs, you usually compare observed performance to Azure VM families and select a size with headroom. For managed services, you map workload demand to service tiers and compute units.

Example performance review logic:

- If CPU is consistently low but memory is high, choose a memory-optimized size.
- If disk latency is the bottleneck, review storage type and IOPS limits.
- If network traffic is bursty, confirm the VM or service tier can handle peak throughput.

A simple Azure VM sizing example:

```bash
az vm list-sizes --location eastus -o table
```

This helps you compare available sizes before finalizing a recommendation.

**Cost estimation**

Assessment should include cost, but cost is not the only decision factor. A cheaper target may be operationally unsuitable, and a more expensive target may reduce risk or downtime.

Cost analysis usually considers:

- Compute
- Storage
- Network egress
- Backup
- Monitoring
- Licensing
- Support and operations overhead

Use cost estimates to compare options such as:

- Rehost to Azure VMs
- Move to App Service
- Move to managed database services
- Keep part of the workload on-premises temporarily

A useful assessment report explains both the estimated Azure cost and the cost of remediation. Sometimes a workload is technically ready but economically poor to modernize immediately.

**Security, identity, and compliance review**

Migration assessment must include security and compliance because target design affects access control and governance.

Review:

- Identity integration with Microsoft Entra ID
- Domain join requirements
- Privileged access model
- Network segmentation
- Encryption at rest and in transit
- Key management
- Logging and audit requirements
- Regulatory constraints

If a workload handles regulated data, you may need to design the landing zone, network controls, and policy assignments before migration. Assessment should identify these requirements early so they do not become cutover blockers.

**Landing zone implications**

Assessment often feeds landing zone design. A landing zone is the Azure environment where workloads land with the right governance, networking, and security controls.

Assessment findings can influence:

- Subscription structure
- Resource group design
- Virtual network and subnet layout
- DNS and name resolution
- Firewall and routing rules
- Policy and RBAC assignments
- Monitoring and alerting

If the workload needs private connectivity to on-premises systems, the assessment should confirm whether VPN or ExpressRoute is required. If the workload needs centralized logging, the landing zone should include Log Analytics and diagnostic settings.

**Practical assessment workflow**

A repeatable workflow helps you avoid missed steps.

1. Identify the workload scope and business owner.
2. Discover servers, apps, databases, and dependencies.
3. Collect performance and configuration data.
4. Classify readiness and compatibility issues.
5. Choose the most suitable Azure target.
6. Estimate sizing, cost, and downtime.
7. Define remediation tasks and migration waves.
8. Validate the plan with stakeholders.
9. Execute a pilot or proof of concept if risk is high.
10. Finalize the migration runbook.

This sequence works well because it moves from facts to decisions to execution. Skipping discovery or readiness usually creates rework later.

**Example assessment decision patterns**

A few common patterns appear often in exam questions and real projects.

- **Legacy line-of-business app on Windows Server**
  - Likely target: Azure VM
  - Reason: minimal change, preserve server dependencies

- **ASP.NET web app with supported runtime and no OS-level dependencies**
  - Likely target: Azure App Service
  - Reason: managed hosting fits the workload

- **SQL Server workload using features not supported by Azure SQL Database**
  - Likely target: Azure SQL Managed Instance or Azure VM-hosted SQL Server
  - Reason: compatibility requirements drive the choice

- **Physical server with moderate dependencies and acceptable downtime**
  - Likely target: Azure VM via migration tooling
  - Reason: rehost is simplest and fastest

- **Workload with major business process changes required**
  - Likely target: refactor, rearchitect, or replace
  - Reason: migration alone does not solve the underlying fit problem

**Assessment pitfalls**

Common mistakes can invalidate an otherwise good migration plan.

- Treating discovery as a one-time task instead of an ongoing process
- Ignoring hidden dependencies such as scheduled jobs or shared storage
- Assuming all web apps need containers before App Service
- Assuming all databases should move to the same managed service
- Using average performance only and ignoring peak demand
- Overlooking identity, DNS, and certificate dependencies
- Choosing a target before validating compatibility
- Confusing disaster recovery replication with migration planning

> Azure Site Recovery is primarily a replication and failover service for disaster recovery scenarios. It can support migration-style cutovers for supported servers, but it is not the same thing as a full assessment platform.

**Useful commands and checks**

You often combine assessment tools with basic Azure CLI checks during planning.

```bash
# Create a resource group for migration work
az group create -n rg-migration -l eastus

# Review available VM sizes in a region
az vm list-sizes --location eastus -o table

# Inspect storage account options for a target design
az storage account list-skus -l eastus -o table
```

For network planning, you may also validate address space and DNS dependencies in your design documents rather than in CLI output alone.

**Exam focus summary**

For AZ-305, focus on the decision logic behind assessment rather than memorizing tool names alone.

- Discovery finds what exists.
- Readiness determines what can move.
- Sizing determines how big the target should be.
- Dependency mapping prevents broken migrations.
- Target selection depends on compatibility and business goals.
- Azure Migrate is central for server discovery and assessment.
- App Service, Azure VMs, Azure SQL Database, and Azure SQL Managed Instance are common targets, but the right choice depends on workload fit.
- Migration planning must include security, networking, cost, and downtime.

A strong answer in an exam scenario explains why a target fits, what blockers exist, and what remediation is needed before cutover.
