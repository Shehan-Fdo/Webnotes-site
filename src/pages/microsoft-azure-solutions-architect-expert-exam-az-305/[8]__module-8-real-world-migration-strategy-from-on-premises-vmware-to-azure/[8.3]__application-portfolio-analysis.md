## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Application portfolio analysis overview**

Application portfolio analysis is the structured process of inventorying applications, understanding their business and technical characteristics, and deciding what to do with each workload. In Azure architecture work, this analysis helps you choose the right migration or modernization path instead of treating every app as a generic lift-and-shift candidate.

The goal is to reduce risk and maximize value. You use portfolio analysis to identify which applications are strategic, which are obsolete, which are expensive to run, and which are good candidates for cloud-native redesign. The output usually feeds migration waves, modernization roadmaps, and business cases.

- **Inventory** all applications, including hidden or shadow IT where possible.
- **Classify** each app by business criticality, technical fit, and dependency profile.
- **Prioritize** based on value, risk, effort, and timing.
- **Decide** whether to retain, retire, rehost, refactor, rearchitect, or replace.

**Why portfolio analysis matters**

Without portfolio analysis, migration programs often fail because they move the wrong workloads first, underestimate dependencies, or modernize systems that should simply be retired. A good analysis gives you a fact-based view of the estate so you can align technical decisions with business outcomes.

It also helps you avoid overengineering. Some applications are best left as-is for a period of time, while others benefit from immediate modernization. The analysis should support incremental decision-making, not force a single migration pattern across the entire estate.

**Core assessment dimensions**

A useful portfolio assessment looks at both business and technical factors. You need enough detail to make a decision, but not so much that discovery becomes endless.

- **Business value**: revenue impact, customer impact, regulatory importance, and strategic relevance.
- **Technical health**: code quality, platform age, supportability, and operational stability.
- **Dependency complexity**: upstream and downstream systems, data flows, identity integration, and network coupling.
- **Cloud readiness**: compatibility with Azure services, operating system support, and deployment model.
- **Operational cost**: infrastructure spend, licensing, support effort, and maintenance burden.
- **Risk**: security exposure, compliance constraints, and outage sensitivity.

A practical approach is to score each application across these dimensions and then group them into action categories. The scoring model does not need to be perfect; it needs to be consistent enough to compare applications against each other.

**Common portfolio outcomes**

Application portfolio analysis usually leads to one of several standard decisions. These are often described with the “6 Rs” model.

| Decision | Meaning | Typical use case |
|---|---|---|
| **Retire** | Decommission the application | No longer needed, duplicate capability, or replaced by another system |
| **Retain** | Keep it where it is for now | Too risky or costly to move immediately |
| **Rehost** | Move with minimal changes | Good candidate for infrastructure migration to Azure VMs |
| **Replatform** | Make limited changes to use managed services | Example: move to Azure App Service or Azure SQL with minimal code change |
| **Refactor** | Modify code to improve cloud fit | Improve scalability, deployment, or resilience |
| **Rearchitect** | Redesign the application architecture | Needed for major scalability, availability, or agility gains |
| **Replace** | Swap with a SaaS or packaged solution | Business capability is better delivered by a commercial product |

> The exact terminology varies by organization. Some teams use five Rs, others use six or seven. The important part is that each application gets a clear disposition.

**Discovery and inventory**

Discovery is the starting point. You cannot analyze what you have not identified. In practice, discovery combines interviews, CMDB (Configuration Management Database) data, infrastructure scans, network analysis, and application owner input.

Useful discovery artifacts include:

- Application name and owner
- Business function and user base
- Hosting platform and environment
- Runtime stack and version
- Data stores and integration points
- Authentication and authorization dependencies
- SLA (Service Level Agreement) and support model
- Compliance or residency requirements

A simple inventory can be maintained in a spreadsheet or a portfolio tool. For larger estates, teams often use discovery platforms and then enrich the data manually with business context.

**Dependency analysis**

Dependencies are often the hardest part of portfolio analysis. An application may look simple until you discover it depends on a legacy database, a shared file share, a hard-coded IP address, or a batch process that runs overnight.

You should identify:

- **Technical dependencies** such as databases, queues, file shares, APIs, and identity providers.
- **Operational dependencies** such as monitoring, backup, patching, and certificate management.
- **Business dependencies** such as downstream reporting, finance close processes, or customer-facing workflows.

A dependency map helps you determine migration order. For example, if App A writes to Database B and App C reads from Database B, moving App A without planning for B and C can break the estate.

**Assessment criteria for Azure decisions**

When deciding how to handle an application in Azure, consider whether the workload is a good fit for IaaS (Infrastructure as a Service), PaaS (Platform as a Service), or SaaS (Software as a Service).

- **IaaS fit** is often best when you need OS-level control, legacy compatibility, or a fast migration path.
- **PaaS fit** is strong when the app can use managed databases, app hosting, or messaging services.
- **SaaS fit** is best when the business capability is commodity and the organization wants to reduce operational overhead.

You also evaluate:

- **Scalability needs**: can the app benefit from autoscaling or elastic capacity?
- **Availability needs**: does it require zone redundancy or multi-region design?
- **Security posture**: can it use managed identity, private endpoints, and centralized policy?
- **Data gravity**: are large datasets or low-latency integrations keeping it on-premises?
- **Modernization effort**: is the engineering cost justified by the business value?

**Practical scoring model**

A scoring model helps you compare applications consistently. One common pattern is to score each dimension from 1 to 5, then calculate a weighted total.

```text
Business value      30%
Technical health    20%
Dependency risk     20%
Cloud readiness     20%
Cost reduction      10%
```

Example interpretation:

- High business value and high cloud readiness often justify modernization.
- Low business value and high maintenance cost often justify retirement.
- High dependency risk and low readiness often justify retain or phased migration.

A scorecard should support discussion, not replace judgment. If an application is legally required or tied to a critical business event, that context can override the numeric score.

**Migration and modernization mapping**

Portfolio analysis should connect directly to Azure landing zone and migration planning. Once you know the disposition, you can map the workload to a target pattern.

- **Rehost** to Azure Virtual Machines when you need speed and minimal change.
- **Replatform** to Azure App Service, Azure SQL Database, Azure Container Apps, or Azure Kubernetes Service when the app can benefit from managed services.
- **Refactor** when you want to improve deployment frequency, resilience, or scalability.
- **Replace** with Microsoft 365, Dynamics 365, or another SaaS product when the business capability is standard.
- **Retire** after confirming no remaining business or compliance need.

**Example decision matrix**

| Condition | Likely decision |
|---|---|
| Low business value, high maintenance cost | Retire |
| High business value, low cloud readiness | Retain or rehost |
| High business value, moderate effort, clear service fit | Replatform |
| Strategic app with scaling and agility needs | Refactor or rearchitect |
| Commodity capability available as SaaS | Replace |

**Azure-specific considerations**

In Azure architecture planning, portfolio analysis should account for platform services and governance from the start.

- **Landing zone alignment**: determine subscription placement, management group structure, and policy requirements.
- **Identity integration**: assess whether the app can use Microsoft Entra ID for authentication and authorization.
- **Network design**: identify whether the app needs private connectivity, hybrid DNS, or VPN/ExpressRoute.
- **Data services**: decide whether to use Azure SQL Database, Azure Database for PostgreSQL, Azure Cosmos DB, or storage services.
- **Operations**: plan for Azure Monitor, Log Analytics, backup, and disaster recovery.

A workload that appears technically ready may still be blocked by governance, compliance, or network constraints. Portfolio analysis should surface those blockers early.

**Practical Azure examples**

A legacy .NET application running on Windows Server may be a good rehost candidate if the team needs a quick move and the app depends on Windows-specific components.

```powershell
# Example: assess a Windows VM workload before migration
Get-ComputerInfo | Select-Object WindowsProductName, WindowsVersion, OsHardwareAbstractionLayer
Get-Service | Where-Object {$_.Status -eq 'Running'} | Select-Object Name, DisplayName
```

A web application with a clean separation between UI, API, and data access may be a better replatform candidate.

```bash
# Example: inspect container images and dependencies
docker images
docker ps
az webapp list --query "[].{name:name, state:state, resourceGroup:resourceGroup}" -o table
```

A database-heavy application with minimal custom logic may be a candidate for managed database migration.

```bash
# Example: check Azure SQL migration readiness at a high level
az sql db list --resource-group rg-apps --server sql-prod --output table
```

**Governance and risk management**

Portfolio analysis is not only a technical exercise. It also supports governance by showing which applications carry the most risk and where controls are missing.

- **Security risk**: unsupported OS versions, weak authentication, or exposed endpoints.
- **Compliance risk**: data residency, retention, audit, and encryption requirements.
- **Operational risk**: single points of failure, manual deployments, and poor monitoring.
- **Financial risk**: oversized infrastructure, unused licenses, and duplicated platforms.

You should document assumptions and exceptions. If an application is retained because of a dependency that cannot yet be removed, note the reason and the planned review date.

> Do not treat “retain” as a permanent answer unless the business explicitly accepts that outcome. Retained applications should still have a remediation or review plan.

**Typical deliverables**

A portfolio analysis effort usually produces several artifacts:

- An application inventory
- A dependency map
- A scoring or classification model
- A migration wave plan
- A modernization backlog
- A risk register
- A business case or cost model

These deliverables help stakeholders understand not just what will move, but why it should move and in what order.

**Common pitfalls**

- **Incomplete discovery**: missing shadow IT or undocumented integrations.
- **Overreliance on technical fit**: ignoring business criticality and timing.
- **Ignoring data dependencies**: moving apps without their data and integration context.
- **One-size-fits-all migration**: forcing every workload into the same pattern.
- **No ownership**: failing to assign business and technical owners for decisions.
- **No review cycle**: treating the portfolio as static when applications and priorities change.

**Exam focus**

For AZ-305, expect application portfolio analysis to appear as part of broader solution design questions. You may need to choose the best migration strategy, identify the right Azure service family, or determine how to prioritize workloads.

Key exam takeaways:

- Use portfolio analysis to align business goals with technical migration choices.
- Evaluate applications by value, risk, dependencies, and cloud readiness.
- Map each application to a disposition such as retire, rehost, replatform, refactor, rearchitect, replace, or retain.
- Use the analysis to drive landing zone planning, migration waves, and modernization decisions.
