## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Lab objective and migration planning scope**

This lab focuses on building a migration plan for an on-premises VMware environment using **Azure Migrate**. The goal is to discover VMware virtual machines, assess Azure readiness, estimate sizing and cost, and identify migration dependencies before you move workloads.

Azure Migrate is the central hub for migration planning in Azure. For VMware environments, you typically use it to:

- Discover VMware VMs and collect performance data.
- Assess whether VMs are ready for Azure.
- Right-size compute and storage targets.
- Estimate monthly Azure costs.
- Identify blockers such as unsupported operating systems, oversized disks, or network dependencies.

A good migration plan separates **discovery**, **assessment**, and **migration execution**. Discovery and assessment help you plan; migration tools handle replication and cutover later.

**Azure Migrate components for VMware**

Azure Migrate uses a VMware appliance deployed as a virtual appliance in your vCenter environment. In practice, you download the appliance package or OVA, deploy it, and register it with your Azure Migrate project.

The appliance is used for:

- **Discovery** of VMware inventory.
- **Performance-based assessment** using collected utilization data.
- **Dependency analysis** when paired with dependency mapping tools.

For migration execution, Azure Migrate uses separate server migration capabilities and replication components. Do not treat discovery and migration as the same workflow.

<details>
<summary>Typical VMware-to-Azure planning flow</summary>
<p>

1. Create an Azure Migrate project.
2. Deploy the VMware appliance in vCenter.
3. Register the appliance with the project.
4. Start discovery of VMs.
5. Run assessments for one or more groups of VMs.
6. Review readiness, sizing, and cost estimates.
7. Refine the migration wave plan.
8. Prepare replication and test migration for selected servers.

**Project setup and prerequisites**

Before you begin, confirm that your environment can support discovery and assessment.

- You need an Azure subscription and permissions to create an Azure Migrate project.
- You need access to the vCenter Server and credentials with read access for discovery.
- The appliance needs network connectivity to vCenter and to Azure endpoints.
- Time synchronization matters because certificate and authentication issues can appear when clocks drift.
- DNS resolution should work for vCenter, the appliance, and Azure service endpoints.

A practical planning checklist looks like this:

- Identify the VMware vCenter version and ESXi estate.
- Confirm the number of VMs in scope.
- Decide whether you want performance-based or as-is sizing.
- Group VMs by application, environment, or migration wave.
- Determine whether you need dependency mapping for multi-tier apps.

**Discovery appliance deployment**

For VMware discovery, you deploy the Azure Migrate appliance as a virtual machine in your VMware environment. The appliance is commonly delivered as an OVA or appliance package, depending on the current portal workflow.

The appliance performs these tasks:

- Connects to vCenter using supplied credentials.
- Enumerates VM inventory.
- Collects performance counters over time.
- Sends metadata to Azure Migrate.

A typical deployment pattern is:

```text
vCenter Server
   |
   |-- Azure Migrate appliance VM
           |
           |-- Azure Migrate project
           |
           |-- Azure services
```

After deployment, you usually complete these steps:

1. Open the appliance configuration manager.
2. Register the appliance with your Azure Migrate project.
3. Provide vCenter credentials.
4. Start discovery.
5. Wait for inventory and performance collection to populate.

> Use the appliance as a discovery and assessment source. Do not assume it is the same component used for server replication and cutover.

**Discovery validation with PowerCLI and vCenter**

PowerCLI is useful for validating that you can connect to vCenter and query inventory before relying on Azure Migrate data. A connectivity check should confirm that the session is established and that you can query a known object.

```powershell
Connect-VIServer -Server vcenter01.contoso.local -User administrator@vsphere.local -Password 'P@ssw0rd!'
Get-View ServiceInstance
Get-VM -Name "APP01"
```

Why this matters:

- `Connect-VIServer` confirms authentication and basic reachability to vCenter.
- `Get-View ServiceInstance` verifies that the API session is usable.
- `Get-VM -Name "APP01"` checks that you can query a specific VM rather than enumerating the entire inventory.

If you need to inspect a VM’s basic configuration, use targeted queries:

```powershell
Get-VM -Name "APP01" | Select-Object Name, PowerState, NumCpu, MemoryGB
```

This is useful for inventory review, but it is not itself a connectivity test.

**Assessment types and when to use them**

Azure Migrate assessments help you decide whether a VM can move to Azure and what size it should be.

Common assessment approaches include:

- **Performance-based assessment**: Uses collected CPU, memory, disk, and network utilization to recommend Azure sizing.
- **As-is assessment**: Uses the current on-premises configuration without performance history.
- **Dependency-informed assessment**: Adds application dependency context to help group related servers.

Use performance-based assessment when:

- You want more realistic sizing.
- Workloads have variable utilization.
- You want to reduce overprovisioning.

Use as-is assessment when:

- You need a quick early estimate.
- Performance data is not yet available.
- You are doing a rough portfolio review.

**Sizing and readiness considerations**

Assessment output usually includes readiness status and recommended Azure VM size. Review the recommendation carefully instead of accepting it blindly.

Key factors include:

- **CPU and memory utilization** over the assessment window.
- **Disk throughput and IOPS** requirements.
- **Network throughput** for chatty workloads.
- **Operating system support** in Azure.
- **Boot type** and firmware compatibility.
- **Disk count and size limits** for the target VM family.

Common readiness outcomes:

- Ready for Azure.
- Ready with conditions.
- Not ready.
- Readiness unknown due to missing data.

A VM may be marked ready but still need manual review if:

- It uses unsupported drivers or software.
- It depends on local hardware integrations.
- It has latency-sensitive dependencies.
- It requires a specific Azure region or availability design.

**Cost estimation and target design**

Azure Migrate can estimate monthly cost based on the selected Azure region, VM size, storage type, and licensing assumptions.

When reviewing cost estimates, check:

- Region pricing differences.
- Reserved instance assumptions, if used.
- Storage account or managed disk type.
- OS licensing and Azure Hybrid Benefit eligibility.
- Network egress assumptions.

A simple planning table helps compare options:

| Factor | On-premises | Azure planning impact |
|---|---:|---|
| CPU | Physical host allocation | Drives VM size recommendation |
| Memory | Installed RAM | Influences target SKU |
| Storage | SAN/NAS or local disks | Maps to managed disks |
| Network | LAN/WAN throughput | Affects sizing and app latency |
| Licensing | Existing agreements | May reduce Azure cost |

**Dependency analysis and application grouping**

Migration planning is stronger when you understand application dependencies. A single VM rarely migrates in isolation in a real enterprise environment.

Dependency analysis helps you:

- Group web, app, and database tiers together.
- Avoid migrating a server before its upstream or downstream dependency.
- Identify hidden services such as DNS, file shares, or authentication endpoints.
- Reduce cutover risk by preserving communication paths.

Typical dependency questions include:

- Which servers talk to this VM?
- Which ports are required?
- Is the dependency local, cross-subnet, or cross-site?
- Does the application require a shared database or file share?

A practical grouping strategy is:

- Group by application first.
- Then group by environment, such as dev, test, or prod.
- Finally sequence by dependency order, such as database before app before web.

**Network planning for migration**

Network planning is critical because discovery success does not guarantee migration success.

You should validate:

- Connectivity from the appliance to vCenter.
- Connectivity from the appliance to Azure endpoints.
- DNS and name resolution.
- Firewall rules for required ports.
- Routing between on-premises and Azure landing zones.

For migration execution later, you also need to plan:

- IP addressing in Azure.
- Subnet design.
- NSGs (Network Security Groups).
- VPN or ExpressRoute connectivity.
- DNS updates and cutover timing.

A TCP-based check is more reliable than ICMP for validating service reachability. For example:

```powershell
Test-NetConnection vcenter01.contoso.local -Port 443
Test-NetConnection management.azure.com -Port 443
```

These checks help confirm that required HTTPS paths are open. ICMP ping may be blocked and should not be treated as a definitive validation method.

**Assessment workflow in the Azure portal**

A typical assessment workflow in Azure Migrate is:

1. Create or open the Azure Migrate project.
2. Add the VMware discovery appliance.
3. Wait for discovered servers to appear.
4. Create a server group for the VMs you want to assess.
5. Run an assessment using performance-based or as-is sizing.
6. Review readiness, sizing, and cost.
7. Export results or use them to refine the migration wave plan.

Useful review points after assessment:

- Confirm that the assessment window is long enough to capture normal workload behavior.
- Check whether peak utilization periods were included.
- Review any VMs with missing performance data.
- Validate that the recommended region matches your landing zone strategy.

**Practical migration plan output**

A migration plan should not stop at discovery results. It should translate assessment data into an actionable wave plan.

Your plan should include:

- VM inventory and ownership.
- Application grouping and dependency order.
- Target Azure region and subscription.
- Sizing recommendations and exceptions.
- Network and identity dependencies.
- Migration wave sequence.
- Test and rollback approach.
- Cutover window and business approval.

A concise wave plan example:

```text
Wave 1: Low-risk internal tools
- 5 VMs
- No external dependencies
- Standard D-series targets
- Weekend test migration

Wave 2: Customer-facing app tier
- 8 VMs
- Depends on SQL backend
- Requires DNS coordination
- Planned maintenance window

Wave 3: Database and shared services
- 3 VMs
- Highest dependency risk
- Requires extended validation
```

**Test migration and cutover planning**

Test migration is a strong validation practice because it lets you verify that a server boots and functions in Azure without affecting production. It is commonly recommended before cutover, but the exact process depends on your organization and workload criticality.

Use test migration to validate:

- VM boot success.
- Network connectivity in the test subnet.
- Application startup.
- DNS behavior.
- User acceptance or smoke testing.

Before cutover, confirm:

- Replication is healthy and current.
- The business has approved the window.
- Rollback steps are documented.
- DNS and load balancer changes are ready.
- Monitoring is in place.

> Test migration is recommended for most workloads, but it is not universally mandatory in every scenario. Follow your migration governance and change-management process.

**Troubleshooting common issues**

If discovery or assessment does not behave as expected, check the following:

- Appliance registration status in Azure Migrate.
- vCenter credentials and permissions.
- Firewall rules and proxy settings.
- DNS resolution from the appliance.
- Time synchronization.
- Performance collection duration.

Common symptoms and likely causes:

- No VMs discovered: vCenter access issue, wrong credentials, or inventory permissions.
- Missing performance data: insufficient collection time or appliance communication issue.
- Incorrect sizing: assessment window too short or workload not representative.
- Readiness warnings: unsupported OS, disk limits, or configuration mismatch.

A useful troubleshooting sequence is:

1. Verify appliance health.
2. Confirm vCenter connectivity.
3. Check Azure endpoint connectivity on TCP 443.
4. Review discovery logs.
5. Re-run assessment after enough data is collected.

**CLI and validation examples**

You can use command-line tools to support planning and validation.

Azure CLI example for project-related work:

```bash

az account show
az group create -n rg-migration-planning -l eastus
```

PowerShell example for vCenter validation:

```powershell

Get-View ServiceInstance

```

Network validation example:

```powershell
Test-NetConnection management.azure.com -Port 443
Test-NetConnection vcenter01.contoso.local -Port 443
```

These examples help you verify access paths, but they do not replace Azure Migrate assessment data.

**Exam-focused takeaways**

For AZ-305, focus on the planning decisions rather than the mechanics alone.

Remember these points:

- Azure Migrate supports discovery and assessment for VMware environments.
- The VMware appliance is deployed as a virtual appliance in vCenter.
- Discovery and migration are related but separate workflows.
- Performance-based assessment gives better sizing when enough data is available.
- Dependency analysis improves wave planning and reduces cutover risk.
- TCP-based validation is more reliable than ping for service reachability.
- Test migration is a best practice and commonly used before cutover, but not always an absolute requirement.

**Quick reference**

- **Discovery**: Find VMs and collect inventory.
- **Assessment**: Determine readiness, sizing, and cost.
- **Dependency mapping**: Understand application relationships.
- **Migration**: Replicate and cut over selected servers.
- **Validation**: Use targeted connectivity checks and test migrations.
