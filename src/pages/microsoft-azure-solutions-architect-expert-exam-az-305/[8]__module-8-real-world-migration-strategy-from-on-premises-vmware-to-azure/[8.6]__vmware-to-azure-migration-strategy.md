## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Migration strategy overview**

A VMware-to-Azure strategy starts with choosing the right target pattern for each workload, not forcing every VM into the same path. In practice, you compare **rehost**, **replatform**, **refactor**, and **replace** against business goals, risk tolerance, licensing, and operational maturity.

- **Rehost**: move the VM with minimal app change; best for speed and low code risk.
- **Replatform**: keep the app mostly intact but change the hosting model, such as moving a database to a managed service.
- **Refactor**: redesign parts of the application for cloud-native services.
- **Replace**: retire the workload and adopt SaaS or a different product.

For exam scenarios, rehost is often the fastest path for legacy VMware estates, but it is not always the best long-term answer. You should identify which workloads need immediate migration, which need modernization, and which should remain on-premises temporarily due to dependencies or compliance.

**When Azure VMware Solution is preferred**

**Azure VMware Solution (AVS)** is a Microsoft-managed private cloud that runs VMware software stack components in Azure. It is often the best fit when you need to preserve VMware operational patterns, migrate quickly with minimal change, or extend an existing VMware environment without redesigning applications.

AVS is commonly preferred over Azure IaaS when you need one or more of the following:

- Keep **vSphere**-based operations and tooling with minimal retraining.
- Preserve **vMotion**, **HA** (High Availability), and **DRS** (Distributed Resource Scheduler) style operations.
- Move large numbers of VMware VMs quickly with low application change.
- Run workloads that depend on VMware-specific constructs or operational processes.
- Use a private cloud for latency-sensitive or tightly controlled workloads while still consuming Azure services nearby.

AVS is not the default choice for every VM. If the goal is to reduce cost, modernize, or standardize on Azure-native operations, Azure IaaS may be a better landing zone. If the workload is a good candidate for managed PaaS, you should consider refactoring instead of preserving the full VMware stack.

**AVS architecture essentials**

AVS provides a dedicated VMware environment inside Azure. The core components you should know are:

- **vSphere**: compute virtualization layer for ESXi hosts and VM management.
- **vSAN**: software-defined storage used by the private cloud cluster.
- **NSX-T**: network virtualization and micro-segmentation platform.
- **HCX**: VMware Hybrid Cloud Extension for workload mobility, network extension, and migration.
- **vCenter Server**: central management plane for the private cloud.

A typical AVS deployment uses a private cloud with multiple hosts in a cluster. You connect it to Azure networking through a dedicated ExpressRoute-based connectivity model. This allows private connectivity to Azure services and on-premises networks.

A practical AVS topology often looks like this:

- On-premises VMware environment
- ExpressRoute or VPN connectivity to Azure
- AVS private cloud in an Azure region
- Azure native services such as Azure Backup, Azure Monitor, and Azure NetApp Files nearby for integration

> AVS is a private cloud service, not a general-purpose Azure VM host. You manage VMware workloads in a VMware-consistent way, but Microsoft manages the underlying infrastructure.

**Azure landing zone and governance**

Migrating VMware workloads into Azure should happen inside a governed landing zone. This prevents ad hoc subscriptions and unmanaged network sprawl.

Key governance elements include:

- **Management groups** to organize subscriptions by platform, environment, or business unit.
- **Subscriptions** to isolate billing, access, and policy boundaries.
- **Azure Policy** to enforce standards such as allowed regions, tagging, private endpoint usage, or SKU restrictions.
- **Resource groups** to group related migrated assets such as VMs, disks, NICs, and load balancers.
- **Role-based access control** (RBAC) to separate platform operations from application ownership.

A common pattern is:

- Root management group
- Platform management group
- Landing zone management groups for production and nonproduction
- Separate subscriptions for shared services, connectivity, and application workloads

This matters because VMware migrations often create many dependent resources at once. Without governance, you can end up with inconsistent naming, weak access control, and difficult cost management.

**Connectivity choices: VPN vs ExpressRoute**

Connectivity is a major design decision for VMware migration. You usually choose between site-to-site VPN and ExpressRoute based on bandwidth, latency, reliability, and migration volume.

| Option | Best for | Strengths | Tradeoffs |
|---|---|---|---|
| VPN | Smaller migrations, pilot projects, lower cost | Fast to deploy, uses internet | Less predictable latency and throughput |
| ExpressRoute | Large-scale migration, AVS, production connectivity | Private connectivity, better consistency, higher throughput | More cost and lead time |

Use **VPN** when you need a quick proof of concept or have modest replication traffic. Use **ExpressRoute** when you need stable connectivity for replication, AVS, or enterprise production traffic. For AVS, ExpressRoute is the standard connectivity model.

Practical network planning points:

- Preserve IP ranges where possible if you use network extension or need minimal app changes.
- Plan DNS carefully so migrated workloads resolve the correct on-premises or Azure endpoints.
- Decide whether to keep the same IPs temporarily, use NAT, or renumber during cutover.
- Validate firewall rules for replication ports, management traffic, and application ports.

**Azure Migrate workflow for VMware VMs**

**Azure Migrate** is the primary discovery, assessment, and migration hub for VMware-to-Azure rehost scenarios. It helps you inventory VMs, assess readiness, and migrate them to Azure VMs.

A typical workflow is:

1. Create an Azure Migrate project.
2. Deploy the **Azure Migrate appliance** in the VMware environment.
3. Connect the appliance to vCenter Server.
4. Discover VMware VMs and collect performance data.
5. Run assessment to check sizing, readiness, and cost estimates.
6. Prepare Azure networking, identity, and target resource groups.
7. Enable replication for selected VMs.
8. Run a **test migration** into an isolated network.
9. Validate the application and dependencies.
10. Perform **cutover** during the maintenance window.
11. Clean up replication and finalize post-migration tasks.

The appliance is important because it performs discovery and communicates with vCenter. In many exam scenarios, the appliance is the first concrete step after project creation.

A simplified deployment example:

```bash
# Example: verify appliance connectivity prerequisites
ping vcenter.contoso.local
nslookup vcenter.contoso.local
Test-NetConnection vcenter.contoso.local -Port 443
```

**Assessment and readiness checks**

Assessment is not just about VM size. You should evaluate whether the workload is actually suitable for migration as-is.

Important readiness checks include:

- **Guest OS support**: older or unsupported operating systems may not be eligible for smooth migration or may require remediation.
- **Application version support**: some legacy apps depend on outdated runtimes, drivers, or middleware.
- **VMware Tools**: outdated tools can affect guest interaction, drivers, and shutdown behavior.
- **Snapshots**: long-lived snapshots can indicate operational risk and may complicate migration planning.
- **Disks and controllers**: unusual disk layouts, shared disks, or special storage controllers may require validation.
- **Special drivers**: custom NIC, storage, or security drivers can cause boot or performance issues after migration.

You should also check for:

- Static IP dependencies
- Hard-coded hostnames
- License bindings to hardware or MAC addresses
- Unsupported clustering or shared storage assumptions
- Time synchronization dependencies

If a VM fails readiness checks, you may need to remediate before migration or choose AVS to preserve the VMware operating model.

**Azure Migrate replication and cutover details**

Replication is the mechanism that keeps the Azure copy of the VM current before cutover. For VMware migrations, Azure Migrate uses an appliance-driven replication workflow.

Typical replication concepts:

- **Replication policy**: defines recovery point retention, app-consistent snapshot frequency, and other replication behavior.
- **Target settings**: subscription, resource group, virtual network, subnet, availability option, and VM size.
- **Mobility service**: installed in the guest to enable replication and orchestration for supported scenarios.

A practical migration sequence is:

1. Select the VM in Azure Migrate.
2. Configure target settings in Azure.
3. Enable replication.
4. Wait for initial replication to complete.
5. Run a test migration to a nonproduction subnet.
6. Validate boot, network, DNS, and application behavior.
7. Schedule cutover.
8. Stop the source VM if required by the runbook.
9. Perform final synchronization.
10. Commit the migration and update DNS, load balancers, and monitoring.

Example target configuration concepts:

```json
{
  "targetResourceGroup": "rg-app-prod",
  "targetVirtualNetwork": "vnet-prod-eastus",
  "targetSubnet": "snet-app",
  "availabilityOption": "AvailabilityZone",
  "vmSize": "Standard_D4s_v5"
}
```

**Application dependency handling**

Multi-tier applications require sequencing. If you migrate the wrong tier first, the application may fail even if each VM is healthy on its own.

Common sequencing guidance:

- **Domain controllers**: migrate carefully and usually early, but maintain at least one healthy controller in the source or target until authentication is stable.
- **DNS servers**: ensure name resolution is available before moving dependent application tiers.
- **Load balancers**: update backend pools and health probes after the target VMs are ready.
- **Shared services**: file shares, license servers, monitoring agents, and certificate services often need special handling.
- **Database tiers**: coordinate with application servers to avoid split-brain or version mismatch.

Practical dependency handling patterns:

- Keep the source and target networks able to resolve each other during transition.
- Use temporary DNS records or lower TTL values before cutover.
- Migrate stateless web tiers before stateful database tiers when the app design allows it.
- Validate service accounts, Kerberos, and time sync for domain-joined workloads.
- For tightly coupled apps, use a full application cutover window rather than VM-by-VM migration.

A simple sequencing example for a three-tier app:

1. Validate identity and DNS services.
2. Migrate database or shared storage if required.
3. Migrate application servers.
4. Migrate web front ends.
5. Update load balancer and DNS records.
6. Run end-to-end validation.

**Azure Site Recovery for VMware migration and DR**

**Azure Site Recovery (ASR)** is often used for disaster recovery, but it also supports migration scenarios. It is especially useful when you want controlled replication, failover testing, and a structured cutover process.

Core ASR concepts:

- **Recovery Services vault**: the container for replication configuration, recovery plans, and failover orchestration.
- **Replication policy**: defines recovery point retention and app-consistent snapshot frequency.
- **Failover types**:
  - **Test failover**: validates recovery without impacting production.
  - **Planned failover**: used for controlled migration with source shutdown and minimal data loss.
  - **Unplanned failover**: used during outage or disaster.
- **Re-protect**: reverses replication direction after failover so the recovered VM becomes protected.
- **Failback**: returns workloads to the original site after recovery or migration if needed.

ASR is useful when you need a DR-first design or when migration and disaster recovery share the same operational model. For exam purposes, remember that failover is not the same as migration until you complete the cutover and commit the target as the new production site.

Example vault setup flow:

```bash
# Azure CLI example: create a resource group and vault
az group create --name rg-dr-eastus --location eastus
az backup vault create \
  --resource-group rg-dr-eastus \
  --name rsv-prod-dr \
  --location eastus
```

> The exact ASR commands and extensions vary by scenario and provider. In exam questions, focus on the sequence and the role of the vault, policy, and failover plan.

**AVS migration with HCX**

**HCX** is the main mobility tool for moving workloads into AVS with minimal disruption. It supports bulk migration, vMotion-based migration, and network extension.

Why HCX matters:

- It reduces the need for immediate IP renumbering.
- It supports large-scale VM movement with less downtime.
- It helps bridge on-premises and AVS networks during transition.

Common HCX capabilities:

- **Bulk migration** for many VMs with scheduled cutover.
- **vMotion migration** for live movement when conditions allow.
- **Network extension** to stretch Layer 2 networks temporarily.
- **WAN optimization** and secure connectivity between sites.

A practical AVS migration pattern is:

1. Deploy AVS private cloud.
2. Configure HCX Manager and service mesh.
3. Pair on-premises HCX with AVS HCX.
4. Extend required networks if IP continuity is needed.
5. Migrate pilot VMs.
6. Validate application behavior.
7. Move production workloads in waves.
8. Remove network extensions after cutover.

**Practical AVS configuration example**

A common AVS design uses:

- One AVS private cloud in a target Azure region
- ExpressRoute connectivity to on-premises
- HCX for migration
- Azure native services for backup, monitoring, and identity integration

Example planning snippet:

```text
On-prem vCenter
  -> HCX Connector
  -> ExpressRoute
  -> AVS Private Cloud
      - vCenter
      - ESXi hosts
      - vSAN datastore
      - NSX-T segments
```

**Modernization opportunities after rehost**

A VMware migration should not stop at lift-and-shift if the business wants lower operational overhead or better elasticity. After rehosting, you can modernize incrementally.

Common modernization paths:

- Move databases to managed services such as Azure SQL or Azure Database for PostgreSQL when compatible.
- Replace file servers with Azure Files or SharePoint where appropriate.
- Use Azure App Service or containers for web tiers that no longer need full VM control.
- Add Azure Monitor, Log Analytics, and Defender for Cloud for improved operations and security.
- Use autoscaling and managed identity to reduce VM dependency.

This is where **replatform** and **refactor** become important. Rehost gets you into Azure quickly; modernization reduces long-term technical debt.

**Decision guidance for exam scenarios**

Use the following decision logic:

- Choose **Azure Migrate to Azure VMs** when you want minimal change and standard Azure IaaS operations.
- Choose **AVS** when you need VMware compatibility, rapid migration with low change, or VMware operational continuity.
- Choose **ASR** when disaster recovery, failover testing, and controlled replication are central to the design.
- Choose **VPN** for smaller, lower-cost, or temporary connectivity needs.
- Choose **ExpressRoute** for production-grade, high-volume, or AVS-connected environments.
- Choose **replatform/refactor** when the workload benefits from managed services and the organization can absorb application change.

**Common pitfalls**

- Migrating without a dependency map.
- Ignoring DNS, certificates, and identity dependencies.
- Underestimating network throughput for replication.
- Forgetting to validate VMware Tools and guest OS support.
- Leaving snapshots in place too long before migration.
- Treating AVS as a generic Azure VM environment.
- Skipping test migration before cutover.
- Failing to define governance, subscription boundaries, and policy before landing workloads.

**Exam-oriented summary**

For AZ-305, the key is to match the migration method to the workload and business constraints. Azure Migrate is the standard tool for discovery, assessment, and rehost to Azure VMs. AVS is the best fit when you need VMware stack continuity, HCX-based mobility, and minimal application change. ASR provides replication and failover orchestration, especially for DR-driven migration plans. Governance, networking, and dependency sequencing are not optional details; they are core design inputs.
