## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Landing zone readiness overview**

A landing zone is the target Azure environment that gives migrated workloads a secure, governed, and supportable place to run. Before you move applications, you need the platform foundations in place so each workload lands in a predictable subscription, network, identity, and monitoring model.

For migration readiness, focus on the platform first and the workload second. If the landing zone is incomplete, you often end up redesigning after cutover, which increases risk and slows operations.

- **Governance** defines who can deploy, where they can deploy, and what they can deploy.
- **Identity** provides centralized authentication and authorization.
- **Networking** connects workloads to each other and to on-premises systems.
- **Security** establishes baseline controls and visibility.
- **Operations** ensures you can monitor, patch, and recover migrated systems.

**Landing zone design principles**

Azure landing zones are commonly aligned to the Cloud Adoption Framework (CAF) and the Azure Landing Zones (ALZ) guidance. In practice, ALZ is a set of design and implementation recommendations, not a single fixed deployment. You adapt it to your organization’s scale, compliance needs, and operating model.

A migration-ready landing zone should be:

- **Scalable** so you can add subscriptions, regions, and workloads without redesign.
- **Governed** so policy and access control are consistent.
- **Observable** so logs, metrics, and alerts are centralized.
- **Networked** so traffic paths are intentional and secure.
- **Recoverable** so backup and disaster recovery are planned before go-live.

**Core platform structure**

A common enterprise structure uses management groups above subscriptions. This lets you apply policy and access control at the right scope.

Typical hierarchy:

```text
Tenant
└── Management groups
    ├── Platform
    │   ├── Identity
    │   ├── Connectivity
    │   └── Management
    └── Landing zones
        ├── Subscription A
        ├── Subscription B
        └── Subscription C
```

Use separate subscriptions when you need isolation for:

- **Platform services** such as shared networking, logging, and identity-related resources.
- **Application workloads** so teams can deploy independently.
- **Nonproduction environments** to reduce blast radius.
- **Regulated workloads** that require stricter controls.

**Identity and access readiness**

Identity is usually the first dependency to validate because every other control depends on it. Use Microsoft Entra ID as the central identity provider and design access around least privilege.

Key practices:

- Use **role-based access control (RBAC)** for Azure resource permissions.
- Prefer group-based assignments over direct user assignments.
- Use **Privileged Identity Management (PIM)** for just-in-time elevation.
- Separate platform administrators from workload operators.
- Use managed identities for applications and automation where possible.

A practical access model often includes:

- **Tenant-level admins** for identity and directory tasks.
- **Platform admins** for shared services.
- **Subscription owners** for workload lifecycle.
- **Application teams** with scoped contributor or custom roles.

Example RBAC assignment:

```bash
az role assignment create \
  --assignee-object-id <group-object-id> \
  --assignee-principal-type Group \
  --role "Contributor" \
  --scope /subscriptions/<subscription-id>
```

For sensitive environments, consider custom roles instead of broad built-in roles. This reduces accidental privilege and helps with audit requirements.

> Avoid using permanent elevated access for migration teams. Use time-bound access and review assignments regularly.

**Governance and policy**

Governance controls make the landing zone repeatable. Azure Policy is the main enforcement tool for standards such as allowed regions, required tags, approved SKUs, and diagnostic settings.

Use policy to:

- Restrict deployments to approved regions.
- Require tags such as `CostCenter`, `Owner`, and `Environment`.
- Deny public IP creation where not allowed.
- Audit or deploy diagnostic settings.
- Enforce encryption and secure configuration baselines.

Example policy assignment using a policy definition resource ID:

```bash
az policy assignment create \
  --name "deny-unapproved-regions" \
  --scope /subscriptions/<subscription-id> \
  --policy /providers/Microsoft.Authorization/policyDefinitions/<policy-definition-guid>
```

If you need a bundle of related controls, use a policy initiative, also called a policy set definition.

Example initiative assignment:

```bash
az policy assignment create \
  --name "baseline-controls" \
  --scope /subscriptions/<subscription-id> \
  --policy-set-definition /providers/Microsoft.Authorization/policySetDefinitions/<initiative-guid>
```

Common governance patterns:

- Use **management groups** to apply policy at scale.
- Use **initiative assignments** for baseline compliance.
- Use **exemptions** sparingly and with expiration dates.
- Use **resource locks** for critical shared services.
- Use **tags** for chargeback and ownership.

**Networking readiness**

Networking is one of the most important landing zone decisions because it affects connectivity, security, and DNS. Decide early whether you will use hub-and-spoke, virtual WAN, or another approved topology.

A typical enterprise pattern is:

- **Hub network** for shared services such as firewall, DNS, VPN, or ExpressRoute.
- **Spoke networks** for application workloads.
- **Central inspection** for north-south and sometimes east-west traffic.
- **Private connectivity** to PaaS services where required.

Key design questions:

- Will workloads need on-premises connectivity on day one?
- Do you need shared DNS resolution across sites?
- Will you use Azure Firewall, third-party NVA, or both?
- Do you need forced tunneling or internet breakout controls?
- Will private endpoints be required for PaaS access?

Example hub-and-spoke route intent:

```text
Spoke subnet -> UDR -> Firewall -> Internet
Spoke subnet -> UDR -> Firewall -> On-premises via VPN/ExpressRoute
Spoke subnet -> Private Endpoint -> Private DNS resolution
```

For private endpoints, use service-specific private DNS zones. The zone name depends on the service, so do not assume one universal zone. For example, Azure SQL uses `privatelink.database.windows.net`, while other services use different zones.

Example private DNS zone link:

```bash
az network private-dns zone create \
  --resource-group rg-network \
  --name privatelink.database.windows.net

az network private-dns link vnet create \
  --resource-group rg-network \
  --zone-name privatelink.database.windows.net \
  --name link-spoke1 \
  --virtual-network vnet-spoke1 \
  --registration-enabled false
```

For on-premises name resolution, forward queries to a DNS server or to Azure DNS Private Resolver inbound endpoints. On-premises DNS does not query Azure Private DNS zones directly.

**Connectivity options**

Choose the connectivity model that matches scale and operational maturity.

| Option | Best for | Notes |
|---|---|---|
| Site-to-site VPN | Smaller or initial migrations | Lower cost, simpler setup, internet-based tunnel |
| ExpressRoute | Enterprise and predictable performance | Private connectivity, higher cost, better consistency |
| Azure Virtual WAN | Distributed or multi-site connectivity | Centralized transit and branch integration |

Practical guidance:

- Use **VPN** for early migration waves or temporary connectivity.
- Use **ExpressRoute** for production workloads with latency or compliance needs.
- Use **Virtual WAN** when you need managed global transit and many branches.

**Security baseline**

Security readiness means more than enabling a few services. You need a baseline that covers configuration, detection, and response.

Important controls:

- Enable **Microsoft Defender for Cloud** for relevant resource types.
- Turn on diagnostic logs for subscriptions, resources, and network components.
- Centralize logs in Log Analytics or another approved SIEM.
- Use **Key Vault** for secrets, keys, and certificates.
- Prefer Azure RBAC for Key Vault access where supported by your design.
- Use private endpoints for sensitive PaaS services when required.

Example Key Vault access with Azure RBAC:

```bash
az role assignment create \
  --assignee-object-id <group-object-id> \
  --role "Key Vault Secrets Officer" \
  --scope /subscriptions/<subscription-id>/resourceGroups/rg-security/providers/Microsoft.KeyVault/vaults/kv-prod
```

If you still use access policies in a legacy design, document why. For new designs, Azure RBAC is usually the preferred model when supported.

Security checklist:

- **Encryption** at rest and in transit.
- **Secret management** outside application code.
- **Network segmentation** between tiers and environments.
- **Threat detection** for workloads and identities.
- **Vulnerability management** for images and VMs.
- **Incident response** runbooks and ownership.

**Monitoring and operations**

A landing zone is not ready until operations can support it. Monitoring should be centralized before the first migration wave so you can validate health immediately after cutover.

Operational foundations:

- Use **Azure Monitor** for metrics and alerts.
- Use **Log Analytics** for queryable logs.
- Use **Activity Log** monitoring for subscription-level changes.
- Use **workbooks** for dashboards and operational views.
- Use **action groups** to route alerts to email, SMS, ITSM, or automation.

Example alert rule creation pattern:

```bash
az monitor metrics alert create \
  --name "cpu-high-vm01" \
  --resource-group rg-monitoring \
  --scopes /subscriptions/<subscription-id>/resourceGroups/rg-app/providers/Microsoft.Compute/virtualMachines/vm01 \
  --condition "avg Percentage CPU > 80" \
  --description "CPU above 80 percent" \
  --window-size 5m \
  --evaluation-frequency 1m
```

Use this as a pattern only; metric alert syntax can vary by resource type and CLI version. Always verify the exact metric name and supported dimensions for the target resource.

Operational readiness also includes:

- Backup policies and retention.
- Recovery Services vault placement.
- Patch management and update rings.
- Change control and deployment approvals.
- Runbooks for common incidents.

**Migration planning inputs**

Before you migrate, inventory the workload and map it to landing zone requirements. This prevents surprises during cutover.

Capture:

- **Application dependencies** including databases, queues, and shared services.
- **Network dependencies** such as ports, DNS, and firewall rules.
- **Identity dependencies** such as service accounts and managed identities.
- **Compliance requirements** such as data residency or retention.
- **Performance requirements** such as latency, throughput, and availability.

A simple readiness checklist:

1. Confirm the target subscription and management group.
2. Validate identity and RBAC assignments.
3. Confirm network connectivity and DNS resolution.
4. Apply policy and tagging standards.
5. Enable logging, monitoring, and backup.
6. Test deployment, access, and rollback.
7. Migrate a pilot workload before broad rollout.

**Common landing zone patterns**

Different workloads need different levels of isolation.

- **Shared landing zone**
  - Good for small environments or early adoption.
  - Lower overhead, but less isolation.
- **Dedicated landing zone per business unit**
  - Good for autonomy and chargeback.
  - Easier to delegate ownership.
- **Dedicated landing zone per application**
  - Good for regulated or high-risk workloads.
  - Strong isolation, higher management overhead.

Choose the pattern that matches governance, scale, and operating model. Do not over-engineer a small migration, but do not under-design a regulated one.

**Practical migration examples**

For a VM migration, you typically need:

- A target subnet with the right NSG rules.
- DNS resolution for internal names.
- A route to on-premises or shared services.
- Backup and monitoring enabled.
- A recovery plan if cutover fails.

For a PaaS migration, you typically need:

- Private endpoint or public access decision.
- DNS zone integration.
- Identity and secret management.
- Policy compliance for the service.
- Logging and alerting for service health.

For a data platform migration, you often need:

- Network isolation.
- Encryption and key management.
- Data retention and backup policies.
- Access reviews and audit logging.
- Performance testing before production cutover.

**What exam questions usually test**

AZ-305 questions often focus on design tradeoffs rather than command syntax. Be ready to choose the right landing zone approach based on requirements.

Expect scenarios about:

- Centralized versus decentralized governance.
- Hub-and-spoke versus Virtual WAN.
- RBAC versus access policies.
- Private endpoints versus public access.
- Management groups versus subscriptions.
- Policy enforcement versus manual controls.
- Shared services versus workload isolation.

When answering, prioritize:

- Security and compliance first.
- Operational simplicity second.
- Cost optimization after the core controls are in place.
- Scalability and repeatability for future migration waves.

**Key takeaways**

A migration-ready landing zone gives you a governed, observable, and secure target environment before workloads move. The most important decisions are identity, network topology, policy scope, and operational monitoring.

If you prepare those foundations early, migration becomes a controlled rollout instead of a series of one-off fixes.
