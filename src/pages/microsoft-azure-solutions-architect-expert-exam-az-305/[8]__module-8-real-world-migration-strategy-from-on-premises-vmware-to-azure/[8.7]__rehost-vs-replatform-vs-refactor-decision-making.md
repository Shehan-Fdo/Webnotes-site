## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Migration strategy context**

Azure architecture decisions often start with the **6 Rs** of migration: retire, retain, relocate, rehost, replatform, refactor, and replace. In practice, AZ-305 questions usually focus on choosing the least disruptive option that still meets business, security, performance, and cost goals.

- **Retire**: decommission workloads that no longer provide value.
- **Retain**: keep workloads on-premises or in the current environment for now.
- **Replace**: move to a SaaS (Software as a Service) product instead of migrating the existing app.
- **Rehost**: lift and shift to Azure with minimal code change.
- **Replatform**: make targeted platform changes to use managed services.
- **Refactor**: redesign the application to better fit cloud-native patterns.

A good decision is not “which option is best in general,” but “which option best satisfies the current constraints.” Those constraints usually include timeline, budget, application complexity, operational ownership, compliance, downtime tolerance, and the amount of technical debt you can carry forward.

**Core definitions**

<dl>
<dt><strong>Rehost</strong></dt>
<dd>Move an application to Azure with minimal or no code changes. The app behavior stays mostly the same, but the hosting environment changes.</dd>
<dt><strong>Replatform</strong></dt>
<dd>Make selective changes to use managed Azure services or platform features while keeping the application architecture largely intact.</dd>
<dt><strong>Refactor</strong></dt>
<dd>Redesign or rewrite parts of the application to improve scalability, agility, resilience, or maintainability, often using cloud-native services.</dd>
</dl>

**How to think about the decision**

Start with business constraints, then validate technical fit. A fast migration deadline may favor rehost, but only if the app is compatible and the operational risk is acceptable. If the app needs better scalability or lower patching overhead, replatform may be the better compromise. If the app must support rapid feature delivery, elastic scale, or modern resilience patterns, refactor becomes more attractive.

- **Rehost** when speed and low change are the priority.
- **Replatform** when you want some cloud benefits without a full redesign.
- **Refactor** when the application needs architectural improvement, not just relocation.
- **Retain** when migration risk is too high or the workload is not ready.
- **Replace** when a SaaS product meets the business need more effectively than custom code.

**Assessment inputs that drive the choice**

Before selecting a strategy, gather evidence from discovery and assessment tools. In Azure migration planning, the most useful outputs are compatibility, readiness, dependency, and cost signals.

- **Compatibility assessment** identifies whether the workload can run in Azure as-is or with changes.
- **Readiness assessment** shows whether the operating system, runtime, and dependencies are supported.
- **Dependency mapping** reveals upstream and downstream systems that may break during migration.
- **Cost estimation** helps compare Azure hosting costs against modernization effort.
- **Performance profiling** shows whether the current architecture can meet target latency and throughput.
- **Security and compliance review** identifies controls that may require redesign.

Practical tools and outputs often include:
- Azure Migrate assessments for server and app readiness.
- Dependency visualization to find hidden network and service dependencies.
- Database assessment reports for compatibility and feature gaps.
- Cost sizing recommendations for VM-based migration targets.

> If the assessment shows unsupported OS versions, hard-coded local dependencies, or incompatible database features, a pure rehost may be risky even if it is the fastest path.

**Decision matrix**

| Requirement / Constraint | Rehost | Replatform | Refactor | Retain | Replace |
|---|---:|---:|---:|---:|---:|
| Fastest migration | Best | Good | Poor | Best short-term | Good if SaaS adoption is quick |
| Minimal code change | Best | Good | Poor | Best | Poor |
| Lower ops overhead | Fair | Good | Best | Fair | Best |
| Cloud-native scalability | Fair | Good | Best | Poor | Good if SaaS supports it |
| Legacy app compatibility | Best | Good | Fair | Best | Poor if SaaS mismatch |
| Tight deadline | Best | Good | Poor | Best | Good |
| Long-term modernization | Poor | Good | Best | Poor | Good if product fit exists |
| High technical debt | Poor | Fair | Best | Fair | Best if business process fits SaaS |

Use this matrix as a starting point, not a rulebook. A workload can score well for rehost and still be a poor choice if it has severe dependency coupling or licensing constraints.

**Rehost: lift and shift**

Rehost moves the workload to Azure with the least application change. It is often the first step when you need to exit a datacenter quickly, reduce hardware risk, or preserve existing behavior while you plan later modernization.

Typical rehost targets include:
- Azure Virtual Machines.
- Azure VMware Solution for VMware-based estates.
- Lift-and-shift database servers when the database engine and version are supported.
- File servers and line-of-business applications that depend on local OS behavior.

Why you choose rehost:
- You need the shortest migration path.
- The application is stable and business-critical.
- The team lacks time or budget for redesign.
- The app has many dependencies that are hard to untangle immediately.

Tradeoffs:
- You keep most operating system and patching responsibility.
- You often carry forward technical debt.
- You may not gain much from managed services.
- Scaling and resilience remain mostly your responsibility.

Example Azure CLI for a Windows VM rehost target:
```bash
az group create -n rg-rehost-prod -l eastus

az vm create \
  -g rg-rehost-prod \
  -n vm-app01 \
  --image Win2022Datacenter \
  --admin-username azureadmin \
  --admin-password 'Use-A-Strong-Password-Here!' \
  --size Standard_D4s_v5
```

For Linux rehost targets, you typically choose a Linux image and manage the app stack yourself:
```bash
az vm create \
  -g rg-rehost-prod \
  -n vm-web01 \
  --image Ubuntu2204 \
  --admin-username azureadmin \
  --generate-ssh-keys \
  --size Standard_D2s_v5
```

**Replatform: lift, tinker, and shift**

Replatform keeps the application mostly intact but changes selected components to managed Azure services. This is the middle ground when you want meaningful operational improvement without a full rewrite.

Common replatform examples:
- Move a database from a self-managed VM to Azure SQL Database or Azure SQL Managed Instance.
- Move application hosting from VMs to Azure App Service.
- Replace self-managed caching with Azure Cache for Redis.
- Replace file-based integration with Azure Storage or messaging services.

Why you choose replatform:
- You want to reduce patching, backups, or OS management.
- The app can tolerate limited code or configuration changes.
- You want better availability or scaling with less operational effort.
- The database or runtime is compatible with a managed service.

Tradeoffs:
- You may need code, connection string, or schema changes.
- Some features may not be supported by the managed service.
- Migration complexity is higher than rehost.
- You still may not achieve full cloud-native agility.

Example: moving a web app to App Service with a managed database backend:
```bash
az group create -n rg-replatform-prod -l eastus

az appservice plan create \
  -g rg-replatform-prod \
  -n asp-web-prod \
  --sku P1v3

az webapp create \
  -g rg-replatform-prod \
  -p asp-web-prod \
  -n mywebapp12345 \
  --runtime "DOTNET:8"

az sql server create \
  -g rg-replatform-prod \
  -n sqlsrvprod12345 \
  -l eastus \
  -u sqladminuser \
  -p 'Use-A-Strong-Password-Here!'

az sql db create \
  -g rg-replatform-prod \
  -s sqlsrvprod12345 \
  -n appdb \
  --service-objective S0
```

Example: moving a database to Azure SQL Managed Instance when you need near-full SQL Server compatibility:
```bash
az sql mi create \
  -g rg-replatform-prod \
  -n mi-prod01 \
  -l eastus \
  --admin-user sqladminuser \
  --admin-password 'Use-A-Strong-Password-Here!' \
  --sku-name GP_Gen5 \
  --storage-size 256
```

**Refactor: redesign for cloud-native value**

Refactor changes the application architecture to better align with Azure platform capabilities. This is the most disruptive option, but it can deliver the greatest long-term benefit.

Typical refactor patterns:
- Break a monolith into services or bounded components.
- Move from synchronous coupling to event-driven messaging.
- Use serverless compute for bursty workloads.
- Externalize state into managed data services.
- Add autoscaling, resilience, and observability patterns.

Why you choose refactor:
- The current design limits scale, agility, or reliability.
- You need faster release cycles and independent deployment.
- You want to reduce coupling and improve fault isolation.
- The business case supports a larger engineering effort.

Tradeoffs:
- Higher cost, time, and delivery risk.
- Requires stronger engineering discipline and testing.
- May need changes across application, data, and integration layers.
- Often needs a phased migration strategy rather than a single cutover.

Example: moving a background processing component to Azure Functions and Service Bus:
```csharp
[FunctionName("OrderProcessor")]
public static async Task Run(
    [ServiceBusTrigger("orders", Connection = "ServiceBusConnection")] string message,
    ILogger log)
{
    log.LogInformation($"Processing order: {message}");
    await Task.CompletedTask;
}
```

Example topology:
- Front end on Azure App Service.
- Async work on Azure Functions.
- Messaging through Azure Service Bus.
- Data in Azure SQL Database or Cosmos DB depending on access patterns.

**When to choose retain instead of migrate**

Retain is often the right answer when migration is not yet justified. In AZ-305 scenarios, this can be the best choice if the workload is too risky, too specialized, or too expensive to move right now.

Choose retain when:
- The application is nearing retirement.
- The workload depends on unsupported hardware or niche software.
- The business value does not justify migration cost.
- Regulatory or latency constraints make migration impractical for now.
- A dependency chain makes partial migration unsafe.

Retain is not failure. It is a deliberate architectural decision when the current environment is still the best fit.

**When to choose replace instead of migrate**

Replace means adopting a SaaS product rather than migrating the existing application. This is often the best choice when the business process matters more than the custom implementation.

Choose replace when:
- The current app is mostly a commodity function.
- A SaaS product covers the required capabilities.
- You want to reduce maintenance and upgrade burden.
- The organization is willing to adapt processes to the product.

Examples:
- Replace a custom ticketing system with a SaaS ITSM platform.
- Replace a custom email or collaboration solution with Microsoft 365 services.
- Replace a bespoke CRM with a SaaS CRM if feature fit is acceptable.

**Data migration considerations**

Data often determines the migration strategy more than the application code does. A workload that looks easy to rehost may become a replatform or refactor candidate once you examine the database.

Key factors:
- **Database compatibility**: unsupported features, CLR dependencies, cross-database queries, or SQL dialect differences can block a simple move.
- **Schema changes**: normalization, denormalization, or partitioning may be needed for cloud scale.
- **Downtime tolerance**: short outage windows may require replication, dual-write, or staged cutover.
- **Data gravity**: large datasets can make repeated movement expensive and slow.
- **Transaction patterns**: high write contention or distributed transactions may favor redesign.
- **Backup and restore strategy**: managed services change how you protect and recover data.

Practical guidance:
- If the database engine and version are supported and the app is tolerant of the target environment, rehost or replatform may work.
- If the database needs feature changes, connection model changes, or managed HA (high availability), replatform is often better.
- If the data model itself is the bottleneck, refactor may be required.

**Scenario-based decision examples**

1. A line-of-business app runs on a supported Windows Server version, uses a standard SQL Server database, and must leave the datacenter in 90 days.
   - Best fit: **rehost**
   - Why: speed matters most, and the app can move with minimal change.

2. A web app runs on VMs, but the team wants less patching and better scaling, and the code already supports externalized configuration.
   - Best fit: **replatform**
   - Why: App Service and a managed database reduce operations without a rewrite.

3. A monolithic order system has frequent release bottlenecks, poor fault isolation, and needs event-driven processing for peak demand.
   - Best fit: **refactor**
   - Why: the architecture itself limits business agility.

4. A legacy app depends on an unsupported OS and a proprietary driver that Azure does not support.
   - Best fit: **retain** temporarily or **replace** if a SaaS alternative exists
   - Why: forcing migration would increase risk without a viable target.

5. A custom HR portal duplicates capabilities already available in a SaaS HR platform.
   - Best fit: **replace**
   - Why: the business process is commodity and the custom app adds maintenance burden.

**How AZ-305 questions usually test this topic**

Exam questions often describe a workload and ask for the most appropriate migration strategy. Look for clues in the wording:

- “Minimal changes,” “fastest migration,” or “lift and shift” usually points to **rehost**.
- “Use managed services,” “reduce administration,” or “limited code changes” usually points to **replatform**.
- “Redesign,” “cloud-native,” “microservices,” or “event-driven” usually points to **refactor**.
- “Not worth migrating,” “unsupported dependency,” or “retire soon” may point to **retain**.
- “SaaS meets the requirement” may point to **replace**.

Be careful with distractors:
- A managed database does not automatically mean refactor; it may be replatform.
- Moving to Azure App Service is not always refactor; it can be replatform if the app remains largely the same.
- A VM migration is not always the best answer if the question emphasizes reduced operations or modernization.

**Practical decision workflow**

1. Identify the business driver: speed, cost, modernization, risk reduction, or feature delivery.
2. Review application constraints: OS, runtime, dependencies, licensing, and integration points.
3. Assess data complexity: compatibility, size, downtime tolerance, and migration method.
4. Compare operational goals: patching, scaling, resilience, and monitoring.
5. Check whether retain or replace is a better business decision than migration.
6. Choose the least disruptive option that still meets the target state.

**Implementation examples and patterns**

For rehost, you often preserve the current deployment model:
- VM-based web tier.
- VM-based application tier.
- VM-based database tier if supported.

For replatform, you often split responsibilities:
- App code on App Service or containers.
- Database on Azure SQL Database or Managed Instance.
- Files in Azure Storage.
- Background jobs in Azure Functions or WebJobs.

For refactor, you often redesign around managed services:
- API layer with App Service or API Management.
- Stateless compute.
- Event-driven messaging.
- Managed identity for authentication.
- Observability with Azure Monitor and Application Insights.

Example App Service configuration snippet:
```json
{
  "name": "WEBSITE_RUN_FROM_PACKAGE",
  "value": "1"
}
```

Example connection string pattern:
```text
Server=tcp:sqlsrvprod12345.database.windows.net,1433;Database=appdb;User ID=sqladminuser;Password=***;Encrypt=True;TrustServerCertificate=False;
```

**Common pitfalls**

- Choosing rehost when the app has hidden dependencies that will fail in Azure.
- Choosing refactor because it sounds modern, even when the business needs a fast exit.
- Ignoring database compatibility and data migration effort.
- Overlooking licensing, supportability, or compliance constraints.
- Treating replatform as a “small rehost” when it still requires testing and change management.
- Forgetting that retain and replace are valid architectural outcomes.

**Exam-ready summary**

- **Rehost**: fastest, least change, most like the current environment.
- **Replatform**: selective modernization with managed services.
- **Refactor**: redesign for cloud-native benefits and long-term agility.
- **Retain**: keep the workload where it is when migration is not justified.
- **Replace**: use SaaS when it better fits the business need.

The best AZ-305 answer is usually the one that aligns business goals with the smallest necessary change. Start with assessment data, validate dependencies and data constraints, and then choose the least disruptive strategy that still meets the target architecture.
