## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Risk management fundamentals**

Risk management is the disciplined process of identifying what can go wrong, estimating how likely it is, measuring the business impact, and deciding how to respond. In Azure architecture, you use it to reduce the chance that a deployment, migration, or platform change causes unacceptable downtime, data loss, or security exposure.

A practical risk workflow includes:

- **Risk identification**: list threats, failure modes, dependencies, and change-related hazards.
- **Risk assessment**: evaluate each risk using likelihood and impact.
- **Risk prioritization**: rank risks so you address the highest business exposure first.
- **Risk mitigation**: reduce likelihood or impact through controls, design changes, or operational safeguards.
- **Risk acceptance**: formally accept a risk when the cost of mitigation exceeds the expected harm.
- **Risk transfer**: shift some exposure to a third party, such as through managed services, support contracts, or insurance.
- **Residual risk**: the risk that remains after controls are applied.

A simple scoring model helps you compare risks consistently:

| Likelihood | Impact | Example interpretation |
|---|---:|---|
| Low | Low | Monitor only |
| Low | High | Mitigate if the service is critical |
| High | Low | Acceptable if recovery is easy |
| High | High | Immediate action required |

For AZ-305, you should connect risk decisions to business requirements such as recovery time objective (<abbr title="Recovery Time Objective">RTO</abbr>) and recovery point objective (<abbr title="Recovery Point Objective">RPO</abbr>). RTO defines how long a service can be unavailable. RPO defines how much data loss is acceptable. These values directly influence whether you choose rollback, restore, or failover.

**How to identify and assess risks**

You usually identify risks by reviewing architecture diagrams, change plans, dependency maps, and operational history. Focus on:

- Single points of failure in compute, storage, networking, identity, and DNS.
- Deployment risks from schema changes, feature flags, and configuration drift.
- Data risks from replication lag, backup gaps, and retention misalignment.
- Operational risks from unclear ownership, missing runbooks, or weak monitoring.
- External risks such as regional outages, third-party API failures, and quota limits.

A useful assessment method is to ask:

1. What fails?
2. How often could it fail?
3. What is the business impact?
4. How quickly can you detect it?
5. How quickly can you recover?

This gives you a realistic view of whether a control is preventive, detective, or corrective.

**Rollback planning as a change-control discipline**

Rollback planning is the preparation you do before a change so you can safely return to a known-good state if the change fails. It is not the same as disaster recovery. Rollback is usually tied to a specific deployment or configuration change. Disaster recovery is broader and covers major outages, corruption, or regional loss.

A strong rollback plan includes:

- A **documented rollback runbook** with exact steps.
- A **decision tree** that defines when rollback is allowed and when you must switch to restore or failover.
- Clear **ownership and approvals** for the rollback decision.
- A **communication plan** for stakeholders, support teams, and users.
- An **escalation path** if rollback does not resolve the issue.
- Validation steps to confirm the system is healthy after rollback.

Rollback should be fast, repeatable, and tested. If rollback is manual, it must still be deterministic. If rollback is automated, you still need human approval gates for high-risk production changes.

**Rollback runbook design**

A rollback runbook should be written before deployment and stored where operators can access it during an incident. It should include:

- Change identifier and deployment version.
- Preconditions for rollback, such as health check failures or error-rate thresholds.
- Exact commands or pipeline actions.
- Dependencies that must be stopped first.
- Data handling instructions, including whether data can be safely discarded.
- Verification checks after rollback.
- Contacts for application, platform, database, and security owners.

Example structure:

```text
1. Confirm incident severity and rollback eligibility.
2. Freeze further deployments.
3. Notify incident commander and service owner.
4. Stop traffic shift or disable the new release.
5. Revert application version.
6. Apply database compensating actions if required.
7. Validate health, logs, metrics, and user flows.
8. Decide whether to keep rollback or escalate to restore/failover.
```

A rollback runbook is most effective when it is paired with a decision tree:

```text
If issue is limited to app code and data is backward-compatible -> rollback release
If schema change is incompatible and data is already mutated -> restore or compensating fix
If platform or region is impaired -> failover
If corruption is widespread or unknown -> restore from backup
```

**Ownership, approvals, and communication**

Rollback decisions should not be ambiguous. Define who can authorize the action and who executes it.

- **Incident commander**: coordinates the response and makes the operational call.
- **Service owner**: confirms business impact and technical risk.
- **Database owner**: approves data-related actions.
- **Change owner**: understands the deployment and its dependencies.
- **Communications lead**: informs stakeholders and updates status channels.

Your communication plan should specify:

- Who gets notified first.
- Which channel is used for internal updates.
- What customer-facing message is approved.
- How often updates are posted.
- When the escalation path is triggered.

This matters because rollback can reduce technical risk while increasing business confusion if communication is poor.

**Backup strategy design for rollback and recovery**

Backups are a core control in risk mitigation and recovery planning. They support rollback when the change cannot be safely reversed in place, and they support restore when data is lost or corrupted.

Design backup strategy around:

- **Backup frequency**: how often backups are taken.
- **Retention**: how long backups are kept.
- **Geo-redundancy**: whether copies exist in another region.
- **Restore testing**: whether backups are actually usable.
- **RPO and RTO**: how much data loss and downtime the business can tolerate.

Common design choices:

- Frequent backups reduce RPO but increase cost and operational overhead.
- Longer retention helps with delayed detection of corruption or ransomware.
- Geo-redundant backups improve resilience to regional failure.
- Immutable or protected backups reduce the chance of deletion or tampering.

For Azure, align backup design with service type:

- **Azure VMs**: use Azure Backup for VM-level protection and consider Azure Site Recovery (<abbr title="Azure Site Recovery">ASR</abbr>) for replication and failover.
- **Azure Files**: use share snapshots and Azure Backup for file share protection.
- **Azure Disks**: use disk snapshots for point-in-time recovery and operational rollback.
- **Databases**: use native backup/restore features, point-in-time restore, and geo-redundant options where available.

A practical backup policy might look like:

```text
- Daily full backup
- Hourly incremental or log backups for critical databases
- 30-day operational retention
- 90-day compliance retention
- Geo-redundant copy for tier-1 workloads
- Monthly restore test in pre-production
```

RPO and RTO influence rollback choices:

- If RPO is near zero, you need replication, log shipping, or synchronous patterns rather than relying on backup restore.
- If RTO is very short, rollback or failover is usually faster than restoring large datasets.
- If the change is reversible and data is intact, rollback is often the fastest option.
- If data is corrupted, restore is safer than rollback.
- If the region is unavailable, failover is the correct response, not rollback.

> If the application has already written bad data, a code rollback alone may not fix the problem. You may need restore, compensating scripts, or both.

**Azure recovery features that support rollback planning**

Azure provides several recovery mechanisms, and you should choose them based on the failure mode.

- **Azure Virtual Machines snapshots**: useful for point-in-time disk recovery, testing, and short-term rollback support.
- **Azure Site Recovery**: replicates VMs to another region and supports orchestrated failover and failback.
- **Azure Files snapshots and backup**: support recovery of deleted or modified files and shares.
- **Managed disk snapshots**: allow you to capture disk state before risky changes.
- **Azure Backup**: centralizes backup policy, retention, and restore operations for supported workloads.
- **Point-in-time restore** for supported databases: useful when corruption or bad writes occur.
- **Geo-redundant storage options**: help protect backups and data against regional loss.

Use these features differently:

- Use **snapshots** for quick operational recovery of disks or files.
- Use **backup** for policy-driven retention and restore.
- Use **ASR** for business continuity and regional failover.
- Use **native database restore** for logical corruption or schema/data mistakes.

**Database rollback patterns**

Database changes are often the hardest part of rollback planning because application code and schema changes can become tightly coupled. The safest pattern is to separate schema changes from app releases and keep them backward compatible.

Recommended patterns:

- **Backward-compatible migrations**: add new columns or tables first, keep old fields until all app instances are updated, then remove old structures later.
- **Expand-and-contract**: expand the schema to support both old and new app versions, then contract after the rollout stabilizes.
- **Compensating scripts**: write scripts that reverse a migration or correct data after a failed change.
- **Feature flags**: deploy code dormant, then enable behavior after validation.
- **Dual-write or read-compatibility**: temporarily support both old and new data paths during transition.

Avoid these anti-patterns:

- Dropping columns immediately after deployment.
- Renaming tables or fields without compatibility handling.
- Releasing schema and app changes in a single irreversible step.
- Assuming a code rollback will undo database writes.

Example migration strategy:

```sql
-- Expand phase
ALTER TABLE dbo.Orders ADD NewStatus nvarchar(50) NULL;
UPDATE dbo.Orders SET NewStatus = OldStatus;

-- App reads NewStatus when present, falls back to OldStatus

-- Contract phase after validation
ALTER TABLE dbo.Orders DROP COLUMN OldStatus;
```

A compensating rollback script might restore a previous value from an audit table:

```sql
UPDATE o
SET o.Status = a.PreviousStatus
FROM dbo.Orders o
JOIN dbo.OrderAudit a ON a.OrderId = o.OrderId
WHERE a.ChangeBatchId = @BatchId;
```

For exam scenarios, remember:

- If the schema change is backward compatible, rollback is simpler.
- If the schema change is destructive, you may need restore from backup.
- If the app can tolerate both versions, you can roll back the app independently.
- If data has been transformed irreversibly, rollback may not be enough.

**Rollback versus restore versus failover**

These three responses solve different problems.

| Response | Best when | Typical goal |
|---|---|---|
| Rollback | A recent deployment or config change caused the issue | Return to the previous known-good version |
| Restore | Data is corrupted, deleted, or logically damaged | Recover a clean point-in-time copy |
| Failover | A site, region, or primary service is unavailable | Keep the service running elsewhere |

Use rollback when:

- The issue started immediately after a deployment.
- The previous version is known to be stable.
- Data changes are compatible or minimal.
- You can revert quickly without losing important writes.

Use restore when:

- Data corruption is confirmed or strongly suspected.
- A bad migration changed data in a non-reversible way.
- You need a point-in-time copy before the bad change.
- The application version itself is not the root cause.

Use failover when:

- The primary region or zone is unavailable.
- The service cannot meet availability targets in place.
- Recovery in the same environment would take too long.
- You have a secondary site or region ready to serve traffic.

**Practical Azure examples**

- A bad app deployment causes 500 errors, but the database is healthy: use rollback.
- A migration deletes a column and the app has already written incompatible data: use restore or compensating recovery.
- An Azure region outage affects a production VM fleet: use ASR failover.
- A user deletes files from Azure Files: use share snapshots or backup restore.
- A managed disk becomes corrupted after a failed patch: restore from snapshot or backup.

**Testing rollback plans**

A rollback plan is only useful if it works under pressure. Test it regularly in lower environments and, where possible, in production-safe ways.

Useful validation methods include:

- **Pre-production restore verification**: restore backups into a test environment and confirm data integrity.
- **Game days**: planned exercises where teams practice incident response and rollback decisions.
- **Chaos testing**: controlled fault injection to validate resilience and recovery behavior.
- **Deployment rehearsals**: simulate release, rollback, and verification steps before production.
- **Tabletop exercises**: walk through the decision tree, approvals, and communications without changing systems.

Testing should verify:

- The runbook is current and complete.
- The team knows who approves rollback.
- The rollback time meets the RTO target.
- The restored data meets the RPO target.
- Monitoring detects failure quickly enough to act before more damage occurs.

A good test plan includes success criteria:

1. Rollback starts within the expected decision window.
2. The system returns to a healthy state.
3. Data integrity checks pass.
4. Stakeholders receive timely updates.
5. The team can explain whether rollback, restore, or failover was the correct choice.

**Operational controls that reduce rollback risk**

You can reduce the need for rollback by making changes safer.

- Use **blue-green deployments** to switch traffic between two environments.
- Use **canary releases** to expose a change to a small percentage of users first.
- Use **feature flags** to decouple deployment from release.
- Use **health probes** and automated gates to stop bad releases early.
- Use **versioned APIs** to preserve compatibility.
- Use **infrastructure as code** so environments are reproducible.

Example Azure CLI checks during a release:

```bash
az vm show -g rg-prod -n appvm01 --query "provisioningState"
az monitor metrics list --resource /subscriptions/<sub>/resourceGroups/rg-prod/providers/Microsoft.Compute/virtualMachines/appvm01 \
  --metric Percentage CPU --interval PT5M
az backup item list --vault-name prodVault --resource-group rg-backup
```

Example of a safe deployment pattern with a feature flag:

```json
{
  "FeatureFlags": {
    "NewCheckoutFlow": false
  }
}
```

You deploy the code first, validate it, then enable the flag. If the feature misbehaves, you disable the flag instead of rolling back the entire release.

**Exam-oriented scenario guidance**

AZ-305 questions often ask you to choose the most appropriate architecture response, not just the most technically powerful one. Use the failure mode to decide.

- If the question mentions a recent deployment and no data corruption, think **rollback**.
- If the question mentions deleted records, schema damage, or bad writes, think **restore**.
- If the question mentions regional outage or site failure, think **failover**.
- If the question asks for minimal data loss, prioritize **RPO** and replication.
- If the question asks for minimal downtime, prioritize **RTO** and automated recovery.

Common scenario patterns:

- A web app update breaks authentication, but the database is fine: rollback the app and keep the data.
- A database migration introduces bad values in a critical table: restore to a point before the migration or run a compensating fix if feasible.
- A zone failure affects a stateless service: fail over to another zone or region depending on design.
- A file share is accidentally modified by users: restore from Azure Files snapshot or backup.
- A VM image update fails during patching: revert using snapshot or redeploy from a known-good image.

**Checklist for a strong rollback-ready design**

- Define risks early and score them by likelihood and impact.
- Tie recovery choices to business RTO and RPO.
- Keep rollback separate from restore and failover decisions.
- Document a rollback runbook with ownership, approvals, and escalation.
- Design backups with frequency, retention, and geo-redundancy in mind.
- Use Azure recovery features that match the workload type.
- Keep database migrations backward compatible when possible.
- Test recovery with restore drills, game days, and chaos exercises.
- Validate that monitoring, communication, and decision-making are part of the plan.
