## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Post-migration optimization overview**

Post-migration optimization is the work you do after workloads are moved to Azure to make them cheaper, faster, safer, and easier to operate. Migration is not the finish line; it is the point where you can finally measure real usage and tune the environment based on actual demand.

The main goal is to align the deployed architecture with business outcomes:
- Reduce waste by right-sizing compute, storage, and databases.
- Improve reliability by removing single points of failure and validating recovery.
- Strengthen security by applying least privilege and continuous posture management.
- Improve performance by tuning tiers, caching, scaling, and network paths.
- Simplify operations by standardizing monitoring, automation, and governance.

A practical optimization cycle usually follows this pattern:
1. Measure current usage and cost.
2. Identify bottlenecks, idle resources, and risk.
3. Apply targeted changes.
4. Validate impact with metrics and tests.
5. Repeat as workloads and demand change.

**Optimization domains**

Post-migration tuning is easiest to manage when you break it into domains. Each domain has different signals and tools, but they should be reviewed together because a change in one area often affects another.

- **Cost optimization** focuses on eliminating unused capacity and matching service tiers to demand.
- **Performance optimization** focuses on latency, throughput, and resource saturation.
- **Reliability optimization** focuses on redundancy, backup, failover, and recovery objectives.
- **Security optimization** focuses on identity, network exposure, secrets, and policy compliance.
- **Operational optimization** focuses on monitoring, automation, and standardization.

**Cost optimization**

Cost optimization starts with visibility. You need to know what is consuming spend before you can reduce it. Azure Cost Management and Billing, resource tags, budgets, and recommendations help you identify waste and enforce accountability.

Common cost actions include:
- **Right-sizing** virtual machines, app tiers, and database tiers based on CPU, memory, and I/O patterns.
- **Shutting down** nonproduction resources outside business hours.
- **Deleting** orphaned disks, snapshots, public IPs, load balancers, and test resources.
- **Using reservations** for predictable steady-state workloads.
- **Using Azure Savings Plan for Compute** when usage is consistent but not tied to a single VM family.
- **Choosing lower-cost storage tiers** for infrequently accessed data.
- **Reviewing data egress** because outbound traffic can become a hidden cost driver.

Useful CLI examples:
```bash
# View current cost analysis for a subscription scope
az costmanagement query \
  --scope /subscriptions/<subscription-id> \
  --timeframe MonthToDate \
  --type ActualCost \
  --dataset '{
    "granularity": "Daily",
    "aggregation": {
      "totalCost": {
        "name": "PreTaxCost",
        "function": "Sum"
      }
    }
  }'
```

```bash
# Create a budget with a simple alert threshold
az consumption budget create \
  --amount 500 \
  --budget-name monthly-budget \
  --category Cost \
  --time-grain Monthly \
  --start-date 2026-04-01 \
  --end-date 2027-04-01
```

> Budget and cost command behavior can vary by Azure CLI version and subscription type. Validate syntax in your environment before using it in automation.

Cost optimization is not only about lowering spend. It is about spending intentionally on the resources that deliver value. A smaller bill is not a success if it causes outages or performance regressions.

**Performance optimization**

Performance tuning after migration should be driven by observed metrics, not assumptions from the source environment. Azure often changes the performance profile of a workload because storage latency, network paths, and service tiers differ from on-premises systems.

Key performance levers include:
- **Compute sizing**: increase or decrease VM size, change series, or move to burstable instances for dev/test.
- **Autoscaling**: scale out stateless services when demand rises and scale in when demand falls.
- **Caching**: use Azure Cache for Redis or application-level caching to reduce repeated reads.
- **Storage tuning**: choose the right disk type, IOPS, and throughput settings.
- **Database tuning**: use Query Store, indexing analysis, and query plan review.
- **Network tuning**: reduce cross-region traffic, place dependent services closer together, and use private connectivity where appropriate.

For application workloads, measure:
- CPU utilization
- Memory pressure
- Disk latency and queue depth
- Request latency
- Error rate
- Throughput per instance

For database workloads, focus on:
- Query duration
- Wait statistics
- Blocking and deadlocks
- Index usage
- Log write latency
- Tempdb or temp storage pressure

Example VM tuning workflow:
```bash
# Check VM size and current performance-related properties
az vm show \
  --resource-group rg-app \
  --name vm-app01 \
  --query "{size:hardwareProfile.vmSize, os:storageProfile.osDisk.osType, location:location}" \
  -o table
```

```bash
# Resize a VM after validating the target size supports the workload
az vm resize \
  --resource-group rg-app \
  --name vm-app01 \
  --size Standard_D8s_v5
```

Performance optimization should be validated with load testing and monitoring. A change that improves one metric can worsen another, such as lowering latency while increasing cost or reducing cache misses while increasing memory usage.

**Database optimization**

Database workloads often become the most visible post-migration tuning target because they expose both application inefficiencies and platform configuration issues. Azure SQL Database and Azure SQL Managed Instance both support monitoring and tuning features, but the exact capabilities depend on service tier and deployment model.

Common database optimization techniques:
- **Use Query Store** to capture query performance history and compare plans over time.
- **Review automatic tuning recommendations** for indexing and plan correction where supported.
- **Tune indexes carefully** by adding missing indexes only when they address real workload patterns.
- **Reduce expensive queries** by rewriting joins, filtering earlier, and avoiding unnecessary scans.
- **Match service tier to workload** using DTU or vCore sizing based on CPU, memory, and I/O demand.
- **Use read scale features where available** to offload read-heavy workloads.
- **Separate OLTP and reporting** when mixed workloads create contention.

> Automatic tuning is not a blanket fix. Index creation, index removal, and plan correction are workload-dependent and should be validated before broad rollout.

Example SQL diagnostics query:
```sql
SELECT
    qsqt.query_sql_text,
    qsp.plan_id,
    rs.avg_duration,
    rs.avg_cpu_time,
    rs.avg_logical_io_reads
FROM sys.query_store_query_text AS qsqt
JOIN sys.query_store_query AS qsq
    ON qsqt.query_text_id = qsq.query_text_id
JOIN sys.query_store_plan AS qsp
    ON qsq.query_id = qsp.query_id
JOIN sys.query_store_runtime_stats AS rs
    ON qsp.plan_id = rs.plan_id
ORDER BY rs.avg_duration DESC;
```

For Azure SQL, also consider:
- **Connection pooling** to reduce login overhead.
- **Parameterization** to improve plan reuse.
- **Maintenance windows** for index and statistics updates.
- **Geo-replication** or failover groups for availability, not just performance.

**Storage optimization**

Storage often accumulates waste after migration because teams keep source-system habits, such as overprovisioned disks or retaining data in premium tiers longer than needed.

Optimization actions include:
- **Choosing the right access tier** for blobs: hot, cool, or archive.
- **Using lifecycle management** to move data automatically between tiers.
- **Deleting stale snapshots and backups** that exceed retention requirements.
- **Matching disk type to workload**: Premium SSD for latency-sensitive workloads, Standard SSD for moderate performance, and Standard HDD for low-cost noncritical data.
- **Using shared access patterns carefully** because unnecessary replication can increase cost.

Example lifecycle policy:
```json
{
  "rules": [
    {
      "enabled": true,
      "name": "move-to-cool-and-archive",
      "type": "Lifecycle",
      "definition": {
        "filters": {
          "blobTypes": ["blockBlob"],
          "prefixMatch": ["logs/"]
        },
        "actions": {
          "baseBlob": {
            "tierToCool": { "daysAfterModificationGreaterThan": 30 },
            "tierToArchive": { "daysAfterModificationGreaterThan": 90 },
            "delete": { "daysAfterModificationGreaterThan": 365 }
          }
        }
      }
    }
  ]
}
```

Storage tuning should also account for application behavior:
- Large sequential reads favor different settings than many small random reads.
- Backup retention can dominate storage cost if not reviewed.
- Cross-region replication improves resilience but increases spend.

**Security optimization**

After migration, security should move from project-mode hardening to continuous governance. The goal is to reduce exposure while keeping the workload usable.

Key security improvements:
- **Identity hardening** with Microsoft Entra ID, least privilege, and role-based access control (RBAC).
- **Network segmentation** using virtual networks, subnets, network security groups, and private endpoints.
- **Secret management** with Azure Key Vault instead of embedded credentials.
- **Posture management** with Microsoft Defender for Cloud and Azure Policy.
- **Just-in-time access** for administrative tasks where supported.
- **Conditional access** and MFA for human access paths.

Important distinctions:
- **Network Security Groups** filter traffic, but they do not replace application-layer validation.
- **Private Link** provides private access to platform services over a private endpoint.
- **Application Gateway** can front private backends, but it is not itself equivalent to a private-access service.
- **Azure Front Door** improves global entry and acceleration, but origin protection still needs backend validation and proper access controls.

Example of a secure app pattern:
- Public entry through Front Door or Application Gateway.
- Backend service exposed only through private endpoint or restricted subnet.
- Secrets stored in Key Vault.
- Managed identity used for service-to-service authentication.
- Policy enforces approved SKUs, locations, and tagging.

**Governance and policy optimization**

Governance keeps the post-migration environment from drifting back into waste and risk. Azure Policy, management groups, tags, and locks help you enforce standards at scale.

Use governance to:
- Require tags such as `CostCenter`, `Owner`, and `Environment`.
- Restrict allowed regions and SKUs.
- Enforce diagnostic settings.
- Prevent public exposure where it is not allowed.
- Audit encryption, backup, and identity settings.

Example policy definition pattern:
```json
{
  "properties": {
    "displayName": "Require tags on resources",
    "policyType": "Custom",
    "mode": "All",
    "parameters": {
      "tagName": {
        "type": "String",
        "metadata": {
          "displayName": "Tag name"
        }
      }
    },
    "policyRule": {
      "if": {
        "field": "[concat('tags[', parameters('tagName'), ']')]",
        "exists": "false"
      },
      "then": {
        "effect": "deny"
      }
    }
  }
}
```

Governance works best when it is paired with automation:
- Use policy to prevent bad deployments.
- Use remediation tasks to fix existing drift.
- Use templates or Bicep to standardize future deployments.

**Monitoring and observability**

You cannot optimize what you cannot measure. After migration, monitoring should shift from basic health checks to workload-specific observability.

Core Azure monitoring tools:
- **Azure Monitor** for metrics, logs, and alerts.
- **Log Analytics** for centralized query and correlation.
- **Application Insights** for application telemetry.
- **Network Watcher** for network diagnostics.
- **Service-specific diagnostics** for databases, storage, and compute.

What to monitor:
- Availability and uptime
- Latency and throughput
- Error rates
- Resource saturation
- Cost anomalies
- Security events
- Backup success and restore readiness

Example Kusto Query Language (KQL) query:
```kusto
AzureActivity
| where TimeGenerated > ago(7d)
| where ActivityStatusValue == "Failed"
| summarize FailedOperations = count() by OperationNameValue, ResourceGroup
| order by FailedOperations desc
```

Good alerting is actionable:
- Alert on symptoms that matter to users.
- Avoid noisy thresholds that trigger constantly.
- Route alerts to the right team with clear ownership.
- Include runbooks or automation for common remediation.

**Automation and operational efficiency**

Automation reduces human error and keeps optimization consistent. It is especially useful after migration because many workloads need recurring actions such as scaling, cleanup, patching, and compliance checks.

Common automation targets:
- Start and stop schedules for nonproduction resources
- Tag enforcement and cleanup
- Backup validation
- Patch orchestration
- Scaling actions based on metrics
- Cost anomaly detection and reporting

Example Azure CLI automation pattern:
```bash
#!/usr/bin/env bash
set -euo pipefail

RG="rg-nonprod"
VM="vm-dev01"

az vm deallocate -g "$RG" -n "$VM"
```

For more advanced operations, use:
- **Azure Automation** for runbooks and scheduled tasks.
- **Logic Apps** for workflow-based remediation.
- **Functions** for event-driven automation.
- **Bicep** or **ARM templates** for repeatable infrastructure changes.

**Reliability and disaster recovery optimization**

Post-migration optimization should confirm that the workload can recover within business targets. Reliability is not only about having backups; it is about proving that restore and failover work.

Review:
- **Backup retention** against compliance and recovery objectives.
- **Recovery point objective (RPO)**, the maximum acceptable data loss.
- **Recovery time objective (RTO)**, the maximum acceptable downtime.
- **Redundancy** at the zone, region, and service level.
- **Failover testing** for critical systems.
- **Dependency mapping** so you know what must recover first.

Practical checks:
- Test restore of a database, VM, or file share.
- Validate application startup after restore.
- Confirm DNS, certificates, and secrets are available.
- Verify that autoscaling and health probes behave correctly after failover.

**Common exam scenarios**

AZ-305 questions often ask you to choose the best post-migration improvement, not the most technically impressive one. Look for the option that best matches the business goal.

Typical scenario patterns:
- A workload is overprovisioned after migration: choose right-sizing or autoscaling.
- A database is slow after migration: inspect Query Store, indexing, and tier selection.
- Costs are rising unexpectedly: use budgets, tags, reservations, and cleanup.
- Security review finds public exposure: use private endpoints, RBAC, and policy.
- Operations are manual and inconsistent: use automation and governance.
- Recovery has not been tested: validate backup and failover procedures.

**Decision guidance**

When selecting an optimization approach, ask:
- Is the issue cost, performance, security, reliability, or operations?
- Is the workload steady-state or variable?
- Is the bottleneck compute, storage, network, or database?
- Can you fix the issue with configuration, or do you need an architectural change?
- What business risk does the change introduce?

A good post-migration plan usually starts with low-risk, high-value changes:
- Remove unused resources.
- Apply tags and budgets.
- Turn on monitoring and alerts.
- Right-size obvious overprovisioning.
- Validate backups and restore paths.
- Then move to deeper tuning such as scaling, caching, and redesign.

**Key takeaways**

Post-migration optimization is an iterative discipline, not a one-time cleanup task. The best Azure architects use telemetry, governance, and automation to keep workloads aligned with business needs as demand changes.

- Measure before you change.
- Optimize across cost, performance, security, reliability, and operations together.
- Prefer small, validated improvements over broad untested changes.
- Use Azure-native tools to enforce standards and reduce drift.
- Reassess regularly because the best configuration today may be inefficient next quarter.
