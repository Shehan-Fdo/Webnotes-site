## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Migration readiness overview**

Before you move workloads, you need a target environment that can support identity, routing, name resolution, access control, and secure administration. Migration failures often come from missing prerequisites rather than from the migration tool itself. In Azure, that means you should validate the landing zone, subscription placement, network topology, and identity integration before cutover.

A good migration design answers four questions:

- Where does the workload live in the Azure governance model?
- How will users, admins, and applications authenticate and authorize?
- How will traffic reach the workload securely and resolve names correctly?
- How will you validate the design before you switch production?

**Landing zone and subscription prerequisites**

A landing zone is the governance and platform foundation for workloads. For migration projects, you usually place subscriptions into a management group hierarchy that reflects business, platform, and workload boundaries. This helps you apply policy, RBAC (role-based access control), and budget controls consistently.

Key planning points:

- **Management groups** define governance scope above subscriptions.
- **Subscriptions** should align to workload, environment, or business unit boundaries.
- **Resource groups** should group resources by lifecycle and deployment unit, not by arbitrary convenience.
- **Policy** should enforce baseline controls such as allowed regions, tagging, private endpoint requirements, and diagnostic settings.
- **Separation of duties** should keep platform administration distinct from application administration.

For migration teams, plan access early:

- Assign only the minimum roles needed for discovery, deployment, and validation.
- Use **Privileged Identity Management (PIM)** for just-in-time elevation.
- Separate build, security, and operations responsibilities where possible.
- Avoid giving migration engineers broad Owner access across all subscriptions unless there is a documented exception.

A practical subscription model often includes:

- A platform subscription for shared services such as hub networking, DNS, and identity integration.
- A landing-zone subscription for each application or workload group.
- A management subscription for monitoring, security tooling, and governance services.

**Identity prerequisites for migration**

Identity is usually the first dependency to validate because users, admins, and apps need a trusted authentication path before the workload is usable. In hybrid migrations, you often integrate on-premises Active Directory Domain Services (AD DS) with Microsoft Entra ID.

Common identity patterns include:

- **Password hash synchronization (PHS)**: hashes of on-premises passwords are synchronized to Microsoft Entra ID. This is the simplest and most resilient sign-in method for many migrations.
- **Pass-through authentication (PTA)**: authentication is validated against on-premises infrastructure in real time. This keeps password validation on-premises but adds dependency on the agent path.
- **Federation**: sign-in is redirected to an identity provider such as AD FS. This is more complex and usually reserved for specific requirements.
- **Cloud sync**: a lightweight synchronization option for some directory scenarios, often simpler than full Entra Connect deployments.

> Use the current product name **Microsoft Entra Domain Services**. The older name **Azure AD DS** may still appear in documentation and exam discussions, but treat it as legacy terminology.

When planning hybrid identity, confirm:

- User and group objects are synchronized correctly.
- UPNs (user principal names) are routable and match sign-in expectations.
- Password policies and sign-in methods are understood before cutover.
- Conditional Access policies do not block migration tooling or admin access unexpectedly.
- Service accounts and application identities are mapped to the target authentication model.

**Hybrid identity integration details**

If your source environment uses on-premises AD DS, you need a synchronization and authentication strategy before migration. Microsoft Entra Connect or cloud sync typically handles directory synchronization, while PHS or PTA handles user sign-in.

Practical considerations:

- **PHS** is often preferred for simplicity and resilience.
- **PTA** is useful when you need on-premises password validation, but you must size and monitor the agents.
- **Seamless SSO** can reduce prompts for domain-joined devices in supported scenarios.
- **Writeback** features may be needed if you must support password reset or device writeback scenarios.

A migration checklist for identity:

1. Verify the source AD DS health and replication.
2. Confirm the synchronization scope for users, groups, and devices.
3. Validate sign-in methods and fallback behavior.
4. Test Conditional Access with pilot users.
5. Confirm break-glass accounts exist and are excluded from risky policy dependencies.
6. Document how service principals, managed identities, and legacy service accounts will be handled.

**Application identity and modernization paths**

Application identity is not just about users. You also need to decide how applications authenticate to Azure services and how legacy apps will be modernized.

Common patterns:

- **Managed identities** for Azure resources that need to access other Azure services without storing secrets.
- **Service principals** for application-to-application access where a managed identity is not possible.
- **OAuth 2.0 / OpenID Connect (OIDC)** for modern web and API authentication.
- **SAML** for enterprise single sign-on with many SaaS and legacy web apps.
- **LDAP/Kerberos** for legacy line-of-business apps that still depend on domain services.

A common migration path for a legacy app:

- If the app needs LDAP or Kerberos, place it on a network that can reach **Microsoft Entra Domain Services** and join it to that managed domain.
- If the app can be modernized, move it to OAuth/OIDC and use Microsoft Entra ID directly.
- If the app needs only Azure resource access, replace stored credentials with a managed identity.

Example: enabling a VM to access Azure Key Vault with a managed identity:

```bash
az vm identity assign \
  --resource-group rg-app-prod \
  --name vm-app01

az role assignment create \
  --assignee-object-id <principal-id> \
  --assignee-principal-type ServicePrincipal \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<sub-id>/resourceGroups/rg-security/providers/Microsoft.KeyVault/vaults/kv-prod
```

**Microsoft Entra Domain Services for legacy app migration**

Microsoft Entra Domain Services provides managed domain capabilities such as domain join, LDAP, Kerberos, and NTLM support without you managing domain controllers. It is useful when you are migrating applications that cannot be rewritten immediately.

Use it when:

- The app requires LDAP bind or Kerberos authentication.
- You need domain join for legacy Windows workloads in Azure.
- You want managed domain services without deploying and patching domain controllers.

Be careful with expectations:

- It is not a full replacement for on-premises AD DS in every scenario.
- You do not get the same administrative control as with self-managed domain controllers.
- Some advanced AD DS features are not available.

A typical migration example:

- A legacy app uses LDAP to authenticate users.
- You synchronize identities from on-premises AD DS to Microsoft Entra ID.
- You enable Microsoft Entra Domain Services in the target virtual network.
- You join the app server to the managed domain.
- You update the app to point LDAP queries at the managed domain controllers.

**Network landing zone and connectivity prerequisites**

Network design determines whether the migrated workload can reach users, dependencies, and platform services. For most enterprise migrations, you should decide whether the target uses a hub-spoke topology, a virtual WAN design, or a simpler flat network.

Common prerequisites:

- IP address space must not overlap with on-premises ranges.
- Subnets should be sized for current and future scale.
- Route propagation and user-defined routes must be planned before deployment.
- Security controls such as NSGs (network security groups) and Azure Firewall should be defined early.
- DNS resolution must work for both internal and external names.

A hub-spoke design is common because it centralizes shared services:

- The **hub** hosts shared connectivity, firewalling, DNS, Bastion, and sometimes VPN or ExpressRoute gateways.
- The **spokes** host application workloads.
- Peering allows traffic flow between hub and spokes, but gateway transit must be designed correctly.

Gateway transit rules:

- Configure `--allow-gateway-transit` on the hub-to-spoke peering only when the hub VNet already has a gateway.
- Configure `--use-remote-gateways` on the spoke-to-hub peering when the spoke should use the hub gateway.
- Do not enable gateway transit unless the design requires it.

Example peering commands:

```bash
az network vnet peering create \
  --resource-group rg-network \
  --vnet-name vnet-hub \
  --name hub-to-spoke1 \
  --remote-vnet vnet-spoke1 \
  --allow-vnet-access \
  --allow-forwarded-traffic \
  --allow-gateway-transit

az network vnet peering create \
  --resource-group rg-network \
  --vnet-name vnet-spoke1 \
  --name spoke1-to-hub \
  --remote-vnet vnet-hub \
  --allow-vnet-access \
  --allow-forwarded-traffic \
  --use-remote-gateways
```

**Private access to PaaS services**

Private access to platform services is a major migration prerequisite because many workloads should not expose public endpoints during or after migration. Azure provides several connectivity options:

- **Private endpoints** give a private IP address in your VNet for a specific Azure PaaS resource.
- **Service endpoints** extend VNet identity to a service over the Azure backbone, but the service still uses its public endpoint.
- **Public endpoints** may still be used temporarily during migration, but they increase exposure and often conflict with zero-trust goals.

Choose the model based on security and operational needs:

- Use **private endpoints** when you need private-only access, tighter network control, and consistent enterprise DNS design.
- Use **service endpoints** when you need simpler connectivity and the service supports them, but you do not require a private IP.
- Use public access only when the service or migration phase requires it and you have compensating controls.

Private DNS is essential for private endpoints. The DNS zone name is service-specific, so you must create the correct zone for each service. For example:

```bash
az network private-dns zone create \
  --resource-group rg-network \
  --name privatelink.database.windows.net
```

That zone is appropriate for Azure SQL private endpoints, but other services use different `privatelink.<service>` zones. Always verify the service-specific DNS zone before deployment.

A typical private endpoint workflow:

1. Create or identify the target PaaS resource.
2. Create the private endpoint in the correct subnet.
3. Link the correct private DNS zone to the VNet.
4. Confirm DNS resolves the service name to the private IP.
5. Test application connectivity before cutover.

**DNS design for hybrid and migration scenarios**

DNS is often the hidden dependency that breaks migrations. You need a design for both Azure-native name resolution and hybrid name resolution.

Common options:

- **Azure-provided DNS** is simple and works well for basic Azure resource resolution.
- **Custom DNS servers** are needed when you must resolve on-premises names, AD DS records, or split-horizon zones.
- **Azure DNS Private Resolver** can bridge on-premises and Azure DNS scenarios without deploying custom DNS VMs in many cases.

Use Azure-provided DNS when:

- The workload is mostly Azure-native.
- You do not need to resolve on-premises internal names from Azure.
- You want the simplest operational model.

Use custom DNS when:

- You need conditional forwarding to on-premises zones.
- You host Active Directory-integrated zones.
- You need to resolve private endpoint zones and internal application names consistently across environments.

A practical hybrid DNS pattern:

- Azure VMs use custom DNS servers or Azure DNS Private Resolver.
- The custom DNS servers forward `privatelink.*` zones to the appropriate resolver path.
- On-premises DNS forwards Azure-specific zones when needed.
- You test both forward and reverse resolution for critical services.

Example of a Windows DNS conditional forwarder concept:

```powershell
Add-DnsServerConditionalForwarderZone `
  -Name "privatelink.database.windows.net" `
  -MasterServers 10.10.0.4,10.10.0.5 `
  -ReplicationScope "Forest"
```

**Secure admin access prerequisites**

Administrative access should be designed before migration, not after. Azure Bastion is a common choice because it provides browser-based RDP (Remote Desktop Protocol) and SSH (Secure Shell) access without exposing public IPs on VMs.

Bastion prerequisites to remember:

- It requires a dedicated subnet named `AzureBastionSubnet`.
- The subnet must be large enough for the chosen SKU and scale requirements.
- The target VNet must be able to route to the workload subnets.
- NSGs and route tables must not block required Bastion traffic.
- Premium features and scaling requirements depend on the SKU.

Example subnet creation:

```bash
az network vnet subnet create \
  --resource-group rg-network \
  --vnet-name vnet-hub \
  --name AzureBastionSubnet \
  --address-prefixes 10.0.10.0/26
```

Operational guidance:

- Use Bastion for admin access to private workloads.
- Use jump hosts only when Bastion is not available or does not meet a specific requirement.
- Restrict inbound management ports on workload subnets.
- Prefer just-in-time access and PIM for privileged operations.

**Connectivity planning for VPN and ExpressRoute**

Migration projects often need private connectivity between on-premises and Azure. The two main options are site-to-site VPN and ExpressRoute.

Use VPN when:

- You need faster deployment and lower cost.
- Bandwidth and latency requirements are moderate.
- Temporary connectivity is acceptable during migration.

Use ExpressRoute when:

- You need private, predictable connectivity.
- You have higher bandwidth or lower latency requirements.
- You are building a long-term enterprise connectivity model.

For ExpressRoute, remember that circuit creation is provider-specific and requires details such as bandwidth, peering location, and service provider information. A generic create command is not enough by itself.

Example pattern:

```bash
az network express-route circuit create \
  --resource-group rg-network \
  --name er-circuit-prod \
  --location "East US" \
  --bandwidth 1000 \
  --sku-tier Standard \
  --sku-family MeteredData
```

You still need provider-specific provisioning details and, in many cases, a service provider to complete the circuit.

For VPN gateways, validate the gateway, connections, and local network gateway objects rather than relying on a single generic inventory command. Useful checks include:

```bash
az network vnet-gateway show \
  --resource-group rg-network \
  --name vng-hub

az network local-gateway list \
  --resource-group rg-network

az network vpn-connection show \
  --resource-group rg-network \
  --name conn-onprem
```

**Validation before cutover**

Validation is a prerequisite, not a post-migration task. You should test identity, routing, DNS, and application access before you move production traffic.

Validation checklist:

- Confirm users can authenticate with the intended sign-in method.
- Confirm admins can reach management endpoints through Bastion or approved jump paths.
- Confirm DNS resolves internal names and private endpoint names correctly.
- Confirm routes send traffic through the intended next hop.
- Confirm NSGs and firewall rules allow only the required ports.
- Confirm application dependencies such as databases, queues, and APIs are reachable.
- Confirm monitoring and logging are enabled for the migrated resources.

Useful validation commands:

```bash
az network nic show-effective-route-table \
  --resource-group rg-app-prod \
  --name nic-app01

az network nic show-effective-nsg \
  --resource-group rg-app-prod \
  --name nic-app01
```

For application testing, use both network and identity checks:

- Test sign-in with a pilot user.
- Test service-to-service authentication with a managed identity or service principal.
- Test legacy authentication paths such as LDAP or Kerberos if the app depends on them.
- Test name resolution from both Azure and on-premises perspectives.

**Common migration scenarios**

A few common patterns appear repeatedly in exam scenarios:

- **Lift-and-shift VM migration**
  - Keep the existing authentication model if possible.
  - Ensure DNS, routing, and admin access are ready before moving the VM.
  - Recreate dependencies such as file shares, certificates, and service accounts.

- **Legacy app modernization**
  - Move from LDAP/Kerberos to Microsoft Entra ID where possible.
  - Use Microsoft Entra Domain Services as an interim step when the app cannot be rewritten immediately.
  - Replace stored secrets with managed identities for Azure service access.

- **Private PaaS migration**
  - Use private endpoints for databases and storage.
  - Create the correct private DNS zones for each service.
  - Validate that application subnets can resolve and reach the private IPs.

- **Hybrid identity migration**
  - Synchronize identities before workload cutover.
  - Pilot Conditional Access and MFA (multifactor authentication) policies.
  - Keep break-glass accounts outside normal dependency chains.

**Exam-focused design reminders**

- Design the landing zone before the workload migration.
- Treat DNS as a first-class dependency.
- Use private endpoints when you need private-only access to PaaS.
- Use Microsoft Entra Domain Services for legacy domain-dependent apps, not as a universal identity replacement.
- Use PHS or PTA based on resilience and operational requirements.
- Use Bastion or equivalent secure admin access instead of public management ports.
- Validate routes, DNS, and authentication before cutover.
- Apply least privilege and PIM to migration teams.
