## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Migration sequencing fundamentals**

Data migration sequencing is the order in which you move data, dependencies, and application components from a source environment to Azure. The right sequence reduces downtime, preserves data integrity, and avoids breaking upstream or downstream systems.

In Azure architecture scenarios, sequencing is rarely just “move the database first.” You usually need to account for:

- **Data dependencies** such as foreign keys, reference data, and shared schemas.
- **Application dependencies** such as APIs, ETL (extract, transform, load) jobs, reports, scheduled tasks, and integration endpoints.
- **Operational constraints** such as downtime windows, compliance requirements, and recovery objectives.
- **Tooling constraints** such as whether the migration is online, offline, bulk, or replicated.

A good sequence answers four questions:

1. What must exist before the cutover target can accept traffic?
2. What can be migrated in parallel without breaking consistency?
3. What must be frozen or paused before final synchronization?
4. What is the exact rollback or failback path if validation fails?

**Core sequencing principle**

The safest order is usually:

1. Prepare the target platform.
2. Migrate or replicate data.
3. Migrate dependent services and integrations.
4. Validate data and application behavior.
5. Cut over traffic.
6. Monitor, reconcile, and only then decommission the source.

That order changes when the migration method changes. For example, an offline bulk move prioritizes a short freeze window, while an online replication-based move prioritizes continuous synchronization and a controlled cutover.

**Common migration scenarios and typical order**

Different Azure migration scenarios have different sequencing patterns. Exam questions often test whether you can choose the correct order for the workload type.

| Scenario | Typical sequence | Why this order matters |
|---|---|---|
| On-premises SQL Server to Azure SQL Database | Assess compatibility, remediate schema, migrate schema and data, sync deltas if supported, validate, cut over app connection strings | Azure SQL Database is a platform service, so compatibility and schema fit matter before data movement |
| SQL Server to Azure SQL Managed Instance | Assess, provision managed instance, configure networking and authentication, migrate schema/data, run validation, cut over application and jobs | Managed Instance supports more SQL Server features, but network and instance-level dependencies still need planning |
| File shares to Azure Storage | Inventory shares, map permissions and ownership, seed bulk copy, sync deltas, switch clients, validate access | File migrations often depend on ACLs, path changes, and client reconfiguration |
| Data warehouse migration | Land source extracts, transform and stage, load target warehouse, validate aggregates, switch BI/reporting, retire old pipelines | Warehouses usually require ETL/ELT sequencing and reconciliation of aggregates, not just raw copy |

**Tool selection and sequencing**

Azure-native tools influence the order of operations. Choose the tool based on the migration pattern, not the other way around.

- **Azure Migrate** is useful for discovery, assessment, and orchestrating server or workload migration planning.
- **Azure Database Migration Service (DMS)** is commonly used for database migrations and supports both offline and online approaches depending on source and target.
- **Azure Data Factory** is appropriate when you need pipeline orchestration, staged loads, transformations, or repeatable data movement across systems.
- **Azure Data Box** is appropriate for large initial seeding when network transfer is too slow or impractical.
- **AzCopy** is useful for file and blob transfers when you need scriptable, high-throughput copy operations.
- **Robocopy** is often used for Windows file share migrations where preserving metadata and retry behavior matters.

> Do not assume a single tool handles discovery, bulk transfer, synchronization, validation, and cutover. In many real migrations, you combine tools.

**Sequencing by migration mode**

Online and offline migrations require different sequencing.

**Offline migration**

Use offline migration when downtime is acceptable and you want a simpler cutover.

- Prepare the target environment first.
- Perform a full data copy or restore.
- Stop writes on the source.
- Run a final delta copy if available.
- Validate the target.
- Switch applications to the target.
- Monitor and keep the source available for rollback until confidence is high.

This approach is common when:
- The dataset is manageable within the downtime window.
- The application cannot tolerate dual writes.
- The source system cannot support continuous replication.

**Online migration**

Use online migration when you need to minimize downtime.

- Provision the target and network connectivity.
- Start continuous replication or synchronization.
- Keep source writes active during the sync phase.
- Monitor replication lag and data drift.
- Freeze writes during the cutover window.
- Apply final synchronization.
- Validate the target.
- Redirect traffic and monitor closely.

This approach is common when:
- The business requires low downtime.
- The source and target support replication or change capture.
- You can tolerate a more complex cutover and rollback plan.

**Important distinction**

Online migration does not mean “no downtime.” It means you reduce downtime to the final cutover window. You still need a write freeze, final sync, and validation gate.

**Sequencing for databases**

Database migrations are often the most exam-relevant because they combine schema, data, security, and application dependencies.

**On-premises SQL Server to Azure SQL Database**

Azure SQL Database is a database-as-a-service target, so you must sequence compatibility checks before migration.

- Assess unsupported features such as SQL Agent jobs, cross-database queries, server-level objects, and certain CLR or file-related dependencies.
- Remediate schema and code before data migration.
- Migrate schema and data using a suitable method such as DMS, bacpac, or scripted deployment depending on the scenario.
- Recreate application-level dependencies such as connection strings, identity, and secrets.
- Replace server-level automation with Azure-native alternatives such as Elastic Jobs, Azure Automation, or Logic Apps where appropriate.
- Validate query performance, indexing, and transaction behavior.
- Cut over the application after final synchronization and acceptance testing.

**SQL Server to Azure SQL Managed Instance**

Managed Instance is often chosen when you need broader SQL Server compatibility.

- Assess instance-level features, linked servers, SQL Agent jobs, credentials, and SQL Server Agent dependencies.
- Provision the managed instance in the correct virtual network and subnet.
- Configure connectivity, DNS, and authentication.
- Migrate databases and, where needed, instance-level objects.
- Validate jobs, maintenance tasks, and application connectivity.
- Cut over application traffic and scheduled tasks.
- Monitor for feature gaps or behavior differences.

Managed Instance can reduce refactoring compared with Azure SQL Database, but you still need to sequence networking and identity before application cutover.

**Practical DMS example**

A typical online database migration with Azure Database Migration Service follows this pattern:

```text
1. Assess source database compatibility.
2. Provision target Azure SQL Database or Managed Instance.
3. Create the DMS instance and network access.
4. Start full load and continuous sync.
5. Monitor latency and errors.
6. Freeze source writes.
7. Wait for replication to catch up.
8. Validate row counts and critical queries.
9. Redirect the application.
10. Keep source in read-only or standby mode until acceptance is complete.
```

**File share migration sequencing**

File share migrations are often underestimated because permissions and client paths matter as much as file content.

- Inventory shares, folders, ACLs (access control lists), and ownership.
- Identify active writers and scheduled jobs that touch the share.
- Seed the target storage account or file share.
- Sync incremental changes until cutover.
- Update application paths, mapped drives, and service accounts.
- Validate permissions, file integrity, and application access.
- Decommission or archive the source only after a stable period.

For large file sets, you may use:

```bash
azcopy copy "D:\Shares\Finance" "https://mystorageaccount.file.core.windows.net/finance?SAS_TOKEN" --recursive=true
```

For Windows file servers, a common pattern is:

```powershell
robocopy \\source\share \\target\share /MIR /COPY:DATSOU /R:2 /W:5 /MT:32 /LOG:C:\logs\share-migration.log
```

Use `robocopy` carefully because mirroring can delete files on the target if the source is missing them. That is useful for final sync, but risky for early seeding if the target is not yet authoritative.

**Data warehouse migration sequencing**

Warehouse migrations usually require a staged approach because reporting and ETL depend on consistent aggregates.

- Inventory source systems, staging tables, transformations, and downstream reports.
- Build the target warehouse schema and loading pipelines.
- Seed historical data in bulk.
- Reconcile facts, dimensions, and aggregates.
- Run parallel loads if the source and target both need to stay active during transition.
- Switch BI tools, semantic models, and reports after validation.
- Retire old ETL jobs only after the new pipeline proves stable.

A common pattern is to migrate in this order:

1. Source extraction.
2. Staging and transformation.
3. Warehouse load.
4. Semantic layer and reporting.
5. Downstream consumers.

That order matters because reports and dashboards should not point to a partially loaded warehouse.

**Sequencing dependent application components**

Data rarely moves alone. You must sequence the surrounding application stack.

- **ETL jobs** should usually be paused or redirected before final cutover to prevent writes to the old target.
- **Scheduled tasks** should be disabled or rescheduled so they do not run against both environments.
- **Reports and dashboards** should be validated against the new data source before users are switched.
- **APIs and integration endpoints** should be updated after the target is ready and tested.
- **Secrets, certificates, and managed identities** should be in place before application cutover.
- **DNS and connection strings** should be updated only after the target is validated.

A practical dependency order is:

1. Identity and access.
2. Network connectivity.
3. Data platform.
4. Data synchronization.
5. Application configuration.
6. Scheduled jobs and integrations.
7. User traffic cutover.

**Validation gates and acceptance criteria**

Validation should happen before, during, and after cutover. Treat validation as a sequence of gates, not a single final check.

**Pre-cutover validation**

- Confirm schema parity.
- Confirm permissions and role assignments.
- Confirm connectivity from application subnets.
- Confirm replication lag is within tolerance.
- Confirm backup and restore strategy exists on the target.

**Cutover validation**

- Compare row counts or object counts for critical tables and shares.
- Reconcile key business totals such as balances, orders, or inventory.
- Run smoke tests for login, read, write, and reporting paths.
- Verify scheduled jobs and integrations are disabled on the source and enabled on the target as intended.

**Post-cutover validation**

- Monitor error rates, latency, and failed jobs.
- Compare business metrics against the source baseline.
- Confirm no unexpected writes continue to the old system.
- Keep a rollback window open until acceptance criteria are met.

Useful acceptance criteria examples:

- Row count variance is zero for critical tables.
- Aggregate variance is within an agreed threshold, such as less than 0.1% for non-financial reporting data.
- All critical user journeys pass smoke tests.
- Replication lag is zero or below the agreed cutover threshold.
- No Sev 1 or Sev 2 incidents occur during the observation window.

**Rollback and failback sequencing**

Rollback is not the same as “switch back if something looks wrong.” The rollback plan must preserve consistency.

For offline migrations:
- Keep the source unchanged until the target is accepted.
- If validation fails, discard or isolate the target changes.
- Restore the source as the active system.
- Re-run remediation before attempting another cutover.

For online migrations:
- Decide in advance whether the source remains writable, read-only, or frozen during the cutover window.
- If cutover fails, stop target writes immediately.
- Assess whether the source can safely resume as the system of record.
- Reconcile any divergence before re-enabling normal operations.
- Avoid assuming you can simply turn source writes back on if the target has already accepted writes.

> Once both environments have diverged, a rollback usually requires a deliberate failback plan, not a simple toggle.

A safe rollback sequence is:

1. Stop traffic to the target.
2. Preserve logs and transaction evidence.
3. Determine whether any writes reached the target.
4. Decide whether source data can be safely resumed.
5. Reconcile or restore data as needed.
6. Repoint applications only after consistency is confirmed.

**DNS, endpoints, and cutover timing**

DNS-based cutover is common, but TTL reduction alone does not guarantee fast client switchover.

- Lower TTL in advance where DNS is used for endpoint changes.
- Verify client behavior, because some resolvers and applications cache DNS longer than the TTL.
- Account for persistent connections, connection pools, and service-side caching.
- Plan for a dual-run period if clients do not refresh immediately.

For application endpoints, you may also need to update:

- Connection strings.
- Private endpoints or firewall rules.
- Load balancer backends.
- API gateway routes.
- Service discovery records.

**When constraints conflict**

Exam questions often ask you to choose the best sequence when goals conflict.

**Minimize downtime**
- Prefer online replication.
- Seed data early.
- Pre-stage dependencies.
- Keep the final cutover window short.

**Preserve data integrity**
- Prefer offline or controlled cutover if the workload is highly sensitive.
- Freeze writes before final sync.
- Validate reconciliation before redirecting traffic.

**Meet compliance requirements**
- Ensure encryption, auditing, and access controls are in place before data movement.
- Restrict who can access source and target during migration.
- Retain logs and evidence for validation and rollback.

**Reduce operational risk**
- Migrate in waves.
- Start with noncritical data or a pilot workload.
- Validate each wave before expanding scope.
- Keep the source available until acceptance is complete.

**Practical sequencing patterns**

**Pattern 1: Lift-and-shift server migration**
1. Assess server dependencies.
2. Replicate the VM or workload.
3. Validate boot, network, and identity.
4. Cut over DNS or load balancer traffic.
5. Monitor and decommission later.

**Pattern 2: Database modernization**
1. Assess compatibility.
2. Remediate schema and code.
3. Provision target service.
4. Seed and sync data.
5. Validate application behavior.
6. Cut over application and integrations.
7. Retire legacy jobs and maintenance tasks.

**Pattern 3: File share modernization**
1. Inventory shares and permissions.
2. Create target storage.
3. Bulk copy initial data.
4. Sync deltas.
5. Update clients and service accounts.
6. Validate access and integrity.
7. Freeze and decommission source.

**Pattern 4: Analytics platform migration**
1. Build target landing zone and warehouse.
2. Migrate ingestion pipelines.
3. Load historical data.
4. Validate aggregates and reports.
5. Switch BI consumers.
6. Decommission old ETL and reporting endpoints.

**CLI and orchestration examples**

Azure CLI can help you automate parts of the sequence, especially provisioning and validation.

```bash
az group create --name rg-migration --location eastus

az storage account create \
  --name mystorageacct123 \
  --resource-group rg-migration \
  --location eastus \
  --sku Standard_LRS

az sql server create \
  --name mysqlserver123 \
  --resource-group rg-migration \
  --location eastus \
  --admin-user sqladminuser \
  --admin-password 'Use-A-Strong-Secret-Here!'
```

A simple orchestration mindset is to separate the migration into phases:

```yaml
phases:
  - assess
  - prepare-target
  - seed-data
  - sync-deltas
  - validate
  - cutover
  - observe
  - decommission
```

That structure helps you reason about dependencies and rollback points.

**Exam decision cues**

Look for these clues in scenario questions:

- If the workload has strict downtime limits, choose an online sync approach and a short cutover window.
- If the target is Azure SQL Database, check for compatibility and feature gaps before migration.
- If the workload includes jobs, reports, and integrations, sequence those components around the data cutover.
- If the scenario mentions large file volumes, consider Data Box or staged copy plus delta sync.
- If the scenario emphasizes data integrity over speed, choose a more controlled cutover with stronger validation.
- If the scenario mentions multiple dependent systems, migrate in waves and validate each wave before proceeding.

**Common mistakes to avoid**

- Migrating data before the target platform is ready.
- Cutting over applications before validation is complete.
- Forgetting scheduled jobs, ETL, or reporting dependencies.
- Assuming DNS TTL alone guarantees immediate client switchover.
- Treating rollback as a simple source re-enable after target writes have occurred.
- Using a tool or migration mode without matching it to the workload’s downtime and consistency requirements.

**Summary**

Data migration sequencing is about controlling risk through order. Prepare the target first, move or replicate data next, then migrate dependent services, validate at each gate, and cut over only when the target is proven ready. The best sequence depends on the workload type, the migration tool, the downtime budget, and the rollback strategy. In AZ-305 scenarios, the correct answer usually reflects both technical dependency order and business constraints.
