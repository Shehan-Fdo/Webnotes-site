## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Why migration projects fail**

Azure migrations usually fail for predictable reasons: teams underestimate application dependencies, move too much too quickly, and treat cutover as a technical event instead of an operational change. The most common pattern is not a single bad decision, but a chain of small misses: incomplete discovery, weak wave planning, poor testing, and no clear rollback path.

A successful migration plan reduces uncertainty before you move production workloads. You need to know what depends on what, which workloads can tolerate downtime, which services have Azure feature gaps, and how you will validate the result after cutover.

- **Discovery gaps** hide shared services, hard-coded endpoints, and legacy integrations.
- **Wave planning** fails when you migrate too many workloads at once or ignore dependency order.
- **Testing gaps** leave performance, security, and user acceptance issues undiscovered until production.
- **Change management gaps** create confusion when support teams are not trained before go-live.

**Discovery and dependency mapping**

The first pitfall is incomplete discovery. Many applications depend on more than the obvious VM, database, or web tier. They may also rely on DNS behavior, Active Directory, file shares, scheduled tasks, certificates, message queues, or third-party agents installed on the operating system.

A migration plan should identify both technical and business dependencies. Technical dependencies tell you what must move together. Business dependencies tell you what must stay available during the move.

- **Application dependencies** often include:
  - Database connections and connection strings
  - Shared storage and file paths
  - Identity services such as Microsoft Entra ID or Active Directory Domain Services
  - Monitoring, backup, and antivirus agents
  - Licensing servers, hardware dongles, or vendor-specific middleware
- **Platform-specific dependencies** can block a lift-and-shift:
  - Unsupported SQL Server features or database behaviors
  - OS or kernel-level requirements
  - Third-party agents that do not run in Azure
  - PaaS feature gaps when the app expects on-premises control

> If an application depends on a feature that Azure does not provide in the target service, you must redesign, refactor, or keep that workload on IaaS (Infrastructure as a Service) longer than planned.

A practical discovery workflow often starts with Azure Migrate for server and application assessment, then expands into dependency visualization and manual validation with application owners.

az login
az account set --subscription "<subscription-id>"

# Example: list discovered Azure Migrate projects

**Wave planning and migration factory pitfalls**

A migration factory is the repeatable operating model that moves workloads in waves. It standardizes assessment, build, test, cutover, and stabilization. The biggest pitfall is treating every workload as if it can move in the same wave.

Poor sequencing creates avoidable outages. For example, moving an application tier before its database, or migrating a shared authentication service after dependent apps, can break multiple workloads at once. Another common mistake is overloading the team by scheduling too many cutovers in a single weekend.

- **Too many workloads at once** increases blast radius and makes troubleshooting slower.
- **Poor sequencing** breaks dependencies between tiers, identity, and shared services.
- **No pilot wave** means you discover process flaws during production migration instead of in a low-risk test.
- **No stabilization window** leaves unresolved issues hidden until the next wave starts.

A better approach is to use a pilot wave with one low-risk workload, then refine the runbook, timing, and rollback steps before scaling out.

| Approach | Risk | Typical outcome |
|---|---:|---|
| Big-bang migration | High | Hard to isolate failures; rollback is complex |
| Pilot wave first | Lower | Reveals tooling, process, and communication gaps early |
| Dependency-based waves | Moderate | Better sequencing and fewer cross-service failures |

**Tooling workflows and where teams go wrong**

Azure provides several migration tools, but each has a specific role. Teams often fail when they use the tool without understanding the workflow.

- **Azure Migrate Server Migration**
  - Discovers servers and assesses readiness.
  - Replicates on-premises VMs to Azure.
  - Supports test migration before cutover.
  - Common pitfall: skipping test migration and discovering driver, boot, or network issues during production cutover.
- **Azure Database Migration Service (DMS)**
  - Assesses database compatibility and migration readiness.
  - Supports online and offline migration patterns depending on source and target.
  - Common pitfall: ignoring schema or feature incompatibilities until the final sync.
- **Azure Site Recovery (ASR)**
  - Replicates workloads to Azure and supports planned failover/cutover.
  - Often used for migration when you want controlled replication and a cutover window.
  - Common pitfall: assuming replication alone proves application readiness.

A realistic workflow for a VM migration might look like this:

1. Discover the server in Azure Migrate.
2. Assess readiness, sizing, and dependency impact.
3. Enable replication with Azure Migrate Server Migration or ASR.
4. Run a test migration into an isolated network.
5. Validate boot, app startup, DNS, and user access.
6. Schedule cutover, quiesce writes, and perform final sync.
7. Switch traffic and monitor closely during stabilization.

**Data consistency and cutover control**

Data consistency is one of the most important migration risks. If users continue writing data during cutover without a controlled sync strategy, you can lose transactions or create split-brain behavior.

Common strategies include:

- **Transactional replication** for databases that need near-real-time synchronization with controlled cutover.
- **Quiescing writes** by placing the application in maintenance mode before final sync.
- **Offline migration** when downtime is acceptable and consistency is simpler to guarantee.
- **Eventual consistency handling** when the target architecture uses asynchronous replication or distributed services.

The pitfall is not choosing a strategy; it is failing to align the strategy with the application’s tolerance for downtime and data loss.

- **Quiescing writes** reduces the chance of missing final transactions.
- **Transactional replication** helps when you need a short cutover window.
- **Eventual consistency** requires user and business acceptance that data may not appear instantly everywhere.
- **Final validation** should confirm row counts, checksums, or application-level reconciliation where possible.

For database migrations, a common mistake is to run the bulk copy, then assume the target is ready without verifying schema differences, collation settings, identity values, triggers, or stored procedure behavior.

```sql
-- Example: basic validation checks after database migration
SELECT COUNT(*) AS OrdersCount FROM dbo.Orders;
SELECT TOP (10) * FROM dbo.Orders ORDER BY CreatedDate DESC;
```

**Testing and validation pitfalls**

Insufficient testing is one of the most expensive migration mistakes because it shifts discovery into production. A test migration is not enough if you do not validate the workload under realistic conditions.

You should test more than connectivity:

- **Functional testing** confirms the application starts and core workflows succeed.
- **Performance testing** checks latency, throughput, and resource sizing.
- **Load testing** verifies the system under expected and peak demand.
- **User acceptance testing (UAT)** confirms business users can complete real tasks.
- **Failover testing** validates recovery behavior and rollback assumptions.

A common failure mode is to test only a single user or a small dataset, then discover that the Azure target is undersized or that storage IOPS (input/output operations per second) cannot sustain production load.

> A migration that passes smoke tests can still fail under real concurrency, real data volume, or real network latency.

Practical validation should include:

- Application startup and service health
- Authentication and authorization flows
- DNS resolution and certificate trust
- Batch jobs, schedulers, and integrations
- Backup, monitoring, and alerting after migration

**Security, compliance, and network design mistakes**

Migration projects often expose hidden security assumptions. A workload may work in the source environment because of permissive firewall rules, flat networks, or local admin access. Those assumptions may not survive in Azure.

- **Network segmentation** can break hard-coded IP dependencies.
- **NSG (Network Security Group)** rules may block required ports after cutover.
- **Identity integration** may fail if domain controllers, trust paths, or managed identities are not planned.
- **Compliance controls** may require logging, encryption, or data residency decisions before migration.

A common pitfall is to replicate the workload first and design the landing zone later. That usually creates rework because routing, DNS, private endpoints, and access control need to be correct before production traffic moves.

**Operational readiness and support handoff**

Migration success does not end at cutover. If the operations team is not prepared, the workload may be technically live but operationally fragile.

- **Training gaps** leave support teams unable to troubleshoot Azure-specific issues.
- **Missing runbooks** slow incident response and increase downtime.
- **No support handoff** means the project team stays involved longer than planned.
- **Unclear ownership** causes delays when alerts fire after go-live.

You should update monitoring thresholds, backup policies, patching responsibilities, and escalation paths before the migration wave closes. If the application team, infrastructure team, and service desk all use different terminology, document the new support model explicitly.

**A realistic end-to-end failure scenario**

Consider a three-tier line-of-business app with a SQL Server backend, a web front end, and a third-party reporting agent installed on the database server. The team discovers the VMs with Azure Migrate, but they miss the reporting agent dependency and do not test the app under peak load.

They also migrate the web tier first, then the database, and cut over without quiescing writes. During the final sync, users continue placing orders, causing data mismatch. After cutover, the app is slow because the Azure VM size was chosen from idle-hour metrics, not peak-hour performance. Support then struggles because the operations team never received a runbook or training.

This kind of failure usually combines several pitfalls:

- Incomplete dependency discovery
- Bad wave sequencing
- No pilot wave
- No write quiescing or transactional sync plan
- Weak performance and UAT coverage
- Poor communication and support handoff

**Practical checklist for avoiding migration pitfalls**

Use this checklist before each wave:

1. Confirm application, infrastructure, and business dependencies.
2. Validate target service fit, including PaaS feature gaps and OS requirements.
3. Choose the right migration method: rehost, replatform, or refactor.
4. Build a pilot wave and document lessons learned.
5. Test migration in an isolated environment.
6. Validate performance, load, and UAT with realistic data.
7. Plan data consistency, final sync, and rollback.
8. Train support teams and publish runbooks.
9. Communicate cutover timing and user impact.
10. Monitor closely after go-live and keep a stabilization window.

**Key takeaways**

Migration pitfalls are usually process failures, not tool failures. Azure Migrate, Database Migration Service, and Azure Site Recovery can support a good plan, but they cannot replace dependency analysis, sequencing discipline, data consistency control, and operational readiness.

If you remember one principle, make it this: move in small, validated waves, and treat cutover as a coordinated business event, not just a technical switch.
