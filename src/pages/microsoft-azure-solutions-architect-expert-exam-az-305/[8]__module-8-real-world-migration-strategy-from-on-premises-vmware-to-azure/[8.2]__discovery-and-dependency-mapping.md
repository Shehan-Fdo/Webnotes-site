## **Course:** Microsoft Azure Solutions Architect Expert (Exam AZ-305)

**Overview**

Discovery and dependency mapping help you understand what exists in an environment before you design a migration, modernization, or landing zone strategy. In Azure architecture work, you use these capabilities to identify servers, applications, network flows, and service relationships so you can plan scope, sequence, risk, and cutover dependencies.

The main goal is not just inventory. It is to answer practical questions such as:

- What workloads exist?
- Which systems talk to each other?
- Which dependencies are hard requirements for a migration wave?
- What breaks if you move one tier before another?

**Why discovery matters**

Discovery gives you the baseline for assessment. Without it, you risk underestimating application coupling, missing shared services, or migrating in the wrong order. Dependency mapping adds context by showing communication patterns between machines and services, which is especially useful for multi-tier applications and legacy estates.

Typical uses include:

- Migration planning for Azure Migrate
- Application rationalization and wave planning
- Identifying shared infrastructure and hidden coupling
- Validating firewall, routing, and segmentation requirements
- Supporting business continuity and rollback planning

**Core concepts**

<dl>
<dt><strong>Discovery</strong></dt>
<dd>The process of identifying assets in an environment, such as servers, databases, applications, and network devices.</dd>
<dt><strong>Dependency mapping</strong></dt>
<dd>The process of identifying communication relationships between discovered assets so you can understand application topology and traffic flow.</dd>
<dt><strong>Application topology</strong></dt>
<dd>The logical structure of an application, including tiers, services, and the connections between them.</dd>
<dt><strong>Assessment</strong></dt>
<dd>The evaluation of discovered assets to determine readiness, sizing, compatibility, and migration approach.</dd>
</dl>

**Azure Migrate and discovery**

Azure Migrate is the primary Azure service used for discovery, assessment, and migration planning. It can discover servers and help you assess readiness for Azure. For dependency mapping, Azure Migrate commonly uses an agent-based approach on Windows and Linux machines to observe network connections and infer relationships.

Key points:

- Discovery can be agentless or agent-based depending on the scenario and tool.
- Dependency visualization is typically built from observed network traffic, not from application code or deep packet inspection of payloads.
- The result is a communication map that helps you understand likely dependencies, not a perfect application architecture diagram.
- The Azure Migrate project organizes discovery and assessment data for a specific environment and scope.

> Dependency mapping is a planning aid, not a source of truth for every application-level relationship. Use it with interviews, CMDB data, firewall logs, and application documentation.

**How dependency mapping works**

In practice, dependency mapping relies on agents or collectors that observe outbound and inbound connections from a machine over time. The service then groups connections into a visual map.

What it can show:

- Source and destination machines
- TCP/UDP communication patterns, depending on the tool and platform support
- Commonly used ports and connection frequency
- Multi-tier relationships such as web to app to database

What it may not fully show:

- Business logic between services
- Calls that are not visible from the monitored host
- Dependencies that occur through external brokers or shared middleware if the observation point does not capture the full chain
- Short-lived or infrequent connections if the observation window is too small

The practical takeaway is that you should treat the map as a strong starting point for dependency analysis, then validate it with application owners and network evidence.

**Common Azure Migrate workflow**

1. Create or select an Azure Migrate project.
2. Add the discovery and assessment tool for servers.
3. Deploy the appliance or agent required for the discovery method.
4. Start discovery and wait for inventory to populate.
5. Enable dependency analysis where supported.
6. Review the dependency map and group machines into application boundaries.
7. Use the results to plan migration waves and test cutover order.

**Example: Azure CLI project creation**

Azure Migrate project creation is typically done through the portal, but you may use Azure CLI for related resource management and automation around the environment. A common pattern is to create the resource group first and then deploy the project through the portal or ARM-based automation.

```bash
az group create \
  --name rg-migrate-eastus \
  --location eastus
```

If you automate surrounding resources, keep the project region aligned with the intended Azure Migrate project location and the target operational model.

**Dependency mapping in migration planning**

Dependency maps are most valuable when you use them to define migration waves. A wave should include systems that are tightly coupled or that must remain available together.

Use the map to identify:

- Front-end and back-end tiers that must move together
- Shared databases or file shares
- Authentication and directory dependencies
- Batch jobs, schedulers, and integration endpoints
- Legacy systems that depend on a single upstream service

A practical migration sequence often looks like this:

1. Discover the environment and validate inventory.
2. Group machines into applications and tiers.
3. Identify shared services and external dependencies.
4. Rank applications by coupling, business criticality, and complexity.
5. Build migration waves that preserve functional integrity.
6. Test each wave in a nonproduction environment before cutover.

**Example topology**

A typical three-tier application might look like this:

```text
Users
  |
  v
Web tier (IIS / NGINX)
  |
  v
App tier (API / business logic)
  |
  v
Database tier (SQL Server / PostgreSQL)
```

A dependency map may also reveal supporting services:

```text
Web tier -> DNS
Web tier -> Identity provider
App tier -> Cache
App tier -> Message queue
Database tier -> Backup service
```

This matters because moving only the web tier without the app tier may create broken requests, while moving the database first may break both upstream tiers.

**Tools and signals you should know**

Azure discovery and dependency mapping often combine multiple sources of truth:

- **Azure Migrate** for discovery, assessment, and dependency visualization
- **Azure Monitor** and logs for operational validation
- **Network security group** flow logs or firewall logs for traffic corroboration
- **CMDB** data for asset ownership and application grouping
- **Application performance monitoring** tools for service-level tracing
- **Interviews** with application and infrastructure owners

The best results come from triangulating these sources rather than trusting one tool alone.

**Practical configuration considerations**

Dependency mapping quality depends on how you deploy and operate the collection mechanism.

Important considerations:

- Install the required agent on each machine you want to observe.
- Ensure outbound connectivity to the required Azure endpoints.
- Allow enough observation time to capture normal business cycles.
- Include batch windows, maintenance jobs, and peak usage periods.
- Verify that firewalls and endpoint protection do not block telemetry.
- Use consistent naming so discovered machines are easy to group.

A simple operational checklist:

- Confirm time synchronization across hosts.
- Validate DNS resolution for internal and external dependencies.
- Check that the agent version is current.
- Review whether servers are behind NAT, proxies, or load balancers.
- Compare observed flows with known application diagrams.

**Example: validating network paths on a Linux host**

```bash
ss -tupn
ip route
nslookup sql01.contoso.local
curl -I http://app01.contoso.local:8080
```

These commands help you confirm whether the observed dependency is real and reachable from the host perspective.

**Example: validating on Windows**

```powershell
Get-NetTCPConnection | Select-Object LocalAddress,LocalPort,RemoteAddress,RemotePort,State
Resolve-DnsName sql01.contoso.local
Test-NetConnection sql01.contoso.local -Port 1433
```

Use these checks to compare the dependency map with actual connectivity and port usage.

**Limitations and exam cautions**

Dependency mapping is useful, but it has limits you should remember for AZ-305:

- It reflects observed communication, not full application intent.
- It may miss dependencies that are dormant during the observation window.
- It may not fully expose application chains that are abstracted behind shared middleware, brokers, or proxies.
- It can show traffic through intermediaries, but that does not always reveal the complete upstream/downstream business dependency.
- It is only as good as the scope, agent coverage, and observation duration.

> Do not overstate the limitation. The correct exam-safe idea is that dependency mapping may not reveal the full application-level dependency chain, not that it cannot detect anything behind proxies or middleware.

**When to use dependency mapping**

Use dependency mapping when you need to:

- Plan a migration wave
- Identify application boundaries
- Reduce cutover risk
- Validate firewall rules and network segmentation
- Understand legacy estates with poor documentation
- Support modernization decisions such as rehost, refactor, or retire

It is especially helpful for:

- Monolithic applications with hidden service calls
- Multi-tier systems with shared databases
- Environments with many unmanaged or poorly documented servers
- Large estates where manual discovery would be too slow

**How to interpret results**

Treat the map as a hypothesis that you validate.

Good interpretation habits:

- Look for repeated connections over time, not one-off traffic.
- Separate business-critical dependencies from incidental traffic.
- Identify shared services that create blast radius.
- Check whether a dependency is internal, external, or cross-subscription.
- Confirm whether the observed port and protocol match the expected application behavior.

A useful question set:

- Is this dependency required for the application to function?
- Is it synchronous or asynchronous?
- Is it local to the application or shared across many workloads?
- Can it be replaced, isolated, or removed before migration?

**Exam focus**

For AZ-305, focus on the architectural purpose of discovery and dependency mapping:

- It supports migration and modernization planning.
- It reduces risk by revealing application coupling.
- It helps define migration waves and cutover order.
- It is not a replacement for application ownership knowledge or operational validation.
- It works best when combined with other inventory and monitoring sources.

**Quick reference**

| Topic | Key idea |
|---|---|
| Discovery | Find assets and inventory the environment |
| Dependency mapping | Show communication between assets |
| Best use | Migration planning and wave design |
| Main limitation | May not reveal the full application-level chain |
| Validation | Compare with logs, interviews, and network evidence |

**Summary**

Discovery tells you what exists. Dependency mapping tells you what depends on what. Together, they help you design safer migrations, avoid breaking application tiers, and make informed Azure architecture decisions. For exam purposes, remember that the value is in planning and risk reduction, while the limitation is incomplete visibility into the full application context.
