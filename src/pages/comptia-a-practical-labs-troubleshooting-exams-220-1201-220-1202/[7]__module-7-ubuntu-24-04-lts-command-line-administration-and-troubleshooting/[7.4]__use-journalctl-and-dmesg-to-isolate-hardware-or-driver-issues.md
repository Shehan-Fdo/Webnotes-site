## **Course:** CompTIA A+ Practical Labs & Troubleshooting (Exams 220-1201 & 220-1202)

**Why these logs matter**

When hardware misbehaves or a driver crashes, Linux often records the event in kernel logs. Two tools help you inspect those messages: `dmesg` for the kernel ring buffer and `journalctl` for the systemd journal. Together, they help you narrow down whether the problem is a device, a driver, or a broader system fault.

The kernel ring buffer is a circular buffer in kernel memory. It stores messages from the kernel itself, including device drivers, filesystem code, and network stack activity. Because it is circular, newer messages can overwrite older ones, so you should check it as soon as possible after the issue appears.

- `dmesg` reads the kernel ring buffer.
- `journalctl` reads the journal managed by `systemd` and `journald`.
- Kernel logs are especially useful for boot problems, device failures, and crash-related events.
- Persistent logging helps when the issue happened before you could inspect the live buffer.

**Using `dmesg` for kernel-level clues**

`dmesg` is the primary tool for reading kernel messages. It is useful when you suspect a hardware fault, a driver problem, or a kernel-side error such as a panic or out-of-memory event. The output often includes timestamps, device names, and subsystem messages that help you connect the error to a specific component.

A practical first pass is to look for recent warnings and errors around the time the issue occurred. If the system is still running, `dmesg` can show the most recent kernel activity. If the system rebooted or the buffer wrapped, you may need the journal for older entries.

```bash
dmesg
dmesg -S
cat /proc/sys/kernel/printk_devkmsg
```

- `dmesg` shows current kernel messages.
- `dmesg -S` checks the ring buffer size.
- `cat /proc/sys/kernel/printk_devkmsg` exposes kernel message handling details.
- Message order matters because the newest entries may overwrite older ones in the ring buffer.

> If the system has already rebooted, the live `dmesg` buffer may no longer contain the event you need. Use `journalctl` for persistent logs when available.

**Using `journalctl` for persistent troubleshooting**

`journalctl` gives you access to the systemd journal, which centralizes logs from the kernel, initrd, services, and other processes. This is useful when the issue spans more than one subsystem or when you need logs from before the current boot session.

Because the journal is persistent when configured that way, it can preserve the context around a failure that `dmesg` no longer shows. That makes it a strong follow-up tool after you identify a suspicious kernel message.

- `journalctl` helps you review logs across boots and services.
- The journal includes kernel and userland messages in one place.
- Persistent logs are valuable after reboots, crashes, or intermittent failures.
- Use it when the problem is not limited to a single live kernel buffer.

**A simple troubleshooting workflow**

1. Reproduce or observe the hardware or driver issue.
2. Run `dmesg` and look for recent kernel warnings, errors, or device-related messages.
3. If the event is missing or the system rebooted, use `journalctl` to review the journal.
4. Correlate timestamps, device names, and error text with the failing component.
5. Decide whether the likely cause is hardware, a driver, or another kernel subsystem.

This workflow keeps you focused on evidence instead of guessing. In A+ troubleshooting, the goal is to identify the failing layer quickly so you can move to the correct fix path.

**What to look for in the output**

Kernel and journal logs often point to the type of failure even when they do not name the exact root cause. You should scan for messages that indicate a device stopped responding, a driver failed to initialize, or the kernel encountered a fault it could not handle.

- Repeated errors from the same device can suggest a hardware problem.
- Driver initialization failures can indicate a bad or incompatible driver.
- Filesystem or memory-related messages can point to broader system instability.
- A kernel panic means the kernel hit a fault it could not handle and stopped the system.

**Practical interpretation tips**

Keep the scope narrow. Start with the time of the failure, then match the log entries to the affected subsystem. If a USB device disconnects, a storage controller reports errors, or a graphics driver fails, the relevant messages usually appear close together in the log stream.

Use conservative conclusions when the evidence is incomplete. A single error line may show the symptom, not the root cause. Combine log output with the observed behavior, such as a frozen screen, reboot, missing device, or failed boot.

**Key takeaways**

- `dmesg` is best for immediate kernel-ring-buffer inspection.
- `journalctl` is best for persistent, centralized log review.
- Both tools help isolate hardware and driver issues by showing kernel-side evidence.
- The fastest path is to match the log timing to the observed failure.
