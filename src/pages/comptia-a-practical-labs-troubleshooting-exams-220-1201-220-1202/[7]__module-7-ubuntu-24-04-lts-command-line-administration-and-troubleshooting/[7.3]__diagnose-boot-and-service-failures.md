## **Course:** CompTIA A+ Practical Labs & Troubleshooting (Exams 220-1201 & 220-1202)

**Boot and service failure overview**

When a system fails to boot cleanly or a service does not start, you need a repeatable troubleshooting path. The evidence points to a consistent workflow on Linux systems: identify what is failing, inspect its status, and then review logs for the reason behind the failure. This same mindset supports A+ troubleshooting on any platform because it keeps you focused on observable symptoms before making changes.

- Start by checking which units are in a failed state.
- Use service status output to confirm whether the unit is loaded, active, or failed.
- Move to logs when the status output does not explain the root cause.

**Checking failed services**

On Ubuntu with `systemd`, the first pass is to list failed units. The commands below show the current failures in slightly different formats, which helps you quickly spot the affected service.

```bash
systemctl --failed
systemctl list-units --state=failed
```

The output includes the unit name and a brief reason. In the status columns, a failed service typically shows `loaded`, `failed`, and `failed` for `LOAD`, `ACTIVE`, and `SUB`. That pattern tells you the unit exists, but it is not running successfully.

- Use `systemctl --failed` for a concise failure list.
- Use `systemctl list-units --state=failed` when you want the same information in a broader unit listing.
- Focus on the service name before changing configuration.

**Inspecting a specific service**

After you identify the failing unit, check its detailed status. The evidence shows this as the next step after the initial failure list. Detailed status helps you confirm whether the service is present, whether it attempted to start, and what immediate error it reported.

```bash
sudo systemctl status nginx
```

Replace `nginx` with the service you are troubleshooting. This is the point where you narrow the problem from “something failed” to one specific unit and its current state.

- Verify the exact unit name before running the status command.
- Read the status output for the most recent failure message.
- Use the status view to decide whether you need logs or startup analysis next.

**Reading logs with journalctl**

`journalctl` is the log viewer used to inspect system and service messages. The evidence identifies it as a core diagnostic tool for failed `systemd` services. When status output is not enough, logs show the sequence of events that led to the failure.

```bash
journalctl -u nginx
```

You can use the service name with `-u` to filter log entries for that unit. This is especially useful when a service fails during startup but the status summary only shows the final error.

- Filter logs to the affected unit instead of scanning the full journal.
- Look for repeated errors, dependency failures, or missing resources.
- Use log timestamps to connect the failure to boot or restart attempts.

**Analyzing boot timing**

For boot-related failures, `systemd-analyze` helps you inspect startup behavior and identify slow or problematic parts of the boot process. The man page shows several useful modes, including `blame`, `critical-chain`, `plot`, and `verify`. These commands support a practical boot diagnosis workflow when the system starts slowly or a unit blocks the boot sequence.

```bash
systemd-analyze blame
systemd-analyze critical-chain
systemd-analyze verify /path/to/unit.service
```

Use `blame` to see which units took the longest, `critical-chain` to trace the startup path, and `verify` to check a unit file for issues. This gives you a structured way to separate a slow boot from a broken service.

- Use `blame` to identify long startup units.
- Use `critical-chain` to see dependency timing.
- Use `verify` when you suspect a unit file problem.

**Practical troubleshooting flow**

A reliable workflow keeps you from guessing. Start with the failure list, confirm the service state, inspect logs, and then analyze boot timing if the issue affects startup. This sequence matches the evidence and gives you a practical path from symptom to cause.

1. Run `systemctl --failed` or `systemctl list-units --state=failed`.
2. Check the specific unit with `systemctl status <service>`.
3. Review logs with `journalctl -u <service>`.
4. Use `systemd-analyze` when the issue involves boot timing or startup order.
5. Verify the unit file if the failure suggests a configuration problem.

> If a service fails during boot, do not assume the boot process itself is broken. Confirm the unit state and logs first so you can distinguish a service failure from a broader startup issue.

**What to remember for exam prep**

For A+ troubleshooting, the key skill is choosing the right diagnostic tool for the symptom. Failed services call for `systemctl`; service-specific history calls for `journalctl`; boot timing and dependency analysis call for `systemd-analyze`. That combination gives you a clean, practical method for diagnosing boot and service failures without unnecessary changes.

- `systemctl` shows current service state.
- `journalctl` shows the failure history.
- `systemd-analyze` helps with boot performance and startup order.
