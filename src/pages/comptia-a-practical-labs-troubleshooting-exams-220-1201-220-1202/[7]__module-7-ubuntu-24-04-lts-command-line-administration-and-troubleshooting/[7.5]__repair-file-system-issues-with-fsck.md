## **Course:** CompTIA A+ Practical Labs & Troubleshooting (Exams 220-1201 & 220-1202)

**What `fsck` does**

`fsck` stands for *file system check*. It is the standard Linux utility for checking and repairing file system inconsistencies. In practice, it helps restore metadata consistency after corruption caused by unexpected power loss, hardware problems, kernel crashes, or software bugs.

File system check tools are often run automatically during boot when the system detects a problem, but you can also invoke them manually for maintenance or recovery. On ext2/ext3/ext4 systems, the underlying checker is typically `e2fsck`.

- `fsck` focuses on consistency, not data recovery.
- It can repair metadata structures such as inodes, block groups, directory structure, and reference counts.
- It is most useful when a system fails to boot, a disk becomes corrupt, a file becomes inaccessible, or the file system switches to read-only mode.

**When you should suspect file system corruption**

Corruption usually shows up as boot failures, inaccessible files, or read-only remounts after the system detects errors. Kernel logs often provide the first clue.

```bash
sudo dmesg | grep -i "ext4\|filesystem\|i/o error"
sudo journalctl -b | grep -i "ext4\|filesystem error"
```

Common warning signs include:

- The system fails to boot with file system error messages.
- Files or directories suddenly become inaccessible.
- You see `Input/output error` when opening files.
- The system drops to read-only mode.
- `dmesg` reports messages such as `EXT4-fs error`.

> Run repair tools only after you confirm the problem is file-system related. If the issue is actually hardware failure, repeated repair attempts can waste time and may not solve the root cause.

**Safe repair workflow**

You should treat file system repair as a controlled recovery task. The key rule is to avoid running `fsck` on a mounted file system, because the live system and the checker can modify metadata at the same time.

1. Identify the affected partition or mount point.
2. Unmount the file system if it is not the root file system.
3. If the root file system is affected, repair it from boot-time recovery or a live USB environment.
4. Run the appropriate checker for the file system type.
5. Review the output and allow automatic fixes only when you understand the impact.

The root file system `/` cannot be unmounted while the system is running, so it is normally checked during boot or from external recovery media.

**Using `fsck` effectively**

`fsck` is a front end for file system-specific tools. For ext2/ext3/ext4, the checker is `e2fsck`; other file systems use different repair utilities.

| File system type | Typical checker |
|---|---|
| ext2/ext3/ext4 | `e2fsck` |
| XFS | `xfs_repair` |
| Btrfs | `btrfs check` |
| NTFS | `ntfsfix` |

A read-only check can be useful when you want to inspect a file system without making changes. For ext file systems, `e2fsck -n` performs a non-destructive check.

```bash
sudo e2fsck -n /dev/sdXn
```

- Use `-n` when you want to inspect damage first.
- Use repair options only after you understand the reported inconsistencies.
- Prefer the correct tool for the file system type instead of forcing a generic command.

**Automatic checks and boot-time behavior**

Linux distributions may run file system checks automatically during boot if they detect inconsistencies. This is useful after unclean shutdowns, but it does not replace manual troubleshooting when the system cannot boot or the damage is severe.

- Automatic checks help catch problems early.
- Manual checks are still needed when the system is unstable or inaccessible.
- Boot-time repair is common for the root file system because it cannot be safely unmounted while running.

**Practical troubleshooting mindset**

`fsck` is a repair tool, but it is not a substitute for backups or hardware diagnostics. If corruption keeps returning, look for the underlying cause such as failing storage, repeated power loss, or other system instability.

- Treat repeated corruption as a symptom, not the final problem.
- Verify logs before and after repair.
- Use the least invasive check that answers your question.
- Keep recovery steps simple and deliberate.

**Quick recall**

- `fsck` means file system check.
- It repairs metadata consistency, not user data.
- Run it on unmounted file systems.
- Use boot-time or live-media recovery for `/`.
- Check logs with `dmesg` and `journalctl` when diagnosing corruption.
