## **Course:** CompTIA A+ Practical Labs & Troubleshooting (Exams 220-1201 & 220-1202)

**Service failure triage on Ubuntu**

When an Ubuntu service fails, you usually start by identifying the failed unit, then inspect its status and logs. The core tools are `systemctl` for service state and `journalctl` for log output. This workflow helps you move from a vague “service is down” ticket to a specific cause.

- `systemctl --failed` shows units in the failed state.
- `systemctl list-units --state=failed` provides the same overview in a slightly different format.
- Failed services typically show `loaded`, `failed`, and `failed` in the `LOAD`, `ACTIVE`, and `SUB` columns.
- `systemctl status servicename` gives a quick status view with recent log lines inline.
- `journalctl -u servicename` shows logs for one service.

**Check the failure state first**

Start with a broad view of what is broken before you focus on one service. This helps you confirm whether the ticket is about a single unit or part of a wider outage. The failed-units list is the fastest way to spot the service name you need to investigate.

```bash
systemctl --failed
systemctl list-units --state=failed
```

A failed unit usually stands out because its state fields indicate failure rather than normal operation. Once you identify the service, move directly to its detailed status.

**Inspect the service status**

`systemctl status` is useful because it combines service state with recent log output. That makes it a practical first stop when you need a quick explanation for why the service did not start.

```bash
sudo systemctl status nginx
```

Use the actual service name from the ticket or from the failed-units list. The status output helps you confirm whether the service is loaded, whether it is active, and whether the failure is recent or persistent.

**Read the logs for the service**

If the status output is not enough, inspect the journal for the specific unit. `journalctl` is the main log source for systemd-managed services, and it is the best place to look for the error that caused the failure.

```bash
journalctl -u nginx
```

You can narrow the output when needed:

- `-f` follows new log entries in real time.
- `-n 100` shows the last 100 lines.
- `--since "1 hour ago"` filters by time.

These filters help you focus on the moment the service failed instead of scanning unrelated history.

**Use the logs to confirm the cause**

Service failures often come from configuration errors, missing dependencies, or startup problems that appear clearly in the journal. Read the log sequence around the failure, not just the final error line, so you can see what happened immediately before the service stopped.

> If the service keeps failing after a restart, do not assume the restart fixed the issue. Recheck `systemctl status` and the journal to confirm the underlying problem is gone.

**Verify recovery after the fix**

After you correct the issue, confirm that the service is no longer failed and that it starts normally. A clean status check is the final proof that the ticket is resolved.

1. Recheck the service with `systemctl status servicename`.
2. Confirm the unit no longer appears in `systemctl --failed`.
3. Review `journalctl -u servicename` if the service still does not stay up.

**Practical troubleshooting pattern**

A reliable workflow keeps you from guessing:

1. Identify failed units with `systemctl --failed`.
2. Inspect the target service with `systemctl status servicename`.
3. Read the service logs with `journalctl -u servicename`.
4. Narrow the time window or follow live logs if needed.
5. Recheck status after the fix to confirm recovery.

This approach works well for common Ubuntu service tickets because it uses the same tools in the same order every time. It gives you a repeatable path from symptom to evidence to resolution.
