## **Course:** CompTIA A+ Practical Labs & Troubleshooting (Exams 220-1201 & 220-1202)

**What a no-boot ticket means**

A no-boot desktop ticket starts with a system that does not reach Windows. The failure can appear as a blank screen, a direct jump into BIOS/UEFI, missing boot options, or a boot loop with no POST (Power-On Self-Test). The practical goal is to separate a power problem, firmware problem, storage problem, and operating system problem before you make changes.

The boot path is useful for triage because it happens in stages:

- **Power on**: the system receives power and begins startup.
- **BIOS/UEFI**: firmware initializes hardware and prepares the machine to boot.
- **POST**: the system checks core components such as memory and keyboard.
- **Boot loader / operating system**: the machine loads Windows.

If the system reaches BIOS/UEFI but cannot find a bootable device, the issue is often storage detection, boot order, or a damaged boot configuration rather than a complete power failure.

**Initial observations and symptom patterns**

Start by identifying exactly how the desktop fails. Evidence from real troubleshooting cases shows several common patterns:

- The PC goes straight to BIOS/UEFI and does not show boot options.
- A drive appears in storage information but not in the boot priority list.
- The system reports that it cannot find any bootable devices.
- The machine gets stuck in a boot cycle and may eventually start after repeated attempts.
- The desktop fails after shutdown or loss of power but may boot normally after a restart.

These patterns help you narrow the fault domain:

- **No power at all** points to PSU, cable, or front-panel power issues.
- **BIOS/UEFI access with no boot device** points to drive detection, boot order, or firmware settings.
- **Repeated boot cycling with no POST** points to hardware instability, seating issues, or a failing component.
- **Windows load failure after POST** points to boot files or system file corruption.

> If the drive is visible in firmware storage information but not available as a boot target, do not assume Windows is healthy. The system may detect the hardware but still fail to locate a valid boot path.

**Safe troubleshooting sequence**

Use a practical, low-risk order so you do not overwrite evidence or create new problems.

1. Verify external power.
   - Check the power cable, outlet, and power strip.
   - Confirm the PSU switch is on, if present.
   - Look for any signs of life such as fans or LEDs.

2. Confirm whether the system reaches POST.
   - Watch for display output, vendor logo, or beep codes.
   - If the machine never reaches POST, focus on hardware and seating.

3. Enter BIOS/UEFI and inspect detection.
   - Check whether the storage device appears in firmware.
   - Review boot priority and boot menu entries.
   - Note whether the system says it cannot find bootable devices.

4. Recheck internal connections if hardware is not detected.
   - Reseat storage cables and power connectors.
   - Reseat memory if the system shows unstable startup behavior.
   - Re-test after each change so you know what affected the result.

5. If the drive is detected but Windows still will not load, move to software repair.
   - Use Windows repair tools when the operating system appears damaged.
   - Keep the scope limited to the boot issue you can verify.

**Firmware and boot configuration checks**

BIOS/UEFI settings can block a valid boot device even when the hardware is present. A desktop that suddenly stops booting after a freeze or forced shutdown may have a changed boot order, a missing entry, or a firmware state that no longer points to the correct drive.

Focus on these checks:

- Confirm the correct drive is first in the boot order.
- Check whether the boot menu shows any bootable devices.
- Verify that the storage device appears consistently in firmware.
- Avoid changing unrelated settings unless you can tie them to the failure.

If the firmware sees the drive in storage information but not in the boot list, the boot loader or partition structure may be damaged. In that case, the issue is no longer just hardware detection.

**Windows repair tools for boot-related corruption**

When the desktop reaches the Windows repair stage or shows signs of corrupted system files, use built-in repair tools. Microsoft documents two key utilities for missing or corrupted system files:

- **DISM** (Deployment Image Servicing and Management) helps provide the files needed for repair.
- **SFC** (System File Checker) scans for corrupted system files and restores missing or damaged Windows components.

A practical repair flow is:

1. Run DISM to prepare or repair the Windows image.
2. Run SFC to scan and restore system files.
3. Reboot and confirm whether Windows starts normally.

These tools are appropriate when Windows becomes unresponsive, freezes, or shows symptoms consistent with corrupted system files. They are not the first step for a machine that cannot even reach firmware or POST.

```powershell
DISM /Online /Cleanup-Image /RestoreHealth
sfc /scannow
```

**When to suspect hardware instead of Windows**

A no-boot desktop is not always an operating system problem. Hardware becomes more likely when you see any of the following:

- The system never reaches POST.
- The machine repeatedly cycles before startup.
- The drive disappears from firmware storage information.
- The desktop only boots after repeated power cycles or after a restart.
- The failure began immediately after a shutdown or power loss.

In those cases, check the physical components before you spend time on software repair. A drive that is not consistently detected cannot reliably boot Windows, even if the operating system is intact.

**Ticket handling and documentation**

A good support ticket records the facts that matter for diagnosis. For initial data collection, capture:

- **User information**
- **Device information**
- **Problem description**

Add the exact symptom, the last known good state, and any changes made before the failure. If the user reports a freeze followed by a forced shutdown, that detail matters because it can point to either file corruption or hardware instability.

Keep your notes specific:

- Record whether the system reaches BIOS/UEFI.
- Record whether the storage device appears in firmware.
- Record whether the boot menu shows any bootable devices.
- Record whether Windows repair tools were attempted.

**Practical takeaway**

A no-boot desktop ticket is solved by moving from power, to POST, to firmware detection, to Windows repair. Do not jump straight to reinstalling the operating system. First confirm whether the machine can see the drive, whether it has a valid boot target, and whether Windows system files need repair.
