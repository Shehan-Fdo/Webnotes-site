## **Course:** CompTIA A+ Practical Labs & Troubleshooting (Exams 220-1201 & 220-1202)

**Ticket intake and triage**

A no-network laptop ticket starts with fast triage. You collect the basic facts, confirm the symptom, and decide whether the issue is local to the laptop or part of a wider outage. Good triage keeps you focused on the most likely cause first and avoids wasting time on unrelated fixes.

- Record the **user**, device, and problem description at intake.
- Confirm whether the laptop fails on wired, wireless, or both connections.
- Check whether other devices on the same network can connect normally.
- Note whether the issue is isolated to one interface or affects all network access.

> If multiple devices fail at the same time, treat it as a broader network issue before you spend time on the laptop.

**Verify the physical and interface state**

Before you move into software troubleshooting, confirm the laptop can actually use the network path you expect. A wired adapter may be disconnected, disabled, or stuck while the system still appears healthy at a glance. The same logic applies to wireless if the adapter is off or not associated with an access point.

- Inspect the cable, port, and link indicators for wired connections.
- Confirm the correct adapter is enabled in the operating system.
- Check whether the laptop is connected to the intended network.
- Re-test after a simple reconnect or adapter toggle.

**Check for address assignment problems**

If the laptop reaches the network but still cannot communicate, the next likely issue is address assignment. The evidence shows a case where the interface requests an IP address but does not receive one, which points to a <abbr title="Dynamic Host Configuration Protocol">DHCP</abbr> problem or a related network path issue. In practical terms, you want to determine whether the laptop is failing to obtain a valid address, gateway, or lease.

- Look for signs that the device is stuck while getting IP configuration.
- Compare the laptop’s behavior with another device on the same network.
- Verify that the network can hand out addresses to other clients.
- Treat a missing address as a strong clue, not a final diagnosis.

```bash
# Example of checking network state on a Linux-style system
nmcli device status
journalctl -xe
ip addr show
```

**Use logs and symptoms to narrow the cause**

Troubleshooting works best when you combine symptoms with available logs. The evidence shows network tools reporting activation attempts and a disconnected state, which helps separate a link problem from a higher-level configuration issue. You do not need to guess the root cause immediately; you need enough evidence to choose the next test.

- Use logs to confirm whether the adapter starts, disconnects, or stalls.
- Compare the laptop’s behavior on different interfaces if available.
- Look for patterns that match a failed connection attempt rather than a total hardware failure.
- Keep the focus on observable behavior instead of assumptions.

**Restore service, then validate**

Once you identify the most likely failure point, restore connectivity with the least disruptive fix available and then verify the result. Root cause analysis matters, but the immediate goal on a help desk ticket is to get the user back online and confirm the laptop stays connected.

1. Apply the smallest practical fix for the suspected issue.
2. Reconnect the laptop to the network.
3. Confirm the device receives a valid network configuration.
4. Test access to a known working resource.
5. Document what changed and whether the issue recurs.

**Escalation and documentation**

If the laptop still cannot connect after basic checks, escalate with clear evidence. Strong documentation helps the next technician avoid repeating the same steps and makes it easier to separate a laptop fault from a DHCP, switch, or broader network problem.

- Include the affected interface, observed symptoms, and test results.
- Note whether the issue is wired, wireless, or both.
- Record any log messages that show the connection attempt.
- Escalate when the problem persists beyond local troubleshooting.

**Key takeaway**

A no-network laptop ticket is usually resolved by moving from intake to interface checks, then to address assignment, then to logs and validation. The practical goal is to isolate whether the failure is on the laptop, the local connection, or the network service that should provide access.
