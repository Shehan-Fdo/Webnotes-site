## **Course:** CompTIA A+ Practical Labs & Troubleshooting (Exams 220-1201 & 220-1202)

**Driver conflict diagnosis workflow**

A driver conflict often shows up as a device that fails to work correctly after a change, such as an update. In Windows, you start by checking the device’s status in **Device Manager**, then you correlate that with **Event Viewer** entries to narrow down the cause.

Device Manager reports error codes for problem devices. Microsoft notes that you can open the device’s **Properties** dialog and read the code in the **Device status** area. If the code is not listed in Microsoft’s guidance, you may need to contact the device vendor or Microsoft Support for help.

- Open **Device Manager** and expand the device category with the issue.
- Right-click the affected device and select **Properties**.
- Read the **Device status** field for the error code.
- Use the code to guide your next troubleshooting step.

> If Device Manager shows an error code, follow the recommended resolution for that code before making broader changes.

**Using Event Viewer to confirm symptoms**

**Event Viewer** helps you connect the device problem to a time-based pattern or related system service activity. In practice, you look for errors or warnings that occur when the device fails, then compare timestamps with the user’s report.

Event logs can show repeated service failures or informational messages that happen at the same time as the problem. That timing can help you distinguish a driver issue from a broader Windows service or system event.

- Look for entries that match the time the issue occurs.
- Compare repeated errors with the device’s failure pattern.
- Note whether multiple services fail together or only one device is affected.
- Use the event details to decide whether the issue is isolated to a driver or part of a wider system problem.

**Practical troubleshooting sequence**

1. Reproduce the problem so you know when it happens.
2. Open **Device Manager** and check the device’s **Properties**.
3. Record the **Device status** error code.
4. Open **Event Viewer** and review entries around the same time.
5. Match the event timing to the device failure.
6. Apply the recommended fix for the error code or escalate to vendor support if the code is not documented.

**What to look for in Device Manager**

Device Manager is the fastest place to confirm that Windows recognizes a device problem. The key signal is the error code shown in the device’s status area. That code tells you whether the issue is likely tied to the driver, the device, or another Windows-level condition.

- **Error code**: the primary clue for the fault.
- **Device status**: the location where Windows displays the code.
- **Affected category**: the device type that contains the problem device.

**What to look for in Event Viewer**

Event Viewer is useful when the problem repeats or appears to happen on a schedule. A cluster of errors at the same time can point to a driver conflict, a service failure, or another system component involved in the same failure window.

- **Timestamp alignment**: events that occur exactly when the issue appears.
- **Repeated failures**: the same service or component failing more than once.
- **Related messages**: informational entries that help confirm the sequence of events.

**Exam focus**

For A+ troubleshooting, remember the division of labor:

- **Device Manager** tells you what Windows thinks is wrong with the device.
- **Event Viewer** helps you see when the problem happens and what else fails at the same time.

That combination is especially useful when a device starts failing after a driver update or when the system shows recurring errors without a clear hardware failure.
