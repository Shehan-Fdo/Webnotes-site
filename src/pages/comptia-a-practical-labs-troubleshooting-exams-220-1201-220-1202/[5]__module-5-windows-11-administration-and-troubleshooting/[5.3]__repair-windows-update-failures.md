## **Course:** CompTIA A+ Practical Labs & Troubleshooting (Exams 220-1201 & 220-1202)

**Windows Update failure basics**

Windows Update failures often point to corruption, missing system files, or diagnostic data that helps you narrow down the cause. In practice, you start by checking the available logs and then move to built-in repair tools when system integrity is in question.

- **Windows Update logs** help you review update errors on Windows 10 and Windows 11.
- `windowsupdate.log` is located at `C:\Windows\Logs\WindowsUpdate`.
- Update Orchestrator events are written to `.etl` files under `C:\ProgramData\USOShared\Logs`.
- If Windows Update returns an error message, the log data can help you troubleshoot the issue.

**Use logs to identify the failure path**

Windows 8.1 and later use <abbr title="Event Tracing for Windows">ETW</abbr> to generate diagnostic logs for Windows Update. That means you may need to interpret log output rather than rely on a single plain-text file. When the update process fails, the logs can show whether the issue is tied to download, orchestration, or installation behavior.

- Review `windowsupdate.log` for update client errors.
- Check `.etl` files when the Update Orchestrator Service is involved.
- Use event logs when you need broader system context around the failure.
- Look for corruption-related errors when updates do not install successfully.

**Repair corrupted system files**

If Windows Update problems coincide with missing or corrupted system files, use built-in repair tools to restore system integrity. Microsoft recommends using <abbr title="Deployment Image Servicing and Management">DISM</abbr> and <abbr title="System File Checker">SFC</abbr> to scan for and repair damaged Windows components.

> Use these tools when Windows becomes unstable, features stop working correctly, or the system freezes while update-related corruption is suspected.

A practical repair sequence is to run DISM first so Windows has the files needed for repair, then run SFC to verify and fix system files.

```powershell
DISM /Online /Cleanup-Image /RestoreHealth
sfc /scannow
```

- **DISM** restores the component store and supplies files needed for repair.
- **SFC** scans protected system files and replaces missing or corrupted copies.
- Run the tools from an elevated command prompt or PowerShell session.
- Recheck Windows Update after the repair completes.

**Use event data for deeper troubleshooting**

When update errors persist, event logs can provide additional clues. The Windows Update documentation points you toward log files, while PowerShell can help you inspect event data on the local computer or a remote system.

- `Get-EventLog` reads events from local or remote computers.
- Use it when you need to inspect logs without opening the graphical Event Viewer.
- Focus on update-related entries and error messages around the failure time.
- Use the Setup event log when corruption or installation failure is suspected.

```powershell
Get-EventLog -LogName System -Newest 50
Get-EventLog -LogName Setup -Newest 50
```

**Troubleshooting approach**

A simple workflow keeps you from skipping the most useful evidence. Start with logs, confirm whether the issue is corruption-related, and then apply repair tools before testing the update again.

1. Check Windows Update logs for the error details.
2. Review relevant event logs for installation or corruption clues.
3. Run DISM to repair the Windows image.
4. Run SFC to repair missing or corrupted system files.
5. Retry Windows Update and confirm the result.

**What to remember for exam prep**

Windows Update failures are not always caused by the update service itself. They can also reflect damaged system files or broader installation corruption, so your troubleshooting should combine log review with repair utilities.

- Logs tell you what failed and when.
- DISM repairs the Windows image.
- SFC repairs protected system files.
- Event logs add context when the update path is unclear.
