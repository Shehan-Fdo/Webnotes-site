## **Course:** CompTIA A+ Practical Labs & Troubleshooting (Exams 220-1201 & 220-1202)

**PowerShell for troubleshooting**

PowerShell is a command-line shell and scripting language that helps you inspect Windows systems quickly. In A+ troubleshooting, you use it to check services, review network adapter status, and search event logs when you need more detail than a graphical tool provides.

- **Services** are a common first check when a feature or application is not behaving as expected.
- **Devices** can be inspected through adapter and hardware-related cmdlets.
- **Logs** help you trace system, security, and application events that may explain a failure.

**Inspect network adapters**

`Get-NetAdapter` returns basic network adapter properties. By default, it shows only visible adapters, which makes it useful when you want a quick view of active or present interfaces.

```powershell
Get-NetAdapter
Get-NetAdapter -Name "Ethernet"
Get-NetAdapter -IncludeHidden
```

Use `Format-List` if you need to view the common properties in a more readable list format. If you are checking a specific adapter, you can target it by name, interface description, or interface index.

- `-Name` filters by adapter name.
- `-InterfaceDescription` filters by the adapter description.
- `-InterfaceIndex` filters by the interface index.
- `-IncludeHidden` shows adapters that are not normally visible.
- `-Physical` limits results to physical adapters.

> `Get-NetAdapter` returns basic properties only; use formatting or additional inspection when you need more detail.

**Work with services and system state**

PowerShell is useful when you need to confirm whether a Windows service is present, running, or stopped. In troubleshooting, this helps you separate a service problem from a broader device or network issue.

- Check whether the expected service exists before you change settings.
- Confirm the service state when an application or feature fails.
- Use service output alongside logs to narrow the cause of the issue.

**Search event logs**

Windows event logs store detailed records of system, security, and application activity. PowerShell lets you search them locally when you need a faster or more targeted view than browsing manually.

- Use logs to correlate an error with the time it occurred.
- Review application, system, and security events for troubleshooting clues.
- Search specific events when you already know the approximate failure window.

A practical workflow is to inspect the device or service first, then check the logs for matching errors or warnings. This gives you both the current state and the history behind the problem.

**Troubleshooting workflow**

1. Identify the symptom and the affected component.
2. Inspect the related service or adapter with PowerShell.
3. Review event logs for matching errors, warnings, or timestamps.
4. Compare the results to determine whether the issue is configuration, connectivity, or a system event.

**Key takeaways**

- PowerShell gives you fast, scriptable visibility into Windows troubleshooting data.
- `Get-NetAdapter` is the main cmdlet in this topic for checking adapter properties.
- Event logs provide the historical context you need to confirm what happened and when.
- Use a step-by-step approach so you can connect symptoms to device state and log evidence.
