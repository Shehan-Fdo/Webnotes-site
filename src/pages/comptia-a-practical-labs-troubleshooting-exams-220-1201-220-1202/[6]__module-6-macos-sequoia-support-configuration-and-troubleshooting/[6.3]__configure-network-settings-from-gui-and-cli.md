## **Course:** CompTIA A+ Practical Labs & Troubleshooting (Exams 220-1201 & 220-1202)

**Network configuration overview**

macOS provides both a graphical interface and command-line tools for network setup. In the GUI, you work through **System Settings** or older **System Preferences** to reach network options. In the CLI, you use tools such as `ifconfig`, `netstat`, `ping`, `ping6`, and `networksetup` to inspect and adjust connectivity.

For A+ practical labs, focus on the relationship between the interface you configure and the results you verify. A change in DNS, IP addressing, or privacy-related network behavior should be followed by a check of address assignment, routing, and name resolution.

- **System Settings** is the main place to manage network options on newer macOS versions.
- **System Preferences** is the older name and layout on earlier versions.
- **CLI tools** help you confirm what the GUI changed and troubleshoot when the GUI is not enough.

**Using the GUI to reach network settings**

You typically start from the Apple menu, then open the network area for the active connection. On macOS Ventura and later, you select the connection and open its details. On older versions, you use the advanced network view. From there, you can inspect DNS settings and adjust the servers used for name resolution.

A practical DNS change often involves removing existing entries and adding new ones. This is useful when you need to point the system to a specific resolver or test whether a DNS issue is causing application failures.

- Open the Apple menu and choose **System Settings** or **System Preferences**.
- Go to **Network**.
- Select the active connection, such as `Wi-Fi` or `Ethernet`.
- Open **Details** or **Advanced**, then choose the **DNS** tab.
- Remove old DNS entries with `-` and add new ones with `+`.

> When troubleshooting, change one setting at a time so you can verify which adjustment fixed the problem.

**Common GUI checks and toggles**

Some network issues are tied to privacy or firewall behavior rather than basic IP settings. Evidence shows that disabling or adjusting features such as the firewall or IP address tracking can restore connectivity in some macOS Sequoia cases. A private Wi-Fi address setting may also affect stability on some networks.

Use conservative changes first. If a device is connected to both Wi-Fi and Ethernet, test with one path at a time so you can isolate the cause.

- **Firewall** settings can affect connectivity.
- **Limit IP address tracking** may need to be disabled for troubleshooting.
- **Private Wi-Fi address** can be changed when connection drops are frequent.
- Multiple active interfaces can complicate diagnosis.

**Inspecting network state from the CLI**

The command line gives you a fast way to confirm whether the system has the expected IPv4 and IPv6 state. `ifconfig` shows interface addresses, while `netstat -rn` shows routing tables. These checks help you confirm whether the machine received an address, whether IPv6 is active, and whether routes exist for the network you expect.

You can also test connectivity directly. `ping -4` checks IPv4 reachability, and `ping6` checks IPv6 reachability. For DNS resolution, `dns-sd -G v4v6` helps verify that both address families resolve correctly.

```bash
ifconfig en0
netstat -rn -f inet
netstat -rn -f inet6
ping -4 8.8.8.8
ping6 2001:4860:4860::8888
dns-sd -G v4v6 example.com
```

- `ifconfig` shows interface status and assigned addresses.
- `netstat -rn -f inet` shows IPv4 routes.
- `netstat -rn -f inet6` shows IPv6 routes.
- `ping` and `ping6` test basic reachability.
- `dns-sd` helps confirm name resolution for both families.

**Dual-stack basics**

Dual-stack means the system uses both IPv4 and IPv6. The evidence indicates macOS supports dual-stack by default, using DHCPv4 for IPv4 and Router Advertisements for IPv6 autoconfiguration. That means you often do not need to manually enable both families, but you do need to verify that both are working when troubleshooting.

If a service fails only on one protocol family, compare the IPv4 and IPv6 results separately. That helps you avoid mistaking a DNS problem for a routing problem.

- **IPv4** commonly uses DHCP.
- **IPv6** can use Router Advertisements and autoconfiguration.
- Test each family independently when connectivity is inconsistent.

**Using `networksetup` for service-level configuration**

The `networksetup` command-line tool manages network services from the terminal. It is useful when you need to list available services or make changes without using the GUI. This is especially helpful on systems where remote access or scripting is preferred.

Keep the workflow simple: identify the service, confirm the current state, then apply the change and verify it. That mirrors the GUI process but is faster for repeatable tasks.

```bash
networksetup -listallnetworkservices
```

- Use `networksetup` when you need CLI-based service management.
- List services first so you target the correct interface.
- Verify the result after each change.

**Troubleshooting approach**

A good troubleshooting sequence starts with the active interface, then moves to DNS, routing, and protocol-specific tests. If the system has a valid address but cannot reach sites, DNS or privacy settings may be the issue. If DNS works but traffic still fails, inspect routes and test direct IP connectivity.

When symptoms appear after an OS update, consider recent changes to firewall behavior, local network permissions, or address tracking settings. The safest approach is to isolate one variable at a time and confirm the effect.

1. Check the active interface in the GUI.
2. Confirm addresses with `ifconfig`.
3. Check routes with `netstat -rn`.
4. Test IPv4 and IPv6 with `ping` and `ping6`.
5. Verify DNS resolution with `dns-sd`.
6. Adjust firewall or privacy-related settings only if the earlier checks do not resolve the issue.

**Key takeaways**

The GUI is best for direct configuration, while the CLI is best for verification and repeatable troubleshooting. For exam prep, remember the path from `System Settings` to `Network`, the role of DNS settings, and the core commands used to inspect addresses, routes, and connectivity.

- GUI changes are useful for configuration.
- CLI checks are useful for validation.
- DNS, routing, and protocol-family tests reveal different failure points.
