## **Course:** CompTIA A+ Practical Labs & Troubleshooting (Exams 220-1201 & 220-1202)

**Boot failure caused by storage layout**

A storage misconfiguration can stop a system from starting even when the operating system (OS) volume is present. The boot process depends on the correct disk, partition, and firmware settings working together. If the system reserved or active partition lives on a different drive than the OS volume, removing that drive can prevent startup.

- A boot loop or an `OS Not Found` message often points to a boot-path problem rather than a missing OS installation.
- The computer may still boot when the original storage device is reattached, which suggests the boot files or active partition are on that device.
- In Windows, the boot volume and the system reserved active primary volume can exist on different disks.

**What to check first**

Start with the simplest boot-path checks before changing partitions or reinstalling the OS. Exam-style troubleshooting favors verifying boot options and storage placement before making destructive changes.

1. Check the firmware boot order and confirm the intended system disk is first.
2. Verify whether the OS volume and the active/system reserved partition are on the same drive.
3. Reconnect any removed storage devices that may contain boot files.
4. Confirm the system is using the expected boot mode for the installed OS.

> If the machine only starts when a specific SSD is installed, do not assume that SSD is the OS drive. It may only contain the active boot partition.

**Understanding the storage mismatch**

A common failure pattern is a computer that boots from one disk for Windows but relies on another disk for startup metadata. This can happen after drive replacement, drive removal, or an incomplete migration.

- The OS volume may be on an NVMe drive while the system reserved active partition remains on a separate SSD.
- Removing the disk that holds the active partition can stop the machine at startup even though the C: drive is still present.
- Disk Management can reveal which disk contains the boot volume and which disk contains the active primary partition.

**Recovery approach**

Use a conservative recovery sequence so you preserve data and avoid making the layout worse.

1. Restore the missing drive if it contains the active or system reserved partition.
2. Recheck boot order in firmware after reconnecting storage.
3. If the system starts, document the current disk layout before changing anything.
4. If you need to move boot responsibility, plan the change carefully so the boot files and OS stay aligned.

**Boot media and repair tools**

When the system cannot start normally, bootable installation media can help you access repair tools. A bootable USB drive must be prepared correctly before it can be used for recovery.

- Format the USB drive with a supported file system.
- Create a bootable partition on the USB drive.
- Write the OS installation image to the USB drive.
- Do not treat the USB drive itself as the installation target.

```text
Bootable USB recovery flow:
1. Prepare installation media.
2. Boot from the USB drive.
3. Use repair options to inspect startup configuration.
4. Correct the storage layout or boot order.
```

**Troubleshooting mindset for A+**

For CompTIA A+ troubleshooting, focus on the symptom and the boot chain. A failed boot after storage changes usually means the system cannot find the correct startup path, not necessarily that the OS is damaged.

- Check boot options first when the system reports `OS Not Found`.
- Treat unexpected boot loops after drive removal as a storage dependency issue.
- Use the disk layout to identify where the boot files actually live.
- Prefer reversible steps before partition changes or OS repair.

**Key takeaways**

Storage misconfiguration can break booting when the OS and boot metadata are split across drives. The fastest recovery path is usually to verify boot order, identify the active partition, and restore the missing storage device if needed. Once the system starts, you can correct the layout with a clear view of which disk is responsible for booting.
