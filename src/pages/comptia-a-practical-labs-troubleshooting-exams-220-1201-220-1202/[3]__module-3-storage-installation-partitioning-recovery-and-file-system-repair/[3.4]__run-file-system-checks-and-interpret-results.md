## **Course:** CompTIA A+ Practical Labs & Troubleshooting (Exams 220-1201 & 220-1202)

**File system check overview**

A file system check verifies the file system and file system metadata on a volume. On Windows, `chkdsk` is the built-in utility used for this task. It can report status only, or it can repair errors when you use repair-related parameters.

Use file system checks when you suspect logical corruption, metadata issues, or disk problems that affect access to files. The goal is to confirm the volume state and decide whether repair is needed.

- `chkdsk` checks a volume for logical and physical errors.
- Without parameters, it shows status only and does not fix errors.
- With `/f`, `/r`, `/x`, or `/b`, it attempts to fix errors on the volume.
- You need local Administrator privileges, or equivalent, to run it.

**Running `chkdsk`**

You typically run `chkdsk` from an elevated Command Prompt. Open Command Prompt as an administrator before you start, because standard user rights are not enough for the command.

```cmd
chkdsk C:
chkdsk C: /f
chkdsk C: /r
```

- `C:` is the target volume in these examples.
- `/f` fixes detected file system errors.
- `/r` looks for bad sectors and attempts recovery of readable information.
- `/x` and `/b` are also repair-related options.

> Interrupting `chkdsk` is not recommended. If you cancel it, the volume may still be left in a usable state, but you should treat the result cautiously and verify the disk again if needed.

**Interpreting results**

The output tells you whether the volume is healthy, whether errors were found, and whether repairs were applied. If you run the command without repair parameters, the result is informational only.

When you review the output, focus on these points:

- Whether the tool reports logical or physical errors.
- Whether the volume status is clean or needs repair.
- Whether the command completed normally or was interrupted.
- Whether a follow-up scan is needed after repair.

A practical interpretation workflow is to check the status first, then decide whether to rerun with repair options. If the volume still shows problems after repair, you may need deeper troubleshooting beyond a basic file system check.

**Linux read-only file system behavior**

On Linux, a serious file system error or hardware issue can cause the system to remount the file system as read-only. This is a protective response that helps prevent further damage, but it also blocks writes, directory creation, and other changes.

- A read-only file system prevents normal write operations.
- Common causes include disk hardware failure, file system corruption, and mount issues.
- The behavior is protective, not a normal operating mode for a writable volume.

<details>
<summary>Quick comparison: what the symptom means</summary>

| Platform | Typical check behavior | What the result can indicate |
|---|---|---|
| Windows | `chkdsk` scans and can repair when requested | Logical or physical errors on the volume |
| Linux | File system may remount as read-only | Corruption, hardware issues, or mount-related problems |

</details>

**Practical exam focus**

For troubleshooting questions, match the tool or symptom to the situation:

1. Use `chkdsk` when you need to check a Windows volume for file system errors.
2. Use repair parameters when the scan shows issues that need fixing.
3. Treat a read-only file system on Linux as a warning sign that the system is protecting data.
4. Verify results after any repair so you know whether the problem is resolved.

**Key takeaways**

File system checks help you confirm volume health and identify corruption before it causes more damage. On Windows, `chkdsk` is the primary tool to inspect and repair a volume. On Linux, a read-only remount often signals that the system detected a serious issue and is limiting writes to protect data.
