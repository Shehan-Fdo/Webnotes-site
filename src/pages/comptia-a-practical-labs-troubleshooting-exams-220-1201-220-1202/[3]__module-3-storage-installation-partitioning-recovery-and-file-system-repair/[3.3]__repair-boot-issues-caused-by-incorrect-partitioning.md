## **Course:** CompTIA A+ Practical Labs & Troubleshooting (Exams 220-1201 & 220-1202)

**Boot issues from incorrect partitioning**

Incorrect partitioning can break the boot path on systems that use a GUID Partition Table (<abbr title="GUID Partition Table">GPT</abbr>) and an <abbr title="Extensible Firmware Interface">EFI</abbr> system partition. When the boot configuration data is missing or damaged, the system may fail to find the boot drive, Startup Repair may fail, and recovery often shifts to command-line tools.

A practical repair approach is to identify the correct disk and partition, then rebuild the boot files instead of converting the drive to <abbr title="Master Boot Record">MBR</abbr>. In the evidence, recovery attempts included `bcdboot`, `bcdedit`, `fixboot`, and offline `sfc`, which shows that boot repair often requires several passes before the system starts normally.

- **GPT** systems keep boot data separate from the main Windows volume.
- The **EFI partition** must be available before you can repair boot files.
- Startup Repair may fail even when the Windows installation is still present.
- `fixboot` can return `element not found` when the boot structure is damaged.
- Offline `sfc /scannow` can find corruption without fully repairing it.

**Typical recovery workflow**

You usually begin from Windows installation media or a recovery drive, then open Command Prompt and inspect the disk layout. The goal is to locate the boot disk, make the system partition available, and point boot files back to the Windows installation.

1. Boot from the Windows installation media.
2. Press <kbd>Shift</kbd> + <kbd>F10</kbd> to open Command Prompt.
3. Use `diskpart` and `list disk` to identify the correct drive by size and GPT status.
4. Select the boot disk with `select disk <number>`.
5. Make the EFI or system partition available so repair tools can access it.
6. Rebuild boot files with `bcdboot` using the Windows directory and the system partition.
7. If needed, use `bcdedit` to inspect or adjust boot configuration entries.
8. Run offline repair tools such as `sfc /scannow /offbootdir=... /offwindir=...` when file corruption is suspected.

```text
diskpart
list disk
select disk 0
```

> Be careful when selecting disks and partitions. Choosing the wrong volume can overwrite the wrong boot files or make recovery harder.

**Repair commands and what they do**

The evidence shows a common repair sequence that renames the existing BCD store, rebuilds boot files, and then checks configuration. This is useful when the boot configuration data is unreadable or the system partition is not linked correctly.

```cmd
attrib BCD -s -h -r
ren BCD BCD.bak
bcdboot E:\Windows /l en-us /s H: /f ALL
```

- `attrib` removes system, hidden, and read-only attributes from `BCD`.
- `ren` preserves the old store as a backup before rebuilding.
- `bcdboot` copies boot files from the Windows installation to the target system partition.
- `/f ALL` targets multiple firmware types when appropriate.
- `bcdedit` is used after the boot files exist and the store can be opened.

**What to check when repair fails**

If repair commands fail, the issue may be one of access, partition selection, or deeper file corruption. The evidence includes cases where `boot configuration could not be opened`, `element not found`, and `sfc` found corrupt files but could not fix them.

- Verify that the EFI partition is mounted and assigned a drive letter.
- Confirm that the Windows installation path matches the offline environment.
- Recheck the disk number and partition letter before rerunning `bcdboot`.
- Review offline `CBS.log` output if `sfc` reports unfixable corruption.
- Avoid switching partition styles unless the recovery plan specifically requires it.

**Partitioning mistakes that can break boot**

Incorrect partitioning can also happen during operating system installation or when another operating system mounts the wrong EFI partition. In the evidence, Linux installation on some systems could overwrite the Windows bootloader if it used the Windows EFI partition instead of its own separate EFI partition.

- A shared or misidentified EFI partition can overwrite another OS bootloader.
- Reformatting the EFI system partition can leave the machine unable to boot.
- Boot problems may appear after updates, reinstallations, or disk layout changes.
- The fix may involve correcting partition references rather than reinstalling the entire OS.

**Troubleshooting mindset**

For exam scenarios, focus on the sequence: identify the disk, expose the correct partition, rebuild boot files, and verify the boot configuration. That order matters because boot tools depend on the partition being accessible before they can repair the startup path.

A good rule is to repair the boot structure first and only consider broader OS recovery if the boot files and partition layout are confirmed correct.
