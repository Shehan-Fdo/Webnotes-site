## **Course:** CompTIA A+ Practical Labs & Troubleshooting (Exams 220-1201 & 220-1202)

**Why baseline capture matters**

A baseline system state is a reference snapshot of how a device looks and behaves when it is healthy. You use it to compare later changes during troubleshooting, security checks, and routine support work. The goal is repeatability: if you can collect the same core details each time, you can spot drift faster and explain what changed.

Command-line tools are especially useful because they support repeatable collection and can be used on local or remote systems where supported. On Windows, PowerShell can query event logs from the local computer and remote computers with `Get-EventLog`. On Linux, `journalctl` is a standard way to review logs. On macOS and ChromeOS Flex, the practical focus is on collecting system information and preserving it in a consistent way for later comparison.

- Capture the same categories each time: system identity, software state, and recent logs.
- Keep the process simple so you can repeat it during incident response or routine audits.
- Save outputs in a dated folder or file name so you can compare snapshots later.

**Windows 11 baseline capture**

Windows supports event-log collection through PowerShell. The `Get-EventLog` cmdlet gets events and event logs from local and remote computers, and it can target a specific log name or list available logs. That makes it useful when you need a quick baseline of recent activity on a workstation or server.

```powershell
Get-EventLog -LogName System -Newest 20
Get-EventLog -List
```

Use this kind of output to establish what the system was doing before a problem appeared. If you are comparing systems, keep the same log name and time window so the results stay consistent.

- Use `-LogName` to focus on a specific log.
- Use `-Newest` to limit the amount of output.
- Use `-ComputerName` when you need to query a remote system.
- Use `-List` when you want to see available logs first.

> `Get-EventLog` is useful for baseline collection, but it only helps if you record the same log sources and time ranges each time.

**macOS Sequoia baseline capture**

On macOS, baseline capture usually centers on system profiling and preserving the output for later review. The evidence here supports the idea of using system-profiler-style information as part of device troubleshooting, which is useful when you need to compare a clean device against one that has changed.

A practical baseline should include the device’s current hardware and software state, then store that output in a file you can revisit. Keep the collection method consistent so you can compare one Mac against another or compare the same Mac over time.

- Record the current system profile before making changes.
- Save the output with a clear date and device identifier.
- Use the same collection method for every Mac you support.

**Ubuntu 24.04 baseline capture**

On Ubuntu, `journalctl` is the key log-review tool for baseline and troubleshooting work. It helps you inspect system logs in a structured way, which is valuable when you want to know what happened before a failure or configuration change.

A baseline on Ubuntu should include recent log activity and any other system details you normally compare during support. The main idea is to preserve a known-good state so you can detect later changes.

```bash
journalctl -n 50
journalctl --since "today"
```

- Use `journalctl` to review recent system activity.
- Narrow the time range when you only need a short baseline window.
- Keep the output consistent so comparisons stay meaningful.

**ChromeOS Flex baseline capture**

For ChromeOS Flex, baseline capture is about documenting the current device state in a way that supports later troubleshooting. Because the platform is often used in managed or lightweight environments, consistency matters more than collecting excessive detail.

Focus on the current configuration and any information that helps you identify the device later. When you compare baselines, you want to know whether the device changed, not just whether it still boots.

- Record the device identity and current configuration.
- Capture the state before applying updates or troubleshooting changes.
- Keep notes concise so you can compare them quickly.

**A repeatable baseline workflow**

1. Identify the device and operating system.
2. Collect the current system state using the platform-appropriate tool.
3. Save the output with a date and device name.
4. Compare the saved baseline against later captures.
5. Note any changes before you make repairs or updates.

This workflow helps you separate pre-existing conditions from new problems. It also gives you a practical record for audits, escalation, and follow-up support.

**What to keep in every baseline**

- Device name or identifier.
- Operating system version.
- Recent log output or event history.
- Any notable configuration details relevant to the issue.
- Timestamp of the capture.

**Troubleshooting value**

A baseline is most useful when you treat it as a reference point, not a one-time report. If a device starts failing, you compare the current state against the saved baseline to see what changed. That comparison can reveal new logs, altered settings, or missing information that points you toward the cause.

For exam prep, remember the practical pattern: collect, save, compare, and document. The exact tool varies by platform, but the support goal stays the same.
