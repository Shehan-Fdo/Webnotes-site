## **Course:** CompTIA A+ Practical Labs & Troubleshooting (Exams 220-1201 & 220-1202)

**IP configuration basics**

IP settings tell you how a device identifies itself on a network and how it reaches other networks. When you verify these settings, you confirm whether the device has a usable address, the correct subnet mask, and a valid default gateway. This is a core troubleshooting step because connectivity problems often start with an incorrect or missing address assignment.

Client devices usually get their settings in one of two ways:

- **Static addressing** means you manually assign the IP settings on the device.
- **Dynamic addressing** means a <abbr title="Dynamic Host Configuration Protocol">DHCP</abbr> server leases the settings to the client.
- End-user devices commonly use DHCP by default because they move between networks more often than servers do.

A quick check of the current configuration can reveal whether the device is ready to communicate or whether you need to investigate the network path, the DHCP server, or the local adapter settings.

**What to verify**

When you inspect IP settings, focus on the values that affect basic reachability:

- **IP address** identifies the device on the network.
- **Subnet mask** defines which part of the address is the network and which part is the host.
- **Default gateway** is the router used to reach other networks.
- **DNS** settings affect name resolution, which matters when addresses work but hostnames do not.

> A `169.254.x.x` address is an <abbr title="Automatic Private IP Addressing">APIPA</abbr> or link-local address. It usually indicates the device could not lease an address from DHCP.

**Windows verification**

On Windows, you typically use command-line tools to view the current configuration faster than through the graphical interface. The most common command is `ipconfig`, which shows the active IP settings for the adapter.

```powershell
ipconfig
ipconfig /all
```

Use the output to confirm:

- The adapter has an assigned address.
- The address matches the expected network.
- The gateway and DNS values are present when required.
- The device is not stuck on an APIPA address.

If you need more detail, `ipconfig /all` provides the full adapter configuration, which is useful when you are checking DHCP status or comparing multiple interfaces.

**macOS verification**

On macOS, you verify IP settings from the network interface details or by using command-line tools in Terminal. The goal is the same: confirm the active address, gateway, and related parameters for the interface that is actually in use.

```bash
ifconfig
networksetup -getinfo Wi-Fi
```

Use the interface output to identify the active adapter and confirm that the device has the expected network configuration. If the system is connected but not reaching other hosts, compare the interface address with the subnet and gateway you expect for that network.

**Ubuntu verification**

On Ubuntu, you can inspect the current network configuration from the command line. This is especially useful when you need a fast check during troubleshooting or when you are working remotely.

```bash
ip addr
ip route
```

Use `ip addr` to confirm the assigned address on the interface, and use `ip route` to verify the default route. Together, these commands help you determine whether the system has both a valid local address and a path to other networks.

**Troubleshooting workflow**

1. Check the current IP address on the device.
2. Confirm the address is in the expected range for the network.
3. Verify the subnet mask and default gateway.
4. Look for APIPA or other unexpected addressing.
5. Compare the configuration with the network’s DHCP or static plan.
6. If the address is missing or incorrect, investigate the adapter, DHCP service, or manual configuration.

This workflow keeps you focused on the most common causes of basic connectivity failures. If the address is correct but the device still cannot communicate, the issue may be outside the local IP configuration.

**Practical reminders**

- Use the CLI when you need faster access to IP details.
- Treat a link-local address as a troubleshooting clue, not a final configuration.
- Verify the interface that is actually connected, especially on systems with multiple adapters.
- Check DNS separately when IP connectivity works but name resolution fails.
