## **Course:** CompTIA A+ Practical Labs & Troubleshooting (Exams 220-1201 & 220-1202)

**Adapter and driver symptoms**

When a network adapter stops working, the issue may be the device itself, the driver, or something disabling the device after it connects. In Windows, a user may see the adapter disappear, become disabled in Device Manager, or fail across multiple networks and connection types.

- A **driver** is the software that lets Windows communicate with hardware.
- A **network adapter** can be disabled by a user action, a script, a policy, or another automated process.
- If the problem affects Wi-Fi, USB tethering, and Ethernet-style devices, you should suspect a broader system-level cause rather than a single access point.

A practical troubleshooting mindset is to verify whether the adapter is being turned off, whether the driver is failing, and whether a security or management action is involved.

**What to check in Device Manager**

Device Manager is the first place to confirm whether the adapter is present and enabled. The evidence shows cases where specific devices, such as wireless adapters and Remote NDIS-based Internet Sharing devices, were disabled automatically.

- Look for the adapter under the correct hardware category.
- Check whether the device shows as disabled or missing.
- Reinstalling a driver may help when the driver is corrupt, but repeated disablement suggests the root cause is elsewhere.
- If the issue returns after reinstall, reset, or even a Windows reinstall, treat it as a persistent configuration or control problem.

> If the adapter disables itself immediately after connection, do not assume the access point is the cause. Check the local system for automation, policy, or device control first.

**Security logging and device disable events**

Windows can log when a device is disabled. Event ID 6420 indicates that a device was disabled, and the log includes details that help identify who performed the action and which device was affected.

- **Event ID 6420** records a successful device disable action.
- The event can show the **Security ID**, account name, device ID, and device name.
- The device ID may appear as the device instance path in Device Manager.
- The device name may match the description shown in Device Manager.

This event is useful when you need to determine whether a person, script, or management tool disabled the adapter. It does not by itself prove intent, but it gives you a concrete audit trail.

**Troubleshooting approach**

Use a step-by-step process so you can separate a bad driver from a disabled device or external control.

1. Open Device Manager and confirm the adapter is present.
2. Check whether the device is disabled or repeatedly changing state.
3. Review relevant Windows security logs for device disable events such as 6420.
4. Reinstall or update the driver if the device appears healthy but still fails.
5. If the issue persists across networks and after reinstall, investigate automation, policy, or other software that may be disabling the adapter.

**Practical interpretation**

A driver reinstall is a valid first response, but it is not the only answer. If the adapter disables automatically after connection, the evidence points toward an automated process or another system component acting on the device. That means you should verify the device state, inspect logs, and look for repeatable disable events before concluding that the hardware is bad.

**Key takeaways**

- **Disabled adapter** symptoms can look like a network outage even when the hardware is still present.
- **Event ID 6420** helps confirm that a device was disabled and can identify the account involved.
- Repeated failures after reinstalling the driver suggest a deeper issue than a simple driver mismatch.
- Use Device Manager and Windows logs together to narrow the cause.
