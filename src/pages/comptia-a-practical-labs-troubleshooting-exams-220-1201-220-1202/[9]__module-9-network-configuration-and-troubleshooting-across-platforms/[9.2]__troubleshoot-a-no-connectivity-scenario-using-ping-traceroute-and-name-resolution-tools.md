## **Course:** CompTIA A+ Practical Labs & Troubleshooting (Exams 220-1201 & 220-1202)

**Connectivity troubleshooting overview**

When a device cannot reach a host or service, you use simple tests to narrow the failure point. The goal is to separate local network problems, routing issues, DNS (Domain Name System) failures, and remote-host problems.

Start with the most direct checks. If you can reach an IP address but not a name, the issue is likely name resolution. If you cannot reach anything by IP, the problem is more likely local connectivity, addressing, or routing.

- **Ping** confirms whether a host responds and gives a basic latency check.
- **Traceroute** shows the path packets take toward a destination.
- **Name resolution tools** help you verify whether a hostname maps to the expected IP address.
- Testing by IP address first helps you avoid confusing DNS failures with general network failure.

**Using ping to establish a baseline**

`ping` sends ICMP (Internet Control Message Protocol) echo requests to a host and waits for replies. It is usually the first tool you use because it quickly tells you whether a destination is reachable.

A successful ping does not prove every service works, but it does show that basic network communication is functioning. If ping to a hostname fails, try the same target by IP address to determine whether DNS is the issue.

```bash
ping -c 4 google.com
ping 192.0.2.10
```

- Use `ping` to test reachability and response time.
- Compare hostname results with IP address results.
- Treat a successful ping to `127.0.0.1` as a local stack check, not proof of external connectivity.

> If you can ping an IP address but not the hostname, focus on DNS before changing routing or cabling.

**Using traceroute to find where traffic stops**

`traceroute` shows the route packets take to reach a destination. It helps you identify the hop where communication slows down or fails.

This is useful when a host is unreachable beyond the local network. A path that stops at a specific hop can point you toward a gateway, upstream router, or transit problem rather than a local device issue.

```bash
traceroute google.com
```

- Use `traceroute` after basic reachability checks.
- Look for the last responding hop when traffic fails.
- Use the path information to decide whether the issue is local, upstream, or remote.

**Checking name resolution**

Name resolution tools help you confirm whether a hostname resolves correctly. The evidence highlights `dig` and `nslookup` as common tools for DNS troubleshooting, including tracing DNS query paths and checking lookup behavior.

If a website works by IP but not by name, verify the DNS answer before assuming the service is down. A bad or missing DNS response can make the internet appear unavailable even when the network is otherwise healthy.

```bash
dig example.com
nslookup example.com
```

- Use `dig` or `nslookup` to test DNS lookup results.
- Compare the returned IP address with the expected value when you have one.
- Use DNS tracing when you need to follow the query path from resolver to authoritative server.

**Practical troubleshooting flow**

1. Check whether the device has a basic network link and local connectivity.
2. Ping `127.0.0.1` to confirm the local TCP/IP stack responds.
3. Ping a known IP address on the local network or internet.
4. Ping the same destination by hostname.
5. Run `traceroute` if the destination is reachable by name or IP only partway.
6. Use `dig` or `nslookup` if hostname resolution fails or returns an unexpected address.

This sequence helps you avoid guessing. Each step narrows the problem domain and gives you a clearer next action.

**What each tool tells you**

- `ping` answers: can you reach the host?
- `traceroute` answers: where does the path fail?
- `dig` and `nslookup` answer: does the name resolve correctly?
- Comparing hostname and IP tests helps you distinguish DNS issues from routing issues.

**Common interpretation patterns**

- Ping works by IP, but not by name: investigate DNS.
- Ping fails by IP and by name: investigate local connectivity, addressing, gateway, or routing.
- Traceroute stops at the first hop: check the local gateway or nearby network path.
- Name resolution returns an unexpected address: verify the DNS server and lookup results.

**Exam focus**

For A+ troubleshooting, you should be able to choose the right tool for the symptom. The key is to test in a logical order and use the results to isolate the failure point instead of changing multiple settings at once.
