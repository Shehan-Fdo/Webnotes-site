## **Course:** CompTIA A+ Practical Labs & Troubleshooting (Exams 220-1201 & 220-1202)

**Security logs and policy blocks**

Windows event logs record activity that helps you trace system behavior, respond to events, and keep systems secure. In Event Viewer, the Windows Logs area includes the security log, which is especially useful when you need to confirm whether access was blocked by policy or whether an audit event occurred.

The security log is not just a general activity feed. It is part of the Windows auditing model, so you use it to look for evidence of logon activity, policy changes, and log management events. When a user reports that something was blocked, the log often tells you whether the issue is tied to permissions, auditing, or a security setting.

- **Event Viewer** is the main tool for reviewing Windows logs.
- The security log helps you trace activity and investigate suspicious or unexpected behavior.
- Policy-related blocks often appear alongside audit or access-control events rather than application errors.

**Security log access and permissions**

Windows lets you customize event log security locally or through Group Policy. You can grant access rights such as `Read`, `Write`, and `Clear` for event logs in general. For the security log, the rules are stricter: you can change only `Read` and `Clear` permissions.

Write access to the security log is reserved for the Windows Local Security Authority (LSA) and identities with the `Manage auditing and security log` privilege enabled. That means a blocked action may reflect a deliberate security control rather than a malfunction.

> If a user cannot view or manage the security log as expected, check whether the issue is caused by log permissions or by a policy that limits access.

- Use local settings when you need to adjust a single system.
- Use Group Policy when you need consistent log security across multiple systems.
- Treat restricted security-log access as a possible policy enforcement issue.

**Events that help identify policy-related blocks**

Some security log events are especially useful when you are checking for policy-related behavior. A cleared audit log, a full log, or a logging service failure can all affect what you can see during troubleshooting.

Common events include:

- `1102` — the audit log was cleared
- `1104` — the security log is now full
- `1100` — the event logging service has shut down
- `1108` — the event logging service encountered an error
- `1105` — event log automatic backup

These events do not always prove a policy block by themselves, but they help you narrow the cause. For example, a full log can prevent new entries from being recorded, and a cleared log can remove evidence you expected to review.

**How to review logs during troubleshooting**

1. Open `Event Viewer`.
2. Check the Windows Logs area, especially the security log.
3. Look for events that match the time of the reported issue.
4. Compare the event details with the user’s description of the block.
5. Check whether permissions, auditing settings, or log state explain the behavior.

When you review the log, focus on the sequence of events rather than a single entry. A policy-related block often becomes clearer when you correlate access attempts with log-clearing events, full-log conditions, or changes in security settings.

**Practical interpretation**

A blocked action can come from several layers:

- A permission setting may prevent access to the log itself.
- A Group Policy setting may enforce centralized security behavior.
- A full or cleared security log may hide the evidence you need.
- A logging service error may interrupt normal audit recording.

Use conservative conclusions when evidence is incomplete. If the log does not show the expected event, consider whether the log was cleared, full, or inaccessible before assuming the action never occurred.

**Key takeaways**

- The security log is a primary source for investigating policy-related blocks in Windows.
- Group Policy and local settings can control event log security.
- Security-log permissions are limited compared with other logs.
- Events such as `1102`, `1104`, `1100`, `1105`, and `1108` help explain missing or blocked audit data.
