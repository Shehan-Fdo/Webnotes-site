## **Course:** CompTIA A+ Practical Labs & Troubleshooting (Exams 220-1201 & 220-1202)

**Safe investigation mindset**

When you investigate a suspicious file or process, your first goal is to avoid making the situation worse. Use simulated artifacts for practice so you can learn the workflow without exposing a real system to risk. In practice, security researchers and incident-response teams analyze suspicious files to determine whether they are threats, unwanted applications, or normal files.

- Treat unknown files as potentially harmful until you verify them.
- Use a controlled lab or simulated sample instead of opening a real suspicious attachment.
- Focus on observation, documentation, and evidence preservation before any remediation.

**What you are trying to determine**

A suspicious artifact is not automatically malware. Your job is to collect enough context to decide whether the item is malicious, unwanted, or benign. Malware analysis tools help teams dissect and understand modern threats, especially when code is designed to evade traditional detection.

- Check whether the file or process matches expected behavior.
- Compare the artifact against known-good baselines when available.
- Note indicators that suggest malicious, unwanted, or normal activity.

**Safe handling and evidence preservation**

Incident response depends on preserving evidence and keeping the investigation defensible. A chain of custody records who handled evidence and helps preserve integrity for possible legal proceedings. If you are working with a sample, keep the handling process controlled and documented.

- Record where the artifact came from and when you found it.
- Limit unnecessary interaction with the file or process.
- Keep notes on every action you take during analysis.

> Do not execute an unknown file on a production system. Use safe, simulated artifacts and a controlled environment for practice and validation.

**Using analysis tools and portals**

Security teams rely on a growing ecosystem of malware analysis tools to inspect suspicious samples. Microsoft also provides a file submission path for malware analysis, where researchers can review suspicious files and determine whether they are threats, unwanted applications, or normal files.

- Use analysis tools to inspect behavior and characteristics.
- Submit suspicious samples only through approved workflows.
- Provide as much relevant context as possible when reporting a file for review.

**Incident response workflow**

Incident response is a process, not a single action. Teams test their incident response plans before relying on them in real events, and simulated artifacts are useful for that practice. A practical workflow keeps the investigation structured and repeatable.

1. Identify the suspicious file or process.
2. Isolate the sample from normal user activity.
3. Collect basic details such as name, location, and observed behavior.
4. Analyze the artifact in a safe environment.
5. Document findings and decide whether escalation is needed.

**What to document during analysis**

Good notes make later review easier and support incident response. Capture the details that help another technician understand what you saw and why you made your conclusion.

- File name, path, and process name.
- User context and time observed.
- Behavior that triggered suspicion.
- Any submission or escalation steps taken.

**Practical exam focus**

For A+ exam prep, expect questions about safe handling, evidence integrity, and the purpose of analysis tools. The key idea is to investigate without risking the system, preserve what you observe, and escalate through approved channels when needed.

- Use safe simulated artifacts for practice.
- Preserve evidence and maintain chain of custody when applicable.
- Choose the right tool or workflow for analysis and reporting.
