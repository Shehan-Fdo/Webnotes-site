## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Azure App Service overview**

Azure App Service is a managed platform for hosting web applications, REST APIs, and background-facing web workloads without managing the underlying operating system. You choose an **App Service plan** first, then create one or more apps that run on that plan. The plan defines the compute, region, scale, and feature set; the app defines the code, runtime, configuration, and identity.

For AZ-204, the key decision is not just “can I host it?” but “which plan and hosting model match the app’s runtime, scaling, networking, and deployment needs?”

- **App Service plan** = the compute container for apps.
- **Web app** = the application resource that runs in the plan.
- **Region** matters because the plan and app must live in the same Azure region for normal operation.
- **OS choice** matters because Windows and Linux plans support different runtime and deployment behaviors.

**App Service plan fundamentals**

An App Service plan is a logical grouping of compute resources. Multiple apps can share the same plan, which is useful when apps have similar performance needs and can share capacity. If one app consumes too much CPU or memory, it can affect the others on the same plan, so plan sharing is a cost and isolation tradeoff.

The plan determines:

- **Pricing tier** and billing model.
- **Scale-out** limits and autoscale support.
- **Feature availability** such as custom domains, TLS/SSL bindings, deployment slots, backups, and Always On.
- **Operating system** and hosting model.

A practical rule: choose the smallest tier that supports the features you need, then scale up only when required.

**Operating system and hosting model choices**

App Service plans support Windows and Linux, but the app kind and runtime must align with the plan. This is a common exam trap: you cannot freely mix a Windows plan with a Linux app configuration.

- **Windows plans**
  - Common for .NET Framework, ASP.NET, and Windows-oriented workloads.
  - Support IIS-based hosting.
  - Often used when you need Windows-specific dependencies.
- **Linux plans**
  - Common for .NET, Node.js, Python, Java, and PHP workloads.
  - Use Linux-based runtime stacks or custom containers.
  - Often preferred for containerized or cross-platform apps.

When you create the app, you must align:

- The **plan OS** with the app’s runtime stack.
- The **runtime stack** with the deployment model.
- The **kind** and configuration with the selected hosting model.

> If you choose a Linux plan, make sure the app is created as a Linux web app or container app in App Service. A Windows app cannot simply be “moved” onto a Linux plan without matching the hosting model.

**Pricing tiers and feature selection**

The tier you choose affects both cost and capabilities. Exam questions often ask which tier is required for a feature.

| Tier family | Typical use | Common feature notes |
|---|---|---|
| Free / Shared | Learning, dev/test, very small workloads | No custom domains, limited scale, no production-grade features |
| Basic | Small production or dev workloads | Supports custom domains and Always On; limited scale and no deployment slots |
| Standard | Production web apps | Supports deployment slots, autoscale, custom domains, TLS/SSL, backups for supported scenarios |
| Premium / Isolated | Higher scale, stronger isolation, advanced networking | More slots, more scale, better performance, private networking options, enterprise scenarios |

Important exam-oriented reminders:

- **Custom domains** require at least **Basic**.
- **Always On** requires at least **Basic**.
- **Deployment slots** require at least **Standard**.
- **TLS/SSL bindings** are available in paid tiers, commonly **Standard** and above for production scenarios.
- **Backups** are supported in higher tiers, but availability depends on OS and plan type; do not assume every Linux scenario supports the same backup behavior as Windows.

**Creating an App Service plan**

You can create a plan in the Azure portal, Azure CLI, ARM templates, or Bicep. The plan is usually created before the app so you can control region, OS, and tier.

Example Azure CLI:

```bash
az group create \
  --name rg-az204-web \
  --location eastus

az appservice plan create \
  --name asp-az204-prod \
  --resource-group rg-az204-web \
  --location eastus \
  --sku S1 \
  --is-linux
```

Notes on the example:

- `--sku S1` creates a **Standard** plan.
- `--is-linux` creates a Linux plan.
- The resource group and region should match your deployment strategy.

A Windows plan example:

```bash
az appservice plan create \
  --name asp-az204-win \
  --resource-group rg-az204-web \
  --location eastus \
  --sku B1
```

**Creating a web app**

After the plan exists, create the web app and attach it to the plan. The runtime stack must match the OS and app type.

Linux web app example:

```bash
az webapp create \
  --resource-group rg-az204-web \
  --plan asp-az204-prod \
  --name myaz204linuxapp123 \
  --runtime "DOTNET:8"
```

Because runtime strings can vary by Azure CLI version and supported stacks, verify the available values in your environment before using them in production.

Windows web app example:

```bash
az webapp create \
  --resource-group rg-az204-web \
  --plan asp-az204-win \
  --name myaz204winapp123 \
  --runtime "DOTNET:8"
```

For Windows apps, the runtime string and supported stack depend on the CLI version and the platform support matrix. If you are unsure, check the portal or run:

```bash
az webapp list-runtimes --os windows
az webapp list-runtimes --os linux
```

**Built-in runtime vs custom container**

A common design choice is whether to use a built-in runtime or a custom container.

- **Built-in runtime**
  - Best when your app fits a supported stack such as .NET, Node.js, Python, Java, or PHP.
  - Simplifies deployment and patching.
  - Reduces operational overhead.
- **Custom container**
  - Best when you need OS-level dependencies, a specific runtime version, or a fully controlled environment.
  - Useful for legacy dependencies or when the app must behave the same across environments.
  - Requires container registry and image management.

Use a built-in runtime when Azure supports your stack cleanly. Use a custom container when portability or dependency control matters more than simplicity.

Example custom container deployment:

```bash
az webapp create \
  --resource-group rg-az204-web \
  --plan asp-az204-prod \
  --name mycontainerapp123 \
  --deployment-container-image-name myregistry.azurecr.io/myapp:1.0
```

**Bicep examples**

Bicep is common in AZ-204 because it expresses infrastructure as code clearly and repeatably.

Windows plan and app:

```bicep
param location string = resourceGroup().location
param planName string = 'asp-az204-win'
param appName string = 'myaz204winapp123'

resource plan 'Microsoft.Web/serverfarms@2023-12-01' = {
  name: planName
  location: location
  sku: {
    name: 'B1'
    tier: 'Basic'
  }
  kind: 'app'
  properties: {
    reserved: false
  }
}

resource app 'Microsoft.Web/sites@2023-12-01' = {
  name: appName
  location: location
  kind: 'app'
  properties: {
    serverFarmId: plan.id
    httpsOnly: true
  }
}
```

Linux plan and app:

```bicep
param location string = resourceGroup().location
param planName string = 'asp-az204-linux'
param appName string = 'myaz204linuxapp123'

resource plan 'Microsoft.Web/serverfarms@2023-12-01' = {
  name: planName
  location: location
  sku: {
    name: 'S1'
    tier: 'Standard'
  }
  kind: 'linux'
  properties: {
    reserved: true
  }
}

resource app 'Microsoft.Web/sites@2023-12-01' = {
  name: appName
  location: location
  kind: 'app,linux'
  properties: {
    serverFarmId: plan.id
    httpsOnly: true
    siteConfig: {
      linuxFxVersion: 'DOTNETCORE|8.0'
    }
  }
}
```

Be careful with runtime strings such as `linuxFxVersion`. The exact value depends on the stack and hosting model, and container-based apps use different settings.

**Deployment slots**

Deployment slots let you stage a new version of your app before swapping it into production. This reduces downtime and supports safer releases.

Typical slot workflow:

1. Deploy the new build to a staging slot.
2. Validate the app in the staging environment.
3. Swap staging into production.
4. Roll back by swapping again if needed.

Slots are especially useful for:

- Blue-green style releases.
- Smoke testing before production exposure.
- Configuration validation with production-like settings.

Important slot facts:

- Slot availability depends on the **tier**.
- Slot count increases in higher tiers.
- Standard tier commonly supports at least one additional slot.
- Premium tiers support more slots than Standard.
- Free and Shared do not support deployment slots.

Slots can also have their own settings. Mark settings as **slot settings** when they should stay with the slot during swaps, such as connection strings or environment-specific app settings.

**Identity and secrets**

App Service supports **managed identity**, which lets your app authenticate to Azure resources without storing credentials in code. This is a common exam topic because it reduces secret management overhead.

Use managed identity when your app needs to access:

- Azure Key Vault.
- Azure Storage.
- Azure SQL Database.
- Azure Service Bus.

Enable a system-assigned identity when the app only needs one identity tied to that resource. Use a user-assigned identity when multiple apps share the same identity or when you want identity lifecycle independence.

Example CLI:

```bash
az webapp identity assign \
  --resource-group rg-az204-web \
  --name myaz204linuxapp123
```

For secrets, prefer:

- **Key Vault references** in app settings.
- Managed identity for runtime access.
- Avoid hardcoding secrets in source or deployment scripts.

> Do not rely on connection strings alone for identity-based access. Many services use managed identity through SDK configuration or app settings rather than a special connection string format.

**Networking: VNet Integration vs Private Endpoint**

Networking is a frequent source of confusion. The two most important concepts are **Virtual Network (VNet) Integration** and **Private Endpoint**.

- **VNet Integration**
  - Primarily enables the app to make **outbound** calls into a VNet.
  - Useful when your app needs to reach private databases, internal APIs, or private services.
  - Does not make the app itself privately reachable from the VNet by default.
- **Private Endpoint**
  - Provides a private IP for the App Service resource through Azure Private Link.
  - Enables **inbound** access to the app over a private network path.
  - Useful when you want to restrict public access and expose the app privately.

Think of it this way:

- Use **VNet Integration** when your app needs to call private resources.
- Use **Private Endpoint** when clients need to reach the app privately.

Support depends on tier and plan type. In exam scenarios, assume advanced networking features require paid tiers and verify whether the question is about inbound or outbound traffic.

**Custom domains and TLS/SSL**

Custom domains let you map a friendly DNS name such as `www.contoso.com` to your app. TLS/SSL secures traffic with HTTPS.

Typical steps:

1. Add the custom domain in App Service.
2. Verify domain ownership with DNS records.
3. Bind a certificate.
4. Enforce HTTPS if required.

Example Azure CLI for hostname binding:

```bash
az webapp config hostname add \
  --resource-group rg-az204-web \
  --webapp-name myaz204linuxapp123 \
  --hostname www.contoso.com
```

TLS/SSL binding usually requires a paid tier. For production apps, you should also enable HTTPS-only traffic:

```bash
az webapp update \
  --resource-group rg-az204-web \
  --name myaz204linuxapp123 \
  --https-only true
```

**Always On and cold starts**

**Always On** keeps the app loaded so it does not unload during idle periods. This reduces cold starts and improves responsiveness for production workloads.

Use it when:

- The app receives intermittent traffic but must respond quickly.
- You host APIs that should not incur startup delay.
- You want to reduce the first-request penalty after inactivity.

Remember:

- Always On is not available in Free or Shared.
- It is available in Basic and higher.
- It is most useful for production-like workloads.

**Monitoring and diagnostics**

Monitoring should be part of app creation, not an afterthought. Azure App Service supports platform logs, application logs, diagnostic logs, and integration with **Application Insights**.

Use these tools together:

- **Application Insights**
  - Tracks requests, dependencies, exceptions, traces, and performance.
  - Best for application-level observability.
  - Can be enabled during app creation or afterward.
- **App Service logs**
  - Capture web server logs, application logging, and detailed error messages.
  - Useful for platform and runtime troubleshooting.
- **Diagnostic settings**
  - Send logs and metrics to Log Analytics, Event Hub, or Storage.
  - Useful for centralized monitoring and retention.

Example enabling Application Insights during deployment is often done by setting the instrumentation key or connection string as an app setting, or by using portal integration. In modern setups, prefer the Application Insights connection string.

Example app settings:

```bash
az webapp config appsettings set \
  --resource-group rg-az204-web \
  --name myaz204linuxapp123 \
  --settings \
    APPLICATIONINSIGHTS_CONNECTION_STRING="InstrumentationKey=00000000-0000-0000-0000-000000000000;IngestionEndpoint=https://eastus-0.in.applicationinsights.azure.com/"
```

For logs:

```bash
az webapp log config \
  --resource-group rg-az204-web \
  --name myaz204linuxapp123 \
  --application-logging filesystem \
  --level information \
  --web-server-logging filesystem
```

Use Application Insights for code-level telemetry and App Service logs for platform-level troubleshooting. They complement each other rather than replace each other.

**Deployment options**

App Service supports several deployment methods. Choose the one that fits your pipeline and operational model.

- **GitHub Actions / Azure DevOps**
  - Best for CI/CD pipelines.
  - Automates build, test, and deployment.
- **Zip deploy**
  - Good for simple package-based deployments.
  - Often used in scripts and automation.
- **Local Git**
  - Useful for lightweight workflows, though less common in enterprise CI/CD.
- **Container registry deployment**
  - Best for custom container apps.
- **FTP/FTPS**
  - Legacy option; generally avoid for modern production workflows.

Example zip deploy:

```bash
az webapp deployment source config-zip \
  --resource-group rg-az204-web \
  --name myaz204linuxapp123 \
  --src app.zip
```

**Practical exam scenario**

You need to host a .NET 8 web app with staging, HTTPS, managed identity, and private access to a backend database.

A strong solution is:

- Create a **Linux Standard** App Service plan.
- Create the web app on that plan using a supported .NET runtime or container.
- Enable **system-assigned managed identity**.
- Use **VNet Integration** for outbound access to the private database.
- Add a **deployment slot** for staging.
- Enable **Application Insights** for telemetry.
- Turn on **HTTPS-only** and bind a custom domain if required.

Why this works:

- Standard tier gives you deployment slots and production features.
- Linux supports modern cross-platform runtimes.
- Managed identity removes secret storage for Azure resource access.
- VNet Integration solves outbound private connectivity.
- Application Insights helps you validate the app after deployment.

**Common troubleshooting points**

If creation or deployment fails, check these first:

- The app OS matches the plan OS.
- The runtime stack is supported in the selected region and CLI version.
- The plan tier supports the feature you are trying to use.
- The app name is globally unique.
- DNS and certificate records are correct for custom domains.
- The app has the right identity and permissions for downstream services.
- Networking rules allow the traffic path you expect.

If the app starts slowly or appears idle:

- Verify **Always On** is enabled on a supported tier.
- Check startup logs and Application Insights traces.
- Confirm the app is not waiting on external dependencies.
- Review container health probes if you use custom containers.

**AZ-204 decision cues**

When you see a question, map the requirement to the feature:

- Need a cheap dev environment → **Free/Shared** if no custom domain or production features are needed.
- Need custom domain and HTTPS → **Basic or higher**.
- Need staging slots → **Standard or higher**.
- Need private outbound access to a database → **VNet Integration**.
- Need private inbound access to the app → **Private Endpoint**.
- Need secretless access to Azure services → **Managed identity**.
- Need safe releases → **Deployment slots**.
- Need reduced cold starts → **Always On** on a supported tier.
