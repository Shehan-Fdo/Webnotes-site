## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Managed identity fundamentals**

A **managed identity** is an identity in Microsoft Entra ID that Azure creates and manages for an application or resource. You use it to authenticate to Azure services without storing secrets, certificates, or connection strings in code or configuration.

Managed identities are especially useful when your app needs to access:

- **Azure Key Vault** for secrets, keys, and certificates
- **Azure Storage** for blobs, queues, or files
- **Azure SQL** or other backend services that support Microsoft Entra authentication
- **Custom APIs** protected by Microsoft Entra ID

The main benefit is operational security: Azure handles credential rotation and lifecycle, while your code requests tokens at runtime.

**Managed identity types**

Azure supports two common forms:

- **System-assigned managed identity**
  - Enabled directly on a single Azure resource.
  - Lifecycle is tied to that resource.
  - Deleted automatically when the resource is deleted.
  - Best when one app instance needs one identity.

- **User-assigned managed identity**
  - Created as a standalone Azure resource.
  - Can be attached to one or more Azure resources.
  - Lifecycle is independent of the host resource.
  - Best when multiple apps share the same identity or you need stable identity reuse.

A practical rule: use system-assigned identity for simple app-to-service access, and user-assigned identity when you need portability, reuse, or separation from the host resource lifecycle.

**How token-based access works**

Your application does not call Key Vault or a backend service with a password. Instead, it requests an access token from the Azure Instance Metadata Service (IMDS) or the platform identity endpoint, then presents that token to the target service.

Typical flow:

1. Your app asks the managed identity endpoint for a token.
2. Azure validates the resource and identity context.
3. Azure returns an OAuth 2.0 access token for the requested audience.
4. Your app sends the token in the `Authorization: Bearer <token>` header.
5. The target service validates the token and authorizes the request.

This pattern removes secret management from your application code and supports least privilege through role assignments or access policies.

**Enabling a managed identity**

You usually enable a system-assigned identity in the Azure portal, ARM template, Bicep, Azure CLI, or Terraform. For Azure App Service, Azure Functions, Azure Container Apps, and many other services, the identity is a simple property on the resource.

Example Azure CLI commands:

```bash
# Enable system-assigned managed identity on an App Service
az webapp identity assign \
  --resource-group rg-app \
  --name my-webapp

# Show the principal and tenant details
az webapp identity show \
  --resource-group rg-app \
  --name my-webapp
```

For a user-assigned identity:

```bash
# Create a user-assigned managed identity
az identity create \
  --resource-group rg-app \
  --name uami-backend

# Attach it to a web app
az webapp identity assign \
  --resource-group rg-app \
  --name my-webapp \
  --identities uami-backend
```

When you enable identity, Azure creates a service principal representation in Microsoft Entra ID. You then grant that identity permissions to the target resource.

**Granting access to Azure Key Vault**

Key Vault access can be controlled in two main ways:

- **Key Vault access policies**
  - Legacy model for many vaults.
  - You explicitly grant permissions such as `Get`, `List`, `Set`, or `Delete`.
  - Simple and direct, but less flexible than Azure RBAC for some scenarios.

- **Azure role-based access control (Azure RBAC)**
  - Modern authorization model for Key Vault data plane in many deployments.
  - Uses built-in roles such as `Key Vault Secrets User` or `Key Vault Administrator`.
  - Better aligns with centralized governance and least privilege.

> Be careful to distinguish between the **management plane** and the **data plane**. Creating a vault is a management-plane action; reading a secret is a data-plane action. Your identity may be allowed to manage the vault but still be blocked from reading secrets.

Example with Azure RBAC:

```bash
# Assign permission to read secrets from a vault
az role assignment create \
  --assignee-object-id <managed-identity-object-id> \
  --assignee-principal-type ServicePrincipal \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.KeyVault/vaults/<vault-name>
```

Example with access policies:

```bash
# Grant secret read permissions using an access policy
az keyvault set-policy \
  --name my-vault \
  --object-id <managed-identity-object-id> \
  --secret-permissions get list
```

**Using Key Vault from code**

Most Azure SDKs support `DefaultAzureCredential`, which tries multiple authentication methods and uses managed identity automatically in Azure.

Example in .NET:

```csharp
using Azure.Identity;
using Azure.Security.KeyVault.Secrets;

var vaultUri = new Uri("https://my-vault.vault.azure.net/");
var client = new SecretClient(vaultUri, new DefaultAzureCredential());

KeyVaultSecret secret = await client.GetSecretAsync("DbPassword");
Console.WriteLine(secret.Value);
```

Why this works:

- In Azure, `DefaultAzureCredential` can use the managed identity assigned to the host.
- Locally, it can fall back to developer credentials such as Azure CLI sign-in or Visual Studio sign-in.
- You avoid hardcoding credentials in source code.

Example in Python:

```python
from azure.identity import DefaultAzureCredential
from azure.keyvault.secrets import SecretClient

vault_url = "https://my-vault.vault.azure.net/"
credential = DefaultAzureCredential()
client = SecretClient(vault_url=vault_url, credential=credential)

secret = client.get_secret("DbPassword")
print(secret.value)
```

**Backend service integration patterns**

Managed identity is not limited to Key Vault. You can use it to call backend services that trust Microsoft Entra ID, including custom APIs and some Azure platform services.

Common patterns include:

- **App to API**
  - Your app acquires a token for the API audience.
  - The API validates the token and authorizes the caller.
  - Useful for microservices and internal service-to-service communication.

- **App to Azure SQL**
  - Your app authenticates with Microsoft Entra ID instead of SQL authentication.
  - You map the managed identity to a database user and grant database roles.
  - This removes SQL passwords from configuration.

- **App to Storage**
  - Your app uses Azure SDK authentication with managed identity.
  - You grant data-plane roles such as `Storage Blob Data Reader` or `Storage Blob Data Contributor`.

**Calling a protected backend API**

If your backend API is protected by Microsoft Entra ID, your app requests a token for the API's application ID URI or scope.

Example in .NET using `Azure.Identity` and `Azure.Core`:

```csharp
using Azure.Core;
using Azure.Identity;
using System.Net.Http.Headers;

var credential = new DefaultAzureCredential();
var tokenRequestContext = new TokenRequestContext(new[] { "api://<api-app-id>/.default" });

AccessToken token = await credential.GetTokenAsync(tokenRequestContext);

using var httpClient = new HttpClient();
httpClient.DefaultRequestHeaders.Authorization =
    new AuthenticationHeaderValue("Bearer", token.Token);

var response = await httpClient.GetAsync("https://my-api.contoso.com/weather");
response.EnsureSuccessStatusCode();
```

Key points:

- The scope `/.default` asks for the static permissions configured for the app.
- The API must validate issuer, audience, and token lifetime.
- The app registration for the API must expose permissions or app roles as needed.

**Using managed identity with Azure SQL**

For Azure SQL Database, managed identity can replace SQL logins in many app scenarios. The application authenticates with an access token, and the database maps the identity to a contained user.

Typical setup steps:

1. Enable managed identity on the app.
2. Configure Microsoft Entra admin on the SQL server if not already set.
3. Connect as an Entra admin and create a user for the managed identity.
4. Grant database roles or permissions.
5. Use token-based authentication in the app.

Example T-SQL:

```sql
CREATE USER [my-webapp] FROM EXTERNAL PROVIDER;
ALTER ROLE db_datareader ADD MEMBER [my-webapp];
ALTER ROLE db_datawriter ADD MEMBER [my-webapp];
```

Example .NET connection using an access token:

```csharp
using Azure.Core;
using Azure.Identity;
using Microsoft.Data.SqlClient;

var credential = new DefaultAzureCredential();
var token = await credential.GetTokenAsync(
    new TokenRequestContext(new[] { "https://database.windows.net/.default" }));

var connectionString = "Server=tcp:myserver.database.windows.net,1433;Database=mydb;Encrypt=True;";

using var connection = new SqlConnection(connectionString);
connection.AccessToken = token.Token;
await connection.OpenAsync();
```

**Local development versus Azure runtime**

Managed identity is an Azure-hosted feature, so it is not available on your local machine. During development, you typically use your developer identity through Azure CLI, Visual Studio, or environment variables.

Practical approach:

- Use `DefaultAzureCredential` in code.
- Sign in locally with `az login`.
- Deploy to Azure and let the same code use managed identity in production.

Example local setup:

```bash
az login
az account set --subscription "<subscription-id>"
```

This keeps your code path consistent across environments while changing only the active credential source.

**Configuration and deployment considerations**

Managed identity is usually paired with environment-based configuration:

- Store only non-secret values in app settings, such as vault URI or API base URL.
- Avoid storing client secrets for Azure resources that support managed identity.
- Use separate identities per environment when you need isolation.
- Prefer user-assigned identity when multiple apps need the same stable identity.

Example App Service app settings:

```bash
az webapp config appsettings set \
  --resource-group rg-app \
  --name my-webapp \
  --settings \
    KeyVaultUri=https://my-vault.vault.azure.net/ \
    BackendApiScope=api://<api-app-id>/.default
```

**Troubleshooting common issues**

Most failures come from authorization, token audience, or propagation delays.

- **403 Forbidden**
  - The identity authenticated successfully but lacks permission.
  - Check Key Vault access policy, Azure RBAC role assignment, or backend API authorization.

- **401 Unauthorized**
  - The token is missing, expired, or targeted at the wrong audience.
  - Verify the scope or resource URI used when requesting the token.

- **Identity not found**
  - The managed identity may not be enabled, or the role assignment has not propagated yet.
  - Wait a few minutes and retry after confirming the principal ID.

- **Works locally but fails in Azure**
  - Local auth may be using your developer account, while Azure runtime uses the managed identity.
  - Confirm the Azure resource identity is enabled and granted access.

Useful diagnostic commands:

```bash
# Inspect the identity on a web app
az webapp identity show \
  --resource-group rg-app \
  --name my-webapp

# Check role assignments for the managed identity
az role assignment list \
  --assignee <managed-identity-object-id> \
  --all -o table

# Read Key Vault access policies
az keyvault show \
  --name my-vault \
  --query properties.accessPolicies
```

**Security best practices**

Managed identity improves security, but you still need disciplined authorization.

- Grant the minimum required role or permission.
- Use separate identities for separate workloads when practical.
- Prefer data-plane roles over broad administrative roles.
- Rotate away from secrets and connection strings where managed identity is supported.
- Audit access regularly through Azure Activity Log, Key Vault logs, and service diagnostics.

A good design principle is to treat managed identity as the authentication mechanism and Azure authorization as the control plane for what the identity can do.

**Exam-focused takeaways**

For AZ-204, expect questions that test whether you can choose the right identity type and configure access correctly.

Remember these distinctions:

- **System-assigned identity** is tied to one resource.
- **User-assigned identity** is reusable across resources.
- **Key Vault** access may use access policies or Azure RBAC.
- **DefaultAzureCredential** is the common SDK choice for Azure-hosted and local development scenarios.
- **Backend services** often require token acquisition for a specific audience or scope.

If a scenario says “no secrets in code,” “automatic credential rotation,” or “app needs to access Key Vault securely,” managed identity is usually the intended solution.
