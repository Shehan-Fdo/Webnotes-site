## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Overview**

Azure applications usually fail in predictable ways: missing telemetry, noisy logs, incorrect scale settings, or unexpected restarts after configuration changes. For AZ-204, you need to know how to enable diagnostics, route logs, interpret telemetry, and choose the right scaling model for App Service and Azure Functions.

The exam often tests practical decisions:
- Where logs go and how you query them.
- Which scaling option applies to a given hosting plan.
- What causes restarts or recycles.
- How to validate health after deployment or configuration changes.

**Logging and telemetry fundamentals**

Logging is the first layer of observability. In Azure, you typically combine platform logs, application logs, and telemetry from Application Insights to understand both infrastructure behavior and app behavior.

Common telemetry sources:
- **Application logs** from your code, such as `ILogger` in ASP.NET Core or `logging` in Python.
- **Platform logs** from App Service, Functions, or container hosts.
- **Metrics** such as CPU, memory, requests, and response time.
- **Distributed traces** that correlate calls across services.

A useful mental model is:
- Logs answer “what happened?”
- Metrics answer “how much or how often?”
- Traces answer “where did the request go?”

**Application Insights**

Application Insights is the primary application performance monitoring service for Azure applications. It helps you detect failures, measure performance, and trace dependencies.

Key features to know:
- **Request telemetry** records incoming requests, duration, and success/failure.
- **Dependency telemetry** tracks outbound calls to SQL, HTTP APIs, queues, and other services.
- **Exceptions** capture unhandled and handled errors.
- **Availability tests** probe an endpoint from multiple regions to verify uptime.
- **Live Metrics Stream** shows near-real-time request rate, failures, and server performance.
- **Distributed transaction correlation** uses operation IDs and trace context to connect related telemetry across services.

For modern .NET apps, Application Insights often uses OpenTelemetry-style correlation concepts under the hood, but exam questions usually focus on the outcome: you can follow one request across multiple components.

Example ASP.NET Core setup:
```csharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddApplicationInsightsTelemetry();

var app = builder.Build();

app.MapGet("/", () => "Hello from Azure");
app.Run();
```

If you use structured logging, include fields that help you query later:
```csharp
logger.LogInformation("Order {OrderId} processed for {CustomerId}", orderId, customerId);
```

**Azure App Service diagnostics**

App Service provides platform diagnostics that help you inspect app behavior without changing code. You can enable and route logs to different destinations depending on retention and analysis needs.

Common App Service log types:
- **Application logging**
- **Web server logging**
- **Detailed error messages**
- **Failed request tracing**
- **Container logs** for Linux containers or custom containers

You can route diagnostic data to:
- **Log Analytics workspace** for Kusto Query Language (KQL) analysis.
- **Storage account** for lower-cost retention and archival.
- **Event Hub** for streaming to external systems or SIEM tools.

A typical diagnostic settings flow:
1. Open the resource in Azure Portal.
2. Select **Diagnostic settings**.
3. Add a setting and choose the log categories and metrics.
4. Send data to one or more destinations.
5. Validate ingestion in the target workspace or storage account.

Example Azure CLI for diagnostic settings:
```bash
az monitor diagnostic-settings create \
  --name appservice-diag \
  --resource /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Web/sites/<appName> \
  --workspace /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.OperationalInsights/workspaces/<lawName> \
  --logs '[{"category":"AppServiceHTTPLogs","enabled":true},{"category":"AppServiceConsoleLogs","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

> Diagnostic category names vary by resource type and platform. Verify the available categories for the specific App Service or Function App before memorizing a command.

**App Service logging commands**

For App Service, the Azure CLI can enable application logging and other log types. Syntax can vary slightly by CLI version, so focus on the intent rather than one exact form.

Example:
```bash
az webapp log config \
  --name <appName> \
  --resource-group <rg> \
  --application-logging true \
  --web-server-logging filesystem \
  --level information
```

Useful follow-up commands:
```bash
az webapp log tail --name <appName> --resource-group <rg>
az webapp log download --name <appName> --resource-group <rg>
```

Use these when you need quick troubleshooting during deployment or when Application Insights is not yet fully configured.

**Kusto Query Language for troubleshooting**

Kusto Query Language, or **KQL**, is the query language used in Log Analytics and Application Insights. It is a common exam skill because it helps you filter, aggregate, and correlate telemetry.

Example queries:
```kusto
AppRequests
| where TimeGenerated > ago(1h)
| where Success == false
| project TimeGenerated, Name, ResultCode, DurationMs, OperationId
| order by TimeGenerated desc
```

```kusto
AppDependencies
| where TimeGenerated > ago(1h)
| summarize count() by Type, Success
| order by count_ desc
```

```kusto
AppTraces
| where Message contains "timeout"
| project TimeGenerated, Message, SeverityLevel, OperationId
```

When troubleshooting, look for:
- Spikes in failed requests.
- Slow dependencies.
- Repeated exceptions with the same operation ID.
- Correlation gaps that suggest missing instrumentation.

**Scaling concepts**

Scaling means adjusting capacity to match demand. In Azure, scaling can be:
- **Vertical scaling**: move to a larger SKU or instance size.
- **Horizontal scaling**: add more instances.
- **Automatic scaling**: Azure adjusts capacity based on rules or platform demand.

For AZ-204, you should distinguish between service-specific scaling behavior and hosting-plan limits.

**App Service scaling**

App Service supports scaling in two main ways:
- **Scale up** by changing the App Service plan SKU.
- **Scale out** by increasing the number of instances.

Important points:
- Autoscale is available for supported App Service plans, typically Standard and above.
- Free and Shared tiers do not support autoscale.
- Scaling out is useful for stateless workloads.
- Session state should be externalized if you want multiple instances to serve traffic reliably.

Example CLI to scale out manually:
```bash
az appservice plan update \
  --name <planName> \
  --resource-group <rg> \
  --number-of-workers 3
```

Example CLI to scale up:
```bash
az appservice plan update \
  --name <planName> \
  --resource-group <rg> \
  --sku P1v3
```

Autoscale is usually configured on the App Service plan, not directly on the web app. A common rule uses CPU or memory thresholds.

Example autoscale rule concept:
- Scale out when CPU exceeds 70% for 10 minutes.
- Scale in when CPU drops below 40% for 10 minutes.

Example ARM-style autoscale resource structure:
```json
{
  "type": "Microsoft.Insights/autoscaleSettings",
  "apiVersion": "2022-10-01",
  "name": "appservice-autoscale",
  "location": "global",
  "properties": {
    "targetResourceUri": "/subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Web/serverfarms/<planName>",
    "enabled": true,
    "profiles": [
      {
        "name": "defaultProfile",
        "capacity": {
          "minimum": "1",
          "maximum": "5",
          "default": "2"
        },
        "rules": [
          {
            "metricTrigger": {
              "metricName": "CpuPercentage",
              "metricResourceUri": "/subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Web/serverfarms/<planName>",
              "operator": "GreaterThan",
              "threshold": 70,
              "timeAggregation": "Average",
              "statistic": "Average",
              "timeWindow": "PT10M",
              "timeGrain": "PT1M"
            },
            "scaleAction": {
              "direction": "Increase",
              "type": "ChangeCount",
              "value": "1",
              "cooldown": "PT5M"
            }
          }
        ]
      }
    ]
  }
}
```

Use this as a conceptual pattern: the exam is more likely to test where autoscale is configured and what triggers it than the exact JSON syntax.

**Azure Functions scaling**

Azure Functions scaling depends heavily on the hosting plan.

| Hosting plan | Scaling behavior | Notes |
|---|---|---|
| Consumption | Automatic event-driven scale | Best for intermittent workloads; cold starts can occur |
| Premium | Automatic scale with pre-warmed instances | Better for low latency and VNET integration scenarios |
| Dedicated/App Service | Uses App Service plan scaling | You manage scale like a web app |

Key exam distinctions:
- **Consumption plan** scales based on triggers and demand.
- **Premium plan** reduces cold start impact and supports more advanced networking and performance needs.
- **Dedicated/App Service plan** does not scale like Consumption; it follows App Service plan rules.

Functions scale is also trigger-aware:
- Queue and Event Hub triggers can scale based on backlog or event rate.
- HTTP-triggered functions scale based on incoming request load.
- Timer-triggered functions do not scale out in the same way because they are schedule-driven.

Example `host.json` settings that affect runtime behavior:
```json
{
  "version": "2.0",
  "logging": {
    "applicationInsights": {
      "samplingSettings": {
        "isEnabled": true,
        "maxTelemetryItemsPerSecond": 5
      }
    }
  }
}
```

**Scale limits and constraints**

Not every service or plan supports the same scaling model. You should know the practical constraints.

Common constraints:
- Autoscale requires a supported SKU or hosting plan.
- Some services scale only by instance count, not by CPU-based rules.
- Scale-out limits depend on region, SKU, and service quotas.
- Premium and Dedicated plans may have minimum and maximum instance counts.
- Functions on Consumption can scale aggressively, but execution limits and timeout behavior still apply.

When answering exam questions, check:
- Is the workload stateless?
- Is low latency required?
- Is the service on a plan that supports autoscale?
- Is there a quota or instance cap?

**Restart and recycle behavior**

Configuration changes can trigger restarts or recycles. In App Service and Functions, many configuration changes cause the app to restart so the new settings take effect, but the exact behavior depends on the setting and platform.

Common restart triggers:
- Changing application settings or connection strings.
- Updating runtime stack or platform configuration.
- Deploying new code.
- Modifying certain site configuration values.
- Swapping deployment slots.

Be careful with wording:
- Many configuration changes can trigger a restart or recycle.
- Some changes apply without a full restart, depending on the setting.

**Deployment slots and warm-up**

Deployment slots help you deploy safely by staging changes before production traffic sees them.

Important slot behaviors:
- A slot swap can warm up the target slot before traffic is switched.
- **Auto-swap** can automatically promote a slot after deployment.
- Swap settings can preserve selected configuration values.
- Warm-up helps reduce cold-start-like effects during release.

Useful slot settings:
- **Swap with preview** for validation before completing the swap.
- **Application initialization** to pre-load the app.
- **Health check** to verify the app is ready before routing traffic.

Example warm-up pattern:
1. Deploy to staging.
2. Run smoke tests against the staging slot.
3. Confirm health endpoint and telemetry.
4. Swap into production.
5. Monitor logs and metrics after the swap.

**Health checks and readiness**

Health checks are not just endpoints in your code. You must configure the platform to probe the path you expose.

Example ASP.NET Core endpoint:
```csharp
app.MapHealthChecks("/health");
```

Azure App Service health check configuration:
- Set the Health check path to `/health`.
- Ensure the endpoint returns a healthy response quickly.
- Avoid expensive dependencies in the health probe if they make the app appear unhealthy too easily.

For containerized apps, health behavior may also involve:
- **Liveness probes** to detect when a container should be restarted.
- **Readiness probes** to determine whether the container should receive traffic.

These concepts matter because a container can be running but not ready to serve requests.

> A health endpoint alone does not make Azure use it. You must configure the App Service or container platform to probe that endpoint.

**Restart behavior in containers**

Containerized workloads add another layer of restart logic. Azure may restart containers when:
- The container exits unexpectedly.
- Health probes fail repeatedly.
- The image or configuration changes.
- The platform recycles the host or redeploys the app.

Practical guidance:
- Keep startup fast and deterministic.
- Log startup failures clearly.
- Expose a lightweight health endpoint.
- Separate readiness checks from deep dependency checks when possible.

Example Docker health check:
```dockerfile
HEALTHCHECK --interval=30s --timeout=5s --start-period=20s --retries=3 \
  CMD curl -f http://localhost:8080/health || exit 1
```

**Practical diagnostics workflow**

A good troubleshooting workflow combines logs, metrics, and scaling signals.

1. **Confirm the symptom**
   - Is the issue latency, errors, or unavailability?
   - Is it isolated to one instance or all instances?

2. **Check platform metrics**
   - CPU, memory, request count, response time, and HTTP 5xx rates.
   - Instance count and recent scale events.

3. **Inspect logs**
   - App Service logs, container logs, and application traces.
   - Use Log Analytics or Application Insights queries.

4. **Correlate with deployments or config changes**
   - Look for slot swaps, app setting changes, or runtime upgrades.
   - Check whether a restart occurred around the same time.

5. **Validate recovery**
   - Re-run the health check.
   - Confirm telemetry resumes.
   - Verify the app scales back to expected capacity.

Example end-to-end KQL investigation:
```kusto
AppRequests
| where TimeGenerated > ago(24h)
| summarize Requests=count(), Failures=countif(Success == false), AvgDuration=avg(DurationMs) by bin(TimeGenerated, 15m)
| order by TimeGenerated desc
```

**End-to-end scenario**

Suppose you deploy an ASP.NET Core API to App Service and notice intermittent 502 errors after a configuration update.

What you do:
- Enable Application Insights and App Service logs.
- Route diagnostics to Log Analytics for retention and KQL analysis.
- Configure a health check path such as `/health`.
- Use deployment slots to test changes before production swap.
- Review whether the configuration change triggered a restart.
- Check whether the app is under-provisioned and needs scale-out.
- Validate that the app remains healthy after the swap and that telemetry shows stable request success.

A practical command sequence might look like this:
```bash
az monitor diagnostic-settings create \
  --name api-diag \
  --resource /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Web/sites/<appName> \
  --workspace /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.OperationalInsights/workspaces/<lawName> \
  --metrics '[{"category":"AllMetrics","enabled":true}]'

az webapp log config \
  --name <appName> \
  --resource-group <rg> \
  --application-logging true \
  --web-server-logging filesystem \
  --level information
```

Then validate:
- The health endpoint returns success.
- Application Insights shows requests and dependencies.
- Log Analytics receives platform logs.
- Autoscale or manual scale settings match demand.
- No unexpected restarts occur after the configuration change.

**Exam-focused distinctions**

- **Application Insights** is for application telemetry and correlation.
- **Diagnostic settings** route platform logs and metrics to destinations.
- **Autoscale** adjusts capacity based on rules or platform demand.
- **Restart/recycle behavior** often follows configuration changes, deployments, or slot swaps.
- **Health checks** require both an endpoint and platform configuration.
- **Functions scaling** depends on the hosting plan, not just the trigger type.

**Quick reference**

- Use **Application Insights** for request, dependency, exception, and trace analysis.
- Use **Diagnostic settings** to send logs and metrics to Log Analytics, Storage, or Event Hub.
- Use **KQL** to query telemetry and troubleshoot failures.
- Use **App Service autoscale** on supported plans for horizontal scaling.
- Use **Functions Consumption** for event-driven scale and **Premium** for reduced cold starts.
- Use **deployment slots** and warm-up to reduce deployment risk.
- Use **health checks** and container probes to detect unhealthy instances early.
