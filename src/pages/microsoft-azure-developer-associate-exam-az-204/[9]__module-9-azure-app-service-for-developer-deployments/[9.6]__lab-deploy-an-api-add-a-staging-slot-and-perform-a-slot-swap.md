## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Overview**

Azure App Service deployment slots let you deploy a new version of an API to a non-production environment, validate it, and then swap it into production with minimal downtime. This pattern is common for blue-green style releases, rollback-safe deployments, and production incident mitigation when you need to promote a known-good build quickly.

A slot swap changes which slot content is serving a hostname; it does not reassign the hostname itself. The production hostname remains bound to the production slot, and the staging hostname remains bound to the staging slot. After the swap, the content and some configuration move between slots according to slot-swap rules.

**Why this matters for AZ-204**

Deployment slots are a practical App Service feature that tests your understanding of:

- Safe release workflows
- Slot-specific configuration
- Warm-up and health validation
- Swap behavior and rollback
- CLI-based management

In real projects, you use slots to reduce risk when deploying an API that backs a mobile app, public website, or partner integration. If the new build fails in staging, you can fix it before users see it. If production has a problem after swap, you can swap back.

**Typical release flow**

1. Deploy the API to the production app or directly to a staging slot.
2. Configure slot-specific settings that must not move during swap.
3. Validate the API in staging using the staging hostname.
4. Swap staging into production.
5. Monitor logs, metrics, and health endpoints.
6. Swap back if the release causes issues.

**App Service slot concepts**

Deployment slots are separate app instances under the same App Service plan and app name. They share compute resources, but each slot has its own hostname and can have its own configuration.

- **Production slot** serves the public hostname.
- **Staging slot** is used for validation before promotion.
- **Slot settings** stay with the slot during a swap.
- **Swappable settings** move with the content during a swap.

A slot is useful when you want to test a release under production-like conditions without exposing it to all users.

**What swaps and what stays sticky**

| Item | Swaps with content? | Notes |
|---|---:|---|
| Application code/content | Yes | The deployed package or app files move between slots. |
| Hostname binding | No | Each hostname stays attached to its original slot. |
| Slot-specific app settings | No | Mark these as slot settings so they remain in place. |
| Connection strings marked sticky | No | Keep environment-specific database values in the slot. |
| TLS/SSL bindings | Usually no | Bindings are slot-specific unless explicitly managed otherwise. |
| Deployment history | No | Each slot keeps its own deployment history. |

**Deploying the API**

You can deploy an API to App Service using ZIP deploy, GitHub Actions, Azure DevOps, or `az webapp deployment source config-zip`. For lab work, CLI deployment is often the fastest way to reproduce the workflow.

Example: create a web app for an API and deploy a package.

```bash
az group create \
  --name rg-api-lab \
  --location eastus

az appservice plan create \
  --resource-group rg-api-lab \
  --name asp-api-lab \
  --sku B1 \
  --is-linux

az webapp create \
  --resource-group rg-api-lab \
  --plan asp-api-lab \
  --name my-api-lab-12345 \
  --runtime "DOTNETCORE:8.0"
```

> The exact runtime string must match current Azure CLI and region support. If the runtime name is rejected, check `az webapp list-runtimes` or the current App Service runtime documentation for your stack.

Deploy a ZIP package:

```bash
az webapp deployment source config-zip \
  --resource-group rg-api-lab \
  --name my-api-lab-12345 \
  --src ./publish/api.zip
```

Common outcomes:

- A successful ZIP deploy returns a deployment status and the app restarts.
- A failed deploy often indicates a bad package path, unsupported runtime, or startup failure.
- If the app starts but returns 500 errors, inspect logs and the startup command.

**Creating a staging slot**

A staging slot is created under the same web app. It gets its own hostname, such as `my-api-lab-12345-staging.azurewebsites.net`.

```bash
az webapp deployment slot create \
  --resource-group rg-api-lab \
  --name my-api-lab-12345 \
  --slot staging
```

After creation, you can deploy a different build to staging or clone the production configuration depending on your workflow.

Useful checks:

```bash
az webapp deployment slot list \
  --resource-group rg-api-lab \
  --name my-api-lab-12345
```

Expected result:

- You should see both `production` and `staging`.
- The staging slot has its own hostname and deployment target.
- The slot shares the same App Service plan, so it consumes shared compute.

**Configuring slot-specific settings**

Use slot settings for values that must remain in the slot after a swap, such as:

- Database connection strings
- API keys for test systems
- Feature flags for validation
- Environment names
- Diagnostic endpoints

For app settings, the common pattern is to set the value and then mark it as sticky using the slot-setting workflow supported by the current CLI version or portal. If your CLI version does not support a direct sticky flag in the same command, use the portal or the slot settings configuration path.

Example app setting:

```bash
az webapp config appsettings set \
  --resource-group rg-api-lab \
  --name my-api-lab-12345 \
  --slot staging \
  --settings ASPNETCORE_ENVIRONMENT=Staging
```

For connection strings, set the value and specify the type explicitly:

```bash
az webapp config connection-string set \
  --resource-group rg-api-lab \
  --name my-api-lab-12345 \
  --slot staging \
  --connection-string-type SQLAzure \
  --settings MyDb="Server=tcp:sqlserver.database.windows.net,1433;Database=appdb;User ID=appuser;Password=Secret123!"
```

Then verify which settings are sticky in the portal or by reviewing the app configuration for the slot. The key exam idea is that slot-specific values do not swap.

> Do not assume every setting is sticky by default. If you forget to mark environment-specific values as slot settings, a swap can move the wrong database or API endpoint into production.

**Validating the staging slot**

Before swapping, test the staging hostname directly. This is where you catch startup errors, broken dependencies, and configuration mismatches.

```bash
curl -i https://my-api-lab-12345-staging.azurewebsites.net/health
```

A healthy API often returns:

- `200 OK`
- A JSON health payload
- A response time similar to production expectations

Example health endpoint:

```json
{
  "status": "Healthy",
  "service": "orders-api",
  "version": "1.4.0"
}
```

If the staging slot fails validation, fix the deployment before swapping. Common issues include:

- Missing environment variables
- Wrong connection string type
- Incorrect startup command
- Package mismatch between build and runtime
- Authentication or authorization misconfiguration

**Performing the slot swap**

A swap promotes staging content into production while preserving slot identity. The production hostname continues to point to the production slot, but after the swap it serves the content that was previously in staging.

```bash
az webapp deployment slot swap \
  --resource-group rg-api-lab \
  --name my-api-lab-12345 \
  --slot staging \
  --target-slot production
```

Expected behavior:

- The staging build becomes live on the production hostname.
- The previous production build moves to the staging slot.
- Sticky settings remain with their original slot.
- Non-sticky settings move with the swapped content.

This is the core blue-green release pattern for App Service.

**Swap behavior in practice**

Imagine production is serving version `1.3.0` and staging has version `1.4.0`.

Before swap:

- Production hostname serves `1.3.0`
- Staging hostname serves `1.4.0`

After swap:

- Production hostname serves `1.4.0`
- Staging hostname serves `1.3.0`

The hostnames do not change owners. The content behind them changes.

**Rollback strategy**

If the new release causes errors, swap back. This is one of the biggest operational benefits of slots.

```bash
az webapp deployment slot swap \
  --resource-group rg-api-lab \
  --name my-api-lab-12345 \
  --slot staging \
  --target-slot production
```

Because the previous production version is now in staging, swapping again restores it. This gives you a fast rollback path without redeploying the old package.

**Warm-up and swap readiness**

App Service can warm up the target slot before completing the swap. Warm-up reduces the chance that users hit a cold start or partially initialized app.

Use warm-up when your API:

- Loads large dependency graphs
- Builds caches on startup
- Connects to external services
- Runs migrations or initialization checks

A common production pattern is to expose a health endpoint and configure the app to report ready only when dependencies are available.

Example startup readiness check:

```csharp
app.MapGet("/health", () => Results.Ok(new { status = "Healthy" }));
```

If the app needs more advanced readiness logic, ensure the endpoint checks the real dependencies you care about, such as SQL Database or Key Vault access.

**Practical release scenario**

A team ships a new orders API version on Friday afternoon. The build passes tests, but the team wants to reduce risk because the release includes a new payment integration.

They:

- Deploy the new build to staging
- Verify `/health`, `/swagger`, and a test order flow
- Confirm the staging slot uses the test payment endpoint
- Swap staging into production during a low-traffic window
- Monitor logs and metrics for errors
- Swap back immediately if payment failures appear

This is a realistic use case for deployment slots in a blue-green deployment strategy.

**CLI workflow summary**

1. Create the App Service plan and web app.
2. Deploy the API to the app or staging slot.
3. Create a staging slot.
4. Configure sticky settings for environment-specific values.
5. Validate the staging hostname.
6. Swap staging into production.
7. Monitor and roll back if needed.

**Troubleshooting notes**

If the swap does not behave as expected, check the following:

- The app is running in a supported App Service plan tier for slots.
- The staging slot exists and is healthy.
- The deployment package is valid and starts correctly.
- Connection strings and app settings are marked correctly as slot-specific.
- The app does not rely on local machine state or ephemeral files.
- The health endpoint returns success before swap.

Common failure symptoms:

- `404` on the staging hostname: wrong deployment target or app not started
- `500` after swap: missing production configuration or startup failure
- Unexpected database changes: connection string was not sticky
- Wrong environment behavior: app setting swapped when it should have stayed in the slot

**Exam focus**

For AZ-204, remember these points:

- Deployment slots support safer releases with minimal downtime.
- A swap changes content routing, not hostname ownership.
- Slot settings stay with the slot during a swap.
- Validate staging before promotion.
- Rollback is usually another swap.
- Use CLI, portal, or CI/CD pipelines to manage the workflow.

**Quick reference**

- `az webapp deployment slot create` creates the staging slot.
- `az webapp deployment slot swap` promotes or rolls back content.
- `az webapp config appsettings set` configures app settings.
- `az webapp config connection-string set` configures connection strings.
- Health checks and warm-up reduce swap risk.
