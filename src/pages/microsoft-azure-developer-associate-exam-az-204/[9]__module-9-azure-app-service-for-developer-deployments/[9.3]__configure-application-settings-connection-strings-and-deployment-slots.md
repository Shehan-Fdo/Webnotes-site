## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Azure App Service configuration overview**

Azure App Service uses a centralized configuration model for app settings, connection strings, and slot-specific values. These settings let you change behavior without redeploying code, which is essential for environment-specific values such as API endpoints, feature flags, and database connections.

App Service configuration is commonly managed in the Azure Portal, Azure CLI, ARM templates, Bicep, and infrastructure-as-code pipelines. For AZ-204, focus on where values are stored, how they are exposed to code, and what changes during a slot swap.

- **App settings** are name/value pairs for runtime configuration.
- **Connection strings** are specialized settings for data stores and are handled differently by App Service.
- **Deployment slots** let you stage and validate changes before promoting them to production.

**Application settings and how they surface in code**

App settings are stored in the App Service configuration and are injected into the app at runtime. In many runtimes, they appear as environment variables. In .NET and ASP.NET Core, App Service also maps them into the configuration system, so you can read them through `IConfiguration`.

Use app settings for values that are not secrets or that are managed as platform configuration. For secrets, prefer Azure Key Vault references when possible.

- **Name** should be stable and descriptive, such as `ApiBaseUrl` or `FeatureFlags__Beta`.
- **Value** should be environment-specific and not hardcoded in source.
- **Slot setting** can pin a value to a specific deployment slot.

Common runtime access patterns:

```csharp
// ASP.NET Core
var apiBaseUrl = builder.Configuration["ApiBaseUrl"];
var featureFlag = builder.Configuration["FeatureFlags:Beta"];
```

```javascript
// Node.js
const apiBaseUrl = process.env.ApiBaseUrl;
```

```python
# Python
import os
api_base_url = os.environ["ApiBaseUrl"]
```

In App Service, the platform exposes app settings as environment variables. In .NET, the configuration provider also maps them into `ConnectionStrings:` and normal key/value configuration patterns when appropriate. In other runtimes, you typically read the environment variable directly.

**Connection strings in App Service**

Connection strings are intended for database and service connection values. App Service stores them separately from general app settings and can expose them to the app in runtime-specific ways.

For exam purposes, remember two ideas:

- App Service treats connection strings as a distinct configuration category.
- The exact consumption pattern depends on the runtime and framework.

In .NET and ASP.NET Core, connection strings are commonly available through the configuration system under `ConnectionStrings:<name>`.

```csharp
var conn = builder.Configuration.GetConnectionString("DefaultConnection");
```

For other runtimes, App Service commonly exposes connection strings as environment variables with type-based prefixes. Examples include:

- `SQLCONNSTR_<name>`
- `SQLAZURECONNSTR_<name>`
- `MYSQLCONNSTR_<name>`
- `POSTGRESQLCONNSTR_<name>`
- `CUSTOMCONNSTR_<name>`

A plain Node.js or Python app does not automatically translate those prefixed variables into a friendly connection string name unless your code or framework does that mapping.

```javascript
// Example: read the prefixed variable directly if needed
const sqlConn = process.env.SQLCONNSTR_DefaultConnection;
```

```python
import os
sql_conn = os.environ.get("SQLCONNSTR_DefaultConnection")
```

> Do not assume every runtime reads `DefaultConnection` directly from `process.env` or `os.environ`. In App Service, the platform’s prefix-based exposure is common for connection strings, while framework mapping behavior varies.

**Where to configure settings in the Azure Portal**

The Azure Portal is the most visible way to manage App Service configuration during development and operations.

Use the Portal when you need to inspect or edit values quickly:

1. Open your App Service resource.
2. Select **Settings** > **Environment variables** or **Configuration**.
3. Add or edit **Application settings**.
4. Add or edit **Connection strings**.
5. Mark a setting as a **deployment slot setting** if it must stay with a slot.
6. Save changes and restart if prompted.

Portal configuration is useful for quick changes, but production teams usually prefer infrastructure-as-code or CI/CD pipelines to keep configuration repeatable.

**Azure CLI configuration**

Azure CLI is useful for automation and scripting. It is commonly used in deployment pipelines and local admin workflows.

Set an app setting:

```bash
az webapp config appsettings set \
  --resource-group rg-app \
  --name myapp \
  --settings ApiBaseUrl=https://api.contoso.com FeatureFlags__Beta=true
```

Set a connection string:

```bash
az webapp config connection-string set \
  --resource-group rg-app \
  --name myapp \
  --connection-string-type SQLAzure \
  --settings DefaultConnection="Server=tcp:myserver.database.windows.net,1433;Database=mydb;User ID=myuser;Password=MyPassword;Encrypt=True;"
```

List current settings:

```bash
az webapp config appsettings list \
  --resource-group rg-app \
  --name myapp
```

```bash
az webapp config connection-string list \
  --resource-group rg-app \
  --name myapp
```

Use the correct connection string type for the target service:

- `SQLAzure`
- `SQLServer`
- `MySql`
- `PostgreSQL`
- `Custom`

**ARM templates and Bicep**

Infrastructure-as-code is the preferred way to define repeatable configuration. In ARM templates and Bicep, app settings and connection strings are usually declared on the `Microsoft.Web/sites` resource.

A typical Bicep pattern looks like this:

```bicep
resource app 'Microsoft.Web/sites@2022-09-01' = {
  name: appName
  location: location
  kind: 'app'
  properties: {
    serverFarmId: plan.id
    siteConfig: {
      appSettings: [
        {
          name: 'ApiBaseUrl'
          value: 'https://api.contoso.com'
        }
        {
          name: 'FeatureFlags__Beta'
          value: 'true'
        }
      ]
      connectionStrings: [
        {
          name: 'DefaultConnection'
          value: 'Server=tcp:myserver.database.windows.net,1433;Database=mydb;User ID=myuser;Password=MyPassword;Encrypt=True;'
          type: 'SQLAzure'
        }
      ]
    }
  }
}
```

To make a setting sticky to a slot, use the `slotConfigNames` child resource to list the names that should not swap.

```bicep
resource slotConfig 'Microsoft.Web/sites/slotConfigNames@2022-09-01' = {
  name: '${app.name}/slotConfigNames'
  properties: {
    appSettingNames: [
      'ApiBaseUrl'
    ]
    connectionStringNames: [
      'DefaultConnection'
    ]
  }
}
```

This tells App Service which names remain with the slot during a swap. The actual app settings and connection strings still live on the site or slot resource; `slotConfigNames` only marks them as sticky.

**Deployment slots and why they matter**

Deployment slots are live app instances with their own hostnames and configuration. They let you deploy and test a new version before sending production traffic to it.

Typical slot workflow:

- Deploy new code to a staging slot.
- Validate the app with test traffic.
- Warm up the slot if needed.
- Swap staging into production.

Slots are especially useful when you need:

- Zero-downtime releases
- Safe rollback
- Pre-production validation
- Controlled configuration changes

**Slot support by App Service plan tier**

Slot availability depends on the App Service plan tier.

| Tier | Deployment slots |
|---|---|
| Free | No |
| Shared | No |
| Basic | Yes, limited number of slots |
| Standard | Yes |
| Premium | Yes |
| Isolated | Yes |

For exam prep, remember that Basic does support slots, but only a small number. Standard and above are the common tiers for slot-based deployment workflows.

**What swaps during a slot swap**

A swap exchanges the content and selected configuration between two slots. The exact behavior depends on whether a setting is marked as slot-specific.

Generally:

- **App content** moves between slots during the swap.
- **Non-sticky app settings** and **non-sticky connection strings** are exchanged.
- **Sticky settings** remain with their original slot.
- **Hostnames** and routing are updated so traffic can move to the target slot.

This distinction is critical. A setting marked as a slot setting does not swap. A setting that is not marked sticky does swap.

| Item | Swaps by default | Can be sticky |
|---|---:|---:|
| App content | Yes | No |
| App settings | Yes, unless sticky | Yes |
| Connection strings | Yes, unless sticky | Yes |
| Hostname binding | Managed by swap | N/A |
| Slot-specific diagnostics/config | Depends on feature | Often yes |

> Do not oversimplify slot swaps as “everything swaps.” The sticky/non-sticky distinction determines what stays and what moves.

**Traffic routing and gradual exposure**

Deployment slots also support traffic routing so you can gradually shift users to a new version. This is useful when you want to reduce risk instead of swapping all traffic at once.

Common approaches:

- **Percentage-based routing** sends a portion of requests to a slot.
- **Manual validation** lets you test the slot before full promotion.
- **Swap with preview** stages the swap so you can validate the target configuration and warm-up behavior before completing the swap.

Use traffic routing when you want canary-style validation or a gradual rollout. Use a full swap when you are confident the slot is ready for production.

**Swap with preview and warm-up behavior**

Swap with preview is a safer promotion path. It prepares the swap and lets you validate the target slot before the final exchange completes.

Practical interpretation:

- The platform stages the swap.
- The source slot is warmed using the target slot’s intended production context.
- You validate the app before completing the swap.

This is not the same as permanently applying all target settings to the source slot. The behavior is more nuanced and is designed to reduce risk during promotion.

**Key Vault references for secrets**

Azure Key Vault references let App Service resolve secrets at runtime without storing the secret value directly in the app configuration. This is often preferred for passwords, API keys, and certificates.

A Key Vault reference looks like this in an app setting:

```text
@Microsoft.KeyVault(SecretUri=https://myvault.vault.azure.net/secrets/DbPassword/abcd1234)
```

How it works:

- App Service uses a managed identity or another authorized identity to access Key Vault.
- The identity must have permission to read the secret.
- App Service resolves the reference and injects the secret value into the app at runtime.

Use Key Vault references when:

- You need centralized secret management.
- You want to avoid storing secrets in App Service configuration.
- You need rotation-friendly secret handling.

**Practical end-to-end workflow**

A common AZ-204 scenario is: configure a database connection, deploy to a staging slot, validate, then swap to production.

1. Create the app setting or connection string in the staging slot.
2. Mark environment-specific values as slot settings if they must stay with staging.
3. Deploy the application to the staging slot.
4. Read the value in code using the runtime’s configuration API.
5. Validate the app against the staging database or service.
6. Swap staging into production when ready.
7. Confirm that sticky settings stayed with the correct slot.

Example flow for .NET:

```csharp
var conn = builder.Configuration.GetConnectionString("DefaultConnection");
using var connection = new SqlConnection(conn);
await connection.OpenAsync();
```

Example flow for Node.js:

```javascript
const conn = process.env.SQLCONNSTR_DefaultConnection;
```

Example flow for Python:

```python
import os
conn = os.environ["SQLCONNSTR_DefaultConnection"]
```

If you use a framework that maps connection strings automatically, prefer the framework’s configuration API instead of reading the raw environment variable.

**Exam-focused decision points**

AZ-204 questions often test whether you choose the right configuration mechanism.

- Use **app settings** for general runtime values and feature flags.
- Use **connection strings** for database and service connection data.
- Use **slot settings** for values that must not move during a swap.
- Use **Key Vault references** for secrets that should not be stored in App Service configuration.
- Use **deployment slots** for safe release validation and rollback.

Typical exam scenario:

- You need to test a new version against production-like traffic without downtime.
- Deploy to a staging slot, validate, then swap.
- Keep the staging database connection fixed to staging by marking it as a slot setting.

Another common scenario:

- You need to rotate a database password without redeploying code.
- Store the secret in Key Vault and reference it from App Service.
- Ensure the app identity can read the secret.

**Troubleshooting and common pitfalls**

Configuration issues often come from naming, runtime mapping, or slot behavior.

- **Wrong name mapping**: the code reads `DefaultConnection`, but App Service exposes `SQLCONNSTR_DefaultConnection` for non-.NET runtimes.
- **Missing slot sticky flag**: a staging-only value swaps into production unexpectedly.
- **Incorrect connection string type**: the platform exposes the value with the wrong prefix or the app interprets it incorrectly.
- **Identity not authorized**: Key Vault reference resolution fails because the managed identity lacks secret access.
- **App restart required**: configuration changes may require a restart to take effect.

A good troubleshooting habit is to verify the value in the portal, confirm whether it is sticky, and then check how the runtime reads configuration.

**Quick reference**

- **App settings**: general runtime configuration
- **Connection strings**: database/service connections
- **Slot settings**: values that stay with a slot
- **Deployment slots**: staged app instances for safe release
- **Traffic routing**: gradual user exposure
- **Key Vault references**: secret resolution without hardcoding values

**Summary**

App Service configuration is a core AZ-204 topic because it connects deployment, runtime behavior, and release safety. You should know how to configure settings in the Portal, CLI, ARM, and Bicep; how different runtimes consume app settings and connection strings; and how slot swaps behave with sticky versus non-sticky values. Deployment slots and Key Vault references are the main tools for safer, production-grade application delivery.
