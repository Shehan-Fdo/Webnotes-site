## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Deployment lifecycle overview**

A typical Azure deployment pipeline moves code from a local workspace to a build artifact or container image, then into a target service, and finally through validation. For AZ-204, you need to recognize both the deployment mechanism and the authentication model behind it.

A practical flow looks like this:

```text
source code
  -> build/test
  -> artifact or image
  -> deploy to Azure
  -> smoke test / validate
```

- **Local workflows** usually mean you deploy from your machine with Azure CLI, Core Tools, or a publish profile.
- **GitHub Actions** automate the same flow with triggers, secrets, and environment protections.
- **Artifacts** are common for App Service and Functions.
- **Images** are common for container-based deployments such as App Service for Containers and Azure Container Apps.

**Common deployment targets**

| Target | Typical deployment style | Notes |
|---|---|---|
| Azure App Service | ZIP deploy, run-from-package, GitHub Actions, publish profile | Good for web apps and APIs |
| Azure Functions | Zip/package deploy, language-specific build and publish flows | Deployment model depends on runtime and hosting plan |
| Azure Container Apps | Build image, push to registry, update app revision | Image pull permissions matter |
| Azure Container Instances | Pull image from registry | Often used for simple container workloads |

**Local deployment workflows**

Local deployment is useful for quick iteration, troubleshooting, and exam scenarios where you need to identify the simplest path to publish code. Azure CLI and project-specific tools are the most common options.

- **Azure CLI** works well for scripted deployments and resource configuration.
- **Azure Functions Core Tools** supports local function development and publish flows.
- **`az webapp deploy`** and ZIP-based methods are common for App Service.
- **Publish profiles** provide a direct deployment credential for App Service and some Function App scenarios.

A common App Service ZIP deployment pattern is:

```bash
zip -r app.zip .
az webapp deploy \
  --resource-group rg-demo \
  --name my-webapp \
  --src-path app.zip \
  --type zip
```

For Linux App Service custom containers, you configure the image rather than pushing source code directly:

```bash
az webapp config container set \
  --resource-group rg-demo \
  --name my-webapp \
  --docker-custom-image-name myregistry.azurecr.io/myapp:1.0.0 \
  --docker-registry-server-url https://myregistry.azurecr.io
```

> App Service container behavior differs by platform and CLI version. For exam purposes, focus on the concept: App Service can run code packages or custom containers, and the deployment command must match the hosting model.

**App Service deployment options**

App Service supports several deployment approaches, and the right one depends on whether you deploy source, a package, or a container.

- **ZIP deploy** uploads a compressed package and extracts it or runs it as a package depending on configuration.
- **Run-from-package** mounts the package read-only, which improves consistency and avoids file drift.
- **GitHub Actions built-in deployment** can publish directly to App Service after build and test steps.
- **Publish profile deployment** uses a service-specific credential and is still common in legacy or simple pipelines.

A run-from-package setting often looks like this:

```bash
az webapp config appsettings set \
  --resource-group rg-demo \
  --name my-webapp \
  --settings WEBSITE_RUN_FROM_PACKAGE=1
```

Use run-from-package when you want predictable deployments and fewer file-system side effects. It is especially helpful when the app writes to the local file system only at runtime and not during deployment.

**Slot deployment and swap strategy**

Deployment slots let you stage a release before sending traffic to production. This is a common exam topic because it reduces downtime and supports safe rollouts.

- **Production slot** serves live traffic.
- **Staging slot** receives the new build first.
- **Swap** moves the warmed-up staging version into production.
- **Slot settings** stay with the slot during swaps.

Example slot configuration:

```bash
az webapp deployment slot create \
  --resource-group rg-demo \
  --name my-webapp \
  --slot staging

az webapp config appsettings set \
  --resource-group rg-demo \
  --name my-webapp \
  --slot staging \
  --settings ConnectionStrings__Db=Server=tcp:demo.database.windows.net;Database=appdb
```

Use slot-specific settings for values that should not move during a swap, such as environment-specific connection strings or feature flags.

**GitHub Actions fundamentals**

GitHub Actions automates build and deployment with YAML workflows. For AZ-204, you should know the main pieces: triggers, jobs, steps, secrets, and environments.

- **Triggers** start the workflow, such as `push`, `pull_request`, or `workflow_dispatch`.
- **Jobs** run on a runner and can depend on each other.
- **Steps** execute commands or actions.
- **Secrets** store credentials such as publish profiles or service principal values.
- **Environments** can require approvals before deployment.

A simple deployment trigger pattern:

```yaml
name: deploy-webapp

on:
  push:
    branches:
      - main
  workflow_dispatch:
```

Use `push` for automatic delivery after merge, `pull_request` for validation-only builds, and `workflow_dispatch` for manual releases or hotfixes. Environment protection rules are useful when you want a human approval gate before production deployment.

**Build, artifact, deploy pattern**

The most reliable workflow separates build from deploy. This makes failures easier to diagnose and allows the same artifact to be promoted through environments.

```yaml
name: build-and-deploy

on:
  push:
    branches: [ main ]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '8.0.x'
      - run: dotnet restore
      - run: dotnet publish -c Release -o publish
      - uses: actions/upload-artifact@v4
        with:
          name: app
          path: publish

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/download-artifact@v4
        with:
          name: app
          path: publish
      - uses: azure/webapps-deploy@v3
        with:
          app-name: my-webapp
          publish-profile: ${{ secrets.AZURE_WEBAPP_PUBLISH_PROFILE }}
          package: publish
```

This pattern is useful because the build job can run tests and produce a known artifact, while the deploy job only handles release.

**Authentication choices in GitHub Actions**

You can authenticate to Azure in several ways, and the best choice depends on the target service and security requirements.

- **OIDC (OpenID Connect)** uses short-lived federated identity tokens and avoids storing long-lived secrets.
- **Service principal secrets** are still common but less preferred than OIDC.
- **Publish profiles** are simple for App Service and often used in exam examples.
- **Managed identity** is used by Azure resources at runtime, not usually by GitHub-hosted runners directly.

A modern OIDC-based Azure login workflow requires:
- `permissions: id-token: write`
- an Entra ID app registration or service principal with a federated credential
- the `azure/login@v2` action configured for the tenant, subscription, and client ID

Example:

```yaml
permissions:
  id-token: write
  contents: read

steps:
  - uses: azure/login@v2
    with:
      client-id: ${{ secrets.AZURE_CLIENT_ID }}
      tenant-id: ${{ secrets.AZURE_TENANT_ID }}
      subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}
```

This works when the federated identity trust is configured correctly in Microsoft Entra ID. The action itself supports OIDC; the missing piece is the federated credential relationship.

**Publish profile deployment**

Publish profiles are a straightforward option for App Service and some Function App deployments. They are easy to use but less flexible than Azure RBAC-based authentication.

```yaml
- uses: azure/webapps-deploy@v3
  with:
    app-name: my-webapp
    publish-profile: ${{ secrets.AZURE_WEBAPP_PUBLISH_PROFILE }}
    package: publish
```

Use a publish profile when you want a quick deployment path and do not need broader Azure resource access. Prefer OIDC or service principal-based authentication for more controlled enterprise pipelines.

**Azure App Service deployment details**

App Service supports both code and container workloads. The deployment method should match the app type.

- **Code apps** commonly use ZIP deploy, run-from-package, or GitHub Actions deployment actions.
- **Linux custom containers** use image configuration and registry access.
- **Windows App Service** often uses package-based deployment or platform-specific tooling.
- **Deployment slots** help with zero-downtime releases.

Common failure modes include:
- wrong runtime stack selected in App Service
- package path does not contain the app entry point
- missing app settings after deployment
- slot swap issues caused by non-sticky settings

**Azure Functions deployment details**

Functions deployment depends on the hosting model, runtime, and language stack. Do not assume one action fits every scenario.

- **Zip/package deployment** is common for many Functions apps.
- **Core Tools publish** is useful during local development.
- **GitHub Actions** can build and deploy, but the exact package path depends on the runtime.
- **Consumption, Premium, and Dedicated plans** can affect build and deployment behavior.

A local publish example:

```bash
func azure functionapp publish my-function-app
```

A GitHub Actions example for a packaged Functions app:

```yaml
- uses: Azure/functions-action@v1
  with:
    app-name: my-function-app
    package: ./publish
```

Use the package path that contains the deployable output for your stack. For some languages, you publish compiled output; for others, you deploy source plus platform build settings. The action is not a universal one-size-fits-all solution.

**Container-based deployments**

Container deployments follow a different pipeline: build image, push image, then update the Azure service to pull that image.

```text
source code
  -> docker build
  -> push to registry
  -> Azure service pulls image
  -> validate running revision
```

For Azure Container Registry (ACR), the workflow usually needs:
- registry access for the identity or credentials used in the pipeline
- permission for the Azure service to pull the image
- a correct image tag or digest

Example build and push:

```yaml
- uses: azure/login@v2
  with:
    client-id: ${{ secrets.AZURE_CLIENT_ID }}
    tenant-id: ${{ secrets.AZURE_TENANT_ID }}
    subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}

- run: az acr login --name myregistry

- run: |
    docker build -t myregistry.azurecr.io/myapp:${{ github.sha }} .
    docker push myregistry.azurecr.io/myapp:${{ github.sha }}
```

`azure/login` authenticates the Azure CLI context, but Docker push still depends on registry authentication. In practice, `az acr login` or explicit Docker credentials are what enable the push, and the identity must have the right ACR permissions.

**Container Apps deployment details**

Azure Container Apps uses container revisions. Updating the image can create a new revision depending on the app configuration and revision mode, but you should not assume every update behaves identically.

- **Revisions** represent versions of the app.
- **Traffic splitting** can route traffic between revisions.
- **Managed identity** can be used to pull from ACR.
- **Registry credentials** may also be configured directly.

Example update:

```bash
az containerapp update \
  --name my-containerapp \
  --resource-group rg-demo \
  --image myregistry.azurecr.io/myapp:1.0.1
```

For ACR access, you can configure managed identity and grant `AcrPull` on the registry:

```bash
az containerapp identity assign \
  --name my-containerapp \
  --resource-group rg-demo

az role assignment create \
  --assignee <principal-id> \
  --role AcrPull \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.ContainerRegistry/registries/myregistry
```

Use managed identity when you want the app to pull images without storing registry passwords. This is the preferred enterprise pattern when supported by the service.

**Deployment triggers and environment protections**

GitHub Actions supports several deployment control points that matter in real projects and exam scenarios.

- **`push`** triggers automatic deployment after merge.
- **`pull_request`** is best for validation and test-only workflows.
- **`workflow_dispatch`** enables manual release control.
- **Environments** can require reviewers before secrets are exposed.
- **Branch protection** helps ensure only reviewed code reaches deployment branches.

A common release pattern is:
1. Run tests on every pull request.
2. Build and publish an artifact on merge to `main`.
3. Deploy to staging automatically.
4. Require approval before production deployment.
5. Swap or promote after validation.

**Practical deployment comparison**

| Scenario | Best fit | Why |
|---|---|---|
| Quick web app release | Publish profile or ZIP deploy | Simple and fast |
| Repeatable enterprise release | GitHub Actions + artifact | Consistent and auditable |
| Functions app with local dev workflow | Core Tools publish | Matches local development |
| Containerized app | Build image + push + deploy image | Matches container lifecycle |
| Safe production rollout | Deployment slots or revisions | Supports validation before traffic shift |

**Troubleshooting and common failure modes**

Deployment issues often come from mismatched credentials, wrong package paths, or service-specific configuration gaps.

- **Authentication failures** usually mean the workflow lacks the right secret, federated credential, or RBAC role.
- **Package failures** often mean the artifact does not contain the expected entry point.
- **Registry failures** usually mean the app cannot pull the image or the pipeline cannot push it.
- **Slot swap issues** often come from non-sticky settings or environment-specific configuration.
- **Runtime failures** can happen when the app setting or startup command does not match the deployed artifact.

Useful checks:

```bash
az webapp log tail --resource-group rg-demo --name my-webapp
az functionapp log stream --resource-group rg-demo --name my-function-app
az containerapp logs show --name my-containerapp --resource-group rg-demo
```

**Exam-focused takeaways**

- Choose the deployment method that matches the hosting model: code package, function package, or container image.
- Prefer **OIDC** for GitHub Actions when possible because it avoids long-lived secrets.
- Use **publish profiles** when the scenario is App Service and the question emphasizes simplicity.
- Use **deployment slots** for safer App Service releases.
- Use **revisions** and traffic management concepts for Container Apps.
- Remember that registry access is separate from Azure login; image push and image pull each need the right permissions.
- Treat Functions deployment as runtime-specific rather than universal.

**Quick reference workflow**

```text
1. Commit code
2. Trigger GitHub Actions
3. Build and test
4. Publish artifact or image
5. Deploy to Azure target
6. Validate logs and health
7. Promote or swap to production
```
