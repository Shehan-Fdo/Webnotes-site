## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Resource groups and deployment scope**

An Azure <abbr title="resource group">RG</abbr> is a logical container for related resources that share a lifecycle. You use it to deploy, manage, monitor, and delete resources as a unit. In practice, the resource group is often the boundary for ownership, environment, and application lifecycle, even though some resources can be moved across groups with constraints.

A resource group belongs to exactly one subscription and one Azure region for metadata storage. The resources inside it can be in different regions, but the group itself has a location value that stores its metadata and supports compliance and recovery scenarios.

- Use a resource group when resources share:
  - A common deployment lifecycle
  - A common owner or team
  - A common environment such as dev, test, or prod
- Avoid mixing unrelated applications in the same group unless you have a clear operational reason.
- Treat the resource group as a management boundary, not a network boundary or security boundary by itself.

> A resource group is not a security boundary. Access control still depends on Azure role-based access control (<abbr title="role-based access control">RBAC</abbr>), locks, policies, and resource-level permissions.

**Subscription context**

Many Azure operations happen in the context of the current subscription. If you target the wrong subscription, you can create resources in the wrong tenant scope, apply tags incorrectly, or deploy to the wrong environment. Always confirm the active subscription before running CLI, PowerShell, or deployment commands.

Common ways to inspect and change context:

```bash
az account show
az account list --output table
az account set --subscription "Contoso-Prod"
```

For PowerShell:

```powershell
Get-AzContext
Get-AzSubscription
Set-AzContext -Subscription "Contoso-Prod"
```

For Azure Resource Manager templates and Bicep, scope matters:
- **Subscription scope** is used for creating resource groups, policy assignments, role assignments, and some higher-level resources.
- **Resource group scope** is used for most application resources.
- **Management group scope** is used for governance across subscriptions.

**Creating a resource group**

You can create a resource group with Azure CLI, PowerShell, ARM templates, or Bicep. The most common exam-relevant pattern is to create it explicitly before deploying application resources into it.

CLI example:

```bash
az group create \
  --name rg-app-prod-weu \
  --location westeurope
```

PowerShell example:

```powershell
New-AzResourceGroup -Name "rg-app-prod-weu" -Location "westeurope"
```

ARM template deployment at subscription scope can create a resource group, but the deployment itself must be started at subscription scope. A simplified pattern looks like this:

```json
{
  "$schema": "https://schema.management.azure.com/schemas/2018-05-01/subscriptionDeploymentTemplate.json#",
  "contentVersion": "1.0.0.0",
  "resources": [
    {
      "type": "Microsoft.Resources/resourceGroups",
      "apiVersion": "2021-04-01",
      "name": "rg-app-prod-weu",
      "location": "westeurope"
    }
  ]
}
```

Bicep commonly targets subscription scope when creating a resource group:

```bicep
targetScope = 'subscription'

param rgName string = 'rg-app-prod-weu'
param location string = 'westeurope'

resource rg 'Microsoft.Resources/resourceGroups@2021-04-01' = {
  name: rgName
  location: location
}
```

> Resource group creation is a subscription-scope operation. Do not assume a resource group is created from a resource-group-scope deployment unless the deployment is explicitly targeting the correct scope.

**Naming conventions**

Naming conventions help you identify resources quickly, automate deployments, and avoid collisions. Azure does not enforce one universal naming standard, but many organizations use a predictable pattern that encodes environment, workload, region, and resource type.

A practical naming pattern often looks like:

```text
<org>-<app>-<env>-<region>-<type>
```

Examples:
- `contoso-payments-prod-weu-rg`
- `contoso-payments-dev-weu-app`
- `contoso-shared-prod-eus-kv`

Good naming conventions are:
- **Consistent** across teams and subscriptions
- **Readable** by humans and scripts
- **Valid** for the Azure resource type
- **Short enough** to fit service-specific limits

Important considerations:
- Some Azure resources require lowercase names.
- Some resources allow only letters, numbers, and hyphens.
- Some names must be globally unique, such as many storage account names and some DNS-related resources.
- Resource group names are not globally unique; they are unique within a subscription.

A few practical examples:
- Resource group: `rg-orders-prod-weu`
- App Service plan: `asp-orders-prod-weu`
- Storage account: `stordersprodweu01`
- Key Vault: `kv-orders-prod-weu`

**Naming strategy by scope**

| Scope | What to name carefully | Why it matters |
|---|---|---|
| Subscription | Management groups, policy assignments, role assignments | Governance and access control clarity |
| Resource group | Workload and environment grouping | Lifecycle management and deletion safety |
| Resource | Service-specific names | Uniqueness, DNS, and service constraints |

**Tags**

Tags are key-value pairs that help you organize, query, and govern Azure resources. They are especially useful when a resource group boundary is too coarse or when you need cross-cutting metadata such as cost center, owner, application, or environment.

Common tag keys:
- `Environment`
- `Owner`
- `CostCenter`
- `Application`
- `BusinessUnit`
- `DataClassification`

Example tags:

```text
Environment=Prod
Owner=platform-team
CostCenter=FIN-204
Application=orders-api
```

Tags are useful for:
- Cost management and chargeback
- Inventory and reporting
- Automation and policy enforcement
- Filtering resources in the portal and CLI

Azure tag keys are generally handled in a case-insensitive way in management experiences, so use a consistent convention rather than relying on case differences to distinguish keys. For example, prefer one standard such as `Environment` everywhere.

**Tagging resources**

Azure CLI can add or update tags on an existing resource:

```bash
az resource tag \
  --ids "/subscriptions/<sub-id>/resourceGroups/rg-orders-prod-weu/providers/Microsoft.Web/sites/orders-api" \
  --tags Environment=Prod Owner=platform-team CostCenter=FIN-204
```

Be careful: tagging commands can replace the existing tag set unless you intentionally merge with the current tags. Check the command behavior for your CLI version and use a read-modify-write approach when preserving existing tags matters.

A safer workflow is:
1. Read the current tags.
2. Merge the new tags with the existing set.
3. Write the full updated set back.

Example with Azure CLI and `jq`:

```bash
current=$(az resource show \
  --ids "/subscriptions/<sub-id>/resourceGroups/rg-orders-prod-weu/providers/Microsoft.Web/sites/orders-api" \
  --query tags -o json)

updated=$(echo "$current" | jq '. + {"Environment":"Prod","Owner":"platform-team"}')

az resource update \
  --ids "/subscriptions/<sub-id>/resourceGroups/rg-orders-prod-weu/providers/Microsoft.Web/sites/orders-api" \
  --set tags="$updated"
```

For resource groups, you can tag the group itself:

```bash
az group update \
  --name rg-orders-prod-weu \
  --set tags.Environment=Prod tags.Owner=platform-team tags.CostCenter=FIN-204
```

**Tagging with ARM and Bicep**

You can define tags declaratively in templates so deployments remain repeatable.

ARM example:

```json
{
  "type": "Microsoft.Storage/storageAccounts",
  "apiVersion": "2023-01-01",
  "name": "stordersprodweu01",
  "location": "westeurope",
  "tags": {
    "Environment": "Prod",
    "Owner": "platform-team"
  },
  "sku": {
    "name": "Standard_LRS"
  },
  "kind": "StorageV2"
}
```

Bicep example:

```bicep
param location string = resourceGroup().location

resource stg 'Microsoft.Storage/storageAccounts@2023-01-01' = {
  name: 'stordersprodweu01'
  location: location
  tags: {
    Environment: 'Prod'
    Owner: 'platform-team'
  }
  sku: {
    name: 'Standard_LRS'
  }
  kind: 'StorageV2'
}
```

**Subscription, resource group, and resource scope in practice**

Scope determines what you can deploy and where you can reference existing objects. In Bicep, use `targetScope` to match the deployment boundary.

Subscription-scope Bicep for creating a resource group:

```bicep
targetScope = 'subscription'

param rgName string
param location string = 'westeurope'

resource rg 'Microsoft.Resources/resourceGroups@2021-04-01' = {
  name: rgName
  location: location
}
```

Resource-group-scope Bicep for deploying application resources:

```bicep
targetScope = 'resourceGroup'

param location string = resourceGroup().location

resource app 'Microsoft.Web/sites@2023-01-01' = {
  name: 'orders-api'
  location: location
  tags: {
    Environment: 'Prod'
  }
  properties: {}
}
```

If you need to deploy into an existing resource group from a higher scope, use explicit scope targeting rather than assuming the current context is correct.

**Azure CLI and context-aware deployment**

Before deploying, verify the active subscription and resource group:

```bash
az account show --output table
az group show --name rg-orders-prod-weu
```

Deploy a resource-group-scope Bicep file:

```bash
az deployment group create \
  --resource-group rg-orders-prod-weu \
  --template-file main.bicep \
  --parameters location=westeurope
```

Deploy a subscription-scope Bicep file that creates a resource group:

```bash
az deployment sub create \
  --location westeurope \
  --template-file rg-create.bicep \
  --parameters rgName=rg-orders-prod-weu location=westeurope
```

**Governance and operational best practices**

Resource groups, naming, and tags work best when they support governance rather than just organization. Use them to make automation safer and reporting easier.

- Use one resource group per application or per environment when lifecycle differs.
- Use tags for metadata that cuts across groups, such as cost center or owner.
- Use naming conventions that are stable and machine-friendly.
- Use Azure Policy to enforce required tags and naming rules where possible.
- Use locks when you need to prevent accidental deletion or modification.

Common policy goals:
- Require `Environment` and `Owner` tags
- Deny resources without approved naming patterns
- Append or modify tags during deployment
- Restrict allowed regions for compliance

**Troubleshooting and exam tips**

If a deployment fails, check these first:
- The active subscription is correct.
- The deployment scope matches the template scope.
- The resource name meets service-specific rules.
- The resource group exists in the expected subscription.
- Required tags or policy conditions are satisfied.

Remember these exam-friendly distinctions:
- A resource group is a logical container, not a network boundary.
- Subscription context affects where resources are created.
- Tags improve governance, but they do not replace RBAC or policy.
- Resource group creation is a subscription-scope action.
