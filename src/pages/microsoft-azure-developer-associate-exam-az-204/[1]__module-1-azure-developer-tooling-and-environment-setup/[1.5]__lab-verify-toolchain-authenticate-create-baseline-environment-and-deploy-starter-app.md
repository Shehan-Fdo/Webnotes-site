## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Lab purpose and expected outcome**

This lab establishes a repeatable baseline for Azure development work: you verify the local toolchain, authenticate to the correct Azure tenant and subscription, create the minimum required cloud resources, and deploy a starter app for validation. The goal is not just to “make it work,” but to confirm that your environment can support later AZ-204 tasks such as build, deploy, monitor, and troubleshoot.

A good lab run ends with three verifiable facts:

- Your local tools match the lab’s required versions closely enough to avoid runtime or CLI mismatches.
- You are signed in to the intended Azure identity and subscription.
- A starter application is deployed and reachable in Azure App Service.

**Toolchain verification**

Before you create anything in Azure, confirm the local stack that the lab depends on. The exact versions vary by lab, but the principle is the same: verify the SDK, runtime, package manager, and Azure CLI before you start.

Common checks:

```bash
az version
az account show
node --version
npm --version
dotnet --version
git --version
python --version
```

If the lab uses Azure CLI extensions, verify them too:

```bash
az extension list -o table
az extension add --name webapp
```

For App Service work, the most important local dependencies are:

- **Azure CLI** for resource creation, deployment, and verification.
- **Git** for source control-based workflows.
- **Language SDK/runtime** that matches the starter app stack.
- **Archive tooling** such as `zip` when you package a published app for deployment.

> On Windows, `zip` is not a default shell command. Use PowerShell compression or a build tool that creates the archive for you.

A practical version check is to compare the lab’s required runtime with what is installed locally. For example:

- Node.js labs often need a specific LTS release.
- .NET labs often require the exact SDK major/minor version used by the starter project.
- Python labs may depend on a matching interpreter and virtual environment tooling.

**Authentication and identity selection**

Authentication is more than “logging in.” In Azure labs, you must confirm the identity, tenant, and subscription before you create resources. This avoids accidental deployment into the wrong subscription or a tenant where you lack permissions.

Interactive sign-in is the most common lab path:

```bash
az login
az account show
az account list -o table
```

If you have multiple subscriptions, explicitly select the one the lab expects:

```bash
az account set --subscription "<subscription-id-or-name>"
az account show --query "{name:name, id:id, tenantId:tenantId}" -o json
```

Useful checks before resource creation:

- Confirm the signed-in user or service principal is the one you intend to use.
- Confirm the subscription has sufficient quota for App Service and related resources.
- Confirm your role assignment allows resource group and App Service creation, typically **Contributor** or higher for the target scope.

For automation or CI/CD, a service principal is common:

```bash
az login --service-principal \
  --username "<appId>" \
  --password "<clientSecret>" \
  --tenant "<tenantId>"
```

If the lab uses federated or managed identity scenarios, the authentication model changes, but the same rule applies: verify the active identity before provisioning anything.

**Baseline environment and minimum Azure resources**

The baseline environment is the smallest set of Azure resources needed to host and verify the starter app. For a typical AZ-204 App Service lab, the minimum is usually:

- A **resource group** to contain and clean up all lab resources.
- An **App Service plan** to define compute and pricing tier.
- A **Web App** to host the starter application.

Depending on the lab, you may also need:

- **Application Insights** for telemetry and log-based troubleshooting.
- **Storage account** if the starter app writes files, queues, or blobs.
- **Database** resources only if the starter app explicitly uses persistence.

If the lab only asks you to deploy and verify a starter app, do not create extra services unless the instructions require them. Extra resources add cost and can obscure the core learning objective.

A typical resource group creation flow:

```bash
az group create \
  --name rg-az204-lab \
  --location eastus
```

A typical App Service plan and web app flow for Linux-based hosting:

```bash
az appservice plan create \
  --name asp-az204-lab \
  --resource-group rg-az204-lab \
  --sku B1 \
  --is-linux

az webapp create \
  --resource-group rg-az204-lab \
  --plan asp-az204-lab \
  --name <unique-app-name> \
  --runtime "<supported-runtime>"
```

Use a runtime string that matches the current Azure CLI and App Service stack support for your region and OS. Runtime tokens are version- and platform-specific, so verify the exact value with `az webapp list-runtimes` or the lab instructions rather than guessing.

```bash
az webapp list-runtimes --os linux -o table
```

**Starter app deployment flow**

The deployment method depends on the stack and the lab’s instructions. The most portable patterns are:

- **Zip deployment** for packaged artifacts.
- **Git-based deployment** for source-driven workflows.
- **`az webapp up`** for quick local-to-Azure publishing in some labs.

For many exam-style labs, a clean flow is:

1. Create the app locally.
2. Build or publish it.
3. Package the output.
4. Deploy the package.
5. Verify the app in the browser and in Azure.

### **Node.js example**

A simple Node.js starter app often uses Express or a minimal HTTP server. A common local flow is:

```bash
mkdir az204-node-lab
cd az204-node-lab
npm init -y
npm install express
```

Example `app.js`:

```javascript
const express = require("express");
const app = express();
const port = process.env.PORT || 3000;

app.get("/", (req, res) => {
  res.send("AZ-204 starter app is running");
});

app.listen(port, "0.0.0.0", () => {
  console.log(`Listening on port ${port}`);
});
```

For App Service on Linux, bind to `0.0.0.0` and listen on the port provided by the platform configuration, commonly via `PORT` or `WEBSITES_PORT` depending on the hosting model and stack. Do not hardcode a port that the platform is not expecting.

A package-and-deploy example:

```bash
zip -r app.zip . -x "node_modules/*"
az webapp deployment source config-zip \
  --resource-group rg-az204-lab \
  --name <unique-app-name> \
  --src app.zip
```

If the app needs build steps, run them before packaging so the deployed artifact contains the compiled output and production dependencies.

### **.NET example**

A .NET starter app typically uses `dotnet new` and `dotnet publish`:

```bash
dotnet new webapp -n Az204Starter
cd Az204Starter
dotnet publish -c Release -o publish
```

Package the published output:

```bash
cd publish
zip -r ../app.zip .
cd ..
```

On Windows, use PowerShell if `zip` is unavailable:

```powershell
Compress-Archive -Path .\publish\* -DestinationPath .\app.zip -Force
```

Deploy the package:

```bash
az webapp deployment source config-zip \
  --resource-group rg-az204-lab \
  --name <unique-app-name> \
  --src app.zip
```

**End-to-end lab example**

This is a complete, practical path you can follow for a typical App Service starter app lab.

1. Sign in and select the correct subscription.
   ```bash
   az login
   az account list -o table
   az account set --subscription "<subscription-name-or-id>"
   az account show --query "{user:user.name, subscription:name, tenant:tenantId}" -o json
   ```

2. Verify the required toolchain.
   ```bash
   az version
   git --version
   node --version
   npm --version
   ```

3. Create the resource group and App Service plan.
   ```bash
   az group create --name rg-az204-lab --location eastus
   az appservice plan create \
     --name asp-az204-lab \
     --resource-group rg-az204-lab \
     --sku B1 \
     --is-linux
   ```

4. Create the web app using a supported runtime.
   ```bash
   az webapp list-runtimes --os linux -o table
   az webapp create \
     --resource-group rg-az204-lab \
     --plan asp-az204-lab \
     --name <unique-app-name> \
     --runtime "<supported-runtime>"
   ```

5. Build and package the app.
   ```bash
   npm install
   zip -r app.zip . -x "node_modules/*"
   ```

6. Deploy the package.
   ```bash
   az webapp deployment source config-zip \
     --resource-group rg-az204-lab \
     --name <unique-app-name> \
     --src app.zip
   ```

7. Verify the app.
   ```bash
   az webapp show \
     --resource-group rg-az204-lab \
     --name <unique-app-name> \
     --query defaultHostName -o tsv
   ```

8. Open the site in a browser and confirm the expected response.

**Verification and diagnostics**

After deployment, verify both Azure-side status and application behavior. A successful deployment is not complete until the app responds correctly.

Useful checks:

- `az webapp show` confirms the resource exists and returns the hostname.
- `az webapp log tail` streams logs, but only after logging is enabled.
- Browser verification confirms the app is reachable from the public endpoint.

Enable logging before tailing logs:

```bash
az webapp log config \
  --resource-group rg-az204-lab \
  --name <unique-app-name> \
  --application-logging filesystem \
  --level information

az webapp log tail \
  --resource-group rg-az204-lab \
  --name <unique-app-name>
```

Common troubleshooting patterns:

- If the site returns an error page, check whether the app listens on the correct port and host.
- If deployment succeeds but the app fails to start, inspect startup logs and runtime compatibility.
- If `az login` succeeds but resource creation fails, verify subscription selection and role assignment.
- If zip deployment fails, confirm the archive contains the app content at the expected root level.

A simple troubleshooting sequence:

1. Confirm the active subscription and resource group.
2. Confirm the web app runtime matches the app stack.
3. Confirm the app listens on the expected port.
4. Enable and inspect logs.
5. Redeploy a clean package if the artifact is malformed.

**Node.js vs .NET deployment flow**

| Aspect | Node.js | .NET |
|---|---|---|
| Local preparation | Install dependencies with `npm install` | Restore and publish with `dotnet publish` |
| Packaging | Zip source or build output, excluding `node_modules` if appropriate | Zip the published output from `publish/` |
| Runtime concern | Must bind to the platform port and `0.0.0.0` | Must target the correct .NET runtime and publish configuration |
| Common failure | Missing dependencies or wrong port binding | Wrong SDK version or incomplete publish output |

**Cleanup**

Always remove lab resources when you finish. This prevents unnecessary charges and keeps your subscription tidy.

```bash
az group delete --name rg-az204-lab --yes --no-wait
```

If you created additional resources outside the resource group, delete them separately. In most labs, placing everything in one resource group makes cleanup simple and reliable.

**Exam and lab takeaways**

- Verify the toolchain before you create Azure resources.
- Select the correct tenant and subscription before provisioning.
- Use the smallest resource set that satisfies the lab.
- Match the app runtime to the hosting stack and deployment method.
- Enable logging before you rely on log streaming.
- Clean up resources after verification.
