## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Why this toolchain matters**

AZ-204 expects you to build, run, debug, and deploy Azure solutions locally before you publish them. That means your workstation needs the right language runtimes, Azure management tools, container tooling, and editor extensions so you can reproduce cloud behavior as closely as possible.

A good setup reduces friction in three places:

- **Development**: you can compile, run, and debug apps locally.
- **Integration**: you can authenticate to Azure, create resources, and deploy from the command line.
- **Validation**: you can test Functions, containers, and app settings before you push to Azure.

**Recommended local workflow**

A typical Azure developer loop looks like this:

1. Install the language runtime and SDK you need.
2. Install Azure CLI and sign in.
3. Install Functions Core Tools if you work with Azure Functions.
4. Install Docker if you build or test containers.
5. Install VS Code and the Azure-related extensions.
6. Verify each tool with version commands before you start a lab.

**.NET SDK installation**

The .NET SDK includes the compiler, runtime components, templates, and `dotnet` CLI. For AZ-204, you usually need a supported LTS (Long Term Support) version that matches the lab or project target framework, such as `.NET 8` for current projects.

Use the SDK, not just the runtime, because you need build and template commands:

```bash
dotnet --info
dotnet --list-sdks
dotnet new --list
```

Common tasks:

- Create a new project with `dotnet new`.
- Restore packages with `dotnet restore`.
- Build with `dotnet build`.
- Run locally with `dotnet run`.
- Publish for deployment with `dotnet publish`.

Example:

```bash
dotnet new webapi -n Contoso.Api
cd Contoso.Api
dotnet run
```

Practical notes:

- Install the SDK version required by your target framework; mismatches can cause restore or build errors.
- If you have multiple SDKs installed, `global.json` can pin a project to a specific version.
- On Windows, `where dotnet` helps confirm which installation is first on `PATH`.
- On macOS/Linux, `which dotnet` serves the same purpose.

Example `global.json`:

```json
{
  "sdk": {
    "version": "8.0.204",
    "rollForward": "latestFeature"
  }
}
```

**Python installation**

Python is commonly used for Azure Functions, automation scripts, and SDK-based tooling. For Azure development, install a supported Python 3 version and verify that both `python` and `pip` resolve correctly in your shell.

Check your installation:

```bash
python --version
pip --version
python -m pip --version
```

Use `python -m pip` when possible because it ensures you install packages into the interpreter you are actually using.

Typical setup steps:

- Install Python 3.x from the official distribution or package manager.
- Create a virtual environment for each project.
- Install dependencies from `requirements.txt`.
- Keep the project isolated from system-wide packages.

Example workflow:

```bash
python -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
pip install azure-functions
pip freeze > requirements.txt
```

On Windows PowerShell:

```powershell
python -m venv .venv
.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
pip install azure-functions
```

For Azure Functions in the current Python programming model, the app entry point is typically `function_app.py` at the project root. Older per-function folder layouts still exist in some samples, so do not confuse the two when following a lab or reading legacy code.

**Azure CLI installation**

The Azure CLI lets you manage Azure resources from the terminal. It is essential for authentication, resource creation, deployment automation, and troubleshooting.

Verify the installation:

```bash
az version
az --help
```

Sign in and inspect your active subscription:

```bash
az login
az account show
az account set --subscription "<subscription-id-or-name>"
```

Useful commands for AZ-204 labs:

```bash
az group create --name rg-az204-dev --location eastus
az storage account create \
  --name <globally-unique-storage-name> \
  --resource-group rg-az204-dev \
  --location eastus \
  --sku Standard_LRS
```

Important storage account constraints:

- The name must be globally unique across Azure.
- Use only lowercase letters and numbers.
- Length must be between 3 and 24 characters.

Common troubleshooting:

- If `az login` opens the wrong browser or fails in a remote shell, use `az login --use-device-code`.
- If commands fail because of the wrong subscription, re-run `az account set`.
- If a lab script expects a resource group, create it first; many Azure resources cannot be created without one.

**Functions Core Tools**

Azure Functions Core Tools let you create, run, and debug Functions apps locally. This is the closest local equivalent to the Azure Functions host and is especially important for trigger and binding testing.

Check the installed version:

```bash
func --version
func --help
```

Typical local workflow:

```bash
func init ContosoFunctions --worker-runtime dotnet-isolated
cd ContosoFunctions
func new
func start
```

For Python:

```bash
func init ContosoFunctions --worker-runtime python
cd ContosoFunctions
func new
func start
```

For local development, the project structure depends on the language and programming model:

- **Python**: current projects commonly use `function_app.py` at the root with `host.json` and `requirements.txt`.
- **.NET isolated**: function classes can live anywhere in the project; the important files are the `.csproj`, `host.json`, and your function class files.
- **Legacy models**: older templates may use per-function folders or in-process patterns, so always match the lab instructions and target runtime.

A minimal `host.json` example:

```json
{
  "version": "2.0",
  "logging": {
    "applicationInsights": {
      "samplingSettings": {
        "isEnabled": true
      }
    }
  }
}
```

A minimal local settings file:

```json
{
  "IsEncrypted": false,
  "Values": {
    "AzureWebJobsStorage": "UseDevelopmentStorage=true",
    "FUNCTIONS_WORKER_RUNTIME": "dotnet-isolated"
  }
}
```

Worker runtime values you should know:

- `dotnet` for .NET in-process apps.
- `dotnet-isolated` for .NET isolated worker apps.
- `python` for Python apps.
- `node` for JavaScript/TypeScript apps.

> `local.settings.json` is for local development only. Do not commit secrets or production connection strings into source control.

Storage emulator note:

- Many Functions scenarios use `AzureWebJobsStorage` locally.
- Azurite is the common local storage emulator, but not every binding or version behaves exactly like Azure Storage.
- If a trigger or binding behaves differently locally, validate against a real Azure Storage account before you assume the emulator is the issue.

**Docker installation and container workflow**

Docker is useful when you need consistent local environments, containerized apps, or deployment targets for Azure Container Apps, App Service containers, or AKS-related workflows.

Verify Docker:

```bash
docker version
docker info
docker ps
```

Common container workflow:

1. Build an image.
2. Run it locally.
3. Inspect logs and ports.
4. Push it to a registry.
5. Deploy it to Azure.

Example Dockerfile for an ASP.NET Core app:

```dockerfile
FROM mcr.microsoft.com/dotnet/aspnet:8.0 AS base
WORKDIR /app

FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src
COPY ["Contoso.Api/Contoso.Api.csproj", "Contoso.Api/"]
RUN dotnet restore "Contoso.Api/Contoso.Api.csproj"
COPY . .
WORKDIR "/src/Contoso.Api"
RUN dotnet publish "Contoso.Api.csproj" -c Release -o /app/publish /p:UseAppHost=false

FROM base AS final
WORKDIR /app
COPY --from=build /app/publish .
ENTRYPOINT ["dotnet", "Contoso.Api.dll"]
```

Build and run:

```bash
docker build -t contoso-api:latest .
docker run --rm -p 8080:8080 contoso-api:latest
```

Container port behavior can vary by base image and app configuration. Some ASP.NET Core container images are already configured for port `8080`, while others may require you to set `ASPNETCORE_URLS` or map a different port. Always confirm the image documentation and your app’s listening address.

Example with an explicit port setting:

```dockerfile
ENV ASPNETCORE_URLS=http://+:8080
EXPOSE 8080
```

Practical container tips:

- Use multi-stage builds to keep runtime images small.
- Match the SDK image version to your target framework.
- Prefer environment variables for runtime configuration.
- Use `docker logs <container>` when the app starts but does not respond.
- Use `docker exec -it <container> sh` or `bash` for interactive troubleshooting when the image includes a shell.

**VS Code and extensions**

Visual Studio Code is the most common editor for AZ-204 labs because it supports Azure sign-in, debugging, Functions, and container workflows through extensions.

Install and verify:

- VS Code itself.
- Azure Account extension for sign-in.
- Azure Functions extension for Functions development.
- C# extension or C# Dev Kit for .NET development.
- Python extension for Python development.
- Docker extension for container management.

Useful extension-driven workflows:

- Sign in to Azure from the VS Code Azure view.
- Create and deploy Functions apps from the command palette.
- Attach the debugger to local .NET or Python processes.
- Browse storage accounts, App Service apps, and Function Apps from the Azure sidebar.

Common commands in VS Code:

- `<kbd>Ctrl</kbd> + <kbd>Shift</kbd> + <kbd>P</kbd>` to open the Command Palette.
- `Azure: Sign In`
- `Azure Functions: Create New Project`
- `Azure Functions: Deploy to Function App`
- `Docker: Add Docker Files to Workspace`

A practical extension set for AZ-204:

| Area | Extension | Why you need it |
|---|---|---|
| Azure auth | Azure Account | Signs you into Azure and enables Azure services in VS Code |
| Functions | Azure Functions | Creates, runs, and deploys Functions projects |
| .NET | C# or C# Dev Kit | Debugs and edits .NET projects effectively |
| Python | Python | Manages interpreters, virtual environments, and debugging |
| Containers | Docker | Builds, runs, and inspects container images |

**End-to-end setup flow**

A clean setup usually follows this order:

1. Install the language SDKs you need, such as `.NET SDK` and `Python`.
2. Install `Azure CLI` and confirm `az login` works.
3. Install `Functions Core Tools` if you will build Azure Functions.
4. Install `Docker` if your labs include containers.
5. Install `VS Code` and the Azure-related extensions.
6. Create a sample project and run a local smoke test.

A simple validation checklist:

- `dotnet --info` returns the expected SDK version.
- `python --version` returns a supported Python 3 version.
- `az version` works and `az account show` returns your subscription.
- `func --version` works if you installed Functions Core Tools.
- `docker ps` runs without errors if Docker is installed and the daemon is active.
- VS Code can sign in to Azure and detect your project type.

**Troubleshooting patterns**

If a lab fails, the issue is often one of these:

- **PATH mismatch**: the shell is using an older SDK or a different interpreter.
- **Wrong subscription**: Azure CLI is authenticated, but not against the subscription you expect.
- **Missing emulator**: local storage-dependent Functions need Azurite or a real storage account.
- **Extension mismatch**: VS Code has the editor, but not the right Azure or language extension.
- **Runtime mismatch**: the project targets a different worker model than the installed tools support.

Useful diagnostics:

```bash
where dotnet
where python
where az
where func
```

On macOS/Linux:

```bash
which dotnet
which python
which az
which func
```

If a Functions app starts but bindings fail, inspect `local.settings.json`, confirm the worker runtime, and check whether the storage connection string is valid for the scenario. If a container starts but does not respond, verify the exposed port, the app’s listening URL, and the `docker run -p` mapping.

**Quick reference**

- Use the **SDK** for building and publishing, not just the runtime.
- Use **virtual environments** for Python projects.
- Use **Azure CLI** for repeatable resource management.
- Use **Functions Core Tools** for local Functions parity.
- Use **Docker** for reproducible runtime environments.
- Use **VS Code extensions** to reduce setup and deployment friction.
