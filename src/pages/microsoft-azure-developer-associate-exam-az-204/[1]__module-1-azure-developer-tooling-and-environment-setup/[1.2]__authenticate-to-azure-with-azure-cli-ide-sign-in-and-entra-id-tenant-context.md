## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Authentication model overview**

Azure development commonly uses Microsoft Entra ID for identity and access control. In practice, you authenticate as a user, a service principal, or a managed identity, and the active identity determines what resources you can see and which APIs you can call.

For AZ-204, focus on two separate ideas:

- **Authentication** answers who you are.
- **Authorization** answers what you can do after identity is established.

A common source of confusion is the difference between the signed-in account, the active tenant, and the active subscription. These are related but not identical:

- **Tenant** is the Microsoft Entra directory that owns identities and app registrations.
- **Subscription** is the Azure billing and resource container.
- **Context** in tools such as Azure CLI stores the current tenant and subscription selection for subsequent commands.

**Azure CLI sign-in and context**

Azure CLI is a local developer tool that can sign you in as a user and then store a profile on your machine. After sign-in, the CLI keeps track of the current account, tenant, and subscription context.

Typical sign-in flow:

```bash
az login
az account show
```

If you belong to multiple tenants or subscriptions, the initial sign-in may not select the one you want. You can inspect and change the active subscription context:

```bash
az account list --output table
az account set --subscription "00000000-0000-0000-0000-000000000000"
az account show --output jsonc
```

Key behavior to remember:

- **`az login`** authenticates you and populates the CLI profile.
- **`az account set`** changes the default subscription for later CLI commands.
- **`az account set` does not re-authenticate you** and does not change the tenant authentication itself.
- If the same user has access to multiple tenants, you may need to switch tenant context explicitly to see the right resources.

To sign in against a specific tenant, use the tenant identifier:

```bash
az login --tenant 11111111-1111-1111-1111-111111111111
```

If you need to verify which tenant and subscription are active, inspect the current account context:

```bash
az account show --query "{user:user.name, tenantId:tenantId, subscriptionId:id, name:name}" -o json
```

**Tenant switching and resource visibility**

Tenant context affects what you can discover in the Azure portal and CLI. If your user exists in multiple tenants, the same email address can map to different directory contexts with different app registrations, role assignments, and visible subscriptions.

Practical implications:

- In the **Azure portal**, the directory shown in the top-right account menu determines which tenant’s resources and identities you browse.
- In the **Azure CLI**, the active tenant in the profile determines which directory the CLI queries for identity and authorization data.
- A subscription can only be selected if your signed-in identity has access to it in that tenant.

A useful troubleshooting pattern is to compare the tenant and subscription you expect with the ones currently active:

```bash
az account show
az account list --all --output table
```

If you need to switch tenants after sign-in, re-run login with the target tenant or explicitly select a subscription that belongs to that tenant. In some cases, you may need to clear stale context first:

```bash
az account clear
az login --tenant 11111111-1111-1111-1111-111111111111
```

> If you see resources in the portal but not in the CLI, or vice versa, suspect a tenant mismatch first, then check subscription selection and role assignment.

**IDE sign-in and developer credentials**

Integrated development environments (IDEs) such as Visual Studio and Visual Studio Code can sign you in with your work or school account. Azure SDKs can often reuse that sign-in automatically through **developer credentials**.

This matters because many Azure SDKs use `DefaultAzureCredential`, which tries several credential sources in a chain. In local development, it can pick up IDE sign-in without extra code if the SDK and extension support it.

Common developer credential sources include:

- **Visual Studio** account sign-in
- **Visual Studio Code** Azure account sign-in
- **Azure CLI** sign-in
- Environment-based credentials for local automation

Example in .NET:

```csharp
using Azure.Identity;
using Azure.Security.KeyVault.Secrets;

var client = new SecretClient(
    new Uri("https://myvault.vault.azure.net/"),
    new DefaultAzureCredential());

var secret = await client.GetSecretAsync("MySecret");
Console.WriteLine(secret.Value.Value);
```

In this pattern, you do not hardcode credentials. Instead, the SDK resolves the best available identity from the local environment.

Important exam detail:

- `DefaultAzureCredential` behavior can vary by **language**, **package version**, and **SDK release**.
- The credential precedence order may differ slightly across implementations.
- If one identity source is available, it may be selected before another, which can change which account the app uses during local debugging.

**Azure SDK credential chain behavior**

`DefaultAzureCredential` is designed to work across local development and hosted environments. It typically tries multiple credential types in sequence until one succeeds.

Common sources in the chain include:

- Environment variables
- Managed identity
- IDE sign-in
- Azure CLI sign-in
- Interactive browser sign-in, depending on configuration and package

Practical consequences:

- In local development, your IDE sign-in may be used automatically.
- In Azure-hosted apps, managed identity usually becomes the preferred source.
- In CI/CD pipelines, environment variables or workload identity are often used instead of user sign-in.

A simplified mental model:

| Scenario | Likely identity source | Why |
|---|---|---|
| Local dev in Visual Studio | IDE sign-in | Fastest developer credential available |
| Local dev in VS Code | Azure extension sign-in | SDK can reuse developer login |
| Local dev from terminal | Azure CLI sign-in | CLI profile is available to SDKs |
| Azure App Service / VM / Function | Managed identity | No secrets to store |
| Automation pipeline | Service principal or federated identity | Non-interactive and repeatable |

**Visual Studio and VS Code tenant verification**

IDE sign-in is not just about being logged in. You also need the correct tenant context.

For Visual Studio:

- Check the signed-in account in the account manager.
- Confirm the organization or directory associated with the account.
- If you work across tenants, ensure the correct directory is selected for the project.

For Visual Studio Code:

- Use the Azure extension account view to confirm the signed-in account.
- Verify the active directory or subscription in the Azure sidebar.
- If the wrong tenant is active, sign out and sign back in with the correct account or directory.

Practical checks:

- Confirm the account shown in the Azure extension matches the tenant you expect.
- Refresh the Azure resources tree after switching directories.
- If resources do not appear, verify that the account has access in that tenant and subscription.

**Managed identity vs service principal vs user sign-in**

These identity types solve different problems. Knowing when to use each one is a common AZ-204 distinction.

| Identity type | Best use | Secret required | Typical environment |
|---|---|---:|---|
| User sign-in | Local development and interactive troubleshooting | No | Developer workstation |
| IDE sign-in | Local development with SDKs | No | Visual Studio / VS Code |
| Azure CLI sign-in | Local development and ad hoc admin tasks | No | Terminal session |
| Managed identity | Azure-hosted workloads | No | App Service, Functions, VMs, AKS, Container Apps |
| Service principal | Automation and app-to-app access | Usually yes, unless federated | Scripts, CI/CD, non-interactive apps |

Use these rules of thumb:

- Choose **user or IDE sign-in** when you are debugging locally and want fast setup.
- Choose **managed identity** when code runs in Azure and should avoid secrets.
- Choose **service principal** when a non-human identity must authenticate from automation or external systems.

**Azure CLI tokens and troubleshooting**

The Azure CLI can also help you understand authentication flow by retrieving access tokens for specific resources. This is useful when troubleshooting token acquisition or verifying which audience a token targets.

Example:

```bash
az account get-access-token --resource https://management.azure.com/
```

You can also request a token for other resources, depending on the API you need to call. This is useful when you want to inspect the token used for a specific service rather than only Azure Resource Manager (ARM).

Common use cases:

- Debugging whether the CLI can obtain a valid token
- Checking token audience for a service-specific API
- Testing whether your current tenant and subscription context are correct
- Comparing CLI-authenticated access with SDK behavior

If you are troubleshooting a custom API or service, the token audience must match the resource you are calling. A token for ARM does not automatically work for every Azure service.

**Tenant and subscription mismatch troubleshooting**

A frequent issue is signing in successfully but still getting authorization or visibility errors.

Example scenario:

- You sign in with `az login`.
- `az account show` displays a subscription you do not expect.
- The portal shows no resources, or the CLI returns `Subscription not found` or `The subscription is not in the current tenant`.

Troubleshooting steps:

1. Check the active tenant and subscription.
   ```bash
   az account show
   ```
2. List all subscriptions available to the signed-in identity.
   ```bash
   az account list --all --output table
   ```
3. Switch to the correct subscription.
   ```bash
   az account set --subscription "My Subscription Name"
   ```
4. If the subscription belongs to another tenant, sign in to that tenant explicitly.
   ```bash
   az login --tenant 11111111-1111-1111-1111-111111111111
   ```
5. If the context still looks stale, clear the profile and sign in again.
   ```bash
   az account clear
   az login
   ```

Common causes:

- You are signed into the wrong tenant.
- Your account has access to multiple directories, but the wrong one is active.
- You have subscription access, but not in the tenant currently selected.
- Role assignment exists, but propagation has not completed yet.

**How the portal, CLI, and SDK relate**

These tools often share the same underlying identity, but they do not always share the same context automatically.

- The **Azure portal** uses the directory and subscription selected in the browser session.
- The **Azure CLI** uses the local CLI profile.
- The **Azure SDK** uses the credential chain, which may prefer IDE sign-in, CLI sign-in, or managed identity depending on environment.

This means you can be authenticated in one tool and still fail in another if:

- The active tenant differs.
- The selected subscription differs.
- The SDK chooses a different credential source than you expected.
- The identity lacks the required role assignment.

**Practical development patterns**

For local development, a common pattern is to use `DefaultAzureCredential` and let it resolve the best available identity.

```csharp
var credential = new DefaultAzureCredential();
```

Then authenticate locally with one of these:

- Sign in to Visual Studio or VS Code.
- Run `az login` in a terminal.
- Set environment variables only when you need deterministic automation.

For Azure-hosted applications, prefer managed identity:

```csharp
var credential = new DefaultAzureCredential();
```

The same code can work in both environments because the credential chain adapts to the host.

For automation outside Azure, use a service principal or federated identity rather than a human account.

**Exam-focused distinctions**

- **`az login`** authenticates the user and creates CLI context.
- **`az account set`** changes the default subscription only.
- **Tenant selection** controls directory visibility and identity scope.
- **IDE sign-in** can be consumed automatically by Azure SDKs through developer credentials.
- **`DefaultAzureCredential`** is convenient, but its exact precedence can vary by SDK and version.
- **`az account get-access-token`** is useful for token inspection and troubleshooting.
- **Managed identity** is the preferred Azure-hosted option when available.
- **Service principal** is the common non-interactive identity for automation.

**Quick reference commands**

```bash
# Sign in to a specific tenant
az login --tenant 11111111-1111-1111-1111-111111111111

# Show current account context
az account show

# List subscriptions visible to the current identity
az account list --all --output table

# Set the default subscription for future CLI commands
az account set --subscription "My Subscription Name"

# Get an access token for ARM
az account get-access-token --resource https://management.azure.com/

# Clear cached CLI account context
az account clear
```

**Key takeaways**

- Authentication and authorization are different; both matter in Azure development.
- Tenant context determines directory visibility, while subscription context determines which subscription the CLI uses by default.
- `az account set` changes the default subscription, not the authentication tenant itself.
- IDE sign-in can be reused by Azure SDKs through `DefaultAzureCredential` without extra code in many local development scenarios.
- `DefaultAzureCredential` may choose different sources depending on language, package version, and environment.
- When troubleshooting, always verify tenant, subscription, and role assignment together.
