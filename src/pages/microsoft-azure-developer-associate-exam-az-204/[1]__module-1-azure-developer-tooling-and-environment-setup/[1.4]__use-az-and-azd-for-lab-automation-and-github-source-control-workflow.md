## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Overview**

`az` and `azd` solve different parts of the Azure developer workflow.

- `az` is the **Azure CLI**: you use it for direct resource management, scripting, inspection, and one-off automation.
- `azd` is the **Azure Developer CLI**: you use it to scaffold, provision, and deploy application infrastructure and app code in a repeatable developer workflow.
- In lab and exam scenarios, you often combine them:
  - `az` for subscription setup, role assignment, verification, and troubleshooting.
  - `azd` for environment-aware app deployment and repeatable provisioning.

A good mental model is:

- `az` = control plane operations and diagnostics.
- `azd` = app-centric environment lifecycle and deployment orchestration.

**Prerequisites and authentication**

Before you automate anything, make sure the tools and identity context are ready.

- Install the Azure CLI and Azure Developer CLI on the workstation or build agent.
- Sign in with the right Azure identity.
- Confirm the active subscription before provisioning.

Typical setup commands:

```bash
az version
azd version

az login
az account show
az account set --subscription "<subscription-id-or-name>"

azd auth login
azd auth list
```

Key points:

- `az login` authenticates the Azure CLI session.
- `azd auth login` authenticates the Azure Developer CLI session.
- `azd` uses the signed-in Azure context to create and manage resources for the selected environment.
- If you switch tenants or subscriptions, recheck both `az account show` and `azd auth login` context before deploying.

> If your lab uses multiple subscriptions, always verify the active subscription before running provisioning commands. Many deployment failures come from targeting the wrong subscription or tenant.

**When to use `az`**

Use `az` when you need precise control over Azure resources or when a task is not app-scoped.

Common uses:

- Create or inspect resource groups.
- Assign roles and permissions.
- Query resource IDs and outputs.
- Validate deployment state.
- Troubleshoot failed resources.
- Export environment values for scripts.

Examples:

```bash
az group create \
  --name rg-az204-lab \
  --location eastus

az storage account show \
  --name mystorageacct123 \
  --resource-group rg-az204-lab

az role assignment create \
  --assignee "<user-or-app-object-id>" \
  --role "Contributor" \
  --scope "/subscriptions/<sub-id>/resourceGroups/rg-az204-lab"
```

Useful inspection commands:

```bash
az account show
az resource list --resource-group rg-az204-lab -o table
az deployment group list --resource-group rg-az204-lab -o table
az deployment group show --resource-group rg-az204-lab --name main
```

**When to use `azd`**

Use `azd` when you want a repeatable application workflow that includes infrastructure and deployment.

Typical `azd` tasks:

- Initialize a project from an app template or existing app structure.
- Define environment-specific settings.
- Provision Azure resources.
- Deploy application code.
- Run the app locally with cloud-backed dependencies.
- Inspect environment values.

Common commands:

```bash
azd init
azd env new dev
azd env select dev
azd provision
azd deploy
azd up
azd env get-values
azd env list
```

Practical interpretation:

- `azd init` prepares a project for `azd` workflows and may scaffold or detect app structure depending on the template and repository layout.
- `azd env new` creates a named environment.
- `azd env select` switches the active environment.
- `azd provision` creates or updates infrastructure for the selected environment.
- `azd deploy` deploys application code to already provisioned infrastructure.
- `azd up` is a convenience command that typically provisions and deploys in one flow for the selected environment.

**Repository and source control workflow**

A clean GitHub workflow keeps lab work reproducible and prevents accidental commits of secrets or generated state.

Recommended repository layout:

```text
repo/
├─ azure.yaml
├─ infra/
│  ├─ main.bicep
│  ├─ main.parameters.json
│  └─ modules/
├─ src/
│  └─ app/
├─ .gitignore
├─ .github/
│  └─ workflows/
└─ README.md
```

What to commit:

- `azure.yaml`
- Bicep files and reusable modules
- Application source code
- Workflow files under `.github/workflows`
- Documentation and sample configuration

What not to commit:

- Secrets, tokens, and private keys
- Local environment files with sensitive values
- Build outputs and package caches
- Machine-specific files

Typical `.gitignore` entries:

```gitignore
# Node
node_modules/
dist/

# Python
__pycache__/
.venv/

# .NET
bin/
obj/

# Azure Developer CLI
.azure/
.env
.env.*
*.local
```

Notes on environment files:

- `azd` environments often store local metadata under `.azure/`.
- Do not commit sensitive `.env` files.
- If you need shared defaults, commit a safe template such as `.env.sample`.
- Keep secrets in GitHub Secrets, Azure Key Vault, or another secure store.

**`azure.yaml` and project binding**

`azure.yaml` tells `azd` how to map your app to infrastructure and deployment targets.

A minimal example:

```yaml
name: az204-lab
services:
  web:
    project: ./src/web
    language: js
    host: appservice
```

A more realistic example with infrastructure and service mapping:

```yaml
name: az204-lab
metadata:
  template: azd-init
infra:
  path: infra
services:
  api:
    project: ./src/api
    language: dotnet
    host: appservice
  web:
    project: ./src/web
    language: js
    host: appservice
```

Why this matters:

- `infra.path` points `azd` to your infrastructure-as-code files.
- `services` tells `azd` what to build and deploy.
- Many templates need service definitions so `azd` can package and publish the app correctly.
- A minimal file may be enough for a simple lab, but real projects usually need explicit service configuration.

**Infrastructure as code with Bicep**

`azd` commonly works with Bicep for Azure Resource Manager deployments.

Example `infra/main.bicep`:

```bicep
param location string = resourceGroup().location
param appName string
param environmentName string

resource plan 'Microsoft.Web/serverfarms@2022-09-01' = {
  name: '${appName}-${environmentName}-plan'
  location: location
  sku: {
    name: 'B1'
    tier: 'Basic'
  }
}

resource app 'Microsoft.Web/sites@2022-09-01' = {
  name: '${appName}-${environmentName}-web'
  location: location
  properties: {
    serverFarmId: plan.id
    httpsOnly: true
  }
}

output webAppName string = app.name
output webAppUrl string = 'https://${app.name}.azurewebsites.net'
```

How `azd` uses this:

- It passes environment values into the deployment.
- It provisions the resources defined in the Bicep template.
- It can surface outputs for later use by the app or pipeline.

**Environment management**

`azd` environments let you isolate dev, test, and lab deployments.

Common workflow:

```bash
azd env new dev
azd env select dev
azd env set LOCATION eastus
azd env set APP_NAME az204lab
azd env get-values
```

Important behavior:

- `azd env set` stores values for the selected environment.
- `azd env get-values` prints shell-friendly key/value output, not JSON.
- You can use those values in scripts or pipeline steps.
- Environment names should be stable and descriptive, such as `dev`, `test`, or `lab01`.

Example output usage:

```bash
eval "$(azd env get-values)"
echo "$AZURE_LOCATION"
echo "$AZURE_ENV_NAME"
```

**Provisioning vs deployment vs validation**

These are related but distinct operations.

- **Provisioning** creates or updates Azure infrastructure.
- **Deployment** publishes application code to the provisioned target.
- **Validation** checks that the environment is healthy and the app behaves as expected.

Typical command mapping:

```bash
azd provision
azd deploy
azd up
```

Validation examples:

```bash
az deployment group show \
  --resource-group rg-az204-lab \
  --name main

az webapp show \
  --name az204lab-dev-web \
  --resource-group rg-az204-lab

curl -I https://az204lab-dev-web.azurewebsites.net
```

Practical guidance:

- Use `azd provision` when you want to create infrastructure without redeploying code.
- Use `azd deploy` when infrastructure already exists and you only need to publish app changes.
- Use `azd up` when you want a single command for the common lab flow.
- Use `az` validation commands when you need to inspect Azure-side state directly.

**GitHub source control workflow**

A GitHub-based workflow makes lab automation repeatable and reviewable.

Recommended process:

1. Create a repository and add the app source, Bicep files, and `azure.yaml`.
2. Add a `.gitignore` that excludes secrets, build outputs, and local environment state.
3. Commit a safe environment template if needed, such as `.env.sample`.
4. Push to GitHub and protect sensitive values with GitHub Secrets.
5. Use GitHub Actions to provision and deploy on push or manual dispatch.

Branching guidance:

- Use `main` for stable lab-ready code.
- Use feature branches for changes to infrastructure or app code.
- Review Bicep changes carefully because infrastructure drift can be expensive.

**GitHub Actions end-to-end workflow**

For OIDC (OpenID Connect) authentication, use `azure/login@v2` and then run `az` or `azd` commands in the same job.

A working example:

```yaml
name: deploy-azd

on:
  workflow_dispatch:
  push:
    branches:
      - main

permissions:
  id-token: write
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Azure login
        uses: azure/login@v2
        with:
          client-id: ${{ secrets.AZURE_CLIENT_ID }}
          tenant-id: ${{ secrets.AZURE_TENANT_ID }}
          subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}

      - name: Install Azure Developer CLI
        run: |
          curl -fsSL https://aka.ms/install-azd.sh | bash

      - name: Set up azd environment
        run: |
          azd env new github --no-prompt || azd env select github
          azd env set AZURE_LOCATION eastus
          azd env set AZURE_SUBSCRIPTION_ID "${{ secrets.AZURE_SUBSCRIPTION_ID }}"

      - name: Provision infrastructure
        run: azd provision --no-prompt

      - name: Deploy application
        run: azd deploy --no-prompt
```

Why this works:

- `azure/login@v2` handles federated authentication for GitHub Actions.
- `permissions.id-token: write` enables OIDC token exchange.
- `contents: read` allows checkout.
- `azd provision` and `azd deploy` are explicit, which makes the pipeline easier to debug than a single combined command.
- `workflow_dispatch` lets you run the workflow manually during labs.

If you need direct Azure CLI steps in the same workflow, you can add them after login:

```yaml
      - name: Inspect subscription
        run: |
          az account show
          az group list -o table
```

**Federated identity setup for GitHub Actions**

GitHub Actions OIDC avoids storing long-lived client secrets in the repository.

Typical setup flow:

1. Create or identify an app registration or service principal in Microsoft Entra ID.
2. Assign the required Azure role at the subscription, resource group, or resource scope.
3. Add a federated credential that trusts your GitHub repository and branch.
4. Store the client ID, tenant ID, and subscription ID as GitHub Secrets.
5. Use `azure/login@v2` in the workflow.

Important distinction:

- `az ad sp create-for-rbac` creates a service principal and usually includes a client secret.
- That is not the same as configuring OIDC federation.
- For GitHub Actions, you still need a federated credential so GitHub can exchange its OIDC token for Azure access without a secret.

**Example lab automation pattern**

A practical lab often follows this sequence:

1. Create the resource group with `az`.
2. Initialize the app with `azd`.
3. Commit the project to GitHub.
4. Configure GitHub Actions with OIDC.
5. Run `azd provision` and `azd deploy` from the pipeline.
6. Validate the deployed endpoint with `az` or `curl`.

Example shell script:

```bash
#!/usr/bin/env bash
set -euo pipefail

RG="rg-az204-lab"
LOCATION="eastus"
ENV_NAME="dev"

az login
az account set --subscription "<subscription-id>"

az group create --name "$RG" --location "$LOCATION"

azd auth login
azd env new "$ENV_NAME" --no-prompt || azd env select "$ENV_NAME"
azd env set AZURE_LOCATION "$LOCATION"
azd provision --no-prompt
azd deploy --no-prompt
```

**Troubleshooting common failures**

Most lab issues fall into a few predictable categories.

- **Wrong subscription selected**
  - Check `az account show`.
  - Re-run `az account set --subscription "<id>"`.
  - Confirm the same subscription is used in GitHub Secrets and workflow inputs.

- **Missing permissions**
  - Verify the identity has the correct role assignment.
  - Check whether the scope is subscription, resource group, or resource level.
  - Remember that provisioning often needs more than read access.

- **Resource group not found**
  - Confirm the resource group name in the environment and Bicep parameters.
  - Use `az group show --name <rg>` before deploying.

- **Deployment output inspection**
  - Use `az deployment group show` or `az deployment sub show` depending on scope.
  - Review outputs and error messages before rerunning.

- **`azd` environment confusion**
  - Run `azd env list` and `azd env select <name>`.
  - Check whether the selected environment matches the branch or lab scenario.

- **GitHub Actions authentication failure**
  - Confirm `permissions.id-token: write`.
  - Confirm the federated credential matches the repository and branch.
  - Confirm the `client-id`, `tenant-id`, and `subscription-id` secrets are correct.

**Practical comparison**

| Task | Prefer `az` | Prefer `azd` |
|---|---:|---:|
| Create a resource group | Yes | Sometimes |
| Deploy app + infra together | No | Yes |
| Inspect role assignments | Yes | No |
| Manage environment-specific app settings | Sometimes | Yes |
| Troubleshoot deployment state | Yes | Sometimes |
| Scaffold app deployment workflow | No | Yes |

**Exam and lab takeaways**

- Use `az` for direct Azure management and troubleshooting.
- Use `azd` for app-centric provisioning and deployment.
- Authenticate both tools correctly before running automation.
- Keep secrets out of GitHub and use OIDC where possible.
- Commit infrastructure and workflow files, but not local state or credentials.
- Distinguish provisioning, deployment, and validation in your scripts and pipelines.
- Prefer explicit steps in CI/CD so failures are easier to diagnose.
