## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Webhook and event handler fundamentals**

A **webhook** is an HTTP callback that lets a service notify your application when an event occurs. Instead of polling, the provider sends a request to your endpoint, which reduces latency and unnecessary traffic. In Azure, you often implement the receiver as an **Azure Function** because it is lightweight, scalable, and easy to secure.

An **event handler** processes incoming events from a source such as Azure Event Grid, Service Bus, Storage, or a third-party SaaS platform. The core design goal is the same: accept the event quickly, validate it, and hand off any longer work to a durable backend process.

- Webhooks are usually **push-based** and depend on public HTTP endpoints.
- Event handlers may be HTTP-triggered or message-triggered depending on the source.
- Azure Functions are a common choice because they support multiple triggers and scale automatically.
- You should design for **idempotency** so duplicate deliveries do not cause incorrect results.

**When to use webhooks versus polling**

Polling asks the client to repeatedly check for changes. Webhooks invert that model and let the source call you only when something happens. This is usually better when you need near-real-time updates, lower cost, or fewer API calls.

Use webhooks when:
- The source system supports outbound HTTP callbacks.
- You need low-latency notifications.
- You want to avoid rate limits caused by frequent polling.
- You can expose a secure endpoint to the internet.

Use polling when:
- The source cannot send callbacks.
- You need to query state that is not event-driven.
- Firewall or network constraints prevent inbound requests.

**Azure Function trigger options for event handling**

Azure Functions support several trigger types that are relevant to event-driven integration. The trigger determines how the function starts and what payload it receives.

- **HTTP trigger**: Receives webhook calls and custom API requests.
- **Event Grid trigger**: Handles Azure event notifications with built-in retry semantics.
- **Queue trigger**: Processes messages asynchronously from Azure Storage queues.
- **Service Bus trigger**: Handles reliable messaging with sessions, dead-lettering, and duplicate detection.
- **Blob trigger**: Reacts to blob changes, often for file processing workflows.

A webhook receiver is often an HTTP-triggered function, while downstream processing may use queue or Service Bus triggers to decouple the request from the work.

**HTTP-triggered Azure Functions for webhooks**

An HTTP-triggered function is the most direct way to implement a webhook endpoint. The function receives an HTTP request, validates it, and returns a response quickly enough to satisfy the sender’s timeout expectations.

Typical responsibilities include:
- Verifying the request signature or shared secret.
- Parsing the event payload.
- Checking for duplicate event IDs.
- Returning a success status code after the event is accepted.
- Offloading heavy work to another component.

A minimal C# example:

```csharp
using System.IO;
using System.Threading.Tasks;
using Microsoft.AspNetCore.Mvc;
using Microsoft.Azure.WebJobs;
using Microsoft.Azure.WebJobs.Extensions.Http;
using Microsoft.AspNetCore.Http;
using Microsoft.Extensions.Logging;

public static class WebhookHandler
{
    [FunctionName("WebhookHandler")]
    public static async Task<IActionResult> Run(
        [HttpTrigger(AuthorizationLevel.Function, "post", Route = "webhook")] HttpRequest req,
        ILogger log)
    {
        string body = await new StreamReader(req.Body).ReadToEndAsync();

        log.LogInformation("Webhook received.");
        log.LogInformation(body);

        return new OkResult();
    }
}
```

A Node.js example:

```javascript
module.exports = async function (context, req) {
  context.log('Webhook received');

  const event = req.body;
  if (!event) {
    context.res = { status: 400, body: 'Invalid payload' };
    return;
  }

  context.res = { status: 200, body: 'Accepted' };
};
```

**Webhook validation and security**

Webhook endpoints are public by design, so you must assume they will be probed, replayed, or spoofed. Security usually depends on a combination of authentication, signature validation, and network controls.

Common protections:
- Use **HTTPS** only.
- Validate a shared secret, HMAC signature, or provider-issued token.
- Reject requests with invalid timestamps to reduce replay risk.
- Restrict access with IP allowlists when the provider publishes stable ranges.
- Use Azure Functions authentication levels appropriately.

> Do not rely on obscurity or a random URL alone. If the endpoint is public, treat every request as untrusted until verified.

Azure Functions authorization levels matter:
- **Anonymous**: No built-in key required; use only if external validation is strong.
- **Function**: Requires a function key and is common for simple webhook endpoints.
- **Admin**: Reserved for host-level operations and should not be used for webhooks.

Example of validating a shared secret header:

```csharp
string secret = req.Headers["X-Webhook-Secret"];
if (secret != Environment.GetEnvironmentVariable("WEBHOOK_SECRET"))
{
    return new UnauthorizedResult();
}
```

For stronger validation, many providers sign the payload. You compute an HMAC (Hash-based Message Authentication Code) over the raw body and compare it to the signature header. Always compare signatures in constant time when possible.

**Event Grid as an event source**

Azure Event Grid is a managed event routing service that delivers events from Azure services and custom publishers to subscribers. It is commonly used for reactive architectures and is a natural fit for Azure Functions.

Important Event Grid behaviors:
- It uses **push delivery** to subscribers.
- It retries delivery when the endpoint fails.
- It expects a fast acknowledgment.
- It can send a subscription validation event during setup.

When you create an Event Grid subscription to an HTTP endpoint, the endpoint must handle the validation handshake. The service sends a validation event, and your endpoint must echo back the validation code.

Example validation response pattern:

```csharp
if (eventType == "Microsoft.EventGrid.SubscriptionValidationEvent")
{
    var validationCode = data["validationCode"];
    return new OkObjectResult(new
    {
        validationResponse = validationCode
    });
}
```

Event Grid event payloads typically include:
- `id`
- `eventType`
- `subject`
- `eventTime`
- `data`
- `dataVersion`

You should use `id` for deduplication because retries can cause duplicate deliveries.

**Designing for retries and idempotency**

Webhook providers and event systems often retry on transient failures. Your handler must tolerate repeated delivery without creating duplicate side effects.

Idempotency strategies:
- Store processed event IDs in a durable store.
- Use upserts instead of inserts where possible.
- Make downstream operations conditional on state.
- Use unique constraints in the database to reject duplicates safely.

A simple deduplication pattern:
1. Read the event ID.
2. Check whether the ID already exists in a processed-events store.
3. If it exists, return success.
4. If it does not exist, process the event and record the ID.

Example SQL table:

```sql
CREATE TABLE ProcessedEvents (
    EventId NVARCHAR(100) NOT NULL PRIMARY KEY,
    ProcessedAt DATETIME2 NOT NULL DEFAULT SYSUTCDATETIME()
);
```

This pattern matters because a webhook sender may retry after a timeout even if your function already completed the work.

**Decoupling webhook receipt from processing**

A webhook endpoint should usually do the minimum amount of work needed to accept the event. If processing is slow, move the work to a queue or Service Bus topic so the HTTP request can return quickly.

A common architecture:
- HTTP-triggered Azure Function receives the webhook.
- The function validates and enqueues the event.
- A queue-triggered or Service Bus-triggered function performs the heavy processing.
- The processing function writes to databases, calls APIs, or updates storage.

This approach improves:
- **Reliability** by buffering spikes.
- **Scalability** by separating ingestion from processing.
- **Responsiveness** by returning HTTP 200/202 quickly.

Example queue output binding in C#:

```csharp
[FunctionName("WebhookHandler")]
public static async Task<IActionResult> Run(
    [HttpTrigger(AuthorizationLevel.Function, "post")] HttpRequest req,
    [Queue("webhook-events", Connection = "AzureWebJobsStorage")] IAsyncCollector<string> queue,
    ILogger log)
{
    string body = await new StreamReader(req.Body).ReadToEndAsync();
    await queue.AddAsync(body);

    return new AcceptedResult();
}
```

**Queue and Service Bus event handlers**

Storage queues and Service Bus are common downstream handlers for webhook workloads. They let you process events asynchronously and absorb bursts.

Storage Queue characteristics:
- Simple and low-cost.
- Good for basic buffering.
- Limited messaging features compared with Service Bus.

Service Bus characteristics:
- Supports **dead-letter queues** for poison messages.
- Supports sessions for ordered processing.
- Offers duplicate detection and richer delivery semantics.
- Better suited for enterprise integration.

Use a queue trigger when:
- You need simple asynchronous processing.
- Ordering is not critical.
- You want a lightweight buffer.

Use a Service Bus trigger when:
- You need reliable enterprise messaging.
- You need dead-lettering or sessions.
- You want stronger control over delivery semantics.

Example Service Bus trigger:

```csharp
[FunctionName("ProcessWebhookEvent")]
public static void Run(
    [ServiceBusTrigger("webhook-events", Connection = "ServiceBusConnection")] string message,
    ILogger log)
{
    log.LogInformation($"Processing message: {message}");
}
```

**Handling provider verification challenges**

Some webhook providers send a challenge request before enabling the subscription. This proves that you control the endpoint.

Common challenge patterns:
- A query parameter or header contains a challenge token.
- The endpoint must echo the token in the response body.
- The provider expects a specific status code and content type.

General approach:
1. Inspect the incoming request for a validation token.
2. Return the token exactly as required.
3. Use the correct HTTP status code.
4. Complete the handshake before accepting normal events.

Because providers differ, you should read the provider’s webhook contract carefully and implement the exact response format they require.

**Practical Azure Function configuration**

Azure Functions use application settings for secrets, connection strings, and environment-specific values. Store webhook secrets in configuration, not in source code.

Example `local.settings.json`:

```json
{
  "IsEncrypted": false,
  "Values": {
    "AzureWebJobsStorage": "UseDevelopmentStorage=true",
    "FUNCTIONS_WORKER_RUNTIME": "dotnet",
    "WEBHOOK_SECRET": "replace-with-a-secure-value",
    "ServiceBusConnection": "Endpoint=sb://..."
  }
}
```

Useful deployment and testing commands:
```bash
func start
func azure functionapp publish my-function-app
az functionapp config appsettings set \
  --name my-function-app \
  --resource-group my-rg \
  --settings WEBHOOK_SECRET='replace-with-a-secure-value'
```

For local testing, you can send a sample request with `curl`:

```bash
curl -X POST "http://localhost:7071/api/webhook" \
  -H "Content-Type: application/json" \
  -H "X-Webhook-Secret: replace-with-a-secure-value" \
  -d '{"id":"evt-123","type":"order.created"}'
```

**Observability and troubleshooting**

Webhook systems are hard to debug without good telemetry because the sender and receiver are separated by the network. You should log enough information to correlate requests without exposing secrets.

Log:
- Event ID
- Event type
- Correlation ID
- Validation outcome
- Processing result
- Retry count if available

Avoid logging:
- Raw secrets
- Full authorization headers
- Sensitive payload fields unless required and approved

Azure Application Insights is useful for:
- Request tracing
- Exception tracking
- Dependency tracking
- Performance monitoring

A practical troubleshooting checklist:
- Confirm the endpoint is reachable over HTTPS.
- Verify the provider can resolve the public URL.
- Check function authorization level and keys.
- Validate the signature or shared secret.
- Inspect timeout behavior and retry patterns.
- Confirm the validation handshake response format.
- Review Application Insights logs for failures.

**Common HTTP responses and their meaning**

| Status code | Typical use |
|---|---|
| `200 OK` | Event accepted and processed successfully |
| `202 Accepted` | Event accepted for asynchronous processing |
| `400 Bad Request` | Payload is malformed or missing required fields |
| `401 Unauthorized` | Secret, token, or signature validation failed |
| `403 Forbidden` | Request is authenticated but not allowed |
| `500 Internal Server Error` | Unexpected processing failure |

Use `202 Accepted` when you enqueue work and do not finish processing inside the webhook request.

**Security and reliability patterns**

A robust webhook handler usually combines several patterns:
- **Authentication** to verify the sender.
- **Validation** to ensure the payload is well formed.
- **Deduplication** to prevent repeated side effects.
- **Asynchronous processing** to keep the endpoint fast.
- **Dead-letter handling** for poison messages.
- **Monitoring** to detect delivery failures early.

Example architecture flow:
1. Provider sends webhook to HTTP-triggered Azure Function.
2. Function validates signature and event schema.
3. Function stores event ID for deduplication.
4. Function enqueues the payload to Service Bus.
5. Worker function processes the message and updates downstream systems.
6. Failed messages move to the dead-letter queue for investigation.

**Exam-focused distinctions**

For AZ-204, focus on how Azure Functions fit into event-driven solutions rather than memorizing one provider’s webhook format.

Key distinctions:
- **HTTP trigger** handles inbound web requests and webhooks.
- **Event Grid trigger** handles Azure event notifications with validation and retries.
- **Queue/Service Bus triggers** handle asynchronous processing after receipt.
- **Idempotency** is essential because retries are normal.
- **Security** depends on secrets, signatures, HTTPS, and access control.

If a scenario asks for a fast, scalable webhook receiver, choose an HTTP-triggered Azure Function with validation and downstream queueing. If it asks for Azure-native event routing, Event Grid is often the better fit.

**Quick reference**

- Use **HTTP-triggered Azure Functions** for webhook endpoints.
- Validate requests with a secret, token, or signature.
- Return quickly and offload long work to a queue or Service Bus.
- Handle duplicate events with idempotent logic.
- Use Event Grid when the source is Azure-native or supports event subscriptions.
- Monitor with Application Insights and design for retries.
