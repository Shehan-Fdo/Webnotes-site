## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Event Grid overview**

Azure Event Grid is a managed event routing service. You use it to publish discrete events and route them to handlers that react quickly, such as webhooks, Azure Functions, Service Bus, Event Hubs, and other supported endpoints. It is not a general-purpose message broker; it is optimized for lightweight event notification and fan-out.

Event Grid uses a publish-subscribe model:

- A **publisher** sends events to a topic.
- A **subscription** defines where matching events go.
- A **handler** receives the event and performs work.

Use Event Grid when you want near-real-time reactions to state changes, especially for integration between Azure services or your own applications.

**Topic types**

Event Grid has two main topic models:

- **Custom topics**
  - You create them explicitly.
  - Your application publishes events to them.
  - They are useful when you own the event source.
- **System topics**
  - Event Grid associates them with supported Azure resources.
  - They are used when Azure services emit events on your behalf.
  - You typically create the event subscription at the resource scope, and Event Grid manages the system topic association.

A custom topic is the right choice when you control the producer. A system topic is the right choice when an Azure resource, such as a storage account or resource group, is the source of events.

**Creating a custom topic**

You create a custom topic when your application needs to publish events directly to Event Grid. The topic is a first-class Azure resource, so you manage it with the portal, Azure CLI, ARM templates, or Bicep.

Typical CLI flow:

1. Create the topic.
2. Retrieve the topic endpoint and access key.
3. Publish events to the topic endpoint.
4. Create one or more event subscriptions.

Example:

```bash
az eventgrid topic create \
  --name my-custom-topic \
  --resource-group rg-events \
  --location eastus
```

Get the endpoint and keys:

```bash
az eventgrid topic show \
  --name my-custom-topic \
  --resource-group rg-events \
  --query "{endpoint: endpoint, provisioningState: provisioningState}" \
  -o json

az eventgrid topic key list \
  --name my-custom-topic \
  --resource-group rg-events
```

Publish to the topic with the topic access key. Event Grid custom topics commonly use the access key for publisher authentication; in some enterprise scenarios, Azure AD-based authentication may be available depending on the resource and SDK support.

```bash
endpoint="https://my-custom-topic.eastus-1.eventgrid.azure.net/api/events"
key="<topic-access-key>"

curl -X POST "$endpoint" \
  -H "aeg-sas-key: $key" \
  -H "Content-Type: application/json" \
  -d '[
    {
      "id": "1",
      "eventType": "order.created",
      "subject": "orders/1001",
      "eventTime": "2026-04-26T12:00:00Z",
      "data": {
        "orderId": 1001,
        "amount": 49.95
      },
      "dataVersion": "1.0"
    }
  ]'
```

> Use the topic access key for publisher authentication unless your scenario explicitly uses Azure AD or managed identity support. Do not assume SAS token publishing for Event Grid topics.

**Creating subscriptions for custom topics**

A subscription tells Event Grid where to deliver matching events from a topic. For custom topics, the subscription scope is the topic resource itself.

Portal flow:

1. Open the custom topic.
2. Select **+ Event Subscription**.
3. Choose a destination type.
4. Configure filters, retry, and dead-letter settings.
5. Save the subscription.

CLI example:

```bash
az eventgrid event-subscription create \
  --name orders-webhook-sub \
  --source-resource-id "/subscriptions/<sub-id>/resourceGroups/rg-events/providers/Microsoft.EventGrid/topics/my-custom-topic" \
  --endpoint https://contoso.example.com/api/events
```

For custom topics, the source is the topic resource ID. This is different from system-topic subscriptions, where the source is the Azure resource that emits the events.

**Creating subscriptions for system topics**

System topics are used for Azure resource events. You usually create the subscription at the source resource scope, and Event Grid creates or associates the system topic behind the scenes.

Common examples include:

- Storage account events
- Resource group events
- Subscription-level events
- Key Vault events
- Azure subscription or resource lifecycle events, where supported

Example for a storage account:

```bash
az eventgrid event-subscription create \
  --name storage-blob-created \
  --source-resource-id "/subscriptions/<sub-id>/resourceGroups/rg-storage/providers/Microsoft.Storage/storageAccounts/mystorageacct" \
  --included-event-types Microsoft.Storage.BlobCreated \
  --endpoint https://contoso.example.com/api/storage-events
```

In this case, you target the storage account resource ID. Event Grid handles the system topic association for that resource type.

**Subscription validation**

Event Grid validates endpoints before it starts delivering events. Validation behavior depends on the destination type.

For **webhooks**, Event Grid sends a `Microsoft.EventGrid.SubscriptionValidationEvent` to the endpoint. Your endpoint must respond with the validation code in a `validationResponse` payload.

Example webhook validation handler:

```csharp
[HttpPost]
public IActionResult Handle([FromBody] JsonElement[] events)
{
    foreach (var e in events)
    {
        if (e.GetProperty("eventType").GetString() == "Microsoft.EventGrid.SubscriptionValidationEvent")
        {
            var data = e.GetProperty("data");
            var code = data.GetProperty("validationCode").GetString();

            return Ok(new
            {
                validationResponse = code
            });
        }
    }

    // Process normal events here
    return Ok();
}
```

For Azure resource destinations and some managed delivery scenarios, Event Grid may also perform validation or authorization checks before accepting the subscription. Common failures include:

- The endpoint is unreachable from Event Grid.
- The endpoint returns a non-2xx status code.
- The endpoint does not echo the validation response correctly.
- Network rules block the request.
- Authentication is missing or misconfigured.

**Destination types**

Event Grid subscriptions can deliver to several supported handlers. The most exam-relevant destinations are:

- **WebHook**
  - Generic HTTPS endpoint.
  - Requires validation and reachable public or appropriately exposed endpoint.
- **Azure Function**
  - Good for lightweight serverless processing.
  - Often used for event-driven automation.
- **Service Bus**
  - Useful when you need durable messaging and downstream consumers.
- **Event Hubs**
  - Useful for high-throughput streaming and telemetry pipelines.
- **Storage Queue**
  - Supported in some scenarios, but less commonly emphasized than the destinations above.
- **Dead-letter storage**
  - Not a delivery destination for normal processing; it stores undeliverable events in a blob container.

Event Grid is an event router, not a queue. If you need ordered consumption, long retention, or competing consumers, use a messaging service such as Service Bus or Event Hubs downstream.

**Filtering events**

Filtering reduces noise and helps you route only the events you need. Event Grid supports both simple and advanced filtering.

Use **subject filtering** when you can match on the event subject path with prefix or suffix rules. This is useful for hierarchical subjects such as `orders/1001/items/1`.

Use **advanced filters** when you need to filter on event payload fields or metadata such as:

- `eventType`
- `data` properties
- `subject`
- `dataVersion`
- `topic`

Example subject filter:

```bash
az eventgrid event-subscription create \
  --name orders-only \
  --source-resource-id "/subscriptions/<sub-id>/resourceGroups/rg-events/providers/Microsoft.EventGrid/topics/my-custom-topic" \
  --endpoint https://contoso.example.com/api/events \
  --subject-begins-with "orders/" \
  --subject-ends-with "/created"
```

Example advanced filter:

```bash
az eventgrid event-subscription create \
  --name high-value-orders \
  --source-resource-id "/subscriptions/<sub-id>/resourceGroups/rg-events/providers/Microsoft.EventGrid/topics/my-custom-topic" \
  --endpoint https://contoso.example.com/api/events \
  --advanced-filter data.amount NumberGreaterThan 1000
```

Use subject filters for path-like routing and advanced filters for content-based routing.

**Delivery properties**

Event Grid subscriptions include delivery controls that matter in production and on the exam.

Key settings:

- **Max delivery attempts**
  - Limits how many times Event Grid retries delivery before dead-lettering or dropping the event.
- **Event time to live (TTL)**
  - Defines how long Event Grid keeps retrying delivery.
- **Dead-letter destination**
  - A blob container where undeliverable events are stored.
- **Retry policy**
  - Controls retry behavior for transient failures.

A practical dead-letter configuration uses an Azure Blob Storage container:

```json
{
  "deadLetterDestination": {
    "endpointType": "StorageBlob",
    "properties": {
      "resourceId": "/subscriptions/<sub-id>/resourceGroups/rg-storage/providers/Microsoft.Storage/storageAccounts/mystorageacct",
      "blobContainerName": "eventgrid-deadletter"
    }
  }
}
```

> Dead-lettering requires a blob container. Event Grid must be able to write to that container using the configured delivery identity or access model.

**Security and authentication**

Security depends on the publisher and destination.

For publishing:

- Custom topics commonly use the topic access key.
- Some scenarios support Azure AD authentication and managed identity.
- Do not assume SAS tokens are the standard publishing mechanism for Event Grid topics.

For delivery:

- Webhooks often use HTTPS and may require a shared access signature or custom auth pattern implemented by your endpoint.
- Azure Functions can use function keys or managed identity depending on the integration path.
- Service Bus and Event Hubs delivery can use managed identity in supported scenarios.
- Storage-based dead-lettering requires write access to the blob container.

Use the least-privilege model that fits the destination. Managed identity is preferred when the destination and API version support it.

**Network and private access**

Event Grid topics and subscriptions may be affected by network restrictions.

Consider:

- Private endpoints for Event Grid resources where supported.
- Firewall rules on the destination service.
- Public network access settings on the topic or destination.
- DNS resolution for private link endpoints.

If your webhook or downstream service is private, Event Grid must still be able to reach it through a supported network path. A common failure mode is a subscription that validates in one environment but fails in production because the endpoint is not publicly reachable or not exposed through the expected private connectivity pattern.

**Azure portal workflow**

Use the portal when you want to inspect settings visually or troubleshoot quickly.

Custom topic workflow:

1. Create the topic.
2. Open the topic resource.
3. Copy the endpoint and access key.
4. Add an event subscription.
5. Configure filters, retry, and dead-lettering.
6. Test publishing.

System topic workflow:

1. Open the supported Azure resource.
2. Select **Events** or **Event Grid**.
3. Create a subscription from that resource.
4. Choose the destination and filters.
5. Save and validate.

The key exam distinction is scope:

- Custom topic subscriptions are created against the topic resource.
- System topic subscriptions are created against the source Azure resource.

**Azure CLI workflow examples**

Create a custom topic and subscription:

```bash
az eventgrid topic create \
  --name my-custom-topic \
  --resource-group rg-events \
  --location eastus

az eventgrid event-subscription create \
  --name my-topic-webhook \
  --source-resource-id "/subscriptions/<sub-id>/resourceGroups/rg-events/providers/Microsoft.EventGrid/topics/my-custom-topic" \
  --endpoint https://contoso.example.com/api/events
```

Create a storage account event subscription:

```bash
az eventgrid event-subscription create \
  --name blob-created-sub \
  --source-resource-id "/subscriptions/<sub-id>/resourceGroups/rg-storage/providers/Microsoft.Storage/storageAccounts/mystorageacct" \
  --included-event-types Microsoft.Storage.BlobCreated \
  --endpoint https://contoso.example.com/api/blob-events
```

List subscriptions:

```bash
az eventgrid event-subscription list \
  --source-resource-id "/subscriptions/<sub-id>/resourceGroups/rg-events/providers/Microsoft.EventGrid/topics/my-custom-topic" \
  -o table
```

Delete a subscription:

```bash
az eventgrid event-subscription delete \
  --name my-topic-webhook \
  --source-resource-id "/subscriptions/<sub-id>/resourceGroups/rg-events/providers/Microsoft.EventGrid/topics/my-custom-topic"
```

**Bicep example**

Use infrastructure as code when you want repeatable deployments. The exact API version can vary by scenario, so verify the latest supported version for your environment before deployment.

```bicep
param location string = resourceGroup().location
param topicName string = 'my-custom-topic'
param subscriptionName string = 'orders-webhook-sub'
param webhookEndpoint string

resource topic 'Microsoft.EventGrid/topics@2022-06-15' = {
  name: topicName
  location: location
  properties: {}
}

resource topicSub 'Microsoft.EventGrid/topics/eventSubscriptions@2022-06-15' = {
  name: '${topic.name}/${subscriptionName}'
  properties: {
    destination: {
      endpointType: 'WebHook'
      properties: {
        endpointUrl: webhookEndpoint
      }
    }
    filter: {
      subjectBeginsWith: 'orders/'
      subjectEndsWith: '/created'
    }
    retryPolicy: {
      maxDeliveryAttempts: 30
      eventTimeToLiveInMinutes: 1440
    }
  }
}
```

Use Bicep to standardize topic creation, subscription scope, and delivery settings. For production, confirm the destination schema and API version against the current Azure Resource Manager specification.

**Azure Functions integration**

Azure Functions is a common Event Grid handler. Depending on the runtime and extension version, you can bind to Event Grid event types such as `EventGridEvent` or `CloudEvent`.

Example function trigger:

```csharp
using Azure.Messaging.EventGrid;
using Microsoft.Azure.Functions.Worker;
using Microsoft.Extensions.Logging;

public class EventGridHandler
{
    private readonly ILogger _logger;

    public EventGridHandler(ILoggerFactory loggerFactory)
    {
        _logger = loggerFactory.CreateLogger<EventGridHandler>();
    }

    [Function("EventGridHandler")]
    public void Run([EventGridTrigger] EventGridEvent eventGridEvent)
    {
        _logger.LogInformation("Event type: {eventType}", eventGridEvent.EventType);
        _logger.LogInformation("Subject: {subject}", eventGridEvent.Subject);
    }
}
```

Use Azure Functions when you want serverless processing, quick integration, and minimal infrastructure. Verify the binding package and runtime version before choosing the event type model.

**Common exam points**

- **Custom topic**
  - You create it explicitly.
  - Your app publishes events to it.
  - Subscriptions are scoped to the topic resource.
- **System topic**
  - It is associated with a supported Azure resource.
  - Subscriptions are scoped to the source resource.
  - Event Grid manages the topic association.
- **Webhook validation**
  - Return `validationResponse` with the validation code.
  - Do not confuse this with query-parameter-based validation flows from other services.
- **Filtering**
  - Use subject filters for path-like matching.
  - Use advanced filters for payload-based conditions.
- **Reliability**
  - Configure retries, TTL, and dead-lettering.
  - Dead-lettering writes to a blob container.
- **Security**
  - Use access keys for custom topic publishing unless your scenario uses Azure AD.
  - Use managed identity where supported for delivery authorization.

**Troubleshooting checklist**

If events do not arrive, check:

- The subscription is created at the correct scope.
- The endpoint is reachable and returns success.
- Validation completed successfully.
- Filters are not excluding the event.
- The event type matches `includedEventTypes`.
- The dead-letter container exists and is writable.
- Network rules or private endpoint settings are not blocking delivery.
- The publisher is using the correct topic endpoint and authentication method.

**Practical mental model**

Think of Event Grid as a routing layer:

- The **topic** is the ingress point for events.
- The **subscription** is the routing rule.
- The **destination** is the handler.
- The **filter** is the selector.
- The **dead-letter container** is the safety net.

If you remember the scope difference between custom topics and system topics, plus validation and filtering behavior, you cover most of the AZ-204 exam surface for Event Grid topic and subscription management.
