## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Event Grid integration overview**

Azure Event Grid is a managed event routing service that pushes events from publishers to subscribers using a publish-subscribe model. For AZ-204, the key idea is that you use Event Grid when you want near-real-time reactions to changes in Azure resources or application events without polling.

Event Grid is commonly used in two exam scenarios:

- **Containerized services** that need to react to events through an HTTP endpoint or another supported destination.
- **Storage events** that notify you when blobs are created, deleted, or renamed in supported storage accounts.

A useful mental model is:

```text
Publisher -> Event Grid topic/system topic -> Event subscription -> Handler
```

For storage, the publisher is usually an Azure Storage account. For custom application events, the publisher is often your own service or containerized app.

**Core Event Grid concepts**

Event Grid separates the source of an event from the handler that processes it. This decoupling improves scalability and makes it easier to add or replace consumers without changing the producer.

Key terms:

- **Event source**: The service or application that emits events.
- **Topic**: The endpoint where events are sent.
- **System topic**: A topic automatically created for Azure resource events, such as storage events.
- **Custom topic**: A topic you create for your own application events.
- **Event subscription**: The routing rule that sends matching events to a destination.
- **Event handler**: The endpoint or service that receives the event.

Event Grid is optimized for event notification, not for durable message queuing. If you need ordered processing, long retention, or competing consumers, Azure Service Bus or Azure Queue Storage may be a better fit.

**Event schemas you should recognize**

Event Grid supports multiple schemas. For exam purposes, you should know the common structure and what changes between schemas.

- **Event Grid schema**: Native Event Grid event format.
- **CloudEvents 1.0**: A standardized event format used across platforms.

A typical Event Grid schema event includes fields such as:

- `id`
- `eventType`
- `subject`
- `eventTime`
- `data`
- `dataVersion`
- `metadataVersion`

Example:

```json
[
  {
    "id": "8f8c1f1d-9f2a-4c5e-9f0d-1d2b3c4d5e6f",
    "eventType": "Microsoft.Storage.BlobCreated",
    "subject": "/blobServices/default/containers/images/blobs/photo.jpg",
    "eventTime": "2026-04-26T12:34:56.789Z",
    "data": {
      "api": "PutBlob",
      "url": "https://examplestorage.blob.core.windows.net/images/photo.jpg",
      "contentType": "image/jpeg",
      "contentLength": 24567
    },
    "dataVersion": "2",
    "metadataVersion": "1"
  }
]
```

CloudEvents uses different field names, such as `type`, `source`, `subject`, `id`, `time`, and `data`.

> Do not assume every event type includes the same optional fields in the same way. Focus on the schema shape and the event metadata you need for filtering and routing.

**Storage events for AZ-204**

Azure Storage events are one of the most important Event Grid scenarios for the exam. The most common source is a **StorageV2** account with Blob Storage enabled. Event Grid system topics are created for the storage account, and the subscription is attached to that system topic.

Common storage event types to know:

- **BlobCreated**: A blob is uploaded or committed.
- **BlobDeleted**: A blob is deleted.
- **BlobRenamed**: A blob is renamed in supported scenarios.
- **BlobTierChanged**: The access tier changes.
- **DirectoryCreated / DirectoryDeleted**: Relevant in hierarchical namespace scenarios, such as ADLS Gen2.
- **BlobInventoryPolicyCompleted**: Less common, but part of storage event coverage.

For AZ-204, the most frequently tested events are:

- `Microsoft.Storage.BlobCreated`
- `Microsoft.Storage.BlobDeleted`
- `Microsoft.Storage.BlobRenamed`

Storage event subscriptions are typically used when you want to:

- Trigger image processing after upload.
- Start a workflow when a document lands in a container.
- Audit or archive deleted blobs.
- Update search indexes or metadata stores after content changes.

**How storage system topics work**

When you enable Event Grid integration for a storage account, Azure creates or associates a **system topic** for that resource. You do not manually publish storage events; Azure Storage emits them automatically.

Typical flow:

1. Create or select a supported storage account.
2. Create an Event Grid subscription on that storage account.
3. Choose the event types you want, such as `BlobCreated`.
4. Configure the handler destination.
5. Add filters if needed.
6. Test by uploading, deleting, or renaming a blob.

Portal flow:

1. Open the storage account.
2. Select **Events**.
3. Choose **+ Event Subscription**.
4. Pick the event types and endpoint.
5. Save the subscription.

CLI example:

```bash
az eventgrid event-subscription create \
  --name blob-created-sub \
  --source-resource-id /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<storageAccount> \
  --included-event-types Microsoft.Storage.BlobCreated Microsoft.Storage.BlobDeleted \
  --endpoint https://<handler-host>/api/events
```

For storage event subscriptions, the source is the storage account resource ID. The system topic is managed by Azure behind the scenes.

**Filtering and routing**

Event Grid subscriptions support filtering so you only process relevant events. This is important in production because it reduces noise and cost.

Common filter types:

- **Subject filters**
  - `subjectBeginsWith`
  - `subjectEndsWith`
- **Advanced filters**
  - Match on fields inside `data`
  - Match on event metadata such as `eventType`

Example subject filter for a specific container:

```bash
az eventgrid event-subscription create \
  --name images-only \
  --source-resource-id /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<storageAccount> \
  --included-event-types Microsoft.Storage.BlobCreated \
  --subject-begins-with /blobServices/default/containers/images/blobs/ \
  --endpoint https://<handler-host>/api/events
```

Advanced filtering is useful when subject filters are not enough. For example, you may want to route only JPEG uploads or only events where a blob property matches a condition.

Example advanced filter concept:

```json
{
  "advancedFilters": [
    {
      "operatorType": "StringEndsWith",
      "key": "data.url",
      "values": [".jpg", ".jpeg"]
    }
  ]
}
```

Use advanced filters when the routing decision depends on event payload fields rather than just the subject path.

**Containerized service integration**

Event Grid can deliver events to a containerized service if the service exposes a reachable HTTP endpoint that can receive Event Grid notifications. In exam scenarios, this usually means a web API running in a container platform with public ingress enabled.

Common hosting options:

- **Azure Container Apps**
- **Azure App Service for Containers**
- **Azure Kubernetes Service (AKS)** with an ingress controller
- **Azure Container Instances** behind a public endpoint or gateway

A practical containerized handler should:

- Accept `POST` requests.
- Respond quickly with a 2xx status code.
- Handle Event Grid validation requests.
- Process events asynchronously when work is expensive.

Example ASP.NET Core endpoint:

```csharp
app.MapPost("/api/events", async (HttpRequest request) =>
{
    using var reader = new StreamReader(request.Body);
    var body = await reader.ReadToEndAsync();

    var events = JsonSerializer.Deserialize<JsonElement[]>(body);

    foreach (var ev in events)
    {
        var eventType = ev.GetProperty("eventType").GetString();

        if (eventType == "Microsoft.EventGrid.SubscriptionValidationEvent")
        {
            var data = ev.GetProperty("data");
            var validationCode = data.GetProperty("validationCode").GetString();

            return Results.Ok(new
            {
                validationResponse = validationCode
            });
        }

        // Process storage or custom events here
    }

    return Results.Ok();
});
```

Event Grid sends a **subscription validation event** when you create the subscription. Your endpoint must echo back the validation response to prove it can receive events.

**Validation and delivery behavior**

Event Grid uses a handshake for webhook-style endpoints:

- It sends a validation event.
- Your endpoint returns the validation response.
- The subscription becomes active.

After validation, Event Grid delivers events using retries if delivery fails. Your handler should be idempotent because duplicate deliveries can happen.

Important delivery behaviors:

- **At-least-once delivery**: You may receive the same event more than once.
- **Retry policy**: Event Grid retries transient failures.
- **Dead-lettering**: Failed events can be sent to a storage destination for later inspection.

> Event Grid is not a guaranteed exactly-once system. Design your handler to detect duplicates using event `id` or your own business key.

**Dead-lettering and retries**

Dead-lettering helps you capture events that could not be delivered after retries. This is useful for troubleshooting and for replaying missed events.

Typical dead-letter destination:

- An Azure Storage container

Use dead-lettering when:

- The endpoint is temporarily unavailable.
- The payload is malformed.
- You want to inspect failed deliveries later.

Example CLI pattern:

```bash
az eventgrid event-subscription create \
  --name blob-events-sub \
  --source-resource-id /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<storageAccount> \
  --included-event-types Microsoft.Storage.BlobCreated \
  --endpoint https://<handler-host>/api/events \
  --deadletter-endpoint /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<deadletterStorage>/blobServices/default/containers/<deadletterContainer>
```

The exact CLI parameters can vary by Azure CLI version and destination type, but the exam concept is consistent: dead-lettering uses a storage destination, not a generic webhook URL.

**Security and authorization**

Security depends on the destination type.

For Azure-hosted endpoints, common patterns include:

- **Webhook validation** for basic endpoint verification.
- **Shared access signatures (SAS)** or keys for storage-based destinations.
- **Managed identity** or Azure role-based access control (RBAC) where supported by the destination and integration path.
- **Azure Active Directory (Azure AD)**-based authentication for supported Azure services.

For custom webhooks, you often secure the endpoint yourself:

- Require a secret path or token.
- Validate headers or signatures if your architecture adds them.
- Restrict ingress at the platform level when possible.

For containerized services, remember the practical constraint:

- Event Grid webhook delivery expects a reachable endpoint.
- Private-only container endpoints are not a simple default fit for arbitrary webhook delivery.
- If you need private networking, use a supported Azure-native destination or an architecture that exposes a secure ingress path through a gateway, load balancer, or public endpoint with strong authentication.

**Publishing custom events to a topic**

When your own application publishes events, you usually send them to a custom topic endpoint. The publish endpoint is the Event Grid topic endpoint, not an application route.

Example with `curl`:

```bash
curl -X POST "https://<topic-name>.<region>-1.eventgrid.azure.net/api/events" \
  -H "aeg-sas-key: <topic-key>" \
  -H "Content-Type: application/json" \
  -d '[
    {
      "id": "1",
      "eventType": "Contoso.Items.Created",
      "subject": "items/42",
      "eventTime": "2026-04-26T12:00:00Z",
      "data": {
        "itemId": 42,
        "name": "Widget"
      },
      "dataVersion": "1.0"
    }
  ]'
```

Use this pattern when your application is the publisher and Event Grid is the router. For storage events, Azure publishes on your behalf.

**Portal and CLI workflow for exam scenarios**

A common AZ-204 task is to connect a storage account to a containerized API through Event Grid.

1. Deploy the containerized API with a public HTTPS endpoint.
2. Ensure the endpoint accepts `POST` requests at a known route, such as `/api/events`.
3. Create or select a StorageV2 account.
4. Open **Events** on the storage account.
5. Create an event subscription.
6. Choose `BlobCreated` or another relevant event type.
7. Set the endpoint to your containerized API.
8. Add filters for container or file type if needed.
9. Save and validate the subscription.
10. Upload a blob and confirm the handler receives the event.

A simple deployment example for Azure Container Apps might look like:

```bash
az containerapp create \
  --name event-handler \
  --resource-group <rg> \
  --environment <aca-env> \
  --image <registry>/event-handler:latest \
  --ingress external \
  --target-port 8080
```

The important exam point is the ingress setting:

- **External ingress** gives Event Grid a reachable HTTPS endpoint.
- **Internal-only ingress** is not suitable for a standard webhook subscription unless you add a supported public entry point.

**Troubleshooting and diagnostics**

When Event Grid delivery fails, check the following first:

- The endpoint returns a 2xx response quickly.
- The validation handshake succeeds.
- The subscription is targeting the correct resource.
- The event type matches the source.
- Filters are not excluding the event.
- The endpoint URL is correct and uses HTTPS.

Useful diagnostics concepts:

- **Delivery attempts**: How many times Event Grid tried to send the event.
- **Retry policy**: The backoff and retry behavior for transient failures.
- **Dead-lettered events**: Events that could not be delivered after retries.
- **Metrics in Azure Monitor**: Delivery success, failure, and dead-letter counts.

If you are debugging a containerized handler, inspect:

- Application logs
- Container logs
- Ingress logs or gateway logs
- Azure Monitor metrics for the Event Grid subscription

Common failure causes:

- The endpoint does not handle the validation event.
- The handler takes too long to respond.
- The route is wrong, such as using `/api/events` in code but configuring a different path in the subscription.
- The storage account is not a supported type for the event scenario.
- The filter is too restrictive.
- The destination is not publicly reachable when webhook delivery is required.

**Exam decision points**

Use these scenario cues to choose the right design:

- If the requirement is to react to blob uploads, use a **storage system topic** and subscribe to `BlobCreated`.
- If the requirement is to notify a containerized API, expose a **public HTTPS endpoint** or use a supported Azure-native destination.
- If the requirement is to filter by file type or container, use **subject filters** first, then advanced filters if needed.
- If the requirement is to recover from failed deliveries, configure **dead-lettering** to storage.
- If the requirement is to publish application events, use a **custom topic** rather than a storage system topic.

**Quick comparison**

| Scenario | Best fit | Why |
|---|---|---|
| Blob upload triggers processing | Storage system topic | Azure Storage emits the event automatically |
| App publishes business events | Custom topic | You control the event payload and publisher |
| Need to route only `.png` uploads | Subject or advanced filter | Reduces unnecessary handler invocations |
| Need to inspect failed deliveries | Dead-letter storage container | Preserves undelivered events |
| Need a containerized webhook receiver | Public HTTPS ingress | Event Grid must reach the endpoint |

**End-to-end example flow**

```text
Blob uploaded to StorageV2
        ↓
Azure Storage emits Microsoft.Storage.BlobCreated
        ↓
Event Grid system topic receives the event
        ↓
Event subscription filters for /containers/images/
        ↓
Event Grid delivers to containerized API /api/events
        ↓
API validates subscription and processes the event
        ↓
If delivery fails repeatedly, event goes to dead-letter storage
```

**Practical implementation checklist**

- Use a supported storage account type for storage events.
- Subscribe to the correct event types.
- Make the handler idempotent.
- Return validation responses correctly.
- Keep the endpoint reachable over HTTPS.
- Add filters to reduce noise.
- Configure dead-lettering for resilience.
- Monitor delivery metrics and logs.

**Scenario practice**

1. A team uploads product images to Blob Storage and wants a microservice to generate thumbnails.
   - Use a storage system topic.
   - Subscribe to `BlobCreated`.
   - Point the subscription to the containerized thumbnail service.

2. A service running in a container app must receive only PDF uploads from a documents container.
   - Use subject filtering for the container path.
   - Add an advanced filter if you need to match file extension or payload fields.
   - Ensure the container app has external HTTPS ingress.

3. A custom application emits order events and another service processes them.
   - Use a custom Event Grid topic.
   - Publish events with the topic endpoint.
   - Subscribe the processor service to the topic.

**Key takeaways**

- Event Grid is a push-based event routing service.
- Storage events are commonly delivered through system topics tied to StorageV2 accounts.
- `BlobCreated`, `BlobDeleted`, and `BlobRenamed` are the most important storage events to know.
- Containerized services need a reachable HTTPS endpoint for webhook delivery.
- Validation, retries, dead-lettering, and filtering are central to reliable designs.
- For AZ-204, focus on choosing the correct topic type, subscription filter, and delivery destination.
