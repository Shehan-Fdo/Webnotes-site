## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Event-driven delivery concepts**

Azure eventing services differ in how they route, deliver, and retry messages. For AZ-204, focus on three related ideas:

- **Filtering** decides which events reach a subscriber.
- **Delivery semantics** define how many times an event may be delivered and in what order.
- **Dead-lettering** captures events or messages that cannot be processed successfully.

These concepts appear across Azure Event Grid, Azure Event Hubs, and Azure Service Bus, but each service implements them differently. You should know the service-specific behavior rather than assuming one model fits all.

**Azure Event Grid filtering**

Azure Event Grid is a routing service for discrete events. You use filtering to reduce noise and send only relevant events to a handler.

Common filtering approaches include:

- **Subject filtering** on the event subject path.
- **Event type filtering** on `eventType`.
- **Advanced filtering** on event data fields.

Subject filters are useful when your event naming convention encodes hierarchy, such as `/orders/1234/created`. Event type filters are useful when you want only specific event categories, such as `Microsoft.Storage.BlobCreated`.

Advanced filters let you inspect event payload fields. Use them when subject and type are not enough.

Example Azure CLI subscription with subject and type filters:

```bash
az eventgrid event-subscription create \
  --name orders-created-subscription \
  --source-resource-id /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<storage-account> \
  --endpoint https://<function-app>.azurewebsites.net/runtime/webhooks/eventgrid?functionName=<function-name>&code=<function-key> \
  --included-event-types Microsoft.Storage.BlobCreated \
  --subject-begins-with /blobServices/default/containers/orders/ \
  --subject-ends-with .json
```

> For Azure Functions, use the Event Grid trigger endpoint generated for the function trigger. Do not confuse it with a generic webhook URL unless your function is configured to accept Event Grid events correctly.

Advanced filtering is often expressed in ARM templates or SDK calls. The shape depends on the operator, so be precise:

```json
{
  "advancedFilters": [
    {
      "operatorType": "StringBeginsWith",
      "key": "data.contentType",
      "values": ["image/"]
    },
    {
      "operatorType": "NumberGreaterThan",
      "key": "data.size",
      "value": 1048576
    }
  ]
}
```

Use advanced filters when:

- You need to route based on payload content.
- You want to reduce downstream processing cost.
- You need to separate events by business attributes, not just topic or type.

**Event Grid delivery semantics**

Event Grid is designed for at-least-once delivery. That means a subscriber may receive the same event more than once. Your handler must be idempotent.

Important implications:

- **Duplicates can happen** during retries or transient failures.
- **Ordering is not guaranteed** across all events.
- **Delivery is asynchronous** and depends on subscriber availability.

Design handlers to tolerate duplicate delivery by using:

- Event IDs for deduplication.
- Idempotent writes.
- Conditional updates or upserts.
- A processed-events store when strict deduplication matters.

A practical pattern is to store the event ID before applying side effects:

```csharp
if (await processedEvents.ExistsAsync(eventId))
{
    return;
}

await processedEvents.MarkAsync(eventId);
await ApplyBusinessChangeAsync(eventData);
```

This pattern prevents repeated side effects when Event Grid retries the same event.

**Event Grid retries and expiration**

Event Grid retries delivery when the destination returns a failure or times out. Retries are automatic and use a backoff strategy.

Key points:

- **2xx responses** indicate success.
- **Non-2xx responses** trigger retries.
- **Timeouts** also count as delivery failures.
- **Retry duration** is limited by the service’s retry policy and event expiration window.

You should return a success response only after your handler has safely accepted the event. If you need to do longer work, acknowledge quickly and offload processing to a queue or durable workflow.

A common pattern is:

1. Receive the event.
2. Validate and persist it.
3. Return `200 OK`.
4. Process asynchronously from a queue or background worker.

This avoids repeated retries caused by slow handlers.

**Event Grid dead-lettering**

Dead-lettering stores events that could not be delivered after retries are exhausted or when the event expires before successful delivery. Dead-lettering is for inspection and manual reprocessing, not automatic replay.

Use dead-lettering when you need:

- A record of failed deliveries.
- A way to inspect malformed or permanently failing events.
- A manual recovery path for operational support.

Typical dead-letter destinations include Azure Blob Storage containers. Configure a dead-letter endpoint when creating the subscription.

Example CLI pattern:

```bash
az eventgrid event-subscription create \
  --name orders-subscription \
  --source-resource-id /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<storage-account> \
  --endpoint https://<handler-endpoint> \
  --deadletter-endpoint /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<deadletter-storage>/blobServices/default/containers/<container>
```

Operational guidance:

- Monitor the dead-letter container regularly.
- Investigate why delivery failed before reprocessing.
- Reprocess events with your own tooling or script; Event Grid does not automatically replay them from dead-letter storage.

**Azure Service Bus filtering**

Azure Service Bus supports message filtering through subscriptions on topics. This is different from Event Grid because Service Bus is a brokered messaging system, not an event routing service.

Use filters when multiple consumers subscribe to the same topic but each consumer needs a different subset of messages.

Common filter types:

- **SQL filters** for property-based expressions.
- **Correlation filters** for matching message properties efficiently.

SQL filters evaluate user-defined properties and some system properties. Prefer user properties for application logic.

Example SQL filter:

```sql
priority = 'high' AND amount > 1000
```

If you need a system property, use the correct system property name for the SDK and service context. Do not assume older label terminology applies in every scenario.

Example topic subscription with a SQL filter:

```csharp
var adminClient = new ServiceBusAdministrationClient(connectionString);

var rule = new CreateRuleOptions(
    "high-value-orders",
    new SqlRuleFilter("priority = 'high' AND amount > 1000"));

await adminClient.CreateSubscriptionRuleAsync(
    topicName: "orders",
    subscriptionName: "finance",
    rule);
```

Correlation filters are useful when messages share a known set of properties, such as `CorrelationId`, `Subject`, or custom application properties.

**Service Bus delivery semantics**

Service Bus supports both at-most-once and at-least-once style processing depending on how you receive and complete messages.

Important concepts:

- **Peek-lock mode** is the default safe processing model.
- The message is locked while you process it.
- You must explicitly complete, abandon, defer, or dead-letter it.
- If the lock expires or processing fails, the message becomes available again.

This gives you reliable processing with the possibility of duplicates, so your consumer should still be idempotent.

Typical receive-complete flow:

```csharp
await foreach (ServiceBusReceivedMessage message in receiver.ReceiveMessagesAsync())
{
    try
    {
        await ProcessAsync(message);
        await receiver.CompleteMessageAsync(message);
    }
    catch
    {
        await receiver.AbandonMessageAsync(message);
    }
}
```

Use `CompleteMessageAsync` only after successful processing. If processing fails and you want another attempt, abandon the message. If the message is invalid or poison, dead-letter it.

**Service Bus retries and lock handling**

Retries in Service Bus happen at two levels:

- **Client retries** for transient transport or service errors.
- **Message redelivery** when a locked message is not completed before the lock expires or is explicitly abandoned.

For client-side retries, configure retry options on the client. The exact property names vary by SDK version, so align your code with the package version used in your project.

Example configuration pattern:

```csharp
var clientOptions = new ServiceBusClientOptions
{
    RetryOptions = new ServiceBusRetryOptions
    {
        Mode = ServiceBusRetryMode.Exponential,
        MaxRetries = 5,
        Delay = TimeSpan.FromSeconds(0.5),
        MaxDelay = TimeSpan.FromSeconds(10)
    }
};

var client = new ServiceBusClient(connectionString, clientOptions);
```

Use retries for transient failures such as:

- Temporary network issues.
- Service throttling.
- Short-lived dependency outages.

Do not rely on retries to fix permanent validation errors. For those, dead-letter the message or reject it in your application logic.

**Service Bus dead-lettering**

Service Bus dead-letter queues (DLQs) are built into queues and subscriptions. A dead-lettered message is moved to the DLQ for later inspection and handling.

Common broker-generated dead-letter reasons include:

- **Max delivery count exceeded**
- **Time-to-live (TTL) expired**
- **Session-related failures** in session-enabled entities
- **Filter evaluation or rule processing failures** in some subscription scenarios

Application-generated dead-lettering is also common when your code detects a poison message or invalid payload.

Example:

```csharp
try
{
    await ProcessAsync(message);
    await receiver.CompleteMessageAsync(message);
}
catch (JsonException ex)
{
    await receiver.DeadLetterMessageAsync(
        message,
        deadLetterReason: "InvalidPayload",
        deadLetterErrorDescription: ex.Message);
}
```

Use the DLQ when:

- The message cannot be processed after retries.
- The payload is malformed.
- Business rules reject the message permanently.
- You need to preserve the message for later analysis.

Inspect DLQ messages with a receiver targeting the dead-letter subqueue:

```csharp
var deadLetterReceiver = client.CreateReceiver(
    queueName: "orders",
    new ServiceBusReceiverOptions
    {
        SubQueue = SubQueue.DeadLetter
    });
```

**Event Hubs delivery semantics**

Azure Event Hubs is a high-throughput event ingestion service. It is not a broker with per-message acknowledgments like Service Bus. Instead, consumers read events from partitions and track progress with checkpoints.

Key ideas:

- **Partitions** distribute load and preserve order within a partition.
- **Consumer groups** let multiple applications read the same stream independently.
- **Checkpoints** record the last successfully processed event.

Event Hubs processing is typically at-least-once. If you restart a processor before checkpointing, you may see events again.

Use the modern processor client pattern with blob storage for checkpointing:

```csharp
using Azure.Messaging.EventHubs;
using Azure.Messaging.EventHubs.Consumer;
using Azure.Messaging.EventHubs.Processor;
using Azure.Storage.Blobs;

var blobContainerClient = new BlobContainerClient(
    storageConnectionString,
    "checkpoints");

var processor = new EventProcessorClient(
    blobContainerClient,
    EventHubConsumerClient.DefaultConsumerGroupName,
    eventHubConnectionString,
    eventHubName);

processor.ProcessEventAsync += async args =>
{
    Console.WriteLine($"Partition: {args.Partition.PartitionId}, Sequence: {args.Data.SequenceNumber}");

    await ProcessAsync(args.Data);

    await args.UpdateCheckpointAsync();
};

processor.ProcessErrorAsync += args =>
{
    Console.WriteLine($"Error on partition {args.PartitionId}: {args.Exception.Message}");
    return Task.CompletedTask;
};

await processor.StartProcessingAsync();
```

Checkpoint after successful processing, not before. If you checkpoint too early, you can lose events during a crash.

**Event Hubs retries and resilience**

Event Hubs clients also use retry policies for transient failures. Use retries for:

- Temporary connectivity issues.
- Service throttling.
- Short-lived authentication or transport interruptions.

For producers, batch events to improve throughput and reduce cost:

```csharp
await using var producer = new EventHubProducerClient(eventHubConnectionString, eventHubName);

using EventDataBatch batch = await producer.CreateBatchAsync();
batch.TryAdd(new EventData(BinaryData.FromString("{\"id\":1,\"type\":\"created\"}")));
batch.TryAdd(new EventData(BinaryData.FromString("{\"id\":2,\"type\":\"created\"}")));

await producer.SendAsync(batch);
```

Batching does not change delivery semantics, but it improves efficiency. Consumers still need checkpointing and idempotent processing.

**Choosing the right service behavior**

Use the service that matches the problem:

| Scenario | Best fit | Why |
|---|---|---|
| Route a business event to multiple handlers | Event Grid | Lightweight event distribution with filtering |
| Process commands or work items reliably | Service Bus | Brokered messaging with locks, DLQ, and retries |
| Ingest telemetry or streaming data | Event Hubs | High-throughput streaming with partitions and checkpoints |

A useful rule of thumb:

- Choose **Event Grid** for event notification.
- Choose **Service Bus** for reliable work dispatch.
- Choose **Event Hubs** for streaming analytics and telemetry.

**Practical design patterns**

Use these patterns to make delivery behavior predictable:

- **Idempotent consumer**
  - Store processed event or message IDs.
  - Make writes safe to repeat.
- **Queue-backed handler**
  - Accept the event quickly.
  - Push long-running work to Service Bus or another queue.
- **Poison-message isolation**
  - Dead-letter invalid messages.
  - Alert on DLQ growth.
- **Checkpoint discipline**
  - In Event Hubs, checkpoint only after successful processing.
- **Filter first**
  - Reduce downstream load by filtering at the source.

**Common exam traps**

Watch for these distinctions:

- Event Grid dead-lettering is not automatic replay.
- Event Grid delivery is at-least-once, so duplicates are possible.
- Service Bus dead-lettering is different from abandoning a message.
- Service Bus SQL filters should use current property names and valid expressions.
- Event Hubs uses checkpoints, not message completion.
- Event Hubs processors must checkpoint after successful work, not before.

**Quick reference**

- **Event Grid**
  - Filters events by subject, type, or payload.
  - Retries failed deliveries.
  - Dead-letters undeliverable events to storage.
- **Service Bus**
  - Uses subscriptions, SQL filters, and correlation filters.
  - Supports peek-lock, completion, abandon, defer, and dead-letter.
  - Uses DLQ for poison or expired messages.
- **Event Hubs**
  - Uses partitions, consumer groups, and checkpoints.
  - Supports at-least-once processing.
  - Requires idempotent consumers for safe reprocessing.
