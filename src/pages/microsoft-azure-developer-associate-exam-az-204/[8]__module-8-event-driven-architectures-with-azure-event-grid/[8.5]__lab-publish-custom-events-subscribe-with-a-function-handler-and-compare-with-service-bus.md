## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Lab goal and architecture**

This lab shows how to publish custom application events to **Azure Event Grid**, handle them with an **Azure Function**, and compare that event-driven flow with **Azure Service Bus**. The core exam idea is choosing the right messaging service for the job:

- **Event Grid** is optimized for lightweight event notification and fan-out.
- **Service Bus** is optimized for reliable command/message delivery with richer broker features.
- A **Function trigger** is a practical way to validate event delivery end to end.

A typical lab topology looks like this:

```text
Publisher app -> Event Grid custom topic -> Event Grid subscription -> Azure Function
Publisher app -> Service Bus queue/topic -> Azure Function or worker
```

**Prerequisites and local tooling**

Before you start, make sure you can create and test the resources end to end.

- An Azure subscription with permission to create resource groups, Function Apps, Event Grid topics, and Service Bus namespaces.
- Azure CLI installed and signed in with `az login`.
- Azure Functions Core Tools for local function development and testing.
- A supported .NET SDK if you build the function in C#.
- Visual Studio, Visual Studio Code, or another editor with Azure Functions support.
- The correct NuGet packages for your runtime:
  - `Microsoft.Azure.WebJobs.Extensions.EventGrid` for Event Grid-triggered functions in the in-process model.
  - `Microsoft.Azure.WebJobs.Extensions.ServiceBus` for Service Bus triggers in the in-process model.
  - For isolated worker apps, use the corresponding `Microsoft.Azure.Functions.Worker.Extensions.*` packages.

> Be careful about runtime compatibility. Azure Functions in-process and isolated worker models use different extension packages and startup patterns.

**Create the Function App first**

A common lab gap is creating the trigger code before the Function App exists. In practice, you need the hosting resources and app settings in place first.

1. Create a resource group.
2. Create a storage account for the Function App.
3. Create the Function App with a runtime and hosting plan.
4. Add application settings for any connection strings or managed identity configuration.
5. Deploy or run the function locally and verify it starts cleanly.

Example Azure CLI setup for a .NET isolated Function App:

```bash
az group create \
  --name rg-az204-events \
  --location eastus

az storage account create \
  --name staz204events001 \
  --resource-group rg-az204-events \
  --location eastus \
  --sku Standard_LRS

az functionapp create \
  --name func-az204-events-001 \
  --resource-group rg-az204-events \
  --storage-account staz204events001 \
  --consumption-plan-location eastus \
  --runtime dotnet-isolated \
  --functions-version 4
```

Add settings for local or cloud use:

```bash
az functionapp config appsettings set \
  --name func-az204-events-001 \
  --resource-group rg-az204-events \
  --settings \
  AzureWebJobsStorage="<storage-connection-string>" \
  FUNCTIONS_WORKER_RUNTIME="dotnet-isolated"
```

**Azure Event Grid custom topics**

An **Event Grid custom topic** is your application’s publish endpoint for custom events. You send event notifications to the topic, and Event Grid routes them to one or more subscriptions.

Key points:

- The publisher sends events to the topic endpoint returned by Azure.
- The topic uses the Event Grid event model unless you explicitly design for another supported schema.
- Subscribers filter and react to events; they do not poll for them.
- Event Grid is best for near-real-time notification, not durable command processing.

Create a custom topic:

```bash
az eventgrid topic create \
  --name eg-topic-az204 \
  --resource-group rg-az204-events \
  --location eastus
```

Get the topic endpoint and access keys:

```bash
az eventgrid topic show \
  --name eg-topic-az204 \
  --resource-group rg-az204-events \
  --query endpoint -o tsv

az eventgrid topic key list \
  --name eg-topic-az204 \
  --resource-group rg-az204-events
```

**Publishing custom events**

The Event Grid topic endpoint is the publish URL. Do not append extra path segments such as `/api/events`.

A custom event payload typically includes:

- `id`: unique event identifier
- `eventType`: application-defined event name
- `subject`: logical resource path or category
- `eventTime`: timestamp
- `data`: event body
- `dataVersion`: schema version for your payload

Example publish request with the native Event Grid schema:

```bash
endpoint=$(az eventgrid topic show \
  --name eg-topic-az204 \
  --resource-group rg-az204-events \
  --query endpoint -o tsv)

key=$(az eventgrid topic key list \
  --name eg-topic-az204 \
  --resource-group rg-az204-events \
  --query key1 -o tsv)

curl -X POST "$endpoint" \
  -H "aeg-sas-key: $key" \
  -H "Content-Type: application/json" \
  -d '[
    {
      "id": "b3f2d7d0-7d5a-4b1f-9d6d-1f3f7b8a1001",
      "eventType": "order.created",
      "subject": "orders/1001",
      "eventTime": "2026-04-26T12:00:00Z",
      "data": {
        "orderId": 1001,
        "customerId": 42,
        "amount": 149.95
      },
      "dataVersion": "1.0"
    }
  ]'
```

If you use CloudEvents, make sure the publisher and subscriber are aligned on the schema and the function binding expects the same event shape. The exam often tests whether you understand that schema choice is part of the contract.

**Create the Function subscriber**

For Event Grid, the function can be triggered by Event Grid events or by an HTTP endpoint that Event Grid calls during subscription validation and event delivery. In the portal, the Event Grid subscription can target a Function App directly, which simplifies the handshake.

Example C# function using the Event Grid trigger:

```csharp
using Azure.Messaging.EventGrid;
using Microsoft.Azure.Functions.Worker;
using Microsoft.Extensions.Logging;

public class OrderCreatedHandler
{
    private readonly ILogger<OrderCreatedHandler> _logger;

    public OrderCreatedHandler(ILogger<OrderCreatedHandler> logger)
    {
        _logger = logger;
    }

    [Function("OrderCreatedHandler")]
    public void Run([EventGridTrigger] EventGridEvent eventGridEvent)
    {
        _logger.LogInformation("Event type: {eventType}", eventGridEvent.EventType);
        _logger.LogInformation("Subject: {subject}", eventGridEvent.Subject);
        _logger.LogInformation("Data: {data}", eventGridEvent.Data?.ToString());
    }
}
```

If you are using the in-process model, the attribute and package differ, but the idea is the same: the function receives the event and processes it immediately.

**Create the Event Grid subscription**

You can create the subscription in the portal or with CLI. The important part is selecting the Function App as the endpoint and letting Event Grid validate the destination.

Portal flow:

1. Open the custom topic.
2. Select **+ Event Subscription**.
3. Enter a subscription name.
4. Choose the event types you want to deliver.
5. Select the endpoint type as **Azure Function**.
6. Pick the Function App and function.
7. Save the subscription.

What happens next:

- Event Grid sends a validation event to confirm the endpoint.
- The Function endpoint responds automatically when configured correctly.
- After validation, Event Grid begins delivering matching events.

CLI example for a Function endpoint:

```bash
az eventgrid event-subscription create \
  --name sub-orders-created \
  --source-resource-id $(az eventgrid topic show \
    --name eg-topic-az204 \
    --resource-group rg-az204-events \
    --query id -o tsv) \
  --endpoint-type azurefunction \
  --endpoint "/subscriptions/<sub-id>/resourceGroups/rg-az204-events/providers/Microsoft.Web/sites/func-az204-events-001/functions/OrderCreatedHandler"
```

**What to observe during validation and delivery**

When the subscription is created successfully, you should see:

- The subscription status becomes active.
- The Function App logs show a validation request and then event invocations.
- A published event appears in the function logs almost immediately.
- If the event filter does not match, the function is not invoked.

Useful verification steps:

1. Check the Event Grid subscription status in the portal.
2. Open the Function App log stream or Application Insights.
3. Publish a test event.
4. Confirm the function invocation count increases.
5. Confirm the payload fields match what you sent.

**Troubleshooting Event Grid delivery**

Common issues are usually configuration or schema mismatches.

- **Wrong endpoint URL**: the topic endpoint is already the publish URL; do not append `/api/events`.
- **Wrong access key**: use `az eventgrid topic key list` and confirm you are using the correct key.
- **Schema mismatch**: the publisher and subscriber must agree on the event format.
- **Filter mismatch**: the subscription may be filtering out your event type or subject.
- **Function not reachable**: the Function App must be deployed, running, and correctly configured.

A quick test for the function endpoint is to inspect logs after publishing:

```bash
az functionapp log tail \
  --name func-az204-events-001 \
  --resource-group rg-az204-events
```

**Azure Service Bus comparison**

**Azure Service Bus** is a message broker for commands and business messages that need stronger delivery guarantees and richer control. It supports queues and topics/subscriptions, dead-lettering, duplicate detection, sessions, scheduled delivery, and peek-lock processing.

Use Service Bus when you need:

- Durable message storage until a consumer processes the message.
- Explicit completion, abandonment, or dead-letter handling.
- Competing consumers or pub/sub with brokered subscriptions.
- More control over retries and poison-message handling.

Use Event Grid when you need:

- Lightweight event notification.
- Push-based fan-out to multiple handlers.
- Low-latency integration with Azure services and serverless handlers.
- Minimal broker-side workflow.

Comparison table:

| Capability | Event Grid | Service Bus |
|---|---|---|
| Primary model | Event notification | Message broker |
| Delivery style | Push | Pull/receive |
| Typical payload | Small event metadata + data | Business command or message |
| Ordering | Not guaranteed | Not guaranteed by default; use sessions or carefully constrained processing when ordering matters |
| Retry behavior | Automatic retries for delivery attempts | Receiver controls completion, abandon, dead-letter, and lock renewal |
| Dead-lettering | Limited to subscription failure scenarios | First-class dead-letter queue support |
| Fan-out | Native via subscriptions | Topics/subscriptions support pub/sub |
| Best fit | Reactive integration | Reliable workflow/message processing |

> Do not assume Service Bus preserves order automatically. If ordering matters, design for sessions or a single ordered processing path.

**Service Bus queue example**

For a queue-based function trigger, the attribute targets a queue only. If you use a topic subscription, you must specify both the topic and subscription names.

Queue trigger example:

```csharp
using Microsoft.Azure.Functions.Worker;
using Microsoft.Extensions.Logging;

public class OrderQueueHandler
{
    private readonly ILogger<OrderQueueHandler> _logger;

    public OrderQueueHandler(ILogger<OrderQueueHandler> logger)
    {
        _logger = logger;
    }

    [Function("OrderQueueHandler")]
    public void Run([ServiceBusTrigger("orders", Connection = "ServiceBusConnection")] string message)
    {
        _logger.LogInformation("Received queue message: {message}", message);
    }
}
```

Topic subscription trigger example:

```csharp
[Function("OrderTopicHandler")]
public void Run(
    [ServiceBusTrigger("orders", "web-subscription", Connection = "ServiceBusConnection")] string message)
{
    _logger.LogInformation("Received topic message: {message}", message);
}
```

**Service Bus receive, complete, abandon, and dead-letter**

In peek-lock mode, the receiver locks the message, processes it, and then decides what to do next.

- **Complete**: remove the message from the queue or subscription.
- **Abandon**: release the lock so the message can be retried.
- **Dead-letter**: move the message to the dead-letter queue when it cannot be processed.

Conceptual flow:

```text
Receive -> Process -> Complete
                 -> Abandon -> Retry
                 -> Dead-letter -> Investigate
```

Example using the SDK with explicit settlement:

```csharp
using Azure.Messaging.ServiceBus;

ServiceBusClient client = new ServiceBusClient("<connection-string>");
ServiceBusReceiver receiver = client.CreateReceiver("orders");

ServiceBusReceivedMessage message = await receiver.ReceiveMessageAsync();

try
{
    // Process message
    Console.WriteLine(message.Body.ToString());

    await receiver.CompleteMessageAsync(message);
}
catch
{
    await receiver.AbandonMessageAsync(message);
}
```

If processing repeatedly fails, the message can eventually be dead-lettered based on max delivery count. That behavior is important in production because it prevents poison messages from blocking the queue indefinitely.

**Practical lab walkthrough**

A good exam-style lab flow is:

1. Create the resource group and storage account.
2. Create the Function App with the correct runtime.
3. Create the Event Grid custom topic.
4. Deploy the Event Grid-triggered function.
5. Create the Event Grid subscription to the Function endpoint.
6. Publish a custom event to the topic endpoint.
7. Confirm the function receives and logs the event.
8. Create a Service Bus queue or topic.
9. Add a Service Bus-triggered function.
10. Send a message and compare delivery, retry, and settlement behavior.

What you should observe:

- Event Grid validates the subscription before delivery begins.
- The function receives the event almost immediately after publish.
- Service Bus messages remain available until the consumer settles them.
- A failed Service Bus message can be retried or dead-lettered, while Event Grid focuses on event delivery attempts rather than consumer-managed settlement.

**Common AZ-204 pitfalls**

- Confusing the Event Grid topic endpoint with a REST API path.
- Using the wrong CLI command for topic keys.
- Assuming Event Grid and Service Bus are interchangeable.
- Forgetting that Service Bus queue triggers and topic subscription triggers use different attribute parameters.
- Treating ordering as a default guarantee in Service Bus.
- Ignoring schema alignment between publisher and subscriber.
- Skipping Function App creation details such as storage, runtime, and app settings.

**Exam-oriented takeaways**

- **Event Grid** is for event notification and reactive integration.
- **Service Bus** is for brokered messaging and controlled processing.
- A Function App is a common subscriber for both services.
- Validation, endpoint configuration, and schema matching are frequent exam traps.
- Dead-lettering and peek-lock behavior are key Service Bus concepts.
- Event Grid subscriptions are push-based and lightweight; Service Bus consumers explicitly settle messages.
