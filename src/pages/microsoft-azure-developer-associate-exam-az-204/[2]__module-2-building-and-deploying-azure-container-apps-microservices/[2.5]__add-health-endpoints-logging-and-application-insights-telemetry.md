## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Overview**

In Azure-hosted ASP.NET Core applications, health endpoints, logging, and Application Insights work together to help you detect failures, diagnose issues, and understand application behavior in production. Health checks expose a lightweight status endpoint for probes and operators. Logging captures application events with context. Application Insights ingests telemetry so you can query, alert, and correlate requests, dependencies, exceptions, and custom events.

A good implementation separates concerns:

- **Health endpoints** answer whether the app and its dependencies are ready or alive.
- **Logging** records what happened and why.
- **Telemetry** in Application Insights helps you analyze trends, failures, and performance over time.

**Health checks in ASP.NET Core**

ASP.NET Core health checks use the `Microsoft.Extensions.Diagnostics.HealthChecks` abstractions and the `AddHealthChecks()` registration pattern. You typically expose one or more endpoints with `MapHealthChecks()` and optionally filter checks by tags.

A basic setup looks like this:

```csharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddHealthChecks()
    .AddCheck("self", () => HealthCheckResult.Healthy("Application is running"));

var app = builder.Build();

app.MapHealthChecks("/health");

app.Run();
```

This creates a simple endpoint that returns a health response. In practice, you often add checks for dependencies such as SQL Server, Redis, or external APIs.

- **Liveness** indicates the process is running and can respond.
- **Readiness** indicates the app can serve traffic and its dependencies are available.
- **Startup** checks can help during warm-up or initialization phases.

> `MapHealthChecks("/health")` does not automatically create separate liveness and readiness semantics. You define that behavior by tagging checks and mapping different endpoints or filters.

A more realistic pattern uses tags:

```csharp
builder.Services.AddHealthChecks()
    .AddCheck("self", () => HealthCheckResult.Healthy(), tags: new[] { "live", "ready" })
    .AddSqlServer(
        connectionString: builder.Configuration.GetConnectionString("SqlConnection")!,
        name: "sql",
        tags: new[] { "ready" });
```

Then map endpoints with predicates:

```csharp
app.MapHealthChecks("/health/live", new HealthCheckOptions
{
    Predicate = check => check.Tags.Contains("live")
});

app.MapHealthChecks("/health/ready", new HealthCheckOptions
{
    Predicate = check => check.Tags.Contains("ready")
});
```

This lets you keep the liveness probe simple while readiness can fail if a dependency is unavailable.

**Dependency checks and practical considerations**

Health checks are most useful when they reflect real service dependencies. Common checks include database connectivity, cache availability, and external service reachability. Keep them fast and deterministic.

- Use health checks for **availability signals**, not deep diagnostics.
- Avoid expensive queries or long-running calls in the endpoint.
- Prefer short timeouts so probes do not hang.
- Cache or throttle checks if the dependency is costly to test.

For SQL Server, the exact API can vary by package version, but the common pattern is to register a connection string and let the health check validate connectivity. Do not assume a custom `healthQuery` parameter exists in every overload.

Example with configuration:

```json
{
  "ConnectionStrings": {
    "SqlConnection": "Server=tcp:myserver.database.windows.net,1433;Initial Catalog=mydb;Persist Security Info=False;User ID=myuser;Password=mypassword;Encrypt=True;TrustServerCertificate=False;"
  }
}
```

And registration:

```csharp
builder.Services.AddHealthChecks()
    .AddSqlServer(builder.Configuration.GetConnectionString("SqlConnection")!);
```

If you need custom behavior, verify the package version and overloads before copying code into production.

**Logging in ASP.NET Core**

ASP.NET Core uses the built-in logging abstractions in `Microsoft.Extensions.Logging`. You write logs through `ILogger<T>` and configure providers and log levels through configuration.

Example usage:

```csharp
public class OrdersController : ControllerBase
{
    private readonly ILogger<OrdersController> _logger;

    public OrdersController(ILogger<OrdersController> logger)
    {
        _logger = logger;
    }

    [HttpPost]
    public IActionResult Create(OrderDto order)
    {
        _logger.LogInformation("Creating order for customer {CustomerId}", order.CustomerId);
        return Ok();
    }
}
```

Use structured logging with message templates:

- **Placeholders** such as `{CustomerId}` create searchable fields.
- Avoid string concatenation when you want queryable telemetry.
- Log at the right level so production noise stays manageable.

Common log levels:

| Level | Use case |
|---|---|
| Trace | Very detailed diagnostic flow |
| Debug | Development diagnostics |
| Information | Normal business events |
| Warning | Unexpected but handled conditions |
| Error | Failed operations that affect a request or job |
| Critical | System-level failures requiring immediate attention |

Configuration example:

```json
{
  "Logging": {
    "LogLevel": {
      "Default": "Information",
      "Microsoft.AspNetCore": "Warning",
      "MyApp": "Debug"
    }
  }
}
```

This reduces framework noise while keeping your application logs useful.

**Application Insights fundamentals**

Application Insights is Azure’s application performance monitoring service. It collects telemetry such as requests, dependencies, exceptions, traces, and custom events. In ASP.NET Core, telemetry is often collected automatically through the Application Insights SDK and logging integration.

Important points:

- **ILogger** is the primary way to write application logs.
- Application Insights can ingest those logs through its logging provider.
- **TelemetryClient** is used for explicit custom telemetry, not as the general entry point for all telemetry.
- Auto-collection can capture requests, dependencies, exceptions, and performance data depending on runtime and hosting model.

A typical setup uses a connection string. The SDK commonly reads one of these binding sources:

- `APPLICATIONINSIGHTS_CONNECTION_STRING`
- `ApplicationInsights:ConnectionString`

Example `appsettings.json` pattern:

```json
{
  "ApplicationInsights": {
    "ConnectionString": "InstrumentationKey=00000000-0000-0000-0000-000000000000;IngestionEndpoint=https://westus-0.in.applicationinsights.azure.com/"
  }
}
```

Environment variable example:

```bash
export APPLICATIONINSIGHTS_CONNECTION_STRING="InstrumentationKey=00000000-0000-0000-0000-000000000000;IngestionEndpoint=https://westus-0.in.applicationinsights.azure.com/"
```

Use the exact binding key your hosting model expects. A custom JSON section is fine when your code binds it explicitly, but the SDK does not universally require that shape.

**Adding Application Insights to ASP.NET Core**

For modern ASP.NET Core apps, you typically add the Application Insights SDK package and configure it in `Program.cs`.

Example:

```csharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddApplicationInsightsTelemetry();

var app = builder.Build();
app.MapGet("/", () => "Hello");
app.Run();
```

If you want to send custom telemetry, inject `TelemetryClient` explicitly:

```csharp
public class CheckoutService
{
    private readonly TelemetryClient _telemetryClient;

    public CheckoutService(TelemetryClient telemetryClient)
    {
        _telemetryClient = telemetryClient;
    }

    public void RecordCheckout(string orderId)
    {
        _telemetryClient.TrackEvent("CheckoutCompleted", new Dictionary<string, string>
        {
            ["OrderId"] = orderId
        });
    }
}
```

Use this for custom events, metrics, and traces that are not already captured by auto-collection or logging.

**Telemetry types and when to use them**

Application Insights supports several telemetry categories. Choose the right one based on the signal you need.

- **Requests**: incoming HTTP requests and response codes.
- **Dependencies**: outbound calls to SQL, HTTP APIs, queues, and other services.
- **Exceptions**: unhandled or explicitly tracked failures.
- **Traces**: log-style diagnostic messages.
- **Events**: business events such as checkout completed.
- **Metrics**: numeric measurements such as queue length or latency.

A practical rule:

- Use `ILogger` for operational and diagnostic messages.
- Use `TrackEvent` for business milestones.
- Use `TrackMetric` for numeric measurements.
- Use `TrackException` when you need to capture handled exceptions explicitly.

**Correlation and distributed tracing**

Correlation lets you connect a request to downstream dependencies and logs. This is essential in microservices and multi-tier systems.

Application Insights and ASP.NET Core can propagate correlation identifiers automatically when configured correctly.

- Incoming request telemetry gets an operation context.
- Outbound HTTP calls can carry trace headers.
- Logs can be correlated with the current request context.

When you inspect telemetry in Application Insights, you can follow a single request across services if each hop preserves correlation headers.

**Availability and health monitoring**

Application Insights availability monitoring is used to test whether an endpoint is reachable from one or more locations. In current Azure monitoring terminology, standard availability tests are the preferred approach; older multi-step web tests are legacy and should not be treated as the default guidance.

Use availability tests when you want to verify:

- DNS resolution and network reachability
- HTTP response codes and response time
- Basic end-to-end availability from outside the app

Use health endpoints when you want internal application state and dependency readiness. They are complementary, not interchangeable.

**Querying telemetry with KQL**

Kusto Query Language (KQL) is used in Application Insights and Log Analytics. You should know how to inspect requests, dependencies, exceptions, and traces.

Requests example:

```kusto
requests
| where timestamp > ago(24h)
| summarize count() by resultCode
| order by count_ desc
```

Dependencies example with explicit aliasing:

```kusto
dependencies
| where timestamp > ago(24h)
| summarize avg_duration = avg(duration), count() by target, type
| order by avg_duration desc
```

Exceptions example:

```kusto
exceptions
| where timestamp > ago(24h)
| project timestamp, type, outerMessage, operation_Name, cloud_RoleName
| order by timestamp desc
```

Traces example:

```kusto
traces
| where timestamp > ago(24h)
| where severityLevel >= 2
| project timestamp, message, severityLevel, operation_Id
| order by timestamp desc
```

These queries help you identify failure patterns, slow dependencies, and noisy logs.

**Configuration patterns you should recognize**

Application Insights can be configured in several ways depending on hosting model and SDK version. For exam purposes, focus on the concepts rather than one rigid file shape.

Common patterns include:

- Connection string in environment variables for App Service or container deployments.
- `appsettings.json` binding under `ApplicationInsights:ConnectionString` when your app reads configuration.
- `AddApplicationInsightsTelemetry()` in `Program.cs`.
- Logging integration through `ILogger` and provider configuration.

Example `Program.cs` with logging and health checks:

```csharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddApplicationInsightsTelemetry();

builder.Services.AddHealthChecks()
    .AddCheck("self", () => HealthCheckResult.Healthy())
    .AddSqlServer(builder.Configuration.GetConnectionString("SqlConnection")!);

builder.Logging.ClearProviders();
builder.Logging.AddConsole();

var app = builder.Build();

app.MapHealthChecks("/health");
app.MapGet("/", (ILogger<Program> logger) =>
{
    logger.LogInformation("Root endpoint hit at {Time}", DateTimeOffset.UtcNow);
    return Results.Ok("Running");
});

app.Run();
```

This shows the three concerns working together:

- Health checks expose service status.
- Logging records runtime behavior.
- Application Insights collects telemetry for analysis.

**Operational best practices**

Good observability is intentional. Do not log everything, and do not make health checks too expensive.

- Keep health endpoints lightweight and fast.
- Separate readiness from liveness when dependency state matters.
- Use structured logs with stable event names.
- Avoid logging secrets, tokens, or personal data.
- Set log levels to reduce noise in production.
- Use custom telemetry sparingly and with clear business value.
- Validate that your Application Insights connection string is correct for the environment.

> Do not rely on a health endpoint as a security boundary. It is an operational signal, not an authorization mechanism.

**Common exam pitfalls**

- Confusing `ILogger` with `TelemetryClient`.
- Assuming one `/health` endpoint automatically means readiness and liveness are separated.
- Treating all performance counters as universally available across Windows, Linux, and containers.
- Copying a SQL health check overload without checking the package version.
- Assuming Application Insights only works through `appsettings.json` instead of environment-based configuration.
- Using outdated availability test terminology without recognizing that standard availability tests are the current guidance.

**Quick reference**

- **Health checks**: expose app and dependency status.
- **Logging**: capture structured diagnostic and business events.
- **Application Insights**: store and analyze telemetry in Azure.
- **KQL**: query telemetry data.
- **Correlation**: connect requests, dependencies, and logs across services.

**Practical workflow**

1. Register health checks for the app and critical dependencies.
2. Expose separate endpoints for liveness and readiness when needed.
3. Use `ILogger<T>` for structured logs throughout the app.
4. Enable Application Insights telemetry collection.
5. Add custom telemetry only for signals not already captured.
6. Query telemetry with KQL to validate behavior and troubleshoot issues.

**Summary**

For AZ-204, you should understand how ASP.NET Core health checks, logging, and Application Insights fit into a production monitoring strategy. Health endpoints provide fast operational signals. Logging gives you structured context. Application Insights centralizes telemetry so you can search, correlate, and alert on application behavior. The strongest implementations keep probes lightweight, logs structured, and telemetry intentional.
