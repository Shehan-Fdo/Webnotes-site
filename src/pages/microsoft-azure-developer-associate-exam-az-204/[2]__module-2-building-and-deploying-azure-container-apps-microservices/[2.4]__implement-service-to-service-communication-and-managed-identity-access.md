## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Service-to-service communication overview**

Service-to-service communication in Azure usually means one workload calls another workload without a human signing in. The main goal is to authenticate the caller, authorize the request, and avoid storing long-lived secrets when possible.

In Azure, the most common modern pattern is to use **Microsoft Entra ID** for identity and token-based access. A calling service obtains an access token for the target resource, then sends that token in the `Authorization: Bearer <token>` header.

Typical scenarios include:

- A web app calling a backend API.
- An Azure Function calling Azure Storage, Key Vault, or a custom API.
- A containerized service calling another internal service.
- A daemon or background worker accessing Azure resources without user interaction.

The key design choice is whether the caller uses:
- A **managed identity** for Azure-hosted workloads.
- An **application registration** with a client secret or certificate.
- A user-delegated flow, which is less common for pure service-to-service calls.

**Core identity concepts**

A service-to-service solution usually involves two identities:

- **Caller identity**: the app or workload that requests a token.
- **Resource identity**: the API or Azure service that validates the token and authorizes access.

For custom APIs, you often define:
- An **application registration** for the API.
- One or more **app roles** or **scopes** for authorization.
- A client application or managed identity that is granted permission to call the API.

For Azure platform services, the resource itself often exposes built-in data-plane permissions through Entra ID integration, role assignments, or service-specific authorization models.

Important terms:

- **Authentication** answers who you are.
- **Authorization** answers what you can do.
- **Token audience** identifies the intended resource.
- **Tenant** is the Entra ID directory that issues and validates identities.
- **Service principal** is the identity object used by an application in a tenant.

**Managed identity fundamentals**

A **managed identity** is an identity in Microsoft Entra ID that is automatically managed by Azure. You use it to let an Azure resource authenticate to other Azure services without storing credentials in code or configuration.

Managed identities are ideal when:
- The workload runs in Azure.
- You want to remove secrets from app settings.
- You need automatic credential rotation.
- You want to simplify operational security.

There are two types:

| Type | Use case | Notes |
|---|---|---|
| **System-assigned** | Identity tied to one Azure resource | Created with the resource and deleted with it |
| **User-assigned** | Shared identity reused across resources | Created independently and attached to one or more resources |

Practical differences:

- Use **system-assigned** when the identity lifecycle should match the resource lifecycle.
- Use **user-assigned** when multiple resources need the same identity or when you want to preserve identity across resource replacement.
- Use user-assigned identities when you need a stable identity for deployment automation or cross-resource reuse.

> Managed identity removes secret management, but it does not remove authorization. You still must grant the identity the correct role or permission on the target resource.

**How managed identity works**

When a managed identity-enabled resource needs a token, it requests one from the Azure Instance Metadata Service (IMDS) or the platform-specific identity endpoint. Azure then issues an access token for the requested resource.

The application does not store:
- Client secrets
- Certificates
- Refresh tokens

Instead, the platform handles the credential exchange.

Common Azure-hosted services that support managed identity include:
- Azure App Service
- Azure Functions
- Azure Virtual Machines
- Azure Virtual Machine Scale Sets
- Azure Container Apps
- Azure Kubernetes Service workloads through **workload identity** patterns, which are separate from managed identity
- Azure Logic Apps Standard
- Azure API Management in some scenarios
- Azure Service Bus, Event Hubs, Storage, and Key Vault integrations through Entra ID

**Managed identity versus workload identity**

Do not treat **AKS workload identity** as the same thing as managed identity. They are related in purpose but different in implementation.

- **Managed identity** is an Azure-managed service identity attached to an Azure resource.
- **Workload identity** uses federated identity with Entra ID and Kubernetes service accounts.
- Workload identity is commonly preferred for modern AKS authentication because it avoids pod-managed secrets and integrates with federation.

Use the correct term in design discussions and exam answers.

**Using managed identity from code**

In application code, you usually use the Azure Identity client libraries. The most common abstraction is `DefaultAzureCredential`, which tries multiple credential sources depending on environment and package version.

Typical sources may include:
- Environment variables
- Managed identity
- Developer sign-in tools such as Azure CLI or Visual Studio
- Shared token cache or other local developer credentials

Because the exact order can vary by SDK version, do not memorize a fixed chain. Instead, understand the intent:
- In local development, it uses your developer identity.
- In Azure, it can use managed identity automatically.
- You can also explicitly choose a credential type when troubleshooting.

Example using `DefaultAzureCredential` with Azure SDK clients:

```csharp
using Azure.Identity;
using Azure.Security.KeyVault.Secrets;

var credential = new DefaultAzureCredential();
var client = new SecretClient(new Uri("https://myvault.vault.azure.net/"), credential);

KeyVaultSecret secret = await client.GetSecretAsync("DbPassword");
Console.WriteLine(secret.Value);
```

This pattern works because the SDK requests a token for the Key Vault resource and the platform supplies the appropriate identity when running in Azure.

**Explicit credential selection**

When you need predictable behavior, use a specific credential class.

Examples:
- `ManagedIdentityCredential` for Azure-hosted workloads.
- `ClientSecretCredential` for app registrations with secrets.
- `ClientCertificateCredential` for certificate-based authentication.
- `AzureCliCredential` for local developer testing.

Example with a user-assigned managed identity:

```csharp
using Azure.Identity;
using Azure.Storage.Blobs;

var clientId = Environment.GetEnvironmentVariable("MANAGED_IDENTITY_CLIENT_ID");
var credential = new ManagedIdentityCredential(clientId);

var blobServiceClient = new BlobServiceClient(
    new Uri("https://mystorageaccount.blob.core.windows.net"),
    credential);
```

Use a user-assigned identity when you need to specify which identity should be used on a resource that has more than one identity attached.

**Azure role assignment and authorization**

Managed identity authentication succeeds only if the identity is authorized on the target resource.

For Azure resources, authorization commonly uses **Azure role-based access control** (Azure RBAC). You assign a role to the managed identity at the correct scope.

Examples:
- `Storage Blob Data Reader` for read-only blob access.
- `Storage Blob Data Contributor` for read/write blob access.
- `Key Vault Secrets User` for reading secrets from Key Vault.
- `Contributor` for broad management access, though this is usually too permissive for data-plane access.

A role assignment example with Azure CLI:

```bash
az role assignment create \
  --assignee-object-id <managed-identity-object-id> \
  --assignee-principal-type ServicePrincipal \
  --role "Storage Blob Data Contributor" \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<storage-account>
```

Key points:
- Use the identity’s **object ID** or principal ID when assigning roles.
- Assign the least privilege role that satisfies the workload.
- Scope the assignment as narrowly as possible.

**Custom APIs and app registrations**

For a custom API, managed identity alone is not enough. You must configure the API to accept Entra ID tokens and authorize callers.

A typical setup includes:
- An app registration for the API.
- An exposed permission model using **scopes** or **app roles**.
- A client app or managed identity granted permission to the API.
- Token validation in the API.

For daemon-to-API communication, **app roles** are often the best fit because there is no signed-in user.

Example API app registration concepts:
- Set an application ID URI such as `api://<api-client-id>`.
- Define app roles like `Api.Read` or `Api.Write`.
- Assign the managed identity to the app role.

Example token request from a managed identity:

```csharp
using Azure.Core;
using Azure.Identity;
using System.Net.Http.Headers;

var credential = new ManagedIdentityCredential();
var token = await credential.GetTokenAsync(
    new TokenRequestContext(new[] { "api://<api-client-id>/.default" }));

using var httpClient = new HttpClient();
httpClient.DefaultRequestHeaders.Authorization =
    new AuthenticationHeaderValue("Bearer", token.Token);

var response = await httpClient.GetAsync("https://api.contoso.com/weather");
response.EnsureSuccessStatusCode();
```

This works only when:
- The API is configured to trust Entra ID tokens.
- The managed identity has been granted an app role or equivalent permission on the API.
- The API validates issuer, audience, and authorization claims.

> Requesting a token does not grant access by itself. The API must recognize the caller and the caller must be assigned permission.

**API validation responsibilities**

A protected API should validate:
- **Issuer**: the token came from the expected Entra ID tenant.
- **Audience**: the token was issued for this API.
- **Expiration**: the token is still valid.
- **Authorization claims**: scopes or app roles match the required permission.

In ASP.NET Core, you typically use the Microsoft identity platform middleware.

Example configuration:

```csharp
builder.Services.AddAuthentication("Bearer")
    .AddJwtBearer("Bearer", options =>
    {
        options.Authority = "https://login.microsoftonline.com/<tenant-id>/v2.0";
        options.Audience = "api://<api-client-id>";
    });

builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("ApiRead", policy =>
        policy.RequireClaim("roles", "Api.Read"));
});
```

Then protect endpoints:

```csharp
app.MapGet("/weather", () => "ok")
   .RequireAuthorization("ApiRead");
```

For user-delegated APIs, you may check `scp` claims. For daemon or managed identity callers, you often check `roles`.

**Azure Storage access patterns**

Azure Storage supports Entra ID-based authorization for many data-plane scenarios, especially Blob and Queue services. This is a common exam topic because it replaces account keys with identity-based access.

Typical patterns:
- Use `BlobServiceClient` with `DefaultAzureCredential`.
- Assign the managed identity a storage data role.
- Access blobs without connection strings or shared keys.

Example:

```csharp
using Azure.Identity;
using Azure.Storage.Blobs;

var credential = new DefaultAzureCredential();
var client = new BlobServiceClient(
    new Uri("https://mystorageaccount.blob.core.windows.net"),
    credential);

await foreach (var container in client.GetBlobContainersAsync())
{
    Console.WriteLine(container.Name);
}
```

For Azure Files, do not assume the same experience as Blob or Queue data-plane auth. Azure Files has its own support model and limitations, so verify the exact protocol and SDK capabilities for the scenario you are designing.

**Key Vault access patterns**

Azure Key Vault is a common managed identity use case because it removes the need to store secrets in app settings.

Typical flow:
1. Enable managed identity on the app.
2. Grant the identity `Key Vault Secrets User` or a more specific role.
3. Use `DefaultAzureCredential` or `ManagedIdentityCredential` in code.
4. Read secrets, keys, or certificates depending on the permission model.

Example:

```csharp
using Azure.Identity;
using Azure.Security.KeyVault.Secrets;

var client = new SecretClient(
    new Uri("https://myvault.vault.azure.net/"),
    new DefaultAzureCredential());

var secret = await client.GetSecretAsync("CosmosDbConnectionString");
Console.WriteLine(secret.Value.Value);
```

Use Key Vault when you need:
- Centralized secret storage
- Rotation support
- Separation of secrets from application code
- Access auditing

**Service-to-service authentication flows**

There are several common flows you should recognize.

| Flow | Best for | Credential storage |
|---|---|---|
| Managed identity | Azure-hosted workloads | None |
| Client secret | Legacy or non-Azure apps | Secret required |
| Certificate credential | Higher assurance app auth | Certificate required |
| On-behalf-of | API calling downstream API with user context | User token exchange |
| Workload identity | AKS and federated workloads | None |

Use managed identity when the caller runs in Azure and the target supports Entra ID authorization. Use client secrets or certificates when the caller cannot use managed identity, such as on-premises or third-party hosting.

**Local development versus Azure runtime**

A common development pattern is to write code once and let the credential adapt to the environment.

Local development:
- You sign in with Azure CLI or Visual Studio.
- `DefaultAzureCredential` uses your developer identity.
- You test against real Azure resources if your account has access.

Azure runtime:
- The same code uses managed identity.
- No code changes are needed if the app is deployed correctly.

Example local setup:

```bash
az login
az account set --subscription <subscription-id>
```

If you need to troubleshoot token acquisition, you can inspect the active account:

```bash
az account show
```

For managed identity troubleshooting on Azure resources, verify:
- The identity is enabled.
- The target resource has the correct role assignment.
- The app is requesting the correct audience.
- The resource endpoint supports Entra ID auth.

**Common troubleshooting issues**

Many authentication failures come from configuration, not code.

- **401 Unauthorized**
  - The token is missing, expired, or for the wrong audience.
  - The API does not trust the issuer.
- **403 Forbidden**
  - The token is valid, but the identity lacks permission.
  - The role assignment is missing or scoped incorrectly.
- **Managed identity not found**
  - The resource has no identity enabled.
  - A user-assigned identity client ID was not specified.
- **Token acquired but API still rejects**
  - The API expects app roles or scopes that were not assigned.
  - The API validates a different audience or tenant.
- **Works locally but fails in Azure**
  - Local developer credentials are masking missing Azure permissions.
  - The managed identity has not been granted access.

A practical debugging approach:
1. Confirm the identity exists and is enabled.
2. Confirm the target resource’s authorization model.
3. Confirm the correct role or app role assignment.
4. Confirm the token audience matches the resource.
5. Inspect logs and token claims if needed.

**Security best practices**

Use the least privilege model everywhere.

- Prefer **managed identity** over secrets.
- Prefer **user-assigned identity** when you need reuse or stable lifecycle.
- Prefer **RBAC** roles that match the data-plane operation.
- Prefer **app roles** for daemon-to-API authorization.
- Rotate away from secrets and certificates where possible.
- Avoid broad roles like `Owner` or `Contributor` unless you truly need management-plane access.
- Store no credentials in source control, app settings, or container images.

Additional guidance:
- Separate management-plane permissions from data-plane permissions.
- Use different identities for different environments when isolation matters.
- Audit role assignments regularly.
- Use private networking and firewall rules alongside identity-based access when appropriate.

**Exam-focused distinctions**

For AZ-204, be able to distinguish these concepts:

- **Managed identity** is for Azure resources to authenticate without secrets.
- **App registration** defines an application identity in Entra ID.
- **Service principal** is the tenant-specific instance of that app.
- **RBAC role assignment** grants access to Azure resources.
- **App role assignment** grants access to a custom API.
- **Scope** is commonly used for delegated user access.
- **Token audience** must match the resource being called.

A useful mental model:
- If you are calling an Azure service, think managed identity plus Azure RBAC.
- If you are calling a custom API, think app registration plus scopes or app roles.
- If you need user context, think delegated permissions or on-behalf-of flow.

**Practical end-to-end example**

A web app calls a backend API, and the API reads a secret from Key Vault.

1. Enable a managed identity on the web app.
2. Register the backend API in Entra ID.
3. Expose an app role such as `Api.Read`.
4. Assign the web app’s managed identity to that app role.
5. Configure the API to validate Entra ID tokens.
6. Grant the API’s managed identity access to Key Vault.
7. Use `DefaultAzureCredential` in the API to read secrets.

This architecture keeps secrets out of the web app and lets each component authenticate independently.

**Quick reference**

- Use `DefaultAzureCredential` for code that should work locally and in Azure.
- Use `ManagedIdentityCredential` when you want to force managed identity.
- Use Azure RBAC for Azure resource access.
- Use app roles or scopes for custom APIs.
- Validate issuer, audience, and authorization claims in every protected API.
- Remember that authentication is not authorization.
