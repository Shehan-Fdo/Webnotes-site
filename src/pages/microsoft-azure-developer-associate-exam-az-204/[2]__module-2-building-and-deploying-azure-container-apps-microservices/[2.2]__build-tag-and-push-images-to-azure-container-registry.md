## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Azure Container Registry overview**

Azure Container Registry (ACR) is a managed, private registry for storing and distributing container images and related artifacts. In AZ-204 scenarios, you typically use ACR to keep images close to Azure workloads, control access with Azure identity, and support repeatable deployment pipelines.

ACR fits into a standard container workflow:

- Build an image from a `Dockerfile`.
- Tag the image with the registry login server and repository name.
- Push the image to ACR.
- Pull the image from Azure services such as Azure Container Apps, Azure Kubernetes Service (AKS), or App Service for Containers.

A registry name is globally unique within Azure and resolves to a login server such as `myregistry.azurecr.io`. The repository name identifies the image family, and the tag identifies a version or variant.

**Core concepts**

A container image is a read-only package that includes the application, runtime, dependencies, and metadata needed to run a container. A container registry stores these images and lets you version them with tags.

<dl>
<dt><strong>Repository</strong></dt>
<dd>A named collection of related images, such as `webapi` or `orderservice`.</dd>
<dt><strong>Tag</strong></dt>
<dd>A label attached to an image, such as `v1`, `1.0.0`, or `latest`.</dd>
<dt><strong>Digest</strong></dt>
<dd>A content-based immutable identifier, usually shown as `sha256:...`.</dd>
<dt><strong>Login server</strong></dt>
<dd>The registry endpoint used by Docker, such as `myregistry.azurecr.io`.</dd>
</dl>

Tags are mutable, so `latest` can point to different image contents over time. Digests are immutable and are safer for exact deployments when you need reproducibility.

**Building images locally**

You usually build an image with Docker from a `Dockerfile`. The build context is the directory sent to the Docker engine during build, so keep it small and exclude unnecessary files with `.dockerignore`.

```dockerfile
# Dockerfile
FROM mcr.microsoft.com/dotnet/aspnet:8.0 AS base
WORKDIR /app
EXPOSE 8080

FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src
COPY ["MyApi/MyApi.csproj", "MyApi/"]
RUN dotnet restore "MyApi/MyApi.csproj"
COPY . .
WORKDIR "/src/MyApi"
RUN dotnet publish "MyApi.csproj" -c Release -o /app/publish /p:UseAppHost=false

FROM base AS final
WORKDIR /app
COPY --from=build /app/publish .
ENTRYPOINT ["dotnet", "MyApi.dll"]
```

Build the image with a repository name and tag:

```bash
docker build -t myapi:1.0.0 .
docker images
```

Practical build guidance:

- Use small base images when possible to reduce attack surface and pull time.
- Prefer multi-stage builds to keep the final image lean.
- Pin versions in base images for predictable builds.
- Add `.dockerignore` entries for `bin/`, `obj/`, `.git/`, and local secrets.

Example `.dockerignore`:

```text
bin/
obj/
.git/
.vscode/
*.user
*.suo
```

**Tagging images for ACR**

Before pushing, tag the local image with the registry login server and repository path. The standard format is:

```text
<registry>.azurecr.io/<repository>:<tag>
```

Example:

```bash
docker tag myapi:1.0.0 myregistry.azurecr.io/myapi:1.0.0
docker tag myapi:1.0.0 myregistry.azurecr.io/myapi:latest
```

Tagging does not copy image layers immediately; it creates another reference to the same local image. This is useful when you want to publish multiple tags for the same build, such as a semantic version and `latest`.

Common tagging strategies:

- Use semantic version tags like `1.4.2` for release tracking.
- Use build numbers or commit SHAs for traceability in CI/CD.
- Use `latest` only for convenience, not as the sole deployment reference.
- Use immutable digests in production deployment manifests when you need strict repeatability.

**Creating and configuring Azure Container Registry**

You can create ACR with the Azure CLI. The registry name must be lowercase and globally unique.

```bash
az group create \
  --name rg-containers \
  --location eastus

az acr create \
  --resource-group rg-containers \
  --name myregistry \
  --sku Basic
```

Useful registry settings:

- **SKU** affects features and scale.
  - Basic is suitable for development and small workloads.
  - Standard adds more throughput and features.
  - Premium adds advanced capabilities such as geo-replication and private link support.
- **Admin user** can be enabled for simple testing, but Azure identity is preferred for production.
- **Network access** can be restricted with firewalls and private endpoints in higher SKUs.

Check the login server:

```bash
az acr show --name myregistry --query loginServer --output tsv
```

**Authenticating to ACR**

You can authenticate with Docker using Azure CLI or a registry token. For exam scenarios, the most common approach is Azure CLI login followed by `az acr login`.

```bash
az login
az acr login --name myregistry
```

This configures Docker credentials for the registry so you can push and pull images.

If you need to use the registry admin account for a quick test, you can enable it and retrieve credentials:

```bash
az acr update --name myregistry --admin-enabled true
az acr credential show --name myregistry
```

> Use the admin account only for short-lived testing. Prefer Microsoft Entra ID-based access control for production and shared environments.

**Pushing images to ACR**

After tagging, push the image to the registry:

```bash
docker push myregistry.azurecr.io/myapi:1.0.0
docker push myregistry.azurecr.io/myapi:latest
```

A successful push uploads image layers and metadata to ACR. If the layers already exist in the registry, Docker uploads only the missing layers, which makes subsequent pushes faster.

A typical end-to-end workflow looks like this:

```bash
az acr login --name myregistry

docker build -t myapi:1.0.0 .
docker tag myapi:1.0.0 myregistry.azurecr.io/myapi:1.0.0
docker push myregistry.azurecr.io/myapi:1.0.0
```

**Build and push directly to ACR**

ACR can build images for you without requiring a local Docker daemon. This is useful in cloud-hosted build pipelines or when you want to keep source code and image creation close to the registry.

```bash
az acr build \
  --registry myregistry \
  --image myapi:1.0.0 \
  --file Dockerfile \
  .
```

This command uploads the build context to ACR, builds the image in Azure, and stores the result in the registry. It is especially useful when:

- Your local machine does not have Docker installed.
- You want consistent cloud-based builds.
- You want to integrate image creation into Azure-native automation.

**Inspecting images and repositories**

You can list repositories and tags in ACR with the Azure CLI:

```bash
az acr repository list --name myregistry --output table
az acr repository show-tags --name myregistry --repository myapi --output table
```

You can also inspect image manifests and digests:

```bash
az acr repository show-manifests \
  --name myregistry \
  --repository myapi \
  --output table
```

Digests are important when you want to deploy a specific image version without tag drift.

**Practical deployment example**

A common deployment pattern is to push an image to ACR and then reference it from an Azure-hosted service.

Example container image reference:

```text
myregistry.azurecr.io/myapi:1.0.0
```

Example Azure Container Apps deployment concept:

```bash
az containerapp create \
  --name myapi-app \
  --resource-group rg-containers \
  --image myregistry.azurecr.io/myapi:1.0.0 \
  --registry-server myregistry.azurecr.io \
  --registry-identity system
```

For AKS, you typically grant the cluster identity permission to pull from ACR, then reference the same image in your Kubernetes manifest.

**Identity and permissions**

ACR integrates with Azure role-based access control (RBAC). The most relevant role for pulling images is `AcrPull`; for pushing, use `AcrPush`.

Example role assignment:

```bash
az role assignment create \
  --assignee <principal-id> \
  --role AcrPush \
  --scope $(az acr show --name myregistry --query id --output tsv)
```

Key permission patterns:

- Grant `AcrPull` to runtime identities that only need to download images.
- Grant `AcrPush` to build identities and CI/CD pipelines that publish images.
- Avoid broad subscription-level permissions when a registry-scoped role is enough.

**Troubleshooting common issues**

Image build and push failures usually come from authentication, naming, or Dockerfile problems.

- `denied: requested access to the resource is denied`
  - You are not authenticated, or your identity lacks `AcrPush`.
  - The registry name or login server may be incorrect.
- `repository does not exist`
  - You may have mistyped the repository name or tag.
  - The image may not have been tagged with the registry prefix.
- `manifest unknown`
  - The tag does not exist in the registry.
  - The push may have failed before completion.
- Build context too large
  - Your `.dockerignore` is missing unnecessary files.
  - You are copying more than the application needs.

Useful verification commands:

```bash
docker images
docker inspect myregistry.azurecr.io/myapi:1.0.0
az acr repository list --name myregistry
az acr repository show-tags --name myregistry --repository myapi
```

**Security and operational best practices**

Container image management is part of your supply chain, so treat it as a security boundary.

- Scan images for vulnerabilities when your pipeline supports it.
- Use minimal base images and keep them updated.
- Avoid embedding secrets in images or Dockerfiles.
- Prefer managed identities and RBAC over static credentials.
- Use immutable tags or digests for production releases.
- Separate build, test, and release tags to avoid accidental promotion.

A simple release flow might use:

- `myapi:ci-1234` for build artifacts
- `myapi:rc-1` for release candidates
- `myapi:1.0.0` for production releases

**Exam focus**

For AZ-204, expect questions about the sequence and purpose of each step:

1. Build the image from a `Dockerfile`.
2. Tag the image with the ACR login server.
3. Authenticate to ACR.
4. Push the image to the registry.
5. Reference the image from an Azure compute service.

You should also recognize when to use:

- `docker build` for local builds.
- `az acr build` for cloud-based builds.
- `docker tag` to prepare an image for a registry.
- `docker push` to publish the image.
- `AcrPull` and `AcrPush` for access control.

**Quick reference**

```bash
# Create registry
az acr create --resource-group rg-containers --name myregistry --sku Basic

# Log in
az acr login --name myregistry

# Build locally
docker build -t myapi:1.0.0 .

# Tag for ACR
docker tag myapi:1.0.0 myregistry.azurecr.io/myapi:1.0.0

# Push to ACR
docker push myregistry.azurecr.io/myapi:1.0.0

# Build in ACR
az acr build --registry myregistry --image myapi:1.0.0 .
```
