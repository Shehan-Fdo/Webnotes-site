## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Lab goal and architecture**

This lab focuses on deploying two microservices to Azure Kubernetes Service (AKS), wiring them together through internal Kubernetes networking, and validating horizontal autoscaling behavior. The typical pattern is a frontend service that calls a backend service over the cluster network, while the frontend is exposed for testing and the backend remains internal.

A correct mental model matters:

- The **client** reaches a Kubernetes Service, not a specific pod.
- The Service load-balances traffic to matching pods through label selectors.
- The frontend pod calls the backend through the backend Service DNS name.
- Horizontal Pod Autoscaler (HPA) scales pods based on observed metrics, not on Service traffic directly.

A simple topology looks like this:

```text
Client -> Frontend Service -> Frontend Pods -> Backend Service -> Backend Pods
```

For internal communication, the frontend should use the backend Service name, such as `http://backend:3000` inside the same namespace, or the fully qualified DNS name when crossing namespaces.

**Core AKS prerequisites**

Before you validate autoscaling, confirm that the cluster can report resource metrics. HPA depends on the Kubernetes metrics pipeline, usually provided by the metrics server in AKS.

Check the cluster and node pool first:

```bash
az aks show \
  --resource-group <rg-name> \
  --name <aks-name> \
  --query "{name:name, kubernetesVersion:kubernetesVersion, fqdn:fqdn}" \
  -o table

kubectl get nodes -o wide
kubectl get pods -A
```

Verify that resource metrics work:

```bash
kubectl top nodes
kubectl top pods -A
```

If `kubectl top` fails, HPA cannot calculate utilization correctly. In that case, fix the metrics pipeline before creating the autoscaler.

If you want the cluster to add nodes when pod demand exceeds current capacity, enable node pool autoscaling on AKS. This is separate from HPA.

```bash
az aks nodepool update \
  --resource-group <rg-name> \
  --cluster-name <aks-name> \
  --name <nodepool-name> \
  --enable-cluster-autoscaler \
  --min-count 1 \
  --max-count 3
```

> HPA scales pods. Cluster autoscaler scales nodes. You often need both for a realistic lab.

**Microservice deployment model**

Use separate Deployments and Services for the frontend and backend. Keep the backend internal by using a `ClusterIP` Service. Expose the frontend only if you need external validation.

Key design points:

- **Backend Deployment** runs the API or data service.
- **Backend Service** provides stable DNS and virtual IP inside the cluster.
- **Frontend Deployment** calls the backend Service.
- **Frontend Service** can be `ClusterIP`, `LoadBalancer`, or accessed through port-forwarding during testing.

A minimal backend Deployment should include labels that match the Service selector:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: backend
spec:
  replicas: 2
  selector:
    matchLabels:
      app: backend
  template:
    metadata:
      labels:
        app: backend
    spec:
      containers:
        - name: backend
          image: ghcr.io/example/backend:1.0.0
          ports:
            - containerPort: 3000
          resources:
            requests:
              cpu: 100m
              memory: 128Mi
            limits:
              cpu: 250m
              memory: 256Mi
```

A matching backend Service:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: backend
spec:
  type: ClusterIP
  selector:
    app: backend
  ports:
    - port: 3000
      targetPort: 3000
```

The frontend Deployment should also define resource requests if you plan to autoscale on CPU utilization. Without CPU requests, HPA cannot compute utilization as a percentage of requested CPU.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: frontend
spec:
  replicas: 2
  selector:
    matchLabels:
      app: frontend
  template:
    metadata:
      labels:
        app: frontend
    spec:
      containers:
        - name: frontend
          image: ghcr.io/example/frontend:1.0.0
          ports:
            - containerPort: 80
          env:
            - name: BACKEND_URL
              value: http://backend:3000
          resources:
            requests:
              cpu: 100m
              memory: 128Mi
            limits:
              cpu: 250m
              memory: 256Mi
```

**Internal service-to-service communication**

Inside the cluster, the frontend should call the backend by Service DNS name. If both workloads are in the same namespace, `http://backend:3000` is enough. If they are in different namespaces, use the fully qualified domain name.

Examples:

```text
http://backend:3000
http://backend.default.svc.cluster.local:3000
```

Use the fully qualified name when you want to remove ambiguity in multi-namespace environments. Kubernetes DNS resolves the Service name to the Service IP, and kube-proxy or the CNI layer forwards traffic to healthy backend pods.

A practical frontend implementation often reads the backend address from an environment variable:

```javascript
const backendUrl = process.env.BACKEND_URL || 'http://backend:3000';
```

If you test from inside the cluster, use a temporary curl pod rather than assuming a specific base image has `wget` or `curl` installed.

```bash
kubectl run curl-test \
  --image=curlimages/curl:8.10.1 \
  --restart=Never \
  --rm -it \
  -- sh
```

Then test the backend:

```bash
curl http://backend:3000
```

**Exposing the frontend for validation**

If the frontend Service is `ClusterIP`, it is only reachable inside the cluster. That is fine for internal communication tests, but you still need a way to validate it from outside the cluster.

Common options:

- `kubectl port-forward` for quick local testing
- `Service type: LoadBalancer` for a temporary public endpoint
- Ingress if the lab includes HTTP routing and a controller

Port-forward example:

```bash
kubectl port-forward svc/frontend 8080:80
```

Then browse or curl:

```bash
curl http://localhost:8080
```

A `LoadBalancer` Service is useful when you want a direct external IP:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: frontend
spec:
  type: LoadBalancer
  selector:
    app: frontend
  ports:
    - port: 80
      targetPort: 80
```

**Horizontal Pod Autoscaler fundamentals**

HPA adjusts the number of pod replicas based on observed metrics such as CPU or memory. In AKS, CPU-based HPA is the most common lab scenario.

Important behavior:

- HPA uses the metrics API to read current usage.
- For CPU utilization targets, the container must define CPU requests.
- HPA compares current usage against the requested CPU, not against the limit.
- If requests are missing, utilization-based scaling is not meaningful.

A typical HPA manifest for the frontend:

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: frontend-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: frontend
  minReplicas: 2
  maxReplicas: 6
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 60
```

Apply it with:

```bash
kubectl apply -f frontend-hpa.yaml
kubectl get hpa
kubectl describe hpa frontend-hpa
```

If the HPA shows `unknown` metrics or no scaling activity, check these items:

- `kubectl top pods` works
- the target Deployment exists
- the pods have CPU requests
- the metrics server is healthy
- the HPA target name matches the Deployment name exactly

**Load generation and scaling validation**

To validate autoscaling, generate sustained CPU load against the frontend or backend, depending on which workload you want to scale. For a frontend HPA, create load that increases frontend CPU usage.

Use a purpose-built image for stress testing rather than relying on shell utilities that may not exist in a minimal container. A reliable approach is to run a temporary load generator pod with a known tool.

Example using `busybox` for repeated HTTP calls is not ideal because image variants differ. Prefer a curl-based image or a dedicated stress image.

A safer curl loop from a temporary pod:

```bash
kubectl run loadgen \
  --image=curlimages/curl:8.10.1 \
  --restart=Never \
  --rm -it \
  -- sh
```

Inside the pod:

```sh
while true; do
  curl -s http://frontend
  sleep 0.2
done
```

For CPU stress on the frontend itself, use a manifest-based approach rather than invalid `kubectl run` resource flags. A separate stress pod is clearer and more portable.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: cpu-stress
spec:
  restartPolicy: Never
  containers:
    - name: stress
      image: polinux/stress
      args: ["--cpu", "2", "--timeout", "300s"]
      resources:
        requests:
          cpu: 100m
          memory: 64Mi
        limits:
          cpu: 500m
          memory: 128Mi
```

Apply and watch scaling:

```bash
kubectl apply -f cpu-stress.yaml
kubectl get pods -w
kubectl get hpa -w
```

**Expected validation results**

Use these outcomes to confirm the lab is working:

- Backend responds from inside the cluster with a successful HTTP response.
- Frontend responds and can reach the backend through the Service name.
- `kubectl top pods` shows live CPU usage.
- HPA increases replicas when CPU utilization stays above the target.
- HPA scales back down after load drops and the stabilization window passes.

Example expected checks:

```bash
kubectl get deploy frontend
kubectl get svc frontend backend
kubectl get hpa frontend-hpa
kubectl describe hpa frontend-hpa
kubectl top pods
```

Sample success patterns:

- `curl http://backend:3000` returns JSON or text from the backend API.
- `curl http://localhost:8080` returns the frontend page or API response.
- `kubectl get hpa` shows `TARGETS` such as `65%/60%`.
- `kubectl get deploy frontend` shows replicas increasing from `2` to a higher value.

**Troubleshooting checklist**

If the frontend cannot reach the backend, check the following in order:

1. Confirm the backend Service exists and has endpoints.
2. Confirm labels on the backend pods match the Service selector.
3. Confirm the frontend uses the correct Service DNS name and port.
4. Confirm both workloads are in the expected namespace.
5. Confirm NetworkPolicy is not blocking traffic.

Useful commands:

```bash
kubectl get svc
kubectl get endpoints backend
kubectl get pods -l app=backend -o wide
kubectl logs deploy/frontend
kubectl logs deploy/backend
```

If HPA does not scale:

- Verify CPU requests exist on the target container.
- Verify metrics are available with `kubectl top`.
- Verify the HPA target references the correct Deployment.
- Verify the load is sustained long enough to exceed the threshold.
- Verify the cluster has enough node capacity or cluster autoscaler is enabled.

**Why requests and limits matter**

Resource requests influence scheduling and autoscaling behavior. CPU requests are especially important for HPA because utilization is calculated as a percentage of requested CPU.

Practical guidance:

- Set requests to the minimum CPU and memory needed for normal operation.
- Set limits to prevent noisy-neighbor behavior or runaway usage.
- Keep requests realistic; too-low requests can cause aggressive scaling.
- Keep limits above requests so the container can burst when needed.

Example comparison:

| Setting | Purpose | HPA impact |
|---|---|---|
| `requests.cpu` | Guaranteed baseline for scheduling | Required for CPU utilization targets |
| `limits.cpu` | Upper bound for container CPU usage | Does not define HPA utilization |
| `requests.memory` | Scheduling and bin packing | Useful for capacity planning |
| `limits.memory` | Prevents excessive memory use | Helps avoid node pressure |

**Exam-ready takeaways**

- Use **ClusterIP** for internal-only services.
- Use Kubernetes DNS names for service-to-service calls.
- Define CPU requests on the autoscaled Deployment.
- Verify metrics with `kubectl top` before relying on HPA.
- Distinguish **HPA** from cluster autoscaler.
- Use a reliable load generator image and a manifest for stress workloads.
- Validate both application behavior and scaling behavior, not just pod creation.
