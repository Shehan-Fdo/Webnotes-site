## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Containerization fundamentals**

Docker packages an application and its dependencies into an image so you can run the same workload consistently across development, test, and production. For a microservice, this reduces environment drift and makes deployment to Azure services such as Azure Container Apps, Azure App Service for Containers, or Azure Kubernetes Service more predictable.

A container is a running instance of an image. The image is built from a `Dockerfile`, which describes the base image, build steps, runtime configuration, and startup command. In practice, you use containers to isolate a service, standardize its runtime, and simplify scaling.

- **Image**: immutable build artifact created from a `Dockerfile`
- **Container**: runtime process created from an image
- **Registry**: storage for images, such as Azure Container Registry
- **Layer**: cached filesystem step that speeds up rebuilds

**Why Docker is useful for microservices**

Microservices often have small, focused responsibilities and independent release cycles. Docker helps you package each service separately, which supports independent versioning and deployment.

- You keep service dependencies local to the image instead of relying on the host machine.
- You can reproduce bugs by running the same image locally and in the cloud.
- You can scale one service without redeploying the entire application.
- You can pin runtime versions, such as a specific .NET, Node.js, Python, or Java base image.

**Core Docker workflow**

A typical workflow is:

1. Write a `Dockerfile` for the service.
2. Build the image with `docker build`.
3. Run the container locally with `docker run`.
4. Tag the image for a registry.
5. Push the image to a registry.
6. Deploy the image from the registry to Azure.

Example build and run commands:

```bash
docker build -t orders-api:1.0 .
docker run --rm -p 8080:8080 orders-api:1.0
```

The `-p 8080:8080` mapping exposes container port `8080` on the host. Use the port your application listens on inside the container.

**Dockerfile structure**

A `Dockerfile` is a sequence of instructions that define how to build the image. For microservices, you usually want a small runtime image and a separate build stage to keep the final image lean.

Common instructions:

- `FROM` selects the base image.
- `WORKDIR` sets the working directory.
- `COPY` adds files into the image.
- `RUN` executes build-time commands.
- `EXPOSE` documents the port the app listens on.
- `ENV` sets environment variables.
- `ENTRYPOINT` defines the main executable.
- `CMD` provides default arguments or the default command.

Example multi-stage build for a .NET microservice:

```dockerfile
# Build stage
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src

COPY *.csproj ./
RUN dotnet restore

COPY . ./
RUN dotnet publish -c Release -o /app/publish

# Runtime stage
FROM mcr.microsoft.com/dotnet/aspnet:8.0 AS runtime
WORKDIR /app

COPY --from=build /app/publish .
EXPOSE 8080
ENV ASPNETCORE_URLS=http://+:8080

ENTRYPOINT ["dotnet", "Orders.Api.dll"]
```

Multi-stage builds matter because they remove SDK tools and intermediate files from the final image, which reduces image size and attack surface.

**Choosing a base image**

Base image selection affects size, compatibility, and security.

| Scenario | Typical choice | Notes |
|---|---|---|
| Build stage | SDK image | Includes compilers and build tools |
| Runtime stage | ASP.NET, Node runtime, Python slim, Java JRE | Contains only what the app needs to run |
| Minimal footprint | Alpine or distroless-style image | Smaller, but may require extra compatibility work |

Use a base image that matches your application runtime exactly. If your app needs native libraries, verify they exist in the runtime image before you optimize for size.

**Ports, networking, and service binding**

A container can only receive traffic on ports that the application actually listens on. `EXPOSE` is documentation for the image; it does not publish the port by itself. You still need `docker run -p` or an orchestrator service definition.

For web microservices, bind to all interfaces inside the container, not just `localhost`.

```bash
dotnet Orders.Api.dll --urls http://0.0.0.0:8080
```

Or with environment variables:

```bash
docker run -e ASPNETCORE_URLS=http://+:8080 -p 8080:8080 orders-api:1.0
```

If the app binds only to `127.0.0.1`, traffic from outside the container cannot reach it.

**Environment variables and configuration**

Containers work best when configuration is externalized. Use environment variables for values that change by environment, such as connection strings, feature flags, and service endpoints.

- Keep secrets out of the image.
- Prefer runtime configuration over hardcoding values.
- Use Azure-managed secret stores when possible instead of embedding credentials.

Example:

```bash
docker run --rm \
  -e ASPNETCORE_ENVIRONMENT=Development \
  -e ConnectionStrings__OrdersDb="Server=sqlhost;Database=Orders;User Id=app;Password=secret;" \
  -p 8080:8080 \
  orders-api:1.0
```

The double underscore syntax is commonly used by .NET configuration providers to map nested settings.

**.dockerignore and build context**

The build context is the set of files sent to the Docker daemon during build. A large context slows builds and can accidentally include secrets or unnecessary artifacts.

Use `.dockerignore` to exclude files such as:

```text
bin/
obj/
.git/
.vscode/
Dockerfile
README.md
*.user
```

This improves build performance and helps prevent accidental leakage of local files into the image.

**Image tagging and versioning**

Tags identify image versions. Avoid relying only on `latest` because it is mutable and can hide what was actually deployed.

Recommended tagging patterns:

- Semantic version tags such as `1.0.0`
- Build-specific tags such as `1.0.0-20260426.1`
- Git commit tags such as `sha-abc1234`

Example:

```bash
docker tag orders-api:1.0 myregistry.azurecr.io/orders-api:1.0.0
docker tag orders-api:1.0 myregistry.azurecr.io/orders-api:latest
```

Use a stable tag for release tracking and a unique tag for traceability.

**Pushing to a registry**

A registry stores images so deployment targets can pull them. Azure Container Registry is a common choice for Azure workloads because it integrates well with Azure identity and deployment pipelines.

Typical flow:

```bash
az acr login --name myregistry
docker tag orders-api:1.0 myregistry.azurecr.io/orders-api:1.0.0
docker push myregistry.azurecr.io/orders-api:1.0.0
```

If you use Azure Container Registry, you can authenticate with Azure CLI, managed identity, or service principals depending on the deployment scenario.

**Running and debugging locally**

Local container execution is the fastest way to validate the image before deployment.

Useful commands:

```bash
docker ps
docker logs <container-id>
docker exec -it <container-id> sh
docker stop <container-id>
docker rm <container-id>
```

For troubleshooting:

- Check whether the container exits immediately because the entrypoint is wrong.
- Inspect logs for missing environment variables or port binding errors.
- Confirm the app listens on the expected port.
- Verify the image contains the published output, not source files only.

If the container starts but is unreachable, the most common causes are incorrect port mapping, binding to `localhost`, or a mismatch between the app port and the published port.

**Health checks and readiness**

Health checks help orchestrators determine whether a container is alive and ready to receive traffic. In Azure-hosted container platforms, health endpoints are especially useful for rolling updates and autoscaling.

Example ASP.NET Core health endpoint:

```csharp
app.MapGet("/health", () => Results.Ok("healthy"));
```

You can also define Docker health checks:

```dockerfile
HEALTHCHECK --interval=30s --timeout=3s --start-period=10s \
  CMD curl -f http://localhost:8080/health || exit 1
```

Use health checks when the service needs warm-up time, depends on downstream systems, or should be removed from rotation when unhealthy.

**Security considerations**

Container security starts with image hygiene.

- Use minimal base images to reduce attack surface.
- Run as a non-root user when possible.
- Avoid baking secrets into images or source control.
- Pin image versions instead of using floating tags for production.
- Scan images for vulnerabilities in your CI/CD pipeline.

Example non-root user pattern:

```dockerfile
FROM mcr.microsoft.com/dotnet/aspnet:8.0
WORKDIR /app
COPY --from=build /app/publish .
RUN useradd -m appuser
USER appuser
ENTRYPOINT ["dotnet", "Orders.Api.dll"]
```

> Do not store passwords, API keys, or connection strings in the `Dockerfile` or image layers. Treat the image as potentially inspectable by anyone with registry access.

**Practical Docker Compose example**

Use Docker Compose when your microservice depends on supporting services such as a database, cache, or message broker.

```yaml
services:
  orders-api:
    build: .
    ports:
      - "8080:8080"
    environment:
      ASPNETCORE_URLS: http://+:8080
      ConnectionStrings__OrdersDb: Server=db;Database=Orders;User Id=sa;Password=Your_password123;
    depends_on:
      - db

  db:
    image: mcr.microsoft.com/mssql/server:2022-latest
    environment:
      ACCEPT_EULA: "Y"
      MSSQL_SA_PASSWORD: "Your_password123"
    ports:
      - "1433:1433"
```

Compose is useful for local integration testing, but it is not a replacement for production orchestration.

**Azure deployment considerations**

When you move a containerized microservice to Azure, the image usually comes from a registry and the platform handles scheduling, scaling, and networking.

Common deployment targets:

- **Azure App Service for Containers** for simpler web app hosting
- **Azure Container Apps** for managed containerized microservices and event-driven scaling
- **Azure Kubernetes Service** for advanced orchestration and control

For AZ-204, focus on the idea that you build the image locally or in CI, push it to a registry, and deploy from that registry to an Azure container host.

**Common pitfalls**

- Building a large image because you copy the entire repository without a `.dockerignore`
- Using the wrong base image for the app runtime
- Forgetting to expose or publish the correct port
- Hardcoding environment-specific values into the image
- Using `latest` in production and losing deployment traceability
- Running the app only on `localhost` inside the container
- Leaving build tools in the final runtime image

**Quick validation checklist**

1. Confirm the `Dockerfile` uses a suitable runtime base image.
2. Confirm the app listens on the container port you publish.
3. Confirm the image builds without local machine dependencies.
4. Confirm the container starts and responds to a health endpoint.
5. Confirm secrets are injected at runtime, not baked into the image.
6. Confirm the image is tagged and pushed to a registry.
7. Confirm the deployment target can pull the image successfully.

**Key takeaways**

Containerizing a microservice with Docker means packaging the service, its runtime, and its configuration into a portable image. The most important habits are to keep the image small, externalize configuration, publish the correct port, and use a registry-backed deployment flow. For Azure, this pattern supports repeatable deployments and fits well with managed container services.
