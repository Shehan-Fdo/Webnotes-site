## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Azure Container Apps overview**

Azure Container Apps is a managed platform for running containerized applications without managing Kubernetes directly. It is useful when you want container deployment, built-in autoscaling, revision support, and service-to-service communication with less operational overhead than Azure Kubernetes Service (AKS).

For AZ-204, focus on how the platform separates concerns:

- **Ingress** and **secrets** are configured under the app’s `configuration`.
- **Environment variables** and **scale rules** are defined in the app’s `template`.
- The platform supports revisions, which matter when you change the app template or other revision-scoped settings.
- You can run both always-on web apps and event-driven workers.

A practical mental model is:

- `configuration` = external exposure, secrets, and app-wide runtime settings
- `template` = container image, env vars, probes, and scaling behavior
- revision mode = how changes are rolled out and how traffic is split

**Core deployment model**

A container app runs inside a Container Apps environment. The environment provides the shared runtime boundary, networking, and observability integration. You deploy one or more containers into an app, and the platform handles routing, scaling, and revision management.

Typical use cases include:

- HTTP APIs
- background workers
- event-driven processors
- microservices that need internal service discovery
- apps that benefit from scale-to-zero

A common AZ-204 scenario is deploying a web API that needs public ingress, a database connection string stored as a secret, environment variables for runtime configuration, and autoscaling based on HTTP traffic.

**Resource structure and where settings belong**

Container Apps configuration is often misunderstood because not every setting lives in the same place.

- **`configuration`**
  - ingress settings
  - secrets
  - registries
  - Dapr settings
  - active revision mode and traffic splitting
- **`template`**
  - container image and command
  - environment variables
  - probes
  - scale rules
  - container resources
  - multiple containers in the same app

This distinction matters because some changes create a new revision, while others are applied as app-level configuration updates. In exam scenarios, assume template changes are revision-scoped unless the question explicitly says otherwise.

**Ingress**

Ingress controls whether your app is reachable from outside the environment and how traffic enters the app. You use it for HTTP APIs, web apps, and some internal service-to-service scenarios.

Common ingress concepts:

- **external** ingress exposes the app publicly
- **internal** ingress keeps the app reachable only inside the Container Apps environment
- **targetPort** tells the platform which container port receives traffic
- **transport** can be HTTP or TCP depending on the scenario
- **traffic splitting** lets you route requests across revisions

A typical public API configuration looks like this:

```yaml
properties:
  configuration:
    ingress:
      external: true
      targetPort: 8080
      transport: auto
```

For internal-only services, set `external: false`. This is useful for backend APIs that should only be called by other apps in the same environment.

**Ingress and authentication**

Ingress exposure is not the same as application authentication.

- Ingress decides whether the app is reachable.
- Authentication and authorization decide who can access it.

In Azure Container Apps, you may protect the app using platform authentication features when available in your deployment model, or you may implement auth in the application itself. In AZ-204 scenarios, be careful not to assume that public ingress automatically means anonymous access is acceptable.

Practical guidance:

- Use public ingress for APIs that must be internet reachable.
- Use internal ingress for private microservices.
- Add authentication at the app layer or platform layer when the workload requires identity-based access.
- If you use Microsoft Entra ID, validate tokens in the app or through the platform’s auth integration where supported.

**Secrets**

Secrets store sensitive values such as connection strings, API keys, and certificates. In Container Apps, secrets are defined at the app level under `configuration`, then referenced by environment variables or other settings.

Important behaviors:

- Secrets are not exposed as plain text in the container definition.
- You reference a secret from an environment variable using `secretRef`.
- Updating a secret does not always require a new revision, but running containers may need a restart or revision update to pick up the new value depending on how the app is deployed and how the secret is consumed.
- Treat secret rotation as an operational event, not just a configuration edit.

Example secret definition and reference:

```yaml
properties:
  configuration:
    secrets:
      - name: db-connection
        value: "Server=tcp:myserver.database.windows.net,1433;Database=appdb;User ID=appuser;Password=REDACTED;"
  template:
    containers:
      - name: api
        image: myregistry.azurecr.io/api:1.0
        env:
          - name: ConnectionStrings__Default
            secretRef: db-connection
```

This pattern is common for ASP.NET Core and other apps that read configuration from environment variables.

**Creating and updating secrets in practice**

You can manage secrets with the Azure CLI, portal, ARM templates, or Bicep. In exam scenarios, know the operational flow:

1. Create or update the secret in the app configuration.
2. Reference the secret from the container environment variable or another supported setting.
3. Restart or redeploy if the app needs to re-read the value immediately.

Example CLI pattern:

```bash
az containerapp secret set \
  --name myapp \
  --resource-group my-rg \
  --secrets db-connection="Server=tcp:myserver.database.windows.net,1433;Database=appdb;User ID=appuser;Password=REDACTED;"
```

Then reference it in the app template or update the app to use the secret-backed variable.

> Secret values are sensitive and should not be committed to source control. Use Key Vault integration when the scenario requires centralized secret management or rotation workflows.

**Environment variables**

Environment variables configure runtime behavior without rebuilding the image. They are ideal for non-secret settings such as feature flags, service endpoints, log levels, and environment names.

Examples:

- `ASPNETCORE_ENVIRONMENT=Production`
- `LOG_LEVEL=Information`
- `SERVICE_URL=https://orders.internal`
- `FEATURE_FLAG_NEW_CHECKOUT=true`

Example template snippet:

```yaml
properties:
  template:
    containers:
      - name: api
        image: myregistry.azurecr.io/api:1.0
        env:
          - name: ASPNETCORE_ENVIRONMENT
            value: Production
          - name: LOG_LEVEL
            value: Information
          - name: ConnectionStrings__Default
            secretRef: db-connection
```

Use environment variables when:

- the value is not sensitive
- you want to change behavior without changing code
- the app already reads configuration from env vars
- you need a simple deployment-time override

**Environment variable precedence**

Most application frameworks support layered configuration. A common pattern is:

- appsettings files provide defaults
- environment variables override defaults
- secret-backed environment variables override plain values when the app reads them from the environment

For ASP.NET Core, double underscores map to nested configuration keys, such as `ConnectionStrings__Default`.

**Scaling fundamentals**

Azure Container Apps uses autoscaling to adjust the number of replicas based on demand. Scaling is driven by rules in the app template and can be based on HTTP traffic, CPU, memory, or event sources.

Key ideas:

- **Scale-to-zero** reduces cost when there is no demand.
- **HTTP scaling** is useful for web APIs and front ends.
- **Event-driven scaling** is useful for queues, topics, and background processing.
- **Replica count** changes automatically based on configured thresholds.

A simple scale configuration might look like this:

```yaml
properties:
  template:
    scale:
      minReplicas: 0
      maxReplicas: 5
      rules:
        - name: http-scale
          http:
            metadata:
              concurrentRequests: "50"
```

This means the app can scale from zero to five replicas and add capacity when concurrent HTTP requests exceed the threshold.

**HTTP scaling behavior**

HTTP scaling is one of the most exam-relevant behaviors.

- The platform monitors incoming HTTP load.
- It scales based on concurrent requests rather than only request rate.
- `minReplicas: 0` enables scale-to-zero for idle periods.
- Cold starts can affect latency when traffic returns after idle time.

Use HTTP scale-to-zero when:

- occasional traffic is acceptable
- cost savings matter
- the app can tolerate startup latency

Avoid or reconsider it when:

- low-latency response is critical
- the app has expensive startup initialization
- the workload receives constant traffic anyway

**Event-driven scaling**

For background processing, Container Apps can scale based on external events such as queue depth or message backlog. This is useful for workers that should run only when work exists.

Common scenarios:

- processing messages from Azure Service Bus
- consuming events from storage or queues
- handling scheduled or bursty workloads

The exact rule metadata depends on the trigger type. In exam prep, focus on the principle:

- the platform uses built-in scaling rules
- each rule maps to a supported trigger source
- the app scales out when the trigger indicates more work

**Scale rules and KEDA**

Azure Container Apps uses <abbr title="Kubernetes Event-driven Autoscaling">KEDA</abbr> concepts under the hood. You do not usually define arbitrary custom KEDA scalers the way you would on a self-managed Kubernetes cluster. Instead, you configure the supported built-in scale rules exposed by the platform.

That means:

- use supported HTTP, CPU, memory, and event-based rules
- choose the rule that matches the workload
- do not assume any KEDA scaler can be plugged in directly

**Revisions and traffic splitting**

A revision is an immutable snapshot of your app template. When you change the template, the platform can create a new revision. This is important for safe deployments and rollbacks.

Revision concepts:

- **single revision mode** routes all traffic to the latest active revision
- **multiple revision mode** lets you keep several revisions active
- traffic splitting supports canary and blue-green deployment patterns

Use revisions when you want:

- controlled rollout
- rollback to a previous version
- testing a new version with a small percentage of traffic

Example traffic split concept:

- revision A: 90%
- revision B: 10%

This is useful for canary testing after a deployment.

**Deployment workflow**

A practical deployment flow for an AZ-204 scenario is:

1. Build the container image.
2. Push the image to a registry such as Azure Container Registry.
3. Create or update the Container Apps environment.
4. Deploy the app with ingress, secrets, env vars, and scaling settings.
5. Validate the revision and traffic routing.
6. Monitor logs and metrics.

Example CLI deployment pattern:

```bash
az containerapp create \
  --name myapp \
  --resource-group my-rg \
  --environment my-env \
  --image myregistry.azurecr.io/api:1.0 \
  --target-port 8080 \
  --ingress external \
  --registry-server myregistry.azurecr.io \
  --secrets db-connection="Server=tcp:myserver.database.windows.net,1433;Database=appdb;User ID=appuser;Password=REDACTED;" \
  --env-vars ASPNETCORE_ENVIRONMENT=Production LOG_LEVEL=Information ConnectionStrings__Default=secretref:db-connection
```

The exact CLI syntax can vary by Azure CLI version, so in practice you should verify the installed command help before running production deployments.

**Bicep example**

Bicep is a common exam-relevant deployment format. A simplified app definition looks like this:

```bicep
param location string = resourceGroup().location
param appName string
param environmentId string
param image string

resource containerApp 'Microsoft.App/containerApps@2024-03-01' = {
  name: appName
  location: location
  properties: {
    managedEnvironmentId: environmentId
    configuration: {
      ingress: {
        external: true
        targetPort: 8080
      }
      secrets: [
        {
          name: 'db-connection'
          value: 'Server=tcp:myserver.database.windows.net,1433;Database=appdb;User ID=appuser;Password=REDACTED;'
        }
      ]
    }
    template: {
      containers: [
        {
          name: 'api'
          image: image
          env: [
            {
              name: 'ASPNETCORE_ENVIRONMENT'
              value: 'Production'
            }
            {
              name: 'ConnectionStrings__Default'
              secretRef: 'db-connection'
            }
          ]
        }
      ]
      scale: {
        minReplicas: 0
        maxReplicas: 5
        rules: [
          {
            name: 'http-scale'
            http: {
              metadata: {
                concurrentRequests: '50'
              }
            }
          }
        ]
      }
    }
  }
}
```

This example shows the correct separation of concerns:

- `configuration` contains ingress and secrets
- `template` contains env vars and scale rules
- `location` is included
- the app references the secret by name

**Service-to-service communication**

Container Apps are often used as a microservices platform. One app can call another using internal ingress and environment-based service discovery patterns.

Typical topology:

- public API app with external ingress
- internal orders service with internal ingress
- internal worker that processes queue messages
- shared environment for service discovery and networking

Example pattern:

- `frontend` exposes external ingress
- `orders-api` uses internal ingress
- `worker` has no ingress and scales from queue events

This design keeps internal services private while still allowing the public app to reach them through the environment.

**Dapr integration**

Dapr, the Distributed Application Runtime, is a relevant Container Apps capability for service invocation, pub/sub, state management, and bindings. It can simplify microservice communication and reduce direct coupling between services.

Use Dapr when you need:

- service invocation without hard-coding service endpoints
- pub/sub messaging
- state store abstraction
- event bindings

In AZ-204 scenarios, Dapr often appears alongside ingress and scaling because it helps you build distributed systems with less plumbing.

**Container apps jobs versus always-on apps**

Container Apps also supports jobs for run-to-completion workloads. This is important when comparing event-driven execution patterns.

Use a **container app** when:

- the service should stay available
- it handles HTTP requests or long-running background processing
- it needs revision-based deployment and ingress

Use a **container app job** when:

- the workload runs to completion
- you need scheduled or event-triggered execution
- you do not need a continuously running service

Examples of jobs:

- nightly data import
- one-time migration
- queue-triggered batch processing

This distinction matters because not every workload should be modeled as an always-on app.

**Practical end-to-end scenario**

Imagine you are deploying an order API for an e-commerce system.

Requirements:

- public HTTPS endpoint for mobile clients
- database connection string stored securely
- runtime environment set to production
- autoscaling based on HTTP concurrency
- internal order processor for asynchronous work

A good design is:

- `orders-api`
  - external ingress
  - secret-backed database connection string
  - `ASPNETCORE_ENVIRONMENT=Production`
  - HTTP scale rule
- `orders-worker`
  - internal or no ingress
  - queue-based scale rule
  - same secret store if it needs database access
- optional Dapr
  - service invocation between API and worker
  - pub/sub for order events

This architecture gives you:

- secure secret handling
- clear separation of public and private services
- cost-efficient scaling
- revision-based rollout control

**Troubleshooting and validation**

When a deployment does not behave as expected, check the following:

- the container listens on the same port as `targetPort`
- ingress is enabled when the app must be reachable
- the secret name matches the `secretRef`
- the environment variable name matches what the app expects
- the scale rule matches the workload type
- the revision is active and receiving traffic
- the app logs show startup or binding errors

Useful CLI commands:

```bash
az containerapp show -n myapp -g my-rg
az containerapp revision list -n myapp -g my-rg
az containerapp logs show -n myapp -g my-rg
```

If the app starts but is not reachable, verify:

- the container is listening on `0.0.0.0`, not only `localhost`
- the ingress `targetPort` matches the container port
- the revision is receiving traffic
- the app is healthy and not crashing on startup

**Exam focus points**

- Know the difference between `configuration` and `template`.
- Know that ingress and secrets are app-level configuration concerns.
- Know that env vars and scaling are template concerns.
- Know how `secretRef` works for secret-backed environment variables.
- Know that HTTP scaling uses concurrency-based behavior and can scale to zero.
- Know that revisions support safe rollout and traffic splitting.
- Know the difference between container apps and container app jobs.
- Know that Dapr can simplify microservice communication.
- Know that authentication is separate from ingress exposure.

**Quick comparison**

| Area | Purpose | Typical location |
|---|---|---|
| Ingress | Expose the app externally or internally | `configuration` |
| Secrets | Store sensitive values | `configuration` |
| Environment variables | Configure runtime behavior | `template` |
| Scale rules | Autoscale replicas | `template` |
| Revisions | Roll out template changes safely | app lifecycle |

**Key takeaways**

Azure Container Apps is a strong fit when you want managed container hosting with built-in ingress, secrets, autoscaling, and revision support. For AZ-204, the most important skill is knowing where each setting belongs and how the platform behaves when traffic, secrets, or deployment versions change.

If you remember only one thing, remember this:

- `configuration` handles exposure and sensitive app settings
- `template` handles runtime behavior and scaling
- revisions control how template changes are deployed
- scale-to-zero saves cost, but cold starts matter
