## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Distributed tracing and correlation overview**

Distributed tracing lets you follow a single request as it moves across multiple services, queues, and dependencies. Correlation adds the ability to tie logs, metrics, and traces together so you can diagnose latency, failures, and fan-out behavior in a distributed system.

In Azure, this is commonly implemented with **Application Insights** as part of Azure Monitor, or with OpenTelemetry exporting telemetry to Azure Monitor or another backend. The key idea is consistent propagation of trace context so each service can continue the same logical operation instead of starting an unrelated one.

- **Trace**: the end-to-end journey of a request through multiple components.
- **Span**: one timed operation within a trace, such as an HTTP call or database query.
- **Correlation ID**: a value used to link related telemetry items.
- **Baggage**: small key-value pairs propagated with trace context for cross-service metadata.

**Core concepts and terminology**

Modern distributed tracing follows the W3C Trace Context standard. The two primary headers are `traceparent` and `tracestate`.

- `traceparent` carries the trace ID, parent span ID, and trace flags.
- `tracestate` carries vendor-specific tracing state.
- `baggage` carries user-defined metadata that should flow across service boundaries.
- Legacy headers such as `Request-Id` and `Correlation-Context` may still appear in older systems, but they are not the preferred modern standard.

A trace usually has one root span and many child spans. Each service should create spans for meaningful work, such as receiving an HTTP request, calling a downstream API, or processing a queue message. This gives you a timeline of where time is spent and where failures occur.

**Why tracing matters in Azure applications**

Distributed systems fail in ways that are hard to see from a single service. Tracing helps you answer questions like:

- Which downstream dependency caused the slowdown?
- Did the request fail before or after the queue message was published?
- Which service instance handled the request?
- Are retries causing duplicate work or hidden latency?

Tracing is especially useful when you have:

- Microservices communicating over HTTP or messaging.
- Serverless functions triggered by queues, events, or HTTP.
- Background workers processing messages asynchronously.
- Fan-out/fan-in workflows where one request triggers many downstream operations.

**Azure telemetry options**

Azure developers commonly use one of these approaches:

- **Application Insights SDK**: direct instrumentation in .NET apps, especially older or explicit setups.
- **Azure Monitor OpenTelemetry**: preferred modern approach for many new applications.
- **Automatic instrumentation**: captures common frameworks and dependencies with minimal code.
- **Manual instrumentation**: used when you need custom spans, custom baggage, or nonstandard propagation.

> Be careful not to assume configuration alone enables telemetry. For example, setting an `ApplicationInsights:ConnectionString` value in `appsettings.json` is not enough by itself unless your app is wired to read and use it. In many setups, the connection string is supplied through environment variables or configuration and then explicitly passed into the telemetry pipeline.

**W3C Trace Context propagation**

The most important skill is preserving the trace context across service boundaries. When a service receives a request, it should read the incoming context and create child spans for its own work. When it sends a request, it should inject the current context into the outgoing headers.

Typical propagation flow:

1. Client sends `traceparent` and optionally `baggage`.
2. Service A receives the request and starts a server span.
3. Service A calls Service B and injects the current context.
4. Service B extracts the context and starts a child span.
5. Telemetry backend reconstructs the full trace.

A valid `traceparent` header looks like this:

```text
00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-00
```

Meaning:

- `00`: version
- `4bf92f3577b34da6a3ce929d0e0e4736`: trace ID
- `00f067aa0ba902b7`: parent span ID
- `00`: trace flags

**.NET tracing with Activity and ActivitySource**

In modern .NET, `System.Diagnostics.Activity` and `ActivitySource` are the core primitives for tracing. `ActivitySource` is the preferred way to create custom spans that integrate with OpenTelemetry and other instrumentation.

A simple setup might look like this:

```csharp
using System.Diagnostics;

var source = new ActivitySource("Orders.Api");

using var activity = source.StartActivity("ProcessOrder", ActivityKind.Server);

if (activity != null)
{
    activity.SetTag("order.id", orderId);
    activity.SetTag("customer.tier", "gold");
}
```

Key points:

- **ActivitySource** creates spans in a way that tracing libraries can observe.
- **ActivityKind** describes the role of the span, such as `Server`, `Client`, `Producer`, or `Consumer`.
- Tags add searchable metadata to spans.
- Use `using` so the span ends correctly even when exceptions occur.

If you need to propagate context manually, prefer W3C-aware APIs and avoid assuming `Activity.Id` is always a valid `traceparent`. `Activity.Id` can be hierarchical in some configurations, so use W3C context APIs or ensure W3C ID format is enabled before relying on it.

**HTTP propagation example in .NET**

For outgoing HTTP calls, tracing libraries often inject headers automatically when instrumentation is enabled. If you need manual propagation, use the current `Activity` context rather than constructing headers from arbitrary values.

```csharp
using System.Diagnostics;
using System.Net.Http.Headers;

var request = new HttpRequestMessage(HttpMethod.Get, "https://inventory/api/items");

var activity = Activity.Current;
if (activity != null)
{
    request.Headers.TryAddWithoutValidation("traceparent", activity.Id);
    if (!string.IsNullOrEmpty(activity.TraceStateString))
    {
        request.Headers.TryAddWithoutValidation("tracestate", activity.TraceStateString);
    }
}

var response = await httpClient.SendAsync(request);
```

> This pattern is only safe when the current activity is using W3C IDs. In modern applications, prefer automatic instrumentation or explicit W3C context propagation APIs rather than building headers from `Activity.Id` blindly.

A more robust approach is to let OpenTelemetry or the Azure SDK instrumentation handle header injection for you.

**Incoming HTTP request handling**

When your service receives a request, you should continue the existing trace if headers are present. ASP.NET Core and OpenTelemetry can do this automatically when configured.

Example middleware-style inspection:

```csharp
app.Use(async (context, next) =>
{
    var traceparent = context.Request.Headers["traceparent"].ToString();
    var tracestate = context.Request.Headers["tracestate"].ToString();

    // Usually you do not parse these manually unless you have a special case.
    // Instrumentation libraries can extract and continue the trace for you.

    await next();
});
```

Practical guidance:

- Let the framework extract context when possible.
- Use manual extraction only for custom protocols or special integration points.
- Keep trace context headers intact when proxying requests.

**Messaging and asynchronous workflows**

Tracing across queues and event buses is more complex because the producer and consumer are separated in time. You must persist the trace context in message metadata and restore it when the consumer processes the message.

Common Azure messaging services include:

- **Azure Service Bus**
- **Azure Storage Queues**
- **Azure Event Hubs**
- **Azure Event Grid**

For message-based systems, the producer should attach trace context to message headers or application properties. The consumer should extract that context and start a new child span.

Example with Service Bus message application properties:

```csharp
using Azure.Messaging.ServiceBus;
using System.Diagnostics;

var message = new ServiceBusMessage(BinaryData.FromString("{\"orderId\":123}"));

if (Activity.Current != null)
{
    message.ApplicationProperties["traceparent"] = Activity.Current.Id;
    if (!string.IsNullOrEmpty(Activity.Current.TraceStateString))
    {
        message.ApplicationProperties["tracestate"] = Activity.Current.TraceStateString;
    }
}

await sender.SendMessageAsync(message);
```

On the consumer side, extract the parent context and create a child activity using a W3C-aware parent context:

```csharp
using Azure.Messaging.ServiceBus;
using System.Diagnostics;

var source = new ActivitySource("Orders.Worker");

async Task ProcessMessageAsync(ProcessMessageEventArgs args)
{
    args.Message.ApplicationProperties.TryGetValue("traceparent", out var traceparentObj);
    args.Message.ApplicationProperties.TryGetValue("tracestate", out var tracestateObj);

    var traceparent = traceparentObj?.ToString();
    var tracestate = tracestateObj?.ToString();

    ActivityContext parentContext = default;
    if (!string.IsNullOrWhiteSpace(traceparent))
    {
        parentContext = ActivityContext.Parse(traceparent, tracestate);
    }

    using var activity = source.StartActivity(
        "ProcessServiceBusMessage",
        ActivityKind.Consumer,
        parentContext);

    // process message
    await Task.CompletedTask;
}
```

This is the preferred pattern because it creates a proper child span from the extracted parent context. It is more reliable than trying to reconstruct parentage with ad hoc span ID parameters.

**Correlation in logs and telemetry**

Correlation is not just for traces. You also want logs to include the same identifiers so you can jump from a failed request to the exact log lines for that request.

Useful fields to include:

- `traceId`
- `spanId`
- `operationId`
- `parentId`
- custom business identifiers such as `orderId` or `tenantId`

In .NET, structured logging with `ILogger` can capture scope values:

```csharp
using (_logger.BeginScope(new Dictionary<string, object>
{
    ["orderId"] = orderId,
    ["traceId"] = Activity.Current?.TraceId.ToString()
}))
{
    _logger.LogInformation("Processing order");
}
```

Benefits:

- Logs become searchable by trace or business key.
- You can correlate application logs with dependency calls.
- You reduce the time needed to diagnose distributed failures.

**Application Insights and Azure Monitor behavior**

Application Insights is commonly used as an Azure Monitor feature for application telemetry. It collects requests, dependencies, exceptions, traces, and custom events, then correlates them into end-to-end operation views.

Typical telemetry types:

- **Requests**: inbound operations handled by your app.
- **Dependencies**: outbound calls to HTTP, SQL, Service Bus, and other services.
- **Exceptions**: unhandled or captured errors.
- **Traces**: log-style diagnostic messages.
- **Custom events/metrics**: business or domain telemetry.

A common .NET configuration pattern is to provide the connection string through environment variables or configuration and wire up telemetry in code:

```csharp
builder.Services.AddApplicationInsightsTelemetry(options =>
{
    options.ConnectionString = builder.Configuration["ApplicationInsights:ConnectionString"];
});
```

Or via environment variable:

```bash
export APPLICATIONINSIGHTS_CONNECTION_STRING="InstrumentationKey=...;IngestionEndpoint=..."
```

Important guidance:

- The connection string must be consumed by the telemetry SDK or OpenTelemetry exporter.
- Configuration alone does not emit telemetry.
- Prefer the current Azure Monitor/OpenTelemetry guidance for new applications when possible.

**OpenTelemetry setup example**

OpenTelemetry provides a vendor-neutral way to collect traces and export them to Azure Monitor or another backend.

```csharp
using OpenTelemetry.Trace;

builder.Services.AddOpenTelemetry()
    .WithTracing(tracing =>
    {
        tracing
            .AddAspNetCoreInstrumentation()
            .AddHttpClientInstrumentation()
            .AddSource("Orders.Api")
            .SetSampler(new ParentBasedSampler(new TraceIdRatioBasedSampler(1.0)))
            .AddOtlpExporter();
    });
```

What this does:

- **AddAspNetCoreInstrumentation** captures inbound HTTP requests.
- **AddHttpClientInstrumentation** captures outbound HTTP calls.
- **AddSource** includes your custom spans.
- **Sampler** controls how much telemetry you keep.
- **AddOtlpExporter** sends telemetry to an OTLP-compatible collector.

If you export to Azure Monitor, you typically configure the Azure Monitor exporter or collector pipeline rather than relying on raw JSON settings alone.

**Sampling and performance**

Tracing every request can be expensive in high-volume systems. Sampling reduces telemetry volume while preserving useful diagnostic data.

Common sampling approaches:

- **AlwaysOn**: capture everything; useful in development or low-volume systems.
- **Ratio-based sampling**: capture a percentage of traces.
- **Parent-based sampling**: preserve consistency across a distributed trace.

Trade-offs:

- Higher sampling gives better visibility but increases cost and storage.
- Lower sampling reduces overhead but may miss rare failures.
- Parent-based sampling helps keep all spans in a trace consistent.

**Custom dimensions and baggage**

Use custom tags for span-local metadata and baggage for data that must flow downstream.

Examples:

- Tags: `order.id`, `payment.method`, `region`
- Baggage: `tenantId`, `correlationId`, `locale`

Example baggage usage:

```csharp
Activity.Current?.AddBaggage("tenantId", tenantId);
Activity.Current?.AddBaggage("locale", "en-US");
```

Guidance:

- Keep baggage small and non-sensitive.
- Do not put secrets, tokens, or personal data in baggage.
- Use tags for observability, not for business logic decisions.

**Practical topology for end-to-end tracing**

A common Azure architecture might look like this:

- Web API receives an HTTP request.
- API writes a message to Service Bus.
- Worker service consumes the message.
- Worker calls a downstream REST API and SQL database.
- All components emit telemetry to Azure Monitor.

In this topology, the trace should flow through:

- HTTP request span in the API
- Producer span for the queue send
- Consumer span in the worker
- HTTP client span for the downstream API
- Dependency span for SQL

This gives you one trace that explains the full transaction path.

**Troubleshooting checklist**

If traces do not connect correctly, check the following:

- The same W3C context is used across services.
- `traceparent` and `tracestate` are forwarded intact.
- Message metadata preserves headers or application properties.
- Automatic instrumentation is enabled for HTTP clients and servers.
- The app is not creating a new root activity for every hop.
- Sampling is not dropping the spans you expect to see.
- Clock skew is not making spans appear out of order.

> If you see separate traces for what should be one transaction, the most common cause is missing propagation at an HTTP boundary or message boundary.

**Exam-focused distinctions**

For AZ-204, focus on the practical differences between tracing, correlation, and telemetry collection.

- **Tracing** shows the path and timing of a request across services.
- **Correlation** links logs, requests, and dependencies using shared identifiers.
- **Instrumentation** is the code or configuration that emits telemetry.
- **Propagation** is the transfer of trace context between services.

You should also recognize when to use:

- **Automatic instrumentation** for standard frameworks and dependencies.
- **Manual instrumentation** for custom operations or nonstandard transports.
- **OpenTelemetry** for portable observability pipelines.
- **Application Insights/Azure Monitor** for Azure-native telemetry analysis.

**Quick reference**

| Concept | Purpose | Typical mechanism |
|---|---|---|
| Trace | End-to-end request path | `traceparent`, `Activity`, OpenTelemetry |
| Span | One timed operation | `ActivitySource.StartActivity()` |
| Correlation | Link related telemetry | Shared trace ID and structured logs |
| Baggage | Propagate small metadata | `Activity.AddBaggage()` |
| Dependency telemetry | Record outbound calls | HTTP/SQL/Service Bus instrumentation |

**Minimal end-to-end example**

```csharp
using System.Diagnostics;
using Azure.Messaging.ServiceBus;

var source = new ActivitySource("Checkout.Api");

using var activity = source.StartActivity("CreateOrder", ActivityKind.Server);
activity?.SetTag("order.id", 123);

var message = new ServiceBusMessage(BinaryData.FromString("{\"orderId\":123}"));

if (Activity.Current != null)
{
    message.ApplicationProperties["traceparent"] = Activity.Current.Id;
    if (!string.IsNullOrEmpty(Activity.Current.TraceStateString))
    {
        message.ApplicationProperties["tracestate"] = Activity.Current.TraceStateString;
    }
}

await sender.SendMessageAsync(message);
```

This example shows the overall pattern, but in production you should prefer framework and SDK instrumentation where possible so context propagation is handled consistently.
