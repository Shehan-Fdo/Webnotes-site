## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Overview**

Troubleshooting in Azure is usually about separating the symptom from the root cause. A failed deployment, a 401 response, a missing trigger invocation, a stuck message, a Cosmos DB 404, or a container that never starts can all come from configuration, identity, networking, runtime, or platform constraints. For AZ-204, you need to recognize the failure pattern, know where to inspect logs, and know which Azure feature is most likely responsible.

A good diagnostic workflow is:

1. Confirm the scope of the failure.
2. Check platform logs and resource health.
3. Validate identity, configuration, and network access.
4. Reproduce with the smallest possible request or deployment.
5. Compare the runtime behavior with the expected Azure service contract.

**Deployment Failure Diagnostics**

Deployment failures often come from template validation, parameter mismatches, permissions, or pipeline environment issues. In Azure Resource Manager (ARM) and Bicep deployments, the first question is whether the failure happens before resource creation or during resource provisioning.

Common checks include:

- **Template validation**: Catch syntax, type, and schema errors before deployment.
- **What-if analysis**: Preview the changes a deployment will make.
- **Deployment operations**: Inspect the exact resource that failed and the error returned by Azure.
- **Pipeline logs**: Review CI/CD output for authentication, artifact, or task failures.

Useful Azure CLI commands:

```bash
# Validate an ARM template
az deployment group validate \
  --resource-group rg-app \
  --template-file main.json \
  --parameters @params.json

# Preview changes
az deployment group what-if \
  --resource-group rg-app \
  --template-file main.bicep \
  --parameters @params.json

# Inspect deployment operations
az deployment group operation list \
  --resource-group rg-app \
  --name deploy-20260426
```

For Bicep, compile errors and type mismatches are often caught locally before deployment. For ARM, malformed JSON, invalid resource IDs, or unsupported API versions are common causes. If a deployment succeeds partially, inspect the deployment operations rather than only the top-level deployment status.

**Common deployment failure patterns**

- **Invalid parameter values**: A string is passed where an integer or object is expected.
- **Missing dependencies**: A resource references another resource that is not yet created.
- **RBAC (Role-Based Access Control) permission gaps**: The deployment identity lacks rights to create or modify a resource.
- **Name or location constraints**: Resource names are globally unique or region-specific.
- **Pipeline authentication failures**: Service connections, federated credentials, or secrets are expired or misconfigured.

For CI/CD, verify the identity used by the pipeline and the scope of its permissions. A deployment can fail even when the template is correct if the service principal or managed identity cannot write to the target resource group.

**Authorization and Managed Identity Troubleshooting**

Authorization failures usually appear as 401 Unauthorized or 403 Forbidden responses. The key distinction is that 401 means authentication failed or no valid token was presented, while 403 means the token is valid but the caller lacks permission.

When troubleshooting Azure authorization, check:

- **Identity type**: System-assigned managed identity, user-assigned managed identity, service principal, or user token.
- **Token acquisition**: Whether the app can obtain a token for the correct resource scope.
- **Role assignment scope**: Subscription, resource group, resource, or data-plane scope.
- **Propagation delay**: RBAC changes can take time to become effective.

Managed identity troubleshooting steps:

1. Confirm the identity is enabled on the resource.
2. Verify the identity has the correct role assignment at the correct scope.
3. Ensure the app requests a token for the right audience or resource.
4. Check whether the code uses the correct identity type, especially when both system-assigned and user-assigned identities exist.
5. Review logs for token acquisition failures, authority errors, or scope mismatches.

Example Azure CLI checks:

```bash
# Show managed identity on an App Service or Function App
az webapp identity show \
  --resource-group rg-app \
  --name myapp

# List role assignments for a principal
az role assignment list \
  --assignee <principal-id> \
  --all \
  --output table
```

In code, prefer the Azure Identity library when possible. For example, `DefaultAzureCredential` can simplify local-to-cloud authentication, but you still need to understand which credential it is using when troubleshooting.

```csharp
using Azure.Identity;
using Azure.Storage.Blobs;

var credential = new DefaultAzureCredential();
var client = new BlobServiceClient(new Uri("https://mystorageaccount.blob.core.windows.net"), credential);
```

If authentication works locally but fails in Azure, compare the local developer identity with the managed identity or service principal used in the cloud. Also verify that the target service supports the authorization model you are using. Some services use Azure RBAC, while others use data-plane permissions or access policies.

**Trigger Failure Diagnostics for Azure Functions**

Trigger failures happen when the function host does not detect, index, or invoke the function as expected. The issue may be in the trigger binding, the host configuration, the extension bundle, the connection string, or the upstream event source.

Common trigger types to know for AZ-204:

- **HTTP trigger**: Invoked by an HTTP request.
- **Timer trigger**: Runs on a schedule.
- **Blob trigger**: Fires when a blob is created or updated.
- **Queue trigger**: Processes messages from Azure Storage queues.
- **Service Bus trigger**: Responds to messages or sessions from Service Bus.
- **Event Grid trigger**: Handles event notifications from Event Grid.

A function may not be indexed if the host cannot load the binding extension or if the trigger configuration is invalid. In that case, check host logs, startup logs, and the Azure portal function list. Do not assume the function is healthy just because the app is running.

**Timer trigger troubleshooting**

- Verify the CRON expression or schedule format.
- Confirm the app is running continuously enough to catch scheduled executions.
- Check time zone behavior and daylight saving effects.
- Review `host.json` if schedule monitoring or concurrency settings affect execution.

Example timer trigger:

```csharp
[FunctionName("CleanupJob")]
public static void Run(
    [TimerTrigger("0 */5 * * * *")] TimerInfo myTimer,
    ILogger log)
{
    log.LogInformation("Timer fired at {time}", DateTime.UtcNow);
}
```

**Queue, blob, and Service Bus trigger troubleshooting**

- Confirm the connection setting name matches the binding attribute.
- Verify the storage account or Service Bus namespace exists and is reachable.
- Check poison message handling and retry behavior.
- Inspect host logs for binding conversion errors or deserialization failures.

For Service Bus triggers, also check:

- **Sessions**: If the entity requires sessions, the function must be session-enabled.
- **Lock renewal**: Long-running handlers may need lock renewal to avoid duplicate processing.
- **Duplicate detection**: A message may appear missing if it is rejected by duplicate detection.
- **Prefetch and concurrency**: High throughput settings can expose ordering or lock issues.

For Event Grid triggers, verify the subscription endpoint, validation handshake, and event schema. If events are not arriving, confirm the subscription is active and the destination endpoint is reachable.

**Messaging Failure Diagnostics**

Messaging issues often involve delivery semantics, lock handling, dead-lettering, or consumer configuration. The service behavior matters: Azure Storage queues, Service Bus, and Event Hubs each fail differently.

**Azure Storage queues**

Storage queue problems usually involve connection strings, message encoding, visibility timeout, or poison message handling. A message can reappear if the consumer does not delete it before the visibility timeout expires.

Example queue processing pattern:

```csharp
public static async Task Run(
    [QueueTrigger("orders")] string message,
    ILogger log)
{
    log.LogInformation("Processing message: {message}", message);
    // Process message
}
```

Troubleshooting points:

- Verify the queue name and storage account connection setting.
- Check whether the message payload exceeds service limits.
- Confirm the consumer deletes or completes the message after successful processing.
- Review retry behavior when the handler throws exceptions.

**Azure Service Bus**

Service Bus supports queues and topics/subscriptions, and it adds features such as sessions, duplicate detection, dead-letter queues, scheduled messages, and lock renewal. These features are common exam targets.

Troubleshooting points:

- **Dead-letter queue (DLQ)**: Messages move to the DLQ after repeated delivery failures, expiration, or explicit dead-lettering by the application.
- **Sessions**: If a queue or subscription requires sessions, the receiver must accept a session.
- **Duplicate detection**: Messages with the same message ID inside the duplicate detection window may be ignored.
- **Lock renewal**: Long processing times can cause lock loss and duplicate delivery.
- **Subscription rules**: Filters control routing, not dead-lettering by themselves.

Example Service Bus trigger:

```csharp
[FunctionName("ProcessOrder")]
public static void Run(
    [ServiceBusTrigger("orders", Connection = "ServiceBusConnection")] string message,
    ILogger log)
{
    log.LogInformation("Received: {message}", message);
}
```

If messages are not arriving, verify:

1. The sender is publishing to the correct queue or topic.
2. The subscription exists and the filter matches the message.
3. The consumer has the correct connection string or managed identity permissions.
4. The entity is not session-enabled when the consumer is non-session-aware.
5. The message is not being dead-lettered due to processing failures or expiration.

**Event Hubs**

Event Hubs troubleshooting often centers on checkpointing, consumer groups, and partition ownership. If a consumer appears stuck or reprocessing events, check the checkpoint store and lease ownership.

Key checks:

- Confirm the consumer group is correct.
- Verify the checkpoint store is accessible and writable.
- Ensure the processor has permission to read the event hub and write checkpoints.
- Check partition balancing and lease acquisition.

A missing or stale checkpoint can cause reprocessing after restart. A consumer group mismatch can make it look like no events are arriving when another consumer group is actually receiving them.

**Cosmos DB Failure Diagnostics**

Cosmos DB issues usually fall into one of four categories: point reads, queries, throughput, or connectivity/consistency. The most important distinction is between a point read and a query.

A **point read** requires the exact `id` and partition key value. If the partition key is wrong, the request targets the wrong logical partition and returns not found. That does not mean the item is somewhere else from the perspective of that read; it means the request is incomplete for the item you want.

Example point read pattern:

```csharp
var item = await container.ReadItemAsync<MyItem>(
    id: "order-123",
    partitionKey: new PartitionKey("customer-42"));
```

Troubleshooting points:

- **404 Not Found**: Check the `id` and partition key pair for point reads.
- **400 Bad Request**: Often indicates malformed SQL, invalid partition key usage, or unsupported query shape.
- **429 Too Many Requests**: The request rate exceeds provisioned throughput.
- **Timeouts**: Can indicate network issues, high RU (Request Unit) consumption, or inefficient queries.

For queries, inspect the query plan and RU consumption. A query that scans many partitions or uses non-sargable predicates can be expensive and slow.

```csharp
var query = new QueryDefinition(
    "SELECT * FROM c WHERE c.customerId = @customerId")
    .WithParameter("@customerId", "customer-42");
```

**Cosmos DB query troubleshooting**

- Prefer point reads when you know the `id` and partition key.
- Ensure the partition key is included in the query when possible.
- Review indexing policy if queries are unexpectedly expensive.
- Check whether the SDK is using the correct consistency level and region.

If reads are inconsistent across regions, verify the configured consistency level and any failover or multi-region write settings. If the SDK cannot connect, confirm endpoint access, firewall rules, private endpoint configuration, and DNS resolution.

**Container Startup Failure Diagnostics**

Container startup failures usually come from image pull issues, port binding problems, missing environment variables, bad entrypoints, or dependency initialization errors. The exact symptoms differ between App Service for Containers, Azure Container Apps, and AKS, but the diagnostic approach is similar.

**App Service for Containers**

For custom containers in App Service, the app must listen on the expected port and bind to the correct interface. In Linux custom containers, the platform often expects the container to listen on the configured port, commonly communicated through `WEBSITES_PORT` or container settings. This is not universal across all container hosting modes, so verify the platform-specific configuration.

Troubleshooting points:

- Confirm the image starts locally with the same environment variables.
- Verify the container listens on `0.0.0.0`, not only `localhost`.
- Check the startup command, entrypoint, and exposed port.
- Review container logs and platform logs.

Example Dockerfile:

```dockerfile
FROM mcr.microsoft.com/dotnet/aspnet:8.0
WORKDIR /app
COPY . .
ENV ASPNETCORE_URLS=http://+:8080
EXPOSE 8080
ENTRYPOINT ["dotnet", "MyApp.dll"]
```

**Azure Container Apps**

Container Apps failures often involve revision startup, probes, secrets, or environment configuration. A deployment manifest must include the full app context, not just a container image snippet. The container app also depends on the environment, ingress settings, and revision configuration.

Common issues:

- Missing or incorrect image reference.
- Secret or environment variable not injected.
- Ingress target port mismatch.
- Health probe failure causing restarts.
- Registry authentication failure.

Example manifest fragment:

```yaml
properties:
  managedEnvironmentId: /subscriptions/<sub>/resourceGroups/<rg>/providers/Microsoft.App/managedEnvironments/<env>
  configuration:
    ingress:
      external: true
      targetPort: 8080
    secrets:
      - name: registry-password
        value: <secret>
  template:
    containers:
      - name: app
        image: myregistry.azurecr.io/myapp:1.0
        env:
          - name: ASPNETCORE_URLS
            value: http://+:8080
```

**AKS troubleshooting**

In Azure Kubernetes Service (AKS), startup failures can come from image pull errors, bad secrets, DNS issues, probe failures, or application crashes. `CrashLoopBackOff` means the container is repeatedly restarting; that can happen because the process exits or because liveness probes keep failing.

Useful commands:

```bash
kubectl get pods
kubectl describe pod myapp-7d9f8c9f7b-abcde
kubectl logs myapp-7d9f8c9f7b-abcde
kubectl logs myapp-7d9f8c9f7b-abcde --previous
kubectl get events --sort-by=.metadata.creationTimestamp
```

Common AKS causes:

- **ImagePullBackOff**: Wrong image name, tag, registry credentials, or network access.
- **CrashLoopBackOff**: Application exits, missing config, or failing liveness probe.
- **Pending**: Insufficient node resources, taints, or scheduling constraints.
- **Readiness probe failures**: The pod starts but is not ready to receive traffic.
- **DNS failures**: Service discovery or external name resolution is broken.
- **Secret or config map issues**: Missing environment variables or invalid mounted data.

If the container runs locally but fails in AKS, compare the runtime environment carefully. Differences in secrets, mounted volumes, service accounts, network policies, and resource limits often explain the gap.

**Practical Diagnostic Workflow**

Use a consistent sequence so you do not chase symptoms in the wrong layer.

1. Identify the failing layer: deployment, identity, trigger, messaging, data, or container runtime.
2. Check the most specific logs first: deployment operations, function host logs, `kubectl logs`, or service diagnostics.
3. Validate configuration values: connection strings, managed identity, partition keys, ports, and environment variables.
4. Reproduce with a minimal request or message.
5. Confirm the platform contract: required port, trigger binding, session support, partition key, or RBAC scope.

**Quick Symptom-to-Cause Mapping**

| Symptom | Likely area | First checks |
|---|---|---|
| Deployment fails before resources are created | ARM/Bicep validation | Template syntax, parameters, `what-if`, permissions |
| 401 or 403 | Authorization | Token acquisition, identity, RBAC scope |
| Function never fires | Trigger binding | Connection setting, extension, schedule, source configuration |
| Messages reappear or duplicate | Messaging | Lock timeout, retries, dead-lettering, duplicate detection |
| Cosmos DB 404 on read | Data access | Exact `id` and partition key |
| Container restarts repeatedly | Runtime startup | Logs, probes, port binding, image pull, secrets |

**Exam-Focused Reminders**

- **Deployment**: Use validation, `what-if`, and deployment operations before guessing.
- **Authorization**: Distinguish authentication from authorization, and verify managed identity scope.
- **Triggers**: Check host logs, binding configuration, and upstream event source health.
- **Messaging**: Know the differences between queues, topics, sessions, duplicate detection, and DLQ behavior.
- **Cosmos DB**: Point reads require the exact `id` and partition key; queries have different failure modes.
- **Containers**: Check image pull, port binding, probes, secrets, and platform-specific startup expectations.
