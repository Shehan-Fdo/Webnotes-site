## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Application observability in Azure**

Application Insights is the Azure Monitor application performance management service used to collect telemetry from apps, services, and dependencies. You use it to understand request rates, failures, latency, dependency health, and user behavior. Structured logging complements telemetry by emitting machine-readable log records with consistent fields, making queries, alerts, and correlation much more effective.

The main goal is not just to “log more,” but to log in a way that supports diagnosis. Good instrumentation helps you answer:
- What failed?
- Where did it fail?
- Which dependency was involved?
- Which user, operation, or request was affected?
- Is the issue isolated or systemic?

**Core telemetry types**

Application Insights organizes data into several useful categories. In practice, you often combine them to reconstruct a request flow.

- **Requests** track incoming operations such as HTTP calls or message handlers.
- **Dependencies** track outbound calls such as SQL, HTTP, Service Bus, or storage operations.
- **Exceptions** capture unhandled or explicitly reported failures.
- **Traces** capture diagnostic messages and structured log entries.
- **Metrics** capture numeric measurements such as queue length, latency, or custom business counters.
- **Page views** and **browser telemetry** apply to client-side apps and web experiences.

A typical request path might look like this:
- An API receives an HTTP request.
- The API writes a trace with a correlation identifier.
- The API calls a database and an external API.
- Application Insights records the request, both dependencies, and any exception.
- You query all related telemetry by operation ID.

**Why structured logging matters**

Structured logging stores data as named fields instead of only as a formatted string. This makes logs easier to filter, aggregate, and join with other telemetry.

For example, compare these approaches:

```text
Unstructured:
Order 12345 failed for customer 9981

Structured:
OrderFailed OrderId=12345 CustomerId=9981 Reason=PaymentDeclined
```

Structured logs are better because you can:
- Filter by `OrderId` or `CustomerId`.
- Count failures by `Reason`.
- Build alerts on specific fields.
- Correlate logs with requests and dependencies.

In .NET, structured logging commonly uses message templates:

```csharp
_logger.LogInformation("Order {OrderId} processed for customer {CustomerId}", orderId, customerId);
```

The placeholders become named properties rather than string concatenation. This is important because query engines can index and search those fields more reliably.

**Application Insights instrumentation options**

You can instrument applications in several ways, often together:

- **Automatic instrumentation** captures common telemetry with minimal code changes.
- **Manual instrumentation** uses SDK APIs to add custom events, metrics, traces, and exceptions.
- **OpenTelemetry** can export traces, metrics, and logs to Azure Monitor in modern setups.
- **Logger integration** sends application logs through the Application Insights provider or Azure Monitor pipeline.

Choose automatic instrumentation when you want quick coverage for standard web and dependency telemetry. Add manual instrumentation when you need business-specific events, custom dimensions, or precise control over what gets recorded.

**Connection string and setup**

Application Insights is typically configured with a connection string. This identifies the resource and the ingestion endpoint.

Example environment variable:

```bash
APPLICATIONINSIGHTS_CONNECTION_STRING="InstrumentationKey=00000000-0000-0000-0000-000000000000;IngestionEndpoint=https://westus-0.in.applicationinsights.azure.com/"
```

In ASP.NET Core, you commonly register Application Insights in `Program.cs`:

```csharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddApplicationInsightsTelemetry();

var app = builder.Build();
app.MapGet("/", () => "Hello world");
app.Run();
```

If you need to set the connection string explicitly:

```csharp
builder.Services.AddApplicationInsightsTelemetry(options =>
{
    options.ConnectionString = builder.Configuration["APPLICATIONINSIGHTS_CONNECTION_STRING"];
});
```

For non-web or worker applications, you still use the SDK or OpenTelemetry exporter, but the telemetry source differs. The key idea is the same: initialize the pipeline early so telemetry is captured from startup onward.

**Structured logging with Microsoft.Extensions.Logging**

The built-in `ILogger<T>` abstraction is the standard logging API in .NET. It supports structured messages, scopes, and log levels.

Example:

```csharp
public class OrderService
{
    private readonly ILogger<OrderService> _logger;

    public OrderService(ILogger<OrderService> logger)
    {
        _logger = logger;
    }

    public async Task ProcessAsync(string orderId, string customerId)
    {
        using (_logger.BeginScope(new Dictionary<string, object>
        {
            ["OrderId"] = orderId,
            ["CustomerId"] = customerId
        }))
        {
            _logger.LogInformation("Processing order");
            // business logic
            _logger.LogInformation("Order processed successfully");
        }
    }
}
```

Scopes are useful because they attach contextual properties to all logs within a logical operation. This helps you correlate multiple log lines without repeating every field manually.

Common log levels:
- **Trace** for very detailed diagnostic information.
- **Debug** for development-time troubleshooting.
- **Information** for normal operational events.
- **Warning** for unexpected but non-fatal conditions.
- **Error** for failures that affect a specific operation.
- **Critical** for severe failures that may stop the application.

Use log levels intentionally:
- Use `Information` for business milestones and successful operations.
- Use `Warning` for recoverable issues or degraded behavior.
- Use `Error` for failed operations that need attention.
- Avoid excessive `Debug` or `Trace` in production unless you can control volume.

**Custom telemetry**

Application Insights supports custom telemetry when built-in signals are not enough.

Common custom items:
- **Custom events** for business actions such as `CheckoutCompleted`.
- **Custom metrics** for counters such as `ItemsProcessed`.
- **Custom traces** for diagnostic breadcrumbs.
- **Custom exceptions** when you catch and rethrow or want to record handled failures.

Example using the Application Insights SDK:

```csharp
using Microsoft.ApplicationInsights;

public class CheckoutService
{
    private readonly TelemetryClient _telemetry;

    public CheckoutService(TelemetryClient telemetry)
    {
        _telemetry = telemetry;
    }

    public void CompleteCheckout(string orderId, decimal total)
    {
        _telemetry.TrackEvent("CheckoutCompleted", new Dictionary<string, string>
        {
            ["OrderId"] = orderId
        }, new Dictionary<string, double>
        {
            ["Total"] = (double)total
        });
    }
}
```

Use custom events for business-level signals that are not naturally represented as HTTP requests or dependencies. This is especially useful in background jobs, queue processors, and domain workflows.

**Correlation and distributed tracing**

Correlation ties telemetry from one operation together across services. In Azure, this is essential for microservices, APIs, and message-driven systems.

Key concepts:
- **Operation ID** identifies a logical end-to-end transaction.
- **Parent ID** identifies the immediate caller or parent span.
- **Trace context** carries identifiers across service boundaries.
- **Distributed tracing** reconstructs the path across multiple components.

In modern .NET and Azure ecosystems, correlation often follows the W3C Trace Context standard using headers such as `traceparent` and `tracestate`.

Practical guidance:
- Propagate correlation headers on outbound HTTP calls.
- Include operation identifiers in logs when needed.
- Avoid generating new correlation IDs for each internal step unless you are intentionally starting a new operation.
- Ensure queue messages or events carry enough context to continue the trace.

Example of adding a custom property to logs for correlation:

```csharp
_logger.LogInformation("Calling payment service for order {OrderId} with CorrelationId {CorrelationId}",
    orderId,
    correlationId);
```

**Dependency tracking**

Dependency telemetry records outbound calls. This is one of the most valuable parts of Application Insights because it shows where latency and failures originate.

Typical dependencies:
- HTTP calls to external APIs
- SQL queries
- Azure Storage operations
- Service Bus sends and receives
- Redis calls

A healthy request may still be slow because of a dependency. Dependency telemetry helps you isolate whether the bottleneck is in your code or in an external service.

Useful fields to inspect:
- Dependency name
- Target host or resource
- Duration
- Success or failure
- Result code
- Type

**KQL queries for investigation**

Application Insights uses Kusto Query Language (KQL) in Logs. You use it to search, aggregate, and correlate telemetry.

Example: recent failed requests

```kusto
requests
| where success == false
| order by timestamp desc
| project timestamp, name, resultCode, duration, operation_Id
```

Example: join requests with dependencies by operation ID

```kusto
requests
| where timestamp > ago(1h)
| project operation_Id, requestName = name, requestDuration = duration
| join kind=leftouter (
    dependencies
    | project operation_Id, dependencyName = name, dependencyDuration = duration, target, success
) on operation_Id
| order by requestDuration desc
```

Example: search structured log properties

```kusto
traces
| where customDimensions.OrderId == "12345"
| project timestamp, message, severityLevel, customDimensions
```

Example: count errors by message template

```kusto
traces
| where severityLevel >= 3
| summarize count() by message
| order by count_ desc
```

KQL is one of the most important skills for this topic because instrumentation is only useful if you can query it effectively.

**Sampling and telemetry volume**

Sampling reduces telemetry volume and cost by keeping only a representative subset of events. This is important for high-throughput systems.

Benefits:
- Lower ingestion cost
- Less storage pressure
- Better performance in noisy systems

Trade-offs:
- You may miss rare events if sampling is too aggressive.
- Counts and rates may need adjustment or estimation.
- Debugging a single incident can be harder if the relevant trace was dropped.

Use sampling carefully:
- Keep enough telemetry to diagnose failures.
- Prefer adaptive sampling when available for dynamic workloads.
- Disable or reduce sampling temporarily during incident investigation if needed.

> Be careful not to confuse low telemetry volume with low traffic. Sampling can hide real spikes or rare failures if you do not account for it.

**Configuration examples**

ASP.NET Core logging configuration in `appsettings.json`:

```json
{
  "Logging": {
    "LogLevel": {
      "Default": "Information",
      "Microsoft": "Warning",
      "System": "Warning",
      "MyApp.Services.OrderService": "Debug"
    }
  }
}
```

This lets you keep framework noise low while increasing verbosity for your own code.

Example `Program.cs` with logging and Application Insights:

```csharp
var builder = WebApplication.CreateBuilder(args);

builder.Logging.ClearProviders();
builder.Logging.AddConsole();
builder.Services.AddApplicationInsightsTelemetry();

builder.Services.AddScoped<OrderService>();

var app = builder.Build();

app.MapPost("/orders/{id}", async (string id, OrderService service) =>
{
    await service.ProcessAsync(id, "cust-001");
    return Results.Accepted();
});

app.Run();
```

Example of logging an exception with context:

```csharp
try
{
    await paymentClient.ChargeAsync(orderId, amount);
}
catch (HttpRequestException ex)
{
    _logger.LogError(ex, "Payment call failed for order {OrderId}", orderId);
    throw;
}
```

Always pass the exception object as the first argument after the log level so the stack trace is preserved.

**Best practices**

- **Log with context** by including identifiers such as `OrderId`, `UserId`, `CorrelationId`, or `TenantId`.
- **Prefer templates** over string interpolation for structured logs.
- **Use scopes** for operation-wide context.
- **Record exceptions once** at the boundary where you can add useful context.
- **Keep telemetry actionable** by avoiding noisy or redundant messages.
- **Protect sensitive data** by excluding secrets, tokens, passwords, and personal data.
- **Instrument business events** that matter to support and product analysis.
- **Validate correlation** across services, queues, and background jobs.
- **Monitor dependency latency** as well as failures.
- **Use KQL regularly** to verify that your telemetry is queryable and useful.

**Common pitfalls**

- Logging only free-form text, which makes filtering difficult.
- Emitting too much `Debug` or `Trace` data in production.
- Forgetting to propagate correlation IDs across service boundaries.
- Logging sensitive values such as access tokens or connection strings.
- Catching exceptions and not rethrowing or recording them properly.
- Relying only on request telemetry and ignoring dependencies.
- Using custom events for everything instead of choosing the right telemetry type.

**Practical troubleshooting workflow**

When an application issue occurs, work from the symptom backward through the telemetry chain.

1. Find the failed request or operation in `requests`.
2. Inspect the `operation_Id` to gather related telemetry.
3. Review `dependencies` for slow or failed outbound calls.
4. Check `traces` for structured context and breadcrumbs.
5. Inspect `exceptions` for stack traces and root causes.
6. Compare the issue against recent deployments, configuration changes, or traffic spikes.

This workflow is effective because it uses correlation to reconstruct the full path of the failure instead of treating logs as isolated text lines.

**When to use Application Insights versus logs**

Application Insights is best for telemetry that describes application behavior over time:
- Request success and latency
- Dependency health
- Exceptions and failures
- User and session behavior
- Custom business events and metrics

Structured logs are best for detailed diagnostic context:
- Step-by-step execution breadcrumbs
- Business identifiers
- Validation failures
- State transitions
- Debugging complex flows

In practice, you use both. Application Insights gives you the observability platform, and structured logging gives you the searchable detail that makes the telemetry useful.

**Quick reference**

- **Application Insights**: Azure service for application telemetry and performance monitoring.
- **Structured logging**: Logging with named fields instead of plain text only.
- **Correlation**: Linking telemetry across a single logical operation.
- **Dependency**: Any outbound call from your application.
- **KQL**: Kusto Query Language used to query telemetry in Azure Monitor Logs.
