## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Operational monitoring in Azure**

Operational monitoring helps you detect failures, performance regressions, and unusual behavior before users report them. In Azure, you usually combine **metrics** for near-real-time signals, **logs** for deep investigation, and **alerts** to notify or automate response. Dashboards and workbooks then turn that data into a view that teams can act on quickly.

For AZ-204, focus on choosing the right signal source and alert type for the problem you are solving:

- **Metrics** are best for fast, numeric signals such as CPU, latency, request count, or availability percentage.
- **Logs** are best for rich queries, correlation, and trend analysis across multiple resources.
- **Activity logs** capture subscription-level control-plane events such as resource creation, deletion, and policy changes.
- **Dashboards** provide a pinned, shared operational view.
- **Workbooks** provide interactive, parameterized analysis and are better for investigation than static dashboards.

**Core monitoring building blocks**

Azure Monitor is the umbrella service for collecting, analyzing, and acting on telemetry. It includes metrics, logs, alerts, action groups, dashboards, and workbooks. In practice, you often wire these together as follows:

1. Application or platform emits telemetry.
2. Azure Monitor ingests metrics and logs.
3. Alert rules evaluate conditions.
4. Action groups notify people or trigger automation.
5. Dashboards and workbooks visualize the current state and trends.

A common exam trap is confusing the data source with the alert type. The source determines what you can measure; the alert type determines how Azure evaluates it.

- **Metric alerts** evaluate numeric time-series data.
- **Log alerts** evaluate query results from Log Analytics.
- **Activity log alerts** evaluate Azure control-plane events.
- **Smart detection** and similar AI-assisted features are not the same as custom alert rules.

**Alert types and when to use them**

Choose the alert type based on the question you need answered.

| Alert type | Best for | Typical scope | Strengths | Limitations |
|---|---|---|---|---|
| Metric alert | Fast thresholding on numeric telemetry | Resource, resource group, subscription | Low latency, simple, efficient | Limited query logic |
| Log alert | Complex conditions, correlation, custom logic | Log Analytics workspace or resource logs | Flexible KQL (Kusto Query Language) | Slightly higher latency, more setup |
| Activity log alert | Control-plane events | Subscription | Detects administrative changes | Not application telemetry |
| Service health alert | Azure platform incidents affecting your resources | Subscription | Useful for outages and advisories | Based on Azure service health data, not app metrics |

A practical rule:

- Use a **metric alert** when you know the threshold and need quick notification.
- Use a **log alert** when you need filtering, joins, aggregation, or a custom incident definition.
- Use an **activity log alert** when you care about who changed what in Azure.
- Use a **service health alert** when you need to know about Azure platform issues that may affect your workloads.

> Do not use activity log alerts as a substitute for application telemetry. They tell you about Azure management events, not whether your app is healthy.

**Metrics: fast signal for operational thresholds**

Metrics are time-series values collected at regular intervals. They are ideal for alerting on conditions that are easy to express numerically.

Common examples:

- CPU percentage above a threshold for several minutes.
- HTTP 5xx error rate rising above normal.
- Queue length growing unexpectedly.
- Availability dropping below a target.
- Response time exceeding an SLA.

Metric alerts are usually the first choice for production operations because they are fast and low overhead. They also work well for autoscaling triggers and simple incident detection.

**Metric alert design considerations**

- Use a **window size** long enough to avoid noise, but short enough to detect incidents quickly.
- Use **aggregation** such as Average, Maximum, Minimum, or Total depending on the metric semantics.
- Use **dimension filters** to target a specific instance, endpoint, or region.
- Use **dynamic thresholds** when the baseline changes frequently and static thresholds are too brittle.
- Use **multiple conditions** only when the platform supports them for the alert type and resource.

For example, a web app may emit a request count metric and a response time metric. A request count spike alone is not necessarily bad; a spike combined with rising latency may indicate a traffic surge or a downstream dependency issue.

**Example: metric alert with Azure CLI**

The modern Azure CLI uses the `az monitor metrics alert` command family for metric alerts.

```bash
az monitor metrics alert create \
  --name "app-high-5xx-rate" \
  --resource-group "rg-prod" \
  --scopes "/subscriptions/<subId>/resourceGroups/rg-prod/providers/Microsoft.Web/sites/myapp" \
  --condition "avg Http5xx > 10" \
  --window-size 5m \
  --evaluation-frequency 1m \
  --severity 2 \
  --description "Alert when HTTP 5xx errors exceed 10 on average over 5 minutes" \
  --action-groups "/subscriptions/<subId>/resourceGroups/rg-prod/providers/microsoft.insights/actionGroups/ag-ops"
```

Notes:

- The exact metric name depends on the resource provider and metric namespace.
- Some resources expose platform metrics directly; others require diagnostic settings or application telemetry.
- For App Service, common metrics include request count, response time, HTTP 5xx, and CPU time, but availability is often better represented by synthetic tests or application telemetry rather than a single platform metric.

**Metric alert example in Bicep**

Use Bicep when you want repeatable infrastructure as code and version-controlled monitoring.

```bicep
param alertName string = 'app-high-latency'
param location string = 'global'
param actionGroupId string
param targetResourceId string

resource metricAlert 'Microsoft.Insights/metricAlerts@2018-03-01' = {
  name: alertName
  location: location
  properties: {
    severity: 2
    enabled: true
    scopes: [
      targetResourceId
    ]
    evaluationFrequency: 'PT1M'
    windowSize: 'PT5M'
    criteria: {
      allOf: [
        {
          criterionType: 'StaticThresholdCriterion'
          name: 'latencyCriterion'
          metricName: 'AverageResponseTime'
          metricNamespace: 'Microsoft.Web/sites'
          timeAggregation: 'Average'
          operator: 'GreaterThan'
          threshold: 2000
          dimensions: []
        }
      ]
    }
    autoMitigate: true
    actions: [
      {
        actionGroupId: actionGroupId
      }
    ]
  }
}
```

Use the actual metric name exposed by the target resource. Metric names are provider-specific, and exam questions often test whether you understand that not every resource exposes the same metrics.

**Logs: flexible analysis with KQL**

Logs are best when you need to ask richer questions than a metric can answer. Azure Monitor Logs stores data in Log Analytics workspaces and supports KQL, which is designed for filtering, grouping, summarizing, and correlating telemetry.

Use log alerts when you need to:

- Detect a pattern across multiple fields.
- Correlate failures with a specific dependency or region.
- Alert on a ratio, trend, or anomaly that is awkward to express as a single metric.
- Combine application logs with platform logs.

A log alert typically evaluates a KQL query on a schedule. If the query returns results that meet the threshold, Azure fires the alert.

**Example KQL for a log alert**

```kusto
AppRequests
| where TimeGenerated > ago(5m)
| where Success == false
| summarize FailedRequests = count() by bin(TimeGenerated, 5m)
| where FailedRequests > 10
```

This pattern is useful when you want to count failures over a time window. The alert rule then triggers if the query returns rows.

**Example: log alert with Azure CLI**

The current Azure CLI family for scheduled query alerts uses `az monitor scheduled-query rule create`.

```bash
az monitor scheduled-query rule create \
  --name "app-failed-requests" \
  --resource-group "rg-prod" \
  --scopes "/subscriptions/<subId>/resourceGroups/rg-prod/providers/Microsoft.OperationalInsights/workspaces/law-prod" \
  --description "Alert when failed requests exceed 10 in 5 minutes" \
  --severity 2 \
  --enabled true \
  --evaluation-frequency "PT1M" \
  --window-size "PT5M" \
  --condition "count 'AppRequests | where TimeGenerated > ago(5m) | where Success == false' > 10" \
  --action "/subscriptions/<subId>/resourceGroups/rg-prod/providers/microsoft.insights/actionGroups/ag-ops"
```

The exact syntax can vary by CLI version and API shape, so treat this as a pattern rather than a universal copy-paste command. In practice, you often define the query and threshold through the rule properties or ARM/Bicep template rather than relying on a single free-form string.

**Log alert design considerations**

- Keep the query efficient; expensive queries can delay alert evaluation.
- Use a time filter such as `TimeGenerated > ago(5m)` to avoid scanning unnecessary data.
- Aggregate before comparing to a threshold.
- Avoid alerting on raw rows unless the rule is intentionally event-based.
- Tune the evaluation frequency and window size to balance speed and noise.

A common incident pattern is a dependency outage. For example, your app may return 500 errors because a downstream API is timing out. A metric alert on 5xx count tells you something is wrong, but a log alert can tell you whether the failures are concentrated in one endpoint, one region, or one dependency.

**Activity log alerts: control-plane monitoring**

Activity log alerts monitor Azure subscription events such as resource creation, deletion, policy changes, role assignments, and service health events. They are useful for governance and operational awareness, especially when you need to know about changes that may affect availability or security.

Typical use cases:

- Alert when a critical resource group is deleted.
- Alert when a network security group rule changes.
- Alert when a role assignment is created for a privileged identity.
- Alert when Azure reports a service health incident affecting your region.

Activity log alerts are not for application-level telemetry. They do not tell you how many requests your app handled or whether your code threw exceptions.

**Example: activity log alert with Azure CLI**

Use the activity log alert command family rather than a generic scheduled query command.

```bash
az monitor activity-log alert create \
  --name "rg-delete-alert" \
  --resource-group "rg-prod" \
  --scopes "/subscriptions/<subId>/resourceGroups/rg-prod" \
  --condition category=Administrative \
  --condition operationName=Microsoft.Resources/subscriptions/resourceGroups/delete \
  --action-group "/subscriptions/<subId>/resourceGroups/rg-prod/providers/microsoft.insights/actionGroups/ag-ops"
```

The condition syntax is parameter-based, not a free-form `and` expression. Depending on the event type, you may need additional filters such as resource group, resource provider, or status.

**Service health alerts: Azure platform incidents**

Service health alerts are built from Azure activity log data, but they are a distinct operational category. They notify you about Azure service issues, planned maintenance, health advisories, and security advisories that may affect your resources.

Use them when:

- You want to know about regional outages before users report them.
- You need to track maintenance windows for compliance or change management.
- You want a centralized notification path for Azure platform incidents.

A service health alert is especially valuable for multi-region applications because it helps you distinguish between an application defect and an Azure platform issue.

**Action groups: what happens when an alert fires**

An alert rule only detects the condition. An **action group** defines the response.

Common action group actions:

- Send email or SMS.
- Trigger a push notification.
- Call a webhook.
- Invoke an Azure Function.
- Start an Automation runbook.
- Send to ITSM or incident management tools through connectors.

Action groups are reusable, so you should define them once and attach them to multiple alerts. This reduces duplication and makes incident routing consistent.

**Example: action group creation**

```bash
az monitor action-group create \
  --name "ag-ops" \
  --resource-group "rg-prod" \
  --short-name "ops" \
  --action email ops-team ops@example.com
```

In production, you often combine multiple receivers:

- Email for visibility.
- Webhook for automation.
- Azure Function for custom remediation.
- ITSM integration for ticket creation.

**Dashboards: shared operational visibility**

Azure dashboards provide a pinned, shareable view of resources, metrics, and charts. They are best for a concise, at-a-glance operational picture.

Use dashboards when you need:

- A team homepage for current health.
- A management view of key KPIs.
- A simple, static layout that non-technical stakeholders can understand.

Dashboards are good for visibility, but they are limited compared with workbooks:

- They are less interactive.
- They are not ideal for deep drill-down.
- They are better for curated tiles than exploratory analysis.

A dashboard often includes:

- Metric charts for latency, errors, and throughput.
- Resource health tiles.
- Log Analytics query charts.
- Links to runbooks or incident documentation.

**Workbook vs dashboard**

| Feature | Dashboard | Workbook |
|---|---|---|
| Primary purpose | Shared overview | Interactive analysis |
| Interactivity | Low to moderate | High |
| Parameters | Limited | Strong support |
| Drill-down | Basic | Rich |
| Best audience | Broad team, leadership | Operators, engineers, investigators |

Use a dashboard when you want a stable operational summary. Use a workbook when you need to investigate incidents, compare time ranges, or filter by environment, region, or instance.

**Workbooks: interactive investigation**

Workbooks combine text, queries, charts, parameters, and links into a single investigative experience. They are especially useful for incident response and post-incident review.

Common workbook scenarios:

- Compare error rates across regions.
- Filter telemetry by deployment version.
- Investigate a spike in latency by endpoint.
- Correlate application logs with infrastructure metrics.

A workbook can include parameters such as environment, subscription, resource group, or time range. This makes it much more useful than a static dashboard for troubleshooting.

**Example workbook query pattern**

```kusto
AppRequests
| where TimeGenerated between (datetime(2026-04-26T10:00:00Z) .. datetime(2026-04-26T11:00:00Z))
| summarize Requests = count(), AvgDuration = avg(DurationMs), Failures = countif(Success == false) by bin(TimeGenerated, 5m)
| order by TimeGenerated asc
```

In a workbook, you would usually parameterize the time range rather than hard-code it.

**Choosing between metric alerts and log alerts**

This distinction appears frequently in AZ-204-style questions.

- Choose a **metric alert** when the signal is already a metric and you need fast detection.
- Choose a **log alert** when the condition requires query logic, joins, or aggregation across fields.
- Choose a **metric alert** for simple thresholds like CPU > 80%.
- Choose a **log alert** for patterns like “more than 10 failed requests from the same region in 5 minutes.”

Borderline scenario examples:

- If you need to alert on request latency for a single service and the platform exposes a reliable latency metric, use a metric alert.
- If you need to alert on “failed logins from more than 3 countries in 10 minutes,” use a log alert.
- If you need to detect a deployment causing errors only on one version, use a log alert with a version dimension or custom property.

**Common real-world incident patterns**

These patterns help you remember which monitoring tool fits the problem.

- **Memory leak in a web app**
  - Metric alert on memory usage or working set.
  - Log alert on increasing exception rate.
  - Workbook to correlate with deployment time.

- **Downstream API outage**
  - Metric alert on 5xx errors or dependency latency.
  - Log alert on failed dependency calls and timeout messages.
  - Dashboard tile for current error rate.

- **Unauthorized infrastructure change**
  - Activity log alert on role assignment or resource deletion.
  - Action group to notify security and operations.
  - Workbook to review the change history.

- **Regional Azure incident**
  - Service health alert.
  - Dashboard tile for affected services.
  - Incident workflow to distinguish platform issue from app issue.

**Implementation workflow**

1. Identify the signal source: metric, log, or activity log.
2. Decide whether the condition is numeric, query-based, or event-based.
3. Create or reuse an action group.
4. Define the alert rule with an appropriate scope, threshold, and evaluation window.
5. Validate the alert with test data or a controlled change.
6. Add the alert or chart to a dashboard or workbook for ongoing visibility.
7. Review and tune thresholds after observing real traffic.

**Practical configuration tips**

- Use separate alert rules for different severities.
- Keep production and non-production alerts distinct.
- Name alerts consistently, such as `app-high-5xx-prod` or `rg-delete-security`.
- Use tags on resources and consistent scopes to simplify management.
- Review alert fatigue regularly; too many low-value alerts reduce response quality.

**Common pitfalls**

- Alerting on the wrong layer:
  - Use activity log alerts for Azure management events, not app failures.
- Using a dashboard as a troubleshooting tool:
  - Dashboards show status; workbooks investigate causes.
- Choosing a threshold without a baseline:
  - Static thresholds should reflect real traffic patterns.
- Ignoring dimensions:
  - A global threshold can hide one bad instance or region.
- Overusing log alerts for simple metrics:
  - Metrics are faster and cheaper for straightforward conditions.
- Confusing service health with application health:
  - Service health tells you about Azure platform issues, not your code.

**Exam-oriented summary**

For AZ-204, remember these distinctions:

- **Metric alerts** are for fast numeric thresholds.
- **Log alerts** are for KQL-based conditions and richer analysis.
- **Activity log alerts** are for subscription and resource management events.
- **Service health alerts** are for Azure platform incidents.
- **Dashboards** are for shared visibility.
- **Workbooks** are for interactive investigation.

If a question asks what to use for a sudden increase in HTTP failures, think metric or log alert depending on whether the signal is already exposed as a metric or must be derived from logs. If it asks about resource deletion or role assignment, think activity log alert. If it asks about a team view of current health, think dashboard. If it asks about drilling into a spike by region or version, think workbook.
