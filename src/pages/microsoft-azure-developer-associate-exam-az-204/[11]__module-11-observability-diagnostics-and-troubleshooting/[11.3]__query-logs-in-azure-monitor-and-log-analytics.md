## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Azure Monitor Logs and Log Analytics overview**

Azure Monitor Logs stores telemetry in a Log Analytics workspace and lets you query it with the <abbr title="Kusto Query Language">KQL</abbr>. You use it to investigate application behavior, infrastructure issues, security events, and custom telemetry. In exam scenarios, the key idea is that logs are optimized for flexible analysis, not for transactional reads.

A Log Analytics workspace is the primary scope for log queries. Depending on how data is ingested, you may query:
- **Workspace data** collected from Azure resources, agents, or applications.
- **Resource-centric logs** from a specific Azure resource.
- **Cross-resource** or **cross-workspace** data when you need a broader investigation.

Typical use cases include:
- Finding failed requests or exceptions.
- Correlating application traces with platform metrics.
- Detecting patterns over time.
- Feeding alert rules and workbooks.

**Workspace access and permissions**

You need appropriate access to query logs. In practice, permissions are controlled through Azure role-based access control (<abbr title="Role-Based Access Control">RBAC</abbr>).

Common roles include:
- **Log Analytics Reader** for read-only query access.
- **Monitoring Reader** for broader monitoring read access.
- **Contributor** or higher when you need to configure workspaces, alerts, or data collection.

If you cannot see data, check:
- Whether you have access to the workspace or resource.
- Whether the data has actually been ingested.
- Whether the time range includes the event.
- Whether table retention or ingestion latency affects visibility.

> If a query returns no rows, do not assume the data is missing. First verify scope, time range, permissions, and table name.

**KQL fundamentals**

KQL is a read-only query language designed for fast log exploration. Queries usually start with a table name and then apply operators to filter, shape, summarize, and join data.

A basic pattern looks like this:

```kusto
AppRequests
| where TimeGenerated > ago(1h)
| where Success == false
| project TimeGenerated, Name, ResultCode, DurationMs, OperationId
| order by TimeGenerated desc
```

Core ideas:
- **Pipe syntax** (`|`) passes results from one operator to the next.
- **Filtering** reduces the dataset early for performance.
- **Projection** selects only the columns you need.
- **Aggregation** turns many rows into summary results.
- **Sorting** helps you inspect the most recent or most severe events.

Common operators you should know:
- `where` filters rows.
- `project` selects and renames columns.
- `extend` creates calculated columns.
- `summarize` aggregates rows.
- `order by` sorts results.
- `take` limits output.
- `distinct` returns unique values.
- `top` returns the highest or lowest values.

**Useful KQL patterns for exam questions**

KQL often appears in questions that ask you to extract fields, correlate records, or prepare alert-friendly output.

- **`parse`** extracts structured values from a text field when the data is semi-structured.
- **`mv-expand`** expands arrays or dynamic lists into multiple rows.
- **`join`** correlates records from two tables or two filtered datasets.
- **`datatable`** creates inline test data for query development and validation.

Examples:

```kusto
AppTraces
| where TimeGenerated > ago(1h)
| parse Message with * "OrderId=" OrderId:int ", Status=" Status
| project TimeGenerated, OrderId, Status, Message
```

```kusto
let Errors = AppRequests
    | where TimeGenerated > ago(1h)
    | where Success == false
    | project OperationId, RequestName = Name, RequestTime = TimeGenerated;
let Exceptions = AppExceptions
    | where TimeGenerated > ago(1h)
    | project OperationId, ExceptionType = Type, ExceptionMessage = OuterMessage;
Errors
| join kind=leftouter Exceptions on OperationId
| project RequestTime, RequestName, ExceptionType, ExceptionMessage
| order by RequestTime desc
```

```kusto
datatable(Service:string, LatencyMs:long)
[
  "api", 120,
  "api", 340,
  "worker", 80
]
| summarize AvgLatency = avg(LatencyMs) by Service
```

```kusto
CustomEvents
| where TimeGenerated > ago(1d)
| extend Tags = todynamic(Properties.Tags)
| mv-expand Tags
| project TimeGenerated, Name, Tags
```

Practical notes:
- Use `parse` when the source string has a predictable pattern.
- Use `mv-expand` when a dynamic field contains multiple values you need to analyze individually.
- Use `join` carefully because it can increase result size and cost.
- Use `datatable` to prototype logic before running against production tables.

**Query editor features you should know**

The Azure Monitor Logs query editor is more than a text box. It supports workflow features that matter in real troubleshooting and exam scenarios.

Important capabilities:
- **Query history** lets you revisit recent queries and reuse them.
- **Saved queries** let you store frequently used queries for later use.
- **Favorites** help you quickly access important queries.
- **Copy query** lets you duplicate a query into another session or share it informally.
- **Share query** is useful when collaborating with teammates or handing off an investigation.
- **Time range selection** controls the data window without changing the query text.
- **Run and stop controls** help you manage expensive or long-running queries.

Practical workflow:
1. Start with a narrow time range.
2. Run a focused query.
3. Save the query if you expect to reuse it.
4. Share or copy the query when collaborating on an incident.
5. Expand the time range only after you confirm the query logic.

**Cross-resource and cross-workspace querying**

Azure Monitor supports querying across multiple resources or workspaces, which is useful when telemetry is distributed across apps, subscriptions, or environments.

Common patterns:
- Query multiple resources in the same workspace context.
- Query multiple workspaces when logs are split by environment or business unit.
- Use resource-scoped queries when you want to focus on one application or Azure resource.

Example cross-workspace pattern:

```kusto
union workspace("prod-workspace").AppRequests,
      workspace("dev-workspace").AppRequests
| where TimeGenerated > ago(24h)
| summarize Requests = count() by Workspace = _ResourceId
```

Example cross-resource pattern:

```kusto
union AppRequests, AppDependencies
| where TimeGenerated > ago(1h)
| summarize Count = count() by Type
```

Limitations and cautions:
- Cross-workspace queries can be slower and more expensive.
- You need access to every workspace or resource included in the query.
- Table names and schemas should be consistent enough for the union to make sense.
- Not every table is available in every workspace.

When investigating distributed systems, use cross-resource queries to correlate:
- Front-end request failures.
- Back-end dependency latency.
- Shared exception patterns.
- Environment-specific differences.

**Query shaping and result interpretation**

The shape of your result set matters, especially when the output feeds dashboards or alerts. Azure Monitor alert rules often expect a result that can be evaluated consistently over time.

For alert-oriented queries:
- Return a **single numeric value** when using a simple threshold alert.
- Use `summarize` to aggregate over a time window.
- Ensure the query returns a stable schema.
- Include dimensions only when the alert rule is designed to split by them.

Example alert-friendly query:

```kusto
AppRequests
| where TimeGenerated > ago(15m)
| where Success == false
| summarize FailedRequests = count() by bin(TimeGenerated, 5m)
```

Why this matters:
- Alert rules evaluate the query output, not just the raw logs.
- If the query returns multiple columns or too many rows, the alert may not behave as expected.
- Aggregation granularity should match the alert frequency and evaluation window.
- Dimensions are useful when you want separate alert instances per service, region, or operation.

Common alert design patterns:
- **Count-based alert** for failures or exceptions.
- **Average-based alert** for latency.
- **Percentile-based alert** for tail latency analysis.
- **Dimension-based alert** for isolating noisy components.

Example with dimensions:

```kusto
AppRequests
| where TimeGenerated > ago(30m)
| summarize Failed = countif(Success == false) by bin(TimeGenerated, 5m), CloudRoleName
```

**Working with time and aggregation**

Time handling is central to log analysis. Most queries filter on `TimeGenerated`, then aggregate into bins.

Useful functions:
- `ago()` for relative time windows.
- `bin()` to group timestamps into fixed intervals.
- `now()` for the current time.
- `datetime_diff()` for comparing timestamps.
- `startofday()` and similar helpers for calendar-based grouping.

Example:

```kusto
AppRequests
| where TimeGenerated between (ago(24h) .. now())
| summarize Requests = count(), AvgDuration = avg(DurationMs) by bin(TimeGenerated, 1h)
| order by TimeGenerated asc
```

Use smaller bins for:
- Near-real-time troubleshooting.
- Alerting.
- Short-lived spikes.

Use larger bins for:
- Trend analysis.
- Capacity planning.
- Reducing noise in high-volume systems.

**Joining and correlating telemetry**

`join` is one of the most important KQL operators for exam preparation because it helps you correlate related records across tables.

Common join kinds:
- `inner` returns matching rows from both sides.
- `leftouter` keeps all rows from the left side and adds matches from the right.
- `fullouter` keeps all rows from both sides.
- `innerunique` reduces duplicate matches on the left side.

Example:

```kusto
let Requests = AppRequests
| where TimeGenerated > ago(1h)
| project OperationId, RequestName = Name, RequestTime = TimeGenerated;
let Dependencies = AppDependencies
| where TimeGenerated > ago(1h)
| project OperationId, DependencyName = Name, Target, DependencyDuration = DurationMs;
Requests
| join kind=leftouter Dependencies on OperationId
| project RequestTime, RequestName, DependencyName, Target, DependencyDuration
```

Practical guidance:
- Join on a stable correlation key such as `OperationId`, `RequestId`, or a custom trace identifier.
- Filter before joining to reduce cost.
- Project only the columns you need after the join.
- Be aware that many-to-many joins can multiply rows.

**Parsing semi-structured data**

Logs often contain text fields with embedded values. `parse` and related functions help you extract those values into columns.

Example:

```kusto
AppTraces
| where TimeGenerated > ago(2h)
| parse Message with * "user=" UserId ", action=" Action ", status=" Status
| summarize Count = count() by UserId, Action, Status
```

Use parsing when:
- The source system writes structured text instead of JSON.
- You need to extract fields for filtering or grouping.
- You want to normalize logs from older applications.

If the data is JSON-like, `todynamic()` and field access can be more reliable than string parsing.

**Expanding arrays and dynamic fields**

Many Azure logs store structured properties in dynamic columns. `mv-expand` turns each element into a separate row so you can analyze list-like data.

Example:

```kusto
CustomEvents
| where TimeGenerated > ago(1d)
| extend Items = todynamic(Properties.Items)
| mv-expand Items
| project TimeGenerated, Name, Item = tostring(Items)
```

Use `mv-expand` when:
- A single event contains multiple tags, IDs, or values.
- You need counts per item rather than per event.
- You want to inspect each element individually.

**Creating test data with `datatable`**

`datatable` is useful for learning, prototyping, and validating query logic without relying on live telemetry.

Example:

```kusto
datatable(TimeGenerated:datetime, Service:string, Success:bool, DurationMs:long)
[
  datetime(2026-04-26 10:00:00), "api", true, 120,
  datetime(2026-04-26 10:05:00), "api", false, 450,
  datetime(2026-04-26 10:10:00), "worker", true, 80
]
| summarize Total = count(), Failures = countif(Success == false) by Service
```

This is especially useful when you want to:
- Test `summarize` logic.
- Validate `join` behavior.
- Demonstrate alert-ready output.
- Build a query before applying it to production tables.

**Query performance and troubleshooting**

Efficient queries are easier to run and easier to reason about.

Best practices:
- Filter early with `where`.
- Project only needed columns.
- Use the narrowest time range that answers the question.
- Avoid unnecessary joins and wide unions.
- Prefer aggregation before expensive transformations when possible.

Troubleshooting steps:
1. Confirm the table name exists in the workspace.
2. Check the time range.
3. Run a simpler query to verify data presence.
4. Add filters and operators one at a time.
5. Inspect intermediate columns with `project`.

Example debugging approach:

```kusto
AppRequests
| where TimeGenerated > ago(1h)
| take 10
```

Then refine:

```kusto
AppRequests
| where TimeGenerated > ago(1h)
| where Success == false
| project TimeGenerated, Name, ResultCode, OperationId
```

**Practical CLI and automation examples**

You often query logs through the portal, but automation is useful for repeatable operations and exam context.

Azure CLI can help you inspect workspaces and related resources:

```bash
az monitor log-analytics workspace list -o table
az monitor log-analytics workspace show --resource-group rg-monitoring --workspace-name law-prod
```

You can also use the Azure Monitor Query SDK in code to run KQL programmatically. A typical flow is:
- Authenticate with Azure identity.
- Select the workspace or resource scope.
- Submit a KQL query.
- Process the returned tables and rows.

Example pseudo-pattern:

```csharp
var query = @"
AppRequests
| where TimeGenerated > ago(1h)
| summarize Requests = count(), Failures = countif(Success == false)
";
```

In automation, keep queries:
- Deterministic.
- Narrow in scope.
- Stable in schema.
- Easy to parse downstream.

**Exam-focused distinctions**

Be ready to distinguish these concepts:
- **Azure Monitor Logs** is the query and analysis layer.
- **Log Analytics workspace** is the storage and query scope for logs.
- **KQL** is the language used to query the data.
- **Metrics** are optimized for numeric time-series monitoring, while logs are better for detailed investigation.

Also remember:
- Logs are not the same as application code traces, though they can include them.
- A query used for troubleshooting may not be suitable for alerting without aggregation.
- Cross-workspace queries are powerful but require access and consistent schema planning.

**Quick reference**

- Use `where` to narrow data early.
- Use `summarize` for counts, averages, and alert-ready output.
- Use `join` to correlate related telemetry.
- Use `parse` to extract fields from text logs.
- Use `mv-expand` to flatten arrays or dynamic lists.
- Use `datatable` to test queries safely.
- Use query history, saved queries, favorites, and sharing to speed up investigations.
- Verify RBAC permissions when data is missing.
- Use cross-resource or cross-workspace queries when telemetry is distributed.
