## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Lab purpose and troubleshooting mindset**

This lab focuses on end-to-end request tracing in an Azure-hosted application, then uses the trace data to identify a runtime failure and correct it. The core skill is not just reading logs, but correlating a user request across the app, platform telemetry, and deployment state so you can isolate the fault quickly.

In practice, you use a combination of:
- **Application telemetry** to see what the app did.
- **Platform logs** to see what Azure reported.
- **Configuration review** to confirm the runtime environment matches expectations.
- **Targeted fixes** to resolve the issue without introducing new regressions.

A good troubleshooting flow is:
1. Reproduce the failure.
2. Capture the request identifier or timestamp.
3. Trace the request through logs and telemetry.
4. Identify the failing dependency, code path, or configuration.
5. Apply the smallest safe fix.
6. Re-test and confirm the request succeeds.

**Tracing a request in Azure**

Request tracing means following a single operation across components such as a web app, API, queue processor, database call, or downstream service. In Azure, this often relies on correlation IDs, distributed tracing, and Application Insights telemetry.

Common telemetry signals include:
- **Requests** for inbound HTTP calls.
- **Dependencies** for outbound calls to storage, SQL, HTTP APIs, or queues.
- **Exceptions** for runtime failures.
- **Traces** for custom log messages.
- **Performance counters** or metrics for latency and saturation.

If Application Insights is enabled, you can often correlate a failed request with its related dependency failures and exception details. This is especially useful when the app returns a generic error page but the root cause is buried in the backend.

Typical Kusto Query Language (KQL) queries in Application Insights include:

```kusto
requests
| where timestamp > ago(1h)
| where success == false
| order by timestamp desc
```

```kusto
exceptions
| where timestamp > ago(1h)
| order by timestamp desc
```

```kusto
dependencies
| where timestamp > ago(1h)
| where success == false
| order by timestamp desc
```

To trace a single operation, you often pivot on `operation_Id`:

```kusto
requests
| where operation_Id == "00000000000000000000000000000000"
| project timestamp, name, resultCode, success, duration, operation_Id
```

```kusto
dependencies
| where operation_Id == "00000000000000000000000000000000"
| project timestamp, name, target, type, resultCode, success, duration
```

```kusto
exceptions
| where operation_Id == "00000000000000000000000000000000"
| project timestamp, type, outerMessage, problemId
```

**What to inspect first**

When a deliberate runtime issue is introduced, the fastest path is to compare the expected behavior with the actual runtime state.

Check these areas in order:
- **Recent deployment changes** such as app settings, connection strings, or code updates.
- **Startup logs** for missing configuration or failed initialization.
- **Request logs** for HTTP status codes and latency spikes.
- **Dependency telemetry** for failed calls to storage, databases, or external APIs.
- **Exception details** for stack traces, null references, authentication failures, or serialization errors.

A deliberate issue in a lab is often one of these:
- A wrong environment variable name or value.
- A missing connection string.
- A broken secret reference.
- A mismatched API endpoint.
- A code path that assumes a resource exists.
- A runtime version mismatch or unsupported package behavior.

**Azure App Service and runtime diagnostics**

For web apps hosted in Azure App Service, you can use platform tools to inspect the runtime environment and logs.

Useful tools and commands:
- **Log stream** to watch live application output.
- **App Service logs** to enable file system or blob logging.
- **Kudu/Advanced Tools** to inspect files, environment variables, and process state.
- **Azure CLI** to query app settings and restart the app after a fix.

Example Azure CLI commands:

```bash
az webapp log tail --name <app-name> --resource-group <rg-name>
```

```bash
az webapp config appsettings list --name <app-name> --resource-group <rg-name> -o table
```

```bash
az webapp config connection-string list --name <app-name> --resource-group <rg-name> -o table
```

```bash
az webapp restart --name <app-name> --resource-group <rg-name>
```

If you need to inspect the app’s environment variables from Kudu, you can use the debug console or the SCM site. In many cases, the issue is visible immediately as a missing or incorrect setting.

> Be careful when changing settings in production-like environments. A fix that resolves one request path can break another if it changes shared configuration or connection behavior.

**Common failure patterns and how to recognize them**

A deliberate runtime issue usually produces a recognizable symptom pattern. Matching the symptom to the likely cause saves time.

- **HTTP 500 Internal Server Error**
  - Often indicates an unhandled exception, missing dependency, or invalid configuration.
  - Check exception telemetry and startup logs first.

- **HTTP 502/503**
  - Often indicates the app is unhealthy, not responding, or failing to start.
  - Check platform logs, container logs, and app startup behavior.

- **HTTP 401/403**
  - Often indicates authentication or authorization failure.
  - Check identity configuration, token audience, role assignments, and secret values.

- **Slow responses**
  - Often indicate dependency latency, retry storms, or inefficient code.
  - Check dependency duration and request duration trends.

- **Intermittent failures**
  - Often indicate transient dependency issues, throttling, or race conditions.
  - Check retry behavior, concurrency, and resource limits.

**Using Application Insights effectively**

Application Insights is the primary observability tool for many AZ-204 troubleshooting scenarios. It helps you answer three questions:
- What request failed?
- What dependency or exception caused it?
- Was the failure isolated or systemic?

Useful investigation views include:
- **Failures** to see failed requests and exceptions.
- **Performance** to identify slow endpoints.
- **Transaction search** to inspect a single request end to end.
- **Live Metrics** to observe near-real-time behavior during reproduction.

A practical workflow is:
1. Open the failed request in Transaction Search.
2. Note the `operation_Id`.
3. Open related dependencies and exceptions.
4. Identify the first failing call in the chain.
5. Compare the failing request with a successful one.

Example KQL to compare successful and failed requests:

```kusto
requests
| where timestamp > ago(1h)
| summarize total=count(), failed=countif(success == false) by name
| order by failed desc
```

Example KQL to inspect top exceptions:

```kusto
exceptions
| where timestamp > ago(24h)
| summarize count() by type
| order by count_ desc
```

**Configuration validation**

Many runtime issues are caused by configuration drift rather than code defects. Validate the app’s settings before changing code.

Check:
- **App settings** for environment-specific values.
- **Connection strings** for storage, SQL, or service endpoints.
- **Managed identity** permissions if the app uses Azure AD authentication.
- **Feature flags** or toggles that may disable a code path.
- **Slot settings** if deployment slots are used.

Example of a typical ASP.NET Core configuration pattern:

```csharp
var builder = WebApplication.CreateBuilder(args);

var connectionString = builder.Configuration["ConnectionStrings:AppDb"];
if (string.IsNullOrWhiteSpace(connectionString))
{
    throw new InvalidOperationException("Missing AppDb connection string.");
}

builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseSqlServer(connectionString));
```

This pattern fails fast, which is useful in labs because it makes the root cause visible immediately instead of producing a vague downstream error.

**Practical fix patterns**

Once you identify the issue, apply the smallest change that restores the expected behavior.

Common fixes include:
- Correcting an app setting key or value.
- Restoring a missing connection string.
- Updating a secret reference or Key Vault reference.
- Fixing a malformed endpoint URL.
- Reverting a bad deployment or configuration change.
- Adjusting code to handle null, missing, or transient values safely.

Example of safer configuration handling:

```csharp
var apiBaseUrl = builder.Configuration["ExternalApi:BaseUrl"];

if (!Uri.TryCreate(apiBaseUrl, UriKind.Absolute, out var baseUri))
{
    throw new InvalidOperationException("ExternalApi:BaseUrl must be a valid absolute URI.");
}
```

Example of resilient HTTP usage:

```csharp
builder.Services.AddHttpClient("external-api", client =>
{
    client.Timeout = TimeSpan.FromSeconds(10);
});
```

**CLI and deployment verification**

After the fix, verify that the runtime state matches the intended configuration and that the app is healthy.

Useful checks:
- Confirm the app setting exists and has the expected value.
- Restart the app if the setting requires a restart.
- Reproduce the failing request.
- Confirm the request now succeeds.
- Review telemetry for reduced exceptions and successful dependencies.

Example verification commands:

```bash
az webapp config appsettings list --name <app-name> --resource-group <rg-name> --query "[?name=='<setting-name>']"
```

```bash
az webapp show --name <app-name> --resource-group <rg-name> --query "{state:state, defaultHostName:defaultHostName}"
```

```bash
az monitor app-insights query \
  --app <app-insights-name> \
  --analytics-query "requests | where timestamp > ago(15m) | order by timestamp desc"
```

**Troubleshooting checklist**

Use this checklist when a request fails and the cause is not obvious:
- Confirm the failure is reproducible.
- Capture the exact time of the failure.
- Find the request in telemetry.
- Trace related dependencies and exceptions.
- Check recent configuration or deployment changes.
- Validate secrets, endpoints, and connection strings.
- Inspect startup and runtime logs.
- Apply the smallest fix.
- Restart only if required.
- Re-test and confirm telemetry shows success.

**What exam scenarios usually test**

AZ-204 lab-style questions often test whether you can:
- Use telemetry to identify the failing component.
- Distinguish between app code failures and platform/configuration failures.
- Read logs and correlate them with a request.
- Fix a deliberate misconfiguration safely.
- Verify the fix with a repeatable test.

The key skill is correlation: you should be able to connect a user-visible failure to a specific runtime cause, then prove the fix worked with logs or metrics rather than assumption.

**Quick reference**

- **Request tracing**: follow one operation across app and dependencies.
- **Correlation ID**: identifier used to connect related telemetry.
- **Application Insights**: Azure service for app telemetry and diagnostics.
- **KQL**: query language used to analyze telemetry data.
- **Kudu**: App Service advanced tools site for inspection and debugging.
- **App settings**: runtime configuration values injected into the app.
- **Connection strings**: endpoint and credential values for data services.
