## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**HTTP-triggered Azure Functions overview**

HTTP-triggered functions expose an endpoint that runs code in response to an HTTP request. They are a common choice for lightweight APIs, webhooks, and glue code between services.

In Azure Functions, the trigger determines how the function starts. For HTTP triggers, you typically define:

- The route template
- The allowed HTTP methods
- The authorization level
- The request and response handling model

HTTP triggers are useful when you need serverless request handling without managing an app server. They are also a common exam topic because they combine bindings, routing, authentication, and local development.

**Core concepts you should know**

- **Trigger**: The binding that starts the function execution.
- **Route**: The URL pattern used to match incoming requests.
- **Authorization level**: The access policy for the endpoint.
- **Worker model**: The runtime model used by your language stack.
- **Binding**: A declarative connection to input or output data.

For AZ-204, focus on how the same HTTP-triggered pattern is implemented differently in .NET isolated worker and Python.

**.NET 8 isolated worker model**

The .NET isolated worker model runs your function code in a separate process from the Functions host. This gives you more control over dependencies, startup, and middleware-like behavior than the in-process model.

Use the isolated worker model when you want:

- Modern .NET support, including .NET 8
- Better isolation from the host runtime
- Explicit control over serialization and dependency injection
- A programming model that aligns more closely with standard .NET application patterns

A typical isolated worker app uses `Program.cs` for startup and one or more function classes for handlers.

```csharp
using Azure.Core.Serialization;
using Microsoft.Azure.Functions.Worker;
using Microsoft.Azure.Functions.Worker.Builder;
using Microsoft.Extensions.DependencyInjection;
using Microsoft.Extensions.Hosting;
using System.Text.Json;

var builder = FunctionsApplication.CreateBuilder(args);

builder.Services.Configure<WorkerOptions>(options =>
{
    options.Serializer = new JsonObjectSerializer(new JsonSerializerOptions
    {
        PropertyNamingPolicy = JsonNamingPolicy.CamelCase,
        WriteIndented = false
    });
});

builder.Build().Run();
```

The serializer configuration matters when you want consistent JSON casing or custom serialization behavior. In isolated worker, `JsonObjectSerializer` comes from `Azure.Core.Serialization`, so you need the correct namespace.

A minimal HTTP-triggered function in .NET 8 isolated worker looks like this:

```csharp
using Microsoft.Azure.Functions.Worker;
using Microsoft.Azure.Functions.Worker.Http;
using Microsoft.Extensions.Logging;
using System.Net;

namespace MyFunctions;

public class PingFunction
{
    private readonly ILogger<PingFunction> _logger;

    public PingFunction(ILogger<PingFunction> logger)
    {
        _logger = logger;
    }

    [Function("Ping")]
    public async Task<HttpResponseData> Run(
        [HttpTrigger(AuthorizationLevel.Function, "get", "post", Route = "ping/{name?}")]
        HttpRequestData req,
        string? name)
    {
        _logger.LogInformation("Processing ping request.");

        var response = req.CreateResponse(HttpStatusCode.OK);
        await response.WriteStringAsync($"pong {name ?? "world"}");
        return response;
    }
}
```

Key points for .NET isolated HTTP functions:

- **`HttpRequestData`** and **`HttpResponseData`** are the primary request and response types.
- **`WriteStringAsync`** is the standard way to write text to the response.
- **Route parameters** can be bound in the isolated model, but the exact binding behavior depends on the trigger signature and template.
- **`AuthorizationLevel.Function`** is a common choice for secured endpoints that use function keys.

> `AuthorizationLevel.Admin` is tied to host/master key access and host-level management scenarios, not normal user-facing function endpoints.

**Startup and hosting in .NET isolated**

The isolated worker startup path depends on the template and packages you use. For HTTP scenarios, you may see either the standard worker model or ASP.NET Core integration.

Use the standard isolated worker model when you want the Functions-native request/response types and a simpler dependency surface. Use ASP.NET Core integration only when your project template and package references explicitly support it.

Typical package references for a .NET 8 isolated HTTP function app include:

```xml
<ItemGroup>
  <PackageReference Include="Microsoft.Azure.Functions.Worker" Version="*" />
  <PackageReference Include="Microsoft.Azure.Functions.Worker.Sdk" Version="*" OutputItemType="Analyzer" />
  <PackageReference Include="Microsoft.Azure.Functions.Worker.Extensions.Http" Version="*" />
  <PackageReference Include="Microsoft.Azure.Functions.Worker.Extensions.Http.AspNetCore" Version="*" />
</ItemGroup>
```

Not every project needs the ASP.NET Core integration package. Do not assume `ConfigureFunctionsWebApplication()` is a universal replacement for all isolated-worker HTTP setups.

**Python HTTP-triggered functions**

Python Azure Functions commonly use the v2 programming model for cleaner decorators and a more compact app definition. The app object defines defaults, and individual route decorators can override them.

A basic Python HTTP-triggered function looks like this:

```python
import azure.functions as func

app = func.FunctionApp(http_auth_level=func.AuthLevel.FUNCTION)

@app.route(route="ping/{name}", methods=["GET", "POST"], auth_level=func.AuthLevel.ANONYMOUS)
def ping(req: func.HttpRequest) -> func.HttpResponse:
    name = req.route_params.get("name", "world")
    return func.HttpResponse(f"pong {name}", status_code=200)
```

Important Python model details:

- **`FunctionApp`** sets app-level defaults, including the default HTTP authorization level.
- **`@app.route(...)`** defines the HTTP trigger and can override the app default.
- **`func.AuthLevel.ANONYMOUS`** allows unauthenticated access.
- **`req.route_params`** gives access to route values.
- **`func.HttpResponse`** returns the response payload and status code.

The earlier app-level auth setting is only a default. If you specify `auth_level` on the route decorator, that value takes precedence for that function.

**Authorization levels and when to use them**

Azure Functions supports several authorization levels for HTTP triggers.

| Level | Typical use | Notes |
|---|---|---|
| `Anonymous` | Public endpoints, webhooks behind other controls | No function key required |
| `Function` | Most app endpoints | Requires a function key |
| `Admin` | Host management and master-key scenarios | Not for normal API consumers |

Use the least privileged option that fits the scenario. For exam questions, choose `Function` for protected API-like endpoints unless the prompt clearly indicates public access or host-level administration.

**Route templates and query handling**

Routes let you create readable endpoints and capture values from the URL path. Query strings are available separately from route parameters.

Examples:

- `/api/ping`
- `/api/ping/alice`
- `/api/orders/123/items/9`

In .NET isolated, route parameters can be bound directly in many common templates:

```csharp
[Function("GetOrder")]
public async Task<HttpResponseData> Run(
    [HttpTrigger(AuthorizationLevel.Function, "get", Route = "orders/{orderId}")] HttpRequestData req,
    string orderId)
{
    var response = req.CreateResponse(HttpStatusCode.OK);
    await response.WriteStringAsync($"Order: {orderId}");
    return response;
}
```

In Python, you usually read route values from `req.route_params`:

```python
order_id = req.route_params.get("orderId")
```

Use query strings when the value is optional or filters the result, and use route parameters when the value identifies the resource.

**Local development workflow**

Local development is usually done with Azure Functions Core Tools. This lets you run and debug the function app on your machine before deployment.

Typical steps:

1. Create the project from the correct template.
2. Configure local settings.
3. Run the Functions host.
4. Test the endpoint with a browser, `curl`, or an API client.
5. Debug and iterate.

Example local commands:

```bash
func init MyHttpFunctions --worker-runtime dotnet-isolated
cd MyHttpFunctions
func new --template "HTTP trigger" --name Ping
func start
```

For Python:

```bash
func init MyPythonFunctions --python
cd MyPythonFunctions
func new --template "HTTP trigger" --name Ping
func start
```

A typical `local.settings.json` for local development might look like this:

```json
{
  "IsEncrypted": false,
  "Values": {
    "AzureWebJobsStorage": "UseDevelopmentStorage=true",
    "FUNCTIONS_WORKER_RUNTIME": "dotnet-isolated",
    "FUNCTIONS_EXTENSION_VERSION": "~4"
  }
}
```

`AzureWebJobsStorage` is commonly required by the Functions host for local development. In deployed environments, the exact storage requirement depends on the hosting plan and configuration, so avoid assuming every HTTP-only app needs the same storage setup in production.

**Testing with curl**

Use `curl` to verify routing, methods, and auth behavior.

```bash
curl -i "http://localhost:7071/api/ping"
curl -i "http://localhost:7071/api/ping/alice"
curl -i -X POST "http://localhost:7071/api/ping/bob"
```

If the function uses `AuthorizationLevel.Function`, include the function key:

```bash
curl -i "http://localhost:7071/api/ping?code=<function-key>"
```

For route-based endpoints, the key can also be passed as a query parameter unless your host configuration changes the default behavior.

**Deployment considerations**

When deploying HTTP-triggered functions, make sure the runtime, worker model, and hosting configuration match your code.

For Azure CLI-based creation, include the platform details that match your app. For Linux-based Python and .NET isolated apps, you often need `--os-type Linux` and a compatible runtime configuration.

Example pattern:

```bash
az group create -n rg-func-http -l eastus

az storage account create \
  -n mystorageacct12345 \
  -g rg-func-http \
  -l eastus \
  --sku Standard_LRS

az functionapp create \
  --resource-group rg-func-http \
  --consumption-plan-location eastus \
  --name my-http-func-app \
  --storage-account mystorageacct12345 \
  --os-type Linux \
  --runtime dotnet-isolated
```

For Python, the runtime and OS must also align with the app stack:

```bash
az functionapp create \
  --resource-group rg-func-http \
  --consumption-plan-location eastus \
  --name my-python-func-app \
  --storage-account mystorageacct12345 \
  --os-type Linux \
  --runtime python
```

Depending on the plan and CLI version, you may also need to specify runtime version details or use a deployment workflow that sets the app settings explicitly. Always verify the generated app settings after creation.

**Configuration and app settings**

Common app settings for HTTP-triggered Functions include:

- `FUNCTIONS_WORKER_RUNTIME`
- `FUNCTIONS_EXTENSION_VERSION`
- `AzureWebJobsStorage`
- Language-specific settings such as Python version or .NET isolated package configuration

In Azure, app settings are the runtime equivalent of environment variables. Use them for secrets, connection strings, and deployment-specific values.

Example app setting update:

```bash
az functionapp config appsettings set \
  --name my-http-func-app \
  --resource-group rg-func-http \
  --settings "MySetting=MyValue"
```

In code, read settings through the standard configuration APIs for your language and worker model.

**Practical response patterns**

HTTP functions often return JSON, plain text, or status-only responses.

Plain text in .NET isolated:

```csharp
var response = req.CreateResponse(HttpStatusCode.OK);
await response.WriteStringAsync("pong");
return response;
```

JSON response in Python:

```python
import json
return func.HttpResponse(
    json.dumps({"message": "pong"}),
    mimetype="application/json",
    status_code=200
)
```

JSON response in .NET isolated:

```csharp
var response = req.CreateResponse(HttpStatusCode.OK);
await response.WriteAsJsonAsync(new { message = "pong" });
return response;
```

Use JSON when the function is acting like an API. Use plain text for simple health checks or diagnostics.

**Common exam traps and troubleshooting**

- **Worker model mismatch**: Make sure the project template, package references, and runtime setting all match the chosen language model.
- **Wrong auth level**: `Anonymous` is public, `Function` requires a key, and `Admin` is for host-level access.
- **Route confusion**: Route parameters are not the same as query strings.
- **Serialization surprises**: In .NET isolated, configure the serializer explicitly if you need specific JSON behavior.
- **Startup assumptions**: Do not assume ASP.NET Core integration is enabled unless the project template and packages support it.
- **Local storage assumptions**: `AzureWebJobsStorage` is often needed locally, but production requirements vary by plan and scenario.

> If a question asks for the simplest secure HTTP endpoint, choose an HTTP trigger with `AuthorizationLevel.Function` unless the prompt explicitly requires anonymous access.

**Quick reference**

| Scenario | .NET 8 isolated | Python |
|---|---|---|
| Define function | `[Function("Name")]` | `@app.route(...)` |
| Request type | `HttpRequestData` | `func.HttpRequest` |
| Response type | `HttpResponseData` | `func.HttpResponse` |
| Read route value | Method parameter or binding | `req.route_params.get(...)` |
| Write response | `WriteStringAsync` / `WriteAsJsonAsync` | `HttpResponse(...)` |
| Default auth | `AuthorizationLevel.Function` in trigger | `FunctionApp(http_auth_level=...)` |

**What to remember for AZ-204**

- HTTP triggers are the standard way to expose serverless endpoints in Azure Functions.
- .NET 8 isolated worker uses a separate process and explicit startup configuration.
- Python v2 uses decorators and a `FunctionApp` object.
- Authorization level selection is a frequent exam detail.
- Route templates, query strings, and response handling are core implementation skills.
- Local testing with Core Tools is part of the normal development workflow.
- Deployment commands must match the runtime, OS, and hosting plan.
