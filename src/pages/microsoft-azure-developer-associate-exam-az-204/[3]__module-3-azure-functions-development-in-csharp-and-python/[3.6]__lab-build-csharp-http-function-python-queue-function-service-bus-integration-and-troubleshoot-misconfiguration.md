## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Lab overview**

This lab combines two common Azure Functions patterns: an HTTP-triggered function in C# and a queue-triggered function in Python, then connects them to Azure Service Bus. The goal is to practice end-to-end serverless integration, including local development, configuration, deployment, and troubleshooting when bindings or app settings are wrong.

The exam-relevant value is in understanding how triggers, bindings, connection settings, and runtime versions interact. You should be able to identify whether a failure comes from code, configuration, identity, or the messaging resource itself.

**Core architecture**

A typical lab topology looks like this:

- A **C# HTTP trigger** receives a request and returns a response or sends a message to Service Bus.
- A **Service Bus queue** buffers work asynchronously.
- A **Python queue trigger** processes messages from the queue.
- Application settings provide connection details for local and cloud execution.

A simple flow is:

1. Client calls the HTTP endpoint.
2. The HTTP function validates input and enqueues a message.
3. Service Bus stores the message until a consumer is available.
4. The queue-triggered function processes the message and completes it on success.

**Azure Functions concepts you must know**

Azure Functions uses a trigger to start execution and bindings to connect to other services without writing boilerplate client code. A trigger is the event source; a binding is the declarative connection to input or output data.

- **Trigger** starts the function, such as `httpTrigger` or `serviceBusTrigger`.
- **Input binding** reads data from another service.
- **Output binding** writes data to another service.
- **Connection setting** points the runtime to the correct resource or identity-based configuration.

For Service Bus, the host manages message settlement. On success, the message is completed. On failure, the message is abandoned and retried until the queue’s `MaxDeliveryCount` is reached, after which Service Bus dead-letters it. You can also dead-letter explicitly in code when the message is invalid.


Retrieve the Service Bus connection string for local development:

```bash
az servicebus namespace authorization-rule keys list \
  --resource-group rg-az204-func-lab \
  --namespace-name sbaz204lab123 \
  --name RootManageSharedAccessKey \
  --query primaryConnectionString -o tsv
```

**C# HTTP-triggered function**

A C# HTTP trigger is often used as the front door for a workflow. It validates the request, transforms payloads, and either returns data directly or forwards work to a queue.

A minimal isolated worker example:

```csharp
using System.Net;
using Microsoft.Azure.Functions.Worker;
using Microsoft.Azure.Functions.Worker.Http;
using Microsoft.Extensions.Logging;

namespace LabFunctions;

public class SubmitWork
{
    private readonly ILogger _logger;

    public SubmitWork(ILoggerFactory loggerFactory)
    {
        _logger = loggerFactory.CreateLogger<SubmitWork>();
    }

    [Function("SubmitWork")]
    public async Task<HttpResponseData> Run(
        [HttpTrigger(AuthorizationLevel.Function, "post")] HttpRequestData req)
    {
        _logger.LogInformation("HTTP request received.");

        var response = req.CreateResponse(HttpStatusCode.OK);
        await response.WriteStringAsync("Request accepted.");
        return response;
    }
}
```

Key points:

- **AuthorizationLevel.Function** requires a function key unless you choose anonymous access.
- The HTTP trigger is useful for synchronous validation and for starting asynchronous workflows.
- In a real lab, you often parse JSON from the request body and enqueue a message for downstream processing.

If you want the HTTP function to send to Service Bus, you can use an output binding or the Service Bus SDK. The binding approach is simpler for exam labs when the goal is to demonstrate declarative integration.

**C# Service Bus output binding example**

An attribute-based output binding is a common AZ-204 pattern:

```csharp
using System.Net;
using System.Text.Json;
using Microsoft.Azure.Functions.Worker;
using Microsoft.Azure.Functions.Worker.Http;
using Microsoft.Extensions.Logging;

namespace LabFunctions;

public class SubmitWorkToQueue
{
    private readonly ILogger _logger;

    public SubmitWorkToQueue(ILoggerFactory loggerFactory)
    {
        _logger = loggerFactory.CreateLogger<SubmitWorkToQueue>();
    }

    [Function("SubmitWorkToQueue")]
    [ServiceBusOutput("work-items", Connection = "ServiceBusConnection")]
    public async Task<string> Run(
        [HttpTrigger(AuthorizationLevel.Function, "post")] HttpRequestData req)
    {
        _logger.LogInformation("Preparing message for Service Bus.");

        var payload = new
        {
            id = Guid.NewGuid().ToString(),
            submittedAtUtc = DateTime.UtcNow,
            source = "http-trigger"
        };

        return JsonSerializer.Serialize(payload);
    }
}
```

Notes:

- The output binding returns the message body to the queue.
- `Connection = "ServiceBusConnection"` points to an app setting containing the connection string or identity-based configuration.
- If you need multiple messages, use a collector or SDK client instead of a single return value.

**Python queue-triggered function**

For Python, the queue-triggered function is the consumer. In Azure Functions Python, the worker model and package version matter, so keep the example aligned with the runtime you are using.

A common Python v2 model example for Service Bus trigger:

```python
import logging
import azure.functions as func

app = func.FunctionApp()

@app.function_name(name="ProcessWorkItem")
@app.service_bus_queue_trigger(
    arg_name="msg",
    queue_name="work-items",
    connection="ServiceBusConnection"
)
def process_work_item(msg: func.ServiceBusMessage) -> None:
    logging.info("Received message: %s", msg.get_body().decode("utf-8"))
```

Important details:

- The `arg_name` must match the parameter name.
- The `connection` setting must resolve to the Service Bus namespace.
- The exact type annotation support can vary by `azure-functions` package version; if your environment does not support `func.ServiceBusMessage` cleanly, rely on the binding object methods available in that version and verify with the installed package docs.

A Python v1 model example using `function.json` is also useful for exam familiarity:

```python
# __init__.py
import logging
import azure.functions as func

def main(msg: func.ServiceBusMessage) -> None:
    logging.info("Python queue trigger processed: %s", msg.get_body().decode("utf-8"))
```

```json
{
  "scriptFile": "__init__.py",
  "bindings": [
    {
      "name": "msg",
      "type": "serviceBusTrigger",
      "direction": "in",
      "queueName": "work-items",
      "connection": "ServiceBusConnection"
    }
  ]
}
```

Use the v1 model when a lab or template expects `function.json`. Use the v2 decorator model when the project is built around the newer Python programming model.

**Local settings and configuration**

Local development uses `local.settings.json` for app settings. This file is not deployed, so it is safe for local secrets only.

```json
{
  "IsEncrypted": false,
  "Values": {
    "AzureWebJobsStorage": "UseDevelopmentStorage=true",
    "FUNCTIONS_WORKER_RUNTIME": "dotnet-isolated",
    "ServiceBusConnection": "Endpoint=sb://sbaz204lab123.servicebus.windows.net/;SharedAccessKeyName=RootManageSharedAccessKey;SharedAccessKey=..."
  }
}
```

For Python, the worker runtime value changes:

```json
{
  "IsEncrypted": false,
  "Values": {
    "AzureWebJobsStorage": "UseDevelopmentStorage=true",
    "FUNCTIONS_WORKER_RUNTIME": "python",
    "ServiceBusConnection": "Endpoint=sb://sbaz204lab123.servicebus.windows.net/;SharedAccessKeyName=RootManageSharedAccessKey;SharedAccessKey=..."
  }
}
```

`AzureWebJobsStorage` is commonly required by the Functions host for state, scale, and some trigger scenarios. The exact requirement depends on the hosting model and extensions in use, so verify the runtime’s current requirements rather than assuming every scenario behaves identically.

**Identity-based Service Bus connections**

For production, identity-based connections are preferred when supported. The configuration pattern is more specific than simply changing an app setting name.

Typical app settings for a Service Bus connection using managed identity:

```text
ServiceBusConnection__fullyQualifiedNamespace = sbaz204lab123.servicebus.windows.net
ServiceBusConnection__credential = managedidentity
```

Depending on the extension and runtime version, you may also need:

- A user-assigned identity client ID setting.
- Role assignment on the Service Bus namespace or queue.
- A supported Functions extension bundle or package version.

Assign the identity the correct role:

```bash
az role assignment create \
  --assignee <principal-id-or-object-id> \
  --role "Azure Service Bus Data Sender" \
  --scope $(az servicebus namespace show \
    --resource-group rg-az204-func-lab \
    --name sbaz204lab123 \
    --query id -o tsv)
```

For a consumer function, use `Azure Service Bus Data Receiver`. If the function both sends and receives, assign both roles or use a broader role only when justified.

**Deploying the functions**

A typical deployment flow is:

1. Build and test locally.
2. Publish the C# function app.
3. Configure app settings in Azure.
4. Deploy the Python function app or a separate Python function project.
5. Verify trigger execution and logs.

Example C# publish command:

```bash
dotnet publish -c Release
func azure functionapp publish <function-app-name>
```

Example app setting update:

```bash
az functionapp config appsettings set \
  --name <function-app-name> \
  --resource-group rg-az204-func-lab \
  --settings ServiceBusConnection="<connection-string>"
```

If you use separate apps for C# and Python, keep their settings isolated. This reduces confusion when one app uses `dotnet-isolated` and the other uses `python`.

**End-to-end test flow**

A practical test sequence helps you isolate failures quickly.

1. Send a POST request to the HTTP function.
2. Confirm the HTTP function returns success.
3. Check the queue length in Service Bus.
4. Verify the Python function logs show message receipt.
5. Confirm the queue message is completed and removed.

Example request:

```bash
curl -X POST https://<function-app-name>.azurewebsites.net/api/SubmitWorkToQueue \
  -H "Content-Type: application/json" \
  -d '{"name":"test","priority":1}'
```

If the HTTP function uses function-level authorization, include the function key:

```bash
curl -X POST "https://<function-app-name>.azurewebsites.net/api/SubmitWorkToQueue?code=<function-key>" \
  -H "Content-Type: application/json" \
  -d '{"name":"test","priority":1}'
```

**Service Bus behavior and message settlement**

Service Bus trigger processing is governed by message settlement rules:

- On successful execution, the Functions host completes the message.
- On unhandled failure, the message is abandoned and retried.
- When `MaxDeliveryCount` is exceeded, Service Bus moves the message to the dead-letter queue.
- You can explicitly dead-letter a message when validation fails and retrying would not help.

This is different from HTTP triggers, where the client receives an HTTP response and no queue settlement occurs.

**Troubleshooting misconfiguration**

Most lab failures come from configuration mismatches rather than code defects. Start with the host logs and then check the binding configuration.

Common symptoms and likely causes:

- **Function not indexed**
  - Missing extension package or bundle.
  - Invalid binding attribute or `function.json`.
  - Wrong worker model for the project type.

- **Service Bus trigger not firing**
  - Incorrect queue name.
  - Wrong connection setting name.
  - Namespace connection string points to the wrong resource.
  - Identity lacks `Azure Service Bus Data Receiver`.

- **HTTP function returns 500**
  - Unhandled exception in request parsing.
  - Missing environment variable.
  - Serialization issue or null reference.

- **Message appears repeatedly**
  - Function throws before completion.
  - Poison message exceeds `MaxDeliveryCount`.
  - The function is not idempotent and reprocessing causes duplicate side effects.

- **Local run works, Azure fails**
  - App settings were not deployed.
  - Runtime version differs between local and cloud.
  - Extension bundle or package version differs.
  - Managed identity is configured in Azure but not locally.

Useful log commands:

```bash
func start --verbose
az functionapp log tail \
  --name <function-app-name> \
  --resource-group rg-az204-func-lab
```

**Extension and host configuration**

When Service Bus bindings fail to load, check the extension setup. In the in-process model or older projects, extension bundles are often used. In isolated worker projects, package references are more common.

Example `host.json` with an extension bundle:

```json
{
  "version": "2.0",
  "extensionBundle": {
    "id": "Microsoft.Azure.Functions.ExtensionBundle",
    "version": "[4.*, 5.0.0)"
  }
}
```

If the bundle version is wrong for your runtime, bindings may not resolve. Keep the bundle aligned with the Functions runtime and the binding extensions you need.

**Practical remediation checklist**

Use this sequence when a lab is broken:

1. Confirm the function app runtime matches the code model.
2. Verify `FUNCTIONS_WORKER_RUNTIME` is correct.
3. Check the queue name and namespace name.
4. Confirm the connection setting exists in local settings and Azure app settings.
5. Verify the Service Bus role assignment if using managed identity.
6. Inspect logs for binding or indexing errors.
7. Test the Service Bus namespace independently with a CLI or SDK client.

**Local emulator and testing notes**

Service Bus does not have a first-party local emulator equivalent to Azurite for Blob Storage. For local testing, you usually connect to a real Service Bus namespace in Azure. That means your local function app still needs valid connection settings.

A practical local workflow is:

- Use Azurite for storage-related host dependencies if needed.
- Use a real Service Bus namespace for queue trigger testing.
- Run the function host locally with `func start`.
- Send test messages with the Azure CLI or a small SDK script.

Example .NET sender for quick validation:

```csharp
using Azure.Messaging.ServiceBus;

var client = new ServiceBusClient(Environment.GetEnvironmentVariable("ServiceBusConnection"));
var sender = client.CreateSender("work-items");

await sender.SendMessageAsync(new ServiceBusMessage("hello from test sender"));
```

**Why this lab matters for AZ-204**

This lab tests several exam skills at once:

- Choosing the correct trigger for the workload.
- Configuring bindings and app settings correctly.
- Understanding how Service Bus decouples producers and consumers.
- Using managed identity or connection strings appropriately.
- Diagnosing runtime, indexing, and binding failures.

If you can explain why a message is not processed, where the configuration lives, and how the host settles messages, you are covering the core AZ-204 serverless integration objectives.

**Quick reference**

- **HTTP trigger**: starts on request and is useful for synchronous entry points.
- **Service Bus trigger**: starts on queue or topic messages and is useful for asynchronous processing.
- **Output binding**: simplifies sending messages without explicit SDK code.
- **Managed identity**: preferred for production when supported.
- **Dead-letter queue**: holds messages that cannot be processed after retries or are explicitly dead-lettered.
