## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Overview**

Azure Functions can react to messages from **Azure Service Bus** and events from **Azure Event Grid** without you managing servers. These triggers solve different problems:

- **Service Bus** is a durable messaging system for work that must be processed reliably, often with ordering, retries, and dead-lettering.
- **Event Grid** is an event routing service for reacting to state changes in Azure services or custom publishers with low-latency push delivery.

For AZ-204, you should know how to create each trigger, how delivery and retry behavior differs, and how to configure the function app securely and correctly.

**Service Bus-triggered functions**

A Service Bus trigger runs when a message arrives on a queue or subscription. The Functions runtime receives the message from the broker, manages the lock, and invokes your function. Your code does not directly call the Service Bus receive API in the trigger path.

Common use cases include:

- Order processing
- Background workflow orchestration
- Decoupling microservices
- Reliable command handling

A trigger can target:

- A **queue**
- A **topic subscription**

If the entity is session-enabled, you must configure the function to process sessions.

**Trigger binding behavior**

The runtime handles message retrieval and settlement behavior based on the trigger configuration and function success or failure.

- If the function completes successfully, the message is typically completed automatically.
- If the function throws an exception, the runtime abandons the message and it may be retried.
- If retries exceed the queue or subscription delivery limit, the message moves to the **dead-letter queue** (DLQ).

> Do not treat a Service Bus trigger as a generic queue consumer with manual polling. The Functions runtime owns the trigger loop, lock handling, and retry integration.

**Function code examples**

C# isolated worker example:

```csharp
using Microsoft.Azure.Functions.Worker;
using Microsoft.Extensions.Logging;

public class ProcessOrder
{
    private readonly ILogger<ProcessOrder> _logger;

    public ProcessOrder(ILogger<ProcessOrder> logger)
    {
        _logger = logger;
    }

    [Function("ProcessOrder")]
    public void Run(
        [ServiceBusTrigger("orders", Connection = "ServiceBusConnection")]
        string message,
        FunctionContext context)
    {
        _logger.LogInformation("Received order message: {Message}", message);
    }
}
```

JavaScript example:

```javascript
module.exports = async function (context, message) {
  context.log('Service Bus message:', message);
};
```

Python example:

```python
import logging
import azure.functions as func

def main(msg: func.ServiceBusMessage):
    logging.info("Service Bus message: %s", msg.get_body().decode("utf-8"))
```

**Configuration and host settings**

Connection information is usually stored in application settings, not in code.

```json
{
  "Values": {
    "AzureWebJobsStorage": "UseDevelopmentStorage=true",
    "FUNCTIONS_WORKER_RUNTIME": "dotnet-isolated",
    "ServiceBusConnection": "<service-bus-connection-string>"
  }
}
```

For newer Service Bus extension versions, configuration is typically under `extensions.serviceBus` in `host.json`. Exact option names vary by extension version, so verify the version you are using before copying examples.

Example shape:

```json
{
  "version": "2.0",
  "extensions": {
    "serviceBus": {
      "prefetchCount": 0,
      "messageHandlerOptions": {
        "autoCompleteMessages": true,
        "maxConcurrentCalls": 16,
        "maxAutoLockRenewalDuration": "00:05:00"
      }
    }
  }
}
```

Use this as a pattern only. In current extension versions, some settings have changed names or moved, especially between legacy and newer worker models.

**Session-enabled entities**

If the queue or subscription requires sessions, the trigger must be session-aware. Sessions preserve ordered processing for messages with the same session ID.

Typical session scenarios:

- Customer-specific workflows
- Ordered event streams
- Correlated multi-step processing

Example:

```csharp
[Function("ProcessSession")]
public void Run(
    [ServiceBusTrigger("session-queue", Connection = "ServiceBusConnection", IsSessionsEnabled = true)]
    string message)
{
    // Process session message
}
```

When using sessions:

- The runtime locks a session, not just a single message.
- Messages in the same session are processed in order.
- You should keep session processing efficient to avoid blocking other messages in that session.

**Manual settlement and advanced message handling**

In some scenarios, you need manual control over settlement. This is useful when you want to complete, abandon, dead-letter, or defer a message based on business rules.

Common settlement actions:

- **Complete**: remove the message from the queue after successful processing.
- **Abandon**: release the message so it can be retried.
- **Dead-letter**: move the message to the DLQ with a reason and description.
- **Defer**: keep the message in the queue but make it unavailable for normal retrieval until later.

Example using the Service Bus SDK inside a function:

```csharp
using Azure.Messaging.ServiceBus;
using Microsoft.Azure.Functions.Worker;

public class ManualSettlement
{
    [Function("ManualSettlement")]
    public async Task Run(
        [ServiceBusTrigger("orders", Connection = "ServiceBusConnection")]
        ServiceBusReceivedMessage message,
        ServiceBusMessageActions messageActions)
    {
        if (message.Subject == "invalid")
        {
            await messageActions.DeadLetterMessageAsync(
                message,
                deadLetterReason: "ValidationFailed",
                deadLetterErrorDescription: "Order payload failed validation");
            return;
        }

        await messageActions.CompleteMessageAsync(message);
    }
}
```

Use manual settlement when:

- You need to inspect the message before deciding the outcome.
- You want custom dead-letter reasons.
- You need to defer processing for later correlation.

**Poison message handling**

A poison message is one that repeatedly fails processing. Service Bus tracks delivery attempts and eventually dead-letters the message when the delivery count exceeds the entity’s configured maximum delivery count.

Practical guidance:

- Set a sensible `maxDeliveryCount` on the queue or subscription.
- Log enough context to diagnose failures.
- Inspect the DLQ regularly.
- Build a replay or remediation process for dead-lettered messages.

Example CLI to inspect queue settings:

```bash
az servicebus queue show \
  --resource-group rg-app \
  --namespace-name sbns-prod \
  --name orders
```

To read from a DLQ, use the `/$DeadLetterQueue` subqueue path with the SDK or tooling.

**Event Grid-triggered functions**

An Event Grid trigger runs when Event Grid delivers an event to the function endpoint. Event Grid is push-based and optimized for event notification, not durable work queue semantics.

Use Event Grid when you want to react to:

- Blob creation or deletion
- Resource changes
- Custom application events
- Infrastructure lifecycle events

Event Grid provides:

- Low-latency delivery
- At-least-once delivery
- Retry with backoff
- Dead-letter support to a storage destination

It does not provide:

- Consumer-controlled completion
- Message locks
- Session ordering
- Queue-style work distribution

**Event schema choices**

Event Grid supports different schemas. The two common ones are:

- **Event Grid schema**
- **CloudEvents 1.0**

The schema you choose affects the payload shape and sometimes the binding type or SDK model you use.

General guidance:

- Use Event Grid schema when integrating with Azure-native event sources and older examples.
- Use CloudEvents when you need broader interoperability with external systems.

**Event Grid trigger function example**

C# isolated worker example:

```csharp
using Azure.Messaging.EventGrid;
using Microsoft.Azure.Functions.Worker;
using Microsoft.Extensions.Logging;

public class BlobCreatedHandler
{
    private readonly ILogger<BlobCreatedHandler> _logger;

    public BlobCreatedHandler(ILogger<BlobCreatedHandler> logger)
    {
        _logger = logger;
    }

    [Function("BlobCreatedHandler")]
    public void Run([EventGridTrigger] EventGridEvent eventGridEvent)
    {
        _logger.LogInformation("Event type: {Type}", eventGridEvent.EventType);
        _logger.LogInformation("Subject: {Subject}", eventGridEvent.Subject);
    }
}
```

If you use CloudEvents, bind to a CloudEvent type instead of `EventGridEvent`.

**Creating an Event Grid subscription**

A common exam scenario is wiring a Storage account blob event to a Function App.

Typical steps:

1. Create the function with an Event Grid trigger.
2. Deploy the function app to Azure.
3. Create an Event Grid subscription on the source resource.
4. Choose the function endpoint as the destination.
5. Configure filters if you only want specific events or subjects.
6. Validate the subscription handshake and event delivery.

Azure CLI example for a storage blob-created subscription:

```bash
az eventgrid event-subscription create \
  --name blob-created-to-function \
  --source-resource-id /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<storageAccount> \
  --endpoint-type azurefunction \
  --endpoint /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Web/sites/<functionApp>/functions/BlobCreatedHandler \
  --included-event-types Microsoft.Storage.BlobCreated \
  --subject-begins-with /blobServices/default/containers/images/blobs/
```

Useful filters include:

- `--included-event-types`
- `--subject-begins-with`
- `--subject-ends-with`
- advanced filters for data fields when supported by the source and subscription type

**Security and authorization**

Event Grid does not call your function like a generic public HTTP API. It uses the function webhook endpoint and performs validation during subscription creation and delivery.

Important points:

- The function endpoint must be reachable by Event Grid.
- The subscription handshake validates the destination before events flow.
- For Azure Functions, the destination is commonly the function endpoint itself, not an arbitrary route.
- Depending on the integration path, you may use a function key or managed identity-based authorization where supported.

For secure production setups:

- Prefer managed identity where the integration supports it.
- Restrict public access where possible.
- Use private networking only if the event delivery path supports it for your scenario.
- Avoid exposing unnecessary HTTP endpoints.

**Local testing and troubleshooting**

Event Grid trigger testing is different from Service Bus testing.

For Service Bus:

- Send a message to the queue or topic.
- Watch function logs.
- Check retries and DLQ if processing fails.

For Event Grid:

- Publish a test event from the source or use the Azure portal test feature where available.
- Confirm the subscription is active.
- Check for validation failures, endpoint reachability issues, and schema mismatches.

Common Event Grid issues:

- Subscription validation fails because the function app is not reachable.
- The function expects the wrong schema type.
- The endpoint returns an authorization error.
- The event filter excludes the event.
- The function throws and Event Grid retries until the retry policy expires.

Example of a simple local test payload for Event Grid schema:

```json
[
  {
    "id": "test-1",
    "eventType": "Contoso.Items.Created",
    "subject": "/items/42",
    "eventTime": "2026-04-26T12:00:00Z",
    "data": {
      "name": "sample"
    },
    "dataVersion": "1.0",
    "metadataVersion": "1"
  }
]
```

**Retry behavior comparison**

Service Bus and Event Grid both support retries, but they behave differently.

| Aspect | Service Bus trigger | Event Grid trigger |
|---|---|---|
| Delivery model | Pull-based message processing by the Functions runtime | Push-based event delivery from Event Grid |
| Locking | Message lock managed by runtime | No message lock |
| Completion | Automatic or manual settlement | No completion API in the trigger |
| Retry control | Delivery count on entity, runtime retries, DLQ | Event Grid retries delivery, then dead-letters if configured |
| Ordering | Possible with sessions | Not guaranteed |
| Best fit | Reliable work processing | Event notification and reactive automation |

Use Service Bus when the work item must be processed exactly once in business terms, or when you need queue semantics and settlement control. Use Event Grid when you need to react to events from Azure resources or custom publishers with low latency.

**Practical implementation guidance**

When you design the function:

- Keep the handler small and fast.
- Offload long-running work to another queue or durable workflow if needed.
- Make processing idempotent because both systems can deliver more than once.
- Log correlation IDs, message IDs, and event IDs for traceability.
- Validate payloads before acting on them.

Example idempotency pattern:

```csharp
if (await _processedEvents.ExistsAsync(eventGridEvent.Id))
{
    _logger.LogInformation("Duplicate event ignored: {Id}", eventGridEvent.Id);
    return;
}

await _processedEvents.MarkAsync(eventGridEvent.Id);
```

**Common exam points**

- **Service Bus trigger** is for queue or topic subscription messages.
- **Session-enabled** entities require session-aware processing.
- **Dead-letter queue** is where poison messages end up after repeated failures.
- **Event Grid trigger** is for event notifications, not durable work queues.
- **CloudEvents** and Event Grid schema are different payload formats.
- **Subscription filters** help reduce noise and route only relevant events.
- **Retry and dead-letter behavior** differ significantly between the two services.

**Quick decision guide**

- Use **Service Bus** when you need:
  - Reliable command processing
  - Ordering with sessions
  - Manual settlement
  - Queue-based backpressure

- Use **Event Grid** when you need:
  - Reactive event handling
  - Azure resource notifications
  - Low-latency fan-out
  - Simple event routing with filters

**Troubleshooting checklist**

1. Confirm the function app is deployed and running.
2. Verify the trigger binding matches the schema and entity type.
3. Check connection settings in application configuration.
4. Inspect logs for authorization, validation, or deserialization errors.
5. For Service Bus, inspect retries and the DLQ.
6. For Event Grid, verify subscription status, filters, and endpoint reachability.
