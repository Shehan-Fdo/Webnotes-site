## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Function App deployment overview**

Azure Functions lets you run event-driven code without managing servers. For AZ-204, you need to know how to deploy Function Apps, how identity is configured for secure access to Azure resources, and how to inspect telemetry in Application Insights when something fails or behaves unexpectedly.

A Function App deployment usually includes:
- The **Function App** resource itself
- A **hosting plan** such as Consumption, Premium, or Dedicated
- A **storage account** for runtime state and triggers
- Optional **Application Insights** for observability
- Optional **deployment slots** for safer releases

The hosting plan affects scaling, cold start behavior, networking options, and slot support. Premium and Dedicated plans are more likely to be used when you need predictable performance, VNet integration, or deployment slots.

**Deployment options and when to use them**

You can deploy Azure Functions in several ways. The right choice depends on your pipeline, runtime model, and whether you need infrastructure automation.

- **Azure CLI / Core Tools**
  - Good for local development and quick publishing.
  - Common with `func azure functionapp publish`.
  - Verify your Core Tools version matches the runtime model you use, especially for isolated worker apps.
- **Zip deploy**
  - Useful for CI/CD pipelines and repeatable releases.
  - Uploads a packaged app artifact to the Function App.
  - Works well with build outputs from GitHub Actions or Azure DevOps.
- **Run-from-package**
  - Mounts a package read-only for consistent deployments.
  - Reduces file-copy drift between deployments.
  - Common in automated pipelines.
- **GitHub Actions / Azure DevOps**
  - Best for production-grade continuous integration and continuous deployment.
  - Lets you build, test, package, and deploy in one workflow.
- **ARM/Bicep / Infrastructure as Code**
  - Best for provisioning the Function App, plan, storage, identity, monitoring, and app settings consistently.
  - Often paired with a separate code deployment step.
  - Important for exam scenarios where repeatability and environment parity matter.

> `AzureWebJobsStorage=UseDevelopmentStorage=true` is only valid for local development with Azurite or a storage emulator. Do not use it as a production app setting.

**Common deployment commands**

Use Azure Functions Core Tools for local publish scenarios and Azure CLI for resource management.

```bash
# Publish a Function App from the local project
func azure functionapp publish func-demo-prod

# Create a resource group
az group create -n rg-func-prod -l eastus

# Create a storage account for the Function App
az storage account create \
  -g rg-func-prod \
  -n stfuncprod001 \
  -l eastus \
  --sku Standard_LRS

# Create a Function App plan
az functionapp plan create \
  -g rg-func-prod \
  -n plan-func-prod \
  -l eastus \
  --sku EP1 \
  --is-linux false
```

For production deployments, prefer a pipeline that packages the app and deploys the artifact rather than relying on manual publish from a workstation.

**Infrastructure as Code pattern**

A typical Bicep deployment provisions the Function App, storage, identity, and monitoring together. This is useful when you want the app settings and identity assignments to be reproducible.

```bicep
param location string = resourceGroup().location
param functionAppName string
param storageAccountName string
param appInsightsName string

resource storage 'Microsoft.Storage/storageAccounts@2023-01-01' = {
  name: storageAccountName
  location: location
  sku: {
    name: 'Standard_LRS'
  }
  kind: 'StorageV2'
  properties: {
    allowBlobPublicAccess: false
  }
}

resource appInsights 'Microsoft.Insights/components@2020-02-02' = {
  name: appInsightsName
  location: location
  kind: 'web'
  properties: {
    Application_Type: 'web'
    WorkspaceResourceId: ''
  }
}
```

In practice, you usually also define the hosting plan, Function App, system-assigned identity, and app settings such as `AzureWebJobsStorage` and the Application Insights connection string.

**Deployment slots and safe releases**

Deployment slots let you stage a new version before swapping it into production. This is especially useful on Premium and Dedicated plans.

- **Production slot**
  - Serves live traffic.
- **Staging slot**
  - Receives the new build for validation.
- **Swap**
  - Moves the staging version into production with minimal downtime.

Slot behavior matters:
- **Sticky settings** stay with a slot during swap.
- Non-sticky settings move with the content unless marked as slot-specific.
- Connection strings and app settings can be configured as slot settings when you need environment-specific values.

Typical slot workflow:
1. Deploy to the staging slot.
2. Validate health, logs, and dependencies.
3. Swap staging into production.
4. Keep a rollback path by swapping back if needed.

**Application settings that matter**

Function Apps rely on app settings for runtime behavior and integration.

- `AzureWebJobsStorage`
  - Required by the Functions runtime for triggers, bindings, and state.
  - Must point to a real storage account in Azure.
- `FUNCTIONS_WORKER_RUNTIME`
  - Identifies the language worker, such as `dotnet`, `node`, `python`, or `java`.
- `FUNCTIONS_EXTENSION_VERSION`
  - Controls the Functions runtime version, often `~4`.
- `APPLICATIONINSIGHTS_CONNECTION_STRING`
  - Preferred modern setting for Application Insights telemetry.
- `APPINSIGHTS_INSTRUMENTATIONKEY`
  - Legacy-style setting still seen in older apps and some migration scenarios.

A modern Application Insights connection string usually looks like this:

```text
InstrumentationKey=00000000-0000-0000-0000-000000000000;IngestionEndpoint=https://eastus-0.in.applicationinsights.azure.com/;LiveEndpoint=https://eastus.livediagnostics.monitor.azure.com/
```

The exact endpoints vary by region and cloud. Prefer the full connection string over only the instrumentation key because it supports current ingestion patterns and future platform changes.

**Managed identity fundamentals**

Managed identity gives your Function App an identity in Microsoft Entra ID without storing credentials in code or configuration. Use it when the function needs to access Azure resources securely.

There are two types:
- **System-assigned managed identity**
  - Tied to one Azure resource.
  - Lifecycle matches the Function App.
  - Good default choice for most apps.
- **User-assigned managed identity**
  - Created as a separate resource.
  - Can be shared across multiple apps.
  - Useful when you need identity reuse or independent lifecycle management.

Managed identity is commonly used for:
- Azure Key Vault secret access
- Azure Storage access
- Azure Service Bus messaging
- Azure Event Hubs ingestion
- Azure SQL authentication
- Azure Container Registry pulls in some scenarios

**How to enable managed identity**

You can enable identity in the portal, CLI, or IaC.

```bash
# Enable system-assigned identity on a Function App
az functionapp identity assign \
  -g rg-func-prod \
  -n func-demo-prod

# Show the principal ID
az functionapp identity show \
  -g rg-func-prod \
  -n func-demo-prod
```

After enabling identity, assign the correct Azure role at the target resource scope.

```bash
# Grant Storage Blob Data Contributor on a storage account
az role assignment create \
  --assignee-object-id <principal-id> \
  --assignee-principal-type ServicePrincipal \
  --role "Storage Blob Data Contributor" \
  --scope $(az storage account show -g rg-func-prod -n stfuncprod001 --query id -o tsv)
```

**Using managed identity in code**

In Azure SDKs, you typically use `DefaultAzureCredential` so the same code works locally and in Azure. Locally, it can use your developer sign-in. In Azure, it can use the managed identity.

```csharp
using Azure.Identity;
using Azure.Storage.Blobs;

var blobServiceClient = new BlobServiceClient(
    new Uri("https://stfuncprod001.blob.core.windows.net"),
    new DefaultAzureCredential());

var containerClient = blobServiceClient.GetBlobContainerClient("input");
await containerClient.CreateIfNotExistsAsync();
```

Other common patterns:
- **Service Bus**
  - Use `ServiceBusClient` with `DefaultAzureCredential`.
- **Event Hubs**
  - Use `EventHubProducerClient` or `EventHubConsumerClient` with `DefaultAzureCredential`.
- **Key Vault**
  - Use `SecretClient` with `DefaultAzureCredential`.

Example for Key Vault:

```csharp
using Azure.Identity;
using Azure.Security.KeyVault.Secrets;

var client = new SecretClient(
    new Uri("https://kv-prod-001.vault.azure.net/"),
    new DefaultAzureCredential());

KeyVaultSecret secret = await client.GetSecretAsync("DbPassword");
```

The key idea is that the Function App authenticates as itself, and Azure RBAC or Key Vault access policies authorize what it can do.

**Portal configuration flow**

When creating or updating a Function App in the portal, you often configure monitoring and identity during the same workflow.

1. Create the Function App and choose the runtime stack.
2. Select or create the storage account.
3. Enable **Application Insights** during creation if available.
4. Choose or create the Application Insights resource.
5. Enable **system-assigned managed identity** if the app needs secure Azure resource access.
6. Add required app settings such as `AzureWebJobsStorage` and `FUNCTIONS_WORKER_RUNTIME`.
7. Save and restart the app if prompted.

If you add Application Insights later, verify that the connection string is present in configuration and that telemetry is flowing after a restart.

**Application Insights setup and telemetry model**

Application Insights is the primary observability tool for Azure Functions. It collects request telemetry, dependencies, exceptions, traces, and custom events so you can diagnose failures and performance issues.

There are two common integration models:
- **Classic instrumentation-key setup**
  - Uses `APPINSIGHTS_INSTRUMENTATIONKEY`.
  - Older model, still seen in legacy apps.
- **Workspace-based Application Insights**
  - Stores telemetry in a Log Analytics workspace.
  - Better for unified querying, retention, and Azure Monitor integration.
  - Common in modern deployments.

Workspace-based Application Insights is often preferred because it aligns with Azure Monitor and Log Analytics. In exam scenarios, know that the telemetry still appears in Application Insights experiences, but the underlying storage and query model are workspace-backed.

**Important telemetry tables and fields**

In Log Analytics, Function App telemetry commonly appears in these tables:
- **requests**
  - Function invocation requests and HTTP-triggered executions.
- **dependencies**
  - Outbound calls to SQL, Storage, HTTP APIs, Service Bus, and other services.
- **exceptions**
  - Unhandled exceptions and captured failures.
- **traces**
  - Log messages from `ILogger`, `context.log`, or equivalent logging APIs.
- **customEvents**
  - Application-defined events for business actions or milestones.
- **customMetrics**
  - Application-defined numeric measurements.

Useful correlation fields:
- `operation_Id`
  - Correlates telemetry across a single invocation or transaction.
- `operation_ParentId`
  - Links child telemetry to a parent operation.
- `cloud_RoleName`
  - Identifies the app or service emitting telemetry.
- `timestamp`
  - Event time.
- `severityLevel`
  - Trace severity in some telemetry types.

For Function Apps, `operation_Id` is usually the safest way to connect requests, traces, dependencies, and exceptions.

**KQL for Function App troubleshooting**

Use Kusto Query Language (KQL) to inspect telemetry in Application Insights or Log Analytics.

A common pattern is to start with `requests`, then join related telemetry by `operation_Id`.

```kusto
requests
| where timestamp > ago(24h)
| where success == false
| project timestamp, operation_Id, name, resultCode, duration, cloud_RoleName
| order by timestamp desc
```

To inspect traces for a specific failed invocation:

```kusto
let failedRequests =
    requests
    | where timestamp > ago(24h)
    | where success == false
    | project operation_Id, requestName = name, requestTime = timestamp;
traces
| where timestamp > ago(24h)
| join kind=inner failedRequests on operation_Id
| project timestamp, requestTime, requestName, message, severityLevel, operation_Id
| order by timestamp desc
```

To inspect dependencies for a specific invocation:

```kusto
let invocationId = "<operation-id>";
dependencies
| where operation_Id == invocationId
| project timestamp, name, target, type, success, duration, resultCode
| order by timestamp asc
```

To inspect exceptions tied to failed requests:

```kusto
requests
| where timestamp > ago(24h)
| where success == false
| join kind=leftouter exceptions on operation_Id
| project requests.timestamp, requests.name, requests.resultCode, exceptions.type, exceptions.outerMessage, exceptions.problemId
| order by requests.timestamp desc
```

To inspect custom events for a function workflow:

```kusto
customEvents
| where timestamp > ago(24h)
| where name has "Order"
| project timestamp, name, operation_Id, customDimensions
| order by timestamp desc
```

For HTTP-triggered functions, the `requests.name` field often includes the HTTP method and route, not just the function name. Use `contains`, `has`, or `startswith` rather than assuming an exact function-name match.

```kusto
requests
| where timestamp > ago(24h)
| where name has "HttpTrigger1"
| project timestamp, name, resultCode, success, operation_Id
```

**Practical telemetry workflow**

When a Function App fails in production, follow a repeatable investigation path.

1. Check `requests` for failed invocations and note `operation_Id`.
2. Open `traces` for the same `operation_Id` to see log messages.
3. Inspect `dependencies` to find downstream service failures.
4. Review `exceptions` for stack traces and root causes.
5. Correlate `customEvents` or `customMetrics` if the app emits business telemetry.
6. Compare the failing invocation with a successful one to spot differences.

This workflow is more reliable than searching by function name alone because telemetry names vary by trigger type, route, and runtime.

**Streaming logs and live troubleshooting**

For live troubleshooting, you can stream logs from the app, but the exact command depends on the hosting model and logging configuration.

- For App Service and Function Apps, `az webapp log tail` is the common Azure CLI log streaming command.
- Before streaming, enable application logging and ensure the app writes to the configured log sink.
- For local development, use `func start` to view runtime logs directly.

Example:

```bash
# Enable application logging to the filesystem
az webapp log config \
  -g rg-func-prod \
  -n func-demo-prod \
  --application-logging filesystem \
  --level information

# Tail logs
az webapp log tail \
  -g rg-func-prod \
  -n func-demo-prod
```

If logs do not appear, verify:
- Application logging is enabled
- The app is running
- The correct subscription and resource group are selected
- The function actually emits logs at the configured level

**Local development versus Azure runtime**

Local development and Azure runtime differ in important ways.

- **Local**
  - Uses `local.settings.json` for secrets and connection strings.
  - Can use Azurite for storage emulation.
  - Often uses your developer identity for Azure SDK authentication.
- **Azure**
  - Uses app settings in the Function App configuration.
  - Requires a real storage account for `AzureWebJobsStorage`.
  - Can use managed identity instead of secrets.

Example `local.settings.json`:

```json
{
  "IsEncrypted": false,
  "Values": {
    "AzureWebJobsStorage": "UseDevelopmentStorage=true",
    "FUNCTIONS_WORKER_RUNTIME": "dotnet-isolated",
    "APPLICATIONINSIGHTS_CONNECTION_STRING": "InstrumentationKey=00000000-0000-0000-0000-000000000000;IngestionEndpoint=https://eastus-0.in.applicationinsights.azure.com/"
  }
}
```

In Azure, replace emulator values with real resource connection details and avoid storing secrets in source control.

**Exam-oriented distinctions**

- **Managed identity vs connection strings**
  - Managed identity removes secret management for Azure resource access.
  - Connection strings are still common for telemetry and some legacy integrations.
- **Application Insights vs Log Analytics**
  - Application Insights is the application monitoring experience.
  - Workspace-based setups store data in Log Analytics and use KQL for analysis.
- **Requests vs traces**
  - `requests` represent invocations or HTTP operations.
  - `traces` represent log output and diagnostic messages.
- **Deployment slots vs direct deployment**
  - Slots reduce risk by allowing validation before production swap.
  - Direct deployment is simpler but riskier for production changes.
- **Core Tools publish vs pipeline deployment**
  - `func azure functionapp publish` is useful for development and quick releases.
  - CI/CD and IaC are better for repeatable production deployments.

**Common pitfalls**

- Using `operation_Name` as the primary correlation key for traces instead of `operation_Id`.
- Assuming `requests.name` is always just the function name.
- Treating `az functionapp log tail` as the standard CLI log-streaming command for Function Apps.
- Using `UseDevelopmentStorage=true` in Azure instead of only local development.
- Forgetting to enable Application Insights or application logging before troubleshooting.
- Assigning managed identity but not granting the required RBAC role on the target resource.
- Deploying to production without validating slot-specific settings and swap behavior.

**Quick reference**

| Area | What to remember | Why it matters |
|---|---|---|
| Deployment | Use Core Tools, zip deploy, pipelines, or IaC | Different scenarios need different release strategies |
| Storage | `AzureWebJobsStorage` must point to real storage in Azure | The runtime depends on it |
| Identity | Prefer managed identity with `DefaultAzureCredential` | Avoids secrets in code and config |
| Monitoring | Use `APPLICATIONINSIGHTS_CONNECTION_STRING` | Modern telemetry configuration |
| Correlation | Use `operation_Id` across tables | Reliable end-to-end tracing |
| Slots | Mark settings sticky when needed | Prevents environment drift during swap |

**Minimal end-to-end example**

A typical production setup might look like this:
- Provision the Function App with Bicep.
- Enable a system-assigned managed identity.
- Assign `Storage Blob Data Contributor` to the identity on the storage account.
- Configure `APPLICATIONINSIGHTS_CONNECTION_STRING`.
- Deploy the app with a pipeline.
- Validate in a staging slot.
- Swap to production after checking `requests`, `traces`, `dependencies`, and `exceptions`.

That sequence reflects the practical AZ-204 mindset: deploy securely, authenticate without secrets, and verify behavior with telemetry.
