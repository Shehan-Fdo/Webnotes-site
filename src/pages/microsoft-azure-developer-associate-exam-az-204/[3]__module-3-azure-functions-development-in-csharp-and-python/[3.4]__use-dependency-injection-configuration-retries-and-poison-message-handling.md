## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Dependency injection in Azure apps**

**Dependency injection** (DI) lets you supply services to classes instead of creating them inside the class. In AZ-204 scenarios, this matters because it improves testability, separates concerns, and makes Azure SDK clients, repositories, and business services easier to swap or mock.

In ASP.NET Core, DI is built in. In Azure Functions, DI is also available, but the registration model depends on the hosting model. For exam purposes, know that you typically register services once at startup and then consume them through constructor injection.

- Use DI when a class depends on:
  - Azure SDK clients
  - Logging
  - Configuration-bound options
  - Data access services
- Prefer constructor injection for required dependencies.
- Use scoped lifetimes when a service should live per request or per function invocation in web-style apps.
- Use singleton lifetimes for thread-safe, reusable clients and stateless services.
- Use transient lifetimes for lightweight services that do not hold state.

A common Azure pattern is to register SDK clients through the built-in service collection rather than creating them repeatedly. This helps reuse connections and aligns with recommended client lifetime behavior.

```csharp
using Azure.Identity;
using Azure.Storage.Blobs;
using Microsoft.Extensions.DependencyInjection;

var services = new ServiceCollection();

services.AddSingleton(new BlobServiceClient(
    new Uri("https://mystorageaccount.blob.core.windows.net"),
    new DefaultAzureCredential()));

services.AddScoped<IOrderProcessor, OrderProcessor>();

var provider = services.BuildServiceProvider();
```

In ASP.NET Core, the same pattern is usually expressed in `Program.cs`:

```csharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllers();
builder.Services.AddSingleton(new BlobServiceClient(
    new Uri(builder.Configuration["Storage:BlobServiceUri"]!),
    new DefaultAzureCredential()));

builder.Services.AddScoped<IOrderRepository, OrderRepository>();
builder.Services.AddTransient<IRetryPolicyFactory, RetryPolicyFactory>();

var app = builder.Build();
app.MapControllers();
app.Run();
```

**Azure Functions DI patterns**

Azure Functions supports DI, but you should be careful about the hosting model and the function app lifecycle. The exam often expects you to know that:

- You register services during startup.
- You inject dependencies into function classes through constructors.
- You avoid static state for services that should be configurable or testable.
- You keep function methods thin and delegate work to services.

For isolated worker Functions, startup commonly uses `HostBuilder`:

```csharp
using Microsoft.Extensions.Hosting;
using Microsoft.Extensions.DependencyInjection;
using Azure.Identity;
using Azure.Storage.Queues;

var host = new HostBuilder()
    .ConfigureFunctionsWorkerDefaults()
    .ConfigureServices(services =>
    {
        services.AddSingleton(new QueueServiceClient(
            new Uri("https://mystorageaccount.queue.core.windows.net"),
            new DefaultAzureCredential()));

        services.AddSingleton<IMessageHandler, MessageHandler>();
    })
    .Build();

host.Run();
```

For in-process Functions, startup is typically done with a `FunctionsStartup` class. The exact model depends on the Functions runtime version, so focus on the concept: register once, inject later.

> In exam questions, if a service must be reused across invocations and is thread-safe, singleton is usually the best fit. If it depends on per-request state, use scoped or transient as appropriate.

**Configuration and the options pattern**

Configuration in Azure apps usually comes from `appsettings.json`, environment variables, Azure App Configuration, Key Vault references, or function app settings. The **options pattern** binds configuration sections to strongly typed classes, which reduces stringly typed code and makes validation easier.

Use the options pattern when:
- You want typed access to configuration values.
- You want to validate settings at startup.
- You want to separate configuration from business logic.
- You need named or reloadable settings.

```json
{
  "Storage": {
    "BlobServiceUri": "https://mystorageaccount.blob.core.windows.net",
    "ContainerName": "orders"
  },
  "Retry": {
    "MaxRetries": 5,
    "DelaySeconds": 2
  }
}
```

```csharp
public sealed class StorageOptions
{
    public string BlobServiceUri { get; set; } = string.Empty;
    public string ContainerName { get; set; } = string.Empty;
}
```

```csharp
var builder = WebApplication.CreateBuilder(args);

builder.Services
    .AddOptions<StorageOptions>()
    .Bind(builder.Configuration.GetSection("Storage"))
    .Validate(o => Uri.IsWellFormedUriString(o.BlobServiceUri, UriKind.Absolute),
        "Storage:BlobServiceUri must be a valid absolute URI")
    .ValidateOnStart();
```

The options interfaces matter in AZ-204 scenarios:

- `IOptions<T>`
  - Use for singleton-style access to configuration.
  - Best when settings do not need to change after startup.
- `IOptionsSnapshot<T>`
  - Use in scoped scenarios, especially ASP.NET Core web requests.
  - Reads updated values per scope.
  - Not typically used in singleton services.
- `IOptionsMonitor<T>`
  - Use when you need change notifications and reload-on-change behavior.
  - Good for long-lived services that should react to config updates.

Named options are useful when you have multiple configurations of the same type, such as multiple storage accounts or multiple retry profiles.

```csharp
builder.Services.AddOptions<StorageOptions>("Primary")
    .Bind(builder.Configuration.GetSection("Storage:Primary"));

builder.Services.AddOptions<StorageOptions>("Secondary")
    .Bind(builder.Configuration.GetSection("Storage:Secondary"));
```

**Practical configuration sources**

In Azure, environment variables are common because they work well with App Service, Functions, containers, and CI/CD pipelines. Use them for secrets only when paired with secure secret management practices.

- Use `appsettings.json` for local defaults.
- Use environment variables for deployment-specific overrides.
- Use Azure Key Vault for secrets and certificates.
- Use Azure App Configuration for centralized feature flags and dynamic settings.
- Use managed identity to access Azure resources without storing credentials.

A typical local development command might look like this:

```bash
dotnet user-secrets set "Storage:ContainerName" "orders"
dotnet user-secrets set "Retry:MaxRetries" "5"
```

**Retries and transient fault handling**

Retries help your app survive transient failures such as throttling, network blips, or temporary service unavailability. In Azure, transient faults are common, so retry logic is a core exam topic.

The key idea is to retry only failures that are likely to succeed on a later attempt. Do not retry permanent failures such as invalid credentials, malformed payloads, or authorization errors.

Common retry strategies:
- **Fixed delay**: wait the same amount of time between attempts.
- **Exponential backoff**: increase delay after each failure.
- **Jitter**: add randomness to reduce retry storms.
- **Circuit breaker**: stop calling a failing dependency temporarily.

Azure SDK clients often include built-in retry policies. In many cases, you should start with the SDK defaults and customize only when you have a clear reason, such as:
- A service-specific throttling pattern
- A need for shorter or longer timeouts
- A workload with strict latency requirements
- A dependency that needs custom handling beyond the SDK defaults

Example Azure SDK client configuration:

```csharp
using Azure.Core;
using Azure.Storage.Blobs;

var options = new BlobClientOptions
{
    Retry =
    {
        Mode = RetryMode.Exponential,
        MaxRetries = 5,
        Delay = TimeSpan.FromSeconds(0.8),
        MaxDelay = TimeSpan.FromSeconds(8),
        NetworkTimeout = TimeSpan.FromSeconds(100)
    }
};

var client = new BlobServiceClient(
    new Uri("https://mystorageaccount.blob.core.windows.net"),
    new DefaultAzureCredential(),
    options);
```

Different Azure SDKs expose retry settings in slightly different ways, but the exam-level principle is consistent: configure the client for the service and workload, and prefer built-in retries before adding another retry layer.

> Avoid stacking multiple retry systems blindly. If the Azure SDK already retries and you also wrap the call in Polly, you can multiply delays and make outages worse.

**When to use SDK retries vs Polly**

Use the Azure SDK retry policy when:
- You are calling Azure services through their SDKs.
- The built-in policy already handles the transient errors you expect.
- You want the simplest and most maintainable solution.

Use **Polly** when:
- You need to wrap non-Azure dependencies.
- You need a broader resilience policy such as retry + circuit breaker + timeout.
- You want a consistent policy across multiple HTTP or custom service calls.

Example with Polly for an HTTP dependency:

```csharp
using Polly;
using Polly.Extensions.Http;

var retryPolicy = HttpPolicyExtensions
    .HandleTransientHttpError()
    .WaitAndRetryAsync(3, attempt => TimeSpan.FromSeconds(Math.Pow(2, attempt)));

builder.Services.AddHttpClient<IMyApiClient, MyApiClient>()
    .AddPolicyHandler(retryPolicy);
```

For Azure SDK clients, prefer the SDK’s own retry settings unless you have a specific cross-cutting resilience requirement.

**Poison-message handling**

A poison message is a message that repeatedly fails processing and should not keep cycling forever. The handling mechanism depends on the messaging service.

For **Azure Service Bus**, poison messages are typically moved to the **dead-letter queue** (DLQ). For **Azure Storage Queues**, poison messages are commonly moved to a separate poison queue, often named `<queue-name>-poison` in Azure Functions scenarios.

**Service Bus dead-lettering**

Service Bus tracks delivery attempts with the message delivery count. When processing fails repeatedly and the delivery count reaches the configured maximum delivery count threshold, the message is dead-lettered after the next failed delivery/abandon cycle. The exact boundary matters: do not think of it as “one more than max” in a vague way.

Service Bus also uses message locks:
- A message is locked while a consumer processes it.
- If processing takes too long, the lock can expire.
- If the lock expires, the message becomes available again and the delivery count can increase on the next delivery.
- You can renew the lock if processing legitimately takes longer than the lock duration.

This interaction matters because a slow handler can look like a failure even if the code is correct. If your processing is long-running, consider lock renewal or redesigning the workflow.

Example dead-letter inspection with the Service Bus client:

```csharp
await using var client = new ServiceBusClient(connectionString);
ServiceBusReceiver receiver = client.CreateReceiver("orders");

ServiceBusReceivedMessage message = await receiver.ReceiveMessageAsync();

if (message != null)
{
    try
    {
        await ProcessAsync(message);
        await receiver.CompleteMessageAsync(message);
    }
    catch (Exception ex)
    {
        await receiver.DeadLetterMessageAsync(
            message,
            deadLetterReason: "ProcessingFailed",
            deadLetterErrorDescription: ex.Message);
    }
}
```

For processor-based code, use the message actions available in the event args rather than assuming the processor itself exposes completion methods directly. That distinction is important in AZ-204 questions.

```csharp
using Azure.Messaging.ServiceBus;

public class OrderProcessor
{
    public async Task HandleAsync(ProcessMessageEventArgs args)
    {
        try
        {
            await ProcessAsync(args.Message);
            await args.CompleteMessageAsync(args.Message);
        }
        catch (Exception ex)
        {
            await args.DeadLetterMessageAsync(
                args.Message,
                "ProcessingFailed",
                ex.Message);
        }
    }

    private static Task ProcessAsync(ServiceBusReceivedMessage message)
    {
        return Task.CompletedTask;
    }
}
```

**Storage Queue poison handling**

Azure Storage Queues do not have a DLQ in the same way Service Bus does. In Azure Functions queue triggers, the runtime can move a repeatedly failing message to a poison queue named `<queue-name>-poison`. This is a Functions runtime behavior for Storage Queue triggers, not a general Azure Functions rule for all triggers.

For manual queue processing, you may need to implement your own poison handling:
- Track dequeue count.
- Move the message to a separate queue after repeated failures.
- Log the payload and failure reason.
- Alert operators or create a remediation workflow.

Example of manual poison handling logic:

```csharp
if (message.DequeueCount >= 5)
{
    await poisonQueueClient.SendMessageAsync(message.Body);
    await queueClient.DeleteMessageAsync(message.MessageId, message.PopReceipt);
    return;
}
```

**How retries and poison handling work together**

Retries and poison handling solve different problems:
- Retries handle transient failures.
- Poison handling isolates messages that are consistently bad.

A good workflow is:
1. Try the operation with a bounded retry policy.
2. If the operation still fails, let the message be retried by the queue or broker.
3. After the configured threshold, move the message to DLQ or poison queue.
4. Investigate, fix the root cause, and reprocess if appropriate.

This prevents infinite retry loops and protects downstream systems from repeated bad input.

**End-to-end AZ-204 scenario**

Imagine an order-processing app:
- An API receives an order and writes a message to Service Bus.
- A Function or worker consumes the message.
- DI provides the order service, blob client, and telemetry client.
- Configuration supplies the storage URI, queue name, and retry settings.
- The Azure SDK retries transient blob failures.
- If the payload is invalid, the message is dead-lettered.
- If the handler keeps failing because of a schema mismatch, the message eventually lands in the DLQ or poison queue depending on the transport.

A practical implementation often looks like this:

```csharp
public sealed class OrderHandler
{
    private readonly BlobServiceClient _blobServiceClient;
    private readonly ILogger<OrderHandler> _logger;
    private readonly IOptionsMonitor<StorageOptions> _options;

    public OrderHandler(
        BlobServiceClient blobServiceClient,
        ILogger<OrderHandler> logger,
        IOptionsMonitor<StorageOptions> options)
    {
        _blobServiceClient = blobServiceClient;
        _logger = logger;
        _options = options;
    }

    public async Task HandleAsync(OrderMessage message, CancellationToken cancellationToken)
    {
        var container = _blobServiceClient.GetBlobContainerClient(_options.CurrentValue.ContainerName);

        try
        {
            var blob = container.GetBlobClient($"{message.OrderId}.json");
            await blob.UploadAsync(BinaryData.FromString(message.RawJson), overwrite: true, cancellationToken);
        }
        catch (RequestFailedException ex) when (ex.Status == 429 || ex.Status >= 500)
        {
            _logger.LogWarning(ex, "Transient storage failure for order {OrderId}", message.OrderId);
            throw;
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Permanent failure for order {OrderId}", message.OrderId);
            throw;
        }
    }
}
```

This pattern shows the exam-relevant flow:
- DI injects dependencies.
- Configuration drives behavior.
- SDK retries handle transient storage issues.
- Exceptions distinguish transient from permanent failures.
- The messaging system eventually isolates poison messages.

**Troubleshooting and exam cues**

When a question asks what to do with a failing message, identify the transport first:
- Service Bus: use DLQ and message actions.
- Storage Queue trigger in Functions: poison queue behavior is runtime-managed.
- Manual queue processing: implement your own poison strategy.

When a question asks about configuration access, choose the right options type:
- `IOptions<T>` for stable settings.
- `IOptionsSnapshot<T>` for per-request scoped reads.
- `IOptionsMonitor<T>` for reloadable settings and change notifications.

When a question asks about retries, ask whether the dependency already has built-in retry support:
- If yes, start there.
- If no, use Polly or custom logic.
- Do not retry permanent errors.

When a question asks about DI lifetimes, match the lifetime to the service behavior:
- Singleton for reusable, thread-safe clients.
- Scoped for request-bound services.
- Transient for lightweight stateless services.

**Quick comparison**

| Area | Best practice | Common exam pitfall |
|---|---|---|
| DI | Register once, inject through constructors | Creating clients repeatedly inside methods |
| Configuration | Bind to typed options and validate early | Reading raw strings everywhere |
| Retries | Use SDK defaults first, then customize | Wrapping every call in multiple retry layers |
| Service Bus poison handling | Use DLQ and message actions | Assuming the processor exposes completion methods directly |
| Storage Queue poison handling | Use Functions poison queue or custom logic | Treating poison queues as a universal Azure pattern |

**Practical study takeaway**

For AZ-204, think in layers:
- DI gives you structure and testability.
- Configuration gives you environment-specific behavior.
- Retries protect you from transient faults.
- Poison handling protects the system from bad messages that should stop retrying.

If you can explain how these four pieces work together in a single Azure app, you are ready for the scenario-based questions.
