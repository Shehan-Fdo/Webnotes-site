## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Why this matters**

Hardcoded secrets create avoidable risk: they leak through source control, build logs, deployment artifacts, and configuration exports. In Azure, the preferred pattern is to store secrets in <abbr title="Azure Key Vault">Key Vault</abbr> and let the application or platform retrieve them at runtime using <abbr title="Managed Service Identity">managed identity</abbr>.

For AZ-204, you need to know two related but different approaches:

- **Key Vault references** are a platform feature for supported Azure services such as Azure App Service and Azure Functions.
- **Direct SDK access** is application code that authenticates to Key Vault and reads secrets itself.

Use Key Vault references when you want the platform to inject a secret into app settings or connection strings. Use SDK access when your code needs dynamic secret retrieval, custom caching, or access from services that do not support references.

**Core concepts**

A managed identity gives an Azure resource an identity in Microsoft Entra ID without storing credentials in code. The app uses that identity to authenticate to Key Vault.

- **System-assigned identity**
  - Tied to one Azure resource.
  - Created and deleted with the resource.
  - Good for simple, single-app scenarios.
- **User-assigned identity**
  - Created as a standalone Azure resource.
  - Can be shared across multiple apps.
  - Useful when you want lifecycle independence from the app.

Key Vault access can be granted in two common ways:

- **Azure RBAC**
  - Assign roles at the vault scope, resource group scope, or subscription scope.
  - Common role for secret read access: `Key Vault Secrets User`.
- **Access policies**
  - Legacy-style vault permissions.
  - You explicitly grant secret permissions to a principal.

> For exam scenarios, distinguish the identity used by the app from the principal that must be authorized in Key Vault. With a user-assigned identity, you grant access to that identity’s principal, not to the web app resource itself.

**When to use Key Vault references vs SDK access**

| Scenario | Best choice | Why |
|---|---|---|
| App Service app setting or connection string needs a secret | Key Vault reference | Platform resolves the secret for you |
| Function app needs a database password in configuration | Key Vault reference | Keeps secret out of app settings |
| App needs to choose secrets dynamically at runtime | SDK access | Code controls retrieval and caching |
| Service does not support Key Vault references | SDK access | No platform reference support |
| You need fine-grained retry, fallback, or custom secret rotation logic | SDK access | More control in code |

**How Key Vault references work**

A Key Vault reference is a special value placed in an app setting or connection string. The platform resolves the reference at runtime and exposes the secret value to the app as the setting value.

A typical reference looks like this:

```text
@Microsoft.KeyVault(SecretUri=https://myvault.vault.azure.net/secrets/DbPassword/)
```

You can also reference a specific version:

```text
@Microsoft.KeyVault(SecretUri=https://myvault.vault.azure.net/secrets/DbPassword/7f3c2d1a8b4e4f5a9c1d2e3f4a5b6c7d)
```

Use a versionless URI when you want the platform to pick up the current secret value. However, do not assume instant propagation after rotation. App Service and Functions cache resolved values, so updates may take time or require a refresh or restart.

**Supported platform behavior**

Key Vault references are a feature of specific Azure hosting services, not a general Azure-wide configuration mechanism.

- Commonly used with:
  - Azure App Service
  - Azure Functions
- Typically entered in:
  - App settings
  - Connection strings
- Not a replacement for:
  - Arbitrary JSON configuration files in your app
  - Direct secret retrieval in services that do not support the feature

If the platform cannot resolve the secret, the app setting may remain unresolved and the application can fail when it tries to use the value. Treat resolution failures as deployment or configuration issues, not as a secret-management fallback.

**Configuration flow for App Service and Functions**

1. Create the Key Vault secret.
2. Enable a managed identity on the app.
3. Grant the identity permission to read secrets from the vault.
4. Add a Key Vault reference in an app setting or connection string.
5. Restart or refresh the app if needed.
6. Verify the resolved value in the hosting service.

**Example: create a secret**

```bash
az keyvault secret set \
  --vault-name myvault \
  --name DbPassword \
  --value 'P@ssw0rd-Example!'
```

**Example: enable a system-assigned identity**

```bash
az webapp identity assign \
  --resource-group rg-app \
  --name mywebapp
```

Check the identity details:

```bash
az webapp identity show \
  --resource-group rg-app \
  --name mywebapp
```

**Example: enable a user-assigned identity**

```bash
az identity create \
  --resource-group rg-app \
  --name uami-app
```

Assign it to the app:

```bash
az webapp identity assign \
  --resource-group rg-app \
  --name mywebapp \
  --identities /subscriptions/<sub-id>/resourceGroups/rg-app/providers/Microsoft.ManagedIdentity/userAssignedIdentities/uami-app
```

> CLI syntax can vary slightly by Azure CLI version. The important exam concept is that the app is associated with the user-assigned identity resource ID, and that identity is the principal you authorize in Key Vault.

**Granting Key Vault access**

If the vault uses Azure RBAC, assign a role that allows secret read access.

```bash
az role assignment create \
  --assignee-object-id <principal-object-id> \
  --assignee-principal-type ServicePrincipal \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<sub-id>/resourceGroups/rg-sec/providers/Microsoft.KeyVault/vaults/myvault
```

If the vault uses access policies, grant secret permissions directly:

```bash
az keyvault set-policy \
  --name myvault \
  --object-id <principal-object-id> \
  --secret-permissions get list
```

Use the identity’s object ID or principal ID, not the app’s resource ID. For a system-assigned identity, retrieve the app’s managed identity principal. For a user-assigned identity, use the user-assigned identity resource’s principal/object ID.

**Adding the Key Vault reference**

In App Service, add the reference in an app setting value:

```text
DbPassword = @Microsoft.KeyVault(SecretUri=https://myvault.vault.azure.net/secrets/DbPassword/)
```

For a connection string, use the same pattern in the connection string value field.

A practical deployment example with Azure CLI:

```bash
az webapp config appsettings set \
  --resource-group rg-app \
  --name mywebapp \
  --settings DbPassword='@Microsoft.KeyVault(SecretUri=https://myvault.vault.azure.net/secrets/DbPassword/)'
```

**How the app authenticates**

The authentication path depends on the approach:

- **Key Vault reference**
  - The hosting platform uses the app’s managed identity to resolve the secret.
  - Your application code receives the resolved value as a normal environment setting.
  - Your code does not call Key Vault for that setting.
- **SDK access**
  - Your application code authenticates to Key Vault using `DefaultAzureCredential`, `ManagedIdentityCredential`, or another Azure Identity credential.
  - Your code calls the Key Vault data plane directly.

This distinction is important for AZ-204. A Key Vault reference reduces code complexity, but it only works where the platform supports it. SDK access is more flexible and is the right choice when the app itself must control secret retrieval.

**Example: direct SDK access in code**

```csharp
using Azure.Identity;
using Azure.Security.KeyVault.Secrets;

var vaultUri = new Uri("https://myvault.vault.azure.net/");
var client = new SecretClient(vaultUri, new DefaultAzureCredential());

KeyVaultSecret secret = await client.GetSecretAsync("DbPassword");
Console.WriteLine(secret.Value);
```

Use this pattern when the application needs to read secrets at runtime rather than through platform configuration.

**Rotation and refresh behavior**

Secret rotation is one of the main reasons to avoid hardcoded values.

- If you use a versioned URI, the app keeps using that exact version until you change the reference.
- If you use a versionless URI, the platform can resolve the current secret value.
- Resolution is not always immediate because the platform caches references.
- After rotation, you may need to:
  - restart the app
  - trigger a configuration refresh
  - wait for cache expiration

Do not describe versionless references as “instant latest version” behavior. The platform eventually picks up the new value, but timing depends on caching and refresh behavior.

**Troubleshooting checklist**

If the reference does not resolve, check the following:

- The managed identity is enabled on the app.
- The identity has `get` permission for the secret.
- The Key Vault firewall or networking rules allow access from the hosting service.
- The secret name and URI are correct.
- The reference is placed in a supported app setting or connection string field.
- The app was restarted after configuration changes if needed.

Useful validation commands:

```bash
az webapp config appsettings list \
  --resource-group rg-app \
  --name mywebapp
```

```bash
az keyvault secret show \
  --vault-name myvault \
  --name DbPassword
```

For App Service, you can also inspect the configuration in the portal and confirm that the setting shows a Key Vault reference rather than a plain text value. If the platform has resolved it successfully, the app receives the secret value at runtime, but the portal still shows the reference string in configuration.

**Common exam pitfalls**

- **Confusing platform references with code access**
  - Key Vault references are not the same as using the Key Vault SDK.
- **Using the wrong principal**
  - Grant access to the managed identity principal, especially for user-assigned identities.
- **Assuming immediate rotation**
  - Cached resolution can delay updates.
- **Treating Key Vault references as universal**
  - They are supported only in specific Azure services.
- **Storing secrets in JSON config**
  - That is still hardcoding unless the app reads the value from a secure source at runtime.

**Practical architecture pattern**

A common secure design for an App Service app looks like this:

- App Service hosts the web app.
- A managed identity is enabled on the app.
- Key Vault stores database passwords, API keys, or connection strings.
- App settings contain Key Vault references.
- The app reads the resolved values from environment configuration.
- For secrets that must be fetched dynamically, the app uses the Key Vault SDK with `DefaultAzureCredential`.

This pattern keeps secrets out of source code and deployment artifacts while preserving operational flexibility.

**Quick decision guide**

- Use **Key Vault references** when:
  - the secret belongs in app settings or connection strings
  - the service supports the feature
  - you want minimal code changes
- Use **managed identity + SDK** when:
  - your code needs direct secret access
  - you need custom caching or rotation logic
  - the service does not support references

**Key takeaways**

- Managed identity removes the need to store credentials for Key Vault access.
- Key Vault references are a platform feature for supported services like App Service and Functions.
- Azure RBAC or access policies can authorize the identity to read secrets.
- Versionless references help with rotation, but caching means updates are not always immediate.
- For AZ-204, know both the configuration flow and the difference between platform resolution and direct SDK access.
