## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Azure Key Vault secret management overview**

Azure Key Vault stores application secrets such as connection strings, API keys, and passwords in a managed service instead of in source code or configuration files. For AZ-204, you should know how to create, read, update, disable, delete, and recover secrets by using the Azure SDK for .NET, and how identity-based access controls protect those operations.

A secret in Key Vault has a **name**, a **value**, and metadata such as enabled state, expiration, and tags. The service preserves the original casing of the name, and you reference secrets consistently by name in code and configuration. When you retrieve a secret, name matching is not case-sensitive for lookup, but the stored casing remains part of the secret’s identity and presentation.

**Core SDK packages and types**

For .NET, the primary package for secrets is `Azure.Security.KeyVault.Secrets`. The most important client and model types are:

- **`SecretClient`**: the main client used to create, get, update, and delete secrets.
- **`KeyVaultSecret`**: represents a secret value plus metadata returned by the service.
- **`SecretProperties`**: exposes metadata without requiring the secret value.
- **`DeletedSecret`**: represents a soft-deleted secret that can be recovered or purged.

Typical package references:

```xml
<ItemGroup>
  <PackageReference Include="Azure.Identity" Version="*" />
  <PackageReference Include="Azure.Security.KeyVault.Secrets" Version="*" />
</ItemGroup>
```

`Azure.Identity` provides credential types used by the SDK clients. In most modern Azure applications, you combine `SecretClient` with `DefaultAzureCredential` or a more specific credential when you want to constrain authentication.

**Authentication patterns for local development and Azure**

`DefaultAzureCredential` is the common choice because it tries multiple authentication sources in a predictable order. In local development, it can use your Azure CLI sign-in, Visual Studio sign-in, Visual Studio Code sign-in, environment variables, or other supported developer credentials. In Azure, it can use managed identity when the app runs in a supported host.

A practical local setup often looks like this:

```bash
az login
az account set --subscription "<subscription-id>"
```

Then your app can authenticate without embedding credentials:

```csharp
using Azure.Identity;
using Azure.Security.KeyVault.Secrets;

var vaultUri = new Uri("https://myvault.vault.azure.net/");
var client = new SecretClient(vaultUri, new DefaultAzureCredential());

KeyVaultSecret secret = await client.GetSecretAsync("DbPassword");
Console.WriteLine(secret.Value);
```

For exam purposes, remember the distinction:

- **Local development** often uses Azure CLI sign-in, Visual Studio, or environment variables.
- **Azure-hosted apps** often use managed identity.
- **`DefaultAzureCredential`** is convenient because it adapts across environments.
- **`ManagedIdentityCredential`** is appropriate when you want to explicitly require managed identity and avoid other credential sources.

Environment variables commonly used by `DefaultAzureCredential` include:

- `AZURE_TENANT_ID`
- `AZURE_CLIENT_ID`
- `AZURE_CLIENT_SECRET`

These are useful for service principal authentication in development or CI/CD, but they are not the preferred long-term choice for production app-to-Azure authentication when managed identity is available.

**Creating and storing secrets**

Use `SetSecretAsync` to create a secret or add a new version of an existing secret name. This is a key exam point: setting a secret does not overwrite the previous version in place. Instead, Key Vault creates a new version under the same secret name.

```csharp
using Azure.Identity;
using Azure.Security.KeyVault.Secrets;

var client = new SecretClient(
    new Uri("https://myvault.vault.azure.net/"),
    new DefaultAzureCredential());

KeyVaultSecret secret = await client.SetSecretAsync(
    "DbPassword",
    "P@ssw0rd-2026!");

Console.WriteLine(secret.Name);
Console.WriteLine(secret.Value);
Console.WriteLine(secret.Properties.Version);
```

Important behaviors:

- **Same name, new version**: each `SetSecretAsync` call creates a new version.
- **Versionless access**: if you request a secret by name only, you get the latest enabled version.
- **Pinned version access**: if you request a specific version, you get that exact version until you change the URI or code.

This versioning model is central to secret rotation questions.

**Reading secrets and metadata**

Use `GetSecretAsync` to retrieve the secret value. Use `GetPropertiesOfSecretVersions` or `GetPropertiesOfSecrets` when you need metadata without exposing the secret value.

```csharp
KeyVaultSecret secret = await client.GetSecretAsync("DbPassword");
Console.WriteLine(secret.Value);
Console.WriteLine(secret.Properties.Enabled);
Console.WriteLine(secret.Properties.ExpiresOn);
```

`SecretProperties` is useful when you need to inspect lifecycle state:

- `Enabled` tells you whether the secret can be retrieved.
- `ExpiresOn` helps you detect expiring credentials.
- `NotBefore` can delay usability until a future time.
- `Tags` can store application-specific metadata.
- `UpdatedOn` and `CreatedOn` help with auditing and rotation tracking.

To inspect versions:

```csharp
await foreach (SecretProperties version in client.GetPropertiesOfSecretVersionsAsync("DbPassword"))
{
    Console.WriteLine($"{version.Name} {version.Version} Enabled={version.Enabled}");
}
```

To list secret names without values:

```csharp
await foreach (SecretProperties secret in client.GetPropertiesOfSecretsAsync())
{
    Console.WriteLine(secret.Name);
}
```

This separation between metadata and value is important for least-privilege design and for troubleshooting secret lifecycle issues.

**Updating, enabling, disabling, and lifecycle control**

You can update secret properties independently of the value. For example, you might disable a secret version during rotation or set an expiration date to enforce renewal.

```csharp
SecretProperties props = (await client.GetSecretAsync("DbPassword")).Properties;
props.Enabled = false;
props.ExpiresOn = DateTimeOffset.UtcNow.AddDays(30);

await client.UpdateSecretPropertiesAsync(props);
```

Lifecycle concepts to remember:

- **Disable** a secret version to prevent retrieval.
- **Enable** it again if you need to restore access.
- **Expire** a secret to mark it as no longer valid.
- **Rotate** by creating a new version and then updating the application to use the new version or a versionless reference.

If a secret is disabled, retrieval of that version fails. If you use a versionless lookup, Key Vault returns the latest enabled version. If all relevant versions are disabled, your application cannot retrieve a usable secret.

**Deleting, recovering, and purging secrets**

Key Vault uses soft delete in supported configurations. When you delete a secret, it is not immediately removed permanently. Instead, it becomes recoverable for a retention period unless purged.

```csharp
DeleteSecretOperation deleteOperation = await client.StartDeleteSecretAsync("DbPassword");
await deleteOperation.WaitForCompletionAsync();

DeletedSecret deleted = deleteOperation.Value;
Console.WriteLine(deleted.Name);
```

Recovery and purge patterns:

- **Recover** a deleted secret if you deleted it accidentally.
- **Purge** a deleted secret only when you need permanent removal and your permissions allow it.
- **Soft delete** helps protect against accidental loss and supports compliance workflows.

A common exam distinction is that deleting a secret version is not the same as disabling it. Disabling blocks use while preserving the version. Deleting moves it into the deleted state.

**Secret versioning and rotation strategy**

Secret rotation is usually implemented by creating a new version under the same secret name, then allowing the application to pick up the new version through a versionless reference or a refresh mechanism.

A practical rotation flow is:

1. Create a new secret version with `SetSecretAsync`.
2. Validate the new version in a staging or canary environment.
3. Update the application to use the versionless secret URI or refresh its cached configuration.
4. Disable or retire the old version after validation.
5. Delete or purge old versions only when policy allows it.

Example of versionless retrieval behavior:

```csharp
var secret = await client.GetSecretAsync("DbPassword");
Console.WriteLine(secret.Properties.Version);
```

If your code does not specify a version, it receives the latest enabled version. If you pin a version in code or configuration, rotation does not automatically change what the app uses.

> Be careful not to assume that `SetSecretAsync` overwrites the existing value. It creates a new version, which is why version-aware design matters for rotation and rollback.

**Azure App Service and Azure Functions integration**

A common production pattern is to avoid direct SDK retrieval in the app and instead use Key Vault references in App Service or Azure Functions configuration. In this model, the platform resolves the secret reference and exposes the resolved value to the app as an application setting value.

Typical setup steps:

1. Enable a managed identity for the App Service or Function App.
2. Grant that identity access to the Key Vault secret.
3. Add an app setting that uses a Key Vault reference.
4. Restart the app or wait for the platform refresh behavior.

Example app setting value:

```text
@Microsoft.KeyVault(SecretUri=https://myvault.vault.azure.net/secrets/DbPassword/)
```

If you want to reference a specific version, include the version segment:

```text
@Microsoft.KeyVault(SecretUri=https://myvault.vault.azure.net/secrets/DbPassword/7f3c2d1a9b4e4b1a8c2d3e4f5a6b7c8d)
```

This pattern is useful when:

- You want to centralize secret storage.
- You want the platform to handle resolution.
- You want to avoid shipping secret values in deployment artifacts.

For exam answers, distinguish this from SDK-based retrieval:

- **SDK retrieval**: your code calls Key Vault directly with `SecretClient`.
- **Key Vault reference**: the hosting platform resolves the secret and injects the value into configuration.

**Access control and authorization**

Key Vault supports authorization through Azure role-based access control (RBAC) or access policies, depending on vault configuration. For AZ-204, you should understand the principle of least privilege and grant only the permissions required for the app.

Common secret permissions include:

- `get`
- `list`
- `set`
- `delete`
- `recover`
- `purge`
- `backup`
- `restore`

A managed identity is often granted a role such as Key Vault Secrets User for read-only access or a more privileged role for write operations in controlled environments.

Example Azure CLI role assignment:

```bash
az role assignment create \
  --assignee <principal-id> \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<subscription-id>/resourceGroups/<rg-name>/providers/Microsoft.KeyVault/vaults/<vault-name>
```

If you are using a managed identity, the principal ID is the identity’s object ID in Microsoft Entra ID. In many CLI examples, `--assignee` is sufficient when you provide the principal ID. Use the least specific form that matches your environment and avoid overcomplicating the assignment unless the scenario requires it.

**Naming rules and practical conventions**

Secret names should be treated as stable identifiers in your application and configuration. Use a consistent naming convention so code, deployment templates, and operational runbooks all reference the same secret reliably.

Practical guidance:

- Use descriptive names such as `DbPassword`, `StorageConnectionString`, or `ApiKey`.
- Avoid embedding environment-specific details in the secret name unless you intentionally separate environments.
- Keep names consistent across code and infrastructure templates.
- Remember that version identifiers are managed by Key Vault and are not something you normally invent manually.

Although retrieval is not case-sensitive for lookup, you should still use a consistent casing convention in code and documentation to reduce confusion.

**Retry and transient fault handling**

Key Vault is a networked cloud service, so transient failures can occur. Your app should handle throttling, temporary network issues, and service interruptions with retries.

The Azure SDK clients support configurable retry behavior through `ClientOptions`.

```csharp
using Azure;
using Azure.Identity;
using Azure.Security.KeyVault.Secrets;

var options = new SecretClientOptions
{
    Retry =
    {
        Mode = RetryMode.Exponential,
        MaxRetries = 5,
        Delay = TimeSpan.FromSeconds(1),
        MaxDelay = TimeSpan.FromSeconds(8)
    }
};

var client = new SecretClient(
    new Uri("https://myvault.vault.azure.net/"),
    new DefaultAzureCredential(),
    options);
```

Why this matters:

- **Exponential backoff** reduces pressure on the service during transient failures.
- **Max retries** helps your app recover from short-lived outages.
- **Timeout-aware design** prevents startup hangs when a secret is unavailable.

For production systems, combine SDK retries with application-level resilience patterns where appropriate.

**End-to-end example**

The following example shows a simple pattern for storing and retrieving a secret in a .NET app using Azure SDKs:

```csharp
using Azure.Identity;
using Azure.Security.KeyVault.Secrets;

var vaultUri = new Uri("https://myvault.vault.azure.net/");
var client = new SecretClient(vaultUri, new DefaultAzureCredential());

// Store a new version
await client.SetSecretAsync("ApiKey", "super-secret-value");

// Retrieve the latest enabled version
KeyVaultSecret secret = await client.GetSecretAsync("ApiKey");
Console.WriteLine($"Secret version: {secret.Properties.Version}");
Console.WriteLine($"Enabled: {secret.Properties.Enabled}");
```

If you need metadata only:

```csharp
await foreach (SecretProperties item in client.GetPropertiesOfSecretsAsync())
{
    Console.WriteLine($"{item.Name} | Enabled={item.Enabled} | Expires={item.ExpiresOn}");
}
```

**Exam-focused takeaways**

- **`SecretClient`** is the main Azure SDK client for Key Vault secrets.
- **`SetSecretAsync`** creates a new version rather than overwriting the old one.
- **`GetSecretAsync`** returns the latest enabled version when no version is specified.
- **`SecretProperties`** lets you inspect enabled state, expiration, and version metadata.
- **`DefaultAzureCredential`** is the common authentication choice for local development and Azure-hosted apps.
- **Managed identity** is the preferred Azure-hosted authentication model in many production scenarios.
- **Key Vault references** in App Service and Functions are resolved by the platform, not by your SDK code.
- **Soft delete** allows recovery after deletion, while purge permanently removes the secret.
- **Retry policies** matter because Key Vault is a remote service and transient faults are expected.
