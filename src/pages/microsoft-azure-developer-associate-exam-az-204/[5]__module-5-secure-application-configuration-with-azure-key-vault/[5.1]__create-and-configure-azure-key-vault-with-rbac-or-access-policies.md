## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Azure Key Vault overview**

Azure Key Vault is a managed service for storing and controlling access to sensitive material such as **secrets**, **keys**, and **certificates**. In AZ-204 scenarios, you use it to keep connection strings, API keys, signing keys, and TLS certificates out of application code and configuration files.

Key Vault separates the stored data from the application that uses it:

- **Secrets** store opaque values such as passwords, connection strings, and tokens.
- **Keys** support cryptographic operations such as encrypt, decrypt, sign, and verify.
- **Certificates** manage X.509 certificate lifecycle and can also expose the associated secret and key material.

A common application pattern is to authenticate with Microsoft Entra ID and retrieve values at runtime instead of hardcoding them. This reduces secret sprawl and supports rotation without redeploying code.

**Creation prerequisites and design choices**

Before you create a vault, decide which tenant, region, and access model you need. A vault belongs to a single Microsoft Entra tenant, so the identities that access it must exist in that tenant or be represented there through guest or service principal objects.

Important creation-time considerations include:

- **Subscription and resource group** determine billing and lifecycle management.
- **Region** should usually match the app or nearby services to reduce latency.
- **Tenant** must match the identity boundary you plan to use.
- **Soft delete** is enabled by default in modern vaults and helps recover deleted items.
- **Purge protection** prevents permanent deletion until the retention period expires.
- **Network access** can be restricted with firewalls, private endpoints, or selected networks.

> If you enable network restrictions, your app may authenticate successfully but still fail to reach the vault from blocked networks. Authentication and network reachability are separate checks.

A practical creation checklist:

1. Choose the correct Microsoft Entra tenant.
2. Select the region closest to the workload.
3. Decide whether to use **Azure RBAC** or **vault access policies**.
4. Review soft delete and purge protection settings.
5. Configure network access for your deployment model.

**Permission models: Azure RBAC vs access policies**

Azure Key Vault supports two authorization models for data-plane access:

- **Azure role-based access control (RBAC)** for Key Vault data-plane permissions.
- **Access policies** stored on the vault for direct secret, key, and certificate permissions.

The vault uses the permission model selected in its configuration. Both models can exist in the environment, but only the selected model authorizes data-plane operations for that vault. Access policies may still be visible when RBAC is active, but they do not grant data-plane access in that mode.

A useful mental model:

- **Management plane**: create the vault, change settings, configure networking, and manage the resource itself.
- **Data plane**: read or write secrets, keys, and certificates inside the vault.

Azure RBAC can govern both planes depending on the role assigned. Access policies govern only the data plane.

| Aspect | Azure RBAC | Access policies |
|---|---|---|
| Where configured | Azure role assignments on the vault, resource group, subscription, or management group scope | Vault-specific access policy list |
| Scope style | Inherited Azure scope model | Vault only |
| Data-plane authorization | Yes, with Key Vault data roles | Yes |
| Management-plane authorization | Yes, with management roles | No |
| Best fit | Centralized governance, least-privilege at scale | Simple vault-specific permissions |

**When to choose each model**

Use **Azure RBAC** when you want centralized governance, consistent role assignment workflows, and easier auditing across many vaults. This is the preferred modern approach for many new deployments.

Use **access policies** when you need a straightforward vault-local permission list or when working with older deployments that already use policies. In exam scenarios, the key is to identify the active permission model on the vault.

**Azure RBAC for Key Vault**

With Azure RBAC, you assign roles to users, groups, service principals, or managed identities. For Key Vault, the important distinction is between:

- **Management-plane roles** such as Contributor or Key Vault Contributor, which let you manage the vault resource.
- **Data-plane roles** such as Key Vault Secrets User, Key Vault Secrets Officer, Key Vault Crypto User, or Key Vault Certificates Officer, which let you access objects inside the vault.

A management role does not automatically grant data access. For example, a user may be able to view the vault resource and change settings but still be unable to read a secret unless they also have a data-plane role.

Common Key Vault data roles include:

- **Key Vault Secrets User**: read secrets.
- **Key Vault Secrets Officer**: manage secrets.
- **Key Vault Crypto User**: perform cryptographic operations with keys.
- **Key Vault Crypto Officer**: manage keys and perform broader key operations.
- **Key Vault Certificates Officer**: manage certificates.
- **Key Vault Reader**: view vault metadata and properties, not secret values.

Typical RBAC assignment examples:

```bash
# Assign a data-plane role to a user by object ID
az role assignment create \
  --assignee-object-id <user-object-id> \
  --assignee-principal-type User \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.KeyVault/vaults/<vault-name>
```

```bash
# Assign a data-plane role to a managed identity's service principal object ID
az role assignment create \
  --assignee-object-id <managed-identity-service-principal-object-id> \
  --assignee-principal-type ServicePrincipal \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.KeyVault/vaults/<vault-name>
```

For managed identities, the object ID used in role assignment is the identity's service principal object ID in Microsoft Entra ID. That distinction matters in exam questions.

**Access policies for Key Vault**

Access policies grant explicit permissions directly on the vault. They are often easier to reason about for a single application, but they do not scale as cleanly as RBAC in larger environments.

Access policy permissions are grouped by object type:

- **Secrets**: `get`, `list`, `set`, `delete`, `recover`, `backup`, `restore`, `purge`
- **Keys**: `get`, `list`, `create`, `delete`, `recover`, `backup`, `restore`, `import`, `encrypt`, `decrypt`, `sign`, `verify`, `wrapKey`, `unwrapKey`, `release`, `rotate`, `getrotationpolicy`, `setrotationpolicy`
- **Certificates**: `get`, `list`, `import`, `create`, `update`, `delete`, `managecontacts`, `getissuers`, `listissuers`, `setissuers`, `deleteissuers`, `manageissuers`, `recover`, `purge`, `backup`, `restore`

The most common application permissions are usually:

- **Secrets**: `get` and sometimes `list`
- **Keys**: `get`, `wrapKey`, `unwrapKey`, `sign`, `verify`, or `decrypt` depending on the crypto workflow
- **Certificates**: `get` and `list` for retrieval, plus management permissions for lifecycle operations

A policy example in Azure CLI:

```bash
az keyvault set-policy \
  --name <vault-name> \
  --object-id <principal-object-id> \
  --secret-permissions get list set delete \
  --key-permissions get list encrypt decrypt sign verify wrapKey unwrapKey \
  --certificate-permissions get list import create update delete managecontacts getissuers listissuers
```

Use access policies when you need a direct, vault-local permission grant and the vault is configured to use the access policy model.

**Portal-based configuration workflow**

Portal questions are common in AZ-204, so know the basic steps for both models.

For **Azure RBAC**:

1. Open the Key Vault in the Azure portal.
2. Go to **Access control (IAM)**.
3. Select **Add role assignment**.
4. Choose a Key Vault data role such as **Key Vault Secrets User**.
5. Select the user, group, or managed identity.
6. Save the assignment and wait for propagation.

For **access policies**:

1. Open the Key Vault in the Azure portal.
2. Go to **Access policies**.
3. Select **Create** or **Add Access Policy**.
4. Choose the object or principal.
5. Select secret, key, and certificate permissions as needed.
6. Save the policy.

If the vault uses RBAC, adding an access policy does not authorize data-plane access. If the vault uses access policies, RBAC data roles do not authorize data-plane access unless the vault is switched to the RBAC permission model.

**Managed identities and app access**

A very common AZ-204 pattern is to give an Azure-hosted app a **managed identity** and then grant that identity access to Key Vault. This avoids storing credentials in the app.

Typical hosts include:

- Azure App Service
- Azure Functions
- Azure Container Apps
- Azure Virtual Machines
- Azure Kubernetes Service workloads using workload identity or managed identity integration

The flow is:

1. Enable a system-assigned or user-assigned managed identity on the app host.
2. Grant that identity a Key Vault data-plane role or access policy.
3. Use the Azure SDK with `DefaultAzureCredential` to authenticate.
4. Retrieve secrets or perform key operations at runtime.

Example .NET code using the Azure SDK:

```csharp
using Azure.Identity;
using Azure.Security.KeyVault.Secrets;

var vaultUri = new Uri("https://<vault-name>.vault.azure.net/");
var client = new SecretClient(vaultUri, new DefaultAzureCredential());

KeyVaultSecret secret = await client.GetSecretAsync("DbPassword");
Console.WriteLine(secret.Value);
```

`DefaultAzureCredential` tries multiple authentication sources in order, including managed identity in Azure. In local development, it can also use Visual Studio, Azure CLI, or environment-based credentials depending on what is available.

**Verification and testing**

After configuration, verify access from both the portal and code.

Portal checks:

- Confirm the vault permission model is set correctly.
- Review role assignments under **Access control (IAM)** if using RBAC.
- Review access policies if using the policy model.
- Check networking settings if requests fail despite correct permissions.

CLI checks:

```bash
# Show vault configuration
az keyvault show --name <vault-name> --query "{name:name, tenantId:properties.tenantId, enableRbacAuthorization:properties.enableRbacAuthorization, publicNetworkAccess:properties.publicNetworkAccess}"
```

```bash
# List secrets if you have permission
az keyvault secret list --vault-name <vault-name>
```

If access fails, distinguish the failure type:

- **401 Unauthorized** often indicates authentication or token issues.
- **403 Forbidden** often indicates missing Key Vault permission or network restriction.
- **404 Not Found** can appear when the caller lacks permission to see the object or the name is wrong.

**Soft delete, purge protection, and recovery**

Soft delete protects against accidental deletion by retaining deleted vault objects for a retention period. Purge protection adds an extra safeguard by preventing immediate permanent deletion of deleted items.

Why these matter:

- They reduce the risk of accidental data loss.
- They help meet compliance and recovery requirements.
- They affect delete and recreate workflows during testing.

Important behaviors:

- Deleted secrets, keys, and certificates can often be recovered during the retention window.
- Purge protection prevents permanent purge until the retention period ends.
- You may need elevated permissions to recover or purge items.

> In exam scenarios, remember that deleting a secret is not always the same as permanently removing it.

**Network access and private connectivity**

Key Vault can be exposed publicly or restricted to selected networks. Network controls are separate from authorization.

Common options include:

- **Public network access** enabled or disabled.
- **Firewall rules** for selected IP ranges or virtual networks.
- **Private endpoints** for private IP access from a virtual network.

Use private endpoints when you want traffic to stay on private network paths. Use firewall rules when you need simpler IP-based restrictions. If the app runs in Azure and uses a private endpoint, make sure DNS resolves the vault name to the private IP.

**Practical deployment example**

A common secure deployment for an app service looks like this:

- App Service has a system-assigned managed identity.
- Key Vault stores a database connection string as a secret.
- The vault uses Azure RBAC.
- The app identity gets the **Key Vault Secrets User** role on the vault.
- The app reads the secret with `DefaultAzureCredential` at startup.

This pattern avoids embedding credentials in app settings and supports secret rotation without code changes.

**Troubleshooting checklist**

If your app cannot access Key Vault, check these items in order:

1. Confirm the app identity exists and is enabled.
2. Confirm the vault permission model matches your authorization method.
3. Confirm the identity has the correct data-plane role or access policy.
4. Confirm the vault network settings allow the caller.
5. Confirm the secret, key, or certificate name is correct.
6. Confirm the app is using the right tenant and subscription context.
7. Wait for role assignment propagation if you just changed permissions.

**Exam-focused distinctions**

- **RBAC** is assigned through Azure role assignments and can be used for both management and data access depending on the role.
- **Access policies** are configured on the vault and apply only to data-plane operations.
- **Managed identities** are the preferred way for Azure-hosted apps to access Key Vault without secrets.
- **Soft delete and purge protection** are important security and recovery features.
- **Network restrictions** can block access even when permissions are correct.
