## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Secret rotation fundamentals**

Secret rotation is the process of replacing credentials before they expire, are leaked, or become risky to keep in use. In Azure solutions, you usually rotate application secrets, storage credentials, certificates, and signing keys on a schedule that matches the risk and operational impact of each credential.

The goal is not just to change a value. You also need a rollout strategy so applications keep working while old and new credentials overlap for a short period.

- **Application secrets** are values your app uses to authenticate to another service.
- **Service credentials** include storage account keys, SAS (Shared Access Signature) tokens, and Microsoft Entra ID app credentials.
- **Overlap** is important so you can deploy the new secret before disabling the old one.
- **Automation** reduces human error and makes rotation repeatable.

A practical rotation lifecycle often looks like this:

```text
Create new credential
        ↓
Store new value securely
        ↓
Update consuming app or configuration reference
        ↓
Validate authentication with new credential
        ↓
Revoke or expire old credential
        ↓
Monitor for failures
```

**What to rotate and when**

Different credential types have different rotation patterns. For AZ-204, you should distinguish between secrets used by applications and credentials used to manage Azure services.

| Credential type | Typical use | Rotation approach | Notes |
|---|---|---|---|
| Application secret | App-to-service authentication | Create new secret, update app, retire old secret | Prefer managed identity when possible |
| Storage account key | Administrative access to storage | Regenerate one key at a time | Keep two keys so one remains active |
| SAS token | Time-limited delegated access | Issue short-lived tokens | Best for limited scope and duration |
| Entra ID app credential | Client authentication for app registration | Add new credential, update clients, remove old one | Use certificates when stronger assurance is needed |
| Key Vault secret | Sensitive app configuration | Rotate in Key Vault and refresh consumers | Use references or reload logic |

For exam scenarios, choose the least risky credential that still meets the requirement. For example, if an app can use managed identity to access Key Vault or Storage, that is usually better than storing a client secret.

**Azure Key Vault rotation patterns**

Azure Key Vault is commonly used to store secrets, keys, and certificates. Rotation support is not identical for every object type, so keep the distinctions clear.

- **Secrets** can be rotated by updating the secret value and version.
- **Keys** can use rotation policy features for supported key types and scenarios.
- **Certificates** are managed through certificate policies and lifecycle actions, not the same rotation policy mechanism used for keys and secrets.

A common pattern is to store a new secret version in Key Vault, then let the application pick up the latest version or a refreshed reference.

```bash
az keyvault secret set \
  --vault-name kv-prod-001 \
  --name DbPassword \
  --value 'new-strong-password'
```

If you need to inspect versions:

```bash
az keyvault secret list-versions \
  --vault-name kv-prod-001 \
  --name DbPassword \
  -o table
```

> Do not assume that creating a new secret version automatically updates every app. The consuming application must either request the latest version, refresh its configuration, or restart/reload depending on how it reads secrets.

**How apps pick up rotated secrets**

In real applications, secret rotation only succeeds if the consumer refreshes the value. Common patterns include direct Key Vault reads, Key Vault references in app hosting services, or configuration providers that reload on change.

- **Direct retrieval** means the app calls Key Vault at runtime.
- **Configuration refresh** means the app periodically reloads settings from a source such as Azure App Configuration.
- **App restart** may be enough for simple workloads, but it is not ideal for zero-downtime rotation.

A common Azure App Configuration and Key Vault pattern is:

1. Store the secret in Key Vault.
2. Reference the secret from Azure App Configuration or the app’s configuration system.
3. Enable refresh behavior in the app.
4. Rotate the secret in Key Vault.
5. Trigger or wait for refresh so the app reads the new value.

Example .NET configuration pattern:

```csharp
var builder = WebApplication.CreateBuilder(args);

builder.Configuration.AddAzureAppConfiguration(options =>
{
    options.Connect(builder.Configuration["AppConfigConnection"])
           .ConfigureKeyVault(kv =>
           {
               kv.SetCredential(new DefaultAzureCredential());
           })
           .ConfigureRefresh(refresh =>
           {
               refresh.Register("Settings:Sentinel", refreshAll: true)
                      .SetCacheExpiration(TimeSpan.FromSeconds(30));
           });
});
```

This pattern is useful because you can rotate a Key Vault secret without redeploying the app, as long as the app refreshes configuration and the secret reference remains valid.

**Managed identity and secret reduction**

Managed identity is often the best answer when the question asks how to reduce secret management overhead. Instead of storing a client secret, the app authenticates with its Azure identity.

- **System-assigned managed identity** is tied to one Azure resource.
- **User-assigned managed identity** can be shared across resources.
- **Least privilege** still applies; identity alone does not grant access.

Use managed identity when the app runs in Azure and the target service supports Microsoft Entra ID authentication. This reduces the number of secrets you must rotate.

**Authorization model troubleshooting**

Authorization failures usually fall into two broad categories:

- **401 Unauthorized** means authentication failed or the token is missing/invalid.
- **403 Forbidden** means authentication succeeded, but the identity lacks permission.

That distinction is critical in AZ-204 troubleshooting questions.

| Symptom | Likely cause | What to check first |
|---|---|---|
| 401 | Bad token, wrong audience, expired credential, missing login | Token acquisition, tenant, scope, client secret, managed identity |
| 403 | Identity authenticated but lacks access | Role assignment, access policy, app role consent, scope permissions |
| Intermittent access | Propagation delay or stale token | Refresh token, wait for role propagation, reauthenticate |

**Key Vault authorization modes**

Azure Key Vault uses one data-plane authorization model at a time for access decisions:

- **Access policy model** grants permissions directly in the vault.
- **Azure RBAC** grants permissions through role assignments.

Do not treat them as additive for the same data-plane evaluation path. If the vault is configured for RBAC authorization, access policies are not used for data-plane access decisions.

How to diagnose:

- Check the vault’s authorization configuration.
- If using **access policies**, inspect the vault’s access policy entries.
- If using **RBAC**, inspect role assignments at the vault, resource group, or subscription scope.

Useful Azure Portal tools:

- **Access control (IAM)** for RBAC role assignments.
- **Access policies** for the legacy vault policy model.
- **Check access** to verify whether a principal has effective permissions.

Example CLI checks:

```bash
az keyvault show \
  --name kv-prod-001 \
  --query properties.enableRbacAuthorization
```

If the result is `true`, use RBAC troubleshooting. If `false`, use access policy troubleshooting.

List RBAC assignments:

```bash
az role assignment list \
  --assignee <principal-id-or-app-id> \
  --scope $(az keyvault show --name kv-prod-001 --query id -o tsv) \
  -o table
```

List access policies:

```bash
az keyvault show \
  --name kv-prod-001 \
  --query properties.accessPolicies
```

**App registration, service principal, and consent**

A common source of confusion is identity terminology.

- **Application (client) ID** identifies the app registration.
- **Object ID** identifies the directory object, such as the app registration or service principal.
- **Service principal** is the tenant-specific identity used at runtime for access control.
- **Tenant ID** identifies the Microsoft Entra tenant.

When a tool asks for `clientId`, it usually expects the application/client ID. When it asks for a principal object, it may want the service principal object ID. Always match the field to the API or portal label.

For permissions, distinguish between:

- **Delegated permissions** for user-based sign-in flows.
- **Application permissions** for daemon or service-to-service flows.
- **Admin consent** when tenant-wide approval is required.
- **App role assignment** when a service principal must be assigned a role in the target app.

Troubleshooting checklist:

1. Confirm the app has the correct permission type.
2. Confirm consent was granted if the permission requires it.
3. Confirm the service principal exists in the tenant.
4. Confirm the token contains the expected `roles` or `scp` claim.
5. Confirm the target API checks the same claim type you granted.

**Token audience and scope issues**

A valid token for one service is not automatically valid for another. The token audience must match the resource.

For Azure Storage with Microsoft Entra ID authentication:

- Resource URI is commonly `https://storage.azure.com/`
- Scope form used in modern token acquisition is `https://storage.azure.com/.default`

Example with Azure CLI:

```bash
az account get-access-token \
  --resource https://storage.azure.com/
```

Example with MSAL-style scope acquisition:

```csharp
var scopes = new[] { "https://storage.azure.com/.default" };
var token = await credential.GetTokenAsync(new TokenRequestContext(scopes));
```

If the audience is wrong, the service may reject the token even though authentication succeeded against Entra ID.

**DefaultAzureCredential and local development failures**

`DefaultAzureCredential` is convenient, but it can hide the exact credential source that failed. It tries multiple credential types in sequence, such as environment variables, managed identity, shared token cache, and developer tools.

Common failure causes:

- Missing environment variables for `EnvironmentCredential`.
- Logged-in developer account belongs to the wrong tenant.
- Managed identity is unavailable locally.
- The app uses a credential source that is not enabled in the current environment.

Example diagnostic pattern:

```csharp
var credential = new DefaultAzureCredential(new DefaultAzureCredentialOptions
{
    ExcludeInteractiveBrowserCredential = true
});

var token = await credential.GetTokenAsync(
    new TokenRequestContext(new[] { "https://vault.azure.net/.default" }));
```

If this fails, inspect which credential in the chain is failing and why. In production, prefer a narrower credential strategy when you know the hosting environment.

**End-to-end authorization failure diagnosis**

A strong AZ-204 answer walks from symptom to root cause to fix.

Scenario: your app gets `403 Forbidden` when reading a Key Vault secret.

1. Inspect the error message.
   - If it says forbidden or access denied, suspect authorization rather than authentication.
2. Check the vault authorization mode.
   - Use RBAC troubleshooting if RBAC is enabled.
   - Use access policy troubleshooting if the vault uses policies.
3. Identify the calling identity.
   - Confirm whether the app uses a managed identity, service principal, or user account.
4. Inspect the token.
   - Decode the JWT and check `aud`, `tid`, `appid`, `oid`, `roles`, or `scp`.
5. Verify permissions.
   - In RBAC mode, assign a role such as Key Vault Secrets User or a more specific role as needed.
   - In access policy mode, grant the required secret permissions.
6. Reauthenticate and retest.
   - Tokens may be cached, so sign out/in or restart the app if needed.

Example token inspection with a JWT decoder is often enough to spot the issue:

- Wrong `aud` means the token was minted for the wrong resource.
- Missing `roles` means the app permission was not granted or consented.
- Missing `scp` means delegated permissions are not present.
- Wrong `oid` means you assigned access to the wrong principal.

**Practical Azure CLI examples**

Check the current signed-in identity:

```bash
az account show -o jsonc
```

Get a token for Key Vault:

```bash
az account get-access-token \
  --resource https://vault.azure.net/
```

Assign a Key Vault RBAC role:

```bash
az role assignment create \
  --assignee <principal-id> \
  --role "Key Vault Secrets User" \
  --scope $(az keyvault show --name kv-prod-001 --query id -o tsv)
```

Grant a Key Vault access policy:

```bash
az keyvault set-policy \
  --name kv-prod-001 \
  --object-id <principal-object-id> \
  --secret-permissions get list
```

Rotate a storage account key:

```bash
az storage account keys renew \
  --account-name mystorageacct \
  --key primary
```

Create a short-lived SAS token for a blob container:

```bash
az storage container generate-sas \
  --account-name mystorageacct \
  --name uploads \
  --permissions rw \
  --expiry 2026-05-01T00:00Z \
  --auth-mode login
```

**Common exam traps**

- **RBAC vs access policy**: know which model the vault uses before assigning permissions.
- **401 vs 403**: authentication and authorization are not the same failure.
- **Client ID vs object ID**: use the identifier the tool expects.
- **Scope vs resource URI**: modern SDKs often use `.default` scopes.
- **Secret rotation vs certificate lifecycle**: these are not the same mechanism.
- **Managed identity vs service principal**: both can authenticate, but they are not interchangeable in troubleshooting.

**Quick comparison for review**

| Topic | Key idea | Typical fix |
|---|---|---|
| 401 | Token problem | Reauthenticate, fix scope, secret, tenant, or credential source |
| 403 | Permission problem | Add role assignment or access policy |
| RBAC mode | Vault uses Azure roles | Use Access control (IAM) |
| Access policy mode | Vault uses vault policies | Use Key Vault Access policies |
| App registration | Defines the app | Use application/client ID |
| Service principal | Runtime identity in tenant | Use object ID when required |

**Rotation and authorization best practices**

- Prefer **managed identity** over stored secrets when Azure services support it.
- Use **Key Vault** to centralize secret storage and auditing.
- Rotate credentials on a schedule and test the rollback path.
- Keep old and new credentials valid briefly during cutover.
- Use **least privilege** for both data-plane and control-plane access.
- Monitor sign-in and access logs for repeated failures after rotation.
- Document which identity, scope, and authorization model each app uses.

**Troubleshooting workflow summary**

1. Identify whether the failure is authentication or authorization.
2. Confirm the target resource and token audience.
3. Determine the identity used by the app.
4. Check whether the service uses RBAC or access policies.
5. Verify role assignment, access policy, or consent.
6. Refresh credentials and retest.
7. Rotate secrets with overlap and monitor for regressions.
