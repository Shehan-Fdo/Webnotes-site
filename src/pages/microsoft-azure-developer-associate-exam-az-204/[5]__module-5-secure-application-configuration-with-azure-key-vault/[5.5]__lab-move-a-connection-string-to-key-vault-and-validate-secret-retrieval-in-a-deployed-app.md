## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Lab goal and architecture**

This lab pattern moves a sensitive connection string out of application settings and into <abbr title="Azure Key Vault">Key Vault</abbr>, then validates that a deployed app can retrieve it at runtime. The main goal is to reduce secret exposure in source control, deployment logs, and app configuration while preserving app functionality.

A typical flow is:

- Create or identify a Key Vault.
- Store the connection string as a secret.
- Grant the app identity permission to read secrets.
- Update the app to resolve the secret from Key Vault.
- Deploy and verify the app still connects to its backend service.

In Azure, this is usually implemented with a **managed identity** so the app does not need to store credentials for Key Vault access.

**Why this pattern matters**

Connection strings often contain usernames, passwords, endpoints, and other sensitive values. Moving them to Key Vault improves security posture and supports rotation without redeploying code in many cases. It also aligns with the principle of least privilege: the app gets only the permissions it needs, and only for the secrets it must read.

This pattern is common for:

- Azure App Service applications
- Azure Functions
- Containerized workloads using managed identity or workload identity
- Any app that can authenticate to Azure and call the Key Vault data plane

**Core Azure services involved**

- **Azure Key Vault** stores secrets, keys, and certificates.
- **Managed identity** provides an Azure-managed identity for the app.
- **Azure App Service** or another hosted app runs the code.
- **Azure RBAC** or Key Vault access policies control secret access, depending on vault configuration.
- **Azure SDK for .NET, Java, JavaScript, or Python** retrieves the secret at runtime.

**Secret storage model**

Key Vault stores the connection string as a **secret**, not as a key or certificate. A secret is an opaque string value with metadata such as version, content type, and expiration.

Example secret creation with Azure CLI:

```bash
az keyvault secret set \
  --vault-name my-keyvault \
  --name DbConnectionString \
  --value "Server=tcp:myserver.database.windows.net,1433;Initial Catalog=mydb;Persist Security Info=False;User ID=myuser;Password=MyStrongPassword;MultipleActiveResultSets=False;Encrypt=True;TrustServerCertificate=False;Connection Timeout=30;"
```

Useful secret operations:

```bash
az keyvault secret show --vault-name my-keyvault --name DbConnectionString
az keyvault secret list --vault-name my-keyvault
az keyvault secret delete --vault-name my-keyvault --name DbConnectionString
```

**Identity and access control**

The app needs permission to read secrets. The recommended approach is to use a **system-assigned managed identity** for a single app or a **user-assigned managed identity** when multiple resources share the same identity.

Common permission models:

| Model | Where configured | Notes |
|---|---|---|
| Key Vault access policy | Vault settings | Traditional model; grants secret permissions directly |
| Azure RBAC | Azure role assignments | Preferred in many newer deployments; uses roles like Key Vault Secrets User |

For secret retrieval, the app typically needs:

- `get` permission on secrets
- Sometimes `list` permission if the app enumerates versions or names, though many apps do not need it

Example using Azure RBAC:

```bash
az role assignment create \
  --assignee-object-id <managed-identity-principal-id> \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.KeyVault/vaults/my-keyvault
```

Example using access policies:

```bash
az keyvault set-policy \
  --name my-keyvault \
  --object-id <managed-identity-principal-id> \
  --secret-permissions get list
```

> If the vault uses the Azure RBAC permission model, access policies may be disabled or ignored for data-plane authorization. Confirm the vault’s configuration before troubleshooting.

**App configuration patterns**

You can reference a Key Vault secret in app settings or retrieve it directly in code.

Common approaches:

- **Key Vault reference in App Service settings**
  - App setting value uses a special syntax that points to a secret URI.
  - Azure resolves the secret for the app at runtime.
- **Direct SDK retrieval**
  - The app uses `DefaultAzureCredential` or another credential to call Key Vault.
  - This gives you more control over caching, retries, and secret versioning.

Example App Service setting using a Key Vault reference:

```text
@Microsoft.KeyVault(SecretUri=https://my-keyvault.vault.azure.net/secrets/DbConnectionString/)
```

If you want a specific version, include it in the URI:

```text
@Microsoft.KeyVault(SecretUri=https://my-keyvault.vault.azure.net/secrets/DbConnectionString/7c3f2f4f6d5b4a1e9c2d3f4a5b6c7d8e)
```

This is useful when you need deterministic behavior during validation, but versionless references are often better for rotation because the app follows the latest enabled version.

**Retrieving the secret in code**

When you retrieve the secret directly, use the Azure Identity and Azure Key Vault Secret client libraries. The app authenticates with its managed identity in Azure and falls back to developer credentials locally.

Example in .NET:

```csharp
using Azure.Identity;
using Azure.Security.KeyVault.Secrets;

var keyVaultUrl = new Uri("https://my-keyvault.vault.azure.net/");
var client = new SecretClient(keyVaultUrl, new DefaultAzureCredential());

KeyVaultSecret secret = await client.GetSecretAsync("DbConnectionString");
string connectionString = secret.Value;
```

Example in JavaScript:

```javascript
import { DefaultAzureCredential } from "@azure/identity";
import { SecretClient } from "@azure/keyvault-secrets";

const vaultUrl = "https://my-keyvault.vault.azure.net/";
const credential = new DefaultAzureCredential();
const client = new SecretClient(vaultUrl, credential);

const secret = await client.getSecret("DbConnectionString");
const connectionString = secret.value;
```

Example in Python:

```python
from azure.identity import DefaultAzureCredential
from azure.keyvault.secrets import SecretClient

vault_url = "https://my-keyvault.vault.azure.net/"
credential = DefaultAzureCredential()
client = SecretClient(vault_url=vault_url, credential=credential)

secret = client.get_secret("DbConnectionString")
connection_string = secret.value
```

**Local development considerations**

Local development should not depend on a hard-coded developer password. Use a developer identity that can authenticate to Azure, such as:

- Azure CLI sign-in
- Visual Studio sign-in
- Azure PowerShell sign-in
- Environment-based service principal credentials for automation

`DefaultAzureCredential` is useful because it tries multiple credential sources in a predictable order. In local development, it often uses your signed-in developer identity; in Azure, it uses the managed identity.

Example local setup:

```bash
az login
az account set --subscription "<subscription-name-or-id>"
```

If the app uses Key Vault references in App Service, local execution may still need a fallback configuration source because the Key Vault reference resolution happens in Azure, not on your workstation.

**Deployment workflow**

A practical deployment sequence is:

1. Create the Key Vault.
2. Add the connection string as a secret.
3. Enable managed identity on the app.
4. Grant the identity secret read access.
5. Update the app configuration or code to use Key Vault.
6. Deploy the app.
7. Validate runtime retrieval.

Example App Service identity enablement:

```bash
az webapp identity assign \
  --resource-group my-rg \
  --name my-webapp
```

Example app setting update for a Key Vault reference:

```bash
az webapp config appsettings set \
  --resource-group my-rg \
  --name my-webapp \
  --settings "ConnectionStrings__Default=@Microsoft.KeyVault(SecretUri=https://my-keyvault.vault.azure.net/secrets/DbConnectionString/)"
```

**Validation techniques**

Validation should prove two things:

- The app can authenticate to Key Vault.
- The app can use the retrieved connection string successfully.

Useful checks include:

- App logs show successful secret retrieval.
- The app can connect to the database or downstream service.
- The secret value is not printed in plaintext logs.
- The app setting contains a Key Vault reference rather than the raw secret.

CLI checks for the app configuration:

```bash
az webapp config appsettings list \
  --resource-group my-rg \
  --name my-webapp
```

CLI checks for Key Vault secret metadata:

```bash
az keyvault secret show \
  --vault-name my-keyvault \
  --name DbConnectionString \
  --query "{name:name, enabled:attributes.enabled, expires:attributes.expires, updated:attributes.updated}" \
  -o table
```

If the app exposes a health endpoint, use it after deployment:

```bash
curl https://my-webapp.azurewebsites.net/health
```

**Troubleshooting common failures**

Most failures fall into a few categories:

- **Authorization failure**
  - The managed identity lacks `get` permission on secrets.
  - The wrong identity is assigned to the app.
  - The vault uses RBAC but the role assignment is missing.
- **Name or URI mismatch**
  - The secret name in the app setting does not match the Key Vault secret name.
  - The URI points to the wrong vault or secret version.
- **Network access restrictions**
  - Key Vault firewall blocks the app.
  - Private endpoint or virtual network integration is misconfigured.
- **Secret resolution timing**
  - The app starts before the identity or permission assignment is fully propagated.
  - A restart may be required after configuration changes.
- **Application parsing issues**
  - The app expects a raw connection string but receives a malformed reference string.
  - The code reads the wrong configuration key.

A quick diagnostic checklist:

1. Confirm the secret exists in Key Vault.
2. Confirm the app identity is enabled.
3. Confirm the identity has secret read permission.
4. Confirm the app setting or code points to the correct secret name.
5. Restart the app after permission or configuration changes.
6. Review logs for authentication or network errors.

**Security and operational best practices**

- Store only the minimum required secret data.
- Use versionless secret URIs when you want automatic rotation pickup.
- Use versioned URIs when you need controlled rollout or rollback.
- Rotate secrets regularly and test the app after rotation.
- Avoid logging connection strings or secret values.
- Prefer managed identity over client secrets for Azure-to-Azure authentication.
- Restrict Key Vault network access where possible.
- Use separate vaults or secret naming conventions for environments such as dev, test, and prod.

**Example topology**

A common production topology looks like this:

- User requests hit **Azure App Service**.
- The app uses **managed identity** to authenticate to **Azure Key Vault**.
- Key Vault returns the connection string secret.
- The app uses that connection string to connect to **Azure SQL Database** or another backend.

This keeps the secret out of code, out of deployment artifacts, and out of most configuration surfaces.

**Practical implementation pattern**

A simple and maintainable pattern is to centralize secret access in a small configuration layer. That layer can:

- Load the secret once at startup.
- Cache the value for a short period if appropriate.
- Fail fast if the secret cannot be retrieved.
- Emit only non-sensitive diagnostic messages.

Example startup logic in pseudocode:

```text
1. Create credential.
2. Create Key Vault client.
3. Read secret by name.
4. Store the value in secure in-memory configuration.
5. Initialize the database client.
6. Verify connectivity.
```

**What to remember for AZ-204**

- Key Vault stores secrets, including connection strings.
- Managed identity is the preferred way for Azure-hosted apps to access Key Vault.
- The app needs secret read permission, either through access policies or Azure RBAC.
- You can use Key Vault references in App Service or retrieve secrets directly with SDKs.
- Validation means proving both secret retrieval and downstream connectivity.
- Troubleshooting usually starts with identity, permissions, secret naming, and network access.
