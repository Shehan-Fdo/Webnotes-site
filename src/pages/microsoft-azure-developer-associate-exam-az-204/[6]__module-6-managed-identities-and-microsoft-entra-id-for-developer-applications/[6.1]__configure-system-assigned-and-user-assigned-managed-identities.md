## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Managed identities overview**

Managed identities let Azure resources authenticate to Azure services without storing credentials in code or configuration. Azure creates and manages the identity in Microsoft Entra ID, and your app requests tokens for Azure resources such as Key Vault, Storage, or Azure SQL.

Use managed identities when you want:
- No secrets in source control or app settings.
- Automatic credential rotation handled by Azure.
- A consistent identity model for apps running in Azure-hosted services.

The two identity types differ mainly in lifecycle and reuse:
- **System-assigned identity** is tied to one Azure resource and is deleted when that resource is deleted.
- **User-assigned identity** is a standalone Azure resource that you can attach to one or more supported Azure resources.

> Managed identity support is service-dependent. Not every Azure service supports both identity types, and the number of user-assigned identities you can attach varies by service.

**Identity types and when to use them**

A system-assigned identity works well when the app and identity should share the same lifecycle. A user-assigned identity works better when you need reuse across multiple resources, independent lifecycle management, or a stable identity that survives app redeployments.

| Aspect | System-assigned | User-assigned |
|---|---|---|
| Lifecycle | Follows the Azure resource | Independent Azure resource |
| Reuse | One resource only | Can be reused across supported resources |
| Deletion | Removed with the resource | Persists until you delete it |
| Best for | Simple, single-app scenarios | Shared access patterns, multi-app reuse |
| Identity stability | Changes with resource recreation | Stable across app changes |

**How token acquisition works**

Your code does not request a password or client secret. Instead, it requests an access token from the Azure platform identity endpoint, and Azure exchanges the managed identity for a token to the target resource.

Common flow:
1. Your app asks the platform for a token for a specific resource scope or resource URI.
2. Azure validates the managed identity assignment.
3. Azure returns an access token.
4. Your app uses the token to call the downstream Azure service.

For Azure SDKs, you usually use `DefaultAzureCredential` or a managed identity credential class. The SDK handles the token request details for you.

```csharp
using Azure.Identity;
using Azure.Security.KeyVault.Secrets;

var client = new SecretClient(
    new Uri("https://myvault.vault.azure.net/"),
    new DefaultAzureCredential());

KeyVaultSecret secret = await client.GetSecretAsync("DbPassword");
Console.WriteLine(secret.Value);
```

**System-assigned managed identity**

A system-assigned managed identity is enabled directly on the Azure resource. Azure creates the service principal for that resource in Microsoft Entra ID and manages its lifecycle.

Use it when:
- The app has a single hosting resource.
- You want the simplest setup.
- You do not need to share the identity with other resources.

Typical examples include:
- Azure App Service
- Azure Functions
- Azure Virtual Machines
- Azure Container Apps
- Some other Azure services that support managed identity

**Enable a system-assigned identity with Azure CLI**

```bash
az webapp identity assign \
  --resource-group rg-app \
  --name my-webapp
```

To inspect the assigned identity:
```bash
az webapp identity show \
  --resource-group rg-app \
  --name my-webapp
```

The output typically includes:
- `principalId`
- `tenantId`
- `type` such as `SystemAssigned`

**Bicep example for system-assigned identity**

```bicep
resource app 'Microsoft.Web/sites@2023-12-01' = {
  name: 'my-webapp'
  location: resourceGroup().location
  identity: {
    type: 'SystemAssigned'
  }
  properties: {
    serverFarmId: appServicePlan.id
  }
}
```

Use the exact API version that matches your deployment target. Resource provider schemas and property shapes can vary by service and version.

**User-assigned managed identity**

A user-assigned managed identity is created as its own Azure resource. You can attach it to supported Azure resources and reuse it across multiple apps or services.

Use it when:
- Multiple apps need the same Azure permissions.
- You want the identity to survive app recreation.
- You need a predictable identity for access control or operational consistency.

Common benefits:
- Reuse across environments or workloads.
- Easier blue-green or slot-based deployment patterns.
- Separation of identity lifecycle from compute lifecycle.

**Create a user-assigned identity with Azure CLI**

```bash
az identity create \
  --resource-group rg-identity \
  --name uai-app-shared
```

Get the identity details:
```bash
az identity show \
  --resource-group rg-identity \
  --name uai-app-shared
```

Important fields include:
- `clientId`
- `principalId`
- `id` for the Azure resource ID

**Attach a user-assigned identity**

Attachment is service-dependent. For example, some services let you attach one or more user-assigned identities, while others support only one or none.

Azure CLI example for a web app:
```bash
az webapp identity assign \
  --resource-group rg-app \
  --name my-webapp \
  --identities /subscriptions/<subId>/resourceGroups/rg-identity/providers/Microsoft.ManagedIdentity/userAssignedIdentities/uai-app-shared
```

If the service supports multiple user-assigned identities, you can attach more than one. If it supports only one, the platform rejects additional assignments.

**Selecting the correct identity at runtime**

When a resource has both system-assigned and user-assigned identities, your code or platform must choose which identity to use. The selection method depends on the credential and hosting environment.

Common selection patterns:
- Use the default identity when only one managed identity is available.
- Specify the user-assigned identity by `client_id` in many SDKs and platform endpoints.
- In some token endpoints, use the managed identity resource ID parameter or object ID parameter when the contract requires it.

For Azure SDKs, `ManagedIdentityCredential` can target a user-assigned identity by client ID:

```csharp
using Azure.Identity;

var credential = new ManagedIdentityCredential(
    clientId: "<user-assigned-client-id>");
```

For `DefaultAzureCredential`, you can set environment variables in local or containerized scenarios:
```bash
export AZURE_CLIENT_ID="<user-assigned-client-id>"
export AZURE_TENANT_ID="<tenant-id>"
```

**IMDS and token requests**

On Azure virtual machines and other compute services that expose the Instance Metadata Service (IMDS), managed identity tokens are commonly requested from the local endpoint.

Example for a system-assigned identity:
```bash
curl 'http://169.254.169.254/metadata/identity/oauth2/token?api-version=2018-02-01&resource=https%3A%2F%2Fvault.azure.net' \
  -H Metadata:true
```

Example for a user-assigned identity using `client_id`:
```bash
curl 'http://169.254.169.254/metadata/identity/oauth2/token?api-version=2018-02-01&resource=https%3A%2F%2Fvault.azure.net&client_id=<user-assigned-client-id>' \
  -H Metadata:true
```

Depending on the endpoint contract and service, user-assigned identity selection may use:
- `client_id`
- `mi_res_id`
- `object_id`

Use the parameter name required by the specific endpoint or SDK. Do not assume they are interchangeable.

**Role assignment and authorization**

Managed identity only solves authentication. You still need authorization through Azure role-based access control (RBAC) or a service-specific access policy.

Typical pattern for Key Vault with RBAC:
1. Enable the managed identity on the app.
2. Grant the identity a role such as `Key Vault Secrets User`.
3. Access the vault using the SDK and managed identity credential.

Azure CLI example:
```bash
az role assignment create \
  --assignee-object-id <principalId> \
  --assignee-principal-type ServicePrincipal \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<subId>/resourceGroups/rg-sec/providers/Microsoft.KeyVault/vaults/myvault
```

For older Key Vault configurations, access policies may still be used, but RBAC is the preferred model in many modern deployments.

**Practical configuration patterns**

A common architecture is:
- App Service or Function App hosts the code.
- The app uses a managed identity.
- The identity gets access to Key Vault, Storage, or SQL.
- Secrets are removed from app settings.

Example topology:
- Web app authenticates to Key Vault with managed identity.
- Web app reads a database connection string secret.
- Web app uses the secret only if the downstream service does not support direct Entra ID auth.

For Azure Storage, you can often avoid secrets entirely by using Entra ID-based authorization with the SDK:
```csharp
using Azure.Identity;
using Azure.Storage.Blobs;

var blobServiceClient = new BlobServiceClient(
    new Uri("https://mystorageaccount.blob.core.windows.net"),
    new DefaultAzureCredential());
```

**Local development considerations**

Managed identities are not available on your local machine in the same way they are in Azure-hosted resources. For local development, use developer credentials such as:
- Azure CLI sign-in
- Visual Studio sign-in
- Azure PowerShell sign-in
- Environment-based service principal credentials

`DefaultAzureCredential` is useful because it can work locally with developer credentials and in Azure with managed identity.

```csharp
var credential = new DefaultAzureCredential();
```

This reduces code changes between local testing and production deployment.

**Troubleshooting checklist**

If authentication fails, check the following:
- The identity is enabled on the target Azure resource.
- The identity has the correct RBAC role or access policy.
- You are requesting a token for the correct resource URI or scope.
- The app is using the intended identity when multiple identities are attached.
- The target service supports the identity type you configured.
- The endpoint or SDK parameter matches the identity selection method required by that service.

For Azure-hosted compute that uses IMDS, verify the platform endpoint is reachable from that environment. For other services, use the service-specific managed identity token path rather than assuming IMDS is always the direct interface.

Common mistakes:
- Granting permissions to the wrong principal.
- Using the wrong resource scope, such as a v2 scope when the service expects a resource URI.
- Forgetting that user-assigned identity attachment is not universal across all Azure services.
- Assuming a recreated system-assigned identity keeps the same principal ID.

**Exam-focused distinctions**

Know these differences for AZ-204:
- **System-assigned** identity is easiest when one app owns one identity.
- **User-assigned** identity is best when you need reuse or stable lifecycle.
- **Authentication** is not the same as authorization; RBAC still matters.
- **SDK support** is usually simpler than manual token handling.
- **Service support** varies, so always verify whether the target Azure service supports the identity type you want.

**Quick decision guide**

Use a system-assigned identity when:
- You want the simplest setup.
- The identity should disappear with the resource.
- Only one resource needs the identity.

Use a user-assigned identity when:
- Multiple resources need the same permissions.
- You want to preserve identity across redeployments.
- You need a stable identity for operational control.

**Minimal deployment example**

A typical deployment sequence looks like this:
1. Create the Azure resource.
2. Enable the managed identity.
3. Capture the identity principal ID.
4. Grant RBAC permissions to the target service.
5. Update the app to use `DefaultAzureCredential` or `ManagedIdentityCredential`.
6. Test access to the downstream resource.

**Key takeaways**

Managed identities remove secrets from your application and shift credential management to Azure. System-assigned identities are lifecycle-bound to one resource, while user-assigned identities are reusable and independently managed. The exact attachment and token-selection behavior depends on the Azure service, so always confirm the supported identity model and the correct endpoint or SDK parameters.
