## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Lab goal and migration pattern**

This lab replaces connection strings and shared keys with <abbr title="Microsoft Entra ID, formerly Azure Active Directory">Entra ID</abbr>-based access. The goal is to move from secret-based authentication to token-based authentication so your app uses an identity that Azure can authorize directly.

A typical migration looks like this:

- You identify each secret in app settings, configuration files, or Key Vault references.
- You enable a managed identity or use a developer identity for local testing.
- You grant the identity the correct Azure role or data-plane permission.
- You update the application code to request tokens and use SDK authentication.
- You validate the new path by calling the service and confirming the old connection string is no longer required.

> <mark>Important</mark>: In exam scenarios, the key idea is not just “remove secrets.” You must also configure authorization correctly at the Azure service and data plane.

**Why identity-based access is preferred**

Connection strings often embed account keys, SAS tokens, or passwords. Those values are powerful but hard to rotate safely and easy to leak. Identity-based access reduces secret sprawl and gives you centralized control through Entra ID, role assignments, and conditional access.

Benefits include:

- **No embedded secrets** in code or configuration for supported services.
- **Centralized access control** through Azure role assignments.
- **Easier rotation** because identities are managed by Azure.
- **Better auditability** through sign-in and role assignment logs.
- **Least privilege** access when you scope permissions carefully.

Identity-based access is especially common for Azure services that support Entra ID authentication directly, such as Azure Blob Storage, Azure Queue Storage, Azure File Storage, Azure Service Bus, Azure Key Vault, Azure SQL Database, and Azure Cosmos DB with the appropriate SDK and configuration model.

**End-to-end lab flow**

A practical lab usually follows this sequence:

1. You inspect the current app and locate connection strings.
2. You enable a managed identity on the app service, function app, container app, or VM.
3. You assign the identity the required Azure role.
4. You change the code to use `DefaultAzureCredential` or another Azure Identity credential.
5. You deploy and test the app.
6. You remove or disable the old secret-based path after validation.

A simple flow view helps:

```text
Before:
App -> connection string / key -> Azure service

After:
App -> managed identity / developer identity -> Entra token -> Azure service
```

**Managed identity basics**

A <abbr title="Managed Service Identity">managed identity</abbr> is an identity managed by Azure for an Azure resource. You do not store credentials for it. Azure handles the lifecycle and token issuance.

There are two common types:

- **System-assigned managed identity**
  - Tied to one Azure resource.
  - Deleted when the resource is deleted.
  - Best when the identity should not outlive the app.
- **User-assigned managed identity**
  - Created as a standalone Azure resource.
  - Can be attached to multiple resources.
  - Best when you want reuse across apps or environments.

Use managed identities when the app runs in Azure. Use a developer credential locally, such as Azure CLI sign-in, Visual Studio sign-in, or environment-based service principal auth.

**Authentication path and token flow**

Token-based calls follow a predictable pattern:

1. Your code asks Azure Identity for a token.
2. The credential chain selects the best available identity.
3. Entra ID issues an access token for the target resource.
4. The SDK sends the token in the `Authorization: Bearer <token>` header.
5. The Azure service validates the token and checks authorization.

This is why the app needs both:

- **Authentication**: proving who the app is.
- **Authorization**: proving what the app can do.

**Using `DefaultAzureCredential`**

`DefaultAzureCredential` is the most common choice for AZ-204 labs because it works in local development and in Azure without code changes.

It tries multiple credential sources in order, including:

- Environment variables
- Managed identity
- Azure CLI sign-in
- Visual Studio / Visual Studio Code sign-in
- Other developer credentials, depending on the runtime and package version

Example in .NET:

```csharp
using Azure.Identity;
using Azure.Storage.Blobs;

var credential = new DefaultAzureCredential();
var blobServiceClient = new BlobServiceClient(
    new Uri("https://mystorageaccount.blob.core.windows.net"),
    credential);
```

Example in JavaScript:

```javascript
const { DefaultAzureCredential } = require("@azure/identity");
const { BlobServiceClient } = require("@azure/storage-blob");

const credential = new DefaultAzureCredential();
const client = new BlobServiceClient(
  "https://mystorageaccount.blob.core.windows.net",
  credential
);
```

Example in Python:

```python
from azure.identity import DefaultAzureCredential
from azure.storage.blob import BlobServiceClient

credential = DefaultAzureCredential()
client = BlobServiceClient(
    account_url="https://mystorageaccount.blob.core.windows.net",
    credential=credential
)
```

Use `DefaultAzureCredential` when you want one code path for local and cloud environments. If you need strict control, you can use a specific credential such as `ManagedIdentityCredential` in Azure-only workloads.

**Service-specific access patterns**

**Azure Blob Storage**

Blob Storage is one of the most common services for identity-based access. You typically grant the app a data-plane role such as `Storage Blob Data Reader`, `Storage Blob Data Contributor`, or `Storage Blob Data Owner`.

Example role assignment:

```bash
az role assignment create \
  --assignee <principal-object-id> \
  --role "Storage Blob Data Contributor" \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<account-name>
```

Example validation with Azure CLI:

```bash
az storage blob list \
  --account-name <account-name> \
  --container-name <container-name> \
  --auth-mode login
```

For CLI validation, make sure:

- You are signed in with the expected identity using `az login`.
- The identity has the correct data-plane role.
- The storage account and operation support Entra ID authentication for the scenario you are testing.

Blob access is a strong fit for token-based auth because the SDKs and CLI support it well.

**Azure Queue Storage**

Queue Storage also supports Entra ID-based access for common scenarios. Use queue data roles such as `Storage Queue Data Contributor` or `Storage Queue Data Reader` depending on the operation.

Example .NET client:

```csharp
using Azure.Identity;
using Azure.Storage.Queues;

var queueClient = new QueueClient(
    new Uri("https://mystorageaccount.queue.core.windows.net/myqueue"),
    new DefaultAzureCredential());
```

Use queue access when your app needs to send, receive, or peek messages without storing account keys.

**Azure File Storage**

Azure Files supports identity-based access in many common scenarios, especially for SMB-based access and certain SDK-driven operations. In AZ-204 study material, focus on the fact that support exists, but validate the exact scenario and client library you are using.

Use it when you want to reduce reliance on storage account keys for file access where supported.

**Azure Table Storage**

Table Storage support is more limited and should be treated carefully in exam prep. Do not assume it behaves exactly like Blob or Queue access in every SDK or scenario. If a question mentions Table Storage, verify whether the scenario is about shared key access, SAS, or a specific Entra ID-capable path.

> <mark>Important</mark>: For AZ-204, Blob and Queue are the most straightforward identity-based storage scenarios. File is also relevant. Table is less commonly treated as a parity example.

**Azure Service Bus**

Service Bus supports Entra ID authentication through Azure roles such as `Azure Service Bus Data Sender`, `Azure Service Bus Data Receiver`, and related roles. In practice, namespace-level authorization is the most common and easiest to reason about in labs.

Example sender client in .NET:

```csharp
using Azure.Identity;
using Azure.Messaging.ServiceBus;

var client = new ServiceBusClient(
    "<namespace>.servicebus.windows.net",
    new DefaultAzureCredential());

var sender = client.CreateSender("orders");
await sender.SendMessageAsync(new ServiceBusMessage("hello"));
```

Use Service Bus when you need token-based messaging without connection strings. If a lab mentions entity-level access, confirm the SDK and scenario because behavior can vary by API and service configuration.

**Azure Key Vault**

Key Vault is a classic identity-based service. You grant the app a role such as `Key Vault Secrets User` or use access policies in older configurations, depending on the vault model.

Example secret retrieval:

```csharp
using Azure.Identity;
using Azure.Security.KeyVault.Secrets;

var client = new SecretClient(
    new Uri("https://myvault.vault.azure.net/"),
    new DefaultAzureCredential());

KeyVaultSecret secret = await client.GetSecretAsync("DbPassword");
```

Even when you use Key Vault, the broader goal of this lab is to reduce direct connection strings in app settings. In some architectures, Key Vault becomes the temporary bridge while you migrate to fully identity-based service access.

**Azure SQL Database and Azure SQL Managed Instance**

Azure SQL uses Entra ID for authentication, but the setup is more specific than storage or messaging services. You need:

- A server-level Entra admin configured on the logical server or managed instance.
- A database user created for the app identity in the target database.
- Appropriate database permissions granted to that user.

The Entra admin alone is not enough. The app identity must exist as a contained database user in the database where it connects.

Example pattern:

```sql
CREATE USER [my-app-managed-identity] FROM EXTERNAL PROVIDER;
ALTER ROLE db_datareader ADD MEMBER [my-app-managed-identity];
ALTER ROLE db_datawriter ADD MEMBER [my-app-managed-identity];
```

Use this when the app needs SQL access without SQL authentication passwords. In labs, the common failure mode is forgetting the database user creation step after assigning the server-level admin.

**Practical migration example**

Suppose your app currently reads a blob connection string from configuration:

```json
{
  "StorageConnectionString": "DefaultEndpointsProtocol=https;AccountName=...;AccountKey=..."
}
```

You migrate it to identity-based access:

1. Remove the connection string from app settings.
2. Enable a managed identity on the app.
3. Assign `Storage Blob Data Contributor` to that identity on the storage account or container scope.
4. Update code to use `BlobServiceClient` with `DefaultAzureCredential`.
5. Deploy and test.

Updated configuration might only keep the account endpoint:

```json
{
  "StorageAccountUrl": "https://mystorageaccount.blob.core.windows.net"
}
```

Updated code:

```csharp
var accountUrl = configuration["StorageAccountUrl"];
var client = new BlobServiceClient(new Uri(accountUrl), new DefaultAzureCredential());
```

This pattern is cleaner because the app no longer needs a secret to authenticate.

**Role assignment and scope**

Authorization depends on both the role and the scope. Scope determines where the permission applies.

Common scopes include:

- **Subscription**
- **Resource group**
- **Resource**
- **Child resource**, such as a storage container or Service Bus namespace in supported scenarios

Use the narrowest scope that still allows the app to function. For exam questions, prefer resource-level or narrower scope when the app only needs one service.

Example role assignment for a user-assigned managed identity:

```bash
az role assignment create \
  --assignee-object-id <managed-identity-object-id> \
  --assignee-principal-type ServicePrincipal \
  --role "Storage Blob Data Reader" \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<account-name>
```

If you are working in a lab, verify the identity object ID carefully. Managed identities can take time to propagate after creation, and role assignment may not be immediately effective.

**Local development validation**

Local validation should mirror the cloud path as closely as possible. The easiest approach is to sign in with Azure CLI and let `DefaultAzureCredential` pick up that identity.

Steps:

1. Sign in with the expected developer account:
   ```bash
   az login
   ```
2. Confirm the active subscription:
   ```bash
   az account show
   ```
3. Run the app locally.
4. Trigger the service call.
5. Confirm the request succeeds without a connection string.

If the app fails locally, check:

- The developer account has the correct role assignment.
- The app is using the expected tenant and subscription.
- The SDK package supports Entra ID for the target service.
- The endpoint and resource URI are correct.

**Azure CLI validation examples**

You can validate access outside the app to isolate whether the issue is code or authorization.

Blob example:

```bash
az storage blob list \
  --account-name mystorageaccount \
  --container-name mycontainer \
  --auth-mode login
```

Service Bus example is usually done through SDK or portal-based checks rather than a simple universal CLI list command, so use the service-specific tooling available in your environment.

SQL validation often uses a client tool such as `sqlcmd` or an application connection string configured for Entra ID authentication.

Example SQL connection test with `sqlcmd` using Entra authentication:

```bash
sqlcmd -S tcp:<server>.database.windows.net,1433 \
  -d <database> \
  -G \
  -U <user-principal-name>
```

The exact flags vary by `sqlcmd` version and authentication mode, so confirm your local toolchain.

**Troubleshooting checklist**

When token-based calls fail, work through the problem in layers:

- **Identity**
  - Is the managed identity enabled?
  - Is the developer signed in to the correct tenant?
- **Authorization**
  - Does the identity have the right role?
  - Is the scope correct?
  - For SQL, is the database user created?
- **Code**
  - Is the correct credential class used?
  - Is the resource URI correct?
  - Are you using the right SDK package version?
- **Service configuration**
  - Is Entra ID authentication supported for the scenario?
  - Is the endpoint correct?
  - Are firewall or network rules blocking access?
- **Propagation**
  - Did you wait for role assignment changes to take effect?

Common symptoms and likely causes:

- `403 Forbidden`
  - The identity authenticated successfully but lacks permission.
- `401 Unauthorized`
  - The token is missing, invalid, expired, or for the wrong audience.
- `Login failed for user '<token-identified principal>'`
  - Often SQL permissions or contained user setup is incomplete.
- `AuthenticationFailed` in SDK logs
  - The credential chain did not find a usable identity or the resource URI is wrong.

**Logging and verification**

Enable SDK logging when you need to see which credential the app selected. This is especially useful when `DefaultAzureCredential` behaves differently locally and in Azure.

Example .NET logging setup:

```csharp
using Azure.Core.Diagnostics;
using System.Diagnostics.Tracing;

using AzureEventSourceListener listener =
    AzureEventSourceListener.CreateConsoleLogger(EventLevel.Informational);
```

For deeper troubleshooting, inspect:

- Application logs
- Azure Activity Log for role assignments
- Entra sign-in logs
- Service-specific diagnostic logs

These logs help you distinguish authentication failures from authorization failures.

**Exam-focused distinctions**

For AZ-204, keep these distinctions clear:

- **Authentication vs authorization**
  - Authentication proves identity.
  - Authorization grants access.
- **Managed identity vs service principal**
  - Managed identity is Azure-managed and tied to a resource or user-assigned object.
  - Service principal is an application identity you manage more directly.
- **Control plane vs data plane**
  - Control plane manages Azure resources.
  - Data plane accesses the data inside the resource.
- **Connection string vs token**
  - Connection strings often contain secrets.
  - Tokens are short-lived and issued by Entra ID.

**Recommended lab mindset**

Treat the lab as a before-and-after verification exercise. Before the change, the app depends on a secret. After the change, the app depends on identity and role assignment.

A good validation outcome looks like this:

- The app starts without a storage key, SQL password, or Service Bus connection string.
- The app successfully acquires a token.
- The Azure service accepts the request.
- The old secret can be removed or rotated without breaking the app.

That outcome proves the migration is complete and the app is using the intended security model.
