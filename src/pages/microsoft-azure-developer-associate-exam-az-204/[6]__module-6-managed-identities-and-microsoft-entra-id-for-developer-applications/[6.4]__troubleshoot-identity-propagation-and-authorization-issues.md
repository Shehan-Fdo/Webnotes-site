## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Identity propagation and authorization: what you are troubleshooting**

In Azure, authentication proves who you are, while authorization determines what you can do. Many access issues come from a mismatch between the identity you expect, the token actually presented, and the permissions that have propagated to the target resource.

Identity propagation problems often appear after you:
- assign a role to a user, group, or managed identity
- enable a managed identity on a resource
- add a service principal to an app registration or enterprise application
- change app roles, scopes, or claims configuration
- move between tenants, subscriptions, or resource groups

A useful mental model is:
- **Identity**: the principal making the request
- **Token**: the signed proof of identity and claims
- **Authorization**: the permission check performed by Azure or the application
- **Propagation**: the delay before a new identity or permission becomes visible everywhere

**Common failure patterns**

Authorization failures usually fall into a few categories:
- `401 Unauthorized` means the request is not authenticated correctly or the token is missing, expired, invalid, or for the wrong audience.
- `403 Forbidden` means the identity is authenticated but lacks permission.
- `404 Not Found` can sometimes hide authorization issues when a service intentionally obscures resources.
- `AADSTS` errors usually indicate Microsoft Entra ID token acquisition or consent problems.

When you troubleshoot, always identify:
- which identity is being used
- which resource is being called
- which token audience or scope is requested
- which role, scope, or policy should grant access
- whether the change has had time to propagate

**How Azure identity flows work**

Azure applications commonly use Microsoft Entra ID for authentication and authorization. The application obtains an access token from the identity provider, then presents that token to Azure services or custom APIs.

Typical flows include:
- **User-delegated flow**: a signed-in user grants the app access to act on their behalf
- **App-only flow**: a daemon or background service uses a service principal or managed identity
- **Managed identity flow**: Azure hosts the identity for the resource, removing the need to store secrets

The token contains claims such as:
- `aud` for audience
- `iss` for issuer
- `tid` for tenant
- `oid` for object ID
- `scp` for delegated scopes
- `roles` for app roles or application permissions
- `exp` for expiration

If the token is valid but the service still denies access, the issue is usually in authorization rather than authentication.

**Propagation delays and why they happen**

Azure is distributed, so changes do not always appear instantly. Role assignments, group membership, app role assignments, and managed identity enablement can take time to propagate across control planes and data planes.

You may see delays when:
- assigning an Azure RBAC role such as `Reader` or `Contributor`
- granting access to Key Vault, Storage, or Cosmos DB using data-plane permissions
- adding a user to a group that is used in access control
- enabling a system-assigned managed identity and immediately using it
- updating app registration permissions and trying to acquire a token right away

Practical implications:
- A portal change may be visible before the service honors it.
- A token may not include new group membership until the user signs out and signs in again.
- A managed identity may need a short wait before the resource provider recognizes it.
- Some services cache authorization decisions for a short period.

> If you just changed permissions and the app still fails, do not assume the configuration is wrong immediately. First verify token freshness, identity selection, and propagation time.

**Azure RBAC versus data-plane authorization**

Azure uses different authorization systems depending on the resource.

| Area | Example | What it controls |
|---|---|---|
| Management plane | Subscription, resource group, resource creation | ARM operations such as create, update, delete |
| Data plane | Blob data, Key Vault secrets, Cosmos DB items | Access to the resource’s actual data |

Azure Role-Based Access Control (RBAC) governs management-plane access and, for some services, data-plane access as well. Other services use their own access models.

Examples:
- Storage accounts can use Azure RBAC for blob data access, but the token must target the correct resource and the role must be a data role such as `Storage Blob Data Reader`.
- Key Vault can use Azure RBAC or access policies depending on configuration.
- App Service authentication settings do not replace backend API authorization.

When troubleshooting, confirm:
- whether the service uses Azure RBAC, access policies, or application-specific roles
- whether the role is assigned at the correct scope
- whether the principal is a user, group, service principal, or managed identity

**Role assignment scope and inheritance**

Azure RBAC permissions flow from broader scopes to narrower scopes:
- management group
- subscription
- resource group
- resource

A role assigned at a higher scope applies to child resources unless denied by a more specific control or service-specific rule.

Useful checks:
- Assign at the narrowest scope that still meets the requirement.
- Verify the scope string exactly matches the target resource.
- Confirm the principal object ID is correct, not just the display name.

Example Azure CLI commands:

```bash
# Show the signed-in account and tenant
az account show

# List role assignments for a principal
az role assignment list --assignee <object-id> --all -o table

# Assign a role at resource group scope
az role assignment create \
  --assignee <object-id> \
  --role "Reader" \
  --scope /subscriptions/<subscription-id>/resourceGroups/<rg-name>
```

**Managed identity troubleshooting**

Managed identities are common in AZ-204 scenarios because they remove secret management. They still require correct assignment and propagation.

There are two types:
- **System-assigned managed identity**: tied to one Azure resource and deleted with it
- **User-assigned managed identity**: created independently and attached to one or more resources

Common issues:
- The identity is enabled but not yet available in Entra ID.
- The target resource has not been granted access.
- The app is requesting a token for the wrong resource.
- The code is using the wrong client ID for a user-assigned identity.
- The runtime environment does not support the identity endpoint you expect.

Example with Azure CLI:

```bash
# Enable a system-assigned managed identity on a web app
az webapp identity assign \
  --name <app-name> \
  --resource-group <rg-name>

# Get the principal ID
az webapp identity show \
  --name <app-name> \
  --resource-group <rg-name> \
  --query principalId -o tsv

# Grant access to a storage account container via RBAC
az role assignment create \
  --assignee <principal-id> \
  --role "Storage Blob Data Reader" \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<storage-name>
```

In code, use the Azure Identity library to request tokens from the managed identity endpoint:

```csharp
using Azure.Identity;
using Azure.Security.KeyVault.Secrets;

var client = new SecretClient(
    new Uri("https://<vault-name>.vault.azure.net/"),
    new DefaultAzureCredential());

KeyVaultSecret secret = await client.GetSecretAsync("MySecret");
Console.WriteLine(secret.Value);
```

If this fails after a recent identity change, retry after a short delay and verify the identity has the correct role at the correct scope.

**Token acquisition and audience mismatches**

A very common cause of `401` errors is requesting a token for the wrong audience. The token must be intended for the service you call.

Examples:
- Storage expects a token for the storage resource.
- Key Vault expects a token for the vault endpoint.
- A custom API expects the API’s application ID URI or scope.
- Microsoft Graph expects a Graph audience.

Symptoms of audience mismatch:
- token is valid but rejected by the service
- `invalid_token`
- `audience is invalid`
- `The access token is from the wrong issuer` in multi-tenant or cross-tenant scenarios

Practical checks:
- Decode the JWT and inspect `aud`, `iss`, `scp`, and `roles`.
- Confirm the app registration exposes the correct scopes.
- Confirm the client requests the correct scope string.

Example using `az account get-access-token` for Azure Resource Manager:

```bash
az account get-access-token --resource https://management.azure.com/
```

For a custom API using MSAL (Microsoft Authentication Library), request the scope explicitly:

```csharp
string[] scopes = new[] { "api://<api-app-id>/user_impersonation" };
var result = await app.AcquireTokenInteractive(scopes).ExecuteAsync();
```

**Delegated permissions, application permissions, and app roles**

Authorization in Entra ID often depends on the permission model.

- **Delegated permissions**: the app acts as the signed-in user
- **Application permissions**: the app acts as itself without a user
- **App roles**: permissions defined by the resource application and assigned to users, groups, or service principals

Use delegated permissions when:
- the action should reflect the user’s identity
- the app needs user context
- the service should enforce per-user access

Use application permissions when:
- the app runs unattended
- a background job or daemon needs access
- there is no interactive user

Troubleshooting tips:
- Delegated tokens carry `scp`.
- App-only tokens usually carry `roles`.
- If the token lacks the expected claim, consent or assignment may be missing.
- Admin consent may be required for application permissions.

**Group membership and claim overage**

Group-based authorization is convenient, but it can cause propagation and token-size issues.

Important behaviors:
- Group membership changes may not appear in an existing token.
- Users often need to sign out and sign in again to refresh claims.
- Large group memberships can trigger group overage, where the token contains a reference instead of all groups.

If your app relies on group claims:
- verify the app registration is configured to emit group claims
- handle overage by querying Microsoft Graph when needed
- avoid assuming every group is always present in the token

Example of a Graph call to inspect group membership:

```http
GET https://graph.microsoft.com/v1.0/me/memberOf
Authorization: Bearer <access-token>
```

**Key Vault authorization issues**

Key Vault is a frequent source of confusion because it can use different authorization models.

Common causes of failure:
- the identity has no `get` permission for secrets, keys, or certificates
- the vault is configured for RBAC but the app expects access policies, or vice versa
- the token is for the wrong resource
- network rules block access before authorization is evaluated

Useful checks:
- Confirm whether the vault uses Azure RBAC or access policies.
- Verify the principal has the correct data-plane role or policy.
- Confirm the secret name and vault URI are correct.
- Check whether firewall or private endpoint rules are blocking the request.

Example RBAC role assignment:

```bash
az role assignment create \
  --assignee <principal-id> \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.KeyVault/vaults/<vault-name>
```

**Storage authorization issues**

Azure Storage supports multiple authorization methods:
- shared key
- SAS (Shared Access Signature)
- Microsoft Entra ID with RBAC

When using Entra ID:
- the token must target the storage service
- the principal needs a data role such as `Storage Blob Data Reader`
- the role must be assigned at the correct scope
- the SDK must be configured to use token credentials

Example .NET code:

```csharp
using Azure.Identity;
using Azure.Storage.Blobs;

var blobServiceClient = new BlobServiceClient(
    new Uri("https://<account-name>.blob.core.windows.net"),
    new DefaultAzureCredential());

var container = blobServiceClient.GetBlobContainerClient("images");
await foreach (var blob in container.GetBlobsAsync())
{
    Console.WriteLine(blob.Name);
}
```

If this returns `403`, check:
- whether the identity has a data-plane role
- whether the role assignment has propagated
- whether the storage account firewall allows the caller
- whether the app is using the correct tenant and subscription context

**App registration and consent problems**

App registrations define how your application authenticates and what permissions it requests. Authorization issues often come from incomplete consent or misconfigured app roles.

Check:
- redirect URI matches the app type
- client secret or certificate is valid
- API permissions are granted
- admin consent has been completed when required
- exposed scopes are configured correctly
- app roles are assigned to the correct principals

Common Entra ID errors:
- `AADSTS65001`: consent required
- `AADSTS7000215`: invalid client secret
- `AADSTS500011`: resource principal not found
- `AADSTS700016`: application not found in the directory

Practical troubleshooting steps:
- Recreate the token after changing permissions.
- Clear cached credentials.
- Verify the tenant ID and client ID.
- Confirm the app registration and enterprise application both exist where expected.

**Using logs and tools to diagnose access failures**

You can usually isolate the problem faster by checking logs before changing configuration.

Useful tools:
- **Azure Activity Log** for management-plane operations
- **Resource logs** for service-specific authorization failures
- **Microsoft Entra sign-in logs** for authentication and consent issues
- **Application logs** for token acquisition and HTTP status codes
- **JWT decoding tools** for inspecting claims

CLI examples:

```bash
# Inspect a token response
az account get-access-token --resource https://vault.azure.net/ -o json

# View current role assignments
az role assignment list --all -o table

# Check the current subscription context
az account show --query "{subscription:name, tenantId:tenantId, user:user.name}" -o json
```

When reading logs, look for:
- the exact principal ID
- the exact resource URI
- the exact error code
- the timestamp relative to the permission change
- whether the failure is intermittent, which often suggests propagation or caching

**A practical troubleshooting workflow**

1. Confirm the error type.
   - Use `401` for authentication problems and `403` for authorization problems as your first clue.
2. Identify the identity.
   - Determine whether the caller is a user, service principal, managed identity, or group-based principal.
3. Inspect the token.
   - Decode the JWT and verify `aud`, `iss`, `scp`, `roles`, and `exp`.
4. Verify the permission model.
   - Check whether the target uses RBAC, access policies, app roles, or custom authorization.
5. Check scope and assignment.
   - Confirm the role is assigned to the right principal at the right scope.
6. Wait for propagation if the change is recent.
   - Reauthenticate and retry after a short delay.
7. Check network and service-specific restrictions.
   - Firewalls, private endpoints, conditional access, and tenant restrictions can block access before authorization succeeds.

**Example decision tree**

- If the token is missing or expired:
  - reacquire the token
  - clear cached credentials
  - verify the client secret or certificate
- If the token is valid but access is denied:
  - verify role assignment
  - verify scope
  - verify the service’s authorization model
- If access works later without changes:
  - suspect propagation delay or caching
- If only some users fail:
  - inspect group membership, consent, or conditional access

**Practical code patterns for resilient auth**

Use retry logic for transient failures, especially immediately after role assignment or identity enablement. Keep retries bounded and log the exact failure.

```csharp
for (int attempt = 1; attempt <= 5; attempt++)
{
    try
    {
        var response = await client.GetAsync("https://<api-endpoint>");
        response.EnsureSuccessStatusCode();
        break;
    }
    catch (HttpRequestException ex) when (attempt < 5)
    {
        await Task.Delay(TimeSpan.FromSeconds(attempt * 2));
    }
}
```

Use `DefaultAzureCredential` for local development and Azure-hosted workloads when appropriate, but remember that it resolves multiple credential sources. If the wrong identity is selected locally, you may troubleshoot the wrong principal.

**What to remember for AZ-204**

- **Authentication** answers “who are you?”
- **Authorization** answers “what can you do?”
- **Propagation** explains why a correct change may not work immediately.
- **Token audience** must match the target service.
- **RBAC scope** must match the resource and permission model.
- **Managed identity** still needs assignment and time to propagate.
- **Logs and claims** are your fastest path to root cause.

A strong troubleshooting habit is to verify the identity, token, scope, and service model in that order. That sequence prevents you from chasing the wrong layer when the real issue is a stale token or a delayed role assignment.
