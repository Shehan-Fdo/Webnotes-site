## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Azure RBAC overview**

Azure Role-Based Access Control (RBAC) is the authorization layer you use to grant identities access to Azure resources. It applies at different scopes, including management group, subscription, resource group, resource, and sometimes data-plane-specific scopes depending on the service.

For AZ-204, the key distinction is between:

- **Management-plane access**: control of Azure resources such as creating a storage account or changing a Key Vault setting.
- **Data-plane access**: access to the data inside the service, such as reading a Cosmos DB item, fetching a secret from Key Vault, or downloading a blob.

Many exam questions hinge on whether the identity needs a management role, a data role, or both.

**Identity types you assign roles to**

You typically grant access to one of these principals:

- **User**: a human operator in Microsoft Entra ID.
- **Group**: preferred for maintainability when multiple users need the same access.
- **Service principal**: an app registration or workload identity used by an application.
- **Managed identity**: an Azure-managed service identity for App Service, Functions, VM, Container Apps, and similar services.

Use managed identities whenever the workload runs in Azure and does not need stored credentials.

**Common role-assignment workflow**

1. Identify the scope: subscription, resource group, resource, or data-plane scope.
2. Confirm the service is configured to use Azure RBAC for data access if required.
3. Assign the least-privilege role to the correct principal.
4. Test with the same identity the app will use at runtime.
5. Troubleshoot authentication first, then authorization.

**Cosmos DB access with Azure RBAC**

Azure Cosmos DB supports Azure AD-based authorization for data access, but the account must be configured to allow it. For many Cosmos DB APIs, especially the NoSQL API, you enable the account to use Azure AD/RBAC for the data plane before role assignments will work.

The important prerequisite is that the Cosmos DB account must be set to use Azure AD authentication for data access, often described as enabling the Azure RBAC data-plane mode or disabling local key-only access for the scenario you want to secure. If the account is not configured for Azure AD data-plane authorization, role assignments alone do not grant item-level access.

Typical Cosmos DB data roles include:

- **Cosmos DB Built-in Data Reader**: read data only.
- **Cosmos DB Built-in Data Contributor**: read and write data.
- API-specific roles may exist for certain workloads or custom role definitions.

A common pattern is to assign the role at the Cosmos DB account scope or a narrower scope if supported by the service and your governance model.

Example Azure CLI role assignment:

```bash
az role assignment create \
  --assignee-object-id <principal-object-id> \
  --assignee-principal-type ServicePrincipal \
  --role "Cosmos DB Built-in Data Contributor" \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.DocumentDB/databaseAccounts/<account-name>
```

If you use a managed identity, the app authenticates with Microsoft Entra ID and requests a token for the Cosmos DB resource. The SDK then uses that token instead of an account key.

Example .NET code using managed identity:

```csharp
using Azure.Identity;
using Microsoft.Azure.Cosmos;

var endpoint = Environment.GetEnvironmentVariable("COSMOS_ENDPOINT");
var credential = new DefaultAzureCredential();

var client = new CosmosClient(endpoint, credential);
var container = client.GetContainer("appdb", "orders");

var response = await container.ReadItemAsync<dynamic>("order-1", new PartitionKey("customer-1"));
Console.WriteLine(response.Resource);
```

Example JavaScript code using managed identity or any Azure AD credential supported by the SDK:

```javascript
const { CosmosClient } = require("@azure/cosmos");
const { DefaultAzureCredential } = require("@azure/identity");

const endpoint = process.env.COSMOS_ENDPOINT;
const credential = new DefaultAzureCredential();

const client = new CosmosClient({ endpoint, aadCredentials: credential });
const container = client.database("appdb").container("orders");

const { resource } = await container.item("order-1", "customer-1").read();
console.log(resource);
```

**Cosmos DB troubleshooting flow**

When authorization fails, check in this order:

1. Confirm the app is using Microsoft Entra ID authentication, not an account key.
2. Verify the Cosmos DB account is configured for Azure AD/RBAC data-plane access.
3. Confirm the principal has the correct Cosmos DB data role.
4. Check the role scope matches the account or database/container scope you intended.
5. Wait for role propagation, then retry.
6. Inspect SDK errors for `401 Unauthorized` versus `403 Forbidden`:
   - `401` often points to token acquisition or wrong audience/resource.
   - `403` often points to missing data-plane permission.

> If the account still expects key-based access for the scenario, RBAC role assignment alone will not fix the failure.

**Key Vault access with Azure RBAC**

Azure Key Vault supports two authorization models:

- **Access policies**: the older model, where you grant secret, key, or certificate permissions directly in the vault.
- **Azure RBAC**: the newer model, where you assign Azure roles to principals.

For RBAC to work, the vault must be configured to use Azure RBAC for permission management. In the portal, this is typically shown as the **Permission model** setting. You must switch it from access policies to Azure role-based access control before RBAC data roles apply.

If the vault is still in access policy mode, RBAC role assignments do not authorize secret or key operations.

Common Key Vault roles include:

- **Key Vault Secrets User**: read secrets.
- **Key Vault Secrets Officer**: manage secrets.
- **Key Vault Crypto User**: use keys for cryptographic operations.
- **Key Vault Administrator**: full data-plane control.

Example Azure CLI to set the permission model to RBAC:

```bash
az keyvault update \
  --name <vault-name> \
  --resource-group <rg> \
  --enable-rbac-authorization true
```

Example role assignment for secret read access:

```bash
az role assignment create \
  --assignee-object-id <principal-object-id> \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.KeyVault/vaults/<vault-name>
```

Example .NET code using managed identity to read a secret:

```csharp
using Azure.Identity;
using Azure.Security.KeyVault.Secrets;

var vaultUri = new Uri(Environment.GetEnvironmentVariable("KEYVAULT_URI"));
var client = new SecretClient(vaultUri, new DefaultAzureCredential());

KeyVaultSecret secret = await client.GetSecretAsync("DbPassword");
Console.WriteLine(secret.Value);
```

**Key Vault troubleshooting flow**

Use this sequence when a secret or key call fails:

1. Confirm the vault permission model is set to Azure RBAC, not access policies.
2. Confirm the principal has the correct Key Vault data role.
3. Confirm the role scope is the vault or a valid parent scope.
4. Confirm the app is using an Azure AD credential such as managed identity, service principal, or workload identity.
5. Check whether the SDK is requesting the correct vault URI.
6. Distinguish errors:
   - `401` usually means authentication or token audience issues.
   - `403` usually means the identity authenticated successfully but lacks permission.

A common exam trap is assuming that adding a role assignment is enough. For Key Vault, the vault configuration must match the authorization model you are using.

**Storage access with Azure RBAC**

Azure Storage uses both management-plane and data-plane roles, and this distinction is important.

- **Management-plane roles** control the storage account resource itself.
  - Example: create or delete the account, change networking, configure replication.
- **Data-plane roles** control access to blobs, queues, tables, and file shares.
  - Example: read a blob, send a queue message, or list file share contents.

A user with `Contributor` on the storage account can manage the account but still cannot read blobs unless they also have a storage data role.

Common Storage data roles include:

- **Storage Blob Data Reader**: read blobs.
- **Storage Blob Data Contributor**: read/write blobs.
- **Storage Queue Data Contributor**: work with queue messages.
- **Storage File Data SMB Share Reader** / contributor roles: access file shares depending on protocol and role.
- **Storage Table Data Contributor**: access table entities.

Common management roles include:

- **Contributor**: manage Azure resources, but not data inside them.
- **Storage Account Contributor**: manage storage account settings.
- **Reader**: view resource settings only.

Example role assignment for blob data access:

```bash
az role assignment create \
  --assignee-object-id <principal-object-id> \
  --role "Storage Blob Data Contributor" \
  --scope /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<account-name>
```

Example .NET code using managed identity to upload a blob:

```csharp
using Azure.Identity;
using Azure.Storage.Blobs;

var blobServiceClient = new BlobServiceClient(
    new Uri(Environment.GetEnvironmentVariable("BLOB_SERVICE_URI")),
    new DefaultAzureCredential());

var container = blobServiceClient.GetBlobContainerClient("images");
var blob = container.GetBlobClient("photo.png");

await blob.UploadAsync("photo.png", overwrite: true);
```

Example Azure CLI to verify the signed-in identity and role assignments:

```bash
az account show
az role assignment list \
  --assignee <principal-object-id> \
  --all \
  --output table
```

**Storage troubleshooting flow**

When Storage authorization fails, separate management-plane and data-plane issues first:

1. Confirm the app is using an Azure AD credential, not a shared key or SAS unless that is intentional.
2. Confirm the identity has a **data-plane** role for the specific service:
   - Blob access needs blob data roles.
   - Queue access needs queue data roles.
   - File share access needs file data roles.
3. Confirm the role scope includes the storage account or the narrower resource path you expect.
4. Confirm the storage account networking rules allow the request if firewalls or private endpoints are enabled.
5. Check whether the SDK is using the correct endpoint:
   - Blob: `https://<account>.blob.core.windows.net`
   - Queue: `https://<account>.queue.core.windows.net`
   - File: `https://<account>.file.core.windows.net`
6. Review the error:
   - `AuthenticationFailed` or token errors suggest credential or audience problems.
   - `AuthorizationPermissionMismatch` or `403` suggests missing data role.

> A management role such as `Contributor` does not replace a storage data role for blob, queue, or file access.

**How Azure AD authentication works in apps**

Your application usually authenticates with one of these approaches:

- **Managed identity**: best for Azure-hosted workloads.
- **Service principal**: useful for local development, CI/CD, or non-Azure hosting.
- **Developer identity**: `az login` or Visual Studio sign-in for local testing.

The Azure SDKs commonly use `DefaultAzureCredential`, which tries multiple credential sources in a predictable order. This makes it useful for local development and production when managed identity is available.

Example service principal environment variables:

```bash
export AZURE_TENANT_ID="<tenant-id>"
export AZURE_CLIENT_ID="<app-client-id>"
export AZURE_CLIENT_SECRET="<client-secret>"
```

Example local development with Azure CLI sign-in:

```bash
az login
az account set --subscription <sub-id>
```

When you use `DefaultAzureCredential`, the same code can work locally and in Azure:

```csharp
var credential = new DefaultAzureCredential();
```

**Practical exam patterns**

- If the question mentions **reading a secret**, think Key Vault data-plane RBAC and vault permission model.
- If the question mentions **reading or writing Cosmos DB items**, think Cosmos DB data roles and Azure AD data-plane enablement.
- If the question mentions **blob upload/download**, think Storage data roles, not just subscription or resource group roles.
- If the question mentions **managed identity**, think passwordless access with Azure AD tokens.
- If the question mentions **Contributor**, verify whether it is only a management-plane role.

**Quick comparison**

| Service | Must be configured for RBAC data access? | Typical data role | Common pitfall |
|---|---:|---|---|
| Cosmos DB | Yes, for Azure AD data-plane access | Cosmos DB Built-in Data Contributor | Role assigned before account is enabled for Azure AD access |
| Key Vault | Yes, if using Azure RBAC instead of access policies | Key Vault Secrets User | Vault still in access policy mode |
| Storage | No special switch for most data-plane RBAC scenarios, but data role is required | Storage Blob Data Contributor | Using management role instead of data role |

**End-to-end authorization checklist**

1. Identify the target service and operation.
2. Confirm whether the operation is management-plane or data-plane.
3. Ensure the service is configured for the intended authorization model.
4. Assign the least-privilege role at the correct scope.
5. Use an Azure AD identity in the app.
6. Test with the same identity and endpoint the app uses in production.
7. If it fails, check token acquisition, role scope, and service-specific prerequisites in that order.

**Key takeaways**

- **Cosmos DB** needs Azure AD/RBAC data-plane enablement before role assignments can authorize data access.
- **Key Vault** requires the vault to be switched to Azure RBAC mode if you want RBAC-based data access.
- **Storage** requires the correct data-plane role; management roles do not grant blob, queue, or file data access.
- **Managed identity** is the preferred runtime identity for Azure-hosted apps.
- **DefaultAzureCredential** is the common SDK choice for passwordless authentication across local and cloud environments.
