## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Identity-first authentication in Azure SDK for Python**

`DefaultAzureCredential` is the recommended starting point for authenticating Azure SDK clients in Python. It is part of the `azure-identity` package and implements a credential chain that tries multiple authentication methods in a predictable order until one succeeds. This lets you write one code path that works in local development, CI/CD, and Azure-hosted environments.

The main benefit is portability. You can develop locally with your own sign-in, then deploy the same code to Azure using managed identity without changing application logic. In practice, you configure the environment, not the code.

- **`azure-identity`** provides authentication credentials for Azure SDK clients.
- **`DefaultAzureCredential`** is a chained credential that selects the best available identity source.
- **Managed identity** is the preferred production option when your app runs in Azure.
- **Environment variables** are useful for service principals in automation and local testing.
- **Developer credentials** support interactive sign-in during local development.

**How the credential chain works**

`DefaultAzureCredential` attempts several credential types in sequence. The exact chain can vary by package version, but the common pattern is:

1. Environment-based credentials
2. Managed identity
3. Developer tool credentials such as Azure CLI sign-in, Visual Studio Code sign-in, or interactive browser sign-in depending on configuration

If one credential fails because it is unavailable, the chain continues. If a credential is available but misconfigured, you may see an authentication failure that stops the chain. That distinction matters when troubleshooting.

```python
from azure.identity import DefaultAzureCredential
from azure.storage.blob import BlobServiceClient

credential = DefaultAzureCredential()
client = BlobServiceClient(
    account_url="https://mystorageaccount.blob.core.windows.net",
    credential=credential
)
```

This pattern works because Azure SDK clients accept a credential object rather than a raw access token. The SDK requests tokens as needed and refreshes them automatically.

**When to use `DefaultAzureCredential`**

Use it when you want one authentication strategy that adapts to the execution environment. It is especially useful for:

- Local development on a workstation
- Integration tests that run in a developer context
- Applications deployed to Azure App Service, Azure Functions, Azure Container Apps, or virtual machines with managed identity
- CI/CD pipelines that authenticate through service principals or federated identity

It is usually the best default choice unless you have a specific reason to use a more targeted credential.

**Equivalent Python credential patterns**

`DefaultAzureCredential` is convenient, but you should understand the underlying credential types because they solve different problems.

| Credential | Typical use | Notes |
|---|---|---|
| `EnvironmentCredential` | Automation and service principal auth | Reads `AZURE_TENANT_ID`, `AZURE_CLIENT_ID`, and secret or certificate settings |
| `ManagedIdentityCredential` | Azure-hosted workloads | Uses the platform identity endpoint; no secret stored in code |
| `AzureCliCredential` | Local developer sign-in | Uses the signed-in Azure CLI session |
| `InteractiveBrowserCredential` | Manual local sign-in | Opens a browser for user authentication |
| `ClientSecretCredential` | Explicit service principal auth | Good when you want direct control over tenant, client ID, and secret |
| `ClientCertificateCredential` | Service principal with certificate | Better than secrets for some enterprise scenarios |

A common equivalent pattern is to choose the credential explicitly instead of relying on the default chain.

```python
from azure.identity import ManagedIdentityCredential
from azure.storage.blob import BlobServiceClient

credential = ManagedIdentityCredential()
client = BlobServiceClient(
    account_url="https://mystorageaccount.blob.core.windows.net",
    credential=credential
)
```

This is useful when you know the app will only run in Azure and you want to avoid ambiguity during troubleshooting.

**Local development options**

For local development, the most common approach is to use Azure CLI authentication. You sign in once, and the SDK can reuse that identity.

```bash
az login
az account show
```

Then your Python code can use `DefaultAzureCredential` without extra secrets:

```python
from azure.identity import DefaultAzureCredential
from azure.keyvault.secrets import SecretClient

credential = DefaultAzureCredential()
client = SecretClient(
    vault_url="https://myvault.vault.azure.net/",
    credential=credential
)
```

If you need to test a specific identity path, you can use a more explicit credential. For example, `AzureCliCredential` forces the app to use the CLI session rather than falling back to other sources.

```python
from azure.identity import AzureCliCredential

credential = AzureCliCredential()
```

**Environment-based authentication**

Environment variables are a common way to authenticate service principals in scripts, build agents, and containerized workloads. The exact variables depend on the credential type and whether you use a secret or certificate.

Typical secret-based configuration:

```bash
export AZURE_TENANT_ID="00000000-0000-0000-0000-000000000000"
export AZURE_CLIENT_ID="11111111-1111-1111-1111-111111111111"
export AZURE_CLIENT_SECRET="your-client-secret"
```

Python code stays unchanged:

```python
from azure.identity import DefaultAzureCredential

credential = DefaultAzureCredential()
```

This works because `DefaultAzureCredential` checks environment configuration early in the chain. If the variables are present and valid, it uses them.

> Avoid hardcoding secrets in source code or committing them to repositories. Use environment variables, managed identity, or a secret store such as Azure Key Vault.

**Managed identity in Azure**

Managed identity is the preferred production pattern for Azure-hosted applications because Azure manages the identity lifecycle. You do not store credentials in code or configuration files.

There are two common forms:

- **System-assigned managed identity**: tied to a single Azure resource and deleted with it
- **User-assigned managed identity**: created as a standalone resource and attached to one or more Azure resources

In Python, `DefaultAzureCredential` usually detects managed identity automatically when the app runs in Azure. If you need to be explicit, use `ManagedIdentityCredential`.

```python
from azure.identity import ManagedIdentityCredential
from azure.keyvault.secrets import SecretClient

credential = ManagedIdentityCredential()
client = SecretClient(
    vault_url="https://myvault.vault.azure.net/",
    credential=credential
)
```

For user-assigned managed identity, you typically provide the client ID:

```python
from azure.identity import ManagedIdentityCredential

credential = ManagedIdentityCredential(client_id="11111111-1111-1111-1111-111111111111")
```

This is useful when a resource has multiple identities or when you want to pin access to a specific identity.

**Service principal patterns**

A service principal is an application identity in Microsoft Entra ID. It is commonly used in automation and non-interactive scenarios. In Python, you can authenticate directly with `ClientSecretCredential` or `ClientCertificateCredential`.

```python
from azure.identity import ClientSecretCredential
from azure.storage.queue import QueueServiceClient

credential = ClientSecretCredential(
    tenant_id="00000000-0000-0000-0000-000000000000",
    client_id="11111111-1111-1111-1111-111111111111",
    client_secret="your-client-secret"
)

client = QueueServiceClient(
    account_url="https://mystorageaccount.queue.core.windows.net",
    credential=credential
)
```

Use this pattern when you need explicit control over the identity source or when your environment does not support managed identity. Prefer certificates over secrets when enterprise policy requires stronger credential handling.

**Practical troubleshooting**

Authentication issues usually fall into one of three categories:

- The credential is unavailable in the current environment
- The identity exists but lacks permission to the target resource
- The tenant, subscription, or resource endpoint is incorrect

A useful debugging technique is to enable identity logging and inspect which credential the chain selected.

```python
import logging
from azure.identity import DefaultAzureCredential

logging.basicConfig(level=logging.INFO)
logging.getLogger("azure.identity").setLevel(logging.DEBUG)

credential = DefaultAzureCredential()
```

You can also test the token acquisition path directly:

```python
token = credential.get_token("https://storage.azure.com/.default")
print(token.token[:20])
```

This helps you separate authentication problems from SDK client configuration problems.

**Common permission model**

Getting a token does not mean you can access the resource. Azure uses authorization after authentication. For example, a managed identity may authenticate successfully but still fail if it lacks a role assignment such as:

- **Storage Blob Data Contributor**
- **Key Vault Secrets User**
- **Reader** or a more specific data-plane role, depending on the service

For many Azure services, you must assign a data-plane role at the resource scope. Control-plane roles like Contributor do not always grant access to data operations.

**Choosing the right credential**

Use this decision pattern:

- Use `DefaultAzureCredential` when you want the SDK to adapt across environments.
- Use `ManagedIdentityCredential` when the app only runs in Azure and you want explicit managed identity behavior.
- Use `AzureCliCredential` or `InteractiveBrowserCredential` for local developer workflows.
- Use `ClientSecretCredential` or `ClientCertificateCredential` for automation outside Azure or when you need direct service principal control.

A simple rule of thumb is:

- **Development**: developer sign-in or `DefaultAzureCredential`
- **Production in Azure**: managed identity
- **Automation outside Azure**: service principal or federated identity

**Example: Azure Blob Storage with `DefaultAzureCredential`**

```python
from azure.identity import DefaultAzureCredential
from azure.storage.blob import BlobServiceClient

account_url = "https://mystorageaccount.blob.core.windows.net"
credential = DefaultAzureCredential()
service_client = BlobServiceClient(account_url=account_url, credential=credential)

container_client = service_client.get_container_client("images")
for blob in container_client.list_blobs():
    print(blob.name)
```

This pattern is common in AZ-204 scenarios because it demonstrates secure access without connection strings.

**Example: Azure Key Vault with explicit credential**

```python
from azure.identity import ManagedIdentityCredential
from azure.keyvault.secrets import SecretClient

credential = ManagedIdentityCredential()
client = SecretClient(
    vault_url="https://myvault.vault.azure.net/",
    credential=credential
)

secret = client.get_secret("db-password")
print(secret.value)
```

This is a good production pattern for apps running in Azure App Service or Azure Functions with managed identity enabled.

**Example: fallback behavior in development**

You can combine explicit and default patterns in a way that keeps local development easy while still supporting Azure deployment.

```python
import os
from azure.identity import DefaultAzureCredential, ManagedIdentityCredential

if os.getenv("WEBSITE_INSTANCE_ID"):
    credential = ManagedIdentityCredential()
else:
    credential = DefaultAzureCredential()
```

This is not always necessary, but it can help when you want to force managed identity in Azure and use developer credentials locally. In many cases, `DefaultAzureCredential` alone is enough.

**Operational best practices**

- Prefer identity-based authentication over connection strings.
- Use managed identity whenever the workload runs in Azure.
- Keep secrets out of code and source control.
- Grant the minimum required role at the narrowest scope.
- Test authentication locally and in Azure because the selected credential may differ.
- Log credential selection during troubleshooting, but avoid logging secrets or full tokens.

**What to remember for the exam**

- `DefaultAzureCredential` is the standard Azure Identity entry point for Python SDK authentication.
- It chains multiple credential sources and chooses the first usable one.
- It supports local development and Azure deployment with the same code.
- Managed identity is the preferred production identity for Azure-hosted apps.
- Service principals are still important for automation and non-Azure environments.
- Authentication and authorization are separate; a valid token does not guarantee access.
