## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Purpose of the validation and recovery workflow**

In Azure development, you do not treat deployment as the finish line. You validate the change, watch for regressions, troubleshoot failures quickly, and roll back safely if the release harms availability or correctness. You also document what changed so the next incident is faster to diagnose.

This topic is about building a repeatable release loop:

- verify the solution works in the target environment
- identify whether failures come from code, configuration, identity, networking, or platform dependencies
- revert to a known-good state when the risk is higher than the benefit
- capture the fix, the root cause, and the operational lesson

**Validation before and after deployment**

Validation starts before production. You want confidence that the build artifact, configuration, and infrastructure assumptions match the environment where the app will run. In Azure, validation often spans local checks, CI/CD pipeline checks, and runtime health checks after deployment.

Common validation targets include:

- **functional behavior**: endpoints return expected responses and business rules still hold
- **configuration correctness**: app settings, connection strings, feature flags, and managed identity permissions are present
- **dependency readiness**: storage accounts, queues, databases, Key Vault secrets, and APIs are reachable
- **performance baselines**: latency, throughput, and resource usage remain within acceptable limits
- **security posture**: authentication, authorization, and secret handling still work as intended

A practical validation sequence often looks like this:

1. Deploy to a nonproduction slot or environment.
2. Run smoke tests against critical paths.
3. Check logs, metrics, and distributed traces.
4. Promote only if health and business checks pass.
5. Keep a rollback path ready until the release stabilizes.

**Typical validation techniques**

Use multiple layers of validation because one test type rarely catches everything.

- **Smoke tests** confirm the app starts and the most important endpoints respond.
- **Integration tests** verify the app can talk to Azure services such as Azure SQL Database, Azure Storage, Service Bus, or Cosmos DB.
- **Synthetic checks** simulate user traffic from a monitoring system.
- **Health probes** tell Azure whether the app instance is ready or alive.
- **Manual verification** is useful for high-risk changes, but it should not be the only gate.

For web apps and APIs, a simple smoke test can be as small as a `GET /health` endpoint:

```bash
curl -i https://myapp-staging.azurewebsites.net/health
```

If you use deployment slots, validate in the staging slot before swapping:

```bash
az webapp deployment slot swap \
  --resource-group rg-app \
  --name myapp \
  --slot staging \
  --target-slot production
```

**Azure health and observability signals**

Validation depends on observability. Azure gives you several signals that help you decide whether the solution is healthy.

- **Application Insights** provides request rates, failures, dependencies, exceptions, and custom events.
- **Azure Monitor** aggregates metrics and logs across resources.
- **Log Analytics** lets you query telemetry with Kusto Query Language (KQL).
- **Activity Log** records control-plane operations such as deployments and configuration changes.
- **Resource health** shows whether Azure platform issues affect the resource.

A useful KQL query for recent failures in Application Insights:

```kusto
requests
| where timestamp > ago(1h)
| where success == false
| project timestamp, name, resultCode, operation_Id, cloud_RoleName
| order by timestamp desc
```

For dependency failures:

```kusto
dependencies
| where timestamp > ago(1h)
| where success == false
| project timestamp, target, name, resultCode, duration
| order by timestamp desc
```

These queries help you separate application bugs from downstream service problems.

**Troubleshooting approach**

Troubleshooting is most effective when you narrow the problem systematically instead of guessing. Start with the symptom, then isolate the layer where the failure occurs.

A practical troubleshooting model is:

- **scope the blast radius**
  - one user, one instance, one region, or all traffic
- **identify the failing layer**
  - client, app code, identity, network, data store, or Azure platform
- **compare good vs bad**
  - recent deployment, config drift, slot swap, secret rotation, or scaling event
- **reproduce safely**
  - use staging, a test subscription, or a controlled request
- **confirm the fix**
  - rerun the same test and verify telemetry improves

**Common failure categories**

- **Configuration errors**
  - wrong app setting name or value
  - missing connection string
  - incorrect feature flag state
  - environment-specific values not deployed
- **Identity and access issues**
  - managed identity not assigned
  - role assignment missing or scoped incorrectly
  - Key Vault access policy or RBAC not configured
  - expired or invalid token
- **Networking issues**
  - private endpoint DNS resolution problems
  - firewall rules blocking access
  - service endpoint or VNet integration misconfiguration
  - outbound SNAT exhaustion in high-throughput apps
- **Code defects**
  - null reference, serialization bug, bad retry logic, race condition
  - incorrect handling of transient failures
  - breaking API contract changes
- **Platform or dependency issues**
  - Azure service throttling
  - database connection limits
  - storage latency spikes
  - regional service degradation

**Useful Azure CLI commands for investigation**

You often need to inspect app settings, logs, and deployment state quickly.

```bash
az webapp config appsettings list \
  --resource-group rg-app \
  --name myapp
```

```bash
az webapp log tail \
  --resource-group rg-app \
  --name myapp
```

```bash
az monitor activity-log list \
  --resource-group rg-app \
  --status Failed \
  --offset 24h
```

```bash
az resource show \
  --ids /subscriptions/<subId>/resourceGroups/rg-app/providers/Microsoft.Web/sites/myapp
```

For containerized workloads, inspect pod or container logs and events as well:

```bash
kubectl logs deploy/myapi
kubectl describe pod myapi-7d9f8c6b7f-abcde
```

**Troubleshooting by Azure service area**

Different Azure services fail in different ways, so your checks should match the service.

- **App Service**
  - check application logs, HTTP status codes, startup failures, and slot settings
  - verify `WEBSITE_RUN_FROM_PACKAGE`, connection strings, and app settings
  - confirm health check path and warm-up behavior
- **Azure Functions**
  - inspect function execution logs and trigger bindings
  - verify storage account connectivity because it is required for many function apps
  - check host.json and extension bundle compatibility
- **Azure Storage**
  - validate SAS token scope and expiry
  - confirm firewall and private endpoint access
  - check retry behavior for transient 5xx and 429 responses
- **Azure SQL Database**
  - verify connection strings, firewall rules, and authentication mode
  - inspect deadlocks, blocking, and query performance
  - confirm schema changes are backward compatible
- **Service Bus**
  - check dead-letter queues, lock renewal, duplicate detection, and message TTL
  - verify namespace authorization and network access
- **Cosmos DB**
  - watch request units (RU/s), throttling, partitioning, and consistency settings
  - confirm the SDK is using the correct endpoint and key or identity

**Rollback strategy**

Rollback means returning to a known-good state when the current release is unsafe or broken. The best rollback is fast, predictable, and already rehearsed.

Use rollback when:

- the defect affects critical user flows
- the fix is not immediately available
- the release causes data corruption risk
- the error rate or latency exceeds your service threshold
- the deployment introduced an unknown dependency issue

Prefer rollback mechanisms that are built into your deployment model.

- **Deployment slots**
  - swap back to the previous slot if the new version fails
  - keep the prior version warm to reduce recovery time
- **Blue-green deployment**
  - route traffic back to the healthy environment
  - preserve the old environment until confidence is high
- **Canary release**
  - reduce traffic to the bad version or stop promotion
  - keep the stable version serving most users
- **Infrastructure as Code (IaC)**
  - redeploy the previous template or parameter set
  - revert resource configuration drift
- **Versioned artifacts**
  - redeploy the last known-good package, container image, or build artifact

A slot-based rollback example:

```bash
az webapp deployment slot swap \
  --resource-group rg-app \
  --name myapp \
  --slot production \
  --target-slot staging
```

If you deploy container images, rollback often means pinning the previous image tag:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapi
spec:
  template:
    spec:
      containers:
        - name: myapi
          image: myregistry.azurecr.io/myapi:1.4.2
```

**Rollback considerations**

Rollback is not always safe if the release changed data shape or external contracts.

> Roll back application code carefully when the new version has already written incompatible data or migrated schemas. In those cases, you may need a forward fix instead of a full revert.

Watch for these risks:

- **database migrations**
  - schema changes may not be reversible
  - down-level code may not understand new columns or records
- **message contracts**
  - consumers may fail if producers changed payload shape
- **feature flags**
  - turning off a feature may be safer than redeploying
- **stateful services**
  - cached or persisted state may survive the rollback and still cause errors

A safer pattern is to use backward-compatible changes:

- add fields before removing them
- deploy consumers before producers when possible
- use feature flags for risky behavior
- keep migrations additive until all instances are upgraded

**Documenting the solution**

Documentation turns a one-time fix into reusable operational knowledge. After you resolve the issue, record what happened while the details are fresh.

Document:

- the symptom and impact
- the environment and version
- the root cause
- the diagnostic steps you used
- the fix and why it worked
- the rollback decision, if any
- the preventive action for next time

A concise incident note might include:

```text
Incident: API returned 500s after deployment
Impact: 18% of requests failed for 12 minutes
Root cause: missing Key Vault access for managed identity
Fix: assigned Key Vault Secrets User role to the app identity
Prevention: add post-deploy check for secret retrieval
```

Good documentation lives in the places your team already uses:

- pull request notes
- release notes
- runbooks
- incident tickets
- architecture decision records
- pipeline comments or deployment logs

**What to include in a runbook**

A runbook should let another engineer repeat the diagnosis and recovery without guessing.

- service name and resource group
- expected health signals and thresholds
- common failure modes
- commands to inspect logs and metrics
- rollback steps and approval requirements
- contact points for dependent teams

Example runbook snippet:

```bash
# Check app health
curl -f https://myapp.azurewebsites.net/health

# Check recent failures
az monitor activity-log list --resource-group rg-app --status Failed --offset 24h

# Tail logs
az webapp log tail --resource-group rg-app --name myapp
```

**Practical release checklist**

Use a checklist to reduce missed steps during deployment and recovery.

1. Confirm the build passed tests and security checks.
2. Deploy to staging or a canary slice.
3. Validate health endpoints, logs, and dependency calls.
4. Compare metrics against baseline.
5. Promote only if error rate and latency are acceptable.
6. Keep rollback instructions ready.
7. Document the outcome and any follow-up work.

**Signals that a rollback is better than a hotfix**

A hotfix is not always the right answer. Roll back first when:

- the issue is severe and the cause is unclear
- the fix would take longer than the rollback
- the release affects many users
- the system is unstable enough that debugging in place is risky
- the change is not yet required for business continuity

A hotfix is more appropriate when:

- the issue is isolated and well understood
- rollback would cause more disruption than the bug
- the fix is small, safe, and can be validated quickly

**Exam-focused takeaways**

For AZ-204, focus on the operational mindset:

- validate changes with tests, health checks, and telemetry
- troubleshoot by isolating the failing layer and checking Azure diagnostics
- use deployment slots, versioned artifacts, and traffic shifting for rollback
- avoid irreversible changes unless you have a recovery plan
- document the incident, fix, and prevention steps so the team can respond faster next time

**Quick reference: validation and recovery tools**

| Area | Tooling | Typical use |
|---|---|---|
| App health | Health endpoints, probes | Confirm the app is alive and ready |
| Telemetry | Application Insights, Azure Monitor | Find failures, latency, and dependency issues |
| Logs | Log Analytics, app logs, container logs | Investigate errors and trace requests |
| Deployment | Slots, canary, blue-green | Reduce risk and enable fast rollback |
| Recovery | Previous artifact, prior image tag, IaC redeploy | Restore a known-good state |
