## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Asynchronous workflow overview**

Azure Service Bus and Azure Event Grid solve different parts of the asynchronous integration problem. Service Bus is a brokered messaging service for reliable command and workflow processing. Event Grid is an event routing service for reactive, event-driven integration.

Use **Service Bus** when you need:
- Guaranteed delivery with brokered queues or topics
- Competing consumers or pub/sub with durable storage
- Message ordering with sessions
- Scheduled delivery, duplicate detection, and dead-lettering
- Explicit message settlement and lock-based processing

Use **Event Grid** when you need:
- Lightweight event distribution to many subscribers
- Near real-time push delivery to webhooks and Azure services
- Event filtering and fan-out
- Integration with Azure resource events, custom events, and partner events

A common architecture uses Event Grid to react to resource changes and Service Bus to buffer work for downstream processing. Event Grid is the notification layer; Service Bus is the work queue.

**Azure Service Bus fundamentals**

Service Bus supports **queues** for point-to-point messaging and **topics/subscriptions** for publish-subscribe messaging. Messages are stored durably until a consumer receives and settles them.

Key concepts:
- **Queue**: one producer, one logical stream, many competing consumers
- **Topic**: one producer, many subscriptions
- **Subscription**: a durable copy of topic messages with its own filters and rules
- **Dead-letter queue (DLQ)**: a subqueue for messages that cannot be processed
- **Peek-lock**: receive a message without removing it until you complete it
- **Receive-and-delete**: remove the message immediately on receive

Service Bus is appropriate when the consumer may be offline, slow, or temporarily unavailable. The broker holds the message until the application is ready.

**Message settlement and lock behavior**

In peek-lock mode, the broker grants a lock for a message. You process the message and then settle it:
- **Complete**: remove the message from the queue or subscription
- **Abandon**: release the lock so the message can be retried
- **Dead-letter**: move the message to the DLQ with a reason
- **Defer**: keep the message in the queue but hide it from normal receives

If the lock expires before settlement, the message becomes available again and may be delivered more than once. Design handlers to be **idempotent** so duplicate processing does not corrupt state.

Example with the modern .NET SDK:

```csharp
using Azure.Messaging.ServiceBus;

var client = new ServiceBusClient(connectionString);
var processor = client.CreateProcessor("orders", new ServiceBusProcessorOptions
{
    MaxConcurrentCalls = 4,
    PrefetchCount = 20,
    AutoCompleteMessages = false,
    MaxAutoLockRenewalDuration = TimeSpan.FromMinutes(5)
});

processor.ProcessMessageAsync += async args =>
{
    string body = args.Message.Body.ToString();

    try
    {
        // Process the message
        Console.WriteLine(body);

        await args.CompleteMessageAsync(args.Message);
    }
    catch (Exception ex)
    {
        Console.WriteLine(ex.Message);
        await args.AbandonMessageAsync(args.Message);
    }
};

processor.ProcessErrorAsync += args =>
{
    Console.WriteLine(args.Exception.ToString());
    return Task.CompletedTask;
};

await processor.StartProcessingAsync();
```

Important processor settings:
- **MaxConcurrentCalls** controls parallel message handlers
- **PrefetchCount** improves throughput by buffering messages locally
- **AutoCompleteMessages** should be `false` when you want explicit settlement
- **MaxAutoLockRenewalDuration** extends the lock automatically while processing

> If processing can exceed the lock duration, use auto-lock renewal or complete work faster. Otherwise the message can reappear and be processed again.

**Sessions, ordering, and concurrency**

Service Bus sessions provide ordered processing for related messages. A session groups messages by a shared **SessionId** and ensures that one consumer processes that session at a time.

Use sessions when:
- You need per-customer or per-order ordering
- You want to serialize work for a logical conversation
- You need session state to store progress

Session-enabled entities change concurrency behavior:
- Messages in the same session are processed sequentially
- Different sessions can be processed in parallel
- A session lock is held while the consumer works on that session

Example session message:

```csharp
var message = new ServiceBusMessage("update order")
{
    SessionId = "order-123",
    MessageId = Guid.NewGuid().ToString()
};

await sender.SendMessageAsync(message);
```

Session state lets you persist small workflow data on the broker side:

```csharp
var sessionReceiver = await client.AcceptSessionAsync("orders", "order-123");
byte[] state = Encoding.UTF8.GetBytes("step=payment-complete");
await sessionReceiver.SetSessionStateAsync(state);

var current = await sessionReceiver.GetSessionStateAsync();
```

Use session state sparingly. It is useful for lightweight workflow checkpoints, but it is not a general-purpose database.

**Duplicate detection and message metadata**

Service Bus can detect duplicate messages when you enable duplicate detection on the entity and set a stable `MessageId`. This is useful when producers retry sends and you want to avoid duplicate work.

Common message metadata:
- **MessageId**: unique identifier for deduplication and tracing
- **CorrelationId**: links related messages across a workflow
- **SessionId**: groups messages for ordered processing
- **ReplyTo**: indicates where responses should be sent
- **ContentType**: describes the payload format
- **Subject**: application-defined label for routing or diagnostics
- **Application properties**: custom key-value metadata for business context

Example:

```csharp
var msg = new ServiceBusMessage(BinaryData.FromString("{\"orderId\":42}"))
{
    MessageId = "order-42-created",
    CorrelationId = "trace-abc123",
    Subject = "OrderCreated",
    ContentType = "application/json"
};

msg.ApplicationProperties["tenant"] = "contoso";
msg.ApplicationProperties["priority"] = "high";
```

Use metadata to support tracing, routing, and idempotency. Do not put large payloads in properties; keep them small and use the body for the main content.

**Scheduled, deferred, and delayed processing**

Service Bus supports delayed workflows in two different ways.

- **Scheduled messages**: send a message for future delivery
- **Deferred messages**: receive a message later by sequence number after explicitly deferring it

Scheduled message example:

```csharp
var scheduled = new ServiceBusMessage("run later");
DateTimeOffset enqueueTime = DateTimeOffset.UtcNow.AddMinutes(10);

await sender.ScheduleMessageAsync(scheduled, enqueueTime);
```

If you need to cancel a scheduled message before it is delivered, keep the returned sequence number:

```csharp
long sequenceNumber = await sender.ScheduleMessageAsync(scheduled, enqueueTime);
await sender.CancelScheduledMessageAsync(sequenceNumber);
```

Deferred message retrieval example:

```csharp
await receiver.DeferMessageAsync(message);

ServiceBusReceivedMessage deferred =
    await receiver.ReceiveDeferredMessageAsync(message.SequenceNumber);
```

Use deferred messages when you want to temporarily set a message aside without losing it. Use scheduled messages when you want the broker to release the message at a future time.

**Dead-lettering and poison message handling**

A message can be dead-lettered by the application or by the broker. Common broker reasons include:
- Exceeded delivery count
- Expired TTL (time to live)
- Invalid session or lock conditions
- Message size or format issues

Application dead-lettering is useful when the message is valid but cannot be processed by business rules.

Example:

```csharp
await args.DeadLetterMessageAsync(
    args.Message,
    deadLetterReason: "ValidationFailed",
    deadLetterErrorDescription: "CustomerId is missing");
```

The DLQ is not a retry queue. It is a holding area for messages that need inspection, repair, or manual replay. Build operational tooling to inspect and resubmit messages when appropriate.

**Service Bus topology and entity selection**

Choose the entity type based on the workflow shape:

| Scenario | Recommended entity | Why |
|---|---|---|
| One consumer processes each task | Queue | Simple point-to-point delivery |
| Many consumers need the same event | Topic + subscriptions | Fan-out with independent filters |
| Ordered per-customer processing | Session-enabled queue or subscription | Preserves order within a session |
| Retryable background work | Queue | Durable buffering and backpressure |
| Multiple downstream teams need different subsets | Topic with filters | Each subscription gets its own view |

A topic subscription behaves like a virtual queue. Each subscription can have its own rules, filters, and dead-letter behavior.

**Event Grid fundamentals**

Event Grid is an event routing service that pushes events to subscribers. It is optimized for lightweight notification and fan-out, not for durable command processing.

Core concepts:
- **Event source**: the system that emits events
- **Topic**: the endpoint that receives events for distribution
- **Subscription**: the delivery configuration for a subscriber
- **Event handler**: the destination that receives the event
- **Event schema**: the payload format used for delivery

Event Grid supports several event source patterns:
- **System topics**: Azure service events such as storage, resource groups, or subscriptions
- **Custom topics**: your application publishes custom events
- **Domain topics**: a grouping model for large-scale event distribution, where supported

Use **system topics** for Azure resource events, **custom topics** for application events, and **domain topics** when you need a multi-tenant or large-scale topic organization model.

**Event schema: Event Grid schema vs CloudEvents**

Event Grid can deliver events in the native Event Grid schema or in the **CloudEvents** schema.

- **Event Grid schema** is the original Azure event format
- **CloudEvents** is a CNCF standard that improves portability across platforms

Event Grid schema example:

```json
[
  {
    "id": "1",
    "eventType": "Contoso.Items.Created",
    "subject": "/items/42",
    "eventTime": "2026-04-26T12:00:00Z",
    "data": {
      "itemId": 42
    },
    "dataVersion": "1.0",
    "metadataVersion": "1"
  }
]
```

CloudEvents example:

```json
[
  {
    "specversion": "1.0",
    "type": "Contoso.Items.Created",
    "source": "/contoso/items",
    "id": "1",
    "time": "2026-04-26T12:00:00Z",
    "subject": "42",
    "datacontenttype": "application/json",
    "data": {
      "itemId": 42
    }
  }
]
```

Choose CloudEvents when you want interoperability with non-Azure systems or standards-based tooling. Choose the native schema when you are working primarily within Azure and want the classic Event Grid format.

**Event Grid delivery, retries, and dead-lettering**

Event Grid uses push delivery with retries. If a subscriber endpoint fails, Event Grid retries according to the subscription delivery policy. Delivery is at-least-once, so duplicate events are possible.

Practical delivery knobs include:
- **Max delivery attempts**: how many times Event Grid retries
- **Event time-to-live**: how long Event Grid keeps retrying
- **Dead-letter destination**: where undeliverable events are stored

Dead-lettering requires a configured storage destination. When enabled, Event Grid writes undeliverable events to a storage account container for later inspection.

Example subscription creation with retry and dead-letter settings:

```bash
az eventgrid event-subscription create \
  --name orders-sub \
  --source-resource-id /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.EventGrid/topics/orders-topic \
  --endpoint https://contoso.example.com/api/events \
  --max-delivery-attempts 10 \
  --event-ttl 1440 \
  --deadletter-endpoint /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<storageAccount>/blobServices/default/containers/<container>
```

> Dead-lettering only works when you configure a valid storage destination. A webhook endpoint is not a dead-letter target.

**Event Grid filtering and routing**

Event Grid subscriptions can filter events so only relevant messages reach a handler. Filtering reduces noise and helps route events to specialized consumers.

Common filter types:
- **Subject filters**: match on the event subject path
- **Advanced filters**: match on event fields such as `data`, `eventType`, or `subject`
- **Included event types**: limit delivery to selected event types

Subject filtering example:

```bash
az eventgrid event-subscription create \
  --name images-only \
  --source-resource-id /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.EventGrid/topics/app-topic \
  --endpoint https://contoso.example.com/api/events \
  --subject-begins-with /images/ \
  --subject-ends-with .png
```

Advanced filtering example:

```bash
az eventgrid event-subscription create \
  --name high-priority-orders \
  --source-resource-id /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.EventGrid/topics/app-topic \
  --endpoint https://contoso.example.com/api/events \
  --advanced-filter data.priority StringIn high critical
```

Use subject filters for path-like routing and advanced filters for business attributes. Keep filters simple and predictable so subscriptions remain easy to reason about.

**Event Grid webhook validation**

When you subscribe a webhook endpoint, Event Grid validates that the endpoint can receive events. Validation behavior depends on the delivery mode and schema.

Common validation patterns:
- **Header-based validation**: Event Grid sends a validation request with `aeg-event-type: SubscriptionValidation`
- **Event payload validation**: Event Grid sends a validation event in the request body, typically as an array
- **CloudEvents handshake**: validation follows the CloudEvents delivery pattern for the chosen transport

A production-ready webhook should inspect the request and respond correctly to validation before processing normal events.

Example ASP.NET Core endpoint:

```csharp
app.MapPost("/api/events", async (HttpRequest request) =>
{
    using var reader = new StreamReader(request.Body);
    var body = await reader.ReadToEndAsync();

    if (request.Headers.TryGetValue("aeg-event-type", out var aegType) &&
        aegType == "SubscriptionValidation")
    {
        var events = JsonSerializer.Deserialize<List<JsonElement>>(body);
        var validationCode = events?[0].GetProperty("data").GetProperty("validationCode").GetString();

        return Results.Json(new
        {
            validationResponse = validationCode
        });
    }

    var eventGridEvents = JsonSerializer.Deserialize<List<JsonElement>>(body);

    foreach (var ev in eventGridEvents ?? [])
    {
        Console.WriteLine(ev.ToString());
    }

    return Results.Ok();
});
```

For CloudEvents, validate according to the CloudEvents transport and payload shape you configure. Do not assume every Event Grid subscription uses the same validation payload.

**Event Grid custom topics, system topics, and domain topics**

These topic types solve different publishing models:

- **Custom topic**: your application publishes events directly
- **System topic**: Azure service emits events from a resource
- **Domain topic**: a higher-level grouping for many related topics or tenants

Use a custom topic when your application is the event source. Use a system topic when Azure is the source. Use a domain topic when you need to manage many event sources or tenants under one namespace model.

Example custom topic publish with the Azure SDK:

```csharp
using Azure;
using Azure.Messaging.EventGrid;

var client = new EventGridPublisherClient(
    new Uri(topicEndpoint),
    new AzureKeyCredential(topicKey));

var events = new[]
{
    new EventGridEvent(
        subject: "orders/42",
        eventType: "Contoso.Orders.Created",
        dataVersion: "1.0",
        data: new { orderId = 42, total = 99.95m })
};

await client.SendEventsAsync(events);
```

**Choosing between Service Bus and Event Grid**

Use the following decision points:
- Choose **Service Bus** when the consumer must process every message reliably and may need retries, locks, sessions, or deferred handling
- Choose **Event Grid** when you want to notify many subscribers about something that happened
- Choose **Service Bus** for commands and work items
- Choose **Event Grid** for events and notifications

A practical pattern is:
1. An application writes a business event to Event Grid.
2. A subscriber validates the event and enqueues a command into Service Bus.
3. A worker processes the command from Service Bus with retries and dead-lettering.

This pattern separates notification from durable processing.

**Practical Service Bus CLI and SDK setup**

Create a namespace and queue with Azure CLI:

```bash
az servicebus namespace create \
  --resource-group <rg> \
  --name <namespaceName> \
  --location eastus

az servicebus queue create \
  --resource-group <rg> \
  --namespace-name <namespaceName> \
  --name orders \
  --enable-partitioning true \
  --max-delivery-count 10
```

Create a topic and subscription:

```bash
az servicebus topic create \
  --resource-group <rg> \
  --namespace-name <namespaceName> \
  --name orders-topic

az servicebus topic subscription create \
  --resource-group <rg> \
  --namespace-name <namespaceName> \
  --topic-name orders-topic \
  --name billing
```

Add a subscription rule filter:

```bash
az servicebus topic subscription rule create \
  --resource-group <rg> \
  --namespace-name <namespaceName> \
  --topic-name orders-topic \
  --subscription-name billing \
  --name high-value \
  --filter-sql-expression "total > 1000"
```

Use SQL filters when you need broker-side routing based on message properties.

**Practical Event Grid CLI and routing setup**

Create a custom topic:

```bash
az eventgrid topic create \
  --name orders-topic \
  --resource-group <rg> \
  --location eastus
```

Create a subscription to a webhook:

```bash
az eventgrid event-subscription create \
  --name orders-webhook \
  --source-resource-id /subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.EventGrid/topics/orders-topic \
  --endpoint https://contoso.example.com/api/events
```

If you need to route only a subset of events, add filters at creation time. Keep the endpoint idempotent because Event Grid can retry and duplicate deliveries can occur.

**Operational considerations**

Design for failure and replay:
- Make consumers **idempotent**
- Log correlation identifiers for tracing
- Use dead-letter queues and dead-letter storage for inspection
- Monitor delivery failures, lock losses, and retry counts
- Tune concurrency and prefetch based on throughput and latency needs

For Service Bus, watch:
- Delivery count
- Dead-letter queue depth
- Lock lost exceptions
- Session lock contention
- Auto-lock renewal duration

For Event Grid, watch:
- Delivery success rate
- Subscription validation failures
- Retry exhaustion
- Dead-letter storage growth
- Endpoint latency and HTTP status codes

**Common exam scenarios**

- You need guaranteed processing of a purchase order even if the worker is offline: use **Service Bus queue**
- You need multiple independent consumers to react to a file upload: use **Event Grid system topic**
- You need ordered processing per customer: use **Service Bus sessions**
- You need to route only high-priority events to one subscriber: use **Event Grid advanced filters** or **Service Bus SQL filters** depending on the broker
- You need to inspect poison messages later: use the **Service Bus DLQ**
- You need to store undeliverable Event Grid events: configure **Event Grid dead-lettering** to a storage container

**Key takeaways**

Service Bus gives you durable, brokered messaging with explicit control over settlement, retries, ordering, and dead-letter handling. Event Grid gives you scalable event distribution with filtering, push delivery, and integration with Azure resource events. In AZ-204 scenarios, the correct choice usually depends on whether you are moving work items reliably or broadcasting notifications reactively.
