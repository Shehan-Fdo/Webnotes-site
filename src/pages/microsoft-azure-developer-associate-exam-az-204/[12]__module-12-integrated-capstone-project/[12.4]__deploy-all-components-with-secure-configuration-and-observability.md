## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Scope and deployment mindset**

This topic is about shipping application components with the right security boundaries, configuration sources, telemetry, and operational checks. In AZ-204 scenarios, you are usually asked to choose the right Azure services and settings so the app can run safely in production, be monitored effectively, and be diagnosed without redeploying.

A good deployment design separates concerns:

- **Secrets** live in Azure Key Vault, not in code or plain app settings.
- **Configuration** lives in Azure App Configuration or environment-specific settings.
- **Telemetry** flows to Azure Monitor and Application Insights.
- **Access** is controlled with Microsoft Entra ID, managed identities, role assignments, and network restrictions.
- **Health** is exposed through endpoints that platform services can probe.

**Secure configuration fundamentals**

Secure configuration starts with identity-first access. Prefer managed identity for Azure-hosted workloads so the app can authenticate to Azure services without storing credentials. Use Microsoft Entra ID for human and service access where possible, and assign the minimum required permissions.

Common secure deployment principles:

- **Least privilege**: grant only the roles needed for the task.
- **Secret minimization**: avoid connection strings when token-based auth is available.
- **Environment isolation**: separate dev, test, and prod resources and settings.
- **Network restriction**: reduce public exposure of dependent services.
- **Configuration externalization**: keep environment-specific values outside the codebase.

A practical pattern is:

- App Service or Azure Functions uses a managed identity.
- The identity reads secrets from Key Vault.
- The app reads non-secret settings from App Configuration.
- Application Insights collects traces, requests, dependencies, and exceptions.
- Azure Monitor alerts notify operators when thresholds are crossed.

**Identity and access control**

Managed identities are the preferred way for Azure resources to access other Azure resources. A system-assigned identity is tied to one resource, while a user-assigned identity can be shared across multiple resources.

Use role-based access control (**RBAC**) for Azure resource management and for data-plane access where the service supports it. For Key Vault, choose one vault-level authorization model for data-plane access:

- **Azure RBAC** mode for Key Vault data access
- **Access policy** mode for Key Vault data access

> Do not assume both models are active for the same vault data-plane path. Pick the model your design uses and assign permissions accordingly.

Typical RBAC examples:

- Grant a web app identity `Key Vault Secrets User` to read secrets.
- Grant a deployment pipeline `Contributor` or a narrower custom role only where needed.
- Grant operators `Monitoring Reader` or `Reader` for observability access.

Azure CLI examples:

```bash
# Get the principal ID of a system-assigned managed identity
principalId=$(az webapp identity show \
  --resource-group rg-app \
  --name my-webapp \
  --query principalId -o tsv)

# Assign Key Vault secret read access using Azure RBAC
az role assignment create \
  --assignee-object-id "$principalId" \
  --assignee-principal-type ServicePrincipal \
  --role "Key Vault Secrets User" \
  --scope $(az keyvault show -g rg-sec -n my-kv --query id -o tsv)
```

For a user-assigned managed identity, use that identity’s service principal object ID. In many CLI flows, `principalId` is the value you need for the role assignment.

**Key Vault for secrets, certificates, and keys**

Azure Key Vault stores secrets, keys, and certificates. In AZ-204 scenarios, it is commonly used for:

- Database connection strings
- API keys
- TLS certificates
- Signing keys
- App secrets that should not be committed to source control

Key Vault design points:

- Use separate vaults per environment when isolation matters.
- Prefer private network access when the workload is private.
- Enable soft delete and purge protection where supported and required by policy.
- Use RBAC or access policies consistently for the vault’s data-plane model.
- Rotate secrets and keys on a schedule.

Secret versioning matters because applications should not depend on a single immutable value. A secret can have multiple versions, and the latest version is typically returned when you reference the secret name without a version.

Practical rotation pattern:

- Store the new secret as a new version.
- Update the application to read the secret by name, not by version, when appropriate.
- Allow the app to refresh configuration or restart gracefully.
- Retire the old version after validation.

Example secret retrieval with Azure CLI:

```bash
az keyvault secret set \
  --vault-name my-kv \
  --name "SqlConnectionString" \
  --value "Server=tcp:myserver.database.windows.net;Database=mydb;User ID=myuser;Password=***"

az keyvault secret show \
  --vault-name my-kv \
  --name "SqlConnectionString" \
  --query value -o tsv
```

In application code, prefer managed identity plus Key Vault SDK or configuration provider integration rather than embedding secrets in environment variables.

**App Configuration for centralized settings**

Azure App Configuration centralizes application settings, feature flags, and environment-specific values. It is useful when you want to change configuration without redeploying the app.

Use App Configuration for:

- Non-secret settings
- Feature flags
- Environment overrides
- Shared configuration across multiple apps
- Dynamic refresh scenarios

App Configuration does not automatically refresh values in every app. You must configure refresh behavior in the application and, for web apps, ensure the request pipeline triggers refresh when needed.

.NET example with Azure App Configuration and Key Vault references:

```csharp
using Azure.Identity;

var builder = WebApplication.CreateBuilder(args);

builder.Configuration.AddAzureAppConfiguration(options =>
{
    options.Connect(new Uri(builder.Configuration["AppConfig:Endpoint"]!), new DefaultAzureCredential())
           .ConfigureKeyVault(kv => kv.SetCredential(new DefaultAzureCredential()))
           .Select("MyApp:*")
           .Select("MyApp:Production:*", label: "Production")
           .UseFeatureFlags();
});

builder.Services.AddAzureAppConfiguration();

var app = builder.Build();

app.UseAzureAppConfiguration();

app.MapGet("/", (IConfiguration config) =>
{
    return Results.Ok(new
    {
        Message = config["MyApp:Message"]
    });
});

app.Run();
```

For runtime refresh, register the keys you want to monitor and set a cache expiration interval. The app must also call the middleware or refresh logic during requests.

```csharp
builder.Configuration.AddAzureAppConfiguration(options =>
{
    options.Connect(new Uri(endpoint), new DefaultAzureCredential())
           .ConfigureRefresh(refresh =>
           {
               refresh.Register("MyApp:Sentinel", refreshAll: true)
                      .SetCacheExpiration(TimeSpan.FromSeconds(30));
           });
});
```

Key App Configuration behaviors:

- **Sentinel key**: a single key that signals a broader config refresh.
- **Cache expiration**: controls how often the app checks for updates.
- **Key Vault references**: App Configuration can store references to Key Vault secrets, but the app still needs permission to resolve them.
- **Feature flags**: useful for gradual rollout and safe toggles.

A common exam distinction is that `AddAzureAppConfiguration(...)` alone is not enough for live refresh. You need refresh registration and pipeline integration.

**Network security for dependent services**

Secure deployments often combine identity with network controls. Identity answers “who can access,” while network controls answer “from where can access happen.”

Common network controls:

- **Private endpoints** for private access to PaaS services
- **Service endpoints** for subnet-based access to supported services
- **Firewall rules** to restrict public access by IP or network
- **Disable public network access** where the service supports it
- **VNet integration** for App Service outbound access to private resources

Use private endpoints when you want the service to be reachable only through a private IP in your virtual network. This is common for Key Vault, Storage, SQL Database, and App Configuration in locked-down environments.

Example deployment considerations:

- App Service uses VNet integration for outbound calls to private endpoints.
- DNS must resolve the private endpoint name correctly.
- The dependent service should have public access disabled if the design requires full isolation.
- Managed identity still handles authentication even when the network path is private.

**Observability overview**

Observability combines telemetry collection, analysis, alerting, and visualization. In Azure, the main building blocks are:

- **Application Insights** for application telemetry
- **Azure Monitor** for metrics, logs, alerts, and dashboards
- **Log Analytics workspace** for centralized log storage and querying
- **Workbooks** for interactive visual analysis
- **Dashboards** for pinned operational views

A strong observability design answers:

- Is the app healthy?
- Is it slow?
- Is it failing?
- Which dependency is causing the issue?
- Did a deployment change behavior?

**Application Insights and OpenTelemetry**

Application Insights is the common Azure application monitoring service. Modern .NET apps can use either the classic Application Insights SDK or OpenTelemetry-based instrumentation, depending on the runtime and team preference.

Use classic SDK when:

- You need a mature, direct Application Insights integration.
- You are maintaining an existing app already using the SDK.
- You want the simplest path for common telemetry in supported frameworks.

Use OpenTelemetry when:

- You want vendor-neutral instrumentation.
- You plan to export to multiple back ends.
- You are standardizing on traces, metrics, and logs across platforms.

Typical telemetry setup includes:

- Connection string or instrumentation key equivalent for Application Insights
- Automatic request and dependency tracking
- Exception capture
- Custom events and metrics
- Correlation across distributed services

.NET example using Application Insights connection string:

```csharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddApplicationInsightsTelemetry(options =>
{
    options.ConnectionString = builder.Configuration["APPLICATIONINSIGHTS_CONNECTION_STRING"];
});

var app = builder.Build();
app.MapGet("/", () => "OK");
app.Run();
```

OpenTelemetry example for ASP.NET Core:

```csharp
using OpenTelemetry.Metrics;
using OpenTelemetry.Resources;
using OpenTelemetry.Trace;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddOpenTelemetry()
    .ConfigureResource(resource => resource.AddService("orders-api"))
    .WithTracing(tracing =>
    {
        tracing.AddAspNetCoreInstrumentation()
               .AddHttpClientInstrumentation()
               .AddOtlpExporter();
    })
    .WithMetrics(metrics =>
    {
        metrics.AddAspNetCoreInstrumentation()
               .AddRuntimeInstrumentation()
               .AddOtlpExporter();
    });

var app = builder.Build();
app.MapGet("/", () => "OK");
app.Run();
```

Practical guidance:

- Use a consistent service name so traces are easy to filter.
- Correlate logs with request IDs and operation IDs.
- Capture dependency telemetry for SQL, HTTP, and messaging calls.
- Add custom telemetry for business events that matter operationally.

**Azure Monitor logs, metrics, and alerts**

Azure Monitor collects platform and application telemetry. It supports both **metrics** and **logs**, and the choice matters.

- **Metrics** are near real-time, numeric, and good for fast alerting.
- **Logs** are richer, queryable, and better for investigation and correlation.

Use metric alerts when:

- You need low-latency detection.
- The signal is numeric and well-defined.
- You want to alert on CPU, memory, request count, or failure rate.

Use log alerts when:

- You need KQL (Kusto Query Language) queries over structured logs.
- You need to detect patterns, joins, or complex conditions.
- You want to alert on application-specific events or exceptions.

Common alert rule types:

- **Metric alert**
- **Log query alert**
- **Activity log alert**
- **Smart detection** or service-specific alerts where available

Alert rule setup usually includes:

- **Scope**: the resource or resource group being monitored
- **Condition**: the metric threshold or log query
- **Action group**: the notification or automation target
- **Severity**: the operational priority
- **Evaluation frequency** and **window size**

Example metric alert CLI:

```bash
az monitor metrics alert create \
  --name "High-5xx-Rate" \
  --resource-group rg-app \
  --scopes $(az webapp show -g rg-app -n my-webapp --query id -o tsv) \
  --condition "avg Http5xx > 5" \
  --window-size 5m \
  --evaluation-frequency 1m \
  --action-group "/subscriptions/<subId>/resourceGroups/rg-monitor/providers/microsoft.insights/actionGroups/ag-ops"
```

Example log alert query:

```kusto
requests
| where success == false
| summarize FailedRequests = count() by bin(timestamp, 5m), cloud_RoleName
| where FailedRequests > 10
```

Action groups can notify by:

- Email
- SMS
- Push notification
- Voice call
- Webhook
- Azure Function
- Logic App
- Automation Runbook

**Workbooks and dashboards**

Workbooks and dashboards help you visualize telemetry and operational state.

Use **workbooks** when you need:

- Interactive filtering
- Parameterized views
- Deep investigation
- Combined metrics and logs

Use **dashboards** when you need:

- A shared operational overview
- Pinned charts and tiles
- A quick status view for teams or leadership

Workbook examples:

- Deployment health by region
- Error rate by endpoint
- Dependency latency over time
- Failed requests correlated with recent releases

A workbook is often better than a static dashboard when you need drill-down and ad hoc analysis.

**Diagnostics and logging for Azure resources**

Application telemetry is only part of the picture. Azure resources also emit platform logs and metrics that you should route to a Log Analytics workspace, storage account, or event hub depending on the use case.

Common diagnostic settings targets:

- **Log Analytics workspace** for querying and alerting
- **Storage account** for long-term retention
- **Event Hub** for SIEM or external analytics integration

Resources commonly configured with diagnostic settings:

- App Service
- Azure Functions
- Key Vault
- Storage accounts
- Azure SQL Database
- API Management
- Application Gateway

Example CLI for diagnostic settings:

```bash
az monitor diagnostic-settings create \
  --name "send-to-law" \
  --resource $(az webapp show -g rg-app -n my-webapp --query id -o tsv) \
  --workspace $(az monitor log-analytics workspace show -g rg-monitor -n law-prod --query id -o tsv) \
  --logs '[{"category":"AppServiceHTTPLogs","enabled":true},{"category":"AppServiceConsoleLogs","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

Bicep example:

```bicep
resource law 'Microsoft.OperationalInsights/workspaces@2023-09-01' existing = {
  name: 'law-prod'
}

resource app 'Microsoft.Web/sites@2023-12-01' existing = {
  name: 'my-webapp'
}

resource diag 'Microsoft.Insights/diagnosticSettings@2021-05-01-preview' = {
  name: 'send-to-law'
  scope: app
  properties: {
    workspaceId: law.id
    logs: [
      {
        category: 'AppServiceHTTPLogs'
        enabled: true
      }
      {
        category: 'AppServiceConsoleLogs'
        enabled: true
      }
    ]
    metrics: [
      {
        category: 'AllMetrics'
        enabled: true
      }
    ]
  }
}
```

**Health checks and deployment readiness**

Health checks let the platform determine whether an app instance is ready to receive traffic. They are especially important for App Service, deployment slots, and containerized workloads.

Use health endpoints to distinguish:

- **Readiness**: can the app accept traffic now?
- **Liveness**: is the process still running and responsive?

In App Service, a health check path can be configured so the platform removes unhealthy instances from rotation. In deployment slot scenarios, health checks help validate a slot before swap.

Practical health endpoint guidance:

- Keep the endpoint lightweight.
- Check only critical dependencies.
- Return a clear success or failure status.
- Avoid expensive downstream calls that can create false negatives.

Example ASP.NET Core health checks:

```csharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddHealthChecks()
    .AddSqlServer(builder.Configuration.GetConnectionString("Sql"))
    .AddAzureKeyVault(new Uri(builder.Configuration["KeyVault:Uri"]!), new DefaultAzureCredential());

var app = builder.Build();

app.MapHealthChecks("/health");
app.MapHealthChecks("/ready");

app.Run();
```

Operational notes:

- Use readiness checks for traffic routing decisions.
- Use liveness checks for restart decisions in container platforms.
- In deployment slots, validate the slot before swapping to production.
- In Functions, expose health only if your hosting model and architecture require it; not every function app needs a custom health endpoint.

**Azure Functions observability and deployment considerations**

Azure Functions can emit telemetry to Application Insights and Azure Monitor. The exact configuration depends on the runtime, hosting plan, and monitoring stack.

Key points:

- Configure Application Insights or OpenTelemetry early in the project.
- Use structured logging with `ILogger`.
- Correlate function executions with dependency calls.
- Review cold start and timeout behavior in telemetry.
- Use host-level logs and function-level logs together for troubleshooting.

Example function logging:

```csharp
public class OrderFunction
{
    private readonly ILogger<OrderFunction> _logger;

    public OrderFunction(ILogger<OrderFunction> logger)
    {
        _logger = logger;
    }

    [Function("OrderFunction")]
    public async Task Run([HttpTrigger(AuthorizationLevel.Function, "post")] HttpRequestData req)
    {
        _logger.LogInformation("Order request received at {Time}", DateTimeOffset.UtcNow);
        await Task.CompletedTask;
    }
}
```

For current observability guidance, prefer the monitoring approach recommended for your Functions runtime and hosting model rather than relying on a single legacy `host.json` pattern.

**Governance with Azure Policy**

Azure Policy helps enforce standards and assess compliance across subscriptions and resource groups. It is useful for secure deployment because it can prevent or flag noncompliant resources before they reach production.

Use Azure Policy to:

- Require tags
- Restrict allowed SKUs or regions
- Enforce private endpoint usage
- Deny public network access on supported services
- Require diagnostic settings
- Audit or deny insecure configurations

Policy concepts:

- **Definition**: the rule
- **Assignment**: where the rule applies
- **Initiative**: a collection of related policies
- **Compliance state**: whether resources meet the rule

Example policy assignment flow:

1. Select or create a policy definition.
2. Assign it to a subscription, management group, or resource group.
3. Review compliance results.
4. Remediate noncompliant resources where possible.

CLI example:

```bash
az policy assignment create \
  --name "require-diagnostics" \
  --scope "/subscriptions/<subId>" \
  --policy "/providers/Microsoft.Authorization/policyDefinitions/<policyDefinitionId>"
```

Initiatives are useful when you want to bundle multiple controls, such as:

- Require diagnostic settings
- Deny public access
- Require tags
- Enforce approved regions

Policy works well with deployment automation because it can validate infrastructure as code deployments and surface compliance drift after deployment.

**Infrastructure as code and secure deployment**

Bicep and ARM templates are common for repeatable deployments. Use them to codify security, observability, and configuration.

Good IaC practices:

- Parameterize environment-specific values.
- Reference existing resources carefully.
- Avoid hardcoding secrets.
- Deploy diagnostic settings alongside the resource.
- Use modules for repeated patterns.

Bicep example for App Service with managed identity and app settings:

```bicep
param location string = resourceGroup().location
param appName string
param appConfigEndpoint string

resource plan 'Microsoft.Web/serverfarms@2023-12-01' = {
  name: '${appName}-plan'
  location: location
  sku: {
    name: 'P1v3'
    tier: 'PremiumV3'
  }
}

resource app 'Microsoft.Web/sites@2023-12-01' = {
  name: appName
  location: location
  identity: {
    type: 'SystemAssigned'
  }
  properties: {
    serverFarmId: plan.id
    siteConfig: {
      appSettings: [
        {
          name: 'AppConfig__Endpoint'
          value: appConfigEndpoint
        }
      ]
    }
  }
}
```

Be aware that API versions and SKU names can vary by region and service support. Validate templates in your target environment before treating them as portable.

**Common exam scenarios and how to reason about them**

When you see a deployment question, identify the operational goal first:

- If the question is about secret storage, choose Key Vault.
- If it is about dynamic configuration, choose App Configuration.
- If it is about telemetry, choose Application Insights and Azure Monitor.
- If it is about access control, choose managed identity and RBAC.
- If it is about network isolation, choose private endpoints and firewall rules.
- If it is about compliance, choose Azure Policy.
- If it is about app readiness, choose health checks.

Typical scenario patterns:

- A web app needs to read a database password without storing it in code.
  - Use managed identity plus Key Vault.
- A feature flag must be changed without redeploying.
  - Use App Configuration feature flags and refresh.
- A service is failing intermittently and you need root cause analysis.
  - Use Application Insights traces, dependencies, and logs.
- A production app must alert on high failure rate.
  - Use a metric alert or log alert with an action group.
- A resource must not expose public endpoints.
  - Use private endpoints, disable public access, and enforce with policy.

**Quick comparison of observability tools**

| Tool | Best for | Notes |
|---|---|---|
| Application Insights | App-level telemetry | Requests, dependencies, exceptions, custom events |
| Azure Monitor metrics | Fast numeric alerting | Near real-time and low overhead |
| Azure Monitor logs | Deep investigation | Query with KQL and correlate events |
| Workbooks | Interactive analysis | Good for drill-down and operational views |
| Dashboards | Shared overview | Good for pinned status tiles |

**Practical deployment checklist**

Before you call a deployment complete, verify:

1. The app uses managed identity where possible.
2. Secrets are stored in Key Vault.
3. Non-secret settings are externalized in App Configuration or app settings.
4. Key Vault and dependent services have the correct network restrictions.
5. Telemetry is flowing to Application Insights and Log Analytics.
6. Diagnostic settings are enabled for key Azure resources.
7. Alerts have scopes, conditions, and action groups.
8. Health endpoints are configured and tested.
9. Azure Policy covers required compliance controls.
10. The app can refresh configuration or secrets without a full redeploy when needed.

**Troubleshooting cues**

If telemetry is missing:

- Check the connection string or exporter configuration.
- Confirm the app has outbound network access.
- Verify the identity has permission to read Key Vault references.
- Check sampling settings and log levels.

If configuration changes do not appear:

- Confirm refresh registration is configured.
- Verify the sentinel key or watched keys changed.
- Check cache expiration.
- Ensure the middleware or refresh call runs in the request path.

If Key Vault access fails:

- Confirm the identity object ID used in the role assignment is correct.
- Verify the vault authorization model matches the permission model you assigned.
- Check private DNS and network access if the vault is private.

If alerts do not fire:

- Confirm the alert scope is correct.
- Check the threshold, aggregation, and evaluation window.
- Verify the action group is attached and enabled.
- Make sure the signal type matches the rule type.
