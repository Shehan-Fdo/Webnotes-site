## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Capstone architecture goals**

A capstone architecture for AZ-204 should demonstrate that you can design a solution that is secure, observable, deployable, and maintainable. The goal is not to maximize the number of Azure services, but to choose a small set of services that work well together and support the application’s business requirements.

A strong design usually separates concerns into layers:

- **Client layer** for web, mobile, or API consumers
- **Application layer** for APIs, background workers, and orchestration
- **Integration layer** for messaging, events, and external systems
- **Data layer** for relational, NoSQL, or object storage
- **Operations layer** for monitoring, secrets, and deployment

The architecture should also reflect common Azure developer patterns:

- Use managed identity instead of storing credentials in code.
- Use Azure Key Vault for secrets when a secret is unavoidable.
- Use Application Insights and Azure Monitor for telemetry.
- Use Infrastructure as Code (IaC) so environments are reproducible.
- Use separate environments such as `dev`, `test`, and `prod`.

**Typical capstone topology**

A common AZ-204 capstone combines a frontend, an API, asynchronous processing, and persistent storage. One practical topology is:

- Static web app or App Service for the UI
- Azure App Service or Azure Functions for the API
- Azure Service Bus for decoupled messaging
- Azure Storage for blobs, queues, or files
- Azure SQL Database or Cosmos DB for application data
- Key Vault for secrets and certificates
- Application Insights for logs, metrics, and traces

This kind of design shows that you understand both synchronous request/response flows and asynchronous workflows.

```text
User -> Web UI -> API -> Data store
                   |
                   v
             Service Bus queue/topic -> Worker -> Database/Storage
```

**Choosing the right Azure services**

Service selection should follow the workload, not preference. Use the service that best matches the access pattern, scaling model, and operational needs.

| Need | Common Azure choice | Why it fits |
|---|---|---|
| Public HTTP API | Azure App Service or Azure Functions | Simple hosting for REST endpoints |
| Event-driven processing | Azure Functions | Triggers respond to queue, timer, or event sources |
| Durable background work | WebJob, Functions, or container worker | Handles long-running or asynchronous tasks |
| Relational data | Azure SQL Database | Supports transactions and SQL querying, with Azure-specific service limits |
| Document or key-value data | Azure Cosmos DB | Flexible schema and global distribution |
| File/blob storage | Azure Blob Storage | Cheap, scalable object storage |
| Decoupled messaging | Azure Service Bus | Reliable queues and topics with dead-lettering |
| Secrets | Azure Key Vault | Centralized secret and key management |
| Telemetry | Application Insights | Request, dependency, and exception tracking |

> Azure SQL Database is not the same as SQL Server on-premises. It supports many familiar relational features, but you should still account for service-specific limits and platform behavior.

**Designing for security**

Security should be built into the architecture from the start. In AZ-204 scenarios, the best answer often removes secrets rather than protecting them after the fact.

Use these patterns:

- **Managed identity** for Azure-to-Azure authentication
- **Role-based access control (RBAC)** for authorization
- **Key Vault references** or SDK access for secrets
- **Private endpoints** when network isolation is required
- **HTTPS only** for public endpoints
- **Least privilege** for service principals, identities, and users

For example, if an App Service needs to read from Key Vault, assign the app a system-assigned managed identity and grant it only the required Key Vault permissions or RBAC role.

```bash
az webapp identity assign \
  --resource-group rg-capstone-dev \
  --name app-capstone-api

az role assignment create \
  --assignee-object-id <principal-id> \
  --assignee-principal-type ServicePrincipal \
  --role "Key Vault Secrets User" \
  --scope /subscriptions/<sub-id>/resourceGroups/rg-capstone-dev/providers/Microsoft.KeyVault/vaults/kv-capstone-dev
```

**Identity and access considerations**

When designing access, distinguish between authentication and authorization:

- **Authentication** proves who the caller is.
- **Authorization** determines what the caller can do.

For Azure services, managed identity is usually the preferred authentication mechanism. For user access, Microsoft Entra ID is the standard identity provider. For service-to-service access, avoid embedding connection strings or account keys unless the service does not support identity-based access.

Service Bus access is a common exam topic:

- Azure RBAC data-plane roles for Service Bus can be assigned at the **namespace**, **resource group**, or **subscription** scope.
- They are **not** assigned at the individual queue or topic entity scope.
- Use the least privileged role that matches the task, such as sender, receiver, or owner permissions.

**Repository structure principles**

A good repository structure makes the solution easy to build, test, and deploy. The structure should reflect the architecture and support independent deployment of components.

Common goals:

- Keep application code separate from infrastructure code.
- Keep shared libraries isolated from service-specific code.
- Make environment-specific configuration explicit.
- Make CI/CD pipelines easy to discover.
- Keep tests close enough to the code to be maintainable, but not mixed into production folders.

A practical monorepo layout for a capstone might look like this:

```text
repo/
├─ src/
│  ├─ api/
│  │  ├─ Api.csproj
│  │  ├─ Controllers/
│  │  ├─ Services/
│  │  └─ Program.cs
│  ├─ worker/
│  │  ├─ Worker.csproj
│  │  ├─ Functions/
│  │  └─ Program.cs
│  ├─ web/
│  │  ├─ package.json
│  │  └─ src/
│  └─ shared/
│     └─ Shared.Contracts.csproj
├─ tests/
│  ├─ api.tests/
│  ├─ worker.tests/
│  └─ integration.tests/
├─ infra/
│  ├─ main.bicep
│  ├─ modules/
│  └─ parameters/
├─ pipelines/
│  ├─ azure-pipelines.yml
│  └─ github-actions.yml
├─ docs/
└─ README.md
```

This layout works well because each concern has a clear home:

- `src/` contains deployable application code.
- `tests/` contains unit, integration, and contract tests.
- `infra/` contains IaC templates and parameter files.
- `pipelines/` contains build and release definitions.
- `docs/` contains architecture diagrams and operational notes.

**Monorepo versus polyrepo**

You can organize the solution as a monorepo or split it into multiple repositories. The right choice depends on team size, release independence, and governance.

| Approach | Best when | Tradeoffs |
|---|---|---|
| Monorepo | Small to medium teams, shared release cadence | Requires discipline to avoid coupling |
| Polyrepo | Independent services, separate ownership, strict boundaries | More overhead for versioning and coordination |

For an AZ-204 capstone, a monorepo is often easier to manage because it keeps the architecture visible in one place and simplifies local development.

**Infrastructure as Code layout**

Infrastructure should be repeatable and parameterized. Bicep is a common choice for AZ-204 because it is concise and integrates well with Azure deployment workflows.

A clean Bicep structure often includes:

- A root file that composes the deployment
- Reusable modules for each resource group or service
- Parameter files for each environment

Example structure:

```text
infra/
├─ main.bicep
├─ main.dev.bicepparam
├─ main.test.bicepparam
├─ main.prod.bicepparam
└─ modules/
   ├─ appservice.bicep
   ├─ functionapp.bicep
   ├─ servicebus.bicep
   ├─ keyvault.bicep
   └─ storage.bicep
```

A simple Bicep deployment command looks like this:

```bash
az deployment group create \
  --resource-group rg-capstone-dev \
  --template-file infra/main.bicep \
  --parameters infra/main.dev.bicepparam
```

When naming Azure resources, do not assume a generic string helper is enough for every resource type. For example, storage account names must be 3-24 characters and contain only lowercase letters and numbers. Validate naming rules per resource.

**Configuration strategy**

Configuration should be externalized so the same build artifact can run in multiple environments. Use environment variables, app settings, and managed identity-based configuration where possible.

Good practices:

- Store environment-specific values in app settings or parameter files.
- Keep secrets out of source control.
- Use consistent naming for settings across services.
- Prefer identity-based connections over connection strings when supported.

Example App Service settings:

```bash
az webapp config appsettings set \
  --resource-group rg-capstone-dev \
  --name app-capstone-api \
  --settings \
    ASPNETCORE_ENVIRONMENT=Development \
    ApplicationInsights__ConnectionString="<connection-string>" \
    ServiceBus__FullyQualifiedNamespace="sb-capstone-dev.servicebus.windows.net"
```

For local development, use `local.settings.json` for Azure Functions or `.env` files for web apps, but keep them out of source control. If a service uses identity-based connections, make sure the local configuration matches the binding model you actually use; do not assume a single setting name works for every binding.

**Local development workflow**

A good repository supports local iteration without requiring full cloud deployment for every change. You should be able to run the API, worker, and tests independently.

Typical local workflow:

1. Restore dependencies.
2. Start supporting services locally or use Azure resources in a dev subscription.
3. Run the API and worker.
4. Execute unit and integration tests.
5. Validate telemetry and configuration.

Example commands:

```bash
dotnet restore
dotnet build
dotnet test

cd src/api
dotnet run

cd src/worker
func start
```

If you use containers, `docker compose` can help you run dependencies such as SQL Server, Azurite, or a local message broker substitute.

```yaml
services:
  api:
    build: ./src/api
    environment:
      ASPNETCORE_ENVIRONMENT: Development
  azurite:
    image: mcr.microsoft.com/azure-storage/azurite
    ports:
      - "10000:10000"
      - "10001:10001"
      - "10002:10002"
```

**Messaging and background processing**

Asynchronous processing is a common capstone requirement because it demonstrates decoupling and resilience. Azure Service Bus is a strong choice when you need reliable delivery, retries, dead-lettering, and ordered processing patterns.

Use Service Bus when:

- The producer should not wait for the consumer.
- You need durable buffering.
- You need competing consumers or pub/sub with topics and subscriptions.
- You need dead-letter handling for poison messages.

Use Azure Functions or a worker service to process messages. With the Azure.Messaging.ServiceBus SDK, a common pattern is to use `ServiceBusClient` with a fully qualified namespace and `DefaultAzureCredential` when identity-based auth is enabled.

```csharp
using Azure.Identity;
using Azure.Messaging.ServiceBus;

var client = new ServiceBusClient(
    "sb-capstone-dev.servicebus.windows.net",
    new DefaultAzureCredential());

ServiceBusSender sender = client.CreateSender("orders");

await sender.SendMessageAsync(new ServiceBusMessage("new order"));
```

For message processing, use a receiver or a processor correctly. `ServiceBusProcessor` uses event handlers rather than direct receive/complete methods.

```csharp
using Azure.Identity;
using Azure.Messaging.ServiceBus;

var client = new ServiceBusClient(
    "sb-capstone-dev.servicebus.windows.net",
    new DefaultAzureCredential());

var processor = client.CreateProcessor("orders");

processor.ProcessMessageAsync += async args =>
{
    string body = args.Message.Body.ToString();
    Console.WriteLine(body);

    await args.CompleteMessageAsync(args.Message);
};

processor.ProcessErrorAsync += args =>
{
    Console.WriteLine(args.Exception.ToString());
    return Task.CompletedTask;
};

await processor.StartProcessingAsync();
```

**Data layer design**

Choose the data store based on access patterns:

- Use **Azure SQL Database** for relational data, joins, and transactional consistency.
- Use **Azure Cosmos DB** for flexible schema, high throughput, and global distribution.
- Use **Azure Blob Storage** for files, images, and large unstructured content.

A capstone often uses more than one data store. For example:

- SQL Database for orders and customers
- Blob Storage for uploaded documents
- Service Bus for order processing events

Keep the data model aligned with the service boundary. Do not force every service to share the same database schema if the architecture is meant to be loosely coupled.

**Observability and diagnostics**

Observability is essential for both development and production. You should be able to answer:

- What happened?
- Where did it fail?
- Which dependency caused the issue?
- How long did it take?

Use Application Insights and Azure Monitor to collect:

- Requests
- Dependencies
- Exceptions
- Traces
- Custom events and metrics

Add correlation IDs so you can trace a request across services. In distributed systems, this is often more useful than raw logs alone.

Example logging in .NET:

```csharp
logger.LogInformation("Processing order {OrderId}", orderId);
logger.LogWarning("Order {OrderId} is delayed", orderId);
logger.LogError(ex, "Failed to process order {OrderId}", orderId);
```

**Deployment strategy**

The repository should support repeatable deployments to multiple environments. A typical flow is:

1. Build and test on pull request.
2. Deploy infrastructure to a dev environment.
3. Deploy application artifacts.
4. Run smoke tests.
5. Promote to test and production with approvals.

Example Azure CLI deployment for a web app artifact:

```bash
az webapp deploy \
  --resource-group rg-capstone-dev \
  --name app-capstone-api \
  --type zip \
  --src-path ./artifacts/api.zip
```

Make sure the path you pass to `--src-path` is correct relative to the current working directory. If you change directories before running the command, adjust the path accordingly.

**Branching and release organization**

A capstone repository should make it easy to understand what is ready, what is in progress, and what is deployed.

Common patterns:

- `main` for production-ready code
- `develop` or feature branches for ongoing work
- pull requests for code review and validation
- tags or release branches for versioned releases

Keep release notes or changelogs if the project is expected to evolve over time. This helps you connect code changes to deployed behavior.

**Testing strategy**

Testing should cover both logic and integration points.

Recommended test layers:

- **Unit tests** for business rules and pure functions
- **Integration tests** for database, storage, and messaging interactions
- **End-to-end tests** for critical user journeys

A practical test pyramid keeps unit tests fast and numerous, while integration tests validate the Azure-specific behavior that unit tests cannot catch.

Example test organization:

```text
tests/
├─ api.tests/
│  ├─ Controllers/
│  └─ Services/
├─ worker.tests/
└─ integration.tests/
   ├─ ServiceBus/
   └─ Storage/
```

**Documentation to include in the repo**

A capstone repository should be self-explanatory. Include documentation that helps another developer or reviewer understand the solution quickly.

Useful files:

- `README.md` with setup, build, and run instructions
- `architecture.md` with service diagrams and design rationale
- `deploy.md` with environment and release steps
- `runbook.md` with troubleshooting and operational guidance

A concise README often includes:

- Prerequisites
- Local setup
- Configuration
- Build and test commands
- Deployment overview
- Troubleshooting tips

**Practical checklist for AZ-204 design questions**

When you evaluate a capstone architecture or repository structure, ask:

- Does the design meet the business requirement with the fewest necessary services?
- Are secrets removed from code and replaced with managed identity where possible?
- Is the repository organized by concern and deployable component?
- Can each environment be deployed reproducibly?
- Are logs, metrics, and traces available for troubleshooting?
- Are asynchronous workloads handled with the right Azure messaging service?
- Are tests included at the right levels?
- Is the solution maintainable by another developer?

**Common mistakes to avoid**

- Mixing infrastructure, application code, and test code in one folder without clear boundaries.
- Hardcoding secrets, connection strings, or environment-specific values.
- Using a database or service that does not match the access pattern.
- Overcomplicating the architecture with unnecessary services.
- Assuming local configuration mirrors cloud authentication automatically.
- Writing deployment scripts that depend on unclear working directories.
- Using SDK samples that do not match the actual client type or method surface.

**Summary**

A strong AZ-204 capstone architecture is simple, secure, and observable. The repository structure should make that architecture easy to build, test, and deploy. Focus on service boundaries, identity-based access, repeatable infrastructure, and clear separation between application code, tests, and deployment assets.
