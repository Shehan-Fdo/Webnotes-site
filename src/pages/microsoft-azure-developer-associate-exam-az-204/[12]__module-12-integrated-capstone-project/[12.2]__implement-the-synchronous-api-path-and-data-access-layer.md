## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Synchronous API path overview**

A synchronous API path means the client sends a request and waits for the response in the same request/response cycle. In ASP.NET Core, that does **not** mean the server must block a single thread for the entire operation; the framework can still use asynchronous I/O internally. The key exam idea is that the client receives the result before continuing, which is appropriate for fast reads, simple writes, and operations that complete quickly.

Use synchronous-style endpoints when the work is short-lived and the caller needs an immediate result. Avoid them for long-running jobs, external batch processing, or workflows that can exceed request timeouts.

- Good fit: lookup endpoints, small CRUD operations, validation checks, and simple projections.
- Poor fit: image processing, report generation, queue-driven workflows, and multi-minute integrations.
- Design goal: keep the request path predictable, fast, and idempotent where possible.

**Controller design for synchronous endpoints**

In ASP.NET Core, controllers usually expose synchronous semantics to the client while using async methods internally for scalability. Prefer `async`/`await` in the controller and service layers even when the endpoint is conceptually synchronous, because this avoids thread starvation under load.

A typical pattern is:
- validate input
- call a service
- map domain data to a response DTO
- return `200 OK`, `201 Created`, `204 No Content`, or an error result

```csharp
[ApiController]
[Route("api/orders")]
public class OrdersController : ControllerBase
{
    private readonly IOrderService _service;

    public OrdersController(IOrderService service)
    {
        _service = service;
    }

    [HttpGet("{id:int}")]
    public async Task<ActionResult<OrderDto>> GetById(int id, CancellationToken cancellationToken)
    {
        var order = await _service.GetByIdAsync(id, cancellationToken);
        if (order is null)
        {
            return NotFound();
        }

        return Ok(order);
    }
}
```

Use `CancellationToken` in controller actions so the request can stop cleanly if the client disconnects or the server times out the request.

**Cancellation, timeouts, and request lifetime**

Cancellation is important in synchronous API paths because the client may abandon the request before the server finishes. Pass the token from the HTTP pipeline into every downstream call that supports it.

- Use `HttpContext.RequestAborted` or an action parameter `CancellationToken`.
- Pass the token into EF Core, Azure SDK calls, and `HttpClient` calls.
- Configure timeouts for external dependencies so the request fails fast instead of hanging.

```csharp
public async Task<OrderDto?> GetByIdAsync(int id, CancellationToken cancellationToken)
{
    return await _dbContext.Orders
        .AsNoTracking()
        .Where(o => o.Id == id)
        .Select(o => new OrderDto
        {
            Id = o.Id,
            CustomerName = o.CustomerName,
            Total = o.Total
        })
        .SingleOrDefaultAsync(cancellationToken);
}
```

> Do not ignore cancellation in a web API. A request that outlives the client wastes compute, increases latency under load, and can hold database connections longer than necessary.

**Data access layer responsibilities**

The data access layer isolates persistence concerns from controllers and business logic. It should handle querying, saving, transaction boundaries, and mapping persistence models to domain or DTO shapes.

A clean data access layer typically:
- hides SQL or provider-specific details from controllers
- centralizes query logic and transaction handling
- keeps `DbContext` usage short-lived
- uses async database APIs for scalability
- returns domain objects or DTOs, not raw database rows

For exam scenarios, recognize that the controller should not contain complex SQL or direct connection management unless the question explicitly asks for it.

**Entity Framework Core patterns**

Entity Framework Core (EF Core) is the most common data access abstraction in Azure-hosted .NET APIs. Use it for standard CRUD, filtering, sorting, pagination, and optimistic concurrency.

Register the context with a scoped lifetime, which matches the HTTP request lifetime.

```csharp
builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseSqlServer(builder.Configuration.GetConnectionString("DefaultConnection")));
```

A scoped `DbContext`:
- is created per request by default
- tracks changes for that request only
- should not be stored in singletons
- should not be reused across concurrent requests

Use `AsNoTracking()` for read-only queries to reduce memory usage and improve performance.

```csharp
var customers = await _dbContext.Customers
    .AsNoTracking()
    .OrderBy(c => c.Name)
    .ToListAsync(cancellationToken);
```

**Connection management and pooling**

A `DbContext` is not the same as a database connection. EF Core opens and closes connections as needed, and ADO.NET connection pooling helps reuse physical connections efficiently.

- Keep `DbContext` lifetimes short.
- Avoid long-lived contexts because they accumulate tracked entities and stale state.
- Let the provider manage pooling; do not manually cache connections unless you have a specific reason.
- Use `DbContextFactory` for background services or scenarios where you need to create contexts outside the request scope.

```csharp
builder.Services.AddDbContextFactory<AppDbContext>(options =>
    options.UseSqlServer(builder.Configuration.GetConnectionString("DefaultConnection")));
```

**Filtering, sorting, and pagination**

Exam-relevant APIs often need query shaping so clients can request only the data they need. This reduces payload size and improves response time.

Common patterns:
- `filter` by status, date range, category, or search term
- `sort` by one or more fields
- `paginate` with `pageNumber` and `pageSize`

```csharp
[HttpGet]
public async Task<ActionResult<IEnumerable<OrderDto>>> GetOrders(
    [FromQuery] string? status,
    [FromQuery] string? sortBy = "createdAt",
    [FromQuery] int pageNumber = 1,
    [FromQuery] int pageSize = 20,
    CancellationToken cancellationToken = default)
{
    var query = _dbContext.Orders.AsNoTracking().AsQueryable();

    if (!string.IsNullOrWhiteSpace(status))
    {
        query = query.Where(o => o.Status == status);
    }

    query = sortBy switch
    {
        "total" => query.OrderBy(o => o.Total),
        _ => query.OrderByDescending(o => o.CreatedAt)
    };

    var items = await query
        .Skip((pageNumber - 1) * pageSize)
        .Take(pageSize)
        .Select(o => new OrderDto
        {
            Id = o.Id,
            CustomerName = o.CustomerName,
            Total = o.Total
        })
        .ToListAsync(cancellationToken);

    return Ok(items);
}
```

For robust pagination, also return metadata such as total count and page information when the client needs it.

```csharp
var totalCount = await query.CountAsync(cancellationToken);
```

**Stored procedures and raw SQL**

Use stored procedures or raw SQL when the query is complex, performance-sensitive, or already exists in the database. Prefer safe parameterization.

```csharp
var customerId = 42;

var orders = await _dbContext.Orders
    .FromSqlInterpolated($"EXEC dbo.GetOrdersByCustomerId {customerId}")
    .AsNoTracking()
    .ToListAsync(cancellationToken);
```

You can also use explicit parameters:

```csharp
var customerIdParam = new SqlParameter("@CustomerId", customerId);

var orders = await _dbContext.Orders
    .FromSqlRaw("EXEC dbo.GetOrdersByCustomerId @CustomerId", customerIdParam)
    .AsNoTracking()
    .ToListAsync(cancellationToken);
```

Use raw SQL carefully:
- keep it parameterized to avoid SQL injection
- ensure the result shape matches the entity or keyless type
- prefer LINQ when the query is simple and maintainable in EF Core

**Optimistic concurrency**

Optimistic concurrency protects data when multiple clients update the same row. EF Core commonly uses a row version or timestamp column to detect conflicts.

```csharp
public class Order
{
    public int Id { get; set; }
    public decimal Total { get; set; }

    [Timestamp]
    public byte[] RowVersion { get; set; } = default!;
}
```

When a conflict occurs, EF Core throws `DbUpdateConcurrencyException`. Handle it by reloading the current values, merging changes, or returning a conflict response.

```csharp
try
{
    await _dbContext.SaveChangesAsync(cancellationToken);
}
catch (DbUpdateConcurrencyException ex)
{
    var entry = ex.Entries.Single();
    var databaseValues = await entry.GetDatabaseValuesAsync(cancellationToken);

    if (databaseValues is null)
    {
        return NotFound();
    }

    return Conflict(new
    {
        Message = "The record was updated by another user.",
        CurrentValues = databaseValues.ToObject()
    });
}
```

Typical conflict strategies:
- **client wins**: overwrite the database with the client’s values
- **store wins**: discard the client change and return the current database state
- **merge**: combine fields manually and retry

**Azure SQL and managed identity**

For Azure SQL Database, managed identity is a preferred authentication approach because it removes stored passwords from the app. The app obtains an access token for Azure SQL and uses it when opening the connection.

A working approach requires:
- enabling a managed identity on the hosting resource
- granting that identity access in Azure SQL
- acquiring a token for the correct SQL resource scope
- setting the token before opening the connection

```csharp
using Azure.Core;
using Azure.Identity;
using Microsoft.Data.SqlClient;

var credential = new DefaultAzureCredential();
var tokenRequestContext = new TokenRequestContext(
    new[] { "https://database.windows.net/.default" });

var token = await credential.GetTokenAsync(tokenRequestContext, cancellationToken);

var connection = new SqlConnection(builder.Configuration.GetConnectionString("DefaultConnection"));
connection.AccessToken = token.Token;

await connection.OpenAsync(cancellationToken);
```

> `connection.AccessToken` is only part of the solution. You still need the correct token acquisition flow and an Azure SQL configuration that supports Microsoft Entra ID authentication.

For EF Core with Azure SQL, you can inject a token into the SQL connection using a connection interceptor or configure the provider to use Microsoft Entra authentication patterns supported by your hosting environment.

**App configuration and secret management**

Do not hardcode secrets in source code. Use environment variables, app settings, managed identity, and Azure Key Vault where appropriate.

Common Azure hosting options:
- **App Service**: application settings and connection strings in the portal
- **Azure Functions**: `local.settings.json` locally, app settings in Azure
- **Container Apps**: environment variables and secret mounts
- **AKS**: Kubernetes secrets or external secret stores

Example app settings:

```json
{
  "ConnectionStrings": {
    "DefaultConnection": "Server=tcp:myserver.database.windows.net,1433;Database=appdb;Encrypt=True;TrustServerCertificate=False;"
  },
  "KeyVault": {
    "VaultUri": "https://myvault.vault.azure.net/"
  }
}
```

Use Key Vault references when you want the platform to resolve secrets at runtime without placing the secret value directly in the app setting.

**Dependency injection for data clients**

Register data clients and services in the DI (dependency injection) container so controllers stay thin and testable.

```csharp
builder.Services.AddScoped<IOrderService, OrderService>();
builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseSqlServer(builder.Configuration.GetConnectionString("DefaultConnection")));
```

For Azure SDK clients, prefer singleton registration when the client is thread-safe and designed for reuse.

```csharp
builder.Services.AddSingleton(_ =>
    new BlobServiceClient(new Uri(builder.Configuration["Storage:BlobServiceUri"]!),
    new DefaultAzureCredential()));
```

Use the Azure SDK client lifetime recommended by the library:
- reuse clients when they are thread-safe
- avoid creating a new client per request
- inject the client into services rather than constructing it in controllers

**Azure SDK usage in synchronous API paths**

When your API reads or writes Azure resources, use the SDK asynchronously and pass cancellation tokens. This is the recommended pattern for web APIs because it scales better under load.

```csharp
public class FileService
{
    private readonly BlobContainerClient _container;

    public FileService(BlobServiceClient blobServiceClient)
    {
        _container = blobServiceClient.GetBlobContainerClient("uploads");
    }

    public async Task<string> UploadAsync(Stream content, string fileName, CancellationToken cancellationToken)
    {
        var blob = _container.GetBlobClient(fileName);
        await blob.UploadAsync(content, overwrite: true, cancellationToken);
        return blob.Uri.ToString();
    }
}
```

Use Azure RBAC where the service supports it and the scenario is configured for it. For example, Azure Storage commonly uses Microsoft Entra ID and Azure RBAC for data access. For other services, authentication models vary, so verify the service-specific support rather than assuming RBAC is always the default.

**API versioning**

Versioning helps you evolve an API without breaking existing clients. In ASP.NET Core, route templates alone are not enough; you must configure API versioning and annotate controllers or actions appropriately.

A common setup uses the `Asp.Versioning.Mvc` package.

```csharp
builder.Services.AddApiVersioning(options =>
{
    options.DefaultApiVersion = new Asp.Versioning.ApiVersion(1, 0);
    options.AssumeDefaultVersionWhenUnspecified = true;
    options.ReportApiVersions = true;
});
```

Controller example:

```csharp
using Asp.Versioning;

[ApiController]
[ApiVersion("1.0")]
[Route("api/v{version:apiVersion}/orders")]
public class OrdersController : ControllerBase
{
    [HttpGet("{id:int}")]
    public IActionResult GetById(int id) => Ok();
}
```

Versioning strategies:
- **URL segment**: `api/v1/orders`
- **query string**: `api/orders?api-version=1.0`
- **header**: custom version header
- **media type**: version in `Accept`

Choose one strategy and keep it consistent. URL segment versioning is easy to discover and common in exam examples.

**Client consumption of versioned APIs**

Clients must send the version in the way the API expects. For URL segment versioning, the version is part of the route.

```bash
curl https://api.contoso.com/api/v1/orders/42
```

For query string versioning:

```bash
curl "https://api.contoso.com/api/orders/42?api-version=1.0"
```

For header-based versioning:

```bash
curl -H "x-api-version: 1.0" https://api.contoso.com/api/orders/42
```

**Synchronous read endpoint patterns**

Synchronous read endpoints often return a single resource or a small collection. Keep them efficient by projecting only the fields the client needs.

- Use `Select` to shape the result.
- Use `AsNoTracking` for read-only queries.
- Use indexes in the database for common filters and sorts.
- Return `404 Not Found` when the resource does not exist.
- Return `400 Bad Request` for invalid query parameters.

Example with validation and projection:

```csharp
[HttpGet("search")]
public async Task<ActionResult<IEnumerable<OrderDto>>> Search(
    [FromQuery] string q,
    CancellationToken cancellationToken)
{
    if (string.IsNullOrWhiteSpace(q))
    {
        return BadRequest("Query text is required.");
    }

    var results = await _dbContext.Orders
        .AsNoTracking()
        .Where(o => o.CustomerName.Contains(q))
        .OrderByDescending(o => o.CreatedAt)
        .Select(o => new OrderDto
        {
            Id = o.Id,
            CustomerName = o.CustomerName,
            Total = o.Total
        })
        .Take(25)
        .ToListAsync(cancellationToken);

    return Ok(results);
}
```

**Error handling and response codes**

Use HTTP status codes that match the outcome of the request.

- `200 OK`: successful read or update
- `201 Created`: resource created
- `204 No Content`: successful update or delete with no body
- `400 Bad Request`: invalid input
- `404 Not Found`: missing resource
- `409 Conflict`: concurrency or state conflict
- `500 Internal Server Error`: unexpected failure

For synchronous APIs, return errors quickly and consistently. Use exception handling middleware or filters for unhandled exceptions, and keep controller actions focused on expected outcomes.

**Practical Azure hosting considerations**

When you deploy a synchronous API to Azure, the hosting environment affects performance and reliability.

- **App Service**: scale out for more concurrent requests; use app settings for configuration.
- **Azure Functions**: use HTTP-triggered functions for lightweight synchronous endpoints, but avoid long-running work.
- **Container Apps**: good for containerized APIs with autoscaling and managed identity.
- **AKS**: suitable for advanced orchestration, but requires more operational overhead.

Keep in mind:
- scale-out helps with concurrency, not with a single slow request
- database bottlenecks often become the limiting factor before the web tier
- connection limits and pool exhaustion can appear under load if contexts are mismanaged

**Common exam traps**

- Assuming synchronous means the server uses a single blocking thread for the whole operation.
- Using route versioning without configuring API versioning services.
- Hardcoding secrets instead of using managed identity or Key Vault.
- Reusing a `DbContext` across requests.
- Forgetting `CancellationToken` in controller and data access methods.
- Returning too much data instead of filtering, sorting, and paginating.
- Using raw SQL without parameterization.
- Ignoring `DbUpdateConcurrencyException` in update flows.

**Quick implementation checklist**

1. Register `DbContext` with scoped lifetime.
2. Add API versioning if the route or client contract requires it.
3. Use async database and SDK calls with `CancellationToken`.
4. Project to DTOs and use `AsNoTracking()` for reads.
5. Add filtering, sorting, and pagination to collection endpoints.
6. Handle concurrency conflicts with `DbUpdateConcurrencyException`.
7. Use managed identity or Key Vault for secrets and credentials.
8. Keep controllers thin and move persistence logic into services or repositories.

**Summary**

The synchronous API path is about delivering a response within the same request/response cycle while keeping the implementation scalable, secure, and maintainable. For AZ-204, focus on how ASP.NET Core controllers, EF Core, Azure SDK clients, API versioning, cancellation, and Azure authentication work together in a real application. The best solutions are usually thin controllers, short-lived data contexts, parameterized queries, and secure configuration through Azure-native identity features.
