## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Azure Service Bus messaging model**

Azure Service Bus is a durable message broker used for asynchronous communication between components. In AZ-204 scenarios, you typically use it when you need decoupling, buffering, retries, ordered processing, or workflows that must survive transient failures.

A message usually contains:
- A **body** with the payload.
- System properties such as `MessageId`, `CorrelationId`, `SessionId`, `ReplyTo`, and `ContentType`.
- Application properties for custom metadata used by receivers, filters, and routing logic.

Use Service Bus when you need brokered messaging rather than direct HTTP calls. It is especially useful when producers and consumers scale independently or when processing must be retried without losing messages.

**Senders and receivers**

The Azure.Messaging.ServiceBus SDK provides sender and receiver abstractions for queues and topics/subscriptions.

Common client types:
- `ServiceBusClient` creates senders, receivers, processors, and session-aware clients.
- `ServiceBusSender` sends messages to a queue or topic.
- `ServiceBusReceiver` receives messages from a queue or subscription.
- `ServiceBusProcessor` provides event-driven message handling for non-session entities.
- `ServiceBusSessionReceiver` handles session-enabled entities.
- `ServiceBusSessionProcessor` provides event-driven handling for session-enabled entities.

Use a sender when you publish messages and a receiver when you need direct control over settlement, batching, or deferred messages. Use a processor when you want the SDK to manage the receive loop and concurrency for you.

**Message properties and metadata**

Message properties help you route, filter, correlate, and process messages without parsing the body first.

Important properties:
- `MessageId` supports duplicate detection and idempotency patterns.
- `CorrelationId` links related messages across workflows.
- `Subject` identifies the message purpose and is commonly used in filters.
- `SessionId` groups related messages into an ordered conversation.
- `ReplyTo` and `ReplyToSessionId` support request/reply patterns.
- `ContentType` describes the payload format, such as `application/json`.
- `ApplicationProperties` stores custom key-value metadata.

Use application properties for business metadata such as:
- `tenantId`
- `orderType`
- `priority`
- `region`
- `eventType`

Keep property names and values consistent. Subscription rules can only filter on properties that are present on the message, so missing or inconsistent metadata can cause messages to bypass intended subscriptions.

**Queues, topics, and subscriptions**

Queues deliver each message to one consumer. Topics fan out messages to multiple subscriptions.

Choose a queue when:
- One consumer should process each message.
- You want a simple work queue.
- You do not need multiple independent subscribers.

Choose a topic with subscriptions when:
- Multiple downstream systems need the same event.
- You need filtering per subscriber.
- You want pub/sub behavior.

Subscriptions can use rules to decide which messages they receive. Rules commonly inspect system properties and application properties.

**Subscription filters and rule types**

Azure Service Bus supports two common rule styles:
- **SQL filters** for expressive comparisons and logical combinations.
- **Correlation filters** for exact-match routing on selected properties.

SQL filters are more flexible. They support expressions such as:
- equality and inequality
- `AND`, `OR`, `NOT`
- `LIKE`
- `IN`
- `IS NULL`

Correlation filters are simpler and faster to reason about. They match exact values on selected fields such as:
- `CorrelationId`
- `MessageId`
- `To`
- `ReplyTo`
- `Subject`
- application properties

Use correlation filters when you want straightforward routing like “all messages where `Subject = 'orders'` and `tenantId = 'contoso'`.” Use SQL filters when you need richer logic such as ranges or compound conditions.

Example SQL filter:
```csharp
var rule = new CreateRuleOptions(
    "HighValueOrders",
    new SqlRuleFilter("amount > 1000 AND region = 'us-east'")
);
```

Example correlation filter:
```csharp
var rule = new CreateRuleOptions(
    "ContosoOrders",
    new CorrelationRuleFilter
    {
        Subject = "orders",
        ApplicationProperties =
        {
            ["tenantId"] = "contoso"
        }
    }
);
```

> Correlation filters perform exact equality matching only. They do not evaluate SQL-like expressions.

**Creating messages with properties**

You set message metadata before sending. The SDK exposes strongly typed properties and a dictionary for custom values.

Example sender code:
```csharp
using Azure.Messaging.ServiceBus;

ServiceBusClient client = new ServiceBusClient(connectionString);
ServiceBusSender sender = client.CreateSender("orders");

var message = new ServiceBusMessage(BinaryData.FromString("{\"orderId\":42,\"amount\":1250}"))
{
    MessageId = Guid.NewGuid().ToString(),
    Subject = "orders",
    ContentType = "application/json",
    CorrelationId = "workflow-123",
    SessionId = "customer-42"
};

message.ApplicationProperties["tenantId"] = "contoso";
message.ApplicationProperties["priority"] = "high";
message.ApplicationProperties["region"] = "us-east";

await sender.SendMessageAsync(message);
```

For multiple messages, use batching when possible:
```csharp
using Azure.Messaging.ServiceBus;

ServiceBusClient client = new ServiceBusClient(connectionString);
ServiceBusSender sender = client.CreateSender("orders");

using ServiceBusMessageBatch batch = await sender.CreateMessageBatchAsync();

foreach (var order in orders)
{
    var msg = new ServiceBusMessage(BinaryData.FromObjectAsJson(order))
    {
        MessageId = order.Id,
        Subject = "orders",
        ContentType = "application/json"
    };

    msg.ApplicationProperties["tenantId"] = order.TenantId;
    msg.ApplicationProperties["region"] = order.Region;

    if (!batch.TryAddMessage(msg))
    {
        await sender.SendMessagesAsync(batch);

        batch.Dispose();
        using ServiceBusMessageBatch nextBatch = await sender.CreateMessageBatchAsync();

        if (!nextBatch.TryAddMessage(msg))
        {
            throw new InvalidOperationException("Message is too large for an empty batch.");
        }

        await sender.SendMessagesAsync(nextBatch);
    }
}

if (batch.Count > 0)
{
    await sender.SendMessagesAsync(batch);
}
```

A safer pattern is to create a new batch when the current one fills and continue processing all remaining messages. In exam scenarios, remember that a failed `TryAddMessage` means the message must be retried in a new batch, not dropped.

**Receiving messages with a non-session processor**

For standard queues and subscriptions that do not require sessions, `ServiceBusProcessor` is the easiest option. It handles the receive loop, concurrency, and error callbacks.

Example:
```csharp
using Azure.Messaging.ServiceBus;

ServiceBusClient client = new ServiceBusClient(connectionString);

ServiceBusProcessor processor = client.CreateProcessor(
    "orders",
    new ServiceBusProcessorOptions
    {
        AutoCompleteMessages = false,
        MaxConcurrentCalls = 4
    });

processor.ProcessMessageAsync += async args =>
{
    string body = args.Message.Body.ToString();

    if (body.Contains("invalid"))
    {
        await args.DeadLetterMessageAsync(args.Message, "ValidationFailed", "Invalid order payload");
        return;
    }

    // Process work here
    await args.CompleteMessageAsync(args.Message);
};

processor.ProcessErrorAsync += args =>
{
    Console.WriteLine(args.Exception);
    return Task.CompletedTask;
};

await processor.StartProcessingAsync();
```

Use a processor when:
- you want event-driven handling
- you do not need manual receive loops
- you want the SDK to manage concurrency and retries

Use a receiver when:
- you need explicit control over settlement
- you want to receive deferred messages
- you need peek-lock style manual processing
- you want to manage message ordering or custom loops

**Manual receive and settlement**

A receiver gives you direct control over how messages are completed, abandoned, dead-lettered, deferred, or renewed.

Common settlement actions:
- `CompleteMessageAsync` removes the message from the queue or subscription.
- `AbandonMessageAsync` releases the message for redelivery.
- `DeadLetterMessageAsync` moves the message to the dead-letter subqueue.
- `DeferMessageAsync` keeps the message in the entity but hides it from normal receives.
- `RenewMessageLockAsync` extends the lock on a message.
- `RenewSessionLockAsync` extends the lock on a session.

Example:
```csharp
using Azure.Messaging.ServiceBus;

ServiceBusClient client = new ServiceBusClient(connectionString);
ServiceBusReceiver receiver = client.CreateReceiver("orders");

ServiceBusReceivedMessage message = await receiver.ReceiveMessageAsync();

if (message is not null)
{
    try
    {
        // Process message
        await receiver.CompleteMessageAsync(message);
    }
    catch
    {
        await receiver.AbandonMessageAsync(message);
    }
}
```

**Deferral and deferred message retrieval**

Deferral is useful when a message arrives before its dependencies are ready. The message stays in the entity, but you must retrieve it later by sequence number.

Example:
```csharp
ServiceBusReceivedMessage message = await receiver.ReceiveMessageAsync();

if (message.Subject == "waiting-for-invoice")
{
    await receiver.DeferMessageAsync(message);
}
```

Later, retrieve deferred messages by sequence number:
```csharp
long[] sequenceNumbers = { 12345, 12346 };
IReadOnlyList<ServiceBusReceivedMessage> deferred =
    await receiver.ReceiveDeferredMessagesAsync(sequenceNumbers);

foreach (var deferredMessage in deferred)
{
    await receiver.CompleteMessageAsync(deferredMessage);
}
```

Use deferral when:
- processing order depends on another message
- a prerequisite is missing temporarily
- you need to postpone handling without losing the message

**Sessions and ordered conversation state**

Sessions provide FIFO ordering within a logical group of messages. All messages with the same `SessionId` are processed in order by a single consumer at a time.

Use sessions when:
- you need ordered processing per customer, order, or workflow
- you want to keep related messages isolated from other groups
- you need per-conversation state

A session-enabled entity must be configured at creation time. You cannot turn a non-session queue or subscription into a session-enabled one later. If you need sessions, create the queue or subscription with sessions enabled from the start.

Typical configuration examples:
- Azure Portal: enable **Requires session** when creating the queue or subscription.
- ARM/Bicep: set `requiresSession` to `true`.
- Azure CLI: create the entity with session support enabled.

Example Bicep:
```bicep
resource queue 'Microsoft.ServiceBus/namespaces/queues@2022-10-01-preview' = {
  name: 'orders'
  parent: namespace
  properties: {
    requiresSession: true
    maxDeliveryCount: 10
    lockDuration: 'PT1M'
  }
}
```

Example Azure CLI:
```bash
az servicebus queue create \
  --resource-group rg-messaging \
  --namespace-name sbns-prod \
  --name orders \
  --requires-session true
```

> A non-session receiver cannot read from a session-enabled queue or subscription. You must use session-aware APIs.

**Receiving from sessions**

To process session-enabled entities, use session-aware APIs. The SDK gives you either a session receiver for manual control or a session processor for automatic handling.

Manual session receiver:
```csharp
using Azure.Messaging.ServiceBus;

ServiceBusClient client = new ServiceBusClient(connectionString);

ServiceBusSessionReceiver sessionReceiver =
    await client.AcceptNextSessionAsync("orders");

ServiceBusReceivedMessage message = await sessionReceiver.ReceiveMessageAsync();

if (message is not null)
{
    await sessionReceiver.CompleteMessageAsync(message);
}
```

Session processor:
```csharp
using Azure.Messaging.ServiceBus;

ServiceBusClient client = new ServiceBusClient(connectionString);

ServiceBusSessionProcessor processor = client.CreateSessionProcessor(
    "orders",
    new ServiceBusSessionProcessorOptions
    {
        MaxConcurrentSessions = 8,
        MaxConcurrentCallsPerSession = 1,
        AutoCompleteMessages = false
    });

processor.ProcessMessageAsync += async args =>
{
    // Messages in the same session are ordered
    await args.CompleteMessageAsync(args.Message);
};

processor.ProcessErrorAsync += args =>
{
    Console.WriteLine(args.Exception);
    return Task.CompletedTask;
};

await processor.StartProcessingAsync();
```

Use `ServiceBusSessionProcessor` when you want the SDK to manage session acquisition and concurrency. Use `ServiceBusSessionReceiver` when you need explicit control over one session at a time.

**Session lock renewal and expiration**

A session lock protects exclusive access to a session while you process its messages. If processing takes too long and you do not renew the lock, the session can be released and another consumer may acquire it.

Important behaviors:
- The session lock is separate from individual message locks.
- Long-running handlers should renew the session lock proactively.
- If the lock expires, the current receiver loses ownership of the session.
- Messages not completed before lock loss may be redelivered.

Example manual renewal:
```csharp
using Azure.Messaging.ServiceBus;

ServiceBusSessionReceiver sessionReceiver =
    await client.AcceptNextSessionAsync("orders");

using CancellationTokenSource cts = new CancellationTokenSource(TimeSpan.FromMinutes(10));

Task renewTask = Task.Run(async () =>
{
    while (!cts.IsCancellationRequested)
    {
        await Task.Delay(TimeSpan.FromSeconds(20), cts.Token);
        await sessionReceiver.RenewSessionLockAsync();
    }
}, cts.Token);
```

In practice, prefer the processor for common workloads because it can manage session handling more safely. For long-running custom logic, ensure your code renews the session lock before expiration.

**Dead-lettering**

Dead-lettering moves a message to the dead-letter subqueue for later inspection. Use it when a message cannot be processed successfully or when it is invalid for the current workflow.

Common dead-letter causes include:
- application validation failure
- `MaxDeliveryCount` exceeded
- message TTL (time to live) expiration
- filter evaluation exceptions in subscriptions
- explicit dead-lettering by your code

Example:
```csharp
await receiver.DeadLetterMessageAsync(
    message,
    deadLetterReason: "ValidationFailed",
    deadLetterErrorDescription: "CustomerId is missing");
```

For session-enabled entities, dead-letter access also requires session-aware APIs. Do not assume a normal receiver can target the dead-letter subqueue of a session-enabled queue or subscription.

You can inspect dead-letter messages with a receiver created for the dead-letter subqueue:
```csharp
var options = new ServiceBusReceiverOptions
{
    SubQueue = SubQueue.DeadLetter
};

ServiceBusReceiver deadLetterReceiver = client.CreateReceiver("orders", options);
```

If the entity requires sessions, use the session-aware dead-letter path instead of a non-session receiver.

**Message properties in subscription rules**

Subscription rules often depend on message metadata rather than the body. This keeps routing fast and avoids deserializing payloads just to decide where a message should go.

Typical rule inputs:
- `Subject` for message type or event name
- `CorrelationId` for workflow grouping
- `ApplicationProperties["tenantId"]` for tenant routing
- `ApplicationProperties["region"]` for geographic routing
- `ApplicationProperties["priority"]` for priority-based delivery

Example message:
```csharp
var msg = new ServiceBusMessage(BinaryData.FromString("payload"))
{
    Subject = "invoice.created",
    CorrelationId = "inv-1001"
};

msg.ApplicationProperties["tenantId"] = "contoso";
msg.ApplicationProperties["amount"] = 2500;
```

Example SQL rule:
```sql
Subject = 'invoice.created' AND tenantId = 'contoso' AND amount > 1000
```

Be careful with property types:
- SQL filters support numeric and string comparisons, but the property type must be compatible.
- Correlation filters use exact matches and are best for simple routing.
- Missing properties can cause a filter to evaluate to false or fail depending on the rule and expression.

**Choosing the right API in exam scenarios**

Use the following decision points:
- Use `ServiceBusSender` to publish one or more messages.
- Use `ServiceBusProcessor` for non-session event-driven consumption.
- Use `ServiceBusReceiver` for manual receive and settlement.
- Use `ServiceBusSessionProcessor` for session-enabled event-driven consumption.
- Use `ServiceBusSessionReceiver` for manual session control.
- Use deferred message APIs when you intentionally postpone processing.
- Use dead-lettering when a message is invalid or cannot be processed after retries.

A practical rule of thumb:
- If you need simplicity, choose a processor.
- If you need control, choose a receiver.
- If you need ordering by group, choose sessions.
- If you need routing, choose message properties and subscription filters.

**Common pitfalls**

- Forgetting to enable sessions on the entity before sending session messages.
- Using a non-session receiver against a session-enabled queue or subscription.
- Dropping a message when a batch is full instead of retrying it in a new batch.
- Completing a message before processing finishes.
- Letting a session lock expire during long-running work.
- Using body content for routing when message properties would be more reliable.
- Assuming correlation filters support SQL expressions.
- Forgetting to retrieve deferred messages by sequence number.

**Practical AZ-204 takeaways**

In exam questions, look for clues in the requirements:
- Ordered processing per customer or order usually means sessions.
- Multiple subscribers with different routing needs usually means topics and subscriptions.
- Filtering by metadata usually means message properties and subscription rules.
- Manual retry control usually means receiver settlement methods.
- Long-running processing usually means lock renewal or a processor with appropriate options.

**Quick reference**

| Scenario | Recommended API | Notes |
|---|---|---|
| Send a single message | `ServiceBusSender.SendMessageAsync` | Set system and application properties first |
| Send many messages | `ServiceBusMessageBatch` | Retry overflow messages in a new batch |
| Non-session consumption | `ServiceBusProcessor` | Best for event-driven handlers |
| Manual non-session receive | `ServiceBusReceiver` | Use for settlement and deferred messages |
| Session consumption | `ServiceBusSessionProcessor` | Preserves order within a session |
| Manual session receive | `ServiceBusSessionReceiver` | Use for explicit session control |
| Route by metadata | SQL or correlation filters | Filters inspect message properties |
| Postpone processing | Deferral | Retrieve later by sequence number |
| Quarantine invalid messages | Dead-lettering | Inspect dead-letter subqueue later |

**Minimal end-to-end example**

This example sends an order message with properties, routes it through a topic subscription, and processes it with a session-aware consumer.

```csharp
using Azure.Messaging.ServiceBus;

var client = new ServiceBusClient(connectionString);

// Sender
var sender = client.CreateSender("orders-topic");
var orderMessage = new ServiceBusMessage(BinaryData.FromString("{\"orderId\":101,\"amount\":1500}"))
{
    Subject = "order.created",
    ContentType = "application/json",
    SessionId = "customer-42",
    CorrelationId = "workflow-101"
};
orderMessage.ApplicationProperties["tenantId"] = "contoso";
orderMessage.ApplicationProperties["region"] = "us-east";

await sender.SendMessageAsync(orderMessage);

// Session processor
var processor = client.CreateSessionProcessor("orders-subscription");
processor.ProcessMessageAsync += async args =>
{
    Console.WriteLine(args.Message.Subject);
    await args.CompleteMessageAsync(args.Message);
};
processor.ProcessErrorAsync += args =>
{
    Console.WriteLine(args.Exception);
    return Task.CompletedTask;
};

await processor.StartProcessingAsync();
```
