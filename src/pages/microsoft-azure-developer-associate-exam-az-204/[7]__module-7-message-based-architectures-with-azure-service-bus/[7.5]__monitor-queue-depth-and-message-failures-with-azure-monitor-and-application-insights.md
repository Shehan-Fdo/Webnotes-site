## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Monitoring goals for queue-based systems**

Queue-driven applications fail in predictable ways: backlog grows, messages retry too many times, poison messages accumulate, and consumers throw exceptions. Your monitoring strategy should answer four questions quickly:

- Is the queue growing faster than consumers can drain it?
- Are messages failing transiently or permanently?
- Are failures isolated to one consumer instance or systemic?
- Can you correlate queue behavior with application exceptions and traces?

Azure Monitor gives you the platform view: metrics, logs, alerts, workbooks, and dashboards. Application Insights gives you the application view: requests, dependencies, exceptions, traces, and custom telemetry. Together, they let you move from “the queue is stuck” to “this code path is failing on this message type” without guessing.

**Core telemetry sources**

For Azure messaging workloads, you usually combine these sources:

- **Platform metrics** from the queue service
  - Queue depth
  - Incoming and outgoing message rates
  - Dead-letter counts where supported
  - Delivery-related counters where available
- **Resource logs** from diagnostic settings
  - Operational events
  - Authorization or connection-related events where emitted
  - Service-specific logs for troubleshooting
- **Application telemetry** from Application Insights
  - Exceptions
  - Dependency calls to the queue service
  - Custom events for message processing stages
  - Traces with correlation identifiers

A good design uses metrics for fast alerting and logs for investigation. Metrics are near real time and low cost for alerting. Logs are richer and better for root-cause analysis.

**Azure Monitor building blocks**

Azure Monitor is the umbrella service for collecting and acting on telemetry. For exam scenarios, focus on these pieces:

- **Metrics**
  - Numeric time-series data.
  - Best for queue depth, throughput, and threshold-based alerts.
- **Logs**
  - Queryable records in Log Analytics.
  - Best for investigating failures, trends, and correlations.
- **Alerts**
  - Metric alerts for fast threshold detection.
  - Log alerts for pattern-based detection.
  - Activity log alerts for subscription-level control-plane events.
- **Workbooks**
  - Interactive visualizations and investigations.
  - Useful for queue depth trends, dead-letter growth, and failure heatmaps.
- **Dashboards**
  - Pinned charts and tiles for operations teams.
  - Good for at-a-glance status, but less flexible than workbooks.

> Use a **metric alert** when you need fast detection of a numeric threshold, such as queue depth above a limit. Use a **log alert** when you need to detect a pattern, such as repeated exceptions or a specific dead-letter reason.

**Choosing the right alert type**

| Alert type | Best for | Example |
|---|---|---|
| Metric alert | Numeric thresholds and fast response | Queue depth exceeds 10,000 for 5 minutes |
| Log alert | Query-based conditions and richer context | More than 20 processing exceptions in 10 minutes |
| Activity log alert | Azure control-plane events | Resource changes, policy actions, or service health events |

For AZ-204, remember that queue depth is usually a metric problem, while message failure investigation is often a log problem.

**Monitoring Azure Service Bus queue depth**

Azure Service Bus exposes platform metrics that help you understand backlog and throughput. Exact metric names can vary by namespace, entity type, and Azure Monitor surface, so verify the available metrics in the portal for your resource. Commonly used signals include:

- **Active messages**
  - Messages ready to be processed.
  - A rising value usually means consumers are falling behind.
- **Dead-lettered messages**
  - Messages moved to the dead-letter queue because they could not be processed normally.
  - Growth here usually indicates poison messages or repeated delivery failures.
- **Scheduled messages**
  - Messages scheduled for future delivery.
  - Useful when backlog is expected and should not be mistaken for a processing problem.
- **Incoming and outgoing messages**
  - Throughput indicators.
  - Helpful for comparing producer rate versus consumer rate.

A practical interpretation pattern is:

- If active messages rise while outgoing messages stay flat, consumers are not keeping up.
- If dead-lettered messages rise, inspect message content, handler exceptions, and dead-letter reasons.
- If incoming messages spike but active messages remain low, consumers may be scaling well or messages may be expiring, completing, or dead-lettering quickly.

**Service Bus diagnostics and logs**

Service Bus diagnostic settings send resource logs and metrics to destinations such as Log Analytics, Storage, or Event Hubs. In practice, Log Analytics is the most useful destination for investigation.

Typical workflow:

1. Open the Service Bus namespace or entity in the Azure portal.
2. Select **Diagnostic settings**.
3. Create a diagnostic setting.
4. Choose the log categories and metrics you need.
5. Send them to a Log Analytics workspace for querying.

A representative Azure CLI pattern looks like this:

```bash
az monitor diagnostic-settings create \
  --name sb-diagnostics \
  --resource "/subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.ServiceBus/namespaces/<namespace>" \
  --workspace "/subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.OperationalInsights/workspaces/<workspace>" \
  --logs '[{"category":"OperationalLogs","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'
```

The exact log category names depend on the resource and diagnostic schema exposed by the service. In exam scenarios, the important point is not memorizing one category name, but knowing that you enable diagnostics on the resource and route them to a workspace for analysis.

**Querying Service Bus logs in Log Analytics**

Once logs are flowing, use Kusto Query Language (KQL) to investigate patterns. The table and schema can vary depending on the diagnostic destination and configuration, so treat queries as starting points and adjust them to your workspace schema.

Example pattern for resource logs:

```kusto
AzureDiagnostics
| where ResourceProvider == "MICROSOFT.SERVICEBUS"
| summarize Count = count() by Category, bin(TimeGenerated, 1h)
| order by TimeGenerated desc
```

If your workspace uses resource-specific tables, query those instead of `AzureDiagnostics`. For example, some environments expose dedicated tables or different column names. The exam takeaway is that you should know how to pivot from metrics to logs, not rely on one fixed table name.

To investigate failures, look for:

- Repeated authorization failures
- Message lock or settlement errors
- Dead-letter-related events
- Throttling or quota-related signals
- Consumer-side exceptions correlated by time

**Monitoring message failures in Service Bus**

Message failure monitoring is more than counting dead letters. You want to understand the failure path:

- **Transient failures**
  - Network blips, temporary downstream outages, or throttling.
  - Usually handled by retry policies.
- **Permanent failures**
  - Invalid payloads, schema mismatches, or business rule violations.
  - Often end in dead-letter queues.
- **Poison messages**
  - Messages that repeatedly fail processing and should be isolated.
  - Usually moved to a dead-letter queue after retry exhaustion.

Useful signals include:

- Dead-letter queue growth
- Delivery count approaching the max delivery count
- Consumer exceptions per message type
- Retry exhaustion
- Abandon/retry loops

A common operational pattern is to alert when dead-letter count increases faster than a baseline, then inspect the dead-letter reason and exception traces to determine whether the issue is data-related or code-related.

**Dead-letter queue inspection**

For Service Bus, the dead-letter queue is a subqueue of the main entity. You inspect it separately to understand why messages failed.

A typical receiver pattern in the Azure.Messaging.ServiceBus SDK is:

```csharp
using Azure.Messaging.ServiceBus;

var client = new ServiceBusClient(connectionString);
var receiver = client.CreateReceiver(
    queueName,
    new ServiceBusReceiverOptions
    {
        SubQueue = SubQueue.DeadLetter
    });

IReadOnlyList<ServiceBusReceivedMessage> messages =
    await receiver.ReceiveMessagesAsync(maxMessages: 10, maxWaitTime: TimeSpan.FromSeconds(5));

foreach (var message in messages)
{
    Console.WriteLine($"MessageId: {message.MessageId}");
    Console.WriteLine($"DeadLetterReason: {message.DeadLetterReason}");
    Console.WriteLine($"DeadLetterErrorDescription: {message.DeadLetterErrorDescription}");
}
```

This is useful when you need to identify whether the message failed because of a business rule, a deserialization issue, or a downstream dependency problem.

**Application Insights for consumer telemetry**

Application Insights is the application performance monitoring service in Azure. For queue consumers, it helps you answer:

- Which message caused the exception?
- Which consumer instance processed it?
- How long did processing take?
- Which dependency failed?
- Did the failure happen before or after settlement?

For modern .NET worker services, the recommended approach is to configure Application Insights through dependency injection and configuration, then provide the connection string through app settings or environment variables.

Example worker service setup:

```csharp
using Microsoft.ApplicationInsights.WorkerService;
using Microsoft.Extensions.DependencyInjection;
using Microsoft.Extensions.Hosting;

var builder = Host.CreateApplicationBuilder(args);

builder.Services.AddApplicationInsightsTelemetryWorkerService();
builder.Services.AddHostedService<QueueProcessor>();

var host = builder.Build();
host.Run();
```

Then set the connection string in configuration, for example in `appsettings.json` or an environment variable:

```json
{
  "ApplicationInsights": {
    "ConnectionString": "InstrumentationKey=<key>;IngestionEndpoint=https://<region>.in.applicationinsights.azure.com/;LiveEndpoint=https://<region>.livediagnostics.monitor.azure.com/"
  }
}
```

In Azure App Service, Azure Functions, or container environments, prefer environment-based configuration:

```bash
APPLICATIONINSIGHTS_CONNECTION_STRING="InstrumentationKey=<key>;IngestionEndpoint=https://<region>.in.applicationinsights.azure.com/"
```

The connection string format is preferred over legacy instrumentation-key-only configuration.

**Application Insights in Azure Functions**

Azure Functions can emit telemetry to Application Insights automatically. The exact programming model matters:

- **In-process model**
  - Uses attributes such as `[FunctionName]`.
  - Still supported in some scenarios, but older.
- **Isolated worker model**
  - Uses the newer .NET isolated process model.
  - Common in current guidance and modern projects.

For exam prep, know that telemetry configuration differs by model, but the goal is the same: capture traces, exceptions, and dependency calls from the function that processes queue messages.

A typical isolated worker setup uses host configuration and the Application Insights connection string in app settings rather than hardcoding telemetry settings in code.

**Correlating queue messages with traces**

Correlation is what turns raw telemetry into a troubleshooting story. The best practice is to propagate distributed tracing context using W3C Trace Context rather than inventing your own header format.

When you send a message, include trace context if your SDK or framework does not do it automatically. In many cases, the Azure SDK and OpenTelemetry-based instrumentation can handle this for you.

Example of adding custom application properties for correlation metadata:

```csharp
var message = new ServiceBusMessage(BinaryData.FromString(payload));
message.ApplicationProperties["messageType"] = "OrderCreated";
message.ApplicationProperties["tenantId"] = tenantId;
message.ApplicationProperties["correlationId"] = correlationId;
```

If you need to propagate trace context manually, use the proper `traceparent` and `tracestate` values from the current activity rather than `Activity.Current?.Id` alone.

```csharp
using System.Diagnostics;

var activity = Activity.Current;
if (activity != null)
{
    message.ApplicationProperties["traceparent"] = activity.Id;
    if (!string.IsNullOrEmpty(activity.TraceStateString))
    {
        message.ApplicationProperties["tracestate"] = activity.TraceStateString;
    }
}
```

In practice, prefer SDK-supported propagation or OpenTelemetry instrumentation where possible. Manual propagation is easy to get wrong.

**Logging failures in the consumer**

Your consumer should log enough context to diagnose failures without exposing sensitive payload data. Capture:

- Message ID
- Correlation ID
- Queue or topic/subscription name
- Processing duration
- Exception type and message
- Retry attempt or delivery count
- Business key or message type

Example:

```csharp
try
{
    var started = DateTimeOffset.UtcNow;

    // Process message
    await ProcessAsync(message);

    logger.LogInformation(
        "Processed message {MessageId} of type {MessageType} in {ElapsedMs} ms",
        message.MessageId,
        message.ApplicationProperties.TryGetValue("messageType", out var type) ? type : "unknown",
        (DateTimeOffset.UtcNow - started).TotalMilliseconds);
}
catch (Exception ex)
{
    logger.LogError(ex,
        "Failed processing message {MessageId}, delivery {DeliveryCount}",
        message.MessageId,
        message.DeliveryCount);

    throw;
}
```

This gives Application Insights enough structured data to group failures and correlate them with queue behavior.

**Querying failures in Application Insights**

Application Insights stores telemetry in tables that you can query with KQL. For queue processors, the most useful tables are often:

- `exceptions`
- `traces`
- `dependencies`
- `requests` for HTTP-triggered components or APIs that enqueue messages
- `customEvents` for business milestones

Example query for recent exceptions:

```kusto
exceptions
| where timestamp > ago(24h)
| summarize Count = count() by type, outerMessage
| order by Count desc
```

Example query for queue processor traces:

```kusto
traces
| where timestamp > ago(24h)
| where message has "Processed message" or message has "Failed processing message"
| project timestamp, message, severityLevel, operation_Id, customDimensions
| order by timestamp desc
```

Example query to correlate exceptions with a specific operation:

```kusto
exceptions
| where timestamp > ago(24h)
| project timestamp, type, outerMessage, operation_Id, problemId
| join kind=leftouter (
    traces
    | where timestamp > ago(24h)
    | project operation_Id, traceMessage = message
) on operation_Id
| order by timestamp desc
```

For a queue consumer, `operation_Id` is often the key to linking traces, dependencies, and exceptions from the same processing attempt.

**Alerting strategy for queue systems**

A strong alerting design uses multiple layers:

- **Queue depth alert**
  - Trigger when active messages exceed a threshold for a sustained period.
  - Detects backlog before users notice latency.
- **Dead-letter alert**
  - Trigger when dead-letter count increases or crosses a threshold.
  - Detects poison messages and schema drift.
- **Exception rate alert**
  - Trigger when consumer exceptions exceed a baseline.
  - Detects code regressions or downstream outages.
- **Latency alert**
  - Trigger when processing time or end-to-end delay exceeds an SLA.
  - Detects slow consumers or dependency degradation.

A practical rule is to alert on symptoms, not just causes. For example, a dead-letter spike is a symptom; the cause may be a serialization bug, a bad deployment, or a downstream API change.

**Example metric alert scenario**

If a queue normally holds fewer than 100 active messages, you might create a metric alert for sustained backlog:

- Metric: active messages
- Condition: greater than 1000
- Evaluation window: 5 minutes
- Action group: email, SMS, webhook, or automation runbook

This is a good fit for Azure Monitor metric alerts because the condition is numeric and time-based.

**Example log alert scenario**

If your consumer logs a specific failure message when deserialization fails, create a log alert:

```kusto
traces
| where timestamp > ago(10m)
| where message has "Deserialization failed"
| summarize Count = count()
```

This is a good fit for a log alert because you are matching a text pattern and can include richer context in the query.

**Workbooks and dashboards for operational visibility**

Workbooks are especially useful for queue monitoring because they can combine metrics and logs in one view. A practical workbook might include:

- Queue depth over time
- Dead-letter count over time
- Top exception types
- Processing latency percentiles
- Recent failed message IDs
- Consumer instance health

Dashboards are useful when operations teams want a simplified view pinned to a portal home page. Use dashboards for quick status and workbooks for investigation.

A useful workbook layout is:

1. Top row: queue depth and dead-letter trend charts.
2. Middle row: exception counts and dependency failures.
3. Bottom row: recent failed messages and drill-through links to logs.

**Storage Queues monitoring**

Azure Storage Queues are simpler than Service Bus, but the monitoring pattern is similar:

- Track queue length to detect backlog.
- Track dequeue count to identify repeated processing attempts.
- Use a poison queue pattern in your application because Storage Queues do not provide a built-in dead-letter queue like Service Bus.

Storage Queues use a visibility timeout and pop receipt model. That gives you a lease-like processing window, but it is not the same as Service Bus peek-lock semantics.

A typical processing pattern is:

```csharp
var message = await queueClient.ReceiveMessageAsync();

if (message != null)
{
    try
    {
        await ProcessAsync(message.MessageText);
        await queueClient.DeleteMessageAsync(message.MessageId, message.PopReceipt);
    }
    catch (Exception ex)
    {
        logger.LogError(ex, "Failed processing storage queue message {MessageId}", message.MessageId);

        if (message.DequeueCount >= 5)
        {
            await poisonQueueClient.SendMessageAsync(message.MessageText);
            await queueClient.DeleteMessageAsync(message.MessageId, message.PopReceipt);
        }
        else
        {
            // Let the visibility timeout expire so the message becomes visible again.
            // Optionally extend visibility if you need more processing time.
        }
    }
}
```

The important detail is that `DequeueCount` reflects how many times the message has been received, not how many times your code has failed after receipt. Use it as a signal for repeated retries, and delete the original message after moving it to a poison queue.

**Alerting on poison queues**

For Storage Queues, create alerts on the poison queue itself rather than only on the main queue. Useful patterns include:

- Poison queue length above zero for a sustained period
- Poison queue growth rate above baseline
- Exception rate in the consumer exceeds threshold
- Custom metric for failed message processing

If you emit a custom metric from your worker, you can alert on it directly:

```csharp
telemetryClient.GetMetric("QueueMessageProcessingFailures").TrackValue(1);
```

This is useful when the platform does not expose the exact business failure you care about.

**End-to-end incident example**

Imagine an order-processing system with a Service Bus queue, a .NET worker, and Application Insights enabled.

What happens:

- Queue depth starts rising from 50 to 5,000.
- Dead-letter messages begin increasing.
- Application Insights shows a spike in `SqlException` or downstream API timeout exceptions.
- Traces show that the consumer fails after deserializing a specific message type.
- The dead-letter reason indicates repeated delivery failures.

How you troubleshoot:

1. Check the queue metric to confirm backlog growth.
2. Open the dead-letter queue and inspect the failing messages.
3. Query Application Insights exceptions for the same time window.
4. Correlate by `operation_Id`, `messageId`, or custom correlation ID.
5. Determine whether the issue is data-related, code-related, or dependency-related.
6. Fix the consumer or quarantine the bad payloads.
7. Watch the backlog drain and dead-letter growth stop.

This is the kind of scenario AZ-204 often tests: not just “what service do you use,” but “which telemetry tells you where the failure is.”

**Practical exam distinctions**

- Use **Azure Monitor metrics** for queue depth and threshold alerts.
- Use **Log Analytics queries** for investigating dead letters and exception patterns.
- Use **Application Insights** for consumer-side exceptions, traces, and dependency failures.
- Use **Workbooks** for operational views and trend analysis.
- Use **Dashboards** for simple team visibility.
- Use **Activity log alerts** for Azure resource changes, not message-processing failures.

**Common pitfalls**

- Hardcoding Application Insights instrumentation keys instead of using the connection string.
- Assuming one Service Bus metric name applies to every namespace and entity without verification.
- Treating dead-letter growth as the root cause instead of a symptom.
- Logging message payloads in full when they may contain sensitive data.
- Using manual trace propagation incorrectly instead of SDK-supported correlation.
- Relying only on queue depth and ignoring exception telemetry.
- Forgetting to create a poison-message strategy for Storage Queues.

**Quick reference**

- **Queue depth high**
  - Check consumer scale, throughput, and downstream latency.
- **Dead-letter count rising**
  - Inspect dead-letter reason, delivery count, and consumer exceptions.
- **Consumer exceptions rising**
  - Query Application Insights `exceptions` and `traces`.
- **Need a fast threshold alert**
  - Use a metric alert.
- **Need pattern-based detection**
  - Use a log alert.
- **Need a visual investigation view**
  - Use a workbook.
