## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Lab overview**

This lab pattern connects three concerns: an API that accepts work, a messaging layer that buffers and decouples processing, and a Function App that performs asynchronous work. The reliability goal is to make message delivery and processing observable, retryable, and safe under transient failures.

The typical flow is:

- Client calls an API endpoint.
- API validates input and publishes a message to <abbr title="Azure Service Bus">ASB</abbr>.
- Azure Functions consumes the message and performs the downstream action.
- You validate retries, dead-lettering, and idempotent behavior.

This architecture helps you absorb spikes, isolate failures, and avoid coupling the API latency to the processing latency.

**Core architecture and why it matters**

A queue or topic acts as a buffer between the producer and consumer. The API can return quickly after enqueueing work, while the Function App can scale independently. This is especially useful when the downstream operation is slow, rate-limited, or occasionally unavailable.

Common reliability goals include:

- **At-least-once delivery**: the consumer may see a message more than once, so processing must be safe to repeat.
- **Poison message handling**: messages that repeatedly fail should move to the dead-letter queue (DLQ).
- **Observability**: logs, correlation IDs, and message metadata should let you trace a request end to end.
- **Backpressure tolerance**: the queue absorbs bursts when the consumer cannot keep up.

> Azure Service Bus and Azure Functions both provide retry and failure-handling features, but you still need application-level idempotency. Do not assume the platform guarantees exactly-once processing.

**Reference topology**

A simple lab topology usually looks like this:

```text
Client -> API App Service / Container App / Function HTTP trigger
       -> Azure Service Bus queue or topic
       -> Azure Function Service Bus trigger
       -> downstream store, API, or business process
```

If you need fan-out, use a topic with subscriptions. If you need one consumer path, a queue is simpler.

**Service Bus concepts you should know**

Azure Service Bus is a managed message broker that supports queues and topics/subscriptions. It is a good fit when you need richer messaging semantics than a basic storage queue.

Key terms:

- **Queue**: one producer-to-one or many consumers pattern with competing consumers.
- **Topic**: one producer-to-many consumers pattern.
- **Subscription**: a virtual queue attached to a topic.
- **Dead-letter queue**: a holding area for messages that cannot be processed.
- **Peek-lock**: the consumer locks a message, processes it, then completes it.
- **Session**: ordered processing for related messages when you need affinity.

Practical reasons to choose Service Bus:

- You need duplicate detection, scheduled delivery, or sessions.
- You want explicit message settlement.
- You need richer retry and dead-letter behavior.
- You want to separate producer and consumer lifecycles.

**API publisher design**

The API should do only the minimum work needed to validate and enqueue the message. Keep the request path short and deterministic.

Recommended responsibilities:

- Validate the request body and required headers.
- Create a message payload with a stable schema.
- Add correlation metadata such as request ID and user context.
- Send the message to Service Bus.
- Return `202 Accepted` when the work is queued, not completed.

A simple .NET example using `Azure.Messaging.ServiceBus`:

```csharp
using Azure.Messaging.ServiceBus;

var client = new ServiceBusClient(connectionString);
var sender = client.CreateSender(queueName);

var payload = new
{
    orderId = "ORD-1001",
    customerId = "CUST-42",
    requestedAtUtc = DateTime.UtcNow
};

var message = new ServiceBusMessage(BinaryData.FromObjectAsJson(payload))
{
    ContentType = "application/json",
    Subject = "order.created",
    MessageId = payload.orderId
};

message.ApplicationProperties["correlationId"] = activityId;
message.ApplicationProperties["source"] = "orders-api";

await sender.SendMessageAsync(message);
```

Important implementation details:

- **MessageId** can help with duplicate detection when enabled on the entity.
- **ApplicationProperties** are useful for routing and tracing.
- **ContentType** helps consumers interpret the payload.
- **BinaryData.FromObjectAsJson** is convenient for JSON payloads.

If you use dependency injection, register the client once and reuse it. Creating a new client per request is inefficient.

**Identity and connection configuration**

For production, prefer Microsoft Entra ID authentication with managed identity over connection strings when possible. This reduces secret management and aligns with least privilege.

Typical setup steps:

- Enable a managed identity on the API host and the Function App.
- Grant the identity the appropriate Service Bus data-plane role on the namespace, queue, or topic scope.
- Configure the app to use identity-based connection settings supported by the SDK or binding extension.

Common roles include:

- **Azure Service Bus Data Sender** for publishing.
- **Azure Service Bus Data Receiver** for consuming.
- **Azure Service Bus Data Owner** only when broader management access is required.

> Do not assume a receiver role alone is enough for every trigger scenario. The exact authorization needs depend on the binding, runtime, and whether you use sessions or manual settlement.

For local development, you often use your developer identity through Azure CLI or Visual Studio sign-in. The app still needs the correct configuration values for the Service Bus client or binding; you do not simply replace code with `DefaultAzureCredential` and omit the expected app settings.

Example client setup with `DefaultAzureCredential`:

```csharp
using Azure.Identity;
using Azure.Messaging.ServiceBus;

var fullyQualifiedNamespace = "mybus.servicebus.windows.net";
var client = new ServiceBusClient(fullyQualifiedNamespace, new DefaultAzureCredential());
var sender = client.CreateSender("orders");
```

This pattern works for SDK-based code when the environment and permissions are configured correctly.

**Azure Functions consumer design**

The Function App processes messages asynchronously. The Service Bus trigger is the most common binding for this lab.

A basic trigger function in .NET:

```csharp
using Microsoft.Azure.Functions.Worker;
using Microsoft.Extensions.Logging;
using Azure.Messaging.ServiceBus;

public class ProcessOrder
{
    private readonly ILogger<ProcessOrder> _logger;

    public ProcessOrder(ILogger<ProcessOrder> logger)
    {
        _logger = logger;
    }

    [Function("ProcessOrder")]
    public async Task Run(
        [ServiceBusTrigger("orders", Connection = "ServiceBusConnection")]
        ServiceBusReceivedMessage message,
        FunctionContext context)
    {
        _logger.LogInformation("Processing message {MessageId}", message.MessageId);

        var body = message.Body.ToString();
        _logger.LogInformation("Payload: {Body}", body);

        await Task.CompletedTask;
    }
}
```

Key points:

- The trigger receives a `ServiceBusReceivedMessage`.
- The binding name in `Connection` must match the app setting or identity-based configuration key expected by the extension.
- The function should be fast and resilient.
- If processing is long-running, consider offloading to another service or breaking the work into smaller steps.

**Manual settlement and message completion**

By default, the Functions runtime typically handles message completion for successful executions. Manual settlement is a special configuration path, not the default.

When manual settlement is enabled, you can explicitly settle the message using `ServiceBusMessageActions`. This is useful when you need to:

- Complete only after multiple internal checks succeed.
- Abandon a message to retry later.
- Defer a message for later retrieval.
- Dead-letter a message with a reason and description.

Example with manual settlement:

```csharp
using Microsoft.Azure.Functions.Worker;
using Microsoft.Extensions.Logging;
using Azure.Messaging.ServiceBus;

public class ProcessOrderWithSettlement
{
    private readonly ILogger<ProcessOrderWithSettlement> _logger;

    public ProcessOrderWithSettlement(ILogger<ProcessOrderWithSettlement> logger)
    {
        _logger = logger;
    }

    [Function("ProcessOrderWithSettlement")]
    public async Task Run(
        [ServiceBusTrigger("orders", Connection = "ServiceBusConnection")]
        ServiceBusReceivedMessage message,
        ServiceBusMessageActions messageActions)
    {
        try
        {
            _logger.LogInformation("Processing {MessageId}", message.MessageId);

            // business logic here

            await messageActions.CompleteMessageAsync(message);
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Failed processing {MessageId}", message.MessageId);
            await messageActions.AbandonMessageAsync(message);
        }
    }
}
```

Use manual settlement only when you need explicit control. If you enable it, you must settle every message path intentionally.

**Retry behavior and failure handling**

Service Bus and Functions both contribute to retry behavior, but they solve different problems.

Platform-level behavior:

- The trigger retries delivery when the function fails.
- The message lock may be renewed while processing.
- After repeated failures, the message can move to the DLQ.
- Host-level retries may be configured depending on the Functions runtime and extension version.

Application-level behavior:

- You should catch only exceptions you can handle meaningfully.
- You should let transient failures bubble when retry is appropriate.
- You should avoid swallowing exceptions without recording enough context.

A practical retry strategy:

- Retry transient downstream failures.
- Fail fast on validation errors.
- Dead-letter messages that are structurally invalid or cannot be processed after repeated attempts.
- Use idempotency keys to prevent duplicate side effects.

**Dead-letter queue inspection**

The DLQ is where you look for poison messages and repeated failures. You can inspect it with the SDK or the Azure portal.

Example using the current `ServiceBusReceiver` peek API shape:

```csharp
using Azure.Messaging.ServiceBus;

var client = new ServiceBusClient(connectionString);
var receiver = client.CreateReceiver("orders", new ServiceBusReceiverOptions
{
    SubQueue = SubQueue.DeadLetter
});

IReadOnlyList<ServiceBusReceivedMessage> messages =
    await receiver.PeekMessagesAsync(10, cancellationToken: CancellationToken.None);

foreach (var msg in messages)
{
    Console.WriteLine($"{msg.MessageId} | {msg.Subject} | {msg.DeadLetterReason}");
}
```

Notes:

- `PeekMessagesAsync` is for inspection, not removal.
- Use the DLQ to diagnose repeated failures, schema mismatches, or downstream outages.
- If you need to reprocess a message, you typically read it from the DLQ and resubmit it after fixing the root cause.

**Idempotency and duplicate handling**

Because delivery is at-least-once, your consumer must tolerate duplicates. This is one of the most important reliability concepts in the lab.

Ways to implement idempotency:

- Use a stable business key such as `orderId`.
- Store processed message IDs in a durable store.
- Make downstream writes conditional.
- Use duplicate detection on the Service Bus entity when appropriate.

Example pattern:

```csharp
if (await repository.HasProcessedAsync(message.MessageId))
{
    _logger.LogInformation("Skipping duplicate {MessageId}", message.MessageId);
    return;
}

await repository.MarkProcessedAsync(message.MessageId);
await repository.SaveOrderAsync(order);
```

This prevents double-charging, double-inserting, or double-notifying when the same message is delivered again.

**Correlation and observability**

Traceability is essential in a distributed lab. Add correlation data at the API boundary and propagate it through the message.

Useful metadata:

- `correlationId`
- `messageId`
- `subject`
- `application properties`
- distributed tracing headers when supported

Example logging pattern:

```csharp
_logger.LogInformation(
    "Received message {MessageId} for order {OrderId} with correlation {CorrelationId}",
    message.MessageId,
    orderId,
    correlationId);
```

Operational checks:

- Verify the API returns success only after the send completes.
- Confirm the queue length increases when the consumer is offline.
- Confirm the Function App drains the queue when restarted.
- Confirm failed messages appear in the DLQ after repeated attempts.

**Testing the end-to-end path**

A reliable lab includes both happy-path and failure-path tests.

Suggested test sequence:

1. Send a valid request to the API.
2. Confirm the API returns `202 Accepted` or another queued-work response.
3. Verify the message appears in the queue or topic subscription.
4. Confirm the Function App processes the message.
5. Check logs for correlation ID continuity.
6. Force a downstream failure and observe retries.
7. Confirm poison messages land in the DLQ after repeated failures.

Example HTTP request:

```bash
curl -X POST "https://api.example.com/orders" \
  -H "Content-Type: application/json" \
  -H "x-correlation-id: 7f3d2f1a" \
  -d '{
    "orderId": "ORD-1001",
    "customerId": "CUST-42",
    "amount": 125.50
  }'
```

**Common failure modes**

- **Authentication failure**: the managed identity lacks the correct Service Bus data-plane role.
- **Binding mismatch**: the Function App app setting name does not match the binding configuration.
- **Serialization issues**: the producer and consumer disagree on payload shape or content type.
- **Duplicate processing**: the consumer is not idempotent.
- **Poison messages**: invalid payloads keep failing until they are dead-lettered.
- **Lock lost**: processing takes too long or the message lock is not renewed.
- **Local development mismatch**: the local environment uses different settings than Azure.

**Practical design guidance**

Use a queue when:

- One consumer path is enough.
- You want simple competing-consumer scaling.
- You do not need subscription-based fan-out.

Use a topic when:

- Multiple downstream systems need the same event.
- Each subscriber has different filtering or processing rules.
- You want to decouple producers from many consumers.

Use manual settlement when:

- You need precise control over completion.
- You must coordinate multiple operations before acknowledging the message.
- You are handling special workflows such as deferred processing or custom dead-lettering.

Use automatic settlement when:

- The function can safely succeed or fail as a unit.
- You want simpler code and fewer moving parts.

**Exam-focused takeaways**

- The API should publish work quickly and return before processing completes.
- Service Bus provides buffering, retries, and dead-lettering for reliability.
- Azure Functions is a good consumer for asynchronous message processing.
- At-least-once delivery means you must design for duplicates.
- Managed identity is preferred over connection strings when feasible.
- Manual settlement is optional and requires explicit configuration and code support.
- DLQ inspection is a key troubleshooting skill.
