## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Service Bus trigger fundamentals**

An Azure Functions **Service Bus trigger** runs your code when a message arrives on an Azure Service Bus queue or subscription. It is a common choice when you need decoupled, asynchronous processing with durable delivery and back-pressure handling.

Service Bus is a message broker that supports **queues** for point-to-point messaging and **topics/subscriptions** for publish-subscribe fan-out. The trigger lets you react to messages without polling, while the Functions runtime handles message acquisition, lock management, and invocation.

- Use a queue trigger when one consumer should process each message.
- Use a topic/subscription trigger when multiple independent consumers need the same event stream.
- Prefer Service Bus when you need dead-lettering, duplicate detection, sessions, or scheduled delivery.

**Binding model and function signatures**

The trigger binding maps a Service Bus message to a function parameter. In the in-process .NET model, you commonly bind to `string`, `byte[]`, or `ServiceBusReceivedMessage`. In the isolated worker model, you typically use `ServiceBusReceivedMessage` and `ServiceBusMessageActions` from the Azure.Messaging.ServiceBus SDK.

A minimal C# example for a queue trigger looks like this:

```csharp
using Azure.Messaging.ServiceBus;
using Microsoft.Azure.Functions.Worker;
using Microsoft.Extensions.Logging;

public class ProcessOrder
{
    private readonly ILogger<ProcessOrder> _logger;

    public ProcessOrder(ILogger<ProcessOrder> logger)
    {
        _logger = logger;
    }

    [Function(nameof(ProcessOrder))]
    public void Run(
        [ServiceBusTrigger("orders", Connection = "ServiceBusConnection")]
        ServiceBusReceivedMessage message)
    {
        _logger.LogInformation("MessageId: {MessageId}", message.MessageId);
        _logger.LogInformation("Body: {Body}", message.Body.ToString());
    }
}
```

For topic/subscription processing, you specify both the topic and subscription names:

```csharp
[Function(nameof(ProcessOrderEvents))]
public void Run(
    [ServiceBusTrigger("orders-topic", "billing", Connection = "ServiceBusConnection")]
    ServiceBusReceivedMessage message)
{
    // Process event
}
```

- `Connection` points to an app setting that contains the Service Bus connection string or identity-based configuration.
- The trigger parameter receives the message payload and metadata such as `MessageId`, `CorrelationId`, `Subject`, and application properties.
- Use `ServiceBusReceivedMessage` when you need headers, delivery count, or lock-related metadata.

**Message settlement and failure behavior**

Service Bus uses **peek-lock** semantics for triggers. The runtime receives a message, locks it, and invokes your function. If the function completes successfully, the runtime completes the message when auto-complete is enabled. If the function fails, the message is not completed and becomes available again after the lock expires or is abandoned according to the runtime and binding behavior.

This is an important exam distinction:

- Service Bus redelivery is driven by **Service Bus delivery semantics**, not by Azure Functions host retry policies alone.
- A failed invocation can be retried by Service Bus after lock expiration or explicit abandonment.
- Duplicate delivery is always possible in at-least-once systems, so your handler should be **idempotent**.

> Do not assume a single failure means exactly one retry. Message redelivery depends on lock duration, delivery count, dead-letter thresholds, and whether the runtime or your code settles the message.

A practical idempotency pattern is to store processed message IDs in durable storage:

```csharp
public async Task<bool> HasAlreadyProcessedAsync(string messageId)
{
    // Example: check a database or cache for a processed marker
    return await _repository.ExistsAsync(messageId);
}

public async Task MarkProcessedAsync(string messageId)
{
    await _repository.InsertAsync(messageId);
}
```

- Use a business key or `MessageId` to detect duplicates.
- Make side effects safe to repeat, such as using upserts or transactional writes.
- Treat retries as normal, not exceptional.

**Auto-complete and manual settlement**

By default, many Service Bus trigger setups use **auto-complete**, meaning the runtime completes the message after the function returns successfully. If the function throws, the message is not completed.

Manual settlement is useful when you need to decide whether to complete, abandon, defer, or dead-letter a message based on business rules. In the isolated worker model, you can use `ServiceBusMessageActions` to settle messages manually when supported by the extension version and configuration.

Example with manual settlement in the isolated worker model:

```csharp
using Azure.Messaging.ServiceBus;
using Microsoft.Azure.Functions.Worker;

public class ValidatePayment
{
    [Function(nameof(ValidatePayment))]
    public async Task Run(
        [ServiceBusTrigger("payments", Connection = "ServiceBusConnection")]
        ServiceBusReceivedMessage message,
        ServiceBusMessageActions messageActions)
    {
        if (message.ApplicationProperties.TryGetValue("valid", out var valid) && valid is bool isValid && isValid)
        {
            await messageActions.CompleteMessageAsync(message);
            return;
        }

        await messageActions.DeadLetterMessageAsync(
            message,
            deadLetterReason: "ValidationFailed",
            deadLetterErrorDescription: "Payment message failed validation.");
    }
}
```

- Use **complete** when processing succeeds and no further action is needed.
- Use **abandon** when you want the message to be retried sooner.
- Use **dead-letter** when the message is invalid or cannot be processed after validation.
- Use **defer** when you want to postpone processing and retrieve the message later by sequence number.

**Sessions and ordered processing**

Service Bus **sessions** provide ordered, correlated processing for messages that share a `SessionId`. A session-enabled queue or subscription requires session-aware processing on the consumer side. A non-session-enabled trigger cannot listen to a session-enabled entity.

Sessions are useful when you need:

- Ordered processing per customer, order, or workflow instance.
- Exclusive handling of a related message group.
- State that persists across multiple messages in the same session.

Session processing changes scaling behavior because the runtime must accept and lock a session before it can process messages from that session. That means concurrency is constrained by active sessions, not just by message count.

A conceptual session-enabled trigger example:

```csharp
[Function(nameof(ProcessSession))]
public async Task Run(
    [ServiceBusTrigger("orders", Connection = "ServiceBusConnection")]
    ServiceBusReceivedMessage message,
    ServiceBusSessionMessageActions sessionActions)
{
    // Process messages in session order
}
```

- Ensure the entity itself is session-enabled.
- Use the same `SessionId` for related messages.
- Expect lower parallelism per session because ordering is preserved.

**Host configuration and tuning**

Service Bus trigger behavior is tuned in `host.json`. The exact schema can vary by extension version, so verify the version used by your function app. Newer extension versions commonly use properties directly under `extensions.serviceBus`, while older versions used nested options such as `messageHandlerOptions`.

A typical newer-style configuration looks like this:

```json
{
  "version": "2.0",
  "extensions": {
    "serviceBus": {
      "prefetchCount": 10,
      "maxConcurrentCalls": 16,
      "autoCompleteMessages": true,
      "maxAutoLockRenewalDuration": "00:05:00"
    }
  }
}
```

- `maxConcurrentCalls` controls how many messages the runtime processes in parallel per instance.
- `prefetchCount` increases throughput by buffering messages locally before invocation.
- `autoCompleteMessages` controls whether the runtime completes messages automatically after success.
- `maxAutoLockRenewalDuration` limits how long the runtime renews the message lock while your function runs.

Older extension versions may use a shape like this:

```json
{
  "version": "2.0",
  "extensions": {
    "serviceBus": {
      "messageHandlerOptions": {
        "maxConcurrentCalls": 16,
        "autoComplete": true,
        "maxAutoRenewDuration": "00:05:00"
      }
    }
  }
}
```

- Check the extension package version before copying a sample.
- Match configuration names to the worker model and extension generation you are using.
- Avoid mixing old and new schemas in the same app.

**Lock duration, renewal, and duplicates**

When a message is received in peek-lock mode, Service Bus locks it for a limited time. If processing takes longer than the lock window, the message can become visible again and be delivered again. The runtime can renew the lock while the function is still running, up to the configured maximum renewal duration.

Important implications:

- Lock renewal reduces the chance of duplicate delivery during long processing.
- Lock renewal does not eliminate duplicates entirely.
- At-least-once delivery means you still need idempotent handlers.
- If processing regularly exceeds the lock window, consider breaking the work into smaller steps or offloading long-running work.

A practical pattern is to enqueue a follow-up message for heavy work instead of holding the trigger execution open:

```csharp
[Function(nameof(StartWork))]
public async Task Run(
    [ServiceBusTrigger("incoming", Connection = "ServiceBusConnection")]
    ServiceBusReceivedMessage message,
    [ServiceBusOutput("work-items", Connection = "ServiceBusConnection")] IAsyncCollector<string> output)
{
    await output.AddAsync(message.Body.ToString());
}
```

**Dead-lettering and poison messages**

A **dead-letter queue** (DLQ) stores messages that cannot be processed successfully. Service Bus can dead-letter messages automatically after too many delivery attempts, or your function can dead-letter them explicitly when the payload is invalid.

Use the DLQ when:

- The message schema is invalid.
- Required business data is missing.
- The message repeatedly fails validation.
- Processing would be unsafe or meaningless.

Operationally, you should monitor the DLQ and build a replay process for recoverable messages.

- Inspect dead-letter reason and description.
- Fix the root cause before replaying messages.
- Re-enqueue only after validation.

**Configuration with connection strings and managed identity**

The trigger needs access to the Service Bus namespace. You can configure this with a connection string or with managed identity, depending on your hosting and extension support.

Connection string app setting example:

```json
{
  "Values": {
    "ServiceBusConnection": "Endpoint=sb://contoso.servicebus.windows.net/;SharedAccessKeyName=RootManageSharedAccessKey;SharedAccessKey=..."
  }
}
```

Managed identity is preferred in production when supported because it reduces secret management. In that model, you typically configure the namespace endpoint and assign the function app a role such as **Azure Service Bus Data Receiver**.

- Use connection strings for quick setup and local development.
- Use managed identity for production security.
- Grant the least privilege needed for the trigger and any output bindings.

**Local development and testing**

For local development, you typically use `local.settings.json` to store the Service Bus connection setting. The Functions runtime reads this file locally, but it is not deployed.

Example:

```json
{
  "IsEncrypted": false,
  "Values": {
    "AzureWebJobsStorage": "UseDevelopmentStorage=true",
    "FUNCTIONS_WORKER_RUNTIME": "dotnet-isolated",
    "ServiceBusConnection": "Endpoint=sb://localhost/;SharedAccessKeyName=RootManageSharedAccessKey;SharedAccessKey=..."
  }
}
```

Useful CLI commands:

```bash
func start
az servicebus namespace create -g rg-app -n contoso-sb
az servicebus queue create -g rg-app --namespace-name contoso-sb -n orders
az servicebus topic create -g rg-app --namespace-name contoso-sb -n orders-topic
az servicebus topic subscription create -g rg-app --namespace-name contoso-sb --topic-name orders-topic -n billing
```

- Use Azure Functions Core Tools to run the app locally.
- Use Azure CLI to provision namespaces, queues, topics, and subscriptions.
- Test with sample messages before deploying to production.

**Operational considerations**

Service Bus triggers are designed for reliability, but you still need to design for failure, scale, and observability. Logging message identifiers and correlation data helps you trace a message across services.

Good practices include:

- Log `MessageId`, `CorrelationId`, and business keys.
- Keep handlers short and focused.
- Use retries carefully for transient downstream failures.
- Prefer idempotent writes to databases and APIs.
- Monitor delivery count, dead-letter volume, and function failures.

A simple logging example:

```csharp
_logger.LogInformation(
    "Processing order {OrderId} message {MessageId} correlation {CorrelationId}",
    orderId,
    message.MessageId,
    message.CorrelationId);
```

**Common exam points**

- A Service Bus trigger listens to a queue or subscription and processes messages asynchronously.
- Message redelivery is governed by Service Bus lock and delivery behavior, not by generic host retries alone.
- Session-enabled entities require session-aware processing.
- Auto-complete is a binding/runtime behavior, not a universal trigger attribute setting.
- Long-running handlers need lock renewal awareness and idempotency.
- Dead-letter queues are the correct place for poison messages.

**Quick comparison**

| Scenario | Recommended approach |
|---|---|
| One consumer, simple work item | Queue trigger |
| Fan-out to multiple consumers | Topic/subscription trigger |
| Ordered per-customer processing | Session-enabled trigger |
| Invalid or unrecoverable message | Dead-letter it |
| Long-running processing | Renew locks or split the work |

**Troubleshooting checklist**

1. Verify the function app has the correct Service Bus connection or identity permissions.
2. Confirm the queue or subscription exists and matches the binding names.
3. Check whether the entity is session-enabled and whether the trigger supports sessions.
4. Inspect function logs for exceptions and delivery count growth.
5. Review `host.json` for extension-version-compatible settings.
6. Check the DLQ for poison messages and settlement failures.

**Summary**

Service Bus triggers let Azure Functions react to durable messages with built-in support for retries, dead-lettering, and scaling. The key exam themes are binding syntax, settlement behavior, session requirements, lock renewal, and idempotent design. If you remember that Service Bus delivery semantics drive redelivery and that session-enabled entities require session-aware consumers, you will avoid the most common mistakes.
