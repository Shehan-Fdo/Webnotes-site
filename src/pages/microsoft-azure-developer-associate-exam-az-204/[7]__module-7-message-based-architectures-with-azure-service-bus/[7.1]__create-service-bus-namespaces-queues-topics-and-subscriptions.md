## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Azure Service Bus overview**

Azure Service Bus is a managed message broker for decoupled application communication. You use it when producers and consumers should not depend on each other’s availability or processing speed. It supports both point-to-point messaging with **queues** and publish/subscribe messaging with **topics** and **subscriptions**.

Service Bus is a good fit when you need:
- Reliable delivery with retries and dead-lettering.
- Message ordering patterns such as sessions.
- Competing consumers for load distribution.
- Fan-out to multiple downstream processors.
- Enterprise messaging features such as duplicate detection and scheduled delivery.

A typical flow is:
- An application sends a message to a queue or topic.
- Service Bus stores the message durably.
- A receiver processes the message and completes it.
- If processing fails, the message can be retried, deferred, or dead-lettered.

**Core resource hierarchy**

Service Bus resources are organized in a hierarchy:
- **Namespace**: The top-level container for messaging entities.
- **Queue**: A single logical destination for messages.
- **Topic**: A publish endpoint that can have multiple subscriptions.
- **Subscription**: A virtual queue attached to a topic.

A namespace is the security and management boundary. You create queues, topics, and subscriptions inside a namespace, and clients connect to the namespace endpoint.

**Namespace creation and configuration**

You usually create a namespace first, then add entities. In Azure, a namespace defines:
- The DNS endpoint, such as `<namespace>.servicebus.windows.net`.
- The pricing tier and available features.
- Authentication and authorization settings.
- Networking controls such as public access, private endpoints, and firewall rules.

Common tiers include:
- **Basic**: Queue-only scenarios with limited features.
- **Standard**: Queues and topics with most common messaging features.
- **Premium**: Dedicated resources, predictable performance, and advanced isolation.

Choose the tier based on feature needs:
- Use **Basic** for simple queue-based workloads.
- Use **Standard** for most application messaging and pub/sub.
- Use **Premium** when you need isolation, higher throughput consistency, or enterprise networking requirements.

> Premium is typically the right choice when you need predictable performance or strict isolation, but it costs more than Standard.

**Creating a namespace with Azure CLI**

You can create a namespace with the Azure CLI and then add entities afterward.

```bash
az servicebus namespace create \
  --resource-group rg-app-messaging \
  --name sbns-orders-prod \
  --location eastus \
  --sku Standard
```

Useful follow-up commands:
```bash
az servicebus queue create \
  --resource-group rg-app-messaging \
  --namespace-name sbns-orders-prod \
  --name orders-queue

az servicebus topic create \
  --resource-group rg-app-messaging \
  --namespace-name sbns-orders-prod \
  --name orders-topic

az servicebus topic subscription create \
  --resource-group rg-app-messaging \
  --namespace-name sbns-orders-prod \
  --topic-name orders-topic \
  --name billing-subscription
```

**Queues**

A queue is a first-in, first-out style destination for messages, although actual processing order can vary with retries, sessions, and lock behavior. A queue is best when one consumer group should process each message once.

Use a queue when:
- One message should be handled by one consumer.
- You want load balancing across multiple instances.
- You do not need fan-out to multiple independent subscribers.

Important queue behaviors:
- **Peek-lock** is the default safe processing model in many SDKs. The receiver locks the message, processes it, then completes it.
- **Auto-forwarding** can route messages from one entity to another.
- **Dead-letter queue** stores messages that cannot be processed or delivered.
- **Duplicate detection** helps prevent repeated processing of the same message.
- **Time to live** controls message expiration.

Example queue creation with common settings:
```bash
az servicebus queue create \
  --resource-group rg-app-messaging \
  --namespace-name sbns-orders-prod \
  --name payments-queue \
  --enable-dead-lettering-on-message-expiration true \
  --max-delivery-count 10 \
  --default-message-time-to-live P7D
```

**Topics and subscriptions**

A topic supports publish/subscribe messaging. Producers send one message to the topic, and Service Bus copies that message into each matching subscription. This is the right model when multiple independent systems need the same event.

Use a topic when:
- Multiple consumers need the same message.
- Each consumer has its own processing logic.
- You want filtering so subscriptions receive only relevant messages.

A subscription behaves like a queue attached to a topic:
- Each subscription receives its own copy of a message.
- Subscriptions can have filters and actions.
- A subscription can dead-letter messages independently.
- Consumers read from subscriptions, not directly from the topic.

Example topic and subscription creation:
```bash
az servicebus topic create \
  --resource-group rg-app-messaging \
  --namespace-name sbns-orders-prod \
  --name orders-topic

az servicebus topic subscription create \
  --resource-group rg-app-messaging \
  --namespace-name sbns-orders-prod \
  --topic-name orders-topic \
  --name shipping-subscription
```

**Filtering subscriptions**

Filters let you route messages to specific subscriptions based on message properties. This is useful when one topic carries multiple event types.

Common filter concepts:
- **SQL filters** evaluate message properties with SQL-like expressions.
- **Correlation filters** match on a small set of properties and are simpler for common routing cases.
- **Actions** can modify properties when a filter matches.

For Service Bus SQL filters, use the system property name `sys.Label` for compatibility in filter expressions. In the current SDK, the message model exposes `Subject` as the application-facing property, but filter syntax still uses the Service Bus system property naming convention.

Example SQL filter:
```sql
sys.Label = 'OrderCreated'
```

Example subscription rule creation with Azure CLI:
```bash
az servicebus topic subscription rule create \
  --resource-group rg-app-messaging \
  --namespace-name sbns-orders-prod \
  --topic-name orders-topic \
  --subscription-name billing-subscription \
  --name order-created-rule \
  --filter-sql-expression "sys.Label = 'OrderCreated'"
```

A practical pattern is:
- Set `Subject` on the message in code for application semantics.
- Use `sys.Label` in SQL filters when you need Service Bus-compatible filtering behavior.

**Creating entities with the Azure portal**

The portal is useful for quick setup and validation:
1. Create a namespace and choose the tier.
2. Open the namespace and create a queue or topic.
3. Configure entity settings such as dead-lettering, sessions, duplicate detection, and TTL.
4. Add subscriptions under a topic.
5. Add rules and filters if you need selective routing.

Portal creation is convenient for learning and inspection, but CLI or infrastructure-as-code is better for repeatable deployments.

**Creating entities with ARM or Bicep**

Infrastructure as code helps you version and repeat messaging setup. Bicep is a common choice for Azure deployments.

Example Bicep snippet:
```bicep
param location string = resourceGroup().location
param namespaceName string = 'sbns-orders-prod'
param queueName string = 'orders-queue'
param topicName string = 'orders-topic'
param subscriptionName string = 'billing-subscription'

resource sbNamespace 'Microsoft.ServiceBus/namespaces@2022-10-01-preview' = {
  name: namespaceName
  location: location
  sku: {
    name: 'Standard'
    tier: 'Standard'
  }
}

resource queue 'Microsoft.ServiceBus/namespaces/queues@2022-10-01-preview' = {
  parent: sbNamespace
  name: queueName
  properties: {
    deadLetteringOnMessageExpiration: true
    maxDeliveryCount: 10
    defaultMessageTimeToLive: 'P7D'
  }
}

resource topic 'Microsoft.ServiceBus/namespaces/topics@2022-10-01-preview' = {
  parent: sbNamespace
  name: topicName
}

resource subscription 'Microsoft.ServiceBus/namespaces/topics/subscriptions@2022-10-01-preview' = {
  parent: topic
  name: subscriptionName
  properties: {
    maxDeliveryCount: 10
    deadLetteringOnFilterEvaluationExceptions: true
  }
}
```

Use ARM or Bicep when:
- You need consistent environments.
- You want source-controlled infrastructure.
- You deploy across dev, test, and production.

**Authentication and authorization**

Service Bus supports Microsoft Entra ID and shared access signatures (SAS). Prefer Entra ID when possible because it integrates with Azure identity and reduces key management overhead. Use SAS when you need key-based access or must support legacy integration patterns.

Common authorization approaches:
- **Microsoft Entra ID**: Recommended for modern applications and managed identities.
- **SAS**: Useful for scoped access with shared access policies.
- **Connection strings**: Often used in quick starts, but less ideal for long-term secret management.

Example of a connection string in application settings:
```bash
Endpoint=sb://sbns-orders-prod.servicebus.windows.net/;SharedAccessKeyName=send-policy;SharedAccessKey=...
```

In production, store secrets in:
- Azure Key Vault.
- Managed identity-based configuration where possible.
- App Service application settings or container secrets only when necessary.

**.NET client examples**

The modern .NET SDK is `Azure.Messaging.ServiceBus`. Use `ServiceBusClient` to create senders and receivers.

Send to a queue:
```csharp
using Azure.Messaging.ServiceBus;

var client = new ServiceBusClient(connectionString);
var sender = client.CreateSender("orders-queue");

var message = new ServiceBusMessage("order-123")
{
    Subject = "OrderCreated",
    ContentType = "application/json"
};

await sender.SendMessageAsync(message);
```

Receive from a queue:
```csharp
using Azure.Messaging.ServiceBus;

var client = new ServiceBusClient(connectionString);
var processor = client.CreateProcessor("orders-queue", new ServiceBusProcessorOptions
{
    AutoCompleteMessages = false,
    MaxConcurrentCalls = 4
});

processor.ProcessMessageAsync += async args =>
{
    Console.WriteLine(args.Message.Subject);
    await args.CompleteMessageAsync(args.Message);
};

processor.ProcessErrorAsync += args =>
{
    Console.WriteLine(args.Exception);
    return Task.CompletedTask;
};

await processor.StartProcessingAsync();
```

Receive from a topic subscription:
```csharp
using Azure.Messaging.ServiceBus;

var client = new ServiceBusClient(connectionString);
var receiver = client.CreateReceiver("orders-topic", "billing-subscription");

ServiceBusReceivedMessage message = await receiver.ReceiveMessageAsync();
if (message != null)
{
    Console.WriteLine(message.Subject);
    await receiver.CompleteMessageAsync(message);
}
```

Key SDK points:
- `CreateSender(queueOrTopicName)` sends to a queue or topic.
- `CreateReceiver(queueName)` receives from a queue.
- `CreateReceiver(topicName, subscriptionName)` receives from a topic subscription.
- `CreateProcessor(...)` is useful for concurrent message handling and automatic pump-style processing.

**Message properties and routing**

Service Bus messages have application properties and system properties. Use them intentionally:
- **Subject**: A convenient application-facing label in the SDK.
- **ContentType**: Describes the payload format, such as `application/json`.
- **CorrelationId**: Helps with tracing and request-response patterns.
- **MessageId**: Supports duplicate detection and idempotency.
- **SessionId**: Enables ordered, session-aware processing.

For routing, prefer stable properties:
- Use `Subject` or custom application properties for business meaning.
- Use filters on topic subscriptions to route messages.
- Use `CorrelationId` when you need to group related messages.

Example with custom properties:
```csharp
var message = new ServiceBusMessage(BinaryData.FromString("{\"orderId\":123}"))
{
    Subject = "OrderCreated",
    ContentType = "application/json",
    CorrelationId = "order-123"
};

message.ApplicationProperties["region"] = "us-east";
message.ApplicationProperties["priority"] = "high";
```

**Reliability features to configure**

Service Bus includes several features that affect how you design entities:
- **Dead-lettering**: Moves poison messages to a separate subqueue for inspection.
- **Max delivery count**: Limits retries before dead-lettering.
- **TTL**: Removes stale messages.
- **Duplicate detection**: Prevents repeated processing of the same logical message.
- **Sessions**: Preserve ordered processing for related messages.
- **Scheduled messages**: Delay delivery until a future time.

A practical queue configuration often includes:
- Dead-lettering enabled.
- A sensible delivery count, such as 5 to 10.
- TTL aligned with business value.
- Duplicate detection when message IDs are stable.

**Common design choices**

Choose the right entity based on the communication pattern:
- **Queue** for one consumer group and work distribution.
- **Topic** for event distribution to multiple consumers.
- **Subscription filters** when consumers need only a subset of events.
- **Sessions** when order matters within a logical conversation.
- **Dead-lettering** when you need to isolate bad messages for later analysis.

A simple topology example:
- `orders-queue` for background order processing.
- `orders-topic` for order events.
- `billing-subscription` for billing events.
- `shipping-subscription` for shipping events.
- `support-subscription` for customer support notifications.

**Operational considerations**

When you create Service Bus resources, also think about:
- Naming conventions that identify environment and workload.
- Access control using least privilege.
- Monitoring with Azure Monitor and diagnostic logs.
- Message size limits and payload design.
- Throughput needs and partitioning where appropriate.

Practical naming example:
- Namespace: `sbns-orders-prod`
- Queue: `orders-queue`
- Topic: `orders-topic`
- Subscription: `billing-subscription`

**Troubleshooting and validation**

If messages do not flow as expected, check:
- The namespace exists in the correct region and subscription.
- The sender and receiver use the same namespace and entity name.
- The authorization policy has the correct rights: `Send`, `Listen`, or `Manage`.
- The subscription rule matches the message properties.
- The receiver completes or abandons messages correctly.
- Dead-letter queues contain poison messages.

Useful CLI inspection commands:
```bash
az servicebus namespace show \
  --resource-group rg-app-messaging \
  --name sbns-orders-prod

az servicebus queue show \
  --resource-group rg-app-messaging \
  --namespace-name sbns-orders-prod \
  --name orders-queue

az servicebus topic subscription show \
  --resource-group rg-app-messaging \
  --namespace-name sbns-orders-prod \
  --topic-name orders-topic \
  --name billing-subscription
```
