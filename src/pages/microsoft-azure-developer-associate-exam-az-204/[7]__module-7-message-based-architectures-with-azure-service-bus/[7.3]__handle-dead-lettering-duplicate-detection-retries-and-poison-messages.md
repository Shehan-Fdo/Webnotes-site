## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Messaging reliability overview**

Reliable message processing is about protecting your system from transient failures, permanent failures, duplicate deliveries, and messages that cannot be processed successfully. In Azure, the main tools are **Azure Service Bus** and **Azure Queue Storage**, but they behave differently:

- Service Bus supports richer brokered messaging features such as **dead-letter queues (DLQ)**, **duplicate detection**, **message locks**, and explicit settlement.
- Queue Storage is simpler and uses **visibility timeout** and **dequeue count** to manage retries and poison messages.
- Your application should distinguish between:
  - **Transient failures**: temporary issues such as network blips, throttling, or downstream service unavailability.
  - **Permanent failures**: invalid payloads, schema mismatches, or business-rule violations that will not succeed on retry.

A good mental model is: retry transient failures, dead-letter or poison-handle permanent failures, and prevent duplicates before they create side effects.

**Message lifecycle and decision flow**

```text
Send message
   |
   v
Receive message
   |
   v
Process successfully? ---- yes ---> Complete / delete message
   |
   no
   v
Transient failure? ---- yes ---> Retry with backoff/jitter ---> Receive again
   |
   no
   v
Permanent failure or max attempts reached?
   |
   v
Dead-letter / poison queue / manual investigation
```

This flow matters because the right action depends on whether the message can succeed later. If you retry a permanent failure, you waste capacity and may create retry storms. If you dead-letter a transient failure too early, you lose a message that could have succeeded after a short delay.

**Azure Service Bus dead-lettering**

Service Bus moves messages to a **dead-letter queue** when they cannot be delivered or processed according to configured rules. The DLQ is a subqueue associated with the original entity, not a separate queue you create manually.

Common reasons for dead-lettering include:

- The application explicitly calls dead-letter settlement.
- The message exceeds the **max delivery count**.
- The message expires before it is processed.
- The message cannot be delivered because of size, header, or protocol issues.
- A subscription or filter rule rejects the message in some scenarios.

Dead-lettering is useful when you want to preserve the original message for later inspection instead of losing it. It is especially important for messages that fail due to bad data, schema drift, or downstream contract changes.

> A dead-lettered message is not automatically fixed or retried by Service Bus. You must inspect, correct, and resubmit it yourself if you want it processed again.

**Inspecting and resubmitting dead-letter messages**

You can inspect DLQ messages with Service Bus tools or code, then decide whether to resubmit them to the original queue/topic or to a repair queue.

Typical approaches:

- Use **Service Bus Explorer** or Azure portal tools to browse the DLQ.
- Use SDK code to receive from the DLQ path and log the body, headers, and dead-letter reason.
- After fixing the payload or downstream issue, send a new message to the original entity.

Example .NET code to read from a DLQ and resubmit after inspection:

```csharp
using Azure.Messaging.ServiceBus;

var connectionString = "<service-bus-connection-string>";
var queueName = "orders";

await using var client = new ServiceBusClient(connectionString);
var dlqReceiver = client.CreateReceiver(queueName, new ServiceBusReceiverOptions
{
    SubQueue = SubQueue.DeadLetter
});
var sender = client.CreateSender(queueName);

ServiceBusReceivedMessage message = await dlqReceiver.ReceiveMessageAsync(TimeSpan.FromSeconds(5));

if (message != null)
{
    Console.WriteLine($"Dead-letter reason: {message.DeadLetterReason}");
    Console.WriteLine($"Dead-letter error description: {message.DeadLetterErrorDescription}");
    Console.WriteLine($"Body: {message.Body}");

    // After validation or repair, resubmit a new message
    var repaired = new ServiceBusMessage(message.Body)
    {
        ContentType = message.ContentType,
        CorrelationId = message.CorrelationId
    };

    await sender.SendMessageAsync(repaired);
    await dlqReceiver.CompleteMessageAsync(message);
}
```

Practical DLQ handling patterns:

- **Quarantine first**: store the message and metadata for investigation before resubmitting.
- **Repair and replay**: fix the payload or mapping issue, then send a new message.
- **Discard with audit**: if the message is invalid and no longer relevant, archive the details and remove it from the DLQ.

**Service Bus lock duration and lock renewal**

Service Bus uses a **peek-lock** model by default for many consumers. When you receive a message, Service Bus locks it for a period so no other consumer processes it concurrently. You must complete, abandon, dead-letter, or defer the message before the lock expires.

Key points:

- **Lock duration** is configured on the queue or subscription.
- If processing takes longer than the lock duration, the lock can expire and the message becomes available again.
- If your handler is long-running, you should **renew the lock** before it expires.
- Lock renewal helps avoid duplicate processing caused by the same message being delivered again while the first handler is still working.

Why this matters with retries:

- If your handler retries an internal dependency call for too long without renewing the lock, the message may reappear and be processed twice.
- If you use application-level retries, keep the total retry time within the lock window or renew the lock explicitly.
- For long-running workflows, consider moving the work to a durable process and completing the message only after you persist the state.

Example of configuring a queue lock duration:

```bash
az servicebus queue create \
  --resource-group rg-app \
  --namespace-name sb-prod \
  --name orders \
  --lock-duration PT1M
```

Practical guidance:

- Use a shorter lock duration when processing is quick and predictable.
- Use lock renewal when processing time varies or depends on external systems.
- Avoid holding a lock while waiting on long, unreliable operations unless you can renew safely.

**Duplicate detection**

Duplicate detection prevents Service Bus from accepting messages that have the same **MessageId** within a configured time window. This is useful when producers may retry sends and you want to avoid duplicate side effects.

Important design rules:

- The entity must have **duplicate detection enabled before you send** messages that rely on it.
- Duplicate detection is based on the message identifier and the configured detection window.
- The detection window has practical limits and tradeoffs:
  - A longer window reduces duplicate risk but increases broker memory and operational overhead.
  - A shorter window reduces overhead but allows older duplicates through.
- Duplicate detection is not a substitute for idempotent consumers. You should still design handlers to tolerate repeated delivery.

Example queue creation with duplicate detection enabled:

```bash
az servicebus queue create \
  --resource-group rg-app \
  --namespace-name sb-prod \
  --name orders \
  --requires-duplicate-detection true \
  --duplicate-detection-history-time-window PT10M
```

Example producer code:

```csharp
var message = new ServiceBusMessage(BinaryData.FromString("{\"orderId\":123}"))
{
    MessageId = "order-123",
    ContentType = "application/json"
};

await sender.SendMessageAsync(message);
```

Design tradeoffs:

- Use duplicate detection when the producer may retry sends after timeouts or ambiguous failures.
- Do not rely on it for business-level uniqueness across all time unless the window is sufficient.
- Combine it with idempotency keys in your application database for stronger protection.

**Retries and backoff strategy**

Retries are appropriate for transient failures such as throttling, temporary network loss, or a briefly unavailable downstream API. The goal is to give the system time to recover without overwhelming it.

Best practices:

- Use **exponential backoff** so each retry waits longer than the previous one.
- Add **jitter** to randomize retry delays and reduce synchronized retry storms.
- Cap the maximum retry count and total retry duration.
- Retry only on errors that are likely to succeed later.
- Avoid retrying permanent failures such as validation errors or unauthorized access.

Why jitter matters:

- If many instances fail at the same time, fixed retry intervals can cause a thundering herd.
- Jitter spreads retries across time and reduces load spikes on the dependency.

Example retry policy in .NET using `Polly`:

```csharp
using Polly;
using Polly.Retry;

AsyncRetryPolicy retryPolicy = Policy
    .Handle<HttpRequestException>()
    .Or<TaskCanceledException>()
    .WaitAndRetryAsync(
        retryCount: 5,
        sleepDurationProvider: attempt =>
        {
            var baseDelay = TimeSpan.FromSeconds(Math.Pow(2, attempt));
            var jitter = TimeSpan.FromMilliseconds(Random.Shared.Next(0, 250));
            return baseDelay + jitter;
        });

await retryPolicy.ExecuteAsync(async () =>
{
    await CallDownstreamServiceAsync();
});
```

Retry decision examples:

- Retry `429 Too Many Requests`, `503 Service Unavailable`, and transient socket failures.
- Do not retry `400 Bad Request` caused by invalid payloads.
- Be careful with retries around non-idempotent operations such as charging a card or creating a record without a uniqueness guard.

**Avoiding retry storms**

Retry storms happen when many clients retry at once and amplify an outage. This can make a temporary issue worse.

To reduce the risk:

- Use exponential backoff with jitter.
- Limit concurrency in consumers.
- Add circuit breakers for repeated downstream failures.
- Prefer queue-based buffering so producers do not directly hammer a failing service.
- Monitor retry rates and dead-letter rates as early warning signals.

A common exam scenario is a service that starts timing out under load. The correct response is usually not “increase retries indefinitely.” Instead, you should back off, limit concurrency, and let the system recover.

**Azure Queue Storage poison messages**

Queue Storage does not have a Service Bus-style DLQ by default. Instead, it uses **visibility timeout** and **dequeue count** to help you detect poison messages.

How it works:

- When a message is retrieved, it becomes invisible for the visibility timeout period.
- If your worker deletes the message before the timeout expires, processing succeeds.
- If the worker fails to delete it, the message becomes visible again after the timeout.
- Each retrieval increments the **dequeue count**.
- When the dequeue count exceeds your threshold, you treat the message as a **poison message** and move it to a separate poison queue or storage location.

This differs from Service Bus in an important way:

- Service Bus supports explicit settlement and built-in DLQ semantics.
- Queue Storage relies on delete-after-success and reappearance after visibility timeout.
- Queue Storage poison handling is usually implemented by your application.

Example Queue Storage processing pattern in .NET:

```csharp
using Azure.Storage.Queues;

var client = new QueueClient("<connection-string>", "orders");
await client.CreateIfNotExistsAsync();

var response = await client.ReceiveMessagesAsync(maxMessages: 1, visibilityTimeout: TimeSpan.FromSeconds(30));

foreach (var msg in response.Value)
{
    try
    {
        Console.WriteLine(msg.MessageText);

        // Process message here

        await client.DeleteMessageAsync(msg.MessageId, msg.PopReceipt);
    }
    catch
    {
        // Let it reappear after visibility timeout, or move to poison queue after threshold
    }
}
```

Poison-message handling pattern:

1. Read the message and inspect `DequeueCount`.
2. Process the message.
3. Delete it on success.
4. If it fails repeatedly, copy it to a poison queue with diagnostic metadata.
5. Delete the original message to stop endless retries.

Example poison-queue threshold logic:

```csharp
if (msg.DequeueCount >= 5)
{
    var poisonQueue = new QueueClient("<connection-string>", "orders-poison");
    await poisonQueue.SendMessageAsync(msg.MessageText);
    await client.DeleteMessageAsync(msg.MessageId, msg.PopReceipt);
}
```

**Service Bus vs Queue Storage: settlement and poison handling**

| Capability | Service Bus | Queue Storage |
|---|---|---|
| Message lock | Peek-lock with explicit settlement | Visibility timeout |
| Success action | Complete message | Delete message |
| Failure action | Abandon, dead-letter, defer | Let visibility timeout expire |
| Built-in poison handling | DLQ | No built-in DLQ; implement poison queue |
| Duplicate detection | Supported on entity | Not native in the same way |
| Long-running handler support | Lock renewal | Extend visibility timeout by re-receiving or redesigning worker flow |

Use Service Bus when you need richer broker features, stronger control over settlement, and built-in dead-lettering. Use Queue Storage when you need a simpler, lower-cost queue and can implement poison handling yourself.

**Troubleshooting scenarios**

A few exam-style cases help distinguish transient from permanent failures:

- A payment API returns `503` for several minutes:
  - Treat as transient.
  - Retry with exponential backoff and jitter.
  - Consider circuit breaking and dead-lettering only after retry exhaustion.

- A message body fails JSON deserialization because a required field is missing:
  - Treat as permanent.
  - Dead-letter the message or move it to a poison queue.
  - Log the schema mismatch and payload details.

- A Service Bus consumer processes a message for 3 minutes, but the lock duration is 30 seconds:
  - The lock expires and the message may be delivered again.
  - Renew the lock or redesign the handler to complete faster.

- A producer times out after sending to Service Bus and retries the send:
  - Duplicate detection can suppress the duplicate if the same `MessageId` is used and the entity has duplicate detection enabled.
  - Without duplicate detection, the consumer must be idempotent.

- A Queue Storage message keeps reappearing after failures:
  - Check `DequeueCount`.
  - Move it to a poison queue after the threshold.
  - Investigate whether the worker is failing permanently or the visibility timeout is too short.

**Practical design patterns**

- **Idempotent consumer**: store processed message IDs so repeated deliveries do not create duplicate side effects.
- **Outbox pattern**: persist business changes and outgoing messages together to reduce inconsistency.
- **Quarantine queue**: isolate suspicious messages for manual review.
- **Replay workflow**: repair dead-lettered or poison messages and resubmit them in controlled batches.
- **Observability**: track dead-letter count, dequeue count, retry count, and processing latency.

A simple idempotency check can look like this:

```csharp
if (await processedMessages.ExistsAsync(message.MessageId))
{
    await receiver.CompleteMessageAsync(message);
    return;
}

await ProcessBusinessLogicAsync(message);
await processedMessages.MarkProcessedAsync(message.MessageId);
await receiver.CompleteMessageAsync(message);
```

**Exam decision cues**

- Choose **dead-lettering** when the message is invalid for the current processing path and needs inspection.
- Choose **duplicate detection** when send retries may create duplicate messages and the entity can enforce a time-windowed uniqueness rule.
- Choose **retries with backoff and jitter** for transient failures.
- Choose **poison-message handling** in Queue Storage when a message repeatedly fails and must be isolated manually.
- Choose **lock renewal** when Service Bus processing may exceed the lock duration.

**Key takeaways**

- Service Bus and Queue Storage solve similar problems but use different failure semantics.
- Dead-letter queues preserve failed messages for diagnosis and replay.
- Duplicate detection must be enabled before sending and works within a bounded time window.
- Long-running Service Bus handlers need lock renewal to avoid duplicate processing.
- Queue Storage poison handling depends on visibility timeout and dequeue count, not built-in DLQ behavior.
- Exponential backoff with jitter is the safest default for transient retries.
