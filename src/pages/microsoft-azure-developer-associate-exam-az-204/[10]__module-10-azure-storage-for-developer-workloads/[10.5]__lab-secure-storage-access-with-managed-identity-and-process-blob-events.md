## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Lab overview**

This lab combines two common Azure application patterns:

- Securely reading blob data without connection strings by using **managed identity** and Azure role-based access control (RBAC).
- Reacting to blob changes by using **Azure Event Grid** and an **Azure Function** trigger.

The goal is to remove shared secrets from the data path and let Azure handle identity, authorization, and event delivery. In production, this pattern is useful for image processing, document ingestion, malware scanning, and metadata extraction pipelines.

A typical flow looks like this:

```text
Blob uploaded to Storage
        ↓
Event Grid publishes blob-created event
        ↓
Azure Function receives event
        ↓
Function uses managed identity to read blob data
        ↓
Function processes or stores the result
```

**Why this matters for AZ-204**

You are expected to understand:

- How to assign a managed identity to an app.
- How to grant data-plane access to Azure Storage.
- How Event Grid subscriptions route storage events to a function.
- How to troubleshoot authorization and validation issues.

**Core concepts**

<dl>
<dt><strong>Managed identity</strong></dt>
<dd>An Azure-managed service principal that lets your app authenticate to Azure resources without storing secrets.</dd>
<dt><strong>Data-plane access</strong></dt>
<dd>Permissions that control access to the contents of a storage account, container, or blob.</dd>
<dt><strong>Event Grid</strong></dt>
<dd>An event routing service that delivers Azure resource events to subscribers such as Azure Functions, webhooks, and queues.</dd>
<dt><strong>RBAC</strong></dt>
<dd>Role-based access control used to grant the minimum required permissions to identities.</dd>
</dl>

**End-to-end lab flow**

1. Create or select a storage account and blob container.
2. Enable a managed identity on the Function App.
3. Grant the identity the correct Storage Blob data role.
4. Create an Event Grid subscription for blob events.
5. Implement the function to parse the event payload and read the blob.
6. Test upload, validate delivery, and troubleshoot failures.

**Architecture and data flow**

The lab usually uses three Azure resources:

- **Storage account** for blob upload and event source.
- **Function App** for event handling and blob processing.
- **Event Grid subscription** to connect storage events to the function.

A practical topology is:

```text
[Client / Portal / CLI]
          |
          v
[Azure Blob Storage] -- emits --> [Event Grid Subscription] -- delivers --> [Azure Function]
          |                                                                  |
          |                                                                  v
          |-----------------------------------------------------------> [Blob read via Managed Identity]
```

The function does not need a storage connection string to read the blob if it uses `DefaultAzureCredential` or another Azure identity credential and has been granted the right role.

**Secure access with managed identity**

Managed identity is the preferred way to let an Azure-hosted app access Storage. It avoids secrets, supports rotation-free authentication, and integrates with Azure RBAC.

There are two common identity types:

- **System-assigned identity**
  - Tied to one Azure resource.
  - Deleted when the resource is deleted.
  - Best for a single Function App or App Service.

- **User-assigned identity**
  - Created as a standalone Azure resource.
  - Can be shared across multiple apps.
  - Useful when several workloads need the same identity.

For this lab, a system-assigned identity is usually the simplest choice.

**Enable the managed identity**

You can enable it in the portal or with Azure CLI.

```bash
az functionapp identity assign \
  --resource-group rg-lab \
  --name func-blob-processor
```

Verify the principal ID:

```bash
az functionapp identity show \
  --resource-group rg-lab \
  --name func-blob-processor
```

If you use a user-assigned identity, you must create it first and attach it to the app.

**Grant Storage permissions**

For blob read access, assign a data-plane role such as:

- **Storage Blob Data Reader** for read-only access.
- **Storage Blob Data Contributor** if the function must also write or delete blobs.

Use the narrowest role that satisfies the lab. For a read-and-process scenario, `Storage Blob Data Reader` is usually enough.

Example at the storage account scope:

```bash
principalId=$(az functionapp identity show \
  --resource-group rg-lab \
  --name func-blob-processor \
  --query principalId -o tsv)

storageId=$(az storage account show \
  --resource-group rg-lab \
  --name stlabsecure \
  --query id -o tsv)

az role assignment create \
  --assignee-object-id "$principalId" \
  --assignee-principal-type ServicePrincipal \
  --role "Storage Blob Data Reader" \
  --scope "$storageId"
```

You can also scope the assignment to a container when you want tighter access control. Container scope is valid for blob data roles, but it is not always the best lab choice if you need the function to enumerate multiple containers or if the lab instructions expect account-level access. Use container scope only when the app truly needs access to one container.

Example container scope:

```bash
containerScope=$(az storage container show \
  --account-name stlabsecure \
  --name input \
  --auth-mode login \
  --query id -o tsv)

az role assignment create \
  --assignee-object-id "$principalId" \
  --assignee-principal-type ServicePrincipal \
  --role "Storage Blob Data Reader" \
  --scope "$containerScope"
```

> If the storage account uses a firewall or private endpoint, RBAC alone is not enough. The function app must also be able to reach the storage endpoint over the network.

**Local development versus Azure runtime**

Local development and deployed Azure runtime behave differently:

- **Locally**
  - `DefaultAzureCredential` can use your signed-in developer identity.
  - You can test blob access if your user account has the right role.
  - Event Grid delivery to a local function usually requires a public endpoint or a tunneling tool such as ngrok for webhook-style testing.

- **In Azure**
  - The function uses its managed identity automatically.
  - Event Grid can deliver directly to the function endpoint.
  - Network restrictions, private endpoints, and firewall rules can block access even when RBAC is correct.

A common local test for blob access is:

```csharp
using Azure.Identity;
using Azure.Storage.Blobs;

var blobClient = new BlobClient(
    new Uri("https://stlabsecure.blob.core.windows.net/input/sample.txt"),
    new DefaultAzureCredential());

var content = await blobClient.DownloadContentAsync();
Console.WriteLine(content.Value.Content.ToString());
```

This works only if the signed-in identity has data-plane access and the storage network rules allow the request.

**Event Grid and blob events**

Azure Storage emits events when blob operations occur. The most commonly used and tested events for this lab are:

- `Microsoft.Storage.BlobCreated`
- `Microsoft.Storage.BlobDeleted`
- `Microsoft.Storage.BlobTierChanged`

These are the events you should expect to see in a typical AZ-204 storage-event scenario. Avoid assuming rename events unless your specific storage feature and event type explicitly support them.

Event Grid delivers events over HTTPS to the subscriber endpoint. The delivery model is push-based, and the subscriber must be reachable and able to respond correctly to validation and event requests.

**Event subscription validation**

When you create an Event Grid subscription to an Azure Function or webhook endpoint, Event Grid performs a validation handshake.

Operationally, this means:

- Event Grid sends a validation event to the endpoint.
- The endpoint must respond correctly to confirm it can receive events.
- For Azure Functions, the platform and trigger binding handle this handshake when the function is configured correctly.
- If validation fails, the subscription creation or activation fails.

This is a common lab failure point. If the subscription does not become active, check:

- The function app is running.
- The trigger binding is correct.
- The endpoint URL is valid.
- The function app is publicly reachable if required by the delivery path.
- The function app is not blocked by authentication settings that prevent Event Grid from reaching the trigger.

**Creating the Event Grid subscription**

You can create the subscription from the portal, Azure CLI, or ARM/Bicep. For a lab, the portal is often easiest to verify visually, but CLI is useful for repeatability.

Example CLI for a storage account event subscription to a webhook-style endpoint:

```bash
storageId=$(az storage account show \
  --resource-group rg-lab \
  --name stlabsecure \
  --query id -o tsv)

az eventgrid event-subscription create \
  --name sub-blob-created \
  --source-resource-id "$storageId" \
  --endpoint-type webhook \
  --endpoint "https://func-blob-processor.azurewebsites.net/runtime/webhooks/EventGrid?functionName=ProcessBlob&code=<function-key>" \
  --included-event-types Microsoft.Storage.BlobCreated
```

For Azure Functions, the exact endpoint shape depends on the hosting model and trigger configuration. In many labs, the portal is the safest way to wire the Event Grid trigger to the function because it handles the function endpoint details for you.

**Function implementation**

The function should do two things:

1. Parse the Event Grid event payload.
2. Use managed identity to read the blob.

A typical Event Grid event contains a `data.url` field that points to the blob. You can extract that URL and create a `BlobClient` from it.

Example in C# for an Azure Functions isolated worker style app:

```csharp
using System.Text.Json;
using Azure.Identity;
using Azure.Storage.Blobs;
using Microsoft.Azure.Functions.Worker;
using Microsoft.Extensions.Logging;

public class ProcessBlob
{
    private readonly ILogger _logger;

    public ProcessBlob(ILoggerFactory loggerFactory)
    {
        _logger = loggerFactory.CreateLogger<ProcessBlob>();
    }

    [Function("ProcessBlob")]
    public async Task Run([EventGridTrigger] JsonElement eventGridEvent)
    {
        var data = eventGridEvent.GetProperty("data");
        var blobUrl = data.GetProperty("url").GetString();

        if (string.IsNullOrWhiteSpace(blobUrl))
        {
            _logger.LogWarning("Event did not contain a blob URL.");
            return;
        }

        var blobClient = new BlobClient(new Uri(blobUrl), new DefaultAzureCredential());
        var download = await blobClient.DownloadContentAsync();

        var content = download.Value.Content.ToString();
        _logger.LogInformation("Blob content length: {Length}", content.Length);
    }
}
```

This example is intentionally practical:

- It reads the blob URL from the event payload.
- It uses `DefaultAzureCredential`, which works locally and in Azure.
- It avoids hard-coded secrets.

If you use the in-process model instead of isolated worker, the trigger attribute and package references differ. Make sure your sample matches the Functions runtime model you actually deploy.

**Required packages and bindings**

For an isolated worker function, you typically need packages similar to:

```xml
<ItemGroup>
  <PackageReference Include="Microsoft.Azure.Functions.Worker" Version="*" />
  <PackageReference Include="Microsoft.Azure.Functions.Worker.Extensions.EventGrid" Version="*" />
  <PackageReference Include="Azure.Identity" Version="*" />
  <PackageReference Include="Azure.Storage.Blobs" Version="*" />
</ItemGroup>
```

For in-process functions, the package set is different. The key point is that the Event Grid trigger binding must match the runtime model, or the function will not start correctly.

**Practical blob payload handling**

The Event Grid payload includes metadata that helps you identify the object and process it safely. Common fields include:

- `data.url` for the blob URL.
- `subject` for the resource path.
- `eventType` for the event name.
- `eventTime` for the time the event occurred.

A simple parser can use `subject` to log the container and blob path:

```csharp
var subject = eventGridEvent.GetProperty("subject").GetString();
_logger.LogInformation("Event subject: {Subject}", subject);
```

If you need to derive the blob name from the URL, you can parse the URI path:

```csharp
var uri = new Uri(blobUrl);
var path = uri.AbsolutePath; // /input/sample.txt
_logger.LogInformation("Blob path: {Path}", path);
```

Use the URL from the event rather than reconstructing the blob path manually when possible. That reduces parsing mistakes.

**Testing the managed identity path**

A good test sequence is:

1. Deploy the function with managed identity enabled.
2. Assign the Storage Blob data role.
3. Upload a test blob.
4. Confirm the function receives the Event Grid event.
5. Confirm the function can read the blob content.

To verify the role assignment, use:

```bash
az role assignment list \
  --assignee-object-id "$principalId" \
  --scope "$storageId" \
  -o table
```

To test blob access from the function, check Application Insights logs or function logs for a successful download message.

If you want to test locally, sign in with Azure CLI:

```bash
az login
az account set --subscription "<subscription-id>"
```

Then run the function locally and ensure your user account has the same blob data role. Local success does not guarantee Azure success if the deployed app has a different identity or network path.

**Common troubleshooting scenarios**

**1. Event Grid subscription validation fails**

Possible causes:

- The function endpoint is not reachable.
- The function app is stopped.
- The trigger binding is incorrect.
- The endpoint URL is malformed.
- Authentication or access restrictions block the validation request.

What to check:

- Function app status.
- Event Grid subscription provisioning state.
- Function logs for validation errors.
- Whether the endpoint is a supported Event Grid trigger target.

**2. Function receives the event but cannot read the blob**

Possible causes:

- Missing `Storage Blob Data Reader` role.
- Role assignment not yet propagated.
- Wrong storage account or container scope.
- Storage firewall or private endpoint blocks access.
- The blob URL in the event is incorrect or points to a different account.

What to check:

- RBAC assignment at the correct scope.
- Network rules on the storage account.
- Whether the function app has outbound access to the storage endpoint.
- Whether the blob exists and is accessible.

**3. Local code works but Azure deployment fails**

Possible causes:

- Local developer identity has access, but the managed identity does not.
- The deployed app uses a different runtime model or package set.
- The Azure environment has stricter networking.
- The function app configuration is missing required settings.

What to check:

- Managed identity is enabled in Azure.
- The correct role is assigned to the managed identity.
- The deployed app settings match the code expectations.
- The function runtime model matches the trigger package.

**4. Blob event never arrives**

Possible causes:

- Wrong event type selected.
- Subscription created on the wrong storage account.
- Blob operation does not emit the expected event.
- Event Grid subscription is filtered too narrowly.
- The function endpoint is not active.

What to check:

- Use `Microsoft.Storage.BlobCreated` for upload scenarios.
- Confirm the subscription source resource ID.
- Review filters and subject prefixes.
- Verify the function app is healthy.

**Practical lab checklist**

- Create a storage account and container.
- Create or deploy a Function App.
- Enable system-assigned managed identity.
- Assign `Storage Blob Data Reader` to the identity.
- Implement an Event Grid-triggered function.
- Parse `data.url` from the event payload.
- Use `DefaultAzureCredential` to read the blob.
- Create an Event Grid subscription for blob-created events.
- Upload a test blob and confirm processing.
- Review logs and fix validation or authorization issues.

**Production considerations**

This pattern scales well because it removes secrets and decouples event generation from processing. In real systems, you should also consider:

- **Idempotency** so repeated events do not cause duplicate processing.
- **Dead-lettering** for failed Event Grid deliveries.
- **Retry behavior** in the function and downstream services.
- **Least privilege** for RBAC assignments.
- **Network isolation** when using private endpoints or storage firewalls.
- **Observability** with Application Insights and structured logs.

A robust implementation often stores processing state, validates event types, and ignores duplicate or irrelevant events.

**Key takeaways**

- Managed identity is the preferred way to access Azure Storage from Azure-hosted code.
- Use Storage Blob data roles, not management-plane roles, for blob content access.
- Event Grid is the event routing layer for blob-created and related storage events.
- The function should read the blob URL from the event payload and use Azure identity-based authentication.
- Validation, RBAC propagation, and network rules are the most common lab blockers.
