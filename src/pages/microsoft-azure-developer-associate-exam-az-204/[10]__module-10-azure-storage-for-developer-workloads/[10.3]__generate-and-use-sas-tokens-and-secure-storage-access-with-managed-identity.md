## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Storage authorization overview**

Azure Storage supports multiple authorization models, and the exam expects you to choose the right one for the scenario. The main options are:

- **Microsoft Entra ID** with Azure role-based access control (RBAC) for identity-based access.
- **Shared Access Signatures (SAS)** for delegated, time-bound access.
- **Storage account keys** for full account-level access, usually avoided in application code.

A good rule is to prefer identity-based access first, then SAS when you need scoped, temporary access, and use account keys only when the scenario explicitly allows it.

**SAS token types and when to use them**

A SAS is a signed URI query string that grants limited access to Storage resources. The signing key determines the SAS type:

| SAS type | Signing key | Supported services | Typical use |
|---|---|---|---|
| **Account SAS** | Storage account key | Blob, Queue, Table, File | Cross-service access with account-level control over allowed services and resource types |
| **Service SAS** | Storage account key | A single service, such as Blob | Fine-grained access to one service or resource |
| **User delegation SAS** | User delegation key from Microsoft Entra ID | **Blob service only** | Preferred Blob SAS when the app uses Microsoft Entra ID |

The distinction matters:

- **Account SAS** can span multiple Storage services, so it is broader than a service SAS.
- **Service SAS** is scoped to one service and is signed with the account key.
- **User delegation SAS** is signed with a user delegation key obtained from Microsoft Entra ID and is supported for the **Blob service only**, not Queue, File, or Table.

> User delegation SAS is the Entra ID-based SAS option for Blob storage only. It does not replace SAS patterns for other Storage services.

**SAS capabilities and constraints**

SAS tokens can restrict access by:

- **Permissions** such as read, write, list, add, create, delete, and update depending on the resource type.
- **Resource scope** such as a container, blob, queue, file share, or table.
- **Start and expiry time** to limit the usable window.
- **IP address range** to reduce exposure.
- **Protocol** to require HTTPS.

For Blob resources, permissions are resource-specific. Do not assume every permission letter is valid for every SAS target.

- **Blob SAS** commonly uses permissions like `r`, `w`, `d`, `l`, `a`, `c`, and `t` depending on whether the target is a blob, container, or snapshot.
- **Container SAS** is constrained to container operations and blob enumeration within that container.
- **Queue, File, and Table SAS** use their own permission sets and do not share Blob permission semantics.

A practical exam takeaway is that SAS permissions are not interchangeable across services.

**Generating SAS with Azure CLI**

You can generate SAS tokens with Azure CLI when you have the right authorization.

For a **service SAS** on a container:

```bash
az storage container generate-sas \
  --account-name mystorageacct \
  --name mycontainer \
  --permissions rl \
  --expiry 2026-05-01T00:00:00Z \
  --auth-mode key \
  --as-user false
```

For a **user delegation SAS** on Blob resources, you sign in with Microsoft Entra ID and request the SAS as the signed-in user:

```bash
az login
az storage blob generate-sas \
  --account-name mystorageacct \
  --container-name mycontainer \
  --name sample.txt \
  --permissions r \
  --expiry 2026-05-01T00:00:00Z \
  --auth-mode login \
  --as-user
```

Important details:

- `--auth-mode login` means Azure CLI uses your Entra ID sign-in instead of the account key.
- `--as-user` requests a **user delegation SAS**.
- Your signed-in identity must have a **data-plane role** that includes the `generateUserDelegationKey` action, such as **Storage Blob Data Contributor** or **Storage Blob Data Owner**.
- Management-plane roles alone are not enough for Blob data access or for generating a user delegation key.

> If you can sign in but still cannot generate a user delegation SAS, check the storage data role assignment first.

**Generating SAS programmatically with Azure SDKs**

In application code, you often generate SAS tokens with the Azure Storage SDK rather than the CLI. The pattern depends on the SAS type.

For a **service SAS** or **account SAS**, you typically use the account key.

Example in C# for a Blob service SAS:

```csharp
using Azure.Storage;
using Azure.Storage.Sas;

var accountName = "mystorageacct";
var accountKey = "<account-key>";
var blobName = "sample.txt";

var credential = new StorageSharedKeyCredential(accountName, accountKey);

var sasBuilder = new BlobSasBuilder
{
    BlobContainerName = "mycontainer",
    BlobName = blobName,
    Resource = "b",
    ExpiresOn = DateTimeOffset.UtcNow.AddHours(1)
};

sasBuilder.SetPermissions(BlobSasPermissions.Read | BlobSasPermissions.Write);

var sasToken = sasBuilder.ToSasQueryParameters(credential).ToString();
Console.WriteLine(sasToken);
```

For a **user delegation SAS**, you first obtain a user delegation key using an Entra ID credential, then sign the SAS with that key.

```csharp
using Azure.Identity;
using Azure.Storage.Blobs;
using Azure.Storage.Sas;

var serviceUri = new Uri("https://mystorageacct.blob.core.windows.net");
var blobServiceClient = new BlobServiceClient(serviceUri, new DefaultAzureCredential());

var userDelegationKey = await blobServiceClient.GetUserDelegationKeyAsync(
    DateTimeOffset.UtcNow,
    DateTimeOffset.UtcNow.AddHours(1));

var sasBuilder = new BlobSasBuilder
{
    BlobContainerName = "mycontainer",
    BlobName = "sample.txt",
    Resource = "b",
    ExpiresOn = DateTimeOffset.UtcNow.AddMinutes(30)
};

sasBuilder.SetPermissions(BlobSasPermissions.Read);

var sasToken = sasBuilder.ToSasQueryParameters(userDelegationKey, "mystorageacct").ToString();
Console.WriteLine(sasToken);
```

This is the key exam pattern:

- Use **account key** for account SAS and service SAS.
- Use **user delegation key** for user delegation SAS.
- Use **Blob service clients** and Entra ID credentials to obtain the delegation key.

**Using SAS in requests**

A SAS is appended to the resource URI as query parameters.

```text
https://mystorageacct.blob.core.windows.net/mycontainer/sample.txt?sv=...&sp=r&se=...&sr=b&sig=...
```

Common SAS fields include:

- `sv` for the Storage service version.
- `sp` for permissions.
- `se` for expiry.
- `st` for start time.
- `sr` for signed resource.
- `sig` for the signature.

Practical guidance:

- Prefer short expirations.
- Use HTTPS-only access.
- Scope the SAS to the narrowest resource possible.
- Avoid embedding long-lived SAS tokens in source code or client apps.

**Managed identity for secure Storage access**

Managed identity lets Azure resources authenticate to Storage without storing secrets. Azure creates an identity in Microsoft Entra ID for the resource, and your app uses that identity to request tokens.

There are two common forms:

- **System-assigned managed identity**: tied to one Azure resource and deleted with it.
- **User-assigned managed identity**: created as a standalone resource and attached to one or more Azure resources.

Use managed identity when your app needs direct access to Storage and you want to avoid account keys, connection strings, or manually managed secrets.

**Enabling managed identity**

For a system-assigned identity, you enable it on the Azure resource.

Azure CLI example for an App Service:

```bash
az webapp identity assign \
  --resource-group rg-app \
  --name mywebapp
```

Azure CLI example for a Function App:

```bash
az functionapp identity assign \
  --resource-group rg-app \
  --name myfuncapp
```

For a user-assigned identity, you create the identity and attach it to the resource.

```bash
az identity create \
  --resource-group rg-app \
  --name my-uami
```

Attach it to a web app:

```bash
az webapp identity assign \
  --resource-group rg-app \
  --name mywebapp \
  --identities my-uami
```

In the portal, you typically:

1. Open the Azure resource.
2. Select **Identity**.
3. Turn on **System assigned** or add a **User assigned** identity.
4. Save the change.

**Assigning Storage roles to managed identity**

To access Blob data, the managed identity needs a **data-plane role** at the correct scope.

Common roles include:

- **Storage Blob Data Reader** for read-only access.
- **Storage Blob Data Contributor** for read/write/delete access to Blob data.
- **Storage Blob Data Owner** for full Blob data access and delegation key generation.

Assign the role at one of these scopes:

- Storage account
- Container
- Specific blob path scope, where supported

Azure CLI example:

```bash
az role assignment create \
  --assignee-object-id <managed-identity-service-principal-object-id> \
  --assignee-principal-type ServicePrincipal \
  --role "Storage Blob Data Contributor" \
  --scope /subscriptions/<sub-id>/resourceGroups/rg-app/providers/Microsoft.Storage/storageAccounts/mystorageacct
```

Notes:

- For managed identities, use the identity’s **service principal object ID**.
- The role assignment must target the **data plane** for Blob access.
- If you only assign a management role, the app still cannot read blob data.

**Using managed identity in application code**

The application uses an Azure Identity credential and a Storage client. The SDK obtains an access token automatically.

C# example for an App Service or Function App using system-assigned managed identity:

```csharp
using Azure.Identity;
using Azure.Storage.Blobs;

var blobServiceClient = new BlobServiceClient(
    new Uri("https://mystorageacct.blob.core.windows.net"),
    new DefaultAzureCredential());

var containerClient = blobServiceClient.GetBlobContainerClient("mycontainer");
await foreach (var blobItem in containerClient.GetBlobsAsync())
{
    Console.WriteLine(blobItem.Name);
}
```

For a **user-assigned managed identity**, specify the client ID or resource ID in the credential options:

```csharp
using Azure.Identity;
using Azure.Storage.Blobs;

var credential = new DefaultAzureCredential(new DefaultAzureCredentialOptions
{
    ManagedIdentityClientId = "<user-assigned-client-id>"
});

var blobServiceClient = new BlobServiceClient(
    new Uri("https://mystorageacct.blob.core.windows.net"),
    credential);
```

Or use `ManagedIdentityCredential` directly:

```csharp
using Azure.Identity;
using Azure.Storage.Blobs;

var credential = new ManagedIdentityCredential("<user-assigned-client-id>");
var blobServiceClient = new BlobServiceClient(
    new Uri("https://mystorageacct.blob.core.windows.net"),
    credential);
```

Typical scenarios:

- **App Service** or **Function App** reads blobs without storing secrets.
- **Virtual Machine** uploads files to Blob Storage using its managed identity.
- **Container Apps** or other Azure-hosted workloads authenticate with the same pattern if managed identity is enabled.

**Managed identity vs SAS**

Use the right tool for the job.

| Scenario | Best fit | Why |
|---|---|---|
| Azure-hosted app needs direct Blob access | Managed identity | No secret to store, easy rotation, least privilege |
| External client needs temporary access to a blob | SAS | Delegated, time-limited access without exposing account keys |
| App needs to grant access to a third party | SAS | You can scope and expire access precisely |
| App needs to access multiple Storage services with one token model | Account SAS | Broader service coverage, but higher risk |
| App uses Entra ID and needs Blob-only delegated access | User delegation SAS | Avoids account key usage for Blob SAS generation |

A common architecture is:

- Your backend uses **managed identity** to access Storage securely.
- Your backend generates a short-lived **SAS** for a browser or partner system.
- The client uses the SAS only for the narrow operation it needs.

**Practical architecture examples**

A web app on App Service can use managed identity to read configuration blobs and generate a short-lived SAS for a user download.

```text
User -> Web App (Managed Identity) -> Blob Storage
                     |
                     +-> Generates SAS for one file
                     |
                     +-> Returns SAS URL to user
```

A Function App can process uploaded blobs without any connection string.

```text
Blob upload -> Storage trigger -> Function App with managed identity -> Read/write blobs
```

A VM can use a user-assigned managed identity when multiple workloads on the same machine need separate identities.

```text
VM
├─ App A -> User-assigned identity A -> Storage account A
└─ App B -> User-assigned identity B -> Storage account B
```

**Troubleshooting and exam traps**

- **Authentication succeeds but data access fails**: check Blob data-plane role assignments, not just subscription or resource-group roles.
- **User delegation SAS generation fails**: verify the identity has a role that includes `generateUserDelegationKey`.
- **Wrong SAS type for the service**: remember user delegation SAS is **Blob-only**.
- **Overly broad permissions**: avoid granting more than the operation requires.
- **Expired or clock-skewed SAS**: ensure start and expiry times are valid and allow for time drift.
- **Using account keys in code**: acceptable in some legacy or controlled scenarios, but not the preferred secure pattern.

**Quick comparison for AZ-204**

- **Account SAS**: signed with the storage account key; works across Storage services.
- **Service SAS**: signed with the storage account key; scoped to one service.
- **User delegation SAS**: signed with a user delegation key from Microsoft Entra ID; Blob service only.
- **Managed identity**: best for Azure-to-Azure authentication without secrets; requires Blob data-plane roles for Blob access.

**Key takeaways**

- Prefer **managed identity** for Azure-hosted applications that access Blob Storage directly.
- Use **SAS** when you need temporary, scoped access for a client or external consumer.
- Use **user delegation SAS** for Blob SAS generation with Microsoft Entra ID.
- Use **account SAS** or **service SAS** when the scenario allows account-key-based signing and the service supports it.
- Always assign the correct **data-plane role** for Blob access and delegation key generation.
