## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Blob Storage fundamentals**

Azure Blob Storage stores unstructured data such as images, logs, backups, documents, and media. In the Azure Storage hierarchy, a **storage account** contains one or more **containers**, and each container contains **blobs**. For AZ-204, you should know how to upload, download, and manage blob metadata using the Azure SDK, Azure CLI, and REST concepts.

Blob types matter because they affect how you write and read data:

- **Block blobs** are the default choice for most files and support efficient uploads of large content in blocks.
- **Append blobs** are optimized for append-only scenarios such as logs.
- **Page blobs** are used for random read/write workloads, commonly for VHDs and some disk scenarios.

Blob access is controlled by authentication and authorization. Common approaches include Microsoft Entra ID with role-based access control, shared access signatures, and account keys. For application code, prefer Entra ID and managed identities when possible.

**Core upload patterns**

Uploading a blob means sending content from memory, a file, a stream, or binary data into a container. The Azure.Storage.Blobs SDK supports several valid upload sources. A common mistake is trying to pass a file path string directly to `UploadAsync`; instead, open the file as a stream or use a binary payload.

Typical upload options include:

- **Stream upload** for files on disk or network streams.
- **BinaryData upload** for in-memory content.
- **Text upload** by converting text to bytes or using a stream writer over a blob stream.
- **Block upload** for large or resumable content patterns.

Example: upload a local file as a stream.

```csharp
using Azure.Identity;
using Azure.Storage.Blobs;

var serviceClient = new BlobServiceClient(
    new Uri("https://<account-name>.blob.core.windows.net"),
    new DefaultAzureCredential());

var containerClient = serviceClient.GetBlobContainerClient("images");
await containerClient.CreateIfNotExistsAsync();

var blobClient = containerClient.GetBlobClient("photo.jpg");

await using var fileStream = File.OpenRead(@"C:\data\photo.jpg");
await blobClient.UploadAsync(fileStream, overwrite: true);
```

Example: upload in-memory content.

```csharp
using Azure.Storage.Blobs;
using Azure;

var blobClient = new BlobClient(
    new Uri("https://<account-name>.blob.core.windows.net/images/readme.txt"),
    new AzureSasCredential("<sas-token>"));

BinaryData content = BinaryData.FromString("Hello from Blob Storage");
await blobClient.UploadAsync(content, overwrite: true);
```

When you need to write text through a stream, keep the blob stream open until the writer flushes.

```csharp
await using var writeStream = await blobClient.OpenWriteAsync(overwrite: true);
await using (var writer = new StreamWriter(writeStream, leaveOpen: true))
{
    await writer.WriteAsync("Line 1");
    await writer.WriteAsync(Environment.NewLine);
    await writer.WriteAsync("Line 2");
    await writer.FlushAsync();
}
```

> `OpenWriteAsync` writes to a blob stream and is not the same as append blob semantics. Use append blobs when you need true append-only behavior.

For large uploads, the SDK can split content into blocks internally. This helps with reliability and throughput, especially when the blob is larger than a single request or when retries are needed. You can also tune transfer options when you need more control.

```csharp
using Azure.Storage.Blobs.Models;

var options = new BlobUploadOptions
{
    TransferOptions = new StorageTransferOptions
    {
        InitialTransferSize = 4 * 1024 * 1024,
        MaximumTransferSize = 4 * 1024 * 1024
    }
};

await using var stream = File.OpenRead(@"C:\data\video.mp4");
await blobClient.UploadAsync(stream, options);
```

**Download patterns**

Downloading retrieves blob content to a stream, file, or memory buffer. Choose the method based on file size and how you plan to consume the data.

Common download approaches:

- **DownloadToAsync** writes directly to a file stream or destination stream.
- **DownloadContentAsync** loads content into memory as `BinaryData`.
- **OpenReadAsync** returns a readable stream with internal buffering for efficient sequential reads.

Example: download to a local file.

```csharp
await using var downloadStream = await blobClient.OpenReadAsync();
await using var output = File.Create(@"C:\data\downloaded-photo.jpg");
await downloadStream.CopyToAsync(output);
```

Example: download directly to a file stream.

```csharp
await using var output = File.Create(@"C:\data\report.pdf");
await blobClient.DownloadToAsync(output);
```

Example: read content into memory for smaller blobs.

```csharp
var response = await blobClient.DownloadContentAsync();
string text = response.Value.Content.ToString();
Console.WriteLine(text);
```

`OpenReadAsync` is useful when you want a stream that reads efficiently without loading the entire blob into memory. It is best for sequential processing, not for treating the blob like a random-access file with explicit range control. If you need precise range retrieval, use download APIs that support range options or request ranges explicitly.

**Metadata and properties**

Blob metadata is a set of user-defined name/value pairs stored with the blob. Metadata is useful for application-specific tags such as `author`, `content-type-source`, `project`, or `ingestion-id`. Metadata keys are case-insensitive in practice but are stored and returned in a normalized form by the service; use consistent naming conventions to avoid confusion.

Important metadata rules:

- **Metadata is not indexed** for search.
- **Metadata is separate from blob properties** such as content type, content length, and last modified time.
- **Metadata updates replace the full metadata set** unless you merge values in code first.
- **Metadata keys and values** must follow service rules for allowed characters and size limits.

Example: set metadata after upload.

```csharp
var metadataBlob = containerClient.GetBlobClient("photo.jpg");

await metadataBlob.SetMetadataAsync(new Dictionary<string, string>
{
    ["project"] = "marketing",
    ["source"] = "mobile-app",
    ["environment"] = "prod"
});
```

Example: read metadata and properties.

```csharp
var properties = await metadataBlob.GetPropertiesAsync();

foreach (var item in properties.Value.Metadata)
{
    Console.WriteLine($"{item.Key} = {item.Value}");
}

Console.WriteLine($"Content-Type: {properties.Value.ContentType}");
Console.WriteLine($"Length: {properties.Value.ContentLength}");
```

If you need to preserve existing metadata, read the current set first, modify it, and write it back.

```csharp
var current = await metadataBlob.GetPropertiesAsync();
var metadata = new Dictionary<string, string>(current.Value.Metadata)
{
    ["reviewed"] = "true"
};

await metadataBlob.SetMetadataAsync(metadata);
```

**Blob properties versus metadata**

Blob properties are service-managed attributes, while metadata is application-managed data. This distinction is important for exam questions and troubleshooting.

| Aspect | Metadata | Properties |
|---|---|---|
| Purpose | App-defined key/value data | Service-defined blob attributes |
| Examples | `project=alpha`, `owner=team-a` | `ContentType`, `ContentLength`, `ETag` |
| Update behavior | Replaced as a set | Set by service or specific APIs |
| Searchability | Not indexed | Not indexed as user data |
| Typical use | Classification, workflow tags | MIME type, caching, validation |

**Authentication and authorization**

For production apps, prefer Microsoft Entra ID with `DefaultAzureCredential` or managed identity. This avoids storing secrets in code and works well in Azure-hosted services.

Common authentication options:

- **DefaultAzureCredential** for local development and Azure deployment.
- **Managed identity** for App Service, Functions, Container Apps, and VMs.
- **SAS token** for delegated, time-limited access.
- **Account key** for administrative scenarios or legacy tooling.

Example: use `DefaultAzureCredential` in an app.

```csharp
using Azure.Identity;
using Azure.Storage.Blobs;

var serviceClient = new BlobServiceClient(
    new Uri("https://<account-name>.blob.core.windows.net"),
    new DefaultAzureCredential());
```

For Azure CLI, authentication often uses the signed-in identity:

```bash
az login
az storage blob upload \
  --account-name mystorageacct \
  --container-name images \
  --name photo.jpg \
  --file ./photo.jpg \
  --auth-mode login
```

If you use an account key, supply the account name and key together:

```bash
az storage blob upload \
  --account-name mystorageacct \
  --account-key "$ACCOUNT_KEY" \
  --container-name images \
  --name photo.jpg \
  --file ./photo.jpg
```

> `--account-key` is not a standalone identity flow. It is a credential value used with the storage account name.

**Azure CLI operations**

The Azure CLI is useful for quick validation, scripting, and troubleshooting. Use it to create containers, upload files, download blobs, and inspect properties.

Common commands:

```bash
az storage container create \
  --account-name mystorageacct \
  --name images \
  --auth-mode login
```

```bash
az storage blob upload \
  --account-name mystorageacct \
  --container-name images \
  --name photo.jpg \
  --file ./photo.jpg \
  --overwrite true \
  --auth-mode login
```

```bash
az storage blob download \
  --account-name mystorageacct \
  --container-name images \
  --name photo.jpg \
  --file ./downloaded-photo.jpg \
  --auth-mode login
```

```bash
az storage blob show \
  --account-name mystorageacct \
  --container-name images \
  --name photo.jpg \
  --auth-mode login
```

`az storage blob show` is a safe way to inspect blob properties and often includes metadata in the output. If you need only metadata, verify the exact CLI version and command behavior in your environment.

**REST API concepts**

At the REST layer, blob upload and download use HTTP verbs and headers. You should understand the shape of the requests even if you primarily use the SDK.

Typical patterns:

- **PUT Blob** uploads a new blob or replaces an existing one.
- **GET Blob** downloads blob content.
- **HEAD Blob** retrieves properties without the body.
- **Set Blob Metadata** updates metadata with `x-ms-meta-*` headers.

Example: upload with REST headers.

```http
PUT https://mystorageacct.blob.core.windows.net/images/photo.jpg HTTP/1.1
x-ms-blob-type: BlockBlob
x-ms-date: Tue, 26 Apr 2026 12:00:00 GMT
x-ms-version: 2020-10-02
Authorization: SharedKey mystorageacct:signature
Content-Length: 1024
Content-Type: image/jpeg
```

Example: set metadata with REST.

```http
PUT https://mystorageacct.blob.core.windows.net/images/photo.jpg?comp=metadata HTTP/1.1
x-ms-date: Tue, 26 Apr 2026 12:00:00 GMT
x-ms-version: 2020-10-02
x-ms-meta-project: marketing
x-ms-meta-source: mobile-app
Authorization: SharedKey mystorageacct:signature
```

Use a broadly supported service version in examples and production code unless a specific version is required by your scenario. Avoid assuming a future or uncommon version number.

**Concurrency, access conditions, and leases**

Blob operations often need concurrency control. Azure Storage uses the **ETag** (entity tag) and conditional headers to help prevent accidental overwrites. A lease can also lock a blob for coordinated updates.

Use these patterns when multiple writers may touch the same blob:

- **ETag checks** with `If-Match` to update only if the blob has not changed.
- **Leases** to coordinate exclusive write access.
- **Overwrite flags** only when you intentionally replace content.

Example: conditional update with ETag.

```csharp
var props = await blobClient.GetPropertiesAsync();

var conditions = new Azure.Storage.Blobs.Models.BlobRequestConditions
{
    IfMatch = props.Value.ETag
};

await blobClient.UploadAsync(
    BinaryData.FromString("updated content"),
    new Azure.Storage.Blobs.Models.BlobUploadOptions
    {
        Conditions = conditions
    });
```

Leases are most relevant when you need exclusive control over a blob during writes, deletes, or metadata/property updates. They are not the general mechanism for tier changes.

**Practical upload and download workflow**

A typical application flow is:

1. Validate the file or payload size.
2. Choose the correct blob type, usually block blob.
3. Upload with the right credential and overwrite behavior.
4. Set metadata and properties after upload if needed.
5. Download or stream the blob when a client requests it.
6. Use ETags or leases when concurrent updates are possible.

Example workflow in C#:

```csharp
using Azure.Identity;
using Azure.Storage.Blobs;

var serviceClient = new BlobServiceClient(
    new Uri("https://<account-name>.blob.core.windows.net"),
    new DefaultAzureCredential());

var container = serviceClient.GetBlobContainerClient("documents");
await container.CreateIfNotExistsAsync();

var blob = container.GetBlobClient("specs.txt");

await blob.UploadAsync(BinaryData.FromString("Initial content"), overwrite: true);

await blob.SetMetadataAsync(new Dictionary<string, string>
{
    ["doc-type"] = "spec",
    ["owner"] = "engineering"
});

var downloaded = await blob.DownloadContentAsync();
Console.WriteLine(downloaded.Value.Content.ToString());
```

**Common pitfalls**

- Passing a file path string directly to `UploadAsync` instead of opening a stream.
- Using `OpenWriteAsync` as if it were append blob behavior.
- Forgetting that `SetMetadataAsync` replaces the full metadata collection.
- Loading large blobs into memory with `DownloadContentAsync` when a stream is better.
- Assuming metadata is searchable or automatically synchronized with blob properties.
- Treating `--account-key` as a complete authentication mode in CLI commands.
- Overstating what `OpenReadAsync` does for random range access.

**Exam focus**

For AZ-204, expect questions that test whether you can choose the right API for the job:

- Use **UploadAsync(Stream)** or **UploadAsync(BinaryData)** for blob content.
- Use **DownloadToAsync** or **OpenReadAsync** for retrieval.
- Use **SetMetadataAsync** and **GetPropertiesAsync** for metadata workflows.
- Use **DefaultAzureCredential** or managed identity for secure app authentication.
- Use **ETag** and conditional requests to avoid overwriting concurrent changes.

**Quick reference**

- **Upload a file:** open a stream, then call `UploadAsync(stream, overwrite: true)`.
- **Upload text:** use `BinaryData.FromString(...)` or a writer over `OpenWriteAsync`.
- **Download a blob:** use `DownloadToAsync(...)` for files or `DownloadContentAsync()` for small payloads.
- **Read metadata:** call `GetPropertiesAsync()` and inspect `.Metadata`.
- **Set metadata:** call `SetMetadataAsync(...)` with the full desired set.
- **Prefer secure auth:** use `DefaultAzureCredential` or managed identity over embedded secrets.
