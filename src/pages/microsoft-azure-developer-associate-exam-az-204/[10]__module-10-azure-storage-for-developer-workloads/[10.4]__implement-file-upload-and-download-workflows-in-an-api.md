## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**File workflow patterns in an API**

File handling in Azure applications usually falls into two patterns:

- **API-mediated transfer**: the API receives the file, validates it, and uploads or streams it to Azure Blob Storage.
- **Direct-to-blob transfer**: the API authorizes the client and returns a short-lived SAS (Shared Access Signature) URL so the client uploads or downloads directly.

The right choice depends on file size, trust boundaries, latency, and how much control you need over validation and auditing. For exam scenarios, think about where authentication happens, where validation happens, and which component owns the data path.

```text
Client
  | 1. Request upload/download authorization
  v
API (authN/authZ, metadata, SAS generation)
  | 2a. Stream file to Blob Storage
  | 2b. Return SAS URL
  v
Azure Blob Storage
```

**Blob Storage basics for file workflows**

Azure Blob Storage is the standard service for unstructured file content such as documents, images, logs, and media. In file workflows, you typically use:

- **Blob containers** to group related files.
- **Block blobs** for most upload/download scenarios.
- **Metadata** and **tags** to store searchable attributes such as owner, content type, or document type.
- **Access tiers** to optimize cost for hot, cool, or archive data.

A practical design is to store the file in Blob Storage and keep only the business record in your database, such as file ID, blob name, owner ID, checksum, size, and status.

**Authentication and authorization choices**

Production APIs should not rely on anonymous access. Use one of these patterns:

- **Managed identity** for the API to access Blob Storage without storing secrets.
- **Microsoft Entra ID** with Azure RBAC for service-to-service access.
- **Shared key** only when you explicitly need account-level signing and understand the security tradeoff.
- **SAS** for temporary delegated access to a specific blob or container.

For an API hosted in Azure, the preferred pattern is often:

1. Enable a managed identity on the API app service, function app, or container app.
2. Assign the identity a role such as `Storage Blob Data Contributor` or `Storage Blob Data Reader`.
3. Use `DefaultAzureCredential` in the SDK to authenticate to Blob Storage.

```csharp
using Azure.Identity;
using Azure.Storage.Blobs;

var serviceClient = new BlobServiceClient(
    new Uri("https://mystorageaccount.blob.core.windows.net"),
    new DefaultAzureCredential());
```

This avoids connection strings in app settings and supports local development through developer sign-in flows.

**Container access levels and why private is the default**

Blob containers can be configured with public access levels, but production APIs usually keep containers private.

- **Private access**: no anonymous read access; recommended for most applications.
- **Blob public access**: anonymous read access to individual blobs if enabled.
- **Container public access**: anonymous read access to blobs and listing in the container if enabled.

For exam and real-world security, prefer private containers and expose files through authenticated API endpoints or short-lived SAS URLs. Public access is only appropriate for truly public assets such as marketing images.

> If a container is private, clients cannot read or write directly unless you provide a SAS or another authorized path.

**Upload workflow: API-mediated**

In an API-mediated upload, the client sends the file to your API, and the API forwards it to Blob Storage. This gives you the most control over validation, auditing, and business rules.

Typical uses:

- You must inspect the file before storing it.
- You need to enforce tenant-specific rules.
- You want to scan for malware before the file becomes available.
- You need to write metadata to a database in the same request flow.

A common ASP.NET Core pattern is to accept `IFormFile`, validate it, and stream it to Blob Storage.

```csharp
[HttpPost("files")]
public async Task<IActionResult> Upload(
    IFormFile file,
    [FromServices] BlobContainerClient containerClient,
    CancellationToken cancellationToken)
{
    if (file is null || file.Length == 0)
        return BadRequest("File is required.");

    var allowedContentTypes = new[] { "application/pdf", "image/png", "image/jpeg" };
    if (!allowedContentTypes.Contains(file.ContentType))
        return BadRequest("Unsupported file type.");

    var blobName = $"{Guid.NewGuid():N}-{Path.GetFileName(file.FileName)}";
    var blobClient = containerClient.GetBlobClient(blobName);

    await using var stream = file.OpenReadStream();
    await blobClient.UploadAsync(
        stream,
        new BlobUploadOptions
        {
            HttpHeaders = new BlobHttpHeaders
            {
                ContentType = file.ContentType
            },
            Metadata =
            {
                ["originalName"] = file.FileName,
                ["uploadedBy"] = User.Identity?.Name ?? "unknown"
            }
        },
        cancellationToken);

    return CreatedAtAction(nameof(Download), new { blobName }, new { blobName });
}
```

Key points:

- **Stream** the file instead of buffering large files in memory.
- Set `ContentType` so downloads return the correct MIME type.
- Store metadata such as original file name and uploader identity.
- Return a stable identifier rather than exposing internal storage details.

**Upload validation patterns**

File upload validation should happen before and after storage. A robust workflow usually includes:

- **Size limits** at the API layer and reverse proxy layer.
- **Extension checks** as a first pass, but never as the only check.
- **MIME sniffing** or content inspection to verify the file type.
- **Antivirus or malware scanning** before the file is marked available.
- **Checksum verification** to detect corruption in transit.
- **ETag** comparison or conditional logic when you need optimistic concurrency.

A simple extension check is not enough because file names can be spoofed. For higher assurance, inspect the file signature, often called the magic number, and compare it to the declared content type.

```csharp
static bool LooksLikePdf(Stream stream)
{
    Span<byte> header = stackalloc byte[4];
    var read = stream.Read(header);
    stream.Position = 0;
    return read == 4 && header.SequenceEqual("%PDF"u8);
}
```

For integrity, you can compute a hash before or after upload and store it with the file record.

```csharp
using var sha256 = System.Security.Cryptography.SHA256.Create();
await using var source = file.OpenReadStream();
await using var hashingStream = new CryptoStream(Stream.Null, sha256, CryptoStreamMode.Write);

await source.CopyToAsync(hashingStream, cancellationToken);
var checksum = Convert.ToHexString(sha256.Hash!);
```

**Direct-to-blob upload with SAS**

A SAS is a signed token that grants limited access to a blob or container for a limited time. This is useful when you want the client to upload directly to Blob Storage without sending the file through your API.

Use SAS when:

- Files are large and you want to reduce API bandwidth.
- You want the browser or mobile app to upload directly.
- You need short-lived, scoped access for a single operation.

There are two common SAS types:

- **Service SAS**: signed with the storage account key.
- **User delegation SAS**: signed with a user delegation key obtained through Microsoft Entra ID.

For Azure-hosted APIs, user delegation SAS is usually preferred because it avoids exposing the account key.

**Important credential requirement for user delegation SAS**

To generate a user delegation SAS in an API, your app must:

1. Authenticate to Blob Storage with Microsoft Entra ID.
2. Call `GetUserDelegationKeyAsync`.
3. Use that key to sign the SAS.

A valid example looks like this:

```csharp
using Azure;
using Azure.Identity;
using Azure.Storage;
using Azure.Storage.Blobs;
using Azure.Storage.Sas;

public async Task<Uri> CreateUploadSasAsync(
    BlobClient blobClient,
    TimeSpan lifetime,
    CancellationToken cancellationToken)
{
    var serviceClient = blobClient.GetParentBlobContainerClient().GetParentBlobServiceClient();

    var startsOn = DateTimeOffset.UtcNow.AddMinutes(-5);
    var expiresOn = DateTimeOffset.UtcNow.Add(lifetime);

    UserDelegationKey delegationKey = await serviceClient.GetUserDelegationKeyAsync(
        startsOn,
        expiresOn,
        cancellationToken);

    var sasBuilder = new BlobSasBuilder
    {
        BlobContainerName = blobClient.BlobContainerName,
        BlobName = blobClient.Name,
        Resource = "b",
        StartsOn = startsOn,
        ExpiresOn = expiresOn
    };

    sasBuilder.SetPermissions(BlobSasPermissions.Create | BlobSasPermissions.Write);

    var sasQuery = sasBuilder.ToSasQueryParameters(
        delegationKey,
        serviceClient.AccountName);

    return blobClient.Uri.ReplaceQuery(sasQuery.ToString());
}
```

This pattern matters because the SAS must be signed with the delegation key. Retrieving the key alone is not enough.

For upload permissions:

- **Write** is the key permission for uploading blob content.
- **Create** is needed in some scenarios when a new blob must be created.
- Add only the permissions required for the specific operation.

> Do not grant broader permissions than necessary. For many upload flows, `Write` is the critical permission; `Create` is only needed when the workflow specifically requires it.

**Direct-to-blob upload request flow**

A common SaaS document upload flow looks like this:

1. The client calls `POST /files/upload-url` with file metadata.
2. The API checks the user, tenant, and file policy.
3. The API creates a blob name and generates a short-lived SAS URL.
4. The client uploads the file directly to Blob Storage with `PUT`.
5. The client notifies the API that upload completed.
6. The API persists metadata and marks the file as available after validation.

This pattern reduces API load and is ideal for large files, but you still need server-side authorization and post-upload validation.

**Practical SAS upload example with Azure CLI**

You can inspect or test storage access with the Azure CLI.

```bash
az login
az storage account show -n mystorageaccount -g my-rg
az storage container create \
  --account-name mystorageaccount \
  --name uploads \
  --auth-mode login
```

If you use a user delegation SAS from the API, the client can upload with `curl`:

```bash
curl -X PUT \
  -H "x-ms-blob-type: BlockBlob" \
  -H "Content-Type: application/pdf" \
  --data-binary @invoice.pdf \
  "https://mystorageaccount.blob.core.windows.net/uploads/invoice.pdf?<sas-token>"
```

**Download workflow: API-mediated**

For downloads, the API can either stream the file itself or return a SAS URL. API-mediated download is useful when you need to enforce authorization, log access, or transform the file.

A typical ASP.NET Core download action streams the blob to the response.

```csharp
[HttpGet("files/{blobName}")]
public async Task<IActionResult> Download(
    string blobName,
    [FromServices] BlobContainerClient containerClient,
    CancellationToken cancellationToken)
{
    var blobClient = containerClient.GetBlobClient(blobName);

    if (!await blobClient.ExistsAsync(cancellationToken))
        return NotFound();

    var download = await blobClient.DownloadStreamingAsync(cancellationToken);

    return File(
        download.Value.Content,
        download.Value.Details.ContentType ?? "application/octet-stream",
        fileDownloadName: blobName);
}
```

When you stream a blob from an API:

- Ensure the caller is authorized before opening the stream.
- Preserve the blob content type when possible.
- Be careful with stream ownership and disposal in your framework code.
- Consider whether the response should be buffered or streamed directly.

**Range requests and resumable downloads**

Range requests let clients request only part of a file, which is useful for large downloads, video playback, and resumable transfers. Blob Storage supports range reads, and ASP.NET Core can forward range support when configured correctly.

A practical API approach is to expose the blob as a file result and allow range processing when appropriate.

```csharp
[HttpGet("files/{blobName}/download")]
public async Task<IActionResult> DownloadWithRange(
    string blobName,
    [FromServices] BlobContainerClient containerClient,
    CancellationToken cancellationToken)
{
    var blobClient = containerClient.GetBlobClient(blobName);

    if (!await blobClient.ExistsAsync(cancellationToken))
        return NotFound();

    var download = await blobClient.DownloadStreamingAsync(cancellationToken);

    return File(
        download.Value.Content,
        download.Value.Details.ContentType ?? "application/octet-stream",
        enableRangeProcessing: true);
}
```

Range support is especially useful when:

- Users resume interrupted downloads.
- Browsers seek within media files.
- Large files should not be re-downloaded from the start.

**Direct-to-blob download with SAS**

You can also generate a read-only SAS for downloads. This is useful when the client should fetch the file directly from Blob Storage, reducing API bandwidth.

A correct user delegation SAS download example must sign the SAS with the delegation key:

```csharp
public async Task<Uri> CreateDownloadSasAsync(
    BlobClient blobClient,
    TimeSpan lifetime,
    CancellationToken cancellationToken)
{
    var serviceClient = blobClient.GetParentBlobContainerClient().GetParentBlobServiceClient();

    var startsOn = DateTimeOffset.UtcNow.AddMinutes(-5);
    var expiresOn = DateTimeOffset.UtcNow.Add(lifetime);

    var delegationKey = await serviceClient.GetUserDelegationKeyAsync(
        startsOn,
        expiresOn,
        cancellationToken);

    var sasBuilder = new BlobSasBuilder
    {
        BlobContainerName = blobClient.BlobContainerName,
        BlobName = blobClient.Name,
        Resource = "b",
        StartsOn = startsOn,
        ExpiresOn = expiresOn,
        ContentDisposition = $"attachment; filename=\"{blobClient.Name}\""
    };

    sasBuilder.SetPermissions(BlobSasPermissions.Read);

    var sas = sasBuilder.ToSasQueryParameters(delegationKey, serviceClient.AccountName);
    return blobClient.Uri.ReplaceQuery(sas.ToString());
}
```

This pattern is ideal when the API only needs to authorize access and does not need to proxy the bytes.

**Chunked and resumable uploads**

Large uploads should use block blob staging so the client can upload in chunks and resume after failures. This is the standard approach for files that are too large or unreliable to send in one request.

The workflow is:

1. Split the file into blocks.
2. Upload each block with a block ID.
3. Commit the block list when all blocks succeed.

```csharp
using Azure.Storage.Blobs;
using Azure.Storage.Blobs.Models;

public async Task UploadLargeFileAsync(
    BlobClient blobClient,
    Stream content,
    int blockSize,
    CancellationToken cancellationToken)
{
    var blockIds = new List<string>();
    var buffer = new byte[blockSize];
    int bytesRead;

    while ((bytesRead = await content.ReadAsync(buffer, cancellationToken)) > 0)
    {
        var blockId = Convert.ToBase64String(Guid.NewGuid().ToByteArray());
        using var blockStream = new MemoryStream(buffer, 0, bytesRead, writable: false);

        await blobClient.StageBlockAsync(blockId, blockStream, cancellationToken: cancellationToken);
        blockIds.Add(blockId);
    }

    await blobClient.CommitBlockListAsync(blockIds, cancellationToken: cancellationToken);
}
```

Use chunked uploads when:

- Files are large.
- Network interruptions are likely.
- You need retry logic per block.
- You want to show upload progress.

For exam scenarios, remember that block blobs support staged blocks and later commit, which is the core of resumable upload design.

**Metadata persistence and lifecycle management**

A complete file workflow usually includes a database record that tracks the file lifecycle. This helps you enforce authorization and keep the application state consistent.

Typical fields:

- `FileId`
- `BlobName`
- `OriginalFileName`
- `OwnerUserId`
- `TenantId`
- `ContentType`
- `Size`
- `Checksum`
- `Status`
- `CreatedUtc`
- `DeletedUtc`

A common lifecycle is:

1. `PendingUpload`
2. `Uploaded`
3. `Scanning`
4. `Available`
5. `Archived` or `Deleted`

This lets you prevent users from downloading a file before validation or malware scanning completes.

**Delete and update workflows**

Deleting a file should remove both the blob and the metadata record, or mark the record as deleted if you need soft delete behavior.

```csharp
[HttpDelete("files/{fileId}")]
public async Task<IActionResult> Delete(
    Guid fileId,
    [FromServices] BlobContainerClient containerClient,
    CancellationToken cancellationToken)
{
    var file = await _repository.FindAsync(fileId, cancellationToken);
    if (file is null)
        return NotFound();

    if (file.OwnerUserId != User.FindFirst("sub")?.Value)
        return Forbid();

    var blobClient = containerClient.GetBlobClient(file.BlobName);
    await blobClient.DeleteIfExistsAsync(DeleteSnapshotsOption.IncludeSnapshots, cancellationToken: cancellationToken);

    file.Status = "Deleted";
    await _repository.SaveChangesAsync(cancellationToken);

    return NoContent();
}
```

For updates, prefer versioning or replacement workflows rather than in-place mutation when the file is user-facing and auditability matters.

**Security and compliance considerations**

File workflows often need more than storage access. In production, consider:

- **Authorization checks** on every upload, download, and delete request.
- **Tenant isolation** so one customer cannot access another customer’s files.
- **Malware scanning** before making files available.
- **Content validation** to prevent dangerous file types.
- **Checksum validation** to detect corruption.
- **Logging and auditing** for access events.
- **Encryption at rest** provided by Azure Storage, with customer-managed keys if required by policy.

A secure API should never trust the client-provided file name, content type, or size alone.

**When to choose each pattern**

| Scenario | Best pattern | Why |
|---|---|---|
| Small files with strict validation | API-mediated upload | Full control over inspection and business rules |
| Large browser uploads | Direct-to-blob with SAS | Reduces API bandwidth and improves scalability |
| Secure downloads with auditing | API-mediated download | Centralized authorization and logging |
| Public static assets | Public blob access or CDN | Simplifies delivery for truly public content |
| Resumable large transfers | Block blob staging | Supports retries and partial progress |

**End-to-end SaaS document example**

A realistic document management flow ties everything together:

- The user signs in to the API with Microsoft Entra ID.
- The API checks tenant membership and role-based access control.
- The API creates a database record with `PendingUpload`.
- The API returns a short-lived upload SAS for a specific blob name.
- The client uploads directly to Blob Storage.
- The API receives a completion callback or polling request.
- The API validates checksum, scans the file, and marks it `Available`.
- The user later requests a download.
- The API either streams the file or returns a read-only SAS.
- The user deletes the document, and the API removes the blob and metadata.

This is the kind of end-to-end flow that often appears in AZ-204 scenarios because it combines identity, storage, security, and application design.

**Common exam pitfalls**

- Confusing **SAS generation** with blob access itself; the SAS must be signed correctly.
- Assuming a private container can be accessed without authorization.
- Using `Create` and `Write` as if both are always required for uploads.
- Buffering large files in memory instead of streaming.
- Forgetting to validate file type, size, and ownership.
- Treating download streams as automatically managed without considering lifetime and disposal.
- Omitting metadata persistence, which makes later authorization and cleanup difficult.

**Quick reference**

- Use **managed identity** and `DefaultAzureCredential` for production API access to Blob Storage.
- Keep containers **private** unless the content is intentionally public.
- Use **user delegation SAS** when the API authenticates with Microsoft Entra ID and needs to delegate access.
- Use **block blobs** for large or resumable uploads.
- Use **range processing** for large downloads and media scenarios.
- Store file metadata in a database so you can enforce authorization and lifecycle rules.
