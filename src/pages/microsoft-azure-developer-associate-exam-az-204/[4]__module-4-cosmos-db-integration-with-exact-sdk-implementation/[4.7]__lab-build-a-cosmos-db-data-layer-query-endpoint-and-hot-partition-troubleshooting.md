## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Lab overview**

This lab focuses on three practical skills you need when building applications on Azure Cosmos DB: creating a data layer, exposing a query endpoint, and diagnosing hot partitions. In real applications, these concerns are tightly connected. Your data model affects query cost, your query patterns affect partition usage, and partition design affects throughput, latency, and scale.

Azure Cosmos DB is a globally distributed, multi-model NoSQL database service. For AZ-204, you should understand how to use the SDK, design partition keys, write efficient queries, and recognize when a workload is concentrating traffic on a single logical partition.

- A **data layer** isolates database access from your API or UI code.
- A **query endpoint** exposes filtered reads without leaking storage details.
- A **hot partition** occurs when one partition key value receives disproportionate traffic.

**Cosmos DB concepts you should know**

Cosmos DB stores JSON documents in containers. A container is the unit of throughput and storage, and it is partitioned by a partition key path such as `/tenantId` or `/category`. The partition key determines how data is distributed and how efficiently queries can be routed.

<dl>
<dt>Database</dt>
<dd>A logical grouping of containers.</dd>
<dt>Container</dt>
<dd>The scalable unit for items, throughput, and partitioning.</dd>
<dt>Item</dt>
<dd>A JSON document stored in a container.</dd>
<dt>Partition key</dt>
<dd>A property used to distribute items and route requests.</dd>
<dt>Logical partition</dt>
<dd>All items that share the same partition key value.</dd>
<dt>Physical partition</dt>
<dd>The internal storage and throughput slice managed by Cosmos DB.</dd>
</dl>

A good partition key has enough cardinality to spread traffic, matches your most common access patterns, and avoids unbounded growth in a single logical partition. If you choose a low-cardinality key such as `status` or `region` for a high-write workload, you can create a hot partition even when total throughput looks sufficient.

**Building the data layer**

In an ASP.NET Core application, the data layer usually wraps the Cosmos DB SDK so controllers or minimal API handlers do not talk to the database directly. This keeps your code testable and centralizes retry logic, serialization settings, and query behavior.

A typical setup uses:

- `CosmosClient` as the shared, thread-safe client
- `Database` and `Container` references for operations
- Repository or service classes for domain-specific methods
- Dependency injection to manage lifetime and configuration

Example registration in `Program.cs`:

```csharp
using Microsoft.Azure.Cosmos;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddSingleton(sp =>
{
    var configuration = sp.GetRequiredService<IConfiguration>();
    var connectionString = configuration["Cosmos:ConnectionString"];
    return new CosmosClient(connectionString);
});

builder.Services.AddSingleton<ICustomerRepository, CosmosCustomerRepository>();

var app = builder.Build();
app.MapControllers();
app.Run();
```

A repository implementation can encapsulate common operations:

```csharp
using Microsoft.Azure.Cosmos;

public sealed class CosmosCustomerRepository : ICustomerRepository
{
    private readonly Container _container;

    public CosmosCustomerRepository(CosmosClient client, IConfiguration configuration)
    {
        var databaseName = configuration["Cosmos:DatabaseName"];
        var containerName = configuration["Cosmos:ContainerName"];
        _container = client.GetContainer(databaseName, containerName);
    }

    public async Task<Customer?> GetAsync(string id, string tenantId)
    {
        try
        {
            var response = await _container.ReadItemAsync<Customer>(
                id,
                new PartitionKey(tenantId));

            return response.Resource;
        }
        catch (CosmosException ex) when (ex.StatusCode == System.Net.HttpStatusCode.NotFound)
        {
            return null;
        }
    }

    public async Task UpsertAsync(Customer customer)
    {
        await _container.UpsertItemAsync(customer, new PartitionKey(customer.TenantId));
    }
}
```

Key implementation points:

- Use a singleton `CosmosClient` because it is expensive to create and designed for reuse.
- Pass the partition key when reading, updating, or deleting a single item.
- Catch `CosmosException` for service-specific failures such as `404 NotFound` or `429 TooManyRequests`.
- Keep serialization consistent between your model and stored JSON.

> If you omit the partition key on point reads, Cosmos DB may need a cross-partition lookup, which is slower and more expensive.

**Modeling items for efficient access**

Your item shape should support both writes and queries. In many applications, you store the partition key value directly in the document so the application can route requests without extra lookups.

Example model:

```csharp
using System.Text.Json.Serialization;

public sealed class Customer
{
    [JsonPropertyName("id")]
    public string Id { get; set; } = default!;

    public string TenantId { get; set; } = default!;

    public string Email { get; set; } = default!;

    public string Status { get; set; } = "Active";

    public DateTimeOffset CreatedUtc { get; set; }
}
```

Practical modeling guidance:

- Use a stable partition key value that does not change often.
- Prefer keys that distribute writes evenly across tenants, users, or devices.
- Avoid using a field that is frequently updated as the partition key.
- Include fields that support your most common filters and sort orders.

**Query endpoint design**

A query endpoint usually accepts filter parameters and translates them into a Cosmos DB SQL query. Cosmos DB uses a SQL-like syntax over JSON documents, and the SDK supports parameterized queries to avoid injection issues and improve readability.

Example minimal API endpoint:

```csharp
app.MapGet("/customers", async (
    string? tenantId,
    string? status,
    ICustomerQueryService service) =>
{
    var results = await service.SearchAsync(tenantId, status);
    return Results.Ok(results);
});
```

Service implementation with parameterized query:

```csharp
using Microsoft.Azure.Cosmos;

public sealed class CosmosCustomerQueryService : ICustomerQueryService
{
    private readonly Container _container;

    public CosmosCustomerQueryService(CosmosClient client, IConfiguration configuration)
    {
        _container = client.GetContainer(
            configuration["Cosmos:DatabaseName"],
            configuration["Cosmos:ContainerName"]);
    }

    public async Task<IReadOnlyList<Customer>> SearchAsync(string? tenantId, string? status)
    {
        var sql = "SELECT * FROM c WHERE (@tenantId IS NULL OR c.TenantId = @tenantId) " +
                  "AND (@status IS NULL OR c.Status = @status)";

        var query = new QueryDefinition(sql)
            .WithParameter("@tenantId", tenantId)
            .WithParameter("@status", status);

        var results = new List<Customer>();
        using var iterator = _container.GetItemQueryIterator<Customer>(query);

        while (iterator.HasMoreResults)
        {
            var page = await iterator.ReadNextAsync();
            results.AddRange(page.Resource);
        }

        return results;
    }
}
```

Important query behaviors:

- Parameterize values with `QueryDefinition`.
- Use `GetItemQueryIterator<T>` for paged results.
- Expect queries to return multiple pages when the result set is large.
- Use filters that align with the partition key when possible.

A query that includes the partition key is usually cheaper than a cross-partition query because Cosmos DB can route the request more precisely. If your endpoint supports tenant-scoped access, include `tenantId` in the query and in the partition key design.

**Point reads versus queries**

Cosmos DB point reads are the fastest and cheapest way to retrieve a single item. They require both the item `id` and the partition key value. Queries are more flexible but can consume more request units and latency.

| Operation | Input needed | Typical use | Cost profile |
|---|---|---|---|
| Point read | `id` + partition key | Fetch one known item | Lowest |
| Query | SQL + parameters | Search by filters | Higher |
| Cross-partition query | SQL across partitions | Broad search | Highest |

Use point reads when:

- You already know the item identity.
- The request is scoped to one partition key value.
- You need the lowest latency for a single record.

Use queries when:

- You need to filter by non-key fields.
- You need a list or search result.
- You cannot determine the exact item ID in advance.

**Pagination and continuation tokens**

Cosmos DB returns query results in pages. The SDK handles continuation tokens internally through iterators, but you should still design your API to support paging for large result sets.

Common patterns:

- Return a limited page size from the API.
- Preserve continuation tokens if the client needs to resume.
- Avoid loading unbounded result sets into memory.

Example with a page size:

```csharp
var requestOptions = new QueryRequestOptions
{
    MaxItemCount = 25
};

using var iterator = _container.GetItemQueryIterator<Customer>(
    queryDefinition,
    requestOptions: requestOptions);
```

If you expose paging to clients, you may return a token alongside the data. The exact token handling depends on your API design and SDK usage, but the principle is the same: do not assume a query returns everything at once.

**Throughput and request units**

Cosmos DB measures throughput consumption in request units, or RUs. A request unit is a normalized measure of compute and I/O cost. Reads, writes, and queries all consume RUs, and the cost depends on item size, indexing, query shape, and partition routing.

Factors that increase RU consumption:

- Large documents
- Broad queries
- Cross-partition scans
- Inefficient filters
- High write rates on indexed fields

Ways to reduce RU usage:

- Use point reads for known items.
- Filter on partition key when possible.
- Keep documents small and focused.
- Index only what you need when using custom indexing policies.
- Avoid `SELECT *` when you only need a subset of fields.

Example of a projection query:

```sql
SELECT c.id, c.Email, c.Status
FROM c
WHERE c.TenantId = @tenantId
```

**Hot partition troubleshooting**

A hot partition happens when one logical partition receives much more traffic than others. This can cause throttling, latency spikes, and uneven RU consumption even if the container has enough total throughput.

Common causes:

- Low-cardinality partition keys
- A single tenant or user generating most traffic
- Time-based keys that concentrate writes
- Sequential keys that cluster access patterns
- Queries that repeatedly target one partition value

Symptoms you may observe:

- `429 TooManyRequests` responses
- High latency for a subset of requests
- One partition consuming most of the RU budget
- Successful overall throughput but poor performance for one tenant

Troubleshooting approach:

1. Identify the partition key value receiving the most traffic.
2. Review application logs and Cosmos DB metrics for throttling and latency.
3. Check whether the partition key has enough cardinality.
4. Confirm whether the workload is write-heavy, read-heavy, or both.
5. Evaluate whether the access pattern can be spread across more partition values.

Useful Azure CLI commands for inspection:

```bash
az cosmosdb show \
  --name <account-name> \
  --resource-group <resource-group>
```

```bash
az monitor metrics list \
  --resource <cosmos-resource-id> \
  --metric TotalRequestUnits,ThrottledRequests,NormalizedRUConsumption \
  --interval PT1M
```

In the Azure portal, look for:

- Throttled requests
- Normalized RU consumption
- Data distribution across partitions
- Latency by operation type

> A single hot logical partition can throttle even when the container still has unused throughput elsewhere.

**Partition key remediation strategies**

If you confirm a hot partition, the fix is usually architectural rather than operational. You often need to change the partition key or change how you distribute writes.

Possible strategies:

- Choose a higher-cardinality partition key.
- Use a composite key pattern in the document, such as `tenantId-userId`.
- Add a synthetic key to spread load, such as a hash bucket.
- Split large tenants into multiple logical buckets if your access pattern allows it.
- Rework queries so they do not concentrate on one value.

Example synthetic key:

```csharp
public sealed class Order
{
    public string Id { get; set; } = default!;
    public string TenantId { get; set; } = default!;
    public string Bucket { get; set; } = default!;
    public string PartitionKey => $"{TenantId}:{Bucket}";
}
```

You can assign `Bucket` using a hash or modulo strategy to spread writes:

```csharp
var bucket = Math.Abs(order.Id.GetHashCode()) % 10;
order.Bucket = bucket.ToString();
```

This approach improves distribution, but it also makes reads more complex because you may need to query multiple buckets to find all items for a tenant. Use it only when the workload justifies the tradeoff.

**Indexing considerations**

Cosmos DB automatically indexes JSON properties by default. This is convenient for development, but indexing every property can increase write cost. For query-heavy applications, the default policy is often acceptable. For write-heavy workloads, you may want to tune indexing.

General guidance:

- Keep default indexing if you are unsure.
- Exclude large or rarely queried paths when write cost matters.
- Ensure fields used in filters and sorts remain indexed.
- Test query RU consumption after indexing changes.

Example custom indexing policy conceptually:

```json
{
  "indexingMode": "consistent",
  "includedPaths": [
    { "path": "/*" }
  ],
  "excludedPaths": [
    { "path": "/largeBlob/?" }
  ]
}
```

**Error handling and retries**

Cosmos DB can throttle requests when a partition or container exceeds its allocated throughput. The SDK includes retry behavior for transient failures, but your application should still handle exceptions explicitly.

Common exceptions:

- `CosmosException` with status `429` for throttling
- `404` for missing items
- `400` for invalid requests or malformed queries
- `409` for conflicts in some write scenarios

Practical handling pattern:

```csharp
try
{
    var response = await _container.UpsertItemAsync(item, new PartitionKey(item.TenantId));
}
catch (CosmosException ex) when (ex.StatusCode == System.Net.HttpStatusCode.TooManyRequests)
{
    // Log and retry with backoff if appropriate
}
```

Use retries carefully:

- Retry transient throttling with exponential backoff.
- Do not retry invalid queries without fixing the request.
- Log partition key values when diagnosing repeated failures.
- Correlate request IDs when available.

**Testing the query endpoint**

You should validate both correctness and performance. A query endpoint can return the right data but still be inefficient if it scans too much data.

Test cases to include:

- Query by partition key only
- Query by partition key plus status
- Query with no filters
- Missing item lookup
- Empty result set
- Large result set with paging

Example `curl` request:

```bash
curl "https://localhost:5001/customers?tenantId=t1&status=Active"
```

Example expected behavior:

- The endpoint returns only items matching the filters.
- The repository uses parameterized queries.
- The API does not expose Cosmos-specific internals to clients.
- The endpoint handles empty results gracefully.

**Practical architecture pattern**

A clean Cosmos DB-backed API often follows this flow:

1. The controller or minimal API receives a request.
2. The service validates input and chooses a read strategy.
3. The repository executes a point read or query.
4. The SDK returns results or a `CosmosException`.
5. The API maps the result to HTTP responses.

This separation helps you:

- Swap query logic without changing controllers.
- Mock the repository in tests.
- Centralize partition key handling.
- Add telemetry for RU usage and latency.

**Telemetry and observability**

For production troubleshooting, collect metrics and logs that help you connect application behavior to Cosmos DB behavior.

Track:

- Request rate
- Latency
- Throttled requests
- RU consumption
- Partition key values for hot paths
- Query duration and result size

If you use Application Insights or another observability tool, include correlation IDs and operation names so you can trace a slow request from the API to the database call.

**Exam-focused takeaways**

For AZ-204, focus on the relationship between data modeling and performance. Cosmos DB is not just a storage API; it is a distributed system where partition design determines scalability.

Remember these points:

- Use a singleton `CosmosClient`.
- Prefer point reads when you know `id` and partition key.
- Parameterize queries with `QueryDefinition`.
- Design partition keys for even distribution and common access patterns.
- Investigate `429` errors and high latency as signs of hot partitions.
- Use metrics and logs to confirm whether the issue is throughput, partitioning, or query shape.

**Quick reference**

- **Best read path:** point read with `id` and partition key
- **Best query path:** parameterized SQL query with partition-aware filters
- **Most common scaling issue:** hot partition from poor key choice
- **Most useful diagnostic signal:** throttled requests and normalized RU consumption
- **Most important design rule:** model for your access pattern, not just for storage
