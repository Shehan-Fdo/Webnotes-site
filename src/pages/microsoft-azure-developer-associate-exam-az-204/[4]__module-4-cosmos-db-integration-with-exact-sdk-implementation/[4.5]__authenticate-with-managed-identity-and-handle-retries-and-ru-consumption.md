## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Managed identity authentication**

<abbr title="Managed Identity">MI</abbr> lets Azure resources authenticate to other Azure services without storing secrets, certificates, or connection strings in code. You use it when your app runs in Azure and needs to access services such as Azure Cosmos DB, Azure Key Vault, Azure Storage, or Azure Service Bus.

The main benefit is operational: you remove secret rotation from the application path and rely on Azure Active Directory, now Microsoft Entra ID, for token issuance. In exam scenarios, managed identity is the preferred choice when the workload is hosted in Azure and the target service supports Entra ID-based authorization.

- **System-assigned identity** is tied to one Azure resource and is deleted with it.
- **User-assigned identity** is a standalone identity that you can attach to multiple resources.
- **Token-based auth** replaces embedded credentials and reduces secret leakage risk.
- **Least privilege** still applies: assign only the roles the app needs.

A common pattern is to enable managed identity on the host, then grant that identity access to the target resource using Azure role assignments or service-specific data-plane roles.

```bash
# Enable system-assigned managed identity on an App Service
az webapp identity assign \
  --resource-group rg-app \
  --name my-webapp

# Show the principal ID
az webapp identity show \
  --resource-group rg-app \
  --name my-webapp \
  --query principalId -o tsv
```

For user-assigned identities, you create the identity first and then attach it to the host.

```bash
# Create a user-assigned managed identity
az identity create \
  --resource-group rg-app \
  --name uami-orders

# Attach it to a web app
az webapp identity assign \
  --resource-group rg-app \
  --name my-webapp \
  --identities /subscriptions/<subId>/resourceGroups/rg-app/providers/Microsoft.ManagedIdentity/userAssignedIdentities/uami-orders
```

**Using managed identity in .NET**

The Azure SDKs use `DefaultAzureCredential` to simplify authentication. It can try multiple credential sources available in the current environment, such as environment variables, managed identity, Visual Studio, Azure CLI, and others depending on SDK version and configuration. Do not assume a single fixed order across all environments; instead, treat it as a chain of available credentials that the SDK evaluates.

```csharp
using Azure.Identity;
using Azure.Security.KeyVault.Secrets;

var client = new SecretClient(
    new Uri("https://myvault.vault.azure.net/"),
    new DefaultAzureCredential());

KeyVaultSecret secret = await client.GetSecretAsync("DbPassword");
Console.WriteLine(secret.Value);
```

When you need to target a specific user-assigned managed identity, configure the credential explicitly. Depending on the hosting model and SDK version, you can use the client ID or the resource ID.

```csharp
using Azure.Identity;

var options = new DefaultAzureCredentialOptions
{
    ManagedIdentityClientId = "<user-assigned-client-id>"
    // or ManagedIdentityResourceId = new ResourceIdentifier(
    //     "/subscriptions/<subId>/resourceGroups/<rg>/providers/Microsoft.ManagedIdentity/userAssignedIdentities/<name>")
};

var credential = new DefaultAzureCredential(options);
```

> Use `ManagedIdentityClientId` or `ManagedIdentityResourceId` when the app has multiple user-assigned identities or when the platform requires explicit selection.

**Cosmos DB authentication and data-plane access**

Azure Cosmos DB supports Microsoft Entra ID for data-plane access in supported SDKs and account configurations. You still need to grant the identity a Cosmos DB data-plane role that allows the required operations. This is separate from Azure RBAC management-plane permissions.

A typical flow is:

1. Enable managed identity on the app.
2. Assign a Cosmos DB data-plane role to that identity.
3. Use an SDK credential such as `DefaultAzureCredential`.
4. Connect to the account endpoint and perform operations.

For SQL API accounts, the role assignment command needs a role assignment name and a full role definition resource ID or a valid built-in role definition reference. A runnable example looks like this:

```bash
# Create a Cosmos DB SQL role assignment for a managed identity
az cosmosdb sql role assignment create \
  --account-name mycosmosacct \
  --resource-group rg-data \
  --role-assignment-name 7f3f2f7d-8f0d-4d8a-9f0a-2f7f0f0a1111 \
  --role-definition-id "/subscriptions/<subId>/resourceGroups/rg-data/providers/Microsoft.DocumentDB/databaseAccounts/mycosmosacct/sqlRoleDefinitions/<roleDefinitionGuid>" \
  --principal-id <managed-identity-principal-id> \
  --scope "/subscriptions/<subId>/resourceGroups/rg-data/providers/Microsoft.DocumentDB/databaseAccounts/mycosmosacct"
```

If you use a built-in role, verify the exact role definition ID for your account and API. The important exam point is that the assignment must reference a valid role definition and the identity’s principal ID.

**Handling retries in Azure SDKs**

Retries matter because cloud calls fail transiently. A transient failure is temporary and often resolves if you retry after a short delay. Common causes include throttling, network blips, and service-side transient errors.

The Azure SDKs generally include retry policies, but you should tune them for the service and workload. For Cosmos DB, retries are especially important because the service can return `429 Too Many Requests` when you exceed provisioned throughput.

- **Transient errors** are retry candidates.
- **Permanent errors** such as invalid credentials should not be retried blindly.
- **Exponential backoff** reduces pressure on the service.
- **Jitter** helps avoid synchronized retry storms.

In .NET, you can configure retry behavior on many Azure clients through options objects.

```csharp
using Azure.Core;
using Azure.Identity;
using Azure.Cosmos;

var options = new CosmosClientOptions
{
    ConnectionMode = ConnectionMode.Direct,
    MaxRetryAttemptsOnRateLimitedRequests = 9,
    MaxRetryWaitTimeOnRateLimitedRequests = TimeSpan.FromSeconds(30)
};

var client = new CosmosClient(
    accountEndpoint: "https://mycosmosacct.documents.azure.com:443/",
    tokenCredential: new DefaultAzureCredential(),
    clientOptions: options);
```

For HTTP-based SDK clients, you may also see retry settings on `ClientOptions`:

```csharp
var options = new SomeAzureClientOptions
{
    Retry =
    {
        Mode = RetryMode.Exponential,
        Delay = TimeSpan.FromMilliseconds(200),
        MaxDelay = TimeSpan.FromSeconds(5),
        MaxRetries = 5
    }
};
```

**Cosmos DB request units and RU consumption**

<abbr title="Request Unit">RU</abbr> means Request Unit, the normalized measure of throughput in Azure Cosmos DB. RU consumption reflects the cost of reads, writes, queries, and stored procedure execution. The exact RU charge depends on item size, indexing, query shape, and consistency-related work.

Think of RU as the currency of Cosmos DB performance:

- **Reads** usually cost less than writes.
- **Queries** can cost more than point reads, especially if they scan many items.
- **Large documents** consume more RU than small documents.
- **Indexes** improve query efficiency but add write overhead.
- **Cross-partition queries** often consume more RU than single-partition operations.

A point read is the most efficient way to fetch a single item when you know the `id` and partition key.

```csharp
var response = await container.ReadItemAsync<Order>(
    id: "order-1001",
    partitionKey: new PartitionKey("customer-42"));

Console.WriteLine($"RU charge: {response.RequestCharge}");
```

A query can be more expensive, especially if it filters across partitions.

```csharp
var query = new QueryDefinition(
    "SELECT * FROM c WHERE c.status = @status")
    .WithParameter("@status", "Pending");

using FeedIterator<Order> iterator = container.GetItemQueryIterator<Order>(
    queryDefinition: query,
    requestOptions: new QueryRequestOptions
    {
        PartitionKey = new PartitionKey("customer-42"),
        MaxItemCount = 50
    });

while (iterator.HasMoreResults)
{
    FeedResponse<Order> page = await iterator.ReadNextAsync();
    Console.WriteLine($"Page RU charge: {page.RequestCharge}");

    foreach (Order order in page)
    {
        Console.WriteLine(order.Id);
    }
}
```

If you project only a subset of fields, make sure your model matches the returned JSON shape. Otherwise, use a matching DTO or read the response as a dynamic/document type.

```csharp
public sealed class OrderSummary
{
    public string Id { get; set; } = default!;
    public string Status { get; set; } = default!;
}

var summaryQuery = new QueryDefinition(
    "SELECT c.id, c.status FROM c WHERE c.status = @status")
    .WithParameter("@status", "Pending");

using FeedIterator<OrderSummary> iterator = container.GetItemQueryIterator<OrderSummary>(summaryQuery);
```

**Why RU matters for retries**

Retries can increase RU consumption if you repeat expensive operations. That is acceptable for transient failures, but you should avoid retrying in a way that amplifies load.

- **429 responses** indicate throttling and often include a retry-after hint.
- **Aggressive retries** can worsen throttling and increase latency.
- **Smaller pages** can reduce per-request spikes.
- **Partition-aware queries** help keep RU usage predictable.

Cosmos DB SDKs typically honor server guidance for throttling and retry after a delay. You should still design your application to reduce the chance of hitting the limit in the first place.

**Practical Cosmos DB tuning strategies**

Use these patterns to reduce RU consumption and improve reliability:

- **Use point reads** instead of queries when you know the item key.
- **Choose a good partition key** to distribute load evenly.
- **Avoid cross-partition scans** unless necessary.
- **Project only needed fields** to reduce payload and query cost.
- **Limit page size** when streaming large result sets.
- **Review indexing policy** to remove unused paths and reduce write cost.
- **Batch related operations** when the API supports it.

A simple indexing policy example:

```json
{
  "indexingMode": "consistent",
  "automatic": true,
  "includedPaths": [
    { "path": "/*" }
  ],
  "excludedPaths": [
    { "path": "/largeBlob/*" }
  ]
}
```

Excluding large, rarely queried properties can reduce write RU consumption. Use this carefully, because excluded fields are not efficiently queryable.

**Retry and RU design pattern**

A robust application usually combines identity, retry, and RU awareness:

1. Authenticate with `DefaultAzureCredential`.
2. Use managed identity in Azure hosting environments.
3. Prefer point reads and partition-key-aware queries.
4. Inspect `RequestCharge` to understand cost.
5. Handle `429` with SDK retries and backoff.
6. Log latency, retry count, and RU charge for hot paths.

```csharp
try
{
    ItemResponse<Order> response = await container.ReadItemAsync<Order>(
        "order-1001",
        new PartitionKey("customer-42"));

    logger.LogInformation("Read order with RU {Ru}", response.RequestCharge);
}
catch (CosmosException ex) when (ex.StatusCode == System.Net.HttpStatusCode.TooManyRequests)
{
    logger.LogWarning("Throttled. RetryAfter={RetryAfter}", ex.RetryAfter);
    throw;
}
```

**Exam-focused distinctions**

- **Managed identity** authenticates the app to Azure services without secrets.
- **Azure RBAC** controls access to Azure resources; Cosmos DB data-plane roles control database operations.
- **Retry policies** handle transient failures, not authorization failures.
- **RU consumption** is a performance and cost concern, not an authentication feature.
- **Partition key choice** strongly affects query cost and scalability.

**Common pitfalls**

- Using a managed identity but forgetting to grant the target service permission.
- Assuming `DefaultAzureCredential` always uses the same source order in every environment.
- Retrying authorization failures such as `401 Unauthorized` or `403 Forbidden`.
- Querying Cosmos DB when a point read would be cheaper.
- Projecting partial fields into a model that does not match the query result shape.
- Ignoring `429` responses and not observing `RetryAfter`.

**Quick reference**

| Topic | What to remember |
|---|---|
| Managed identity | Secretless Azure authentication for hosted workloads |
| `DefaultAzureCredential` | Simplifies auth by using available credentials in the environment |
| Cosmos DB RU | Measures request cost and throughput consumption |
| `429 Too Many Requests` | Throttling signal; retry after delay |
| Point read | Lowest-cost way to fetch a single item by `id` and partition key |
| Query tuning | Use partition keys, projections, and indexing wisely |

**Minimal end-to-end example**

```csharp
using Azure.Identity;
using Azure.Cosmos;

var credential = new DefaultAzureCredential();

var client = new CosmosClient(
    "https://mycosmosacct.documents.azure.com:443/",
    credential,
    new CosmosClientOptions
    {
        MaxRetryAttemptsOnRateLimitedRequests = 9,
        MaxRetryWaitTimeOnRateLimitedRequests = TimeSpan.FromSeconds(30)
    });

var container = client.GetContainer("appdb", "orders");

var item = await container.ReadItemAsync<Order>(
    "order-1001",
    new PartitionKey("customer-42"));

Console.WriteLine($"Order status: {item.Resource.Status}");
Console.WriteLine($"RU charge: {item.RequestCharge}");
```

This pattern is strong for AZ-204 because it combines secure authentication, service-aware retries, and cost-aware data access.
