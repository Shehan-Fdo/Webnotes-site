## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**CRUD with Azure SDKs: the core pattern**

CRUD means **Create, Read, Update, Delete**. In Azure SDKs, you usually work with a service-specific client, then a subclient or operation object for the resource you want to manage. The exact method names vary by service, but the workflow is consistent:

- Create a client with credentials.
- Target a container, queue, table, vault, or database.
- Call a create/read/update/delete method.
- Handle response metadata such as **ETag**, status codes, and version IDs.

For AZ-204, you should know both the conceptual pattern and the practical differences between the .NET and Python SDKs. The biggest exam traps are version-sensitive method names, sync vs async usage, and confusing management-plane APIs with data-plane APIs.

**SDK selection and package basics**

Use the correct package for the service and the correct client type for the operation. Azure SDKs are split by service and by plane:

- **Data plane** SDKs manage service data, such as blobs, secrets, queue messages, and Cosmos DB items.
- **Management plane** SDKs manage Azure resources, such as storage accounts or Key Vault resources.

Common package examples:

```bash
dotnet add package Azure.Storage.Blobs
dotnet add package Azure.Storage.Queues
dotnet add package Azure.Data.Tables
dotnet add package Azure.Security.KeyVault.Secrets
dotnet add package Azure.Cosmos
```

```bash
pip install azure-storage-blob azure-storage-queue azure-data-tables azure-keyvault-secrets azure-cosmos azure-identity
```

Use `Azure.Identity` in .NET and `azure-identity` in Python when you authenticate with Microsoft Entra ID. For local development, `DefaultAzureCredential` is the usual choice because it can use environment variables, Visual Studio, Azure CLI, managed identity, and other sources.

**Authentication and client creation**

Most modern Azure SDKs support token-based authentication. Prefer Microsoft Entra ID over shared keys when the service and scenario allow it.

```csharp
using Azure.Identity;
using Azure.Storage.Blobs;

var blobServiceClient = new BlobServiceClient(
    new Uri("https://mystorageaccount.blob.core.windows.net"),
    new DefaultAzureCredential());
```

```python
from azure.identity import DefaultAzureCredential
from azure.storage.blob import BlobServiceClient

credential = DefaultAzureCredential()
blob_service_client = BlobServiceClient(
    account_url="https://mystorageaccount.blob.core.windows.net",
    credential=credential
)
```

Key points:

- **Client constructors** differ by package version, but the pattern is always endpoint plus credential.
- Use `account_url` in Python when the SDK documents it.
- Use `new Uri(...)` in .NET for endpoint-based clients.
- If you use a connection string or key, do so only when the scenario requires it.

**Blob Storage CRUD**

Blob Storage is a common AZ-204 data-plane example. You typically create a service client, then a container client, then a blob client.

Typical operations:

- **Create**: upload a blob.
- **Read**: download or inspect properties.
- **Update**: overwrite content or metadata.
- **Delete**: remove the blob.

```csharp
using Azure.Storage.Blobs;
using Azure.Storage.Blobs.Models;

var blobServiceClient = new BlobServiceClient(
    new Uri("https://mystorageaccount.blob.core.windows.net"),
    new DefaultAzureCredential());

BlobContainerClient container = blobServiceClient.GetBlobContainerClient("images");
await container.CreateIfNotExistsAsync();

BlobClient blob = container.GetBlobClient("logo.png");

using var stream = File.OpenRead("logo.png");
await blob.UploadAsync(stream, overwrite: true);

BlobProperties props = await blob.GetPropertiesAsync();
Console.WriteLine($"ETag: {props.ETag}");

var download = await blob.DownloadContentAsync();
byte[] bytes = download.Value.Content.ToArray();

await blob.SetMetadataAsync(new Dictionary<string, string>
{
    ["source"] = "marketing",
    ["env"] = "dev"
});

await blob.DeleteIfExistsAsync();
```

```python
from azure.identity import DefaultAzureCredential
from azure.storage.blob import BlobServiceClient

credential = DefaultAzureCredential()
blob_service_client = BlobServiceClient(
    account_url="https://mystorageaccount.blob.core.windows.net",
    credential=credential
)

container_client = blob_service_client.get_container_client("images")
container_client.create_container()

blob_client = container_client.get_blob_client("logo.png")

with open("logo.png", "rb") as data:
    blob_client.upload_blob(data, overwrite=True)

props = blob_client.get_blob_properties()
print(props.etag)

download = blob_client.download_blob()
content = download.readall()

blob_client.set_blob_metadata({
    "source": "marketing",
    "env": "dev"
})

blob_client.delete_blob()
```

Important Blob Storage notes:

- `upload_blob(..., overwrite=True)` in Python and `UploadAsync(..., overwrite: true)` in .NET are common overwrite patterns.
- Metadata is separate from content. Updating metadata does not rewrite the blob body.
- Use `GetPropertiesAsync` / `get_blob_properties()` when you need the current **ETag** or content headers.
- Blob operations are generally strongly consistent for committed writes, but do not overgeneralize list or replication behavior.
- For large files, use streaming APIs rather than loading the entire file into memory.

**Conditional access and optimistic concurrency**

Many Azure SDK operations support conditional requests using **ETag** values. An ETag is a version identifier for a resource. You use it to avoid overwriting changes made by another client.

Common patterns:

- Read the resource and capture its ETag.
- Send the ETag back on update or delete.
- Fail the operation if the resource changed since you read it.

```csharp
var blob = container.GetBlobClient("logo.png");
BlobProperties props = await blob.GetPropertiesAsync();

await blob.SetMetadataAsync(
    new Dictionary<string, string> { ["reviewed"] = "true" },
    conditions: new Azure.Storage.Blobs.Models.BlobRequestConditions
    {
        IfMatch = props.ETag
    });
```

```python
from azure.core.exceptions import ResourceModifiedError
from azure.core.match_conditions import MatchConditions

props = blob_client.get_blob_properties()

try:
    blob_client.set_blob_metadata(
        {"reviewed": "true"},
        etag=props.etag,
        match_condition=MatchConditions.IfNotModified
    )
except ResourceModifiedError:
    print("Blob changed before update")
```

Use conditional headers when you need **optimistic concurrency**. Do not assume every SDK method accepts the same conditional parameters; support varies by service and operation.

**Queue Storage CRUD and message operations**

Queue Storage is not a classic CRUD store, but it is a common Azure SDK data-plane pattern and often appears in exam prep. The main operations are create queue, send message, receive message, update message, and delete message.

```csharp
using Azure.Storage.Queues;

var queueClient = new QueueClient(
    new Uri("https://mystorageaccount.queue.core.windows.net/orders"),
    new DefaultAzureCredential());

await queueClient.CreateIfNotExistsAsync();

await queueClient.SendMessageAsync("order-1001");

QueueMessage[] messages = await queueClient.ReceiveMessagesAsync(maxMessages: 1);
foreach (var message in messages)
{
    Console.WriteLine(message.MessageText);

    await queueClient.UpdateMessageAsync(
        message.MessageId,
        message.PopReceipt,
        "order-1001-processing",
        visibilityTimeout: TimeSpan.FromSeconds(30));

    await queueClient.DeleteMessageAsync(message.MessageId, message.PopReceipt);
}
```

```python
from azure.storage.queue import QueueClient

queue_client = QueueClient(
    account_url="https://mystorageaccount.queue.core.windows.net",
    queue_name="orders",
    credential=credential
)

queue_client.create_queue()
queue_client.send_message("order-1001")

messages = queue_client.receive_messages(messages_per_page=1)
for message in messages:
    print(message.content)

    queue_client.update_message(
        message.id,
        message.pop_receipt,
        content="order-1001-processing",
        visibility_timeout=30
    )

    queue_client.delete_message(message.id, message.pop_receipt)
```

Queue-specific reminders:

- A received message is not removed automatically.
- You need the **pop receipt** to update or delete a message after receiving it.
- Visibility timeout controls when a message becomes visible again if processing fails.
- Queue operations are message-oriented, so “update” usually means changing message content or visibility, not editing a stored entity.

**Table Storage CRUD**

Azure Table Storage is a key-value NoSQL store with partitioning. It is useful for simple, scalable entity storage and is a common SDK pattern to know.

Core concepts:

- **PartitionKey** groups related entities.
- **RowKey** uniquely identifies an entity within a partition.
- Together, they form the entity identity.

```csharp
using Azure.Data.Tables;

var tableClient = new TableClient(
    new Uri("https://mystorageaccount.table.core.windows.net"),
    "users",
    new DefaultAzureCredential());

await tableClient.CreateIfNotExistsAsync();

var entity = new TableEntity("A", "1001")
{
    ["Name"] = "Ada",
    ["Role"] = "Admin"
};

await tableClient.AddEntityAsync(entity);

TableEntity read = await tableClient.GetEntityAsync<TableEntity>("A", "1001");
read["Role"] = "Reader";

await tableClient.UpdateEntityAsync(read, ETag.All, TableUpdateMode.Replace);

await tableClient.DeleteEntityAsync("A", "1001");
```

```python
from azure.data.tables import TableClient
from azure.core.exceptions import ResourceNotFoundError

table_client = TableClient(
    endpoint="https://mystorageaccount.table.core.windows.net",
    table_name="users",
    credential=credential
)

table_client.create_table()

entity = {
    "PartitionKey": "A",
    "RowKey": "1001",
    "Name": "Ada",
    "Role": "Admin"
}

table_client.create_entity(entity)

read = table_client.get_entity(partition_key="A", row_key="1001")
read["Role"] = "Reader"
table_client.update_entity(mode="replace", entity=read)

table_client.delete_entity(partition_key="A", row_key="1001")
```

Table Storage notes:

- Use `AddEntityAsync` / `create_entity` for create.
- Use `GetEntityAsync` / `get_entity` for point reads.
- Use `UpdateEntityAsync` / `update_entity` for replace or merge semantics.
- Use `DeleteEntityAsync` / `delete_entity` for removal.
- Table Storage supports **merge** and **replace** update modes; merge changes only specified properties, while replace overwrites the entity.

**Cosmos DB CRUD**

Azure Cosmos DB is a globally distributed NoSQL database. For AZ-204, focus on point operations, partition keys, and the difference between create, upsert, replace, patch, and query.

A typical flow is:

- Create a `CosmosClient`.
- Get a database and container.
- Use point operations when you know the item ID and partition key.
- Use queries only when you need filtering across items.

```csharp
using Microsoft.Azure.Cosmos;

var client = new CosmosClient("https://mycosmos.documents.azure.com:443/", "<key>");
Database database = await client.CreateDatabaseIfNotExistsAsync("appdb");
Container container = await database.CreateContainerIfNotExistsAsync(
    id: "users",
    partitionKeyPath: "/department");

var user = new
{
    id = "1001",
    department = "A",
    name = "Ada"
};

await container.CreateItemAsync(user, new PartitionKey("A"));

ItemResponse<dynamic> read = await container.ReadItemAsync<dynamic>("1001", new PartitionKey("A"));
dynamic updated = read.Resource;
updated.name = "Ada Lovelace";

await container.ReplaceItemAsync(updated, "1001", new PartitionKey("A"));

await container.DeleteItemAsync<dynamic>("1001", new PartitionKey("A"));
```

```python
from azure.cosmos import CosmosClient

client = CosmosClient("https://mycosmos.documents.azure.com:443/", credential="<key>")
database = client.create_database_if_not_exists(id="appdb")
container = database.create_container_if_not_exists(
    id="users",
    partition_key="/department"
)

user = {
    "id": "1001",
    "department": "A",
    "name": "Ada"
}

container.create_item(body=user)

read = container.read_item(item="1001", partition_key="A")
read["name"] = "Ada Lovelace"

container.replace_item(item="1001", body=read)

container.delete_item(item="1001", partition_key="A")
```

Cosmos DB exam-relevant details:

- Use the correct **partition key** for point operations.
- `CreateItem` inserts a new item and fails if the item already exists.
- `UpsertItem` creates the item if it does not exist, or replaces it if it does.
- `ReplaceItem` overwrites the full item.
- `PatchItem` updates selected fields and is often more efficient than replace.
- Query operations are not the same as point reads; they cost more and are used when you do not know the exact item key.

Patch example:

```csharp
await container.PatchItemAsync<dynamic>(
    id: "1001",
    partitionKey: new PartitionKey("A"),
    patchOperations: new[]
    {
        PatchOperation.Set("/name", "Ada Lovelace"),
        PatchOperation.Add("/loginCount", 1)
    });
```

```python
container.patch_item(
    item="1001",
    partition_key="A",
    patch_operations=[
        {"op": "set", "path": "/name", "value": "Ada Lovelace"},
        {"op": "incr", "path": "/loginCount", "value": 1}
    ]
)
```

Use `Patch` when you want to change a few fields without sending the whole document. Use `Replace` when you want the stored item to exactly match your in-memory object.

**Key Vault Secrets CRUD**

Key Vault Secrets stores sensitive values such as API keys, connection strings, and certificates-related secrets. For modern applications, prefer Microsoft Entra ID and role-based access control (**RBAC**) where possible. Access policies still exist in some vaults, but RBAC is the recommended authorization model for new designs.

```csharp
using Azure.Identity;
using Azure.Security.KeyVault.Secrets;

var secretClient = new SecretClient(
    new Uri("https://myvault.vault.azure.net/"),
    new DefaultAzureCredential());

await secretClient.SetSecretAsync("DbPassword", "P@ssw0rd!");
KeyVaultSecret secret = await secretClient.GetSecretAsync("DbPassword");

await secretClient.StartDeleteSecretAsync("DbPassword");
```

```python
from azure.identity import DefaultAzureCredential
from azure.keyvault.secrets import SecretClient

secret_client = SecretClient(
    vault_url="https://myvault.vault.azure.net/",
    credential=credential
)

secret_client.set_secret("DbPassword", "P@ssw0rd!")
secret = secret_client.get_secret("DbPassword")

poller = secret_client.begin_delete_secret("DbPassword")
deleted_secret = poller.result()
```

Key Vault notes:

- `set_secret` / `SetSecretAsync` creates a **new version** every time; there is no in-place update of an existing secret version.
- Reading a secret without a version returns the latest enabled version.
- Deletion is typically a **soft delete** operation, and the delete workflow may be asynchronous.
- Purge is separate from delete when soft delete and purge protection are enabled.
- Use RBAC when the vault is configured for it; use access policies only when required by the vault’s authorization model.

**.NET vs Python SDK patterns**

The two languages share the same service concepts, but the code style differs.

| Area | .NET | Python |
|---|---|---|
| Async model | `async` / `await` is standard | Sync and async variants exist; many examples use sync calls |
| Client creation | Strongly typed constructors with `Uri` | Keyword arguments are common, but signatures vary by package |
| Response handling | Often returns response wrappers like `Response<T>` | Often returns model objects directly |
| Streams | Common for blob upload/download | Common for blob upload/download |
| Exceptions | `RequestFailedException` | Service-specific exceptions from `azure.core.exceptions` |

Practical guidance:

- In .NET, prefer async methods such as `UploadAsync`, `GetPropertiesAsync`, and `DeleteIfExistsAsync`.
- In Python, check whether the package exposes sync or async clients before writing code.
- Do not assume method names are identical across services or versions.
- When in doubt, verify the exact SDK version and its documentation before memorizing a signature.

**End-to-end CRUD flow examples**

A good way to study is to trace one resource through its full lifecycle.

Blob example flow:

1. Create a container if needed.
2. Upload a blob.
3. Read properties or content.
4. Update metadata or overwrite content.
5. Delete the blob.

Cosmos DB example flow:

1. Create database and container.
2. Insert an item with a partition key.
3. Read the item by ID and partition key.
4. Replace or patch the item.
5. Delete the item.

Table Storage example flow:

1. Create the table.
2. Insert an entity with `PartitionKey` and `RowKey`.
3. Read the entity by key.
4. Merge or replace properties.
5. Delete the entity.

Queue Storage example flow:

1. Create the queue.
2. Send a message.
3. Receive the message.
4. Update visibility or content if needed.
5. Delete the message after successful processing.

**Common exam pitfalls**

- Confusing **data-plane** SDKs with management-plane SDKs.
- Using the wrong partition key in Cosmos DB point operations.
- Assuming every update method performs a partial update.
- Forgetting that Key Vault secret set operations create new versions.
- Treating queue message updates like entity updates in a database.
- Hardcoding shared keys when a token credential is more appropriate.
- Copying method names across languages without checking the package version.

**Practical CLI and environment setup**

Use the Azure CLI to sign in and support local development with `DefaultAzureCredential`.

```bash
az login
az account show
az account set --subscription "<subscription-id>"
```

For local environment variables, set credentials only when needed for development:

```bash
export AZURE_TENANT_ID="<tenant-id>"
export AZURE_CLIENT_ID="<client-id>"
export AZURE_CLIENT_SECRET="<client-secret>"
```

In production, prefer managed identity where possible:

- Assign the identity to the app service, function, VM, or container.
- Grant the identity the minimum required data-plane role.
- Let `DefaultAzureCredential` pick up the managed identity automatically.

**Quick reference**

- **Blob Storage**: upload, download, metadata, delete.
- **Queue Storage**: send, receive, update visibility, delete message.
- **Table Storage**: add, get, update, delete entity.
- **Cosmos DB**: create, read, replace, patch, delete, upsert.
- **Key Vault Secrets**: set, get, list, delete, purge.

**Study takeaway**

For AZ-204, focus on the resource lifecycle, the client hierarchy, and the differences between services. The SDKs are consistent in design, but the exact method names and parameters are service-specific. If you understand how to create a client, target a resource, and perform safe CRUD with the right credential and concurrency model, you can adapt to most Azure SDK questions.
