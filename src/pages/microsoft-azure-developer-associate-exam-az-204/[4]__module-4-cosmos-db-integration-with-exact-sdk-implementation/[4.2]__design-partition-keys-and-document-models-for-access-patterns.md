## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Why partitioning and document shape matter**

In Azure Cosmos DB, your data model is part of your performance model. The way you choose a **partition key** and shape documents determines how evenly data spreads, how expensive queries become, and whether your application can scale without hot partitions.

A good design aligns with the way your application reads and writes data. You want to minimize cross-partition work, keep related data together when it is frequently accessed together, and avoid partition keys that create skew or exceed the logical partition limit for a single key value.

**Core Cosmos DB concepts**

Cosmos DB stores JSON documents in containers. A container is partitioned, and each item belongs to exactly one logical partition based on the partition key value.

- **Logical partition**: all items with the same partition key value.
- **Physical partition**: an internal storage and throughput unit managed by Cosmos DB.
- **Partition key path**: the JSON property path used to determine the partition key, such as `/tenantId`.
- **Item**: a single JSON document in a container.
- **Request Unit (RU)**: the normalized measure of throughput cost for reads, writes, and queries.

A partition key is not just a storage choice. It is the main routing mechanism for reads, writes, and queries. If you know the partition key value, Cosmos DB can often target a single logical partition and avoid scanning the entire container.

**How to choose a partition key**

Choose a partition key that matches your most common access patterns and distributes load evenly. The best key usually has high cardinality, is stable, and appears in most queries.

- **High cardinality** means many distinct values, which helps spread data and throughput.
- **Stable** means the value does not change often after item creation.
- **Frequently queried** means you can use the key to perform point reads or targeted queries.
- **Evenly distributed** means no single value receives disproportionate traffic.

Good candidates often include:
- `tenantId` in multi-tenant applications
- `userId` for user-centric workloads
- `deviceId` for telemetry or IoT data
- `orderId` when each order is accessed independently

Poor candidates often include:
- Boolean values such as `isActive`
- Low-cardinality fields such as `country` in a global app
- Frequently changing fields such as `status`
- Sequential values that can create hot spots if they concentrate writes

> A single partition key value must not grow beyond the logical partition limit. In exam scenarios, treat this as a design constraint that affects how much data and traffic you can place under one key value, rather than assuming one universal fixed number for every service capability.

**Access-pattern-first design**

Start with the queries and operations your app performs most often, then design the document model around them. Cosmos DB is optimized when you can answer a request with a point read or a query scoped to one partition.

Typical access patterns include:
- Get one item by ID
- List all orders for a customer
- Read the latest telemetry for a device
- Update a user profile and related preferences together
- Search by status across many tenants

For each pattern, ask:
- What is the partition key value?
- Is the request a point read or a query?
- Does the query stay within one partition?
- Do I need to denormalize data to avoid joins or fan-out queries?

**Document modeling strategies**

Cosmos DB uses JSON documents, so you often model data differently than in a relational database. The goal is to shape documents around read efficiency and partition locality.

**1. Embed related data when it is read together**

Use embedded arrays or nested objects when child data is usually retrieved with the parent.

Example:
```json
{
  "id": "order-1001",
  "customerId": "cust-42",
  "orderDate": "2026-04-26",
  "status": "Pending",
  "shippingAddress": {
    "line1": "1 Main St",
    "city": "Seattle",
    "region": "WA"
  },
  "items": [
    { "sku": "A100", "name": "Keyboard", "qty": 1, "price": 49.99 },
    { "sku": "B200", "name": "Mouse", "qty": 2, "price": 19.99 }
  ]
}
```

Use embedding when:
- The child data is small and bounded.
- You usually read the parent and children together.
- You want to avoid extra queries.

Avoid embedding when:
- The child collection can grow without bound.
- The child data is updated independently and frequently.
- The document could become too large or too expensive to rewrite.

**2. Denormalize for read performance**

Cosmos DB often favors duplication over joins across containers. If a query needs data from multiple entities, consider storing a read-optimized copy of the needed fields in the same document or partition.

For example, an order document may include:
- Customer name
- Customer tier
- Shipping city
- Product summary fields

This reduces query complexity and RU consumption, but you must accept that duplicated fields can become stale unless you update them consistently.

**3. Use separate documents for independently changing entities**

If a child entity changes often or has its own lifecycle, store it separately.

Examples:
- User profile vs. user activity events
- Product catalog vs. inventory updates
- Order header vs. large audit trail

This keeps writes smaller and avoids rewriting large parent documents for minor updates.

**4. Model for bounded growth**

A document should not grow indefinitely. Large arrays can become a problem if they are append-heavy or frequently rewritten.

Common patterns:
- Split event streams into separate documents by time window.
- Store comments or logs as separate items keyed by parent ID.
- Use a summary document plus detail documents.

**Partition key patterns**

**Single-tenant or user-centric workloads**

If most operations are scoped to a user, `userId` is often a strong partition key.

Benefits:
- Point reads by user become efficient.
- User-related data stays together.
- Queries for one user avoid cross-partition scans.

Risks:
- Very active users can create hot partitions.
- A single user with too much data can hit logical partition limits.

**Multi-tenant workloads**

For SaaS applications, `tenantId` is often the best partition key because it isolates tenant data and supports tenant-scoped queries.

Benefits:
- Tenant data is naturally grouped.
- Security and data isolation are easier to reason about.
- Most tenant operations can stay within one partition.

Risks:
- Large tenants may dominate throughput.
- You may need a synthetic key or hierarchical strategy if one tenant is much larger than others.

**Synthetic partition keys**

When no single natural field distributes data well, create a synthetic key by combining fields.

Examples:
- `tenantId-userId`
- `customerId-orderType`
- `deviceId-yyyyMM`

This can improve distribution and query targeting, but it can also make queries harder if your app often needs only one component of the key.

Example document:
```json
{
  "id": "evt-9001",
  "tenantId": "contoso",
  "userId": "u-123",
  "pk": "contoso|u-123",
  "eventType": "login",
  "timestamp": "2026-04-26T10:15:00Z"
}
```

**Hierarchical partition keys**

Cosmos DB supports hierarchical partition key scenarios in newer capabilities. These allow you to model partitioning with multiple components, which can help when one dimension alone is not enough to distribute data or target queries efficiently.

Use hierarchical approaches when:
- You need better distribution than a single key provides.
- Your access patterns naturally filter by multiple dimensions.
- You want to reduce hot partitions in large multi-tenant systems.

Design still matters:
- Put the most selective and commonly filtered components first when it helps routing.
- Keep the key stable.
- Ensure the combined key still matches your query patterns.

**Point reads vs. queries**

A **point read** is the cheapest and fastest way to retrieve a single item. It requires the item `id` and the full partition key value.

Example:
```csharp
ItemResponse<Order> response = await container.ReadItemAsync<Order>(
    id: "order-1001",
    partitionKey: new PartitionKey("cust-42"));
```

Use point reads when:
- You know the exact item ID.
- You know the partition key.
- You want the lowest latency and RU cost.

A **query** is more flexible but usually costs more than a point read.

Example:
```csharp
var query = new QueryDefinition(
    "SELECT c.id, c.orderDate, c.total FROM c WHERE c.customerId = @customerId")
    .WithParameter("@customerId", "cust-42");

using FeedIterator<OrderSummary> iterator =
    container.GetItemQueryIterator<OrderSummary>(
        query,
        requestOptions: new QueryRequestOptions
        {
            PartitionKey = new PartitionKey("cust-42")
        });
```

Use queries when:
- You need filtering, sorting, or projection.
- You do not know the exact item ID.
- You can still scope the query to one partition.

> Make sure your projected JSON fields match your C# model property names or use explicit serialization attributes. For example, `id` often maps to `Id` in .NET models depending on serializer settings.

**JOIN behavior in Cosmos DB SQL API**

Cosmos DB SQL API supports `JOIN` within a single item/document scope, not relational joins across containers.

Use intra-document `JOIN` when:
- You need to flatten arrays inside one document.
- You want to query nested collections within the same item.

Example:
```sql
SELECT c.id, i.sku, i.qty
FROM c
JOIN i IN c.items
WHERE c.customerId = @customerId
```

Do not expect:
- Cross-container relational joins
- Server-side joins across unrelated partition keys like a relational database engine

If you need cross-entity data, you usually:
- Denormalize
- Query multiple containers in application code
- Use change feed or ETL to build read models

**Indexing and query cost**

Cosmos DB indexes JSON properties by default unless you exclude paths. Good indexing supports flexible queries, but unnecessary indexing increases write cost.

Use indexing policy tuning when:
- You have large properties that are rarely queried.
- You want to reduce write RU by excluding paths.
- You need to optimize a known query shape.

Example indexing policy snippet:
```json
{
  "indexingMode": "consistent",
  "automatic": true,
  "includedPaths": [
    { "path": "/*" }
  ],
  "excludedPaths": [
    { "path": "/largeText/?" }
  ]
}
```

This excludes a specific property path from indexing. Use it carefully:
- Exclude only fields you do not query or filter on.
- Confirm the JSON shape matches the path.
- Remember that excluding a path affects queryability and may not be appropriate for fields used in filters or sorts.

**Partitioning and throughput**

Throughput in Cosmos DB is consumed by reads, writes, and queries. Poor partition design can create hot partitions where one logical partition receives disproportionate traffic.

Symptoms of a bad design:
- High RU consumption on one key value
- Throttling even when total throughput seems sufficient
- Slow queries that scan many partitions
- Uneven storage growth

Mitigations:
- Choose a higher-cardinality partition key.
- Add a synthetic suffix to spread writes.
- Split large logical entities into multiple items.
- Revisit access patterns and denormalize read models.

**Practical design workflow**

1. Identify the top 5 read and write operations.
2. Determine which operations need single-item access.
3. Find the most common filter used in those operations.
4. Choose a partition key that supports the most important scoped queries.
5. Model documents so the common read path needs minimal joins or fan-out.
6. Validate that no single key value will accumulate too much data or traffic.
7. Test RU usage with realistic data volumes and query shapes.

**Example design scenarios**

**Scenario: Order management**

If users mostly read orders by customer:
- Partition key: `/customerId`
- Document shape: order header plus line items
- Query pattern: list orders for one customer
- Benefit: customer-scoped queries stay within one partition

If order line items are huge or updated independently:
- Store order header and line items separately
- Keep `customerId` on both items
- Use the same partition key to preserve locality

**Scenario: IoT telemetry**

If devices send frequent telemetry:
- Partition key: `/deviceId`
- Document shape: one event per item
- Optional synthetic key: `deviceId|yyyyMM` for very high-volume devices
- Benefit: writes distribute across devices and queries can target one device

**Scenario: SaaS tenant data**

If each tenant has many users:
- Partition key: `/tenantId`
- Add `userId` to documents for secondary filtering
- Denormalize tenant metadata into user-facing documents when needed
- Consider hierarchical partitioning or synthetic keys if one tenant is much larger than others

**CLI and provisioning examples**

Use Azure CLI to create a Cosmos DB SQL container with a partition key. Exact autoscale syntax can vary by CLI version and resource scope, so verify the current command help before using it in production.

Create a database and container with manual throughput:
```bash
az cosmosdb sql database create \
  --account-name mycosmosacct \
  --resource-group rg-app \
  --name appdb

az cosmosdb sql container create \
  --account-name mycosmosacct \
  --resource-group rg-app \
  --database-name appdb \
  --name orders \
  --partition-key-path "/customerId" \
  --throughput 400
```

Inspect the command help before configuring autoscale:
```bash
az cosmosdb sql container create -h
az cosmosdb sql database create -h
```

If you use autoscale, confirm the supported parameter names for your CLI version and whether throughput is configured at the database or container level in your deployment model.

**Common exam traps**

- Confusing a partition key with an item ID.
- Choosing a low-cardinality field because it is easy to query.
- Assuming a query is cheap just because it filters on an indexed field.
- Expecting relational joins across containers.
- Ignoring the cost of rewriting large documents.
- Designing for one query while breaking the top write path.
- Forgetting that point reads require both `id` and partition key value.

**Quick comparison**

| Design choice | Best when | Tradeoff |
|---|---|---|
| Embed data | Data is read together | Larger documents, more rewrite cost |
| Separate documents | Data changes independently | More queries or application joins |
| Natural partition key | One field matches access patterns | May skew if values are uneven |
| Synthetic key | Need better distribution | More complex queries |
| Exclude index paths | Large fields are rarely queried | Reduced query flexibility |

**Key takeaways**

- Design from access patterns first, not from entity diagrams alone.
- Pick a partition key that is stable, high-cardinality, and aligned with your most common queries.
- Keep related data together when it is usually read together.
- Use denormalization to reduce cross-partition work.
- Remember that Cosmos DB SQL API supports `JOIN` within a document, not relational joins across containers.
- Validate partition size, traffic distribution, and RU cost with realistic workloads.
