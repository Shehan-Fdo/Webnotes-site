## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Overview**

These patterns show up across Azure data services and application code: you need to query safely, return results in manageable chunks, write many records efficiently, and avoid overwriting someone else’s changes. In practice, you often combine them in the same workflow, especially when building APIs over Azure SQL, Cosmos DB, or storage-backed systems.

The core goals are:

- Reduce injection risk with **parameterized queries**
- Control response size with **pagination**
- Improve throughput with **bulk operations**
- Prevent lost updates with **optimistic concurrency**

**Parameterized queries**

A parameterized query separates the SQL text from the values you supply. This matters because the database treats parameters as data, not executable code. That is the primary defense against SQL injection when you build queries from user input.

Use parameters whenever values come from request bodies, route values, query strings, or external systems. Do not concatenate raw input into SQL.

- **Safer execution**: user input cannot change the query structure
- **Better reuse**: the database can often reuse execution plans
- **Cleaner code**: values stay typed and explicit

A typical ADO.NET example uses `SqlCommand` parameters:

```csharp
using var connection = new SqlConnection(connectionString);
using var command = new SqlCommand(
    "SELECT Id, DisplayName FROM Users WHERE Email = @Email AND IsActive = @IsActive",
    connection);

command.Parameters.AddWithValue("@Email", email);
command.Parameters.AddWithValue("@IsActive", true);

await connection.OpenAsync();
using var reader = await command.ExecuteReaderAsync();
```

Prefer strongly typed parameter APIs when possible. `AddWithValue` is convenient, but it can infer an unexpected type or length and sometimes hurt performance. For production code, specify the type explicitly:

```csharp
command.Parameters.Add("@Email", SqlDbType.NVarChar, 256).Value = email;
command.Parameters.Add("@IsActive", SqlDbType.Bit).Value = true;
```

For Entity Framework Core, parameterization is usually automatic when you use LINQ or interpolated APIs that preserve parameters:

```csharp
var user = await db.Users
    .FirstOrDefaultAsync(u => u.Email == email);
```

If you must use raw SQL, keep it parameterized:

```csharp
var users = await db.Users
    .FromSqlInterpolated($"SELECT * FROM Users WHERE Email = {email}")
    .ToListAsync();
```

> Avoid building SQL like `"... WHERE Email = '" + email + "'"`. That pattern is vulnerable to injection and is difficult to maintain.

**Pagination**

Pagination limits how many records you return in one response. This improves latency, reduces memory pressure, and prevents large payloads from overwhelming clients or APIs. It is essential for list endpoints, search results, and any dataset that can grow over time.

There are two common approaches:

- **Offset pagination**: use `OFFSET` and `FETCH` or `Skip` and `Take`
- **Keyset pagination**: use a stable sort key and request the next page after the last item seen

Offset pagination is easy to implement and works well for moderate datasets:

```sql
SELECT Id, DisplayName, CreatedUtc
FROM Users
ORDER BY CreatedUtc DESC, Id DESC
OFFSET @Offset ROWS FETCH NEXT @PageSize ROWS ONLY;
```

In C# with EF Core:

```csharp
var page = await db.Users
    .OrderByDescending(u => u.CreatedUtc)
    .ThenByDescending(u => u.Id)
    .Skip((pageNumber - 1) * pageSize)
    .Take(pageSize)
    .ToListAsync();
```

Offset pagination is simple, but it can become inefficient on large tables because the database still has to walk past earlier rows. It can also produce duplicates or gaps if rows are inserted or deleted between requests.

Keyset pagination is often better for large or frequently changing datasets. You use the last seen sort key as the cursor:

```sql
SELECT TOP (@PageSize) Id, DisplayName, CreatedUtc
FROM Users
WHERE (CreatedUtc < @LastCreatedUtc)
   OR (CreatedUtc = @LastCreatedUtc AND Id < @LastId)
ORDER BY CreatedUtc DESC, Id DESC;
```

This approach is more stable and usually faster because it can seek into an index. Use it when you need consistent navigation through large result sets, such as activity feeds or audit logs.

A practical API response often includes a continuation token or cursor:

```json
{
  "items": [
    { "id": 101, "displayName": "Ava" },
    { "id": 100, "displayName": "Noah" }
  ],
  "nextCursor": "2026-04-26T10:15:00Z|100"
}
```

When designing pagination:

- Always define a deterministic `ORDER BY`
- Return a stable sort key with the page
- Validate `pageSize` to prevent abuse
- Prefer cursors for large, append-heavy datasets
- Document whether pages are snapshot-like or live

**Bulk operations**

Bulk operations reduce round trips and improve throughput when you need to insert, update, or delete many rows. Instead of sending one command per row, you batch work so the database can process it more efficiently.

Common scenarios include:

- Importing CSV or JSON data
- Syncing records from an external system
- Seeding reference data
- Processing queue messages in batches

For SQL Server, `SqlBulkCopy` is a common high-performance insert option:

```csharp
using var bulkCopy = new SqlBulkCopy(connectionString)
{
    DestinationTableName = "dbo.Users",
    BatchSize = 5000,
    BulkCopyTimeout = 60
};

bulkCopy.ColumnMappings.Add("Id", "Id");
bulkCopy.ColumnMappings.Add("DisplayName", "DisplayName");
bulkCopy.ColumnMappings.Add("Email", "Email");

await bulkCopy.WriteToServerAsync(dataReader);
```

`SqlBulkCopy` is optimized for large inserts, but it is not a general-purpose ORM feature. Use it when raw throughput matters more than per-row business logic.

For smaller batches, a table-valued parameter or batched `INSERT` statements can be appropriate. In EF Core, `AddRange` plus a single `SaveChangesAsync()` is better than calling `SaveChangesAsync()` inside a loop:

```csharp
db.Users.AddRange(users);
await db.SaveChangesAsync();
```

For Cosmos DB, bulk support depends on the SDK and configuration. In general, you improve throughput by batching operations where the service supports it and by grouping items by partition key. Partition-aware batching reduces cross-partition overhead.

When using bulk operations:

- Keep transactions and batch sizes reasonable
- Watch for lock escalation and log growth
- Validate data before sending it in bulk
- Handle partial failures explicitly
- Measure throughput and latency under realistic load

> Bulk writes can increase pressure on the database log, locks, and network. Bigger batches are not always better.

**Optimistic concurrency**

Optimistic concurrency assumes conflicts are rare. You read a record, make changes, and then save only if nobody else changed the record in the meantime. If the record changed, the save fails and you decide how to resolve the conflict.

This pattern is common in web apps because many users can read the same data, but only a few will update the same row at the same time.

The usual mechanism is a concurrency token:

- In SQL Server, a `rowversion` column is a common choice
- In EF Core, you can mark a property as a concurrency token
- In Cosmos DB, the `_etag` value is used for conditional updates

Example SQL Server table:

```sql
CREATE TABLE Users (
    Id INT PRIMARY KEY,
    DisplayName NVARCHAR(100) NOT NULL,
    Email NVARCHAR(256) NOT NULL,
    RowVersion ROWVERSION NOT NULL
);
```

In EF Core:

```csharp
public class User
{
    public int Id { get; set; }
    public string DisplayName { get; set; } = "";
    public string Email { get; set; } = "";
    [Timestamp]
    public byte[] RowVersion { get; set; } = Array.Empty<byte>();
}
```

When you save changes, EF Core includes the original token in the `WHERE` clause. If the row changed, zero rows are affected and EF Core throws `DbUpdateConcurrencyException`.

```csharp
try
{
    await db.SaveChangesAsync();
}
catch (DbUpdateConcurrencyException ex)
{
    // Reload, merge, or ask the user to retry
}
```

In Cosmos DB, optimistic concurrency often uses `If-Match` with the current ETag:

```csharp
var item = await container.ReadItemAsync<MyDoc>(id, new PartitionKey(pk));
item.Resource.Status = "Processed";

await container.ReplaceItemAsync(
    item.Resource,
    id,
    new PartitionKey(pk),
    new ItemRequestOptions { IfMatchEtag = item.ETag });
```

If the ETag no longer matches, the service rejects the update with a precondition failure. That protects you from overwriting a newer version.

Use optimistic concurrency when:

- Multiple clients can edit the same record
- You want high read throughput without locking readers
- Conflicts are possible but not constant

Handle conflicts by:

1. Reloading the latest version
2. Comparing user changes with current data
3. Merging automatically when safe
4. Prompting the user when manual resolution is needed

**Putting the patterns together**

These techniques often work as a pipeline. For example, an API might:

1. Accept a filter and page cursor
2. Run a parameterized query
3. Return a page of results
4. Let the client edit one item
5. Save the update with a concurrency token
6. Process many queued updates in bulk

That design gives you safer input handling, predictable response sizes, better write performance, and fewer lost updates.

A practical endpoint might look like this:

```csharp
[HttpGet("users")]
public async Task<IActionResult> GetUsers([FromQuery] string? email, [FromQuery] int pageSize = 25)
{
    pageSize = Math.Clamp(pageSize, 1, 100);

    var query = db.Users.AsQueryable();

    if (!string.IsNullOrWhiteSpace(email))
    {
        query = query.Where(u => u.Email == email);
    }

    var items = await query
        .OrderByDescending(u => u.CreatedUtc)
        .ThenByDescending(u => u.Id)
        .Take(pageSize)
        .ToListAsync();

    return Ok(items);
}
```

This example shows the main ideas:

- Input is filtered through typed query composition
- Results are limited
- Ordering is deterministic
- The query remains parameterized

**Exam and implementation tips**

- Use **parameterized queries** for any user-supplied value
- Use **pagination** for list endpoints and large datasets
- Use **bulk operations** when many rows must be processed efficiently
- Use **optimistic concurrency** to prevent lost updates
- Choose the right pattern for the data store and access pattern
- Prefer service-native features when available, such as `rowversion`, ETags, or bulk APIs

**Common pitfalls**

- Concatenating SQL strings from user input
- Returning unbounded result sets
- Using offset pagination on very large, volatile tables without considering performance
- Saving each row individually in a loop
- Ignoring concurrency exceptions and silently overwriting data
- Forgetting to order results before paging

**Quick reference**

- **Safe query input**: parameters, not string concatenation
- **Small pages**: lower latency and memory use
- **Large imports**: batch or bulk APIs
- **Conflict detection**: rowversion, ETag, or concurrency tokens
- **Conflict handling**: reload, merge, retry, or prompt
