## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Change feed overview**

Azure Cosmos DB change feed is an ordered stream of changes made to items in a container. You use it to build event-driven systems that react to inserts and updates without polling the database. It is especially useful when you need to project data into another store, trigger downstream workflows, or integrate with messaging services.

The change feed is not a general-purpose event bus. It is a durable read model of committed writes in a container, and consumers must manage their own processing state through leases and checkpoints. In exam scenarios, think of it as the right choice when the source of truth is Cosmos DB and you want to react to data changes reliably.

- **Typical uses**
  - Synchronize data to Azure Search.
  - Publish messages to Azure Service Bus.
  - Trigger Azure Functions for workflow automation.
  - Build materialized views or read-optimized projections.

- **Core idea**
  - You read changes after they are committed to the container.
  - You process them independently from the write path.
  - You track progress so processing can resume after failures or scale-out.

**Change feed behavior and guarantees**

The change feed preserves order within a logical partition, but not across partitions. This matters when your processing logic depends on sequence. If you need strict ordering for a business entity, design that entity to stay within one partition key value.

The feed exposes changes in a way that supports incremental processing. Consumers typically read from the beginning of the feed or from the current point in time, depending on whether they need historical backfill or only new changes.

- **Ordering**
  - Guaranteed within a logical partition.
  - Not guaranteed across different partitions.
  - Design partition keys around the ordering boundary you care about.

- **Starting position**
  - Start from the beginning for backfill or reprocessing.
  - Start from now for new event processing only.
  - Use the chosen start position consistently across instances.

- **Change types**
  - Inserts and updates appear as item changes.
  - Delete visibility depends on the change feed mode and SDK/API capabilities.
  - In latest-version mode, you usually see the current item state, not a full history of intermediate versions.

> <mark>Important:</mark> The exact delete behavior depends on the change feed mode and account/API capabilities. For exam prep, remember that delete events are not always available in the basic latest-version experience.

**Change feed modes**

Azure Cosmos DB supports different change feed experiences. The common exam focus is the latest-version feed, which gives you the latest state of changed items. Newer capabilities can expose all versions and deletes for scenarios that need full history.

- **Latest-version mode**
  - Returns the latest state of each changed item.
  - Best for projections, sync, and downstream processing.
  - Does not provide a full historical event stream.

- **All-versions-and-deletes mode**
  - Exposes a richer history of changes.
  - Useful when you need audit-like processing or delete awareness.
  - Requires support from the account, API, and SDK version.

- **When to choose which**
  - Use latest-version mode for most event-driven integration patterns.
  - Use all-versions-and-deletes mode when consumers must react to deletions or reconstruct change history.

**Lease containers and load balancing**

A lease container stores processing state for the change feed processor. Leases let multiple instances coordinate work, balance partitions, and recover from failures. Each lease typically corresponds to a partition range being processed.

This is one of the most important exam concepts: the lease container is what makes horizontal scaling and failover possible. Without leases, multiple instances could duplicate work or lose track of progress.

- **Lease responsibilities**
  - Track which processor owns which partition range.
  - Store checkpoint progress.
  - Enable load balancing across instances.
  - Support failover when an instance stops.

- **Operational behavior**
  - A processor acquires leases for available partitions.
  - If a processor becomes unavailable, another instance can take over.
  - Checkpoints allow the new owner to resume from the last processed position.

- **Lease container guidance**
  - Use a dedicated container for leases.
  - In SQL API accounts, the lease container is commonly created with partition key path `/id`.
  - Keep the lease container separate from application data.

```bash
az cosmosdb sql container create \
  --account-name mycosmosacct \
  --database-name appdb \
  --name leases \
  --partition-key-path "/id" \
  --throughput 400
```

**Checkpointing and reprocessing**

Checkpointing records how far a processor has read in the change feed. The processor writes checkpoints after it successfully processes a batch or at configured intervals, depending on the SDK and processor implementation. If the app crashes before a checkpoint is written, the same changes may be reprocessed after restart.

This is a tradeoff between throughput and duplicate processing. More frequent checkpoints reduce replay after failure but add overhead. Less frequent checkpoints improve throughput but increase the amount of work that may be repeated.

- **Checkpoint purpose**
  - Persist progress for each lease.
  - Prevent re-reading the entire feed after restart.
  - Support recovery and scale-out.

- **Checkpoint timing**
  - Usually after successful batch processing.
  - Can be influenced by batch size and processor settings.
  - Should happen only after downstream work succeeds.

- **Design implication**
  - Make downstream handlers idempotent.
  - Expect at-least-once processing semantics.
  - Use deduplication if duplicate delivery is harmful.

**SDK building blocks**

For custom processors, the Azure Cosmos DB SDK provides classes such as `ChangeFeedProcessorBuilder` and `ChangeFeedProcessor`. These are the core abstractions for building a background worker that reads the feed, balances leases, and invokes your handler.

The exact namespace and package version depend on the SDK generation, but the exam usually expects you to understand the role of these types rather than memorize every overload.

- **`ChangeFeedProcessorBuilder`**
  - Configures the processor.
  - Sets the monitored container and lease container.
  - Registers the delegate that handles changes.
  - Applies options such as instance name and start position.

- **`ChangeFeedProcessor`**
  - Runs the configured processor.
  - Manages lease acquisition and checkpointing behavior.
  - Starts and stops the background feed reader.

- **Common supporting concepts**
  - Monitored container: the source data container.
  - Lease container: the coordination and checkpoint container.
  - Delegate handler: your code that processes each batch.

```csharp
var processor = monitoredContainer
    .GetChangeFeedProcessorBuilder<MyItem>(
        processorName: "orders-processor",
        onChangesDelegate: async (IReadOnlyCollection<MyItem> changes, CancellationToken ct) =>
        {
            foreach (var item in changes)
            {
                await HandleItemAsync(item, ct);
            }
        })
    .WithInstanceName(Environment.MachineName)
    .WithLeaseContainer(leaseContainer)
    .WithStartTime(DateTime.UtcNow.AddMinutes(-5))
    .Build();
```

**Custom processor implementation**

A custom processor is appropriate when you need full control over startup, shutdown, batching, and downstream integration. You often use it in worker services, container apps, or long-running background services.

Your handler should process a batch, honor cancellation, and only checkpoint after successful work. If you publish to another system, do so in a way that tolerates retries.

```csharp
public async Task HandleChangesAsync(
    IReadOnlyCollection<MyItem> changes,
    CancellationToken cancellationToken)
{
    foreach (var change in changes)
    {
        cancellationToken.ThrowIfCancellationRequested();

        await _serviceBusSender.SendMessageAsync(
            new ServiceBusMessage(JsonSerializer.Serialize(change)),
            cancellationToken);
    }
}
```

A safe shutdown pattern is to stop the processor in the host lifecycle rather than trying to await inside a cancellation registration callback.

```csharp
public async Task StopAsync(CancellationToken cancellationToken)
{
    if (_processor != null)
    {
        await _processor.StopAsync();
    }
}
```

- **Implementation tips**
  - Keep handlers idempotent.
  - Pass `CancellationToken` to awaited operations.
  - Avoid long blocking work inside the feed callback.
  - Separate transformation from transport when possible.

**Azure Functions Cosmos DB trigger**

Azure Functions is a common serverless way to consume the change feed. The trigger simplifies lease management and checkpointing, which makes it a strong exam answer when the scenario asks for minimal infrastructure.

The exact attribute and package names vary by Functions model and extension version, so focus on the binding concepts: monitored container, lease container, connection setting, and optional tuning parameters.

```csharp
using Microsoft.Azure.Functions.Worker;
using Microsoft.Extensions.Logging;

public class OrderChangeFunction
{
    private readonly ILogger<OrderChangeFunction> _logger;

    public OrderChangeFunction(ILogger<OrderChangeFunction> logger)
    {
        _logger = logger;
    }

    [Function("OrderChangeFunction")]
    public void Run(
        [CosmosDBTrigger(
            databaseName: "appdb",
            containerName: "orders",
            Connection = "CosmosConnection",
            LeaseContainerName = "leases",
            CreateLeaseContainerIfNotExists = true)]
        IReadOnlyList<MyItem> input,
        FunctionContext context)
    {
        _logger.LogInformation("Received {Count} changes", input.Count);
    }
}
```

- **Trigger settings to know**
  - `databaseName`: source database.
  - `containerName`: monitored container.
  - `Connection`: app setting with the Cosmos DB connection string or identity-based configuration.
  - `LeaseContainerName`: dedicated lease container.
  - `CreateLeaseContainerIfNotExists`: convenience for development and simple deployments.

- **Tuning options**
  - Batch size affects how many items are delivered per invocation.
  - Lease settings affect load distribution and failover behavior.
  - Start position controls whether the function reads historical changes or only new ones.

> <mark>Important:</mark> For production, verify the exact extension version and attribute namespace used by your Functions runtime. The binding model differs between in-process and isolated worker apps.

**Starting from beginning vs now**

A common implementation choice is whether to process historical changes or only new writes. This affects backfill, migration, and replay scenarios.

- **Start from beginning**
  - Use when you need to rebuild a projection.
  - Useful for initial indexing or migration.
  - Can generate a large backlog.

- **Start from now**
  - Use when you only care about future changes.
  - Good for live event processing.
  - Reduces initial catch-up time.

- **Practical guidance**
  - Use a controlled backfill job for historical data.
  - Switch to live processing after the backlog is drained.
  - Keep the start strategy explicit in deployment scripts.

**Event-driven integration patterns**

The change feed is often the first step in a larger pipeline. You read from Cosmos DB, transform the data, and push it to another service that better fits the downstream workload.

- **Cosmos DB to Service Bus**
  - Use when you need reliable decoupling and downstream retries.
  - Good for order processing, notifications, and workflow orchestration.
  - The change feed consumer becomes the producer of messages.

- **Cosmos DB to Azure Functions**
  - Use when you want serverless reaction to data changes.
  - Good for lightweight enrichment or fan-out.
  - Functions can call other services or update secondary stores.

- **Cosmos DB to Azure AI Search**
  - Use when you need searchable projections.
  - Good for catalog indexing and document search.
  - The consumer maps Cosmos documents to search documents.

```text
Cosmos DB container
        |
        v
Change feed processor / Azure Function
        |
        +--> Azure Service Bus queue
        +--> Azure AI Search index
        +--> Notification or workflow service
```

- **Why this pattern works**
  - The database remains the source of truth.
  - Downstream systems can scale independently.
  - Failures in one consumer do not block writes to Cosmos DB.

**Deletes, updates, and projection design**

Delete handling is a frequent exam trap. In the basic latest-version feed, you typically receive the latest state of changed items, which is enough for many projection scenarios but not for full audit trails. If your downstream system must remove documents when source items are deleted, make sure your chosen feed mode exposes delete information or use a separate tombstone strategy.

- **Projection strategies**
  - Upsert the latest item into a search index.
  - Emit a delete message when delete events are available.
  - Store a soft-delete flag if hard delete events are not available.

- **Design choices**
  - Use item IDs consistently across systems.
  - Preserve partition key information if downstream routing depends on it.
  - Make delete handling explicit in the consumer contract.

**Exam-oriented comparison**

| Scenario | Best fit | Why |
|---|---|---|
| React to Cosmos DB writes with minimal code | Azure Functions Cosmos DB trigger | Managed lease and checkpoint handling |
| Need custom batching and downstream routing | `ChangeFeedProcessor` | Full control over processing logic |
| Need search indexing from operational data | Change feed to Azure AI Search | Keeps search projection in sync |
| Need reliable decoupling and retries | Change feed to Azure Service Bus | Buffering and consumer isolation |

**Practical configuration example**

A typical app uses one container for business data and one dedicated lease container. The processor reads from the monitored container, writes checkpoints to the lease container, and forwards events to a downstream service.

```json
{
  "CosmosConnection": "AccountEndpoint=https://example.documents.azure.com:443/;AccountKey=***;",
  "DatabaseName": "appdb",
  "OrdersContainer": "orders",
  "LeasesContainer": "leases",
  "ServiceBusConnection": "Endpoint=sb://example.servicebus.windows.net/;SharedAccessKeyName=RootManageSharedAccessKey;SharedAccessKey=***"
}
```

```csharp
var processor = cosmosClient
    .GetContainer("appdb", "orders")
    .GetChangeFeedProcessorBuilder<Order>(
        "orders-processor",
        async (changes, ct) =>
        {
            foreach (var order in changes)
            {
                await serviceBusSender.SendMessageAsync(
                    new ServiceBusMessage(JsonSerializer.Serialize(order)),
                    ct);
            }
        })
    .WithInstanceName("worker-1")
    .WithLeaseContainer(cosmosClient.GetContainer("appdb", "leases"))
    .Build();

await processor.StartAsync();
```

**Common pitfalls**

- **Duplicate processing**
  - Happens when a batch is processed but not checkpointed.
  - Solve with idempotent handlers and deduplication keys.

- **Wrong partition design**
  - Causes ordering issues across related records.
  - Keep related events in the same logical partition when ordering matters.

- **Lease container misuse**
  - Sharing a lease container across unrelated processors can cause confusion.
  - Use a dedicated lease container per monitored feed or processor group.

- **Ignoring cancellation**
  - Can delay shutdown and cause partial work.
  - Pass `CancellationToken` through all async calls.

- **Assuming delete events always exist**
  - Delete visibility depends on feed mode and SDK support.
  - Verify the mode before designing delete-dependent logic.

**How to choose the right approach**

Use the change feed when Cosmos DB is your system of record and you need to react to data changes asynchronously. Choose Azure Functions for simple serverless handlers, or a custom processor when you need more control over batching, retries, and downstream orchestration.

For AZ-204, the key ideas are:

- The change feed is a durable stream of committed changes from Cosmos DB.
- Leases coordinate scale-out and failover.
- Checkpoints control replay and recovery.
- Ordering is partition-scoped, not global.
- Downstream consumers should be idempotent.
- Event-driven patterns often forward changes to Service Bus, Functions, or Search.
