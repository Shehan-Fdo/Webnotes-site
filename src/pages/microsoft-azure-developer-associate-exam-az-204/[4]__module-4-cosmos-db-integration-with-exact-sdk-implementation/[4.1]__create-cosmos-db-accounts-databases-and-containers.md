## **Course:** Microsoft Azure Developer Associate (Exam AZ-204)

**Cosmos DB resource hierarchy**

Azure Cosmos DB is a globally distributed, multi-model database service. In Azure, you typically create resources in this order:

- **Account**: the top-level Cosmos DB resource that defines API, region, consistency, networking, and replication settings.
- **Database**: a logical grouping of containers or collections.
- **Container**: the unit that stores items/documents and defines partitioning, indexing, and throughput placement.

The exact resource names vary by API, but the hierarchy is consistent. For exam purposes, distinguish between the Azure resource you deploy and the data model you use inside that resource.

**API-specific resource names**

Cosmos DB supports multiple APIs, and the resource names differ by API even when the concepts are similar.

| API | Account type | Database resource | Container resource | Notes |
|---|---|---|---|---|
| NoSQL (SQL API) | `Microsoft.DocumentDB/databaseAccounts` | Database | Container | Most common for AZ-204 examples |
| MongoDB | `MongoDB`-compatible account | Database | Collection | Uses MongoDB wire protocol semantics |
| Cassandra | Cassandra account | Keyspace | Table | Data is modeled as tables and rows |
| Gremlin | Gremlin account | Database | Graph | Graphs use vertices and edges |
| Table | Table API account | Table | Table | Uses key/value-style entities |

For exam prep, remember that the portal and ARM/Bicep resource types are Azure-specific, while the logical names exposed to your application depend on the chosen API.

**Account creation considerations**

When you create a Cosmos DB account, you choose settings that affect availability, performance, and cost.

- **API selection** determines the programming model and supported features.
- **Region placement** affects latency and replication.
- **Consistency level** controls read guarantees.
- **Networking** can be public, private endpoint-based, or restricted by firewall rules.
- **Backup mode** can be periodic or continuous, depending on account capabilities and region support.
- **Capacity mode** is either provisioned throughput or serverless for supported APIs and scenarios.

> **Important:** Serverless and provisioned throughput are different capacity modes. In serverless, you do not provision RU/s (request units per second) on databases or containers. In provisioned mode, you assign throughput at the database or container level for supported APIs.

**Throughput and scaling model**

Cosmos DB uses **RU/s** (Request Units per second) in provisioned mode to represent throughput. The important exam distinction is where throughput is assigned and what actually scales.

- **Partition key** defines the logical partitioning boundary for items in a container.
- **Physical partitions** are the internal storage and throughput units that Cosmos DB scales across.
- **Container** is the resource that holds the partition key, indexing policy, unique key policy, and optional throughput configuration.
- **Database-level shared throughput** lets multiple containers consume a common RU/s pool.
- **Container-level throughput** dedicates RU/s to a single container.

A container is not the scaling boundary by itself. The partition key determines how data is distributed, and physical partitions determine how the service scales storage and throughput behind the scenes.

**Provisioned throughput patterns**

Use the right throughput placement based on workload isolation and cost.

- **Database shared throughput**
  - Good when several small containers share a common workload.
  - Simplifies management when no single container needs dedicated RU/s.
  - Throughput is assigned to the database, not each container.
- **Container dedicated throughput**
  - Good when one container has a distinct workload or needs predictable performance.
  - Useful when you want to isolate hot data or noisy neighbors.
  - Throughput is assigned to the container.
- **Autoscale throughput**
  - Automatically adjusts RU/s within a configured maximum.
  - Useful for spiky workloads.
  - Still provisioned mode, not serverless.
- **Serverless**
  - No RU/s provisioning.
  - Best for intermittent, low-volume workloads.
  - Not suitable when you need predictable high throughput or features that require provisioned capacity.

> **Important:** Do not mix serverless with provisioned throughput settings in the same account design. If you choose serverless, omit throughput settings on databases and containers.

**Common creation constraints**

Cosmos DB has practical limits and rules that affect how you create databases and containers.

- **Minimum throughput** exists for provisioned resources and varies by mode and API.
- **Shared throughput** is only available for certain APIs and resource types.
- **Container throughput** and **database throughput** are mutually exclusive for a given resource scope.
- **Partition key choice** is critical because it affects distribution, query efficiency, and hot partition risk.
- **Unique key policies** must be defined when the container is created.
- **Indexing policies** can be customized at creation time and are often easier to define up front than to retrofit later.

For exam scenarios, the most common design mistake is choosing a poor partition key or assigning throughput at the wrong scope.

**Creating a SQL API account and resources with Bicep**

Bicep is a common way to deploy Cosmos DB resources in Azure. For the SQL API, you typically create the account first, then the database, then the container.

```bicep
param accountName string
param location string = resourceGroup().location
param databaseName string = 'appdb'
param containerName string = 'orders'

resource cosmosAccount 'Microsoft.DocumentDB/databaseAccounts@2024-05-15' = {
  name: accountName
  location: location
  kind: 'GlobalDocumentDB'
  properties: {
    databaseAccountOfferType: 'Standard'
    locations: [
      {
        locationName: location
        failoverPriority: 0
        isZoneRedundant: false
      }
    ]
    consistencyPolicy: {
      defaultConsistencyLevel: 'Session'
    }
    capabilities: [
      {
        name: 'EnableServerless'
      }
    ]
  }
}

resource sqlDatabase 'Microsoft.DocumentDB/databaseAccounts/sqlDatabases@2024-05-15' = {
  name: '${cosmosAccount.name}/${databaseName}'
  properties: {
    resource: {
      id: databaseName
    }
  }
}

resource sqlContainer 'Microsoft.DocumentDB/databaseAccounts/sqlDatabases/containers@2024-05-15' = {
  name: '${cosmosAccount.name}/${databaseName}/${containerName}'
  properties: {
    resource: {
      id: containerName
      partitionKey: {
        paths: [
          '/customerId'
        ]
        kind: 'Hash'
      }
      indexingPolicy: {
        indexingMode: 'consistent'
        automatic: true
        includedPaths: [
          {
            path: '/*'
          }
        ]
        excludedPaths: [
          {
            path: '/"_etag"/?'
          }
        ]
      }
    }
  }
}
```

This example shows a serverless account, so it does not include `throughput` on the database or container. If you deploy provisioned throughput instead, remove the serverless capability and add throughput at the database or container scope as appropriate.

**Provisioned SQL API example with container throughput**

```bicep
param accountName string
param location string = resourceGroup().location
param databaseName string = 'appdb'
param containerName string = 'orders'

resource cosmosAccount 'Microsoft.DocumentDB/databaseAccounts@2024-05-15' = {
  name: accountName
  location: location
  kind: 'GlobalDocumentDB'
  properties: {
    databaseAccountOfferType: 'Standard'
    locations: [
      {
        locationName: location
        failoverPriority: 0
      }
    ]
    consistencyPolicy: {
      defaultConsistencyLevel: 'Session'
    }
  }
}

resource sqlDatabase 'Microsoft.DocumentDB/databaseAccounts/sqlDatabases@2024-05-15' = {
  name: '${cosmosAccount.name}/${databaseName}'
  properties: {
    resource: {
      id: databaseName
    }
  }
}

resource sqlContainer 'Microsoft.DocumentDB/databaseAccounts/sqlDatabases/containers@2024-05-15' = {
  name: '${cosmosAccount.name}/${databaseName}/${containerName}'
  properties: {
    resource: {
      id: containerName
      partitionKey: {
        paths: [
          '/customerId'
        ]
        kind: 'Hash'
      }
    }
    options: {
      throughput: 400
    }
  }
}
```

Use container throughput when one container needs isolated performance. Use database shared throughput when several containers can share the same RU/s pool.

**Creating resources with Azure CLI**

The Azure CLI is useful for quick provisioning and exam practice. The exact commands vary slightly by API and CLI version, but the workflow is consistent.

```bash
az cosmosdb create \
  --name my-cosmos-account \
  --resource-group rg-app \
  --locations regionName=eastus failoverPriority=0 \
  --default-consistency-level Session
```

Create a SQL database:

```bash
az cosmosdb sql database create \
  --account-name my-cosmos-account \
  --resource-group rg-app \
  --name appdb
```

Create a SQL container with a partition key:

```bash
az cosmosdb sql container create \
  --account-name my-cosmos-account \
  --resource-group rg-app \
  --database-name appdb \
  --name orders \
  --partition-key-path /customerId
```

Create a container with throughput:

```bash
az cosmosdb sql container create \
  --account-name my-cosmos-account \
  --resource-group rg-app \
  --database-name appdb \
  --name orders \
  --partition-key-path /customerId \
  --throughput 400
```

For shared throughput, create the database with throughput and omit container throughput:

```bash
az cosmosdb sql database create \
  --account-name my-cosmos-account \
  --resource-group rg-app \
  --name appdb \
  --throughput 400
```

**Creating resources with the Azure SDK**

For application provisioning, you can create Cosmos DB resources using the Azure SDK. This is relevant when your app bootstraps its own database objects or when you automate setup in deployment code.

In .NET, the `Azure.Cosmos` SDK supports database and container creation for the SQL API.

```csharp
using Azure.Cosmos;

var client = new CosmosClient(connectionString);

DatabaseResponse dbResponse = await client.CreateDatabaseIfNotExistsAsync("appdb");

ContainerProperties containerProperties = new ContainerProperties(
    id: "orders",
    partitionKeyPath: "/customerId")
{
    IndexingPolicy = new IndexingPolicy
    {
        Automatic = true,
        IndexingMode = IndexingMode.Consistent
    },
    UniqueKeyPolicy = new UniqueKeyPolicy
    {
        UniqueKeys =
        {
            new UniqueKey
            {
                Paths = { "/orderNumber" }
            }
        }
    }
};

ContainerResponse containerResponse = await dbResponse.Database.CreateContainerIfNotExistsAsync(
    containerProperties,
    throughput: 400);
```

Use SDK-based creation when your application needs to initialize resources on first run, but prefer infrastructure-as-code for repeatable production deployments.

**Custom indexing and unique keys**

Indexing and uniqueness are often configured when you create the container.

- **Indexing policy**
  - Controls which paths are indexed and how queries are served.
  - Can reduce write cost when you exclude unused paths.
  - Should match your query patterns.
- **Unique key policy**
  - Enforces uniqueness across one or more paths within a partition.
  - Must be defined at container creation time.
  - Helps prevent duplicate business keys such as order numbers or usernames.

Example container definition with a custom indexing policy and unique key policy:

```json
{
  "id": "orders",
  "partitionKey": {
    "paths": ["/customerId"],
    "kind": "Hash"
  },
  "indexingPolicy": {
    "indexingMode": "consistent",
    "automatic": true,
    "includedPaths": [
      { "path": "/*" }
    ],
    "excludedPaths": [
      { "path": "/\"_etag\"/?" }
    ]
  },
  "uniqueKeyPolicy": {
    "uniqueKeys": [
      {
        "paths": ["/orderNumber"]
      }
    ]
  }
}
```

The `_etag` path is commonly excluded because it is system-managed and usually not needed for application queries.

**Non-SQL API creation patterns**

Other APIs use different resource names, but the provisioning idea is the same: create the account, then the logical database-like resource, then the container-like resource.

- **MongoDB API**
  - Create a MongoDB-compatible Cosmos DB account.
  - Create a database, then a collection.
  - Use MongoDB drivers and MongoDB-style commands in your application.
- **Cassandra API**
  - Create a Cassandra account.
  - Create a keyspace, then a table.
  - Model data for wide-column access patterns.
- **Gremlin API**
  - Create a Gremlin account.
  - Create a database, then a graph.
  - Use vertices and edges for graph traversal workloads.
- **Table API**
  - Create a Table API account.
  - Create tables and use partition key plus row key semantics.

Example CLI pattern for a MongoDB API account is conceptually similar, but the resource type and child resources differ from SQL API. In practice, you should use the API-specific CLI or SDK commands for the chosen model.

**When to choose each creation method**

Different provisioning methods fit different stages of development.

- **Portal**
  - Best for learning, quick validation, and manual inspection.
  - Useful when you want to verify settings visually.
- **Azure CLI**
  - Best for scripting and repeatable local automation.
  - Good for quick exam practice and troubleshooting.
- **Bicep/ARM**
  - Best for production infrastructure-as-code.
  - Ensures repeatable deployments and source control.
- **Azure SDK**
  - Best when application code must create or verify resources at runtime.
  - Common in bootstrap or integration-test scenarios.

A practical production pattern is to use Bicep for account/database/container deployment and the SDK for data operations.

**Common exam scenarios**

1. You need a low-cost environment for intermittent testing.
   - Choose **serverless** if the API and feature set support it.
   - Do not configure RU/s on databases or containers.
2. You need predictable performance for one hot workload.
   - Choose **container-level throughput**.
   - Pick a partition key that distributes load well.
3. You have several small containers with light traffic.
   - Choose **database shared throughput**.
   - Keep the workload simple and cost-effective.
4. You need to enforce a business identifier.
   - Add a **unique key policy** at container creation.
5. You need to optimize query cost.
   - Customize the **indexing policy** before deployment.

**Practical design guidance**

Good Cosmos DB design starts with the data model, not the deployment command.

- Choose a partition key with high cardinality and even distribution.
- Avoid keys that create hot partitions, such as a single tenant or date-only value, unless the workload is intentionally scoped that way.
- Keep item size and query patterns in mind when deciding whether to use shared or dedicated throughput.
- Prefer infrastructure-as-code for repeatable deployments.
- Use SDK creation only when the application truly owns resource lifecycle.

**Troubleshooting and validation**

If creation fails, check the following first:

- The account mode matches the throughput settings you are trying to apply.
- The API type matches the resource type and child resource names.
- The partition key path exists in the documents you write.
- The throughput value meets the minimum for the chosen mode.
- The indexing or unique key policy is valid JSON and uses supported paths.
- The region and feature combination support the requested capability.

A quick validation workflow is to create the account, inspect the resource in the portal, and then confirm the database and container settings with CLI or SDK calls.

**Key takeaways**

- Cosmos DB resources are created in a hierarchy: account, database, container.
- The exact resource names depend on the API.
- Partition key design is central to performance and scaling.
- Provisioned throughput can be assigned at the database or container level in supported scenarios.
- Serverless does not use RU/s provisioning.
- Indexing and unique key policies are best defined at container creation.
- Bicep, CLI, and SDK all have valid use cases in AZ-204 scenarios.
