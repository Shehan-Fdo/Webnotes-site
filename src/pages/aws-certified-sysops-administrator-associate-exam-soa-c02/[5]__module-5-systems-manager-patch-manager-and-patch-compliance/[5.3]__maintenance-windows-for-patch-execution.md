## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Overview**

AWS Systems Manager <abbr title="Systems Manager">SSM</abbr> maintenance windows let you define a recurring time window for operational tasks such as patching, reboots, scripts, and other controlled maintenance. For exam purposes, the key idea is that you separate *when* work is allowed from *what* work runs and *which* managed nodes receive it.

Patch execution is a common maintenance window use case because it gives you a predictable schedule, controlled concurrency, and a way to avoid patching during business hours. You typically combine a maintenance window with Patch Manager, patch baselines, and a target selection strategy based on instance IDs, tags, or resource groups.

**Core concepts**

- **Maintenance window**: A scheduled period during which Systems Manager can run tasks.
- **Target**: The set of managed nodes or resources the window applies to.
- **Task**: The action run during the window, such as `AWS-RunPatchBaseline`.
- **Patch baseline**: The rule set that defines approved patches and approval behavior.
- **Patch group**: A `Patch Group` tag used by Patch Manager to associate instances with a baseline.

A maintenance window does not itself patch instances. It provides the schedule and execution framework; the patching logic comes from the task document and the patch baseline.

**Maintenance window scheduling**

You define a schedule using either a cron expression or a rate expression. Use cron when you need precise calendar control, and use rate when you want a simple recurring interval.

- **Cron expressions** are best when you need:
  - Specific days of week or month
  - A fixed local maintenance pattern
  - More precise operational windows
- **Rate expressions** are best when you need:
  - Simple recurring intervals such as every 7 days
  - Less complex scheduling logic

A maintenance window also includes a **duration** and a **cutoff**.

- **Duration** is the total length of the window.
- **Cutoff** is the number of hours before the end of the window when new tasks stop starting.

This matters because a task can be scheduled for the window but still fail to start if the cutoff has passed. In exam terms, cutoff protects you from starting work too late in the window.

Example schedule concepts:

```text
cron(0 2 ? * SUN *)
# Every Sunday at 02:00 UTC

rate(7 days)
# Every 7 days
```

A practical interpretation:

- If the window duration is 4 hours and the cutoff is 1 hour, tasks can start only during the first 3 hours.
- If a task has not started before the cutoff, Systems Manager does not begin it even if the window is still technically open.

**Maintenance window lifecycle and operations**

Maintenance windows can be enabled or disabled. This is useful when you want to pause patching without deleting the schedule.

- **Enabled** windows can accept and run tasks.
- **Disabled** windows remain defined but do not execute tasks.
- **Execution history** helps you verify what ran, when it ran, and whether it succeeded.
- **Next execution** is useful for troubleshooting schedule expectations.

Operationally, you should check the window state when patching does not occur as expected. A disabled window, a missed schedule, or a cutoff issue can all look like a patching failure at first glance.

**Target selection for maintenance windows**

Maintenance window targets determine which managed nodes the window can act on. In practice, you can target by instance IDs, tags, or resource groups.

- **Instance IDs**
  - Best when you need a fixed, explicit set of nodes
  - Useful for small or highly controlled fleets
  - Less flexible for dynamic environments
- **Tags**
  - Best when instances are grouped logically, such as `Patch Group=prod-linux`
  - Useful when instances scale in and out
  - Common for patching workflows
- **Resource groups**
  - Best when you already manage resources through AWS Resource Groups
  - Useful for broader operational grouping across services

A common exam distinction is that the *targeting method* and the *task registration* are separate concerns:

- The maintenance window target defines the selection criteria.
- The maintenance window task references that target or directly references instances, depending on how you register the task.

If you target by tag, the target registration should reflect that selection criteria. If you target by instance IDs, the task can be bound to those specific target IDs. Do not confuse the target definition with the task action itself.

Example target registration by tag:

```bash
aws ssm register-target-with-maintenance-window \
  --window-id mw-0abc1234def567890 \
  --resource-type INSTANCE \
  --targets Key=tag:Patch Group,Values=prod-linux \
  --name ProdLinuxPatchTargets \
  --description "Linux instances tagged for production patching"
```

Example target registration by instance IDs:

```bash
aws ssm register-target-with-maintenance-window \
  --window-id mw-0abc1234def567890 \
  --resource-type INSTANCE \
  --targets Key=InstanceIds,Values=i-0123456789abcdef0,i-0fedcba9876543210 \
  --name ExplicitPatchTargets
```

Example target registration by resource group:

```bash
aws ssm register-target-with-maintenance-window \
  --window-id mw-0abc1234def567890 \
  --resource-type INSTANCE \
  --targets Key=resource-groups:Name,Values=ProdLinuxFleet \
  --name ProdLinuxResourceGroup
```

**Patch groups and baselines**

Patch groups are a Patch Manager concept, not a generic Systems Manager grouping feature. You use the `Patch Group` tag to associate instances with a patch baseline.

The important exam point is this:

- A patch baseline defines what patches are approved.
- A patch group maps instances to a baseline through the `Patch Group` tag.
- You can still patch instances without relying on patch groups if your workflow targets instances directly and uses the desired baseline or document behavior.

This means patch groups are helpful for consistent fleet management, but they are not the only way to run patching.

Example baseline creation:

```bash
aws ssm create-patch-baseline \
  --name "ProdLinuxBaseline" \
  --operating-system AMAZON_LINUX_2 \
  --approval-rules 'PatchRules=[{PatchFilterGroup={PatchFilters=[{Key=CLASSIFICATION,Values=[Security]},{Key=SEVERITY,Values=[Critical,Important]}]},ApproveAfterDays=7}]' \
  --description "Approve critical and important security patches after 7 days"
```

Example associating a patch group to a baseline:

```bash
aws ssm register-patch-baseline-for-patch-group \
  --baseline-id pb-0123456789abcdef0 \
  --patch-group prod-linux
```

Example tagging an instance for the patch group:

```bash
aws ec2 create-tags \
  --resources i-0123456789abcdef0 \
  --tags Key=Patch Group,Value=prod-linux
```

This association is what lets Patch Manager know which baseline to use for instances carrying that tag.

**Maintenance window tasks for patching**

The most common patching task uses the `AWS-RunPatchBaseline` document. This document is typically run as a command task in a maintenance window because it performs patch scanning and installation on managed nodes.

Use the task type that matches the document and execution model you are using. For the common maintenance-window patching workflow, `RUN_COMMAND` is the expected task type for `AWS-RunPatchBaseline`. In some workflows, Systems Manager Automation can also orchestrate patch-related actions, but do not assume every patching workflow uses the same task type.

Example task registration:

```bash
aws ssm register-task-with-maintenance-window \
  --window-id mw-0abc1234def567890 \
  --targets Key=WindowTargetIds,Values=wt-0a1b2c3d4e5f67890 \
  --task-arn AWS-RunPatchBaseline \
  --task-type RUN_COMMAND \
  --priority 1 \
  --max-concurrency 10 \
  --max-errors 1 \
  --service-role-arn arn:aws:iam::123456789012:role/SSMMaintenanceWindowRole \
  --name "RunPatchBaseline"
```

Important exam notes:

- **Priority** controls the order in which multiple tasks start.
- Lower numbers run before higher numbers.
- If multiple tasks are registered, priority matters when the window is short.
- **Max concurrency** limits how many targets run at once.
- **Max errors** controls how many failures are tolerated before Systems Manager stops further execution.

> The service role requirement depends on the task type and configuration. Many maintenance window tasks need an IAM role that allows Systems Manager to act on your behalf, but do not treat `--service-role-arn` as universally required in every possible maintenance window scenario.

**Patch execution behavior**

Patch execution usually follows this pattern:

1. Systems Manager identifies the target managed nodes.
2. The maintenance window opens according to the schedule.
3. The task starts if the cutoff has not passed.
4. `AWS-RunPatchBaseline` scans and installs approved patches.
5. The instance may reboot if the patching operation requires it.
6. Systems Manager records the execution result.

Reboot behavior is important because a reboot can temporarily make an instance unavailable. If your patching policy allows reboots, plan for application downtime and health checks after the reboot.

Common reboot-related options for patching include:

- **Reboot if needed**
  - Useful for normal patch cycles
  - Lets the document decide whether a reboot is required
- **No reboot**
  - Useful when you want to defer rebooting
  - May leave the instance in a partially updated state until reboot occurs later

Example patch command invocation outside a maintenance window:

```bash
aws ssm send-command \
  --document-name "AWS-RunPatchBaseline" \
  --targets "Key=InstanceIds,Values=i-0123456789abcdef0" \
  --parameters Operation=Install,RebootOption=RebootIfNeeded
```

This is useful for ad hoc patching, but maintenance windows are preferred when you need repeatable, controlled execution.

**Validation and post-patch checks**

After patching, validate both the patch result and the instance health. Patching is not complete just because the command returned.

- Confirm the instance is still a managed node in Systems Manager.
- Check whether the instance rebooted successfully.
- Verify application health and service availability.
- Confirm the patch baseline and patch group association if the wrong patches were applied or no patches were applied.

Practical validation steps often include:

```bash
aws ssm describe-maintenance-window-executions \
  --window-id mw-0abc1234def567890
```

```bash
aws ssm describe-maintenance-window-execution-task-invocations \
  --window-execution-id mw-exec-0123456789abcdef0 \
  --task-id task-0123456789abcdef0
```

Use the execution IDs returned by the maintenance window APIs. The task ID is the ID of the registered maintenance window task, not a generic placeholder.

You can also inspect command output in Systems Manager Run Command history or in the configured output destination, such as Amazon S3 or Amazon CloudWatch Logs, if you enabled logging.

**Troubleshooting signals**

When patching fails, check the basics first. Many issues are caused by agent, IAM, or targeting problems rather than the patch document itself.

- **SSM Agent**
  - Verify the agent is installed and running
  - Confirm the version supports the features you need
  - Restart the agent if it is unhealthy
- **Managed node registration**
  - Confirm the instance appears as a managed node in Systems Manager
  - Verify the instance profile allows SSM communication
- **IAM permissions**
  - Confirm the instance role can communicate with Systems Manager endpoints
  - Confirm the maintenance window service role can run the task
- **Network access**
  - Ensure the instance can reach Systems Manager endpoints, either through the internet or VPC endpoints
- **Logging**
  - Check Run Command output, CloudWatch Logs, or S3 output locations
  - Review maintenance window execution history for task start failures
- **Targeting**
  - Confirm the instance has the correct `Patch Group` tag if you use patch groups
  - Confirm the maintenance window target matches the intended selection method

A useful operational checklist:

1. Verify the maintenance window is enabled.
2. Verify the next execution time is correct.
3. Verify the target selection matches the intended fleet.
4. Verify the task priority and cutoff allow the task to start.
5. Verify the instance is a healthy managed node.
6. Verify output logs for patch scan/install and reboot results.

**Exam-focused distinctions**

- **Maintenance window vs patch baseline**
  - The window schedules work.
  - The baseline defines approved patches.
- **Patch group vs direct targeting**
  - Patch groups map instances to baselines through tags.
  - Direct targeting can patch specific instances without relying on tag-based baseline association.
- **Cron vs rate**
  - Cron is precise and calendar-based.
  - Rate is interval-based and simpler.
- **Cutoff vs duration**
  - Duration is the full window length.
  - Cutoff prevents new task starts near the end of the window.
- **Task priority**
  - Lower numbers run first.
  - Important when multiple tasks compete for a short window.
- **Reboot behavior**
  - Reboots may be required for full patch completion.
  - Plan validation after reboot to confirm service recovery.

**Practical patching workflow example**

A common production pattern looks like this:

1. Create a patch baseline for the operating system and patch policy.
2. Associate the baseline with a `Patch Group` value.
3. Tag instances with `Patch Group=prod-linux`.
4. Create a maintenance window with a weekly cron schedule.
5. Register the tagged instances as maintenance window targets.
6. Register `AWS-RunPatchBaseline` as a `RUN_COMMAND` task.
7. Set concurrency, error thresholds, and cutoff to match the fleet size.
8. Review execution history and logs after each run.

Example weekly schedule:

```bash
aws ssm create-maintenance-window \
  --name "ProdLinuxPatchWindow" \
  --schedule "cron(0 2 ? * SUN *)" \
  --duration 4 \
  --cutoff 1 \
  --allow-unassociated-targets
```

Example task tuning:

```bash
aws ssm register-task-with-maintenance-window \
  --window-id mw-0abc1234def567890 \
  --targets Key=WindowTargetIds,Values=wt-0a1b2c3d4e5f67890 \
  --task-arn AWS-RunPatchBaseline \
  --task-type RUN_COMMAND \
  --priority 1 \
  --max-concurrency 20 \
  --max-errors 2
```

This pattern gives you predictable patching with controlled blast radius and clear operational visibility.

**Key takeaways**

- Maintenance windows control *when* patching runs.
- Patch baselines control *what* patches are approved.
- Patch groups connect instances to baselines through the `Patch Group` tag.
- Targets can be selected by instance IDs, tags, or resource groups.
- Cutoff prevents tasks from starting too late in the window.
- Priority matters when multiple tasks share the same window.
- Reboots and post-patch validation are part of the operational workflow, not an afterthought.
