## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Patch management in Systems Manager**

AWS Systems Manager patching usually involves three separate permission planes:

- **Instance-side permissions** for the managed node to register with Systems Manager and report inventory/compliance.
- **Service-side permissions** for Systems Manager to orchestrate patch execution through Run Command or maintenance windows.
- **Artifact/logging permissions** for writing output to Amazon S3, CloudWatch Logs, or encrypting data with AWS Key Management Service (AWS KMS).

A common exam trap is assuming one IAM role covers everything. In practice, you troubleshoot by asking which actor failed:

- The instance could not register.
- The maintenance window could not assume its task role.
- The command ran but patching failed on the operating system.
- The command succeeded but logs or artifacts could not be written.

**Core actors and what they do**

| Actor | Purpose | Typical failure symptom |
|---|---|---|
| EC2 instance profile / managed node role | Lets the SSM Agent register, inventory, and report status | Instance never appears as managed, or shows as offline |
| Maintenance window service role | Lets Systems Manager run the maintenance window and invoke tasks | Window exists but tasks never start |
| Maintenance window task role | Lets the task perform the action it is allowed to run | Task starts but `SendCommand` or related action is denied |
| S3 / CloudWatch Logs / KMS permissions | Store output, logs, and encrypted artifacts | Patch runs, but output is missing or access denied |

**IAM basics for patching**

For EC2 instances, the instance profile usually includes the AWS managed policy `AmazonSSMManagedInstanceCore`. This policy is the standard baseline for Systems Manager managed nodes and covers the instance-side calls needed by the SSM Agent, such as registration, inventory, association, patch snapshot retrieval, and compliance reporting.

Typical instance-side capabilities include:

- Registering and maintaining managed node identity.
- Describing associations and patch-related metadata.
- Reporting inventory and compliance data.
- Retrieving patch snapshot information used by patch operations.

A minimal custom policy is possible, but for exam purposes the safest answer is usually to attach `AmazonSSMManagedInstanceCore` unless the question explicitly asks for least-privilege customization.

Example instance profile policy attachment:

```bash
aws iam attach-role-policy \
  --role-name MyEC2SSMRole \
  --policy-arn arn:aws:iam::aws:policy/AmazonSSMManagedInstanceCore
```

> The instance profile does not usually initiate patch execution itself. It enables the node to communicate with Systems Manager and report state. The orchestration side is handled by the Systems Manager service and its roles.

**Trust policies and role assumption**

Trust policy problems are a frequent cause of maintenance window failures. The role may have the right permissions but still fail because the service cannot assume it.

For a maintenance window task role, the trust policy must allow the Systems Manager service principal to assume the role. The exact principal depends on the feature, but for maintenance windows and Run Command orchestration, the trust relationship must explicitly permit Systems Manager.

Example trust policy pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "ssm.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

For some workflows, the service-linked role for Systems Manager is also relevant. If the service-linked role is missing or broken, Systems Manager may not be able to perform orchestration tasks correctly. In exam scenarios, look for wording like:

- “Systems Manager cannot assume the role”
- “Access denied on `sts:AssumeRole`”
- “Maintenance window task never starts”
- “Service role missing or misconfigured”

**Permissions commonly involved in patching**

Patch execution often uses `AWS-RunPatchBaseline`, a Systems Manager document that runs patching logic on the managed node. The permissions model is broader than just one `SendCommand` action.

Commonly relevant permissions include:

- `ssm:SendCommand` to invoke Run Command.
- `ssm:CreateAssociation` and `ssm:UpdateAssociation` when patching is driven by associations.
- `ssm:GetCommandInvocation` and related read actions for status retrieval.
- `ssm:DescribeInstanceInformation` for inventory and managed node visibility.
- `ssm:ListCommandInvocations` and `ssm:ListCommands` for troubleshooting and console visibility.
- `ssm:DescribeDocument` or document access permissions when document lookup or sharing is involved.
- `ssm:GetDocument` in some document retrieval workflows.
- `ssm:DescribeMaintenanceWindows`, `ssm:DescribeMaintenanceWindowTasks`, and `ssm:DescribeMaintenanceWindowExecutions` for maintenance window troubleshooting.

For exam prep, remember this distinction:

- **Run Command permissions** authorize the service to invoke the document.
- **Instance profile permissions** authorize the node to receive instructions and report results.
- **Logging/storage permissions** authorize output delivery.

**Maintenance window execution flow**

A maintenance window patch task has multiple moving parts. If one fails, the symptom may look like a patch problem even when the root cause is IAM.

Typical flow:

1. You create or select a maintenance window.
2. You register a task, often using `AWS-RunPatchBaseline`.
3. You assign a maintenance window service role and task role.
4. Systems Manager assumes the service role.
5. Systems Manager invokes the task using the task role.
6. The SSM Agent on the instance receives the command.
7. The instance applies patches and reports status.
8. Logs and output are written to the configured destination.

A simplified flow diagram:

```text
Maintenance Window
   |
   v
Service Role assumed by Systems Manager
   |
   v
Task Role authorizes task execution
   |
   v
SendCommand / Run Command document
   |
   v
SSM Agent on managed node
   |
   v
Patch install + status reporting + logs
```

**What to check when a task never starts**

If a maintenance window task never starts, do not assume the task role is the only issue. Check:

- The maintenance window is enabled and within its schedule.
- The target selection matches the intended instances or resource group.
- The maintenance window service role exists and trusts Systems Manager.
- The task role exists and has the required permissions.
- The task is registered against the correct document and parameters.
- The instance is a managed node and is online.

Useful console locations:

- **Systems Manager > Maintenance Windows > Executions**
- **Systems Manager > Maintenance Windows > Tasks**
- **Systems Manager > Run Command > Command history**
- **Systems Manager > Managed nodes**

**Patch baseline failures**

A patch baseline can fail even when IAM is correct. Common causes include:

- The baseline does not approve the patch.
- The patch is explicitly rejected or blocked.
- The operating system or platform does not match the baseline.
- The patch group is not associated with the baseline.
- The instance is not tagged or targeted correctly for the patch group.
- The patch snapshot is stale or unavailable.
- The maintenance window runs at a time when the patch is not yet approved.

Patch baselines are especially important for exam questions because the failure may be policy-related rather than permission-related.

Example baseline concepts:

- **Approved patches** are allowed to install.
- **Rejected patches** are excluded even if available.
- **Approval rules** can be based on classification, severity, product, or release date.
- **Compliance status** may show noncompliant even when the patch job ran successfully if the baseline rules still mark patches as missing.

**SSM Agent troubleshooting**

If the instance is not managed, patching cannot begin. Troubleshoot the agent first.

Check these items on the instance:

- The SSM Agent is installed and running.
- The instance profile is attached.
- The instance can reach Systems Manager endpoints.
- The instance metadata service is available when required.
- The node is in a supported region and platform.
- The activation or registration details are correct for on-premises or hybrid nodes.

Linux checks:

```bash
sudo systemctl status amazon-ssm-agent
sudo systemctl restart amazon-ssm-agent
sudo journalctl -u amazon-ssm-agent -n 100 --no-pager
```

Windows checks:

```powershell
Get-Service AmazonSSMAgent
Restart-Service AmazonSSMAgent
Get-Content "C:\ProgramData\Amazon\SSM\Logs\amazon-ssm-agent.log" -Tail 100
```

Common registration failure causes:

- The instance profile was attached after launch but the agent has not refreshed credentials yet.
- The instance cannot access the Instance Metadata Service (IMDS).
- IMDSv2 is required but the agent or environment is misconfigured.
- The subnet has no route to Systems Manager endpoints.
- The security group or network ACL blocks outbound access.
- The managed node activation is expired or incorrect for hybrid systems.

**Private subnet and VPC endpoint requirements**

In private subnets, patching often fails because the instance cannot reach AWS endpoints. Systems Manager typically needs interface endpoints for:

- `ssm`
- `ssmmessages`
- `ec2messages`

If you use patching features that download content or write artifacts, you may also need:

- An S3 gateway endpoint or interface access pattern, depending on your architecture.
- KMS access if encryption is enabled.
- CloudWatch Logs access if logs are sent there.

Endpoint policy considerations matter in locked-down environments:

- The endpoint policy must allow the required service actions.
- The instance role must still have IAM permissions.
- Security groups on interface endpoints must allow traffic from the instance subnet.

A common exam pattern is a private EC2 instance that has the correct IAM role but no endpoint path to Systems Manager. In that case, the instance never becomes a managed node or goes offline after registration.

**CloudTrail clues for IAM troubleshooting**

CloudTrail is useful when you need to map a failure to a missing permission or trust issue.

Look for events such as:

- `AssumeRole`
- `SendCommand`
- `StartAutomationExecution`
- `CreateAssociation`
- `UpdateAssociation`
- `RegisterManagedInstance`
- `UpdateInstanceInformation`
- `PutInventory`
- `PutComplianceItems`

How to interpret them:

- `AccessDenied` on `sts:AssumeRole` usually points to a trust policy or service role problem.
- `AccessDenied` on `ssm:SendCommand` usually points to the task role or caller permissions.
- Missing `UpdateInstanceInformation` or `PutInventory` often points to the instance profile.
- Successful `SendCommand` with failed command invocation suggests the instance-side agent, OS, or patch baseline is the issue.

**Command invocation status and failure output**

When a patch command runs, the console and CLI expose useful status fields. Check:

- Command status
- Invocation status
- Response code
- Standard output content
- Standard error content
- Execution time
- Target instance ID

Example CLI inspection:

```bash
aws ssm list-commands
aws ssm list-command-invocations --details
aws ssm get-command-invocation \
  --command-id 12345678-1234-1234-1234-123456789012 \
  --instance-id i-0123456789abcdef0
```

Important status meanings:

- `Pending` means the command has not started.
- `InProgress` means the agent received it and is working.
- `Success` means the command completed successfully.
- `Failed` means the command ran but returned an error.
- `Cancelled` means execution was stopped.
- `TimedOut` means the command exceeded the allowed time.

**Windows-specific patch troubleshooting**

Windows patching often fails for reasons beyond IAM.

Check these areas:

- Windows Update service is running.
- Windows Update Agent is healthy.
- The servicing stack is current enough to process the patch.
- The instance can reach Microsoft update sources or your internal update infrastructure.
- Group Policy is not blocking updates.
- Reboot behavior is understood and allowed by the maintenance window.

Useful repair steps include:

```powershell
net stop wuauserv
net stop bits
net stop cryptsvc
ren C:\Windows\SoftwareDistribution SoftwareDistribution.old
ren C:\Windows\System32\catroot2 catroot2.old
net start cryptsvc
net start bits
net start wuauserv
```

If servicing stack corruption is suspected, you may need to repair the image or apply prerequisite updates before patching can succeed. In exam scenarios, a Windows patch failure is often caused by update service health, not IAM.

**Linux-specific patch troubleshooting**

Linux patch failures often involve repository access or package manager state.

Check:

- `yum`, `dnf`, or `apt` repository reachability.
- Package manager locks or stale metadata.
- Disk space.
- Reboot requirements.
- Kernel or dependency conflicts.

Examples:

```bash
sudo yum clean all
sudo yum makecache
sudo dnf updateinfo list security
sudo apt-get update
sudo apt-get -y upgrade
df -h
```

If the instance is in a private subnet, repository access may depend on NAT, proxy, or internal mirrors. That is separate from Systems Manager connectivity.

**Logging, S3, CloudWatch Logs, and KMS**

Do not confuse patch execution permissions with artifact-writing permissions.

If you configure output to S3 or CloudWatch Logs:

- The role used for logging must allow `s3:PutObject`, `s3:GetBucketLocation`, and related bucket actions as needed.
- CloudWatch Logs delivery requires log group and stream permissions.
- If encryption uses KMS, the relevant principal needs `kms:Encrypt`, `kms:Decrypt`, `kms:GenerateDataKey`, and sometimes `kms:DescribeKey`.

This is especially important when the patch command succeeds but the console shows missing output or encrypted artifact errors.

Example S3 output configuration:

```bash
aws ssm send-command \
  --document-name "AWS-RunPatchBaseline" \
  --targets "Key=InstanceIds,Values=i-0123456789abcdef0" \
  --output-s3-bucket-name my-ssm-logs \
  --output-s3-key-prefix patching/
```

Use this pattern carefully: output storage is not the same as patch execution. A missing S3 permission can break logging without preventing the patch from running.

**Practical troubleshooting decision tree**

1. Confirm the instance is a managed node.
2. Confirm the maintenance window or command actually started.
3. Check whether Systems Manager can assume the required role.
4. Check whether the task role or caller has `SendCommand`-related permissions.
5. Check the SSM Agent, network path, and endpoint access.
6. Check patch baseline rules, OS compatibility, and patch group targeting.
7. Check output destinations, encryption, and log delivery.

**Common exam scenarios**

- A private EC2 instance has the right instance profile, but patching fails because the subnet lacks Systems Manager VPC endpoints.
- A maintenance window exists, but tasks never run because the service role trust policy does not allow Systems Manager to assume it.
- `SendCommand` succeeds, but the instance returns `Failed` because the SSM Agent is stopped.
- The patch job runs, but compliance remains noncompliant because the baseline does not approve the missing patch.
- Logs are missing because the role can run the command but cannot write to S3 or CloudWatch Logs.
- A Windows instance fails patching because Windows Update services are corrupted, not because of IAM.

**Quick reference: what to verify first**

- **Managed node status** in Systems Manager.
- **Instance profile** attached before troubleshooting deeper.
- **Trust policy** for the maintenance window service role and task role.
- **VPC endpoints** for private subnets.
- **Patch baseline** approval and target association.
- **Command invocation output** and error codes.
- **OS-specific update services** and repository access.

**Key takeaways**

- Instance profiles support registration, inventory, and reporting; they do not replace service-side orchestration roles.
- Maintenance windows require both correct permissions and correct trust relationships.
- Patch failures are often caused by baseline rules, network reachability, or agent health rather than IAM alone.
- CloudTrail, command invocation details, and maintenance window execution history are the fastest ways to isolate the failing layer.
- Separate execution permissions from logging and encryption permissions when you troubleshoot output-related failures.
