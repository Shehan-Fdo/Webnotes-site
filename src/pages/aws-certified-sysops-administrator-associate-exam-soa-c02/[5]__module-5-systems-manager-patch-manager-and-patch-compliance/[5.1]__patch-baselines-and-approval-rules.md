## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Patch Manager overview**

AWS Systems Manager Patch Manager helps you define which patches are approved for installation on managed instances and then automate scanning and installation. A **patch baseline** is the policy object that controls patch approval behavior for a specific operating system. In exam scenarios, think of the baseline as the rule set, the patch group as the targeting mechanism, and the maintenance window or command execution as the delivery mechanism.

Patch Manager is most useful when you need consistent patching across fleets with different environments such as dev, staging, and production. You use it to standardize patch selection, reduce drift, and produce compliance reports that show whether instances match the approved patch state.

**Core concepts**

A patch baseline is built from a few distinct parts:

- **Approved patches** define what is allowed.
- **Rejected patches** define what is explicitly blocked.
- **Approval rules** define dynamic approval logic based on patch metadata.
- **Global filters** narrow the universe of patches the rules can consider.

These pieces work together, but they are not interchangeable. Approval rules are usually the most important exam topic because they let you approve patches automatically based on age, classification, severity, product, or other OS-specific metadata.

**How baselines are selected**

Patch baselines are OS-specific, and AWS provides a default baseline for each supported operating system in each Region context. You can create custom baselines, but you do not directly edit the AWS-managed default baseline in the same way you edit a custom one. In practice, you usually create a custom baseline and associate it with a patch group when you need controlled patch policy.

Selection follows a simple idea:

- If an instance is associated with a patch group, Patch Manager uses the baseline registered to that patch group.
- If no patch group association applies, the default baseline for that OS is used.
- If multiple associations exist, the effective baseline is the one registered for the patch group that matches the instance’s `Patch Group` tag value.

> For exam questions, remember that patch group targeting depends on the instance tag key `Patch Group` and the baseline registration for that exact group name.

**Patch group tagging mechanics**

Patch groups are not a special AWS resource type; they are a convention based on instance tags. You tag the instance with the key `Patch Group` and a value such as `Production`, `Staging`, or `Linux-Blue`. Then you register that patch group name to a baseline.

Typical flow:

1. Create or choose a baseline.
2. Tag instances with `Patch Group=<group-name>`.
3. Register the patch group name to the baseline.
4. Run scan or install operations against the tagged instances.

Example instance tag:

```text
Key: Patch Group
Value: Production
```

Example registration command:

```bash
aws ssm register-patch-baseline-for-patch-group \
  --baseline-id pb-0123456789abcdef0 \
  --patch-group Production
```

This registration step matters. Tagging alone does not activate the baseline; the patch group must be registered to the baseline before Patch Manager uses it.

**Baseline components in detail**

**Approved patches**

Approved patches are patches that the baseline explicitly allows. You can approve patches directly by ID or indirectly through approval rules. Direct approval is useful when you want precise control over a known patch set.

**Rejected patches**

Rejected patches are patches you want excluded even if they might otherwise match an approval rule. Rejections are useful for blocking problematic updates, vendor-specific exceptions, or patches that require a later rollout.

**Approval rules**

Approval rules let you approve patches automatically based on metadata and timing. A common rule says, “Approve security updates after 7 days, except critical severity updates after 1 day.” This is the preferred approach for ongoing maintenance because it reduces manual baseline updates.

**Global filters**

Global filters constrain the patch universe before approval logic is applied. They are OS-specific and vendor-specific. This is important for the exam: not every filter key is valid for every operating system, and some metadata fields only exist for certain vendors or package ecosystems.

**OS-specific constraints**

Patch baselines are not universal across all platforms. The supported filter keys and patch metadata depend on the operating system and vendor data exposed by that platform. For example:

- Linux distributions may expose package name, severity, classification, and release metadata differently.
- Windows baselines use Microsoft patch metadata.
- macOS and other platforms have their own supported fields and limitations.

Do not assume a filter key is portable across operating systems. In exam questions, always match the baseline rule to the platform’s supported metadata.

**Approval rule behavior**

Approval rules are evaluated against patch metadata and age. The most common field is `ApproveAfterDays`, which delays approval until a patch has been available for a specified number of days. This helps you avoid approving brand-new patches immediately.

A practical rule might say:

- Approve security patches after 7 days.
- Approve critical patches after 1 day.
- Approve all other patches after 30 days.

That pattern gives you a staged rollout model where higher-risk patches are validated before broad deployment.

Example baseline creation snippet:

```bash
aws ssm create-patch-baseline \
  --name "Prod-Linux-Baseline" \
  --operating-system AMAZON_LINUX_2 \
  --description "Production Linux baseline" \
  --approval-rules '{
    "PatchRules": [
      {
        "PatchFilterGroup": {
          "PatchFilters": [
            { "Key": "CLASSIFICATION", "Values": ["Security"] }
          ]
        },
        "ApproveAfterDays": 7,
        "ComplianceLevel": "CRITICAL"
      }
    ]
  }'
```

This example is illustrative of the structure and intent: approval rules contain patch filter groups and timing controls. In real use, always verify the exact CLI syntax for the operating system and AWS CLI version you are using.

**Scanning vs installing**

Patch Manager supports two distinct operations:

- **Scan** checks what is missing relative to the baseline.
- **Install** applies approved patches.

These operations are related but not identical.

- A scan reports compliance status without changing the instance.
- An install attempts to bring the instance into compliance by applying approved patches.

Approval rules affect both operations, but in different ways:

- During a scan, the instance is evaluated against the baseline to determine whether approved patches are missing.
- During an install, only approved patches are candidates for installation.

This distinction matters because a patch can be available on the instance but still not count as compliant if it is approved by the baseline and not installed. Conversely, a patch that is not yet approved by the baseline is typically not considered part of the desired compliant state.

**Compliance interpretation**

Patch compliance is based on whether the instance has the patches approved by the baseline installed at the time of the scan or install evaluation. Rejected patches can influence reporting and desired state, but the main compliance question is whether the instance matches the baseline’s approved patch set.

A useful mental model:

- Approved and installed: compliant.
- Approved but missing: noncompliant.
- Not approved yet: not part of the desired state until the baseline approves it.
- Explicitly rejected: excluded from approval logic and may remain a compliance concern depending on the baseline and scan context.

> Avoid oversimplifying compliance as “approved patches installed and rejected patches absent.” The actual result depends on the baseline, the scan/install operation, and the patch state on the instance.

**End-to-end compliance scenario**

Imagine a production Linux fleet:

- Baseline approves security patches after 7 days.
- Critical security patches are approved after 1 day.
- Instances are tagged `Patch Group=Production`.
- The `Production` patch group is registered to the production baseline.

Scenario:

1. A new critical kernel patch is released on Monday.
2. On Tuesday, the baseline approves it because `ApproveAfterDays=1`.
3. A maintenance window runs an install on Wednesday.
4. Instances that successfully install the patch become compliant.
5. Instances that fail installation or were offline remain noncompliant until the next successful install and scan.

This is the kind of scenario AWS exam questions often test: policy timing, patch group targeting, and the difference between approval and installation.

**Creating and managing baselines**

A typical baseline lifecycle includes:

1. Create a custom baseline.
2. Add approval rules and optional explicit approvals or rejections.
3. Register a patch group to the baseline.
4. Associate instances with the patch group tag.
5. Run scans and installs on a schedule.
6. Review compliance and adjust rules as needed.

Example of a more complete baseline definition:

```bash
aws ssm create-patch-baseline \
  --name "Staging-Windows-Baseline" \
  --operating-system WINDOWS \
  --approval-rules '{
    "PatchRules": [
      {
        "PatchFilterGroup": {
          "PatchFilters": [
            { "Key": "MSRC_SEVERITY", "Values": ["Critical", "Important"] }
          ]
        },
        "ApproveAfterDays": 3
      }
    ]
  }' \
  --approved-patches "KB5030211" "KB5030310" \
  --rejected-patches "KB5000001"
```

Use direct approvals and rejections sparingly. They are best for exceptions, not for the entire policy. Approval rules scale better for ongoing operations.

**Patch baseline selection and precedence**

When multiple patching choices exist, the effective baseline is determined by the instance’s patch group association first, then by the default baseline if no patch group match exists. This is why patch group registration is so important in production environments.

Practical precedence model:

| Situation | Effective baseline |
|---|---|
| Instance tagged and registered patch group exists | Registered custom baseline |
| Instance tagged but group not registered | Default baseline for that OS |
| No patch group tag | Default baseline for that OS |

This is a common exam trap. Tagging an instance without registering the patch group to a baseline does not guarantee the custom baseline is used.

**Real-world operating model**

Teams often use different baselines for different environments:

- **Dev**: faster approval, shorter delay, broader patch acceptance.
- **Staging**: moderate delay, closer to production policy.
- **Production**: conservative approval, longer delay, explicit exception handling.

A common pattern is to use the same OS baseline family but different patch group registrations:

- `Patch Group=Dev`
- `Patch Group=Staging`
- `Patch Group=Production`

Each group can point to a different baseline, allowing environment-specific patch timing without changing instance-level automation.

**CLI examples for targeting and execution**

Scan a patch group:

```bash
aws ssm send-command \
  --document-name "AWS-RunPatchBaseline" \
  --targets "Key=tag:Patch Group,Values=Production" \
  --parameters "Operation=Scan" \
  --comment "Scan production instances for patch compliance"
```

Install approved patches:

```bash
aws ssm send-command \
  --document-name "AWS-RunPatchBaseline" \
  --targets "Key=tag:Patch Group,Values=Production" \
  --parameters "Operation=Install" \
  --comment "Install approved patches on production instances"
```

These examples use the correct `--targets` style: a list of `Key=...,Values=...` entries, not JSON objects.

You can also run patching through a maintenance window task when you want repeatable scheduling. That is often the preferred production pattern because it centralizes change timing and reduces ad hoc execution.

**Troubleshooting and exam pitfalls**

- **Patch group not taking effect**
  - Confirm the instance has the `Patch Group` tag.
  - Confirm the patch group name is registered to the intended baseline.
  - Confirm the instance OS matches the baseline OS.

- **Unexpected compliance results**
  - Check whether you ran `Scan` or `Install`.
  - Verify the approval delay has elapsed.
  - Confirm the patch is actually supported by the baseline’s OS/vendor metadata.

- **Baseline appears ignored**
  - Check for a default baseline fallback.
  - Confirm the patch group registration exists in the same Region.
  - Verify the instance is managed by Systems Manager and has the required IAM permissions and SSM Agent connectivity.

- **Rule does not match expected patches**
  - Recheck supported filter keys for that OS.
  - Confirm the vendor metadata is present for the patch.
  - Avoid assuming a filter key from one platform works on another.

**Quick comparison**

| Concept | Purpose | Exam focus |
|---|---|---|
| Patch baseline | Defines approved patch policy | Policy source of truth |
| Approval rule | Dynamically approves patches | Timing and metadata filters |
| Approved patches | Explicit allow list | Exception handling and precision |
| Rejected patches | Explicit deny list | Blocking known-bad updates |
| Patch group | Instance targeting via tag | `Patch Group` tag convention |
| Scan | Evaluate compliance | No changes made |
| Install | Apply approved patches | Changes instance state |

**Best practices**

- Use custom baselines for production control.
- Use patch groups to separate environments.
- Prefer approval rules over large manual patch lists.
- Keep approval delays long enough to validate patches, but short enough to meet patching SLAs.
- Validate OS-specific filter keys before relying on them in automation.
- Schedule scans regularly and installs during maintenance windows.
- Review compliance reports after each patch cycle to catch drift early.

**What to remember for SOA-C02**

- Patch baselines define what is approved for installation.
- Patch groups are instance tags using the key `Patch Group`.
- The patch group must be registered to the baseline to take effect.
- Default baselines exist per OS/Region context, and custom baselines are used through patch group association.
- Approval rules use patch metadata and timing such as `ApproveAfterDays`.
- Scan and install are different operations with different outcomes.
- Supported filter keys are OS-specific, so do not assume universal patch metadata.
