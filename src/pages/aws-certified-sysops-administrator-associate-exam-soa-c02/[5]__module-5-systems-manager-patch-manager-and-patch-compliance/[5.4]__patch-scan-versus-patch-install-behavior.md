## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Patch Manager behavior at a glance**

AWS Systems Manager Patch Manager uses two distinct operations: **scan** and **install**. Scan evaluates patch compliance without changing the instance, while install applies approved patches and can optionally reboot the instance. For exam purposes, the key distinction is that scan is read-only and install is state-changing.

Patch Manager typically works through a **patch baseline**, which defines approval rules for patches by operating system, classification, severity, or explicit patch IDs. The baseline determines what counts as approved, but the operation you run determines whether the instance only reports missing patches or actually remediates them.

- **Scan** checks the instance against the baseline and reports missing or noncompliant patches.
- **Install** downloads and applies approved patches that are missing.
- **Reboot behavior** depends on the patch and the selected options; some updates require a restart to complete.
- **Compliance data** from scan is useful for reporting, auditing, and change planning.

**Core difference between scan and install**

A scan operation is a compliance assessment. It inventories installed packages, compares them to the patch baseline, and records what is missing. It does not modify the operating system. An install operation performs the same evaluation first, then downloads and installs approved patches that are not present.

This distinction matters in production because you often want to separate visibility from change. You may scan frequently to understand drift and schedule install during a maintenance window.

| Operation | Purpose | Changes instance state | Typical use |
|---|---|---:|---|
| `Scan` | Detect missing patches | No | Compliance reporting, pre-change validation |
| `Install` | Apply approved patches | Yes | Maintenance windows, remediation |

**How Patch Manager evaluates patches**

Patch Manager compares the instance against the patch baseline and the instance’s operating system metadata. The baseline can include rules such as:

- Approved classifications, such as security or bugfix updates
- Approved severities, such as critical or important
- Auto-approval delays for newly released patches
- Explicit approval or rejection of specific patch IDs

The scan result reflects the baseline at the time of execution. If a patch is not approved by the baseline, it is not considered installable by Patch Manager even if the package exists in the repository.

> A scan can show a patch as missing even when the patch is available from the vendor repository, because “available” is not the same as “approved by baseline.”

**Typical operational flow**

1. Associate the instance with the correct patch baseline.
2. Run a scan to identify missing patches.
3. Review compliance results and confirm the maintenance window.
4. Run install to apply approved patches.
5. Re-scan to verify compliance after remediation.

This workflow helps you separate detection from enforcement and reduces the risk of unexpected downtime.

**AWS Systems Manager commands and examples**

You can run patch operations with the AWS CLI using Systems Manager Run Command. The document name differs by platform, but the behavior is the same: scan reports, install remediates.

```bash
aws ssm send-command \
  --document-name "AWS-RunPatchBaseline" \
  --targets "Key=InstanceIds,Values=i-0123456789abcdef0" \
  --parameters "Operation=Scan" \
  --comment "Patch compliance scan"
```

```bash
aws ssm send-command \
  --document-name "AWS-RunPatchBaseline" \
  --targets "Key=InstanceIds,Values=i-0123456789abcdef0" \
  --parameters "Operation=Install" \
  --comment "Install approved patches"
```

For a Linux instance, you may also see patching behavior controlled by the Systems Manager Agent and the underlying package manager. Patch Manager orchestrates the process, but the OS package tools perform the actual installation.

**Practical interpretation of results**

A scan result usually tells you:

- Which patches are missing
- Which patches are approved but not installed
- Whether the instance is compliant with the baseline
- Whether a reboot is pending or recommended after installation

An install result usually tells you:

- Which approved patches were applied
- Which patches failed to install
- Whether the instance needs a reboot to finish patching

If a patch fails during install, the instance may remain partially updated. In that case, a follow-up scan helps you confirm the remaining gaps.

**Reboot considerations**

Some patches require a reboot to complete. Patch Manager can be configured to reboot automatically when needed, or you can defer rebooting until a controlled window. The right choice depends on workload sensitivity and maintenance policy.

- Use automatic reboot when the service can tolerate interruption and you want patch completion in one run.
- Use deferred reboot when you need to coordinate failover, clustering, or application shutdown.
- Always verify post-reboot compliance with another scan.

**Common exam-relevant scenarios**

- You need to prove patch compliance without changing production systems: use `Scan`.
- You need to remediate missing security updates during a maintenance window: use `Install`.
- You want to detect drift before a change freeze: run a scan first.
- You need to confirm whether a baseline is too restrictive: compare scan results against approved rules.

**Troubleshooting and operational tips**

If scan results do not match expectations, check these items:

- The instance is managed by Systems Manager and has the correct IAM permissions.
- The instance has network access to Systems Manager endpoints.
- The patch baseline is associated with the instance or patch group.
- The operating system and repository metadata are current.
- The patch operation ran with the intended baseline and parameters.

If install does not apply patches, common causes include:

- The patch is not approved by the baseline
- The package repository is unavailable
- A dependency or lock conflict exists
- The instance requires a reboot before additional patches can be installed

**Why this matters for SOA-C02**

For SysOps, the exam often tests whether you can choose the right operational action. Patch Manager scan versus install is a classic example of assessment versus remediation. If the requirement is visibility, choose scan. If the requirement is change, choose install. That distinction is central to safe operations, compliance reporting, and maintenance planning.
