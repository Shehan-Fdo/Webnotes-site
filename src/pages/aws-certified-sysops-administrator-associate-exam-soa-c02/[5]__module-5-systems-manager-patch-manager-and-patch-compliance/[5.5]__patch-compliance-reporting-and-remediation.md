## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Patch compliance overview**

AWS Systems Manager Patch Manager helps you assess and remediate missing operating system and application patches across managed nodes. Patch compliance is the record of whether a node meets the rules defined in its patch baseline. You use it to answer two different questions:

- Which patches are missing or noncompliant?
- Which instances need remediation, and how do you prove that remediation happened?

Patch compliance is not the same as inventory. Inventory tells you what software, packages, and instance metadata exist. Patch compliance tells you whether the installed patch state matches the baseline and scan results. In exam scenarios, distinguish these data sources carefully because they solve different operational problems.

**Core patching concepts**

Patch Manager evaluates managed nodes against a **patch baseline**. A baseline defines which patches are approved, rejected, or auto-approved for a given operating system. The baseline is then associated with a **patch group** or used as the default baseline for that operating system.

Key terms:

- **Patch baseline**: The approval policy for patches on a platform.
- **Patch group**: A tag-based grouping mechanism used to target a baseline to a set of instances.
- **Managed node**: An instance or server managed by Systems Manager.
- **Compliance item**: A record that represents a compliance result for a resource and category.
- **Remediation**: The act of installing approved missing patches.

A patch baseline commonly uses rules such as:

- Approve patches by classification or severity.
- Approve patches after a delay using `ApproveAfterDays`.
- Approve patches until a cutoff date using `ApproveUntilDate`.
- Explicitly approve or reject specific patch IDs.

Example baseline creation for Linux:

```bash
aws ssm create-patch-baseline \
  --name "Linux-Prod-Baseline" \
  --operating-system AMAZON_LINUX_2 \
  --description "Approved patches for production Linux" \
  --approval-rules '{
    "PatchRules": [
      {
        "PatchFilterGroup": {
          "PatchFilters": [
            {"Key":"CLASSIFICATION","Values":["Security"]},
            {"Key":"SEVERITY","Values":["Critical","Important"]}
          ]
        },
        "ApproveAfterDays": 7
      }
    ]
  }'
```

This example shows the structure of a real baseline rule: you approve patches based on filters and timing, not by setting a top-level compliance field.

**How compliance is generated**

Patch compliance is produced when Systems Manager scans a node against a baseline. A scan does not install patches; it only evaluates the current state. A remediation run installs approved missing patches.

Typical workflow:

1. You define a patch baseline.
2. You associate the baseline with a patch group or use the default baseline for the operating system.
3. You run a scan to collect compliance data.
4. You review missing patches and noncompliant nodes.
5. You run remediation to install approved patches.
6. You rescan to confirm compliance.

The Systems Manager document used for patching is `AWS-RunPatchBaseline`. You use it with Run Command or a maintenance window to perform either scan or install operations.

Example Run Command scan:

```bash
aws ssm send-command \
  --document-name "AWS-RunPatchBaseline" \
  --targets "Key=tag:PatchGroup,Values=ProdLinux" \
  --parameters "Operation=Scan" \
  --comment "Patch compliance scan for production Linux"
```

Example Run Command install:

```bash
aws ssm send-command \
  --document-name "AWS-RunPatchBaseline" \
  --targets "Key=tag:PatchGroup,Values=ProdLinux" \
  --parameters "Operation=Install" \
  --comment "Patch remediation for production Linux"
```

**Viewing patch compliance in the console**

In the Systems Manager console, you typically review patch compliance from the Patch Manager or Compliance views. You can filter results by:

- Patch group
- Operating system
- Compliance status
- Severity
- Instance ID
- Patch baseline

This is useful when you need to isolate a subset of the fleet, such as all critical Linux instances in a production patch group. The console view is operationally helpful for quick triage, but audit workflows often require exported reports or CLI-based aggregation.

**CLI reporting and audit output**

For audit and fleet review, you can query compliance data with the Systems Manager and Compliance APIs. Common patterns include listing compliance items, summarizing compliance status, and retrieving patch state for a managed node.

Useful commands:

```bash
aws ssm list-compliance-items \
  --resource-types ManagedInstance \
  --filters Key=ComplianceType,Values=Patch
```

```bash
aws ssm list-compliance-summaries \
  --filters Key=ComplianceType,Values=Patch
```

```bash
aws ssm describe-instance-patch-states \
  --instance-ids i-0123456789abcdef0
```

```bash
aws ssm describe-instance-patches \
  --instance-id i-0123456789abcdef0 \
  --filters Key=State,Values=Missing
```

What these outputs help you determine:

- `list-compliance-items` shows compliance records for patching.
- `list-compliance-summaries` gives an aggregated view of compliance counts.
- `describe-instance-patch-states` summarizes patch state for a node.
- `describe-instance-patches` lists individual patches and their state.

For exam purposes, remember that compliance summaries are aggregated views, while patch state and patch lists are more granular. If you need evidence for an audit, you usually combine summary data with detailed node-level results.

**Reports and fleet-wide review**

Patch compliance reporting is often used for recurring audit evidence. In practice, you may generate reports from Systems Manager compliance data, then export or analyze them in other services. A common pattern is to send compliance data to Amazon S3 or query it through AWS-native reporting tools, depending on the organization’s setup.

Useful report-oriented outputs include:

- Compliance summaries by patch type
- Node-level patch state
- Missing patch lists by instance
- Remediation history over repeated scans

If you need a fleet-wide view, be careful about scope:

- Systems Manager compliance is naturally scoped to the account and Region where the managed nodes exist.
- Cross-account or cross-Region aggregation is not automatic.
- Centralized reporting usually requires additional setup, such as organization-wide collection, data export, or a separate reporting pipeline.

This distinction matters on the exam because a “global compliance dashboard” is not the default behavior of Patch Manager.

**Patch compliance trends and history**

Compliance is not a one-time snapshot. Repeated scans create a history of how the fleet changes over time. This is important for:

- Measuring remediation effectiveness
- Tracking recurring drift
- Proving patch SLAs
- Identifying instances that repeatedly fail to patch

A practical operational cycle looks like this:

- Scan on a schedule.
- Remediate approved missing patches.
- Rescan after the maintenance window.
- Compare compliance counts over time.

If compliance does not improve after remediation, investigate:

- Unsupported packages
- Reboot requirements
- Repository access problems
- Baseline approval rules that exclude the patch
- Instances that were offline during the maintenance window

**Maintenance windows for remediation**

Maintenance windows let you schedule patching during approved time periods. This is the standard way to automate patch remediation at scale. When you register a patch task in a maintenance window, you usually use `AWS-RunPatchBaseline` as the task document.

Example maintenance window task:

```bash
aws ssm register-task-with-maintenance-window \
  --window-id mw-0abc123def4567890 \
  --targets "Key=WindowTargetIds,Values=target-1234567890abcdef0" \
  --task-type RUN_COMMAND \
  --task-arn "AWS-RunPatchBaseline" \
  --service-role-arn "arn:aws:iam::123456789012:role/SSMMaintenanceWindowRole" \
  --priority 1 \
  --max-concurrency "10" \
  --max-errors "1" \
  --task-invocation-parameters '{
    "RunCommand": {
      "Parameters": {
        "Operation": ["Install"]
      }
    }
  }'
```

Important operational settings:

- **Priority** controls execution order when multiple tasks exist.
- **Max concurrency** limits how many nodes patch at once.
- **Max errors** defines how many failures you tolerate before stopping or marking the task unhealthy.
- **Service role** allows the maintenance window to invoke Systems Manager actions.

These settings matter in large fleets because they reduce blast radius and help you avoid patching every node simultaneously.

**Patch groups and baseline association**

Patch groups let you target different baselines to different fleets, such as production versus development. You tag instances with a patch group value and associate that value with a baseline.

Example association:

```bash
aws ssm register-patch-baseline-for-patch-group \
  --baseline-id pb-0abc123def4567890 \
  --patch-group "ProdLinux"
```

To set a default baseline for a platform, use the default baseline registration command for that operating system context. The exact behavior depends on the platform and baseline type, so do not assume a patch group is the only way to select a baseline.

Operationally:

- Use patch groups when you need different approval policies for different fleets.
- Use default baselines when you want a platform-wide fallback policy.
- Verify which baseline is active before scanning or installing, because compliance results depend on the baseline used for the operation.

**Interpreting compliance results**

Patch compliance results usually include:

- Missing patches
- Installed patches
- Approved but not yet installed patches
- Rejected patches
- Unknown or unsupported items

Edge cases to watch:

- **Superseded patches** may not appear as missing if a newer approved patch replaces them.
- **Unsupported packages** may not be patchable through Patch Manager.
- **Reboot-required patches** may show as installed only after the reboot completes and the node reports back.
- **Offline nodes** cannot be scanned or remediated until they reconnect.
- **Repository issues** can cause install failures even when the baseline approves the patch.

A patch can be approved in the baseline but still not appear as installed if the package manager cannot resolve it on the node. That is why compliance reporting and remediation troubleshooting often require both Systems Manager data and OS-level package logs.

**Practical troubleshooting flow**

When a node remains noncompliant after remediation, check the problem in layers:

1. Confirm the node is a managed node and is online.
2. Confirm the correct patch group or default baseline is in use.
3. Confirm the baseline approves the patch.
4. Confirm the maintenance window or Run Command executed successfully.
5. Check the instance package manager logs for install errors.
6. Rescan to refresh compliance data.

Example Linux log locations:

```bash
/var/log/amazon/ssm/amazon-ssm-agent.log
/var/log/yum.log
/var/log/dnf.log
```

Example Windows log locations:

```text
C:\ProgramData\Amazon\SSM\Logs\amazon-ssm-agent.log
C:\Windows\Logs\CBS\CBS.log
```

**Common exam distinctions**

Patch compliance is often confused with other Systems Manager features. Keep these distinctions clear:

- **Patch compliance vs inventory**
  - Inventory reports what is installed.
  - Patch compliance reports whether installed software meets patch policy.

- **Patch compliance vs State Manager**
  - State Manager enforces desired state associations.
  - Patch Manager evaluates and remediates patch state.

- **Patch compliance vs AWS Config**
  - AWS Config tracks configuration drift for supported resources.
  - Patch Manager tracks patch status on managed nodes.

- **Patch compliance vs Security Hub**
  - Security Hub aggregates security findings.
  - Patch Manager reports patch status and remediation results.

**Practical audit pattern**

A common audit-ready workflow is:

1. Run a scheduled scan across the fleet.
2. Review compliance summaries by patch group and severity.
3. Export or store the results for evidence.
4. Remediate missing critical and security patches during a maintenance window.
5. Rescan and compare the new compliance state with the prior run.
6. Retain the history for trend analysis and SLA reporting.

This gives you both point-in-time evidence and trend evidence, which is often more useful than a single compliance snapshot.

```bash
# Remediate approved missing patches
aws ssm send-command \

aws ssm list-compliance-summaries \

```

**Exam takeaways**

- Use `AWS-RunPatchBaseline` for scan and install operations.
- Use patch groups to map fleets to baselines.
- Use maintenance windows for controlled, repeatable remediation.
- Use compliance summaries for audit-style reporting and detailed patch state for troubleshooting.
- Remember that compliance is account- and Region-scoped unless you build additional aggregation.
- Distinguish patch compliance from inventory and other Systems Manager compliance-related features.
