## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Patch groups in Systems Manager**

AWS Systems Manager uses **patch groups** to map a set of managed nodes to a specific patch baseline. The mapping is driven by a tag key/value pair on the instance and a registered patch group name in Systems Manager. In practice, you tag instances with the patch group value you want, then register that same value against a baseline.

A patch group is not a separate resource type on the instance. It is a coordination mechanism between:

- the instance tag value
- the patch baseline registration
- the operating system family of the managed node

This is useful when different environments need different patch approval rules, such as production versus development, or Linux versus Windows workloads with different maintenance requirements.

**How patch group selection works**

Systems Manager determines the baseline at execution time by checking the instance’s current tag value and looking for a matching patch group registration. If a match exists, Systems Manager uses that baseline for patching and compliance evaluation. If no patch-group-specific baseline exists, Systems Manager falls back to the default baseline for that operating system. If neither exists, patching or compliance evaluation can fail because there is no baseline to evaluate against.

- **Instance tag**: identifies the patch group membership on the managed node.
- **Registered patch group**: links a patch group name to a patch baseline.
- **Default baseline**: used when no patch group mapping exists for that OS.
- **Execution-time lookup**: the current tag value matters when the patch operation runs.

> The patch group name must exactly match the instance tag value you intend to use for that baseline association.

**Tagging strategy**

A good tagging strategy keeps patching predictable and auditable. Use a small, controlled set of tag values that represent patch policy, not just arbitrary labels. For example, `ProdLinux`, `DevLinux`, and `ProdWindows` are clearer than free-form values that drift over time.

Recommended practices:

- Use a consistent tag key for patch group membership, commonly `Patch Group`.
- Keep tag values stable and environment-oriented.
- Separate patch policy from other operational tags such as `Environment`, `Owner`, or `CostCenter`.
- Apply tags through automation so new instances inherit the correct patch group immediately.
- Review tag governance so users cannot create uncontrolled patch group values.

A practical pattern is to use one tag key for patch policy and other keys for inventory and ownership:

```text
Patch Group=ProdLinux
Environment=Production
Owner=PlatformTeam
PatchWindow=Sun-0200
```

**Registering a patch baseline to a patch group**

You associate a patch baseline with a patch group using `register-patch-baseline-for-patch-group`. The command requires both the baseline ID and the patch group name. The patch group name must match the instance tag value exactly.

```bash
aws ssm register-patch-baseline-for-patch-group \
  --baseline-id pb-0abc1234def567890 \
  --patch-group ProdLinux
```

Typical workflow:

1. Create or choose a patch baseline for the OS.
2. Register the baseline to the patch group name.
3. Tag the managed instances with the same patch group value.
4. Run patch scans or installs through a maintenance window or Run Command.

You can verify baseline associations with:

```bash
aws ssm describe-patch-groups
aws ssm describe-patch-baselines
```

**Default baseline versus patch group baseline**

Systems Manager uses a clear precedence model:

- If the instance tag value matches a registered patch group, that baseline is used.
- If no patch-group mapping exists, Systems Manager uses the default baseline for that OS.
- If no default baseline exists either, patching and compliance evaluation may not have a valid baseline.

This matters because patch compliance is not created by the association alone. Compliance is reported after a scan or install operation compares the instance state to the approved baseline.

**Maintenance windows and patching**

Maintenance windows are the common way to schedule patching at scale. The maintenance window task does not choose the baseline by targeting the patch group tag directly. Instead, the baseline is selected from the instance’s patch group mapping when the patch document runs.

A typical patching setup uses:

- a maintenance window for timing
- a target set of managed instances
- the `AWS-RunPatchBaseline` document
- a patch baseline selected through patch group association

Example maintenance window task registration for a Run Command task:

```bash
aws ssm register-task-with-maintenance-window \
  --window-id mw-0abc1234def567890 \
  --task-type RUN_COMMAND \
  --task-arn AWS-RunPatchBaseline \
  --targets Key=tag:Patch Group,Values=ProdLinux \
  --service-role-arn arn:aws:iam::123456789012:role/MaintenanceWindowRole \
  --max-concurrency "10" \
  --max-errors "1" \
  --priority 1
```

Use this pattern carefully:

- The maintenance window target set controls which instances receive the task.
- The patch baseline used by `AWS-RunPatchBaseline` comes from the instance’s patch group mapping.
- The target tag should be used to select the right fleet, not to replace baseline registration.

> Do not assume the command target alone determines the patch baseline. The baseline comes from the registered patch group association.

**Run Command patching**

You can also invoke patching directly with `AWS-RunPatchBaseline`. This is useful for ad hoc patch scans or installs, but the same baseline-selection rules apply.

```bash
aws ssm send-command \
  --document-name AWS-RunPatchBaseline \
  --targets "Key=tag:Patch Group,Values=ProdLinux" \
  --parameters '{"Operation":["Scan"]}'
```

Common operations:

- `Scan` to evaluate compliance without installing patches.
- `Install` to apply approved patches.

Example install operation:

```bash
aws ssm send-command \
  --document-name AWS-RunPatchBaseline \
  --targets "Key=tag:Patch Group,Values=ProdLinux" \
  --parameters '{"Operation":["Install"]}'
```

Use Run Command when you need immediate execution, but prefer maintenance windows for repeatable patch governance.

**Compliance and reporting**

Patch compliance is based on the results of a scan or install compared against the approved baseline. The association itself does not generate compliance data until Systems Manager runs a patch operation.

Important points:

- A successful scan reports missing patches relative to the baseline.
- An install operation updates the instance and then can be followed by another scan.
- Compliance views are most useful after scheduled patching runs.
- Baseline association alone does not prove the instance is compliant.

Useful CLI examples:

```bash
aws ssm list-compliance-items \
  --resource-id mi-0123456789abcdef0 \
  --resource-types ManagedInstance \
  --filters Key=ComplianceType,Values=Patch
```

```bash
aws ssm describe-instance-patch-states \
  --instance-ids mi-0123456789abcdef0
```

The exact compliance query you need depends on whether you want instance-level patch state, compliance items, or baseline associations.

**Operational design patterns**

A strong patching design separates fleets by patch policy and lifecycle stage. This reduces accidental overlap and makes approvals easier to reason about.

Common patterns:

- **Environment-based groups**: `Dev`, `Test`, `Prod`
- **OS-based groups**: `ProdLinux`, `ProdWindows`
- **Risk-based groups**: `Critical`, `Standard`, `Delayed`
- **Region-based groups**: `ProdLinux-us-east-1`, if regional policy differs

A simple topology might look like this:

```text
Managed Instance
  └─ Tag: Patch Group=ProdLinux
       └─ Registered to baseline: pb-0abc1234def567890
            └─ Used by: AWS-RunPatchBaseline in maintenance window mw-0abc1234def567890
```

This model helps you:

- apply different approval rules to different fleets
- stage patches before production rollout
- keep compliance reporting aligned with business risk
- avoid one-size-fits-all patching

**Automation and governance**

Automate tag assignment whenever possible. If instances are launched through Auto Scaling groups, launch templates, or configuration management, ensure the patch group tag is applied at creation time. This prevents unmanaged instances from missing their baseline mapping.

Practical controls:

- enforce tags through infrastructure as code
- use IAM policies or tag policies to limit invalid values
- validate tags in CI/CD pipelines
- monitor for instances missing the patch group tag

Example launch template tag specification:

```json
{
  "TagSpecifications": [
    {
      "ResourceType": "instance",
      "Tags": [
        { "Key": "Patch Group", "Value": "ProdLinux" },
        { "Key": "Environment", "Value": "Production" }
      ]
    }
  ]
}
```

**Troubleshooting checklist**

When patching does not behave as expected, check the mapping in this order:

1. Confirm the instance is a managed node in Systems Manager.
2. Verify the instance has the expected `Patch Group` tag key and value.
3. Confirm the patch group value is registered to the intended baseline.
4. Check whether a default baseline exists for the OS.
5. Review the maintenance window target set or Run Command target selection.
6. Run a scan and inspect the resulting compliance state.

Common failure modes:

- The instance tag value does not exactly match the registered patch group name.
- The baseline exists, but it is registered to a different patch group.
- The instance is targeted for patching, but no baseline is available for that OS.
- The maintenance window runs against the wrong fleet because the target tag is too broad.
- The instance was not managed by Systems Manager at the time of execution.

**Exam focus**

For exam scenarios, remember these distinctions:

- **Tagging** selects membership in a patch group.
- **Baseline registration** binds that patch group to a baseline.
- **Maintenance windows** schedule patch execution.
- **Run Command** can execute patching on demand.
- **Compliance** comes from scan/install results, not from the association alone.

If a question asks how to apply different patch policies to different groups of instances, the best answer usually involves:

- tagging instances with a patch group value
- registering each patch group to a specific baseline
- using maintenance windows or Run Command to execute `AWS-RunPatchBaseline`
