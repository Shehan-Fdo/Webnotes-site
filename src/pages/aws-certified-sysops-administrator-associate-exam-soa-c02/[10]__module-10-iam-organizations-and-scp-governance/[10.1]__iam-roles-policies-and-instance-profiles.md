## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**IAM authorization model**

AWS Identity and Access Management (IAM) controls *who* can do *what* on *which* resources. For exam scenarios, separate the concepts of authentication and authorization:

- **Authentication** proves identity, such as a user signing in or a service assuming a role.
- **Authorization** is the policy decision that allows or denies an action.
- **Temporary credentials** are usually issued by AWS Security Token Service (STS) when a role is assumed.

A role is not a long-term identity like an IAM user. It is an assumable identity with permissions attached, designed for workloads, cross-account access, federated access, and temporary elevated access.

**Core IAM building blocks**

- **IAM user**: A long-term identity for a person or application, usually avoided for workloads when roles are possible.
- **IAM group**: A collection of users that share permissions through attached policies.
- **IAM role**: An identity with permissions that can be assumed by trusted principals.
- **IAM policy**: A JSON document that defines permissions.
- **Instance profile**: The EC2-specific container that lets an instance use an IAM role.

A common exam trap is confusing a role with an instance profile. The role is the permission identity; the instance profile is the EC2 delivery mechanism that makes the role available to the instance metadata service.

**Policy types and where they apply**

IAM authorization often depends on multiple policy types working together:

- **Identity-based policies** attach to users, groups, or roles and say what the identity can do.
- **Resource-based policies** attach to resources such as S3 buckets, KMS keys, SQS queues, SNS topics, and some Secrets Manager resources.
- **Permissions boundaries** set the maximum permissions an IAM user or role can ever receive from identity-based policies.
- **Service control policies (SCPs)** in AWS Organizations set account-level guardrails, but they do not grant permissions by themselves.

The evaluation result is always the intersection of what is allowed and what is not explicitly denied. An explicit `Deny` wins over any `Allow`.

**Policy evaluation logic**

The exact interplay between identity-based and resource-based policies matters for exam questions.

- In the **same account**, an action is allowed when the identity-based policy allows it and no explicit deny blocks it.
- A **resource-based policy** can also grant access directly to a principal, sometimes without needing an identity-based allow in the same way as cross-account access.
- In **cross-account** scenarios, you usually need both sides to cooperate:
  - The caller’s identity policy must allow the action.
  - The target resource policy must trust or allow the caller, or the caller must assume a role in the target account.

A practical way to think about it:

- **Same-account access** often works with either identity-based permission alone or a resource policy alone, depending on the service and principal type.
- **Cross-account access** usually requires an explicit trust path, either by assuming a role or by a resource policy that names the external principal.

> If you see an exam question about cross-account S3 or KMS access, check whether the design uses **role assumption** or **direct resource policy access**. The preferred answer depends on whether the external principal should operate as a temporary identity in the target account or be granted direct access to the resource.

**IAM policy structure**

A policy document contains statements, and each statement can include several common elements:

- **Effect**: `Allow` or `Deny`.
- **Action**: The API operations permitted or denied.
- **Resource**: The ARN(s) the statement applies to.
- **Principal**: Who is trusted or allowed, used in resource-based policies and trust policies.
- **Condition**: Context-based constraints such as source IP, MFA, VPC endpoint, or tag matching.
- **NotAction**: Matches all actions except the listed ones.
- **NotResource**: Matches all resources except the listed ones.

Example identity policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadOnlyS3Bucket",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::example-bucket",
        "arn:aws:s3:::example-bucket/*"
      ]
    }
  ]
}
```

Example trust policy for a role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowEC2AssumeRole",
      "Effect": "Allow",
      "Principal": {
        "Service": "ec2.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

**Conditions and common operators**

Conditions are essential for least privilege and for exam questions about contextual access control.

- **StringEquals**: Exact string match.
- **StringLike**: Pattern match with wildcards.
- **ArnEquals**: Exact ARN match.
- **Bool**: Boolean condition, often used for MFA.
- **IpAddress**: Restrict by source IP range.
- **Null**: Test whether a key is present.
- **ForAllValues** and **ForAnyValue**: Evaluate multi-value condition keys.

Common condition keys include:

- `aws:SourceIp`
- `aws:PrincipalArn`
- `aws:RequestedRegion`
- `aws:MultiFactorAuthPresent`
- `aws:PrincipalTag/<tag-key>`
- `aws:ResourceTag/<tag-key>`

Example with a condition:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowOnlyFromCorporateNetwork",
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::example-bucket/*",
      "Condition": {
        "IpAddress": {
          "aws:SourceIp": "203.0.113.0/24"
        }
      }
    }
  ]
}
```

**Permissions boundaries**

A permissions boundary is a guardrail for an IAM user or role. It does not grant permissions; it limits the maximum permissions that identity-based policies can provide.

Key exam distinction:

- **Applies to IAM users and roles**
- **Does not apply to resource-based policies in the same way**
- **Does not replace explicit denies**
- **Useful for delegated administration**

A realistic scenario:

- A developer can create IAM roles for an application.
- You attach a permissions boundary that allows only `s3`, `logs`, and `cloudwatch` actions.
- Even if the developer attaches a broader identity policy, the boundary prevents the role from exceeding the approved scope.

Example boundary pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "BoundaryForAppRoles",
      "Effect": "Allow",
      "Action": [
        "s3:*",
        "logs:*",
        "cloudwatch:*"
      ],
      "Resource": "*"
    }
  ]
}
```

**Role assumption and STS session duration**

When a principal assumes a role, STS issues temporary credentials with an expiration time. The session duration affects how long the credentials remain valid.

Important points:

- The role has a **maximum session duration** configured on the role.
- The caller can request a shorter session, but not longer than the role allows.
- Temporary credentials expire automatically.
- Long-running automation must refresh credentials before expiration.

Role chaining has a stricter limit:

- When one role assumes another role, the session duration is typically capped at **1 hour**.
- This matters in multi-hop access patterns and federated workflows.
- If an exam question mentions a chain of role assumptions and a long-running task, check for session expiration as the root cause.

Example STS call:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/AppReadOnlyRole \
  --role-session-name app-session \
  --duration-seconds 3600
```

The returned credentials are temporary and must be exported or injected into the calling process.

**IAM roles versus instance profiles**

An IAM role is the permission set. An instance profile is the EC2 wrapper that holds exactly one role for use by an instance.

Think of it this way:

- **Role**: the permissions and trust policy
- **Instance profile**: the container attached to EC2
- **IMDS**: Instance Metadata Service, the local endpoint the instance uses to retrieve credentials

A role can exist without being attached to EC2. An instance profile exists to make a role usable by EC2.

Simple relationship diagram:

```text
IAM Role
  └─ trust policy allows ec2.amazonaws.com
  └─ permissions policy grants S3/CloudWatch/etc.
        ↓
Instance Profile
  └─ contains one IAM role
        ↓
EC2 Instance
  └─ retrieves temporary credentials from IMDS
```

This distinction is a frequent exam trap because the console often hides the difference. In the AWS Management Console, you usually attach an instance profile to an instance, but the underlying permission model is still role-based.

**EC2 credential delivery and why order matters**

An EC2 instance can obtain credentials from several sources, and the order matters during troubleshooting.

Typical provider chain concepts:

- Environment variables can override other sources.
- Shared credentials files may be used by local tools.
- EC2 instance metadata credentials are used when the workload runs on EC2 with an attached role.
- SDKs and CLI tools resolve credentials in a precedence order.

Why this matters:

- If a script works locally but fails on EC2, the instance may be using the wrong credential source.
- If stale environment variables exist, they can mask the instance role.
- If IMDS is disabled or blocked, the instance cannot retrieve role credentials.

Example on EC2:

```bash
aws sts get-caller-identity
aws s3 ls s3://example-bucket
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/
```

If the metadata endpoint returns the role name, the instance profile is attached and IMDS is reachable.

**Attaching and detaching an instance profile**

You often need to update the role used by an existing EC2 instance. In the console, you edit the IAM role association for the instance. In the API and CLI, you work with the instance profile association.

Console workflow:

1. Open the EC2 instance details.
2. Choose **Actions** > **Security** > **Modify IAM role**.
3. Select the desired role.
4. Save the change and verify the instance can retrieve new credentials.

CLI workflow:

```bash
aws ec2 describe-iam-instance-profile-associations
```

Attach or replace an instance profile association:

```bash
aws ec2 associate-iam-instance-profile \
  --instance-id i-0123456789abcdef0 \
  --iam-instance-profile Name=AppInstanceProfile
```

If the instance already has one, you may need to replace it:

```bash
aws ec2 replace-iam-instance-profile-association \
  --association-id iip-assoc-0abc123def4567890 \
  --iam-instance-profile Name=AppInstanceProfile
```

Detach the profile:

```bash
aws ec2 disassociate-iam-instance-profile \
  --association-id iip-assoc-0abc123def4567890
```

**Cross-account access patterns**

There are two common ways to access resources across accounts.

| Pattern | How it works | When to prefer it |
|---|---|---|
| **Assume role** | The caller assumes a role in the target account and gets temporary credentials there | Best for broad access, operational tasks, and when you want the caller to act as a principal in the target account |
| **Direct resource policy access** | The resource policy names the external principal directly | Best for services like S3 or KMS when you want direct access without role assumption |

For S3:

- A bucket policy can allow an external account’s role or user directly.
- A role assumption model is often cleaner when the external party needs multiple actions across many resources.

For KMS:

- Key policies are central because KMS authorization is policy-driven and often requires both key policy and identity policy alignment.
- Cross-account KMS access commonly uses a combination of key policy and IAM permissions, or a role in the target account.

Exam guidance:

- Choose **assume role** when the question emphasizes temporary credentials, centralized control, or operating inside the target account.
- Choose **resource policy** when the question emphasizes direct access to a specific resource, especially S3 bucket access or KMS key usage.

**Resource-based policy examples**

S3 bucket policy allowing a role from another account:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCrossAccountRead",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::210987654321:role/ExternalAppRole"
      },
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::example-bucket",
        "arn:aws:s3:::example-bucket/*"
      ]
    }
  ]
}
```

KMS key policy conceptually must allow the external principal or the role that will use the key. In practice, KMS often requires careful alignment between the key policy and the caller’s IAM permissions.

**Troubleshooting and validation tools**

Use AWS-native tools to validate policy behavior before production changes.

- **IAM Policy Simulator**: Test whether a principal can perform an action on a resource.
- **Access Analyzer**: Identify unintended external access to resources and validate resource policies.
- **CloudTrail**: Inspect denied API calls and confirm which principal attempted the action.
- **`sts get-caller-identity`**: Verify which identity the CLI is actually using.

Example validation flow:

```bash
aws sts get-caller-identity
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/AppRole \
  --action-names s3:GetObject \
  --resource-arns arn:aws:s3:::example-bucket/test.txt
```

Use Access Analyzer when you need to answer questions like:

- Is this bucket policy exposing data outside the account?
- Does this KMS key policy allow unintended principals?
- Which external accounts can access this resource?

**Common exam traps**

- **Role vs instance profile**: The role is the permission identity; the instance profile is the EC2 attachment container.
- **Allow is not enough**: An explicit deny anywhere in the evaluation path overrides allows.
- **Boundary confusion**: A permissions boundary limits identity permissions; it is not a substitute for a resource policy.
- **Cross-account assumptions**: Direct access and role assumption are not interchangeable.
- **Session expiration**: Temporary credentials expire, and role chaining shortens usable duration.
- **Policy scope**: `NotAction` and `NotResource` can create broad permissions if paired with a wide resource scope.

**Practical EC2 role setup example**

A common pattern is an EC2 instance that needs read access to one S3 bucket and write access to CloudWatch Logs.

1. Create a role trusted by EC2.
2. Attach a least-privilege policy.
3. Create an instance profile containing the role.
4. Attach the instance profile to the EC2 instance.
5. Verify access from the instance.

Example permissions policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadAppBucket",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::app-data",
        "arn:aws:s3:::app-data/*"
      ]
    },
    {
      "Sid": "WriteLogs",
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "*"
    }
  ]
}
```

Verification from the instance:

```bash
aws sts get-caller-identity
aws s3 ls s3://app-data
aws logs describe-log-groups
```

**How to reason through exam questions**

When you see an IAM question, ask these in order:

1. Who is the principal?
2. Is the access same-account or cross-account?
3. Is the permission coming from an identity policy, resource policy, or both?
4. Is there a permissions boundary or SCP limiting the result?
5. Is the principal using temporary credentials from STS?
6. Is the workload on EC2 using an instance profile and IMDS?

That sequence usually reveals the correct answer faster than memorizing isolated facts.

**Quick reference**

- **Role**: assumable identity with permissions and trust policy.
- **Instance profile**: EC2 container for exactly one role.
- **STS**: issues temporary credentials for assumed roles.
- **Permissions boundary**: maximum permissions for a user or role.
- **Resource policy**: grants access directly to a resource.
- **Policy simulator**: tests authorization outcomes.
- **Access Analyzer**: finds unintended external access.
