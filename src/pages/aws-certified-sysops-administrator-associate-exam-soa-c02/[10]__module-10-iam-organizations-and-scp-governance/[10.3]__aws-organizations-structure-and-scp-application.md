## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**AWS Organizations hierarchy**

AWS Organizations lets you centrally manage multiple AWS accounts under a single organization. The hierarchy matters because policy inheritance follows the tree from the root down through organizational units (OUs) to accounts.

- **Organization**: The top-level container for all accounts.
- **Root**: The top of the hierarchy inside an organization; every account and OU belongs somewhere under a root.
- **Organizational unit (OU)**: A grouping mechanism for accounts and nested OUs.
- **Account**: The leaf node where workloads run and where effective permissions are evaluated.

A common governance pattern is to place broad guardrails at the root, then apply narrower controls to OUs for workload-specific restrictions. This reduces blast radius because you can isolate environments such as `prod`, `dev`, and `sandbox` with different policy sets.

**Policy types in AWS Organizations**

AWS Organizations supports more than service control policies (SCPs). For exam scenarios, distinguish the policy types because they solve different problems.

- **SCPs**: Define the maximum permissions available to accounts in the organization or OU path.
- **Tag policies**: Standardize tag keys and values across accounts for governance and cost allocation.
- **Backup policies**: Centralize backup plan assignment and retention governance for supported resources.
- **AI services opt-out policies**: Control whether accounts can use certain AWS AI services for data processing, depending on organization settings.

SCPs are the main access-control topic, but tag and backup policies often appear in broader governance questions. They do not grant permissions to users or roles; they enforce organizational standards.

**How SCP evaluation works**

SCPs are guardrails, not identity policies. They do not grant access by themselves. Instead, they limit what IAM identities and roles can do in member accounts.

The effective permission set is the intersection of:
- Identity-based policies
- Resource-based policies
- Permission boundaries
- SCPs
- Session policies, when present

For SCPs specifically:
- An action must be allowed by the applicable SCP allow path.
- Any explicit `Deny` in any applicable SCP blocks the action.
- If no applicable SCP allows an action, the action is denied by default.

This means SCPs behave like a filter on top of normal IAM authorization. Even if IAM allows an action, the SCP layer can still block it.

> **Critical:** SCPs do not override each other in a simple last-wins model. The effective result is the intersection of all applicable SCPs, and any explicit deny in any applicable SCP always wins.

**Root, OU, and account attachment behavior**

SCPs can be attached to the root, OUs, and accounts. The effective policy for an account includes all SCPs attached along its path.

- **Root attachment**: Applies to every account in the organization unless the account is moved to a different root in a multi-root scenario, which is uncommon in exam contexts.
- **OU attachment**: Applies to all accounts in that OU and nested child OUs.
- **Account attachment**: Applies only to that account.

A practical governance strategy is:
- Attach a baseline SCP at the root, often `FullAWSAccess` or an equivalent allow baseline.
- Add restrictive SCPs at OUs for environment-specific controls.
- Use account-level SCPs sparingly for exceptional cases.

This approach keeps the blast radius small. A restrictive SCP at the root affects every account, so it is harder to recover from mistakes. OU-level controls are usually safer for experimentation and staged rollout.

**Allow-list and deny-list patterns**

SCPs can be used in two common styles.

| Pattern | Behavior | Typical use |
|---|---|---|
| Allow-list | Only explicitly allowed actions are permitted | Strong governance, regulated environments |
| Deny-list | Most actions are allowed except explicit denies | Simpler rollout, broad baseline with targeted restrictions |

With an allow-list model, you usually need a baseline allow such as `FullAWSAccess` or a custom SCP that allows the required services. If you attach only a `Deny`-only SCP and no allow path exists, the account still cannot perform actions because SCPs do not grant permissions.

Example deny-only SCP:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyLeavingOrg",
      "Effect": "Deny",
      "Action": [
        "organizations:LeaveOrganization"
      ],
      "Resource": "*"
    }
  ]
}
```

This blocks the specified action, but it does not by itself authorize anything else.

**Management account behavior**

The management account has special behavior in AWS Organizations.

- It is the account that creates and manages the organization.
- Some Organizations actions are only available from the management account.
- SCPs do not restrict the management account in the same way they restrict member accounts for most service usage scenarios.

For exam purposes, remember that the management account is not treated like a normal member account. If a question asks where to perform organization administration, the answer is usually the management account.

**SCP inheritance and limits**

SCPs inherit downward through the hierarchy. If you attach an SCP to a root, every child OU and account inherits it. If you attach an SCP to an OU, all child OUs and accounts inherit it.

Important nuances:
- The effective SCP set is the combination of all SCPs attached at each level in the path.
- Explicit denies in any applicable SCP always block the action.
- The number of SCPs you can attach to a root, OU, or account is limited by AWS quotas, so design with consolidation in mind.
- The total size and number of statements matter when you build large guardrail sets.

Because of inheritance, moving an account between OUs can immediately change what it can do. That is a common exam scenario.

**Example: account move changes access**

Suppose you have:
- Root: `FullAWSAccess`
- OU `Dev`: allows broad developer services
- OU `Prod`: denies `ec2:TerminateInstances`

An account in `Dev` can terminate instances if IAM allows it. If you move that account to `Prod`, the inherited deny now blocks termination.

```bash
aws organizations move-account \
  --account-id 111122223333 \
  --source-parent-id ou-dev-aaaa1111 \
  --destination-parent-id ou-prod-bbbb2222
```

The source parent can be the current OU ID or the root ID. The destination parent must be a valid OU or the root, depending on where you are moving the account.

After the move, verify the effective SCPs on the new path before assuming access remains the same.

**Practical verification workflow**

To validate SCP effects, inspect the organization structure and the policies attached at each level.

1. Confirm the account’s current parent.
2. List SCPs attached to the root, parent OU, nested OUs, and the account.
3. Check whether `FullAWSAccess` or another allow baseline is present.
4. Look for explicit denies that match the failing action.
5. Test the action with the affected role or user and compare the result to the SCP path.

Useful CLI commands:

```bash
aws organizations describe-account --account-id 111122223333
aws organizations list-parents --child-id 111122223333
aws organizations list-policies-for-target --target-id r-abcd --filter SERVICE_CONTROL_POLICY
aws organizations list-policies-for-target --target-id ou-abcd-11111111 --filter SERVICE_CONTROL_POLICY
aws organizations list-policies-for-target --target-id 111122223333 --filter SERVICE_CONTROL_POLICY
```

If you need to inspect the policy content:

```bash
aws organizations describe-policy --policy-id p-12345678
```

In the console, you typically verify:
- The account’s placement in the hierarchy
- The SCPs attached to each parent
- The effective restrictions implied by the path

**Troubleshooting SCP failures**

When an action fails unexpectedly, check both IAM and Organizations layers.

- **IAM allows but SCP denies**: The action is blocked by organizational guardrails.
- **No allow path exists**: The action is denied even if there is no explicit deny.
- **Explicit deny exists anywhere in the path**: The action is blocked immediately.
- **Management account confusion**: The action may work in the management account but fail in a member account.

A practical checklist:
- Verify the account is in the expected OU.
- Confirm the root has the intended baseline policy.
- Confirm no parent OU has an explicit deny.
- Confirm the account itself does not have a restrictive SCP.
- Confirm the IAM principal still has permission after SCP filtering.
- Confirm the service action is not one of the AWS service operations that require broader permissions.

**Service-linked role and AWS service pitfalls**

Overly restrictive SCPs can break AWS-managed operations. This is especially important when you deny IAM or Organizations actions too aggressively.

Watch for:
- **Service-linked roles**: Many AWS services create and use service-linked roles automatically.
- **IAM actions**: Denying `iam:CreateServiceLinkedRole`, `iam:PassRole`, or related permissions can break service setup and automation.
- **Organizations actions**: Denying `organizations:*` too broadly can interfere with account management workflows.
- **CloudFormation and automation**: Stack operations may fail if the service cannot create or pass required roles.

A safer pattern is to deny only the specific risky actions you want to block, rather than broad service families, unless you have tested the impact.

Example of a targeted deny:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "PreventInstanceTermination",
      "Effect": "Deny",
      "Action": "ec2:TerminateInstances",
      "Resource": "*"
    }
  ]
}
```

This is usually safer than denying all EC2 actions, which could disrupt legitimate operations.

**Root versus OU governance strategy**

Root-level SCPs are best for organization-wide invariants. OU-level SCPs are best for environment-specific controls.

Use root-level SCPs for:
- Global security guardrails
- Organization-wide compliance requirements
- Actions that should never be allowed anywhere

Use OU-level SCPs for:
- Environment-specific restrictions
- Staged rollout of new controls
- Different policies for `prod`, `dev`, and `sandbox`

This distinction matters because root-level mistakes affect every account. OU-level mistakes affect only the accounts in that branch.

**Exam-style scenario**

A company has:
- Root with `FullAWSAccess`
- OU `Sandbox` with an SCP that denies `ec2:TerminateInstances`
- OU `Prod` with an SCP that denies `s3:DeleteBucket`
- A shared IAM role that allows both actions

If you move an account from `Sandbox` to `Prod`:
- `ec2:TerminateInstances` may become allowed if no SCP in the new path denies it and an allow path exists.
- `s3:DeleteBucket` becomes denied because the new OU inherits the `Prod` deny.

This is the key exam idea: access changes with the account’s position in the hierarchy, not just with IAM policy changes.

**Common misconceptions**

- SCPs do not grant permissions; they only limit them.
- A deny in any applicable SCP is enough to block an action.
- An allow-only SCP model still needs a valid allow path for the action.
- The management account is special and is not governed like a normal member account.
- Tag policies and backup policies are part of Organizations governance, but they are not access-control policies like SCPs.

**Quick reference**

- **SCP**: Organization-level permission guardrail.
- **Root**: Applies to all descendant OUs and accounts.
- **OU**: Best place for environment-specific guardrails.
- **Account move**: Can change inherited permissions immediately.
- **Explicit deny**: Always wins within applicable SCPs.
- **Allow baseline**: Needed when using allow-list style SCPs.

**Command examples for investigation**

```bash
# Show the account's parent
aws organizations list-parents --child-id 111122223333

# List SCPs attached to the root
aws organizations list-policies-for-target \
  --target-id r-abcd \
  --filter SERVICE_CONTROL_POLICY

# List SCPs attached to an OU
aws organizations list-policies-for-target \
  --target-id ou-abcd-11111111 \
  --filter SERVICE_CONTROL_POLICY

# Inspect a specific SCP
aws organizations describe-policy --policy-id p-12345678
```

**Key exam takeaways**

- The effective SCP result is the intersection of all applicable SCPs in the path.
- Any explicit deny in any applicable SCP blocks the action.
- If you use allow-list SCPs, you need an allow baseline such as `FullAWSAccess` or equivalent allows.
- Moving an account between OUs can immediately change access.
- Root-level policies have the widest blast radius; OU-level policies are safer for targeted governance.
- AWS Organizations includes other governance policy types, especially tag and backup policies, which may appear in scenario questions.
