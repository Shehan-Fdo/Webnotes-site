## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**SCP fundamentals and evaluation model**

<abbr title="Service control policy">SCP</abbr>s are AWS Organizations guardrails that define the maximum permissions available to accounts, organizational units (OUs), and the organization root. They do not grant permissions by themselves; they only limit what IAM permissions can ever allow.

The key exam idea is the intersection model:

- Effective access is the intersection of:
  - the SCPs attached to the account’s path in the organization
  - the IAM policy attached to the principal
  - any permission boundary on the principal
  - any session policy, if present
- If an action is not allowed by the SCP set, the request is denied even if IAM allows it.
- If an action is allowed by SCPs but denied by IAM, the request is still denied.
- If no SCP is attached at all, the account is still subject to the organization’s SCP inheritance model; in practice, AWS Organizations commonly uses a baseline allow policy such as `FullAWSAccess` to avoid unintentionally blocking everything.

A useful mental model is that SCPs set the outer ceiling, while IAM policies decide what a user or role can do inside that ceiling.

**Default behavior, allow-list designs, and implicit deny**

AWS Organizations often starts with a broad baseline SCP such as `FullAWSAccess`. That baseline means member accounts can use any service action that IAM allows, unless another SCP narrows the scope.

When you remove or replace the baseline with an allow-list SCP design, the behavior changes significantly:

- Only actions explicitly allowed by the effective SCP set are available.
- Anything not explicitly allowed is implicitly denied.
- You must include every required action, including supporting actions for the service workflow.
- This is why allow-list SCPs are usually deployed with a baseline allow SCP plus one or more narrower allow SCPs for specific OUs or accounts.

> In an allow-list model, omission is not harmless. If a required action is missing from the effective SCP set, the action is denied.

Example allow-list pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "BaselineAllowCommonServices",
      "Effect": "Allow",
      "Action": [
        "ec2:*",
        "s3:*",
        "cloudwatch:*",
        "logs:*",
        "iam:PassRole"
      ],
      "Resource": "*"
    }
  ]
}
```

This kind of SCP is only safe if it includes all services and actions the account actually needs. In real environments, you usually scope it more carefully and add separate SCPs for exceptions.

**Deny-list SCPs and guardrail use cases**

Deny-list SCPs are often easier to manage because they block only specific risky actions while leaving the rest of the organization’s baseline access intact.

Common deny-list use cases include:

- preventing account tampering, such as `organizations:LeaveOrganization`
- blocking disabling of security services, such as CloudTrail or AWS Config
- restricting destructive actions in production, such as `ec2:TerminateInstances`
- limiting use of expensive or unsupported services
- enforcing region restrictions

A deny statement in an SCP is powerful because it overrides IAM allows. This makes SCPs ideal for organization-wide guardrails.

**SCPs versus permission boundaries**

SCPs and permission boundaries are both guardrails, but they operate at different scopes and solve different problems.

| Control | Scope | What it limits | Typical use |
|---|---|---|---|
| SCP | Account, OU, organization root | Maximum permissions available in the account | Organization-wide guardrails |
| Permission boundary | IAM user or role | Maximum permissions that that principal can ever receive | Delegation inside an account |

Important interaction points:

- If an SCP denies an action, no principal in the account can perform it, regardless of IAM policy or permission boundary.
- If a permission boundary denies an action, the principal cannot perform it even if IAM and SCP allow it.
- The effective permission is the intersection of all applicable controls.
- Permission boundaries are useful when you want to delegate IAM administration inside an account without giving unrestricted power.
- SCPs are useful when you want to prevent entire accounts from using certain services or actions.

Exam nuance:

- Use SCPs to control what an account can do.
- Use permission boundaries to control what a delegated admin role inside that account can create or attach.

**Root user and IAM principal behavior in member accounts**

SCPs apply to all principals in member accounts, including the root user of the member account. That is a common exam trap.

Key points:

- The root user in a member account is not exempt from SCPs.
- IAM users and roles in the member account are also constrained by SCPs.
- If an SCP denies `ec2:TerminateInstances`, neither the root user nor an IAM role in that member account can terminate instances.
- SCPs do not apply to the management account itself, so the management account is not governed by SCPs in the same way as member accounts.

This distinction matters when troubleshooting. If an action works in the management account but fails in a member account, an SCP is a likely cause.

**Delegated administration in AWS Organizations**

Delegated administration lets you register a member account to administer a specific AWS service for the organization. This reduces the need to use the management account for day-to-day operations.

Typical delegated admin pattern:

- The management account enables trusted access for a service.
- The management account registers a member account as the delegated administrator.
- The delegated admin account performs service-specific organization-wide tasks.
- Member accounts remain under the organization’s SCP guardrails.

Common services that support delegated administration include services such as AWS Security Hub, AWS Config, AWS IAM Identity Center, AWS CloudTrail, AWS Firewall Manager, and AWS Organizations-integrated security services. Exact support varies by service and region, so verify the service documentation when designing a solution.

A practical example is central security operations:

- Management account enables trusted access for Security Hub.
- A security tooling account is registered as delegated admin.
- Security engineers use the delegated admin account to aggregate findings across the organization.
- SCPs still prevent the delegated admin account from performing actions that the organization has denied.

**Typical permissions needed for delegated admin setup**

Delegated administration usually requires two layers of authorization:

- Organizations permissions in the management account to enable trusted access and register the delegated admin
- Service-specific permissions in the delegated admin account to operate the service

Examples of Organizations actions often involved:

```bash
aws organizations enable-aws-service-access --service-principal securityhub.amazonaws.com
aws organizations register-delegated-administrator \
  --account-id 111122223333 \
  --service-principal securityhub.amazonaws.com
```

To inspect the registration state, you commonly use Organizations APIs such as:

```bash
aws organizations list-delegated-administrators --service-principal securityhub.amazonaws.com
aws organizations list-aws-service-access-for-organization
aws organizations list-policies-for-target --target-id 111122223333 --filter SERVICE_CONTROL_POLICY
aws organizations describe-policy --policy-id p-abc123example
```

For SCP troubleshooting, these APIs are more relevant than `describe-effective-policy` for SCP inspection.

**How SCPs affect delegated admin accounts**

A delegated administrator is still just a member account from the SCP perspective. That means:

- the delegated admin account can be blocked from using services or actions by SCPs
- the delegated admin account can be prevented from creating or modifying resources needed for the delegated service
- the delegated admin account may need explicit SCP exceptions for service-linked role creation, logging, or cross-account read operations

This is a frequent exam scenario: the organization wants centralized administration, but the SCP blocks a required supporting action.

Example:

- Security Hub delegated admin needs to create or manage service-linked roles.
- If the SCP denies `iam:CreateServiceLinkedRole`, the delegated admin workflow can fail.
- You may need a narrow exception for the specific service principal rather than a broad IAM exception.

**Region restriction SCPs**

Region restriction SCPs are common in exam questions because they are simple in concept but tricky in practice.

A common pattern is to deny actions outside approved regions using `aws:RequestedRegion`:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyOutsideApprovedRegions",
      "Effect": "Deny",
      "Action": "*",
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": [
            "us-east-1",
            "us-west-2"
          ]
        }
      }
    }
  ]
}
```

Important caveats:

- `aws:RequestedRegion` does not apply uniformly to every AWS service action.
- Global or regionless services may require exceptions.
- Some control-plane actions are global even when the service has regional resources.
- Exception lists are workload-specific and should be tested carefully.

A more realistic pattern often includes explicit exceptions for global services and service-linked role workflows:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyAllOutsideApprovedRegionsExceptGlobalServices",
      "Effect": "Deny",
      "NotAction": [
        "iam:*",
        "route53:*",
        "cloudfront:*",
        "organizations:*",
        "sts:*",
        "support:*"
      ],
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": [
            "us-east-1",
            "us-west-2"
          ]
        }
      }
    }
  ]
}
```

This is only an example pattern, not a universal template. In real deployments, you often need additional exceptions for services such as STS, KMS, CloudTrail, or service-linked role creation depending on the architecture.

**Troubleshooting SCP-related access failures**

When a request fails because of an SCP, the error often looks like a standard `AccessDenied` or `UnauthorizedOperation` message. The challenge is identifying that the denial came from Organizations policy evaluation rather than IAM alone.

Useful troubleshooting steps:

1. Check the error context in the service console or CLI response.
2. Review CloudTrail for the denied event and inspect the `errorCode` and `errorMessage`.
3. Confirm which account and principal made the request.
4. Review the SCPs attached to the account’s OU path.
5. Verify whether the action is explicitly allowed in the effective SCP set.
6. Check whether a permission boundary or session policy is also involved.

Example CloudTrail clues:

- `AccessDenied`
- `Client.UnauthorizedOperation`
- `explicit deny in a service control policy`
- `not authorized to perform: <action>`

Useful Organizations commands:

```bash
aws organizations list-policies-for-target \
  --target-id 111122223333 \
  --filter SERVICE_CONTROL_POLICY

aws organizations describe-policy \
  --policy-id p-abc123example
```

If you need to understand the OU path, you can walk upward from the account to the root:

```bash
aws organizations list-parents --child-id 111122223333
aws organizations list-organizational-units-for-parent --parent-id r-xxxx
```

The root ID follows the documented `r-xxxx` pattern, but the exact value is unique to your organization.

**Policy size, attachment limits, and design constraints**

SCP design is not only about logic; it is also constrained by quotas.

Common practical constraints include:

- maximum SCP size
- maximum number of SCPs attached to a target
- maximum number of policies in an organization
- maximum number of OUs and account attachment combinations
- service-specific limits for delegated administration and trusted access

These limits matter when you build layered guardrails:

- A large deny-list can become hard to maintain.
- Too many small SCPs can make troubleshooting difficult.
- Exception-heavy region policies can approach size limits quickly.
- Delegated admin designs may require multiple SCPs to preserve both security and operability.

For exam purposes, remember the design implication: if a policy model becomes too complex, you may need to simplify the OU structure or consolidate guardrails.

**Exam-focused patterns and examples**

A few common scenarios appear repeatedly in SOA-C02-style questions:

- You want to stop developers from disabling CloudTrail.
  - Use an SCP deny on `cloudtrail:StopLogging`, `cloudtrail:DeleteTrail`, and related actions.
- You want a security account to manage organization-wide findings.
  - Enable trusted access and register a delegated administrator.
- You want to restrict workloads to approved regions.
  - Use a region-deny SCP, but test global service exceptions.
- You want to let teams create roles but not exceed a safe permission set.
  - Use permission boundaries inside the account, not SCPs alone.
- You want to prevent a member account from leaving the organization.
  - Deny `organizations:LeaveOrganization` in an SCP.

Example deny guardrail:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ProtectSecurityServices",
      "Effect": "Deny",
      "Action": [
        "cloudtrail:StopLogging",
        "cloudtrail:DeleteTrail",
        "config:StopConfigurationRecorder",
        "config:DeleteConfigurationRecorder"
      ],
      "Resource": "*"
    }
  ]
}
```

**Practical CLI workflow for validation**

A safe validation workflow is to test IAM logic separately, then confirm organization-level behavior.

1. Verify the SCPs attached to the target account.
2. Confirm the IAM policy on the user or role.
3. Check whether a permission boundary exists.
4. Test the action in a non-production account.
5. Review CloudTrail if the request is denied.
6. Adjust the SCP only after confirming the exact missing permission.

Example CLI sequence:

```bash
aws organizations list-policies-for-target \
  --target-id 111122223333 \
  --filter SERVICE_CONTROL_POLICY

aws organizations describe-policy \
  --policy-id p-abc123example

aws sts get-caller-identity
aws ec2 describe-instances
```

If the action fails in the account but succeeds in IAM policy simulation, remember that the simulator does not evaluate SCP inheritance. It is still useful for IAM debugging, but not sufficient for organization-wide authorization validation.

**Key takeaways**

- SCPs define the maximum permissions for member accounts, not the permissions themselves.
- SCPs apply to IAM principals and the root user in member accounts.
- The management account is not governed by SCPs.
- Allow-list SCPs require explicit coverage for every needed action.
- Permission boundaries and SCPs both restrict access, but at different layers.
- Delegated administration still operates inside SCP guardrails.
- Region restriction SCPs need careful exception handling for global and regionless services.
- CloudTrail and Organizations APIs are the best tools for diagnosing SCP-related denials.
