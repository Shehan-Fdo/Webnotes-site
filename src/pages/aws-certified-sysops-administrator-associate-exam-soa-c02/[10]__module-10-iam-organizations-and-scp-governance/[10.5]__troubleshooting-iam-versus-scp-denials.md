## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Why this matters**

When AWS returns `AccessDenied`, the root cause is often not obvious. The denial may come from an **IAM** policy, a **service control policy** (SCP), a permissions boundary, a session policy, a resource-based policy, or a service-specific condition. For exam and operations work, you need a fast way to identify which layer blocked the request and why.

A useful mental model is:

- **IAM** controls what a principal can do within an account.
- **SCPs** set the maximum permissions available to accounts in an AWS Organizations hierarchy.
- **Resource policies** can grant access directly to a resource.
- **Boundaries and session policies** further limit what a role or session can do.

The key is not to guess from the error text alone. You inspect the principal, the account, the target resource, and the organization context together.

**Core authorization model**

AWS authorization is evaluated across multiple policy types, but not as a simple “all layers must allow” intersection in every case. In practice, an explicit deny anywhere wins, and some resource-based policy scenarios can allow access without a matching identity-based allow.

Important points:

- **Explicit deny** always overrides allow.
- **SCPs** define the outer ceiling for member accounts in AWS Organizations.
- **Identity-based policies** attach to users, groups, and roles.
- **Resource-based policies** attach to resources such as S3 buckets, KMS keys, SNS topics, and SQS queues.
- **Permissions boundaries** cap the maximum permissions for a user or role.
- **Session policies** can further reduce permissions for an assumed role session.

A practical rule:

- If the request is denied, look for the most restrictive layer first.
- If the request is allowed by a resource policy but still fails, check whether an SCP, boundary, or condition is blocking it.
- If the request is in the management account, SCPs generally do not restrict principals there directly, but other IAM and resource-policy limits still apply.

**How SCP inheritance works**

SCPs are attached at the **root**, **OU** (organizational unit), or **account** level. The effective SCP permissions for an account are the result of inheritance from all parent containers plus any SCPs attached directly to the account.

Key behavior:

- The root SCPs apply to all accounts in the organization unless an account is outside the organization.
- OU SCPs apply to all accounts in that OU and its child OUs.
- Account-level SCPs apply only to that account.
- Multiple SCPs combine to form the effective permission set.
- An SCP can be either:
  - an **allow list** model, where only explicitly allowed actions are permitted, or
  - a **deny list** model, where a broad allow such as `FullAWSAccess` is attached and specific denies remove access.

A common exam trap is assuming that attaching one SCP to an OU is enough to understand the result. You must inspect the full path from root to OU to account.

If the root has a deny on `ec2:TerminateInstances`, the account cannot override that deny with a lower-level SCP. If the root uses an allow-list model, then every required action must be explicitly allowed somewhere in the inherited SCP set.

**Fast way to distinguish IAM from SCP denials**

Use the context of the principal and the account first.

- **Member account principal**
  - Could be blocked by IAM, SCP, permissions boundary, session policy, or resource policy.
- **Management account principal**
  - SCPs do not directly restrict the management account the same way they restrict member accounts.
  - IAM, resource policies, and service-specific controls can still deny access.
- **IAM user or role in a member account**
  - SCPs apply regardless of whether the principal is a user or role.
- **Root user in a member account**
  - SCPs can still constrain actions in the member account.
- **Assumed role session**
  - Check the role policy, the trust policy, the session policy, and any permissions boundary.

A useful troubleshooting question is:

- “Would this action be allowed if the IAM policy were perfect?”

If the answer is still no, suspect SCPs, boundaries, or resource policies.

**Common denial patterns**

Different layers often leave different clues.

- **IAM denial**
  - The principal lacks an allow in its identity policy.
  - The policy may miss the exact action, resource ARN, or condition.
- **SCP denial**
  - The action is not allowed by the organization’s effective SCP set.
  - The denial often appears even when the IAM policy looks correct.
- **Permissions boundary denial**
  - The role or user policy allows the action, but the boundary caps it.
- **Resource policy denial**
  - The resource does not trust the principal, account, or organization.
- **Condition-based denial**
  - The action is allowed only in certain Regions, with certain tags, or from certain networks.

Examples of condition-based SCP denials:

- Deny all actions outside approved Regions using `aws:RequestedRegion`.
- Deny creation of resources unless a required tag exists using `aws:RequestTag` or `aws:TagKeys`.
- Deny access unless the request comes from a specific VPC endpoint using `aws:SourceVpce`.

These are especially tricky because the policy may look permissive at first glance.

**Console troubleshooting path**

The AWS Management Console is often the fastest place to confirm whether an SCP is attached.

1. Open **AWS Organizations**.
2. Go to **Policies** and select **Service control policies**.
3. Review the SCPs attached to the **root**, relevant **OU**, and target **account**.
4. Open the account’s **organization path** to confirm inheritance.
5. Compare the effective SCP set with the denied action.
6. Check whether the account is in the **management account** or a **member account**.
7. If needed, switch to **IAM** and inspect the user or role policies, permissions boundary, and session context.

The console is useful because it shows the hierarchy visually. That matters when a deny comes from a parent OU rather than the account itself.

**CLI checks for IAM versus SCP troubleshooting**

Use the CLI to inspect the organization structure and the policies attached at each level.

```bash
aws organizations list-parents --child-id 111122223333
aws organizations list-policies-for-target \
  --target-id 111122223333 \
  --filter SERVICE_CONTROL_POLICY
aws organizations describe-policy \
  --policy-id p-abc123example
aws organizations list-policies-for-target \
  --target-id ou-abcd-12345678 \
  --filter SERVICE_CONTROL_POLICY
```

These commands help you answer:

- Which OU or root is the account under?
- Which SCPs are attached to the root, OU, and account?
- Does the effective SCP set include an explicit deny?
- Is the account using an allow-list model that omits the action?

For IAM-side inspection:

```bash
aws iam get-user
aws iam list-attached-user-policies --user-name Alice
aws iam list-user-policies --user-name Alice
aws iam get-role --role-name AppRole
aws iam list-attached-role-policies --role-name AppRole
aws iam get-role-policy --role-name AppRole --policy-name InlinePolicyName
aws iam get-policy-version --policy-arn arn:aws:iam::123456789012:policy/ExamplePolicy --version-id v1
```

For an assumed role session, also inspect the session source and any session policy used during `sts:AssumeRole`.

**Policy simulation and its limits**

IAM policy simulation is useful for identity-based policies and some resource-policy scenarios, but it does not replace SCP review.

Use it to test whether a principal’s IAM policy allows an action:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/AppRole \
  --action-names ec2:RunInstances \
  --resource-arns arn:aws:ec2:us-east-1:123456789012:instance/*
```

What this tells you:

- Whether the identity policy would allow the action.
- Whether a permissions boundary or session policy may matter, depending on the simulation inputs.

What it does not fully solve:

- It does not provide a general-purpose SCP simulator equivalent to IAM policy simulation.
- It does not automatically prove the effective organization-level permission set.
- It does not replace checking the root, OU, and account SCP attachments.

For SCP troubleshooting, you still need to inspect the organization hierarchy and the attached policies directly.

**Side-by-side comparison**

| Control type | Where it attaches | What it limits | Common troubleshooting clue |
|---|---|---|---|
| IAM identity policy | User, group, role | What the principal can do | Missing allow or wrong resource ARN |
| SCP | Root, OU, account | Maximum permissions in member accounts | Action denied even though IAM looks correct |
| Permissions boundary | User or role | Upper bound on identity permissions | Policy allows action, boundary does not |
| Session policy | STS session | Temporary session permissions | Assumed role works differently than base role |
| Resource policy | Resource | Who can access the resource | Resource trusts some principals but not yours |

This comparison is useful because many exam questions hinge on identifying the correct layer from a short scenario.

**Practical troubleshooting workflow**

1. Confirm the exact API action that failed.
2. Identify the principal type: IAM user, IAM role, assumed role, root, or service-linked role.
3. Determine whether the account is a member account or the management account.
4. Check the IAM identity policy for an allow on the exact action and resource.
5. Check for a permissions boundary or session policy.
6. Review the resource policy if the target supports one.
7. Inspect the organization path and attached SCPs from root to OU to account.
8. Look for explicit denies and condition-based restrictions.
9. Reproduce the request with the same principal and Region.
10. Review CloudTrail for the denied event and surrounding context.

**CloudTrail evidence**

CloudTrail helps you confirm the exact API call, the principal, and the error code. It is especially useful when the console error is vague.

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=RunInstances
```

Use CloudTrail to look for:

- The exact event name.
- The user identity type.
- The source IP or VPC endpoint.
- The Region.
- The error code and error message.
- Whether the failure happened before or after a related service call.

CloudTrail is not always enough by itself to distinguish IAM from SCP, but it often reveals whether the request reached the service and which principal made it.

**Service-linked role edge cases**

Some AWS services create or use **service-linked roles**. SCPs can block the permissions needed to create or use those roles, which can look like a service failure rather than an authorization failure.

Watch for:

- A service trying to create a service-linked role and failing.
- A deployment that works in one account but not another because the SCP blocks `iam:CreateServiceLinkedRole`.
- A service operation that depends on a role the account cannot create due to an SCP deny.

If a service suddenly stops provisioning resources, check whether the SCP blocks IAM actions required by the service itself.

**Region and tag restrictions**

SCPs often enforce guardrails rather than direct service access. These guardrails can produce confusing denials.

Common examples:

- Restricting actions to approved Regions.
- Requiring tags on create operations.
- Blocking destructive actions except for break-glass roles.
- Denying access unless requests come from approved networks.

Example SCP pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyOutsideApprovedRegions",
      "Effect": "Deny",
      "Action": "*",
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": [
            "us-east-1",
            "us-west-2"
          ]
        }
      }
    }
  ]
}
```

This kind of policy can make a valid IAM permission fail if the request is sent to a disallowed Region.

**Root user considerations**

The root user in a member account is not a bypass for SCPs. In member accounts, SCPs still constrain what can be done, even if the principal is root. That said, root access is still powerful and should be tightly controlled.

For troubleshooting:

- Do not assume root means “not an SCP issue.”
- Do assume root can still be blocked by organization guardrails in member accounts.
- Check whether the action is restricted by a deny in the SCP hierarchy.

**Service-linked role and resource-policy examples**

A resource policy can sometimes grant access even when identity policy is minimal, but SCPs can still block the action in member accounts.

Example S3 bucket policy allowing a specific role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowAppRoleRead",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:role/AppRole"
      },
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::example-bucket/*"
    }
  ]
}
```

If the role is in a member account and an SCP denies `s3:GetObject`, the bucket policy alone will not save the request.

**Exam-focused decision cues**

Use these cues to narrow the cause quickly:

- If the error appears across many principals in the same account, suspect an SCP or account-wide boundary.
- If only one role fails, suspect IAM policy, boundary, or session policy.
- If access works from one resource but not another, suspect resource policy or ARN mismatch.
- If access fails only in one Region, suspect a Region-restricting SCP or condition.
- If a service cannot create infrastructure, suspect a deny on the IAM actions the service needs.

A safe exam answer pattern is:

- Check IAM first for the principal’s allow.
- Check SCPs next for organization guardrails.
- Check resource policies and conditions last if the target resource supports them.

**Quick reference checklist**

- Confirm the exact denied action.
- Identify the principal and account type.
- Review IAM identity policies.
- Check permissions boundaries and session policies.
- Inspect resource policies.
- Walk the SCP hierarchy from root to OU to account.
- Look for explicit denies and condition keys.
- Verify Region, tags, and network conditions.
- Use CloudTrail to confirm the request context.

**Key takeaway**

An `AccessDenied` message is a symptom, not a diagnosis. The fastest way to solve it is to trace the request through IAM, SCP inheritance, and resource-level controls in that order, while paying close attention to conditions and the account’s place in AWS Organizations.
