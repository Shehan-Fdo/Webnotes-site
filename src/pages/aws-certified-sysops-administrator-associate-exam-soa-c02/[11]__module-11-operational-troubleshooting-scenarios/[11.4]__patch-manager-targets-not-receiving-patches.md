## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Patch Manager troubleshooting overview**

AWS Systems Manager Patch Manager applies operating system and application patches to managed nodes using a patch baseline, a patching operation, and a target selection method. When targets do not receive patches, the root cause is usually one of three areas:

- **Node readiness**: the instance is not properly managed by Systems Manager.
- **Baseline mapping**: the node is not matched to the intended patch baseline or patch group.
- **Execution timing**: the patch operation ran, but the target was skipped, failed, or could not complete.

A useful mental model is:

```text
Managed node ready?
        |
        v
Patch baseline matched?
        |
        v
Patch operation executed?
        |
        v
Reboot/compliance completed?
```

If you separate those layers, troubleshooting becomes much faster.

**Patch Manager prerequisites**

Patch Manager only works when the managed node is healthy and reachable by Systems Manager. Check these prerequisites first:

- **SSM Agent** is installed and running on the instance.
- The instance profile includes the **AmazonSSMManagedInstanceCore** permissions or an equivalent least-privilege policy.
- The node can reach Systems Manager endpoints through the internet, a NAT gateway, or **VPC endpoints** for Systems Manager services.
- The instance appears in Systems Manager as a managed node.
- The operating system is supported by the patching workflow you are using.

Common endpoint dependencies include:

- `ssm`
- `ec2messages`
- `ssmmessages`

For private subnets, interface VPC endpoints are often the cleanest design because they remove dependence on public internet access.

> If the node is not a healthy managed node in the same Region where you run the Systems Manager operation, Patch Manager cannot patch it.

**How Patch Manager decides what to patch**

Patch Manager uses a **patch baseline** to define which patches are approved. The baseline is not “on” the instance; it is a Systems Manager configuration object that determines patch eligibility during a patching run.

A baseline can include rules such as:

- auto-approve patches by **classification**
- auto-approve patches by **severity**
- approve patches after a **delay period**
- explicitly approve or reject specific patch IDs
- define a default baseline for an operating system

Example baseline logic:

- Approve `Security` patches immediately.
- Approve `Bugfix` patches after 7 days.
- Approve only `Critical` and `Important` severities.
- Reject a specific kernel update that breaks a legacy driver.

That kind of rule set can make it look like targets are “not receiving patches” when the real issue is that the patches are not approved by the baseline.

**Patch groups and target mapping**

Patch groups are based on the instance tag key `Patch Group` and a tag value such as `WebServers` or `LinuxProd`. The important detail is that the tag key must be exactly `Patch Group`.

Example:

```text
Key:   Patch Group
Value: WebServers
```

Then you associate that patch group value with a baseline. The association is by the patch group value, not by inventing a different tag key name.

A common mistake is:

- tagging instances with `PatchGroup` instead of `Patch Group`
- associating the wrong patch group value with the baseline
- assuming the default baseline will apply when a custom patch group is present

**Baseline association example**

```bash
aws ssm register-patch-baseline-for-patch-group \
  --baseline-id pb-0123456789abcdef0 \
  --patch-group WebServers
```

If the instance has `Patch Group=WebServers`, it can use that baseline association. If the tag value is different, the instance may fall back to the default baseline for that OS.

**Why targets appear to miss patches**

The most common causes are:

- **Wrong patch group tag**
  - The instance uses `PatchGroup` instead of `Patch Group`.
  - The tag value does not match the baseline association.
- **No managed node connectivity**
  - SSM Agent is stopped.
  - IAM role is missing permissions.
  - VPC endpoints or internet access are missing.
- **Baseline approval rules exclude the patch**
  - The patch is not approved yet because of a delay.
  - The patch is rejected by classification or severity rules.
- **Maintenance window did not run as expected**
  - The task was not triggered.
  - The target set was empty.
  - The execution failed before patching began.
- **Reboot behavior prevented completion**
  - The node needed a reboot to finish installation.
  - The reboot was suppressed or deferred.
  - Compliance still shows missing patches until the reboot completes and inventory refreshes.
- **Operating system or repository issues**
  - The package repository is unreachable.
  - The instance cannot download patch metadata.
  - The OS package manager is in a broken state.

**Reboot behavior and compliance**

Reboots are a frequent reason patching appears incomplete. Some patches install only after a reboot, and compliance may remain noncompliant until the reboot happens and Systems Manager updates its view of the node.

Important points:

- `RebootIfNeeded` controls whether Patch Manager reboots the node when required.
- If you choose not to reboot, the patching run may finish with pending changes.
- A node with a pending reboot can still appear to have missing patches.
- Compliance reporting may lag until the instance completes the reboot and the next inventory or compliance update occurs.

Example patch command:

```bash
aws ssm send-command \
  --document-name "AWS-RunPatchBaseline" \
  --targets "Key=tag:Patch Group,Values=WebServers" \
  --parameters '{"Operation":["Install"],"RebootOption":["RebootIfNeeded"]}'
```

If you use `NoReboot`, verify whether the OS requires a restart before concluding that patching failed.

**Maintenance windows and execution scope**

Maintenance windows schedule patching, but they do not themselves patch the node. The patch task runs only if the maintenance window execution, target registration, and task invocation all succeed.

Useful checks:

- confirm the maintenance window exists in the same Region as the target node
- confirm the target registration matches the intended instances or tags
- confirm the task is associated with the correct maintenance window
- confirm the execution actually started

Example commands:

```bash
aws ssm describe-maintenance-windows
aws ssm describe-maintenance-window-targets --window-id mw-0123456789abcdef0
aws ssm describe-maintenance-window-tasks --window-id mw-0123456789abcdef0
aws ssm describe-maintenance-window-executions --window-id mw-0123456789abcdef0
```

To inspect a specific task execution, you often need the execution identifiers, not just the window and task IDs:

```bash
aws ssm describe-maintenance-window-execution-task-invocations \
  --window-execution-id mw-exec-0123456789abcdef0 \
  --task-id task-0123456789abcdef0
```

If you are troubleshooting a missed run, always verify the execution record before assuming Patch Manager skipped the node.

**Useful CLI checks**

Start with the managed node inventory:

```bash
aws ssm describe-instance-information
```

Look for:

- `PingStatus` should be `Online`
- `PlatformType` should match the intended patching workflow
- `AgentVersion` should be current enough for your environment
- the instance should appear in the expected Region

Check patch baseline associations:

```bash
aws ssm describe-patch-baselines
aws ssm describe-patch-groups
aws ssm describe-effective-patches \
  --baseline-id pb-0123456789abcdef0 \
  --operating-system AMAZON_LINUX_2
```

Check patch compliance for a node:

```bash
aws ssm describe-instance-patch-states \
  --instance-ids i-0123456789abcdef0
```

Check patch compliance details:

```bash
aws ssm describe-instance-patch-state \
  --instance-id i-0123456789abcdef0
```

Check associations when you suspect a document or schedule is not applying:

```bash
aws ssm describe-associations
```

The response typically includes fields such as `AssociationId`, `Status`, and `Overview`. Do not assume a field named `AssociationStatus` exists at the top level.

**Practical troubleshooting flow**

1. Verify the instance is a healthy managed node.
   - Confirm `PingStatus=Online`.
   - Confirm the SSM Agent is running.
   - Confirm IAM and network access are correct.

2. Verify the patch group mapping.
   - Check the instance tag `Patch Group`.
   - Confirm the tag value matches the baseline association.
   - Confirm the correct baseline is the default or explicitly associated baseline.

3. Verify the baseline rules.
   - Check whether the patch is approved.
   - Check approval delay, severity, and classification filters.
   - Check whether a rejected patch ID is blocking the update you expected.

4. Verify the execution path.
   - If using a maintenance window, confirm the window execution ran.
   - If using `send-command`, confirm the command invocation succeeded.
   - Check for failures caused by missing permissions or unreachable repositories.

5. Verify reboot and compliance state.
   - Determine whether the patch requires a reboot.
   - Check whether the node rebooted successfully.
   - Re-run compliance checks after the reboot.

**Example end-to-end troubleshooting scenario**

A Linux instance in a private subnet is not receiving patches during a weekly maintenance window.

1. You check `describe-instance-information` and see the node is `Online`.
2. You confirm the instance profile includes Systems Manager permissions.
3. You inspect the instance tags and find `PatchGroup=WebServers` instead of `Patch Group=WebServers`.
4. You check the baseline association and see the baseline is associated with `WebServers`.
5. You correct the tag key to `Patch Group`.
6. You rerun the maintenance window.
7. The patch run now starts, but the node still shows missing kernel patches.
8. You inspect the baseline and see kernel updates are approved only after a 7-day delay.
9. You conclude the node is healthy; the baseline approval policy is intentionally delaying the patch.

That sequence shows why patching problems are often a combination of tagging, baseline policy, and execution timing rather than a single failure.

**Baseline rule example**

A baseline can be configured to auto-approve patches like this:

```json
{
  "PatchRules": [
    {
      "PatchFilterGroup": {
        "PatchFilters": [
          { "Key": "CLASSIFICATION", "Values": ["Security"] },
          { "Key": "SEVERITY", "Values": ["Critical", "Important"] }
        ]
      },
      "ApproveAfterDays": 0
    },
    {
      "PatchFilterGroup": {
        "PatchFilters": [
          { "Key": "CLASSIFICATION", "Values": ["Bugfix"] }
        ]
      },
      "ApproveAfterDays": 7
    }
  ]
}
```

This kind of baseline is useful because it balances speed and risk:

- security patches move quickly
- lower-priority fixes wait for validation
- production systems avoid surprise regressions

But it also means a target may be working correctly and still not receive a patch immediately.

**Real-world log and output clues**

When patching fails, the command output or logs often point to the layer that broke.

Examples of useful clues:

- `AccessDeniedException` suggests IAM permissions are missing.
- `TargetNotConnected` suggests the node is not reachable by Systems Manager.
- `InvalidInstanceId` suggests the instance is not a managed node in that Region.
- `No patches were installed` can mean the baseline did not approve any applicable patches.
- `Reboot required` or similar messages suggest the patching run needs a restart to complete.

Example command output pattern:

```text
{
  "CommandInvocations": [
    {
      "Status": "Failed",
      "StatusDetails": "DeliveryTimedOut",
      "InstanceId": "i-0123456789abcdef0"
    }
  ]
}
```

That kind of result usually points to connectivity or agent health rather than baseline logic.

**Exam-focused distinctions**

- **Patch baseline** controls what is approved.
- **Patch group** controls which baseline applies.
- **Maintenance window** controls when patching runs.
- **SSM Agent and IAM role** control whether the node can receive the command.
- **Reboot behavior** controls whether patching can fully complete.

These are separate layers, and exam questions often hide the failure in one layer while describing symptoms in another.

**Quick reference checklist**

- Confirm the instance is a managed node in the correct Region.
- Confirm `SSM Agent` is running and healthy.
- Confirm the instance role has Systems Manager permissions.
- Confirm network access to Systems Manager endpoints.
- Confirm the instance tag key is exactly `Patch Group`.
- Confirm the patch group value matches the baseline association.
- Confirm the baseline approves the patch you expect.
- Confirm the maintenance window or command actually executed.
- Confirm reboot requirements were handled.
- Recheck compliance after the reboot and inventory refresh.

**When to use `AWS-RunPatchBaseline`**

Use `AWS-RunPatchBaseline` when you want Systems Manager to install or scan patches according to the approved baseline. It is the standard public document for patching workflows.

Example install run:

```bash
aws ssm send-command \
  --document-name "AWS-RunPatchBaseline" \
  --targets "Key=tag:Patch Group,Values=LinuxProd" \
  --parameters '{"Operation":["Install"],"RebootOption":["RebootIfNeeded"]}'
```

Example scan run:

```bash
aws ssm send-command \
  --document-name "AWS-RunPatchBaseline" \
  --targets "Key=tag:Patch Group,Values=LinuxProd" \
  --parameters '{"Operation":["Scan"]}'
```

Use `Scan` to assess compliance before you install. Use `Install` when you are ready to apply approved patches.

**Summary**

When Patch Manager targets are not receiving patches, troubleshoot in this order:

- node readiness
- patch group mapping
- baseline approval rules
- maintenance window or command execution
- reboot and compliance refresh

Most failures are not caused by Patch Manager “ignoring” the instance. They are caused by a missing prerequisite, a tag mismatch, a baseline rule that excludes the patch, or an execution that never completed.
