## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**What “registration” means in Systems Manager**

AWS Systems Manager <abbr title="Systems Manager">SSM</abbr> Agent is the local component that lets an instance or server become a managed node. Registration is the point where the agent authenticates to Systems Manager, identifies the node, and begins exchanging inventory, command, and session data with the service.

If registration fails, the node usually cannot appear as managed, cannot receive Run Command tasks, and cannot start Session Manager sessions. The root cause is often not the agent binary itself, but one of four areas:

- **IAM** permissions for the instance or activation
- **Network** reachability to required endpoints
- **Agent** installation, version, or service health
- **Activation** or identity problems for on-premises or hybrid nodes

**How SSM Agent communicates**

SSM Agent does not use a single generic connection to “Systems Manager.” It uses service-specific endpoints and behavior depending on the feature in use. In practice, the agent commonly needs access to:

- `ssm`
- `ec2messages`
- `ssmmessages`

For many features, the agent polls and exchanges messages with these services over HTTPS. Session Manager and other interactive features rely on the `ssmmessages` channel, while command and inventory workflows use the Systems Manager control plane and related messaging services.

> If the instance is in a private subnet, missing VPC endpoints are a common cause of registration failure. Do not assume internet access is required; private connectivity through interface endpoints can be enough when configured correctly.

**First checks: is the agent running?**

Before troubleshooting AWS-side configuration, verify the local service.

On Linux:

```bash
sudo systemctl status amazon-ssm-agent
sudo journalctl -u amazon-ssm-agent --no-pager -n 100
```

On Windows PowerShell:

```powershell
Get-Service AmazonSSMAgent
Get-Content "C:\ProgramData\Amazon\SSM\Logs\amazon-ssm-agent.log" -Tail 100
```

Useful checks:

- **Service state** should be `active` on Linux or `Running` on Windows.
- **Logs** often show whether the failure is DNS, TLS, proxy, IAM, or activation related.
- **Version** matters because older agents may lack fixes or endpoint behavior needed in your environment.

If the service is stopped, start it and enable it at boot:

```bash
sudo systemctl enable --now amazon-ssm-agent
```

**IAM role and instance profile problems**

For EC2 instances, the most common registration issue is a missing or incorrect instance profile. The instance needs an IAM role attached through an instance profile, and that role must allow Systems Manager to manage the node.

The AWS managed policy typically used is `AmazonSSMManagedInstanceCore`. Do not memorize a hand-written JSON approximation as the policy definition; the managed policy can evolve over time. For exam purposes, know that it grants the agent the permissions needed to register, describe itself, send inventory, and communicate with Systems Manager services.

Key points:

- **Instance profile attached** to the EC2 instance
- **Correct managed policy** or equivalent permissions on the role
- **No conflicting explicit deny** in SCPs, permission boundaries, or session policies
- **Trust policy** allows EC2 to assume the role

Example trust policy for an EC2 instance role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "ec2.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

How to verify the instance profile from the CLI:

```bash
aws ec2 describe-instances \
  --instance-ids i-0123456789abcdef0 \
  --query 'Reservations[0].Instances[0].IamInstanceProfile'
```

If the profile is missing, attach one and then restart the agent or reboot the instance if needed.

**Network reachability and endpoint design**

Registration can fail even when IAM is correct if the node cannot reach the required endpoints.

Common causes:

- **No internet route** from a public subnet instance
- **Missing VPC interface endpoints** in private subnets
- **Security group egress** blocks HTTPS traffic
- **Network ACLs** block outbound or return traffic
- **DNS resolution** fails for AWS service names
- **Proxy** settings are missing or incorrect

For private subnets, you often need interface endpoints for Systems Manager-related services. In many environments, the minimum set includes:

- `com.amazonaws.<region>.ssm`
- `com.amazonaws.<region>.ec2messages`
- `com.amazonaws.<region>.ssmmessages`

Depending on your use case, you may also need endpoints for related services such as CloudWatch Logs, S3, or KMS.

Example endpoint verification from an instance:

```bash
curl -I https://ssm.us-east-1.amazonaws.com
curl -I https://ec2messages.us-east-1.amazonaws.com
curl -I https://ssmmessages.us-east-1.amazonaws.com
```

A successful TCP/TLS path does not prove IAM is correct, but it does confirm basic network reachability.

**Proxy configuration**

If your environment requires an outbound proxy, SSM Agent must be configured using the agent’s supported configuration format for that platform and version. Do not assume that dropping `http_proxy` and `https_proxy` into an arbitrary JSON file is sufficient.

Practical guidance:

- **Use the documented agent configuration method** for your OS and agent version.
- **Include bypass rules** for local metadata and internal addresses where appropriate.
- **Test proxy reachability** separately with `curl` or `Invoke-WebRequest`.
- **Check logs** for proxy authentication or certificate errors.

Example shell environment variables for a manual test session:

```bash
export HTTP_PROXY=http://proxy.example.com:8080
export HTTPS_PROXY=http://proxy.example.com:8080
export NO_PROXY=169.254.169.254,localhost,127.0.0.1
curl -I https://ssm.us-east-1.amazonaws.com
```

This is only a test pattern, not a universal SSM Agent configuration method.

**EC2 instance metadata and credential retrieval**

For EC2, the agent usually obtains credentials from the instance profile through the instance metadata service <abbr title="Instance Metadata Service">IMDS</abbr>. If IMDS is disabled or restricted too tightly, the agent may not be able to retrieve credentials.

Check for:

- **IMDS availability**
- **IMDSv2 token requirements** if the environment enforces them
- **Hop limit** settings that may block access from the agent process
- **Local firewall rules** that interfere with `169.254.169.254`

Example metadata test:

```bash
TOKEN=$(curl -X PUT "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")

curl -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/meta-data/iam/security-credentials/
```

If this fails, the agent may not be able to retrieve role credentials even when the role is attached.

**On-premises and hybrid activation issues**

For non-EC2 managed nodes, registration uses a Systems Manager activation. The workflow is more than a single command. You must create an activation in Systems Manager, install the agent, register the node with the activation code and ID, and ensure the service starts successfully afterward.

Typical workflow:

1. Create an activation in Systems Manager.
2. Install SSM Agent on the server.
3. Register the agent with the activation code and activation ID.
4. Start or restart the agent service.
5. Confirm the node appears as managed.

Example activation creation:

```bash
aws ssm create-activation \
  --default-instance-name "onprem-web-01" \
  --iam-role "AmazonSSMHybridInstanceRole" \
  --registration-limit 1 \
  --region us-east-1
```

Example registration command:

```bash
sudo amazon-ssm-agent -register \
  -code "activation-code" \
  -id "activation-id" \
  -region "us-east-1"
```

Then ensure the service is running:

```bash
sudo systemctl restart amazon-ssm-agent
sudo systemctl status amazon-ssm-agent
```

Important considerations:

- **Activation expiration** can cause registration failures if you wait too long.
- **Registration limit** can be exhausted after the allowed number of nodes.
- **Region mismatch** between activation and agent registration breaks onboarding.
- **Clock skew** can affect TLS and authentication flows.

**Interpreting common console states**

The Systems Manager console can show different states that are easy to misread.

- **Online** means the node is currently communicating.
- **Connection lost** usually means the node was previously managed and has stopped communicating, but transient delays can also appear during initial onboarding or brief outages.
- **Inactive** often points to activation or registration problems for hybrid nodes.
- **Not managed** usually means the node has not successfully registered or lacks the required role and connectivity.

Treat these as clues, not final diagnoses. Always confirm with logs and endpoint checks.

**Logs and what to look for**

SSM Agent logs are the fastest way to narrow the issue.

Linux log locations commonly include:

```bash
/var/log/amazon/ssm/amazon-ssm-agent.log
/var/log/amazon/ssm/errors.log
```

Windows log locations commonly include:

```text
C:\ProgramData\Amazon\SSM\Logs\amazon-ssm-agent.log
C:\ProgramData\Amazon\SSM\Logs\errors.log
```

Look for patterns such as:

- **AccessDenied** or `UnauthorizedOperation` for IAM issues
- **dial tcp** or `timeout` for network problems
- **certificate** or `x509` errors for TLS trust issues
- **proxy** authentication failures
- **activation** or `registration` errors for hybrid nodes

Example log search on Linux:

```bash
sudo grep -Ei 'error|accessdenied|timeout|proxy|certificate|registration' \
  /var/log/amazon/ssm/amazon-ssm-agent.log | tail -n 50
```

**Practical troubleshooting sequence**

Use a layered approach so you do not chase symptoms.

1. Confirm the agent service is running.
2. Check the agent logs for the first clear error.
3. Verify the instance profile or activation details.
4. Confirm DNS and HTTPS reachability to required endpoints.
5. Validate proxy settings if the environment uses one.
6. Check IMDS access for EC2 instances.
7. Restart the agent after fixing the root cause.

Example restart and verification:

```bash
sudo systemctl restart amazon-ssm-agent
sudo systemctl status amazon-ssm-agent
```

**Real-world examples**

A private subnet EC2 instance with the correct IAM role but no VPC endpoints often fails because it cannot reach `ssm`, `ec2messages`, or `ssmmessages`. The fix is usually to add the endpoints, allow HTTPS in security groups, and confirm DNS resolution.

A hybrid server with a valid activation code but the wrong region often registers nowhere because the activation exists in one region while the agent points to another. The fix is to re-register with the correct region and, if needed, create a fresh activation.

An EC2 instance with a role attached but `Connection lost` in the console may simply have a stopped agent, a broken proxy, or a temporary network outage. Do not assume the instance was always healthy before; verify logs and connectivity first.

**Exam focus**

For SOA-C02, focus on the relationship between symptoms and root causes:

- **No managed node entry** often means registration never completed.
- **Connection lost** suggests the node was managed but is no longer communicating, though transient states can occur.
- **Private subnet failures** usually point to missing VPC endpoints or routing.
- **Hybrid node failures** usually point to activation, region, or service startup issues.
- **IAM failures** usually point to a missing instance profile or insufficient permissions.

If you remember only one thing, remember this: SSM Agent registration is a three-part problem of identity, network, and service health.
