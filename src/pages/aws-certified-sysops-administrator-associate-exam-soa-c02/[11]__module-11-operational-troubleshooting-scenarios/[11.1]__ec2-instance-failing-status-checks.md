## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**EC2 status checks: what they measure**

Amazon EC2 performs automated health checks to tell you whether an instance is reachable and whether the underlying AWS infrastructure is healthy. These checks are not the same as application monitoring. An instance can pass both checks while your web server, database, or agent is still broken.

The console and CloudWatch expose the key metrics you need to know:

- **`StatusCheckFailed`**: a composite metric that is `1` when either the instance check or the system check fails.
- **`StatusCheckFailed_Instance`**: indicates the guest OS or instance-level problem, such as a hung kernel, failed boot, or network stack issue inside the instance.
- **`StatusCheckFailed_System`**: indicates an AWS infrastructure problem, such as host hardware failure, network impairment, or power issues on the underlying physical server.

In the EC2 console, these are commonly shown as the familiar **2/2 status checks** concept:

- **2/2 passed** means both the instance reachability check and the system reachability check are healthy.
- **1/2 failed** means one of the two checks is failing.
- **0/2 failed** means both checks are failing.

The console typically labels the two checks as:

- **System reachability**
- **Instance reachability**

That distinction matters for troubleshooting because it tells you where to focus first.

**How to interpret the failure type**

A failed **instance status check** usually points to something inside the operating system or boot path:

- Corrupted filesystem
- Misconfigured network settings
- Exhausted disk space
- Broken bootloader or kernel panic
- Overloaded CPU or memory causing the OS to stop responding
- Security software or startup scripts preventing normal boot

A failed **system status check** usually points to the AWS side of the boundary:

- Underlying host failure
- Network connectivity issue on the physical server
- Hardware impairment
- Power or virtualization layer problem

If both checks are green, but users still cannot reach the application, the problem is likely at the application or service layer:

- Web server stopped
- Application process crashed
- Port blocked by security group, NACL, or host firewall
- Load balancer target health failing
- DNS or configuration issue

> A healthy EC2 status check does not guarantee a healthy application.

**Where to see the signals in AWS**

You can inspect status checks in both the EC2 console and CloudWatch.

- In the **EC2 console**, open the instance details and review the **Status checks** tab or the instance list columns.
- In **CloudWatch**, the metrics appear under the EC2 namespace and can be used for alarms and automation.

A practical CloudWatch alarm setup often uses the composite metric:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name ec2-status-check-failed-i-0123456789abcdef0 \
  --namespace AWS/EC2 \
  --metric-name StatusCheckFailed \
  --dimensions Name=InstanceId,Value=i-0123456789abcdef0 \
  --statistic Maximum \
  --period 60 \
  --evaluation-periods 2 \
  --threshold 0 \
  --comparison-operator GreaterThanThreshold \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:ec2-alerts
```

For more targeted alerting, you can alarm separately on instance and system failures:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name ec2-instance-check-failed-i-0123456789abcdef0 \
  --namespace AWS/EC2 \
  --metric-name StatusCheckFailed_Instance \
  --dimensions Name=InstanceId,Value=i-0123456789abcdef0 \
  --statistic Maximum \
  --period 60 \
  --evaluation-periods 2 \
  --threshold 0 \
  --comparison-operator GreaterThanThreshold
```

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name ec2-system-check-failed-i-0123456789abcdef0 \
  --namespace AWS/EC2 \
  --metric-name StatusCheckFailed_System \
  --dimensions Name=InstanceId,Value=i-0123456789abcdef0 \
  --statistic Maximum \
  --period 60 \
  --evaluation-periods 2 \
  --threshold 0 \
  --comparison-operator GreaterThanThreshold
```

**Console workflow for troubleshooting**

When an instance fails a status check, use the console to narrow the cause before taking disruptive action.

1. Open the EC2 console and select the instance.
2. Check whether the failure is **instance**, **system**, or both.
3. Review the **Status checks** details and recent events.
4. Use **Get system log** or **Get instance screenshot** for boot-level clues.
5. If the instance supports it, use the **EC2 Serial Console** for deeper access during boot or login failures.

The screenshot and serial console are especially useful when the instance is stuck early in boot:

- **Screenshot** helps you see the last visible screen state, such as kernel panic, bootloader errors, or login prompts that never complete.
- **Serial Console** is better when you need interactive troubleshooting and the network stack is unavailable.

Use the screenshot first when you want a quick non-invasive clue. Use the serial console when you need to inspect boot output, edit boot parameters, or diagnose a system that never becomes network reachable.

**Instance status check failure: common remediation paths**

When the failure is inside the instance, your response depends on whether the OS is still partially accessible.

- If you can still connect, inspect logs and resource usage:
  - `journalctl -xe`
  - `dmesg`
  - `top`
  - `df -h`
  - `free -m`
  - `ss -tulpn`
- If the instance is unresponsive, consider rebooting first if the issue may be transient.
- If the instance remains unhealthy, stop and start the instance if the root volume is EBS-backed and the workload can tolerate a new public IP.

A reboot may clear temporary OS issues, but it will not fix persistent filesystem corruption or broken startup configuration.

**Offline repair for EBS-backed root volume corruption**

If the instance is EBS-backed and the root filesystem is corrupted, the standard recovery pattern is offline repair using a helper instance.

1. Stop the affected instance.
2. Detach the root EBS volume.
3. Attach the volume to a healthy helper instance in the same Availability Zone.
4. Mount the volume on the helper instance.
5. Run filesystem repair tools:
   - Linux: `fsck` for ext-family filesystems, or the appropriate tool for the filesystem type.
   - Windows: `chkdsk` for NTFS volumes.
6. Review and repair boot configuration, logs, or application files if needed.
7. Unmount the volume.
8. Detach it from the helper instance.
9. Reattach it to the original instance as the root volume.
10. Start the original instance and verify the status checks.

Example Linux repair flow:

```bash
sudo lsblk
sudo mkdir -p /mnt/recovery
sudo mount /dev/nvme1n1p1 /mnt/recovery
sudo fsck -f /dev/nvme1n1p1
sudo umount /mnt/recovery
```

Example Windows repair flow is typically done from an attached volume in a helper Windows instance using `chkdsk` from an elevated command prompt.

> Be careful not to run filesystem repair on the wrong device. Confirm the volume mapping before making changes.

**System status check failure: AWS-side recovery options**

When the system check fails, the problem is usually outside the guest OS. In many cases, AWS can recover the instance to new hardware.

EC2 **instance recovery** is designed for supported instance types and architectures, and it is not guaranteed in every case. Recovery success depends on factors such as:

- Instance type support
- Platform and architecture compatibility
- Whether the instance is backed by EBS
- The nature of the underlying failure
- Whether the instance has local instance store data that cannot be preserved

When recovery succeeds, AWS often preserves:

- The **instance ID**
- The **private IP address**
- The attached EBS volumes

However, do not assume every attached resource or ephemeral behavior is preserved in the same way for every scenario. Instance store data is ephemeral and should be treated as lost on stop, terminate, or host replacement events.

A common exam-relevant distinction:

- **Auto Recovery** attempts to move the instance to healthy hardware while keeping the instance identity intact in many cases.
- **Stop/Start** replaces the underlying host and may change the public IP unless you use an Elastic IP.
- **Terminate/Replace** creates a new instance and is not the same as recovery.

**CloudWatch alarm and recovery automation**

You can automate recovery with a CloudWatch alarm on `StatusCheckFailed_System` or the composite `StatusCheckFailed` metric. In the AWS Management Console, the workflow is:

1. Open **CloudWatch**.
2. Create an alarm on the EC2 metric namespace.
3. Select the instance and the metric:
   - `StatusCheckFailed`
   - `StatusCheckFailed_Instance`
   - `StatusCheckFailed_System`
4. Set the threshold to trigger when the metric is greater than `0`.
5. Choose an action such as **Recover this instance** or send an SNS notification.

A console-based alarm is useful when you want a quick operational setup without scripting. For example, you might create a system-check alarm that triggers recovery automatically and an instance-check alarm that only notifies the operations team.

**Practical console example**

A typical console configuration for a production instance might look like this:

- Alarm name: `prod-web-01-system-check`
- Metric: `StatusCheckFailed_System`
- Statistic: `Maximum`
- Period: `1 minute`
- Evaluation periods: `2`
- Threshold: `> 0`
- Action: **Recover this instance**
- Notification: SNS topic for on-call paging

This setup helps you react quickly to host-level failures while keeping the instance identity stable when recovery is possible.

**Troubleshooting decision tree**

Use the check type to choose the next action.

- **Instance check failed**
  - Investigate OS boot, filesystem, disk space, and network configuration.
  - Use screenshot or serial console for boot diagnostics.
  - Consider offline repair if the root volume is corrupted.
- **System check failed**
  - Treat it as an AWS infrastructure issue.
  - Use CloudWatch alarm recovery or stop/start if appropriate.
  - Verify whether the instance type and architecture support recovery.
- **Both checks passed**
  - Focus on application health, service status, security rules, and load balancer target health.

**Useful commands for live triage**

If you still have access to the instance, these commands help separate OS problems from application problems:

```bash
# Linux
uptime
systemctl --failed
journalctl -p err -b
df -h
free -m
ss -lntp
```

```powershell
# Windows
Get-Service | Where-Object {$_.Status -ne 'Running'}
Get-EventLog -LogName System -Newest 50
Get-Volume
Get-NetTCPConnection
```

These checks help you determine whether the issue is at the operating system layer or only in a specific service.

**Exam focus points**

- Know the difference between **instance reachability** and **system reachability**.
- Remember the **2/2 status check** model in the EC2 console.
- Map `StatusCheckFailed`, `StatusCheckFailed_Instance`, and `StatusCheckFailed_System` to the correct failure domain.
- Understand that application outages can happen even when both status checks are green.
- Know the offline repair pattern for an **EBS-backed** root volume: detach, attach to helper, repair with `fsck` or `chkdsk`, then reattach.
- Know that **instance recovery** is useful but not universal, and recovery behavior depends on support and conditions.
- Use **screenshot** and **serial console** output when boot troubleshooting requires visibility beyond normal network access.

**Quick comparison**

| Situation | Likely cause | Best next step |
|---|---|---|
| `StatusCheckFailed_Instance` = 1 | OS or boot issue | Inspect logs, screenshot, serial console, offline repair |
| `StatusCheckFailed_System` = 1 | AWS host or infrastructure issue | Enable recovery, consider stop/start if needed |
| Both checks = 0 | No EC2 health issue | Troubleshoot application, service, or network configuration |

**Key takeaway**

EC2 status checks tell you whether the instance and the underlying AWS host are healthy, not whether your application is healthy. For SysOps troubleshooting, the fastest path is to identify which check failed, use the right console evidence, and choose the least disruptive recovery method that matches the failure domain.
