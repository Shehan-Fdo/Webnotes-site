## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**SCP fundamentals**

A <abbr title="Service Control Policy">SCP</abbr> is an AWS Organizations policy that sets the maximum permissions available to accounts, organizational units (OUs), and the organization root. It does not grant permissions by itself; it only filters what identity-based policies and resource-based policies can allow.

In practice, an SCP can block an operational task even when:
- The IAM user or role has an allow policy.
- The resource policy allows access.
- The action works in a standalone account outside Organizations.

The key exam idea is that an explicit deny or an implicit deny from an SCP wins before the request can succeed.

- **Guardrail** policies limit what accounts can do.
- **Identity policies** still need to allow the action.
- **Resource policies** cannot override an SCP deny.

**How SCPs block operational tasks**

SCPs commonly block SysOps actions when you try to:
- Stop or terminate critical instances.
- Delete CloudWatch logs, S3 buckets, or backup vaults.
- Disable CloudTrail, Config, or GuardDuty.
- Modify IAM, Organizations, or KMS settings.
- Use services or Regions that the organization has not approved.

A typical failure pattern is an API call returning `AccessDenied` or `UnauthorizedOperation`, even though the IAM policy looks correct. The denial may come from the SCP attached to the account or OU.

> An SCP deny is often invisible if you only inspect IAM permissions. Always check the organization hierarchy.

**Evaluation logic**

AWS evaluates permissions in layers:
1. The request must be allowed by the SCPs that apply to the account.
2. The request must be allowed by an identity-based policy or session policy.
3. If applicable, the request must also be allowed by a resource-based policy.
4. Service-specific conditions, permission boundaries, and other controls may further restrict access.

If any layer denies the action, the request fails.

A useful mental model is:
- **SCPs define the ceiling**
- IAM defines the role/user permissions under that ceiling
- Resource policies define who can access the resource itself

**Common SCP patterns that break operations**

SCPs often block tasks through broad deny statements such as:
- Denying all actions except a small allow list.
- Denying actions in specific Regions.
- Denying changes to security services.
- Denying `iam:*`, `organizations:*`, or `sts:AssumeRole` except for break-glass roles.
- Denying write actions on production resources.

Example of a restrictive SCP that blocks instance termination:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyTerminateInstances",
      "Effect": "Deny",
      "Action": "ec2:TerminateInstances",
      "Resource": "*"
    }
  ]
}
```

Even if an operator has `ec2:TerminateInstances` in IAM, the SCP deny prevents the action.

**Example: Region restriction**

A common operational issue is a Region deny that blocks console or CLI work in an unapproved Region.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyOutsideApprovedRegions",
      "Effect": "Deny",
      "Action": "*",
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": [
            "us-east-1",
            "us-west-2"
          ]
        }
      }
    }
  ]
}
```

This can block:
- EC2 launches in other Regions
- S3 control-plane actions that are Region-scoped
- Lambda, RDS, and many other service operations

Be careful with global services and service-linked behavior. Some actions do not behave like normal Region-scoped API calls, so test before enforcing a broad Region deny.

**How to diagnose an SCP block**

When an operational task fails, use a structured troubleshooting approach.

1. Confirm the exact API action.
   - Identify the service and operation from the console error, CloudTrail event, or CLI output.
   - Example: `ec2:StopInstances`, `s3:DeleteBucket`, `cloudtrail:StopLogging`.

2. Check the IAM principal.
   - Determine whether the caller is a user, role, federated session, or service role.
   - Verify attached policies and permission boundaries.

3. Inspect the organization path.
   - Find the account’s OU and inherited SCPs.
   - Remember that SCPs apply from the root down through each OU to the account.

4. Look for explicit denies.
   - Search for `Effect: Deny`.
   - Check for deny statements using `NotAction`, `Condition`, or wildcard patterns.

5. Test with policy simulation where possible.
   - Use IAM policy simulation for identity policies.
   - Use the Organizations policy tools and effective permissions view if available in your environment.

6. Review CloudTrail.
   - CloudTrail often shows the denied API call and the principal ARN.
   - The event may not always name the SCP directly, but it helps identify the blocked action.

**CLI examples for investigation**

Use the AWS CLI to inspect the account and organization structure.

```bash
aws organizations describe-account --account-id 123456789012
aws organizations list-parents --child-id 123456789012
aws organizations list-policies-for-target --target-id ou-abcd-11111111 --filter SERVICE_CONTROL_POLICY
aws organizations describe-policy --policy-id p-abc123xyz
```

To inspect the caller’s identity and current session context:

```bash
aws sts get-caller-identity
```

To review a CloudTrail event for a denied action:

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=TerminateInstances \
  --max-results 10
```

**Practical troubleshooting example**

Suppose an operator cannot stop an EC2 instance.

- The IAM role includes `ec2:StopInstances`.
- The console returns `UnauthorizedOperation`.
- CloudTrail shows the `StopInstances` call from the role.
- The account is in an OU with an SCP that denies `ec2:StopInstances`.

In that case, the fix is not to add more IAM permissions. You must adjust the SCP, move the account to a different OU, or use an approved break-glass path if one exists.

**Designing safer SCPs**

SCPs are most effective when they protect against accidental or high-risk changes without blocking routine operations.

Good patterns include:
- Deny destructive actions only in production OUs.
- Allow a narrow set of break-glass roles to bypass specific denies.
- Restrict by Region using `aws:RequestedRegion`.
- Protect security services such as CloudTrail, Config, GuardDuty, and Security Hub.
- Deny changes to shared networking or logging resources.

Example of a safer deny with an exception for a break-glass role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyStopInstancesExceptBreakGlass",
      "Effect": "Deny",
      "Action": "ec2:StopInstances",
      "Resource": "*",
      "Condition": {
        "ArnNotLike": {
          "aws:PrincipalArn": "arn:aws:iam::*:role/BreakGlassAdmin"
        }
      }
    }
  ]
}
```

This pattern still requires careful testing. Conditions can behave differently for assumed roles, so validate the principal ARN format in your environment.

**Common pitfalls**

- **Assuming IAM is enough**  
  IAM can allow an action, but SCPs can still block it.

- **Using overly broad denies**  
  `Deny` on `*` without exceptions can break routine administration and automation.

- **Forgetting inherited policies**  
  An SCP at the root applies to all child OUs and accounts unless explicitly exempted by design.

- **Blocking service-linked roles**  
  Some AWS services need service-linked roles or specific permissions to function. Overly broad denies can break managed services.

- **Ignoring automation**  
  CI/CD pipelines, backup jobs, patching tools, and monitoring agents may need permissions that humans do not.

**Operational examples of blocked tasks**

- Stopping EC2 instances during maintenance
- Deleting an S3 bucket used for logs
- Disabling CloudTrail to reduce costs
- Modifying security group rules
- Creating IAM users or access keys
- Launching resources in an unapproved Region
- Changing KMS key policies or disabling keys

These are common exam scenarios because they test whether you recognize the difference between IAM permission issues and organization-level guardrails.

**Best practices for exam and real-world operations**

- Use SCPs to enforce boundaries, not to replace IAM design.
- Start with a permissive baseline and add targeted denies.
- Test policies in a non-production OU before broad rollout.
- Document approved exceptions and break-glass procedures.
- Monitor denied API calls with CloudTrail and AWS Organizations change tracking.
- Keep security and operations teams aligned on which tasks must remain possible.

**Quick comparison**

| Control type | What it does | Can it override an SCP deny? |
|---|---|---|
| IAM policy | Grants permissions to a user or role | No |
| Resource policy | Grants access to a resource | No |
| SCP | Sets the maximum permissions in an account or OU | N/A |
| Permission boundary | Limits a principal’s maximum permissions | No |

**Exam takeaway**

If a task should work but fails with access denied, and the IAM policy appears correct, think about the organization hierarchy first. In AWS Organizations, an SCP can block the operation even when everything else looks valid.
