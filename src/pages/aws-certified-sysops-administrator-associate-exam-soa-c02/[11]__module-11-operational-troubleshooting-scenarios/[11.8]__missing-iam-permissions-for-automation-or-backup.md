## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**What this topic covers**

Automation and backup workflows often fail because the identity running the task does not have the exact AWS Identity and Access Management (IAM) permissions required. In AWS, the caller may be a user, role, service-linked role, or service principal such as `backup.amazonaws.com`, `events.amazonaws.com`, or `ssm.amazonaws.com`. When permissions are missing, the failure may appear as `AccessDenied`, `UnauthorizedOperation`, `Client.UnauthorizedOperation`, or a service-specific error that hides the real cause.

The key skill is to identify **who** is calling, **what** action is denied, and **which** resource or condition is blocking the request. In SysOps scenarios, this usually involves automation tools such as AWS Systems Manager, AWS Backup, Amazon EventBridge, AWS Lambda, AWS CloudFormation, or scheduled scripts running on EC2 instances.

**Common failure patterns**

- A backup plan starts, but the backup vault or source resource cannot be accessed.
- A Systems Manager Automation document fails because the execution role cannot call `ec2:DescribeInstances` or `ssm:SendCommand`.
- A Lambda function triggered by EventBridge cannot write logs or access a target resource.
- A cron job on EC2 uses an instance profile that lacks `s3:GetObject`, `kms:Decrypt`, or `rds:DescribeDBInstances`.
- A service-linked role exists, but the service still cannot perform cross-service actions because the trust policy or permissions policy is incomplete.

**How to troubleshoot permission failures**

Start by reading the exact error message and mapping it to the denied API action. AWS errors often reveal the missing action, the resource ARN, or the condition key that failed. If the message is vague, use CloudTrail to inspect the event history and identify the principal, event name, and error code.

1. Identify the principal that made the call.
   - Check whether the caller is an IAM user, IAM role, assumed role session, AWS service, or instance profile.
   - For automation, confirm which role the workflow assumes.
2. Identify the denied action.
   - Look for the API operation in the error, such as `s3:PutObject`, `ec2:CreateSnapshot`, or `kms:Encrypt`.
   - Match the action to the service that owns it.
3. Identify the resource and conditions.
   - Verify the ARN in the policy.
   - Check for condition keys such as `aws:SourceArn`, `aws:SourceAccount`, `kms:ViaService`, or tag-based restrictions.
4. Confirm the trust relationship.
   - If a service assumes a role, the role trust policy must allow that service principal.
   - If a user assumes a role, the user must have `sts:AssumeRole`.
5. Review explicit denies.
   - An explicit deny in any identity-based policy, resource-based policy, permissions boundary, or SCP overrides allows.
6. Test with least privilege.
   - Add only the missing actions and resources needed for the workflow.
   - Re-run the automation and verify the next denied action, if any.

**IAM policy areas that commonly cause issues**

A workflow can fail even when an identity policy looks correct. The problem may be in the trust policy, a resource policy, or an organization-level control.

- **Identity-based policy**
  - Grants actions to a user, group, or role.
  - Commonly missing `Describe`, `List`, or `PassRole` permissions.
- **Trust policy**
  - Controls who can assume a role.
  - Commonly missing the service principal or the correct `sts:AssumeRole` relationship.
- **Resource-based policy**
  - Controls access on the target resource, such as an S3 bucket policy or KMS key policy.
  - Commonly blocks cross-account automation or service access.
- **Permissions boundary**
  - Limits the maximum permissions a role or user can receive.
  - Can silently cap an otherwise correct policy.
- **Service control policy (SCP)**
  - Applies at the AWS Organizations level.
  - Can deny backup, snapshot, or logging actions across accounts.

**Example: EC2 automation role missing permissions**

A common pattern is an EC2 instance profile used by a script that uploads logs to Amazon S3 and decrypts data with AWS Key Management Service (KMS). The role may allow `s3:PutObject` but not `kms:Decrypt`, causing uploads or reads to fail when the object is encrypted.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "WriteLogsToBucket",
      "Effect": "Allow",
      "Action": [
        "s3:PutObject",
        "s3:AbortMultipartUpload"
      ],
      "Resource": "arn:aws:s3:::my-automation-logs/*"
    },
    {
      "Sid": "AllowDecryptForS3Objects",
      "Effect": "Allow",
      "Action": [
        "kms:Decrypt",
        "kms:GenerateDataKey"
      ],
      "Resource": "arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555"
    }
  ]
}
```

If the script still fails, check whether the KMS key policy also allows the role. For KMS, both the IAM policy and the key policy can matter. This is a frequent exam trap because an IAM allow alone is not always sufficient.

**Example: Systems Manager Automation execution role**

Systems Manager Automation documents often require a role that can describe resources, modify them, and pass a service role to another service. Missing `iam:PassRole` is a classic cause of failure when the automation launches an EC2 action, creates a backup job, or starts another managed service operation.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AutomationReadAccess",
      "Effect": "Allow",
      "Action": [
        "ec2:DescribeInstances",
        "ec2:DescribeVolumes",
        "ssm:GetAutomationExecution"
      ],
      "Resource": "*"
    },
    {
      "Sid": "AllowPassSpecificRole",
      "Effect": "Allow",
      "Action": "iam:PassRole",
      "Resource": "arn:aws:iam::123456789012:role/SSMAutomationServiceRole"
    }
  ]
}
```

Use `iam:PassRole` only when a service needs to assume another role on your behalf. Scope it to the exact role ARN whenever possible.

**Example: AWS Backup permissions**

AWS Backup uses service roles and service-linked roles to access protected resources. If a backup job fails, verify that the role can read the source resource, create snapshots or recovery points, and write to the backup vault. For encrypted backups, KMS permissions are often required.

Typical permissions may include:

- `backup:StartBackupJob`
- `backup:DescribeBackupJob`
- `ec2:CreateSnapshot`
- `rds:CreateDBSnapshot`
- `kms:CreateGrant`
- `kms:Decrypt`
- `kms:GenerateDataKey`
- `logs:CreateLogStream`
- `logs:PutLogEvents`

A backup workflow may also need a KMS grant rather than only direct key usage. If the backup service cannot use the key, inspect the key policy and any grant constraints.

**Trust policy examples**

A role used by a service must trust that service. If the trust relationship is wrong, the service cannot assume the role even if the permissions policy is correct.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "backup.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

For an EC2 instance profile, the trust policy should allow the EC2 service principal:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "ec2.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

If the role is assumed by a human or another role, the trust policy must name the correct AWS principal and the caller must have `sts:AssumeRole`.

**Practical CLI checks**

You can use the AWS Command Line Interface (CLI) to validate the identity, inspect policies, and reproduce the failing action.

```bash
aws sts get-caller-identity
aws iam get-role --role-name SSMAutomationServiceRole
aws iam list-attached-role-policies --role-name SSMAutomationServiceRole
aws iam get-role-policy --role-name SSMAutomationServiceRole --policy-name InlinePolicyName
aws cloudtrail lookup-events --lookup-attributes AttributeKey=EventName,AttributeValue=StartBackupJob
```

Useful commands for testing access:

```bash
aws s3 ls s3://my-automation-logs/
aws s3 cp test.txt s3://my-automation-logs/
aws kms decrypt --ciphertext-blob fileb://ciphertext.bin --key-id arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555
aws ec2 describe-instances --instance-ids i-0123456789abcdef0
```

If a command fails, compare the denied action with the policy statements. The goal is to add the minimum missing permission, not to grant broad access.

**Using CloudTrail for root-cause analysis**

CloudTrail records the API call, the principal, and the error. This is often the fastest way to determine whether the issue is an identity policy, a trust policy, or a resource policy.

Look for:

- `eventSource`
- `eventName`
- `userIdentity`
- `errorCode`
- `errorMessage`
- `requestParameters`
- `resources`

A CloudTrail event can show that the caller is an assumed role session such as `arn:aws:sts::123456789012:assumed-role/SSMAutomationServiceRole/AutomationExecution`. That tells you which role to inspect.

**When to suspect a resource policy or KMS key policy**

Some services require access to the target resource itself, not just the caller’s identity policy.

- **S3 bucket policy**
  - May deny writes from the automation role.
  - May require `aws:SourceArn` or `aws:SourceAccount`.
- **KMS key policy**
  - May not allow the role or service to use the key.
  - May require grants for AWS-managed workflows.
- **SNS topic policy**
  - May block EventBridge or Lambda from publishing notifications.
- **SQS queue policy**
  - May block a service from sending messages.

For encrypted resources, remember that the caller often needs both the service action and the KMS action. For example, writing an encrypted object to S3 may require `s3:PutObject` and `kms:Encrypt` or `kms:GenerateDataKey`.

**Least-privilege design tips**

- Start with the exact service role or instance profile used by the workflow.
- Grant only the actions the workflow needs.
- Scope resources to specific ARNs instead of `*` when possible.
- Use condition keys to restrict access by account, source ARN, or tags.
- Separate read, write, and admin roles for different automation tasks.
- Review permissions after adding new services, encryption, or cross-account targets.

**Exam-focused distinctions**

- **Identity policy problem**
  - The role exists, but the action is not allowed.
- **Trust policy problem**
  - The service or caller cannot assume the role.
- **Resource policy problem**
  - The target resource denies access even though the role allows it.
- **KMS problem**
  - The role may need both IAM permission and key policy permission.
- **SCP problem**
  - The organization may block the action regardless of local permissions.

**Quick troubleshooting checklist**

- Confirm the exact principal running the automation.
- Read the denied action from the error or CloudTrail.
- Check the role trust policy.
- Check attached and inline policies.
- Check resource policies and KMS key policies.
- Check permissions boundaries and SCPs.
- Retest after each minimal change.

**Practical example: EventBridge-triggered Lambda writing to CloudWatch Logs and S3**

A Lambda function triggered by EventBridge may fail because it cannot write logs or access an S3 bucket. The execution role needs permissions for logging and the target resource, and the bucket policy must allow the function if cross-account access is involved.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "LambdaLogging",
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogGroup",
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "*"
    },
    {
      "Sid": "ReadInputAndWriteOutput",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": [
        "arn:aws:s3:::input-bucket/*",
        "arn:aws:s3:::output-bucket/*"
      ]
    }
  ]
}
```

If the function still fails, inspect the bucket policy and any KMS encryption settings on the bucket. A missing bucket policy statement or key policy permission can look like an IAM issue at first glance.

**Key takeaway**

When automation or backup fails, do not guess. Identify the caller, the denied action, and the policy layer that blocks it. In AWS, the fix is often a small, precise permission change, but the real challenge is finding whether the problem is in the identity policy, trust policy, resource policy, KMS key policy, permissions boundary, or SCP.
