## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**What “not triggering” usually means**

A CloudWatch alarm can fail to notify for several different reasons, and the fix depends on whether the problem is with the metric, the alarm configuration, or the action target. In practice, you troubleshoot in this order:

- Confirm the metric is being published.
- Confirm the alarm is evaluating the correct namespace, metric name, dimensions, statistic, and period.
- Confirm the alarm has enough datapoints to evaluate.
- Confirm the alarm action is valid and permitted.
- Confirm the alarm is in the expected state and not stuck in `INSUFFICIENT_DATA`.

CloudWatch alarm states are `OK`, `ALARM`, and `INSUFFICIENT_DATA`. A newly created alarm often starts in `INSUFFICIENT_DATA` until enough datapoints arrive. Composite alarms and some advanced behaviors can affect troubleshooting, so avoid treating the state model as overly simplistic.

**Core alarm evaluation model**

CloudWatch evaluates alarms by comparing metric datapoints against a threshold over a time window. The alarm does not trigger just because a single value exists; it triggers when the evaluation logic matches the configured conditions.

Key settings that affect triggering:

- **Namespace**: The metric must be in the exact namespace the alarm watches.
- **Metric name**: The alarm must reference the correct metric.
- **Dimensions**: Dimensions must match exactly, including case and value.
- **Statistic or extended statistic**: `Average`, `Sum`, `Minimum`, `Maximum`, or percentile-based values must match the use case.
- **Period**: The aggregation interval must align with how often the metric is published.
- **Evaluation periods**: The alarm needs the configured number of consecutive periods.
- **Datapoints to alarm**: The alarm may require only some of the evaluation periods to breach.
- **Treat missing data**: Missing datapoints can keep the alarm in `INSUFFICIENT_DATA`, or cause it to stay `OK`, depending on the setting.

A common exam trap is assuming that a metric exists in CloudWatch, so the alarm should work. The metric may exist, but the alarm can still fail if the dimensions or period do not match the published data.

**First things to verify**

Start with the metric itself. If the metric is not present or is sparse, the alarm cannot evaluate correctly.

- Check whether the service publishes the metric automatically.
- Check whether the metric is custom and requires `PutMetricData`.
- Check whether the metric is in the same Region as the alarm.
- Check whether the dimensions match the source resource exactly.
- Check whether the metric has recent datapoints.

Useful CLI commands:

```bash
aws cloudwatch list-metrics \
  --namespace AWS/EC2 \
  --metric-name CPUUtilization
```

```bash
aws cloudwatch get-metric-statistics \
  --namespace AWS/EC2 \
  --metric-name CPUUtilization \
  --dimensions Name=InstanceId,Value=i-0123456789abcdef0 \
  --start-time 2026-04-26T00:00:00Z \
  --end-time 2026-04-26T01:00:00Z \
  --period 300 \
  --statistics Average
```

For custom metrics, publish a datapoint with valid CLI syntax:

```bash
aws cloudwatch put-metric-data \
  --namespace "MyApp" \
  --metric-data '[
    {
      "MetricName": "CheckoutErrors",
      "Dimensions": [
        {"Name": "Service", "Value": "checkout"}
      ],
      "Value": 1,
      "Unit": "Count"
    }
  ]'
```

If the metric graph shows no datapoints, the alarm history will not help you prove missing datapoints. Alarm history records state changes and action attempts, not raw metric datapoints.

**Alarm configuration mistakes**

Many “alarm not triggering” cases come from a mismatch between the alarm definition and the metric stream.

Common configuration problems:

- **Wrong dimension set**: The alarm watches `InstanceId=i-abc`, but the metric is published with `AutoScalingGroupName=...`.
- **Wrong statistic**: You alarm on `Average`, but the issue is visible only in `Sum`.
- **Wrong period**: The metric publishes every 5 minutes, but the alarm evaluates every 1 minute.
- **Too few datapoints**: The alarm needs 3 of 3 breaching periods, but only 1 datapoint breaches.
- **Threshold direction error**: You configured `GreaterThanThreshold` when the condition should be `LessThanThreshold`.
- **Missing data behavior**: The alarm is configured to ignore missing data, so it never enters `ALARM` during gaps.

Example alarm creation for a custom metric:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name "CheckoutErrorsHigh" \
  --namespace "MyApp" \
  --metric-name "CheckoutErrors" \
  --dimensions Name=Service,Value=checkout \
  --statistic Sum \
  --period 60 \
  --evaluation-periods 3 \
  --datapoints-to-alarm 2 \
  --threshold 5 \
  --comparison-operator GreaterThanOrEqualToThreshold \
  --treat-missing-data notBreaching \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:ops-alerts
```

This alarm triggers only if at least 2 of the last 3 one-minute periods meet or exceed the threshold.

**Missing data and sparse metrics**

Missing data is one of the most common reasons alarms appear “stuck.”

CloudWatch lets you choose how to treat missing datapoints:

- **`missing`**: The alarm may move to `INSUFFICIENT_DATA`.
- **`breaching`**: Missing data counts as bad.
- **`notBreaching`**: Missing data counts as good.
- **`ignore`**: CloudWatch keeps the current state.

Use the setting that matches the metric behavior:

- Use `breaching` for heartbeat-style metrics where silence means failure.
- Use `notBreaching` for metrics that are naturally sparse and should not alert on gaps.
- Use `ignore` when you want to preserve the last known state during temporary gaps.

> Be careful with sparse custom metrics. If your application publishes only on error, the alarm may never evaluate the way you expect unless you design the metric and missing-data policy together.

**Alarm actions and notification targets**

If the alarm enters `ALARM` but nothing happens, the issue is usually the action target rather than the evaluation logic.

Direct alarm actions commonly include:

- **Amazon SNS** topic notifications
- **Auto Scaling** actions
- Some service-specific actions supported by CloudWatch alarms

For SNS-based notifications, the SNS topic must be in the same Region as the alarm for direct alarm action delivery. The topic policy must also allow CloudWatch to publish.

Example SNS topic policy statement:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCloudWatchPublish",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudwatch.amazonaws.com"
      },
      "Action": "sns:Publish",
      "Resource": "arn:aws:sns:us-east-1:123456789012:ops-alerts"
    }
  ]
}
```

If the policy is missing or too restrictive, the alarm can change state but fail to deliver the notification.

Things to verify for actions:

- The alarm has an action configured for the `ALARM` state.
- The target ARN is correct.
- The target exists in the same Region when required.
- The target policy allows the publish or invocation.
- The IAM role or service permissions are correct for the action type.

**How to inspect alarm state and history**

Use the CLI to confirm the alarm state and recent transitions.

```bash
aws cloudwatch describe-alarms \
  --alarm-names "CheckoutErrorsHigh"
```

```bash
aws cloudwatch describe-alarm-history \
  --alarm-name "CheckoutErrorsHigh" \
  --history-item-type StateUpdate
```

Alarm history is useful for seeing:

- State transitions
- Action attempts
- Configuration changes

It does not show the underlying metric datapoints. If you need datapoint-level troubleshooting, use metric graphs, `get-metric-statistics`, or `get-metric-data`.

**Practical troubleshooting workflow**

1. Confirm the alarm is in the correct Region.
2. Confirm the metric exists and has recent datapoints.
3. Confirm the namespace, metric name, and dimensions match exactly.
4. Confirm the statistic and period match the metric publication pattern.
5. Confirm the threshold and comparison operator are correct.
6. Confirm the evaluation periods and datapoints-to-alarm settings are realistic.
7. Confirm missing-data handling matches the metric behavior.
8. Confirm the alarm action target exists and is permitted.
9. Confirm the alarm has actually entered `ALARM`.
10. Review alarm history for state changes and action failures.

**Common real-world scenarios**

A few patterns appear often in operations work and exam questions:

- **EC2 CPU alarm never fires**
  - The instance may not be publishing the expected metric because the alarm points to the wrong `InstanceId`.
  - The threshold may be too high for the workload.
  - The period may be too short for the workload’s burst pattern.

- **Custom application error alarm stays in `INSUFFICIENT_DATA`**
  - The application does not publish the metric often enough.
  - The metric is only emitted on errors, so missing-data behavior prevents evaluation.
  - The namespace or dimensions do not match the alarm.

- **Alarm enters `ALARM` but no email arrives**
  - The SNS topic policy does not allow CloudWatch to publish.
  - The subscription is not confirmed.
  - The alarm action points to the wrong topic ARN.
  - The topic and alarm are in different Regions for direct action delivery.

- **Alarm never leaves `OK` even though the service is unhealthy**
  - The metric is the wrong one for the symptom.
  - The statistic hides the issue, such as using `Average` when `Maximum` would reveal spikes.
  - The evaluation window is too long or too short for the failure pattern.

List alarms:

```bash
aws cloudwatch describe-alarms \
  --state-value ALARM
```

Get recent metric data:

```bash
aws cloudwatch get-metric-data \
  --metric-data-queries '[
    {
      "Id": "cpu1",
      "MetricStat": {
        "Metric": {
          "Namespace": "AWS/EC2",
          "MetricName": "CPUUtilization",
          "Dimensions": [
            {"Name": "InstanceId", "Value": "i-0123456789abcdef0"}
          ]
        },
        "Period": 300,
        "Stat": "Average"
      },
      "ReturnData": true
    }
  ]' \
  --start-time 2026-04-26T00:00:00Z \
  --end-time 2026-04-26T01:00:00Z
```

Create a test alarm for validation:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name "TestAlarm" \
  --namespace "MyApp" \
  --metric-name "Heartbeat" \
  --statistic Sum \
  --period 60 \
  --evaluation-periods 1 \
  --threshold 0 \
  --comparison-operator LessThanOrEqualToThreshold \
  --treat-missing-data breaching \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:ops-alerts
```

This pattern is useful for heartbeat monitoring, where missing data should be treated as a failure.

**Exam-focused takeaways**

- CloudWatch alarms depend on correct metric publication, not just alarm creation.
- `OK`, `ALARM`, and `INSUFFICIENT_DATA` are the key states to know.
- Alarm history shows state changes and action attempts, not datapoints.
- SNS is the most common direct notification target, and the topic must be in the same Region for direct alarm actions.
- Missing data handling is often the difference between a working alarm and a silent one.
- Dimension mismatches are a frequent cause of alarms that never trigger.

**Quick mental model**

If the alarm is silent, ask three questions:

- Is the metric present?
- Is the alarm evaluating the right data?
- Can the action target actually receive the notification?

That sequence resolves most CloudWatch alarm troubleshooting cases.
