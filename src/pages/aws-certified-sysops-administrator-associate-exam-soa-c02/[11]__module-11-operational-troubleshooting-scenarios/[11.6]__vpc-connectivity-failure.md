## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**VPC connectivity failure: what it usually means**

A VPC connectivity failure is any break in the path between a source and destination inside AWS or between AWS and an external network. The failure may be caused by routing, security controls, DNS, endpoint configuration, appliance insertion, or an upstream network issue. In practice, you troubleshoot by proving each hop in the path: name resolution, route selection, security filtering, and return traffic.

Common symptoms include:

- Instances cannot reach other subnets, peered VPCs, on-premises networks, or the internet.
- Applications time out even though instances are running and reachable by SSH or SSM.
- DNS resolves to the wrong address, or resolves correctly but traffic still fails.
- Connectivity works in one direction only, which usually indicates a return-path problem.

**Core troubleshooting model**

Use a layered approach so you do not confuse DNS failures with routing failures or security-group issues. Start with the simplest checks and move outward.

1. Confirm the destination name and IP address.
2. Verify the route table for the source subnet.
3. Check security groups and network ACLs (NACLs).
4. Validate the target-side route table and return path.
5. Inspect appliances, VPN, Direct Connect, Transit Gateway (TGW), and endpoint settings.
6. Use flow logs and reachability tools to prove where packets stop.

**Routing failures and route table analysis**

Route tables are the most common cause of VPC connectivity problems. A route must exist for both the forward path and the return path. If the route is missing, too specific, or points to the wrong target, traffic fails.

Important route targets include:

- `local` for intra-VPC traffic
- Internet gateway (`igw-...`) for public internet access
- NAT gateway for private subnet outbound internet access
- Virtual private gateway (`vgw-...`) for Site-to-Site VPN or Direct Connect
- Transit Gateway attachment for hub-and-spoke designs
- VPC peering connection (`pcx-...`) for peered VPC traffic
- Interface endpoint ENI for PrivateLink-based access

A route can also enter a **blackhole** state. This means the route exists in the table, but the target is no longer usable. Blackhole states are especially important in peering, TGW, VPN, and endpoint scenarios.

- **VPC peering blackhole**: the peering connection was deleted or is not active, but the route remains.
- **TGW blackhole**: the attachment or propagated route is no longer valid.
- **VPN blackhole**: the VPN tunnel or virtual private gateway path is down, but the route still appears.
- **Endpoint-related blackhole**: the endpoint or associated target is removed, leaving stale routing references in some designs.

Use the console or CLI to inspect route state:

```bash
aws ec2 describe-route-tables \
  --filters Name=vpc-id,Values=vpc-0123456789abcdef0
```

Look for routes where `State` is `blackhole` or where the target resource is missing. In advanced architectures, also verify propagated routes from TGW or VPN because a route may appear present while the underlying attachment is unhealthy.

> A route table entry that looks correct is not enough. If the target is blackholed, traffic still fails.

**Security groups and network ACLs**

Security groups are stateful, while NACLs are stateless. That difference matters when traffic works in one direction but not the other.

- **Security groups** allow return traffic automatically for established connections.
- **NACLs** require explicit inbound and outbound rules for both directions.

Typical mistakes include:

- Missing ephemeral port return traffic in NACLs.
- Allowing inbound traffic to the destination but blocking outbound responses.
- Referencing the wrong security group or CIDR block.
- Using overly restrictive rules after a subnet redesign.

For example, if an instance initiates HTTPS to a private endpoint or external service, the return traffic uses ephemeral ports on the client side. If the NACL blocks ephemeral ports, the connection times out even though the destination port is allowed.

**DNS and DHCP options set misconfiguration**

DNS issues often look like connectivity failures because the application cannot resolve the endpoint it needs. In AWS, VPC DNS behavior depends on the VPC settings and the DHCP options set.

Key points:

- `enableDnsSupport` must be enabled for DNS resolution in the VPC.
- `enableDnsHostnames` is important when you need AWS-provided private DNS names for instances.
- A custom **DHCP options set** can override the DNS servers handed to instances.
- If you point DHCP to custom DNS servers that are unreachable from the VPC, all name resolution can fail.
- If custom DNS servers do not know about private hosted zones or internal service names, applications may resolve public addresses instead of private ones.

A common failure pattern is this:

- The instance receives custom DNS server addresses from DHCP.
- Those DNS servers are on-premises and reachable only through VPN.
- The VPN is down or the route is missing.
- DNS queries fail, so the application appears to have a connectivity problem.

Check the DHCP options set attached to the VPC:

```bash
aws ec2 describe-vpcs --vpc-ids vpc-0123456789abcdef0
aws ec2 describe-dhcp-options --dhcp-options-ids dopt-0123456789abcdef0
```

On the instance, verify what DNS server it is using:

```bash
cat /etc/resolv.conf
```

If you suspect DNS, test resolution directly:

```bash
nslookup internal-app.example.com
dig internal-app.example.com
```

If the name resolves to an unexpected public IP, the issue may be split-horizon DNS, private hosted zone association, or custom resolver behavior rather than routing.

**VPC Reachability Analyzer**

VPC Reachability Analyzer is a diagnostic tool that checks the network path between a source and destination and identifies where the path is blocked. It is especially useful when you want to prove whether the issue is routing, security, or an unsupported path.

Use it when:

- You need to validate connectivity before a change.
- You suspect a route table, security group, NACL, or gateway issue.
- You want a faster root-cause hint than manual hop-by-hop testing.

It can help identify:

- Missing routes
- Security group blocks
- NACL blocks
- Unsupported or incomplete paths
- Problems involving ENIs, instances, gateways, and some endpoint scenarios

Example CLI workflow:

```bash
aws ec2 create-network-insights-path \
  --source eni-0123456789abcdef0 \
  --destination eni-0fedcba9876543210 \
  --protocol tcp \
  --destination-port 443

aws ec2 start-network-insights-analysis \
  --network-insights-path-id nip-0123456789abcdef0
```

Then inspect the analysis result for the first blocking component:

```bash
aws ec2 describe-network-insights-analyses \
  --network-insights-analysis-ids nia-0123456789abcdef0
```

Use Reachability Analyzer as a proof tool, not just a guessing tool. It is most valuable when you already know the intended source and destination.

**Interface VPC endpoints and Private DNS**

Interface endpoints are a common source of confusion because traffic may fail even when the endpoint exists. The key question is whether the client resolves the service name to the endpoint ENI or to the public AWS service endpoint.

With **Private DNS enabled**:

- Standard service names such as `ssm.<region>.amazonaws.com` resolve to private IP addresses in the VPC.
- Traffic stays inside AWS private networking.
- You must ensure the VPC has DNS support enabled.

With **Private DNS disabled**:

- The standard service name resolves to the public AWS endpoint.
- You must use the endpoint-specific DNS name or a custom private hosted zone if you want private routing.
- If the subnet has no internet access, the public resolution path fails.

Example verification steps:

```bash
nslookup ssm.us-east-1.amazonaws.com
dig ssm.us-east-1.amazonaws.com
```

If Private DNS is enabled, the answer should typically map to private IPs in the endpoint ENI range. You can confirm the endpoint ENIs with:

```bash
aws ec2 describe-vpc-endpoints \
  --vpc-endpoint-ids vpce-0123456789abcdef0
```

Then inspect the ENI addresses:

```bash
aws ec2 describe-network-interfaces \
  --network-interface-ids eni-0123456789abcdef0 eni-0fedcba9876543210
```

A practical failure pattern:

- The endpoint exists and the security group allows traffic.
- Private DNS is disabled.
- The instance resolves the public service name.
- The subnet has no NAT gateway or internet route.
- The connection times out.

**VPN, Direct Connect, and Transit Gateway troubleshooting**

Hybrid connectivity failures often come from route propagation, tunnel state, BGP (Border Gateway Protocol) issues, or asymmetric routing.

For **Site-to-Site VPN**:

- Check tunnel status and phase 1/phase 2 health.
- Verify that the customer gateway and virtual private gateway or TGW attachment are configured correctly.
- Confirm that routes are propagated into the VPC route table or TGW route table.
- Ensure on-premises routers advertise the correct prefixes and accept AWS prefixes.

Common failure patterns:

- One tunnel is down, but the other is up and carrying traffic poorly.
- The on-premises firewall blocks UDP 500/4500 or ESP.
- BGP is established, but the route is not being advertised or accepted.
- The VPC route table points to the VPN gateway, but the on-premises return route is missing.

For **Direct Connect**:

- Verify the physical link, virtual interface, and BGP session.
- Confirm that the correct prefixes are advertised on both sides.
- Check whether the private virtual interface is associated with the right virtual private gateway or TGW.
- Remember that Direct Connect does not automatically fix return routing; on-premises routes must still point back to AWS.

Common failure patterns:

- BGP is down because of a wrong peer IP or ASN (Autonomous System Number).
- The virtual interface is up, but the route is not propagated to the intended VPC or TGW route table.
- The on-premises network prefers a different path, causing asymmetric routing.

For **Transit Gateway**:

- Verify attachment state for each VPC, VPN, or Direct Connect attachment.
- Check TGW route tables for propagation and association.
- Confirm that the source attachment can reach the destination attachment through the correct TGW route table.
- Look for blackhole routes when an attachment is deleted or detached.

A simple TGW troubleshooting checklist:

1. Confirm the source subnet route points to the TGW attachment.
2. Confirm the TGW route table has a route to the destination attachment.
3. Confirm the destination VPC route table has a return route back to the source CIDR.
4. Check for overlapping CIDRs that prevent clean routing.
5. Inspect whether appliance insertion or inspection routing is forcing traffic through a middlebox.

**Advanced architectures: appliances, inspection, and appliance mode**

In more advanced designs, traffic may intentionally pass through a firewall, IDS/IPS, third-party appliance, or AWS Network Firewall. These designs add extra failure points.

Possible causes of path failure include:

- **AWS Network Firewall** rule groups blocking the flow.
- A **third-party appliance** dropping traffic because of policy, licensing, or health issues.
- **Appliance mode** not enabled where symmetric routing is required.
- Return traffic bypassing the appliance and being dropped as asymmetric traffic.
- Incorrect route table associations that send traffic to the wrong inspection VPC.

When using TGW with inspection appliances, verify:

- The appliance ENIs are healthy.
- The appliance subnet route tables send traffic back to TGW correctly.
- Appliance mode is enabled if the design requires symmetric flow through the same attachment.
- The firewall policy allows the protocol, port, and destination.

A common failure pattern is asymmetric routing:

- Client traffic enters the inspection VPC through TGW.
- The appliance forwards it to the destination.
- The return path bypasses the appliance because of a different route table.
- The appliance or stateful firewall drops the return packet.

**Practical CLI checks**

Use these commands to gather evidence quickly:

```bash
aws ec2 describe-route-tables --filters Name=association.subnet-id,Values=subnet-0123456789abcdef0
aws ec2 describe-network-acls --filters Name=association.subnet-id,Values=subnet-0123456789abcdef0
aws ec2 describe-security-groups --group-ids sg-0123456789abcdef0
aws ec2 describe-vpc-endpoints --filters Name=vpc-id,Values=vpc-0123456789abcdef0
aws ec2 describe-vpn-connections --vpn-connection-ids vpn-0123456789abcdef0
aws ec2 describe-transit-gateway-attachments --filters Name=resource-id,Values=vpc-0123456789abcdef0
```

For packet-level evidence, use VPC Flow Logs. They help you see accepted and rejected traffic, but they do not tell you the full path.

```bash
aws ec2 create-flow-logs \
  --resource-type VPC \
  --resource-ids vpc-0123456789abcdef0 \
  --traffic-type ALL \
  --log-destination-type cloud-watch-logs \
  --log-group-name /aws/vpc/flowlogs
```

Flow log interpretation:

- `ACCEPT` means the packet passed the logged layer.
- `REJECT` often points to NACLs or security group issues, depending on the log context.
- Missing logs can indicate the traffic never reached the interface you expected.

**How to isolate the failure domain**

Use a structured test plan so you do not chase symptoms.

1. Test DNS resolution from the source host.
2. Test TCP reachability to the destination port with `curl`, `nc`, or `telnet`.
3. Compare source and destination route tables.
4. Check security groups and NACLs for both directions.
5. Validate endpoint, VPN, TGW, or Direct Connect health.
6. Use Reachability Analyzer to confirm the first blocking hop.
7. Review flow logs for rejected or missing traffic.

Example connectivity tests:

```bash
nc -vz internal-app.example.com 443
curl -vk https://internal-app.example.com
traceroute 10.0.2.15
```

For AWS service endpoints, test the resolved address and the service response separately. A successful DNS lookup does not guarantee that the endpoint is reachable.

**Common exam-style failure patterns**

- A private subnet cannot reach the internet because it has no NAT gateway route.
- An instance cannot reach another VPC because the peering route is missing on one side.
- Traffic to an on-premises subnet fails because the VPN route is present but the on-premises return route is missing.
- A route shows `blackhole` after a TGW attachment or peering connection is deleted.
- DNS fails because the DHCP options set points to unreachable custom resolvers.
- An interface endpoint exists, but Private DNS is disabled and the client resolves the public AWS hostname.
- A firewall appliance drops return traffic because appliance mode or symmetric routing is not configured correctly.

**Quick reference: what to check first**

- **Name resolution**: does the hostname resolve to the expected private or public address?
- **Source route table**: is there a valid route to the destination?
- **Destination route table**: is there a return route?
- **Security controls**: do SGs and NACLs allow the flow and return ports?
- **Path devices**: are VPN, DX, TGW, firewall, or appliance components healthy?
- **Route state**: is any route in `blackhole`?
- **Endpoint behavior**: is Private DNS enabled where needed?

**Key takeaway**

Most VPC connectivity failures are not caused by a single AWS feature. They are usually the result of a mismatch between DNS, routing, security policy, and return-path design. The fastest way to solve them is to verify each layer in order and use Reachability Analyzer, flow logs, and route-state checks to prove where the path breaks.
