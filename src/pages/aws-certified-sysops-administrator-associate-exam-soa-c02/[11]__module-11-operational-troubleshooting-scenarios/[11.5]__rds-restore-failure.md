## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**RDS restore failure: what it means**

Amazon Relational Database Service (Amazon RDS) restore operations fail when AWS cannot create a new database instance or cluster from a backup source, snapshot, or replica state. In exam scenarios, the key idea is that restore is not an in-place overwrite of the existing database. RDS creates a <mark>new DB instance or cluster</mark>, and failures usually come from permissions, encryption, engine/version mismatch, storage constraints, subnet/AZ limitations, or backup retention settings.

Restore behavior differs by source:

- **Automated backups** support point-in-time recovery (PITR, point-in-time recovery) within the retention window.
- **Manual snapshots** restore a specific saved state, but not to an arbitrary timestamp.
- **Read replicas** can be promoted or used as recovery sources in some engines, but they are not the same as backups.
- **Aurora** restores at the cluster level and has different mechanics from standard RDS instances.

> A restore does not replace the original DB instance. If you need the old instance preserved, keep it running or snapshot it first.

**Core restore types and what they support**

Use the restore source to match the recovery goal. The exam often tests whether you choose the right recovery method for the required time window.

| Restore source | What it restores | Typical use | Key limitation |
|---|---|---|---|
| Automated backup | Snapshot plus transaction logs for PITR | Recover to a specific time | Requires automated backups to be enabled and retained |
| Manual snapshot | Exact saved state at snapshot time | Clone or recover a known-good state | No PITR beyond snapshot time |
| Read replica | Replication state, then promotion | Reduce recovery time for some workloads | Not a backup substitute; replication lag can lose recent writes |
| Aurora cluster snapshot | Cluster data and metadata | Restore Aurora cluster | Cluster-level restore, not instance-level overwrite |

**Automated backups and PITR availability**

Automated backups are the foundation for point-in-time restore. If you disable automated backups, you lose the ability to restore to a specific time from transaction logs. You can still restore from any existing manual snapshots, but only to the snapshot time.

Important exam points:

- **Backup retention period** controls how long automated backups and logs remain available.
- Setting retention to `0` disables automated backups for many RDS engines.
- PITR is only available while the backup window and logs are retained.
- Deleting a DB instance can optionally retain the final automated backup depending on the deletion settings and engine behavior.

Example CLI to inspect backup retention:

```bash
aws rds describe-db-instances \
  --db-instance-identifier mydb \
  --query 'DBInstances[0].BackupRetentionPeriod'
```

If the value is `0`, PITR is not available from automated backups.

**Restore failure causes you should recognize**

Most restore failures fall into a few predictable categories.

- **Encryption mismatch**
  - A snapshot encrypted with AWS Key Management Service (AWS KMS) requires access to the same key or a valid copy workflow.
  - Shared encrypted snapshots cannot be restored directly by another account unless the key policy and snapshot sharing are configured correctly.
- **Engine or version mismatch**
  - You cannot restore a snapshot into an unsupported engine version.
  - Some major version upgrades require a restore-and-upgrade workflow rather than a direct restore.
- **Storage and configuration mismatch**
  - The target storage type, allocated storage, or IOPS (input/output operations per second) settings may not be valid for the selected engine or instance class.
  - Some storage changes are allowed during restore, but not all combinations are valid.
- **Subnet or AZ constraints**
  - The target DB subnet group must contain subnets in the required Availability Zones (AZs).
  - A restore into a specific AZ can fail if the subnet group or capacity is unavailable.
- **Instance class limitations**
  - Not every DB instance class supports every engine, edition, or feature.
  - Restoring to a smaller class may fail if the engine requires more memory or unsupported capabilities.
- **Multi-AZ and topology constraints**
  - A restore can create a new Multi-AZ deployment, but the target configuration must be valid for the engine and region.
  - Some restore paths cannot preserve the exact topology of the source, especially when moving between single-AZ and Multi-AZ configurations.
- **Read replica state**
  - A replica may be lagging or unavailable, so promoting it can lose recent transactions.
  - A failed replica does not guarantee a clean recovery point.
- **Permission and sharing issues**
  - The IAM (Identity and Access Management) principal may lack `rds:RestoreDBInstanceFromDBSnapshot`, `rds:RestoreDBInstanceToPointInTime`, or KMS permissions.
  - Cross-account restore from shared snapshots often fails when the snapshot is shared but the KMS key is not usable by the target account.

**Restore from automated backup vs manual snapshot**

These are commonly confused on the exam.

- **Automated backup restore**
  - Restores to a specific time within the retention window.
  - Uses the latest automated snapshot plus transaction logs.
  - Best when you need to recover from accidental deletes, bad writes, or corruption at a known time.
- **Manual snapshot restore**
  - Restores the exact state captured when the snapshot was taken.
  - Best for cloning environments, long-term retention, or a known-good checkpoint.
  - Does not support recovery to arbitrary timestamps.

Practical rule:

- Use **automated backup/PITR** when you need precision.
- Use **manual snapshot** when you need a stable checkpoint or long-lived backup.

**Restore creates a new DB instance**

RDS restore operations always create a new DB instance or cluster. You cannot restore into an existing DB instance identifier.

This matters because:

- You must choose a new DB instance identifier.
- Application endpoints may need to be updated after restore.
- DNS or connection strings may still point to the failed source instance.
- If you want to preserve the original instance name, you typically rename or replace resources after validation.

Example restore from snapshot:

```bash
aws rds restore-db-instance-from-db-snapshot \
  --db-instance-identifier restored-prod-db \
  --db-snapshot-identifier prod-snapshot-2026-04-26 \
  --db-instance-class db.m6g.large \
  --availability-zone us-east-1a
```

**Point-in-time restore workflow**

Use PITR when you need to recover to a moment before corruption or accidental deletion.

1. Identify the approximate failure time.
2. Confirm automated backups are enabled and retained long enough.
3. Choose a restore time before the incident.
4. Restore to a new DB instance.
5. Validate data consistency and application behavior.
6. Cut over traffic to the restored instance.

Example CLI:

```bash
aws rds restore-db-instance-to-point-in-time \
  --source-db-instance-identifier mydb \
  --target-db-instance-identifier mydb-pitr-recovery \
  --restore-time 2026-04-26T14:15:00Z \
  --db-instance-class db.m6g.large
```

Common PITR failure points:

- Backup retention expired before the target time.
- Automated backups were disabled.
- The source instance was deleted and no retained backup exists.
- The target configuration is invalid for the engine or region.

**Cross-Region and cross-account restore**

Cross-Region restore is usually snapshot-based. Cross-account restore often depends on snapshot sharing and KMS permissions.

Typical workflow for a shared encrypted snapshot:

1. Share the manual snapshot with the target AWS account.
2. Ensure the KMS key policy allows the target account to use the key, or copy the snapshot with a key the target can access.
3. In the target account, copy the shared snapshot to a target Region if needed.
4. Restore a new DB instance from the copied snapshot.

Common failure points:

- The snapshot is shared, but the KMS key is not shared or not usable.
- The target Region does not have the copied snapshot.
- The target account lacks permission to copy or restore the snapshot.
- The source engine or version is not supported in the destination Region.

For encrypted snapshots, a direct restore across accounts is often blocked unless the encryption workflow is handled correctly. In practice, copying the snapshot with a target-owned KMS key is the safest pattern.

**Read replicas and recovery scenarios**

Read replicas are useful for read scaling and some recovery patterns, but they are not a substitute for backups.

Use a read replica when:

- You want to offload read traffic.
- You need a warm standby with replication lag.
- You want a faster recovery option than restoring from backup in some cases.

Be careful:

- Replication lag means the replica may not contain the latest committed transactions.
- A replica failure can still require snapshot restore if the replica is corrupted or unavailable.
- Promoting a replica changes its role; it is no longer a replica after promotion.

Example promotion command:

```bash
aws rds promote-read-replica \
  --db-instance-identifier mydb-replica
```

Exam angle:

- If the question asks for the most recent recoverable state, backups/PITR are usually better.
- If the question asks for a low-latency standby, a replica may be appropriate.
- If the replica is the only surviving copy, promotion may be the fastest path, but not the most complete one.

**Engine-specific restore constraints**

Different engines have different restore rules. The exam may not require every detail, but you should know the broad differences.

- **MySQL / MariaDB**
  - Commonly restored from snapshots or PITR.
  - Version compatibility matters for restore and upgrade paths.
  - Storage and parameter group compatibility can affect restore success.
- **PostgreSQL**
  - PITR is supported through automated backups.
  - Major version differences often require careful restore-and-upgrade planning.
  - Extensions and parameter settings may need validation after restore.
- **Oracle**
  - Restore behavior can be constrained by edition, option groups, and storage configuration.
  - Some recovery workflows are more sensitive to parameter and option compatibility.
- **SQL Server**
  - Restore must respect edition, version, and storage constraints.
  - Native backup/restore concepts differ from snapshot-based RDS restore, but RDS snapshot restore still creates a new instance.
- **Aurora**
  - Restores at the cluster level from cluster snapshots or PITR.
  - Aurora replicas and cluster endpoints behave differently from standard RDS replicas.
  - You restore a new cluster, then create or attach instances as needed.

If the exam asks which restore is supported, check:

- Engine family
- Major version
- Edition
- Storage type
- Whether the source is a DB instance or Aurora cluster

**Storage, AZ, and instance class limitations**

Restore can fail when the target configuration is not compatible with the source or with the selected engine.

- **Instance class**
  - The chosen class must support the engine and workload.
  - Some classes are not available in every Region.
- **Availability Zone**
  - If you specify an AZ, the restore depends on capacity in that AZ.
  - A subnet group must include valid subnets for the deployment.
- **Storage**
  - Provisioned IOPS (PIOPS) and storage type combinations must be valid.
  - Some engines require minimum storage sizes or specific storage families.
- **Multi-AZ**
  - Restoring into Multi-AZ may require additional capacity and valid subnet coverage.
  - If the target Region or AZ lacks capacity, the restore can fail even when the snapshot is valid.

A common exam trap is assuming that a valid snapshot guarantees a valid restore. It does not. The target environment must also be valid.

**Troubleshooting failed restore events**

When a restore fails, check the RDS event messages, CloudTrail, and the DB instance status. The event text often points to the exact blocker.

Example failure-style message:

> `RestoreDBInstanceFromDBSnapshot failed: InvalidParameterCombination: Cannot restore encrypted snapshot to target account without access to the KMS key.`

How to interpret it:

- The snapshot is encrypted.
- The target account cannot use the source KMS key.
- You need to adjust key sharing or copy the snapshot with a target-owned key.

Another example:

> `RestoreDBInstanceToPointInTime failed: Backup retention period is not enabled for source DB instance.`

How to interpret it:

- PITR is unavailable because automated backups were disabled or retention expired.
- You must use a manual snapshot or another recovery source.

Useful checks:

```bash
aws rds describe-events \
  --source-type db-instance \
  --source-identifier mydb \
  --duration 60
```

```bash
aws rds describe-db-snapshots \
  --db-instance-identifier mydb \
  --snapshot-type manual
```

```bash
aws rds describe-db-instances \
  --db-instance-identifier mydb \
  --query 'DBInstances[0].{Status:DBInstanceStatus,AZ:AvailabilityZone,Class:DBInstanceClass,BackupRetention:BackupRetentionPeriod}'
```

**Practical restore decision guide**

Choose the recovery method based on the failure scenario.

- **Accidental delete or bad update within retention window**
  - Use PITR from automated backups.
- **Need a stable clone for testing**
  - Restore from a manual snapshot.
- **Replica is healthy but primary is down**
  - Consider promoting the replica if data loss from lag is acceptable.
- **Encrypted snapshot shared across accounts**
  - Copy the snapshot with correct KMS permissions, then restore.
- **Restore fails after changing instance class or AZ**
  - Validate engine support, subnet group, storage type, and capacity.
- **Need to recover after backup retention was disabled**
  - Use the latest manual snapshot, if available.

**Exam-focused pitfalls**

- **Do not assume restore is in-place**
  - RDS restore creates a new resource.
- **Do not confuse replicas with backups**
  - Replicas help availability, not guaranteed point-in-time recovery.
- **Do not assume encrypted snapshots are portable**
  - KMS permissions are part of the restore path.
- **Do not assume PITR works forever**
  - It depends on backup retention and automated backup enablement.
- **Do not ignore engine differences**
  - Aurora, Oracle, PostgreSQL, MySQL, and SQL Server have different restore constraints.

**Quick CLI reference**

Restore from snapshot:

```bash
aws rds restore-db-instance-from-db-snapshot \
  --db-instance-identifier restored-db \
  --db-snapshot-identifier prod-snapshot \
  --db-instance-class db.t3.medium
```

Restore to point in time:

```bash
aws rds restore-db-instance-to-point-in-time \
  --source-db-instance-identifier prod-db \
  --target-db-instance-identifier prod-db-recovery \
  --use-latest-restorable-time
```

Promote a read replica:

```bash
aws rds promote-read-replica \
  --db-instance-identifier prod-replica
```

Copy a snapshot across Regions:

```bash
aws rds copy-db-snapshot \
  --source-db-snapshot-identifier arn:aws:rds:us-east-1:123456789012:snapshot:prod-snapshot \
  --target-db-snapshot-identifier prod-snapshot-copy \
  --source-region us-east-1 \
  --kms-key-id arn:aws:kms:us-west-2:123456789012:key/abcd-1234
```

**What to remember for the exam**

- Restore creates a <mark>new DB instance</mark>, not an overwrite.
- Automated backups enable PITR; manual snapshots do not.
- Backup retention and automated backup settings directly affect recovery options.
- Cross-account encrypted snapshot restore usually fails without correct KMS access.
- Read replicas are useful for availability, but they are not a full backup strategy.
- Engine, version, storage, AZ, and instance class compatibility all matter during restore.
