## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Instance metadata overview**

Amazon EC2 instance metadata is a local service that exposes information about the running instance, its networking, and its environment. You use it from inside the instance, not over the public internet. It is a common troubleshooting tool because it helps you confirm what AWS has attached to the instance and what configuration the instance sees at runtime.

The metadata service is especially useful when you need to verify:
- Instance identity and region
- Private and public IP addresses
- Security group and subnet-related details
- IAM role credentials delivered to the instance
- User data and launch-time configuration

The current recommended service is **IMDSv2** (Instance Metadata Service v2). It uses a session-oriented token flow and is more resistant to SSRF (Server-Side Request Forgery) attacks than IMDSv1.

**IMDSv2 usage**

To query metadata with IMDSv2, you first request a token, then include that token in subsequent requests.

```bash
TOKEN=$(curl -sX PUT "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")

curl -s \
  -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/meta-data/instance-id
```

Common metadata paths include:
- `latest/meta-data/instance-id`
- `latest/meta-data/ami-id`
- `latest/meta-data/hostname`
- `latest/meta-data/local-ipv4`
- `latest/meta-data/public-ipv4`
- `latest/meta-data/iam/security-credentials/`
- `latest/user-data`

Use metadata when you need to distinguish between what you intended to launch and what the instance actually received. For example, if an application cannot reach a database, metadata can confirm whether the instance is in the expected subnet or whether it has the expected private address.

> If IMDS is disabled or restricted by instance metadata options, metadata queries fail even though the instance is otherwise healthy. That is often a deliberate security setting.

**Instance metadata options**

Metadata access is controlled by instance metadata options. These settings determine whether IMDS is enabled, which version is required, and how the service is reached.

Key options include:
- `HttpTokens`
  - `required` forces IMDSv2
  - `optional` allows IMDSv1 or IMDSv2
- `HttpEndpoint`
  - `enabled` allows metadata access
  - `disabled` blocks metadata access
- `HttpPutResponseHopLimit`
  - limits how many network hops the token response can traverse
  - helps reduce exposure to containers or nested network paths
- `InstanceMetadataTags`
  - can expose instance tags through metadata when enabled

A practical hardening pattern is to require IMDSv2 and disable unnecessary metadata exposure.

```bash
aws ec2 modify-instance-metadata-options \
  --instance-id i-0123456789abcdef0 \
  --http-tokens required \
  --http-endpoint enabled \
  --http-put-response-hop-limit 1
```

**System logs and boot diagnostics**

EC2 system logs help you inspect early boot output, kernel messages, and startup failures. They are useful when an instance fails to boot, hangs during initialization, or cannot be reached over SSH or RDP.

System logs are often the first place to look when:
- The instance is stuck in a boot loop
- Cloud-init or user data fails
- A kernel panic occurs
- Filesystems fail to mount
- Network initialization does not complete

You can retrieve the system log from the EC2 console or with the CLI.

```bash
aws ec2 get-console-output \
  --instance-id i-0123456789abcdef0
```

For newer Nitro-based instances, console output may be limited after boot, but it still provides valuable startup information. If the instance uses a serial console or supported boot diagnostics, you may see additional output depending on configuration and platform support.

Typical log content can include:
- BIOS or UEFI startup messages
- Kernel boot messages
- `cloud-init` status
- Service startup failures
- Login prompts or panic traces

A useful troubleshooting pattern is to compare the system log with the expected boot sequence. If user data is supposed to install packages, but the log shows DNS failures or package repository errors, the issue is likely in network or repository access rather than EC2 itself.

**Status checks**

EC2 status checks tell you whether AWS can see the instance as healthy at the infrastructure and operating system levels. They are a fast way to separate host problems from guest OS problems.

There are two primary status checks:

- **System status check**
  - monitors the AWS infrastructure hosting the instance
  - checks for problems such as hardware failure, network reachability on the host, or underlying system issues
  - if it fails, the issue is usually outside the guest OS

- **Instance status check**
  - monitors the guest operating system
  - checks whether the OS is reachable and responding
  - if it fails, the issue is usually inside the instance, such as a misconfigured network stack, corrupted filesystem, or failed boot process

You can view status checks in the EC2 console or query them with the CLI.

```bash
aws ec2 describe-instance-status \
  --instance-ids i-0123456789abcdef0 \
  --include-all-instances
```

The output includes fields such as:
- `SystemStatus`
- `InstanceStatus`
- `Details`
- `Status`

A common interpretation pattern is:
- Both checks pass: the instance and host are healthy
- System check fails, instance check passes: investigate AWS infrastructure or stop/start the instance
- Instance check fails, system check passes: investigate the guest OS, boot process, or application-level startup dependencies
- Both fail: the instance may be severely impaired or unreachable at multiple layers

**What status checks do and do not mean**

Status checks are not application health checks. They do not tell you whether Apache, Nginx, a database, or your custom service is working correctly. They only indicate whether the instance and underlying host are healthy enough at the infrastructure and OS reachability level.

That distinction matters in operations:
- A web server can pass both status checks and still return HTTP 500 errors
- An instance can fail the instance check while the application logs show a kernel or filesystem issue
- A system check failure can occur even if the OS was previously healthy, because the host itself has a problem

**Troubleshooting workflow**

When an EC2 instance is unhealthy, use a layered approach.

1. Check the status checks.
   - Determine whether the failure is system-level or instance-level.
   - Use this to narrow the scope of investigation.

2. Review the system log.
   - Look for boot errors, kernel panics, or user data failures.
   - Confirm whether the OS reached a normal startup state.

3. Query instance metadata from inside the instance if possible.
   - Verify instance ID, IP addresses, IAM role, and region.
   - Confirm the instance sees the expected configuration.

4. Inspect OS logs.
   - On Linux, check `/var/log/messages`, `/var/log/syslog`, `journalctl`, and cloud-init logs.
   - On Windows, check Event Viewer and EC2Launch or EC2Config logs depending on the AMI generation.

5. Remediate based on the failure domain.
   - Reboot for transient guest issues
   - Stop/start to move the instance to new hardware when a system check fails
   - Fix boot scripts, filesystem issues, or network configuration when an instance check fails

**Practical Linux examples**

Use metadata to confirm the instance identity and network configuration.

```bash
curl -s http://169.254.169.254/latest/meta-data/instance-id
curl -s http://169.254.169.254/latest/meta-data/local-ipv4
curl -s http://169.254.169.254/latest/meta-data/public-ipv4
curl -s http://169.254.169.254/latest/meta-data/iam/security-credentials/
```

If IMDSv2 is required, the same queries need a token.

```bash
TOKEN=$(curl -sX PUT "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 60")

curl -s \
  -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/meta-data/placement/region
```

Check boot-related logs on a Linux instance.

```bash
sudo journalctl -b
sudo journalctl -u cloud-init
sudo tail -n 100 /var/log/cloud-init-output.log
```

If the instance fails to come up after a user data change, inspect the user data script for syntax errors, missing package repositories, or commands that block boot completion.

**Practical Windows examples**

On Windows EC2 instances, metadata is also available from the same local endpoint. PowerShell can query it directly.

```powershell
$token = Invoke-RestMethod -Method Put `
  -Uri "http://169.254.169.254/latest/api/token" `
  -Headers @{"X-aws-ec2-metadata-token-ttl-seconds"="21600"}

Invoke-RestMethod `
  -Uri "http://169.254.169.254/latest/meta-data/instance-id" `
  -Headers @{"X-aws-ec2-metadata-token"=$token}
```

For boot and startup issues, review:
- Event Viewer
- EC2Launch logs
- Windows system and application logs

These logs help identify driver failures, domain join problems, or startup scripts that prevent the instance from becoming reachable.

**Common exam scenarios**

- An instance cannot be reached by SSH, but the system status check passes.
  - Focus on the guest OS, security groups, NACLs, route tables, and instance status check.
- The instance status check fails after a kernel update.
  - Review the system log and OS logs for boot failures or kernel issues.
- Metadata queries fail from inside the instance.
  - Check whether IMDS is disabled, `HttpTokens` is required, or the hop limit blocks access.
- A user data script does not appear to run.
  - Inspect the system log and cloud-init output for syntax or dependency errors.
- The instance has the wrong private IP or IAM role behavior.
  - Use metadata to confirm the actual runtime configuration.

**Security considerations**

Metadata is powerful, so treat it as sensitive internal infrastructure data.

- Prefer **IMDSv2** over IMDSv1
- Require tokens with `HttpTokens=required`
- Keep `HttpPutResponseHopLimit` as low as practical
- Disable metadata access if the workload does not need it
- Be cautious with containers and proxy configurations that may expose metadata unintentionally

> Metadata credentials are temporary IAM role credentials, not long-term access keys. They still grant permissions, so protect access to the metadata service.

**Quick comparison**

| Feature | What it tells you | Typical use |
|---|---|---|
| Instance metadata | Runtime instance details from inside the instance | Verify identity, IPs, IAM role, user data |
| System log | Early boot and console output | Diagnose boot failures and startup issues |
| System status check | AWS host and infrastructure health | Detect underlying hardware or network host problems |
| Instance status check | Guest OS reachability and health | Detect OS boot or configuration problems |

**Key takeaways**

- Metadata answers “what does the instance think it is?”
- System logs answer “what happened during boot?”
- Status checks answer “is the host or guest healthy?”
- IMDSv2 is the preferred and more secure metadata access method
- Use these tools together to isolate whether a problem is in AWS infrastructure, the guest OS, or the application layer
