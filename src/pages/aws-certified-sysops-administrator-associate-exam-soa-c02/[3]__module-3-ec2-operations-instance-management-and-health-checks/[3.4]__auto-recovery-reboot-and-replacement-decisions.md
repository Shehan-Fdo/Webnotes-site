## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Decision framework for EC2 remediation**

Amazon EC2 troubleshooting usually starts with the two status checks: **system status check** and **instance status check**. The exam expects you to choose the least disruptive action that addresses the failure mode.

- **System status check failure** usually points to AWS infrastructure issues such as host hardware, networking, or underlying platform impairment.
- **Instance status check failure** usually points to issues inside the guest OS, such as kernel problems, file system corruption, exhausted resources, or misconfiguration.
- **Reboot** is the first low-impact action when the guest OS may be unhealthy but the underlying host is fine.
- **Recover** is the AWS-managed action for supported EC2 configurations when the underlying host or system status is impaired.
- **Stop and start** is a broader remediation step for standalone instances, but it is not a universal host-failure recovery mechanism and it changes instance identity details in important ways.
- **Replace** is often the right architectural answer for immutable or disposable workloads, especially when the instance is managed by an Auto Scaling group.

A useful exam shortcut is to map the failure to the layer:
- guest OS problem → reboot first
- AWS host/system problem → recover if supported
- persistent or unrecoverable issue → stop/start or replace, depending on architecture

**EC2 status checks and what they imply**

EC2 publishes two primary health signals. They are not interchangeable, and the remediation choice depends on which one fails.

| Check type | What it indicates | Typical action |
|---|---|---|
| **System status check** | AWS infrastructure, host, or network path problem outside the guest OS | Recover, or stop/start if recovery is unavailable |
| **Instance status check** | Guest OS, software, or local configuration problem | Reboot first; investigate OS and application health |

A failed system status check does not automatically mean the instance is permanently broken. It often means the underlying host or network path is impaired. In supported cases, EC2 recovery can move the instance to healthy hardware while preserving the instance identity as much as possible.

A failed instance status check does not always mean a reboot will fix it. Reboot helps when the OS is hung, services are stuck, or transient kernel issues exist. If the root cause is persistent corruption, bad boot configuration, or a broken application dependency, you may need stop/start, repair, or replacement.

**Reboot: when to use it**

Use **reboot** when the instance is reachable at the infrastructure level but the guest OS appears unhealthy.

Common reasons:
- OS is hung or unresponsive
- memory pressure or process exhaustion is causing instability
- a service restart alone is not enough
- temporary kernel or driver issues may clear on restart

Reboot is usually the least disruptive action because it keeps the instance on the same host and preserves the instance ID, private IP, and attached EBS volumes.

Example CLI:
```bash
aws ec2 reboot-instances --instance-ids i-0123456789abcdef0
```

Operationally, reboot is often the first step for an **instance status check** failure. However, do not assume reboot is always sufficient. If the instance fails to boot again, or the same failure recurs, move to deeper investigation or a stronger remediation path.

> A reboot does not fix every guest OS problem. If the root cause is file system corruption, broken startup scripts, or persistent configuration drift, you may need recovery, stop/start, or replacement.

**Recover: AWS-managed remediation for supported instances**

The **recover** action is designed for instances that fail a system status check due to underlying host impairment. AWS attempts to move the instance to healthy hardware while keeping the instance available in a supported way.

Key points:
- Recovery is intended for **system status check** failures, not ordinary application bugs.
- It is available only for certain instance types and configurations.
- It is not universally supported for every placement option or specialized setup.
- It can be initiated automatically through a CloudWatch alarm or manually from the console or API in supported cases.

Manual recovery via CLI:
```bash
aws ec2 recover-instances --instance-id i-0123456789abcdef0
```

The console path is typically:
1. Open the EC2 console.
2. Select the instance.
3. Choose **Actions**.
4. Open **Instance state** or **Monitor and troubleshoot**.
5. Select **Recover instance** if the option is available.

Recovery is most useful when:
- the instance is EBS-backed
- the workload is not in an Auto Scaling group that already handles replacement
- you want AWS to attempt remediation without changing the instance identity more than necessary

Important limitations to remember:
- Recovery is not available for all instance families or configurations.
- Some placement group or specialized networking/storage setups may not support it.
- If the instance is not eligible, the recovery option may be absent.
- If the root cause is not host-level, recovery may not help.

**Stop and start: broader remediation with identity changes**

**Stop/start** is a stronger action than reboot. It stops the instance, releases the current host, and starts it again later. This can clear host-level issues and can also help with some guest OS problems, but it is not the same as recovery.

Use stop/start when:
- the instance is standalone and recovery is unavailable
- you need to move away from a problematic host
- you are willing to accept changes to instance identity details

Typical effects:
- the **instance ID** stays the same
- the **private IPv4 address** usually stays the same for a VPC-backed EBS instance
- the **public IPv4 address** changes unless you use an Elastic IP
- the instance may start on different underlying hardware
- instance store data is lost if the instance uses instance store volumes

Example CLI:
```bash
aws ec2 stop-instances --instance-ids i-0123456789abcdef0
aws ec2 start-instances --instance-ids i-0123456789abcdef0
```

Use stop/start carefully in exam scenarios:
- It is not a generic “fix the host” answer for every failure.
- It is not the first choice for a simple instance status check failure if reboot is likely enough.
- It is often the fallback when recovery is unavailable or unsupported.

**Replacement: when to terminate and recreate**

**Replacement** means you stop treating the instance as a unique snowflake and instead rebuild it from code, an AMI, user data, configuration management, or an Auto Scaling group.

Choose replacement when:
- the workload is disposable or immutable
- the instance is in an Auto Scaling group
- the instance is too damaged to trust after remediation
- you want a clean, repeatable recovery path rather than manual repair

Replacement is often the best answer for:
- stateless web servers
- worker nodes
- container hosts managed by an orchestrator
- instances built from golden images and bootstrap scripts

For standalone instances, replacement is usually a design decision:
- if the instance is critical and stateful, you may prefer recovery or stop/start first
- if the instance is stateless, rebuilding is often faster and safer than trying to repair it

**Auto Scaling groups and health checks**

Auto Scaling groups (ASGs) change the decision tree. If an instance is part of an ASG, the group usually replaces unhealthy instances rather than relying on manual recovery.

ASG health sources:
- **EC2 status checks**
- **Elastic Load Balancing (ELB)** health checks

Behavior:
- if an instance fails health checks, the ASG can terminate and launch a replacement
- this is usually preferred for immutable or horizontally scaled workloads
- recovery is less relevant because the group’s job is to maintain desired capacity

Practical implication:
- if the question mentions an ASG, think **replace** rather than **recover**
- if the question mentions a standalone instance, think **reboot**, **recover**, or **stop/start** depending on the failure type and supportability

**CloudWatch alarm-driven recovery**

You can automate EC2 recovery with a **CloudWatch alarm** that watches the relevant status check metric. This is a common exam pattern.

Typical metric:
- `StatusCheckFailed_System`

A common alarm pattern uses:
- one evaluation period
- a short period such as 60 seconds
- alarm action set to EC2 recovery

Example CLI:
```bash
aws cloudwatch put-metric-alarm \
  --alarm-name ec2-system-recovery \
  --namespace AWS/EC2 \
  --metric-name StatusCheckFailed_System \
  --dimensions Name=InstanceId,Value=i-0123456789abcdef0 \
  --statistic Minimum \
  --period 60 \
  --evaluation-periods 1 \
  --threshold 1 \
  --comparison-operator GreaterThanOrEqualToThreshold \
  --treat-missing-data missing \
  --alarm-actions arn:aws:automate:us-east-1:ec2:recover
```

Why this matters:
- a single failed period is often enough to trigger recovery quickly
- longer evaluation windows can delay remediation
- `treat-missing-data` affects how the alarm behaves when metrics are absent, which matters during outages or instance termination

Alarm tuning considerations:
- use a short period for faster response when the workload is critical
- avoid overly long evaluation windows if you want timely recovery
- understand that missing data can keep an alarm from transitioning as expected
- test the alarm path before relying on it in production

**Console setup for recovery alarms**

A practical console workflow helps on the exam because it shows you understand the operational path, not just the CLI.

1. Open the EC2 console and select the instance.
2. Go to the **Monitoring** tab.
3. Open CloudWatch alarms or create a new alarm.
4. Choose the EC2 metric for system status failure.
5. Set the threshold to trigger on failure.
6. Add the EC2 recovery action.
7. Save and verify the alarm state.

If the console does not offer recovery as an action, that usually indicates a support limitation for the instance type or configuration.

**Instance metadata, identity, and persistence concerns**

Recovery, reboot, and stop/start differ in how they affect identity and persistence.

- **Reboot**
  - keeps the same instance ID
  - keeps the same private IP
  - keeps the same host placement unless AWS intervenes for maintenance
- **Recover**
  - attempts to preserve the instance identity while moving it to healthy infrastructure
  - is designed for supported system-level failures
  - may preserve more operational continuity than stop/start
- **Stop/start**
  - keeps the instance ID
  - usually keeps the private IP in a VPC for EBS-backed instances
  - changes the public IP unless you use an Elastic IP
  - may change the underlying host and can affect ephemeral data
- **Replace**
  - creates a new instance identity
  - may require reattachment of dependencies, registration, or bootstrap

This distinction matters when the question mentions:
- licensing tied to instance ID
- security groups or NACLs that depend on IP behavior
- applications that store state locally
- scripts that use instance metadata service values such as `instance-id`

Example metadata query:
```bash
curl http://169.254.169.254/latest/meta-data/instance-id
```

**When to prefer terminate and replace**

Terminate and replace is often the cleanest answer when the instance is not meant to be repaired in place.

Prefer replacement when:
- the instance is managed by an ASG
- the instance is built from immutable infrastructure
- the root cause is unknown and repair would be risky
- the instance has repeated failures after reboot or recovery attempts

Do not confuse replacement with recovery:
- **recovery** tries to preserve the instance
- **replacement** accepts that the current instance is no longer the right object to keep

**Troubleshooting patterns for exam questions**

Use the failure signal and workload type to choose the action.

- **System status check failed on a standalone EBS-backed instance**
  - try recovery if supported
  - if recovery is unavailable, consider stop/start
- **Instance status check failed and the OS seems hung**
  - reboot is usually the first step
- **Instance status check failed repeatedly after reboot**
  - investigate OS corruption, startup scripts, disk issues, or application dependencies
  - consider stop/start or replacement depending on statefulness
- **Instance in an Auto Scaling group is unhealthy**
  - let the ASG replace it
- **SSH is failing but status checks are healthy**
  - check security groups, NACLs, routing, key pairs, `sshd`, and OS firewall before assuming EC2 host failure

A healthy system status check does not guarantee application reachability. A failed SSH connection can come from:
- security group rules
- network ACLs
- route table issues
- wrong key pair or username
- `sshd` not running
- host-based firewall rules
- application-level authentication problems

**Practical decision matrix**

| Scenario | Best first action | Why |
|---|---|---|
| Instance status check failed, OS appears hung | Reboot | Least disruptive guest OS reset |
| System status check failed, supported standalone instance | Recover | AWS moves workload off impaired host |
| System status check failed, recovery unsupported | Stop/start | Forces placement on new host |
| Instance in ASG is unhealthy | Replace | ASG maintains desired capacity |
| Stateless workload with repeated corruption | Replace | Faster and safer than repeated repair |
| SSH unreachable, checks healthy | Network/OS troubleshooting | Problem may not be EC2 health related |

**Exam traps to avoid**

- Do not assume every failure means “replace the instance.”
- Do not assume reboot fixes host-level problems.
- Do not assume recovery is available for every instance type or placement configuration.
- Do not assume stop/start preserves public IP.
- Do not assume SSH failure means the instance status check failed.
- Do not assume a CloudWatch alarm with a long evaluation window is ideal for fast recovery.

**Quick CLI reference**

```bash
# Reboot
aws ec2 reboot-instances --instance-ids i-0123456789abcdef0

# Stop
aws ec2 stop-instances --instance-ids i-0123456789abcdef0

# Start
aws ec2 start-instances --instance-ids i-0123456789abcdef0

# Manual recovery
aws ec2 recover-instances --instance-id i-0123456789abcdef0

# Describe status checks
aws ec2 describe-instance-status --instance-ids i-0123456789abcdef0 --include-all-instances
```

**Key takeaways**

- **Reboot** for likely guest OS issues.
- **Recover** for supported system status check failures caused by host impairment.
- **Stop/start** when recovery is unavailable or you need a broader reset.
- **Replace** for Auto Scaling groups and immutable workloads.
- Always match the action to the failure layer, workload architecture, and support limitations.
