## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Overview**

Amazon EC2 status checks are the first signal you use when an instance becomes unreachable. They tell you whether the problem is inside the guest operating system, on the underlying host, or in the network path to the instance. For exam purposes, you need to distinguish between symptoms, likely causes, and the right recovery action.

EC2 exposes two primary status checks:

- **System status check**: evaluates the AWS infrastructure that runs the instance.
- **Instance status check**: evaluates the guest operating system and software stack.

A failure in either check can explain why SSH, RDP, HTTP, or application traffic stops working. The key is to avoid guessing and use the check results, console output, logs, and network controls to narrow the fault domain.

**Status check types and what they mean**

| Check type | What it tests | Common failure causes | Typical response |
|---|---|---|---|
| **System status check** | AWS-managed host, networking, and power | Underlying hardware issue, impaired host, network reachability issue at the infrastructure layer | Wait for AWS recovery, stop/start, or recover instance if supported |
| **Instance status check** | Guest OS and instance software | OS hang, full disk, failed boot, bad kernel, misconfigured firewall, broken SSH/RDP service | Use console access, reboot, fix OS, or restore from backup |

A failed status check does not always mean the instance is permanently broken. It often means you need to identify whether the issue is transient, host-related, or caused by the operating system configuration.

**How to read the symptoms**

A practical troubleshooting flow starts with the simplest observable facts:

- If the instance is running but you cannot connect, check both status checks first.
- If the **system status check** fails, suspect the underlying AWS host or network path.
- If the **instance status check** fails, suspect the guest OS, boot process, or local configuration.
- If both checks pass, the issue is usually outside the instance, such as security groups, network ACLs, route tables, DNS, or application-level binding.

Common user-visible symptoms include:

- SSH times out on port `22`.
- RDP fails on port `3389`.
- The instance responds to ping only if ICMP is allowed and the OS is healthy.
- The application port is closed or returns no response.
- The instance appears healthy in the console but the service is not listening.

**Core troubleshooting workflow**

1. Check the EC2 console status checks.
2. Confirm the instance state is `running`.
3. Review recent changes to security groups, network ACLs, route tables, IAM roles, user data, and OS configuration.
4. Inspect system logs and console output.
5. Test network reachability from a known-good source.
6. Decide whether to reboot, stop/start, recover, or restore from backup.

This order matters because it helps you avoid disruptive actions before you know whether the fault is in the guest OS or the AWS host.

**AWS CLI commands you should know**

Use the AWS CLI to inspect instance health and recent events.

```bash
aws ec2 describe-instance-status \
  --instance-ids i-0123456789abcdef0 \
  --include-all-instances
```

This returns the current system and instance status checks, along with any scheduled events. Read the output fields rather than trying to filter on status-check values in the request.

To inspect console output, which can reveal boot failures or kernel panics:

```bash
aws ec2 get-console-output \
  --instance-id i-0123456789abcdef0 \
  --latest
```

To view instance metadata and confirm networking details from inside the instance, use IMDS (Instance Metadata Service) if you can access the OS:

```bash
curl http://169.254.169.254/latest/meta-data/public-ipv4
curl http://169.254.169.254/latest/meta-data/local-ipv4
curl http://169.254.169.254/latest/meta-data/instance-id
```

> If the instance is unreachable, you may not be able to use IMDS. In that case, rely on the console, EBS volume attachment, or recovery actions.

**System status check failures**

A system status check failure indicates a problem with the AWS infrastructure hosting the instance, not necessarily with your operating system. Typical causes include:

- Underlying hardware failure.
- Network impairment on the host.
- Power or virtualization issues on the physical server.
- AWS maintenance or host degradation.

What you should do:

- Wait briefly and recheck if AWS is already remediating the issue.
- Reboot the instance if the failure appears transient and the instance is otherwise healthy.
- Stop and start the instance to move it to new hardware when the instance is EBS-backed and stop/start is supported.
- Use instance recovery for supported instance types and configurations when available.

**Important recovery behavior**

- **Stop/start** moves the instance to new underlying hardware.
- The **public IPv4 address** is not retained across stop/start unless you associate an Elastic IP address.
- The **private IPv4 address** is typically retained for the same instance when it is stopped and started.
- **Instance recovery** is available only for certain instance types and configurations, and it is designed for system status check failures, not guest OS failures.

**When to use instance recovery**

Instance recovery is useful when the underlying host is impaired but the instance configuration supports recovery. It attempts to move the instance to healthy hardware while preserving the instance identity and network configuration where possible.

Use it when:

- The system status check fails.
- The instance is EBS-backed and recovery is supported.
- You want to preserve the instance ID and private IP if the recovery succeeds.

Do not assume recovery is available for every instance. Some instance types, placement configurations, or storage setups may not support it.

**Instance status check failures**

An instance status check failure means the guest OS is not responding correctly. This is usually inside the instance rather than in AWS infrastructure.

Common causes include:

- OS boot failure.
- Kernel panic.
- Full root volume.
- Corrupted filesystem.
- Misconfigured network stack.
- Firewall rules blocking access.
- SSH daemon or RDP service not running.
- CPU or memory exhaustion causing the OS to become unresponsive.

Useful checks inside the OS:

```bash
# Linux: service name varies by distribution
sudo systemctl status ssh
sudo systemctl status sshd
sudo journalctl -xe
sudo dmesg | tail -n 50
df -h
free -m
ip addr
ss -tulpn
```

For Windows, inspect:

- Event Viewer logs.
- Services console.
- RDP settings.
- Disk space on the system volume.
- Windows Update or driver issues.

If the OS is reachable through the console or serial access, you can often fix the problem without replacing the instance.

**Network reachability troubleshooting**

If both status checks pass but the instance is still unreachable, focus on the network path and security controls.

Check these layers in order:

- **Security group**: stateful allow rules for inbound and outbound traffic.
- **Network ACL**: stateless subnet-level allow and deny rules.
- **Route table**: correct route to the internet gateway, NAT gateway, transit gateway, or peering connection.
- **Subnet association**: instance is in the expected subnet.
- **Elastic IP**: public access is mapped correctly if needed.
- **DNS**: hostname resolves to the expected address.
- **Application listener**: service is bound to the correct interface and port.

Example security group rule for SSH:

```bash
aws ec2 authorize-security-group-ingress \
  --group-id sg-0123456789abcdef0 \
  --protocol tcp \
  --port 22 \
  --cidr 203.0.113.0/24
```

Example to inspect route tables:

```bash
aws ec2 describe-route-tables \
  --filters Name=association.subnet-id,Values=subnet-0123456789abcdef0
```

A common exam trap is assuming the instance is broken when the real issue is a missing inbound rule or a route table that no longer points to an internet gateway.

**Reboot, stop/start, recover, or replace**

Choose the least disruptive action that matches the failure domain.

| Action | Best for | What it does | Caveats |
|---|---|---|---|
| **Reboot** | Transient OS issues or minor service problems | Restarts the guest OS | Does not move the instance to new hardware |
| **Stop/start** | Host-level impairment on EBS-backed instances | Stops the instance and starts it on new hardware | Public IPv4 changes unless an Elastic IP is attached |
| **Instance recovery** | Supported instances with system status check failure | Attempts to recover the instance onto healthy hardware | Not universally available |
| **Restore from backup** | Corrupted OS, bad configuration, or data loss | Rebuilds from AMI, snapshot, or backup | May change instance ID and IPs |
| **Replace instance** | Irreparable failure or unsupported recovery | Launches a new instance | Requires reconfiguration and cutover |

Use reboot first only when the instance status check suggests a guest OS issue and the instance is otherwise likely recoverable. Use stop/start or recovery when the system status check points to host impairment.

**Console output and logs**

Console output is valuable when the instance cannot be reached over the network. It can show:

- Boot loader errors.
- Filesystem check failures.
- Kernel panics.
- Missing boot device errors.
- Service startup failures.

Example Linux boot troubleshooting clues:

```text
Kernel panic - not syncing: VFS: Unable to mount root fs on unknown-block(0,0)
```

That kind of message suggests a boot or volume problem, not a security group issue.

If you can detach the root EBS volume and attach it to a helper instance, you can inspect logs and repair files offline.

Typical offline repair workflow:

1. Stop the affected instance.
2. Detach the root EBS volume.
3. Attach it to a healthy helper instance.
4. Mount the volume.
5. Inspect logs, configuration files, and filesystem integrity.
6. Fix the issue.
7. Reattach the volume and start the original instance.

**EBS-backed instance repair patterns**

For EBS-backed instances, offline repair is often the safest way to fix OS-level problems.

Examples of what you might correct:

- `/etc/fstab` entries that prevent boot.
- Broken SSH configuration in `/etc/ssh/sshd_config`.
- Incorrect network configuration.
- Full `/var` or root filesystem.
- Bad startup scripts or systemd units.

Example of checking a mounted root volume:

```bash
sudo mkdir -p /mnt/recovery
sudo mount /dev/nvme1n1p1 /mnt/recovery
sudo ls /mnt/recovery/var/log
sudo cat /mnt/recovery/etc/fstab
```

Be careful not to edit the wrong volume or mount point. Confirm the device mapping before making changes.

**Windows-specific considerations**

For Windows instances, failed reachability often comes from:

- RDP disabled.
- Windows firewall rules.
- Corrupted boot configuration.
- Disk full conditions.
- Pending updates or driver issues.
- Password or account problems.

Useful recovery options include:

- Using EC2 console output to inspect boot issues.
- Using Systems Manager if the instance is managed and reachable.
- Detaching the root volume and repairing offline from another Windows instance or supported recovery workflow.
- Restoring from a known-good AMI or snapshot.

**Monitoring and alerting**

You should not wait for users to report reachability problems. Use monitoring to detect them early.

Common signals:

- EC2 status check alarms.
- CloudWatch metrics for instance and system status checks.
- CPU, disk, and network anomalies.
- Application health checks from a load balancer or external probe.

Example CloudWatch alarm concept:

- Trigger when `StatusCheckFailed_Instance` is `1` for several consecutive periods.
- Trigger when `StatusCheckFailed_System` is `1` for several consecutive periods.

These alarms help you distinguish between guest OS failure and infrastructure failure before the outage spreads.

**Exam-focused distinctions**

- **System status check failure** points to AWS infrastructure or host impairment.
- **Instance status check failure** points to the guest OS or software stack.
- **Reboot** does not move the instance to new hardware.
- **Stop/start** usually changes the public IPv4 address unless an Elastic IP is associated.
- **Instance recovery** is not universal; it depends on instance support.
- **Security groups** are stateful.
- **Network ACLs** are stateless.
- **Both status checks passing** means you should look beyond EC2 health, especially at networking and application configuration.

**Practical troubleshooting example**

Suppose an application server stops responding to SSH and HTTP.

1. Check EC2 status checks.
2. If the system check fails, try recovery or stop/start.
3. If the instance check fails, inspect console output and logs.
4. If both checks pass, verify:
   - Security group inbound rules.
   - Network ACL rules.
   - Route table and internet gateway association.
   - The service is listening on the correct port.
   - The OS firewall allows traffic.
5. If the root cause is a bad update or filesystem issue, repair offline or restore from backup.

This sequence prevents you from wasting time on the wrong layer.

**Quick reference**

- **System check failed**: think host or AWS infrastructure.
- **Instance check failed**: think OS, boot, or software.
- **No status check failure**: think security group, NACL, route, DNS, or application.
- **Need to preserve identity**: consider instance recovery or stop/start on EBS-backed instances.
- **Need a clean rebuild**: restore from AMI or snapshot.

**Common pitfalls**

- Assuming every unreachable instance needs replacement.
- Using stop/start when a simple reboot would fix a transient OS issue.
- Forgetting that public IPv4 is not preserved across stop/start without an Elastic IP.
- Treating security groups and network ACLs as interchangeable.
- Ignoring console output, which often reveals the real failure faster than network testing.
- Assuming recovery is available for every instance type or configuration.
