## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Lab purpose and mental model**

AWS Systems Manager (SSM) lets you manage instances without opening inbound SSH or RDP. In a connectivity break-and-restore lab, you intentionally disrupt one or more dependencies, observe the failure mode, and then restore the path. This is useful because SSM access depends on multiple layers working together: the instance must run the SSM Agent, the instance profile must allow SSM actions, and the network must reach SSM endpoints either through the public internet or through VPC endpoints.

The key exam skill is to identify *which layer is broken* from the symptom. A failed Session Manager connection is not always a network issue. It can also be an IAM role problem, a missing agent, a disabled endpoint, or a security group or NACL rule that blocks egress.

**Core SSM connectivity path**

For Session Manager and other SSM features to work, the instance typically needs:

- **SSM Agent** installed and running on the instance
- **Instance profile IAM role** with the correct Systems Manager permissions
- **Network egress** to SSM service endpoints
- **DNS resolution** and time synchronization
- **Optional VPC endpoints** when the instance has no internet route

A common architecture looks like this:

```text
EC2 instance
  -> SSM Agent
  -> IAM instance profile
  -> HTTPS 443
  -> ssmmessages / ec2messages / ssm endpoints
  -> Systems Manager console or CLI
```

If the instance is in a private subnet, you usually restore connectivity by using interface VPC endpoints for:

- `com.amazonaws.<region>.ssm`
- `com.amazonaws.<region>.ec2messages`
- `com.amazonaws.<region>.ssmmessages`

**What the agent actually does**

The SSM Agent runs on the instance and maintains outbound connections to AWS-managed endpoints. It does not require inbound SSH or RDP for Session Manager access. That is why Session Manager is often used to reduce exposure of bastion hosts and public IPs.

On Amazon Linux 2 and many other Linux distributions, you can check the agent with:

```bash
sudo systemctl status amazon-ssm-agent
sudo systemctl restart amazon-ssm-agent
sudo journalctl -u amazon-ssm-agent -n 50 --no-pager
```

On Windows, you can inspect the service with PowerShell:

```powershell
Get-Service AmazonSSMAgent
Restart-Service AmazonSSMAgent
Get-WinEvent -LogName Application -MaxEvents 20
```

If the agent is stopped, Session Manager cannot establish a shell even if IAM and networking are correct.

**IAM role and permissions**

The instance profile must allow the instance to register with Systems Manager and open sessions. The standard managed policy is `AmazonSSMManagedInstanceCore`. In a lab, breaking IAM is a useful way to simulate a registration failure.

Typical symptoms of an IAM problem:

- The instance does not appear as a managed node in Systems Manager
- The console shows the instance as offline or not managed
- Session Manager fails before a shell opens
- The agent logs show authorization or credential errors

A minimal instance profile trust relationship for EC2 looks like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "ec2.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

And the attached policy is commonly:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "ssm:UpdateInstanceInformation",
        "ssmmessages:CreateControlChannel",
        "ssmmessages:CreateDataChannel",
        "ssmmessages:OpenControlChannel",
        "ssmmessages:OpenDataChannel",
        "ec2messages:AcknowledgeMessage",
        "ec2messages:DeleteMessage",
        "ec2messages:FailMessage",
        "ec2messages:GetEndpoint",
        "ec2messages:GetMessages",
        "ec2messages:SendReply"
      ],
      "Resource": "*"
    }
  ]
}
```

In practice, use the AWS managed policy unless you have a reason to scope permissions more tightly.

**Network requirements and common break points**

SSM uses outbound HTTPS on port 443. The instance must be able to reach the relevant endpoints. In a public subnet, this may happen through an internet gateway and a public IP or through a NAT gateway if the instance is private. In a private subnet, interface VPC endpoints are the usual solution.

Break points you can test in a lab:

- Remove the route to the internet or NAT gateway
- Disable or misconfigure the VPC endpoint
- Block outbound 443 in the security group
- Block ephemeral return traffic with a restrictive network ACL
- Break DNS resolution in the VPC
- Remove the instance profile role

A security group example that would break connectivity if outbound is too restrictive:

```text
Outbound rules:
- Allow TCP 443 to 0.0.0.0/0
```

If you remove that rule and the instance relies on public endpoints, the agent cannot reach Systems Manager.

A network ACL can also break the path if it blocks return traffic. Because NACLs are stateless, you must allow both outbound 443 and the ephemeral response ports.

**VPC endpoints for private connectivity**

When you want SSM to work without internet access, create interface endpoints in the VPC. This is a common exam scenario because it preserves private subnet isolation while still allowing management access.

Recommended endpoints:

- `ssm`
- `ec2messages`
- `ssmmessages`

Important configuration points:

- Enable **Private DNS** so the standard AWS service names resolve to the endpoint IPs
- Attach a security group to the endpoint that allows inbound 443 from the instance subnets or instance security group
- Ensure the endpoint subnets have route table and DNS support enabled at the VPC level

Example endpoint creation with the AWS CLI:

```bash
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0123456789abcdef0 \
  --vpc-endpoint-type Interface \
  --service-name com.amazonaws.us-east-1.ssm \
  --subnet-ids subnet-aaa subnet-bbb \
  --security-group-ids sg-0123456789abcdef0 \
  --private-dns-enabled
```

Repeat for `ec2messages` and `ssmmessages`.

**How to diagnose a broken connection**

Start from the symptom and work outward. The fastest path is usually to check the agent, then IAM, then network, then endpoints.

1. Verify the instance is running and reachable at the EC2 layer.
2. Check that the SSM Agent is installed and active.
3. Confirm the instance profile is attached and has `AmazonSSMManagedInstanceCore`.
4. Inspect the instance’s outbound network path to port 443.
5. If the instance is private, confirm the three interface endpoints exist and have Private DNS enabled.
6. Review agent logs for credential, DNS, or TLS errors.

Useful Linux checks:

```bash
sudo systemctl is-active amazon-ssm-agent
sudo tail -n 100 /var/log/amazon/ssm/amazon-ssm-agent.log
curl -I https://ssm.us-east-1.amazonaws.com
nslookup ssm.us-east-1.amazonaws.com
```

Useful Windows checks:

```powershell
Test-NetConnection ssm.us-east-1.amazonaws.com -Port 443
Resolve-DnsName ssm.us-east-1.amazonaws.com
Get-Content "C:\ProgramData\Amazon\SSM\Logs\amazon-ssm-agent.log" -Tail 50
```

If `curl` or `Test-NetConnection` fails, the issue is usually network or DNS. If the agent log shows access denied, the issue is usually IAM. If the service is stopped, the issue is local to the instance.

**Typical failure symptoms and likely causes**

| Symptom | Likely cause | What to check first |
|---|---|---|
| Instance does not appear in Systems Manager | Missing IAM role, agent not running, no network path | Instance profile, agent status |
| Session Manager times out | No outbound 443, missing VPC endpoint, DNS issue | Security group, route table, endpoints |
| Instance shows online but session fails | Session permissions or endpoint mismatch | IAM policy, `ssmmessages` endpoint |
| Agent logs show credential errors | Wrong or missing instance profile | EC2 role attachment |
| Agent logs show DNS failures | VPC DNS disabled or broken resolver path | VPC DNS settings, DHCP options |
| Agent logs show TLS or connection errors | Proxy, firewall, or endpoint security group issue | Proxy config, SG/NACL rules |

**Restoring connectivity in a lab**

A restore workflow should reverse the exact change you made. This helps you prove causality and avoid masking the real issue.

1. Re-enable the agent if it was stopped.
2. Reattach the correct instance profile if it was removed.
3. Restore outbound 443 in the security group or NACL.
4. Recreate or repair the VPC endpoints if the instance is private.
5. Re-enable Private DNS on the endpoints if name resolution is failing.
6. Restart the agent and retest the session.

Example recovery commands on Linux:

```bash
sudo systemctl restart amazon-ssm-agent
sudo systemctl enable amazon-ssm-agent
sudo journalctl -u amazon-ssm-agent -n 20 --no-pager
```

If you changed the IAM role on the instance, it can take a short time for the agent to refresh credentials. A restart often speeds up validation, but the underlying fix is the role attachment itself.

**Practical lab scenarios**

A good lab usually includes one break at a time so you can isolate the cause.

- **Scenario 1: Stop the agent**
  - Expected result: instance disappears from managed nodes or session fails immediately
  - Restore: start the service again

- **Scenario 2: Remove the instance profile**
  - Expected result: agent cannot authenticate to Systems Manager
  - Restore: attach `AmazonSSMManagedInstanceCore`

- **Scenario 3: Block outbound 443**
  - Expected result: agent cannot reach AWS endpoints
  - Restore: allow HTTPS egress

- **Scenario 4: Remove the NAT route**
  - Expected result: private instance loses internet-based SSM access
  - Restore: add the route back or use VPC endpoints

- **Scenario 5: Delete one interface endpoint**
  - Expected result: private connectivity fails partially or completely
  - Restore: recreate `ssm`, `ec2messages`, and `ssmmessages`

**CLI commands you should know**

Use the AWS CLI to inspect the instance profile and endpoint state.

```bash
aws ec2 describe-instances \
  --instance-ids i-0123456789abcdef0 \
  --query 'Reservations[0].Instances[0].IamInstanceProfile'
```

```bash
aws ec2 describe-vpc-endpoints \
  --filters Name=vpc-id,Values=vpc-0123456789abcdef0
```

```bash
aws ssm describe-instance-information
```

If the instance is not listed in `describe-instance-information`, the problem is usually registration, IAM, or network reachability.

**Why this matters for the exam**

SOA-C02 often tests your ability to choose the least disruptive and most secure management path. Session Manager is preferred when you want to avoid inbound ports and bastions. You should recognize that SSM connectivity depends on multiple components and that the correct fix is not always “open port 22.”

The exam may describe a private EC2 instance that must be managed without internet access. The best answer is usually to use interface VPC endpoints with Private DNS and the correct instance role, not to assign a public IP.

**Quick troubleshooting checklist**

- **Agent**: Is `amazon-ssm-agent` running?
- **Role**: Does the instance have `AmazonSSMManagedInstanceCore`?
- **Network**: Can the instance reach HTTPS outbound?
- **DNS**: Does the instance resolve SSM endpoints?
- **Endpoints**: Are `ssm`, `ec2messages`, and `ssmmessages` present for private subnets?
- **Logs**: Do the agent logs show auth, DNS, or connection errors?

**Key takeaway**

SSM connectivity is a chain, not a single setting. To break and restore it confidently, you need to isolate whether the failure is caused by the agent, IAM, routing, security controls, or VPC endpoints. The fastest fix is the one that restores the exact missing dependency without weakening the instance’s security posture.
