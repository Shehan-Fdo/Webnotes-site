## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Overview**

Amazon Elastic Compute Cloud (<abbr title="Elastic Compute Cloud">EC2</abbr>) instances can be launched either directly with the AWS Command Line Interface (<abbr title="Command Line Interface">CLI</abbr>) or declaratively with AWS CloudFormation. The exam often tests when to use each approach, how to pass networking and identity settings, and how to bootstrap an instance at launch time.

Use the CLI when you need an immediate, one-off launch. Use CloudFormation when you want repeatable infrastructure, parameterized environments, and controlled change management. Both approaches can attach security groups, subnets, IAM roles, storage, and startup scripts, but CloudFormation adds drift-aware lifecycle management and stack-level rollback behavior.

**Direct EC2 launch with AWS CLI**

The core command for launching an instance is `aws ec2 run-instances`. You supply an Amazon Machine Image (<abbr title="Amazon Machine Image">AMI</abbr>), instance type, key pair, network placement, security groups, and optional bootstrap data.

```bash
aws ec2 run-instances \
  --image-id ami-0123456789abcdef0 \
  --instance-type t3.micro \
  --key-name my-keypair \
  --subnet-id subnet-0abc1234def567890 \
  --security-group-ids sg-0123abcd4567ef890 \
  --iam-instance-profile Name=WebInstanceProfile \
  --user-data file://userdata.sh \
  --tag-specifications 'ResourceType=instance,Tags=[{Key=Name,Value=web-1},{Key=Env,Value=dev}]'
```

Key launch-time options you should recognize:

- **AMI**: Defines the operating system and preinstalled software.
- **Instance type**: Controls CPU, memory, network, and storage performance.
- **Subnet**: Places the instance in a specific Availability Zone.
- **Security groups**: Act as stateful virtual firewalls.
- **IAM instance profile**: Attaches an IAM role to the instance for AWS API access.
- **User data**: Runs bootstrap commands on first boot.
- **Tags**: Help with cost allocation, automation, and inventory.

A common exam pattern is to ask which option enables the instance to call AWS services without long-lived credentials. The answer is the IAM instance profile, not access keys baked into the AMI or user data.

**User data and bootstrap behavior**

User data is a startup script or cloud-init payload that runs during instance initialization. It is commonly used to install packages, start services, fetch configuration, or register the host with a load balancer or configuration system.

Example Linux bootstrap script:

```bash
#!/bin/bash
yum update -y
yum install -y httpd
systemctl enable httpd
systemctl start httpd
echo "<h1>Bootstrapped by user data</h1>" > /var/www/html/index.html
```

When you pass user data with the CLI, you typically provide a file or encoded string. AWS CLI handles the encoding when you use `file://`.

Important behavior to remember:

- **User data runs once by default** on first boot for most Linux AMIs.
- **Changing user data later does not modify a running instance automatically**.
- **Replacing the instance** is often required if you need the new bootstrap logic to take effect.
- **Cloud-init** is the common initialization system on Linux AMIs.

For Windows instances, user data is often used with EC2Launch or EC2Config depending on the AMI generation.

**CloudFormation launch workflow**

CloudFormation launches EC2 as part of a stack. You define the instance in a template, then create or update the stack. This is the preferred approach when you need repeatability, version control, and environment-specific configuration.

Minimal EC2 instance example:

```yaml
AWSTemplateFormatVersion: '2010-09-09'
Description: Minimal EC2 instance launch

Parameters:
  AmiId:
    Type: AWS::EC2::Image::Id
  InstanceType:
    Type: String
    Default: t3.micro
  SubnetId:
    Type: AWS::EC2::Subnet::Id
  SecurityGroupIds:
    Type: List<AWS::EC2::SecurityGroup::Id>
  KeyName:
    Type: AWS::EC2::KeyPair::KeyName

Resources:
  WebInstance:
    Type: AWS::EC2::Instance
    Properties:
      ImageId: !Ref AmiId
      InstanceType: !Ref InstanceType
      SubnetId: !Ref SubnetId
      SecurityGroupIds: !Ref SecurityGroupIds
      KeyName: !Ref KeyName
      Tags:
        - Key: Name
          Value: web-1
      UserData:
        Fn::Base64: !Sub |
          #!/bin/bash
          yum update -y
          yum install -y httpd
          systemctl enable httpd
          systemctl start httpd

Outputs:
  InstanceId:
    Value: !Ref WebInstance
  PublicIp:
    Value: !GetAtt WebInstance.PublicIp
```

This template shows the basic end-to-end pattern:

- Parameters make the template reusable.
- `AWS::EC2::Instance` creates the instance directly.
- `Fn::Base64` is required for `UserData`.
- `!Sub` lets you interpolate values into the script.
- Outputs expose values for operators or downstream stacks.

> CloudFormation can launch a single EC2 instance directly. It is not limited to Auto Scaling groups.

**CloudFormation stack creation commands**

A common workflow is `validate-template`, then `create-stack`, then `describe-stack-events` if something fails.

```bash
aws cloudformation validate-template \
  --template-body file://ec2-instance.yaml
```

```bash
aws cloudformation create-stack \
  --stack-name web-ec2-dev \
  --template-body file://ec2-instance.yaml \
  --parameters \
    ParameterKey=AmiId,ParameterValue=ami-0123456789abcdef0 \
    ParameterKey=InstanceType,ParameterValue=t3.micro \
    ParameterKey=SubnetId,ParameterValue=subnet-0abc1234def567890 \
    ParameterKey=SecurityGroupIds,ParameterValue=sg-0123abcd4567ef890 \
    ParameterKey=KeyName,ParameterValue=my-keypair \
  --capabilities CAPABILITY_IAM
```

Use `CAPABILITY_IAM` when the template creates IAM resources without custom names. Use `CAPABILITY_NAMED_IAM` when the template creates named IAM roles, instance profiles, or policies.

If the stack includes IAM resources, a safer deployment example is:

```bash
aws cloudformation create-stack \
  --stack-name web-ec2-dev \
  --template-body file://ec2-instance-with-iam.yaml \
  --parameters \
    ParameterKey=AmiId,ParameterValue=ami-0123456789abcdef0 \
    ParameterKey=SubnetId,ParameterValue=subnet-0abc1234def567890 \
    ParameterKey=SecurityGroupIds,ParameterValue=sg-0123abcd4567ef890 \
  --capabilities CAPABILITY_NAMED_IAM
```

For updates, use `update-stack` or `deploy` depending on your workflow:

```bash
aws cloudformation deploy \
  --stack-name web-ec2-dev \
  --template-file ec2-instance.yaml \
  --parameter-overrides \
    AmiId=ami-0123456789abcdef0 \
    InstanceType=t3.micro \
    SubnetId=subnet-0abc1234def567890 \
    SecurityGroupIds=sg-0123abcd4567ef890 \
    KeyName=my-keypair \
  --capabilities CAPABILITY_IAM
```

If a parameter is declared as a list, pass values in the format expected by the CLI and template. For a single security group, some workflows accept one value; for multiple groups, use a comma-delimited list such as `sg-aaa,sg-bbb`.

**CloudFormation and IAM instance profiles**

If the instance needs AWS API access, define an IAM role and instance profile, then attach the profile to the EC2 instance with `IamInstanceProfile`.

```yaml
Resources:
  WebRole:
    Type: AWS::IAM::Role
    Properties:
      AssumeRolePolicyDocument:
        Version: '2012-10-17'
        Statement:
          - Effect: Allow
            Principal:
              Service: ec2.amazonaws.com
            Action: sts:AssumeRole
      ManagedPolicyArns:
        - arn:aws:iam::aws:policy/AmazonSSMManagedInstanceCore

  WebInstanceProfile:
    Type: AWS::IAM::InstanceProfile
    Properties:
      Roles:
        - !Ref WebRole

  WebInstance:
    Type: AWS::EC2::Instance
    Properties:
      ImageId: ami-0123456789abcdef0
      InstanceType: t3.micro
      IamInstanceProfile: !Ref WebInstanceProfile
```

This is a common exam trap: creating a role or instance profile is not enough. The instance must reference `IamInstanceProfile` to actually use it.

**Networking and placement choices**

EC2 launch behavior depends heavily on network placement. CloudFormation and the CLI both let you control where the instance lands.

- **SubnetId** places the instance in a specific subnet and Availability Zone.
- **SecurityGroupIds** defines inbound and outbound filtering.
- **NetworkInterfaces** gives you more control than simple subnet and security group properties.
- **AssociatePublicIpAddress** is relevant when launching into a public subnet and you need direct internet reachability.

Example with explicit network interface configuration:

```yaml
Resources:
  WebInstance:
    Type: AWS::EC2::Instance
    Properties:
      ImageId: ami-0123456789abcdef0
      InstanceType: t3.micro
      NetworkInterfaces:
        - DeviceIndex: '0'
          SubnetId: subnet-0abc1234def567890
          GroupSet:
            - sg-0123abcd4567ef890
          AssociatePublicIpAddress: true
      Tags:
        - Key: Name
          Value: web-eni
```

Use explicit network interfaces when you need advanced control, such as multiple ENIs, fixed private IPs, or precise attachment behavior.

**Launch templates and versioning**

A launch template stores reusable instance settings such as AMI, instance type, network configuration, block device mappings, metadata options, and user data. It is commonly used with Auto Scaling groups and can also be referenced by other launch workflows.

Example launch template:

```yaml
Resources:
  WebLaunchTemplate:
    Type: AWS::EC2::LaunchTemplate
    Properties:
      LaunchTemplateData:
        ImageId: ami-0123456789abcdef0
        InstanceType: t3.micro
        MetadataOptions:
          HttpTokens: required
        UserData:
          Fn::Base64: !Sub |
            #!/bin/bash
            yum update -y
            yum install -y httpd
```

Version selection matters:

- **`$Default`** uses the template’s default version.
- **`$Latest`** uses the most recently created version.
- **Explicit version numbers** are useful for controlled rollouts and reproducibility.

Example CLI usage:

```bash
aws ec2 run-instances \
  --launch-template LaunchTemplateName=web-template,Version=$Default
```

Use explicit versions when you want predictable deployments. Use `$Latest` when you intentionally want new launches to pick up the newest template revision.

**IMDSv2 and metadata options**

The Instance Metadata Service version 2 (<abbr title="Instance Metadata Service version 2">IMDSv2</abbr>) improves protection against certain metadata access attacks by requiring session-oriented requests. In CloudFormation, metadata options are supported on `AWS::EC2::Instance` and `AWS::EC2::LaunchTemplate`.

Example:

```yaml
MetadataOptions:
  HttpTokens: required
  HttpEndpoint: enabled
```

Practical meaning:

- **`HttpTokens: required`** enforces IMDSv2.
- **`HttpEndpoint: enabled`** keeps metadata access available.
- **`HttpPutResponseHopLimit`** can matter in container or proxy scenarios.

This is often tested as a security hardening choice for EC2 launch workflows.

**Parameters, mappings, and conditions**

CloudFormation lets you vary EC2 configuration by environment without duplicating templates. This is a common exam pattern.

- **Parameters** accept values at stack creation or update.
- **Mappings** translate environment or region values into AMIs or settings.
- **Conditions** control whether resources or properties are created.

Example with environment-based instance sizing:

```yaml
Parameters:
  Environment:
    Type: String
    AllowedValues: [dev, test, prod]

Mappings:
  EnvMap:
    dev:
      InstanceType: t3.micro
    prod:
      InstanceType: t3.small

Resources:
  WebInstance:
    Type: AWS::EC2::Instance
    Properties:
      ImageId: ami-0123456789abcdef0
      InstanceType: !FindInMap [EnvMap, !Ref Environment, InstanceType]
```

This pattern reduces template sprawl and keeps launch behavior consistent across environments.

**Outputs and cross-stack references**

Outputs expose values from a stack, such as instance ID, private IP, or public IP. They are useful for operators and for other stacks that need to consume the result of an EC2 launch.

Example outputs:

```yaml
Outputs:
  InstanceId:
    Value: !Ref WebInstance
  PrivateIp:
    Value: !GetAtt WebInstance.PrivateIp
```

For multi-stack designs, you can export values and import them elsewhere.

```yaml
Outputs:
  WebInstanceId:
    Value: !Ref WebInstance
    Export:
      Name: WebInstanceIdExport
```

Another stack can consume it with `Fn::ImportValue`. This is useful when a network stack and compute stack are managed separately.

**Rollback, stack events, and troubleshooting**

CloudFormation tracks resource creation and failure states. If a stack fails during creation, CloudFormation typically rolls back the resources it created unless rollback is disabled.

Useful troubleshooting commands:

```bash
aws cloudformation describe-stack-events --stack-name web-ec2-dev
```

```bash
aws cloudformation describe-stacks --stack-name web-ec2-dev
```

What to look for:

- **CREATE_FAILED** events show the first failing resource.
- **Status reason** often identifies invalid AMIs, bad subnet IDs, missing permissions, or malformed user data.
- **Dependency order** matters when a resource depends on IAM, networking, or security group creation.

Operational controls to know:

- **Stack policy** can protect critical resources from updates.
- **Rollback triggers** can integrate CloudWatch alarms to abort a deployment if health degrades.
- **Termination protection** helps prevent accidental stack deletion.

> If an EC2 stack fails, inspect the earliest failed event first. Later failures are often cascading errors.

**Common replacement and update behavior**

Some EC2 property changes are disruptive. In CloudFormation, a property update may be in-place or may require replacement depending on the resource and property.

Commonly disruptive changes include:

- **AMI changes** often require replacement for `AWS::EC2::Instance`.
- **Subnet changes** often require replacement because the instance is tied to placement.
- **Instance type changes** may require replacement depending on the resource and update path.
- **User data changes** on `AWS::EC2::Instance` are not typically applied in place; plan for replacement or a bootstrap mechanism that runs on every boot.

For operational safety, use immutable deployment patterns when possible:

- Launch a new instance with the updated template.
- Validate health.
- Shift traffic with a load balancer or DNS.
- Retire the old instance.

**CLI versus CloudFormation comparison**

| Aspect | AWS CLI | CloudFormation |
|---|---|---|
| Launch style | Imperative, one command at a time | Declarative, template-driven |
| Repeatability | Depends on operator consistency | High, because the template is versioned |
| Bootstrap | `--user-data` or launch template data | `UserData` in template or launch template |
| IAM access | `--iam-instance-profile` | `IamInstanceProfile` property |
| Rollback | No orchestration by the CLI itself | Stack rollback on failure |
| Multi-resource orchestration | Manual | Built in |
| Best use case | Ad hoc or scripted launches | Standardized environments and controlled change |

The key exam distinction is that the CLI launches an instance directly, while CloudFormation manages the resource lifecycle as part of a stack.

**Practical launch patterns**

1. Launch a single instance directly with the CLI when you need a quick test host.
2. Launch a single instance with CloudFormation when you need repeatability and parameter control.
3. Use a launch template when you want a reusable launch specification for multiple launches or Auto Scaling.
4. Use an IAM instance profile whenever the instance must call AWS services.
5. Use user data to install software and configure the host at first boot.

**Exam-focused reminders**

- `AWS::EC2::Instance` launches a single EC2 instance directly.
- `AWS::EC2::LaunchTemplate` is reusable launch metadata; it is not the same as an Auto Scaling group.
- `AWS::AutoScaling::LaunchConfiguration` is the older Auto Scaling construct and is separate from EC2 launch templates.
- `IamInstanceProfile` must be attached to the instance to use the role.
- `UserData` is base64-encoded in CloudFormation.
- `create-stack` and `deploy` are common CloudFormation workflows for EC2.
- `describe-stack-events` is the first place to check when a stack fails.
