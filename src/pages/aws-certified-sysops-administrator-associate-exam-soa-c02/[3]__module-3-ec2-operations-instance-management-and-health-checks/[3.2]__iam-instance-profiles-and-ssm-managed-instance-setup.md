## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**IAM instance profiles: what they are and why they matter**

An **instance profile** is the EC2-facing container for an IAM role. You attach the profile to an EC2 instance so applications on that instance can call AWS APIs without storing long-term credentials on disk.

The role is the identity and permission policy; the instance profile is the delivery mechanism that EC2 uses to expose that role to the instance metadata service.

- **Role**: defines permissions and trust policy.
- **Instance profile**: holds one role for EC2 use.
- **Temporary credentials**: delivered through the instance metadata service and rotated automatically.

A common exam trap is to confuse the role with the profile. EC2 attaches the profile, not the role directly.

**Core concepts for EC2 access**

When an instance has an attached profile, the AWS SDKs and CLI can retrieve temporary credentials from the instance metadata service (IMDS, Instance Metadata Service). This is the standard way to grant least-privilege access to S3, CloudWatch, Systems Manager, and other services.

Use instance profiles when you need:

- Application access to AWS APIs from EC2.
- Secure, short-lived credentials instead of static access keys.
- A standard way to authorize Systems Manager managed instance registration.

> <mark>Security note:</mark> For exam scenarios, prefer IMDSv2 and restrict IMDSv1 by requiring token-based metadata access.

**Creating an IAM role and instance profile**

You usually create the role first, then attach it to an instance profile. In the AWS Console, the profile is often created automatically when you create a role for EC2, but it is still the object EC2 attaches.

Typical trust policy for an EC2 role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": { "Service": "ec2.amazonaws.com" },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

A minimal Systems Manager role usually includes the managed policy `AmazonSSMManagedInstanceCore`. That policy grants the permissions the SSM Agent needs to register and communicate with Systems Manager.

**Console steps: create a role and instance profile**

1. Open the IAM console and choose **Roles**.
2. Choose **Create role**.
3. Select **AWS service** and choose **EC2** as the use case.
4. Attach the policy `AmazonSSMManagedInstanceCore`.
5. Name the role, such as `EC2-SSM-Role`.
6. Finish creation.

If you create the role for EC2 in the console, AWS creates the instance profile with the same name in the background.

**CLI steps: create and attach an instance profile**

```bash
aws iam create-role \
  --role-name EC2-SSM-Role \
  --assume-role-policy-document file://ec2-trust-policy.json

aws iam attach-role-policy \
  --role-name EC2-SSM-Role \
  --policy-arn arn:aws:iam::aws:policy/AmazonSSMManagedInstanceCore

aws iam create-instance-profile \
  --instance-profile-name EC2-SSM-Profile

aws iam add-role-to-instance-profile \
  --instance-profile-name EC2-SSM-Profile \
  --role-name EC2-SSM-Role
```

To verify the attachment:

```bash
aws iam get-instance-profile --instance-profile-name EC2-SSM-Profile
```

**Attaching an existing profile to an existing EC2 instance**

This is a frequent operational task and a common exam requirement. You can attach or replace the instance profile on a running instance.

**Console procedure**

1. Open the EC2 console.
2. Select the target instance.
3. Choose **Actions**.
4. Choose **Security**.
5. Choose **Modify IAM role**.
6. Select the existing instance profile from the dropdown.
7. Save the change.

**CLI procedure**

1. Identify the instance ID and the profile name.
2. Associate the profile with the instance.

```bash
aws ec2 associate-iam-instance-profile \
  --instance-id i-0123456789abcdef0 \
  --iam-instance-profile Name=EC2-SSM-Profile
```

If the instance already has a profile, use `replace-iam-instance-profile-association` instead of a fresh association.

```bash
aws ec2 describe-iam-instance-profile-associations

aws ec2 replace-iam-instance-profile-association \
  --association-id iip-assoc-0123456789abcdef0 \
  --iam-instance-profile Name=EC2-SSM-Profile
```

**Instance metadata service and IMDSv2 enforcement**

The EC2 instance metadata service provides credentials and instance details to software running on the instance. For secure access, you should require IMDSv2, which uses session-oriented requests with a token.

Relevant metadata options include:

- `HttpTokens=required`: forces IMDSv2.
- `HttpEndpoint=enabled`: keeps metadata available.
- `HttpPutResponseHopLimit`: controls how far metadata responses can travel.

Example CLI to enforce IMDSv2 on an existing instance:

```bash
aws ec2 modify-instance-metadata-options \
  --instance-id i-0123456789abcdef0 \
  --http-tokens required \
  --http-endpoint enabled
```

You can also set this at launch time in the launch template or instance configuration. For exam purposes, remember that requiring IMDSv2 reduces the risk of credential theft from SSRF (Server-Side Request Forgery) style attacks.

**How SSM managed instance registration works**

A **Systems Manager managed instance** is an EC2 instance that has successfully registered with AWS Systems Manager. Once registered, you can use Session Manager, Run Command, Patch Manager, Inventory, and other SSM features.

For registration, the instance needs all of the following:

- An IAM role attached through an instance profile.
- The `AmazonSSMManagedInstanceCore` permissions or equivalent.
- The SSM Agent installed and running.
- Network access to Systems Manager endpoints, either through the internet/NAT or VPC endpoints.
- Correct region alignment between the instance and the Systems Manager endpoints you use.

The SSM Agent uses the instance profile credentials to call Systems Manager APIs and register itself as a managed node.

**Service-linked role for Systems Manager**

Systems Manager uses a service-linked role in some workflows. A service-linked role is an IAM role that AWS creates and manages for a service.

For Systems Manager:

- AWS may create the service-linked role automatically when you use a feature that requires it.
- You can also create it explicitly if needed.
- It is used by the service, not by the EC2 instance itself.

This is different from the EC2 instance role. The instance role authorizes the instance; the service-linked role authorizes AWS Systems Manager to perform service-side actions.

**SSM Agent prerequisites by OS and AMI type**

The SSM Agent is the software that runs on the instance and communicates with Systems Manager.

Common exam-relevant points:

- Many modern Amazon Linux AMIs include the SSM Agent by default.
- Many current Ubuntu, Windows Server, and Amazon Linux 2/2023 images often include it or make installation straightforward, but you should verify.
- Older AMIs, custom images, and some minimal distributions may not include the agent.
- If the agent is missing, you must install it manually and ensure it starts at boot.

Examples of manual installation:

```bash
# Amazon Linux / RHEL-like systems
sudo yum install -y amazon-ssm-agent
sudo systemctl enable amazon-ssm-agent
sudo systemctl start amazon-ssm-agent
```

```bash
# Ubuntu example
sudo snap install amazon-ssm-agent --classic
sudo systemctl enable snap.amazon-ssm-agent.amazon-ssm-agent.service
sudo systemctl start snap.amazon-ssm-agent.amazon-ssm-agent.service
```

On Windows, the agent is commonly preinstalled on supported AMIs, but you may need to install or repair it on custom images.

**Network requirements for SSM**

The agent must reach Systems Manager endpoints. You can do this in two main ways:

- Through the public internet via an internet gateway or NAT gateway.
- Privately through VPC endpoints.

For private subnets, the preferred exam answer is usually VPC interface endpoints with private DNS enabled.

Required endpoints are typically:

- `com.amazonaws.<region>.ssm`
- `com.amazonaws.<region>.ec2messages`
- `com.amazonaws.<region>.ssmmessages`

These are **interface endpoints**, not gateway endpoints.

> <mark>Important:</mark> Enable **private DNS** for the SSM interface endpoints in most private-subnet designs so the standard service names resolve to the VPC endpoints.

Example endpoint creation:

```bash
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0123456789abcdef0 \
  --service-name com.amazonaws.us-east-1.ssm \
  --vpc-endpoint-type Interface \
  --subnet-ids subnet-aaa subnet-bbb \
  --security-group-ids sg-0123456789abcdef0 \
  --private-dns-enabled
```

Repeat for `ec2messages` and `ssmmessages`.

**Security groups and endpoint access**

For interface endpoints, the endpoint security group must allow inbound HTTPS from the instance security group or subnet CIDR. The instance must also allow outbound HTTPS to the endpoints.

Typical pattern:

- Instance security group: outbound TCP 443 allowed.
- Endpoint security group: inbound TCP 443 from the instance security group.
- NACLs: allow ephemeral return traffic and HTTPS.

**End-to-end setup flow**

Use this sequence when building a new managed instance from scratch.

1. Create or choose an IAM role for EC2.
2. Attach `AmazonSSMManagedInstanceCore`.
3. Ensure the role is in an instance profile.
4. Configure the launch template or instance settings to require IMDSv2.
5. Launch the EC2 instance with the profile attached.
6. Confirm the SSM Agent is installed and running.
7. Provide network access to SSM endpoints through NAT or VPC interface endpoints.
8. Wait for the instance to appear in Systems Manager as a managed node.
9. Open Session Manager and start a shell session.

Example launch template snippet for metadata options:

```json
{
  "MetadataOptions": {
    "HttpTokens": "required",
    "HttpEndpoint": "enabled",
    "HttpPutResponseHopLimit": 2
  }
}
```

**Session Manager connection example**

Once the instance is managed, you can connect without SSH keys or inbound port 22.

```bash
aws ssm start-session --target i-0123456789abcdef0
```

This is often the preferred access method because it removes the need for bastion hosts and inbound SSH rules.

**Troubleshooting managed instance registration**

When an instance does not show up in Systems Manager, check the failure domain in order.

- **IAM permissions**: confirm the instance profile is attached and includes `AmazonSSMManagedInstanceCore`.
- **Permissions boundaries or SCPs**: a permissions boundary or Service Control Policy can block required actions even when the policy is attached.
- **Region mismatch**: the instance must talk to the correct regional SSM endpoints.
- **Agent status**: verify the SSM Agent is installed, enabled, and running.
- **Metadata access**: confirm the instance can reach IMDS and that IMDSv2 requirements are not breaking older software.
- **Network path**: verify outbound HTTPS to SSM endpoints or correct VPC interface endpoints.
- **Private DNS**: if using endpoints, ensure private DNS is enabled so the standard service names resolve correctly.
- **Endpoint security groups**: confirm the endpoint allows inbound 443 from the instance.
- **OS support**: very old or custom AMIs may need manual agent installation.

Useful checks on Linux:

```bash
sudo systemctl status amazon-ssm-agent
sudo journalctl -u amazon-ssm-agent -n 100 --no-pager
curl -s http://169.254.169.254/latest/meta-data/instance-id
```

For IMDSv2 testing:

```bash
TOKEN=$(curl -X PUT "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")

curl -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/meta-data/instance-id
```

If the metadata call fails, the agent may not be able to retrieve credentials.

**Practical exam distinctions**

- Use an **instance profile** when EC2 needs AWS permissions.
- Use **Session Manager** when you want shell access without SSH keys or open inbound ports.
- Use **IMDSv2** when the question asks for stronger metadata security.
- Use **interface VPC endpoints** for private SSM access in isolated subnets.
- Use the **service-linked role** concept when the question is about Systems Manager service-side permissions, not instance permissions.

**Quick comparison**

| Item | Purpose | Who uses it |
|---|---|---|
| IAM role | Defines permissions and trust | EC2 and AWS services |
| Instance profile | Attaches role to EC2 | EC2 |
| SSM Agent | Registers and communicates with SSM | Instance OS |
| Service-linked role | Lets AWS service act on your behalf | Systems Manager service |

**Minimal working example**

A common secure setup for a private EC2 instance looks like this:

- EC2 instance in a private subnet.
- Instance profile with `AmazonSSMManagedInstanceCore`.
- SSM Agent installed and running.
- IMDSv2 required.
- Interface endpoints for `ssm`, `ec2messages`, and `ssmmessages`.
- Private DNS enabled on the endpoints.
- Security groups allowing HTTPS to the endpoints.

With that in place, you can register the instance and connect through Session Manager without exposing SSH to the internet.

**Common mistakes to avoid**

- Attaching the role but forgetting the instance profile association.
- Launching the instance without the SSM Agent.
- Using only the `ssm` endpoint and forgetting `ec2messages` and `ssmmessages`.
- Creating endpoints without private DNS in a design that expects standard DNS names.
- Assuming the service-linked role replaces the EC2 instance role.
- Ignoring SCPs or permissions boundaries that silently block registration.
- Forgetting to require IMDSv2 in hardened environments.
