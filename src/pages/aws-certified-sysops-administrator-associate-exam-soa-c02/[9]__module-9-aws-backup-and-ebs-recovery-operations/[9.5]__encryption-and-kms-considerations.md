## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**AWS KMS encryption hierarchy**

AWS Key Management Service (AWS KMS) centers on a **KMS key** and the cryptographic material behind it. Older documentation may still say *customer master key (CMK)*, but the current term is KMS key.

A KMS key is not used directly to encrypt large payloads. Instead, AWS uses envelope encryption:

- The KMS key protects a **data key**.
- The data key encrypts the actual data.
- The plaintext data key is discarded after use.

This design improves performance and limits how often KMS must process large data operations. In practice, you usually see KMS involved in control-plane authorization and data-key generation, not in bulk encryption of file contents.

Key hierarchy terms to know:

- **KMS key**: The logical key resource in KMS.
- **Backing key material**: The cryptographic material associated with a KMS key.
- **Data key**: A short-lived key used to encrypt application data.
- **Envelope encryption**: Encrypt data with a data key, then encrypt the data key with a KMS key.

Common API actions map to this model:

- `Encrypt` and `Decrypt` for direct KMS encryption of small data blobs.
- `GenerateDataKey` to obtain a plaintext data key and an encrypted copy.
- `GenerateDataKeyWithoutPlaintext` when a service needs only the encrypted data key.
- `ReEncrypt` to rewrap ciphertext under a different KMS key.
- `DescribeKey` to inspect metadata, not grants or ciphertext.
- `ListGrants` to inspect active grants.

**KMS key types and when to use them**

AWS offers several key ownership models. The differences matter for policy control, rotation, and operational responsibility.

| Key type | Who manages it | Rotation | Policy control | Typical use |
|---|---|---:|---|---|
| AWS owned key | AWS | Automatic and fully managed | No customer policy control | Default encryption for some AWS services |
| AWS managed key | AWS for your account and service | Automatic for many services | Limited; you can use it but not fully administer it | Simple service encryption with minimal setup |
| Customer managed key | You | Configurable | Full key policy and IAM control | Most exam scenarios requiring explicit access control |
| Customer provided key material | You provide imported material | You control lifecycle of imported material | Full control, but with import constraints | Special compliance or portability cases |

Use these distinctions carefully:

- **AWS owned keys** are invisible to you as a resource.
- **AWS managed keys** are created and managed by AWS for a service in your account; you can view them and use them, but you do not control their policy in the same way as customer managed keys.
- **Customer managed keys** are the main choice when you need explicit permissions, auditing, rotation control, or cross-account sharing.
- **Imported key material** is useful when you must supply your own cryptographic material, but you must manage expiration and re-import behavior.

**Aliases and operational naming**

An **alias** is a friendly name that points to a KMS key. Aliases help you avoid hard-coding key IDs in scripts and service settings.

Operationally, aliases are useful because:

- They make automation easier when you rotate or replace keys.
- They let you reference a stable name like `alias/app-prod` instead of a changing key ID.
- Many AWS services accept an alias when you configure encryption settings.

Important alias behavior:

- An alias points to one KMS key at a time.
- You can repoint an alias to a different key.
- The alias itself is not the key; permissions still apply to the target KMS key.
- Some CLI operations are clearer with a key ID or ARN, especially when troubleshooting.

Example:

```bash
aws kms create-alias \
  --alias-name alias/app-prod \
  --target-key-id 1234abcd-12ab-34cd-56ef-1234567890ab
```

To inspect the target of an alias:

```bash
aws kms list-aliases --key-id 1234abcd-12ab-34cd-56ef-1234567890ab
```

**Key policies, IAM, and access evaluation**

KMS authorization is often the hardest exam topic because access depends on both the key policy and IAM, but not always in the same way.

For **same-account access**:

- The key policy must allow the action, either directly or by delegating to IAM.
- IAM permissions in the same account are evaluated together with the key policy.
- A common pattern is to allow the account root principal in the key policy and then use IAM policies to grant users and roles access.

For **cross-account access**:

- The key policy in the owning account must explicitly allow the external principal or account.
- IAM in the external account is not enough by itself.
- The external principal also needs permission to call the relevant KMS action from its own identity policy in many practical setups, but the key policy is the critical gate on the owning side.

A standard delegation-style key policy looks like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "EnableAccountPermissions",
      "Effect": "Allow",
      "Principal": { "AWS": "arn:aws:iam::111122223333:root" },
      "Action": "kms:*",
      "Resource": "*"
    }
  ]
}
```

This pattern does not grant every IAM user full access by itself. Instead, it allows the account to use IAM policies to manage access to the key.

Common failure modes:

- The IAM policy allows `kms:Decrypt`, but the key policy does not delegate access.
- The key policy allows the principal, but the IAM policy omits the action.
- The service role has S3 or EBS permissions, but the KMS key policy does not allow the service to use the key.
- A cross-account role trusts the source account, but the destination KMS key policy does not name the role or account.

**Grants and service access**

A **grant** is a lightweight permission mechanism that KMS uses to allow temporary or service-specific access to a key. Services often create grants automatically when they need to use a key on your behalf.

Use grants when:

- A service needs to use a key without changing the key policy repeatedly.
- You want scoped, revocable access for a specific principal or service.
- You need to support workflows where permissions are created dynamically.

Grant concepts to know:

- **Grant token**: A token that lets a newly created grant be used immediately before eventual consistency completes.
- **Grant constraints**: Limits on what the grantee can do, often tied to encryption context.
- **Retiring principal**: A principal that can retire a grant if permitted.

Example grant creation:

```bash
aws kms create-grant \
  --key-id 1234abcd-12ab-34cd-56ef-1234567890ab \
  --grantee-principal arn:aws:iam::111122223333:role/AppRole \
  --operations Decrypt Encrypt GenerateDataKey
```

Inspect grants with:

```bash
aws kms list-grants --key-id 1234abcd-12ab-34cd-56ef-1234567890ab
```

Practical notes:

- `DescribeKey` shows metadata, not grant details.
- Services such as EBS, RDS, and S3 may create grants automatically when they need to use a customer managed key.
- If a workflow fails immediately after grant creation, a grant token may be relevant in custom application code.

**Rotation and key lifecycle**

Rotation behavior depends on the key type.

- **Automatic rotation** is available for customer managed symmetric KMS keys.
- **AWS managed keys** are rotated automatically by AWS for many services, but you do not configure the schedule or policy details.
- **Imported key material** does not rotate automatically in the same way; you manage the imported material lifecycle.
- **Manual rotation** in KMS usually means creating a new key and updating aliases, applications, or service settings to use the new key.

Why rotation matters:

- It limits the amount of data protected by one key version.
- It supports compliance requirements.
- It gives you a clean operational path when you need to retire a key.

Example operational rotation pattern:

1. Create a new customer managed KMS key.
2. Update the alias to point to the new key.
3. Reconfigure services or applications if they reference the key directly.
4. Re-encrypt existing data if required by the workload.

**Deletion, disablement, and recovery window**

KMS key deletion is intentionally slow to reduce accidental data loss.

Key lifecycle states:

- **Enabled**: The key can be used.
- **Disabled**: The key cannot be used for cryptographic operations.
- **Pending deletion**: The key is scheduled for deletion after a waiting period.

Deletion safeguards:

- KMS requires a waiting period, commonly 7 to 30 days, before deletion completes.
- During the waiting period, you can cancel deletion.
- Disabling a key is often safer than deleting it when you are troubleshooting or planning a migration.

Operational impact:

- If you disable a key used by EBS, encrypted volumes cannot be attached or used normally until the key is re-enabled.
- If you disable a key used by RDS, database operations that require the key can fail.
- If you disable a key used by S3 objects encrypted with SSE-KMS, the objects remain stored, but any operation that requires decryption fails until the key is enabled again.

Example CLI workflow:

```bash
aws kms disable-key --key-id 1234abcd-12ab-34cd-56ef-1234567890ab
aws kms schedule-key-deletion --key-id 1234abcd-12ab-34cd-56ef-1234567890ab --pending-window-in-days 7
aws kms cancel-key-deletion --key-id 1234abcd-12ab-34cd-56ef-1234567890ab
```

> Be careful with aliases during deletion workflows. An alias can be repointed, but deleting the underlying key breaks any data still encrypted under that key.

**CloudTrail auditing and troubleshooting**

AWS CloudTrail records KMS API activity and is one of the first places you should check when encryption fails.

Useful event patterns:

- `Decrypt` failures when an application cannot read encrypted data.
- `GenerateDataKey` failures when a service cannot create a data key.
- `CreateGrant` events when AWS services set up access automatically.
- `DisableKey`, `ScheduleKeyDeletion`, and `CancelKeyDeletion` for lifecycle changes.

Troubleshooting approach:

1. Check CloudTrail for the exact KMS action and error code.
2. Confirm the principal in the event matches the expected IAM role or service principal.
3. Verify the key policy, IAM policy, and any grants.
4. Check whether the key is enabled and not pending deletion.
5. Confirm the encryption context if the application uses one.

Example CloudTrail lookup with the AWS CLI:

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=Decrypt
```

**Encryption context**

The **encryption context** is a set of non-secret key-value pairs that KMS binds to ciphertext. It is not secret, but it helps ensure that ciphertext is only decrypted in the intended context.

Use it when you want:

- Additional integrity checks.
- Application-level binding between ciphertext and metadata.
- Better auditability in logs and troubleshooting.

Example:

```bash
aws kms encrypt \
  --key-id 1234abcd-12ab-34cd-56ef-1234567890ab \
  --plaintext fileb://secret.txt \
  --encryption-context AppName=Payroll,Environment=Prod
```

If the decryption request uses a different encryption context, decryption fails.

**S3 encryption with KMS**

Amazon S3 supports server-side encryption with KMS keys, commonly called **SSE-KMS**.

Why use SSE-KMS:

- You want centralized key control.
- You need audit trails for object encryption and decryption.
- You want to use customer managed keys for compliance or separation of duties.

Basic bucket encryption configuration:

```json
{
  "Rules": [
    {
      "ApplyServerSideEncryptionByDefault": {
        "SSEAlgorithm": "aws:kms",
        "KMSMasterKeyID": "arn:aws:kms:us-east-1:111122223333:key/1234abcd-12ab-34cd-56ef-1234567890ab"
      }
    }
  ]
}
```

CLI example:

```bash
aws s3api put-bucket-encryption \
  --bucket my-secure-bucket \
  --server-side-encryption-configuration file://sse-kms.json
```

Permissions to remember:

- The uploader needs permission to use the KMS key.
- The bucket policy may need to enforce SSE-KMS.
- For replication, the replication role often needs a combination of S3 permissions and KMS permissions such as `kms:Decrypt`, `kms:Encrypt`, `kms:GenerateDataKey`, and sometimes `kms:ReEncrypt*`, depending on the source, destination, and replication mode.

Replication nuance:

- The exact KMS permissions depend on whether the source and destination buckets are encrypted, whether replication is same-account or cross-account, and whether the destination uses a different KMS key.
- Do not assume `kms:ReEncrypt*` is always required; many replication setups rely on `Decrypt`, `Encrypt`, and `GenerateDataKey` instead.

**EBS encryption with KMS**

Amazon Elastic Block Store (EBS) uses KMS keys to encrypt volumes and snapshots.

Common exam points:

- You can enable **default EBS encryption** so new volumes are encrypted automatically.
- You can choose the KMS key used for encryption.
- Snapshots of encrypted volumes remain encrypted.
- Copying encrypted snapshots across Regions or accounts requires KMS permissions and often a destination key.

Example to enable default EBS encryption:

```bash
aws ec2 enable-ebs-encryption-by-default
aws ec2 modify-ebs-default-kms-key-id \
  --kms-key-id arn:aws:kms:us-east-1:111122223333:key/1234abcd-12ab-34cd-56ef-1234567890ab
```

Operational impact:

- If the key is disabled, new volume operations and some snapshot workflows fail.
- Existing encrypted data remains encrypted; access depends on the key being available.

**RDS and Secrets Manager with KMS**

Amazon RDS uses KMS for encrypted databases and snapshots. AWS Secrets Manager also uses KMS to encrypt secret values.

RDS considerations:

- Choose a customer managed key when you need explicit control over access and rotation.
- Snapshot copy and cross-account restore scenarios often require destination-side KMS permissions.
- If the key is unavailable, database operations that need the key can fail.

Secrets Manager considerations:

- Secrets are encrypted at rest with a KMS key.
- The service can use the AWS managed key for the account or a customer managed key.
- Use a customer managed key when you need tighter control, cross-account sharing, or explicit audit boundaries.

Example secret creation with a customer managed key:

```bash
aws secretsmanager create-secret \
  --name prod/db/password \
  --secret-string '{"username":"appuser","password":"ExamplePass123!"}' \
  --kms-key-id arn:aws:kms:us-east-1:111122223333:key/1234abcd-12ab-34cd-56ef-1234567890ab
```

**AWS Backup and KMS**

AWS Backup uses KMS to encrypt backup vaults and backup copies. The authorization model is easy to oversimplify, so be precise.

What matters:

- The backup vault is associated with a KMS key.
- The KMS key policy must allow the AWS Backup service principal and, in some scenarios, the backup role or account principals involved in the backup workflow.
- Cross-account backup copy requires explicit permissions on both the source and destination sides.

Practical exam takeaway:

- Do not assume service-linked roles alone are enough.
- Check the vault key policy, the backup plan, and the copy destination permissions together.

**Multi-Region keys**

A **multi-Region key** is a set of related KMS keys in different Regions that share the same key material and key ID structure for replication purposes.

Use multi-Region keys when:

- You need active-active or disaster recovery encryption patterns.
- You want to replicate encrypted workloads across Regions with consistent key identity.
- You need to simplify cross-Region failover for applications that reference KMS keys.

Important details:

- Each Region has its own KMS key resource and its own key policy.
- The replica is related to the primary key, not an unrelated new key.
- You still manage access in each Region through the local key policy and IAM permissions.
- Replication copies key material and metadata, but not every operational setting behaves like a single global key.

**CLI and operational examples**

Inspect a key:

```bash
aws kms describe-key --key-id arn:aws:kms:us-east-1:111122223333:key/1234abcd-12ab-34cd-56ef-1234567890ab
```

Encrypt and decrypt a small payload:

```bash
aws kms encrypt \
  --key-id alias/app-prod \
  --plaintext fileb://message.txt \
  --output text --query CiphertextBlob > ciphertext.b64
```

```bash
aws kms decrypt \
  --ciphertext-blob fileb://ciphertext.bin
```

Generate a data key for application use:

```bash
aws kms generate-data-key \
  --key-id alias/app-prod \
  --key-spec AES_256
```

A simple application pattern:

```python
# Pseudocode
ciphertext_key = kms.generate_data_key(KeyId="alias/app-prod", KeySpec="AES_256")
plaintext_data_key = ciphertext_key["Plaintext"]
encrypted_data_key = ciphertext_key["CiphertextBlob"]

# Use plaintext_data_key to encrypt local data
# Store encrypted_data_key alongside the ciphertext
```

**Exam-focused troubleshooting checklist**

When encryption fails, check these in order:

1. Confirm the KMS key is **enabled** and not pending deletion.
2. Verify the principal has the required KMS action, such as `Decrypt`, `Encrypt`, or `GenerateDataKey`.
3. Check the key policy for same-account delegation or explicit cross-account access.
4. Review grants if a service is involved.
5. Confirm the correct Region and key ARN are being used.
6. Validate the encryption context if the application supplies one.
7. Check CloudTrail for the exact denied action and principal.

**High-value distinctions to remember**

- **Encrypt** is not the same as **GenerateDataKey**.
- **DescribeKey** does not reveal grants.
- **Alias** is a pointer, not the key itself.
- **Disabled** is reversible; **scheduled deletion** is time-delayed but destructive.
- **AWS managed keys** are service-managed; **customer managed keys** are customer-controlled.
- **Cross-account access** depends heavily on the key policy in the owning account.
- **SSE-KMS** on S3, EBS encryption, RDS encryption, Secrets Manager, and AWS Backup all use KMS differently enough that you should check the service-specific permissions model.
