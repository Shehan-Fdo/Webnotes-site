## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Backup validation and restore testing**

Backups are only useful if you can restore them successfully. In AWS, validation means proving that a backup is usable, complete, and recoverable within your recovery objectives. Restore testing goes one step further by actually performing a recovery in a controlled environment and checking that the application, data, and dependencies behave as expected.

A strong backup strategy focuses on three questions:

- Can you find the right recovery point?
- Can you restore it within the required time?
- Does the restored system work correctly after recovery?

This topic is closely tied to **RPO** (Recovery Point Objective) and **RTO** (Recovery Time Objective). RPO defines how much data loss is acceptable. RTO defines how long recovery can take. Validation and testing help you confirm that your backup design meets both targets.

**Core concepts**

- **Backup validation** confirms that backup artifacts exist, are current, and can be read or restored.
- **Restore testing** confirms that a backup can be recovered into a usable state.
- **Recovery verification** checks that the restored workload functions as intended, not just that the restore job completed.
- **Isolation** reduces risk by keeping test restores separate from production traffic and data paths.

**Why validation matters**

A backup job that reports success does not guarantee a successful recovery. Common failure causes include:

- Corrupted backup data
- Missing permissions for restore operations
- Incorrect encryption key access
- Application-level inconsistency
- Dependency failures after restore, such as DNS, IAM, or database connection issues

Validation gives you confidence before an incident occurs. It also helps you detect problems such as expired credentials, broken lifecycle policies, or misconfigured backup vault access.

**AWS Backup and restore testing**

AWS Backup centralizes backup management for supported AWS services and can orchestrate restore jobs. For some resource types and features, AWS Backup also supports restore testing workflows that help you validate recovery in a controlled way. The exact capabilities vary by resource type, so you should always check the documented restore metadata requirements for the specific service you are restoring.

Important points:

- Restore testing is not identical for every resource type.
- The metadata required for a restore job depends on the source service.
- Some restores require additional parameters such as target network settings, IAM roles, or instance configuration.
- You should treat restore testing as a service-specific process, not a one-size-fits-all command.

**Validation methods**

You can validate backups in several ways, depending on the workload.

- **Metadata checks**
  - Confirm the backup exists in the expected vault or repository.
  - Verify creation time, retention, and encryption settings.
  - Confirm the recovery point is within the required retention window.

- **Integrity checks**
  - Restore the backup and verify file hashes or application checksums.
  - Compare restored data against known-good records.
  - Check logs for restore warnings or partial failures.

- **Functional checks**
  - Start the application and confirm it serves requests.
  - Run database consistency checks.
  - Verify that dependent services can connect and authenticate.

- **Operational checks**
  - Measure restore duration against the RTO.
  - Confirm that runbooks are accurate.
  - Validate that the team can execute the restore under pressure.

**Practical restore testing workflow**

A safe restore test usually follows a repeatable process:

1. Select a recovery point that matches the scenario you want to test.
2. Restore into an isolated environment, such as a separate account, VPC, subnet, or test stack.
3. Verify that the restored resource is complete and accessible.
4. Run application-level checks and compare results with expected behavior.
5. Record the restore duration, issues found, and corrective actions.

This workflow helps you test both the backup data and the operational process.

**Example: EBS volume restore validation**

When restoring an Amazon Elastic Block Store (Amazon EBS) volume, the restored volume may attach to a Nitro-based instance as an NVMe device. The device name you see in the operating system is often different from the original Linux device name. Do not assume a specific partition name such as `/dev/nvme1n1p1`; the restored volume may appear as the whole block device, such as `/dev/nvme1n1`, depending on how the source volume was formatted and partitioned.

Example commands to inspect the attached device:

```bash
lsblk
sudo blkid
sudo file -s /dev/nvme1n1
sudo mount /dev/nvme1n1 /mnt/restore-test
```

If the source volume used partitions, inspect the partition table first:

```bash
sudo fdisk -l /dev/nvme1n1
```

Then mount the correct partition if one exists:

```bash
sudo mount /dev/nvme1n1p1 /mnt/restore-test
```

Use the actual device layout discovered on the instance rather than assuming a fixed path.

> Do not hardcode restored EBS device names in runbooks. Nitro instances expose EBS volumes as NVMe devices, and the exact mapping can differ from the original attachment name.

**Example: AWS Backup restore job submission**

AWS Backup restore jobs require resource-type-specific metadata. The exact keys and values depend on the service being restored, and they must match the documented restore metadata schema for that resource type. For example, an EC2 instance restore may require different metadata than an RDS database restore or an EFS file system restore.

A generic pattern looks like this:

```bash
aws backup start-restore-job \
  --recovery-point-arn arn:aws:backup:us-east-1:123456789012:recovery-point:example \
  --metadata '{"key1":"value1","key2":"value2"}' \
  --iam-role-arn arn:aws:iam::123456789012:role/AWSBackupRestoreRole
```

Use the service documentation to determine the correct metadata keys before submitting the job. If the metadata is incomplete or incorrect, the restore can fail or produce an unusable resource.

**Common validation targets by workload**

- **Amazon EBS**
  - Confirm the volume attaches successfully.
  - Verify filesystem integrity and mountability.
  - Check application data consistency after mount.

- **Amazon RDS**
  - Confirm the restored DB instance becomes available.
  - Validate schema, row counts, and application connectivity.
  - Check parameter group and option group settings if relevant.

- **Amazon EFS**
  - Confirm the file system mounts from the test environment.
  - Verify file permissions, ownership, and application access.
  - Test read/write behavior for critical paths.

- **Amazon EC2**
  - Confirm the instance boots and reaches the expected state.
  - Validate user data, IAM role behavior, and network access.
  - Check application startup and service health.

- **Amazon DynamoDB**
  - Confirm the table or point-in-time recovery result contains expected items.
  - Validate indexes, throughput settings, and application reads.

**Isolated test environments**

Restore testing is safest when you isolate the restored resource from production. Isolation can mean:

- A separate AWS account
- A separate VPC
- A separate subnet
- Different security groups
- A dedicated test IAM role
- No public exposure unless explicitly required

Isolation reduces the chance of accidental writes, duplicate processing, or traffic conflicts. It also lets you test destructive scenarios without affecting live users.

**Important limitations**

- Not every resource type supports the same restore-testing workflow.
- Some workloads require manual post-restore steps before they are usable.
- Application consistency may require quiescing the workload before backup.
- A successful restore does not guarantee the application is healthy without further checks.

**Operational best practices**

- Test restores on a schedule, not only during incidents.
- Document the exact restore steps for each resource type.
- Store restore credentials and IAM permissions in a controlled, auditable way.
- Validate encryption key access for both backup and restore paths.
- Measure restore time and compare it to your RTO.
- Keep test results and remediation actions in a change log.

**Example: restore validation checklist**

- Recovery point exists and is within retention.
- Restore permissions are available.
- Encryption keys are accessible.
- Restored resource is isolated from production.
- Data is complete and readable.
- Application starts successfully.
- Dependencies resolve correctly.
- Restore duration meets the RTO target.

**Troubleshooting restore failures**

Common issues include:

- **Permissions**
  - The restore role lacks access to the backup vault, KMS key, or target resource.
- **Encryption**
  - The backup is encrypted with a key that the restore role cannot use.
- **Metadata**
  - The restore job uses incorrect or incomplete service-specific metadata.
- **Networking**
  - The restored resource cannot reach required endpoints or subnets.
- **Application state**
  - The data restored correctly, but the application still needs manual repair or reconfiguration.

When troubleshooting, start with the restore job status, then inspect service logs, IAM policies, and resource configuration. If the restore succeeded but the workload is broken, focus on application dependencies and configuration drift.

**Exam focus**

For the exam, remember these ideas:

- Backups must be validated by restoring and verifying usability.
- Restore testing should be isolated from production whenever possible.
- AWS Backup restore metadata is service-specific.
- A completed restore job is not the same as a successful application recovery.
- EBS restores on Nitro instances often appear as NVMe devices, so verify the actual device mapping before mounting.
