## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**EBS snapshot fundamentals**

Amazon Elastic Block Store (<abbr title="Elastic Block Store">EBS</abbr>) snapshots are point-in-time backups of EBS volumes attached to Amazon EC2 instances. You use them to protect data, clone volumes, migrate data across Availability Zones, and recover from corruption or accidental deletion.

A snapshot is created from a live volume, and the source volume can remain in use while the snapshot is being taken. AWS captures the blocks that have changed since the previous snapshot in the same snapshot lineage, which makes later snapshots incremental. The first snapshot is a full baseline; subsequent snapshots store only changed blocks, while AWS preserves the ability to reconstruct the full volume state.

- Snapshots are **region-scoped** resources.
- You restore a snapshot by creating a new volume from it.
- You cannot overwrite an existing EBS volume in place with a snapshot.
- You can replace the old volume at the application or instance level after restoring a new one.

**Why snapshots matter**

Snapshots are a core recovery tool for SysOps because they support operational backup, disaster recovery, and environment cloning without requiring the instance to be stopped in many cases. They are especially useful for application data volumes, boot volume recovery, and moving data between accounts or Regions.

Common use cases include:

- Recovering from accidental file deletion or filesystem corruption.
- Creating a test or staging copy of production data.
- Migrating data to a new instance or Availability Zone.
- Preserving a known-good state before patching or schema changes.
- Restoring a failed root volume to recover an unbootable instance.

**Creating a snapshot**

You can create a snapshot from the AWS Management Console, AWS CLI, or SDKs. In the console, you typically select the EBS volume, choose the snapshot action, and optionally add tags and a description. In the CLI, you create the snapshot from the volume ID.

```bash
aws ec2 create-snapshot \
  --volume-id vol-0123456789abcdef0 \
  --description "Pre-patch backup for app-data"
```

If you want to tag the snapshot at creation time, use `--tag-specifications` with the correct structure for the AWS CLI shell you are using.

```bash
aws ec2 create-snapshot \
  --volume-id vol-0123456789abcdef0 \
  --description "Pre-patch backup for app-data" \
  --tag-specifications 'ResourceType=snapshot,Tags=[{Key=Name,Value=app-data-prepatch},{Key=Environment,Value=prod}]'
```

- Use **tags** to support cost allocation, automation, and retention workflows.
- Use a clear **description** to identify the purpose and change window.
- Snapshot creation is asynchronous; the command returns before the snapshot is fully completed.

**Console workflow for snapshot creation**

In the AWS Management Console, the typical flow is:

1. Open the EC2 console and go to **Elastic Block Store > Volumes**.
2. Select the source volume attached to the instance.
3. Choose **Actions > Create snapshot**.
4. Enter a description and add tags if needed.
5. Create the snapshot and monitor its status under **Snapshots**.

The source volume can continue serving traffic during snapshot creation. For application consistency, you may still need to quiesce the filesystem or application first, depending on the workload.

> For databases and write-heavy applications, a crash-consistent snapshot may not be enough. Use application-aware backup steps, such as flushing buffers or pausing writes, when consistency requirements demand it.

**Consistency and application state**

EBS snapshots are typically crash-consistent by default. That means the snapshot reflects the volume as if power were lost at the capture point. For many Linux filesystems this is acceptable, but it does not guarantee that in-memory application state is committed to disk.

To improve consistency:

- Stop the application briefly if downtime is acceptable.
- Flush filesystem buffers before snapshotting.
- Use database-native backup procedures when required.
- Coordinate with AWS Systems Manager or scripts for repeatable backup workflows.

Example Linux flush sequence before snapshotting:

```bash
sync
sudo fsfreeze -f /data
# create snapshot here
sudo fsfreeze -u /data
```

Use `fsfreeze` only when the filesystem and workload support it. Freezing the wrong mount or leaving it frozen too long can disrupt the instance.

**Snapshot restore basics**

Restoring a snapshot means creating a new EBS volume from that snapshot. You must choose the same AWS Region as the snapshot because snapshots are regional. The Availability Zone is selected when you create the new volume, and it must match the target instance’s AZ if you plan to attach it there.

Key restore rules:

- A snapshot can create a new volume in the same Region.
- The new volume can be placed in any Availability Zone within that Region.
- You cannot attach a volume across AZ boundaries.
- If you need the data in another Region, copy the snapshot first, then restore from the copy.

CLI example for restoring a volume:

```bash
aws ec2 create-volume \
  --snapshot-id snap-0123456789abcdef0 \
  --availability-zone us-east-1a \
  --volume-type gp3
```

You can then attach the new volume to an instance in the same AZ.

```bash
aws ec2 attach-volume \
  --volume-id vol-0abcdef1234567890 \
  --instance-id i-0123456789abcdef0 \
  --device /dev/sdf
```

On Nitro-based instances, the operating system may expose the device as an NVMe device with a different name than the one you specified. Do not assume the Linux device path will match the attachment name exactly.

**Console workflow for restore**

In the AWS Management Console, the restore flow is:

1. Open **EC2 > Snapshots**.
2. Select the snapshot.
3. Choose **Actions > Create volume from snapshot**.
4. Select the target **Region** and **Availability Zone**.
5. Choose the volume type, size, and encryption settings if applicable.
6. Create the volume.
7. Attach it to an instance if needed.

After attachment, verify the device mapping in the guest OS before mounting.

**Mounting and validation after restore**

After you attach the restored volume, you usually need to discover the actual device name, inspect partitions, mount the filesystem, and validate the data. This is especially important on Nitro instances where NVMe naming can differ from the requested device name.

Typical Linux validation steps:

```bash
lsblk
sudo blkid
sudo file -s /dev/nvme1n1
sudo mkdir -p /mnt/recovered
sudo mount /dev/nvme1n1p1 /mnt/recovered
df -h /mnt/recovered
```

Use the actual device and partition discovered on the instance. If the volume has no partition table, mount the raw device only if the filesystem is directly on it.

Practical validation checklist:

- Confirm the volume appears in `lsblk`.
- Confirm the filesystem type with `blkid` or `file -s`.
- Mount read-only first if you want a safer inspection.
- Check application files, logs, and ownership.
- Run filesystem-specific repair tools only when needed and after careful assessment.

Example read-only mount:

```bash
sudo mount -o ro /dev/nvme1n1p1 /mnt/recovered
```

**Root volume recovery**

Snapshots are commonly used to recover a failed boot volume. If the instance will not boot, you can restore the root volume by creating a new volume from the snapshot and replacing the old root volume.

A common recovery pattern is:

1. Stop the affected instance if possible.
2. Create a snapshot of the root volume if you still need the current state.
3. Create a new volume from a known-good snapshot.
4. Detach the broken root volume.
5. Attach the restored volume as the root device.
6. Start the instance and verify boot.

If the instance is already unbootable, you may need to detach the root volume and attach it to a helper instance for offline inspection and repair. This is often used to fix bootloader issues, filesystem corruption, or configuration mistakes.

Another common approach is to create an Amazon Machine Image (<abbr title="Amazon Machine Image">AMI</abbr>) from a snapshot-backed root volume when you want a reusable bootable template. An AMI is not the same thing as a snapshot, but snapshots are a key building block for AMI-backed recovery and cloning.

**Cross-account and cross-Region copy**

You can copy snapshots to another Region, and you can share snapshots with another AWS account. These are different operations with different implications.

For unencrypted snapshots:

- You can share the snapshot with another account by modifying snapshot permissions.
- The receiving account can create a volume from the shared snapshot.
- The snapshot remains in the source account, and the recipient gets access to use it, not ownership.

For encrypted snapshots:

- Sharing is more constrained.
- The snapshot must use a customer managed AWS Key Management Service (<abbr title="AWS Key Management Service">KMS</abbr>) key.
- The KMS key policy or grants must allow the target account to use the key.
- The receiving account must also have permission to create volumes from the snapshot.
- You cannot treat encrypted snapshot sharing like unencrypted sharing; KMS permissions are part of the workflow.

Copying a snapshot is also the normal way to re-encrypt it with a different customer managed KMS key. Re-encryption happens during the copy operation when you specify a destination KMS key; it is not automatic.

CLI example for copying and re-encrypting:

```bash
aws ec2 copy-snapshot \
  --source-region us-east-1 \
  --source-snapshot-id snap-0123456789abcdef0 \
  --encrypted \
  --kms-key-id arn:aws:kms:us-east-1:123456789012:key/abcd1234-5678-90ef-abcd-1234567890ab \
  --description "Copied and re-encrypted snapshot"
```

**Snapshot sharing permissions**

To share an unencrypted snapshot, you typically modify the snapshot’s create volume permissions.

```bash
aws ec2 modify-snapshot-attribute \
  --snapshot-id snap-0123456789abcdef0 \
  --attribute createVolumePermission \
  --operation-type add \
  --user-ids 123456789012
```

Important constraints:

- Sharing does not copy the snapshot into the other account automatically.
- The recipient must create their own volume from the shared snapshot.
- Encrypted snapshots require KMS access in addition to snapshot sharing.
- If the source snapshot uses the AWS managed key for EBS encryption, it is not shareable in the same way as a snapshot encrypted with a customer managed key.

**Snapshot lifecycle and incremental behavior**

Because snapshots are incremental, deleting one snapshot does not necessarily delete all data associated with that point in time immediately. AWS retains only the blocks needed by the remaining snapshots in the chain. If you delete an older snapshot, AWS keeps any blocks that are still referenced by newer snapshots.

This has practical implications:

- Deleting a snapshot may not reduce storage as much as expected.
- The newest snapshot in a chain may depend on blocks first referenced by older snapshots.
- You should use lifecycle policies and retention planning rather than manual deletion alone.

Operationally, this means you can safely delete snapshots according to retention policy, but you should understand that the storage footprint is shared across the chain.

**Volume modification after restore**

After restoring a volume, you can modify certain attributes if needed. For supported volume types, you can change:

- Volume size
- Volume type
- IOPS for supported types
- Throughput for supported types

You cannot change the Availability Zone of an existing volume. If you need a different AZ, create a new volume in that AZ from the snapshot.

Example:

```bash
aws ec2 modify-volume \
  --volume-id vol-0abcdef1234567890 \
  --volume-type gp3 \
  --iops 6000 \
  --throughput 250
```

Use this when you need to tune performance after restore, but remember that the instance and workload must support the new settings.

**Practical recovery patterns**

EBS snapshots fit several common operational patterns:

- **Pre-change backup**: create a snapshot before patching, upgrades, or schema changes.
- **Clone for testing**: restore a copy into a non-production instance for validation.
- **Disaster recovery**: keep periodic snapshots for rollback and rebuild.
- **Forensic analysis**: restore a snapshot to inspect data without altering the original volume.
- **Root volume replacement**: recover a broken boot disk by creating a new volume from a known-good snapshot.

A useful mental model is: snapshot first, restore second, attach third, validate fourth.

**Exam-focused distinctions**

- A snapshot is a backup of an **EBS volume**, not of the whole instance.
- The source volume can stay online during snapshot creation.
- Snapshots are **regional**, while volumes live in a specific AZ.
- You restore by creating a new volume, not by editing the snapshot in place.
- Unencrypted snapshots are easier to share than encrypted ones.
- Encrypted snapshot sharing depends on **KMS** permissions and key policy.
- Copying a snapshot is the way to move it across Regions or re-encrypt it with a different key.
- Root volume recovery often involves replacing the boot volume or creating an AMI for repeatable recovery.

**Quick reference workflow**

1. Create a snapshot from the EBS volume attached to EC2.
2. Wait for the snapshot to complete.
3. Create a new volume from the snapshot in the correct Region and AZ.
4. Attach the volume to the target instance.
5. Discover the actual device name in the guest OS.
6. Mount and validate the filesystem.
7. Replace the old volume in the application or instance workflow if needed.

**Troubleshooting tips**

- If the volume does not appear, check the instance’s AZ and attachment state.
- If the device name is unexpected, inspect `lsblk` and `/dev/disk/by-id/`.
- If the filesystem will not mount, verify the partition table and filesystem type first.
- If a shared encrypted snapshot is inaccessible, review both snapshot permissions and KMS key policy.
- If restore performance is slow, consider whether the volume type, size, or initialization state needs tuning.
