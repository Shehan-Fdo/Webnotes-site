## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**What tag-based backup selection does**

AWS Backup can select resources for a backup plan by matching **resource tags** instead of listing each resource ARN manually. This is useful when you want backup coverage to follow application metadata such as `Environment=Prod`, `App=Payments`, or `Backup=Daily`.

Tag-based selection is especially valuable when resources are created and deleted frequently. Instead of updating the backup plan every time a new resource appears, you tag the resource correctly and AWS Backup includes it in the next evaluation cycle.

- **Resource tags** are key-value metadata attached to AWS resources.
- **Selection** is the rule set AWS Backup uses to decide which resources a backup plan protects.
- **Dynamic coverage** means newly tagged resources can be included without editing the selection each time.

**Why this matters for the exam**

The exam often tests whether you understand the difference between:
- selecting resources by **ARN**
- selecting by **resource type**
- selecting by **tags**

Tag-based selection is the most flexible option when you want policy-driven coverage across many resources that share a common classification.

**How AWS Backup evaluates tags**

AWS Backup evaluates the selection against the tags on supported resources. The selection is not a live inventory dashboard; it is a rule that AWS Backup uses to determine which resources are in scope for backup jobs.

Important exam points:
- A resource must have the expected tag key and value to match the selection.
- If you change a tag, the resource can move into or out of the selection on the next evaluation.
- If you create a new resource with the right tags, it can be picked up automatically.
- If you remove or change the tag, the resource may stop being protected by that selection.

> Tag-based selection is not the same as a backup vault filter or a restore filter. It is a way to define which resources a backup plan protects.

**Selection model and API shape**

AWS Backup selection configuration is defined in a backup selection object. In the AWS CLI and API, you typically provide:
- `SelectionName`
- `IamRoleArn`
- one or more resource matching fields such as `Resources`, `NotResources`, `ListOfTags`, or other supported selection criteria depending on the selection type

For tag-based selection, the important idea is that the selection is expressed as tag criteria, not arbitrary boolean logic over custom condition keys.

A simplified example looks like this:

```json
{
  "SelectionName": "prod-tag-selection",
  "IamRoleArn": "arn:aws:iam::123456789012:role/AWSBackupDefaultServiceRole",
  "ListOfTags": [
    {
      "ConditionType": "STRINGEQUALS",
      "ConditionKey": "Environment",
      "ConditionValue": "Prod"
    }
  ]
}
```

In practice, the exact CLI shape can vary by command version and selection mode, but the exam-relevant concept is consistent: you define a backup selection by matching tags on resources, and AWS Backup uses that selection to include eligible resources in the plan.

**Common tag-selection patterns**

Use tag-based selection when you want to group resources by application, environment, or ownership.

- `Environment=Prod` for production-only backups
- `Backup=Required` for resources that must always be protected
- `App=Billing` for application-scoped backup coverage
- `Owner=TeamA` for delegated operational ownership

A common enterprise pattern is to require a standard backup tag on all protected resources and enforce it through provisioning controls.

**CLI example: create a tag-based backup selection**

The following example shows the general workflow for creating a selection that targets production resources.

```bash
aws backup create-backup-selection \
  --backup-plan-id 1234abcd-56ef-78gh-90ij-klmnopqrstuv \
  --backup-selection '{
    "SelectionName": "prod-selection",
    "IamRoleArn": "arn:aws:iam::123456789012:role/AWSBackupDefaultServiceRole",
    "ListOfTags": [
      {
        "ConditionType": "STRINGEQUALS",
        "ConditionKey": "Environment",
        "ConditionValue": "Prod"
      }
    ]
  }'
```

Key points:
- `IamRoleArn` must allow AWS Backup to access the target resources.
- The tag key and value must match exactly unless the selection type explicitly supports a different comparison.
- The selection is attached to a specific backup plan.

**Failure scenario and fix**

A common failure is when a resource is tagged, but the backup job still does not include it.

Example situation:
- You tag an EC2 instance with `Environment=Production`
- Your selection expects `Environment=Prod`
- The instance is not selected

Fix:
1. Check the exact tag key and value on the resource.
2. Compare them with the selection criteria.
3. Correct either the resource tag or the selection rule.
4. Confirm the IAM role can access the resource type.

Another failure case is using the wrong role:
- The selection is correct
- The IAM role lacks permissions for the resource type
- AWS Backup cannot back up the resource even though it matches the tag

**Troubleshooting checklist**

- Verify the resource supports AWS Backup.
- Verify the tag key spelling and case.
- Verify the tag value spelling and case.
- Verify the backup plan is associated with the selection.
- Verify the IAM role trusts AWS Backup and has the required permissions.
- Verify the resource is in a supported Region and account scope.

**Tag-based selection vs other selection methods**

| Method | Best for | Strengths | Limitations |
|---|---|---|---|
| Tag-based selection | Groups of resources with shared metadata | Flexible, scalable, policy-driven | Depends on consistent tagging |
| Resource ARN selection | Specific known resources | Precise and explicit | Hard to maintain at scale |
| Resource type selection | Broad coverage by service type | Simple for standard coverage | Less granular than tags |

Use tag-based selection when you want backup coverage to follow business metadata rather than a static list of resource identifiers.

**How tags change over time**

Tag-based backup selection is dynamic. That means the set of protected resources can change as tags change.

- New resource with matching tags: can be included automatically
- Existing resource with tag removed: can fall out of scope
- Existing resource retagged for a different app or environment: can move between backup groups

This behavior is useful for automation, but it also creates risk if your tagging standards are inconsistent. A missing tag can create a backup gap without any change to the backup plan itself.

**Practical tagging examples**

For EC2, you can tag an instance like this:

```bash
aws ec2 create-tags \
  --resources i-0123456789abcdef0 \
  --tags Key=Environment,Value=Prod Key=Backup,Value=Required
```

For RDS, tag the specific database resource you intend to protect. Use the correct ARN for the DB instance or cluster, depending on the deployment model.

```bash
aws rds add-tags-to-resource \
  --resource-name arn:aws:rds:us-east-1:123456789012:db:mydbinstance \
  --tags Key=Environment,Value=Prod Key=Backup,Value=Required
```

For Amazon EFS, you can tag the file system:

```bash
aws efs tag-resource \
  --resource-id fs-12345678 \
  --tags Key=Environment,Value=Prod Key=Backup,Value=Required
```

**Operational best practices**

- Standardize tag keys across teams.
- Use a small, controlled set of values.
- Enforce tags at provisioning time with IaC or service controls.
- Audit for untagged resources regularly.
- Use backup tags that clearly express intent, such as `Backup=Required`.

A strong pattern is to combine tagging with automation:
- Infrastructure as Code creates the resource
- The template applies the required tags
- AWS Backup selection picks it up automatically

**Multi-account and AWS Organizations**

In larger environments, tag-based backup selection is often managed centrally.

AWS Organizations and AWS Backup policies can help you:
- enforce backup coverage across accounts
- standardize backup behavior
- reduce the chance that a workload is deployed without protection

This is exam-relevant because centralized governance can ensure that tag-based backup rules are applied consistently across many accounts, especially when using delegated administration and organization-wide backup policies.

**Example architecture**

A common design looks like this:

- Application teams deploy resources in member accounts
- Resources are tagged with `Environment`, `App`, and `Backup`
- A central backup administrator manages backup policies
- AWS Backup uses tag-based selection to include matching resources
- Backup copies can be sent to another Region or account for resilience

**Mini case study**

A team launches a new production EC2 instance for a customer portal. The instance is created from an automation pipeline, but the pipeline forgets to apply `Backup=Required`.

Result:
- The instance runs normally
- It is not included in the tag-based backup selection
- No backup job protects it until the tag is fixed

Resolution:
1. Add the missing tag.
2. Confirm the selection criteria match the tag exactly.
3. Verify the backup plan runs successfully on the next cycle.
4. Update the provisioning pipeline so the tag is always applied.

This is a classic exam scenario because the infrastructure is healthy, but the backup coverage is incomplete due to metadata drift.

**Exam reminders**

- Tag-based selection is about **metadata-driven inclusion**.
- Exact tag matching is the key concept unless the documented selection mode says otherwise.
- A resource can become protected or unprotected when tags change.
- IAM permissions still matter even when the tag matches.
- Central governance with AWS Organizations can enforce consistent backup coverage.

**Quick memory aid**

- **Tags define intent**
- **Selection defines scope**
- **IAM defines access**
- **Backup plan defines schedule and retention**
