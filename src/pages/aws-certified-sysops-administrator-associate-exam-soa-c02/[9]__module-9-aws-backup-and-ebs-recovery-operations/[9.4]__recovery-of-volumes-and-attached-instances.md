## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Overview**

Amazon Elastic Block Store (<abbr title="Elastic Block Store">EBS</abbr>) volumes are durable block storage devices for Amazon EC2 instances, but they are still a common recovery point in operations. You typically recover from volume corruption, accidental deletion, filesystem damage, boot failure, or application-level data loss by using snapshots, replacement volumes, or instance rebuild workflows.

The key exam idea is to choose the recovery method that preserves the right data with the least downtime. In practice, you often combine multiple tools: snapshots for point-in-time recovery, detached replacement volumes for data repair, and instance replacement when the root volume or operating system is compromised.

**Core recovery concepts**

- **EBS volume** recovery focuses on the data layer.
- **Instance** recovery focuses on the compute layer and attached storage state.
- **Snapshots** provide point-in-time backups of EBS volumes.
- **AMI** (<abbr title="Amazon Machine Image">AMI</abbr>) rebuilds are useful when the instance itself is the problem.
- **Rescue instances** help you inspect or repair a broken volume offline.

**EBS snapshot fundamentals**

An EBS snapshot is a point-in-time backup of an EBS volume stored in Amazon S3-managed infrastructure. You use snapshots to create new volumes, move data across Availability Zones, and copy backups across Regions. Snapshots are incremental after the first full snapshot, so subsequent snapshots store only changed blocks.

Important operational properties:

- A snapshot is tied to a source volume at the time it is created, but it can be copied to other Regions and shared across accounts if permissions allow.
- You can restore a snapshot into a new volume in the same Region or a different Region after copying it.
- A restored volume must be created in an Availability Zone, because EBS volumes are AZ-scoped.
- The new volume size must be equal to or larger than the snapshot’s original volume size.
- You can restore data from a snapshot even if the original volume is lost.

> A snapshot captures crash-consistent state by default. For application-consistent recovery, you usually quiesce the filesystem or application before taking the snapshot.

**Recovery decision guide**

| Situation | Best option | Why |
|---|---|---|
| Data volume is corrupted but instance is healthy | Detach and replace the volume | Preserves instance and repairs storage only |
| Root volume is damaged and instance will not boot | Replace root volume or launch a new instance from AMI | Faster than deep repair in many cases |
| Need to recover a single file or inspect logs offline | Attach the volume to a rescue instance | Lets you mount and inspect data safely |
| Need to restore to a previous point in time | Create a new volume from a snapshot | Simple and reliable rollback path |
| Need to rebuild the whole server consistently | Launch from AMI or automation | Best for immutable or repeatable infrastructure |

**Snapshot-based recovery workflow**

The most common recovery pattern is to create a new volume from a snapshot, attach it to an instance, and then validate the filesystem and application data. This is safer than trying to repair a live corrupted volume.

Typical workflow:

1. Identify the correct snapshot and confirm the source volume size.
2. Create a new EBS volume from the snapshot in the same Availability Zone as the target instance.
3. Detach the damaged volume if you are replacing it.
4. Attach the restored volume to the instance.
5. Mount the filesystem and validate data integrity.
6. Update application configuration if the device name or mount point changes.

Example AWS CLI workflow:

```bash
aws ec2 create-volume \
  --availability-zone us-east-1a \
  --snapshot-id snap-0123456789abcdef0 \
  --volume-type gp3 \
  --tag-specifications 'ResourceType=volume,Tags=[{Key=Name,Value=recovered-data-volume}]'
```

After the volume becomes `available`, attach it:

```bash
aws ec2 attach-volume \
  --volume-id vol-0123456789abcdef0 \
  --instance-id i-0123456789abcdef0 \
  --device /dev/sdf
```

On modern Nitro-based instances, the device may appear as an NVMe device such as `/dev/nvme1n1` rather than the exact Linux device name you specified. Always verify with `lsblk`.

```bash
lsblk -f
sudo blkid
```

**Rescue instance technique**

A rescue instance is a temporary EC2 instance used to inspect or repair a broken volume offline. This is especially useful when the root filesystem will not boot, when `fsck` is needed, or when you need to recover configuration files before rebuilding.

Use this approach when:

- The instance fails during boot.
- The root filesystem is damaged.
- You need to edit boot configuration or remove a bad package.
- You want to copy logs or application data before replacement.

Safe recovery pattern:

1. Stop the affected instance.
2. Detach the broken volume.
3. Attach it to a rescue instance as a secondary volume.
4. Identify the device and partitions with `lsblk`.
5. Mount the correct partition read-only if possible.
6. Copy out data or repair the filesystem.
7. Unmount, detach, and reattach to the original instance or replace it.

Example inspection commands:

```bash
lsblk -f
sudo file -s /dev/nvme1n1
sudo mkdir -p /mnt/rescue
sudo mount -o ro /dev/nvme1n1p1 /mnt/rescue
sudo cp -a /mnt/rescue/etc /tmp/recovered-etc
```

If the volume has no partition table, the device may mount directly without a `p1` suffix. If the filesystem is on the whole disk, use the disk device itself.

For logs, do not assume a single distro-specific path. Check the system’s logging model:

```bash
sudo ls /mnt/rescue/var/log
sudo journalctl --directory=/mnt/rescue/var/log/journal
```

**Root volume recovery**

Root volume recovery is different from data volume recovery because the instance boot process depends on the root disk, bootloader, and kernel initialization files. If the root volume is corrupted, you often recover by replacing the root volume or launching a new instance from a known-good AMI.

Key boot components to understand:

- **Bootloader**: the software that starts the operating system, such as GRUB on many Linux systems.
- **initramfs**: an initial RAM filesystem used during early boot to load drivers and mount the real root filesystem.
- **Kernel**: the core operating system component that must load successfully before user space starts.

Common root-volume recovery options:

- Restore the root volume from a snapshot and reattach it.
- Launch a new instance from the latest AMI and attach the old volume for data extraction.
- Use a rescue instance to repair boot configuration or filesystem damage.
- Replace the instance entirely if the environment is immutable or managed by automation.

> If the root volume is encrypted with AWS Key Management Service (<abbr title="Key Management Service">KMS</abbr>), the restored or copied volume must be created with permissions that allow decryption and re-encryption.

**Instance recovery versus volume recovery**

| Method | What it restores | When to use | Limitations |
|---|---|---|---|
| Snapshot restore | EBS data at a point in time | Volume corruption, rollback, file recovery | Does not restore instance state by itself |
| AMI rebuild | Instance configuration plus root volume template | OS rebuild, repeatable deployments | May not include latest data on separate volumes |
| Instance recovery | The same instance on new hardware | Underlying host failure for supported instances | Not a fix for application or disk corruption |
| Volume replacement | Only the affected EBS volume | Data disk failure or root disk swap | Requires careful attachment and validation |

Instance recovery is an EC2 feature that attempts to move an impaired instance to new hardware while preserving the instance ID, private IP, and metadata where supported. It is useful for underlying host issues, but it does not repair a corrupted filesystem or application data problem.

**End-to-end recovery scenario**

Imagine a production PostgreSQL database running on an EC2 instance with a separate EBS data volume. The application reports corruption after an unexpected shutdown.

A practical recovery path is:

1. Stop writes at the application layer if the instance is still reachable.
2. Take a snapshot of the affected volume if the filesystem is still mountable enough to do so.
3. Stop the instance and detach the data volume.
4. Attach the volume to a rescue instance.
5. Run filesystem checks and inspect database files.
6. If repair is not safe, create a new volume from the latest clean snapshot.
7. Attach the restored volume to a replacement instance or the original instance.
8. Validate database startup and application connectivity.

This workflow minimizes the chance of making corruption worse. If the database is critical, you may prefer restoring to a new instance and validating there before cutover.

**Filesystem and volume validation**

After attaching a recovered volume, verify the filesystem before mounting it read-write. This helps you avoid mounting a damaged filesystem and causing further corruption.

Useful commands:

```bash
lsblk -f
sudo blkid
sudo fsck -n /dev/nvme1n1p1
sudo mount -o ro /dev/nvme1n1p1 /mnt/recovery
```

Notes:

- Use `fsck -n` for a non-destructive check first.
- Only run repair mode after you understand the risk and have a backup or snapshot.
- Confirm the correct device name before running repair commands.
- If the volume is encrypted, ensure the instance role or operator credentials can access the KMS key.

**Cross-Region and cross-account recovery**

Snapshots are often used for disaster recovery because they can be copied to another Region and shared with another account. This supports regional failover and backup isolation.

Typical uses:

- **Cross-Region DR**: copy snapshots to a secondary Region for regional outage recovery.
- **Cross-account backup**: share snapshots with a backup account for separation of duties.
- **Long-term retention**: keep snapshots in a backup Region even if the source volume is deleted.

Example copy command:

```bash
aws ec2 copy-snapshot \
  --source-region us-east-1 \
  --source-snapshot-id snap-0123456789abcdef0 \
  --description "DR copy of production data volume" \
  --region us-west-2
```

If the snapshot is encrypted, the destination Region must have access to the appropriate KMS key strategy, and you may need to specify a destination KMS key when creating the copy.

**Practical CLI checks**

Before recovery, inspect the current state of volumes and attachments:

```bash
aws ec2 describe-volumes --volume-ids vol-0123456789abcdef0
aws ec2 describe-snapshots --snapshot-ids snap-0123456789abcdef0
aws ec2 describe-instances --instance-ids i-0123456789abcdef0
```

Useful fields to review:

- Volume state: `available`, `in-use`, or `deleting`
- Attachment state: `attached`, `detaching`, or `busy`
- Snapshot progress: ensure the snapshot is complete before restoring
- Instance state: `running`, `stopped`, or `terminated`

**Operational best practices**

- Take regular snapshots and test restores, not just backups.
- Tag snapshots and volumes with application, environment, and retention metadata.
- Use automation to create replacement instances from AMIs and infrastructure as code.
- Keep rescue-instance procedures documented and practiced.
- Prefer restoring to a new resource rather than repairing in place when data integrity is uncertain.
- Verify encryption, IAM permissions, and KMS access before a disaster occurs.

**Common exam traps**

- A snapshot is not the same as an AMI.
- An EBS volume cannot be attached to multiple instances at the same time unless it is a supported multi-attach configuration and the filesystem/application supports it.
- A restored volume must be created in an Availability Zone.
- Cross-Region snapshot copy uses `aws ec2 copy-snapshot` with `--source-region` and `--region`, not a `--destination-region` parameter.
- Device names on Nitro instances may not match the name you specify at attach time.
- Instance recovery does not fix application-level corruption.

**Quick recovery checklist**

- Confirm whether the problem is with the instance, the root volume, or a data volume.
- Choose snapshot restore, rescue-instance repair, instance recovery, or AMI rebuild.
- Verify encryption and permissions.
- Restore to a new volume when possible.
- Mount read-only first if you are unsure about integrity.
- Validate application startup before returning traffic.
