## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**AWS Backup overview**

AWS Backup is a centralized backup service that helps you define backup policies, assign resources, store recovery points, and restore data across supported AWS services. It is useful when you want consistent backup governance instead of managing service-specific backup tooling for every workload.

At a high level, AWS Backup separates the backup lifecycle into:
- **Backup plans** for scheduling and retention
- **Backup selections** for choosing protected resources
- **Backup vaults** for storing recovery points
- **Restore jobs** for recovery operations
- **Copy jobs** for moving backups across Regions or accounts

AWS Backup supports many AWS services, but feature parity is not identical across all of them. For exam purposes, always think in terms of:
- Which services are supported
- Which services support continuous backups or point-in-time recovery
- Which services support cold storage
- Which services require service-specific restore metadata

**Backup plans**

A backup plan defines *when* backups run and *how long* they are retained. A plan can contain one or more rules, and each rule can specify:
- Backup frequency and start window
- Retention period
- Lifecycle transitions to cold storage where supported
- Copy actions to other vaults, Regions, or accounts
- Optional continuous backup for supported services

A backup plan is not the same as a backup selection. The plan defines policy; the selection defines scope.

Key exam points:
- A plan can have multiple rules, and each rule is evaluated independently for the resources it applies to.
- A resource can match more than one rule, which can result in multiple recovery points being created according to each rule.
- Continuous backup is a service capability, not a universal plan setting for every resource type.
- Default backup plans may exist in an account, but you should not assume they protect anything until resources are explicitly assigned or organization-level policies apply.

**Plan versions and management**

When you update a backup plan, AWS Backup creates a new version of the plan. This matters because:
- Existing recovery points remain governed by the retention and lifecycle settings that applied when they were created.
- New assignments and future backups use the latest plan version.
- Versioning helps you track policy changes without rewriting history.

In the console, you typically:
- Create a plan
- Add one or more rules
- Assign resources to the plan
- Review the generated plan version and rule details

In the API or CLI, the same concepts appear as plan JSON, rule objects, and selection objects.

Example plan skeleton:
```json
{
  "BackupPlanName": "daily-weekly-plan",
  "Rules": [
    {
      "RuleName": "daily",
      "TargetBackupVaultName": "primary-vault",
      "ScheduleExpression": "cron(0 5 ? * * *)",
      "StartWindowMinutes": 60,
      "CompletionWindowMinutes": 180,
      "Lifecycle": {
        "MoveToColdStorageAfterDays": 30,
        "DeleteAfterDays": 365
      }
    }
  ]
}
```

**Backup selections and scope**

Backup selections define which resources are protected by a plan. You can target resources using:
- Resource ARNs
- Tags
- IAM role permissions that allow AWS Backup to access the resources

Selections are important because a plan without a selection does nothing.

Common selection patterns:
- Tag-based selection for broad coverage, such as `Backup=Daily`
- Explicit ARN selection for a small set of critical resources
- Organization-wide governance using backup policies where supported

Practical considerations:
- Tag-based selection is easier to scale, but it depends on consistent tagging.
- Explicit selection is precise, but it is harder to maintain at scale.
- In AWS Organizations, backup policies can centrally apply backup intent across accounts, reducing per-account drift.
- Cross-account protection still depends on the right IAM, AWS Organizations, and service permissions.

Example selection using tags:
```json
{
  "SelectionName": "prod-tag-selection",
  "IamRoleArn": "arn:aws:iam::123456789012:role/AWSBackupDefaultServiceRole",
  "ListOfTags": [
    {
      "ConditionType": "STRINGEQUALS",
      "ConditionKey": "Backup",
      "ConditionValue": "Daily"
    }
  ]
}
```

**Backup vaults**

A backup vault is the logical container that stores recovery points. Vaults help you separate backups by environment, account, application, or compliance boundary.

Use vaults to:
- Organize recovery points
- Apply access control
- Enable notifications
- Enforce immutability with vault lock
- Support copy workflows between vaults

A vault is not just storage. It is also a policy boundary.

**Vault access policy**

A vault access policy controls who can interact with the vault. It is similar in spirit to an S3 bucket policy, but it applies to backup vault actions.

Use vault policies to:
- Allow cross-account copy into a destination vault
- Restrict who can delete or read recovery points
- Limit access to specific principals or conditions

Be careful with cross-account and cross-Region copy. A working workflow often requires:
- Source-side permissions to read and copy the recovery point
- Destination-side permissions to accept the copy
- KMS permissions on both sides if the vaults use customer managed keys
- Organization or account trust where applicable

Example vault access policy pattern:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowBackupServiceCopyIntoVault",
      "Effect": "Allow",
      "Principal": {
        "Service": "backup.amazonaws.com"
      },
      "Action": [
        "backup:CopyIntoBackupVault"
      ],
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "aws:SourceAccount": "123456789012"
        }
      }
    }
  ]
}
```

> A single statement is often not enough for real cross-account or cross-Region copy. You may also need IAM permissions, destination vault policy, and KMS key policy updates.

**Vault notifications**

Vault notifications let you send events to Amazon EventBridge or Amazon SNS depending on the integration pattern you configure. Use notifications to track:
- Backup job success or failure
- Restore job success or failure
- Copy job status changes
- Vault-related operational events where supported

A practical pattern is to route AWS Backup events to EventBridge and then fan out to SNS, Lambda, or incident tooling.

Example EventBridge rule pattern for backup job state changes:
```json
{
  "source": ["aws.backup"],
  "detail-type": ["Backup Job State Change"],
  "detail": {
    "state": ["COMPLETED", "FAILED", "ABORTED"]
  }
}
```

Event names and detail fields can vary by job type, so validate the exact event schema for backup, copy, and restore workflows.

**KMS integration**

AWS Backup encrypts recovery points using AWS Key Management Service (KMS) keys. Vaults can use:
- AWS managed keys
- Customer managed keys

Important behaviors:
- If you rotate a KMS key, existing recovery points remain decryptable as long as the old key material remains available through KMS.
- If you replace or disable a key, restores can fail for recovery points encrypted under that key.
- Cross-account copy with encryption usually requires destination-side KMS permissions and sometimes source-side decrypt permissions.

Operationally, treat the KMS key as part of the backup dependency chain.

**Vault lock**

Vault lock helps you enforce write-once-read-many-style retention controls for backup vaults. It is a major exam topic because it supports compliance and immutability requirements.

Vault lock has two modes:
- **Governance mode**: administrators with sufficient permissions can still make some changes
- **Compliance mode**: once locked, retention settings become immutable within the configured constraints

Key distinctions:
- In compliance mode, after the lock is enforced, you cannot shorten retention or remove the lock.
- A cooling-off period can exist before the compliance lock becomes permanent, giving you a short window to cancel or adjust the configuration.
- Minimum and maximum retention settings help ensure backups cannot be kept too briefly or too long.

Typical setup workflow:
1. Create or choose a backup vault.
2. Configure vault lock with minimum retention, maximum retention, and mode.
3. If using compliance mode, review the cooling-off period carefully.
4. Enforce the lock after validation.
5. Confirm that future recovery points inherit the retention constraints.

Example CLI pattern:
```bash
aws backup put-backup-vault-lock-configuration \
  --backup-vault-name prod-vault \
  --min-retention-days 30 \
  --max-retention-days 3650 \
  --changeable-for-days 3
```

> In compliance mode, once the lock is enforced, you should assume retention is immutable for the locked vault.

**Backup plan lifecycle and cold storage**

AWS Backup can transition eligible recovery points to cold storage after a configured number of days. This is useful when you want lower-cost long-term retention.

Use cold storage when:
- You need long retention for compliance
- Restore frequency is low
- The supported service and backup type allow lifecycle transitions

Be aware:
- Not every service or backup type supports cold storage.
- Cold storage usually increases restore time.
- Lifecycle settings are part of the backup rule, not a separate vault feature.

Example lifecycle rule:
```json
{
  "Lifecycle": {
    "MoveToColdStorageAfterDays": 30,
    "DeleteAfterDays": 365
  }
}
```

**Continuous backup and point-in-time recovery**

Continuous backup provides point-in-time recovery for supported services. This is not a universal AWS Backup feature for every resource.

Use continuous backup when:
- You need recovery to a specific point in time
- The service supports it
- You want reduced recovery point loss compared with scheduled snapshots

Examples of service-specific behavior:
- Some services support continuous backups and point-in-time restore
- Others only support scheduled backups
- Restore metadata and recovery steps differ by service

For exam prep, remember that continuous backup is service-dependent, not a generic backup-plan toggle.

**Restore operations**

Restore jobs recover data from a recovery point into a target resource. The restore process is highly service-specific.

General restore flow:
1. Choose the recovery point.
2. Select the target resource or restore parameters.
3. Provide required metadata.
4. Start the restore job.
5. Monitor job status and validate the restored resource.

Common restore considerations:
- Restores may create a new resource rather than overwrite the original.
- Some services require subnet, security group, or instance metadata.
- Some services restore only within the same Region unless you copied the recovery point elsewhere.
- Cross-account restore usually requires the recovery point to exist in the destination account or a shared/copy-enabled workflow.

Example CLI restore pattern:
```bash
aws backup start-restore-job \
  --recovery-point-arn arn:aws:backup:us-east-1:123456789012:recovery-point:abcd1234 \
  --metadata file://restore-metadata.json \
  --iam-role-arn arn:aws:iam::123456789012:role/AWSBackupDefaultServiceRole
```

Service-specific metadata examples:
- **EBS**: restore metadata often includes volume size, availability zone, and encryption-related settings depending on the restore target.
- **RDS**: restore metadata can include DB instance class, subnet group, parameter group, and engine-specific options.
- **EFS**: restore behavior is tied to file system settings and may require target configuration details.
- **DynamoDB**: restore typically recreates a table and may require table-specific settings such as billing mode and indexes.

Because metadata keys vary by service, do not memorize a single universal restore JSON shape.

**Cross-Region and cross-account copy**

Copy jobs move recovery points to another vault in the same or a different Region, and optionally to another account. This is a common disaster recovery pattern.

Why you use copy:
- Regional isolation
- Account isolation
- Compliance separation
- Long-term archival in a dedicated vault

Typical topology:
- Source account, primary Region, production vault
- Destination account or secondary Region, DR vault
- Copy rule in the backup plan or manual copy job

Example copy rule concept:
```json
{
  "RuleName": "daily-copy-to-dr",
  "TargetBackupVaultName": "primary-vault",
  "ScheduleExpression": "cron(0 5 ? * * *)",
  "CopyActions": [
    {
      "DestinationBackupVaultArn": "arn:aws:backup:us-west-2:210987654321:backup-vault:dr-vault",
      "Lifecycle": {
        "DeleteAfterDays": 365
      }
    }
  ]
}
```

Common failure points:
- Missing destination vault policy
- Missing KMS permissions on source or destination keys
- Incorrect IAM role trust
- Unsupported resource type for the chosen copy path
- Region mismatch for a service that does not support the desired restore pattern

**Backup indexing and search**

AWS Backup can support indexing and search features for recovery points in some contexts. These features help you discover backups faster, especially in larger environments.

Use indexing when:
- You manage many recovery points
- You need faster discovery of restore candidates
- You want to search across backup metadata

For exam purposes:
- Treat indexing as an enhancement, not the core backup workflow
- Do not confuse indexing with the backup plan itself
- Verify whether the feature is available for the specific service and Region

**CLI examples**

Create a backup vault:
```bash
aws backup create-backup-vault \
  --backup-vault-name prod-vault
```

Create a backup plan:
```bash
aws backup create-backup-plan \
  --backup-plan file://backup-plan.json
```

Assign resources to a plan:
```bash
aws backup create-backup-selection \
  --backup-plan-id 1234abcd-56ef-78gh-90ij-klmnopqrstuv \
  --backup-selection file://selection.json
```

Start a backup job manually:
```bash
aws backup start-backup-job \
  --backup-vault-name prod-vault \
  --resource-arn arn:aws:ec2:us-east-1:123456789012:volume/vol-0123456789abcdef0 \
  --iam-role-arn arn:aws:iam::123456789012:role/AWSBackupDefaultServiceRole
```

List recovery points:
```bash
aws backup list-recovery-points-by-backup-vault \
  --backup-vault-name prod-vault
```

Start a restore job:
```bash
aws backup start-restore-job \
  --recovery-point-arn arn:aws:backup:us-east-1:123456789012:recovery-point:abcd1234 \
  --metadata file://restore-metadata.json \
  --iam-role-arn arn:aws:iam::123456789012:role/AWSBackupDefaultServiceRole
```

**Exam distinctions and troubleshooting**

Focus on the difference between policy, scope, and storage:
- **Plan**: defines schedule and retention
- **Selection**: defines which resources are protected
- **Vault**: stores recovery points and enforces access/immutability controls

Troubleshooting checklist:
- Confirm the resource is supported by AWS Backup
- Confirm the resource is actually selected by the plan
- Confirm the IAM role has permission to back up and restore the resource
- Confirm the vault policy allows the operation
- Confirm the KMS key policy allows encryption and decryption
- Confirm the restore metadata matches the service type
- Confirm the Region and account are correct for copy or restore

Common exam traps:
- Assuming one restore metadata format works for every service
- Assuming vault lock can always be removed later
- Assuming a backup plan alone protects resources without a selection
- Assuming cross-account copy works with only one policy statement
- Assuming continuous backup is available for every supported resource type

**Practical mental model**

Think of AWS Backup as a control plane with four layers:
- **Intent**: backup plan
- **Targeting**: backup selection
- **Storage**: backup vault
- **Recovery**: restore and copy jobs

If you can explain how those layers interact, you can answer most SOA-C02 questions on backup governance, retention, and recovery design.
