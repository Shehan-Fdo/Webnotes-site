## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**EventBridge rule fundamentals**

Amazon EventBridge is a managed event routing service that lets you react to operational changes without polling. Rules evaluate incoming events on an event bus and route matching events to one or more targets. For SysOps scenarios, you use rules to automate remediation, notifications, workflow triggers, and scheduled maintenance.

EventBridge rules are **regional** resources. A rule in one Region does not automatically process events in another Region, so you must create the rule in the same Region where the relevant event bus and targets live.

- **Default bus** receives AWS service events and many SaaS partner events automatically.
- **Custom bus** is for your own application events or centralized event routing patterns.
- **Partner bus** is used for supported SaaS integrations that publish into EventBridge.
- **Rules** match events by pattern or schedule and then invoke targets.

A rule is not the same as an event bus:
- The **bus** is the ingestion and distribution layer.
- The **rule** is the matching and routing logic.
- The **target** is the downstream service or resource that receives the event.

**Event bus types and when to use them**

The default event bus is the most common starting point for AWS service events such as EC2, EBS, Auto Scaling, IAM, and many others. AWS services publish to the default bus when they support EventBridge integration. You typically use the default bus for operational automation inside a single account and Region.

Custom buses are useful when you want to:
- Separate application events from AWS service events.
- Centralize events from multiple producers.
- Apply different rules, archives, and replay policies to different event streams.
- Forward events across accounts or organizational boundaries.

Partner buses are useful when a SaaS provider integrates directly with EventBridge. In exam scenarios, the key distinction is that the partner service publishes to a partner event bus, while your rules still perform the routing.

**Rule creation, update, and deletion**

You can manage rules in the console, CLI, CloudFormation, or Terraform. The exam often expects you to know the lifecycle operations, not just the matching logic.

Typical CLI workflow:
1. Create the rule.
2. Add one or more targets.
3. Grant target-specific permissions if required.
4. Test with sample events or real service activity.
5. Update the pattern, schedule, or targets as requirements change.
6. Delete the rule when it is no longer needed.

Example CLI commands:

```bash
aws events put-rule \
  --name ec2-state-change-stop-nonprod \
  --event-pattern '{
    "source": ["aws.ec2"],
    "detail-type": ["EC2 Instance State-change Notification"],
    "detail": {
      "state": ["running"]
    }
  }' \
  --state ENABLED
```

```bash
aws events put-targets \
  --rule ec2-state-change-stop-nonprod \
  --targets '[
    {
      "Id": "lambda1",
      "Arn": "arn:aws:lambda:us-east-1:123456789012:function:StopNonProdInstances"
    }
  ]'
```

```bash
aws events delete-rule \
  --name ec2-state-change-stop-nonprod
```

If you delete a rule, you should also remove unused targets and any related permissions or queue policies to avoid stale access paths.

**Event pattern matching**

EventBridge event patterns are JSON filters. They match fields in the event envelope and nested `detail` content. Patterns are exacting but flexible, and exam questions often test whether you know which operator to use.

Common operators and matching behavior:
- **Exact match**: match a specific string or number.
- **`prefix`**: match values that begin with a string.
- **`anything-but`**: match anything except listed values.
- **`exists`**: match when a field is present or absent.
- **`numeric`**: compare numeric values with operators such as `>`, `<`, `>=`, `<=`, or ranges.
- **`cidr`**: match IP addresses in a CIDR block.
- **`null`**: match fields explicitly set to null.

Example pattern using multiple operators:

```json
{
  "source": ["aws.ec2"],
  "detail-type": ["EC2 Instance State-change Notification"],
  "detail": {
    "state": [
      "stopped",
      "terminated"
    ],
    "instance-id": [
      { "prefix": "i-" }
    ]
  }
}
```

Example using `anything-but` and `numeric`:

```json
{
  "source": ["aws.s3"],
  "detail-type": ["Object Created"],
  "detail": {
    "object": {
      "size": [
        { "numeric": [">", 104857600] }
      ]
    },
    "bucket": {
      "name": [
        { "anything-but": ["test-bucket", "dev-bucket"] }
      ]
    }
  }
}
```

Example using `exists` and `null`:

```json
{
  "detail": {
    "errorCode": [
      { "exists": true }
    ],
    "responseElements": [
      { "null": true }
    ]
  }
}
```

Practical guidance:
- Keep patterns as narrow as possible to avoid noisy invocations.
- Match on stable fields such as `source`, `detail-type`, and key `detail` attributes.
- Test patterns with real sample events before relying on them in production.
- Remember that pattern complexity can make troubleshooting harder, so prefer readable filters over clever ones.

**Scheduled rules**

Scheduled rules trigger on a time-based schedule rather than on incoming event content. EventBridge rules support `rate()` and `cron()` expressions, and they are evaluated in **UTC**. Time zone support is a feature of EventBridge Scheduler, not EventBridge rules.

Use scheduled rules for:
- Nightly cleanup jobs.
- Periodic backups or snapshot orchestration.
- Health checks and drift detection.
- Batch jobs that must run on a fixed cadence.

Example `rate()` schedule:
```json
{
  "scheduleExpression": "rate(1 hour)"
}
```

Example `cron()` schedule for a maintenance window:
```json
{
  "scheduleExpression": "cron(0 3 ? * SUN *)"
}
```

A practical backup automation example:
- Run a rule every night at 03:00 UTC.
- Target a Lambda function that tags EBS snapshots.
- Use the function to create or verify snapshots, then notify an SNS topic on success or failure.

Example rule creation:

```bash
aws events put-rule \
  --name nightly-backup-orchestrator \
  --schedule-expression 'cron(0 3 * * ? *)' \
  --state ENABLED
```

**Targets and invocation permissions**

A rule can send matching events to multiple targets. Common targets include Lambda, SNS, SQS, Step Functions, ECS tasks, Kinesis Data Streams, API destinations, and some AWS service integrations.

Target delivery is not always automatic. Many targets require a resource-based policy or service-specific permission so EventBridge can invoke them.

Typical permission considerations:
- **Lambda**: add a resource-based permission allowing `events.amazonaws.com` to invoke the function.
- **SNS**: the topic policy must allow EventBridge to publish.
- **SQS**: the queue policy must allow `events.amazonaws.com` to send messages.
- **Step Functions**: the state machine needs a policy that allows EventBridge to start an execution.
- **ECS**: the target role must allow EventBridge to run tasks.
- **API destinations**: use a connection and IAM authorization where applicable.

Lambda permission example:

```bash
aws lambda add-permission \
  --function-name StopNonProdInstances \
  --statement-id EventBridgeInvoke \
  --action lambda:InvokeFunction \
  --principal events.amazonaws.com \
  --source-arn arn:aws:events:us-east-1:123456789012:rule/ec2-state-change-stop-nonprod
```

SQS queue policy example:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowEventBridgeSendMessage",
      "Effect": "Allow",
      "Principal": {
        "Service": "events.amazonaws.com"
      },
      "Action": "sqs:SendMessage",
      "Resource": "arn:aws:sqs:us-east-1:123456789012:ops-events"
    }
  ]
}
```

> A target can be configured correctly in the rule and still fail if the target resource policy does not allow EventBridge to invoke it.

**Input transformation and input paths**

EventBridge can reshape event payloads before sending them to a target. This is useful when the downstream service expects a smaller or differently structured message than the original event.

Use input transformation when you want to:
- Reduce payload size.
- Pass only relevant fields.
- Normalize event data for a Lambda function or API destination.
- Add static text or context to the message.

Common approaches:
- **Input path**: select a field from the event and pass it through.
- **Input transformer**: map event fields to template variables and build a custom JSON payload.
- **Constant input**: send a fixed message.

Example input transformer concept:
- Extract `instance-id`, `state`, and `time`.
- Send a compact JSON object to a Lambda function or SNS topic.

Example CLI target with input transformer:

```bash
aws events put-targets \
  --rule ec2-state-change-stop-nonprod \
  --targets '[
    {
      "Id": "notify1",
      "Arn": "arn:aws:sns:us-east-1:123456789012:ops-alerts",
      "InputTransformer": {
        "InputPathsMap": {
          "instance": "$.detail.instance-id",
          "state": "$.detail.state",
          "time": "$.time"
        },
        "InputTemplate": "{\"instance\":\"<instance>\",\"state\":\"<state>\",\"time\":\"<time>\"}"
      }
    }
  ]'
```

Input transformation is especially useful when you route operational events to human-readable notifications or to automation functions that only need a subset of the event.

**Cross-account routing**

Cross-account event routing is common in centralized operations models. A source account can send events to a custom event bus in a central account, where rules then fan out to security, operations, or compliance targets.

Authorization model:
- The **destination event bus policy** must allow the sending account or principal to put events.
- The sender uses its own identity to call `PutEvents`.
- You do not usually grant a separate permission on the destination bus to the sender beyond the bus policy that authorizes event ingestion.

Example destination bus policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowSourceAccountPutEvents",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:root"
      },
      "Action": "events:PutEvents",
      "Resource": "arn:aws:events:us-east-1:444455556666:event-bus/central-ops-bus"
    }
  ]
}
```

Operationally, this pattern is useful when you want:
- Centralized incident response.
- Shared audit pipelines.
- Separation of duties between application teams and platform teams.

**Archives and replay**

Archives let you retain events on an event bus for later analysis or recovery. Replay lets you reprocess archived events through rules after you fix a downstream issue or deploy a new consumer.

Key points:
- Archives apply to **event buses**, not individual rules.
- Replay can be scoped by **time window**.
- Replay can be limited to **specific rules** so you do not re-trigger every consumer.
- Replay is useful after a target outage, a bug in a Lambda function, or a missed automation window.

Example use case:
- You archive all production change events on a custom bus.
- A downstream remediation Lambda fails for two hours.
- You fix the function and replay only the affected time window to the remediation rule.

**Monitoring, metrics, and troubleshooting**

EventBridge publishes metrics that help you confirm whether rules and targets are working. In CloudWatch, watch for delivery and failure signals rather than assuming the rule is healthy because it exists.

Useful operational checks:
- Rule is **enabled**.
- Pattern matches the expected event shape.
- Target permissions are correct.
- Target service is healthy.
- Input transformation produces valid JSON.
- Dead-letter handling is configured where appropriate.

Common metrics and signals to review:
- **Invocations**: how many times EventBridge attempted to invoke a target.
- **FailedInvocations**: how many target deliveries failed.
- **MatchedEvents**: how many events matched the rule.
- **TriggeredRules**: how often the rule fired.

Troubleshooting workflow:
1. Confirm the event actually appears on the correct bus.
2. Verify the rule is in the same Region as the event bus.
3. Validate the event pattern against a sample event.
4. Check target permissions and resource policies.
5. Review CloudWatch metrics and logs for the target service.
6. If using a dead-letter queue, inspect failed messages there.

**Dead-letter queues and failure handling**

EventBridge can send failed target deliveries to a dead-letter queue when configured on the target. This captures delivery failures from EventBridge to the target, not failures that happen later inside the target after it has already been invoked.

Use a dead-letter queue when:
- You need to inspect undeliverable events.
- You want a fallback path for transient target failures.
- You need operational visibility into failed routing attempts.

Important distinction:
- A Lambda function can fail after EventBridge successfully invokes it.
- A target DLQ does not capture every possible application error inside the function.
- For application-level failures, use Lambda logs, retries, and your own error handling.

**Quotas and operational limits**

EventBridge has service quotas and practical limits that can affect design and exam answers. These limits can change, so verify current values in the AWS documentation when implementing production systems.

Common constraints to remember:
- EventBridge rules are regional.
- A rule can have multiple targets, but target count is limited.
- Event patterns have size and complexity limits.
- Event payloads have size limits.
- Large or overly broad patterns can increase noise and cost.
- Some targets have their own throughput or concurrency limits.

Operational implications:
- If you need many consumers, consider a bus-centric fan-out design.
- If you need long-term retention and reprocessing, use archives.
- If you need precise time-zone scheduling, use EventBridge Scheduler instead of rules.

**Practical examples**

Example 1: Stop non-production EC2 instances when they enter a running state.

```json
{
  "source": ["aws.ec2"],
  "detail-type": ["EC2 Instance State-change Notification"],
  "detail": {
    "state": ["running"]
  }
}
```

Flow:
- EC2 emits a state-change event.
- The rule matches `running`.
- A Lambda target tags or stops instances based on environment metadata.

Example 2: Notify operations when an S3 object larger than 100 MB is created.

```json
{
  "source": ["aws.s3"],
  "detail-type": ["Object Created"],
  "detail": {
    "object": {
      "size": [
        { "numeric": [">", 104857600] }
      ]
    }
  }
}
```

Flow:
- S3 emits an object-created event.
- The rule filters for large uploads.
- An SNS topic notifies the operations team.

Example 3: Nightly maintenance job.

```bash
aws events put-rule \
  --name nightly-maintenance \
  --schedule-expression 'cron(0 2 * * ? *)' \
  --state ENABLED
```

Then attach a target such as a Lambda function or Step Functions state machine that:
- Checks patch status.
- Creates snapshots.
- Sends a summary notification.

**Exam-oriented distinctions**

- **EventBridge rules vs EventBridge Scheduler**
  - Rules are best for event pattern matching and simple UTC-based schedules.
  - Scheduler is better for one-time or time-zone-aware scheduling.

- **Event bus vs rule**
  - The bus receives events.
  - The rule filters and routes them.

- **Rule vs target permission**
  - The rule defines what matches.
  - The target policy defines whether EventBridge can invoke the destination.

- **Archive vs DLQ**
  - Archive stores events for replay.
  - DLQ stores failed deliveries to a target.

- **Default bus vs custom bus**
  - Default bus is for AWS service events and common operational automation.
  - Custom bus is for application events and centralized routing.

**What to remember for the exam**

- EventBridge rules are **regional** and operate on an event bus.
- AWS service events commonly arrive on the **default bus**.
- Use event patterns for content-based routing and `rate()` or `cron()` for schedules.
- EventBridge rules use **UTC** schedules.
- Target invocation often requires a **resource policy** or service-specific permission.
- Use `anything-but`, `prefix`, `exists`, `numeric`, `cidr`, and `null` for advanced filtering.
- Archives and replay work at the **bus** level.
- Monitor **MatchedEvents**, **Invocations**, and **FailedInvocations** in CloudWatch.
- Input transformation helps you tailor payloads for downstream targets.
