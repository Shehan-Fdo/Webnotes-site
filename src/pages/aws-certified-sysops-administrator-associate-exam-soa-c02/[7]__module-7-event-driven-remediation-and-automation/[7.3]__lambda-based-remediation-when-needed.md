## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Lambda-based remediation: when and why you use it**

Lambda-based remediation is the pattern you use when a managed, native remediation path is not enough. In AWS operations, the usual first choice is a service-native fix such as AWS Config automatic remediation with an SSM Automation document. You move to Lambda when the workflow needs custom logic, conditional branching, cross-service coordination, or an action that AWS Config cannot express directly.

Typical reasons to choose Lambda include:

- **Custom logic** that depends on resource tags, account structure, or business rules.
- **Multi-step workflows** that must call several AWS APIs in sequence.
- **Unsupported actions** where no managed remediation document exists.
- **Cross-account or cross-region orchestration** that needs explicit control.
- **Human approval gates** or ticketing integration before making changes.

A good exam mindset is: use the simplest native remediation first, then add Lambda only when the remediation must be programmable.

**Native AWS Config remediation versus Lambda**

AWS Config remediation is often misunderstood. Native remediation does not directly invoke Lambda as the remediation target. Instead, AWS Config remediation commonly uses an <abbr title="Systems Manager">SSM</abbr> Automation document, which performs the fix. Lambda can still be part of the overall design, but usually as a custom event-driven workflow rather than the native Config remediation engine itself.

Key distinctions:

- **AWS Config automatic remediation**
  - Triggers when a rule evaluates a resource as noncompliant.
  - Uses a remediation configuration tied to a managed or custom SSM Automation document.
  - Best for standardized, repeatable fixes.
- **Lambda-based remediation**
  - Triggered by events from AWS Config, EventBridge, Security Hub, CloudWatch alarms, or custom application events.
  - Best for logic-heavy or service-spanning remediation.
  - Requires you to manage the function, permissions, retries, and idempotency.

> Native Config remediation is usually the right answer when the question says “automatic remediation” and the action can be expressed as an SSM Automation document. Use Lambda when the scenario explicitly needs custom code or unsupported branching.

**Common event sources for remediation**

You often build Lambda remediation around one of these event sources:

- **AWS Config**
  - Detects configuration drift or policy violations.
  - Can emit configuration change notifications and compliance state changes.
  - Useful when the remediation should follow a compliance rule.
- **Amazon EventBridge**
  - Routes events from AWS services and custom applications.
  - Good for decoupling detection from remediation.
  - Lets you filter and fan out to multiple targets.
- **AWS Security Hub**
  - Aggregates findings from AWS services and partner tools.
  - Useful for security-driven remediation workflows.
  - Findings often need workflow status updates after remediation.
- **Amazon CloudWatch alarms**
  - Good for metric-based remediation, such as restarting a service or scaling capacity.
- **Custom application events**
  - Useful when your app detects a condition and emits a remediation event.

**AWS Config remediation model in practice**

For exam purposes, remember the operational pieces of AWS Config remediation:

- A **Config rule** evaluates resources.
- A **remediation configuration** defines what to do when the rule is noncompliant.
- The remediation action is usually an **SSM Automation document**.
- The remediation can be **automatic** or manual.
- The remediation needs the right **parameters**, such as resource IDs, region, or role ARN.
- AWS Config uses a **service-linked role** or an explicitly supplied IAM role to perform remediation.

Common setup considerations:

- The remediation document must accept the parameters AWS Config passes.
- The role must allow Systems Manager automation and the target service actions.
- The target resource must be in a state where the fix is actually possible.
- Some remediations are not in-place and require replacement workflows.

Example of a remediation configuration conceptually:

```json
{
  "ConfigRuleName": "restricted-ssh",
  "TargetType": "SSM_DOCUMENT",
  "TargetId": "AWS-DisablePublicAccessForSecurityGroup",
  "Automatic": true,
  "Parameters": {
    "GroupId": {
      "ResourceValue": {
        "Value": "RESOURCE_ID"
      }
    }
  }
}
```

This is a conceptual shape, not a universal document name. In real deployments, you must use a valid SSM Automation document that matches the resource type and remediation goal.

**When Lambda is the better remediation target**

Use Lambda when the remediation needs logic that SSM Automation alone does not handle cleanly.

Examples:

- A security group rule must be removed only if the source CIDR is not on an allow list.
- An EBS volume must be replaced with an encrypted copy, which requires snapshotting, creating a new encrypted volume, and reattaching it.
- A public S3 bucket must be remediated only after checking whether the bucket is used by a known static website.
- A finding must be enriched with tags, account metadata, or CMDB data before action is taken.
- A remediation must open a ticket, notify Slack, and then apply the fix.

Lambda is also useful when you need to call APIs that are not supported by a managed remediation document or when you need to coordinate multiple services in one transaction-like workflow.

**End-to-end pattern: AWS Config -> EventBridge -> Lambda**

A common architecture is:

1. AWS Config detects noncompliance.
2. AWS Config publishes a compliance event.
3. EventBridge matches the event pattern.
4. EventBridge invokes a Lambda function.
5. Lambda evaluates context and performs the remediation.
6. Lambda logs the action and optionally updates a ticket or notification channel.

This pattern is useful when you want Config for detection but need custom code for the fix.

Example EventBridge rule pattern for AWS Config compliance changes:

```json
{
  "source": ["aws.config"],
  "detail-type": ["Config Rules Compliance Change"],
  "detail": {
    "messageType": ["ComplianceChangeNotification"],
    "newEvaluationResult": {
      "complianceType": ["NON_COMPLIANT"]
    }
  }
}
```

A Lambda target can then inspect the event and decide whether to act.

Example Lambda handler in Python:

```python
import json
import boto3

ec2 = boto3.client("ec2")

def lambda_handler(event, context):
    print(json.dumps(event))

    detail = event.get("detail", {})
    resource_id = detail.get("resourceId")
    resource_type = detail.get("resourceType")

    if resource_type == "AWS::EC2::SecurityGroup":
        ec2.revoke_security_group_ingress(
            GroupId=resource_id,
            IpPermissions=[
                {
                    "IpProtocol": "tcp",
                    "FromPort": 22,
                    "ToPort": 22,
                    "IpRanges": [{"CidrIp": "0.0.0.0/0"}]
                }
            ]
        )

    return {"status": "remediated", "resourceId": resource_id}
```

This example is intentionally simple. In production, you should validate the event, make the function idempotent, and handle partial failures.

**Security Hub remediation workflow**

Security Hub is a common source for Lambda-based remediation because findings often need enrichment and workflow updates.

Important points:

- Security Hub findings are structured records with fields such as product ARN, severity, resource details, and workflow status.
- A remediation workflow often updates the finding status after the fix.
- You may need to set the finding workflow to `RESOLVED`, `NOTIFIED`, or another appropriate state depending on your process.
- EventBridge is commonly used to route Security Hub findings to Lambda.

A typical EventBridge pattern for Security Hub findings:

```json
{
  "source": ["aws.securityhub"],
  "detail-type": ["Security Hub Findings - Imported"],
  "detail": {
    "findings": {
      "RecordState": ["ACTIVE"],
      "Workflow": {
        "Status": ["NEW"]
      }
    }
  }
}
```

After remediation, you can update the finding:

```bash
aws securityhub batch-update-findings \
  --finding-identifiers Id="finding-id",ProductArn="arn:aws:securityhub:us-east-1:123456789012:product/123456789012/default" \
  --workflow Status=RESOLVED
```

Use this carefully. Only mark a finding resolved when the underlying issue is actually fixed.

**Lambda failure handling and delivery semantics**

Remediation design must account for failure. Lambda can fail in several ways:

- **Timeout**
  - The function runs longer than its configured timeout.
  - Common when remediation waits on eventual consistency or long-running service operations.
- **Unhandled exception**
  - The code crashes before completing the fix.
- **Throttling**
  - The function or downstream service is rate-limited.
- **Permission failure**
  - The execution role lacks the required API permissions.
- **Idempotency conflict**
  - The same event is delivered more than once and the function repeats an action unsafely.

Invocation type matters:

- **Synchronous invocation**
  - The caller waits for the result.
  - Common when EventBridge invokes Lambda.
  - Retries and delivery behavior are handled by the invoking service.
- **Asynchronous invocation**
  - Lambda queues the event and processes it later.
  - Lambda retries failed executions automatically for asynchronous invokes.
  - You can configure destinations for success or failure.

EventBridge retry behavior is separate from Lambda async behavior:

- EventBridge retries delivery to the target for a limited period.
- If configured, EventBridge can send undeliverable events to an EventBridge DLQ.
- Lambda asynchronous invocation can use its own failure destination or DLQ-like handling through destinations.

Example async destination configuration concept:

```bash
aws lambda put-function-event-invoke-config \
  --function-name remediate-security-group \
  --maximum-retry-attempts 2 \
  --maximum-event-age-in-seconds 3600 \
  --destination-config '{
    "OnFailure": {
      "Destination": "arn:aws:sqs:us-east-1:123456789012:remediation-failures"
    },
    "OnSuccess": {
      "Destination": "arn:aws:sns:us-east-1:123456789012:remediation-success"
    }
  }'
```

Use destinations when you need post-execution handling for asynchronous Lambda invocations. Do not confuse this with EventBridge target DLQs.

**IAM permissions and trust relationships**

Remediation workflows usually need multiple permission layers:

- **Event source permissions**
  - EventBridge must be allowed to invoke the Lambda function.
- **Lambda execution role**
  - The function needs permission to call the target service APIs.
- **Service-linked or automation role**
  - AWS Config or SSM may need a role to perform remediation.
- **Cross-account permissions**
  - If the remediation spans accounts, you need explicit trust and resource policies.

Example Lambda execution role policy for security group remediation:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "ec2:RevokeSecurityGroupIngress",
        "ec2:DescribeSecurityGroups"
      ],
      "Resource": "*"
    }
  ]
}
```

This is not a perfect least-privilege example because some EC2 actions do not support resource-level restrictions. The `*` is often required because of service limitations, not because broad access is ideal.

Example trust policy for Lambda:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "lambda.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Example permission for EventBridge to invoke Lambda:

```bash
aws lambda add-permission \
  --function-name remediate-security-group \
  --statement-id allow-eventbridge \
  --action lambda:InvokeFunction \
  --principal events.amazonaws.com \
  --source-arn arn:aws:events:us-east-1:123456789012:rule/config-remediation-rule
```

**Practical remediation examples**

**1. Security group rule removal**

Use Lambda when you need to remove only a specific ingress rule after checking context.

```python
import boto3

ec2 = boto3.client("ec2")

def lambda_handler(event, context):
    group_id = event["detail"]["resourceId"]

    ec2.revoke_security_group_ingress(
        GroupId=group_id,
        IpPermissions=[
            {
                "IpProtocol": "tcp",
                "FromPort": 22,
                "ToPort": 22,
                "IpRanges": [{"CidrIp": "0.0.0.0/0"}]
            }
        ]
    )
```

**2. EBS encryption replacement workflow**

You cannot enable encryption in place on an existing unencrypted EBS volume. A Lambda remediation must typically:

1. Create a snapshot of the unencrypted volume.
2. Copy the snapshot with encryption enabled.
3. Create a new encrypted volume from the encrypted snapshot.
4. Stop or detach the instance volume if required.
5. Attach the new encrypted volume.
6. Validate the instance and clean up old resources.

This is a good example of why Lambda is useful: the workflow is multi-step and conditional.

**3. S3 public access correction**

A Lambda function can inspect bucket policy, ACLs, and public access block settings before applying a fix.

```bash
aws s3api put-public-access-block \
  --bucket my-bucket \
  --public-access-block-configuration \
  BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true
```

You can combine this with policy inspection and notification logic in Lambda.

**4. Security Hub finding response**

A Lambda function can parse the finding, determine the affected resource, apply the fix, and then update the finding workflow status.

```python
import boto3

securityhub = boto3.client("securityhub")

def mark_resolved(finding_id, product_arn):
    securityhub.batch_update_findings(
        FindingIdentifiers=[{"Id": finding_id, "ProductArn": product_arn}],
        Workflow={"Status": "RESOLVED"}
    )
```

**Operational best practices**

- **Make the function idempotent**
  - The same event may be delivered more than once.
  - Check current state before changing anything.
- **Log enough context**
  - Include resource ID, account ID, region, and event ID.
- **Use least privilege**
  - Grant only the API actions the function needs.
- **Set sane timeouts**
  - Keep the timeout long enough for the remediation, but not so long that failures linger.
- **Use retries carefully**
  - Retries help transient failures, but repeated destructive actions can cause damage.
- **Add guardrails**
  - Use allow lists, tag checks, or approval steps for risky changes.
- **Test in a nonproduction account**
  - Remediation code can have side effects that are hard to reverse.

**Choosing between Config remediation, EventBridge, and Lambda**

| Approach | Best for | Strengths | Limitations |
|---|---|---|---|
| AWS Config automatic remediation | Standard compliance fixes | Native, simple, policy-driven | Limited to supported SSM Automation documents |
| EventBridge + Lambda | Custom event-driven fixes | Flexible, decoupled, multi-service | Requires code, retries, and permissions management |
| Security Hub + Lambda | Security findings workflows | Good for triage and enrichment | Needs careful finding-state handling |
| CloudWatch alarm + Lambda | Metric-triggered actions | Fast response to operational issues | Not ideal for configuration compliance |

**Exam focus points**

- AWS Config automatic remediation usually uses **SSM Automation**, not Lambda directly.
- Lambda is the right answer when the remediation needs **custom code** or **multi-step logic**.
- EventBridge is often the glue between detection and remediation.
- Security Hub remediation often includes **finding workflow updates** after the fix.
- Lambda failure handling depends on whether the invocation is **synchronous** or **asynchronous**.
- For complex fixes, think about **idempotency**, **permissions**, and **rollback** before choosing the remediation design.

**Quick mental model**

- Detect with Config, Security Hub, CloudWatch, or an app event.
- Route with EventBridge if you need decoupling.
- Remediate with SSM Automation when the action is standard.
- Remediate with Lambda when the action is custom, conditional, or multi-step.
- Record the outcome in logs, tickets, or finding status.
