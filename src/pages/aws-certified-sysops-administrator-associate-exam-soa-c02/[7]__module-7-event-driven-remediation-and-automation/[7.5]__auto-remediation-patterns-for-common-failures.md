## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Overview**

Auto-remediation means you detect a noncompliant or unhealthy resource and trigger a predefined action to restore the desired state. In AWS, the most common building blocks are <abbr title="AWS Config">Config</abbr>, <abbr title="Amazon EventBridge">EventBridge</abbr> (formerly CloudWatch Events), <abbr title="AWS Systems Manager">Systems Manager</abbr> Automation, Lambda, and service-native controls such as S3 Block Public Access or EC2 Auto Recovery.

The exam often tests whether you choose the right tool for the failure mode:

- **Config** for configuration compliance and drift detection.
- **EventBridge** for event-driven reactions to AWS service events.
- **Systems Manager Automation** for repeatable operational runbooks.
- **Lambda** for custom logic, enrichment, and branching.
- **Native service features** when the simplest fix is built into the service.

**Core remediation pattern**

A typical flow looks like this:

1. A resource changes or drifts from policy.
2. A detector evaluates the state.
3. A rule or evaluation emits a remediation trigger.
4. An automation action restores the resource.
5. A verification step confirms compliance.
6. Failures route to logging, alerting, or escalation.

```text
Resource drift or failure
        ↓
Detector: Config rule / EventBridge / CloudWatch alarm
        ↓
Trigger: remediation rule / target / alarm action
        ↓
Executor: SSM Automation / Lambda / native API call
        ↓
Verify: re-evaluate compliance or health
        ↓
Escalate if failed: SNS / ticket / DLQ / on-call
```

**When to use each service**

| Service | Best for | Strengths | Limits |
|---|---|---|---|
| AWS Config | Detecting configuration drift and compliance | Managed rules, remediation integration, continuous evaluation | Not a general event bus |
| EventBridge | Reacting to AWS events and custom events | Flexible routing, filtering, multiple targets | Not every API activity is available unless the source emits events |
| Systems Manager Automation | Repeatable operational runbooks | Step-based workflows, approvals, retries, rollback patterns | Requires correct document design and IAM permissions |
| Lambda | Custom decision logic | Flexible code, branching, enrichment | More code to maintain; can become a glue layer |
| CloudWatch alarms | Metric-based health signals | Fast alerting, threshold-based actions | Not a configuration compliance engine by itself |

> For exam questions, remember that **CloudWatch Events** is the older name for **EventBridge**. Older study material may use either term.

**AWS Config remediation patterns**

AWS Config is the most exam-relevant service for configuration drift. It continuously evaluates resources against rules and can trigger remediation when a rule becomes noncompliant. This is ideal for common failures such as public S3 buckets, overly permissive security groups, unencrypted volumes, or disabled logging.

A Config remediation usually has two parts:

- A **Config rule** that detects noncompliance.
- A **remediation action** that fixes the resource or invokes a workflow.

Managed remediation is often the simplest option when AWS provides a ready-made action. If no managed remediation exists, you can use a custom action such as an SSM Automation document or a Lambda function.

**Managed remediation vs custom remediation**

- **Managed remediation**
  - Use when AWS provides a built-in remediation action for the rule.
  - Best for standard fixes with low complexity.
  - Reduces maintenance and implementation risk.

- **Custom remediation**
  - Use when the fix requires branching, validation, or multiple API calls.
  - Best when you need pre-checks, post-checks, or rollback logic.
  - Commonly implemented with SSM Automation or Lambda.

**Common Config remediation prerequisites**

Before remediation can run, you usually need:

- A **service role** that allows Config to invoke the remediation target.
- Permissions for the remediation target to modify the affected resource.
- For SSM Automation, a valid **Automation document** and any required instance or service permissions.
- For EC2 instance actions, the instance may need the **SSM Agent**, an IAM instance profile, and network access to Systems Manager endpoints if the workflow targets the instance itself.
- A clear rule to avoid remediation loops.

**Example: public S3 bucket remediation**

A common pattern is to detect a bucket policy or ACL that makes a bucket public, then automatically restore a secure baseline.

Possible remediation actions:

- Remove public bucket policy statements.
- Enable S3 Block Public Access.
- Reapply a known-good bucket policy.
- Notify security if the bucket is business-critical and should not be changed automatically.

A safe design often uses a staged approach:

1. Detect public access with Config.
2. Attempt automatic correction if the bucket is in a known managed set.
3. Verify the bucket is no longer public.
4. Send an alert if the bucket remains public or the fix fails.

**Example: security group drift**

Security group remediation is a common exam scenario. The goal is usually to remove an overly permissive inbound rule, such as SSH from `0.0.0.0/0`.

Be careful with the exact API shape:

- Older rules may be removed by tuple-based parameters.
- Modern security group rules often have **security group rule IDs**, and revocation by ID is more precise.
- In automation, prefer the exact rule identifier when available.

```bash
aws ec2 describe-security-group-rules \
  --filters Name=group-id,Values=sg-0123456789abcdef0

aws ec2 revoke-security-group-ingress \
  --group-id sg-0123456789abcdef0 \
  --security-group-rule-ids sgr-0123456789abcdef0
```

If you only know the rule tuple, you may need to specify the exact source, protocol, and port combination that matches the existing rule.

**Systems Manager Automation patterns**

Systems Manager Automation is the preferred choice when remediation needs a repeatable runbook. It is better than Lambda when the workflow is operationally oriented and can be expressed as a sequence of AWS API calls, shell commands, or document steps.

Use Automation when you need:

- Multi-step remediation with clear ordering.
- Built-in retries and branching.
- Approval gates for risky actions.
- Reusable runbooks across many accounts or Regions.
- Standardized operational response.

A typical Automation document can:

- Call AWS APIs with `aws:executeAwsApi`.
- Run commands on managed instances with `aws:runCommand`.
- Branch with `aws:branch`.
- Pause for approval with `aws:approve`.
- Write outputs for later steps.

> In Automation, `aws:runCommand` invokes an SSM Command document. It does not take raw `InstanceIds` in the same way a direct Run Command call does; the document schema and parameters must be structured correctly.

**Example Automation pattern for EC2 hardening**

If a Config rule detects an instance with a risky setting, an Automation document can:

1. Confirm the instance is still running and in scope.
2. Apply the fix.
3. Verify the setting.
4. Tag the instance as remediated.
5. Notify on failure.

```yaml
schemaVersion: '0.3'
description: "Remediate a risky EC2 setting"
assumeRole: '{{ AutomationAssumeRole }}'
parameters:
  InstanceId:
    type: String
  AutomationAssumeRole:
    type: String
mainSteps:
  - name: CheckInstance
    action: aws:executeAwsApi
    inputs:
      Service: ec2
      Api: DescribeInstances
      InstanceIds:
        - '{{ InstanceId }}'
  - name: ApplyFix
    action: aws:runCommand
    inputs:
      DocumentName: AWS-RunShellScript
      InstanceIds:
        - '{{ InstanceId }}'
      Parameters:
        commands:
          - 'sudo systemctl restart myservice'
  - name: VerifyFix
    action: aws:executeAwsApi
    inputs:
      Service: ec2
      Api: DescribeInstances
      InstanceIds:
        - '{{ InstanceId }}'
```

**Lambda remediation patterns**

Lambda is useful when remediation requires custom logic that is awkward in Automation. Examples include parsing event payloads, deciding whether to remediate based on tags, checking multiple conditions, or integrating with external systems.

Use Lambda when you need:

- Complex branching or data transformation.
- Custom validation before remediation.
- Integration with ticketing, chat, or third-party APIs.
- A lightweight function to orchestrate other AWS services.

Avoid Lambda when:

- The task is a straightforward operational runbook that Automation can handle.
- You need a long-running workflow.
- You want a managed, auditable step sequence with explicit runbook actions.

A common pattern is:

- EventBridge or Config invokes Lambda.
- Lambda checks tags, environment, or severity.
- Lambda either remediates directly or starts an Automation document.
- Lambda publishes failures to SNS or a queue for later processing.

**EventBridge patterns**

EventBridge is best for event-driven remediation, especially when the trigger is an AWS service event rather than a compliance rule. It can route events to Lambda, SSM Automation, SNS, SQS, Step Functions, or other targets.

Use EventBridge when:

- You want to react to a specific service event.
- You need content-based filtering.
- You want to fan out to multiple targets.
- You need to connect event sources to remediation workflows.

Be precise in exam scenarios:

- EventBridge can receive many AWS service events.
- CloudTrail management events can be routed through EventBridge when CloudTrail is enabled and the relevant events are being captured.
- Not every API call is automatically available as an EventBridge event in every account or configuration.

**Example: stopped EC2 instance**

A common remediation is to restart or replace an instance that stops unexpectedly.

Possible approaches:

- If the instance is critical and should always run, use EventBridge to detect the stop event and start it again.
- If the instance is part of an Auto Scaling Group, let the group replace it instead of manually remediating.
- If the stop is intentional, use tags or maintenance windows to suppress remediation.

```json
{
  "source": ["aws.ec2"],
  "detail-type": ["EC2 Instance State-change Notification"],
  "detail": {
    "state": ["stopped"]
  }
}
```

A target Lambda function can then call `StartInstances`:

```bash
aws ec2 start-instances --instance-ids i-0123456789abcdef0
```

**CloudWatch alarms and remediation**

CloudWatch alarms are metric-based and are useful when the failure can be represented as a metric threshold. They are not a general configuration drift detector, but they can trigger remediation if you publish a custom metric that reflects state.

Use CloudWatch alarms when:

- You have a measurable health indicator.
- You want fast threshold-based action.
- You are monitoring performance, availability, or custom application state.

Do not rely on CloudWatch alarms alone for:

- Detecting whether a security group is public.
- Detecting whether an S3 bucket policy is compliant.
- Detecting configuration drift across many resource types.

A practical pattern is:

- Application or Lambda publishes a custom metric.
- CloudWatch alarm triggers when the metric indicates failure.
- Alarm action invokes SNS, Lambda, or another remediation path.

**Failure handling and safety controls**

Auto-remediation can create outages if it is too aggressive. The exam often rewards designs that include guardrails.

Important safety controls:

- **Scope control**
  - Remediate only tagged resources or approved accounts.
  - Exclude production or critical workloads unless explicitly allowed.

- **Idempotency**
  - Make the remediation safe to run more than once.
  - Check current state before changing anything.

- **Verification**
  - Re-evaluate compliance after the fix.
  - Confirm the resource is healthy before closing the loop.

- **Loop prevention**
  - Avoid triggering the same remediation repeatedly on the same failure.
  - Use tags, state markers, or suppression windows.

- **Escalation**
  - Send failures to SNS, email, chat, ticketing, or a queue.
  - Use a dead-letter queue (DLQ) for asynchronous targets when supported.
  - Escalate after retries are exhausted.

**Retry and escalation patterns**

A robust remediation pipeline usually includes:

- Immediate retry for transient API failures.
- Backoff for throttling or eventual consistency.
- DLQ for failed asynchronous event delivery.
- SNS notification for human review.
- Ticket creation for persistent or high-severity failures.

```text
Trigger → Remediation attempt → Success
                    ↓
                 Failure
                    ↓
          Retry / backoff / recheck
                    ↓
          Still failing?
             ↓        ↓
           Yes        No
             ↓         ↓
      DLQ / SNS / ticket   Re-evaluate compliance
```

**End-to-end scenario: public S3 bucket**

This is a common exam-style architecture:

1. AWS Config evaluates the bucket against a managed rule for public access.
2. The bucket becomes noncompliant.
3. Config triggers a remediation action.
4. The remediation action runs an SSM Automation document or Lambda function.
5. The workflow removes public access and reapplies the approved policy.
6. Config re-evaluates the bucket.
7. If the bucket is still noncompliant, the workflow sends an alert to SNS and opens a ticket.

Why this pattern works:

- Config provides compliance visibility.
- Automation provides repeatable correction.
- Verification prevents false success.
- Escalation handles exceptions that need human review.

**End-to-end scenario: overly permissive security group**

A second common pattern is to remove a risky inbound rule.

1. Config detects a security group rule allowing SSH from the internet.
2. A remediation action identifies the exact rule ID.
3. The workflow revokes the rule.
4. The workflow verifies the rule is gone.
5. If the rule reappears, the workflow checks whether another system is reapplying it.
6. If repeated drift occurs, the issue is escalated to the platform team.

This scenario often appears in questions about why remediation keeps failing:

- Another automation may be recreating the rule.
- Infrastructure as Code may be reapplying the desired state.
- A manual process may be bypassing governance.
- The remediation may be targeting the wrong rule identifier.

**Choosing between Lambda and Automation**

| Requirement | Prefer Lambda | Prefer SSM Automation |
|---|---|---|
| Custom branching logic | Yes | Sometimes |
| Step-by-step operational runbook | No | Yes |
| Native AWS API orchestration | Sometimes | Yes |
| Human approval step | No | Yes |
| Simple event transformation | Yes | No |
| Auditable remediation steps | Limited | Strong |

A good exam heuristic:

- Choose **Lambda** for logic.
- Choose **Automation** for runbooks.
- Choose **Config** for compliance.
- Choose **EventBridge** for event routing.

**Practical CLI examples**

Check Config rule compliance:

```bash
aws configservice describe-compliance-by-config-rule \
  --config-rule-names s3-bucket-public-read-prohibited
```

Start an Automation execution:

```bash
aws ssm start-automation-execution \
  --document-name MyRemediationDocument \
  --parameters InstanceId=i-0123456789abcdef0,AutomationAssumeRole=arn:aws:iam::123456789012:role/AutomationRole
```

Invoke a Lambda remediation function:

```bash
aws lambda invoke \
  --function-name remediate-public-s3-bucket \
  --payload '{"bucket":"my-example-bucket"}' \
  response.json
```

Create a simple EventBridge rule targetting a Lambda function:

```bash
aws events put-rule \
  --name remediate-ec2-stop \
  --event-pattern '{"source":["aws.ec2"],"detail-type":["EC2 Instance State-change Notification"],"detail":{"state":["stopped"]}}'
```

**Common exam traps**

- Confusing **detection** with **remediation**.
  - Config detects compliance.
  - Automation or Lambda performs the fix.

- Assuming EventBridge sees every API call.
  - It depends on the event source and configuration.
  - CloudTrail integration matters for API activity scenarios.

- Using CloudWatch alarms for configuration drift.
  - Alarms are metric-based, not compliance-based.

- Choosing Lambda when a managed remediation exists.
  - Prefer the simplest supported option.

- Forgetting permissions.
  - Config, EventBridge, Lambda, and Automation all need IAM permissions to invoke actions and modify resources.

- Ignoring verification.
  - A fix is not complete until you confirm the resource is healthy or compliant again.

**Quick decision guide**

- Use **AWS Config** when the question is about compliance, drift, or policy enforcement.
- Use **EventBridge** when the question is about reacting to a service event.
- Use **Systems Manager Automation** when the question is about a repeatable operational fix.
- Use **Lambda** when the question needs custom logic or integration.
- Use **CloudWatch alarms** when the question is about metric thresholds or health signals.

**Study takeaway**

The strongest remediation designs are simple, scoped, and verifiable. In exam scenarios, look for the service that matches the trigger type, the remediation complexity, and the need for auditability. If the failure is configuration drift, start with Config. If it is an event, start with EventBridge. If it is a runbook, start with Automation. If it is custom logic, use Lambda. Then add verification, retries, and escalation so the fix is safe in production.
