## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**What Systems Manager Automation does**

AWS Systems Manager Automation lets you define repeatable operational workflows that perform corrective actions on AWS resources. You use it when you want a controlled, auditable runbook to fix a known condition such as a stopped instance, a misconfigured volume, or a resource that needs a snapshot before change.

Automation is especially useful for:
- **Remediation** after an event, alarm, or compliance finding.
- **Standardized operations** such as patching, snapshotting, or instance replacement.
- **Controlled change** with approvals, branching, retries, and rollback-style steps.
- **Cross-service orchestration** without writing a large custom application.

For exam purposes, think of Automation as the AWS-native workflow engine for operational runbooks. If the task is mostly AWS API calls, state changes, and guardrails, Automation is often a better fit than Lambda.

**When to use Automation vs Lambda**

Use Automation when the workflow is:
- Centered on AWS Systems Manager and AWS resource operations.
- Better expressed as a step-by-step runbook.
- In need of built-in execution history, approvals, and document versioning.
- Triggered by operational events that should be remediated consistently.

Use Lambda when the workflow:
- Needs custom code, complex parsing, or non-AWS integrations.
- Requires low-latency event handling or bespoke logic.
- Is easier to maintain as application code than as a runbook.

A practical rule: if you can describe the corrective action as “take snapshot, verify condition, stop instance, replace resource, notify team,” Automation is a strong candidate. If you need heavy data transformation or custom business logic, Lambda is usually the better tool.

**Automation document basics**

Automation uses a document that defines the runbook. The document includes:
- **Schema version**: commonly `0.3` for Automation documents.
- **Parameters**: inputs such as instance IDs, role ARNs, or snapshot names.
- **Main steps**: actions that call AWS APIs or Systems Manager actions.
- **Outputs**: values passed from one step to another or returned to the caller.

A document can be:
- A **default version** that callers use unless they specify another version.
- A **specific version** for controlled change management.
- **Shared** with other accounts or made public when appropriate.

Versioning matters in exam scenarios because you may want a stable, approved runbook version for remediation while testing a newer version separately. If a target system must use a known-good workflow, reference the specific document version rather than relying on the default.

**Execution model and targeting**

Automation can run against:
- A **single resource**, such as one EC2 instance or one EBS volume.
- **Multiple targets**, where you supply target parameters and the runbook iterates over matching resources.

Target selection is commonly done by:
- Explicit IDs, such as `InstanceId`.
- Resource tags, such as `tag:Environment=Prod`.
- Parameter-driven targeting from an event payload or remediation rule.

This matters because corrective actions often need to scale from one affected resource to many. For example, a compliance rule might target all instances with a specific tag, while an alarm-triggered runbook might target only the instance that breached a threshold.

**Common corrective-action patterns**

Automation is often used in these patterns:
- **Snapshot before change**: create an EBS snapshot before modifying or replacing a volume.
- **Backup before remediation**: create an AMI or snapshot before stopping, terminating, or reconfiguring an instance.
- **Quarantine or isolate**: change security groups, detach a resource, or stop an instance.
- **Restore or replace**: launch a replacement instance, attach a known-good volume, or revert configuration.
- **Notify and record**: send a message to SNS or create an audit trail after the action completes.

A safe remediation workflow usually starts with a protection step. For example, before changing a production volume, you might snapshot it first so you can recover if the remediation causes unexpected impact.

**Runbook step types and control flow**

Automation runbooks support step-by-step control flow. Common capabilities include:
- Calling AWS APIs through built-in actions.
- Branching based on step results or parameter values.
- Failing fast when a prerequisite is not met.
- Retrying transient failures.
- Capturing outputs from one step and using them in later steps.

Typical control-flow concepts:
- **onFailure**: defines what happens if a step fails, such as aborting the runbook or jumping to a rollback step.
- **maxAttempts**: retries a step when the failure is likely transient.
- **timeoutSeconds**: prevents a step from hanging indefinitely.
- **outputs**: exposes values from a step for later use.

Example of a simple step structure:

```yaml
schemaVersion: '0.3'
description: "Snapshot before remediation"
parameters:
  InstanceId:
    type: String
  AutomationAssumeRole:
    type: String
mainSteps:
  - name: CreateSnapshot
    action: aws:executeAwsApi
    inputs:
      Service: ec2
      Api: CreateSnapshot
      VolumeId: '{{ VolumeId }}'
    outputs:
      - Name: SnapshotId
        Selector: '$.SnapshotId'
        Type: String
  - name: Notify
    action: aws:executeAwsApi
    inputs:
      Service: sns
      Api: Publish
      TopicArn: '{{ TopicArn }}'
      Message: 'Snapshot created: {{ CreateSnapshot.SnapshotId }}'
```

The output syntax is important because later steps often depend on values returned by earlier steps. In exam questions, look for references like `{{ StepName.OutputName }}` or JSONPath selectors.

**Common Automation actions**

Automation documents commonly use actions such as:
- `aws:executeAwsApi` to call AWS service APIs.
- `aws:executeScript` to run small scripts for logic that is awkward in pure API steps.
- `aws:runCommand` to run commands on managed instances.
- `aws:changeInstanceState` to start or stop an EC2 instance.
- `aws:branch` to choose a path based on conditions.
- `aws:waitForAwsResourceProperty` to wait until a resource reaches a desired state.

Important correction for exam prep:
- `aws:changeInstanceState` supports **start** and **stop** operations for EC2 instances.
- It does **not** use reboot as a desired state.

If you need a reboot, use a different approach such as `aws:runCommand` with an OS-level reboot command or another workflow that explicitly performs reboot logic.

**Example: start or stop an instance**

```yaml
schemaVersion: '0.3'
description: "Stop an instance for maintenance"
parameters:
  InstanceId:
    type: String
mainSteps:
  - name: StopInstance
    action: aws:changeInstanceState
    inputs:
      InstanceIds:
        - '{{ InstanceId }}'
      DesiredState: stopped
```

And to start it later:

```yaml
schemaVersion: '0.3'
description: "Start an instance after maintenance"
parameters:
  InstanceId:
    type: String
mainSteps:
  - name: StartInstance
    action: aws:changeInstanceState
    inputs:
      InstanceIds:
        - '{{ InstanceId }}'
      DesiredState: running
```

**Approvals and human-in-the-loop control**

For exam scenarios, be careful with approval concepts. Automation runbooks are not typically described as using a standard `aws:approve` step. If a workflow needs human approval, that is usually handled through:
- **Change Manager** approval workflows.
- A pause-and-resume pattern outside the runbook.
- An external approval process that triggers the next step after authorization.

If a question mentions approval, check whether it is really asking about Systems Manager Change Manager rather than Automation itself.

**Rollback and recovery patterns**

Corrective actions should be designed with rollback in mind. Common patterns include:
- Create a snapshot or AMI before making changes.
- Store the original configuration in a parameter or output.
- Branch to a rollback step if validation fails.
- Stop the workflow and notify operators if the system is in an unsafe state.

Example rollback-oriented flow:
1. Create a snapshot.
2. Apply the remediation.
3. Verify the resource state.
4. If verification fails, restore from the snapshot or alert an operator.

A rollback step is especially useful when the remediation changes infrastructure state, such as replacing a volume, modifying a security group, or restarting a service.

**Event-driven remediation sources**

Automation often starts from an event or finding. Common sources include:
- **Amazon EventBridge** rules.
- **AWS Config** remediation actions.
- **Amazon CloudWatch** alarms.
- **AWS Security Hub** findings.
- **Amazon GuardDuty** findings.

These integrations are exam-relevant because they show how corrective actions become automatic. A finding or alarm can trigger a runbook that performs a predefined fix, reducing mean time to recovery.

**AWS Config remediation**

AWS Config can trigger Automation as a remediation action when a resource is noncompliant. This is useful for guardrail-style operations such as:
- Reapplying a required tag.
- Reverting an insecure security group rule.
- Taking a snapshot before a change.
- Stopping or isolating a noncompliant resource.

A remediation configuration typically includes:
- The Config rule name.
- The target Automation document.
- The target parameter mappings.
- An assume role for Automation.
- Optional automatic remediation settings.

Example shape of a remediation configuration:

```json
{
  "ConfigRuleName": "restricted-ssh",
  "TargetType": "SSM_DOCUMENT",
  "TargetId": "AWS-DisablePublicAccessForSecurityGroup",
  "Automatic": true,
  "MaximumAutomaticAttempts": 5,
  "RetryAttemptSeconds": 60,
  "Parameters": {
    "GroupId": {
      "ResourceValue": {
        "Value": "RESOURCE_ID"
      }
    }
  }
}
```

The exact document name and parameters vary by use case, but the key exam idea is that Config can automatically invoke an Automation document when a rule detects noncompliance.

**EventBridge integration**

EventBridge can route operational events to Systems Manager Automation. This is common for:
- Security findings from Security Hub or GuardDuty.
- Operational alarms from CloudWatch.
- Custom application events that indicate a known failure mode.

A typical EventBridge rule sends the event to an SSM Automation target using a role that allows EventBridge to invoke the target action. The target role and permissions depend on the integration pattern, and you may also need permission to pass an Automation assume role if the runbook uses one.

Example EventBridge rule pattern:

```json
{
  "source": ["aws.guardduty"],
  "detail-type": ["GuardDuty Finding"],
  "detail": {
    "severity": [{ "numeric": [">=", 7] }]
  }
}
```

Example target concept:
- Rule matches the event.
- Target invokes an Automation document.
- Event fields are mapped to runbook parameters.
- Runbook performs the corrective action.

For exam questions, focus on the flow: **event source → EventBridge rule → Automation document → remediation**.

**CloudWatch alarm-driven remediation**

CloudWatch alarms can trigger remediation when a metric indicates a problem. Examples include:
- Restarting or replacing a failed instance.
- Scaling out capacity.
- Running a diagnostic or snapshot workflow.
- Notifying operators after a threshold breach.

This is useful when the condition is metric-based rather than compliance-based. For example, a disk utilization alarm might trigger a runbook that snapshots a volume and then expands it.

**CLI examples**

To start an Automation execution from the AWS CLI, pass parameters as a JSON map of string arrays.

```bash
aws ssm start-automation-execution \
  --document-name "MyRemediationRunbook" \
  --parameters '{
    "InstanceId":["i-0123456789abcdef0"],
    "AutomationAssumeRole":["arn:aws:iam::123456789012:role/SSMAutomationRole"]
  }'
```

If you want to specify a document version:

```bash
aws ssm start-automation-execution \
  --document-name "MyRemediationRunbook" \
  --document-version "3" \
  --parameters '{
    "InstanceId":["i-0123456789abcdef0"],
    "AutomationAssumeRole":["arn:aws:iam::123456789012:role/SSMAutomationRole"]
  }'
```

To inspect an execution:

```bash
aws ssm describe-automation-executions
aws ssm get-automation-execution --automation-execution-id "12345678-1234-1234-1234-123456789012"
```

These commands are useful for troubleshooting because they show step status, failure messages, and execution history.

**IAM roles and permissions**

Automation usually needs an execution role. The runbook often accepts this as a parameter such as `AutomationAssumeRole`. That role must trust the Systems Manager service and allow the actions the runbook performs.

Example trust policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "ssm.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Example permissions policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "ec2:CreateSnapshot",
        "ec2:DescribeVolumes",
        "ec2:StopInstances",
        "ec2:StartInstances",
        "sns:Publish"
      ],
      "Resource": "*"
    }
  ]
}
```

Key points:
- The role is not automatically used unless the runbook or execution context specifies it.
- The role must have permission for every AWS API call the runbook makes.
- If another service invokes Automation and passes a role, that invoking service may also need permission to pass the role.

**Document sharing and ownership**

Automation documents can be:
- Owned by your account.
- Shared with other accounts.
- Publicly shared when appropriate and allowed.

This matters when you use AWS-managed documents or cross-account operational patterns. In exam questions, a shared document may be referenced by name or ARN, and the caller still needs permission to execute it.

Also remember:
- The **default version** is what callers use if they do not specify a version.
- A **specific version** is safer for controlled remediation.
- Shared documents may be updated by the owner, so pinning a version can reduce surprise changes.

**Execution logging and traceability**

Automation provides execution history that helps you troubleshoot and audit corrective actions. You can inspect:
- Step-by-step status.
- Input parameters.
- Output values.
- Failure messages.
- Timing and retries.

For operational visibility, you may also send notifications or log related events to:
- **Amazon CloudWatch Logs** for centralized logging.
- **Amazon SNS** for operator notifications.
- **AWS CloudTrail** for API-level audit history.

When troubleshooting, check:
1. Whether the document version is the one you expected.
2. Whether the assume role has the required permissions.
3. Whether the target resource ID or tag mapping is correct.
4. Whether a step failed due to timeout, bad input, or missing outputs.

**Practical remediation example**

A common corrective action is to snapshot a volume before making a change, then notify the team.

```yaml
schemaVersion: '0.3'
description: "Snapshot an EBS volume before remediation"
parameters:
  VolumeId:
    type: String
  TopicArn:
    type: String
  AutomationAssumeRole:
    type: String
mainSteps:
  - name: CreateSnapshot
    action: aws:executeAwsApi
    inputs:
      Service: ec2
      Api: CreateSnapshot
      VolumeId: '{{ VolumeId }}'
      Description: 'Pre-remediation snapshot'
    outputs:
      - Name: SnapshotId
        Selector: '$.SnapshotId'
        Type: String

  - name: Notify
    action: aws:executeAwsApi
    inputs:
      Service: sns
      Api: Publish
      TopicArn: '{{ TopicArn }}'
      Message: 'Created snapshot {{ CreateSnapshot.SnapshotId }} for volume {{ VolumeId }}'
```

This pattern is common because it reduces risk before a disruptive action. It also creates an audit trail that helps you recover if the remediation causes issues.

**How exam questions usually frame Automation**

Look for wording such as:
- “Automatically remediate noncompliant resources.”
- “Run a predefined workflow when an alarm or finding occurs.”
- “Take a snapshot before making changes.”
- “Use a Systems Manager document to perform corrective actions.”
- “Require controlled, auditable operational steps.”

If the question emphasizes:
- AWS-native operational workflow, choose **Automation**.
- Custom code or complex logic, choose **Lambda**.
- Approval workflow and change governance, consider **Change Manager**.
- Compliance-driven auto-remediation, consider **AWS Config remediation**.

**Common pitfalls**

- Confusing `aws:changeInstanceState` with reboot behavior.
- Treating approval as a standard Automation step instead of a separate governance flow.
- Forgetting to pass `AutomationAssumeRole` when the runbook needs permissions.
- Using the wrong parameter format in the CLI.
- Ignoring document versioning when a stable remediation workflow is required.
- Assuming EventBridge only needs a single permission without considering the target role and any pass-role requirements.
- Skipping backup or snapshot steps before a destructive remediation.

**Quick decision guide**

- Use **Automation** for repeatable AWS operational runbooks.
- Use **AWS Config remediation** when the trigger is noncompliance.
- Use **EventBridge** when the trigger is an event or finding.
- Use **CloudWatch alarms** when the trigger is a metric threshold.
- Use **Lambda** when the remediation needs custom code.
