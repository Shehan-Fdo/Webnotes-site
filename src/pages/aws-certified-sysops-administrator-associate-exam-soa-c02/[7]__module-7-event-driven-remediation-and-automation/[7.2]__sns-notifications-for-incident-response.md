## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Amazon SNS for incident response**

Amazon Simple Notification Service (<abbr title="Simple Notification Service">SNS</abbr>) is a managed pub/sub messaging service that helps you fan out alerts to multiple responders and systems with low operational overhead. In incident response, you usually use SNS as the notification layer between detection systems and human or automated responders.

SNS is a strong fit when you need:

- Fast broadcast of the same alert to many subscribers.
- Loose coupling between the alert producer and the response targets.
- Simple integration with AWS services such as CloudWatch alarms, EventBridge rules, AWS Lambda, and AWS Systems Manager automation.

A common incident-response pattern is:

1. A detector raises an event or alarm.
2. The event is published to an SNS topic.
3. Multiple subscribers receive the notification through their chosen protocol.
4. A responder or automation workflow takes action.

**Core concepts**

An SNS **topic** is the logical channel that publishers send messages to. Subscribers attach to the topic and receive messages according to protocol and filter rules. This makes SNS useful for incident response because one alarm can notify on-call engineers, trigger automation, and create tickets at the same time.

Key terms:

- **Publisher**: The service or application that sends the message.
- **Topic**: The message distribution channel.
- **Subscription**: The binding between a topic and a delivery endpoint.
- **Endpoint**: The destination that receives the message, such as an email address, Lambda function, HTTP endpoint, or SQS queue.
- **Fanout**: Delivering one message to multiple subscribers.

SNS is regional. You create topics in a specific AWS Region, and subscriptions are associated with that topic in that Region. If you need multi-Region alerting, you design for it explicitly with separate regional topics, cross-Region event routing, or custom forwarding logic. SNS does not automatically replicate topics across Regions.

**When to use SNS in incident response**

Use SNS when you need broad notification and simple integration rather than complex workflow orchestration.

Typical use cases:

- CloudWatch alarm notifications for CPU, latency, error rate, or health checks.
- Security alerts from EventBridge rules or AWS services.
- Pager and email notifications for on-call teams.
- Triggering Lambda to enrich or route alerts.
- Sending messages to SQS for downstream processing or ticket creation.

SNS is not the best choice when you need:

- Long-running orchestration with branching logic.
- Human approval workflows with stateful steps.
- Guaranteed exactly-once processing.
- Rich incident management features such as escalation policies and schedules.

For those needs, you often combine SNS with AWS Step Functions, EventBridge, Lambda, or a third-party incident platform.

**Subscription protocols and delivery targets**

SNS supports multiple delivery protocols. Choose the protocol based on the responder or system you want to reach.

| Protocol | Typical use | Notes |
|---|---|---|
| Email | Human notification | Requires subscription confirmation before delivery |
| Email-JSON | Human-readable structured email | Useful when you want JSON payloads in email |
| HTTP / HTTPS | Webhooks and integrations | HTTPS is preferred for security |
| Amazon SQS | Queue-based downstream processing | Good for durable buffering and decoupling |
| AWS Lambda | Automated response | Invokes a function on publish |
| SMS | Text message alerts | Useful for urgent human notification, subject to regional and account limits |
| Application | Mobile push via platform applications/endpoints | Mobile push is configured through SNS platform applications, not as a generic subscription protocol in the same sense as email or SMS |

For incident response, the most common patterns are:

- **Email** for low-friction human alerts.
- **SMS** for urgent, short-form notifications.
- **Lambda** for automated enrichment or remediation.
- **SQS** for durable handoff to another system.
- **HTTPS** for webhook-based incident tools.

> Confirm email subscriptions before expecting delivery. A `publish` call can succeed even if the email endpoint has not been confirmed yet.

**Creating a topic and subscriptions**

You can create and use SNS topics from the AWS CLI. This is useful for repeatable incident-response setup and exam scenarios.

```bash
aws sns create-topic --name incident-alerts
```

Example output:

```json
{
  "TopicArn": "arn:aws:sns:us-east-1:123456789012:incident-alerts"
}
```

Subscribe an email endpoint:

```bash
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:incident-alerts \
  --protocol email \
  --notification-endpoint oncall@example.com
```

Subscribe a Lambda function:

```bash
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:incident-alerts \
  --protocol lambda \
  --notification-endpoint arn:aws:lambda:us-east-1:123456789012:function:process-incident
```

Publish a test message:

```bash
aws sns publish \
  --topic-arn arn:aws:sns:us-east-1:123456789012:incident-alerts \
  --subject "High CPU alarm" \
  --message "Instance i-0123456789abcdef0 exceeded CPU threshold"
```

Important operational detail:

- Email subscriptions must be confirmed before they receive messages.
- Lambda subscriptions require permission for SNS to invoke the function.
- HTTPS endpoints must be reachable and should validate the SNS message signature if you process messages directly.

**CloudWatch alarm integration**

A common incident-response setup is CloudWatch Alarm -> SNS Topic -> responders. This is one of the most testable patterns for SOA-C02.

Example alarm action:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name HighCPU-ProdWeb \
  --metric-name CPUUtilization \
  --namespace AWS/EC2 \
  --statistic Average \
  --period 300 \
  --evaluation-periods 2 \
  --threshold 80 \
  --comparison-operator GreaterThanThreshold \
  --dimensions Name=InstanceId,Value=i-0123456789abcdef0 \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:incident-alerts
```

Why this works well:

- CloudWatch detects the condition.
- SNS fans out the alarm to multiple destinations.
- You can add or remove responders without changing the alarm itself.

Practical considerations:

- Use separate topics for different severities, such as `critical`, `warning`, and `informational`.
- Include enough context in the alarm description or message to help responders triage quickly.
- Route only actionable alerts to paging channels to reduce noise.

**Message structure and formatting**

SNS messages can carry plain text or structured JSON, depending on the publisher and protocol. For incident response, structured messages are often better because they preserve context for automation and downstream tools.

Example JSON payload:

```json
{
  "alarmName": "HighCPU-ProdWeb",
  "severity": "critical",
  "resource": "i-0123456789abcdef0",
  "region": "us-east-1",
  "state": "ALARM",
  "summary": "CPU utilization exceeded 80% for 10 minutes",
  "runbook": "https://internal.example.com/runbooks/high-cpu"
}
```

Good incident messages usually include:

- What happened.
- Which resource is affected.
- When it started.
- Severity or priority.
- Suggested next action or runbook link.

If you publish from Lambda or EventBridge, you can enrich the message before sending it to SNS so responders get more context.

**Filtering and routing**

SNS subscription filter policies let you route messages to different subscribers based on message attributes. This is useful when you want one topic to carry many event types but only deliver relevant alerts to each team.

Example message attributes:

```bash
aws sns publish \
  --topic-arn arn:aws:sns:us-east-1:123456789012:incident-alerts \
  --message "Database connection failures detected" \
  --message-attributes '{
    "severity": {"DataType":"String","StringValue":"critical"},
    "service": {"DataType":"String","StringValue":"orders"},
    "environment": {"DataType":"String","StringValue":"prod"}
  }'
```

Example filter policy for a subscription:

```json
{
  "severity": ["critical"],
  "environment": ["prod"]
}
```

Use filtering when:

- Multiple teams share one topic.
- You want to reduce alert noise.
- You need different escalation paths for different severities or services.

Avoid overcomplicating routing when a simple topic-per-team or topic-per-severity design is easier to operate.

**Retries, delivery status, and failure handling**

SNS retries delivery for supported protocols according to protocol-specific behavior. This matters in incident response because transient failures should not silently drop alerts.

Operational points:

- Delivery behavior differs by protocol.
- HTTP/S endpoints may receive retries if the endpoint is unavailable.
- Lambda invocations can fail and be retried according to service behavior.
- SQS provides durable buffering if the downstream consumer is unavailable.
- SMS and email are not equivalent to durable queueing; they are notification channels.

Dead-letter queue (<abbr title="dead-letter queue">DLQ</abbr>) support is not universal for every SNS subscription type. It depends on the protocol and configuration. Do not assume every endpoint can use a DLQ in the same way.

Use failure handling patterns such as:

- SQS subscription for durable handoff.
- Lambda error handling and retries.
- Endpoint monitoring for webhook health.
- Separate backup notification channels for critical incidents.

> Do not rely on SNS alone as your incident archive. If message retention matters, send a copy to SQS, Kinesis, or a logging system.

**Security and access control**

SNS topics are protected by IAM (Identity and Access Management) policies. For incident response, you usually want to restrict who can publish and who can subscribe.

Common controls:

- Topic access policy to allow only approved publishers.
- IAM permissions for operators and automation roles.
- Encryption at rest with AWS KMS (Key Management Service) for sensitive messages.
- HTTPS endpoints for secure delivery.
- Signature verification for webhook consumers.

Example topic policy allowing CloudWatch alarms to publish:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCloudWatchPublish",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudwatch.amazonaws.com"
      },
      "Action": "sns:Publish",
      "Resource": "arn:aws:sns:us-east-1:123456789012:incident-alerts"
    }
  ]
}
```

If you use customer-managed KMS keys, ensure the publishing service and subscribers have the necessary permissions. Otherwise, encrypted topics can block delivery or publishing.

**Incident-response design patterns**

A few practical topologies show up often in AWS environments:

- **Single topic, multiple subscribers**
  - CloudWatch alarm publishes to one topic.
  - Email, SMS, and Lambda subscribers all receive the same alert.
  - Best for small teams and simple operations.

- **Severity-based topics**
  - `critical-alerts`, `warning-alerts`, and `audit-alerts`.
  - Different subscribers and escalation paths per topic.
  - Best when paging should be reserved for high-severity events.

- **SNS to Lambda to enrichment**
  - SNS receives the raw alert.
  - Lambda adds metadata such as owner, runbook, or recent deployment info.
  - Lambda forwards to chat, ticketing, or paging systems.

- **SNS to SQS to processor**
  - SNS fans out to SQS for durable processing.
  - A consumer service reads from SQS and creates incidents or tickets.
  - Best when downstream processing must survive temporary outages.

- **EventBridge to SNS**
  - EventBridge matches AWS events and routes selected ones to SNS.
  - Useful for operational events, security findings, or custom application events.

**Practical examples**

Example CloudFormation topic and email subscription:

```yaml
Resources:
  IncidentTopic:
    Type: AWS::SNS::Topic
    Properties:
      TopicName: incident-alerts

  OnCallEmailSubscription:
    Type: AWS::SNS::Subscription
    Properties:
      TopicArn: !Ref IncidentTopic
      Protocol: email
      Endpoint: oncall@example.com
```

Example Lambda subscription permission:

```yaml
Resources:
  IncidentProcessorPermission:
    Type: AWS::Lambda::Permission
    Properties:
      Action: lambda:InvokeFunction
      FunctionName: !Ref IncidentProcessor
      Principal: sns.amazonaws.com
      SourceArn: !Ref IncidentTopic
```

Example AWS CLI to inspect subscriptions:

```bash
aws sns list-subscriptions-by-topic \
  --topic-arn arn:aws:sns:us-east-1:123456789012:incident-alerts
```

Example to confirm a subscription from the console or email link:

1. Open the confirmation email.
2. Click the confirmation link.
3. Verify the subscription status changes to `Confirmed`.

**Troubleshooting checklist**

When SNS notifications do not arrive, check the full path from publisher to endpoint.

- Verify the topic ARN and Region.
- Confirm the subscription exists and is active.
- Confirm email subscriptions are confirmed.
- Check IAM permissions for the publisher.
- Check endpoint health for HTTP/S or Lambda targets.
- Review CloudWatch metrics and logs for delivery failures.
- Validate message attributes if filter policies are in use.
- Confirm KMS permissions if the topic is encrypted.

For Lambda targets, also check:

- Function invocation permissions.
- Lambda errors and throttling.
- Dead-letter or retry configuration on the function side, if applicable.

For HTTP/S targets, also check:

- TLS certificate validity.
- Endpoint response codes.
- Request signature validation.
- Network reachability and firewall rules.

**Exam-focused takeaways**

For SOA-C02, remember these high-value points:

- SNS is a pub/sub service used to fan out incident alerts.
- CloudWatch alarms commonly publish to SNS topics.
- SNS supports multiple protocols, but delivery behavior differs by protocol.
- Email subscriptions require confirmation before delivery.
- SNS does not automatically replicate topics across Regions.
- Filter policies help route alerts by severity, service, or environment.
- SQS is the better choice when you need durable buffering.
- Lambda is a common subscriber when you want automated response or enrichment.

**Quick mental model**

Think of SNS as the alert distribution layer:

- Detection happens elsewhere.
- SNS distributes the alert.
- Subscribers decide what to do next.

That separation is what makes SNS valuable in incident response: you can change responders without changing the detector.
