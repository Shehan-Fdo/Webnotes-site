## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**RDS health monitoring overview**

Amazon Relational Database Service (Amazon RDS) health monitoring is about detecting availability issues, storage pressure, failover events, and performance degradation before they become outages. For exam purposes, separate three layers of visibility:

- **Service health**: whether the DB instance is running, available, or undergoing maintenance.
- **Resource health**: CPU, memory, disk, network, connections, and I/O behavior.
- **Storage health**: allocated storage, free space, growth trends, and autoscaling readiness.

RDS monitoring is primarily built from Amazon CloudWatch metrics, RDS events, logs, Enhanced Monitoring, and Performance Insights. The right tool depends on whether you need coarse operational alerting, OS-level detail, or SQL-level performance analysis.

**Core health signals**

RDS exposes several signals that help you determine whether a database is healthy enough to serve traffic.

- **DB instance status**: `available`, `backing-up`, `modifying`, `storage-full`, `rebooting`, `failed`, or `maintenance` are operational states you should watch.
- **Availability**: a healthy instance is reachable and accepting connections; a degraded instance may still exist but fail application checks.
- **Failover state**: Multi-AZ deployments can be healthy overall while the primary is being replaced during failover.
- **Maintenance state**: pending maintenance can explain reboots, brief interruptions, or parameter changes.
- **Event stream**: RDS emits events for failovers, backups, storage changes, maintenance, and configuration updates.

A practical exam mindset is to treat “instance exists” and “instance is healthy” as different conditions. A DB can be provisioned and still be unavailable to applications because of storage exhaustion, failover, or maintenance.

**CloudWatch metrics for RDS health**

Amazon CloudWatch is the first place to check for operational trends and alarms. It gives you time-series metrics for the DB instance and, in some cases, the Aurora cluster.

Common RDS instance metrics include:

- **CPUUtilization**: sustained high CPU can indicate query pressure, missing indexes, or too-small instance classes.
- **FreeableMemory**: low memory often correlates with swapping, cache pressure, or connection overload.
- **ReadIOPS / WriteIOPS**: useful for workload intensity and storage subsystem pressure.
- **ReadLatency / WriteLatency**: rising latency can indicate storage bottlenecks.
- **DiskQueueDepth**: queued I/O can show storage saturation.
- **FreeStorageSpace**: critical for standard RDS storage exhaustion detection.
- **DatabaseConnections**: helps detect connection storms or leaks.
- **NetworkReceiveThroughput / NetworkTransmitThroughput**: useful for replication and client traffic analysis.

For exam prep, remember that CloudWatch is best for alerting and trend detection, not deep SQL diagnostics.

**Status checks and instance health indicators**

RDS does not expose EC2-style system status checks, but you still need to monitor instance health through service state and application reachability.

- **DB instance status** tells you whether AWS considers the instance operational.
- **Availability checks** from the application side confirm that the endpoint accepts connections and queries.
- **Maintenance and failover events** explain temporary unavailability.
- **Replica lag** and replication health matter for read replicas and Aurora replicas.

A good operational pattern is to combine:
- a CloudWatch alarm on a metric,
- an RDS event subscription for state changes,
- and an external synthetic check that attempts a real connection.

**RDS events and event subscriptions**

RDS events are important because not every operational issue appears first as a metric. Events can notify you about failovers, backups, parameter changes, storage autoscaling, and maintenance actions.

You can route RDS events to Amazon Simple Notification Service (Amazon SNS) by creating an event subscription.

Typical uses:
- notify the operations team when a failover starts or completes,
- alert on pending maintenance,
- track storage autoscaling actions,
- capture backup failures or restore events.

Example subscription creation:

```bash
aws rds create-event-subscription \
  --subscription-name rds-ops-events \
  --sns-topic-arn arn:aws:sns:us-east-1:123456789012:rds-alerts \
  --source-type db-instance \
  --event-categories availability backup configuration failover maintenance notification recovery \
  --enabled
```

Useful event categories include:
- **availability**
- **backup**
- **configuration**
- **failover**
- **maintenance**
- **notification**
- **recovery**

> Event categories and exact event names vary by engine and RDS feature. Use subscriptions for notification routing, but verify the specific event types you need in your environment.

**Enhanced Monitoring**

Enhanced Monitoring provides OS-level visibility from the database host, which is more detailed than standard CloudWatch metrics. It is useful when you need to understand memory pressure, process activity, thread states, or kernel-level behavior.

You enable it when creating or modifying the DB instance, choose an IAM role for RDS to publish the metrics, and set the monitoring interval.

Key configuration points:
- **Monitoring role**: an IAM role that allows RDS to publish Enhanced Monitoring data.
- **Monitoring interval**: the granularity in seconds, commonly `1`, `5`, `10`, `15`, `30`, or `60`.
- **Retention and access**: data is delivered to CloudWatch Logs, where you can inspect it with Logs Insights.

Example enabling Enhanced Monitoring:

```bash
aws rds modify-db-instance \
  --db-instance-identifier mydb \
  --monitoring-interval 60 \
  --monitoring-role-arn arn:aws:iam::123456789012:role/rds-monitoring-role \
  --apply-immediately
```

When to use it:
- investigate memory exhaustion or swap usage,
- correlate OS-level spikes with database latency,
- inspect process and thread behavior during incidents.

When not to rely on it alone:
- it does not replace SQL-level analysis,
- it is more detailed than needed for simple storage alarms,
- it is not the best first tool for query tuning.

**Performance Insights**

Performance Insights helps you identify database load and top SQL contributors. It is especially useful when the issue is not storage capacity but query contention, lock waits, or CPU saturation.

Use it when:
- you need to find the top SQL statements causing load,
- you want to understand wait events,
- CloudWatch shows a problem but not the cause.

Performance Insights is not a replacement for CloudWatch alarms. It complements them by showing who is consuming database resources and why.

Example enabling Performance Insights:

```bash
aws rds modify-db-instance \
  --db-instance-identifier mydb \
  --enable-performance-insights \
  --performance-insights-retention-period 7 \
  --apply-immediately
```

Practical guidance:
- use CloudWatch for alerting and trend detection,
- use Enhanced Monitoring for host-level detail,
- use Performance Insights for SQL and wait analysis.

**Standard RDS storage monitoring**

For standard RDS engines, storage is allocated to the DB instance volume. The most important metric is **FreeStorageSpace**, which tells you how much usable storage remains.

Watch for:
- steady downward trends in free space,
- sudden drops after bulk loads or index rebuilds,
- log growth from verbose logging,
- temporary spikes during maintenance or large transactions.

Other storage-related signals:
- **Read/Write latency** can rise before storage is fully exhausted.
- **DiskQueueDepth** can increase when the storage subsystem is under pressure.
- **FreeableMemory** can indirectly affect storage behavior if the engine starts spilling or thrashing.

A common operational mistake is to watch only allocated storage size. Allocated size does not tell you whether the database is close to running out of free space.

**RDS storage autoscaling**

RDS storage autoscaling helps prevent storage exhaustion by automatically increasing allocated storage when free space falls below a threshold. You configure a **maximum allocated storage** value, and RDS grows the volume automatically up to that limit.

Important behavior:
- the trigger is based on low free storage, not on the maximum value itself,
- RDS performs a single automatic storage modification to a larger size,
- autoscaling does not shrink storage,
- the feature is engine and storage-type dependent.

Typical configuration parameters:
- **Allocated storage**: current size of the DB volume.
- **Maximum allocated storage**: upper limit RDS can grow to.
- **Free-space threshold**: internal trigger condition based on remaining free storage.
- **Supported engines/storage**: varies by engine and deployment model.

Example enabling storage autoscaling:

```bash
aws rds modify-db-instance \
  --db-instance-identifier mydb \
  --max-allocated-storage 500 \
  --apply-immediately
```

Operational notes:
- set the maximum high enough to absorb growth, but not so high that runaway data growth goes unnoticed,
- monitor storage trends even when autoscaling is enabled,
- use alarms to warn you before autoscaling is needed.

> Autoscaling is a safety net, not a capacity planning strategy. If the database grows unexpectedly, you still need alarms and investigation.

**Storage monitoring alarms**

CloudWatch alarms are the main exam-relevant control for proactive response. A good alarm strategy uses thresholds that warn before service impact.

Common alarm patterns:
- **FreeStorageSpace low**: alert when free space drops below a safe threshold.
- **CPUUtilization high**: detect sustained compute pressure.
- **DatabaseConnections high**: catch connection storms.
- **ReadLatency / WriteLatency high**: identify storage bottlenecks.
- **ReplicaLag high**: detect replication delay.

Example alarm for low free storage:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name rds-free-storage-low \
  --namespace AWS/RDS \
  --metric-name FreeStorageSpace \
  --dimensions Name=DBInstanceIdentifier,Value=mydb \
  --statistic Minimum \
  --period 300 \
  --evaluation-periods 2 \
  --threshold 10737418240 \
  --comparison-operator LessThanThreshold \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:rds-alerts
```

Why `Minimum` here:
- for free-space alarms, the lowest point in the period is often the safest indicator,
- it reduces the chance of missing a brief but important dip.

For storage usage trend analysis, you may also inspect `Maximum` or `Average` depending on the metric and the question you are answering. The statistic should match the operational goal.

**CloudWatch Logs export**

RDS can export database logs to CloudWatch Logs, which helps with error analysis, slow query review, and audit-style troubleshooting. Supported log types vary by engine.

Examples:
- MySQL may support logs such as `error`, `general`, `slowquery`, and `audit`.
- PostgreSQL uses different log names and formats.
- SQL Server and Oracle have their own logging patterns.

Example enabling log exports for a MySQL-compatible engine:

```bash
aws rds modify-db-instance \
  --db-instance-identifier mydb \
  --cloudwatch-logs-export-configuration EnableLogTypes=error,slowquery \
  --apply-immediately
```

Use logs export when:
- you need to centralize database logs,
- you want to query logs with CloudWatch Logs Insights,
- you need longer-term retention than the DB engine’s local log files.

A safer Logs Insights approach is to query for known engine-specific patterns rather than generic text matching. For example, inspect the actual slow query format before building a filter.

Example Logs Insights query pattern:

```sql
fields @timestamp, @message
| sort @timestamp desc
| limit 50
```

Then refine based on the engine’s log structure, such as query time fields, statement text, or error codes.

**Aurora monitoring differences**

Amazon Aurora uses a shared cluster volume model, so storage monitoring differs from standard RDS. You monitor both the **cluster** and the **instances**.

Important Aurora concepts:
- the cluster volume is shared across instances,
- storage grows automatically as data is written,
- instance metrics still matter for CPU, memory, connections, and latency,
- failover behavior is often visible through cluster and replica state changes.

Key Aurora metrics and signals:
- **VolumeBytesUsed**: amount of storage consumed in the cluster volume.
- **ReplicaLag**: delay between writer and reader replicas.
- **CPUUtilization**: instance-level compute pressure.
- **FreeableMemory**: instance memory pressure.
- **DatabaseConnections**: connection load on writer or readers.
- **Failover events**: indicate writer replacement or promotion.

Interpretation guidance:
- use **VolumeBytesUsed** to understand cluster storage growth,
- use instance metrics to diagnose performance on a specific writer or reader,
- use **ReplicaLag** to detect read replica staleness or replication pressure.

Example metric query for Aurora storage usage:

```bash
aws cloudwatch get-metric-statistics \
  --namespace AWS/RDS \
  --metric-name VolumeBytesUsed \
  --dimensions Name=DBClusterIdentifier,Value=my-aurora-cluster \
  --start-time 2026-04-25T00:00:00Z \
  --end-time 2026-04-26T00:00:00Z \
  --period 300 \
  --statistics Maximum
```

Why `Maximum` can be useful:
- it helps you see peak storage usage during the period,
- it avoids understating short-lived spikes.

Aurora-specific operational note:
- storage growth is usually automatic, but you still need to watch for abnormal growth patterns, replication lag, and failover readiness.

**Aurora failover and health indicators**

Aurora failover is usually faster than traditional RDS Multi-AZ failover, but it still affects availability. Monitor both the cluster writer role and reader health.

Watch for:
- writer promotion events,
- reader lag,
- connection drops during failover,
- cluster endpoint behavior,
- instance status transitions.

A healthy Aurora deployment should show:
- a reachable writer endpoint,
- at least one healthy reader if configured,
- acceptable replica lag,
- stable storage growth.

**Practical monitoring workflow**

1. Check the DB instance or cluster status in the RDS console or CLI.
2. Review CloudWatch metrics for CPU, memory, connections, latency, and storage.
3. Inspect RDS events for failover, maintenance, or autoscaling activity.
4. Use Enhanced Monitoring if the issue looks host-level.
5. Use Performance Insights if the issue looks query- or wait-related.
6. Review exported logs for engine-specific errors or slow queries.

Example status check:

```bash
aws rds describe-db-instances \
  --db-instance-identifier mydb \
  --query 'DBInstances[0].{Status:DBInstanceStatus,Class:DBInstanceClass,Storage:AllocatedStorage,FreeStorage:StorageThroughput}'
```

You should verify the actual output fields available in your environment; the key idea is to inspect status, storage, and configuration together.

**Exam-relevant distinctions**

- **CloudWatch**: metrics and alarms for trends and thresholds.
- **RDS events**: notifications for state changes and operational actions.
- **Enhanced Monitoring**: OS-level host detail.
- **Performance Insights**: SQL and wait analysis.
- **Logs export**: engine logs in CloudWatch Logs.
- **Standard RDS storage**: watch `FreeStorageSpace`.
- **Aurora storage**: watch `VolumeBytesUsed` and cluster behavior.

A common exam trap is confusing standard RDS instance storage with Aurora cluster storage. They are monitored differently, and the most important metrics are not identical.

**Troubleshooting patterns**

- If the database is slow and storage is low, check free space, log growth, and I/O latency.
- If the database is slow but storage is healthy, check CPU, memory, connections, and top SQL.
- If readers lag, check replication pressure and workload imbalance.
- If the instance is unavailable, check events for failover, maintenance, or storage exhaustion.
- If autoscaling occurred, confirm the new allocated size and investigate why growth happened.

**Quick reference**

| Area | What to watch | Why it matters |
|---|---|---|
| Health | DB instance status, events, failover state | Confirms availability and operational state |
| Storage | `FreeStorageSpace`, `VolumeBytesUsed` | Detects exhaustion and growth trends |
| Performance | CPU, memory, latency, connections | Identifies resource bottlenecks |
| Replication | `ReplicaLag` | Shows read replica freshness |
| Deep diagnostics | Enhanced Monitoring, Performance Insights | Explains host and SQL behavior |

**Key takeaways**

- Monitor both **health** and **storage**; they are related but not the same.
- Use CloudWatch alarms for early warning and SNS for notifications.
- Enable Enhanced Monitoring when you need OS-level detail.
- Use Performance Insights when the problem is query load or wait events.
- For standard RDS, `FreeStorageSpace` is the primary storage metric.
- For Aurora, focus on cluster volume metrics such as `VolumeBytesUsed` plus instance-level health and `ReplicaLag`.
- Storage autoscaling prevents immediate exhaustion, but you still need capacity planning and alerting.
