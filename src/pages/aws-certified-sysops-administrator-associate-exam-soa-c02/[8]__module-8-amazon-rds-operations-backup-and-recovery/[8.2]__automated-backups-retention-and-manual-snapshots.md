## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Backup concepts you must separate**

AWS exam questions often mix up three related but different backup models:

- **Automated backups** are service-managed and usually tied to a retention window.
- **Manual snapshots** are user-created point-in-time copies that persist until you delete them.
- **AWS Backup recovery points** are backup artifacts stored in a **backup vault** and governed by a **backup plan** and lifecycle rules.

The key exam skill is knowing what is restorable, how long it is retained, and whether you can restore to a point in time or only to a specific saved copy.

**Automated backups vs snapshots vs recovery points**

A good mental model is:

- **Automated backups**
  - Created by the service on a schedule.
  - Retained for a configured period.
  - Often support point-in-time recovery within the retention window.
  - May be deleted or retained differently when the source resource is deleted, depending on the service.
- **Manual snapshots**
  - Created on demand.
  - Persist until you delete them.
  - Restore from a specific saved state, not a continuous time range.
- **AWS Backup recovery points**
  - Represent a backup stored in a vault.
  - Can be copied across accounts and Regions if configured.
  - Can transition to colder storage where supported by the service and backup type.

> For exam purposes, do not assume every service supports the same restore model. Some support point-in-time recovery, while others are snapshot-based only.

**Amazon RDS automated backups and manual snapshots**

Amazon Relational Database Service (RDS) is a common exam target because it supports both automated backups and manual snapshots, but they behave differently.

- **Automated backups**
  - Include transaction logs plus daily snapshots for supported engines.
  - Enable point-in-time restore within the retention window.
  - Retention is configurable, commonly from 0 to 35 days.
  - If you delete the DB instance, you can choose to retain automated backups in some cases, which preserves restore capability for the retention period.
- **Manual snapshots**
  - Are user-initiated and kept until you delete them.
  - Are useful for long-term retention, change windows, and pre-upgrade checkpoints.
  - Can be copied to another Region or account, subject to encryption and KMS permissions.

RDS automated backups are not the same as a manual snapshot. Automated backups are a combination of snapshot data and logs that let you restore to a specific time. Manual snapshots restore only to the exact saved snapshot state.

**RDS retention and deletion behavior**

Retention behavior is a common trap.

- If automated backups are enabled, RDS keeps them for the configured retention period.
- If you delete the DB instance, you may be able to retain automated backups for later restore, depending on how deletion is performed and the engine/service behavior.
- If automated backups are not retained after deletion, you lose point-in-time restore capability once the instance is gone.
- Manual snapshots remain available after the source DB instance is deleted.

A practical exam interpretation:

- Use **automated backups** when you need recovery from recent accidental changes.
- Use **manual snapshots** when you need a durable restore point beyond the automated retention window.

**RDS restore examples**

Point-in-time restore is the right choice when you need to recover to a moment just before a bad deployment.

```bash
aws rds restore-db-instance-to-point-in-time \
  --source-db-instance-identifier prod-db \
  --target-db-instance-identifier prod-db-restore \
  --restore-time 2026-04-26T14:15:00Z
```

Restoring from a manual snapshot is the right choice when you want a fixed checkpoint.

```bash
aws rds restore-db-instance-from-db-snapshot \
  --db-instance-identifier prod-db-restore \
  --db-snapshot-identifier prod-db-snapshot-2026-04-26
```

**Aurora backup behavior**

Amazon Aurora has its own backup semantics that are often tested alongside RDS.

- Aurora supports **continuous backups** to enable point-in-time recovery within the retention window.
- Aurora also supports **manual snapshots** that persist until deleted.
- Aurora cluster snapshots are cluster-level artifacts, not the same as instance-level backups in standard RDS.
- Restores often create a new cluster, which is important when planning cutovers or recovery drills.

Aurora questions often test whether you understand that the restore target is typically a new cluster, not an in-place overwrite.

**DynamoDB point-in-time recovery**

Amazon DynamoDB is different from RDS because it does not use traditional snapshots in the same way for its primary backup model.

- **Point-in-time recovery (PITR)** lets you restore a table to any second in the last 35 days.
- PITR is continuous and designed for accidental deletes or bad writes.
- You restore to a new table rather than overwriting the original table.
- On-demand backups are separate from PITR and behave more like saved backup artifacts.

Use PITR when you need fine-grained recovery from recent data loss. Use on-demand backups when you want a named backup you can keep and restore later.

```bash
aws dynamodb restore-table-to-point-in-time \
  --source-table-name Orders \
  --target-table-name Orders-Restore \
  --restore-date-time 2026-04-26T13:00:00Z
```

**Amazon FSx backup behavior**

Amazon FSx services support backups, but the exact behavior depends on the FSx type.

- FSx backups are typically managed as service-native backup objects.
- Retention and restore semantics vary by FSx flavor.
- Some FSx deployments support automatic daily backups and manual backups.
- Restores usually create a new file system or volume rather than modifying the original in place.

For exam questions, focus on the service-specific restore target and whether the backup is automatic, manual, or policy-driven.

**EBS snapshots and consistency**

Amazon Elastic Block Store (EBS) snapshots are incremental backups of volumes stored in Amazon S3 behind the scenes.

- EBS snapshots are **crash-consistent** by default.
- Crash-consistent means the volume is captured as if power was lost at that moment.
- For many workloads, crash consistency is acceptable.
- For application consistency, you should quiesce the filesystem or application before taking the snapshot.
- On supported Windows workloads, AWS Backup can use **Volume Shadow Copy Service (VSS)** to help create application-consistent backups.

A common exam distinction:

- **Crash-consistent**: good for many Linux filesystems and general recovery.
- **Application-consistent**: better for databases and transactional systems because in-memory writes are flushed before capture.

**EBS snapshot lifecycle and restore**

EBS snapshots persist until you delete them, which makes them useful for long-term retention and cross-account workflows.

- Snapshots are incremental after the first full snapshot.
- You can restore a snapshot by creating a new volume.
- You can copy snapshots across Regions.
- You can share unencrypted snapshots with other accounts.
- Encrypted snapshots require careful KMS planning.

Example restore workflow:

```bash
aws ec2 create-volume \
  --availability-zone us-east-1a \
  --snapshot-id snap-0123456789abcdef0 \
  --volume-type gp3
```

**Encrypted EBS snapshot sharing**

Encrypted snapshot sharing is a frequent exam trap.

- You cannot simply share an encrypted snapshot and expect another account to use it.
- The destination account must be allowed to use the **AWS Key Management Service (KMS)** key that encrypted the snapshot, or you must copy the snapshot using a key the destination can access.
- In practice, the source account often shares the snapshot and the KMS key permissions, and the recipient copies the snapshot into their own account using a key they control.
- The recipient usually restores from the copied snapshot, not from the original shared encrypted snapshot.

> If the KMS key policy or grants do not allow access, the snapshot is effectively unusable to the other account.

**AWS Backup core objects**

AWS Backup centralizes backup management across supported services. Know the main objects:

- **Backup plan**
  - Defines when backups run and how long they are retained.
  - Can include lifecycle rules.
- **Backup selection**
  - Defines which resources are protected.
  - Can be based on tags or explicit resource ARNs.
- **Backup vault**
  - Logical container for recovery points.
  - Can be protected with access policies and encryption.
- **Recovery point**
  - The actual backup artifact created by AWS Backup.
  - Can be restored, copied, or transitioned according to policy and service support.

A backup plan does not itself store data. It schedules and governs creation of recovery points in a vault.

**AWS Backup lifecycle and retention**

AWS Backup retention is policy-driven.

- You define how long a recovery point stays in the warm tier.
- Where supported, you can transition recovery points to colder storage.
- Not every service or backup type supports every lifecycle option.
- Retention rules are applied per backup plan and can differ by rule.

A practical example of a backup plan JSON:

```json
{
  "BackupPlanName": "daily-prod-backups",
  "Rules": [
    {
      "RuleName": "daily",
      "TargetBackupVaultName": "PrimaryVault",
      "ScheduleExpression": "cron(0 5 * * ? *)",
      "StartWindowMinutes": 60,
      "CompletionWindowMinutes": 180,
      "Lifecycle": {
        "DeleteAfterDays": 35
      }
    }
  ]
}
```

Create the plan with:

```bash
aws backup create-backup-plan \
  --backup-plan file://backup-plan.json
```

**AWS Backup restore implications**

AWS Backup is useful when you want centralized governance, but restore behavior still depends on the underlying service.

- Some resources restore to a new resource.
- Some support point-in-time recovery semantics through AWS Backup integration.
- Some are snapshot-based and restore from a recovery point only.
- Cross-account and cross-Region copy are common reasons to use AWS Backup instead of service-native backups alone.

Think of AWS Backup as the control plane for backup policy, not as a replacement for every service’s native restore model.

**Service comparison for exam questions**

| Service | Primary backup style | Retention behavior | Restore style |
|---|---|---|---|
| RDS | Automated backups and manual snapshots | Automated backups expire by retention; manual snapshots persist until deleted | Point-in-time or snapshot restore |
| Aurora | Continuous backups and snapshots | Continuous backups follow retention; snapshots persist until deleted | Point-in-time or cluster snapshot restore |
| DynamoDB | PITR and on-demand backups | PITR limited to 35 days; on-demand backups persist until deleted | Restore to new table |
| EBS | Snapshots | Snapshots persist until deleted | Create new volume from snapshot |
| FSx | Service-specific backups | Depends on FSx type and policy | Restore to new file system or volume |

**How to choose the right backup type**

Use the backup type that matches your recovery objective:

- Choose **automated backups** when you need recent recovery with minimal operational overhead.
- Choose **manual snapshots** when you need a durable checkpoint before a change.
- Choose **PITR** when you need recovery to a precise time within a retention window.
- Choose **AWS Backup** when you need centralized policy, cross-account governance, or multi-service backup management.

A simple decision rule:

1. If you need to recover from a mistake made minutes ago, use point-in-time recovery if the service supports it.
2. If you need a known-good checkpoint before a migration or patch, use a manual snapshot or on-demand backup.
3. If you need organization-wide backup policy and copy controls, use AWS Backup.

**Practical scenario**

Imagine you patch a production database and discover corruption two hours later.

- If the service supports PITR and the event is within the retention window, restore to a time just before the patch.
- If PITR is unavailable or the retention window expired, restore from the latest manual snapshot or on-demand backup.
- If the database was protected by AWS Backup, you may restore from the relevant recovery point in the vault, subject to the service’s restore semantics.

This is why exam questions often ask not just “Can you back it up?” but “What can you restore, to where, and for how long?”

**Common exam traps**

- **Automated backup retention** is not the same as manual snapshot retention.
- **Crash-consistent** EBS snapshots are not automatically application-consistent.
- **Encrypted snapshot sharing** requires KMS permissions, not just snapshot sharing.
- **AWS Backup recovery points** are not the same as the backup plan or vault.
- **PITR** is time-based recovery, while snapshots are fixed restore points.
- **Not all services** support the same lifecycle or cold storage options.

**Operational commands you may see**

List RDS snapshots:

```bash
aws rds describe-db-snapshots \
  --db-instance-identifier prod-db
```

List EBS snapshots:

```bash
aws ec2 describe-snapshots \
  --owner-ids self
```

List AWS Backup recovery points:

```bash
aws backup list-recovery-points-by-backup-vault \
  --backup-vault-name PrimaryVault
```

These commands help you verify what exists before you attempt a restore.

**Key takeaways**

- Automated backups are time-bound and often support point-in-time recovery.
- Manual snapshots are durable restore points that you delete explicitly.
- AWS Backup adds centralized policy, vaults, selections, and recovery points.
- EBS snapshots are crash-consistent by default unless you take extra steps for application consistency.
- Encrypted snapshot sharing depends on KMS access, not just snapshot permissions.
- Service-specific behavior matters: RDS, Aurora, DynamoDB, EBS, and FSx do not all restore the same way.
