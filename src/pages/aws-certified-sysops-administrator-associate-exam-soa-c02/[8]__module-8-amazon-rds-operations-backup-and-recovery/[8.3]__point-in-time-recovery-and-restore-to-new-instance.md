## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Point-in-Time Recovery (PITR) overview**

Point-in-Time Recovery (PITR) lets you restore an Amazon Relational Database Service (Amazon RDS) database to a specific second within the backup retention window. You use it when you need to recover from accidental deletes, bad deployments, application bugs, or data corruption without overwriting the current production instance.

PITR works by combining:
- A **full automated backup** of the DB instance.
- A continuous stream of **transaction logs** that AWS retains during the backup retention period.

AWS restores the backup and replays logs up to the requested time. This gives you a database state just before the incident, as long as the time falls within the retention window.

**When to use PITR**

Use PITR when you need:
- Recovery from human error, such as a mistaken `DELETE` or schema change.
- Recovery from application-level corruption that happened recently.
- A safe way to inspect historical data without modifying the original instance.
- A rollback target for testing after a failed release.

PITR is different from a manual snapshot restore:
- A **manual snapshot** restores only to the snapshot time.
- PITR restores to a specific time between snapshots, with finer granularity.

**How PITR works in practice**

RDS maintains automated backups and transaction logs for supported engines. When you request a restore to a point in time, AWS:
- Selects the appropriate automated backup.
- Restores it into a **new DB instance**.
- Applies transaction logs up to the chosen timestamp.
- Brings the new instance online with a new endpoint.

> PITR does not overwrite the source DB instance. It always creates a new instance, which is important for safe recovery and validation.

**Restore to new instance behavior**

A point-in-time restore creates a separate DB instance rather than replacing the original. This is useful because you can:
- Validate the recovered data before cutover.
- Compare the restored instance with production.
- Redirect applications only after confirming correctness.
- Keep the original instance available for forensic analysis.

The restored instance gets:
- A new DB instance identifier.
- A new endpoint and port.
- Its own security group associations and parameter group settings, subject to restore options.
- A state that reflects the selected restore time.

**Typical restore workflow**

1. Identify the incident time.
2. Choose a restore time just before the issue occurred.
3. Restore the DB instance to a new instance.
4. Verify data consistency and application behavior.
5. Update application configuration or DNS to point to the restored instance if needed.

**AWS CLI example: restore to a specific time**

You can restore an RDS instance with the AWS CLI using `restore-db-instance-to-point-in-time`.

```bash
aws rds restore-db-instance-to-point-in-time \
  --source-db-instance-identifier prod-db \
  --target-db-instance-identifier prod-db-restore \
  --restore-time 2026-04-26T14:15:00Z \
  --db-instance-class db.m6g.large \
  --publicly-accessible \
  --no-multi-az
```

Key points:
- `--source-db-instance-identifier` is the original DB instance.
- `--target-db-instance-identifier` is the new DB instance name.
- `--restore-time` must be within the automated backup retention window.
- You can choose a different instance class if needed, subject to engine support.
- The restore creates a new instance; it does not modify the source.

**Restore to latest restorable time**

If you want the most recent recoverable state, restore to the latest restorable time instead of specifying a timestamp.

```bash
aws rds restore-db-instance-to-point-in-time \
  --source-db-instance-identifier prod-db \
  --target-db-instance-identifier prod-db-latest \
  --use-latest-restorable-time
```

This is useful when:
- You need the newest possible recovery point.
- You do not know the exact incident timestamp.
- You want to minimize data loss after an outage.

**Console restore flow**

In the Amazon RDS console, the general flow is:
1. Open the source DB instance.
2. Choose **Actions**.
3. Select **Restore to point in time**.
4. Choose either a specific time or the latest restorable time.
5. Enter a new DB instance identifier.
6. Review networking, security, and instance class settings.
7. Start the restore.

The restored instance appears as a separate resource and must be validated before use.

**Important restore characteristics**

- **Backup retention window** controls how far back you can restore.
- **Automated backups** must be enabled for PITR.
- **Transaction logs** are required to restore between backup points.
- **New instance creation** means you need to plan for endpoint changes.
- **Downtime avoidance** is possible because the original instance stays available during validation.

**Common exam-relevant distinctions**

| Concept | What it does | Key limitation |
|---|---|---|
| Automated backup restore | Restores from a backup set | Less granular than PITR |
| Manual snapshot restore | Restores to snapshot time only | No log replay to arbitrary time |
| Point-in-Time Recovery | Restores to a specific second | Limited to retention window |
| Restore to new instance | Creates a separate DB instance | Requires application cutover |

**Operational considerations**

After restore, you often need to handle:
- **Endpoint updates** in application configuration.
- **DNS changes** if the app uses a CNAME or Route 53 record.
- **Secrets rotation** if credentials differ from the original setup.
- **Parameter group review** to ensure the restored instance behaves as expected.
- **Security group checks** so the application can connect.

If the restored instance is for testing, you may want to:
- Disable public access.
- Place it in a non-production subnet group.
- Restrict inbound access to only your admin hosts.

**Example: validate the restored instance**

```bash
aws rds describe-db-instances \
  --db-instance-identifier prod-db-restore \
  --query 'DBInstances[0].{Status:DBInstanceStatus,Endpoint:Endpoint.Address,Class:DBInstanceClass}'
```

You can also connect with a client to confirm the recovered data:

```bash
mysql -h prod-db-restore.xxxxxx.us-east-1.rds.amazonaws.com -u admin -p
```

Or for PostgreSQL:

```bash
psql "host=prod-db-restore.xxxxxx.us-east-1.rds.amazonaws.com port=5432 dbname=appdb user=admin sslmode=require"
```

**Practical recovery pattern**

A safe production recovery pattern is:
- Restore to a new instance.
- Run application smoke tests.
- Compare row counts, recent transactions, and critical tables.
- Promote the restored instance by updating the application endpoint.
- Keep the original instance until you confirm the recovery is correct.

This pattern reduces the risk of making a bad recovery decision under pressure.

**What PITR does not do**

PITR does not:
- Restore directly into the original instance in place.
- Recover data outside the backup retention period.
- Guarantee recovery from logical corruption if the corruption existed before the chosen restore time.
- Replace the need for application-level backups or export strategies in all cases.

**Troubleshooting and edge cases**

If PITR is unavailable or fails, check:
- Automated backups are enabled.
- The requested restore time is within the retention window.
- The DB instance is in a supported state.
- You have enough quota and capacity for a new instance.
- The target identifier is unique.

If the restored database is missing expected data, consider:
- Whether the chosen restore time was too early.
- Whether the corruption happened before the selected timestamp.
- Whether the application continued writing bad data after the incident.

**Best practices**

- Keep automated backups enabled on production databases.
- Set a retention period that matches your recovery objectives.
- Practice restore procedures before an incident.
- Document the application cutover process for restored instances.
- Use least-privilege access to the restored database during validation.
- Test both the restore and the application reconnect path, not just the database startup.

**Exam focus**

For the SysOps Administrator exam, remember:
- PITR restores to a **specific time** within the retention window.
- The restore creates a **new DB instance**.
- It relies on **automated backups plus transaction logs**.
- It is commonly used for **accidental deletion** or **recent corruption** recovery.
- You must plan for **endpoint and application cutover** after restore.
