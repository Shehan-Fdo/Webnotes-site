## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**RDS Multi-AZ overview**

Amazon Relational Database Service (Amazon RDS) Multi-AZ is a high availability feature that protects a database from infrastructure failure by placing redundant resources in separate Availability Zones (AZs). For exam purposes, you should distinguish between the older **Multi-AZ DB instance** deployment model and the newer **Multi-AZ DB cluster** model, because they differ in topology, read access, and failover behavior.

Multi-AZ is about availability, not scaling reads. If you need read scaling, you usually add read replicas or use a Multi-AZ DB cluster reader endpoint where supported. If you need automatic failover with minimal disruption, Multi-AZ is the primary RDS pattern.

- **Availability goal**: keep the database online through AZ-level failure, storage failure, or instance failure.
- **Primary use case**: production workloads that need higher resilience and predictable failover.
- **Not a backup replacement**: Multi-AZ improves availability, but you still need backups and point-in-time recovery.

**Multi-AZ DB instance vs Multi-AZ DB cluster**

The exam often tests whether you know which architecture provides a readable standby, how many instances exist, and how failover works.

| Feature | Multi-AZ DB instance | Multi-AZ DB cluster |
|---|---|---|
| Instances | One primary + one standby | One writer + two readers |
| Read access | Standby is not readable | Reader instances are readable |
| Endpoints | Single database endpoint | Writer endpoint and reader endpoint |
| Replication model | Synchronous storage replication across AZs | Synchronous replication across AZs with a cluster architecture |
| Failover | Promotes standby to primary | Promotes a reader to writer |
| Typical intent | High availability for a single writable instance | High availability plus read scaling and faster failover characteristics |
| Backup source | Automated backups are taken from the primary/standby-managed storage without impacting the active writer in the same way as a single-AZ design | Backups are managed across the cluster architecture |

A classic Multi-AZ DB instance is the older and more common exam reference. A Multi-AZ DB cluster is a newer architecture with a dedicated writer endpoint and reader endpoint, and the reader instances are intended to be read-capable. Do not describe the reader as “maybe readable”; in this architecture, it is readable by design.

**How Multi-AZ works**

In a classic Multi-AZ DB instance deployment, RDS maintains a synchronous standby in another AZ. The standby is not used for application reads. RDS automatically handles failover by switching the DNS endpoint to the standby when the primary becomes unavailable or when a maintenance event requires it.

Important operational points:

- **Synchronous storage replication** keeps the standby storage aligned with the primary.
- **Commit behavior** can vary by engine and deployment mode, so avoid oversimplifying it as a universal “write is acknowledged only after standby acknowledgment” rule.
- **Automatic failover** updates the database endpoint to point to the new primary.
- **DNS caching** in clients can delay recovery if applications cache the old endpoint too aggressively.

For Multi-AZ DB clusters, the architecture includes a writer and reader instances across AZs. The cluster provides separate endpoints for write and read traffic, which makes it more suitable when you want high availability and a built-in read path without adding separate read replicas.

**Operational behavior and failover**

Failover is the core operational event you must understand for the exam. RDS detects a failure, promotes a standby or reader, and updates the endpoint mapping. The application sees a brief interruption, then reconnects to the new primary.

Typical failover triggers include:

- Primary instance failure
- AZ impairment
- Planned maintenance
- Storage-related issues
- Manual failover initiated by an administrator

What happens during failover:

1. RDS detects the failure or maintenance event.
2. A standby or reader is promoted to writer.
3. The database endpoint is remapped.
4. Existing connections drop and must reconnect.
5. The application resumes after retry logic succeeds.

> Failover is not zero-downtime. Design for brief connection loss and automatic retry.

**Backups, snapshots, and recovery**

Multi-AZ is often confused with backup strategy. They solve different problems.

- **Automated backups** enable point-in-time recovery (PITR), which lets you restore to a specific time within the retention window.
- **Manual snapshots** are user-managed restore points.
- **Multi-AZ** protects availability, not historical recovery.

Operationally, RDS can take backups from the standby in classic Multi-AZ deployments to reduce impact on the primary. This is an important exam concept because it explains why Multi-AZ can improve backup operations without making backups the same as HA.

Key backup facts:

- Automated backups are enabled per DB instance or cluster.
- Retention can be configured from 0 to 35 days for many engines.
- Deleting a DB instance can optionally retain the final snapshot.
- Restoring from backup creates a new DB instance or cluster; it does not “rewind” the existing one.

**Engine and edition considerations**

Multi-AZ support depends on engine, edition, and deployment model. In scenario questions, check whether the engine supports the requested architecture before assuming it is available.

Common guidance:

- **Amazon Aurora** uses its own cluster architecture and is not the same as classic RDS Multi-AZ.
- **MySQL, PostgreSQL, MariaDB, Oracle, and SQL Server** support Multi-AZ in RDS, but exact options vary by engine and edition.
- **SQL Server** has edition-specific limitations; not every deployment option is available in every edition.
- **Older or specialized engine versions** may not support the newer Multi-AZ DB cluster model.

When the question says “high availability for a standard RDS database,” think Multi-AZ DB instance first. When it says “reader endpoint,” “writer endpoint,” or “built-in read-capable standby,” think Multi-AZ DB cluster.

**Connection management and RDS Proxy**

Failover causes connection drops, so application connection handling matters. RDS Proxy is often the best answer when the question asks how to reduce failover disruption for applications with many short-lived connections.

Why use RDS Proxy:

- It pools and reuses database connections.
- It reduces connection storms after failover.
- It helps applications reconnect faster during failover events.
- It can improve resilience for Lambda and microservice workloads.

Practical pattern:

- Put the application behind RDS Proxy.
- Point the application to the proxy endpoint instead of the DB endpoint.
- Let the proxy manage backend database connections during failover.

Example architecture:

```text
Application -> RDS Proxy -> RDS Multi-AZ DB instance or Multi-AZ DB cluster
```

RDS Proxy does not replace Multi-AZ. It complements it by reducing the application impact of failover.

**CLI examples**

Use the AWS CLI to inspect and manage Multi-AZ settings. Be precise about which command applies to which deployment model.

Describe a DB instance and check Multi-AZ status:

```bash
aws rds describe-db-instances \
  --db-instance-identifier mydb \
  --query 'DBInstances[0].{MultiAZ:MultiAZ,Engine:Engine,Status:DBInstanceStatus,AZ:AvailabilityZone,SecondaryAZ:SecondaryAvailabilityZone}'
```

Describe a DB cluster and inspect endpoints:

```bash
aws rds describe-db-clusters \
  --db-cluster-identifier mycluster \
  --query 'DBClusters[0].{Engine:Engine,Status:Status,WriterEndpoint:Endpoint,ReaderEndpoint:ReaderEndpoint,MultiAZ:MultiAZ}'
```

Modify a classic DB instance to enable Multi-AZ:

```bash
aws rds modify-db-instance \
  --db-instance-identifier mydb \
  --multi-az \
  --apply-immediately
```

> `--apply-immediately` can cause an immediate change window. In production, you often prefer a maintenance window unless the change is urgent.

Test failover for a supported classic Multi-AZ DB instance:

```bash
aws rds reboot-db-instance \
  --db-instance-identifier mydb \
  --force-failover
```

This command is for supported Multi-AZ DB instance failover testing. It is not a universal failover command for every RDS deployment type.

Create a Multi-AZ DB cluster from a supported engine and parameter set:

```bash
aws rds create-db-cluster \
  --db-cluster-identifier mycluster \
  --engine mysql \
  --master-username admin \
  --master-user-password 'ChangeMe123!' \
  --allocated-storage 100 \
  --storage-type io1 \
  --multi-az
```

The exact parameters vary by engine and current RDS feature support, so verify the engine-specific requirements before using this pattern in production.

**Console workflow**

In the AWS Management Console, Multi-AZ configuration is usually part of the database creation or modification flow.

Typical steps for a classic Multi-AZ DB instance:

1. Open the RDS console and choose **Create database** or **Modify**.
2. Select the engine and edition.
3. In **Availability and durability**, enable **Multi-AZ deployment**.
4. Review storage, backup retention, and maintenance window settings.
5. Create or apply the change.

Typical steps for a Multi-AZ DB cluster:

1. Open the RDS console and choose **Create database**.
2. Select a supported engine and the Multi-AZ DB cluster option.
3. Configure the writer and reader capacity or instance settings.
4. Confirm the writer endpoint and reader endpoint behavior.
5. Review networking, security groups, and backup settings.
6. Create the cluster and verify the instances appear in the cluster view.

Useful console signals:

- **Available** means the DB is ready for connections.
- **Modifying** often appears during scaling or configuration changes.
- **Failing over** indicates an active promotion event.
- **Backing up** can appear during snapshot or automated backup activity.
- **Maintenance** may indicate a scheduled patch or infrastructure update.

**Events, status, and troubleshooting**

RDS events are important for operational awareness and exam questions. You should know where to look when a failover or maintenance event occurs.

Common places to inspect:

- **RDS console events** for recent failover, backup, and maintenance messages
- **DB instance or cluster status** for current lifecycle state
- **CloudWatch metrics** for performance symptoms
- **Application logs** for connection resets and retry behavior

Useful event categories include:

- Failover started or completed
- Backup started or completed
- Maintenance scheduled or applied
- Storage or host replacement
- Parameter group changes requiring reboot

CloudWatch metrics can help you distinguish a failover from a performance issue:

- **CPUUtilization** for compute pressure
- **FreeStorageSpace** for storage exhaustion
- **DatabaseConnections** for connection spikes or drops
- **ReadIOPS** and **WriteIOPS** for workload patterns
- **FreeableMemory** for memory pressure

For read replicas, `ReplicaLag` is a useful metric. Do not use `ReplicaLag` as a general Multi-AZ metric for classic Multi-AZ deployments, because it is read-replica-specific.

**Practical monitoring and alerting**

Set alarms around availability and failover symptoms rather than only raw performance.

Recommended alarms:

- DB instance or cluster status changes
- Elevated `DatabaseConnections` with repeated connection failures
- Low `FreeStorageSpace`
- High `CPUUtilization` during failover recovery
- Application-level health checks failing after endpoint remap

Example CloudWatch alarm creation:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name rds-mydb-low-free-storage \
  --metric-name FreeStorageSpace \
  --namespace AWS/RDS \
  --statistic Average \
  --period 300 \
  --threshold 21474836480 \
  --comparison-operator LessThanThreshold \
  --dimensions Name=DBInstanceIdentifier,Value=mydb \
  --evaluation-periods 2
```

**Performance and latency considerations**

Multi-AZ improves resilience, but it can add operational overhead.

- **Write latency** may increase because the system must maintain synchronous durability across AZs.
- **Read performance** does not improve in classic Multi-AZ DB instances because the standby is not readable.
- **Failover recovery time** depends on engine, workload, client retry logic, and DNS behavior.
- **Storage and network design** matter because cross-AZ traffic is part of the architecture.

For exam scenarios, choose Multi-AZ when the requirement is high availability and automatic failover, not when the requirement is read scaling or lowest possible write latency.

**When to choose each option**

Use this decision logic:

- Choose **classic Multi-AZ DB instance** when you need a straightforward highly available RDS database with one writer and one standby.
- Choose **Multi-AZ DB cluster** when you need a writer plus readable readers with a built-in reader endpoint and stronger read availability characteristics.
- Choose **read replicas** when you need asynchronous read scaling, cross-region replication, or reporting workloads.
- Choose **RDS Proxy** when connection management and failover resilience are important for the application layer.
- Choose **backups and snapshots** when you need recovery to a point in time or a specific restore point.

**Exam traps and common mistakes**

- **Confusing Multi-AZ with read replicas**: Multi-AZ is for availability; read replicas are for read scaling and replication use cases.
- **Assuming the standby is readable**: that is false for classic Multi-AZ DB instances.
- **Assuming all engines support all Multi-AZ modes**: support varies by engine and edition.
- **Treating backups as HA**: backups help recovery, not uptime.
- **Using the wrong metric**: `ReplicaLag` is not a classic Multi-AZ metric.
- **Ignoring client reconnect behavior**: failover is only useful if the application retries correctly.

**Quick reference**

- **Classic Multi-AZ DB instance**: one primary, one standby, automatic failover, standby not readable.
- **Multi-AZ DB cluster**: one writer, two readers, writer and reader endpoints, readers are readable.
- **RDS Proxy**: reduces connection disruption during failover.
- **Automated backups**: support point-in-time recovery.
- **Manual snapshots**: user-controlled restore points.
- **CloudWatch and RDS events**: primary tools for operational visibility.

**Practical scenario examples**

If an application requires minimal downtime during AZ failure and does not need read scaling, choose a classic Multi-AZ DB instance.

If a workload needs high availability and a readable secondary path for read traffic, choose a Multi-AZ DB cluster where supported.

If a developer asks for “a standby that can serve reads,” do not answer with classic Multi-AZ DB instance. That requirement points to a Multi-AZ DB cluster or read replica architecture, depending on the exact use case.

If the question asks how to reduce failover impact on a Lambda-based application with many connections, the best answer is often RDS Proxy in front of Multi-AZ.
