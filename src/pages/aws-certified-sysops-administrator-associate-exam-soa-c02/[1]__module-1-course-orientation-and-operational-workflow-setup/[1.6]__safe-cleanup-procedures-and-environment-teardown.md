## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Safe cleanup mindset**

Safe cleanup is the controlled removal of AWS resources after testing, migration, incident response, or project completion. The goal is to stop unnecessary spend without deleting data, breaking shared services, or leaving behind security exposure. In practice, you verify dependencies first, preserve required artifacts, then delete resources in the correct order.

A good teardown process is repeatable. You should be able to answer three questions before deleting anything:

- What depends on this resource?
- What data must be retained?
- What can be deleted immediately versus later?

For exam purposes, focus on the operational discipline: identify dependencies, protect stateful data, remove resources from the outside in, and confirm that billing and monitoring reflect the cleanup.

**Why cleanup matters**

AWS resources often continue to incur charges even when they are idle. Common examples include:

- **Elastic IP addresses** that are allocated but not associated
- **EBS volumes** that remain after instance termination
- **NAT gateways** that bill by the hour and by data processed
- **Load balancers** that continue running until deleted
- **Snapshots** and **AMIs** that persist as stored backups
- **CloudWatch logs** and retained metrics that accumulate over time

Cleanup also reduces risk:

- Orphaned security groups can hide stale rules and make audits harder.
- Unused IAM roles, access keys, and instance profiles increase attack surface.
- Forgotten public endpoints can expose test systems after a project ends.

**Plan the teardown**

Before deleting resources, inventory what exists and map dependencies. Use tags, naming conventions, and AWS resource discovery tools to identify the environment boundary.

Useful discovery commands:

```bash
aws resourcegroupstaggingapi get-resources \
  --tag-filters Key=Environment,Values=dev \
  --output table
```

```bash
aws ec2 describe-instances \
  --filters "Name=tag:Environment,Values=dev" \
  --query "Reservations[].Instances[].{Id:InstanceId,State:State.Name,Type:InstanceType}" \
  --output table
```

```bash
aws elbv2 describe-load-balancers --output table
aws rds describe-db-instances --output table
aws ecs list-clusters --output table
```

Practical planning steps:

1. Identify the environment scope, such as a dev, test, or temporary lab account.
2. List all tagged resources and confirm whether any shared services are in use.
3. Check for cross-stack or cross-account dependencies, especially DNS, IAM, and networking.
4. Decide what must be archived, exported, or snapshotted before deletion.
5. Define the deletion order and rollback point.

**Protect data before deletion**

Stateful services require special care. You usually preserve data first, then remove the compute or orchestration layer.

Common retention actions:

- Create an **RDS snapshot** before deleting a database.
- Create an **EBS snapshot** before deleting a volume if the data is needed later.
- Export logs from **CloudWatch Logs** to S3 if they must be retained.
- Save configuration files, secrets references, and deployment manifests.
- Capture AMIs if you need to recreate a known-good instance image.

Examples:

```bash
aws rds create-db-snapshot \
  --db-instance-identifier app-db \
  --db-snapshot-identifier app-db-pre-teardown
```

```bash
aws ec2 create-snapshot \
  --volume-id vol-0123456789abcdef0 \
  --description "Pre-teardown snapshot for app data"
```

```bash
aws logs create-export-task \
  --log-group-name /aws/lambda/app-prod \
  --from 1714000000000 \
  --to 1714086400000 \
  --destination my-log-archive-bucket \
  --destination-prefix exports/app-prod
```

> Delete only after you confirm the backup is complete, accessible, and tested enough for your recovery needs.

**Delete in the correct order**

Teardown usually works best from dependent resources outward to foundational resources. If you delete the foundation first, dependent resources may fail to clean up or may become harder to identify.

A typical order is:

1. Stop or scale down application workloads.
2. Remove autoscaling and orchestration attachments.
3. Delete load balancers, target groups, and listeners.
4. Delete application data stores only after snapshots are taken.
5. Remove networking components such as NAT gateways, route tables, and subnets.
6. Delete security groups, IAM roles, and policies that are no longer referenced.
7. Release public IPs and delete DNS records.
8. Remove monitoring artifacts, alarms, dashboards, and log groups if retention is complete.

Examples of common cleanup commands:

```bash
aws ec2 disassociate-address --association-id eipassoc-0123456789abcdef0
aws ec2 release-address --allocation-id eipalloc-0123456789abcdef0
```

```bash
aws elbv2 delete-load-balancer --load-balancer-arn arn:aws:elasticloadbalancing:us-east-1:123456789012:loadbalancer/app/my-alb/50dc6c495c0c9188
aws elbv2 delete-target-group --target-group-arn arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/my-tg/6d0ecf831eec9f09
```

```bash
aws ec2 delete-security-group --group-id sg-0123456789abcdef0
aws iam delete-role-policy --role-name app-role --policy-name inline-policy
aws iam delete-role --role-name app-role
```

**Common resource-specific considerations**

Different AWS services have different teardown behaviors. Knowing the service-specific gotchas helps you avoid orphaned resources and failed deletions.

- **EC2**
  - Terminating an instance does not always delete attached EBS volumes unless `DeleteOnTermination` is enabled.
  - Elastic IPs must be released separately.
  - Instance profiles and IAM roles may remain after instance termination.

- **EBS**
  - Volumes persist independently of instances.
  - Snapshots are incremental and continue to incur storage charges.
  - Delete only after confirming no recovery requirement remains.

- **RDS**
  - Deleting a DB instance can optionally create a final snapshot.
  - Automated backups may remain for a retention period.
  - Read replicas and Multi-AZ components must be removed according to dependency order.

- **VPC**
  - You cannot delete a VPC until dependent resources are removed.
  - Common blockers include NAT gateways, internet gateways, route tables, network ACLs, endpoints, and ENIs.
  - Security groups and subnets often fail deletion if still referenced.

- **S3**
  - Buckets must be emptied before deletion.
  - Versioned buckets require removal of all object versions and delete markers.
  - Lifecycle policies can help automate retention and expiration.

- **CloudFormation**
  - Stack deletion can fail if resources were created outside the stack or have retention policies.
  - `DeletionPolicy: Retain` preserves resources intentionally, so verify whether retained items are expected.
  - Nested stacks and stack sets may require coordinated teardown.

Example of a retention policy in CloudFormation:

```yaml
Resources:
  DatabaseSnapshot:
    Type: AWS::RDS::DBInstance
    DeletionPolicy: Snapshot
```

**Use tags and automation**

Tags make cleanup safer because they let you target only the resources that belong to a specific environment. A consistent tag set such as `Environment=dev`, `Project=alpha`, and `Owner=team-a` helps you avoid deleting shared infrastructure.

Example tag-based deletion workflow:

```bash
aws ec2 describe-instances \
  --filters "Name=tag:Environment,Values=dev" \
  --query "Reservations[].Instances[].InstanceId" \
  --output text
```

You can then script cleanup carefully:

```bash
for id in $(aws ec2 describe-instances \
  --filters "Name=tag:Environment,Values=dev" \
  --query "Reservations[].Instances[].InstanceId" \
  --output text); do
  aws ec2 terminate-instances --instance-ids "$id"
done
```

Automation options include:

- AWS CLI scripts for repeatable teardown
- Infrastructure as Code (IaC) deletion through CloudFormation, Terraform, or similar tools
- Lifecycle policies for S3 objects and log retention
- Scheduled cleanup jobs for temporary environments

Automation reduces human error, but you still need guardrails such as tag filters, dry runs, and approval steps.

**Validate before and after cleanup**

Verification is part of safe teardown. Before deletion, confirm that the resource is not in use. After deletion, confirm that the environment is actually gone and that billing-impacting items are released.

Checks to perform:

- Confirm no active connections or running workloads remain.
- Verify snapshots, exports, or backups exist and are readable.
- Confirm DNS records no longer point to removed endpoints.
- Check that Elastic IPs, NAT gateways, and load balancers are deleted.
- Review CloudWatch alarms and logs for any remaining references.
- Inspect the billing console or cost explorer for lingering spend.

Useful verification commands:

```bash
aws ec2 describe-addresses --output table
aws ec2 describe-nat-gateways --output table
aws ec2 describe-volumes --filters Name=status,Values=available --output table
aws s3 ls
```

For CloudFormation-managed environments:

```bash
aws cloudformation describe-stacks --stack-name my-stack
aws cloudformation list-stack-resources --stack-name my-stack
```

If a deletion fails, read the dependency error carefully. AWS usually tells you what still references the resource. Common fixes include detaching network interfaces, removing route table associations, deleting listeners before load balancers, or clearing bucket contents before bucket deletion.

**Security cleanup**

Teardown is also a security task. Remove credentials and access paths that were created for the environment.

Typical security cleanup items:

- Delete temporary IAM users, roles, and inline policies.
- Revoke or rotate access keys used for testing.
- Remove security group rules that opened temporary ports.
- Delete SSH keys, SSM Session Manager assumptions, or bastion access paths if they were environment-specific.
- Remove secrets from AWS Secrets Manager or confirm retention requirements before deletion.

Example IAM cleanup:

```bash
aws iam list-access-keys --user-name temp-user
aws iam delete-access-key --user-name temp-user --access-key-id AKIAEXAMPLE
aws iam delete-user --user-name temp-user
```

**Cost-control best practices**

A safe teardown process should also stop future charges quickly. Prioritize resources with high ongoing cost:

- NAT gateways
- Load balancers
- RDS instances and read replicas
- EBS volumes and snapshots
- Elastic IPs
- Data transfer-heavy services

Helpful habits:

- Set a clear end date for temporary environments.
- Use budgets and alerts for lab or test accounts.
- Apply lifecycle policies to logs and object storage.
- Prefer ephemeral infrastructure for short-lived testing.
- Review cost allocation tags regularly.

**Troubleshooting teardown failures**

Deletion failures usually come from hidden dependencies or service-specific constraints. When cleanup stalls, work backward from the error message.

Common causes:

- A subnet still contains an ENI (Elastic Network Interface).
- A security group is referenced by another security group or resource.
- A bucket still contains versioned objects.
- A load balancer still has listeners or target groups attached.
- A VPC still has an internet gateway, NAT gateway, or endpoint attached.
- An RDS instance still has replicas or pending final snapshot requirements.

A practical troubleshooting approach:

1. Read the exact AWS error message.
2. Identify the blocking dependency.
3. Remove or detach the dependent resource.
4. Retry deletion.
5. Recheck for orphaned resources after success.

**Exam focus**

For SOA-C02, expect questions that test whether you can choose the safest and most cost-effective cleanup action. The best answer often includes:

- Preserving data with snapshots or exports before deletion
- Deleting resources in dependency order
- Using tags to scope cleanup
- Releasing chargeable resources such as Elastic IPs and NAT gateways
- Confirming that retained resources are intentional

A strong operational answer is not simply “delete the stack.” It is “verify dependencies, preserve required data, remove dependent resources, then delete the environment and confirm no billable artifacts remain.”

**Quick teardown checklist**

- Inventory resources with tags and service discovery.
- Identify shared versus environment-specific components.
- Snapshot or export required data.
- Stop workloads and remove dependencies.
- Delete resources from dependent to foundational layers.
- Release public IPs and remove DNS records.
- Clean up IAM, security groups, and secrets.
- Verify deletion and check for remaining charges.
