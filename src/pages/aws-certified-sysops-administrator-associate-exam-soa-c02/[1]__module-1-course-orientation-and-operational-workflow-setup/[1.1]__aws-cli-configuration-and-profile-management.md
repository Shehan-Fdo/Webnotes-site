## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**AWS CLI configuration overview**

The AWS Command Line Interface (AWS CLI) uses a layered configuration model so you can separate credentials, regions, output preferences, and profile-specific settings. For exam purposes, focus on how the CLI resolves conflicts, how profiles are stored, and how to troubleshoot broken or inconsistent setups.

The CLI commonly reads from:

- **Command-line options** passed directly in the command.
- **Environment variables** such as `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, and `AWS_DEFAULT_REGION`.
- **Named profiles** in `~/.aws/config` and `~/.aws/credentials`.
- **Role-based or federated sources** such as assumed roles, AWS Identity Center, or credential processes.

A practical mental model is: the most specific setting wins. If you set a region on the command line, it overrides profile and environment values for that invocation.

**Core configuration files**

The AWS CLI stores most persistent settings in two files under your home directory:

- `~/.aws/config`
- `~/.aws/credentials`

The files serve different purposes:

- `~/.aws/credentials` stores long-term access keys and session credentials.
- `~/.aws/config` stores non-secret settings such as region, output format, pager behavior, and role assumptions.

A typical layout looks like this:

```ini
# ~/.aws/config
[default]
region = us-east-1
output = json

[profile dev]
region = us-west-2
output = table

[profile prod]
region = eu-west-1
cli_pager = less
```

```ini
# ~/.aws/credentials
[default]
aws_access_key_id = AKIAEXAMPLEDEFAULT
aws_secret_access_key = defaultSecretKeyExample

[dev]
aws_access_key_id = AKIAEXAMPLEDEV
aws_secret_access_key = devSecretKeyExample

[prod]
aws_access_key_id = AKIAEXAMPLEPROD
aws_secret_access_key = prodSecretKeyExample
```

Important details:

- The `default` profile is special and is used when you do not specify `--profile`.
- In `~/.aws/config`, named profiles use the `profile ` prefix, for example `[profile dev]`.
- In `~/.aws/credentials`, named profiles do **not** use the prefix, for example `[dev]`.
- A profile can exist in one file but not the other, although many real setups use both.

**Profile creation and editing**

You can create or update profiles interactively with `aws configure`. This is the most common way to set up a profile quickly.

```bash
aws configure
aws configure --profile dev
```

The second form is valid and commonly used to configure a named profile interactively. It writes values into the appropriate files for that profile.

Typical prompts include:

- AWS Access Key ID
- AWS Secret Access Key
- Default region name
- Default output format

For non-interactive updates, use `aws configure set`:

```bash
aws configure set region us-west-2 --profile dev
aws configure set output json --profile dev
aws configure set aws_access_key_id AKIAEXAMPLEDEV --profile dev
aws configure set aws_secret_access_key devSecretKeyExample --profile dev
```

Useful read operations:

```bash
aws configure get region --profile dev
aws configure list
aws configure list-profiles
```

Notes:

- `aws configure list` shows the effective values and where they came from, but the exact source labels can vary by CLI version and by setting.
- `aws configure list-profiles` lists configured profile names.
- The AWS CLI does **not** provide a standard `aws configure unset` command in v2. To remove a setting, edit the files manually or overwrite the value where appropriate.

**Credential precedence and conflict resolution**

When multiple sources define the same setting, the AWS CLI resolves them by precedence. For exam prep, remember this practical order:

1. Command-line parameters
2. Environment variables
3. Profile settings from `--profile` or `AWS_PROFILE`
4. Shared configuration and credentials files
5. Role-based or external credential sources used by the selected profile

For credentials specifically, the CLI may also use:

- `credential_process`
- Assume-role profiles with `role_arn`
- AWS Identity Center cached tokens
- EC2 or container metadata when running on AWS compute

A few examples:

- If you run `aws s3 ls --region ap-south-1`, that region overrides profile and environment region settings.
- If `AWS_DEFAULT_REGION=us-east-1` is set, it can override the region in your profile for CLI operations.
- If `AWS_PROFILE=prod` is set, the CLI uses the `prod` profile unless you pass `--profile dev`.

> If multiple credential sources are present, the CLI uses the first valid source in its resolution path. Conflicting environment variables often cause the most confusing failures.

**Environment variables you should know**

Environment variables are useful for temporary overrides, automation, and CI/CD pipelines.

Common variables:

- `AWS_PROFILE` selects a named profile.
- `AWS_DEFAULT_REGION` sets the default region for the CLI.
- `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY` provide static credentials.
- `AWS_SESSION_TOKEN` supplies the session token for temporary credentials.
- `AWS_PAGER` controls the pager used by the CLI.
- `AWS_CA_BUNDLE` points to a custom certificate bundle.

Region note:

- For the AWS CLI, `AWS_DEFAULT_REGION` is the standard environment variable for default region selection.
- `AWS_REGION` is more commonly emphasized in SDK and runtime contexts and is not the primary CLI region variable in the same way.
- If both are present, prefer `AWS_DEFAULT_REGION` as the exam-safe answer for CLI behavior.

Examples:

```bash
export AWS_PROFILE=dev
export AWS_DEFAULT_REGION=us-west-2
export AWS_PAGER=""
```

To clear a conflicting variable in a shell session:

```bash
unset AWS_PROFILE
unset AWS_DEFAULT_REGION
unset AWS_ACCESS_KEY_ID
unset AWS_SECRET_ACCESS_KEY
unset AWS_SESSION_TOKEN
```

**Temporary credentials and session handling**

Temporary credentials include a session token and expire after a fixed duration. The CLI uses them for:

- STS (AWS Security Token Service) assumed roles
- AWS Identity Center sessions
- Federated access flows
- Some `credential_process` integrations

Temporary credentials are represented by:

- `aws_access_key_id`
- `aws_secret_access_key`
- `aws_session_token`

Example profile for assumed role access:

```ini
[profile audit]
role_arn = arn:aws:iam::123456789012:role/AuditRole
source_profile = dev
region = us-east-1
```

In this setup:

- `source_profile` points to a profile that has source credentials.
- The CLI uses those source credentials to call STS and obtain temporary role credentials.
- The session token is handled automatically by the CLI after the role is assumed.

Temporary credentials are also common with AWS Identity Center, where the CLI caches tokens and refreshes them when possible.

**AWS Identity Center and `sso-session` configuration**

Modern AWS CLI v2 supports AWS Identity Center, which is the current name for AWS Single Sign-On. The newer session-based flow uses `sso-session` blocks to centralize SSO settings.

Example:

```ini
# ~/.aws/config
[sso-session corp]
sso_start_url = https://d-abc123.awsapps.com/start
sso_region = us-east-1
sso_registration_scopes = sso:account:access

[profile dev-sso]
sso_session = corp
sso_account_id = 123456789012
sso_role_name = DeveloperAccess
region = us-west-2
output = json
```

Typical workflow:

1. Run `aws sso login --profile dev-sso`.
2. Use the profile with AWS CLI commands.
3. Let the CLI refresh cached SSO tokens as needed.

Why this matters:

- It reduces long-term key usage.
- It is the preferred modern approach for many enterprise environments.
- It separates the reusable SSO session from individual account-role profiles.

**Manual profile management and file structure**

You often need to edit profiles directly when migrating, repairing, or reviewing configuration.

Rules to remember:

- Use `[default]` in both files for the default profile.
- Use `[profile name]` in `~/.aws/config`.
- Use `[name]` in `~/.aws/credentials`.
- Keep section names exact; a typo creates a separate profile instead of updating the intended one.

Example of multiple profiles:

```ini
# ~/.aws/config
[default]
region = us-east-1
output = json

[profile dev]
region = us-west-2
output = table

[profile prod]
region = eu-central-1
output = json
cli_pager = less
```

```ini
# ~/.aws/credentials
[default]
aws_access_key_id = AKIADEFAULTEXAMPLE
aws_secret_access_key = defaultSecretExample

[dev]
aws_access_key_id = AKIADEVEXAMPLE
aws_secret_access_key = devSecretExample

[prod]
aws_access_key_id = AKIAPRODEXAMPLE
aws_secret_access_key = prodSecretExample
```

Common manual-edit mistakes:

- Using `[dev]` in `~/.aws/config` instead of `[profile dev]`.
- Creating a profile in `~/.aws/credentials` but forgetting the matching config section.
- Leaving stale keys in `default` and accidentally using them when `--profile` is omitted.

**Useful configuration options**

Several settings affect CLI behavior beyond credentials and region.

- `output` controls the default output format.
  - Common values: `json`, `table`, `text`, `yaml`, `yaml-stream`
- `cli_pager` controls paging of long output.
  - Set it to `less` or another pager if you want interactive scrolling.
  - Disable paging with `AWS_PAGER=""` or `--no-cli-pager`.
- `cli_binary_format` controls how binary parameters are handled.
  - This matters for commands that pass binary blobs, such as some Lambda or S3-related workflows.
- `cli_timestamp_format` controls timestamp display formatting in supported outputs.
- `region` sets the default AWS Region for the profile.

Example:

```ini
[profile ops]
region = us-east-2
output = yaml
cli_pager = less
cli_binary_format = base64
cli_timestamp_format = iso8601
```

Pager guidance:

- Use `--no-cli-pager` when you want a one-off command to print directly.
- Use `AWS_PAGER=""` when you want to disable paging for the current shell session.
- Avoid relying on empty `cli_pager` values as the primary method; behavior can vary by version and environment.

Examples:

```bash
aws --no-cli-pager s3 ls
AWS_PAGER="" aws s3 ls
```

**Troubleshooting common profile problems**

Profile issues usually come from syntax errors, missing source credentials, or conflicting environment variables.

1. **Invalid profile section name**
   - Symptom: the CLI ignores the profile or reports it cannot find it.
   - Fix: ensure `~/.aws/config` uses `[profile name]` and `~/.aws/credentials` uses `[name]`.

2. **Missing source profile**
   - Symptom: an assume-role profile fails with an error about the source profile or credentials.
   - Fix: verify that `source_profile` exists and contains valid credentials.

```ini
[profile prod-admin]
role_arn = arn:aws:iam::123456789012:role/AdminRole
source_profile = base
region = us-east-1

[base]
aws_access_key_id = AKIABASEEXAMPLE
aws_secret_access_key = baseSecretExample
```

3. **Conflicting environment variables**
   - Symptom: the CLI uses the wrong account, region, or credentials.
   - Fix: inspect and clear variables such as `AWS_PROFILE`, `AWS_DEFAULT_REGION`, `AWS_ACCESS_KEY_ID`, and `AWS_SECRET_ACCESS_KEY`.

```bash
env | grep '^AWS_'
unset AWS_PROFILE AWS_DEFAULT_REGION AWS_ACCESS_KEY_ID AWS_SECRET_ACCESS_KEY AWS_SESSION_TOKEN
```

4. **Expired temporary credentials**
   - Symptom: `ExpiredToken` or authentication failures after a period of time.
   - Fix: refresh the session with `aws sso login`, re-assume the role, or regenerate the temporary credentials.

5. **Wrong region**
   - Symptom: resources appear missing or commands fail against the wrong endpoint.
   - Fix: confirm the effective region with `aws configure list` and override with `--region` if needed.

**Practical CLI examples**

List buckets with a named profile:

```bash
aws s3 ls --profile dev
```

Show the current identity:

```bash
aws sts get-caller-identity --profile prod
```

Check effective configuration:

```bash
aws configure list --profile dev
```

Set a region for a profile:

```bash
aws configure set region ap-southeast-1 --profile dev
```

Use a profile without changing the shell environment:

```bash
aws ec2 describe-instances --profile audit --region us-east-1
```

**Migration and import workflows**

In operational environments, you may need to move credentials between machines or standardize local setups.

Common approaches:

- Manually copy the relevant profile sections in `~/.aws/config` and `~/.aws/credentials`.
- Recreate profiles with `aws configure` or `aws configure set`.
- Use `aws configure import` when supported by your CLI version and workflow to bring in credentials from a supported source.

Practical migration advice:

- Prefer named profiles over editing the `default` profile for shared workstations.
- Keep long-term access keys out of scripts when possible.
- Use role assumption or AWS Identity Center for day-to-day access.

**Security and operational best practices**

- Use named profiles for each account or role.
- Prefer temporary credentials over long-term access keys.
- Store secrets only in the credentials file or a secure secret manager, not in shell history.
- Avoid hardcoding credentials in scripts.
- Use `aws sts get-caller-identity` to verify the active identity before making changes.
- Rotate access keys regularly if you must use them.
- Remove stale profiles and unused keys to reduce confusion.

**Exam-focused distinctions**

- `aws configure --profile <name>` is valid and commonly used.
- `AWS_DEFAULT_REGION` is the key environment variable to remember for CLI region selection.
- `aws configure list` helps you inspect effective values, but source labels are not guaranteed to be identical across versions.
- `--no-cli-pager` is a global option for suppressing paging.
- `aws configure unset` is not a standard AWS CLI v2 command.
- `source_profile` is required for many assume-role profiles that depend on another credential source.
- AWS Identity Center uses `sso-session` and cached tokens, which is the modern CLI v2 pattern.

**Quick reference**

| Task | Command or file |
|---|---|
| Configure default profile | `aws configure` |
| Configure named profile | `aws configure --profile dev` |
| Set a value | `aws configure set region us-west-2 --profile dev` |
| View a value | `aws configure get region --profile dev` |
| List profiles | `aws configure list-profiles` |
| Inspect effective config | `aws configure list` |
| Disable pager for one command | `aws --no-cli-pager s3 ls` |
| Disable pager for session | `AWS_PAGER=""` |
| Default config file | `~/.aws/config` |
| Default credentials file | `~/.aws/credentials` |

**Troubleshooting checklist**

- Confirm the intended profile exists in both files if it needs both config and credentials.
- Check for typos in section names and profile prefixes.
- Verify `AWS_PROFILE` and `AWS_DEFAULT_REGION` are not overriding your settings.
- Use `aws sts get-caller-identity` to confirm the active account and role.
- Refresh SSO sessions or temporary credentials when tokens expire.
- Edit the files manually if you need to remove a setting or repair a malformed section.
