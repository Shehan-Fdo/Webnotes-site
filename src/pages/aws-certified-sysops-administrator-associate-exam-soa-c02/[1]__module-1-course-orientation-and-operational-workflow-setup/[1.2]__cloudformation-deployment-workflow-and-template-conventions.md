## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**CloudFormation deployment workflow**

AWS CloudFormation lets you model infrastructure as code using templates, then create, update, and delete stacks from those templates. For exam purposes, focus on the full lifecycle: author the template, validate it, package local artifacts if needed, create the stack, review events and outputs, update safely, detect drift, and clean up with the right deletion behavior.

A deployment is not just a single command. It is a sequence of checks and state transitions. You should know how CloudFormation behaves when a stack succeeds, fails, rolls back, or becomes stuck in a rollback state.

- **Stack**: a deployed collection of AWS resources managed as one unit.
- **Template**: the JSON or YAML document that defines resources, parameters, mappings, conditions, and outputs.
- **Change set**: a preview of proposed changes before execution.
- **Drift**: a difference between the stack template and the actual live resource configuration.

**Core deployment commands**

You commonly use the AWS CLI to validate, package, create, update, and delete stacks. `aws cloudformation package` is used to upload local artifacts, such as Lambda code or nested templates, to Amazon S3 and rewrite template references. It does not deploy the stack by itself; it prepares a template for deployment.

```bash
aws cloudformation validate-template \
  --template-body file://template.yaml

aws cloudformation package \
  --template-file template.yaml \
  --s3-bucket my-artifact-bucket \
  --output-template-file packaged.yaml

aws cloudformation deploy \
  --template-file packaged.yaml \
  --stack-name my-app \
  --capabilities CAPABILITY_IAM CAPABILITY_NAMED_IAM \
  --parameter-overrides EnvironmentType=prod InstanceType=t3.micro
```

Use `deploy` when you want a simpler workflow that can create or update a stack. Use `create-stack` and `update-stack` when you need more explicit control over stack behavior, rollback settings, or change management.

**Stack creation and update behavior**

When you create a stack, CloudFormation provisions resources in dependency order. When you update a stack, CloudFormation compares the new template and parameters to the current stack and applies only the required changes. If an update fails, CloudFormation rolls back by default, but rollback can also fail and leave the stack in a state such as `UPDATE_ROLLBACK_FAILED`.

> Do not assume a failed update always returns the stack to a clean stable state. You may need to inspect events and continue or cancel rollback before retrying.

Important command options and behaviors:

- **Capabilities**: required when the template creates IAM resources or named IAM resources.
  - `CAPABILITY_IAM` allows IAM resource creation.
  - `CAPABILITY_NAMED_IAM` is required when you specify custom names for IAM resources.
- **Parameters**: pass values with `--parameters` or `--parameter-overrides`.
  - Missing required parameters can block stack creation or update.
  - Parameter defaults in the template reduce deployment friction.
- **Termination protection**: prevents accidental stack deletion.
  - Enable it for critical production stacks.
  - It does not prevent updates.
- **Rollback controls**:
  - `create-stack` supports `--disable-rollback` and `--on-failure`.
  - `--on-failure` supports `DO_NOTHING`, `ROLLBACK`, and `DELETE`.
  - These are not interchangeable with update behavior; use them intentionally.

Example explicit stack creation:

```bash
aws cloudformation create-stack \
  --stack-name web-app \
  --template-body file://template.yaml \
  --parameters ParameterKey=EnvironmentType,ParameterValue=prod \
  --capabilities CAPABILITY_IAM CAPABILITY_NAMED_IAM \
  --enable-termination-protection
```

Example update with capabilities and parameter overrides:

```bash
aws cloudformation update-stack \
  --stack-name web-app \
  --template-body file://template.yaml \
  --parameters ParameterKey=EnvironmentType,ParameterValue=prod \
  --capabilities CAPABILITY_IAM CAPABILITY_NAMED_IAM
```

**Stack events, statuses, and troubleshooting**

Stack events are your primary source of truth during deployment. They show which resource is being created, updated, deleted, or rolled back, and they often include the failure reason.

Common statuses to recognize:

- `CREATE_IN_PROGRESS`, `CREATE_COMPLETE`, `CREATE_FAILED`
- `UPDATE_IN_PROGRESS`, `UPDATE_COMPLETE`, `UPDATE_FAILED`
- `UPDATE_ROLLBACK_IN_PROGRESS`, `UPDATE_ROLLBACK_COMPLETE`, `UPDATE_ROLLBACK_FAILED`
- `DELETE_IN_PROGRESS`, `DELETE_COMPLETE`, `DELETE_FAILED`

Use events to identify the first failing resource, not just the final stack status. The earliest failure often reveals the root cause, such as:

- Missing IAM permissions
- Invalid parameter values
- Bad resource properties
- Dependency ordering problems
- Quota limits or service-specific constraints

Useful CLI commands:

```bash
aws cloudformation describe-stacks --stack-name web-app
aws cloudformation describe-stack-events --stack-name web-app
aws cloudformation list-stack-resources --stack-name web-app
```

A practical troubleshooting pattern is to inspect the most recent events, identify the first `FAILED` resource, then check the service-specific error message. For example, an EC2 security group rule error may be caused by an invalid CIDR block, while an RDS failure may be caused by an unsupported engine version or subnet configuration.

**Change sets and safe updates**

Change sets let you preview the effect of a template update before you execute it. This is useful when you want to see whether a change will replace a resource, modify it in place, or delete it.

```bash
aws cloudformation create-change-set \
  --stack-name web-app \
  --change-set-name web-app-change-1 \
  --template-body file://template.yaml \
  --capabilities CAPABILITY_IAM CAPABILITY_NAMED_IAM \
  --parameters ParameterKey=EnvironmentType,ParameterValue=prod

aws cloudformation describe-change-set \
  --stack-name web-app \
  --change-set-name web-app-change-1

aws cloudformation execute-change-set \
  --stack-name web-app \
  --change-set-name web-app-change-1
```

Use change sets when:

- You need review before production deployment.
- You want to detect replacements that could cause downtime.
- You are coordinating a controlled release.

**Template conventions and structure**

A CloudFormation template is usually organized into sections that make intent clear and support reuse.

Common sections:

- **AWSTemplateFormatVersion**: optional version marker.
- **Description**: human-readable summary.
- **Parameters**: input values supplied at deployment time.
- **Mappings**: static lookup tables.
- **Conditions**: logic for conditional resource creation or property selection.
- **Resources**: the actual AWS resources.
- **Outputs**: values returned after stack creation.
- **Metadata**: auxiliary information for tools or operators.

A minimal YAML template:

```yaml
AWSTemplateFormatVersion: '2010-09-09'
Description: Simple web application stack

Parameters:
  EnvironmentType:
    Type: String
    AllowedValues:
      - dev
      - test
      - prod
    Default: dev

Resources:
  AppBucket:
    Type: AWS::S3::Bucket

Outputs:
  BucketName:
    Value: !Ref AppBucket
```

**Parameters and validation**

Parameters make templates reusable across environments. Use constraints to reduce deployment errors and enforce valid input.

Important parameter properties:

- **AllowedValues**: restricts input to a fixed set.
- **AllowedPattern**: validates a string with a regular expression.
- **MinLength** and **MaxLength**: constrain string length.
- **MinValue** and **MaxValue**: constrain numeric input.
- **NoEcho**: masks sensitive values in console output and many logs.

Example:

```yaml
Parameters:
  EnvironmentType:
    Type: String
    Description: Deployment environment
    AllowedValues:
      - dev
      - stage
      - prod
    Default: dev

  DBPassword:
    Type: String
    NoEcho: true
    MinLength: 12
    AllowedPattern: '[A-Za-z0-9@#%+=:_-]+'
```

Use `NoEcho` for secrets, but do not treat it as full secret management. The value can still be exposed in some contexts, such as resource metadata or outputs if you explicitly place it there.

**Resource ordering and dependencies**

CloudFormation usually infers dependencies from references, but you can force ordering with `DependsOn` when implicit references are not enough. This matters when a resource must exist before another resource can be created, even if the template does not directly reference it.

Example:

```yaml
Resources:
  AppSecurityGroup:
    Type: AWS::EC2::SecurityGroup
    Properties:
      GroupDescription: App SG

  AppInstance:
    Type: AWS::EC2::Instance
    DependsOn: AppSecurityGroup
    Properties:
      ImageId: ami-0123456789abcdef0
      InstanceType: t3.micro
      SecurityGroupIds:
        - !Ref AppSecurityGroup
```

Use `DependsOn` sparingly. Prefer natural references when possible, because they are easier to maintain and reflect actual resource relationships.

**Intrinsic functions**

Intrinsic functions let you build dynamic templates without hardcoding values. Know the common ones and when to use them.

Frequently tested functions:

- **`Ref`**: returns a parameter value or a resource-specific identifier.
- **`Fn::GetAtt`**: returns an attribute from a resource.
- **`Fn::Join`**: concatenates values with a delimiter.
- **`Fn::Sub`**: substitutes variables into strings.
- **`Fn::Select`**: selects an item from a list by index.
- **`Fn::Split`**: splits a string into a list.
- **`Fn::FindInMap`**: looks up values in a mapping.
- **`Fn::ImportValue`**: imports an exported value from another stack.
- **`Fn::If`**: returns one of two values based on a condition.
- **`Fn::Equals`**: compares two values for a condition.

Examples:

```yaml
Mappings:
  RegionMap:
    us-east-1:
      AMI: ami-11111111111111111
    us-west-2:
      AMI: ami-22222222222222222

Parameters:
  EnvironmentType:
    Type: String
    AllowedValues: [dev, prod]

Conditions:
  IsProd: !Equals [!Ref EnvironmentType, prod]

Resources:
  AppInstance:
    Type: AWS::EC2::Instance
    Properties:
      ImageId: !FindInMap [RegionMap, !Ref AWS::Region, AMI]
      InstanceType: !If [IsProd, m5.large, t3.micro]
      Tags:
        - Key: Name
          Value: !Sub '${AWS::StackName}-${EnvironmentType}'
```

Example of `Fn::Join`, `Fn::Select`, `Fn::Split`, and `Fn::ImportValue`:

```yaml
Outputs:
  SubnetId:
    Value: !Select [0, !Split [",", !ImportValue SharedNetwork-PrivateSubnetIds]]
  BucketArn:
    Value: !Join ["", ["arn:aws:s3:::", !Ref AppBucket]]
```

**Conditions and conditional resources**

Conditions let you create or configure resources only when a condition evaluates to true. This is useful for environment-specific deployments, optional features, and cost control.

Typical uses:

- Create extra logging resources only in production.
- Use a snapshot ID only when restoring a database.
- Enable alarms or scaling policies only in selected environments.

Example with a defined parameter context:

```yaml
Parameters:
  EnvironmentType:
    Type: String
    AllowedValues: [dev, prod]
    Default: dev

  SnapshotId:
    Type: String
    Default: ""

Conditions:
  UseSnapshot: !Not [!Equals [!Ref SnapshotId, ""]]
  IsProd: !Equals [!Ref EnvironmentType, prod]

Resources:
  Database:
    Type: AWS::RDS::DBInstance
    Properties:
      DBInstanceClass: db.t3.micro
      Engine: mysql
      DBSnapshotIdentifier: !If [UseSnapshot, !Ref SnapshotId, !Ref AWS::NoValue]
```

Use `AWS::NoValue` to omit a property when a condition is false.

**Macros and transforms**

Transforms let CloudFormation preprocess templates before deployment. This is important for reusable abstractions and serverless applications.

Common transforms:

- **`AWS::Serverless`**: used by AWS SAM (Serverless Application Model) templates.
- **`AWS::Include`**: inserts template fragments from S3.
- **Custom macros**: user-defined template transformations.

Example SAM transform:

```yaml
Transform: AWS::Serverless-2016-10-31

Resources:
  HelloFunction:
    Type: AWS::Serverless::Function
    Properties:
      Handler: app.lambda_handler
      Runtime: python3.12
      CodeUri: src/
      Events:
        Api:
          Type: Api
          Properties:
            Path: /hello
            Method: get
```

Use transforms when you want higher-level syntax or reusable fragments. Remember that the transformed template is what CloudFormation ultimately processes.

**Nested stacks and modular design**

Nested stacks help you split large templates into smaller, reusable components. This improves maintainability and can simplify updates.

Example parent stack:

```yaml
Resources:
  NetworkStack:
    Type: AWS::CloudFormation::Stack
    Properties:
      TemplateURL: https://s3.amazonaws.com/my-bucket/network.yaml
      Parameters:
        VpcCidr: 10.0.0.0/16
```

Use nested stacks when:

- A template becomes too large to manage comfortably.
- You want to reuse a common component, such as networking or logging.
- You need clearer separation of concerns across teams.

**StackSets and multi-account deployment**

StackSets extend CloudFormation to deploy stacks across multiple accounts and Regions from a single template. This is useful for organization-wide baselines, guardrails, and shared services.

Typical StackSet workflow:

```bash
aws cloudformation create-stack-set \
  --stack-set-name baseline-network \
  --template-body file://network.yaml \
  --capabilities CAPABILITY_IAM CAPABILITY_NAMED_IAM

aws cloudformation create-stack-instances \
  --stack-set-name baseline-network \
  --accounts 111122223333 444455556666 \
  --regions us-east-1 us-west-2
```

Use StackSets when you need consistent deployment at scale. Know the difference between a stack, a nested stack, and a StackSet:

- **Stack**: one deployment in one account and Region.
- **Nested stack**: a stack managed by another stack.
- **StackSet**: one template deployed to many accounts and/or Regions.

**Deletion behavior and retention**

Deletion behavior is controlled by resource-level policies and stack deletion actions. `DeletionPolicy` applies when CloudFormation deletes a resource from the stack, including stack deletion and some replacement scenarios where the old resource is removed. It does not mean the resource is always retained just because you edited the template; the policy is evaluated when CloudFormation actually deletes that resource.

Common `DeletionPolicy` values:

- **Delete**: remove the resource.
- **Retain**: keep the resource after stack deletion or resource removal.
- **Snapshot**: create a snapshot before deletion, where supported.

Example:

```yaml
Resources:
  Database:
    Type: AWS::RDS::DBInstance
    DeletionPolicy: Snapshot
    Properties:
      DBInstanceClass: db.t3.micro
      Engine: mysql
```

Use `Retain` for data you cannot lose, such as production databases or S3 buckets with important content. Be aware that retained resources can become orphaned and require manual cleanup later.

**Stack policies**

Stack policies protect specific resources from accidental updates. They are not IAM policies. A stack policy uses CloudFormation-specific actions such as `Update:Modify`, `Update:Replace`, and `Update:Delete`, and it must be written as a full stack policy document with resource matching.

Example stack policy structure:

```json
{
  "Statement": [
    {
      "Effect": "Deny",
      "Action": "Update:Delete",
      "Principal": "*",
      "Resource": "LogicalResourceId/ProductionDatabase"
    }
  ]
}
```

Use stack policies to protect critical resources, especially in production. They are useful when you want to allow most stack updates but block destructive changes to a subset of resources.

**Drift detection and remediation**

Drift detection compares the deployed stack resources against the template and stack parameters to identify configuration changes made outside CloudFormation. Drift matters because manual changes can break repeatability and make future updates unpredictable.

Operational workflow:

1. Run drift detection on the stack.
2. Review which resources are drifted and how they differ.
3. Decide whether to update the template to match the live state or revert the live resource to match the template.
4. Re-run drift detection after remediation.

CLI example:

```bash
aws cloudformation detect-stack-drift --stack-name web-app
aws cloudformation describe-stack-drift-detection-status --stack-drift-detection-id abc12345
aws cloudformation describe-stack-resource-drifts --stack-name web-app
```

Remediation options:

- Update the stack template or parameters if the live change is intentional.
- Revert manual changes if the template is the source of truth.
- Replace the resource if it cannot be reconciled safely.

**Practical deployment checklist**

Before deployment, verify the template and inputs so you avoid common failures.

1. Validate the template syntax with `validate-template`.
2. Confirm required parameters and defaults.
3. Check whether IAM resources require `CAPABILITY_IAM` or `CAPABILITY_NAMED_IAM`.
4. Review dependencies and `DependsOn` usage.
5. Decide whether you need a change set.
6. Confirm termination protection for critical stacks.
7. Review stack events after deployment.

**Exam-focused distinctions**

- **`deploy` vs `create-stack`/`update-stack`**:
  - `deploy` is convenient for common workflows.
  - `create-stack` and `update-stack` expose more explicit controls.
- **Rollback vs failure**:
  - A failed update usually triggers rollback.
  - Rollback can fail, leaving the stack in a rollback-failed state.
- **Package vs deploy**:
  - `package` prepares artifacts and rewrites references.
  - `deploy` or `create-stack` actually provisions the stack.
- **Drift vs change set**:
  - Drift detects out-of-band changes after deployment.
  - Change sets preview planned changes before deployment.

**Quick reference examples**

A simple parameterized stack with outputs:

```yaml
AWSTemplateFormatVersion: '2010-09-09'
Description: Parameterized S3 bucket example

Parameters:
  BucketNamePrefix:
    Type: String
    AllowedPattern: '^[a-z0-9.-]{3,63}$'
    MinLength: 3
    MaxLength: 63

Resources:
  AppBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: !Sub '${BucketNamePrefix}-${AWS::AccountId}'

Outputs:
  BucketName:
    Value: !Ref AppBucket
```

A conditional resource with a dependency:

```yaml
Conditions:
  CreateAlarm: !Equals [!Ref EnvironmentType, prod]

Resources:
  AlarmTopic:
    Type: AWS::SNS::Topic
    Condition: CreateAlarm

  Alarm:
    Type: AWS::CloudWatch::Alarm
    DependsOn: AlarmTopic
    Condition: CreateAlarm
    Properties:
      AlarmName: !Sub '${AWS::StackName}-cpu-high'
      ComparisonOperator: GreaterThanThreshold
      EvaluationPeriods: 2
      MetricName: CPUUtilization
      Namespace: AWS/EC2
      Period: 60
      Statistic: Average
      Threshold: 80
```

CloudFormation rewards disciplined template design. If you understand how parameters, conditions, intrinsic functions, transforms, dependencies, and deployment controls work together, you can predict stack behavior and troubleshoot failures quickly.
