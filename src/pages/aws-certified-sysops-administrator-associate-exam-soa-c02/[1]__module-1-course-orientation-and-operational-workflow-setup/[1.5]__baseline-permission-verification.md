## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Baseline Permission Verification**

Baseline permission verification means confirming the minimum set of permissions an identity needs before you troubleshoot a failure or expand access. In AWS, access decisions are the result of multiple policy layers, so you verify the caller, the attached permissions, and any explicit denies before assuming the service is broken.

This topic matters because many AWS issues are not service outages at all. They are authorization failures caused by missing IAM permissions, restrictive resource policies, permission boundaries, session policies, SCPs (Service Control Policies), or KMS (Key Management Service) key policy constraints. A clean baseline helps you separate “can’t access” from “service unavailable.”

**What you verify first**

Start with the identity and the exact action being attempted. Then confirm whether the request is allowed by the effective permission set.

- **Caller identity**: determine whether the request comes from an IAM user, IAM role, federated session, AWS service role, or root user.
- **Target action**: identify the API action, such as `s3:GetObject`, `ec2:DescribeInstances`, or `iam:PassRole`.
- **Target resource**: confirm the ARN (Amazon Resource Name) and whether the service supports resource-level permissions.
- **Region and account**: verify the request is in the correct Region and AWS account.
- **Deny sources**: look for explicit denies in identity policies, resource policies, SCPs, permission boundaries, or session policies.

A useful mental model is that AWS authorization is evaluated with a default deny posture. An allow must exist somewhere in the effective policy set, and no explicit deny can override it.

**Core permission layers**

AWS evaluates permissions across several layers. You need to understand each layer because a missing allow in one place can look like a service problem.

- **Identity-based policies** attach to IAM users, groups, or roles and grant permissions to the identity.
- **Resource-based policies** attach to the resource, such as an S3 bucket policy, KMS key policy, or SNS topic policy.
- **Permissions boundaries** limit the maximum permissions an IAM principal can have, even if identity policies are broader.
- **Session policies** further restrict temporary credentials from STS (AWS Security Token Service).
- **SCPs** in AWS Organizations set the maximum permissions available to accounts or organizational units.
- **ACLs** (Access Control Lists) still matter for some services, especially legacy S3 object ownership scenarios, though they are less central than policies.
- **Service-linked roles** are managed by AWS services and may be required for service operations.

> An explicit deny anywhere in the evaluation path wins over any allow.

**How to verify the baseline**

A practical verification flow keeps you from chasing the wrong layer.

1. Confirm the identity with `aws sts get-caller-identity`.
2. Reproduce the exact failing API call with the same credentials and Region.
3. Check the identity policies attached to the user or role.
4. Check resource policies on the target resource.
5. Check organizational controls such as SCPs and permission boundaries.
6. Check whether the action requires a service-linked role, pass role permission, or KMS access.
7. Review CloudTrail for the denied event and the error code.

Example identity check:

```bash
aws sts get-caller-identity
```

Typical output includes the account, ARN, and user ID. This tells you whether you are using the expected principal.

Example policy inspection for an IAM role:

```bash
aws iam list-attached-role-policies --role-name MyAppRole
aws iam list-role-policies --role-name MyAppRole
aws iam get-role --role-name MyAppRole
```

Example for a user:

```bash
aws iam list-attached-user-policies --user-name Alice
aws iam list-user-policies --user-name Alice
```

**Effective permissions and policy evaluation**

Effective permissions are the result of combining all applicable policy types. The most important rule is that AWS starts with deny and only allows what is explicitly permitted.

- **Identity policy allow** is necessary for most actions.
- **Resource policy allow** may be required for cross-account access or service-specific access patterns.
- **Boundary and SCP limits** can remove permissions even when identity policies allow them.
- **Explicit deny** always overrides allow.
- **Implicit deny** means no applicable allow was found.

A common exam scenario is a role that has `AdministratorAccess`, but the action still fails. The cause is often an SCP, a permission boundary, or a resource policy that blocks the request.

**Example: IAM policy allowing S3 read access**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadObjects",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::example-bucket",
        "arn:aws:s3:::example-bucket/*"
      ]
    }
  ]
}
```

This policy is enough for basic read access only if no other layer blocks it and the bucket policy does not require additional conditions.

**Using the IAM policy simulator**

The IAM policy simulator helps you test whether a principal can perform an action on a resource. It is useful when you need to prove whether the issue is authorization or something else.

- Use it to test a specific action and resource ARN.
- Include context keys when conditions are involved, such as `aws:SourceIp`, `aws:PrincipalTag`, or `s3:x-amz-server-side-encryption`.
- Use it to compare expected access with actual access.

CLI example for policy simulation:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/MyAppRole \
  --action-names s3:GetObject \
  --resource-arns arn:aws:s3:::example-bucket/test.txt
```

If the result is `allowed`, the issue may be elsewhere, such as network path, endpoint policy, bucket policy conditions, or KMS encryption permissions.

**CloudTrail as a verification source**

CloudTrail records API activity and is one of the best tools for baseline permission verification. When a request fails, CloudTrail can show the exact API name, principal, source IP, and error message.

Look for:

- `AccessDenied`
- `UnauthorizedOperation`
- `Client.UnauthorizedOperation`
- `KMS.AccessDeniedException`
- `InvalidClientTokenId`
- `AccessDeniedException`

CloudTrail helps you distinguish between:
- **Authentication failures**: bad credentials, expired tokens, wrong session
- **Authorization failures**: valid credentials but missing permission
- **Service-side failures**: the request was allowed but the service could not complete it

**Common AWS CLI checks**

Use the AWS CLI to verify the current session and the permissions around the target service.

```bash
aws configure list
aws sts get-caller-identity
aws ec2 describe-regions
aws s3 ls
```

These commands help you confirm:
- which credentials are active
- whether the session is valid
- whether the principal can perform basic read operations

For a role assumption workflow, verify the temporary credentials:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/ReadOnlyRole \
  --role-session-name baseline-check
```

Then export the returned temporary credentials and retest the failing action.

**Service-specific permission gotchas**

Some services require more than the obvious action permission.

- **EC2** often needs `ec2:Describe*` for visibility and `iam:PassRole` when launching instances with an instance profile.
- **S3** may require both bucket-level and object-level permissions, plus bucket policy alignment.
- **KMS** requires both IAM permission and key policy permission in many scenarios.
- **Lambda** may need `lambda:InvokeFunction` plus permissions for any downstream service it accesses.
- **CloudWatch Logs** often requires permissions on both log groups and log streams.
- **Systems Manager** may require instance profile permissions, SSM agent health, and network reachability in addition to IAM access.

> If an AWS service uses encryption, always check KMS permissions before assuming the service itself is failing.

**KMS and encryption baseline checks**

KMS is a frequent source of hidden authorization failures. Even if the primary service permission is correct, the request can fail if the principal cannot use the key.

Verify:
- `kms:Decrypt` for reading encrypted data
- `kms:Encrypt` for writing encrypted data
- `kms:GenerateDataKey` for envelope encryption workflows
- key policy access for the principal or service
- grants, if the service uses them

Example KMS permission snippet:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "kms:Decrypt",
        "kms:GenerateDataKey"
      ],
      "Resource": "arn:aws:kms:us-east-1:123456789012:key/abcd1234-5678-90ab-cdef-EXAMPLE11111"
    }
  ]
}
```

**Cross-account access verification**

Cross-account access requires both sides to align. You need permission in the caller account and trust or resource permission in the target account.

- The caller identity must be allowed to call the action.
- The target resource policy must trust the caller principal or account.
- The target role trust policy must allow `sts:AssumeRole` from the caller.
- SCPs in either account can still block the request.

Example trust policy for a role in the target account:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:root"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Example assume-role test:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::444455556666:role/CrossAccountReadRole \
  --role-session-name verify-access
```

**Practical troubleshooting pattern**

When access fails, use a repeatable sequence so you do not miss the real blocker.

1. Capture the exact error message and API action.
2. Confirm the active identity with `sts get-caller-identity`.
3. Check whether the action is allowed by the identity policy.
4. Check for explicit deny in resource policy, boundary, SCP, or session policy.
5. Verify Region, account, and resource ARN.
6. Review CloudTrail for the denied event.
7. Test with the IAM policy simulator if the result is still unclear.

This approach is especially useful when multiple teams manage different layers of access. For example, an application team may own the IAM role, while a platform team owns the SCPs and a security team owns the KMS key policy.

**Quick reference: what to inspect**

| Area | What you check | Why it matters |
|---|---|---|
| Identity | User, role, session, root | Determines who is making the request |
| Identity policy | Attached allow/deny statements | Grants or blocks actions directly |
| Resource policy | Bucket, key, topic, queue, etc. | Enables cross-account or resource-scoped access |
| Permissions boundary | Maximum allowed permissions | Caps what the principal can do |
| Session policy | Temporary credential restrictions | Narrows assumed-role permissions |
| SCP | Organization-level guardrails | Can block actions even for admins |
| KMS key policy | Key usage permissions | Often required for encrypted resources |
| CloudTrail | Error code and request details | Confirms the exact failure mode |

**Exam-focused interpretation**

For SysOps questions, baseline permission verification usually means you should not jump straight to service remediation. First determine whether the principal has the right effective permissions.

- If the question mentions `AccessDenied`, think authorization first.
- If the user can list resources but not read or modify one object, think resource policy or object-level permission.
- If an admin role still fails, think SCP, boundary, or explicit deny.
- If encryption is involved, think KMS key policy and grants.
- If cross-account access is involved, think trust policy plus resource policy.

**Key takeaways**

Baseline permission verification is the process of proving whether AWS access is blocked by missing or restricted permissions before you troubleshoot anything else. The most reliable checks are the caller identity, the exact API action, the target resource ARN, and the full effective policy set. In practice, CloudTrail, IAM policy simulation, and targeted CLI tests give you the fastest path to the real cause.
