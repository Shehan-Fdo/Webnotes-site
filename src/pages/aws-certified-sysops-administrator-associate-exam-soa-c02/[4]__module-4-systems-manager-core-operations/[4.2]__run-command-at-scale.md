## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Run Command at Scale: what it is**

AWS Systems Manager Run Command lets you execute shell commands or scripts on managed nodes without logging in interactively. At scale, the key value is centralized, auditable, and repeatable execution across many instances, on-premises managed nodes, or hybrid fleets that are registered with Systems Manager.

Use it when you need to:

- Patch or remediate many instances quickly.
- Collect diagnostics from a fleet.
- Restart services or rotate configuration files.
- Run one-time operational tasks without building a custom orchestration system.

Run Command is not the same as a long-running workflow engine. It is best for discrete commands with clear success/failure outcomes and output capture.

**Core concepts**

A Run Command execution is a **command** sent to one or more targets. Each target receives an **invocation**. The command has an overall status, while each invocation has its own status.

Common status values to know:

- **Pending**: the command is accepted but not yet dispatched.
- **InProgress**: the command is currently running on the target.
- **Success**: the command completed successfully.
- **Failed**: the command completed with a non-zero exit or execution error.
- **TimedOut**: the command exceeded its timeout.
- **Cancelled**: the command was stopped before completion.
- **Delayed**: dispatch is postponed, often due to concurrency controls or service-side throttling.

For exam purposes, distinguish between:

- **Command status**: the aggregate view across all targets.
- **Invocation status**: the per-instance result.

A command can be partially successful even if some invocations fail.

**Prerequisites and fleet readiness**

Run Command depends on the Systems Manager agent and network reachability to Systems Manager endpoints.

You need:

- The **SSM Agent** installed and running on the managed node.
- An IAM instance profile attached to the instance with permissions for Systems Manager.
- Network access to Systems Manager endpoints:
  - Public internet access, or
  - NAT gateway/instance for private subnets, or
  - VPC endpoints for private connectivity in isolated networks.

Common interface endpoints for private fleets include:

- `com.amazonaws.<region>.ssm`
- `com.amazonaws.<region>.ec2messages`
- `com.amazonaws.<region>.ssmmessages`

If you omit these in a private, isolated VPC, the instance may appear managed but fail to receive or report command activity reliably.

> If the node cannot reach Systems Manager endpoints, Run Command may show as pending or fail to deliver even when IAM permissions are correct.

**Targeting instances at scale**

Run Command supports multiple targeting methods, including:

- Instance IDs.
- Tags.
- Resource groups.
- Managed node IDs.

Tag targeting is the most common at scale, but you must understand selector behavior carefully.

Multiple target objects are generally treated as a **union** of matches, not an intersection. If you specify separate tag selectors, you usually target instances matching either selector, not only instances matching both.

Example:

```bash
aws ssm send-command \
  --document-name "AWS-RunShellScript" \
  --targets "Key=tag:Role,Values=web" "Key=tag:Environment,Values=prod" \
  --parameters 'commands=["uptime"]'
```

This does **not** mean “web AND prod” in the way many people expect. It can target the union of the matching sets. To achieve an AND-like result, use a single selector strategy that uniquely identifies the intended fleet, such as a composite tag design like `Role=web-prod`, or prebuild a resource group that represents the intersection.

Practical targeting guidance:

- Use a single, well-designed tag key/value scheme for fleet segmentation.
- Prefer resource groups when you need a reusable operational boundary.
- Validate the target set before sending destructive commands.
- Avoid relying on multiple separate tag selectors to express logical AND.

**Command document selection and versioning**

Run Command uses an SSM document to define what runs on the target. Common documents include:

- `AWS-RunShellScript` for Linux and macOS.
- `AWS-RunPowerShellScript` for Windows.
- Custom documents for organization-specific tasks.

You can specify a document version to pin behavior and avoid surprises when the default version changes.

Examples:

```bash
aws ssm send-command \
  --document-name "AWS-RunShellScript" \
  --document-version "1" \
  --targets "Key=InstanceIds,Values=i-0123456789abcdef0" \
  --parameters 'commands=["sudo systemctl restart nginx"]'
```

Why version pinning matters:

- It improves repeatability.
- It reduces risk from document updates.
- It helps with change control and rollback.

If you omit the version, Systems Manager uses the default version of the document.

**CLI examples for common operations**

Send a command to a single instance:

```bash
aws ssm send-command \
  --document-name "AWS-RunShellScript" \
  --instance-ids "i-0123456789abcdef0" \
  --parameters 'commands=["df -h","free -m"]'
```

Send a command to a tagged fleet:

```bash
aws ssm send-command \
  --document-name "AWS-RunShellScript" \
  --targets "Key=tag:Role,Values=web" \
  --max-concurrency "10%" \
  --max-errors "1" \
  --timeout-seconds 600 \
  --parameters 'commands=["sudo yum update -y"]'
```

Retrieve command metadata:

```bash
aws ssm list-commands
aws ssm list-command-invocations --command-id "12345678-1234-1234-1234-123456789012" --details
```

Get output for a specific invocation:

```bash
aws ssm get-command-invocation \
  --command-id "12345678-1234-1234-1234-123456789012" \
  --instance-id "i-0123456789abcdef0"
```

**Concurrency, error thresholds, and scale controls**

At scale, the most important safety controls are **MaxConcurrency** and **MaxErrors**.

- **MaxConcurrency** limits how many targets run at once.
- **MaxErrors** limits how many failures are tolerated before Systems Manager stops dispatching to additional targets.

Important nuance:

- When the failure threshold is reached, Systems Manager stops sending the command to more targets.
- Already-dispatched invocations continue running; they are not cancelled automatically.

Examples:

```bash
aws ssm send-command \
  --document-name "AWS-RunShellScript" \
  --targets "Key=tag:Role,Values=web" \
  --max-concurrency "5" \
  --max-errors "2" \
  --parameters 'commands=["sudo systemctl restart app"]'
```

Use percentages when the fleet size changes frequently:

- `10%` concurrency is useful for rolling operations.
- `1%` error tolerance may be too strict for large fleets if transient failures are expected.
- Absolute counts are easier to reason about for small, stable fleets.

Practical interpretation:

- Use low concurrency for risky commands.
- Increase concurrency for read-only diagnostics.
- Set error thresholds to prevent broad blast radius.

**Output handling at scale**

For a few instances, you can inspect output with `get-command-invocation`. For a fleet, that does not scale well. Use centralized output destinations.

Run Command can send output to:

- **Amazon S3** for durable, queryable storage and long-term retention.
- **Amazon CloudWatch Logs** for near-real-time operational visibility and log-based alerting.

Typical use cases:

- Use **S3** when you need archival, later analysis, or integration with Athena and data lakes.
- Use **CloudWatch Logs** when you need fast troubleshooting, dashboards, and alarms.

Example with both destinations:

```bash
aws ssm send-command \
  --document-name "AWS-RunShellScript" \
  --targets "Key=tag:Role,Values=web" \
  --output-s3-bucket-name "my-ssm-output-bucket" \
  --output-s3-key-prefix "run-command/web" \
  --cloud-watch-output-config "CloudWatchOutputEnabled=true,CloudWatchLogGroupName=/ssm/run-command" \
  --parameters 'commands=["hostname","uptime"]'
```

Why this matters at scale:

- Per-instance API polling becomes inefficient and noisy.
- Centralized logs make it easier to troubleshoot partial failures.
- S3 and CloudWatch Logs support fleet-wide review after the command finishes.

A practical retrieval pattern:

1. Send the command with S3 and/or CloudWatch Logs enabled.
2. Use `list-command-invocations --details` for summary status.
3. Inspect S3 objects or CloudWatch log streams for detailed stdout/stderr.
4. Correlate failures by instance ID and command ID.

**Stopping and cancelling commands**

Run Command supports cancellation, which is important when a command is unsafe or misconfigured.

Use cancellation when:

- You targeted the wrong fleet.
- The command is causing service disruption.
- You need to stop further execution quickly.

Example:

```bash
aws ssm cancel-command --command-id "12345678-1234-1234-1234-123456789012"
```

Behavior to remember:

- Cancellation prevents further dispatch where possible.
- In-flight invocations may continue until they complete or time out.
- Cancellation is not the same as rollback; you must design rollback separately.

For exam scenarios, if a command is already running on many nodes, cancelling it does not guarantee immediate termination on every node.

**Permissions and roles**

There are three distinct permission areas to keep straight:

- **Operator IAM permissions**: the human or automation role that calls `ssm:SendCommand`, `ssm:ListCommands`, `ssm:GetCommandInvocation`, and related APIs.
- **Instance profile permissions**: the IAM role attached to the managed node that allows the SSM Agent to communicate with Systems Manager and access any resources needed by the script.
- **Service-linked role behavior**: some Systems Manager features use AWS-managed service-linked roles behind the scenes for orchestration and integration.

Practical permission considerations:

- The operator needs permission to use the specific document and target scope.
- The instance profile should be minimal but sufficient for the command’s runtime actions.
- If the command writes to S3 or CloudWatch Logs, the instance or service path must be able to publish output as required.

A safer operator policy pattern is to scope by document and allowed actions, then restrict target selection through organizational controls and tagging strategy. Be cautious with overly broad permissions that allow arbitrary document execution.

Example of a minimal operator-oriented policy shape:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowRunCommandRead",
      "Effect": "Allow",
      "Action": [
        "ssm:ListCommands",
        "ssm:ListCommandInvocations",
        "ssm:GetCommandInvocation"
      ],
      "Resource": "*"
    },
    {
      "Sid": "AllowSendCommandForApprovedDocuments",
      "Effect": "Allow",
      "Action": "ssm:SendCommand",
      "Resource": [
        "arn:aws:ssm:us-east-1:123456789012:document/AWS-RunShellScript",
        "arn:aws:ec2:us-east-1:123456789012:instance/*"
      ]
    }
  ]
}
```

This is only a conceptual example. In real environments, you should validate the exact resource-level permissions and condition keys supported for your document and target model.

**Operational limits and scale considerations**

At scale, practical limits matter as much as feature behavior.

Keep in mind:

- There are service quotas for command size, target counts, and API throughput.
- Large fleets may require batching or staged rollouts.
- Very large parameter payloads can make commands harder to manage and audit.
- Tag-based targeting is convenient, but the target set should remain predictable and bounded.

Operational best practices:

- Break very large fleets into waves.
- Use canary targets first.
- Prefer idempotent scripts so retries are safe.
- Keep commands short and deterministic.
- Store complex logic in version-controlled scripts or documents rather than inline shell fragments.

**Command lifecycle and troubleshooting**

A typical lifecycle looks like this:

1. You submit the command.
2. Systems Manager validates the request and resolves targets.
3. The command is dispatched according to concurrency settings.
4. Each managed node receives and executes the document.
5. Output is returned and optionally stored in S3 or CloudWatch Logs.

Troubleshooting checklist:

- Confirm the instance is a managed node in Systems Manager.
- Verify the SSM Agent is running.
- Check network access to SSM endpoints.
- Confirm the operator has permission to use the document.
- Validate the target selector matches the intended fleet.
- Review command and invocation status for `Failed`, `TimedOut`, or `Cancelled`.
- Inspect stdout/stderr in S3 or CloudWatch Logs for script-level errors.

Common failure patterns:

- The command is accepted but never reaches the instance: usually network or agent connectivity.
- The command reaches the instance but fails immediately: usually script syntax, missing dependencies, or permissions on the instance.
- Some instances succeed and others fail: often environment drift, package differences, or tag targeting mistakes.

**Run Command compared with related Systems Manager features**

| Feature | Best for | Typical behavior |
|---|---|---|
| Run Command | One-time or ad hoc remote execution | Executes commands on selected managed nodes |
| State Manager | Ongoing configuration enforcement | Keeps instances in a desired state over time |
| Maintenance Windows | Controlled execution windows | Schedules tasks during approved time periods |
| Automation | Multi-step operational workflows | Orchestrates runbooks, approvals, and branching |

Use this distinction to answer exam questions about “what service should you use”:

- Choose **Run Command** for immediate, one-off execution.
- Choose **State Manager** for recurring compliance or configuration drift correction.
- Choose **Maintenance Windows** when timing and change control matter.
- Choose **Automation** when the task requires multiple steps, approvals, or conditional logic.

**Practical examples**

Fleet diagnostics example:

```bash
aws ssm send-command \
  --document-name "AWS-RunShellScript" \
  --targets "Key=tag:Environment,Values=prod" \
  --max-concurrency "20%" \
  --max-errors "5" \
  --output-s3-bucket-name "company-ssm-logs" \
  --output-s3-key-prefix "diagnostics/prod" \
  --parameters 'commands=["uname -a","df -h","sudo systemctl status nginx --no-pager"]'
```

Windows patch-related check example:

```powershell
aws ssm send-command `
  --document-name "AWS-RunPowerShellScript" `
  --targets "Key=tag:Role,Values=windows-app" `
  --parameters 'commands=["Get-Service","Get-Process | Select-Object -First 10"]'
```

Canary rollout pattern:

- Target a small subset first, such as a dedicated canary tag.
- Review output in CloudWatch Logs or S3.
- Expand to the broader fleet only after validation.

**Exam takeaways**

- **Run Command** is for remote command execution on managed nodes.
- Multiple target selectors can be misleading if you assume they create an AND filter.
- Use **MaxConcurrency** and **MaxErrors** to control blast radius.
- Use **S3** and **CloudWatch Logs** for fleet-scale output collection.
- Know the difference between command status and invocation status.
- Cancellation stops further dispatch but does not instantly kill all running invocations.
- Pin document versions when repeatability matters.
- Private instances need network reachability to Systems Manager endpoints, not just the agent.
- Distinguish operator permissions from instance profile permissions and service-linked role behavior.
