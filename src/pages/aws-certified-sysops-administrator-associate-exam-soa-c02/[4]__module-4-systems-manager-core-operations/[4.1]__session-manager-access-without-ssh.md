## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**What Session Manager is**

AWS Systems Manager Session Manager lets you connect to managed instances without opening inbound ports such as `22` for SSH or `3389` for Remote Desktop Protocol (RDP). It uses the Systems Manager control plane plus the SSM Agent on the instance to create an authenticated, auditable session.

This is especially useful when you want to reduce attack surface in a private subnet, avoid bastion hosts, and centralize access control with AWS Identity and Access Management (IAM). For exam purposes, the key idea is that the instance initiates outbound communication to AWS, so you do not need inbound access from the internet or a VPN just to start a session.

**Why you use it**

Session Manager is a common operational choice when you need secure administrative access to EC2 instances in production.

- **No inbound SSH/RDP** reduces exposed network paths.
- **IAM-based access control** lets you grant or deny session access with policies.
- **Auditability** improves incident response because sessions can be logged.
- **Private subnet support** works well when instances have no public IP address.
- **Centralized operations** avoids managing bastion hosts for routine access.

A practical example is troubleshooting a private EC2 instance in a production virtual private cloud (VPC). Instead of temporarily opening port `22` in a security group, you attach the correct IAM role, confirm the SSM Agent is healthy, and start a session from the AWS CLI or console.

**How the connection works**

The connection flow is easy to remember:

1. You authenticate to AWS with IAM.
2. You request a session through Systems Manager.
3. The SSM Agent on the instance maintains outbound connectivity to the Systems Manager endpoints.
4. AWS brokers the session between your client and the instance.
5. You interact with the instance through the Session Manager channel.

A simple mental model is:

```text
You -> IAM auth -> Systems Manager -> SSM Agent on EC2 -> shell session
```

The important exam takeaway is that Session Manager does not require an inbound listener on the instance for the session itself. The instance must still be able to reach the Systems Manager service endpoints, either through the internet, a NAT gateway, or VPC endpoints.

**Core prerequisites**

To use Session Manager successfully, you need all of the following:

- **SSM Agent** installed and running on the instance.
- **IAM instance profile** attached to the instance.
- **IAM permissions** for the user or role starting the session.
- **Network connectivity** from the instance to Systems Manager endpoints.
- **Supported operating system** and instance configuration.

For EC2, the instance profile typically uses the AWS managed policy `AmazonSSMManagedInstanceCore`. That policy grants the permissions the agent needs to register and communicate with Systems Manager.

A typical instance role trust policy looks like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "ec2.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Attach the managed policy to the role:

```bash
aws iam attach-role-policy \
  --role-name MyEC2SSMRole \
  --policy-arn arn:aws:iam::aws:policy/AmazonSSMManagedInstanceCore
```

For the human operator, you also need permissions such as `ssm:StartSession` and often `ssm:DescribeSessions`. In tightly controlled environments, you can scope access to specific instances, tags, or documents.

**Typical IAM permissions for operators**

A minimal operator policy often includes session-start and read-only session visibility actions.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "StartAndInspectSessions",
      "Effect": "Allow",
      "Action": [
        "ssm:StartSession",
        "ssm:DescribeSessions",
        "ssm:GetConnectionStatus",
        "ssm:TerminateSession"
      ],
      "Resource": "*"
    }
  ]
}
```

In practice, you usually restrict `Resource` to specific instance ARNs and session documents. You can also use conditions such as tags to limit who can connect to which systems.

**Starting a session**

The most common way to open a shell session is with the AWS CLI:

```bash
aws ssm start-session --target i-0123456789abcdef0
```

This command starts a standard interactive session using Session Manager’s default behavior. You do not normally specify a special shell document just to get a basic session.

If you want to use the console, you can select the instance in Systems Manager and choose Session Manager. The console path is often easier for quick troubleshooting, while the CLI is better for automation and repeatability.

**Session types and related documents**

Session Manager supports more than one kind of session behavior. The exact document you use depends on the task.

| Use case | Typical document or behavior | Notes |
|---|---|---|
| Interactive shell access | Default `start-session` behavior | Most common for admin access |
| SSH tunneling | `AWS-StartSSHSession` | Used to tunnel SSH through Session Manager |
| Port forwarding | `AWS-StartPortForwardingSession` | Useful for databases, web apps, and internal services |
| Port forwarding to a remote host | `AWS-StartPortForwardingSessionToRemoteHost` | Useful when the instance acts as a jump point |

Example: port forward local port `3306` to a private database reachable from the instance.

```bash
aws ssm start-session \
  --target i-0123456789abcdef0 \
  --document-name AWS-StartPortForwardingSession \
  --parameters '{"portNumber":["3306"],"localPortNumber":["3306"]}'
```

Example: SSH tunneling through Session Manager when you still need SSH semantics but do not want inbound port `22` exposed.

```bash
aws ssm start-session \
  --target i-0123456789abcdef0 \
  --document-name AWS-StartSSHSession \
  --parameters '{"portNumber":["22"]}'
```

> Do not confuse Session Manager with opening an interactive shell over SSH. Session Manager can provide shell access without SSH, but SSH tunneling is a separate use case that still speaks SSH on the instance side.

**Logging and auditing**

One of the strongest reasons to use Session Manager is traceability. You can record session activity to Amazon CloudWatch Logs, Amazon Simple Storage Service (Amazon S3), or both, depending on your configuration.

This helps you:

- Review who connected and when.
- Preserve command output for investigations.
- Support compliance requirements.
- Detect suspicious administrative activity.

A common operational pattern is to enable session logging centrally and restrict direct shell access by policy. That way, even privileged access is visible and reviewable.

Example of enabling logging in Session Manager preferences through the Systems Manager console or a document-based configuration:

```json
{
  "cloudWatchLogGroupName": "/aws/ssm/session-manager",
  "s3BucketName": "my-ssm-session-logs",
  "s3KeyPrefix": "prod/",
  "kmsKeyId": "alias/ssm-session-logs"
}
```

The exact preference structure can vary by feature and console workflow, so treat the console-generated configuration as the source of truth. The exam focus is on the outcome: sessions can be logged and encrypted.

**Network requirements**

Session Manager works best when the instance can reach Systems Manager endpoints reliably.

Common connectivity patterns:

- **Internet access** through a public subnet and route to an internet gateway.
- **NAT gateway** for private subnets that need outbound internet access.
- **VPC endpoints** for private connectivity without internet exposure.

For private environments, interface VPC endpoints are often the preferred design. Typical endpoints include:

- `com.amazonaws.<region>.ssm`
- `com.amazonaws.<region>.ec2messages`
- `com.amazonaws.<region>.ssmmessages`

If you omit these endpoints in a private subnet with no NAT, the instance may appear offline in Systems Manager even if the IAM role is correct.

**Troubleshooting workflow**

When an instance does not show up as managed or you cannot start a session, check the problem in layers.

1. Confirm the instance is running and reachable from the AWS control plane.
2. Verify the instance profile is attached and uses `AmazonSSMManagedInstanceCore`.
3. Check that the SSM Agent is installed, running, and up to date.
4. Confirm outbound connectivity to Systems Manager endpoints.
5. Review security groups, network ACLs, and VPC endpoints.
6. Validate the operator’s IAM permissions for `ssm:StartSession`.

A very common failure is an outdated, stopped, or missing SSM Agent. Another common issue is a private subnet without NAT or the required VPC endpoints. Missing IAM permissions are also frequent, but do not overlook agent health.

Useful commands on Linux instances:

```bash
sudo systemctl status amazon-ssm-agent
sudo systemctl restart amazon-ssm-agent
sudo journalctl -u amazon-ssm-agent -n 50
```

On Amazon Linux 2 and many other Linux distributions, the agent service name is `amazon-ssm-agent`. On Windows, you can inspect the service in Services or with PowerShell.

```powershell
Get-Service AmazonSSMAgent
Restart-Service AmazonSSMAgent
```

**Real-world scenario**

Suppose you have a production EC2 instance in a private subnet that hosts an internal application. The instance has no public IP, and security policy forbids opening inbound SSH.

With Session Manager, you can:

- Attach an IAM role with `AmazonSSMManagedInstanceCore`.
- Ensure the instance has outbound access through a NAT gateway or VPC endpoints.
- Confirm the agent is running.
- Grant your admin role `ssm:StartSession`.
- Connect with `aws ssm start-session --target i-...`.

This gives you a secure break-glass path for maintenance without changing the network perimeter. If the application is down, you can inspect logs, restart services, or patch the host without ever exposing port `22`.

**Session Manager versus SSH and RDP**

| Capability | Session Manager | SSH | RDP |
|---|---|---|---|
| Inbound port required | No | Yes, usually `22` | Yes, usually `3389` |
| IAM integration | Yes | Not native | Not native |
| Session logging | Built in | External tooling needed | External tooling needed |
| Works well in private subnets | Yes | Only with network path | Only with network path |
| Best for browser/CLI admin access | Yes | Yes | No |
| Best for Windows GUI desktop | No | No | Yes |

Session Manager is not a replacement for every remote access method. It is best for secure administrative shell access and port forwarding. If you need a full graphical desktop on Windows, RDP still serves a different purpose.

**Exam-focused takeaways**

- **Session Manager** removes the need to open inbound SSH or RDP ports for administrative access.
- **AmazonSSMManagedInstanceCore** is the standard managed policy for the EC2 instance role.
- **SSM Agent** must be installed, running, and able to reach Systems Manager endpoints.
- **VPC endpoints** are a common solution for private subnets without internet access.
- **Logging** to CloudWatch Logs or S3 improves auditing and compliance.
- **`ssm:StartSession`** is the key operator permission for initiating access.
- **Port forwarding** and **SSH tunneling** are separate Session Manager use cases, not the same as a basic shell session.

**Quick reference commands**

```bash
# Start a basic Session Manager shell
aws ssm start-session --target i-0123456789abcdef0

# List active sessions
aws ssm describe-sessions --state Active

# Terminate a session
aws ssm terminate-session --session-id session-id-here
```

**Common mistakes to avoid**

- Assuming the instance role alone is enough without checking the agent.
- Treating Session Manager as if it requires inbound SSH.
- Using the wrong document for port forwarding or SSH tunneling.
- Forgetting private subnet connectivity to `ssm`, `ec2messages`, and `ssmmessages`.
- Overlooking operator IAM permissions when the instance itself is healthy.

**Memory aid**

Think of Session Manager as:

- **Identity** from IAM
- **Agent** on the instance
- **Outbound** connectivity from the host
- **Audit** through logging

That four-part model covers most exam questions and most real troubleshooting scenarios.
