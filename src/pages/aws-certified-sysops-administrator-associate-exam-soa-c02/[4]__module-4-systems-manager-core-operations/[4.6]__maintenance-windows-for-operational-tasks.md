## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**What maintenance windows are**

AWS Systems Manager <abbr title="Systems Manager">SSM</abbr> maintenance windows let you run operational tasks on a controlled schedule. You use them to reduce risk by limiting when disruptive actions happen, such as patching, reboots, configuration changes, or automation workflows.

A maintenance window is not the task itself. It is the scheduling and execution container that defines:

- when the window opens
- which managed nodes are in scope
- which tasks run
- task order and concurrency
- error thresholds and stop behavior

This topic is exam-relevant because it sits at the intersection of operations, patching, automation, and instance management.

**Core prerequisites**

Before a maintenance window can do useful work, the target resources and permissions must be ready.

- **Managed node registration** is required for most instance-based tasks.
  - The instance must be managed by Systems Manager.
  - The SSM Agent must be installed and running.
  - The instance must have network access to SSM endpoints, either through the internet or VPC endpoints.
- **IAM service role** is required for the maintenance window to run tasks.
  - The role must allow Systems Manager to assume it.
  - It must include permissions for the task type you register, such as Run Command, Automation, Lambda, or Step Functions where supported.
- **Targeting must be explicit**.
  - You can target instances directly or use resource tags.
  - You can also register window targets and reference them from tasks.

> If the instance is not managed by Systems Manager, the window can still exist, but instance-oriented tasks will not execute successfully.

**Maintenance window concepts**

A maintenance window has a few distinct parts that are easy to confuse in exam questions.

- **Window schedule**: when the window opens.
- **Window target**: the set of resources the window applies to.
- **Task**: the action that runs during the window.
- **Task priority**: the order in which tasks run.
- **Concurrency**: how many targets can be processed at once.
- **Error threshold**: how many failures are allowed before the task stops.

A useful mental model is:

- the window says *when*
- the target says *where*
- the task says *what*
- the service role says *who is allowed to do it*

**Scheduling behavior**

Maintenance windows support recurring schedules and one-time schedules. In practice, you usually define the schedule with a cron or rate expression.

- **cron expressions** are best when you need precise recurring schedules.
- **rate expressions** are best when you want a simple repeating interval.
- **UTC** is the safest assumption unless you explicitly manage time zone behavior in the service or console.
- **one-time windows** are useful for a single change event, migration, or emergency maintenance.

Typical scheduling examples:

```bash
# Every Sunday at 02:00 UTC
cron(0 2 ? * SUN *)

# Every 7 days
rate(7 days)
```

Practical scheduling guidance:

- Use **cron** when you need a specific day and time.
- Use **rate** when you care about elapsed time rather than calendar alignment.
- Prefer a dedicated scheduling service such as EventBridge Scheduler for pure scheduling use cases outside Systems Manager operations.
- Use maintenance windows when the schedule must directly govern Systems Manager operational tasks.

**Task types supported by maintenance windows**

Maintenance windows support several task types, and exam questions often test whether you know which one fits the use case.

- **RUN_COMMAND**
  - Runs an SSM document on managed nodes.
  - Common for shell scripts, PowerShell, package actions, and operational commands.
- **AUTOMATION**
  - Runs an SSM Automation document.
  - Best for multi-step workflows, approvals, and coordinated changes.
- **LAMBDA**
  - Invokes an AWS Lambda function during the window.
  - Useful when the operational action is implemented as serverless code.
- **STEP_FUNCTIONS**
  - Starts an AWS Step Functions state machine in supported contexts.
  - Useful for orchestrating more complex workflows.

Use the task type that matches the operational control plane you need:

- choose **RUN_COMMAND** for direct instance actions
- choose **AUTOMATION** for structured workflows and change orchestration
- choose **LAMBDA** for custom event-driven operational logic
- choose **STEP_FUNCTIONS** when the workflow spans multiple services and steps

**Targets and target registration**

Maintenance windows distinguish between the resources the window covers and the targets a task uses.

- **Window targets** define the resource set associated with the window.
- **Task targets** define which of those resources a specific task acts on.
- Tasks can reference window targets rather than repeating the same selection logic.

This matters because a single maintenance window can host multiple tasks with different scopes or priorities.

Common targeting patterns:

- target by **instance IDs** for a fixed set of nodes
- target by **tags** for dynamic fleet membership
- target by **resource group** when you want a broader operational grouping
- reference a **window target ID** from a task when you want the task to inherit the window’s target set

Example target registration with tags:

```bash
aws ssm register-target-with-maintenance-window \
  --window-id mw-0123456789abcdef0 \
  --resource-type INSTANCE \
  --targets Key=tag:PatchGroup,Values=ProdLinux \
  --owner-information "Production Linux fleet"
```

Example task registration that references a window target:

```bash
aws ssm register-task-with-maintenance-window \
  --window-id mw-0123456789abcdef0 \
  --task-type RUN_COMMAND \
  --task-arn AWS-RunShellScript \
  --service-role-arn arn:aws:iam::123456789012:role/SSMMaintenanceWindowRole \
  --priority 1 \
  --max-concurrency "10" \
  --max-errors "1" \
  --targets Key=WindowTargetIds,Values=target-abc123 \
  --task-invocation-parameters '{
    "RunCommand": {
      "Comment": "Apply standard maintenance script",
      "Parameters": {
        "commands": ["sudo yum update -y"]
      }
    }
  }'
```

> For `RUN_COMMAND`, the `--task-arn` value must be an SSM document name or ARN, such as `AWS-RunShellScript` or a custom document. Do not use a patching document as a generic command example.

**Task priority, concurrency, and error handling**

Task execution is controlled by priority and failure thresholds.

- **Priority**
  - Lower numbers run first.
  - Use this to ensure prerequisites happen before dependent tasks.
- **Max concurrency**
  - Limits how many targets are processed at the same time.
  - Helps reduce blast radius.
- **Max errors**
  - Stops the task when failures exceed the threshold.
  - Helps prevent widespread failure during a bad deployment or script issue.

Operationally, this means you can stage work safely:

- run a precheck task first
- patch a subset of nodes next
- reboot only if needed
- validate after the change

Example of a safe ordering strategy:

1. Inventory or health check task with priority `1`
2. Patch or configuration task with priority `2`
3. Validation task with priority `3`

**Run Command in maintenance windows**

`RUN_COMMAND` is the most common maintenance window task type for direct instance operations. It uses SSM documents to execute commands on managed nodes.

Typical use cases:

- install OS updates
- restart services
- rotate logs
- run health checks
- apply configuration changes

Example using a shell script document:

```bash
aws ssm register-task-with-maintenance-window \
  --window-id mw-0123456789abcdef0 \
  --task-type RUN_COMMAND \
  --task-arn AWS-RunShellScript \
  --service-role-arn arn:aws:iam::123456789012:role/SSMMaintenanceWindowRole \
  --priority 1 \
  --max-concurrency "5" \
  --max-errors "1" \
  --targets Key=WindowTargetIds,Values=target-abc123 \
  --task-invocation-parameters '{
    "RunCommand": {
      "Parameters": {
        "commands": ["sudo systemctl restart nginx"]
      }
    }
  }'
```

Use `RUN_COMMAND` when:

- the action is simple and instance-local
- you do not need a multi-step workflow
- you want direct command execution with minimal orchestration

**Automation in maintenance windows**

`AUTOMATION` is the better choice when the task needs multiple steps, branching, or approvals. Automation documents can coordinate actions across resources and services.

Typical use cases:

- patch orchestration with prechecks and postchecks
- snapshot creation before change
- coordinated instance stop/start workflows
- remediation with conditional logic

Example Automation task registration:

```bash
aws ssm register-task-with-maintenance-window \
  --window-id mw-0123456789abcdef0 \
  --task-type AUTOMATION \
  --task-arn AWS-CreateImage \
  --service-role-arn arn:aws:iam::123456789012:role/SSMMaintenanceWindowRole \
  --priority 1 \
  --max-concurrency "1" \
  --max-errors "0" \
  --targets Key=WindowTargetIds,Values=target-abc123 \
  --task-invocation-parameters '{
    "Automation": {
      "Parameters": {
        "InstanceId": ["{{RESOURCE_ID}}"]
      }
    }
  }'
```

Use `AUTOMATION` when:

- you need orchestration rather than a single command
- you want reusable operational runbooks
- you need safer change control with explicit steps

**Lambda and Step Functions tasks**

Maintenance windows can also invoke Lambda functions and, in supported contexts, Step Functions state machines.

- **Lambda**
  - Good for custom operational logic.
  - Useful when you want to integrate with APIs, tagging, notifications, or custom validation.
- **Step Functions**
  - Good for longer workflows with branching, retries, and service integration.
  - Useful when the maintenance activity spans multiple AWS services.

Exam distinction:

- do not treat these as arbitrary “other integrations”
- choose them only when the task type is explicitly supported and the workflow fits the service model

**Patching workflow details**

Maintenance windows are commonly used for patching fleets of managed nodes. Patching is not just “run updates”; it has a workflow and policy layer.

Key patching concepts:

- **Patch baseline**
  - Defines which patches are approved or rejected.
  - Can be managed by AWS or custom-defined.
- **Approval rules**
  - Control which patches become eligible.
  - Often based on classification, severity, product, or age.
- **Patch scan**
  - Detects missing patches without installing them.
  - Useful for compliance reporting and prechecks.
- **Patch install**
  - Applies approved patches.
  - May require a reboot depending on the patch and OS.

Common patching behavior:

- Linux patching often uses package manager behavior under the hood.
- Windows patching may require reboots more frequently.
- Reboot behavior should be planned, not assumed.
- A maintenance window can schedule patching, but the patch baseline determines what is actually eligible.

Example patching command with the standard patch baseline document:

```bash
aws ssm send-command \
  --document-name AWS-RunPatchBaseline \
  --targets "Key=tag:PatchGroup,Values=ProdLinux" \
  --parameters Operation=Install,RebootOption=RebootIfNeeded
```

Important patching notes:

- `Operation=Scan` checks compliance without installing.
- `Operation=Install` applies patches.
- `RebootOption=RebootIfNeeded` allows the instance to reboot if required.
- Patch groups and baselines must align for predictable results.

**End-to-end maintenance window workflow**

A complete operational flow usually looks like this:

1. Ensure the target instances are managed by Systems Manager.
2. Confirm the SSM Agent is installed and healthy.
3. Create or choose an IAM service role for maintenance window execution.
4. Define a maintenance window schedule.
5. Register window targets using instance IDs, tags, or resource groups.
6. Register one or more tasks with priority, concurrency, and error thresholds.
7. Validate execution history and task results after the window runs.

Example workflow commands:

```bash
# 1. Create a maintenance window
aws ssm create-maintenance-window \
  --name "prod-weekly-maintenance" \
  --schedule "cron(0 2 ? * SUN *)" \
  --duration 4 \
  --cutoff 1 \
  --allow-unassociated-targets

# 2. Register a target
aws ssm register-target-with-maintenance-window \
  --window-id mw-0123456789abcdef0 \
  --resource-type INSTANCE \
  --targets Key=tag:PatchGroup,Values=ProdLinux

# 3. Register a patching task
aws ssm register-task-with-maintenance-window \
  --window-id mw-0123456789abcdef0 \
  --task-type RUN_COMMAND \
  --task-arn AWS-RunPatchBaseline \
  --service-role-arn arn:aws:iam::123456789012:role/SSMMaintenanceWindowRole \
  --priority 1 \
  --max-concurrency "10" \
  --max-errors "1" \
  --targets Key=WindowTargetIds,Values=target-abc123 \
  --task-invocation-parameters '{
    "RunCommand": {
      "Parameters": {
        "Operation": ["Install"],
        "RebootOption": ["RebootIfNeeded"]
      }
    }
  }'

# 4. Inspect executions
aws ssm describe-maintenance-window-executions \
  --window-id mw-0123456789abcdef0
```

**Validation and troubleshooting**

After the window runs, you should verify both the window execution and the task execution details.

Useful commands:

```bash
aws ssm describe-maintenance-window-executions \
  --window-id mw-0123456789abcdef0

aws ssm describe-maintenance-window-execution-tasks \
  --window-execution-id mw-exec-0123456789abcdef0

aws ssm describe-maintenance-window-execution-task-invocations \
  --window-execution-id mw-exec-0123456789abcdef0 \
  --task-id task-0123456789abcdef0
```

Troubleshooting checklist:

- confirm the instance is a managed node
- confirm the SSM Agent is running
- confirm the IAM service role has the required permissions
- confirm the target selection matches the intended fleet
- confirm the task type matches the document or workflow
- confirm the schedule actually opened during the expected time window
- confirm the task IDs and window execution IDs belong to the same execution chain

**Common exam distinctions**

Maintenance windows are often compared with other AWS operational tools.

| Feature | Maintenance Windows | EventBridge Scheduler | EventBridge Rules |
|---|---|---|---|
| Primary purpose | Schedule Systems Manager operational tasks | Schedule actions and one-time or recurring jobs | React to events or simple schedules |
| Best for | Patching, Run Command, Automation, Lambda, Step Functions tasks | Pure scheduling use cases | Event-driven automation and simple scheduled triggers |
| Targeting | Managed nodes and registered targets | Service/API targets | Event patterns or schedules |
| Operational control | High, with task priority and error thresholds | Scheduling-focused | Event routing-focused |

Use this distinction carefully:

- choose **maintenance windows** when the schedule must govern Systems Manager operations
- choose **EventBridge Scheduler** when you mainly need a schedule
- choose **EventBridge Rules** when the trigger is event-driven or when a simple rule-based schedule is sufficient

**Service-specific maintenance scenarios**

Some AWS services have their own maintenance or patching controls, but they are not the same as Systems Manager maintenance windows.

- **Amazon RDS**
  - Uses maintenance windows for database engine and OS maintenance.
  - This is service-managed maintenance, not Systems Manager task orchestration.
- **Amazon EC2**
  - Uses Systems Manager maintenance windows for fleet operations.
  - This is where patching and command execution are commonly centralized.
- **Amazon EKS and other managed services**
  - May have their own upgrade or maintenance mechanisms.
  - Do not assume they use Systems Manager maintenance windows unless explicitly supported.

The exam may test whether you can distinguish:

- service-native maintenance windows
- Systems Manager maintenance windows
- general-purpose scheduling services

**Practical exam takeaways**

- Use maintenance windows to control when Systems Manager tasks run.
- Ensure instances are managed by Systems Manager before scheduling instance tasks.
- Use the correct task type for the job: `RUN_COMMAND`, `AUTOMATION`, `LAMBDA`, or `STEP_FUNCTIONS`.
- Remember that lower priority numbers run first.
- Use tags or window targets to scope work cleanly.
- Understand patch baselines, approval rules, scan versus install, and reboot behavior.
- Prefer EventBridge Scheduler for pure scheduling, and use maintenance windows when operational execution is the goal.
