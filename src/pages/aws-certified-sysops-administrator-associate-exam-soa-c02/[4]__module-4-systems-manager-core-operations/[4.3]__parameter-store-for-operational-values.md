## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Parameter Store overview**

AWS Systems Manager Parameter Store is a managed service for storing operational data such as configuration values, feature flags, AMI IDs, database endpoints, and secrets. You use it when you want centralized, hierarchical, and versioned values that applications, scripts, and automation can read at runtime.

Parameter Store is especially useful when you need:
- **Centralized configuration** across multiple accounts, Regions, or environments.
- **Version history** for controlled updates and rollback.
- **Encryption** for sensitive values using AWS Key Management Service (AWS KMS).
- **Integration** with AWS Systems Manager, AWS Lambda, Amazon EC2, AWS CloudFormation, and CI/CD pipelines.

A common pattern is to store non-secret operational values in plain text and sensitive values as `SecureString`. This lets you separate application configuration from code and reduce hard-coded settings in AMIs, containers, and deployment scripts.

**Core concepts**

Parameter Store organizes values as named parameters. Each parameter has a name, type, value, and optional metadata. Names are often hierarchical, which makes it easier to group values by application, environment, or component.

Typical hierarchy examples:
- `/prod/webapp/db/host`
- `/prod/webapp/db/password`
- `/dev/webapp/feature/new-checkout`
- `/shared/network/vpc-id`

Key parameter types:
- **String**: Plain text value for non-sensitive data.
- **StringList**: Comma-separated list of strings.
- **SecureString**: Encrypted value stored with AWS KMS.

Important metadata and behavior:
- **Versioning**: Every update creates a new version.
- **Labels**: Human-friendly pointers to versions, useful for promotion workflows.
- **Policies**: Optional expiration, no-change, or notification policies for governance.
- **Tiering**: Standard and Advanced tiers differ in size, throughput, and feature availability.

Parameter Store is not a full secrets manager replacement in every case. It works well for operational values and some secrets, but if you need automatic secret rotation, database credential lifecycle management, or richer secret-specific workflows, AWS Secrets Manager is often the better fit.

**When to use Parameter Store**

Use Parameter Store when you need configuration values that change independently of application code. This is common in immutable infrastructure, blue/green deployments, and multi-environment systems.

Good use cases:
- AMI IDs for launch templates or Auto Scaling groups.
- Application endpoints and ports.
- Feature flags and environment toggles.
- License keys or API tokens that do not require rotation workflows.
- Shared infrastructure values such as VPC IDs, subnet IDs, and security group IDs.
- Bootstrapping values for EC2 user data or container startup scripts.

Avoid using Parameter Store as a general-purpose database or for large binary objects. It is designed for small operational values, not bulk storage.

**Parameter hierarchy and naming**

Hierarchical naming is one of the most useful features for operational management. It lets you scope access and organize values by path.

Example hierarchy:
```text
/app1/dev/db/host
/app1/dev/db/name
/app1/dev/db/password
/app1/prod/db/host
/app1/prod/db/name
/app1/prod/db/password
```

Benefits of hierarchy:
- Easier discovery and grouping.
- Fine-grained IAM access control using path prefixes.
- Cleaner automation when scripts read a whole subtree.
- Better separation between environments.

Naming guidance:
- Use consistent prefixes such as `/org/app/env/component/key`.
- Keep names descriptive but not excessively long.
- Avoid embedding secrets in names.
- Use lowercase and hyphens or slashes consistently.

**Security and encryption**

`SecureString` parameters are encrypted at rest with AWS KMS. You can use the default AWS managed key for Systems Manager or a customer managed KMS key when you need tighter control, auditing, or cross-account design considerations.

Security practices:
- Store only sensitive values as `SecureString`.
- Restrict access with IAM policies scoped to parameter paths.
- Use KMS key policies and grants to control decryption.
- Avoid printing decrypted values in logs or shell history.
- Prefer least privilege for both `ssm:GetParameter` and `kms:Decrypt`.

Example IAM policy for read access to a specific path:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadAppParameters",
      "Effect": "Allow",
      "Action": [
        "ssm:GetParameter",
        "ssm:GetParameters",
        "ssm:GetParametersByPath"
      ],
      "Resource": "arn:aws:ssm:us-east-1:123456789012:parameter/app1/prod/*"
    },
    {
      "Sid": "DecryptSecureParameters",
      "Effect": "Allow",
      "Action": "kms:Decrypt",
      "Resource": "arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555"
    }
  ]
}
```

> If you grant access to `SecureString` parameters, you usually need both SSM read permissions and KMS decrypt permissions. Missing either one causes retrieval failures.

**Working with parameters from the CLI**

The AWS CLI is the most direct way to manage Parameter Store values during operations and troubleshooting.

Create a plain text parameter:
```bash
aws ssm put-parameter \
  --name "/app1/prod/api/endpoint" \
  --type "String" \
  --value "https://api.example.com" \
  --overwrite
```

Create a secure parameter:
```bash
aws ssm put-parameter \
  --name "/app1/prod/db/password" \
  --type "SecureString" \
  --value "SuperSecretValue" \
  --key-id "alias/aws/ssm" \
  --overwrite
```

Read a parameter:
```bash
aws ssm get-parameter \
  --name "/app1/prod/api/endpoint"
```

Read and decrypt a secure parameter:
```bash
aws ssm get-parameter \
  --name "/app1/prod/db/password" \
  --with-decryption
```

Read multiple parameters:
```bash
aws ssm get-parameters \
  --names "/app1/prod/api/endpoint" "/app1/prod/db/password" \
  --with-decryption
```

Read all parameters under a path:
```bash
aws ssm get-parameters-by-path \
  --path "/app1/prod/" \
  --recursive \
  --with-decryption
```

Delete a parameter:
```bash
aws ssm delete-parameter \
  --name "/app1/prod/api/endpoint"
```

Operational tips:
- Use `--overwrite` carefully because it creates a new version.
- Use `--with-decryption` only when you need the plaintext value.
- Prefer `get-parameters-by-path` for bootstrap scripts that load multiple related values.

**Versioning and labels**

Every update to a parameter creates a new version. This is useful for rollback and controlled promotion between environments.

Example workflow:
1. Store version 1 of `/app1/prod/config/timeout`.
2. Update the value to create version 2.
3. Attach a label such as `current` to version 2.
4. Roll back by moving the label back to version 1 if needed.

Labels are helpful when you want stable references without hard-coding a version number. This supports deployment workflows where you promote a tested configuration version into production.

Practical example:
```bash
aws ssm label-parameter-version \
  --name "/app1/prod/config/timeout" \
  --labels "current" \
  --parameter-version 2
```

**Policies and governance**

Parameter policies help you manage lifecycle and compliance for operational values. They are useful when values should expire, trigger notifications, or enforce change discipline.

Common policy types:
- **Expiration**: Parameter becomes invalid after a date.
- **ExpirationNotification**: Sends a notification before expiration.
- **NoChangeNotification**: Alerts when a parameter has not changed for a defined period.

Use policies when:
- A value must be rotated on a schedule.
- You want to detect stale configuration.
- You need operational reminders for temporary values.

These policies are especially useful for temporary credentials, maintenance windows, or time-bound feature toggles.

**Integration with EC2 and automation**

Parameter Store is frequently used during instance bootstrapping. EC2 user data, AWS Systems Manager Run Command, and automation documents can retrieve parameters at runtime so you do not bake environment-specific values into images.

Example user data snippet:
```bash
#!/bin/bash
yum install -y awscli

DB_HOST=$(aws ssm get-parameter \
  --name "/app1/prod/db/host" \
  --query "Parameter.Value" \
  --output text)

DB_PASSWORD=$(aws ssm get-parameter \
  --name "/app1/prod/db/password" \
  --with-decryption \
  --query "Parameter.Value" \
  --output text)

echo "DB_HOST=${DB_HOST}" >> /etc/app.env
echo "DB_PASSWORD=${DB_PASSWORD}" >> /etc/app.env
```

Operational considerations:
- Ensure the instance profile has permission to read the parameters.
- Ensure the instance profile can decrypt `SecureString` values.
- Avoid writing secrets to world-readable files.
- Prefer temporary in-memory use when possible.

**Integration with CloudFormation and deployment workflows**

CloudFormation can reference Parameter Store values to reduce template duplication and environment drift. This is useful for AMI IDs, subnet IDs, and other shared infrastructure values.

Example CloudFormation parameter reference:
```yaml
Parameters:
  LatestAmiId:
    Type: AWS::SSM::Parameter::Value<AWS::EC2::Image::Id>
    Default: /aws/service/ami-amazon-linux-latest/al2023-ami-kernel-default-x86_64

Resources:
  WebServer:
    Type: AWS::EC2::Instance
    Properties:
      ImageId: !Ref LatestAmiId
      InstanceType: t3.micro
```

This pattern helps you:
- Avoid hard-coding AMI IDs.
- Keep templates portable across Regions.
- Update infrastructure by changing a parameter rather than editing the template.

**Parameter Store vs Secrets Manager**

Both services store sensitive data, but they solve different operational problems.

| Capability | Parameter Store | Secrets Manager |
|---|---|---|
| Store configuration values | Yes | Yes |
| Store secrets | Yes, with `SecureString` | Yes |
| Automatic rotation | Limited / not native for general use | Yes |
| Versioning | Yes | Yes |
| Hierarchical naming | Yes | No path hierarchy |
| Fine-grained path access | Yes | Less path-oriented |
| Best for operational values | Yes | Sometimes |
| Best for secret lifecycle management | Sometimes | Yes |

Choose Parameter Store when:
- You need configuration and secrets in one simple hierarchy.
- You want low-cost storage for many operational values.
- You need easy integration with automation and path-based access.

Choose Secrets Manager when:
- You need built-in rotation.
- You manage database credentials or other secrets with lifecycle automation.
- You want secret-specific features and workflows.

**Common exam and operations scenarios**

In SysOps scenarios, Parameter Store often appears as the best answer when the requirement is to centralize operational values without changing application code.

Typical clues:
- “Store application configuration centrally.”
- “Use a secure, encrypted parameter for a password.”
- “Allow different environments to read different values.”
- “Avoid hard-coding AMI IDs in templates.”
- “Retrieve values at instance startup.”

Watch for distinctions:
- If the question emphasizes rotation, Secrets Manager may be the better choice.
- If the question emphasizes path-based organization and operational configuration, Parameter Store is usually the right answer.
- If the question asks for a value that changes frequently but is not secret, use `String`.
- If the question asks for a secret that must be encrypted, use `SecureString`.

**Troubleshooting and best practices**

When Parameter Store access fails, check permissions in layers:
- IAM permission to call SSM APIs.
- KMS permission to decrypt `SecureString`.
- Correct Region and account.
- Correct parameter path and spelling.
- Whether the parameter exists in the expected tier or environment.

Best practices:
- Use consistent naming conventions.
- Separate environments with path prefixes.
- Grant read access by path, not by individual parameter when appropriate.
- Use `SecureString` only for sensitive values.
- Keep secrets out of logs, shell history, and source control.
- Use labels for controlled promotion and rollback.
- Prefer automation over manual console edits for repeatability.

A practical operational pattern is to store shared values in a common path and environment-specific values in separate branches:
```text
/shared/network/vpc-id
/shared/network/route53-zone-id
/dev/app1/db/host
/prod/app1/db/host
```

This reduces duplication while preserving environment isolation.

**Quick reference**

- **String**: Plain text operational value.
- **StringList**: Comma-separated list of values.
- **SecureString**: Encrypted value using AWS KMS.
- **Version**: Immutable revision created on update.
- **Label**: Friendly alias for a version.
- **Path**: Hierarchical naming structure for organization and access control.
- **Policy**: Lifecycle rule such as expiration or notification.

**Summary**

Parameter Store is a strong fit for operational values that need centralized management, versioning, and optional encryption. It simplifies configuration delivery across EC2, Lambda, CloudFormation, and automation workflows while supporting secure access control through IAM and KMS. For SysOps, the key is to choose it when you need structured configuration and controlled retrieval, and to choose Secrets Manager when secret rotation and secret lifecycle management are the primary requirement.
