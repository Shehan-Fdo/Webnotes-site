## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Inventory collection overview**

AWS Systems Manager Inventory is a managed-node metadata collection feature. You use it to gather structured information about your fleet, such as installed applications, network configuration, services, OS details, and custom metadata. The main exam idea is that Inventory helps you answer “what is running where?” rather than “is it compliant?” or “what should be patched?”

Inventory data is collected from **managed nodes** registered with Systems Manager. In practice, that means the instance or server must be onboarded correctly before inventory can be gathered.

- **Managed node registration** is required before collection starts.
- **SSM Agent** must be installed and running on the node.
- The node needs an **IAM instance profile** or equivalent credentials that allow Systems Manager communication.
- The node must have network reachability to Systems Manager endpoints, either through the internet, NAT, or VPC endpoints.
- Inventory is typically collected by a **State Manager association** that runs on a schedule.

> Inventory is regional and account-scoped by default. If you need centralized reporting across multiple accounts or Regions, you usually add **Resource Data Sync** to export data to S3 for aggregation.

**Prerequisites and onboarding**

Before inventory works, you need the node to be visible to Systems Manager. In exam scenarios, failures often come from missing prerequisites rather than from the inventory feature itself.

Common prerequisites include:

- **SSM Agent** installed and updated on supported platforms.
- An **IAM role** attached to EC2 instances with permissions for Systems Manager.
- Proper **managed node activation** for on-premises or hybrid servers.
- Network access to Systems Manager, EC2 messages, and SSMMessages endpoints.
- A supported operating system and inventory type for the data you want to collect.

Typical EC2 instance profile permissions include the managed policy `AmazonSSMManagedInstanceCore`:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "ssm:UpdateInstanceInformation",
        "ssmmessages:*",
        "ec2messages:*",
        "logs:*"
      ],
      "Resource": "*"
    }
  ]
}
```

For hybrid nodes, you register the server as a managed instance using an activation code and ID, then install and start the agent.

**How collection works**

Inventory collection is usually configured as a scheduled association that runs the Systems Manager document for inventory gathering. The schedule controls freshness: more frequent runs give you more current data, but they also increase operational overhead.

A common pattern is:

1. Create or use the default inventory association.
2. Target the managed nodes or a tag-based fleet.
3. Set a schedule expression.
4. Let Systems Manager collect and store the inventory snapshot.

Example association command:

```bash
aws ssm create-association \
  --name "AWS-GatherSoftwareInventory" \
  --targets "Key=tag:Environment,Values=Prod" \
  --schedule-expression "rate(1 day)"
```

The exact association parameters can vary by document version and platform. In practice, inventory collection is often managed through the Systems Manager console or State Manager, where you can review targets, schedules, and execution history.

**Console workflow**

In the Systems Manager console, inventory setup is typically done through the Inventory area or through State Manager associations, depending on the workflow you choose.

A common console flow is:

1. Open **Systems Manager**.
2. Choose **Inventory** or **State Manager**.
3. Create or review an inventory association.
4. Select target instances by tags, resource groups, or instance IDs.
5. Choose the inventory types to collect.
6. Set the schedule and save.

The console is useful for exam scenarios where you need to confirm whether inventory is configured, which nodes are targeted, and whether the last run succeeded.

**Inventory types and schema**

Inventory is schema-driven. Each inventory type has a defined structure, and custom inventory must follow that schema. You cannot send arbitrary JSON and expect Systems Manager to accept it.

Common built-in inventory categories include:

- **AWS:Application** for installed software
- **AWS:InstanceInformation** for instance metadata
- **AWS:Network** for network details
- **AWS:WindowsUpdate** for Windows update-related data
- **AWS:Service** for service state information
- **AWS:File** for file metadata
- **AWS:AWSComponent** for Systems Manager component details

Supported inventory types vary by platform. For example, Windows nodes expose Windows-specific data such as services and updates, while Linux and macOS support different subsets of inventory categories. Always assume platform differences matter in exam questions.

Custom inventory is useful when built-in types do not capture the operational data you need, such as application version, owner, environment, or local configuration state.

Example custom inventory upload:

```bash
aws ssm put-inventory \
  --instance-id i-0123456789abcdef0 \
  --items '[
    {
      "TypeName": "Custom:AppMetadata",
      "SchemaVersion": "1.0",
      "CaptureTime": "2026-04-26T12:00:00Z",
      "Content": [
        { "Key": "ApplicationName", "Value": "payments-api" },
        { "Key": "Version", "Value": "2.4.1" },
        { "Key": "Owner", "Value": "platform-team" }
      ]
    }
  ]'
```

Important custom inventory rules:

- **TypeName** must be unique within the inventory namespace you use.
- **SchemaVersion** should match the schema you define and maintain.
- **Content** must match the expected key/value structure for the inventory type.
- Updates typically overwrite the latest values for that item type on the node.
- Versioning matters when you evolve the schema over time.

**Reporting and search**

Inventory is valuable because it gives you searchable fleet visibility. You can query by managed node, inventory type, and schema-supported attributes. The exact filter keys depend on the inventory type, so do not assume every field name is valid for every query.

Example query for application inventory:

```bash
aws ssm list-inventory-entries \
  --instance-id i-0123456789abcdef0 \
  --type-name "AWS:Application"
```

If you filter, the filter key must match a valid attribute in that inventory schema. For example, a filter on `Name` may work for one type but not another.

The console supports:

- Viewing inventory by **managed node**
- Filtering by **inventory type**
- Grouping by attributes such as OS, application name, or service state
- Searching for nodes that match a specific software or configuration profile

This is useful when you need to answer questions like:

- Which instances still have an old agent version?
- Which servers have a specific application installed?
- Which nodes are missing a required package?
- Which Windows hosts have a service stopped?

**Retention and data scope**

Inventory data is stored in Systems Manager and is available for operational reporting, but it is not a long-term data warehouse. For exam purposes, remember these scope ideas:

- Inventory is **regional**.
- Inventory is **account-scoped**.
- Inventory snapshots are useful for current-state reporting.
- Historical analysis is limited unless you export data externally.

If you need centralized, longer-term reporting, use **Resource Data Sync** to export Systems Manager data to an S3 bucket. This is the standard approach for aggregating inventory across multiple accounts or Regions.

A typical centralized reporting pattern looks like this:

- Source accounts and Regions collect inventory locally.
- Each source configures a Resource Data Sync.
- Data lands in a central S3 bucket.
- Analytics tools such as Athena, QuickSight, or custom ETL jobs query the exported data.

> Resource Data Sync centralizes reporting, but it does not remove the need to configure collection correctly in each source account and Region.

**Resource Data Sync considerations**

Resource Data Sync is often tested as the answer when you need fleet-wide reporting across accounts or Regions. The operational details matter:

- The destination is an **S3 bucket**.
- You need the right **bucket policy** and permissions.
- You must configure sync in each source scope you want to export from.
- The exported data format and structure depend on the sync configuration.

Use it when you need:

- Cross-account visibility
- Cross-Region aggregation
- Long-term retention outside Systems Manager
- Reporting pipelines for analytics tools

**Inventory vs compliance vs patching vs State Manager**

These features are easy to confuse on the exam. Inventory tells you what exists. Compliance tells you whether a resource meets a rule. Patch Manager helps you assess and apply patches. State Manager enforces desired state over time.

| Feature | Primary purpose | Typical output | Example use case |
|---|---|---|---|
| Inventory | Collect metadata about managed nodes | Installed apps, services, OS details | Find all hosts with a specific package |
| Compliance | Evaluate adherence to rules | Compliant / noncompliant status | Detect a missing security baseline |
| Patch Manager | Assess and install patches | Patch compliance and patch execution | Apply critical OS updates |
| State Manager | Maintain desired configuration | Association execution and state enforcement | Keep an agent installed and running |

A practical exam distinction:

- Use **Inventory** when you need discovery and reporting.
- Use **Compliance** when you need policy or baseline evaluation.
- Use **Patch Manager** when you need patch lifecycle management.
- Use **State Manager** when you need recurring configuration enforcement.

**Real-world scenarios**

Inventory is often the first step in remediation workflows.

Scenario 1: You need to find all servers running an outdated agent version.

- Use inventory to identify the affected nodes.
- Export or filter the results.
- Trigger a remediation runbook or patching workflow.

Scenario 2: Security asks which instances have a vulnerable application installed.

- Query application inventory.
- Group by application name and version.
- Use the result to scope patching or replacement.

Scenario 3: Operations wants a fleet-wide view of Windows services.

- Collect service inventory.
- Filter for stopped or disabled services.
- Use State Manager or automation to correct drift.

Scenario 4: You need a monthly software audit across multiple accounts.

- Collect inventory in each account and Region.
- Export with Resource Data Sync.
- Query the central S3 data set with Athena.

**Troubleshooting inventory collection**

When inventory is missing, troubleshoot from the node outward.

1. Confirm the node is registered as a managed node.
2. Verify the SSM Agent is installed and running.
3. Check the IAM role or hybrid activation permissions.
4. Confirm network access to Systems Manager endpoints.
5. Review the association schedule and execution history.
6. Validate that the inventory type and filter keys match the schema.

Useful CLI commands:

```bash
aws ssm describe-instance-information
aws ssm list-associations
aws ssm describe-association --association-id a-0123456789abcdef0
aws ssm list-association-executions --association-id a-0123456789abcdef0
aws ssm describe-association-execution-targets --association-id a-0123456789abcdef0 --execution-id 12345678-1234-1234-1234-123456789012
```

If the association exists but data is stale, check the schedule and the last execution result. If the node is not appearing at all, the problem is usually onboarding, permissions, or connectivity rather than inventory itself.

**Operational limits and best practices**

- Use tag-based targeting to scale inventory across fleets.
- Keep schedules frequent enough for operational needs, but not so frequent that they create unnecessary overhead.
- Use custom inventory only when built-in types do not meet the requirement.
- Keep custom schemas stable and version them carefully.
- Centralize reporting with Resource Data Sync when you need cross-account or cross-Region visibility.
- Treat inventory as a discovery tool, not as a compliance engine.

**Exam focus points**

- Inventory requires a managed node, SSM Agent, and correct IAM/network setup.
- Inventory collection is usually scheduled through State Manager.
- Inventory is regional and account-scoped by default.
- Resource Data Sync exports data to S3 for centralized reporting.
- Inventory is for discovery and reporting, not compliance enforcement.
- Custom inventory must follow schema rules and cannot accept arbitrary JSON.
- Console reporting supports filtering, grouping, and node-level inspection.

**Quick mental model**

Think of Systems Manager Inventory as the answer to:

- What software is installed?
- What services are present?
- What OS and configuration details exist?
- Which nodes match a given operational profile?

If the question is about drift enforcement, patching, or compliance status, you are likely in a different Systems Manager feature set.
