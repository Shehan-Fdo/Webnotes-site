## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Automation documents overview**

AWS Systems Manager Automation lets you define repeatable operational workflows as documents, then execute them against AWS resources or managed nodes. It is commonly used for remediation, maintenance, recovery, and controlled change tasks where you want a documented sequence with approvals, branching, and outputs.

Automation is a better fit than ad hoc scripting when you need:

- **Repeatability** across environments and accounts
- **Auditability** through execution history and step status
- **Safety controls** such as approvals, branching, and parameter validation
- **Integration** with other AWS APIs and Systems Manager capabilities

Automation documents are often used with Maintenance Windows, EventBridge, and Change Manager-style operational processes. For exam purposes, distinguish Automation from Run Command:

- **Automation** orchestrates multi-step workflows and can call AWS APIs, other automations, and scripts.
- **Run Command** is better for direct command execution on instances.
- **Maintenance Windows** schedule either Run Command tasks or Automation tasks, depending on the operation.

**Document types and schema**

Automation documents are Systems Manager documents, usually authored in YAML or JSON. The document schema defines the document type and the step model. Common document types include:

- **Automation** documents for orchestration workflows
- **Command** documents for Run Command
- **Policy** documents for state enforcement in some Systems Manager contexts

A typical Automation document includes:

- `schemaVersion`
- `description`
- `parameters`
- `mainSteps`

Example skeleton:

```yaml
schemaVersion: '0.3'
description: "Example Automation document"
parameters:
  InstanceId:
    type: String
    description: EC2 instance to operate on
mainSteps:
  - name: ExampleStep
    action: aws:executeAwsApi
    inputs:
      Service: ec2
      Api: DescribeInstances
      InstanceIds:
        - '{{ InstanceId }}'
```

Important syntax points:

- Use `{{ parameterName }}` for interpolation.
- Keep YAML indentation consistent; indentation errors are a common failure source.
- Step names must be unique within a document.
- Inputs are action-specific; do not assume one action accepts the same fields as another.

**Document ownership, sharing, and permissions**

Automation documents have ownership and sharing behavior that matters in multi-account environments.

- **Private documents** are owned by the account that created them and are only directly usable there unless shared.
- **Shared documents** can be shared with specific AWS accounts, organizational units, or made public depending on document sharing settings and governance controls.
- **Public sharing** should be treated carefully because it exposes the document definition to broader use.
- **Cross-account execution** depends on both document sharing and the permissions of the role used to execute the automation.

Practical exam point:

- If a document is shared to another account, the receiving account can execute it only if the document permissions and the execution role permissions both allow the required actions.
- The document may reference AWS resources in the target account, but the automation role must still have permission to call the underlying APIs.

**Execution modes and targeting**

Automation can run in different ways depending on how you invoke it.

- **Single execution** runs once with the supplied parameters.
- **Multi-target execution** runs the same document against multiple targets, often by instance IDs, tags, or resource groups where supported by the invocation method.
- **Rate controls** such as `MaxConcurrency` and `MaxErrors` help you control blast radius during multi-target operations.

Use multi-target execution when you want the same remediation or maintenance action applied to many resources. Use single execution when the workflow is resource-specific or when you need a controlled approval path.

Example CLI invocation:

```bash
aws ssm start-automation-execution \
  --document-name "MyAutomationDocument" \
  --parameters InstanceId=i-0123456789abcdef0
```

Example with multiple targets:

```bash
aws ssm start-automation-execution \
  --document-name "MyAutomationDocument" \
  --targets "Key=tag:Environment,Values=Prod" \
  --max-concurrency "10" \
  --max-errors "1"
```

Notes on targeting:

- Targeting semantics vary by invocation path and document design.
- Some documents are written to accept a single resource ID parameter rather than broad target selection.
- Resource groups can be used in some Systems Manager workflows, but do not assume every Automation document or execution mode supports them identically.

**Core step actions**

Automation documents use step actions to perform work. The most exam-relevant actions are:

- **`aws:executeAwsApi`**: Call an AWS service API directly.
- **`aws:executeScript`**: Run a script, commonly for data transformation or logic that is awkward in pure YAML.
- **`aws:executeAutomation`**: Invoke another Automation document from a parent runbook.
- **`aws:branch`**: Branch execution based on conditions.
- **`aws:assertAwsResourceProperty`**: Assert that a resource property matches an expected value.
- **`aws:waitForAwsResourceProperty`**: Poll until a resource property reaches a desired state.
- **`aws:approve`**: Approval step in supported Automation document contexts; use it carefully and verify the document schema and service support for the specific workflow.
- **`aws:changeInstanceState`**: Change EC2 instance state, such as stop or start, when appropriate.
- **`aws:createImage`**: Create an Amazon Machine Image (AMI) from an EC2 instance in Automation.
- **`aws:createSnapshot`**: Create an EBS snapshot.
- **`aws:runCommand`**: Run a Systems Manager command document from Automation.

A few practical distinctions:

- **`aws:executeAwsApi`** is the most flexible when you need to call a specific AWS API operation.
- **`aws:executeScript`** is useful for parsing outputs, building dynamic values, or making decisions that are cumbersome in YAML.
- **`aws:executeAutomation`** supports nested runbooks and reusable building blocks.
- **`aws:assertAwsResourceProperty`** is for validation, not remediation.
- **`aws:waitForAwsResourceProperty`** is for polling eventual consistency or asynchronous state changes.

Example `aws:executeAwsApi`:

```yaml
- name: DescribeInstance
  action: aws:executeAwsApi
  inputs:
    Service: ec2
    Api: DescribeInstances
    InstanceIds:
      - '{{ InstanceId }}'
  outputs:
    - Name: StateName
      Selector: '$.Reservations[0].Instances[0].State.Name'
      Type: String
```

**Outputs, selectors, and data passing**

Automation step outputs are extracted from the action response using selectors, usually JSONPath expressions. Outputs can then be passed to later steps.

Key ideas:

- **Selectors** point to a field in the returned JSON.
- **Type** defines how the output is interpreted, such as `String` or `StringList`.
- **Interpolation** passes values between steps using `{{ StepName.OutputName }}`.

Example:

```yaml
mainSteps:
  - name: GetInstance
    action: aws:executeAwsApi
    inputs:
      Service: ec2
      Api: DescribeInstances
      InstanceIds:
        - '{{ InstanceId }}'
    outputs:
      - Name: InstanceState
        Selector: '$.Reservations[0].Instances[0].State.Name'
        Type: String

  - name: CheckState
    action: aws:assertAwsResourceProperty
    inputs:
      Service: ec2
      Api: DescribeInstances
      InstanceIds:
        - '{{ InstanceId }}'
      PropertySelector: '$.Reservations[0].Instances[0].State.Name'
      DesiredValues:
        - running
```

Practical cautions:

- JSONPath selectors must match the actual response shape.
- If the API returns arrays, you often need to index into the first element.
- A selector that works for one API may fail for another because response structures differ.
- Output names are case-sensitive in interpolation.

**Branching and conditional logic**

Branching lets you choose different paths based on resource state or parameter values. Use it when the workflow must handle multiple scenarios, such as already-running instances, stopped instances, or different OS types.

Example pattern:

```yaml
- name: DecideNextStep
  action: aws:branch
  inputs:
    Choices:
      - NextStep: StartInstance
        Variable: '{{ InstanceState }}'
        StringEquals: stopped
      - NextStep: RebootInstance
        Variable: '{{ InstanceState }}'
        StringEquals: running
    Default: UnsupportedState
```

Use branching to:

- Skip unnecessary actions
- Avoid failing on expected states
- Route to different remediation paths

Be careful to validate the exact branching syntax for the document schema you are using. Automation branching is strict, and malformed conditions are a common source of document failure.

**Nested automation and reusable runbooks**

Nested automation is a strong design pattern when you want small reusable documents that compose into larger workflows.

Typical use cases:

- A parent runbook orchestrates approvals and high-level logic.
- A child runbook performs a specific task such as snapshot creation or instance recovery.
- Outputs from the child are passed back to the parent for follow-up actions.

Example parent step:

```yaml
- name: RunChildAutomation
  action: aws:executeAutomation
  inputs:
    DocumentName: ChildRunbook
    RuntimeParameters:
      InstanceId:
        - '{{ InstanceId }}'
  outputs:
    - Name: ChildResult
      Selector: '$.AutomationExecutionId'
      Type: String
```

Why this matters:

- It improves reuse across teams.
- It reduces duplication in large operational libraries.
- It makes testing easier because you can validate child workflows independently.

**Approval steps and human-in-the-loop workflows**

Automation can include approval gates in supported contexts. This is useful for production changes, destructive actions, or recovery steps that should not proceed without review.

Use approvals when:

- The action is irreversible or high risk
- You need a change-management checkpoint
- A human must confirm the context before continuing

Practical guidance:

- Treat approval support as document- and schema-dependent.
- Do not assume every Automation document uses the same approval behavior.
- Pair approvals with clear step descriptions and timeout handling.

**Common operational patterns**

Automation is frequently used for remediation and recovery. Common exam scenarios include:

- **Instance recovery** after failure detection
- **EBS snapshot and restore** workflows
- **AMI creation** before patching or major changes
- **Patch orchestration** with Maintenance Windows and Automation
- **Service restart or configuration drift correction**

Example: create an AMI before maintenance

```yaml
- name: CreatePreChangeImage
  action: aws:createImage
  inputs:
    InstanceId: '{{ InstanceId }}'
    ImageName: 'prechange-{{ InstanceId }}-{{ global:DATE_TIME }}'
    NoReboot: true
```

Notes:

- `aws:createImage` is an Automation action, not a direct `aws:executeAwsApi` wrapper.
- The action creates an AMI and associated snapshots, but you should not assume it returns snapshot IDs for cleanup logic.
- If you need snapshot-level cleanup, capture snapshot IDs from the appropriate snapshot-creating step or API response.

Example: create an EBS snapshot

```yaml
- name: CreateVolumeSnapshot
  action: aws:createSnapshot
  inputs:
    VolumeId: '{{ VolumeId }}'
    Description: 'Snapshot before change'
```

Example: recover an instance-oriented workflow

```yaml
- name: StopInstance
  action: aws:changeInstanceState
  inputs:
    InstanceIds:
      - '{{ InstanceId }}'
    DesiredState: stopped
```

Important distinction:

- A **stop/start cycle** is not the same as a **reboot**.
- Use stop/start when you need a full power cycle or to change instance attributes that require stopping.
- Use reboot when you want an in-place restart without changing the stopped state.

**Automation with Maintenance Windows**

Maintenance Windows schedule operational tasks at controlled times. They are often used with Automation for patching, reboots, and recurring maintenance.

Use Maintenance Windows when you need:

- A defined maintenance schedule
- Controlled concurrency and error thresholds
- Repeated execution during approved windows

Use Automation inside a Maintenance Window when the task requires orchestration rather than a single command.

Typical pattern:

1. Create or choose an Automation document.
2. Register the document as a Maintenance Window task.
3. Define targets, concurrency, and error thresholds.
4. Run the window on schedule.

This combination is common for patch orchestration:

- Maintenance Window controls when the task runs.
- Automation controls the sequence of actions.
- Patch Manager may use documents and Maintenance Windows as part of its orchestration model.

**Troubleshooting and execution history**

Automation execution history is essential for debugging. In the console or CLI, inspect:

- **Execution status**: success, failed, timed out, canceled, or in progress
- **Step status**: which step failed and why
- **Outputs**: whether the expected data was produced
- **Failure message**: API errors, permission issues, or selector mismatches

CLI examples:

```bash
aws ssm describe-automation-executions
aws ssm get-automation-execution --automation-execution-id a1b2c3d4-5678-90ab-cdef-EXAMPLE11111
```

What to check first:

- IAM permissions for the automation assume role
- Document syntax and indentation
- Parameter names and types
- API response selectors
- Resource state at execution time
- Whether the target resource exists in the expected Region

Common failure causes:

- Missing `iam:PassRole` permission for the execution role
- Incorrect JSONPath selector
- Wrong parameter type, such as passing a string where a list is expected
- Using an API field that does not exist for that action
- Attempting to operate on a resource in the wrong Region

**IAM and execution roles**

Automation often runs with an assumed role. That role needs permission to call the AWS APIs used by the document.

Key permissions concepts:

- The caller needs permission to start the automation.
- The automation service role needs permission to perform the actions in the document.
- If the document assumes another role, that role must trust Systems Manager and allow the required actions.

Practical example:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "ec2:DescribeInstances",
        "ec2:StopInstances",
        "ec2:StartInstances",
        "ec2:CreateImage"
      ],
      "Resource": "*"
    }
  ]
}
```

Remember:

- Least privilege still applies.
- A document can fail even if the document syntax is correct when the role lacks permission.
- Cross-account execution usually requires careful role trust and document sharing setup.

**CLI and document authoring workflow**

A practical authoring flow is:

1. Draft the document in YAML.
2. Validate indentation and parameter names.
3. Test against a non-production resource.
4. Review execution history and step outputs.
5. Refine selectors, branching, and permissions.

Useful CLI commands:

```bash
aws ssm create-document \
  --content file://automation-doc.yaml \
  --name MyAutomationDocument \
  --document-type Automation
```

```bash
aws ssm update-document \
  --content file://automation-doc.yaml \
  --name MyAutomationDocument
```

```bash
aws ssm start-automation-execution \
  --document-name MyAutomationDocument \
  --parameters InstanceId=i-0123456789abcdef0
```

Common YAML pitfalls:

- Mixing tabs and spaces
- Incorrect quoting around interpolation expressions
- Using the wrong list shape for parameters
- Forgetting that some inputs require arrays even for one value
- Reusing step names or output names

**Exam-focused comparisons**

| Feature | Best use | Key exam clue |
|---|---|---|
| Automation | Multi-step orchestration | Use for remediation, recovery, approvals, and nested workflows |
| Run Command | Direct command execution | Use for shell or PowerShell commands on managed nodes |
| Maintenance Windows | Scheduled operations | Use to control when Automation or Run Command tasks run |
| Patch Manager | Patch compliance and patching workflows | Uses Systems Manager documents and scheduling/orchestration features |

**Practical recovery example**

A common recovery pattern is to snapshot a volume, stop the instance if needed, make the change, and then start it again or recover service.

Example high-level flow:

1. Validate the instance state.
2. Create a snapshot or AMI.
3. Apply the change.
4. Verify the resource state.
5. Notify or hand off to a downstream workflow.

This pattern is useful because it gives you a rollback point before a risky operation. For exam questions, choose the workflow that matches the resource type and the desired recovery granularity:

- Use an **AMI** when you want to preserve an instance configuration for later launch.
- Use an **EBS snapshot** when you want to preserve block storage data.
- Use **reboot** when you only need an in-place restart.
- Use **stop/start** when you need a full power cycle or instance-state change.

**Best practices**

- Keep documents small and composable.
- Use parameters instead of hard-coded IDs.
- Validate outputs before using them in later steps.
- Prefer explicit state checks before destructive actions.
- Use approvals for production-impacting workflows.
- Test in a sandbox account or non-production Region first.
- Review execution history after every change.

- **Automation** = orchestration

- **Maintenance Window** = schedule
- **Nested automation** = reusable workflow blocks
