## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Multi-AZ VPC architecture overview**

A Multi-AZ <abbr title="Availability Zone">AZ</abbr> design places critical resources across two or more Availability Zones in the same AWS Region to reduce the impact of an AZ failure. In a <abbr title="Virtual Private Cloud">VPC</abbr>, this usually means creating matching public and private subnets in each AZ, then attaching routing, security, and load balancing layers that keep the application reachable even when one AZ becomes unavailable.

The main goal is not just redundancy, but predictable failover. You want traffic to continue flowing through healthy subnets, instances, and managed services without requiring manual intervention during a zone outage.

- Use at least two AZs for production workloads that need high availability.
- Keep subnet design symmetrical across AZs when possible.
- Place stateless tiers behind load balancers and keep state in managed services or replicated storage.
- Avoid single points of failure such as one NAT Gateway, one bastion host, or one application instance.

**Core VPC building blocks**

A resilient Multi-AZ VPC usually includes:
- A VPC with a non-overlapping CIDR block, such as `10.0.0.0/16`.
- Public subnets in each AZ for internet-facing components.
- Private application subnets in each AZ for EC2 instances, containers, or internal services.
- Private data subnets in each AZ for databases or tightly controlled back-end systems.
- An Internet Gateway for internet access from public subnets.
- One NAT Gateway per AZ for outbound internet access from private subnets.
- Route tables associated per subnet tier, not shared blindly across all subnets.

A subnet is considered **public** when its route table has a default route to an Internet Gateway. Public reachability also requires the resource to have a public IPv4 address or Elastic IP, and security groups or network ACLs must allow the traffic.

**Typical Multi-AZ topology**

A common pattern is:
- AZ-a:
  - Public subnet for load balancer nodes, bastion access, or NAT Gateway
  - Private app subnet for application servers
  - Private data subnet for database nodes
- AZ-b:
  - Public subnet for the same public-tier functions
  - Private app subnet for application servers
  - Private data subnet for database nodes

This layout supports:
- Horizontal scaling across zones
- Zone-aware routing
- Reduced blast radius
- Maintenance without full application downtime

**Routing design**

Route tables determine how traffic leaves each subnet. In a Multi-AZ design, you usually create separate route tables for public and private tiers, and often one private route table per AZ.

Example route intent:
- Public subnet route table:
  - `10.0.0.0/16 -> local`
  - `0.0.0.0/0 -> igw-xxxxxxxx`
- Private app subnet in AZ-a:
  - `10.0.0.0/16 -> local`
  - `0.0.0.0/0 -> nat-az-a`
- Private app subnet in AZ-b:
  - `10.0.0.0/16 -> local`
  - `0.0.0.0/0 -> nat-az-b`

This per-AZ NAT pattern prevents cross-AZ dependency for outbound internet access. If AZ-a fails, instances in AZ-b still have a local egress path.

```bash
aws ec2 create-route-table \
  --vpc-id vpc-0123456789abcdef0

aws ec2 create-route \
  --route-table-id rtb-0123456789abcdef0 \
  --destination-cidr-block 0.0.0.0/0 \
  --gateway-id igw-0123456789abcdef0

aws ec2 associate-route-table \
  --route-table-id rtb-0123456789abcdef0 \
  --subnet-id subnet-0123456789abcdef0
```

> A NAT Gateway is a managed service for outbound internet access from private subnets. A **public NAT Gateway** uses an Elastic IP, while a **private NAT Gateway** does not use an Elastic IP for internet egress.

**Internet access patterns**

Use the right egress pattern for the workload:
- Public subnet resources:
  - Use an Internet Gateway route
  - Assign a public IPv4 address or Elastic IP when direct internet reachability is required
- Private subnet resources:
  - Use a NAT Gateway for outbound-only internet access
  - Keep inbound access blocked unless explicitly required
- Internal-only services:
  - Use no internet route at all
  - Reach them through internal load balancers, VPC endpoints, or private connectivity

For AWS service access from private subnets, prefer **VPC endpoints** when possible. They reduce NAT traffic, lower cost, and keep service access on the AWS network.

Common endpoint choices:
- Gateway endpoints for S3 and DynamoDB
- Interface endpoints for services such as Systems Manager, CloudWatch Logs, ECR, and Secrets Manager

**NAT Gateway placement and resilience**

A NAT Gateway is AZ-specific. If you place one NAT Gateway in a single AZ and route all private subnets through it, you create a cross-AZ dependency and a failure domain. In a Multi-AZ architecture, deploy one NAT Gateway per AZ and route each private subnet to the NAT Gateway in the same AZ.

Benefits:
- Better availability during AZ impairment
- Lower cross-AZ data transfer exposure
- Cleaner failure isolation

Trade-offs:
- Higher cost because each NAT Gateway has hourly and data processing charges
- More route table management

If cost is a concern, you can sometimes centralize egress, but that weakens availability. For exam scenarios, prefer the design that best matches the availability requirement.

**Elastic IP usage**

An **Elastic IP** is a static public IPv4 address that you can remap to another resource. It is commonly used for:
- Public NAT Gateways
- Bastion hosts that need a stable public address
- Legacy systems that require fixed inbound allowlists

Do not assume every public-facing resource needs an Elastic IP. Many workloads use load balancers with DNS names instead. Also, a NAT Gateway only needs an Elastic IP when it is a public NAT Gateway.

**Load balancing across AZs**

For internet-facing or internal applications, use an Elastic Load Balancing service to distribute traffic across targets in multiple AZs.

Common patterns:
- Application Load Balancer for HTTP/HTTPS and Layer 7 routing
- Network Load Balancer for TCP/UDP and very high throughput
- Gateway Load Balancer for network appliances

Best practices:
- Enable cross-zone load balancing when appropriate for even traffic distribution
- Register targets in multiple AZs
- Keep health checks aligned with application readiness
- Avoid placing the only load balancer target in one AZ

A load balancer helps absorb AZ failure by sending traffic only to healthy targets in remaining zones.

**Database and stateful tier design**

Stateful components need special attention because failover is not always automatic at the instance level.

Common AWS-managed options:
- Amazon RDS Multi-AZ for synchronous standby replication and automatic failover
- Amazon Aurora with multiple AZs and reader endpoints
- Amazon ElastiCache with replication groups and Multi-AZ failover where supported

Design guidance:
- Put database subnets in at least two AZs
- Use a DB subnet group spanning multiple AZs
- Keep application code stateless when possible
- Store session state in managed stores such as ElastiCache or DynamoDB if the app must scale horizontally

For self-managed databases on EC2:
- Replicate data across AZs
- Automate failover
- Test recovery procedures regularly

**Security layers**

Security in a Multi-AZ VPC is layered:
- **Security groups** act as stateful instance-level firewalls
- **Network ACLs** act as stateless subnet-level filters
- IAM controls who can manage the infrastructure
- Route tables control where traffic can go

Practical guidance:
- Use security groups for most application access control
- Use network ACLs only when you need coarse subnet-level restrictions or explicit deny rules
- Keep inbound rules minimal on public subnets
- Restrict administrative access through VPN, AWS Systems Manager Session Manager, or tightly controlled bastions

Example security group pattern:
- Load balancer SG:
  - Allow `443` from `0.0.0.0/0`
- App SG:
  - Allow app port from load balancer SG only
- DB SG:
  - Allow database port from app SG only

```bash
aws ec2 authorize-security-group-ingress \
  --group-id sg-0123456789abcdef0 \
  --protocol tcp \
  --port 443 \
  --cidr 0.0.0.0/0

aws ec2 authorize-security-group-ingress \
  --group-id sg-0abcdef0123456789 \
  --protocol tcp \
  --port 3306 \
  --source-group sg-0123456789abcdef0
```

**Deployment workflow**

A repeatable deployment sequence helps avoid miswired routes and uneven subnet layouts.

1. Create the VPC with a CIDR block that leaves room for growth.
2. Create at least two public subnets and two private subnets across different AZs.
3. Attach an Internet Gateway to the VPC.
4. Create route tables for public and private tiers.
5. Add the default route to the Internet Gateway for public subnets.
6. Deploy one NAT Gateway per AZ in public subnets.
7. Point each private subnet route table to the NAT Gateway in the same AZ.
8. Create security groups for load balancer, application, and database tiers.
9. Deploy the load balancer across public subnets in multiple AZs.
10. Place application instances or services in private subnets across multiple AZs.
11. Configure database or stateful services for Multi-AZ failover.
12. Validate failover by simulating instance, subnet, or AZ impairment.

**Example CLI workflow**

The exact commands vary by resource type, but the following pattern is representative.

```bash
# Create a VPC
aws ec2 create-vpc --cidr-block 10.0.0.0/16

# Create subnets in different AZs
aws ec2 create-subnet \
  --vpc-id vpc-0123456789abcdef0 \
  --cidr-block 10.0.1.0/24 \
  --availability-zone us-east-1a

aws ec2 create-subnet \
  --vpc-id vpc-0123456789abcdef0 \
  --cidr-block 10.0.2.0/24 \
  --availability-zone us-east-1b

# Attach an Internet Gateway
aws ec2 create-internet-gateway
aws ec2 attach-internet-gateway \
  --vpc-id vpc-0123456789abcdef0 \
  --internet-gateway-id igw-0123456789abcdef0
```

For NAT Gateway creation, verify the current AWS CLI syntax for your version and region. The general workflow is:
- Allocate an Elastic IP for a public NAT Gateway
- Create the NAT Gateway in a public subnet
- Update the private route table to point `0.0.0.0/0` to that NAT Gateway

**Validation and troubleshooting**

When a Multi-AZ deployment fails, check the layers in order:
- DNS and load balancer health
- Security group rules
- Route table associations
- NAT Gateway placement and status
- Subnet CIDR overlap or exhaustion
- Instance health checks and application readiness
- AZ-specific service limits or failures

Useful checks:
- Confirm the subnet is associated with the intended route table.
- Confirm the public subnet has a route to the Internet Gateway.
- Confirm the instance has a public IPv4 address or Elastic IP if direct internet access is expected.
- Confirm private subnets use a NAT Gateway in the same AZ when possible.
- Confirm target groups show healthy targets in more than one AZ.

Example inspection commands:
```bash
aws ec2 describe-route-tables \
  --filters Name=vpc-id,Values=vpc-0123456789abcdef0

aws ec2 describe-nat-gateways \
  --filter Name=vpc-id,Values=vpc-0123456789abcdef0

aws elbv2 describe-target-health \
  --target-group-arn arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/app-tg/abcdef1234567890
```

**Common exam traps**

- A subnet is not public just because it is named “public”; it must have a route to an Internet Gateway.
- A resource in a public subnet is not reachable from the internet unless it also has a public IPv4 address or Elastic IP and the security controls allow it.
- One NAT Gateway for all AZs is simpler, but it reduces availability.
- A NAT Gateway is for outbound internet access from private subnets, not for inbound publishing.
- Cross-zone load balancing does not replace Multi-AZ target placement.
- RDS Multi-AZ is for high availability, not read scaling; use read replicas or Aurora readers for read-heavy workloads.
- VPC endpoints can reduce NAT usage and improve security, but they do not replace internet access for arbitrary external destinations.

**Design principles to remember**

- Build for failure at the AZ level, not just the instance level.
- Keep each tier present in multiple AZs.
- Prefer managed services that handle failover for you.
- Minimize cross-AZ dependencies where they are not needed.
- Use routing, security, and health checks together; none of them alone provides high availability.

**Quick reference**

| Component | Multi-AZ recommendation | Why it matters |
|---|---|---|
| Subnets | At least two AZs per tier | Reduces AZ blast radius |
| Internet access | IGW for public, NAT per AZ for private | Preserves reachability and isolation |
| Load balancer | Span multiple AZs | Keeps traffic flowing during failures |
| Database | Multi-AZ or replicated design | Supports automatic failover |
| Security groups | Tier-specific rules | Limits lateral movement |
| VPC endpoints | Use for AWS services when possible | Reduces NAT cost and exposure |

**Summary**

Multi-AZ VPC architecture is about combining subnet placement, routing, load balancing, and managed failover so that one AZ outage does not take down the application. For SysOps exam scenarios, choose designs that preserve availability, avoid single points of failure, and keep public, private, and data tiers clearly separated.
