## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Why VPC endpoints matter for Systems Manager and S3**

VPC endpoints let you keep traffic to supported AWS services on the AWS network instead of sending it through an internet gateway or NAT gateway. For exam purposes, the key idea is not “private access to everything,” but private connectivity to specific services with the right endpoint type and supporting configuration.

For Systems Manager (SSM), endpoints are especially important when you manage EC2 instances in private subnets with no internet route. For S3, endpoints let instances reach buckets without public internet egress, which is useful for patching, artifact retrieval, backups, and application data access.

**Core endpoint types**

There are two endpoint families you should distinguish clearly:

- **Gateway endpoints**
  - Supported for Amazon S3 and DynamoDB.
  - Added to route tables.
  - Use a prefix list destination for the service and a VPC endpoint target in the route table.
  - Do not use security groups.
- **Interface endpoints**
  - Powered by AWS PrivateLink.
  - Appear as elastic network interfaces (ENIs) in your subnets.
  - Use private IP addresses in your VPC.
  - Can be secured with security groups.
  - Common for Systems Manager services.

A useful exam shortcut is this: S3 typically uses a gateway endpoint, while SSM uses interface endpoints.

**Systems Manager private connectivity pattern**

A private EC2 instance that uses Systems Manager Session Manager or Run Command usually needs access to several SSM-related service endpoints. The exact set depends on the feature, but the common pattern is to create interface endpoints for the Systems Manager control plane and related dependencies.

Typical interface endpoints include:

- `com.amazonaws.<region>.ssm`
- `com.amazonaws.<region>.ec2messages`
- `com.amazonaws.<region>.ssmmessages`

In many environments, you also add:

- `com.amazonaws.<region>.logs` for CloudWatch Logs integration
- `com.amazonaws.<region>.kms` if you use AWS Key Management Service (KMS) for encryption-related workflows
- `com.amazonaws.<region>.s3` as a gateway endpoint for patching content, documents, or artifacts that are stored in S3

The instance does not need inbound SSH or RDP for Session Manager. That is one of the main operational benefits: you can administer instances without opening management ports to the internet or a bastion host.

**How Session Manager traffic flows**

Session Manager uses the SSM Agent on the instance. The agent communicates outbound to AWS services. In a private subnet, that outbound path can stay inside the VPC if you provide the required interface endpoints.

A typical flow looks like this:

- Your workstation calls the Systems Manager API.
- The SSM service coordinates the session.
- The instance’s SSM Agent connects to the message channels through `ssmmessages` and related endpoints.
- Traffic stays private inside the AWS network path rather than using public internet routing.

This is not a replacement for VPN in general. VPN is a separate private connectivity mechanism for reaching your VPC from on-premises or another network. VPC endpoints solve service connectivity from within the VPC.

**S3 gateway endpoint behavior**

An S3 gateway endpoint is created in a Region and serves S3 traffic for that Region’s S3 service path. The important exam point is that it is Region-scoped and integrated with route tables, not that it behaves like a normal next-hop route to an instance or appliance.

When you create the endpoint:

- You associate it with one or more route tables.
- AWS adds a route for the S3 prefix list to the endpoint target.
- Instances in subnets using those route tables can reach S3 privately.

A route table entry is conceptually like this:

```text
Destination: pl-xxxxxxxx  -> Target: vpce-xxxxxxxx
```

That is a simplified representation. The key is that the destination is the S3 managed prefix list, and the target is the gateway endpoint. It is not a generic route to an IP address.

**Practical SSM and S3 architecture**

A common private management design for a VPC looks like this:

- Private subnets host EC2 instances.
- Interface endpoints for SSM services are deployed in those subnets.
- A gateway endpoint for S3 is associated with the private subnet route tables.
- Security groups allow the instance to reach the interface endpoint ENIs on HTTPS port 443.
- No internet gateway or NAT gateway is required for the management path itself.

This design is especially useful for:

- Highly restricted production environments
- Regulated workloads
- Instances in isolated subnets
- Patch and inventory workflows that must avoid public internet egress

**Example endpoint setup with AWS CLI**

You can create the interface endpoints and the S3 gateway endpoint with the AWS CLI. Replace the placeholders with your VPC, subnet, and route table IDs.

```bash
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0123456789abcdef0 \
  --vpc-endpoint-type Interface \
  --service-name com.amazonaws.us-east-1.ssm \
  --subnet-ids subnet-aaa111 subnet-bbb222 \
  --security-group-ids sg-0123456789abcdef0 \
  --private-dns-enabled

aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0123456789abcdef0 \
  --vpc-endpoint-type Interface \
  --service-name com.amazonaws.us-east-1.ec2messages \
  --subnet-ids subnet-aaa111 subnet-bbb222 \
  --security-group-ids sg-0123456789abcdef0 \
  --private-dns-enabled

aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0123456789abcdef0 \
  --vpc-endpoint-type Interface \
  --service-name com.amazonaws.us-east-1.ssmmessages \
  --subnet-ids subnet-aaa111 subnet-bbb222 \
  --security-group-ids sg-0123456789abcdef0 \
  --private-dns-enabled

aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0123456789abcdef0 \
  --vpc-endpoint-type Gateway \
  --service-name com.amazonaws.us-east-1.s3 \
  --route-table-ids rtb-0123456789abcdef0 rtb-0fedcba9876543210
```

**DNS settings you should verify**

Private DNS is important for interface endpoints. When enabled, standard AWS service hostnames resolve to the private IP addresses of the endpoint ENIs inside your VPC.

Check the VPC DNS attributes with the correct top-level fields:

```bash
aws ec2 describe-vpcs \
  --vpc-ids vpc-0123456789abcdef0 \
  --query 'Vpcs[0].{DnsSupport:EnableDnsSupport,DnsHostnames:EnableDnsHostnames}'
```

You generally want both of these enabled for private DNS-based endpoint access:

- **EnableDnsSupport**: allows DNS resolution in the VPC
- **EnableDnsHostnames**: enables DNS hostnames for resources in the VPC

If private DNS is disabled on an interface endpoint, clients may still reach the endpoint using its specific DNS names, but standard service names may continue to resolve publicly.

**Security group and network considerations**

Interface endpoints are ENIs, so security groups matter.

- Allow inbound TCP 443 from the instance security group to the endpoint security group.
- Allow outbound TCP 443 from the instance if your egress rules are restrictive.
- Ensure NACLs do not block ephemeral return traffic.
- Place endpoints in subnets that are reachable from the workloads that need them.

Gateway endpoints do not use security groups. Access control is enforced through route tables and endpoint policies.

**Endpoint policies**

Endpoint policies let you restrict which actions or resources can use the endpoint.

For S3, you can limit access to specific buckets or prefixes. For interface endpoints, policies can restrict which AWS API actions are allowed through the endpoint.

Example S3 endpoint policy:

```json
{
  "Statement": [
    {
      "Sid": "AllowReadOnlyToSpecificBucket",
      "Effect": "Allow",
      "Principal": "*",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-private-bucket",
        "arn:aws:s3:::my-private-bucket/*"
      ]
    }
  ]
}
```

Use endpoint policies as an additional control, not as the only control. IAM permissions and bucket policies still matter.

**SSM feature dependencies and S3**

Be precise about what needs S3.

- **Session Manager connectivity**
  - Primarily depends on SSM-related interface endpoints.
  - Does not universally require S3 just to start a session.
- **Patch Manager and document/artifact workflows**
  - Often require S3 access for patch content, documents, or supporting artifacts.
  - This is why S3 gateway endpoints are commonly paired with SSM endpoints in private subnets.
- **Inventory and automation workflows**
  - May involve S3 depending on how you store or retrieve documents and outputs.

The exam-friendly takeaway is that SSM connectivity and S3 access are related but distinct requirements.

**Troubleshooting scenario**

Imagine a private EC2 instance in a subnet with no NAT gateway. You want to use Session Manager, but the instance never appears as managed in Systems Manager.

A practical troubleshooting path is:

1. Confirm the instance has the SSM Agent installed and running.
2. Verify the instance profile includes the `AmazonSSMManagedInstanceCore` policy or equivalent permissions.
3. Check that the VPC has the required interface endpoints for `ssm`, `ec2messages`, and `ssmmessages`.
4. Confirm private DNS is enabled on the interface endpoints.
5. Verify the endpoint security group allows inbound 443 from the instance security group.
6. Check route tables and confirm the subnet is associated with the correct tables.
7. If patching or document retrieval fails, confirm S3 access through a gateway endpoint or another approved path.

This kind of issue often looks like a “network problem,” but the root cause may be IAM, DNS, security groups, or missing endpoints.

**End-to-end example: private patching workflow**

A common real-world use case is patching a fleet of private instances.

- The instances run in isolated or private subnets.
- Session Manager provides administrative access without SSH.
- SSM Agent communicates through interface endpoints.
- Patch content or related artifacts are retrieved from S3 through a gateway endpoint.
- Logs can be sent to CloudWatch Logs through an interface endpoint if you want centralized auditing.

This design gives you a controlled management plane with no public IP addresses on the instances.

**Comparison of endpoint choices**

| Need | Best fit | Why |
|---|---|---|
| Private access to SSM APIs | Interface endpoints | SSM uses PrivateLink-style service endpoints |
| Private access to S3 | Gateway endpoint | Route-table-based, no security groups required |
| Private admin access without SSH | Session Manager + SSM endpoints | Removes inbound management ports |
| Private patch/artifact retrieval | S3 gateway endpoint | Keeps S3 traffic off the internet path |

**Exam points to remember**

- **Interface endpoints** are used for Systems Manager services.
- **Gateway endpoints** are used for S3 and DynamoDB.
- Private DNS is important for seamless service-name resolution with interface endpoints.
- S3 gateway endpoints are associated with route tables, not security groups.
- Session Manager reduces the need for bastions and inbound SSH/RDP.
- SSM management and S3 access are separate concerns, though many private workflows need both.
- VPN and VPC endpoints solve different problems and are not interchangeable.

**Quick verification commands**

Use these commands to inspect your setup:

```bash
aws ec2 describe-vpc-endpoints \
  --filters Name=vpc-id,Values=vpc-0123456789abcdef0 \
  --query 'VpcEndpoints[*].{Id:VpcEndpointId,Type:VpcEndpointType,Service:ServiceName,State:State,PrivateDns:PrivateDnsEnabled}'
```

```bash
aws ec2 describe-route-tables \
  --route-table-ids rtb-0123456789abcdef0 \
  --query 'RouteTables[0].Routes'
```

```bash
aws ssm describe-instance-information \
  --query 'InstanceInformationList[*].{Id:InstanceId,Platform:PlatformName,Ping:PingStatus}'
```

These checks help you confirm endpoint presence, route association, and whether the instance is actually managed by Systems Manager.

**Common mistakes**

- Assuming an S3 gateway endpoint alone enables Session Manager.
- Forgetting `ec2messages` or `ssmmessages` when building private SSM access.
- Enabling the endpoint but leaving private DNS disabled.
- Expecting a gateway endpoint to behave like a normal routed next hop.
- Testing endpoint usage with a generic S3 command and assuming the result proves the endpoint path.
- Overlooking IAM permissions on the instance profile.

**Summary**

For SOA-C02, focus on the relationship between private subnets, Systems Manager, and S3. Use interface endpoints for SSM services, use a gateway endpoint for S3, and verify DNS, route tables, security groups, and IAM together. The goal is private, reliable administration and data access without exposing management ports or forcing traffic through the public internet.
