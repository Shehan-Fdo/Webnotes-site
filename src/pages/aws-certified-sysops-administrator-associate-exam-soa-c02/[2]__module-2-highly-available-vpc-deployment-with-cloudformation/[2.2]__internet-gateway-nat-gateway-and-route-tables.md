## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Core networking model**

In Amazon Virtual Private Cloud (Amazon VPC), route tables decide where traffic goes, while gateways provide the actual path to external networks. For exam purposes, you should separate three ideas:

- **Internet Gateway (IGW)** enables direct internet routing for resources in a VPC.
- **NAT Gateway** enables outbound-only internet access for private IPv4 instances.
- **Route tables** determine which destination CIDR blocks use which target.

A subnet is often described as *public* when its route table contains a route to an IGW for internet-bound traffic. In practice, an instance also needs a public IPv4 address and permissive security group and network ACL rules to be reachable from the internet. A subnet is *private* when it does not have a direct route to an IGW for its internet traffic.

**Internet Gateway fundamentals**

An IGW is a horizontally scaled, highly available VPC component that allows communication between instances in your VPC and the internet. It supports both IPv4 and IPv6, but the behavior differs:

- For **IPv4**, instances need a public IPv4 address or an Elastic IP address for inbound internet reachability.
- For **IPv6**, instances use globally routable IPv6 addresses; there is no NAT for IPv6 through an IGW.
- The IGW is attached to a VPC, not to a subnet.

A common exam trap is assuming that attaching an IGW alone makes instances reachable. It does not. You still need:

- A route in the subnet’s route table pointing the relevant destination to the IGW.
- A public IPv4 address or Elastic IP for IPv4 inbound access.
- Security group and network ACL rules that allow the traffic.

**Route table behavior**

Route tables are the decision engine for subnet traffic. Each subnet can be associated with only one route table at a time, but one route table can be associated with multiple subnets. If you do not explicitly associate a subnet with a custom route table, it uses the VPC’s main route table.

AWS selects routes using **longest-prefix match**:

- The most specific matching route wins.
- This applies to both static and propagated routes.
- A more specific propagated route can beat a less specific static route.

For example, if a route table contains both `10.0.0.0/16 -> local` and `10.0.1.0/24 -> eni-1234567890abcdef0`, traffic to `10.0.1.25` uses the `/24` route because it is more specific.

A route table commonly includes:

- A **local** route for the VPC CIDR block.
- A default route such as `0.0.0.0/0` for IPv4 internet-bound traffic.
- An IPv6 default route such as `::/0` when IPv6 internet access is required.
- Routes to peering connections, Transit Gateway, VPN, or Direct Connect attachments.

> Route tables do not “make” a subnet public or private by themselves in an absolute sense; they define the next hop. Public reachability still depends on addressing and security configuration.

**Internet Gateway route patterns**

For a public subnet, the route table usually includes a default route to the IGW:

```text
Destination      Target
10.0.0.0/16      local
0.0.0.0/0        igw-0abc1234def567890
```

For IPv6-enabled public subnets, you typically add:

```text
Destination              Target
2001:db8:1234:1::/64     local
::/0                     igw-0abc1234def567890
```

The exact local route depends on the VPC’s assigned CIDR blocks. AWS creates the local route automatically for each VPC CIDR association.

Use an IGW when you need:

- Inbound internet access to public-facing instances.
- Outbound internet access from instances that already have public IPv4 addresses.
- IPv6 internet connectivity for instances with IPv6 addresses.

**NAT Gateway fundamentals**

A NAT Gateway provides network address translation for instances in a private subnet so they can initiate outbound IPv4 connections to the internet, while preventing unsolicited inbound connections from the internet.

Key points:

- NAT Gateway supports **IPv4 only**.
- It is created in a **public subnet**.
- The public subnet must have a route to an IGW.
- The NAT Gateway must have an Elastic IP address.
- Private subnets route `0.0.0.0/0` to the NAT Gateway, not to the IGW.

This dependency chain is important in exam scenarios:

1. Create or identify a public subnet.
2. Ensure the public subnet’s route table sends `0.0.0.0/0` to the IGW.
3. Place the NAT Gateway in that public subnet.
4. Allocate and associate an Elastic IP with the NAT Gateway.
5. Update private subnet route tables so `0.0.0.0/0` points to the NAT Gateway.

A private instance using a NAT Gateway can reach the internet for updates, package downloads, API calls, and patching, but it cannot accept unsolicited inbound connections from the internet through the NAT Gateway.

**NAT Gateway route pattern**

A private subnet route table often looks like this:

```text
Destination      Target
10.0.0.0/16      local
0.0.0.0/0        nat-0abc1234def567890
```

If the VPC uses IPv6, remember that NAT Gateway does not translate IPv6. For IPv6 outbound internet access from private subnets, AWS uses an **egress-only internet gateway** rather than a NAT Gateway.

**Public subnet vs private subnet in exam terms**

A useful exam distinction is:

- **Public subnet**
  - Has a route to an IGW for internet-bound traffic.
  - Often hosts load balancers, bastion hosts, and NAT Gateways.
  - Instances may still be unreachable if they lack public IPv4 addresses or if security rules block traffic.

- **Private subnet**
  - Does not route internet-bound traffic directly to an IGW.
  - Often hosts application servers, databases, and internal services.
  - Can still reach the internet through a NAT Gateway for outbound IPv4 traffic.

Do not confuse subnet type with instance behavior:

- A subnet can be configured with a route to an IGW, but an instance without a public IPv4 address still cannot be reached directly over IPv4 from the internet.
- A private subnet can still have outbound internet access if it routes through a NAT Gateway.

**Route table association and limits**

Practical constraints matter in troubleshooting and exam questions:

- A subnet can be associated with only one route table at a time.
- A route table can be associated with multiple subnets.
- The main route table is the default for subnets without an explicit association.
- Route tables are regional resources within a VPC context.

This means you can use one public route table for several public subnets and one private route table for several private subnets, which is a common architecture pattern.

**Longest-prefix match examples**

AWS route selection uses the most specific destination match. This is critical when routes overlap.

Example:

```text
Destination      Target
10.0.0.0/16      local
10.0.1.0/24      pcx-0123456789abcdef0
10.0.1.128/25    nat-0abc1234def567890
```

Traffic to `10.0.1.200` matches `10.0.1.128/25` because `/25` is more specific than `/24` and `/16`.

This behavior matters more than whether a route is static or propagated. In AWS, route choice is based on prefix specificity, not a simple “static beats propagated” rule.

**Common architecture pattern**

A standard two-tier VPC design uses one public subnet and one private subnet per Availability Zone.

```text
Internet
   |
   v
[Internet Gateway]
   |
   +-----------------------------+
   | Public subnet route table   |
   | 0.0.0.0/0 -> IGW            |
   +-----------------------------+
            |
            v
     [NAT Gateway in public subnet]
            |
            v
   +-----------------------------+
   | Private subnet route table  |
   | 0.0.0.0/0 -> NAT Gateway    |
   +-----------------------------+
            |
            v
     Private EC2 instances
```

In this design:

- Public subnets host the NAT Gateway and any internet-facing resources.
- Private subnets host application or database instances.
- The NAT Gateway itself depends on the public subnet’s route to the IGW.
- Private instances depend on the NAT Gateway for outbound IPv4 internet access.

For high availability, deploy one NAT Gateway per Availability Zone and route each private subnet to the NAT Gateway in the same AZ. This reduces cross-AZ dependency and avoids a single point of failure.

**CLI examples**

Use the AWS CLI to inspect and manage these components.

Describe route tables:

```bash
aws ec2 describe-route-tables \
  --filters Name=vpc-id,Values=vpc-0123456789abcdef0
```

Create a route to an IGW:

```bash
aws ec2 create-route \
  --route-table-id rtb-0123456789abcdef0 \
  --destination-cidr-block 0.0.0.0/0 \
  --gateway-id igw-0123456789abcdef0
```

Create a route to a NAT Gateway:

```bash
aws ec2 create-route \
  --route-table-id rtb-0123456789abcdef0 \
  --destination-cidr-block 0.0.0.0/0 \
  --nat-gateway-id nat-0123456789abcdef0
```

Associate a subnet with a route table:

```bash
aws ec2 associate-route-table \
  --route-table-id rtb-0123456789abcdef0 \
  --subnet-id subnet-0123456789abcdef0
```

Create and attach an Internet Gateway:

```bash
aws ec2 create-internet-gateway
aws ec2 attach-internet-gateway \
  --internet-gateway-id igw-0123456789abcdef0 \
  --vpc-id vpc-0123456789abcdef0
```

Allocate an Elastic IP for a NAT Gateway:

```bash
aws ec2 allocate-address --domain vpc
```

Create a NAT Gateway in a public subnet:

```bash
aws ec2 create-nat-gateway \
  --subnet-id subnet-0123456789abcdef0 \
  --allocation-id eipalloc-0123456789abcdef0
```

**Troubleshooting workflow**

When connectivity fails, verify the path from the instance outward and back.

1. Confirm the subnet association.
   - Check which route table the subnet uses.
   - Verify the route table contains the expected default route.

2. Confirm the target type.
   - Public subnet traffic should point to an IGW.
   - Private subnet outbound IPv4 traffic should point to a NAT Gateway.

3. Confirm instance addressing.
   - For IPv4 inbound access through an IGW, the instance needs a public IPv4 address or Elastic IP.
   - For NAT-based outbound access, the instance stays private and does not need a public IPv4 address.

4. Confirm security controls.
   - Security groups must allow the required inbound and outbound ports.
   - Network ACLs must not block the traffic.

5. Confirm gateway placement.
   - The NAT Gateway must be in a public subnet.
   - That public subnet must itself route to the IGW.

6. Confirm route specificity.
   - If a more specific route exists, it overrides the default route.
   - Check for overlapping CIDR blocks that send traffic somewhere unexpected.

**End-to-end example**

Consider a VPC with `10.0.0.0/16`, two subnets, and one NAT Gateway.

Public subnet:
- `10.0.1.0/24`
- Route table:
  - `10.0.0.0/16 -> local`
  - `0.0.0.0/0 -> igw-aaa111`
- Contains:
  - NAT Gateway `nat-bbb222`
  - Optional bastion host or load balancer

Private subnet:
- `10.0.2.0/24`
- Route table:
  - `10.0.0.0/16 -> local`
  - `0.0.0.0/0 -> nat-bbb222`
- Contains:
  - Application EC2 instance with no public IPv4 address

Expected behavior:

- The application instance can download packages from the internet.
- The application instance cannot be reached directly from the internet.
- The NAT Gateway can send traffic to the internet because it sits in a public subnet with a route to the IGW.
- If the public subnet loses its IGW route, the NAT Gateway stops working, and private instances lose outbound internet access.

**Exam traps and clarifications**

- **IGW attachment alone is not enough**: you need the route table entry and correct instance addressing.
- **NAT Gateway is not for inbound access**: it is outbound-only for private IPv4 instances.
- **NAT Gateway is not in a private subnet**: it must live in a public subnet.
- **Route selection uses longest-prefix match**: do not assume a default route always wins.
- **IPv6 is different**: NAT Gateway does not translate IPv6; use an egress-only internet gateway for private IPv6 outbound access.
- **One subnet, one route table**: but one route table can serve many subnets.

**Quick comparison**

| Feature | Internet Gateway | NAT Gateway |
|---|---|---|
| Primary use | Direct internet connectivity | Outbound-only internet access for private IPv4 instances |
| Placement | Attached to VPC | Deployed in a public subnet |
| IPv4 inbound | Yes, with public IPv4 and security rules | No |
| IPv4 outbound | Yes | Yes |
| IPv6 support | Yes, with IPv6 addressing and routes | No |
| Requires Elastic IP | No | Yes |
| Common subnet type | Public subnet | Public subnet for the gateway; private subnets use it as target |

**Practical memory aid**

Think of the path like this:

- **IGW** = the VPC’s door to the internet.
- **NAT Gateway** = a receptionist in the public subnet who lets private instances call out but does not let strangers call in.
- **Route table** = the map that tells packets which door to use.

**What to remember for SOA-C02**

Focus on the dependency chain and route selection logic:

- The route table decides the next hop.
- The IGW enables direct internet routing for public resources.
- The NAT Gateway enables outbound IPv4 internet access for private resources.
- The NAT Gateway must be in a public subnet with an IGW route.
- Longest-prefix match determines which route AWS uses when routes overlap.
