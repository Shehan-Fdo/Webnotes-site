## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Core model: security groups vs network ACLs**

AWS uses two different layers of subnet and instance traffic control:

- **Security groups (SGs)** are stateful, allow-only firewalls attached to <abbr title="Elastic Network Interface">ENI</abbr>s.
- **Network ACLs (NACLs)** are stateless packet filters associated with subnets.
- SGs evaluate traffic at the ENI level and automatically allow return traffic for allowed flows.
- NACLs evaluate traffic at the subnet boundary and require explicit rules for both directions.

For exam questions, first identify whether the control is tied to an ENI or a subnet. That usually tells you whether the issue is SG-related or NACL-related.

**Security groups: behavior and design**

Security groups are the primary control for instance and service access. They are attached to ENIs, and resources such as EC2 instances use them through their ENIs. A single ENI can have multiple security groups, and the effective result is the union of all allowed rules across those groups.

Key properties:

- **Stateful**: if inbound traffic is allowed, the response traffic is automatically allowed back out, and vice versa.
- **Allow-only**: there are no explicit deny rules.
- **Rule evaluation**: AWS checks whether any rule in any attached SG allows the traffic.
- **Multiple ENIs**: if an instance has more than one ENI, each ENI can have different SGs and different exposure.

Common use cases:

- Web tier SG allows inbound `443` from the internet or from an ALB SG.
- App tier SG allows inbound application ports only from the web tier SG.
- Database tier SG allows inbound database ports only from the app tier SG.

Example of a tiered SG design:

```text
Internet -> ALB SG -> Web SG -> App SG -> DB SG
```

A practical EC2 example with the AWS CLI:

```bash
aws ec2 authorize-security-group-ingress \
  --group-id sg-0abc1234def567890 \
  --ip-permissions '[
    {
      "IpProtocol": "tcp",
      "FromPort": 443,
      "ToPort": 443,
      "IpRanges": [{"CidrIp": "0.0.0.0/0", "Description": "HTTPS from internet"}]
    }
  ]'
```

Reference another security group instead of a CIDR when you want tier-to-tier trust:

```bash
aws ec2 authorize-security-group-ingress \
  --group-id sg-0abc1234def567890 \
  --ip-permissions '[
    {
      "IpProtocol": "tcp",
      "FromPort": 3306,
      "ToPort": 3306,
      "UserIdGroupPairs": [{"GroupId": "sg-0123456789abcdef0"}]
    }
  ]'
```

**Self-referencing security groups**

A self-referencing SG rule allows members of the same SG to communicate with each other. This is common for clustered systems and peer discovery.

Use cases:

- Application nodes that replicate to each other.
- Distributed caches or cluster membership traffic.
- Stateful services where every node must talk to every other node on a specific port.

Example:

```bash
aws ec2 authorize-security-group-ingress \
  --group-id sg-0abc1234def567890 \
  --ip-permissions '[
    {
      "IpProtocol": "tcp",
      "FromPort": 7946,
      "ToPort": 7946,
      "UserIdGroupPairs": [{"GroupId": "sg-0abc1234def567890"}]
    }
  ]'
```

**Default security group behavior**

The default SG is often tested because its behavior is easy to confuse with custom SGs:

- It allows inbound traffic only from itself.
- It allows all outbound traffic by default.
- Instances launched without an explicit SG use the default SG for the VPC.

That means two instances in the default SG can talk to each other, but the default SG does not automatically allow traffic from the internet.

**Network ACLs: behavior and design**

Network ACLs are subnet-level packet filters. Every resource in the subnet is affected by the subnet’s NACL, including EC2 instances, load balancers, NAT gateways, and other ENI-attached resources.

Key properties:

- **Stateless**: you must allow both the request and the return traffic explicitly.
- **Allow and deny**: NACLs support explicit deny rules.
- **Rule order matters**: rules are evaluated in ascending order, and the first matching rule wins.
- **Separate directions**: inbound and outbound rules are evaluated independently.
- **Subnet scope**: one NACL applies to all traffic entering and leaving the subnet.

A NACL is useful when you need coarse subnet-wide guardrails, such as blocking a known bad CIDR or enforcing a broad network policy across all resources in a tier.

**Default network ACL behavior**

The default NACL is permissive:

- It allows all inbound traffic.
- It allows all outbound traffic.

A custom NACL starts more restrictive and requires explicit allow rules. If you replace the default NACL with a custom one and forget return-path rules, traffic often fails even though the SG looks correct.

**How NACL rule evaluation works**

NACL rules are numbered, and the lowest matching rule number takes precedence within each direction.

Important exam points:

- Rule numbers are unique within a direction, but the same number can exist in inbound and outbound rule sets.
- A broad allow at a lower number can override a later deny for the same traffic.
- A deny rule only blocks traffic if it is the first matching rule.

Example of a subnet NACL for a public web tier:

```text
Inbound:
100 allow TCP 443 from 0.0.0.0/0
110 allow TCP 1024-65535 from 0.0.0.0/0   # return traffic for responses to outbound connections
120 deny  all  from 0.0.0.0/0

Outbound:
100 allow TCP 1024-65535 to 0.0.0.0/0     # response traffic for inbound client connections
110 allow TCP 443 to 0.0.0.0/0            # if the subnet initiates HTTPS outbound
120 deny  all  to 0.0.0.0/0
```

This pattern is about the subnet boundary, not the application itself. For an internet-facing ALB subnet, inbound `443` and outbound ephemeral return traffic are common. For an instance subnet behind the ALB, the exact ports differ based on the application and health check flow.

**Ephemeral ports and return traffic**

Ephemeral ports are temporary high-numbered ports used by clients and by operating systems for outbound connections. In AWS exam scenarios, you must know which side uses the ephemeral port in each flow.

Typical TCP flow:

- Client sends from ephemeral source port to server destination port `443`.
- Server replies from source port `443` to client destination ephemeral port.

For a stateful SG, you usually only need to allow the initiating direction. For a stateless NACL, you must allow both directions:

- Inbound rule for the server port, such as `443`.
- Outbound rule for ephemeral return traffic, such as `1024-65535`, or the narrower ephemeral range used by your OS if you have standardized it.

For outbound connections initiated by instances:

- Outbound rule must allow the destination service port, such as `443`.
- Inbound return rule must allow ephemeral destination ports on the instance side.

> Be careful not to confuse the direction of the packet with the role of the application. The return packet is always reversed at the transport layer, so the source and destination ports swap.

**Operational tiers and traffic patterns**

A tiered architecture usually has different controls at each layer.

**Public web tier**

Typical pattern:

- SG allows inbound `80`/`443` from the internet or from an ALB SG.
- SG allows outbound to app tier, package repositories, logging endpoints, or the internet through a NAT gateway if needed.
- NACL allows inbound client traffic and outbound ephemeral return traffic if the subnet is public-facing.

**Application tier**

Typical pattern:

- SG allows inbound only from the web tier SG on the application port.
- SG allows outbound to the database tier, internal APIs, or external services as required.
- NACL allows the specific inbound application port and the outbound return path.

**Database tier**

Typical pattern:

- SG allows inbound database ports only from the app tier SG.
- SG usually has very limited outbound access.
- NACL allows inbound database traffic and outbound ephemeral return traffic for responses.

**Load balancer tier**

An internet-facing ALB often sits in public subnets:

- SG allows inbound `80`/`443` from the internet.
- SG allows outbound to target instances on the target port.
- NACL must allow client-facing inbound traffic and return traffic.

For an ALB, the subnet NACL must reflect the load balancer’s traffic pattern, which is not identical to the target instance subnet.

**NAT gateway and private subnet traffic**

A NAT gateway enables instances in a private subnet to initiate outbound connections to the internet without being directly reachable from the internet.

Exam-relevant points:

- The NAT gateway lives in a public subnet and uses an Elastic IP.
- The NAT gateway itself does not use security groups.
- The private subnet’s SGs and NACLs control the instance side of the flow.
- The private subnet NACL must allow outbound traffic to the NAT gateway path and inbound return traffic from the translated flows.
- The public subnet NACL must allow the NAT gateway’s traffic to and from the internet as required by the flow.

A private instance using a NAT gateway typically needs:

- Outbound SG rule to the destination service, such as `443`.
- Outbound NACL rule to the destination service or to the NAT path, depending on your subnet design.
- Inbound NACL rule for ephemeral return traffic.

**IPv4 and IPv6 considerations**

Both SGs and NACLs support IPv6.

Important differences to remember:

- IPv6 uses IPv6 CIDR blocks such as `::/0`.
- SG rules can allow IPv6 sources and destinations directly.
- NACL rules can also allow or deny IPv6 traffic with IPv6 CIDRs.
- Return traffic logic is the same conceptually: SGs remain stateful, NACLs remain stateless.

Example SG rule for IPv6 HTTPS:

```bash
aws ec2 authorize-security-group-ingress \
  --group-id sg-0abc1234def567890 \
  --ip-permissions '[
    {
      "IpProtocol": "tcp",
      "FromPort": 443,
      "ToPort": 443,
      "Ipv6Ranges": [{"CidrIpv6": "::/0", "Description": "HTTPS over IPv6"}]
    }
  ]'
```

Example NACL rule for IPv6 return traffic:

```text
Inbound:
100 allow tcp 443 from ::/0
110 allow tcp 1024-65535 from ::/0

Outbound:
100 allow tcp 1024-65535 to ::/0
110 allow tcp 443 to ::/0
```

**Troubleshooting denied flows**

When traffic fails, isolate the layer by checking the path in order:

1. Confirm the route table sends traffic where you expect.
2. Check the SG on the source ENI and destination ENI.
3. Check the subnet NACL on both subnets.
4. Verify the application is listening on the expected port.

A practical troubleshooting example:

- Symptom: a browser cannot reach an internal web app on `443`.
- SG check: the instance SG allows `443` from the ALB SG, so SGs look correct.
- NACL check: the subnet NACL denies inbound `443` from the ALB subnet CIDR at rule `120`.
- Result: the packet is dropped at the subnet boundary before it reaches the instance.

Another example:

- Symptom: an instance can reach an external API on `443`, but responses time out.
- SG check: outbound `443` is allowed.
- NACL check: outbound `443` is allowed, but inbound ephemeral return ports are missing.
- Result: the request leaves the subnet, but the response is blocked on the way back.

**How to distinguish SG vs NACL vs route table issues**

Use the symptom pattern:

- **SG issue**: traffic is blocked only for a specific source or destination, while other subnet traffic may still work.
- **NACL issue**: traffic fails for everything in the subnet in that direction or for a broad CIDR pattern.
- **Route table issue**: traffic never reaches the next hop, such as an IGW, NAT gateway, or peering connection.

A quick mental model:

- SG decides whether the resource is willing to talk.
- NACL decides whether the subnet permits the packet.
- Route table decides where the packet goes.

**Common exam traps**

- **Stateful vs stateless**: SGs do not need explicit return rules; NACLs do.
- **Default behaviors**: default SG is restrictive inbound and permissive outbound; default NACL allows all traffic.
- **Rule order**: NACLs use first match wins.
- **Direction confusion**: inbound and outbound NACL rules are separate, and return traffic uses reversed ports.
- **Scope confusion**: SGs attach to ENIs; NACLs attach to subnets.
- **Multiple SGs**: attached SGs are additive for allowed traffic.
- **Self-reference**: SGs can reference themselves for cluster communication.
- **IPv6**: remember `::/0` and IPv6 CIDRs in both SGs and NACLs.

**Practical configuration examples**

A common three-tier SG pattern:

```text
Web SG:
- Inbound 443 from 0.0.0.0/0
- Outbound 8080 to App SG

App SG:
- Inbound 8080 from Web SG
- Outbound 3306 to DB SG

DB SG:
- Inbound 3306 from App SG
- Outbound limited or none
```

A corresponding subnet NACL pattern for the app subnet:

```text
Inbound:
100 allow tcp 8080 from web subnet CIDR
110 allow tcp 1024-65535 from app subnet CIDR
120 deny all

Outbound:
100 allow tcp 3306 to db subnet CIDR
110 allow tcp 1024-65535 to web subnet CIDR
120 deny all
```

This is only a conceptual example. In real designs, you should align the NACL with the actual traffic matrix, including health checks, admin access, and any outbound dependencies.

**Operational guidance**

Use SGs for most resource-level access control because they are easier to reason about and safer for dynamic environments. Use NACLs when you need subnet-wide guardrails, explicit denies, or a second control layer for compliance and segmentation.

A good exam answer usually favors:

- SGs for application-specific trust relationships.
- NACLs for coarse subnet filtering and explicit deny requirements.
- Route tables for path selection, not access control.

**Quick reference**

| Topic | Security Group | Network ACL |
|---|---|---|
| Scope | ENI | Subnet |
| State | Stateful | Stateless |
| Rules | Allow only | Allow and deny |
| Return traffic | Automatic | Explicitly allowed |
| Default behavior | Inbound denied except self-reference; outbound allowed | Allow all inbound and outbound |
| Best use | Fine-grained resource access | Subnet-wide filtering and explicit denies |
