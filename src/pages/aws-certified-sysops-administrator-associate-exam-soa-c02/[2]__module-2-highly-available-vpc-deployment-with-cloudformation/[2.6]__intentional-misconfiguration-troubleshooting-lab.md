## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Lab purpose and troubleshooting mindset**

An intentional misconfiguration lab trains you to identify the *smallest change* that explains a failure. In AWS, outages often come from a single missing permission, a wrong security group rule, a broken DNS setting, or a service dependency that was never enabled. For SOA-C02, you should practice moving from symptom to root cause using evidence from logs, metrics, and service status rather than guessing.

A strong workflow is:

- Confirm the symptom and scope.
- Identify the affected layer: identity, network, compute, storage, database, or application.
- Check recent changes first.
- Validate dependencies such as IAM roles, routes, endpoints, certificates, and service agents.
- Fix one issue at a time and retest.

**Recommended lab design**

Build a small environment with at least one public subnet, one private subnet, an EC2 instance, an Application Load Balancer (ALB), an Amazon RDS database, CloudWatch Logs, and Systems Manager (SSM). Then introduce one fault at a time so you can observe the failure mode and the recovery path.

A practical topology might include:

- A public subnet with an internet-facing ALB and a bastion or admin host.
- A private subnet with an application instance and an RDS database.
- A NAT gateway for private subnet outbound access.
- VPC endpoints for SSM and CloudWatch where you want private-only management.
- CloudWatch alarms and logs for visibility.

**Core troubleshooting workflow**

Use a repeatable sequence so you do not miss the obvious issue.

1. Reproduce the problem.
2. Check the most recent change in the console, IaC, or deployment pipeline.
3. Inspect service health and logs.
4. Verify IAM permissions and resource policies.
5. Validate network paths and DNS.
6. Confirm OS-level services and application configuration.
7. Test again after each fix.

```bash
# Quick triage commands on Linux EC2
curl -I http://localhost
curl -I http://127.0.0.1:8080
ss -tulpn
ip route
resolvectl status
systemctl status amazon-ssm-agent
journalctl -u amazon-ssm-agent --no-pager -n 50
```

**IAM and permissions misconfigurations**

IAM (Identity and Access Management) issues are common in labs because they create clear, repeatable failures. A resource may exist and be reachable, but the caller still receives `AccessDenied` because the role, policy, or trust relationship is wrong.

Common failure points include:

- An EC2 instance profile is missing or attached to the wrong role.
- The role lacks permissions for S3, CloudWatch Logs, SSM, KMS, or Secrets Manager.
- A service-linked role is missing for a managed AWS service.
- The trust policy does not allow the service to assume the role.
- A permissions boundary or SCP (Service Control Policy) blocks the action.
- A resource policy denies access even when the identity policy allows it.

Typical symptoms:

- `AccessDenied` or `UnauthorizedOperation` in the console or CLI.
- SSM managed instance does not appear online.
- Application cannot read from S3 or decrypt with KMS.
- CloudWatch Logs are not delivered.
- Auto Scaling, ELB, or RDS operations fail because the service role is incomplete.

Example instance profile policy for basic SSM and CloudWatch Logs access:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "SSMCore",
      "Effect": "Allow",
      "Action": [
        "ssm:UpdateInstanceInformation",
        "ssmmessages:CreateControlChannel",
        "ssmmessages:CreateDataChannel",
        "ssmmessages:OpenControlChannel",
        "ssmmessages:OpenDataChannel",
        "ec2messages:AcknowledgeMessage",
        "ec2messages:DeleteMessage",
        "ec2messages:FailMessage",
        "ec2messages:GetEndpoint",
        "ec2messages:GetMessages",
        "ec2messages:SendReply"
      ],
      "Resource": "*"
    },
    {
      "Sid": "CloudWatchLogsWrite",
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogGroup",
        "logs:CreateLogStream",
        "logs:PutLogEvents",
        "logs:DescribeLogStreams"
      ],
      "Resource": "*"
    }
  ]
}
```

When troubleshooting IAM, verify all three parts:

- **Identity policy**: does the principal allow the action?
- **Trust policy**: can the service assume the role?
- **Resource policy**: does the target resource allow the caller?

> If a service uses a role but the trust relationship is wrong, the permissions may look correct while the service still fails.

**Systems Manager troubleshooting**

SSM is a frequent exam scenario because it depends on multiple layers working together. A managed instance can fail to register for several reasons, and the fix depends on whether the problem is agent, IAM, network, or endpoint related.

Common SSM failure points:

- The `amazon-ssm-agent` service is not installed or not running.
- The instance profile lacks the `AmazonSSMManagedInstanceCore` permissions.
- The instance cannot reach SSM endpoints from a private subnet.
- DNS resolution is broken, so endpoint names do not resolve.
- A VPC endpoint policy blocks SSM traffic.
- The instance clock is skewed and TLS authentication fails.

Useful checks on Linux:

```bash
systemctl status amazon-ssm-agent
sudo systemctl enable --now amazon-ssm-agent
sudo journalctl -u amazon-ssm-agent -n 100 --no-pager
curl -s https://ssm.us-east-1.amazonaws.com >/dev/null && echo OK
```

Useful checks on Windows:

```powershell
Get-Service AmazonSSMAgent
Start-Service AmazonSSMAgent
Get-WinEvent -LogName "AmazonSSM" -MaxEvents 20
```

For private subnets, SSM needs either:

- A NAT gateway with outbound internet access, or
- Interface VPC endpoints for `ssm`, `ssmmessages`, and `ec2messages`

If you use interface endpoints, also verify:

- Private DNS is enabled when you expect standard service names to resolve privately.
- The endpoint security group allows inbound HTTPS from the instance.
- The endpoint policy allows the required SSM actions.

**Networking and outbound access failures**

Networking issues are often the fastest way to break a lab because they affect many services at once. In SOA-C02 scenarios, you should be able to distinguish between missing internet access, broken routing, security group restrictions, and endpoint/DNS problems.

Key components to validate:

- **Internet gateway (IGW)**: required for public subnet internet access.
- **NAT gateway**: required for private subnet outbound internet access.
- **Route tables**: must point to the correct target for `0.0.0.0/0`.
- **Security groups**: stateful, so return traffic is allowed automatically.
- **Network ACLs (NACLs)**: stateless, so both inbound and outbound rules must allow traffic.
- **VPC endpoints**: useful for private access to AWS services without NAT.

Common misconfigurations:

- Private subnet route table lacks a default route to the NAT gateway.
- Public subnet route table lacks a default route to the IGW.
- NAT gateway is in the wrong subnet or has no Elastic IP.
- Security group blocks outbound HTTPS or inbound application ports.
- NACL blocks ephemeral ports, causing intermittent failures.
- Interface endpoint DNS is disabled, so the instance still tries public service names.
- Endpoint policy denies the action even though the route exists.

Example route table patterns:

```text
Public subnet:
0.0.0.0/0 -> igw-1234567890abcdef0

Private subnet:
0.0.0.0/0 -> nat-1234567890abcdef0
```

Useful commands:

```bash
ip route
traceroute 8.8.8.8
curl -I https://aws.amazon.com
nslookup ssm.us-east-1.amazonaws.com
dig ssm.us-east-1.amazonaws.com
```

If a private instance cannot reach AWS APIs, check in this order:

1. Route table default route.
2. NAT gateway health and subnet placement.
3. Security group egress rules.
4. NACL rules for outbound and ephemeral return traffic.
5. DNS resolver settings.
6. VPC endpoint configuration and policy.

**DNS and resolver problems**

DNS failures can look like application outages, but the root cause is often local name resolution. In labs, you can intentionally break `/etc/resolv.conf`, disable the VPC DNS settings, or point the instance at an invalid resolver.

Symptoms include:

- Hostnames fail to resolve, but IP connectivity still works.
- AWS service endpoints cannot be reached by name.
- Package managers fail to download metadata.
- SSM and CloudWatch agent connections fail because endpoint names do not resolve.

Checks:

```bash
cat /etc/resolv.conf
resolvectl status
nslookup example.com
dig amazon.com
```

On EC2, also verify the VPC settings:

- `enableDnsSupport` is enabled.
- `enableDnsHostnames` is enabled when needed.
- The DHCP options set provides valid DNS servers.

**Application-layer misconfigurations**

Application failures often come from a mismatch between what the app expects and what the infrastructure provides. These are especially useful lab faults because the infrastructure may look healthy while the app still returns errors.

Common examples:

- Listener rules send traffic to the wrong target group.
- The target group health check path is incorrect.
- The app listens on the wrong port.
- The security group allows the wrong protocol or port.
- HTTP to HTTPS redirects are misconfigured.
- The SSL/TLS certificate is expired, missing, or attached to the wrong listener.
- The backend expects a header, path, or host-based rule that the ALB does not provide.

Example ALB listener logic to validate:

```text
HTTP :80  -> redirect to HTTPS :443
HTTPS:443 -> forward to target-group-app
```

Troubleshooting steps:

- Confirm the listener protocol and port.
- Confirm the target group protocol and port match the application.
- Check health check path, matcher, and timeout.
- Verify the certificate ARN on the HTTPS listener.
- Inspect application logs for 4xx and 5xx errors.

A common misconfiguration is a target group health check path such as `/health` when the app only serves `/status`. Another is forwarding HTTPS traffic to a backend that only listens on HTTP, which produces connection failures even though the instance is running.

**CloudWatch Logs and observability issues**

CloudWatch Logs is often the fastest way to confirm whether the failure is in the app, the OS, or the network. If logs are missing, the issue may be the agent, permissions, log group configuration, or network access to the Logs API.

Common problems:

- The CloudWatch agent is not installed or not running.
- The agent config file points to the wrong log file path.
- The instance role lacks `logs:CreateLogStream` or `logs:PutLogEvents`.
- The log group does not exist and the role cannot create it.
- The instance in a private subnet cannot reach the Logs endpoint.
- The application writes logs to a different file than expected.

Example CloudWatch agent config snippet:

```json
{
  "logs": {
    "logs_collected": {
      "files": {
        "collect_list": [
          {
            "file_path": "/var/log/messages",
            "log_group_name": "/ec2/app/system",
            "log_stream_name": "{instance_id}/messages"
          },
          {
            "file_path": "/var/log/nginx/access.log",
            "log_group_name": "/ec2/app/nginx",
            "log_stream_name": "{instance_id}/access"
          }
        ]
      }
    }
  }
}
```

Useful checks:

```bash
sudo systemctl status amazon-cloudwatch-agent
sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \
  -a fetch-config -m ec2 -c file:/opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json -s
sudo tail -n 50 /opt/aws/amazon-cloudwatch-agent/logs/amazon-cloudwatch-agent.log
```

Use logs to trace the failure path:

- Application startup errors
- Permission denied errors
- Database connection failures
- TLS handshake failures
- DNS lookup failures
- File path or disk space issues

**RDS and database misconfigurations**

RDS problems often appear as application timeouts, authentication failures, or failover events. In a lab, you can intentionally break connectivity, parameter settings, or storage behavior to practice isolating the cause.

Common RDS issues:

- Security group does not allow the application subnet or instance.
- The database port is wrong in the application config.
- The parameter group change requires a reboot to take effect.
- The option group is missing a required feature.
- Multi-AZ failover causes a temporary connection drop.
- Storage autoscaling is disabled or storage is near capacity.
- Backup or restore settings do not match the recovery expectation.

Important behaviors:

- Some parameter changes are dynamic, but others require a reboot.
- Multi-AZ improves availability, but failover still interrupts active connections.
- Storage full conditions can cause write failures before the instance is marked unhealthy.
- Restored databases may have different endpoint names, credentials, or parameter groups.

Troubleshooting checklist:

- Confirm the DB endpoint and port in the application.
- Verify the DB security group allows inbound from the app tier.
- Check parameter group and option group associations.
- Review RDS events for failover, maintenance, or storage warnings.
- Confirm the instance class and storage type meet workload needs.

Example application connection string pattern:

```bash
mysql -h mydb.abcdefg123.us-east-1.rds.amazonaws.com -P 3306 -u appuser -p
```

If the app fails after a parameter change, ask whether the change needs a reboot. If the database was restored from a snapshot, confirm that the app is not still pointing to the old endpoint or using stale credentials.

**Storage and filesystem issues**

Storage faults are useful because they create visible OS-level symptoms. A full disk, wrong mount point, or bad permissions can break logging, package installation, and application startup.

Common issues:

- The EBS volume is not attached or not mounted.
- The mount point is wrong in `/etc/fstab`.
- The filesystem is full.
- The application user lacks write permission.
- The volume type or IOPS is insufficient for the workload.

Useful commands:

```bash
df -h
lsblk
mount
sudo blkid
sudo cat /etc/fstab
```

If the application cannot write logs or temp files, check:

- Disk space
- Inode exhaustion
- Ownership and permissions
- SELinux or AppArmor restrictions, if applicable

**OS-level and package repository issues**

Operating system misconfigurations are easy to overlook because they sit below the application layer. They are especially useful in labs where you want to simulate a broken bootstrap or user data script.

Common examples:

- The package repository URL is wrong.
- DNS cannot resolve the repository host.
- The instance time is incorrect, causing TLS certificate validation failures.
- A startup script fails because a package install never completed.
- A service is enabled but not started after reboot.

Useful checks:

```bash
timedatectl
chronyc tracking
sudo yum repolist
sudo dnf repolist
sudo apt update
systemctl list-unit-files --type=service | grep enabled
```

Time synchronization matters because many AWS and TLS operations depend on valid timestamps. If the clock is far off, you may see certificate trust failures, authentication errors, or token validation problems.

**Certificate and TLS trust issues**

Certificate problems often masquerade as network failures. The connection may reach the server, but the handshake fails because the certificate is invalid, expired, self-signed, or not trusted by the client.

Common causes:

- The certificate expired.
- The certificate does not match the hostname.
- The wrong certificate is attached to the ALB listener.
- The client trust store is missing the issuing CA.
- The system clock is incorrect.

Symptoms:

- Browser certificate warnings.
- `curl` reports SSL errors.
- Application clients fail with handshake or trust errors.

Useful checks:

```bash
openssl s_client -connect example.com:443 -servername example.com
curl -v https://example.com
```

**Hands-on misconfiguration scenarios**

Use these scenarios to practice a full troubleshoot-fix-verify loop.

1. **Broken IAM role on EC2**
   - Remove the instance profile or detach the SSM permissions.
   - Observe that the instance disappears from Systems Manager.
   - Reattach the correct role and verify the managed instance returns.

2. **Private subnet without outbound access**
   - Remove the private route table default route to the NAT gateway.
   - Observe package updates and SSM fail.
   - Restore the route and confirm outbound access works.

3. **Missing VPC endpoint DNS**
   - Create interface endpoints for SSM but disable private DNS.
   - Observe that the instance still tries public endpoints and fails in a private-only subnet.
   - Enable private DNS or add NAT access.

4. **Wrong ALB health check path**
   - Set the target group health check to a path the app does not serve.
   - Observe targets become unhealthy.
   - Correct the path and confirm healthy status.

5. **RDS security group mismatch**
   - Remove inbound access from the app security group.
   - Observe database connection timeouts.
   - Restore the rule and retest the application.

6. **CloudWatch Logs permission gap**
   - Remove `logs:PutLogEvents` from the instance role.
   - Observe the agent cannot publish logs.
   - Restore permissions and confirm log delivery.

7. **Clock skew on an instance**
   - Stop time sync or set an incorrect time.
   - Observe TLS and authentication failures.
   - Re-enable NTP and verify the clock.

**Verification commands and evidence collection**

After each fix, collect proof that the issue is resolved. This is important in labs and in the exam because you need to confirm the change actually solved the problem.

```bash
# Network and DNS
curl -I https://aws.amazon.com
nslookup ssm.us-east-1.amazonaws.com

# Services
systemctl status amazon-ssm-agent
systemctl status amazon-cloudwatch-agent

# Application
curl -I http://localhost:8080
tail -n 50 /var/log/app.log

# Database
mysql -h <db-endpoint> -P 3306 -u <user> -p
```

Useful AWS-side checks:

- EC2 instance status checks
- Security group and NACL rules
- Route tables and gateway attachments
- VPC endpoint status and policy
- CloudWatch metrics and alarms
- RDS events and failover history
- ALB target health

**Exam-focused takeaways**

For SOA-C02, the key skill is not memorizing every error message. It is recognizing which AWS layer is most likely responsible and then validating that layer with the right tool.

Remember these patterns:

- **Access denied** usually points to IAM, trust policy, resource policy, or SCP issues.
- **Private subnet failures** often point to NAT, route tables, VPC endpoints, or DNS.
- **SSM failures** usually involve the agent, instance profile, or endpoint reachability.
- **Healthy infrastructure but broken app** often means listener rules, ports, certificates, or health checks.
- **Database timeouts** often mean security groups, endpoints, parameter changes, or failover behavior.
- **Missing logs** usually mean agent config, permissions, or network access to CloudWatch Logs.
- **Odd TLS or auth failures** can come from certificate trust or time synchronization.

A disciplined lab approach helps you build the habit of proving each hypothesis before changing the next thing. That is the same habit the exam rewards.
