## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Multi-service deployment overview**

AWS CloudFormation lets you define infrastructure as code (IaC) so you can deploy related services together in a repeatable way. In a multi-service environment, you usually combine networking, compute, load balancing, databases, security controls, and observability into one coordinated deployment.

The exam focus is not just on syntax. You need to know how CloudFormation handles dependency ordering, how to split large systems into manageable stacks, and how to avoid update failures when one service depends on another.

- Use **CloudFormation** when you want repeatable, version-controlled infrastructure.
- Use **nested stacks** when a single template becomes too large or when you want to reuse a component.
- Use **StackSets** when you need the same stack across multiple accounts or Regions.
- Use **outputs** to pass values between related stacks and to expose important deployment details.

**Core deployment patterns**

A multi-service environment often includes these layers:

- **Foundation layer**
  - VPC, subnets, route tables, internet gateway, NAT gateway, security groups, and VPC endpoints.
- **Shared services layer**
  - IAM roles, KMS keys, logging buckets, parameter stores, and monitoring resources.
- **Application layer**
  - Auto Scaling groups, ECS services, Lambda functions, target groups, and application load balancers.
- **Data layer**
  - RDS, DynamoDB, ElastiCache, and backup policies.

CloudFormation works best when you model these layers explicitly. That makes dependencies easier to reason about and reduces the chance of circular references.

**Dependency management and creation order**

CloudFormation creates resources in a dependency-aware order. It infers many dependencies automatically when one resource references another. For example, if an EC2 instance references a security group ID, CloudFormation knows the security group must exist first.

You can also force ordering with `DependsOn`, but use it sparingly. Prefer implicit dependencies from references because they are easier to maintain and less brittle.

```yaml
Resources:
  AppSecurityGroup:
    Type: AWS::EC2::SecurityGroup
    Properties:
      GroupDescription: Allow app traffic
      VpcId: !Ref Vpc

  AppInstance:
    Type: AWS::EC2::Instance
    Properties:
      ImageId: ami-0123456789abcdef0
      InstanceType: t3.micro
      SecurityGroupIds:
        - !Ref AppSecurityGroup
```

In this example, `AppInstance` depends on `AppSecurityGroup` because of the reference. You do not need `DependsOn` unless a dependency is not expressed through a property reference.

- Use **implicit dependencies** for most relationships.
- Use **DependsOn** when CloudFormation cannot infer the order.
- Avoid **circular dependencies** by separating resources that reference each other.

> A common exam trap is a circular dependency between a security group and a resource that needs the security group ID while the security group also references the resource. Break the cycle by moving one side into a separate resource or by using parameters and outputs across stacks.

**Avoiding circular dependencies**

Circular dependencies happen when resource A needs resource B and resource B also needs resource A during creation. In multi-service deployments, this often appears with:

- Security groups referencing each other.
- IAM roles and policies that depend on resources created later.
- Load balancers, target groups, and listeners with tightly coupled references.
- Nested stacks that try to consume outputs before the child stack finishes.

Ways to avoid them:

- Split the stack into smaller units.
- Pass values in one direction only.
- Use parameters for inputs and outputs for results.
- Create shared resources first, then attach dependent resources later.
- Use custom resources only when CloudFormation cannot model the dependency cleanly.

**Nested stacks**

Nested stacks let you compose a large deployment from smaller templates. The parent stack creates child stacks using `AWS::CloudFormation::Stack`. This is useful when you want modularity, reuse, or clearer ownership boundaries.

A nested stack is not the same as cross-stack export/import. The parent accesses child outputs directly through `Fn::GetAtt ChildStack.Outputs.OutputName`.

```yaml
Resources:
  NetworkStack:
    Type: AWS::CloudFormation::Stack
    Properties:
      TemplateURL: https://s3.amazonaws.com/my-bucket/network.yaml
      Parameters:
        EnvironmentName: prod

  AppStack:
    Type: AWS::CloudFormation::Stack
    Properties:
      TemplateURL: https://s3.amazonaws.com/my-bucket/app.yaml
      Parameters:
        VpcId: !GetAtt NetworkStack.Outputs.VpcId
        PrivateSubnet1Id: !GetAtt NetworkStack.Outputs.PrivateSubnet1Id
```

This pattern is common for multi-service environments because it separates concerns:

- **Network stack**
  - VPC, subnets, routing, endpoints.
- **Security stack**
  - IAM roles, KMS keys, security groups.
- **Application stack**
  - Compute, listeners, target groups, autoscaling.
- **Data stack**
  - Databases, caches, backups.

Nested stacks help you update one layer without rewriting the entire template. They also make it easier to reuse a standard network or application module across environments.

**Nested stack outputs and parent access**

Child stack outputs are exposed to the parent through `Fn::GetAtt`. The child template defines `Outputs`, and the parent reads them after the child stack is created.

Child template example:

```yaml
Resources:
  Vpc:
    Type: AWS::EC2::VPC
    Properties:
      CidrBlock: 10.0.0.0/16

Outputs:
  VpcId:
    Value: !Ref Vpc
```

Parent template example:

```yaml
Resources:
  NetworkStack:
    Type: AWS::CloudFormation::Stack
    Properties:
      TemplateURL: https://s3.amazonaws.com/my-bucket/network.yaml

  AppStack:
    Type: AWS::CloudFormation::Stack
    Properties:
      TemplateURL: https://s3.amazonaws.com/my-bucket/app.yaml
      Parameters:
        VpcId: !GetAtt NetworkStack.Outputs.VpcId
```

- Use **Outputs** in the child template to expose values.
- Use `!GetAtt ChildStack.Outputs.OutputName` in the parent.
- Do not rely on `Export`/`ImportValue` for parent-child communication.

**Cross-stack references and exports**

Cross-stack references are useful when separate stacks need to share values and are not in a parent-child relationship. The exporting stack publishes a value, and another stack imports it with `Fn::ImportValue`.

```yaml
Outputs:
  SharedVpcId:
    Value: !Ref Vpc
    Export:
      Name: SharedVpcId
```

Consumer stack:

```yaml
Resources:
  AppSecurityGroup:
    Type: AWS::EC2::SecurityGroup
    Properties:
      GroupDescription: App SG
      VpcId: !ImportValue SharedVpcId
```

Important limitations:

- An exported value cannot be modified while another stack imports it.
- You must remove or update all imports before changing the export name or value.
- Export names must be unique within a Region per account.
- Cross-stack references are best for shared infrastructure, not for tightly coupled parent-child stacks.

**Parameters, secure values, and dynamic references**

Parameters let you customize a template at deployment time. For sensitive values, use `NoEcho` to hide the value in stack events and outputs, but do not treat it as encryption.

For better secret handling, use dynamic references to AWS Systems Manager Parameter Store or AWS Secrets Manager. CloudFormation resolves the value at deployment time without storing the plaintext in the template.

```yaml
Parameters:
  DbPassword:
    Type: String
    NoEcho: true

Resources:
  DbSecretUser:
    Type: AWS::IAM::User
    Properties:
      UserName: app-user
```

Dynamic reference example:

```yaml
Resources:
  AppTaskDefinition:
    Type: AWS::ECS::TaskDefinition
    Properties:
      ContainerDefinitions:
        - Name: app
          Image: 123456789012.dkr.ecr.us-east-1.amazonaws.com/app:latest
          Environment:
            - Name: DB_PASSWORD
              Value: '{{resolve:secretsmanager:prod/db:SecretString:password}}'
```

Use dynamic references when:

- You want to keep secrets out of template files.
- You need runtime resolution from a managed secret store.
- You want to rotate secrets without editing the template.

Use `NoEcho` when:

- You must accept a secret as a parameter.
- You want to reduce accidental exposure in console output.

> `NoEcho` hides values in many CloudFormation views, but it does not make the value secret if you place it in a resource property that is later exposed elsewhere.

**Conditions and `AWS::NoValue`**

Conditions let you create resources or properties only when a condition is true. This is useful for environment-specific deployments such as dev, test, and prod.

`AWS::NoValue` removes a property when the condition evaluates to false.

```yaml
Parameters:
  EnableMultiAz:
    Type: String
    AllowedValues: [true, false]
    Default: false

Conditions:
  UseMultiAz: !Equals [!Ref EnableMultiAz, true]

Resources:
  DbInstance:
    Type: AWS::RDS::DBInstance
    Properties:
      DBInstanceClass: db.t3.micro
      AllocatedStorage: 20
      MultiAZ: !If [UseMultiAz, true, !Ref AWS::NoValue]
```

Use this pattern when a property is optional and should disappear entirely in one branch. Do not return placeholder values if the property must be omitted.

**Transforms, macros, and modules**

CloudFormation supports higher-level abstractions that help in multi-service environments.

- **Transforms**
  - Built-in or custom preprocessing steps that expand templates before deployment.
  - Common example: `AWS::Serverless` for SAM-based serverless applications.
- **Macros**
  - Custom template processors that rewrite or generate CloudFormation syntax.
  - Use them when you need reusable logic that CloudFormation does not natively support.
- **Modules**
  - Reusable, packaged building blocks with a defined interface.
  - Use them when you want standardized resource patterns across teams.

Practical guidance:

- Use **transforms** when you want a supported shorthand for a known deployment style.
- Use **macros** when you need template generation or custom syntax expansion.
- Use **modules** when you want reusable infrastructure components with guardrails.

Example transform:

```yaml
Transform: AWS::Serverless-2016-10-31
Resources:
  Function:
    Type: AWS::Serverless::Function
    Properties:
      Handler: app.lambda_handler
      Runtime: python3.12
      CodeUri: src/
      Events:
        Api:
          Type: Api
          Properties:
            Path: /hello
            Method: get
```

**Custom resources**

Custom resources let you run code during stack operations when CloudFormation does not support a native resource type or when you need to orchestrate an external system.

Use a custom resource when you need to:

- Create or configure a third-party service.
- Perform a one-time setup step.
- Generate values that depend on external APIs.
- Bridge a gap in CloudFormation coverage.

Avoid custom resources when a native resource type exists. Native resources are easier to update, rollback, and troubleshoot.

**StackSets for multi-account and multi-Region deployment**

StackSets extend CloudFormation across multiple accounts and Regions. This is a major exam topic for enterprise-scale deployments.

Two permission models matter:

- **Self-managed permissions**
  - You provide an administration role in the management account and an execution role in target accounts.
  - Best when you manage accounts manually or outside AWS Organizations.
- **Service-managed permissions**
  - AWS Organizations manages permissions for you.
  - Best when target accounts are in an organization and you want simpler administration.

Deployment targeting behavior:

- You can target specific **accounts** and **Regions**.
- With service-managed permissions, you can target an **organizational unit (OU)** so new accounts in that OU can receive the stack set automatically, depending on deployment settings.
- StackSets are ideal for baseline controls such as CloudTrail, Config, IAM roles, or security agents.

Example StackSet use cases:

- Deploy a standard VPC flow log configuration to all accounts.
- Install an SSM Agent-related baseline across Regions.
- Apply guardrail IAM roles to a set of sandbox accounts.

**StackSet deployment modes**

StackSets support different ways to roll out instances.

- **Automatic deployment**
  - New accounts in a targeted OU can receive stack instances automatically.
  - Useful for landing zone and governance patterns.
- **Manual deployment**
  - You explicitly choose accounts and Regions.
  - Useful when rollout must be controlled or phased.

Operational considerations:

- Use **failure tolerance** and **maximum concurrency** to control blast radius.
- Prefer smaller batches for risky changes.
- Review drift and failed instances regularly.

**Practical StackSet example**

A simple service-managed StackSet can deploy a standard S3 bucket policy or logging baseline across accounts in an OU. The exact template is ordinary CloudFormation; the difference is the deployment scope.

```yaml
AWSTemplateFormatVersion: '2010-09-09'
Description: Baseline logging bucket
Resources:
  LogBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketEncryption:
        ServerSideEncryptionConfiguration:
          - ServerSideEncryptionByDefault:
              SSEAlgorithm: AES256
```

You then create the StackSet and target an OU rather than a single account. This is how you standardize controls across a multi-account environment.

**Update behavior and replacement risk**

CloudFormation updates resources in place when possible, but some property changes require replacement. In multi-service environments, replacement can cause downtime if the resource is in the traffic path.

Examples of replacement-sensitive resources include:

- VPCs and subnets
- Some load balancer and listener changes
- RDS instance class or storage-related changes in certain cases
- Security group changes that alter immutable properties

Be careful with security groups:

- Many **rule** changes are in-place.
- Some **resource property** changes can require replacement.
- Security groups are tied to a VPC and cannot be moved to another VPC.

Use change sets to preview replacements before applying them.

```bash
aws cloudformation create-change-set \
  --stack-name prod-app \
  --template-body file://template.yaml \
  --change-set-name preview-update \
  --capabilities CAPABILITY_NAMED_IAM
```

Then review the change set to see which resources will be modified, replaced, or deleted.

**Rollback, failure states, and troubleshooting**

When a stack update fails, CloudFormation may roll back changes to preserve the last stable state. In complex deployments, rollback can fail if a resource is already partially deleted or if a dependent resource cannot be restored.

Common stack states to recognize:

- `CREATE_IN_PROGRESS`
  - Stack creation is underway.
- `CREATE_COMPLETE`
  - Deployment succeeded.
- `CREATE_FAILED`
  - One or more resources failed during creation.
- `ROLLBACK_IN_PROGRESS`
  - CloudFormation is undoing failed changes.
- `ROLLBACK_COMPLETE`
  - Rollback finished successfully.
- `UPDATE_IN_PROGRESS`
  - Update is underway.
- `UPDATE_ROLLBACK_FAILED`
  - Rollback could not complete and manual intervention is needed.

Troubleshooting workflow:

1. Open the stack events and identify the first failure, not just the final error.
2. Read the `StatusReason` for the failed resource.
3. Check whether the failure is due to permissions, invalid parameters, quota limits, or dependency ordering.
4. Inspect nested stack events if the parent stack only shows a generic failure.
5. Fix the template or parameter issue and retry with a change set or a new stack update.

Useful CLI commands:

```bash
aws cloudformation describe-stack-events --stack-name prod-app
aws cloudformation describe-stacks --stack-name prod-app
aws cloudformation get-template --stack-name prod-app
```

When a nested stack fails, inspect the child stack directly. The parent often reports only that the nested stack failed, while the child contains the real error.

**Practical multi-service architecture example**

A common pattern is a three-tier application:

- **Network stack**
  - Creates VPC, public/private subnets, route tables, NAT gateway.
- **Database stack**
  - Creates RDS in private subnets and outputs the endpoint.
- **Application stack**
  - Creates an Auto Scaling group or ECS service behind an Application Load Balancer.
  - Consumes VPC and database outputs from the other stacks.

Example parent template structure:

```yaml
Resources:
  NetworkStack:
    Type: AWS::CloudFormation::Stack
    Properties:
      TemplateURL: https://s3.amazonaws.com/my-bucket/network.yaml

  DatabaseStack:
    Type: AWS::CloudFormation::Stack
    Properties:
      TemplateURL: https://s3.amazonaws.com/my-bucket/database.yaml
      Parameters:
        PrivateSubnet1Id: !GetAtt NetworkStack.Outputs.PrivateSubnet1Id
        PrivateSubnet2Id: !GetAtt NetworkStack.Outputs.PrivateSubnet2Id
        VpcId: !GetAtt NetworkStack.Outputs.VpcId

  AppStack:
    Type: AWS::CloudFormation::Stack
    Properties:
      TemplateURL: https://s3.amazonaws.com/my-bucket/app.yaml
      Parameters:
        VpcId: !GetAtt NetworkStack.Outputs.VpcId
        DbEndpoint: !GetAtt DatabaseStack.Outputs.DbEndpoint
```

This structure keeps each service focused and reduces the chance of a single template becoming unmanageable.

**Operational best practices**

- Keep templates small enough to review, but large enough to preserve logical ownership.
- Use nested stacks for reusable building blocks.
- Use StackSets for organization-wide baselines.
- Use outputs and parameters to pass only the values that are truly needed.
- Preview updates with change sets before production deployment.
- Store templates in version control and deploy from a controlled artifact bucket.
- Tag resources consistently so you can trace ownership and cost.
- Prefer native CloudFormation resources over custom resources when possible.

**Exam focus summary**

For SOA-C02, be ready to distinguish these concepts:

- **Nested stacks** vs **cross-stack exports**
  - Parent-child communication uses `Fn::GetAtt ChildStack.Outputs.OutputName`.
  - Separate stacks use `Export` and `ImportValue`.
- **StackSets** vs **stacks**
  - StackSets deploy across accounts and Regions.
  - Stacks deploy within one account and Region.
- **Service-managed** vs **self-managed permissions**
  - Service-managed uses AWS Organizations.
  - Self-managed uses roles you create and manage.
- **Implicit dependencies** vs **DependsOn**
  - Prefer references when possible.
  - Use `DependsOn` only when necessary.
- **NoEcho** vs **dynamic references**
  - `NoEcho` hides parameter values.
  - Dynamic references retrieve secrets from managed stores at deploy time.
- **Replacement** vs **in-place update**
  - Some changes preserve resources.
  - Others replace them and may cause downtime.

**Quick reference commands**

```bash
aws cloudformation validate-template --template-body file://template.yaml
aws cloudformation create-stack \
  --stack-name app-prod \
  --template-body file://template.yaml \
  --capabilities CAPABILITY_NAMED_IAM

aws cloudformation update-stack \
  --stack-name app-prod \
  --template-body file://template.yaml \
  --capabilities CAPABILITY_NAMED_IAM
```

Use `validate-template` to catch syntax issues early, but remember that validation does not guarantee a successful deployment. Resource-level failures still occur during stack creation or update.
