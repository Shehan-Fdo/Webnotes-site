## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Monitoring and alerting overview**

AWS monitoring is about collecting signals, turning them into actionable metrics, and routing notifications or automated responses when thresholds or patterns indicate risk. For SysOps, you need to know how to observe infrastructure health, application behavior, security events, and configuration drift without creating noisy or expensive alerting systems.

CloudWatch is the core service for metrics, logs, alarms, and dashboards. EventBridge handles event routing and automation. CloudTrail records API activity for audit and investigation. AWS Config tracks configuration changes and compliance. In practice, you often combine these services: CloudWatch detects symptoms, CloudTrail explains who changed what, and Config shows whether the change violated policy.

**CloudWatch metrics**

CloudWatch metrics are time-series data points published by AWS services, custom applications, or the CloudWatch agent. Metrics are organized by namespace, metric name, and dimensions. Dimensions are the key-value pairs that identify a specific resource or slice of data, such as `InstanceId`, `AutoScalingGroupName`, or `LoadBalancer`.

- **Namespace** groups related metrics, such as `AWS/EC2` or `AWS/Lambda`.
- **Metric name** identifies the measurement, such as `CPUUtilization` or `Invocations`.
- **Dimensions** narrow the metric to a resource or logical grouping.
- **Statistic** determines how CloudWatch aggregates values, such as `Average`, `Sum`, `Minimum`, `Maximum`, or percentile statistics like `p99`.

CloudWatch metric timing is a common exam trap. AWS service metrics may be available at different periods depending on the service and metric. Standard-resolution custom metrics are typically published at 60-second granularity, while high-resolution custom metrics can be published at 1-second or 10-second resolution. Do not confuse publication frequency with the alarm period you choose.

- Use **period** to define the evaluation window for an alarm, such as 60 seconds or 300 seconds.
- Use **statistics** to choose how the metric is summarized during each period.
- Use **high-resolution metrics** when you need faster detection and are willing to pay more.

Example custom metric publication:

```bash
aws cloudwatch put-metric-data \
  --namespace "MyApp" \
  --metric-data '[
    {
      "MetricName": "QueueDepth",
      "Dimensions": [
        {"Name": "QueueName", "Value": "orders"}
      ],
      "Value": 42,
      "Unit": "Count"
    }
  ]'
```

High-resolution example:

```bash
aws cloudwatch put-metric-data \
  --namespace "MyApp" \
  --metric-data '[
    {
      "MetricName": "LatencyMs",
      "Value": 128,
      "Unit": "Milliseconds",
      "StorageResolution": 1
    }
  ]'
```

**CloudWatch alarms**

Alarms evaluate metrics against thresholds and move through states. The three primary states are `OK`, `ALARM`, and `INSUFFICIENT_DATA`.

- **OK** means the metric is within the defined threshold.
- **ALARM** means the threshold condition is met for the configured evaluation logic.
- **INSUFFICIENT_DATA** means CloudWatch does not have enough data to evaluate the alarm reliably.

Alarm behavior depends on several settings:

- **Period** controls how often CloudWatch evaluates the metric.
- **Evaluation periods** define how many consecutive periods are considered.
- **Datapoints to alarm** lets you require a subset of periods to breach the threshold, such as 2 out of 3.
- **Missing data treatment** determines how CloudWatch handles gaps:
  - `breaching`
  - `notBreaching`
  - `ignore`
  - `missing`

This matters in exam scenarios because sparse metrics, stopped instances, or intermittent publishing can cause false alarms or missed alarms. For example, if an application publishes a metric only when traffic exists, treating missing data as breaching may create noise during quiet periods.

A practical alarm example for EC2 CPU:

- Trigger when `CPUUtilization` is greater than `80` percent.
- Evaluate over `3` periods of `60` seconds.
- Require `2` datapoints to alarm.
- Treat missing data as `missing` if the instance may stop publishing when unhealthy.

Example CLI alarm:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name "ec2-high-cpu" \
  --namespace "AWS/EC2" \
  --metric-name "CPUUtilization" \
  --dimensions Name=InstanceId,Value=i-0123456789abcdef0 \
  --statistic Average \
  --period 60 \
  --evaluation-periods 3 \
  --datapoints-to-alarm 2 \
  --threshold 80 \
  --comparison-operator GreaterThanThreshold \
  --treat-missing-data missing \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:ops-alerts
```

Alarm actions commonly include:

- **SNS** notifications to email, SMS, Lambda, or downstream systems.
- **Auto Scaling** policies to scale out or scale in.
- **Systems Manager** automation or incident workflows.
- **EC2 stop/terminate/recover** actions for specific alarm types and use cases.

> Alarm design should reflect the metric’s behavior. A threshold that is too tight creates alert fatigue; a threshold that is too loose delays response.

**Metric math and composite alarms**

Metric math lets you combine multiple metrics in a single expression to create derived signals. Use it when raw metrics do not directly represent the condition you care about, such as error rate or utilization ratio.

A valid metric math alarm uses metric IDs and expressions in one definition. For example, you can calculate error percentage from request counts and 5xx errors.

Conceptual example:

```text
error_rate = 100 * m2 / m1
```

Where:
- `m1` is total requests
- `m2` is 5xx errors

In the console or API, you define each metric with an ID and then reference those IDs in an expression. This is more accurate than trying to write standalone `SUM(RequestCount)` style pseudo-syntax.

Composite alarms reduce noise by combining multiple underlying alarms with Boolean logic. They are useful when you want to alert only when several conditions occur together.

Example composite alarm rule:

```text
ALARM("HighCPU") AND ALARM("High5xxRate")
```

Use composite alarms to:
- suppress duplicate notifications,
- correlate symptoms,
- avoid paging on a single weak signal.

**CloudWatch Logs**

CloudWatch Logs stores log events in log groups and log streams. A log group is the container for related logs, and a log stream is an ordered sequence of events from one source, such as one EC2 instance or one Lambda execution environment.

Log ingestion can happen through several paths:

- **Service-integrated logging** from AWS services such as Lambda, API Gateway, VPC Flow Logs, and others.
- **CloudWatch agent** on EC2 or on-premises servers.
- **AWS CLI or SDKs** using the Logs API.
- **Subscription filters** that forward logs to Kinesis Data Streams, Kinesis Data Firehose, or Lambda for processing.

Operational behavior to remember:

- Some services create log groups automatically when first used.
- Lambda commonly creates its log group on first invocation if permissions allow.
- You should set retention explicitly; otherwise, logs may remain indefinitely and increase cost.
- Log streams are often created per source instance, container, or execution environment.

Example CloudWatch agent configuration snippet:

```json
{
  "logs": {
    "logs_collected": {
      "files": {
        "collect_list": [
          {
            "file_path": "/var/log/messages",
            "log_group_name": "/ec2/system/messages",
            "log_stream_name": "{instance_id}"
          }
        ]
      }
    }
  }
}
```

Example agent install pattern on Linux varies by distribution and version. On many systems you install the package from the downloaded bundle or use automation such as Systems Manager. Package managers may be `yum` or `dnf` depending on the OS version.

Example log tailing with AWS CLI v2:

```bash
aws logs tail /aws/lambda/my-function --follow
```

> This command requires AWS CLI v2 and a recent version. It is not available in AWS CLI v1.

**Logs Insights and alerting workflows**

CloudWatch Logs Insights is for interactive querying and investigation. You do not create a native CloudWatch alarm directly on a Logs Insights query result, but Logs Insights is still valuable in alerting workflows because it helps you identify patterns, validate thresholds, and refine metric filters.

Use Logs Insights when you need to:
- search for error patterns,
- count exceptions over time,
- inspect request traces or application messages,
- confirm whether a metric alarm reflects a real issue.

Example Logs Insights query:

```sql
fields @timestamp, @message
| filter @message like /ERROR|Exception/
| sort @timestamp desc
| limit 20
```

If you need an alarm from log content, use a **metric filter** to turn matching log events into a custom metric, then alarm on that metric.

Example metric filter concept:
- Match `ERROR` in application logs.
- Increment a custom metric named `ErrorCount`.
- Alarm when `ErrorCount` exceeds a threshold over a period.

**Dashboards**

CloudWatch dashboards provide a visual operational view of metrics, alarms, and logs. They are useful for NOC-style monitoring, incident response, and executive summaries.

Dashboard widgets can display:
- metric graphs,
- single-value metrics,
- alarm status,
- log query results,
- text notes and runbook links.

Important limitations and setup details:

- Dashboards are regional resources, but you can include cross-region metrics in widgets if you specify them correctly.
- Cross-account visibility requires the appropriate cross-account observability setup and permissions.
- Widgets are configured from metrics, alarms, or logs queries, and each widget can be customized for period, statistic, and display type.
- Dashboards are read-oriented; they do not replace alarms.

Example metric widget JSON fragment:

```json
{
  "type": "metric",
  "properties": {
    "metrics": [
      [ "AWS/EC2", "CPUUtilization", "InstanceId", "i-0123456789abcdef0" ]
    ],
    "period": 60,
    "stat": "Average",
    "region": "us-east-1",
    "title": "EC2 CPU"
  }
}
```

**EventBridge and CloudWatch Events history**

Amazon EventBridge is the modern event bus and rule engine. Older exam material may still refer to **CloudWatch Events**, which is the historical name for what is now EventBridge.

EventBridge rules match events based on event patterns and route them to targets such as Lambda, SNS, SQS, Step Functions, or API destinations. Event patterns can match many fields, including:
- `source`
- `detail-type`
- `detail`
- `account`
- `region`
- `resources`
- `time`

This is broader than the simplified idea that rules only match `source`, `detail-type`, and `detail`.

Example event pattern:

```json
{
  "source": ["aws.ec2"],
  "detail-type": ["EC2 Instance State-change Notification"],
  "detail": {
    "state": ["stopped", "terminated"]
  }
}
```

Use EventBridge when you want:
- near-real-time reaction to AWS service events,
- decoupled automation,
- event routing without polling.

**CloudTrail and AWS Config in monitoring workflows**

CloudTrail records API calls and console activity. It is not a metrics service, but it is essential for investigating why a resource changed or who performed an action that triggered an alarm.

Use CloudTrail to:
- trace configuration changes,
- investigate security incidents,
- correlate operational events with API activity.

AWS Config records resource configuration history and evaluates compliance against rules. It helps answer whether a resource drifted from the desired state.

Use Config to:
- detect unauthorized changes,
- audit resource settings over time,
- support compliance reporting.

A common workflow is:
1. CloudWatch alarm detects abnormal behavior.
2. CloudTrail shows the API call that changed the resource.
3. AWS Config shows the before-and-after configuration state.
4. EventBridge routes the event to automation or incident response.

**Practical alerting patterns**

A good monitoring design uses layered signals rather than a single alarm for everything.

Common patterns include:

- **Infrastructure health**
  - EC2 CPU, status checks, disk space, memory via agent metrics.
- **Application health**
  - 5xx error rate, latency, queue depth, custom business metrics.
- **Security and audit**
  - CloudTrail event patterns, Config compliance changes, GuardDuty findings routed through EventBridge.
- **Operational automation**
  - Alarm triggers SNS, Lambda, or Auto Scaling.

Example SNS-based notification flow:

```text
CloudWatch Alarm -> SNS Topic -> Email / Pager / Lambda
```

Example Auto Scaling flow:

```text
High CPU Alarm -> Auto Scaling Policy -> Add EC2 capacity
```

Example incident enrichment flow:

```text
CloudWatch Alarm -> EventBridge -> Lambda -> Slack/Webhook/Incident tool
```

**Alarm tuning and accuracy**

Alarm quality depends on how well you match the metric’s natural behavior.

- Use a longer period for noisy metrics.
- Use multiple evaluation periods to reduce flapping.
- Use `datapoints to alarm` for partial-breach logic.
- Use percentile statistics for latency when outliers matter.
- Use `Sum` for count-based metrics and `Average` for utilization-style metrics when appropriate.

Examples:
- For request latency, `p95` or `p99` may be more meaningful than `Average`.
- For request counts, `Sum` over a period is often better than `Average`.
- For CPU, `Average` is usually acceptable, but short spikes may require shorter periods or high-resolution metrics.

**Costs and quotas**

Monitoring can become expensive if you do not control data volume and retention.

Cost and quota considerations include:

- **Custom metrics** are billed, and high-cardinality dimensions increase cost and complexity.
- **High-resolution metrics** cost more than standard-resolution metrics.
- **Logs ingestion** and **log retention** both affect cost.
- **Long retention** is useful for compliance but increases storage cost.
- **Too many alarms** can create operational noise and management overhead.
- **Metric filters** and subscription pipelines can add processing cost.

Practical cost controls:
- Set log retention policies by environment.
- Avoid unnecessary custom dimensions.
- Use high-resolution metrics only where faster detection matters.
- Archive older logs to S3 if long-term retention is required.

**Common exam scenarios**

- An EC2 instance is unhealthy, but CPU is normal.
  - Check status checks, memory, disk, and application logs.
- A Lambda function fails intermittently.
  - Review CloudWatch Logs, metric filters, and error metrics.
- A deployment caused a spike in 5xx errors.
  - Use CloudWatch alarms, Logs Insights, and CloudTrail to correlate the change.
- A compliance rule is violated after a security group change.
  - Use AWS Config and CloudTrail, then route the event through EventBridge.

**Quick reference**

| Topic | Key point |
|---|---|
| Metrics | Time-series data with namespaces, names, dimensions, and statistics |
| Alarms | States: `OK`, `ALARM`, `INSUFFICIENT_DATA` |
| Missing data | Choose treatment carefully to avoid false positives or missed alerts |
| Logs | Stored in log groups and streams; retention should be set explicitly |
| Logs Insights | Investigate logs and refine alerting, but do not alarm directly on query results |
| Dashboards | Visualize metrics, alarms, and logs; support cross-region and cross-account setups with proper configuration |
| EventBridge | Modern event routing; CloudWatch Events is the legacy name |
| CloudTrail | Audit API activity and investigate changes |
| AWS Config | Track configuration history and compliance |

**Operational checklist**

1. Define the symptom you want to detect.
2. Choose the best signal: metric, log pattern, event, or configuration change.
3. Select the right statistic, period, and evaluation logic.
4. Decide how to handle missing data.
5. Route the alarm to SNS, Auto Scaling, or automation.
6. Add dashboards and logs for investigation.
7. Review cost, retention, and noise regularly.

**Troubleshooting tips**

- If an alarm stays in `INSUFFICIENT_DATA`, verify the metric is publishing and the dimensions match exactly.
- If an alarm flaps, increase the period or evaluation periods, or use `datapoints to alarm`.
- If logs are missing, check IAM permissions, log group creation behavior, and agent configuration.
- If a dashboard widget is blank, confirm region, account access, and metric namespace/dimensions.
- If an event rule does not fire, validate the event pattern against the actual event structure.
