## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Patch compliance enforcement overview**

AWS Systems Manager Patch Manager helps you assess and remediate missing operating system and application patches on managed nodes. In exam scenarios, the key idea is not just “install patches,” but to define what counts as compliant, scan for drift, and enforce a repeatable remediation process.

Patch compliance is typically built from three layers:

- **Patch baseline**: defines which patches are approved or rejected.
- **Patch scan**: checks the instance against the baseline and reports compliance.
- **Patch install**: applies approved patches and can reboot if required.

A compliant node has the expected approved patches installed according to the baseline and the latest scan/install results. A noncompliant node is missing one or more approved patches, has failed installs, or has not successfully reported a current assessment.

**Core Patch Manager concepts**

Patch Manager works through AWS Systems Manager managed nodes, usually EC2 instances with the Systems Manager agent installed and an IAM instance profile attached. For EC2, the instance profile should include `AmazonSSMManagedInstanceCore` or equivalent permissions so the node can communicate with Systems Manager.

Important terms:

- **Managed node**: an instance or server registered with Systems Manager.
- **Patch group**: a tag-based grouping used to associate instances with a specific patch baseline.
- **Baseline**: the patch policy that determines approval and rejection rules.
- **Compliance state**: the reported result of scan and install activity.

Patch Manager is most useful when you need consistent patch policy across fleets, especially when different environments require different approval windows or reboot behavior.

**Patch baselines and default behavior**

Patch baselines define the patch policy for a platform such as Amazon Linux, Ubuntu, Windows Server, or other supported operating systems. AWS provides default baselines for many platforms, and you can also create custom baselines.

Default behavior matters in exam questions:

- If you do not associate a custom baseline to a patch group, the AWS-managed default baseline is used for that platform.
- A custom baseline overrides the default only for the instances associated with that patch group.
- Different patch groups can point to different baselines, which lets you separate production, staging, and development policies.

Use a custom baseline when you need:

- A longer approval delay before patches become eligible.
- Explicit rejection of specific patches.
- Different reboot behavior or approval rules for different environments.
- More control than the AWS-managed default baseline provides.

> If no custom baseline is associated, do not assume “no policy.” The AWS default baseline still applies.

**Approval rules and timing**

Approval rules determine when a patch becomes eligible for installation. The most common setting is `ApproveAfterDays`, which delays approval for a number of days after patch release.

Practical implications:

- A patch released today may not be approved until the delay period passes.
- This delay helps reduce risk by allowing time for vendor or community issues to surface.
- Compliance can remain noncompliant during the delay because the patch is not yet approved, even though it exists.

Other rule elements can include:

- **ComplianceLevel**: the severity or importance assigned to approved patches.
- **Patch filters**: criteria such as product, classification, severity, or advisory identifiers.
- **Rejected patches**: patches explicitly excluded from installation.

A useful mental model is:

- Approved patches are eligible.
- Rejected patches are never installed by the baseline.
- Unapproved patches are ignored until they meet the approval rule.

**Baseline lifecycle and management**

Patch baselines are not static. In operations and exam scenarios, you may need to create, modify, copy, or delete them.

Common lifecycle actions:

1. Create a baseline for a platform and define approval rules.
2. Associate the baseline with a patch group.
3. Modify the baseline as patch policy changes.
4. Copy a baseline when you need a similar policy for another environment.
5. Delete obsolete baselines after disassociation.

Example CLI patterns:

```bash
aws ssm create-patch-baseline \
  --name "prod-linux-baseline" \
  --operating-system AMAZON_LINUX_2 \
  --approval-rules 'PatchRules=[{PatchFilterGroup={PatchFilters=[{Key=CLASSIFICATION,Values=[Security]},{Key=SEVERITY,Values=[Critical,Important]}]},ApproveAfterDays=7,ComplianceLevel=CRITICAL}]' \
  --description "Production baseline with 7-day approval delay"
```

```bash
aws ssm copy-patch-baseline \
  --baseline-id pb-0123456789abcdef0 \
  --name "prod-linux-baseline-copy"
```

```bash
aws ssm modify-patch-baseline \
  --baseline-id pb-0123456789abcdef0 \
  --approval-rules 'PatchRules=[{PatchFilterGroup={PatchFilters=[{Key=CLASSIFICATION,Values=[Security]}]},ApproveAfterDays=3,ComplianceLevel=HIGH}]'
```

```bash
aws ssm delete-patch-baseline \
  --baseline-id pb-0123456789abcdef0
```

For exam prep, remember that baseline management is a control-plane activity. You are defining policy, not directly patching instances.

**Patch groups and tagging**

Patch groups are how you map instances to baselines. The tag key must match the patch group key used in your baseline association workflow.

Typical workflow:

1. Create a patch baseline.
2. Associate the baseline with a patch group name.
3. Tag instances with the matching patch group value.
4. Run scans or installs against those instances.

Example tagging:

```bash
aws ec2 create-tags \
  --resources i-0123456789abcdef0 \
  --tags Key=Patch Group,Value=prod-linux
```

The exact tag key must match the patch group configuration you use in Systems Manager. Do not treat the key as arbitrary if your baseline association expects a specific patch group name.

**Scan versus install**

Patch compliance enforcement depends on the difference between scanning and installing.

- **Scan**: evaluates the instance against the baseline and reports missing approved patches.
- **Install**: applies approved patches and can trigger a reboot if needed.

A scan does not change the system state. It only updates compliance data. An install changes the system state and may improve compliance if the patches succeed.

Typical outcomes:

- **Compliant**: all required approved patches are present.
- **Noncompliant**: one or more approved patches are missing.
- **Failed scan**: Systems Manager could not complete the assessment.
- **Failed install**: patch application failed, so compliance may remain unresolved.

A stale or missing scan can prevent an up-to-date compliance assessment. The issue is not simply “the node is old”; it is that the latest reporting data is incomplete or outdated.

**Reboot behavior**

Reboot handling is a common exam topic because many patches require a restart to complete installation.

Patch Manager commonly supports:

- **RebootIfNeeded**: reboot only when the patching operation requires it.
- **NoReboot**: do not reboot automatically, even if patches need it.

Operational tradeoffs:

- Use `RebootIfNeeded` when you want Patch Manager to complete patching in one run and you can tolerate a restart.
- Use `NoReboot` when you need to defer restarts to a maintenance window, application drain period, or manual change control process.

Example install command:

```bash
aws ssm send-command \
  --document-name "AWS-RunPatchBaseline" \
  --targets "Key=InstanceIds,Values=i-0123456789abcdef0" \
  --parameters "Operation=Install,RebootOption=RebootIfNeeded" \
  --comment "Install approved patches on one instance"
```

If you choose `NoReboot`, the instance may still show incomplete patch remediation until the reboot occurs and the node is rescanned.

**Maintenance Windows and recurring patching**

Maintenance Windows are the standard way to schedule recurring patch operations. They let you define when patching runs, which targets are included, and which task executes.

Use Maintenance Windows when you need:

- Recurring patch cycles.
- Controlled change windows.
- Coordinated patching across fleets.
- Separation of scan and install tasks.

A common pattern is:

- Run a scan before the window to identify missing patches.
- Run an install task during the window.
- Reboot if needed, or defer reboot to a controlled step.

Example maintenance window task registration:

```bash
aws ssm register-task-with-maintenance-window \
  --window-id mw-0123456789abcdef0 \
  --task-type RUN_COMMAND \
  --task-arn AWS-RunPatchBaseline \
  --targets "Key=WindowTargetIds,Values=target-0123456789abcdef0" \
  --service-role-arn arn:aws:iam::123456789012:role/MaintenanceWindowRole \
  --task-invocation-parameters '{
    "RunCommand": {
      "Parameters": {
        "Operation": ["Install"],
        "RebootOption": ["RebootIfNeeded"]
      }
    }
  }'
```

For exam questions, distinguish between:

- **One-time execution**: a direct `send-command` run for ad hoc remediation.
- **Recurring maintenance window execution**: scheduled, repeatable patching.

**One-time patching versus scheduled patching**

You do not always need a Maintenance Window. For a one-off remediation, you can invoke the patch document directly with Systems Manager Run Command.

Use one-time execution when:

- You need to patch a single instance immediately.
- You are responding to an urgent vulnerability.
- You are testing a baseline or patch behavior before rolling it out.

Use Maintenance Windows when:

- You need repeatable patching.
- You want a controlled operational process.
- You need to coordinate patching across many instances.

Example one-time scan:

```bash
aws ssm send-command \
  --document-name "AWS-RunPatchBaseline" \
  --targets "Key=InstanceIds,Values=i-0123456789abcdef0" \
  --parameters "Operation=Scan" \
  --comment "Ad hoc patch compliance scan"
```

Example one-time install:

```bash
aws ssm send-command \
  --document-name "AWS-RunPatchBaseline" \
  --targets "Key=InstanceIds,Values=i-0123456789abcdef0" \
  --parameters "Operation=Install,RebootOption=NoReboot" \
  --comment "Install without reboot for controlled restart later"
```

**Compliance interpretation**

Patch compliance reports help you decide whether to remediate, defer, or investigate.

Common interpretations:

- **Compliant**: the instance matches the approved patch set.
- **Noncompliant**: approved patches are missing and should be installed.
- **Missing**: the patch is not present on the node.
- **Failed**: the scan or install did not complete successfully.
- **Stale data**: the report may not reflect the current system state if the latest scan or install result is old or incomplete.

In practice, a node can be operationally healthy but still noncompliant if it has not yet received an approved patch. Conversely, a node can be marked noncompliant because a patch install failed even though the patch was attempted.

**IAM permissions and execution roles**

Patch Manager depends on IAM roles in two places:

- **Instance profile role**: attached to the managed node so it can communicate with Systems Manager.
- **Maintenance Window service role**: allows the maintenance window to invoke tasks on your behalf.

For EC2 instances, the instance profile should include permissions equivalent to `AmazonSSMManagedInstanceCore`. Without this, the node may not register correctly or may fail to receive commands.

For maintenance windows and patching workflows, ensure the role can:

- Send Run Command tasks.
- Read instance and patch metadata as needed.
- Write logs to CloudWatch Logs if configured.
- Access S3 if you store command output there.

A common troubleshooting pattern is to verify:

1. The instance is a managed node.
2. The instance profile is attached and correct.
3. The maintenance window role can invoke the task.
4. The patch baseline is associated with the correct patch group.

**Auto Scaling groups and immutable deployments**

Patch compliance becomes more interesting in Auto Scaling environments. You often have two strategies:

- **In-place patching**: patch running instances directly.
- **Immutable replacement**: build a new AMI or launch template version, then replace instances.

In-place patching is simpler for small fleets, but it can be risky if patching or rebooting affects availability. Immutable patterns are often better when:

- You want consistent instance state.
- You can replace instances instead of modifying them.
- You use blue/green or rolling deployment workflows.

For Auto Scaling groups:

- Patching instances directly can conflict with scale-in events or health checks.
- Reboots may trigger replacement if the instance becomes unhealthy.
- A replacement-based strategy can reduce drift and simplify compliance.

A practical production pattern is to patch a golden image pipeline, then roll out new instances rather than repeatedly patching long-lived nodes.

**Troubleshooting patch compliance**

When patch compliance is not behaving as expected, check the workflow from policy to execution.

Common issues:

- The instance is not a managed node.
- The instance profile is missing `AmazonSSMManagedInstanceCore` or equivalent permissions.
- The patch group tag does not match the baseline association.
- The baseline approval rules do not yet approve the patch.
- The maintenance window task is misconfigured.
- The reboot was deferred, so compliance is not complete.
- The scan did not run successfully, leaving stale compliance data.

A realistic incident example:

A production Linux instance shows noncompliant for a critical security patch. The team confirms the patch was released yesterday, but the baseline has `ApproveAfterDays=7`. The instance is behaving normally, but Patch Manager correctly withholds approval until the delay expires. The fix is not to force the patch immediately unless change policy allows it; the fix is to understand the approval window and decide whether to override the baseline for emergency remediation.

Another example:

A Windows fleet is patched during a maintenance window, but several nodes remain noncompliant. Investigation shows the install ran with `NoReboot`, and the patches require a restart to finalize. After a controlled reboot and a follow-up scan, the nodes become compliant.

**Practical command patterns**

List baselines:

```bash
aws ssm describe-patch-baselines
```

Describe a baseline:

```bash
aws ssm describe-patch-baselines \
  --filters Key=NAME,Values=prod-linux-baseline
```

List patch groups associated with a baseline:

```bash
aws ssm describe-patch-groups \
  --filters Key=BaselineId,Values=pb-0123456789abcdef0
```

Run a scan across a target set:

```bash
aws ssm send-command \
  --document-name "AWS-RunPatchBaseline" \
  --targets "Key=tag:Patch Group,Values=prod-linux" \
  --parameters "Operation=Scan"
```

Run an install across a target set:

```bash
aws ssm send-command \
  --document-name "AWS-RunPatchBaseline" \
  --targets "Key=tag:Patch Group,Values=prod-linux" \
  --parameters "Operation=Install,RebootOption=RebootIfNeeded"
```

**Exam-focused distinctions**

Remember these high-value distinctions for SOA-C02:

- **Baseline policy** decides what is approved.
- **Scan** reports compliance; it does not remediate.
- **Install** remediates; it may require a reboot.
- **Maintenance Windows** schedule recurring patching.
- **Run Command** can perform one-time patching.
- **Patch groups** map instances to baselines.
- **Default baselines** apply when no custom baseline is associated.
- **Approval delays** can make a patch intentionally unavailable for a period after release.
- **Reboot settings** affect whether patch completion is immediate or deferred.
- **IAM roles** must allow both the managed node and the execution workflow to function.

**Study takeaway**

Patch compliance enforcement is about policy, timing, and execution. In exam scenarios, always identify which part of the workflow is failing: baseline selection, approval timing, instance registration, scheduling, reboot handling, or IAM permissions. Once you map the failure to the correct layer, the remediation choice becomes much clearer.
