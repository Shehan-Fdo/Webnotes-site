## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**What SCPs do in AWS Organizations**

Service control policies (SCPs) are an <abbr title="AWS Organizations">Organizations</abbr> governance control that sets the maximum permissions available to principals in member accounts. They do not grant permissions by themselves. Instead, they filter what IAM permissions, resource policies, and session permissions can ultimately allow.

SCPs are most useful when you need to enforce guardrails across many accounts, such as blocking destructive actions, restricting regions, or preventing users from turning off security services. For exam purposes, remember the key distinction:

- **IAM policies** grant permissions within an account.
- **Resource policies** grant access to a resource.
- **SCPs** limit the ceiling for member accounts in the organization.

> SCPs do not apply to the management account. They do apply to member accounts, including the root user in those member accounts.

**Prerequisites and organization model**

SCPs only work when the account is part of an AWS Organization with the required features enabled. In practice, you need:

- An AWS Organization with **all features enabled**.
- SCPs attached to an organizational unit (OU) or account.
- A policy strategy that matches your governance model.

If the organization is not using all features, SCPs are not effective for enforcement. This is a common exam trap because the policy may exist, but the organization setup prevents it from taking effect.

A practical governance hierarchy looks like this:

```text
Organization root
├── Security OU
│   ├── Log archive account
│   └── Audit account
├── Shared services OU
└── Workloads OU
    ├── Dev account
    └── Prod account
```

You usually attach broad guardrails at the root or OU level, then add narrower exceptions lower in the tree.

**How SCP evaluation works**

Think of authorization as layered filtering:

```text
Request
  ↓
SCPs in the account's org path
  ↓
IAM identity policy + session policy
  ↓
Resource policy
  ↓
Final decision
```

An action succeeds only when it is allowed by the applicable SCP model and also allowed by the normal AWS authorization layers. An SCP allow does not automatically grant access. It only means the action is not blocked by the organization guardrail.

Important evaluation points:

- **Explicit deny wins** across policy types.
- SCPs can block an action even when IAM allows it.
- IAM can still deny an action even if the SCP permits it.
- Resource policies can also be required for access to the target resource.

**Default FullAWSAccess versus custom SCPs**

AWS Organizations commonly starts with the `FullAWSAccess` SCP attached at the root. This is a full-access baseline that allows all actions unless another SCP explicitly denies them.

That default matters:

- With `FullAWSAccess` attached, you can use **deny-only SCPs** as exceptions.
- If you remove `FullAWSAccess`, the model becomes much stricter.
- Without a broad allow SCP, only actions explicitly allowed by your SCP set remain available.

This is one of the most important exam concepts. A deny-only strategy works well when `FullAWSAccess` remains in place. If you replace it with a restrictive allow-list, you must explicitly permit every action you want to keep.

| Model | Typical structure | Result |
|---|---|---|
| Deny-only with `FullAWSAccess` | Allow all, then deny exceptions | Easier to manage for broad organizations |
| Allow-list SCP | Explicitly allow only approved actions | Stronger restriction, higher maintenance |
| No broad allow SCP | Only explicitly allowed actions remain | Very restrictive, easy to lock down accounts |

> Removing `FullAWSAccess` without a replacement allow-list can lock down member accounts far more than intended.

**Deny-only SCP example**

A deny-only SCP is often used to block a dangerous action while leaving everything else available through `FullAWSAccess`.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyDisableGuardDuty",
      "Effect": "Deny",
      "Action": [
        "guardduty:DeleteDetector"
      ],
      "Resource": "*"
    }
  ]
}
```

This prevents users from deleting the GuardDuty detector in accounts where the SCP applies. In real governance, you often pair this with detective controls and delegated administration so security teams can still manage the service centrally.

**Allow-list SCP example**

An allow-list SCP is more restrictive. It is used when you want to define the only actions that are permitted in a scope.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCoreReadOnlyActions",
      "Effect": "Allow",
      "Action": [
        "ec2:Describe*",
        "s3:Get*",
        "s3:List*",
        "cloudwatch:Get*",
        "cloudwatch:List*"
      ],
      "Resource": "*"
    }
  ]
}
```

This style is useful for tightly controlled environments, but it requires careful maintenance because any missing action is denied by default in the SCP model.

**Policy language limitations**

SCPs use IAM-style JSON syntax, but they are not identical to IAM policies in what they support. Be careful with assumptions:

- Not every IAM policy feature is supported in SCPs.
- Some condition keys and policy elements may not behave the same way as in IAM.
- SCPs are intended for coarse governance, not fine-grained application authorization.
- You should verify service support before relying on a complex SCP condition.

Practical exam takeaway: if a policy works in IAM, do not assume it will work the same way in an SCP. Keep SCPs simple when possible.

**Common governance use cases**

SCPs are best for organization-wide guardrails that reduce risk across accounts.

Typical uses include:

- Blocking root-level or destructive actions.
- Preventing users from disabling security services.
- Restricting which AWS Regions can be used.
- Limiting access to expensive or high-risk services.
- Enforcing separation between production and non-production accounts.

A common pattern is to deny everything outside approved Regions:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyOutsideApprovedRegions",
      "Effect": "Deny",
      "NotAction": [
        "iam:*",
        "organizations:*",
        "route53:*",
        "cloudfront:*",
        "support:*"
      ],
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": [
            "us-east-1",
            "us-west-2"
          ]
        }
      }
    }
  ]
}
```

This pattern needs care because some services are global or have control-plane operations that are not region-scoped in the way you expect.

**Global services and region restrictions**

Region restriction SCPs are a frequent source of confusion. Some AWS services are global, or they use a mix of global and regional APIs. That means a simple `aws:RequestedRegion` condition may not cover every operation the way you expect.

Watch for these pitfalls:

- **IAM** is global for many administrative actions.
- **CloudFront** is global.
- **Route 53** is global.
- Some services call supporting APIs in a different region than the one you think you are using.
- A region deny can accidentally block service setup or management workflows.

When you design region guardrails, test the full workflow, not just the obvious API call.

**Service-linked roles and AWS service operations**

Some AWS services depend on service-linked roles to perform actions in your account. If an SCP blocks the actions those services need, the service may fail in unexpected ways.

Key points:

- Service-linked roles are created and managed by AWS services.
- SCPs can still block the service from calling required APIs.
- A governance policy that is too strict can break automation, scaling, or managed service operations.

This matters for services such as Auto Scaling, Elastic Load Balancing, GuardDuty, Security Hub, and others that rely on service-managed permissions.

**Delegated administration and governance boundaries**

Not all organization-level administration happens through member-account permissions. Some AWS services support delegated administrators, where a designated account manages the service for the organization.

Important distinctions:

- SCPs govern what principals in member accounts can do.
- Delegated admin controls who administers the service at the organization level.
- The management account often retains special control-plane authority.
- SCPs are not a substitute for service-specific delegation models.

For exam scenarios, separate these ideas:

- **Who can administer the organization or service?**
- **What actions are allowed inside member accounts?**
- **What does the SCP block regardless of IAM?**

**Practical CLI workflow**

You typically manage SCPs with AWS Organizations commands. The policy content must be valid JSON, and the account must be in an organization with SCPs enabled.

Create a policy:

```bash
aws organizations create-policy \
  --content file://deny-guardduty-delete.json \
  --description "Prevent GuardDuty detector deletion" \
  --name DenyGuardDutyDelete \
  --type SERVICE_CONTROL_POLICY
```

Attach it to an OU:

```bash
aws organizations attach-policy \
  --policy-id p-abc123example \
  --target-id ou-abcd-11111111
```

List policies attached to a target:

```bash
aws organizations list-policies-for-target \
  --target-id ou-abcd-11111111 \
  --filter SERVICE_CONTROL_POLICY
```

Describe a policy:

```bash
aws organizations describe-policy \
  --policy-id p-abc123example
```

Check the organization structure:

```bash
aws organizations list-roots
aws organizations list-organizational-units-for-parent --parent-id r-examplerootid
aws organizations list-accounts-for-parent --parent-id ou-abcd-11111111
```

**Troubleshooting AccessDenied**

When a user gets `AccessDenied`, do not assume IAM is the only cause. SCPs are a common hidden blocker.

A practical troubleshooting flow:

1. Confirm the account is a member account in an organization with SCPs enabled.
2. Check which SCPs are attached to the account’s OU path.
3. Look for explicit denies in the SCPs.
4. Verify IAM identity policies and resource policies.
5. Check whether the action depends on a service-linked role or a global service workflow.
6. Confirm the request region if a region restriction is in place.

Useful commands and checks:

```bash
aws organizations list-policies-for-target \
  --target-id 123456789012 \
  --filter SERVICE_CONTROL_POLICY
```

```bash
aws organizations describe-effective-policy \
  --policy-type SERVICE_CONTROL_POLICY \
  --target-id 123456789012
```

> `aws iam simulate-principal-policy` is useful for IAM evaluation, but it does not evaluate SCPs. Use it only as part of a broader investigation.

A simple incident example:

- A developer can launch EC2 instances in IAM, but the launch fails in the account.
- The SCP denies `ec2:RunInstances` in production OUs.
- IAM simulation shows allow, but the real request still fails because the SCP blocks it.
- The fix is to adjust the OU attachment or create a narrower exception SCP.

**Incident scenario: preventing accidental security-service disablement**

Imagine a production OU where users have broad IAM permissions. A team member tries to disable GuardDuty to reduce noise during testing.

What happens:

- IAM may allow the action.
- The SCP denies `guardduty:DeleteDetector`.
- The request fails with `AccessDenied`.
- Security monitoring remains active.

Why this is valuable:

- It protects the organization from accidental or malicious security degradation.
- It centralizes the guardrail so individual account admins cannot bypass it.

**Incident scenario: region lockout gone wrong**

A company wants to restrict workloads to `us-east-1` and `us-west-2`. They attach a region-deny SCP but forget to exempt a global service workflow.

What happens:

- Normal regional workloads are blocked outside approved Regions.
- A global service operation also fails because the policy is too broad.
- Teams see unexpected `AccessDenied` errors during setup or management.

How to avoid it:

- Test the policy in a sandbox OU first.
- Review global services and shared control-plane actions.
- Use a staged rollout with a narrow pilot account.

**Exam-focused distinctions**

- **SCPs do not grant permissions**; they only limit them.
- **SCPs do not apply to the management account**.
- **Member account root users are still constrained** by SCPs.
- **FullAWSAccess** is the common baseline allow policy.
- **Deny-only SCPs** are easiest to use when FullAWSAccess remains attached.
- **Allow-list SCPs** require explicit permission for every allowed action.
- **IAM simulation does not test SCPs**.
- **Organization setup matters**; SCPs require the right Organizations configuration.

**Quick comparison: IAM, resource policies, and SCPs**

| Policy type | Primary purpose | Grants access? | Scope |
|---|---|---:|---|
| IAM identity policy | Defines what a principal can do | Yes | Account-level principal |
| Resource policy | Defines who can access a resource | Yes | Specific resource |
| SCP | Sets the maximum permissions in member accounts | No | Organization / OU / account |

**Practical mental model**

Use this rule of thumb:

- If IAM says “yes” but SCP says “no,” the action fails.
- If SCP says “yes” but IAM says “no,” the action fails.
- If both allow it and no resource policy blocks it, the action succeeds.

That model is enough for most SOA-C02 questions and helps you reason through real incidents quickly.

**Study checklist**

- Know that SCPs are an Organizations guardrail, not a permission grant.
- Know that SCPs require an organization with all features enabled.
- Understand the difference between deny-only and allow-list SCP strategies.
- Remember that `FullAWSAccess` changes the meaning of deny-only policies.
- Be able to explain why IAM simulation is not enough for SCP troubleshooting.
- Recognize region restriction and global service edge cases.
- Remember that SCPs do not affect the management account.
