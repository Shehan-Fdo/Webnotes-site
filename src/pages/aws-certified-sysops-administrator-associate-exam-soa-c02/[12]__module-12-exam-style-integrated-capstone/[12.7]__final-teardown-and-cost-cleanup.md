## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Why teardown matters**

Final teardown is the last operational step after labs, migrations, or proof-of-concept work. In AWS, many resources continue to incur charges even when they are idle, so you need a deliberate cleanup process rather than a quick console shutdown. The goal is to remove compute, storage, networking, and monitoring artifacts in the correct order while preserving anything you still need for audit or troubleshooting.

A good teardown process also reduces exam-lab mistakes. If you know which resources are billable and which dependencies block deletion, you can clean up faster and avoid orphaned assets such as unattached EBS volumes, idle Elastic IPs, and forgotten load balancers.

**What typically costs money**

AWS billing is resource-specific, so “stopped” does not always mean “free.” You should identify the common cost drivers before you delete anything.

- **Compute** resources:
  - EC2 instances stop billing for instance-hours when stopped, but attached storage and some networking charges may remain.
  - Auto Scaling groups can relaunch instances if you do not scale them down first.
  - NAT Gateways continue to accrue hourly and data-processing charges until deleted.
- **Storage** resources:
  - EBS volumes continue billing after detachment.
  - Snapshots remain billed in S3-backed snapshot storage.
  - S3 buckets incur storage, request, lifecycle transition, and replication costs.
- **Networking** resources:
  - Elastic IPs can be charged when allocated but not associated with a running instance.
  - Load balancers bill while provisioned.
  - Transit Gateway attachments, VPNs, and Direct Connect-related components can also persist charges.
- **Observability** resources:
  - CloudWatch logs, custom metrics, alarms, dashboards, and retained log data can continue to cost money.
  - AWS Config, VPC Flow Logs, and CloudTrail data events may generate ongoing charges.
- **Security and identity** resources:
  - IAM itself is not typically a direct cost driver, but access keys, roles, and policies can keep other resources reachable.
  - KMS keys can incur charges depending on key type and usage patterns.

**Teardown order and dependency awareness**

Delete resources in an order that respects dependencies. If you remove a parent resource too early, child resources may become harder to identify or delete cleanly.

1. Disable or scale down automation first.
   1. Suspend Auto Scaling processes if needed.
   2. Reduce desired capacity to zero where appropriate.
   3. Disable scheduled actions, event triggers, and pipeline deployments.
2. Remove traffic entry points.
   1. Delete or detach load balancers.
   2. Remove Route 53 records that point to the environment.
   3. Release Elastic IPs after they are no longer associated.
3. Delete compute resources.
   1. Terminate EC2 instances.
   2. Delete launch templates, launch configurations, and Auto Scaling groups.
   3. Remove container services or task definitions if they are no longer needed.
4. Clean up storage.
   1. Empty S3 buckets before deletion.
   2. Delete EBS volumes that are no longer required.
   3. Remove snapshots only after confirming you do not need rollback points.
5. Remove networking components.
   1. Delete NAT Gateways.
   2. Delete unused subnets, route tables, internet gateways, and VPC endpoints.
   3. Delete security groups and network ACLs only after all dependencies are gone.
6. Remove monitoring and logging artifacts.
   1. Delete CloudWatch alarms and dashboards.
   2. Reduce log retention or delete log groups if appropriate.
   3. Disable Config recorders, Flow Logs, and CloudTrail trails if they were created for the lab.
7. Verify identity and encryption leftovers.
   1. Disable and schedule deletion for unused KMS keys.
   2. Remove temporary IAM users, roles, and access keys.
   3. Revoke temporary credentials and session access.

**Practical CLI cleanup examples**

The AWS CLI is useful when you need repeatable teardown steps or when the console makes dependency chains hard to see. Always confirm the target region and account before deleting anything.

```bash
aws sts get-caller-identity
aws configure list
aws ec2 describe-instances --filters "Name=tag:Project,Values=lab"
```

To terminate an EC2 instance:

```bash
aws ec2 terminate-instances --instance-ids i-0123456789abcdef0
aws ec2 wait instance-terminated --instance-ids i-0123456789abcdef0
```

To delete an unattached EBS volume:

```bash
aws ec2 describe-volumes --filters "Name=status,Values=available"
aws ec2 delete-volume --volume-id vol-0123456789abcdef0
```

To empty and delete an S3 bucket:

```bash
aws s3 rm s3://my-lab-bucket --recursive
aws s3api delete-bucket --bucket my-lab-bucket
```

To delete a CloudWatch log group:

```bash
aws logs describe-log-groups --log-group-name-prefix "/aws/lambda/"
aws logs delete-log-group --log-group-name "/aws/lambda/my-function"
```

To release an Elastic IP:

```bash
aws ec2 describe-addresses
aws ec2 release-address --allocation-id eipalloc-0123456789abcdef0
```

**Common resource-specific cleanup patterns**

Some resources require a specific sequence before deletion. Knowing these patterns prevents failed delete attempts and partial cleanup.

- **S3 buckets**
  - You must empty the bucket before deletion.
  - Versioned buckets require deleting all object versions and delete markers.
  - Multipart uploads may need aborting if they are still in progress.
- **EBS volumes and snapshots**
  - Detach a volume before deletion unless it is already unused.
  - Delete snapshots only after confirming they are not part of a backup or AMI strategy.
  - Remember that deleting an AMI does not automatically delete its backing snapshots.
- **Load balancers**
  - Deregister targets and remove listeners before deleting the load balancer.
  - Check for associated security groups and target groups that may remain after deletion.
- **NAT Gateways**
  - Delete route table entries that point to the NAT Gateway first.
  - Then delete the NAT Gateway itself to stop hourly charges.
- **Auto Scaling**
  - Set desired capacity to zero or delete the group to prevent instance recreation.
  - Remove scaling policies and scheduled actions if they are still attached.
- **CloudFormation stacks**
  - Delete the stack to remove most resources in dependency order.
  - Use stack policies carefully; they can block deletion of protected resources.
  - If deletion fails, inspect retained resources and nested stacks.

**Verification after teardown**

Cleanup is not complete until you verify that the environment is actually gone. Billing surprises often come from one overlooked resource in a different region.

- Check all active regions, not just the one you used for the lab.
- Review the AWS Billing and Cost Management console for recent usage.
- Inspect resource lists for leftovers:
  - EC2 instances, volumes, snapshots, and Elastic IPs
  - S3 buckets and object versions
  - Load balancers, target groups, and NAT Gateways
  - CloudWatch log groups, alarms, and dashboards
  - IAM users, roles, policies, and access keys
- Use tags such as `Project`, `Owner`, or `Environment` to filter and confirm deletion.
- Confirm that automation is disabled so deleted resources are not recreated.

A quick region sweep with the CLI can help you find leftovers:

```bash
for region in us-east-1 us-east-2 us-west-1 us-west-2; do
  echo "== $region =="
  aws ec2 describe-instances --region "$region" --query 'Reservations[].Instances[].InstanceId' --output text
  aws ec2 describe-volumes --region "$region" --filters Name=status,Values=available --query 'Volumes[].VolumeId' --output text
  aws ec2 describe-addresses --region "$region" --query 'Addresses[].AllocationId' --output text
done
```

**Cost-control habits for labs and exams**

The best cleanup strategy starts before you create resources. If you build with teardown in mind, you reduce both cost and risk.

- Tag every resource with a consistent lab tag set.
  - Example: `Project=SOA-C02-Lab`, `Owner=you`, `Expires=2026-04-26`.
- Prefer ephemeral architectures.
  - Use Auto Scaling groups, launch templates, and CloudFormation stacks so you can delete one parent object instead of many children.
- Set budgets and alerts.
  - AWS Budgets can notify you when spend crosses a threshold.
  - Cost Anomaly Detection can help catch unexpected usage patterns.
- Use lifecycle policies.
  - S3 lifecycle rules can expire objects and transition data automatically.
  - EBS snapshot lifecycle management can reduce long-term storage growth.
- Minimize always-on services.
  - Replace NAT Gateways with temporary alternatives only when appropriate for a lab.
  - Turn off nonessential logging once validation is complete.
- Record the teardown plan.
  - Keep a checklist of resources created during the lab.
  - Include region, account, and tag filters so you can verify deletion quickly.

**Troubleshooting deletion failures**

Deletion often fails because another resource still depends on the target. Read the error message carefully; it usually identifies the blocker.

- If an S3 bucket will not delete, check for:
  - Object versions
  - Delete markers
  - Replication configuration
  - Object Lock or retention settings
- If a security group will not delete, check for:
  - ENIs (Elastic Network Interfaces) still attached
  - Load balancers using the group
  - RDS or Lambda VPC attachments
- If a subnet will not delete, check for:
  - ENIs
  - NAT Gateways
  - VPC endpoints
  - EC2 instances or managed services still present
- If a KMS key will not disappear immediately, remember:
  - You usually schedule deletion rather than remove it instantly.
  - Pending deletion has a waiting period.
- If a CloudFormation stack is stuck, inspect:
  - Events for the first failed delete
  - Retained resources
  - Nested stacks and custom resources

> Be careful with shared environments. Deleting a resource that another team or lab still uses can break unrelated workloads and create avoidable downtime.

**A practical teardown checklist**

Use a repeatable checklist so you do not miss billable resources.

1. Confirm the correct account and region.
2. Stop automation that could recreate resources.
3. Terminate compute and scale groups to zero.
4. Empty and delete storage resources.
5. Delete networking components that are no longer referenced.
6. Remove monitoring, logging, and audit artifacts.
7. Schedule deletion for unused encryption keys.
8. Revoke temporary access and remove lab IAM entities.
9. Verify all regions for leftovers.
10. Review billing and cost dashboards for unexpected usage.

**Exam mindset**

For SOA-C02, expect questions that test whether you understand resource dependencies, billing implications, and safe cleanup order. The correct answer is often the one that removes the cost driver first while preserving the ability to delete dependent resources cleanly.

Focus on these ideas:

- **Dependency order** matters more than raw speed.
- **Stopped** resources may still cost money.
- **Verification** across regions is essential.
- **Tags** and automation make cleanup easier.
- **CloudFormation** can simplify teardown when the environment was built as code.
