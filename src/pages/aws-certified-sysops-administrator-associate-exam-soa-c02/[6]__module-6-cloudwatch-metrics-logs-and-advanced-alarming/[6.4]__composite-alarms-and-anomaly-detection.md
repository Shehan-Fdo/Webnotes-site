## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**CloudWatch alarm strategy**

Amazon CloudWatch alarms help you detect when a metric crosses a threshold, but real systems often need more than a single metric. Composite alarms and anomaly detection reduce noise and improve signal quality by combining alarm states or learning expected metric behavior.

Use **composite alarms** when you want to evaluate the state of other alarms and trigger actions only when a logical condition is met. Use **anomaly detection** when you want CloudWatch to learn a normal pattern and alert on statistically unusual behavior instead of a fixed threshold.

- **Metric alarms** evaluate one metric or a metric math expression.
- **Composite alarms** evaluate alarm states, not raw metrics.
- **Anomaly detection** builds a dynamic band around expected behavior.

**Composite alarms**

A composite alarm uses a rule that references the state of other alarms. It does not directly evaluate a metric. Instead, it evaluates whether referenced alarms are in `ALARM`, `OK`, or `INSUFFICIENT_DATA` state and combines those states with Boolean logic.

Supported rule elements include:

- `ALARM("AlarmName")`
- `OK("AlarmName")`
- `INSUFFICIENT_DATA("AlarmName")`
- `AND`
- `OR`
- `NOT`
- `TRUE`
- `FALSE`

A composite alarm is useful when you want to reduce alert fatigue, suppress notifications during maintenance, or create higher-level service health signals from multiple lower-level alarms.

<dl>
<dt><strong>Why use it</strong></dt>
<dd>To aggregate multiple alarms into one operational signal and avoid paging on every low-level symptom.</dd>
<dt><strong>What it evaluates</strong></dt>
<dd>The state of other alarms, not metric values.</dd>
<dt><strong>What it can do</strong></dt>
<dd>Trigger actions such as notifications or incident workflows when the composite itself enters `ALARM`.</dd>
</dl>

**Rule syntax and behavior**

A composite alarm rule is a logical expression. Example:

```text
ALARM(High5xxErrors) AND ALARM(HighLatency) AND NOT ALARM(MaintenanceWindow)
```

This means the composite alarm enters `ALARM` only when both `High5xxErrors` and `HighLatency` are in `ALARM`, and `MaintenanceWindow` is not in `ALARM`.

> A composite alarm does not automatically suppress other alarms. It changes only its own state and actions. If you want maintenance suppression, you design the rule and actions so the composite alarm behaves that way.

Important exam points:

- `TRUE` and `FALSE` are valid literals and can be used for testing or forcing a rule outcome.
- A composite alarm can reference alarms in the same Region; it is not a general cross-Region construct.
- Cross-account use is possible only with specific CloudWatch setup and permissions; do not assume unrestricted cross-account or cross-Region composition.
- Composite alarms can help suppress notifications by gating actions, but they do not mute underlying alarms by themselves.

**Common composite alarm patterns**

- **Service health aggregation**
  - Combine multiple component alarms into one service-level alarm.
  - Example: database, application, and load balancer alarms feed a single service composite.

- **Maintenance suppression**
  - Use a maintenance alarm or scheduled condition in the rule.
  - Example: page only when a production alarm is active and maintenance is not active.

- **Noise reduction**
  - Alert only when multiple symptoms occur together.
  - Example: CPU high alone may not page, but CPU high plus latency plus error rate may.

- **Escalation tiers**
  - Use one composite for warning and another for critical conditions.
  - Example: warning when one subsystem degrades, critical when several alarms fire together.

**Creating composite alarms**

You can create composite alarms in the AWS Management Console, AWS CLI, or CloudFormation.

In the **console**, you typically provide:

- Alarm name
- Alarm rule
- Actions enabled
- SNS topic or other action targets
- Tags and optional description

In the **CLI**, you use `put-composite-alarm`:

```bash
aws cloudwatch put-composite-alarm \
  --alarm-name "ProdServiceCritical" \
  --alarm-rule 'ALARM("High5xxErrors") AND ALARM("HighLatency") AND NOT ALARM("MaintenanceWindow")' \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:ops-critical
```

In **CloudFormation**, use `AWS::CloudWatch::CompositeAlarm`:

```yaml
Resources:
  ProdServiceCritical:
    Type: AWS::CloudWatch::CompositeAlarm
    Properties:
      AlarmName: ProdServiceCritical
      AlarmRule: 'ALARM("High5xxErrors") AND ALARM("HighLatency") AND NOT ALARM("MaintenanceWindow")'
      AlarmActions:
        - arn:aws:sns:us-east-1:123456789012:ops-critical
```

Operational notes:

- Composite alarms can have actions for `ALARM`, and optionally `OK` actions depending on your design.
- They do not use `Threshold`, `ComparisonOperator`, or `MetricName` because they are not metric alarms.
- You manage them like other CloudWatch alarms, but the key input is the rule expression.

**When to choose composite alarms**

Choose a composite alarm when you need to:

- Combine multiple alarm states into one higher-level signal.
- Reduce duplicate notifications.
- Gate paging on multiple conditions.
- Model operational health across a subsystem.

Do not use a composite alarm when you need to:

- Detect a numeric threshold directly from a metric.
- Perform arithmetic on metrics to create a derived value.
- Alert on a single metric expression result.

For derived values, use **metric math alarms** instead. Metric math evaluates expressions over metrics, while composite alarms evaluate alarm states.

| Feature | Metric alarm / metric math | Composite alarm |
|---|---|---|
| Evaluates raw metrics | Yes | No |
| Evaluates expressions over metrics | Yes | No |
| Evaluates other alarm states | No | Yes |
| Good for thresholding | Yes | No |
| Good for alert aggregation | No | Yes |
| Good for suppression logic | Limited | Yes |

**Anomaly detection**

CloudWatch anomaly detection uses machine learning to create a band of expected behavior around a metric. Instead of a fixed threshold, CloudWatch learns the normal pattern from historical data and flags values that fall outside the band.

This is useful for metrics with seasonality or changing baselines, such as:

- Request count with daily cycles
- Latency with predictable traffic patterns
- CPU utilization with recurring business-hour peaks

Anomaly detection is less suitable for:

- Sparse metrics with many missing points
- Irregular or highly erratic metrics
- Metrics with too little history to learn a stable pattern
- Metrics whose behavior changes abruptly for legitimate reasons

**How the band works**

CloudWatch generates an upper and lower expected band. The width of the band depends on the selected sensitivity, often expressed as the number of standard deviations. A narrower band is more sensitive and produces more alarms; a wider band is less sensitive and reduces false positives.

- Lower sensitivity value or fewer standard deviations: tighter band, more alerts.
- Higher sensitivity value or more standard deviations: wider band, fewer alerts.

You usually create separate alarms for upper and lower breaches because the alarm comparison operator is directional.

- Upper breach: metric goes above the upper band.
- Lower breach: metric goes below the lower band.

**Anomaly detection alarm configuration**

An anomaly detection alarm uses a metric math expression for the band and then compares the metric against that band.

Typical structure:

- One metric ID for the base metric
- One expression using `ANOMALY_DETECTION_BAND(...)`
- `ThresholdMetricId` pointing to the band expression
- A comparison operator such as `GreaterThanUpperThreshold` or `LessThanLowerThreshold`

Example CLI:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name "HighLatencyAnomaly" \
  --comparison-operator GreaterThanUpperThreshold \
  --evaluation-periods 2 \
  --datapoints-to-alarm 2 \
  --threshold-metric-id ad1 \
  --metrics '[
    {
      "Id": "m1",
      "MetricStat": {
        "Metric": {
          "Namespace": "AWS/ApplicationELB",
          "MetricName": "TargetResponseTime",
          "Dimensions": [
            {"Name": "LoadBalancer", "Value": "app/my-alb/1234567890abcdef"}
          ]
        },
        "Period": 60,
        "Stat": "Average"
      },
      "ReturnData": true
    },
    {
      "Id": "ad1",
      "Expression": "ANOMALY_DETECTION_BAND(m1, 2)",
      "Label": "Expected band",
      "ReturnData": true
    }
  ]' \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:ops-alerts
```

Key details:

- The anomaly band expression must be included in the `--metrics` array.
- The alarm evaluates the band expression, not a static threshold.
- The metric statistic must be appropriate for the metric and use case.
- The period should match the metric’s sampling pattern and operational needs.

**Practical configuration notes**

- Use a stable metric with enough history for training.
- Prefer metrics with regular seasonality or predictable patterns.
- Review the generated band before relying on it for paging.
- Tune sensitivity to balance false positives and missed detections.
- Revisit the alarm after traffic patterns or deployments change.

**Console workflow for anomaly detection**

In the CloudWatch console, you typically:

1. Choose the metric.
2. Select anomaly detection or create an alarm from the metric graph.
3. Enable the anomaly band.
4. Adjust sensitivity.
5. Choose whether the alarm triggers on upper breach, lower breach, or both by creating separate alarms.
6. Configure actions and notification targets.

**CloudFormation example for anomaly detection**

```yaml
Resources:
  HighLatencyAnomaly:
    Type: AWS::CloudWatch::Alarm
    Properties:
      AlarmName: HighLatencyAnomaly
      ComparisonOperator: GreaterThanUpperThreshold
      EvaluationPeriods: 2
      DatapointsToAlarm: 2
      Metrics:
        - Id: m1
          MetricStat:
            Metric:
              Namespace: AWS/ApplicationELB
              MetricName: TargetResponseTime
              Dimensions:
                - Name: LoadBalancer
                  Value: app/my-alb/1234567890abcdef
            Period: 60
            Stat: Average
          ReturnData: true
        - Id: ad1
          Expression: ANOMALY_DETECTION_BAND(m1, 2)
          Label: Expected band
          ReturnData: true
      ThresholdMetricId: ad1
      AlarmActions:
        - arn:aws:sns:us-east-1:123456789012:ops-alerts
```

**Composite alarms plus anomaly detection together**

A strong real-world design is to use anomaly detection for a noisy metric and then wrap it in a composite alarm with other signals.

Example topology:

- `LatencyAnomalyHigh` detects unusual latency.
- `High5xxErrors` detects elevated server errors.
- `MaintenanceWindow` indicates planned maintenance.
- `CriticalServiceHealth` is a composite alarm:

```text
ALARM("LatencyAnomalyHigh") AND ALARM("High5xxErrors") AND NOT ALARM("MaintenanceWindow")
```

This design helps you:

- Detect unusual behavior without a fixed threshold.
- Avoid paging on a single noisy symptom.
- Suppress alerts during planned work by controlling the composite rule.
- Escalate only when multiple indicators point to a real incident.

**Exam-relevant distinctions**

- **Composite alarms** are for alarm-state logic.
- **Anomaly detection alarms** are for dynamic metric thresholds.
- **Metric math alarms** are for computed metric values.
- Composite alarms do not evaluate metrics directly.
- Anomaly detection does not replace good metric selection or operational tuning.

**Troubleshooting and limitations**

If a composite alarm does not behave as expected:

- Verify each referenced alarm name exactly matches the rule.
- Check whether the referenced alarms are in the expected state.
- Confirm the composite rule logic with parentheses where needed.
- Ensure actions are attached to the composite alarm, not only the underlying alarms.

If an anomaly detection alarm is noisy or unstable:

- Increase the band width by adjusting sensitivity.
- Use a more stable statistic, such as `Average` or a percentile where appropriate.
- Confirm the metric has enough historical data.
- Avoid sparse metrics unless the use case tolerates occasional instability.
- Consider whether a fixed threshold is simpler and more reliable.

> Anomaly detection is not a universal replacement for thresholds. If the metric is sparse, irregular, or operationally simple, a standard alarm may be more dependable.

**Quick comparison**

| Topic | Composite alarm | Anomaly detection alarm |
|---|---|---|
| Primary purpose | Combine alarm states | Detect unusual metric behavior |
| Input | Other alarms | Metric plus learned band |
| Logic type | Boolean rule | Statistical model |
| Best for | Aggregation, suppression, escalation | Seasonal or dynamic baselines |
| Direct metric evaluation | No | Yes |
| Typical exam clue | “Only alert when multiple alarms fire” | “Alert when behavior deviates from normal” |

**Key takeaways**

- Use composite alarms to build higher-level operational signals from existing alarms.
- Use anomaly detection to alert on deviations from learned normal behavior.
- Remember that composite alarms evaluate alarm states, not metrics.
- Remember that anomaly detection alarms usually need separate upper and lower alarms if you want both directions covered.
- For SOA-C02, be ready to choose between metric alarms, metric math, composite alarms, and anomaly detection based on the monitoring goal.
