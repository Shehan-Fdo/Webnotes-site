## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**Why metric filters matter**

CloudWatch Logs metric filters let you extract operational signals from raw log streams and publish them as CloudWatch metrics. That makes them useful when the thing you want to alarm on is not already exposed as a native metric, such as application errors, authentication failures, or specific business events.

A typical pipeline looks like this:

```text
Log event -> Metric filter match -> CloudWatch metric datapoint -> Alarm evaluation -> Notification or automation
```

Use metric filters when you need to count occurrences of a pattern over time, then trigger alarms, dashboards, or automation from that count. They are especially useful for legacy applications, custom applications, and situations where you cannot easily instrument code with embedded metrics.

**Core concepts**

A metric filter scans incoming log events in a log group and checks whether each event matches a filter pattern. When a match occurs, CloudWatch publishes a metric datapoint to a namespace you choose.

- **Log group**: The container that receives log streams from a source such as EC2, Lambda, ECS, or an application.
- **Metric filter**: A rule attached to a log group that matches log events and emits metric data.
- **Metric namespace**: The logical container for the custom metric, such as `MyApp/Logs`.
- **Metric name**: The metric identifier, such as `ErrorCount`.
- **Metric value**: The value published for each match, often `1`.
- **Dimensions**: Optional key-value pairs that split a metric into multiple time series.

Metric filters are not the same as CloudWatch Logs Insights queries. Insights is for interactive analysis and ad hoc querying; metric filters are for continuous metric publication and alarm integration.

**Filter pattern syntax**

CloudWatch Logs metric filters support specific filter pattern syntaxes, and exam questions often test the difference between them.

- **Space-delimited patterns**: Match fields in logs that are separated by spaces.
- **JSON patterns**: Match JSON log fields by key and value.
- **Standalone terms and phrases**: Match literal text in a log event.
- **Regular expressions**: Supported in metric filters, but only in the filter pattern context and with service-specific constraints. Do not assume full general-purpose regex behavior everywhere.

> Metric filters use filter patterns, not arbitrary SQL-like queries. If a question asks how to match logs continuously and publish a metric, think metric filter syntax first, not Logs Insights.

**Space-delimited example**

A space-delimited pattern is useful for Apache-style or custom text logs.

```text
[ip, user, timestamp, request, status_code = 5*, bytes]
```

This pattern matches events where the `status_code` field starts with `5`, which is useful for counting server errors.

Example log line:

```text
10.0.0.1 alice [26/Apr/2026:12:00:01 +0000] "GET /api/orders" 500 1234
```

If the event matches, CloudWatch can publish a metric value of `1` for that event.

**JSON example**

For JSON logs, you can match on fields directly.

```text
{ $.level = "ERROR" && $.service = "checkout" }
```

Example log event:

```json
{
  "level": "ERROR",
  "service": "checkout",
  "message": "Payment authorization failed"
}
```

This is a common pattern for application logs because JSON is easy to parse and filter.

**Regular expression example**

Regex is useful when the log format is inconsistent or when you need pattern matching beyond simple equality.

```text
{ $.message = /timeout|timed out|deadline exceeded/ }
```

Use regex carefully. For exam purposes, remember that metric filters are still constrained by CloudWatch Logs filter pattern rules, so you should not treat them like a full regex engine over arbitrary text.

**How matching and aggregation work**

A metric filter does not create one metric stream per log event. Instead, it publishes datapoints into the selected metric over time.

- Each matching log event can contribute a metric value, commonly `1`.
- CloudWatch aggregates datapoints within the metric period when alarms evaluate the metric.
- If multiple log events match during the same period, the metric statistic reflects the sum, count, or other selected statistic depending on how the metric is used.
- For a simple error counter, the usual design is to publish `1` per match and alarm on `Sum`.

Example:

- 3 matching error log events occur in one minute.
- The metric filter publishes three datapoints with value `1`.
- The one-minute `Sum` becomes `3`.
- An alarm with threshold `>= 1` or `>= 3` can enter `ALARM` depending on the configuration.

This is important for troubleshooting: if you expect one alarm per error but see only one datapoint, remember that CloudWatch metrics are aggregated by period, not by individual log line.

**Dimensions and default value constraints**

Dimensions let you break a metric into multiple time series, which is useful when you want to alarm separately by service, environment, or error type.

Example dimensioned metric filter:

```json
{
  "filterPattern": "{ $.level = \"ERROR\" }",
  "metricTransformations": [
    {
      "metricNamespace": "MyApp/Logs",
      "metricName": "ErrorCount",
      "metricValue": "1",
      "defaultValue": 0,
      "dimensions": {
        "Service": "$.service",
        "Environment": "$.env"
      }
    }
  ]
}
```

Exam-relevant constraints:

- If you use **dimensions**, you generally cannot use `defaultValue` in the same metric transformation.
- `defaultValue` is intended to publish a value when there are no matches, but dimensioned metrics create multiple time series and do not behave like a single undifferentiated metric.
- A metric filter with dimensions is best when you want per-dimension visibility, not when you need a guaranteed zero every period.
- If you need a zero baseline for alarms, design carefully; many log-based alarms rely on `TreatMissingData` rather than `defaultValue`.

> If a question combines dimensions and default values, be cautious: the safe exam answer is that `defaultValue` is not used with dimensioned metric filters.

**End-to-end example: log event to alarm**

Suppose your application emits JSON logs and you want to alarm when checkout failures occur.

1. You create a metric filter on the `app/checkout` log group.
2. The filter matches events where `level = ERROR` and `service = checkout`.
3. Each match publishes `1` to `MyApp/Logs:CheckoutErrorCount`.
4. A CloudWatch alarm evaluates the metric every minute.
5. If the sum is `>= 1` for one period, the alarm transitions to `ALARM`.
6. An SNS topic sends a notification to the operations team.

Example log event:

```json
{
  "timestamp": "2026-04-26T12:00:01Z",
  "level": "ERROR",
  "service": "checkout",
  "message": "Card authorization failed"
}
```

Example alarm logic:

- Period: `60 seconds`
- Statistic: `Sum`
- Threshold: `>= 1`
- Evaluation periods: `1`
- Datapoints to alarm: `1`

This means one matching error in a minute is enough to trigger the alarm.

**CloudWatch alarm missing-data behavior**

Log-based alarms often face sparse data, so missing-data treatment matters.

CloudWatch alarm options for missing data include:

- **breaching**: Treat missing data as violating the threshold.
- **notBreaching**: Treat missing data as healthy.
- **ignore**: Keep the current alarm state when data is missing.
- **missing**: Use missing data as missing, which can lead to `INSUFFICIENT_DATA`.

How to think about these options for log-based alarms:

- Use **breaching** when the absence of expected log data is itself a problem, such as a heartbeat-like log event.
- Use **notBreaching** when logs are intermittent and you only care about actual error matches.
- Use **ignore** when you want the alarm state to remain stable during gaps.
- Use **missing** when you want CloudWatch to surface uncertainty explicitly.

For most error-count alarms, `notBreaching` is common because no error logs usually means the system is healthy. For heartbeat-style log monitoring, `breaching` may be more appropriate if the expected log line stops appearing.

**Testing and validation**

Before you rely on a metric filter for production alarms, validate it.

In the console, you can use the metric filter test feature against sample log events to confirm the pattern matches what you expect. This is the fastest way to catch syntax mistakes, field mismatches, and unexpected log formats.

CLI workflow example:

```bash
aws logs test-metric-filter \
  --filter-pattern '{ $.level = "ERROR" && $.service = "checkout" }' \
  --log-event-messages '{
    "level":"ERROR",
    "service":"checkout",
    "message":"Card authorization failed"
  }'
```

You can also inspect the resulting metric in CloudWatch and verify that datapoints appear after matching log events are ingested.

Useful validation steps:

- Confirm the log event format in the source application.
- Test the filter pattern against representative sample events.
- Verify the metric namespace and name.
- Check the alarm period and statistic.
- Confirm the alarm action target, such as SNS or Auto Scaling.

**Practical CLI and CloudFormation examples**

Create a metric filter with the AWS CLI:

```bash
aws logs put-metric-filter \
  --log-group-name /app/checkout \
  --filter-name CheckoutErrors \
  --filter-pattern '{ $.level = "ERROR" && $.service = "checkout" }' \
  --metric-transformations metricName=CheckoutErrorCount,metricNamespace=MyApp/Logs,metricValue=1
```

Create a space-delimited filter:

```bash
aws logs put-metric-filter \
  --log-group-name /app/web \
  --filter-name Http5xx \
  --filter-pattern '[ip, user, timestamp, request, status_code = 5*, bytes]' \
  --metric-transformations metricName=Http5xxCount,metricNamespace=MyApp/Logs,metricValue=1
```

CloudFormation example:

```yaml
Resources:
  CheckoutErrorFilter:
    Type: AWS::Logs::MetricFilter
    Properties:
      LogGroupName: /app/checkout
      FilterPattern: '{ $.level = "ERROR" && $.service = "checkout" }'
      MetricTransformations:
        - MetricNamespace: MyApp/Logs
          MetricName: CheckoutErrorCount
          MetricValue: "1"
```

CloudFormation alarm example:

```yaml
Resources:
  CheckoutErrorAlarm:
    Type: AWS::CloudWatch::Alarm
    Properties:
      AlarmDescription: Alarm when checkout errors appear in logs
      Namespace: MyApp/Logs
      MetricName: CheckoutErrorCount
      Statistic: Sum
      Period: 60
      EvaluationPeriods: 1
      DatapointsToAlarm: 1
      Threshold: 1
      ComparisonOperator: GreaterThanOrEqualToThreshold
      TreatMissingData: notBreaching
      AlarmActions:
        - !Ref OpsTopic
```

**Troubleshooting workflow**

When a log-based alarm does not behave as expected, troubleshoot from the source outward.

1. Verify that the application is actually writing the expected log line.
2. Confirm the log group name and ingestion path.
3. Test the filter pattern against a known sample event.
4. Check whether the metric is being published in the correct namespace.
5. Confirm the alarm period, statistic, and threshold.
6. Review missing-data treatment.
7. Check whether dimensions are splitting the metric into multiple time series.
8. Verify that the alarm action target is configured and reachable.

Common failure modes:

- The log format changed, so the filter no longer matches.
- The filter pattern uses JSON syntax against plain text logs.
- The alarm uses `Average` when the design expects `Sum`.
- Dimensions create separate metrics, so the alarm watches the wrong time series.
- Missing data is treated as `missing`, causing `INSUFFICIENT_DATA`.
- The metric filter is attached to the wrong log group.

**Scenario-driven troubleshooting example**

You expect an alarm when checkout failures occur, but the alarm never fires.

- The application logs this line:
  ```json
  {"level":"ERROR","service":"checkout","message":"Card authorization failed"}
  ```
- Your filter pattern is:
  ```text
  { $.level = "ERROR" && $.service = "payments" }
  ```
- The filter never matches because `service` is `checkout`, not `payments`.
- No metric datapoints are published.
- The alarm remains `OK` or `INSUFFICIENT_DATA`, depending on missing-data settings.

Fix:

```text
{ $.level = "ERROR" && $.service = "checkout" }
```

This kind of mismatch is common in exams and in real operations. Always validate the exact field names and values in the log source.

**When to use metric filters versus other options**

Use metric filters when you need:

- Continuous counting of log events.
- Alarm integration from logs.
- Simple operational signals from application output.
- A low-friction way to monitor legacy or custom systems.

Prefer other tools when you need:

- Interactive search and analysis of logs over time: use Logs Insights.
- Complex parsing or transformation before storage: use subscription filters, Lambda, or a log pipeline.
- Native application metrics: instrument the application directly or use embedded metric format when appropriate.

**Exam focus points**

- Metric filters convert log patterns into CloudWatch metrics.
- Filter patterns are not the same as Logs Insights queries.
- Multiple matching log events in one period aggregate into the metric statistic, usually `Sum`.
- Dimensions split metrics into separate time series and affect how you alarm.
- `defaultValue` and dimensions have important constraints; do not assume they work together.
- Missing-data treatment can change whether a log-based alarm stays `OK`, becomes `ALARM`, or enters `INSUFFICIENT_DATA`.
- Always validate the filter before depending on it for alerting.

**Quick reference**

| Topic | Key point |
|---|---|
| Metric filter purpose | Turn log matches into CloudWatch metrics |
| Best alarm statistic | Usually `Sum` for event counts |
| JSON logs | Match fields with JSON filter patterns |
| Text logs | Use space-delimited or literal patterns |
| Dimensions | Split one metric into multiple time series |
| `defaultValue` | Not used with dimensioned metric filters |
| Missing data | Configure alarm behavior explicitly |
| Validation | Test patterns before production use |

**Mental model**

Think of metric filters as a bridge between logs and metrics. Logs tell you what happened; metrics let CloudWatch alarm on how often it happened. The exam usually rewards candidates who can trace that bridge end to end and explain why a specific pattern, statistic, or missing-data setting is chosen.
