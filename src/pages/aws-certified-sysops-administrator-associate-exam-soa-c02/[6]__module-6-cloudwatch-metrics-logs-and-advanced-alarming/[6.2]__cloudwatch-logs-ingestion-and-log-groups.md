## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**CloudWatch Logs overview**

Amazon CloudWatch Logs is the managed service you use to collect, store, search, and monitor log data from AWS services, applications, and on-premises systems. For exam purposes, focus on the ingestion path, the log group hierarchy, retention and access controls, and the differences between log group classes.

CloudWatch Logs is not just a storage bucket for text. It is a managed log pipeline with:
- **Producers** such as EC2 instances, Lambda functions, ECS tasks, VPC Flow Logs, Route 53 Resolver query logs, and custom applications.
- **Destinations** such as log groups, subscription targets, and export workflows.
- **Consumers** such as CloudWatch Logs Insights, metric filters, alarms, and external analytics systems.

A useful mental model is:
- A **log group** is the container.
- A **log stream** is an ordered sequence of events from one source.
- A **log event** is one record with a timestamp and message.

**Core data model**

A log group is the top-level container for logs that share common retention, access, and class settings. Within a log group, log streams separate events by source instance, container, function version, or other producer identity.

- **Log group**: logical container for related logs, such as `/aws/lambda/my-function` or `/aws/ec2/web-tier`.
- **Log stream**: sequence of events from one source, often one instance or one execution environment.
- **Log event**: a single record with a timestamp and message payload.
- **Timestamp**: used for display, filtering, and query semantics; ingestion order still matters for how events are written and returned within a stream.

CloudWatch Logs stores events in the order they are ingested into a stream. Timestamps are still important for search and analysis, but do not assume the console is simply sorting by timestamp alone.

**Log group naming and organization**

You should design log group names to make ownership, environment, and workload easy to identify. Consistent naming helps with retention policies, IAM permissions, and automation.

Common patterns include:
- `/aws/lambda/<function-name>` for Lambda-managed logs.
- `/aws/ec2/<application>` for custom application logs shipped from instances.
- `/aws/ecs/<cluster>/<service>` for container workloads.
- `/company/<env>/<app>` for custom naming conventions.

Good naming practices:
- Use a stable prefix for environment, such as `prod`, `stage`, or `dev`.
- Keep names predictable for automation and IAM policy scoping.
- Avoid embedding rapidly changing values that create unnecessary new log groups.

**Log ingestion sources**

CloudWatch Logs can receive data from many AWS services and from your own agents or SDK calls.

Typical ingestion sources:
- **AWS Lambda** automatically writes function logs to CloudWatch Logs.
- **Amazon EC2** instances can ship logs with the CloudWatch agent or the older unified agent.
- **Amazon ECS** and **Amazon EKS** workloads can send container logs through the `awslogs` driver or a logging pipeline.
- **AWS services** such as VPC Flow Logs, API Gateway, Route 53 Resolver, and others can deliver logs directly.
- **Custom applications** can call the CloudWatch Logs API directly.

For exam scenarios, identify whether the service writes logs automatically or whether you must configure an agent, driver, or API integration.

**Lambda logging behavior**

Lambda automatically creates a log group and log stream pattern for each function if logging is enabled and permissions allow it. The function runtime writes `stdout` and `stderr` output to CloudWatch Logs.

Key points:
- Lambda typically uses the log group `/aws/lambda/<function-name>`.
- Each execution environment can create its own log stream.
- The function execution role should include the standard managed policy `AWSLambdaBasicExecutionRole` or equivalent permissions for `logs:CreateLogGroup`, `logs:CreateLogStream`, and `logs:PutLogEvents`.
- If you pre-create the log group, Lambda still needs permission to write to it.

Example IAM policy for Lambda logging:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "LambdaLogging",
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogGroup",
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "*"
    }
  ]
}
```

A simple Node.js example:
```javascript
exports.handler = async (event) => {
  console.log("Received event:", JSON.stringify(event));
  console.error("Example error line");
  return { statusCode: 200, body: "ok" };
};
```

**EC2 and on-premises ingestion**

For EC2 instances and on-premises servers, you usually install the CloudWatch agent and configure it to collect files, system logs, or application logs. The agent can also publish metrics, but for this topic the important part is log shipping.

Typical use cases:
- Ship application logs from a Linux server.
- Collect Windows Event Logs.
- Forward custom log files from on-premises hosts over the internet or through a private network path.

Example CloudWatch agent configuration:
```json
{
  "logs": {
    "logs_collected": {
      "files": {
        "collect_list": [
          {
            "file_path": "/var/log/syslog",
            "log_group_name": "/company/prod/linux/system",
            "log_stream_name": "{instance_id}",
            "timezone": "UTC"
          }
        ]
      }
    }
  }
}
```

Example agent start commands:
```bash
sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \
  -a fetch-config \
  -m ec2 \
  -c file:/opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json \
  -s
```

> Use a log file path that actually exists on the target operating system. Linux distributions differ, and Windows uses Event Logs rather than Unix-style paths.

**ECS, EKS, and container logging**

Container workloads often use a logging driver or sidecar to send logs to CloudWatch Logs. The simplest pattern is the `awslogs` log driver for ECS tasks.

Example ECS container definition snippet:
```json
{
  "logConfiguration": {
    "logDriver": "awslogs",
    "options": {
      "awslogs-group": "/company/prod/orders",
      "awslogs-region": "us-east-1",
      "awslogs-stream-prefix": "ecs"
    }
  }
}
```

Important considerations:
- The task execution role must allow log delivery if the platform requires it.
- Stream prefixes help you distinguish tasks, containers, and environments.
- For Kubernetes, a DaemonSet or logging agent often forwards container logs to CloudWatch Logs.

**Log groups and retention**

A log group defines the retention policy for all streams inside it unless you override it by changing the group setting. Retention controls how long CloudWatch Logs keeps events before deleting them.

Retention choices:
- **Never expire**: logs remain until you delete them manually.
- **Finite retention**: logs expire after a set period such as 1 day, 7 days, 30 days, 90 days, 1 year, and so on.

Why retention matters:
- Controls storage cost.
- Supports compliance and data minimization.
- Prevents old logs from accumulating indefinitely.

Example CLI to set retention:
```bash
aws logs put-retention-policy \
  --log-group-name "/company/prod/orders" \
  --retention-in-days 30
```

You can also remove a retention policy to return to indefinite retention:
```bash
aws logs delete-retention-policy \
  --log-group-name "/company/prod/orders"
```

**Log group classes**

CloudWatch Logs log groups support two classes: **Standard** and **Infrequent Access**.

- **Standard**: supports the full CloudWatch Logs feature set and is the default choice for active operational logs.
- **Infrequent Access**: lowers storage cost for logs you query less often, but it has feature limitations compared with Standard.

Practical guidance:
- Use **Standard** for logs that need frequent querying, subscriptions, or broader feature support.
- Use **Infrequent Access** for older or less frequently accessed logs where cost reduction matters more than advanced functionality.

> For exam prep, remember that class choice affects feature availability. Do not assume every CloudWatch Logs capability works the same way in both classes.

**Creating and managing log groups**

You can create log groups manually or let services create them automatically. Manual creation is useful when you want to predefine retention, encryption, tags, or class before the first log event arrives.

Example CLI:
```bash
aws logs create-log-group \
  --log-group-name "/company/prod/orders"
```

Set a log group class at creation time when needed:
```bash
aws logs create-log-group \
  --log-group-name "/company/prod/archive" \
  --log-group-class INFREQUENT_ACCESS
```

Add tags for cost allocation and governance:
```bash
aws logs tag-log-group \
  --log-group-name "/company/prod/orders" \
  --tags Environment=prod Application=orders
```

**Permissions and access control**

CloudWatch Logs access is controlled with IAM. You typically need permissions for creating groups, creating streams, writing events, reading logs, and managing subscriptions or exports.

Common actions:
- `logs:CreateLogGroup`
- `logs:CreateLogStream`
- `logs:PutLogEvents`
- `logs:DescribeLogStreams`
- `logs:FilterLogEvents`
- `logs:GetLogEvents`
- `logs:StartQuery`
- `logs:StopQuery`

A minimal read-only policy for troubleshooting might allow:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "logs:DescribeLogGroups",
        "logs:DescribeLogStreams",
        "logs:GetLogEvents",
        "logs:FilterLogEvents",
        "logs:StartQuery",
        "logs:StopQuery"
      ],
      "Resource": "*"
    }
  ]
}
```

Use resource-level permissions where possible, especially for production environments with many teams.

**Writing log events with the API**

If you write directly to CloudWatch Logs, you create or target a log stream and send batches of events with `PutLogEvents`. The API expects events to be ordered by timestamp within the batch and to respect service limits.

Example flow:
1. Create the log group if needed.
2. Create the log stream.
3. Send batches of events with `put-log-events`.

Example CLI using a JSON file for the event batch:
```json
[
  {
    "timestamp": 1710000000000,
    "message": "application started"
  },
  {
    "timestamp": 1710000005000,
    "message": "ready to serve requests"
  }
]
```

```bash
aws logs put-log-events \
  --log-group-name "/company/prod/orders" \
  --log-stream-name "i-0123456789abcdef0" \
  --log-events file://events.json
```

Operational notes:
- Use millisecond Unix epoch timestamps.
- Batch events in the format expected by the AWS CLI and your shell environment.
- Do not rely on a strict sequence-token workflow as the only mental model; AWS has relaxed some sequencing constraints, but you still need to handle API responses and concurrency correctly in real systems.

**Subscription filters and log delivery**

Subscription filters stream matching log events to another service for near-real-time processing. Common destinations include Kinesis Data Streams, Kinesis Data Firehose, and Lambda.

Use cases:
- Send security logs to a central analytics pipeline.
- Forward application logs to a SIEM.
- Trigger Lambda-based enrichment or alerting.

Example subscription filter to Kinesis Data Streams:
```bash
aws logs put-subscription-filter \
  --log-group-name "/company/prod/orders" \
  --filter-name "error-stream" \
  --filter-pattern '"ERROR"' \
  --destination-arn "arn:aws:kinesis:us-east-1:123456789012:stream/log-archive"
```

For Lambda destinations, the Lambda function must allow CloudWatch Logs to invoke it. You also need the correct destination ARN and permissions.

Example Lambda permission:
```bash
aws lambda add-permission \
  --function-name log-processor \
  --statement-id CloudWatchLogsInvoke \
  --action lambda:InvokeFunction \
  --principal logs.amazonaws.com \
  --source-arn arn:aws:logs:us-east-1:123456789012:log-group:/company/prod/orders:*
```

**Metric filters and alarms**

Metric filters turn log patterns into CloudWatch metrics. This is useful when you want to alarm on a log condition without building a custom parser.

Examples:
- Count `ERROR` lines.
- Track `5xx` responses.
- Detect authentication failures.

Example metric filter:
```bash
aws logs put-metric-filter \
  --log-group-name "/company/prod/orders" \
  --filter-name "ErrorCount" \
  --filter-pattern '"ERROR"' \
  --metric-transformations metricName=ErrorCount,metricNamespace=App/Logs,metricValue=1
```

Then create a CloudWatch alarm on the custom metric.

**CloudWatch Logs Insights**

Logs Insights is the query engine for CloudWatch Logs. It helps you search and aggregate logs across one or more log groups.

Why it matters:
- Faster troubleshooting than manual scrolling.
- Useful for ad hoc analysis and dashboards.
- Supports structured fields when logs are JSON.

Example query:
```sql
fields @timestamp, @message
| filter @message like /ERROR/
| sort @timestamp desc
| limit 20
```

If your application emits JSON, you can query fields directly:
```sql
fields @timestamp, level, requestId, message
| filter level = "error"
| sort @timestamp desc
| limit 50
```

Remember:
- Query performance depends on the amount of data scanned.
- Structured logging improves query usefulness.
- Log group class can affect which features are available.

**Exporting and archiving logs**

CloudWatch Logs can export data to Amazon S3 for long-term retention, compliance, or offline analytics. This is different from subscription filters, which are near real-time.

Use export when:
- You need a historical archive.
- You want to analyze logs with Athena, Glue, or other S3-based tools.
- You want to reduce active log storage costs.

Example export task:
```bash
aws logs create-export-task \
  --task-name "orders-archive-2024-01" \
  --log-group-name "/company/prod/orders" \
  --from 1711929600000 \
  --to 1712016000000 \
  --destination "my-log-archive-bucket"
```

**Encryption and security**

CloudWatch Logs supports encryption at rest with AWS Key Management Service (AWS KMS). Use a customer managed key when you need tighter control over key policy, rotation, or cross-team governance.

Security considerations:
- Restrict who can read sensitive logs.
- Encrypt logs that may contain personal or regulated data.
- Use least privilege for producers and consumers.
- Review whether logs contain secrets before shipping them.

**Troubleshooting ingestion**

When logs do not appear, check the producer, permissions, network path, and destination configuration.

Common checks:
- Confirm the application or agent is running.
- Verify the IAM role includes the required CloudWatch Logs actions.
- Confirm the log group name and region are correct.
- Check whether the log stream exists and is writable.
- Review agent or service logs for delivery errors.
- Confirm the destination service accepts the subscription or export.

Useful CLI commands:
```bash
aws logs describe-log-groups --log-group-name-prefix "/company/prod/"
aws logs describe-log-streams --log-group-name "/company/prod/orders"
aws logs filter-log-events --log-group-name "/company/prod/orders" --filter-pattern "ERROR"
```

**Exam-focused distinctions**

- **Log group vs log stream**: the group is the container; the stream is the ordered source-specific feed.
- **Automatic vs manual ingestion**: Lambda and some AWS services write automatically; EC2 and on-premises systems usually need an agent.
- **Retention vs export**: retention deletes old logs from CloudWatch Logs; export copies logs to S3.
- **Subscription filter vs metric filter**: subscription filters forward logs; metric filters create metrics from log patterns.
- **Standard vs Infrequent Access**: Standard supports the full feature set; Infrequent Access is for lower-cost storage with limitations.

**Quick reference**

<dl>
<dt>Best for active troubleshooting</dt>
<dd>Standard log groups with short retention and Logs Insights queries.</dd>
<dt>Best for long-term archive</dt>
<dd>Export to S3 or use longer retention with careful cost control.</dd>
<dt>Best for alerting on patterns</dt>
<dd>Metric filters plus CloudWatch alarms.</dd>
<dt>Best for streaming to another system</dt>
<dd>Subscription filters to Kinesis Data Streams, Firehose, or Lambda.</dd>
</dl>

**Practical design pattern**

A common production pattern is:
- Application writes structured JSON logs.
- CloudWatch agent or service integration ships logs to a Standard log group.
- A short retention policy keeps the active operational window.
- Metric filters create alarms for critical errors.
- Subscription filters forward selected logs to a central security or analytics pipeline.
- Older logs are exported to S3 for archive and compliance.

This pattern balances observability, cost, and operational speed.
