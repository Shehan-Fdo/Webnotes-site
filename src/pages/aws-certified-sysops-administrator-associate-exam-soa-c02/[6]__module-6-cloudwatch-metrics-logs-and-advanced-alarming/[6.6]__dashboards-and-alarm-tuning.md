## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**CloudWatch dashboards: purpose and design**

Amazon CloudWatch dashboards give you a single place to visualize operational health across AWS resources, applications, and custom metrics. You use them to spot trends, compare services, and create a shared view for operations, development, and incident response.

A good dashboard answers operational questions quickly:
- Is the system healthy right now?
- Which component is degrading?
- Is the issue isolated or widespread?
- Are alarms firing for the right reasons?

Dashboards are read-only views of metrics and alarms. They do not store data themselves; they query CloudWatch metrics and present them in widgets. This makes them lightweight and easy to share across teams.

**Core dashboard concepts**

- **Widgets** are the building blocks of a dashboard.
  - Metric widgets show time-series graphs, single values, or stacked views.
  - Text widgets document runbooks, ownership, or incident context.
  - Alarm widgets show alarm state and help you monitor critical conditions.
  - Log widgets can surface CloudWatch Logs data for troubleshooting.
- **Metrics** are the data points that widgets display.
  - AWS service metrics are published automatically by many services.
  - Custom metrics come from your applications or scripts.
- **Namespaces** group related metrics, such as `AWS/EC2`, `AWS/Lambda`, or your own application namespace.
- **Dimensions** filter metrics to a specific resource or logical subset, such as `InstanceId`, `AutoScalingGroupName`, or `FunctionName`.

A dashboard is most useful when it reflects the service architecture rather than a random list of graphs. Group widgets by user journey, tier, or dependency chain so you can move from symptom to cause.

**When to use dashboards**

Use dashboards when you need:
- A real-time operational overview for a service or environment.
- A shared incident response view during outages.
- A management summary of availability, latency, and capacity.
- A way to compare multiple resources, such as several EC2 instances or Lambda functions.

Use multiple dashboards when one view becomes too dense. For example, separate dashboards for:
- Platform health
- Application performance
- Database and storage
- Business KPIs

**Creating and managing dashboards**

You can create dashboards in the AWS Management Console, with the AWS CLI, or through infrastructure as code. The CLI is useful for repeatable deployments and version control.

Example dashboard body:
```json
{
  "widgets": [
    {
      "type": "metric",
      "x": 0,
      "y": 0,
      "width": 12,
      "height": 6,
      "properties": {
        "metrics": [
          [ "AWS/EC2", "CPUUtilization", "InstanceId", "i-0123456789abcdef0" ],
          [ ".", "CPUUtilization", ".", "i-0fedcba9876543210" ]
        ],
        "period": 300,
        "stat": "Average",
        "region": "us-east-1",
        "title": "EC2 CPU Utilization"
      }
    },
    {
      "type": "text",
      "x": 12,
      "y": 0,
      "width": 12,
      "height": 6,
      "properties": {
        "markdown": "## Runbook\n- Check Auto Scaling group health\n- Review ALB target health\n- Inspect application logs"
      }
    }
  ]
}
```

Create or update a dashboard with the CLI:
```bash
aws cloudwatch put-dashboard \
  --dashboard-name production-operations \
  --dashboard-body file://dashboard.json
```

View dashboard metadata:
```bash
aws cloudwatch list-dashboards
aws cloudwatch get-dashboard --dashboard-name production-operations
```

**Dashboard design best practices**

- Put the most important widgets in the top-left area because users scan dashboards in that order.
- Use consistent time ranges and periods across related widgets.
- Prefer fewer, high-signal widgets over many low-value graphs.
- Use annotations to mark deployments, incidents, or scaling events when available.
- Include both symptom metrics and cause metrics.
  - Symptom examples: latency, error rate, throttles.
  - Cause examples: CPU, memory, queue depth, disk I/O.
- Use text widgets to document:
  - Service owner
  - Escalation path
  - Runbook links
  - Known failure modes

A dashboard should support action, not just observation. If a graph does not help you decide what to do next, remove it.

**Alarm fundamentals**

CloudWatch alarms watch a metric and change state when the metric crosses a threshold or meets a rule. Alarms are central to alerting, automation, and self-healing.

Alarm states:
- **OK** means the metric is within the expected range.
- **ALARM** means the threshold condition is met.
- **INSUFFICIENT_DATA** means CloudWatch does not have enough data to evaluate the alarm.

Alarms can trigger actions such as:
- Sending notifications through Amazon SNS
- Stopping, terminating, rebooting, or recovering EC2 instances
- Scaling Auto Scaling groups
- Invoking AWS Lambda functions in some workflows through event-driven integrations

**Alarm types and evaluation**

CloudWatch supports several alarm patterns:
- **Static threshold alarms** compare a metric to a fixed value.
- **Anomaly detection alarms** use a learned band to detect unusual behavior.
- **Composite alarms** combine multiple alarms with Boolean logic to reduce noise.
- **Metric math alarms** evaluate expressions built from one or more metrics.

Alarm evaluation depends on:
- The metric period
- The statistic used, such as `Average`, `Sum`, `Minimum`, or `Maximum`
- The number of evaluation periods
- The datapoints to alarm setting
- Missing data treatment

These settings determine whether an alarm is sensitive, stable, or noisy.

**Tuning alarms to reduce noise**

Alarm tuning is the process of making alarms accurate enough to catch real problems without overwhelming operators with false positives.

Key tuning levers:
- **Threshold**
  - Set it based on real operational limits, not arbitrary numbers.
  - Use historical data to understand normal behavior.
- **Period**
  - Short periods detect issues faster but can increase noise.
  - Longer periods smooth spikes but may delay detection.
- **Evaluation periods**
  - More periods reduce false positives.
  - Fewer periods make alarms more responsive.
- **Datapoints to alarm**
  - Require a subset of breaching datapoints to avoid alerting on one-off spikes.
  - Example: 3 out of 5 periods must breach before alarm state changes.
- **Missing data treatment**
  - `breaching` treats missing data as bad.
  - `notBreaching` treats missing data as healthy.
  - `ignore` keeps the current state.
  - `missing` leaves the alarm in `INSUFFICIENT_DATA` when data is absent.

A common tuning mistake is alerting on every brief spike. For example, CPU can briefly jump during deployments or batch jobs without indicating a real incident. In that case, a longer period or a multi-period alarm is usually better than a very low threshold.

**Example alarm creation**

Create a basic CPU alarm for an EC2 instance:
```bash
aws cloudwatch put-metric-alarm \
  --alarm-name ec2-high-cpu-i-0123456789abcdef0 \
  --alarm-description "Alarm when EC2 CPU exceeds 80%" \
  --namespace AWS/EC2 \
  --metric-name CPUUtilization \
  --dimensions Name=InstanceId,Value=i-0123456789abcdef0 \
  --statistic Average \
  --period 300 \
  --evaluation-periods 2 \
  --datapoints-to-alarm 2 \
  --threshold 80 \
  --comparison-operator GreaterThanThreshold \
  --treat-missing-data missing \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:ops-alerts
```

This configuration waits for two consecutive 5-minute periods above 80% before alarming. That reduces noise from short-lived spikes.

**Choosing the right metric**

The best alarm metric is usually the one that reflects user impact or resource saturation.

Examples:
- For web applications:
  - `5XXErrorRate`
  - `TargetResponseTime`
  - `RequestCount`
- For EC2:
  - `CPUUtilization`
  - `StatusCheckFailed`
  - custom memory or disk metrics
- For queues:
  - `ApproximateNumberOfMessagesVisible`
  - `ApproximateAgeOfOldestMessage`
- For databases:
  - `CPUUtilization`
  - `FreeStorageSpace`
  - `DatabaseConnections`
- For Lambda:
  - `Errors`
  - `Throttles`
  - `Duration`
  - `ConcurrentExecutions`

Prefer metrics that correlate with customer experience or imminent capacity exhaustion. CPU alone is often insufficient; pair it with latency, error rate, or queue depth.

**Composite alarms**

Composite alarms combine multiple underlying alarms using logic such as `AND`, `OR`, and `NOT`. They are useful when you want to reduce alert fatigue.

Use composite alarms when:
- Multiple symptoms must occur together before you page someone.
- You want one alert for a service-level incident instead of many resource-level alerts.
- You need to suppress noisy child alarms during maintenance or known events.

Example logic:
- Page only when both `HighLatency` and `HighErrorRate` are in ALARM.
- Notify a lower-priority channel when either `HighCPU` or `HighMemory` is in ALARM.

Composite alarms do not evaluate metrics directly; they evaluate other alarms. This makes them ideal for incident-level alerting.

**Anomaly detection**

Anomaly detection helps identify behavior that deviates from a learned baseline. It is useful when normal patterns vary by time of day or day of week.

Use anomaly detection when:
- Traffic is seasonal or cyclical.
- A fixed threshold would be too rigid.
- You want to detect unusual spikes or drops without hand-tuning every threshold.

Be careful:
- It needs enough historical data to learn a useful band.
- It may not be ideal for brand-new workloads.
- It should complement, not replace, critical static alarms.

**Missing data and alarm state behavior**

Missing data is one of the most important tuning decisions because it affects whether an alarm is trustworthy.

Common scenarios:
- A metric stops publishing because the instance is down.
- A custom metric is emitted only when requests occur.
- A network issue delays metric ingestion.

Guidance:
- Use `breaching` when missing data itself indicates a problem.
- Use `notBreaching` when silence is expected and should not page you.
- Use `ignore` when you want the alarm to preserve its last known state.
- Use `missing` when you want to surface uncertainty explicitly.

For custom metrics, think carefully about whether the absence of data means healthy or unhealthy. For example, a queue depth metric may be absent if no queue exists, but a heartbeat metric missing from a critical host often indicates a failure.

**Practical alarm tuning workflow**

1. Identify the user-visible symptom you want to detect.
2. Choose the metric that best represents that symptom.
3. Review historical graphs to find normal ranges and seasonal patterns.
4. Set an initial threshold above normal variation but below unacceptable impact.
5. Choose a period and evaluation window that balance speed and stability.
6. Decide how to handle missing data.
7. Test the alarm with a controlled load or a known failure.
8. Review false positives and false negatives after deployment.
9. Adjust thresholds, periods, or composite logic as needed.

This iterative approach is better than trying to perfect the alarm on the first attempt.

**Example: alarm tuning for an Auto Scaling web tier**

Suppose an application uses an Application Load Balancer, an Auto Scaling group, and EC2 instances.

A practical dashboard might include:
- ALB request count
- ALB target response time
- HTTP 4XX and 5XX rates
- EC2 CPU utilization
- EC2 status checks
- Auto Scaling group desired vs. in-service capacity
- Queue depth if asynchronous work exists

A practical alarm set might include:
- High 5XX error rate for the load balancer
- High target response time
- Instance status check failure
- Queue age above a service-level objective
- Composite alarm that pages only when latency and errors are both elevated

This design helps you distinguish between a noisy CPU spike and a real customer-impacting incident.

**CLI examples for inspection and troubleshooting**

List alarms:
```bash
aws cloudwatch describe-alarms
```

Inspect a specific alarm:
```bash
aws cloudwatch describe-alarms --alarm-names ec2-high-cpu-i-0123456789abcdef0
```

List metrics for a namespace:
```bash
aws cloudwatch list-metrics --namespace AWS/EC2
```

Get recent metric data:
```bash
aws cloudwatch get-metric-statistics \
  --namespace AWS/EC2 \
  --metric-name CPUUtilization \
  --dimensions Name=InstanceId,Value=i-0123456789abcdef0 \
  --start-time 2026-04-26T00:00:00Z \
  --end-time 2026-04-26T06:00:00Z \
  --period 300 \
  --statistics Average
```

These commands help you validate whether an alarm threshold matches actual metric behavior.

**Common exam and operational pitfalls**

- Setting alarms on metrics that do not correlate with user impact.
- Using thresholds that are too low and generating alert fatigue.
- Ignoring missing data behavior, which can cause unexpected `INSUFFICIENT_DATA` states.
- Building dashboards with too many widgets and no clear hierarchy.
- Monitoring only symptoms and not root-cause indicators.
- Forgetting to tune alarms after application changes, scaling changes, or seasonal traffic shifts.
- Using one-size-fits-all thresholds across different environments.

**Quick comparison: dashboards vs alarms**

| Feature | Dashboards | Alarms |
|---|---|---|
| Primary purpose | Visualize health and trends | Detect conditions and trigger actions |
| Output | Graphs, text, summaries | State changes: OK, ALARM, INSUFFICIENT_DATA |
| Best use | Situational awareness | Notification and automation |
| Data source | Metrics, logs, alarms | Metrics or other alarms |
| Operational value | Faster diagnosis | Faster response |

Dashboards help you understand what is happening. Alarms help you act when it matters.

**Key takeaways**

- Use dashboards to organize metrics around services, dependencies, and incident response.
- Use alarms to detect actionable conditions, not every metric fluctuation.
- Tune thresholds, periods, and missing data handling to reduce noise.
- Prefer metrics that reflect customer impact or resource exhaustion.
- Use composite alarms and anomaly detection to improve signal quality where appropriate.
- Revisit dashboards and alarms regularly as workloads, traffic patterns, and architectures change.
