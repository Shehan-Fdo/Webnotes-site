## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**CloudWatch metrics overview**

Amazon CloudWatch metrics are time-ordered data points that represent the state or performance of AWS resources and your applications. You use them to observe trends, trigger alarms, and automate operational responses. For the SOA-C02 exam, focus on how metrics are collected, how they are grouped, and when you need to publish your own measurements.

A metric is identified by a combination of:
- **Namespace**: the logical container for related metrics, such as `AWS/EC2` or `AWS/Lambda`.
- **Metric name**: the specific measurement, such as `CPUUtilization` or `Invocations`.
- **Dimensions**: name/value pairs that further identify the metric, such as `InstanceId=i-1234567890abcdef0`.

CloudWatch stores metric data as statistical aggregates over time periods, not as raw event logs. That means you typically work with values such as average, sum, minimum, maximum, and sample count.

**Standard metrics**

Standard metrics are the metrics AWS services publish automatically. You usually do not need to install an agent or write code to get them. These metrics are the first place you look when troubleshooting service health or capacity.

Common examples include:
- **EC2** instance metrics such as `CPUUtilization`, `NetworkIn`, `NetworkOut`, `DiskReadBytes`, and `DiskWriteBytes`.
- **EBS** volume metrics such as `VolumeReadOps`, `VolumeWriteOps`, and `VolumeQueueLength`.
- **RDS** metrics such as `CPUUtilization`, `FreeStorageSpace`, and `DatabaseConnections`.
- **Lambda** metrics such as `Invocations`, `Errors`, `Duration`, and `Throttles`.
- **ELB/ALB** metrics such as `RequestCount`, `HTTPCode_ELB_5XX_Count`, and `TargetResponseTime`.

Standard metrics are useful because they:
- Require minimal setup.
- Integrate directly with alarms and dashboards.
- Provide a baseline for capacity planning and incident response.
- Help you identify whether a problem is in the AWS service layer, the instance layer, or the application layer.

> CloudWatch does not automatically give you every possible operating system or application metric. If you need deeper visibility, you often publish custom metrics or use the CloudWatch agent.

**Metric resolution and retention**

CloudWatch supports different collection granularities. Standard resolution metrics are typically collected at 1-minute intervals for many AWS services, while high-resolution custom metrics can be published at 1-second intervals. High-resolution metrics are useful for fast-moving workloads, but they can increase cost and are not always necessary.

Retention is based on the age of the data:
- High-resolution data is available at finer granularity for a shorter time.
- Older data is rolled up into larger time periods.
- This rollup lets you view long-term trends without storing every raw point forever.

For exam purposes, remember that CloudWatch is designed for operational monitoring and trending, not for long-term forensic storage.

**Custom metrics**

Custom metrics are measurements that you define and publish yourself. Use them when AWS service metrics do not capture the business or application signal you need. Examples include:
- Number of orders processed per minute.
- Queue depth for a proprietary worker system.
- Login failures by application version.
- Cache hit ratio for a custom service.

You publish custom metrics with the CloudWatch API, AWS CLI, SDKs, or embedded metric formats from supported runtimes. A custom metric becomes part of a namespace you choose, which helps you organize application-specific telemetry.

Typical reasons to create custom metrics:
- You need application-level visibility beyond infrastructure metrics.
- You want to alarm on business KPIs, not just resource utilization.
- You need to track internal service health indicators.
- You want to correlate application behavior with AWS resource behavior.

**Namespaces, dimensions, and metric design**

Namespaces prevent metric name collisions and help you separate AWS-managed metrics from your own telemetry. A good namespace is descriptive and stable, such as `MyApp/Payments` or `Company/Orders`.

Dimensions are powerful because they let you slice metrics by attributes. For example:
- `InstanceId`
- `AutoScalingGroupName`
- `Environment`
- `ServiceName`
- `Region`

Use dimensions carefully:
- Add dimensions when they help you filter, aggregate, or alarm on a meaningful subset.
- Avoid excessive cardinality, because too many unique dimension combinations can create operational noise and cost.
- Keep dimension values consistent so dashboards and alarms remain predictable.

A metric with dimensions is more specific than one without them. For example, `CPUUtilization` for all EC2 instances is less useful than `CPUUtilization` for a single `InstanceId` when you are troubleshooting a spike.

**Publishing custom metrics with the AWS CLI**

You can publish a custom metric with `put-metric-data`. This is a common exam-relevant workflow because it shows how custom telemetry enters CloudWatch.

```bash
aws cloudwatch put-metric-data \
  --namespace "MyApp/Orders" \
  --metric-data '[
    {
      "MetricName": "OrdersProcessed",
      "Dimensions": [
        {"Name": "Environment", "Value": "prod"},
        {"Name": "ServiceName", "Value": "checkout"}
      ],
      "Value": 42,
      "Unit": "Count"
    }
  ]'
```

You can also publish multiple data points in one call:
```bash
aws cloudwatch put-metric-data \
  --namespace "MyApp/Orders" \
  --metric-data '[
    {
      "MetricName": "Latency",
      "Dimensions": [{"Name": "Environment", "Value": "prod"}],
      "Value": 187,
      "Unit": "Milliseconds"
    },
    {
      "MetricName": "Errors",
      "Dimensions": [{"Name": "Environment", "Value": "prod"}],
      "Value": 3,
      "Unit": "Count"
    }
  ]'
```

Practical guidance:
- Use `Unit` consistently so graphs and alarms are meaningful.
- Publish values at a regular interval when you want trend analysis.
- Prefer aggregated application metrics over extremely granular event counts when possible.

**CloudWatch agent and operating system metrics**

AWS service metrics do not include all guest OS metrics. If you need memory usage, disk space, swap, or detailed process-level data, you typically install the CloudWatch agent on EC2 instances or on-premises servers.

The agent can collect:
- Memory utilization
- Disk usage
- Disk I/O
- Swap usage
- Additional system-level metrics
- Logs, if configured

Example agent configuration snippet:
```json
{
  "metrics": {
    "namespace": "CWAgent",
    "metrics_collected": {
      "mem": {
        "measurement": [
          "mem_used_percent"
        ]
      },
      "disk": {
        "measurement": [
          "used_percent"
        ],
        "resources": [
          "*"
        ]
      }
    }
  }
}
```

Example installation and start commands on Linux:
```bash
sudo yum install -y amazon-cloudwatch-agent
sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \
  -a fetch-config \
  -m ec2 \
  -c file:/opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json \
  -s
```

Use the agent when:
- Native AWS metrics are insufficient.
- You need memory or filesystem visibility.
- You want to monitor application-specific host telemetry.

**Alarms and metric evaluation**

CloudWatch alarms evaluate metric data against thresholds. This is where standard and custom metrics become operationally useful. You can alarm on:
- High CPU utilization on EC2.
- Low free storage on RDS.
- Elevated error counts in a custom application metric.
- Queue depth above a threshold.

Alarm behavior depends on:
- The metric and dimensions selected.
- The statistic used, such as `Average` or `Sum`.
- The evaluation period and number of datapoints.
- Missing data treatment.

Example alarm creation:
```bash
aws cloudwatch put-metric-alarm \
  --alarm-name "High-EC2-CPU" \
  --alarm-description "Alarm when EC2 CPU exceeds 80 percent" \
  --namespace "AWS/EC2" \
  --metric-name "CPUUtilization" \
  --dimensions Name=InstanceId,Value=i-1234567890abcdef0 \
  --statistic Average \
  --period 300 \
  --evaluation-periods 2 \
  --threshold 80 \
  --comparison-operator GreaterThanThreshold \
  --treat-missing-data missing
```

Key alarm choices:
- Use `Average` for smoothing noisy metrics.
- Use `Sum` for event counts over a period.
- Use `Maximum` when a brief spike matters.
- Use `treat-missing-data` carefully so absent data does not hide failures or create false alarms.

**Metric math and derived signals**

CloudWatch metric math lets you combine metrics to create more useful signals. This is especially helpful when raw metrics do not directly express the condition you care about.

Examples:
- Error rate = `Errors / Invocations * 100`
- CPU headroom = `100 - CPUUtilization`
- Cache hit ratio = `Hits / (Hits + Misses) * 100`

Metric math is useful because:
- It reduces the need to publish redundant custom metrics.
- It lets you alarm on ratios and percentages.
- It helps normalize metrics across different traffic levels.

Example expression in a dashboard or alarm context:
```text
m1 = Invocations
m2 = Errors
e1 = 100 * m2 / m1
```

**Standard vs custom metrics**

| Aspect | Standard metrics | Custom metrics |
|---|---|---|
| Source | AWS services | You, your app, or the CloudWatch agent |
| Setup | Automatic or minimal | Requires publishing/configuration |
| Scope | Infrastructure and managed service health | Application, business, or host-specific signals |
| Examples | `CPUUtilization`, `Invocations`, `FreeStorageSpace` | Orders processed, queue depth, login failures |
| Best use | Baseline monitoring and AWS troubleshooting | Gaps in native telemetry and business KPIs |

Use standard metrics first, then add custom metrics when you need a signal that AWS does not provide.

**Common exam scenarios**

For the exam, expect questions that ask you to choose the right metric source or monitoring approach:
- If you need EC2 CPU, network, or EBS activity, use standard AWS metrics.
- If you need memory usage on EC2, use the CloudWatch agent and custom metrics.
- If you need application error rate, publish custom metrics or use metric math.
- If you need a business KPI such as completed orders, publish a custom metric in a dedicated namespace.
- If you need fast detection of short spikes, consider high-resolution custom metrics.

**Operational best practices**

- Use meaningful namespaces and consistent dimension names.
- Keep metric names stable so dashboards and alarms do not break.
- Publish only the metrics you actually use.
- Prefer aggregated metrics over overly granular event streams.
- Set alarms on symptoms that matter to users, not just on raw resource usage.
- Combine metrics with logs and traces when you need root-cause analysis.

**Troubleshooting checklist**

When a metric looks wrong or missing:
- Confirm the namespace and metric name.
- Verify the dimensions match exactly.
- Check the region; CloudWatch metrics are regional.
- Confirm the instance or service is actually publishing data.
- Review IAM permissions if a custom publisher cannot send data.
- Validate the statistic and period used in the graph or alarm.

A common mistake is querying the right metric name with the wrong dimension set. In CloudWatch, dimensions define a distinct time series, so a mismatch can make the metric appear absent.

**Quick reference**

- **Standard metrics**: AWS publishes them automatically.
- **Custom metrics**: You publish them for application or host-specific needs.
- **Namespace**: Groups related metrics.
- **Dimensions**: Identify a specific metric time series.
- **Alarm**: Evaluates a metric against a threshold.
- **Metric math**: Combines metrics into derived values.
- **CloudWatch agent**: Collects OS-level and additional telemetry.
