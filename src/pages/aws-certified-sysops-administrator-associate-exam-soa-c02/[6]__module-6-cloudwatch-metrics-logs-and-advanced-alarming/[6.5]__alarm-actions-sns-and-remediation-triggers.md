## **Course:** AWS Certified SysOps Administrator – Associate (Exam SOA-C02)

**CloudWatch alarm actions: what they do**

Amazon CloudWatch alarms evaluate a metric or composite condition and change state to `OK`, `ALARM`, or `INSUFFICIENT_DATA`. An alarm action runs only when the alarm transitions into a state that has an action configured. This matters for operations because actions are event-driven, not continuously executed.

Alarm actions are commonly used for:

- **Notification** through Amazon Simple Notification Service (SNS)
- **Remediation** through AWS Systems Manager or custom workflows
- **Recovery** for supported Amazon Elastic Compute Cloud (EC2) instances
- **Scaling** for Amazon EC2 Auto Scaling groups

A key exam point is that not every action type supports every alarm state:

- **SNS** can be attached to `ALARM`, `OK`, and `INSUFFICIENT_DATA`
- **EC2 recovery** is for `ALARM` only
- **Auto Scaling** actions are for `ALARM` only
- **Systems Manager Automation** is usually triggered indirectly, not as a native direct alarm action in the same way as SNS or Auto Scaling

> Alarm actions fire on state transition, so repeated evaluations in the same state do not keep re-running the action unless the state changes again or you re-enable/reset the workflow.

**Alarm action prerequisites and constraints**

Before an action can run, the alarm must actually change state. If the metric stays in `ALARM`, the action does not re-fire on every evaluation period. This is important when you are troubleshooting “why didn’t my action run again?”

Other constraints to remember:

- **Regional scope**: alarm actions are regional
- **State-specific support**: some actions are only valid for certain states
- **Metric suitability**: the metric must support the intended operation
- **Permissions**: the alarm service role or target policy must allow the action
- **Action enablement**: actions can be disabled during maintenance or troubleshooting

CloudWatch also lets you disable alarm actions temporarily. This is useful when you are making changes, suppressing noisy alerts, or performing planned maintenance. Disabling actions does not stop the alarm from changing state; it only prevents the configured actions from executing.

**SNS as the primary notification target**

SNS is the most common alarm action because it is simple, flexible, and supports multiple subscribers. A CloudWatch alarm can publish a message to an SNS topic when the alarm enters `ALARM`, `OK`, or `INSUFFICIENT_DATA`, depending on how you configure it.

Typical SNS delivery flow:

1. CloudWatch alarm changes state.
2. CloudWatch publishes a notification to an SNS topic.
3. SNS fan-outs the message to one or more subscriptions.
4. Subscribers receive the notification through their protocol.

Common SNS subscription protocols:

- **Email** for human notification
- **SMS** for mobile alerts
- **HTTPS** for webhook integrations
- **Lambda** for custom processing
- **SQS** for durable queue-based workflows
- **Application endpoints** for mobile push

Subscription confirmation is required for many protocols. For example, email subscriptions remain pending until the recipient confirms the subscription link. If you forget this step, the alarm may publish successfully but no one receives the alert.

A typical SNS topic policy must allow CloudWatch to publish to the topic. In many same-account, same-Region setups, a basic policy is enough. In cross-account or more restricted environments, you may need additional conditions or explicit permissions.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCloudWatchPublish",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudwatch.amazonaws.com"
      },
      "Action": "sns:Publish",
      "Resource": "arn:aws:sns:us-east-1:123456789012:OpsAlerts"
    }
  ]
}
```

Practical SNS setup example:

```bash
aws sns create-topic --name OpsAlerts
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:OpsAlerts \
  --protocol email \
  --notification-endpoint ops-team@example.com
```

CloudWatch alarm messages delivered through SNS include alarm metadata such as:

- Alarm name
- Previous state and new state
- Reason for the transition
- Metric details
- Threshold and comparison operator
- Timestamp

That message format is useful for downstream automation because it gives you enough context to route or parse the event.

**When to use SNS versus direct remediation**

Use SNS when you want:

- Human notification
- Fan-out to multiple systems
- A simple decoupled integration point
- A bridge to Lambda, SQS, or external tooling

Use direct remediation actions when the target is supported and the response is simple, such as EC2 recovery or Auto Scaling scaling actions. Use SNS when you need flexibility or when the next step is not a single AWS-native action.

**EC2 recovery and Auto Scaling actions**

CloudWatch can trigger EC2 recovery for supported instance types and conditions. This is a common exam topic because it is easy to confuse with reboot or stop/start behavior.

EC2 recovery is appropriate when:

- The instance is impaired due to underlying host issues
- You want AWS to move the instance to healthy hardware
- The instance and configuration support recovery

EC2 recovery limitations to remember:

- It is an **ALARM-only** action
- It is not a general-purpose application remediation tool
- It depends on instance support and the nature of the failure
- It is not the same as rebooting the OS

Auto Scaling actions are used to adjust capacity in response to alarms. They are also **ALARM-only** actions.

A common pattern is to alarm on an EC2 metric for an Auto Scaling group, such as `CPUUtilization`, using the `AWS/EC2` namespace with the `AutoScalingGroupName` dimension. That is valid for group-level scaling behavior.

Example alarm concept:

- Namespace: `AWS/EC2`
- Metric: `CPUUtilization`
- Dimension: `AutoScalingGroupName=my-asg`
- Action: scale out when CPU is high

This is a normal and correct design for Auto Scaling policies. The important part is that the metric and dimension match the resource you want to scale.

Example CLI sketch:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name HighASGCPU \
  --namespace AWS/EC2 \
  --metric-name CPUUtilization \
  --dimensions Name=AutoScalingGroupName,Value=my-asg \
  --statistic Average \
  --period 300 \
  --evaluation-periods 2 \
  --threshold 70 \
  --comparison-operator GreaterThanThreshold \
  --alarm-actions arn:aws:autoscaling:us-east-1:123456789012:scalingPolicy:abcd1234-5678-90ef-ghij-klmnopqrstuv:autoScalingGroupName/my-asg:policyName/ScaleOutPolicy
```

**Systems Manager remediation patterns**

AWS Systems Manager is often used for remediation, but CloudWatch alarms do not usually invoke Automation as a simple native alarm action in the same way they invoke SNS or Auto Scaling. In practice, you usually connect the alarm to an intermediary service.

Common patterns:

- **CloudWatch Alarm -> SNS -> Lambda -> Systems Manager Automation**
- **CloudWatch Alarm -> EventBridge rule on alarm state change -> Lambda -> Systems Manager Automation**
- **CloudWatch Alarm -> SNS -> SQS -> worker or orchestrator -> Automation**

A practical Automation pattern is to use a custom runbook that performs a controlled fix, such as restarting a service, applying a patch, or collecting diagnostics.

Example Automation document skeleton:

```yaml
schemaVersion: '0.3'
description: "Restart a service on a Linux instance after alarm notification"
assumeRole: '{{ AutomationAssumeRole }}'
parameters:
  InstanceId:
    type: String
  ServiceName:
    type: String
mainSteps:
  - name: RestartService
    action: aws:runCommand
    inputs:
      DocumentName: AWS-RunShellScript
      InstanceIds:
        - '{{ InstanceId }}'
      Parameters:
        commands:
          - sudo systemctl restart {{ ServiceName }}
```

This is a better remediation pattern than trying to use a Run Command document as if it were an Automation runbook. `AWS-RunShellScript` is a Run Command document, so it is typically invoked by Automation through `aws:runCommand`, not started directly as an Automation document.

Example orchestration flow:

1. Alarm enters `ALARM`.
2. Alarm publishes to SNS.
3. Lambda receives the SNS message.
4. Lambda starts an Automation execution.
5. Automation runs the approved remediation steps.

Example Lambda starter:

```python
import boto3
ssm = boto3.client("ssm")

def handler(event, context):
    ssm.start_automation_execution(
        DocumentName="My-RemediateHighCPU",
        Parameters={
            "InstanceId": ["i-0123456789abcdef0"]
        }
    )
```

**EventBridge versus alarm actions**

EventBridge is useful for broader orchestration, but it is not the same as a direct CloudWatch alarm action target.

Use this distinction:

- **Direct alarm actions**: SNS, EC2 recovery, Auto Scaling
- **Indirect event handling**: EventBridge rules that react to alarm state change events

CloudWatch alarm state changes can be routed into event-driven workflows through AWS event integration patterns, but you should not assume EventBridge is a native alarm action target. For exam purposes, remember that EventBridge is often the orchestration layer, not the alarm action itself.

This is useful when you need:

- Multiple downstream consumers
- Filtering by alarm name, state, or tags
- Centralized routing across many alarms
- Complex branching logic before remediation

**Cross-account and cross-Region considerations**

Alarm actions are regional. That means the alarm and its direct action target generally live in the same Region.

Important distinctions:

- **Same-Region SNS** is the normal and simplest pattern
- **Cross-account SNS** can work with the right topic policy and permissions
- **Cross-Region direct alarm actions** are not a simple native pattern
- **Cross-Region designs** usually require forwarding, replication, or an intermediary service

For cross-account SNS, you may need:

- A topic policy that allows the source account or service principal
- Subscription confirmation in the target account
- Additional conditions for restricted environments

For cross-Region workflows, consider:

- SNS topic in the same Region as the alarm
- Lambda forwarding to another Region
- EventBridge or custom automation to relay the event

**Disabling and re-enabling actions**

During maintenance windows or troubleshooting, you may want the alarm to keep evaluating but not trigger actions.

Operational uses:

- Prevent noisy notifications during planned work
- Avoid auto-remediation while you are testing
- Temporarily suppress scaling during deployments

Example CLI:

```bash
aws cloudwatch disable-alarm-actions --alarm-names HighASGCPU
aws cloudwatch enable-alarm-actions --alarm-names HighASGCPU
```

This is different from disabling the alarm itself. The alarm still changes state and records history; only the configured actions are suppressed.

**Exam-style decision guide**

| Scenario | Best choice | Why |
|---|---|---|
| Notify an operations team when CPU is high | SNS | Simple fan-out and human alerting |
| Restart a service on an instance after a threshold breach | SNS or EventBridge plus Automation | Needs orchestration, not a direct alarm action |
| Recover an impaired EC2 instance | EC2 recovery | Native alarm-driven recovery for supported instances |
| Scale out an Auto Scaling group | Auto Scaling action | Native capacity adjustment |
| Send the same alarm to email and SMS | SNS | Multiple subscriptions from one topic |
| Suppress actions during maintenance | Disable alarm actions | Keeps evaluation but stops execution |

**Practical troubleshooting checklist**

If an alarm action does not run, check:

1. The alarm actually transitioned state.
2. The action is valid for that state.
3. The target service is in the same Region, if required.
4. The SNS topic policy allows CloudWatch to publish.
5. Subscriptions are confirmed and active.
6. The alarm action is enabled.
7. The metric and dimensions match the intended resource.
8. The target workflow has permission to execute.

If SNS publishes but no one receives the message, check:

- Subscription confirmation
- Email spam filters
- HTTPS endpoint response codes
- Lambda permissions and logs
- SQS queue policy and redrive settings

**High-value exam takeaways**

- CloudWatch alarm actions run on **state transition**, not on every evaluation.
- SNS is the most flexible alarm action and supports `ALARM`, `OK`, and `INSUFFICIENT_DATA`.
- EC2 recovery and Auto Scaling actions are **ALARM-only**.
- Systems Manager remediation is usually **indirect**, often through SNS, Lambda, or EventBridge-based orchestration.
- Alarm actions are **regional**, and cross-Region designs usually need an intermediary.
- You can **disable alarm actions** without disabling the alarm itself.
- `AWS/EC2` with `AutoScalingGroupName` is a valid pattern for group-level scaling alarms.
