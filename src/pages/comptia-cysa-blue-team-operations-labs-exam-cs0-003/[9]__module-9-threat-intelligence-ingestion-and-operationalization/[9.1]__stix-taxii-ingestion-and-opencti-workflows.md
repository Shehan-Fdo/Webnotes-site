## **Course:** CompTIA CySA+ Blue Team Operations Labs (Exam CS0-003)

**STIX and TAXII overview**

<abbr title="Structured Threat Information Expression">STIX</abbr> is a language and serialization format for exchanging cyber threat intelligence. It represents suspicion, compromise, attribution, and related context with objects and relationships that can be stored as JSON and machine-read.

<abbr title="Trusted Automated Exchange of Intelligence Information">TAXII</abbr> is the transport layer commonly used to move STIX content between systems. In OpenCTI, TAXII-based ingestion supports importing STIX 2.1 objects through exposed collections and feeds.

- STIX 2.1 is the version referenced in the OpenCTI TAXII push workflow.
- STIX helps you express threat data in a structured format that tools can ingest and analyze.
- TAXII provides a standardized way to exchange that intelligence between platforms.

**OpenCTI ingestion model**

OpenCTI centralizes automated import under `Data > Ingestion`. The built-in ingestion functions include Live Streams, TAXII Feeds, TAXII Push, RSS Feeds, and JSON/CSV Feeds. These options let you bring external intelligence into the platform in a controlled way.

OpenCTI connectors act as gateways for importing data from external systems. Their behavior depends on how each connector is built, so the available data types and configuration options vary by source.

- **Connectors** handle automated import from external systems.
- **TAXII Feeds** pull intelligence from a TAXII source into OpenCTI.
- **TAXII Push** accepts STIX 2.1 objects through an exposed TAXII collection.
- **Automated import** often formats incoming data into a STIX bundle before platform ingestion.

**TAXII push workflow**

TAXII Push is the OpenCTI ingester used when another system sends STIX 2.1 objects into OpenCTI through a TAXII collection. This is useful when the external platform can publish to a TAXII-compatible endpoint and OpenCTI can expose the collection for inbound ingestion.

A practical integration pattern is to confirm the discovery URL and ensure inbound access from the sending system. This matters when a SIEM or other tool supports STIX/TAXII ingestion but does not have a native OpenCTI connector.

> Make sure the discovery endpoint is reachable from the source system before you troubleshoot the content format.

1. Identify the OpenCTI TAXII endpoint and discovery URL.
2. Confirm network access from the external system to the discovery URL.
3. Verify the source can send STIX 2.1 objects to the exposed collection.
4. Validate that OpenCTI receives and imports the objects as expected.

**Feed configuration and source organization**

OpenCTI recommends a clean separation for imported data sources. Create a dedicated user for each source and follow a naming convention such as `[F] Source name` so the account is easy to identify. Assign that user to the `Connectors` group to simplify permissions related to data creation.

OpenCTI also recommends creating a dedicated organization for the source. This improves traceability and keeps imported intelligence organized by origin.

- Use a dedicated user per feed source for clarity.
- Place the user in the `Connectors` group.
- Create a source-specific organization for traceability.
- Keep source naming consistent across ingestion assets.

**Using TAXII feeds in OpenCTI**

TAXII Feeds are configured from `Data > Ingestion` in OpenCTI. The workflow is straightforward: add a new feed, open the TAXII Feeds section, and configure the feed parameters.

ReversingLabs documents this pattern for a premium TAXII feed integration, showing that OpenCTI can ingest standardized STIX-formatted threat intelligence from external providers. The important operational point is that the feed configuration must match the source credentials and connector settings.

- Open `Data > Ingestion`.
- Select `TAXII Feeds`.
- Add a new feed with the `+` button.
- Enter the feed settings required by the source.

**Analysis and validation in OpenCTI**

After ingestion, OpenCTI lets you review imported intelligence in the `Analyses` area. The platform organizes content into tabs such as Reports, Groupings, Malware Analyses, and Notes. These views help you confirm that imported objects are available for analyst review and further enrichment.

Reports act as containers for structured intelligence production. Groupings collect objects that share a context. Malware Analyses capture metadata and results from static or dynamic analysis of malware instances or families.

- Use **Reports** to structure intelligence from a source.
- Use **Groupings** to collect related objects without treating them as formal intelligence production.
- Use **Malware Analyses** when the imported data includes malware analysis results.
- Use **Notes** to capture analyst observations inside the platform.

**Operational takeaways**

STIX gives you the data model, TAXII gives you the transport, and OpenCTI gives you the ingestion and analysis workflow. For exam prep, focus on how these pieces fit together: source authentication, endpoint reachability, ingestion method, and post-import validation.

A reliable workflow is to configure the source, verify the TAXII discovery path, ingest STIX 2.1 content, and then confirm the objects appear in OpenCTI analysis views.
