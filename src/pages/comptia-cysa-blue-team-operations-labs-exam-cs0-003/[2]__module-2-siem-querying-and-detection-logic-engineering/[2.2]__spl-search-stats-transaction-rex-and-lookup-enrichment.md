## **Course:** CompTIA CySA+ Blue Team Operations Labs (Exam CS0-003)

**SPL search fundamentals**

Search Processing Language (SPL) is the query language used in Splunk to find and analyze events. A search starts with `search`, which applies the terms you specify and forms the base of the query. In security work, you use it to narrow large datasets before applying more focused commands.

An **event** is a single entry of data associated with a timestamp. It can be one line or multiple lines, such as a log entry, configuration file, or stack trace. That matters because many SPL workflows begin by identifying the right event type before extracting fields or summarizing results.

- `search` starts the query and filters by terms.
- Events can contain one or multiple lines.
- Use search terms to reduce noise before deeper analysis.
- Keep the query focused on the data you need to investigate.

**Statistical analysis with `stats`**

The `stats` command is used for statistical aggregation in SPL. It helps you summarize events into counts, totals, and other grouped results so you can spot patterns quickly. This is useful when you want to move from raw event volume to a concise view of activity.

Use `stats` when you need to compare values across categories or identify repeated behavior. In blue team workflows, that often means turning many matching events into a smaller set of meaningful results.

- Use `stats` to aggregate matching events.
- Summaries help reveal trends and repeated activity.
- Pair it with search filters to keep the output relevant.
- Use it when raw event detail is less important than the pattern.

**Grouping related events with `transaction`**

The `transaction` command groups events that are conceptually related and may span a duration of time. This is especially useful for multi-step activity, such as a user session or a business process that produces several separate events.

Because transactions can represent a sequence of related actions, they help you reconstruct activity from scattered logs. Use them when the investigation depends on understanding how events connect over time.

- `transaction` groups related events into a single logical unit.
- It is useful for multi-step activity.
- It helps you analyze sessions or other duration-based workflows.
- Use it when event order and relationship matter.

**Field extraction with `rex`**

The `rex` command uses regular expressions to extract fields from event data. It is useful when the data you need is embedded in text and not already available as a structured field. Splunk documentation includes examples of `rex` for extracting values from event content.

Regular expressions let you match patterns precisely, so `rex` is a practical choice when you need to pull out usernames, IDs, or other text fragments from logs. Use it carefully so the pattern matches the data you actually expect.

- `rex` extracts fields from text using regular expressions.
- Use it when the value is present in the event but not yet a field.
- Pattern accuracy matters because extraction depends on the text structure.
- It is useful for turning raw text into searchable fields.

```spl
... | rex field=_raw "(?<user>[A-Za-z0-9._-]+)"
```

**Lookup enrichment**

Lookup enrichment adds external reference data to search results. In practice, you use it to make event data more useful by attaching context from a lookup table, such as a mapping or classification list. This helps you interpret results faster during investigation.

Enrichment is most valuable when raw events do not contain enough context on their own. By adding lookup data, you can turn a basic match into a more informative result set.

- Use lookups to add context to search results.
- Enrichment helps you interpret events faster.
- It is useful when event data needs external reference information.
- Keep lookup data aligned with the fields in your search results.

**Putting the commands together**

These commands often work best as part of one workflow. You search for relevant events, extract fields with `rex` when needed, summarize with `stats`, group related activity with `transaction`, and enrich the output with lookup data. That sequence helps you move from raw logs to investigation-ready results.

A practical approach is to start broad, then narrow and enrich:

1. Search for the relevant events.
2. Extract missing fields with `rex`.
3. Summarize patterns with `stats`.
4. Group related events with `transaction` when the activity spans time.
5. Add lookup context to improve interpretation.

> Use the simplest command that answers the question first. Add `transaction`, `rex`, or lookup enrichment only when the investigation needs more structure or context.

**Investigation mindset**

In security analysis, the goal is not just to retrieve events but to shape them into evidence. SPL gives you the tools to filter, extract, summarize, group, and enrich data so you can understand what happened and why it matters.

Focus on the relationship between the commands:
- `search` finds the candidate events.
- `rex` exposes hidden fields.
- `stats` condenses the data.
- `transaction` rebuilds related activity.
- lookups add context that raw logs may not contain.
