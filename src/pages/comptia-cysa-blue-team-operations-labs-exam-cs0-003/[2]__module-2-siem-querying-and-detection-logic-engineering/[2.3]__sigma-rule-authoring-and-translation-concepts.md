## **Course:** CompTIA CySA+ Blue Team Operations Labs (Exam CS0-003)

**Sigma rule purpose and format**

Sigma is a text-based, generic detection format that uses YAML to describe suspicious or malicious log activity. You use it to write detections once and share them across different tools and environments, especially within SIEMs (Security Information and Event Management systems).

Sigma rules help standardize detection content so analysts can collaborate more easily. Because the format is human-readable, you can inspect the rule structure, understand the intent, and adapt it for different back ends.

- **YAML** gives Sigma a structured, readable rule format.
- **Platform-independent** content supports reuse across SIEM and detection tools.
- **Detection sharing** is a core benefit of the Sigma ecosystem.

**Rule authoring basics**

A Sigma rule is built from metadata and detection logic. The specification defines a file structure with components such as title, identification, name, taxonomy, status, description, license, author, references, date, modified, log source, detection, search identifier, condition, false positives, level, and tags.

When you author a rule, start with the behavior you want to detect, then express it in Sigma’s YAML structure. Keep the rule focused on observable log activity and make the intent clear enough that another analyst can review or translate it.

- **Metadata** explains what the rule is and who maintains it.
- **Log source** identifies where the event data comes from.
- **Detection** defines the matching logic.
- **Condition** ties the search elements together.

**Example rule structure**

A Sigma rule can describe a specific event, such as an account lockout, using fields like `title`, `id`, `status`, `description`, `references`, `author`, `date`, `modified`, and `tags`. The example in the Sigma documentation shows how a rule can be written as a complete YAML file for a real detection use case.

```yaml
title: "Okta User Account Locked Out"
id: 14701da0-4b0f-4ee6-9c95-2ffb4e73bb9a
status: test
description: "Detects when a user account is locked out."
references:
  - https://developer.okta.com/docs/reference/api/system-log/
  - https://developer.okta.com/docs/reference/api/event-types/
author: Austin Songer @austinsonger
date: 2021-09-12
modified: 2022-10-09
tags:
  - attack.impact
```

This structure shows the practical pattern: define the event, document the context, and tag the behavior so the rule is easier to classify and share.

**Translation concepts**

Sigma is designed to be converted into other query languages. That means you write a rule in Sigma once, then translate it into the syntax used by a target platform such as a SIEM query language.

This translation model reduces friction when teams use different tools. Instead of rewriting the same detection logic repeatedly, you maintain a common source rule and generate platform-specific queries from it.

- **Source rule** stays in Sigma format.
- **Target query** matches the destination platform’s syntax.
- **Shared logic** improves consistency across tools.

**Why translation matters**

Translation supports portability. A well-written Sigma rule can be adapted for different detection systems while preserving the original intent. This is useful when teams need to share detections across environments or standardize response workflows.

It also helps with operational efficiency. By using a common detection language, security teams can collaborate on rules, reduce duplication, and improve consistency in alerting.

- **Portability** lets you reuse detections across platforms.
- **Consistency** helps preserve detection intent.
- **Efficiency** reduces repeated rule writing.

**Practical authoring guidance**

Write the rule around a clear behavior, not around a tool-specific query. Use the metadata fields to explain context, and use the detection section to capture the observable indicators from logs.

Keep the rule readable and specific enough to support translation. If the rule is too vague, the translated query may become noisy or less useful. If it is too narrow, it may miss relevant events.

1. Identify the suspicious behavior you want to detect.
2. Map that behavior to available log fields.
3. Write the Sigma rule in YAML.
4. Review metadata, detection logic, and conditions.
5. Translate the rule into the target query language.
6. Validate that the translated query preserves the intended behavior.

**Key takeaways**

Sigma rules are a shared, YAML-based way to describe detections in log data. Their value comes from portability, readability, and the ability to translate one rule into multiple platform-specific queries.

For exam prep, focus on the relationship between rule structure, detection intent, and translation. If you understand how Sigma separates the human-authored rule from the platform-specific query, you can reason about how detection content moves between tools.
