## **Course:** CompTIA CySA+ Blue Team Operations Labs (Exam CS0-003)

**Traffic analysis workflow**

Network traffic analysis helps you identify malicious activity by reviewing packet captures (pcaps) and protocol behavior. In practice, you often use Wireshark to inspect small captures directly, isolate a suspicious stream, and then trace the conversation back to the host or service involved.

A practical workflow is to start broad, then narrow your focus:
- Review the capture for obvious protocol activity and unusual endpoints.
- Follow a TCP stream when you need to reconstruct a conversation.
- Compare what you see against expected behavior for the protocol.
- Preserve the capture and any extracted indicators for later reporting.

When the capture is large, you may need to use other tools or workflows before returning to Wireshark for detailed inspection. The right approach depends on the size of the pcap and the question you are trying to answer.

**DNS and web traffic review**

Domain Name System (DNS) traffic often provides the first clue about where a host is trying to connect. In a suspicious trace, look for unusual domain lookups, repeated queries, or names that do not match the expected application behavior. DNS can help you pivot from a hostname to the rest of the session.

For web traffic, focus on the relationship between HTTP and HTTPS:
- **HTTP** is readable in transit when not protected by encryption.
- **HTTPS** uses Transport Layer Security (TLS) to protect the session content.
- Web traffic analysis often starts with the request path, host header, and timing between requests.

A useful habit is to compare normal browsing patterns with the capture you are investigating. Unexpected destinations, repeated callbacks, or traffic that appears to support command-and-control activity can stand out quickly when you inspect the web layer carefully.

**TLS inspection and encrypted sessions**

TLS protects the contents of many web sessions, which means you may not see the full application payload directly. Even when the content is encrypted, you can still analyze metadata such as endpoints, timing, and certificate-related details to understand the session.

In some cases, you can decrypt traffic in a controlled test environment if you have the required key material. This is especially useful when you need to inspect hidden content in a packet capture. Use that approach only when you have the proper setup and authorization.

> Encryption can hide the payload, but it does not hide the fact that a session exists, who it talks to, or when it happens.

**SMB traffic analysis**

Server Message Block (SMB) traffic is important when you are looking for file sharing, lateral movement, or remote resource access. In a capture, SMB activity may reveal how a host interacts with shared folders or remote systems.

When reviewing SMB, pay attention to:
- The source and destination hosts involved in the session.
- Repeated access to shared resources.
- Unexpected file transfer or browsing behavior.
- Activity that does not fit the normal role of the endpoint.

SMB is especially useful in incident response because it can show movement between systems that may not be obvious from endpoint logs alone.

**RDP traffic analysis**

Remote Desktop Protocol (RDP) is a common remote access protocol and a frequent target in attacks against exposed systems. Since RDP supports encrypted operating modes, its content can be difficult to inspect directly in a packet capture.

RDP analysis is still valuable because you can:
- Identify that RDP is being used.
- Trace the source and destination of the session.
- Look for signs of unauthorized remote access.
- Decrypt traffic in a test environment when you have the required key file.

Because RDP encryption hides content, signature-based detection can be harder. That makes protocol context, session timing, and endpoint relationships especially important during analysis.

**Malicious traffic investigation habits**

When you analyze suspicious traffic, keep your environment safe and your evidence intact. A common practice is to work in an isolated virtual machine with the network interface disabled when you are handling potentially harmful captures or extracted files.

Useful habits include:
- Treat the pcap as evidence and avoid modifying it.
- Extract only what you need for analysis.
- Record indicators of compromise (IOCs) such as IP addresses, domains, and suspicious file references.
- Preserve a timeline of notable events so you can explain the sequence of activity.

If the traffic appears to support malware or command-and-control behavior, focus on the communication pattern as well as the payload. Regular beaconing, repeated callbacks, and unusual remote sessions are often more informative than a single packet.

**Practical analysis focus by protocol**

| Protocol | What to look for | Why it matters |
|---|---|---|
| DNS | Suspicious domains, repeated lookups, unusual resolution patterns | Helps you identify where a host is trying to connect |
| HTTP/S | Hosts, paths, request timing, encrypted vs. readable content | Reveals web activity and possible callback behavior |
| TLS | Endpoints, timing, certificate-related details, hidden payloads | Supports analysis when content is encrypted |
| SMB | Shared resources, file access, lateral movement indicators | Helps detect internal spread or remote file activity |
| RDP | Remote access sessions, source/destination pairs, encrypted traffic | Useful for spotting unauthorized administration or exposure |

**Investigation summary**

Use protocol knowledge to turn raw packets into a story about host behavior. DNS points you toward destinations, HTTP and HTTPS show application activity, TLS explains why content may be hidden, SMB exposes file-sharing behavior, and RDP highlights remote access risk. Together, these protocols help you move from packet-level evidence to a defensible incident narrative.
