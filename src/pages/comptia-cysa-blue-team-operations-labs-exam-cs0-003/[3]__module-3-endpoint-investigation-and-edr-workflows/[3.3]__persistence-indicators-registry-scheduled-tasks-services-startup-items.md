## **Course:** CompTIA CySA+ Blue Team Operations Labs (Exam CS0-003)

**Persistence indicators overview**

Persistence is the ability of a threat actor to keep access after initial compromise. In Windows environments, common indicators include registry autoruns, scheduled tasks, services, and startup items. These mechanisms are legitimate administration features, which makes abuse harder to notice during incident response.

CySA+ focuses on recognizing indicators of malicious activity and using monitoring data to detect them. For this topic, the practical goal is to identify where persistence is likely to appear and what artifacts help you confirm it.

- **Registry** entries can launch programs or scripts at boot or user logon.
- **Scheduled tasks** can run at specific times or under conditions and are often abused for hidden execution.
- **Services** can provide recurring execution through native Windows management.
- **Startup items** can trigger programs when a user signs in.

**Registry-based persistence**

Registry autorun locations are a common persistence mechanism because they are simple, native, and easy to blend into normal system configuration. The evidence highlights `Run` and `RunOnce` locations under both `HKLM` and `HKCU`.

Common locations to review include:

- `HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run`
- `HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\RunOnce`
- `HKCU\SOFTWARE\Microsoft\Windows\CurrentVersion\Run`
- `HKCU\SOFTWARE\Microsoft\Windows\CurrentVersion\RunOnce`

These keys matter because they can point to executables, scripts, or command lines that run automatically. `HKLM` affects the machine, while `HKCU` affects the current user profile. When you investigate, compare the value name, command path, and whether the target is expected for that host.

**Scheduled tasks as persistence**

Windows Scheduled Tasks automate jobs that run at a set time or under a condition. Threat actors abuse them because they are native to Windows and can appear routine. The evidence also notes that scheduled tasks may be used to download and execute scripts, sometimes in memory, which reduces obvious disk artifacts.

Detection is important here. The sources emphasize monitoring with Windows Event Logs and Sysmon logs, and using tools such as Wazuh to alert when new tasks are created.

- Review task creation events and changes to existing tasks.
- Look for suspicious command lines, unusual triggers, or unexpected run contexts.
- Treat task scheduler activity as high priority when it appears alongside other persistence indicators.

> **Warning:** A scheduled task is not malicious by itself. Focus on the task action, trigger, creator, and whether the behavior matches the system’s normal administration pattern.

**Services and startup items**

Services are another traditional persistence method because they can start automatically and run with elevated privileges depending on configuration. Startup items are also common because they execute when a user logs in and are easy for attackers to hide among legitimate shortcuts or scripts.

The evidence groups these with other native persistence methods, so the practical approach is to treat them as part of the same investigation workflow:

- Check for newly added or modified services.
- Review startup folders for unexpected shortcuts or scripts.
- Correlate service or startup changes with registry edits and task creation.

Startup folders are directories where shortcuts to programs or scripts can be placed to execute automatically. That makes them useful for legitimate automation and equally useful for persistence.

**Investigation workflow**

When you suspect persistence, work from the broadest indicator to the most specific artifact. Start with telemetry that shows execution or configuration change, then confirm the persistence mechanism on disk or in the registry.

1. Identify the suspicious behavior from endpoint telemetry, logs, or alerting.
2. Check for registry autorun entries, scheduled tasks, services, and startup items.
3. Compare the artifact against known-good system behavior and expected software.
4. Correlate timestamps, parent processes, and command lines to build the timeline.
5. Validate whether the item is legitimate administration or malicious persistence.

This workflow helps you avoid overreacting to normal automation while still catching native Windows abuse.

**Detection focus for CySA+**

CySA+ expects you to connect indicators with monitoring and response. For persistence, that means you should know where to look and what evidence supports your conclusion.

- **Windows Event Logs** help track task creation and related activity.
- **Sysmon** can generate useful endpoint events for new tasks and process behavior.
- **Registry inspection** helps confirm autorun abuse.
- **Cross-artifact correlation** strengthens your finding when one indicator alone is ambiguous.

A practical mindset is to assume persistence may involve more than one mechanism. For example, a suspicious command-line process may modify the registry and then create a scheduled task. That chain is more informative than any single artifact on its own.

**Key takeaways**

Persistence on Windows often uses built-in features rather than exotic malware techniques. Registry autoruns, scheduled tasks, services, and startup items are all legitimate mechanisms that attackers can repurpose. Your job is to identify the artifact, understand its normal purpose, and determine whether the behavior fits the environment.

- Native features are attractive to attackers because they are trusted and persistent.
- Scheduled tasks deserve close review because they are widely abused and can be hard to detect.
- Registry and startup artifacts often provide the clearest confirmation of autorun behavior.
- Correlation across logs and system artifacts gives you the strongest evidence.
