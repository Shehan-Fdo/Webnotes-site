## **Course:** CompTIA CySA+ Blue Team Operations Labs (Exam CS0-003)

**Command-line analysis in blue team investigations**

PowerShell is a built-in Windows scripting language that defenders use for automation, log analysis, incident response, and system monitoring. Attackers also abuse it because it blends into normal administration and can support fileless or low-noise activity. In practice, you look at the command line, the process tree, and any network behavior together rather than treating a PowerShell alert as a standalone event.

- **PowerShell** is legitimate, but suspicious arguments can indicate malware installation or execution.
- **LOLBins** are trusted system tools abused for malicious activity, which makes behavior-based detection important.
- Command-line review helps you separate routine admin use from encoded, obfuscated, or download-based activity.

**What to look for in suspicious PowerShell activity**

A useful investigation starts with the exact command executed. Alerts often point to suspicious argument values, and those values are the first place to check for signs of abuse. Encoded commands, download commands, and obfuscated scripts are common indicators because they hide intent or retrieve additional payloads.

- Look for `-EncodedCommand` or other signs of encoded content.
- Check for download behavior that may retrieve malware or scripts.
- Treat heavily obfuscated syntax as a higher-risk signal.
- Compare the command against the user’s normal administrative behavior.

**Process context and parent-child relationships**

The parent process often tells you whether PowerShell is being used normally or launched in a suspicious chain. If Office applications such as Word, Excel, or Outlook start PowerShell, that can suggest malicious macro activity or another user-driven execution path that deserves closer review.

- Identify the process that launched PowerShell.
- Pay attention when Office apps are the parent process.
- Review whether the launch path matches expected admin tooling.
- Correlate the process tree with the time of the alert.

**Network activity and user context**

PowerShell may connect to external servers during malicious activity, especially when it is used to fetch payloads or communicate with infrastructure outside the environment. Network connections to unknown IP addresses, suspicious domains, or file download endpoints strengthen the case that the activity is not routine administration. You should also verify which user executed the command so you can judge whether the action fits that account’s role.

- Check for outbound connections to unfamiliar hosts.
- Review any file downloads initiated by the script.
- Confirm the user account and whether the action is expected.
- Use user context to separate admin work from compromise indicators.

**Execution policy and defensive control**

PowerShell execution policy restrictions can reduce risk from untrusted scripts. The evidence highlights three common policy modes: `Restricted`, `RemoteSigned`, and `AllSigned`. These controls do not stop every abuse case, but they help limit what runs and support safer administration.

- `Restricted` blocks all scripts from running.
- `RemoteSigned` allows locally created scripts but requires signatures for downloaded scripts.
- `AllSigned` requires all scripts to be signed by trusted publishers.

> Execution policy is a control, not a guarantee. Attackers can still abuse legitimate tools, so you still need command-line, process, and network analysis.

**Practical investigation flow**

1. Review the exact PowerShell command line.
2. Check for encoded, download, or obfuscated arguments.
3. Identify the parent process and process tree.
4. Inspect network connections and destination hosts.
5. Validate the user account and expected activity.
6. Decide whether the behavior matches normal administration or suspicious execution.

**Key takeaways**

PowerShell is a common administrative tool and a common attacker tool, so context matters. Suspicious arguments, unusual parent processes, and external network activity are the most useful signals in the evidence provided. When you combine those clues with user context, you can triage alerts more confidently and focus on likely malicious execution.
