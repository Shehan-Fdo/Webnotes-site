## **Course:** CompTIA CySA+ Blue Team Operations Labs (Exam CS0-003)

**Containment in incident response**

Containment limits the spread and impact of an incident while you preserve enough evidence to continue investigation. In the SANS <abbr title="Preparation, Identification, Containment, Eradication, Recovery, Lessons Learned">PICERL</abbr> framework, containment is the phase where you stop active harm and reduce exposure before eradication and recovery.

- Use containment to interrupt attacker activity, not to fully remediate.
- Match the response to the asset type and the business impact.
- Prefer actions that are reversible when you still need investigation data.

> Containment choices vary by platform. A network block, host isolation, or identity revocation can all be valid, but each affects users and evidence differently.

**Asset-based containment options**

Different assets support different containment actions. For physical or virtual endpoints, you often isolate the host from the network so you can stop lateral movement while keeping the system available for analysis. For cloud or service workloads, you may need to restrict access, disable exposed interfaces, or apply service-level controls that reduce attack reach without taking every dependent system offline.

- **Endpoints** are often contained by isolating the device from the network.
- **Cloud workloads** are often contained by tightening access or limiting service exposure.
- **Shared services** may require narrower controls so you do not disrupt unrelated users.

When you choose an option, consider whether the asset is user-facing, shared, or part of a larger service chain. A broad shutdown may stop the threat quickly, but it can also create avoidable business disruption. A narrower control may take longer to fully contain the incident, but it can preserve operations while you investigate.

**Identity-based response actions**

Identity compromise often requires actions against tokens, sessions, or accounts rather than only against devices. OAuth 2.0 <abbr title="Open Authorization 2.0">OAuth 2.0</abbr> token revocation lets an authorization server invalidate a previously issued refresh or access token. The revocation request can invalidate the token itself and, when applicable, other tokens based on the same authorization grant.

- Revoking a **refresh token** can invalidate access tokens issued from that refresh token.
- Some systems support revoking the token at the endpoint level through a dedicated revocation request.
- Identity actions may affect only one grant, one client, or one session depending on the platform.

Amazon Cognito notes that revoking a refresh token ends the user session tied to that token and invalidates previously issued access tokens from it, while other refresh tokens for the same user are not affected. It also distinguishes between token revocation and broader sign-out actions. This matters during containment because you may want to end one compromised session without forcing every session offline.

**Token revocation behavior to remember**

Token revocation is not always identical across platforms. RFC 7009 defines the OAuth 2.0 token revocation endpoint as a way for clients to tell the authorization server that a refresh or access token is no longer needed. The request invalidates the actual token and may also invalidate other tokens from the same authorization grant.

- The revocation endpoint is used to clean up security credentials after compromise.
- Some platforms support direct access token revocation only for certain token types.
- JSON Web Token <abbr title="JSON Web Token">JWT</abbr> access tokens may only be indirectly revoked in some systems by revoking the associated refresh token.

PingFederate documents that its revocation endpoint accepts only `POST` requests and that JWT access tokens do not always support direct revocation. In those cases, revocation may depend on the associated refresh token and token-manager configuration. Use conservative wording in incident notes: verify the platform’s revocation model before assuming a token is fully invalidated.

**Practical containment decision points**

Use the least disruptive action that still stops the threat. If the incident is tied to a single endpoint, isolate that host first. If the compromise is identity-driven, revoke the relevant token or session. If the affected asset is a shared cloud service, apply access restrictions that contain the incident without breaking unrelated workloads.

1. Identify the asset type and the scope of compromise.
2. Choose the narrowest containment action that stops attacker activity.
3. Confirm whether the action affects one token, one session, one user, or multiple grants.
4. Monitor for residual access or alternate sessions that remain valid.
5. Escalate to broader containment only if the threat persists.

**Operational cautions**

Containment can have side effects. A token revocation may end access for a user who is still legitimate on another device. A host isolation action may preserve evidence but interrupt business processes. A service-level restriction may reduce exposure while leaving some attacker footholds intact.

- Verify what the platform actually revokes.
- Check whether other sessions remain active.
- Document the business impact of the containment choice.
- Coordinate with recovery so containment does not block restoration.

**Key takeaway**

Containment is a targeted response, not a one-size-fits-all shutdown. Asset type determines whether you isolate a host, restrict a cloud service, or revoke identity credentials. Identity-based containment is especially important because token and session behavior varies by platform, and the same action may affect only one grant or may end multiple access paths.
