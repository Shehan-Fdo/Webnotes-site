## **Course:** CompTIA Network+ (Exam N10-009)

**High availability overview**

High availability (HA) is the design goal of keeping an IT system, component, or application operating continuously with minimal or zero downtime. The practical aim is to reduce service interruption when a device, node, or path fails.

HA is not a single product or feature. It is a set of strategies that help a service stay available even when parts of the infrastructure stop working.

- **Redundancy** adds extra components so a failure does not stop the service.
- **Failover** moves the workload to another system when the primary system fails.
- **Clustering** groups multiple systems so they can act as one service.
- **Load balancing** spreads work across multiple systems to improve availability and handling of demand.

**Failover and clustering**

A failover cluster uses multiple independent computers, called nodes, that work together to support applications and services. These clustered roles are monitored, and if one node fails, another node takes over the workload automatically.

This design helps avoid a single point of failure for critical services. In practice, the cluster is built so the remaining nodes can continue the workload with minimal disruption.

- Nodes are separate computers that cooperate as part of the cluster.
- Clustered roles are the applications or services the cluster supports.
- Failover is the automatic transfer of workload after a failure.
- Health monitoring helps the cluster detect when a role or node is no longer healthy.

> High availability reduces downtime, but it does not eliminate the need for testing, monitoring, and maintenance.

**Common HA characteristics**

HA infrastructure is designed to handle different loads and failures while maintaining service quality. The goal is not just recovery after a problem, but continuous operation during normal use and during component failure.

High-availability clusters are often described as servers grouped together to operate as a single, unified system. They may share storage and support the same workloads as the primary system they protect.

- **Shared storage** can let multiple nodes access the same data set.
- **Separate networks** can support resilience and traffic handling.
- **Regular testing** confirms that nodes are ready to take over.
- **Minimal downtime** is the expected outcome when a failure occurs.

**Redundancy and fault tolerance**

Redundancy is the foundation of many HA designs. You add extra capacity or duplicate components so one failure does not bring down the service. Fault tolerance goes further by aiming to keep the system operating through failures, often with added complexity and cost.

Examples of redundancy include extra storage devices, additional power supplies, and larger-scale server designs that can continue operating when a component fails.

- **Fault tolerance** increases resilience by allowing operation through failure.
- **Additional complexity** is a common tradeoff for stronger availability.
- **Higher cost** often comes with duplicate hardware and supporting infrastructure.

**Load balancing**

Load balancing distributes traffic or work across multiple systems so no single server becomes the only place where the service runs. This improves availability and can help the environment handle more demand.

In high-availability designs, load balancing is useful when a shared resource must remain accessible even if one system becomes unavailable.

- It spreads requests across multiple servers.
- It reduces dependence on one system.
- It supports continuity when one server needs to be removed or fails.

**Planning and tradeoffs**

High availability is a practical design choice, not a guarantee. You balance uptime goals against cost, complexity, and operational effort. More resilient systems usually require more components, more configuration, and more ongoing validation.

When you evaluate an HA design, focus on whether it removes single points of failure and whether the remaining systems can actually take over the workload.

- More resilience usually means more hardware and configuration.
- Testing is necessary to confirm failover works as expected.
- Monitoring helps you detect failures before they become outages.

**Quick comparison**

| Concept | Purpose | Key idea |
|---|---|---|
| Redundancy | Prevent service loss | Duplicate critical components |
| Failover | Restore service automatically | Another node takes over |
| Clustering | Coordinate multiple systems | Nodes act as one service |
| Load balancing | Share demand | Traffic is distributed across systems |

**Exam focus**

For Network+ purposes, focus on the relationship between availability, redundancy, and failover. You should be able to explain how clustered systems reduce downtime and why HA designs are used for critical services.

- HA aims for continuous service with minimal interruption.
- Clusters use multiple nodes to support the same workload.
- Failover helps the service keep running after a node failure.
- Load balancing can be part of an HA strategy.
