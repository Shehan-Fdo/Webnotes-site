## **Course:** CompTIA Network+ (Exam N10-009)

**Shared Responsibility Model Overview**

The **shared responsibility model** defines how security and operational tasks are divided between the cloud service provider and the customer. The exact split depends on the service model you use, so you must confirm which side handles infrastructure, platform services, and application-level controls before you deploy.

In practical terms, the provider secures the cloud service itself, while you remain responsible for what you place in it and how you configure it. This division helps reduce operational burden, but it does not remove your security obligations.

- The **provider** typically secures the underlying cloud environment.
- The **customer** typically secures data, configurations, and workloads they manage.
- Responsibility changes based on whether the service is `SaaS`, `PaaS`, or `IaaS`.

**Service Model Differences**

The shared responsibility split becomes clearer when you compare common cloud models. In `IaaS` (Infrastructure as a Service), you manage virtual machines, operating systems, and applications, while the provider handles the foundational infrastructure. In `PaaS` (Platform as a Service), you deploy applications without managing the virtual machines or operating systems, so the provider takes on more of the platform burden. In `SaaS` (Software as a Service), the provider manages the application and supporting service, and you mainly use the software and protect your data and access.

| Model | Provider responsibility | Customer responsibility |
|---|---|---|
| `IaaS` | Physical infrastructure and core cloud platform | Virtual machines, operating systems, applications |
| `PaaS` | Platform and runtime environment | Application deployment and app-level configuration |
| `SaaS` | Application and service delivery | User access, data handling, and usage controls |

> Responsibility varies by service model, so do not assume one cloud service has the same security split as another.

**What You Should Verify**

Before adopting a cloud service, check which tasks remain yours. The evidence consistently shows that customers still manage some combination of guest operating systems, application software, configuration, and security controls, even when the provider handles the lower layers.

- Confirm who patches the `guest OS` if the service includes one.
- Confirm who manages application software and updates.
- Confirm who configures security controls such as firewall rules or security groups.
- Confirm how the service model affects your compliance obligations.

**Why the Model Matters**

The shared responsibility model helps you avoid gaps in security ownership. If you assume the provider handles everything, you may leave applications, data, or access settings exposed. If you assume too much responsibility on your side, you may duplicate controls or misconfigure the service.

A good cloud security plan starts with a clear ownership map. That map should show which tasks belong to the provider, which belong to you, and which controls need shared coordination.

**Quick Review**

- `SaaS` gives you the least operational management.
- `PaaS` removes operating system management from your workload.
- `IaaS` gives you the most control, but also the most responsibility.
- The provider and customer each own different parts of security and operations.
