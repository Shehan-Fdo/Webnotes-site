## **Course:** CompTIA Network+ (Exam N10-009)

**Cloud troubleshooting mindset**

Cloud troubleshooting focuses on how traffic is routed, how quickly users reach resources, and whether the environment stays available during failures. In cloud environments, DNS (Domain Name System) routing policies often determine which resource a user reaches, so routing behavior is a common place to start when performance or availability changes.

Cloud services also add shared responsibility and distributed dependencies. You need to consider not only the application, but also the region, the DNS policy, the health of endpoints, and the security controls around name resolution.

- **DNS routing** can change which resource answers a query.
- **Latency** affects user experience when resources are spread across regions.
- **Availability** depends on failover behavior and health checks.
- **Security** remains a core requirement for cloud DNS.

**Cloud DNS routing policies**

Cloud DNS routing policies help direct users to the most appropriate endpoint. These policies support high availability, lower latency, and controlled traffic distribution for testing or deployment changes.

| Policy | Practical use |
|---|---|
| `Simple routing` | Sends all traffic to one resource |
| `Weighted routing` | Splits traffic by percentage for load testing or blue/green deployments |
| `Failover routing` | Supports active-passive disaster recovery with health checks |
| `Latency-based routing` | Sends users to the region with the lowest latency |
| `Geolocation routing` | Routes based on the user’s physical location |
| `Geolocation bias` | Adjusts routing with custom bias rules |

When troubleshooting, confirm that the policy matches the intended design. A routing issue may not be a failure at all; it may be the expected result of a policy sending traffic to a different region or endpoint.

> If you are testing latency-based behavior, remember that latency data is based on traffic between users and cloud data centers, so real-world latency can differ when resources are not in an AWS Region.

**Latency-based routing considerations**

Latency-based routing improves performance by directing requests to the AWS Region that provides the lowest latency. To use it, you create latency records for resources in multiple AWS Regions, and the DNS service selects the record for the region with the best latency for the user.

This matters during troubleshooting because a user may report slow access even though the service is healthy. In that case, you should verify whether the user is being routed to the expected region and whether the region selection still matches the current network conditions.

- Check whether latency records exist for the relevant regions.
- Verify that DNS is returning the record for the lowest-latency region.
- Compare the user’s experience with the routing policy in use.
- Remember that geographic proximity does not always equal the lowest measured latency.

**Failover and resilience checks**

Failover routing supports high availability by moving traffic when a primary resource becomes unavailable. In cloud troubleshooting, this means you should verify both the health of the primary endpoint and the behavior of the secondary endpoint.

A failover design is only useful if health checks work correctly and the backup path is ready. If users cannot reach the backup resource, the issue may be with the failover target, the DNS record, or the application behind the target.

1. Confirm that the primary resource is healthy or intentionally marked unavailable.
2. Verify that health checks are passing or failing as expected.
3. Check that the secondary resource is reachable and correctly configured.
4. Validate that DNS responses match the intended failover state.

**Security considerations in cloud DNS**

DNS security is part of cloud troubleshooting because reliability, performance, and security are all required. A DNS issue may involve more than routing; it may also involve protection of the name-resolution path and the services that depend on it.

You should treat DNS as a critical cloud control plane component. If DNS is misconfigured or exposed, users may experience outages, incorrect routing, or degraded trust in the service.

- **Reliability** keeps name resolution available.
- **Performance** helps users reach the right resource quickly.
- **Security** protects DNS from misuse and supports safe cloud operation.

**Practical troubleshooting approach**

Start with the symptom, then trace it back through DNS, routing policy, region selection, and endpoint health. This approach helps you separate application problems from cloud routing behavior.

Use a structured check sequence:

1. Identify the user impact, such as slowness, wrong-region access, or total failure.
2. Review the DNS routing policy in use.
3. Check whether the selected endpoint or region matches the expected design.
4. Validate health checks and failover behavior.
5. Confirm that security controls and DNS configuration are intact.

**Key takeaways**

Cloud troubleshooting often centers on DNS routing and distributed availability. The most useful checks are usually the simplest ones: confirm the routing policy, verify the selected region or endpoint, and make sure health checks and security controls support the intended design.
