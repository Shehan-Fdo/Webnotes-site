## **Course:** CompTIA Network+ (Exam N10-009)

**APIs and REST overview**

An <abbr title="Application Programming Interface">API</abbr> lets one system request data or actions from another system through defined endpoints. In network and automation contexts, APIs provide a controlled way to interact with services without manual intervention.

<abbr title="Representational State Transfer">REST</abbr> is an architectural style for web-based APIs. It is commonly used because it maps well to resource access over the web and supports standard HTTP methods.

- REST stands for **Representational State Transfer**.
- A REST API is also called a **RESTful API**.
- REST is an architectural style, not a protocol or a standard.
- RESTful services expose resources that clients request from a server.

**How REST works**

A client initiates communication by sending a request to a server, often through a web URL. The server responds with a resource, which may be returned in different formats depending on the service.

Common HTTP methods used with REST include:

- `GET` to retrieve a resource
- `POST` to create or submit data
- `PUT` to update a resource
- `DELETE` to remove a resource

> REST services must follow their guiding principles and constraints to be considered RESTful.

**Why APIs matter in networking**

APIs are useful in network automation because they give you a structured way to access services and resources. Instead of relying only on manual configuration, you can use API requests to interact with systems programmatically.

- APIs help automate repetitive tasks.
- REST is widely used for web-based APIs.
- Network and API access controls are important because endpoints can expose sensitive functions and data.
- Authentication, authorization, and usage limits help restrict access to approved requests.

**Access control considerations**

When you expose an API endpoint, treat it like a front door to the service. If it is not protected, attackers may try to steal data, overload the system, or alter behavior.

- Use **authentication** to verify identity.
- Use **authorization** to limit what a requester can do.
- Use **usage limits** to reduce abuse and overload.
- Keep communication private and tamper-resistant where possible.

**Practical REST request pattern**

A typical REST interaction follows a simple request-and-response flow.

1. You send a request to a resource URL.
2. The server processes the request.
3. The server returns the requested resource or a status response.

```http
GET /api/devices/10.1.1.1 HTTP/1.1
Host: example.com
Accept: application/json
```

**Key takeaways**

REST is a common way to build web APIs because it uses familiar web concepts and standard HTTP methods. In network environments, APIs and REST support automation, but they also require strong access controls to protect exposed endpoints.
