## **Course:** CompTIA Network+ (Exam N10-009)

**Automation tools in network operations**

Automation tools help you apply repeatable changes across infrastructure instead of configuring each system by hand. In the Network+ context, the key value is consistency: you reduce manual errors, speed up deployment, and make changes easier to review and reproduce.

Network automation tools are often grouped by what they do:

- **Configuration management** tools apply predefined settings across devices.
- **Infrastructure provisioning** tools create or modify infrastructure resources.
- **Orchestration** tools coordinate multiple steps in a workflow.
- **Scripting frameworks** let you write custom logic for specialized tasks.

Ansible and Nornir are commonly used for configuration management, while Terraform is associated with infrastructure provisioning. Python is often used when you need custom automation logic or integration between tools.

**Ansible and Terraform in practice**

Ansible is commonly used to configure systems after they exist, while Terraform is used to define and provision infrastructure as code. In many environments, you use them together: Terraform creates the infrastructure, and Ansible configures it.

This division of labor matters because it helps you choose the right tool for the job. If you need to build resources, Terraform fits well. If you need to push configuration changes to devices or hosts, Ansible is a better fit.

- **Terraform** focuses on provisioning infrastructure resources.
- **Ansible** focuses on configuration and operational tasks.
- **Together**, they support a full automated deployment workflow.

> Use conservative tool selection: do not force one tool to do every job when a workflow benefits from separate provisioning and configuration stages.

**CI/CD pipelines for automation**

<abbr title="Continuous Integration and Continuous Delivery">CI/CD</abbr> pipelines help you move automation through controlled stages instead of running changes manually. A typical workflow starts with a code push, then runs validation, planning, approval, deployment, configuration, testing, and notification.

A practical pipeline can include these stages:

1. Validate the infrastructure and automation code.
2. Generate a plan so you can review intended changes.
3. Approve the change before applying it.
4. Provision infrastructure.
5. Generate inventory for the newly created systems.
6. Configure the systems.
7. Run smoke tests.
8. Send a notification when the workflow completes.

A GitLab CI example from the evidence uses stages such as `validate`, `plan`, `apply`, `configure`, and `test`. It also separates Terraform and Ansible directories with variables like `TF_DIR` and `ANSIBLE_DIR`.

```yaml
variables:
  TF_DIR: terraform
  ANSIBLE_DIR: ansible

validate:
  stage: validate
  script:
    - cd $TF_DIR && terraform init && terraform validate
    - cd ../$ANSIBLE_DIR && ansible-lint playbook.yml
    - cd ../$ANSIBLE_DIR && ansible-playbook --syntax-check playbook.yml
```

**Version control and change control**

Version control gives you a controlled way to store, review, and track automation code. When your automation lives in a repository, you can review changes before they reach production, compare revisions, and keep a history of what changed and when.

This is especially important for infrastructure as code because the code itself becomes the source of truth for how systems are built and configured. Version control supports:

- **Code review** before deployment.
- **Traceability** for changes over time.
- **Rollback planning** when a change needs to be reversed.
- **Collaboration** across teams working on the same automation.

Version control also fits naturally with CI/CD because the pipeline can run automatically when code changes are committed or merged.

**Choosing the right automation approach**

Different tools solve different problems, so your choice should match the task. Configuration management tools are best when you need to enforce device settings consistently. Provisioning tools are best when you need to create infrastructure resources. Scripting is useful when you need custom logic or glue between systems.

| Tool type | Primary use | Typical strength |
|---|---|---|
| Ansible | Configuration management | Repeatable device and host configuration |
| Terraform | Infrastructure provisioning | Declarative infrastructure as code |
| Python | Scripting and integration | Custom automation logic |
| Nornir | Network automation workflows | Programmatic device automation |

A practical network automation strategy often combines these tools rather than replacing one with another. That approach supports repeatability, review, and controlled deployment across the full lifecycle.

**Key exam takeaways**

- Automation reduces manual work and improves consistency.
- Infrastructure as code lets you define systems in a repeatable way.
- Version control helps you track, review, and manage automation changes.
- CI/CD pipelines make automation safer by adding validation and approval steps.
- Terraform and Ansible are often used together in a provisioning-plus-configuration workflow.
