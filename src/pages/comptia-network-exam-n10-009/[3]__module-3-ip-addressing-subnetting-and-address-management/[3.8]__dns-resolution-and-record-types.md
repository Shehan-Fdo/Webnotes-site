## **Course:** CompTIA Network+ (Exam N10-009)

**DNS resolution overview**

DNS, or <abbr title="Domain Name System">DNS</abbr>, is a hierarchical naming system that translates human-friendly domain names into machine-readable IP addresses. You use it when a browser needs to find the server behind a name such as `example.com`.

The resolution process is usually hidden from the user, but it is essential for web access and other network services. DNS also supports troubleshooting because many connectivity problems start with name resolution rather than the application itself.

- DNS acts like a distributed phone book for network names.
- A resolver queries DNS to find the address or service information it needs.
- Name resolution affects web browsing, email delivery, and service discovery.

**How DNS resolution works**

When you enter a name in a browser, the client asks DNS for the information needed to reach the destination. The lookup may involve multiple steps across the DNS hierarchy before the answer is returned.

In practical terms, you should think of DNS as a lookup system that maps names to different kinds of data, not just IP addresses. That distinction matters because different record types support different services.

1. The client submits a query for a domain name.
2. DNS returns the record type that matches the request.
3. The client uses the returned data to connect to the target service.

> DNS issues can affect name resolution, Active Directory functionality, and general network connectivity.

**Core record types**

DNS stores information in resource records. Each record type serves a specific purpose, and the most common ones are the ones you are most likely to see in exam scenarios and basic administration.

| Record type | Purpose | Typical use |
|---|---|---|
| `A` | Maps a name to an IPv4 address | Hostname to IPv4 lookup |
| `AAAA` | Maps a name to an IPv6 address | Hostname to IPv6 lookup |
| `CNAME` | Points one name to another canonical name | Alias management |
| `MX` | Identifies mail servers | Email routing |
| `NS` | Identifies authoritative name servers | Delegation and zone lookup |
| `PTR` | Maps an IP address back to a name | Reverse lookup |
| `SRV` | Identifies a service location | Service discovery |
| `SOA` | Describes the start of authority for a zone | Zone control data |
| `TXT` | Stores text information | Verification and policy data |

**Address records: `A` and `AAAA`**

`A` records map hostnames to IPv4 addresses, while `AAAA` records map hostnames to IPv6 addresses. These are the most direct records for reaching a host by name.

A host can have more than one address record. That can support redundancy or multiple interfaces, and it is common to see multiple `A` records for the same name in simple load-sharing setups.

- `A` returns a 32-bit IPv4 address.
- `AAAA` returns a 128-bit IPv6 address.
- Multiple address records can exist for the same hostname.

```dns
example.com.     3600 IN A     93.184.216.34
example.com.     3600 IN AAAA  2606:2800:220:1:248:1893:25c8:1946
```

**Alias and delegation records**

`CNAME` records create an alias from one name to another name. You use them when you want one hostname to follow another canonical hostname.

`NS` records identify the authoritative name servers for a zone. They are important for delegation because they tell resolvers where to ask for the zone’s data.

`SOA` records mark the start of authority for a zone and contain the zone’s core administrative data.

- `CNAME` points to another hostname instead of an IP address.
- `NS` identifies the servers responsible for a zone.
- `SOA` defines the zone’s authority information.

**Mail and service records**

`MX` records direct email to the correct mail servers. They are a key part of mail delivery because senders use them to find where to route messages for a domain.

`SRV` records support service discovery. They help clients locate services for protocols that need more than a simple host-to-IP mapping.

`TXT` records store text data. They are often used for verification or policy-related information.

- `MX` supports email routing.
- `SRV` supports service discovery.
- `TXT` stores arbitrary text data for DNS-related use cases.

**Reverse lookup and troubleshooting**

`PTR` records support reverse DNS lookup, which maps an IP address back to a hostname. This is useful when you need to identify a system from its address.

DNS troubleshooting often starts with checking whether the client has valid IP configuration and whether the DNS server can return authoritative data. If name resolution fails, verify both client-side connectivity and server-side DNS behavior.

- Check client IP configuration first.
- Confirm the DNS server is reachable.
- Verify the record exists and matches the expected name.
- Use reverse lookup when you need to map an address back to a host.

**Practical exam focus**

For Network+ questions, focus on what each record does rather than memorizing every possible DNS record type. The most common exam patterns ask you to match a record with a use case.

- `A` and `AAAA` map names to addresses.
- `CNAME` creates aliases.
- `MX` handles mail routing.
- `NS` identifies authoritative servers.
- `PTR` supports reverse lookup.
- `SRV` helps clients find services.
- `SOA` defines zone authority.
- `TXT` stores text-based DNS data.

**Quick recall**

<dl>
<dt><strong>Forward lookup</strong></dt>
<dd>Resolve a name to an IP address using records such as <code>A</code> or <code>AAAA</code>.</dd>
<dt><strong>Reverse lookup</strong></dt>
<dd>Resolve an IP address to a name using a <code>PTR</code> record.</dd>
<dt><strong>Authoritative data</strong></dt>
<dd>The DNS server’s trusted zone information for a domain.</dd>
