## **Course:** CompTIA Network+ (Exam N10-009)

**Zero-trust architecture overview**

Zero-trust architecture (ZTA) is an approach that removes inherent trust from the network. Instead of assuming that users, devices, or applications are safe because they are inside a perimeter, you verify each request against policy.

This model reflects a shift away from the traditional “castle-and-moat” mindset. In zero trust, location alone does not grant access. You evaluate identity, context, and authorization before allowing a connection.

- **No implicit trust** means every request must be checked.
- **Least privilege** limits access to only what is needed.
- **Continuous verification** helps keep access decisions current as conditions change.

**Core principles**

Zero trust is built around the idea that the network should be treated as hostile until proven otherwise. Access is granted only after verification, and that verification is repeated as needed.

<dl>
<dt>Identity verification</dt>
<dd>You confirm the user, device, or application before access is allowed.</dd>
<dt>Context-aware policy</dt>
<dd>You consider factors such as device security and request conditions.</dd>
<dt>Fine-grained authorization</dt>
<dd>You allow access to specific resources instead of broad network reach.</dd>
</dl>

The practical goal is to reduce the attack surface and limit the impact of a breach. If an attacker gains one foothold, zero trust is designed to prevent easy movement across the environment.

**How zero trust changes access control**

Zero trust focuses on access to specific applications and data rather than opening the entire network. This is why zero trust network access (ZTNA) is often associated with the model. ZTNA verifies users and grants access based on identity and context policies.

> Zero trust does not trust a user, device, or application just because it is already connected or located inside the network.

Key effects of this model include:

- Hiding applications and services from broad discovery
- Restricting lateral movement
- Reducing exposure on the public internet
- Applying authorization per request rather than per location

**Identity and access management alignment**

Identity and access management (IAM) is central to zero trust because access decisions depend on proving who or what is requesting access. The model also helps address shadow access, which refers to unintended resource access that can appear in complex cloud and automated environments.

In practice, you strengthen zero trust by making identity checks and authorization rules more precise. This supports a security posture where access is continuously evaluated instead of assumed.

**Implementation focus**

A zero-trust rollout starts with understanding where the model adds the most value. You then apply identity, authentication, and context-based controls to the workloads and applications that need them most.

1. Identify the workloads, applications, and data that need tighter access control.
2. Apply strong identity verification and authorization rules.
3. Use context signals, such as device security, to refine access decisions.
4. Limit access to specific applications instead of broad network segments.
5. Review and adjust policies as the environment changes.

A practical implementation should reduce unnecessary pathways to data and support flexible, identity-aware access. This is especially useful in cloud, hybrid, and remote-work environments.

**Benefits and exam takeaways**

Zero trust is valuable because it improves security without relying on network location as a trust signal. It helps organizations protect data, reduce attack surface, and better control access in modern environments.

- **Security posture** improves when access is verified continuously.
- **Attack surface** shrinks when services are not broadly exposed.
- **Breach impact** is lower when access is tightly scoped.
- **Policy control** becomes more precise with identity-based rules.

For exam purposes, remember the main idea: zero trust assumes no inherent trust, verifies every request, and grants only the access that is explicitly allowed.
