## **Course:** CompTIA Network+ (Exam N10-009)

**AAA overview**

<abbr title="Authentication, Authorization, and Accounting">AAA</abbr> is a security framework that controls access to network resources, enforces policy, and records usage. It supports network management and cybersecurity by screening users and tracking activity while they are connected.

AAA breaks into three separate functions:

- **Authentication** verifies who you are.
- **Authorization** determines what you can access or do.
- **Accounting** records what you did for auditing and review.

In practice, AAA helps you apply access control in a way that aligns with security goals and least privilege. A user may prove identity successfully but still receive limited access based on role or policy.

**Authentication**

Authentication is the identity check. You present credentials such as a username and password, and the AAA system compares them with stored records.

Common points to remember:

- It answers the question: “Who are you?”
- It happens before access is granted.
- It often uses a centralized AAA server for credential validation.

**Authorization**

Authorization happens after authentication succeeds. It defines the resources, commands, or operations a user can use.

- It answers the question: “What are you allowed to do?”
- It can limit access to specific devices, folders, services, or administrative actions.
- It is based on privileges, roles, or policy.

**Accounting**

Accounting tracks activity after access is granted. It creates a record of user actions for auditing, troubleshooting, and oversight.

- It answers the question: “What did you do?”
- It supports logging and review.
- It helps administrators monitor usage and investigate events.

**AAA protocols**

AAA commonly uses centralized protocols that separate the client device from the authentication server. Two names you should recognize are RADIUS and TACACS+.

| Protocol | Practical role | Notes |
|---|---|---|
| RADIUS | Access server AAA protocol | Exchanges user account information between a client and an external server; returns privilege information |
| TACACS+ | AAA protocol similar to RADIUS | Used for centralized access control; some systems check reachability with UDP port 49 before using TCP port 49 |
| LDAP | Directory-based authentication option | Can provide similar functions in some environments |

RADIUS provides an extra layer of login security by sending user account information to a server that authenticates the user and returns privilege details. TACACS+ and LDAP can provide similar centralized functions. Some devices keep local user accounts as a backup if the external AAA service fails.

> Use conservative wording on the exam: protocol behavior can vary by implementation, but the core AAA roles stay the same.

**Certificate basics**

Certificates are part of trust and identity validation in secure network communication. At a high level, a certificate helps prove that a public key belongs to a specific entity.

Key ideas:

- A certificate supports trust by binding identity to a key.
- It is commonly used when systems need to verify a server or user.
- It helps protect communication by making identity checks more reliable.

When you see certificate-related questions, focus on the trust relationship:

- Who is being identified?
- What key is being trusted?
- What system is validating that trust?

**How AAA and certificates relate**

AAA controls access decisions, while certificates help establish trust in identity. They solve different parts of the security problem.

- AAA decides whether a user can get in and what they can do.
- Certificates help verify that a system or party is legitimate.
- Both support secure access, but they are not the same mechanism.

**Exam focus**

For Network+ N10-009, be ready to distinguish these concepts quickly:

1. Identify the AAA function from the scenario.
2. Match the protocol to its role in centralized access control.
3. Separate authentication from authorization and accounting.
4. Recognize that certificates support identity and trust, not user privilege assignment.

**Quick memory aid**

- **Authentication** = identity
- **Authorization** = permissions
- **Accounting** = logging
- **Certificate** = trusted identity binding
