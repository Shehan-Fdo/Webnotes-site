## **Course:** CompTIA Network+ (Exam N10-009)

**Remote access and file transfer overview**

Remote access protocols let you connect to systems from another location, while file transfer protocols move data between hosts. For the Network+ exam, focus on which protocols are secure, which are not, and what kind of traffic they carry.

A practical way to study these protocols is to pair each service with its port and security behavior. Insecure protocols typically send data in plaintext, which makes credentials and content easier to intercept. Secure alternatives use encryption to protect the session.

- **FTP** uses port `21` and transfers files without encryption.
- **Telnet** uses port `23` and sends commands and credentials in plaintext.
- **SSH** uses port `22` and provides encrypted remote access.
- **SFTP** uses SSH for encrypted file transfer.
- Secure protocols reduce exposure to interception and man-in-the-middle attacks.

**FTP and SFTP**

File Transfer Protocol (FTP) is a classic file transfer service, but it does not encrypt the session. That means usernames, passwords, and file contents can be exposed if traffic is captured. For exam purposes, treat FTP as an insecure protocol.

Secure File Transfer Protocol (SFTP) is different from FTP even though the names are similar. SFTP uses SSH to encrypt the connection, so it protects credentials and transferred data.

| Protocol | Typical Port | Security Profile | Primary Use |
|---|---:|---|---|
| FTP | `21` | Insecure | File transfer |
| SFTP | `22` | Secure | Encrypted file transfer |

> Use caution with port-based questions: `21` is associated with FTP, while `22` is associated with SSH and SFTP.

**Telnet and SSH**

Telnet is a remote access protocol that transmits data in plaintext. Because it lacks encryption, it is vulnerable to interception and should be recognized as insecure.

SSH, or Secure Shell, encrypts the connection between the client and server. It is the secure replacement you should choose when remote command-line access is required.

- **Telnet** is insecure remote access.
- **SSH** is encrypted remote access.
- SSH is also the transport used by SFTP.
- Plaintext sessions expose commands and credentials.

**DHCP in network access setup**

Dynamic Host Configuration Protocol (DHCP) automates IP address assignment and other network settings. It helps devices join a network without manual configuration.

DHCP follows a simple exchange pattern:

1. A device sends a discovery message to find DHCP servers.
2. A server responds with an offer that includes an available IP address and configuration details.
3. The device requests the offered configuration from the chosen server.
4. The server acknowledges the lease and settings.

DHCP commonly provides:
- IP address
- Subnet mask
- Default gateway
- DNS server addresses
- Lease duration

This protocol is not a remote access tool, but it is part of the broader network services you should recognize when devices connect and begin communicating.

**Security comparison mindset**

When you see a protocol question, ask whether the service protects data in transit. If it does not encrypt traffic, assume the credentials and payload may be visible to attackers.

- **Insecure examples** include FTP and Telnet.
- **Secure examples** include SSH and SFTP.
- Encryption helps protect against interception.
- Older or weaker protocol versions can increase risk if they remain enabled.

**Exam focus reminders**

For quick recall, connect the protocol to its function and security posture:

- FTP: file transfer, plaintext, port `21`
- SFTP: secure file transfer, SSH-based, port `22`
- Telnet: remote access, plaintext, port `23`
- SSH: secure remote access, encrypted, port `22`
- DHCP: automatic IP configuration

A strong exam answer usually comes from matching the protocol name to the right use case and identifying whether the traffic is secure or insecure.
