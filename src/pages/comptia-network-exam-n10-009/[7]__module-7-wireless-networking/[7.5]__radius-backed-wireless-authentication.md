## **Course:** CompTIA Network+ (Exam N10-009)

**Wireless authentication overview**

Wireless enterprise authentication uses a centralized server to verify a user or device before the access point grants network access. In 802.1X-based designs, the wireless client is the **supplicant**, the access point is the **authenticator**, and the authentication server makes the access decision.

RADIUS, which stands for **Remote Authentication Dial-In User Service**, is commonly used as the authentication server protocol in enterprise Wi-Fi. The access point forwards authentication traffic to the RADIUS server, which evaluates the request and tells the access point whether to allow or deny access.

- **Supplicant**: the client device requesting access.
- **Authenticator**: the wireless access point enforcing the decision.
- **Authentication server**: the trusted server that validates credentials and policy.

**Why enterprise Wi-Fi uses RADIUS**

Enterprise Wi-Fi avoids the weaknesses of a single shared pre-shared key (PSK). A PSK is the same for every user, is difficult to revoke for one person, and can be exposed through sniffing and offline attacks. RADIUS-backed authentication gives you per-user or per-device control instead of one shared network secret.

This model is a better fit for business networks because it supports centralized access control and stronger identity checks. It also aligns with the broader authentication workflow used in enterprise environments, where a server validates identity before access is granted.

- PSK networks use one shared key for everyone.
- Enterprise authentication can revoke access for one user without changing the key for all users.
- Centralized authentication improves control over who joins the network.

**802.1X and EAP**

IEEE 802.1X is the port-based access control framework used for wired and wireless networks. For Wi-Fi, it works with the **Extensible Authentication Protocol (EAP)** to carry authentication methods between the client and the server.

The access point does not decide trust on its own. Instead, it relays the authentication exchange to the server, which processes the request and returns the result. This separation is what makes enterprise wireless authentication scalable and manageable.

- **802.1X** defines the access-control framework.
- **EAP** carries the authentication method.
- **RADIUS** commonly transports the authentication exchange to the server.

**Certificate-based authentication with EAP-TLS**

A common enterprise approach is **EAP-TLS** (Extensible Authentication Protocol-Transport Layer Security), which uses certificates instead of passwords. In this model, the server and device complete a mutual TLS handshake before the access point allows the connection.

Certificate-based Wi-Fi is especially useful in security-sensitive environments because it relies on cryptography rather than shared credentials. It is more resistant to phishing, brute force attacks, dictionary attacks, and credential spraying than password-based methods.

- Use a **Certificate Authority (CA)** to issue certificates.
- Use a **RADIUS server** to validate the device during the handshake.
- Use a process for distributing the CA certificate and enrolling clients, often through **Mobile Device Management (MDM)**.

> In EAP-TLS deployments, the RADIUS server must complete mutual TLS with the device before it approves access.

**Typical deployment pieces**

A practical enterprise Wi-Fi deployment generally needs four things:

1. A Certificate Authority.
2. A RADIUS server.
3. A properly configured access point.
4. A way to distribute the CA certificate and enroll client devices.

This setup lets the access point delegate authentication to the RADIUS service while the client proves identity with a certificate. Some platforms provide the CA, RADIUS, and MDM integrations together to simplify rollout.

**Troubleshooting and operational notes**

When you debug enterprise Wi-Fi authentication, focus on the authentication path rather than the radio link alone. If the client cannot authenticate, the issue may be in certificate enrollment, RADIUS reachability, or access point configuration.

- Verify that the client has the correct certificate.
- Confirm that the access point is configured for 802.1X or WPA-Enterprise.
- Check that the RADIUS server is reachable and responding.
- Use verbose debugging tools when available; for FreeRADIUS, `radiusd -X` is recommended for debugging.

**Exam focus**

For Network+ purposes, remember the relationship between the components and the reason enterprise Wi-Fi uses them:

- **802.1X** provides port-based access control.
- **RADIUS** centralizes authentication decisions.
- **EAP-TLS** uses certificates for stronger wireless authentication.
- **PSK** is simpler but weaker for enterprise use.
