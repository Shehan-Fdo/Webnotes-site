## **Course:** CompTIA Network+ (Exam N10-009)

**Version control fundamentals**

Version control tracks the history of changes so you can recover earlier versions, review what changed, and understand who made the changes and when. It helps you work with files such as scripts, notes, and configuration data without creating a long chain of manually named copies.

Git is a **distributed version control system** (DVCS), which means each developer has a full copy of the project and its history. That design supports collaboration without requiring a constant connection to a central repository.

- Version control makes change history easier to inspect and compare.
- Git supports independent work while keeping a shared project history.
- GitHub adds an online platform for storing and sharing repositories.

**Git and GitHub roles**

Git handles the local version control workflow. GitHub provides a hosted place to back up work, collaborate, and manage repositories online. In practice, you use Git to record and organize changes, then use GitHub to share those changes with others or keep an accessible remote copy.

A practical benefit of this model is that you can keep an online backup of your work while also preserving a clear history of edits. This is especially useful when multiple people contribute to the same project.

- **Git** tracks and manages versions locally.
- **GitHub** hosts repositories and supports collaboration.
- A remote copy gives you backup and sharing benefits.

**What version history lets you answer**

A version control system gives you a transparent record of project activity. That record helps you answer common review questions during development or troubleshooting.

- Which changes were made?
- Who made the changes?
- When were the changes made?
- Why were the changes needed?

This history is valuable because it keeps contributors aligned while they work independently. You can inspect the project state at different points instead of relying on file names or memory.

**Common Git workflow areas**

Git covers the core tasks you expect in a version control workflow, including getting a repository, recording changes, viewing commit history, undoing changes, working with remotes, and tagging releases. It also supports branching, merging, rebasing, and other tools for managing parallel work.

- Get a repository and start tracking files.
- Record changes as commits.
- Review commit history to understand progress.
- Work with remotes to sync local and hosted copies.
- Use branches to isolate work before merging.

**Branching and collaboration**

Branches let you separate work so you can develop features or changes without immediately affecting the main line of development. Git supports basic branching and merging, branch management, remote branches, and rebasing.

In collaborative environments, branching helps you test changes and resolve issues before combining work. That makes it easier to keep the main project stable while work continues elsewhere.

- Branches isolate work from the main line.
- Merging combines completed changes.
- Rebasing is another way to integrate work history.

**GitHub in team workflows**

GitHub supports account setup, contributing to projects, maintaining projects, managing organizations, and scripting GitHub tasks. It is commonly used for both open source and commercial software development, and it fits well into distributed workflows where contributors do not need constant central access.

GitHub also appears in network automation communities as a place to share tools, examples, and structured data projects. That makes it useful beyond traditional software code.

- Use GitHub to contribute to shared projects.
- Use GitHub to maintain repositories and organizations.
- Use GitHub to support automation and collaboration work.

**Practical setup and access**

Git can be installed and used from the command line, and many workflows also integrate with other tools. First-time setup, help access, and server-side options are part of the broader Git ecosystem.

Common server and hosting topics include protocols, SSH public key generation, Git daemon, smart HTTP, GitWeb, GitLab, and third-party hosted options. These features support different ways of publishing and accessing repositories.

```bash
git status
git add .
git commit -m "Save changes"
git push
```

- `git status` shows the current working state.
- `git add` stages changes for commit.
- `git commit` records a snapshot with a message.
- `git push` sends local commits to a remote repository.

**Study focus for exam prep**

For exam prep, focus on the relationship between local version control and hosted collaboration. You should be able to explain why distributed version control matters, what Git tracks, and how GitHub extends Git with sharing and team features.

- Define version control clearly.
- Distinguish Git from GitHub.
- Explain why distributed history helps collaboration.
- Recognize branching, merging, remotes, and commits as core workflow concepts.
