## **Course:** CompTIA Network+ (Exam N10-009)

**Overview**

Command-line tools give you fast, direct visibility into network behavior. They are useful when you need to confirm whether a host is reachable, identify where a path slows down, or inspect local addressing and name resolution.

These tools are commonly run from a CLI (Command-Line Interface) such as Windows Command Prompt, Linux terminal, or macOS Terminal. In troubleshooting, you use them to narrow a problem before you change settings or replace hardware.

**Core troubleshooting approach**

A practical troubleshooting flow starts with identifying the problem, forming a theory, testing it, and then verifying the result. Command-line tools fit into that process because they let you test one assumption at a time.

- **Identify** the symptom and the affected scope.
- **Test** a likely cause with a focused command.
- **Verify** whether the result matches your expectation.
- **Document** what you changed and what you observed.

> Use a methodical sequence instead of running random commands. That makes it easier to isolate the fault and explain your findings.

**Ping**

`ping` is one of the first tools you use when checking connectivity. It sends Internet Control Message Protocol (ICMP) echo requests to a target and waits for replies. If the host responds, you know the destination is reachable at that moment.

`ping` also gives you round-trip time, which helps you spot delay or instability. If the command fails, the result can point you toward a local issue, a routing problem, or a remote host that is down.

- Test basic reachability to a host or IP address.
- Compare response time across multiple attempts.
- Use it early in the troubleshooting process.

```bash
ping 1.1.1.1
ping 9.9.9.9
```

**Traceroute and path testing**

`traceroute` shows the path packets take from source to destination. It helps you identify where traffic slows down or stops along the route. This is useful when a host is unreachable but local connectivity appears normal.

On some systems, the command name differs, but the purpose stays the same: reveal the hop-by-hop path and highlight where latency or failure begins.

- Trace the route to a remote destination.
- Look for a hop where delay increases sharply.
- Compare the path to a known-good destination.

```bash
traceroute example.com
```

**PathPing**

`PathPing` combines ideas from `ping` and `traceroute`. It helps you see both the route and the quality of each hop over time. That makes it useful when you suspect intermittent loss or a bottleneck somewhere in the path.

- Use it when a simple `ping` does not explain the problem.
- Use it when you need both path visibility and loss information.
- Use it to compare hop behavior over a longer test window.

```bash
pathping example.com
```

**Name resolution tools**

`nslookup` and `dig` help you test Domain Name System (DNS) resolution. If a user can reach an IP address but not a hostname, these tools help you confirm whether the name resolves correctly.

These commands are especially useful when the network is up but applications still fail because they cannot translate names into addresses.

- Check whether a hostname resolves to the expected address.
- Compare results from different DNS servers.
- Confirm whether the issue is DNS-related rather than general connectivity.

```bash
nslookup example.com
dig example.com
```

**Local interface and address inspection**

Tools such as `ipconfig` and `ifconfig` show local interface settings. They help you confirm the device’s IP address, subnet mask, default gateway, and other basic network parameters.

Use them when you need to verify whether the client received the correct configuration or whether the interface is up with the expected address.

- Confirm the assigned IP address.
- Check the default gateway.
- Verify whether the interface has the expected configuration.

```bash
ipconfig
ifconfig
```

**Connection and socket visibility**

`netstat` helps you inspect active connections and listening ports. It is useful when you want to see whether a service is listening locally or whether a connection is established to a remote system.

This makes it helpful for distinguishing between a network problem and an application problem.

- Review active sessions.
- Check listening ports.
- Look for unexpected or missing connections.

```bash
netstat
```

**Packet capture and deeper inspection**

`tcpdump` captures packets at the command line. It is useful when you need to see actual traffic rather than just the result of a higher-level test. This can help you confirm whether packets leave the host, whether replies return, and whether the traffic matches your expectation.

Because packet capture is more detailed than basic reachability tests, use it when simpler commands do not isolate the issue.

- Capture traffic for a specific host or protocol.
- Confirm whether requests and replies both appear.
- Use it to validate what the network is actually carrying.

```bash
tcpdump -i eth0
```

**How to choose the right tool**

Pick the simplest command that answers your current question. Start with reachability, then move to path, name resolution, interface state, and packet-level inspection as needed.

| Question | Useful tool | What it tells you |
|---|---|---|
| Is the host reachable? | `ping` | Whether the target responds to ICMP |
| Where does traffic fail? | `traceroute` | The hop where delay or failure appears |
| Is DNS working? | `nslookup`, `dig` | Whether a name resolves correctly |
| Is the local interface configured correctly? | `ipconfig`, `ifconfig` | Addressing and gateway details |
| What connections are active? | `netstat` | Open ports and established sessions |
| What packets are actually moving? | `tcpdump` | Packet-level traffic evidence |

**Practical troubleshooting sequence**

1. Run `ping` to confirm basic reachability.
2. Use `traceroute` or `pathping` if the destination is not responding as expected.
3. Check `nslookup` or `dig` if the problem involves a hostname.
4. Inspect local settings with `ipconfig` or `ifconfig`.
5. Review active connections with `netstat`.
6. Capture packets with `tcpdump` if you need deeper evidence.

This sequence keeps your investigation focused and reduces guesswork. It also helps you separate network-layer issues from DNS, host configuration, or application behavior.
