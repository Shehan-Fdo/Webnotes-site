## **Course:** CompTIA Network+ (Exam N10-009)

**Overview**

Connectivity issues often show up as devices that cannot join the network, slow performance, dropped packets, or intermittent access. The key is to match the symptom to the likely layer or service involved, then test the simplest causes first.

A practical troubleshooting approach starts by identifying the problem, forming a theory, testing it, and then verifying the fix. This keeps you focused on observable symptoms instead of guessing.

**DHCP and IP Address Assignment**

<abbr title="Dynamic Host Configuration Protocol">DHCP</abbr> automatically assigns IP configuration to clients. It typically provides an IP address, subnet mask, default gateway, and sometimes DNS information. If DHCP fails, a device may not receive the settings it needs to communicate on the network.

Common DHCP-related problems include:

- **No lease**: The client does not obtain an IP address from the DHCP server.
- **Wrong subnet**: A device receives an address that does not match the local network.
- **Duplicate assignment**: Two devices end up with the same IP address.
- **Rogue server**: An unauthorized DHCP server hands out incorrect settings.

> A duplicate IP address can cause connectivity loss for the affected devices and may generate error messages or event log entries.

When you troubleshoot DHCP, check whether the server is reachable, whether the scope has available addresses, and whether the client is requesting the correct options. On managed switches, DHCP snooping can help identify unauthorized DHCP activity.

**Duplicate IP Addresses**

An IP address conflict occurs when two or more devices use the same address. This can happen because of manual misconfiguration, a bad DHCP scope, or a rogue DHCP server.

Symptoms often include:

- Intermittent connectivity
- Failed logons or application access
- Error messages about address conflicts
- Event log alerts on network devices

If the conflict is on a DHCP-managed network, review the lease assignment and confirm that the server is not distributing overlapping addresses.

**Physical Layer and Cabling Issues**

Physical media problems can look like general connectivity failures, but they often stem from signal degradation or cable faults. Two common causes are attenuation and crosstalk.

- **Attenuation** is signal loss over distance.
- **Crosstalk** happens when one cable interferes with another, especially when cables run in parallel or when wire pairs are crossed or crushed.
- **Near-end crosstalk** often appears close to the transmitting end, commonly near the connector.

These issues can produce slow performance, dropped packets, or unintelligible traffic. If you suspect cabling, test the cable from both ends and inspect connectors, bends, and physical damage.

**Collisions and Duplex Problems**

Collisions occur when two devices transmit at the same time on a half-duplex Ethernet segment. Legacy hubs and half-duplex configurations can still create this behavior.

- Collisions are normal on half-duplex Ethernet.
- Excessive collisions can indicate a duplex mismatch or an overloaded shared segment.
- Devices typically detect the collision, wait a random time, and retransmit.

If you see repeated collision-related symptoms on modern switched networks, verify duplex settings and check for legacy hardware in the path.

**Broadcast and Traffic-Related Symptoms**

Some connectivity issues are not caused by a single bad device but by excessive traffic on the network. Broadcast storms and similar conditions can overwhelm devices and reduce usable bandwidth.

Typical signs include:

- Slow network performance
- Dropped packets
- Unresponsive hosts
- Widespread access problems rather than a single-device failure

When the issue affects many devices at once, look for loops, misbehaving network equipment, or abnormal traffic patterns.

**Troubleshooting Workflow**

Use a structured process so you can isolate the cause efficiently.

1. Identify the problem and define the symptoms.
2. Establish a theory of probable cause.
3. Test the theory.
4. Plan and implement the fix.
5. Verify full functionality.
6. Document the findings.

This workflow helps you separate DHCP issues, addressing conflicts, and physical-layer faults from broader network outages.

**Quick Symptom-to-Cause Mapping**

| Symptom | Likely Cause | What to Check |
|---|---|---|
| Device has no IP address | DHCP failure | Server reachability, scope availability, client options |
| Two devices lose access after address assignment | Duplicate IP address | DHCP leases, static addressing, event logs |
| Slow or dropped traffic | Attenuation or crosstalk | Cable length, damage, termination quality |
| Repeated retransmissions on a shared segment | Collisions | Duplex mode, hubs, legacy devices |
| Many devices affected at once | Broadcast storm or traffic overload | Loops, abnormal traffic, switching behavior |

**Practical Troubleshooting Notes**

Start with the most visible and least disruptive checks. If a single host fails, focus on its address assignment and local cabling. If multiple hosts fail, look for shared infrastructure problems such as DHCP scope issues, rogue servers, or a bad switch segment.

Use logs when available. DHCP event logs and switch alerts can confirm whether the issue is an address conflict, a lease problem, or an unauthorized server.
