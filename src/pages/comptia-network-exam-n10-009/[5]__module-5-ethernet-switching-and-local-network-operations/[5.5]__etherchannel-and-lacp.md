## **Course:** CompTIA Network+ (Exam N10-009)

**EtherChannel and link aggregation overview**

EtherChannel groups multiple physical Ethernet links into one logical connection. This approach increases available bandwidth, improves redundancy, and lets network devices treat the bundle as a single link.

Link aggregation is also called by several related names, including:

- Ethernet bonding
- Ethernet teaming
- Link bonding
- Link bundling
- Link teaming
- NIC bonding
- NIC teaming
- Port aggregation
- Port channeling
- Port trunking

A link aggregation group (LAG) can be used between devices such as switches, servers, network-attached storage (NAS), and multi-port access points. Network management functions can treat the LAG as one link, and you can include it in a virtual local area network (VLAN).

**LACP fundamentals**

Link Aggregation Control Protocol (<abbr title="Link Aggregation Control Protocol">LACP</abbr>) is an IEEE standard used to negotiate link aggregation automatically. It is part of the IEEE 802.3ad specification and is commonly associated with EtherChannel-style bundles.

LACP helps you:

- Combine multiple physical links into one logical link
- Share traffic across active links for load balancing
- Fail over automatically if a link in the group fails
- Reduce setup errors by negotiating the bundle between devices

A common implementation limit is up to 16 Ethernet links in one aggregation group, with up to 8 active links and the remaining links kept on standby.

**LACP modes**

LACP uses two negotiation modes:

- **Active**: the interface actively sends LACP packets to start negotiation
- **Passive**: the interface responds to LACP negotiation but does not initiate it

For a successful LACP connection, at least one side must be in active mode. If both sides are passive, the bundle does not form.

| Mode pair | Link aggregation works |
|---|---:|
| Active + Active | Yes |
| Active + Passive | Yes |
| Passive + Passive | No |

**Configuration and compatibility checks**

Before you expect LACP to work, verify the basic requirements on both ends of the link. The bundle depends on matching configuration and supported negotiation behavior.

1. Confirm that at least one interface is set to `active`.
2. Verify that the connected devices support LACP.
3. Check that the member links are the same type.
4. Confirm that the aggregation group size stays within the device limit.

> If the bundle does not form, start by checking the negotiation mode on both sides. A passive-only pair will not establish LACP.

**Operational benefits**

EtherChannel and LACP are useful when you want more capacity without presenting multiple separate links to the network. They also improve resilience because traffic can continue over remaining links when one member fails.

Key practical outcomes include:

- Higher effective throughput across the bundle
- Better fault tolerance than a single physical link
- Simplified management because the group behaves like one logical interface
- Automatic recovery behavior when a member link drops

**Troubleshooting focus**

When you troubleshoot link aggregation, keep the problem scope narrow and verify the bundle from the physical layer upward. Start with the member links, then confirm the negotiation state, and finally check whether the logical group appears as expected.

Useful checks include:

- Link status on each physical interface
- LACP mode on both ends
- Whether the device has reached its active-link limit
- Whether the devices are treating the bundle as one logical link

**Exam takeaway**

For Network+ purposes, remember the relationship between the terms:

- **EtherChannel**: the logical bundle concept
- **LAG**: the grouped links acting as one connection
- **LACP**: the protocol that negotiates and maintains the bundle

The core idea is simple: you combine multiple Ethernet links to increase bandwidth and resilience, and LACP helps automate that process.
