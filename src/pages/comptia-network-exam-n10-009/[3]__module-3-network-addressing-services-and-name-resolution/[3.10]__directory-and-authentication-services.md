## **Course:** CompTIA Network+ (Exam N10-009)

**Overview**

Directory and authentication services centralize user access so you can control who logs in and what they can do. These services commonly work with the <abbr title="Authentication, Authorization, and Accounting">AAA</abbr> framework, which separates identity verification from privilege assignment and activity tracking.

A directory or authentication server is typically external to the device being accessed. The device sends user account information to the server, and the server returns the result of the authentication request along with any privilege information needed to define access.

- **Authentication** verifies identity before access is granted.
- **Authorization** determines what the user can do after login.
- **Accounting** records activity for tracking or auditing.

**AAA and centralized access**

AAA is useful because it reduces the need to manage every account locally on every device. Instead, you can use a centralized service to control access across multiple systems. This approach also makes it easier to apply consistent login policies and privilege levels.

In practice, the device acts as a client to the authentication service. The server validates the user and can return privilege details that affect what the user is allowed to do, such as restricting file transfer access.

- Centralized services simplify account management.
- Returned privilege data can limit specific actions on a device.
- Local device accounts can remain available as a backup if the external service fails.

**RADIUS**

<abbr title="Remote Authentication Dial-In User Service">RADIUS</abbr> is an access server AAA protocol. It provides a standardized way to exchange authentication information between a client on the device and an external RADIUS server.

RADIUS adds an extra layer of login security by offloading authentication to the server. The client relays user account information, the server authenticates the user, and the server returns privilege information that defines device access.

- **RADIUS client** runs on or is associated with the device being managed.
- **RADIUS server** is external to the device and performs authentication.
- Returned privileges can restrict actions such as FTP access.

**TACACS+**

<abbr title="Terminal Access Controller Access-Control System Plus">TACACS+</abbr> provides functions similar to RADIUS. It is also used for centralized authentication and access control.

One practical detail is that some systems check reachability to a TACACS+ server using UDP port `49` to avoid long timeout issues, while subsequent communication uses TCP port `49`.

> Use conservative troubleshooting: verify server reachability first, then confirm the expected transport and port behavior for the platform you are managing.

**LDAP**

<abbr title="Lightweight Directory Access Protocol">LDAP</abbr> is another directory-based authentication option that provides similar functions to RADIUS in the context of centralized login services. It is commonly used when you want directory-backed identity information to support authentication and access decisions.

LDAP is part of the same general family of centralized access services, so the key exam idea is not the vendor implementation but the role it plays: it helps validate users and support controlled access to network resources.

**Backup local accounts**

You can create local device user accounts as a backup to RADIUS, TACACS+, or LDAP authentication. This gives you a fallback path if the external authentication service is unreachable or fails.

This is important operationally because centralized authentication improves control, but it also introduces a dependency on the external service. A local account can preserve administrative access during outages.

- Keep backup accounts available for recovery.
- Use them when external authentication is unavailable.
- Treat them as a fallback, not the primary access method.

**Practical comparison**

| Service | Primary role | Notable detail |
|---|---|---|
| RADIUS | Centralized AAA access server protocol | Returns privilege information that can limit device actions |
| TACACS+ | Centralized authentication and access control | Reachability may be checked with UDP `49`; later traffic uses TCP `49` |
| LDAP | Directory-based authentication service | Supports centralized identity and access decisions |

**Key takeaways**

Directory and authentication services help you manage access from a central point instead of relying only on local device accounts. For Network+ purposes, focus on the shared AAA model, the role of RADIUS, TACACS+, and LDAP, and the value of local fallback accounts when external authentication fails.
