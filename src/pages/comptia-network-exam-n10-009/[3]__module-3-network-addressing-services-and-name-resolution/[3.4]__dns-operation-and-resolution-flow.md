## **Course:** CompTIA Network+ (Exam N10-009)

**DNS resolution overview**

DNS, or <abbr title="Domain Name System">DNS</abbr>, translates a human-readable name such as `google.com` into an IP address that devices can use for communication. The lookup process is usually fast because clients check local caches before asking external servers.

A typical resolution flow moves from the client outward:

- Browser cache is checked first.
- OS cache is checked next.
- A recursive resolver handles the lookup if the answer is not already cached.
- Root nameservers direct the query toward the correct top-level domain.
- TLD nameservers point to the domain’s authoritative nameserver.
- The authoritative nameserver returns the final IP address.

> If caching already contains the answer, the lookup may stop at the browser or OS level instead of completing the full chain.

**Recursive resolution and stub resolvers**

Your device usually does not perform the entire DNS search by itself. Instead, the operating system uses a stub resolver to send the query to a recursive DNS resolver, which may belong to your ISP or a public DNS service.

The recursive resolver either returns a cached response or performs the full lookup on your behalf. This design reduces client complexity and improves performance through shared caching.

- **Stub resolver**: The client-side component that sends DNS queries.
- **Recursive resolver**: The server that performs the lookup chain and returns the answer.
- **Caching**: A stored response that avoids repeating the full lookup.

**DNS hierarchy**

DNS is organized as a hierarchical database. That structure supports distributed administration and scalable name resolution.

- The root sits at the top of the hierarchy and is represented by `.`.
- Top-level domains, such as `.com`, `.org`, and `.net`, sit below the root.
- Authoritative nameservers hold the final records for a domain.
- Each level helps narrow the search until the correct host record is found.

**Common record types**

DNS uses record types to store different kinds of name-to-data mappings. The evidence specifically highlights the following record types:

| Record type | Purpose |
|---|---|
| `A` | Maps a domain name to an IPv4 address |
| `CNAME` | Points one name to another name |
| `MX` | Identifies mail exchange servers |
| `NS` | Identifies nameservers for a zone |

These records support both web access and broader network services. For example, an application may use a fully qualified domain name instead of a raw IP address, which makes DNS availability critical to the application’s operation.

**Troubleshooting DNS-related symptoms**

When DNS fails, the network may still be reachable even though name-based access does not work. That distinction helps you separate name resolution problems from general connectivity problems.

Common symptoms include:

- You can ping an IP address, but not the same host by fully qualified domain name.
- Web browsing appears broken even though the server is reachable by IP.
- Applications that rely on names fail while basic network connectivity still works.

A practical check is to test by IP address first. If IP connectivity works, focus on DNS resolution rather than physical or routing issues.

**Resolution flow summary**

1. You enter a domain name in the browser.
2. The browser checks its cache.
3. The operating system checks its cache.
4. The stub resolver sends the query to a recursive resolver.
5. The recursive resolver checks its cache or queries the DNS hierarchy.
6. Root and TLD servers direct the query to the authoritative nameserver.
7. The authoritative nameserver returns the final IP address.
8. The client uses that IP address to reach the destination.

**Key takeaways**

DNS converts names into addresses, and caching makes the process fast in most cases. When troubleshooting, separate name resolution failures from general network failures by testing direct IP connectivity. Understanding the hierarchy and record types helps you trace where the lookup succeeds or breaks.
