## **Course:** CompTIA Network+ (Exam N10-009)

**DHCP fundamentals**

<abbr title="Dynamic Host Configuration Protocol">DHCP</abbr> is a standard protocol used to dynamically distribute IP addressing and configuration information to clients. It reduces manual configuration by letting a server provide network settings automatically when devices join the network.

A DHCP server commonly provides these core settings:

- **IP address** for the client
- **Subnet mask** to define the local network range
- **Default gateway** for traffic outside the subnet
- **DNS** server addresses for name resolution
- Other options, such as <abbr title="Windows Internet Name Service">WINS</abbr> server addresses, when configured

The system administrator configures the DHCP server with the options that are handed out to clients. In practice, this makes DHCP useful for environments with many devices that need consistent configuration.

**Client-server operation**

DHCP uses a client-server model. The client requests network settings, and the server allocates them from an address pool. The assigned settings are delivered as options for a lease time, which means the configuration is temporary and can be renewed.

Typical DHCP behavior includes:

- A client requests parameters when it connects to the network
- The server responds with available configuration from its pool
- The client receives an address and related settings for a lease period
- The client may request updates periodically as the lease approaches expiration

This process removes the need to manually configure each device one by one and helps keep addressing consistent across the network.

**DHCP relay**

A DHCP relay forwards DHCP requests and replies between clients and the server when they are on different networks, such as different subnets. This is important because DHCP broadcasts do not normally cross broadcast domain boundaries on their own.

Use relay when:

- The DHCP server is not on the same subnet as the client
- You need clients in one broadcast domain to obtain addresses from a server in another
- You want centralized DHCP management without placing a server on every subnet

> DHCP relay is the practical solution when clients cannot reach the DHCP server directly because of subnet separation.

**Troubleshooting DHCP-related issues**

When DHCP is not working, start by confirming the client has the correct IP configuration and that the network path to the DHCP service is available. Problems with IP configuration can cause limited connectivity, no connectivity, or inability to reach resources outside the local subnet.

Useful checks include:

- Verify the client has the expected IP address, subnet mask, and default gateway
- Confirm the client can communicate on the local network
- Check whether the DHCP server is reachable directly or through relay
- Review whether the client is receiving the correct DNS settings
- Look for symptoms that suggest the client did not receive a valid lease

If a device can reach local addresses but not destinations outside its subnet, the default gateway or DHCP-provided settings may be incorrect. If name-based access fails while IP-based access works, the DNS settings delivered by DHCP may be the issue.

**Key takeaways**

DHCP automates IP configuration and is central to efficient network administration. It supplies the basic addressing information clients need, and relay extends that service across subnets so centralized DHCP can support multiple broadcast domains.

- **Automation** reduces manual IP configuration
- **Lease-based assignment** lets clients receive and renew settings
- **Relay** enables DHCP across different subnets
- **Core options** usually include IP address, subnet mask, default gateway, and DNS
