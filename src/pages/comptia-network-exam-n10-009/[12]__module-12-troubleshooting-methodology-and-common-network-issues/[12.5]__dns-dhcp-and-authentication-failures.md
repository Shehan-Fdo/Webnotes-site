## **Course:** CompTIA Network+ (Exam N10-009)

**Overview**

DNS (Domain Name System), DHCP (Dynamic Host Configuration Protocol), and authentication failures often appear together because each one can block access in a different way. DHCP assigns addressing information, DNS resolves names to addresses, and authentication confirms that a user or device is allowed to connect. When one layer fails, the symptom may look like a general network outage even though the root cause is narrower.

A practical troubleshooting approach is to identify the problem, test likely theories, verify the result, and then document what changed. That sequence helps you avoid guessing when multiple services could be involved.

**DHCP failure symptoms and checks**

DHCP automatically distributes IP addressing and configuration information to clients. At minimum, a DHCP server usually provides an IP address, subnet mask, and default gateway, and it may also provide DNS-related settings.

Common DHCP problems include:

- **No lease available** when the scope has no free addresses.
- **Service stopped** when the DHCP server service is not running.
- **Authorization issue** when a domain-joined DHCP server is not authorized.
- **BAD_ADDRESS entries** when the lease table contains problematic records.
- **Client-side mismatch** when the client is on the wrong subnet or cannot reach the server.

On the server side, verify the service and scope health first:

1. Confirm that the DHCP Server service is started and running.
2. Check that the DHCP server is authorized if the environment requires it.
3. Verify that leases are available in the scope for the client subnet.
4. Review `BAD_ADDRESS` listings in address leases.
5. Check for other devices or conditions that may interfere with address assignment.

> If clients cannot obtain an address, fix DHCP before moving to DNS or authentication. Without a valid IP configuration, later checks are unreliable.

**DNS failure symptoms and checks**

DNS translates host names into IP addresses. When DNS fails, users may still have network connectivity but cannot reach services by name. That often looks like a web, mail, or login problem even though the underlying issue is name resolution.

Watch for these signs:

- Users can reach an IP address but not the hostname.
- Applications fail when they rely on name lookup.
- Old or incorrect records remain in use after address changes.
- Missing records or misconfigured zones cause resolution errors.

Useful troubleshooting focus areas include:

- Confirm the client has the correct DNS server settings from DHCP or manual configuration.
- Check whether the expected record exists and points to the correct address.
- Look for stale or orphaned records after devices receive new IP addresses.
- Validate configuration before reloading DNS changes when your process includes that step.

DNS problems often follow DHCP issues because incorrect DHCP options can hand out the wrong DNS server or incomplete configuration.

**Authentication failures**

Authentication failures happen when a user or device cannot prove identity to the service that controls access. In network troubleshooting, these failures may appear after addressing and name resolution are already working, so you should not assume the network is down just because a login fails.

Typical indicators include:

- A device gets an IP address but still cannot access protected resources.
- Name resolution works, but sign-in or access control fails.
- The failure is limited to one service, account, or network segment.

When you investigate authentication, confirm the basics first:

- The client has a valid IP address and can reach the network.
- DNS resolves the target service correctly.
- The user or device is attempting the correct authentication method for the environment.
- Access restrictions are intentional and not the result of a misconfiguration.

**Troubleshooting order**

Use a layered approach so you do not chase symptoms out of order.

1. Verify physical and link connectivity if the device cannot communicate at all.
2. Confirm DHCP assignment and check whether the client has a valid IP configuration.
3. Test DNS name resolution if IP connectivity exists but services fail by name.
4. Check authentication if the client reaches the network and resolves names but still cannot access resources.
5. Document the root cause and the fix.

This order matters because DHCP problems can prevent DNS from being configured correctly, and DNS problems can make authentication services appear unavailable even when they are reachable by IP.

**Quick comparison**

| Service | Primary role | Common failure effect |
|---|---|---|
| DHCP | Assigns IP configuration automatically | Client cannot join the network or gets unusable settings |
| DNS | Resolves names to IP addresses | Services fail by name even though IP connectivity may work |
| Authentication | Verifies identity and access | User or device reaches the network but cannot access protected resources |

**Exam focus**

For Network+ troubleshooting questions, pay attention to the symptom pattern:

- No IP address or bad lease behavior points to **DHCP**.
- IP works but names fail points to **DNS**.
- Network access exists but login or authorization fails points to **authentication**.
- Multiple failures can cascade, so start with the lowest layer that is broken.
