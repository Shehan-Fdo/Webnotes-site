## **Course:** HTML Essentials: Complete Guide (Beginner to Advanced)

**Overview**

- The `<dialog>` element represents a **dialog box** or other interactive component.
- Use it for modal or non-modal UI, such as a dismissible alert, inspector, or subwindow.
- It is widely available across browsers since March 2022.
- Some parts of the feature may still vary in support.

**Core behavior**

- A **modal** dialog blocks interaction with the page behind it.
- A non-modal dialog stays available without blocking the rest of the page.
- `showModal()` opens a dialog as modal.
- A modal dialog appears in the top layer.
- The page content outside the dialog becomes inert while the modal is open.
- If the dialog is inside an `iframe`, only the containing document is blocked.

**JavaScript methods**

- `showModal()` displays the dialog as a modal dialog.
- `showModal()` has no parameters.
- `showModal()` throws `InvalidStateError` if the dialog is already open and non-modal.
- The `show()` method is referenced by the API behavior for non-modal opening.
- The `closedBy` property reflects the `closedby` attribute value.

**Closing behavior**

- The `closedby` attribute controls how users can dismiss a dialog.
- `any` allows light dismiss, platform-specific close actions, and developer-specified mechanisms.
- `closerequest` allows platform-specific close actions and developer-specified mechanisms.
- `none` allows only developer-specified mechanisms.
- If `closedby` is missing or invalid, the dialog falls back to the **Auto** state.
- In Auto state, a dialog opened with `showModal()` behaves like `closedby="closerequest"`.
- In Auto state, a dialog opened by other means behaves like `closedby="none"`.

**Attributes and constraints**

- The `<dialog>` element supports global attributes.
- Do not use `tabindex` on `<dialog>`.
- `closedby` is not baseline across all widely used browsers.
- Browser support for some dialog features may vary.

**Accessibility and UX**

- Native dialog support helps reduce the work needed for accessible modal UI.
- You still need intentional accessibility and UX decisions.
- Use modal dialogs only when you need to block the current flow.
- Use non-modal dialogs when the user should keep interacting with the page.

**Practical usage**

- Treat `modal` as a state, not as a synonym for every dialog.
- Choose the dialog type based on whether the page behind it should remain active.
- Use a developer-specified close control when you need predictable dismissal.
- Prefer clear close behavior so users know how to exit the dialog.
