## **Course:** CompTIA Cybersecurity Analyst (Exam CS0-003)

**PowerShell in Windows-centric analysis**

PowerShell is a core Windows automation and analysis tool. In a security operations context, you use it to query event logs, support investigation workflows, and work with data that helps with threat detection, incident response, and digital forensics.

Windows event logs are especially valuable because they capture activity that can reveal brute-force attempts, suspicious logons, and other endpoint behavior. PowerShell gives you direct access to that data from local or remote systems.

- **PowerShell** is widely used for legitimate administration and can also be abused by threat actors.
- Windows event logs support threat hunting, incident response, and forensic review.
- Logging and parsing activity helps you turn raw endpoint events into investigation evidence.

**Event log access with PowerShell**

Microsoft provides cmdlets for event-log interrogation. `Get-EventLog` retrieves events or a list of event logs from local or remote computers. By default, it reads the local computer, and you use `-ComputerName` when you need remote access.

`Get-EventLog` supports filtering by log name and several useful parameters, including time, source, entry type, and message content. That makes it practical for narrowing large logs to a smaller set of events during analysis.

> `Get-EventLog` uses a deprecated Win32 API. Microsoft recommends `Get-WinEvent` for modern Windows Event Log features, though `Get-EventLog` may still work well for the Application, System, and Security logs.

- `Get-EventLog` can list logs with `-List`.
- `-After` and `-Before` help you bound searches by time.
- `-Source`, `-EntryType`, and `-Message` help you filter for relevant activity.
- `-ComputerName` supports remote log review.

```powershell
Get-EventLog -LogName Security -Newest 20
Get-EventLog -LogName Security -After (Get-Date).AddHours(-24) -EntryType FailureAudit
Get-EventLog -ComputerName server01 -LogName System -Source Service Control Manager
```

**Security log analysis workflow**

When you analyze Security Event Logs, you often start by confirming that auditing is enabled and then look for event IDs that indicate successful or failed authentication. One example is failed logon activity, which can help you identify brute-force attempts.

PowerShell can be used to parse these logs and support detection workflows. In practice, that means you collect the relevant events, filter them to the time window you care about, and then review the results for patterns that match suspicious behavior.

1. Confirm that security auditing is enabled on the endpoint.
2. Collect the relevant log data with PowerShell.
3. Filter for the event IDs or fields tied to the behavior you are investigating.
4. Review the output for repeated failures, unusual sources, or other anomalies.

- Event ID `4625` indicates a failed logon.
- Event ID `4624` indicates a successful logon.
- Group Policy can be used to enable logon failure auditing.
- Centralized logging or event forwarding can support broader analysis.

**Why this matters for CySA+**

CompTIA CySA+ focuses on incident detection, prevention, and response through continuous security monitoring. PowerShell fits that workflow because it helps you inspect Windows telemetry quickly and consistently.

For exam prep, focus on the practical relationship between the tool and the data:

- Use PowerShell to access and filter Windows event logs.
- Prefer `Get-WinEvent` for modern event-log work when possible.
- Use `Get-EventLog` when working with supported legacy scenarios or common logs.
- Treat PowerShell logs and event logs as investigation inputs, not just admin output.

**Key takeaways**

PowerShell is not just a scripting language; it is a Windows-native analysis interface. In security work, its value comes from fast access to event data, flexible filtering, and support for both local and remote review.

Keep these points in mind:

- Windows event logs are a high-value source for hunting and response.
- `Get-EventLog` retrieves event data from local or remote systems.
- `Get-WinEvent` is the preferred modern cmdlet.
- Security auditing and event IDs such as `4624` and `4625` are central to logon analysis.
