## **Course:** CompTIA Cybersecurity Analyst (Exam CS0-003)

**Session hijacking overview**

Session hijacking happens when an attacker steals or reuses an active session token or session identifier to impersonate a logged-in user. The token proves authentication to the web application, so the attacker does not need the original password.

Because the attack occurs after login, it can bypass controls that already succeeded during authentication, including <abbr title="Multi-Factor Authentication">MFA</abbr>. In practice, the attacker inherits the victim’s current permissions until the session expires or is revoked.

- **Bearer token** behavior matters because possession is enough to use it.
- **Session cookie** theft can let an attacker continue an authenticated browser session.
- **Post-authentication** compromise is often harder to spot than a failed login attempt.

**Common ways tokens are exposed**

Attackers usually target the places where tokens are stored, transmitted, or reused. The evidence highlights three common paths: malware, phishing, and web application flaws.

- Malware can copy browser cookies or other session data.
- Phishing sites can capture tokens after the user signs in.
- Web application vulnerabilities can expose session identifiers.

> A stolen token can be more valuable than a stolen password because it may already satisfy MFA and other login checks.

**Why suspicious token activity stands out**

Suspicious token activity often shows up as a valid session being used in a way that does not match normal user behavior. Since the attacker is reusing a legitimate token, the application may treat the request as trusted.

Look for patterns that suggest the token is being used by someone other than the rightful user:

- Access from an unexpected location or device.
- Repeated use of the same session in a way that does not match the user’s normal pattern.
- Activity that continues even though the user did not initiate it.
- Signs that a session remains active after the user should have logged out or reauthenticated.

**Detection and response focus**

You do not rely on password failure alone to detect this attack. Instead, you watch for abnormal use of an otherwise valid session and verify whether the token should still be trusted.

1. Confirm whether the session token is active and whether the user should still be authenticated.
2. Compare the session’s behavior with the user’s normal access pattern.
3. Revoke the token if the activity is suspicious.
4. Force reauthentication and investigate the source of exposure.

**Reducing risk**

The evidence points to a few practical controls that reduce the chance of token theft and reuse.

- Use **HTTPS** to protect tokens in transit.
- Secure cookies so session data is harder to steal or misuse.
- Keep session lifetimes short so stolen tokens expire sooner.
- Use continuous verification to re-check trust during the session.

**Exam takeaway**

Session hijacking is a post-login attack that reuses a stolen token to impersonate a legitimate user. The key exam clue is that MFA may already be satisfied, so the compromise appears as suspicious activity inside an otherwise valid session rather than a failed sign-in.
