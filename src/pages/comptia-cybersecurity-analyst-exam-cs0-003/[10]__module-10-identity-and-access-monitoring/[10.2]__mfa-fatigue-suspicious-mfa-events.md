## **Course:** CompTIA Cybersecurity Analyst (Exam CS0-003)

**MFA fatigue attack overview**

MFA fatigue, also called MFA bombing or MFA spamming, is a social engineering attack that targets the user rather than the authentication system. The attacker repeatedly sends approval prompts to a legitimate user’s device until the user accepts one out of annoyance, confusion, or exhaustion.

The attack usually starts after the attacker already has valid credentials. Those credentials often come from phishing, credential stuffing, or a prior data breach. MFA then becomes the last barrier, so the attacker tries to wear down the user instead of breaking the technology.

- **Social engineering** is the core tactic.
- The attacker relies on repeated prompts, not malware or cryptographic failure.
- Push-based MFA is especially exposed when users can approve a prompt with a single tap.

**How the attack works**

The attacker first attempts to sign in with stolen credentials. If MFA blocks the login, the attacker triggers repeated authentication requests to the victim’s registered device. The flood of prompts can continue for minutes or hours.

A successful attack often looks like this:

1. The attacker obtains a username and password.
2. The attacker starts repeated login attempts.
3. The victim receives a stream of MFA prompts.
4. The victim approves one prompt to stop the notifications.
5. The attacker gains access to the account or connected resource.

> If you see repeated MFA prompts followed by a sudden approval, treat it as suspicious until you confirm the user intended the action.

**Suspicious MFA event indicators**

Suspicious MFA activity often appears in authentication logs before a full compromise is obvious. You should look for patterns that show pressure, inconsistency, or a change in normal user behavior.

Common indicators include:

- Repeated denials followed by a single approval.
- A sudden burst of MFA prompts for one account.
- Successful logins from a new device.
- Logins from an unusual location.
- Multiple successful MFA logins for the same user in a short period.
- Prompt frequency that does not match the user’s normal behavior.

These signals matter because the attack succeeds through persistence. The authentication method may still be working as designed, but the user is being manipulated into approving access.

**Detection and triage**

When you investigate suspicious MFA events, focus on the sequence of authentication activity and whether it matches the user’s normal pattern. A single approval is less meaningful than the surrounding context.

Use the following approach:

1. Review the authentication log timeline for repeated prompts and denials.
2. Check whether the successful approval came from a new device or unusual location.
3. Confirm with the user whether they initiated the login.
4. Look for other signs of account misuse after the approval.
5. Contain access quickly if the event does not match expected behavior.

Useful log questions include:

- Did the user receive multiple prompts in a short time?
- Did the user deny several requests before one approval?
- Did the login originate from a device or location the user does not normally use?
- Did the account show multiple successful MFA logins that do not fit normal activity?

**Prevention and hardening**

MFA fatigue is harder to stop when the only control is a simple push approval. Stronger controls reduce the chance that a user can be pressured into approving access.

Practical defenses include:

- **Rate limiting** authentication prompts to reduce flooding.
- **Contextual prompts** that show useful details to the user.
- **Number matching** so the user must enter or match a code instead of tapping approve.
- **Risk-based authentication** that reacts to unusual sign-in conditions.
- **Phishing-resistant FIDO2** methods for stronger verification.
- User training that teaches people to deny unexpected prompts and report them immediately.

These controls work best together. Technical controls reduce the attack surface, while training helps users recognize that repeated prompts are a warning sign, not a normal inconvenience.

**Exam focus**

For CS0-003, connect suspicious MFA logs to the attack pattern, not just the authentication result. A user who receives repeated prompts and then approves one may be experiencing MFA fatigue, especially if the login also shows a new device or unusual location.

Key takeaways:

- MFA fatigue is a **social engineering** attack.
- It often begins with stolen credentials.
- Repeated prompts, denials, and sudden approvals are important clues.
- New devices and unusual locations strengthen suspicion.
- Response should include verification, containment, and stronger MFA controls.
