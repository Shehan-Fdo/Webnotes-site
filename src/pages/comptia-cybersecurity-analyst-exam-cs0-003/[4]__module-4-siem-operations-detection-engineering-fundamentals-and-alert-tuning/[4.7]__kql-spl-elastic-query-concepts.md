## **Course:** CompTIA Cybersecurity Analyst (Exam CS0-003)

**Query language purpose and scope**

Security analysts use query languages to explore logs, detect anomalies, and investigate incidents. The core idea is the same across platforms: you ask a data store to filter, search, aggregate, and correlate records so you can turn raw telemetry into findings.

Kusto Query Language (KQL) is designed for readable, expressive analysis of structured, semi-structured, and unstructured data. Splunk Processing Language (SPL) is used to search and analyze indexed events and metrics. Elastic query concepts focus on querying data in the Elastic ecosystem, often through Elasticsearch-style search and query syntax.

- **KQL** supports deep analysis of telemetry, metrics, and logs.
- **SPL** is built around event search, reporting, and operational analysis.
- **Elastic** query approaches are commonly used to search and filter indexed data.
- All three support investigation workflows such as threat hunting and incident review.

**KQL in Microsoft Sentinel data lake**

Microsoft Sentinel data lake lets you store and analyze high-volume, low-fidelity logs, asset inventories, and historical records for up to 12 years. Because storage and compute are decoupled, you can query the same copy of data with multiple tools without moving or duplicating it.

You can use KQL in the Microsoft Defender portal to explore data lake resources and federated tables. The KQL queries page lets you edit and run queries, while the Jobs page lets you manage jobs that promote data to the analytics tier or create aggregate tables in the data lake tier.

- Run KQL queries interactively for exploration.
- Create jobs to promote data or build aggregate tables.
- Run jobs on demand or on a schedule.
- Manage jobs by enabling, disabling, editing, or deleting them.

> KQL queries in the data lake require the appropriate onboarding and access prerequisites before you can run them.

**KQL query use cases**

Sample KQL queries help you investigate security incidents and monitor suspicious activity. Microsoft Sentinel includes out-of-the-box queries in the KQL query editor under the Queries tab.

A practical example is trend analysis of Entra ID sign-in logs to detect unusual location changes. The sample query described in the evidence looks for anomalous sign-in locations by computing trend lines of location diversity and highlighting accounts with the steepest increase.

- Use sample queries as starting points for investigations.
- Adapt built-in queries to your environment and data sources.
- Look for trends, outliers, and unusual changes in behavior.
- Use KQL jobs when you want repeatable or scheduled analysis.

**SPL concepts in Splunk**

Splunk documentation describes SPL as a language for querying and analyzing data in Splunk Cloud and Splunk Enterprise. It works with events, metrics, and related data structures that support search and reporting.

An **event** is a set of values associated with a timestamp. It can be a single-line or multi-line record such as a log entry, configuration file, or stack trace. Splunk also supports **transactions**, which group related events that span time and represent a single activity, such as a customer session.

- Events are timestamped records.
- Transactions group conceptually related events.
- Metrics include a timestamp, one or more measurements, and optional dimensions.
- SPL is useful when you need search, filtering, and reporting over indexed data.

**Elastic query concepts**

Elastic query concepts center on searching and analyzing data in the Elastic stack. In practice, this means working with query syntax that can retrieve indexed data and support operational or security analysis.

A useful comparison point is that Elastic-style querying often aims to express search intent against indexed content, similar to how KQL and SPL express intent in their own ecosystems. The exact syntax and tooling differ, but the analyst goal stays the same: find relevant records quickly and accurately.

- Use Elastic queries to search indexed data.
- Apply filters to narrow results to relevant events.
- Compare results across time or fields when investigating behavior.
- Treat query design as part of the investigation workflow, not just a retrieval step.

**Comparing the three query approaches**

| Language | Primary environment | Common analyst use |
|---|---|---|
| KQL | Microsoft Sentinel, Azure Data Explorer, Azure Monitor, Microsoft Fabric | Threat hunting, investigations, log analysis |
| SPL | Splunk Cloud, Splunk Enterprise | Search, reporting, event analysis |
| Elastic query concepts | Elastic ecosystem | Search and analysis of indexed data |

KQL is especially strong for readable analytics over logs and telemetry. SPL is centered on event-oriented search and reporting. Elastic query concepts are useful when you need to work within Elastic-based search and analysis workflows.

**Practical study takeaways**

When you see a question about these languages, focus on the platform context and the analyst task. The exam is more likely to test what each language is used for than exact syntax.

- Match the language to the platform.
- Identify whether the task is search, aggregation, trend analysis, or investigation.
- Remember that KQL in Sentinel can run interactively or through jobs.
- Remember that SPL works with events, metrics, and transactions.
- Remember that Elastic querying is tied to indexed search and analysis.
