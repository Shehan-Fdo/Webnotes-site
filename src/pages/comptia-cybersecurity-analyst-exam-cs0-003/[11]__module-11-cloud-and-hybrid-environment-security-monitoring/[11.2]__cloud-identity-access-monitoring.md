## **Course:** CompTIA Cybersecurity Analyst (Exam CS0-003)

**Cloud identity and access monitoring**

Cloud environments depend on identity and access management (IAM) to control who can access resources and under what conditions. Monitoring focuses on the identity layer because access changes, trust changes, and unusual logins often reveal risk before data is touched.

In multi-cloud environments, the challenge is not the security goal itself but the different implementations across providers. You need visibility into identity structures, policy changes, and authentication events so you can spot suspicious behavior consistently.

- **IAM** is the core control plane for access decisions in cloud platforms.
- Multi-cloud security becomes harder because each provider uses different terminology and models.
- Monitoring should cover both routine access and changes to identity configuration.

**What to watch in cloud identity logs**

Cloud identity monitoring is most useful when you look for changes that alter who can authenticate, what they can do, or how trust is established. Suspicious activity often involves identity structure changes rather than direct data access.

Common signals include:
- Creation or deletion of identity groups.
- Changes to role trust relationships.
- New trust anchors or profiles that extend access to external workloads.
- Unexpected logouts, redirects, or missing context that may indicate identity or session issues.

> Access changes are high-risk because they can quietly expand permissions or hide who actually has them.

**Why identity events matter**

Identity and access events are valuable because they often show attacker movement early. A malicious actor may try to create persistence by changing roles, groups, or trust settings instead of immediately attacking data.

Monitoring also helps you separate normal cloud operations from suspicious behavior:
- Legitimate administration usually follows known change windows and approved workflows.
- Suspicious changes often appear outside normal patterns or affect rarely used identity objects.
- Access anomalies become more meaningful when compared with baseline behavior.

**Log sources and investigation workflow**

Cloud identity monitoring depends on logs that capture authentication and management activity. CloudTrail is one example of a log source used to ingest AWS management events, and SIEM (Security Information and Event Management) tools can correlate those events with other telemetry.

A practical investigation flow is:
1. Review the identity or access event that triggered the alert.
2. Check whether the change matches an approved administrative action.
3. Correlate the event with other logs, such as cloud audit data or SIEM alerts.
4. Determine whether the activity changes permissions, trust, or access scope.
5. Escalate if the event is unexpected or affects critical identity controls.

**Shared responsibility and monitoring scope**

Cloud providers and customers share security responsibilities, but the exact boundary varies by service. Identity and access management, applications, and data are often the areas where customer responsibility is most visible.

That means you should monitor:
- Identity configuration changes you control.
- Access policies and trust relationships.
- Authentication activity tied to workloads and users.
- Misconfigurations that expose services or weaken access boundaries.

**Practical detection mindset**

A good detection strategy looks for behavior that should not happen in normal operations. Decoy resources, anomaly detection, and cloud log analysis can all help surface suspicious access attempts.

Useful indicators include:
- Access to resources that should not be used in normal business.
- Identity changes that do not match change management records.
- Unusual activity patterns compared with baseline logon times or expected event volume.
- Configuration changes that broaden access without a clear business reason.

**Key takeaways**

Cloud identity and access monitoring is about visibility into the control plane of access. If you can see who changed identity settings, what trust was modified, and when authentication behavior shifts, you can detect many cloud threats earlier.

- Focus on identity, roles, groups, and trust relationships.
- Use logs and SIEM correlation to validate suspicious activity.
- Compare events against baseline behavior and approved changes.
- Treat unexpected access changes as high-priority investigation items.
