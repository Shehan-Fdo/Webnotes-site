## **Course:** CompTIA Cybersecurity Analyst (Exam CS0-003)

**Cloud audit logs and API visibility**

Cloud audit logs help you trace actions taken by identities and services in cloud environments. In AWS, <abbr title="Identity and Access Management">IAM</abbr> and AWS Security Token Service (<abbr title="AWS STS">AWS STS</abbr>) integrate with AWS CloudTrail, which records API calls as events. CloudTrail captures calls from the console and from API requests, and it can deliver events continuously to an Amazon S3 bucket when you configure a trail.

These logs are useful because they show what request was made, who made it, and supporting context such as source information. If you do not configure a trail, you can still review recent events in CloudTrail event history.

- Look for API activity tied to identity changes, role use, or application access.
- Use the log source that preserves request context, not just the final outcome.
- Treat sudden changes in cloud identity behavior as investigation triggers.

**Suspicious API call patterns**

Suspicious API calls often show up as unexpected changes to cloud identities, applications, or access paths. A key example is malicious OAuth application activity in Microsoft environments, where threat actors create or modify an enterprise application to gain access to email, Teams, SharePoint, or OneDrive data. In these cases, the API activity is not just administrative noise; it can be the mechanism used to establish or expand access.

When you investigate, focus on whether the API action matches normal administration. A legitimate change usually has a clear business reason, while suspicious activity may involve new app creation, modified permissions, or access that appears out of sequence with normal change control.

- Compare the API call against the expected role of the account or service.
- Check whether the action affects authentication, authorization, or application consent.
- Correlate the event with user sign-in activity and other audit records.

**Privilege changes as an attack signal**

Privilege changes are a common indicator of account manipulation. Security content for privilege escalation includes events such as a new local user account, a user privilege level change, an account modification, or a user granted an admin role. These events matter because attackers often move from basic access to elevated control quickly.

In hybrid cloud environments, privilege escalation can happen across multiple identity sources and systems. That makes the detection window narrow and the impact high. You should pay attention to any event that changes a user’s effective permissions, especially when it happens without a matching administrative workflow.

- Watch for new accounts created with elevated access.
- Flag role changes that do not align with approved change activity.
- Review whether the privilege increase happened alongside suspicious login behavior.

**Investigation workflow**

1. Identify the suspicious API call or privilege change in the audit log.
2. Confirm the identity, source, and time of the event.
3. Compare the action to expected administrative behavior.
4. Correlate the event with sign-in logs, application activity, and other audit records.
5. Determine whether the change expanded access, created persistence, or enabled data access.

This workflow helps you separate routine administration from malicious activity. The goal is not only to find the event, but to understand whether it changed the security posture of the account, application, or environment.

**Practical detection focus**

Audit data is most valuable when you use it to answer a few direct questions: Who made the change? What API was called? Did the action create, modify, or elevate access? Was the event part of a normal workflow?

- **Identity context** matters because the same API call can be benign for one role and suspicious for another.
- **Change context** matters because privilege changes often indicate account manipulation.
- **Correlation** matters because one event rarely proves compromise on its own.

> If a cloud action changes permissions, trust, or application access without a clear business reason, treat it as suspicious until you verify it.

**Key takeaways**

- Cloud audit logs provide the evidence trail for API activity and privilege changes.
- AWS CloudTrail records IAM and AWS STS API calls as events.
- Malicious OAuth application changes can be used to access sensitive cloud data.
- Privilege escalation often appears as account creation, account modification, or admin role assignment.
- Correlating audit logs with sign-in and application activity improves confidence in your findings.
