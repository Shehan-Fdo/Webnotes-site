## **Course:** CompTIA Cybersecurity Analyst (Exam CS0-003)

**Shared responsibility model**

The shared responsibility model defines how security duties split between the cloud service provider and you. The exact boundary depends on the service model and the specific service you use.

Cloud providers typically handle the underlying cloud infrastructure. That includes the physical facilities and the lower layers that support the service. You handle what you deploy and configure in the cloud.

- **Provider-managed** duties commonly include physical security and core platform operations.
- **Customer-managed** duties commonly include data, application security, and configuration.
- Responsibility changes with the service and how you integrate it into your environment.

> Misaligned responsibility boundaries can create visibility gaps, misconfigurations, and compliance risk.

**Service model differences**

Cloud workload responsibility varies across <abbr title="Software as a Service">SaaS</abbr>, <abbr title="Platform as a Service">PaaS</abbr>, and <abbr title="Infrastructure as a Service">IaaS</abbr>. As you move from SaaS to IaaS, you generally take on more operational control and more security responsibility.

- **SaaS**: The provider manages most of the stack, while you focus on usage, data handling, and access.
- **PaaS**: You deploy applications without managing virtual machines or operating systems.
- **IaaS**: You manage virtual machines, operating systems, and applications.

A practical way to remember the split is that the provider secures more of the stack in SaaS, less in PaaS, and least in IaaS.

| Model | Typical customer responsibility | Typical provider responsibility |
|---|---|---|
| SaaS | Data use, access, and configuration choices | Most of the application and platform stack |
| PaaS | Applications and data | Runtime, platform, and underlying infrastructure |
| IaaS | Virtual machines, operating systems, applications | Physical infrastructure and core cloud services |

**What you still own in the cloud**

Even when the provider secures the platform, you remain responsible for the parts you control. That usually includes secure configuration, patching where applicable, and protecting the data and applications you place in the service.

- **Guest operating system** updates and security patches are customer responsibilities in many shared models.
- **Application software** remains under your control.
- **Security group firewall** configuration is a customer task in AWS-style shared responsibility examples.
- **Data protection** and access control remain critical customer concerns.

**Why the model matters**

The shared responsibility model helps reduce your operational burden, but it does not remove your security obligations. You still need to understand which controls belong to you and which belong to the provider before you deploy workloads.

- It helps you avoid assuming the provider covers every control.
- It supports better planning for compliance and risk management.
- It clarifies how responsibilities shift across different cloud services.

**Exam focus**

For exam purposes, focus on the boundary between provider and customer. If a question asks who manages the operating system, application, or configuration, check the service model first.

1. Identify whether the workload is SaaS, PaaS, or IaaS.
2. Determine which layer is being discussed: infrastructure, platform, operating system, application, or data.
3. Assign responsibility based on the layer and the service model.

**Quick memory aid**

- **Provider**: infrastructure, facilities, and core service operation.
- **Customer**: data, applications, and configuration.
- **Service model**: determines how much of the stack you manage.
