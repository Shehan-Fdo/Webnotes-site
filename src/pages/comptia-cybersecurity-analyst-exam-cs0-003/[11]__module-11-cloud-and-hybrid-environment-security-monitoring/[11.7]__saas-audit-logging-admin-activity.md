## **Course:** CompTIA Cybersecurity Analyst (Exam CS0-003)

**SaaS audit logging overview**

SaaS audit logs are time-stamped records of actions that occur in a cloud service. They help you answer who did what, when, and in some cases where or from which service the action came. In practice, these logs support security analysis, incident response, and compliance evidence.

Audit logging is especially important in environments where users, administrators, and system processes all generate activity. The logs can show file access, downloads, edits, deletions, sharing changes, sign-ins, and configuration changes. That makes them useful for spotting suspicious behavior and reconstructing events after an incident.

- **Audit logs** provide chronological visibility into activity.
- **Admin activity** is part of the same investigative picture as user activity.
- **Retention and access** matter because logs are only useful if you can search and review them later.

**Microsoft 365 audit logging**

Microsoft 365 audit logging is turned on by default for Microsoft 365 organizations. If auditing is not enabled, the portal prompts you to start recording user and admin activity. You can search these activities in the Microsoft Purview portal.

The audit log tables group related activities by product or service. They include the friendly name shown in the Activities drop-down list and the name used in PowerShell. This helps you map what you see in the portal to the underlying activity name used for filtering or automation.

- Search the audit log in the Microsoft Purview portal.
- Use the activity tables to identify the correct event names.
- Expect coverage across multiple Microsoft 365 services.

> Audit logs are only useful when you know which activity names to search for and which service generated them.

**What to monitor in SaaS activity**

Expanded SaaS logging increases visibility across services such as Exchange Online, SharePoint, and Teams. That broader coverage helps you detect suspicious behavior earlier and gives you more context during investigations.

Common high-value activity areas include:
- Email access and search behavior
- Messaging activity
- File interactions
- User sign-ins
- Settings changes
- Downloads, deletions, and sharing actions

These events can reveal unusual patterns, such as a spike in file downloads or deletions, which may indicate compromise or misuse. The goal is not to watch everything manually, but to focus on behaviors that are more likely to signal risk.

**Defender for Cloud Apps activity visibility**

Microsoft Defender for Cloud Apps provides visibility into activities from connected apps. After you connect an app with an App connector, Defender for Cloud Apps scans historical activity for a retroactive period that varies by app and then keeps updating with new activity.

The activity names and types come directly from the connected app’s third-party API. Defender for Cloud Apps displays them as received and does not define or modify them. That means you should interpret the activity in the context of the source application.

- Connected apps feed activity data into the platform.
- Historical activity may be scanned after onboarding.
- Activity labels reflect the source app’s API naming.

**Investigating user and admin actions**

When you investigate SaaS activity, start with centralized visibility. A consolidated view helps you connect events across services and avoid missing related actions. This is useful when identities and data are spread across multiple SaaS platforms.

A practical investigation flow is:
1. Review the alert or suspicious event in a centralized console.
2. Identify the user, admin, app, or service involved.
3. Correlate related actions across logs and services.
4. Look for unusual sequences such as mass downloads, deletions, or configuration changes.
5. Use the activity context to decide whether the behavior is expected, risky, or malicious.

SaaS investigations work best when you normalize events and compare them across services. Without that context, isolated log entries can be hard to interpret.

**Why audit logs matter for security and compliance**

Audit logs support both technical response and organizational accountability. They help you determine the scope of an incident, identify affected data, and understand whether a user, administrator, or automated process performed the action.

They also support compliance because they create a record of activity that can be reviewed later. In cloud SaaS environments, where many business processes depend on third-party services, auditability is part of maintaining trust and control.

- Use logs to reconstruct incidents.
- Use logs to validate administrative changes.
- Use logs to support compliance reporting.

**Key takeaways**

SaaS audit logging gives you the evidence needed to investigate activity across cloud services. For exam purposes, focus on the relationship between visibility, detection, and response:

- Logs capture user, admin, and system actions.
- Microsoft 365 auditing is enabled by default.
- Purview and Defender for Cloud Apps are common places to search and investigate.
- High-risk behaviors often show up as unusual file, email, messaging, or configuration activity.
- Centralized, normalized logs make cross-SaaS investigations more effective.
