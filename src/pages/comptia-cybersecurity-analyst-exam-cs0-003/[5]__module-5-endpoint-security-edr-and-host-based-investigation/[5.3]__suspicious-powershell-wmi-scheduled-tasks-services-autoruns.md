## **Course:** CompTIA Cybersecurity Analyst (Exam CS0-003)

**Persistence patterns on Windows**

Threat actors often use native Windows features to keep access after reboot or cleanup attempts. Scheduled tasks, services, autorun items, startup items, the registry, and PowerShell-based execution all support persistence because they blend into normal system behavior.

In incident response, you should assume persistence is present until you prove otherwise. Traditional methods still appear frequently because they work and do not require exotic tooling.

- **Scheduled tasks** are a common persistence mechanism.
- **Services** can start automatically and run without user interaction.
- **Autoruns** and startup items can launch payloads at logon or boot.
- **Registry** artifacts often reveal persistence configuration.
- **PowerShell** activity can indicate scripted execution or follow-on actions.

**Suspicious scheduled tasks**

Scheduled tasks deserve close review because they can be created, modified, or deleted through Task Scheduler, Windows Management Instrumentation (WMI), PowerShell, or API-based methods. Detection strategy DET0441 from MITRE ATT&CK notes that suspicious tasks may execute through `svchost.exe` or `taskeng.exe`, and that hidden or anomalous tasks are especially important when they appear under `SYSTEM` or unusual user contexts.

You should look for tasks that do not match the host’s normal purpose, especially if they are hidden, newly created, or tied to privileged accounts.

- Review task creation and modification activity.
- Check for execution from `svchost.exe` or `taskeng.exe`.
- Pay attention to tasks under `SYSTEM`.
- Treat hidden or anomalous task names as suspicious.
- Correlate task activity with process creation and registry changes.

> Hidden scheduled tasks may evade common tools such as `schtasks /query`, Autoruns, and the Task Scheduler GUI when attackers manipulate the task security descriptor.

**Hidden tasks and registry artifacts**

Some malware hides scheduled tasks by changing the Security Descriptor Definition Language (SDDL) permissions on the task. This can deny read access and prevent common audit tools from showing the task even though it still exists and runs.

That means you should not rely on a single visibility source. If a task seems to disappear from one tool, compare it with registry evidence and other telemetry.

- Use multiple sources to confirm task existence.
- Compare GUI results with registry artifacts.
- Treat missing visibility as a possible evasion technique.
- Look for unusual access control changes on task-related entries.

**PowerShell indicators**

Suspicious PowerShell activity can appear as unusual provider events, unexpected host application details, or commands that do not fit the user’s normal behavior. In the evidence, a PowerShell session shows provider startup events such as `Registry` and `function`, along with a command like `Get-CimInstance Win32_Processor | Format-List -Property NumberOfCores`.

PowerShell is not inherently malicious, so context matters. Focus on whether the command line, parent process, and timing make sense for the environment.

- Inspect the `HostApplication` and command line.
- Look for unusual provider events.
- Compare activity against the user’s normal workflow.
- Correlate PowerShell with other persistence artifacts.

**WMI and API-based execution**

Windows Management Instrumentation can be used to create or trigger scheduled tasks, and it can also support broader remote administration or scripted execution patterns. Because WMI can be part of legitimate administration, you need supporting evidence before labeling it malicious.

Use WMI findings as a pivot point rather than a standalone conclusion.

- Correlate WMI activity with task creation.
- Check whether the action aligns with administrative change windows.
- Look for related PowerShell or process creation events.
- Validate whether the host normally uses WMI for automation.

**Services and cleanup behavior**

Malware may disable host security services and then remove traces such as the initial dropper, event log entries, and prefetch files. When that happens, you often need to rely on surviving artifacts rather than the deleted payload itself.

Registry artifacts and file system metadata are especially useful when the malware has already cleaned up parts of its footprint.

- Check for disabled or altered security services.
- Review registry evidence for persistence and configuration.
- Use file system metadata when files are missing.
- Expect event logs and prefetch data to be incomplete after cleanup.

**What to prioritize during investigation**

A practical workflow is to start with persistence locations, then expand into process and registry evidence. This helps you connect a suspicious task, service, or PowerShell event to the actual payload or parent activity.

1. Review scheduled tasks, services, autoruns, and startup items.
2. Check PowerShell logs and command lines for unusual execution.
3. Correlate findings with registry artifacts and file system metadata.
4. Validate whether WMI or API-based creation explains the behavior.
5. Look for cleanup signs such as deleted logs, prefetch files, or dropped binaries.

**Defender mindset**

The main goal is to determine how the attacker keeps access and what evidence still remains after cleanup. Native Windows mechanisms are attractive to attackers because they are common, flexible, and often overlooked during quick triage.

If one artifact is missing, do not stop. Use the remaining telemetry to reconstruct the persistence chain.
