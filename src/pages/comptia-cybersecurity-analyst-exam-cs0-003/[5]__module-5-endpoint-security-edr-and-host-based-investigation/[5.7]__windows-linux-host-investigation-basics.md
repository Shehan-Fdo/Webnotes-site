## **Course:** CompTIA Cybersecurity Analyst (Exam CS0-003)

**Host investigation overview**

Host-based digital forensics focuses on evidence from individual systems such as computers, laptops, and mobile devices. In a CySA+ context, you use host investigation to identify malicious activity, understand attacker behavior, and collect evidence from file systems, logs, and memory.

Windows and Linux both matter during incident response because attackers often leave traces on endpoints and servers. Linux is especially common in server and security tooling environments, while Windows remains a major target and a frequent source of forensic artifacts.

- **Host-based forensics** examines the endpoint itself rather than network traffic.
- **File systems** can reveal suspicious files, timestamps, permissions, and deleted or hidden content.
- **System logs** help you reconstruct activity and confirm what happened on the host.
- **Memory** can matter when malware runs only in RAM and leaves little on disk.

> Memory analysis is important because some malicious activity is fileless and may not leave strong disk evidence.

**Core command-line navigation**

Basic command-line navigation is often the first step in a live investigation. You use it to move through directories, confirm your location, and inspect files that may contain evidence.

```bash
# Linux
pwd
ls -la
ls -lrt
ls -lah
cd /var/log

# Windows
dir
cd
```

- `ls` and `dir` list directory contents.
- `pwd` confirms the current working directory.
- `cd` moves you into log, system, or user directories.
- `ls -la` shows hidden files and detailed metadata.
- `ls -lrt` helps you spot recently changed files.
- `ls -lah` makes file sizes easier to read during triage.

When you investigate a compromised host, start by checking for unusual file names, unexpected extensions, hidden files, and permission changes. Recently modified files and strange ownership or access settings can point to compromise.

**Useful Linux host checks**

Linux investigation often starts with quick identity and context checks. These commands help you confirm what system you are on, who is logged in, and where you are working.

```bash
uname -a
whoami
pwd
```

- `uname -a` shows OS, kernel, and architecture details.
- `whoami` identifies the active user.
- `pwd` confirms your current location before you inspect files.
- `/var/log` is a common place to review logs.
- `/tmp` may contain temporary files or malware-related artifacts.
- `/home` often contains user data and user-specific evidence.

These checks help you avoid confusion during triage. If you are reviewing suspicious activity, always validate the host context before drawing conclusions from files or logs.

**Investigation workflow on a live host**

A practical host investigation is usually iterative. You move from broad context to specific evidence, then back again if something looks suspicious.

1. Confirm the system identity and your user context.
2. Navigate to likely evidence locations such as log and temporary directories.
3. List files with detailed output so you can inspect timestamps and permissions.
4. Look for unusual names, hidden files, or unexpected changes.
5. Correlate what you see with logs or other host artifacts.

This workflow is useful because there is no universal command sequence for every incident. You choose the next step based on what the host is showing you.

**What to look for**

The goal is not just to run commands, but to interpret what they reveal. Small anomalies often matter more than obvious alerts.

- Recently modified files in sensitive directories.
- Unusual permissions such as `777`.
- Suspicious file names or extensions.
- Unexpected resources being accessed.
- A user appearing in a group unexpectedly.
- Signs that activity may be happening in memory rather than on disk.

These indicators do not prove compromise by themselves, but they help you prioritize deeper analysis. Use them to decide where to collect more evidence.

**Windows and Linux comparison**

| Task | Linux | Windows |
|---|---|---|
| List directory contents | `ls` | `dir` |
| Change directory | `cd` | `cd` |
| Check current location | `pwd` | `cd` with no path is not the same as `pwd`; use the appropriate shell context |
| Common evidence areas | `/var/log`, `/tmp`, `/home` | User and system locations relevant to the host under review |

The main difference is not the investigation goal, but the syntax and the artifact locations. You still look for the same kinds of clues: logs, timestamps, permissions, and suspicious files.

**Practical interpretation tips**

Host investigation works best when you stay methodical. If a file looks suspicious, ask whether its timestamp matches the incident window, whether its permissions are unusual, and whether its location makes sense for the system role.

- Use detailed listings to expose hidden files and metadata.
- Compare current findings against expected system behavior.
- Treat memory-only behavior as a possibility when disk evidence is thin.
- Focus on evidence that helps explain how the attacker gained access or what they touched.

**Exam focus**

For CySA+ CS0-003, expect host investigation questions to emphasize practical analysis rather than memorization. You should be able to recognize which command helps you confirm context, which directory is likely to contain evidence, and which artifact type supports incident reconstruction.

- Know the purpose of `ls`, `dir`, `cd`, `pwd`, `uname -a`, and `whoami`.
- Understand why logs, file metadata, and memory all matter.
- Be ready to identify suspicious changes in files, permissions, or user context.
- Remember that Linux and Windows investigations use similar logic with different commands and paths.
