## **Course:** CompTIA Cybersecurity Analyst (Exam CS0-003)

**Identity and authentication log basics**

Authentication logs record access attempts and help you see who tried to sign in, when they tried, and whether the attempt succeeded. In security operations, these logs are a primary source for spotting failed logins, unusual behavior, and access issues across identity providers and applications.

Identity logs are especially useful when you need to confirm whether a user completed multifactor authentication (MFA, multi-factor authentication) or whether a federated identity flow passed the expected authentication signal. They also help you understand how a cloud identity provider and a relying application interact during sign-in.

- **Authentication events** include successful and failed sign-ins.
- **Visibility** into login behavior supports threat detection and troubleshooting.
- **Federated sign-in** logs may be needed when one identity provider handles MFA for another.

**MFA and federated identity signals**

In federated setups, one identity system can satisfy MFA requirements for another. For example, Okta can be used to meet Microsoft Entra ID MFA requirements for a WS-Federation Office 365 app. This is relevant when you want the federated identity provider to handle the MFA challenge for an Okta-federated domain or support Windows Hello for Business enrollment.

A key practical point is that not every log source clearly shows third-party MFA completion. Microsoft Q&A indicates that Audit Logs and Sign-in Logs may not expose enough detail to prove that a user authenticated with Okta MFA. When you investigate these flows, you may need to validate the authentication outcome at the identity provider that actually performed the MFA step.

> If the relying platform does not record the third-party MFA result clearly, check the upstream identity provider logs instead of assuming the sign-in logs are complete.

- **Okta MFA** can satisfy Microsoft Entra ID MFA requirements in supported federated scenarios.
- **WS-Federation** is one of the sign-in patterns where this integration appears.
- **Log completeness** varies by platform, so verify the source of truth for MFA events.

**What to look for in logs**

When reviewing identity and authentication logs, focus on the event details that help you reconstruct the sign-in path. You want to know whether the attempt was successful, whether MFA was required, and whether the event aligns with expected user behavior.

- **Success or failure** of the authentication attempt.
- **MFA requirement** or step-up behavior during the sign-in.
- **Timing** of the event relative to other suspicious activity.
- **Identity provider** that processed the authentication.

These fields help you distinguish normal access from suspicious activity. They also support incident response when you need to confirm whether a user was challenged, blocked, or redirected during authentication.

**Suspicious sign-in detection**

Authentication logs are useful for detecting malicious attempts such as password spray, brute force, and repeated failed sign-ins. Microsoft guidance in the evidence notes that sign-in activity may take about 5–7 minutes to appear in Entra logs under normal conditions, which affects how quickly you can detect and respond.

That delay matters operationally. If you rely only on the admin center view, you may see events later than you want. Polling the Microsoft Graph API for `signInLogs` or `auditLogs` may not remove the underlying latency if it reads from the same data pipeline, so plan alerting around the actual event delay.

- **Failed sign-ins** can indicate password spray or brute force activity.
- **Log latency** can delay detection and response.
- **Graph API** access does not necessarily eliminate the same ingestion delay.

**Practical analysis workflow**

1. Review the sign-in or authentication event for the user and time window.
2. Check whether MFA was required or expected for the session.
3. Compare the event with upstream identity provider logs if the platform log does not show MFA completion.
4. Look for repeated failures, unusual timing, or patterns that suggest automated attack activity.
5. Correlate the authentication event with other security telemetry before you conclude the sign-in was benign.

**Operational takeaways**

Identity and authentication logs are most valuable when you treat them as evidence of the authentication path, not just a pass/fail record. They help you confirm MFA behavior, investigate federated sign-ins, and detect suspicious access attempts.

For CySA+ exam prep, remember these core ideas:

- Logs may not always show third-party MFA details in the relying platform.
- Federated identity flows can shift the authoritative MFA evidence to the upstream provider.
- Detection speed depends on log ingestion latency, so alerting strategy matters.
- Authentication logs are a key source for identifying suspicious login patterns.
