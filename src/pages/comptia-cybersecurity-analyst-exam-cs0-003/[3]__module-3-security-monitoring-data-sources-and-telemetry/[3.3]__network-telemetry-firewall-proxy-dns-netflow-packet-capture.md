## **Course:** CompTIA Cybersecurity Analyst (Exam CS0-003)

**Network telemetry overview**

Network telemetry is the collection of network-generated data that helps you monitor, detect, and investigate suspicious activity. In security operations, you use it to build a timeline, confirm whether traffic is expected, and pivot from one clue to another during triage. The evidence emphasizes network analysis, traffic measurement, and the use of logs and packet-level data to support incident response.

Telemetry works best when you treat each source as a different lens on the same event. Some sources show policy decisions, some show name resolution, some show flow patterns, and some show the actual contents of packets.

- **Firewall logs** show allowed or blocked traffic decisions.
- **Proxy logs** help you see web access and intermediary requests.
- **DNS** data helps you trace name lookups tied to suspicious destinations.
- **NetFlow** summarizes traffic patterns for analysis and detection.
- **Packet capture** provides deeper detail when you need to inspect traffic content.

**Firewall telemetry**

Firewall data is useful when you need to confirm whether a connection was permitted, denied, or repeatedly attempted. It gives you a policy-oriented view of network activity, which makes it valuable for spotting scanning, blocked outbound connections, or unusual access attempts.

Use firewall telemetry to answer practical questions:

- Was the connection allowed or denied?
- Did the same source repeat the attempt?
- Does the traffic match the expected service or port?
- Is the destination unusual for that host or user?

Firewall logs are often most useful when you correlate them with other sources. A blocked connection may matter less on its own than when it lines up with DNS lookups, proxy activity, or a flow spike.

**Proxy and DNS telemetry**

Proxy logs and DNS records help you connect user activity to external destinations. A proxy can show web requests that pass through an intermediary, while DNS shows the name resolution step that often happens before a connection is made. Together, they help you trace where a host tried to go and how it got there.

These sources are especially helpful when you are investigating suspicious browsing, command-and-control lookups, or traffic that appears to bypass direct destination visibility.

- **Proxy logs** can reveal requested URLs or web destinations.
- **DNS logs** can reveal queried hostnames and repeated lookups.
- **Correlation** between DNS and proxy data helps you validate a sequence of activity.
- **Anomalies** such as unusual domains or repeated failed lookups can support triage.

> If encrypted traffic limits visibility, proxy and DNS telemetry can still provide useful context even when payload content is not available.

**NetFlow and traffic analysis**

NetFlow is a flow-based telemetry source that summarizes network conversations rather than storing full packet contents. The evidence points to network traffic analysis and measurement as a core use case, which makes flow data useful for spotting trends, spikes, and unusual communication patterns.

Flow data is practical when you need broad visibility across many hosts or long time ranges. It helps you identify who talked to whom, how much data moved, and whether a pattern stands out from normal behavior.

- **Flow records** are efficient for large-scale monitoring.
- **Traffic spikes** can indicate abuse, misconfiguration, or an active incident.
- **Baseline comparison** helps you spot abnormal communication patterns.
- **Pivoting** from a flow record can lead you to firewall, DNS, proxy, or packet data.

A flow view is not the same as packet inspection. It gives you summary-level evidence, which is often enough for triage but not always enough for deep validation.

**Packet capture**

Packet capture gives you the most detailed network view in this set because it records packet-level traffic for inspection. The evidence notes that analysts may carve network logs or use additional telemetry during incident work, and packet capture fits that deeper investigative role.

Use packet capture when you need to verify protocol behavior, inspect session details, or confirm what actually crossed the wire. It is especially useful after a higher-level source suggests something suspicious and you need stronger evidence.

- **Packet data** supports detailed inspection.
- **Protocol analysis** helps validate what a connection was doing.
- **Session detail** can confirm or refute assumptions from logs.
- **Targeted capture** is more practical than trying to inspect everything.

**How to use telemetry during triage**

A practical workflow is to start broad and then narrow your focus. You usually begin with summary sources, then move toward more detailed evidence as needed.

1. Review flow or log data to identify the suspicious host, time, or destination.
2. Check firewall, proxy, and DNS telemetry to confirm the sequence of activity.
3. Compare the observed traffic to normal behavior or known business use.
4. Move to packet capture if you need deeper validation or protocol detail.
5. Document the timeline and the evidence that supports your conclusion.

This approach matches the evidence’s emphasis on rapid triage, network analysis goals, and using multiple tools to carve logs or add telemetry.

**Key comparisons**

| Source | Best for | Typical value |
|---|---|---|
| Firewall | Allow/deny decisions | Confirms policy enforcement |
| Proxy | Web access visibility | Shows intermediary requests |
| DNS | Name resolution context | Links hosts to queried domains |
| NetFlow | Traffic patterns | Highlights volume and communication trends |
| Packet capture | Deep inspection | Verifies packet and protocol details |

**Practical exam focus**

For CySA+ style questions, focus on what each telemetry source contributes and when you would use it. The exam often expects you to choose the source that best matches the investigative need rather than the one with the most detail.

- Use **firewall** data for access control and blocked traffic questions.
- Use **proxy** and **DNS** data for destination and lookup context.
- Use **NetFlow** for broad traffic analysis and anomaly spotting.
- Use **packet capture** when you need the deepest technical confirmation.
