## **Course:** CompTIA Cybersecurity Analyst (Exam CS0-003)

**Windows event logging and Sysmon overview**

Windows event logs provide the baseline telemetry for endpoint activity, while <abbr title="System Monitor">Sysmon</abbr> adds deeper visibility into system behavior. Sysmon is a Microsoft Sysinternals tool that records detailed low-level events on Windows devices, including process creation, network connections, file and registry changes, driver and service activity, and Windows Management Instrumentation (WMI) actions.

Sysmon matters because native Windows logging can leave gaps for security analysis. In practice, you use it to improve detection, support threat hunting, and reconstruct what happened during an incident.

- **Windows Event Logs** are the default source of operating system activity on Windows.
- **Sysmon** extends visibility beyond the default logs.
- **Telemetry** from Sysmon helps you understand how software behaves over time.
- **Forensics** benefits from the extra detail in process, network, and registry events.

> Sysmon events are written to `Applications and Services Logs -> Microsoft -> Windows -> Sysmon -> Operational`, and event timestamps are recorded in UTC.

**What Sysmon records**

Sysmon focuses on activity that is useful for security monitoring and investigation. The evidence highlights several event categories that are especially relevant when you are looking for suspicious behavior.

- Process initiation and execution
- Network connections and communication
- File modifications
- Registry modifications
- Driver and service activity
- WMI actions

These event types help you connect a process to its network behavior and its changes to the system. That makes Sysmon useful for spotting stealthy threats that may not stand out in standard logs.

**How to use Windows logs in analysis**

When you investigate Windows activity, you usually correlate native event logs with Sysmon data. That gives you both the broad operating-system view and the more detailed endpoint telemetry.

1. Identify the time window of interest.
2. Review the relevant Windows event logs for account, service, or system activity.
3. Check Sysmon for process creation, network connections, and file or registry changes.
4. Correlate the events to build a timeline of user and system behavior.
5. Look for unusual sequences, such as a new process followed by network activity or configuration changes.

This approach is especially useful when you need to detect lateral movement, suspicious execution, or post-compromise activity.

**Linux syslog and auth logs**

Linux telemetry is not a one-to-one match with Windows telemetry. The logging model differs, so you interpret Linux events through its own system logs rather than trying to map them directly to Windows event IDs.

Two common sources are:

- **syslog** for general system messages
- **auth logs** for authentication-related activity

Authentication logs are important when you are checking account behavior, login attempts, and other access-related events. They help you understand whether a user authenticated successfully, failed repeatedly, or performed actions that may indicate misuse.

**Windows vs. Linux logging differences**

Windows and Linux both provide visibility into system activity, but they do so in different ways. Windows is relatively uniform, which simplifies telemetry collection. Linux environments vary more, so the exact log locations and formats can differ by distribution and configuration.

| Area | Windows | Linux |
|---|---|---|
| Logging style | More uniform across systems | More variable across distributions |
| Enhanced endpoint telemetry | Sysmon adds detailed visibility | Uses native system logs such as syslog and auth logs |
| Common investigation focus | Process, network, registry, service activity | Authentication and system messages |

The practical takeaway is that you should choose the right log source for the platform instead of assuming the same investigation workflow works everywhere.

**Threat hunting and incident response use**

Windows Event Logs, Sysmon, and Linux auth logs all support threat hunting and incident response, but they answer slightly different questions. Windows logs help you understand system and account activity, Sysmon adds high-fidelity endpoint telemetry, and Linux auth logs help you validate access behavior.

- Use Windows logs for baseline system and account events.
- Use Sysmon when you need deeper process and network visibility.
- Use Linux auth logs when you need to review authentication behavior.
- Use syslog when you need broader system messages and operational context.

**Practical analyst mindset**

You get the best results when you treat logs as evidence streams rather than isolated records. A single event rarely tells the full story. Instead, you build context by comparing timestamps, event types, and related activity across the available sources.

- Start with the platform-specific log source.
- Confirm the sequence of events before drawing conclusions.
- Prefer conservative interpretations when the log context is incomplete.
- Use detailed telemetry to support, not replace, investigation reasoning.
