## **Course:** CompTIA Cybersecurity Analyst (Exam CS0-003)

**Cloud audit logs and telemetry basics**

Cloud audit logs record administrative activity and access within cloud resources. They help you answer “who did what, where, and when?” and support security, auditing, and compliance work. In practice, audit telemetry is most useful when you can collect it consistently, retain it long enough for investigations, and analyze it across services or even across cloud providers.

- **Audit logs** capture administrative actions and access events.
- **Telemetry** becomes more valuable when you centralize it for search and correlation.
- Cloud logs are distinct from application logs and should be treated as a separate security data source.

**What the logs are used for**

Cloud logging supports threat detection, investigation, and remediation. The evidence emphasizes enabling, collecting, and storing audit logs, then using them with native detection tools, a cloud monitoring service, or a SIEM (Security Information and Event Management) platform. Time synchronization and log retention also matter because they improve event correlation and preserve evidence for later review.

- Use logs to detect known threats and anomalies.
- Keep retention aligned to investigation and compliance needs.
- Centralize analysis so you can correlate events across services.

**Google Cloud audit logging**

Google Cloud Audit Logs provide visibility into administrative activities and accesses within Google Cloud resources. The logs help you answer the same core questions as other cloud audit systems: who acted, what changed, where it happened, and when it occurred. The evidence also notes that audit logs support security, auditing, and compliance monitoring.

Google Cloud separates audit logging into streams such as admin activity and data access. Admin activity logs record API calls or other administrative actions that modify configuration or metadata, and they are always enabled. Data access logs are also important for troubleshooting and should generally remain enabled.

- **Admin activity** logs track configuration and metadata changes.
- **Data access** logs help with troubleshooting and visibility into access.
- Audit logs are separate from application logs.

> Keep data access audit logs enabled unless you have a clear reason not to, because they help with troubleshooting and investigation.

**AWS CloudTrail Lake for centralized analysis**

AWS CloudTrail Lake is described as a managed data lake for capturing, storing, accessing, and analyzing user and API activity on AWS and non-AWS sources. It supports immutable storage for audit and security investigations and allows SQL-based queries for search and analysis. The evidence also shows that Azure administrative or audit logs can be ingested into CloudTrail Lake, which makes it useful in multicloud environments.

A practical pattern is to ingest external cloud activity logs into a centralized event data store, then query them during investigations. This reduces the need to jump between separate consoles when you are tracing activity across platforms.

- Store activity events in one place for audit and security work.
- Use SQL queries to search and analyze events.
- Ingest non-AWS sources when you need cross-cloud visibility.

**Microsoft cloud logging and threat detection**

Microsoft’s cloud security guidance emphasizes enabling threat detection capabilities, collecting and storing audit logs, and centralizing security analysis with a SIEM. It also calls out log retention and time synchronization as part of a complete logging strategy. The goal is to monitor known resource types for expected threats and anomalies, then support detection, investigation, and remediation.

This is a good exam pattern to remember: logging is not just storage. It is a pipeline from collection to analysis to response.

- Enable native threat detection where available.
- Collect logs through a cloud monitoring service.
- Centralize analysis in a SIEM for correlation and alerting.

**Operational best practices**

Good cloud logging programs focus on completeness, consistency, and usability. You want logs that are enabled by default where possible, retained long enough for investigations, and centralized enough to support cross-service analysis. If you work in multicloud environments, the same principles apply even when the provider tools differ.

1. Enable audit logging for the services you need to monitor.
2. Retain logs long enough for security investigations and compliance.
3. Centralize logs in a monitoring platform or SIEM.
4. Synchronize time sources so events line up correctly.
5. Query and review logs regularly for anomalies and suspicious activity.

**Exam focus**

For CS0-003, expect questions that test the purpose of cloud audit logs, the value of centralized telemetry, and the difference between logging and detection. Be ready to identify why audit logs matter, how they support investigations, and why retention and time synchronization improve security operations.

- Audit logs answer who, what, where, and when.
- Centralization improves search and correlation.
- Retention and time sync strengthen investigations.
