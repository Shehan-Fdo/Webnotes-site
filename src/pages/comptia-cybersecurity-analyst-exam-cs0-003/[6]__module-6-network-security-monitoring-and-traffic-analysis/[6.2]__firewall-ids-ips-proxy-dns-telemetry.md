## **Course:** CompTIA Cybersecurity Analyst (Exam CS0-003)

**Network control and visibility**

Firewalls, intrusion detection systems (IDS), intrusion prevention systems (IPS), proxies, and DNS telemetry each contribute a different view of network activity. A firewall filters traffic, IDS and IPS focus on suspicious or malicious behavior, proxies mediate and observe client requests, and DNS telemetry exposes name-resolution patterns that can reveal hidden activity.

- **Firewall**: filters traffic based on rules and acts as a control point for allowed and denied connections.
- **IDS**: detects suspicious activity and alerts without blocking traffic.
- **IPS**: detects suspicious activity and can block or prevent it inline.
- **Proxy**: sits between clients and destinations to mediate requests and improve visibility into web access.
- **DNS telemetry**: shows domain lookups and can expose beaconing, tunneling, or unusual domain activity.

A practical security program uses these sources together because one control rarely shows the full picture. For example, a firewall may allow a connection, while DNS logs and proxy records reveal the destination pattern behind it.

**Firewall, IDS, and IPS differences**

The main distinction is where each control sits in the traffic path and what it does with suspicious traffic. A firewall primarily filters traffic, while IDS and IPS inspect traffic for signs of attack. IDS is passive from a blocking perspective, and IPS is active because it can stop traffic before it reaches a target.

- **Firewall**
  - Enforces allow/deny decisions.
  - Helps reduce exposure by limiting reachable services.
- **IDS**
  - Monitors traffic and raises alerts.
  - Supports investigation and detection workflows.
- **IPS**
  - Monitors traffic and takes preventive action.
  - Helps stop known or highly suspicious activity inline.

> Use conservative wording on exams: if a tool only alerts, treat it as IDS; if it blocks or prevents, treat it as IPS.

**Proxy and DNS telemetry**

Proxy logs and DNS logs add context that packet-level controls may not provide. Proxy data helps you see web access patterns, while DNS telemetry helps you see which domains a host tries to resolve. DNS logs are especially useful when attackers use domain lookups for command-and-control (C2), beaconing, or tunneling.

- **DNS telemetry indicators**
  - Weird or gibberish domain names.
  - Repeated failed lookups, such as `NXDOMAIN`.
  - Queries at fixed intervals, which can suggest beaconing.
  - Very long or random-looking domain strings, which can suggest DNS tunneling.
  - Newly registered or rare domains that are unusual for the environment.
- **Proxy telemetry indicators**
  - Repeated access to the same destination.
  - Unusual browsing or download patterns.
  - Requests that help tie a user or host to a web destination.

DNS logs are often valuable when firewalls or antivirus tools miss the behavior because the malicious activity blends into normal name resolution.

**Network telemetry mindset**

Network telemetry is the collection of metadata or traffic data from network devices for analysis. In practice, it helps you understand traffic patterns, detect anomalies, monitor performance, and respond to incidents faster. Telemetry can come from routers, switches, servers, and related monitoring tools.

- **Telemetry value**
  - Improves visibility into what is happening on the network.
  - Helps identify anomalies before they affect users.
  - Supports troubleshooting and incident response.
- **Telemetry sources**
  - Device logs.
  - Flow or metadata exports.
  - DNS records.
  - Proxy records.
  - Firewall events.

When you analyze telemetry, focus on patterns rather than single events. A single DNS lookup may be harmless, but repeated lookups at exact intervals or to rare domains can indicate a larger issue.

**How to compare the controls**

| Control | Primary role | Typical output | Best use |
|---|---|---|---|
| Firewall | Filter traffic | Allow/deny decisions | Reduce exposure and enforce policy |
| IDS | Detect activity | Alerts | Investigate suspicious traffic |
| IPS | Detect and prevent activity | Alerts plus blocking | Stop malicious traffic inline |
| Proxy | Mediate requests | Request and destination logs | Observe web access and user activity |
| DNS telemetry | Record name resolution | Query and response logs | Spot beaconing, tunneling, and rare domains |

**Exam focus**

For CS0-003, expect questions that ask you to match a control to its function or choose the best data source for investigation. Read the action carefully:

1. If the question asks for blocking or prevention, choose **IPS** or a firewall rule.
2. If it asks for alerting or detection without blocking, choose **IDS**.
3. If it asks for web request visibility, choose **proxy** logs.
4. If it asks for domain-resolution behavior, choose **DNS telemetry**.

A good rule is to map the tool to the layer of visibility it provides: enforcement, detection, mediation, or name resolution.
