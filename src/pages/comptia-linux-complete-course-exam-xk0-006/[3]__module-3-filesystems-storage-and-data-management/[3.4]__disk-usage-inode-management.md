**Disk usage fundamentals**

Linux storage problems usually fall into two categories: you run out of **blocks** (actual disk space) or you run out of **inodes** (filesystem metadata entries). Both conditions can stop applications from writing files, but they fail for different reasons.

A filesystem can still report free gigabytes while refusing new files if inode allocation is exhausted. This is common on systems that create huge numbers of small files, such as mail spools, cache directories, package build trees, and log-heavy services.

- **Blocks** store file contents.
- **Inodes** store metadata such as ownership, permissions, timestamps, and pointers to data blocks.
- A directory entry maps a filename to an inode number.
- Large files consume blocks quickly; many tiny files consume inodes quickly.

**How Linux reports usage**

Use `df` to inspect filesystem-level space usage and `du` to inspect directory or file-level usage. These tools answer different questions.

```bash
df -h
df -i
du -sh /var/log
du -sh /home/*
```

- `df -h` shows human-readable space usage by mounted filesystem.
- `df -i` shows inode usage by mounted filesystem.
- `du -sh` summarizes the apparent disk usage of a path.
- `du -sh /path/*` helps identify large subdirectories.

`df` reads filesystem metadata, so it is best for determining whether the filesystem itself is full. `du` walks the directory tree, so it is best for finding which files or directories consume space. On busy systems, `du` can take longer and may miss files that are deleted while it runs.

**Inodes and why they matter**

An inode is a filesystem data structure that describes a file. The inode does not store the filename; it stores the file’s attributes and the location of its data blocks. The filename lives in the directory entry.

<dl>
<dt><strong>Inode</strong></dt>
<dd>Metadata record for a file or directory.</dd>
<dt><strong>Hard link</strong></dt>
<dd>Another directory entry pointing to the same inode.</dd>
<dt><strong>Symbolic link</strong></dt>
<dd>A separate file that stores a path to another file.</dd>
</dl>

Because each file and directory consumes at least one inode, a filesystem with many small files can exhaust inodes before it exhausts space. Deleting large files may free blocks but not necessarily solve an inode shortage if the file count remains high.

> <mark>Important:</mark> inode exhaustion can prevent file creation even when `df -h` shows available space.

**Finding what is consuming space**

Start with the filesystem that is full, then narrow down the largest directories. A practical workflow is to identify the mount point, inspect top-level directories, and drill down.

```bash
df -hT
du -xhd1 /var
du -xhd1 /var/log
du -xhd1 /home
```

- `-T` shows filesystem type.
- `-x` keeps `du` on one filesystem and avoids crossing mount points.
- `-d1` limits depth to one level for quick triage.

For interactive investigation, `ncdu` is often faster than repeated `du` commands.

```bash
ncdu /
```

`ncdu` provides a navigable interface for sorting directories by size. It is especially useful on servers with deep trees or many nested application directories.

**Finding what is consuming inodes**

To diagnose inode pressure, check inode usage at the filesystem level and then count files in suspect directories.

```bash
df -i
find /var/spool -xdev -type f | wc -l
find /tmp -xdev -type f | wc -l
```

- `df -i` shows inode totals, used inodes, and free inodes.
- `find ... -type f | wc -l` counts files.
- `-xdev` prevents crossing filesystem boundaries.

If a directory contains millions of small files, the inode count can become the limiting factor even if the total byte usage is modest.

**Common causes of disk and inode exhaustion**

Typical causes are predictable and often tied to service behavior rather than a single large file.

- **Logs** grow without rotation or retention limits.
- **Package caches** accumulate after updates.
- **Temporary files** remain after crashes or failed jobs.
- **Build artifacts** pile up in development or CI environments.
- **Mail queues** and spool directories can create many small files.
- **Container layers and images** consume both blocks and metadata.
- **User home directories** may contain large downloads, archives, or caches.

**Practical cleanup approaches**

Before deleting anything, confirm what the files are and whether a service depends on them. Prefer safe cleanup methods over manual deletion when possible.

1. Identify the filesystem and the top consumer.
2. Confirm whether the issue is blocks, inodes, or both.
3. Stop or pause the service if it is actively recreating files.
4. Remove or rotate unneeded data.
5. Recheck usage with `df -h` and `df -i`.

Examples of safe cleanup actions:

```bash
journalctl --vacuum-time=7d
journalctl --vacuum-size=500M
apt clean
dnf clean all
rm -rf /tmp/*
```

- `journalctl --vacuum-time` removes older systemd journal entries.
- `journalctl --vacuum-size` trims journal storage to a target size.
- `apt clean` and `dnf clean all` remove package cache data.
- `rm -rf /tmp/*` is only appropriate when you understand local policy and active processes.

> <mark>Warning:</mark> do not delete files blindly from `/var`, `/run`, or application data directories without checking whether a service owns them.

**Filesystem and inode allocation behavior**

Different filesystems allocate inodes differently. Some create a fixed inode table at filesystem creation time, while others allocate metadata more dynamically. The practical takeaway is that inode availability is not always proportional to free space.

When formatting a filesystem, inode density may be influenced by filesystem type and mkfs options. For example, ext-family filesystems traditionally create a fixed number of inodes based on expected file size and count. If you expect many small files, choose a filesystem and creation strategy that supports that workload.

```bash
mkfs.ext4 -N 2000000 /dev/sdb1
mkfs.ext4 -i 4096 /dev/sdb1
```

- `-N` sets the number of inodes directly.
- `-i` sets bytes-per-inode, which affects inode density.

Use these options carefully and only when creating a new filesystem. They are not general-purpose tuning knobs for an already-mounted filesystem.

**Monitoring and alerting**

Disk and inode exhaustion are easier to prevent than to fix. Monitor both metrics and alert before the filesystem reaches critical thresholds.

Common checks include:

- filesystem percent used
- inode percent used
- growth rate over time
- largest directories under key mount points
- log volume and retention behavior

Example monitoring commands:

```bash
watch -n 5 'df -h /var; df -i /var'
```

For scripting, you can extract usage values and trigger alerts when thresholds are exceeded.

```bash
#!/usr/bin/env bash
usage=$(df -P /var | awk 'NR==2 {gsub("%","",$5); print $5}')
inode=$(df -Pi /var | awk 'NR==2 {gsub("%","",$5); print $5}')

if [ "$usage" -ge 90 ] || [ "$inode" -ge 90 ]; then
  echo "Alert: /var is near capacity"
fi
```

This kind of check is useful in cron jobs, monitoring agents, or lightweight health scripts.

**Log rotation and retention**

Logs are a frequent source of disk growth. Proper rotation limits file size, preserves history, and reduces the risk of both block and inode exhaustion.

A typical `logrotate` policy may compress old logs, keep a fixed number of rotations, and create new files with the correct ownership.

```conf
/var/log/myapp/*.log {
    daily
    rotate 7
    compress
    delaycompress
    missingok
    notifempty
    create 0640 myapp adm
}
```

- `daily` rotates logs every day.
- `rotate 7` keeps seven archived copies.
- `compress` reduces storage usage.
- `delaycompress` avoids compressing the most recent rotated file immediately.
- `create` ensures the new log file has the right permissions.

**Troubleshooting workflow**

When a system reports “No space left on device,” do not assume the disk is full. The error can also appear when inodes are exhausted or when a quota is reached.

1. Run `df -h` to check block usage.
2. Run `df -i` to check inode usage.
3. Use `du -xhd1` on the affected mount point.
4. Inspect logs, caches, and temporary directories.
5. Verify whether quotas or reserved blocks are involved.
6. Re-test the failing application after cleanup.

Useful commands for deeper inspection:

```bash
lsof +L1
find /var -xdev -type f -size +1G -ls
find /var -xdev -type f | sort | uniq -c | sort -nr | head
```

- `lsof +L1` finds deleted files that are still held open by processes.
- Large files may still consume space until the process closes them.
- Counting files can reveal directories with unusually high file counts.

**Reserved blocks and root access**

Some filesystems reserve a percentage of blocks for privileged use. This helps the system remain partially functional even when user space is full. On ext-family filesystems, the reserved block percentage is often visible with `tune2fs`.

```bash
tune2fs -l /dev/sda1 | grep -i 'Reserved block count\|Block count\|Reserved block percentage'
```

Reserved blocks can allow root-owned maintenance tasks to continue, but they do not solve the underlying capacity issue. Use them as a safety margin, not as a substitute for cleanup or expansion.

**Expanding capacity**

If cleanup is not enough, increase capacity at the filesystem or storage layer. The exact method depends on the environment.

- Extend the underlying virtual disk or LVM logical volume.
- Grow the filesystem after the block device expands.
- Add a new mount point and move data if expansion is not practical.
- Rebalance data across filesystems to reduce hot spots.

Example with LVM:

```bash
lvextend -r -L +20G /dev/vg0/var
```

- `-r` grows the filesystem after extending the logical volume.
- This is often the simplest path when the filesystem supports online growth.

**Key exam takeaways**

- **`df`** answers “How full is the filesystem?”
- **`du`** answers “What is using the space?”
- **`df -i`** answers “Are inodes exhausted?”
- Many small files can exhaust inodes before blocks.
- Deleted-but-open files can keep space allocated.
- Log rotation, cache cleanup, and monitoring prevent most incidents.
- Capacity expansion is the long-term fix when growth is legitimate.

**Quick reference**

```bash

df -i        # inode usage

```
