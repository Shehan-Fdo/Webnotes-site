**Filesystem types: what they are and why they matter**

A filesystem defines how data is organized, stored, and retrieved on a block device or virtual storage target. In Linux, you choose a filesystem based on workload, reliability needs, feature set, and tooling support. Exam questions often test both the filesystem type and the mount behavior that goes with it.

Common filesystem families include:

- **ext4**: general-purpose, widely supported, stable default for many Linux systems
- **XFS**: strong for large files, large filesystems, and parallel I/O
- **Btrfs**: copy-on-write filesystem with snapshots, subvolumes, and built-in volume management features
- **vfat/FAT32**: broad compatibility, often used for removable media and EFI System Partitions
- **NTFS**: common for interoperability with Windows systems
- **tmpfs**: memory-backed temporary storage
- **iso9660**: read-only optical media images
- **NFS** and **CIFS/SMB**: network filesystems rather than local disk filesystems

A practical workflow usually looks like this:

1. Partition or identify the block device.
2. Create the filesystem with the appropriate `mkfs` tool.
3. Mount it with the right filesystem type and options.
4. Add a persistent entry in `/etc/fstab` if it should mount at boot.

```bash
lsblk -f
blkid
mkfs.ext4 /dev/sdb1
mkfs.xfs /dev/sdc1
mkfs.btrfs /dev/sdd1
mkdir -p /data
mount /dev/sdb1 /data
```

**Creating filesystems**

The `mkfs` family formats a device with a filesystem. The command name usually matches the filesystem type.

- `mkfs.ext4` creates an ext4 filesystem
- `mkfs.xfs` creates an XFS filesystem
- `mkfs.btrfs` creates a Btrfs filesystem
- `mkfs.vfat` creates a FAT filesystem
- `mkfs.ntfs` may be available through external packages, depending on distribution

Use the correct tool before mounting; mounting does not create a filesystem. If a device has no filesystem signature, `mount` will fail.

> Be careful: formatting destroys existing data on the target device.

**Mounting basics**

Mounting attaches a filesystem to a directory called a mount point. Linux presents a single directory tree, so the mount point becomes the access path to the storage.

Key concepts:

- **Mount point**: an existing directory such as `/data` or `/mnt/backup`
- **Source**: a block device, UUID, label, network share, or pseudo-filesystem
- **Filesystem type**: the format or protocol, such as `ext4`, `xfs`, `btrfs`, `nfs`, or `tmpfs`
- **Options**: behavior controls such as read-only, no execution, or automatic mounting

Useful inspection commands:

```bash
findmnt
mount
df -Th
lsblk -f
blkid
```

`findmnt` is especially useful because it shows the active mount tree and options in a readable form.

**Common mount command patterns**

```bash
mount /dev/sdb1 /data
mount -t ext4 /dev/sdb1 /data
mount -o ro /dev/sdb1 /mnt/readonly
mount -o noexec,nodev,nosuid /dev/sdb1 /srv/share
```

- `-t` specifies the filesystem type when auto-detection is not enough or when clarity matters.
- `-o` passes mount options.
- `ro` mounts read-only.
- `rw` mounts read-write, which is often the default if the filesystem supports it.

**Persistent mounts with `/etc/fstab`**

The `/etc/fstab` file defines filesystems that should be mounted automatically, usually at boot or when `mount -a` runs. Each line has six fields:

```text
<fs_spec>  <fs_file>  <fs_vfstype>  <fs_mntops>  <fs_freq>  <fs_passno>
```

Example:

```fstab
UUID=1234-ABCD  /data   ext4  defaults,noatime  0  2
LABEL=BACKUP    /backup xfs   defaults          0  0
```

Field meanings:

- **fs_spec**: device, UUID, label, network source, or pseudo-filesystem name
- **fs_file**: mount point
- **fs_vfstype**: filesystem type
- **fs_mntops**: comma-separated mount options
- **fs_freq**: dump backup flag, rarely used on modern systems
- **fs_passno**: `fsck` order at boot; root is usually `1`, other local filesystems often `2`, and network or non-checkable filesystems often `0`

Common source forms in `fstab`:

- `UUID=...` is preferred for stable device identification
- `LABEL=...` is readable and convenient when labels are managed carefully
- `/dev/sdXn` works but can change across boots
- `server:/export/path` is used for NFS
- `//server/share` is used for CIFS/SMB

A typical local-disk entry:

```fstab
UUID=9f3a2c1d-1111-2222-3333-444455556666  /var/www  ext4  defaults,noatime  0  2
```

**What `defaults` means**

`defaults` is a shorthand for a standard set of mount options. On Linux, it typically expands to:

- `rw`
- `suid`
- `dev`
- `exec`
- `auto`
- `nouser`
- `async`

This is a shorthand, not a guarantee of every implementation detail on every filesystem or distribution. In exam scenarios, treat it as the standard baseline behavior unless another option overrides it.

**Common mount options and what they do**

Mount options fall into two broad groups:

- **Generic options**: apply to many filesystems
- **Filesystem-specific options**: only meaningful for a particular filesystem type

Generic options you should know:

- `ro`: read-only
- `rw`: read-write
- `noexec`: prevents execution of binaries from the mount
- `nosuid`: ignores setuid and setgid bits
- `nodev`: ignores device files on the mount
- `noatime`: does not update access times on reads
- `relatime`: updates access times only under certain conditions; common default on many systems
- `sync`: synchronous writes; safer but slower
- `async`: asynchronous writes; faster and usually default
- `user`: allows a non-root user to mount the filesystem, and typically only that user can unmount it
- `users`: allows any user to mount the filesystem, but only the user who mounted it can unmount it
- `auto`: mount at boot or when `mount -a` runs
- `noauto`: do not mount automatically
- `remount`: change options on an already mounted filesystem
- `bind`: create a bind mount of an existing directory tree

Examples:

```bash
mount -o remount,ro /data
mount -o noexec,nodev,nosuid /home
mount -o bind /var/www /srv/www
```

> A bind mount must already exist before you remount it with new options. For read-only bind mounts, many systems require a two-step approach or a source-and-target form that your distribution supports.

A safer pattern for a read-only bind mount is:

```bash
mount --bind /var/www /srv/www
mount -o remount,bind,ro /var/www /srv/www
```

**Filesystem-specific options**

Some options only apply to a specific filesystem type. These are common exam targets because they look like generic mount flags but are not.

Examples:

- **ext4**: `errors=remount-ro`, `acl`, `barrier`, `data=ordered`
- **XFS**: `inode64`, `noquota`, `prjquota`, `uquota`, `gquota`
- **Btrfs**: `compress=zstd`, `compress-force=zstd`, `subvol=`, `subvolid=`, `space_cache`, `ssd`
- **NFS**: `vers=4.2`, `hard`, `soft`, `timeo=`, `retrans=`
- **CIFS/SMB**: `username=`, `password=`, `vers=3.0`, `uid=`, `gid=`, `iocharset=`
- **tmpfs**: `size=`, `mode=`, `nr_inodes=`

Examples:

```bash
mount -t ext4 -o errors=remount-ro /dev/sdb1 /data
mount -t xfs -o prjquota /dev/sdc1 /projects
mount -t btrfs -o compress=zstd,subvol=@home /dev/sdd1 /home
mount -t tmpfs -o size=1G,mode=1777 tmpfs /tmp
```

**Filesystem behavior and recovery**

Different filesystems handle corruption and repair differently.

- **ext4** uses `fsck.ext4` and is commonly checked at boot if `fs_passno` is set appropriately.
- **XFS** uses `xfs_repair` for offline repair. `fsck.xfs` is typically a stub that exits with a message rather than performing a real check.
- **Btrfs** uses `btrfs check` and `btrfs scrub` for different maintenance tasks; `scrub` verifies data against checksums while mounted, while `check` is an offline consistency tool.
- **vfat** has `fsck.vfat` for basic checks.
- **NTFS** support depends on the toolchain and distribution; modern Linux often uses `ntfs3` in-kernel support or `ntfs-3g` in userspace.

For exam purposes, remember that not every filesystem uses `fsck` the same way. Some have dedicated repair tools, and some rely on wrappers or stubs.

**Choosing a filesystem**

Choose based on the workload and operational requirements.

- **ext4** is a safe default for general-purpose Linux servers and desktops.
- **XFS** is strong for large files, high throughput, and enterprise storage.
- **Btrfs** is useful when you need snapshots, subvolumes, compression, or integrated volume features.
- **tmpfs** is ideal for temporary, memory-backed data such as `/tmp` or application scratch space.
- **vfat** is useful when compatibility matters more than Linux permissions.
- **NFS** and **CIFS** are appropriate for shared storage over the network.

A simple comparison:

| Filesystem | Strengths | Common use |
|---|---|---|
| ext4 | Stable, broad support, easy administration | General-purpose local storage |
| XFS | Scales well, good for large files | Data volumes, media, logs |
| Btrfs | Snapshots, compression, subvolumes | Flexible storage layouts |
| tmpfs | Fast, memory-backed | Temporary files, runtime data |
| vfat | Broad compatibility | USB media, EFI partitions |

**Special mount targets**

Not every mount is a disk partition.

- **Pseudo-filesystems** such as `proc`, `sysfs`, `devtmpfs`, and `tmpfs` expose kernel or runtime data.
- **Network filesystems** such as NFS and CIFS mount remote storage.
- **Loopback mounts** attach a file as if it were a block device.
- **Bind mounts** expose an existing directory at another path.

Examples:

```bash
mount -t proc proc /proc
mount -t sysfs sysfs /sys
mount -t tmpfs tmpfs /run
mount -o loop /path/to/image.iso /mnt/iso
```

Modern systems often mount many pseudo-filesystems automatically through `systemd` or the init system, so you may not need to manage them manually in `/etc/fstab`.

**Automounting and persistence**

There are several ways to make mounts available on demand or at boot.

- **`/etc/fstab` with `auto`** mounts during boot or when `mount -a` runs.
- **`noauto`** prevents automatic mounting; you mount it manually.
- **`x-systemd.automount`** in `fstab` creates a systemd automount unit so the filesystem mounts on first access.
- **systemd mount units** provide explicit unit-based control over mount behavior.
- **Mount namespaces** let processes see different mount tables, which is useful in containers and isolation scenarios.

Example `fstab` automount entry:

```fstab
UUID=1234-ABCD  /archive  ext4  noauto,x-systemd.automount  0  2
```

How this behaves:

- The mount point exists immediately.
- The filesystem is mounted only when something accesses `/archive`.
- This reduces boot delays for slow or remote storage.

A systemd mount unit example:

```ini
[Unit]
Description=Mount archive volume

[Mount]
What=UUID=1234-ABCD
Where=/archive
Type=ext4
Options=defaults,noatime

[Install]
WantedBy=multi-user.target
```

An automount unit can pair with it to trigger on access. This is different from `noauto`, which simply disables automatic mounting without on-demand activation.

**Testing and troubleshooting**

After editing `/etc/fstab`, validate carefully.

1. Check syntax and identify the mount point.
2. Test the entry with `mount -a`.
3. Verify the result with `findmnt` or `df -Th`.
4. Review logs if the mount fails.

```bash
mount -a
findmnt /data
journalctl -b | grep -i mount
```

Important caveats:

- `mount -a` skips entries marked `noauto`.
- `mount -a` does not fully simulate boot ordering or dependency timing.
- A mount can succeed manually but still fail at boot if the device is not ready early enough.
- For network filesystems, boot timing and network availability matter.

If a mount fails, check:

- the device exists
- the filesystem type is correct
- the mount point directory exists
- the UUID or label matches the actual device
- the options are valid for that filesystem

**Practical examples**

Mount an ext4 data volume by UUID:

```bash
mkdir -p /data
mount UUID=1234-ABCD /data
```

Mount an XFS project volume with quotas:

```bash
mount -t xfs -o prjquota /dev/sdc1 /projects
```

Create and mount a Btrfs filesystem with compression:

```bash
mkfs.btrfs /dev/sdd1
mkdir -p /srv/btrfs
mount -t btrfs -o compress=zstd /dev/sdd1 /srv/btrfs
```

Create a temporary in-memory workspace:

```bash
mkdir -p /tmp/work
mount -t tmpfs -o size=512M,mode=1777 tmpfs /tmp/work
```

Add a persistent mount:

```fstab
UUID=9f3a2c1d-1111-2222-3333-444455556666  /data  ext4  defaults,noatime  0  2
```

**Exam-focused distinctions**

- **Filesystem type** answers “how is the data organized?”
- **Mount option** answers “how should Linux treat this mounted filesystem?”
- **`mkfs`** creates the filesystem.
- **`mount`** attaches it.
- **`fstab`** makes it persistent.
- **`systemd` mount and automount units** are an alternative persistence and activation mechanism.
- **`fsck`** is not universal in the same way across filesystems; know the filesystem-specific repair tool.

**Quick memory cues**

- `ext4` for general use
- `XFS` for large-scale performance
- `Btrfs` for snapshots and compression
- `tmpfs` for RAM-backed temporary storage
- `noexec`, `nosuid`, `nodev` for hardening
- `noauto` to prevent automatic mounting
- `x-systemd.automount` for on-demand mounting
- `UUID=` for stable `fstab` entries
