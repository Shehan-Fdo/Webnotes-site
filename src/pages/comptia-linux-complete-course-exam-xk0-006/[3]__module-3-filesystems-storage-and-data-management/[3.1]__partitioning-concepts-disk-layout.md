## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Partitioning fundamentals**

Disk partitioning divides a physical or virtual disk into logical regions that the operating system can manage independently. In Linux, a partition may contain a filesystem, swap space, an LVM physical volume, or an encrypted container. The partition itself is only the container; formatting and higher-level storage choices happen on top of it.

A useful mental model is:

- **Disk**: the whole block device, such as `/dev/sda` or `/dev/nvme0n1`
- **Partition table**: metadata that describes how the disk is divided
- **Partition**: a slice of the disk, such as `/dev/sda1`
- **Payload**: what lives inside the partition, such as `ext4`, `xfs`, `swap`, LVM, or LUKS encryption

This distinction matters on the exam because a partition is not always directly formatted with a filesystem. For example, `/dev/sda2` might be an LVM physical volume, while `/dev/sda3` might be a swap partition.

**Partition table types: MBR and GPT**

The two common partition table schemes are **MBR** (Master Boot Record) and **GPT** (GUID Partition Table). They differ in capacity, resiliency, and how tools label partitions.

- **MBR** uses a legacy layout with boot code in the first sector and a partition table stored there as well.
- **GPT** uses a modern layout with a primary header at the start of the disk and a backup header at the end.
- **GPT** is preferred for modern systems because it supports more partitions, larger disks, and redundancy.
- **MBR** is still encountered on older systems and in compatibility scenarios.

MBR has a small boot code area at the beginning of the disk. That area can be used by boot loaders, but it is limited and is one reason MBR is less flexible than GPT. GPT includes a **protective MBR** so older tools see the disk as occupied and do not accidentally overwrite it.

Common differences you should recognize:

| Feature | MBR | GPT |
|---|---|---|
| Maximum practical disk size | Limited by 32-bit addressing | Much larger support |
| Number of partitions | Typically 4 primary, or 3 primary + extended | Many partitions |
| Boot metadata | Boot code in first sector | Primary and backup headers |
| Redundancy | Minimal | Stronger, with backup table/header |
| Legacy compatibility | High | Good, with protective MBR |

**Partition type identifiers and labels**

Partition tables store more than just start and end sectors. They also store type information that tells tools and firmware how a partition is intended to be used.

- On **MBR**, the partition type is a one-byte code, often shown as a hex value such as `82` for Linux swap or `83` for Linux filesystem.
- On **GPT**, the partition type is a **GUID** (Globally Unique Identifier), such as the Linux filesystem GUID or the Linux swap GUID.
- Tools may also show a human-readable label or name, especially on GPT.

Examples of how tools display this information:

```bash
sudo fdisk -l /dev/sda
```

Typical output concepts to look for:

- `Id` or `Type` on MBR partitions
- `Type` or `GUID` on GPT partitions
- partition names such as `Linux filesystem`, `Linux swap`, `EFI System`, or `Linux LVM`

With `parted`, you often see a type label rather than a raw code:

```bash
sudo parted /dev/sda print
```

Example output fields:

- `Number`
- `Start`
- `End`
- `Size`
- `File system`
- `Name`
- `Flags`

With `lsblk`, you can inspect type and filesystem-related metadata:

```bash
lsblk -f
```

This is especially useful because `lsblk` distinguishes between the partition type and the filesystem or volume type mounted on it.

> A partition type does not always mean a filesystem. A partition can be marked for swap, LVM, RAID, or encryption rather than formatted directly.

**Alignment and optimal boundaries**

Modern disks and SSDs perform best when partitions start on aligned boundaries, commonly **1 MiB alignment**. Alignment means the partition begins at a sector boundary that matches the underlying device’s physical block size and erase block behavior.

Why alignment matters:

- **Performance**: misaligned partitions can cause extra read-modify-write operations
- **Compatibility**: 1 MiB alignment works well across many disk types and firmware environments
- **SSD longevity**: aligned writes reduce unnecessary wear
- **RAID and advanced format disks**: alignment helps avoid crossing stripe or erase boundaries

Most modern partitioning tools align automatically, but you should still verify it. `parted` often defaults to aligned starts, and `fdisk` on modern systems also tends to create aligned partitions.

Example:

```bash
sudo parted /dev/sda unit MiB print
```

Look for partitions starting at `1.00MiB`, `2.00MiB`, or another aligned boundary rather than at sector 63 or other legacy offsets.

**Common partition roles**

Linux systems often use several special partitions or storage areas, each with a distinct purpose.

- **`/boot`**: stores the kernel, initramfs, and bootloader files
- **EFI System Partition (ESP)**: required for UEFI boot; usually FAT32 and mounted at `/boot/efi`
- **swap**: provides virtual memory backing and hibernation support in some setups
- **root (`/`)**: the main filesystem tree
- **`/home`**: user data, often separated for easier reinstall or resizing
- **LVM PVs**: partitions used as physical volumes for flexible volume management
- **encrypted containers**: partitions that hold LUKS-encrypted data or volumes

`/boot` becomes especially important in encrypted or complex boot setups. If the root filesystem is encrypted, the bootloader and kernel must still be able to load enough files to unlock the system. In many designs, `/boot` remains unencrypted so the system can start, while the root volume is encrypted. On UEFI systems, the ESP is also required for firmware boot files.

**Swap layout choices**

Swap can be implemented as either a **swap partition** or a **swap file**.

- **Swap partition**
  - Dedicated partition marked as swap
  - Simple and predictable
  - Useful in some server or installer-driven layouts
  - Easy for early boot and hibernation scenarios
- **Swap file**
  - Regular file on an existing filesystem
  - Easier to resize or remove
  - Common on modern distributions
  - Convenient when you do not want to reserve a fixed partition

When to prefer each:

- Use a **swap partition** when you want a fixed, simple layout or when your boot/hibernate design benefits from it.
- Use a **swap file** when flexibility matters and the filesystem supports it cleanly.
- Use a **swap partition** more often in minimal or highly controlled deployments.
- Use a **swap file** more often on general-purpose desktops and servers where resizing is likely.

Create and enable a swap file example:

```bash
sudo fallocate -l 4G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
```

Persist it in `/etc/fstab`:

```fstab
/swapfile none swap defaults 0 0
```

Create a swap partition example:

```bash
sudo mkswap /dev/sda3
sudo swapon /dev/sda3
```

**Filesystem versus partition type**

A common exam trap is confusing the partition type with the filesystem inside it.

- A partition type tells the system what the partition is intended for.
- A filesystem tells the OS how data is stored and retrieved.
- Some partitions are not formatted with a filesystem at all.

Examples:

- `/dev/sda1` may be `ext4` and mounted at `/`
- `/dev/sda2` may be `swap` and never mounted
- `/dev/sda3` may be an LVM physical volume and contain no filesystem directly
- `/dev/sda4` may be a LUKS container that holds an encrypted filesystem or LVM stack

Useful inspection commands:

```bash
lsblk -o NAME,FSTYPE,FSVER,TYPE,MOUNTPOINTS,PARTTYPE,PARTLABEL
blkid
sudo fdisk -l /dev/sda
sudo parted /dev/sda print
```

`PARTTYPE` and `PARTLABEL` help you distinguish partition metadata from the filesystem reported in `FSTYPE`.

**LVM as a partitioning design choice**

**LVM** (Logical Volume Manager) adds a flexible storage layer between partitions and filesystems. Instead of formatting a partition directly, you can use it as an LVM physical volume (**PV**), combine PVs into a volume group (**VG**), and create logical volumes (**LVs**) on top.

Why use LVM:

- Resize volumes more easily than fixed partitions
- Combine multiple disks or partitions into one storage pool
- Create snapshots for backup or testing
- Separate storage concerns without rigid partition boundaries

Typical flow:

1. Create a partition intended for LVM.
2. Mark or identify it as an LVM PV.
3. Create a VG from one or more PVs.
4. Create LVs for `/`, `/home`, `var`, or other mount points.
5. Format the LV with a filesystem.

Example:

```bash
sudo pvcreate /dev/sdb1
sudo vgcreate vgdata /dev/sdb1
sudo lvcreate -n lvhome -L 50G vgdata
sudo mkfs.ext4 /dev/vgdata/lvhome
```

This is why partitioning is often a design decision rather than a final storage format. The partition may simply be the foundation for a more flexible storage stack.

**Encryption and partition layout**

Disk encryption is often implemented with **LUKS** (Linux Unified Key Setup). A partition can hold a LUKS container, and inside that container you may place a filesystem directly or build LVM on top of it.

Common patterns:

- **Encrypted root with separate `/boot`**
  - `/boot` remains unencrypted
  - root filesystem is inside LUKS
  - common on laptops and secure systems
- **Encrypted LVM**
  - partition → LUKS → LVM → filesystems
  - flexible and secure
- **Encrypted data partition**
  - partition → LUKS → filesystem
  - useful for sensitive data volumes

Example stack:

```bash
/dev/sda1  ->  /boot
/dev/sda2  ->  LUKS container  ->  LVM PV  ->  VG  ->  LVs  ->  filesystems
```

This layering is important because boot firmware and bootloaders may not be able to read encrypted volumes directly without additional support. That is why `/boot` is often separate in encrypted installations.

**Inspecting and verifying partition changes**

After creating or modifying partitions, you should verify that the kernel and user-space tools see the updated layout. The partition table on disk and the kernel’s in-memory view do not always update at the same moment.

Useful commands:

```bash
sudo partprobe /dev/sda
sudo partx -u /dev/sda
sudo udevadm settle
lsblk
```

What these do:

- `partprobe` asks the kernel to re-read the partition table
- `partx -u` updates the kernel’s partition table entries
- `udevadm settle` waits for device events to finish processing
- `lsblk` shows the current block device tree and often reflects changes after udev settles

If a partition does not appear immediately:

- Re-run `lsblk`
- Check `dmesg` for kernel messages
- Use `partx` or `partprobe`
- Confirm the device is not in use by mounted filesystems or active LVM/RAID layers

Example verification sequence:

```bash
sudo parted /dev/sda print
sudo partprobe /dev/sda
sudo udevadm settle
lsblk -f
```

**Boot-related edge cases**

Some layouts require special attention during installation or recovery.

- **UEFI systems** need an EFI System Partition, usually FAT32, mounted at `/boot/efi`
- **Legacy BIOS systems** may use MBR boot code or, on GPT disks, a small BIOS boot partition for GRUB support
- **Encrypted root** setups often require a separate `/boot`
- **Complex storage stacks** may need `/boot` outside LVM or outside encryption if the bootloader cannot access the full stack

Exam-relevant distinctions:

- GPT stores a **backup header** and partition table at the end of the disk, which helps recovery if the primary header is damaged.
- MBR stores boot code and partition metadata in the first sector, making that area critical and more fragile.
- GPT’s **protective MBR** helps prevent older tools from misidentifying the disk as empty.

**Practical command examples**

Create a GPT partition table and a partition aligned at 1 MiB:

```bash
sudo parted /dev/sdb mklabel gpt
sudo parted -a optimal /dev/sdb mkpart primary ext4 1MiB 100%
sudo parted /dev/sdb print
```

Create an MBR table and a Linux swap partition:

```bash
sudo fdisk /dev/sdc
```

Inside `fdisk`, you would typically:

- create a new DOS/MBR label
- add a partition
- set its type to Linux swap
- write changes

Then verify:

```bash
sudo fdisk -l /dev/sdc
lsblk -f /dev/sdc
```

Create a filesystem on a normal data partition:

```bash
sudo mkfs.xfs /dev/sdb1
sudo mount /dev/sdb1 /mnt/data
```

Create swap and activate it:

```bash
sudo mkswap /dev/sdb2
sudo swapon /dev/sdb2
swapon --show
```

Inspect partition identifiers and labels:

```bash
lsblk -o NAME,TYPE,FSTYPE,PARTTYPE,PARTLABEL,UUID,MOUNTPOINTS
blkid /dev/sdb1 /dev/sdb2
```

**Exam focus and troubleshooting mindset**

When you see a partitioning question, ask what layer the question is really about:

- **Disk layout**: MBR or GPT
- **Partition purpose**: filesystem, swap, LVM, encryption, boot
- **Boot method**: BIOS or UEFI
- **Alignment**: performance and compatibility
- **Verification**: kernel view versus on-disk table

Common mistakes to avoid:

- Treating a partition type as the same thing as a filesystem
- Forgetting that GPT has a backup header
- Forgetting that MBR has boot code in the first sector
- Assuming every partition must be formatted directly
- Ignoring the need for `/boot` or an ESP in certain boot designs
- Overlooking alignment when creating new partitions

A strong answer on the exam usually explains not just what a partition is, but why a given layout is chosen for booting, performance, flexibility, or security.
