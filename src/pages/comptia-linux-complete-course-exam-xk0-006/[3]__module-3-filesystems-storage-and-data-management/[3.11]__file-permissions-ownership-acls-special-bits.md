## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Access control fundamentals**

Linux file access is built on a layered model: ownership, permission bits, ACLs (Access Control Lists), and special bits. You use these controls to decide who can read data, modify files, traverse directories, or execute programs. The model is simple at first glance, but the details matter because directories and files behave differently, and a single incorrect bit can block access or create a security risk.

At the core, every file has:
- an owning **user**
- an owning **group**
- a set of **mode bits** for user, group, and others

You inspect these with `ls -l` and `stat`.

```bash
ls -l /etc/passwd
stat /etc/passwd
```

A typical long listing looks like this:

```text
-rw-r--r-- 1 root root  2483 Apr 26 10:00 /etc/passwd
```

The first character shows the file type:
- `-` regular file
- `d` directory
- `l` symbolic link
- `c` character device
- `b` block device
- `p` named pipe
- `s` socket

The next nine characters are the permission triplets:
- user permissions
- group permissions
- others permissions

**Permission bits**

Linux uses three basic permissions:
- **read** (`r`)
- **write** (`w`)
- **execute** (`x`)

For files, these mean:
- `r`: view file contents
- `w`: modify file contents
- `x`: run the file as a program or script

For directories, they mean something different:
- `r`: list directory contents
- `w`: create, delete, or rename entries in the directory
- `x`: traverse the directory and access files inside it

That directory distinction is important. A directory with `r` but no `x` can often be listed, but you cannot access entries by name. A directory with `x` but no `r` can be traversed if you already know the name, but not listed.

**Ownership and identity**

Every process runs with a user identity and one or more group identities. File access checks compare the process credentials against the file owner, file group, and others class.

- **User owner**: the account that owns the file
- **Group owner**: the group associated with the file
- **Others**: everyone else

You view ownership with:

```bash
ls -l
stat filename
```

You change ownership with:
- `chown` for user and group
- `chgrp` for group only

```bash
sudo chown alice:developers report.txt
sudo chgrp developers report.txt
```

Common patterns:
- `chown alice file` changes the user owner only
- `chown alice:staff file` changes both user and group
- `chown :staff file` changes only the group

> Changing ownership usually requires root privileges or equivalent capabilities.

**Numeric and symbolic modes**

You can set permissions with symbolic or numeric notation.

Symbolic examples:
```bash
chmod u+x script.sh
chmod g-w,o-r file.txt
chmod u=rw,g=r,o= file.txt
```

Numeric examples:
- `7` = `rwx`
- `6` = `rw-`
- `5` = `r-x`
- `4` = `r--`
- `0` = `---`

Examples:
```bash
chmod 644 file.txt
chmod 755 script.sh
chmod 700 private.key
```

Interpretation:
- `644` means owner can read/write, group and others can read
- `755` means owner can read/write/execute, group and others can read/execute
- `700` means only owner has access

A useful mental model is:
- files often use `644` for data files
- executable scripts and binaries often use `755`
- secrets often use `600` or `700`

**Default permissions and umask**

When a process creates a file, the system starts from a default mode and subtracts permissions using the **umask**. The umask is not a direct permission setting; it is a mask that removes bits from the default.

Typical defaults:
- files start from `666` before umask
- directories start from `777` before umask

Common umask values:
- `022` produces files like `644` and directories like `755`
- `027` produces files like `640` and directories like `750`
- `077` produces files like `600` and directories like `700`

Check your current umask with:

```bash
umask
```

Set it temporarily with:

```bash
umask 027
```

For example, with umask `027`:
- a new file becomes `640`
- a new directory becomes `750`

This matters in shared environments because a permissive umask can expose newly created files to unintended readers.

**Directory permissions in practice**

Directory permissions are often the source of troubleshooting confusion. Consider these cases:

- `r` without `x`:
  - you may see names
  - you cannot access entries by path
- `x` without `r`:
  - you can access known names
  - you cannot list the directory
- `w` without `x`:
  - directory modifications are limited and often not useful alone
  - deletion and creation typically require both `w` and `x`

A practical example:

```bash
mkdir testdir
touch testdir/file1
chmod 600 testdir
ls testdir
cat testdir/file1
```

Even if the file inside is readable, lack of directory execute permission prevents traversal.

**Special bits**

Linux has three special permission bits that modify standard behavior:
- **setuid**
- **setgid**
- **sticky bit**

These are often shown in the execute positions of `ls -l`.

**setuid**

When set on an executable, **setuid** causes the program to run with the file owner's effective user ID. This is commonly used for privileged utilities that need limited elevated access.

Example display:
```text
-rwsr-xr-x 1 root root  ... /usr/bin/passwd
```

The `s` in the user execute position means setuid is active.

Set it with:
```bash
chmod u+s /path/to/program
```

Remove it with:
```bash
chmod u-s /path/to/program
```

Use setuid carefully:
- it increases privilege
- bugs in setuid programs can become security issues
- many systems restrict or audit setuid binaries

**setgid**

On executables, **setgid** makes the program run with the file group’s effective group ID.

Example display:
```text
-rwxr-sr-x 1 root mail ... /usr/bin/someprog
```

On directories, setgid has a different and very useful meaning:
- new files and subdirectories inherit the directory’s group
- subdirectories may also inherit the setgid bit

This is common in shared project directories.

Set it with:
```bash
chmod g+s /shared/project
```

Remove it with:
```bash
chmod g-s /shared/project
```

Example shared directory pattern:
```bash
sudo chgrp developers /srv/project
sudo chmod 2775 /srv/project
```

The leading `2` in `2775` sets setgid on the directory. This helps keep group ownership consistent for collaborative work.

**sticky bit**

The **sticky bit** on a directory restricts deletion and renaming so that only the file owner, directory owner, or root can remove entries. This is most commonly used on world-writable directories like `/tmp`.

Example display:
```text
drwxrwxrwt 10 root root ... /tmp
```

The `t` in the others execute position means sticky bit is active.

Set it with:
```bash
chmod +t /shared/dropbox
```

Remove it with:
```bash
chmod -t /shared/dropbox
```

Without the sticky bit, any user with write access to a shared directory could delete another user’s files.

**ACLs**

ACLs provide more granular permissions than the traditional user/group/others model. Use them when the basic mode bits are not enough, such as when you need to grant access to a specific user without changing group structure.

Common ACL commands:
- `getfacl` to view ACLs
- `setfacl` to modify ACLs

View ACLs:
```bash
getfacl file.txt
```

Grant a user read access:
```bash
setfacl -m u:alice:r-- file.txt
```

Grant a group read/write access:
```bash
setfacl -m g:developers:rw- file.txt
```

Remove a specific ACL entry:
```bash
setfacl -x u:alice file.txt
```

Clear all ACLs:
```bash
setfacl -b file.txt
```

A file with ACLs often shows a `+` in `ls -l`:

```text
-rw-r-----+ 1 root developers ... file.txt
```

That `+` indicates extended ACL entries exist.

**ACL mask**

ACLs include a **mask** that limits the effective permissions of named users and groups. This is a common source of confusion because an ACL entry may appear to grant `rw-`, but the mask can reduce the effective result.

Example:
```bash
setfacl -m u:alice:rw- file.txt
setfacl -m m:r-- file.txt
getfacl file.txt
```

If the mask is `r--`, Alice’s effective permissions are limited to read-only even if her entry says `rw-`.

When troubleshooting ACLs:
- check the named entry
- check the mask
- check the base mode bits

**Default ACLs**

Default ACLs apply to newly created files and directories inside a directory. They are useful for collaborative directories where you want inherited access beyond what umask and setgid provide.

Example:
```bash
setfacl -m d:g:developers:rwx /srv/project
setfacl -m d:u:alice:rwx /srv/project
```

View them with:
```bash
getfacl /srv/project
```

Default ACLs are especially useful when:
- multiple users need access
- you want inheritance without manual chmod/chown after each creation
- group ownership alone is not enough

**Practical comparison**

| Mechanism | Best for | Limitation |
|---|---|---|
| Mode bits | Simple owner/group/others control | Not granular enough for exceptions |
| Ownership | Assigning primary control | Does not grant access by itself |
| setgid directory | Shared group collaboration | Still group-based only |
| Sticky bit | Protecting shared writable directories | Does not grant access |
| ACLs | Per-user or per-group exceptions | More complex to audit |

**Common commands and workflows**

Inspect permissions:
```bash
ls -l file
stat file
getfacl file
```

Change mode:
```bash
chmod 640 file
chmod u=rw,g=r,o= file
```

Change owner:
```bash
sudo chown alice:developers file
```

Change group:
```bash
sudo chgrp developers file
```

Set a collaborative directory:
```bash
sudo mkdir -p /srv/team
sudo chgrp developers /srv/team
sudo chmod 2770 /srv/team
sudo setfacl -m d:g:developers:rwx /srv/team
```

Protect a shared drop area:
```bash
sudo mkdir -p /srv/drop
sudo chmod 1777 /srv/drop
```

**Troubleshooting access problems**

When access fails, work through the checks in order:
1. Confirm the file type and path are correct.
2. Check the owner, group, and mode bits with `ls -l`.
3. Check directory execute permission on every parent directory.
4. Check ACLs with `getfacl`.
5. Check whether setuid, setgid, or sticky bit is affecting behavior.
6. Verify the process is running as the expected user and group.

A useful diagnostic sequence:

```bash
namei -l /srv/project/report.txt
ls -ld /srv /srv/project
ls -l /srv/project/report.txt
getfacl /srv/project/report.txt
id
```

`namei -l` is especially helpful because it shows permissions on each path component, not just the final file.

**Security considerations**

Permissions are not just about convenience; they are a primary security boundary on Linux.

- Use the least privilege necessary.
- Prefer group-based access for stable teams.
- Use ACLs for exceptions, not as a replacement for good group design.
- Avoid world-writable directories unless they are intentionally shared and protected with sticky bit.
- Review setuid binaries regularly because they carry elevated risk.
- Keep secrets at `600` or tighter when possible.

> If a file is executable but should not be, remove execute permission immediately. Unnecessary execute bits can create accidental exposure or misuse.

**Exam focus points**

For exam-style questions, remember these high-value distinctions:
- `x` on a directory means traverse, not execute code
- `w` on a directory means create/delete/rename entries
- `setuid` affects the effective user ID of executables
- `setgid` on directories helps group inheritance
- sticky bit on directories prevents users from deleting each other’s files
- ACLs extend the basic permission model and may show a `+` in `ls -l`
- umask subtracts permissions from default creation modes

**Quick reference**

```bash
# View
ls -l
stat file
getfacl file

# Change mode
chmod 644 file
chmod u+x script.sh
chmod 2775 /shared/project
chmod 1777 /tmp-like-dir

# Ownership
chown user:group file
chgrp group file

# ACLs
setfacl -m u:alice:r-- file
setfacl -m d:g:developers:rwx /shared/project
setfacl -b file

# Identity and defaults
id
umask
namei -l /path/to/file
```

**Summary**

Linux access control starts with ownership and mode bits, then expands with ACLs and special bits when you need more precise behavior. You should choose the simplest mechanism that meets the requirement: mode bits for standard access, group ownership for collaboration, ACLs for exceptions, setgid for inherited group ownership, sticky bit for shared writable directories, and setuid only when elevated execution is truly necessary.
