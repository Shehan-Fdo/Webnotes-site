**NFS overview**

<abbr title="Network File System">NFS</abbr> lets a Linux system share directories over the network so remote clients can mount them as if they were local filesystems. It is common in mixed Linux environments for home directories, shared project data, software repositories, and lab storage.

NFS is a client/server protocol:
- The **server** exports directories.
- The **client** mounts exported directories.
- Access control is enforced by export rules, filesystem permissions, and identity mapping.

NFS is usually preferred when you want:
- Centralized storage with simple Unix-style permissions
- Transparent access from multiple hosts
- Shared read/write data without copying files between systems

**NFS versions and protocol behavior**

NFSv3 and NFSv4 differ in how they use supporting services and how they present exports.

- **NFSv3** often relies on auxiliary RPC services such as `rpcbind`, `mountd`, `lockd`, and `statd`.
- **NFSv4** is more self-contained and can work with fewer auxiliary daemons because it uses a more integrated protocol model.
- **NFSv4** introduces a pseudo-filesystem namespace, so exports may appear under a single root-like hierarchy.

Practical exam point:
- If a server is using NFSv3, you usually check more supporting services.
- If a server is using NFSv4, the service stack is simpler, but export path design and permissions still matter.

**Server-side setup**

On the server, you export directories by defining them in `/etc/exports` and then applying the export table.

Common service names vary by distribution:
- On many systemd-based distributions, the main service is `nfs-server`.
- `rpcbind` is commonly required for NFSv3 and related RPC services.
- Some distributions start NFS-related daemons automatically when you enable the server service.

Typical service checks:
```bash
systemctl status nfs-server
systemctl status rpcbind
systemctl enable --now nfs-server
systemctl enable --now rpcbind
```

On some systems, the service name may differ slightly, but the exam-relevant idea is:
- Verify the NFS server is running.
- Verify required dependencies are active.
- Confirm the export table is loaded.

Useful server commands:
```bash
exportfs -v
exportfs -ra
showmount -e localhost
```

- `exportfs -v` shows active exports and options.
- `exportfs -ra` re-exports all entries from `/etc/exports`.
- `showmount -e` lists exported directories, especially useful for NFSv3 visibility checks.

> If you change `/etc/exports`, you usually need to re-export or reload the configuration before clients see the update.

**`/etc/exports` syntax**

The `/etc/exports` file defines which directories are shared and which clients may access them. Each line contains a shared path followed by one or more client specifications and options.

Basic example:
```bash
/srv/nfs/projects 192.168.1.0/24(rw,sync,no_subtree_check)
```

Examples with different client specifications:
```bash
/srv/nfs/data client1.example.com(ro,sync)
/srv/nfs/data *.example.com(rw,sync)
/srv/nfs/data 192.168.10.25(rw,sync) 192.168.10.26(rw,sync)
/srv/nfs/home 192.168.20.0/24(rw,sync,no_root_squash)
```

What the syntax means:
- A **hostname** grants access to a specific host.
- A **wildcard** such as `*.example.com` matches multiple hosts.
- A **CIDR subnet** such as `192.168.1.0/24` grants access to a network range.
- Multiple client specs on one line let you mix hostnames, subnets, and individual IP addresses.

Common export options:
- `rw` allows read/write access.
- `ro` allows read-only access.
- `sync` writes changes to disk before replying, improving safety.
- `async` may improve performance but increases risk if the server crashes.
- `root_squash` maps remote root to an unprivileged user, which is the safer default.
- `no_root_squash` preserves remote root privileges and is risky unless you fully trust the client.
- `subtree_check` verifies file location within exported subtrees but can add overhead.
- `no_subtree_check` is often used to avoid subtree-related issues.

**Export traversal and nested exports**

When you export parent and child directories, traversal behavior matters.

- `nohide` can make a mounted child export appear under its parent in some NFSv3 scenarios.
- `crossmnt` is more relevant to NFSv4 namespace traversal and allows crossing mount points within an exported tree.

Practical distinction:
- Use `nohide` when you need NFSv3 clients to see nested exports more naturally.
- Use `crossmnt` when designing an NFSv4 export tree that should allow traversal across mount boundaries.

**Applying and removing exports**

After editing `/etc/exports`, reload the export table:
```bash
exportfs -ra
```

To unexport a share:
```bash
exportfs -u /srv/nfs/projects
```

You can also unexport all shares:
```bash
exportfs -ua
```

This is useful when you need to temporarily remove access without deleting the configuration file.

**Client-side mounting**

On the client, you mount the exported directory using `mount` or `/etc/fstab` for persistence.

Temporary mount:
```bash
mount -t nfs server1:/srv/nfs/projects /mnt/projects
```

NFSv4 example:
```bash
mount -t nfs4 server1:/export/projects /mnt/projects
```

Persistent mount in `/etc/fstab`:
```fstab
server1:/srv/nfs/projects  /mnt/projects  nfs  defaults,_netdev  0  0
```

Notes:
- `_netdev` tells the system the mount depends on networking.
- Use `nfs4` or `nfs` depending on your environment and distro defaults.
- If the mount should wait for the network, systemd-based systems may also use automount behavior.

Useful client checks:
```bash
mount | grep nfs
findmnt /mnt/projects
df -hT /mnt/projects
```

**Automount with `autofs`**

`autofs` mounts NFS shares on demand instead of keeping them mounted all the time. This reduces boot delays and avoids stale mounts when servers are unavailable.

Typical use cases:
- Home directories
- Infrequently used shared storage
- Large environments with many mounts

Example `/etc/auto.master` entry:
```bash
/misc  /etc/auto.misc
```

Example `/etc/auto.misc` map:
```bash
projects  -fstype=nfs4,rw  server1:/srv/nfs/projects
data      -fstype=nfs,rw   server1:/srv/nfs/data
```

Service management:
```bash
systemctl enable --now autofs
systemctl status autofs
```

With `autofs`, the mount appears when a user accesses the path and disappears after inactivity.

**Permissions and identity mapping**

NFS permissions are often confusing because access depends on both server-side filesystem permissions and matching user identities.

Key rule:
- NFS does not magically translate usernames across systems.
- It relies heavily on numeric `UID` and `GID` values.

Troubleshooting checklist:
- Confirm the user has the same `UID` and `GID` on client and server.
- Check directory ownership and mode bits on the server.
- Verify export options such as `root_squash`.
- Confirm the client is mounting the expected export path.

Example:
- If user `alice` has `UID 1001` on the client but `UID 2001` on the server, she may see permission denied even if the username matches.
- Shared groups work best when group IDs are consistent across systems.

Server-side permission example:
```bash
chown -R alice:shared /srv/nfs/projects
chmod 2775 /srv/nfs/projects
```

The setgid bit (`2` in `2775`) helps new files inherit the directory group, which is useful for shared team directories.

**Security and authentication**

Basic NFS exports often rely on host-based trust and Unix permissions, but stronger security is available.

Kerberos-based security options:
- `sec=krb5` provides Kerberos authentication only.
- `sec=krb5i` provides authentication plus integrity protection.
- `sec=krb5p` provides authentication, integrity, and privacy through encryption.

Example mount:
```bash
mount -t nfs4 -o sec=krb5p server1:/srv/nfs/secure /mnt/secure
```

When to use each:
- Use `krb5` when you need identity verification but not message protection.
- Use `krb5i` when you want to detect tampering.
- Use `krb5p` when you need confidentiality for sensitive data.

Other security points:
- `root_squash` is a common safety measure on exports.
- `no_root_squash` should be limited to trusted administrative scenarios.
- Firewalls must allow the required NFS ports and related RPC services, especially for NFSv3.

**Troubleshooting NFS**

A good troubleshooting approach separates server issues from client issues.

Server-side checks:
- Is `nfs-server` running?
- Is `rpcbind` running when needed?
- Is the directory exported?
- Are firewall rules allowing NFS traffic?
- Are the underlying filesystem permissions correct?

Client-side checks:
- Is the server reachable by name and IP?
- Is the mount path correct?
- Is the client using the expected NFS version?
- Are local permissions or UID/GID mismatches causing access failures?

Useful commands:
```bash
showmount -e server1
rpcinfo -p server1
mount -v -t nfs server1:/srv/nfs/projects /mnt/projects
```

`rpcinfo -p` is especially useful for NFSv3-related service discovery.

**`nfsstat` for performance and behavior**

`nfsstat` shows NFS client and server statistics and helps you spot performance or mount behavior issues.

Examples:
```bash
nfsstat
nfsstat -c
nfsstat -s
nfsstat -m
```

What to look for:
- High retransmissions may indicate network problems.
- Unexpected mount options may explain performance or locking behavior.
- Client and server counters can reveal whether requests are succeeding or being retried.

This is useful when a mount works but feels slow, unstable, or inconsistent.

**Real-world workflow**

A common admin workflow looks like this:
1. Create the shared directory on the server.
2. Set ownership and permissions.
3. Add the export to `/etc/exports`.
4. Reload exports with `exportfs -ra`.
5. Verify with `exportfs -v` or `showmount -e`.
6. Mount the share on the client.
7. Test read/write access.
8. Use `nfsstat` and logs if performance or access issues appear.

Example server setup:
```bash
mkdir -p /srv/nfs/projects
chown -R root:shared /srv/nfs/projects
chmod 2775 /srv/nfs/projects
echo "/srv/nfs/projects 192.168.1.0/24(rw,sync,no_subtree_check)" >> /etc/exports
exportfs -ra
systemctl enable --now nfs-server
```

Example client test:
```bash
mkdir -p /mnt/projects
mount -t nfs server1:/srv/nfs/projects /mnt/projects
touch /mnt/projects/testfile
ls -l /mnt/projects
```

**Exam-focused distinctions**

- **Server export control** happens in `/etc/exports`.
- **Client mount control** happens with `mount`, `/etc/fstab`, or `autofs`.
- **NFSv3** usually needs more helper services than NFSv4.
- **Kerberos** strengthens NFS authentication and can add integrity or privacy.
- **UID/GID consistency** is often the real reason a mount “works” but access fails.
- **`exportfs -u`** removes an export without deleting the config line.
- **`nfsstat`** helps diagnose usage patterns and performance issues.
