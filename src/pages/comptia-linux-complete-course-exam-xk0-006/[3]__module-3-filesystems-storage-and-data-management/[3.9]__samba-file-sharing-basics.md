## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Samba overview**

Samba is the standard open-source implementation of the <abbr title="Server Message Block">SMB</abbr> protocol family on Linux and Unix-like systems. It lets Linux systems share files and printers with Windows clients and also access SMB shares hosted by Windows or other SMB servers.

At a practical level, Samba solves two common problems:

- You want Linux directories to appear as network shares for Windows, macOS, and Linux clients.
- You want Linux clients to mount or browse shares hosted by Windows file servers.

Samba is not just one daemon. It is a suite of components that may include file-serving daemons, authentication helpers, name-resolution support, and administrative tools. Package and service names vary by distribution, so always verify the local layout with your system’s package manager and service manager.

**Core SMB concepts**

SMB is the network protocol used for shared files, printers, and some related services. Modern SMB traffic usually uses TCP port `445`. Legacy NetBIOS-based browsing may also involve ports `137`, `138`, and `139`, but that older model is increasingly disabled in modern environments.

Key terms:

- **SMB**: The protocol used for network file and printer sharing.
- **CIFS**: Common Internet File System, an older Microsoft dialect name often used loosely for SMB.
- **NetBIOS**: Legacy name service and browsing support that predates modern direct-host SMB access.
- **Workgroup**: A simple peer-to-peer grouping used in small SMB environments.
- **Domain**: A centralized authentication environment, often backed by Active Directory.

Modern clients usually connect directly to `\\server\share` over port `445` without relying on NetBIOS browsing. That means basic SMB access can work even when legacy browsing services are absent.

**Typical Samba components**

The exact service names depend on the distribution, but the functional roles are consistent:

- **File-serving daemon**: Handles SMB file and printer requests.
- **Name service daemon**: Supports legacy NetBIOS name resolution and browsing when enabled.
- **Client tools**: Let you list shares, test authentication, and mount remote shares.
- **Configuration file**: Usually `/etc/samba/smb.conf`.

Common tools include:

- `smbclient` for interactive access and share listing
- `testparm` for validating Samba configuration
- `mount.cifs` for mounting SMB shares on Linux
- `smbstatus` for viewing active Samba sessions and locks
- `pdbedit` for Samba user database management in some setups

**Basic server configuration**

A Samba server typically defines global settings and one or more shares in `/etc/samba/smb.conf`. A minimal configuration might look like this:

```ini
[global]
    workgroup = WORKGROUP
    server string = Linux Samba Server
    security = user
    map to guest = Bad User
    log file = /var/log/samba/%m.log
    max log size = 1000

[public]
    path = /srv/samba/public
    browseable = yes
    read only = no
    guest ok = yes
```

Important settings:

- **`workgroup`** should match the expected SMB workgroup or domain context for simple environments.
- **`security = user`** is the common mode for authenticated access.
- **`map to guest = Bad User`** can map unknown users to guest access when you intentionally allow it.
- **`path`** points to the exported directory on the Linux host.
- **`browseable`** controls whether the share appears in network browsing lists.
- **`read only = no`** allows writes when permissions also permit them.
- **`guest ok = yes`** allows guest access to that share when combined with the broader guest policy.

> Guest access is not controlled by `guest ok` alone. It also depends on `map to guest`, the guest account mapping, and filesystem permissions on the exported path.

**Share types and access models**

Samba shares are usually configured as either authenticated or guest-accessible. You should choose the model based on the data sensitivity and the client population.

- **Authenticated shares**
  - Require a valid Samba user account.
  - Are appropriate for private team data.
  - Usually map to Linux filesystem permissions and ownership.

- **Guest shares**
  - Allow access without a username and password.
  - Are useful for public drop zones or read-only content.
  - Should be tightly scoped and carefully permissioned.

A common guest-oriented share example:

```ini
[public]
    path = /srv/samba/public
    browseable = yes
    read only = yes
    guest ok = yes
    guest only = yes
```

For this to work reliably, you still need the underlying directory permissions to allow the guest-mapped account to read the files. If the filesystem denies access, Samba cannot override it.

**User and password management**

Samba can use local Linux users for authentication, but Samba also maintains its own password database in many configurations. A common workflow is:

1. Create or verify a Linux user.
2. Add that user to Samba’s password database.
3. Enable the account for SMB access.

Example:

```bash
sudo useradd -m alice
sudo passwd alice
sudo smbpasswd -a alice
sudo smbpasswd -e alice
```

Notes:

- `smbpasswd -a` adds the user to Samba’s password database.
- `smbpasswd -e` enables the Samba account.
- The Linux account and Samba account usually share the same username, but they are not the same database.

In some environments, Samba integrates with centralized identity sources such as LDAP or Active Directory. For Linux+ basics, focus on local-user authentication and the relationship between Samba credentials and filesystem permissions.

**Filesystem permissions and ownership**

Samba does not replace Linux permissions. It sits on top of them. A user may authenticate successfully and still be denied if the directory permissions do not allow the requested action.

Typical permission strategy:

- Create a dedicated directory such as `/srv/samba/projects`.
- Assign a group that represents the share’s users.
- Grant group write access if the share is collaborative.
- Use setgid on the directory so new files inherit the group.

Example:

```bash
sudo mkdir -p /srv/samba/projects
sudo groupadd smbteam
sudo usermod -aG smbteam alice
sudo chown -R root:smbteam /srv/samba/projects
sudo chmod 2770 /srv/samba/projects
```

Why this matters:

- **Ownership** determines who controls the directory.
- **Group membership** lets multiple users collaborate.
- **Setgid** on a directory helps keep new files in the same group.
- **Mode bits** still govern read, write, and execute access.

If you need more granular control, use ACLs (Access Control Lists):

```bash
sudo setfacl -m u:alice:rwx /srv/samba/projects
sudo getfacl /srv/samba/projects
```

**SELinux considerations**

On SELinux-enabled systems, Samba access can fail even when Unix permissions look correct. The preferred approach is usually to label the shared directory with a Samba file context rather than immediately enabling broad booleans.

Example labeling workflow:

```bash
sudo semanage fcontext -a -t samba_share_t "/srv/samba/projects(/.*)?"
sudo restorecon -Rv /srv/samba/projects
```

Useful booleans exist for broader access patterns, but they are not the default fix for ordinary shares:

```bash
sudo setsebool -P samba_export_all_ro on
sudo setsebool -P samba_export_all_rw on
```

Use these booleans carefully:

- They can simplify access to nonstandard paths.
- They are broader than a normal directory label.
- They are better treated as exceptions, not the first choice.

**Firewall and network ports**

For modern SMB access, TCP `445` is the most important port. Legacy browsing may also use `137/udp`, `138/udp`, and `139/tcp`, but many environments no longer rely on them.

Common firewall examples:

```bash
sudo firewall-cmd --permanent --add-service=samba
sudo firewall-cmd --reload
```

Or with UFW:

```bash
sudo ufw allow Samba
```

Remember:

- Opening the firewall is different from verifying that a service is listening.
- A local socket check does not prove remote reachability.
- You should test from a client host when validating network access.

**Service management**

Service names vary by distribution. Some systems use `smb` and `nmb`, while others use `smbd` and `nmbd`. Do not assume every distro uses the same unit names.

General approach:

```bash
systemctl status smb
systemctl status smbd
systemctl status nmb
systemctl status nmbd
```

Enable the appropriate services for your platform:

```bash
sudo systemctl enable --now smb
sudo systemctl enable --now smbd
```

Use the names that exist on the target system. The important exam concept is not the exact unit name, but the role each service plays.

> `nmbd` is legacy NetBIOS name service support. It is not required for basic SMB file access on modern networks, and many environments disable NetBIOS browsing entirely.

**Client-side access and testing**

Use `smbclient` to test connectivity and enumerate shares. When listing shares, point it at the server, not a specific share path:

```bash
smbclient -L //server -U alice
```

To connect directly to a share:

```bash
smbclient //server/projects -U alice
```

Inside the interactive prompt, common commands include:

```text
ls
cd subdir
get report.txt
put notes.txt
exit
```

Mounting a share on Linux uses the CIFS filesystem driver:

```bash
sudo mkdir -p /mnt/projects
sudo mount -t cifs //server/projects /mnt/projects -o username=alice
```

A more complete example with a credentials file:

```bash
sudo nano /root/.smbcred
```

```ini
username=alice
password=SecretPass123
domain=WORKGROUP
```

```bash
sudo chmod 600 /root/.smbcred
sudo mount -t cifs //server/projects /mnt/projects -o credentials=/root/.smbcred,vers=3.0
```

Why use a credentials file:

- It avoids exposing passwords in shell history.
- It is easier to reuse in `/etc/fstab`.
- It supports cleaner automation.

Example `/etc/fstab` entry:

```fstab
//server/projects  /mnt/projects  cifs  credentials=/root/.smbcred,vers=3.0,_netdev  0  0
```

**Troubleshooting workflow**

Start with the simplest checks and move outward from the server to the client.

1. Verify the Samba configuration syntax.
   ```bash
   testparm
   ```
2. Confirm the service is running.
   ```bash
   systemctl status smb
   systemctl status smbd
   ```
3. Check listening sockets on the server.
   ```bash
   ss -tulpn | grep -E ':(445|139)\b'
   ```
4. Confirm the share exists and the path is correct.
   ```bash
   ls -ld /srv/samba/projects
   ```
5. Check filesystem permissions and SELinux labels.
   ```bash
   ls -Zd /srv/samba/projects
   getfacl /srv/samba/projects
   ```
6. Test from a client with `smbclient`.
   ```bash
   smbclient -L //server -U alice
   ```

Useful logs and diagnostics:

- Samba logs often live under `/var/log/samba/`.
- `journalctl -u smb` or `journalctl -u smbd` can show service errors.
- `smbstatus` shows active sessions, open files, and locks on the server.

Common failure patterns:

- **Authentication failure**: Wrong Samba password, disabled account, or username mismatch.
- **Access denied**: Linux permissions, ACLs, or SELinux context problem.
- **Share not visible**: `browseable = no`, browsing disabled, or legacy discovery assumptions.
- **Connection refused**: Service not running or firewall blocking the port.
- **Name resolution failure**: DNS, `/etc/hosts`, or legacy NetBIOS issues.

**Printer sharing basics**

Samba can also share printers, though modern Linux printing often integrates with CUPS (Common Unix Printing System). Traditional Samba printer shares may still appear in older configurations.

A classic example:

```ini
[printers]
    comment = All Printers
    path = /var/spool/samba
    printable = yes
    browseable = no
```

Important caveats:

- This is a traditional pattern and may not match every modern deployment.
- Many current systems rely on CUPS integration rather than a simple spool directory.
- Printer sharing often requires both Samba and printing subsystem configuration.

**Practical share design patterns**

Use the share model that matches the data and the users.

- **Public read-only share**
  - Good for software installers, documents, or reference files.
  - Use `guest ok = yes` and `read only = yes`.
  - Keep the directory world-readable or guest-readable as appropriate.

- **Team collaboration share**
  - Good for departmental file exchange.
  - Use authenticated users, a shared group, and group write permissions.
  - Consider ACLs for exceptions.

- **Drop box share**
  - Good for uploads where users should not read each other’s files.
  - Use write permissions carefully and consider separate upload workflows.
  - Test file creation and ownership behavior.

Example collaborative share:

```ini
[projects]
    path = /srv/samba/projects
    browseable = yes
    read only = no
    valid users = @smbteam
    force group = smbteam
    create mask = 0660
    directory mask = 2770
```

Why these options matter:

- **`valid users`** restricts access to a defined set.
- **`force group`** helps keep files in the collaboration group.
- **`create mask`** limits file permissions on creation.
- **`directory mask`** controls new directory permissions.

**Exam focus points**

For Linux+ study, focus on the relationship between Samba, Linux permissions, and network access.

- Samba provides SMB file and printer sharing for heterogeneous networks.
- `smbclient` is useful for listing and testing shares.
- `mount -t cifs` is the standard Linux client mount method.
- `testparm` validates `smb.conf`.
- Filesystem permissions and SELinux still apply.
- TCP `445` is the primary modern SMB port.
- Legacy NetBIOS browsing exists but is not required for basic SMB access.

**Quick reference commands**

```bash
testparm
smbclient -L //server -U alice
smbclient //server/projects -U alice
smbstatus
mount -t cifs //server/projects /mnt/projects -o credentials=/root/.smbcred,vers=3.0
journalctl -u smb
```

**Common configuration file locations**

- `/etc/samba/smb.conf` for Samba server configuration
- `/var/log/samba/` for logs on many systems
- `/srv/samba/` or `/home/share/` for exported data directories
- `/etc/fstab` for persistent client mounts

**Summary**

Samba bridges Linux and SMB-based clients by exporting directories and printers over the network. The most important exam ideas are share configuration, authentication, filesystem permissions, SELinux labeling, and basic client testing. If you remember that Samba does not bypass Linux security controls, you will avoid many common troubleshooting mistakes.
