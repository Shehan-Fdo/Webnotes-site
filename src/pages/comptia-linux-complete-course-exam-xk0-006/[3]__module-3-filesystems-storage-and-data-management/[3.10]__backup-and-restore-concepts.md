**Backup fundamentals**

A backup is a separate copy of data that you can use to recover from deletion, corruption, ransomware, hardware failure, or human error. In Linux environments, backups protect not only user files but also configuration, package state, databases, and service data.

A good backup plan answers four questions:

- **What** do you protect?
- **How often** do you protect it?
- **Where** do you store it?
- **How** do you restore it when needed?

Backups are only useful if you can restore them. That means you should design for recovery, not just for storage.

**Common backup goals**

Different systems need different recovery objectives. Two terms appear often in backup planning:

- **RPO**: Recovery Point Objective, the maximum acceptable data loss measured in time.
- **RTO**: Recovery Time Objective, the maximum acceptable downtime before service must be restored.

These goals influence backup frequency, retention, and storage location. For example, a database with a 15-minute RPO needs much more frequent backups or replication than a file server with a daily RPO.

**Backup types**

Backups are commonly classified by how much data they capture and how they track change over time.

| Type | What it captures | Strengths | Tradeoffs |
|---|---|---|---|
| Full | All selected data | Simple restores, self-contained | Slowest and largest |
| Incremental | Changes since the last backup of any type | Fastest and smallest backups | Restore may require multiple backup sets |
| Differential | Changes since the last full backup | Faster restore than incremental | Grows larger until next full backup |
| Mirror / sync | Current state of source data | Easy to browse and restore files | Can propagate deletions and corruption if not versioned |
| Snapshot | Point-in-time view of a filesystem, volume, or storage layer | Fast creation, useful for short-term rollback | Not a substitute for off-host backup by itself |

A snapshot can be part of a backup strategy, but it is not enough on its own if it stays on the same system or storage array. If the host, array, or site fails, the snapshot may fail with it.

**Backup scope and what to include**

A complete Linux backup plan usually includes more than `/home`.

- **User data**: documents, project files, mail, and application data.
- **System configuration**: `/etc`, service unit overrides, cron jobs, network config, SSH configuration, and firewall rules.
- **Application state**: databases, web content, container definitions, and queue data.
- **Package inventory**: installed packages and repositories, so you can rebuild the system.
- **Boot and recovery data**: bootloader configuration, initramfs, and partition layout when needed.
- **Secrets and keys**: SSH host keys, TLS certificates, and application credentials, handled carefully.

> Back up secrets only when you have a secure storage and access-control plan. A backup that exposes private keys or credentials can create a security incident.

**Backup media and destinations**

A backup should be stored separately from the original data. Common destinations include:

- **Local disk** for fast restores and short-term retention.
- **External USB storage** for portable offline copies.
- **Network storage** such as NFS, SMB, or an SSH-accessible backup server.
- **Object storage** for offsite retention and immutability features.
- **Tape** for long-term archival in some enterprise environments.

A practical design often follows the 3-2-1 rule:

- Keep **3** copies of data.
- Store them on **2** different media types.
- Keep **1** copy offsite or offline.

**Backup tools and approaches**

Linux offers many ways to back up data. The right tool depends on whether you need file-level copies, archive files, synchronization, or disk images.

**`tar` archives**

`tar` bundles files and directories into one archive. It is common for configuration backups and portable restores.

```bash
tar -czf /backups/etc-2026-04-26.tar.gz /etc
```

This creates a compressed archive of `/etc`. By default, GNU `tar` often strips the leading `/` from stored paths unless you use `-P` or `--absolute-names`, so restores usually extract into the current directory or a chosen target path.

To preserve absolute paths, use:

```bash
tar -czPf /backups/etc-2026-04-26.tar.gz /etc
```

Useful options include:

- `-c` create archive
- `-x` extract archive
- `-t` list archive contents
- `-f` specify archive file
- `-z` gzip compression
- `-J` xz compression
- `-p` preserve permissions on extract
- `-P` preserve absolute paths

Example restore into a staging directory:

```bash
mkdir -p /restore-test
tar -xzf /backups/etc-2026-04-26.tar.gz -C /restore-test
```

**`rsync` file synchronization**

`rsync` is useful for mirroring directories and performing efficient file transfers. It transfers changed files and can use a delta algorithm to send only changed blocks within files during transfer. It is not a block-level backup tool.

```bash
rsync -aHAX --delete /home/backupuser/ /mnt/backup/home/
```

Common flags:

- `-a` archive mode
- `-H` preserve hard links
- `-A` preserve ACLs
- `-X` preserve extended attributes
- `--delete` remove files from destination that no longer exist at source

> Be careful with `--delete`. It can remove files from the destination and is dangerous if the source path is wrong.

A restore should be done offline or in a carefully controlled maintenance window to avoid overwriting live data unexpectedly:

```bash
rsync -aHAX /mnt/backup/home/ /home/
```

**`dd` disk imaging**

`dd` copies raw blocks from one device or file to another. It is useful for full disk images, boot media cloning, and forensic-style captures.

```bash
dd if=/dev/sda of=/backups/sda.img bs=64K status=progress conv=noerror,sync
```

Important considerations:

- A live mounted filesystem can produce an inconsistent image unless it is quiesced, unmounted, or snapshotted first.
- `dd` copies everything, including empty space, so images can be large.
- `conv=noerror,sync` helps continue past read errors and pad blocks, but it does not make a damaged disk fully reliable.

You can restore an image with:

```bash
dd if=/backups/sda.img of=/dev/sda bs=64K status=progress
```

Use extreme caution: this overwrites the target device completely.

**Filesystem snapshots**

Snapshots capture a point-in-time view of a filesystem or volume. They are fast and useful for short-term rollback, especially before upgrades or risky changes.

Common snapshot technologies include:

- **LVM** snapshots
- **Btrfs** snapshots
- **ZFS** snapshots

Example LVM snapshot creation:

```bash
lvcreate -L 2G -s -n root_snap /dev/vg0/root
```

Example Btrfs snapshot:

```bash
btrfs subvolume snapshot -r /mnt/data /mnt/data_snap
```

Snapshots are valuable when you need quick rollback, but they do not replace off-host backups. If the underlying storage fails, snapshots on that storage are lost too.

**Databases and application-aware backups**

File copies alone may not be enough for databases because open files can change during backup. Use application-aware methods when possible.

Examples:

- **MySQL/MariaDB**: use logical dumps with `mysqldump` or physical backup tools designed for the engine.
- **PostgreSQL**: use `pg_dump` for logical backups or `pg_basebackup` for physical backups.
- **SQLite**: use the database’s backup mode or a consistent copy method.

Example MySQL dump:

```bash
mysqldump -u root -p --all-databases > /backups/all-databases.sql
```

Example PostgreSQL dump:

```bash
pg_dump -U postgres mydb > /backups/mydb.sql
```

These backups are often easier to restore than raw file copies because they capture consistent database state.

**Compression and archiving**

Compression reduces storage use and transfer time, but it adds CPU overhead. Choose based on your environment:

- **gzip**: fast, widely supported
- **bzip2**: better compression, slower
- **xz**: high compression, slower
- **zstd**: modern balance of speed and compression, where available

Example with `tar` and `zstd`:

```bash
tar --zstd -cpf /backups/home-2026-04-26.tar.zst /home
```

Compression is most useful for text-heavy data and configuration files. It may provide little benefit for already compressed media such as JPEG, MP4, or many database files.

**Verification and integrity**

A backup is not trustworthy until you verify it. Verification can include checksum validation, archive listing, and test restores.

Create a checksum:

```bash
sha256sum /backups/etc-2026-04-26.tar.gz > /backups/etc-2026-04-26.tar.gz.sha256
```

Verify it later:

```bash
sha256sum -c /backups/etc-2026-04-26.tar.gz.sha256
```

The checksum file must reference the exact filename that was hashed. If you rename or move the archive, the verification file may no longer match unless you update it accordingly.

Other verification methods:

- Compare file counts and sizes.
- List archive contents with `tar -tf`.
- Perform a restore test into a scratch directory.
- Validate application-level consistency after restore.

Example archive inspection:

```bash
tar -tzf /backups/etc-2026-04-26.tar.gz | head
```

**Restore planning**

A restore plan should be documented before disaster strikes. You should know what to restore first, where to restore it, and how to validate success.

A typical restore order is:

1. Restore storage or base OS if the system is lost.
2. Restore configuration and service definitions.
3. Restore application data.
4. Restore user data.
5. Validate services, permissions, and logs.

For a file-level restore, extract only what you need:

```bash
tar -xzf /backups/etc-2026-04-26.tar.gz -C /restore-test etc/ssh/sshd_config
```

For a directory mirror restore, use `rsync` carefully and confirm the source and destination paths before running it.

For a disk image restore, ensure the target device is correct and unmounted:

```bash
lsblk
umount /dev/sdb1
dd if=/backups/sda.img of=/dev/sdb bs=64K status=progress
```

**Disaster recovery concepts**

Disaster recovery is broader than backup. It includes the people, procedures, and infrastructure needed to recover service after a major outage.

Key ideas:

- **Offsite copies** protect against fire, theft, and site-wide failure.
- **Offline copies** help defend against ransomware and accidental deletion.
- **Immutability** prevents backups from being altered or deleted during a retention window.
- **Retention policy** defines how long backups are kept.
- **Versioning** lets you recover older states after silent corruption or delayed detection.

A backup strategy should support both short-term rollback and long-term recovery. For example, snapshots may help you recover from a bad update today, while offsite archives help you recover from a disaster next month.

**Automation and scheduling**

Backups should run automatically and consistently. Common scheduling tools include `cron`, `systemd` timers, and enterprise backup software.

Example `cron` entry:

```cron
0 2 * * * /usr/local/bin/backup-home.sh
```

Example `systemd` timer approach:

```ini
[Unit]
Description=Nightly backup timer

[Timer]
OnCalendar=*-*-* 02:00:00
Persistent=true

[Install]
WantedBy=timers.target
```

Automation reduces human error, but you still need monitoring. A backup job that fails silently is almost as bad as no backup at all.

**Practical backup script example**

A simple backup script can archive a directory, create a checksum, and log the result.

```bash
#!/usr/bin/env bash
set -euo pipefail

SRC="/etc"
DEST="/backups"
DATE="$(date +%F)"
ARCHIVE="$DEST/etc-$DATE.tar.gz"

mkdir -p "$DEST"
tar -czf "$ARCHIVE" "$SRC"
sha256sum "$ARCHIVE" > "$ARCHIVE.sha256"
```

This script is simple, but in production you would usually add:

- logging
- error handling
- retention cleanup
- notification on failure
- offsite replication

**Common mistakes**

- Backing up only `/home` and forgetting `/etc` and application data.
- Assuming a snapshot is a complete backup strategy.
- Restoring without testing in a safe environment first.
- Using `rsync --delete` without confirming the source path.
- Imaging a live disk without considering filesystem consistency.
- Storing backups on the same host as the original data.
- Failing to verify checksums or perform test restores.

**Exam focus**

For Linux+ exam purposes, know how to:

- Distinguish full, incremental, differential, and mirror-style backups.
- Explain why snapshots are useful but not sufficient alone for offsite protection.
- Choose between `tar`, `rsync`, `dd`, and database-specific tools.
- Verify backups with checksums and test restores.
- Recognize the importance of RPO, RTO, retention, and offsite storage.
- Restore data safely without overwriting live systems unintentionally.

**Quick command reference**

```bash
# Archive /etc
tar -czf /backups/etc.tar.gz /etc

# List archive contents
tar -tzf /backups/etc.tar.gz

# Mirror a directory
rsync -aHAX /home/ /mnt/backup/home/

# Create a disk image
dd if=/dev/sda of=/backups/sda.img bs=64K status=progress conv=noerror,sync

# Check archive integrity
sha256sum -c /backups/etc.tar.gz.sha256
```
