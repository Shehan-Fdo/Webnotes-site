**RAID fundamentals**

<abbr title="Redundant Array of Independent Disks">RAID</abbr> combines multiple physical disks into a single logical storage unit to improve performance, availability, or both. In Linux, you typically implement software RAID with `mdadm`, while hardware RAID uses a controller that presents one logical disk to the operating system.

RAID is not a backup strategy. It protects against some disk failures, but it does not protect against accidental deletion, ransomware, corruption, or site loss. Use RAID to improve uptime and storage resilience, then pair it with backups and monitoring.

- **Availability**: RAID can keep data online after a disk failure.
- **Performance**: Some RAID levels improve read and/or write throughput.
- **Capacity tradeoff**: Redundancy reduces usable space.
- **Rebuild risk**: Large arrays can take time to resync after a failure, increasing exposure during rebuild.

**Why RAID matters in Linux administration**

You use RAID when a single disk is too risky for the workload. Common examples include:
- system volumes that must survive a disk failure
- file servers that need continuous access
- databases that benefit from faster reads or mirrored writes
- lab systems where you want to practice storage management safely

RAID design depends on the workload:
- **Read-heavy** workloads often benefit from mirroring or striping.
- **Write-heavy** workloads need careful consideration because parity-based RAID can add overhead.
- **Critical data** usually needs redundancy plus backups, not RAID alone.

**Common RAID levels**

| Level | Minimum disks | Fault tolerance | Usable capacity | Typical use |
|---|---:|---:|---|---|
| RAID 0 | 2 | None | Sum of all disks | Performance only |
| RAID 1 | 2 | 1 disk per mirror set | About 50% | Simple redundancy |
| RAID 5 | 3 | 1 disk | `(n-1)` disks | Balanced capacity and redundancy |
| RAID 6 | 4 | 2 disks | `(n-2)` disks | Better fault tolerance |
| RAID 10 | 4 | 1 disk per mirror pair | About 50% | High performance and resilience |

- **RAID 0** stripes data across disks for speed, but any disk failure destroys the array.
- **RAID 1** mirrors data to another disk, making recovery straightforward.
- **RAID 5** uses distributed parity and tolerates one disk failure.
- **RAID 6** uses dual parity and tolerates two disk failures.
- **RAID 10** combines mirroring and striping for strong performance and redundancy.

> RAID 5 and RAID 6 rebuilds can be slow on large disks. During rebuild, performance drops and the array is more vulnerable to a second failure.

**How striping, mirroring, and parity work**

- **Striping** splits data across disks in chunks called stripes or stripe units.
- **Mirroring** writes the same data to two or more disks.
- **Parity** stores calculated recovery information so the array can reconstruct lost data.

A simple mental model:
- striping improves speed by parallelizing I/O
- mirroring improves availability by duplicating data
- parity improves capacity efficiency by avoiding full duplication

**RAID implementation in Linux**

Linux commonly uses software RAID through the `md` subsystem and `mdadm`. The kernel assembles arrays from component devices such as `/dev/sdb` and `/dev/sdc`, then exposes the array as `/dev/md0` or similar.

Typical workflow:
1. Partition the disks with matching RAID partitions.
2. Create the array with `mdadm`.
3. Format the array with a filesystem.
4. Mount it and make it persistent.
5. Save the RAID metadata configuration for boot-time assembly.

Example: create a RAID 1 array from two partitions.

```bash
sudo mdadm --create /dev/md0 --level=1 --raid-devices=2 /dev/sdb1 /dev/sdc1
```

Check array status:

```bash
cat /proc/mdstat
sudo mdadm --detail /dev/md0
```

Create a filesystem and mount it:

```bash
sudo mkfs.ext4 /dev/md0
sudo mkdir -p /mnt/raid1
sudo mount /dev/md0 /mnt/raid1
```

**Persistent configuration**

To assemble arrays automatically at boot, save metadata and update the initramfs or system configuration as needed.

Common steps:
- write the array definition to `/etc/mdadm.conf` or the distribution equivalent
- update the initramfs if your distro requires it
- add the mount entry to `/etc/fstab`

Example `fstab` entry:

```fstab
/dev/md0  /mnt/raid1  ext4  defaults,nofail  0  2
```

- `nofail` helps boot continue if the array is unavailable.
- Use it carefully for non-root arrays where delayed availability is acceptable.

**Creating arrays safely**

Before creating RAID, ensure the disks are empty or that you have backed up the data. RAID creation usually overwrites metadata at the start or end of the device.

A practical partitioning example with `parted`:

```bash
sudo parted /dev/sdb --script mklabel gpt mkpart primary 1MiB 100%
sudo parted /dev/sdc --script mklabel gpt mkpart primary 1MiB 100%
sudo parted /dev/sdb --script set 1 raid on
sudo parted /dev/sdc --script set 1 raid on
```

Then create the array:

```bash
sudo mdadm --create /dev/md0 --level=5 --raid-devices=3 /dev/sdb1 /dev/sdc1 /dev/sdd1
```

**Monitoring and maintenance**

You should monitor RAID health regularly because degraded arrays can appear healthy at a glance while silently losing redundancy.

Useful commands:
```bash
cat /proc/mdstat
sudo mdadm --detail /dev/md0
sudo mdadm --examine /dev/sdb1
lsblk
```

What to look for:
- `active` versus `degraded`
- failed or removed member disks
- resync or rebuild progress
- mismatched event counters or metadata issues

Common maintenance actions:
- mark a failed disk as failed
- remove the failed member
- replace the disk
- add the new disk back into the array
- wait for rebuild or resync to complete

Example replacement flow:

```bash
sudo mdadm /dev/md0 --fail /dev/sdb1
sudo mdadm /dev/md0 --remove /dev/sdb1
sudo mdadm /dev/md0 --add /dev/sde1
```

**Resync, rebuild, and recovery**

A **resync** compares members and makes them consistent. A **rebuild** reconstructs data onto a replacement disk after failure. The exact terminology varies by RAID level and implementation, but both processes consume I/O and CPU.

During recovery:
- expect slower performance
- avoid heavy maintenance tasks if possible
- verify the array after rebuild completes
- confirm the filesystem is healthy with tools such as `fsck` when appropriate and offline

> Do not run filesystem repair tools on a mounted filesystem unless the tool explicitly supports it. For ext filesystems, repair is typically done offline.

**RAID and boot considerations**

If the root filesystem lives on RAID, boot support must be available early in the boot process. That usually means:
- the initramfs includes RAID assembly support
- the bootloader can read the required boot files
- the array metadata layout is compatible with boot requirements

Practical concerns:
- some systems keep `/boot` on a separate non-RAID partition for simplicity
- UEFI and BIOS boot paths may differ in how they access disks
- degraded root arrays may still boot if the remaining members are sufficient

**Hardware RAID vs software RAID**

| Aspect | Hardware RAID | Software RAID |
|---|---|---|
| Managed by | RAID controller | Linux kernel and `mdadm` |
| OS visibility | One logical disk | Array device like `/dev/md0` |
| Portability | Lower if controller fails | Higher if metadata is readable |
| CPU usage | Lower on host | Uses host CPU |
| Flexibility | Controller-dependent | Very flexible |

- **Hardware RAID** can simplify OS setup, but controller failure may complicate recovery.
- **Software RAID** is often preferred on Linux servers because it is transparent and portable.
- Choose based on supportability, performance needs, and operational familiarity.

**RAID level selection guidance**

Use the workload to guide your choice:
- **RAID 0** when speed matters and data loss is acceptable, such as scratch space.
- **RAID 1** for small systems, boot volumes, and simple redundancy.
- **RAID 5** when you want better capacity efficiency and can tolerate one-disk failure.
- **RAID 6** when arrays are large or rebuild times are long.
- **RAID 10** when you need strong performance and resilience for databases or busy services.

A simple rule of thumb:
- choose mirroring for simplicity
- choose parity for capacity efficiency
- choose RAID 10 for performance-sensitive production workloads

**Practical verification commands**

After creating or modifying an array, verify all layers:
```bash
lsblk -o NAME,SIZE,TYPE,MOUNTPOINT
sudo mdadm --detail /dev/md0
df -hT /mnt/raid1
```

If the array is not assembling automatically:
```bash
sudo mdadm --assemble --scan
sudo systemctl restart mdmonitor 2>/dev/null || true
```

If your distribution uses a different service name or boot integration, check the vendor documentation. Service names and initramfs tooling vary across distributions.

**Exam-focused takeaways**

- RAID improves availability and/or performance, but it is not a backup.
- `mdadm` is the standard Linux software RAID tool.
- RAID 0 has no redundancy.
- RAID 1 mirrors data.
- RAID 5 tolerates one disk failure.
- RAID 6 tolerates two disk failures.
- RAID 10 combines striping and mirroring.
- Monitor arrays with `/proc/mdstat` and `mdadm --detail`.
- Plan for rebuild time, boot integration, and replacement procedures.

**Quick reference example**

```bash
# Create a RAID 1 array
sudo mdadm --create /dev/md0 --level=1 --raid-devices=2 /dev/sdb1 /dev/sdc1

# Check status
cat /proc/mdstat
sudo mdadm --detail /dev/md0

# Format and mount
sudo mkfs.ext4 /dev/md0
sudo mkdir -p /srv/data
sudo mount /dev/md0 /srv/data
```
