**Persistent mount fundamentals**

`/etc/fstab` is the standard configuration file for defining filesystems that should mount automatically at boot or on demand. It gives you a repeatable way to attach local disks, partitions, network filesystems, and pseudo-filesystems without manually running `mount` after every reboot.

Use persistent mounts when you need:

- **Boot-time availability** for data volumes, application storage, or shared directories
- **Consistent mount options** such as `noatime`, `ro`, or `nodev`
- **Recovery-friendly configuration** that survives restarts and maintenance windows
- **Centralized documentation** of how storage is intended to behave

A typical `fstab` entry maps a device or identifier to a mount point and defines the filesystem type, options, dump behavior, and fsck order.

```text
<device>  <mountpoint>  <fstype>  <options>  <dump>  <pass>
```

The six fields are:

- **Device**: The block device, UUID, LABEL, or network source to mount
- **Mount point**: The directory where the filesystem becomes accessible
- **Filesystem type**: Such as `ext4`, `xfs`, `vfat`, `nfs`, or `tmpfs`
- **Options**: Comma-separated mount behavior settings
- **Dump**: Legacy backup flag, usually `0`
- **Pass**: Filesystem check order at boot

**Why identifiers matter**

Device names like `/dev/sdb1` can change across boots, especially when hardware order changes. For persistent mounts, prefer stable identifiers such as `UUID=` or `LABEL=`.

```bash
blkid
lsblk -f
```

Example output-driven approach:

```text
UUID="3f2a1c2e-7b1d-4d7a-9f2d-1a2b3c4d5e6f" TYPE="ext4"
LABEL="DATA" TYPE="xfs"
```

Using UUIDs reduces the risk of mounting the wrong disk after a reboot or hardware change.

**Common `fstab` entry patterns**

A local ext4 data volume mounted at `/data`:

```fstab
UUID=3f2a1c2e-7b1d-4d7a-9f2d-1a2b3c4d5e6f  /data  ext4  defaults  0  2
```

A read-only CD-ROM:

```fstab
/dev/sr0  /mnt/cdrom  iso9660  ro,noauto,user  0  0
```

A temporary filesystem in RAM:

```fstab
tmpfs  /run/cache  tmpfs  defaults,size=512M  0  0
```

A network share using NFS (Network File System):

```fstab
server:/exports/projects  /mnt/projects  nfs  defaults,_netdev  0  0
```

**Mount options you should know**

Mount options control behavior, performance, and security. They are often the most important part of an `fstab` entry.

- **`defaults`**: Uses a standard set of options, typically `rw,suid,dev,exec,auto,nouser,async`
- **`ro` / `rw`**: Mount read-only or read-write
- **`noatime`**: Do not update access times on reads; can reduce write activity
- **`nodev`**: Ignore device files on the mounted filesystem
- **`nosuid`**: Ignore setuid and setgid bits
- **`noexec`**: Prevent execution of binaries from the mount
- **`user` / `users`**: Allow non-root users to mount, with different permission behavior
- **`auto` / `noauto`**: Mount automatically at boot or only manually
- **`_netdev`**: Indicates the mount depends on networking and should not be treated like a local disk
- **`x-systemd.automount`**: Let systemd mount the filesystem on first access
- **`nofail`**: Boot continues even if the mount is unavailable

> Be careful with `noauto`, `_netdev`, and `nofail` together. They solve different problems: manual mounting, network dependency, and boot tolerance.

**Filesystem check field and boot order**

The last two columns are often misunderstood:

- **Dump field**: Historically used by the `dump` backup utility; most systems set this to `0`
- **Pass field**: Controls `fsck` order during boot
  - `0` means do not check
  - `1` is usually reserved for the root filesystem
  - `2` is used for other filesystems that should be checked after root

A common pattern is:

```fstab
UUID=...  /      ext4  defaults  0  1
UUID=...  /home  ext4  defaults  0  2
```

For filesystems that should not be checked by `fsck`, use `0` in the pass field. This is common for `xfs`, `tmpfs`, and many network mounts.

**Safe workflow for editing `/etc/fstab`**

You should validate changes carefully because a bad entry can prevent a system from booting normally.

1. Back up the file before editing.
2. Add or modify one entry at a time.
3. Test the mount manually before rebooting.
4. Verify the syntax and behavior.
5. Reboot only after the entry works as expected.

Example workflow:

```bash
sudo cp /etc/fstab /etc/fstab.bak
sudoedit /etc/fstab
sudo mount -a
findmnt --verify
findmnt /data
```

`mount -a` attempts to mount all filesystems listed in `fstab` that are eligible to mount. It is one of the fastest ways to catch syntax errors, missing directories, or invalid options.

`findmnt --verify` checks mount table consistency and can help identify malformed entries.

**Creating the mount point**

The mount point directory must exist before the filesystem can mount there.

```bash
sudo mkdir -p /data
sudo chown root:root /data
sudo chmod 755 /data
```

If the directory contains files before mounting, those files become hidden while the filesystem is mounted. That can confuse troubleshooting, so create empty mount points when possible.

**Finding the right device identifier**

Use tools that show stable identifiers and filesystem metadata.

```bash
lsblk -f
blkid /dev/sdb1
```

Example `lsblk -f` output concept:

```text
NAME   FSTYPE LABEL UUID                                 MOUNTPOINT
sda1   ext4         1111-2222-3333-4444                  /
sdb1   xfs   DATA   3f2a1c2e-7b1d-4d7a-9f2d-1a2b3c4d5e6f /data
```

If you are configuring a removable or secondary disk, UUID-based entries are usually safer than `/dev/sdX` names.

**Example `/etc/fstab` file**

```fstab
# <file system>                                  <mount point>  <type>  <options>                 <dump> <pass>
UUID=1111-2222-3333-4444                         /              ext4    defaults                 0      1
UUID=3f2a1c2e-7b1d-4d7a-9f2d-1a2b3c4d5e6f         /data          xfs     defaults,noatime         0      2
LABEL=BACKUP                                      /backup        ext4    nofail,x-systemd.automount 0     2
server:/exports/projects                          /mnt/projects  nfs     rw,_netdev,nofail        0      0
tmpfs                                             /run/cache     tmpfs   defaults,size=512M      0      0
```

This example shows several common patterns:

- Root filesystem with `pass` set to `1`
- Local data volume with `noatime`
- Optional backup volume with `nofail`
- NFS share with `_netdev`
- RAM-backed temporary storage with `tmpfs`

**Troubleshooting mount problems**

When a mount fails, check the error message first. It often points directly to the issue.

Common causes include:

- **Missing mount point** directory
- **Wrong filesystem type**
- **Invalid UUID or LABEL**
- **Typo in mount options**
- **Filesystem corruption**
- **Network not available** for remote mounts
- **Permissions or SELinux/AppArmor constraints** in some environments

Useful commands:

```bash
journalctl -b
dmesg | tail -n 50
mount -av
findmnt /data
```

If a bad `fstab` entry blocks boot, you may need to boot into rescue mode or emergency mode, comment out the faulty line, and repair the configuration.

> A malformed `fstab` line can cause boot delays or drop the system into emergency mode. Always test with `mount -a` before rebooting.

**Special cases and practical considerations**

- **Network filesystems**: Use `_netdev` so systemd treats the mount as network-dependent.
- **Removable media**: Use `noauto` if you do not want the system to try mounting absent media at boot.
- **Security-sensitive mounts**: Add `nodev`, `nosuid`, and `noexec` where appropriate, especially for shared or user-writable locations.
- **Performance tuning**: `noatime` can reduce metadata writes on busy filesystems.
- **Systemd integration**: `x-systemd.automount` can improve boot speed and reduce failures for infrequently used mounts.

**Editing tips**

Use `sudoedit` or a root-aware editor to avoid permission issues.

```bash
sudoedit /etc/fstab
```

Keep comments in the file to document why a mount exists, especially for non-obvious options.

```fstab
# Shared project storage; mount only when accessed
server:/exports/projects /mnt/projects nfs rw,_netdev,nofail,x-systemd.automount 0 0
```

**Quick verification checklist**

- The mount point directory exists
- The device identifier is stable
- The filesystem type matches the actual filesystem
- The options are valid for that filesystem
- `mount -a` succeeds without errors
- The filesystem appears in `findmnt`
- The system reboots cleanly with the entry enabled
