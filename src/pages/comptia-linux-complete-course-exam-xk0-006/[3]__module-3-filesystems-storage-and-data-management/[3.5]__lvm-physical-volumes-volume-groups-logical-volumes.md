## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**LVM overview**

<abbr title="Logical Volume Manager">LVM</abbr> lets you abstract storage so you can resize, combine, and move capacity more flexibly than with fixed partitions. Instead of binding a filesystem directly to a single partition, you place disks or partitions into a storage pool and carve out logical volumes from that pool.

This model is useful when you expect storage growth, need snapshots, or want to separate filesystem layout from physical disk layout. In practice, you use LVM when you want operational flexibility without redesigning partitions every time capacity changes.

- **PV**: a physical volume, usually a disk or partition initialized for LVM use.
- **VG**: a volume group, which aggregates one or more PVs into a shared pool.
- **LV**: a logical volume, which is the block device you format and mount.

**LVM building blocks**

A physical volume is the LVM layer that prepares a block device for inclusion in a volume group. LVM stores its own metadata in metadata areas on the PVs, and the exact layout and backup behavior depends on configuration and distribution defaults. The important exam idea is that PVs are the raw storage members, not filesystems.

A volume group pools capacity from one or more PVs. The VG is where free extents live, and LVs consume those extents. If you add another disk later, you can extend the VG and then grow one or more LVs without moving data off the existing filesystem first.

A logical volume is the device you actually format with a filesystem such as `ext4`, `xfs`, or `btrfs`. The filesystem sits on top of the LV, and the mount process does not care whether the underlying block device is a physical partition or an LVM LV.

- **Extent**: the allocation unit LVM uses internally.
- **PE**: physical extent, the unit of space inside a VG.
- **LE**: logical extent, the LV’s mapping to VG extents.

**Typical LVM layout**

A common layout starts with a disk, then a partition or whole-disk PV, then a VG, then one or more LVs.

```text
Disk -> PV -> VG -> LV -> Filesystem -> Mount point
```

Example topology:

```text
/dev/sdb1  ->  pvcreate  ->  vg_data  ->  lv_projects
/dev/sdc1  ->  pvcreate  ->  vg_data  ->  lv_backups
```

This design lets you place different workloads on separate LVs while still sharing the same storage pool.

**Core commands**

You should know the standard LVM command set and what each command changes.

- `pvcreate`: initialize a block device for LVM.
- `pvs`: display PV information.
- `pvdisplay`: show detailed PV metadata.
- `vgcreate`: create a VG from one or more PVs.
- `vgs`: display VG summary information.
- `vgdisplay`: show detailed VG information.
- `lvcreate`: create an LV.
- `lvs`: display LV summary information.
- `lvdisplay`: show detailed LV information.
- `pvresize`: resize a PV after the underlying disk grows.
- `vgextend`: add a PV to an existing VG.
- `vgreduce`: remove a PV from a VG when space has been evacuated.
- `lvextend`: grow an LV.
- `lvreduce`: shrink an LV.
- `lvremove`: delete an LV.
- `vgremove`: delete a VG.
- `pvremove`: remove LVM metadata from a PV.

A quick inventory view is often enough for troubleshooting:

```bash
pvs
vgs
lvs
```

For more detail:

```bash
pvdisplay /dev/sdb1
vgdisplay vg_data
lvdisplay /dev/vg_data/lv_projects
```

**Creating LVM storage**

A typical workflow is to initialize the device, create the VG, and then create one or more LVs.

```bash
sudo pvcreate /dev/sdb1
sudo vgcreate vg_data /dev/sdb1
sudo lvcreate -n lv_projects -L 20G vg_data
```

Then you format and mount the LV:

```bash
sudo mkfs.ext4 /dev/vg_data/lv_projects
sudo mkdir -p /projects
sudo mount /dev/vg_data/lv_projects /projects
```

To make the mount persistent, add an `/etc/fstab` entry using the UUID or device path.

```bash
sudo blkid /dev/vg_data/lv_projects
```

Example `/etc/fstab` line:

```fstab
UUID=1234-ABCD  /projects  ext4  defaults  0  2
```

> Use the UUID when possible because device names can change across boots or hardware changes.

**Extending storage**

One of LVM’s biggest advantages is online growth. If the VG has free extents, you can extend an LV and then grow the filesystem.

For an `ext4` filesystem, you can usually grow the filesystem while mounted:

```bash
sudo lvextend -L +10G /dev/vg_data/lv_projects
sudo resize2fs /dev/vg_data/lv_projects
```

You can also grow the LV and filesystem in one step with `-r` when supported:

```bash
sudo lvextend -r -L +10G /dev/vg_data/lv_projects
```

For `xfs`, you grow the filesystem with `xfs_growfs` after extending the LV:

```bash
sudo lvextend -L +10G /dev/vg_data/lv_projects
sudo xfs_growfs /projects
```

- **Online growth** is common and usually safe when the filesystem supports it.
- **Filesystem-specific tools** matter because `ext4` and `xfs` use different resize utilities.
- **Free extents** in the VG must exist before the LV can grow.

If you add a new disk to the system, the usual sequence is:

```bash
sudo pvcreate /dev/sdc1
sudo vgextend vg_data /dev/sdc1
sudo lvextend -r -L +50G /dev/vg_data/lv_projects
```

**Shrinking storage**

Shrinking is more delicate than growing. You must understand the filesystem type and the correct order of operations. `ext4` can be shrunk offline, but `xfs` cannot be shrunk with standard tools.

For an `ext4` filesystem, the safe pattern is:

1. Unmount the filesystem.
2. Check the filesystem.
3. Shrink the filesystem to a size slightly smaller than the target LV size.
4. Reduce the LV to the final size.
5. Remount and verify.

Example:

```bash
sudo umount /projects
sudo e2fsck -f /dev/vg_data/lv_projects
sudo resize2fs /dev/vg_data/lv_projects 14G
sudo lvreduce -L 15G /dev/vg_data/lv_projects
sudo mount /dev/vg_data/lv_projects /projects
```

The filesystem is intentionally shrunk a bit smaller first so the LV reduction has a safety margin. That reduces the risk of cutting the filesystem too close to its actual used space.

> Never shrink an LV before shrinking the filesystem on it. That can destroy data.

For `xfs`, the standard approach is to back up the data, recreate the filesystem at the smaller size, and restore the data. In exam terms, remember that `xfs` is grow-only with normal tooling.

**Removing storage**

Removal depends on whether the VG still contains multiple PVs and whether the LV is still in use.

A common cleanup sequence is:

1. Unmount the filesystem.
2. Remove the LV.
3. Remove the VG if it is no longer needed.
4. Remove the PV metadata from the underlying device.

Example for a single-PV VG:

```bash
sudo umount /projects
sudo lvremove /dev/vg_data/lv_projects
sudo vgremove vg_data
sudo pvremove /dev/sdb1
```

If the VG spans multiple PVs and you want to remove only one PV, you first move or remove the extents from that PV, then reduce the VG membership.

```bash
sudo pvmove /dev/sdc1
sudo vgreduce vg_data /dev/sdc1
sudo pvremove /dev/sdc1
```

- **`vgreduce`** is for removing a PV from a VG.
- **`vgremove`** deletes the entire VG.
- **`pvremove`** clears LVM metadata from the device so it is no longer recognized as a PV.

**Snapshots**

LVM snapshots capture a point-in-time view of an LV. They are useful for backups, testing, and short-term rollback scenarios. Traditional snapshots use copy-on-write behavior: when data changes on the origin LV, the snapshot preserves the old blocks by copying them to snapshot storage first.

Create a snapshot with enough space for expected change activity:

```bash
sudo lvcreate -s -n lv_projects_snap -L 2G /dev/vg_data/lv_projects
```

Mount the snapshot read-only if you want a consistent backup source:

```bash
sudo mkdir -p /mnt/snap
sudo mount -o ro /dev/vg_data/lv_projects_snap /mnt/snap
```

A snapshot can become invalid if its snapshot space fills up, which makes the snapshot unusable. The origin LV remains intact; the snapshot is what is affected.

- **Copy-on-write** means changed blocks are preserved before being overwritten.
- **Snapshot size** should reflect expected write activity during the snapshot window.
- **Short-lived use** is best because heavy writes can exhaust snapshot space quickly.

**Thin provisioning**

Thin provisioning lets you allocate logical space to LVs without reserving all physical space immediately. This is useful when you want to overcommit storage based on expected usage patterns, such as virtual machine disks.

A thin pool is the storage reservoir, and thin LVs consume space from that pool as data is written.

Example workflow:

```bash
sudo lvcreate -L 100G -n tp_data vg_data
sudo lvconvert --type thin-pool vg_data/tp_data
sudo lvcreate -V 20G -T vg_data/tp_data -n vm01
```

Depending on the LVM version and distribution, thin-pool creation may also be done directly with `lvcreate` options designed for thin pools. The key exam concept is the relationship: thin pool first, thin volume second.

- **Thin pool**: shared backing storage for thin LVs.
- **Thin LV**: appears large to the guest or filesystem but consumes space on demand.
- **Overcommit**: you can provision more virtual space than physically available, but you must monitor usage carefully.

> If a thin pool fills up, thin volumes can fail or become read-only depending on configuration and system behavior. Monitor pool usage proactively.

**Metadata and backups**

LVM keeps metadata about VGs and LVs in metadata areas associated with the PVs, and many systems also maintain automatic backups or archives of metadata changes. The exact behavior depends on configuration, but the common locations are:

- `/etc/lvm/backup/`
- `/etc/lvm/archive/`

These files are useful when you need to inspect or restore previous metadata states after an administrative mistake.

You can also back up metadata manually:

```bash
sudo vgcfgbackup vg_data
```

And restore from a known-good metadata file if needed:

```bash
sudo vgcfgrestore vg_data
```

Use these tools carefully, because restoring metadata can make the system’s view of storage diverge from the actual on-disk state if you choose the wrong version.

**Practical inspection and troubleshooting**

When something looks wrong, start by checking the LVM layer before blaming the filesystem.

```bash
sudo pvs
sudo vgs
sudo lvs -a -o +devices
```

Useful questions to ask:

- Is the PV present and recognized?
- Does the VG have free extents?
- Is the LV active?
- Is the filesystem mounted on the expected device?
- Did the mount use the correct UUID or device path?

You can activate or deactivate volumes when needed:

```bash
sudo lvchange -ay /dev/vg_data/lv_projects
sudo lvchange -an /dev/vg_data/lv_projects
```

If a device path changes, prefer `/dev/mapper/vg_data-lv_projects` or the `/dev/vg_data/lv_projects` symlink only after confirming the mapping with `lvs` or `lsblk`.

**Common exam distinctions**

LVM questions often test whether you know which layer does what.

- **PV**: prepares a block device for LVM.
- **VG**: pools PV capacity.
- **LV**: provides the block device for filesystems and applications.
- **Filesystem**: sits on the LV, not on the VG.
- **Mount point**: is where the filesystem becomes accessible in the directory tree.

A few high-value distinctions:

- You extend the VG when you add more PV capacity.
- You extend the LV when you want to give more space to a filesystem.
- You resize the filesystem after changing the LV size, unless you use a helper option like `-r`.
- You shrink filesystems before shrinking LVs, and only when the filesystem supports shrinking.

**Quick command reference**

```bash
# Initialize a PV
pvcreate /dev/sdb1

# Create a VG
vgcreate vg_data /dev/sdb1

# Create an LV
lvcreate -n lv_projects -L 20G vg_data

# Format and mount
mkfs.ext4 /dev/vg_data/lv_projects
mount /dev/vg_data/lv_projects /projects

# Extend storage
lvextend -r -L +10G /dev/vg_data/lv_projects

# Create a snapshot
lvcreate -s -n lv_projects_snap -L 2G /dev/vg_data/lv_projects

# Remove storage
lvremove /dev/vg_data/lv_projects
vgremove vg_data
pvremove /dev/sdb1
```

**What to remember for the exam**

LVM gives you flexible storage management by separating physical disks from logical storage objects. PVs feed VGs, VGs supply extents, and LVs become the block devices you format and mount. The most important operational skills are creating storage, extending it safely, understanding filesystem-specific resize behavior, and knowing the difference between a snapshot and the origin volume.
