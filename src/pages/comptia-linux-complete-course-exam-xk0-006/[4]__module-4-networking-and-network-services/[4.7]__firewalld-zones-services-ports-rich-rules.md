## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**firewalld overview**

`firewalld` is a dynamic firewall manager for Linux that uses zones, services, ports, and rich rules to control traffic without rebuilding the entire ruleset each time you make a change. It commonly sits on top of `nftables` on modern systems, while still presenting a simpler administrative model.

The main idea is to assign network connections to a **zone** that reflects trust level, then allow only the traffic that matches that zone’s policy. This is useful when the same host has multiple interfaces, such as a public NIC, a private management network, and a VPN tunnel.

- **Zones** define trust boundaries and default behavior.
- **Services** are named collections of ports and protocols.
- **Ports** open specific TCP or UDP endpoints.
- **Rich rules** add fine-grained matching for source, destination, protocol, logging, and actions.

**Why firewalld is used**

`firewalld` is designed for systems where network conditions change often. You can add or remove access while the firewall stays active, which is safer than stopping the firewall to edit static rules. This makes it practical for servers, laptops, and cloud instances that move between networks.

A typical workflow is:
1. Identify the active zone for the interface.
2. Add the required service or port.
3. Use a rich rule when the access needs extra conditions.
4. Make the change permanent if it should survive reboot.

**Zones**

A zone is a named policy profile. Each zone has a trust level and a set of allowed services, ports, and rules. Traffic is evaluated against the zone assigned to the interface or source address.

Common zones include:
- **drop**: silently drops incoming packets.
- **block**: rejects incoming packets with an error response.
- **public**: for untrusted networks; allows only selected services.
- **external**: for NAT gateways and internet-facing interfaces.
- **internal**: for trusted internal networks.
- **dmz**: for systems exposed to the outside but isolated from the internal network.
- **work** and **home**: for moderately trusted environments.
- **trusted**: allows all traffic.

The default zone matters because interfaces without an explicit zone assignment use it. You should verify the default before making assumptions about what traffic is allowed.

```bash
firewall-cmd --get-default-zone
firewall-cmd --get-active-zones
firewall-cmd --list-all
```

- **Default zone** applies when no interface or source mapping overrides it.
- **Active zones** show which interfaces are currently attached to which zone.
- **List-all** reveals services, ports, masquerading, and rich rules for a zone.

**Zone assignment and inspection**

You can assign a zone to an interface or source network. Interface assignment is common for physical or virtual NICs, while source-based assignment is useful for VPN ranges or management subnets.

```bash
# Show zones and interfaces
firewall-cmd --get-active-zones

# Assign an interface to a zone temporarily
firewall-cmd --zone=public --change-interface=eth0

# Make the interface assignment permanent
firewall-cmd --permanent --zone=public --change-interface=eth0

# Reload to apply permanent changes
firewall-cmd --reload
```

> Be careful when changing the zone on a remote system. If you move your SSH access into a zone that does not allow SSH, you can lock yourself out.

You can also bind a source network to a zone:

```bash
firewall-cmd --permanent --zone=internal --add-source=192.168.10.0/24
firewall-cmd --reload
```

This is useful when you want all hosts from a subnet to inherit the same policy, regardless of which interface they reach the server through.

**Services**

A service is a predefined label that maps to one or more ports, protocols, and sometimes helper settings. Using services is usually better than opening raw ports because the name documents intent and reduces mistakes.

Examples of common services:
- `ssh`
- `http`
- `https`
- `dns`
- `smtp`
- `ntp`
- `cockpit`

```bash
# List available service definitions
firewall-cmd --get-services

# Allow SSH in the current zone temporarily
firewall-cmd --add-service=ssh

# Allow HTTP permanently in the public zone
firewall-cmd --permanent --zone=public --add-service=http

# Remove a service
firewall-cmd --permanent --zone=public --remove-service=http
```

Service definitions are stored as XML files, usually under `/usr/lib/firewalld/services/` or `/etc/firewalld/services/`. The `/etc` location is for local overrides and custom definitions.

A custom service definition is useful when an application uses multiple ports or a nonstandard port set. For example, a monitoring tool might need both TCP and UDP on different ports.

```xml
<?xml version="1.0" encoding="utf-8"?>
<service>
  <short>myapp</short>
  <description>Custom application service</description>
  <port protocol="tcp" port="8443"/>
  <port protocol="tcp" port="9443"/>
</service>
```

After creating a custom service file, reload firewalld and add the service by name:

```bash
firewall-cmd --reload
firewall-cmd --permanent --zone=public --add-service=myapp
```

**Ports**

A port rule opens a specific port and protocol directly. Use this when no service definition exists or when you need a one-off exception.

```bash
# Open TCP 8080 temporarily
firewall-cmd --add-port=8080/tcp

# Open UDP 5353 permanently in the internal zone
firewall-cmd --permanent --zone=internal --add-port=5353/udp

# Remove a port
firewall-cmd --permanent --zone=internal --remove-port=5353/udp
```

- **TCP** is connection-oriented and common for web, SSH, and file transfer.
- **UDP** is connectionless and common for DNS, streaming, and discovery protocols.
- Always specify the protocol, because `8080` and `8080/tcp` are not the same rule.

Use ports when:
- the application is not packaged with a firewalld service definition,
- you are testing a temporary listener,
- you need a narrow exception that does not justify a custom service.

You can inspect open ports in a zone with:

```bash
firewall-cmd --zone=public --list-ports
firewall-cmd --zone=public --list-services
firewall-cmd --zone=public --list-all
```

**Services vs ports**

Services are usually preferred because they are easier to read and maintain. Ports are more explicit and sometimes necessary, but they can become harder to manage when an application uses multiple endpoints.

| Aspect | Service | Port |
|---|---|---|
| Readability | High | Lower |
| Scope | Can include multiple ports/protocols | Single port/protocol |
| Best use | Standard applications | Custom or temporary access |
| Maintenance | Easier | More manual |

A practical rule is to use a service when one exists, and use a port only when you need direct control.

**Permanent vs runtime configuration**

`firewalld` separates runtime changes from permanent configuration.

- **Runtime** changes take effect immediately but disappear after reload or reboot unless saved permanently.
- **Permanent** changes are stored on disk and applied after reload.

```bash
# Runtime change
firewall-cmd --add-service=ssh

# Permanent change
firewall-cmd --permanent --add-service=ssh

# Apply permanent changes without reboot
firewall-cmd --reload
```

This distinction is important during troubleshooting. If a rule works only until reload, it was probably added only to runtime. If a rule does not seem to take effect, you may have edited permanent settings but not reloaded.

**Rich rules**

Rich rules provide advanced matching and actions beyond simple service or port openings. They let you filter by source address, destination address, protocol, port, service, logging, and rate limiting.

Rich rules are useful when you need:
- access from only one subnet or host,
- logging for suspicious traffic,
- accept/reject/drop behavior based on conditions,
- more precise control than a zone-wide service rule.

Example rich rules:

```bash
# Allow SSH only from one host
firewall-cmd --permanent --zone=public \
  --add-rich-rule='rule family="ipv4" source address="192.168.10.50" service name="ssh" accept'

# Allow HTTPS from a subnet
firewall-cmd --permanent --zone=public \
  --add-rich-rule='rule family="ipv4" source address="192.168.20.0/24" service name="https" accept'

# Reject SMTP from everywhere else
firewall-cmd --permanent --zone=public \
  --add-rich-rule='rule family="ipv4" port port="25" protocol="tcp" reject'
```

A rich rule typically follows this pattern:

```bash
rule [family="ipv4|ipv6"] [source ...] [destination ...] [service|port|protocol ...] [log ...] [limit ...] action
```

Common actions:
- **accept**: allow the packet.
- **reject**: deny and notify the sender.
- **drop**: silently discard the packet.

Logging can help with audits and troubleshooting:

```bash
firewall-cmd --permanent --zone=public \
  --add-rich-rule='rule family="ipv4" source address="203.0.113.0/24" log prefix="FW-DENY " level="info" drop'
```

This logs matching traffic before dropping it. Use logging carefully, because excessive logging can create noise or performance overhead.

**Practical examples**

A web server on a public interface often needs only HTTP and HTTPS:

```bash
firewall-cmd --permanent --zone=public --add-service=http
firewall-cmd --permanent --zone=public --add-service=https
firewall-cmd --reload
```

An SSH management network may allow SSH only from a trusted subnet:

```bash
firewall-cmd --permanent --zone=internal \
  --add-rich-rule='rule family="ipv4" source address="10.10.0.0/24" service name="ssh" accept'
firewall-cmd --reload
```

A development host may expose a temporary application on port 3000:

```bash
firewall-cmd --add-port=3000/tcp
```

If the application becomes permanent, move from a temporary port rule to a permanent rule or a custom service definition.

**Common inspection commands**

Use these commands to verify what firewalld is doing:

```bash
firewall-cmd --state
firewall-cmd --get-default-zone
firewall-cmd --get-active-zones
firewall-cmd --zone=public --list-all
firewall-cmd --zone=public --list-rich-rules
firewall-cmd --zone=public --list-services
firewall-cmd --zone=public --list-ports
```

You can also inspect the underlying configuration files:

- `/etc/firewalld/zones/`
- `/etc/firewalld/services/`
- `/usr/lib/firewalld/zones/`
- `/usr/lib/firewalld/services/`

The `/usr/lib` files are vendor defaults, while `/etc` contains administrator overrides and custom definitions.

**Reloading and restarting**

`firewall-cmd --reload` reloads configuration without fully stopping the service. This is usually the safest way to apply permanent changes.

```bash
firewall-cmd --reload
```

A full restart is less common and may briefly disrupt traffic:

```bash
systemctl restart firewalld
```

Use reload for routine changes and restart only when necessary, such as after deeper service issues or when troubleshooting unexpected behavior.

**Troubleshooting workflow**

When a connection fails, check the firewall in a structured way:

1. Confirm the service is running.
2. Identify the active zone for the interface.
3. Verify the required service or port is allowed.
4. Check whether the rule is runtime or permanent.
5. Review rich rules that may override broader access.
6. Confirm the application is actually listening on the expected port.

```bash
systemctl status firewalld
firewall-cmd --get-active-zones
firewall-cmd --zone=public --list-all
ss -tulpn
```

If the application is not listening, the firewall is not the root cause. If it is listening but traffic still fails, the zone or rule set is the likely issue.

**Best practices**

- Prefer **services** over raw ports when a service definition exists.
- Keep the default zone restrictive, especially on public-facing systems.
- Use **rich rules** for exceptions instead of broad zone changes.
- Make changes permanent only after validating them in runtime.
- Document custom services and source-based zone assignments.
- Test remote access carefully to avoid locking yourself out.

**Quick reference**

- `firewall-cmd --get-active-zones` shows interface-to-zone mapping.
- `firewall-cmd --add-service=ssh` allows a named service.
- `firewall-cmd --add-port=8080/tcp` opens a specific port.
- `firewall-cmd --add-rich-rule='...'` adds conditional access control.
- `firewall-cmd --reload` applies permanent changes.
- `ss -tulpn` confirms what the host is actually listening on.

**Key exam takeaways**

- A **zone** is the primary trust and policy boundary in firewalld.
- A **service** is a named abstraction for one or more ports and protocols.
- A **port** rule is direct and specific, but less descriptive than a service.
- A **rich rule** provides advanced matching and actions for precise control.
- Runtime and permanent settings are separate, so always verify which one you changed.
