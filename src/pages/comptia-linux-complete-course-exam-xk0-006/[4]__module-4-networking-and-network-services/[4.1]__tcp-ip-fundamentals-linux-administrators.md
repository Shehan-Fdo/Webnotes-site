## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**TCP/IP model and OSI mapping**

The **TCP/IP model** is the practical networking model used by Linux and the Internet. The **OSI (Open Systems Interconnection)** model is a conceptual reference that helps you reason about where problems occur. Exam questions often expect you to map the two models and identify the layer responsible for a symptom.

- **Application layer** in TCP/IP maps roughly to OSI layers 5–7.
- **Transport layer** maps to OSI layer 4.
- **Internet layer** maps to OSI layer 3.
- **Link layer** maps to OSI layers 1–2.

A useful troubleshooting habit is to move from the bottom up:
- Verify link status and addressing.
- Verify routing and neighbor resolution.
- Verify transport reachability and service listening.
- Verify application behavior.

Common examples:
- A host can ping its gateway but not a remote server: likely routing, firewall, or remote service issues.
- A host cannot reach its gateway: likely link, VLAN, IP, mask, or neighbor resolution issues.
- A service is reachable by IP but not by name: likely DNS.

**TCP/IP layers and what they do**

- **Link layer**
  - Handles local network delivery on the same segment.
  - Uses MAC (Media Access Control) addresses.
  - Includes Ethernet, Wi-Fi, and VLAN behavior.
- **Internet layer**
  - Handles logical addressing and routing across networks.
  - Uses IPv4 and IPv6.
  - Includes ICMP and ICMPv6 for control and diagnostics.
- **Transport layer**
  - Provides end-to-end communication between processes.
  - Uses TCP and UDP.
- **Application layer**
  - Carries protocols such as SSH, HTTP, DNS, NTP, and SMTP.

**IPv4 and IPv6 addressing basics**

Linux administrators must understand how addresses, masks, and gateways work together.

- **IPv4** uses 32-bit addresses, commonly written in dotted decimal form.
- **IPv6** uses 128-bit addresses, written in hexadecimal colon-separated form.
- A **subnet mask** or prefix length defines the network portion.
- A **default gateway** forwards traffic to destinations outside the local subnet.

Examples:
- `192.168.10.25/24` means the host is on network `192.168.10.0/24`.
- `2001:db8:10::25/64` means the host is on an IPv6 /64 network.

Useful commands:
```bash
ip addr show
ip route show
ip -6 route show
hostname -I
```

Key points:
- `ip addr` shows assigned addresses and interface state.
- `ip route` shows the IPv4 routing table.
- `ip -6 route` shows the IPv6 routing table.
- `hostname -I` gives a quick view of assigned addresses, but it does not replace `ip`.

**ARP for IPv4 and NDP for IPv6**

Local delivery depends on mapping network-layer addresses to link-layer addresses.

- **ARP (Address Resolution Protocol)** resolves an IPv4 address to a MAC address on the local network.
- **NDP (Neighbor Discovery Protocol)** performs the IPv6 equivalent and also supports router discovery and neighbor reachability.

How they work:
- When a host wants to send to a local IPv4 destination, it broadcasts an ARP request: “Who has this IP?”
- The owner replies with its MAC address.
- For IPv6, the host uses NDP with ICMPv6 neighbor solicitation and neighbor advertisement messages instead of ARP broadcasts.

Troubleshooting implications:
- If ARP fails, IPv4 traffic may fail even when the IP configuration looks correct.
- If NDP fails, IPv6 neighbors may appear unreachable or incomplete.
- A stale or incorrect neighbor cache can cause intermittent connectivity.

Useful commands:
```bash
ip neigh show
arp -n
ping -c 1 192.168.10.1
ping -6 -c 1 2001:db8:10::1
```

Notes:
- `ip neigh` is the modern tool for viewing neighbor cache entries.
- `arp -n` may still appear on systems with legacy tools installed.
- An entry in `FAILED`, `INCOMPLETE`, or `STALE` state can help explain connectivity problems.

**ICMP and ICMPv6 for diagnostics**

**ICMP (Internet Control Message Protocol)** and **ICMPv6** are control protocols used for error reporting and network diagnostics. They are essential for troubleshooting and are frequently tested.

Common ICMP/ICMPv6 message types and uses:
- **Echo request/reply**
  - Used by `ping` and `ping6`/`ping -6`.
  - Confirms basic reachability and round-trip time.
- **Destination unreachable**
  - Indicates a host, network, port, or protocol is unreachable.
  - Can point to routing problems, firewall blocks, or a closed UDP/TCP destination.
- **Time exceeded**
  - Generated when a packet’s TTL (Time To Live) in IPv4 or Hop Limit in IPv6 reaches zero.
  - Used by `traceroute` to map the path to a destination.
- **Fragmentation needed / Packet Too Big**
  - Important for PMTU (Path Maximum Transmission Unit) discovery.
  - Helps hosts avoid sending packets larger than the path can carry.

Practical examples:
```bash
ping -c 4 8.8.8.8
ping -6 -c 4 2001:4860:4860::8888
traceroute 8.8.8.8
traceroute -6 2001:4860:4860::8888
```

Interpretation tips:
- Successful ping proves basic IP reachability, not application health.
- `Destination Host Unreachable` may indicate no route, ARP/NDP failure, or a local gateway issue.
- `Request timed out` can mean filtering, packet loss, or a host that does not answer ICMP.
- `Time exceeded` responses from intermediate hops are expected during traceroute.

> ICMP is not “just ping.” Blocking all ICMP can break troubleshooting and can interfere with PMTU discovery, causing large packets to fail mysteriously.

**MTU, fragmentation, and PMTU discovery**

The **MTU (Maximum Transmission Unit)** is the largest packet size a link can carry without fragmentation. Linux administrators need to understand MTU because mismatches often cause “works for small packets, fails for large transfers” symptoms.

Important concepts:
- IPv4 can fragment packets if necessary, though fragmentation is inefficient.
- IPv6 routers do not fragment packets in transit; the sender must adjust using PMTU discovery.
- **PMTU discovery** uses ICMP feedback to learn the smallest MTU along the path.
- If ICMP is blocked, PMTU discovery can fail and large connections may stall or hang.

Common symptoms:
- SSH connects, but file transfers stall.
- Web pages partially load.
- VPN traffic works for small packets but fails for larger ones.
- DNS queries succeed, but some TCP sessions fail after handshake.

Useful checks:
```bash
ip link show dev eth0
ping -M do -s 1472 8.8.8.8
tracepath 8.8.8.8
```

Notes:
- `ping -M do` on IPv4 asks the kernel not to fragment the packet.
- The `-s` value plus headers helps you test the path MTU.
- `tracepath` is useful because it can reveal PMTU changes along the route.

**Ports, sockets, and common port ranges**

A **port** identifies a process or service on a host. A **socket** is the combination of IP address, port, and protocol used for communication.

Port categories:
- **Well-known ports**: `0-1023`
  - Reserved for common services such as SSH (`22`), HTTP (`80`), HTTPS (`443`), DNS (`53`), and NTP (`123`).
- **Registered ports**: `1024-49151`
  - Used by many vendor and application services.
- **Dynamic/private ports**: `49152-65535`
  - Typically used as ephemeral client source ports.

Why this matters:
- Servers usually listen on well-known or registered ports.
- Clients usually connect from dynamic source ports.
- Firewall rules often distinguish between inbound service ports and outbound ephemeral ports.

Useful commands:
```bash
ss -tulpen
ss -tanp
lsof -iTCP -sTCP:LISTEN -P -n
```

Interpretation:
- `ss -tulpen` shows listening TCP and UDP sockets with process details.
- `ss -tanp` shows active TCP connections and their states.
- `LISTEN` means the service is waiting for connections.
- `ESTAB` means a TCP session is established.
- `CLOSE-WAIT`, `SYN-SENT`, and `SYN-RECV` can indicate application or network issues.

**TCP behavior and flags**

**TCP (Transmission Control Protocol)** is connection-oriented and reliable. It uses a handshake, acknowledgments, retransmissions, and flow control to deliver data in order.

Common TCP flags:
- **SYN**
  - Starts a connection.
- **ACK**
  - Acknowledges received data or handshake steps.
- **FIN**
  - Gracefully closes a connection.
- **RST**
  - Abruptly resets a connection.
- **PSH**
  - Requests prompt delivery to the application.
- **URG**
  - Marks urgent data; rarely used in modern practice.

Three-way handshake:
1. Client sends `SYN`.
2. Server replies `SYN, ACK`.
3. Client sends `ACK`.

Connection teardown:
1. One side sends `FIN`.
2. The other side acknowledges.
3. The other side sends its own `FIN`.
4. Final `ACK` completes closure.

Troubleshooting clues:
- Repeated `SYN` without `SYN, ACK` suggests filtering, routing, or a dead service.
- `RST` often means the port is closed or the application rejected the connection.
- Many retransmissions suggest packet loss, congestion, or MTU issues.

**UDP behavior**

**UDP (User Datagram Protocol)** is connectionless and does not provide reliability, ordering, or retransmission. It is lightweight and common for DNS, VoIP, streaming, and some monitoring tools.

Implications:
- A UDP service may appear “up” even when no handshake exists.
- Loss or filtering may be harder to detect than with TCP.
- Application-level checks are often needed to confirm UDP service health.

Useful examples:
```bash
ss -u -a -n
nc -u -v -z 192.168.10.20 53
```

**Linux network configuration locations and services**

Linux networking is managed differently across distributions, so exam questions may ask you to recognize common configuration systems.

Common management approaches:
- **NetworkManager**
  - Common on desktop and many server installations.
  - Stores connection profiles and can manage Wi-Fi, Ethernet, VLANs, bonds, and VPNs.
  - CLI tools include `nmcli` and `nmtui`.
- **systemd-networkd**
  - Common on servers and minimal installations.
  - Uses `.network`, `.netdev`, and `.link` files.
- **Traditional distro files**
  - Debian/Ubuntu may use `/etc/network/interfaces` on older setups.
  - RHEL-like systems may use `ifcfg-*` files on older releases.
  - Modern systems increasingly prefer NetworkManager or systemd-networkd.

Examples:
```bash
nmcli device status
nmcli connection show
nmcli connection show "Wired connection 1"
```

Example NetworkManager profile snippet:
```ini
[connection]
id=Wired connection 1
type=ethernet
autoconnect=true

[ipv4]
method=manual
address1=192.168.10.25/24,192.168.10.1
dns=1.1.1.1;8.8.8.8;

[ipv6]
method=auto
```

Example systemd-networkd file:
```ini
# /etc/systemd/network/20-eth0.network
[Match]
Name=eth0

[Network]
Address=192.168.10.25/24
Gateway=192.168.10.1
DNS=1.1.1.1
```

Service control examples:
```bash
systemctl status NetworkManager
systemctl restart NetworkManager
systemctl status systemd-networkd
systemctl restart systemd-networkd
```

Practical guidance:
- Use `nmcli` when NetworkManager controls the interface.
- Use `networkctl` and `.network` files when systemd-networkd is in use.
- Check which service owns the interface before editing config files.

**Core Linux networking commands**

These commands are foundational for exam prep and real troubleshooting.

```bash
ip link show
ip addr show
ip route show
ip neigh show
ss -tulpen
ping -c 4 192.168.10.1
traceroute 8.8.8.8
dig example.com
tcpdump -i eth0 -nn
```

What each command helps you verify:
- `ip link show`
  - Interface state, MAC address, and MTU.
- `ip addr show`
  - IP addresses and prefix lengths.
- `ip route show`
  - Default route and network routes.
- `ip neigh show`
  - ARP/NDP cache.
- `ss`
  - Listening and established sockets.
- `ping`
  - Basic reachability and latency.
- `traceroute`
  - Path and hop behavior.
- `dig`
  - DNS resolution.
- `tcpdump`
  - Packet-level evidence.

**Packet capture basics with tcpdump and Wireshark**

Packet capture helps you confirm what is actually on the wire instead of guessing from symptoms. `tcpdump` is the standard CLI tool; Wireshark provides a graphical view and deeper inspection.

Useful `tcpdump` examples:
```bash
tcpdump -i eth0 -nn host 192.168.10.20
tcpdump -i eth0 -nn port 22
tcpdump -i eth0 -nn icmp
tcpdump -i eth0 -nn -vvv -c 10
```

Reading packets:
- **Source and destination IPs** show who is talking.
- **Source and destination ports** show which applications are involved.
- **Flags** reveal TCP state, such as `S`, `S.`, `F.`, or `R`.
- **Length** helps identify whether payload is present.
- **TTL/Hop Limit** can hint at path length or routing behavior.
- **Checksums** may be offloaded by the NIC, so apparent checksum errors are not always real.

Wireshark interpretation basics:
- Filter by `ip.addr == 192.168.10.20` or `tcp.port == 443`.
- Follow a TCP stream to see application exchange.
- Look for retransmissions, duplicate ACKs, and resets.
- Use the packet details pane to inspect Ethernet, IP, TCP, and application headers.

Example capture workflow:
1. Start a capture on the relevant interface.
2. Reproduce the problem.
3. Filter by host, port, or protocol.
4. Check whether packets leave the host and whether replies return.
5. Compare packet behavior with the expected protocol flow.

**Routing and forwarding**

Routing determines where packets go when the destination is not on the local subnet.

Key concepts:
- The **default route** handles traffic for unknown networks.
- More specific routes override less specific ones.
- Linux chooses the best route based on prefix length and metrics.
- A host route targets a single IP address.

Useful commands:
```bash
ip route show
ip route get 8.8.8.8
ip -6 route show
```

Example route output interpretation:
- `default via 192.168.10.1 dev eth0` means all non-local IPv4 traffic goes to the gateway.
- `192.168.10.0/24 dev eth0 proto kernel scope link` means the local subnet is directly reachable.
- `unreachable` or missing routes explain why traffic cannot leave the host.

**DNS in the TCP/IP stack**

**DNS (Domain Name System)** is an application-layer service, but it is often the first thing users notice when networking fails.

Troubleshooting points:
- If IP connectivity works but names fail, check DNS configuration.
- DNS may use UDP port `53` for most queries and TCP port `53` for large responses or zone transfers.
- Resolver configuration is commonly stored in `/etc/resolv.conf`, though on modern systems it may be managed by `systemd-resolved` or NetworkManager.

Useful commands:
```bash
cat /etc/resolv.conf
resolvectl status
dig example.com
dig @1.1.1.1 example.com
```

**Practical troubleshooting workflow**

Use a layered approach so you do not skip obvious causes.

1. Verify the interface is up and has the expected address.
2. Verify the link partner and local neighbor resolution.
3. Verify the default route and specific routes.
4. Test reachability with `ping` and `traceroute`.
5. Confirm the service is listening with `ss`.
6. Check firewall rules and packet capture if needed.
7. Validate DNS if the issue is name-related.

Example workflow:
```bash
ip link show eth0
ip addr show eth0
ip route show
ip neigh show
ping -c 3 192.168.10.1
ping -c 3 8.8.8.8
ss -tulpen | grep ':22'
tcpdump -i eth0 -nn host 192.168.10.20 and port 22
```

**Common troubleshooting patterns**

- **No carrier or interface down**
  - Check cable, switch port, Wi-Fi association, or virtual NIC attachment.
- **Wrong IP or mask**
  - The host may think a remote system is local, or vice versa.
- **Missing default gateway**
  - Local traffic works, but remote networks fail.
- **ARP/NDP failure**
  - The host cannot resolve the next hop’s MAC address.
- **ICMP blocked**
  - Ping and traceroute may fail even when the service works.
- **Port closed or filtered**
  - `RST` or silence on a TCP connect attempt suggests service or firewall issues.
- **MTU mismatch**
  - Small packets work, large packets fail or stall.
- **DNS failure**
  - IP-based access works, but hostnames do not.

**Quick reference: protocol and tool mapping**

| Area | What to check | Common tools |
|---|---|---|
| Link | Carrier, MTU, MAC, interface state | `ip link`, `ethtool` |
| Addressing | IP, prefix, gateway | `ip addr`, `ip route` |
| Neighbor resolution | ARP/NDP cache | `ip neigh`, `arp` |
| Reachability | ICMP echo, path | `ping`, `traceroute`, `tracepath` |
| Transport | Ports, sockets, flags | `ss`, `lsof`, `nc` |
| Application | DNS, HTTP, SSH, logs | `dig`, `curl`, service logs |
| Wire evidence | Actual packets | `tcpdump`, Wireshark |

**Exam-focused takeaways**

- Know the TCP/IP to OSI mapping and use it to isolate faults.
- Understand ARP for IPv4 and NDP for IPv6 as local neighbor resolution mechanisms.
- Recognize ICMP and ICMPv6 message types and why they matter for diagnostics.
- Remember port ranges and how sockets relate to services.
- Distinguish TCP from UDP and know the major TCP flags.
- Understand MTU, fragmentation, and PMTU discovery because they cause subtle failures.
- Be comfortable with Linux tools such as `ip`, `ss`, `nmcli`, `tcpdump`, `ping`, and `traceroute`.
