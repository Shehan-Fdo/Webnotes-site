**Network service validation workflow**

Validating a network service means proving that each layer works, not just that a port is open. You start with name resolution, then address family selection, routing, firewall policy, transport behavior, and finally the application protocol itself. This layered approach helps you avoid false conclusions when a service fails for reasons outside the daemon.

A practical workflow is:

1. Confirm the service is running locally.
2. Test local socket binding and listening state.
3. Validate reachability from the same host and from a remote host.
4. Inspect packet flow when the result is unclear.
5. Correlate network symptoms with service logs.

Useful baseline commands:

```bash
ss -tulpn
systemctl status sshd
journalctl -u sshd --since "10 minutes ago"
curl -v http://127.0.0.1:8080/
```

- `ss` shows listening sockets and the process owning them.
- `systemctl` confirms whether systemd considers the service healthy.
- `journalctl` often reveals authentication, binding, or startup errors that packet capture cannot explain.
- `curl -v` exposes DNS, TCP, TLS, and HTTP details in one test.

**Local validation before remote testing**

A service can fail remotely even when it works on `localhost`. That usually points to binding, firewall, or routing issues rather than application failure. Check whether the daemon listens on `127.0.0.1`, `0.0.0.0`, or a specific interface address.

```bash
ss -ltnp
ip addr show
```

- `127.0.0.1` limits access to the local host.
- `0.0.0.0` listens on all IPv4 interfaces.
- `::` listens on all IPv6 interfaces.
- A service bound only to loopback will not accept remote connections even if the port appears open locally.

**Address family and dual-stack validation**

Modern systems often prefer IPv6 when both IPv4 and IPv6 are available. This can hide problems if one family works and the other does not. The browser and many client libraries may use <abbr title="Happy Eyeballs is a connection strategy that races IPv6 and IPv4 attempts">Happy Eyeballs</abbr>, which quickly falls back to the other family and masks partial outages.

Test both families explicitly:

```bash
ping -4 192.0.2.10
ping -6 2001:db8::10
curl -4 -v http://example.com/
curl -6 -v http://example.com/
getent ahosts example.com
```

- `-4` and `-6` force the address family.
- `getent ahosts` shows the resolved addresses and their ordering.
- If IPv6 fails but IPv4 succeeds, the application may still appear healthy because the client falls back silently.
- If DNS returns both `A` and `AAAA` records, the client may choose either path depending on policy and reachability.

> Be careful when testing only by hostname: a successful connection may hide an IPv6 routing problem or a broken AAAA path.

**Common service validation examples**

Different protocols expose different failure modes. Validate the application protocol, not just the port.

```bash
# SSH
ssh -vvv user@server.example.com

# HTTP
curl -v http://server.example.com/
curl -vk https://server.example.com/

# DNS
dig @192.0.2.53 example.com A
dig @192.0.2.53 example.com AAAA

# SMTP
nc -vz mail.example.com 25
openssl s_client -starttls smtp -connect mail.example.com:25

# NTP
ntpq -p
chronyc sources -v

# PostgreSQL
psql -h db.example.com -p 5432 -U appuser -d appdb
```

- `ssh -vvv` shows key exchange, authentication, and host key negotiation.
- `curl -v` reveals redirects, TLS negotiation, and HTTP status codes.
- `dig` confirms whether the DNS server answers and which records it returns.
- `openssl s_client -starttls smtp` validates STARTTLS on the correct SMTP port.
- `ntpq` and `chronyc` help confirm time synchronization, which matters for certificates and Kerberos-like systems.
- Database clients validate both transport and application authentication.

**Packet-level reasoning basics**

Packet analysis helps you distinguish between “no packet arrived,” “packet arrived but was dropped,” and “packet arrived but the application rejected it.” Read packets as evidence, not just as traffic.

Key fields to inspect:

- **Flags** in TCP packets:
  - `SYN` starts a connection.
  - `SYN, ACK` confirms the server accepted the initial request.
  - `ACK` advances an established session.
  - `RST` indicates a reset, often from a closed port or an application refusing the connection.
  - `FIN` indicates a graceful close.
- **Sequence and acknowledgment numbers**:
  - They show ordering and whether data is being acknowledged.
  - Large gaps or repeated retransmissions suggest loss or filtering.
- **TTL** (Time To Live):
  - Low or changing values can help infer path length or asymmetric routing.
  - Unexpectedly low TTL may indicate a nearby proxy or NAT device.
- **Window size**:
  - Very small or zero windows can indicate receiver congestion or application backpressure.
- **ICMP codes**:
  - They provide the reason for an unreachable or blocked path.
  - Destination Unreachable can mean network unreachable, host unreachable, port unreachable, fragmentation needed, or administratively prohibited traffic.

A simple mental model:

- `SYN` sent, no reply: packet loss, firewall drop, routing issue, or host down.
- `SYN` followed by `RST`: port closed or service not listening.
- `SYN` followed by `SYN, ACK`, then failure later: application, TLS, or post-handshake issue.
- ICMP error returned: the network is actively telling you why forwarding failed.

**tcpdump and tshark for validation**

`tcpdump` is ideal for quick packet capture on Linux. `tshark` adds field-level decoding and is useful when you need structured output.

```bash
sudo tcpdump -i eth0 -nn host 192.0.2.10 and port 443
sudo tcpdump -i eth0 -nn -vvv tcp port 22
sudo tshark -i eth0 -f "tcp port 443" -Y "tcp.flags.syn == 1 || tcp.flags.reset == 1"
```

- `-i eth0` selects the interface.
- `-nn` avoids DNS and service-name lookups, which keeps output precise.
- `-vvv` increases detail.
- Capture filters reduce traffic before capture; display filters in `tshark` refine what you inspect after capture.

A more targeted handshake capture can be:

```bash
sudo tcpdump -i eth0 -nn 'tcp port 443 and (tcp[tcpflags] & tcp-syn != 0 or tcp[tcpflags] & tcp-ack != 0 or tcp[tcpflags] & tcp-rst != 0)'
```

This still requires interpretation, but it is better than assuming every packet with `SYN` or `ACK` is part of the initial handshake. Look at the packet sequence, not just the presence of a flag.

**Interpreting packet captures like a Wireshark analyst**

Even without Wireshark, you can reason about the same fields. Wireshark-style thinking is about correlating layers:

- Ethernet or Wi-Fi frame delivery
- IP addressing and routing
- TCP or UDP transport behavior
- Application protocol response

Useful packet clues:

- **TCP three-way handshake**
  - `SYN` from client
  - `SYN, ACK` from server
  - `ACK` from client
- **Retransmissions**
  - Suggest loss, filtering, or path instability.
- **Duplicate ACKs**
  - Often indicate missing segments.
- **Zero window**
  - The receiver cannot accept more data right now.
- **ICMP fragmentation needed**
  - Often points to MTU or Path MTU Discovery issues.
- **TCP RST from middlebox**
  - May indicate firewall policy, proxy behavior, or security appliance intervention.

A quick `tshark` field view:

```bash
tshark -i eth0 -Y "tcp.port == 443" -T fields \
  -e frame.number -e ip.src -e ip.dst -e tcp.flags -e tcp.seq -e tcp.ack -e tcp.window_size_value
```

This helps you compare sequence numbers, acknowledgments, and window sizes across packets.

**MTU, fragmentation, and Path MTU Discovery**

A service can appear reachable while larger packets fail. This often happens when the path cannot carry the packet size and ICMP messages are blocked, preventing Path MTU Discovery from working correctly.

Symptoms include:

- Small requests succeed, large uploads fail.
- TLS handshakes work, but data transfer stalls.
- VPNs, tunnels, or overlay networks behave inconsistently.
- Some hosts work while others fail because of different path MTUs.

Useful tests:

```bash
ping -M do -s 1472 192.0.2.10
tracepath 192.0.2.10
ip link show eth0
```

- `ping -M do` asks Linux not to fragment the packet.
- `-s 1472` is a common IPv4 test size for a 1500-byte MTU, but adjust for headers and environment.
- `tracepath` can reveal MTU changes along the route.
- If ICMP “fragmentation needed” messages are filtered, the sender may never learn the correct path MTU.

> MTU problems often look like application bugs because the connection starts successfully and fails only when payload size increases.

**NAT and port forwarding as failure domains**

<abbr title="Network Address Translation">NAT</abbr> changes packet source or destination addresses, and port forwarding maps one external port to an internal host. Both can break validation if you only test from inside the same network.

Common pitfalls:

- Hairpin NAT may fail, so internal clients cannot reach the public address of an internal service.
- Port forwarding may point to the wrong internal IP or port.
- Stateful NAT devices may allow outbound connections but block unsolicited inbound traffic.
- Packet captures on the host may show the service listening while the edge device silently drops the traffic.

Validation approach:

- Test from inside the host.
- Test from another host on the same LAN.
- Test from outside the NAT boundary.
- Capture on both sides if possible.

Example topology reasoning:

- Client on the internet connects to `203.0.113.10:443`.
- Firewall forwards to `192.168.1.20:8443`.
- If the service listens on `127.0.0.1:8443`, forwarding will fail even though the process is running.
- If the firewall rewrites ports incorrectly, the server may receive traffic on the wrong destination port.

**Firewall troubleshooting: host firewall vs upstream firewall**

You need to isolate where traffic is dropped. A host firewall drops packets on the Linux system itself, while an upstream firewall or security group drops them before they reach the host.

A practical isolation sequence:

1. Confirm the service listens locally.
2. Test from the host to `127.0.0.1` and the host’s own IP.
3. Capture on the host interface.
4. Check host firewall rules.
5. Test from another host on the same subnet.
6. If packets never arrive, inspect upstream firewall, router ACLs, or cloud security groups.

Examples:

```bash
sudo firewall-cmd --list-all
sudo nft list ruleset
sudo iptables -S
sudo ufw status verbose
```

For `firewalld`, remember the difference between runtime and permanent changes:

```bash
sudo firewall-cmd --add-service=http
sudo firewall-cmd --add-service=https
sudo firewall-cmd --runtime-to-permanent
```

- The first two commands affect the current runtime configuration only.
- `--runtime-to-permanent` copies the active runtime state into the permanent configuration.
- If you want persistence directly, use `--permanent` with the add command and then reload.

Example:

```bash
sudo firewall-cmd --permanent --add-service=http
sudo firewall-cmd --permanent --add-service=https
sudo firewall-cmd --reload
```

**Reading ICMP and transport errors**

ICMP is not just “ping traffic.” It is a diagnostic channel that often explains why a path fails.

Common meanings:

- **Destination Unreachable**
  - Network unreachable: no route to the network.
  - Host unreachable: the next hop or host cannot be reached.
  - Port unreachable: the host received the packet, but no process is listening on that UDP port.
  - Administratively prohibited: a firewall or ACL blocked the traffic.
  - Fragmentation needed: the packet is too large for the next hop MTU.
- **Time Exceeded**
  - TTL expired, often due to routing loops or a path that is longer than expected.
- **Redirect**
  - A router suggests a better next hop, though redirects are often disabled or ignored in secure environments.

For UDP, remember the protocol has no handshake. Success is usually inferred from an application response, not from the transport layer itself.

```bash
nc -vu 192.0.2.53 53
dig @192.0.2.53 example.com
```

- `nc -vu` can send a UDP datagram, but it cannot reliably prove delivery or application success.
- `dig` is better for DNS because it expects a response and validates the protocol exchange.
- A lack of output from a UDP probe may mean the packet was dropped, the service did not respond, or the application requires a different query format.

**Loopback, containers, namespaces, and virtual interfaces**

Packet capture placement matters. If you capture on the wrong interface, you may miss the traffic entirely.

Common cases:

- **Loopback**
  - Use `-i lo` to see local-only traffic.
  - Services bound to `127.0.0.1` never appear on a physical NIC.
- **Containers**
  - Traffic may traverse veth pairs, bridges, or overlay interfaces.
  - Capture on the host bridge or inside the container namespace depending on where the failure occurs.
- **Virtual machines**
  - Traffic may appear on `virbr0`, `br0`, or a tap interface.
- **VPNs**
  - Traffic may be encapsulated on `tun0`, `wg0`, or similar interfaces.

Examples:

```bash
sudo tcpdump -i lo -nn port 8080
sudo tcpdump -i br0 -nn host 172.17.0.2
sudo ip netns exec myns tcpdump -i any -nn port 53
```

- `ip netns exec` lets you inspect traffic inside a network namespace.
- `-i any` can be useful for broad visibility, but it may hide interface-specific behavior.
- For container issues, compare host-side and namespace-side captures to see where packets stop.

**Correlating logs with packet captures**

Packet traces show what happened on the wire; logs show what the application thought happened. Use both.

Helpful commands:

```bash
journalctl -u nginx --since "15 minutes ago"
journalctl -u sshd --since "15 minutes ago"
tail -f /var/log/messages
tail -f /var/log/syslog
```

What to correlate:

- A TCP `RST` with an application log entry about bind failure or permission denial.
- Repeated connection attempts with authentication failures in the service log.
- TLS handshake failures with certificate or cipher mismatch messages.
- UDP requests with no response and a log entry showing malformed input or rate limiting.

A strong troubleshooting pattern is:

- Capture packets.
- Check the service log at the same timestamp.
- Compare source IP, destination IP, port, and protocol.
- Decide whether the failure is network-layer, transport-layer, or application-layer.

**Practical decision tree**

Use this sequence when a service is unreachable:

1. Confirm the daemon is running and listening.
2. Test the service locally with `localhost` or `127.0.0.1`.
3. Test the host’s own IP address.
4. Force IPv4 and IPv6 separately.
5. Check firewall rules on the host.
6. Verify routing, NAT, and port forwarding.
7. Capture packets on the correct interface.
8. Inspect ICMP, TCP flags, sequence numbers, and retransmissions.
9. Review service logs with `journalctl` or the application log file.
10. Test from a second network location to isolate upstream filtering.

**Exam-focused takeaways**

- A listening port does not guarantee remote reachability.
- A successful ping does not prove the application protocol works.
- TCP `RST` usually means the host actively refused the connection.
- UDP requires application-level confirmation, not just packet transmission.
- IPv4 and IPv6 can fail independently, and client fallback can hide the problem.
- MTU and fragmentation issues often appear only with larger payloads.
- NAT and port forwarding can break reachability even when the service is healthy locally.
- Packet captures are most useful when paired with logs and a clear hypothesis.
