## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Linux interface configuration overview**

Linux uses the `ip` suite from `iproute2` to inspect and manage network interfaces, addresses, routes, and related kernel networking state. It replaces older tools such as `ifconfig`, `route`, and `arp` in most modern distributions.

The `ip` command is preferred because it is consistent, scriptable, and exposes more of the kernel’s networking model. You use it to answer questions like:

- Which interfaces are up?
- Which IPv4 and IPv6 addresses are assigned?
- What is the default gateway?
- Which route will a packet take?
- Is a link administratively down or physically disconnected?

A useful mental model is that Linux networking is split into:
- **Link layer** state: interface name, MTU, MAC address, carrier, and admin status
- **Address layer** state: IPv4 and IPv6 addresses assigned to interfaces
- **Routing layer** state: how traffic is forwarded to local networks and remote destinations

**Core `ip` syntax and object model**

The `ip` command uses objects such as `link`, `addr`, `route`, `neigh`, and `rule`. This object-based design makes it easier to remember than separate legacy commands.

Common forms include:

```bash
ip link show
ip addr show
ip route show
ip neigh show
ip rule show
```

You can also shorten many object names:
- `ip a` for addresses
- `ip r` for routes
- `ip l` for links

The most important objects for interface configuration are:
- **`link`**: interface state and properties
- **`addr`**: IP addresses assigned to interfaces
- **`route`**: routing table entries
- **`neigh`**: neighbor cache entries, including ARP (Address Resolution Protocol) on IPv4 and NDP (Neighbor Discovery Protocol) on IPv6

**Inspecting interfaces**

Use `ip link show` to view interface names, flags, and link-layer details.

```bash
ip link show
```

Typical output includes:
- Interface index and name, such as `2: enp0s3`
- Flags such as `UP`, `LOWER_UP`, or `BROADCAST`
- `mtu` value
- MAC address shown as `link/ether`
- Queueing discipline, often `qdisc`
- State such as `state UP` or `state DOWN`

Example:

```bash
2: enp0s3: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc fq_codel state UP mode DEFAULT group default qlen 1000
    link/ether 08:00:27:4b:2c:11 brd ff:ff:ff:ff:ff:ff
```

Interpretation:
- **`UP`** in flags means the interface is administratively enabled
- **`LOWER_UP`** usually means the physical or virtual link is detected
- **`state DOWN`** may indicate no carrier or an administratively disabled interface
- **`mtu`** is the Maximum Transmission Unit, the largest packet size before fragmentation or dropping

To inspect one interface only:

```bash
ip link show dev enp0s3
```

**Managing link state**

You can bring an interface up or down with `ip link set`.

```bash
sudo ip link set dev enp0s3 up
sudo ip link set dev enp0s3 down
```

Use this when you need to:
- Enable a newly created or disabled interface
- Test connectivity changes
- Reset a link during troubleshooting

Important distinction:
- `up` changes the **administrative state**
- It does not guarantee the cable, virtual switch, or upstream network is working

You can also change interface properties such as MTU and MAC address:

```bash
sudo ip link set dev enp0s3 mtu 9000
sudo ip link set dev enp0s3 address 02:42:ac:11:00:02
```

Practical notes:
- MTU changes are common in storage, virtualization, and high-performance networks
- MAC address changes are often used for testing, cloning, or privacy, but may be restricted by policy

> Changing the MAC address or MTU can disrupt connectivity if the network expects a specific hardware identity or frame size.

**Viewing and assigning IP addresses**

Use `ip addr show` to inspect assigned IPv4 and IPv6 addresses.

```bash
ip addr show
```

Example output:

```bash
2: enp0s3: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500
    inet 192.168.1.50/24 brd 192.168.1.255 scope global dynamic enp0s3
    inet6 fe80::a00:27ff:fe4b:2c11/64 scope link
```

Key fields:
- **`inet`**: IPv4 address
- **`inet6`**: IPv6 address
- **CIDR prefix** such as `/24` or `/64`
- **`brd`**: broadcast address for IPv4
- **`scope`**: reachability scope, such as `host`, `link`, or `global`
- **`dynamic`**: address obtained automatically, often via DHCP (Dynamic Host Configuration Protocol)

To add an address temporarily:

```bash
sudo ip addr add 192.168.1.50/24 dev enp0s3
```

To remove it:

```bash
sudo ip addr del 192.168.1.50/24 dev enp0s3
```

For IPv6:

```bash
sudo ip -6 addr add 2001:db8:10::50/64 dev enp0s3
sudo ip -6 addr del 2001:db8:10::50/64 dev enp0s3
```

Useful behaviors:
- You can assign multiple addresses to one interface
- Temporary `ip addr add` changes usually do not persist after reboot
- Persistent configuration is handled by tools such as NetworkManager, systemd-networkd, or distro-specific network scripts

**Understanding address scope and types**

Linux uses scope to describe where an address is valid.

Common scopes:
- **`host`**: valid only on the local machine, such as `127.0.0.1`
- **`link`**: valid only on the local network segment, such as IPv6 link-local addresses
- **`global`**: routable beyond the local link, subject to routing and policy

Examples:
- `127.0.0.1/8` is loopback and uses `scope host`
- `fe80::/10` is IPv6 link-local and uses `scope link`
- `192.168.1.50/24` is usually `scope global` even though it is private RFC 1918 space

Address types you may see:
- **Primary** address: main address on an interface
- **Secondary** address: additional address on the same interface
- **Tentative** IPv6 address: still undergoing Duplicate Address Detection (DAD)
- **Deprecated** IPv6 address: still usable for existing connections, but not preferred for new ones

**Routing with `ip route`**

Routing determines where packets go after Linux decides they are not local.

View the routing table:

```bash
ip route show
```

Typical output:

```bash
default via 192.168.1.1 dev enp0s3
192.168.1.0/24 dev enp0s3 proto kernel scope link src 192.168.1.50
```

Interpretation:
- **`default`** route is used when no more specific route matches
- **`via`** identifies the next-hop gateway
- **`dev`** identifies the outgoing interface
- **`proto kernel`** often indicates a route created automatically from an interface address
- **`src`** is the preferred source address for traffic on that route

Add a default route:

```bash
sudo ip route add default via 192.168.1.1 dev enp0s3
```

Add a specific network route:

```bash
sudo ip route add 10.20.0.0/16 via 192.168.1.254 dev enp0s3
```

Delete a route:

```bash
sudo ip route del 10.20.0.0/16
```

Replace a route safely:

```bash
sudo ip route replace default via 192.168.1.1 dev enp0s3
```

Why `replace` matters:
- It avoids errors if the route already exists
- It is useful in scripts and automation

**Route selection basics**

Linux chooses routes by:
1. Matching the most specific prefix first
2. Considering policy routing rules if present
3. Using the best metric when multiple routes match equally

Important terms:
- **Prefix length**: longer prefixes are more specific, such as `/24` over `/16`
- **Metric**: lower values are usually preferred
- **Gateway**: the next hop used to reach remote networks

Example:
- `10.10.5.0/24` beats `10.10.0.0/16` for destination `10.10.5.20`
- A route with metric `100` is usually preferred over one with metric `200` if both are otherwise equal

**Neighbor table and ARP/NDP**

The neighbor table maps IP addresses to link-layer addresses.

View it with:

```bash
ip neigh show
```

Example:

```bash
192.168.1.1 dev enp0s3 lladdr 08:00:27:aa:bb:cc REACHABLE
```

This is the IPv4 ARP equivalent on Ethernet networks. For IPv6, the same table also stores NDP information.

Common states:
- **REACHABLE**: recently confirmed
- **STALE**: valid but not recently used
- **DELAY**: waiting before probing
- **FAILED**: resolution failed

You can flush neighbor entries during troubleshooting:

```bash
sudo ip neigh flush dev enp0s3
```

Use this carefully because it forces Linux to relearn neighbors.

**Policy routing with `ip rule`**

Policy routing lets Linux choose different routing tables based on source address, destination, TOS (Type of Service), or other selectors.

View rules:

```bash
ip rule show
```

Typical default rules:
- `lookup local`
- `lookup main`
- `lookup default`

A custom rule example:

```bash
sudo ip rule add from 192.168.50.0/24 table 100
sudo ip route add default via 192.168.50.1 dev enp0s3 table 100
```

Use policy routing when:
- A host has multiple uplinks
- Different source networks must use different gateways
- You need traffic separation for VPNs or specialized services

**Practical troubleshooting workflow**

When a host cannot reach the network, check layers in order.

1. Verify the interface exists and is up:
   ```bash
   ip link show
   ```
2. Verify the address assignment:
   ```bash
   ip addr show dev enp0s3
   ```
3. Verify the route table:
   ```bash
   ip route show
   ```
4. Verify neighbor resolution:
   ```bash
   ip neigh show
   ```
5. Test local and remote connectivity:
   ```bash
   ping -c 3 127.0.0.1
   ping -c 3 192.168.1.1
   ping -c 3 8.8.8.8
   ```
6. Inspect path selection:
   ```bash
   ip route get 8.8.8.8
   ```

`ip route get` is especially useful because it shows the exact route Linux would use for a destination.

Example:

```bash
ip route get 8.8.8.8
```

Possible output:

```bash
8.8.8.8 via 192.168.1.1 dev enp0s3 src 192.168.1.50 uid 1000
```

This tells you:
- The chosen gateway
- The outgoing interface
- The source address Linux selected

**Temporary vs persistent configuration**

Commands like `ip addr add` and `ip route add` change the live kernel state only. They are ideal for:
- Testing
- Emergency recovery
- Short-lived lab work

They are not enough for permanent configuration on most systems.

Persistence is usually handled by:
- **NetworkManager**
- **systemd-networkd**
- Distribution-specific network configuration files

Examples of persistent configuration vary by distro, but the concept is the same:
- Define interface name
- Assign address and prefix
- Set gateway and DNS
- Enable the connection at boot

**Common interface naming patterns**

Modern Linux often uses predictable interface names such as:
- `enp0s3`
- `ens33`
- `wlp2s0`

These names are based on device location or type rather than the older `eth0` style. This reduces ambiguity when multiple adapters exist.

Useful naming clues:
- `en` often indicates Ethernet
- `wl` often indicates wireless LAN
- `p` and numbers often indicate PCI bus location

**Real-world examples**

Bring up an interface, assign an address, and add a default route for a lab host:

```bash
sudo ip link set dev enp0s3 up
sudo ip addr add 192.168.10.25/24 dev enp0s3
sudo ip route add default via 192.168.10.1 dev enp0s3
```

Add a second address to the same interface for a service alias:

```bash
sudo ip addr add 192.168.10.26/24 dev enp0s3
```

Check the resulting state:

```bash
ip addr show dev enp0s3
ip route show
```

Remove the temporary configuration:

```bash
sudo ip route del default via 192.168.10.1 dev enp0s3
sudo ip addr del 192.168.10.25/24 dev enp0s3
sudo ip addr del 192.168.10.26/24 dev enp0s3
```

**Legacy tools and why they matter**

You may still encounter `ifconfig`, `route`, and `arp` on older systems or in older documentation. They are useful to recognize, but `ip` is the modern standard.

Comparison:
- `ifconfig` roughly maps to `ip link` and `ip addr`
- `route` roughly maps to `ip route`
- `arp` roughly maps to `ip neigh`

For Linux+ exam prep, know the modern commands first and recognize the legacy equivalents when reading older material.

**Exam-focused takeaways**

- **`ip link`** manages interface state and link-layer properties
- **`ip addr`** manages IP addresses on interfaces
- **`ip route`** manages routing tables
- **`ip neigh`** shows neighbor cache entries
- **`ip rule`** supports policy routing
- Temporary changes made with `ip` usually do not persist across reboot
- Route selection depends on specificity, metrics, and policy rules
- Troubleshooting should move from interface state to addressing, routing, and neighbor resolution

**Quick reference**

```bash
ip link show
ip link set dev enp0s3 up
ip addr show dev enp0s3
ip addr add 192.168.1.50/24 dev enp0s3
ip route show
ip route add default via 192.168.1.1 dev enp0s3
ip route get 8.8.8.8
ip neigh show
ip rule show
```
