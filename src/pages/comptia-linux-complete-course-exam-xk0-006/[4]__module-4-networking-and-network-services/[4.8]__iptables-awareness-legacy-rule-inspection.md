**iptables in context**

`iptables` is the classic Linux userspace tool for managing packet filtering rules in the kernel’s netfilter framework. You use it to inspect, add, delete, and modify rules for chains such as `INPUT`, `OUTPUT`, and `FORWARD`.

Modern Linux systems may use either the legacy `iptables` backend or the nftables compatibility layer. For exam purposes, you should recognize both the command syntax and the fact that the visible ruleset may not always represent the native packet filter engine in use.

- **netfilter** is the kernel packet-filtering subsystem.
- **iptables** is the userspace command that configures netfilter tables and chains.
- **nftables** is the newer packet-filtering framework that can coexist with `iptables` compatibility tools.
- **legacy rules** usually refers to the older `iptables` rule format and tooling, often contrasted with nftables-native configuration.

A practical mental model is:

- `iptables` is the interface you type.
- netfilter is where enforcement happens.
- the backend may be `legacy` or `nf_tables` depending on distribution and package selection.

> Be careful not to assume that `iptables` output always reflects the native kernel ruleset on every system. On nft-backed systems, compatibility tools may present translated rules rather than the underlying nftables configuration.

**Why rule inspection matters**

You inspect firewall rules to confirm what traffic is allowed, blocked, logged, or redirected. This is essential when troubleshooting connectivity, validating security posture, or auditing a host after a change.

Typical reasons to inspect rules include:

- verifying why a service is unreachable
- checking whether a port is allowed before opening it publicly
- confirming that a rule order is correct
- identifying counters that show whether traffic is matching a rule
- comparing active rules with saved configuration files

**Core tables and chains**

`iptables` organizes rules into tables, and each table contains chains.

Common tables:

- **filter**: default table for packet filtering
- **nat**: network address translation
- **mangle**: packet alteration
- **raw**: exemptions from connection tracking
- **security**: SELinux-related packet marking in some environments

Common built-in chains:

- `INPUT`: traffic destined for the local host
- `OUTPUT`: traffic originating from the local host
- `FORWARD`: traffic routed through the host
- `PREROUTING`: packets before routing decision
- `POSTROUTING`: packets after routing decision

A simple way to remember the flow:

- incoming traffic to the host goes through `INPUT`
- outgoing traffic from the host goes through `OUTPUT`
- transit traffic goes through `FORWARD`

**Inspecting rules safely**

When you inspect firewall rules, you usually want to avoid changing them. The most common read-only commands are `-L`, `-S`, and `-t`.

```bash
sudo iptables -L
sudo iptables -L -n -v
sudo iptables -S
sudo iptables -t nat -S
```

Key inspection options:

- `-L` lists rules in a chain or table
- `-S` prints rules in a form close to the command syntax used to create them
- `-t <table>` selects a table such as `filter` or `nat`
- `-n` prevents DNS and service-name lookups
- `-v` shows verbose details, including packet and byte counters
- `--line-numbers` adds rule numbers for easier reference when deleting or inserting rules

> `iptables -L` alone does not show packet and byte counters. Use `-v` when you need counters or more detail.

A useful inspection workflow is:

1. Check the backend and version.
2. List the active rules with numeric output.
3. Review counters to see which rules are being hit.
4. Compare the live ruleset with saved configuration if needed.

```bash
iptables --version
sudo iptables -L -n -v --line-numbers
sudo iptables -t nat -L -n -v --line-numbers
sudo iptables -S
```

**Understanding backend differences**

On many systems, `iptables --version` reveals whether the command is using the legacy backend or the nftables compatibility backend.

Examples you may see:

```bash
iptables v1.8.x (legacy)
iptables v1.8.x (nf_tables)
```

Interpretation:

- **legacy** indicates the older iptables backend.
- **nf_tables** usually indicates the iptables-nft compatibility layer.
- The exact packaging and wrapper behavior can vary by distribution.

This matters because:

- the same `iptables` command may manage different underlying implementations
- saved output from `iptables-save` may reflect the compatibility layer rather than a native nftables ruleset
- troubleshooting should account for the active backend before assuming rule behavior

**Listing rules with counters and line numbers**

Counters help you determine whether a rule is actually matching traffic. Line numbers help you reference a specific rule for deletion or insertion.

```bash
sudo iptables -L INPUT -n -v --line-numbers
```

What each part does:

- `-L INPUT` lists only the `INPUT` chain
- `-n` avoids name resolution delays
- `-v` shows counters and extra detail
- `--line-numbers` prefixes each rule with a number

Use line numbers when you need to remove a rule precisely:

```bash
sudo iptables -D INPUT 3
```

That deletes rule number 3 from the `INPUT` chain. Because rule order matters, always re-list the chain after changes.

**Reading rule output**

A typical verbose listing includes:

- chain policy
- packet and byte counters
- protocol
- source and destination
- match conditions
- target action such as `ACCEPT`, `DROP`, `REJECT`, or `LOG`

Example:

```bash
sudo iptables -L INPUT -n -v --line-numbers
```

Possible interpretation:

- a high packet counter suggests the rule is actively matching traffic
- a zero counter may mean the rule is unused, newly added, or not reached because of earlier rules
- the first matching rule usually determines the action, so order is critical

Important rule-order concepts:

- rules are evaluated top to bottom
- the first match wins in most chains
- a broad `DROP` rule placed early can block later `ACCEPT` rules
- default chain policy applies only if no rule matches

**Saving and restoring rules**

Legacy iptables environments often use `iptables-save` and `iptables-restore` to export and import rulesets.

```bash
sudo iptables-save
sudo iptables-save > /root/iptables.rules
sudo iptables-restore < /root/iptables.rules
```

Use cases:

- backing up a known-good firewall configuration
- migrating rules between systems
- restoring after a reboot on systems that do not persist rules automatically

Cautions:

- `iptables-save` output is useful for backup and review, but on nft-backed systems it may represent the compatibility layer rather than the native nftables configuration
- always verify the active backend before assuming the saved file is the authoritative source of truth
- test restores carefully, especially on remote systems, to avoid locking yourself out

**Common inspection commands and what they show**

| Command | Purpose | Notes |
|---|---|---|
| `iptables -L` | List rules | No counters unless `-v` is added |
| `iptables -L -n -v` | List rules with counters | Avoids DNS and service lookups |
| `iptables -S` | Show rules in command-like form | Helpful for auditing and scripting |
| `iptables -t nat -L -n -v` | Inspect NAT rules | Useful for port forwarding and masquerading |
| `iptables-save` | Dump the current ruleset | Good for backups and comparisons |
| `iptables-restore` | Load a saved ruleset | Use carefully on live systems |

**Practical troubleshooting examples**

If SSH is unreachable, inspect the `INPUT` chain first:

```bash
sudo iptables -L INPUT -n -v --line-numbers
```

Look for:

- a default policy of `DROP`
- an `ACCEPT` rule for TCP port 22
- an earlier rule that drops or rejects SSH traffic
- counters that show whether packets are reaching the rule

If web traffic is not forwarding through a host acting as a router, inspect `FORWARD` and `nat`:

```bash
sudo iptables -L FORWARD -n -v --line-numbers
sudo iptables -t nat -L -n -v --line-numbers
```

Look for:

- forwarding rules that permit the traffic path
- masquerade or source NAT rules in `POSTROUTING`
- destination NAT or port-forwarding rules in `PREROUTING`

If a rule appears correct but traffic still fails, check:

- rule order
- chain policy
- interface matches such as `-i` and `-o`
- source and destination address matches
- protocol and port matches
- whether the packet is hitting a different table or chain than expected

**Example rule set and interpretation**

```bash
sudo iptables -S
```

Example output shape:

```bash
-P INPUT DROP
-P FORWARD DROP
-P OUTPUT ACCEPT
-A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
-A INPUT -i lo -j ACCEPT
-A INPUT -p tcp --dport 22 -j ACCEPT
```

How to read it:

- default `INPUT` policy is `DROP`
- established connections are allowed back in
- loopback traffic is allowed
- SSH is explicitly allowed
- anything else to the host is blocked unless another rule matches

This is a common secure baseline because it permits only necessary inbound traffic while allowing outbound traffic by default.

**Useful real-world commands**

```bash
# Show backend version
iptables --version

# List all rules with counters and line numbers
sudo iptables -L -n -v --line-numbers

# Inspect only the NAT table
sudo iptables -t nat -L -n -v --line-numbers

# Show rules in a script-friendly format
sudo iptables -S

# Save the current ruleset
sudo iptables-save > /root/iptables.rules

# Restore a saved ruleset
sudo iptables-restore < /root/iptables.rules
```

**Legacy vs nftables awareness**

You do not need to become an nftables expert for basic Linux+ firewall awareness, but you should know the distinction.

- **iptables-legacy** uses the older kernel interface and rule handling.
- **iptables-nft** translates iptables syntax to nftables under the hood.
- Some distributions default to one backend while still providing the other for compatibility.
- Troubleshooting should begin by identifying which backend is active.

A cautious exam-ready rule of thumb:

- if `iptables --version` shows `legacy`, you are using the older backend
- if it shows `nf_tables`, you are using the nft-compatible backend
- if behavior seems inconsistent, verify whether another firewall manager is also active

**Interaction with other firewall tools**

On some systems, `firewalld`, `ufw`, or direct nftables rules may coexist with or replace manual iptables management.

Be aware that:

- a higher-level firewall service may rewrite rules automatically
- manual `iptables` changes may not persist after reboot unless saved and restored
- multiple tools managing the same packet filter can create confusing results

When troubleshooting, identify the controlling layer before editing rules.

**Exam focus points**

- Know the purpose of `iptables` as a netfilter management tool.
- Know the main chains: `INPUT`, `OUTPUT`, `FORWARD`, `PREROUTING`, `POSTROUTING`.
- Know how to inspect rules with `-L`, `-S`, `-t`, `-n`, `-v`, and `--line-numbers`.
- Know that `-v` is required for packet and byte counters.
- Know that rule order matters and the first match usually wins.
- Know that `iptables-save` and `iptables-restore` are used for backup and restore.
- Know that backend differences between legacy and nft-compatible modes affect interpretation.
