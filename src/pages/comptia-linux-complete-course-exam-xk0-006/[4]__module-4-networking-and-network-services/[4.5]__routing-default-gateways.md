**Routing fundamentals**

Linux routing decides where packets go after they leave the local host. The kernel consults the routing table to choose a next hop based on destination IP address, route prefix length, and route metrics. If the destination is on the local subnet, the packet is sent directly at Layer 2; otherwise, it is forwarded to a gateway.

A **default gateway** is the route used when no more specific route matches. In IPv4, this is usually `0.0.0.0/0`; in IPv6, it is `::/0`. You use a default route for general Internet access or any off-subnet traffic.

- **Destination match** uses longest-prefix match first.
- **Metric** breaks ties between routes with the same prefix.
- **Next hop** is the router IP used to reach remote networks.
- **Interface** identifies the outgoing device when needed.

A simple topology looks like this:

```text
Host 192.168.10.50/24 ---- 192.168.10.1 (gateway) ---- Internet
```

If the host needs to reach `8.8.8.8`, it sends the packet to `192.168.10.1` because no more specific route exists.

**Route types you should recognize**

Linux route tables can contain several route types, and exam questions often ask what they mean.

- **Unicast** routes forward traffic normally to a next hop or directly connected network.
- **Local** routes represent addresses assigned to the host itself.
- **Broadcast** routes handle broadcast traffic on IPv4 networks.
- **Multicast** routes handle multicast destinations.
- **Blackhole** routes silently discard traffic.
- **Unreachable** routes discard traffic and return an error to the sender.
- **Prohibit** routes discard traffic and return an administratively prohibited error.
- **Throw** routes stop lookup in the current table and continue policy routing evaluation.

Examples:

```bash
ip route show
ip route add blackhole 203.0.113.0/24
ip route add unreachable 198.51.100.0/24
```

Use blackhole or unreachable routes when you want to intentionally block a prefix, such as during testing or to prevent accidental access to a network.

**Viewing routes with iproute2**

The modern Linux toolset is `iproute2`, especially `ip route` and `ip rule`. These commands replace older `route` and `netstat` workflows in most distributions.

Common inspection commands:

```bash
ip route show
ip route show table main
ip route show table all
ip -4 route show
ip -6 route show
ip route get 8.8.8.8
```

Useful output fields include:

- **default via**: the gateway used for unmatched traffic.
- **dev**: the outgoing interface.
- **proto**: how the route was learned, such as `kernel`, `static`, or `dhcp`.
- **scope**: how far the route applies, such as `link` for directly connected networks.
- **metric**: route preference value.

Example output:

```bash
default via 192.168.10.1 dev eth0 metric 100
192.168.10.0/24 dev eth0 proto kernel scope link src 192.168.10.50
```

The `ip route get` command is especially useful because it shows the exact route the kernel would use for a destination.

```bash
ip route get 1.1.1.1
```

**Adding and deleting routes**

You can add a route temporarily with `ip route add`. This changes the live kernel routing table but usually does not survive reboot unless you also configure persistence.

```bash
ip route add 10.20.30.0/24 via 192.168.10.254 dev eth0
ip route add default via 192.168.10.1 dev eth0
ip route del 10.20.30.0/24
ip route del default
```

When you add a route, be precise about the destination, gateway, and interface. If the gateway is not on a directly connected network, the kernel cannot use it unless another route already reaches that gateway.

- Use `via` for a next-hop router.
- Use `dev` for the outgoing interface.
- Use `metric` to prefer one route over another.
- Use `src` when you need to influence the preferred source address.

Example with a source address:

```bash
ip route add 10.50.0.0/16 via 192.168.10.1 dev eth0 src 192.168.10.50
```

**Host routes, network routes, and default routes**

Route prefix length determines how specific a route is.

- A **host route** targets one IP address, usually `/32` in IPv4 or `/128` in IPv6.
- A **network route** targets a subnet, such as `/24` or `/64`.
- A **default route** matches everything else.

Examples:

```bash
ip route add 192.168.10.77/32 dev eth0
ip route add 10.10.0.0/16 via 192.168.10.1 dev eth0
ip route add default via 192.168.10.1 dev eth0
```

Use a host route when you need a special path for one destination, such as a management host or VPN endpoint. Use a network route when a whole subnet should go through a specific router.

**Policy-based routing and multiple routing tables**

Linux can use more than one routing table. This is called **policy-based routing**. The kernel evaluates rules in `ip rule` order and then consults the selected table.

This is useful when:

- You have multiple uplinks.
- You need traffic from one source IP to use a specific gateway.
- You want VPN traffic separated from normal traffic.
- You need special routing for a service or subnet.

View rules and tables:

```bash
ip rule show
ip route show table main
ip route show table 100
ip route show table all
```

A simple policy routing example:

```bash
ip route add default via 192.168.10.1 dev eth0 table 100
ip rule add from 192.168.10.50/32 table 100
ip rule show
ip route show table 100
```

In this example, traffic sourced from `192.168.10.50` uses table `100`, while other traffic can still use the main table.

You can also route by destination, firewall mark, or other selectors depending on the rule configuration.

**Persistence across reboots**

Temporary `ip route` changes disappear after reboot unless you save them in the network configuration system used by the distribution.

Common persistence methods vary by distro and network manager.

- **NetworkManager**: use `nmcli` or connection profiles.
- **systemd-networkd**: use `.network` files.
- **Debian/Ubuntu legacy**: older `/etc/network/interfaces` setups may still appear.
- **RHEL/CentOS/Rocky/Alma legacy**: older `ifcfg-*` files may still be used on systems not fully migrated.
- **Netplan**: on Ubuntu-based systems, routes may be defined in YAML and applied through NetworkManager or systemd-networkd.

NetworkManager example:

```bash
nmcli connection show
nmcli connection modify "Wired connection 1" ipv4.gateway 192.168.10.1
nmcli connection modify "Wired connection 1" +ipv4.routes "10.20.30.0/24 192.168.10.254"
nmcli connection modify "Wired connection 1" ipv4.never-default no
nmcli connection down "Wired connection 1"
nmcli connection up "Wired connection 1"
```

You can also inspect the active profile:

```bash
nmcli device show eth0
nmcli connection show "Wired connection 1"
```

Legacy Debian-style example:

```ini
auto eth0
iface eth0 inet static
    address 192.168.10.50/24
    gateway 192.168.10.1
    up ip route add 10.20.30.0/24 via 192.168.10.254 dev eth0
    down ip route del 10.20.30.0/24 via 192.168.10.254 dev eth0
```

Legacy RHEL-style example:

```ini
DEVICE=eth0
BOOTPROTO=none
ONBOOT=yes
IPADDR=192.168.10.50
PREFIX=24
GATEWAY=192.168.10.1
```

For systems using `systemd-networkd`, routes are often defined in a `.network` file:

```ini
[Match]
Name=eth0

[Network]
Address=192.168.10.50/24
Gateway=192.168.10.1

[Route]
Destination=10.20.30.0/24
Gateway=192.168.10.254
```

For Netplan, a route may look like this:

```yaml
network:
  version: 2
  ethernets:
    eth0:
      addresses:
        - 192.168.10.50/24
      gateway4: 192.168.10.1
      routes:
        - to: 10.20.30.0/24
          via: 192.168.10.254
```

**IPv6 routing and default gateways**

IPv6 routing works similarly to IPv4, but there are a few exam-relevant differences.

- The default route is `::/0`.
- A directly connected subnet is usually `/64`.
- IPv6 gateways are often reached using a **link-local** address.
- Router Advertisements (RAs) can install default routes automatically.

Example:

```bash
ip -6 route show
ip -6 route add default via fe80::1 dev eth0
ip -6 route add 2001:db8:20::/64 via fe80::1 dev eth0
```

When using a link-local gateway, you usually need the interface because link-local addresses are only valid on that link:

```bash
ip -6 route add default via fe80::1 dev eth0
```

If Router Advertisements are enabled, the system may learn a default route automatically from the router. This is common on IPv6 networks and can explain why a manual default route is unnecessary or overridden.

Check IPv6 routes and neighbors:

```bash
ip -6 route show
ip -6 neigh show
```

**Troubleshooting a bad default gateway**

A misconfigured default gateway is a common cause of “can reach local subnet but not the Internet” problems.

Typical symptoms:

- `ping` works for local hosts but fails for remote IPs.
- DNS lookups may fail because the resolver cannot reach external servers.
- `ip route get` shows the wrong gateway or interface.
- The default route is missing, duplicated, or points to an unreachable next hop.

Step-by-step troubleshooting example:

1. Check the current routes.
   ```bash
   ip route show
   ip route get 8.8.8.8
   ```
2. Confirm the gateway is on the local subnet.
   ```bash
   ip addr show eth0
   ping -c 3 192.168.10.1
   ```
3. Verify the gateway is reachable at Layer 2 and Layer 3.
   ```bash
   ip neigh show
   ping -c 3 192.168.10.1
   ```
4. Remove the bad default route if needed.
   ```bash
   ip route del default
   ```
5. Add the correct default route.
   ```bash
   ip route add default via 192.168.10.1 dev eth0
   ```
6. Re-test external reachability.
   ```bash
   ping -c 3 8.8.8.8
   ip route get 8.8.8.8
   ```

If the gateway is wrong but still on the same subnet, the host may ARP for it successfully yet still fail to reach remote networks. If the gateway is off-subnet, the kernel may reject the route or traffic may never leave the host.

> Be careful when changing routes over SSH. Removing the active default route can disconnect your session if the management path depends on it.

**Flushing routes and clearing stale entries**

`ip route flush` removes routes from a table or prefix range. This is useful when stale routes, DHCP leftovers, or test routes interfere with troubleshooting.

Examples:

```bash
ip route flush table main
ip route flush cache
ip -6 route flush table main
```

Use flushing carefully because it can remove working routes and temporarily break connectivity. In practice, you often flush only a specific prefix or table rather than the entire main table.

Examples of targeted cleanup:

```bash
ip route flush 10.20.30.0/24
ip route flush table 100
```

If a DHCP client or NetworkManager keeps re-adding a route, fix the persistent profile rather than repeatedly flushing the live table.

**Legacy verification tools**

Older commands still appear in documentation and exam contexts, even though `iproute2` is preferred.

- `route -n` shows the IPv4 routing table in numeric form.
- `netstat -rn` shows routing tables on older systems.
- `netstat -r` may display routes with name resolution.
- `ss -r` can show routing-related socket information on some systems, though it is not a primary route viewer.

Examples:

```bash
route -n
netstat -rn
ss -r
```

Use these when you encounter older servers or when a question references legacy output. For modern troubleshooting, prefer `ip route` and `ip rule`.

**Practical troubleshooting workflow**

A reliable workflow helps you isolate whether the issue is addressing, routing, or name resolution.

1. Verify the interface and address.
   ```bash
   ip addr show
   ip link show
   ```
2. Verify the local route and default route.
   ```bash
   ip route show
   ip -6 route show
   ```
3. Test the gateway.
   ```bash
   ping -c 3 192.168.10.1
   ping -c 3 fe80::1%eth0
   ```
4. Test a remote IP, not a hostname.
   ```bash
   ping -c 3 8.8.8.8
   ping -c 3 2001:4860:4860::8888
   ```
5. Check DNS only after routing works.
   ```bash
   resolvectl status
   cat /etc/resolv.conf
   ```
6. Inspect policy routing if the path is unexpected.
   ```bash
   ip rule show
   ip route show table all
   ```

**Key exam takeaways**

- The **default route** handles traffic with no more specific match.
- `ip route` is the primary tool for viewing and changing routes.
- `ip rule` and multiple tables enable **policy-based routing**.
- Host routes, network routes, and blackhole/unreachable routes are all valid route types.
- Persistence depends on the network stack in use, especially NetworkManager, systemd-networkd, Netplan, or legacy config files.
- IPv6 often uses link-local gateways and may learn defaults from Router Advertisements.
- `ip route flush` is useful for cleanup, but use it carefully.
- Legacy tools like `route -n` and `netstat -rn` still matter for older systems and exam questions.
