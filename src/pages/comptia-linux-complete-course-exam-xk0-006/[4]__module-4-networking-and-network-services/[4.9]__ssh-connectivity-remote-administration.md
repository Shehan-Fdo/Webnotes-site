**SSH fundamentals**

<abbr title="Secure Shell">SSH</abbr> is the standard encrypted protocol for remote login, command execution, file transfer, and tunneling on Linux systems. It replaces insecure tools such as Telnet because it protects credentials and session data in transit.

SSH uses a client-server model:

- The **client** initiates the connection, usually with `ssh`.
- The **server** listens for incoming connections through `sshd`, the OpenSSH daemon.
- Authentication can use passwords, public keys, certificates, or other methods configured on the server.

The default SSH port is `22/tcp`, but you can change it for hardening or policy reasons. Changing the port reduces noise from automated scans, but it does not replace real security controls.

**Core client usage**

The basic connection syntax is:

```bash
ssh user@server.example.com
ssh -p 2222 user@server.example.com
ssh -i ~/.ssh/id_ed25519 user@server.example.com
```

Common client options include:

- `-p` to specify a non-default port.
- `-i` to select a private key file.
- `-l` to specify the login name when you do not want to use `user@host`.
- `-v`, `-vv`, and `-vvv` for increasing debug verbosity.
- `-X` or `-Y` for X11 forwarding when permitted by policy.

A useful client configuration file is `~/.ssh/config`. It lets you define host aliases and default connection behavior:

```sshconfig
Host prod-web
    HostName 203.0.113.10
    User admin
    Port 2222
    IdentityFile ~/.ssh/id_ed25519
    IdentitiesOnly yes
```

This is especially helpful when you manage many systems or use different keys for different environments.

**Key-based authentication**

Public key authentication is the preferred method for remote administration because it is stronger and easier to automate than passwords. You keep the private key on your client and place the public key on the server.

Typical key types include:

- `ed25519` for modern, compact, and fast keys.
- `rsa` for compatibility with older systems, though modern deployments often prefer Ed25519.
- `ecdsa` in some environments, though Ed25519 is often the simpler default choice.

Generate a key pair with:

```bash
ssh-keygen -t ed25519 -C "admin@laptop"
```

Then copy the public key to the server:

```bash
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@server.example.com
```

If `ssh-copy-id` is unavailable, you can append the public key manually:

```bash
cat ~/.ssh/id_ed25519.pub | ssh user@server.example.com 'umask 077; mkdir -p ~/.ssh; cat >> ~/.ssh/authorized_keys'
```

Important key files and permissions:

- `~/.ssh/id_ed25519` is the private key and must remain secret.
- `~/.ssh/id_ed25519.pub` is the public key and can be shared.
- `~/.ssh/authorized_keys` on the server contains allowed public keys.
- `~/.ssh/known_hosts` stores host keys for servers you trust.

> Do not treat `ssh-keyscan` as proof of identity. It retrieves a host key, but you still need an out-of-band fingerprint check before trusting a new host.

**SSH agent usage**

The SSH agent holds decrypted private keys in memory so you do not need to re-enter a passphrase for every connection. This is useful when you connect to multiple systems or use Git over SSH.

Start the agent and add a key:

```bash
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519
ssh-add -l
```

Agent behavior to remember:

- The agent runs in your user session and stores unlocked keys temporarily.
- `ssh-add -l` lists loaded keys.
- `ssh-add -D` removes all keys from the agent.
- A passphrase protects the private key at rest, while the agent reduces repeated prompts during active use.

Agent forwarding lets a remote server use your local agent for onward SSH connections:

```bash
ssh -A jumpbox.example.com
```

Use it carefully:

- It is convenient for jump hosts and bastion workflows.
- It increases risk if the intermediate host is compromised.
- Prefer it only when necessary and only on trusted systems.

**Server-side configuration**

The main server configuration file is usually `/etc/ssh/sshd_config`. Client-side defaults live in `/etc/ssh/ssh_config`, while per-user client settings live in `~/.ssh/config`.

Common server directives include:

```sshconfig
Port 2222
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
AllowUsers alice bob
AllowGroups sshadmins
Banner /etc/issue.net
```

Key directives and why they matter:

- `Port` changes the listening port.
- `PermitRootLogin no` blocks direct root login and encourages privilege escalation with `sudo`.
- `PasswordAuthentication no` forces key-based access when you are ready to disable passwords.
- `PubkeyAuthentication yes` enables public key login.
- `AllowUsers` restricts SSH access to named accounts.
- `AllowGroups` restricts SSH access to members of approved groups.
- `Banner` displays a legal or policy notice before authentication.

Other useful hardening settings may include:

- `PermitEmptyPasswords no`
- `X11Forwarding no` if you do not need GUI forwarding
- `MaxAuthTries` to limit brute-force attempts
- `ClientAliveInterval` and `ClientAliveCountMax` to manage idle sessions
- `LoginGraceTime` to limit how long a login attempt can remain open

After editing `sshd_config`, validate and reload safely:

```bash
sudo sshd -t
sudo systemctl reload sshd
```

On some distributions, the service name is `ssh` instead of `sshd`:

```bash
sudo systemctl reload ssh
```

**Port changes and access control**

If you change the SSH port, update every layer that enforces network access:

- The SSH daemon must listen on the new port.
- The firewall must allow the new port.
- SELinux may need the port labeled for SSH if it is enforcing.

Example firewall and SELinux steps:

```bash
sudo firewall-cmd --permanent --add-port=2222/tcp
sudo firewall-cmd --reload
sudo semanage port -a -t ssh_port_t -p tcp 2222
```

If the port already exists in SELinux, modify it instead of adding it:

```bash
sudo semanage port -m -t ssh_port_t -p tcp 2222
```

Verify the daemon is listening:

```bash
ss -tlnp | grep sshd
sudo systemctl status sshd
```

If the service is named differently:

```bash
ss -tlnp | grep ':2222'
sudo systemctl status ssh
```

**Host keys and known_hosts**

SSH server identity is verified with host keys. The first time you connect, the client records the server’s host key in `~/.ssh/known_hosts`. Later connections compare the presented key to the stored one.

If a host key changes unexpectedly, SSH warns you because the change could indicate a man-in-the-middle attack or a legitimate rebuild.

Useful commands:

```bash
ssh-keygen -R server.example.com
ssh-keygen -R 203.0.113.10
ssh-keygen -F server.example.com
```

Practical workflow for a changed host key:

1. Confirm the change with an out-of-band source, such as console access or a trusted fingerprint from the admin team.
2. Remove the stale entry with `ssh-keygen -R`.
3. Reconnect only after you verify the new fingerprint independently.

`ssh-keyscan` can help collect host keys for inventory or automation, but it does not authenticate the server by itself:

```bash
ssh-keyscan -t ed25519 server.example.com
```

Use it for convenience, not trust establishment.

**Remote administration patterns**

SSH is more than interactive login. It is a transport for administration tasks.

Run a single remote command:

```bash
ssh admin@server.example.com 'uname -r && uptime'
```

Copy files with `scp` or `sftp`:

```bash
scp report.txt admin@server.example.com:/tmp/
sftp admin@server.example.com
```

Modern OpenSSH `scp` commonly uses the SFTP protocol by default. If you need legacy SCP protocol behavior for compatibility, use `scp -O`. This matters when you troubleshoot path handling or older automation.

Useful `scp` flags:

- `-r` for recursive directory copy.
- `-P` for the remote port in `scp` specifically.
- `-p` to preserve timestamps and modes.
- `-O` to force legacy SCP protocol when required.

Example:

```bash
scp -P 2222 -r ./site admin@server.example.com:/var/www/
```

For interactive file transfer, `sftp` is often safer and more predictable than legacy SCP semantics:

```bash
sftp -P 2222 admin@server.example.com
```

**Tunneling and port forwarding**

SSH can securely forward traffic through an encrypted channel. This is useful for database access, internal services, and jump-host workflows.

Local forwarding exposes a remote service on your local machine:

```bash
ssh -L 8080:127.0.0.1:80 user@server.example.com
```

Remote forwarding exposes a local service on the remote machine:

```bash
ssh -R 9090:127.0.0.1:3000 user@server.example.com
```

Dynamic forwarding creates a SOCKS proxy:

```bash
ssh -D 1080 user@server.example.com
```

Common use cases:

- Accessing an internal web app through a bastion host.
- Reaching a database that should not be publicly exposed.
- Creating a temporary secure proxy for troubleshooting.

**Jump hosts and bastions**

A jump host is an intermediate system that you connect through to reach a protected target. This is common in segmented networks and production environments.

You can chain connections manually:

```bash
ssh -J jumpuser@jump.example.com targetuser@target.internal
```

Or define it in `~/.ssh/config`:

```sshconfig
Host jump
    HostName jump.example.com
    User jumpuser

Host target
    HostName target.internal
    User targetuser
    ProxyJump jump
```

This approach improves usability and supports controlled access paths.

**Troubleshooting SSH**

When SSH fails, isolate the problem by checking the network, the daemon, authentication, and policy controls.

A practical troubleshooting sequence:

1. Confirm name resolution and reachability.
   ```bash
   ping -c 1 server.example.com
   nc -vz server.example.com 22
   ```
2. Check the SSH service and listening port on the server.
   ```bash
   sudo systemctl status sshd
   ss -tlnp | grep sshd
   ```
3. Review logs for authentication or configuration errors.
   ```bash
   sudo journalctl -u sshd -b
   sudo journalctl -u ssh -b
   ```
4. Test the client with verbose output.
   ```bash
   ssh -vvv user@server.example.com
   ```
5. Verify permissions on key files and SSH directories.
   ```bash
   chmod 700 ~/.ssh
   chmod 600 ~/.ssh/authorized_keys
   chmod 600 ~/.ssh/id_ed25519
   ```

Common causes of failure:

- Wrong username or host name.
- Firewall blocking the SSH port.
- `sshd` not running or listening on the expected port.
- Incorrect file permissions on `~/.ssh` or `authorized_keys`.
- SELinux blocking a non-default port.
- A mismatched or stale host key in `known_hosts`.
- Server policy denying the account through `AllowUsers`, `AllowGroups`, or `PermitRootLogin`.

If you change the SSH port, remember to test the new port explicitly:

```bash
ssh -p 2222 user@server.example.com
nc -vz server.example.com 2222
```

**Security best practices**

SSH hardening is about reducing attack surface and limiting blast radius.

Recommended practices:

- Use key-based authentication instead of passwords.
- Protect private keys with passphrases.
- Disable direct root login.
- Restrict access with `AllowUsers` or `AllowGroups`.
- Change the default port only as a supplemental control.
- Keep OpenSSH updated.
- Use `sudo` for privileged actions instead of logging in as root.
- Limit agent forwarding to trusted jump hosts only.
- Review `known_hosts` warnings instead of bypassing them.

A simple hardened server profile might look like this:

```sshconfig
Port 2222
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
AllowGroups sshadmins
X11Forwarding no
Banner /etc/issue.net
```

```bash
# Generate a key
ssh-keygen -t ed25519

# Copy a key to a server
ssh-copy-id user@server.example.com

# Connect on a custom port
ssh -p 2222 user@server.example.com

# Start agent and add key
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519

# Check listening ports
ss -tlnp | grep ssh

# Validate sshd config
sudo sshd -t

# Reload SSH service
sudo systemctl reload sshd

# Remove stale host key
ssh-keygen -R server.example.com
```
