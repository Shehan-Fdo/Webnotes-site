**DNS resolution overview**

<abbr title="Domain Name System">DNS</abbr> translates human-readable names like `www.example.com` into IP addresses such as `93.184.216.34`. On Linux, name resolution is not just “ask DNS”; it is a layered process that can consult local files, caching services, and one or more resolvers before a query ever leaves the host.

The practical goal is simple: when an application needs a hostname, the system returns an address quickly and consistently. The exam focus is usually on the order of lookup sources, the role of resolver configuration, and the tools you use to verify what is happening.

- **Hostname resolution** maps names to addresses and sometimes addresses back to names.
- **Forward lookup** resolves a name to an IP address.
- **Reverse lookup** resolves an IP address to a name using <abbr title="Pointer">PTR</abbr> records.
- **Recursive resolution** means a resolver does the work of finding the final answer for the client.
- **Authoritative DNS** is the source that owns the zone data for a domain.

**How Linux chooses a name source**

Linux typically follows the <abbr title="Name Service Switch">NSS</abbr> configuration in `/etc/nsswitch.conf` to decide where to look for names. This file controls the lookup order for hosts, users, groups, and other databases.

A common hosts line looks like this:

```ini
hosts: files dns
```

That means:

- Check local static entries in `/etc/hosts` first.
- If no match exists, query DNS next.

This order matters because local entries can override DNS. That is useful for maintenance windows, lab environments, or temporary overrides, but it can also cause confusion when a hostname resolves differently than expected.

**Local resolution sources**

Before DNS is queried, Linux may use local sources:

- `/etc/hosts`
  - Static hostname-to-IP mappings.
  - Useful for bootstrapping, small environments, and overrides.
- `/etc/nsswitch.conf`
  - Controls the lookup sequence.
  - Can include sources such as `files`, `dns`, `myhostname`, or `resolve` depending on the distribution.
- Caching services
  - `systemd-resolved`, `nscd`, or a local DNS forwarder may answer from cache.

A typical `/etc/hosts` entry looks like this:

```ini
127.0.0.1   localhost
192.0.2.10  web01.example.com web01
```

Use `/etc/hosts` carefully:

> If you place a hostname in `/etc/hosts`, it can mask the real DNS record and make troubleshooting misleading.

**Resolver configuration**

The classic resolver configuration file is `/etc/resolv.conf`. It tells the system which DNS servers to use and which search domains to apply.

Example:

```ini
nameserver 192.0.2.53
nameserver 192.0.2.54
search example.com lab.example.com
options timeout:2 attempts:3
```

Key fields:

- **nameserver**
  - IP address of a DNS server to query.
  - Multiple entries provide redundancy.
- **search**
  - Domains appended automatically when you query an unqualified name.
  - For example, `web01` may be tried as `web01.example.com`.
- **options**
  - Resolver behavior such as timeout and retry count.

Modern systems may generate `/etc/resolv.conf` dynamically. On some distributions, `systemd-resolved` manages it indirectly, so editing the file manually may not persist. In those cases, you configure DNS through the network manager, `systemd-resolved`, or the interface configuration tool used by the distribution.

**What happens during resolution**

A typical lookup follows this path:

1. An application calls a resolver function such as `getaddrinfo()`.
2. The system checks `nsswitch.conf` for the `hosts` lookup order.
3. If `files` is listed first, `/etc/hosts` is checked.
4. If no local match exists, the resolver sends a query to the configured DNS server.
5. The DNS server may answer from cache or perform recursion.
6. The final IP address is returned to the application.

This layered design explains why a hostname can resolve even when external DNS is unavailable: a local file or cache may already contain the answer.

**DNS record types you should know**

DNS stores different record types for different purposes.

- **A**
  - Maps a hostname to an IPv4 address.
- **AAAA**
  - Maps a hostname to an IPv6 address.
- **CNAME**
  - Creates an alias from one name to another name.
- **MX**
  - Identifies mail exchangers for a domain.
- **NS**
  - Identifies authoritative name servers for a zone.
- **PTR**
  - Supports reverse lookups from IP to name.
- **SOA**
  - Start of Authority record; contains zone metadata.
- **TXT**
  - Stores arbitrary text, often used for verification and policy records.

A simple zone fragment might look like this:

```dns
$ORIGIN example.com.
@   IN SOA ns1.example.com. admin.example.com. (
        2026042601 ; serial
        3600       ; refresh
        900        ; retry
        1209600    ; expire
        300 )      ; minimum

@   IN NS  ns1.example.com.
@   IN NS  ns2.example.com.
www IN A   192.0.2.80
mail IN A   192.0.2.25
@   IN MX 10 mail.example.com.
```

**Forward and reverse lookups**

Forward lookups are the most common: you type a name, and DNS returns an address. Reverse lookups are useful for logging, access control, and diagnostics.

Reverse IPv4 lookups use the `in-addr.arpa` namespace. Reverse IPv6 lookups use `ip6.arpa`.

Example reverse record:

```dns
80.2.0.192.in-addr.arpa. IN PTR www.example.com.
```

Reverse lookups are not always configured, so a missing PTR record does not necessarily mean the forward record is wrong.

**Search domains and unqualified names**

An unqualified name is a hostname without a full domain, such as `web01`. The resolver may append search domains from `/etc/resolv.conf` or DHCP-provided settings.

Example behavior:

- Query `web01`
- Try `web01.example.com`
- Try `web01.lab.example.com`

This is convenient in internal networks, but it can also create ambiguity if multiple search domains exist. For production troubleshooting, use fully qualified domain names <abbr title="Fully Qualified Domain Name">FQDN</abbr>s such as `web01.example.com`.

**Common Linux DNS tools**

Use these commands to inspect and test resolution:

```bash
getent hosts www.example.com
dig www.example.com A
dig +short www.example.com
dig -x 192.0.2.80
host www.example.com
resolvectl query www.example.com
```

What each tool is good for:

- **getent**
  - Queries NSS, so it reflects the same lookup path applications use.
  - Useful for checking `/etc/hosts` plus DNS behavior together.
- **dig**
  - Detailed DNS query tool.
  - Best for troubleshooting records, servers, and response details.
- **host**
  - Simple DNS lookup utility.
  - Good for quick checks.
- **resolvectl**
  - Useful on systems using `systemd-resolved`.
  - Shows resolver state and cache-aware queries.

A practical comparison:

```bash
getent hosts web01
dig web01.example.com
resolvectl status
cat /etc/nsswitch.conf
cat /etc/resolv.conf
```

**Troubleshooting workflow**

When DNS resolution fails, work from the local system outward.

1. Check whether the name is in `/etc/hosts`.
2. Verify the `hosts:` line in `/etc/nsswitch.conf`.
3. Inspect `/etc/resolv.conf` or the active resolver manager.
4. Test the configured DNS server directly with `dig @server name`.
5. Confirm network reachability to port 53 over UDP and, if needed, TCP.
6. Check for caching issues or stale records.
7. Verify the record exists on the authoritative server if you administer the zone.

Useful commands:

```bash
cat /etc/hosts
grep '^hosts:' /etc/nsswitch.conf
cat /etc/resolv.conf
dig @192.0.2.53 www.example.com
dig @192.0.2.53 www.example.com +tcp
ping -c 1 192.0.2.53
ss -lunp | grep ':53'
ss -ltnp | grep ':53'
```

Important troubleshooting notes:

- **UDP 53** is the normal transport for most DNS queries.
- **TCP 53** is used for large responses, zone transfers, and some fallback cases.
- Firewalls can block either transport and cause intermittent failures.
- A successful `ping` does not prove DNS works; it only proves basic IP reachability.

**Caching and performance**

Caching reduces repeated lookups and improves response time. A cache may exist in the local resolver, on a recursive DNS server, or in an application.

Benefits of caching:

- Faster repeated lookups.
- Lower load on upstream DNS servers.
- Better resilience during brief upstream outages.

Risks of caching:

- Stale data after a record change.
- Confusing results when TTLs <abbr title="Time To Live">TTL</abbr> have not expired.
- Different answers from different resolvers during propagation.

If you change a DNS record, remember that clients may continue to use cached data until the TTL expires. Lower TTL values help during migrations, but they also increase query volume.

**Recursive, iterative, and authoritative roles**

DNS resolution often involves several server roles.

- **Recursive resolver**
  - Accepts a client query and finds the final answer.
  - Common in enterprise networks and ISP infrastructure.
- **Root server**
  - Directs the resolver to the correct top-level domain servers.
- **TLD server**
  - Points the resolver to the authoritative servers for the domain.
- **Authoritative server**
  - Returns the actual record data for the zone.

In practice, your Linux host usually talks to a recursive resolver, not directly to root or authoritative servers. That is why `/etc/resolv.conf` usually lists internal or ISP DNS servers rather than root server addresses.

**Practical examples**

Check whether a hostname resolves through NSS:

```bash
getent ahosts example.com
```

Query a specific DNS server directly:

```bash
dig @192.0.2.53 example.com A
dig @192.0.2.53 example.com AAAA
```

Inspect reverse DNS:

```bash
dig -x 192.0.2.80
```

See resolver configuration on a systemd-based host:

```bash
resolvectl status
resolvectl dns
resolvectl domain
```

Test an override in `/etc/hosts`:

```ini
192.0.2.200 app.internal.example
```

Then verify:

```bash
getent hosts app.internal.example
dig app.internal.example
```

If `getent` returns the `/etc/hosts` address but `dig` returns a different DNS answer, that is expected: `getent` follows NSS, while `dig` queries DNS directly.

**Common problems and causes**

- **Wrong lookup order**
  - `dns files` instead of `files dns` can change behavior in unexpected ways.
- **Stale `/etc/hosts` entry**
  - Local override no longer matches the real service address.
- **Bad resolver server**
  - The configured DNS server is unreachable or misconfigured.
- **Missing search domain**
  - Short names fail because the expected domain is not appended.
- **Firewall filtering**
  - DNS traffic on port 53 is blocked.
- **Caching delay**
  - New records are not visible everywhere yet.
- **Split DNS**
  - Internal and external clients receive different answers by design.

**Exam-focused takeaways**

- **`/etc/nsswitch.conf`** controls the order of name service lookups.
- **`/etc/hosts`** can override DNS for local resolution.
- **`/etc/resolv.conf`** defines DNS servers and search domains, though it may be managed dynamically.
- **`getent`** is the best quick check for what applications will likely see.
- **`dig`** is the best tool for detailed DNS troubleshooting.
- **PTR records** support reverse lookups, but they are not always present.
- **Caching and TTL** explain why DNS changes do not appear instantly.

**Minimal troubleshooting checklist**

1. Confirm the hostname spelling.
2. Check `/etc/hosts`.
3. Check `hosts:` in `/etc/nsswitch.conf`.
4. Check resolver settings in `/etc/resolv.conf` or `resolvectl`.
5. Query the DNS server directly with `dig @server`.
6. Verify network access to port 53.
7. Consider cache and TTL effects.
