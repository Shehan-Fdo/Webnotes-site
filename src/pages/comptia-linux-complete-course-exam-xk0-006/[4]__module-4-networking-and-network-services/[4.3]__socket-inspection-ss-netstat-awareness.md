**Socket inspection overview**

A socket is an endpoint for network communication. On Linux, socket inspection helps you answer practical questions such as which process is listening on a port, whether a connection is established, and which protocol family is in use. This is essential for troubleshooting services, validating firewall behavior, and confirming that daemons bind to the expected interfaces.

The modern tool for this job is `ss` from the `iproute2` suite. It is faster and more flexible than legacy `netstat`, especially on systems with many connections. `netstat` still appears in older documentation and scripts, so you should recognize it, but prefer `ss` for current administration.

- **Listening sockets** show services waiting for inbound connections.
- **Established sockets** show active sessions between peers.
- **Local address and port** identify where the service binds.
- **Peer address and port** identify the remote endpoint.

**Why socket inspection matters**

You use socket inspection when a service is running but unreachable, when a port conflict prevents startup, or when you need to confirm that a daemon is bound only to `127.0.0.1` instead of all interfaces. It also helps you distinguish between a process that is listening and one that is merely installed or enabled.

Common troubleshooting scenarios include:

- A web server starts, but nothing responds on port `80` or `443`.
- An application binds to the wrong interface and is not reachable remotely.
- Two services attempt to use the same port.
- A firewall rule appears correct, but the service is not actually listening.

**`ss` basics**

`ss` displays socket statistics and connection state. It can show TCP, UDP, UNIX domain sockets, routing-related socket information, and process ownership when run with sufficient privileges.

Useful options include:

- `-t` for TCP sockets
- `-u` for UDP sockets
- `-l` for listening sockets
- `-n` for numeric output without DNS or service-name resolution
- `-p` for process information
- `-a` for all sockets, including listening and non-listening
- `-s` for summary statistics
- `-x` for UNIX domain sockets

A common inspection pattern is:

```bash
ss -tuln
```

This shows TCP and UDP listening sockets using numeric addresses and ports. It is one of the most useful quick checks for service availability.

To include process ownership, use:

```bash
sudo ss -tulpn
```

This helps you map a port to the process and PID (process ID) that owns it.

**Reading `ss` output**

A typical `ss` listing includes columns such as:

- **Netid**: protocol family or socket type
- **State**: socket state such as `LISTEN`, `ESTAB`, or `UNCONN`
- **Recv-Q**: bytes waiting to be read
- **Send-Q**: bytes waiting to be sent
- **Local Address:Port**: local bind address and port
- **Peer Address:Port**: remote endpoint, if applicable
- **Process**: owning process details when requested

Example:

```bash
sudo ss -tulpn
```

```text
Netid State  Recv-Q Send-Q Local Address:Port  Peer Address:Port Process
tcp   LISTEN 0      128    0.0.0.0:22         0.0.0.0:*       users:(("sshd",pid=812,fd=3))
tcp   LISTEN 0      511    127.0.0.1:5432      0.0.0.0:*       users:(("postgres",pid=1042,fd=5))
udp   UNCONN 0      0      0.0.0.0:68         0.0.0.0:*       users:(("dhclient",pid=601,fd=6))
```

Interpretation:

- `0.0.0.0:22` means SSH listens on all IPv4 interfaces.
- `127.0.0.1:5432` means PostgreSQL listens only on localhost.
- `UNCONN` for UDP is normal because UDP is connectionless.

> `ss` output can differ slightly by distribution and version. Focus on the socket state, bind address, and owning process rather than exact formatting.

**Filtering with `ss`**

You can narrow output to the sockets you care about.

Examples:

```bash
ss -t state established
```

```bash
ss -ltn sport = :80
```

```bash
ss -tn dst 192.0.2.10
```

```bash
ss -x src /run/systemd/private
```

Practical uses:

- Check only established TCP sessions during an incident.
- Confirm whether port `80` is listening.
- Inspect traffic to a specific remote host.
- Review UNIX domain sockets used by local services.

You can also combine filters with process output:

```bash
sudo ss -ltnp 'sport = :443'
```

This is useful when multiple daemons or containers may be involved.

**`ss` and service troubleshooting**

When a service fails to respond, use `ss` to verify the network layer before moving to application logs.

A simple workflow is:

1. Confirm the service process is running.
2. Check whether the expected port is listening.
3. Verify the bind address.
4. Confirm the firewall or security policy allows access.
5. Test connectivity from the local host and a remote host.

Example:

```bash
systemctl status nginx
sudo ss -ltnp | grep ':80\|:443'
curl -I http://127.0.0.1
```

If `nginx` is active but `ss` shows only `127.0.0.1:80`, remote clients cannot connect unless a reverse proxy or tunnel is in place. If `ss` shows `0.0.0.0:80` but remote access still fails, the issue is likely elsewhere, such as firewall rules or routing.

**TCP states you should recognize**

`ss` is especially useful because it exposes TCP state transitions.

Common states include:

- **LISTEN**: waiting for inbound connections
- **ESTAB**: connection established
- **SYN-SENT**: client has sent a connection request
- **SYN-RECV**: server has received a request and is responding
- **TIME-WAIT**: connection recently closed, waiting to ensure delayed packets expire
- **CLOSE-WAIT**: remote side closed, local application has not closed yet

Why this matters:

- A large number of `TIME-WAIT` sockets can be normal after heavy short-lived traffic.
- Persistent `CLOSE-WAIT` sockets may indicate an application that is not closing connections properly.
- `SYN-RECV` buildup can suggest connection pressure or a possible flood.

Example:

```bash
ss -tan state time-wait
```

```bash
ss -tan state close-wait
```

**UDP and UNIX sockets**

UDP does not establish a session in the same way TCP does, so you usually inspect bound sockets rather than connection state.

Example:

```bash
ss -uln
```

UNIX domain sockets are local IPC (inter-process communication) endpoints used by services such as databases, systemd components, and desktop services.

Example:

```bash
ss -xln
```

This is useful when a service communicates locally without using TCP/IP.

**`netstat` awareness**

`netstat` is the older tool traditionally used for network statistics and socket inspection. It comes from the `net-tools` package, which is considered legacy on many modern Linux distributions. You may still encounter it in older scripts, training material, or troubleshooting guides.

Common `netstat` patterns include:

```bash
netstat -tuln
sudo netstat -tulpn
netstat -an
```

What to know:

- `netstat` and `ss` can both show listening and established sockets.
- `ss` is generally faster and more current.
- Some minimal installations may not include `netstat` by default.
- If you see `netstat` in documentation, translate the intent to `ss` when possible.

Equivalent examples:

| Task | `ss` | `netstat` |
|---|---|---|
| Show listening TCP/UDP sockets | `ss -tuln` | `netstat -tuln` |
| Show listening sockets with processes | `sudo ss -tulpn` | `sudo netstat -tulpn` |
| Show all TCP sockets numerically | `ss -tan` | `netstat -tan` |

> Prefer `ss` in new work. Use `netstat` only when maintaining legacy systems or following older instructions that have not been updated.

**Practical command patterns**

Use these commands frequently during troubleshooting:

```bash
# Show all listening TCP and UDP sockets
ss -tuln

# Show listening sockets with process details
sudo ss -tulpn

# Show established TCP sessions
ss -tan state established

# Show sockets for a specific port
ss -ltn 'sport = :22'

# Show summary statistics
ss -s
```

If you need to identify what is bound to a port and then inspect the owning service:

```bash
sudo ss -ltnp 'sport = :8080'
ps -fp <PID>
systemctl status <service-name>
```

This sequence helps you move from socket to process to service manager.

**Binding, interfaces, and reachability**

A service can listen on different addresses, and that choice affects accessibility.

Common bind patterns:

- `127.0.0.1` or `::1`: local-only access
- `0.0.0.0` or `::`: all IPv4 or IPv6 interfaces
- Specific interface IP: only that interface can accept connections

Example configuration concept:

```ini
# Example service bind setting
ListenAddress=127.0.0.1
```

If a daemon binds to localhost, remote clients cannot reach it directly. This is often intentional for databases, admin services, and local proxies. If you need remote access, you usually change the bind address carefully and then verify firewall and authentication controls.

**IPv4 and IPv6 considerations**

`ss` can show both IPv4 and IPv6 sockets. On dual-stack systems, a service may listen on an IPv6 wildcard address and still accept IPv4 connections depending on kernel and application settings.

Examples:

```bash
ss -ltn
ss -ltn6
ss -ltn4
```

Why this matters:

- A service may appear absent if you only inspect one protocol family.
- IPv6 wildcard binding can affect how IPv4 traffic is handled.
- Troubleshooting should include both families when connectivity is unclear.

**Process ownership and permissions**

The `-p` option shows process information, but you often need elevated privileges to see full details. Without `sudo`, output may be incomplete or hidden depending on system policy.

Example:

```bash
sudo ss -ltnp
```

This is useful when:

- Multiple services could own the same port on different addresses.
- You need the PID for a targeted restart or investigation.
- You are checking whether a container runtime or proxy is handling the socket.

**Common exam and admin takeaways**

- Use `ss` for modern socket inspection.
- Add `-n` to avoid name resolution delays and ambiguity.
- Add `-p` when you need the owning process.
- Use `-l` to focus on listening sockets.
- Use `netstat` awareness to read older documentation and legacy systems.
- Interpret socket state, bind address, and process ownership together rather than in isolation.

**Quick reference**

```bash
ss -tuln        # listening TCP/UDP sockets
ss -tulpn       # listening sockets with process info
ss -tan         # all TCP sockets
ss -tan state established
ss -s           # summary statistics
netstat -tuln   # legacy equivalent
```

**Troubleshooting checklist**

1. Confirm the service is running with `systemctl status <service>`.
2. Check whether the expected port is listening with `ss -tulpn`.
3. Verify the bind address is correct for local or remote access.
4. Confirm the protocol family matches your test path, including IPv6.
5. Check firewall, routing, and SELinux or AppArmor policy if the socket looks correct.
6. Use `curl`, `nc`, or `telnet`-style tests to validate connectivity from the client side.

**Practical example: diagnosing a web service**

Suppose a web application should listen on port `8080`, but remote clients cannot connect.

```bash
systemctl status myapp
sudo ss -ltnp 'sport = :8080'
curl -I http://127.0.0.1:8080
```

Possible outcomes:

- `LISTEN 127.0.0.1:8080`: service is local-only.
- `LISTEN 0.0.0.0:8080`: service is reachable on all IPv4 interfaces if firewall permits.
- No output: service is not listening, or it uses a different port.

That sequence quickly separates application failure from network exposure issues.
