## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Network troubleshooting mindset**

A good troubleshooting workflow reduces guesswork. You start by defining the symptom, then narrow the fault domain from the user or application layer down through the network stack. In Linux environments, this usually means checking local host state first, then link, IP configuration, routing, name resolution, and finally remote service reachability.

A disciplined workflow helps you avoid changing multiple variables at once. That matters because network issues often overlap: a service may appear down because DNS fails, a route is missing, a firewall blocks traffic, or the interface never obtained an address.

- **Scope** the problem before touching configuration.
- **Reproduce** the issue with a known test.
- **Isolate** the layer where failure begins.
- **Verify** each fix with a repeatable command.

**A practical troubleshooting sequence**

Use a top-down or bottom-up approach depending on the symptom. If a user cannot reach a website, start with the application and name resolution. If a host cannot even ping its gateway, start with the local interface and routing table.

1. Confirm the exact failure.
   - Ask what works and what does not.
   - Identify the source host, destination, protocol, and port.
   - Determine whether the issue is constant or intermittent.
2. Check local interface status.
   - Verify the interface is up and has the expected address.
   - Confirm link state, duplex, and speed if relevant.
3. Validate routing.
   - Ensure the default route and specific routes exist.
   - Check for asymmetric or conflicting routes.
4. Test name resolution.
   - Compare DNS lookup results with direct IP connectivity.
   - Verify resolver configuration and search domains.
5. Test remote reachability.
   - Use `ping`, `traceroute`, `tracepath`, `nc`, or `curl` depending on the service.
6. Inspect firewall and security controls.
   - Check host firewall rules, SELinux context issues, and upstream ACLs if applicable.
7. Review logs and counters.
   - Look for interface errors, dropped packets, and service logs.
8. Apply one change at a time and retest.

**Core Linux commands for each layer**

Linux provides multiple tools that map cleanly to the troubleshooting workflow. Use the modern `ip` suite first, then fall back to older tools only when needed.

```bash
ip link show
ip addr show
ip route show
ss -tulpen
ping -c 4 192.0.2.1
traceroute 198.51.100.10
dig example.com
resolvectl status
curl -I https://example.com
nc -vz 198.51.100.10 443
```

- `ip link show` checks whether the interface is administratively up and whether the kernel sees the device.
- `ip addr show` confirms assigned IPv4 and IPv6 addresses.
- `ip route show` reveals the routing table and default gateway.
- `ss -tulpen` lists listening sockets and can confirm whether a service is bound to the expected address and port.
- `ping` tests basic IP reachability and round-trip time.
- `traceroute` and `tracepath` help identify where packets stop.
- `dig` and `resolvectl` help separate DNS issues from transport issues.
- `curl` and `nc` test application-layer connectivity more accurately than ICMP alone.

**Layer-by-layer checks**

The network stack is easiest to troubleshoot when you move from local to remote in a consistent order. Each layer answers a different question.

**1. Physical and link layer**

At the lowest level, verify that the interface is present and active. A cable issue, disabled port, or failed NIC can look like a higher-level problem.

```bash
ip link show eth0
ethtool eth0
```

- **Link state** tells you whether the interface detects carrier.
- **Speed and duplex** mismatches can cause poor performance or packet loss.
- **Driver errors** may appear in `dmesg` or `journalctl -k`.

Useful checks:

```bash
dmesg | grep -iE 'eth0|link up|link down|firmware|error'
journalctl -k -b | grep -i network
```

**2. IP addressing**

If the interface is up but unreachable, confirm that it has the correct address, prefix length, and gateway. A wrong subnet mask can make local peers appear remote, and a missing address prevents any IP communication.

```bash
ip addr show eth0
```

Look for:

- Correct IPv4 address and prefix, such as `192.168.10.25/24`
- Correct IPv6 address and scope
- No duplicate or stale addresses
- Expected state such as `UP` and `LOWER_UP`

If DHCP is involved, verify lease acquisition:

```bash
journalctl -u NetworkManager -b
journalctl -u systemd-networkd -b
```

**3. Routing**

Routing determines where packets go after leaving the host. A missing default route is a common cause of “local network works, internet does not.”

```bash
ip route show
ip route get 8.8.8.8
```

- **Default route** should point to the correct gateway.
- **Specific routes** should not override the intended path.
- `ip route get` shows the exact path the kernel would use for a destination.

Example routing table:

```bash
default via 192.168.10.1 dev eth0
192.168.10.0/24 dev eth0 proto kernel scope link src 192.168.10.25
10.20.0.0/16 via 192.168.10.254 dev eth0
```

If the route is missing, add it temporarily for testing:

```bash
sudo ip route add default via 192.168.10.1 dev eth0
```

> Be careful with temporary changes. They disappear after reboot unless you persist them in your network manager or distribution-specific config.

**4. Name resolution**

DNS (Domain Name System) problems often masquerade as network outages. If `ping 1.1.1.1` works but `ping example.com` fails, the network path is likely fine and resolution is the issue.

```bash
resolvectl status
cat /etc/resolv.conf
dig example.com
getent hosts example.com
```

- `resolvectl status` shows the active resolver state on systemd-based systems.
- `/etc/resolv.conf` may be managed dynamically, so check whether it is a symlink.
- `dig` gives detailed DNS response data.
- `getent hosts` uses the system name service switch and reflects real application behavior.

Common DNS failure causes:

- Wrong nameserver entry
- Broken search domain
- Split DNS misconfiguration
- Firewall blocking UDP/TCP port 53
- Caching resolver serving stale data

**5. Transport and application reachability**

A host can be reachable at the IP layer while the target service is still unavailable. Use protocol-aware tests instead of relying only on ICMP.

```bash
nc -vz 203.0.113.20 22
curl -I https://app.example.com
openssl s_client -connect app.example.com:443 -servername app.example.com
ss -tulpen | grep ':443'
```

- `nc -vz` checks whether a TCP port accepts connections.
- `curl -I` verifies HTTP response headers.
- `openssl s_client` helps diagnose TLS (Transport Layer Security) handshake issues.
- `ss -tulpen` confirms whether the local service is listening.

**Common failure patterns and what they usually mean**

| Symptom | Likely area | Typical next check |
|---|---|---|
| Cannot ping gateway | Interface, VLAN, IP config | `ip link`, `ip addr`, switch port, VLAN tagging |
| Can ping IP but not hostname | DNS | `resolvectl`, `dig`, `/etc/resolv.conf` |
| Can reach gateway but not internet | Routing, NAT, firewall | `ip route`, upstream gateway, firewall rules |
| Can reach host but not service port | Service, local firewall, ACL | `ss`, `nc`, `firewall-cmd`, `nft list ruleset` |
| Intermittent packet loss | Link, congestion, MTU, duplex | `ethtool`, `ping -M do`, `mtr` |
| Works locally, fails remotely | Routing, policy, reverse path filtering | `ip rule`, `ip route`, logs |

**Useful diagnostic commands**

These commands help you gather evidence quickly without making changes.

```bash
hostname -I
ip -br addr
ip -br link
ip neigh show
mtr -rw 198.51.100.10
tcpdump -i eth0 host 198.51.100.10
```

- `hostname -I` prints assigned addresses quickly.
- `ip -br` gives a compact view for fast triage.
- `ip neigh show` checks ARP (Address Resolution Protocol) and NDP (Neighbor Discovery Protocol) state.
- `mtr` combines ping and traceroute behavior for path analysis.
- `tcpdump` confirms whether packets leave or arrive on the interface.

Example packet capture:

```bash
sudo tcpdump -i eth0 -nn host 198.51.100.10 and port 443
```

Use packet capture when you need to answer one of these questions:

- Is the packet leaving the host?
- Is a reply returning?
- Is the packet being dropped before the application sees it?
- Is the issue on the local host or beyond it?

**Firewall and policy checks**

Host firewalls can block traffic even when routing and DNS are correct. On Linux, common firewall frameworks include `nftables` and `firewalld`.

```bash
sudo nft list ruleset
sudo firewall-cmd --list-all
sudo firewall-cmd --list-ports
sudo firewall-cmd --list-services
```

- **nftables** is the modern packet filtering framework on many distributions.
- **firewalld** provides a zone-based management layer over the underlying firewall.
- Check both inbound and outbound policy if the environment uses restrictive egress rules.

If SELinux is enabled, it can also block network-related service behavior even when the firewall is open.

```bash
getenforce
ausearch -m avc -ts recent
journalctl -t setroubleshoot -b
```

**Service-side validation**

Sometimes the network is fine and the service is the problem. Confirm that the daemon is running, listening on the expected address, and bound to the correct port.

```bash
systemctl status sshd
ss -tulpen | grep sshd
journalctl -u sshd -b
```

Look for:

- Service not running
- Service bound only to `127.0.0.1`
- Wrong port number
- TLS certificate errors
- Authentication failures that appear as connectivity issues

**A repeatable troubleshooting workflow**

Use this sequence when you need a reliable method under exam or production pressure.

1. Identify the symptom precisely.
   - Note source, destination, protocol, and time.
2. Reproduce the issue with a minimal test.
   - Prefer a single command that fails consistently.
3. Check local host state.
   - Verify interface, address, and route.
4. Test the next hop.
   - Ping the gateway or local peer.
5. Test name resolution separately.
   - Compare hostname and IP-based tests.
6. Test the target service.
   - Use `nc`, `curl`, or protocol-specific tools.
7. Inspect logs and packet flow.
   - Use `journalctl`, `tcpdump`, and `ss`.
8. Make one change, then retest.
   - Avoid stacking changes that hide the real cause.

**Example workflow: website unreachable**

Suppose `curl https://app.example.com` fails.

```bash
curl -I https://app.example.com
dig app.example.com
ping -c 4 203.0.113.20
ip route get 203.0.113.20
ss -tulpen | grep ':443'
```

Interpretation:

- If `dig` fails, focus on DNS.
- If `ping` to the IP fails, focus on routing or reachability.
- If `ping` works but `curl` fails, inspect TCP, TLS, proxy settings, or the remote service.
- If the local service is involved, confirm it is listening on the correct interface and port.

**Configuration examples**

A static IPv4 configuration on a system using NetworkManager might look like this:

```bash
nmcli con mod "Wired connection 1" ipv4.addresses 192.168.10.25/24
nmcli con mod "Wired connection 1" ipv4.gateway 192.168.10.1
nmcli con mod "Wired connection 1" ipv4.dns "192.168.10.53 1.1.1.1"
nmcli con mod "Wired connection 1" ipv4.method manual
nmcli con up "Wired connection 1"
```

A systemd-networkd example:

```ini
[Match]
Name=eth0

[Network]
Address=192.168.10.25/24
Gateway=192.168.10.1
DNS=192.168.10.53
DNS=1.1.1.1
```

These examples matter because troubleshooting often ends with correcting configuration, not just identifying the fault. Always verify the active configuration after changes.

**Troubleshooting habits that save time**

- **Change one variable at a time** so you can attribute the result.
- **Test from both ends** when possible, especially for server issues.
- **Use the simplest command first** before moving to packet capture.
- **Record the exact command and output** so you can compare before and after.
- **Prefer evidence over assumptions**; a reachable gateway does not prove the application path is healthy.

**Quick reference checklist**

- Interface up and carrier present
- Correct IP address and prefix
- Correct default route
- DNS resolution working
- Target host reachable by IP
- Target port open and service listening
- Firewall and SELinux not blocking traffic
- Logs and packet capture confirm the path

**When to escalate**

Escalate beyond the local host when your checks show the problem is outside your control.

- The host has a correct address and route, but packets never return.
- Packet capture shows outbound traffic leaving but no reply.
- The issue depends on a switch, VLAN, VPN, or upstream firewall.
- The remote service is down or refusing connections.
- You need access to infrastructure logs, switch configuration, or cloud security groups.
