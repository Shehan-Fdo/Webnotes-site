## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Performance interpretation mindset**

Basic performance interpretation is about turning symptoms into likely bottlenecks. You do not need perfect certainty from one metric; you need to correlate CPU, memory, disk, and network signals with process behavior and system context.

A useful workflow is:

1. Identify the symptom: slowness, timeouts, high latency, or unresponsiveness.
2. Check whether the issue is system-wide or limited to one process, service, or host.
3. Compare current values to baseline, not just to a generic â€śgoodâ€ť number.
4. Correlate metrics across layers before you act.

A system can show high load average without being CPU-bound, high CPU without being â€śslow,â€ť or low free memory without being in trouble. The exam expects you to recognize these distinctions.

**Core tools for quick triage**

Use a small set of commands first, then drill down.

```bash
uptime
top
htop
vmstat 1
iostat -xz 1
free -h
ss -tuna
sar -u 1 3
sar -r 1 3
sar -n DEV 1 3
dmesg -T | tail -50
journalctl -p warning -b
```

- `uptime` shows load average and how long the system has been running.
- `top` and `htop` show CPU, memory, and process-level consumption.
- `vmstat` helps you see runnable processes, swap activity, and I/O wait trends.
- `iostat` helps you separate CPU pressure from storage latency.
- `ss` shows socket states and can reveal connection buildup.
- `sar` gives historical samples when available through `sysstat`.
- `dmesg` and `journalctl` can reveal kernel-level warnings, OOM events, or device errors.

**CPU interpretation**

CPU issues are not just â€śhigh usage.â€ť You want to know whether the CPU is busy doing useful work, waiting on I/O, or spending time in kernel overhead.

Key indicators include:

- **User CPU (`us`)**: time spent running application code.
- **System CPU (`sy`)**: time spent in kernel space.
- **Nice CPU (`ni`)**: time spent on niced processes.
- **Idle (`id`)**: unused CPU time.
- **I/O wait (`wa`)**: time the CPU is idle while at least one task waits on I/O completion.
- **Steal (`st`)**: time taken by the hypervisor on virtual machines.

Example `top` output patterns:

```text
%Cpu(s): 72.0 us, 18.0 sy,  0.0 ni,  2.0 id,  8.0 wa,  0.0 hi,  0.0 si,  0.0 st
```

- High `us` with low `wa` usually points to CPU-bound application work.
- High `sy` can indicate kernel-heavy activity, many syscalls, networking overhead, or storage stack pressure.
- Elevated `wa` suggests tasks are waiting on I/O, but it does not prove the storage device is the only cause; blocked network filesystems, slow remote storage, or other I/O paths can contribute.
- Nonzero `st` on a VM suggests the guest wants CPU time but the host is not giving it.

Load average is related but not identical to CPU usage. It counts runnable tasks and tasks in uninterruptible sleep, often due to I/O. Compare load average to the number of logical CPUs and the workload pattern.

Rules of thumb:

- Load average near the number of logical CPUs can be normal on a busy system.
- Load average consistently above CPU count suggests contention, but you still need to check whether the pressure is CPU, memory, or I/O related.
- A short spike above CPU count is not automatically a problem.
- On systems with hyperthreading, logical CPU count is higher than physical core count, so interpret carefully.

Example:

```bash
nproc
uptime
```

If `nproc` returns `8`, a load average of `8.0` may be acceptable under sustained work. A load of `20.0` with high `wa` and slow disk response is more suspicious than the same load with high `us` during a compile job.

Useful commands for CPU analysis:

```bash
mpstat -P ALL 1
pidstat -u 1
ps -eo pid,ppid,comm,%cpu,%mem --sort=-%cpu | head
```

- `mpstat -P ALL` shows per-CPU imbalance.
- `pidstat -u` identifies which process is consuming CPU over time.
- `ps` helps you find top consumers quickly.

**Memory interpretation**

Memory troubleshooting starts with understanding the difference between free memory, cached memory, and actual pressure. Linux uses RAM aggressively for page cache, so low `free` memory alone is not a problem.

A typical `free -h` output includes:

```text
               total        used        free      shared  buff/cache   available
Mem:            16Gi        10Gi       1.2Gi       512Mi       4.8Gi       5.1Gi
Swap:            4Gi       256Mi       3.7Gi
```

Interpretation:

- **Available** is the most useful quick indicator of how much memory can be allocated without heavy reclaim.
- **Buff/cache** is reclaimable cache and should not be treated as wasted memory.
- **Used** includes memory in use by applications and kernel structures.
- **Swap used** is not automatically bad, especially if it is old or inactive, but growing swap activity can indicate pressure.

Signs of memory pressure:

- `available` is low and falling.
- Swap usage grows along with `si` and `so` in `vmstat`.
- The system becomes sluggish even when CPU is not saturated.
- The kernel logs out-of-memory activity.

Example `vmstat` pattern:

```text
procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
 2  1  204800  120000  80000  900000  512 1024  2048  4096  500  900 10  8 40 42  0
```

- `r` is runnable processes.
- `b` is blocked processes.
- `si` and `so` show swap-in and swap-out activity.
- Sustained nonzero `si`/`so` with low `available` suggests memory pressure.
- A high `b` value can indicate tasks blocked on I/O, which may coexist with memory pressure.

Major page faults matter because they can cause visible latency. A page fault is normal when a process accesses memory not currently mapped in RAM, but a **major page fault** requires disk access to fetch data. Frequent major faults can make an application feel slow even if CPU usage looks moderate.

Useful commands:

```bash
vmstat 1
sar -B 1 3
pidstat -r 1
ps -eo pid,comm,min_flt,maj_flt --sort=-maj_flt | head
```

- `sar -B` reports paging activity.
- `pidstat -r` shows memory and page fault behavior by process.
- `maj_flt` helps identify processes that are repeatedly faulting in data from disk.

Out-of-memory behavior is a critical exam topic. When Linux cannot satisfy memory demands, the kernel may invoke the OOM killer to terminate processes and recover memory.

Common indicators:

- `dmesg` shows messages about `Out of memory` or `Killed process`.
- `journalctl -k` contains kernel OOM logs.
- A service exits unexpectedly during memory pressure.
- The system may survive by killing one or more processes, but the workload becomes unstable.

Example:

```bash
dmesg -T | grep -i -E 'oom|killed process|out of memory'
journalctl -k | grep -i oom
```

> If a service dies and the logs show the OOM killer, the root cause is memory exhaustion, not an application crash in isolation.

**Disk and storage interpretation**

Storage problems often show up as slow applications, high load average, and elevated I/O wait. The key is to distinguish device saturation, filesystem space issues, and latency from the storage stack.

Start with `iostat`:

```bash
iostat -xz 1
```

Important fields:

- **`%util`**: how busy the device is. Near 100% can indicate saturation, but it must be interpreted with latency and queue depth.
- **`await`**: average time for I/O requests to complete, including queue time.
- **`r/s` and `w/s`**: read and write operations per second.
- **`rkB/s` and `wkB/s`**: throughput.
- **`avgqu-sz`**: average queue depth.

Interpretation rules:

- High `await` with high `avgqu-sz` often suggests the device is backing up.
- High `%util` with rising latency suggests the device may be saturated.
- High throughput with low latency can be normal on fast storage.
- A small queue depth with high latency can indicate a slow device, remote storage delay, or filesystem-level contention.

Example pattern:

```text
Device            r/s     w/s   rkB/s   wkB/s  avgqu-sz  await  %util
sda              12.0    80.0   480.0  4096.0      18.5  120.0   99.8
```

This suggests the disk is busy and requests are waiting a long time. That is more consistent with storage latency or saturation than with CPU pressure.

Filesystem space issues are different from device performance issues. A filesystem can be slow because it is nearly full, fragmented, or under metadata pressure, even if the underlying disk is not maxed out.

Check:

```bash
df -h
df -i
```

- `df -h` shows space usage.
- `df -i` shows inode usage.
- A filesystem can have free blocks but no free inodes, which prevents new files from being created.
- Near-full filesystems can slow down some workloads, especially those that create many files or update metadata heavily.

Queue depth and latency are especially important for exam scenarios. A storage device can have a modest throughput number but still be a bottleneck if requests wait too long. Latency matters more than raw bandwidth for interactive workloads and databases.

Practical storage clues:

- `wa` is high and `iostat` shows high `await`.
- `dmesg` shows I/O errors, resets, or filesystem warnings.
- `df -h` shows a nearly full filesystem.
- `df -i` shows inode exhaustion.
- `b` in `vmstat` stays elevated, indicating blocked tasks.

**Network interpretation**

Network issues can look like application slowness, timeouts, or intermittent failures. Do not assume â€śslow networkâ€ť means low bandwidth only. Latency, packet loss, retransmissions, interface errors, and congestion all matter.

Start with interface statistics:

```bash
ip -s link show eth0
sar -n DEV 1 3
ss -s
ss -tuna
```

What to look for:

- **RX/TX errors**: physical or driver-level problems.
- **Dropped packets**: congestion, buffer exhaustion, or interface issues.
- **Overruns**: receive path could not keep up.
- **Retransmissions**: TCP is resending packets, often due to loss or congestion.
- **Socket buildup**: many connections in `SYN-RECV`, `ESTAB`, `TIME-WAIT`, or `CLOSE-WAIT` can indicate application or network stress.

Example `ip -s link` pattern:

```text
2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500
    RX:  bytes  packets  errors  dropped overrun mcast
    123456789  987654   12      340     0      0
    TX:  bytes  packets  errors  dropped carrier collsns
    234567890  876543   0       5       0       0
```

- Nonzero errors or drops deserve investigation.
- A few drops may be normal on a busy system, but growing counters usually indicate a problem.
- If throughput is low but latency is high, the issue may be congestion or retransmission rather than bandwidth exhaustion.

Use `ss` to inspect socket states:

```bash
ss -tuna state established
ss -tuna state syn-recv
ss -tuna state time-wait
```

Interpretation:

- Many `SYN-RECV` entries can suggest connection backlog pressure or a flood of incomplete handshakes.
- Many `CLOSE-WAIT` sockets often indicate an application is not closing connections properly.
- Many `TIME-WAIT` sockets can be normal on busy client systems, but excessive buildup may indicate connection churn.

Bandwidth saturation versus latency:

- **Bandwidth saturation** means the link is carrying near its capacity.
- **Latency/congestion** means packets are delayed, queued, or retransmitted even if raw throughput is not maxed out.
- A link can have low bandwidth usage and still perform poorly if packet loss is high.

**Correlating symptoms with likely causes**

Use multiple signals together instead of relying on one metric.

| Symptom | Likely direction | What to check next |
|---|---|---|
| High load average, high `wa`, low CPU `us` | I/O wait or blocked tasks | `iostat -xz 1`, `dmesg`, `df -h` |
| High CPU `us`, low `wa` | CPU-bound workload | `top`, `pidstat -u`, application profiling |
| Low `available`, rising `si/so` | Memory pressure | `free -h`, `vmstat 1`, OOM logs |
| Slow app, high retransmits, interface drops | Network congestion or errors | `ip -s link`, `ss -s`, `sar -n DEV` |
| High load, many blocked tasks, disk `await` high | Storage latency | `iostat -xz 1`, filesystem checks, device logs |
| VM sluggish, nonzero `st` | Host contention | Hypervisor metrics, host CPU pressure |

**Exam-style scenarios**

1. A server has load average `18.0`, CPU `us` is low, `wa` is high, and `iostat` shows `await` above 100 ms.
   - Likely bottleneck: storage or another blocked I/O path.
   - Best next step: inspect disk latency, filesystem health, and kernel logs.

2. A host has `free -h` showing low `free` memory but high `buff/cache` and healthy `available`.
   - Likely status: normal Linux cache usage.
   - Best next step: confirm with `vmstat` and process memory usage before changing anything.

3. A VM shows high load average, moderate CPU usage, and nonzero `st`.
   - Likely bottleneck: CPU contention on the host.
   - Best next step: check hypervisor allocation and host load.

4. A web server has many `TIME-WAIT` sockets and low throughput, but interface errors are zero.
   - Likely issue: connection churn or application behavior, not physical link failure.
   - Best next step: inspect application keepalive settings and request patterns.

5. A database host has low disk throughput but high `await` and a growing queue depth.
   - Likely issue: latency-bound storage, not bandwidth saturation.
   - Best next step: check device health, RAID/controller logs, and storage backend performance.

**Practical troubleshooting sequence**

1. Confirm the symptom with `uptime`, `top`, and `free -h`.
2. Check whether CPU, memory, disk, or network is the dominant pressure.
3. Use `vmstat 1` for a quick cross-subsystem view.
4. Drill into the suspected subsystem with `iostat`, `ss`, `ip -s link`, or process-level tools.
5. Review logs for kernel warnings, OOM events, device resets, or filesystem errors.
6. Compare against baseline and recent changes before making a fix.

**Configuration and monitoring examples**

A simple recurring snapshot can help you capture transient issues:

```bash
#!/bin/bash
date
uptime
free -h
vmstat 1 2
iostat -xz 1 2
ss -s
```

For historical monitoring, `sysstat` can collect samples that you review later:

```bash
sudo systemctl enable --now sysstat
sar -u
sar -r
sar -n DEV
sar -d
```

This is useful when the problem is intermittent and not visible during live troubleshooting.

**What â€śnormalâ€ť often looks like**

Normal depends on workload, but these patterns are common:

- `free -h` shows modest `free` memory and substantial `buff/cache`, with healthy `available`.
- `top` shows some CPU use, but not persistent saturation across all cores.
- `vmstat` shows `r` near or below CPU count most of the time, with low `si/so`.
- `iostat` shows low to moderate `await` and a manageable queue depth.
- `ip -s link` shows no growing error or drop counters.
- `ss` shows expected socket states for the application type.

**What to remember for the exam**

- **Load average** is not the same as CPU usage; it includes runnable and uninterruptible tasks.
- **Available memory** is more meaningful than free memory alone.
- **I/O wait** suggests blocked I/O, but you still need correlation to identify the exact bottleneck.
- **Major page faults** and **swap activity** are strong memory-pressure clues.
- **`await`**, **queue depth**, and **`%util`** help you judge storage latency and saturation.
- **Drops, errors, and retransmissions** are key network health indicators.
- Always compare metrics to the workload, the CPU count, and the recent baseline.
