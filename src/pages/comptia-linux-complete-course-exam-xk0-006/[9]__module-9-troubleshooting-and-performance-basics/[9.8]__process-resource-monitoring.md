**Process fundamentals**

A Linux process is a running instance of a program. Monitoring processes helps you identify performance bottlenecks, runaway jobs, hung services, and resource leaks. In troubleshooting, you usually start by answering four questions: what is running, how much resource it uses, who started it, and what it is blocking.

Key identifiers and concepts:

- **PID**: Process ID, the unique numeric identifier for a process.
- **PPID**: Parent process ID, used to trace process lineage.
- **UID**: User ID, useful for ownership and permission checks.
- **TTY**: Controlling terminal, if any.
- **State**: Current process state such as running, sleeping, stopped, or zombie.

Common process states you may see in tools like `ps` and `top`:

- **R**: Running or runnable.
- **S**: Interruptible sleep, often waiting for input or I/O.
- **D**: Uninterruptible sleep, usually waiting on disk or kernel I/O.
- **T**: Stopped or traced.
- **Z**: Zombie, terminated but not yet reaped by the parent.

A zombie process does not consume CPU, but it still occupies a process table entry until the parent collects its exit status. A large number of zombies usually indicates a parent process problem, not a CPU problem.

**Viewing processes with `ps`**

`ps` gives a snapshot of current processes. It is ideal when you want a precise, scriptable view rather than a live display.

Common options:

- `ps -ef`: Full-format listing of all processes in a traditional UNIX style.
- `ps aux`: BSD-style listing with CPU and memory columns.
- `ps -u <user>`: Processes owned by a specific user.
- `ps -p <pid>`: Show a specific process.
- `ps --forest`: Display parent/child relationships as a tree.

Examples:

```bash
ps -ef | less
ps aux --sort=-%cpu | head
ps -u alice
ps -p 1234 -o pid,ppid,user,%cpu,%mem,stat,cmd
ps --forest -ef
```

Useful output columns:

- **PID**: Process ID.
- **PPID**: Parent process ID.
- **%CPU**: CPU usage since the last refresh or snapshot.
- **%MEM**: Memory usage as a percentage of RAM.
- **VSZ**: Virtual memory size.
- **RSS**: Resident set size, the non-swapped physical memory in use.
- **STAT**: Process state plus flags.
- **CMD**: Command name or full command line.

`ps` is especially useful when you need to filter and format output for automation. The `-o` option lets you choose exactly which fields to display.

```bash
ps -eo pid,ppid,user,stat,etime,%cpu,%mem,cmd --sort=-%mem
```

**Finding processes by name with `pgrep` and `pidof`**

When you know the process name but not the PID, use `pgrep` or `pidof`.

- **`pgrep`** searches running processes by name or pattern and returns matching PIDs.
- **`pidof`** returns the PID(s) of a named program, often useful for simple scripts.

Examples:

```bash
pgrep nginx
pgrep -u alice -a bash
pgrep -f 'python .*app.py'
pidof sshd
pidof -s nginx
```

Important differences:

- `pgrep` is more flexible and can match the full command line with `-f`.
- `pidof` is simpler and often returns only the PID list for the executable name.
- `pgrep -a` shows the PID and full command, which is helpful for verification.

**Sending signals with `kill`, `pkill`, and `killall`**

Signals tell a process to stop, reload, or terminate. The default signal for `kill` is `SIGTERM`, which asks the process to exit cleanly.

Common signals:

- **SIGTERM** (`15`): Graceful termination request.
- **SIGKILL** (`9`): Immediate termination by the kernel; cannot be caught or ignored.
- **SIGHUP** (`1`): Historically hangup; often used to reload configuration.
- **SIGSTOP**: Pause a process immediately.
- **SIGCONT**: Resume a stopped process.

Examples:

```bash
kill 1234
kill -TERM 1234
kill -9 1234
pkill nginx
pkill -u alice -TERM python
killall httpd
```

Use `SIGTERM` first, then escalate to `SIGKILL` only if the process does not exit. `SIGKILL` can leave temporary files, partial writes, or inconsistent application state.

> `kill -9` is a last resort. If a process is stuck in uninterruptible sleep (`D` state), even `SIGKILL` may not terminate it until the kernel I/O wait clears.

**Process trees and parent-child troubleshooting**

Many services spawn helper processes, workers, or shells. A process tree helps you identify the parent responsible for launching a child and whether a problem is isolated or inherited.

Useful tools:

- `ps --forest`: Shows hierarchy in a `ps` snapshot.
- `pstree`: Displays a live tree view of processes.
- `pstree -p`: Includes PIDs.
- `pstree -a`: Shows command-line arguments.

Examples:

```bash
pstree
pstree -p 1
pstree -ap | less
ps -ef --forest | less
```

Process trees are useful when:

- A daemon forks worker processes.
- A shell script launches background jobs.
- You need to identify the parent of a zombie process.
- A service manager such as `systemd` restarts or supervises child processes.

**Interactive monitoring with `top` and `htop`**

`top` is the standard live process monitor. It updates continuously and is useful for spotting spikes in CPU, memory pressure, and load.

Typical `top` header fields:

- **uptime**: How long the system has been running.
- **load average**: Average number of runnable or uninterruptible tasks over 1, 5, and 15 minutes.
- **Tasks**: Total processes and their states.
- **%Cpu(s)**: CPU time breakdown.
- **MiB Mem / MiB Swap**: Memory and swap usage.

Interpreting load average:

- On a single-core system, a load of `1.00` means the CPU is fully utilized.
- On a multi-core system, compare load to the number of CPU cores.
- A load much higher than the core count often means CPU saturation, I/O wait, or too many runnable tasks.

Interpreting task counts:

- **running**: Actively using CPU or ready to run.
- **sleeping**: Waiting for an event.
- **stopped**: Suspended by a signal or job control.
- **zombie**: Exited but unreaped.

Common `top` columns:

- **PID**: Process ID.
- **USER**: Owner.
- **PR**: Scheduling priority.
- **NI**: Nice value.
- **VIRT**: Total virtual memory.
- **RES**: Resident memory.
- **SHR**: Shared memory.
- **S**: State.
- **%CPU**: CPU usage.
- **%MEM**: Memory usage.
- **TIME+**: Total CPU time used.
- **COMMAND**: Command name.

Practical `top` usage:

```bash
top
top -u alice
top -p 1234
```

Inside `top`, common interactive keys include:

- `P`: Sort by CPU usage.
- `M`: Sort by memory usage.
- `T`: Sort by cumulative CPU time.
- `k`: Kill a process.
- `r`: Renice a process.
- `1`: Show per-CPU usage.
- `H`: Show threads.

`htop` is a more user-friendly alternative if installed. It provides color, easier navigation, and tree views, but `top` remains the baseline tool you should know for the exam.

**Memory monitoring and interpretation**

Memory troubleshooting requires more than checking “used” versus “free.” Linux uses spare RAM for cache, so high memory usage is not automatically a problem.

Useful commands:

```bash
free -h
vmstat 1
cat /proc/meminfo | less
```

Key memory concepts:

- **Buffers/cache**: Memory used to speed up disk access; can be reclaimed when needed.
- **Swap**: Disk-backed overflow for memory pages.
- **OOM killer**: Out-of-memory mechanism that terminates processes when memory is exhausted.

`free -h` output interpretation:

- **available** is often the most useful field for estimating how much memory can still be allocated without heavy swapping.
- A system with low `free` memory but high `available` memory is usually healthy.
- High swap usage with sustained paging can indicate memory pressure.

`vmstat` helps correlate memory, CPU, and I/O behavior:

```bash
vmstat 1 5
```

Important `vmstat` fields:

- **r**: Runnable processes.
- **b**: Blocked processes.
- **si/so**: Swap in/out.
- **bi/bo**: Block input/output.
- **us/sy/id/wa/st**: CPU user, system, idle, wait, and steal time.

**CPU and scheduler analysis**

CPU issues are not always caused by one process using 100% CPU. They can also come from too many runnable tasks, kernel overhead, or I/O wait.

Useful tools:

```bash
mpstat -P ALL 1
uptime
sar -u 1 3
```

What to look for:

- High **user** CPU suggests application workload.
- High **system** CPU suggests kernel activity, often networking, storage, or many context switches.
- High **iowait** suggests the CPU is idle while waiting for storage.
- High **steal** time in virtual machines suggests the hypervisor is contending for CPU.

Nice values and priority:

- **nice** adjusts user-space scheduling priority.
- Lower nice values mean higher priority.
- Regular users can usually increase nice values to reduce priority.
- Root can decrease nice values to increase priority.

Examples:

```bash
nice -n 10 long_job.sh
renice 5 -p 1234
renice -n -5 -p 1234
```

**I/O monitoring and open files**

Disk bottlenecks often appear as slow applications, high load average, or processes stuck in `D` state. You need to distinguish CPU pressure from storage pressure.

Useful commands:

```bash
iostat -xz 1
iotop
lsof
lsof -p 1234
lsof /var/log/messages
lsof +L1
```

Interpretation tips:

- **iostat** shows device utilization, throughput, and latency.
- **iotop** identifies processes generating disk I/O.
- **lsof** lists open files, sockets, and device handles.

`lsof` is especially useful when:

- A file was deleted but disk space is not freed because a process still has it open.
- You need to identify which process is holding a log file or socket.
- You are troubleshooting a port conflict.
- You want to see what files a service is actively using.

Examples:

```bash
lsof -i :80
lsof -iTCP -sTCP:LISTEN
lsof +L1
```

`lsof +L1` finds open files that have been unlinked from the filesystem but are still held open by a process.

**Job control and background tasks**

Shell job control helps you manage processes started from an interactive shell. This is especially important for long-running commands that should survive terminal closure.

Common job control commands:

- `jobs`: List jobs in the current shell.
- `bg`: Resume a stopped job in the background.
- `fg`: Bring a job to the foreground.
- `Ctrl+Z`: Suspend the foreground job.
- `nohup`: Run a command immune to hangup signals.
- `disown`: Remove a job from the shell’s job table.

Example workflow:

```bash
long_task.sh &
jobs
fg %1
Ctrl+Z
bg %1
disown %1
```

When to use `disown`:

- You started a background job in an interactive shell and want it to continue after logout.
- You do not want the shell to send it a hangup-related notification or manage it as a job.

`nohup` and `disown` are related but not identical:

- `nohup` prevents the process from receiving `SIGHUP` by default.
- `disown` removes the job from the shell’s control, which is useful after the job has already started.

Example:

```bash
nohup backup.sh > backup.log 2>&1 &
disown
```

**Persistent monitoring and historical logging with `sysstat`**

Real troubleshooting often requires historical data, not just a live snapshot. The `sysstat` package provides tools such as `sar`, `iostat`, `mpstat`, and `pidstat` for collecting and reviewing performance trends.

Common uses:

- Identify when CPU, memory, or I/O spikes occurred.
- Compare current performance to historical baselines.
- Correlate service slowdowns with system load.

Typical `sar` examples:

```bash
sar -u 1 5
sar -r 1 5
sar -b 1 5
sar -q 1 5
sar -n DEV 1 5
```

Useful `sar` views:

- **`-u`**: CPU utilization.
- **`-r`**: Memory usage.
- **`-b`**: I/O and transfer rates.
- **`-q`**: Run queue and load average.
- **`-n DEV`**: Network interface statistics.

Historical log files are commonly stored under `/var/log/sa/` or a similar distribution-specific path. You can review past activity with date-based reports:

```bash
sar -u -f /var/log/sa/sa26
sar -r -f /var/log/sa/sa26
sar -b -f /var/log/sa/sa26
```

Persistent collection is often controlled by a timer or cron job. On many systems, `sysstat` enables periodic sampling so you can inspect performance after an incident rather than only during it.

**Processes, services, and cgroups**

On modern Linux systems, many processes run under `systemd` service units and cgroups (control groups). This matters because resource limits may be enforced at the service or slice level, not just by the process itself.

Why this matters:

- A process may appear healthy individually but still be constrained by a service limit.
- CPU, memory, and I/O usage can be capped by `systemd`.
- A service restart may change the process tree and PID values.

Useful commands:

```bash
systemctl status nginx
systemctl show nginx -p MainPID -p MemoryCurrent -p CPUUsageNSec
systemd-cgls
systemd-cgtop
```

Examples of resource controls:

```ini
# /etc/systemd/system/myapp.service
[Service]
ExecStart=/usr/local/bin/myapp
MemoryMax=512M
CPUQuota=50%
TasksMax=200
```

Interpretation:

- **MemoryMax** limits the maximum memory a service can use.
- **CPUQuota** limits CPU time as a percentage of one CPU core per configured quota model.
- **TasksMax** limits the number of processes or threads.

If a service is slow or being killed, check both the process metrics and the unit’s configured limits. A process monitor alone may not reveal a cgroup restriction.

**Practical troubleshooting workflow**

1. Identify the symptom: high CPU, memory pressure, slow I/O, or a hung service.
2. Confirm the top consumers with `top`, `ps`, `pidstat`, or `iotop`.
3. Check process state and parentage with `ps`, `pstree`, or `pgrep`.
4. Inspect open files and sockets with `lsof`.
5. Review service limits and cgroup constraints with `systemctl` and `systemd-cgtop`.
6. Use `kill`, `pkill`, or `renice` only after you understand the cause.
7. If the issue is recurring, collect historical data with `sar` and `sysstat`.

Example investigation sequence:

```bash
top
ps -eo pid,ppid,user,stat,%cpu,%mem,cmd --sort=-%cpu | head -20
pgrep -a nginx
pstree -ap 1234
lsof -p 1234
systemctl status nginx
sar -u -f /var/log/sa/sa26
```

**Exam-focused distinctions**

- `ps` is a snapshot; `top` is live.
- `pgrep` and `pidof` find processes by name; `ps` can filter by many attributes.
- `kill` sends a signal; it does not “force close” in the GUI sense.
- `SIGTERM` is polite; `SIGKILL` is abrupt.
- `load average` is not the same as CPU percentage.
- High memory usage is not always a problem because cache is reclaimable.
- `lsof` shows open files, sockets, and deleted-but-open files.
- `disown` helps background jobs survive shell logout.
- `sar` provides historical performance data, which is essential for intermittent issues.
- `systemd` and cgroups can impose resource limits that affect process behavior.
