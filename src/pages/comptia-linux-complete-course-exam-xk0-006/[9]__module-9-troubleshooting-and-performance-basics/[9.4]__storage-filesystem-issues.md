**Storage troubleshooting mindset**

Storage problems often present as symptoms rather than root causes: a service fails to start, a mount disappears, a system boots into emergency mode, or writes suddenly fail. You need to separate the layer that is failing:

- **Hardware layer**: disk, cable, controller, power, RAID member, or enclosure.
- **Block layer**: partition table, LVM, dm-crypt, mdadm, device mapper.
- **Filesystem layer**: corruption, full blocks, inode exhaustion, quotas, permissions, ACLs.
- **Mount layer**: bad `/etc/fstab` entry, wrong UUID, unsupported options, missing mountpoint.
- **Application layer**: app-specific path, ownership, SELinux or ACL restrictions, or a full log directory.

A good workflow is to confirm the device exists, confirm the filesystem is healthy enough to mount, confirm the mount options are correct, and then verify the application can actually write where it expects.

**Identify the problem before changing anything**

Start with read-only inspection whenever possible. Useful commands include:

```bash
lsblk -f
blkid
findmnt
df -hT
df -i
mount | column -t
journalctl -b -p err
dmesg -T | tail -n 50
```

- `lsblk -f` shows disks, partitions, filesystems, labels, and UUIDs.
- `blkid` is useful when you need the exact UUID or filesystem type for `/etc/fstab`.
- `findmnt` shows the active mount tree and is better than guessing from `mount` output.
- `df -hT` shows space usage and filesystem type.
- `df -i` checks inode exhaustion, which can look like “disk full” even when blocks remain.
- `journalctl -b -p err` and `dmesg` often reveal the real storage error, such as I/O failures, ext4 journal issues, or XFS metadata problems.

**Disk health and hardware-layer diagnostics**

Not every storage issue is a filesystem problem. If the kernel logs show I/O errors, resets, timeouts, or read failures, suspect the hardware path first.

Common indicators include:

- Repeated `I/O error`, `Buffer I/O error`, or `blk_update_request` messages.
- SATA link resets, controller timeouts, or NVMe reset events.
- SMART warnings, reallocated sectors, pending sectors, or uncorrectable errors.
- RAID member dropouts or degraded arrays.

Typical tools:

```bash
smartctl -a /dev/sda
smartctl -t short /dev/sda
smartctl -t long /dev/sda
nvme smart-log /dev/nvme0
```

- **SMART** (Self-Monitoring, Analysis and Reporting Technology) can show whether a disk is failing.
- A rising count of reallocated or pending sectors suggests media degradation.
- A disk that passes SMART is not guaranteed healthy, but SMART failures are a strong warning.
- For SSDs, also watch wear indicators and media errors.

> If the kernel reports read/write failures, do not run aggressive repair tools first. Copy data off the device or image it if possible, then investigate the hardware path.

**Partitioning and device layout**

Before filesystem repair, confirm the block device layout. A storage issue may be caused by a missing partition, wrong partition type, or a device name change after reboot.

Useful commands:

```bash
lsblk
fdisk -l
parted -l
parted /dev/sda print
```

- `fdisk` is useful for interactive partition work, especially on MBR and basic GPT layouts.
- `parted` is better for scripted work, large disks, and GPT-aware changes.
- On GPT disks, partition numbers matter more than labels like `primary`; GPT does not use MBR-style primary/extended semantics.
- If you need GPT-specific management, `gdisk` or `sgdisk` may be more appropriate than `fdisk`.

Example GPT partition creation for a Linux volume:

```bash
parted /dev/sdb --script mklabel gpt
parted /dev/sdb --script mkpart linux-data ext4 1MiB 20GiB
parted /dev/sdb --script set 1 lvm on
```

**Filesystem types and repair tools**

Different filesystems require different tools and different expectations.

| Filesystem | Typical check/repair tool | Notes |
|---|---|---|
| ext4/ext3/ext2 | `e2fsck` | Common offline repair tool; run unmounted when possible |
| XFS | `xfs_repair` | Repair only offline; use `xfs_info` for geometry, not mount options |
| Btrfs | `btrfs check` | Primarily an offline consistency checker; repair options are limited and risky |
| FAT/VFAT | `fsck.vfat` | Often used for removable media and EFI partitions |
| NTFS | `ntfsfix` | Limited repair helper, not a full Windows chkdsk replacement |

For ext-family filesystems:

```bash
umount /dev/sdb1
e2fsck -f /dev/sdb1
```

- `e2fsck` checks and repairs ext filesystems.
- Use `-f` to force a check even if the filesystem appears clean.
- Never run `fsck` on a mounted filesystem unless you know the filesystem and tool support online checking safely; most repairs require the filesystem to be unmounted.

For XFS:

```bash
xfs_repair /dev/sdb2
xfs_info /mnt/data
```

- `xfs_repair` is the standard offline repair tool.
- `xfs_info` reports XFS geometry and superblock information for a mounted filesystem.
- XFS does not support shrinking; it supports online growth when mounted.

For Btrfs:

```bash
btrfs filesystem show
btrfs scrub start -Bd /mnt/btrfsdata
btrfs check /dev/sdb3
```

- `btrfs scrub` reads data and metadata and can repair checksum errors only when redundant copies exist, such as mirrored profiles or multiple devices.
- On a single-device, non-redundant Btrfs filesystem, scrub can detect corruption but cannot reconstruct bad data from redundancy.
- `btrfs check` is an offline consistency checker and is not a routine repair equivalent to `e2fsck`.
- Use `btrfs check --repair` only with extreme caution and only when you understand the risk of data loss.

**Mount failures and `/etc/fstab` problems**

A very common boot issue is a bad `/etc/fstab` entry. Symptoms include emergency mode, failed mounts, or a system that boots but leaves a filesystem unavailable.

Check the current mount state:

```bash
findmnt
cat /etc/fstab
blkid
```

A safe validation workflow is:

1. Verify the UUID or device path in `/etc/fstab` matches the real device.
2. Confirm the mountpoint directory exists.
3. Test the entry with `mount -a`.
4. Review `journalctl -b` if the mount fails.

Example `/etc/fstab` entry:

```fstab
UUID=3f2a1c4e-8d2b-4d7f-9f0a-1f2c3d4e5f60  /data  ext4  defaults,noatime  0  2
```

Example recovery steps after a bad boot entry:

```bash
mount -o remount,rw /
nano /etc/fstab
mount -a
systemctl daemon-reload
```

- `mount -a` only attempts entries that are not already mounted, so a clean run does not guarantee every boot scenario will succeed.
- If the system is in emergency mode, you may need to remount `/` read-write before editing files.
- If a UUID changed after disk replacement, update `/etc/fstab` with the new UUID from `blkid`.

If a mountpoint is missing, create it first:

```bash
mkdir -p /data
```

If the filesystem is mounted read-only after an error, inspect the kernel log before forcing a remount:

```bash
dmesg -T | tail -n 50
mount -o remount,rw /data
```

- Remounting read-write only works if the filesystem is already mounted.
- If the kernel forced the filesystem read-only because of corruption or I/O errors, remounting may fail until the underlying issue is fixed.

**Bind mounts and special mount behavior**

Bind mounts are useful for exposing a directory at another path, often for chroots, containers, or restricted views.

```bash
mount --bind /var/log /mnt/logview
mount -o remount,bind,ro /mnt/logview
```

- Create the bind mount first, then remount it read-only if needed.
- Some systems require the source and target to be specified explicitly during the remount step.
- Verify the result with `findmnt /mnt/logview`.

**Space exhaustion versus inode exhaustion**

A filesystem can be “full” in two different ways:

- **Block exhaustion**: no data blocks remain for file contents.
- **Inode exhaustion**: no file metadata entries remain, even if free space exists.

Check both:

```bash
df -hT /var
df -i /var
```

- If `df -h` shows free space but `df -i` shows 100% inode use, the filesystem cannot create new files.
- This often happens in directories with millions of tiny files, such as mail spools, caches, package caches, or log directories.
- Cleaning a few large files will not help inode exhaustion; you must remove many files or move to a filesystem with more inodes.

**Permissions, ownership, and ACL symptoms that look like storage failures**

Not every “cannot write” problem is a filesystem failure. A directory may be mounted correctly but still reject writes because of permissions, ownership, or access control lists.

Check:

```bash
ls -ld /data /data/app
getfacl /data/app
namei -l /data/app/config
```

- **Ownership** determines which user and group can write.
- **Mode bits** control read, write, and execute access.
- **ACLs** (Access Control Lists) can grant or deny access beyond traditional Unix permissions.
- A service account may fail to write logs even when the filesystem has plenty of free space.

Example fix:

```bash
chown -R appuser:appgroup /data/app
chmod 750 /data/app
setfacl -m u:appuser:rwx /data/app
```

Use `namei -l` when a path fails somewhere in the middle, because one parent directory with missing execute permission can block access to the entire path.

**Quotas and user-space limits**

Disk usage problems may be caused by quotas rather than actual filesystem exhaustion. Quotas limit how much space or how many inodes a user or group can consume.

Check quota status:

```bash
quota -u alice
quota -g developers
repquota -a
```

- A user can receive “disk full” errors even when the filesystem still has free space.
- Quotas may apply to blocks, inodes, or both.
- If a service runs under a dedicated account, check that account’s quota before resizing the filesystem.

Typical quota administration:

```bash
quotacheck -cum /home
quotaon /home
edquota -u alice
```

**Swap troubleshooting**

Swap issues can look like memory pressure, sluggishness, or failed hibernation. Linux uses swap partitions or swap files to extend virtual memory.

Identify swap:

```bash
swapon --show
free -h
cat /proc/swaps
```

Create and enable a swap file:

```bash
fallocate -l 4G /swapfile
chmod 600 /swapfile
mkswap /swapfile
swapon /swapfile
```

Persist it in `/etc/fstab`:

```fstab
/swapfile none swap sw 0 0
```

Troubleshoot swap problems:

- If `swapon` fails, check permissions, file type, and whether the file has holes or unsupported attributes.
- If a swap partition is missing after reboot, verify the UUID in `/etc/fstab`.
- If the system is swapping heavily, inspect memory pressure with `free -h`, `vmstat 1`, and process memory usage.
- If hibernation fails, confirm the swap area is large enough and correctly configured for resume.

Disable swap temporarily:

```bash
swapoff /swapfile
```

**Resizing filesystems**

Resizing depends on the filesystem type and whether LVM is involved. Always distinguish between resizing the block device and resizing the filesystem.

For ext4 growth on a regular partition:

```bash
resize2fs /dev/sdb1
```

- `resize2fs` can grow ext filesystems after the underlying partition or volume has been expanded.
- Shrinking ext4 requires the filesystem to be unmounted and usually follows a careful sequence: unmount, check with `e2fsck`, shrink the filesystem, then shrink the partition or volume.

Example ext4 shrink workflow:

```bash
umount /dev/sdb1
e2fsck -f /dev/sdb1
resize2fs /dev/sdb1 20G
```

For XFS growth:

```bash
xfs_growfs /mnt/data
```

- XFS can grow online while mounted.
- XFS cannot shrink, so plan capacity carefully.

If LVM is involved, resize the logical volume first, then grow the filesystem:

```bash
lvextend -L +20G /dev/vgdata/lvhome
resize2fs /dev/vgdata/lvhome
```

For XFS on LVM:

```bash
lvextend -L +20G /dev/vgdata/lvdata
xfs_growfs /mnt/data
```

**RAID and mdadm troubleshooting**

Software RAID issues can appear as missing disks, degraded arrays, or filesystems that mount read-only after a member fails.

Inspect arrays:

```bash
cat /proc/mdstat
mdadm --detail /dev/md0
lsblk
```

- A degraded array may still function, but performance and redundancy are reduced.
- If a member disk fails, replace it and resync the array before assuming the filesystem is healthy.
- Filesystems on top of degraded RAID can still suffer corruption if the array loses another member.

Common recovery actions:

```bash
mdadm --fail /dev/md0 /dev/sdb1
mdadm --remove /dev/md0 /dev/sdb1
mdadm --add /dev/md0 /dev/sdc1
```

After RAID recovery, verify the filesystem with the appropriate tool for that filesystem type.

**Boot failures caused by storage problems**

A bad root filesystem entry, missing disk, or failed RAID member can drop the system into emergency mode. In that case, your goal is to regain minimal access, correct the configuration, and reboot cleanly.

Typical recovery flow:

1. Read the boot error and identify the missing device or failed mount.
2. Remount the root filesystem read-write if needed.
3. Fix `/etc/fstab`, RAID metadata, or the mountpoint path.
4. Run the appropriate filesystem check only if the filesystem is unmounted or the tool supports the required mode.
5. Reboot and verify with `findmnt`, `df -hT`, and `journalctl -b`.

Example emergency-mode repair for a bad UUID:

```bash
blkid
nano /etc/fstab
mount -a
reboot
```

If the root filesystem itself is damaged, you may need to boot from rescue media or a live environment so you can unmount and repair it safely.

**Practical command patterns**

Use these patterns to avoid common mistakes:

```bash
# Check space and inodes
df -hT /home
df -i /home

# Find the real device behind a mount
findmnt /home
lsblk -f

# Check filesystem health offline
umount /dev/sdb1
e2fsck -f /dev/sdb1

# Repair XFS offline
xfs_repair /dev/sdb2

# Inspect Btrfs status and scrub
btrfs filesystem show
btrfs scrub start -Bd /mnt/btrfs

# Check RAID status
cat /proc/mdstat
mdadm --detail /dev/md0

# Inspect disk health
smartctl -a /dev/sda
```

**Exam-focused distinctions**

- **`df` vs `du`**: `df` reports filesystem usage; `du` reports directory or file usage.
- **`df -i`**: checks inode exhaustion, not block usage.
- **`mount -a`**: tests `/etc/fstab` entries, but it does not guarantee boot success.
- **`xfs_info`**: shows XFS geometry and superblock details, not general mount options.
- **`btrfs check`**: is not a routine repair equivalent to ext filesystem tools.
- **`scrub` vs `check`**: scrub verifies data against checksums and can repair only when redundancy exists; check inspects filesystem structures offline.
- **Permissions vs filesystem health**: a write failure may be caused by ownership, ACLs, or mode bits rather than corruption.
- **RAID vs filesystem**: a healthy filesystem on a failing RAID array is still at risk.

**Quick troubleshooting checklist**

1. Confirm the device exists with `lsblk`, `blkid`, or `findmnt`.
2. Check kernel logs for I/O errors, timeouts, or filesystem corruption.
3. Determine whether the issue is space, inodes, permissions, quotas, or hardware.
4. Verify `/etc/fstab`, UUIDs, and mountpoints.
5. Use the correct offline repair tool for the filesystem type.
6. Check RAID, swap, and disk health if the problem persists.
7. Re-test with `mount -a`, `df -hT`, `df -i`, and application-level writes.
