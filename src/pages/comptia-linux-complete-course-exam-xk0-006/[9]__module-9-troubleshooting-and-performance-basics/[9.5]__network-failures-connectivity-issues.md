## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Network failure troubleshooting mindset**

Linux connectivity problems usually fall into a few layers: physical link, interface state, addressing, routing, name resolution, firewall policy, and remote service availability. The fastest way to troubleshoot is to move from the local host outward and verify one layer at a time.

A practical workflow is:

1. Confirm the interface is up and has a valid address.
2. Verify the default route and any specific routes.
3. Test local and remote reachability with `ping`, `ip`, and `ss`.
4. Check DNS resolution separately from raw IP connectivity.
5. Inspect firewall rules and service status if the network path is fine but the application still fails.

**Common failure categories**

- **Physical or virtual link issues**: cable unplugged, switch port down, VM NIC disconnected, or carrier loss.
- **Interface misconfiguration**: wrong IP address, subnet mask, gateway, or MTU.
- **Routing problems**: missing default route, incorrect static route, asymmetric path, or policy routing mismatch.
- **DNS failures**: hostnames fail while IP addresses work.
- **Firewall or ACL blocks**: packets reach the host but are dropped by `nftables`, `iptables`, `firewalld`, or upstream controls.
- **Service-level issues**: the host is reachable, but the daemon is not listening or is bound to the wrong address.

**Interface and link verification**

Start by checking whether the kernel sees the interface and whether it has carrier and an address. The `ip` suite is the primary tool on modern Linux systems.

```bash
ip link show
ip addr show
ip route show
```

Useful checks include:

- **`ip link show`**: confirms interface state such as `UP`, `DOWN`, and `LOWER_UP`.
- **`ip addr show`**: displays assigned IPv4 and IPv6 addresses.
- **`ip route show`**: shows the routing table, including the default gateway.
- **`ethtool eth0`**: checks link speed, duplex, and whether the physical link is detected.
- **`nmcli device status`**: shows NetworkManager device state on systems that use it.

If an interface is administratively down, bring it up:

```bash
sudo ip link set dev eth0 up
```

If you need to assign a temporary address for testing, you can do so with `ip addr add`:

```bash
sudo ip addr add 192.0.2.25/24 dev eth0
sudo ip route add default via 192.0.2.1
```

These changes are usually temporary unless you apply them through the system’s persistent network configuration.

**Addressing and subnet problems**

A host can appear “connected” but still fail to reach other systems if the address or prefix is wrong. Subnet mistakes often cause local communication to work while remote traffic fails.

Watch for:

- **Wrong prefix length**: a `/24` vs `/16` mismatch changes which hosts are considered local.
- **Duplicate IP addresses**: intermittent connectivity, ARP confusion, or address conflicts.
- **Incorrect gateway**: traffic leaves the host but never reaches the intended network.
- **IPv4/IPv6 mismatch**: services may listen on one protocol family only.

Helpful commands:

```bash
ip addr show dev eth0
ip neigh show
arp -n
```

`ip neigh` and `arp -n` help you inspect neighbor discovery and ARP (Address Resolution Protocol) state. If a neighbor entry is `FAILED` or constantly changing, suspect L2/L3 reachability issues, duplicate addresses, or switch problems.

**Routing failures**

Routing determines where packets go after they leave the local subnet. If the route table is incomplete or wrong, the host may reach local peers but not remote networks.

Check the route table:

```bash
ip route show
ip -6 route show
```

Look for:

- **Default route**: required for reaching destinations outside the local subnet.
- **Specific routes**: needed for internal networks behind routers or VPNs.
- **Multiple gateways**: can create unexpected path selection.
- **Policy routing**: advanced setups may use multiple tables and rules.

Trace the chosen path with:

```bash
ip route get 8.8.8.8
traceroute 8.8.8.8
tracepath 8.8.8.8
```

`ip route get` shows the kernel’s selected route for a destination. `traceroute` and `tracepath` help identify where packets stop. `tracepath` is useful when you want to infer path MTU (Maximum Transmission Unit) issues without requiring elevated privileges on some systems.

> A missing default route is one of the most common causes of “local network works, internet does not.”

**DNS resolution issues**

DNS (Domain Name System) problems often look like network outages because applications fail when they cannot resolve hostnames. Always test IP connectivity separately from name resolution.

Compare these tests:

```bash
ping -c 3 8.8.8.8
ping -c 3 example.com
getent hosts example.com
resolvectl status
cat /etc/resolv.conf
```

Interpretation:

- If `ping` to an IP works but `ping` to a hostname fails, suspect DNS.
- If `getent hosts` fails, the system resolver cannot resolve the name.
- If `/etc/resolv.conf` points to the wrong server, resolution may fail or be slow.
- If `resolvectl status` shows no active DNS servers, the resolver configuration may be incomplete.

Common DNS causes include:

- Wrong nameserver entries.
- Broken search domains.
- Local caching resolver failures.
- Firewall blocks to port `53` UDP/TCP.
- Split DNS misconfiguration on VPN or multi-homed hosts.

Example resolver configuration:

```conf
nameserver 192.0.2.53
nameserver 192.0.2.54
search corp.example
```

**Firewall and packet filtering**

If the host has correct addressing and routing but traffic still fails, inspect local packet filtering. Modern Linux systems may use `nftables`, `iptables`, or a higher-level manager such as `firewalld`.

Check active rules:

```bash
sudo nft list ruleset
sudo iptables -S
sudo firewall-cmd --list-all
```

Things to verify:

- **Inbound policy**: the service port may be blocked.
- **Outbound policy**: some environments restrict client traffic.
- **Stateful rules**: return traffic may be dropped if connection tracking is misconfigured.
- **Zone assignment**: `firewalld` may apply the wrong zone to the interface.

A simple `nftables` example that allows SSH and established traffic:

```nft
table inet filter {
  chain input {
    type filter hook input priority 0;
    policy drop;

    ct state established,related accept
    iif "lo" accept
    tcp dport 22 accept
  }
}
```

If you change firewall rules, test carefully to avoid locking yourself out of remote access.

**Service availability and socket binding**

Sometimes the network is healthy, but the application is not listening where you expect. Confirm the service is running and bound to the correct address and port.

```bash
systemctl status sshd
ss -tulpn
lsof -i :22
```

Use these checks to identify:

- **Service down**: the daemon is stopped or failed.
- **Wrong bind address**: the service listens only on `127.0.0.1` instead of the external interface.
- **Wrong port**: the client connects to a port the service does not use.
- **Port conflict**: another process already occupies the socket.

Example `ss` output focus:

```bash
ss -tulpn | grep ':80\|:443\|:22'
```

If a service should listen on all interfaces, its configuration may need a bind address such as `0.0.0.0` for IPv4 or `::` for IPv6, depending on the daemon.

**MTU and fragmentation problems**

Path MTU issues can cause traffic to fail in ways that look random: small packets work, large packets fail, VPNs break, or some websites load partially. MTU mismatches often appear after tunnels, VLANs, or cloud networking changes.

Useful checks:

```bash
ip link show dev eth0
ping -M do -s 1472 8.8.8.8
tracepath 8.8.8.8
```

Notes:

- `ping -M do` forces “do not fragment” behavior on IPv4.
- If a large ping fails but smaller packets succeed, suspect MTU or fragmentation filtering.
- VPNs and overlays often require a lower MTU than the underlying physical network.

**Wireless-specific issues**

On laptops and wireless hosts, connectivity problems may come from association, authentication, or signal quality rather than IP configuration.

Check status with:

```bash
nmcli device wifi list
nmcli device show wlan0
iw dev wlan0 link
```

Common wireless causes:

- Incorrect passphrase or authentication failure.
- Weak signal or interference.
- Roaming between access points with different policies.
- Captive portal blocking general internet access.

**Name service and host lookup order**

Linux may resolve names through multiple sources, not just DNS. The lookup order can include `/etc/hosts`, DNS, LDAP, or other NSS (Name Service Switch) sources.

Inspect the configuration:

```bash
cat /etc/nsswitch.conf
cat /etc/hosts
```

If a hostname resolves incorrectly, check whether `/etc/hosts` overrides DNS. This is common in lab environments and during migrations.

Example:

```conf
hosts: files dns
```

This means the system checks local files before DNS.

**Troubleshooting with packet capture**

When higher-level checks do not reveal the issue, capture traffic to see whether packets leave, return, or get dropped.

```bash
sudo tcpdump -i eth0 host 192.0.2.10
sudo tcpdump -i eth0 port 53
sudo tcpdump -i eth0 icmp
```

Use packet capture to answer questions such as:

- Are SYN packets leaving the host?
- Do replies return?
- Is DNS query traffic reaching the resolver?
- Are ICMP errors indicating unreachable networks or fragmentation problems?

A basic workflow is:

1. Start a capture on the relevant interface.
2. Reproduce the failure.
3. Inspect whether the expected packets appear.
4. Compare the observed traffic with the route, firewall, and service configuration.

**Logs and system diagnostics**

Logs often reveal whether the failure is at the link, authentication, or service layer. On systemd-based systems, `journalctl` is especially useful.

```bash
journalctl -u NetworkManager
journalctl -u systemd-networkd
journalctl -u sshd
dmesg | grep -i -E 'eth|wlan|link|firmware'
```

Look for:

- Link up/down events.
- DHCP (Dynamic Host Configuration Protocol) lease failures.
- Authentication failures on wireless or VPN connections.
- Driver or firmware errors.
- Interface renaming or device disappearance.

**DHCP failures**

If a host depends on DHCP, it may fail to connect because it never receives an address, gateway, or DNS server. Symptoms often include an APIPA-style fallback address on some systems or no address at all.

Check lease and client state:

```bash
nmcli device show eth0
journalctl -u NetworkManager | grep -i dhcp
sudo dhclient -v eth0
```

Possible causes:

- DHCP server unreachable.
- Broadcast traffic blocked.
- Wrong VLAN or switch port.
- Lease exhaustion.
- Client identifier mismatch.

**Practical isolation checklist**

Use this sequence when a host cannot reach a destination:

1. Verify the interface is present and up with `ip link show`.
2. Confirm the host has the expected IP address with `ip addr show`.
3. Check the default route with `ip route show`.
4. Test raw IP reachability with `ping`.
5. Test name resolution with `getent hosts` or `resolvectl status`.
6. Inspect firewall rules with `nft list ruleset` or `firewall-cmd --list-all`.
7. Confirm the target service is listening with `ss -tulpn`.
8. Capture packets with `tcpdump` if the failure remains unclear.

**Real-world examples**

A server can ping its gateway but not the internet:

```bash
ip route show
ping -c 3 192.0.2.1
ping -c 3 8.8.8.8
```

If the gateway responds but public IPs do not, the issue is often upstream routing, NAT, or firewall policy outside the host.

A workstation can reach websites by IP but not by name:

```bash
ping -c 3 1.1.1.1
getent hosts example.com
cat /etc/resolv.conf
```

This points to DNS rather than general connectivity.

A service is reachable locally but not remotely:

```bash
ss -tulpn | grep ':8080'
curl http://127.0.0.1:8080
curl http://192.0.2.25:8080
```

If localhost works and the LAN address fails, check the bind address and firewall rules.

**Key takeaways**

- Start with the local host and move outward through link, address, route, DNS, firewall, and service layers.
- Separate IP connectivity from hostname resolution.
- Use `ip`, `ss`, `nft`, `journalctl`, and `tcpdump` as your core diagnostic toolkit.
- Confirm whether the problem is local, upstream, or application-specific before making changes.
- Prefer small, verifiable tests over broad configuration edits.
