**Performance bottlenecks: the big picture**

A bottleneck is the subsystem that limits overall throughput. In Linux, the slowest resource often shifts between CPU, memory, storage, and I/O depending on workload. Your job is to identify the constrained resource, confirm it with metrics, and apply the least disruptive fix.

A useful mental model is:

- **CPU-bound** workloads spend most time executing instructions.
- **Memory-bound** workloads spend time waiting for RAM or swapping.
- **Disk-bound** workloads wait on storage latency or throughput.
- **I/O-bound** workloads wait on reads, writes, network filesystems, or device queues.

The key is to avoid guessing. Use process, system, and storage metrics together so you can distinguish a busy system from a truly saturated one.

**CPU bottlenecks**

CPU pressure appears when runnable tasks outnumber available compute time. High CPU usage alone does not always mean a problem; a server can be healthy at 90% CPU if latency stays acceptable. The real warning signs are sustained load, long run queues, and processes spending time waiting to be scheduled.

Important indicators include:

- **Load average** higher than the number of CPU cores for a sustained period.
- **User time** or **system time** staying high in `top` or `htop`.
- High **run queue** length in `vmstat`.
- One core pegged at 100% while others stay idle, which can indicate poor parallelism or CPU affinity issues.
- High **steal time** in virtual machines, which means the hypervisor is taking CPU time away from the guest.

Useful commands:

```bash
uptime
top
htop
mpstat -P ALL 1
vmstat 1
pidstat -u 1
```

What to look for:

- `top` shows `%us` for user-space work, `%sy` for kernel work, and `%wa` for I/O wait.
- `mpstat -P ALL 1` helps you see whether one core is saturated or all cores are busy.
- `pidstat -u 1` identifies the process consuming CPU over time.
- `vmstat 1` shows `r` for runnable tasks and `us`, `sy`, `id`, `wa`, and `st` for CPU state.

Common causes:

- Tight loops or inefficient code.
- Too many worker threads or processes.
- Compression, encryption, or compilation workloads.
- Interrupt storms from devices or network traffic.
- Poor container or VM CPU limits.

Practical response:

1. Confirm the bottleneck with `top`, `vmstat`, and `mpstat`.
2. Identify the top process or thread with `pidstat`, `top -H`, or `ps -eo pid,ppid,cmd,%cpu --sort=-%cpu`.
3. Reduce contention by tuning thread counts, scheduling, or affinity.
4. Scale vertically by adding CPU resources or horizontally by distributing work.
5. If the workload is bursty, consider cgroups or systemd resource controls.

Example process inspection:

```bash
ps -eo pid,ppid,cmd,%cpu,%mem --sort=-%cpu | head
top -H -p 1234
taskset -cp 1234
```

**Memory bottlenecks**

Memory pressure occurs when available RAM is insufficient for active workloads. Linux uses free memory aggressively for page cache, so low `free` output is not automatically a problem. You care about reclaimability, swap activity, and memory pressure, not just raw free space.

Key indicators:

- Frequent swapping or sustained swap-in/swap-out activity.
- High memory pressure and low available memory.
- Out-of-memory (OOM) kills when the kernel terminates processes to protect the system.
- Large page faults or thrashing, where the system repeatedly evicts and reloads pages.
- Slowdowns that improve when memory-hungry processes stop.

Useful commands:

```bash
free -h
vmstat 1
sar -r 1
sar -W 1
ps -eo pid,cmd,%mem,rss --sort=-%mem
smem -tk
cat /proc/meminfo
```

What to look for:

- `free -h` shows `available`, which is more useful than `free`.
- `vmstat 1` shows `si` and `so` for swap in and swap out.
- `sar -W 1` helps confirm swap activity over time.
- `ps` and `smem` help identify memory-heavy processes.
- `/proc/meminfo` provides detailed kernel memory accounting.

Important concepts:

- **Page cache** speeds file access and is reclaimable under pressure.
- **Swap** extends memory onto disk but is much slower than RAM.
- **Thrashing** happens when the system spends more time paging than doing useful work.
- **OOM killer** is a last-resort kernel mechanism that frees memory by terminating processes.

Practical response:

- Stop or restart memory-hungry services.
- Tune application memory usage or worker counts.
- Add RAM if the workload legitimately needs more.
- Reduce cache pressure by adjusting application behavior rather than disabling page cache.
- Use swap carefully; it can absorb spikes, but heavy swap use usually signals a sizing problem.

Example inspection:

```bash
watch -n 1 free -h
vmstat 1
grep -E 'MemAvailable|SwapTotal|SwapFree|Dirty|Cached' /proc/meminfo
```

> If `si` and `so` stay nonzero under load, the system is likely memory constrained or thrashing. A large `buff/cache` value by itself is not a problem.

**Disk bottlenecks**

Disk bottlenecks happen when storage cannot deliver data fast enough or with low enough latency. This can affect databases, log-heavy services, package installs, backups, and any workload that depends on frequent reads and writes.

Storage performance is usually described by:

- **Throughput**: how much data per second the device can move.
- **Latency**: how long each I/O request takes.
- **IOPS**: input/output operations per second, important for small random I/O.
- **Queue depth**: how many requests are waiting for the device.

Useful commands:

```bash
iostat -xz 1
iotop -oPa
pidstat -d 1
lsblk -o NAME,TYPE,SIZE,FSTYPE,MOUNTPOINT,ROTA,MODEL
df -h
du -sh /var/log/*
```

What to look for:

- `iostat -xz 1` shows `%util`, `await`, `r/s`, `w/s`, and average request sizes.
- High `%util` with high `await` suggests the device is saturated.
- `iotop` shows which processes are generating disk I/O.
- `pidstat -d 1` helps attribute read/write activity to specific processes.
- `lsblk` reveals whether a device is rotational (`ROTA=1`) or SSD-backed (`ROTA=0`).

Common causes:

- Slow spinning disks under random I/O.
- Small synchronous writes from databases or logging.
- Filesystem fragmentation or metadata-heavy workloads.
- Full disks causing allocator and journal overhead.
- Misconfigured RAID, thin provisioning, or virtual storage backends.

Practical response:

1. Confirm whether the issue is latency, throughput, or queueing.
2. Identify the process generating the I/O.
3. Move hot data to faster storage such as SSD or NVMe.
4. Reduce synchronous writes with batching or caching where safe.
5. Tune filesystem and mount options only after measuring the workload.

Example storage inspection:

```bash
iostat -xz 1
iotop -oPa
pidstat -d 1
```

A simple interpretation guide:

| Metric | Meaning | Typical concern |
|---|---|---|
| `%util` | Device busy percentage | Near 100% for sustained periods |
| `await` | Average I/O wait time | Rising latency under load |
| `svctm` | Service time | Less reliable on newer systems; use cautiously |
| `r/s`, `w/s` | Read/write operations per second | Helps identify workload shape |
| `rkB/s`, `wkB/s` | Read/write throughput | Useful for bandwidth-heavy workloads |

**I/O bottlenecks beyond local disk**

I/O is broader than block storage. Network filesystems, pipes, sockets, and device interrupts can also become bottlenecks. A system may show low CPU usage but still feel slow because tasks are blocked on external resources.

Common I/O-related symptoms:

- Processes in uninterruptible sleep (`D` state).
- High `%wa` in CPU summaries.
- Slow NFS, SMB, or object-storage-backed mounts.
- Network saturation affecting storage or application traffic.
- Excessive context switching or interrupt handling.

Useful commands:

```bash
ps -eo pid,stat,cmd | awk '$2 ~ /D/ {print}'
ss -tuna
nload
sar -n DEV 1
cat /proc/interrupts
```

What to look for:

- `D` state often means a process is waiting on disk or another kernel I/O path.
- `sar -n DEV 1` helps determine whether network throughput is the limiting factor.
- `/proc/interrupts` can reveal a noisy device or uneven interrupt distribution.
- Network filesystem latency may appear as application slowness even when local disk is fine.

Practical response:

- Check the storage backend if a mounted share is slow.
- Verify DNS, network latency, and packet loss for remote dependencies.
- Balance interrupts across CPUs if a single core is overloaded by device handling.
- Use asynchronous I/O or batching where the application supports it.

**How to diagnose systematically**

A repeatable workflow prevents false conclusions. Start broad, then narrow down to the subsystem and process.

1. Check overall system health with `uptime`, `top`, and `vmstat 1`.
2. Decide whether the system is CPU-, memory-, disk-, or I/O-bound.
3. Identify the top process or kernel path with `pidstat`, `iotop`, or `ps`.
4. Correlate with storage and memory metrics using `iostat`, `free`, and `/proc/meminfo`.
5. Confirm whether the issue is persistent, periodic, or workload-specific.
6. Apply the smallest effective fix and remeasure.

A practical triage command set:

```bash
uptime
vmstat 1
mpstat -P ALL 1
free -h
iostat -xz 1
pidstat -u -d 1
```

**Tuning and mitigation strategies**

The right fix depends on the bottleneck type. You usually get the best result by reducing contention before buying hardware.

- **CPU**
  - Reduce thread oversubscription.
  - Tune process priority with `nice` or `renice`.
  - Pin workloads with `taskset` when locality matters.
  - Scale out stateless services.

- **Memory**
  - Lower application cache size or worker count.
  - Add RAM for legitimate working-set growth.
  - Investigate leaks with long-running process monitoring.
  - Use swap as a buffer, not as a substitute for adequate RAM.

- **Disk**
  - Move hot paths to SSD or NVMe.
  - Batch writes and reduce sync frequency.
  - Separate logs, databases, and backups onto different volumes when possible.
  - Monitor filesystem usage and keep free space available.

- **I/O**
  - Reduce remote dependency latency.
  - Optimize network paths and storage backends.
  - Spread interrupts and I/O queues across CPUs.
  - Use application-level buffering or asynchronous operations.

**Real-world examples**

A database server with high `await`, high `%util`, and processes in `D` state is likely storage-bound. Adding CPU will not help until the storage path improves.

A build server with high load average, high `%us`, and low `%wa` is likely CPU-bound. More cores or fewer parallel jobs will help.

A web server with low CPU but frequent swap activity and rising latency is likely memory-bound. Adding RAM or reducing cache pressure is the correct direction.

A file server backed by NFS that shows low local disk activity but slow application response is likely waiting on network I/O or the remote storage backend.

**Exam focus**

For Linux+ troubleshooting, remember these distinctions:

- **Load average** is not the same as CPU usage.
- **Free memory** is not the same as available memory.
- **High disk utilization** does not always mean high throughput.
- **I/O wait** means the CPU is idle while tasks wait on storage or another device.
- **Swap activity** is a strong signal of memory pressure.
- **Per-process tools** help you identify the offender, not just the symptom.

**Quick reference commands**

```bash
# CPU
top
mpstat -P ALL 1
pidstat -u 1

# Memory
free -h
vmstat 1
cat /proc/meminfo

# Disk and I/O
iostat -xz 1
iotop -oPa
pidstat -d 1

# System-wide clues
uptime
ps -eo pid,stat,cmd
```

**Troubleshooting mindset**

Always ask three questions:

- What resource is saturated?
- Which process or kernel path is responsible?
- What change will reduce contention without creating a new bottleneck?

That approach keeps you from treating symptoms in isolation and helps you choose the right fix for the workload.
