**Service failure fundamentals**

A Linux service can fail for many reasons: a bad configuration file, a missing dependency, a port conflict, permission problems, resource exhaustion, or an upstream service that is unavailable. On modern Linux systems, `systemd` is usually the service manager, so most troubleshooting starts with `systemctl` and `journalctl`.

A useful mental model is:

- **Unit state** tells you whether `systemd` considers the service active, failed, or inactive.
- **Process state** tells you whether the underlying daemon is actually running.
- **Logs** tell you why startup or runtime behavior failed.

When a service fails, do not restart blindly. First identify whether the failure is caused by the service itself or by something it depends on.

**Core `systemd` commands for failure analysis**

Use `systemctl` to inspect unit status and dependency relationships. Use `journalctl` to read logs from the current boot or a specific unit.

```bash
systemctl status nginx
systemctl is-active nginx
systemctl is-failed nginx
journalctl -u nginx
journalctl -u nginx -b
journalctl -xe
```

- `systemctl status <unit>` shows the current state, recent log lines, and the main process if available.
- `systemctl is-active <unit>` returns whether the unit is active.
- `systemctl is-failed <unit>` helps confirm a failure state.
- `journalctl -u <unit>` filters logs for a specific service.
- `journalctl -b` limits output to the current boot, which is useful after a reboot.
- `journalctl -xe` shows recent high-priority messages and often surfaces the first visible error.

**Reading service status output**

`systemctl status` is often the fastest way to identify the problem. It can reveal:

- The service unit name and loaded state.
- Whether the service exited cleanly or crashed.
- The exit code or signal that stopped the process.
- Recent log entries that point to the root cause.

Example output patterns to recognize:

- `Active: active (running)` means the service is up.
- `Active: failed` means `systemd` marked the unit as failed.
- `code=exited, status=1/FAILURE` often indicates the daemon started but rejected its configuration or runtime environment.
- `Main PID: ...` missing or gone usually means the process exited or never started successfully.

If a service fails immediately after start, the issue is often in configuration, permissions, or a missing file. If it runs for a while and then fails, look for runtime errors, resource limits, or dependency outages.

**Dependency issues and startup ordering**

Many services require other units to be available first. `systemd` models this with dependency relationships such as `Requires=`, `Wants=`, `After=`, and `Before=`. These relationships affect startup order and failure behavior.

- **Requires=** means the dependent unit must start successfully, or the service fails.
- **Wants=** means the dependency is desired but not mandatory.
- **After=** controls ordering only; it does not require the other unit to be active.
- **Before=** is the inverse ordering relationship.

A service may fail because:

- A required mount point is not available.
- A database service is down before a web application starts.
- A network target is not reached when the service expects connectivity.
- A socket, port, or file lock is already in use.

Use dependency inspection to understand the chain:

```bash
systemctl list-dependencies nginx
systemctl list-dependencies --reverse network-online.target
systemctl show nginx -p Requires -p Wants -p After -p Before
```

- `systemctl list-dependencies <unit>` shows what the unit pulls in.
- `--reverse` shows what depends on a unit, which helps identify blast radius.
- `systemctl show` exposes unit properties in a script-friendly format.

**Common failure causes**

Service failures usually fall into a few repeatable categories:

- **Bad configuration**
  - Syntax errors in config files.
  - Invalid paths, users, or directives.
  - Missing included files.
- **Missing dependency**
  - Required service not started.
  - Required filesystem not mounted.
  - Required network not ready.
- **Permission or ownership problems**
  - Daemon user cannot read config, keys, or data directories.
  - SELinux or AppArmor policy blocks access on systems that use them.
- **Port conflicts**
  - Another process already listens on the required port.
- **Resource exhaustion**
  - Disk full, inode exhaustion, memory pressure, or process limits.
- **Environment mismatch**
  - Wrong environment variables, working directory, or runtime path.
- **Crash loop**
  - Service starts, fails, and restarts repeatedly until rate limiting stops it.

A practical first-pass checklist is:

1. Confirm the unit name and current state.
2. Read the latest logs for the unit.
3. Validate configuration syntax if the daemon provides a test command.
4. Check dependencies, mounts, and network readiness.
5. Verify permissions, ownership, and available resources.
6. Inspect for port conflicts or stale lock files.

**Configuration validation examples**

Many daemons provide built-in validation commands. Use them before restarting a service after a config change.

```bash
nginx -t
apachectl configtest
sshd -t
named-checkconf
postfix check
```

- `nginx -t` tests NGINX configuration syntax.
- `apachectl configtest` validates Apache configuration.
- `sshd -t` checks OpenSSH daemon configuration.
- `named-checkconf` validates BIND configuration.
- `postfix check` performs Postfix consistency checks.

If the validation command reports an error, fix that before restarting. A service may fail even when the syntax is valid if referenced files or directories are missing.

**Practical log analysis**

Logs often show the exact reason for failure. Focus on the first error, not the last repeated message. A restart loop can generate many follow-on errors that hide the root cause.

```bash
journalctl -u nginx --since "10 minutes ago"
journalctl -u nginx -p err
journalctl -u nginx -o cat
```

- `--since` narrows the time window.
- `-p err` shows error-level messages and above.
- `-o cat` removes metadata for easier reading.

Typical log clues include:

- `permission denied` for file or directory access problems.
- `no such file or directory` for missing paths or sockets.
- `address already in use` for port conflicts.
- `dependency failed` when a required unit did not start.
- `failed to start` with a specific exit status from the daemon.

**Dependency failure scenarios**

Dependency problems often appear as a service that is technically configured correctly but cannot start because the environment is incomplete.

Examples:

- A web service depends on a database:
  - The web app starts before the database is ready.
  - The app exits because it cannot connect.
- A service depends on a mounted volume:
  - The mount is absent at boot.
  - The daemon cannot find its data directory.
- A network service depends on external connectivity:
  - The service starts too early in boot.
  - DNS or routing is not yet available.

To reduce these issues, use appropriate ordering and readiness targets. For network-dependent services, `network-online.target` is more appropriate than `network.target` when the service truly needs a usable network connection.

```ini
[Unit]
Description=Example app
After=network-online.target mariadb.service
Wants=network-online.target

[Service]
ExecStart=/usr/local/bin/example-app
Restart=on-failure

[Install]
WantedBy=multi-user.target
```

- `After=` ensures the app starts later than the network and database.
- `Wants=` requests the network-online target without making it mandatory.
- `Restart=on-failure` helps recover from transient startup errors, but it does not fix the root cause.

**Restart behavior and rate limiting**

`systemd` can restart failed services automatically, but repeated failures may trigger start-limit protection. This prevents endless crash loops.

Useful settings include:

- `Restart=` controls when restarts happen.
- `RestartSec=` sets the delay between attempts.
- `StartLimitBurst=` and `StartLimitIntervalSec=` control how many failures are allowed in a time window.

Example:

```ini
[Service]
ExecStart=/usr/local/bin/mydaemon
Restart=on-failure
RestartSec=5s

[Unit]
StartLimitIntervalSec=60
StartLimitBurst=3
```

If a service hits the start limit, `systemctl status` may show that it is failed due to repeated restarts. In that case, fix the underlying issue and then reset the failure state:

```bash
systemctl reset-failed mydaemon
systemctl start mydaemon
```

**Troubleshooting workflow**

Use a consistent workflow so you do not miss the real cause.

1. Check the unit state with `systemctl status <unit>`.
2. Read the unit logs with `journalctl -u <unit>`.
3. Validate the configuration with the daemon’s test command.
4. Inspect dependencies with `systemctl list-dependencies <unit>`.
5. Check for resource, permission, and port issues.
6. Restart only after you have corrected the cause.

This approach is faster than repeated restarts because it narrows the problem before changing state.

**Useful inspection commands**

```bash
systemctl cat nginx
systemctl show nginx -p FragmentPath -p DropInPaths -p ExecStart -p User
ss -tulpn
lsof -i :80
df -h
df -i
free -h
```

- `systemctl cat` shows the unit file and any drop-in overrides.
- `systemctl show` exposes key properties such as the executable path and service user.
- `ss -tulpn` identifies listening ports and owning processes.
- `lsof -i :80` helps find what is using a specific port.
- `df -h` checks disk space.
- `df -i` checks inode exhaustion.
- `free -h` checks memory availability.

**Overrides and drop-in fixes**

Sometimes the packaged unit file is correct, but the environment needs a local override. Use drop-in files instead of editing vendor unit files directly.

```bash
systemctl edit mydaemon
```

Example override:

```ini
[Service]
Environment="APP_ENV=production"
Restart=on-failure
RestartSec=3s
```

Drop-ins are useful when you need to:

- Change environment variables.
- Adjust restart behavior.
- Add dependencies or ordering.
- Modify resource limits.

After editing, reload the manager configuration:

```bash
systemctl daemon-reload
systemctl restart mydaemon
```

**When the dependency is a mount or filesystem**

Services that store data on a separate filesystem can fail if the mount is missing or delayed.

- Check mount status with `mount` or `findmnt`.
- Confirm the path exists and is writable.
- Verify the unit ordering if the service starts before the mount is ready.

```bash
findmnt /var/lib/myapp
systemctl status var-lib-myapp.mount
```

If the service needs a local directory, ensure ownership matches the service account:

```bash
chown -R myapp:myapp /var/lib/myapp
chmod 750 /var/lib/myapp
```

**When the dependency is the network**

Network-related failures are common for services that bind to interfaces, resolve hostnames, or contact remote APIs.

- Confirm the interface is up.
- Confirm DNS resolution works.
- Confirm the target port is reachable.
- Distinguish between `network.target` and `network-online.target`.

```bash
ip addr
ip route
resolvectl status
ping -c 3 8.8.8.8
getent hosts example.com
```

A service that needs a remote database should usually wait for the network to be truly online, not merely configured.

**Exam focus points**

- **`systemctl status`** is the first command to check for service failures.
- **`journalctl -u`** is the primary log source for a specific unit.
- **Dependency ordering** affects startup timing, but ordering alone does not guarantee availability.
- **`Requires=`** is stronger than `Wants=`.
- **`After=`** controls order, not readiness.
- **`network-online.target`** is more appropriate than `network.target` for services that need actual connectivity.
- **`systemctl reset-failed`** clears a failed state after you fix the root cause.
- **Validation commands** such as `nginx -t` or `sshd -t` help catch configuration errors before restart.

**Quick reference**

| Problem type | Common symptom | Useful command |
|---|---|---|
| Bad config | Service exits immediately | `nginx -t`, `systemctl status <unit>` |
| Missing dependency | Unit fails during startup | `systemctl list-dependencies <unit>` |
| Port conflict | Address already in use | `ss -tulpn`, `lsof -i :port` |
| Permission issue | Permission denied in logs | `journalctl -u <unit>`, `ls -l`, `namei -l <path>` |
| Resource exhaustion | Random crashes or startup failure | `df -h`, `df -i`, `free -h` |
| Crash loop | Repeated restarts then failure | `systemctl status <unit>`, `systemctl reset-failed <unit>` |

**Practical example: diagnosing a failed web service**

Suppose `nginx` fails after a configuration change.

```bash
systemctl status nginx
journalctl -u nginx -b
nginx -t
ss -tulpn | grep ':80'
```

A likely workflow is:

1. `systemctl status nginx` shows the service failed.
2. `journalctl -u nginx -b` reveals a syntax or permission error.
3. `nginx -t` confirms the configuration problem.
4. You fix the config and restart the service.

If the logs instead show `address already in use`, inspect the port owner and stop or reconfigure the conflicting service.

**Practical example: dependency failure at boot**

A custom app depends on MariaDB and a mounted data volume.

```ini
[Unit]
Description=Custom app
Requires=mariadb.service
After=mariadb.service network-online.target
Wants=network-online.target

[Service]
ExecStart=/opt/custom/bin/app
Restart=on-failure

[Install]
WantedBy=multi-user.target
```

If the app fails at boot:

- Check whether MariaDB is active.
- Confirm the data volume is mounted.
- Verify the app can read its config and data directories.
- Review logs for connection or filesystem errors.

**Key takeaway**

Service failures are easiest to solve when you separate the problem into three layers: the unit state, the logs, and the dependencies. `systemd` gives you the tools to inspect all three, and the fastest fix usually comes from identifying the first real error rather than restarting repeatedly.
