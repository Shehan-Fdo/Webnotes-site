## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Boot failure workflow**

Boot problems usually fall into a few stages: firmware, bootloader, kernel, initramfs, and userspace. You diagnose them by identifying the last stage that worked, then fixing the component immediately after it.

- **Firmware** issues appear before GRUB or the kernel starts.
- **Bootloader** issues often show GRUB errors, a rescue prompt, or a missing menu.
- **Kernel/initramfs** issues may boot partway, then stop with missing root device, filesystem, or driver errors.
- **Userspace** issues can reach the login screen but fail to start services or the graphical target.

A practical first step is to capture the exact error text and note whether the system uses BIOS or UEFI. That determines the repair path and where boot files live.

**Common symptoms and likely causes**

Boot failures are often caused by a small set of problems:

- **Wrong `root=` parameter** in the boot entry
- **Missing or corrupted initramfs**
- **Broken GRUB configuration**
- **Deleted or resized partitions**
- **Filesystem corruption**
- **Incorrect `/etc/fstab` entries**
- **Firmware boot entry problems on UEFI systems**
- **Disk or controller changes** after hardware replacement or virtualization edits

If the machine still reaches a shell, use that shell to inspect disks, mounts, and logs before making changes.

**GRUB basics**

GRUB (Grand Unified Bootloader) is the most common Linux bootloader. It loads the kernel and initramfs, then passes control to the kernel with boot parameters.

- On **BIOS** systems, GRUB may store its core image in the MBR (Master Boot Record) area or, on GPT disks, in a dedicated BIOS Boot Partition.
- On **UEFI** systems, GRUB is usually installed as an EFI executable under the EFI System Partition (ESP), often mounted at `/boot/efi`.
- GRUB configuration is commonly generated from `/etc/default/grub` and scripts in `/etc/grub.d/`.

Typical configuration files and paths:

```bash
/etc/default/grub
/etc/grub.d/
/boot/grub2/grub.cfg
/boot/grub/grub.cfg
/boot/efi/EFI/
```

The exact `grub.cfg` path varies by distribution. Some systems use `/boot/grub2/grub.cfg`, while others use `/boot/grub/grub.cfg`.

**GRUB rescue and GRUB prompt**

GRUB has different interactive modes, and they are not equally capable.

- `grub>` is the normal GRUB command line and supports more commands, including module loading on many systems.
- `grub rescue>` is a limited recovery shell with a small command set.
- In `grub rescue>`, you typically can inspect disks and set variables, but you usually cannot rely on module loading or full command support.

Useful commands at the normal GRUB prompt include:

```text
ls
set
linux
initrd
boot
insmod
normal
```

At `grub rescue>`, available commands are more limited and vary by build, but commonly include:

```text
ls
set
unset
root
prefix
normal
```

> `grub rescue>` is intentionally minimal. If you need `insmod` or richer recovery actions, you usually must first locate the correct GRUB modules and transition into normal mode.

**Recovering from a GRUB prompt**

If the system drops to a GRUB prompt, you can often boot manually by identifying the partition that contains `/boot` or the GRUB files.

1. List disks and partitions:
   ```text
   ls
   ```
2. Inspect partitions until you find the one containing kernel and initramfs files:
   ```text
   ls (hd0,gpt1)/
   ls (hd0,gpt2)/
   ```
3. Set the GRUB root to the partition that contains the boot files:
   ```text
   set root=(hd0,gpt2)
   ```
4. Load the kernel and initramfs using paths relative to that GRUB root:
   ```text
   linux /vmlinuz-6.8.0-xx root=/dev/sda2 ro
   initrd /initrd.img-6.8.0-xx
   boot
   ```

The exact filenames depend on the distribution and where GRUB’s `root` points. If the partition contains `/boot`, the kernel may be referenced as `/vmlinuz-...` and the initramfs as `/initrd.img-...`. If GRUB’s root is set differently, the paths change accordingly.

**Repairing GRUB from a live environment**

When the installed system will not boot, a live USB or rescue environment lets you mount the system and reinstall the bootloader.

A common recovery flow is:

1. Boot the live environment.
2. Identify the Linux root partition and, if separate, the `/boot` and EFI System Partition.
3. Mount the installed system under `/mnt`.
4. Bind-mount runtime filesystems if needed.
5. Chroot into the installed system.
6. Reinstall GRUB and regenerate the configuration.

Example layout:

```bash
lsblk -f
```

Mount the root filesystem:

```bash
mount /dev/sda2 /mnt
```

If `/boot` is separate:

```bash
mount /dev/sda3 /mnt/boot
```

If using UEFI, mount the ESP:

```bash
mount /dev/sda1 /mnt/boot/efi
```

Bind-mount runtime filesystems and chroot:

```bash
mount --bind /dev /mnt/dev
mount --bind /proc /mnt/proc
mount --bind /sys /mnt/sys
chroot /mnt
```

Then repair GRUB using the appropriate platform-specific command.

**BIOS recovery**

On BIOS systems, reinstalling GRUB usually writes boot code to the disk and rebuilds the GRUB core image location. On GPT disks, that may involve a BIOS Boot Partition rather than the MBR alone.

Example:

```bash
grub-install /dev/sda
grub-mkconfig -o /boot/grub2/grub.cfg
```

Some distributions use different output paths, such as `/boot/grub/grub.cfg`. Use the path expected by the installed system.

Key points:

- Install to the disk, not a partition, for BIOS bootloader recovery.
- Verify the disk name carefully before running `grub-install`.
- If the system uses GPT with BIOS boot, ensure the BIOS Boot Partition exists and is large enough.

**UEFI recovery**

On UEFI systems, GRUB is installed to the EFI System Partition and the firmware boot entry must point to the correct EFI binary.

A typical repair sequence is:

```bash
grub-install --target=x86_64-efi --efi-directory=/boot/efi --bootloader-id=linux
grub-mkconfig -o /boot/grub2/grub.cfg
```

Depending on the environment, you may also need to create or repair the NVRAM (non-volatile RAM) boot entry, or use `--removable` if firmware boot entries are broken or unavailable.

Examples:

```bash
grub-install --target=x86_64-efi --efi-directory=/boot/efi --bootloader-id=linux --recheck
```

Fallback-style install for some recovery cases:

```bash
grub-install --target=x86_64-efi --efi-directory=/boot/efi --removable
```

Useful verification commands:

```bash
efibootmgr -v
ls /boot/efi/EFI/
```

Important UEFI considerations:

- The EFI binary path must match the firmware’s expectations.
- Some systems require a vendor-specific directory under `/boot/efi/EFI/<vendor>/`.
- Installing files into the ESP does not always restore bootability if the firmware boot entry is missing or wrong.

**Kernel and initramfs recovery**

If GRUB loads but the kernel cannot find the root filesystem, the problem is often the boot parameters, storage drivers, or initramfs contents.

Common kernel command-line issues:

- Wrong `root=` device
- Missing `ro` or `rw` expectation mismatch
- Incorrect `resume=` device for hibernation
- Missing `rd.lvm.lv=`, `rd.md=`, or `rd.luks.uuid=` parameters on systems using LVM, RAID, or encryption

You can inspect the current boot entry in GRUB and edit it temporarily by pressing `e` at the menu. This is useful for testing a corrected `root=` value before making permanent changes.

Example kernel line:

```text
linux /vmlinuz-6.8.0-xx root=/dev/mapper/vg0-root ro quiet
```

If the initramfs is missing drivers or storage hooks, regenerate it from the installed system.

Examples:

```bash
dracut -f
```

or on some distributions:

```bash
update-initramfs -u
```

Use the tool that matches the distribution. The goal is to include the modules needed to access the root filesystem early in boot.

**Filesystem repair**

Filesystem corruption can prevent the kernel from mounting the root filesystem or can trigger emergency mode.

General guidance:

- Repair filesystems from rescue media when possible.
- Unmount the filesystem before running repair tools.
- Use the tool that matches the filesystem type.

Examples:

```bash
fsck -f /dev/sda2
e2fsck -f /dev/sda2
xfs_repair /dev/sda2
btrfs check /dev/sda2
```

Notes by filesystem:

- **ext2/ext3/ext4**: `fsck` or `e2fsck` can repair inconsistencies.
- **XFS**: `xfs_repair` is used offline only; the filesystem must be unmounted. If the root filesystem is XFS, boot from rescue media or another environment before repairing it.
- **Btrfs**: `btrfs check` is available, but use it carefully because some repair options are risky.

> Do not run repair tools on a mounted filesystem unless the tool explicitly supports that mode. For root filesystem repair, boot into rescue mode or from external media.

**`fstab` and mount validation**

Incorrect `/etc/fstab` entries can stop boot or force emergency mode. This is especially common after disk UUID changes, partition renumbering, or adding new storage.

Check the file:

```bash
cat /etc/fstab
blkid
```

Validate mount entries:

```bash
mount -a
```

`mount -a` attempts to mount entries that are not already mounted and reports syntax or device problems, but it does not fully prove that every boot-critical mount will succeed in the same way during startup. Options such as `noauto` or some `x-systemd.*` settings can change behavior.

Useful checks:

- Confirm UUIDs with `blkid`
- Verify mount points exist
- Ensure filesystem types are correct
- Check for typos in options and dump/pass fields

Example `fstab` entry:

```fstab
UUID=1234-ABCD  /      ext4  defaults  0 1
UUID=5678-EFGH  /boot  ext4  defaults  0 2
```

**Emergency mode and rescue mode**

If systemd cannot mount critical filesystems or start essential services, it may drop into emergency mode or rescue mode.

- **Emergency mode** provides a minimal shell with the root filesystem often mounted read-only.
- **Rescue mode** starts a minimal set of services and is useful for repair tasks.
- You may need the root password or console access to continue.

Typical actions in these modes:

- Remount root read-write if needed:
  ```bash
  mount -o remount,rw /
  ```
- Inspect logs:
  ```bash
  journalctl -xb
  ```
- Check failed units:
  ```bash
  systemctl --failed
  ```

These modes are useful because they let you repair configuration without a full live-USB boot, but they still depend on the system reaching a minimally functional state.

**Logs and diagnostics**

Logs help you identify whether the failure is bootloader, kernel, or userspace related.

Useful commands:

```bash
journalctl -xb
dmesg
systemctl status
systemctl list-units --failed
```

What to look for:

- Missing device nodes or UUID mismatches
- Filesystem mount failures
- Kernel panic messages
- Driver or module load failures
- SELinux or permission denials that block services after boot

If the system does not boot far enough for `journalctl`, use the live environment to inspect logs under `/var/log/` or mount the system and review journal files if available.

**Practical recovery checklist**

1. Identify the stage of failure.
2. Confirm BIOS or UEFI firmware mode.
3. Check disk and partition layout with `lsblk -f` and `blkid`.
4. Verify `/etc/fstab` and boot parameters.
5. Repair the filesystem if corruption is present.
6. Rebuild the initramfs if storage drivers or root access are missing.
7. Reinstall GRUB for the correct firmware mode.
8. Regenerate the GRUB configuration.
9. Verify firmware boot entries on UEFI systems.
10. Reboot and confirm the system reaches the expected target.

**Example recovery session**

```bash
lsblk -f
mount /dev/sda2 /mnt
mount /dev/sda1 /mnt/boot/efi
mount --bind /dev /mnt/dev
mount --bind /proc /mnt/proc
mount --bind /sys /mnt/sys
chroot /mnt
grub-install --target=x86_64-efi --efi-directory=/boot/efi --bootloader-id=linux
grub-mkconfig -o /boot/grub2/grub.cfg
dracut -f
exit
reboot
```

This example assumes a UEFI system with `/dev/sda2` as root and `/dev/sda1` as the ESP. Adjust device names and output paths for the actual distribution.

**Exam focus**

For Linux+ exam prep, know how to:

- Distinguish BIOS from UEFI recovery steps
- Interpret GRUB and kernel boot errors
- Use rescue media and `chroot` effectively
- Reinstall GRUB and regenerate configuration files
- Repair filesystems safely
- Validate `/etc/fstab` and boot parameters
- Use logs to confirm the failure point
