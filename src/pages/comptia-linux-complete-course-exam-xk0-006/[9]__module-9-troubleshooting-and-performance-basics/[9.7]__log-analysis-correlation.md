**Log analysis and correlation**

Log analysis is the process of reading system, service, and application logs to understand what happened, when it happened, and which component was involved. Correlation goes a step further: you compare multiple log sources and system signals to connect related events into a single timeline. In practice, this helps you troubleshoot outages, confirm security incidents, and distinguish root cause from symptoms.

A strong workflow is to start broad, narrow by time and host, then correlate across services. For example, a failed web login may appear in the web server log, the authentication log, and the database log. If you only inspect one file, you may miss the real failure point.

**Why log analysis matters**

Logs are often the fastest way to answer operational questions:

- What changed before the failure?
- Which service restarted?
- Did authentication fail, or did the backend database stop responding?
- Was the issue local to one host or visible across several systems?

Logs also provide evidence for incident response. They help you verify whether a problem is a configuration error, a resource exhaustion issue, a permission problem, or suspicious activity.

**Common Linux log locations**

Linux distributions vary, but these locations are common:

- `/var/log/` contains persistent logs on many systems.
- `journalctl` reads the systemd journal.
- `/var/log/syslog` is common on Debian-based systems.
- `/var/log/messages` is common on Red Hat-based systems.
- `/var/log/auth.log` or `/var/log/secure` records authentication events.
- `/var/log/kern.log` may contain kernel messages.
- Application-specific logs often live under `/var/log/<service>/` or the application’s own data directory.

> Not every system writes the same events to the same files. Modern distributions may rely heavily on the systemd journal, while some services still write traditional text logs.

**Systemd journal basics**

The systemd journal stores structured logs and supports filtering by time, boot, service, priority, and other fields. It is especially useful when you need to correlate events across a boot cycle or inspect logs from a service unit.

Common commands:

```bash
journalctl
journalctl -u sshd
journalctl -p err
journalctl --since "2026-04-26 08:00:00" --until "2026-04-26 09:00:00"
journalctl -b
journalctl -k
```

Useful options:

- `-u` filters by systemd unit.
- `-p` filters by priority, such as `err`, `warning`, or `info`.
- `--since` and `--until` define a time window.
- `-b` shows logs from the current boot.
- `-k` shows kernel messages.

When troubleshooting a service, start with the unit log and then widen the scope to related services and the kernel.

**Reading and filtering text logs**

Traditional log files are usually plain text, so you use standard Unix tools to search and summarize them.

Examples:

```bash
grep -i "error" /var/log/syslog
grep -E "fail|denied|refused" /var/log/secure
tail -f /var/log/messages
less /var/log/auth.log
```

Common tools and patterns:

- `grep` finds matching lines.
- `less` lets you page through large logs.
- `tail -f` follows a live log as new entries arrive.
- `head` and `tail` help inspect the beginning or end of a file.
- `awk` and `cut` extract fields from structured text.
- `sort` and `uniq -c` help count repeated events.

Example of counting repeated messages:

```bash
grep "sshd" /var/log/auth.log | awk '{print $1, $2, $3, $NF}' | sort | uniq -c
```

This kind of summary helps you spot bursts of failures or repeated restarts.

**Correlation techniques**

Correlation means connecting events that share a time window, host, user, process, or request ID. You rarely solve a problem from one log line alone.

Useful correlation dimensions:

- **Time**: compare events that happened just before and after the failure.
- **Host**: check whether multiple servers show the same symptom.
- **Service**: follow the request path across web, app, and database layers.
- **User**: match authentication events to account activity.
- **Process**: identify which daemon or child process generated the message.
- **Request ID**: if applications emit transaction IDs, use them to trace a single request.

A practical correlation workflow:

1. Identify the first visible symptom.
2. Note the exact timestamp and affected host.
3. Check the primary service log.
4. Check authentication, kernel, and dependent service logs.
5. Compare the same time window on related systems.
6. Confirm whether the earliest error is the root cause or a downstream failure.

**Example incident timeline**

Suppose a user reports that a web app returns `502 Bad Gateway`.

You might inspect:

```bash
journalctl --since "2026-04-26 10:15:00" --until "2026-04-26 10:25:00" -u nginx
journalctl --since "2026-04-26 10:15:00" --until "2026-04-26 10:25:00" -u php-fpm
journalctl --since "2026-04-26 10:15:00" --until "2026-04-26 10:25:00" -u mariadb
```

Possible interpretation:

- Nginx reports upstream connection failures.
- PHP-FPM shows worker exhaustion or crashes.
- MariaDB shows connection refusals or slow queries.
- The earliest failure may be in the database, with the web error appearing later.

This is correlation in action: the visible error is not always the root cause.

**Priorities and severity**

Log entries often include a severity level. In the systemd journal, priorities range from emergency to debug. Filtering by severity helps you focus on actionable events.

Typical levels:

- `emerg`
- `alert`
- `crit`
- `err`
- `warning`
- `notice`
- `info`
- `debug`

Use higher-severity filters when you need to reduce noise:

```bash
journalctl -p warning
journalctl -p err..alert
```

Severity filtering is useful during outages because it surfaces failures without requiring you to read every informational message.

**Authentication and security analysis**

Authentication logs are critical for detecting brute-force attempts, privilege escalation, and unauthorized access.

Look for:

- repeated failed logins
- invalid users
- unexpected `sudo` usage
- SSH key failures
- logins from unusual source addresses
- account lockouts

Examples:

```bash
grep "Failed password" /var/log/auth.log
grep "sudo" /var/log/secure
journalctl -u sshd --since "today"
```

Security correlation often requires comparing:

- SSH logs
- PAM (Pluggable Authentication Modules) messages
- `sudo` logs
- audit logs from `auditd`
- firewall logs

If you see a successful login after many failures, check whether the source IP, username, and time align with a known maintenance window or indicate suspicious activity.

**Kernel and hardware clues**

Not all problems are application problems. Kernel logs can reveal hardware faults, driver issues, filesystem errors, and out-of-memory conditions.

Examples:

```bash
journalctl -k
dmesg
dmesg -T | tail -n 50
```

Look for:

- disk I/O errors
- filesystem remounts as read-only
- memory pressure or OOM killer events
- NIC or link failures
- USB or device enumeration issues
- kernel panics or taint messages

Kernel messages are especially valuable when a service fails without leaving a clear application log.

**Log rotation and retention**

Logs can grow quickly, so Linux systems use rotation and retention policies to manage disk usage. If you cannot find an older event, it may have been rotated or expired.

Common rotation tools and files:

- `logrotate` manages text log rotation.
- `/etc/logrotate.conf` contains global settings.
- `/etc/logrotate.d/` contains per-service rules.
- Compressed archives often end in `.gz`.

Examples:

```bash
ls /var/log/
zgrep "error" /var/log/syslog.1.gz
logrotate -d /etc/logrotate.conf
```

Important considerations:

- Rotated logs may be compressed.
- Some services reopen logs automatically; others need a reload.
- The journal may also have size and retention limits.

If a log seems missing, check whether rotation, retention, or journald limits removed it.

**Practical command patterns**

These commands are common during troubleshooting:

```bash
# Follow a live log
tail -f /var/log/messages

# Search for authentication failures
grep -i "failed" /var/log/auth.log

# Inspect the current boot only
journalctl -b

# Show logs for a service since a specific time
journalctl -u nginx --since "1 hour ago"

# Search compressed rotated logs
zgrep "kernel" /var/log/messages-*.gz
```

Use `less` when you need to move back and forth through a large log, and use `grep` when you already know the pattern you want.

**Structured logging and fields**

The systemd journal stores metadata such as:

- `_PID` for process ID
- `_SYSTEMD_UNIT` for the service unit
- `_HOSTNAME` for the host
- `PRIORITY` for severity
- `MESSAGE` for the log text

You can query fields directly:

```bash
journalctl _SYSTEMD_UNIT=sshd.service
journalctl _PID=1234
journalctl PRIORITY=3
```

Structured fields make correlation easier because you can filter on machine-readable metadata instead of parsing text manually.

**Remote and centralized logging**

In larger environments, logs are often forwarded to a central system for retention and analysis. This improves correlation across hosts and makes it easier to search historical data.

Common approaches include:

- forwarding syslog to a remote collector
- shipping logs with agents
- storing logs in a SIEM (Security Information and Event Management) platform
- using centralized dashboards for alerts and trends

A simple syslog forwarding example:

```conf
*.* @@loghost.example.com:514
```

This sends logs to a remote server using TCP. Use the correct protocol and port for your environment and logging policy.

Centralized logging is useful when:

- multiple hosts participate in one incident
- local logs rotate too quickly
- you need long-term retention
- you want alerting and dashboards

**Troubleshooting strategy**

A reliable troubleshooting sequence reduces guesswork:

1. Confirm the symptom and exact time.
2. Identify the affected host and service.
3. Check the service log first.
4. Check dependent services and kernel logs.
5. Compare with other hosts for scope.
6. Look for the earliest error, not just the loudest one.
7. Validate the fix by reproducing or monitoring the event again.

This approach helps you avoid chasing downstream errors. For example, a database timeout may cause application failures, but the first log entry may be a disk latency warning or a network interruption.

**Real-world examples**

Example: find SSH failures in the last hour:

```bash
journalctl -u sshd --since "1 hour ago" | grep -i "failed"
```

Example: inspect a boot for storage errors:

```bash
journalctl -b -k | grep -iE "error|fail|timeout|I/O"
```

Example: correlate a service restart with a crash:

```bash
journalctl -u httpd --since "2026-04-26 09:00:00" --until "2026-04-26 10:00:00"
```

Look for:

- `Started` and `Stopped` messages
- exit codes
- segmentation faults
- watchdog timeouts
- dependency failures

**Best practices**

- Use a narrow time window first, then expand if needed.
- Correlate across at least two sources before concluding root cause.
- Preserve logs during incidents if you suspect rotation may remove evidence.
- Prefer structured fields when available.
- Record timestamps in UTC when coordinating across systems.
- Verify whether the system uses text logs, journald, or both.
- Treat repeated messages as a pattern, not isolated noise.

**Common pitfalls**

- Assuming the last error is the root cause.
- Checking only one log file.
- Ignoring time zone differences.
- Forgetting that logs may have rotated.
- Overlooking kernel messages when an application log is empty.
- Misreading severity levels or filtering too narrowly.

**Key takeaway**

Effective log analysis is about building a timeline. Start with the symptom, gather evidence from the relevant logs, and correlate events across services and hosts until the earliest meaningful failure becomes clear. That method is essential for both troubleshooting and security investigation.
