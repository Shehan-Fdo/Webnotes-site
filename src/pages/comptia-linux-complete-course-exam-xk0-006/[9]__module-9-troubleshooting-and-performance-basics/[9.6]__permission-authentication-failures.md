## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Permission vs. authentication failures**

Linux access problems usually fall into two broad categories:

- **Authentication** failures happen when the system cannot verify who you are.
- **Authorization** failures happen when the system knows who you are but denies access to a resource.

This distinction matters because the fix is different. If authentication fails, you troubleshoot credentials, identity sources, PAM (Pluggable Authentication Modules), SSH, or account state. If authorization fails, you inspect file permissions, ownership, ACLs (Access Control Lists), SELinux/AppArmor policy, sudo rules, or service-specific access controls.

A common exam trap is assuming every `Permission denied` message means file mode bits are wrong. In practice, the denial may come from a parent directory, a MAC (Mandatory Access Control) policy, a mount option, or a service policy.

**Core Linux permission model**

Linux evaluates access using several layers. You should check them in order because the first denial often explains the symptom.

- **User, group, other** permissions control read, write, and execute access on files and directories.
- **Ownership** determines which user and group receive the owner and group permission sets.
- **Special bits** such as setuid, setgid, and sticky bit modify normal behavior.
- **ACLs** can grant or deny access beyond the basic mode bits.
- **Security policy** from SELinux or AppArmor can block access even when mode bits look correct.
- **Mount options** such as `ro`, `noexec`, or `nosuid` can prevent expected actions.

Use `ls -l` for mode bits and ownership, and `getfacl` when ACLs may be involved.

```bash
ls -l /srv/share/report.txt
getfacl /srv/share/report.txt
stat /srv/share/report.txt
```

**Reading file and directory permissions**

File permissions and directory permissions are not the same. A file with `r--` can be read, but a directory with `r--` alone is not enough to traverse into it. For directories:

- **Read (`r`)** lets you list names in the directory.
- **Write (`w`)** lets you create, delete, or rename entries.
- **Execute (`x`)** lets you traverse the directory and access metadata for entries.

For files:

- **Read (`r`)** lets you view contents.
- **Write (`w`)** lets you modify contents.
- **Execute (`x`)** lets you run the file as a program or script.

A frequent failure pattern is a user who can see a file name but cannot open it because one of the parent directories lacks execute permission.

```bash
namei -l /opt/app/config/settings.conf
```

`namei -l` is especially useful because it shows permissions on every path component, not just the target file.

**Common permission failure symptoms**

- `Permission denied` when opening a file usually indicates missing file or directory access.
- `Operation not permitted` often appears when a privileged action is blocked by policy, immutable attributes, or capability restrictions.
- `No such file or directory` can still happen when the file exists but a parent directory is inaccessible or a symlink target cannot be resolved.
- `Read-only file system` indicates a mount or filesystem state issue rather than a normal permission problem.

When troubleshooting, verify the exact command, the target path, and the user context. A command that works with `sudo` but fails without it usually points to authorization, not authentication.

**Ownership and group-based access**

Ownership is the first thing to check after mode bits. Every file has a user owner and a group owner.

- Use `chown` to change user ownership.
- Use `chgrp` or `chown :group` to change group ownership.
- Use `usermod -aG` to add a user to a supplementary group.

```bash
sudo chown alice:devops /srv/share/report.txt
sudo chgrp devops /srv/share/report.txt
sudo usermod -aG devops alice
```

Group membership changes usually require the user to log out and back in before the new group list is active. You can verify current groups with `id`.

```bash
id alice
groups alice
```

**Special permissions**

Special bits change how access works and are common in exam scenarios.

- **setuid** on an executable runs it with the file owner’s effective user ID.
- **setgid** on an executable runs it with the file group’s effective group ID.
- **setgid** on a directory causes new files and subdirectories to inherit the directory’s group.
- **sticky bit** on a directory prevents users from deleting files they do not own, even if the directory is writable.

Examples:

```bash
chmod u+s /usr/local/bin/helper
chmod g+s /srv/project
chmod +t /tmp/shared
```

Use `ls -l` to spot special bits:

- `s` in the user or group execute position indicates setuid or setgid.
- `t` in the others execute position indicates sticky bit.

> Be careful with setuid and setgid on custom binaries. They can create privilege escalation risk if the program is not written securely.

**ACLs and extended access control**

ACLs let you grant permissions to specific users or groups without changing the basic owner/group/other model. They are useful when multiple teams need different access to the same file tree.

Common commands:

```bash
getfacl /srv/share
setfacl -m u:alice:rw /srv/share/report.txt
setfacl -m g:devops:rwx /srv/share
setfacl -x u:alice /srv/share/report.txt
```

Important points:

- A file may look inaccessible in `ls -l` but still be accessible through an ACL.
- A directory ACL may include a **default ACL** that applies to new files created inside it.
- The effective permission is the combination of mode bits, ACLs, and any security policy.

Use `getfacl` when the mode bits do not explain the denial.

**Authentication stack and PAM**

Authentication failures happen before access is granted. Linux often uses PAM to centralize authentication, account checks, password policy, and session setup.

PAM can fail because of:

- Incorrect password or expired password
- Locked or expired account
- Invalid PAM module configuration
- Missing or broken identity source such as LDAP, Kerberos, or SSSD
- Denied login shell or restricted access rules

Common files and tools:

```bash
cat /etc/pam.d/sshd
cat /etc/pam.d/login
passwd -S alice
chage -l alice
```

If a user cannot log in locally or over SSH, check account status first:

```bash
sudo passwd -S alice
sudo chage -l alice
sudo usermod -U alice
```

`passwd -S` shows whether the password is locked, and `chage -l` shows password aging information. An account can be valid but still fail login because the password is expired or the account is locked.

**SSH authentication failures**

SSH is a common source of authentication and authorization confusion. A user may authenticate successfully but still be denied by server-side policy.

Typical checks:

- Confirm the username and host are correct.
- Verify the account is not locked or expired.
- Check `~/.ssh/authorized_keys` permissions and ownership.
- Ensure the private key is readable only by the owner.
- Review `/etc/ssh/sshd_config` for login restrictions.

```bash
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys
chmod 600 ~/.ssh/id_ed25519
chown -R alice:alice ~/.ssh
```

SSH is strict about permissions. If the home directory, `.ssh` directory, or `authorized_keys` file is too open, `sshd` may ignore the key and fall back to password authentication or deny access entirely.

Useful diagnostics:

```bash
ssh -vvv alice@server.example.com
sudo journalctl -u sshd -b
```

**sudo and privilege escalation failures**

`sudo` failures are often authorization failures, not authentication failures. The user may know the password but still lack permission to run the command.

Common causes:

- The user is not in the allowed sudoers group.
- The command is not permitted by `/etc/sudoers` or `/etc/sudoers.d/*`.
- The command path does not match the sudoers rule.
- The user entered the wrong password or the timestamp expired.
- PAM or policy plugins deny the request.

Check sudo access safely with:

```bash
sudo -l
visudo
```

Example sudoers entry:

```bash
alice ALL=(ALL) /usr/bin/systemctl restart nginx
```

If a command is denied, verify the exact path with `which` or `command -v`:

```bash
command -v systemctl
```

**SELinux and AppArmor denials**

Mandatory Access Control can block access even when traditional permissions are correct.

SELinux clues:

- `Permission denied` with correct mode bits
- AVC (Access Vector Cache) denial messages in logs
- Wrong file context or process context

Useful commands:

```bash
getenforce
sestatus
ls -Z /var/www/html/index.html
restorecon -Rv /var/www/html
ausearch -m avc -ts recent
```

AppArmor clues:

- Denials appear in system logs or `dmesg`
- Profiles may be in enforce or complain mode

Useful commands:

```bash
aa-status
aa-complain /usr/sbin/nginx
aa-enforce /usr/sbin/nginx
```

If a service works after temporarily disabling a profile, the issue is likely policy-related rather than file permissions.

**Filesystem and mount-related denials**

Sometimes the problem is not permissions at all.

- A filesystem mounted `ro` prevents writes.
- `noexec` prevents execution of binaries and scripts from that mount.
- `nosuid` disables setuid and setgid behavior.
- Network filesystems may enforce server-side permissions or root squashing.

Check mounts with:

```bash
findmnt -o TARGET,SOURCE,FSTYPE,OPTIONS
mount | grep '/srv'
```

Example mount options:

```bash
UUID=1234-ABCD  /data  ext4  defaults,noexec,nosuid  0  2
```

If a script fails with `Permission denied` on an executable mount, confirm both the execute bit and the mount options.

**Troubleshooting workflow**

Use a consistent sequence so you do not miss the real cause.

1. Reproduce the failure as the affected user.
2. Read the exact error message and note the path or command.
3. Check ownership and mode bits with `ls -l`.
4. Check every parent directory with `namei -l`.
5. Inspect ACLs with `getfacl`.
6. Verify account state with `id`, `passwd -S`, and `chage -l`.
7. Review sudo rules if privilege escalation is involved.
8. Check SSH, PAM, SELinux, AppArmor, and logs if the issue persists.
9. Confirm mount options and filesystem state.
10. Apply the smallest safe fix and retest.

A practical example:

```bash
id alice
namei -l /srv/share/project/data.csv
getfacl /srv/share/project/data.csv
sudo journalctl -xe
```

**Real-world examples**

A user cannot read a file in a shared directory:

```bash
ls -ld /srv/share /srv/share/project
ls -l /srv/share/project/data.csv
getfacl /srv/share/project/data.csv
```

Possible fixes:

- Add execute permission to the parent directory.
- Add the user to the correct group.
- Set an ACL for the user.
- Correct SELinux context if the path is under a labeled service directory.

A user cannot SSH in with a valid key:

```bash
ls -ld ~ ~/.ssh
ls -l ~/.ssh/authorized_keys
journalctl -u sshd -b
```

Possible fixes:

- Tighten permissions on `.ssh` and `authorized_keys`.
- Correct ownership of the home directory and SSH files.
- Ensure the account is not locked or expired.
- Verify the key is actually present in `authorized_keys`.

A service cannot write to a directory even though it is writable:

```bash
findmnt -o TARGET,OPTIONS /var/lib/app
ls -ldZ /var/lib/app
```

Possible fixes:

- Remove `ro` mount state.
- Adjust SELinux context with `restorecon`.
- Confirm the service runs under the expected user and group.

**Quick reference**

| Problem type | Likely cause | Useful command |
|---|---|---|
| Cannot log in | Locked, expired, or invalid account | `passwd -S`, `chage -l` |
| `Permission denied` on file | Mode bits, ACLs, or parent directory | `ls -l`, `namei -l`, `getfacl` |
| SSH key ignored | Bad permissions or ownership | `ls -ld ~/.ssh`, `chmod 700 ~/.ssh` |
| `sudo` denied | Missing sudoers rule | `sudo -l`, `visudo` |
| Access denied with correct permissions | SELinux/AppArmor | `getenforce`, `aa-status`, logs |
| Write fails unexpectedly | Read-only mount or policy | `findmnt`, `mount` |

**Key exam takeaways**

- **Authentication** proves identity; **authorization** decides access.
- Check parent directories, not just the target file.
- ACLs can override what `ls -l` seems to show.
- SSH and sudo have their own permission expectations.
- SELinux and AppArmor can deny access even when Unix permissions are correct.
- Mount options can block execution, writing, or setuid behavior.
