## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Container troubleshooting fundamentals**

Container issues usually fall into a few categories: the container never starts, it starts but exits immediately, it cannot reach the network, it cannot access required files or volumes, or the application inside the container fails because of environment or permission problems. Your first job is to identify whether the failure is at the image, runtime, host, or application layer.

A practical troubleshooting flow is:

1. Confirm the container exists and inspect its state.
2. Read logs from the container and the runtime.
3. Verify image integrity, tags, and entrypoint/command behavior.
4. Check networking, storage mounts, and permissions.
5. Reproduce the issue with a minimal run command.

Common container runtimes include Docker and Podman. The troubleshooting concepts are similar even when the command syntax differs slightly.

**Container lifecycle and state checks**

A container can be `created`, `running`, `paused`, `exited`, or `dead`. The state tells you where to focus.

- **Running** containers may still be unhealthy if the application inside is broken.
- **Exited** containers often indicate a bad command, missing dependency, or crash on startup.
- **Paused** containers stop process execution but keep state in memory.
- **Dead** containers usually indicate a runtime cleanup problem or an unrecoverable failure.

Useful commands:

```bash
docker ps -a
podman ps -a
docker inspect <container_name_or_id>
podman inspect <container_name_or_id>
```

Look for these fields in `inspect` output:

- `State.Status`
- `State.ExitCode`
- `State.Error`
- `State.OOMKilled`
- `Config.Entrypoint`
- `Config.Cmd`
- `Mounts`
- `NetworkSettings`

If a container exits immediately, the exit code is often the fastest clue:

- `0` usually means the process completed normally.
- `1` often indicates a generic application error.
- `126` usually means the command was found but not executable.
- `127` usually means the command was not found.
- `137` often indicates the process was killed, commonly by the kernel out-of-memory (OOM) killer or a forced stop.
- `139` usually indicates a segmentation fault.

**Logs and runtime output**

Logs are the primary source of truth when the application starts and then fails. Container stdout and stderr are typically captured by the runtime.

```bash
docker logs <container_name_or_id>
docker logs -f <container_name_or_id>
podman logs <container_name_or_id>
podman logs --follow <container_name_or_id>
```

If logs are empty, consider these possibilities:

- The process never started.
- The entrypoint failed before the application initialized.
- Logging is redirected to a file inside the container.
- The application writes to a location that is not mounted or not writable.

For system-level clues, check the host journal and runtime service logs:

```bash
journalctl -u docker
journalctl -u podman
journalctl -u containerd
```

On systems using `systemd`, runtime service failures can prevent containers from starting even when the image is valid.

**Image and command troubleshooting**

A common failure is a mismatch between the image’s default command and the container’s runtime overrides. The image may define an `ENTRYPOINT` and `CMD`, and your run command can replace or append to them depending on the runtime behavior.

Typical causes include:

- The image tag points to the wrong version.
- The image was built for a different CPU architecture.
- The entrypoint script lacks execute permission.
- The container command references a missing binary.
- Environment variables required by the app are absent.

Useful checks:

```bash
docker image ls
podman images
docker inspect <image_name_or_id>
podman inspect <image_name_or_id>
```

To test a container image interactively, override the entrypoint and inspect the filesystem:

```bash
docker run --rm -it --entrypoint /bin/sh myapp:latest
podman run --rm -it --entrypoint /bin/sh myapp:latest
```

If the image is multi-architecture, verify the host architecture matches the image platform:

```bash
uname -m
docker image inspect myapp:latest --format '{{.Os}}/{{.Architecture}}'
podman image inspect myapp:latest --format '{{.Os}}/{{.Architecture}}'
```

**Networking problems**

Container networking issues often appear as DNS failures, connection timeouts, or inability to reach published ports. The problem may be inside the container, on the host firewall, or in the container network configuration.

Common checks:

- Confirm the container has an IP address.
- Verify the application listens on the expected port.
- Ensure the port is published to the host.
- Test DNS resolution inside the container.
- Check firewall rules and SELinux or AppArmor restrictions if applicable.

Commands:

```bash
docker port <container_name_or_id>
podman port <container_name_or_id>
docker network ls
podman network ls
docker network inspect bridge
podman network inspect podman
```

Inside the container, test name resolution and connectivity:

```bash
getent hosts example.com
ping -c 1 8.8.8.8
curl -I http://localhost:8080
curl -I http://<service_name>:<port>
```

If the containerized app listens only on `127.0.0.1`, it may work inside the container but remain unreachable from outside. Bind services to `0.0.0.0` when you need external access.

Example application binding:

```bash
python3 -m http.server 8080 --bind 0.0.0.0
```

Port publishing example:

```bash
docker run -d --name web -p 8080:80 nginx
podman run -d --name web -p 8080:80 nginx
```

If DNS fails inside the container, inspect the resolver configuration:

```bash
cat /etc/resolv.conf
```

A broken DNS setup can come from the host, the runtime, or custom network settings.

**Storage, mounts, and permissions**

Volume and bind-mount issues are common because the container process often runs as a non-root user or with a different UID (user ID) and GID (group ID) than the host files. A mount can exist but still be unusable if permissions are wrong.

Check mounts and ownership:

```bash
docker inspect <container_name_or_id> --format '{{json .Mounts}}'
podman inspect <container_name_or_id> --format '{{json .Mounts}}'
ls -ld /host/path
ls -l /host/path
id
```

Typical problems include:

- The host path does not exist, so the runtime creates an unexpected directory.
- The container user cannot write to the mounted directory.
- SELinux labels block access on enforcing systems.
- Read-only mounts prevent the application from creating files.

Example bind mount:

```bash
docker run --rm -it -v /srv/app/data:/data myapp
podman run --rm -it -v /srv/app/data:/data myapp
```

If SELinux is enabled, you may need a label option such as `:Z` or `:z` with Podman or Docker on compatible systems:

```bash
podman run --rm -it -v /srv/app/data:/data:Z myapp
```

> Be careful with broad permission fixes such as `chmod 777`. They can hide the real problem and weaken host security.

If the container needs a specific UID/GID, align ownership on the host or run the container with matching IDs when appropriate:

```bash
docker run --rm -it --user 1000:1000 -v /srv/app/data:/data myapp
podman run --rm -it --user 1000:1000 -v /srv/app/data:/data myapp
```

**Environment variables and configuration**

Many containers fail because required configuration is missing or malformed. Environment variables often control database connections, API keys, ports, and runtime modes.

Inspect the environment:

```bash
docker inspect <container_name_or_id> --format '{{json .Config.Env}}'
podman inspect <container_name_or_id> --format '{{json .Config.Env}}'
```

Run with explicit variables:

```bash
docker run --rm -e APP_ENV=production -e DB_HOST=db myapp
podman run --rm -e APP_ENV=production -e DB_HOST=db myapp
```

A `.env` file can simplify repeatable runs:

```bash
docker run --rm --env-file .env myapp
podman run --rm --env-file .env myapp
```

If the application reads a config file, verify the file path inside the container and confirm the file is mounted or baked into the image.

**Resource limits and OOM conditions**

Containers can fail because they run out of memory, CPU, or disk space. The kernel may kill a process if memory pressure becomes too high.

Check for OOM evidence:

```bash
docker inspect <container_name_or_id> --format '{{.State.OOMKilled}}'
podman inspect <container_name_or_id> --format '{{.State.OOMKilled}}'
dmesg | grep -i oom
journalctl -k | grep -i -E 'oom|killed process'
```

If a container is repeatedly killed:

- Increase memory limits if the workload needs more RAM.
- Reduce application memory usage.
- Add swap only if your environment allows it and performance impact is acceptable.
- Review memory leaks in the application.

Example memory limit:

```bash
docker run --memory 512m --memory-swap 1g myapp
podman run --memory 512m --memory-swap 1g myapp
```

**Security controls and host restrictions**

Security layers can block container behavior even when the image and command are correct. On Linux, SELinux and AppArmor are common sources of access denials.

Symptoms include:

- Permission denied errors on mounted files.
- Network access blocked by policy.
- Container startup failures after a host policy change.

Useful checks:

```bash
getenforce
sestatus
aa-status
journalctl -t audit
```

If SELinux is enforcing, audit logs may show denied operations. You should confirm the denial before changing labels or policies.

**Debugging workflow**

A disciplined workflow reduces guesswork and avoids changing too many variables at once.

1. Reproduce the problem with the simplest possible run command.
2. Check `docker ps -a` or `podman ps -a` for state and exit code.
3. Read container logs.
4. Inspect the container configuration and mounts.
5. Test the image interactively with a shell.
6. Verify networking, DNS, and published ports.
7. Check host logs, kernel messages, and security policy denials.
8. Compare the failing container with a known-good container or image version.

A minimal diagnostic run can help isolate the issue:

```bash
docker run --rm -it --entrypoint /bin/sh myapp:latest
podman run --rm -it --entrypoint /bin/sh myapp:latest
```

Inside the shell, inspect the environment and filesystem:

```bash
env
pwd
ls -la
cat /etc/os-release
which myapp
```

**Practical troubleshooting examples**

If a web container exits with code `127`, the entrypoint likely references a missing executable. Check the image’s `ENTRYPOINT` and `CMD`, then verify the binary exists in the image.

If a database container starts but clients cannot connect, confirm the service listens on the correct interface and port, and ensure the port is published to the host.

If a container can read but not write to a mounted directory, inspect ownership and SELinux labels before changing application code.

If a container works on one host but not another, compare:

- Runtime version
- Kernel version
- CPU architecture
- SELinux or AppArmor state
- Available memory and disk space
- Network driver and firewall rules

**Quick reference commands**

```bash
docker ps -a
docker logs <container>
docker inspect <container>
docker exec -it <container> /bin/sh
docker network inspect bridge
docker port <container>
journalctl -u docker
journalctl -k | grep -i oom
```

**Key takeaways**

Container troubleshooting is mostly about narrowing the failure domain. Start with state and logs, then move to image, command, network, storage, permissions, and host policy. The fastest fixes usually come from reading the exit code, checking mounts and environment variables, and verifying that the application listens on the expected interface and port.
