## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Container lifecycle overview**

Container lifecycle management covers the full path from image acquisition to container removal. In practice, you manage both the **image lifecycle** and the **container lifecycle**:

- Images are pulled, tagged, built, updated, and removed.
- Containers are created, started, stopped, restarted, inspected, checkpointed, exported, and deleted.
- Runtime settings such as ports, volumes, environment variables, and resource limits shape how containers behave after launch.

Podman is especially important for Linux+ because it is daemonless and rootless-friendly, but Docker concepts transfer well. The main exam skill is knowing which command changes the image, which command changes the container, and which command only observes state.

**Core concepts**

A container is a running or stopped instance of an image. An image is a read-only template built from layers. The lifecycle is easier to understand if you separate these states:

- **Image**: immutable template used to create containers.
- **Container**: writable runtime instance with its own filesystem layer.
- **Volume**: persistent storage outside the container lifecycle.
- **Network**: connectivity and port mapping used by the container.
- **Process**: the application running inside the container.

A container can be stopped and started again without being recreated, as long as the container object still exists. If you remove the container, its writable layer is gone unless data was stored in a mounted volume or bind mount.

**Image lifecycle management**

Image management is the starting point for container operations. You usually pull a known version, verify it, optionally tag it for local use, and then run or build from it.

- Pull a specific version when you need repeatability.
- Avoid relying on `latest` in production or exam scenarios unless the question explicitly asks for it.
- Tagging creates a local alias for an image reference.
- Building creates a new image from a `Containerfile` or `Dockerfile`.
- Removing dangling images helps reclaim space, but only after you confirm they are not needed.

Common commands:

```bash
podman pull docker.io/library/nginx:1.26
podman images
podman tag docker.io/library/nginx:1.26 local-nginx:stable
podman rmi local-nginx:stable
podman image prune
```

Docker equivalents:

```bash
docker pull nginx:1.26
docker images
docker tag nginx:1.26 local-nginx:stable
docker rmi local-nginx:stable
docker image prune
```

A build uses a file that describes the image layers and metadata:

```bash
podman build -t myapp:1.0 -f Containerfile .
docker build -t myapp:1.0 -f Dockerfile .
```

A simple build file might look like this:

```dockerfile
FROM alpine:3.20
RUN apk add --no-cache curl
COPY app.sh /usr/local/bin/app.sh
RUN chmod +x /usr/local/bin/app.sh
CMD ["/usr/local/bin/app.sh"]
```

**Create vs. run vs. start**

This distinction is a common exam target.

- `podman create` makes a container object from an image but does not start the main process.
- `podman start` launches an existing stopped container.
- `podman run` combines create and start in one step.
- `podman restart` stops and starts the same container again.
- `podman exec` runs an additional process inside a running container.

Example workflow:

```bash
podman create --name web1 -p 8080:80 nginx:1.26
podman ps -a
podman start web1
podman logs web1
```

The same idea in Docker:

```bash
docker create --name web1 -p 8080:80 nginx:1.26
docker ps -a
docker start web1
docker logs web1
```

Use `create` when you want to define the container first and start it later, such as in automation or when you need to inspect settings before launch. Use `run` when you want immediate execution.

**Common lifecycle commands**

These commands control the container state:

```bash
podman ps
podman ps -a
podman stop web1
podman start web1
podman restart web1
podman rm web1
podman inspect web1
podman logs web1
```

State transitions typically look like this:

- **Created**: container exists but is not running.
- **Running**: main process is active.
- **Paused**: processes are temporarily suspended.
- **Stopped**: process exited or was stopped.
- **Removed**: container metadata and writable layer are deleted.

Useful operational notes:

- `podman stop` sends a graceful termination signal first, then forcefully kills the container if it does not exit within the timeout.
- `podman kill` sends a signal immediately; by default that signal is `SIGKILL`, but you can choose another signal with `--signal`.
- `podman rm` removes a stopped container; use `-f` only when you intentionally want to stop and remove in one action.

Examples:

```bash
podman stop --time 10 web1
podman kill --signal SIGTERM web1
podman rm -f web1
```

**Runtime options that affect lifecycle**

Lifecycle management is not just about starting and stopping. You also need to control how the container behaves while it runs.

- **Ports**: map host ports to container ports with `-p`.
- **Volumes**: persist data with named volumes or bind mounts.
- **Environment variables**: pass configuration with `-e` or `--env-file`.
- **Resource limits**: constrain CPU and memory usage.
- **Restart policy**: define what happens after failure or reboot.
- **User and security settings**: run as a non-root user, adjust SELinux labels, or drop capabilities.

Example with common runtime flags:

```bash
podman run -d \
  --name app1 \
  -p 8080:80 \
  -e APP_ENV=prod \
  --env-file ./app.env \
  --memory 512m \
  --cpus 1.5 \
  --restart=always \
  nginx:1.26
```

Docker equivalent:

```bash
docker run -d \
  --name app1 \
  -p 8080:80 \
  -e APP_ENV=prod \
  --env-file ./app.env \
  --memory 512m \
  --cpus 1.5 \
  --restart=always \
  nginx:1.26
```

An environment file keeps configuration out of the command line:

```bash
APP_ENV=prod
DB_HOST=db01
DB_PORT=5432
```

Then use it with:

```bash
podman run --env-file ./app.env myapp:1.0
```

**Restart policies and auto-restart behavior**

Restart policies help containers recover from crashes or host reboots. They are especially important for services that should stay available.

- `no` means do not restart automatically.
- `on-failure` restarts only when the process exits with a non-zero code.
- `always` restarts regardless of exit reason.
- `unless-stopped` restarts unless you explicitly stop the container.

Examples:

```bash
podman run -d --restart=on-failure:3 myapp:1.0
podman update --restart=always web1
podman inspect web1 --format '{{.HostConfig.RestartPolicy.Name}}'
```

If a container exits with code `0`, that often means the process completed successfully. A non-zero exit code usually indicates an error, but you should interpret it in context. For example, a one-shot job may exit `0` by design.

**Inspecting containers and troubleshooting state**

Inspection tells you what the container is configured to do and what happened to it.

```bash
podman inspect web1
podman inspect --format '{{.State.Status}}' web1
podman inspect --format '{{.State.ExitCode}}' web1
podman inspect --format '{{.Config.Env}}' web1
```

Useful troubleshooting commands:

```bash
podman logs web1
podman logs -f web1
podman top web1
podman stats
podman port web1
podman exec -it web1 sh
```

Common failure modes:

- **Port conflict**: another process already uses the host port.
- **Permission issue**: the container cannot read a mounted file or directory.
- **SELinux denial**: a bind mount is blocked by labeling rules.
- **Bad image tag**: the requested image version does not exist.
- **Crash loop**: the container starts and exits repeatedly because the main process fails.

Examples:

```bash
ss -tulpn | grep ':8080'
podman run -d -p 8080:80 nginx:1.26
```

If port `8080` is already in use, choose another host port or stop the conflicting service.

For bind mounts on SELinux systems, you often need a label option:

```bash
podman run -d \
  -v /srv/web:/usr/share/nginx/html:Z \
  -p 8080:80 \
  nginx:1.26
```

The `:Z` option gives the mount a private SELinux label for one container. Use `:z` when multiple containers share the same content and the labeling model supports it.

**Resource management**

Resource limits prevent one container from consuming too much of the host.

- `--memory` limits RAM usage.
- `--cpus` limits CPU share.
- `--cpuset-cpus` pins a container to specific cores.
- `--memory-swap` controls swap behavior where supported.
- `--pids-limit` limits process creation.

Examples:

```bash
podman run -d --memory 256m --cpus 0.5 --pids-limit 100 myapp:1.0
podman update --memory 512m --cpus 1.0 web1
```

Use limits when you run multiple services on the same host, when you want predictable performance, or when you need to prevent runaway workloads.

**Volumes, bind mounts, and persistence**

Container filesystems are temporary. If you need data to survive container removal, store it outside the container layer.

- **Named volumes** are managed by the container engine.
- **Bind mounts** map a host path into the container.
- **Tmpfs mounts** store data in memory and disappear on stop or reboot.

Examples:

```bash
podman volume create dbdata
podman run -d --name db -v dbdata:/var/lib/postgresql/data postgres:16
podman volume ls
podman volume inspect dbdata
```

Bind mount example:

```bash
podman run -d \
  -v /srv/app/config:/etc/myapp:ro \
  myapp:1.0
```

> Do not store important data only in a container writable layer. If you remove the container, that data is lost.

Volume cleanup is safer when you verify what is still referenced. `podman volume prune` removes only volumes not referenced by any container, so the risk is deleting unreferenced data that you still intended to keep.

**Checkpoint, restore, export, and import**

These features are less common in daily administration but can appear in lifecycle questions.

- **Checkpoint/restore** saves and resumes a container’s process state, typically with CRIU (Checkpoint/Restore In Userspace), where supported.
- **Export/import** moves filesystem contents, not full runtime metadata.
- **Save/load** works with images, not containers.

Examples:

```bash
podman checkpoint web1
podman restore web1
podman export web1 -o web1.tar
podman import web1.tar web1-imported:latest
podman save -o nginx.tar nginx:1.26
podman load -i nginx.tar
```

Use export/import when you need a filesystem snapshot. Use save/load when you need to move an image between systems. Do not confuse either with a full backup of application state unless the data is stored in volumes and included separately.

**Practical lifecycle workflow**

A typical service workflow might look like this:

1. Pull the image you want.
2. Create or run the container with the required ports, volumes, and environment.
3. Verify the container is running.
4. Inspect logs and state if the service fails.
5. Update the container or image when needed.
6. Stop and remove the container when it is no longer required.

Example:

```bash
podman pull docker.io/library/httpd:2.4
podman run -d --name web2 -p 8081:80 docker.io/library/httpd:2.4
podman ps
podman logs web2
podman stop web2
podman rm web2
```

**Troubleshooting examples**

A container exits immediately:

```bash
podman ps -a
podman logs app1
podman inspect --format '{{.State.ExitCode}}' app1
```

If the exit code is non-zero, check the application command, missing files, or permissions.

A bind mount fails on SELinux systems:

```bash
podman run -d -v /srv/data:/data:Z alpine sleep 3600
```

If the container still cannot access the mount, verify host permissions and SELinux context with:

```bash
ls -ldZ /srv/data
```

A port is already in use:

```bash
ss -tulpn | grep ':80'
podman run -d -p 8082:80 nginx:1.26
```

A container needs an interactive shell for debugging:

```bash
podman exec -it web1 sh
docker exec -it web1 sh
```

**Podman and Docker transfer points**

Podman and Docker share many familiar commands, but Podman is often preferred on modern Linux systems because it supports rootless operation and integrates well with systemd.

- `podman run` and `docker run` behave similarly for most basic tasks.
- `podman create`, `start`, `stop`, `rm`, `inspect`, and `logs` map closely to Docker equivalents.
- Podman can generate systemd units for containers in some workflows, which helps with service management.
- Docker commonly uses a daemon model, while Podman does not require a central daemon for normal operation.

Quick comparison:

| Task | Podman | Docker |
|---|---|---|
| Create container | `podman create` | `docker create` |
| Start container | `podman start` | `docker start` |
| Run container | `podman run` | `docker run` |
| Inspect state | `podman inspect` | `docker inspect` |
| View logs | `podman logs` | `docker logs` |
| Remove container | `podman rm` | `docker rm` |

**Exam focus reminders**

- Know the difference between an image and a container.
- Know that `create` does not start the container.
- Know that `run` creates and starts in one step.
- Know how restart policies affect availability.
- Know how to persist data with volumes or bind mounts.
- Know how to interpret exit codes and logs.
- Know how to troubleshoot ports, permissions, and SELinux issues.
- Know the difference between image cleanup and container cleanup.
