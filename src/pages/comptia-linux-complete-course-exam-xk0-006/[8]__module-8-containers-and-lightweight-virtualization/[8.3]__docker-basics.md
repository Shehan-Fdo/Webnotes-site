## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Docker overview**

Docker is a container platform that packages an application and its runtime dependencies into a portable unit. On Linux, Docker uses kernel features such as namespaces and cgroups to isolate processes and control resource usage. The result is a lightweight alternative to full virtual machines when you want consistent deployment across development, test, and production systems.

At a high level, you work with:
- **Images** as read-only templates.
- **Containers** as running instances of images.
- **Registries** as places to store and distribute images.
- **Volumes** and **bind mounts** for persistent or shared data.

Docker is useful when you need repeatable environments, fast startup times, and simple packaging for services. It is not a replacement for every virtualization use case; it is best when the workload can share the host kernel.

**Core concepts**

A Docker image is built from layers. Each instruction in a Dockerfile typically creates a new layer, which helps with caching and reuse. A container adds a writable layer on top of the image, so changes inside the container do not modify the original image.

- **Image**: Immutable template used to create containers.
- **Container**: Isolated process running from an image.
- **Layer**: Filesystem change that can be cached and reused.
- **Registry**: Remote repository for images, such as Docker Hub or a private registry.
- **Tag**: Human-readable label for an image version, such as `nginx:1.27`.
- **Digest**: Content-addressed identifier that uniquely identifies an image.

A useful mental model is:
- Build an image from a Dockerfile.
- Push the image to a registry.
- Pull the image on another host.
- Run one or more containers from that image.

**Docker architecture**

Docker uses a client-server model:
- The **Docker client** sends commands such as `docker run` or `docker build`.
- The **Docker daemon** (`dockerd`) performs image builds, container lifecycle actions, networking, and storage management.
- The **container runtime** handles low-level container execution. Modern Docker installations commonly use `containerd` and `runc` under the hood.

Typical communication flow:
- You run `docker ps` on the client.
- The client talks to the daemon through the Unix socket, usually `/var/run/docker.sock`.
- The daemon returns container state and metadata.

> Be careful with access to `/var/run/docker.sock`. Anyone who can control the Docker daemon effectively has root-level control over the host in many environments.

**Installing and verifying Docker**

Package names vary by distribution, but the workflow is similar: install the engine, start the service, and verify the daemon is running.

Common verification commands:
```bash
docker --version
docker info
systemctl status docker
```

If Docker is installed correctly, `docker info` shows server details, storage driver, and available resources. If the daemon is not running, start it with:
```bash
sudo systemctl enable --now docker
```

On many Linux systems, you can add your user to the `docker` group to avoid using `sudo` for every command:
```bash
sudo usermod -aG docker $USER
newgrp docker
```

> Membership in the `docker` group is highly privileged. Treat it as administrative access.

**Working with images**

Images are the starting point for containers. You can download them from a registry, inspect them, tag them, and remove them when no longer needed.

Common image commands:
```bash
docker pull nginx:latest
docker images
docker inspect nginx:latest
docker tag nginx:latest myrepo/nginx:web
docker rmi nginx:latest
```

Practical notes:
- `docker pull` downloads an image from a registry.
- `docker images` lists local images.
- `docker inspect` shows detailed metadata in JSON format.
- `docker tag` creates another name for the same image ID.
- `docker rmi` removes an image if no container depends on it.

Use explicit tags when possible. `latest` is only a tag name, not a guarantee of newest or safest version.

**Running containers**

The most common command is `docker run`. It creates a container from an image, starts it, and optionally attaches your terminal.

Examples:
```bash
docker run hello-world
docker run -it ubuntu:24.04 bash
docker run -d --name web -p 8080:80 nginx:1.27
```

Key options:
- `-it` gives you an interactive terminal.
- `-d` runs the container in detached mode.
- `--name` assigns a readable container name.
- `-p host:container` publishes a port from the container to the host.

Container lifecycle commands:
```bash
docker ps
docker ps -a
docker start web
docker stop web
docker restart web
docker rm web
```

Use `docker ps` to view running containers and `docker ps -a` to include stopped ones. Remove containers when they are no longer needed; stopping a container does not delete it.

**Container networking**

Docker provides built-in networking so containers can communicate with each other and with the host. The default network drivers you should know are:
- **bridge**: Default for standalone containers on a single host.
- **host**: Shares the host network stack.
- **none**: Disables networking.
- **overlay**: Used for multi-host networking in orchestrated environments.

Inspect networks with:
```bash
docker network ls
docker network inspect bridge
```

Create and use a custom bridge network:
```bash
docker network create app-net
docker run -d --name db --network app-net postgres:16
docker run -d --name api --network app-net myapi:1.0
```

Custom networks are useful because they provide automatic DNS-based service discovery by container name. That makes it easier for one container to reach another without hardcoding IP addresses.

Port publishing is separate from container-to-container networking:
- `-p 8080:80` exposes container port `80` on host port `8080`.
- `-P` publishes all exposed ports to random host ports.

**Storage: volumes and bind mounts**

Containers are ephemeral by design. If you write data only inside the container filesystem, you lose it when the container is removed. Use persistent storage when the data matters.

Two common approaches:
- **Volumes**: Managed by Docker and stored in Docker’s data directory.
- **Bind mounts**: Map a host path directly into the container.

Examples:
```bash
docker volume create appdata
docker run -d --name db -v appdata:/var/lib/postgresql/data postgres:16
docker run -d --name web -v /srv/site:/usr/share/nginx/html:ro nginx:1.27
```

Use volumes when you want Docker-managed persistence and easier portability. Use bind mounts when you need direct access to host files, such as source code during development.

Useful commands:
```bash
docker volume ls
docker volume inspect appdata
docker volume rm appdata
```

**Dockerfile basics**

A Dockerfile defines how to build an image. It is the standard way to create repeatable images for applications.

Example Dockerfile:
```dockerfile
FROM ubuntu:24.04

RUN apt-get update && apt-get install -y curl ca-certificates \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /app
COPY . /app
EXPOSE 8080
CMD ["python3", "-m", "http.server", "8080"]
```

Common instructions:
- `FROM` sets the base image.
- `RUN` executes commands during build.
- `WORKDIR` sets the working directory.
- `COPY` copies files into the image.
- `EXPOSE` documents the intended port.
- `CMD` defines the default command.
- `ENTRYPOINT` defines the executable that always runs.

Build and run:
```bash
docker build -t myapp:1.0 .
docker run -d --name myapp -p 8080:8080 myapp:1.0
```

Practical build tips:
- Combine related `RUN` commands to reduce layers.
- Clean package caches in the same layer to keep images smaller.
- Prefer pinned base image tags for reproducibility.
- Use `.dockerignore` to exclude unnecessary files from the build context.

Example `.dockerignore`:
```text
.git
node_modules
*.log
```

**Inspecting and troubleshooting**

Docker provides inspection tools that help you understand what is running and why something failed.

Useful commands:
```bash
docker logs web
docker exec -it web sh
docker inspect web
docker stats
docker top web
```

What each command helps with:
- `docker logs` shows container output.
- `docker exec` opens a shell or runs a command inside a running container.
- `docker inspect` reveals configuration, mounts, network settings, and state.
- `docker stats` shows live CPU, memory, network, and I/O usage.
- `docker top` shows processes inside the container.

A typical troubleshooting flow:
1. Check whether the container is running with `docker ps -a`.
2. Read logs with `docker logs <name>`.
3. Inspect configuration with `docker inspect <name>`.
4. Enter the container with `docker exec -it <name> sh`.
5. Verify ports, environment variables, and mounted paths.

**Environment variables and configuration**

Containers often need runtime configuration. Use environment variables for values that change between environments.

Examples:
```bash
docker run -d --name app \
  -e APP_ENV=production \
  -e DB_HOST=db \
  myapp:1.0
```

You can also use an env file:
```bash
docker run --env-file .env myapp:1.0
```

This is useful when you want to keep configuration separate from the image. Do not bake secrets into images unless you fully understand the security implications and lifecycle.

**Security considerations**

Docker improves isolation, but containers are not the same as virtual machines. They share the host kernel, so kernel vulnerabilities can affect container security.

Important practices:
- Run containers as a non-root user when possible.
- Use minimal base images to reduce attack surface.
- Keep images updated and rebuild regularly.
- Limit capabilities when the application does not need them.
- Avoid mounting the Docker socket into containers unless absolutely necessary.
- Treat secrets carefully; prefer secret management tools in production platforms.

Example of dropping capabilities:
```bash
docker run --cap-drop ALL --cap-add NET_BIND_SERVICE nginx:1.27
```

Example of running as a non-root user in a Dockerfile:
```dockerfile
FROM alpine:3.20
RUN adduser -D appuser
USER appuser
CMD ["sh"]
```

**Common exam-relevant distinctions**

Docker concepts are often tested by comparing similar terms:
- **Image vs container**: An image is the template; a container is the running instance.
- **Volume vs bind mount**: A volume is Docker-managed; a bind mount maps a host path.
- **`CMD` vs `ENTRYPOINT`**: `CMD` provides defaults; `ENTRYPOINT` defines the main executable.
- **`EXPOSE` vs `-p`**: `EXPOSE` documents intent; `-p` publishes the port.
- **`docker stop` vs `docker rm`**: Stop halts a container; remove deletes it.

**Practical workflow example**

A simple workflow for a web service might look like this:
1. Pull a base image or write a Dockerfile.
2. Build the image with `docker build`.
3. Run the container with port mapping and a volume.
4. Check logs and inspect the container if it fails.
5. Push the image to a registry when it is ready for deployment.

Example:
```bash
docker build -t webapp:1.0 .
docker run -d --name webapp \
  -p 8080:8080 \
  -v webdata:/var/lib/webapp \
  webapp:1.0
docker logs -f webapp
```

**Quick reference**

| Task | Command |
|---|---|
| List images | `docker images` |
| List running containers | `docker ps` |
| List all containers | `docker ps -a` |
| Pull an image | `docker pull IMAGE:TAG` |
| Run a container | `docker run IMAGE:TAG` |
| Stop a container | `docker stop NAME` |
| Remove a container | `docker rm NAME` |
| View logs | `docker logs NAME` |
| Inspect details | `docker inspect NAME` |

**Key takeaways**

Docker packages applications into portable images and runs them as isolated containers. For Linux+ exam purposes, focus on the image/container relationship, basic lifecycle commands, networking and storage options, and the security implications of daemon access and privileged containers. If you can explain how to build, run, inspect, and persist a containerized service, you understand the core of Docker basics.
