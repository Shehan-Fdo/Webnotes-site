## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Core distinction**

A **container image** is a read-only template that packages an application and its dependencies. A **container** is a running instance of that image with a writable layer added at start time. In practice, you build or pull images, then create and run containers from them.

This distinction matters because images are portable artifacts you store, tag, scan, and distribute, while containers are ephemeral runtime processes you start, stop, inspect, and remove. If you understand that split, most container workflows become easier to reason about.

- **Image**: immutable content used as the launch blueprint.
- **Container**: runtime object created from an image.
- **Layering**: images are usually composed of stacked filesystem layers.
- **Writable layer**: container-specific changes live here until the container is removed.

**How images work**

Images are typically built from a `Dockerfile` or similar build definition. Each instruction creates a layer, and layers are cached to speed up rebuilds. A tag such as `nginx:1.27` points to a specific image version, but tags are mutable labels, not guarantees of identity.

A useful mental model is:

- The image contains the application, libraries, default configuration, and metadata.
- The container adds runtime state such as process IDs, network interfaces, and temporary file changes.
- Multiple containers can run from the same image at the same time.
- Containers do not modify the base image; they write to their own layer.

Common image metadata includes:

- **Entrypoint**: the default executable started when the container launches.
- **Cmd**: default arguments or fallback command.
- **Environment variables**: runtime defaults such as `APP_ENV=production`.
- **Exposed ports**: documentation of intended network ports.
- **Volumes**: declared mount points for persistent or shared data.

**How containers work**

A container is not a full virtual machine. It shares the host kernel and uses Linux kernel features such as namespaces and cgroups to isolate processes and limit resources. That is why containers start quickly and use less overhead than traditional VMs.

Key isolation mechanisms include:

- **Namespaces**: isolate process IDs, mounts, networking, users, and other views of the system.
- **cgroups**: control CPU, memory, I/O, and other resource usage.
- **Union filesystem**: combines image layers with the container’s writable layer.

Because the container shares the host kernel, kernel compatibility matters. You cannot run a Linux container on a host without a compatible Linux kernel, although higher-level tooling may abstract some of that complexity.

**Image lifecycle vs container lifecycle**

Images and containers have different lifecycles, and exam questions often test that difference.

| Object | Typical actions | Persistence |
|---|---|---|
| Image | build, pull, tag, scan, push, remove | Stored until deleted |
| Container | create, start, stop, restart, inspect, exec, remove | Runtime state is temporary |

A container can be stopped and started again, but any changes made inside the writable layer disappear when the container is removed unless the data is stored in a volume or bind mount.

> If you need data to survive container removal, do not store it only inside the container filesystem.

**Practical CLI workflow**

The exact commands vary by runtime, but the workflow is similar across tools such as Docker, Podman, and containerd-based front ends.

Build an image:

```bash
docker build -t webapp:1.0 .
```

List local images:

```bash
docker images
```

Run a container from an image:

```bash
docker run -d --name webapp01 -p 8080:80 webapp:1.0
```

Inspect running containers:

```bash
docker ps
docker inspect webapp01
```

View logs:

```bash
docker logs webapp01
```

Enter a running container for troubleshooting:

```bash
docker exec -it webapp01 /bin/sh
```

Stop and remove a container:

```bash
docker stop webapp01
docker rm webapp01
```

Remove an unused image:

```bash
docker rmi webapp:1.0
```

**Dockerfile example**

A `Dockerfile` defines how an image is assembled. Each instruction should be intentional because it affects caching, size, and reproducibility.

```dockerfile
FROM alpine:3.20

RUN apk add --no-cache nginx

COPY index.html /usr/share/nginx/html/index.html

EXPOSE 80

CMD ["nginx", "-g", "daemon off;"]
```

Why this matters:

- `FROM` selects the base image.
- `RUN` adds packages or performs build steps.
- `COPY` places files into the image.
- `EXPOSE` documents the intended port.
- `CMD` defines the default process.

**Image tags, digests, and reproducibility**

Tags are convenient for humans, but digests are better for exact identity. A digest is a cryptographic hash that identifies the image content precisely.

- `alpine:3.20` is a tag reference.
- `alpine@sha256:...` is a digest reference.
- Use digests when you need repeatable deployments.
- Use tags when you want readable versioning during development.

Example:

```bash
docker pull alpine@sha256:exampledigest
```

In real environments, pinning by digest reduces the risk of pulling a different image later if a tag is retargeted.

**Containers and persistence**

Containers are designed to be disposable. That is a feature, not a limitation. You should keep application state outside the container when the data must survive restarts or replacement.

Common persistence patterns:

- **Named volumes** for managed persistent storage.
- **Bind mounts** for host-path sharing during development or specific operational needs.
- **External services** for databases, object storage, or queues.

Example with a volume:

```bash
docker volume create appdata
docker run -d --name db01 -v appdata:/var/lib/postgresql/data postgres:16
```

Example with a bind mount:

```bash
docker run -d --name devweb -v "$PWD":/usr/share/nginx/html:ro -p 8080:80 nginx:stable
```

**Networking basics**

Containers usually get their own network namespace and virtual interface. The runtime connects them to a bridge, overlay, or host network depending on configuration.

- **Bridge network**: common default for single-host container networking.
- **Host network**: container shares the host network stack.
- **Overlay network**: used for multi-host container communication in orchestrated environments.

Port publishing maps a host port to a container port:

```bash
docker run -d --name api01 -p 9000:8080 myapi:latest
```

This means traffic to host port `9000` is forwarded to container port `8080`.

**Resource control and isolation**

Containers can be constrained so one workload does not consume all host resources. This is especially important on shared systems and in production.

Examples:

```bash
docker run -d --name limitedapp --memory 256m --cpus 0.5 myapp:latest
```

Resource controls help you:

- Prevent noisy-neighbor problems.
- Match container usage to host capacity.
- Test application behavior under constrained conditions.

**Security considerations**

Images and containers both introduce security responsibilities.

- **Minimize image size** to reduce attack surface.
- **Use trusted base images** from reputable sources.
- **Scan images** for known vulnerabilities before deployment.
- **Run as non-root** when possible.
- **Avoid baking secrets into images** because image layers are easy to inspect.
- **Use read-only filesystems** where practical.
- **Drop unnecessary capabilities** to reduce privilege.

Example of running as a non-root user in a Dockerfile:

```dockerfile
FROM alpine:3.20
RUN adduser -D appuser
USER appuser
CMD ["sh"]
```

If you need secrets, prefer runtime injection mechanisms such as environment variables from a secret store, mounted secret files, or orchestrator-managed secrets rather than hardcoding them into the image.

**Inspection and troubleshooting**

When a container does not behave as expected, inspect both the image and the runtime state.

Useful commands:

```bash
docker inspect nginx:stable
docker inspect webapp01
docker logs --tail 50 webapp01
docker exec -it webapp01 sh
```

What to check:

- **Image config**: entrypoint, command, environment, exposed ports.
- **Container state**: running, exited, restart count, exit code.
- **Mounts**: verify volumes and bind mounts are attached correctly.
- **Network settings**: confirm port mappings and IP assignment.
- **Logs**: look for application startup failures or permission errors.

A common failure pattern is a container that exits immediately because the main process ends. In containers, the foreground process is the container’s lifecycle anchor.

**Image management best practices**

Treat images as versioned artifacts.

- Tag images with meaningful versions such as `1.4.2` or a build number.
- Keep a `latest` tag only as a convenience, not as a deployment target.
- Rebuild images when base images receive security updates.
- Remove unused images to save disk space.
- Use multi-stage builds to keep runtime images small.

Example multi-stage build:

```dockerfile
FROM golang:1.23 AS build
WORKDIR /src
COPY . .
RUN go build -o app .

FROM alpine:3.20
COPY --from=build /src/app /usr/local/bin/app
CMD ["app"]
```

This pattern keeps build tools out of the final image, which reduces size and attack surface.

**Container runtime concepts**

Different tools may use different terminology, but the concepts are similar.

- **Image store**: local repository of downloaded or built images.
- **Container runtime**: component that starts and manages containers.
- **OCI**: Open Container Initiative, a set of standards for image and runtime formats.
- **Registry**: remote server that stores and distributes images.

Typical registry workflow:

```bash
docker tag webapp:1.0 registry.example.com/team/webapp:1.0
docker push registry.example.com/team/webapp:1.0
docker pull registry.example.com/team/webapp:1.0
```

**Exam focus**

For Linux+ exam purposes, be ready to distinguish these ideas quickly:

- An **image** is the template; a **container** is the running instance.
- Images are immutable; containers add a writable layer.
- Containers share the host kernel.
- Persistent data belongs in volumes or mounts, not only in the container filesystem.
- Tags are convenient, but digests are more precise.
- `build`, `pull`, and `push` are image operations; `run`, `stop`, `exec`, and `logs` are container operations.

**Quick comparison**

| Feature | Image | Container |
|---|---|---|
| State | Static template | Running instance |
| Mutability | Read-only | Writable layer on top |
| Purpose | Distribution and reuse | Execution |
| Persistence | Stored artifact | Temporary runtime object |
| Typical commands | `build`, `pull`, `tag`, `push` | `run`, `stop`, `exec`, `logs`, `rm` |

**Practical mental model**

Think of an image as a packaged recipe and a container as the meal prepared from that recipe. You can make many meals from the same recipe, and each meal can be customized while it is being served. When the meal is gone, the recipe remains.

That model helps you answer most image-versus-container questions correctly and choose the right storage, security, and deployment approach in real Linux environments.
