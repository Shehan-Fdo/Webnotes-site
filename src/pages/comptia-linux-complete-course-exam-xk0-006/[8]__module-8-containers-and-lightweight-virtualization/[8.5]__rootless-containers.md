**Rootless container fundamentals**

Rootless containers run without a container engine process or container payload having full root privileges on the host. Instead, the runtime maps an unprivileged user into a private namespace so the container can behave like it has root inside the container while remaining constrained outside it.

This model reduces the blast radius of a compromise. If an attacker escapes a container, they usually land as an unprivileged host user rather than `root`. That makes rootless mode especially useful on shared systems, developer workstations, CI runners, and environments where you want stronger default isolation without giving users administrative access.

- **User namespaces** map container UID/GID values to host UID/GID ranges.
- **Least privilege** is the main security benefit.
- **Compatibility tradeoffs** still exist for networking, storage, and privileged ports.

**How rootless containers work**

A rootless runtime relies on kernel features that let an unprivileged user create isolated namespaces. The most important are:

- **User namespace**: remaps identities so container `root` is not host `root`.
- **Mount namespace**: isolates filesystem mounts.
- **Network namespace**: isolates interfaces, routes, and ports.
- **PID namespace**: isolates process IDs so the container sees its own process tree.

The host must allow user namespace creation. On many distributions, this is enabled by default, but some hardened systems restrict it. If user namespaces are disabled or limited by policy, rootless containers may not start.

A typical mapping uses subordinate ID ranges from `/etc/subuid` and `/etc/subgid`:

```text
alice:100000:65536
alice:165536:65536
```

This means the user `alice` can map container IDs into host-owned subordinate ranges. The runtime uses these ranges to translate ownership for files created inside the container.

**Why rootless matters**

Rootless mode is not just a convenience feature. It changes the security model.

- **No direct host root access** is required to run containers.
- **Reduced risk from daemon compromise** because the engine runs as the user, not as a system-wide privileged service.
- **Safer multi-user systems** where developers need containers but should not get `sudo`.
- **Better alignment with least privilege** and defense-in-depth practices.

Rootless mode is not a replacement for all security controls. A malicious container can still consume CPU, memory, disk, and network resources within the user’s permissions. You still need image trust, patching, and resource controls.

**Podman and Docker in rootless mode**

Podman is commonly associated with rootless operation because it can run containers without a long-lived privileged daemon. Docker also supports rootless mode, but it still uses a `dockerd` daemon running as the unprivileged user.

| Feature | Podman rootless | Docker rootless |
|---|---|---|
| Daemon model | Daemonless CLI with per-container processes | User-scoped `dockerd` daemon |
| Default fit | Strong for rootless and systemd integration | Supported, but more setup-oriented |
| Privilege model | Runs as the invoking user | Runs as the invoking user |
| Typical use | Developer and server workloads | Developer environments and compatible workflows |

For exam purposes, remember the key distinction: **Podman is daemonless by design**, while **Docker rootless still uses a daemon**, just not a root-owned one.

**User namespaces and ID mapping**

User namespaces let a process appear as `root` inside the container while remaining unprivileged on the host. The runtime needs a mapping between container IDs and host subordinate IDs.

Common tools and files:

- `/etc/subuid` and `/etc/subgid` define subordinate ranges.
- `newuidmap` and `newgidmap` help set up mappings.
- `unshare` can demonstrate namespace behavior.
- `podman unshare` opens a shell with the container user mapping context.

Example inspection:

```bash
grep "^alice:" /etc/subuid /etc/subgid
```

Example namespace test:

```bash
unshare -Ur bash
id
touch /tmp/testfile
```

Inside the namespace, `id` may show `uid=0(root)`, but that identity is only valid within the namespace. On the host, the process still belongs to the unprivileged user.

**Storage behavior in rootless mode**

Rootless containers cannot freely use host paths the way rootful containers can. File ownership and filesystem features matter more because the container process does not have unrestricted permission to chown arbitrary files on the host.

Common storage approaches:

- **Rootless overlay storage**: often uses `fuse-overlayfs` when kernel overlay support is not available for unprivileged users.
- **Per-user storage locations**: container data usually lives under the user’s home directory or XDG paths.
- **Bind mounts**: work only for paths the user can already access.

Typical Podman storage locations:

```bash
~/.local/share/containers/storage
~/.config/containers
```

A practical bind mount example:

```bash
podman run --rm -it \
  -v "$HOME/projects:/work:Z" \
  alpine sh
```

Notes on storage:

- **`fuse-overlayfs`** is common in rootless setups because it avoids requiring privileged overlay mounts.
- **SELinux labels** may matter on SELinux-enabled systems; `:Z` or `:z` can relabel content for container access.
- **Ownership mismatches** can cause confusing permission errors when host files are not mapped into the container’s subordinate ID range.

> Rootless containers do not magically bypass filesystem permissions. If your user cannot read or write a host path, the container usually cannot either.

**Networking in rootless mode**

Networking is one of the biggest differences between rootless and rootful containers. Because unprivileged users cannot normally create raw network devices or bind low ports directly, rootless engines use user-space networking helpers.

Common approaches include:

- **slirp4netns**: user-mode networking for rootless containers.
- **pasta**: another user-space networking option used by some runtimes.
- **Port forwarding**: the runtime forwards host ports to the container namespace.

Example Podman port mapping:

```bash
podman run --rm -p 8080:80 nginx
```

In rootless mode, binding to ports below `1024` is often restricted because those are privileged ports on the host. Some systems can change the host-wide sysctl `net.ipv4.ip_unprivileged_port_start`, but that is a host policy decision and may be unavailable or intentionally locked down.

Example of checking the current setting:

```bash
sysctl net.ipv4.ip_unprivileged_port_start
```

If you need a low port in a rootless environment, common options are:

- Use a higher host port such as `8080`.
- Put a reverse proxy or load balancer in front of the container.
- Adjust host policy only if you control the system and understand the security impact.

**Cgroups and resource limits**

Control groups, or **cgroups**, let the kernel enforce resource limits such as CPU, memory, and I/O. Rootless containers can use cgroups only when the host supports delegation properly, and modern systems prefer **cgroup v2** for this model.

What to expect:

- **Delegated cgroup controllers** allow a user to manage a subtree safely.
- **Without proper delegation**, some limits may fail to apply.
- **Resource enforcement** is not guaranteed in every rootless setup.

Example of checking cgroup version:

```bash
stat -fc %T /sys/fs/cgroup
```

If the output is `cgroup2fs`, the host uses cgroup v2.

Example resource limit with Podman:

```bash
podman run --rm --memory=256m --cpus=1 alpine sh
```

If the host does not delegate the needed controllers, the runtime may report an error or ignore the requested limit depending on the engine and configuration. Do not assume a silent fallback.

**Practical Podman rootless workflow**

A typical rootless Podman workflow looks like this:

1. Confirm subordinate ID ranges exist for your user.
2. Verify user namespaces are allowed.
3. Pull or build an image as your user.
4. Run the container with user-space networking and per-user storage.
5. Expose only the ports you need.

Example:

```bash
podman pull docker.io/library/alpine:latest
podman run --rm -it alpine sh
```

Example with a web service:

```bash
podman run -d --name web -p 8080:80 nginx
podman ps
podman logs web
```

Useful inspection commands:

```bash
podman info
podman inspect web
podman top web
podman stats
```

If a container fails to start, check:

- Whether user namespaces are enabled.
- Whether subordinate UID/GID ranges exist.
- Whether the image storage backend is available.
- Whether the requested port or mount is allowed for the user.

**Practical Docker rootless workflow**

Docker rootless mode requires setup for the unprivileged user and starts a user-scoped `dockerd` process. The daemon listens on a user-owned socket rather than the system socket.

Typical setup commands:

```bash
dockerd-rootless-setuptool.sh install
systemctl --user enable docker
systemctl --user start docker
```

Client access often uses a user-specific socket:

```bash
export DOCKER_HOST=unix:///run/user/1000/docker.sock
docker info
docker run --rm hello-world
```

Important points:

- **`dockerd` still exists** in rootless Docker.
- **The daemon runs as your user**, not as root.
- **Environment variables** such as `DOCKER_HOST` may be needed depending on setup.

**Troubleshooting rootless containers**

Rootless issues usually fall into a few categories: identity mapping, storage, networking, and cgroups.

Common checks:

```bash
id
grep "^$USER:" /etc/subuid /etc/subgid
sysctl kernel.unprivileged_userns_clone
podman info --debug
```

Typical symptoms and causes:

- **Permission denied on bind mounts**: the host path is not accessible to your user or the label is wrong on SELinux systems.
- **Port bind failure on low ports**: the port is privileged or blocked by host policy.
- **Container start failure**: user namespaces or subordinate IDs are missing.
- **Resource limits not applied**: cgroup delegation is incomplete.

Useful diagnostics for Podman:

```bash
podman --log-level=debug run --rm alpine true
```

Useful diagnostics for Docker rootless:

```bash
docker info
journalctl --user -u docker
```

For Docker rootless, the user service is the relevant place to inspect logs because the daemon runs under the user session. For Podman, there is usually no persistent user daemon to query in the same way.

**Security considerations**

Rootless containers improve security, but they do not eliminate risk.

- **Image trust** still matters because malicious code can run with your user’s permissions.
- **Kernel vulnerabilities** can still affect the host if the kernel is exploitable.
- **Shared resources** can still be abused for denial of service within user limits.
- **Secrets handling** remains important because mounted files and environment variables can leak data.

Best practices:

- Run containers as non-root inside the container when possible.
- Use minimal images.
- Limit capabilities where supported.
- Avoid mounting sensitive host directories.
- Keep the host kernel and runtime updated.
- Prefer rootless mode for developer and multi-user environments when feasible.

**Exam focus summary**

For Linux+ exam purposes, focus on these ideas:

- **Rootless containers** run without host root privileges.
- **User namespaces** are the core enabling technology.
- **Subordinate UID/GID ranges** support identity mapping.
- **Podman** is naturally aligned with rootless, daemonless operation.
- **Docker rootless** still uses a user-scoped daemon.
- **Networking and low ports** are common friction points.
- **cgroup v2** is the preferred modern foundation for rootless resource control.
