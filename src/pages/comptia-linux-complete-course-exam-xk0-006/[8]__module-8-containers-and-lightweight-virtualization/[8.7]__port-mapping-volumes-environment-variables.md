## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Container configuration fundamentals**

Containers are designed to be disposable, but the applications inside them still need network access, persistent storage, and runtime configuration. Port mapping, volumes, and environment variables solve those three problems:

- **Port mapping** exposes a container service to the host or external network.
- **Volumes** preserve data beyond the container lifecycle.
- **Environment variables** inject configuration without rebuilding the image.

These features are central to Docker and other OCI (Open Container Initiative) runtimes, and they appear frequently in Linux administration and troubleshooting tasks.

**Port mapping and network exposure**

A container usually listens on an internal port inside its own network namespace. Port mapping connects a host port to that container port so clients can reach the service.

Typical Docker syntax:

```bash
docker run -d --name web \
  -p 8080:80 \
  nginx:latest
```

In this example:

- `8080` is the host port.
- `80` is the container port.
- Traffic sent to `http://host:8080` is forwarded to port `80` in the container.

Use port mapping when:

- You want to publish a service from a container to the host.
- You need to avoid conflicts with another service already using the same host port.
- You want to run multiple instances of the same service on one host by assigning different host ports.

Common forms of `-p`:

- `-p hostPort:containerPort` maps a specific host port.
- `-p ip:hostPort:containerPort` binds only to a specific host interface.
- `-p hostPort:containerPort/protocol` publishes TCP or UDP explicitly.

Examples:

```bash
docker run -d --name dns \
  -p 53:53/udp \
  bind9:latest
```

```bash
docker run -d --name admin \
  -p 127.0.0.1:8081:80 \
  nginx:latest
```

The second example binds the service only to localhost, which is useful for local testing or admin interfaces that should not be exposed externally.

**Port mapping vs. exposing ports**

Container images often document the ports they use with `EXPOSE` in a Dockerfile:

```dockerfile
EXPOSE 80
```

`EXPOSE` is documentation and metadata; it does not publish the port by itself. You still need `-p` or an equivalent runtime option to make the service reachable from outside the container.

> `EXPOSE` describes intent. `-p` creates the actual host-to-container forwarding rule.

**Inspecting mapped ports**

You can verify published ports with:

```bash
docker ps
docker port web
docker inspect web --format '{{json .NetworkSettings.Ports}}'
```

Useful troubleshooting checks:

- Confirm the application is listening inside the container.
- Confirm the host firewall allows the published port.
- Confirm the container port matches the service configuration.
- Confirm you mapped the correct protocol, especially for UDP services.

If a service listens on the wrong interface inside the container, port mapping alone will not fix it. The application must bind to `0.0.0.0` or the container’s network interface, not just `127.0.0.1`.

**Volumes and persistent storage**

Containers are ephemeral, so data written inside the writable layer disappears when the container is removed. Volumes solve this by storing data outside the container filesystem.

Common storage options:

- **Named volumes** are managed by the container engine.
- **Bind mounts** map a host directory or file into the container.
- **tmpfs mounts** store data in memory and vanish when the container stops.

Named volume example:

```bash
docker volume create dbdata

docker run -d --name mysql \
  -e MYSQL_ROOT_PASSWORD=secret \
  -v dbdata:/var/lib/mysql \
  mysql:8
```

This keeps database files in `dbdata` even if the container is recreated.

Bind mount example:

```bash
docker run -d --name app \
  -v /srv/app/config:/etc/app \
  myapp:latest
```

Bind mounts are useful when:

- You want to edit files directly on the host.
- You need to mount existing host data into a container.
- You are developing and want live code changes reflected immediately.

Named volumes are usually better when:

- You want portability across hosts.
- You want the runtime to manage storage location and lifecycle.
- You want to reduce accidental host-path coupling.

**Volume syntax and behavior**

Docker uses the general form:

```bash
-v source:destination[:options]
```

Examples:

- `-v dbdata:/var/lib/mysql`
- `-v /home/user/site:/usr/share/nginx/html:ro`

The `:ro` option makes the mount read-only, which is useful for static content or configuration files that should not be modified by the container.

You can also use the more explicit `--mount` syntax:

```bash
docker run -d --name web \
  --mount type=bind,source=/srv/site,target=/usr/share/nginx/html,readonly \
  nginx:latest
```

`--mount` is often easier to read in complex setups because it names each field explicitly.

**Volume management commands**

Common commands:

```bash
docker volume ls
docker volume inspect dbdata
docker volume rm dbdata
docker volume prune
```

For bind mounts, the host path must exist or be created as needed. Permissions matter:

- The container user must be able to read or write the mounted path.
- SELinux systems may require labels such as `:Z` or `:z` in some Docker setups.
- Ownership mismatches are a common cause of permission errors.

Example with SELinux relabeling:

```bash
docker run -d --name web \
  -v /srv/site:/usr/share/nginx/html:Z \
  nginx:latest
```

**Environment variables for configuration**

Environment variables let you pass runtime settings into a container without changing the image. They are commonly used for:

- Application mode selection
- Database credentials
- Service endpoints
- Feature flags
- Time zone and locale settings

Set variables with `-e`:

```bash
docker run -d --name app \
  -e APP_ENV=production \
  -e LOG_LEVEL=info \
  myapp:latest
```

You can also load variables from a file:

```bash
docker run -d --name app \
  --env-file .env \
  myapp:latest
```

Example `.env` file:

```bash
APP_ENV=production
LOG_LEVEL=info
DB_HOST=db
DB_PORT=5432
```

Environment variables are convenient, but they are not a secure secret store. Avoid placing sensitive values in shell history, plain text compose files, or world-readable files when better secret-management options are available.

**Environment variables in Dockerfiles**

You can define defaults in an image with `ENV`:

```dockerfile
FROM alpine:latest
ENV APP_ENV=production
ENV PORT=8080
```

`ENV` values become part of the image metadata and are available to processes in the container. Runtime `-e` values override image defaults.

Use `ARG` for build-time values, not runtime configuration:

```dockerfile
FROM alpine:latest
ARG VERSION=1.0
RUN echo "Building version $VERSION"
```

`ARG` exists only during image build, while `ENV` persists into the running container.

**How these features work together**

A practical container deployment often combines all three:

```bash
docker run -d --name wordpress \
  -p 8080:80 \
  -e WORDPRESS_DB_HOST=db \
  -e WORDPRESS_DB_USER=wpuser \
  -e WORDPRESS_DB_PASSWORD=secret \
  -v wpdata:/var/www/html \
  wordpress:latest
```

This setup:

- Publishes the web server on host port `8080`.
- Passes database connection settings through environment variables.
- Persists application files in a named volume.

A database container might use a separate volume:

```bash
docker run -d --name db \
  -e MYSQL_DATABASE=wordpress \
  -e MYSQL_USER=wpuser \
  -e MYSQL_PASSWORD=secret \
  -e MYSQL_ROOT_PASSWORD=rootsecret \
  -v dbdata:/var/lib/mysql \
  mysql:8
```

**Docker Compose example**

Compose is common in Linux administration because it makes multi-container configuration repeatable.

```yaml
services:
  web:
    image: nginx:latest
    ports:
      - "8080:80"
    volumes:
      - webdata:/usr/share/nginx/html:ro
    environment:
      NGINX_HOST: example.local
      NGINX_PORT: 80

  db:
    image: mysql:8
    environment:
      MYSQL_ROOT_PASSWORD: rootsecret
      MYSQL_DATABASE: appdb
    volumes:
      - dbdata:/var/lib/mysql

volumes:
  webdata:
  dbdata:
```

Key points:

- `ports` publishes container ports to the host.
- `volumes` persists data or mounts content.
- `environment` injects runtime variables.
- Named volumes are declared at the bottom for reuse.

**Troubleshooting checklist**

When a containerized service does not behave as expected, check these areas in order:

1. Verify the container is running.
   ```bash
   docker ps
   docker logs web
   ```
2. Verify the application listens on the expected port inside the container.
   ```bash
   docker exec -it web ss -tulpn
   ```
3. Verify the port mapping.
   ```bash
   docker port web
   ```
4. Verify host firewall rules and SELinux/AppArmor policy if applicable.
5. Verify mounted paths and permissions.
   ```bash
   docker inspect web --format '{{json .Mounts}}'
   ls -ld /srv/site
   ```
6. Verify environment variables inside the container.
   ```bash
   docker exec -it app printenv | sort
   ```

Common failure patterns:

- Host port already in use.
- Wrong container port published.
- Service binds only to `127.0.0.1` inside the container.
- Volume path permissions prevent writes.
- Missing or misspelled environment variable names.
- A `.env` file contains quotes or spaces that the application does not expect.

**Security and operational considerations**

Treat these features as operational controls, not just convenience options.

- **Port mapping** should expose only the ports you need.
- **Volumes** should be mounted read-only when write access is unnecessary.
- **Environment variables** should not be used for highly sensitive secrets unless the platform’s secret handling is appropriate.
- **Bind mounts** can give containers broad access to host files, so use them carefully.
- **Named volumes** reduce accidental exposure of arbitrary host directories.

> A container is not automatically isolated from risk just because it is packaged. The mount and port settings determine much of its real exposure.

**Quick comparison**

| Feature | Purpose | Best for | Common risk |
|---|---|---|---|
| Port mapping | Publish a service | Web apps, APIs, admin tools | Exposing too much to the network |
| Named volume | Persist data | Databases, application state | Orphaned data if not managed |
| Bind mount | Share host files | Development, config, local data | Host path permission and security issues |
| Environment variable | Pass configuration | Runtime settings, endpoints, modes | Secrets leakage or misconfiguration |

**Exam focus reminders**

For Linux+ tasks, be ready to recognize:

- The difference between container ports and host ports.
- The difference between `EXPOSE` and actual published ports.
- The difference between named volumes and bind mounts.
- The difference between `ENV` and `ARG`.
- How to inspect mappings, mounts, and runtime variables with CLI tools.

A strong mental model is:

- **Ports** control network reachability.
- **Volumes** control data persistence.
- **Environment variables** control runtime configuration.
