## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Container fundamentals**

Containers package an application with its runtime dependencies while sharing the host kernel. That makes them lighter than virtual machines (VMs), faster to start, and easier to scale. For Linux+ purposes, the key idea is that a container is not a full guest operating system; it is a process isolated by kernel features.

A container typically includes:

- The application binary and libraries
- Runtime files and configuration
- A minimal filesystem view
- Resource and namespace isolation enforced by the kernel

Containers are useful when you want:

- Repeatable deployments across environments
- Fast startup and teardown
- Stronger process isolation than plain host processes
- Efficient use of CPU, memory, and storage

> A container improves isolation, but it is not a security boundary equivalent to a hardened VM. You still need least privilege, image hygiene, and host hardening.

**Containers vs. virtual machines**

Containers and VMs solve similar deployment problems, but they isolate at different layers.

| Aspect | Container | Virtual machine |
|---|---|---|
| Kernel | Shares host kernel | Runs its own guest kernel |
| Startup time | Usually seconds or less | Usually slower |
| Overhead | Lower | Higher |
| Isolation model | Process-level isolation | Hardware-assisted OS isolation |
| Image size | Usually smaller | Usually larger |

Use containers when you need portability and density. Use VMs when you need a stronger isolation boundary, a different kernel, or a full OS environment.

**Core isolation primitives**

Linux containers rely on several kernel mechanisms working together.

- **Namespaces** isolate what a process can see.
- **cgroups** (control groups) limit and account for resource usage.
- **Capabilities** split root privileges into smaller units.
- **Filesystem layering** provides a writable container view over read-only image layers.
- **Security modules** such as SELinux or AppArmor add policy enforcement.

These mechanisms do not make a container magical; they simply constrain a process so it behaves like it has its own environment.

**Namespaces**

Namespaces give a process its own view of system resources. Different namespace types isolate different parts of the system.

- **PID namespace** isolates process IDs so a container can have its own `PID 1`.
- **Mount namespace** isolates mounted filesystems and mount points.
- **Network namespace** isolates interfaces, routes, ports, and firewall state.
- **UTS namespace** isolates hostname and domain name.
- **IPC namespace** isolates interprocess communication objects such as shared memory and message queues.
- **User namespace** maps container users to different host users and can reduce privilege.
- **Cgroup namespace** hides cgroup paths and hierarchy details.

A process can belong to multiple namespaces at once. That combination creates the container illusion.

Example inspection commands:

```bash
lsns
lsns -p 1234
readlink /proc/1234/ns/*
```

You can also inspect namespace membership through `/proc`:

```bash
cat /proc/1234/status | grep NSpid
```

**PID namespace behavior**

The first process inside a PID namespace becomes `PID 1` for that namespace. This matters because `PID 1` has special responsibilities:

- It adopts orphaned child processes
- It must handle signals correctly
- It often acts as the container init process

If `PID 1` does not reap zombies, the container can accumulate defunct processes. In practice, many container images use a tiny init process or an init-aware entrypoint to handle this.

**cgroups and resource control**

cgroups limit how much CPU, memory, I/O, and other resources a container can consume. They are essential for preventing one container from starving the host or other containers.

Common resource controls include:

- **CPU shares or quotas** to limit processor usage
- **Memory limits** to cap RAM consumption
- **Block I/O limits** to control disk throughput
- **PIDs limits** to restrict process creation
- **Device access controls** to limit hardware exposure

On modern systems, cgroup v2 is common. You can inspect the hierarchy with:

```bash
mount | grep cgroup
cat /proc/cgroups
systemd-cgls
systemd-cgtop
```

Example of checking a process’s cgroup:

```bash
cat /proc/1234/cgroup
```

A practical container runtime may translate flags like `--memory`, `--cpus`, or `--pids-limit` into cgroup settings.

**Capabilities and privilege reduction**

Traditional Unix privilege treats root as all-powerful. Linux capabilities split that power into smaller pieces such as:

- `CAP_NET_ADMIN` for network administration
- `CAP_SYS_ADMIN` for many system-level operations
- `CAP_CHOWN` for changing file ownership
- `CAP_NET_BIND_SERVICE` for binding to privileged ports

Containers often run with a reduced capability set. This limits damage if the application is compromised.

Practical examples:

```bash
docker run --cap-drop ALL --cap-add NET_BIND_SERVICE nginx
podman run --cap-drop=all --cap-add=net_bind_service nginx
```

Dropping unnecessary capabilities is one of the simplest hardening steps you can take.

**Filesystem isolation and image layers**

Container filesystems are usually built from layered images. Read-only layers provide the base application and dependencies, while a thin writable layer captures runtime changes.

This model gives you:

- Efficient storage reuse across containers
- Fast image distribution
- Easy rollback to a known image version

Common concepts:

- **Image**: a template used to create containers
- **Container**: a running instance of an image
- **Layer**: a filesystem diff that can be shared
- **Union filesystem**: a mechanism that merges layers into one view

Example image workflow:

```bash
podman pull alpine:latest
podman images
podman run --rm -it alpine:latest sh
```

Inside the container, changes to the writable layer are temporary unless you commit them or mount persistent storage.

**Bind mounts and volumes**

Containers often need persistent or shared data. You can provide storage in two common ways.

- **Bind mounts** map a host path into the container.
- **Volumes** are managed storage locations handled by the container engine.

Bind mounts are useful when you want direct access to host files. Volumes are usually better for portability and lifecycle management.

Examples:

```bash
docker run -d --name web -v /srv/web:/usr/share/nginx/html:ro nginx
podman run -d --name db -v mydbdata:/var/lib/postgresql/data postgres
```

Use read-only mounts when the container only needs to consume data. That reduces accidental modification risk.

**Container networking**

Containers usually get their own network namespace and connect to the host through virtual interfaces, bridges, or overlays. The exact implementation depends on the runtime and platform.

Common patterns:

- **Bridge networking** for local container-to-host communication
- **Host networking** for direct host stack sharing
- **Macvlan** for giving containers addresses on the physical network
- **Overlay networking** for multi-host container communication

Example inspection commands:

```bash
ip addr
ip link
ip route
ss -tulpn
```

A container may expose a service port to the host with port mapping:

```bash
docker run -d -p 8080:80 nginx
podman run -d -p 8080:80 nginx
```

Port mapping is a translation layer, not a security feature by itself. You still need firewall rules and service hardening.

**Security boundaries and limitations**

Containers reduce risk by isolating processes, but they do not eliminate shared-kernel risk. If the host kernel has a vulnerability, a container escape may be possible.

Important limitations:

- Containers share the host kernel
- A privileged container can weaken isolation significantly
- Misconfigured mounts can expose host files
- Excessive capabilities can expand attack surface
- Running as root inside a container can still be dangerous if mapped to host privileges

Hardening practices:

- Run as a non-root user when possible
- Drop unnecessary capabilities
- Use read-only root filesystems where practical
- Avoid `--privileged` unless you truly need it
- Keep images minimal and updated
- Restrict host mounts
- Apply SELinux or AppArmor policies
- Scan images for known vulnerabilities

> Avoid `--privileged` unless you are troubleshooting or have a specific, justified need. It disables many isolation protections.

**SELinux and AppArmor**

Security modules add mandatory access control (MAC) on top of container isolation.

- **SELinux** uses labels and policy rules to control access.
- **AppArmor** uses path-based profiles to confine processes.

These tools can prevent a container from accessing files or operations even if the process has Linux permissions. On many systems, container engines integrate with these policies automatically.

Useful checks:

```bash
getenforce
sestatus
aa-status
```

If a container fails due to policy enforcement, inspect audit logs before weakening the policy.

**Rootless containers**

Rootless containers run without full root privileges on the host. They rely heavily on user namespaces and subordinate UID/GID mappings.

Benefits:

- Reduced host risk
- Better fit for developer workstations
- Less dependence on privileged daemon access

Trade-offs:

- Some networking features are limited
- Certain mounts and device operations are restricted
- Performance or compatibility may differ from rootful containers

Example rootless workflow:

```bash
podman info
podman run --rm -it alpine sh
```

Rootless mode is a strong default when your workload does not need privileged host access.

**Common container runtime commands**

The exact tool may be Docker, Podman, or another OCI (Open Container Initiative) runtime, but the workflow is similar.

```bash
podman pull ubuntu:24.04
podman run --name testbox -it ubuntu:24.04 bash
podman ps
podman inspect testbox
podman logs testbox
podman stop testbox
podman rm testbox
```

Useful inspection commands:

```bash
docker inspect <container>
podman inspect <container>
docker stats
podman stats
```

For troubleshooting, check:

- Container logs
- Exit codes
- Image tags and digests
- Mounts and environment variables
- Resource limits
- Namespace and cgroup assignments

**Container lifecycle**

A container usually follows this lifecycle:

1. You pull or build an image.
2. You create and start a container from the image.
3. The container runs one main process.
4. The process exits or the container is stopped.
5. You remove the container if it is no longer needed.

This lifecycle is intentionally disposable. If you need persistent state, store it outside the container or in a volume.

**Practical configuration examples**

A simple web container with a read-only mount and port mapping:

```bash
podman run -d \
  --name webapp \
  -p 8080:80 \
  -v /srv/web:/usr/share/nginx/html:ro \
  nginx:stable
```

A more restrictive run with dropped capabilities and a memory limit:

```bash
docker run -d \
  --name api \
  --memory 256m \
  --pids-limit 100 \
  --cap-drop ALL \
  --cap-add NET_BIND_SERVICE \
  myapi:latest
```

A Compose example for multi-container applications:

```yaml
services:
  app:
    image: myapp:latest
    ports:
      - "8080:8080"
    environment:
      APP_ENV: production
  redis:
    image: redis:7
    volumes:
      - redisdata:/data

volumes:
  redisdata:
```

Compose is useful when you need repeatable multi-service stacks with shared configuration.

**Troubleshooting checklist**

When a container does not behave as expected, verify the basics first.

1. Confirm the image exists and the tag is correct.
2. Check whether the container is running or exited.
3. Review logs for application errors.
4. Inspect mounts, environment variables, and port mappings.
5. Check resource limits and cgroup settings.
6. Verify SELinux or AppArmor is not blocking access.
7. Confirm the process inside the container is listening on the expected interface and port.

Helpful commands:

```bash
podman ps -a
podman logs <container>
podman inspect <container> | less
ss -tulpn
journalctl -u podman
```

**Exam focus**

For Linux+ exam purposes, focus on these ideas:

- Containers share the host kernel, unlike VMs.
- Namespaces isolate views of system resources.
- cgroups control resource usage.
- Capabilities reduce the power of root.
- Images are layered and containers are disposable.
- Rootless containers improve safety where supported.
- Security modules such as SELinux and AppArmor can block container actions.
- Misconfiguration, not just software bugs, often causes container issues.

**Quick recall**

- **Namespace**: isolates what a process can see
- **cgroup**: limits what a process can use
- **Capability**: splits root privilege into smaller permissions
- **Image**: immutable template
- **Container**: running instance of an image
- **Volume**: persistent managed storage
- **Rootless**: container runs without host root privileges
