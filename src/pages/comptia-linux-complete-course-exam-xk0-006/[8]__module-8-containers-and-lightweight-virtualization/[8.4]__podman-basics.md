## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Podman overview**

Podman is an OCI (Open Container Initiative) container engine for building, running, and managing containers and pods on Linux. It is designed to work without a long-running daemon, which makes it different from Docker-style daemon-based workflows. A major advantage is **rootless** operation: you can run containers as a regular user, reducing the blast radius of a container escape or misconfiguration.

Podman is often used for:
- Running single containers for development or services
- Managing groups of containers with **pods**
- Building images from `Containerfile` or `Dockerfile` syntax
- Integrating with system services through `systemd`

A useful mental model is that Podman focuses on the container lifecycle and uses standard Linux kernel features such as namespaces, cgroups, and capabilities. It does not require a central daemon to keep containers alive.

**Core concepts**

Podman supports the same broad container workflow you expect from other engines, but the implementation details matter for Linux administration and exam scenarios.

- **Container**: An isolated process with its own filesystem view, network namespace, and process tree.
- **Image**: A read-only template used to create containers.
- **Containerfile**: The preferred build file name in Podman; `Dockerfile` syntax is usually compatible.
- **Pod**: A group of one or more containers that share the same network namespace and can share IPC and UTS namespaces.
- **Rootless container**: A container started by a non-root user.
- **Rootful container**: A container started by `root` or via `sudo`.

Podman stores images and containers in user-specific locations for rootless use, typically under `~/.local/share/containers/`, while rootful storage is usually under `/var/lib/containers/`. Exact storage drivers can vary by distribution and configuration.

**Why Podman matters**

Podman is especially useful when you want container workflows that fit naturally into Linux administration.

- **No daemon dependency** means fewer background services to manage.
- **Rootless support** improves security and simplifies least-privilege operation.
- **Pod support** helps when multiple containers must behave like a single application stack.
- **systemd integration** makes it easier to run containers as managed services at boot.
- **OCI compatibility** improves portability across container tooling.

In practice, you use Podman when you want to run containers on a Linux host without introducing a privileged daemon as a single point of failure.

**Installation and verification**

Package names vary by distribution, but the workflow is consistent: install Podman, verify the version, and confirm that the user environment is ready for rootless containers.

```bash
# Fedora/RHEL family
sudo dnf install -y podman

# Debian/Ubuntu family
sudo apt update
sudo apt install -y podman

# Verify installation
podman --version
podman info
```

`podman info` is useful because it shows storage, runtime, cgroup, and network configuration. If rootless support is not working as expected, this command often reveals the cause.

> Rootless containers may require subordinate UID/GID mappings. If user namespaces are not configured correctly, Podman can fail to start containers for non-root users.

**Images and registries**

Podman pulls images from registries such as Quay, Docker Hub, or private registries. Image names follow the familiar pattern `registry/namespace/name:tag`, though defaults may apply if you omit parts of the reference.

Common image operations:
- `podman pull` downloads an image
- `podman images` lists local images
- `podman rmi` removes an image
- `podman search` looks up images in configured registries

```bash
podman pull docker.io/library/alpine:latest
podman images
podman rmi docker.io/library/alpine:latest
```

When you omit a tag, `latest` is often assumed, but relying on `latest` is risky in production because it can change over time. Prefer explicit tags or digests for repeatable deployments.

**Running containers**

The most common task is starting a container from an image. You control the container’s lifecycle with `podman run`, `podman start`, `podman stop`, and `podman rm`.

```bash
podman run --name webtest -d -p 8080:80 docker.io/library/nginx:latest
podman ps
podman logs webtest
podman stop webtest
podman rm webtest
```

Key flags:
- `-d` runs the container in detached mode
- `--name` assigns a human-readable name
- `-p` publishes a container port to the host
- `--rm` removes the container when it exits
- `-it` allocates a TTY and keeps stdin open for interactive use

Example interactive shell:
```bash
podman run --rm -it docker.io/library/alpine:latest sh
```

Use `podman exec` to run a command inside an existing container:
```bash
podman exec -it webtest sh
```

**Container lifecycle and inspection**

Podman provides commands to inspect and troubleshoot containers without guessing.

- `podman ps` shows running containers
- `podman ps -a` shows all containers, including stopped ones
- `podman inspect` returns detailed JSON metadata
- `podman logs` displays container output
- `podman top` shows processes inside the container
- `podman stats` reports live resource usage

```bash
podman inspect webtest
podman top webtest
podman stats webtest
```

Use `inspect` when you need exact values for mounts, environment variables, network settings, or image metadata. Use `logs` when the application starts but fails quickly. Use `top` when you need to confirm the main process is running.

**Networking basics**

Podman can expose container services to the host and can also create pod-level networking. The simplest pattern is port publishing.

```bash
podman run -d --name apache -p 8080:80 docker.io/library/httpd:latest
curl http://localhost:8080
```

Important networking ideas:
- **Port publishing** maps a host port to a container port
- **Bridge networking** is common for isolated container connectivity
- **Pod networking** lets containers in the same pod share one IP address and port space
- **DNS resolution** may depend on the network mode and container runtime configuration

If a service is reachable inside the container but not from the host, check:
- The published port mapping
- Host firewall rules
- Whether the service listens on the expected interface
- SELinux policy on enforcing systems

**Pods**

A Podman pod is a group of containers that share networking and can share other namespaces. This is useful when one container provides the main application and another provides a sidecar service such as logging, proxying, or metrics collection.

```bash
podman pod create --name apppod -p 8080:80
podman run -d --pod apppod --name app docker.io/library/nginx:latest
podman pod ps
podman pod inspect apppod
```

Pods are helpful when containers must communicate over `localhost` inside the same pod. This mirrors patterns used in Kubernetes, which makes Podman useful for local development and testing.

**Building images**

Podman can build images from a `Containerfile` or `Dockerfile`. The build process creates a new image layer stack from instructions such as `FROM`, `RUN`, `COPY`, and `CMD`.

Example `Containerfile`:
```dockerfile
FROM docker.io/library/alpine:latest
RUN apk add --no-cache curl
COPY app.sh /usr/local/bin/app.sh
RUN chmod +x /usr/local/bin/app.sh
CMD ["/usr/local/bin/app.sh"]
```

Build and run:
```bash
podman build -t myapp:1.0 .
podman run --rm myapp:1.0
```

Build tips:
- Use small base images to reduce attack surface
- Pin versions for reproducibility
- Minimize layers where practical
- Avoid storing secrets in image layers

**Rootless operation**

Rootless containers are one of Podman’s most important features. They let you run containers without full root privileges, which is safer for development and many production use cases.

Benefits:
- Reduced privilege exposure
- Better alignment with least-privilege principles
- Easier multi-user container hosting
- Less dependence on privileged daemons

Common rootless considerations:
- User namespaces must be enabled
- Subordinate UID/GID ranges may be required
- Some privileged ports and kernel features may be restricted
- Host networking and device access can be limited

Check user namespace mappings:
```bash
podman unshare cat /proc/self/uid_map
podman unshare cat /proc/self/gid_map
```

If you need to expose a service on a privileged port such as 80 or 443, rootless containers may require alternatives such as a reverse proxy on the host, port forwarding, or a higher unprivileged port like 8080.

**Storage and volumes**

Containers are ephemeral by design, so persistent data should live outside the container filesystem. Podman supports bind mounts and named volumes.

Bind mount example:
```bash
podman run -d --name dbtest \
  -v /srv/data:/var/lib/data:Z \
  docker.io/library/postgres:latest
```

Named volume example:
```bash
podman volume create pgdata
podman run -d --name dbtest -v pgdata:/var/lib/postgresql/data docker.io/library/postgres:latest
```

Notes:
- `-v` mounts host or managed storage into the container
- `:Z` relabels content for a single container on SELinux systems
- `:z` shares content among multiple containers
- Use volumes for databases and stateful applications

**SELinux and security context**

On SELinux-enabled systems, container access to host files can fail if labels are incorrect. Podman works well with SELinux, but you must use the right mount options.

Common symptoms:
- Permission denied on mounted files
- Container can see the mount but cannot write to it
- Service starts but fails when accessing host data

Typical fix:
```bash
podman run -v /srv/web:/usr/share/nginx/html:Z -d --name web docker.io/library/nginx:latest
```

Security-related features to remember:
- **Capabilities** can be dropped to reduce privilege
- **Read-only** containers reduce write risk
- **Secrets** should not be baked into images
- **SELinux labeling** helps isolate container access

Example hardening pattern:
```bash
podman run -d \
  --name app \
  --read-only \
  --cap-drop=all \
  --security-opt=no-new-privileges \
  docker.io/library/nginx:latest
```

**systemd integration**

Podman can generate systemd unit files so containers start and stop like regular services. This is useful for boot persistence and centralized service management.

Generate a unit:
```bash
podman generate systemd --name webtest --files
```

For newer workflows, Podman also supports Quadlet on many distributions, which lets you define containers declaratively as systemd-managed files. Exact support depends on version and distro packaging.

Why this matters:
- You can manage containers with `systemctl`
- Services can start at boot
- Restart policies become easier to standardize
- Logging and dependency handling fit Linux service management

**Common management commands**

These commands cover most day-to-day administration tasks.

```bash
podman ps
podman ps -a
podman images
podman pull alpine
podman run --name test -it alpine sh
podman exec -it test sh
podman logs test
podman stop test
podman rm test
podman rmi alpine
podman system df
podman system prune
```

Useful cleanup commands:
- `podman rm -f <container>` force-removes a container
- `podman image prune` removes unused images
- `podman system prune` removes unused containers, images, networks, and build cache

> Be careful with prune commands on shared systems. They can remove resources that another workflow still expects to exist.

**Troubleshooting workflow**

When a container fails, use a consistent sequence instead of guessing.

1. Check whether the container is running with `podman ps -a`.
2. Inspect the exit code and logs with `podman logs <name>`.
3. Review the container configuration with `podman inspect <name>`.
4. Confirm port mappings with `podman port <name>`.
5. Verify file permissions, SELinux labels, and mount paths.
6. Test the image interactively with `podman run --rm -it <image> sh`.

Example:
```bash
podman ps -a
podman logs webtest
podman inspect webtest | less
podman port webtest
```

Common failure causes:
- Wrong command or entrypoint in the image
- Missing environment variables
- Port already in use on the host
- Incorrect volume permissions
- SELinux denial on mounted content
- Network or DNS misconfiguration

**Exam focus points**

For Linux+ exam scenarios, focus on what Podman does differently from older container workflows.

- **Daemonless** operation reduces background service dependence
- **Rootless** containers are a major security advantage
- **Pods** group containers that share networking
- **Containerfile** is the preferred build file name, though Dockerfile syntax is usually compatible
- **systemd** integration supports service-style management
- **OCI** compatibility improves portability

You should be able to recognize the right command for common tasks:
- Start a container: `podman run`
- List containers: `podman ps`
- View logs: `podman logs`
- Enter a running container: `podman exec`
- Build an image: `podman build`
- Manage images: `podman images`, `podman rmi`
- Create a pod: `podman pod create`

**Practical example workflow**

A simple end-to-end workflow helps connect the concepts.

```bash
# Pull an image
podman pull docker.io/library/nginx:latest

# Run it with a published port
podman run -d --name web -p 8080:80 docker.io/library/nginx:latest

# Verify it is running
podman ps
curl http://localhost:8080

# Inspect details
podman inspect web

# Stop and remove it
podman stop web
podman rm web
```

This sequence shows the typical lifecycle: obtain an image, create a container, verify service access, inspect configuration, and clean up when finished.

**Key takeaways**

Podman is a Linux-native container engine that emphasizes security, portability, and service integration. Its daemonless design and rootless support make it a strong fit for modern Linux administration. For exam purposes, know the core commands, understand how pods differ from single containers, and be comfortable with image management, port publishing, persistent storage, and troubleshooting.
