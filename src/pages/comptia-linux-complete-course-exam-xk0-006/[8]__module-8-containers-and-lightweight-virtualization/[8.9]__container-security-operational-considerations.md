## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Container security overview**

Containers package an application and its dependencies into an isolated runtime, but they do not create a full security boundary by themselves. You still rely on the host kernel, the container runtime, image provenance, and your deployment controls. For exam prep, focus on the difference between *isolation* and *hardening*: containers reduce blast radius, but misconfiguration can still expose the host, secrets, or network.

A secure container strategy usually includes:

- Minimal base images
- Non-root execution
- Read-only filesystems where possible
- Dropped Linux capabilities
- Seccomp and AppArmor or SELinux confinement
- Image scanning and patch management
- Resource limits and logging
- Controlled networking and secret handling

**Image security and supply chain hygiene**

Image security starts before a container runs. You want to trust the source, reduce the attack surface, and keep layers small and current. Smaller images usually mean fewer packages, fewer CVEs (Common Vulnerabilities and Exposures), and faster pulls.

Practical image hygiene includes:

- Using official or trusted base images
- Pinning versions instead of using floating tags like `latest`
- Rebuilding images regularly to pick up patched packages
- Removing build tools and package caches from runtime images
- Verifying image digests when you need repeatable deployments

Example Dockerfile pattern for a multi-stage build:

```dockerfile
FROM golang:1.22 AS build
WORKDIR /src
COPY . .
RUN CGO_ENABLED=0 go build -o /out/myapp ./cmd/myapp

FROM debian:stable-slim
RUN useradd --system --create-home --shell /usr/sbin/nologin appuser
WORKDIR /app
COPY --from=build /out/myapp /usr/local/bin/myapp
USER appuser
ENTRYPOINT ["/usr/local/bin/myapp"]
```

Why this helps:

- The build toolchain stays in the build stage and does not ship in the final image.
- The runtime image contains only the binary and minimal OS libraries.
- The application runs as a non-root user.

If you need to inspect image metadata, use standard tooling such as:

```bash
docker image inspect myapp:1.0
docker history myapp:1.0
docker images --digests
```

For vulnerability analysis, use current image-scanning tools available in your environment, such as `docker scout`, registry-integrated scanners, or CI/CD security scanners.

**Runtime hardening**

At runtime, the goal is to limit what a compromised container can do. Containers inherit kernel features, so you reduce risk by removing unnecessary privileges and constraining access.

Key runtime controls:

- **Non-root user**: Avoid running as `root` inside the container unless the workload requires it.
- **Capabilities**: Drop Linux capabilities you do not need.
- **Read-only root filesystem**: Prevent unexpected writes to system paths.
- **Tmpfs mounts**: Provide writable temporary storage without persisting data.
- **Seccomp**: Filter system calls to reduce kernel attack surface.
- **AppArmor/SELinux**: Apply mandatory access control policies where supported.
- **No-new-privileges**: Prevent privilege escalation through setuid binaries or file capabilities.

Example `docker run` hardening options:

```bash
docker run -d \
  --name myapp \
  --user 1000:1000 \
  --read-only \
  --tmpfs /tmp \
  --cap-drop ALL \
  --security-opt no-new-privileges \
  --pids-limit 200 \
  --memory 256m \
  --cpus 0.5 \
  myapp:1.0
```

Notes on these flags:

- `--user 1000:1000` runs the process as a non-root UID and GID.
- `--read-only` blocks writes to the container root filesystem.
- `--tmpfs /tmp` gives the app a writable temporary directory.
- `--cap-drop ALL` removes all Linux capabilities; add back only what you need.
- `--security-opt no-new-privileges` blocks privilege escalation.
- `--pids-limit`, `--memory`, and `--cpus` help contain resource abuse.

> Be careful with `--cap-add`: every added capability expands the container's power. Add only the minimum required capability, such as `NET_BIND_SERVICE` for binding to privileged ports without full root access.

**Linux capabilities and privilege reduction**

Linux capabilities split the power of `root` into smaller units. This lets you grant specific privileges without giving full administrative access. For example, a web server may need to bind to port 80 but not mount filesystems or load kernel modules.

Common examples:

- `CAP_NET_BIND_SERVICE`: bind to ports below 1024
- `CAP_CHOWN`: change file ownership
- `CAP_SETUID` and `CAP_SETGID`: change user or group identity
- `CAP_SYS_ADMIN`: very broad and usually too powerful for containers

Example of adding only one capability:

```bash
docker run -d \
  --cap-drop ALL \
  --cap-add NET_BIND_SERVICE \
  myweb:1.0
```

If the application can listen on a high port, prefer that over adding a capability. For example, run the app on port `8080` and map it externally:

```bash
docker run -d -p 80:8080 myweb:1.0
```

This keeps the container unprivileged while still serving traffic on port 80 from the host.

**Namespaces, cgroups, and isolation limits**

Containers rely on kernel isolation features:

- **Namespaces** isolate process IDs, mounts, networking, users, and other resources.
- **cgroups** (control groups) limit CPU, memory, I/O, and process counts.

These features improve containment, but they are not equivalent to a virtual machine. A kernel vulnerability can affect multiple containers on the same host. That is why host patching and runtime restrictions matter.

Operationally, you should:

- Keep the host kernel updated
- Use a supported container runtime
- Avoid privileged containers unless absolutely necessary
- Restrict access to the Docker socket or container runtime socket
- Separate workloads with different trust levels onto different hosts when possible

> Mounting `/var/run/docker.sock` into a container gives that container effectively root-equivalent control over the host Docker daemon. Treat it as highly sensitive.

**Secrets management**

Never bake secrets into images. Images are often copied, cached, and stored in registries, which makes embedded secrets hard to revoke and easy to leak. Instead, inject secrets at runtime using your orchestrator or secret store.

Good practices:

- Store secrets outside the image
- Use environment variables only when the risk is acceptable and the secret is short-lived
- Prefer mounted secret files or platform secret objects
- Rotate credentials regularly
- Limit secret scope to the minimum required service

Example of passing a secret file into a container:

```bash
docker run -d \
  --name dbclient \
  -v /run/secrets/db_password:/run/secrets/db_password:ro \
  myclient:1.0
```

Inside the container, the application reads the file rather than embedding the password in the image.

In orchestration platforms, secrets are often mounted as files or injected through dedicated secret APIs. The exact mechanism varies, but the principle is the same: keep secrets out of image layers and source control.

**Networking considerations**

Container networking affects both security and operations. By default, containers may be attached to bridge networks, user-defined networks, or host networking depending on the runtime and configuration. Each choice changes exposure.

Security-oriented networking practices:

- Expose only required ports
- Bind services to specific interfaces when possible
- Use user-defined bridge networks for service-to-service isolation
- Avoid `--network host` unless you need host-level networking behavior
- Restrict east-west traffic between containers when the platform supports it
- Use network policies in orchestrated environments

Example of publishing a port only on localhost:

```bash
docker run -d -p 127.0.0.1:8080:8080 myapp:1.0
```

This keeps the service reachable from the host but not directly from other network interfaces.

Example of creating and using a user-defined network:

```bash
docker network create appnet
docker run -d --name db --network appnet postgres:16
docker run -d --name api --network appnet myapi:1.0
```

User-defined networks improve service discovery and let you isolate groups of containers from unrelated workloads.

**Logging, monitoring, and auditability**

Operational security depends on visibility. If you cannot observe container behavior, you cannot detect compromise, resource exhaustion, or configuration drift.

Focus on:

- Centralized logs
- Time synchronization
- Health checks
- Metrics for CPU, memory, and restart counts
- Audit logs for image pulls, deployments, and administrative actions

Example health check in a Dockerfile:

```dockerfile
HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
  CMD curl -f http://localhost:8080/health || exit 1
```

Operational commands for inspection:

```bash
docker ps
docker logs myapp
docker inspect myapp
docker stats
```

Use these to confirm:

- The container is running the expected image
- The process is healthy
- Resource usage is within limits
- Mounts, networks, and environment variables match the intended configuration

**Resource limits and denial-of-service resistance**

Containers can fail due to legitimate load or malicious abuse. Resource limits help prevent one workload from starving others on the same host.

Important controls:

- Memory limits to prevent runaway allocation
- CPU quotas or shares to prevent noisy-neighbor issues
- PID limits to reduce fork bombs
- Disk quotas or storage driver controls where available
- Ulimits for file descriptors and process counts

Example:

```bash
docker run -d \
  --memory 512m \
  --memory-swap 512m \
  --cpus 1.0 \
  --pids-limit 100 \
  myapp:1.0
```

Operationally, set limits based on observed usage, not guesswork. Too-low limits can cause crashes or throttling; too-high limits reduce the value of containment.

**Orchestrator and host operational concerns**

In real deployments, containers often run under an orchestrator such as Kubernetes, but the same security principles apply. You still need image trust, least privilege, network segmentation, and secret management.

Common operational concerns include:

- Scheduling workloads onto trusted nodes
- Using namespaces or projects to separate teams and environments
- Applying admission controls or policy checks before deployment
- Rolling updates and rollback plans
- Node maintenance and patch windows
- Backup and restore for persistent volumes
- Image registry access control

If you administer Docker directly, protect the daemon socket and limit who can join the `docker` group. Membership in that group often grants near-root access on the host.

**Practical hardening checklist**

Use this as a deployment review before production rollout:

1. Confirm the image source is trusted and the tag or digest is pinned.
2. Remove build tools and unnecessary packages from the runtime image.
3. Run the process as a non-root user.
4. Drop all capabilities and add back only what is required.
5. Enable read-only filesystem access where the app allows it.
6. Mount temporary and persistent data separately.
7. Apply seccomp and MAC policies such as AppArmor or SELinux.
8. Set CPU, memory, and PID limits.
9. Keep secrets out of the image and source repository.
10. Expose only the ports and interfaces the service needs.
11. Enable logging, health checks, and monitoring.
12. Patch the host, runtime, and base images regularly.

**Troubleshooting common security and operational issues**

When a hardened container fails, the cause is often one of the controls you added. Troubleshoot systematically:

- If the app cannot write files, check whether the filesystem is read-only and whether the app has a writable mount.
- If the app cannot bind to port 80, check whether it runs as non-root and whether you need `CAP_NET_BIND_SERVICE`.
- If the app crashes on startup, inspect logs and verify that required environment variables and secret files exist.
- If the container is killed under load, review memory and PID limits.
- If networking fails, verify the container network, published ports, and firewall rules.

Useful commands:

```bash
docker logs --tail 50 myapp
docker exec -it myapp sh
docker inspect myapp | less
```

**Exam focus summary**

For Linux+ exam prep, remember these high-value points:

- Containers share the host kernel, so they are not a complete security boundary.
- Least privilege matters at build time and runtime.
- Non-root execution, capability dropping, and read-only filesystems are core hardening steps.
- Secrets should be injected at runtime, not baked into images.
- Resource limits and health checks are operational controls as much as security controls.
- Image scanning, patching, and trusted registries reduce supply-chain risk.
- Docker socket access is highly privileged and should be tightly controlled.
