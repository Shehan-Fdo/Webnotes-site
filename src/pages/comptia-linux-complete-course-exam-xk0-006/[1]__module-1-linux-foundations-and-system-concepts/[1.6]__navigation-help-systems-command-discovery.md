## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Why this topic matters**

Linux administration depends on quickly answering three questions: where am I, how do I get help, and what command should I use. In real troubleshooting, you often need to confirm your current directory, identify whether a command is a shell builtin or an external binary, and find the right documentation without guessing.

This topic is heavily practical. The exam expects you to recognize the right tool for the job, not just memorize definitions. You should know when to use `pwd`, `cd`, `type`, `help`, `man`, `apropos`, `find`, `locate`, `whereis`, and related commands.

**Navigation basics**

The shell keeps track of your current working directory, which is the directory where relative paths are resolved. Navigation is usually done with `cd`, while `pwd` confirms your location.

- `pwd` prints the current working directory.
- `cd` changes directories.
- `cd` with no arguments usually returns you to your home directory.
- `cd -` switches to the previous directory and is useful for quick back-and-forth work.
- Relative paths are resolved from the current directory; absolute paths start at `/`.

```bash
pwd
cd /var/log
pwd
cd -
pwd
cd
```

A few path concepts are worth keeping straight:

- `.` means the current directory.
- `..` means the parent directory.
- `~` expands to your home directory.
- `~user` expands to another user’s home directory if permitted.

```bash
cd ..
cd ./projects
ls ~/Downloads
```

**Shell command discovery**

When you type a command, the shell does not always run an external executable. It may run an alias, function, builtin, or a file found in `PATH`. Understanding this order helps you troubleshoot “command not found” errors and unexpected behavior.

Use `type` to identify what the shell will execute.

```bash
type cd
type ls
type -a python
```

- `type` shows whether a name is an alias, function, builtin, keyword, or file.
- `type -a` lists all matches in resolution order, which is useful when multiple commands share the same name.
- `type -t` prints only the type name, which is handy in scripts or quick checks.

`type -a` is especially important when an alias or function shadows an executable. For example, if `ls` is aliased, `type -a ls` shows the alias first and then the underlying binaries found in `PATH`.

```bash
alias ls='ls --color=auto'
type -a ls
```

`command -v` is another common discovery tool. It is often used in scripts because it returns a concise path or name and is generally more portable than shell-specific introspection commands.

```bash
command -v bash
command -v cd
```

- `command -v` reports how the shell would resolve a command.
- `command -V` gives a more verbose description.
- `command` can also bypass shell functions and aliases when you need the external command.

> Be careful with shell-specific behavior. `echo` is a shell builtin in Bash, but its exact behavior can vary across shells, especially with escape handling and option parsing. For portable scripts, prefer `printf` when output formatting matters.

**Builtins, external commands, and `help`**

Shell builtins are commands implemented inside the shell itself. They are often faster and can affect shell state, such as `cd`, `export`, `alias`, `unset`, and `help`. External commands are separate executables stored on disk.

Use `help` to view documentation for Bash builtins.

```bash
help cd
help export
help help
```

- `help` works for builtins in Bash and similar shells that support it.
- `help -d` gives a short description.
- `help -m` shows a man-page-like format in Bash.
- `help` does not document external commands; use `man` for those.

A practical workflow is to check the command type first, then choose the right help source.

1. Run `type <name>` to see what kind of command it is.
2. Use `help <name>` if it is a builtin.
3. Use `man <name>` if it is an external command.
4. Use `command -v <name>` when you only need the resolved path or a quick existence check.

**Manual pages and documentation lookup**

`man` is the primary documentation system on Linux. It opens manual pages organized into sections, and many commands appear in more than one section.

Common sections include:

- `1` user commands
- `2` system calls
- `3` library functions
- `5` file formats and configuration files
- `8` system administration commands

```bash
man ls
man 5 passwd
man 8 useradd
```

Use section numbers when a name is ambiguous. For example, `passwd` can refer to a user command or a file format entry, depending on the section.

Useful `man` options include:

- `man -k <keyword>` searches the manual page database by keyword.
- `man -f <name>` shows the one-line description for a topic.
- `man -a <name>` opens all matching manual pages in sequence.
- `man -w <name>` prints the location of the manual page files.
- `man -K <string>` searches the full text of all manual pages; this can be slow.

```bash
man -k network
man -f passwd
man -a printf
man -w ls
man -K "SELinux context"
```

`apropos` is closely related to `man -k`. It searches the manual page database for keywords and is useful when you do not know the exact command name.

```bash
apropos archive
apropos "file permissions"
```

- `whatis` is similar to `man -f`, but it depends on the manual database being available and reasonably current.
- If the database is stale or missing, `whatis` and `man -f` may return incomplete results.

`man` navigation is also exam-relevant:

- `/pattern` searches forward within a page.
- `n` repeats the search forward.
- `N` repeats the search backward.
- `q` quits the pager.
- `h` shows pager help in `less`, which is commonly used by `man`.

**Command location and path discovery**

Sometimes you need to know where a command lives on disk. This matters when debugging `PATH` issues, verifying package ownership, or comparing multiple installed versions.

`which` prints the path of an executable found in `PATH`.

```bash
which python3
which ls
```

- `which` is useful for a quick lookup.
- `which` usually reports only the first match.
- `which` may not show aliases, functions, or shell builtins.
- `type -a` is more complete when you need to see all resolution matches.

`whereis` searches for a command’s binary, source, and manual page locations.

```bash
whereis bash
whereis ssh
```

- `whereis` can be helpful when you want binaries and docs together.
- `whereis` does not search the entire filesystem.
- It relies on predefined locations, so it may miss unusual installations.

**Finding files with `find`**

`find` searches the filesystem in real time and is the most flexible discovery tool for files and directories. It is slower than `locate`, but it is accurate because it walks the directory tree directly.

```bash
find /var/log -name "*.log"
find ~ -type f -name "*.conf"
```

Common `find` tests and actions include:

- `-name` matches by filename.
- `-type f` finds regular files.
- `-type d` finds directories.
- `-user` and `-group` filter by ownership.
- `-perm` filters by permissions.
- `-size` filters by file size.
- `-mtime`, `-atime`, and `-ctime` filter by timestamps.
- `-maxdepth` and `-mindepth` control traversal depth.

```bash
find /etc -maxdepth 1 -type f
find /var -mindepth 2 -maxdepth 3 -name "*.conf"
find /home -type f -size +100M
find /tmp -type f -mtime -2
```

Permission matching with `-perm` is a common exam trap:

- `-perm 0644` matches files with exactly mode `0644`.
- `-perm -0644` matches files that have at least those permission bits set.
- `-perm /0644` matches files with any of those bits set, depending on the `find` implementation and mode semantics.

```bash
find /srv -type f -perm -0644
```

Use parentheses and `-o` to combine expressions correctly. Parentheses must be escaped or quoted so the shell does not interpret them.

```bash
find /var/log \( -name "*.log" -o -name "*.gz" \) -type f
find /etc -type f \( -name "*.conf" -o -name "*.cfg" \)
```

Actions let you do something with each match:

- `-print` is the default action in many cases.
- `-exec ... \;` runs a command once per match.
- `-exec ... +` batches multiple matches into fewer command invocations.
- `-delete` removes matched files and should be used carefully.

```bash
find /tmp -type f -name "*.tmp" -delete
find /var/log -type f -name "*.log" -exec gzip {} \;
find /home -type f -name "*.txt" -exec chmod 640 {} +
```

> `-delete` is powerful and irreversible. Always test with `-print` first, and consider adding `-maxdepth` or a narrower starting path before deleting anything.

A practical `find` workflow is:

1. Start with a narrow path.
2. Add a type filter such as `-type f` or `-type d`.
3. Add a name, size, time, or ownership test.
4. Test with `-print`.
5. Add an action such as `-exec` or `-delete` only after verifying the matches.

**Fast file lookup with `locate`**

`locate` searches a prebuilt database instead of walking the filesystem live. It is much faster than `find` for broad searches, but results can be stale if the database has not been updated recently.

```bash
locate ssh_config
locate "*.service"
```

- `locate` is ideal when you need a quick guess at where a file might be.
- `locate` may return paths that no longer exist if the database is outdated.
- `locate` can miss recently created files until the database is refreshed.
- Database contents may exclude paths based on configuration, permissions, or indexing rules.

Database maintenance is usually handled by scheduled jobs such as cron or systemd timers, depending on the distribution and package used. The update command is commonly `updatedb`.

```bash
sudo updatedb
locate bashrc
```

If `locate` behaves unexpectedly, check whether the database is current and whether the system’s indexing configuration excludes the directory you expect.

**Choosing the right tool**

Different tools answer different questions. The exam often tests whether you can choose the most efficient command for a scenario.

| Need | Best tool | Why |
|---|---|---|
| Find your current directory | `pwd` | Shows the exact working directory |
| Move around the filesystem | `cd` | Changes the shell’s current location |
| Identify what a command resolves to | `type` | Shows alias, function, builtin, or file |
| See all matches for a name | `type -a` | Reveals shadowing and multiple binaries |
| Check if a command exists in `PATH` | `command -v` | Concise and script-friendly |
| Read builtin help | `help` | Builtin documentation source |
| Read command documentation | `man` | Full manual pages |
| Search manuals by topic | `apropos` or `man -k` | Finds commands by keyword |
| Find files live on disk | `find` | Accurate, flexible, slower |
| Search a file database quickly | `locate` | Fast, but database-dependent |
| Find binary, source, and man page locations | `whereis` | Good for quick package-related lookup |

**Real-world troubleshooting examples**

If a command behaves strangely, first determine what the shell is actually running.

```bash
type -a ls
type -a python
command -v ssh
```

If `ls` is aliased, you may need to bypass the alias with `command ls` or call the full path directly.

If you need documentation for a shell builtin, use `help` instead of `man`.

```bash
help cd
help export
```

If you need to find a configuration file but do not know the exact name, search the manuals or use `locate` first, then verify with `find`.

```bash
apropos ssh
locate sshd_config
find /etc -name "sshd_config"
```

If you need to search for a file under a specific directory and only within a limited depth, `find` is the correct choice.

```bash
find /etc -maxdepth 2 -type f -name "*.conf"
```

If you need to locate a command’s documentation and binary locations together, `whereis` can be a quick first pass.

```bash
whereis passwd
whereis man
```

**Exam-focused reminders**

- `type` tells you what the shell will run; `which` usually tells you only the first executable in `PATH`.
- `type -a` is the best choice when aliases, functions, or multiple binaries may shadow each other.
- `help` is for shell builtins, not external commands.
- `man -k` and `apropos` help when you know the topic but not the exact command name.
- `man -a` and `man -w` are useful when a command has multiple manual entries or you need the file location.
- `find` searches live data; `locate` searches an index.
- `updatedb` refreshes the `locate` database, but the exact scheduling and path exclusions vary by distribution.
- `find -exec ... +` is more efficient than `-exec ... \;` when many files match.
- `find -delete` is destructive and should be tested carefully.

**Quick command reference**

```bash
pwd
cd /path/to/dir
type -a command
command -v command
help builtin
man command
man -k keyword
apropos keyword
whereis command
find /path -type f -name "*.conf"
locate filename
sudo updatedb
```
