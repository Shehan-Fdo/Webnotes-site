## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Linux origins and the GNU/Linux relationship**

Linux begins as a kernel project, not a complete operating system. Linus Torvalds releases the first Linux kernel in 1991, and the project quickly becomes useful because it combines with GNU user-space tools, libraries, and utilities. That relationship matters for exam questions: the kernel handles hardware interaction and process management, while the surrounding GNU components provide the shell, compilers, core utilities, and development tools that make the system practical.

A simple mental model is:

```text
Hardware -> Linux kernel -> system libraries -> GNU utilities -> user applications
```

The term **GNU/Linux** reflects that layered system. In everyday speech, many people say “Linux” to mean the whole operating system, but technically Linux is the kernel. On exams, be ready to distinguish the kernel from the full distribution.

- **Kernel**: core that manages CPU, memory, devices, and system calls
- **Distribution**: packaged operating system built from the kernel plus tools, package manager, and defaults
- **User space**: everything outside the kernel, including shells and applications

**Key milestones in Linux and free software history**

Linux history is tied to the broader free software movement. The timeline helps explain why licensing and community governance matter.

- 1983: Richard Stallman announces the GNU Project to build a free Unix-like operating system
- 1985: The Free Software Foundation (FSF) is founded to support free software development
- 1989: GNU General Public License version 1 (GPLv1) is published
- 1991: Linux kernel is first released
- 1992: Linux is relicensed under **GPLv2**, which is the important version to remember for exam accuracy
- 1993: Major early distributions such as Slackware and Debian help broaden adoption
- 1994: Linux kernel 1.0 is released
- 1996: Linux kernel 2.0 adds SMP (Symmetric Multiprocessing) support
- 2001: Linux kernel 2.4 improves scalability and hardware support
- 2003: Linux kernel 2.6 becomes a major adoption marker for enterprise and desktop use
- 2011 onward: Linux becomes dominant in cloud, containers, embedded systems, and mobile infrastructure

The history is not just trivia. It explains why Linux culture emphasizes collaboration, source availability, and the right to modify and redistribute software.

**Free software philosophy and the four freedoms**

Free software is defined by user freedom, not price. A program can be free software and still cost money; the key is whether users receive the freedoms to use and share it.

The four freedoms are central:

1. **Freedom 0**: run the program for any purpose
2. **Freedom 1**: study how the program works and change it
3. **Freedom 2**: redistribute copies
4. **Freedom 3**: distribute modified versions

These freedoms depend on source code access and legal permission to modify and share. That is why free software licenses usually require source availability or a written offer for source when binaries are distributed.

> A common exam trap is confusing “free software” with “free of charge.” The philosophy is about user control, not zero cost.

The free software movement also emphasizes that software should not lock users into vendor control. In practice, this means:

- You can audit code for security and privacy
- You can fix bugs without waiting for the vendor
- You can adapt software to new hardware or workflows
- You can redistribute improvements to the community

**Open source philosophy and the OSI definition**

Open source overlaps with free software, but the emphasis is different. Free software focuses on user freedom and ethics; open source focuses on development methodology, collaboration, and practical benefits such as reliability and faster innovation.

The Open Source Initiative (OSI) maintains the **Open Source Definition (OSD)**. A license is open source if it meets criteria such as:

- Free redistribution
- Source code availability
- Permission to create derived works
- Integrity of the author’s source code, when required
- No discrimination against persons or groups
- No discrimination against fields of endeavor
- Distribution of license rights with the program
- License must not be specific to a product
- License must not restrict other software
- License must be technology-neutral

A useful comparison:

| Aspect | Free software view | Open source view |
|---|---|---|
| Main focus | User freedom and ethics | Practical development model |
| Typical language | “freedom” and “rights” | “collaboration” and “efficiency” |
| Licensing goal | Preserve user freedoms | Encourage open development |
| Exam relevance | Understand the four freedoms | Know OSI criteria and license compatibility |

The two movements often support the same licenses, but they frame the reasons differently.

**Common license families and what they mean**

Linux distributions combine many licenses, so you need to recognize the major families and their obligations.

- **GPL (GNU General Public License)**: strong copyleft; derivative works must remain under GPL-compatible terms when distributed
- **LGPL (Lesser GPL)**: weaker copyleft, often used for libraries; allows linking under certain conditions without forcing the entire application to become GPL
- **MIT/BSD/Apache-style permissive licenses**: allow broad reuse with fewer redistribution requirements
- **MPL (Mozilla Public License)**: file-level copyleft; modified files remain under MPL, but larger combined works can be distributed under different terms
- **Proprietary licenses**: source code is typically closed and redistribution is restricted

A practical rule of thumb:

- **Permissive licenses** are easier to combine into proprietary or mixed projects
- **Copyleft licenses** require you to preserve freedoms in redistributed derivatives
- **GPL-compatible** means a license can be combined with GPL code without violating GPL terms

> Be careful with “public domain” and “Creative Commons” assumptions. Public domain is not a software license in the usual compliance sense, and many CC licenses are designed for content, not software. In Linux package compliance, you usually need a software-appropriate license with clear redistribution and source terms.

**Copyleft, permissive licensing, and compliance impact**

Copyleft is often tested because it affects packaging, distribution, and derivative work analysis. The key question is not “Is the code open source?” but “What obligations apply when you distribute it?”

- **Strong copyleft**: GPL family; derivative works must preserve the same freedoms
- **Weak copyleft**: LGPL or MPL; allows more flexible combination
- **Permissive**: MIT, BSD, Apache; minimal redistribution restrictions beyond attribution and notices

Typical compliance obligations include:

- Include copyright and license notices
- Provide source code or a valid source offer when required
- Preserve modification notices
- Provide installation information when the license requires it for user products
- Avoid adding extra restrictions that conflict with the upstream license

A practical compliance workflow often looks like this:

```bash
# Inspect package metadata and license fields
rpm -qi bash | grep -i license
dpkg -s bash | grep -i '^License\|^Maintainer'

# Search a source tree for common license files
find . -maxdepth 2 -type f \( -iname 'LICENSE*' -o -iname 'COPYING*' -o -iname 'NOTICE*' \) -print

# Review declared licenses in build metadata
grep -RIn --exclude-dir=.git -E 'SPDX-License-Identifier|License:' .
```

The `find` approach is usually more reliable than searching for one exact phrase, because projects may use `COPYING`, `LICENSE.md`, SPDX tags, or package metadata instead of a full license text.

**Static linking, dynamic linking, and derivative work questions**

Exam questions often ask whether linking creates a derivative work. The safest answer is that it depends on the license and distribution model, and legal interpretation can vary by jurisdiction. For Linux+ purposes, focus on the practical compliance implications.

- **Static linking** combines library code into the executable at build time
- **Dynamic linking** loads shared libraries at runtime
- **GPL libraries** may impose stronger redistribution obligations if you distribute a combined work
- **LGPL libraries** are designed to allow linking under more flexible conditions

Scenario examples:

- If you statically link a GPL library into your application and distribute the result, you may need to release the application under GPL-compatible terms
- If you dynamically link to an LGPL library, you can often keep your application under different terms, provided you meet the LGPL conditions
- If you merely use a permissively licensed library, redistribution is usually simpler, but you still need to preserve notices and attribution

A useful exam mindset is to ask:

1. Are you distributing the software or only using it internally?
2. Is the component GPL, LGPL, permissive, or proprietary?
3. Is the link static or dynamic?
4. Does the license require source distribution, notices, or relinking rights?

**Kernel modules and license compatibility**

Kernel modules run in kernel space and interact closely with the Linux kernel. That makes licensing more sensitive than ordinary user-space applications.

User-space software and kernel-space modules are treated differently because kernel modules can use kernel interfaces, internal symbols, and module-loading mechanisms that are tightly controlled by the kernel project. Some kernel symbols are exported for modules, but access to certain symbols may be limited to modules that are considered GPL-compatible.

Important points:

- A module that is not GPL-compatible may be unable to use some kernel symbols marked for GPL-only use
- Loading a proprietary module often taints the kernel, but tainting behavior depends on the module and kernel state, so avoid treating it as an absolute rule
- Kernel module licensing is not the same as user-space application licensing
- A module can be technically loadable yet still create compliance or support concerns

Example module metadata:

```c
#include <linux/module.h>
#include <linux/kernel.h>

MODULE_LICENSE("GPL");
MODULE_AUTHOR("Example");
MODULE_DESCRIPTION("Sample kernel module");

static int __init sample_init(void)
{
    pr_info("sample loaded\n");
    return 0;
}

static void __exit sample_exit(void)
{
    pr_info("sample unloaded\n");
}

module_init(sample_init);
module_exit(sample_exit);
```

If you change `MODULE_LICENSE("GPL")` to a non-GPL-compatible string, the kernel may treat the module differently, including restricting access to some symbols and potentially marking the kernel as tainted.

**Dual licensing, CLAs, and copyright ownership**

Real projects often use more than one licensing strategy. These concepts are important in enterprise and upstream contribution workflows.

- **Dual licensing**: the same code is offered under two different licenses, and the recipient chooses one
- **Contributor License Agreement (CLA)**: a contributor grants the project additional rights to use the contribution
- **Copyright assignment**: the contributor transfers copyright ownership to another entity
- **Copyright ownership**: determines who can relicense the code and under what terms

Why this matters:

- A company may dual-license code to support both open source and commercial customers
- A project may require a CLA to simplify future relicensing or enforcement
- Copyright assignment can centralize control, but it may reduce contributor comfort
- Without clear rights, a project may not be able to relicense contributions later

Practical example:

- A vendor releases a library under GPL for community use and under a commercial license for proprietary products
- A contributor signs a CLA so the vendor can incorporate the patch into both licensing tracks
- If the contributor does not have the right to assign or license the code, the project may face compliance risk

**Distribution, packaging, and compliance in Linux environments**

Linux administrators and package maintainers often need to verify license obligations before shipping software in a distro, container image, or appliance.

Common compliance tasks include:

- Tracking package licenses in metadata
- Shipping license texts with binaries
- Preserving notices in documentation or about dialogs
- Keeping source code available when required
- Recording third-party components and versions
- Checking whether firmware blobs or drivers have separate terms

Useful commands and checks:

```bash
# Debian/Ubuntu package metadata
apt show bash | grep -i license

# RPM-based package metadata
rpm -qi bash | sed -n '/License/Ip'

# Inspect installed files for license documents
dpkg -L bash | grep -Ei 'license|copying|notice'

# Check a source package for SPDX identifiers
grep -RIn --exclude-dir=.git 'SPDX-License-Identifier' /path/to/source
```

For distro packaging, license compliance is not optional. A package may be technically functional but still unacceptable if it fails redistribution requirements.

**Public domain, Creative Commons, and software licensing distinctions**

Public domain and Creative Commons licenses are often mentioned in licensing discussions, but they are not interchangeable with software licenses.

- **Public domain**: the author waives rights or the work is no longer protected; this can be legally complex across jurisdictions
- **Creative Commons**: designed primarily for creative works such as text, images, and media
- **Software licenses**: written to address source code, binaries, linking, redistribution, and patent concerns

Why this matters in Linux package compliance:

- A package manager needs clear redistribution rights for source and binaries
- A CC BY license may be fine for documentation but not ideal for code
- Public domain dedication may be acceptable in some contexts, but many projects still prefer a formal software license for clarity

**Exam-style scenarios**

1. You package a GPL application for a distro repository.
   - You must preserve the GPL terms, include notices, and provide source or a valid source offer if you distribute binaries.

2. You build a proprietary app that dynamically links to an LGPL library.
   - You may keep the app proprietary if you satisfy the LGPL requirements, such as allowing relinking and preserving notices.

3. You statically link a GPL library into a closed-source appliance image.
   - This is high risk for compliance because static linking can create a derivative work obligation when distributed.

4. You load a third-party kernel module with a non-GPL-compatible license.
   - The module may still load, but it can be restricted from GPL-only symbols and may taint the kernel depending on the module and kernel state.

5. You find a project with only a `NOTICE` file and SPDX tags, not a full `LICENSE` file.
   - Do not assume the license is missing; inspect package metadata, source headers, and build files before concluding anything.

**How to think about licensing questions on the exam**

When you see a licensing question, classify the situation before answering.

- Identify whether the software is user space or kernel space
- Identify whether the license is copyleft or permissive
- Determine whether you are distributing the software
- Check whether the work is modified, linked, or merely used
- Look for obligations such as source disclosure, notices, or compatibility

A compact decision model:

```text
Is it distributed?
  -> No: usually fewer license obligations
  -> Yes: inspect license terms carefully

Is it copyleft?
  -> Yes: preserve freedoms and source obligations
  -> No: preserve notices and attribution as required

Is it kernel code or module code?
  -> Yes: check GPL compatibility and symbol restrictions
  -> No: user-space linking rules apply
```

**Practical command-line habits for license review**

You do not usually “detect” a license with one command. Instead, you combine metadata, file inspection, and source review.

```bash
# Find likely license files
find /path/to/project -type f \( -iname 'LICENSE*' -o -iname 'COPYING*' -o -iname 'NOTICE*' \)

# Search for SPDX tags in source headers
grep -RIn --exclude-dir=.git 'SPDX-License-Identifier' /path/to/project

# Inspect package metadata
rpm -qi package-name
dpkg -s package-name

# Review installed documentation
ls /usr/share/doc/package-name/
```

Use these checks to confirm, not to guess. Many projects use abbreviated notices, multiple licenses, or per-file licensing.

**Summary of high-value exam facts**

- Linux is the kernel; distributions combine the kernel with GNU tools and other software
- Linux is released in 1991 and relicensed under **GPLv2** in 1992
- The free software movement centers on the four freedoms
- Open source emphasizes practical development and the OSI definition
- Copyleft licenses impose redistribution obligations; permissive licenses are more flexible
- Static and dynamic linking can change compliance obligations
- Kernel modules have special licensing considerations, especially around GPL-compatible symbols
- Dual licensing, CLAs, and copyright ownership affect how projects can reuse and relicense code
- Public domain and Creative Commons are not automatically equivalent to software licenses
