**Environment variables: what they are and why they matter**

Environment variables are name/value pairs that influence how processes behave. You use them to control locale, search paths, editor preferences, proxy settings, and many application defaults without changing the application itself.

A shell keeps two related sets of variables:

- **Shell variables** exist only in the current shell unless exported.
- **Environment variables** are exported to child processes.

The key exam idea is scope:

- `VAR=value` sets a shell variable in the current shell.
- `export VAR=value` makes it part of the environment for child processes.
- `unset VAR` removes a variable from the current shell.
- A parent shell does not receive changes made by a child process.

Example:

```bash
EDITOR=vim
export EDITOR
echo "$EDITOR"
```

If you start a program from that shell, the program inherits `EDITOR=vim`. If you open a new terminal or a separate shell process, it only sees the variable if the startup files set it again.

**Common commands for managing variables**

Use the shell builtins directly when you need to inspect or change variables.

```bash
printenv
env
set
export
unset EDITOR
readonly APP_MODE=prod
```

Practical meanings:

- `printenv` shows environment variables.
- `env` prints the environment, and can also run a command with a modified environment.
- `set` shows shell variables, functions, and environment content in many shells.
- `export` marks variables for inheritance by child processes.
- `unset` removes a variable or function from the current shell.
- `readonly` makes a variable immutable in the current shell.

A useful pattern is to run a command with a temporary variable assignment:

```bash
HTTP_PROXY=http://proxy.example.com:8080 apt update
EDITOR=nano git commit
```

This syntax means “set the variable only for this command.” It does not permanently change the shell.

You can also use `env` for the same purpose:

```bash
env HTTP_PROXY=http://proxy.example.com:8080 apt update
```

That is especially useful when you want to start from a controlled environment or when you need to pass a variable only to one process.

> `unset` removes the variable from the current shell, but it does not affect already running child processes that inherited the old value.

**Inheritance, subshells, and process scope**

Environment variables flow downward through process creation. When a shell launches a command, the child process receives exported variables. When a shell launches a subshell, the subshell also inherits them, but changes stay local unless you export them again in that new shell.

Important distinctions:

- **Current shell**: your interactive session where you type commands.
- **Child process**: a program started by the shell.
- **Subshell**: a shell process created from another shell, often by parentheses or command substitution.

Examples:

```bash
VAR=parent
export VAR

( VAR=subshell; echo "$VAR" )
echo "$VAR"
```

The subshell prints `subshell`, but the parent shell still prints `parent`.

This matters when troubleshooting because a variable may appear inside a script or subshell but not in your interactive shell, or vice versa.

**Read-only variables and verification**

`readonly` is useful when you want to protect a value from accidental changes. Verify it by trying to modify or unset it.

```bash
readonly APP_ENV=production
echo "$APP_ENV"

APP_ENV=dev
unset APP_ENV
```

Expected behavior:

- Reassignment fails.
- `unset` fails for a readonly variable.
- The variable remains available until the shell exits.

You can inspect variable attributes with `declare -p` or `readonly -p` in Bash:

```bash
declare -p APP_ENV
readonly -p
```

**Temporary assignments and command scope**

The `VAR=value command` form is one of the most important shell patterns for exam questions and real administration. It sets the variable only for that command invocation.

Use it when you want to:

- Test a setting without changing your shell.
- Override a default for one run.
- Avoid polluting your environment.

Examples:

```bash
LANG=C sort file.txt
PATH=/opt/custom/bin:$PATH mytool
GIT_EDITOR=vim git commit
```

This is different from:

```bash
export GIT_EDITOR=vim
git commit
```

The first affects only one command. The second affects all child processes started afterward in that shell.

**Shell startup files: login shells vs non-login shells**

Shell initialization is a common source of confusion because different shell types read different files. For Bash, the startup path depends on whether the shell is login or non-login, interactive or non-interactive.

A quick review table helps:

| Shell type | Typical Bash files read | Notes |
|---|---|---|
| Login shell, interactive | `/etc/profile`, then one of `~/.bash_profile`, `~/.bash_login`, or `~/.profile` | System-wide first, then user-level login file |
| Non-login, interactive | `~/.bashrc` | Common for terminal tabs and new interactive shells |
| Non-interactive shell | `BASH_ENV` if set | Often used by scripts, but can be risky |

For POSIX `sh`-style shells, the analogous variable is `ENV`, which can point to a file to source for interactive shells.

**System-wide Bash initialization order**

On many Linux systems, Bash login shells read `/etc/profile` first. That file may then source scripts in `/etc/profile.d/*.sh` to modularize system-wide settings.

Typical flow:

1. Bash starts as a login shell.
2. It reads `/etc/profile`.
3. `/etc/profile` may iterate over `/etc/profile.d/*.sh`.
4. The shell then reads the first user file it finds:
   - `~/.bash_profile`
   - or `~/.bash_login`
   - or `~/.profile`

This order matters because system administrators often place shared defaults in `/etc/profile` or `/etc/profile.d/*.sh`, while user-specific login settings belong in the home directory.

A common pattern is for the user login file to source `~/.bashrc` so interactive settings are available in login shells too:

```bash
# ~/.bash_profile
if [ -f ~/.bashrc ]; then
    . ~/.bashrc
fi
```

Or:

```bash
# ~/.profile
if [ -f "$HOME/.bashrc" ]; then
    . "$HOME/.bashrc"
fi
```

Why this matters:

- Login shells often start from a display manager, SSH session, or console login.
- Non-login interactive shells often start from a terminal emulator.
- If you put interactive aliases and prompt settings only in `~/.bashrc`, they may not appear in login shells unless you source `~/.bashrc` from `~/.bash_profile` or `~/.profile`.

> If you edit the wrong startup file, the variable may seem “missing” even though the shell is working correctly.

**Bash startup file comparison**

| File | Scope | Common use |
|---|---|---|
| `/etc/profile` | System-wide login shells | Global environment defaults |
| `/etc/profile.d/*.sh` | System-wide modular login settings | Package-managed environment snippets |
| `~/.bash_profile` | User login shells | User login environment, often sources `~/.bashrc` |
| `~/.bash_login` | User login shells fallback | Used only if `~/.bash_profile` is absent |
| `~/.profile` | User login shells fallback, POSIX-friendly | Portable login settings |
| `~/.bashrc` | User interactive non-login shells | Aliases, prompt, functions, interactive exports |

**Practical Bash configuration pattern**

A reliable setup separates login and interactive concerns:

```bash
# ~/.bash_profile
# Login-shell settings
export PATH="$HOME/bin:$PATH"
export EDITOR=vim

if [ -f ~/.bashrc ]; then
    . ~/.bashrc
fi
```

```bash
# ~/.bashrc
# Interactive-shell settings
alias ll='ls -alF'
alias gs='git status'
export HISTCONTROL=ignoreboth
```

This pattern prevents duplication and keeps interactive customizations available in both login and terminal shells.

**Non-interactive shells: BASH_ENV and why it can be risky**

Non-interactive Bash shells do not read `~/.bashrc` by default. Instead, Bash can source the file named by `BASH_ENV` for non-interactive shells.

Example:

```bash
export BASH_ENV="$HOME/.bash_env"
```

Then a script or non-interactive Bash invocation may source that file before running commands.

Use cases:

- Standardizing environment setup for scripts.
- Injecting a controlled set of variables for automation.
- Ensuring consistent behavior in CI/CD pipelines.

Risks:

- `BASH_ENV` can unexpectedly alter script behavior.
- It may slow down scripts if the file is large.
- It can create hard-to-debug side effects if it exports aliases, functions, or overrides commands.

For exam purposes, remember:

- `BASH_ENV` affects non-interactive Bash shells.
- It is not the same as `~/.bashrc`.
- It can be powerful, but it is easy to misuse.

**POSIX `sh` and the `ENV` variable**

For POSIX-style shells, `ENV` is the comparable startup variable. It is commonly used by shells such as `sh`-compatible interactive shells to source a file at startup.

Example:

```bash
export ENV="$HOME/.shenv"
```

Use this when comparing shell initialization across shell families. The exact behavior varies by shell implementation, so avoid assuming Bash and `sh` behave identically.

**Troubleshooting: variable not appearing where you expect**

A common real-world problem is editing `~/.bashrc` and then logging in over SSH, only to find the variable missing. That happens because the login shell may read `~/.bash_profile` or `~/.profile`, not `~/.bashrc` directly.

Troubleshooting workflow:

1. Check the shell type:
   ```bash
   echo "$0"
   shopt -q login_shell && echo login || echo non-login
   ```
2. Check which file you edited.
3. Confirm the file is sourced by the relevant startup file.
4. Open a new session, not just a new command.
5. Verify the variable:
   ```bash
   printenv MYVAR
   echo "$MYVAR"
   ```

Example fix:

```bash
# ~/.bash_profile
if [ -f ~/.bashrc ]; then
    . ~/.bashrc
fi
```

If the variable still does not appear, confirm that the shell is actually Bash and not another shell such as `zsh` or `dash`.

**Real-world examples**

Environment variables are common in administration workflows.

Proxy settings for package managers:

```bash
export http_proxy=http://proxy.example.com:8080
export https_proxy=http://proxy.example.com:8080
sudo -E apt update
```

`sudo -E` preserves selected environment variables if policy allows it. This is useful when a package manager or tool must reach the network through a proxy.

Editor settings for Git and `sudo` workflows:

```bash
export EDITOR=vim
git commit
sudo visudo
```

Many tools honor `EDITOR` or `VISUAL`. Setting them in your shell makes command-line editing more predictable.

Path customization for local tools:

```bash
export PATH="$HOME/.local/bin:$PATH"
```

This is a common way to make user-installed binaries available without using absolute paths.

**System administration and security considerations**

Environment variables are convenient, but they are also part of the attack surface.

Be careful with:

- `PATH` manipulation, which can cause the shell to run the wrong executable.
- `LD_LIBRARY_PATH` and similar loader variables, which can affect dynamic linking.
- `BASH_ENV` and `ENV`, which can inject startup behavior into scripts or shells.
- Passing sensitive values in the environment, because child processes can inherit them.

Best practices:

- Keep startup files small and readable.
- Use absolute paths in scripts when possible.
- Export only what child processes need.
- Avoid placing secrets in shell startup files unless you understand the exposure.
- Prefer per-command assignments for temporary overrides.

**Quick exam review**

- `export` sends variables to child processes.
- `unset` removes a variable from the current shell.
- `VAR=value command` and `env VAR=value command` set a variable only for one command.
- Bash login shells read `/etc/profile` first, then user login files.
- `/etc/profile` often sources `/etc/profile.d/*.sh`.
- `~/.bash_profile` commonly sources `~/.bashrc` for login shells.
- `~/.bashrc` is typically for interactive non-login shells.
- `BASH_ENV` affects non-interactive Bash shells and can be risky.
- `ENV` is the POSIX `sh`-style counterpart for shell initialization.
