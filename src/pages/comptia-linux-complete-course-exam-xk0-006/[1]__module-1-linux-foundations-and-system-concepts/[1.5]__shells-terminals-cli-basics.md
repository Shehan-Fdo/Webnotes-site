## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Linux CLI fundamentals**

The command-line interface (CLI) is the text-based way you interact with Linux. You type commands, the shell interprets them, and the kernel performs the requested work through system calls. In practice, the CLI is fast, scriptable, and precise, which is why it remains central to administration, troubleshooting, and automation.

A useful mental model is:

- **Terminal emulator**: the graphical app you open, such as GNOME Terminal, Konsole, or xterm.
- **TTY**: a text terminal device, including virtual consoles like `<kbd>Ctrl</kbd> + <kbd>Alt</kbd> + <kbd>F3</kbd>`.
- **Pseudo-terminal (pty)**: the software-backed terminal pair used by terminal emulators and SSH sessions.
- **Shell**: the command interpreter, such as Bash, Zsh, or Fish.

The terminal is the interface, while the shell is the program that reads and executes your commands. A shell session usually runs inside a terminal or pty, but those are not the same thing.

**Terminal types and session behavior**

Linux supports several ways to access the CLI, and each has slightly different behavior:

- **Virtual consoles** provide direct text logins on the local machine.
- **Terminal emulators** provide a graphical window connected to a pty.
- **Remote sessions** such as SSH create a network-backed pty and usually start a login shell by default, depending on server and shell configuration.
- **Serial consoles** are common in servers, embedded systems, and recovery scenarios.

Understanding the distinction matters when you troubleshoot startup files, environment variables, or job control. For example, a graphical terminal may start an interactive non-login shell, while an SSH login typically starts a login shell that reads login startup files.

**Shell prompt basics**

The prompt tells you the shell is ready for input. It often includes the username, host, and current directory, but the exact format depends on shell configuration.

Common prompt variables in Bash include:

- **`PS1`**: primary prompt for interactive commands.
- **`PS2`**: secondary prompt for continued input, such as an unfinished command or multiline statement.
- **`PS4`**: used during shell tracing with `set -x`.

Example prompt customization:

```bash
export PS1='\u@\h:\w\$ '
```

This prompt shows:

- `\u` for username
- `\h` for hostname
- `\w` for the full working directory
- `\$` for `#` if root, otherwise `$`

A secondary prompt appears when the shell expects more input:

```bash
echo "Hello
```

The shell typically shows `PS2` until you close the quote.

**Shells and common command interpreters**

A shell is both an interactive environment and a scripting language. Different shells have different syntax details, but many core concepts are shared.

Common shells include:

- **Bash**: the default on many Linux systems and the most exam-relevant shell.
- **Zsh**: feature-rich interactive shell with advanced completion and customization.
- **Dash**: lightweight POSIX-compatible shell often used for system scripts.
- **Fish**: user-friendly interactive shell with different syntax and behavior.

Bash is especially important because many Linux systems use it for interactive logins and because its startup files are frequently tested.

**Login shells, interactive shells, and startup files**

A shell’s behavior depends on whether it is:

- **Login** or **non-login**
- **Interactive** or **non-interactive**

A login shell reads files intended for session initialization. An interactive shell reads files intended for interactive use. Bash chooses startup files based on both conditions.

For Bash, the common selection order is:

- **Login shell**
  - Reads `/etc/profile`
  - Then reads the first readable file in this order:
    - `~/.bash_profile`
    - `~/.bash_login`
    - `~/.profile`
- **Interactive non-login shell**
  - Reads `~/.bashrc`
- **Non-interactive shell**
  - Usually reads the file named by the `BASH_ENV` environment variable, if set

This order is important because only the first of `~/.bash_profile`, `~/.bash_login`, or `~/.profile` is used for a Bash login shell. A common practice is to source `~/.bashrc` from `~/.bash_profile` so interactive settings are available in login sessions too.

Example `~/.bash_profile`:

```bash
# ~/.bash_profile
if [ -f ~/.bashrc ]; then
  . ~/.bashrc
fi

export PATH="$HOME/bin:$PATH"
```

Example `~/.bashrc`:

```bash
# ~/.bashrc
alias ll='ls -alF'
alias grep='grep --color=auto'
export EDITOR=vim
```

> Be careful not to place interactive-only commands in files that run for non-interactive shells, or scripts and remote commands may fail unexpectedly.

**Environment variables and shell variables**

Environment variables are inherited by child processes. Shell variables exist only in the current shell unless exported.

Common examples include:

- **`PATH`**: directories searched for commands
- **`HOME`**: user’s home directory
- **`SHELL`**: user’s default shell
- **`USER`** or **`LOGNAME`**: current username
- **`EDITOR`**: preferred text editor
- **`LANG`** and **`LC_*`**: locale settings

View variables with:

```bash
printenv PATH
echo "$HOME"
env | sort
```

Set a variable for the current shell:

```bash
MYVAR="test value"
```

Export it to child processes:

```bash
export MYVAR="test value"
```

Persist environment variables by placing exports in startup files:

```bash
# ~/.bashrc or ~/.profile
export PATH="$HOME/bin:$PATH"
export EDITOR=nano
```

Use `~/.profile` for login-session environment settings and `~/.bashrc` for interactive shell behavior. That separation helps avoid alias and prompt code running in non-interactive contexts.

**Aliases, functions, and command lookup**

Aliases are shortcuts for commands. They are useful for convenience, but they can also confuse troubleshooting if they override expected behavior.

Examples:

```bash
alias ll='ls -alF'
alias rm='rm -i'
```

List aliases with:

```bash
alias
```

Remove one alias with:

```bash
unalias ll
```

Shell functions are more powerful than aliases and can accept arguments:

```bash
mkcd() {
  mkdir -p "$1" && cd "$1"
}
```

Command lookup generally follows this order in Bash:

1. Alias
2. Shell function
3. Builtin
4. Hashed command path
5. `PATH` search

Check what a command resolves to with:

```bash
type ls
type -a python
which bash
```

`type` is often more informative than `which` because it can identify aliases, functions, builtins, and files.

**PATH and command resolution**

`PATH` is a colon-separated list of directories searched when you type a command without a full path. If `PATH` is broken, commands may appear missing even though they are installed.

Example:

```bash
echo "$PATH"
export PATH="/usr/local/bin:/usr/bin:/bin:$HOME/bin"
```

Troubleshooting tips:

- Use an absolute path such as `/usr/bin/ls` to confirm the binary exists.
- Check whether `PATH` was overwritten instead of appended.
- Verify startup files for accidental `export PATH=/some/dir` lines that remove standard directories.
- Use `command -v name` to confirm resolution.

A broken `PATH` is a common cause of “command not found” errors after editing shell startup files.

**Input, output, and redirection**

Linux treats standard streams consistently:

- **stdin**: standard input
- **stdout**: standard output
- **stderr**: standard error

Redirection lets you control where data goes.

Examples:

```bash
command > output.txt
command 2> errors.txt
command >> output.txt
command < input.txt
```

Redirect both stdout and stderr:

```bash
command > all.log 2>&1
```

Use pipes to send stdout from one command into another:

```bash
ps aux | grep sshd
```

A practical example with `tar`:

```bash
tar -czf backup.tgz /etc 2>backup.err
```

`tar` writes the archive to the file named by `-f`; errors go to `stderr`. If you want to capture command output, make sure the command actually writes to stdout.

**Quoting, escaping, and expansion**

The shell expands text before running a command. Understanding expansion prevents many mistakes.

Common forms:

- **Single quotes** preserve literal text.
- **Double quotes** preserve most text but still allow variable and command expansion.
- **Backslash** escapes a single character.

Examples:

```bash
echo '$HOME'
echo "$HOME"
echo file\ name.txt
```

Useful expansions include:

- **Variable expansion**: `$HOME`
- **Command substitution**: `$(date)`
- **Globbing**: `*.log`
- **Brace expansion**: `file{1,2,3}.txt`

Example:

```bash
mkdir -p /tmp/demo/{in,out,log}
```

Use quotes whenever a value may contain spaces, tabs, or wildcard characters.

**Job control**

Job control lets you manage foreground and background processes from an interactive shell. It is especially useful when a command takes a long time or you need to suspend work temporarily.

Common controls:

- `<kbd>Ctrl</kbd> + <kbd>C</kbd>` sends **SIGINT** (signal interrupt) to stop a foreground process.
- `<kbd>Ctrl</kbd> + <kbd>Z</kbd>` sends **SIGTSTP** to suspend a foreground process.
- `bg` resumes a suspended job in the background.
- `fg` brings a job to the foreground.
- `jobs` lists current jobs.

Example workflow:

```bash
sleep 300
# press Ctrl+Z
jobs
bg %1
fg %1
```

Start a job in the background:

```bash
long_task &
```

Useful signal examples:

- **SIGTERM** (`15`): polite request to terminate; use first for normal shutdown.
- **SIGKILL** (`9`): forceful termination; use only when a process ignores SIGTERM.
- **SIGHUP** (`1`): historically hangup; often used to reload or signal daemons and to notify sessions that the terminal disconnected.
- **SIGINT** (`2`): interrupt from keyboard.
- **SIGSTOP** (`19` on many Linux systems): uncatchable stop signal.

Send signals with `kill`:

```bash
kill -TERM 1234
kill -9 1234
kill -HUP 1234
```

Use `pkill` or `killall` when you want to target by name:

```bash
pkill -TERM nginx
killall -HUP sshd
```

> Use `SIGKILL` only after `SIGTERM` fails. It prevents cleanup, so processes cannot close files or release resources gracefully.

**Terminal control and session management**

A terminal session can be detached or preserved with tools such as `screen` or `tmux`. These tools are useful over SSH because they let you keep work running after disconnecting.

Example `tmux` workflow:

```bash
tmux new -s work
# detach with Ctrl+b then d
tmux attach -t work
```

This is especially helpful for long-running builds, maintenance tasks, and remote administration.

**Shell initialization and troubleshooting**

When a shell behaves unexpectedly, check startup files first. Problems often come from syntax errors, bad aliases, or overwritten variables.

Common symptoms and causes:

- **Login failure after editing startup files**
  - A syntax error in `~/.bash_profile`, `~/.profile`, or `/etc/profile`
- **Missing commands**
  - `PATH` was overwritten or not exported
- **Broken prompt**
  - `PS1` contains invalid escape sequences or unclosed quotes
- **Unexpected command behavior**
  - An alias or function shadows the real command
- **SSH session looks different from local terminal**
  - Different startup files are being read, or shell initialization differs between login and interactive shells

Useful recovery commands:

```bash
bash --noprofile --norc
bash -x ~/.bashrc
env -i bash --noprofile --norc
```

These commands help you isolate whether the problem comes from environment variables, startup files, or shell configuration.

If you suspect a bad alias or function, inspect the command resolution:

```bash
type -a ls
alias
declare -f mkcd
```

If `PATH` is broken, temporarily restore a minimal safe value:

```bash
export PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
```

**Practical configuration examples**

Persist an alias and environment variable in `~/.bashrc`:

```bash
alias ll='ls -alF'
export HISTSIZE=5000
export HISTFILESIZE=10000
```

Persist a login-session `PATH` change in `~/.profile`:

```bash
export PATH="$HOME/bin:$PATH"
```

If you want interactive Bash sessions to share the same settings, source `~/.bashrc` from `~/.bash_profile`:

```bash
# ~/.bash_profile
[ -f ~/.bashrc ] && . ~/.bashrc
```

This pattern keeps login and interactive behavior consistent without duplicating configuration.

**Exam-focused distinctions**

Know these differences clearly:

- **Terminal emulator** is the app window; **shell** is the command interpreter.
- **TTY** is a text terminal device; **pty** is a software terminal used by emulators and SSH.
- **Login shell** reads login startup files; **interactive non-login shell** reads `~/.bashrc`.
- **Environment variables** are inherited; **shell variables** are not unless exported.
- **Alias** is a simple substitution; **function** can contain logic and parameters.
- **SIGTERM** is graceful; **SIGKILL** is immediate and non-catchable.

These distinctions show up in troubleshooting, configuration, and command behavior. If you can explain them clearly, you can usually reason through most CLI questions on the exam.

**Quick reference commands**

```bash
# Identify shell and prompt-related state
echo "$SHELL"
echo "$0"
echo "$PS1"

# Inspect command resolution
type -a ls
command -v python3

# View and set environment
printenv | sort
export EDITOR=vim

# Manage jobs
jobs
bg %1
fg %1

# Send signals
kill -TERM 1234
kill -9 1234

# Start a clean shell for troubleshooting
bash --noprofile --norc
```

**Common troubleshooting checklist**

1. Confirm whether you are in a login shell or interactive shell.
2. Check `PATH`, aliases, and functions before assuming a package is missing.
3. Review `~/.bash_profile`, `~/.bash_login`, `~/.profile`, and `~/.bashrc` for syntax errors.
4. Test with a clean shell using `bash --noprofile --norc`.
5. Use `type`, `command -v`, and `echo "$PATH"` to isolate resolution problems.
6. Use `SIGTERM` before `SIGKILL` when stopping processes.
