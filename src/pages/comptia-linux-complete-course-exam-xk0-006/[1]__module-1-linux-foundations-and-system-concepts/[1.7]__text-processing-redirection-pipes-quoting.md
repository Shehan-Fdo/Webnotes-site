## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Shell parsing model**

The shell does more than run commands. It first parses your line, expands patterns and variables, applies redirection, and then executes the command. Understanding that order helps you predict why a command behaves differently when quoted, redirected, or placed in a pipeline.

A useful mental model is:

1. Read the command line.
2. Perform expansions.
3. Set up redirections and pipes.
4. Execute the command.

That order explains many exam questions about why a wildcard expands, why `$(...)` output splits into multiple words, or why `2>&1` must appear in the right place.

- **Word splitting** happens after parameter and command substitution unless you quote the result.
- **Globbing** expands wildcard patterns like `*` and `?` after word splitting.
- **Redirection** changes where standard streams go before the command runs.

**Quoting and escaping**

Quoting controls how the shell treats special characters. It is one of the most tested topics because it affects variables, wildcards, spaces, and command substitution.

- **Single quotes** preserve literal text.
  - Everything inside `'...'` is taken literally.
  - Variables, command substitution, and backslashes do not expand inside single quotes.
  - Example: `echo '$HOME * $(date)'` prints the text exactly.
- **Double quotes** preserve most text but still allow expansion.
  - Variables, command substitution, and arithmetic expansion still work.
  - Wildcards do not expand inside double quotes.
  - Example: `echo "$HOME"` expands the variable, but `echo "*.txt"` prints the pattern literally.
- **Backslash escaping** protects a single character.
  - `\ ` preserves a space.
  - `\*` prevents glob expansion.
  - `\$HOME` prints `$HOME` literally.
  - Backslash is narrower than quotes because it escapes only the next character, not a whole string.

> A common exam trap: `echo $(cmd)` is not the same as `echo "$(cmd)"`. The unquoted form can split and glob the output.

**Practical quoting examples**

Use quoting to protect filenames, variables, and patterns that contain spaces or shell metacharacters.

```bash
# Unsafe if the filename contains spaces
cat $file

# Safe for spaces and most special characters
cat "$file"

# Literal wildcard, not pathname expansion
echo "*.log"

# Escape one special character
rm file\ name.txt
```

- Use **single quotes** when you want literal text.
- Use **double quotes** when you need expansion but want to preserve spaces.
- Use **backslash** for one-off escaping in the middle of a token.

**Brace expansion and shell parsing**

Brace expansion is a shell feature that generates multiple strings before most other expansions. It is not the same as globbing.

- Example: `echo file{1,2,3}.txt` becomes `file1.txt file2.txt file3.txt`.
- Example: `mkdir -p /tmp/{in,out,log}` creates three directories.
- Brace expansion does not inspect the filesystem.
- Quoting prevents brace expansion if the braces are inside quotes.
- Redirection operators are parsed separately, so `echo hi > file{1,2}` is not a valid way to redirect to multiple files.

Brace expansion is useful for quick generation, but do not confuse it with wildcard matching.

**Pathname expansion and globbing**

Pathname expansion, also called globbing, matches filenames in the current directory or a specified path. The shell expands the pattern before the command runs.

Common patterns:

- `*` matches zero or more characters.
- `?` matches exactly one character.
- `[abc]` matches one character from a set.
- `[a-z]` matches a range.
- `[!0-9]` or `[^0-9]` matches one character not in the set, depending on shell behavior.

Examples:

```bash
ls *.log
rm report?.txt
cp file[0-9].csv /backup/
```

Important behavior:

- If a glob matches nothing, many shells leave it unchanged by default.
- Quoting prevents pathname expansion.
  - `echo *.txt` expands to matching files.
  - `echo "*.txt"` prints the literal pattern.
- Backslash also prevents expansion for a single character.
  - `echo \*.txt` prints `*.txt`.

> If you need literal glob characters in a filename, quote them when referencing the file.

**Command substitution pitfalls**

Command substitution runs a command and replaces it with its output. The shell then performs word splitting and globbing on the result unless you quote it.

- `files=$(ls)` is fragile because `ls` output is formatted for humans, not scripts.
- `echo $(printf 'a b\nc*d\n')` can become multiple words and may expand `c*d` as a glob if matching files exist.
- `echo "$(printf 'a b\nc*d\n')"` preserves the output as one argument, though embedded newlines remain.

Safer patterns:

```bash
# Prefer direct command use over parsing ls
for f in *.txt; do
  printf '%s\n' "$f"
done

# Quote command substitution when you need one string
name="$(hostname)"
printf 'Host: %s\n' "$name"
```

- Use **command substitution** for short, controlled output.
- Quote `$(...)` unless you intentionally want splitting.
- Avoid parsing `ls` output in scripts.
- Prefer arrays or null-delimited tools when handling arbitrary filenames.

**Redirection basics**

Linux commands use three standard streams:

- **stdin**: standard input, file descriptor 0
- **stdout**: standard output, file descriptor 1
- **stderr**: standard error, file descriptor 2

Redirection changes where these streams go.

Common forms:

```bash
command > out.txt
command 1> out.txt
command 2> errors.txt
command >> append.txt
command < input.txt
```

- `>` overwrites a file.
- `>>` appends to a file.
- `<` reads stdin from a file.
- `2>` redirects errors.
- `>/dev/null` discards standard output.
- `2>/dev/null` discards standard error.

Examples:

```bash
grep root /etc/passwd > matches.txt
grep root /etc/passwd 2> errors.txt
command >/dev/null
command 2>/dev/null
```

**Combining redirections**

You often need to redirect both output and errors, or send them to different places.

Examples:

```bash
# Send stdout to a file, stderr to another file
command >out.txt 2>err.txt

# Append stdout, overwrite stderr
command >>out.txt 2>err.txt

# Send both stdout and stderr to the same file
command >all.txt 2>&1
```

Order matters:

```bash
# stderr goes to the original stdout destination, then stdout goes to file
command 2>&1 >all.txt
```

In the previous example, stderr does not end up in `all.txt` because `2>&1` happens before stdout is redirected. The usual portable pattern is:

```bash
command >all.txt 2>&1
```

> Redirection order is a frequent exam topic. Read it left to right.

**Descriptor duplication and closing**

File descriptors can be duplicated or closed explicitly.

- `2>&1` duplicates stderr to wherever stdout currently points.
- `1>&2` sends stdout to stderr.
- `2>/dev/null` discards errors.
- `>/dev/null` discards normal output.
- `2>&-` closes stderr.
- `1>&-` closes stdout.

Examples:

```bash
# Silence normal output but keep errors
command >/dev/null

# Silence errors but keep normal output
command 2>/dev/null

# Silence everything
command >/dev/null 2>&1
```

Use these patterns when you want to suppress noisy commands in scripts or cron jobs.

**Input redirection and stdin in pipelines**

Some commands read from stdin by default, while others expect filenames. Input redirection is useful when a command reads from a file instead of a pipe.

Examples:

```bash
sort < unsorted.txt
wc -l < access.log
tr '[:lower:]' '[:upper:]' < input.txt
```

A pipeline connects stdout of one command to stdin of the next:

```bash
cat access.log | grep ERROR
```

But many commands do not need `cat` in front of them because they can read files directly:

```bash
grep ERROR access.log
sort access.log
wc -l access.log
```

Use input redirection when:

- A command expects stdin and you want to feed it a file.
- You want to keep a command in a pipeline but source input from a file.
- You need to combine file input with other redirections.

Example with multiple redirections:

```bash
sort < unsorted.txt > sorted.txt 2> sort.err
```

This reads from `unsorted.txt`, writes sorted output to `sorted.txt`, and sends errors to `sort.err`.

**Pipes**

A pipe `|` sends stdout from one command directly into stdin of another command. Pipes are ideal for chaining text-processing tools.

Examples:

```bash
ps aux | grep sshd
dmesg | less
journalctl -u sshd | tail
```

Key points:

- Pipes connect **stdout** only, not stderr.
- If you want errors in the pipeline, redirect them explicitly.
- A command in the middle of a pipeline should usually read from stdin and write to stdout.

Example:

```bash
grep -i error app.log | sort | uniq -c | sort -nr
```

This pipeline finds error lines, sorts them, counts duplicates, and sorts by count descending.

**Text-processing commands**

These commands are commonly used with pipes and redirection.

- **`cat`**: concatenate and display files.
  - Useful for quick inspection, but often unnecessary in pipelines.
- **`grep`**: search text by pattern.
  - `grep -E` uses extended regular expressions.
  - `grep -F` treats the pattern as a fixed string.
  - `grep -i` ignores case.
  - `grep -v` inverts the match.
- **`sort`**: sort lines.
  - `sort -n` sorts numerically.
  - `sort -r` reverses order.
  - `sort -k` sorts by a field.
- **`uniq`**: collapse adjacent duplicate lines.
  - `uniq -c` counts repeated lines.
  - `uniq -d` shows only duplicates.
  - `uniq` usually requires sorted input to count all duplicates correctly.
- **`tr`**: translate or delete characters.
  - `tr '[:lower:]' '[:upper:]'` uppercases text.
  - `tr -d '\r'` removes carriage returns.
- **`cut`**: extract columns or fields.
- **`wc`**: count lines, words, and bytes.
- **`head`** and **`tail`**: show the beginning or end of input.
- **`tee`**: copy stdin to stdout and a file at the same time.
- **`awk`**: field-oriented text processing.
- **`sed`**: stream editing.

Examples:

```bash
grep -E 'error|fail' app.log
grep -F '192.168.1.10' access.log
sort -n scores.txt
sort -k2,2 -n data.txt
uniq -c words.txt
tr -d '\r' < windows.txt > unix.txt
```

**`grep` patterns and when to use them**

Choose the right `grep` mode for the job.

- `grep -E` is useful for alternation and extended regex syntax.
  - Example: `grep -E 'warn|error|fail' syslog`
- `grep -F` is faster and safer for literal strings.
  - Example: `grep -F 'a.b[c]' file.txt`
- `grep -i` ignores case.
- `grep -n` shows line numbers.
- `grep -r` searches recursively.

Practical examples:

```bash
grep -E '^(root|admin):' /etc/passwd
grep -F 'GET /index.html' access.log
grep -in 'failed' auth.log
```

Use `-F` when the search term contains regex metacharacters that you want treated literally.

**`sort`, `uniq`, and counting duplicates**

`uniq` only compares adjacent lines, so you usually sort first.

```bash
sort names.txt | uniq
sort names.txt | uniq -c
sort names.txt | uniq -d
```

Useful combinations:

```bash
# Count repeated IP addresses
cut -d' ' -f1 access.log | sort | uniq -c | sort -nr

# Sort by the second field numerically
sort -k2,2 -n scores.txt
```

- `sort -n` compares numeric values, not text.
- `sort -r` reverses the sort order.
- `sort -k` lets you choose the field or key range.

**`awk` field and record handling**

`awk` is powerful for column-based processing. It splits input into records and fields.

- **Record separator** is usually a newline.
- **Field separator** defaults to whitespace.
- `-F` sets the field separator.
- `BEGIN` runs before input is processed.
- `END` runs after input is processed.

Examples:

```bash
awk '{print $1, $3}' /etc/passwd
awk -F: '{print $1, $7}' /etc/passwd
awk -F, '{sum += $3} END {print sum}' data.csv
```

More advanced example:

```bash
awk -F: 'BEGIN {OFS=","} {print $1, $3, $7}' /etc/passwd
```

- `FS` is the input field separator.
- `OFS` is the output field separator.
- `RS` is the record separator.
- `ORS` is the output record separator.

If you need to process nonstandard record formats, `awk` can change `RS`, but that is less common in basic exam scenarios.

**`xargs` and whitespace-safe patterns**

`xargs` builds command lines from stdin. It is useful when a command does not accept stdin directly.

Basic example:

```bash
find . -name '*.log' | xargs grep -n 'ERROR'
```

However, plain `xargs` splits on whitespace, which is unsafe for filenames with spaces, tabs, or newlines.

Safer patterns:

```bash
find . -name '*.log' -print0 | xargs -0 grep -n 'ERROR'
find . -type f -print0 | xargs -0 rm -f
```

- Use `-0` with `find -print0` when filenames may contain spaces or special characters.
- Use `xargs -r` on GNU systems to avoid running the command with no input.
- Prefer `xargs` when you need to batch many arguments efficiently.

> For arbitrary filenames, `-print0` and `xargs -0` are the standard safe pairing.

**Pipelines versus loops**

Pipelines are concise, but loops are sometimes clearer when you need per-file logic.

```bash
find /var/log -name '*.log' -print0 |
  xargs -0 grep -H 'failed'
```

Versus:

```bash
for f in /var/log/*.log; do
  grep -H 'failed' "$f"
done
```

Use a loop when:

- You need to preserve filenames exactly.
- You need conditional logic per file.
- You want to avoid `xargs` argument limits or quoting issues.

**Process substitution**

Process substitution lets a command read from a temporary file-like path produced by another command.

Examples:

```bash
diff <(sort file1.txt) <(sort file2.txt)
comm <(sort a.txt) <(sort b.txt)
```

This is convenient, but it is **shell-specific** and not part of POSIX (Portable Operating System Interface). Do not assume it works in every shell or on every system.

- Use it in Bash and similar shells when available.
- Avoid it in scripts that must be strictly portable.

**Practical redirection and pipeline scenarios**

1. Compare two command outputs without creating files:
   ```bash
   diff <(grep -v '^#' config1) <(grep -v '^#' config2)
   ```

2. Save normal output and suppress errors:
   ```bash
   command >output.txt 2>/dev/null
   ```

3. Save both output and errors separately:
   ```bash
   command >output.txt 2>errors.txt
   ```

4. Append output while keeping errors on screen:
   ```bash
   command >>output.txt
   ```

5. Feed a file into a command that reads stdin:
   ```bash
   wc -l < /var/log/messages
   ```

6. Count repeated values:
   ```bash
   cut -d: -f1 /etc/passwd | sort | uniq -c | sort -nr
   ```

**Exam-oriented quoting and expansion examples**

- `echo $HOME` expands the variable and then word splitting may occur if the value contains spaces.
- `echo "$HOME"` preserves the value as one argument.
- `echo $(printf 'a b\n')` can become two words.
- `echo "$(printf 'a b\n')"` preserves the output as one argument.
- `echo *.txt` expands matching filenames.
- `echo "*.txt"` prints the literal pattern.
- `echo \*.txt` also prints the literal pattern.
- `echo file{1,2}.txt` becomes two words from brace expansion.

A good rule is:

- Quote variables and command substitutions unless you explicitly need splitting.
- Use globs unquoted when you want filename matching.
- Use backslash for a single protected character.
- Use single quotes for literal strings.

**Common mistakes to avoid**

- Parsing `ls` output in scripts.
- Forgetting that `uniq` needs sorted input for reliable counting.
- Assuming pipes carry stderr automatically.
- Using `xargs` without `-0` on filenames that may contain spaces.
- Expecting `2>&1 >file` to behave like `>file 2>&1`.
- Quoting a glob when you actually want pathname expansion.
- Forgetting that `$(...)` output can split and glob if unquoted.

**Quick reference**

| Topic | Key behavior | Example |
|---|---|---|
| Single quotes | Literal text | `'*.txt'` |
| Double quotes | Preserve spaces, allow expansion | `"$HOME"` |
| Backslash | Escape one character | `\*` |
| Globbing | Filename matching | `*.log` |
| Command substitution | Replace with command output | `$(date)` |
| stdout | Normal output | `>` |
| stderr | Error output | `2>` |
| Pipe | stdout to stdin | `cmd1 \| cmd2` |
| `uniq -c` | Count adjacent duplicates | `sort \| uniq -c` |
| `grep -E` | Extended regex | `grep -E 'a|b'` |
| `grep -F` | Fixed string | `grep -F 'a.b'` |
| `xargs -0` | Null-delimited safe input | `find ... -print0 \| xargs -0` |

**Study takeaway**

Shell quoting and redirection are not separate topics; they are part of the same parsing pipeline. If you know when the shell expands text, when it redirects streams, and when it passes literal characters through, you can predict command behavior and avoid the most common Linux+ exam traps.
