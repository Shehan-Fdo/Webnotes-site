## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Kernel vs user space**

Linux splits execution into **user space** and **kernel space** to protect the system and control access to hardware and privileged operations. User applications run with limited privileges, while the kernel runs with full control over memory, devices, scheduling, and security enforcement.

This separation is enforced by the CPU through **privilege rings** or equivalent hardware privilege levels. On x86 systems, the common mapping is:

- **Ring 0**: kernel mode, highest privilege
- **Ring 3**: user mode, least privilege

Linux maps most application code to user mode and kernel code to ring 0. When a program needs a privileged service, it does not directly touch hardware or kernel memory. Instead, it requests help from the kernel through a controlled transition.

> The exact hardware implementation can vary by architecture, but the core idea is the same: user code is restricted, and the kernel mediates sensitive operations.

**Why the boundary matters**

The kernel/user boundary is central to stability and security:

- A user process cannot arbitrarily read another process’s memory.
- A user process cannot directly program devices or change page tables.
- The kernel can validate requests before acting on them.
- Faults in user programs usually stay contained instead of crashing the whole system.

This boundary is also where Linux enforces permissions, capabilities, and mandatory access control (MAC). File permissions decide whether a user can open a file, capabilities split root power into smaller privileges, and MAC frameworks such as SELinux or AppArmor can further restrict what a process may do even if traditional permissions allow it.

**Execution model overview**

A Linux program becomes a process when the kernel loads it into memory and creates the execution context needed to run it. The execution model is not just “a program runs”; it is a lifecycle that includes creation, scheduling, state changes, waiting, and termination.

A typical lifecycle looks like this:

1. A program file exists on disk, such as `/bin/ls` or a custom binary.
2. The kernel creates a **process image** when the program is executed.
3. The process receives a PID (process ID), memory mappings, open file descriptors, credentials, and a scheduling context.
4. The scheduler gives the process CPU time.
5. The process may create child processes or threads.
6. The process eventually exits.
7. The parent reaps the exit status with `wait()` or `waitpid()`.

A process image includes more than the executable code:

- **Text segment**: program instructions
- **Data segment**: initialized global/static data
- **BSS**: uninitialized global/static data
- **Heap**: dynamically allocated memory
- **Stack**: function calls, local variables, return addresses
- **Open file descriptors**: inherited or newly opened resources
- **Credentials**: UID, GID, supplementary groups, capabilities, security labels

**Process creation and termination**

Linux commonly creates new processes with `fork()` and then replaces the child’s program image with `execve()`.

- `fork()` creates a new child process that is initially a near-copy of the parent.
- `execve()` loads a new executable into the current process image.
- `exit()` terminates the process and returns an exit status to the kernel.
- `wait()` and `waitpid()` let the parent collect the child’s status and release the kernel’s bookkeeping.

A common pattern is:

```c
pid_t pid = fork();

if (pid == 0) {
    execl("/bin/ls", "ls", "-l", NULL);
    _exit(1); // only reached if exec fails
} else if (pid > 0) {
    int status;
    waitpid(pid, &status, 0);
}
```

This pattern matters because `fork()` and `execve()` are separate. The child is created first, then the child’s image is replaced. That separation gives shells, daemons, and supervisors flexibility.

**Parent, child, and zombie reaping**

When a child exits before the parent calls `wait()` or `waitpid()`, the child becomes a **zombie**. A zombie is not running; it is a dead process that still has an entry in the process table so the parent can read its exit status.

- The child has terminated.
- The kernel keeps minimal metadata.
- The parent must reap it.
- If the parent never reaps it, zombies accumulate until the parent exits or is fixed.

Useful commands:

```bash
ps -o pid,ppid,state,cmd -p <PID>
```

```bash
top
```

```bash
ps -ef | grep defunct
```

In `ps`, a zombie often appears with state `Z` or as `<defunct>`. In `top`, you may see the `Z` state or a count of zombie tasks. The parent PID (`PPID`) helps you identify which process should reap the child.

**Processes vs threads**

A **process** is an execution container with its own virtual address space and resources. A **thread** is a schedulable execution path within a process that shares the process’s memory and many resources with sibling threads.

Key differences:

- **Process**
  - Has its own address space
  - Stronger isolation
  - Higher creation and context-switch overhead than a thread
- **Thread**
  - Shares memory with other threads in the same process
  - Useful for concurrency inside one application
  - Can corrupt shared state if synchronization is poor

Linux schedules both processes and threads as tasks, but threads share more resources. That is why thread bugs can be harder to debug: one thread can overwrite memory used by another thread in the same process.

Practical view:

- `ps -eLf` shows threads.
- `top -H` shows thread-level activity.
- `pstree -p` helps visualize parent/child relationships.

Example:

```bash
ps -eLf | head
top -H -p <PID>
pstree -p <PID>
```

**User-to-kernel transitions**

User code enters the kernel through controlled events such as:

- **System calls**: requests for kernel services like `open()`, `read()`, `write()`, `fork()`, `execve()`, `waitpid()`, `mmap()`, and `socket()`
- **Interrupts**: hardware events such as timer interrupts or device interrupts
- **Traps**: intentional software-triggered transitions, often used for system calls or debugging
- **Exceptions**: CPU-detected faults such as divide-by-zero, invalid memory access, or page faults

These transitions are the main ways execution crosses from user mode into kernel mode.

A system call is the normal path for asking the kernel to do privileged work:

```c
#include <unistd.h>
#include <fcntl.h>

int fd = open("/etc/hosts", O_RDONLY);
```

Under the hood, the process traps into the kernel, the kernel validates the request, performs the operation, and returns control to user space. Tools like `strace` make this visible:

```bash
strace -f -o trace.log ls /etc
```

`strace` is useful because it shows the exact system calls a program makes, which helps you understand how user-space code depends on kernel services.

**Signals, faults, and process state changes**

Signals are asynchronous notifications delivered to a process or thread. They are not the same as interrupts, but they often interact with the execution model in similar ways from the application’s perspective.

Common examples:

- `SIGINT` from <kbd>Ctrl</kbd> + <kbd>C</kbd>
- `SIGTERM` for graceful termination
- `SIGKILL` for immediate termination
- `SIGCHLD` when a child exits
- `SIGSEGV` for invalid memory access

A page fault is a CPU exception that may be normal or fatal:

- A valid page fault can occur when memory is demand-paged in.
- An invalid page fault can terminate the process with `SIGSEGV`.

Process states commonly include:

- `R` running or runnable
- `S` interruptible sleep
- `D` uninterruptible sleep, often waiting on I/O
- `T` stopped or traced
- `Z` zombie

You can observe these states with:

```bash
ps -o pid,ppid,state,stat,cmd -p <PID>
```

```bash
watch -n 1 'ps -eo pid,ppid,state,cmd | head -20'
```

**Scheduling and context switching**

The scheduler decides which runnable task gets CPU time. Because the CPU can run only a limited number of tasks at once, Linux rapidly switches between them.

A **context switch** saves the current task’s CPU state and restores another task’s state. The saved state includes things like:

- Program counter / instruction pointer
- Stack pointer
- CPU registers
- Memory management context
- Scheduling metadata

Context switching is necessary for multitasking, but it has overhead. Too many switches can reduce performance, especially if tasks compete for CPU or frequently block on I/O.

Useful observations:

```bash
vmstat 1
```

```bash
pidstat -w 1
```

```bash
top
```

High context-switch rates can indicate lock contention, excessive threading, or I/O-heavy workloads.

**Memory protection and virtual memory**

Each process sees its own **virtual address space**. The kernel and MMU (Memory Management Unit) translate virtual addresses to physical memory using page tables.

This design provides:

- Isolation between processes
- Controlled sharing through shared memory, `mmap()`, or libraries
- Protection against accidental or malicious memory access
- Efficient demand paging and copy-on-write behavior

Copy-on-write is important after `fork()`:

- Parent and child initially share the same physical pages read-only.
- When one writes, the kernel copies the page.
- This makes process creation efficient.

Inspect memory-related information with:

```bash
cat /proc/<PID>/maps
```

```bash
cat /proc/<PID>/status
```

```bash
pmap <PID>
```

**/proc and process inspection**

The `/proc` filesystem exposes live kernel and process information as pseudo-files. It is one of the best ways to study the execution model.

Useful paths:

- `/proc/<PID>/status` for state, UIDs, GIDs, and memory summary
- `/proc/<PID>/cmdline` for the command line
- `/proc/<PID>/fd/` for open file descriptors
- `/proc/<PID>/maps` for memory mappings
- `/proc/<PID>/task/` for threads
- `/proc/cpuinfo` and `/proc/meminfo` for system context

Examples:

```bash
cat /proc/$$/status
ls -l /proc/$$/fd
tr '\0' ' ' < /proc/$$/cmdline
```

`$$` expands to the current shell’s PID, which makes it easy to inspect your own shell process.

**Practical fork/exec demonstration**

A simple demonstration helps connect the theory to real behavior:

```c
#include <stdio.h>
#include <unistd.h>
#include <sys/wait.h>

int main(void) {
    pid_t pid = fork();

    if (pid == 0) {
        printf("child: pid=%d ppid=%d\n", getpid(), getppid());
        execl("/bin/echo", "echo", "hello from child", NULL);
        _exit(1);
    }

    if (pid > 0) {
        int status;
        printf("parent: pid=%d child=%d\n", getpid(), pid);
        waitpid(pid, &status, 0);
        printf("child exited with status=%d\n", WEXITSTATUS(status));
    }

    return 0;
}
```

Compile and run:

```bash
gcc -Wall -O2 demo.c -o demo
./demo
```

This shows the full lifecycle:

- Parent creates child with `fork()`
- Child replaces itself with `/bin/echo` using `execl()`
- Parent waits and reaps the child
- Exit status returns to the parent

**Permissions, capabilities, and MAC at the boundary**

Linux security controls determine what a process can do once it crosses into the kernel.

- **Permissions**: traditional Unix mode bits control file access by owner, group, and others.
- **Capabilities**: split root privileges into smaller units such as `CAP_NET_ADMIN` or `CAP_SYS_TIME`.
- **MAC**: mandatory access control adds policy enforcement beyond standard permissions.

This matters at the kernel boundary because the kernel checks credentials on every sensitive operation. For example:

- Opening a protected file depends on file permissions and possibly ACLs.
- Binding to a privileged port may require a capability.
- Access to a device node may be blocked by permissions or MAC policy.
- Even root can be constrained by MAC rules.

Practical checks:

```bash
id
getcap /usr/bin/ping
ls -l /dev/sda
ls -Z /etc/passwd
```

If SELinux is enabled, `ls -Z` shows security contexts. If AppArmor is active, profile enforcement may appear in logs or status tools. The key exam idea is that kernel mediation is where these controls are enforced.

**Observing user and kernel activity in practice**

You can use several tools to see the execution model in action:

- `ps` for process state and parent/child relationships
- `top` or `htop` for live CPU and memory usage
- `strace` for system calls and signals
- `/proc` for detailed process metadata
- `lsof` for open files and sockets
- `pstree` for hierarchy visualization

Examples:

```bash
ps -eo pid,ppid,user,state,cmd --forest
```

```bash
strace -f -e trace=process,file ./demo
```

```bash
lsof -p <PID>
```

These tools help you answer practical questions such as:

- Is the process sleeping on I/O or using CPU?
- Which parent created this child?
- Is the child a zombie because the parent never waited?
- Which system calls does the program use?
- What files or sockets does the process hold open?

**Exam-focused takeaways**

- User space runs with limited privilege; kernel space runs with full privilege.
- On x86, user mode is typically ring 3 and kernel mode is ring 0.
- Processes are isolated execution containers; threads share a process’s memory.
- `fork()` creates a child, `execve()` loads a new program, and `wait()`/`waitpid()` reap children.
- System calls, interrupts, traps, and exceptions are the main entry points into the kernel.
- Zombies exist when a child exits but the parent has not reaped it.
- `/proc`, `ps`, `top`, `strace`, and `pstree` are essential for observing execution behavior.
- Permissions, capabilities, and MAC are enforced by the kernel at the user/kernel boundary.
