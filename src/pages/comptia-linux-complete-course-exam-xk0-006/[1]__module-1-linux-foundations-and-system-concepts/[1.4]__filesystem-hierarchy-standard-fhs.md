## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Filesystem Hierarchy Standard overview**

The <abbr title="Filesystem Hierarchy Standard">FHS</abbr> defines the conventional directory layout used by Linux systems. It helps you predict where software, configuration, logs, device files, and user data belong, even when distributions differ in packaging or defaults.

The main value of FHS is consistency. When you know the role of each top-level directory, you can troubleshoot faster, write safer scripts, and avoid placing files in locations that may be overwritten by package managers or lost during upgrades.

- **Predictability** helps you locate files quickly across distributions.
- **Separation** keeps operating system files distinct from variable data and user content.
- **Maintainability** reduces upgrade and backup problems.

**Core directory categories**

FHS directories generally fall into a few functional groups:

- **Essential boot and system binaries** such as `/bin`, `/sbin`, `/lib`, and `/lib64`
- **Configuration** under `/etc`
- **Variable runtime data** under `/var`
- **Temporary files** under `/tmp` and `/var/tmp`
- **User home directories** under `/home`
- **Optional or third-party software** under `/opt`
- **Mount points and removable media** under `/mnt`, `/media`, and sometimes `/run`

A useful mental model is: static system files live in one place, while data that changes frequently lives elsewhere. That separation is why `/var/log`, `/var/spool`, and `/var/cache` exist instead of placing everything under `/etc` or `/usr`.

**Top-level directories and their purpose**

| Directory | Purpose | Common contents |
|---|---|---|
| `/` | Root of the filesystem tree | All other directories |
| `/bin` | Essential user commands | `ls`, `cp`, `mv`, `cat` |
| `/sbin` | Essential system administration commands | `ip`, `fsck`, `mount` |
| `/boot` | Bootloader and kernel files | `vmlinuz`, `initramfs`, `grub/` |
| `/dev` | Device files | Disk, terminal, and pseudo-device nodes |
| `/etc` | Host-specific configuration | `passwd`, `fstab`, service configs |
| `/home` | User home directories | Personal files and dotfiles |
| `/lib`, `/lib64` | Essential shared libraries and kernel modules support | Shared objects, loader files |
| `/media` | Auto-mounted removable media | USB drives, optical media |
| `/mnt` | Temporary mount point | Manual mounts for admins |
| `/opt` | Add-on application packages | Vendor software, standalone apps |
| `/proc` | Virtual process and kernel information | Process entries, kernel parameters |
| `/root` | Home directory for the root user | Root’s personal files |
| `/run` | Volatile runtime data since boot | PID files, sockets, state data |
| `/srv` | Data served by system services | Web, FTP, or other service data |
| `/sys` | Kernel and device model interface | Hardware and driver information |
| `/tmp` | Temporary files | Short-lived app and user temp files |
| `/usr` | Userland applications and read-only shareable data | Most installed programs, libraries, docs |
| `/var` | Variable data | Logs, spools, caches, databases |

**Important directories in detail**

**`/etc` for configuration**

`/etc` stores system-wide configuration files that are specific to the host. These files are usually plain text and are edited by administrators.

Examples:

```bash
cat /etc/fstab
cat /etc/hosts
ls /etc/systemd/system/
```

Common uses:

- Network settings
- Service configuration
- Authentication and account-related files
- Boot and mount configuration

> Do not place application data in `/etc`. It is for configuration, not for large or frequently changing content.

**`/var` for changing data**

`/var` contains files that grow or change during normal operation. This includes logs, mail queues, print spools, caches, and package manager metadata.

Examples:

```bash
ls /var/log
du -sh /var/log/*
journalctl -xe
```

Typical subdirectories:

- `/var/log` for logs
- `/var/cache` for cached data
- `/var/spool` for queued jobs
- `/var/lib` for persistent application state
- `/var/tmp` for temporary files that may survive reboot

This directory matters for capacity planning. If `/var` fills up, services may fail even when the root filesystem still has free space.

**`/usr` for installed software**

`/usr` holds most userland programs, libraries, documentation, and shareable read-only data. On modern Linux systems, many commands that used to live in `/bin` or `/sbin` are now provided from `/usr/bin` or `/usr/sbin`, with compatibility links often present.

Common subdirectories:

- `/usr/bin` for most user commands
- `/usr/sbin` for non-essential admin tools
- `/usr/lib` and `/usr/lib64` for libraries
- `/usr/share` for architecture-independent data
- `/usr/local` for locally installed software outside the package manager

Example:

```bash
which python3
ls -l /usr/local/bin
```

Use `/usr/local` when you install software manually and want to keep it separate from distribution-managed packages.

**`/bin`, `/sbin`, `/lib`, and `/lib64`**

Traditionally, `/bin` contains essential commands needed for basic operation and recovery, while `/sbin` contains essential administrative tools. `/lib` and `/lib64` store the shared libraries required by those binaries.

In many modern distributions, these directories may be merged or symlinked into `/usr`, but the functional meaning remains important for exam purposes and system understanding.

Examples:

```bash
ls -l /bin
ls -l /sbin
ldd /bin/ls
```

Why this matters:

- Recovery environments need core tools available early in boot.
- Shared libraries must be accessible for essential binaries to run.
- Scripts should not assume a command is in `/bin` if portability matters; use `command -v` or rely on `PATH`.

**`/boot` for startup files**

`/boot` contains files required to start the system, including the kernel image and bootloader configuration.

Typical contents:

- Kernel images such as `vmlinuz-*`
- Initial RAM disk images such as `initramfs-*`
- Bootloader files such as GRUB configuration

Example:

```bash
ls -lh /boot
grub2-mkconfig -o /boot/grub2/grub.cfg
```

If `/boot` is on a separate partition, it must be mounted correctly before kernel updates or bootloader changes.

**`/dev`, `/proc`, `/sys`, and `/run` as virtual filesystems**

These directories are not ordinary storage locations. They expose kernel and runtime information through virtual filesystems.

- **`/dev`** contains device nodes that represent hardware and pseudo-devices.
- **`/proc`** exposes process and kernel information, often as text files.
- **`/sys`** exposes the kernel device model and hardware attributes.
- **`/run`** stores volatile runtime state since boot.

Examples:

```bash
ls -l /dev/sda
cat /proc/cpuinfo
cat /proc/meminfo
ls /sys/class/net
ls /run
```

Practical use:

- Use `/proc` to inspect process and kernel state.
- Use `/sys` to inspect or sometimes tune device-related settings.
- Use `/run` for PID files and sockets created at runtime.

**`/home` and `/root` for user data**

`/home` contains regular users’ home directories. Each user typically stores personal files, configuration dotfiles, downloads, and application data there.

`/root` is the home directory for the superuser account. It is separate from `/home` to keep administrative work isolated from normal user accounts.

Examples:

```bash
echo $HOME
ls -la /home/alice
ls -la /root
```

Best practice:

- Store user-owned data in home directories.
- Keep administrative scripts and backups aware of hidden files such as `.bashrc`, `.ssh`, and `.config`.

**`/tmp` and `/var/tmp` for temporary files**

Both directories store temporary data, but they differ in expected lifetime.

- **`/tmp`** is for short-lived temporary files and may be cleared on reboot or by cleanup policies.
- **`/var/tmp`** is also temporary, but it is intended to persist longer, often across reboots.

Examples:

```bash
mktemp /tmp/example.XXXXXX
ls -ld /tmp /var/tmp
```

Use temporary directories for scratch files, but do not rely on them for persistent storage.

> Never store important data in `/tmp` unless your application explicitly manages persistence elsewhere.

**`/mnt`, `/media`, and removable storage**

These directories are used for mounting filesystems.

- **`/mnt`** is commonly used for temporary manual mounts by administrators.
- **`/media`** is commonly used for auto-mounted removable media such as USB drives.

Examples:

```bash
sudo mkdir -p /mnt/recovery
sudo mount /dev/sdb1 /mnt/recovery
ls /media/$USER
```

This distinction is mostly conventional, but it improves clarity when multiple filesystems are mounted.

**`/opt` for optional software**

`/opt` is used for add-on application packages, often vendor-supplied software that is self-contained.

Typical layout:

```bash
/opt/vendor/app/
  bin/
  lib/
  conf/
```

Use `/opt` when you want to keep a third-party application isolated from the distribution package database or when the vendor expects a fixed directory structure.

**`/srv` for service data**

`/srv` stores data served by system services, such as web content or FTP files. Not every distribution uses it heavily, but it is the standard location for service-specific data when you want a clear separation between code and served content.

Example:

```bash
sudo mkdir -p /srv/www
sudo chown -R www-data:www-data /srv/www
```

This can be useful for web servers, file servers, and similar services where content should be easy to identify and back up.

**Common admin tasks and commands**

You often need to inspect the filesystem layout during troubleshooting, storage planning, or package verification.

```bash
# Show mounted filesystems and mount points
findmnt

# Display disk usage by top-level directory
du -sh /* 2>/dev/null

# Show filesystem types and free space
df -hT

# Find where a command lives
command -v ssh

# Inspect a package-owned file location on RPM-based systems
rpm -ql bash | head

# Inspect a package-owned file location on Debian-based systems
dpkg -L bash | head
```

These commands help you answer practical questions such as:

- Is `/var` filling up?
- Is `/boot` mounted?
- Is a command coming from `/usr/bin` or `/usr/local/bin`?
- Did a package install files into the expected hierarchy?

**FHS and package management**

Package managers expect files to live in standard locations. If you place files in the wrong directory, upgrades may overwrite them, backups may miss them, and support tools may not find them.

General guidance:

- Put configuration in `/etc`.
- Put persistent application state in `/var/lib`.
- Put logs in `/var/log`.
- Put user data in `/home` or `/srv` depending on the use case.
- Put local custom software in `/usr/local` or `/opt`, not in package-managed directories.

Example of a safer local install path:

```bash
sudo install -m 0755 mytool /usr/local/bin/mytool
```

This keeps custom tools separate from distribution packages and makes them easier to manage.

**Exam-focused distinctions**

A few FHS ideas are especially important for Linux+:

- **Static vs. variable**
  - Static files are usually under `/usr`, `/bin`, `/sbin`, and `/lib`.
  - Variable files are usually under `/var`.
- **Configuration vs. data**
  - Configuration belongs in `/etc`.
  - Data belongs in `/home`, `/srv`, or `/var/lib` depending on purpose.
- **Temporary vs. persistent**
  - `/tmp` is short-lived.
  - `/var/tmp` is longer-lived.
- **System vs. user**
  - System-wide settings are in `/etc`.
  - User-specific settings are in home directories.

**Practical examples**

If you are troubleshooting a service that fails after reboot, check whether it depends on runtime files in `/run` or temporary files in `/tmp`.

```bash
systemctl status nginx
ls -l /run/nginx
journalctl -u nginx --since today
```

If a disk is unexpectedly full, inspect `/var` first because logs, caches, and spools often grow there.

```bash
sudo du -sh /var/*
sudo journalctl --disk-usage
sudo du -sh /var/log
```

If you are creating a backup plan, separate configuration, data, and logs so you can restore them independently.

- Back up `/etc` for system configuration.
- Back up `/home` for user files.
- Back up `/var/lib` for application state when needed.
- Back up `/srv` for hosted content.

**Quick memory aid**

Think of the hierarchy this way:

- `/etc` = editable configuration
- `/usr` = installed software and shared read-only resources
- `/var` = changing data
- `/home` = user files
- `/tmp` = temporary scratch space
- `/boot` = startup files
- `/dev`, `/proc`, `/sys`, `/run` = kernel and runtime interfaces

That model is usually enough to place most files correctly and answer exam questions about where Linux stores specific types of data.
