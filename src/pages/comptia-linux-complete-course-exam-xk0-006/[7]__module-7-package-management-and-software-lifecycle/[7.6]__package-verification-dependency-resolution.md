## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Package verification fundamentals**

Package verification answers two different questions:

- Is the installed package database consistent?
- Do the installed files still match what the package manager recorded?

These are related but not identical. A package can be installed correctly yet have modified files, or a system can have dependency problems even when most files are intact. On the exam, separate **file integrity verification** from **dependency resolution**.

Package managers typically verify:

- File presence and size
- File checksum or digest
- Permissions and ownership
- Configuration file changes
- Package signatures or repository metadata signatures

Verification is most useful when you suspect:

- Accidental file modification
- Unauthorized changes
- Broken upgrades
- Missing libraries after partial installs
- Conflicts caused by mixed repositories or version pinning

**RPM-based verification**

On RPM-based systems, `rpm` provides the most direct file verification tools. The key distinction is:

- `rpm -qf /path/to/file` identifies the package that owns a file
- `rpm -V package-name` verifies the installed files in that package
- `rpm -K package.rpm` or `rpm --checksig package.rpm` checks the package file’s signature and digest information before installation

Example workflow:

```bash
rpm -qf /etc/ssh/sshd_config
rpm -V openssh-server
rpm -K openssh-server-*.rpm
```

`rpm -V` compares installed files against the package metadata stored in the RPM database. If a file differs, RPM prints status characters that help you interpret the mismatch.

Common verification output indicators include:

- `S` — file size differs
- `M` — file mode differs
- `5` — digest differs
- `D` — device major/minor number differs
- `L` — symlink path differs
- `U` — user ownership differs
- `G` — group ownership differs
- `T` — modification time differs
- `P` — capabilities differ

A leading `c` often indicates a configuration file. That matters because config files are expected to change more often than binaries.

Example:

```bash
rpm -V bash
```

Possible output:

```text
S.5....T.  c /etc/bashrc
```

This suggests the file size, digest, and timestamp differ, and the file is a configuration file. You would investigate whether the change is intentional before reinstalling.

> `rpm -V` verifies package-installed files; it does not tell you whether a package is trusted. Signature verification and trust are separate concerns.

If you need to repair a package after verification, common approaches include:

- Reinstall the package with `dnf reinstall package-name`
- Reinstall from a known-good repository or local RPM
- Restore a modified config file from backup if the change was accidental

**Debian and Ubuntu verification**

Debian-based systems use `dpkg` and APT. The main verification command is `dpkg -V package-name`, which compares installed files against package metadata.

Example:

```bash
dpkg -V openssh-server
```

`dpkg -V` output uses a compact status format similar in spirit to RPM. The exact output can vary, but the goal is the same: identify files that differ from the package record.

Useful related commands:

- `dpkg -S /path/to/file` finds the package that owns a file
- `dpkg -l` lists installed packages
- `apt show package-name` displays package metadata
- `apt policy package-name` shows candidate versions and repository selection

Example:

```bash
dpkg -S /etc/ssh/sshd_config
apt policy openssh-server
```

`apt update` refreshes package metadata and verifies repository metadata signatures through the signed `Release` or `InRelease` workflow. It does not verify installed package files. Use it when dependency resolution seems wrong because the local package index is stale.

If package installation is interrupted or dependencies are left unresolved, use:

```bash
apt -f install
```

or:

```bash
apt --fix-broken install
```

This attempts to complete or repair broken dependency states. It is not a substitute for file integrity verification.

**Dependency resolution on Debian-based systems**

APT resolves dependencies automatically during install, upgrade, and remove operations. When it cannot satisfy a dependency, you usually need to determine whether the issue is:

- A missing package in the enabled repositories
- A version conflict
- A held package
- A broken package state
- A repository priority or pinning issue

Useful diagnostic commands:

```bash
apt-cache policy package-name
apt policy package-name
apt-cache depends package-name
apt-cache rdepends package-name
```

`apt-cache policy` and `apt policy` are especially useful when the wrong version is selected. They show:

- Installed version
- Candidate version
- Repository priorities
- Which repository provides each version

Example:

```bash
apt policy libssl3
```

If a package is held, APT may refuse to upgrade it. Common hold mechanisms include:

```bash
apt-mark showhold
apt-mark hold package-name
apt-mark unhold package-name
```

For example, if a kernel or database package must stay on a known-good version, you can hold it temporarily while resolving other upgrades.

When dependencies conflict, you may need to:

1. Check the candidate version with `apt policy`
2. Verify the package source with `apt-cache policy`
3. Remove or downgrade conflicting packages if appropriate
4. Run `apt -f install` to complete the dependency graph
5. Reinstall the affected package if files or scripts are inconsistent

Example conflict workflow:

```bash
apt policy nginx
apt-cache depends nginx
apt remove nginx nginx-common
apt install nginx
apt -f install
```

**Dependency resolution on RPM-based systems**

RPM-based distributions usually rely on `dnf` or `zypper` for dependency resolution. These tools resolve dependencies from enabled repositories and can install missing requirements automatically.

With `dnf`, common commands include:

```bash
dnf install package-name
dnf remove package-name
dnf upgrade
dnf reinstall package-name
dnf repoquery --requires package-name
dnf repoquery --whatprovides 'libssl.so.3()(64bit)'
```

If a package is unavailable, the issue is often repository-related rather than a broken package manager. You may need to:

- Enable an additional repository
- Enable a module stream
- Refresh metadata
- Choose a compatible package version
- Resolve modular conflicts

Examples:

```bash
dnf repolist
dnf module list
dnf module enable nodejs:20
dnf install nodejs
```

If a required dependency is missing from the enabled repositories, DNF cannot resolve it until you add the correct source. This is common on minimal installs, EOL systems, or systems with disabled optional repositories.

For version-specific control, RPM-based systems often use a version lock mechanism. On DNF systems, `versionlock` is commonly used:

```bash
dnf install 'dnf-command(versionlock)'
dnf versionlock add kernel-*
dnf versionlock list
dnf versionlock delete kernel-*
```

This is the RPM-family equivalent of APT hold/pinning concepts. It prevents selected packages from being upgraded past a chosen version.

**Zypper verification and resolution**

On SUSE-based systems, `zypper` handles installation and dependency resolution. It can also verify installed packages, but do not confuse that with full RPM file verification.

Useful commands:

```bash
zypper install package-name
zypper remove package-name
zypper update
zypper verify
zypper ve
zypper info package-name
zypper se -s package-name
zypper wp 'libssl.so.3'
```

`zypper verify` or `zypper ve` checks for dependency and package consistency issues in the installed system. It is useful for detecting broken package relationships, but `rpm -V` remains the better tool for file-level integrity checks.

If you need to find which package provides a file or capability, `zypper what-provides` or `zypper wp` can help. For file paths, be precise with the query:

```bash
zypper wp /usr/lib64/libssl.so.3
```

If a package is missing, you may need to enable a repository or module before installation:

```bash
zypper lr -u
zypper refresh
zypper addrepo https://example.repo.local/repo repo-name
zypper install package-name
```

**Interpreting verification results**

Verification output tells you what changed, not why it changed. Use the result to decide whether the change is expected.

- A changed config file may be normal after local customization
- A changed binary may indicate corruption or tampering
- A missing file may indicate accidental deletion or a failed upgrade
- A dependency error may indicate repository mismatch rather than package damage

Practical interpretation strategy:

- Compare the file against backups or version control
- Check package changelogs and update history
- Reinstall only after confirming the change is not intentional
- Preserve local config files when reinstalling packages

Example RPM reinstall with config preservation awareness:

```bash
dnf reinstall openssh-server
```

Example Debian package reinstall:

```bash
apt install --reinstall openssh-server
```

**Common dependency problems and fixes**

Dependency issues usually fall into a few patterns:

- **Missing dependency**: the required package is not installed
- **Version mismatch**: the installed version is too old or too new
- **Conflicting packages**: two packages provide incompatible files or capabilities
- **Broken transaction**: installation stopped halfway
- **Repository mismatch**: the needed package exists, but not in enabled sources

Typical fixes include:

- Refresh metadata with `apt update`, `dnf makecache`, or `zypper refresh`
- Inspect candidate versions with `apt policy` or `dnf repoquery`
- Enable the correct repository or module stream
- Remove the conflicting package
- Reinstall the affected package
- Complete interrupted package configuration

Examples:

```bash
apt update
apt policy libpq5
apt install --reinstall postgresql-client
```

```bash
dnf makecache
dnf repoquery --whatprovides 'perl(JSON::XS)'
dnf install perl-JSON-XS
```

```bash
zypper refresh
zypper se -s python3
zypper install python3
```

> If a dependency cannot be found, do not assume the package manager is broken. First confirm that the correct repository, module stream, or architecture is enabled.

**Package holds, pinning, and version control**

Sometimes you need to prevent upgrades while keeping the rest of the system current. This is useful for:

- Kernel stability
- Application compatibility
- Vendor-certified stacks
- Temporary rollback protection during maintenance

APT uses holds and pinning:

```bash
apt-mark hold package-name
apt-mark unhold package-name
apt-cache policy package-name
```

You can also use pinning in `/etc/apt/preferences` or `/etc/apt/preferences.d/` to influence version selection.

RPM-based systems commonly use:

- `dnf versionlock`
- Repository priority controls
- Module stream selection
- Excluding packages during a transaction

Example:

```bash
dnf install --exclude=kernel\* package-name
```

This excludes matching packages from that transaction. Use it carefully because it can leave the system partially upgraded if the excluded package is a dependency.

**Practical troubleshooting workflow**

When a package or dependency problem appears, work from the least invasive checks to the most invasive fixes:

1. Identify the affected package and file ownership.
2. Verify whether the file or package metadata changed.
3. Check repository metadata and candidate versions.
4. Inspect dependency relationships.
5. Confirm whether a hold, pin, or module stream is blocking resolution.
6. Reinstall or repair only after you understand the cause.

Example workflow on an RPM system:

```bash
rpm -qf /usr/bin/ssh
rpm -V openssh-clients
dnf repoquery --requires openssh-clients
dnf policy openssh-clients
dnf reinstall openssh-clients
```

Example workflow on a Debian system:

```bash
dpkg -S /usr/bin/ssh
dpkg -V openssh-client
apt policy openssh-client
apt-cache depends openssh-client
apt --fix-broken install
```

**Exam focus**

Be ready to distinguish these command families:

- `rpm -V` and `dpkg -V` verify installed files
- `rpm -qf` and `dpkg -S` identify file ownership
- `rpm -K` and `--checksig` verify package file signatures and digests
- `apt policy` and `apt-cache policy` diagnose version selection
- `apt -f install`, `dnf reinstall`, and `zypper verify` help with dependency or package-state issues
- `dnf versionlock` and `apt-mark hold` prevent unwanted upgrades

Also remember:

- Repository metadata verification is not the same as package file verification
- Dependency resolution may require enabling the right repository or module stream
- Configuration files often change legitimately, so interpret verification output carefully
