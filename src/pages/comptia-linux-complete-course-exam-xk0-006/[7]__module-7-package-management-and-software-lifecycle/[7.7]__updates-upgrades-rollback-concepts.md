**Updates vs. upgrades vs. rollback**

Linux maintenance spans three related but distinct activities:

- **Updates** apply newer package versions, security fixes, or bug fixes within the same release.
- **Upgrades** move a system to a newer release or major version, or replace a package with a newer compatible build.
- **Rollback** restores a previous working state after a failed change, patch, or upgrade.

Understanding the difference matters because the tools, risk level, and recovery options are not the same. A routine package update is usually low risk. A distribution upgrade can change libraries, services, and configuration formats. A rollback may require snapshots, backups, or boot-time recovery steps rather than a simple package command.

**Package management basics**

Most Linux systems use a package manager to install, remove, and update software while tracking dependencies. Common examples include:

- **APT** on Debian and Ubuntu
- **DNF** on Fedora and RHEL-like systems
- **YUM** on older RHEL-like systems
- **Zypper** on SUSE/openSUSE
- **Pacman** on Arch-based systems

Package managers typically handle:

- Dependency resolution
- Repository metadata
- Package signatures and trust
- Version selection and conflict detection
- Transaction logging

A package update usually means you refresh repository metadata, then apply available package changes.

```bash
# Debian/Ubuntu
sudo apt update
sudo apt upgrade

# Fedora/RHEL-like
sudo dnf makecache
sudo dnf upgrade

# SUSE/openSUSE
sudo zypper refresh
sudo zypper update
```

**Repository metadata and cache**

Package managers rely on repository metadata to know what versions are available. Metadata can become stale, so you refresh it before checking for updates.

- `apt update` downloads current package lists.
- `dnf makecache` refreshes cached metadata.
- `zypper refresh` updates repository data.
- `pacman -Sy` refreshes sync databases, though partial upgrades are generally discouraged on Arch-based systems.

> `dnf check-update` checks for available updates; it does not refresh metadata by itself. Use `dnf makecache` or `dnf upgrade --refresh` when you want a refresh plus update workflow.

A stale cache can cause missing packages, outdated dependency resolution, or confusing version comparisons.

**Security updates and maintenance updates**

Updates are often grouped by purpose:

- **Security updates** fix vulnerabilities.
- **Bugfix updates** correct defects without changing intended behavior.
- **Feature updates** add capabilities, but may also introduce new defaults or dependencies.

In enterprise environments, you often stage updates:

- Test in a lab or nonproduction host first.
- Apply to a small pilot group.
- Roll out broadly after validation.
- Keep a maintenance window for service restarts and reboots.

This reduces the chance that a kernel, libc, or service update disrupts production.

**Common update commands**

Different package managers use different verbs, but the goal is similar.

```bash
# Debian/Ubuntu
sudo apt update
sudo apt full-upgrade

# Fedora/RHEL-like
sudo dnf upgrade

# SUSE/openSUSE
sudo zypper update

# Arch-based
sudo pacman -Syu
```

Notes:

- `apt upgrade` avoids removing packages in some cases; `apt full-upgrade` can install or remove packages to complete the transaction.
- `dnf upgrade` updates installed packages to the latest available versions in enabled repositories.
- `zypper update` applies available package updates.
- `pacman -Syu` performs a full sync and upgrade in one step.

**Kernel updates and reboot awareness**

Kernel updates are common and important because they fix security issues and hardware support problems. However, a running system continues using the old kernel until you reboot.

Useful checks include:

```bash
uname -r
rpm -q kernel
dpkg -l 'linux-image*'
```

- `uname -r` shows the running kernel.
- `rpm -q kernel` lists installed kernel packages on RPM-based systems.
- `dpkg -l 'linux-image*'` lists installed kernel images on Debian-based systems.

A package query alone does not tell you whether a reboot is required. You infer that from the type of update, package manager hints, or the presence of a newer installed kernel than the running one.

**Reboot indicators and service restarts**

Some updates require only a service restart; others require a full reboot.

- Restart a service when its shared library or configuration changes and the daemon can reload cleanly.
- Reboot when the kernel, init system, or core runtime libraries change in a way that affects the running system.
- Use maintenance windows for systems with strict uptime requirements.

Examples:

```bash
sudo systemctl restart sshd
sudo systemctl restart nginx
sudo reboot
```

You should also verify whether a package manager or vendor tool reports a reboot requirement. On some systems, a file such as `/var/run/reboot-required` may be created by update tooling.

**Major release upgrades**

A major release upgrade moves the system to a new distribution version, such as from one Fedora release to the next or from one Ubuntu LTS release to another. This is more disruptive than a normal package update because it may change:

- Default configuration files
- Library ABIs (Application Binary Interfaces)
- Service behavior
- Package names or availability
- Deprecated features

Examples of major upgrade workflows:

```bash
# Ubuntu
sudo do-release-upgrade

# Fedora / DNF-based systems using the system-upgrade plugin
sudo dnf system-upgrade download --releasever=41
sudo dnf system-upgrade reboot

# openSUSE / SUSE distribution sync
sudo zypper dup
```

Important distinctions:

- `do-release-upgrade` is the standard Ubuntu release upgrade path.
- `dnf system-upgrade` is a plugin-based workflow used on Fedora and some DNF-based systems; it is not the universal RHEL major upgrade method.
- `zypper dup` performs a distribution upgrade and is also used for regular distribution synchronization, not only major version jumps.
- RHEL major version upgrades are often handled with **Leapp** rather than a generic DNF release-upgrade workflow.

> Major upgrades are higher risk than package updates. Always back up data and verify application compatibility before proceeding.

**Rollback concepts**

Rollback means returning to a known-good state after a failed update, upgrade, or configuration change. The best rollback method depends on what changed.

Common rollback approaches:

- **Package rollback**: reinstall a previous package version if the repository still provides it.
- **Snapshot rollback**: revert a filesystem, volume, or VM snapshot.
- **Backup restore**: recover files or full systems from backup media.
- **Configuration rollback**: restore previous config files from version control or backups.
- **Boot rollback**: select an older kernel or boot entry from the bootloader.

Rollback is easiest when you plan for it before making changes.

**Package rollback and version pinning**

Some package managers can downgrade packages or hold versions.

```bash
# Debian/Ubuntu
sudo apt install package-name=1.2.3-1
sudo apt-mark hold package-name

# Fedora/RHEL-like
sudo dnf downgrade package-name
sudo dnf versionlock add package-name

# SUSE/openSUSE
sudo zypper addlock package-name
sudo zypper install --oldpackage package-name
```

Use these carefully:

- Downgrades may break dependencies.
- Version locks can prevent security fixes if left in place too long.
- A downgrade is not always possible if the older package is no longer in the repository.

**Filesystem snapshots**

Snapshots are one of the most useful rollback tools because they capture a point-in-time state quickly.

Common snapshot technologies:

- **LVM snapshots**
- **Btrfs snapshots**
- **ZFS snapshots**

Example with Btrfs:

```bash
sudo btrfs subvolume snapshot /@ /@-prechange
```

This creates a snapshot of the root subvolume before a risky change. To use it for rollback, you usually need to:

- Boot into a recovery environment or alternate boot entry.
- Mount the filesystem.
- Change the default subvolume or boot configuration to point to the snapshot.
- Reboot into the restored state.

A snapshot alone does not automatically become the active root filesystem.

Example with LVM:

```bash
sudo lvcreate -L 2G -s -n root_prechange /dev/vg0/root
```

LVM snapshots can be merged back into the origin, but the origin volume typically must be inactive. That often means you perform the merge from rescue mode or after rebooting into another environment.

**Virtual machine rollback**

Virtual machines often support snapshots, which can be useful before patching or testing.

Typical workflow:

1. Shut down or quiesce the VM if the hypervisor requires it.
2. Create a snapshot.
3. Apply updates or configuration changes.
4. Revert if the change fails.
5. Verify guest services after rollback.

Example with libvirt:

```bash
virsh snapshot-create-as vm01 pre-upgrade "Before patching"
virsh snapshot-list vm01
virsh snapshot-revert vm01 pre-upgrade
```

Snapshot behavior depends on the hypervisor and snapshot type:

- Internal snapshots and external snapshots behave differently.
- Memory state may or may not be included.
- Disk-only snapshots do not restore running RAM state.
- Some guests require shutdown before revert.

Always confirm the snapshot model before relying on it for recovery.

**Backups vs. snapshots**

Snapshots and backups are related but not interchangeable.

| Feature | Snapshot | Backup |
|---|---|---|
| Speed | Very fast | Slower |
| Storage location | Often same storage system | Usually separate storage |
| Protection from disk failure | Limited | Stronger |
| Best use | Quick rollback | Disaster recovery |
| Retention | Short-term | Long-term |

Use snapshots for fast recovery from a bad change. Use backups for hardware failure, ransomware, accidental deletion, and long-term retention.

**Configuration management and rollback**

Configuration management tools help you make changes repeatable and reversible.

Examples:

- **Ansible** can redeploy known-good configuration.
- **Git** can track config file history.
- **Puppet** and **Chef** can reapply desired state.

A practical pattern is to store critical config files in version control:

```bash
git init /etc-config-backup
cp /etc/ssh/sshd_config /etc-config-backup/
git add sshd_config
git commit -m "Baseline SSH config"
```

If a change breaks access, you can restore the previous version quickly.

**Safe update workflow**

A disciplined workflow reduces outage risk.

1. Identify the change and its scope.
2. Check release notes and dependency impact.
3. Back up data or create a snapshot.
4. Apply the update in a test environment if possible.
5. Update the production system during a maintenance window.
6. Verify services, logs, and application behavior.
7. Reboot if required.
8. Keep a rollback path ready until validation is complete.

This process is especially important for kernel updates, database servers, authentication services, and remote access systems.

**Verification after updates**

After applying changes, verify that the system is healthy.

Useful checks include:

```bash
systemctl --failed
journalctl -p err -b
df -h
free -h
ss -tulpn
```

- `systemctl --failed` shows failed units.
- `journalctl -p err -b` shows error-level logs from the current boot.
- `df -h` checks filesystem usage.
- `free -h` checks memory availability.
- `ss -tulpn` confirms listening services and ports.

You should also test application-specific functions, not just service status. A web server can be “running” while authentication, database connectivity, or TLS certificates are broken.

**Troubleshooting failed updates**

If an update fails, common causes include:

- Broken dependencies
- Repository mismatch
- Disk space exhaustion
- Package conflicts
- Interrupted transactions
- Incompatible third-party repositories

Useful recovery actions:

- Clear or refresh metadata.
- Remove or disable problematic repositories.
- Free disk space in `/var`, `/boot`, or package cache locations.
- Re-run the transaction after fixing the root cause.
- Boot into rescue mode if the system no longer starts.

Examples:

```bash
sudo dnf clean all
sudo dnf makecache
sudo apt --fix-broken install
sudo zypper refresh
```

**Bootloader and recovery entries**

Rollback may require selecting an older kernel or recovery entry at boot time.

Common boot-time options:

- GRUB menu selection
- Rescue mode
- Single-user or emergency mode
- Alternate kernel entry

If a new kernel fails, booting an older installed kernel can restore access long enough to remove the bad package or investigate logs.

**Practical comparison of change types**

| Change type | Risk | Typical tool | Rollback method |
|---|---:|---|---|
| Security patch | Low to medium | `apt upgrade`, `dnf upgrade`, `zypper update` | Package downgrade, snapshot, backup |
| Kernel update | Medium | Package manager | Reboot into older kernel, snapshot |
| App version update | Medium | Package manager or app installer | Restore config, downgrade, snapshot |
| Major release upgrade | High | `do-release-upgrade`, `dnf system-upgrade`, `zypper dup` | Snapshot, backup restore, rescue mode |
| VM patching | Medium | Hypervisor tools | Snapshot revert |

**Exam-focused takeaways**

- **Update** means apply newer packages within the same release.
- **Upgrade** can mean a package version increase or a full distribution release change.
- **Rollback** requires a preplanned recovery method, not just a package command.
- **Snapshots** are fast but are not a substitute for backups.
- **Kernel changes** often require a reboot before they take effect.
- **Major release upgrades** are riskier and need compatibility checks, backups, and validation.
- **Repository metadata** must be current for reliable package operations.

**Quick command reference**

```bash
# Refresh metadata
sudo apt update
sudo dnf makecache
sudo zypper refresh

# Apply updates
sudo apt upgrade
sudo dnf upgrade
sudo zypper update

# Major upgrades
sudo do-release-upgrade
sudo dnf system-upgrade download --releasever=41
sudo dnf system-upgrade reboot
sudo zypper dup

# Rollback helpers
sudo dnf downgrade package-name
sudo apt install package-name=version
sudo lvcreate -s -n snap /dev/vg0/root
sudo btrfs subvolume snapshot /@ /@-prechange
```
