**DNF overview on RHEL**

`dnf` is the modern package manager on Red Hat Enterprise Linux (RHEL) and related distributions. It resolves dependencies, downloads packages from configured repositories, and records transactions so you can inspect or roll back changes later. For Linux+ exam prep, focus on the workflow: discover packages, inspect repositories, install or remove software, manage modules and groups, and recover from mistakes using history.

`dnf` is the successor to `yum`, and many `yum` commands still work as compatibility aliases. In practice, `dnf` is the preferred tool, and `upgrade` is the preferred update action.

- **Package management**: install, remove, upgrade, downgrade, and reinstall software.
- **Repository control**: enable, disable, add, and inspect software sources.
- **Transaction safety**: review history before and after changes.
- **Content streams**: use modules for versioned application stacks such as `nodejs` or `php`.

**Core command patterns**

You usually work with `dnf` in a few repeatable ways. First, search and inspect. Then install or modify packages. Finally, verify the result and record the transaction if needed.

```bash
dnf search nginx
dnf info nginx
dnf list installed
dnf list available
dnf provides /usr/bin/python3
```

- `search` finds package names and summaries.
- `info` shows package metadata, version, architecture, and repository.
- `list installed` confirms what is already present.
- `provides` maps a file or capability to a package.

**Installing and removing packages**

The most common workflow is package installation. Use `-y` for noninteractive automation when you already trust the transaction, and use `--assumeno` when you want to preview the result without applying it.

```bash
dnf install httpd
dnf remove httpd
dnf reinstall httpd
dnf downgrade httpd
dnf upgrade httpd
dnf update httpd
```

- **install** adds a package and its dependencies.
- **remove** uninstalls a package and leaves configuration files only when the package policy preserves them.
- **reinstall** is useful when files are corrupted or missing.
- **downgrade** returns to an older available version when the repository still provides it.
- **upgrade** is the preferred command for moving to newer versions.
- `update` remains a compatibility alias in DNF, but `upgrade` is clearer for exam and admin use.

> `dnf downgrade` only works if an older version is available in enabled repositories or local package sources.

**Noninteractive and safety options**

Automation often depends on predictable behavior. DNF supports options that help you script changes safely.

```bash
dnf -y install vim-enhanced
dnf --assumeno install vim-enhanced
dnf --assumeyes upgrade
```

- `-y` or `--assumeyes` answers yes to prompts.
- `--assumeno` previews the transaction and answers no.
- Use these options carefully in scripts because they can hide prompts that would otherwise prevent mistakes.

**Repository workflow**

RHEL package sources are defined in repository files under `/etc/yum.repos.d/`. These files usually contain one or more repository stanzas with settings such as `baseurl`, `enabled`, `gpgcheck`, and `gpgkey`. DNF reads these definitions to decide where to fetch packages and metadata.

```ini
[rhel-baseos]
name=RHEL BaseOS
baseurl=https://example.invalid/rhel/9/BaseOS/x86_64/os/
enabled=1
gpgcheck=1
gpgkey=file:///etc/pki/rpm-gpg/RPM-GPG-KEY-redhat-release
```

- **baseurl** points to the repository content.
- **enabled** controls whether DNF uses the repo.
- **gpgcheck** enforces package signature verification.
- **gpgkey** identifies the trusted signing key.

You can inspect repository status with:

```bash
dnf repolist
dnf repolist all
dnf repoinfo
```

- `repolist` shows enabled repositories.
- `repolist all` shows enabled and disabled repositories.
- `repoinfo` provides more detail about each repository.

To add or modify repositories, use `dnf config-manager` from the `dnf-plugins-core` package.

```bash
dnf install -y dnf-plugins-core
dnf config-manager --add-repo https://example.invalid/custom.repo
dnf config-manager --set-enabled myrepo
dnf config-manager --set-disabled myrepo
dnf config-manager --setopt=myrepo.gpgcheck=1
```

- `--add-repo` creates a new repo file from a URL or local file.
- `--set-enabled` and `--set-disabled` toggle repository state.
- `--setopt` changes a repository option without editing the file manually.

**Metadata cache and refresh behavior**

DNF uses cached metadata to speed up operations. If repository content changes, you may need to refresh the cache.

```bash
dnf makecache
dnf clean all
dnf clean metadata
```

- `makecache` rebuilds repository metadata cache.
- `clean metadata` removes cached metadata so DNF fetches fresh data later.
- `clean all` removes metadata and package cache data.

Use `makecache` when you want to proactively refresh repository data. Use `clean metadata` when you want to force a later refresh by clearing stale cache.

**Package groups**

Groups let you install a collection of related packages, such as a desktop environment or server role. This is useful when you want a functional software set rather than a single package.

```bash
dnf group list
dnf group info "Development Tools"
dnf group install "Development Tools"
dnf group remove "Development Tools"
```

- `group list` shows available, installed, and hidden groups.
- `group info` reveals the packages included in a group.
- `group install` installs the selected group.
- `group remove` removes the group’s packages when possible.

Groups are common in RHEL workflows for server roles, build environments, and troubleshooting toolsets. If a group is not visible, repository content or module state may be affecting availability.

**Modules and streams**

RHEL modules provide versioned application streams. A module stream lets you choose a supported major version of software such as `nodejs`, `php`, or `postgresql` without replacing the entire operating system package set.

A typical module workflow is:

1. Inspect available streams.
2. Enable the desired stream.
3. Install the module profile or packages from that stream.
4. Verify the active stream and installed packages.

```bash
dnf module list nodejs
dnf module info nodejs:18
dnf module enable nodejs:18
dnf module install nodejs:18/common
dnf module reset nodejs
```

- `module list` shows streams and profiles.
- `module info` displays stream details and defaults.
- `module enable` selects a stream without installing packages yet.
- `module install <name>:<stream>/<profile>` installs the stream’s profile when one is defined.
- `module reset` clears the stream selection so you can choose again.

> Module syntax can vary by content set and profile availability. If a profile is not obvious, inspect `dnf module info <name>:<stream>` before installing.

You can also install packages after enabling a stream if the module exposes them directly.

```bash
dnf module enable php:8.2
dnf install php php-cli php-fpm
```

This pattern is useful when you want the stream selected first and then choose specific packages from that stream.

**Version locking**

Version locking prevents selected packages from being upgraded past a known-good version. On RHEL, install the plugin package first, then add locks with `dnf versionlock`.

```bash
dnf install -y dnf-plugins-core
dnf versionlock add bash-5.1.8-9.el9.x86_64
dnf versionlock list
dnf versionlock delete bash
```

- **dnf-plugins-core** provides the versionlock plugin on RHEL.
- `versionlock add` pins a package NEVRA, which is Name-Epoch-Version-Release-Architecture.
- `versionlock list` verifies what is currently locked.
- `versionlock delete` removes a lock.

Use version locking when you need to preserve compatibility with an application, driver, or tested baseline. It is especially useful in controlled environments where a package update could break a service.

**History, rollback, and recovery**

DNF records each transaction, which makes troubleshooting and rollback much safer than manual RPM changes. Use history before you change anything important, and use it again if a package update causes a problem.

```bash
dnf history
dnf history info 12
dnf history undo 12
dnf history redo 12
```

- `history` lists transactions with IDs, dates, and actions.
- `history info` shows the packages involved in a transaction.
- `history undo` reverses a transaction when possible.
- `history redo` repeats a transaction.

This is one of the most important exam topics because it supports safe recovery. If a package update breaks a service, you can inspect the transaction, identify the change, and revert it.

DNF also stores useful logs and state information for troubleshooting.

- `/var/log/dnf.log` records DNF activity.
- `/var/log/dnf.rpm.log` may contain RPM transaction details on some systems.
- `dnf history` is the first place to check for package change tracking.

**Practical troubleshooting workflow**

When a package operation fails, work through the problem systematically.

1. Check repository availability.
2. Refresh metadata if the repo changed.
3. Inspect package or module state.
4. Review history if the issue followed an update.
5. Reinstall, downgrade, or undo as needed.

```bash
dnf repolist
dnf makecache
dnf info httpd
dnf history
dnf history undo last
```

- Repository errors often come from disabled repos, bad URLs, or missing subscription access.
- Metadata errors often clear after `dnf makecache` or `dnf clean all`.
- Dependency conflicts may require enabling the correct module stream or repository.
- Transaction rollback is safer than manual file replacement when DNF history can reverse the change.

**Common RHEL subscription and repository considerations**

On RHEL, some repositories are subscription-based. You may need valid registration and entitlement before packages appear. Repository IDs also vary by major version and architecture, so treat examples as patterns rather than universal names.

- **BaseOS** provides core operating system packages.
- **AppStream** provides additional packages and module streams.
- Optional repositories may be required for development tools or supplemental packages.
- Repository names differ between RHEL 8, RHEL 9, and future releases.

A common admin task is enabling a repository only when needed, then disabling it again to reduce accidental package drift.

```bash
dnf config-manager --set-enabled myrepo
dnf install mypackage
dnf config-manager --set-disabled myrepo
```

**Useful package inspection commands**

Before installing or changing software, inspect what DNF knows about the package.

```bash
dnf info podman
dnf list podman
dnf repoquery podman
dnf provides '*/httpd.conf'
```

- `repoquery` is useful for dependency and package relationship questions.
- `provides` helps you find which package owns a file or capability.
- `list` is a quick way to compare installed and available versions.

**Exam-focused distinctions**

- **upgrade vs update**: prefer `upgrade`; `update` is a compatibility alias.
- **makecache vs clean metadata**: `makecache` refreshes cache; `clean metadata` removes it.
- **module enable vs module install**: enabling selects a stream; installing adds packages from that stream.
- **group install vs package install**: groups install a curated set; packages install individual components.
- **history undo vs reinstall**: undo reverses a transaction; reinstall replaces files for a package already installed.

**Quick reference workflow**

```bash
dnf repolist
dnf search <package>
dnf info <package>
dnf install <package>
dnf group list
dnf module list <name>
dnf history
```

Use this sequence when you are unsure what to do next. It keeps you in a safe, observable workflow and reduces the chance of installing the wrong stream, repo, or package version.
