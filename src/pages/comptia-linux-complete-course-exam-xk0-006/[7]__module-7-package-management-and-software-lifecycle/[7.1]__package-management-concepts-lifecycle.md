**Package management overview**

Package management controls how software is discovered, installed, updated, verified, and removed on a Linux system. In practice, it is also a lifecycle discipline: you decide where packages come from, which versions are allowed, how dependencies are resolved, and when old artifacts are cleaned up.

A package manager gives you repeatable software deployment and a consistent way to track ownership of files. It also reduces drift between systems by making package state explicit instead of relying on manual compilation or ad hoc copying.

- **Package**: a bundle of files, metadata, scripts, and dependency information.
- **Repository**: a trusted source of packages and metadata.
- **Dependency**: another package or library required for correct operation.
- **Lifecycle**: install, verify, update, hold, pin, remove, purge, and clean up.

**Package formats and ecosystems**

Different distributions use different package formats and tools, but the lifecycle concepts are similar.

- Debian-based systems use `.deb` packages with `dpkg` as the low-level tool and `apt` as the higher-level resolver.
- Red Hat-based systems use `.rpm` packages with `rpm` as the low-level tool and `dnf` or `yum` as the higher-level resolver.
- Arch-based systems use `pacman`.
- Universal packaging systems such as Snap and Flatpak isolate applications and often manage their own runtimes.

The low-level tool typically installs a local package file and queries package databases. The higher-level tool resolves dependencies, fetches metadata, and manages repositories.

**Repository metadata, trust, and cache behavior**

Package managers do not usually query every repository live for every command. They maintain local metadata caches that describe available versions, dependencies, and signatures. This improves speed and makes dependency resolution possible even when the network is slow or temporarily unavailable.

On Debian and Ubuntu, `apt update` refreshes repository metadata, while `apt upgrade` or `apt full-upgrade` uses that metadata to decide what to install. If metadata is stale, you may see missing versions, dependency errors, or packages that appear unavailable even though they exist upstream.

Common cache and metadata actions include:

- `apt update` to refresh package lists.
- `apt clean` to remove downloaded package archives from the local cache.
- `apt autoclean` to remove only obsolete cached package files.
- `dnf makecache` to prefetch repository metadata.
- `dnf clean all` to clear metadata and package caches.
- `rpm --rebuilddb` to rebuild the RPM database if it becomes corrupted, though this is a recovery action rather than routine maintenance.

Example cache workflow on Debian-based systems:

```bash
sudo apt update
apt-cache policy nginx
sudo apt install nginx
sudo apt clean
```

> Cache cleanup does not uninstall software. It only removes locally stored package files or metadata used for future transactions.

**Installing packages**

Installation can come from a repository or from a local package file. Repository installs are preferred because the package manager can resolve dependencies automatically and track updates.

Examples:

```bash
sudo apt install curl
sudo dnf install curl
sudo pacman -S curl
sudo rpm -ivh ./package.rpm
sudo dpkg -i ./package.deb
```

Important distinctions:

- `apt install` and `dnf install` resolve dependencies automatically.
- `dpkg -i` and `rpm -ivh` install a local file but may fail if dependencies are missing.
- `apt` and `dnf` can often repair dependency issues after a local install by fetching missing packages from configured repositories.

A practical Debian workflow for a local package is:

```bash
sudo dpkg -i ./example.deb
sudo apt -f install
```

The second command asks APT to fix broken dependencies by installing what is missing.

**Querying packages and ownership**

You often need to answer two questions: “Is the package installed?” and “Which package owns this file?”

Useful queries include:

```bash
dpkg -l | grep nginx
apt list --installed | grep nginx
rpm -q nginx
dnf list installed nginx
pacman -Q nginx
```

To identify file ownership:

```bash
dpkg -S /usr/sbin/nginx
rpm -qf /usr/sbin/nginx
pacman -Qo /usr/bin/bash
```

This is useful during troubleshooting because it tells you which package should be repaired, upgraded, or removed.

**Integrity verification and file checks**

Package databases tell you what should be installed, but they do not always prove that files are unchanged. Integrity checks help detect tampering, accidental edits, or corruption.

On Debian-based systems, `debsums` compares installed files against package checksums when available:

```bash
sudo debsums -s
sudo debsums nginx
```

If `debsums` reports mismatches, you may need to reinstall the package or investigate unauthorized changes.

On RPM-based systems, package verification uses `rpm -V`:

```bash
rpm -V openssh-server
```

Verification output can indicate changes in size, permissions, checksum, ownership, or timestamps. This is especially useful when you suspect file drift on a managed host.

**Updating, upgrading, and lifecycle control**

Updating software is not the same as installing it. Updates change package versions while trying to preserve system stability. The exact behavior depends on the tool and distribution policy.

Common commands:

```bash
sudo apt update
sudo apt upgrade
sudo apt full-upgrade
sudo dnf upgrade
sudo pacman -Syu
```

Key differences:

- `apt upgrade` avoids removing packages and may hold back changes that require removals.
- `apt full-upgrade` can remove or replace packages to complete dependency transitions.
- `dnf upgrade` handles dependency changes more flexibly than older `yum` workflows.
- `pacman -Syu` synchronizes repositories and upgrades installed packages in one step.

For lifecycle planning, you should distinguish between security updates, feature updates, and major version transitions. Major transitions can introduce ABI changes, new dependencies, or package renames.

**Holding and pinning packages on Debian and Ubuntu**

Sometimes you need to prevent a package from changing even when updates are available. This is common for production systems, kernel versions, database servers, or applications with strict compatibility requirements.

A package hold blocks normal upgrades:

```bash
sudo apt-mark hold nginx
apt-mark showhold
sudo apt-mark unhold nginx
```

Holds are simple and effective, but they are coarse-grained. If you need more control, use **APT pinning** with preferences files. Pinning lets you assign priorities to versions, releases, or origins.

Example `/etc/apt/preferences.d/nginx`:

```text
Package: nginx
Pin: version 1.24.*
Pin-Priority: 1001
```

Example pinning by release:

```text
Package: *
Pin: release a=stable
Pin-Priority: 500

Package: *
Pin: release a=testing
Pin-Priority: 100
```

General priority behavior:

- `Pin-Priority` above `1000` can force downgrades or preferred selection.
- `500` is a normal default priority.
- Lower values reduce preference.
- Negative values effectively block selection.

Use pinning when you need version control across a repository mix, such as stable plus backports, or when you must keep one package on a known-good branch.

**Removal, purge, and cleanup**

Removing software is not always the same as deleting all traces of it. Package managers separate binary removal from configuration cleanup.

On Debian-based systems:

```bash
sudo apt remove nginx
sudo apt purge nginx
sudo apt autoremove
```

Differences:

- `apt remove` uninstalls the package but leaves system-wide configuration files in place when possible.
- `apt purge` removes the package and its configuration files managed by the package.
- `apt autoremove` removes dependencies that were installed automatically and are no longer needed.

On RPM-based systems:

```bash
sudo dnf remove nginx
sudo rpm -e nginx
```

On Arch:

```bash
sudo pacman -R nginx
sudo pacman -Rns nginx
```

The `-Rns` form removes the package, its unneeded dependencies, and configuration files that are no longer required.

Use purge when you want a clean reinstall or when stale configuration could cause the same problem to recur. Use remove when you want to preserve local configuration for later reuse.

**Dependency management and orphan cleanup**

Dependencies are central to package lifecycle management. A package may pull in libraries, helper tools, fonts, plugins, or runtime components. Over time, some of those dependencies become unused.

Examples of cleanup tools and patterns:

```bash
sudo apt autoremove
sudo dnf autoremove
sudo pacman -Rns $(pacman -Qdtq)
```

Notes:

- `apt autoremove` is the most common Debian/Ubuntu cleanup command for orphaned dependencies.
- `dnf autoremove` removes packages installed as dependencies that are no longer required.
- `pacman -Qdtq` lists orphaned dependencies on Arch-based systems.

Be careful with aggressive cleanup on shared systems. A package may appear unused locally but still be required by a manually installed application or a workflow outside the package manager’s view.

**Package database and cache troubleshooting**

When package operations fail, the cause is often stale metadata, broken dependencies, or database corruption.

Useful troubleshooting actions:

```bash
sudo apt update
sudo apt --fix-broken install
sudo dpkg --configure -a
sudo dnf clean all
sudo dnf makecache
sudo rpm --rebuilddb
```

What these do:

- `apt --fix-broken install` attempts to resolve incomplete dependency states.
- `dpkg --configure -a` finishes configuring unpacked packages.
- `dnf clean all` clears cached metadata and package files.
- `dnf makecache` refreshes metadata after cleanup.
- `rpm --rebuilddb` rebuilds the RPM database if queries or installs behave inconsistently.

If a package manager reports lock contention, another package process may still be running. Wait for it to finish before forcing recovery steps.

**Repository configuration and package sources**

Repositories are defined by configuration files and trust material. On Debian-based systems, sources may live in `/etc/apt/sources.list` and `/etc/apt/sources.list.d/`. On RPM-based systems, repository definitions often live in `/etc/yum.repos.d/`.

Example APT source entry:

```text
deb http://archive.ubuntu.com/ubuntu noble main restricted universe multiverse
```

Example DNF repository file:

```ini
[example-repo]
name=Example Repository
baseurl=https://repo.example.com/linux/$releasever/$basearch/
enabled=1
gpgcheck=1
gpgkey=https://repo.example.com/RPM-GPG-KEY-example
```

Trust matters because package managers should verify repository metadata and package signatures before installation. If signature verification is disabled, you lose an important defense against tampering.

**Snap lifecycle details**

Snap packages are self-contained and can include multiple revisions. This makes rollback and channel management practical, especially for desktop applications and some server tools.

Common commands:

```bash
snap list
snap info firefox
snap install firefox
snap refresh firefox
snap revert firefox
snap remove firefox
```

Lifecycle details to know:

- `snap list` shows installed snaps and their current revisions.
- `snap info` shows available channels such as `stable`, `candidate`, `beta`, and `edge`.
- `snap refresh` updates to a newer revision.
- `snap revert` rolls back to the previous revision if it is still available.
- `snap remove` uninstalls the snap.

You can also manage refresh timing:

```bash
sudo snap set system refresh.timer=mon,wed,fri,02:00-04:00
```

This is useful when you want updates during a maintenance window.

**Flatpak lifecycle details**

Flatpak focuses on desktop applications and sandboxing. It separates applications from runtimes, so lifecycle management includes both app updates and runtime cleanup.

Common commands:

```bash
flatpak list
flatpak info org.gimp.GIMP
flatpak install flathub org.gimp.GIMP
flatpak update
flatpak uninstall org.gimp.GIMP
flatpak uninstall --unused
```

Important concepts:

- `flatpak list` shows installed apps and runtimes.
- `flatpak update` updates both applications and runtimes.
- `flatpak uninstall --unused` removes runtimes and extensions no longer needed by installed apps.
- Permissions can be inspected and overridden with `flatpak override`.

Example permission management:

```bash
flatpak override --user --filesystem=home org.gimp.GIMP
flatpak override --show org.gimp.GIMP
```

Use overrides carefully because they can weaken sandbox isolation. Prefer the minimum permission needed for the task.

**Practical lifecycle workflow**

A disciplined package lifecycle usually follows the same pattern:

1. Identify the package and source.
2. Refresh metadata.
3. Check version, repository, and trust status.
4. Install or update.
5. Verify ownership and integrity.
6. Hold or pin if stability is required.
7. Remove or purge when the package is no longer needed.
8. Clean caches and orphaned dependencies.

Example Debian-based workflow:

```bash
sudo apt update
apt-cache policy nginx
sudo apt install nginx
dpkg -S /usr/sbin/nginx
sudo debsums -s nginx
sudo apt-mark hold nginx
sudo apt remove nginx
sudo apt purge nginx
sudo apt autoremove
sudo apt clean
```

This sequence shows the full lifecycle from discovery to cleanup.

**Exam focus and common distinctions**

- **Install vs update**: installation adds a package; update changes an installed package version.
- **Remove vs purge**: remove uninstalls; purge also removes package-managed configuration.
- **Hold vs pin**: hold blocks upgrades; pin controls version selection with priorities.
- **Metadata vs cache**: metadata describes available packages; cache stores local copies and lists.
- **Verification vs ownership**: verification checks integrity; ownership identifies which package provided a file.
- **Snap vs Flatpak**: both are sandboxed formats, but Snap uses revisions and channels, while Flatpak emphasizes runtimes and permissions.

**Key takeaways**

Package management is not just about installing software. It is about controlling software state over time: where packages come from, how they are verified, how updates are applied, and how you prevent unwanted changes.

If you can explain repository metadata, cache refresh, dependency resolution, holds, pinning, purge behavior, integrity checks, and sandboxed app lifecycle tools, you cover the core exam-level concepts for package lifecycle management.
