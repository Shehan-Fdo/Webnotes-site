## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**RPM fundamentals**

RPM (Red Hat Package Manager) is a low-level package format and toolchain used by many Linux distributions, especially Red Hat Enterprise Linux (RHEL), Fedora, CentOS Stream, and related systems. You use `rpm` to inspect package metadata, verify installed files, query ownership, and install or remove local `.rpm` files.

Unlike higher-level tools such as `dnf` or `yum`, `rpm` works directly with package files and the local RPM database. That makes it useful for inspection, troubleshooting, offline installs, and exam scenarios where you need to reason about package contents and signatures.

- **Package file**: a local `.rpm` archive containing payload and metadata.
- **RPM database**: the local database of installed packages, usually under `/var/lib/rpm`.
- **Verification**: checking signatures, checksums, file attributes, and package integrity.

**Core RPM lifecycle**

A typical RPM workflow is:

1. Inspect the package before installation.
2. Verify the package signature and metadata.
3. Install, upgrade, or reinstall the package.
4. Query the installed package and its files.
5. Remove the package when no longer needed.

`rpm` does not resolve dependencies automatically the way `dnf` does. If a package depends on other libraries or tools, you must satisfy those dependencies yourself or use a higher-level package manager.

> `rpm` is powerful but unforgiving: if you bypass dependency checks or install the wrong architecture, you can break application behavior or system consistency.

**Inspecting package files before installation**

Before installing, you should inspect the package to confirm what it contains and whether it matches your system.

Common inspection commands:

```bash
rpm -qpi package.rpm
rpm -qpl package.rpm
rpm -qpc package.rpm
rpm -qp --scripts package.rpm
rpm -qp --triggers package.rpm
```

What these do:

- `-qpi` shows package information such as name, version, release, architecture, summary, license, and vendor.
- `-qpl` lists files contained in the package payload.
- `-qpc` lists configuration files packaged as `%config`.
- `--scripts` displays pre/post install and uninstall scriptlets.
- `--triggers` shows trigger scripts that run in response to package events.

These options are especially useful when you need to understand side effects before installation. For example, a package may create users, restart services, or modify configuration files through scriptlets.

**Signature and integrity verification**

RPM supports package signature verification using GPG (GNU Privacy Guard) keys. Before trusting a package from a repository or third party, you should verify the signature and import the vendor key if appropriate.

Typical commands:

```bash
rpm -K package.rpm
rpm --checksig package.rpm
rpm --import /etc/pki/rpm-gpg/RPM-GPG-KEY-example
rpm -qa gpg-pubkey*
rpm -qi gpg-pubkey
```

Key points:

- `-K` and `--checksig` validate the package signature and digest.
- `rpm --import` adds a public key to the RPM keyring for signature verification.
- `rpm -qa gpg-pubkey*` lists imported public keys as installed pseudo-packages.
- `rpm -qi gpg-pubkey` shows key metadata.

If a package is unsigned or signed by an unknown key, treat it as untrusted until you confirm the source. For exam and admin practice, know that signature verification is separate from dependency resolution.

**Installing packages**

The basic install command is:

```bash
sudo rpm -ivh package.rpm
```

Common flags:

- `-i` installs a new package.
- `-v` increases verbosity.
- `-h` shows hash marks for progress.

Useful install-related options:

- `--test` performs a dry run without writing changes.
- `--nodeps` ignores dependency checks.
- `--replacepkgs` reinstalls an already installed package.
- `--replacefiles` allows file conflicts to be overwritten.
- `--noscripts` skips package scriptlets.
- `--excludedocs` omits documentation files.

Example dry run:

```bash
sudo rpm -ivh --test package.rpm
```

Example forced install in a lab scenario:

```bash
sudo rpm -ivh --nodeps package.rpm
```

Use `--nodeps` only when you understand the consequences. It can leave the system in a broken state if required libraries or tools are missing.

**Upgrading and reinstalling**

RPM distinguishes between install and upgrade behavior.

```bash
sudo rpm -Uvh package.rpm
sudo rpm -Fvh package.rpm
sudo rpm -ivh --replacepkgs package.rpm
```

Comparison:

| Action | Command | Behavior |
|---|---|---|
| Install | `rpm -ivh` | Installs a package if it is not already present |
| Upgrade | `rpm -Uvh` | Installs new version and removes older version if applicable |
| Freshen | `rpm -Fvh` | Upgrades only if an older version is already installed |
| Reinstall | `rpm -ivh --replacepkgs` | Reinstalls the same version |

Use `-U` when you want to move to a newer version and preserve upgrade semantics. Use `-F` when you want to avoid installing packages that are not already present.

**Removing packages**

Package removal is part of the RPM lifecycle and is often tested alongside installation.

```bash
sudo rpm -e package-name
```

Important details:

- Use the package name, not the `.rpm` filename.
- RPM refuses removal if other installed packages depend on it, unless you force around dependency checks with other tools or unsafe methods.
- Removing a package does not automatically remove configuration files marked as preserved or user-created data outside the package.

To identify the exact installed package name:

```bash
rpm -qa | grep '^package-name'
rpm -q package-name
```

**Querying installed packages**

RPM query mode is one of its most important exam topics. You use it to inspect what is installed, where files live, and which package owns a file.

Common queries:

```bash
rpm -q package-name
rpm -qa
rpm -qi package-name
rpm -ql package-name
rpm -qc package-name
rpm -qd package-name
rpm -qf /usr/bin/command
rpm -q --scripts package-name
rpm -q --triggers package-name
```

What they return:

- `-q` queries a package by name.
- `-qa` lists all installed packages.
- `-qi` shows detailed package information.
- `-ql` lists installed files.
- `-qc` lists configuration files.
- `-qd` lists documentation files.
- `-qf` identifies which package owns a file.
- `--scripts` and `--triggers` inspect package behavior.

Example:

```bash
rpm -qf /usr/bin/ssh
rpm -ql openssh-server
rpm -qi bash
```

**Query formatting for precise output**

`--queryformat` lets you customize output for scripting or targeted inspection. This is useful when you need only specific fields.

Example:

```bash
rpm -q --queryformat '%{NAME} %{VERSION}-%{RELEASE}.%{ARCH}\n' bash
rpm -q --queryformat 'Package: %{NAME}\nVendor: %{VENDOR}\nInstall Date: %{INSTALLTIME:date}\n' bash
```

Common tags include:

- `%{NAME}`
- `%{VERSION}`
- `%{RELEASE}`
- `%{ARCH}`
- `%{SUMMARY}`
- `%{INSTALLTIME}`
- `%{LICENSE}`
- `%{VENDOR}`

Use `--queryformat` when you need machine-friendly output or when the default query output is too verbose.

**Package scripts and triggers**

RPM packages can include scriptlets that run during install, upgrade, and removal. These can create users, update caches, regenerate indexes, or restart services.

Common scriptlet types:

- `%pre` runs before installation.
- `%post` runs after installation.
- `%preun` runs before removal.
- `%postun` runs after removal.
- `%trigger` runs when related packages are installed, upgraded, or removed.

Inspection example:

```bash
rpm -qp --scripts package.rpm
rpm -q --scripts installed-package
rpm -qp --triggers package.rpm
```

Why this matters:

- Scriptlets can explain unexpected behavior during install.
- Trigger scripts can affect packages that were not directly installed.
- Troubleshooting often requires checking whether a package modified system state during these phases.

**Verifying installed packages**

After installation, you can verify package integrity against the RPM database.

```bash
rpm -V package-name
rpm -Va
```

Verification checks may report differences in:

- file size
- permissions
- ownership
- checksum
- modification time
- device type
- symlink target
- capabilities

A verification result does not always mean corruption. It may reflect legitimate local changes, such as edited configuration files or updated timestamps. Focus on whether the changed file is expected.

**Troubleshooting dependency and repository issues**

Dependency failures are common when installing local RPMs directly.

Example failure:

```bash
sudo rpm -ivh app.rpm
# error: Failed dependencies:
#   libexample.so.1()(64bit) is needed by app-1.0-1.x86_64
```

Ways to handle this:

- Install the missing dependency package first.
- Use `dnf install ./package.rpm` when repository resolution is appropriate.
- Confirm architecture compatibility with `rpm -qpi package.rpm`.
- Check whether the package was built for a different distribution release.

For third-party packages, verify the source and import the correct GPG key before installation:

```bash
sudo rpm --import https://vendor.example.com/RPM-GPG-KEY-vendor
rpm -K package.rpm
```

If the package comes from a repository, the repository metadata and key management are usually handled by `dnf`, but the underlying RPM signature trust model still matters.

**RPM database locations and recovery**

The RPM database stores installed package metadata, and corruption can prevent queries or package operations.

Common location:

- `/var/lib/rpm`

If the database becomes inconsistent, recovery may involve rebuilding the database. A common administrative approach is:

```bash
sudo rm -f /var/lib/rpm/__db*
sudo rpm --rebuilddb
```

Notes:

- `__db*` files are Berkeley DB environment files on systems that still use them.
- `--rebuilddb` reconstructs the database from installed package headers.
- Always back up the RPM database or system state before recovery work when possible.

If queries fail unexpectedly, check filesystem health, available disk space, and whether the database directory is readable and writable.

**Practical inspection workflow**

A safe workflow for a local package is:

1. Inspect metadata with `rpm -qpi`.
2. List files with `rpm -qpl`.
3. Check scripts with `rpm -qp --scripts`.
4. Verify signature with `rpm -K`.
5. Perform a dry run with `rpm -ivh --test`.
6. Install with `rpm -ivh` or upgrade with `rpm -Uvh`.
7. Confirm installation with `rpm -q` and `rpm -V`.

Example:

```bash
rpm -qpi nginx-1.26.0-1.el9.x86_64.rpm
rpm -qpl nginx-1.26.0-1.el9.x86_64.rpm
rpm -K nginx-1.26.0-1.el9.x86_64.rpm
sudo rpm -ivh --test nginx-1.26.0-1.el9.x86_64.rpm
sudo rpm -ivh nginx-1.26.0-1.el9.x86_64.rpm
rpm -q nginx
rpm -V nginx
```

**Exam-focused distinctions**

- `rpm` works directly with package files and the local database.
- `dnf` resolves dependencies and is usually preferred for normal package management.
- `-i` installs, `-U` upgrades, `-F` freshens, and `-e` removes.
- `--test` is a dry run.
- `--nodeps` bypasses dependency checks and is risky.
- `--import` adds trusted GPG keys for signature validation.
- `--queryformat` gives precise output for scripting and inspection.
- `--scripts` and `--triggers` reveal package behavior beyond file lists.

**Quick reference**

```bash
# Inspect a package file
rpm -qpi package.rpm
rpm -qpl package.rpm
rpm -qp --scripts package.rpm

# Verify signature
rpm -K package.rpm
rpm --import /etc/pki/rpm-gpg/RPM-GPG-KEY-example

# Install, upgrade, remove
rpm -ivh package.rpm
rpm -Uvh package.rpm
rpm -e package-name

# Query installed packages
rpm -qa
rpm -qi package-name
rpm -ql package-name
rpm -qf /path/to/file

# Verify installed files
rpm -V package-name

# Repair database
rpm --rebuilddb
```
