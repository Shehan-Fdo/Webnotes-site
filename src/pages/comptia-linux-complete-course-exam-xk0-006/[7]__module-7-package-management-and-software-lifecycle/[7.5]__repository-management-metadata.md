## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Repository fundamentals**

A software repository is a trusted source of packages plus the metadata clients need to discover, verify, and install them. On Debian-based systems, APT uses package indexes such as `Packages`, `Release`, and signature files. On RPM-based systems, DNF/YUM uses repository configuration files and `repodata/` metadata generated from RPM payloads.

Repository management matters because it controls:

- **Availability** of packages for a given distribution release and architecture
- **Integrity** through cryptographic signatures and checksums
- **Consistency** across fleets by pinning, mirroring, and snapshots
- **Automation** for updates, rollbacks, and offline installs

A package manager does not install from a repository directory alone; it relies on metadata to know what exists, what version is current, and whether the source is trusted.

**APT repository structure and client workflow**

APT repositories are usually organized by distribution codename, component, and architecture. A common layout looks like this:

```text
dists/
  bookworm/
    main/
      binary-amd64/
        Packages.gz
        Release
pool/
  main/
    p/
      package-name/
        package-name_1.2.3_amd64.deb
```

APT clients read metadata from `Release` and `Packages` indexes, then download the referenced `.deb` files from the `pool/` area or another published path. The `Packages` file lists package names, versions, dependencies, checksums, and file locations.

Useful commands:

```bash
apt-cache policy
apt-cache madison nginx
apt list --upgradable
```

- `apt-cache policy` shows candidate versions and repository priorities.
- `apt-cache madison` is useful when you want to compare versions across sources.
- `apt list --upgradable` helps confirm what metadata the client currently sees.

**Adding APT repositories**

You can add repositories manually or with helper tools. The helper workflow is common on desktops and in exam scenarios.

```bash
sudo add-apt-repository ppa:graphics-drivers/ppa
sudo apt update
```

For a standard third-party repository, you often add a source list entry:

```bash
echo "deb [arch=amd64 signed-by=/usr/share/keyrings/vendor.gpg] https://repo.example.com/debian bookworm main" \
  | sudo tee /etc/apt/sources.list.d/vendor.list
sudo apt update
```

Key points:

- **`add-apt-repository`** is a convenience wrapper that can add PPAs or repository entries depending on the distro.
- **`apt-add-repository`** is often the same command name on some systems via a symlink or package-provided alias.
- Modern APT prefers per-repository keyrings referenced with `signed-by=`.
- Repository entries should match the correct suite/codename, component, and architecture.

**APT key management and trust model**

APT historically used `apt-key`, but that model is deprecated. The modern approach is to store repository keys in a dedicated keyring and reference them from the source entry.

Deprecated pattern:

```bash
sudo apt-key add vendor.asc
```

Modern pattern:

```bash
curl -fsSL https://repo.example.com/vendor.gpg | sudo gpg --dearmor -o /usr/share/keyrings/vendor.gpg
echo "deb [signed-by=/usr/share/keyrings/vendor.gpg] https://repo.example.com/debian bookworm main" \
  | sudo tee /etc/apt/sources.list.d/vendor.list
sudo apt update
```

Important details:

- **`apt-key`** is deprecated because it trusts keys globally for all repositories.
- Per-repository keyrings reduce the blast radius of a compromised key.
- You should verify the key fingerprint out of band when possible.
- A repository signature protects metadata, not necessarily the package payload alone; the client verifies both metadata and package checksums during normal operation.

> Do not rely on `apt-key` for new deployments. Use `/usr/share/keyrings/` or another dedicated keyring path and `signed-by=` in the source definition.

**APT pinning and package selection**

APT pinning lets you influence which repository or version APT prefers. This is useful when you mix stable, testing, backports, or vendor repositories.

Example pin file:

```bash
sudo tee /etc/apt/preferences.d/vendor.pref >/dev/null <<'EOF'
Package: *
Pin: origin repo.example.com
Pin-Priority: 100
EOF
```

Another example:

```bash
sudo tee /etc/apt/preferences.d/stable.pref >/dev/null <<'EOF'
Package: *
Pin: release a=stable
Pin-Priority: 700
EOF
```

Interpretation:

- **Pin-Priority** affects candidate selection, but behavior depends on whether the package is installed, available, or would require a downgrade.
- Higher values generally win, but APT has special thresholds for installing, upgrading, and downgrading.
- `Pin: release a=stable` matches release metadata fields such as archive/suite values, so confirm the exact field names in the repository metadata.
- Use pinning carefully when mixing repositories to avoid accidental dependency drift.

Practical inspection:

```bash
apt-cache policy package-name
```

This shows the candidate version and the priority assigned to each source.

**Creating a local APT repository**

A local repository is useful for offline systems, controlled rollouts, or internal package distribution. A simple flat repository can be created from a directory of `.deb` files.

Example workflow:

```bash
mkdir -p /srv/repo/debs
cp *.deb /srv/repo/debs/
cd /srv/repo/debs
dpkg-scanpackages . /dev/null | gzip -9c > Packages.gz
```

Then publish it over HTTP or file access:

```bash
python3 -m http.server 8000
```

Client entry:

```bash
echo "deb [trusted=yes] http://repo-host:8000/ ./" | sudo tee /etc/apt/sources.list.d/local.list
sudo apt update
```

Notes:

- `dpkg-scanpackages` scans a directory of `.deb` files and generates a `Packages` index for a flat layout.
- For a more complete repository, you usually build a `dists/` structure and sign the metadata.
- `trusted=yes` bypasses signature checks and is appropriate only for controlled lab use, not production.

A more production-like APT publish flow often includes:

1. Build or collect `.deb` packages.
2. Generate `Packages` and `Release` metadata.
3. Sign the `Release` file to produce `InRelease` or `Release.gpg`.
4. Publish the repository via HTTP/HTTPS.
5. Point clients at the correct suite and component.

Example signing flow:

```bash
apt-ftparchive packages pool/main > dists/bookworm/main/binary-amd64/Packages
gzip -9c dists/bookworm/main/binary-amd64/Packages > dists/bookworm/main/binary-amd64/Packages.gz
apt-ftparchive release dists/bookworm > dists/bookworm/Release
gpg --default-key "Repo Signing Key" --clearsign -o dists/bookworm/InRelease dists/bookworm/Release
gpg --default-key "Repo Signing Key" -abs -o dists/bookworm/Release.gpg dists/bookworm/Release
```

**RPM repository structure and DNF workflow**

RPM-based systems use repository metadata stored under `repodata/`. The repository usually contains RPM files in a publish directory and metadata generated from those packages.

Typical layout:

```text
repo/
  x86_64/
    package-name-1.2.3-1.el9.x86_64.rpm
  repodata/
    repomd.xml
    *.xml.gz
```

DNF reads `repomd.xml`, which points to the metadata files describing package names, dependencies, checksums, and file lists.

Useful commands:

```bash
dnf repolist
dnf repolist all
dnf info package-name
dnf list available
```

- `dnf repolist` confirms which repositories are enabled.
- `dnf repolist all` shows enabled and disabled repositories.
- `dnf info` and `dnf list available` help verify whether metadata is current.

**DNF repository configuration**

Repository definitions live in `.repo` files under `/etc/yum.repos.d/` or a similar vendor path.

Example:

```ini
[internal-baseos]
name=Internal BaseOS
baseurl=https://repo.example.com/rhel/9/BaseOS/x86_64/os/
enabled=1
gpgcheck=1
gpgkey=https://repo.example.com/keys/RPM-GPG-KEY-example
skip_if_unavailable=0
exclude=kernel*
```

Common options:

- **`baseurl`** points to a specific repository location.
- **`mirrorlist`** points to a service that returns a list of mirrors.
- **`enabled`** controls whether the repo is active.
- **`gpgcheck`** enables package signature verification.
- **`repo_gpgcheck`** verifies repository metadata signatures when supported and configured by the distro and DNF/YUM version.
- **`skip_if_unavailable`** allows the system to continue if the repo is unreachable.
- **`exclude`** prevents selected packages from being installed or updated from that repo.
- **`enabled_metadata`** is not a universal option; if present in a distro-specific build or plugin context, confirm its behavior before relying on it.

**Managing DNF repositories**

The `dnf config-manager` workflow is common for enabling, disabling, and editing repositories. On many systems, it comes from the `dnf-plugins-core` package.

Examples:

```bash
sudo dnf install -y dnf-plugins-core
sudo dnf config-manager --add-repo https://repo.example.com/example.repo
sudo dnf config-manager --set-enabled internal-baseos
sudo dnf config-manager --set-disabled internal-testing
```

You can also inspect the generated repo file:

```bash
cat /etc/yum.repos.d/example.repo
```

Practical notes:

- Install the plugin package before expecting `config-manager` to exist.
- `--add-repo` can import a remote `.repo` file or create a local entry.
- Use `--setopt` for temporary overrides during troubleshooting or testing.

Example temporary override:

```bash
sudo dnf --setopt=internal-baseos.skip_if_unavailable=1 makecache
```

**Generating RPM repository metadata**

On RPM systems, `createrepo_c` generates the metadata that DNF consumes.

Basic workflow:

```bash
sudo dnf install -y createrepo_c
createrepo_c /srv/repo/rpms
```

If you add or remove packages later, update the metadata:

```bash
createrepo_c --update /srv/repo/rpms
```

Metadata maintenance tools:

- **`createrepo_c`** creates or refreshes `repodata/`.
- **`modifyrepo_c`** can add supplemental metadata files to an existing repository.
- `sqlite` metadata may be generated depending on tool options and client compatibility needs.

Example of adding custom metadata:

```bash
modifyrepo_c /tmp/comps.xml /srv/repo/rpms/repodata
```

This is useful when you need to publish group information or other repository metadata alongside package indexes.

**RPM signing and verification**

RPM package signing protects package integrity and origin. DNF verifies signatures when `gpgcheck=1` is set and the public key is available.

Import a repository key:

```bash
sudo rpm --import https://repo.example.com/keys/RPM-GPG-KEY-example
rpm -qa gpg-pubkey\*
```

Verify a package manually:

```bash
rpm -K package-name.rpm
rpm -Kv package-name.rpm
```

Sign packages during build or release workflows:

```bash
rpm --addsign package-name.rpm
```

Repository metadata signing may also be used, but support and defaults vary by distribution and toolchain. Treat `repo_gpgcheck` as an additional control, not a universal assumption.

> Always verify the key fingerprint before importing a repository key in production. A valid signature from the wrong key is still a trust failure.

**Mirrors, snapshots, and publish workflows**

A mirror copies repository content from an upstream source. A snapshot preserves a point-in-time state so clients can install reproducible versions later. These are related but not identical.

Comparison:

| Concept | Behavior | Typical use |
|---|---|---|
| Mirror | Follows upstream changes over time | Bandwidth reduction, local caching |
| Snapshot | Freezes a repository state | Reproducible builds, rollback, compliance |
| Publish | Exposes curated content to clients | Internal package distribution |

APT snapshot-style workflow example:

```bash
rsync -aH --delete rsync://mirror.example.com/debian/ /srv/mirror/debian/
apt-ftparchive release /srv/mirror/debian/dists/bookworm > /srv/mirror/debian/dists/bookworm/Release
gpg --clearsign -o /srv/mirror/debian/dists/bookworm/InRelease /srv/mirror/debian/dists/bookworm/Release
```

In a real snapshot system, you usually publish a dated or versioned path such as `/srv/repo/snapshots/2026-04-26/` and keep that path immutable. Clients then point to that snapshot rather than a moving target.

RPM mirror workflow example:

```bash
reposync -p /srv/mirror/rhel9 --repoid=internal-baseos --download-metadata
createrepo_c --update /srv/mirror/rhel9/internal-baseos
```

If you mirror metadata, ensure the metadata and packages stay in sync. A stale `repodata/` directory can cause checksum or dependency errors.

**Troubleshooting repository problems**

Repository failures usually fall into a few categories: bad URLs, bad signatures, stale metadata, architecture mismatch, or proxy/network issues.

Common checks:

```bash
apt update
dnf makecache
curl -I https://repo.example.com/
```

APT-specific checks:

```bash
grep -R "deb " /etc/apt/sources.list /etc/apt/sources.list.d/
apt-cache policy
```

DNF-specific checks:

```bash
dnf repolist -v
dnf clean all
dnf makecache
```

Common failure cases:

- **Bad `baseurl` or `mirrorlist` syntax** causes DNF to fail before metadata download.
- **Architecture mismatch** occurs when the repo publishes `x86_64` packages but the client expects `aarch64`, or when the APT `arch=` filter is wrong.
- **Expired or rotated GPG keys** cause signature verification failures until the new key is installed.
- **Stale metadata** can produce missing dependency or checksum errors after packages are added or removed.
- **Proxy or TLS issues** can block metadata retrieval even when the repository is otherwise correct.

Useful diagnostics:

```bash
apt-cache policy package-name
dnf repoquery package-name
rpm -q --qf '%{NAME} %{VERSION}-%{RELEASE}\n' package-name
```

If a repository fails signature verification, check the key fingerprint:

```bash
gpg --show-keys --fingerprint /usr/share/keyrings/vendor.gpg
rpm -qi gpg-pubkey
```

**Practical administration patterns**

For enterprise systems, repository management is usually a controlled pipeline:

1. Build or collect packages.
2. Scan or generate metadata.
3. Sign packages and/or repository metadata.
4. Publish to a web server, object store, or file share.
5. Point clients to the correct repository definition.
6. Validate access, signatures, and package visibility.
7. Refresh metadata on clients with `apt update` or `dnf makecache`.

A simple internal repository server might use:

```bash
sudo mkdir -p /srv/repo
sudo chown -R www-data:www-data /srv/repo
sudo systemctl enable --now nginx
```

Then publish the repository over HTTPS and restrict access with firewall rules, authentication, or network segmentation as needed.

**Exam focus reminders**

- Know the difference between **package payloads** and **repository metadata**.
- Know how APT and DNF discover repositories and verify trust.
- Know that `apt-key` is deprecated and that per-repository keyrings are preferred.
- Know that `createrepo_c` builds RPM metadata and `modifyrepo_c` can extend it.
- Know how mirrors differ from snapshots.
- Know how to troubleshoot with `apt-cache policy`, `apt update`, `dnf repolist -v`, and `dnf makecache`.
- Know that signing and metadata freshness are separate concerns.
