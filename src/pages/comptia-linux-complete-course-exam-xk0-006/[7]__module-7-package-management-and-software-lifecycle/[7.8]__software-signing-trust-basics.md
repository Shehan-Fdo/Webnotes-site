**Software signing and trust model**

Linux package trust is built on a chain of confidence, not on blind installation. You usually trust a repository key, then trust the repository metadata signed by that key, and then trust the package hashes listed in that metadata. This matters because a package manager can confirm that what it downloaded matches what the repository published, but that does not automatically mean the software is safe or benign.

A practical threat model helps here:

- A mirror can be compromised and serve altered files.
- A man-in-the-middle attack can tamper with downloads if transport security is weak.
- A malicious or stolen signing key can make harmful content look legitimate.
- A vendor update server can be breached and distribute trojanized packages.

The goal of signing is integrity and authenticity:

- **Integrity** means the file was not changed after signing.
- **Authenticity** means the signer is who you expect, assuming you trust the key.

**Core concepts**

A few terms appear repeatedly in package trust workflows:

- **Public key cryptography** uses a key pair: a private key to sign and a public key to verify.
- **Digital signature** proves that a file or metadata object was signed by the private key holder and has not changed.
- **Checksum** is a hash value such as SHA256 used to detect accidental or malicious modification.
- **Repository metadata** is the signed index that tells the package manager which packages exist and what hashes they should have.
- **Trust anchor** is the key or certificate you accept as the basis for verification.

A checksum alone is not a signature. A checksum can tell you a file changed, but it does not tell you who created it. A signature gives you both integrity and a trust relationship through the signer’s public key.

**How Linux package trust works**

Different package ecosystems implement the same basic idea in slightly different ways.

- **APT** on Debian and Ubuntu verifies repository metadata, typically the inline-signed `InRelease` file first, and falls back to `Release` plus `Release.gpg` when needed.
- **DNF/YUM** on RPM-based systems verifies repository metadata and can import repository signing keys from configured locations.
- **RPM** can verify a local package file’s signature and digest with tools such as `rpm --checksig`.
- **GPG** is the underlying OpenPGP tool commonly used to sign and verify release files and keys.

The important exam distinction is this:

- Package managers usually verify the repository metadata first.
- The metadata contains hashes for packages.
- The package manager then checks the downloaded package against those hashes.
- The package file itself is not always individually signature-checked during install in the way many learners assume.

> APT does not generally verify the package payload directly as a signed object during install; it verifies signed repository metadata and then trusts the hashes listed there.

**APT trust workflow**

APT uses repository signing to protect package installation and updates. The repository publishes signed metadata, and APT checks that metadata before it trusts package lists and package hashes.

Common files and concepts:

- `InRelease` is a single file containing repository metadata and an inline signature.
- `Release` is the metadata file.
- `Release.gpg` is the detached signature for `Release`.
- `trusted.gpg.d/` is a directory used for trusted key files on many systems.
- `apt-key` is deprecated for normal repository management and should not be your preferred workflow.

A modern repository setup often uses a dedicated keyring file and a `signed-by=` option in the source entry.

Example source entry:

```bash
deb [signed-by=/usr/share/keyrings/vendor-archive-keyring.gpg] https://repo.example.com/debian stable main
```

Typical verification flow:

1. APT downloads `InRelease` if available.
2. If `InRelease` is unavailable, APT uses `Release` and `Release.gpg`.
3. APT verifies the repository signature with the trusted key.
4. APT checks package hashes from the signed metadata.
5. APT installs the package only if the metadata and hashes are valid.

Useful commands:

```bash
apt update
apt-cache policy package-name
apt show package-name
```

To inspect trusted keys on older systems, you may still see:

```bash
apt-key list
```

But for study and administration, remember the modern pattern:

- Store vendor keys in `/usr/share/keyrings/` or `/etc/apt/keyrings/`.
- Reference them with `signed-by=`.
- Avoid adding broad trust to the global keyring unless required.

**RPM, DNF, and YUM trust workflow**

RPM-based systems use repository metadata and package signatures in a similar trust chain. DNF and YUM manage repository metadata verification, while `rpm` can inspect a local package file.

Useful commands:

```bash
rpm --checksig package.rpm
rpm -Kv package.rpm
dnf repolist
dnf update
```

What these do:

- `rpm --checksig package.rpm` checks the package signature and digests on the local file.
- `rpm -Kv package.rpm` is a verbose verification form.
- `dnf` verifies repository metadata and uses configured keys for repository trust.
- `dnf` may prompt you to import a repository key the first time it sees a new signed repository.

A repository file may include a `gpgkey=` line:

```ini
[example]
name=Example Repo
baseurl=https://repo.example.com/rpm/$basearch
enabled=1
gpgcheck=1
repo_gpgcheck=1
gpgkey=https://repo.example.com/RPM-GPG-KEY-example
```

Key points:

- `gpgcheck=1` enables package signature checking.
- `repo_gpgcheck=1` enables repository metadata signature checking.
- `gpgkey=` points to a URL or file location from which the key can be imported.
- The key is not automatically trusted just because the line exists; the package manager must import it and the user must accept it, or it must already be trusted.

**Signature verification vs checksum verification**

These are related but not identical.

| Aspect | Signature verification | Checksum verification |
|---|---|---|
| Purpose | Proves authenticity and integrity | Detects file changes |
| Trust source | Public key | Published hash value |
| Typical tools | `gpg`, `rpm --checksig`, package manager metadata checks | `sha256sum`, `sha512sum` |
| Best use | Repository metadata, signed release files, signed packages | Confirming a download matches a vendor-published hash |

A common real-world workflow is to verify both:

1. Download the software archive.
2. Download the vendor checksum file.
3. Download the vendor signature for the checksum or release file.
4. Verify the signature.
5. Verify the archive hash matches the checksum.

Example:

```bash
gpg --verify SHA256SUMS.sig SHA256SUMS
sha256sum -c SHA256SUMS
```

This pattern is useful when a vendor publishes a tarball outside a package manager. The signature protects the checksum file, and the checksum file protects the archive.

**Manual verification with GPG**

`gpg` is the standard tool for OpenPGP signatures. It is commonly used to verify release files, checksums, and detached signatures.

Detached signature example:

```bash
gpg --verify Release.gpg Release
```

Inline-signed file example:

```bash
gpg --verify InRelease
```

If you need to inspect a public key before trusting it, you can use:

```bash
gpg --show-keys vendor-key.asc
gpg --fingerprint vendor@example.com
```

Why fingerprints matter:

- A key ID alone is too short to trust in high-risk situations.
- A fingerprint is a stronger identifier for confirming the correct key.
- You should compare fingerprints from an independent source when possible.

**Trust management and key hygiene**

Trust is only as strong as key management. A valid signature from a compromised key is still dangerous.

Good practices:

- Keep vendor keys scoped to the repository that needs them.
- Rotate or remove keys when vendors change signing infrastructure.
- Review repository files for unexpected URLs or key locations.
- Prefer HTTPS for key and metadata transport, even though signatures provide the real trust guarantee.
- Use least privilege for repository trust instead of importing keys globally.

Common storage locations:

- `/usr/share/keyrings/` for packaged keyrings
- `/etc/apt/keyrings/` for administrator-managed APT keys
- `/etc/pki/rpm-gpg/` for RPM-based key material on many distributions

A practical APT repository setup might look like this:

```bash
curl -fsSL https://repo.example.com/keys/vendor.asc | gpg --dearmor | sudo tee /usr/share/keyrings/vendor-archive-keyring.gpg >/dev/null
echo "deb [signed-by=/usr/share/keyrings/vendor-archive-keyring.gpg] https://repo.example.com/debian stable main" | sudo tee /etc/apt/sources.list.d/vendor.list
sudo apt update
```

**When to use each tool**

Choose the tool based on what you are verifying:

- Use `apt update` when you want the package manager to validate repository metadata and refresh package lists.
- Use `rpm --checksig` when you want to inspect a downloaded RPM file before installation.
- Use `gpg --verify` when you have a signed release file, checksum file, or detached signature.
- Use `sha256sum -c` when you already trust the checksum file and want to confirm the download matches it.

This distinction is important in incident response and troubleshooting. For example, if a package manager refuses an update, you need to know whether the failure is due to:

- an expired or missing repository key,
- a changed repository metadata signature,
- a mismatched package hash,
- or a corrupted download.

**Troubleshooting trust failures**

Common errors often point to a specific layer in the trust chain.

- Missing public key:
  - The repository key is not installed or not imported.
  - The package manager cannot verify metadata.
- Signature mismatch:
  - The metadata or package changed after signing.
  - The mirror is stale or compromised.
- Hash mismatch:
  - The downloaded file does not match the signed metadata.
  - The cache or mirror may be corrupted.
- Expired key:
  - The vendor rotated keys or the key lifetime ended.
  - You may need to import a new key and remove the old one.

Example troubleshooting commands:

```bash
apt update
apt policy
dnf repolist -v
rpm -q gpg-pubkey
gpg --verify Release.gpg Release
sha256sum package.tar.gz
```

If a repository suddenly fails after working for months, suspect one of these:

- key rotation by the vendor,
- expired metadata,
- a changed repository URL,
- or a local cache issue.

**Security scenarios to remember**

A few realistic incidents make the trust model easier to remember:

- A compromised mirror serves a modified package, but signature verification blocks it because the metadata hash no longer matches.
- An attacker steals a repository signing key, and the package manager accepts malicious updates until the key is revoked or replaced.
- A user downloads a tarball from a vendor site over HTTP, and the file is altered in transit; a checksum alone would not prove authenticity, but a verified signature would.
- A stale local cache causes a hash mismatch after a repository update, which looks like corruption but is actually an out-of-sync metadata problem.

These scenarios show why signatures matter more than plain hashes when you need to know who published the file.

**Exam focus points**

For Linux+ exam prep, focus on these distinctions:

- **Signature** means authenticity plus integrity.
- **Checksum** means integrity only.
- **Repository metadata** is what package managers usually trust first.
- **APT** relies on signed metadata such as `InRelease` or `Release` plus `Release.gpg`.
- **RPM** can verify local package signatures with `rpm --checksig`.
- **DNF/YUM** manage repository trust and key import for RPM-based systems.
- **`apt-key`** is legacy; modern APT uses per-repository keyrings and `signed-by=`.
- **`gpg --verify`** is the standard manual verification tool for signed files.

**Quick reference**

```bash
# APT
apt update
apt-cache policy package-name

# RPM / DNF
rpm --checksig package.rpm
dnf update

# GPG
gpg --verify Release.gpg Release
gpg --verify InRelease

# Checksums
sha256sum -c SHA256SUMS
```
