**APT on Ubuntu: the package-management workflow**

`apt` is the high-level package manager you use on Ubuntu for most day-to-day software tasks. It works with Debian package metadata, resolves dependencies, and pulls packages from configured repositories. In practice, you use it to refresh package lists, install and remove software, upgrade systems, inspect package details, and manage repository sources.

A good workflow matters because package operations affect system stability. You want to update package indexes before installing, verify what will change before upgrading, and understand how to recover when a repository or package is unavailable.

- **APT** stands for Advanced Package Tool.
- `apt` is the user-facing command; lower-level tools such as `dpkg` handle local `.deb` package installation and database operations.
- Ubuntu repositories are usually split into components such as `main`, `universe`, `restricted`, and `multiverse`, depending on support and licensing.

**Core `apt` workflow**

The most common sequence is to refresh metadata, inspect available changes, then install or upgrade packages.

```bash
sudo apt update
apt list --upgradable
sudo apt upgrade
```

- `apt update` downloads the latest package indexes from configured repositories.
- `apt upgrade` installs newer versions of already installed packages without removing packages to satisfy dependencies.
- `apt full-upgrade` may remove or replace packages if needed to complete dependency changes.

Use `apt` instead of `apt-get` for interactive administration because `apt` provides clearer output and combines common workflows. For scripts, `apt-get` is still often preferred for stable output, but `apt` is acceptable in many modern environments if you control the execution context.

**Repository metadata and package discovery**

Before installing anything, you should know what package versions are available and where they come from. Ubuntu stores repository definitions in `/etc/apt/sources.list` and `/etc/apt/sources.list.d/`.

```bash
apt-cache policy nginx
apt-cache search container runtime
apt show curl
```

- `apt-cache policy` shows installed, candidate, and repository versions.
- `apt-cache search` finds packages by name or description.
- `apt show` displays package metadata such as dependencies, maintainer, and description.

If you need to inspect the exact repository configuration, check the source files directly.

```bash
cat /etc/apt/sources.list
ls /etc/apt/sources.list.d/
```

A typical Ubuntu source entry looks like this:

```text
deb http://archive.ubuntu.com/ubuntu jammy main restricted universe multiverse
deb http://archive.ubuntu.com/ubuntu jammy-updates main restricted universe multiverse
deb http://security.ubuntu.com/ubuntu jammy-security main restricted universe multiverse
```

- `deb` indicates a binary package repository.
- The distribution codename identifies the release, such as `jammy` or `noble`.
- Repository components control which package sets are available.

**Installing packages**

You install packages with `apt install`. This command resolves dependencies automatically and prompts before making changes unless you use noninteractive options.

```bash
sudo apt install vim
sudo apt install nginx curl
```

- Installing multiple packages in one command reduces repeated dependency resolution.
- `apt` may install additional libraries or helper packages required by the requested software.
- You can install a local `.deb` file directly, and `apt` will attempt to resolve dependencies from configured repositories.

```bash
sudo apt install ./example-package_1.2.3_amd64.deb
```

When you need to confirm what will happen before committing, use simulation mode.

```bash
apt install --simulate nginx
```

- `--simulate` shows the planned actions without changing the system.
- This is useful for change review and troubleshooting dependency conflicts.

**Upgrading packages and the system**

Package upgrades are a routine maintenance task. `apt upgrade` updates installed packages to newer versions that do not require removing packages. `apt full-upgrade` is more aggressive and can remove packages if necessary to satisfy dependency changes.

```bash
sudo apt update
sudo apt upgrade
sudo apt full-upgrade
```

- Run `apt update` first so the package lists are current.
- Use `apt upgrade` for conservative maintenance.
- Use `apt full-upgrade` when kernel, library, or meta-package transitions require dependency changes.

You can also upgrade a single package.

```bash
sudo apt install --only-upgrade openssh-server
```

This is useful when you want to patch one service without changing the rest of the system.

**Removing packages cleanly**

Removal has two common forms: removing the package itself and removing its configuration files.

```bash
sudo apt remove apache2
sudo apt purge apache2
sudo apt autoremove
```

- `apt remove` uninstalls the package but usually leaves configuration files behind.
- `apt purge` removes the package and its system-wide configuration files.
- `apt autoremove` deletes orphaned dependencies that are no longer needed.

Use `purge` when you want a clean reset of a service or application. Use `remove` when you may reinstall later and want to preserve configuration.

**Package state and verification**

You should verify what is installed and whether a package is held back or marked for a specific version.

```bash
dpkg -l | grep '^ii' | grep nginx
apt list --installed | grep nginx
apt policy nginx
```

- `dpkg -l` shows package status from the local database.
- `apt list --installed` is easier to read for quick checks.
- `apt policy` shows the candidate version and pinning priority.

If a package is on hold, it will not upgrade normally.

```bash
apt-mark showhold
sudo apt-mark hold linux-image-generic
sudo apt-mark unhold linux-image-generic
```

- Holding packages is useful when you need to delay a risky upgrade.
- Be careful with holds on security-sensitive packages because they can block important fixes.

**Repository management**

Ubuntu often uses `add-apt-repository` to enable additional sources, especially PPAs (Personal Package Archives). Use this carefully because third-party repositories can introduce compatibility or trust issues.

```bash
sudo add-apt-repository universe
sudo add-apt-repository ppa:graphics-drivers/ppa
sudo apt update
```

- After adding a repository, run `apt update` to fetch its package lists.
- Third-party repositories should be trusted, maintained, and compatible with your Ubuntu release.
- Prefer official repositories when possible for better support and security.

You can also manage repository keys and source files manually, though modern Ubuntu discourages legacy global keyring approaches. If you add a third-party source, verify the signing method and repository origin before trusting it.

> Avoid mixing repositories from different Ubuntu releases unless you understand the dependency impact. Cross-release packages can break system libraries and create upgrade conflicts.

**Dependency resolution and troubleshooting**

`apt` resolves dependencies automatically, but conflicts still happen. Common causes include broken package states, partial upgrades, held packages, or mismatched repositories.

```bash
sudo apt --fix-broken install
sudo dpkg --configure -a
sudo apt clean
sudo apt update
```

- `--fix-broken install` attempts to repair dependency problems.
- `dpkg --configure -a` completes interrupted package configuration.
- `apt clean` removes downloaded package archives from the local cache.
- `apt update` refreshes metadata after repository changes.

If a package is unavailable, check whether the repository is enabled and whether the package name is correct.

```bash
apt-cache search package-name
apt-cache policy package-name
```

If a package install fails because of a broken local `.deb`, inspect the package file and dependency chain. `dpkg` can install a local package, but it does not fetch missing dependencies automatically.

```bash
sudo dpkg -i package-file.deb
sudo apt -f install
```

**Useful inspection commands**

These commands help you understand package contents, dependencies, and ownership.

```bash
apt show htop
apt depends nginx
dpkg -S /usr/bin/ssh
dpkg -L curl
```

- `apt depends` shows dependency relationships.
- `dpkg -S` identifies which package owns a file.
- `dpkg -L` lists files installed by a package.

This is especially useful when troubleshooting missing binaries, overwritten files, or unexpected package behavior.

**Cache and disk usage**

APT stores downloaded package archives locally so you can reinstall without re-downloading immediately. Over time, the cache can consume disk space.

```bash
du -sh /var/cache/apt/archives
sudo apt autoclean
sudo apt clean
```

- `autoclean` removes package files that are no longer downloadable from repositories.
- `clean` removes most or all cached package archives.
- Use these when disk space is tight or after large upgrade cycles.

**Practical admin workflow**

A safe maintenance routine on Ubuntu usually looks like this:

1. Refresh package metadata with `sudo apt update`.
2. Review pending changes with `apt list --upgradable`.
3. Check package details with `apt show <package>` or `apt-cache policy <package>`.
4. Install or upgrade with `sudo apt install <package>` or `sudo apt upgrade`.
5. Remove unused dependencies with `sudo apt autoremove`.
6. Reboot if kernel, libc, or core system components were upgraded and the change requires it.

This workflow reduces surprises and makes it easier to trace what changed on the system.

**Example: installing and validating a web server**

```bash
sudo apt update
sudo apt install nginx
systemctl status nginx
curl -I http://localhost
```

- `apt install nginx` installs the package and its dependencies.
- `systemctl status nginx` confirms the service is active.
- `curl -I` checks that the HTTP service responds locally.

**Example: pinning or holding a package**

If you need to prevent an upgrade while testing compatibility, you can hold the package.

```bash
sudo apt-mark hold docker.io
apt-mark showhold
sudo apt-mark unhold docker.io
```

Use this sparingly and document why the hold exists so future maintenance is not blocked unexpectedly.

**Example: repairing a broken package state**

```bash
sudo dpkg --configure -a
sudo apt --fix-broken install
sudo apt update
sudo apt upgrade
```

This sequence is useful after interrupted installs, power loss, or dependency conflicts introduced by partial upgrades.

**Key distinctions to remember**

- **`apt` vs `dpkg`**: `apt` manages repositories and dependencies; `dpkg` manages local package installation state.
- **`upgrade` vs `full-upgrade`**: `upgrade` avoids removals; `full-upgrade` can remove packages to complete transitions.
- **`remove` vs `purge`**: `remove` leaves config files; `purge` deletes them.
- **`update` vs `upgrade`**: `update` refreshes package lists; `upgrade` installs newer versions.

**Exam and troubleshooting focus**

For Linux administration tasks, you should be able to explain why a package is not installing, how to refresh repository metadata, and how to safely remove software without leaving unnecessary dependencies behind. You should also know where repository definitions live and how to inspect package versions before making changes.

Common troubleshooting questions usually come down to one of these issues:

- The package lists are stale because `apt update` was not run.
- A repository is missing, disabled, or incompatible with the current Ubuntu release.
- A package is held back by `apt-mark hold`.
- A dependency conflict requires `apt --fix-broken install` or `apt full-upgrade`.
- A local `.deb` was installed with `dpkg -i` and still needs dependency resolution.
