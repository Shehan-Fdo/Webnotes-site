## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Purpose of the final readiness checklist**

This topic is a last-pass review of the Linux administration tasks and concepts that commonly appear in performance-based and multiple-choice exam scenarios. Use it to confirm that you can recognize the right tool, choose the right file, and apply the right command under time pressure.

A strong final review focuses on *recognition plus execution*: you should know what each command does, when to use it, and what output or side effect to expect.

**User and group account management**

User and group administration is a core Linux skill area. You should be comfortable creating accounts, changing identity settings, and understanding how password and shadow files work together.

- `useradd` creates a new local user account.
- `usermod` changes account properties such as primary group, supplementary groups, home directory, shell, or UID.
- `userdel` removes a user account; use `-r` to remove the home directory and mail spool when appropriate.
- `groupadd`, `groupmod`, and `groupdel` manage local groups.
- `passwd` sets or changes a password and can lock or unlock accounts depending on options and distribution behavior.
- `/etc/passwd` stores account metadata such as username, UID, GID, home directory, and login shell.
- `/etc/shadow` stores hashed passwords and password aging data; it is readable only by privileged users.
- `/etc/group` stores group definitions and membership lists.

Typical account workflow:

```bash
sudo useradd -m -s /bin/bash alice
sudo passwd alice
sudo usermod -aG wheel alice
sudo groupadd developers
sudo usermod -aG developers alice
sudo userdel -r olduser
```

Important account concepts:

- **UID 0** identifies the root account.
- **Primary group** is the default group assigned to new files created by the user.
- **Supplementary groups** grant additional access without changing the primary group.
- **Password aging** can enforce rotation and inactivity rules through shadow-related settings.
- **Skeleton directories** such as `/etc/skel` provide default files for new home directories.

> If a user cannot log in, check the shell, password status, account lock state, and whether the home directory exists with correct ownership and permissions.

**Shell, environment, and command-line fundamentals**

Many exam tasks depend on understanding how the shell interprets commands. You should know how environment variables, quoting, redirection, and pipelines affect command behavior.

- `PATH` controls where the shell searches for executables.
- `env`, `printenv`, and `export` help inspect or set environment variables.
- `echo $VAR` expands a variable in the current shell.
- Single quotes preserve literal text; double quotes allow variable expansion.
- Command substitution uses `$(...)` to insert command output into another command.
- Pipes (`|`) send standard output from one command to another.
- Redirection controls standard input, standard output, and standard error.

Common examples:

```bash
export PATH="$HOME/bin:$PATH"
export EDITOR=vim
echo "Home is $HOME"
grep -i error /var/log/messages 2>/dev/null | less
tar -czf backup.tgz /etc > backup.log 2>&1
```

Redirection patterns to remember:

- `>` overwrites standard output.
- `>>` appends standard output.
- `<` reads input from a file.
- `2>` redirects standard error.
- `&>` redirects both standard output and standard error on shells that support it.
- `2>&1` merges standard error into standard output.

Quoting and expansion rules matter:

- Use single quotes when you want literal strings.
- Use double quotes when you need variable expansion but want to preserve spaces.
- Use braces like `${VAR}` to avoid ambiguity in variable names.

**Text processing and file inspection**

You should be able to inspect files quickly and transform output with standard Unix tools. These commands often appear in troubleshooting and log-analysis scenarios.

- `cat` prints file contents.
- `less` pages through long output interactively.
- `head` and `tail` show the beginning or end of a file.
- `grep` searches for patterns in text.
- `sed` performs stream editing and simple substitutions.
- `awk` extracts fields and performs pattern-based processing.
- `sort` orders lines.
- `uniq` removes or counts adjacent duplicates.
- `cut` extracts columns or delimited fields.
- `tr` translates or deletes characters.
- `find` locates files by name, type, size, time, or permissions.

Practical examples:

```bash
grep -R "PermitRootLogin" /etc/ssh
sed 's/old/new/g' config.txt
awk -F: '{print $1, $3}' /etc/passwd
sort access.log | uniq -c | sort -nr
cut -d: -f1,3 /etc/passwd
tr '[:lower:]' '[:upper:]' < input.txt
find /var/log -type f -name "*.log" -mtime -7
```

Use these tools together when you need to answer questions like:

- Which users have UID values above a threshold?
- Which log entries contain a specific service name?
- Which files changed in the last day?
- Which lines are duplicated or out of order?

**Permissions, ownership, and SSH file security**

File permissions are a frequent exam topic because they affect both access control and service behavior. You should know numeric and symbolic modes, ownership changes, and special permission bits.

- `chmod` changes permissions.
- `chown` changes owner and group.
- `chgrp` changes group ownership.
- `umask` sets default permissions for newly created files.
- Setuid, setgid, and sticky bits modify execution or deletion behavior.

Examples:

```bash
chmod 640 report.txt
chmod u+x script.sh
chown root:root /usr/local/bin/tool
chgrp developers shared.txt
umask 027
```

Special permission reminders:

- **setuid** on an executable runs it with the file owner’s effective UID.
- **setgid** on a directory causes new files to inherit the directory group.
- **sticky bit** on a shared directory prevents users from deleting files they do not own.

SSH-related file permissions are especially important:

- `~/.ssh` should usually be `700`.
- Private keys such as `id_rsa` should usually be `600`.
- Public keys such as `id_rsa.pub` are readable.
- `known_hosts` stores host key fingerprints for servers you have connected to.
- If a host key changes unexpectedly, SSH warns about a possible man-in-the-middle issue.

> Do not weaken SSH file permissions to “make it work.” Fix ownership and mode values instead.

**Networking, name resolution, and firewall basics**

Networking questions often test whether you can identify the right configuration file or command for a given situation. You should know how to inspect interfaces, verify routes, and troubleshoot name resolution.

- `ip addr`, `ip link`, and `ip route` are the primary modern inspection commands.
- `ss -tulpn` shows listening sockets and active connections.
- `ping`, `traceroute`, `dig`, `host`, and `nslookup` help diagnose connectivity and DNS issues.
- `nmcli` manages NetworkManager connections on many modern distributions.
- `hostnamectl` shows or sets the system hostname on systemd-based systems.

Common configuration files and concepts:

- `/etc/hosts` provides static hostname-to-IP mappings.
- `/etc/resolv.conf` defines DNS resolver settings, though it may be managed dynamically.
- Persistent interface configuration may live under distribution-specific paths such as NetworkManager profiles or legacy network scripts.
- Firewall tools vary by distribution: `firewall-cmd` for firewalld, `ufw` for uncomplicated firewall, and `nft` for nftables.

Examples:

```bash
ip addr show
ip route show
nmcli device status
nmcli connection show
ping -c 4 8.8.8.8
dig example.com
```

Firewall examples:

```bash
sudo firewall-cmd --add-service=ssh --permanent
sudo firewall-cmd --reload

sudo ufw allow 22/tcp
sudo ufw status verbose

sudo nft list ruleset
```

Name resolution checklist:

- Confirm the IP address is reachable before blaming DNS.
- Check `/etc/hosts` for local overrides.
- Verify `/etc/resolv.conf` points to valid resolvers.
- Confirm the correct search domain and hostname are configured when needed.

**Storage, filesystems, RAID, and encryption**

Storage tasks often combine partitioning, filesystem creation, mounting, and persistence. You should be able to recognize the right tool for each layer.

- `lsblk`, `blkid`, `fdisk`, `parted`, and `gdisk` help inspect and manage block devices.
- `mkfs` creates filesystems.
- `mount` and `umount` attach and detach filesystems.
- `/etc/fstab` defines persistent mounts.
- `df -h` shows filesystem usage; `du -sh` shows directory usage.

Examples:

```bash
lsblk -f
sudo mkfs.ext4 /dev/sdb1
sudo mount /dev/sdb1 /mnt/data
sudo blkid /dev/sdb1
df -hT
du -sh /var/log
```

Advanced storage concepts to review:

- **RAID** combines disks for redundancy, performance, or both.
- Software RAID is commonly managed with `mdadm`.
- Logical Volume Manager (**LVM**) adds flexible volume resizing and snapshot capabilities.
- Encryption is commonly implemented with `LUKS` (Linux Unified Key Setup) and `cryptsetup`.
- Filesystem features such as journaling, quotas, and mount options affect reliability and capacity planning.

RAID and encryption examples:

```bash
sudo mdadm --create /dev/md0 --level=1 --raid-devices=2 /dev/sdb /dev/sdc
sudo cryptsetup luksFormat /dev/sdb2
sudo cryptsetup open /dev/sdb2 securedata
sudo mkfs.xfs /dev/mapper/securedata
```

Mount persistence example:

```fstab
UUID=1234-ABCD  /data  ext4  defaults,noatime  0  2
```

Review why you use each layer:

- Partitioning defines disk layout.
- Filesystems organize files and metadata.
- Mounting makes storage available in the directory tree.
- LVM and RAID improve flexibility and resilience.
- Encryption protects data at rest.

**Backup, restore, and archiving**

Backups are a practical administration skill and a common exam scenario. You should know how to preserve metadata, copy trees safely, and compress archives.

- `tar` creates and extracts archives.
- `rsync` synchronizes files efficiently and preserves metadata when configured correctly.
- `cp -a` copies recursively while preserving attributes.
- Compression tools include `gzip`, `bzip2`, `xz`, and `zip`.

Examples:

```bash
tar -czf etc-backup.tar.gz /etc
tar -xzf etc-backup.tar.gz -C /restore
rsync -aHAX --delete /home/ /backup/home/
cp -a /srv/project /srv/project.bak
gzip large.log
```

Key backup ideas:

- Use `-a` with `rsync` or `cp` when you need to preserve ownership, timestamps, and permissions.
- Use `--delete` with `rsync` only when the destination should mirror the source.
- Test restores, not just backups.
- Confirm whether you need compression, encryption, or both.

**Scheduling with cron and systemd timers**

Scheduling is often tested because Linux offers more than one mechanism. You should know when to use `cron` and when to use `systemd` timers.

- `cron` is time-based scheduling using crontab entries.
- `systemd` timers trigger associated service units and integrate with the system journal.
- `at` schedules one-time jobs.
- `anacron` helps run periodic jobs on systems that are not always on.

Cron syntax:

```cron
# m h dom mon dow command
15 2 * * 1 /usr/local/bin/backup.sh
```

Useful commands:

```bash
crontab -e
crontab -l
sudo systemctl list-timers
sudo systemctl status backup.timer
sudo systemctl status backup.service
```

A basic systemd timer pair:

```ini
# /etc/systemd/system/backup.service
[Unit]
Description=Run backup script

[Service]
Type=oneshot
ExecStart=/usr/local/bin/backup.sh
```

```ini
# /etc/systemd/system/backup.timer
[Unit]
Description=Daily backup timer

[Timer]
OnCalendar=daily
Persistent=true
Unit=backup.service

[Install]
WantedBy=timers.target
```

Enable and verify:

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now backup.timer
systemctl list-timers --all
journalctl -u backup.service
```

Compare the two approaches:

- Use **cron** for simple time expressions and legacy compatibility.
- Use **systemd timers** when you want dependency handling, logging integration, and persistent missed-run behavior.
- Use `Persistent=true` so a timer can catch up after downtime.

**Boot process, recovery, and troubleshooting**

Boot and recovery topics often appear in scenario-based questions. You should understand the boot chain and know where to intervene when a system fails to start.

- **GRUB** is a common bootloader on Linux systems.
- The kernel loads an **initramfs** (initial RAM filesystem) early in boot to provide drivers and tools needed before the real root filesystem mounts.
- `systemd` typically becomes PID 1 and manages targets and services.
- Rescue and emergency targets provide minimal environments for repair.

Useful commands and concepts:

```bash
systemctl get-default
systemctl isolate rescue.target
systemctl isolate emergency.target
journalctl -b
journalctl -xb
```

Boot troubleshooting checklist:

- Confirm the bootloader configuration is intact.
- Check whether the kernel and initramfs match the installed system.
- Review `journalctl -b` for boot-time errors.
- Use rescue mode for multi-user repair tasks.
- Use emergency mode when only a minimal shell is available and filesystems may need manual repair.

Recovery concepts to remember:

- A broken `/etc/fstab` entry can prevent normal boot.
- A missing driver in initramfs can stop the root filesystem from mounting.
- A failed service may block dependent targets or delay login.
- Kernel command-line edits in GRUB can temporarily change boot behavior for troubleshooting.

**Service management and logs**

You should be fluent with `systemctl` and the journal because service state and logs are central to Linux administration.

- `systemctl status` shows service state and recent log lines.
- `systemctl start`, `stop`, `restart`, `reload`, and `enable` control services.
- `journalctl` queries systemd logs.
- `systemctl is-enabled` and `systemctl is-active` help verify configuration and runtime state.

Examples:

```bash
sudo systemctl status sshd
sudo systemctl restart nginx
sudo systemctl enable --now chronyd
journalctl -u sshd --since today
journalctl -p err -b
```

Interpretation tips:

- **enabled** means the service starts at boot.
- **active** means the service is currently running.
- `reload` is preferred when a service supports configuration reload without full restart.
- Use `journalctl -xe` or `journalctl -xb` when you need recent high-priority errors.

**Security and access control**

Security review should include authentication, authorization, and basic hardening. Focus on what you can verify quickly during an exam.

- `sudo` delegates privileged commands according to policy.
- `/etc/sudoers` and files in `/etc/sudoers.d/` define sudo rules.
- `chmod`, `chown`, and ACLs (`getfacl`, `setfacl`) control access.
- SELinux or AppArmor may enforce mandatory access control depending on distribution.

Examples:

```bash
sudo visudo
getfacl file.txt
setfacl -m u:alice:rwx shared/
sestatus
aa-status
```

Security reminders:

- Use `visudo` to edit sudoers safely.
- Check whether SELinux is enforcing, permissive, or disabled when troubleshooting access issues.
- Verify that service files, keys, and scripts have the minimum required permissions.
- Review firewall rules alongside service configuration; a service can be healthy but unreachable.

**Quick final drill**

Use this as a last-minute self-test. If you cannot explain each item, review that section again.

1. Create a user, set a password, and add the user to a supplementary group.
2. Explain the difference between `/etc/passwd` and `/etc/shadow`.
3. Show how `PATH` affects command lookup.
4. Demonstrate quoting, pipes, redirection, and command substitution.
5. Use `grep`, `awk`, `sed`, `sort`, `uniq`, `cut`, `tr`, `head`, `tail`, `less`, and `find` on a sample log or config file.
6. Mount a filesystem and make the mount persistent in `/etc/fstab`.
7. Create a compressed archive with `tar` and restore it.
8. Compare a `cron` job with a `systemd` timer and verify both.
9. Inspect a failed service with `systemctl` and `journalctl`.
10. Troubleshoot a boot issue using rescue or emergency mode concepts.
11. Verify DNS, routes, and listening ports with `ip`, `dig`, `ss`, and `nmcli`.
12. Check SSH key and directory permissions before blaming authentication.

**Final exam mindset**

The exam often rewards careful reading. Identify the task domain first, then choose the smallest correct tool. If a question asks for persistence, think about configuration files and enablement. If it asks for verification, think about status commands, logs, and output inspection. If it asks for recovery, think about safe minimal access and reversible changes.

A good final readiness pass is not about memorizing every flag. It is about knowing the workflow: inspect, change, verify, and document.
