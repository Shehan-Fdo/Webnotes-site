## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Scenario-based exam strategy**

Scenario questions test whether you can choose the best Linux action under constraints, not whether you can recite definitions. You usually need to identify the symptom, confirm the likely layer involved, and select the least disruptive fix that matches the environment.

- Start by classifying the problem domain:
  - **Boot and service** issues
  - **Storage and filesystems**
  - **Networking and name resolution**
  - **Permissions and authentication**
  - **Package and update management**
  - **Logging and monitoring**
- Read for constraints:
  - What changed recently
  - What must stay online
  - Whether the fix must be persistent
  - Which distribution family is implied
- Prefer verification before and after changes:
  - `systemctl status`
  - `journalctl -xe`
  - `ip a`
  - `ss -tulpn`
  - `df -h`
  - `mount`
  - `getent hosts`

A good exam answer often avoids the most dramatic fix. For example, if a service fails after a config change, you usually validate the config, inspect logs, and restart only the affected service rather than rebooting the host.

**Troubleshooting workflow**

Use a repeatable process so you do not jump to conclusions. The exam often rewards methodical diagnosis over memorized commands.

1. Identify the symptom and scope.
   - Is the issue local, host-wide, or network-wide?
   - Is it intermittent or constant?
   - Does it affect one user, one service, or all users?
2. Check recent changes.
   - Package updates
   - Config edits
   - Permission changes
   - Hardware changes
   - Network changes
3. Inspect the relevant status and logs.
   - `systemctl status <service>`
   - `journalctl -u <service>`
   - `dmesg`
   - Distribution-specific log files such as `/var/log/messages` or `/var/log/syslog`
4. Test the smallest safe fix.
   - Reload a service
   - Correct a config file
   - Restore permissions
   - Add a missing route or DNS setting
5. Verify the result.
   - Re-run the failing command
   - Confirm persistence after restart or reconnect

> If the question asks for a persistent fix, do not choose a one-time runtime command unless it also updates the relevant configuration.

**Boot and service scenarios**

Boot problems often involve `systemd`, kernel modules, initramfs, or filesystem issues. Service problems usually involve a unit file, dependency failure, or a bad configuration file.

- Use `systemctl` to inspect service state:
  - `systemctl status nginx`
  - `systemctl is-enabled sshd`
  - `systemctl list-dependencies multi-user.target`
- Use `journalctl` for service-specific logs:
  - `journalctl -u nginx --since "10 minutes ago"`
  - `journalctl -b` for current boot messages
- Use `dmesg` for kernel and driver messages:
  - `dmesg | tail -n 50`

Common scenario patterns:
- A service starts manually but fails after reboot:
  - Check whether it is enabled with `systemctl is-enabled`
  - Confirm the unit file is not masked
  - Verify dependencies and environment files
- A hardware device disappears after a kernel update:
  - Check whether the correct kernel module is available and loaded
  - Confirm firmware is present if the device requires it
  - Verify the initramfs includes the needed driver when the device is required early in boot
- A module should load at boot:
  - Create a file in `/etc/modules-load.d/` containing only the module name on its own line
  - Example:
```bash
echo e1000e | sudo tee /etc/modules-load.d/e1000e.conf
```
  - This creates persistent module-load configuration; it does not immediately load the module into the running kernel

To load a module immediately, use `modprobe`:
```bash
sudo modprobe e1000e
lsmod | grep e1000e
```

**Networking scenarios**

Networking questions often mix runtime configuration with persistent configuration. The exam may ask you to restore connectivity, set a static address, fix DNS, or diagnose a listening service.

- Inspect interface state:
  - `ip link`
  - `ip addr show`
  - `ip route`
- Test connectivity in layers:
  - `ping -c 4 8.8.8.8`
  - `ping -c 4 example.com`
  - `getent hosts example.com`
  - `curl -I http://host`
- Check listening sockets:
  - `ss -tulpn`

A static address with NetworkManager usually needs both the address and the method set to manual for persistence.

```bash
nmcli con mod "Wired connection 1" ipv4.addresses 192.168.1.50/24
nmcli con mod "Wired connection 1" ipv4.gateway 192.168.1.1
nmcli con mod "Wired connection 1" ipv4.dns "1.1.1.1 8.8.8.8"
nmcli con mod "Wired connection 1" ipv4.method manual
nmcli con up "Wired connection 1"
```

- Use `ipv4.method manual` for a true static setup
- Use `nmcli con up` to apply the profile after editing it
- Verify with `ip addr` and `ip route`

For temporary testing, you can assign an address with `ip`, but that change is not persistent:
```bash
sudo ip addr add 192.168.1.50/24 dev eth0
sudo ip route add default via 192.168.1.1
```

DNS troubleshooting often requires checking both resolver configuration and name service behavior.

- Confirm the resolver:
  - `cat /etc/resolv.conf`
  - `resolvectl status` on systems using `systemd-resolved`
- Test name resolution:
  - `getent hosts host.example.com`
  - `dig host.example.com`
  - `nslookup host.example.com`

> A host can ping an IP address and still fail by name if DNS is misconfigured.

**SSH and remote access scenarios**

SSH issues are common in scenario questions because they combine service state, firewall rules, keys, and authentication policy. Service names can be distribution-specific: Debian and Ubuntu commonly use `ssh`, while RHEL and Fedora commonly use `sshd`.

- Check the service:
  - `systemctl status ssh`
  - `systemctl status sshd`
- Confirm the daemon is listening:
  - `ss -tulpn | grep :22`
- Review logs:
  - Debian/Ubuntu often use `/var/log/auth.log`
  - RHEL/Fedora often use `/var/log/secure`
  - `journalctl -u ssh` or `journalctl -u sshd`

Typical fixes:
- Ensure the service is enabled:
```bash
sudo systemctl enable --now sshd
```
- Open the firewall if needed:
```bash
sudo firewall-cmd --add-service=ssh --permanent
sudo firewall-cmd --reload
```
- Validate key permissions:
  - `~/.ssh` should usually be `700`
  - `~/.ssh/authorized_keys` should usually be `600`
- Check SSH server configuration:
  - `/etc/ssh/sshd_config`
  - Restart or reload after changes:
```bash
sudo systemctl restart sshd
```

If password login fails, consider whether the server disables it, whether the account is locked, or whether PAM (Pluggable Authentication Modules) policy blocks access.

**Storage and filesystem scenarios**

Storage questions often involve mounting, permissions, disk usage, or filesystem integrity. The exam may ask you to make storage available at boot or recover from a mount failure.

- Check block devices:
  - `lsblk`
  - `blkid`
  - `fdisk -l`
- Check usage:
  - `df -h`
  - `du -sh /path/*`
- Check mounts:
  - `mount`
  - `findmnt`

Persistent mounts belong in `/etc/fstab`. Use UUIDs when possible because device names can change.

```bash
UUID=1234-ABCD  /data  ext4  defaults  0  2
```

- `defaults` applies common mount options
- The last two fields control dump and fsck order
- Test an `fstab` entry safely with:
```bash
sudo mount -a
```

If a mount fails after boot:
- Check the filesystem type
- Confirm the UUID or device path is correct
- Verify the mount point exists
- Inspect `journalctl -b` and `dmesg` for errors

For filesystem repair, choose the tool that matches the filesystem type. For example, `fsck` is for many traditional Linux filesystems, while other filesystems use different utilities. Do not run repair tools on a mounted filesystem unless the tool and filesystem support it.

**Permissions and identity scenarios**

Permission problems often appear as “access denied,” “cannot read file,” or “service cannot write to directory.” The correct fix depends on ownership, mode bits, ACLs (Access Control Lists), SELinux context, or group membership.

- Check ownership and permissions:
  - `ls -l`
  - `stat file`
- Check group membership:
  - `id user`
  - `groups user`
- Check ACLs:
  - `getfacl file`
- Check SELinux context when applicable:
  - `ls -Z`
  - `restorecon -Rv /path`

Common patterns:
- A user can read a file but not write it:
  - The mode bits may allow read only
  - The directory permissions may block creation or deletion
- A service cannot access a file even though permissions look correct:
  - The service account may not own the file
  - The parent directory may block traversal
  - SELinux may deny access despite Unix permissions appearing correct

Example ownership and mode fix:
```bash
sudo chown appuser:appgroup /srv/app/config.yml
sudo chmod 640 /srv/app/config.yml
```

If the issue involves ACLs, add a targeted rule instead of broadening permissions for everyone:
```bash
sudo setfacl -m u:alice:rw /srv/shared/report.txt
```

**Package and update scenarios**

Package questions usually ask you to install, remove, query, or verify software using the distribution’s package manager. The exact tool depends on the family.

- Debian/Ubuntu:
  - `apt update`
  - `apt install package`
  - `apt remove package`
  - `apt policy package`
- RHEL/Fedora:
  - `dnf install package`
  - `dnf remove package`
  - `dnf info package`
  - `dnf provides /path/to/file`

Scenario clues often include:
- A command is missing after an update
- A service fails because a package was removed
- A dependency is unresolved
- A repository is disabled or unreachable

Practical checks:
```bash
rpm -q bash
dpkg -l | grep bash
dnf repoquery package
apt-cache policy package
```

If a hardware driver seems missing after an update, consider:
- Whether the correct kernel module package is installed for the running kernel
- Whether firmware files are present
- Whether the initramfs needs regeneration so the driver loads early enough

**Logs and monitoring scenarios**

Logs are often the fastest path to the answer, but the exam may expect you to know where to look based on the distribution and service type.

- Systemd journal:
  - `journalctl`
  - `journalctl -b`
  - `journalctl -p err`
  - `journalctl -u service`
- Traditional log files:
  - `/var/log/messages` on many RHEL-like systems
  - `/var/log/syslog` on many Debian-like systems
  - `/var/log/auth.log` on many Debian-like systems
  - `/var/log/secure` on many RHEL-like systems

Use filtering to reduce noise:
```bash
journalctl -u nginx --since today
journalctl -p warning --since "1 hour ago"
```

If the question asks for a log location, qualify your answer by distribution family rather than assuming one universal path.

**SELinux and AppArmor scenarios**

Mandatory access control can block actions even when Unix permissions are correct. Linux+ scenarios may include services failing after file moves, custom web content not loading, or daemons denied access to ports and directories.

- SELinux tools:
  - `getenforce`
  - `sestatus`
  - `ls -Z`
  - `restorecon`
  - `semanage`
- AppArmor tools:
  - `aa-status`
  - profile files under `/etc/apparmor.d/`

Common SELinux pattern:
- A file was moved into a new directory
- The service can read the file by path, but access is denied
- The fix is often to restore the correct context rather than disable SELinux

Example:
```bash
sudo restorecon -Rv /var/www/html
```

If a service needs a nonstandard port or path, use a policy-aware change rather than permanently disabling enforcement.

**Exam-style decision making**

When multiple answers look plausible, choose the one that best matches the requirement and the least disruptive path.

- If the question asks for a temporary test, use a runtime command
- If it asks for persistence, edit the relevant config file or profile
- If it asks for a service restart, do not reboot the host unless necessary
- If it asks for a network fix, distinguish between interface state, routing, and DNS
- If it asks for a permission fix, prefer the narrowest change that solves the problem

A useful mental model is:
- **Symptom**: what fails
- **Layer**: where the failure occurs
- **Tool**: what command confirms it
- **Fix**: what change persists appropriately
- **Verify**: how you prove it works

**Practical mini-scenarios**

**Scenario 1: Web service fails after config edit**

You changed a web server config and now the service will not start.

```bash
sudo nginx -t
sudo systemctl status nginx
sudo journalctl -u nginx --since "10 minutes ago"
sudo systemctl restart nginx
```

- Validate syntax before restarting
- Read the service logs for the exact error
- Fix the config, then restart only the affected service

**Scenario 2: Host has IP connectivity but no DNS**

You can ping an IP address, but hostnames fail.

```bash
ip addr
ip route
cat /etc/resolv.conf
getent hosts example.com
```

- Confirm the interface has an address
- Confirm a default route exists
- Confirm the resolver points to a valid DNS server
- Test name resolution separately from raw connectivity

**Scenario 3: User cannot access a shared directory**

A user reports permission denied on a shared path.

```bash
ls -ld /srv/share
getfacl /srv/share
id alice
```

- Check directory execute permission for traversal
- Check group membership
- Check ACLs before broadening mode bits

**Scenario 4: Service starts manually but not at boot**

The daemon works after you start it, but it is absent after reboot.

```bash
systemctl is-enabled myservice
systemctl enable myservice
systemctl status myservice
```

- Enable the unit for boot-time startup
- Confirm dependencies and any required environment files
- Verify after a reboot or service restart

**High-value commands to remember**

- `systemctl status <unit>`
- `systemctl enable --now <unit>`
- `journalctl -u <unit>`
- `ip addr show`
- `ip route`
- `ss -tulpn`
- `nmcli`
- `mount -a`
- `lsblk`
- `df -h`
- `getfacl`
- `restorecon`
- `modprobe`
- `lsmod`
- `dmesg`

**How to answer scenario questions quickly**

1. Identify the most likely subsystem from the symptom.
2. Eliminate answers that are only temporary when persistence is required.
3. Prefer commands that verify before changing.
4. Match distro-specific details carefully, especially service names and log paths.
5. Choose the least disruptive fix that satisfies the requirement.
