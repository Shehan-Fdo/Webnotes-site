## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Why this topic matters**

Linux+ questions often test whether you can spot the *wrong assumption* in a scenario. The command may look familiar, but the context, shell, filesystem, package manager, or service state changes the correct answer. These notes focus on the most common exam traps and how to reason through them.

**Command parsing and option traps**

Many Linux mistakes come from how the shell and commands parse arguments. A filename that starts with `-` can be treated as an option unless you stop option parsing with `--`.

- Use `--` to end option parsing when a path or value might begin with `-`.
- Example:
  ```bash
  rm -- -important-file
  cp -- -weird-name.txt /tmp/
  grep -- -pattern file.txt
  ```
- Prefer explicit paths when a command might otherwise search `PATH` or the current directory.
- Remember that `.` is the current directory, but it is not always in `PATH`.
- Use absolute paths when you need to avoid ambiguity:
  ```bash
  /usr/local/bin/backup.sh
  ./backup.sh
  ```

Common parsing pitfalls include:
- **Word splitting** when unquoted variables expand into multiple arguments.
- **Tilde expansion** only happening in specific shell contexts.
- **Command substitution** such as `$(...)` producing unexpected whitespace or multiple words.
- **Globbing** expanding `*` and `?` before the command runs.

A safe scripting pattern is to quote variables unless you intentionally want splitting:

```bash
for file in "$@"; do
  printf 'Processing: %s\n' "$file"
done
```

**Paths, `PATH`, and current directory assumptions**

A frequent exam trap is assuming a command or script will run just because it exists somewhere on disk. The shell only finds executables in directories listed in `PATH`, unless you provide a path explicitly.

- `./script.sh` runs the script in the current directory.
- `script.sh` works only if the current directory is in `PATH` or the command is installed in a searched directory.
- `command -v name` shows how the shell resolves a command name.
- `which` may not report shell aliases, functions, or builtins consistently because its behavior depends on the implementation.

Practical checks:

```bash
pwd
echo "$PATH"
command -v python3
type -a ls
```

Be careful with relative paths in cron jobs, systemd units, and scripts. Those environments often use a different working directory than your interactive shell.

> If a script works manually but fails in automation, check the working directory, `PATH`, and whether the script uses relative file paths.

**Shell expansion and quoting mistakes**

Shell expansion happens before the command runs, so the same command can behave differently depending on quoting.

- **Single quotes** prevent most expansion.
- **Double quotes** preserve spaces but still allow variable and command expansion.
- Unquoted variables can split into multiple words and trigger globbing.
- `~` expands to a home directory only in shell parsing contexts, not inside quotes or after some characters.

Examples:

```bash
name="Linux Admin"
echo $name      # word splitting risk
echo "$name"    # safe for spaces

files=$(ls *.log)
printf '%s\n' "$files"
```

Use `find` and loops instead of parsing `ls` output in scripts:

```bash
find /var/log -type f -name '*.log' -print0 | while IFS= read -r -d '' file; do
  printf 'Found: %s\n' "$file"
done
```

**Package manager misconceptions**

Linux distributions use different package workflows, and exam questions often test whether you know which tool performs which task.

- `apt` handles package installation, removal, and dependency resolution on Debian-based systems.
- `dnf` handles package installation, removal, updates, and dependency resolution on modern RHEL-family systems.
- `rpm` and `dpkg` work at a lower level and do not resolve dependencies automatically the way `dnf` and `apt` do.
- `rpm -qf /path/to/file` and `dpkg -S /path/to/file` identify which package owns a file.

Examples:

```bash
apt install nginx
dnf install nginx
rpm -qf /usr/bin/ssh
dpkg -S /usr/bin/ssh
```

Common traps:
- Installing a package by exact version only works if that version exists in enabled repositories.
- Dependency conflicts can block installation even when the package name is correct.
- `dnf config-manager` requires the appropriate plugin package on many systems.
- Repository names are distro-specific; do not assume a repo called `extras` exists everywhere.
- `rpm` can install a local package file, but it does not solve dependencies automatically.
- `dpkg -i` can leave packages unconfigured until dependencies are fixed with `apt -f install` or equivalent workflow.

A useful mental model:
- Use `apt` or `dnf` for normal package management.
- Use `rpm` or `dpkg` for inspection, queries, or low-level package operations.

**Service management and systemd traps**

`systemctl` questions often hinge on the difference between runtime state and boot-time enablement.

- `start` launches a service now.
- `stop` stops it now.
- `restart` stops and starts it again.
- `reload` asks the service to re-read configuration without a full restart, if supported.
- `reload-or-restart` reloads when possible, otherwise restarts.
- `enable` creates boot-time startup links or equivalent policy.
- `disable` removes boot-time startup.
- `mask` prevents a unit from starting, even manually, by linking it to `/dev/null`.
- `preset` applies vendor preset policy, which may differ from your manual choice.

Examples:

```bash
systemctl status sshd
systemctl enable --now nginx
systemctl disable --now cups
systemctl mask bluetooth.service
systemctl unmask bluetooth.service
systemctl reload-or-restart httpd
```

Important nuance:
- `enable` does not guarantee the service will start if the unit is masked, broken, or blocked by dependencies.
- A service can be enabled but inactive.
- A service can be active but not enabled.
- Preset policy may override assumptions during provisioning or image builds.

For troubleshooting, inspect logs and unit details:

```bash
systemctl cat nginx
systemctl list-dependencies sshd
journalctl -u nginx --since today
journalctl -xeu nginx
```

`journalctl -xeu` is useful because `-x` adds explanatory text when available and `-e` jumps to the end of the journal; it is not a special “error mode.”

**Filesystem and mount misconceptions**

Mount questions often test whether you understand the difference between the filesystem on disk, the mount point, and the options applied at mount time.

- A filesystem must be mounted before its contents are accessible through the mount point.
- The mount point is a directory, not the device itself.
- `mount -o remount,ro /` is a common way to remount the root filesystem read-only.
- Some systems require the mount point rather than the device path for remount operations.

Examples:

```bash
mount
findmnt
mount -o remount,ro /
mount -o remount,rw /var
```

Common misconceptions:
- A read-only remount does not erase data; it changes access mode.
- Permissions still matter on mounted filesystems.
- A filesystem can be mounted but inaccessible if ownership, ACLs, SELinux labels, or export rules block access.
- `fstab` syntax errors can prevent boot-time mounts.
- `nofail` allows boot to continue if a mount fails.
- `defaults` is a shorthand for a standard set of mount options, but it may not be appropriate for every filesystem.
- `x-systemd.*` options are systemd-specific helpers and should be used carefully.

Example `/etc/fstab` entries:

```fstab
UUID=1234-ABCD  /data  ext4  defaults,nofail  0  2
UUID=5678-EFGH  /backup  xfs  defaults,x-systemd.automount  0  0
```

Be aware of filesystem-specific behavior:
- The setgid bit on a directory can influence group inheritance for new files and subdirectories, but the exact behavior depends on the filesystem and mount options.
- Do not assume every filesystem treats special bits identically.

**Permissions, ownership, and special bits**

Permission questions often hide a detail about the object type or the execution context.

- `rwx` on a directory means something different from `rwx` on a file.
- Execute permission on a directory means you can traverse it.
- Read permission on a directory lets you list names, but not necessarily access file contents.
- The owner, group, and mode bits are only part of the access decision; ACLs and MAC policies can also apply.

Examples:

```bash
ls -ld /srv/app
stat /srv/app/config.yml
chmod 750 /srv/app
chown root:app /srv/app
```

Special bits:
- **setuid** on an executable runs it with the file owner’s effective user ID.
- **setgid** on an executable runs it with the file group’s effective group ID.
- **setgid** on a directory may affect group inheritance, but behavior is filesystem-dependent.
- **sticky bit** on a directory restricts deletion of files to the file owner, directory owner, or root.

Examples:

```bash
chmod u+s /usr/local/bin/tool
chmod g+s /shared
chmod +t /tmp/shared
```

**Cron and timer traps**

Cron is a classic exam area because the syntax changes depending on where the job is defined.

- User crontabs edited with `crontab -e` do **not** include a user field.
- `/etc/crontab` and files in `/etc/cron.d/` do include the user field.
- Cron jobs run with a minimal environment, so `PATH` may be limited.
- Scripts may fail because they are not executable, have the wrong shebang, or reference relative paths.

Examples:

```cron
# User crontab
*/5 * * * * /usr/local/bin/backup.sh

# /etc/crontab or /etc/cron.d/backup
*/5 * * * * root /usr/local/bin/backup.sh
```

Troubleshooting checklist:
- Verify the script is executable: `chmod +x /usr/local/bin/backup.sh`
- Verify the interpreter line: `#!/bin/bash` or another valid shebang
- Use absolute paths inside the script
- Check cron logs or journal entries
- Confirm the service is installed and running, such as `crond` or `cron`

Systemd timers are often a better fit for modern systems because they integrate with the journal and unit dependencies.

```bash
systemctl list-timers
systemctl status backup.timer
journalctl -u backup.service
```

**SELinux and AppArmor misconceptions**

Mandatory access control can block actions even when Unix permissions look correct.

- **SELinux** uses labels and policy rules.
- **AppArmor** uses path-based profiles.
- A denial does not always mean the file mode bits are wrong.

SELinux troubleshooting examples:

```bash
getenforce
sestatus
ausearch -m avc -ts recent
journalctl -t setroubleshoot --since today
sealert -a /var/log/audit/audit.log
restorecon -Rv /var/www/html
semanage fcontext -a -t httpd_sys_content_t '/srv/web(/.*)?'
restorecon -Rv /srv/web
```

AppArmor examples:

```bash
aa-status
aa-enforce /etc/apparmor.d/usr.sbin.nginx
aa-complain /etc/apparmor.d/usr.sbin.nginx
```

Practical trap:
- `restorecon` restores the default SELinux context from policy.
- `semanage fcontext` changes the persistent mapping so future relabels keep the custom context.
- Minimal installations may not include SELinux troubleshooting tools such as `sealert`.

**Networking and name resolution traps**

Network questions often mix up local name mapping, DNS, and interface configuration.

- `/etc/hosts` provides local static name resolution.
- DNS resolves names through configured name servers.
- Hostname lookup order depends on `nsswitch.conf`.
- A successful `ping` to a hostname may reflect `/etc/hosts`, DNS, or cached resolution.

Examples:

```bash
getent hosts server1
cat /etc/nsswitch.conf
cat /etc/hosts
resolvectl status
```

Common misconceptions:
- Editing `/etc/hosts` does not update DNS.
- DNS failures and local host mapping failures are different problems.
- A hostname may resolve locally even when external DNS is broken.
- `ping 127.0.0.1` tests loopback, not network connectivity.
- `ping localhost` depends on name resolution and may not behave the same as `ping 127.0.0.1`.

Useful checks:

```bash
ip addr
ip route
ss -tulpn
dig example.com
nslookup example.com
```

**Logging and journal traps**

Logs are often the fastest way to distinguish a configuration problem from a service failure.

- `journalctl -u service` filters by unit.
- `journalctl -p err` filters by priority.
- `journalctl --since today` narrows the time window.
- `journalctl -f` follows new entries.

Examples:

```bash
journalctl -u sshd --since "1 hour ago"
journalctl -p warning --since today
tail -f /var/log/messages
```

Be careful not to assume every system uses the same log files:
- Some distributions rely heavily on the journal.
- Some still write traditional text logs under `/var/log`.
- Log locations and service names vary by distribution.

**System recovery and boot misconceptions**

Recovery questions often test whether you know the difference between a temporary fix and a persistent one.

- Editing a file in a live system may not survive reboot if the real issue is in `fstab`, a unit file, or a config management system.
- A kernel parameter or bootloader change may be needed for single-user or rescue mode access.
- A service failure at boot may be caused by a dependency, a missing mount, or a bad unit override.

Examples:

```bash
systemctl rescue
systemctl emergency
journalctl -b -p err
```

When troubleshooting boot issues:
- Check `fstab` for invalid mount options.
- Check unit dependencies with `systemctl list-dependencies`.
- Verify that required filesystems are mounted before services start.
- Confirm that the target is correct, such as `multi-user.target` versus `graphical.target`.

**Scenario-based exam traps**

These are the kinds of details that often change the correct answer:

- A command works in your shell but fails in a script because the script lacks `PATH`, quoting, or execute permission.
- A package installs but the service does not start because the unit is disabled, masked, or failing dependency checks.
- A file is present but inaccessible because SELinux or AppArmor blocks it.
- A mount exists but the application cannot write because the filesystem is read-only or the directory permissions are wrong.
- A hostname resolves on one machine but not another because `/etc/hosts` differs from DNS.
- A cron job runs manually but not automatically because the environment, shebang, or absolute path is wrong.

**Quick exam checklist**

Before choosing an answer, ask:
1. Is this a shell parsing issue, a path issue, or a permissions issue?
2. Is the command using the right tool for the distribution and task?
3. Is the problem runtime state or boot-time state?
4. Is access blocked by Unix permissions, ACLs, SELinux, or AppArmor?
5. Is the failure local name resolution, DNS, or network connectivity?
6. Is the syntax different in a user crontab, system crontab, or timer unit?

**Practical command set to remember**

```bash
command -v name
type -a name
getent hosts host1
findmnt
systemctl status service
journalctl -xeu service
getenforce
aa-status
rpm -qf /path/to/file
dpkg -S /path/to/file
```

These commands help you verify assumptions quickly instead of guessing from the symptom alone.
