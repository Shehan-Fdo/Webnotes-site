## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**How to use this review**

This note set is organized as a domain-by-domain study guide for Linux administration. Use it to connect exam objectives to real commands, configuration files, and troubleshooting workflows. The goal is not memorization alone; it is to understand how Linux systems behave under normal operation, failure, and recovery.

A good exam strategy is to study each domain in three layers:

- **Concepts**: what the feature does and when you use it.
- **Commands and files**: the tools and paths you must recognize quickly.
- **Scenarios**: how you diagnose and fix common problems.

**Domain 1: System Management and Administration**

This domain covers the daily work of managing a Linux host: users, services, logs, processes, packages, and the boot process. You should be comfortable moving between command-line inspection and persistent configuration.

- **Systemd** is the default service manager on many modern distributions.
  - `systemctl status sshd`
  - `systemctl enable --now firewalld.service`
  - `systemctl list-units --type=service`
- **Targets** replace older runlevels.
  - `systemctl get-default`
  - `systemctl set-default multi-user.target`
  - `systemctl isolate rescue.target` changes the current transaction, but it is not the same as booting directly into rescue mode.
- **Logs** are commonly queried with `journalctl`.
  - `journalctl -xe`
  - `journalctl -u sshd`
  - `journalctl -p err`
- **Processes** are inspected and controlled with standard tools.
  - `ps aux`
  - `top` or `htop`
  - `kill -TERM <pid>` and `kill -KILL <pid>` only when needed
- **Scheduling** includes cron and systemd timers.
  - `crontab -e`
  - `systemctl list-timers`
  - `systemctl --user` manages a user-level systemd instance, not a cron replacement.

A practical boot-troubleshooting workflow often starts with the bootloader, then initramfs, then systemd:

```bash
journalctl -b
systemctl --failed
systemctl status <unit>
```

If a service fails repeatedly, inspect the unit file and its dependencies:

```bash
systemctl cat nginx
systemctl list-dependencies nginx
```

**Boot and recovery concepts**

Boot issues often involve one of four layers: firmware, bootloader, kernel/initramfs, or userspace. You should know how to isolate each layer.

- **GRUB** is the common bootloader on Linux systems.
  - Edit boot parameters temporarily at the GRUB menu for one-time recovery.
  - `systemd.unit=rescue.target` at boot is different from switching targets after boot.
- **Initramfs** is the early userspace image used to mount the real root filesystem.
  - `rd.break` can drop you into an early shell on some dracut-based systems.
- **Rescue mode** is useful when the system cannot reach normal multi-user operation.
  - Mount the root filesystem read-write if you need to repair configuration.
- **Emergency mode** is more minimal than rescue mode and is used when critical mounts fail.

A common recovery pattern is:

1. Boot into a rescue or emergency environment.
2. Identify the root filesystem and mount it.
3. Check logs and configuration.
4. Repair the issue.
5. Rebuild initramfs or bootloader config if needed.

**Package management**

Package tools differ by distribution family, but the workflow is similar: query, install, verify, update, and remove.

- Debian-based systems commonly use `apt` and `dpkg`.
  - `apt update`
  - `apt install curl`
  - `dpkg -l | grep nginx`
  - `apt show <package>`
- RHEL-based systems commonly use `dnf` and `rpm`.
  - `dnf check-update`
  - `dnf install vim-enhanced`
  - `rpm -qf /usr/bin/ssh`
- Repository management matters because packages are only as trustworthy as the configured sources.
  - `dnf repolist`
  - `apt policy`
  - `apt-cache policy <package>` is still available, though `apt show` is often more direct for package details.

Package verification and signature checking are exam-relevant because they support supply-chain trust.

- Use repository signatures and package signatures where available.
- Verify installed files with package managers when troubleshooting unexpected changes.
- Know that package metadata can reveal dependencies, conflicts, and file ownership.

**Users, groups, and permissions**

Linux access control combines ownership, mode bits, ACLs (Access Control Lists), and sometimes SELinux or AppArmor.

- **Ownership** is controlled by user and group.
  - `chown alice:dev /srv/project`
  - `chgrp dev /srv/project`
- **Mode bits** control read, write, and execute.
  - `chmod 640 file.txt`
  - `chmod 750 /srv/app`
- **Special bits** matter in shared directories and executables.
  - setuid, setgid, and sticky bit
- **ACLs** provide more granular permissions than mode bits alone.
  - `setfacl -m u:alice:rw /srv/project/report.txt`
  - `getfacl /srv/project/report.txt`

For directories, remember that execute permission means traversal. A file ACL that grants `rw` may be correct for a document, but a directory usually needs `x` as well for access.

**Domain 2: Networking**

Networking questions often test both configuration and diagnosis. You should know how to inspect interfaces, routes, DNS, firewall rules, and name resolution behavior.

- **Interface inspection**
  - `ip addr show`
  - `ip link show`
  - `ip route show`
- **Temporary configuration**
  - `ip addr add 192.0.2.10/24 dev eth0`
  - `ip route add default via 192.0.2.1`
- **Persistent configuration** depends on the distribution.
  - NetworkManager profiles are common on desktops and many servers.
  - systemd-networkd is common in some server and container-oriented environments.
  - Traditional `/etc/network/interfaces` still appears on some Debian-based systems.

A practical NetworkManager workflow uses `nmcli`:

```bash
nmcli device status
nmcli connection show
nmcli connection modify "Wired connection 1" ipv4.addresses 192.0.2.10/24
nmcli connection modify "Wired connection 1" ipv4.gateway 192.0.2.1
nmcli connection modify "Wired connection 1" ipv4.dns 1.1.1.1,8.8.8.8
nmcli connection up "Wired connection 1"
```

**DNS and name resolution**

Name resolution can involve `/etc/hosts`, local caching, `systemd-resolved`, or external resolvers.

- `/etc/hosts` is checked before DNS in many configurations.
- `/etc/resolv.conf` may be managed dynamically.
- `resolvectl status` helps inspect `systemd-resolved` behavior when that service is in use.
- `dig`, `host`, and `nslookup` are useful for testing DNS queries.

Example checks:

```bash
getent hosts example.com
dig example.com
resolvectl query example.com
```

If DNS fails, verify:

- the interface has the correct IP address and gateway
- the resolver configuration points to valid name servers
- the firewall allows outbound DNS if relevant
- the local cache is not masking stale data

**Firewalling**

Firewall tools vary by distribution, but the exam expects you to understand the purpose of packet filtering and service exposure.

- `firewalld` uses zones and services.
  - `firewall-cmd --get-active-zones`
  - `firewall-cmd --add-service=ssh --permanent`
  - `firewall-cmd --reload`
- `nftables` is the modern packet-filtering framework on many systems.
  - `nft list ruleset`
- `iptables` may still appear on older systems or in compatibility contexts.

A simple firewall workflow is:

1. Identify the service and port.
2. Confirm the service is listening.
3. Open only the required port or service.
4. Test from a remote host.

```bash
ss -tulpn
firewall-cmd --list-all
```

**Remote access**

SSH is the primary remote administration protocol.

- Client-side commands:
  - `ssh user@host`
  - `scp file.txt user@host:/tmp/`
  - `sftp user@host`
- Server-side hardening:
  - disable password authentication when possible
  - restrict root login
  - use key-based authentication
  - limit allowed users or groups

A hardened `/etc/ssh/sshd_config` example:

```conf
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
AllowUsers alice bob
```

After changes, validate and reload carefully:

```bash
sshd -t
systemctl reload sshd
```

> If you disable password authentication before confirming key access, you can lock yourself out. Always keep a tested recovery path.

**Domain 3: Security**

Security questions often combine permissions, authentication, auditing, and hardening. The exam expects you to recognize both preventive controls and troubleshooting steps.

**Authentication and PAM**

PAM (Pluggable Authentication Modules) controls how many Linux services authenticate users.

- PAM configuration is commonly found in `/etc/pam.d/`.
- Authentication failures may come from:
  - incorrect module order
  - expired passwords
  - account lockouts
  - missing access rules
- Useful checks:
  - `faillock --user alice`
  - `chage -l alice`
  - `passwd -S alice`

When troubleshooting login issues, check both the application logs and PAM-related logs. A service may fail authentication even when the password is correct if account policy blocks access.

**SELinux and AppArmor**

Mandatory access control adds another layer beyond Unix permissions.

- **SELinux** uses labels and policy.
  - `getenforce`
  - `sestatus`
  - `ls -Z`
  - `restorecon -Rv /var/www/html`
- **AppArmor** uses profiles and path-based rules.
  - `aa-status`
  - `aa-enforce`
  - `aa-complain`

A common SELinux troubleshooting pattern is:

```bash
ausearch -m avc -ts recent
sealert -a /var/log/audit/audit.log
```

If a service works when SELinux is permissive but fails when enforcing, do not disable SELinux permanently without understanding the policy issue. Fix labels, booleans, or policy rules first.

**Auditing and integrity**

Security administration also includes detecting unauthorized change.

- **Auditd** records security-relevant events.
  - `auditctl -l`
  - `ausearch`
  - `aureport`
- **File integrity** tools help detect unexpected modifications.
  - `AIDE` is a common example.
  - Baseline the system, then compare later to detect drift.

**SSH hardening and key management**

Key-based authentication is preferred for administrative access.

- Generate keys with `ssh-keygen`.
- Copy keys with `ssh-copy-id`.
- Protect private keys with a passphrase when appropriate.
- Restrict server-side access using `AllowUsers`, `AllowGroups`, and `Match` blocks.

Example client workflow:

```bash
ssh-keygen -t ed25519 -C "admin@example"
ssh-copy-id alice@server1
ssh -i ~/.ssh/id_ed25519 alice@server1
```

**Domain 4: Storage and Filesystems**

Storage tasks include partitioning, filesystems, mounting, LVM, quotas, and recovery. You should understand both the command-line tools and the persistent configuration files.

**Disk and partition management**

- `lsblk` shows block devices and mount points.
- `blkid` shows UUIDs and filesystem types.
- `fdisk`, `parted`, and `gdisk` create and modify partitions.

Example inspection:

```bash
lsblk -f
blkid
parted /dev/sdb print
```

**Filesystems and mounting**

Mounting is both temporary and persistent.

- Temporary mount:
  - `mount /dev/sdb1 /mnt/data`
- Persistent mount:
  - `/etc/fstab`

Example `/etc/fstab` entry:

```fstab
UUID=1234-ABCD  /data  ext4  defaults,noatime  0  2
```

Useful mount options include:

- `defaults` for standard behavior
- `noatime` to reduce metadata writes
- `ro` for read-only mounts
- `nodev`, `nosuid`, and `noexec` for hardening where appropriate

Validate `fstab` carefully before rebooting:

```bash
mount -a
findmnt /data
```

**LVM basics**

LVM (Logical Volume Manager) is common in enterprise Linux because it simplifies resizing and storage abstraction.

- Physical volume: `pvcreate`
- Volume group: `vgcreate`
- Logical volume: `lvcreate`

Example workflow:

```bash
pvcreate /dev/sdb1
vgcreate vgdata /dev/sdb1
lvcreate -n lvapp -L 20G vgdata
mkfs.ext4 /dev/vgdata/lvapp
mount /dev/vgdata/lvapp /srv/app
```

Resizing is a frequent exam scenario:

- Extend the logical volume first.
- Grow the filesystem afterward if needed.
- Some filesystems can grow online; shrinking is more limited and often requires offline work.

Example:

```bash
lvextend -L +5G /dev/vgdata/lvapp
resize2fs /dev/vgdata/lvapp
```

**Quotas and usage**

Disk quotas limit per-user or per-group consumption.

- Enable quotas in the filesystem mount options and configuration.
- Use `quota`, `repquota`, and `edquota` to inspect and manage limits.

**Filesystem repair**

`fsck` is a front-end that dispatches to filesystem-specific checkers. It is usually used offline on unmounted filesystems.

- Check before repair when possible.
- Do not run repair tools on a mounted filesystem unless the filesystem and procedure explicitly allow it.
- Not all filesystems use `fsck` in the same way.

**Domain 5: Scripting, Automation, and DevOps Concepts**

This domain focuses on repeatability. You should be able to automate routine tasks, schedule jobs, and debug scripts.

**Shell scripting basics**

A script should be readable, safe, and predictable.

```bash
#!/usr/bin/env bash
set -euo pipefail

backup_dir="/var/backups"
timestamp="$(date +%F_%H%M%S)"

tar -czf "${backup_dir}/etc-${timestamp}.tar.gz" /etc
```

Key practices:

- use `#!/usr/bin/env bash` when portability matters
- quote variables unless you intentionally want word splitting
- enable strict mode when appropriate
- test scripts with `bash -n` and `shellcheck` if available

**Cron**

Cron is still widely used for periodic jobs.

- User crontab: `crontab -e`
- System cron files: `/etc/crontab`, `/etc/cron.d/`, `/etc/cron.daily/`

Cron syntax example:

```cron
# minute hour day-of-month month day-of-week command
15 2 * * 1 /usr/local/bin/backup.sh
```

Remember:

- cron uses the system time and timezone
- environment variables are limited
- commands should use absolute paths

**Systemd timers**

Systemd timers are often preferred when you want logging, dependency handling, and tighter integration with services.

Example timer unit:

```ini
[Unit]
Description=Run backup daily

[Timer]
OnCalendar=daily
Persistent=true

[Install]
WantedBy=timers.target
```

Example service unit:

```ini
[Unit]
Description=Backup job

[Service]
Type=oneshot
ExecStart=/usr/local/bin/backup.sh
```

Enable both units:

```bash
systemctl enable --now backup.timer
systemctl list-timers
```

**Script debugging**

When a script fails, isolate the failure point.

- run with `bash -x script.sh`
- check exit codes with `$?`
- inspect environment differences between interactive shells and cron
- log output explicitly

**Domain 6: Containers and Virtualization**

Container and virtualization topics emphasize isolation, image handling, and basic lifecycle management. The exam usually focuses on operational understanding rather than deep orchestration design.

**Containers**

Containers package an application and its dependencies while sharing the host kernel.

- **Images** are templates.
- **Containers** are running instances.
- **Registries** store and distribute images.
- **Rootless containers** run without full root privileges, improving safety in many environments.

Common Podman workflows:

```bash
podman pull docker.io/library/nginx:latest
podman images
podman run -d --name web -p 8080:80 docker.io/library/nginx:latest
podman ps
podman logs web
podman stop web
podman rm web
```

Operational notes:

- the image may need to be pulled from a registry if it is not local
- port mappings expose container services to the host
- persistent data usually requires volumes or bind mounts

Example with a bind mount:

```bash
podman run -d --name web \
  -p 8080:80 \
  -v /srv/web:/usr/share/nginx/html:ro \
  docker.io/library/nginx:latest
```

Container security considerations:

- use minimal images when possible
- avoid running as root inside the container unless required
- limit capabilities and exposed ports
- keep images updated and rebuild regularly
- scan images when tooling is available

**Virtualization**

Virtual machines (VMs) provide hardware-level isolation through a hypervisor.

- **Type 1** hypervisors run directly on hardware.
- **Type 2** hypervisors run on top of a host OS.
- Common Linux tooling may include `virsh`, `virt-install`, and `virt-manager` in KVM/QEMU environments.

Basic VM administration concepts:

- create and start a VM
- attach storage and network interfaces
- inspect console access
- manage snapshots carefully
- understand the difference between guest tools and host tools

**Domain 7: Troubleshooting and Diagnostics**

Troubleshooting is not a separate skill from administration; it is the method you use to validate every change. The exam often presents symptoms rather than direct questions.

**A practical diagnostic order**

1. Confirm the symptom.
2. Identify the scope: one user, one host, one service, or the whole network.
3. Check logs.
4. Check configuration.
5. Check dependencies and permissions.
6. Test the fix.
7. Verify persistence after reboot or restart.

**Common scenario patterns**

- **Service will not start**
  - `systemctl status <unit>`
  - `journalctl -u <unit>`
  - check config syntax and file permissions
- **Network host unreachable**
  - `ip addr`
  - `ip route`
  - `ping`, `traceroute`, `ss`
  - verify firewall and DNS
- **Login fails**
  - check PAM, account expiration, shell, and SSH settings
  - inspect `/var/log/secure`, `/var/log/auth.log`, or journal output depending on distribution
- **Disk full**
  - `df -h`
  - `du -sh *`
  - inspect logs and rotated files
  - clear safely, then prevent recurrence
- **Boot failure**
  - inspect GRUB, kernel parameters, initramfs, and filesystem integrity
  - use rescue media if needed

**Useful command set**

```bash
df -h
du -sh /var/*
free -h
uptime
ss -tulpn
lsof +D /var/log
```

**Cloud and remote systems**

Cloud and remote administration concepts often appear as operational awareness rather than deep cloud engineering. You should know how Linux systems behave when provisioned automatically.

- **cloud-init** is commonly used to initialize cloud instances.
  - user-data can create users, install packages, and set SSH keys
- **Metadata services** provide instance-specific configuration.
  - if metadata access fails, provisioning scripts may not complete
- **Remote management** may rely on SSH, serial console access, or provider-specific tooling

Example cloud-init user-data:

```yaml
#cloud-config
users:
  - name: alice
    groups: sudo
    ssh_authorized_keys:
      - ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIExampleKey
package_update: true
packages:
  - vim
  - curl
```

If a cloud instance does not come up as expected, check:

- cloud-init logs
- network metadata access
- SSH key injection
- security group or firewall rules
- instance console output

**Domain-to-objective mapping**

Use this quick map to verify that your study covers the major Linux+ areas.

| Domain area | What you should be able to do | Typical evidence |
|---|---|---|
| System management | Manage services, logs, boot targets, packages | `systemctl`, `journalctl`, `apt`, `dnf` |
| Networking | Configure interfaces, routes, DNS, and firewalls | `ip`, `nmcli`, `ss`, `firewall-cmd`, `nft` |
| Security | Control access, harden SSH, inspect MAC and audit logs | `chmod`, `setfacl`, `sshd_config`, `getenforce`, `auditctl` |
| Storage | Partition, mount, repair, and resize storage | `lsblk`, `fstab`, `LVM`, `fsck`, `mount` |
| Automation | Schedule and debug recurring tasks | `cron`, `systemd timers`, shell scripts |
| Containers/VMs | Run and manage isolated workloads | `podman`, `virsh`, image registries |
| Troubleshooting | Diagnose failures methodically | logs, status checks, dependency inspection |

**Final exam prep reminders**

- Learn the difference between temporary changes and persistent configuration.
- Practice reading logs before changing settings.
- Know where each distribution stores its network, auth, and service configuration.
- Be able to explain why a command works, not just what it does.
- Treat security controls as layered: permissions, authentication, MAC, auditing, and network filtering all work together.
