**Core concepts**

Shell redirection controls where a command reads input and sends output. In scripts, this lets you automate data flow without interactive prompts, capture logs, separate errors from normal output, and feed commands from files or inline text.

The three standard streams are:

- **stdin**: standard input, file descriptor `0`
- **stdout**: standard output, file descriptor `1`
- **stderr**: standard error, file descriptor `2`

A shell can redirect each stream independently. This is different from piping, which connects the stdout of one command to the stdin of another.

**Standard output and standard error**

By default, most commands write normal results to stdout and errors to stderr. This separation matters because you can log them differently or suppress one without hiding the other.

Common redirection forms:

```bash
command > output.txt
command 1> output.txt
command 2> errors.txt
command >> output.txt
command 2>> errors.txt
```

- `>` overwrites the target file.
- `>>` appends to the target file.
- `1>` is explicit stdout redirection.
- `2>` redirects stderr only.

A practical example:

```bash
find /etc -name '*.conf' > matches.txt 2> errors.txt
```

This stores matches in one file and permission or path errors in another. That separation is useful in scripts that must continue even when some paths are inaccessible.

You can also combine streams:

```bash
command > all-output.txt 2>&1
```

This sends stderr to the same destination as stdout. In modern Bash, you can also use:

```bash
command &> all-output.txt
```

> Be careful with redirection order. `command >file 2>&1` sends both streams to `file`, but `command 2>&1 >file` first points stderr at the current stdout, then redirects stdout to `file`, so stderr may still go to the terminal.

**Input redirection**

Input redirection replaces interactive typing with data from a file or another source.

```bash
sort < names.txt
wc -l < access.log
```

- `<` feeds a file into stdin.
- This is useful when a command expects keyboard input but you want repeatable automation.
- Many commands can read from stdin by default, so `<` often makes scripts cleaner than passing filenames as arguments.

A common pattern is to process a file line by line:

```bash
while IFS= read -r line; do
  printf 'Line: %s\n' "$line"
done < input.txt
```

- `IFS=` prevents trimming leading and trailing whitespace.
- `read -r` prevents backslash interpretation.
- Redirecting the loop’s stdin from a file keeps the loop self-contained.

**Pipes versus redirection**

Pipes connect commands in a chain. Redirection connects a command to a file, device, or here-document.

```bash
grep 'error' app.log | sort | uniq -c
```

- `|` passes stdout from one command to stdin of the next.
- Pipes are ideal for transforming streams.
- Redirection is ideal for reading from or writing to files directly.

You often combine both:

```bash
grep -i 'fail' /var/log/auth.log | tee failed-auth.txt
```

`tee` writes to stdout and a file at the same time, which is useful when you want to inspect output and preserve it.

**Appending, truncating, and safe logging**

Use append redirection for logs that should preserve history.

```bash
echo "$(date -Is) backup started" >> /var/log/backup.log
```

Use overwrite redirection when the file should contain only the latest run.

```bash
generate_report > report.txt
```

If a script writes to a file that already exists, `>` truncates it immediately. That can be dangerous if the command later fails. A safer pattern is to write to a temporary file and move it into place after success.

```bash
tmpfile=$(mktemp)
if generate_report > "$tmpfile"; then
  mv "$tmpfile" report.txt
else
  rm -f "$tmpfile"
fi
```

**File descriptors**

A file descriptor is a numeric handle to an open file or stream. In shell scripting, the standard descriptors are `0`, `1`, and `2`, but you can use higher numbers for custom channels.

Examples:

```bash
exec 3> debug.log
echo "debug message" >&3
exec 3>&-
```

- `exec 3> debug.log` opens descriptor `3` for writing.
- `>&3` sends output to descriptor `3`.
- `exec 3>&-` closes descriptor `3`.

You can duplicate descriptors:

```bash
command 2>&1
```

This duplicates stdout onto stderr, so both streams share the same destination. Duplicating an existing descriptor is not the same as creating a new source of input or output.

A useful input example is:

```bash
exec 3< input.txt
while read -r line <&3; do
  printf '%s\n' "$line"
done
exec 3<&-
```

Here descriptor `3` is opened for reading from `input.txt`, then the loop reads from that descriptor. This works because descriptor `3` already exists and is open for input.

**Here-documents**

A here-document lets you embed multiline input directly in a script.

```bash
cat <<EOF
line one
line two
EOF
```

- `<<EOF` sends the following lines to stdin until the delimiter line `EOF` appears alone.
- If you quote the delimiter token, for example `<<'EOF'`, the shell does not perform parameter expansion, command substitution, or arithmetic expansion inside the body.
- If the delimiter is unquoted, expansions occur in the body.

Example with expansion:

```bash
name="Linux"
cat <<EOF
Hello, $name
Today is $(date +%F)
EOF
```

Example without expansion:

```bash
cat <<'EOF'
Hello, $name
Today is $(date +%F)
EOF
```

Here-documents are useful for generating config files, feeding commands to utilities, or embedding test data in scripts.

A more realistic example:

```bash
cat > /etc/myapp.conf <<'EOF'
[main]
enabled=true
log_level=info
EOF
```

> Use caution when writing to privileged paths. A typo in the target path can overwrite the wrong file.

**Here-strings**

A here-string sends a single string to stdin.

```bash
grep 'root' <<< "$line"
```

- `<<<` is convenient for one-line input.
- It is often simpler than a here-document when you only need one value.
- Quote variables to preserve spaces and prevent word splitting.

**Suppressing output and errors**

Sometimes you want to silence a command in scripts.

```bash
command >/dev/null 2>&1
```

- `/dev/null` is the null device; anything written there is discarded.
- Redirect stdout and stderr separately when you need fine control.
- Use suppression sparingly because it can hide useful diagnostics.

Examples:

```bash
rm -f missing-file 2>/dev/null
```

```bash
command >/dev/null
```

The first example hides error messages from a harmless cleanup step. The second hides normal output but still shows errors.

**Practical script patterns**

A script often combines several redirection techniques.

```bash
#!/usr/bin/env bash

logfile=/var/log/myjob.log
inputfile=/tmp/items.txt

{
  echo "Job started: $(date -Is)"
  while IFS= read -r item; do
    printf 'Processing: %s\n' "$item"
  done < "$inputfile"
  echo "Job finished: $(date -Is)"
} >> "$logfile" 2>&1
```

This pattern:

- groups multiple commands with `{ ...; }`
- appends all output to one log file
- merges stderr into stdout
- reads input from a file inside the loop

Another useful pattern is capturing command output while still displaying it:

```bash
backup_output=$(tar -czf backup.tgz /home 2>&1)
status=$?
printf '%s\n' "$backup_output"
exit "$status"
```

This captures both stdout and stderr in a variable, but note that command substitution strips trailing newlines. For large output, prefer a file or `tee`.

**Common pitfalls**

- `>` truncates files immediately, so test paths carefully.
- `2>&1` depends on redirection order.
- Unquoted variables can split into multiple words and break redirection targets.
- `read` normally reads one line at a time, so it is not a general-purpose consumer for multiline here-documents.
- A descriptor like `0<&3` only works if descriptor `3` is already open for reading.

**Quick reference**

| Syntax | Meaning |
|---|---|
| `cmd > file` | Redirect stdout to `file` |
| `cmd >> file` | Append stdout to `file` |
| `cmd 2> file` | Redirect stderr to `file` |
| `cmd < file` | Read stdin from `file` |
| `cmd > file 2>&1` | Send stdout and stderr to `file` |
| `cmd | other` | Pipe stdout into another command |
| `cmd <<EOF` | Feed a here-document to stdin |
| `cmd <<'EOF'` | Here-document with no expansion |
| `cmd <<< "text"` | Feed a here-string to stdin |

**Exam focus**

For Linux+ scripting tasks, make sure you can:

- distinguish stdin, stdout, and stderr
- choose between pipes and redirection
- append versus overwrite output correctly
- use here-documents with and without expansion
- explain descriptor duplication with `2>&1`
- redirect input into loops and scripts safely
- suppress output intentionally without hiding important errors
