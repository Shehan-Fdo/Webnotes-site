**Variables in shell scripts**

Variables store data that you reuse across commands, tests, and function calls. In POSIX shells and Bash, you typically assign values without spaces around `=` and reference them with `$name` or `${name}`. Braces are especially useful when you need to avoid ambiguity in longer strings.

- Assign with `name=value`; do not use spaces around the equals sign.
- Read with `$name` or `${name}`.
- Use `readonly` for values that should not change.
- Use `export` when child processes need the variable.
- Prefer `local` inside functions to avoid leaking values into the global scope.

```bash
#!/usr/bin/env bash

name="linuxadmin"
path="/var/log"
readonly env="production"

echo "User: ${name}"
echo "Logs: ${path}"
```

Variables can be untyped in shell, but the shell still treats them as strings unless a command interprets them differently. Arithmetic expansion and test expressions convert values as needed. This is why quoting matters: unquoted variables can split into multiple words or expand to filenames unexpectedly.

- Quote variables unless you intentionally want word splitting or glob expansion.
- Use `${var:-default}` to provide a fallback when a variable is unset or empty.
- Use `${var:=default}` to assign a default value if needed.
- Use `${var:?message}` to fail fast when a required variable is missing.

```bash
backup_dir="${BACKUP_DIR:-/srv/backups}"
echo "Using backup directory: $backup_dir"

: "${DB_HOST:?DB_HOST must be set}"
```

**Environment variables and scope**

A shell variable exists only in the current shell unless you export it. Environment variables are inherited by child processes, which is useful for tools that read configuration from the environment. This distinction matters when you run scripts, subshells, or commands launched from a parent shell.

- Shell variables affect only the current shell process.
- Exported variables become part of the environment.
- Subshells receive exported values, but changes in a subshell do not flow back to the parent.
- Command substitution runs in a subshell, so assignments inside it do not persist outside.

```bash
export EDITOR=vim
child_shell="$SHELL"
printf 'Shell: %s\n' "$child_shell"

output="$(date)"
echo "$output"
```

**Conditionals and exit status**

Shell conditionals rely on exit status, not just text comparisons. A command that succeeds returns status `0`; failure returns a nonzero status. The `if`, `elif`, and `else` keywords let you branch based on command results or test expressions. This is central to robust automation because you can check whether files exist, services are active, or commands completed successfully.

- `if` evaluates a command or test and runs the matching branch.
- `elif` adds additional checks without nesting deeply.
- `else` handles the default case.
- `test`, `[ ]`, and `[[ ]]` are common ways to evaluate conditions.
- `[[ ]]` is a Bash keyword with safer handling for pattern matching and fewer quoting pitfalls than `[ ]`.

```bash
#!/usr/bin/env bash

if [[ -f /etc/passwd ]]; then
  echo "passwd file exists"
elif [[ -d /etc/passwd ]]; then
  echo "Unexpected directory"
else
  echo "Missing"
fi
```

Use the right test operator for the data type you are checking. File tests, string tests, and numeric comparisons each have different operators. Mixing them up can produce subtle bugs.

| Category | Common operators | Example |
|---|---|---|
| File tests | `-f`, `-d`, `-e`, `-r`, `-w`, `-x` | `[[ -x /usr/bin/ssh ]]` |
| String tests | `=`, `!=`, `-z`, `-n` | `[[ -z "$user" ]]` |
| Numeric tests | `-eq`, `-ne`, `-lt`, `-le`, `-gt`, `-ge` | `[[ "$count" -gt 10 ]]` |

> Use numeric operators only for numbers. Comparing numeric strings with `>` or `<` in a single-bracket test can produce lexicographic results instead of arithmetic comparisons.

**Logical operators and compound tests**

You can combine conditions with `&&`, `||`, and grouping. In scripts, these operators help you express simple logic compactly, but they should not replace readable `if` blocks when the logic becomes complex.

- `&&` runs the next command only if the previous one succeeds.
- `||` runs the next command only if the previous one fails.
- `!` negates a condition or command result.
- Group related checks with parentheses or braces when needed.

```bash
[[ -f /etc/ssh/sshd_config ]] && echo "SSH config present"
[[ -r /etc/shadow ]] || echo "Need elevated privileges"

if ! systemctl is-active --quiet sshd; then
  echo "sshd is not running"
fi
```

**Loops for repetition**

Loops let you repeat actions over a list, a range, or until a condition changes. They are essential for batch operations such as processing files, checking multiple hosts, or retrying a command. Choose the loop type based on what drives the repetition.

- `for` iterates over items in a list.
- `while` repeats while a condition succeeds.
- `until` repeats until a condition succeeds.
- `break` exits a loop early.
- `continue` skips to the next iteration.

```bash
#!/usr/bin/env bash

for user in alice bob carol; do
  echo "Checking $user"
done
```

A `for` loop is often the cleanest choice when you already have a list of items. Be careful with word splitting: if items may contain spaces, use arrays in Bash or a safer input method.

```bash
files=(/var/log/*.log)

for file in "${files[@]}"; do
  [[ -e "$file" ]] || continue
  echo "Processing $file"
done
```

A `while` loop is useful when input is streamed line by line or when you need to keep going until a command fails. It is common in log processing and service polling.

```bash
while read -r line; do
  printf 'Line: %s\n' "$line"
done < /etc/hosts
```

`until` is the inverse of `while`. It is useful for retry logic when you want to keep trying until a service becomes available.

```bash
#!/usr/bin/env bash

count=0
until curl -fsS http://localhost:8080/health >/dev/null; do
  count=$((count + 1))
  if [[ $count -ge 10 ]]; then
    echo "Service did not become ready"
    exit 1
  fi
  sleep 2
done

echo "Service is ready"
```

**Arithmetic and incrementing values**

Shell arithmetic is commonly done with `$(( ... ))`. This is the standard way to calculate counters, indexes, and simple totals. Use arithmetic expansion instead of external tools when possible because it is faster and easier to read.

- Use `$((count + 1))` for arithmetic expansion.
- Use `((count++))` or `((++count))` for incrementing.
- Use `(( expression ))` as a conditional test in Bash.
- Avoid relying on string concatenation when you mean numeric addition.

```bash
count=0
((count++))
echo "$count"

total=$((5 + 7))
echo "$total"
```

**Functions for reusable logic**

Functions group commands into reusable units. They improve readability, reduce duplication, and make scripts easier to test. In shell scripts, functions can accept positional parameters just like scripts do, using `$1`, `$2`, and so on inside the function body.

- Define with `name() { ...; }` or `function name { ...; }` in Bash.
- Access arguments with `$1`, `$2`, and `"$@"`.
- Return status with `return`; use `echo` or `printf` for data output.
- Use `local` for function-scoped variables in Bash.
- Keep functions small and focused on one task.

```bash
#!/usr/bin/env bash

greet() {
  local user="$1"
  printf 'Hello, %s\n' "$user"
}

greet "admin"
```

A function should usually communicate success or failure through its exit status and send textual output to standard output. This separation makes it easier to compose functions in pipelines and conditionals.

```bash
check_port() {
  local host="$1"
  local port="$2"

  if nc -z "$host" "$port"; then
    return 0
  fi

  return 1
}

if check_port localhost 22; then
  echo "SSH is reachable"
fi
```

**Positional parameters and argument handling**

Scripts and functions receive arguments through positional parameters. `$0` is the script name, `$1` through `$9` are the first nine arguments, and `${10}` and beyond require braces. `"$@"` preserves each argument as a separate word, which is usually what you want when forwarding parameters.

- Use `"$@"` to forward all arguments safely.
- Use `"$#"` to count arguments.
- Use `shift` to move through arguments one by one.
- Use `${10}` for the tenth argument and beyond.

```bash
#!/usr/bin/env bash

echo "Script: $0"
echo "Arg count: $#"

for arg in "$@"; do
  echo "Arg: $arg"
done
```

**Case statements for multi-branch logic**

`case` is often cleaner than long `if` chains when you compare one value against several patterns. It is especially useful for menu handling, service actions, and parsing simple command options.

- Use `case` when one variable can match multiple patterns.
- Patterns can include wildcards such as `*` and `?`.
- End each branch with `;;` unless you intentionally want fall-through behavior in Bash-specific constructs.

```bash
#!/usr/bin/env bash

action="$1"

case "$action" in
  start)
    systemctl start nginx
    ;;
  stop)
    systemctl stop nginx
    ;;
  restart)
    systemctl restart nginx
    ;;
  *)
    echo "Usage: $0 {start|stop|restart}"
    exit 1
    ;;
esac
```

**Practical scripting patterns**

Real scripts often combine variables, tests, loops, and functions. A common pattern is to validate input, perform a repeated action, and stop on error. This structure makes scripts predictable and easier to troubleshoot.

- Validate required variables before doing work.
- Use functions for repeated tasks such as logging or checks.
- Loop over files, hosts, or services when the same action repeats.
- Check command exit status after each critical step.

```bash
#!/usr/bin/env bash

log() {
  printf '[%s] %s\n' "$(date +%F\ %T)" "$1"
}

backup_file() {
  local src="$1"
  local dest_dir="$2"

  [[ -f "$src" ]] || return 1
  cp -- "$src" "$dest_dir/"
}

src_dir="/etc"
dest_dir="/srv/backups"

for file in "$src_dir"/hosts "$src_dir"/passwd; do
  if backup_file "$file" "$dest_dir"; then
    log "Backed up $file"
  else
    log "Failed to back up $file"
  fi
done
```

**Debugging and safe scripting habits**

Good shell scripts fail clearly and avoid surprising expansions. Quoting, explicit tests, and careful variable handling prevent many common bugs. When troubleshooting, enable tracing to see how the shell expands commands.

- Use `set -x` to trace commands as they execute.
- Use `set -u` to treat unset variables as errors.
- Use `set -e` carefully; it can stop scripts early, but it may also surprise you in compound commands.
- Prefer `printf` over `echo` for predictable output.

```bash
#!/usr/bin/env bash
set -u

name="${1:-guest}"
printf 'Welcome, %s\n' "$name"
```

> `set -e` is helpful for simple scripts, but it does not replace explicit error handling. In complex scripts, check exit codes where failure handling matters.

**Common exam-relevant distinctions**

Understanding the differences between similar constructs helps you choose the right tool and avoid syntax errors.

- `[` is a command; `[[` is a Bash keyword with better parsing.
- `test` and `[` are broadly portable; `[[` is Bash-specific.
- `return` exits a function; `exit` ends the script.
- `"$@"` preserves arguments; `$*` joins them into one string in many contexts.
- `while read -r` prevents backslash interpretation in input lines.

| Concept | Best use | Notes |
|---|---|---|
| Variables | Store reusable values | Quote expansions |
| Conditionals | Branch on status or tests | Prefer readable logic |
| Loops | Repeat actions | Watch for word splitting |
| Functions | Reuse logic | Use `local` in Bash |

**Quick workflow for writing a script**

1. Define the inputs and defaults with variables.
2. Validate required values with tests or parameter expansion.
3. Put repeated logic into functions.
4. Use `if`, `case`, or loops to control flow.
5. Check exit status and handle errors explicitly.
6. Test with tracing before running in production.

**Example: combining all four concepts**

```bash
#!/usr/bin/env bash

set -u

log() {
  printf '[%s] %s\n' "$(date +%F\ %T)" "$1"
}

check_service() {
  local svc="$1"
  if systemctl is-active --quiet "$svc"; then
    return 0
  fi
  return 1
}

services=(sshd cron)

for svc in "${services[@]}"; do
  if check_service "$svc"; then
    log "$svc is running"
  else
    log "$svc is not running"
  fi
done
```

This pattern is common in Linux administration because it is readable, testable, and easy to extend. You use variables to hold state, conditionals to make decisions, loops to process multiple items, and functions to keep the script organized.
