**Cron fundamentals**

`cron` is a time-based job scheduler used to run commands automatically at specific times, dates, or intervals. You use it for recurring maintenance tasks such as log cleanup, backups, report generation, and health checks.

The `cron` daemon reads schedule definitions and executes jobs in the background. On most modern Linux systems, the service is managed by `systemd`, but the scheduling syntax remains the same.

- **cron daemon**: the background service that checks schedules and launches jobs.
- **crontab**: a per-user schedule file or the command used to edit it.
- **system crontab**: a global schedule file that can specify which user runs the job.
- **anacron**: a companion tool for systems that may be powered off when a job is due.

**Why use cron**

Cron is best when you need predictable repetition and simple time-based automation. It is not ideal for event-driven tasks or jobs that must react immediately to file changes; tools like `systemd` timers, `inotify`, or event hooks may fit better there.

- Use cron for:
  - nightly backups
  - log rotation support tasks
  - periodic cleanup
  - scheduled reports
- Avoid cron when:
  - the task depends on system events rather than time
  - you need missed jobs to run immediately after boot without extra tooling
  - you need complex dependency handling

**Cron schedule syntax**

A cron entry has five time fields followed by the command to run:

```text
* * * * * command-to-run
│ │ │ │ │
│ │ │ │ └─ day of week (0-7, where 0 and 7 are Sunday)
│ │ │ └─── month (1-12)
│ │ └───── day of month (1-31)
│ └─────── hour (0-23)
└───────── minute (0-59)
```

Each field can contain a single value, a list, a range, or a step value.

- `*` means every valid value in that field.
- `1,15,30` means multiple specific values.
- `1-5` means a range.
- `*/10` means every 10 units.
- `5-55/5` means every 5 units within a range.

Examples:

```cron
# Every day at 2:30 AM
30 2 * * * /usr/local/bin/backup.sh

# Every 15 minutes
*/15 * * * * /usr/bin/logger "heartbeat"

# Weekdays at 6:00 PM
0 18 * * 1-5 /usr/local/bin/endofday.sh
```

> Cron uses the system's local time unless the environment or implementation specifies otherwise. Time zone changes and daylight saving time can affect execution timing.

**Special strings**

Many cron implementations support shorthand expressions that improve readability:

- `@reboot` runs once at startup.
- `@yearly` or `@annually` runs once a year.
- `@monthly` runs once a month.
- `@weekly` runs once a week.
- `@daily` or `@midnight` runs once a day.
- `@hourly` runs once an hour.

Example:

```cron
@reboot /usr/local/bin/startup-check.sh
@daily /usr/local/bin/backup.sh
```

**User crontabs**

You manage a user's personal cron jobs with `crontab`.

Common commands:

```bash
crontab -e     # edit the current user's crontab
crontab -l     # list the current user's crontab
crontab -r     # remove the current user's crontab
crontab -u alice -l   # list another user's crontab (requires privilege)
```

User crontabs are typically stored in system-managed locations such as `/var/spool/cron/` or `/var/spool/cron/crontabs/`, depending on the distribution.

- User crontabs do not include a username field in each line.
- The command runs with the permissions of the crontab owner.
- Environment variables can be set at the top of the crontab.

Example:

```cron
SHELL=/bin/bash
PATH=/usr/local/bin:/usr/bin:/bin
MAILTO=admin@example.com

0 3 * * * /usr/local/bin/backup.sh
```

**System crontab and cron directories**

System-wide scheduling is often configured in `/etc/crontab` and in drop-in directories such as `/etc/cron.d/`.

A system crontab line includes an extra username field:

```cron
# minute hour day-of-month month day-of-week user command
15 4 * * * root /usr/local/bin/cleanup.sh
```

Common cron directories:

- `/etc/cron.hourly/`
- `/etc/cron.daily/`
- `/etc/cron.weekly/`
- `/etc/cron.monthly/`

These directories are typically processed by `run-parts`, which executes scripts found there according to the configured schedule.

- Use these directories for simple periodic maintenance.
- Use `/etc/cron.d/` when you need a package-managed or service-specific schedule.
- Use `/etc/crontab` for centrally managed system jobs.

**Permissions and access control**

Cron can restrict who may use it.

- `/etc/cron.allow` lists users permitted to use `crontab`.
- `/etc/cron.deny` lists users denied access.
- If `cron.allow` exists, it usually takes precedence.
- If neither file exists, behavior depends on the distribution, but many systems allow only privileged users or follow default package policy.

Security considerations:

- Keep scripts executable and owned appropriately.
- Use absolute paths in cron jobs because cron provides a minimal environment.
- Avoid relying on interactive shell aliases or profile settings.
- Restrict writable directories to prevent job tampering.

**Environment and execution context**

Cron jobs run with a limited environment. That means commands that work in your shell may fail in cron if they depend on missing variables, relative paths, or interactive prompts.

Common issues:

- `PATH` may not include `/usr/local/bin`.
- Shell features may differ from your login shell.
- Output is mailed to the job owner unless redirected.
- Jobs may fail silently if you do not capture stdout and stderr.

Practical example:

```cron
SHELL=/bin/bash
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin

0 1 * * * /usr/local/bin/backup.sh >> /var/log/backup.log 2>&1
```

**Logging and troubleshooting**

When a cron job does not run as expected, check the scheduler logs and the job’s own output.

Typical troubleshooting steps:

1. Verify the cron service is running.
2. Confirm the schedule syntax is valid.
3. Check file permissions and ownership.
4. Use absolute paths in the command.
5. Redirect output to a log file.
6. Review system logs for cron activity.

Useful commands:

```bash
systemctl status cron
systemctl status crond
journalctl -u cron
journalctl -u crond
grep CRON /var/log/syslog
grep crond /var/log/cron
```

> Service names and log locations vary by distribution. Debian-based systems often use `cron`, while Red Hat-based systems often use `crond`.

A quick validation technique is to schedule a job to run every minute and write to a file:

```cron
* * * * * /usr/bin/date >> /tmp/cron-test.log 2>&1
```

If the file does not update, inspect the environment, permissions, and service logs.

**Practical examples**

A backup job with log capture:

```cron
30 2 * * * /usr/local/bin/backup.sh --source /home --dest /backup >> /var/log/backup.log 2>&1
```

A cleanup job that removes files older than 7 days:

```cron
0 4 * * 0 /usr/bin/find /var/tmp -type f -mtime +7 -delete
```

A report job that runs on weekdays:

```cron
0 8 * * 1-5 /usr/local/bin/daily-report.sh
```

A startup task:

```cron
@reboot /usr/local/bin/initialize-cache.sh
```

**Related tools and alternatives**

Cron is simple and widely supported, but other scheduling tools may be better for certain workloads.

- **systemd timers**: better integration with service management, logging, and missed-run handling.
- **at**: one-time scheduled execution.
- **anacron**: useful for machines that are not always on.
- **batch**: runs jobs when system load is low.

Use cron when the task is recurring and time-based. Use `systemd` timers when you want tighter integration with modern Linux service management.

**Exam-focused takeaways**

- Know the five-field cron format and the meaning of each position.
- Remember that user crontabs do not include a username field.
- Remember that `/etc/crontab` and `/etc/cron.d/` do include a username field.
- Know the common cron directories and their purpose.
- Expect cron jobs to run with a minimal environment.
- Use absolute paths and redirect output for reliable automation.
- Distinguish cron from `at`, `anacron`, and `systemd` timers.
