## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Automation in Linux administration**

Automation reduces repetitive work, improves consistency, and lowers the chance of human error. In sysadmin work, you usually automate tasks that are frequent, time-sensitive, or risky when done manually. Good automation also makes actions auditable and repeatable.

Common goals include:

- **Consistency** across hosts and environments
- **Speed** for routine operations
- **Reliability** for scheduled or event-driven tasks
- **Traceability** through logs and version-controlled configuration
- **Safety** by reducing manual intervention in production

Typical automation layers include shell scripts, scheduled jobs, configuration management, and service managers. You choose the tool based on how often the task runs, what triggers it, and how much state it must manage.

**Common sysadmin automation use cases**

Automation is most useful when the same action must happen many times or when a system event should trigger a controlled response.

- **User lifecycle tasks**
  - Create accounts, set defaults, expire access, and remove stale users.
  - Apply consistent home directory permissions and shell settings.
- **Package and patch management**
  - Install baseline packages on new systems.
  - Apply updates during maintenance windows.
  - Verify package state after deployment.
- **Service management**
  - Start, stop, restart, or enable services after configuration changes.
  - Recover from failures with health checks and restart policies.
- **Log rotation and cleanup**
  - Archive logs, remove old files, and enforce retention policies.
  - Prevent disk exhaustion from uncontrolled growth.
- **Backups and restores**
  - Run scheduled backups and verify integrity.
  - Copy data to remote storage and alert on failures.
- **Configuration drift control**
  - Reapply desired settings after manual changes.
  - Keep files, permissions, and services aligned with policy.
- **Monitoring and alerting**
  - Trigger notifications when thresholds are crossed.
  - Run remediation steps for known conditions.
- **File ingestion workflows**
  - Process files dropped into a directory.
  - Move, validate, or import data after arrival.
- **Hardware or device events**
  - React to USB insertion, network changes, or storage attachment.
  - Hand off work to a service rather than doing heavy processing in the event hook.

**Choosing the right automation tool**

Different tools solve different problems. A strong exam answer explains why one tool fits better than another.

| Tool | Best for | Strengths | Limitations |
|---|---|---|---|
| Shell script | Small local tasks | Simple, portable, easy to chain | Harder to scale and maintain |
| `cron` / `at` | Time-based scheduling | Reliable scheduling, low overhead | Not event-driven |
| `systemd` timers | Scheduled jobs on modern Linux | Tight integration with services and logs | Requires systemd-based systems |
| `systemd` path units | File or directory change triggers | Event-driven, service handoff | Not for complex processing directly |
| `udev` rules | Device events | Detects hardware changes | Should not run long tasks inline |
| Ansible | Multi-host configuration | Idempotent, scalable, declarative | Requires inventory and control node |
| `rsync` + script | File synchronization | Efficient incremental transfer | Needs wrapper logic for orchestration |

**Shell scripts for local automation**

Shell scripts are the simplest automation layer. Use them for tasks that are local, linear, and easy to test. They are especially useful for glue logic between commands.

A robust script should:

- Set safe shell options such as `set -euo pipefail` when appropriate
- Validate inputs before making changes
- Log actions and failures
- Use absolute paths in scheduled contexts
- Exit with meaningful status codes

Example backup script:

```bash
#!/usr/bin/env bash
set -euo pipefail

SRC="/srv/data"
DEST="/backup/data-$(date +%F).tar.gz"

tar -czf "$DEST" "$SRC"
logger -t backup "Created backup $DEST"
```

Use scripts when you need custom logic, but keep them small enough to understand and maintain. If the workflow grows into multi-host orchestration or complex state management, move to a higher-level tool.

**Scheduling recurring work with cron**

`cron` runs commands at fixed times. It is ideal for backups, cleanup, reports, and periodic checks.

A user crontab entry:

```cron
# minute hour day-of-month month day-of-week command
15 2 * * * /usr/local/bin/backup.sh
```

A system-wide cron file may include a user field:

```cron
15 2 * * * root /usr/local/bin/backup.sh
```

Important cron practices:

- Use full paths for commands and files
- Redirect output to logs or mail
- Remember that cron uses a minimal environment
- Test commands interactively before scheduling them

Example with logging:

```cron
0 3 * * 0 /usr/local/bin/cleanup.sh >> /var/log/cleanup.log 2>&1
```

Use `at` for one-time delayed execution rather than recurring schedules.

```bash
echo "/usr/local/bin/reboot-check.sh" | at 23:30
atq
atrm 3
```

**systemd timers for scheduled jobs**

`systemd` timers are a modern alternative to cron on systems that use `systemd`. They pair a timer unit with a service unit, which improves logging, dependency handling, and integration with the service manager.

A timer is useful when you want:

- Better visibility in `journalctl`
- Dependency ordering before or after services
- Calendar-based or monotonic scheduling
- Easy enable/disable behavior with units

Example service unit:

```ini
# /etc/systemd/system/backup.service
[Unit]
Description=Run backup job

[Service]
Type=oneshot
ExecStart=/usr/local/bin/backup.sh
```

Example timer unit:

```ini
# /etc/systemd/system/backup.timer
[Unit]
Description=Daily backup timer

[Timer]
OnCalendar=*-*-* 02:15:00
Persistent=true

[Install]
WantedBy=timers.target
```

Enable and start the timer:

```bash
systemctl daemon-reload
systemctl enable --now backup.timer
systemctl list-timers
```

Use timers when you want scheduled execution with service semantics. `Persistent=true` helps catch missed runs after downtime.

**Event-driven automation with systemd path units**

Path units trigger a service when a file or directory changes. They are useful for drop directories, spool folders, and workflows where another process places files for processing.

For a directory drop workflow, monitor the directory itself rather than a single file. `PathExists=` only watches for a specific path to appear, so it is not the right choice for “any new file in this directory” behavior. Use `PathChanged=` or `PathModified=` on the directory, then hand off work to a service.

Example path unit:

```ini
# /etc/systemd/system/incoming.path
[Unit]
Description=Watch incoming directory

[Path]
PathChanged=/srv/incoming

[Install]
WantedBy=multi-user.target
```

Example service unit:

```ini
# /etc/systemd/system/incoming.service
[Unit]
Description=Process incoming files

[Service]
Type=oneshot
ExecStart=/usr/local/bin/process-incoming.sh
```

Enable the path unit:

```bash
systemctl daemon-reload
systemctl enable --now incoming.path
systemctl status incoming.path
```

A processing script might look like this:

```bash
#!/usr/bin/env bash
set -euo pipefail

for file in /srv/incoming/*; do
  [ -e "$file" ] || exit 0
  mv "$file" /srv/processed/
done
```

Key points for path units:

- Trigger on directory changes, not just one filename
- Keep the service short-lived and idempotent
- Avoid doing heavy work directly in the path trigger
- Expect multiple events for a burst of file activity

**Device-triggered automation with udev**

`udev` handles device events such as USB insertion, disk appearance, and hardware attribute matching. It is excellent for detection, but not for long-running jobs. If you need to mount media, copy data, or run a backup, hand off to a systemd service or another asynchronous mechanism.

A rule can match a device and start a service:

```udev
# /etc/udev/rules.d/99-usb-backup.rules
ACTION=="add", SUBSYSTEM=="block", ENV{ID_FS_LABEL}=="BACKUP", TAG+="systemd", ENV{SYSTEMD_WANTS}="usb-backup.service"
```

Example service:

```ini
# /etc/systemd/system/usb-backup.service
[Unit]
Description=USB backup handler

[Service]
Type=oneshot
ExecStart=/usr/local/bin/usb-backup.sh
```

Example script:

```bash
#!/usr/bin/env bash
set -euo pipefail

MNT="/mnt/backup-usb"
DEV="/dev/disk/by-label/BACKUP"

mkdir -p "$MNT"
mount "$DEV" "$MNT"
rsync -a /srv/data/ "$MNT/data/"
umount "$MNT"
```

Important `udev` guidance:

- Use `udev` to detect the event, not to perform the full workflow inline
- Keep rules simple and deterministic
- Prefer systemd service handoff for mounting, copying, or cleanup
- Be careful with device matching attributes such as label, vendor, or serial

> `RUN+=` in a udev rule is not appropriate for complex or long-running tasks. Use it only for very small, fast actions if at all, and prefer a service handoff for real work.

**Configuration management with Ansible**

Ansible is a common choice for repeatable multi-host administration. It uses an inventory of managed hosts and modules that aim for idempotence, meaning repeated runs converge on the same desired state.

Why Ansible is useful:

- You can apply the same configuration to many systems
- You can describe desired state instead of writing imperative steps
- You can run ad hoc commands or full playbooks
- You can use privilege escalation with `-b` when needed

Ad hoc example:

```bash
ansible all -m package -a "name=vim state=present" -b
```

The `package` module is a generic wrapper, so behavior depends on the target platform and available package manager. That makes it convenient, but you should still understand what backend it uses on each host.

A simple playbook:

```yaml
- name: Ensure web tools are installed
  hosts: webservers
  become: true
  tasks:
    - name: Install packages
      ansible.builtin.package:
        name:
          - curl
          - vim
        state: present
```

Ansible best practices:

- Use playbooks for repeatable workflows
- Use roles to organize larger automation projects
- Keep variables in inventory or group vars
- Test with `--check` and `--diff` when supported

Example dry run:

```bash
ansible-playbook site.yml --check --diff
```

**Service orchestration and recovery**

Automation often includes service lifecycle management. You may need to restart a daemon after a config change, enable it at boot, or verify it is healthy.

Common `systemctl` actions:

```bash
systemctl status sshd
systemctl restart nginx
systemctl enable --now firewalld
systemctl is-enabled cron
```

Use `enable --now` when you want a service or timer to start immediately and also persist across reboots. `start` alone affects only the current session.

You can also automate recovery with service unit settings:

```ini
[Service]
Restart=on-failure
RestartSec=5
```

This helps with transient failures, but it is not a substitute for fixing the root cause.

**Logging, verification, and safety**

Automation is only useful if you can verify it. Always plan how you will observe success and failure.

Useful verification commands:

```bash
journalctl -u backup.service
systemctl list-timers
systemctl status incoming.path
ansible-playbook site.yml --check
```

Good logging practices:

- Send script output to syslog with `logger` or to a file
- Use `journalctl` for systemd-managed jobs
- Include timestamps and context in messages
- Capture both standard output and standard error when scheduling jobs

Safety practices:

- Test in a non-production environment first
- Make scripts idempotent when possible
- Use lock files or `flock` to prevent overlapping runs
- Validate paths before deleting or moving data

Example lock usage:

```bash
flock -n /var/lock/backup.lock /usr/local/bin/backup.sh
```

**Practical decision guide**

Choose the simplest tool that fits the trigger and scope.

- Use a **shell script** for a local task with custom logic
- Use `cron` or `systemd` timers for time-based repetition
- Use a **path unit** for file arrival or directory change events
- Use `udev` to detect hardware events and hand off work
- Use **Ansible** for consistent multi-host configuration
- Use `systemctl enable --now` when the automation must survive reboot

**Exam-focused takeaways**

For Linux+ exam scenarios, focus on matching the tool to the use case.

- Scheduled backups usually map to `cron` or `systemd` timers
- File drop processing usually maps to `systemd` path units plus a service
- USB or disk detection usually maps to `udev` plus a service handoff
- Repeated host configuration usually maps to Ansible
- Service startup after configuration changes usually maps to `systemctl restart` or `enable --now`

A strong answer explains not just what the tool does, but why it is appropriate for the trigger, scale, and safety requirements of the task.
