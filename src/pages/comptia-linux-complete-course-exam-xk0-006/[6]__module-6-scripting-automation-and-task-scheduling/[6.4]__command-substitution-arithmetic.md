**Command substitution overview**

Command substitution lets you run a command and replace the substitution syntax with that command’s output. You use it when a script needs data produced at runtime, such as a hostname, a file count, a timestamp, or the result of a filter.

The modern, preferred syntax is `$(...)`. It is easier to read, supports nesting cleanly, and avoids many quoting problems associated with legacy backticks.

- Use **command substitution** when you need command output as text in another command or assignment.
- Use it in variable assignments, arguments, and string construction.
- Expect the shell to remove trailing newlines from the captured output.

Common examples:

```bash
today=$(date +%F)
host=$(hostname)
files=$(find /var/log -type f | wc -l)
echo "Today is $today on $host with $files log files"
```

**Syntax and quoting rules**

The `$(...)` form works inside double quotes, which is often what you want when the output may contain spaces. Inside single quotes, the shell treats the content literally, so substitution does not occur.

- Double quotes preserve spaces and most characters in the substituted text.
- Single quotes disable substitution entirely.
- Unquoted substitution can trigger word splitting and pathname expansion, which may break scripts.

Examples:

```bash
name=$(printf 'Alice Smith')
echo "$name"     # safe
echo $name       # word splitting may occur
echo '$(date)'   # literal text, no substitution
```

A practical rule is to quote command substitutions unless you intentionally want the shell to split the result into separate words.

**Legacy backticks and nesting**

The older form uses backticks: `` `command` ``. It still appears in older scripts, but it is harder to read and escape. Nested backticks require escaping inner backticks, which makes complex expressions error-prone.

- Prefer `$(...)` for new scripts.
- Use backticks only when maintaining older code or when a target shell lacks modern syntax.
- Remember that backticks are difficult to nest and to escape correctly.

Compare the two forms:

```bash
# Preferred
path=$(dirname "$(pwd)")

# Legacy and harder to read
path=`dirname \`pwd\``
```

If you need nested substitutions, `$(...)` is much clearer:

```bash
result=$(echo "User: $(whoami), Home: $(echo "$HOME")")
echo "$result"
```

**Multiple substitutions in one command**

You can combine several substitutions in a single line. This is common in prompts, reports, and scripts that assemble structured output.

```bash
echo "User=$(whoami) Host=$(hostname) Shell=$(basename "$SHELL")"
```

You can also nest substitutions when one command needs the output of another:

```bash
count=$(grep -c "$(whoami)" /var/log/auth.log 2>/dev/null)
echo "Login-related entries: $count"
```

> If the inner command produces spaces or special characters, quote it carefully before passing it to the outer command.

**Command substitution in practice**

Command substitution is useful for capturing command output into variables, command arguments, and configuration values.

- Capture a timestamp for log files.
- Build a dynamic filename.
- Feed one command’s output into another command.
- Store a filtered list for later processing.

Examples:

```bash
backup_file="/backup/home-$(date +%F).tar.gz"
tar -czf "$backup_file" "$HOME"
```

```bash
user_count=$(cut -d: -f1 /etc/passwd | wc -l)
printf 'Local users: %s\n' "$user_count"
```

```bash
kernel=$(uname -r)
echo "Running kernel: $kernel"
```

**Arithmetic expansion basics**

Arithmetic expansion evaluates an expression and replaces it with the numeric result. The standard form is `$(( expression ))`.

This is not the same as running an external calculator command. The shell evaluates the expression internally, so it is fast and convenient for counters, offsets, and simple calculations.

- Use **arithmetic expansion** for numeric results inside a command or assignment.
- Use integers only; shell arithmetic is integer-based.
- Expect shell variables used in arithmetic contexts to be interpreted as numbers.

Examples:

```bash
x=5
y=2
sum=$((x + y))
echo "$sum"   # 7
```

```bash
files=10
files=$((files + 1))
echo "$files"
```

Many POSIX-compliant shells allow bare variable names inside arithmetic expansion, so `x` is treated as the value of variable `x` without needing `$x` inside `$((...))`.

**Arithmetic expansion versus arithmetic evaluation**

The shell also supports arithmetic evaluation with `(( ... ))` in many shells, especially Bash and other common interactive shells. This form evaluates the expression for its side effects and returns an exit status rather than printing a value.

- `$((...))` produces a value you can assign or embed in text.
- `((...))` evaluates an expression as a command and is often used in conditionals or loops.
- `((...))` is widely available in Bash and other shells, but it is not the same as command substitution.

Examples:

```bash
# Expansion: produces text
count=$((count + 1))
echo "$count"

# Evaluation: useful for tests and increments
((count++))
if ((count > 10)); then
  echo "Threshold reached"
fi
```

A common exam point is that `((...))` does not need a leading `$` because it is a shell arithmetic command, not a substitution. In contrast, `$((...))` expands to a string result.

**POSIX sh versus Bash behavior**

POSIX specifies command substitution with `$(...)` and arithmetic expansion with `$((...))`. However, not every historical `/bin/sh` implementation supported all modern shell features equally well. For exam purposes, distinguish between what the standard defines and what a particular shell implementation provides.

- `$(...)` is the portable, standard command substitution form.
- Backticks are legacy and less readable.
- `$((...))` is part of POSIX shell arithmetic expansion.
- `((...))` is a shell feature commonly associated with Bash and similar shells, not a POSIX requirement.

Some arithmetic features are shell-specific:

- Exponentiation with `**` is Bash/ksh-style behavior, not guaranteed by POSIX sh.
- Base prefixes such as `10#08` are shell-specific and not portable across all POSIX sh implementations.
- Division by zero is an error in shell arithmetic.

Examples:

```bash
# Portable arithmetic
n=$((7 + 3))

# Bash/ksh-style base handling; not portable everywhere
n=$((10#08))
```

> Do not assume Bash-only arithmetic syntax works in `/bin/sh` scripts unless you know the target shell supports it.

**Arithmetic syntax and operators**

Shell arithmetic supports common integer operators such as addition, subtraction, multiplication, division, modulus, and grouping with parentheses.

Typical operators:

- `+` addition
- `-` subtraction
- `*` multiplication
- `/` integer division
- `%` remainder
- `()` grouping
- `<<` and `>>` bit shifts
- `&`, `|`, `^` bitwise operations
- `!` logical negation in arithmetic contexts
- `~` bitwise negation

Examples:

```bash
a=17
b=5
echo $((a / b))   # 3
echo $((a % b))   # 2
echo $((a * b))   # 85
```

Arithmetic expansion uses integer math, so `5 / 2` becomes `2`, not `2.5`.

**Variables, unset values, and errors**

Unset variables are usually treated as `0` in arithmetic expansion. That makes counters easy to initialize implicitly. However, a variable containing a non-numeric string is not the same as an unset variable.

- Unset variables often evaluate as `0`.
- A non-numeric assigned value can trigger an arithmetic error.
- Do not rely on silent conversion of arbitrary strings.

Examples:

```bash
unset x
echo $((x + 1))   # usually 1

x=abc
echo $((x + 1))   # error in Bash and many shells
```

This distinction matters in troubleshooting: an unset variable may be harmless, but a malformed value can break a script.

**Command substitution and arithmetic together**

You often combine both features in real scripts. Command substitution gathers data, and arithmetic expansion processes it.

Examples:

```bash
files=$(find /etc -maxdepth 1 -type f | wc -l)
next=$((files + 1))
echo "Files: $files, next index: $next"
```

```bash
size=$(du -sb /var/log | awk '{print $1}')
mb=$((size / 1024 / 1024))
echo "Log directory size: ${mb} MB"
```

```bash
users=$(cut -d: -f1 /etc/passwd | wc -l)
echo "User count is $((users + 0))"
```

**Practical CLI patterns**

These patterns appear often in administration scripts and exam questions.

- Capture command output into a variable.
- Use the variable in arithmetic.
- Quote the substitution when it may contain spaces.
- Redirect errors separately if you want to suppress or inspect them.

Examples:

```bash
timestamp=$(date +%Y%m%d-%H%M%S)
archive="/tmp/home-$timestamp.tar.gz"
tar -czf "$archive" "$HOME"
```

```bash
free_kb=$(awk '/MemAvailable/ {print $2}' /proc/meminfo)
free_mb=$((free_kb / 1024))
echo "Available memory: ${free_mb} MB"
```

```bash
pid_count=$(pgrep -u "$USER" | wc -l)
echo "Your processes: $pid_count"
```

**Troubleshooting and common pitfalls**

Many shell bugs come from quoting mistakes, unsupported syntax, or malformed expressions.

- Unquoted substitutions can split on spaces and tabs.
- Backticks are hard to nest and easy to misread.
- A missing `)` or `}` causes a syntax error.
- Arithmetic expressions fail on invalid numeric input.
- Command substitution inside pipelines can hide failures if you do not check exit status.

Examples of problems and fixes:

```bash
# Problem: word splitting
files=$(printf 'a b c')
for f in $files; do
  echo "$f"
done

# Better: preserve intended structure or use arrays in Bash
printf '%s\n' "$files"
```

```bash
# Problem: malformed arithmetic
echo $((5 + ))   # syntax error

# Fix
echo $((5 + 1))
```

```bash
# Problem: stderr is not captured by command substitution
result=$(some_command 2>/tmp/error.log)

# If you want stdout and stderr together, redirect both
result=$(some_command 2>&1)
```

If you need to inspect both output streams separately, redirect them to different files or use a more explicit logging approach.

**Exam-focused distinctions**

For Linux+ exam prep, focus on what each construct does and when to choose it.

- **Command substitution** replaces syntax with command output.
- **Arithmetic expansion** replaces syntax with a numeric result.
- `$(...)` is the preferred command substitution syntax.
- `$((...))` is the standard arithmetic expansion syntax.
- `((...))` is an arithmetic evaluation command used for tests and increments.
- POSIX guarantees the standard forms, while Bash adds extra arithmetic conveniences and syntax.

A quick memory aid:

- Use `$(...)` for text from commands.
- Use `$((...))` for numbers.
- Use `((...))` when you want arithmetic as a shell test or increment.

**Quick reference**

| Feature | Syntax | Result | Notes |
|---|---|---|---|
| Command substitution | `$(command)` | Text output | Preferred over backticks |
| Legacy command substitution | `` `command` `` | Text output | Harder to nest and escape |
| Arithmetic expansion | `$((expression))` | Numeric text | Standard shell feature |
| Arithmetic evaluation | `((expression))` | Exit status | Common in Bash-style shells |

**Key takeaways**

Command substitution and arithmetic expansion are core shell features for building dynamic scripts. Use `$(...)` for readable, nestable command output capture, and use `$((...))` for integer math. Keep quoting in mind, distinguish standard POSIX behavior from Bash-specific extensions, and watch for syntax and numeric errors when values are malformed.
