**Exit codes and error handling**

Linux commands and scripts communicate success or failure through an <abbr title="exit status">exit code</abbr>, also called a return code. You use these values to automate decisions, detect failures early, and make scripts safer in production. In shell scripting, error handling is not just about printing messages; it is about checking command outcomes, stopping on critical failures, and preserving the correct status for callers.

A command usually returns `0` for success and a nonzero value for failure. The exact nonzero value often indicates the type of problem, but the meaning is command-specific. For example, `grep` uses `1` to mean “no matches,” which is not always an error in the same sense as a syntax failure.

- `0` means success.
- Nonzero values mean some kind of failure or special condition.
- The shell stores the most recent exit code in `$?`.

```bash
ls /etc/passwd
echo $?

grep -q "nologin" /etc/passwd
echo $?
```

**Understanding exit status**

The shell captures the exit status of the last foreground pipeline or command. You inspect it with `$?` immediately after the command finishes. If you run another command first, `$?` changes, so timing matters.

Common conventions include:

- `0` — success
- `1` — general error or no match
- `2` — misuse of shell builtins or invalid usage
- `126` — command found but not executable
- `127` — command not found
- `128+n` — command terminated by signal `n`

These conventions are widely used, but not every program follows them exactly. For exam purposes, know the common meanings and the fact that command-specific exit codes may vary.

```bash
command_that_fails
status=$?
echo "Exit status: $status"
```

A practical pattern is to save the exit code before doing anything else:

```bash
cp /important/file /backup/
status=$?
logger "backup copy exit status: $status"
```

**Checking command success in the shell**

You often test exit codes with conditional statements. The shell treats `0` as true and nonzero as false in command contexts, which is the opposite of many programming languages.

```bash
if grep -q "PermitRootLogin" /etc/ssh/sshd_config; then
  echo "Directive exists"
else
  echo "Directive missing"
fi
```

You can also use the `&&` and `||` operators for short-circuit logic:

```bash
mkdir -p /var/log/myapp && echo "Directory ready"
test -f /etc/hosts || echo "hosts file missing"
```

Use these carefully. `cmd1 && cmd2` runs `cmd2` only if `cmd1` succeeds. `cmd1 || cmd2` runs `cmd2` only if `cmd1` fails. This is useful for compact scripts, but it can hide complex logic if overused.

**`test`, `[ ]`, and `[[ ]]`**

The `test` command and its bracket forms are common in shell conditionals. They return exit codes rather than printing values.

```bash
test -d /etc && echo "etc is a directory"
[ -r /etc/passwd ] && echo "readable"
[[ -n "$USER" ]] && echo "user set"
```

Key points:

- `test` and `[` are POSIX-compatible shell builtins in many shells.
- `[[ ]]` is a Bash and some other shell feature with safer parsing and richer operators.
- The result of the expression is represented by the exit code, not by output.

**Pipelines and error propagation**

A pipeline passes the output of one command to another, but by default the shell usually reports the exit status of the last command in the pipeline. That can hide failures earlier in the chain.

```bash
grep "ERROR" /var/log/messages | wc -l
echo $?
```

If `grep` fails to read the file, `wc` may still succeed, and `$?` may show success from `wc`. To detect failures in pipelines, use `pipefail` in shells that support it, such as Bash:

```bash
set -o pipefail
grep "ERROR" /var/log/messages | wc -l
echo $?
```

With `pipefail`, the pipeline returns a failure if any command in the pipeline fails. This is especially useful in automation, CI/CD, and administrative scripts where silent partial failure is dangerous.

**Fail-fast scripting with `set` options**

Shell options help you stop scripts when something goes wrong. The most common are:

- `set -e` — exit immediately if a simple command fails
- `set -u` — treat unset variables as an error
- `set -o pipefail` — fail a pipeline if any component fails

A common defensive pattern is:

```bash
#!/usr/bin/env bash
set -euo pipefail

backup_dir="/backup"
src="/etc/ssh/sshd_config"

mkdir -p "$backup_dir"
cp "$src" "$backup_dir/"
echo "Backup completed"
```

These options improve reliability, but they are not magic. `set -e` has exceptions and can behave unexpectedly in compound commands, conditionals, and subshells. You still need explicit checks for commands whose failure is acceptable or expected.

> Be careful with `set -e` in scripts that intentionally test failure conditions. A command that is supposed to fail may terminate the script unless you handle it explicitly.

**Explicit error handling patterns**

For robust scripts, check return codes and handle failures near the point of occurrence. This makes scripts easier to debug and safer to maintain.

```bash
if ! useradd appuser; then
  echo "Failed to create user" >&2
  exit 1
fi
```

You can also use a reusable helper function:

```bash
die() {
  echo "Error: $*" >&2
  exit 1
}

cp /etc/hosts /backup/hosts || die "backup copy failed"
```

This pattern is useful when you want a clear error message and a controlled exit status. Use standard error (`>&2`) for error messages so that normal output stays separate from diagnostics.

**Exit codes in functions and scripts**

Functions return exit statuses just like commands. In shell, `return` sets the function’s exit status, while `exit` terminates the entire script or shell session.

```bash
check_file() {
  [ -f "$1" ] || return 1
  return 0
}

check_file /etc/passwd
echo $?
```

Use `return` inside functions and `exit` at the top level when you want to stop the whole script. If a function fails and you want the script to stop, combine it with `set -e` or check the function explicitly.

**Signals and exit codes**

When a process is terminated by a signal, shells often report an exit status of `128 + signal number`. For example, if a process receives `SIGINT` (`2`), the exit code is often `130`. This is useful when diagnosing interrupted jobs or aborted scripts.

Common examples:

- `130` — interrupted by `Ctrl` + `C` (`SIGINT`)
- `137` — killed by `SIGKILL` (`9`)
- `143` — terminated by `SIGTERM` (`15`)

These values help you distinguish a normal program failure from an external termination event.

**Practical troubleshooting workflow**

When a command fails, you should identify the failing command, capture its exit status, and inspect related logs or stderr output. A disciplined workflow reduces guesswork.

1. Re-run the command and capture the exit code immediately with `echo $?`.
2. Check whether the command wrote useful diagnostics to stderr.
3. Verify input paths, permissions, and environment variables.
4. If the command is in a pipeline, test each stage separately.
5. Review service logs or system logs when the command interacts with daemons.

```bash
systemctl restart sshd
echo $?

journalctl -u sshd -n 20 --no-pager
```

**Real-world examples**

A backup script should stop if the source file is missing:

```bash
#!/usr/bin/env bash
set -euo pipefail

src="/etc/fstab"
dest="/var/backups/fstab.$(date +%F)"

[ -r "$src" ] || { echo "Source not readable" >&2; exit 1; }
cp "$src" "$dest"
```

A monitoring check may treat “no matches” as a valid result rather than a fatal error:

```bash
if grep -q "FAILED" /var/log/myapp.log; then
  echo "Failures detected"
else
  echo "No failures found"
fi
```

A pipeline can be made safer with `pipefail`:

```bash
set -o pipefail
tar -czf /tmp/home.tgz /home | tee /tmp/backup.log
status=$?
echo "Pipeline status: $status"
```

**Best practices**

- Check exit codes for commands that affect state, such as `cp`, `mv`, `rm`, `useradd`, and service management commands.
- Use `set -euo pipefail` in scripts where failure should stop execution.
- Handle expected failures explicitly instead of relying on global shell options alone.
- Send error messages to stderr with `>&2`.
- Preserve the original exit code when writing wrapper scripts or helper functions.
- Test pipelines carefully, because the last command may hide earlier failures.

**Common pitfalls**

- Reading `$?` too late after another command has already run.
- Assuming every nonzero exit code means the same thing.
- Using `set -e` without understanding its exceptions.
- Ignoring pipeline failures because only the last command succeeded.
- Mixing normal output and error output, which makes automation harder.

**Quick reference**

| Exit code | Typical meaning |
|---:|---|
| `0` | Success |
| `1` | General failure or no match |
| `2` | Misuse or invalid usage |
| `126` | Found but not executable |
| `127` | Command not found |
| `130` | Interrupted by `Ctrl` + `C` |
| `137` | Killed by `SIGKILL` |
| `143` | Terminated by `SIGTERM` |

**Key takeaway**

Exit codes are the shell’s primary success-and-failure signal. You use them to branch logic, stop scripts safely, and diagnose problems quickly. Good error handling means checking results early, handling expected failures deliberately, and making pipelines and scripts fail in predictable ways.
