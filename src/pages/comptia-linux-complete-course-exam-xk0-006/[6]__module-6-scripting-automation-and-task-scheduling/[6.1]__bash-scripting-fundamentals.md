## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Bash scripting overview**

Bash (<abbr title="Bourne Again SHell">Bash</abbr>) is the default shell on many Linux systems and a common scripting language for automation, administration, and exam scenarios. You use it to combine commands, make decisions, process text, and handle files without manual repetition.

A Bash script is just a text file containing shell commands. The shell reads and executes them line by line, with special syntax for variables, tests, loops, functions, and redirection. Good scripts are predictable, readable, and defensive.

- Use Bash when you need simple automation, system checks, or glue between existing commands.
- Prefer built-in shell features when they are faster and easier than spawning external processes.
- Write scripts for repeatability, not just convenience.

**Script structure and metadata**

A script usually starts with a shebang, which tells the kernel which interpreter to use. For Bash, that is commonly `#!/usr/bin/env bash` or `#!/bin/bash`. The `env` form is more portable when Bash may live in a nonstandard path.

Beyond the shebang, scripts often include metadata comments that help humans and tools understand the file. Common conventions include a short description, author, version, usage, and license. These comments do not affect execution, but they improve maintainability.

```bash
#!/usr/bin/env bash
# Description: Back up a directory to a timestamped archive
# Author: Webnotes
# Version: 1.0
# Usage: ./backup.sh /path/to/source /path/to/destination

set -euo pipefail
```

- **Shebang** selects the interpreter for direct execution.
- **Comments** document intent, assumptions, and usage.
- **Metadata** is especially useful in shared scripts and exam labs.
- `set -euo pipefail` is a common safety baseline for stricter error handling.

> A script without a shebang may still run if you invoke it with `bash script.sh`, but direct execution depends on the file being executable and having a valid interpreter line.

**Running scripts and permissions**

To run a script directly, you need execute permission. To inspect or edit it, you only need read permission. The shell does not care whether the file has a `.sh` extension; the shebang and permissions matter more.

```bash
chmod +x backup.sh
./backup.sh
bash backup.sh
```

- `chmod +x` adds execute permission.
- `./script` runs the file as a program.
- `bash script` explicitly starts Bash and reads the file.
- Use `bash script` when testing a script that is not yet executable.

**Variables and quoting**

Variables store data for reuse. In Bash, assignment has no spaces around `=`.

```bash
name="Linux"
count=5
path="/var/log"
```

Use quotes carefully. Double quotes preserve spaces while still allowing variable expansion. Single quotes prevent expansion. Unquoted variables can split into multiple words and trigger pathname expansion, which often causes bugs.

```bash
echo "$name"
echo '$name'
echo "$path"
```

- **Double quotes** are the default safe choice for variable expansion.
- **Single quotes** are useful for literal strings.
- Unquoted expansions are risky when values may contain spaces, tabs, or glob characters.
- Use braces like `${var}` when you need to separate a variable name from adjacent text.

Common parameter expansion patterns help you manipulate values without external tools:

```bash
file="report.txt"
echo "${file%.txt}"     # remove shortest .txt suffix
echo "${file#report}"   # remove shortest report prefix
echo "${#file}"         # length of the string
```

- `${#var}` returns string length.
- `${var#pattern}` removes the shortest matching prefix.
- `${var##pattern}` removes the longest matching prefix.
- `${var%pattern}` removes the shortest matching suffix.
- `${var%%pattern}` removes the longest matching suffix.

**String manipulation basics**

Bash can do simple string work directly, which is often enough for exam-level scripting and small automation tasks. Case conversion and trimming are especially useful when normalizing user input or parsing command output.

```bash
word="linux"
echo "${word^^}"   # LINUX
echo "${word,,}"   # linux
```

```bash
text="   hello world   "
echo "${text#"${text%%[![:space:]]*}"}"   # trim leading whitespace
echo "${text%"${text##*[![:space:]]}"}"   # trim trailing whitespace
```

- **Case conversion** uses `^^` for uppercase and `,,` for lowercase.
- Pattern removal can trim known prefixes or suffixes.
- Bash trimming tricks are useful, but external tools like `sed` or `awk` may be clearer for complex text processing.
- Prefer simple parameter expansion when the transformation is small and predictable.

**Positional parameters and special variables**

Scripts often accept input from the command line. Positional parameters let you access arguments by number.

```bash
#!/usr/bin/env bash
echo "Script name: $0"
echo "First arg: $1"
echo "Second arg: $2"
echo "All args: $*"
echo "All args safely: $@"
echo "Arg count: $#"
```

- `$0` is the script name or invocation path.
- `$1`, `$2`, and so on are positional arguments.
- `$#` is the number of arguments.
- `"$@"` preserves each argument as a separate word and is usually the safest way to forward arguments.
- `"$*"` joins arguments into one word when quoted, which is less common.

Other useful special variables include:

- `$$` for the current shell process ID.
- `$?` for the exit status of the last command.
- `$!` for the process ID of the most recent background job.
- `$_` for the last argument of the previous command in many shells.

**Exit status and command chaining**

Bash uses exit status to signal success or failure. By convention, `0` means success and nonzero means failure. You often combine commands based on these results.

```bash
grep -q "ERROR" app.log && echo "Errors found"
grep -q "ERROR" app.log || echo "No errors found"
```

- `&&` runs the next command only if the previous one succeeds.
- `||` runs the next command only if the previous one fails.
- Exit status is central to scripting because many commands communicate results this way.
- Use `set -e` carefully; it can stop scripts early, which is useful for safety but can surprise you in complex logic.

**Tests and comparisons**

The `test` command and its `[ ... ]` form are used for conditional checks. `[[ ... ]]` is a Bash keyword with safer parsing and more features, so it is usually preferred in Bash scripts.

Numeric comparisons use operators like `-eq`, `-ne`, `-lt`, `-le`, `-gt`, and `-ge`. String comparisons use operators like `=`, `!=`, `<`, and `>` inside `[[ ... ]]`, or `=` and `!=` in `[ ... ]`.

```bash
if [ "$count" -gt 10 ]; then
  echo "Large count"
fi

if [[ "$name" == "admin" ]]; then
  echo "Administrator"
fi
```

- **Numeric tests** compare integer values.
- **String tests** compare text values.
- Use quotes around variables in tests to avoid errors with empty values.
- `[[ ... ]]` handles pattern matching and avoids some word-splitting issues.

Common file tests include:

- `-e` file exists
- `-f` regular file
- `-d` directory
- `-r` readable
- `-w` writable
- `-x` executable
- `-s` nonempty file

**Input and output redirection**

Redirection controls where input comes from and where output goes. This is fundamental to Bash scripting because it lets you capture results, suppress noise, and connect commands together.

```bash
command > output.txt
command >> output.txt
command < input.txt
command 2> errors.txt
command &> all.txt
```

- `>` redirects standard output (`stdout`) and overwrites the file.
- `>>` appends standard output.
- `<` redirects standard input (`stdin`) from a file.
- `2>` redirects standard error (`stderr`).
- `&>` redirects both standard output and standard error in Bash.
- `2>&1` merges standard error into standard output.

File descriptors are numeric handles for streams:

- `0` = standard input
- `1` = standard output
- `2` = standard error

You can combine redirection with pipelines and `tee` to inspect output while saving it.

```bash
grep "kernel" /var/log/messages | tee kernel.log
command | tee output.txt | less
```

- **Pipes** send standard output from one command to another command’s standard input.
- `tee` duplicates a stream to both the terminal and a file.
- Use `tee` when you want visibility during long pipelines or troubleshooting.

> Redirection order matters. `command >file 2>&1` is not the same as `command 2>&1 >file` because the file descriptors are duplicated at different times.

**Command grouping and subshells**

Grouping changes how commands are executed and how variable scope behaves. Parentheses create a **subshell**, while braces create a **command group** in the current shell.

```bash
(
  cd /tmp
  pwd
)

{
  echo "Start"
  echo "Still in current shell"
}
```

- `( ... )` runs commands in a subshell.
- `{ ...; }` runs commands in the current shell.
- A subshell gets its own environment copy, so variable changes do not persist outside it.
- A brace group does not create a new process, so variable changes remain visible afterward.

Example of scope difference:

```bash
x=1
(
  x=2
  echo "Inside subshell: $x"
)
echo "Outside subshell: $x"

{
  x=3
  echo "Inside group: $x"
}
echo "Outside group: $x"
```

- Use a subshell when you want isolation.
- Use a brace group when you want to apply redirection to multiple commands without forking.

**Functions and local scope**

Functions organize repeated logic and make scripts easier to read. They also help you isolate behavior and reuse code with arguments.

```bash
greet() {
  local name="$1"
  echo "Hello, $name"
}
```

- **local variables** exist only inside the function.
- Without `local`, variables are global by default in Bash.
- Function arguments are accessed with positional parameters just like script arguments.
- Use `return` for exit status, not for returning strings.

```bash
is_root() {
  [[ $EUID -eq 0 ]]
}

if is_root; then
  echo "Running as root"
fi
```

Scope rules to remember:

- Variables set inside a function are global unless declared `local`.
- Variables set in a subshell do not affect the parent shell.
- A function can read global variables unless shadowed by a local variable.
- `export` makes a variable available to child processes, but not automatically to the parent shell.

**Conditionals and branching**

`if`, `elif`, and `else` let you choose behavior based on command results or tests. Bash also supports `case` for cleaner multi-branch matching.

```bash
if [[ -f /etc/passwd ]]; then
  echo "passwd exists"
elif [[ -d /etc/passwd ]]; then
  echo "Unexpected directory"
else
  echo "Missing"
fi
```

```bash
case "$1" in
  start)
    echo "Starting"
    ;;
  stop)
    echo "Stopping"
    ;;
  restart)
    echo "Restarting"
    ;;
  *)
    echo "Usage: $0 {start|stop|restart}"
    ;;
esac
```

- Use `if` for boolean-style checks and numeric comparisons.
- Use `case` for matching one value against several patterns.
- `case` is often cleaner for command-line argument handling.

**Loops and loop control**

Loops repeat commands over a list or until a condition changes. Bash supports `for`, `while`, and `until`.

```bash
for file in *.log; do
  echo "Processing $file"
done
```

```bash
while read -r line; do
  echo "$line"
done < input.txt
```

```bash
until ping -c1 8.8.8.8 >/dev/null 2>&1; do
  echo "Waiting for network..."
  sleep 1
done
```

- `for` is common for iterating over items.
- `while` is common for reading streams or repeating until a condition fails.
- `until` repeats until the test succeeds.

Loop control keywords help you manage nested logic:

```bash
for dir in /etc /var /tmp; do
  for file in "$dir"/*; do
    [[ -e "$file" ]] || continue
    [[ -f "$file" ]] || continue
    [[ "$file" == *.bak ]] && continue
    echo "$file"
    break
  done
done
```

- `break` exits the current loop.
- `continue` skips to the next iteration.
- In nested loops, `break 2` exits two levels, and `continue 2` skips two levels.
- Use loop control to avoid deeply nested conditionals.

**Reading user input safely**

The `read` builtin collects input from the user or a stream. It is commonly used for prompts, menus, and interactive scripts.

```bash
read -r -p "Enter hostname: " host
echo "You entered: $host"
```

- `-p` shows a prompt.
- `-r` prevents backslash escaping, which is usually safer.
- Quote the variable when you use it later.
- Avoid `echo` for sensitive input; use `read -s` for silent entry such as passwords.

Timeouts and defaults are useful in automation:

```bash
if read -r -t 5 -p "Continue? [y/N] " answer; then
  echo "Answer: $answer"
else
  echo "Timed out"
fi
```

- `-t` sets a timeout in seconds.
- `-n` limits the number of characters read.
- Validate user input before using it in commands.

**Arrays and iteration patterns**

Arrays let you store multiple values in one variable. Bash supports indexed arrays and associative arrays.

```bash
servers=("web1" "web2" "db1")
echo "${servers[0]}"
echo "${servers[@]}"
echo "${#servers[@]}"
```

- Indexed arrays use numeric indexes.
- `${array[@]}` expands each element separately when quoted.
- `${#array[@]}` returns the number of elements.

Safe iteration usually uses quoted expansions:

```bash
for server in "${servers[@]}"; do
  echo "Checking $server"
done
```

Associative arrays map keys to values:

```bash
declare -A ports
ports[http]=80
ports[https]=443
echo "${ports[http]}"
```

- **Associative arrays** require `declare -A`.
- Keys are strings, not numeric indexes.
- Not all shells support associative arrays; Bash does, but portability to other shells may be limited.
- When iterating keys, use `"${!ports[@]}"` to preserve keys safely.

```bash
for key in "${!ports[@]}"; do
  printf '%s -> %s\n' "$key" "${ports[$key]}"
done
```

**Arithmetic and numeric handling**

Bash can perform integer arithmetic directly. Use arithmetic expansion for calculations and arithmetic evaluation for assignments and comparisons.

```bash
a=5
b=2
echo $((a + b))
((a++))
((total += 10))
```

- `$(( ... ))` expands to a numeric result.
- `(( ... ))` evaluates arithmetic and returns an exit status.
- Bash arithmetic is integer-based; it does not handle floating-point math natively.
- Use `bc` or `awk` for decimal calculations when needed.

**Debugging and validation**

Debugging is part of scripting fundamentals because shell scripts fail in subtle ways. Start with syntax checks, then trace execution, then use external linting.

```bash
bash -n script.sh
bash -x script.sh
```

- `bash -n` checks syntax without running the script.
- `bash -x` traces commands as they execute.
- `set -x` enables tracing inside a script.
- `set +x` disables tracing after the section you want to inspect.

A practical debugging pattern is to combine tracing with safer error handling:

```bash
#!/usr/bin/env bash
set -euo pipefail
set -x

name="${1:-guest}"
echo "Hello, $name"
```

- `-u` treats unset variables as errors.
- `pipefail` makes a pipeline fail if any command in it fails.
- Use `trap` for cleanup or error reporting when scripts manage temporary files.

```bash
tmpfile=$(mktemp)
trap 'rm -f "$tmpfile"' EXIT
```

External validation tools such as `shellcheck` are valuable because they catch quoting mistakes, unused variables, and portability issues.

```bash
shellcheck script.sh
```

- **shellcheck** is a static analysis tool for shell scripts.
- It is especially useful for catching word splitting, globbing, and test syntax problems.
- Use it before deploying scripts or submitting lab work.

**Practical script patterns**

A small but realistic script combines input validation, functions, redirection, and safe quoting.

```bash
#!/usr/bin/env bash
set -euo pipefail

usage() {
  echo "Usage: $0 <file>"
}

main() {
  local file="${1:-}"
  [[ -n "$file" ]] || { usage; exit 1; }
  [[ -f "$file" ]] || { echo "Not a file: $file" >&2; exit 1; }

  grep -n "ERROR" "$file" | tee errors.txt
}

main "$@"
```

- Put reusable logic in functions.
- Validate arguments before using them.
- Send errors to `stderr` with `>&2`.
- Forward all script arguments with `"$@"`.

**Common Bash pitfalls**

Bash is powerful, but small syntax mistakes can change behavior dramatically. Many exam questions focus on these edge cases because they reveal whether you understand shell semantics.

- Forgetting quotes around variables can split words or expand globs.
- Using `=` instead of `-eq` in numeric tests can produce wrong results.
- Expecting variables from a subshell to persist in the parent shell is a common mistake.
- Using `read` without `-r` can mangle backslashes.
- Iterating over `"$@"` incorrectly can collapse arguments with spaces.
- Assuming `export` makes a variable available to the current shell is incorrect; it only affects child processes.

**Exam-focused command reference**

```bash
# Syntax check
bash -n script.sh

# Trace execution
bash -x script.sh

# Lint
shellcheck script.sh

# Make executable
chmod +x script.sh

# Run with Bash explicitly
bash script.sh arg1 arg2
```

**Key takeaways**

Bash scripting fundamentals center on safe text handling, reliable control flow, and understanding how the shell executes commands. The most important habits are quoting variables, checking exit status, using the right test operators, and knowing when scope changes in subshells or functions.

If you can read and write scripts that use variables, conditionals, loops, functions, redirection, and debugging tools, you are covering the core skills expected in Linux administration and exam prep.
