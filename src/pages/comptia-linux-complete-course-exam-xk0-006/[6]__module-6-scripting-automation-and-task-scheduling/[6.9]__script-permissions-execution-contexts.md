**Script permissions and execution contexts**

Linux decides whether a script can run by combining file permissions, the interpreter line, mount options, and the user’s execution context. A script may be readable but not executable, executable but blocked by the filesystem, or runnable only through an interpreter. Understanding these layers helps you troubleshoot “Permission denied,” “bad interpreter,” and “command not found” errors quickly.

**Core permission model**

Linux uses the standard permission triplet for the file owner, group, and others. For scripts, the most important bit is usually the execute permission, but read permission also matters because most interpreters must read the script contents.

- `r` allows the file contents to be read.
- `w` allows the file contents to be modified.
- `x` allows the file to be executed as a program.
- Ownership determines which permission class applies first: user, group, or others.
- ACLs (Access Control Lists) can add more specific rules beyond the basic mode bits.

A common point of confusion is that `chmod +x script.sh` makes the file executable, but it does not guarantee the script will run successfully. The interpreter must still exist, the file must be readable, and the filesystem must allow execution.

```bash
ls -l deploy.sh
chmod 750 deploy.sh
stat deploy.sh
```

Typical mode examples:

- `750` means owner can read, write, and execute; group can read and execute; others have no access.
- `644` means owner can read and write; group and others can read only.
- `700` is useful for private administrative scripts.
- `755` is common for shared utility scripts and binaries.

**Shebang and interpreter selection**

A script usually starts with a shebang line, which tells the kernel which interpreter to use. The shebang must be the first line of the file.

```bash
#!/bin/bash
echo "Hello from Bash"
```

The kernel does not “understand” shell syntax directly. When you run `./script.sh`, the kernel reads the shebang and launches the interpreter specified there. If the shebang is missing, the shell may still run the file if you invoke it explicitly with an interpreter, such as `bash script.sh`.

- `#!/bin/bash` runs the script with Bash.
- `#!/usr/bin/env python3` uses `env` to locate `python3` in the current `PATH`.
- `#!/bin/sh` requests the system shell, which may be `dash`, `bash`, or another POSIX-compatible shell depending on the distribution.
- The interpreter path must be valid and executable.
- A malformed shebang often causes `bad interpreter` or `No such file or directory`.

Use `env` when you want portability across systems where the interpreter may live in different locations.

```bash
#!/usr/bin/env bash
set -euo pipefail
echo "Portable Bash script"
```

**Execution contexts**

Execution context describes *who* runs the script, *what environment* it inherits, and *what privileges* it has. The same script can behave differently depending on whether you run it interactively, from cron, through `sudo`, or as a service.

Important context factors include:

- **Effective user ID (EUID)** determines the privileges the process actually uses.
- **Real user ID (RUID)** identifies the user who launched the process.
- **Group membership** affects access to files and directories.
- **Environment variables** such as `PATH`, `HOME`, `SHELL`, and `LANG` influence behavior.
- **Current working directory** affects relative paths.
- **Umask** controls default permissions for files created by the script.

A script that works in your terminal may fail in automation because the environment is smaller or different. For example, cron jobs often have a minimal `PATH`, so commands that work interactively may fail unless you use absolute paths.

```bash
#!/bin/bash
PATH=/usr/sbin:/usr/bin:/sbin:/bin
/usr/bin/logger "Backup started"
```

**Running a script: direct execution vs interpreter invocation**

There are two common ways to run a script:

1. Direct execution: `./script.sh`
2. Interpreter invocation: `bash script.sh`

Direct execution requires the execute bit and a valid shebang. Interpreter invocation only requires the script to be readable by the interpreter.

```bash
chmod 644 report.sh
bash report.sh
./report.sh   # fails without execute permission
```

This distinction is useful when testing scripts. If a script fails with `./script.sh` but works with `bash script.sh`, the problem is often permissions, shebang, or line endings rather than the script logic itself.

**Common permission and execution problems**

Several issues can prevent a script from running even when the file exists.

- **Permission denied** usually means the execute bit is missing, the directory lacks execute permission, or the filesystem is mounted with `noexec`.
- **bad interpreter** often means the shebang path is wrong or the interpreter is missing.
- **command not found** may indicate a missing `PATH` entry, a typo, or Windows-style line endings.
- **Exec format error** can occur when the file is not a valid executable format and lacks a usable shebang.
- **Text file busy** is uncommon but can appear when a file is being modified while executed on some systems.

> If a script is executable but still fails, check the directory permissions and mount options as well as the file mode. You need execute permission on every directory in the path, not just on the script itself.

Useful troubleshooting commands:

```bash
ls -l /opt/tools/cleanup.sh
namei -l /opt/tools/cleanup.sh
file /opt/tools/cleanup.sh
mount | grep noexec
head -n 1 /opt/tools/cleanup.sh
```

**Directory permissions matter too**

To access and execute a script, you need execute permission on the directories that contain it. Directory execute permission means you can traverse the directory.

- Without `x` on a directory, you cannot access files inside it by path.
- Without `r` on a directory, you cannot list its contents.
- A script in `/opt/private/bin` may be inaccessible even if the file itself is `755` if one parent directory lacks `x`.

Example:

```bash
chmod 750 /opt/private
chmod 750 /opt/private/bin
chmod 755 /opt/private/bin/maint.sh
```

In this case, only the owner and group can traverse the path to the script.

**Mount options and filesystem restrictions**

Some filesystems are mounted with `noexec`, which prevents execution of binaries and scripts from that mount point. This is a common security control for removable media, shared directories, and temporary storage.

```bash
mount | grep /tmp
findmnt -no OPTIONS /tmp
```

If `/tmp` is mounted with `noexec`, you may still be able to read or copy a script there, but you cannot execute it directly from that location.

- `noexec` blocks direct execution from the mount.
- `nosuid` disables setuid and setgid behavior.
- `nodev` prevents device files from being interpreted.

A common workaround is to run the script through an interpreter from a permitted location, but that does not bypass all security controls and may still be restricted by policy.

**Setuid, setgid, and scripts**

Traditional Linux permissions include setuid and setgid bits, but these are generally not reliable for scripts. Most modern Linux systems ignore setuid/setgid on scripts for security reasons because of race conditions between opening and executing the file.

- Setuid on a binary can change the effective user to the file owner.
- Setuid on a script is typically ignored.
- If you need privileged automation, use `sudo`, a service account, or a properly designed privileged helper.

Example of checking special bits:

```bash
ls -l /usr/local/bin/admin-task
chmod u+s /usr/local/bin/admin-task
```

For exam purposes, remember that scripts should not be treated like setuid binaries.

**Environment inheritance and automation**

Scripts inherit environment variables from the parent process unless the launcher sanitizes them. This matters in login shells, `sudo`, cron, systemd, and SSH sessions.

Common environment differences:

- `PATH` may be shorter in non-interactive sessions.
- `HOME` may point to a service account home directory.
- `LANG` and `LC_*` affect sorting, messages, and parsing.
- `SHELL` may not match the interpreter actually used.
- `sudo` may reset or restrict environment variables.

Examples:

```bash
sudo env
sudo -E ./script.sh
env -i /bin/bash ./script.sh
```

Use `env -i` to test how a script behaves with a nearly empty environment. This is a good way to expose hidden dependencies on inherited variables.

**Cron, systemd, and login shells**

Different execution contexts load different startup files and environment settings.

- Interactive login shells may read `/etc/profile` and `~/.bash_profile`.
- Interactive non-login shells may read `~/.bashrc`.
- Cron jobs run with a minimal environment and no terminal.
- systemd services run with a controlled environment defined in the unit file.
- SSH remote commands may not behave like full interactive sessions.

Cron example:

```bash
SHELL=/bin/bash
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
0 2 * * * /usr/local/bin/backup.sh >> /var/log/backup.log 2>&1
```

systemd service example:

```ini
[Unit]
Description=Nightly backup

[Service]
Type=oneshot
ExecStart=/usr/local/bin/backup.sh
User=backup
Group=backup
Environment=PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin

[Install]
WantedBy=multi-user.target
```

These contexts are important because a script may rely on aliases, shell functions, or interactive shell settings that are not available in automation.

**Line endings and file format issues**

Scripts created on Windows systems may contain CRLF (carriage return + line feed) line endings. Linux interpreters can misread the shebang or commands because of the extra carriage return.

Symptoms:

- `bad interpreter: /bin/bash^M: no such file or directory`
- Unexpected command parsing errors
- Shebang appears correct but still fails

Fix with:

```bash
dos2unix script.sh
sed -i 's/\r$//' script.sh
```

Also verify the file type:

```bash
file script.sh
```

**Practical permission workflow**

When preparing a script for execution, follow a simple workflow:

1. Write the script and include a valid shebang.
2. Make the file readable and executable as needed.
3. Verify the parent directories are traversable.
4. Confirm the filesystem does not use `noexec`.
5. Test both direct execution and interpreter invocation.
6. Check logs or error output if the script runs in automation.

Example:

```bash
cat > /usr/local/bin/rotate-logs.sh <<'EOF'
#!/usr/bin/env bash
set -euo pipefail
/usr/bin/find /var/log/myapp -type f -name '*.log' -mtime +7 -delete
EOF

chmod 750 /usr/local/bin/rotate-logs.sh
chown root:adm /usr/local/bin/rotate-logs.sh
/usr/local/bin/rotate-logs.sh
```

**Security best practices**

Script permissions should be as restrictive as practical.

- Use `750` or `700` for administrative scripts.
- Use `755` only when broad execution is appropriate.
- Avoid world-writable script locations.
- Keep scripts in trusted directories such as `/usr/local/bin` or a controlled application path.
- Use absolute paths for critical commands inside scripts.
- Validate input before using it in shell commands.
- Prefer `set -euo pipefail` in Bash scripts to reduce silent failures.

Example defensive pattern:

```bash
#!/usr/bin/env bash
set -euo pipefail

BACKUP_DIR=/var/backups/app
TARGET=/usr/bin/tar

mkdir -p "$BACKUP_DIR"
"$TARGET" -czf "$BACKUP_DIR/app-$(date +%F).tar.gz" /srv/app
```

**Quick comparison of execution methods**

| Method | Requires `x` bit | Requires shebang | Uses interpreter from shebang | Notes |
|---|---:|---:|---:|---|
| `./script.sh` | Yes | Yes | Yes | Most direct method |
| `bash script.sh` | No | No | No | Bash reads the file directly |
| `sh script.sh` | No | No | No | Uses `sh`, not necessarily Bash |
| `python3 script.py` | No | No | No | Interpreter is chosen explicitly |

**Troubleshooting checklist**

When a script fails, check the problem in layers:

- Confirm the file exists and is the expected file type.
- Check permissions with `ls -l` and `stat`.
- Check directory traversal with `namei -l`.
- Inspect the first line for a valid shebang.
- Verify the interpreter path exists.
- Look for CRLF line endings.
- Check mount options for `noexec`.
- Compare interactive and non-interactive environments.
- Review ownership, ACLs, and SELinux or AppArmor restrictions if applicable.

**Key takeaways**

Script execution is not just about `chmod +x`. Linux evaluates file mode bits, directory permissions, interpreter selection, mount options, and the runtime environment. If you understand execution context, you can predict why a script works in one place and fails in another, and you can fix the issue systematically instead of guessing.
