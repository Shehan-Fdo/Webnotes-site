**Environment variables: what they are and why they matter**

Environment variables are name/value pairs that influence how processes and shells behave. They are inherited by child processes, so they are a common way to pass locale settings, search paths, editor preferences, proxy settings, and application-specific options.

A shell variable exists only in the current shell unless you export it. An environment variable is a shell variable marked for inheritance by child processes.

- **Shell variable**: visible only in the current shell session unless exported.
- **Environment variable**: available to the shell and child processes.
- **Inheritance**: child processes receive a copy of the parent environment at launch time.
- **Scope**: changes in a child process do not flow back to the parent shell.

Common examples include `PATH`, `HOME`, `USER`, `SHELL`, `LANG`, `TERM`, and `EDITOR`.

```bash
# Shell-only variable
MYVAR="local only"

# Exported environment variable
export MYVAR="available to child processes"

# Alternative one-line form
MYVAR="available to child processes"
export MYVAR
```

**Inspecting variables in practice**

Use the right command depending on whether you want shell variables, exported variables, or the full environment.

```bash
# Show a shell variable if it exists
echo "$MYVAR"

# Show exported variables
export -p

# Show shell variables and functions in Bash
declare -p MYVAR

# Show the current process environment
env
printenv
```

- `declare -p` is useful in Bash when you want to see whether a variable exists and how it is defined.
- `export -p` shows variables marked for export.
- `env` and `printenv` show environment variables, not shell-only variables.

> A variable can exist in the shell without being exported. If a script cannot see it, check whether it is exported.

**Exporting versus shell-only values**

This distinction matters when you troubleshoot scripts, cron jobs, and service startup. A shell-only variable can affect interactive commands in your current session, but a child process will not inherit it unless you export it.

```bash
MYAPP_MODE=debug
echo "$MYAPP_MODE"

bash -c 'echo "$MYAPP_MODE"'   # empty unless exported

export MYAPP_MODE=debug
bash -c 'echo "$MYAPP_MODE"'   # prints debug
```

**How Bash startup files work**

Bash reads different startup files depending on whether the shell is a login shell, interactive shell, or non-interactive shell. This is a frequent exam topic because the same variable may appear in one session type but not another.

- **Login shell**: typically starts when you log in on a console, via SSH, or with `bash -l`.
- **Interactive shell**: a shell that reads commands from a terminal.
- **Non-interactive shell**: a shell running a script or command string.

Bash startup behavior is roughly:

- Login shell:
  - reads `/etc/profile`
  - then reads the first readable file among `~/.bash_profile`, `~/.bash_login`, or `~/.profile`
- Interactive non-login shell:
  - reads `~/.bashrc`
- Non-interactive shell:
  - does not read the interactive startup files by default
  - may read the file named by `BASH_ENV` if it is set

```bash
# Check whether the current shell is interactive
case $- in
  *i*) echo "interactive" ;;
  *)   echo "non-interactive" ;;
esac

# Check whether Bash is running as a login shell
shopt -q login_shell && echo "login shell" || echo "not a login shell"
```

**BASH_ENV and non-interactive Bash**

`BASH_ENV` is important when scripts unexpectedly pick up environment settings. In non-interactive Bash, if `BASH_ENV` is set, Bash expands that value and sources the referenced file before running the script.

This can be useful for controlled automation, but it can also create confusing behavior if a script inherits an unexpected `BASH_ENV` from the environment.

```bash
# Example: Bash sources this file for non-interactive shells
export BASH_ENV="$HOME/.bash_env"

cat > "$HOME/.bash_env" <<'EOF'
export APP_ENV=testing
alias ll='ls -alF'
EOF

bash -c 'echo "$APP_ENV"'
```

- Use `BASH_ENV` carefully in automation.
- Prefer explicit script initialization when you need predictable behavior.
- Remember that aliases usually do not help in non-interactive scripts unless alias expansion is enabled and the shell reads the alias definition.

**Files Bash reads versus other shells**

Shell startup files are not universal. Exam questions often test whether you know which files belong to Bash and which are POSIX-oriented or shell-specific.

- `/etc/profile`
  - system-wide login shell initialization for POSIX-style shells and Bash login shells
- `~/.profile`
  - user login initialization file, commonly used by POSIX shells
  - often used when a user wants settings shared across multiple shells
- `~/.bash_profile`
  - Bash login shell file
  - often sources `~/.bashrc` to reuse interactive settings
- `~/.bashrc`
  - Bash interactive non-login shell file
- `~/.bash_login`
  - Bash login shell fallback if `~/.bash_profile` is absent
- `~/.zshrc`, `~/.zprofile`, `~/.config/fish/config.fish`
  - shell-specific files for other shells

> `~/.profile` is generally POSIX-shell oriented, while `~/.bashrc` and `~/.bash_profile` are Bash-specific. Do not assume every shell reads the same files.

A common Bash pattern is to source `~/.bashrc` from `~/.bash_profile` so login shells get the same interactive settings.

```bash
# ~/.bash_profile
if [ -f ~/.bashrc ]; then
  . ~/.bashrc
fi
```

**System-wide environment configuration**

System-wide settings are usually split between login shell initialization and environment files.

- `/etc/profile`
  - sets system-wide login shell environment
  - often sources scripts in `/etc/profile.d/`
- `/etc/profile.d/*.sh`
  - drop-in scripts for package or administrator customizations
- `/etc/environment`
  - defines system-wide environment variables in a simple `KEY=value` format
  - is not a shell script
  - is commonly processed by PAM (Pluggable Authentication Modules) or login managers, not by Bash itself

`/etc/environment` is useful for variables that should exist broadly for users and graphical sessions, such as locale or proxy-related values, but it does not support shell syntax like `export`, command substitution, or conditionals.

```bash
# /etc/environment
LANG=en_US.UTF-8
EDITOR=vim
HTTP_PROXY=http://proxy.example.com:8080
HTTPS_PROXY=http://proxy.example.com:8080
```

- Use `/etc/profile` or `/etc/profile.d/` for shell logic.
- Use `/etc/environment` for simple system-wide variable assignment.
- Do not place shell commands in `/etc/environment`.

**Typical login flow**

When you log in, the environment is assembled from system and user files. The exact path varies by distribution and login method, but the general idea is consistent.

1. The login program or display manager sets a base environment.
2. PAM and related login components may read `/etc/environment`.
3. A login shell reads `/etc/profile`.
4. The shell reads one user login file such as `~/.bash_profile` or `~/.profile`.
5. Interactive shells may later read `~/.bashrc`.

This explains why a variable may appear in SSH sessions but not in a terminal tab, or vice versa.

**Practical troubleshooting workflow**

When a variable is missing, verify each layer in order.

1. Check whether the variable exists in the current shell.
2. Check whether it is exported.
3. Check which shell you are using.
4. Check whether the session is login, interactive, or non-interactive.
5. Check the relevant startup files for that shell.
6. Check whether `BASH_ENV` or another shell-specific mechanism is altering startup behavior.

```bash
# Current shell and session clues
echo "$SHELL"
echo "$0"
ps -p $$ -o args=

# Variable visibility
declare -p MYVAR 2>/dev/null
export -p | grep -E 'MYVAR|PATH'

# Environment seen by a child process
bash -c 'env | grep -E "MYVAR|PATH"'
```

**PATH management**

`PATH` is one of the most important environment variables because it controls where the shell looks for executables. A bad `PATH` can make commands appear missing even when they are installed.

- Put custom directories before system directories only when you intentionally want them to take precedence.
- Avoid overwriting `PATH` without preserving existing entries.
- Use `export PATH=...` so child processes inherit it.

```bash
# Safe pattern: prepend a directory
export PATH="$HOME/bin:$PATH"

# Add a directory only if it is not already present
case ":$PATH:" in
  *":$HOME/bin:"*) ;;
  *) export PATH="$HOME/bin:$PATH" ;;
esac
```

**Common configuration patterns**

A clean Bash setup often separates interactive settings from login settings.

```bash
# ~/.bash_profile
# Login-shell settings
export PATH="$HOME/bin:$PATH"

if [ -f ~/.bashrc ]; then
  . ~/.bashrc
fi
```

```bash
# ~/.bashrc
# Interactive-shell settings
alias ll='ls -alF'
alias gs='git status'
export EDITOR=vim
```

- Put aliases and prompt customization in `~/.bashrc`.
- Put login-only environment setup in `~/.bash_profile` or `~/.profile`.
- Keep shared environment values in a file that is sourced consistently.

**POSIX shell considerations**

If you need portability across shells, prefer POSIX-compliant syntax and `~/.profile`-style initialization. Not every shell reads Bash files, and not every shell supports Bash-only features such as arrays, `shopt`, or `declare` in the same way.

- Use `.` instead of `source` for POSIX portability.
- Use simple `VAR=value` assignments and `export VAR`.
- Avoid Bash-only syntax in files intended for `/bin/sh` or other POSIX shells.

```sh
# POSIX-compatible example
PATH="$HOME/bin:$PATH"
export PATH

if [ -f "$HOME/.profile.local" ]; then
  . "$HOME/.profile.local"
fi
```

**Troubleshooting examples**

If a script sees a variable in your terminal but not when run by cron, systemd, or `bash -c`, the issue is usually startup context, not the variable itself.

- Cron jobs often run with a minimal environment.
- Non-interactive shells do not read `~/.bashrc` by default.
- `BASH_ENV` can unexpectedly inject settings into scripts.
- GUI sessions may use `/etc/environment` and display-manager-specific startup files instead of shell login files.

```bash
# Compare interactive and non-interactive behavior
echo "interactive: $MYVAR"
bash -c 'echo "non-interactive: $MYVAR"'
```

**Key exam takeaways**

- **Export** variables when child processes must inherit them.
- **Shell-only** variables stay in the current shell unless exported.
- **Bash login shells** read `/etc/profile` and a user login file.
- **Bash interactive non-login shells** read `~/.bashrc`.
- **Non-interactive Bash** may read `BASH_ENV`.
- **`~/.profile`** is generally POSIX-oriented, while Bash-specific files vary by shell.
- **`/etc/environment`** is for simple system-wide variables, not shell commands.
- **Troubleshooting** starts by checking shell type, export status, and startup file order.
