## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Overview**

`at` schedules a command or script to run once at a specific time in the future. Use it when you need a delayed action that does not repeat, such as sending a reminder, restarting a service later, or running a maintenance command after hours.

Unlike `cron`, which is designed for recurring schedules, `at` is for a single execution. The job is stored by the `atd` service and executed later by a background daemon. Because the job runs in a non-interactive shell, you should write commands carefully and avoid assuming your full login session is available.

- **Use case**: one-time reminders, delayed maintenance, deferred shutdowns, or post-deployment checks.
- **Execution model**: the command is queued now and executed later by `atd`.
- **Shell context**: the job runs non-interactively with a limited environment.

**Core concepts**

`at` reads commands from standard input, from a file, or from a shell pipeline. You can type commands interactively, end input with <kbd>Ctrl</kbd> + <kbd>D</kbd>, and the job is queued immediately.

Common time expressions include:

- `now + 10 minutes`
- `5pm`
- `tomorrow`
- `noon`
- `midnight`
- `10:30 2026-05-01`

The exact accepted formats can vary slightly by implementation, but the general idea is the same: you provide a one-time execution time in natural language or a timestamp-like format.

**Basic usage**

A simple interactive example:

```bash
at now + 10 minutes
at> echo "Reminder: meeting starts soon" | wall
at> <Ctrl+D>
```

A non-interactive example using a pipeline:

```bash
echo 'wall "Reminder: meeting starts soon"' | at now + 10 minutes
```

A script file example:

```bash
at -f /home/alice/cleanup.sh now + 1 hour
```

`-f` tells `at` to read commands from a file instead of standard input. In GNU `at`, it is an option you can combine with a time expression; placement can vary by implementation, so focus on its purpose rather than memorizing a single rigid syntax.

**How jobs are stored and executed**

When you submit a job, `at` assigns it a job number and places it in a queue. The daemon later runs the job at the scheduled time.

Important behavior:

- The job runs in a shell, usually `/bin/sh` unless configured otherwise.
- The environment is limited and not identical to your interactive login shell.
- Output is typically mailed to the submitting user unless redirected.
- The working directory is usually the directory where you submitted the job, but you should not rely on that for critical tasks.

> Do not assume the job inherits your full interactive environment. If a command depends on variables, paths, or files, define them explicitly in the job or use absolute paths.

**Practical examples**

Use absolute paths and explicit environment setup when the job depends on them:

```bash
cat <<'EOF' | at now + 15 minutes
PATH=/usr/sbin:/usr/bin:/sbin:/bin
/usr/bin/systemctl restart nginx
EOF
```

Run a backup script later:

```bash
echo '/usr/local/bin/backup-home.sh' | at 23:00
```

Send a message to logged-in users:

```bash
echo 'wall "System maintenance begins in 10 minutes"' | at now + 10 minutes
```

Schedule a cleanup command:

```bash
echo 'find /tmp -type f -mtime +7 -delete' | at 02:30 tomorrow
```

**Managing queued jobs**

Use `atq` to list jobs you have queued:

```bash
atq
```

Typical output includes the job number, scheduled time, queue, and user.

Remove a queued job with `atrm`:

```bash
atrm 12
```

You can also use `at -r` as an alternative removal form on some systems, but `atrm` is the standard tool to remember for exam purposes.

**Viewing a job before it runs**

Use `at -c` to display the shell script that `at` generated for a queued job:

```bash
at -c 12
```

This is useful for troubleshooting because it shows the environment setup and the exact commands that will run.

**Access control**

`at` access is controlled by allow/deny files, typically in `/etc`:

- `/etc/at.allow`
- `/etc/at.deny`

General behavior:

- If `/etc/at.allow` exists, only users listed there may use `at`.
- If `/etc/at.allow` does not exist and `/etc/at.deny` exists, users listed in `at.deny` are blocked and others may be allowed.
- If neither file exists, behavior depends on the implementation and distribution; many systems allow access by default, but you should verify the local policy.
- `root` can usually submit jobs regardless of user restrictions.

This is a common exam topic because it tests both command knowledge and Linux security policy.

**Comparison with cron**

| Feature | `at` | `cron` |
|---|---|---|
| Schedule type | One-time | Repeating |
| Typical use | Run once later | Run on a schedule |
| Job storage | Queue managed by `atd` | Crontab entries managed by `cron`/`crond` family |
| Best for | Delayed actions | Recurring maintenance |

Use `at` when you need a single delayed action. Use `cron` when the task must repeat on a schedule.

**Daemon and service notes**

The background service is commonly `atd` on Linux distributions that use the GNU `at` package. The exact service management command depends on the init system:

```bash
systemctl status atd
systemctl enable --now atd
```

If the service is not running, queued jobs will not execute on time.

**Troubleshooting**

Common issues include:

- The command works manually but fails in `at` because the job lacks your interactive shell setup.
- Relative paths fail because the job runs from a different context than expected.
- Output is mailed instead of displayed, so you may miss errors unless you check mail.
- The `atd` service is stopped or disabled.
- Access is blocked by `/etc/at.allow` or `/etc/at.deny`.

Useful checks:

```bash
systemctl status atd
atq
mail
```

If a job fails, inspect the generated script and make the command more explicit:

```bash
at -c 12
```

**Exam reminders**

- `at` is for **one-time** jobs, not recurring schedules.
- `atq` lists queued jobs.
- `atrm` removes queued jobs.
- `at -c` shows the queued job script.
- `atd` must be running for jobs to execute.
- Use absolute paths and explicit environment settings for reliability.
- Access control may be governed by `/etc/at.allow` and `/etc/at.deny`.

**Quick reference**

```bash
# Schedule a job
echo 'echo "Hello from at" | wall' | at now + 5 minutes

# List jobs
atq

# Remove job 7
atrm 7

# Show job 7 contents
at -c 7

# Check daemon
systemctl status atd
```
