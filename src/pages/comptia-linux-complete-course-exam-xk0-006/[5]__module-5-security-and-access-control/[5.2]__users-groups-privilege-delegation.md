## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Users, groups, and privilege delegation**

Linux access control starts with identities. You manage users, groups, and delegated privileges so people can do their jobs without giving everyone full administrative access. On the exam, expect questions about account files, password and lock states, sudo behavior, PAM (Pluggable Authentication Modules), ACLs (Access Control Lists), and special permission bits.

A useful mental model is:

- **Users** identify who is acting.
- **Groups** collect users for shared access.
- **Privilege delegation** lets you grant limited administrative power without sharing the root password.

**Identity and the root account**

Every process and file operation is evaluated against a user identity and one or more group identities. The most important identity is the superuser, `root`, which has UID (User ID) `0`.

- **UID 0** bypasses normal discretionary access control checks.
- `root` can read, write, and execute almost anything unless another control layer blocks it, such as SELinux, AppArmor, or filesystem restrictions.
- Delegated admin access is safer than sharing `root` because it preserves accountability and limits blast radius.

The `root` account is not the same as “an admin user.” A delegated admin typically uses a normal account and elevates only for specific commands. That distinction matters on the exam.

Common ways to become `root` or run privileged commands:

```bash
su -
sudo -i
sudo command
runuser -u root -- command
pkexec command
```

- `su` switches to another user, often `root`, and usually requires that user’s password.
- `sudo` runs a command as another user, usually `root`, using the caller’s password and policy rules.
- `runuser` is typically used by `root` or scripts to switch users without a login shell.
- `pkexec` uses PolicyKit for desktop or service-oriented privilege escalation on systems that support it.

> `su`, `runuser`, and `pkexec` are not interchangeable with `sudo`; they solve similar problems with different authentication and policy models.

**User account files and what they store**

Linux traditionally stores account data in text files under `/etc`, though many environments also use centralized identity services such as LDAP or SSSD.

Key files:

- `/etc/passwd` stores account metadata such as username, UID, GID (Group ID), home directory, and login shell.
- `/etc/shadow` stores password hashes and password aging data.
- `/etc/group` stores group names, GIDs, and supplementary membership.
- `/etc/gshadow` stores protected group password and administrative group data.

Example entries:

```text
alice:x:1001:1001:Alice Example:/home/alice:/bin/bash
alice:$y$j9T$...:19700:0:99999:7:::
developers:x:2001:alice,bob
developers:!::
```

Important points:

- `/etc/passwd` is world-readable on many systems because it contains account metadata, not password hashes.
- `/etc/shadow` is restricted because it contains sensitive authentication data.
- `/etc/gshadow` exists for group password protection and group administrators, but group passwords are rarely used in modern Linux environments.
- Actual group membership is usually managed by editing `/etc/group`, `/etc/gshadow`, or using tools such as `usermod` and `gpasswd`.

**Creating, modifying, and removing users**

User management commands are common exam material. You should know what each command changes and what it does not change.

Create a user:

```bash
sudo useradd -m -s /bin/bash alice
sudo passwd alice
```

- `useradd` creates the account.
- `-m` creates a home directory.
- `-s` sets the login shell.
- `passwd` sets the password.

Modify a user:

```bash
sudo usermod -aG developers alice
sudo usermod -d /home/alice2 -m alice
sudo usermod -L alice
sudo usermod -U alice
```

- `-aG` appends supplementary groups without removing existing ones.
- `-d` changes the home directory path.
- `-m` moves the home directory contents when used with `-d`.
- `-L` locks the password.
- `-U` unlocks the password.

Be careful with `usermod -G`:

- Without `-a`, `-G` replaces the user’s supplementary groups.
- This is a common exam trap because it can silently remove access.

Remove a user:

```bash
sudo userdel alice
sudo userdel -r alice
```

- `userdel` removes the account entry.
- `userdel -r` also removes the home directory and mail spool if configured to do so.
- It does **not** remove every file owned by that user elsewhere on the system.

If you need to find leftover files owned by a deleted user:

```bash
sudo find / -xdev -uid 1001
```

**Password aging, lock states, and expiration**

Password and account aging are controlled through shadow data and related tools.

Inspect aging information:

```bash
sudo chage -l alice
```

Set expiration and aging:

```bash
sudo chage -M 90 -m 7 -W 14 alice
sudo chage -E 2026-12-31 alice
```

- `-M` sets maximum password age.
- `-m` sets minimum password age.
- `-W` sets warning days before expiration.
- `-E` sets account expiration date.

Locking and unlocking are subtle:

```bash
sudo passwd -l alice
sudo passwd -u alice
```

- `passwd -l` locks the password by changing the shadow entry.
- `passwd -u` attempts to unlock it.
- Unlocking does not guarantee the account is usable if other restrictions remain, such as account expiration, shell restrictions, or policy rules.
- Some systems refuse to unlock if the result would be a passwordless account.

Check the current state with:

```bash
sudo passwd -S alice
sudo chage -l alice
```

**Groups and group administration**

Groups simplify access management by letting you grant permissions to a team rather than to each user individually.

Types of groups:

- **Primary group**: the default group associated with a user account.
- **Supplementary group**: additional groups that extend access.

View group membership:

```bash
id alice
groups alice
getent group developers
```

Create and manage groups:

```bash
sudo groupadd developers
sudo groupmod -n devteam developers
sudo groupdel developers
```

Add users to groups:

```bash
sudo usermod -aG developers alice
sudo gpasswd -a bob developers
sudo gpasswd -d bob developers
```

Group password and administration details:

- Group passwords are uncommon in modern Linux.
- `gpasswd` can manage group membership and group administrators.
- `/etc/gshadow` stores protected group data, including group admins and password placeholders.
- In practice, group membership is usually controlled by an administrator rather than by users joining groups with a shared password.

**Privilege delegation with sudo**

`sudo` is the standard tool for controlled privilege escalation. It lets you define who can run what, as whom, and from where.

Basic usage:

```bash
sudo command
sudo -u postgres psql
sudo -i
sudo -l
```

- `sudo command` runs a single command with elevated privileges if policy allows it.
- `-u` runs as a different target user.
- `-i` starts a login shell as the target user.
- `-l` lists allowed commands.

The policy file is usually `/etc/sudoers`, but you should edit it with `visudo`:

```bash
sudo visudo
```

Why `visudo` matters:

- It validates syntax before saving.
- It reduces the risk of locking yourself out with a broken sudoers file.

A simple sudoers rule:

```sudoers
alice ALL=(root) /usr/bin/systemctl restart nginx
```

This means:

- `alice` may run the exact command `/usr/bin/systemctl restart nginx`
- from any host listed as `ALL`
- as the target user `root`

Important sudoers concepts:

- Command matching is strict.
- Arguments are part of the command specification.
- `alice ALL=(root) /usr/bin/systemctl restart nginx` does not automatically allow other `systemctl` actions.
- If you want broader access, you must define it explicitly, often with command aliases.

Example with aliases:

```sudoers
User_Alias ADMINS = alice,bob
Runas_Alias DBUSERS = postgres,mysql
Cmnd_Alias WEBCMDS = /usr/bin/systemctl restart nginx, /usr/bin/systemctl reload nginx

ADMINS ALL=(DBUSERS) WEBCMDS
```

This grants members of `ADMINS` permission to run the listed web commands as either `postgres` or `mysql` if the policy permits that exact run-as set.

Useful sudoers options:

```sudoers
Defaults logfile="/var/log/sudo.log"
Defaults timestamp_timeout=5
Defaults use_pty
```

- `logfile` records sudo activity.
- `timestamp_timeout` controls how long authentication is cached.
- `use_pty` improves logging and reduces some command injection risks.

Troubleshooting sudo:

- Use `sudo -l` to see what the policy allows.
- Check `/var/log/auth.log` or `/var/log/secure` depending on distribution.
- Validate syntax with `visudo -c`.
- Remember that `sudo` may be restricted by host, user, run-as user, command path, and command arguments.

**Common privilege-delegation alternatives**

Linux+ often expects you to recognize tools that overlap with `sudo`.

- `su` switches to another user and usually requires the target user’s password.
- `runuser` is similar to `su` but is commonly used by `root` or scripts and does not prompt like a normal user switch.
- `pkexec` uses PolicyKit rules and is common in desktop or service contexts.
- `sudo` is the most common command-line delegation tool on servers.

Practical differences:

- Use `sudo` when you want fine-grained command control and auditability.
- Use `su` when you need a full shell as another user and have that user’s credentials.
- Use `runuser` in scripts or administrative contexts where `root` is already trusted.
- Use `pkexec` when a graphical or policy-managed privilege prompt is appropriate.

**PAM and authentication policy**

PAM (Pluggable Authentication Modules) is the framework that lets Linux services share authentication, account, session, and password policy logic.

PAM affects:

- `login`
- `sshd`
- `sudo`
- display managers
- `su`
- many other authentication-aware services

A typical PAM stack has four module types:

- `auth` verifies identity.
- `account` checks account validity and restrictions.
- `password` changes credentials.
- `session` sets up and tears down the user session.

Example PAM snippets vary by distribution, but common modules include:

```text
auth       required     pam_faillock.so preauth
auth       sufficient   pam_unix.so
account    required     pam_unix.so
password   requisite    pam_pwquality.so retry=3
session    required     pam_limits.so
```

Use these modules carefully:

- `pam_unix.so` handles traditional local authentication.
- `pam_pwquality.so` enforces password complexity rules in the password stack.
- `pam_faillock.so` can lock accounts after repeated failures.
- `pam_limits.so` applies resource limits from `/etc/security/limits.conf`.

> PAM configuration is distribution-specific. The exact file layout and module order can differ between Debian-based and Red Hat-based systems.

Practical examples:

- If SSH logins are failing after too many bad passwords, check `pam_faillock` or equivalent lockout policy.
- If a user cannot open enough files or processes, check `pam_limits` and `/etc/security/limits.conf`.
- If password changes are rejected, inspect the password stack and complexity policy.

Example limits configuration:

```conf
alice hard nofile 4096
@developers soft nproc 2048
```

**ACLs for finer-grained access**

ACLs let you grant permissions to specific users or groups without changing the owning user or primary group.

Check ACLs:

```bash
getfacl /srv/projects/app1
```

Set an ACL:

```bash
sudo setfacl -m u:alice:rwx /srv/projects/app1
sudo setfacl -m g:developers:r-x /srv/projects/app1
```

Default ACLs on directories apply to newly created files and subdirectories:

```bash
sudo setfacl -d -m g:developers:rwx /srv/projects/app1
```

Important caveats:

- Default ACLs affect new objects only.
- Existing permissions on the directory still matter.
- The process `umask` can reduce the permissions that new files receive.
- To make inherited access useful, you often need both a sensible directory mode and a default ACL.

A practical setup might look like this:

```bash
sudo chgrp developers /srv/projects/app1
sudo chmod 2775 /srv/projects/app1
sudo setfacl -d -m g:developers:rwx /srv/projects/app1
```

Why this works better:

- `chmod 2775` sets the setgid bit on the directory so new files inherit the directory’s group.
- The default ACL grants the group explicit permissions on new content.
- Together, they create predictable collaboration behavior.

Remove ACLs:

```bash
sudo setfacl -b /srv/projects/app1
sudo setfacl -k /srv/projects/app1
```

- `-b` removes all ACL entries.
- `-k` removes default ACLs from a directory.

**Special permission bits: setuid, setgid, and sticky bit**

Special bits are part of privilege delegation because they change how execution and file creation behave.

- **setuid** on an executable runs the program with the file owner’s effective UID.
- **setgid** on an executable runs the program with the file’s group privileges.
- **setgid** on a directory causes new files and subdirectories to inherit the directory’s group.
- **sticky bit** on a directory prevents users from deleting files they do not own, even if they have write access to the directory.

View special bits:

```bash
ls -l /usr/bin/passwd
ls -ld /tmp
```

Typical examples:

- `/usr/bin/passwd` often has setuid so users can change their own passwords through a controlled privileged helper.
- `/tmp` often has the sticky bit so users cannot delete each other’s files.

Set special bits:

```bash
sudo chmod u+s /path/to/program
sudo chmod g+s /srv/projects/app1
sudo chmod +t /shared/tmp
```

Numeric examples:

- `4755` sets setuid.
- `2755` sets setgid.
- `1777` sets sticky bit.

**Troubleshooting and exam-style edge cases**

Account and privilege problems often come from policy, not just file permissions.

Common issues to check:

- The user is not in the expected supplementary group.
- `usermod -aG` was forgotten, so existing groups were replaced.
- The sudoers rule matches the wrong command path.
- The command arguments do not match exactly.
- The account is locked, expired, or has a non-login shell.
- PAM is denying access due to lockout, password policy, or limits.
- ACLs exist, but the directory mode or `umask` prevents useful access.
- The setgid bit is missing on a shared directory, so new files get the wrong group.

Useful diagnostics:

```bash
id alice
sudo -l -U alice
sudo visudo -c
getfacl /srv/projects/app1
chage -l alice
passwd -S alice
```

A few practical scenarios:

1. If a user can log in but cannot use `sudo`, check group membership, sudoers rules, and whether the account is listed in the correct admin group for that distribution.
2. If a shared project directory is not inheriting group ownership, check the directory’s setgid bit and the process `umask`.
3. If a password change fails, inspect PAM password policy and account aging.
4. If a locked account still appears to authenticate in some contexts, verify whether the password is locked, the account is expired, or a service-specific policy is overriding the expected behavior.

**What to remember for the exam**

- `root` has UID `0` and full privilege, but delegated admin is safer and more auditable.
- `sudo` is policy-based and command-specific; command paths and arguments matter.
- `su`, `runuser`, and `pkexec` are related tools with different use cases.
- Groups are the main way to scale access control, but `usermod -aG` is safer than `-G` alone.
- `/etc/passwd`, `/etc/shadow`, `/etc/group`, and `/etc/gshadow` each serve distinct roles.
- PAM controls authentication, account checks, password changes, and session setup.
- ACLs and special bits solve access problems that basic mode bits cannot.
- Default ACLs and setgid help with collaboration, but directory mode and `umask` still matter.
