## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Security mindset in Linux**

Linux security starts with reducing trust, limiting exposure, and assuming that any service, user, or package can become a risk. You protect systems by combining preventive controls, detection, and recovery planning.

The most important principles are:

- **Least privilege**: give users, services, and processes only the access they need.
- **Defense in depth**: layer controls so one failure does not expose the system.
- **Attack surface reduction**: remove unused packages, services, ports, and accounts.
- **Separation of duties**: split administrative responsibilities where possible.
- **Secure defaults**: prefer restrictive permissions and explicit allow rules.
- **Auditability**: log actions so you can investigate incidents and prove what happened.

A Linux administrator should think in terms of risk, not just configuration. A system can be technically functional and still be insecure if it exposes unnecessary services, uses weak authentication, or allows broad file permissions.

**Threat model and common attack paths**

Threat awareness means understanding how attackers typically gain access and what they do after entry. On Linux, common entry points include exposed network services, weak credentials, vulnerable web applications, misconfigured SSH, and supply-chain issues from untrusted packages or scripts.

Typical attacker goals include:

- **Privilege escalation** to gain root access.
- **Persistence** through cron jobs, systemd units, shell profiles, or backdoors.
- **Credential theft** from config files, SSH keys, browser caches, or memory.
- **Lateral movement** to other hosts using reused credentials or trust relationships.
- **Data exfiltration** through encrypted tunnels, DNS, HTTP, or cloud sync tools.
- **Denial of service** by exhausting CPU, memory, disk, or network resources.

Common threat categories to recognize:

- **Malware**: trojans, worms, ransomware, rootkits, and spyware.
- **Phishing and social engineering**: tricking users into revealing credentials or running malicious commands.
- **Misconfiguration**: open permissions, exposed admin interfaces, default passwords, or overly permissive firewall rules.
- **Vulnerability exploitation**: abusing unpatched software or kernel flaws.
- **Insider threats**: accidental or intentional misuse by authorized users.
- **Supply-chain attacks**: compromised repositories, packages, scripts, containers, or CI/CD pipelines.

> A Linux host is often compromised through a small mistake: a leaked private key, a writable script run by root, or a service exposed to the internet without hardening.

**Core Linux security controls**

Linux security is built from file permissions, authentication, authorization, process isolation, and network filtering. You should understand how these controls work together.

- **Users and groups**: separate identities and assign access by role.
- **File permissions**: control read, write, and execute access with owner/group/other bits.
- **Special permissions**: `setuid`, `setgid`, and the sticky bit change execution or deletion behavior.
- **ACLs (Access Control Lists)**: provide finer-grained permissions than standard mode bits.
- **Capabilities**: split root privileges into smaller units such as `CAP_NET_ADMIN`.
- **MAC (Mandatory Access Control)**: enforce policy with SELinux or AppArmor.
- **Firewalling**: restrict inbound and outbound traffic with `nftables`, `iptables`, or frontends such as `firewalld`.
- **Logging and auditing**: record authentication, authorization, and system events.
- **Encryption**: protect data at rest and in transit.

A secure design uses multiple controls together. For example, a web server may run as a non-root user, listen only on required ports, store secrets outside the web root, and be confined by SELinux or AppArmor.

**Authentication and account security**

Authentication verifies identity, while authorization determines what an identity can do. Strong authentication reduces the chance that stolen credentials become a full compromise.

Key practices:

- Use **strong passwords** where passwords are allowed.
- Prefer **SSH keys** over passwords for remote administration.
- Protect private keys with passphrases and correct file permissions.
- Disable direct root login over SSH when possible.
- Use **MFA (multi-factor authentication)** for sensitive access if supported.
- Lock or remove unused accounts.
- Review `/etc/passwd`, `/etc/shadow`, and `/etc/group` for unexpected entries.

Useful commands:

```bash
# Inspect local accounts and groups
getent passwd
getent group

# Check password aging and account status
chage -l alice
passwd -S alice

# Lock or unlock an account
sudo passwd -l alice
sudo passwd -u alice
```

SSH hardening example:

```bash
# /etc/ssh/sshd_config
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
AllowUsers admin1 admin2
MaxAuthTries 3
LoginGraceTime 20
```

After changing SSH configuration, validate and reload safely:

```bash
sudo sshd -t
sudo systemctl reload sshd
```

> If you disable password authentication, make sure key-based login works before closing your current session.

**File permissions, ownership, and special bits**

Permissions are one of the most important Linux security mechanisms. They determine who can access files and directories.

Basic model:

- **Owner**: the user who owns the file.
- **Group**: users in the file’s group.
- **Other**: everyone else.

Common commands:

```bash
ls -l /etc/shadow
stat /etc/passwd
chmod 640 report.txt
chown root:syslog /var/log/custom.log
```

Special permission bits:

- **setuid**: executable runs with the file owner’s privileges.
- **setgid**: executable runs with the file group’s privileges; on directories, new files inherit the directory group.
- **Sticky bit**: on shared directories, only the file owner or root can delete files.

Examples:

```bash
# Set setgid on a shared directory
sudo chmod 2775 /srv/shared

# Set sticky bit on a public temp directory
sudo chmod 1777 /tmp

# Find files with special permissions
find / -perm /4000 -o -perm /2000 2>/dev/null
```

Security implications:

- `setuid` binaries can be dangerous if vulnerable, because they may grant elevated privileges.
- World-writable directories without the sticky bit can allow users to delete each other’s files.
- Overly broad permissions on scripts, keys, or config files can expose secrets or enable tampering.

**ACLs and capabilities**

ACLs let you grant access to specific users or groups without changing the main mode bits. They are useful when standard owner/group/other permissions are too coarse.

Example:

```bash
# Give user bob read access to a file
sudo setfacl -m u:bob:r-- project.txt

# View ACLs
getfacl project.txt

# Remove ACL entry
sudo setfacl -x u:bob project.txt
```

Capabilities reduce the need for full root privileges. Instead of giving a process all-powerful access, you grant only the capability it needs.

Examples:

- `CAP_NET_BIND_SERVICE`: bind to ports below 1024.
- `CAP_NET_ADMIN`: manage network settings.
- `CAP_SYS_ADMIN`: very broad and often considered highly sensitive.

You can inspect capabilities with:

```bash
getcap /usr/bin/ping
sudo setcap cap_net_bind_service=+ep /usr/local/bin/myserver
```

Use capabilities carefully. They are safer than setuid in many cases, but they still expand privilege and should be limited to trusted binaries.

**Mandatory Access Control: SELinux and AppArmor**

Mandatory Access Control adds policy enforcement beyond traditional Unix permissions. Even if a process has file permissions, MAC can still deny access.

Two common systems:

- **SELinux (Security-Enhanced Linux)**: label-based policy, common on Red Hat family systems.
- **AppArmor**: path-based profiles, common on Ubuntu and some other distributions.

SELinux concepts:

- **Context/label**: security label attached to files, processes, and ports.
- **Enforcing mode**: policy is active and blocks violations.
- **Permissive mode**: policy logs violations but does not block them.
- **Disabled**: policy is off.

Useful commands:

```bash
getenforce
sestatus
ls -Z /var/www/html
ps -eZ | head
```

Common SELinux troubleshooting workflow:

1. Check whether SELinux is enforcing.
2. Inspect labels on the file, directory, or port.
3. Review audit logs for denials.
4. Fix the label or policy instead of disabling SELinux.

Example label repair:

```bash
sudo restorecon -Rv /var/www/html
sudo semanage fcontext -a -t httpd_sys_content_t '/srv/web(/.*)?'
sudo restorecon -Rv /srv/web
```

AppArmor concepts:

- **Profile**: rule set for a program.
- **Complain mode**: logs violations without blocking.
- **Enforce mode**: blocks violations.

Useful commands:

```bash
sudo aa-status
sudo aa-enforce /etc/apparmor.d/usr.sbin.nginx
sudo aa-complain /etc/apparmor.d/usr.sbin.nginx
```

MAC is especially valuable for services exposed to untrusted input, such as web servers, mail servers, and file-sharing daemons.

**Network security and service exposure**

Network security on Linux is about controlling what listens, what connects out, and what traffic is allowed between hosts.

Start by identifying listening services:

```bash
ss -tulpn
sudo lsof -i -P -n | grep LISTEN
```

Reduce exposure by disabling unused services:

```bash
systemctl list-unit-files --type=service
sudo systemctl disable --now telnet.socket
sudo systemctl disable --now cups
```

Firewall example with `nftables`:

```bash
sudo nft add table inet filter
sudo nft add chain inet filter input '{ type filter hook input priority 0; policy drop; }'
sudo nft add rule inet filter input ct state established,related accept
sudo nft add rule inet filter input iif lo accept
sudo nft add rule inet filter input tcp dport 22 accept
```

Firewall example with `firewalld`:

```bash
sudo firewall-cmd --permanent --add-service=ssh
sudo firewall-cmd --permanent --remove-service=cockpit
sudo firewall-cmd --reload
```

Security tips:

- Allow only required ports and protocols.
- Restrict management interfaces to trusted networks.
- Use VPNs or jump hosts for administrative access when appropriate.
- Prefer encrypted protocols such as SSH, HTTPS, and SFTP over plaintext alternatives.
- Be cautious with port forwarding and NAT rules that expose internal services.

**Patch management and software trust**

Keeping software updated is one of the most effective defenses against exploitation. Many Linux compromises happen because a known vulnerability remains unpatched.

Good patching practices:

- Apply security updates regularly.
- Track kernel, library, and application updates.
- Reboot when kernel updates require it.
- Test updates in staging when uptime is critical.
- Remove obsolete packages and repositories.

Package management examples:

```bash
# Debian/Ubuntu
sudo apt update
apt list --upgradable
sudo apt upgrade

# RHEL/Fedora
sudo dnf check-update
sudo dnf upgrade
```

Trust considerations:

- Install packages from trusted repositories.
- Verify checksums or signatures for downloaded software.
- Avoid running unknown scripts as root.
- Review container images and base layers before deployment.

Supply-chain risk is especially important in automation. A script fetched from the internet and piped into a shell can execute arbitrary code immediately.

> Avoid patterns like `curl ... | bash` unless you fully trust the source and have verified the content.

**Logging, auditing, and detection**

Detection helps you notice compromise early and reconstruct events after an incident. Linux provides logs from the kernel, services, authentication stack, and security tools.

Common log locations:

- `/var/log/messages` or `/var/log/syslog`
- `/var/log/auth.log`
- `journalctl` for systemd-based logging
- `/var/log/audit/audit.log` for auditd

Useful commands:

```bash
journalctl -xe
journalctl -u sshd
sudo tail -f /var/log/auth.log
sudo ausearch -m AVC,USER_AUTH,USER_ACCT
```

Auditd is useful for tracking sensitive actions:

```bash
sudo auditctl -w /etc/sudoers -p wa -k sudoers_watch
sudo auditctl -w /etc/shadow -p wa -k shadow_watch
```

What to watch for:

- Repeated failed logins.
- New privileged accounts or sudoers changes.
- Unexpected service restarts or new listening ports.
- SELinux or AppArmor denials that indicate policy violations.
- File changes in sensitive directories such as `/etc`, `/usr/local/bin`, and web roots.

Detection is most effective when logs are centralized to a remote system, because local logs can be altered by an attacker with sufficient access.

**Backups, recovery, and resilience**

Security includes the ability to recover from compromise, ransomware, or accidental deletion. Backups should be protected, tested, and isolated from normal user access.

Best practices:

- Follow the 3-2-1 rule: three copies, two media types, one offsite copy.
- Encrypt backups when they contain sensitive data.
- Test restores regularly.
- Protect backup credentials and repositories.
- Keep immutable or offline backups for ransomware resilience.

Example backup command:

```bash
rsync -aHAX --delete /etc/ /backup/etc/
```

For sensitive systems, consider:

- Snapshot-based backups for fast rollback.
- Separate backup accounts with limited permissions.
- Write-once or object-lock storage where available.

**Threat awareness in real environments**

Security issues often appear as patterns rather than isolated events. You should recognize suspicious behavior in context.

Examples of red flags:

- A web server process spawning a shell.
- A user account logging in from an unusual country or at an unusual time.
- A cron job appearing in `/etc/cron.d` without change control.
- A binary in `/tmp`, `/dev/shm`, or a hidden directory with execute permissions.
- A service listening on a new port after a package update.
- A `setuid` file that was not present during baseline review.

Practical checks:

```bash
# Find recently modified files in sensitive areas
sudo find /etc -type f -mtime -2
sudo find /usr/local/bin -type f -perm /111

# Review running processes and parents
ps -eo pid,ppid,user,cmd --forest

# Check open network connections
ss -tunap
```

A baseline helps you distinguish normal from abnormal. Without one, every alert looks suspicious and every change is harder to validate.

**Hardening checklist**

Use a repeatable hardening process when bringing up a Linux host:

1. Install only required packages.
2. Create a non-root administrative account.
3. Disable direct root SSH login.
4. Enforce key-based authentication where possible.
5. Apply updates and enable automatic security patching if appropriate.
6. Remove unused services and close unused ports.
7. Set correct file permissions and ownership.
8. Enable SELinux or AppArmor in enforcing mode.
9. Configure logging and auditing.
10. Back up configuration and verify restore procedures.

Example baseline commands:

```bash
sudo systemctl disable --now avahi-daemon
sudo systemctl disable --now bluetooth
sudo ufw allow OpenSSH
sudo ufw enable
```

The exact tools vary by distribution, but the goal stays the same: minimize what is exposed, monitor what remains, and make recovery possible.
