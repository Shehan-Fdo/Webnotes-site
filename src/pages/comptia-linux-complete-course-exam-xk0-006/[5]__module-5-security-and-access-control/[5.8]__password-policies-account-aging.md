**Password policy fundamentals**

Linux password policy controls how long credentials remain valid, how often users must change them, and what happens when a password expires. These settings help reduce the risk of stale credentials, shared accounts, and long-lived passwords that are easier to compromise.

Account aging is usually managed through the shadow password system, which stores password metadata separately from `/etc/passwd`. The shadow file tracks fields such as last password change date, minimum and maximum password age, warning period, inactivity period, and account expiration.

- `/etc/passwd` contains user account identity data.
- `/etc/shadow` stores password hashes and aging metadata.
- **Password aging** limits how long a password can be used before change is required.
- **Account expiration** disables the account after a specific date.
- **Inactivity** can lock an account after password expiration if the user does not change it in time.

A typical shadow entry looks like this:

```text
alice:$y$j9T$...:19800:7:90:14:30:20000:
```

Field meanings in `/etc/shadow` are:

1. Username
2. Encrypted password or password placeholder
3. Last password change date, in days since 1970-01-01
4. Minimum password age
5. Maximum password age
6. Password warning period
7. Password inactivity period
8. Account expiration date
9. Reserved field

**Inspecting current policy**

You often need to verify aging settings before changing them. The `chage` command is the primary tool for viewing and modifying password aging values.

```bash
chage -l alice
```

Example output may include:

```text
Last password change                                    : Jan 15, 2026
Password expires                                        : Apr 15, 2026
Password inactive                                       : May 15, 2026
Account expires                                         : never
Minimum number of days between password change          : 7
Maximum number of days between password change          : 90
Number of days of warning before password expires       : 14
```

Use this output to confirm whether a user is approaching expiration, whether the account is already inactive, and whether the policy matches your security requirements.

**Managing aging with `chage`**

`chage` updates password aging attributes for a specific user. It is the most direct and exam-relevant command for account aging tasks.

```bash
sudo chage -M 90 -m 7 -W 14 alice
```

This sets:

- **Maximum age** to 90 days
- **Minimum age** to 7 days
- **Warning period** to 14 days

Common `chage` options include:

- `-l` to list current settings
- `-M` to set maximum password age
- `-m` to set minimum password age
- `-W` to set warning days before expiration
- `-I` to set inactivity days after expiration
- `-E` to set account expiration date
- `-d` to set last password change date

Examples:

```bash
sudo chage -E 2026-12-31 alice
sudo chage -I 30 alice
sudo chage -d 0 alice
```

- `-E 2026-12-31` expires the account on that date.
- `-I 30` disables the account 30 days after password expiration if the password is not changed.
- `-d 0` forces the user to change the password at next login.

> Be careful when setting `-d 0` on service or automation accounts. It can break unattended logins if the account is used by scheduled jobs or applications.

**Using `passwd` for password state changes**

The `passwd` command is not only for changing passwords; it also supports some aging-related actions.

```bash
sudo passwd -e alice
```

This expires the password immediately, forcing a change at next login. This is useful when you want to reset a user’s credential lifecycle without editing shadow data directly.

Other useful actions include:

```bash
sudo passwd -l alice
sudo passwd -u alice
```

- `-l` locks the password by prefixing the hash with a lock marker.
- `-u` unlocks the password.
- Locking prevents password-based login, but other authentication methods may still work depending on system configuration.

**Default policy for new users**

System-wide defaults influence new accounts. These defaults are commonly configured in `/etc/login.defs`, though behavior can vary by distribution and user management tools.

Typical settings include:

```ini
PASS_MAX_DAYS   90
PASS_MIN_DAYS   7
PASS_WARN_AGE   14
```

These values define the default aging policy for newly created local users. They do not always retroactively change existing accounts, so you should verify whether a user-specific policy already exists.

You may also encounter `useradd` defaults and skeleton configuration, but `login.defs` is the classic place to review aging defaults.

**Applying policy to existing users**

To enforce a consistent policy across existing local accounts, you can script `chage` against a list of users.

```bash
for user in alice bob carol; do
  sudo chage -M 90 -m 7 -W 14 "$user"
done
```

This approach is practical when onboarding a group of users or remediating weak password lifecycle settings after an audit.

If you need to identify accounts with weak or missing aging controls, inspect `/etc/shadow` or query with `chage -l` for each account. In larger environments, you may combine this with `awk`, `getent`, or configuration management.

**Shadow file and aging fields**

The shadow file stores aging data in numeric form. Understanding the fields helps when troubleshooting or when `chage` output seems inconsistent.

- The last password change field is stored as a day count, not a calendar date.
- A value of `0` in the last change field usually means the password must be changed immediately.
- Empty or special values may indicate no expiration, depending on the field.
- Direct editing of `/etc/shadow` is possible but risky and usually unnecessary.

Example inspection:

```bash
sudo awk -F: '$1=="alice" {print $0}' /etc/shadow
```

This is useful for verification, but `chage` remains the safer administrative interface.

**Account expiration versus password expiration**

These are related but different controls:

- **Password expiration** requires the user to change the password after a set interval.
- **Account expiration** disables the account entirely after a date.
- **Password inactivity** can disable the account after the password expires and remains unchanged.

Use password expiration for routine credential rotation. Use account expiration for temporary users, contractors, lab accounts, or access that should end on a known date.

| Control | Purpose | Typical use |
|---|---|---|
| Password expiration | Forces password change | Regular security rotation |
| Account expiration | Ends account validity | Temporary access |
| Inactivity period | Disables stale expired passwords | Cleanup after missed change window |
| Minimum age | Prevents rapid repeated changes | Stops password cycling to bypass history |

**Practical administration workflow**

1. Check the current policy with `chage -l username`.
2. Compare the values to your security baseline.
3. Update aging settings with `chage` if needed.
4. Verify the result with another `chage -l` query.
5. Test login behavior carefully, especially for service or shared accounts.

Example:

```bash
sudo chage -l alice
sudo chage -M 90 -m 7 -W 14 -I 30 alice
sudo chage -l alice
```

**Troubleshooting common issues**

If a user reports that they cannot log in, check whether the password expired, the account expired, or the account is locked.

- If the password expired, the user may be prompted to change it at login.
- If the account expired, login is denied regardless of password correctness.
- If the account is locked, the password hash may be disabled.
- If the inactivity period elapsed, the account may be disabled after expiration.

Useful checks:

```bash
sudo chage -l alice
sudo passwd -S alice
sudo grep '^alice:' /etc/shadow
```

`passwd -S` can show whether the account is locked or active, while `chage -l` shows aging status.

**Security and operational considerations**

Password aging is only one part of authentication policy. Strong security also depends on password quality, MFA (multi-factor authentication) where available, least privilege, and account lifecycle management.

- Avoid overly short maximum ages that cause users to choose predictable passwords.
- Use a minimum age to reduce password cycling.
- Set warning periods so users have time to update credentials before lockout.
- Treat service accounts differently from human accounts when policy requires it.
- Prefer centralized identity management when available, but know the local tools for exam and troubleshooting scenarios.

> Do not edit `/etc/shadow` manually unless you have a specific recovery reason and understand the format. A small mistake can lock out users or corrupt authentication data.

**Quick command reference**

```bash
chage -l alice          # Show aging details
sudo chage -M 90 alice  # Set max password age
sudo chage -m 7 alice   # Set minimum password age
sudo chage -W 14 alice  # Set warning days
sudo chage -I 30 alice  # Set inactivity period
sudo chage -E 2026-12-31 alice  # Set account expiration
sudo passwd -e alice    # Force password change at next login
sudo passwd -l alice    # Lock account password
sudo passwd -u alice    # Unlock account password
```

**Exam focus**

For Linux+ tasks, remember the distinction between the tools and the policy locations:

- `chage` manages password aging.
- `passwd` changes or locks passwords and can force expiration.
- `/etc/shadow` stores aging metadata.
- `/etc/login.defs` provides default aging values for new accounts.

If you can explain what each field does, how to inspect it, and how to enforce a policy safely, you cover the core of password policies and account aging.
