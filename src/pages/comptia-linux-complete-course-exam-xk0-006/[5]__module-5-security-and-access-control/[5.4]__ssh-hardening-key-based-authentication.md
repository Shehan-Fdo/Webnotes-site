## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**SSH fundamentals and why hardening matters**

<abbr title="Secure Shell">SSH</abbr> is the standard encrypted protocol for remote administration on Linux. It protects confidentiality and integrity in transit, but a default SSH service is not automatically hardened. Hardening reduces brute-force risk, limits lateral movement, and makes authentication behavior predictable.

The main security goals are:

- Prefer **public key authentication** over passwords.
- Reduce the attack surface by disabling features you do not need.
- Restrict who can log in, from where, and how often.
- Verify host identity to reduce man-in-the-middle risk.

A typical SSH deployment has two sides:

- **Client**: your workstation or automation host running `ssh`, `scp`, `sftp`, and `ssh-keygen`.
- **Server**: the remote Linux system running `sshd` and reading `/etc/ssh/sshd_config`.

**Key files, directories, and permissions**

SSH is very sensitive to file ownership and mode bits. Incorrect permissions are a common reason key-based authentication fails.

Client-side locations:

- `~/.ssh/` stores user SSH material.
- `~/.ssh/id_ed25519` is a common private key file.
- `~/.ssh/id_ed25519.pub` is the matching public key.
- `~/.ssh/config` stores per-host client settings.
- `~/.ssh/known_hosts` stores trusted server host keys.

Server-side locations:

- `/etc/ssh/sshd_config` is the main daemon configuration.
- `/etc/ssh/ssh_host_*` stores server host private keys and public keys.
- `~/.ssh/authorized_keys` contains allowed public keys for a user account.

Typical permission expectations:

- `~/.ssh` should usually be `700`.
- Private keys should usually be `600`.
- `authorized_keys` should usually be `600`.
- Home directories and `.ssh` content must be owned by the correct user.
- Server host private keys under `/etc/ssh/` should be readable only by root.

Example permission checks:

```bash
ls -ld ~/.ssh
ls -l ~/.ssh/id_ed25519 ~/.ssh/id_ed25519.pub ~/.ssh/authorized_keys
stat ~/.ssh ~/.ssh/id_ed25519 ~/.ssh/authorized_keys
```

Common failure modes when permissions are too open:

- `sshd` ignores `authorized_keys`.
- The client refuses to use a private key.
- Authentication falls back to password or fails entirely.
- Logs show messages about bad ownership or permissions.

**Generating and managing SSH keys**

`ssh-keygen` creates and manages key pairs. For modern Linux environments, `ed25519` is usually the best default because it is compact, fast, and widely supported in current OpenSSH versions.

Common key type choices:

- **Ed25519**: preferred for most modern systems.
- **RSA**: useful for older compatibility requirements, but use a strong size such as 3072 or 4096 bits.
- **ECDSA**: supported, but often chosen less frequently than Ed25519.

Generate a modern key pair:

```bash
ssh-keygen -t ed25519 -a 100 -C "admin@workstation"
```

Useful management commands:

```bash
ssh-keygen -l -f ~/.ssh/id_ed25519.pub
ssh-keygen -p -f ~/.ssh/id_ed25519
ssh-keygen -y -f ~/.ssh/id_ed25519
ssh-keygen -R server.example.com
```

What these do:

- `-l` shows the fingerprint and key size.
- `-p` changes the private key passphrase.
- `-y` prints the public key from a private key file.
- `-R` removes a stale host entry from `known_hosts`.

A passphrase on the private key is strongly recommended. It protects the key if the file is copied or stolen. For automation, you may use an agent or a dedicated service account with tighter controls, but you should still treat the private key as a secret.

**Key-based authentication workflow**

Key-based authentication uses a public/private key pair:

- You keep the **private key** on the client and protect it.
- You place the **public key** on the server in `authorized_keys`.
- The server proves the client owns the private key without receiving it.

A simple deployment flow:

1. Generate a key pair on the client.
2. Copy the public key to the server account.
3. Verify permissions on `~/.ssh` and `authorized_keys`.
4. Test login in a second session before disabling passwords.
5. Harden `sshd_config` and reload the daemon.

Copy a public key to a server:

```bash
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@server.example.com
```

If `ssh-copy-id` is unavailable, append the public key manually:

```bash
cat ~/.ssh/id_ed25519.pub | ssh user@server.example.com 'umask 077; mkdir -p ~/.ssh; cat >> ~/.ssh/authorized_keys'
```

Verify the server-side file:

```bash
ssh user@server.example.com 'ls -ld ~/.ssh && ls -l ~/.ssh/authorized_keys && tail -n 1 ~/.ssh/authorized_keys'
```

**Client-side SSH hardening**

Client configuration in `~/.ssh/config` helps you control identity selection, forwarding, and host-specific behavior. This is especially useful when you manage multiple servers or multiple keys.

Example client config:

```sshconfig
Host prod-*
    User admin
    IdentityFile ~/.ssh/id_ed25519_prod
    IdentitiesOnly yes
    ForwardAgent no
    ServerAliveInterval 30
    ServerAliveCountMax 3

Host jumpbox
    HostName jump.example.com
    User ops
    IdentityFile ~/.ssh/id_ed25519_jump
    IdentitiesOnly yes
```

Important options:

- **IdentityFile** selects a specific private key.
- **IdentitiesOnly yes** prevents SSH from offering many unrelated keys.
- **ForwardAgent no** keeps agent forwarding disabled.
- **ForwardAgent yes** enables agent forwarding and should be used only when needed.
- **ServerAliveInterval** and `ServerAliveCountMax` help detect dead sessions.

Agent forwarding is a risk because a compromised remote host can potentially use your agent connection to reach other systems. Prefer jump hosts, short-lived credentials, or separate keys instead of forwarding when possible.

**Server-side hardening in `sshd_config`**

The SSH daemon configuration controls authentication methods, login restrictions, and session behavior. After changes, validate the config before reloading.

Check syntax:

```bash
sshd -t
```

Reload without dropping active sessions:

```bash
systemctl reload sshd
```

Restart only when necessary:

```bash
systemctl restart sshd
```

A practical hardening baseline:

```sshconfig
Port 22
PermitRootLogin prohibit-password
PasswordAuthentication no
KbdInteractiveAuthentication no
PubkeyAuthentication yes
PermitEmptyPasswords no
MaxAuthTries 3
LoginGraceTime 20
AllowUsers admin ops
X11Forwarding no
AllowAgentForwarding no
AllowTcpForwarding no
ClientAliveInterval 300
ClientAliveCountMax 2
Banner /etc/issue.net
```

Key points:

- **PermitRootLogin prohibit-password** allows root login without password-based authentication.
- **PasswordAuthentication no** disables password logins, but only after you confirm key-based access works.
- **KbdInteractiveAuthentication no** disables keyboard-interactive prompts unless you explicitly need them.
- **PermitEmptyPasswords no** blocks accounts with empty passwords.
- **MaxAuthTries** limits repeated authentication attempts.
- **LoginGraceTime** limits how long a client can stay connected before authenticating.
- **AllowUsers** and `AllowGroups` restrict who can log in.
- **Banner** displays a legal warning or policy notice before authentication.

> Be careful with `AuthenticationMethods`: it defines required authentication method combinations. For example, `publickey,password` means the server requires both public key authentication and the SSH password method, and password auth must still be enabled for that to work.

Example multi-factor style requirement:

```sshconfig
AuthenticationMethods publickey,password
PubkeyAuthentication yes
PasswordAuthentication yes
```

This is useful when you want a key plus password combination. It is not the same as “either/or.”

**Host key verification and known_hosts**

SSH does not only authenticate users; it also authenticates servers through host keys. The first time you connect, the client stores the server host key in `~/.ssh/known_hosts`. On later connections, SSH compares the presented key to the stored one.

Why this matters:

- It helps detect man-in-the-middle attacks.
- It warns you when a server host key changes unexpectedly.
- It prevents silent trust of a different machine using the same hostname.

Useful commands:

```bash
ssh-keygen -F server.example.com
ssh-keygen -R server.example.com
ssh-keyscan -H server.example.com >> ~/.ssh/known_hosts
```

Operational notes:

- `ssh-keyscan` is useful for inventory or bootstrap workflows, but it does not verify trust by itself.
- A host key change may be legitimate after a rebuild, but clients will warn until you update `known_hosts`.
- If you see a host key warning unexpectedly, verify the change out of band before accepting it.

Server host keys are usually stored under `/etc/ssh/` with names like:

- `/etc/ssh/ssh_host_ed25519_key`
- `/etc/ssh/ssh_host_rsa_key`

If host keys are lost or intentionally rotated, regenerate them carefully and expect clients to flag the change. On many systems, host keys are generated automatically at install time or on first boot.

**PAM interaction and authentication behavior**

<abbr title="Pluggable Authentication Modules">PAM</abbr> can affect SSH authentication, especially for keyboard-interactive logins, account restrictions, password policies, and multi-factor setups. Even if `sshd_config` allows a method, PAM may still deny access.

Practical implications:

- `PasswordAuthentication` and keyboard-interactive behavior may depend on PAM configuration.
- Account lockouts, password expiry, and access rules can come from PAM.
- MFA solutions often integrate through PAM or keyboard-interactive flows.

When troubleshooting, check both:

- `/etc/ssh/sshd_config`
- PAM files such as `/etc/pam.d/sshd`

**Troubleshooting key-based login**

When public key authentication fails, work through the problem systematically.

1. Confirm the client is using the expected key.
2. Confirm the server account has the correct public key in `~/.ssh/authorized_keys`.
3. Check ownership and permissions on `~/.ssh` and `authorized_keys`.
4. Verify the server allows public key authentication.
5. Review logs for the exact rejection reason.

Useful client-side diagnostics:

```bash
ssh -vvv user@server.example.com
```

Useful server-side log checks:

```bash
journalctl -u sshd -b
tail -f /var/log/auth.log
tail -f /var/log/secure
```

Common error patterns:

- `Permission denied (publickey)`:
  - The wrong key is offered.
  - The public key is missing from `authorized_keys`.
  - Permissions are too open.
  - The server disallows public key auth.
- `Authentication refused: bad ownership or modes`:
  - `~/.ssh` or `authorized_keys` has incorrect ownership or permissions.
- `Host key verification failed`:
  - The server host key changed or `known_hosts` contains stale data.
- `Too many authentication failures`:
  - The client offered too many keys before the correct one.
  - Use `IdentitiesOnly yes` and a specific `IdentityFile`.

A quick server-side permission repair pattern:

```bash
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys
chown -R "$USER:$USER" ~/.ssh
```

**Safe password-authentication deactivation**

Disabling password authentication is a common hardening step, but do it carefully to avoid locking yourself out.

Recommended staged approach:

1. Open a second SSH session before making changes.
2. Confirm key-based login works reliably.
3. Edit `/etc/ssh/sshd_config` and set `PasswordAuthentication no`.
4. Keep `PubkeyAuthentication yes`.
5. Validate the config with `sshd -t`.
6. Reload `sshd` with `systemctl reload sshd`.
7. Test a new login before closing the original session.

Example change set:

```sshconfig
PubkeyAuthentication yes
PasswordAuthentication no
KbdInteractiveAuthentication no
PermitEmptyPasswords no
```

If you need a rollback, restore the previous config and reload the daemon. Avoid editing from your only active session unless you have console access or another recovery path.

**Additional hardening controls often tested**

Several SSH options are frequently relevant in hardening scenarios:

- **AllowUsers** limits access to named users.
- **AllowGroups** limits access to named groups.
- **DenyUsers** and `DenyGroups` block specific accounts or groups.
- **MaxSessions** limits multiplexed sessions per connection.
- **ClientAliveInterval** and `ClientAliveCountMax` help clear dead sessions.
- **X11Forwarding no** disables X11 forwarding when not needed.
- **AllowTcpForwarding no** blocks port forwarding if you do not need it.
- **PermitTunnel no** disables tunneling features.
- **Banner** provides policy or legal notice text.

Example access restriction:

```sshconfig
AllowUsers alice bob
AllowGroups sshadmins
```

Example banner file:

```text
Authorized use only. Activity may be monitored and recorded.
```

**Practical operational examples**

Check the SSH service status:

```bash
systemctl status sshd
```

View the active daemon configuration path and options:

```bash
sshd -T | less
```

Test a connection with a specific key:

```bash
ssh -i ~/.ssh/id_ed25519 -o IdentitiesOnly=yes user@server.example.com
```

Copy files securely:

```bash
scp -i ~/.ssh/id_ed25519 report.txt user@server.example.com:/tmp/
```

Use SFTP for interactive file transfer:

```bash
sftp -i ~/.ssh/id_ed25519 user@server.example.com
```

**Exam-focused distinctions**

- **Public key authentication** proves possession of a private key; the private key never leaves the client.
- **Password authentication** depends on a user password and is easier to brute-force.
- **Host keys** identify the server; user keys identify the client.
- **`known_hosts`** stores trusted server host keys on the client.
- **`authorized_keys`** stores allowed client public keys on the server.
- **`sshd_config`** controls server behavior; `~/.ssh/config` controls client behavior.

**Common hardening checklist**

- Use Ed25519 keys for modern systems.
- Protect private keys with a passphrase.
- Restrict `~/.ssh` and `authorized_keys` permissions.
- Verify host keys and manage `known_hosts`.
- Disable password authentication only after testing key login.
- Limit root login and user access with `PermitRootLogin`, `AllowUsers`, and `AllowGroups`.
- Reduce brute-force exposure with `MaxAuthTries` and `LoginGraceTime`.
- Disable unused forwarding and X11 features.
- Check PAM when authentication behavior does not match `sshd_config`.
