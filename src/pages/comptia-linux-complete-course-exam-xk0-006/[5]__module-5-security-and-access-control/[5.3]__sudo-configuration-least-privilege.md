## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Overview**

`sudo` lets you delegate administrative tasks without sharing the root password. It is a core least-privilege tool because you can grant narrowly scoped access to specific commands, users, groups, hosts, and run-as identities.

The main policy file is `/etc/sudoers`, but you should edit it with `visudo` so syntax errors do not lock you out. In practice, `sudo` is about controlled elevation, auditability, and reducing the blast radius of privileged access.

**Why least privilege matters**

Least privilege means each user or service gets only the access required to do its job. With `sudo`, that usually means:

- Allowing one command instead of full root access
- Restricting which arguments are valid
- Delegating access to a group instead of individual accounts
- Using `sudoedit` instead of running a full editor as root
- Requiring a password for sensitive actions, or intentionally disabling it only where justified

This approach reduces accidental damage, limits lateral movement after compromise, and makes administrative intent easier to audit.

**sudo architecture and policy flow**

When you run `sudo`, the program checks policy rules and then decides whether to execute the command with elevated privileges. The decision depends on:

- The invoking user
- The target host
- The command path and arguments
- The run-as user and group
- Any matching `Defaults` options
- Command tags such as `NOPASSWD` or `PASSWD`

`sudo` does not simply grant “root or nothing.” It can run commands as another user or group, which is useful for service accounts and tightly scoped delegation.

**sudoers file basics**

The sudo policy file is usually `/etc/sudoers`, and additional rules may live in `/etc/sudoers.d/`. Always validate changes with `visudo` or `visudo -c`.

Common editing pattern:

```bash
sudo visudo
```

Check syntax without editing:

```bash
sudo visudo -c
```

Validate a drop-in file:

```bash
sudo visudo -cf /etc/sudoers.d/admins
```

A safe workflow is:

1. Edit with `visudo`.
2. Save the file.
3. Re-check syntax.
4. Test with a non-root account in a separate session.

> A syntax error in sudoers can prevent all privileged access until corrected from recovery mode or another administrative path.

**Rule structure and matching behavior**

A sudoers rule is a command specification that can include user, host, run-as, tags, and command details. The exact grammar is more nuanced than a simple one-line template, but the practical idea is:

- Who may run the command
- On which host
- As which user or group
- With which command and arguments
- With which tags or defaults

A common pattern looks like this:

```sudoers
alice ALL=(root) /usr/bin/systemctl restart sshd
```

This means user `alice` may run that exact command specification on any host as `root`.

Important matching behavior:

- `sudo` matches the full command line, including arguments.
- The command path should usually be absolute.
- Wildcards can broaden access, so use them carefully.
- Multiple rules may match; the effective result depends on the final matching tags and rule order.
- Later entries can override earlier tag settings in practical use, so keep rules explicit and organized.

**Users, groups, and host lists**

You can authorize individual users or groups.

Examples:

```sudoers
alice ALL=(root) /usr/bin/systemctl restart sshd
%ops ALL=(root) /usr/bin/systemctl restart sshd
```

Key points:

- `%group` refers to a Unix group.
- Group-based delegation is easier to maintain than per-user rules.
- Host lists let you restrict a rule to specific systems.
- `ALL` is convenient but should be used only when the scope is truly intended.

A group-based rule is often the best starting point for team administration because you can add or remove users from the group without editing sudoers repeatedly.

**Run-as users and run-as groups**

The run-as portion controls which identity the command uses. This is useful when a task should run as a service account rather than as root.

Examples:

```sudoers
alice ALL=(postgres) /usr/bin/psql
alice ALL=(root:root) /usr/bin/systemctl restart sshd
alice ALL=(www-data) /usr/bin/tee /var/www/html/status.txt
```

Notes:

- `(postgres)` means run as the `postgres` user.
- `(root:root)` specifies both user and group.
- Run-as groups are useful when file ownership or service permissions matter.
- Not every command needs root; sometimes a dedicated service account is safer.

**Command aliases and runas aliases**

Aliases make sudoers easier to read and maintain, especially in larger environments.

Common alias types:

- `User_Alias`
- `Host_Alias`
- `Runas_Alias`
- `Cmnd_Alias`

Example:

```sudoers
User_Alias   WEBADMINS = alice, bob
Runas_Alias   WEBRUN = root
Cmnd_Alias    WEB_CMDS = /bin/systemctl restart nginx, /bin/systemctl reload nginx

WEBADMINS ALL=(WEBRUN) WEB_CMDS
```

Benefits:

- Reduces repetition
- Makes policy easier to audit
- Helps you group related permissions logically

Common pitfalls:

- Aliases do not bypass command matching rules.
- A `Cmnd_Alias` still needs exact, valid command paths and arguments.
- Overly broad aliases can accidentally grant more access than intended.

**Command wildcards and argument restrictions**

Sudoers can use wildcards in command specifications, but wildcards are powerful and risky. They are often the source of unintended privilege escalation if used loosely.

Examples:

```sudoers
alice ALL=(root) /usr/bin/systemctl restart nginx
alice ALL=(root) /usr/bin/systemctl restart nginx.service
alice ALL=(root) /usr/bin/rsync /srv/backups/*
```

Practical guidance:

- Prefer exact command paths and exact arguments.
- Use wildcards only when the command syntax is tightly controlled.
- Remember that shell-style wildcards in sudoers are not the same as shell expansion.
- If you allow a command with user-controlled arguments, verify whether those arguments can change behavior in unsafe ways.

A safer pattern is to allow a single fixed administrative action rather than a general-purpose command with many options.

**NOPASSWD and PASSWD tags**

Command tags control whether `sudo` prompts for a password.

- `NOPASSWD` allows execution without a password prompt.
- `PASSWD` requires authentication.

Examples:

```sudoers
alice ALL=(root) NOPASSWD: /usr/bin/systemctl restart nginx
bob ALL=(root) PASSWD: /usr/bin/systemctl restart nginx
```

Use cases:

- `NOPASSWD` is useful for tightly controlled automation, such as a monitored service restart.
- `PASSWD` is better for interactive administrative actions where you want an extra confirmation step.

Security impact:

- `NOPASSWD` reduces friction but increases risk if the account is compromised.
- If you use `NOPASSWD`, keep the command scope narrow and auditable.
- For automation, consider a dedicated service account with only the required command access.

**Defaults and policy tuning**

`Defaults` entries adjust sudo behavior globally, per user, per host, or per command.

Examples:

```sudoers
Defaults        env_reset
Defaults        secure_path="/usr/sbin:/usr/bin:/sbin:/bin"
Defaults:alice  timestamp_timeout=5
Defaults        logfile="/var/log/sudo.log"
```

Common options:

- `env_reset` clears most environment variables for safety.
- `secure_path` limits the PATH used by sudo commands.
- `timestamp_timeout` controls how long authentication remains valid.
- `logfile` can send sudo logs to a dedicated file on systems that support it.

Why these matter:

- Environment variables can influence command behavior.
- A controlled PATH reduces the risk of running the wrong binary.
- Shorter timestamp timeouts improve security on shared systems.

**sudoedit as a safer alternative**

`sudoedit` lets you edit files as root without running the editor itself with full root privileges. This is often safer than `sudo vi /etc/somefile`.

Example:

```bash
sudoedit /etc/ssh/sshd_config
```

Why it is safer:

- The editor runs as your user.
- `sudoedit` copies the file to a temporary location, lets you edit it, then writes it back with elevated privileges.
- It reduces the risk of a compromised editor plugin or malicious editor behavior.

Use `sudoedit` whenever the task is “edit this file” rather than “run this program as root.”

**Testing delegated access**

Use `sudo -l` to list what a user can run. This is one of the most important validation tools for exam prep and real administration.

Examples:

```bash
sudo -l
sudo -l -U alice
sudo -l -U bob -h server1
```

What to look for:

- Allowed commands
- Run-as identities
- Whether `NOPASSWD` applies
- Host-specific restrictions
- Any defaults that affect behavior

This is the best way to verify whether a rule works for a specific delegated scenario, not just whether `sudo` itself is installed.

**Common least-privilege patterns**

A good sudo policy usually favors narrow, task-specific permissions.

Examples:

- Allow restarting one service:
  ```sudoers
  %webops ALL=(root) /bin/systemctl restart nginx
  ```
- Allow viewing a protected log file through a controlled command:
  ```sudoers
  %support ALL=(root) /usr/bin/journalctl -u sshd
  ```
- Allow editing one configuration file safely:
  ```sudoers
  %admins ALL=(root) sudoedit /etc/ssh/sshd_config
  ```
- Allow a backup job to run as a service account:
  ```sudoers
  backup ALL=(backup) NOPASSWD: /usr/local/bin/run-backup
  ```

Better patterns:

- Use a dedicated service account for recurring automation.
- Restrict commands to a single purpose.
- Prefer wrapper scripts only when the script itself is tightly controlled and root-owned.
- Keep the allowed command surface as small as possible.

**Logging and auditing**

`sudo` can log command usage, which helps with accountability and troubleshooting. Logging may go to the system journal, syslog, or a dedicated sudo log depending on distribution and configuration.

Useful checks:

```bash
sudo journalctl -t sudo
sudo grep sudo /var/log/auth.log
sudo grep sudo /var/log/secure
```

Notes:

- Log locations vary by distribution.
- Some systems use `journalctl`, while others write to `/var/log/auth.log` or `/var/log/secure`.
- You usually do not need `sudo` just to read logs if your account already has read access.

Audit value:

- Shows who ran what and when
- Helps detect unexpected privilege use
- Supports incident response and compliance

**Troubleshooting and verification**

If a sudo rule does not work, check the following:

- The command path is absolute and correct.
- The arguments match exactly what the rule allows.
- The user is in the expected group.
- The host matches the rule.
- The run-as user is permitted.
- A later rule or tag is not overriding the expected behavior.
- The sudoers file has no syntax errors.

Useful commands:

```bash
which systemctl
sudo -l -U alice
sudo visudo -c
sudo -V
```

`sudo -V` shows version and policy-related information, which can help when behavior differs across distributions.

**Security best practices**

- Use `visudo` for every sudoers change.
- Prefer group-based delegation with `%group` over individual user entries when appropriate.
- Use exact command paths and exact arguments.
- Avoid broad wildcards unless you fully understand the risk.
- Prefer `sudoedit` for file changes.
- Use `NOPASSWD` only for narrow, well-audited automation.
- Review `sudo -l` output regularly for privilege creep.
- Keep rules readable and documented so future admins can audit them quickly.

**Quick reference**

| Concept | Meaning | Why it matters |

| `visudo` | Safe sudoers editor/validator | Prevents syntax lockouts |
| `%group` | Group-based authorization | Simplifies delegation |

| `Cmnd_Alias` | Named command set | Reduces repetition |

**Exam focus**

For Linux+ exam readiness, make sure you can:

- Explain why `sudo` supports least privilege better than shared root access
- Edit and validate `/etc/sudoers` safely with `visudo`
- Interpret rules using users, groups, hosts, run-as identities, and command specifications
- Recognize the security tradeoff between `NOPASSWD` and `PASSWD`
- Use `sudo -l -U user` to test delegated access
- Choose `sudoedit` when editing protected files
- Understand that command matching includes paths and arguments, not just the binary name
