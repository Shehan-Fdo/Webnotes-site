**Why permissions matter**

Linux file permissions are a primary control for limiting who can read, modify, or execute data and programs. They protect sensitive files, reduce accidental damage, and help enforce least privilege. In practice, permission mistakes are one of the most common causes of data exposure and privilege escalation.

At a high level, permissions answer three questions:

- **Who** can access the object
- **What** they can do with it
- **How** that access changes system risk

A weak permission model can expose credentials, allow unauthorized code execution, or let users overwrite critical configuration. A strong model supports confidentiality, integrity, and operational stability.

**Core permission model**

Linux uses ownership plus permission bits on files and directories. Every object has:

- **User owner**: usually the creator or assigned administrator
- **Group owner**: a shared access boundary for a team or service
- **Others**: everyone else on the system

The standard permission bits are:

- **Read (`r`)**
- **Write (`w`)**
- **Execute (`x`)**

For files, these bits mean:

- **Read**: view file contents
- **Write**: modify file contents
- **Execute**: run the file as a program or script

For directories, the meaning changes:

- **Read**: list filenames in the directory
- **Write**: create, delete, or rename entries
- **Execute**: traverse the directory and access entries by name

That directory distinction is important. A directory without execute permission is effectively inaccessible even if you can read it, because you cannot traverse into it.

**Security implications of each permission**

Each bit has a different risk profile depending on the object type.

- **Read on sensitive files** can expose passwords, API keys, SSH material, logs, and personal data.
- **Write on executable files** can turn a benign program into a malicious one.
- **Write on configuration files** can redirect services, weaken authentication, or disable logging.
- **Execute on scripts and binaries** can allow code execution, but execute alone does not imply read access to the file contents.
- **Write on directories** is especially powerful because it can enable file replacement, deletion, or symlink attacks.

A common security principle is to avoid granting more access than needed:

- Give users **read-only** access when they only need to inspect data.
- Give services access only to the specific files they require.
- Avoid world-writable directories unless they are intentionally designed for shared temporary use, such as `/tmp`.

**Permission notation and interpretation**

You will often see permissions in symbolic and numeric form.

Symbolic example:

```bash
-rw-r----- 1 alice finance 2048 Apr 26  report.txt
```

Interpretation:

- `-` indicates a regular file
- `rw-` for the owner means read and write
- `r--` for the group means read only
- `---` for others means no access

Numeric example:

- `644` means owner `rw-`, group `r--`, others `r--`
- `600` means owner `rw-`, group `---`, others `---`
- `755` means owner `rwx`, group `r-x`, others `r-x`

The numeric form is compact and common in scripts and provisioning. The symbolic form is easier to read during audits.

**Practical command examples**

Use `ls -l` to inspect permissions:

```bash
ls -l /etc/shadow
ls -ld /var/www /tmp
```

Use `stat` for detailed metadata:

```bash
stat /etc/passwd
```

Change permissions with `chmod`:

```bash
chmod 640 report.txt
chmod u+x deploy.sh
chmod go-rwx secret.key
```

Change ownership with `chown` and group with `chgrp`:

```bash
sudo chown root:finance /srv/finance/report.txt
sudo chgrp developers app.log
```

These commands are powerful because they directly alter who can access data. Use them carefully, especially on shared systems.

**Real-world security examples**

A few common scenarios show why permissions matter.

- **Private keys** should usually be readable only by the owning user or root. If a private key is world-readable, anyone who can access the system may impersonate that identity.
- **Web content** often needs read access for the web server, but not write access. If the web server can write to its own served files, an attacker who compromises the service may inject malicious content.
- **System configuration** under `/etc` should be tightly controlled. If non-administrators can edit authentication or service files, they may weaken security or disrupt services.
- **Shared project directories** may use a group-based model so team members can collaborate without exposing files to all users.

**Special permission bits**

Linux also includes special permission bits that change the security model.

- **Setuid (`s`)**: a program runs with the file owner’s privileges
- **Setgid (`s`)**: a program runs with the file group’s privileges
- **Sticky bit (`t`)**: on directories, only the file owner, directory owner, or root can delete entries

These bits can be useful, but they increase risk.

- **Setuid** can be dangerous because a vulnerable program may grant elevated privileges to an attacker.
- **Setgid** can help with shared collaboration or controlled access, but it can also widen access if misused.
- **Sticky bit** is commonly used on shared writable directories like `/tmp` to prevent users from deleting each other’s files.

Example:

```bash
ls -ld /tmp
```

Typical output shows the sticky bit on a world-writable directory.

**Access control beyond basic permissions**

Basic permissions are only one layer. Linux often uses additional controls:

- **Groups** for shared access boundaries
- **ACLs (Access Control Lists)** for finer-grained per-user or per-group rules
- **umask** to control default permissions on newly created files
- **Capabilities** to grant specific privileged actions without full root access

These mechanisms help when the standard user/group/other model is too coarse.

For example, a team may need one user to have read access to a file without making it readable to the entire group. An ACL can do that:

```bash
setfacl -m u:carol:r-- project-plan.txt
getfacl project-plan.txt
```

**umask and default exposure**

The `umask` value determines which permission bits are removed from newly created files and directories. It is a preventive control that helps avoid accidental overexposure.

Common examples:

- `umask 022` often results in files that are readable by others unless the application restricts them further
- `umask 077` creates private files and directories by default

Check the current value with:

```bash
umask
```

Set it for the current shell with:

```bash
umask 077
```

This is especially important for scripts that create credentials, backups, or reports containing sensitive information.

**Directory permissions and attack surface**

Directory permissions are often more security-sensitive than file permissions because they control namespace access.

- **Read** lets users see filenames, which can leak information about projects, users, or system structure.
- **Write** lets users add or remove entries, which can enable tampering.
- **Execute** lets users enter the directory and access objects inside it.

A directory that is writable but not sticky can be abused by other users to delete or replace files. A directory that is executable but not readable can still be traversed if the user knows the filename.

This matters for shared locations such as:

- `/tmp`
- application upload directories
- team collaboration folders
- spool or cache directories

**Common misconfigurations and risks**

Typical permission mistakes include:

- **World-readable secrets** such as keys, tokens, or database passwords
- **World-writable scripts** that can be modified by untrusted users
- **Overly permissive directories** that allow file replacement or deletion
- **Incorrect ownership** that gives the wrong user control over sensitive data
- **Setuid binaries** that are unnecessary or poorly audited

A quick audit pattern is to look for files that are writable by non-owners or executable where they should not be.

```bash
find / -perm -0002 -type f 2>/dev/null
find / -perm -4000 -type f 2>/dev/null
```

These commands help identify world-writable files and setuid files. Review results carefully; not every match is a vulnerability, but each one deserves attention.

**Hardening workflow**

A practical permission-hardening workflow is:

1. Identify the asset and its sensitivity.
2. Determine the minimum access required.
3. Assign ownership to the correct user and group.
4. Set restrictive permissions.
5. Add ACLs only when the basic model is insufficient.
6. Verify the result with `ls -l`, `stat`, or `getfacl`.
7. Recheck after application updates or deployment changes.

Example hardening for a private key:

```bash
sudo chown root:root /etc/ssl/private/server.key
sudo chmod 600 /etc/ssl/private/server.key
```

Example for a shared project directory:

```bash
sudo chown alice:devteam /srv/project
sudo chmod 2770 /srv/project
```

The `2` in `2770` sets the setgid bit on the directory so new files inherit the group, which helps keep collaboration consistent.

**Troubleshooting permission problems**

When access fails, do not guess. Check the full path, not just the target file.

- Verify ownership and mode on every parent directory
- Confirm the user is in the expected group
- Check whether an ACL overrides the basic mode bits
- Confirm the process is running as the expected account
- Inspect SELinux or AppArmor if permissions appear correct but access still fails

Useful commands:

```bash
namei -l /srv/project/report.txt
id carol
getfacl /srv/project/report.txt
```

`namei -l` is especially useful because it shows permissions on each path component. A user may have access to the file but still be blocked by one parent directory.

**Key takeaways**

- Permissions are a first-line control for limiting exposure and preventing tampering.
- File and directory permissions behave differently, especially for `r`, `w`, and `x`.
- Misconfigured ownership or mode bits can create serious security issues.
- Special bits like setuid, setgid, and sticky bit improve workflows but increase risk if misused.
- ACLs and `umask` extend the basic model when you need finer control or safer defaults.

**Quick reference**

| Item | Security effect | Common risk |
|---|---|---|
| `r` on file | Read contents | Data disclosure |
| `w` on file | Modify contents | Tampering |
| `x` on file | Execute program | Code execution |
| `r` on directory | List names | Information leakage |
| `w` on directory | Add/remove entries | Deletion or replacement |
| `x` on directory | Traverse path | Access by name |
| setuid | Run as owner | Privilege escalation |
| sticky bit | Restrict deletion in shared dirs | Misuse if omitted |
| ACL | Fine-grained access | Complexity and audit burden |
| umask | Default creation mask | Accidental overexposure |

**Exam focus**

For exam scenarios, focus on the relationship between permissions and risk:

- If a file contains secrets, restrict read access.
- If a file is executable, control who can modify it.
- If a directory is shared, consider group ownership, setgid, and sticky bit.
- If standard permissions are not enough, use ACLs.
- If new files are too open by default, adjust `umask`.
