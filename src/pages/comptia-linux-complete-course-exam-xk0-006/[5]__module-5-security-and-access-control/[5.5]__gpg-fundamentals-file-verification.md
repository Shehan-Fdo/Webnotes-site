**GPG overview**

<abbr title="GNU Privacy Guard">GPG</abbr> is the common open-source implementation of the <abbr title="OpenPGP">OpenPGP</abbr> standard. You use it to protect data integrity, authenticate file origin, and optionally encrypt content for confidentiality. For Linux+ purposes, the most important use case is verifying that a downloaded file or package really came from the expected publisher and was not altered in transit.

GPG supports two related trust workflows:

- **Encryption** protects confidentiality so only the intended recipient can read the data.
- **Digital signatures** protect integrity and authenticity so you can verify who signed the data and whether it changed.

A signature is created with the signer’s private key and verified with the signer’s public key. The public key does not reveal the private key; it is used to validate the signature mathematically against the signed content.

**Core concepts**

GPG uses a public/private key pair:

- **Private key**
  - Kept secret by the owner.
  - Used to sign data and decrypt data addressed to that key.
- **Public key**
  - Shared with others.
  - Used to verify signatures and encrypt data to the key owner.

A key pair is usually identified by:

- **Key ID** or **long key ID**
- **Fingerprint**
- **User ID** such as a name and email address

The fingerprint is the most reliable identifier. Key IDs are shorter and easier to spoof visually, so you should prefer fingerprints when confirming trust.

**Why file verification matters**

File verification helps you detect:

- Corruption during download
- Tampering by a malicious intermediary
- Wrong or incomplete files
- Mismatched release artifacts

In practice, you often verify:

- Source tarballs
- Package repository metadata
- Release archives
- Vendor-provided checksums
- Detached signatures for installers or binaries

> A checksum proves that a file matches a published digest. A signature proves that a trusted key signed the file or checksum. Signatures provide stronger authenticity than checksums alone.

**Common verification methods**

There are two main verification patterns:

- **Checksum verification**
  - Compares a file’s hash to a published hash value.
  - Detects accidental or malicious changes.
  - Does not prove who created the file unless the checksum file itself is signed.
- **Signature verification**
  - Uses a public key to verify a signed file or signed checksum list.
  - Confirms both integrity and signer identity, assuming you trust the key.

Typical hash algorithms include:

- SHA-256
- SHA-512
- SHA-1 in older systems, though it is generally avoided for new trust decisions

**Useful GPG commands**

You commonly use `gpg` to import keys, inspect keys, sign files, and verify signatures.

```bash
# Import a vendor public key
gpg --import vendor-public.key

# List keys in your keyring
gpg --list-keys

# Show fingerprints for verification
gpg --fingerprint

# Inspect a key file without importing it
gpg --show-keys vendor-public.key
```

To verify a detached signature, you provide the signature file and the target file:

```bash
gpg --verify release.tar.gz.sig release.tar.gz
```

For a clearsigned file, the signed content is embedded in the `.asc` file, so you verify only the signed file:

```bash
gpg --verify message.asc
```

For a detached signature, the signature file is separate from the data file. The file extension does not determine the signature type; the signature format does.

**Detached signatures vs clearsigned files**

| Type | What it contains | Typical use |
|---|---|---|
| Detached signature | Signature only, separate from the data | Large downloads, release archives |
| Clearsigned file | Signed text plus the original text in one file | Release notes, manifests, text documents |

Detached signatures are common for binary releases because they keep the original file unchanged. Clearsigned files are common when the signed content is human-readable text.

**Checksum verification workflow**

A checksum file usually contains one hash per file, often in a standard format such as:

```text
<hash>  <filename>
```

Example:

```bash
sha256sum myfile.iso
sha256sum -c SHA256SUMS
```

The `-c` option checks files listed in the checksum manifest. For this to work correctly, the manifest must reference files that are available in the current directory or in the paths listed inside the manifest.

A typical workflow is:

1. Download the file and the checksum manifest.
2. Verify the checksum manifest’s signature, if one is provided.
3. Run the checksum check against the downloaded file.
4. Compare the result to the expected value.

Example:

```bash
gpg --verify SHA256SUMS.sig SHA256SUMS
sha256sum -c SHA256SUMS
```

This is stronger than checking a raw checksum alone because the signed manifest proves the checksum list came from the publisher.

**Practical release verification example**

A common vendor release flow looks like this:

```bash
wget https://example.com/releases/SHA256SUMS
wget https://example.com/releases/SHA256SUMS.asc
wget https://example.com/releases/app-1.2.3-linux-x86_64.tar.gz

gpg --verify SHA256SUMS.asc SHA256SUMS
sha256sum -c SHA256SUMS
```

If the checksum file lists the exact filename, `sha256sum -c` reports whether the file matches. If the filename differs, the check fails even if the file content is correct.

**Key management and trust**

Importing a public key is not the same as trusting it. You can have a key in your keyring without having verified that it belongs to the expected publisher.

Important trust practices:

- Verify the fingerprint through an independent channel.
- Prefer vendor documentation, HTTPS, or an out-of-band source for fingerprint confirmation.
- Use a dedicated keyring for repository keys when possible.
- Avoid relying on `apt-key` for per-repository trust in modern Debian and Ubuntu systems.

Inspect a keyring file with:

```bash
gpg --show-keys /etc/apt/keyrings/vendor.gpg
```

This is useful when a repository key is stored in a dedicated keyring and referenced by an APT source entry.

**APT repository signing**

Modern Debian-based systems use signed repository metadata. APT verifies that repository indexes are signed by a trusted key before installing packages.

A typical repository entry may reference a keyring file:

```text
deb [signed-by=/etc/apt/keyrings/vendor.gpg] https://repo.example.com/debian stable main
```

This limits trust to that repository instead of granting the key broad system-wide trust.

**Signing files yourself**

You may need to sign your own release artifacts or internal files.

Detached signature example:

```bash
gpg --detach-sign --armor release.tar.gz
```

This creates an ASCII-armored signature file, often named `release.tar.gz.asc`.

Verify it with:

```bash
gpg --verify release.tar.gz.asc release.tar.gz
```

You can also create a clearsigned text file:

```bash
gpg --clearsign release-notes.txt
```

That produces a signed text file that includes the original content.

**Encryption vs signing**

| Feature | Purpose | Uses private key? | Uses public key? |
|---|---|---:|---:|
| Encrypt | Confidentiality | Recipient private key decrypts | Sender uses recipient public key |
| Sign | Integrity and authenticity | Signer private key signs | Verifier uses signer public key |

For file verification, signing is usually the relevant feature. Encryption is only needed when the file content must remain secret.

**Troubleshooting verification failures**

Common causes of verification problems include:

- Missing public key
- Wrong file paired with the signature
- Corrupted or incomplete download
- Expired or revoked key
- Signature made with a different key than expected
- Checksum file not matching the file name or path

Useful diagnostics:

```bash
gpg --verify file.sig file
gpg --list-keys
gpg --fingerprint
gpg --check-sigs
```

If a detached signature fails, confirm that you are verifying the exact file that was signed. If a clearsigned file fails, verify the `.asc` file directly and do not supply a separate target file.

> A signature failure does not always mean tampering. It can also mean the wrong file, the wrong key, or a bad download.

**Best practices**

- Verify fingerprints from a trusted, separate source.
- Use signatures for authenticity, not checksums alone.
- Verify the signed checksum file before trusting the hashes inside it.
- Keep repository keys in dedicated keyring files.
- Prefer modern repository trust models with `signed-by=`.
- Re-download files if verification fails before assuming compromise.
- Treat expired or revoked keys as a warning sign that needs investigation.

**Quick command reference**

```bash
# Import a key
gpg --import vendor-public.key

# List keys and fingerprints
gpg --list-keys
gpg --fingerprint

# Verify a detached signature
gpg --verify file.sig file

# Verify a clearsigned file
gpg --verify file.asc

# Create a detached ASCII-armored signature
gpg --detach-sign --armor file.tar.gz

# Create a clearsigned text file
gpg --clearsign notes.txt

# Check SHA-256 hashes from a manifest
sha256sum -c SHA256SUMS
```

**Exam focus**

For Linux+ exam scenarios, be ready to:

- Distinguish encryption from signing
- Explain why a public key verifies a signature
- Choose the correct `gpg --verify` syntax for detached versus clearsigned files
- Recognize that a signed checksum file is stronger than a raw checksum file
- Identify modern repository trust practices using keyrings and `signed-by=`
