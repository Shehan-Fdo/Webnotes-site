## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Why auditing and log review matter**

Linux systems generate logs from the kernel, services, authentication stack, and security tools. You use these records to confirm what happened, when it happened, and which process or user was involved. In troubleshooting, logs often provide the fastest path from symptom to root cause. In security work, logs and audit trails help you detect unauthorized access, policy violations, and configuration drift.

A strong workflow combines three layers:

- **Service logs** for application and daemon behavior
- **System logs** for boot, kernel, and authentication events
- **Audit logs** for security-relevant actions such as file access, privilege use, and policy enforcement

**Core log locations and sources**

Linux distributions vary, but the same general sources appear across systems. Some use only the systemd journal, while others also write traditional text logs through a syslog daemon such as `rsyslog` or `syslog-ng`.

- `/var/log/` contains many traditional log files, such as:
  - `/var/log/messages` for general system messages on some distributions
  - `/var/log/syslog` on Debian-based systems
  - `/var/log/secure` or `/var/log/auth.log` for authentication events
  - `/var/log/audit/audit.log` for auditd records
- `systemd-journald` stores structured logs in the journal
- `auditd` records security events from the Linux Audit subsystem

> Many modern systems rely primarily on the journal. Do not assume `/var/log/messages` exists or is populated unless the distribution and logging stack are configured to write there.

**systemd journal basics**

The journal is a binary, indexed log store managed by `systemd-journald`. It captures metadata such as the unit name, PID, UID, boot ID, and priority. This makes filtering more precise than plain text searching.

Common `journalctl` usage:

```bash
journalctl
journalctl -xe
journalctl -u sshd
journalctl -p warning
journalctl --since "2026-04-26 08:00:00" --until "2026-04-26 10:00:00"
journalctl -f
```

Key points:

- `journalctl -p warning` shows messages at `warning` and higher severity, not only exact matches.
- `-xe` is useful for recent errors and explanatory context.
- `-f` follows new entries in real time.
- `-u <unit>` filters by systemd unit, such as `sshd`, `nginx`, or `NetworkManager`.

**Journal priorities and filtering**

The journal uses standard severity levels:

- `emerg`
- `alert`
- `crit`
- `err`
- `warning`
- `notice`
- `info`
- `debug`

Use priority filtering when you want to reduce noise. For example, if you are investigating a service failure, start with `err` or `warning`, then widen the scope if needed.

```bash
journalctl -p err
journalctl -p warning --since today
journalctl -p info -u cron
```

**Boot-based log review**

Boot selection is useful when a problem appears only after a reboot or kernel update. `journalctl` can list boots and query a specific boot by index or boot ID.

```bash
journalctl --list-boots
journalctl -b
journalctl -b -1
journalctl -b <boot-id>
```

Use cases:

- Compare the current boot with the previous boot after a failed update
- Inspect early boot messages when a service never starts
- Confirm whether a kernel panic or initramfs issue occurred before login

**Traditional text log inspection**

Text logs remain common and are still important for exam troubleshooting. Use standard Unix tools to inspect them efficiently.

```bash
head -n 20 /var/log/messages
tail -n 50 /var/log/secure
tail -f /var/log/syslog
less /var/log/audit/audit.log
cat /var/log/dnf.log
grep -i "failed" /var/log/secure
awk '/sshd/ {print $0}' /var/log/messages
sed -n '1,40p' /var/log/syslog
```

Practical guidance:

- Use `head` for the beginning of a file, such as startup messages.
- Use `tail` for the newest entries.
- Use `less` for interactive review and searching.
- Use `grep` for pattern matching.
- Use `awk` and `sed` when you need field extraction or line-range filtering.

**Interpreting `systemctl status` output**

`systemctl status` is often the first place to check when a service fails. It combines service state, recent journal lines, and process information.

```bash
systemctl status sshd
```

Typical clues include:

- `Active: active (running)` means the service is up
- `Active: failed` means systemd detected a failure
- `Loaded:` shows whether the unit file is present and enabled
- Recent log lines often show the exact error message
- Main PID and exit code help identify whether the process crashed or exited cleanly

Use `systemctl status` as a quick triage tool, then move to `journalctl -u <unit>` for deeper review.

**Syslog and journald relationship**

`journald` can store logs locally and can also forward entries to a syslog daemon through its forwarding interface. A syslog daemon such as `rsyslog` may then write selected messages to traditional text files. Some systems also read from the journal and route entries into `/var/log/messages`, `/var/log/syslog`, or other files.

Important distinctions:

- `journald` is the primary collector on many systemd systems
- `rsyslog` and `syslog-ng` are log routers and writers
- File-based logs may be absent if no syslog daemon is configured
- The journal may contain more metadata than plain text logs

**Common syslog-style troubleshooting examples**

A few realistic log snippets help you recognize patterns quickly.

```text
sshd[1842]: Failed password for invalid user admin from 192.0.2.55 port 51422 ssh2
```

This usually indicates a brute-force attempt or a mistyped username. Check whether the source IP is expected and whether account lockout or rate limiting is enabled.

```text
systemd[1]: nginx.service: Main process exited, code=exited, status=1/FAILURE
```

This suggests the service started but the main process exited with an error. Review the service-specific logs and configuration syntax.

```text
kernel: EXT4-fs error (device sda2): ext4_find_entry:1453: inode #...
```

This points to a filesystem issue. You may need to inspect disk health, run filesystem checks during maintenance, and verify recent hardware errors.

**Audit subsystem overview**

The Linux Audit subsystem records security-relevant events at the kernel level. `auditd` collects these events and writes them to `/var/log/audit/audit.log`. Audit is especially useful when you need accountability for file access, privilege escalation, identity changes, or policy enforcement.

Audit answers questions such as:

- Who accessed a sensitive file?
- Which process changed a permission or ownership?
- Did a user run a privileged command?
- Was a rule triggered by a specific syscall or path?

**Audit service and configuration**

The audit daemon is typically managed with systemd.

```bash
systemctl status auditd
systemctl enable --now auditd
```

Primary configuration files include:

- `/etc/audit/auditd.conf` for daemon behavior
- `/etc/audit/rules.d/*.rules` for persistent rules on many distributions
- `/etc/audit/audit.rules` on systems that load a compiled rules file
- `/var/log/audit/audit.log` for event records

Common `auditd.conf` settings to know:

- `log_file` sets the audit log path, usually `/var/log/audit/audit.log`
- `max_log_file` controls the maximum size of a log file before rotation behavior begins
- `num_logs` controls how many rotated logs to keep
- `max_log_file_action` defines what happens when the log reaches its size limit
- `space_left_action` and `admin_space_left_action` define responses to low disk space

These settings matter because audit logs can grow quickly on busy systems. If retention is too small, you may lose evidence. If disk space is exhausted, audit behavior may change depending on policy.

**Audit rule management**

Use `auditctl` for runtime rule management. Persistent rules are usually stored in rule files and loaded at boot.

```bash
auditctl -l
auditctl -D
auditctl -w /etc/passwd -p wa -k identity_changes
auditctl -a always,exit -F arch=b64 -S execve -k exec_monitor
```

Key operations:

- `-l` lists active rules
- `-D` deletes all currently loaded rules
- `-w` watches a file or directory path
- `-p` sets permissions to watch: `r`, `w`, `x`, `a`
- `-k` assigns a searchable key
- `-a always,exit` adds a syscall rule for events that exit the kernel syscall path
- `-S` names the syscall to monitor
- `-F arch=b64` limits the rule to 64-bit architecture

> Be careful not to mix file-watch syntax and syscall syntax incorrectly. `-w` rules use path and permission flags, while syscall rules use syscall names and field filters appropriate to the event type.

**Practical audit examples**

File watch rules are useful for monitoring sensitive configuration files.

```bash
auditctl -w /etc/shadow -p wa -k shadow_watch
auditctl -w /etc/sudoers -p wa -k sudoers_watch
```

These rules help you detect:

- Password database changes
- Privilege policy changes
- Unauthorized edits to critical files

A syscall-based rule can track command execution:

```bash
auditctl -a always,exit -F arch=b64 -S execve -k command_exec
```

This is useful when you need to see which commands were launched, especially during incident response. On busy systems, however, it can generate a large volume of events, so use it carefully.

To search audit logs by key:

```bash
ausearch -k sudoers_watch
ausearch -k command_exec
```

To summarize audit events:

```bash
aureport
aureport -x
aureport -f
```

**Audit log review workflow**

A practical audit workflow is usually iterative:

1. Identify the suspicious time window or host activity.
2. Check the journal for service failures, authentication errors, or boot issues.
3. Review `/var/log/audit/audit.log` for security-relevant events.
4. Search by key, user ID, executable, or file path.
5. Correlate audit entries with syslog or journal timestamps.

Example search flow:

```bash
journalctl --since "2026-04-26 09:00" --until "2026-04-26 09:30"
ausearch -ts 09:00:00 -te 09:30:00
ausearch -k identity_changes
```

**Reading audit records**

Audit records are often split across multiple lines. A single event may include a syscall record, a path record, and a summary record. Learn to correlate them by event ID.

Example:

```text
type=SYSCALL msg=audit(1714125600.123:420): arch=c000003e syscall=2 success=yes exit=3 a0=...
type=PATH msg=audit(1714125600.123:420): item=0 name="/etc/sudoers" inode=...
type=PROCTITLE msg=audit(1714125600.123:420): proctitle=vim /etc/sudoers
```

Interpretation:

- The shared event ID `:420` ties the records together
- `SYSCALL` shows the system call and result
- `PATH` shows the file involved
- `PROCTITLE` shows the command line used

**Authentication and privilege review**

Authentication logs are among the most frequently reviewed records. They help you detect failed logins, sudo usage, and account changes.

Common places to check:

- `journalctl -u sshd`
- `/var/log/secure`
- `/var/log/auth.log`
- `audit.log` for privileged actions

Examples of useful patterns:

```bash
grep -i "failed password" /var/log/secure
grep -i "sudo" /var/log/auth.log
journalctl | grep -i "authentication failure"
```

Look for:

- Repeated failed logins from the same source
- Successful logins after many failures
- Unexpected `sudo` usage
- Account lockouts or password changes

**Log rotation and retention**

Logs must be rotated to prevent uncontrolled growth. Traditional logs often use `logrotate`, while audit logs are governed by `auditd` settings and system storage policy.

Typical rotation concerns:

- File size thresholds
- Retention count
- Compression of old logs
- Permission preservation
- Service reload after rotation

Example `logrotate` snippet:

```conf
/var/log/messages {
    weekly
    rotate 4
    compress
    missingok
    notifempty
    postrotate
        /bin/systemctl kill -s HUP rsyslog.service >/dev/null 2>&1 || true
    endscript
}
```

Audit retention is different from general log rotation because audit logs are security records. If the audit log fills, the daemon may take configured actions such as warning, suspending, or halting depending on policy.

**Troubleshooting strategy**

When a system misbehaves, start broad and narrow down.

- Check service state with `systemctl status`
- Review recent journal entries with `journalctl -xe`
- Filter by unit, time, or priority
- Inspect text logs if the distribution writes them
- Search audit logs for file access or privilege events
- Correlate timestamps across sources

A good troubleshooting mindset is to ask:

- What changed recently?
- Which service failed first?
- Is the issue authentication, networking, storage, or permissions?
- Do the logs show a symptom or the root cause?

**Quick command reference**

```bash
# Journal review
journalctl -xe
journalctl -u sshd
journalctl -b -1
journalctl --list-boots

# Text log review
tail -f /var/log/messages
less /var/log/secure

# Audit review
auditctl -l
auditctl -D
ausearch -k sudoers_watch
aureport -x
```

**Exam focus points**

- Know the difference between the journal, syslog-style text logs, and audit logs
- Understand that `journalctl -p` uses a severity threshold, not a single exact match
- Know how to list boots and inspect a previous boot with `journalctl`
- Recognize when to use `systemctl status` versus `journalctl -u`
- Distinguish audit file watches from syscall rules
- Remember that `auditctl -D` removes loaded rules
- Know where audit configuration and logs usually live
- Understand that log retention and rotation are operational concerns, not just storage details

**Common pitfalls**

- Assuming every distribution uses `/var/log/messages`
- Treating `journalctl` output as plain text files
- Mixing file watch syntax and syscall syntax in audit rules
- Forgetting that audit logs can become large and require retention planning
- Searching only one log source when the answer requires correlating multiple sources
