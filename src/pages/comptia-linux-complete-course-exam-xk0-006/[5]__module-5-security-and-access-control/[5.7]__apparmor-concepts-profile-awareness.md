## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**AppArmor overview**

AppArmor is a Linux <abbr title="Mandatory Access Control">MAC</abbr> framework that confines programs by path-based rules. Instead of labeling every object like some other security models, AppArmor focuses on the executable and the files, capabilities, and operations that a specific program may use.

It is useful when you want to reduce the impact of a compromise. If an application is exploited, the attacker is still constrained by the profile unless they can escape confinement.

- **Path-based control** means rules usually reference full file paths such as `/usr/bin/nginx` or `/var/www/html/**`.
- **Profile-driven** policy means each application can have its own rule set.
- **Enforcement modes** let you test policy before fully blocking actions.
- **Audit visibility** helps you see denials and refine rules.

**Core AppArmor concepts**

AppArmor profiles define what a program can access. A profile is typically attached to a specific executable path and may include file permissions, capability rules, network rules where supported, and execution transitions.

A profile usually has two broad states:

- **Enforce**: violations are blocked and logged.
- **Complain**: violations are logged but not blocked.

This makes AppArmor practical for incremental rollout. You can observe real application behavior in complain mode, then tighten the policy once you understand what the program actually needs.

**Profile structure and syntax**

A basic profile names the executable and lists allowed resources. The syntax is declarative and path-oriented.

```apparmor
#include <tunables/global>

profile /usr/sbin/mydaemon flags=(attach_disconnected) {
  # Allow read access to configuration
  /etc/mydaemon.conf r,

  # Allow logs to be written
  /var/log/mydaemon.log w,

  # Allow access to runtime files
  /run/mydaemon.pid rw,

  # Allow read-only access to libraries and shared data
  /usr/lib/** r,
  /lib/** r,

  # Allow a limited capability
  capability net_bind_service,
}
```

Key syntax ideas:

- **Profile name** often matches the executable path.
- **Permissions** are appended to paths, such as `r`, `w`, `m`, `ix`, or `px`.
- **Globs** like `*`, `**`, and `?` help match multiple paths.
- **Includes** such as `#include <tunables/global>` help standardize variables and common definitions.

> AppArmor syntax is strict. A rule that looks plausible may still be invalid if the attachment or transition form is wrong, so verify with `apparmor_parser` before relying on it.

**Common file permissions in profiles**

AppArmor file rules use short permission letters. These are not the same as traditional UNIX file mode bits, although they often map conceptually to read/write/execute behavior.

- `r` allows read access.
- `w` allows write access.
- `m` allows memory mapping with executable permissions in some contexts.
- `k` allows file locking.
- `l` allows link access in some cases.
- `ix` means execute and inherit the current profile.
- `px` means execute and transition to another profile if one exists.
- `cx` means execute and transition to a child profile.
- `ux` means execute unconfined, subject to policy and mode.
- Uppercase variants such as `Px`, `Cx`, and `Ux` are the more restrictive forms that do not fall back the same way lowercase forms may.

The execute modes are important for exam awareness because they control what happens when a confined process launches another program.

**Execute transition behavior**

AppArmor execute rules are more nuanced than a simple “safe versus unsafe” split. The mode determines whether the new process stays in the current profile, transitions to another profile, or runs with reduced confinement.

- **`ix`** keeps the current confinement.
- **`px`** attempts a transition to a named profile for the target executable.
- **`cx`** transitions to a child profile defined under the current profile.
- **`ux`** runs the program with unconfined execution semantics, which is risky.
- **Uppercase forms** generally avoid fallback behavior and are stricter about transition outcomes.

Practical examples:

```apparmor
/usr/bin/vim ix,
/usr/bin/python3 px,
/usr/bin/helper cx,
/usr/bin/legacy-tool Ux,
```

Use these carefully:

- **`ix`** is useful when the launched helper should remain under the same policy.
- **`px`** is useful when a separate profile exists for the helper binary.
- **`cx`** is useful for tightly related helper behavior defined as a child profile.
- **`ux`** and **`Ux`** are generally avoided unless you have a strong reason and understand the risk.

**Child profiles and hats**

AppArmor supports nested policy structures, but the syntax must be correct. Child profiles are usually written as subprofiles or hats with proper attachment semantics, not as arbitrary nested blocks.

A hat is a special subprofile used for a controlled privilege change within the same application. It is often entered with `change_hat` from the application and is useful for separating high-risk operations from the main process.

Example of a child profile style:

```apparmor
profile /usr/sbin/webapp {
  /usr/sbin/webapp ix,

  profile helper {
    /usr/bin/helper rix,
    /var/lib/webapp/helper/** rw,
  }
}
```

In practice, the exact attachment and transition behavior depends on how the application is designed and how the profile is loaded. For exam purposes, focus on the idea that child profiles narrow privileges for a specific subtask.

**Capabilities in AppArmor**

AppArmor can allow Linux capabilities explicitly. Capabilities are privileged kernel operations such as binding to low ports or changing ownership.

```apparmor
capability net_bind_service,
capability dac_override,
```

Common examples:

- **`net_bind_service`** allows binding to ports below 1024.
- **`dac_override`** bypasses discretionary access control checks.
- **`sys_admin`** is extremely powerful and should be avoided unless absolutely necessary.

Use capability rules sparingly. If a service only needs to bind to port 80 or 443, `net_bind_service` is more appropriate than running the service as root.

**Network rules**

AppArmor can express some network-related policy on systems and kernels that support it, but this is not a universal replacement for firewalling. Network mediation support is limited and distribution-dependent, so do not assume it is the primary control for network security.

Example syntax may appear as:

```apparmor
network inet stream,
network inet6 dgram,
```

Use this as a supplemental control, not as your main network security boundary. For most Linux administration tasks, firewall tools and service binding controls remain the primary network defenses.

**Loading, testing, and managing profiles**

You usually manage profiles with the AppArmor utilities and the kernel interface exposed through the AppArmor subsystem.

Useful commands:

```bash
sudo aa-status
sudo apparmor_status
sudo apparmor_parser -r /etc/apparmor.d/usr.sbin.mydaemon
sudo aa-enforce /etc/apparmor.d/usr.sbin.mydaemon
sudo aa-complain /etc/apparmor.d/usr.sbin.mydaemon
sudo aa-disable /etc/apparmor.d/usr.sbin.mydaemon
```

What these do:

- **`aa-status`** shows loaded profiles and whether they are in enforce or complain mode.
- **`apparmor_status`** provides similar status information on some systems.
- **`apparmor_parser -r`** reloads a profile after changes.
- **`aa-enforce`** switches a profile to enforce mode.
- **`aa-complain`** switches a profile to complain mode.
- **`aa-disable`** disables a profile by removing it from active loading on the next reload or service restart, depending on the distribution workflow.

> `aa-disable` is not an instant magic switch on every system. In practice, you often need to reload AppArmor or restart the affected service for the change to take effect.

**Typical workflow for creating a profile**

1. Start with complain mode so the application keeps working while you observe denials.
2. Run the service under realistic load.
3. Review audit messages and refine the profile.
4. Reload the profile.
5. Switch to enforce mode once the policy is stable.

Example workflow:

```bash
sudo aa-complain /usr/sbin/mydaemon
sudo systemctl restart mydaemon
sudo journalctl -k | grep -i apparmor
sudo aa-logprof
sudo aa-enforce /usr/sbin/mydaemon
```

This approach reduces the risk of breaking production services while still improving confinement.

**Audit logs and troubleshooting**

AppArmor denials usually appear in kernel logs or the system journal. You can inspect them to understand what the application tried to do.

Useful commands:

```bash
sudo journalctl -k | grep -i apparmor
sudo dmesg | grep -i apparmor
sudo grep -i apparmor /var/log/syslog
```

What to look for:

- **Denied path** showing the file or directory the process tried to access.
- **Operation** such as read, write, execute, or capability use.
- **Profile name** identifying which policy blocked the action.
- **Requested permission** indicating what rule you may need to add.

A denial message often tells you exactly which rule is missing, which makes AppArmor easier to tune than a purely opaque security failure.

**Practical profile example**

A simple web service profile might allow the binary to execute, read its config, write logs, and access a runtime directory.

```apparmor
#include <tunables/global>

profile /usr/sbin/simple-web flags=(attach_disconnected) {
  /usr/sbin/simple-web ix,
  /etc/simple-web/** r,
  /var/log/simple-web/** rw,
  /run/simple-web/** rw,
  /usr/share/simple-web/** r,
  capability net_bind_service,
}
```

Why this matters:

- The service can read configuration and static assets.
- The service can write logs and runtime files.
- The service can bind to a privileged port without full root privileges.
- The profile blocks unrelated filesystem access by default.

**AppArmor versus other access control models**

AppArmor is often compared with SELinux, but they are not the same.

| Aspect | AppArmor | SELinux |
|---|---|---|
| Policy basis | Path-based | Label-based |
| Typical complexity | Lower | Higher |
| Learning curve | Easier for many admins | Steeper |
| Rule targeting | Executable paths and file paths | Security contexts and labels |
| Common admin workflow | Profile tuning | Context and policy management |

AppArmor is often preferred when you want a more approachable confinement model. SELinux can be more granular in some environments, but it is also more complex to administer.

**Best practices**

- **Start in complain mode** when introducing a new profile.
- **Keep profiles narrow** and allow only what the application needs.
- **Use separate profiles** for helpers and privileged subprocesses.
- **Prefer explicit paths** over broad wildcards.
- **Review logs regularly** to catch unexpected access attempts.
- **Avoid unnecessary unconfined execution** modes.
- **Document profile intent** so future changes do not weaken security accidentally.

**Exam focus points**

- AppArmor is a **mandatory access control** system.
- It uses **path-based profiles** rather than labels.
- Profiles can run in **enforce** or **complain** mode.
- Execute transitions such as **`ix`**, **`px`**, **`cx`**, and **`ux`** control how child processes are confined.
- `aa-status`, `aa-enforce`, `aa-complain`, and `aa-logprof` are common administration tools.
- Denials are typically reviewed in **kernel logs** or the **system journal**.

**Quick reference**

```bash
# Check status
aa-status

# Put a profile into complain mode
aa-complain /etc/apparmor.d/usr.sbin.mydaemon

# Reload a profile
apparmor_parser -r /etc/apparmor.d/usr.sbin.mydaemon

# Review denials
journalctl -k | grep -i apparmor

# Generate or refine rules from logs
aa-logprof
```

AppArmor is most effective when you treat it as a living policy: observe, refine, enforce, and revisit after application updates. That operational mindset is often what exam questions are testing, not just the definition of the framework.
