## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**SELinux fundamentals**

Security-Enhanced Linux (**SELinux**) is a mandatory access control (**MAC**) system that adds policy-based restrictions on top of standard Linux discretionary access control (**DAC**). DAC answers whether the file owner can read or write; SELinux answers whether the policy allows the action even if DAC permits it.

SELinux is most common on Red Hat family systems, but the concepts apply broadly wherever SELinux is enabled. You use it to reduce the impact of compromised services by confining processes to only the resources they need.

- **Policy enforcement** limits what a process can do, even as root.
- **Contexts** label processes, files, ports, and users.
- **Booleans** toggle common policy behaviors without editing policy modules.
- **Troubleshooting** focuses on denials, labels, and policy state.

SELinux is not a replacement for file permissions. It is an additional control layer that often explains why a service fails even when ownership and mode bits look correct.

**SELinux modes and policy state**

SELinux can run in different modes, and knowing the current mode is the first troubleshooting step.

- **Enforcing** blocks actions that violate policy and logs denials.
- **Permissive** logs denials but does not block them.
- **Disabled** turns SELinux off at boot and removes enforcement entirely.

Check the current mode with:

```bash
getenforce
sestatus
```

Typical output from `sestatus` includes the current mode, the loaded policy, and whether policy is enabled at boot. To switch modes temporarily, use:

```bash
sudo setenforce 0   # permissive
sudo setenforce 1   # enforcing
```

> `setenforce` changes the runtime mode only. If SELinux is disabled in the boot configuration, you must edit the boot settings and reboot to enable it.

A useful workflow is to set a problem system to permissive while you collect logs, then return it to enforcing after you fix the labels or booleans.

**Security contexts and labels**

SELinux decisions are based on security contexts, also called labels. A context usually has four fields:

```text
user:role:type:level
```

The most important field for Linux+ troubleshooting is the **type**. Type enforcement is the core of most SELinux policy decisions.

Example context:

```text
system_u:system_r:httpd_t:s0
```

Common context types include:

- **Process context**: the label assigned to a running process.
- **File context**: the label assigned to files and directories.
- **Port context**: the label assigned to network ports.
- **User context**: the SELinux user mapping, which is separate from the Linux login name.

View labels with:

```bash
ls -Z /var/www/html
ps -eZ | grep httpd
id -Z
```

The `-Z` option is the key indicator that you are viewing SELinux labels rather than standard file metadata.

**Type enforcement and domain transitions**

SELinux uses types to control access between processes and objects. A process runs in a **domain** such as `httpd_t`, and files are labeled with types such as `httpd_sys_content_t`. Policy allows or denies access based on the relationship between the process domain and the object type.

This matters because a service may need a specific file type to function:

- Web content often needs `httpd_sys_content_t`.
- Writable web directories may need `httpd_sys_rw_content_t`.
- SSH keys and system files have their own expected labels.

If a process transitions into a different domain, SELinux can apply a different set of permissions automatically. This is one reason SELinux is effective at containing services.

**File contexts and relabeling**

File contexts tell SELinux how files should be labeled. If a file is placed in the wrong directory or copied manually, it may inherit the wrong label and trigger denials.

Inspect file labels with:

```bash
ls -Z /srv/www
```

View the expected context rules with:

```bash
semanage fcontext -l | grep httpd
```

Apply the correct label to a file or directory tree with:

```bash
sudo restorecon -Rv /var/www/html
```

If you need to define a persistent custom label rule, use `semanage fcontext` and then relabel:

```bash
sudo semanage fcontext -a -t httpd_sys_content_t '/srv/web(/.*)?'
sudo restorecon -Rv /srv/web
```

- `semanage fcontext` changes the policy mapping.
- `restorecon` applies the expected label from policy.
- `chcon` changes a label immediately, but the change is usually not persistent across relabels.

Use `chcon` only for quick testing or temporary fixes:

```bash
sudo chcon -t httpd_sys_content_t /srv/web/index.html
```

For long-term administration, prefer `semanage fcontext` plus `restorecon`.

**SELinux booleans**

Booleans are on/off policy switches that enable or disable specific SELinux behaviors. They are useful when a service needs a common exception that policy already supports.

List booleans with:

```bash
getsebool -a
```

Query a specific boolean:

```bash
getsebool httpd_can_network_connect
```

Enable or disable a boolean temporarily:

```bash
sudo setsebool httpd_can_network_connect on
```

Make the change persistent across reboots:

```bash
sudo setsebool -P httpd_can_network_connect on
```

Common examples include:

- **`httpd_can_network_connect`**: allows web services to initiate outbound network connections.
- **`httpd_enable_homedirs`**: allows Apache to serve content from user home directories.
- **`ftp_home_dir`**: allows FTP access to home directories.
- **`virt_use_nfs`**: allows virtualization services to use NFS-backed storage.

Booleans are often the cleanest fix when a service is blocked by a policy feature that is intentionally disabled by default.

**Practical web server example**

A common SELinux issue occurs when Apache serves content from a nonstandard directory.

Suppose you place files in `/srv/web` and Apache returns `403 Forbidden` even though permissions look correct. The likely causes are wrong file labels or a missing boolean.

Check the labels:

```bash
ls -Zd /srv/web
ls -Z /srv/web/index.html
ps -eZ | grep httpd
```

If the directory is not labeled for web content, fix it:

```bash
sudo semanage fcontext -a -t httpd_sys_content_t '/srv/web(/.*)?'
sudo restorecon -Rv /srv/web
```

If Apache must write to that directory, use a writable content type for the writable path only:

```bash
sudo semanage fcontext -a -t httpd_sys_rw_content_t '/srv/web/uploads(/.*)?'
sudo restorecon -Rv /srv/web/uploads
```

If Apache needs outbound network access, enable the relevant boolean:

```bash
sudo setsebool -P httpd_can_network_connect on
```

This approach is safer than disabling SELinux because it preserves confinement while granting only the needed access.

**Troubleshooting workflow**

When SELinux blocks something, you want to identify the denial, confirm the label mismatch or policy restriction, and then apply the smallest safe fix.

1. Confirm SELinux is active.
   ```bash
   getenforce
   sestatus
   ```
2. Reproduce the problem and inspect logs.
   ```bash
   sudo ausearch -m avc -ts recent
   sudo journalctl -t setroubleshoot --since "10 min ago"
   ```
3. Check the labels on the process, file, and directory.
   ```bash
   ls -Z /path/to/resource
   ps -eZ | grep service_name
   ```
4. Determine whether the issue is a label problem or a boolean.
   - Wrong file type usually needs `restorecon` or `semanage fcontext`.
   - Missing capability often needs `setsebool -P`.
5. Apply the least disruptive fix.
   - Relabel files.
   - Adjust a boolean.
   - Only as a last resort, use permissive mode for diagnosis.
6. Retest in enforcing mode.

A useful command for reading AVC (Access Vector Cache) denials is:

```bash
sudo ausearch -m avc -ts recent
```

You can also use:

```bash
sudo sealert -a /var/log/audit/audit.log
```

if the `setroubleshoot` tools are installed. The report often explains the denied access and suggests a fix.

**Reading AVC denials**

An AVC denial usually includes the source context, target context, class, and requested permission. For example, a denial may show that `httpd_t` tried to read a file labeled `user_home_t`.

What to look for:

- **Source context**: the process that was denied.
- **Target context**: the file, port, or object being accessed.
- **Class**: the object type, such as `file`, `dir`, or `tcp_socket`.
- **Permission**: the action denied, such as `read`, `write`, or `name_connect`.

This tells you whether to relabel the target, change a boolean, or adjust the service configuration.

**Ports and SELinux**

SELinux also labels ports. A service may fail to bind to a port if the port is not allowed for that service type.

View port labels with:

```bash
semanage port -l | grep http
```

Add a custom port for a service type:

```bash
sudo semanage port -a -t http_port_t -p tcp 8080
```

If the port already exists under another type, modify it instead of adding a duplicate:

```bash
sudo semanage port -m -t http_port_t -p tcp 8080
```

This is useful when you run a web service on a nonstandard port and want SELinux to permit it without disabling protection.

**SELinux users and role concepts**

SELinux includes its own user, role, and type model. This is separate from Linux user accounts.

- **SELinux user**: maps a login to allowed roles and ranges.
- **Role**: limits which domains a user can enter.
- **Type**: defines the domain or object label used for access decisions.
- **Level**: supports multi-level security in advanced deployments.

For Linux+ purposes, the most important practical idea is that the SELinux user mapping can differ from the Linux username. You can inspect mappings with:

```bash
semanage login -l
```

This is more relevant in hardened or multi-user environments than in basic server administration, but it helps explain why a login may not behave as expected under SELinux.

**Best practices**

- Keep SELinux in **enforcing** mode on production systems.
- Fix labels and booleans before considering broader policy changes.
- Use `restorecon` for policy-based relabeling.
- Use `semanage fcontext` for persistent custom file context rules.
- Use `setsebool -P` when a documented policy toggle solves the problem.
- Avoid disabling SELinux unless you are isolating a problem temporarily.

**Exam focus**

For Linux+ exam scenarios, expect to identify:

- Whether SELinux is enforcing, permissive, or disabled.
- The difference between DAC and MAC.
- How to read and correct file contexts.
- When to use `restorecon` versus `chcon`.
- How booleans change service behavior.
- How to interpret AVC denials and use logs to troubleshoot.

A strong mental model is: permissions say *who owns the file*, while SELinux says *whether the policy allows the action*. That distinction explains many real-world service failures.
