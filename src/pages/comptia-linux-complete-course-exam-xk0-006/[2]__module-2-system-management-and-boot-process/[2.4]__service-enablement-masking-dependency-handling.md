## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Service management overview**

`systemd` is the init system and service manager on most modern Linux distributions. It starts units in parallel, tracks dependencies, and manages service lifecycle through unit files and symlinks. For exam purposes, you need to know the difference between making a service available, making it start automatically, and preventing it from starting at all.

A **unit** is any resource `systemd` manages, such as a service, socket, timer, mount, or target. A **service unit** usually ends in `.service`, but many commands let you omit the suffix when the meaning is clear.

Common service states include:

- **enabled**: `systemd` has install-time symlinks so the unit starts automatically when its target is reached.
- **disabled**: the unit does not start automatically from its normal enablement links, but it can still start manually or through other activation paths.
- **masked**: the unit is linked to `/dev/null`, so `systemd` refuses to start it.
- **static**: the unit has no `[Install]` section and cannot be enabled directly; it is usually started by dependencies or triggers.
- **indirect**: the unit can be enabled only through another unit or alias relationship, depending on the package design.

**Enable, disable, and start behavior**

`enable` and `start` solve different problems. `enable` changes boot-time behavior by creating symlinks. `start` activates the unit immediately in the current session. You often use both when you want a service running now and after reboot.

Typical commands:

```bash
sudo systemctl enable nginx
sudo systemctl start nginx
sudo systemctl enable --now nginx
sudo systemctl disable nginx
sudo systemctl stop nginx
```

Key points:

- **enable** creates symlinks under locations such as `/etc/systemd/system/*.wants/` so the unit is pulled in by its target at boot.
- **start** launches the service now without changing whether it starts on the next boot.
- **enable --now** combines both actions.
- **disable** removes the enablement symlinks, but the unit can still be started manually or by another unit, socket, timer, path, or dependency.
- **stop** ends the running process now without changing future boot behavior.

A disabled service is not “dead”; it is simply not part of the normal automatic startup path. This distinction matters on exams and in troubleshooting.

**Masking and unmasking**

Masking is stronger than disabling. A masked unit points to `/dev/null`, which blocks activation attempts from any source that resolves through `systemd` unit loading.

```bash
sudo systemctl mask apache2
sudo systemctl unmask apache2
sudo systemctl status apache2
```

Use masking when you want to prevent a service from starting under any normal `systemd` path, such as:

- avoiding conflicts between mutually exclusive daemons
- blocking a vulnerable or deprecated service
- preventing accidental activation during maintenance

Important behavior:

- A masked unit cannot be started manually with `systemctl start`.
- A masked unit cannot be enabled.
- If another unit has a hard dependency on a masked unit, the dependent transaction typically fails because the required unit cannot be started.
- If another unit merely **wants** a masked unit, `systemd` usually continues the transaction and logs that the wanted unit could not be started.

> A masked unit is intentionally unreachable through normal `systemd` activation. Use it carefully, especially on shared or production systems.

**Dependency types and startup ordering**

`systemd` uses dependencies to decide what must start, what should start, and in what order. The most important dependency relationships for Linux+ are `Requires=`, `Wants=`, `After=`, and `Before=`.

| Dependency | Meaning | Effect on transaction | Ordering |
|---|---|---|---|
| `Requires=` | Hard dependency | Fails if the required unit cannot be started | No ordering by itself |
| `Wants=` | Soft dependency | Continues even if the wanted unit fails | No ordering by itself |
| `After=` | Start this unit after another unit | Does not pull in the other unit | Ordering only |
| `Before=` | Start this unit before another unit | Does not pull in the other unit | Ordering only |

A common exam trap is confusing dependency with ordering:

- `Requires=` and `Wants=` decide whether another unit is pulled into the transaction.
- `After=` and `Before=` decide sequence, not inclusion.
- You often combine them, for example `Requires=network-online.target` with `After=network-online.target`.

Example unit snippet:

```ini
[Unit]
Description=Example application
Requires=network-online.target
After=network-online.target

[Service]
ExecStart=/usr/local/bin/example-app
Restart=on-failure

[Install]
WantedBy=multi-user.target
```

In this example, the application is pulled in with `multi-user.target` when enabled, and it waits until the network is considered online before starting.

**Enablement and install sections**

The `[Install]` section tells `systemctl enable` what symlinks to create. Without it, a unit is usually static and cannot be enabled directly.

Common directives in `[Install]`:

- `WantedBy=`: creates a symlink in a target’s `.wants/` directory
- `RequiredBy=`: creates a symlink in a target’s `.requires/` directory
- `Alias=`: creates an alternate name for the unit
- `Also=`: enables related units at the same time

Example:

```ini
[Install]
WantedBy=multi-user.target
Also=example-app.socket
```

Practical implications:

- A service with `WantedBy=multi-user.target` can be enabled to start during normal multi-user boot.
- A static service may still start if another unit references it with `Requires=` or `Wants=`.
- A package may ship a service as static because it is meant to be activated by a socket, timer, or target rather than enabled directly.

**Socket, timer, and path activation**

Not all services start from boot targets. `systemd` can activate units on demand.

- **Socket activation**: a `.socket` unit listens on a port or Unix socket and starts the matching service when traffic arrives.
- **Timer activation**: a `.timer` unit starts a service on a schedule, similar to `cron`.
- **Path activation**: a `.path` unit starts a service when a file or directory changes.

Examples:

```bash
sudo systemctl enable --now example-app.socket
sudo systemctl enable --now log-cleanup.timer
sudo systemctl enable --now watch-config.path
```

Why this matters:

- A disabled service may still start if its socket, timer, or path unit is active.
- Masking the service prevents that service from starting, even if the trigger unit fires.
- When troubleshooting, check the trigger unit as well as the service unit.

**Inspecting dependencies and relationships**

Use `systemctl` to understand why a unit starts or fails.

Useful commands:

```bash
systemctl status nginx
systemctl is-enabled nginx
systemctl is-active nginx
systemctl list-dependencies nginx
systemctl list-dependencies --reverse nginx
systemctl cat nginx
systemctl show nginx -p FragmentPath -p UnitFileState -p ActiveState -p SubState
```

What each command helps you verify:

- `status` shows current state, recent logs, and the main process.
- `is-enabled` tells you whether the unit is enabled, disabled, masked, or static.
- `list-dependencies` shows what the unit pulls in.
- `list-dependencies --reverse` shows what depends on the unit.
- `cat` displays the unit file and any drop-ins.
- `show` exposes machine-readable properties for scripting and troubleshooting.

A useful workflow is to ask:

1. Is the unit installed and loadable?
2. Is it enabled, disabled, masked, or static?
3. Is another unit pulling it in?
4. Is ordering delaying startup?
5. Is the failure in the service itself, or in a dependency such as networking, storage, or a socket?

**Practical enablement examples**

Enable a web server to start at boot and now:

```bash
sudo systemctl enable --now nginx
```

Disable it so it no longer starts from its normal boot links:

```bash
sudo systemctl disable nginx
```

Stop it immediately but leave boot behavior unchanged:

```bash
sudo systemctl stop nginx
```

Mask a service to prevent accidental startup:

```bash
sudo systemctl mask cups
```

Unmask it later if you need it again:

```bash
sudo systemctl unmask cups
```

Check whether a unit is static or enabled:

```bash
systemctl is-enabled rsyslog
```

Possible outputs include:

- `enabled`
- `disabled`
- `masked`
- `static`
- `indirect`

**Dependency handling in real troubleshooting**

If a service fails to start, the cause is often not the service binary itself. Common dependency-related causes include:

- the network is not yet online
- a required mount point is missing
- a database service is unavailable
- a socket unit is masked or inactive
- a configuration file is missing or unreadable

Example: a service that needs a mounted data volume.

```ini
[Unit]
Description=Data processor
Requires=var-data.mount
After=var-data.mount

[Service]
ExecStart=/usr/local/bin/data-processor
```

If `/var/data` is not mounted, the service may fail because its required mount cannot be satisfied. In contrast, if the mount is only ordered with `After=` but not required, the service may still attempt to start and fail later when it tries to access files.

**Drop-in overrides and custom dependency changes**

You do not usually edit vendor unit files directly. Instead, create a drop-in override.

```bash
sudo systemctl edit nginx
```

This opens an override file such as `/etc/systemd/system/nginx.service.d/override.conf`.

Example override:

```ini
[Unit]
After=network-online.target
Wants=network-online.target
```

This approach is safer because package updates do not overwrite your changes. It is also easier to audit and remove.

If you need to inspect the full merged unit, use:

```bash
systemctl cat nginx
```

**Common exam distinctions**

- **enable vs start**: enable affects future boots; start affects the current session.
- **disable vs mask**: disable removes automatic startup links; mask blocks startup entirely.
- **Wants= vs Requires=**: wants is soft; requires is hard.
- **After= vs Requires=**: after controls order only; requires pulls in the dependency.
- **static vs disabled**: static units cannot be enabled directly; disabled units can be enabled later.
- **preset vs enable**: preset applies preset policy files; it does not always mean “restore vendor default” in every environment.

**Preset behavior**

`systemctl preset` applies preset policy from files such as `/usr/lib/systemd/system-preset/*.preset` and `/etc/systemd/system-preset/*.preset`.

```bash
sudo systemctl preset nginx
sudo systemctl preset-all
```

Use presets when you want to apply the distribution’s or administrator’s policy for which units should be enabled or disabled. Presets are policy-driven, so the result depends on the preset files present on the system. They are not a universal reset button.

**Troubleshooting checklist**

When a service does not start as expected, check in this order:

1. Confirm the unit name and state with `systemctl status <unit>`.
2. Check whether it is masked with `systemctl is-enabled <unit>`.
3. Inspect dependencies with `systemctl list-dependencies <unit>`.
4. Review logs with `journalctl -u <unit>`.
5. Verify related targets, sockets, timers, mounts, and network readiness.
6. Confirm the unit file syntax and any drop-in overrides.

Example log review:

```bash
journalctl -u nginx --since "10 minutes ago"
```

If the service is masked, unmask it before trying to start it. If it is disabled, decide whether you need to enable it, or whether another activation path should be used instead.

**Quick reference**

- Use `enable` when you want automatic startup at boot.
- Use `start` when you want the service running now.
- Use `disable` when you want to stop automatic startup from enablement links.
- Use `mask` when you want to block all normal activation.
- Use `Requires=` for hard dependencies.
- Use `Wants=` for soft dependencies.
- Use `After=` and `Before=` for ordering only.
- Use `systemctl list-dependencies` and `journalctl` to trace startup problems.
