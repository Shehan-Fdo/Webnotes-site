## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Boot sequence overview**

The Linux boot process is a chain of trust and handoff points. Each stage has a narrow job: firmware initializes hardware, the bootloader finds and loads the kernel, the kernel starts drivers and mounts the root filesystem, and `systemd` or another init system brings the machine to a usable state.

A failure at any stage produces different symptoms, so recovery depends on identifying where the chain broke. In practice, you troubleshoot by checking firmware settings, bootloader configuration, kernel and initramfs pairing, storage visibility, and root filesystem availability.

- **Firmware** starts the machine and chooses a boot target.
- **Bootloader** loads the kernel and initramfs into memory.
- **Kernel** initializes hardware and mounts the real root filesystem.
- **Init system** starts services and user space.

**Firmware and boot mode**

Modern systems usually boot in either legacy BIOS mode or UEFI (Unified Extensible Firmware Interface) mode. The boot path and recovery steps differ significantly between them.

Legacy BIOS reads the first sector of the boot disk, traditionally the Master Boot Record (MBR). The MBR contains two important parts:

- **Boot code** in the first 446 bytes, which is tiny and only responsible for loading the next stage.
- **Partition table** in the remaining bytes, which describes disk partitions.

Because the MBR is small, it cannot hold a full bootloader. It only loads the next stage from disk. This also means the partition table is limited in size and layout, which is one reason GPT (GUID Partition Table) replaced it on modern systems.

The legacy BIOS boot process often depends on the **active** or **bootable** partition flag. The BIOS loads MBR code, and that code may look for the partition marked active to continue booting. If the wrong partition is marked active, or none is marked active on a BIOS-based install, the system may fail to boot even if the files are present.

UEFI uses firmware variables and boot entries instead of the MBR boot code path. It loads an `.efi` bootloader file from the EFI System Partition (ESP), a FAT32 partition usually mounted at `/boot/efi`.

- Common UEFI loader paths include:
  - `/boot/efi/EFI/<vendor>/grubx64.efi`
  - `/boot/efi/EFI/<vendor>/shimx64.efi`
  - `/boot/efi/EFI/BOOT/BOOTX64.EFI`
- Vendor and distro differences matter:
  - Fedora and RHEL-family systems often use `shimx64.efi` plus `grubx64.efi`.
  - Debian and Ubuntu commonly place files under `/boot/efi/EFI/ubuntu/`.
  - Some systems use a fallback path under `EFI/BOOT/` for removable-media style recovery.

> Secure Boot can block unsigned or tampered boot components. If Secure Boot is enabled, recovery may require signed `shim`, signed GRUB, and a signed kernel. A manually copied or rebuilt bootloader may fail to start until it is signed or Secure Boot is temporarily disabled in firmware.

**Bootloader concepts**

The bootloader bridges firmware and the Linux kernel. The most common bootloader in Linux environments is GRUB 2, though systemd-boot and vendor-specific loaders also exist.

GRUB typically performs these tasks:

- Presents a boot menu.
- Reads configuration from files such as `grub.cfg`.
- Loads the selected kernel and initramfs.
- Passes kernel command-line parameters such as `root=`, `ro`, `quiet`, or `rd.lvm.lv=`.

GRUB has two important modes:

- **Normal mode** loads the menu and configuration.
- **Rescue mode** appears when GRUB cannot find its configuration, modules, or root filesystem.

In rescue mode, you can often recover temporarily by setting `root` and `prefix` manually, loading modules, and booting the kernel by hand.

```text
grub rescue> ls
grub rescue> set root=(hd0,gpt2)
grub rescue> set prefix=(hd0,gpt2)/boot/grub
grub rescue> insmod normal
grub rescue> normal
```

If `normal` mode still does not load, you may need to load modules manually and boot a kernel directly from the GRUB prompt.

```text
grub> ls
grub> set root=(hd0,gpt2)
grub> linux /boot/vmlinuz-6.8.0-xx root=/dev/mapper/vg0-root ro
grub> initrd /boot/initramfs-6.8.0-xx.img
grub> boot
```

- **`root`** in GRUB points to the partition GRUB should read.
- **`prefix`** points to the directory containing GRUB modules and configuration.
- **`linux`** loads the kernel image and kernel parameters.
- **`initrd`** or `initrd16` loads the initramfs image.
- **`boot`** starts the selected entry immediately.

**Kernel and initramfs relationship**

The kernel image and initramfs image must match the installed version closely. A common recovery mistake is booting `vmlinuz-<version>` with the wrong `initramfs-<version>.img`, or using a stale initramfs after a kernel update.

Typical naming patterns include:

- `/boot/vmlinuz-6.8.0-xx`
- `/boot/initramfs-6.8.0-xx.img`
- `/boot/initrd.img-6.8.0-xx`

The exact naming depends on the distribution:

- RHEL, Fedora, and related systems often use `initramfs-<version>.img`.
- Debian and Ubuntu often use `initrd.img-<version>`.
- Some systems keep symlinks such as `/boot/vmlinuz` or `/boot/initrd.img` pointing to the current version.

To verify the installed kernel and image pair, check both the package database and the `/boot` directory.

```bash
uname -r
ls -l /boot/vmlinuz* /boot/initramfs* /boot/initrd.img*
rpm -q kernel-core kernel
dpkg -l 'linux-image*' 'initramfs-tools*'
```

- `uname -r` shows the running kernel, which may differ from the installed target if you are recovering offline.
- Package queries help confirm which kernel version should have a matching initramfs.
- A missing or mismatched initramfs often causes early boot failures such as dropped-to-emergency-shell behavior or missing storage drivers.

**Initramfs recovery and regeneration**

The initramfs (initial RAM filesystem) contains temporary userspace tools and drivers needed before the real root filesystem is available. It often includes storage, filesystem, encryption, LVM (Logical Volume Manager), and RAID modules.

If the initramfs is missing drivers or is corrupted, rebuild it from a rescue environment or chroot.

For dracut-based systems:

```bash
dracut --force --kver 6.8.0-xx
dracut --force --regenerate-all
```

For Debian/Ubuntu systems using initramfs-tools:

```bash
update-initramfs -u -k 6.8.0-xx
update-initramfs -c -k 6.8.0-xx
```

- Use `-u` to update an existing image.
- Use `-c` to create a new image.
- Use `--force` when you need to overwrite a damaged image.
- Regenerate after kernel updates, storage changes, or crypt/LVM/RAID configuration changes.

Many distributions generate initramfs content from configuration files and hooks. Typical examples include:

- `/etc/dracut.conf` and `/etc/dracut.conf.d/*.conf`
- `/etc/initramfs-tools/initramfs.conf`
- `/etc/initramfs-tools/modules`
- `/etc/crypttab`
- `/etc/fstab`

If you add a new storage controller, move root to LVM, or enable disk encryption, you may need to regenerate the initramfs so the required modules and unlock scripts are included.

**Booting encrypted root, LVM, and RAID**

Complex storage stacks must be available before the root filesystem can mount. The initramfs usually contains the tools needed to assemble them.

- **Encrypted root**:
  - The initramfs must include `cryptsetup` support.
  - The kernel command line may need `rd.luks.uuid=` or `cryptdevice=` depending on the distro.
  - The unlock prompt appears before the root filesystem mounts.
- **LVM**:
  - The initramfs must activate volume groups and logical volumes.
  - Common tools include `lvm` and `lvm2`.
  - The root parameter often points to `/dev/mapper/<vg>-<lv>`.
- **RAID**:
  - The initramfs must assemble arrays with `mdadm` or equivalent tooling.
  - The root filesystem may live on `/dev/md0` or a device mapper path layered on top of RAID.

Practical recovery often requires checking that the initramfs generation process includes the right hooks and modules.

```bash
# RHEL/Fedora-style examples
dracut --force --add crypt --add lvm --add mdraid --kver 6.8.0-xx

# Debian/Ubuntu-style examples
echo cryptsetup >> /etc/initramfs-tools/modules
echo dm_mod >> /etc/initramfs-tools/modules
update-initramfs -u -k all
```

- Include storage modules before rebuilding the image.
- Verify that `/etc/crypttab` and `/etc/fstab` reference the correct UUIDs.
- Confirm that the kernel command line matches the storage layout.

**GRUB and bootloader file locations**

Bootloader files live in different places depending on boot mode and distribution.

- BIOS/legacy GRUB commonly uses:
  - `/boot/grub2/grub.cfg`
  - `/boot/grub/grub.cfg`
  - `/boot/grub2/i386-pc/`
- UEFI GRUB commonly uses:
  - `/boot/efi/EFI/<distro>/grubx64.efi`
  - `/boot/efi/EFI/<distro>/shimx64.efi`
  - `/boot/grub2/grub.cfg` or `/boot/grub/grub.cfg` for configuration
- systemd-boot commonly uses:
  - `/boot/loader/loader.conf`
  - `/boot/loader/entries/*.conf`

The exact path varies by distribution and whether `/boot` is separate from the root filesystem. When recovering, always confirm where the firmware expects the loader and where the configuration actually resides.

Useful inspection commands:

```bash
efibootmgr -v
lsblk -f
find /boot /boot/efi -maxdepth 3 -type f | sort
cat /etc/fstab
```

- `efibootmgr -v` shows UEFI boot entries and their file paths.
- `lsblk -f` helps identify the ESP, `/boot`, and root partitions.
- `find` confirms whether the expected `.efi`, kernel, and initramfs files exist.

**Recovery from a live environment**

A common recovery workflow is to boot from rescue media, mount the installed system, chroot into it, and rebuild the boot components.

1. Identify the root, boot, and ESP partitions.
2. Mount the installed system under `/mnt`.
3. Bind-mount system directories needed for chroot.
4. Chroot into the installed system.
5. Reinstall or repair the bootloader.
6. Rebuild the initramfs.
7. Regenerate the bootloader configuration.
8. Reboot and verify.

```bash
mount /dev/mapper/vg0-root /mnt
mount /dev/sda2 /mnt/boot
mount /dev/sda1 /mnt/boot/efi

mount --bind /dev /mnt/dev
mount --bind /proc /mnt/proc
mount --bind /sys /mnt/sys
mount --bind /run /mnt/run

chroot /mnt
```

Inside the chroot, repair the bootloader and regenerate configuration.

```bash
# BIOS GRUB example
grub2-install /dev/sda
grub2-mkconfig -o /boot/grub2/grub.cfg

# UEFI GRUB example
grub2-install --target=x86_64-efi --efi-directory=/boot/efi --bootloader-id=linux
grub2-mkconfig -o /boot/grub2/grub.cfg
```

On Debian/Ubuntu systems, the equivalent commands may be:

```bash
grub-install /dev/sda
update-grub
```

> If the ESP is full or corrupted, bootloader repair may fail even when the kernel and root filesystem are healthy. Check free space with `df -h /boot/efi`, verify the FAT filesystem, and repair it with `fsck.vfat` or `dosfsck` from a rescue environment before reinstalling the EFI loader.

**UEFI recovery details**

UEFI recovery often involves both firmware entries and files on the ESP. If the firmware entry is missing, you can recreate it with `efibootmgr`.

```bash
efibootmgr -c -d /dev/sda -p 1 -L "Linux" -l '\EFI\linux\grubx64.efi'
efibootmgr -v
```

- `-d` selects the disk.
- `-p` selects the partition number of the ESP.
- `-l` specifies the EFI loader path using backslashes.
- `-L` sets the boot entry label.

If Secure Boot is enabled, ensure the loader chain is signed:

- `shimx64.efi` is commonly used as the first-stage signed loader.
- `grubx64.efi` may need to be signed or packaged by the distro.
- The kernel may need to be signed depending on the platform policy.

If a signed component is replaced by an unsigned one, the system may stop at firmware or return to the boot menu without a clear Linux error.

**Troubleshooting common boot failures**

Boot failures usually map to a specific stage in the chain.

- **Firmware cannot find a boot entry**:
  - Check UEFI boot order with `efibootmgr -v`.
  - Verify the ESP is mounted and contains the expected `.efi` files.
- **GRUB rescue prompt appears**:
  - Manually set `root` and `prefix`.
  - Load `normal` and boot temporarily.
  - Reinstall GRUB from a rescue environment.
- **Kernel loads but root filesystem is not found**:
  - Check the `root=` kernel parameter.
  - Verify the initramfs includes storage, LVM, RAID, or encryption support.
  - Confirm UUIDs and device names.
- **System drops to emergency shell**:
  - Inspect `/etc/fstab` for bad UUIDs or mount options.
  - Check journal logs with `journalctl -xb`.
- **Boot loops after kernel update**:
  - Verify the kernel and initramfs versions match.
  - Rebuild the initramfs and regenerate the bootloader config.

Useful commands during troubleshooting:

```bash
journalctl -xb
blkid
lsinitrd /boot/initramfs-6.8.0-xx.img
file /boot/vmlinuz-6.8.0-xx
```

- `lsinitrd` shows what is inside a dracut-generated initramfs.
- `file` confirms that the kernel image is present and readable.
- `blkid` helps verify UUIDs used by `root=`, `fstab`, and `crypttab`.

**Practical recovery patterns**

A few exam-relevant patterns appear often in real systems:

- A BIOS system with the wrong active partition flag may need the boot flag corrected before GRUB can continue.
- A UEFI system with a missing or renamed loader file may boot only after the `.efi` file is restored to the expected ESP path.
- A system with a new storage controller may require a regenerated initramfs so the kernel can see the root disk early.
- A system with encrypted root may fail if the initramfs lacks `cryptsetup` hooks or the correct unlock parameters.
- A system with LVM or RAID may fail if the initramfs does not activate the volume group or assemble the array before mounting root.

**Key commands to remember**

```bash
# Inspect boot entries and disks
efibootmgr -v
lsblk -f
blkid

# Rebuild initramfs
dracut --force --kver <version>
update-initramfs -u -k <version>

# Rebuild GRUB config
grub2-mkconfig -o /boot/grub2/grub.cfg
update-grub

# Reinstall bootloader
grub2-install /dev/sda
grub-install /dev/sda
```

**Exam focus**

For Linux+ recovery questions, identify the stage first, then choose the tool that matches that stage.

- Firmware and partitioning issues point to BIOS/UEFI setup, ESP health, or boot entries.
- GRUB prompt issues point to manual `root`/`prefix` recovery or GRUB reinstall.
- Kernel/initramfs issues point to version matching, driver inclusion, and image regeneration.
- Storage-stack issues point to `cryptsetup`, LVM activation, RAID assembly, and initramfs configuration.

A strong answer usually includes both the immediate fix and the reason the system failed, such as a missing driver, a bad UUID, a corrupted ESP, or a Secure Boot signature mismatch.
