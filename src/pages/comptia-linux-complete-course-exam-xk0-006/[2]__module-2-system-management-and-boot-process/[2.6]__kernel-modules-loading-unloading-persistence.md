**Kernel modules overview**

Linux kernel modules are pieces of code that extend the running kernel without requiring a reboot. They are commonly used for device drivers, filesystems, network protocols, and other features that may not need to be built into the kernel permanently. This modular design keeps the base kernel smaller and lets you add or remove functionality as needed.

A module is loaded into kernel space, where it runs with high privilege. Because of that, module management affects system stability and security. You should know how to inspect modules, load them manually, remove them safely, and make module settings persist across reboots.

- **Module**: a loadable kernel object, usually stored as `.ko` or `.ko.xz`.
- **Built-in code**: compiled into the kernel image and cannot be unloaded as a module.
- **Dependency**: one module may require another module to be loaded first.

Common module locations include:

```bash
/lib/modules/$(uname -r)/
```

That directory usually contains module files organized by kernel version, along with metadata used by module tools.

**Inspecting loaded modules**

You can view currently loaded modules with `lsmod`, which reads information from `/proc/modules`. This is the fastest way to see what is active right now.

```bash
lsmod
cat /proc/modules
```

Typical `lsmod` output includes the module name, size, and usage count. The usage count helps you understand whether a module is in use by other modules or by the kernel.

Useful inspection commands include:

- `modinfo <module>`: show metadata such as description, author, license, filename, and parameters.
- `lsmod`: list loaded modules.
- `cat /proc/modules`: view the kernel’s module table directly.
- `find /lib/modules/$(uname -r) -name '*.ko*'`: locate module files on disk.

Example:

```bash
modinfo e1000e
```

This is useful when you need to confirm whether a driver exists, where it is stored, or which parameters it accepts.

**Loading modules**

The standard tool for loading a module is `modprobe`. It resolves dependencies automatically, which makes it safer and more convenient than inserting a module file directly.

```bash
sudo modprobe <module_name>
```

Examples:

```bash
sudo modprobe vfat
sudo modprobe e1000e
```

If you need to load a module file directly, `insmod` inserts a specific `.ko` file without resolving dependencies. Use it only when you know the dependency chain is already satisfied.

```bash
sudo insmod /lib/modules/$(uname -r)/kernel/drivers/net/ethernet/intel/e1000e/e1000e.ko
```

Key differences:

- **modprobe**: preferred for normal administration; handles dependencies and aliases.
- **insmod**: low-level insertion of a single module file; no dependency resolution.
- **depmod**: builds module dependency metadata used by `modprobe`.

If a module fails to load, check:

- whether the module exists for the running kernel version
- whether Secure Boot or module signing policies block it
- whether the kernel already includes the feature built in
- whether required dependencies are missing
- whether the module is blacklisted

A practical troubleshooting sequence:

```bash
uname -r
modinfo <module_name>
sudo modprobe <module_name>
dmesg | tail -n 20
```

`dmesg` often shows the real reason for failure, such as symbol mismatches, unsupported hardware, or signature problems.

**Unloading modules**

To remove a module, use `modprobe -r` or `rmmod`. As with loading, `modprobe -r` is usually preferred because it understands dependencies.

```bash
sudo modprobe -r <module_name>
sudo rmmod <module_name>
```

Important behavior:

- A module cannot be removed if it is in use.
- If another module depends on it, you must remove dependents first.
- Some modules are intentionally difficult or impossible to unload if the kernel or hardware is actively using them.

Before unloading, check usage:

```bash
lsmod | grep <module_name>
```

If the usage count is nonzero, investigate what is holding the module open. For example, a filesystem module may be in use because a filesystem is mounted, or a network driver may be active because an interface is up.

> Be careful when unloading storage, network, or display drivers on a remote system. Removing the wrong module can disconnect your session or make the system unstable.

**Module parameters**

Many modules accept parameters that change behavior at load time. You can view them with `modinfo` and pass them to `modprobe`.

```bash
modinfo e1000e | less
```

Load with parameters:

```bash
sudo modprobe e1000e InterruptThrottleRate=3000
```

You can also set parameters persistently in configuration files under `/etc/modprobe.d/`.

Example:

```conf
options e1000e InterruptThrottleRate=3000
```

Module parameters are useful for tuning hardware behavior, enabling debug output, or working around device quirks.

**Persistence across reboots**

Modules loaded manually with `modprobe` or `insmod` are not automatically persistent after reboot unless something configures them to load again. Persistence is usually managed through module configuration files or system startup mechanisms.

Common persistence methods:

- **Automatic dependency loading**: a module is loaded when hardware or another subsystem requests it.
- **`/etc/modules-load.d/`**: lists modules to load at boot.
- **`/etc/modprobe.d/`**: stores options, aliases, blacklists, and install/remove rules.
- **Initramfs**: early boot image may include modules needed before the root filesystem is available.

Example boot-load file:

```conf
# /etc/modules-load.d/storage.conf
vfat
nls_utf8
```

This tells the system to load those modules during boot.

Example parameter file:

```conf
# /etc/modprobe.d/e1000e.conf
options e1000e InterruptThrottleRate=3000
```

Example blacklist file:

```conf
# /etc/modprobe.d/blacklist.conf
blacklist pcspkr
```

Blacklisting prevents automatic loading, but it does not always stop a manual `modprobe` unless additional rules are used. If you need stronger control, you may also use an install override.

Example:

```conf
install pcspkr /bin/false
```

This blocks normal loading through `modprobe` by replacing the install action.

**Dependency management**

Module dependencies are tracked in metadata generated by `depmod`. When the kernel updates or modules are added, dependency information may need to be rebuilt.

```bash
sudo depmod -a
```

This command scans the module tree and updates dependency maps such as `modules.dep`. It is especially useful after installing out-of-tree modules or when module metadata appears stale.

Related files under `/lib/modules/$(uname -r)/` may include:

- `modules.dep`
- `modules.alias`
- `modules.symbols`
- `modules.builtin`

These files help `modprobe` locate modules, resolve aliases, and distinguish built-in functionality from loadable objects.

**Aliases and automatic loading**

Hardware and subsystems often request modules by alias rather than by exact module name. `modprobe` uses alias metadata to map a request to the correct module.

Example:

```bash
modprobe -c | grep '^alias'
```

This is useful when a device is detected but the exact driver name is not obvious. The kernel may request a module automatically when it sees matching hardware IDs.

**Built-in versus loadable modules**

Not every kernel feature is a module. Some are compiled directly into the kernel.

- **Built-in feature**: always present after boot; cannot be unloaded.
- **Loadable module**: can be inserted and removed dynamically.

You can often infer built-in status from `modinfo` or `lsmod` behavior:

- If a feature works but does not appear in `lsmod`, it may be built in.
- If `modprobe` says the module is not found, the feature may be built in or absent from the installed kernel package.

**Practical workflow**

When you need to manage a module, follow a simple workflow:

1. Identify the hardware or feature you need.
2. Check whether the module is already loaded with `lsmod`.
3. Inspect module details with `modinfo`.
4. Load it with `modprobe` if needed.
5. Verify operation with `dmesg`, `ip link`, `lsblk`, or application-specific checks.
6. Add configuration under `/etc/modprobe.d/` or `/etc/modules-load.d/` if persistence is required.
7. Rebuild initramfs if the module must be available very early in boot.

Example verification for a network driver:

```bash
lsmod | grep e1000e
ip link show
dmesg | grep -i e1000e
```

**Common commands at a glance**

| Command | Purpose |
|---|---|
| `lsmod` | List loaded modules |
| `modinfo <module>` | Show module metadata |
| `modprobe <module>` | Load a module with dependencies |
| `modprobe -r <module>` | Unload a module safely |
| `insmod <file.ko>` | Insert a specific module file |
| `rmmod <module>` | Remove a module directly |
| `depmod -a` | Rebuild dependency metadata |
| `dmesg` | View kernel messages |

**Troubleshooting tips**

- If `modprobe` fails, check `dmesg` first.
- If a module is missing after reboot, confirm whether you configured persistence in the correct file.
- If a module loads but the device still does not work, verify firmware, permissions, and kernel support.
- If a module is blacklisted, confirm whether another configuration file overrides it.
- If you are working on a production system, prefer `modprobe` over `insmod` and test changes during a maintenance window.

**Exam focus**

For Linux+ exam purposes, remember these distinctions:

- `modprobe` is the normal tool for loading and unloading modules.
- `insmod` and `rmmod` are lower-level and do not manage dependencies.
- `lsmod` shows what is currently loaded.
- `modinfo` shows module details and parameters.
- Persistence usually comes from `/etc/modules-load.d/` and `/etc/modprobe.d/`.
- `depmod` updates dependency metadata after module changes.
