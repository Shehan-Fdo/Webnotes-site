**Targets vs. runlevels**

Linux distributions historically used <abbr title="runlevel">runlevels</abbr> to describe system operating states. Modern systemd-based systems use **targets** instead. For exam purposes, you should understand both the legacy mapping and the practical systemd commands you use today.

A runlevel is a numbered state from the older SysV init model. A target is a named synchronization point in systemd that groups services and dependencies. In practice, targets are more flexible because they can pull in multiple units, support parallel startup, and express richer boot relationships.

- **Legacy model**: `init` and `/etc/inittab` controlled runlevels on SysV systems.
- **Modern model**: `systemd` uses targets such as `multi-user.target` and `graphical.target`.
- **Exam focus**: know the common mappings, but treat them as compatibility concepts, not exact one-to-one behavior on every distro.

**Common runlevel-to-target mapping**

The mapping below is the standard exam-level reference. Exact behavior can vary by distribution, especially for runlevel `2`.

| Runlevel | Typical systemd target | Meaning |
|---|---|---|
| `0` | `poweroff.target` | Halt and power off the system |
| `1` | `rescue.target` | Single-user rescue mode |
| `2` | `multi-user.target` | Multi-user text mode on many distros; distro-specific on some systems |
| `3` | `multi-user.target` | Multi-user text mode |
| `4` | `multi-user.target` | Unused or custom on many systems |
| `5` | `graphical.target` | Multi-user mode with graphical login |
| `6` | `reboot.target` | Reboot the system |

> `init 1`, `init 3`, and `init 5` are historical compatibility commands on systemd systems. They are not the preferred operational method and may not behave identically across all distributions.

**What targets actually do**

Targets are special unit types that represent a desired system state. They do not usually perform work themselves; instead, they group other units and define ordering and dependency relationships.

Common examples include:

- `default.target`
- `multi-user.target`
- `graphical.target`
- `rescue.target`
- `emergency.target`
- `reboot.target`
- `poweroff.target`

A target can be active when the system has reached that boot state. For example, if `graphical.target` is active, the system has reached the graphical boot target, even if no user is logged in yet.

**Inspecting the current boot state**

To check the current state, use systemd inspection commands rather than `get-default`, which only shows the configured default target for the next boot.

```bash
systemctl list-units --type=target
systemctl is-active graphical.target
systemctl is-active multi-user.target
systemctl list-dependencies graphical.target
```

Useful interpretations:

- `systemctl list-units --type=target` shows which targets are currently loaded and active.
- `systemctl is-active <target>` tells you whether a specific target is active.
- `systemctl list-dependencies <target>` shows what units are pulled in by that target.

To see the default target that will be used on the next boot:

```bash
systemctl get-default
```

**Default target vs. current target**

The default target is the boot target systemd selects during startup. It is not the same as the current active state if you have already isolated to another target.

- `systemctl get-default` shows the configured default.
- `systemctl isolate <target>` changes the current runtime state.
- `systemctl set-default <target>` changes the boot-time default for future boots.

**Switching targets safely**

You can move the running system into another target with `systemctl isolate`. This is useful for maintenance, troubleshooting, or changing between graphical and text modes.

```bash
sudo systemctl isolate multi-user.target
sudo systemctl isolate graphical.target
sudo systemctl isolate rescue.target
```

Important behavior:

- Switching from `graphical.target` to `multi-user.target` may stop the display manager and end graphical user sessions.
- On remote systems, this can interrupt services that depend on the graphical stack.
- Use caution when isolating targets on production or remote hosts.

**Setting the default boot target**

To make a target the default for future boots, use `set-default`.

```bash
sudo systemctl set-default multi-user.target
sudo systemctl set-default graphical.target
systemctl get-default
```

This is the preferred way to change whether the machine boots into text mode or graphical mode.

**Practical mapping examples**

A few common scenarios help anchor the concepts:

- A server that should boot without a GUI usually uses `multi-user.target`.
- A desktop system usually uses `graphical.target`.
- Recovery work often uses `rescue.target` or `emergency.target`.
- A reboot request maps to `reboot.target`, while shutdown maps to `poweroff.target`.

**Legacy commands and compatibility**

Older documentation may refer to `runlevel`, `telinit`, or `init`. On systemd systems, these are compatibility interfaces.

```bash
runlevel
who -r
init 3
init 5
telinit 1
```

Use these with caution:

- `runlevel` reports the previous and current runlevel in legacy terms.
- `who -r` can show the current runlevel on many systems.
- `init 3` and `init 5` are compatibility aliases, not the preferred method.
- `init 1` is historically associated with single-user mode, but `rescue.target` is the clearer systemd concept.

**Emergency and rescue modes**

These modes are often confused, but they serve different purposes.

- **Rescue mode**: starts a minimal system with basic services and a root shell, usually requiring the root password.
- **Emergency mode**: starts with even fewer services, often mounting only the root filesystem read-only or minimally, for severe recovery situations.

Typical commands:

```bash
sudo systemctl rescue
sudo systemctl emergency
```

Use these when normal boot or service startup fails and you need local repair access.

**Quick reference commands**

```bash
# Show the current default target
systemctl get-default

# Show active targets
systemctl list-units --type=target

# Check whether a target is active
systemctl is-active multi-user.target

# Switch the current session to text mode
sudo systemctl isolate multi-user.target

# Switch the current session to graphical mode
sudo systemctl isolate graphical.target

# Set the default boot target
sudo systemctl set-default graphical.target

# View dependencies of a target
systemctl list-dependencies graphical.target
```

**Exam tips**

- Remember that `multi-user.target` is the modern equivalent for most non-GUI runlevel use cases.
- Treat runlevel `2` carefully: on many systemd distributions it maps to `multi-user.target`, but its exact meaning can differ from runlevel `3`.
- Do not confuse `get-default` with the current boot state.
- Know that `graphical.target` being active means the system reached the graphical boot target, not necessarily that a user session is open.
- Prefer `systemctl` commands over legacy `init` commands in modern administration.

**Mental model**

Think of runlevels as the old numbered labels and targets as the modern named states. The important exam skill is translating between the two while using the correct systemd commands to inspect, change, and persist the system’s boot mode.
