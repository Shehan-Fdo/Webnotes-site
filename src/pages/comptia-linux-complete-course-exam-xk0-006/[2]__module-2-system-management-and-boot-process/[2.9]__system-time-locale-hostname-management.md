**System time fundamentals**

Linux timekeeping affects authentication, logs, certificates, scheduled jobs, package management, and distributed services. You usually manage time at three layers: the system clock, the hardware clock, and network time synchronization.

- The **system clock** is the running kernel time used by processes and logs.
- The **hardware clock** or RTC (Real-Time Clock) stores time on the motherboard when the system is off.
- **NTP** (Network Time Protocol) keeps the system clock accurate over time.

`timedatectl` is the main high-level tool on systemd-based systems. It shows current time state and lets you change timezone, NTP status, and RTC behavior.

```bash
timedatectl
timedatectl status
```

Typical output includes local time, universal time, RTC time, timezone, NTP service state, and whether the RTC is set to local time.

**Viewing and setting time**

You can inspect the current time with standard commands, but `timedatectl` is preferred for systemd hosts because it integrates with time synchronization and timezone settings.

```bash
date
date -u
timedatectl set-time '2026-04-26 14:30:00'
```

- `date` shows or sets the system clock.
- `date -u` displays UTC (Coordinated Universal Time).
- `timedatectl set-time` is useful when NTP is disabled or unavailable.

> If NTP is active, manual time changes may be rejected or immediately corrected. Disable synchronization first if you need a one-time manual adjustment.

**Timezone management**

A timezone defines how UTC is displayed as local time. Linux commonly stores timezone data in `/usr/share/zoneinfo/`, and `/etc/localtime` points to the selected zone file.

- `/etc/localtime` is often a symlink to a file under `/usr/share/zoneinfo/`.
- On some distributions, `/etc/timezone` stores the timezone name as text for persistence or package tooling.
- `timedatectl` updates timezone settings cleanly on systemd systems.

```bash
timedatectl list-timezones | grep -i london
timedatectl set-timezone Europe/London
ls -l /etc/localtime
readlink -f /etc/localtime
cat /etc/timezone 2>/dev/null
```

A common persistence model is:

- `/etc/localtime` links to the zone database file.
- `/etc/timezone` may record the zone name on Debian-based systems.
- The running system clock remains in UTC internally, while applications render local time using the timezone.

**Hardware clock and RTC behavior**

The hardware clock can be configured to store either UTC or local time. Most Linux systems prefer UTC because it avoids daylight saving time confusion and simplifies dual-boot setups.

```bash
timedatectl set-local-rtc 0
hwclock --show
hwclock --systohc
hwclock --hctosys
```

- `hwclock --systohc` writes the system time to the hardware clock.
- `hwclock --hctosys` reads the hardware clock into the system clock.
- `timedatectl set-local-rtc 0` tells systemd to treat the RTC as UTC.

> If the RTC is set to local time, daylight saving changes and dual-boot environments can cause drift or apparent time jumps.

**NTP synchronization and service choices**

Modern Linux systems often use `systemd-timesyncd`, `chrony`, or `ntpd` to synchronize time. `timedatectl` can enable or disable the systemd NTP client, but exam prep should also cover service-specific tools.

```bash
timedatectl set-ntp true
timedatectl timesync-status
systemctl status systemd-timesyncd
```

For `chrony`:

```bash
systemctl status chronyd
chronyc tracking
chronyc sources -v
chronyc sourcestats -v
```

For `ntpd`:

```bash
systemctl status ntpd
ntpq -p
```

Common configuration files include:

- `/etc/chrony.conf` or `/etc/chrony/chrony.conf`
- `/etc/ntp.conf`
- `/etc/systemd/timesyncd.conf`

Practical verification steps:

- Check whether the service is running with `systemctl status`.
- Confirm synchronization with `chronyc tracking`, `ntpq -p`, or `timedatectl timesync-status`.
- Review logs with `journalctl -u chronyd` or `journalctl -u systemd-timesyncd`.

**Time troubleshooting in VMs and containers**

Virtual machines often inherit time drift from the host or pause/resume behavior. Containers usually share the host kernel clock, so you do not normally set time inside a container.

- After VM suspend/resume, verify time immediately.
- If a VM drifts, check guest tools or hypervisor time sync settings.
- In containers, adjust the host time or synchronization service instead of the container.

Useful checks:

```bash
timedatectl
journalctl -b | grep -i time
```

**Locale fundamentals**

A locale defines language, character encoding, collation, date formats, number formats, and other regional conventions. It affects shell output, application messages, sorting, and text processing.

Common locale components include:

- Language, such as `en`
- Region, such as `US`
- Encoding, such as `UTF-8`

Examples:

- `en_US.UTF-8`
- `de_DE.UTF-8`
- `fr_FR.UTF-8`

Use `locale` to inspect the current environment.

```bash
locale
locale -a
```

- `locale` shows active locale variables.
- `locale -a` lists installed locales available on the system.

**Locale variables and precedence**

Locale behavior is controlled by environment variables such as:

- `LANG`
- `LC_ALL`
- `LC_TIME`
- `LC_NUMERIC`
- `LC_MESSAGES`
- `LC_COLLATE`

`LC_ALL` overrides all other locale variables. `LANG` acts as the default when more specific `LC_*` variables are unset.

```bash
export LANG=en_US.UTF-8
export LC_TIME=de_DE.UTF-8
export LC_ALL=C
```

The `C` locale is the traditional POSIX locale and is useful for predictable scripting, sorting, and parsing.

**Persistent locale configuration**

For system-wide persistence, Linux distributions use different files and tools.

- Debian-based systems often use `/etc/default/locale`.
- Red Hat-based systems commonly use `/etc/locale.conf`.
- Some systems also rely on locale generation tools to ensure the locale exists before it is selected.

Examples:

```bash
cat /etc/default/locale
cat /etc/locale.conf
```

To set a persistent locale on systemd systems, `localectl` is the preferred tool.

```bash
localectl status
localectl set-locale LANG=en_US.UTF-8
```

On non-Debian systems, you may need to generate the locale first:

```bash
localedef -i en_US -f UTF-8 en_US.UTF-8
locale -a | grep en_US
```

On Debian-based systems, locale generation is often handled with:

```bash
dpkg-reconfigure locales
locale-gen
```

> Always verify that the target locale is installed before setting it. If it is missing, applications may fall back to `C` or display warnings.

**Keyboard layout management**

Locale management on systemd systems often includes keyboard layout settings because both are handled through `localectl`. This is especially important on servers accessed through physical consoles or rescue environments.

```bash
localectl status
localectl list-keymaps | grep -i us
localectl set-keymap us
localectl set-x11-keymap us pc105
```

- `set-keymap` changes the virtual console keyboard layout.
- `set-x11-keymap` changes the graphical session layout.
- `list-keymaps` helps you verify available console layouts before applying one.

A mismatch between keyboard layout and locale can make password entry and shell access difficult, especially on non-US keyboards.

**Locale troubleshooting and text behavior**

Locale settings affect sorting, case conversion, and character handling. If you see unexpected output, check whether the shell session inherited the intended locale.

```bash
printf '%s\n' a B c | sort
LC_ALL=C printf '%s\n' a B c | sort
```

- UTF-8 locales support multilingual text and are the default choice for modern Linux systems.
- The `C` locale can produce byte-wise sorting that differs from human language expectations.
- Applications may need a logout/login or service restart to pick up changed locale variables.

**Hostname fundamentals**

A hostname identifies a system on a network. Linux distinguishes between the short hostname, the static hostname, the transient hostname, and the pretty hostname in some environments.

- The **static hostname** persists across reboots.
- The **transient hostname** can be assigned by DHCP, cloud-init, or network services.
- The **pretty hostname** is a human-friendly label used by some desktop tools.

Use `hostnamectl` to inspect and manage hostnames on systemd systems.

```bash
hostnamectl
hostname
hostname -f
```

- `hostname` shows the current hostname.
- `hostname -f` attempts to show the FQDN (Fully Qualified Domain Name).
- `hostnamectl` shows static, transient, and pretty hostnames when available.

**Setting hostnames correctly**

A hostname change should be consistent across the system, DNS, and local name resolution. For a stable server identity, set the static hostname and ensure name resolution works.

```bash
hostnamectl set-hostname web01
hostnamectl set-hostname web01.example.com
```

If you need a pretty hostname:

```bash
hostnamectl set-hostname "Web Server 01" --pretty
```

A good practice is to use:

- a short hostname for local identification, such as `web01`
- an FQDN for DNS-aware environments, such as `web01.example.com`

**FQDN, DNS, and `/etc/hosts`**

The FQDN combines the hostname and domain name. DNS resolves names across the network, while `/etc/hosts` provides local static mappings that can help during boot or troubleshooting.

Example `/etc/hosts` entry:

```bash
127.0.0.1   localhost
192.0.2.10  web01.example.com web01
```

This helps when:

- DNS is unavailable during boot
- you need local resolution for the machineâ€™s own name
- you are troubleshooting name lookup issues

Check resolution with:

```bash
getent hosts web01
getent hosts web01.example.com
hostname -f
```

If `hostname -f` fails, the system may not have a correct DNS record, `/etc/hosts` entry, or domain search configuration.

**Transient hostname behavior**

Transient hostnames are often supplied by DHCP clients, cloud metadata services, or network managers. They can override what you expect if the system is configured to accept remote naming information.

- DHCP may provide a hostname option.
- NetworkManager or cloud-init may set a transient name at boot.
- The transient hostname can differ from the static hostname until you explicitly set the static value.

Troubleshooting steps:

```bash
hostnamectl status
nmcli general hostname
cat /etc/hostname
```

If a service keeps changing the hostname, inspect its configuration and disable hostname updates if appropriate. For example, DHCP client settings may include hostname requests or acceptance of server-provided names.

**Practical hostname troubleshooting**

When hostname resolution fails, check the full chain:

1. Confirm the local hostname with `hostnamectl`.
2. Verify `/etc/hostname` contains the expected static name.
3. Check `/etc/hosts` for a matching local entry.
4. Confirm DNS records for the FQDN.
5. Test resolution with `getent hosts` and `hostname -f`.

Example:

```bash
cat /etc/hostname
cat /etc/hosts
hostnamectl
getent hosts $(hostname)
getent hosts $(hostname -f)
```

If the machine is part of a domain, ensure the domain suffix is correct and that the hostname matches the DNS A/AAAA record and reverse lookup expectations where required.

**Quick comparison of persistence files**

| Area | Common file or tool | Purpose |
|---|---|---|
| Timezone | `/etc/localtime` | Links system time display to a zoneinfo file |
| Timezone metadata | `/etc/timezone` | Stores timezone name on some distributions |
| Hardware clock | `hwclock` | Reads or writes RTC time |
| Locale | `/etc/default/locale` | Common Debian-based persistent locale file |
| Locale | `/etc/locale.conf` | Common system-wide locale file on systemd distros |
| Hostname | `/etc/hostname` | Stores the static hostname |
| Name resolution | `/etc/hosts` | Local hostname-to-IP mappings |

**Workflow summary**

Use a consistent workflow when configuring a Linux system:

1. Set the timezone with `timedatectl set-timezone` and verify `/etc/localtime`.
2. Confirm whether the RTC uses UTC or local time, then sync it with `hwclock` if needed.
3. Enable and verify NTP synchronization with `timedatectl`, `chronyc`, or `ntpq`.
4. Check installed locales with `locale -a` before setting a new one.
5. Persist locale settings with `localectl` or the appropriate distro file.
6. Set the static hostname with `hostnamectl` and confirm DNS and `/etc/hosts` resolution.
7. Watch for transient hostname changes from DHCP, cloud-init, or network management tools.

**Key takeaways**

- Time, locale, and hostname settings affect both user experience and system reliability.
- `timedatectl`, `localectl`, and `hostnamectl` are the main systemd-era management tools.
- Persistence matters: verify the underlying config files, not just the current session.
- Troubleshooting often requires checking both local files and network services.
