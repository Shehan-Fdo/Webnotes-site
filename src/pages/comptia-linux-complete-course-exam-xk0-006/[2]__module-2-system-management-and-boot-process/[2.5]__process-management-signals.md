## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Process fundamentals**

A Linux process is a running instance of a program with its own PID (process ID), memory space, open file descriptors, credentials, and scheduling state. The kernel creates processes with `fork()` and replaces a child’s program image with `execve()`-style calls. In day-to-day administration, you manage processes to diagnose hangs, stop runaway jobs, control services, and keep background work alive after logout.

Key relationships to remember:

- **PID** identifies a process.
- **PPID** identifies the parent process.
- **UID/GID** affect permissions and ownership of resources.
- **TTY** shows the controlling terminal, if any.
- **PGID** groups related processes for job control and signal delivery.
- **SID** identifies a session, which matters for terminal detachment and `SIGHUP` behavior.

A process can be CPU-bound, I/O-bound, sleeping, stopped, or a zombie. The state tells you whether the process is runnable, waiting, suspended, or already exited but not yet reaped.

**Viewing and interpreting process state**

Use `ps`, `top`, `htop`, and `/proc` to inspect process behavior. `ps` is best for snapshots; `top` is best for live monitoring.

```bash
ps -ef
ps aux
ps -o pid,ppid,pgid,sid,stat,tty,cmd -p 1234
top
```

Common `ps`/`top` state codes:

- **R**: running or runnable on the CPU.
- **S**: interruptible sleep, usually waiting for an event.
- **D**: uninterruptible sleep, often waiting on I/O; this can indicate storage or driver issues.
- **T**: stopped or traced, often from job control or debugging.
- **Z**: zombie; the process has exited, but the parent has not reaped its exit status.
- **I**: idle kernel thread on some systems and tools.

Practical interpretation:

- `R` is normal for active work.
- `S` is common for daemons and interactive programs waiting on input or timers.
- `D` often deserves investigation because the process cannot be interrupted normally until the kernel wait completes.
- `T` usually means `SIGSTOP`, `SIGTSTP`, or a debugger paused the task.
- `Z` means the process is dead but still occupies a PID entry until the parent calls a wait function.

Useful inspection commands:

```bash
ps -o pid,ppid,stat,cmd -p 4321
cat /proc/4321/status
cat /proc/4321/stat
```

**Finding and controlling processes**

Use `pgrep` and `pkill` when you want to match by name, user, or other attributes. Use `kill` when you already know the PID.

```bash
pgrep -a sshd
pkill -u alice firefox
kill 4321
kill -TERM 4321
kill -KILL 4321
```

Important points:

- `kill` sends a signal; it does not always terminate immediately.
- `SIGTERM` is the polite request to exit cleanly.
- `SIGKILL` cannot be caught, blocked, or ignored, so it is the last resort.
- `SIGSTOP` stops a process immediately and cannot be handled by the process.
- `SIGCONT` resumes a stopped process.

List signal names and numbers with:

```bash
kill -l
trap -l
man 7 signal
```

`kill -l` is especially useful on exams because it maps signal names to numbers on your system, which can vary across architectures.

**Signals and how they work**

A signal is an asynchronous notification delivered to a process or thread. Signals are used for termination, interruption, stopping, continuation, timer events, child status changes, and custom application behavior.

Common signals:

- **SIGHUP**: hangup; often used to tell daemons to reload configuration.
- **SIGINT**: interrupt from the keyboard, usually `Ctrl` + `C`.
- **SIGQUIT**: quit from the keyboard, often `Ctrl` + `\`, and may produce a core dump.
- **SIGTERM**: graceful termination request.
- **SIGKILL**: forced termination.
- **SIGSTOP**: unconditional stop.
- **SIGTSTP**: terminal stop, usually `Ctrl` + `Z`.
- **SIGCONT**: continue a stopped process.
- **SIGCHLD**: child state changed; parent may need to reap the child.
- **SIGUSR1`/`SIGUSR2`**: user-defined signals for application-specific behavior.

Send a signal with `kill`:

```bash
kill -HUP 1234
kill -TERM 1234
kill -STOP 1234
kill -CONT 1234
```

Send to a process group by using a negative PGID. This is important in shell workflows because pipelines and job-controlled tasks often share a process group.

```bash
# Send SIGTERM to process group 2345
kill -TERM -2345

# Send SIGINT to the current foreground job's process group from a shell
kill -INT -- -$(ps -o pgid= -p $$ | tr -d ' ')
```

Practical shell workflow examples:

- A background pipeline may contain multiple processes in one process group.
- `kill`ing the group stops the whole job, not just one command.
- This is useful when a pipeline hangs and you want to terminate every stage consistently.

Example:

```bash
yes | grep y | awk '{print $1}' &
jobs -l
ps -o pid,pgid,cmd --forest -g "$(ps -o pgid= -p $! | tr -d ' ')"
kill -TERM -- -$(ps -o pgid= -p $! | tr -d ' ')
```

**Job control in the shell**

Job control lets the shell manage foreground and background tasks. The shell tracks jobs by job number and process group, not just by PID.

Core commands:

```bash
sleep 100 &
jobs
fg %1
bg %1
disown %1
```

Keyboard shortcuts:

- `Ctrl` + `C` sends `SIGINT` to the foreground job.
- `Ctrl` + `Z` sends `SIGTSTP` to the foreground job.
- `Ctrl` + `\` sends `SIGQUIT`.

What happens in practice:

- A foreground job owns the terminal until it exits or is stopped.
- A background job continues running but cannot read from the terminal by default.
- `fg` brings a stopped or background job to the foreground.
- `bg` resumes a stopped job in the background.

**Detaching jobs from the terminal**

When you log out, the terminal may send `SIGHUP` to jobs associated with that session. To keep work running, detach it from the terminal or prevent hangup termination.

Common tools:

- **`nohup`**: runs a command immune to `SIGHUP` and redirects output to `nohup.out` unless you specify otherwise.
- **`disown`**: shell builtin that removes a job from the shell’s job table so the shell no longer manages it.
- **`setsid`**: starts a command in a new session, detaching it from the controlling terminal.

Examples:

```bash
nohup long_task.sh >task.log 2>&1 &
disown %1
setsid rsync -a /src /dest >/tmp/rsync.log 2>&1 < /dev/null &
```

Use cases:

- `nohup` is simple when you want a command to survive logout.
- `disown` is useful after you already started a background job in the current shell.
- `setsid` is useful when you want a clean session without a controlling terminal.

> `nohup` does not magically make a program daemon-like; if the program still reads from stdin or expects a terminal, you may need to redirect input and output explicitly.

**Daemonization and session behavior**

Traditional daemons detach from the terminal, create or join a new session, and run in the background. Modern systems often use `systemd` to supervise services, so manual daemonization is less common, but the concepts still matter.

Typical daemonization steps in classic Unix style:

1. `fork()` and let the parent exit.
2. Call `setsid()` to create a new session and detach from the controlling terminal.
3. Optionally `fork()` again so the process cannot reacquire a terminal.
4. Change working directory, set `umask`, and close inherited file descriptors as needed.

This matters because a process tied to a terminal can receive `SIGHUP` when the terminal closes, while a detached session is less likely to be affected.

**Blocking, masking, and pending signals**

A process can temporarily block signals so they are not delivered immediately. The kernel stores blocked signals in the signal mask. If a blocked signal arrives, it becomes pending until the process unblocks it.

Key concepts:

- **Signal mask**: the set of signals currently blocked.
- **Pending signal**: a signal that has been generated but not yet delivered.
- **`sigprocmask()`**: system call used to block, unblock, or set the signal mask.
- **`sigaction()`**: preferred API for installing signal handlers with precise control.
- **`sigpending()`**: checks which signals are pending.

Why this matters:

- Blocking is useful during critical sections where shared state must not be interrupted.
- Signals like `SIGKILL` and `SIGSTOP` cannot be blocked or handled.
- Standard signals are usually not queued multiple times; repeated delivery may collapse into one pending instance, while real-time signals are queued.

Conceptual example in C:

```c
#include <signal.h>
#include <stdio.h>
#include <unistd.h>

static void handler(int sig) {
    write(STDOUT_FILENO, "caught signal\n", 14);
}

int main(void) {
    struct sigaction sa = {0};
    sigemptyset(&sa.sa_mask);
    sa.sa_handler = handler;
    sa.sa_flags = SA_RESTART;
    sigaction(SIGINT, &sa, NULL);

    sigset_t set;
    sigemptyset(&set);
    sigaddset(&set, SIGINT);
    sigprocmask(SIG_BLOCK, &set, NULL);

    sleep(5);

    sigprocmask(SIG_UNBLOCK, &set, NULL);
    pause();
    return 0;
}
```

**Signal handling in applications**

A signal handler is code that runs when a signal is delivered. Handlers must be careful because they can interrupt normal execution at almost any point.

Best practices:

- Keep handlers short.
- Use only async-signal-safe functions inside handlers.
- Set a flag and let the main loop do the real work when possible.
- Use `sigaction()` instead of older `signal()` interfaces for predictable behavior.

Typical pattern:

```c
volatile sig_atomic_t stop_requested = 0;

void handle_term(int sig) {
    (void)sig;
    stop_requested = 1;
}
```

Then the main loop checks `stop_requested` and exits cleanly.

**Child processes and reaping**

When a process creates children, it must eventually collect their exit status. If it does not, exited children become zombies until reaped.

Relevant calls:

- **`wait()`**: waits for any child to change state.
- **`waitpid()`**: waits for a specific child, a process group, or any child with options.
- **`waitid()`**: more flexible interface that can wait for child state changes and provide detailed status information.
- **`WNOHANG`**: option that makes `waitpid()` return immediately if no child has changed state.

Behavior differences:

- `wait()` is simple but less selective.
- `waitpid()` is commonly used in servers and shells because it can target a specific child and avoid blocking with `WNOHANG`.
- `waitid()` is useful when you need finer-grained control over child state transitions and status reporting.

Example with `waitpid()`:

```c
pid_t pid = waitpid(-1, &status, WNOHANG);
if (pid == 0) {
    /* no child ready */
} else if (pid > 0) {
    /* child reaped */
}
```

Example with `waitid()`:

```c
siginfo_t info;
if (waitid(P_ALL, 0, &info, WEXITED | WNOHANG) == 0) {
    if (info.si_pid != 0) {
        /* child state changed */
    }
}
```

**Zombie and orphan processes**

A **zombie** is a child that has exited but still has an entry in the process table because the parent has not collected its exit status. Zombies do not consume CPU, but they do consume a PID slot and metadata.

An **orphan** is a running child whose parent has exited. The kernel re-parents orphans to PID 1 or to the system’s init-like subreaper, commonly `systemd`, which then becomes responsible for reaping them when they exit.

Important distinctions:

- A zombie is already dead.
- An orphan is still running.
- A zombie usually indicates a parent bug or failure to call a wait function.
- An orphan is normal in some workflows and is not inherently a problem.

Troubleshooting approach:

```bash
ps -o pid,ppid,stat,cmd -p 1234
ps -eo pid,ppid,stat,cmd | grep ' Z '
```

If you see zombies:

- Identify the parent process.
- Check whether the parent is stuck, crashed, or ignoring `SIGCHLD`.
- Restart or fix the parent if necessary.
- Killing the zombie itself does nothing because it is already dead.

**Process groups, sessions, and terminal control**

Process groups and sessions explain how shells manage jobs and how signals reach multiple related processes.

Definitions:

- **Process group**: a collection of processes that can receive signals together.
- **Session**: a collection of process groups, usually created by a login shell or `setsid`.
- **Controlling terminal**: the terminal associated with a session, if any.

Why this matters:

- Foreground jobs receive keyboard-generated signals from the terminal.
- Background jobs are separated from terminal input.
- A pipeline often shares one process group, so a single signal can affect all stages.

Useful inspection:

```bash
ps -o pid,ppid,pgid,sid,tty,stat,cmd -p $$
```

**Practical admin workflows**

Stopping a runaway process:

```bash
ps -eo pid,ppid,stat,cmd | grep '[f]fmpeg'
kill -TERM 24567
sleep 2
kill -KILL 24567
```

Restarting a hung background job safely:

```bash
jobs -l
bg %1
kill -CONT %1
```

Keeping a long task alive after logout:

```bash
nohup tar -czf backup.tgz /data >backup.log 2>&1 &
disown
```

Investigating a stuck I/O wait:

```bash
ps -o pid,stat,wchan,cmd -p 9876
top
```

If the state is `D`, look for storage, network filesystem, or driver issues rather than trying repeated `SIGKILL` attempts.

**Exam-oriented command summary**

- `ps` shows process snapshots.
- `top` shows live process activity.
- `kill -l` lists signals.
- `trap -l` lists signal names in shell contexts.
- `man 7 signal` documents signal semantics.
- `kill -TERM`, `kill -KILL`, `kill -STOP`, and `kill -CONT` are core signal commands.
- `nohup`, `disown`, and `setsid` help detach jobs from terminals.
- `wait`, `waitpid`, and `waitid` manage child exit status.
- `sigaction`, `sigprocmask`, and `sigpending` are core programming interfaces for signal handling.

**Quick comparison**

| Topic | Key idea | Typical use |
|---|---|---|
| `SIGTERM` | polite shutdown | normal service stop |
| `SIGKILL` | forced stop | last resort |
| `SIGSTOP` | immediate pause | suspend a task |
| `SIGCONT` | resume a task | restart stopped job |
| `nohup` | ignore hangup | survive logout |
| `disown` | remove shell job tracking | detach from shell |
| `setsid` | new session | fully detach from terminal |

**Common pitfalls**

- Sending `SIGKILL` first instead of trying `SIGTERM`.
- Confusing a zombie with a running orphan.
- Assuming `kill` only targets one process when a negative PGID targets a whole group.
- Forgetting that `D` state usually indicates uninterruptible I/O wait.
- Using `signal()` in new code instead of `sigaction()`.
- Expecting blocked signals to disappear instead of becoming pending.

**Minimal shell examples to remember**

```bash
# List signals
kill -l

# Stop and resume a job
sleep 100 &
kill -STOP %1
kill -CONT %1

# Detach a job
nohup myscript.sh >/tmp/myscript.log 2>&1 &
disown %1

# Inspect a process group
ps -o pid,ppid,pgid,sid,stat,cmd -p 1234
```
