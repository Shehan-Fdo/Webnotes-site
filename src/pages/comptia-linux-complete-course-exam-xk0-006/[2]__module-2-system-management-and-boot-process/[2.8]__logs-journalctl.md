## **Course:** CompTIA Linux+ Complete Course (Exam XK0-006)

**Logging fundamentals**

Linux systems generate logs from the kernel, services, authentication stack, and user applications. On modern distributions, the primary log collector is often **systemd-journald**, which stores structured events in the binary journal. Traditional text logs may still exist through **syslog**-style daemons such as `rsyslog` or `syslog-ng`, and many systems use both approaches together.

Logs help you troubleshoot boot failures, service crashes, authentication problems, storage issues, and network outages. For exam purposes, you should know where logs live, how to query them efficiently, and how retention and permissions affect access.

- `journalctl` reads the systemd journal.
- `/var/log/` commonly contains classic text logs such as `/var/log/messages`, `/var/log/syslog`, and `/var/log/auth.log`, depending on distribution.
- `dmesg` shows kernel ring buffer messages, which are useful for hardware and driver issues.
- `logger` can inject test messages into the journal or syslog pipeline.

**systemd-journald and the journal**

`systemd-journald` collects messages from the kernel, services, stdout/stderr streams, and other sources. The journal is structured, so you can filter by fields instead of only searching plain text. This is a major advantage over classic logs because it supports precise queries, boot scoping, time windows, and metadata such as unit name, PID, UID, and message IDs.

The journal can be stored in memory or on disk:

- **Volatile storage** lives under `/run/log/journal/` and disappears on reboot.
- **Persistent storage** lives under `/var/log/journal/` and survives reboot.
- If `/var/log/journal/` does not exist, many systems default to volatile storage unless configured otherwise.

To enable persistent storage on systems that do not already use it:

1. Create the directory:
   ```bash
   sudo mkdir -p /var/log/journal
   ```
2. Set ownership and permissions if needed:
   ```bash
   sudo systemd-tmpfiles --create --prefix /var/log/journal
   ```
3. Restart journald or reboot:
   ```bash
   sudo systemctl restart systemd-journald
   ```
4. Verify that logs now persist after reboot:
   ```bash
   journalctl --list-boots
   ```

> If the journal is only volatile, older entries vanish after reboot, which can make incident analysis difficult.

**Reading logs with journalctl**

`journalctl` is the main tool for querying the systemd journal. Without options, it shows the full log in a pager. In practice, you usually narrow the output by boot, time, unit, priority, or field.

Common display options:

- `-b` or `--boot` shows the current boot by default.
- `-b -1` shows the previous boot.
- `-f` follows new entries like `tail -f`.
- `-n 50` shows the last 50 entries.
- `--no-pager` disables paging for scripts.
- `-o` or `--output` changes formatting.

Useful output formats include:

- `short` for human-readable timestamps and concise lines.
- `short-precise` for higher timestamp precision.
- `short-monotonic` for elapsed time since boot, which is useful when correlating events during startup.
- `verbose` for full metadata, including all journal fields.
- `json` or `json-pretty` for machine parsing.
- `export` for a binary-safe, field-oriented export format that preserves all journal fields and is useful for archival or transfer.
- `cat` for message text only.

Examples:

```bash
journalctl -b -n 20
journalctl -u sshd --output=short-monotonic
journalctl -u nginx --output=export > nginx-journal.export
journalctl --output=json-pretty | less
```

Use `--output=short-monotonic` when you need to compare event ordering during boot or service startup. Use `--output=export` when you need a lossless, field-preserving representation for tooling or offline analysis.

**Field-based querying**

A key exam topic is journal field filtering. Journal fields use the form `FIELD=value`, and matching is exact for the field value. This is different from free-text searching.

Common fields include:

- `_SYSTEMD_UNIT=` for the systemd unit name
- `MESSAGE_ID=` for a specific structured event ID
- `_PID=` for process ID
- `_UID=` for user ID
- `_COMM=` for the executable name as reported by the kernel
- `_EXE=` for the full executable path
- `_TRANSPORT=` for the source, such as `kernel` or `syslog`
- `PRIORITY=` for severity

Examples of exact-match field queries:

```bash
journalctl _SYSTEMD_UNIT=sshd.service
journalctl MESSAGE_ID=fc2e22bc6ee647b6b90729ab34a250b1
journalctl _UID=1000
journalctl _COMM=sshd
journalctl _EXE=/usr/sbin/sshd
```

You can combine multiple field filters to narrow results further. Journalctl applies them as an intersection, so entries must satisfy all specified filters.

```bash
journalctl _SYSTEMD_UNIT=sshd.service _UID=1000
journalctl _COMM=sudo PRIORITY=3
journalctl _TRANSPORT=kernel _BOOT_ID=abcd1234...
```

This is especially useful when you want logs for one service and one user session, or one executable and one severity level.

**Searching message content**

Field filters are exact and structured, but you can also search message text. This is useful when you only know part of the error string.

- `-g` or `--grep` searches the message field with a regular expression.
- `-k` shows kernel messages.
- `-p` filters by priority.
- `-u` filters by unit.

Examples:

```bash
journalctl -g 'failed password'
journalctl -u sshd -g 'Accepted'
journalctl -p warning
journalctl -k -g 'ext4'
```

Use field filters when you know the source. Use `--grep` when you only know the text of the event.

**Boot, time, and session filtering**

Boot scoping is common in troubleshooting because many problems happen during startup.

- `journalctl -b` shows the current boot.
- `journalctl -b -1` shows the previous boot.
- `journalctl --list-boots` lists available boots with indexes and boot IDs.

Time filtering helps isolate incidents:

```bash
journalctl --since "2026-04-25 08:00:00" --until "2026-04-25 09:00:00"
journalctl --since yesterday
journalctl --since "1 hour ago"
```

For user-session analysis, combine user and session-related fields:

```bash
journalctl _UID=1000
journalctl _SYSTEMD_USER_UNIT=pipewire.service
journalctl _COMM=gnome-shell
```

This is useful when investigating desktop login failures, per-user services, or session-specific application crashes.

**Executable, unit, and message-ID examples**

Exam questions often ask you to identify the right filter for a scenario.

- To inspect a service, use `_SYSTEMD_UNIT=` or `-u`.
- To inspect a specific executable, use `_COMM=` or `_EXE=`.
- To inspect a structured event, use `MESSAGE_ID=`.

Practical examples:

```bash
journalctl -u chronyd
journalctl _COMM=cron
journalctl _EXE=/usr/bin/python3
journalctl MESSAGE_ID=39f53479d3a045ac8e11786248231fbf
```

`MESSAGE_ID=` is exact-match only and is commonly used by systemd components and applications that emit structured events. If you do not know the ID, search by text first, then refine with fields.

**Access control and permissions**

Reading the full journal is not always allowed for regular users. Access depends on distribution policy and group membership.

- Root can usually read all logs.
- Users in the `systemd-journal` group may read more journal data on many systems.
- Some logs may still be restricted to root, especially sensitive authentication or audit-related entries.
- Per-user journals may be readable by the owning user, but not by others.

Check group membership with:

```bash
id
groups
```

If needed, add a user to the journal-reading group:

```bash
sudo usermod -aG systemd-journal alice
```

Then log out and back in for the group change to take effect.

> Do not assume every user can read every log. Permission issues are a common reason `journalctl` appears to show incomplete data.

**File-based journal operations**

`journalctl` can read journal data from specific files or directories, which is useful for offline analysis, forensic work, or mounted disk images.

- `--file` reads a specific journal file.
- `--directory` reads journal files from a directory.

Examples:

```bash
journalctl --file=/var/log/journal/abcd1234/system.journal
journalctl --directory=/mnt/offline-root/var/log/journal
```

This is helpful when:

- You are analyzing logs from another system.
- You mounted a disk image and want to inspect its journal.
- You need to compare a rotated or archived journal file with the live system.

**Namespaces and per-user journals**

Advanced systemd setups may use **journal namespaces**, which isolate logs for different services or workloads. Namespaces can help separate noisy or sensitive logs, but they also mean you may need to query the correct namespace to see expected entries.

Per-user journals store logs for user sessions and user services. These are especially relevant on desktop systems and systems using `systemd --user`.

Key points:

- User journals can contain session services, desktop components, and user-level daemons.
- System journals contain system services and kernel messages.
- Namespace-aware setups may require explicit targeting depending on distribution and configuration.

Examples that may matter in advanced troubleshooting:

```bash
journalctl --user
journalctl --user -u pipewire
journalctl --namespace=webapp
```

If a log seems missing, verify whether you are looking at the system journal, a user journal, or a separate namespace.

**Classic logs, rotation, and retention**

Not all logs are in the journal. Traditional logging still matters, especially on distributions that use `rsyslog` or `syslog-ng` to write text files under `/var/log/`.

Common behaviors:

- Logs may be rotated by `logrotate`.
- Rotated logs may be compressed with `gzip`, `xz`, or `zstd`.
- File names often gain numeric suffixes such as `.1`, `.2`, or compressed extensions like `.gz`.
- Retention policies differ by distribution and service.

Examples:

```bash
ls -lh /var/log
zgrep -i ssh /var/log/auth.log*.gz
less /var/log/messages
```

Understand the interaction between journald and classic log rotation:

- The journal has its own retention controls based on size and age.
- `logrotate` manages text logs, not the binary journal.
- A system may keep both, but one source may have more history than the other.

**Retention and storage management**

Journal size can grow quickly on busy systems. Retention is controlled by journald configuration and available disk space.

Relevant settings in `/etc/systemd/journald.conf` include:

- `Storage=` to choose `volatile`, `persistent`, `auto`, or `none`
- `SystemMaxUse=` to cap total journal disk usage
- `SystemKeepFree=` to preserve free space
- `SystemMaxFileSize=` to limit individual journal files
- `MaxRetentionSec=` to limit age-based retention

Example configuration:

```ini
[Journal]
Storage=persistent
SystemMaxUse=500M
SystemKeepFree=1G
MaxRetentionSec=1month
```

After changing configuration, restart journald or reboot:

```bash
sudo systemctl restart systemd-journald
```

To inspect current journal usage:

```bash
journalctl --disk-usage
```

To vacuum old entries:

```bash
sudo journalctl --vacuum-time=2weeks
sudo journalctl --vacuum-size=500M
```

**Troubleshooting workflow**

A practical workflow helps you move from symptom to cause quickly.

1. Identify the affected service or subsystem.
2. Check the current boot with `journalctl -b`.
3. Narrow by unit, user, executable, or priority.
4. Search for the error text with `--grep`.
5. Compare with the previous boot if the issue happened during startup.
6. Check classic logs if the service writes there instead of the journal.

Example workflow for a failing SSH service:

```bash
systemctl status sshd
journalctl -u sshd -b
journalctl -u sshd -p warning
journalctl -u sshd --since "10 minutes ago"
journalctl -u sshd -g 'failed'
```

Example workflow for a boot issue:

```bash
journalctl -b -1 -p err
journalctl -b -1 --output=short-monotonic
journalctl --list-boots
```

**Common exam scenarios**

- You need logs from the previous boot: use `journalctl -b -1`.
- You need only kernel messages: use `journalctl -k`.
- You need a service’s logs: use `journalctl -u service.name`.
- You need a specific user’s activity: use `_UID=`.
- You need exact structured events: use `MESSAGE_ID=`.
- You need machine-readable output: use `--output=json` or `--output=export`.
- You need to follow live logs: use `journalctl -f`.
- You need logs from an offline system: use `--directory` or `--file`.

**Quick reference**

| Task | Command pattern | Notes |
|---|---|---|
| Show current boot | `journalctl -b` | Default troubleshooting view |
| Show previous boot | `journalctl -b -1` | Useful after failed startup |
| Follow logs live | `journalctl -f` | Similar to `tail -f` |
| Show service logs | `journalctl -u nginx` | Unit-based filtering |
| Search text | `journalctl -g 'error'` | Message content search |
| Show kernel logs | `journalctl -k` | Kernel ring buffer view |
| Show monotonic time | `journalctl -o short-monotonic` | Best for boot timing |
| Export fields | `journalctl -o export` | Lossless structured output |
| Read offline journal | `journalctl --directory=/mnt/...` | For mounted images |

**Key takeaways**

`journalctl` is the central tool for modern Linux log analysis. You should know how to filter by boot, time, unit, user, executable, and exact journal fields, and you should understand when to use different output formats. Also remember that access may be restricted, persistent storage may need to be enabled manually, and classic text logs still matter on many distributions.
