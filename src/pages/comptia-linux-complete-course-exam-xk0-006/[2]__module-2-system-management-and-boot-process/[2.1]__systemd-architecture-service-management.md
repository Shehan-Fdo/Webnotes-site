**systemd overview**

`systemd` is the init system and service manager used by most modern Linux distributions. It starts the system during boot, manages services and other resources after boot, and tracks dependencies between components. It replaces older SysV init workflows with a dependency-driven model built around units, targets, and jobs.

At a high level, `systemd` does three important things:

- **Boot orchestration**: it brings the system up in a controlled order.
- **Service supervision**: it starts, stops, restarts, and monitors daemons.
- **Resource activation**: it can activate services on demand through sockets, paths, timers, and other triggers.

`systemd` is not only about services. It also manages mount points, devices, sockets, timers, slices, and targets. For exam purposes, think of it as the central control plane for system startup and runtime service management.

**Core architecture**

`systemd` uses a unit-based model. A unit is a named object that describes a resource or action. Common unit types include:

- `.service` for daemons and background processes
- `.socket` for socket activation
- `.path` for path-based activation
- `.timer` for scheduled activation
- `.target` for grouping and synchronization points
- `.mount` for mount points
- `.device` for kernel device nodes
- `.slice` for cgroup-based resource grouping

Units are stored in standard locations such as:

- `/usr/lib/systemd/system/` for vendor-provided unit files
- `/etc/systemd/system/` for administrator overrides and custom units
- `/run/systemd/system/` for runtime-generated units and transient changes

The search order matters. Files in `/etc/systemd/system/` override vendor files in `/usr/lib/systemd/system/`. This is why drop-ins and custom unit files belong under `/etc/systemd/system/`.

**Unit files and service directives**

A unit file is an INI-style configuration file. Service units commonly contain `[Unit]`, `[Service]`, and `[Install]` sections.

Important service directives include:

- **`ExecStart=`**: command used to start the service
- **`ExecStartPre=`**: commands run before the main start command
- **`ExecStartPost=`**: commands run after the service starts
- **`ExecReload=`**: command used when the service is reloaded
- **`ExecStop=`**: command used when the service is stopped
- **`User=`**: user account the service runs as
- **`Group=`**: group account the service runs as
- **`Restart=`**: restart policy for failures
- **`Type=`**: startup behavior, such as `simple`, `forking`, or `notify`

A minimal example:

```ini
[Unit]
Description=Example web service
After=network-online.target

[Service]
Type=simple
User=webapp
Group=webapp
ExecStartPre=/usr/bin/test -f /etc/webapp/config.yml
ExecStart=/usr/local/bin/webapp --config /etc/webapp/config.yml
ExecReload=/bin/kill -HUP $MAINPID
ExecStop=/usr/bin/pkill -TERM -u webapp webapp
Restart=on-failure

[Install]
WantedBy=multi-user.target
```

`ExecStartPre=` and `ExecStartPost=` are useful when you need validation or setup around the main process. `ExecReload=` is used when the daemon supports re-reading configuration without a full restart. `ExecStop=` is useful when the service needs a clean shutdown action beyond a simple signal.

**Service states and status fields**

`systemctl status` is one of the most useful troubleshooting commands because it shows the current state, recent log lines, and the main process information.

Common service states include:

- **`active`**: the unit is running or available
- **`inactive`**: the unit is not running
- **`failed`**: the unit exited unsuccessfully or hit a start failure
- **`activating`**: the unit is in the process of starting
- **`deactivating`**: the unit is stopping

You may also see substates such as:

- `running`
- `exited`
- `dead`
- `waiting`
- `listening`
- `plugged`

These fields help you distinguish between a service that is truly running and one that is merely enabled or installed.

Useful inspection commands:

```bash
systemctl status sshd
systemctl show sshd -p ActiveState -p SubState -p FragmentPath -p UnitFileState
systemctl cat sshd
```

Use them for different purposes:

- **`systemctl status`**: human-readable runtime status and recent journal output
- **`systemctl show`**: machine-readable property dump for scripting or precise inspection
- **`systemctl cat`**: the full unit file plus any drop-in overrides

**Enablement, presets, and vendor policy**

A unit can be installed, enabled, disabled, masked, or static. These are different concepts from runtime state.

- **enabled**: starts automatically at boot through symlinks in target directories
- **disabled**: installed but not configured to start automatically
- **masked**: linked to `/dev/null` so it cannot be started
- **static**: has no `[Install]` section and is usually started by dependency only

To inspect unit file states, use:

```bash
systemctl list-unit-files --state=enabled,disabled,masked
systemctl is-enabled sshd
systemctl is-enabled chronyd
```

Vendor default enablement policies are applied with presets. This is common in package installation workflows and image builds.

```bash
systemctl preset sshd.service
systemctl preset-all
```

- **`systemctl preset`** applies the preset policy to a specific unit.
- **`systemctl preset-all`** applies preset rules to all installed units.

Presets are useful when you want the system to follow distribution defaults rather than your own manual enable/disable choices.

**Starting, stopping, restarting, and reloading**

Basic service control uses `systemctl`:

```bash
systemctl start nginx
systemctl stop nginx
systemctl restart nginx
systemctl reload nginx
systemctl reload-or-restart nginx
```

Use the right action for the change you made:

- **`restart`**: stop and start the service again
- **`reload`**: ask the service to re-read configuration without full restart, if supported
- **`reload-or-restart`**: reload when possible, otherwise restart

A key distinction is between `systemctl daemon-reload` and `systemctl reload`:

- **`systemctl daemon-reload`** tells `systemd` to re-read unit files and drop-ins after you change service definitions.
- **`systemctl reload`** tells the service process itself to reload its runtime configuration.

Example:

1. You edit `/etc/systemd/system/nginx.service.d/override.conf`.
2. Run `systemctl daemon-reload` so `systemd` notices the unit change.
3. Run `systemctl restart nginx` if the service needs a new process to use the updated unit settings.

Another example:

1. You change `/etc/nginx/nginx.conf`.
2. Run `systemctl reload nginx` if NGINX supports reload.
3. Use `restart` only if reload is unsupported or the change requires a new process.

> `daemon-reload` does not restart services. It only refreshes `systemd`'s view of unit definitions.

**Jobs and the transaction model**

When you ask `systemd` to do something, it creates a job. Jobs are queued actions such as start, stop, restart, reload, or isolate. `systemd` resolves dependencies and builds a transaction so related units are handled in the correct order.

Inspect queued jobs with:

```bash
systemctl list-jobs
```

This is useful when a service appears stuck in `activating` or when a dependency chain delays startup. If a job is waiting on another unit, `list-jobs` helps you see the active transaction.

**Dependencies and reverse dependency tracing**

`systemd` uses dependency relationships to decide what should start before or after a unit. Common dependency directions include `Requires=`, `Wants=`, `After=`, and `Before=`.

To see what a unit depends on:

```bash
systemctl list-dependencies sshd
```

To see what depends on a unit, use reverse dependency tracing:

```bash
systemctl list-dependencies --reverse network-online.target
```

This is especially useful when troubleshooting why stopping one unit affects others, or when you need to understand what will be impacted by a target change.

**Targets and boot flow**

Targets are synchronization points that group units into a boot state. They are similar to runlevels in concept, but more flexible.

Common targets include:

- **`multi-user.target`**: non-graphical multi-user system
- **`graphical.target`**: multi-user system with graphical login
- **`rescue.target`**: single-user maintenance mode with basic services
- **`emergency.target`**: minimal environment with very limited services
- **`default.target`**: the default boot target, usually a symlink to another target

Check or change the default target:

```bash
systemctl get-default
systemctl set-default multi-user.target
```

Use `isolate` carefully:

```bash
systemctl isolate rescue.target
```

`isolate` starts the target and stops units that are not part of it. This can terminate active services and user sessions. It differs from `set-default`, which only changes the target used on the next boot.

Operationally:

- **`rescue.target`** usually provides a maintenance shell and mounts local filesystems, so you can repair the system with more access.
- **`emergency.target`** is more restrictive and may provide only a root shell with minimal mounts and services.

> `isolate` can disrupt remote access and running workloads. Use it only when you understand which units will be stopped.

**Socket activation and path units**

`systemd` can start services on demand. This reduces boot time and allows services to remain inactive until needed.

Socket activation uses a `.socket` unit paired with a `.service` unit. `systemd` listens on the socket and starts the service when a connection arrives.

Example `.socket` unit:

```ini
[Unit]
Description=Demo socket activation

[Socket]
ListenStream=127.0.0.1:9090
Accept=no

[Install]
WantedBy=sockets.target
```

Example matching service:

```ini
[Unit]
Description=Demo socket service

[Service]
ExecStart=/usr/local/bin/demo-server
```

Path units watch files or directories and trigger a service when a path changes.

Example `.path` unit:

```ini
[Unit]
Description=Watch config directory

[Path]
PathModified=/etc/demo/config.yml

[Install]
WantedBy=multi-user.target
```

Matching service:

```ini
[Unit]
Description=Reload demo service

[Service]
Type=oneshot
ExecStart=/usr/bin/systemctl reload demo.service
```

Path units are useful for automation such as config-driven reloads, file drop processing, or event-based workflows.

**Template units and instance units**

Template units let you define one reusable unit file for multiple instances. The template name contains `@`, such as `[email protected]`. An instance adds a specific identifier, such as `[email protected]`.

Example:

```bash
systemctl start [email protected]
systemctl enable [email protected]
```

A template unit file might look like this:

```ini
[Unit]
Description=Worker instance %i

[Service]
ExecStart=/usr/local/bin/worker --id %i
Restart=on-failure
```

Common specifiers include:

- **`%i`**: instance name
- **`%n`**: full unit name
- **`%p`**: prefix before `@`

Templates are common for running multiple similar services with different ports, tenants, or roles.

**Overrides, drop-ins, and precedence**

You usually do not edit vendor unit files directly. Instead, you create overrides. This keeps package updates from overwriting your changes.

There are two main approaches:

- **`systemctl edit unit`** creates a drop-in override file under `/etc/systemd/system/unit.d/override.conf`
- **`systemctl edit --full unit`** opens a full copy of the unit file for editing

Use them differently:

- **`systemctl edit`** when you want to change only a few directives and preserve vendor updates
- **`systemctl edit --full`** when you need to replace the entire unit definition or make broad changes

Example drop-in:

```ini
[Service]
Restart=always
RestartSec=5
```

Drop-in precedence works like this:

1. Vendor unit file in `/usr/lib/systemd/system/`
2. Administrator drop-ins in `/etc/systemd/system/unit.d/*.conf`
3. Full override unit in `/etc/systemd/system/unit`

To remove or disable overrides:

- Delete the drop-in file from `/etc/systemd/system/unit.d/`
- Remove the full override file if one exists
- Run `systemctl daemon-reload`
- Restart the service if the runtime process must pick up the change

You can also use:

```bash
systemctl revert nginx
```

This removes local overrides and restores the vendor unit view.

**Troubleshooting workflow**

A practical troubleshooting sequence helps you isolate whether the problem is with the unit file, the service process, dependencies, or the boot environment.

1. Check the runtime state:
   ```bash
   systemctl status nginx
   ```
2. Inspect logs:
   ```bash
   journalctl -u nginx -b
   ```
3. Verify the unit definition and overrides:
   ```bash
   systemctl cat nginx
   systemctl show nginx -p FragmentPath -p DropInPaths
   ```
4. Check whether the unit is enabled, masked, or static:
   ```bash
   systemctl is-enabled nginx
   systemctl list-unit-files --state=enabled,disabled,masked
   ```
5. Clear a failed state after fixing the issue:
   ```bash
   systemctl reset-failed nginx
   ```

`reset-failed` clears the failed status so the unit can be started again cleanly. It does not fix the underlying problem; it only resets the state marker.

**Practical service management examples**

Start and enable a daemon:

```bash
systemctl enable --now sshd
```

Disable and stop a daemon:

```bash
systemctl disable --now cups
```

Mask a service to prevent any start attempt:

```bash
systemctl mask avahi-daemon
systemctl unmask avahi-daemon
```

Check whether a service is active and enabled:

```bash
systemctl is-active sshd
systemctl is-enabled sshd
```

Inspect all loaded units of a type:

```bash
systemctl list-units --type=service
systemctl list-units --type=socket
```

**Shutdown, rescue, and emergency operations**

Targets are important for recovery and maintenance. If the system is unstable, you may need to boot into a reduced environment.

- **`rescue.target`**: intended for maintenance with local filesystems mounted and a root shell available
- **`emergency.target`**: minimal recovery mode with the smallest possible set of services and mounts

Typical use cases:

- Use `rescue.target` when you need package repair, filesystem checks, or configuration edits with basic system access.
- Use `emergency.target` when normal boot fails very early or when filesystem issues prevent standard services from starting.

You can switch to a target temporarily with `isolate`, but on production systems it is often safer to set the default target for the next boot or use bootloader kernel parameters when recovery is needed.

**Exam-focused distinctions**

Keep these differences clear:

- **Enable vs start**: enable affects boot-time behavior; start affects the current session.
- **Reload vs restart**: reload keeps the process running if supported; restart replaces it.
- **daemon-reload vs reload**: daemon-reload refreshes unit definitions; reload refreshes service configuration.
- **set-default vs isolate**: set-default changes the next boot target; isolate changes the current runtime target and stops unrelated units.
- **status vs show vs cat**: status is human-friendly runtime state; show is property-based; cat displays the unit file and drop-ins.
- **preset vs enable**: preset applies vendor policy; enable applies your explicit admin choice.
