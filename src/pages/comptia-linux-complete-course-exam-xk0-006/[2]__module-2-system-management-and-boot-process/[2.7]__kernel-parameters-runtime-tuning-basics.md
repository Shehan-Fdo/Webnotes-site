**Kernel parameters: what they are and why they matter**

Kernel parameters are settings that influence how the Linux kernel behaves. You use them to adjust boot-time behavior, hardware handling, memory management, networking, security controls, and other low-level runtime characteristics.

These settings matter because they let you:

- Fix hardware or driver issues without recompiling the kernel.
- Tune performance for specific workloads.
- Change security-related behavior such as address-space protections or module loading rules.
- Make temporary tests before committing changes permanently.

A useful mental model is:

- **Boot-time parameters** are passed to the kernel by the bootloader.
- **Runtime parameters** are changed after boot, usually through `/proc`, `/sys`, or `sysctl`.
- **Persistent tuning** is stored in configuration files and applied at boot or during early userspace startup.

**Boot-time kernel parameters**

Boot-time parameters are most often used when the kernel must know something before it initializes hardware or mounts filesystems. Common examples include selecting a root device, enabling debug output, or disabling a problematic feature.

Typical places to set them include:

- GRUB 2 configuration, usually through `/etc/default/grub`
- Bootloader menu entries
- Kernel command line in recovery or rescue environments

Example GRUB configuration:

```bash
GRUB_CMDLINE_LINUX="quiet splash mitigations=auto"
```

After editing GRUB settings, you usually regenerate the bootloader config:

```bash
sudo update-grub
```

On some distributions, the equivalent command is:

```bash
sudo grub2-mkconfig -o /boot/grub2/grub.cfg
```

Common boot-time parameters include:

- `ro` or `rw` to control initial root filesystem mount mode.
- `quiet` to reduce boot messages.
- `debug` to increase kernel logging.
- `systemd.unit=rescue.target` to boot into rescue mode on systemd-based systems.
- `init=/bin/bash` for emergency recovery when the normal init process fails.

> Use boot-time parameters carefully. A wrong kernel command line can prevent a system from booting normally, so test changes in a controlled environment first.

**Viewing the current kernel command line**

You can inspect the active boot parameters from `/proc/cmdline`:

```bash
cat /proc/cmdline
```

This is useful when you need to confirm what the bootloader actually passed to the kernel.

**Runtime tuning with `sysctl`**

`sysctl` manages kernel parameters exposed through the `/proc/sys` interface. These are often called **sysctl variables** or **kernel tunables**. They affect the running system and are commonly used for networking, memory behavior, file handling, and security limits.

View all current tunables:

```bash
sysctl -a
```

Query a specific value:

```bash
sysctl net.ipv4.ip_forward
```

Set a value temporarily:

```bash
sudo sysctl -w net.ipv4.ip_forward=1
```

The equivalent `/proc` write is:

```bash
echo 1 | sudo tee /proc/sys/net/ipv4/ip_forward
```

`sysctl` is preferred because it is clearer, logs the change more cleanly, and supports configuration files.

**Common runtime tuning examples**

A few frequently encountered tunables include:

- `net.ipv4.ip_forward` enables IPv4 forwarding for routing or NAT use cases.
- `vm.swappiness` influences how aggressively the kernel swaps memory pages.
- `fs.file-max` sets the system-wide file descriptor ceiling.
- `kernel.hostname` changes the system hostname at runtime.
- `net.core.somaxconn` affects the listen backlog ceiling for sockets.

Example:

```bash
sudo sysctl -w vm.swappiness=10
sudo sysctl -w fs.file-max=2097152
```

These changes are temporary unless you persist them in a configuration file.

**Persistent sysctl configuration**

To make tunables survive reboot, place them in a sysctl configuration file. Modern distributions commonly use `/etc/sysctl.d/*.conf` drop-ins, while `/etc/sysctl.conf` still exists for compatibility.

Example drop-in file:

```bash
sudo tee /etc/sysctl.d/99-custom.conf >/dev/null <<'EOF'
net.ipv4.ip_forward = 1
vm.swappiness = 10
fs.file-max = 2097152
EOF
```

Apply the configuration immediately:

```bash
sudo sysctl --system
```

`sysctl --system` loads sysctl settings from multiple locations in a defined order, including system defaults and drop-in directories. It is broader than loading a single file.

If you want to load one file directly, use:

```bash
sudo sysctl -p /etc/sysctl.d/99-custom.conf
```

> Be precise about scope: `sysctl -p` loads a specific file, while `sysctl --system` processes the full sysctl configuration stack.

**Order of precedence and file layout**

Sysctl files are typically read in an ordered sequence. Later files can override earlier ones, which is why administrators often use a high-numbered filename such as `99-custom.conf`.

Typical pattern:

- Vendor defaults in `/usr/lib/sysctl.d/`
- Local overrides in `/etc/sysctl.d/`
- Legacy settings in `/etc/sysctl.conf`

Practical guidance:

- Use `/etc/sysctl.d/` for custom settings.
- Use descriptive filenames with numeric prefixes.
- Avoid editing vendor files directly unless the distribution explicitly recommends it.

**Inspecting and validating tunables**

Before changing a setting, verify the current value and understand the expected range.

Examples:

```bash
sysctl net.ipv4.ip_forward
sysctl vm.swappiness
sysctl fs.file-max
```

You can also inspect the `/proc/sys` tree directly:

```bash
ls /proc/sys/net/ipv4
cat /proc/sys/vm/swappiness
```

If a setting does not appear, it may be:

- Unsupported by the running kernel
- Controlled by a module that is not loaded
- Named differently than expected
- Restricted by permissions

**Kernel modules and module parameters**

Some runtime behavior is controlled by kernel module parameters rather than global sysctl values. These parameters are specific to the module and driver.

Check available parameters with:

```bash
modinfo e1000e
```

Look for the `parm:` lines in the output. The parameter names and valid values are driver-specific, so you must verify them for the exact module you are tuning.

Example of a module parameter configuration file:

```bash
echo "options e1000e InterruptThrottleRate=3000" | sudo tee /etc/modprobe.d/e1000e.conf
```

This is only valid if the `e1000e` driver supports that parameter and value on your system. Do not assume a parameter name is universal across drivers.

To load or reload a module after changing parameters, you may need to:

1. Unload the module if it is safe to do so.
2. Reload it with the new options.
3. Reboot if the module is in active use and cannot be safely removed.

Example:

```bash
sudo modprobe -r e1000e
sudo modprobe e1000e
```

**Runtime tuning versus persistent tuning**

Use temporary changes when you are testing behavior or troubleshooting. Use persistent changes when you are confident the setting is correct and should survive reboot.

| Goal | Best approach | Example |
|---|---|---|
| Test a networking change | Temporary `sysctl -w` | `sysctl -w net.ipv4.ip_forward=1` |
| Keep a tuning across reboots | `/etc/sysctl.d/*.conf` | `99-custom.conf` |
| Adjust a driver-specific option | `/etc/modprobe.d/*.conf` | `options e1000e ...` |
| Change boot behavior | Bootloader kernel command line | `quiet` or `systemd.unit=rescue.target` |

**Networking-related tuning**

Networking is one of the most common areas for kernel tuning. You may enable forwarding, adjust connection backlog limits, or tune TCP behavior for a server workload.

Example NAT or router-style setup:

```bash
sudo sysctl -w net.ipv4.ip_forward=1
```

If you are building a router or firewall, you often pair forwarding with packet filtering rules. On modern Linux systems, nftables is the default firewall framework on many distributions. Legacy `iptables` syntax may still work through compatibility layers, but it is not the preferred default on current systems.

Example legacy compatibility NAT rule:

```bash
sudo iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE
```

Use this only when you know your environment relies on `iptables` compatibility. For new deployments, prefer nftables where appropriate.

**Memory and VM tuning**

Virtual memory tunables affect how the kernel uses RAM and swap. These settings can help with latency-sensitive services, database hosts, or systems with constrained memory.

Common examples:

- `vm.swappiness`
  - Lower values reduce swap usage.
  - Higher values make the kernel more willing to move pages to swap.
- `vm.vfs_cache_pressure`
  - Influences reclaim pressure on inode and dentry caches.
- `vm.dirty_ratio` and `vm.dirty_background_ratio`
  - Affect how much dirty data can accumulate before writeback.

Example:

```bash
sudo sysctl -w vm.vfs_cache_pressure=50
sudo sysctl -w vm.dirty_background_ratio=5
```

These values are workload-dependent. A database server, a desktop, and a virtualization host may need very different tuning.

**Security-related kernel settings**

Some kernel parameters harden the system by reducing attack surface or limiting risky behavior.

Examples include:

- Disabling IP forwarding on non-router hosts.
- Restricting unprivileged access to certain kernel features.
- Controlling module loading policy.
- Adjusting kernel pointer exposure or logging behavior.

A common security practice is to keep only the tunables you need and document why they exist. Over-tuning can create instability or weaken security if the wrong value is chosen.

**Troubleshooting workflow**

When a tuning change does not behave as expected, use a structured approach:

1. Confirm the current value with `sysctl` or `/proc/sys`.
2. Check whether the change is temporary or persistent.
3. Verify the correct file location and filename order.
4. Confirm the setting is valid for the running kernel or module.
5. Review logs for errors or warnings.

Useful commands:

```bash
sysctl net.ipv4.ip_forward
journalctl -k
dmesg | tail -n 50
```

If a module parameter does not apply, check whether the module is loaded and whether the parameter name is correct:

```bash
lsmod | grep e1000e
modinfo e1000e | grep -i parm
```

**Practical examples**

Enable IPv4 forwarding temporarily:

```bash
sudo sysctl -w net.ipv4.ip_forward=1
```

Persist the same setting:

```bash
echo "net.ipv4.ip_forward = 1" | sudo tee /etc/sysctl.d/99-ip-forward.conf
sudo sysctl --system
```

Lower swap aggressiveness for a server:

```bash
sudo sysctl -w vm.swappiness=10
echo "vm.swappiness = 10" | sudo tee /etc/sysctl.d/99-swappiness.conf
sudo sysctl --system
```

Check a boot-time parameter:

```bash
cat /proc/cmdline
```

Inspect a driver’s supported options:

```bash
modinfo e1000e
```

**Exam focus summary**

For Linux+ exam prep, remember these core distinctions:

- **Kernel command line parameters** are set at boot and affect early initialization.
- **sysctl tunables** change runtime kernel behavior and can be made persistent with config files.
- **Module parameters** are driver- or module-specific and must be verified with `modinfo`.
- **Temporary changes** are best for testing.
- **Persistent changes** belong in `/etc/sysctl.d/` or `/etc/modprobe.d/` depending on the setting.

A strong answer on the exam usually shows that you know not just the command, but also when to use it and what scope it affects.
