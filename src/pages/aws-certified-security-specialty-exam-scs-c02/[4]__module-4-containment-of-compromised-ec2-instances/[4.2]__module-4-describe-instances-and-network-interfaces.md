## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**EC2 instances and network interfaces**

Amazon EC2 (Elastic Compute Cloud) instances are the compute layer, but most networking and security questions in this topic center on the <abbr title="Elastic Network Interface">ENI</abbr>. An ENI is a virtual network card that you attach to an instance inside a specific subnet and Availability Zone (<abbr title="Availability Zone">AZ</abbr>). It carries the instance’s private IPv4 address, optional secondary private IPv4 addresses, IPv6 addresses, security groups, and attachment metadata.

For exam purposes, think of the instance as the compute container and the ENI as the network identity. Many security and failover designs are really ENI designs.

- **Primary ENI** is created when the instance launches and is attached as device index `0`.
- **Secondary ENIs** are added later and can be detached and reattached more flexibly.
- **Subnet and AZ** placement is fixed for an ENI; you cannot attach it to an instance in a different AZ.
- **Security groups** are associated with the ENI, not the instance itself.
- **Private IPs** belong to the ENI and stay in the same subnet.

**Instance lifecycle and ENI behavior**

The primary network interface is created with the instance and is the default interface used for the instance’s main network path. Operationally, it differs from secondary ENIs because it is tied to the instance launch configuration and device index `0`. Whether it is deleted when the instance terminates depends on the attachment’s `DeleteOnTermination` setting.

Secondary ENIs are separate resources. By default, they are preserved when the instance terminates unless you explicitly delete them. This distinction is a common exam trap: do not assume all ENIs disappear with the instance.

- **Primary ENI**
  - Attached at launch.
  - Usually the default route for the instance’s traffic.
  - Deletion on termination depends on `DeleteOnTermination`.
- **Secondary ENI**
  - Created independently.
  - Can be detached and reattached to another compatible instance in the same AZ.
  - Usually preserved after instance termination unless deleted.

> The ENI itself is not “the instance.” If you terminate an instance, the instance goes away, but the ENI may remain depending on its lifecycle settings and whether it is secondary.

**Key ENI attributes**

When you inspect an ENI, several fields matter for security, troubleshooting, and exam questions:

- **Description** helps identify the purpose of the interface, such as `NAT appliance` or `web-tier-eni`.
- **Attachment status** shows whether the ENI is attached, available, or in transition.
- **Requester-managed** indicates the ENI was created by an AWS service, such as a load balancer or managed service.
- **DeleteOnTermination** controls whether the ENI is removed when the attached instance terminates.
- **Source/destination check** controls whether the instance can forward traffic not addressed to itself.
- **Private IPs and IPv6 addresses** show the actual addresses assigned to the interface.

A practical inspection command:

```bash
aws ec2 describe-network-interfaces \
  --filters Name=attachment.instance-id,Values=i-0123456789abcdef0
```

To inspect a specific ENI in more detail:

```bash
aws ec2 describe-network-interfaces \
  --network-interface-ids eni-0123456789abcdef0
```

**Primary vs secondary private IPs**

An ENI can have one primary private IPv4 address and zero or more secondary private IPv4 addresses. The primary private IP is the main address for that interface. Secondary private IPs are useful for multi-tenant services, blue/green cutovers, and failover patterns.

Important distinctions:

- **Primary private IP**
  - Assigned automatically when the ENI is created.
  - Stays with that ENI unless the ENI is replaced.
  - Often used by the instance’s default hostname and internal identity.
- **Secondary private IP**
  - Can be assigned and reassigned within the same subnet.
  - Useful for moving service identity without moving the whole ENI.
  - Must remain in the same subnet when reassigned.

A common exam distinction is that you can move a secondary private IP between ENIs in the same subnet, but you cannot move an ENI across subnets. The ENI is bound to its subnet and AZ.

**IPv4, IPv6, and public addressing**

IPv4 and IPv6 behave differently in AWS networking, and exam questions often test the difference.

- **Private IPv4**
  - Lives on the ENI.
  - Can have a primary and secondary addresses.
  - Used for internal routing and security group evaluation.
- **Elastic IP address (EIP)**
  - A public IPv4 address that you can remap.
  - Can be associated with an instance or directly with an ENI.
  - Reassociation is a common failover mechanism.
- **IPv6**
  - Assigned to the ENI, not “NATed” in the same way as IPv4.
  - There is no direct equivalent of an EIP for IPv6 in the same sense as public IPv4 remapping.
  - Security groups and route tables still control exposure.

For public IPv4, you often associate an EIP to the ENI so the public address follows the network interface rather than the instance identity. This is useful when you want to preserve a stable public endpoint during instance replacement.

Example EIP association at the ENI level:

```bash
aws ec2 associate-address \
  --allocation-id eipalloc-0123456789abcdef0 \
  --network-interface-id eni-0123456789abcdef0
```

If you associate an EIP to an instance, AWS still resolves that to the underlying primary ENI. For exam purposes, remember that the ENI is the actual attachment point.

**Source/destination check**

The source/destination check is critical for instances that forward traffic. By default, EC2 instances drop packets that are not addressed to their own IPs. That is correct for normal servers, but it breaks NAT instances, routing appliances, firewalls, and some inspection designs.

You disable source/destination check when the instance must act as a router or NAT device.

```bash
aws ec2 modify-network-interface-attribute \
  --network-interface-id eni-0123456789abcdef0 \
  --no-source-dest-check
```

Use cases:

- **NAT instance** forwards private subnet traffic to the internet.
- **Inspection appliance** routes traffic through a security control point.
- **Transit or failover appliance** handles traffic for other hosts.

> If source/destination check remains enabled on a forwarding instance, traffic may fail in ways that look like routing or security group problems.

**Security groups and network controls**

Security groups are stateful virtual firewalls attached to ENIs. Because they attach to the ENI, not the instance, moving an ENI also moves its security group context.

Practical implications:

- A replacement instance can inherit the same security posture if you attach the same ENI.
- Multiple ENIs on one instance can have different security groups.
- Troubleshooting should include both the instance route table and the ENI’s security groups.

Example of checking security groups on an ENI:

```bash
aws ec2 describe-network-interfaces \
  --network-interface-ids eni-0123456789abcdef0 \
  --query 'NetworkInterfaces[0].Groups'
```

**Attachment rules and lifecycle constraints**

ENI attachment has strict placement rules:

- The ENI and the target instance must be in the same **AZ**.
- The ENI must be in the same **subnet** as the target instance if you are reusing the same private IP identity.
- You cannot attach an ENI across AZs.
- You cannot “move” an ENI to another subnet; you must create a new ENI in the target subnet if needed.

This is one of the most common exam traps. If the standby instance is in a different AZ, you cannot attach the same ENI to it. In that case, you typically use an EIP remap, a load balancer, or a new ENI in the target AZ.

**Failover and reassociation patterns**

There are two different failover ideas that are easy to confuse:

1. Move the **ENI** to a standby instance in the same AZ.
2. Reassign a **secondary private IP** to another ENI in the same subnet.

These are not the same operation.

- **Move the ENI**
  - Preserves the ENI’s private IPs, security groups, and attachment identity.
  - Requires the standby instance to be in the same AZ.
  - Useful for stateful appliances or bastion replacement.
- **Move a secondary private IP**
  - Preserves the IP identity but not the whole ENI.
  - Requires the target ENI to be in the same subnet.
  - Useful for application failover where the service listens on a specific private address.

A simple comparison:

| Item | Can move across AZs? | Can move across subnets? | Typical use |
|---|---:|---:|---|
| Primary private IPv4 | No | No | Instance identity |
| Secondary private IPv4 | No | No | Service failover within subnet |
| ENI | No | No | Stateful network identity |
| Elastic IP | Yes | Yes | Public IPv4 failover |

**CLI workflow for ENI failover**

A safe workflow is to identify the attachment, detach the ENI, and then attach it to the standby instance in the same AZ.

1. Find the ENI and its attachment ID.
2. Detach the ENI from the current instance.
3. Attach the ENI to the standby instance.
4. Verify routes, security groups, and source/destination check settings.

Example:

```bash
aws ec2 describe-network-interfaces \
  --network-interface-ids eni-0123456789abcdef0 \
  --query 'NetworkInterfaces[0].Attachment.AttachmentId'
```

Then detach:

```bash
aws ec2 detach-network-interface \
  --attachment-id eni-attach-0123456789abcdef0
```

Then attach to the standby instance:

```bash
aws ec2 attach-network-interface \
  --network-interface-id eni-0123456789abcdef0 \
  --instance-id i-0fedcba9876543210 \
  --device-index 1
```

The `device-index` matters. The primary ENI is device index `0`; secondary ENIs use higher indexes. If you are replacing the primary network path, you must understand how the instance was launched and whether the target instance can accept that attachment pattern.

**NAT instance and inspection appliance example**

A common real-world design uses a dedicated EC2 instance as a NAT or inspection appliance. In that pattern, the instance has at least one ENI in a public subnet and one ENI in a private subnet.

Typical setup:

- **Public ENI**
  - Has a route to the internet gateway.
  - May hold an EIP.
- **Private ENI**
  - Receives traffic from private subnets.
  - Forwards traffic after inspection or translation.
- **Source/destination check**
  - Disabled on the forwarding instance.
- **Route tables**
  - Private subnet route table points `0.0.0.0/0` to the appliance ENI or instance path, depending on design.

Example route table intent:

```text
Private subnet route table
- 10.0.0.0/16 -> local
- 0.0.0.0/0   -> nat-instance-private-eni
```

Example security posture:

- Allow inbound from private subnet CIDRs on the private ENI.
- Allow outbound to the internet on the public ENI.
- Restrict management access to a bastion or VPN source range.

This pattern is useful for exam questions about centralized egress, packet inspection, or custom firewalling. In production, AWS managed NAT Gateway is usually preferred for simpler outbound internet access, but the exam may ask about the instance-based design because it exposes ENI behavior.

**Bastion replacement and blue/green cutover**

ENIs are also useful for replacing a bastion host or performing a blue/green cutover with minimal DNS changes.

Example scenario:

- You have a bastion host with a known private IP and security groups.
- You launch a new instance in the same AZ.
- You detach the ENI from the old instance and attach it to the new one.
- Users keep connecting to the same private IP or EIP.

This works well when you need to preserve:

- Security group rules
- Private IP identity
- Monitoring or allowlist dependencies
- Existing automation that targets the ENI

For blue/green application cutover, a secondary private IP can be even cleaner if the application listens on that address and the target ENI is in the same subnet.

**Troubleshooting attachment failures**

When an ENI attachment fails, check these items first:

- **AZ mismatch** between the ENI and target instance.
- **Subnet mismatch** when you are trying to reuse a private IP identity.
- **Instance type limits** for number of ENIs and private IPs.
- **Requester-managed** status, which may prevent manual attachment or deletion.
- **Attachment state** if the ENI is still detaching.
- **Security groups** and route tables after the move.

Useful commands:

```bash
aws ec2 describe-instances \
  --instance-ids i-0fedcba9876543210 \
  --query 'Reservations[0].Instances[0].NetworkInterfaces'
```

```bash
aws ec2 describe-network-interfaces \
  --filters Name=availability-zone,Values=us-east-1a
```

If you need to inspect the instance metadata service (<abbr title="Instance Metadata Service">IMDS</abbr>) from inside the instance, the network interface metadata is available under the MAC-address-specific path. Listing MACs alone is not enough to get subnet or security group details.

Example:

```bash
curl http://169.254.169.254/latest/meta-data/network/interfaces/macs/
curl http://169.254.169.254/latest/meta-data/network/interfaces/macs/02:7b:64:1a:2c:3d/
curl http://169.254.169.254/latest/meta-data/network/interfaces/macs/02:7b:64:1a:2c:3d/subnet-id
curl http://169.254.169.254/latest/meta-data/network/interfaces/macs/02:7b:64:1a:2c:3d/security-group-ids
```

**Common exam traps**

- **ENI mobility** is limited by AZ and subnet rules.
- **EIP mobility** is much more flexible than ENI mobility.
- **Security groups** belong to the ENI, not the instance.
- **Source/destination check** must be disabled for forwarding roles.
- **Primary ENI deletion** depends on `DeleteOnTermination`; do not assume it always disappears.
- **Secondary ENIs** are usually preserved when the instance terminates.
- **Secondary private IPs** can be reassigned within the same subnet, but not across subnets.
- **IPv6** does not use the same public remapping model as IPv4 EIPs.

**Quick memory aid**

- **Instance** = compute
- **ENI** = network identity
- **Security group** = firewall on the ENI
- **EIP** = remappable public IPv4
- **Source/destination check** = forwarding guardrail

**Practical command set**

```bash
# List ENIs attached to an instance
aws ec2 describe-network-interfaces \
  --filters Name=attachment.instance-id,Values=i-0123456789abcdef0

# Show attachment details
aws ec2 describe-network-interfaces \
  --network-interface-ids eni-0123456789abcdef0 \
  --query 'NetworkInterfaces[0].Attachment'

# Disable source/destination check
aws ec2 modify-network-interface-attribute \
  --network-interface-id eni-0123456789abcdef0 \
  --no-source-dest-check

# Associate an EIP with an ENI
aws ec2 associate-address \
  --allocation-id eipalloc-0123456789abcdef0 \
  --network-interface-id eni-0123456789abcdef0
```

**What to remember for SCS-C02**

For the exam, focus on how AWS uses ENIs to separate compute from network identity. The most important distinctions are attachment rules, lifecycle behavior, security group placement, and the difference between moving an ENI, moving a secondary private IP, and remapping an EIP. If you can reason through those three operations, you can answer most ENI-related security and failover questions correctly.
