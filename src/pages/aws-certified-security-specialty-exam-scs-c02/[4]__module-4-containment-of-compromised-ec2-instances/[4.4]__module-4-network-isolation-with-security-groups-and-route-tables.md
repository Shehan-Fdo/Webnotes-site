## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Network isolation in AWS VPCs**

Network isolation is the practice of controlling which resources can talk to each other and which paths traffic can take. In AWS, you usually combine **security groups**, **network ACLs (NACLs)**, **route tables**, and **VPC endpoints** to build layered controls.

A strong exam answer usually distinguishes between:

- **Security groups** for instance-level, stateful filtering
- **NACLs** for subnet-level, stateless filtering
- **Route tables** for path selection, not packet filtering
- **Endpoints** for private access to AWS services without public internet exposure

The key idea is that these controls solve different problems. If you mix them up, you can design a network that looks secure but still fails connectivity or leaks traffic through an unintended path.

**Core VPC traffic model**

Traffic in a VPC is evaluated in layers:

1. The subnet’s **route table** decides where packets go.
2. The subnet’s **NACL** allows or denies traffic at the subnet boundary.
3. The instance’s **security group** allows or denies traffic at the ENI (Elastic Network Interface) level.
4. Service-specific controls such as **endpoint policies** may further restrict access.

This layered model matters because a packet must satisfy all applicable controls. A single deny or missing route can break connectivity.

**Security groups**

Security groups are **stateful virtual firewalls** attached to ENIs. They control inbound and outbound traffic for resources such as EC2 instances, RDS databases, and load balancers.

Because they are stateful:

- Return traffic is automatically allowed if the initiating flow was allowed.
- You do not need to create separate rules for ephemeral return ports for established connections.
- You can focus on the direction that matters for the initial request.

Common exam-relevant properties:

- **Allow rules only**: security groups do not support explicit deny rules.
- **Stateful behavior**: response traffic is automatically permitted.
- **Referenced sources**: you can allow traffic from another security group, not just CIDR ranges.
- **ENI attachment**: the security group applies to the network interface, which is useful for multi-ENI designs.

A typical inbound rule for a private web tier might look like this:

```bash
aws ec2 authorize-security-group-ingress \
  --group-id sg-0123456789abcdef0 \
  --protocol tcp \
  --port 443 \
  --source-group sg-0fedcba9876543210
```

This pattern is common when an application tier should only accept HTTPS from an ALB or another trusted tier.

**Security group design patterns**

Use security groups when you want:

- Fine-grained application access control
- Stateful behavior that simplifies return traffic
- Identity-like references between tiers
- Easier operational management than subnet-wide rules

Examples:

- Web tier allows inbound `443` from an ALB security group.
- App tier allows inbound `8080` from the web tier security group.
- Database tier allows inbound `3306` or `5432` only from the app tier security group.

Security groups are usually the primary control for instance-to-instance isolation inside a VPC.

> Security groups cannot explicitly deny traffic. If you need deny semantics or subnet-wide stateless filtering, use a NACL.

**Network ACLs**

NACLs are **stateless** subnet-level filters. They evaluate traffic entering and leaving a subnet, and they require explicit rules for both directions of a flow.

Because they are stateless:

- Inbound and outbound rules are evaluated independently.
- You must allow return traffic explicitly.
- You often need ephemeral port rules for response traffic.

NACLs are useful when you want:

- A coarse subnet boundary control
- Explicit deny capability
- A guardrail around entire subnets
- Additional protection for public or shared subnets

A common NACL rule set for a private application subnet might include:

```text
Rule 100  ALLOW  TCP  443     10.0.0.0/16      inbound
Rule 110  ALLOW  TCP  1024-65535 10.0.0.0/16    inbound
Rule 120  DENY   ALL  ALL     0.0.0.0/0        inbound

Rule 100  ALLOW  TCP  443     10.0.0.0/16      outbound
Rule 110  ALLOW  TCP  1024-65535 10.0.0.0/16    outbound
Rule 120  DENY   ALL  ALL     0.0.0.0/0        outbound
```

This is only an example. In practice, you tailor the ports and CIDRs to the actual flows you need.

**Bidirectional ephemeral port handling**

For stateless NACLs, you must think carefully about ephemeral ports:

- When a client initiates outbound HTTPS to a server, the client uses an ephemeral source port.
- The server responds from port `443` to the client’s ephemeral port.
- The NACL on the client subnet must allow inbound traffic to the ephemeral port range.
- The NACL on the server subnet must allow outbound traffic from `443` and inbound traffic on `443` for the request path.

A practical example for a private subnet that initiates outbound web access through a NAT gateway:

- Outbound from instance: allow TCP `443` to `0.0.0.0/0`
- Inbound return traffic: allow TCP `1024-65535` from the relevant source CIDR or broader allowed range
- Security group statefulness handles return traffic at the instance level, but the NACL still needs explicit rules

For exam questions, remember:

- **Security groups** handle return traffic automatically.
- **NACLs** do not.

**Route tables**

Route tables determine where traffic is sent. They do not allow or deny traffic by themselves; they only choose the next hop.

A subnet can be associated with only **one route table** at a time. If you do not explicitly associate a subnet with a custom route table, it uses the **main route table** for the VPC.

Important route table behaviors:

- Every subnet must have a route table association, explicit or implicit.
- The **local route** for the VPC CIDR is always present and enables communication inside the VPC.
- Route selection uses **longest-prefix match**.
- If a route target is deleted or becomes unavailable, the route can become a **blackhole** route.
- A blackhole route appears in the route table but does not forward traffic successfully.

Example route table for a private subnet:

```text
Destination        Target
10.0.0.0/16        local
0.0.0.0/0          nat-0abc123def4567890
pl-68a54001        vpce-gateway-0123456789abcdef0
```

Interpretation:

- `10.0.0.0/16 -> local` keeps VPC-internal traffic inside the VPC.
- `0.0.0.0/0 -> nat-...` sends internet-bound traffic from the private subnet to a NAT gateway.
- `pl-... -> vpce-gateway-...` is a gateway endpoint route for a service prefix list such as S3 or DynamoDB.

**Route selection and precedence**

When multiple routes could match, AWS uses the most specific match.

Practical implications:

- A more specific CIDR route overrides a broader one.
- The implicit `local` route handles traffic to the VPC CIDR.
- If you add a more specific route to a peering connection, Transit Gateway, or endpoint prefix list, it can steer traffic away from the default path.
- A missing or blackholed specific route can break only part of the traffic, which makes troubleshooting tricky.

Example:

- `10.0.0.0/16 -> local`
- `10.0.1.0/24 -> pcx-1234`

Traffic to `10.0.1.0/24` follows the peering route because it is more specific than the broader local route entry for the VPC CIDR. In practice, AWS still preserves the local route for the VPC CIDR, but more specific routes can direct traffic for overlapping destinations outside the local VPC range.

**Internet gateway vs NAT gateway**

These two are often confused on the exam.

An **internet gateway (IGW)** provides internet connectivity for resources with public IP addresses in public subnets. It supports inbound and outbound internet traffic when the route table and security controls allow it.

A **NAT gateway** allows instances in private subnets to initiate outbound connections to the internet or AWS public endpoints without exposing those instances to inbound internet traffic.

Key differences:

- **IGW**
  - Used with public subnets
  - Requires public IP or Elastic IP association on the instance path
  - Supports inbound reachability from the internet when security controls allow it
- **NAT gateway**
  - Used from private subnets
  - Supports outbound-only internet access from private resources
  - Does not make the private instance directly reachable from the internet

A private subnet route table often includes:

```text
0.0.0.0/0 -> nat-0abc123def4567890
```

That route means the subnet can reach the internet outbound through the NAT gateway. It does **not** mean the subnet becomes publicly reachable.

> NAT gateways are stateful managed services. If outbound traffic fails, look first at the subnet route table, security groups, NACLs, DNS, or the destination-side controls—not at a missing “return route” in the private subnet route table.

**Route tables and private subnet egress**

A private subnet needs a default route to a NAT gateway if instances must reach the internet for package updates, external APIs, or software downloads.

Typical flow:

- Instance in private subnet sends traffic to `0.0.0.0/0`.
- Route table forwards it to the NAT gateway in a public subnet.
- NAT gateway uses the public subnet’s route to the IGW.
- Return traffic comes back through the NAT gateway automatically.

This is why private subnets can have outbound internet access without inbound internet exposure.

**VPC endpoints**

VPC endpoints let you access AWS services privately without traversing the public internet.

There are two main types:

- **Gateway endpoints**
- **Interface endpoints**

Use them differently:

- **Gateway endpoints**
  - Supported only for **Amazon S3** and **Amazon DynamoDB**
  - Add route table entries
  - Are the preferred choice when you want private access to those services at the routing layer
- **Interface endpoints**
  - Use **AWS PrivateLink**
  - Support most other AWS services
  - Create elastic network interfaces in your subnets
  - Do not add route table entries
  - Are controlled by security groups and endpoint policies

This distinction is exam critical.

**Gateway endpoints**

Gateway endpoints are attached to route tables and use service prefix lists.

Example route table entry:

```text
Destination        Target
pl-68a54001        vpce-gateway-0123456789abcdef0
```

This means traffic destined for the S3 or DynamoDB prefix list is routed to the gateway endpoint.

Use gateway endpoints when:

- You need private access to S3 or DynamoDB
- You want a simple, route-based design
- You want to avoid NAT charges for those service calls

**Interface endpoints**

Interface endpoints create ENIs in your subnets and are protected by security groups.

Use interface endpoints when:

- You need private access to services such as CloudWatch Logs, STS, ECR, Secrets Manager, KMS, or many other AWS services
- You want to keep traffic on the AWS network
- You need security group control at the endpoint

Because interface endpoints do not add route table entries, connectivity depends on:

- DNS resolution to the endpoint
- Security group rules on the endpoint ENI
- NACL rules for the subnet
- Endpoint policy permissions

**Endpoint policies**

Endpoint policies are an additional authorization layer for VPC endpoints.

They are useful when you want to restrict which principals, actions, or resources can use the endpoint.

Examples of what endpoint policies can help with:

- Limiting S3 access to specific buckets
- Restricting DynamoDB table access
- Constraining which AWS principals can use the endpoint path

Endpoint policies do not replace IAM policies or resource policies. They add another control point.

A simplified S3 gateway endpoint policy example:

```json
{
  "Statement": [
    {
      "Sid": "AllowSpecificBucketOnly",
      "Effect": "Allow",
      "Principal": "*",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": [
        "arn:aws:s3:::my-private-bucket",
        "arn:aws:s3:::my-private-bucket/*"
      ]
    }
  ]
}
```

**When to use each endpoint type**

| Need | Best choice | Why |
|---|---|---|
| Private S3 access | Gateway endpoint | Route-table integration and no NAT required |
| Private DynamoDB access | Gateway endpoint | Same routing model as S3 |
| Private access to most AWS APIs | Interface endpoint | Supports many services via PrivateLink |
| Security group control on the endpoint | Interface endpoint | Endpoint ENIs support SGs |
| Lowest operational complexity for S3/DynamoDB | Gateway endpoint | Simple and cost-effective |

**End-to-end isolated VPC design**

A common exam scenario is a fully isolated application stack with controlled outbound access.

Example architecture:

- Public subnet:
  - NAT gateway
  - Optional ALB
- Private app subnet:
  - EC2 instances or containers
  - Security group allowing only app-specific inbound traffic
  - Route table with default route to NAT gateway
- Private data subnet:
  - Database instances
  - Security group allowing only app tier access
  - No internet route
- VPC endpoints:
  - Gateway endpoint for S3
  - Interface endpoint for STS, CloudWatch Logs, Secrets Manager, or KMS as needed

A simplified flow:

1. The app instance receives traffic only from the ALB or trusted app tier security group.
2. The app subnet route table sends internet-bound traffic to the NAT gateway.
3. The app uses a gateway endpoint for S3 so bucket access stays private.
4. The app uses interface endpoints for AWS APIs that do not support gateway endpoints.
5. The database subnet has no default route to the internet and only accepts traffic from the app security group.

**Example configuration flow**

1. Create the VPC and subnets.
2. Attach a route table to each subnet.
3. Add the local route automatically provided by AWS.
4. Add a default route to a NAT gateway only for subnets that need outbound internet access.
5. Create security groups for each tier and allow only required ports and sources.
6. Add NACL rules if you need subnet-level deny controls or additional guardrails.
7. Create gateway endpoints for S3 or DynamoDB if private access is required.
8. Create interface endpoints for other AWS services and attach restrictive security groups.
9. Add endpoint policies to limit service access.
10. Test connectivity from the workload subnet and verify logs, routes, and DNS.

**Practical CLI examples**

Create a route table association:

```bash
aws ec2 associate-route-table \
  --route-table-id rtb-0123456789abcdef0 \
  --subnet-id subnet-0abc123def4567890
```

Add a default route to a NAT gateway:

```bash
aws ec2 create-route \
  --route-table-id rtb-0123456789abcdef0 \
  --destination-cidr-block 0.0.0.0/0 \
  --nat-gateway-id nat-0abc123def4567890
```

Create a gateway endpoint for S3:

```bash
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0123456789abcdef0 \
  --service-name com.amazonaws.us-east-1.s3 \
  --vpc-endpoint-type Gateway \
  --route-table-ids rtb-0123456789abcdef0
```

Create an interface endpoint for STS:

```bash
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0123456789abcdef0 \
  --service-name com.amazonaws.us-east-1.sts \
  --vpc-endpoint-type Interface \
  --subnet-ids subnet-0abc123def4567890 subnet-0def4567890abc123 \
  --security-group-ids sg-0123456789abcdef0 \
  --private-dns-enabled
```

**NACL example with numbered rules**

A concrete NACL example helps with exam questions because rule order matters.

```text
Inbound rules
100  ALLOW  TCP  443        10.0.0.0/16
110  ALLOW  TCP  1024-65535  10.0.0.0/16
120  DENY   ALL  ALL         0.0.0.0/0

Outbound rules
100  ALLOW  TCP  443        10.0.0.0/16
110  ALLOW  TCP  1024-65535  10.0.0.0/16
120  DENY   ALL  ALL         0.0.0.0/0
```

How to read it:

- Lower rule numbers are evaluated first.
- The first matching rule wins.
- You must allow both directions for stateless flows.
- If you need internet egress, you usually allow the destination ports you need and the ephemeral return range as appropriate for the flow.

**Troubleshooting checklist**

When connectivity fails, isolate the layer systematically.

1. Check the **route table**.
   - Is the subnet associated with the expected route table?
   - Is the destination covered by a valid route?
   - Is the route target active, or is it a blackhole?
   - Is the local route present for VPC-internal traffic?

2. Check the **security group**.
   - Is the required inbound port allowed?
   - Is the source correct: CIDR, prefix, or source security group?
   - Is outbound traffic allowed to the destination?

3. Check the **NACL**.
   - Are both inbound and outbound rules present?
   - Are ephemeral ports allowed where needed?
   - Is a deny rule shadowing the allow rule because of rule order?

4. Check the **endpoint configuration**.
   - Is the correct endpoint type used?
   - For gateway endpoints, is the route table associated?
   - For interface endpoints, is private DNS enabled if expected?
   - Does the endpoint security group allow the traffic?

5. Check the **destination-side controls**.
   - Does the target instance or service allow the source?
   - Is the destination security group correct?
   - Are IAM, endpoint policies, or resource policies blocking access?

6. Check **DNS and application settings**.
   - Is the hostname resolving to the expected private or public address?
   - Is the application using the correct port and protocol?
   - Is TLS or certificate validation failing?

**Common exam traps**

- Assuming a security group can deny traffic explicitly.
- Assuming a NACL is stateful.
- Thinking a NAT gateway makes a subnet publicly reachable.
- Forgetting that gateway endpoints are only for S3 and DynamoDB.
- Assuming interface endpoints add route table entries.
- Ignoring the main route table when a subnet has no explicit association.
- Missing the fact that a blackhole route can exist after a target is deleted.
- Confusing route selection with packet filtering.

**Quick comparison**

| Feature | Security Group | NACL | Route Table |
|---|---|---|---|
| Scope | ENI | Subnet | Subnet |
| Stateful | Yes | No | No |
| Explicit deny | No | Yes | No |
| Default behavior | Deny all inbound, allow all outbound unless changed | Deny unless allowed | No filtering, only routing |
| Return traffic handling | Automatic | Must be allowed explicitly | Not applicable |
| Best use | Instance-level access control | Subnet boundary control | Path selection |

**Key takeaways**

- Use **security groups** for stateful, instance-level isolation.
- Use **NACLs** when you need stateless subnet filtering or explicit deny rules.
- Use **route tables** to choose the next hop, not to enforce access control.
- Use **NAT gateways** for outbound internet access from private subnets.
- Use **gateway endpoints** for S3 and DynamoDB.
- Use **interface endpoints** for most other AWS services.
- Use **endpoint policies** to further restrict private service access.

**Exam mindset**

When you see a scenario, ask:

- Is the problem about who can talk to whom? Think security groups.
- Is it about subnet-wide allow/deny or ephemeral ports? Think NACLs.
- Is it about where traffic goes? Think route tables.
- Is it about private access to AWS services? Think VPC endpoints.
- Is it about outbound internet from a private subnet? Think NAT gateway.
