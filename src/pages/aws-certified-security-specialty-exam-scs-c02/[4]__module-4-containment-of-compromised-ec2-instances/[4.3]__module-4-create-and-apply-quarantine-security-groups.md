## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Quarantine security groups: purpose and incident-response role**

A **quarantine security group** is a restrictive Amazon Virtual Private Cloud (Amazon VPC) security group that you attach to a suspected or compromised Elastic Compute Cloud (Amazon EC2) instance to sharply reduce network exposure during investigation. The goal is not to “fix” the host immediately; it is to contain risk, preserve evidence, and prevent lateral movement while you assess the incident.

In practice, you use quarantine groups when you need to:
- Stop inbound access from untrusted networks.
- Prevent the instance from reaching internal services, databases, or other workloads.
- Keep only the minimum connectivity needed for forensics, patching, or management.
- Replace a broad production security posture with a known, tightly controlled one.

A quarantine group is most effective when it is preplanned. You should define it before an incident, document who can apply it, and ensure it aligns with your incident response runbooks.

**How security groups behave during quarantine**

Security groups are **stateful** virtual firewalls attached to ENIs (Elastic Network Interfaces). Statefulness means return traffic is automatically allowed when a connection is initiated from an allowed direction. This matters because a quarantine group can be very restrictive and still permit specific management flows without requiring separate return rules.

Key properties to remember:
- **Instance-level containment** is achieved by changing the security group on the ENI, not by modifying the operating system.
- **Outbound rules matter** as much as inbound rules if you want to block exfiltration or command-and-control traffic.
- **Multiple security groups** can be attached to one ENI, so you must remove permissive groups if you want true quarantine.
- **Security groups are allow lists only**; they do not support explicit deny rules.

A common mistake is attaching a quarantine group while leaving a permissive group in place. That does not isolate the instance.

**Typical quarantine design**

A quarantine security group usually allows only the traffic required for response operations. The exact rules depend on your environment, but a conservative design often includes:
- No inbound access except from a controlled bastion, VPN, or AWS Systems Manager (SSM) Session Manager path.
- No outbound access except to approved forensic or management endpoints.
- Optional access to internal logging, patch, or snapshot services if required by the response workflow.

A simple pattern is:
- Deny by omission through a minimal allow list.
- Use separate groups for normal operations and quarantine operations.
- Swap the instance from the production group to the quarantine group during containment.

**Example quarantine security group rules**

The following example shows a restrictive group for an EC2 instance under investigation. It allows only SSH from a trusted forensic subnet and HTTPS to an internal SSM or logging endpoint if needed.

```bash
aws ec2 create-security-group \
  --group-name quarantine-sg \
  --description "Quarantine SG for incident response" \
  --vpc-id vpc-0123456789abcdef0
```

```bash
aws ec2 authorize-security-group-ingress \
  --group-id sg-0123456789abcdef0 \
  --protocol tcp \
  --port 22 \
  --cidr 10.0.10.0/24
```

```bash
aws ec2 authorize-security-group-egress \
  --group-id sg-0123456789abcdef0 \
  --protocol tcp \
  --port 443 \
  --cidr 10.0.20.15/32
```

If you want a stricter posture, you can remove all egress rules and rely on out-of-band access methods such as SSM, provided the instance can still reach the required AWS endpoints through a VPC endpoint or controlled path.

**Applying quarantine to an instance**

You usually apply quarantine by updating the ENI attached to the instance. The exact method depends on whether the instance has one or more ENIs and whether it is in an Auto Scaling Group.

1. Identify the instance and its network interface.
2. Remove permissive security groups from the ENI.
3. Attach the quarantine security group.
4. Verify that only the intended rules remain effective.
5. Confirm that the instance is isolated from sensitive internal and external paths.

Example workflow:

```bash
aws ec2 describe-instances \
  --instance-ids i-0123456789abcdef0 \
  --query 'Reservations[0].Instances[0].NetworkInterfaces[0].NetworkInterfaceId' \
  --output text
```

```bash
aws ec2 modify-network-interface-attribute \
  --network-interface-id eni-0123456789abcdef0 \
  --groups sg-0123456789abcdef0
```

If the instance is managed by an Auto Scaling Group, be careful: the group may replace or reconfigure the instance after health checks fail. In that case, you may need to suspend scaling processes or isolate the workload at a higher control point, depending on your incident response strategy.

**When to use quarantine groups**

Use a quarantine security group when you need fast, reversible containment and the instance must remain online for investigation. This is especially useful when:
- You suspect credential theft, malware, or unauthorized access.
- You need to preserve volatile evidence before terminating the instance.
- You want to prevent further spread while collecting logs, memory, or disk snapshots.
- You need to keep the host reachable only through a controlled management channel.

You may choose a different response if the host is already irrecoverable or if policy requires immediate termination and rebuild. Quarantine is about containment first, remediation second.

**Operational considerations**

A quarantine group is only one layer of containment. You should also consider:
- **IAM** permissions that could allow the instance role to access sensitive AWS resources.
- **Network ACLs (NACLs)** if you need subnet-level controls, though they are stateless and less flexible for per-instance response.
- **VPC endpoints** to preserve access to AWS services without opening internet egress.
- **CloudTrail**, VPC Flow Logs, and GuardDuty findings to support investigation.
- **SSM Session Manager** for access without inbound SSH, if the instance is already configured for it.

If the instance has a public IP address, removing the security group does not remove the public IP. It only changes what traffic is allowed. For stronger containment, you may also need to detach public exposure at the architecture level.

**Quarantine versus other controls**

| Control | Scope | Strength | Typical use |
|---|---:|---:|---|
| Security group quarantine | ENI / instance | High for network containment | Incident response on a specific host |
| NACL restriction | Subnet | Medium | Broad subnet-level containment |
| Route table change | Subnet / VPC path | High for path disruption | Block traffic to/from a network segment |
| IAM policy restriction | Principal / role | High for AWS API access | Stop resource access from compromised credentials |

Security groups are usually the fastest and most precise option for isolating a single EC2 instance.

**Practical incident-response pattern**

A common response pattern is:
- Preserve the instance.
- Snapshot attached EBS volumes.
- Capture logs and metadata.
- Apply the quarantine security group.
- Restrict IAM permissions if the instance role is suspected compromised.
- Investigate and decide whether to rebuild, restore, or terminate.

This sequence helps you avoid destroying evidence before you understand the scope of compromise.

**Example: quarantine with Systems Manager access**

If the instance is managed by SSM, you can design the quarantine group to allow only the minimum network path needed for SSM connectivity through VPC endpoints.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowSSMAndLogs",
      "Effect": "Allow",
      "Action": [
        "ssm:StartSession",
        "ssm:DescribeInstanceInformation",
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "*"
    }
  ]
}
```

This does not replace the security group; it complements it. The network path must still exist for the agent to reach the required endpoints.

**Common pitfalls**

- Leaving the original production security group attached.
- Allowing broad outbound internet access during quarantine.
- Forgetting that security groups are stateful and assuming they block all return traffic in every case.
- Quarantining the wrong ENI on a multi-homed instance.
- Ignoring Auto Scaling behavior that may undo your containment.
- Using quarantine as a substitute for credential revocation or IAM containment.

> A quarantine security group reduces network exposure, but it does not neutralize compromised credentials, malware persistence, or data already exfiltrated.

**Exam focus**

For the AWS Certified Security – Specialty exam, expect questions that test whether you can:
- Contain a compromised EC2 instance quickly.
- Choose security groups over broader controls when isolating a single host.
- Preserve access for forensics while blocking unnecessary traffic.
- Recognize that multiple attached security groups can undermine quarantine.
- Use least privilege and stateful behavior correctly in a response scenario.

**Quick reference**

- **Security group**: Stateful allow-list firewall attached to an ENI.
- **Quarantine group**: Minimal allow-list security group used to isolate a suspected resource.
- **Best use case**: Rapid containment of a single EC2 instance during incident response.
- **Key caution**: Remove all permissive groups; attaching one restrictive group is not enough if others remain.
