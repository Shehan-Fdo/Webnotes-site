## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Snapshot volumes and evidence preservation**

Amazon Elastic Block Store (Amazon EBS) snapshots are point-in-time backups of EBS volumes stored in Amazon S3-managed infrastructure. For security and incident response, you use snapshots to preserve disk state before you modify, quarantine, or terminate a compromised instance. The goal is to capture evidence with minimal change, maintain chain of custody, and keep the original volume intact for later analysis.

A snapshot is not a mounted copy of the volume. It is a backup artifact that you can use to create a new volume later. That distinction matters in exams and in real incident response: you preserve the original, snapshot it, and then work from a restored copy.

**Core evidence-preservation workflow**

When you suspect compromise, you typically follow a controlled sequence:

1. Isolate the instance at the network layer.
2. Preserve the volume by creating a snapshot.
3. Record metadata and timestamps.
4. Copy or share the snapshot only if required and only with the right controls.
5. Restore the snapshot to a new volume for analysis.

This workflow reduces the chance that evidence changes while you investigate.

**Why snapshots are useful for forensics**

EBS snapshots help you preserve:

- File system state at a point in time.
- Malware binaries, logs, and persistence artifacts.
- Deleted-but-not-overwritten data that may still exist in blocks.
- Boot records and configuration files relevant to root cause analysis.

Snapshots are especially valuable when the instance is still running and you need to capture evidence quickly. However, a running system can still change while you snapshot it, so you should treat the snapshot as a near-point-in-time capture rather than a perfect freeze of all application state.

**Snapshot consistency and application state**

EBS snapshots are crash-consistent by default. That means AWS captures the volume as if the system lost power at that moment. For many file systems, this is sufficient to restore the volume and inspect data. For applications with in-memory transactions or write caches, crash consistency may not be enough.

To improve consistency:

- Quiesce the application before snapshotting when possible.
- Flush file system buffers.
- Use database-native backup or freeze mechanisms if the workload supports them.
- Coordinate with the operating system to reduce write activity.

Examples include pausing writes, using database backup modes, or freezing a Linux file system before snapshot creation. If you cannot quiesce the workload, document that the snapshot is crash-consistent and preserve the context.

**CLI snapshot creation**

You can create a snapshot with the AWS CLI. The command captures the volume and lets you add a description for tracking.

```bash
aws ec2 create-snapshot \
  --volume-id vol-0123456789abcdef0 \
  --description "IR-2026-04-26 suspected compromise on app server"
```

Useful follow-up commands:

```bash
aws ec2 describe-snapshots \
  --snapshot-ids snap-0123456789abcdef0
```

```bash
aws ec2 wait snapshot-completed \
  --snapshot-ids snap-0123456789abcdef0
```

The waiter is useful when you need to block automation until the snapshot is ready. Always verify the waiter name against the current AWS CLI version in your environment.

**Tagging and evidence tracking**

Tags help you preserve context and support governance. Use tags to record incident identifiers, owner, environment, and retention intent.

```bash
aws ec2 create-tags \
  --resources snap-0123456789abcdef0 \
  --tags Key=CaseId,Value=IR-2026-04-26 \
         Key=Owner,Value=SecurityTeam \
         Key=Purpose,Value=ForensicPreservation
```

Good evidence tags are:

- Case or ticket ID.
- Analyst name or team.
- Source instance ID.
- Source volume ID.
- Retention or legal hold marker.

Tags do not replace a chain-of-custody process, but they make later review easier.

**Restoring a snapshot for analysis**

To inspect data, you create a new volume from the snapshot and attach it to an analysis instance. The restored volume is a separate resource, so you can preserve the original snapshot unchanged.

```bash
aws ec2 create-volume \
  --snapshot-id snap-0123456789abcdef0 \
  --availability-zone us-east-1a \
  --volume-type gp3
```

Then attach it to a forensic instance:

```bash
aws ec2 attach-volume \
  --volume-id vol-0fedcba9876543210 \
  --instance-id i-0123456789abcdef0 \
  --device /dev/sdf
```

> EBS does not make the attachment read-only for you. If you need read-only handling, enforce it at the operating system or filesystem layer after attachment.

On Linux, you might mount the volume read-only:

```bash
sudo mkdir -p /mnt/evidence
sudo mount -o ro /dev/nvme1n1p1 /mnt/evidence
```

If the volume contains a partition table, identify the correct device or partition first with tools such as `lsblk`, `blkid`, or `fdisk -l`.

**Encryption and KMS considerations**

Encrypted EBS snapshots remain encrypted. When you create a volume from an encrypted snapshot, the new volume is also encrypted. You can re-encrypt by copying the snapshot and specifying a different AWS Key Management Service (AWS KMS) key.

Important behaviors:

- Unencrypted snapshots can be copied and encrypted during copy.
- Encrypted snapshots can be copied and re-encrypted with a different KMS key.
- You must have permission to use the destination KMS key.
- If you share encrypted snapshot data, the recipient also needs access to the KMS key material or a key they can use, depending on the sharing model.

Example copy with encryption:

```bash
aws ec2 copy-snapshot \
  --source-region us-east-1 \
  --source-snapshot-id snap-0123456789abcdef0 \
  --description "Encrypted forensic copy" \
  --encrypted \
  --kms-key-id arn:aws:kms:us-west-2:123456789012:key/abcd1234-5678-90ab-cdef-1234567890ab
```

For cross-Region copy, run the command in the destination Region context and specify the source Region and source snapshot ID. Do not rely on a `--destination-region` parameter for `copy-snapshot`.

**Sharing snapshots and access control**

You can share unencrypted snapshots with other AWS accounts by modifying snapshot permissions. This is useful when a separate security account or forensic account needs to restore a copy.

```bash
aws ec2 modify-snapshot-attribute \
  --snapshot-id snap-0123456789abcdef0 \
  --attribute createVolumePermission \
  --operation-type add \
  --user-ids 123456789012
```

For encrypted snapshots, sharing is more complex:

- You must share the snapshot permissions.
- You must also allow the recipient to use the KMS key, or provide a copy encrypted under a key they can use.
- Snapshot sharing alone is not sufficient for encrypted data.

This is a common exam trap: snapshot permissions and KMS permissions are separate controls.

**Snapshot copy for isolation and retention**

Copying a snapshot is useful when you want to:

- Move evidence to another Region for resilience.
- Store a separate copy under a different KMS key.
- Preserve a clean forensic copy while continuing operational recovery elsewhere.
- Apply different retention or access controls.

Copying does not change the source snapshot. It creates a new snapshot object with its own lifecycle and permissions.

**AWS Backup and evidence preservation**

AWS Backup can protect EBS volumes and automate backup policies, but it is not a generic substitute for forensic snapshot handling. Use it when you need centralized backup governance, retention policies, and scheduled backups. Use direct EBS snapshot workflows when you need immediate incident-response preservation and fine-grained control.

Key points:

- AWS Backup can manage backup plans and vault retention.
- Recovery points are stored in backup vaults.
- Vault access policies and KMS permissions still matter.
- Cross-account and cross-Region behavior depends on the supported resource type and AWS Backup features, not a blanket “share anything anywhere” model.

For exam purposes, distinguish between:

- **Operational backup**: scheduled, policy-driven, long-term recovery.
- **Forensic preservation**: immediate capture, evidence handling, and controlled access.

**Retention and deletion protection**

Evidence should not disappear because of an automated cleanup rule. Use retention controls and access restrictions to protect snapshots.

Common controls include:

- IAM permissions that restrict `ec2:DeleteSnapshot`.
- Tag-based policies for evidence resources.
- AWS Backup vault retention where appropriate.
- Separate forensic accounts or vaults for isolation.
- Manual approval workflows for deletion.

If legal hold or investigation retention is required, document it outside the snapshot itself and enforce it with policy and process.

**Incident response topology**

A practical forensic pattern looks like this:

- Production account contains the compromised EC2 instance and EBS volume.
- Security account receives a copied snapshot or a restored volume, depending on permissions and encryption.
- Forensic analysis instance is isolated in a restricted subnet.
- Analysts mount the restored volume read-only at the OS level and collect artifacts.

This separation helps preserve the original environment while allowing investigation in a controlled account.

**Example response sequence**

1. Stop user access to the instance through security groups, NACLs, or instance isolation.
2. Capture instance metadata, volume IDs, and timestamps.
3. Create the snapshot.
4. Wait for completion.
5. Copy the snapshot if you need another Region or account boundary.
6. Restore a new volume from the snapshot.
7. Attach the volume to a forensic instance.
8. Mount it read-only and collect evidence.
9. Record hashes, notes, and chain-of-custody details.

**Practical Linux evidence handling**

After attaching the restored volume, inspect it carefully before mounting:

```bash
lsblk
sudo blkid
sudo file -s /dev/nvme1n1
```

If the volume contains LVM (Logical Volume Manager), you may need to activate the volume group first:

```bash
sudo pvscan
sudo vgscan
sudo vgchange -ay
sudo lvscan
```

Then mount the logical volume read-only:

```bash
sudo mount -o ro /dev/mapper/vg0-root /mnt/evidence
```

If you need to preserve timestamps or copy files out, use tools that avoid unnecessary metadata changes where possible.

**Common pitfalls**

- Taking a snapshot without recording the source volume and instance context.
- Assuming a snapshot is application-consistent when the workload was not quiesced.
- Forgetting that encrypted snapshot sharing requires KMS access too.
- Mounting evidence volumes read-write by accident.
- Treating AWS Backup as a direct replacement for incident-response snapshot handling.
- Deleting the source volume before the snapshot is verified and retained.

**Exam-focused distinctions**

| Topic | Key idea | Security implication |
|---|---|---|
| EBS snapshot | Point-in-time backup of a volume | Preserves disk state for recovery or forensics |
| Crash consistency | Captures volume as if power failed | May not preserve in-memory application state |
| Encryption | Snapshot and restored volume remain encrypted | KMS permissions control access |
| Snapshot sharing | Works for unencrypted snapshots; encrypted needs KMS access | Sharing is not just an EC2 permission problem |
| AWS Backup | Policy-driven backup service | Good for governance, not a blanket forensic sharing tool |


**What to remember for SCS-C02**

- Use snapshots to preserve EBS volume state before remediation.
- Treat snapshots as crash-consistent unless you quiesce the workload.
- Restore to a new volume for analysis; do not alter the original evidence.
- Encrypted snapshots require KMS-aware handling.
- Snapshot sharing and KMS permissions are separate concerns.
- Mount restored evidence read-only at the OS level, not at the EBS attachment layer.
- AWS Backup helps with backup governance, but forensic preservation still needs careful snapshot and access control design.
