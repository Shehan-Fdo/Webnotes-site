## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Incident response goals for a suspected EC2 compromise**

A suspected Amazon Elastic Compute Cloud (EC2) compromise requires two parallel goals: contain the threat quickly and preserve evidence for later analysis. In practice, you want to stop attacker activity, prevent reinfection, and avoid destroying artifacts that explain initial access, persistence, lateral movement, and data access.

The best response depends on the instance role, network design, and whether the workload is still needed. If the instance is business-critical, you often isolate first and investigate second. If the instance is disposable, you may snapshot, replace, and rebuild from a known-good baseline.

- **Containment** means reducing attacker reach without erasing evidence.
- **Preservation** means capturing disk, memory, network, and control-plane artifacts before changes.
- **Eradication** means removing persistence, credentials, and malicious code.
- **Recovery** means restoring service from a trusted source and validating it is clean.

**Immediate triage and decision points**

Start by determining whether the instance is actively compromised, merely suspicious, or already offline. Look for unusual CPU, network, or disk activity; unexpected processes; new users; altered startup scripts; and unexplained outbound connections. Correlate host findings with AWS control-plane logs and security services.

Useful sources for triage include:

- **Amazon CloudWatch** metrics and logs for spikes in CPU, network, or disk I/O.
- **AWS CloudTrail** for API activity such as `AuthorizeSecurityGroupIngress`, `CreateAccessKey`, `AttachRolePolicy`, `RunInstances`, or `ModifyInstanceAttribute`.
- **Amazon GuardDuty** for findings such as suspicious DNS, crypto-mining, credential exfiltration, or unusual API calls.
- **AWS Security Hub** for aggregated findings and cross-service prioritization.
- **AWS Config** for configuration drift, resource changes, and compliance history.

If the instance is still reachable, avoid making unnecessary changes. Every command you run can alter timestamps, logs, and process state. Prefer read-only checks first.

```bash
# Basic host triage
uptime
who
w
ps aux --sort=-%cpu | head
ps aux --sort=-%mem | head
ss -tulpn
lsof -i -P -n | head
last -a | head
journalctl -p warning -n 100
```

**Containment options in AWS**

Security groups are often the first containment control, but they are not always sufficient. If you suspect the security group is already compromised, overly permissive, or shared across multiple workloads, isolate at a deeper network layer.

- **Security group isolation**
  - Remove inbound rules.
  - Restrict outbound rules to only what is required for investigation or business continuity.
  - Replace the instance’s security group with a quarantine group if needed.
- **Subnet or NACL isolation**
  - Move the instance into a quarantine subnet if architecture allows.
  - Use a **network ACL (NACL)** to block inbound and outbound traffic at the subnet level.
  - This is useful when security groups are insufficient or you need a broader containment boundary.
- **Route table isolation**
  - Remove or replace routes to the internet gateway, NAT gateway, VPN, or peering connections.
  - Blackhole traffic by pointing the subnet to a restricted route table.
- **Instance-level isolation**
  - Stop the instance only if you are ready to preserve state and accept service interruption.
  - Detach public IP exposure by removing Elastic IP association if appropriate.

> If you suspect the security group itself is compromised or reused by other instances, do not rely on it alone for containment. Use subnet, NACL, or route-table isolation to reduce blast radius.

A common quarantine pattern is to move the instance into a restricted subnet with no default route to the internet and only allow access from a forensic bastion or VPN endpoint.

```text
Quarantine VPC
  ├─ Forensic subnet
  │   └─ Bastion / analysis host
  └─ Isolated subnet
      └─ Suspected EC2 instance
```

**Preserving evidence before remediation**

Preserve the instance and its attached storage before you terminate, rebuild, or patch it. Forensic value drops quickly once the root volume is modified or the instance is rebooted repeatedly.

Key evidence sources include:

- **Root EBS volume**
  - Snapshot the root volume before changes.
  - If the instance is stopped, detach the root volume and attach it to a forensic instance as read-only when possible.
- **Additional EBS data volumes**
  - Identify every attached volume, not just the root volume.
  - Snapshot each data volume separately.
  - Preserve application data, logs, databases, caches, and staging directories that may contain attacker artifacts.
- **Multi-volume instances**
  - Check all block device mappings.
  - Do not assume the root volume contains all relevant evidence.
  - Snapshot and label each volume with the instance ID, device name, and timestamp.
- **Instance metadata**
  - Capture instance ID, AMI ID, launch template, IAM instance profile, security groups, subnet, route table, and attached ENIs.
- **Memory and volatile state**
  - If available and justified, capture memory or use a live-response tool before shutdown.
  - Memory can contain injected code, decrypted secrets, command history, and active network connections.

Example: list attached volumes and snapshot them all.

```bash
INSTANCE_ID=i-0123456789abcdef0

aws ec2 describe-instances \
  --instance-ids "$INSTANCE_ID" \
  --query 'Reservations[0].Instances[0].BlockDeviceMappings[*].{Device:DeviceName,VolumeId:Ebs.VolumeId}' \
  --output table

aws ec2 create-snapshot \
  --volume-id vol-aaa111 \
  --description "Forensic snapshot for $INSTANCE_ID root volume"

aws ec2 create-snapshot \
  --volume-id vol-bbb222 \
  --description "Forensic snapshot for $INSTANCE_ID data volume /dev/xvdf"
```

If you need to preserve evidence in another account, copy or share snapshots to a dedicated forensic account with tightly controlled access.

```bash
# Share a snapshot with a forensic account
aws ec2 modify-snapshot-attribute \
  --snapshot-id snap-0123456789abcdef0 \
  --attribute createVolumePermission \
  --operation-type add \
  --user-ids 123456789012

# Copy a snapshot into a forensic account or Region
aws ec2 copy-snapshot \
  --source-region us-east-1 \
  --source-snapshot-id snap-0123456789abcdef0 \
  --description "Copied for forensic analysis"
```

**Host-level investigation**

Once evidence is preserved, inspect the instance for persistence and attacker tooling. Focus on how the attacker gained execution, what they changed, and how they may return.

Common persistence locations on Linux include:

- `crontab` entries and `/etc/cron*`
- `systemd` units and drop-in files
- shell profile files such as `~/.bashrc`, `~/.profile`, or `/etc/profile`
- SSH authorized keys in `~/.ssh/authorized_keys`
- startup scripts, init scripts, and application-specific hooks
- unexpected binaries in `/tmp`, `/var/tmp`, `/dev/shm`, or hidden directories

Useful commands:

```bash
# Look for suspicious persistence and recent changes
find /etc -type f -mtime -7 2>/dev/null | head -50
find /home -type f \( -name ".bashrc" -o -name ".profile" -o -name "authorized_keys" \) -mtime -7 2>/dev/null
systemctl list-unit-files --type=service
systemctl list-timers
crontab -l
ls -la /etc/cron* /var/spool/cron 2>/dev/null
grep -R "curl\|wget\|nc \|bash -i\|python -c" /etc /home 2>/dev/null | head -50
```

On Windows EC2 instances, inspect:

- Scheduled Tasks
- Services
- Run keys in the registry
- Startup folders
- PowerShell history
- Event logs
- Remote Desktop Protocol (RDP) logons and new local administrators

**IAM and credential evidence**

A suspected EC2 compromise is not only a host problem. Attackers often pivot through credentials exposed on the instance or through the instance profile. Preserve and investigate IAM-related evidence beyond the instance profile itself.

Look for:

- **Instance profile role usage**
  - Determine what permissions the role grants and whether they were abused.
  - Review CloudTrail for API calls made with the role’s temporary credentials.
- **Compromised access keys**
  - Search for access keys on the instance, in application config files, environment variables, and developer tooling.
  - Check whether any IAM user access keys were created, rotated, or used recently.
- **STS sessions**
  - Investigate AWS Security Token Service (STS) activity such as `AssumeRole`, `AssumeRoleWithWebIdentity`, or unusual session names.
  - Correlate session issuance time with suspicious API calls.
- **Recent IAM policy changes**
  - Review `AttachUserPolicy`, `PutUserPolicy`, `AttachRolePolicy`, `PutRolePolicy`, `CreatePolicyVersion`, and trust policy changes.
  - Look for privilege escalation paths such as adding `AdministratorAccess` or modifying a role trust relationship.
- **Credential sources on the host**
  - Environment variables
  - Application config files
  - `.aws/credentials` and `.aws/config`
  - CI/CD tokens, API keys, and secrets managers caches

Example CloudTrail queries can be done in Athena, CloudTrail Lake, or your SIEM. A simple pattern is to filter by the instance role ARN, source IP, and suspicious API actions.

```sql
SELECT eventTime, eventName, userIdentity.arn, sourceIPAddress, requestParameters
FROM cloudtrail_logs
WHERE userIdentity.arn LIKE '%role/EC2InstanceRole%'
  AND eventTime > timestamp '2026-04-26 00:00:00'
ORDER BY eventTime DESC;
```

**Network forensics and deeper traffic analysis**

If the instance is suspected of command-and-control, data exfiltration, or lateral movement, host logs may not be enough. Use network evidence when available.

Options include:

- **VPC Traffic Mirroring**
  - Mirror ENI traffic to an analysis appliance for packet capture and inspection.
  - Useful for observing protocols, destinations, and payload patterns.
  - Best when enabled before or early in the incident.
- **Packet capture on the host**
  - Use `tcpdump` or similar tools if you can do so without destroying evidence.
  - Capture only what you need and document the time window.
- **VPC Flow Logs**
  - Review accepted and rejected flows for destination IPs, ports, and traffic volume.
  - Flow logs do not show payloads, but they help identify suspicious connections.
- **DNS logs**
  - Check Route 53 Resolver query logs or host resolver logs for suspicious domains.

Example packet capture:

```bash
sudo tcpdump -i eth0 -nn -s0 -w /tmp/ec2-incident.pcap
```

Example VPC Flow Logs interpretation:

- Repeated outbound connections to uncommon ports may indicate beaconing.
- Large sustained transfers to external IPs may indicate exfiltration.
- Rejected inbound scans may indicate exposure, but not necessarily compromise.

**AWS service-side detection and corroboration**

Use managed detection services to validate your hypothesis and prioritize response actions.

- **GuardDuty**
  - Detects anomalous API calls, suspicious DNS, port probing, crypto-mining, and known malicious IP activity.
  - Findings can point to the initial compromise vector or post-compromise behavior.
- **Security Hub**
  - Aggregates findings from GuardDuty, Inspector, Config, and partner tools.
  - Helps you see whether the issue is isolated or part of a broader security posture problem.
- **AWS Config**
  - Shows resource configuration history and drift.
  - Useful for identifying when security groups, IAM roles, or instance settings changed.
- **Amazon Inspector**
  - Helps identify vulnerable packages or exposed software on supported workloads.
  - Useful for determining whether the compromise may have exploited a known weakness.

**Containment when the instance must stay online**

Sometimes you cannot stop the instance immediately because it supports a critical service. In that case, reduce exposure while preserving service continuity.

- Remove public exposure by detaching the Elastic IP or moving behind a load balancer with stricter controls.
- Restrict inbound access to a bastion, VPN, or private endpoint.
- Limit outbound access to only required destinations.
- Disable or rotate credentials that may be exposed on the host.
- Increase logging and monitoring before making larger changes.

If the instance is part of an Auto Scaling group, remember that the group may launch a replacement from the same compromised launch template or AMI. Update the launch template or image before allowing replacement instances to come online.

```bash
# Example: create a new launch template version with a clean AMI
aws ec2 create-launch-template-version \
  --launch-template-id lt-0123456789abcdef0 \
  --source-version 1 \
  --launch-template-data '{"ImageId":"ami-0clean1234567890"}'

# Point the Auto Scaling group to the new version
aws autoscaling update-auto-scaling-group \
  --auto-scaling-group-name prod-web-asg \
  --launch-template LaunchTemplateId=lt-0123456789abcdef0,Version=2
```

**Eradication and recovery**

Do not trust a compromised instance to become clean just because you removed one malicious file. Attackers often leave multiple persistence mechanisms, stolen credentials, and backdoors.

A safer recovery pattern is:

1. Preserve evidence and isolate the instance.
2. Rebuild from a known-good AMI or golden image.
3. Patch and harden the new instance before production use.
4. Rotate all potentially exposed secrets.
5. Validate that persistence mechanisms are absent.
6. Reintroduce the workload gradually with enhanced monitoring.

Validation should include:

- Confirming the new AMI is from a trusted pipeline.
- Verifying package versions and configuration baselines.
- Checking for unexpected users, services, scheduled tasks, and startup entries.
- Rotating application secrets, database passwords, API keys, and SSH keys that may have been exposed.
- Reviewing CloudTrail and GuardDuty for continued suspicious activity after cutover.
- Confirming the old instance is terminated only after evidence is preserved and no further analysis is needed.

> If the instance hosted secrets, assume they are compromised until proven otherwise. Rotate them before returning the workload to production.

**Practical recovery example**

A common response for a compromised web server is to quarantine the instance, snapshot all volumes, launch a clean replacement from a hardened AMI, and then swap traffic after validation.

```bash
# Example: launch a clean replacement from a trusted AMI
aws ec2 run-instances \
  --image-id ami-0clean1234567890 \
  --instance-type t3.medium \
  --subnet-id subnet-0123456789abcdef0 \
  --security-group-ids sg-0123456789abcdef0 \
  --iam-instance-profile Name=WebServerRole
```

Before reintroducing traffic, verify:

- The replacement instance has no unexpected outbound connections.
- The application starts without pulling binaries from untrusted sources.
- Secrets come from a managed secret store or parameter store, not hardcoded files.
- Logging, monitoring, and alerting are active.
- The old compromised instance is not still reachable from the network.

**Exam-focused response pattern**

For SCS-C02 scenarios, the best answer usually balances containment, evidence preservation, and recovery hygiene. Look for the option that:

- isolates the instance without destroying evidence,
- snapshots all relevant volumes,
- checks CloudTrail, GuardDuty, Security Hub, and Config,
- investigates IAM and STS activity,
- uses network-level containment if security groups are not enough,
- rebuilds from a clean AMI,
- rotates secrets,
- and prevents reinfection through updated launch templates or hardened baselines.

**Quick reference checklist**

- Identify the instance, role, subnet, security groups, and attached volumes.
- Contain with security groups, NACLs, subnet isolation, or route-table changes.
- Snapshot the root volume and every attached data volume.
- Preserve IAM evidence, including access keys, STS sessions, and policy changes.
- Review GuardDuty, Security Hub, AWS Config, CloudTrail, and CloudWatch.
- Use VPC Traffic Mirroring or packet capture if deeper network forensics are needed.
- Rebuild from a clean AMI and validate before production cutover.
- Rotate secrets and verify persistence is removed.
- Update Auto Scaling launch templates or images to prevent reinfection.
