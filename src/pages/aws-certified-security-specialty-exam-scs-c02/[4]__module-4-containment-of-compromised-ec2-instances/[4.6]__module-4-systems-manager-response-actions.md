## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Systems Manager in incident response**

AWS Systems Manager (SSM) gives you a controlled way to inspect, contain, and remediate EC2 instances and some hybrid managed nodes without logging in manually. In security incidents, you use it to reduce exposure, collect evidence, and automate repeatable actions while preserving as much forensic value as possible.

The main exam idea is choosing the right SSM capability for the job:

- **Session Manager** for interactive access without opening inbound SSH or RDP.
- **Run Command** for one-to-many command execution on managed nodes.
- **Automation** for multi-step workflows, approvals, and service integrations.
- **State Manager** for ongoing configuration enforcement.
- **OpsCenter** and **Incident Manager** for operational tracking and incident workflows.

A good response plan minimizes host changes, uses least privilege, and records what happened in a durable audit trail.

**Core response workflow**

A typical containment and investigation flow looks like this:

1. Detect suspicious activity from a service such as GuardDuty, Security Hub, CloudWatch alarms, or custom detections.
2. Trigger an automated response with EventBridge, Systems Manager Automation, or a Lambda function.
3. Contain the instance by changing security groups, isolating network access, or stopping risky processes.
4. Collect evidence such as memory-adjacent process data, instance metadata, logs, snapshots, or AMIs.
5. Record the incident in OpsCenter or Incident Manager and hand off for deeper analysis.

**Choosing the right SSM feature**

Use the right tool based on scope and control requirements:

| Capability | Best for | Notes |
|---|---|---|
| Session Manager | Interactive troubleshooting and live investigation | No inbound ports required; access is authenticated and logged. |
| Run Command | Quick commands across one or many instances | Good for containment scripts, log collection, and checks. |
| Automation | Repeatable response playbooks | Supports approvals, branching, and AWS API actions. |
| State Manager | Persistent desired state | Better for ongoing compliance than one-time response. |
| OpsCenter | Central incident record | Tracks operational issues, related resources, and runbooks. |
| Incident Manager | Coordinated incident response | Useful for escalation, timelines, and response plans. |

**Session Manager for secure interactive access**

Session Manager lets you connect to a managed instance through the SSM Agent and the Systems Manager service. You do not need inbound SSH or RDP rules if the instance can reach SSM endpoints or the public service endpoints.

Use it when you need to inspect a system manually, but prefer it over direct network access because it reduces attack surface and improves auditability.

Typical prerequisites:

- The instance has the **SSM Agent** installed and running.
- The instance profile includes permissions such as `AmazonSSMManagedInstanceCore`.
- Network access exists to Systems Manager endpoints, either through the internet or VPC endpoints.
- The instance is a supported managed node type.

Example session start:

```bash
aws ssm start-session --target i-0123456789abcdef0
```

For port forwarding, use Session Manager documents such as:

```bash
aws ssm start-session \
  --target i-0123456789abcdef0 \
  --document-name AWS-StartPortForwardingSession \
  --parameters "portNumber=3389,localPortNumber=13389"
```

For SSH tunneling through Session Manager:

```bash
aws ssm start-session \
  --target i-0123456789abcdef0 \
  --document-name AWS-StartSSHSession \
  --parameters "portNumber=22"
```

Session Manager is especially useful when you want to avoid exposing bastions or opening inbound ports. It is also a strong exam answer when the requirement is “secure shell access without public SSH.”

**Run Command for fast containment and collection**

Run Command executes a document on one or more managed nodes. It is ideal for short, repeatable actions such as stopping a process, collecting logs, checking configuration, or disabling a service.

Example command invocation:

```bash
aws ssm send-command \
  --document-name AWS-RunShellScript \
  --targets "Key=InstanceIds,Values=i-0123456789abcdef0" \
  --parameters 'commands=["sudo systemctl stop suspicious-service","sudo ps aux | head"]'
```

On Windows, use `AWS-RunPowerShellScript` and pass PowerShell commands through the `commands` parameter:

```powershell
aws ssm send-command `
  --document-name AWS-RunPowerShellScript `
  --targets "Key=InstanceIds,Values=i-0123456789abcdef0" `
  --parameters 'commands=["Get-Process | Sort-Object CPU -Descending | Select-Object -First 10","Get-NetTCPConnection | Select-Object -First 20"]'
```

Important operational points:

- **Output** is stored separately from the command itself and can be sent to S3 or CloudWatch Logs.
- **Execution** is asynchronous; you must query status after submission.
- **Scope** can be a single instance, a tag set, or a resource group.
- **Safety** depends on the document, the target set, and the permissions of the caller.

Common use cases:

- Kill a suspicious process.
- Disable a scheduled task or service.
- Pull log files to a central bucket.
- Check whether a host is still communicating with a known bad IP.

**Automation for response playbooks**

Automation is the best fit when you need a multi-step, repeatable response workflow. It can call AWS APIs, run scripts, branch on conditions, and pause for approval.

Use Automation when the response requires:

- Multiple actions in sequence.
- Conditional logic.
- Human approval before a destructive step.
- Integration with other AWS services.
- A durable execution history.

A simple containment playbook might:

1. Describe the instance.
2. Replace its security groups with a quarantine group.
3. Create an EBS snapshot of attached volumes.
4. Tag the instance and snapshot with incident metadata.
5. Send a notification to the security team.
6. Create or update an OpsCenter item.

Example Automation document skeleton:

```yaml
schemaVersion: '0.3'
description: "Quarantine an EC2 instance and capture evidence"
parameters:
  InstanceId:
    type: String
mainSteps:
  - name: QuarantineInstance
    action: aws:executeAwsApi
    inputs:
      Service: ec2
      Api: ModifyInstanceAttribute
      InstanceId: '{{ InstanceId }}'
      Groups:
        - sg-0abc1234def567890
  - name: NotifyTeam
    action: aws:executeAwsApi
    inputs:
      Service: sns
      Api: Publish
      TopicArn: arn:aws:sns:us-east-1:123456789012:security-alerts
      Message: 'Instance quarantined: {{ InstanceId }}'
```

> `ModifyInstanceAttribute` for security groups applies to instances in a VPC and replaces the instance’s security groups. Use it carefully and verify the target is correct.

For evidence collection, a real playbook should discover attached volumes before snapshotting them. A robust workflow usually:

- Calls `DescribeInstances`.
- Extracts the attached `VolumeId` values.
- Creates snapshots for each volume.
- Tags snapshots with incident ID, timestamp, and owner.

Example snapshot step pattern:

```yaml
- name: DescribeInstance
  action: aws:executeAwsApi
  inputs:
    Service: ec2
    Api: DescribeInstances
    InstanceIds:
      - '{{ InstanceId }}'
```

In practice, you often use `aws:executeScript` or a Lambda-backed step to parse the response and loop over attached volumes. That is more realistic than hardcoding a single volume ID.

**Approvals and change control**

Automation can pause for manual review before a destructive or high-impact step. This is useful when you want a human to confirm quarantine, snapshotting, termination, or credential revocation.

Use approval gates when:

- The action could disrupt production.
- The evidence collection step is safe, but remediation is not.
- You need separation of duties.
- The incident severity is uncertain.

Common patterns include:

- A security analyst approves quarantine before the instance is isolated.
- A responder approves snapshot creation before any termination.
- A manager approves account-wide remediation in a sensitive environment.

In exam scenarios, approval steps are a strong signal that Automation is the right service rather than Run Command.

**Triggering response actions**

SSM response actions are often triggered by event-driven services rather than manually.

Common trigger sources:

- **Security Hub** findings routed through EventBridge rules.
- **GuardDuty** findings that match a severity threshold or finding type.
- **CloudWatch** alarms that indicate suspicious behavior.
- **Custom EventBridge** rules from application or security tooling.
- **Lambda** functions that normalize findings and start Automation.

A common pattern is:

1. GuardDuty emits a finding.
2. EventBridge matches the finding type.
3. EventBridge starts an Automation document.
4. Automation quarantines the instance and creates evidence artifacts.
5. OpsCenter records the issue for follow-up.

Example EventBridge target pattern:

```json
{
  "source": ["aws.guardduty"],
  "detail-type": ["GuardDuty Finding"],
  "detail": {
    "severity": [{ "numeric": [">=", 7] }]
  }
}
```

This event can target an SSM Automation document directly or invoke Lambda to enrich the finding first.

**OpsCenter and Incident Manager integration**

OpsCenter and Incident Manager are often overlooked, but they matter in security response workflows.

Use **OpsCenter** when you want:

- A central operational record for the issue.
- Related AWS resources linked to the incident.
- Runbook references and remediation tracking.
- A place for responders to coordinate work items.

Use **Incident Manager** when you need:

- A formal incident response process.
- Escalation paths and on-call routing.
- Timeline tracking and response plans.
- Coordination across teams during a live incident.

A practical workflow is to have Automation create or update an OpsItem, then link the incident to an Incident Manager response plan for escalation and communications.

**Evidence collection and forensic safety**

When you collect evidence, preserve the system as much as possible. The goal is to reduce contamination and maintain chain of custody.

Good forensic-safe practices include:

- Prefer snapshots and AMIs over ad hoc file copying when you need disk evidence.
- Tag artifacts with incident ID, time, and collector identity.
- Minimize interactive changes on the host.
- Use read-only or low-impact commands where possible.
- Record who initiated the action and from where.
- Store evidence in controlled buckets with restricted access.

Useful evidence sources:

- **EBS snapshots** for disk state.
- **AMIs** when you need a bootable image of the instance.
- **Run Command output** for command results and quick triage.
- **Session Manager logs** for interactive session transcripts.
- **CloudTrail** for API activity and who initiated the response.
- **VPC Flow Logs** and application logs for network and app context.

**Where to find evidence**

Different SSM features leave different records:

- **Run Command output** shows command results and status for each target.
- **Automation execution history** shows step-by-step workflow progress, inputs, outputs, and failures.
- **Session Manager logs** capture interactive session transcripts if logging is configured.
- **CloudTrail** records API calls such as `StartSession`, `SendCommand`, and Automation-related actions.

Do not assume all evidence is retained in the same place. For exam questions, match the artifact to the service:

- Need command output from a fleet action? Use Run Command history and its output destination.
- Need the exact workflow path and step failures? Use Automation execution history.
- Need a transcript of an interactive investigation? Use Session Manager logs.

**Logging and transcript destinations**

Session Manager preferences control where session transcripts are sent, such as CloudWatch Logs or S3, and can influence session behavior. Encryption at rest is handled by the destination service and KMS, not by Session Manager alone.

Example preference concepts:

- Send transcripts to an S3 bucket for long-term retention.
- Send transcripts to CloudWatch Logs for near-real-time review.
- Use KMS keys to encrypt the destination storage.
- Restrict access to the log bucket or log group with IAM and bucket policies.

A simplified preference example:

```json
{
  "s3BucketName": "security-session-logs",
  "s3KeyPrefix": "ssm/",
  "cloudWatchLogGroupName": "ssm-session-logs",
  "cloudWatchEncryptionEnabled": true,
  "kmsKeyId": "arn:aws:kms:us-east-1:123456789012:key/abcd-1234"
}
```

The exact fields depend on how you configure Session Manager preferences, but the exam takeaway is that transcript routing and storage encryption are separate concerns.

**Document permissions and version control**

SSM documents are powerful, so control who can create, update, share, and execute them.

Key controls:

- Restrict document creation and modification to trusted administrators.
- Limit who can share documents across accounts.
- Use version pinning when executing a document to avoid unexpected changes.
- Review default and latest versions carefully before production use.
- Prefer immutable, reviewed documents for response automation.

Version pinning matters because a document can change after approval. If your automation references `$DEFAULT` or `$LATEST`, you may execute a newer version than the one you reviewed.

Example version pinning:

```bash
aws ssm send-command \
  --document-name MySecurityCheck \
  --document-version 3 \
  --targets "Key=InstanceIds,Values=i-0123456789abcdef0"
```

**IAM and permissions**

SSM response actions require permissions in multiple places:

- The caller needs permission to start sessions, send commands, or start automations.
- The instance profile needs permission to communicate with Systems Manager.
- Automation may need a service role to call EC2, SNS, CloudWatch, or other APIs.
- Cross-service actions may require resource-specific permissions or service-linked roles.

Be careful with EC2 permissions in response workflows:

- Some EC2 actions support resource-level permissions.
- Others require broader permissions or are constrained by the API model.
- For exam questions, focus on least privilege, but recognize that some APIs are not perfectly resource-scoped.

A minimal instance profile commonly includes:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "ssm:UpdateInstanceInformation",
        "ssmmessages:CreateControlChannel",
        "ssmmessages:CreateDataChannel",
        "ssmmessages:OpenControlChannel",
        "ssmmessages:OpenDataChannel",
        "ec2messages:AcknowledgeMessage",
        "ec2messages:DeleteMessage",
        "ec2messages:FailMessage",
        "ec2messages:GetEndpoint",
        "ec2messages:GetMessages",
        "ec2messages:SendReply"
      ],
      "Resource": "*"
    }
  ]
}
```

For managed instances, the managed policy `AmazonSSMManagedInstanceCore` is the common starting point.

**Common response patterns**

Use these patterns to map exam scenarios to the right service:

- **Isolate a compromised EC2 instance**
  - Use Automation or Lambda to replace security groups with a quarantine group.
  - Confirm the instance is in a VPC.
  - Preserve evidence with snapshots before destructive remediation.

- **Collect logs from multiple hosts**
  - Use Run Command with a document that copies logs to S3.
  - Use CloudWatch Logs for centralized output if needed.

- **Investigate a suspicious host interactively**
  - Use Session Manager instead of opening SSH.
  - Enable transcript logging for auditability.

- **Enforce a baseline continuously**
  - Use State Manager associations.
  - Remediate drift on a schedule or at boot.

- **Coordinate a live incident**
  - Use Incident Manager for escalation and response plans.
  - Use OpsCenter to track the operational issue and related resources.

**Troubleshooting response actions**

Common failure modes are often the difference between a working response and a stalled one.

Watch for:

- **Missing SSM Agent**
  - The instance will not appear as managed.
  - Session Manager and Run Command will fail.

- **Incorrect instance profile permissions**
  - The agent cannot register or communicate.
  - Commands may time out or never start.

- **Blocked network access to SSM endpoints**
  - VPC endpoints or internet access may be missing.
  - Private subnets need the right interface endpoints and routing.

- **Unsupported node type**
  - Not every resource is a managed node.
  - Check whether the target is actually supported by Systems Manager.

- **Wrong document or parameters**
  - `AWS-RunShellScript` is for Linux and macOS managed nodes.
  - `AWS-RunPowerShellScript` is for Windows managed nodes.
  - Parameter names must match the document schema.

- **Insufficient permissions for the caller**
  - `StartSession`, `SendCommand`, or `StartAutomationExecution` may be denied.
  - Cross-account document sharing may also block execution.

- **Quarantine step not taking effect**
  - The instance may not be in a VPC.
  - Security group replacement may not be the right containment method for the workload.

**Exam-focused decision guide**

When you read an SCS-C02 question, ask what the response must accomplish:

- If the goal is **interactive inspection without inbound access**, choose Session Manager.
- If the goal is **run a command on one or many instances**, choose Run Command.
- If the goal is **multi-step containment with approvals and API calls**, choose Automation.
- If the goal is **continuous enforcement**, choose State Manager.
- If the goal is **incident coordination and tracking**, choose Incident Manager or OpsCenter.

A few realistic examples:

- A GuardDuty finding indicates crypto-mining on an EC2 instance. Use EventBridge to start Automation, quarantine the instance, snapshot volumes, and create an OpsItem.
- A responder needs to inspect a production host but cannot open SSH. Use Session Manager and log the transcript to S3.
- A fleet-wide configuration drift must be corrected every hour. Use State Manager, not Run Command.
- A security team needs to run a one-time log collection script on 50 instances. Use Run Command.

**Practical containment example**

A common containment sequence is:

1. Replace the instance’s security groups with a quarantine group that only allows SSM and forensic access.
2. Preserve disk evidence with EBS snapshots.
3. Capture metadata, tags, and process state.
4. Notify the incident channel.
5. Record the event in OpsCenter or Incident Manager.

Example CLI for a quarantine change in a VPC:

```bash
aws ec2 modify-instance-attribute \
  --instance-id i-0123456789abcdef0 \
  --groups sg-0deadbeef12345678
```

Use this carefully because it replaces the instance’s security groups. Verify the target instance, the VPC context, and the quarantine group before execution.

**Practical evidence collection example**

You can use Run Command to gather quick triage data:

```bash
aws ssm send-command \
  --document-name AWS-RunShellScript \
  --targets "Key=InstanceIds,Values=i-0123456789abcdef0" \
  --parameters 'commands=["uname -a","whoami","ps aux --sort=-%cpu | head -20","ss -tulpn"]' \
  --output-s3-bucket-name security-evidence-bucket \
  --output-s3-key-prefix triage/i-0123456789abcdef0/
```

This is useful for fast collection, but for stronger forensic preservation you usually pair it with snapshots or AMIs.

**Key takeaways**

- Use **Session Manager** for secure interactive access without inbound SSH or RDP.
- Use **Run Command** for quick, repeatable commands across managed nodes.
- Use **Automation** for incident playbooks, approvals, and AWS API-driven remediation.
- Use **OpsCenter** and **Incident Manager** to track and coordinate response work.
- Preserve evidence with snapshots, AMIs, logs, and CloudTrail, and minimize host changes.
- Validate prerequisites first: SSM Agent, instance profile, network reachability, and supported node type.
- Pin document versions and restrict document permissions to avoid unsafe changes.
