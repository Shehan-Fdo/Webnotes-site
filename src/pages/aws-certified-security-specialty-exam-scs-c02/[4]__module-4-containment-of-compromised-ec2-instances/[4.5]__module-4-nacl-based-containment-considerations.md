## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**NACL containment overview**

A <abbr title="Network Access Control List">NACL</abbr> is a stateless, subnet-level packet filter for Amazon <abbr title="Elastic Compute Cloud">EC2</abbr> in a <abbr title="Virtual Private Cloud">VPC</abbr>. You use it when you need coarse-grained containment at the subnet boundary, especially during incident response, quarantine, or blast-radius reduction.

Because NACLs are stateless, you must explicitly allow both directions of traffic. That makes them useful for isolation, but also easy to misconfigure if you forget return traffic, ephemeral ports, or AWS-managed dependencies.

- **Subnet scope**: A NACL applies to every resource in the associated subnet.
- **Stateless behavior**: Return traffic is not automatically allowed.
- **Ordered evaluation**: Rules are processed from the lowest rule number upward until a match is found.
- **Default deny**: If no rule matches, traffic is denied.

**What NACLs can and cannot filter**

NACLs filter by network attributes, not identity. They are a good fit for IP-based containment, but they do not understand who the caller is or what application is running.

- **Can filter** by:
  - CIDR block
  - Protocol
  - Port range
  - ICMP type/code
  - Allow or deny action
- **Cannot filter** by:
  - Security group
  - IAM user or role
  - Application identity
  - Instance tags
  - Process name or host user

This distinction matters on the exam: if the requirement is identity-aware access control, a security group, IAM policy, or host-based control is usually the better answer. If the requirement is to rapidly quarantine a subnet regardless of instance state, a NACL is often the right containment tool.

**Rule limits and operational constraints**

NACLs have practical limits that matter during containment. In AWS, a NACL supports a finite number of inbound and outbound rules, commonly tested as a maximum of 20 rules per direction by default, with room for quota increases in some accounts and Regions. The exact quota can vary, so treat the limit as a real operational constraint and verify current service quotas when designing response playbooks.

Rule exhaustion is a real incident-response risk:

- If you need many explicit exceptions, you can run out of rule slots.
- If you exhaust the rule space, you may be unable to add a needed allow rule for management access or service dependencies.
- During containment, a “deny-all plus exceptions” strategy can fail if the exception list is too long for the available rules.
- Rule numbering strategy matters because you need room to insert emergency exceptions ahead of broader deny rules.

A practical response plan should reserve rule numbers for:
- emergency management access
- AWS service dependencies
- temporary forensic access
- rollback or recovery rules

**Why NACLs are different from security groups**

Security groups are stateful and instance- or ENI-attached, while NACLs are stateless and subnet-attached. That difference changes how containment behaves in practice.

- **Security groups**
  - Stateful
  - Allow rules only
  - Best for least-privilege workload access
  - Existing connections generally continue if the state table already exists
- **NACLs**
  - Stateless
  - Allow and deny rules
  - Best for subnet quarantine or coarse filtering
  - Existing connections can be disrupted when the subnet association changes or when new packets are blocked

For incident response, this means a NACL can cut off traffic more abruptly than a security group. That is useful for containment, but it can also interrupt legitimate sessions, replication, and management channels.

**Association changes and existing connections**

When you associate a different NACL with a subnet, the new rules apply to traffic going forward. Because NACLs are stateless, they do not preserve session state the way security groups do.

Operationally, this means:

- Existing TCP sessions may fail once return traffic is no longer permitted.
- Long-lived connections such as SSH, database sessions, or application tunnels can drop.
- UDP traffic is especially sensitive because there is no connection state to preserve.
- ICMP-based diagnostics may stop working unless explicitly allowed.

During containment, assume that changing the subnet’s NACL can immediately affect active workloads. Plan for that impact before you quarantine a production subnet.

**Ephemeral ports and return traffic**

Because NACLs are stateless, you must allow the return path explicitly. This is where ephemeral ports matter.

For typical client-server traffic:

- The client uses an ephemeral source port.
- The server listens on a well-known destination port such as 443 or 22.
- Return traffic from the server must be allowed back to the client’s ephemeral port range.

Common exam-relevant guidance:

- For **TCP**, allow the service port in one direction and the ephemeral port range in the reverse direction.
- For **UDP**, you still need explicit reverse-path allowances, but the lack of session state makes timing and port matching less forgiving.
- For **ICMP**, allow the specific type/code pairs you need, not just “ICMP” broadly, if you want tighter control.

A common Linux ephemeral port range is `32768-60999`, but the actual range depends on the operating system and configuration. In AWS exam scenarios, the key point is not the exact range; it is that you must account for client-side ephemeral ports when designing NACL rules.

**Quarantine design pattern**

A strong containment pattern is a deny-all NACL with a small set of explicit exceptions. You usually apply it to a subnet that contains compromised or suspicious instances, then preserve only the traffic needed for management, forensics, and required AWS dependencies.

A practical quarantine design often includes:

- inbound deny-all by default
- outbound deny-all by default
- explicit allow for approved management source IPs
- explicit allow for VPC endpoints used by your tooling
- explicit allow for DNS to the VPC resolver if name resolution is required
- explicit allow for Systems Manager if you rely on it for access
- explicit allow for metadata access only if the instance must reach the instance metadata service locally

> Be careful: a NACL cannot selectively allow “only this instance” inside a subnet. If you quarantine a subnet, every resource in that subnet is affected.

**Example quarantine NACL structure**

The exact rule numbers are illustrative, but the pattern is important: place narrow allows before broad denies, and mirror the logic in both directions.

```text
Inbound rules
100  ALLOW  TCP  443    203.0.113.10/32        # admin workstation or jump host
110  ALLOW  TCP  22     203.0.113.10/32        # SSH if needed
120  ALLOW  TCP  443    10.0.0.0/16            # VPC endpoint access, if applicable
130  ALLOW  UDP  53     10.0.0.2/32            # VPC resolver example
140  ALLOW  ICMP ALL    10.0.0.0/16            # optional diagnostics
900  DENY   ALL  ALL    0.0.0.0/0

Outbound rules
100  ALLOW  TCP  443    203.0.113.10/32        # response to admin access
110  ALLOW  TCP  22     203.0.113.10/32
120  ALLOW  TCP  443    10.0.0.0/16            # VPC endpoint access
130  ALLOW  UDP  53     10.0.0.2/32            # DNS queries if needed
140  ALLOW  TCP  1024-65535 203.0.113.10/32    # return traffic for TCP sessions
150  ALLOW  UDP  1024-65535 10.0.0.0/16        # return traffic for UDP-based flows
900  DENY   ALL  ALL    0.0.0.0/0
```

This is not a universal template. You must tailor the exception list to the actual management path and service dependencies. The important exam concept is that both inbound and outbound rules must be designed together.

**AWS-managed service access during quarantine**

A common failure mode is locking down a subnet so aggressively that you lose access to the very tools you need for investigation. Preserve access to AWS-managed services deliberately.

- **AWS Systems Manager (SSM)**
  - If you use Session Manager for shell access, the instance needs network reachability to SSM endpoints.
  - In private subnets, prefer <abbr title="Virtual Private Cloud">VPC</abbr> endpoints for `ssm`, `ec2messages`, and `ssmmessages`.
  - If you block those paths, you may lose remote shell access even though the instance is still running.
- **EC2 Instance Metadata Service (IMDS)**
  - IMDS is local to the instance at `169.254.169.254`.
  - NACLs are subnet-level filters and generally are not the right control for IMDS access because metadata access is link-local and host-local in nature.
  - For hardening, use IMDSv2 and instance-level controls rather than relying on a NACL.
- **DNS resolver behavior**
  - Instances in a VPC commonly use the AmazonProvidedDNS resolver at the VPC base address plus two, such as `10.0.0.2` in a `10.0.0.0/16` VPC.
  - If you block DNS, name resolution fails and many tools appear broken even when network paths still exist.
  - Allow both UDP and, when needed, TCP 53 to the resolver.
- **VPC endpoints**
  - Interface endpoints let you keep traffic inside AWS without opening internet egress.
  - During containment, endpoints for SSM, CloudWatch Logs, S3, or ECR can preserve investigation and patching workflows.
  - If your response plan depends on endpoints, include them in the quarantine exception set.

**Practical containment workflow**

1. Identify the subnet that contains the suspected workload.
2. Confirm which management and telemetry paths must remain open.
3. Build a quarantine NACL with explicit allow rules for those paths.
4. Associate the NACL with the subnet.
5. Validate access from a known-good management source.
6. Monitor for dropped traffic, failed health checks, and broken dependencies.
7. Refine the exception list or move the workload to a dedicated quarantine subnet if needed.

**CLI examples**

You can inspect and manage NACLs with the AWS CLI. These commands are useful in exam scenarios and real operations.

```bash
aws ec2 describe-network-acls --filters Name=vpc-id,Values=vpc-0123456789abcdef0
aws ec2 create-network-acl --vpc-id vpc-0123456789abcdef0
aws ec2 create-network-acl-entry \
  --network-acl-id acl-0123456789abcdef0 \
  --ingress \
  --rule-number 100 \
  --protocol tcp \
  --port-range From=443,To=443 \
  --cidr-block 203.0.113.10/32 \
  --rule-action allow
aws ec2 replace-network-acl-association \
  --association-id aclassoc-0123456789abcdef0 \
  --network-acl-id acl-0123456789abcdef0
```

To verify the current association and rule order, inspect the ACL before and after changes:

```bash
aws ec2 describe-network-acls \
  --network-acl-ids acl-0123456789abcdef0 \
  --query 'NetworkAcls[0].Entries'
```

**When to use NACL containment versus other controls**

Choose the control that matches the response goal.

| Control | Best use case | Strength | Limitation |
|---|---|---|---|
| NACL | Subnet quarantine, coarse IP/port containment | Fast, stateless, deny support | No identity awareness; rule limits |
| Security group | Workload-level least privilege | Stateful, simpler for app access | No explicit deny; less suited to quarantine |
| Route table blackholing | Stop traffic to a destination or path | Can isolate at routing layer | Less granular; may not stop all desired flows |
| Host-based firewall | Process- or host-specific control | Fine-grained on the instance | Requires host access and agent health |

Exam decision-making often comes down to scope and speed:

- Use a **NACL** when you need to quarantine a subnet quickly.
- Use a **security group** when you need normal application access control.
- Use **route table changes** when you want to remove a path to an external network or segment.
- Use a **host firewall** when you need local, process-aware filtering and you can still manage the host.

**Common pitfalls**

- Forgetting to allow return traffic for ephemeral ports.
- Blocking DNS and then misdiagnosing the issue as an application failure.
- Locking out SSM endpoints and losing remote access during an incident.
- Assuming a NACL can filter by user, role, or application identity.
- Running out of rule slots when trying to preserve too many exceptions.
- Forgetting that NACL changes can disrupt existing sessions immediately.
- Allowing only inbound rules and forgetting that outbound rules are equally important.

**Exam-oriented takeaways**

- NACLs are subnet-level, stateless filters with explicit allow and deny rules.
- They are best for coarse containment and quarantine, not identity-based authorization.
- You must design inbound and outbound rules together, including ephemeral ports.
- Preserve access to management and AWS services before you lock down a subnet.
- Rule limits and association behavior are operational constraints that can make or break an incident response plan.
