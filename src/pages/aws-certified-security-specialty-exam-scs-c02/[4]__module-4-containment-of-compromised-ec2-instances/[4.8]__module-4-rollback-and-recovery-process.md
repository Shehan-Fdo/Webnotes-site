## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Rollback and recovery overview**

Rollback and recovery is the set of actions you use to return a workload to a trusted state after a failed deployment, security incident, data corruption event, or regional disruption. In SCS-C02 scenarios, you are often deciding between three outcomes:

- **Rollback**: revert a recent change to a previous known-good state.
- **Restore**: recover data or infrastructure from a backup, snapshot, or replicated copy.
- **Rebuild/fail over**: redeploy clean infrastructure or switch traffic to a healthy environment when rollback is not possible.

The exam often tests whether you can choose the safest and fastest recovery path while preserving evidence, minimizing blast radius, and respecting encryption, access control, and organizational governance.

**Core recovery decision model**

Use a simple decision tree when analyzing a scenario:

1. Determine whether the issue is caused by a recent change, data loss, compromise, or regional failure.
2. Check whether the affected component supports in-place rollback.
3. Verify whether backups, snapshots, replicas, or versioned artifacts exist.
4. Confirm whether encryption keys, permissions, and network dependencies still allow restore.
5. Decide whether to restore in place, restore to a new environment, or fail over.

A practical rule: if the environment may be compromised, prefer rebuilding from trusted artifacts over “fixing” the existing instance.

**Rollback versus restore versus rebuild**

Rollback is best when the change is recent and reversible. Restore is best when you need to recover data or a service state from a point in time. Rebuild is best when the trust boundary is broken or the service cannot safely be returned to its prior state.

| Action | Best use case | Typical limitation |
|---|---|---|
| Rollback | Bad deployment, config error, feature flag mistake | Not possible if state is destructive or non-reversible |
| Restore | Deleted data, corruption, ransomware, accidental overwrite | Requires valid backup and working encryption/access |
| Rebuild | Compromise, immutable infrastructure, broken dependencies | Takes longer and may require DNS/traffic cutover |

> If you suspect credential theft, malware, or persistence, do not assume rollback is safe. A clean rebuild is often the correct security answer.

**AWS-native backup and restore building blocks**

AWS provides multiple recovery primitives, and the exam expects you to know when each one applies.

- **AWS Backup** centralizes backup policy, scheduling, retention, and cross-account/cross-Region copy for supported services.
- **Snapshots** capture point-in-time storage state for services such as Amazon EBS and Amazon RDS.
- **Versioning** protects object data in Amazon S3 against overwrite and delete operations.
- **Replication** supports faster recovery by maintaining copies in another Region or account.
- **Infrastructure as Code (IaC)** such as AWS CloudFormation or AWS CDK helps you recreate environments consistently.

Use these together rather than treating backup as a single feature. For example, you may restore data from a snapshot, redeploy application servers from CloudFormation, and reattach IAM policies and security groups from code.

**Service-specific recovery nuances**

Different AWS services recover differently, and the exam often distinguishes between data-plane recovery and control-plane rebuilds.

- **Amazon EBS**
  - Use snapshots to restore volumes.
  - You can create a new volume from a snapshot and attach it to an instance.
  - Snapshots are incremental, but restore creates a full usable volume.
- **Amazon RDS**
  - Use automated backups, manual snapshots, or point-in-time recovery (PITR).
  - You typically restore to a new DB instance, then cut over applications.
- **Amazon Aurora**
  - Use cluster snapshots, PITR, or restore to a new cluster.
  - Aurora supports fast recovery patterns compared with traditional database rebuilds.
  - Aurora backtracking can rewind a cluster to a previous time without restoring from backup, but it is not a substitute for long-term backup retention.
- **Amazon EFS**
  - Use AWS Backup or EFS backup features to recover file systems.
  - Recovery usually means creating a new file system and restoring data into it.
  - Plan for mount target and security group recreation after restore.
- **Amazon FSx**
  - Recovery depends on the FSx variant.
  - Use backups and restore workflows to create a new file system from a backup.
  - Recreate network placement, DNS aliases, and client mappings as needed.
- **Amazon S3**
  - Use versioning, MFA delete, object lock, and replication to protect against deletion or tampering.
  - Recovery may mean restoring previous object versions or copying from a replica bucket.
- **Amazon DynamoDB**
  - Use on-demand backups, PITR, and global tables for resilience.
  - Restore creates a new table; it does not overwrite the original table in place.
- **Amazon EC2**
  - Recovery often means launching a new instance from an AMI or rebuilding from automation.
  - If the instance stores state locally, you need a separate data recovery plan.

**Database recovery patterns**

Database recovery is a common exam theme because the right answer depends on whether you need the latest data, a clean copy, or a fast failover.

- **Point-in-time recovery (PITR)** lets you restore to a specific timestamp before corruption or deletion.
- **Read replica promotion** can reduce downtime when the primary is unavailable, but it is a failover pattern, not a backup.
- **Cross-Region snapshot copy** helps with regional disaster recovery and data residency planning.
- **Backtracking in Aurora** is useful for rapid reversal of operational mistakes, especially schema or application changes.

Practical distinctions:

- Use **PITR** when a bad write or delete happened and you need a precise recovery point.
- Use **replica promotion** when the primary is down and the replica is current enough to serve traffic.
- Use **snapshot restore** when you need a durable recovery point or a clean rebuild.
- Use **backtracking** when you need to undo recent changes quickly and the feature is supported.

Example RDS restore workflow:

```bash
aws rds restore-db-instance-to-point-in-time \
  --source-db-instance-identifier prod-db \
  --target-db-instance-identifier prod-db-restore \
  --restore-time 2026-04-26T10:15:00Z
```

Example Aurora cluster restore workflow:

```bash
aws rds restore-db-cluster-from-snapshot \
  --db-cluster-identifier aurora-recovery \
  --snapshot-identifier aurora-prod-snap-20260426
```

**Encryption and KMS dependencies**

Recovery can fail even when a backup exists if the encryption key is unavailable. This is a high-value exam concept.

- **AWS KMS (Key Management Service)** keys protect encrypted backups, snapshots, and objects.
- If the **customer managed key (CMK)** is disabled, pending deletion, or deleted, restore operations may fail or become impossible.
- If the key policy or grants no longer allow the recovery role to use the key, the restore can fail with access errors.
- Cross-account restore requires that the destination account can decrypt or re-encrypt data according to the service’s supported workflow.

Operational implications:

- Keep key policies and grants aligned with recovery roles.
- Protect recovery keys with separate governance and break-glass access.
- Test restores after key rotation, policy changes, and account reorganizations.
- Document whether the backup is encrypted with an AWS managed key or a customer managed key, because that affects portability and control.

> A backup is only useful if you can decrypt it during recovery. Key availability is part of the recovery plan, not an afterthought.

**Multi-account recovery governance**

In mature AWS environments, recovery is not just a technical action; it is a governance process. AWS Organizations and AWS Control Tower help you manage guardrails, account structure, and delegated administration across multiple accounts.

- Use **AWS Organizations** to centralize policy, backup governance, and account-level controls.
- Use **AWS Control Tower** to standardize landing zones, account vending, and baseline guardrails.
- Use separate accounts for production, backup, security tooling, and disaster recovery when possible.
- Apply service control policies (SCPs) to prevent accidental deletion of critical backups or logs.
- Use cross-account backup vaults or copy jobs to reduce the chance that a single compromised account can destroy all recovery points.

A common secure pattern is:

- Production account creates backups.
- Backup account stores immutable or long-retention copies.
- Security account monitors backup health and restore events.
- Recovery account hosts warm standby or pilot-light resources.

This separation improves blast-radius control and supports forensic preservation if the source account is compromised.

**Backup immutability and anti-tamper controls**

Recovery planning should assume that an attacker may try to delete backups before you can restore them.

- Use **S3 Object Lock** for immutable retention where supported.
- Use backup vault access policies to restrict deletion.
- Use MFA and least privilege for backup administration.
- Separate backup operators from workload administrators.
- Enable logging for backup, restore, and delete actions.

Useful controls include:

- CloudTrail for API audit trails.
- AWS Config for configuration drift and compliance checks.
- Amazon EventBridge for alerting on backup failures or restore events.
- AWS Backup vault lock for write-once, delete-protected retention.

**Testing restores and proving recovery works**

A backup is not a recovery capability until you have tested it. The exam may ask how you validate readiness, not just how you create backups.

- Perform periodic restore drills on a schedule that matches business criticality.
- Test both data restoration and application rehydration.
- Verify that restored resources pass health checks and can serve traffic.
- Record evidence such as timestamps, restore identifiers, checksums, and application validation results.
- Include encryption key access, IAM permissions, DNS updates, and security group rules in the test.

A practical recovery drill should answer:

- Can you restore within the recovery time objective (RTO)?
- Is the restored data within the recovery point objective (RPO)?
- Can the application authenticate, connect, and process requests after restore?
- Are logs, monitoring, and alerting working in the recovered environment?

Example restore validation checklist:

- Confirm backup job succeeded.
- Restore to an isolated test account or VPC.
- Run application smoke tests.
- Verify data integrity with row counts or checksums.
- Capture evidence in a ticket or change record.
- Destroy the test restore if it is no longer needed.

**Operationalizing periodic recovery drills**

Use automation so recovery testing is repeatable and auditable.

1. Schedule backup creation and restore tests with AWS Backup plans or automation pipelines.
2. Restore into a non-production environment or isolated subnet.
3. Run scripted validation against the restored service.
4. Publish results to a ticketing system or compliance repository.
5. Review failures and update runbooks, IAM permissions, and key policies.

Example validation script pattern:

```bash
#!/usr/bin/env bash
set -euo pipefail

DB_HOST="restored-db.example.internal"
EXPECTED_ROWS=1000

actual_rows=$(psql "host=${DB_HOST} dbname=app user=readonly" -tAc "select count(*) from orders;")
if [ "$actual_rows" -lt "$EXPECTED_ROWS" ]; then
  echo "Restore validation failed"
  exit 1
fi

echo "Restore validation passed"
```

**Disaster recovery orchestration patterns**

Recovery orchestration is about sequencing dependencies correctly. The exam may present a scenario where data is restored but the app still fails because IAM, DNS, or network components were not rebuilt.

Common patterns:

- **Backup and restore**: lowest cost, highest RTO.
- **Pilot light**: minimal core services are always on; scale up during recovery.
- **Warm standby**: a smaller but functional environment is always running.
- **Multi-site active/active**: highest resilience, most complex and expensive.

Scenario guidance:

- Use backup and restore for non-critical or cost-sensitive workloads.
- Use pilot light when you need faster recovery than a full rebuild.
- Use warm standby when downtime must be limited and you can afford duplicate infrastructure.
- Use active/active when the workload can tolerate distributed complexity and needs very low RTO.

**Rollback limitations and when rebuild is required**

Not every failure can be rolled back. Some changes are destructive or irreversible.

- Dropped data without a backup usually requires rebuild and data re-entry.
- Compromised AMIs, containers, or deployment artifacts should be replaced, not repaired.
- If secrets may have been exposed, rotate them before or during recovery.
- If the application state is inconsistent across tiers, a clean redeploy may be safer than partial rollback.
- If the service does not support in-place rollback, restore to a new resource and cut over.

Examples of rebuild-required situations:

- A compromised EC2 instance with unknown persistence mechanisms.
- A deleted RDS instance with no snapshot or PITR window.
- A corrupted EFS file system with no valid backup.
- A broken deployment where schema migrations cannot be reversed safely.

**Scenario-based examples**

A few exam-style distinctions help you choose the right answer quickly.

- A developer deletes a table in Amazon Aurora 10 minutes ago:
  - Use PITR or Aurora backtracking if available.
  - Restore to a new cluster, validate, then cut over.
- A production EC2 instance is compromised:
  - Rebuild from a trusted AMI and redeploy from IaC.
  - Rotate credentials and inspect logs before reintroducing traffic.
- An Amazon EFS file share is corrupted:
  - Restore to a new file system from backup.
  - Recreate mount targets and update clients.
- A Region becomes unavailable:
  - Promote replicas or restore from cross-Region copies in the DR Region.
  - Update DNS, load balancers, and application endpoints.
- A KMS key used for encrypted snapshots is disabled:
  - Recovery may be blocked until the key is re-enabled or an alternate decrypt path exists.
  - If the key is deleted, the backup may be unrecoverable.

**CLI examples for recovery operations**

AWS CLI is useful for understanding the mechanics of recovery, even if the exam is conceptual.

Create an EBS snapshot:

```bash
aws ec2 create-snapshot \
  --volume-id vol-0123456789abcdef0 \
  --description "Pre-change backup"
```

Restore an EBS volume from a snapshot:

```bash
aws ec2 create-volume \
  --availability-zone us-east-1a \
  --snapshot-id snap-0123456789abcdef0
```

Copy a snapshot to another Region:

```bash
aws ec2 copy-snapshot \
  --source-region us-east-1 \
  --source-snapshot-id snap-0123456789abcdef0 \
  --destination-region us-west-2
```

Restore a DynamoDB table from backup:

```bash
aws dynamodb restore-table-from-backup \
  --target-table-name OrdersRestored \
  --backup-arn arn:aws:dynamodb:us-east-1:123456789012:table/Orders/backup/01500000000000-abc123
```

**Exam signals to watch for**

Look for wording that hints at the correct recovery strategy.

- “Recent change” often suggests rollback or backtracking.
- “Deleted data” often suggests restore or PITR.
- “Compromised system” often suggests rebuild and credential rotation.
- “Regional outage” often suggests failover or cross-Region recovery.
- “Encrypted backup cannot be restored” often points to KMS dependency issues.
- “Need to prove recovery” often points to restore testing and evidence collection.

**Practical recovery checklist**

Use this checklist to structure your answer in scenario questions:

1. Identify the failure type and blast radius.
2. Preserve evidence if compromise is possible.
3. Confirm backup, snapshot, replica, or version history availability.
4. Verify KMS key, IAM, and network access for restore.
5. Choose rollback, restore, failover, or rebuild.
6. Validate the recovered workload with tests and logs.
7. Document the recovery outcome and update the runbook.

**Key takeaways**

Rollback and recovery in AWS is about matching the failure mode to the right recovery primitive. The best answer is not always the fastest one; it is the one that preserves trust, meets RTO and RPO targets, and respects encryption and governance constraints. For SCS-C02, be ready to distinguish backup from replication, restore from failover, and rollback from rebuild.
