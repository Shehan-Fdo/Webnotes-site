## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Scenario framing**

A lab account is a common exam setting for suspicious activity questions because it isolates blast radius, but it still needs a real incident response mindset. You usually need to answer three questions quickly:

- What happened?
- What is still happening?
- What evidence must you preserve before you change anything?

In AWS, the first useful signals often come from **CloudTrail** for API activity, **GuardDuty** for threat detections, **AWS Config** for configuration drift, and **VPC Flow Logs** or application logs for network or host confirmation. Do not assume one service is enough for every case. If the suspicious behavior is network-heavy or host-heavy, CloudTrail may show the control-plane change, but not the full payload or packet-level proof.

**Investigation workflow**

A practical workflow is to build a timeline from identity, configuration, and network evidence. Start with the alert source, then correlate across services.

```text
Alert or user report
        ↓
CloudTrail event timeline
        ↓
AWS Config resource change history
        ↓
GuardDuty finding details
        ↓
VPC Flow Logs / CloudWatch Logs / OS logs
        ↓
Containment and evidence preservation
```

Use this sequence because each service answers a different question:

- **CloudTrail** tells you who called which API, from where, and with what result.
- **AWS Config** tells you what changed in resource configuration and when.
- **GuardDuty** tells you what suspicious behavior was detected and how severe it appears.
- **VPC Flow Logs** help confirm traffic patterns between sources, destinations, and ports.
- **CloudWatch Logs** or workload logs help confirm application-level behavior when the issue is inside the instance or service.

**CloudTrail as the primary identity and API source**

CloudTrail records management events by default and can also record data events for selected resources. This distinction matters in exam questions.

- **Management events** cover control-plane actions such as `RunInstances`, `AuthorizeSecurityGroupIngress`, `PutBucketPolicy`, `CreateAccessKey`, and `AssumeRole`.
- **Data events** cover high-volume resource activity such as S3 object-level access and Lambda function invocation, depending on what you enable.

Use management events when you need to know who changed permissions, networking, logging, or compute state. Use data events when the suspicious activity is about object access, exfiltration, or sensitive file reads.

A typical CloudTrail event review focuses on:

- `eventTime`
- `eventName`
- `eventSource`
- `awsRegion`
- `sourceIPAddress`
- `userAgent`
- `userIdentity`
- `requestParameters`
- `responseElements`
- `errorCode`

For IAM users, `userIdentity.type` is often `IAMUser`. For temporary credentials, you may see `AssumedRole`, `FederatedUser`, or related session context fields. Do not assume every identity type has the same session structure. For example, an IAM user event does not behave like an assumed-role session event.

Example event shape:

```json
{
  "eventTime": "2026-04-26T12:14:22Z",
  "eventSource": "ec2.amazonaws.com",
  "eventName": "AuthorizeSecurityGroupIngress",
  "awsRegion": "us-east-1",
  "sourceIPAddress": "203.0.113.10",
  "userAgent": "aws-cli/2.15.0",
  "userIdentity": {
    "type": "AssumedRole",
    "arn": "arn:aws:sts::111122223333:assumed-role/LabAdmin/session-1234"
  },
  "requestParameters": {
    "groupId": "sg-0123456789abcdef0",
    "ipPermissions": [
      {
        "ipProtocol": "tcp",
        "fromPort": 22,
        "toPort": 22,
        "ipRanges": [{"cidrIp": "0.0.0.0/0"}]
      }
    ]
  },
  "responseElements": {
    "return": true
  }
}
```

That event suggests a security group was opened to the internet. The next question is whether the change was intentional, whether it enabled access to a sensitive host, and whether any follow-on activity occurred after the change.

**CloudTrail Lake and log search**

CloudTrail Lake can help you query event history, but the table name depends on the event data store configuration. Do not memorize a single default table name. In exam scenarios, the important point is that you can query the event data store for a time-bounded investigation.

Example query pattern:

```sql
SELECT eventTime, eventName, eventSource, sourceIPAddress, userIdentity.arn
FROM <your_event_data_store_table>
WHERE eventTime >= '2026-04-26T12:00:00Z'
  AND eventTime <  '2026-04-26T13:00:00Z'
  AND eventSource IN ('ec2.amazonaws.com', 'iam.amazonaws.com', 's3.amazonaws.com')
ORDER BY eventTime ASC;
```

Useful filters during suspicious activity review:

- `eventName IN (...)` for targeted actions such as `CreateAccessKey`, `PutBucketPolicy`, `AttachRolePolicy`, `AuthorizeSecurityGroupIngress`
- `sourceIPAddress` to identify unusual geolocation or proxy use
- `userIdentity.arn` to trace the principal
- `errorCode IS NOT NULL` to find failed attempts that may indicate probing

**CloudTrail log integrity and tamper detection**

CloudTrail is not just a log source; it is also a forensic control. In exam questions, you should know how to protect and validate it.

- Enable **log file validation** so CloudTrail can create digest files that help detect tampering.
- Centralize trails into a dedicated **log archive account** with restricted write access.
- Use a separate security or audit account to read logs without being able to modify them.
- Store logs in S3 with versioning and tight bucket policies.
- Consider cross-account delivery so the source account cannot easily erase its own evidence.

If you suspect tampering:

- Check whether the trail is still delivering logs.
- Verify the S3 bucket policy and KMS key policy if encryption is used.
- Review CloudTrail digest files and validation status.
- Compare the expected event volume against the observed gap.

> If the attacker has access to the same account that stores the logs, assume local evidence may be incomplete until you confirm delivery to an immutable or separate archive location.

**AWS Config for configuration drift**

AWS Config answers a different question than CloudTrail: not “who called the API?” but “what is the current and historical configuration of the resource?”

Use Config when you need to confirm:

- When a security group rule changed
- Whether an S3 bucket became public
- Whether an IAM policy or role trust policy changed
- Whether encryption, logging, or versioning was disabled
- Whether a resource is compliant with a rule or conformance pack

Config is especially useful when CloudTrail shows the change event, but you need the before-and-after state. For example, CloudTrail may show `PutBucketPolicy`, while Config shows the exact policy diff and the time the bucket became noncompliant.

**GuardDuty for threat detection**

GuardDuty is a managed threat detection service. It analyzes signals such as CloudTrail, VPC Flow Logs, and DNS logs to identify suspicious behavior.

Use GuardDuty findings to prioritize response when you see:

- Unusual API calls from a new location
- Credential exfiltration indicators
- Reconnaissance activity
- Port scanning or brute-force behavior
- Suspicious S3 access patterns
- Instance compromise indicators

GuardDuty is not a replacement for raw evidence. It is a high-value lead that tells you where to look next. In an exam scenario, a GuardDuty finding often points you to the right account, principal, resource, or network path, but you still confirm with CloudTrail, Config, or logs.

**VPC Flow Logs and network confirmation**

When the suspicious activity involves traffic patterns, use VPC Flow Logs to confirm whether a connection actually occurred. Flow logs help you answer:

- Did traffic reach the instance or ENI?
- Which source and destination addresses were involved?
- Which ports were used?
- Was the traffic accepted or rejected?

Flow logs are especially useful when CloudTrail shows a security group change and you need to verify whether the new rule was used. They are also useful when GuardDuty reports port scanning or unusual outbound connections.

Example interpretation:

- `ACCEPT` on port 22 after a security group change suggests possible SSH access.
- `REJECT` on repeated outbound connections may indicate blocked exfiltration or failed beaconing.
- A sudden spike in traffic to an unfamiliar IP may indicate command-and-control or data transfer.

**Evidence preservation before containment**

Preserve evidence before you make disruptive changes whenever possible. In a lab account, you may still need to act quickly, but you should avoid destroying the timeline.

Good preservation actions include:

- Exporting CloudTrail logs to a secured S3 bucket
- Taking EBS snapshots of affected volumes
- Capturing instance metadata, tags, and security group state
- Saving GuardDuty finding details and timestamps
- Recording IAM policy versions and trust policies
- Preserving Config snapshots or configuration history
- Copying relevant CloudWatch Logs to a durable location

For EC2 instances, EBS snapshots are often the first forensic artifact to preserve because they capture disk state without booting the instance. If the instance is still running and you need memory evidence, that is a more advanced workflow and is not always available in exam scenarios, so do not overstate it.

**Containment actions in a lab account**

Containment should reduce risk without erasing evidence. The exact action depends on the principal and the resource.

For identity compromise:

- Disable or rotate IAM user access keys.
- Reset the IAM user password if console access is possible.
- Remove or detach suspicious policies.
- Revoke sessions where supported for the identity type and service.
- Review and restrict role trust policies if a role was abused.

For account-level emergency response:

- Disable root access keys if they exist.
- Enforce a root password reset and MFA review.
- Apply an SCP (Service Control Policy) in AWS Organizations to restrict the lab account if it is part of an organization.
- Temporarily deny high-risk actions such as IAM changes, KMS key changes, or network exposure changes.

For resource containment:

- Remove public ingress rules.
- Isolate the instance in a quarantine security group.
- Stop or snapshot the instance if that preserves evidence better than termination.
- Restrict S3 bucket access and block public access.
- Disable suspicious event sources or integrations.

> Changing an IAM user’s access key does not instantly revoke already-issued temporary sessions. For assumed-role or federated sessions, you usually need to rely on session expiration or supported revocation mechanisms rather than assuming a key rotation alone ends active access.

**Trust policy and permission review**

When a role is involved, inspect both the permission policy and the trust policy.

- The **permission policy** controls what the role can do after it is assumed.
- The **trust policy** controls who can assume the role.

A suspicious activity review often finds one of these patterns:

- Overly broad trust, such as allowing an unexpected principal to assume the role
- Overly broad permissions, such as `AdministratorAccess` or wildcard actions on sensitive resources
- Privilege escalation paths through `iam:PassRole`, `sts:AssumeRole`, or policy attachment permissions

Example of a safer trust policy pattern that limits assumption to a specific principal and optional external ID for third-party access:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::444455556666:role/TrustedAutomationRole"
      },
      "Action": "sts:AssumeRole",
      "Condition": {
        "StringEquals": {
          "sts:ExternalId": "lab-automation-2026"
        }
      }
    }
  ]
}
```

Use conditions that match the actual access pattern. Do not rely on redundant or misleading conditions that simply repeat the principal ARN.

**IAM Access Analyzer in suspicious access review**

IAM Access Analyzer helps identify resources that are accessible from outside the account or organization. It is useful when you suspect unintended exposure.

Use it to check for:

- Public S3 bucket access
- Cross-account access to roles, KMS keys, or other resources
- Resource policies that allow external principals
- Unexpected sharing after a policy change

In a suspicious activity scenario, Access Analyzer can quickly confirm whether a policy change created unintended exposure. It does not replace CloudTrail, but it helps you assess blast radius.

**Decision guidance: CloudTrail vs Config vs GuardDuty vs CloudWatch Logs**

When exam questions ask which service to use, choose based on the question being asked.

| Need | Best fit | Why |
|---|---|---|
| Who made the API call | CloudTrail | Records identity, source, time, and API action |
| What changed in resource configuration | AWS Config | Tracks configuration history and compliance |
| What suspicious behavior was detected | GuardDuty | Produces threat findings from multiple signals |
| What traffic actually flowed | VPC Flow Logs | Shows accepted/rejected network flows |
| What the application or OS logged | CloudWatch Logs or host logs | Captures workload-level detail |

A common trap is using CloudTrail for everything. CloudTrail is essential, but it does not show packet payloads, full application traces, or all host activity.

**End-to-end incident example: exposed SSH and suspicious login**

A realistic lab-account scenario might unfold like this:

1. A GuardDuty finding reports `UnauthorizedAccess:EC2/SSHBruteForce`.
2. CloudTrail shows `AuthorizeSecurityGroupIngress` added `0.0.0.0/0` to port 22 on a security group.
3. AWS Config confirms the security group remained open for 18 minutes.
4. VPC Flow Logs show repeated accepted connections from an unfamiliar IP range.
5. CloudTrail shows an `AssumeRole` event or console login from a new source IP shortly after the exposure.
6. You preserve the instance with an EBS snapshot, export the relevant logs, and then quarantine the security group.

This timeline helps you distinguish between a harmless lab change and a real compromise. The key is correlation across services, not a single alert.

**End-to-end incident example: S3 policy change and data access**

Another common scenario is suspicious S3 access:

1. CloudTrail records `PutBucketPolicy` or `PutBucketPublicAccessBlock`.
2. AWS Config shows the bucket became publicly accessible or cross-account accessible.
3. IAM Access Analyzer flags external access.
4. CloudTrail data events show `GetObject` activity on sensitive objects.
5. GuardDuty may raise a finding if the access pattern looks anomalous.
6. You restore the policy, block public access, preserve the logs, and review whether any objects were downloaded.

This is where data events matter. If the question is about object-level reads or writes, management events alone are not enough.

**Practical CLI examples**

Use the AWS CLI to inspect recent events, resource configuration, and findings.

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=AuthorizeSecurityGroupIngress \
  --start-time 2026-04-26T12:00:00Z \
  --end-time 2026-04-26T13:00:00Z
```

```bash
aws configservice get-resource-config-history \
  --resource-type AWS::EC2::SecurityGroup \
  --resource-id sg-0123456789abcdef0 \
  --chronological-order Reverse
```

```bash
aws guardduty list-findings --detector-id <detector-id>
```

```bash
aws ec2 describe-security-groups \
  --group-ids sg-0123456789abcdef0
```

```bash
aws iam get-role \
  --role-name LabAdmin
```

```bash
aws s3api get-bucket-policy \
  --bucket suspicious-lab-bucket
```

These commands help you verify the current state after you identify the suspicious event. In an exam, you do not need to memorize every flag, but you should know which service answers which question.

**Containment checklist for a lab account**

Use a disciplined sequence so you do not lose evidence.

1. Record the alert time, affected account, region, and resource IDs.
2. Export or copy logs and findings to a protected location.
3. Snapshot affected EBS volumes or preserve other critical artifacts.
4. Remove public exposure or quarantine the affected resource.
5. Disable or rotate compromised credentials.
6. Review IAM roles, trust policies, and recent policy changes.
7. Check for cross-account access with IAM Access Analyzer.
8. Confirm whether the activity stopped using CloudTrail and Flow Logs.

**Exam-oriented takeaways**

- Use **CloudTrail** to identify the actor and API activity.
- Use **AWS Config** to confirm configuration drift and historical state.
- Use **GuardDuty** to prioritize suspicious behavior and likely compromise.
- Use **VPC Flow Logs** and workload logs to confirm network or host-level evidence.
- Preserve evidence before containment whenever possible.
- Remember that management events and data events serve different investigative needs.
- Treat log integrity as part of the security posture, not an afterthought.
- In account-level emergencies, consider root credential review, SCP restrictions, and isolation of the lab account.
