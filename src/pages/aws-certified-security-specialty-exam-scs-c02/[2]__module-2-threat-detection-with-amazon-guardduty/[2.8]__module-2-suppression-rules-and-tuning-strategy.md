## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Overview**

Suppression and tuning reduce alert noise without hiding real risk. In AWS security services, the right approach depends on where the finding originates, whether you want to mute a single finding or a class of findings, and whether you need to preserve evidence for audit and response.

The exam often tests whether you can distinguish between:
- **Suppress at source** by changing the detector, scan scope, or configuration that generates the finding.
- **Suppress in the aggregator** by using Security Hub workflow updates or automation rules.
- **Archive or mute** findings in the source service when the service supports it.
- **Route exceptions** by environment, account, tag, or resource scope.

A good tuning strategy keeps signal high, preserves traceability, and avoids permanently disabling detections that matter.

**Core concepts**

<dl>
<dt><strong>Suppression</strong></dt>
<dd>Reducing alerting for a known-acceptable condition while keeping the finding available for audit, reporting, or later review.</dd>
<dt><strong>Archiving</strong></dt>
<dd>Moving a finding out of the active queue in a source service such as GuardDuty, while retaining it for history and investigation.</dd>
<dt><strong>Workflow status</strong></dt>
<dd>Security Hub finding state used for triage: <code>NEW</code>, <code>NOTIFIED</code>, or <code>RESOLVED</code>.</dd>
<dt><strong>Automation rule</strong></dt>
<dd>Security Hub rule that matches findings and applies actions such as changing workflow status, adding notes, or updating severity.</dd>
</dl>

> <mark>Important:</mark> Security Hub workflow status does not include `SUPPRESSED`. If you need suppression behavior, use automation rules, notes, workflow updates, or source-service controls.

**Security Hub tuning strategy**

Security Hub is the central place where findings from many services converge, so it is often the best place to standardize triage. However, Security Hub does not provide a universal “suppress all findings” switch. Instead, you tune findings with workflow updates, automation rules, and careful filtering.

Use Security Hub tuning when:
- You want a centralized exception process across multiple accounts.
- You need to preserve the finding but reduce paging or ticket creation.
- You want to tag or classify findings for downstream workflows.
- You need to manage imported findings from AWS services and partner products consistently.

Common Security Hub actions:
- Set workflow status to `NOTIFIED` or `RESOLVED`.
- Add a note explaining the exception.
- Use automation rules to apply consistent handling.
- Filter findings in dashboards, insights, or EventBridge consumers.

**Security Hub workflow handling**

Security Hub workflow status is a triage signal, not a suppression engine.

- `NEW` means the finding has not been triaged.
- `NOTIFIED` means the finding has been acknowledged or routed.
- `RESOLVED` means the issue is considered addressed or accepted.

A practical pattern is:
- Keep the finding in Security Hub.
- Add a note with the exception rationale.
- Set workflow to `RESOLVED` only when the condition is truly accepted or fixed.
- Use automation rules to standardize this for known benign cases.

Example CLI update for a Security Hub finding:

```bash
aws securityhub batch-update-findings \
  --finding-identifiers ProductArn=arn:aws:securityhub:us-east-1:123456789012:product/123456789012/default,Id=example-finding-id \
  --workflow-status RESOLVED \
  --note Text="Accepted risk for non-production benchmark host; tracked in change request CR-1042",UpdatedBy="security-team"
```

Notes:
- `ProductArn` must match the originating product for the finding.
- `--workflow-status` is the correct parameter for workflow changes.
- `--note` records the reason for the decision.

**Security Hub automation rules**

Automation rules are the closest thing to built-in suppression logic in Security Hub, but they are not the same as a source-service suppression feature.

Use automation rules to:
- Match findings by attributes such as account, severity, type, resource tags, or compliance status.
- Change workflow status.
- Add notes.
- Update severity or user-defined fields where supported.

Typical use cases:
- Auto-resolve low-risk findings in non-production accounts.
- Route findings from a known scanner account to a specific team.
- Mark recurring benchmark findings as reviewed after validation.

A conceptual automation rule pattern:

```json
{
  "RuleName": "ResolveKnownNonProdFindings",
  "RuleOrder": 1,
  "Description": "Auto-handle approved non-production findings",
  "IsTerminal": true,
  "Criteria": {
    "ProductName": [
      {
        "Comparison": "EQUALS",
        "Value": "Inspector"
      }
    ],
    "AwsAccountId": [
      {
        "Comparison": "EQUALS",
        "Value": "111122223333"
      }
    ],
    "SeverityLabel": [
      {
        "Comparison": "EQUALS",
        "Value": "LOW"
      }
    ]
  },
  "Actions": [
    {
      "Type": "FINDING_FIELDS_UPDATE",
      "FindingFieldsUpdate": {
        "Workflow": {
          "Status": "RESOLVED"
        },
        "Note": {
          "Text": "Approved non-production exception",
          "UpdatedBy": "security-automation"
        }
      }
    }
  ]
}
```

Use this as a conceptual pattern only. In practice, automation rules are created in the Security Hub console or through the Security Hub automation rule APIs, and the exact schema must match the service model.

**Security Hub vs suppression**

Security Hub does not offer a universal suppression status. In exam terms, think of Security Hub as:
- A **finding aggregator** and triage layer.
- A place to **standardize workflow**.
- A place to **document exceptions**.
- Not a replacement for source-service tuning.

If you need to stop a finding from being generated repeatedly, prefer source-side tuning first.

**GuardDuty tuning strategy**

GuardDuty is a threat detection service, so tuning usually happens at the detector level or by archiving specific findings. GuardDuty does not use Security Hub-style workflow statuses.

Use GuardDuty tuning when:
- A detector repeatedly generates benign findings for a known workload.
- You want to reduce noise from a specific finding type or resource pattern.
- You need to exclude a trusted source or known benign behavior from alerting.

GuardDuty tuning mechanisms include:
- **Trusted IP lists** for known safe IP ranges.
- **Threat lists** for known malicious IPs or domains.
- **Finding archiving** for accepted or investigated findings.
- **Detector-level suppression rules** for recurring patterns.

> GuardDuty suppression is not the same as Security Hub workflow resolution. GuardDuty archived findings remain in history, but they no longer appear as active findings.

**GuardDuty archived findings**

Archiving is useful when the finding is valid but no longer actionable.

- Archived findings remain searchable.
- Archived findings can still support audit and trend analysis.
- Archiving does not stop the detector from generating future matching findings unless a suppression rule or source condition prevents recurrence.

A practical operational pattern:
- Investigate the finding.
- Confirm it is benign or accepted.
- Archive the current finding.
- Add a suppression rule if the pattern is expected to recur.

**GuardDuty suppression rules**

GuardDuty suppression rules are detector-scoped and match on finding attributes. They are used to automatically archive matching findings.

Typical supported criteria include:
- Finding type.
- Severity.
- Account.
- Resource.
- Service-specific attributes.
- Region or detector context, depending on the finding type.

Operational behavior:
- Matching findings are archived automatically.
- The rule applies to future findings that match the criteria.
- You should keep criteria narrow to avoid hiding real threats.

Conceptual CLI pattern:

```bash
aws guardduty create-filter \
  --detector-id a1b2c3d4e5f6 \
  --name "ArchiveKnownScannerNoise" \
  --action ARCHIVE \
  --finding-criteria '{
    "service.archived": {
      "Criterion": "false"
    },
    "severity": {
      "Gte": 2
    },
    "type": {
      "Eq": ["Recon:EC2/Portscan"]
    }
  }'
```

Use this as a conceptual example of the intent: create a detector filter that archives a known benign pattern. The exact criteria keys must match the GuardDuty finding schema for the finding type you are targeting.

**GuardDuty trusted IP lists and threat lists**

These lists are useful for reducing noise, but they are not a blanket suppression mechanism.

- **Trusted IP lists** help GuardDuty treat known safe IPs differently for certain detections.
- **Threat lists** help GuardDuty identify known malicious indicators.
- They are best used for stable, well-understood network boundaries.

Use them when:
- You have fixed corporate egress IPs.
- You have approved partner ranges.
- You need to reduce false positives from known scanners or monitoring systems.

Avoid overusing them:
- Do not add broad ranges just to silence alerts.
- Do not treat them as a substitute for investigation.
- Do not assume they apply to every finding type.

**Inspector tuning strategy**

Amazon Inspector focuses on vulnerability and exposure findings. The main tuning approach is usually not “suppression” in the Security Hub sense. Instead, you reduce noise by controlling what gets scanned, what is in scope, and what is considered acceptable risk.

Use Inspector tuning when:
- You want to exclude resources or packages from scanning scope.
- You want to avoid repeated findings on intentionally vulnerable test assets.
- You want to manage findings through remediation and lifecycle controls.

Inspector tuning mechanisms include:
- **Scan scope control** for accounts, regions, and resource types.
- **Exclusion or exception handling** through resource selection and tagging.
- **Remediation** to remove the underlying vulnerability.
- **Finding lifecycle management** through Security Hub if Inspector findings are forwarded there.

Inspector does not provide a broad universal suppression workflow equivalent to Security Hub automation rules. For exam purposes, prefer the language of:
- scope,
- exclusion,
- remediation,
- and downstream triage.

**Inspector example: scope control with tags**

A common pattern is to exclude non-production or lab resources from scanning by using tags and resource selection.

Example approach:
- Tag lab instances with `Environment=Lab`.
- Configure scanning scope to include only `Environment=Prod`.
- Keep a separate account or OU for test workloads.

Example tag-based selection concept:

```json
{
  "resourceTags": [
    {
      "key": "Environment",
      "value": "Prod"
    }
  ]
}
```

This is a strategy pattern, not a literal Inspector API payload. The exact API shape depends on the Inspector version and resource type.

**When to suppress, archive, or suppress at source**

Choose the least risky control that solves the noise problem.

| Situation | Best choice | Why |
|---|---|---|
| Known benign finding from a source service | Suppress at source or archive in source service | Prevents repeated noise while preserving history |
| Finding needs centralized triage across many accounts | Security Hub automation rule | Standardizes handling and keeps audit trail |
| Temporary exception during a change window | Security Hub note + workflow update | Preserves context and supports rollback |
| Non-production benchmark or lab asset | Scope exclusion or dedicated account | Avoids generating noise in the first place |
| Third-party imported finding | Security Hub automation or downstream filter | Source control may not be available |

A useful rule of thumb:
- If you can stop the finding from being created safely, do that.
- If you cannot, archive or automate handling at the source.
- If the finding is already centralized, tune it in Security Hub.

**Operational patterns for prod vs non-prod**

Environment-based exception handling is a common exam theme.

Recommended patterns:
- Separate production and non-production accounts.
- Use tags such as `Environment=Prod`, `Environment=Dev`, and `Environment=Lab`.
- Apply stricter alerting in production.
- Allow broader suppression or auto-resolution only in non-production, and only for approved findings.

Examples:
- Auto-resolve low-severity benchmark findings in dev accounts.
- Keep the same benchmark findings open in prod until explicitly reviewed.
- Exclude ephemeral test instances from Inspector scans.
- Archive GuardDuty findings from known scanner IPs only in lab accounts.

This approach reduces noise without weakening production controls.

**EventBridge and downstream automation**

Suppression decisions affect alerting pipelines. If you only change workflow status in Security Hub, downstream consumers may still receive events unless they filter on the new state.

Common integration points:
- **Amazon EventBridge** rules that route findings to ticketing or chat systems.
- SIEM pipelines that ingest Security Hub findings.
- Lambda functions that enrich or suppress notifications.
- SOAR workflows that open or close incidents.

Practical guidance:
- Filter on `Workflow.Status` in EventBridge if you only want active findings.
- Exclude archived GuardDuty findings if your pipeline should only process active alerts.
- Keep suppression logic consistent across Security Hub, EventBridge, and ticketing tools.

Example EventBridge pattern concept:

```json
{
  "source": ["aws.securityhub"],
  "detail-type": ["Security Hub Findings - Imported"],
  "detail": {
    "findings": {
      "Workflow": {
        "Status": ["NEW", "NOTIFIED"]
      }
    }
  }
}
```

This pattern keeps resolved findings out of notification workflows.

**End-to-end scenarios**

1. A GuardDuty port scan is generated by a known vulnerability scanner in a lab account.
   - You confirm the scanner is approved.
   - You create a GuardDuty suppression rule or archive the finding.
   - You keep the rule narrow so it only matches the lab account and the specific finding type.
   - If the finding is forwarded to Security Hub, you may also add a Security Hub automation rule to mark it resolved or route it away from paging.

2. An Inspector vulnerability appears on a temporary test instance.
   - You verify the instance is non-production.
   - You exclude the instance from future scans by scope or tag-based selection.
   - You remediate or terminate the instance if it is no longer needed.
   - If the finding is already in Security Hub, you can resolve it there with a note, but the real fix is at the scan scope.

3. A third-party product sends repeated low-value findings into Security Hub.
   - You cannot tune the source product directly.
   - You create a Security Hub automation rule that matches the product, account, and severity.
   - You add a note and set workflow to `RESOLVED` only for approved cases.
   - You keep EventBridge and ticketing filters aligned so the same finding does not page twice.

**Exam decision points**

- Use **Security Hub** when the goal is centralized triage and consistent handling across sources.
- Use **GuardDuty** when the goal is detector-level noise reduction or archiving of threat findings.
- Use **Inspector** when the goal is to reduce scan scope or remove the underlying vulnerability.
- Prefer **source-side tuning** over downstream suppression when possible.
- Treat **workflow status** as triage metadata, not as a universal suppression feature.
- Remember that **archived** does not mean deleted; it means no longer active in the source service.

**Practical CLI reminders**

Security Hub workflow update:

```bash
aws securityhub batch-update-findings \
  --finding-identifiers ProductArn=arn:aws:securityhub:us-east-1:123456789012:product/123456789012/default,Id=example-finding-id \
  --workflow-status NOTIFIED \
  --note Text="Acknowledged and routed to platform team",UpdatedBy="secops"
```

GuardDuty archive pattern:

```bash
aws guardduty create-filter \
  --detector-id a1b2c3d4e5f6 \
  --name "ArchiveApprovedScanner" \
  --action ARCHIVE \
  --finding-criteria '{"type":{"Eq":["Recon:EC2/Portscan"]},"accountId":{"Eq":["123456789012"]}}'
```

Inspector tuning pattern:
- Prefer account, region, and tag scoping.
- Use remediation to eliminate the vulnerability.
- Use Security Hub only for downstream triage if Inspector findings are aggregated there.

**Common mistakes to avoid**

- Treating `SUPPRESSED` as a Security Hub workflow status.
- Using Security Hub automation rules as if they were source-service suppression.
- Assuming GuardDuty archive stops future detections by itself.
- Trying to solve Inspector noise with a generic suppression workflow instead of scope control.
- Overbroad filters that hide real production issues.
- Forgetting to align EventBridge rules with the chosen workflow state.

**Summary**

Suppression strategy in AWS is about choosing the right control at the right layer. Security Hub handles centralized triage and automation, GuardDuty handles detector-level archiving and filtering, and Inspector relies more on scope control and remediation than on suppression. For exam success, focus on where the finding originates, how it is consumed downstream, and whether you need to mute, archive, resolve, or prevent it at the source.
