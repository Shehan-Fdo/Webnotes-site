## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Overview**

This topic is about turning on the AWS services that produce, collect, and centralize security-relevant telemetry. In practice, you usually combine account-level controls, organization-wide administration, and Region-by-Region enablement.

The exam often tests whether you know:
- Which services are **regional** versus **global** in operational impact.
- Which features require **delegated administration** in AWS Organizations.
- Which services need supporting resources such as **S3 buckets**, **KMS keys**, **IAM roles**, or **delivery channels** before they work.

A useful mental model is:
- **CloudTrail** records API activity.
- **AWS Config** records configuration state.
- **GuardDuty** analyzes signals for threats.
- **Security Hub** aggregates findings and standards.
- **Detective** helps investigate relationships and timelines.
- **Security Lake** centralizes security data for analytics.
- **WAF**, **Shield**, and **Session Manager** add protection and logging around workloads and access.

**AWS Organizations and centralized security**

Many security services support centralized administration through AWS Organizations. For exam purposes, distinguish between:
- **Management account**: the original organization owner.
- **Delegated administrator**: a member account authorized to administer a service for the organization.
- **Trusted access**: permission for a service to integrate with Organizations.

Centralized security usually follows this pattern:
1. Enable trusted access for the service in Organizations.
2. Register a delegated administrator, if the service supports it.
3. Configure organization-wide auto-enable or aggregation settings.
4. Ensure each target Region is configured where the service is regional.

> Organization-wide enablement does not always mean one setting covers every Region. Many services still require per-Region setup.

**AWS CloudTrail**

CloudTrail records AWS API calls and related events. It is foundational for auditability and incident response.

Key event types:
- **Management events**: control-plane actions such as `CreateUser`, `RunInstances`, or `PutBucketPolicy`.
- **Data events**: high-volume object or function activity such as S3 object access or Lambda invocation.
- **Network activity events**: service-specific network-related events where supported.

Important deployment concepts:
- A **trail** delivers events to S3 and optionally CloudWatch Logs.
- An **organization trail** captures activity across all accounts in an AWS Organization.
- A **multi-Region trail** reduces blind spots by collecting events from all Regions.
- **CloudTrail Lake** stores and queries events in an event data store.

Typical organization trail prerequisites:
- AWS Organizations is enabled.
- **Trusted access** for CloudTrail is enabled.
- A **delegated administrator** may be used for centralized management.
- The trail is configured to apply to the organization and, if needed, all Regions.

A practical trail setup often includes:
- An S3 bucket for log delivery.
- A bucket policy that allows CloudTrail to write logs.
- Optional SSE-KMS encryption with a KMS key policy that allows CloudTrail to use the key.
- Optional CloudWatch Logs integration for near-real-time alerting.

Example bucket policy pattern for CloudTrail delivery:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AWSCloudTrailAclCheck",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudtrail.amazonaws.com"
      },
      "Action": "s3:GetBucketAcl",
      "Resource": "arn:aws:s3:::my-cloudtrail-logs"
    },
    {
      "Sid": "AWSCloudTrailWrite",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudtrail.amazonaws.com"
      },
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::my-cloudtrail-logs/AWSLogs/123456789012/*",
      "Condition": {
        "StringEquals": {
          "s3:x-amz-acl": "bucket-owner-full-control"
        }
      }
    }
  ]
}
```

If you encrypt the bucket with SSE-KMS, the KMS key policy must also allow CloudTrail to use the key. A common exam point is that S3 permissions alone are not enough.

Example KMS key policy fragment:
```json
{
  "Sid": "AllowCloudTrailUseOfTheKey",
  "Effect": "Allow",
  "Principal": {
    "Service": "cloudtrail.amazonaws.com"
  },
  "Action": [
    "kms:GenerateDataKey*",
    "kms:Decrypt",
    "kms:DescribeKey"
  ],
  "Resource": "*"
}
```

Useful CLI examples:
```bash
aws cloudtrail create-trail \
  --name org-audit-trail \
  --s3-bucket-name my-cloudtrail-logs \
  --is-multi-region-trail \
  --is-organization-trail

aws cloudtrail put-event-selectors \
  --trail-name org-audit-trail \
  --event-selectors '[
    {
      "ReadWriteType": "All",
      "IncludeManagementEvents": true,
      "DataResources": [
        {
          "Type": "AWS::S3::Object",
          "Values": ["arn:aws:s3:::my-sensitive-bucket/"]
        }
      ]
    }
  ]'
```

For S3 data events, use the object ARN pattern carefully. In many cases, the selector targets the bucket object namespace, not just the bucket ARN alone.

**AWS Config**

AWS Config records resource configurations and evaluates them against rules. It is essential for drift detection, compliance evidence, and change history.

Core components:
- **Configuration recorder**: captures configuration changes.
- **Delivery channel**: sends snapshots and configuration history to S3 and optional SNS.
- **Recorder status**: the recorder must be started before data is collected.
- **Config rules**: evaluate resources against desired state.
- **Conformance packs**: bundle rules and remediation into reusable compliance packages.

Operationally, AWS Config does not record until both the recorder and delivery channel are configured. The order matters:
1. Create the S3 bucket for Config snapshots.
2. Create the delivery channel.
3. Create or update the configuration recorder.
4. Start the recorder.

Example CLI flow:
```bash
aws configservice put-configuration-recorder \
  --configuration-recorder '{
    "name": "default",
    "roleARN": "arn:aws:iam::123456789012:role/AWSConfigRole",
    "recordingGroup": {
      "allSupported": true,
      "includeGlobalResourceTypes": true
    }
  }'

aws configservice put-delivery-channel \
  --delivery-channel '{
    "name": "default",
    "s3BucketName": "my-config-bucket"
  }'

aws configservice start-configuration-recorder \
  --configuration-recorder-name default
```

The IAM role for Config must allow:
- Writing to the S3 bucket.
- Publishing to SNS if used.
- Describing and recording supported resources.

Example trust policy fragment:
```json
{
  "Effect": "Allow",
  "Principal": {
    "Service": "config.amazonaws.com"
  },
  "Action": "sts:AssumeRole"
}
```

Exam-relevant details:
- **Global resource types** are often recorded in a designated home Region.
- **Organization Config** can aggregate compliance across accounts.
- **Aggregator** setup is separate from recording and requires source accounts or organization sources.
- **Rules** can be managed or custom.
- **Remediation** can invoke SSM Automation documents.

Example organization aggregator pattern:
```bash
aws configservice put-configuration-aggregator \
  --configuration-aggregator-name org-aggregator \
  --organization-aggregation-source '{
    "RoleArn": "arn:aws:iam::123456789012:role/AWSConfigAggregatorRole",
    "AllAwsRegions": true
  }'
```

**GuardDuty**

GuardDuty is a threat detection service that analyzes signals such as CloudTrail events, VPC Flow Logs, DNS logs, and service-specific telemetry. It produces findings rather than raw logs.

Enablement concepts:
- A **detector** exists per account and Region.
- Organization administration uses a **delegated administrator**.
- You can configure **auto-enable** for new member accounts.
- Some features are optional and must be enabled separately.

Common data sources and features:
- **S3 Protection**: detects suspicious S3 object access patterns.
- **EKS Audit Log Monitoring**: analyzes Kubernetes audit logs.
- **Malware Protection**: scans supported storage or workload paths depending on feature support.
- **DNS Protection**: detects malicious domain activity.
- **RDS Protection** and other workload-specific protections where available.

Typical organization setup:
1. Enable GuardDuty in the delegated administrator account.
2. Register member accounts or enable organization auto-enable.
3. Turn on desired data sources and protection plans in each Region.
4. Verify findings flow to the admin account.

Example CLI pattern:
```bash
aws guardduty create-detector \
  --enable

aws guardduty update-organization-configuration \
  --detector-id a1b2c3d4e5f6 \
  --auto-enable '{
    "EKS_RUNTIME_MONITORING": "NEW",
    "S3_DATA_EVENTS": "NEW",
    "EBS_MALWARE_PROTECTION": "NEW"
  }'
```

> GuardDuty is not “one detector for the whole organization.” You still manage Region-specific detectors and organization settings.

**Security Hub**

Security Hub aggregates findings from AWS services and partner products, then evaluates security posture against standards.

Important concepts:
- **Findings aggregation**: centralizes alerts from multiple sources.
- **Standards**: benchmark frameworks such as CIS or AWS Foundational Security Best Practices.
- **Cross-Region aggregation**: consolidates findings from multiple Regions into a home Region.
- **Organization administration**: centralizes enablement and member management.

Basic account-level enablement:
```bash
aws securityhub enable-security-hub \
  --enable-default-standards
```

For organization use, you also need:
- A delegated administrator.
- Trusted access with Organizations.
- Auto-enable or invitation-based member association.
- Cross-Region aggregation if you want a single reporting Region.

Example organization administration pattern:
```bash
aws securityhub enable-organization-admin-account \
  --admin-account-id 123456789012

aws securityhub update-organization-configuration \
  --auto-enable
```

Operational notes:
- Enabling default standards in one account does not automatically centralize the organization.
- Findings can be imported from AWS services such as GuardDuty, Inspector, Macie, and Config where supported.
- Security Hub is often the dashboard layer, not the source of raw telemetry.

**IAM Access Analyzer**

IAM Access Analyzer helps identify unintended access by analyzing resource-based policies and access paths.

Key outputs:
- **External access findings**: resources shared outside the account or organization boundary.
- **Unused access findings**: permissions granted but not used over time.
- **Archive rules**: suppress expected findings based on conditions.
- **Analyzers**: can be scoped to an account or an organization.

Analyzer types:
- **Account analyzer**: evaluates resources in one account.
- **Organization analyzer**: evaluates across accounts in an organization.

Example CLI:
```bash
aws accessanalyzer create-analyzer \
  --analyzer-name org-analyzer \
  --type ORGANIZATION
```

Example archive rule:
```bash
aws accessanalyzer create-archive-rule \
  --analyzer-name org-analyzer \
  --rule-name suppress-known-partner-access \
  --filter '{
    "resourceOwnerAccount": [
      {
        "eq": ["123456789012"]
      }
    ]
  }'
```

Exam focus:
- Access Analyzer is about **policy analysis**, not packet inspection.
- It is especially useful for S3 buckets, IAM roles, KMS keys, Lambda permissions, and other resource-based policies where supported.
- Support varies by resource type and can change over time, so focus on the concept rather than memorizing a fixed service list.

**Security Lake**

Security Lake centralizes security data into an analytics-ready data lake. It is designed for normalized security data collection and downstream analysis.

Core workflow:
1. Create or designate the Security Lake administrator.
2. Set up the data lake in supported Regions.
3. Choose sources and subscriptions.
4. Configure permissions and integrations for consumers.

Important concepts:
- **Data lake**: the central storage layer.
- **Sources**: AWS services, custom sources, and partner sources where supported.
- **Subscribers**: analytics or SIEM tools that consume the data.
- **Normalization**: data is stored in a common schema for easier analysis.

Practical considerations:
- Security Lake is regional and has supported-Region constraints.
- You must plan for source onboarding and permissions.
- It often integrates with Athena, Glue, and S3-based analytics workflows.

Example conceptual CLI pattern:
```bash
aws securitylake create-data-lake \
  --regions us-east-1 us-west-2 \
  --encryption-configuration '{
    "kmsKeyId": "arn:aws:kms:us-east-1:123456789012:key/abcd-1234"
  }'
```

> Exact Security Lake onboarding commands and parameters can vary by CLI version and feature set. For exam prep, focus on the enablement sequence and the need for supported Regions, sources, and subscribers.

**VPC Flow Logs**

VPC Flow Logs capture IP traffic metadata for VPCs, subnets, or network interfaces. They are useful for network forensics, troubleshooting, and anomaly detection.

What Flow Logs do:
- Record accepted and rejected traffic metadata.
- Help identify security group or NACL issues.
- Support delivery to CloudWatch Logs, S3, or Kinesis Data Firehose depending on setup.

Important configuration choices:
- **Resource scope**: VPC, subnet, or ENI.
- **Traffic type**: `ACCEPT`, `REJECT`, or `ALL`.
- **Aggregation interval**: commonly 1 minute or 10 minutes.
- **Log format**: controls which fields are included.
- **Destination**: CloudWatch Logs, S3, or Firehose.

Example CLI pattern for a VPC flow log to CloudWatch Logs:
```bash
aws ec2 create-flow-logs \
  --resource-type VPC \
  --resource-ids vpc-0123456789abcdef0 \
  --traffic-type ALL \
  --log-destination-type cloud-watch-logs \
  --log-group-name /aws/vpc/flowlogs \
  --deliver-logs-permission-arn arn:aws:iam::123456789012:role/FlowLogsDeliveryRole \
  --max-aggregation-interval 60 \
  --log-format '${version} ${account-id} ${interface-id} ${srcaddr} ${dstaddr} ${srcport} ${dstport} ${protocol} ${action} ${log-status}'
```

Troubleshooting points:
- The destination must allow delivery permissions.
- CloudWatch Logs requires a log group and a role that can publish logs.
- S3 destinations require a bucket policy that allows the Flow Logs service to write.
- If you use Firehose, the delivery stream must be configured correctly.

**AWS WAF and AWS Shield**

AWS WAF protects web applications from common exploits and bot-like patterns. AWS Shield protects against DDoS attacks.

WAF concepts:
- **Web ACL**: the top-level rule container.
- **Rules and rule groups**: match requests and take actions.
- **Association**: attach a Web ACL to CloudFront, ALB, API Gateway, AppSync, or other supported resources.
- **Logging**: send request logs to Kinesis Data Firehose for analysis.

Example WAF logging association pattern:
```bash
aws wafv2 put-logging-configuration \
  --logging-configuration '{
    "ResourceArn": "arn:aws:wafv2:us-east-1:123456789012:regional/webacl/example/abcd1234",
    "LogDestinationConfigs": [
      "arn:aws:firehose:us-east-1:123456789012:deliverystream/aws-waf-logs"
    ]
  }'
```

Shield concepts:
- **Shield Standard** is automatically available and protects against common network and transport-layer DDoS events.
- **Shield Advanced** adds enhanced detection, response support, cost protection for certain scaling events, and deeper visibility into attacks.
- Shield is primarily a protection and response service; it is not a general-purpose telemetry source like CloudTrail.

Exam distinction:
- Use **WAF** for HTTP/S request filtering.
- Use **Shield** for DDoS resilience.
- Use both together for internet-facing applications.

**Session Manager**

AWS Systems Manager Session Manager provides secure shell or command access to managed instances without opening inbound ports. It can also log sessions for audit and investigation.

Prerequisites:
- The instance must have the **SSM Agent** installed and running.
- The instance profile must allow Systems Manager actions.
- The instance must be able to reach Systems Manager endpoints, either directly or through VPC endpoints.
- Logging destinations must be configured if you want session transcripts.

Common logging options:
- **CloudWatch Logs**
- **Amazon S3**
- **KMS encryption** for stored transcripts

Example Session Manager preference document pattern:
```json
{
  "schemaVersion": "1.0",
  "description": "Session Manager preferences",
  "sessionType": "Standard_Stream",
  "inputs": {
    "s3BucketName": "my-session-logs",
    "s3KeyPrefix": "ssm/",
    "cloudWatchLogGroupName": "/aws/ssm/session-logs",
    "cloudWatchEncryptionEnabled": true,
    "kmsKeyId": "arn:aws:kms:us-east-1:123456789012:key/abcd-1234",
    "runAsEnabled": true,
    "runAsDefaultUser": "ssm-user"
  }
}
```

Practical note:
- Session logging is configured through Session Manager preferences and related Systems Manager settings, not by inventing a custom session document schema.
- If you need encrypted logs, ensure the KMS key policy allows the Systems Manager service and the logging destination to use the key.

**How to think about enablement scope**

A common exam challenge is deciding what must be enabled:
- **Per account and per Region**: GuardDuty detectors, Security Hub, Config recorders, VPC Flow Logs, WAF associations.
- **Centralized with organization support**: CloudTrail organization trails, GuardDuty delegated admin, Security Hub delegated admin, Config aggregators, Security Lake administration, IAM Access Analyzer organization analyzers.
- **Global or mostly centralized but still operationally scoped**: CloudTrail management, IAM Access Analyzer, and some Security Hub administration tasks.

A practical comparison:

| Service | Primary purpose | Typical enablement scope | Key prerequisite |
|---|---|---|---|
| CloudTrail | API auditing | Organization trail plus multi-Region trail | S3 bucket policy, optional KMS key policy |
| AWS Config | Configuration history and compliance | Per account and Region, then aggregate | Recorder, delivery channel, recorder started |
| GuardDuty | Threat detection | Per account and Region, centrally administered | Delegated admin, auto-enable settings |
| Security Hub | Finding aggregation and standards | Per account and Region, centrally administered | Delegated admin, member association |
| IAM Access Analyzer | Access analysis | Account or organization analyzer | Analyzer creation and archive rules |
| Security Lake | Central security data lake | Supported Regions with admin setup | Data lake, sources, subscribers |
| VPC Flow Logs | Network metadata | Per VPC/subnet/ENI | Destination permissions |
| WAF | HTTP/S filtering | Per protected resource | Web ACL association and logging |
| Shield | DDoS protection | Automatic Standard, Advanced by subscription | Resource protection and response setup |
| Session Manager | Secure access and session logging | Per managed instance and account settings | SSM Agent, IAM role, logging config |

**Troubleshooting and exam traps**

- A trail that exists but does not deliver logs usually points to an S3 policy, KMS policy, or Region scope issue.
- Config compliance data missing from an account often means the recorder is not started or the delivery channel is incomplete.
- GuardDuty findings not appearing organization-wide often indicate delegated admin or auto-enable misconfiguration.
- Security Hub enabled in one account does not automatically mean centralized aggregation is active.
- Flow Logs that are created but silent often fail because the destination role, bucket policy, or log group permissions are wrong.
- Session Manager access without logs usually means the instance is reachable but logging preferences or encryption settings are incomplete.

**Practical deployment sequence**

1. Enable AWS Organizations trusted access for the security service.
2. Register a delegated administrator where supported.
3. Turn on organization-wide auto-enable or member enrollment.
4. Configure per-Region resources such as trails, recorders, detectors, and Web ACL associations.
5. Add logging destinations, encryption, and retention controls.
6. Validate that findings, logs, and compliance data appear in the central account.

**Key takeaways**

- **CloudTrail** gives you the audit trail; make sure S3 and KMS permissions are correct.
- **AWS Config** needs both a recorder and a delivery channel before it records.
- **GuardDuty** and **Security Hub** require careful organization and Region setup.
- **IAM Access Analyzer** helps you find unintended access and unused permissions.
- **Security Lake** centralizes security data, but only after source and Region planning.
- **VPC Flow Logs**, **WAF**, **Shield**, and **Session Manager** add network, web, DDoS, and access visibility around your workloads.
