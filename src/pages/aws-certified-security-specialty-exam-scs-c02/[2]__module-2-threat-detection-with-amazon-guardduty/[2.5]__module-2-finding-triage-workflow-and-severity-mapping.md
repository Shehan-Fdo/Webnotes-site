## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Finding triage workflow overview**

Finding triage is the process of turning a raw alert into a decision: ignore, suppress, investigate, escalate, or remediate. In AWS, this usually starts with a service such as **Amazon GuardDuty**, **AWS Security Hub**, **AWS CloudTrail**, **Amazon Inspector**, or a partner integration, then moves through enrichment, prioritization, ownership, and closure.

A practical triage workflow usually looks like this:

1. Detect the finding in a source service or in Security Hub.
2. Enrich it with context such as account, region, resource tags, asset criticality, and recent activity.
3. Map severity to business impact, not just technical impact.
4. Decide whether the finding is an incident, a compliance issue, or a remediation ticket.
5. Assign an owner and track the workflow state until closure.

In exam scenarios, the key skill is not memorizing every alert type. It is knowing how to decide what matters first and what action is appropriate for the organization.

**Security Hub as the aggregation layer**

**AWS Security Hub** is the central place to aggregate, normalize, and prioritize findings across accounts and services. It ingests findings from AWS services and partner products, then presents them in a common format so you can compare alerts consistently.

Security Hub is useful because it gives you:

- **Normalization** across sources, so different services can be triaged in one place.
- **Aggregation** across multiple accounts and Regions.
- **Insights** that highlight patterns, such as repeated misconfigurations or repeated suspicious activity.
- **Automation rules** that can update workflow fields, add labels, suppress noise, or route findings.

A common operating model is:

- Member accounts generate findings.
- A delegated administrator or central security account aggregates them.
- Security analysts triage centrally.
- Remediation owners act in the workload account or service team account.

This model matters for the exam because it separates detection from ownership. The central team often triages and routes, but the workload owner usually fixes the issue.

**Finding lifecycle and workflow states**

Security Hub findings have a workflow state that helps track triage progress. The current workflow states are:

- **NEW**: The finding has not been triaged yet.
- **NOTIFIED**: The finding has been sent to an owner or downstream system, but not necessarily resolved.
- **RESOLVED**: The issue has been addressed or the finding is no longer actionable.
- **SUPPRESSED**: The finding is intentionally ignored or hidden from active triage, usually because it is expected, low value, or handled by automation.

> Be careful not to treat suppression as a security fix. A suppressed finding is a triage decision, not proof that the underlying issue is gone.

In practice, a finding often moves through this path:

- `NEW` when first ingested.
- `NOTIFIED` after routing to a ticketing system, chat channel, or owner.
- `RESOLVED` after remediation, validation, or accepted risk.
- `SUPPRESSED` when automation or policy determines it should not consume analyst time.

Security Hub automation rules can update workflow fields based on criteria such as product name, severity, resource type, account, or tags. This is useful for recurring low-value findings, but you should use it carefully so you do not suppress real risk.

**Severity mapping fundamentals**

Severity mapping is the process of converting a technical alert into a business-priority decision. A finding with a low technical severity can still be high priority if it affects a crown-jewel asset, a regulated dataset, or an internet-facing system.

Security Hub uses severity labels and a numeric severity score. The common severity labels are:

- **INFORMATIONAL**
- **LOW**
- **MEDIUM**
- **HIGH**
- **CRITICAL**

The numeric score is typically on a 0–100 scale, with source services normalizing their own severity into Security Hub’s model. Exact values can vary by source and finding type, so do not assume every service uses the same raw scoring logic.

A practical mapping approach is:

- **0**: Informational or no immediate security impact.
- **1–39**: Low.
- **40–69**: Medium.
- **70–89**: High.
- **90–100**: Critical.

Use the score as a starting point, not the final answer. The final priority should also consider:

- **Asset criticality**: Is the resource production, internet-facing, or a crown jewel?
- **Data classification**: Does it involve sensitive, regulated, or confidential data?
- **Blast radius**: Could the issue affect one instance, one account, or the whole organization?
- **Exposure**: Is the resource public, reachable from the internet, or accessible cross-account?
- **Compensating controls**: Are there guardrails, segmentation, or detection controls already in place?
- **Exploitability**: Is the issue actively abused in the wild or easy to weaponize?
- **Business timing**: Is the system in a change freeze, peak sales window, or incident response period?

A simple decision rubric is:

| Technical severity | Asset criticality | Exposure | Suggested priority |
|---|---|---|---|
| Low | Low | Internal only | Low |
| Medium | High | Internet-facing | High |
| High | High | Sensitive data | Critical |
| Informational | High | Publicly exposed | Medium or High, depending on context |

**How source services influence triage**

Different AWS services produce findings for different reasons, so the triage workflow changes by source.

**GuardDuty findings**

**Amazon GuardDuty** detects suspicious or malicious activity such as credential misuse, reconnaissance, crypto mining, or unusual API calls. GuardDuty findings are often high-value triage inputs because they can indicate active threat behavior.

When triaging GuardDuty:

- Check whether the activity is expected for the workload.
- Correlate the finding with recent CloudTrail events.
- Look for related VPC Flow Logs, DNS logs, or service-specific logs.
- Determine whether the actor is a known automation role, a human user, or an unknown principal.
- Assess whether the activity is isolated or part of a broader campaign.

Example triage logic:

- A single failed API call from a known admin role may be low priority.
- Repeated `AssumeRole` anomalies from an unfamiliar IP may be high priority.
- Evidence of credential exfiltration or instance compromise may be critical.

A GuardDuty finding often becomes an incident when you see signs of active compromise, lateral movement, or data access. If it is only a policy violation or a benign anomaly, it may remain a ticket or be suppressed after review.

**CloudTrail-based findings**

**AWS CloudTrail** records API activity and is essential for attribution, timeline reconstruction, and control-plane investigation. It is not always the only source you need, but it is often the first place you look for who did what and when.

Use CloudTrail to answer:

- Which principal made the call?
- From which source IP or AWS service?
- What resource was targeted?
- Was the action successful or denied?
- What happened immediately before and after the event?

Example commands:

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=PutBucketPolicy \
  --max-results 10
```

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=Username,AttributeValue=alice \
  --start-time 2026-04-25T00:00:00Z \
  --end-time 2026-04-26T00:00:00Z
```

CloudTrail is especially useful for suspicious IAM changes, policy edits, access key use, and resource modifications. However, for data-plane or network-centric incidents, you may also need VPC Flow Logs, DNS logs, application logs, or service logs to complete the picture.

**Inspector and configuration findings**

**Amazon Inspector** and configuration-focused findings often point to vulnerabilities, exposed packages, or misconfigurations. These findings are usually not incidents by themselves, but they can become urgent if the affected asset is internet-facing, privileged, or hosts sensitive data.

Typical triage questions:

- Is the vulnerable package actually installed and reachable?
- Is the affected workload public or isolated?
- Is there a compensating control such as WAF, segmentation, or patching automation?
- Is there an exploit available in the wild?
- Is the asset business-critical?

A critical vulnerability on a dev box may be lower priority than a medium vulnerability on a public production service.

**Compliance findings versus incidents**

Not every finding is a security incident. Many findings are better treated as compliance or remediation tickets.

Use this distinction:

- **Incident**: Active compromise, unauthorized access, data exposure, or ongoing malicious behavior.
- **Remediation ticket**: Misconfiguration, missing patch, weak control, or policy drift without evidence of active exploitation.
- **Compliance issue**: Control failure that violates a standard, benchmark, or internal policy.

This distinction matters because it determines response speed, escalation path, and whether you activate incident response procedures.

**Practical severity mapping rubric**

A useful triage rubric combines technical and business factors into a final priority.

1. Start with the source severity.
2. Adjust for asset criticality.
3. Adjust for exposure and blast radius.
4. Adjust for data sensitivity and regulatory impact.
5. Adjust for evidence of active exploitation.
6. Decide the response class: incident, ticket, or suppression.

Example:

- GuardDuty reports suspicious `AssumeRole` activity on a production account.
- The role can access customer data.
- The activity comes from an unfamiliar IP and occurs repeatedly.
- CloudTrail shows follow-on calls to `GetObject` and `ListBuckets`.

Final decision:

- Treat as a **high-priority incident**.
- Isolate the principal or credentials.
- Preserve logs.
- Notify the incident response owner.
- Open a remediation ticket only after containment.

Another example:

- Security Hub reports an S3 bucket policy deviation on a non-production bucket.
- The bucket contains no sensitive data.
- Access is limited to internal users.
- No evidence suggests exploitation.

Final decision:

- Treat as a **remediation ticket**.
- Assign to the workload owner.
- Track to closure.
- Consider suppression only if the condition is expected and recurring.

**Security Hub insights and automation rules**

Security Hub **insights** help you spot patterns across many findings. They are useful for triage because they reduce noise and show where the biggest clusters of risk are.

Common insight use cases:

- Top findings by severity.
- Repeated findings by account.
- Findings by resource type.
- Findings by product source.
- Findings by workflow status.

Automation rules help you operationalize triage. You can use them to:

- Set workflow status.
- Add notes or labels.
- Suppress known-benign findings.
- Route findings to the right team.
- Standardize handling for recurring patterns.

Example automation logic:

- If the finding source is a known scanner and the resource is a dev account, set workflow to `SUPPRESSED`.
- If the finding severity is `CRITICAL` and the account is production, add a label and notify the incident channel.
- If the finding matches a known maintenance window, mark it for review by the platform team.

A realistic automation rule example in JSON-like form:

```json
{
  "RuleName": "SuppressKnownDevScannerNoise",
  "Description": "Suppress recurring scanner findings in non-production accounts",
  "Criteria": {
    "ProductName": ["Inspector"],
    "AwsAccountId": ["111122223333"],
    "SeverityLabel": ["LOW", "MEDIUM"]
  },
  "Actions": [
    {
      "Type": "FINDING_FIELDS_UPDATE",
      "Workflow": {
        "Status": "SUPPRESSED"
      }
    }
  ]
}
```

Use automation to reduce repetitive noise, but keep human review for high-severity or ambiguous findings.

**Operational triage flow across accounts**

In a multi-account AWS organization, triage usually follows a central-to-local model.

- **Member account**: The finding originates here, and the resource owner has the most context.
- **Central security account**: Security Hub aggregates findings and provides visibility.
- **Security operations team**: Triage, enrichment, and prioritization happen here.
- **Workload owner**: Remediation happens here, often with change control and service ownership.

A good operational flow is:

1. Security Hub ingests the finding in the central account.
2. The analyst checks severity, source, and affected account.
3. The analyst enriches with tags, asset inventory, and CloudTrail context.
4. The analyst routes the finding to the workload owner or incident channel.
5. The owner remediates and confirms closure.
6. The analyst updates workflow to `RESOLVED` or `SUPPRESSED` if appropriate.

This model is exam-relevant because it shows how centralized visibility and decentralized remediation work together.

**Example end-to-end walkthrough**

Scenario: GuardDuty detects suspicious API activity in a production account.

1. GuardDuty raises a finding for unusual `AssumeRole` behavior.
2. Security Hub aggregates the finding and assigns a high severity.
3. The analyst checks CloudTrail and sees the role used to enumerate S3 buckets.
4. The affected role has access to customer data.
5. The analyst confirms the source IP is not part of a known corporate range.
6. The team disables the access key or revokes the session, depending on the principal type.
7. The workload owner rotates credentials and reviews IAM permissions.
8. The analyst marks the finding `RESOLVED` after validation.
9. If the finding is expected noise from a known automation pattern, the team may instead suppress future matches with an automation rule.

This is the kind of scenario where you must combine technical evidence with business context. The finding is not just “high severity” because the service says so; it is high priority because it affects a sensitive production asset and shows signs of active misuse.

**CLI examples for triage and closure**

List findings with filters:

```bash
aws securityhub get-findings \
  --filters '{
    "SeverityLabel": [{"Value": "HIGH", "Comparison": "EQUALS"}],
    "WorkflowStatus": [{"Value": "NEW", "Comparison": "EQUALS"}],
    "RecordState": [{"Value": "ACTIVE", "Comparison": "EQUALS"}]
  }'
```

Update workflow status and note using the batch update API:

```bash
aws securityhub batch-update-findings \
  --finding-identifiers '[
    {
      "Id": "arn:aws:securityhub:us-east-1:111122223333:finding/abcd1234",
      "ProductArn": "arn:aws:securityhub:us-east-1::product/aws/securityhub"
    }
  ]' \
  --workflow '{"Status":"RESOLVED"}' \
  --note '{"Text":"Confirmed benign after investigation and closed by owner.","UpdatedBy":"security-ops"}'
```

Archive a finding after triage when it is no longer active for your workflow:

```bash
aws securityhub batch-update-findings \
  --finding-identifiers '[
    {
      "Id": "arn:aws:securityhub:us-east-1:111122223333:finding/abcd1234",
      "ProductArn": "arn:aws:securityhub:us-east-1::product/aws/securityhub"
    }
  ]' \
  --record-state ARCHIVED
```

> Use `ARCHIVED` carefully. Archiving is a workflow decision, not a substitute for remediation.

**Prioritization criteria for exam scenarios**

When the exam asks what to do first, use a consistent decision order:

- **Active exploitation** beats theoretical risk.
- **Production** beats non-production.
- **Sensitive data** beats public or low-value data.
- **Internet exposure** beats internal-only access.
- **Large blast radius** beats isolated impact.
- **Weak compensating controls** increase urgency.
- **Regulatory impact** can elevate even moderate technical issues.

A quick mental model:

- If the finding shows active compromise, treat it as an incident.
- If it shows a control gap without exploitation, treat it as remediation.
- If it is expected and low value, suppress it with a documented rule.
- If it is ambiguous, enrich before deciding.

**Common mistakes to avoid**

- Treating every high-severity finding as an incident without context.
- Suppressing findings before confirming they are truly benign.
- Ignoring asset criticality and data sensitivity.
- Using CloudTrail alone when network or application logs are needed.
- Confusing workflow status with record state.
- Assuming source severity equals final business priority.

**Exam takeaways**

- Security Hub is the central triage and aggregation layer.
- GuardDuty often signals suspicious behavior that needs correlation and enrichment.
- CloudTrail is critical for attribution and timeline analysis, but not always sufficient alone.
- Severity mapping must combine technical severity with business context.
- Use insights and automation rules to reduce noise and standardize handling.
- Decide whether the finding is an incident, a remediation ticket, or a compliance issue.
- Track findings through workflow states until they are resolved or intentionally suppressed.
