## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Why this integration matters**

Amazon GuardDuty is a managed threat detection service that emits findings when it detects suspicious activity in your AWS environment. Amazon EventBridge is the event routing layer you use to react to those findings in near real time.

Routing GuardDuty findings to EventBridge lets you:

- Trigger automated remediation.
- Send alerts to chat, ticketing, or paging systems.
- Enrich findings with context before escalation.
- Fan out the same security event to multiple targets.

For exam purposes, focus on the pattern: **GuardDuty detects, EventBridge routes, targets respond**.

**How GuardDuty publishes findings**

GuardDuty publishes findings as events to EventBridge in the same AWS Region where the finding is generated. You typically match on the GuardDuty event source and finding attributes, then send the event to one or more targets.

A GuardDuty finding event commonly includes fields such as:

- `source`: `aws.guardduty`
- `detail-type`: `GuardDuty Finding`
- `detail.type`: the finding type, such as `UnauthorizedAccess:IAMUser/InstanceCredentialExfiltration`
- `detail.severity`: numeric severity value
- `detail.accountId`: AWS account that owns the finding
- `detail.region`: Region where the finding occurred
- `detail.id`: finding identifier
- `detail.resource`: affected resource details

> Do not rely on `detail.service.action` as a primary routing field. It may appear in some findings, but it is not a stable or universal match key for EventBridge rules.

A practical rule usually filters on:

- `source`
- `detail-type`
- `detail.severity`
- `detail.type`
- `detail.accountId`
- `detail.region`

**Event pattern basics**

EventBridge rules use event patterns to match GuardDuty findings. You can route only high-severity findings, only specific finding types, or all findings from a given account or Region.

Example event pattern for high-severity GuardDuty findings:

```json
{
  "source": ["aws.guardduty"],
  "detail-type": ["GuardDuty Finding"],
  "detail": {
    "severity": [
      { "numeric": [">=", 7] }
    ]
  }
}
```

Example pattern for a specific finding type:

```json
{
  "source": ["aws.guardduty"],
  "detail-type": ["GuardDuty Finding"],
  "detail": {
    "type": ["UnauthorizedAccess:IAMUser/InstanceCredentialExfiltration"]
  }
}
```

Example pattern for multiple finding types:

```json
{
  "source": ["aws.guardduty"],
  "detail-type": ["GuardDuty Finding"],
  "detail": {
    "type": [
      "UnauthorizedAccess:IAMUser/InstanceCredentialExfiltration",
      "Impact:EC2/BitcoinTool.B!DNS"
    ]
  }
}
```

Use specific finding types when you want precise automation. Use severity thresholds when you want a broader triage workflow.

**Core architecture**

A common design is:

- GuardDuty generates a finding.
- EventBridge rule matches the event.
- EventBridge sends the event to a target.
- The target performs alerting or remediation.

Typical targets include:

- AWS Lambda for custom remediation logic
- Amazon SNS for email or SMS notifications
- Amazon SQS for buffering and downstream processing
- AWS Step Functions for orchestrated response workflows
- Amazon Kinesis Data Firehose for archival or analytics pipelines
- Systems Manager Automation for predefined operational runbooks

A simple topology looks like this:

```text
GuardDuty Finding
      |
      v
EventBridge Rule
      |
      +--> SNS Topic -> Email / Chat integration
      +--> Lambda -> Remediation
      +--> Step Functions -> Multi-step response
      +--> SQS -> Security operations queue
```

**Creating the rule**

You can create the rule in the console or with the AWS CLI. The rule and target must be in the same Region as the EventBridge event source and the target service, unless you intentionally forward events across Regions with an additional routing design.

Example CLI rule creation:

```bash
aws events put-rule \
  --name guardduty-high-severity \
  --event-pattern '{
    "source": ["aws.guardduty"],
    "detail-type": ["GuardDuty Finding"],
    "detail": {
      "severity": [{ "numeric": [">=", 7] }]
    }
  }'
```

Attach a target such as an SNS topic:

```bash
aws events put-targets \
  --rule guardduty-high-severity \
  --targets "Id"="1","Arn"="arn:aws:sns:us-east-1:123456789012:security-alerts"
```

If you target Lambda, remember that `put-targets` only attaches the target. You still need a Lambda resource-based policy that allows EventBridge to invoke the function.

**Lambda target permissions**

EventBridge invokes Lambda through the function’s resource policy. Without that permission, the rule matches but invocation fails.

Example permission grant:

```bash
aws lambda add-permission \
  --function-name guardduty-remediate \
  --statement-id allow-eventbridge-invoke \
  --action lambda:InvokeFunction \
  --principal events.amazonaws.com \
  --source-arn arn:aws:events:us-east-1:123456789012:rule/guardduty-high-severity
```

A minimal Lambda target flow:

```python
import json

def lambda_handler(event, context):
    finding = event.get("detail", {})
    finding_type = finding.get("type")
    severity = finding.get("severity")
    account_id = finding.get("accountId")
    region = finding.get("region")

    print(json.dumps({
        "type": finding_type,
        "severity": severity,
        "accountId": account_id,
        "region": region
    }))
```

**SNS alerting pattern**

SNS is the simplest notification target when you want human-readable alerts.

Example target setup:

```bash
aws sns create-topic --name security-alerts
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:security-alerts \
  --protocol email \
  --notification-endpoint security-team@example.com
```

Use SNS when you want:

- Fast notification delivery
- Multiple subscribers
- Minimal custom code

Use Lambda or Step Functions when you need:

- Enrichment
- Conditional branching
- Automated containment
- Integration with external systems

**Automated remediation patterns**

GuardDuty findings often trigger response actions rather than just alerts. Common examples include:

- Disabling or rotating compromised credentials
- Quarantining an EC2 instance by changing security groups
- Isolating a workload by modifying network ACLs or route tables
- Revoking temporary access keys
- Opening an incident ticket
- Tagging resources for investigation

A Lambda-based remediation flow often:

1. Receives the GuardDuty event.
2. Extracts the finding type and affected resource.
3. Looks up context in AWS APIs.
4. Applies a containment action.
5. Logs the action and notifies the security team.

Example pseudo-flow for EC2 isolation:

```python
def isolate_instance(ec2_client, instance_id, quarantine_sg_id):
    ec2_client.modify_instance_attribute(
        InstanceId=instance_id,
        Groups=[quarantine_sg_id]
    )
```

Be careful with automated remediation. Some findings indicate real compromise, but others may be lower-confidence or require analyst review before action.

**Filtering strategy**

Choose filters based on operational maturity:

- **Severity-based filtering** for broad triage
  - Route `severity >= 7` to paging.
  - Route `severity < 7` to a queue or dashboard.
- **Type-based filtering** for precise automation
  - Trigger isolation only for known compromise patterns.
- **Account-based filtering** for multi-account environments
  - Separate production from sandbox or development.
- **Region-based filtering** for regional operations
  - Keep response local to the affected Region when possible.

A useful pattern is to send all findings to a central security account while also routing critical findings to local responders.

**Cross-account and multi-Region considerations**

In AWS Organizations, GuardDuty can be enabled centrally with delegated administration. Findings may be aggregated in a security account, but EventBridge routing still depends on where the event is emitted and where the rule exists.

Key points:

- GuardDuty findings are regional.
- EventBridge rules are regional.
- Lambda targets are regional.
- Cross-account delivery requires explicit permissions and a deliberate event-sharing design.
- Cross-Region routing is not automatic; use forwarding or centralized aggregation patterns if you need it.

A common enterprise design is:

- Member accounts generate findings.
- GuardDuty delegates to a security account.
- EventBridge rules in the security account route findings to central response targets.
- Optional forwarding sends selected events to regional response systems.

**Permissions and least privilege**

To make the integration work, you need the right permissions in the right places.

Common requirements include:

- Permission to create and manage EventBridge rules and targets.
- Permission for EventBridge to invoke Lambda, publish to SNS, or start Step Functions.
- Permission for the security team or automation role to read GuardDuty findings.
- Resource policies for cross-account targets when applicable.

For Lambda, the resource policy is essential. For SNS, the topic policy may need to allow `events.amazonaws.com` to publish. For Step Functions, the state machine must allow EventBridge to start executions.

Example SNS topic policy fragment:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowEventBridgePublish",
      "Effect": "Allow",
      "Principal": {
        "Service": "events.amazonaws.com"
      },
      "Action": "sns:Publish",
      "Resource": "arn:aws:sns:us-east-1:123456789012:security-alerts"
    }
  ]
}
```

**Troubleshooting**

If the rule does not fire, check the following:

- The rule is in the same Region as the GuardDuty event.
- The event pattern matches the actual event structure.
- The target exists and is reachable.
- The target resource policy allows EventBridge access.
- The Lambda function is not failing silently.
- CloudWatch Logs contain invocation errors.
- The finding is not filtered out by an overly narrow pattern.

Useful CLI checks:

```bash
aws events list-rules
aws events list-targets-by-rule --rule guardduty-high-severity
aws events test-event-pattern \
  --event-pattern file://pattern.json \
  --event file://sample-finding.json
```

If you are unsure about a pattern, capture a real finding from EventBridge or CloudWatch Logs and compare the exact JSON structure before tightening filters.

**Exam focus**

For AWS Certified Security – Specialty, know these distinctions:

- GuardDuty detects threats; EventBridge routes events.
- EventBridge rules match on event patterns, not on arbitrary runtime logic.
- Lambda targets require a resource-based permission.
- SNS is a common notification target.
- Findings are regional, so routing design must account for Region boundaries.
- Automated remediation should be scoped carefully to avoid disruptive false positives.

A strong exam answer usually identifies the correct service chain and the permission model, not just the target service.

**Quick reference**

- **Source service:** `aws.guardduty`
- **Detail type:** `GuardDuty Finding`
- **Common match fields:** `detail.type`, `detail.severity`, `detail.accountId`, `detail.region`
- **Common targets:** Lambda, SNS, SQS, Step Functions
- **Key permission issue:** target service must allow EventBridge to invoke or publish

**Practical example: security alert pipeline**

A simple production-ready pattern is:

- Match all findings with `severity >= 7`.
- Send them to an SNS topic for immediate notification.
- Send the same events to a Lambda function for enrichment.
- Store the enriched output in S3 or a security data lake.

This gives you both human response and machine processing without overcomplicating the initial rule.

**When to use EventBridge instead of polling**

Use EventBridge when you want:

- Near real-time response
- Lower operational overhead
- Event-driven automation
- Native integration with AWS services

Avoid polling GuardDuty APIs unless you have a specific batch-processing requirement. Event-driven routing is simpler, faster, and more scalable for security operations.
