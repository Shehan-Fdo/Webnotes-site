## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Security Hub finding model and CLI workflow**

AWS Security Hub aggregates security findings from AWS services and partner products into a normalized format called the <abbr title="AWS Security Finding Format">ASFF</abbr> (AWS Security Finding Format). When you query findings with the AWS CLI, you usually work with `securityhub get-findings`, then narrow the result set with filters and shape the output with JMESPath.

The exam expects you to understand both the data model and the mechanics of CLI retrieval:

- **Findings** are records with fields such as `Id`, `ProductArn`, `AwsAccountId`, `Severity`, `Title`, `Description`, and timestamps.
- **Filters** are sent to the Security Hub API as JSON and are evaluated server-side before results are returned.
- **Pagination** can happen at the service layer and again in the AWS CLI paginator.
- **JMESPath** is used to project, sort, and reduce the returned JSON.

A practical workflow is:

1. Identify the scope, such as account, region, severity, or workflow state.
2. Query findings with `get-findings`.
3. Use `--query` to extract only the fields you need.
4. Inspect a single finding in detail with `get-findings --filters` or `batch-get-findings` when you already know the finding `Id`.

**Core CLI commands**

```bash
aws securityhub get-findings --region us-east-1
aws securityhub batch-get-findings --region us-east-1 --finding-identifiers file://finding-ids.json
aws securityhub get-findings --region us-east-1 --max-results 10
```

`batch-get-findings` is useful when you already have one or more finding identifiers and want the full record without broad searching. `get-findings` is the primary search tool.

**Finding filter structure**

The `--filters` parameter takes JSON that mirrors the Security Hub API filter structure. Each top-level key is a finding field, and each field contains one or more comparison objects.

A single field can include multiple comparison objects, and multiple values inside a comparison object are combined as an OR within that comparison type. Different filter fields are combined as an AND across the request.

```json
{
  "SeverityLabel": [
    { "Value": "HIGH", "Comparison": "EQUALS" },
    { "Value": "CRITICAL", "Comparison": "EQUALS" }
  ],
  "RecordState": [
    { "Value": "ACTIVE", "Comparison": "EQUALS" }
  ]
}
```

This example means:

- Return findings where `SeverityLabel` is `HIGH` or `CRITICAL`.
- Also require `RecordState` to be `ACTIVE`.

That AND/OR behavior is a common exam point. If you need broader matching, add more objects to the same field. If you need tighter matching, add another field.

**Commonly tested filter fields**

Security Hub supports many filterable fields. The most commonly tested ones include:

- **Severity and classification**
  - `SeverityLabel`
  - `SeverityNormalized`
  - `Confidence`
  - `Criticality`
- **Workflow and record state**
  - `WorkflowStatus`
  - `RecordState`
  - `VerificationState`
  - `ComplianceStatus`
- **Timestamps**
  - `CreatedAt`
  - `UpdatedAt`
  - `FirstObservedAt`
  - `LastObservedAt`
  - `NoteUpdatedAt`
- **Text and metadata**
  - `Title`
  - `Description`
  - `NoteText`
  - `SourceUrl`
  - `Types`
  - `GeneratorId`
- **Resource and ownership**
  - `AwsAccountId`
  - `Region`
  - `ProductArn`
  - `ResourceType`
  - `ResourceId`

Not every field is available in every context, and some are more useful for specific finding sources. For example, `VerificationState` is relevant when a finding can be verified or unverified by the source, while `NoteText` and `NoteUpdatedAt` help you search analyst annotations.

**Filtering examples**

Use a JSON file for anything beyond a trivial one-liner. It is easier to read, reuse, and validate.

```json
{
  "SeverityLabel": [
    { "Value": "CRITICAL", "Comparison": "EQUALS" }
  ],
  "WorkflowStatus": [
    { "Value": "NEW", "Comparison": "EQUALS" }
  ],
  "RecordState": [
    { "Value": "ACTIVE", "Comparison": "EQUALS" }
  ]
}
```

```bash
aws securityhub get-findings \
  --region us-east-1 \
  --filters file://filters.json \
  --output json
```

You can also filter by text or identifiers:

```json
{
  "Title": [
    { "Value": "public", "Comparison": "CONTAINS" }
  ],
  "SourceUrl": [
    { "Value": "https://", "Comparison": "PREFIX" }
  ],
  "AwsAccountId": [
    { "Value": "111122223333", "Comparison": "EQUALS" }
  ]
}
```

For resource targeting, be careful with `ResourceId` and `ResourceType`:

- `ResourceType` is the resource category in ASFF, such as an EC2 instance or S3 bucket type.
- `ResourceId` must match the identifier format used in the finding, which may be an ARN or service-specific ID.
- Do not assume a generic resource name will match; use the exact value present in the finding.

**Time-based filtering**

Time filters are important for incident triage and exam scenarios. Security Hub date/time filters use comparison operators appropriate to the field, and you should use the API’s supported time comparisons rather than inventing a `Start`/`End` shape that does not match the field.

Typical use cases:

- Find recently updated findings.
- Search for findings first observed in a specific window.
- Review notes updated after an analyst action.

Example:

```json
{
  "UpdatedAt": [
    { "DateRange": { "Value": 7, "Unit": "DAYS" }, "Comparison": "GREATER_THAN_OR_EQUAL" }
  ]
}
```

> Be careful with date filters: the exact supported structure depends on the API field and comparison type. For exam prep, remember that Security Hub uses field-specific date comparisons and that you should validate the request shape against the API model.

A practical pattern is to combine time and severity:

```json
{
  "SeverityLabel": [
    { "Value": "HIGH", "Comparison": "EQUALS" },
    { "Value": "CRITICAL", "Comparison": "EQUALS" }
  ],
  "UpdatedAt": [
    { "DateRange": { "Value": 24, "Unit": "HOURS" }, "Comparison": "GREATER_THAN_OR_EQUAL" }
  ]
}
```

**Inspecting a single finding**

When you already know the finding identifier, use `batch-get-findings` to retrieve the full record. This is often better than searching broadly when you are doing incident response or validating a specific alert.

```json
[
  {
    "Id": "arn:aws:securityhub:us-east-1:111122223333:subscription/aws-foundational-security-best-practices/v/1.0.0/EC2.1/finding-id",
    "ProductArn": "arn:aws:securityhub:us-east-1::product/aws/securityhub"
  }
]
```

```bash
aws securityhub batch-get-findings \
  --region us-east-1 \
  --finding-identifiers file://finding-ids.json \
  --output json
```

If you need to inspect a finding from a search result, first query for the `Id` and `ProductArn`, then pass those values into `batch-get-findings`.

**Pagination mechanics**

Security Hub and the AWS CLI both paginate, but they do different jobs.

- **Service-side pagination**: the Security Hub API returns a page of results and a token for the next page.
- **CLI-side pagination**: the AWS CLI can automatically keep calling the service until it collects all pages, then present the combined result.
- **`--max-results`**: sets the maximum number of items the service should return in a single API response, subject to service limits.
- **`--page-size`**: tells the CLI paginator to request smaller pages from the service.
- **`--starting-token`**: resumes a paginated CLI operation from a previous token.

This distinction matters on the exam. `--max-results` does not mean “the total number of items the CLI will ever return” in all cases; it affects the service request size. `--page-size` is a CLI pagination control that can help with throttling or large responses. `--starting-token` is for resuming, not for filtering.

Examples:

```bash
aws securityhub get-findings \
  --region us-east-1 \
  --max-results 25
```

```bash
aws securityhub get-findings \
  --region us-east-1 \
  --page-size 25
```

```bash
aws securityhub get-findings \
  --region us-east-1 \
  --starting-token eyJ2IjoiMSJ9
```

If you want only the first page, use `--no-paginate`. That is useful when you are testing a query or troubleshooting response shape.

```bash
aws securityhub get-findings \
  --region us-east-1 \
  --no-paginate \
  --max-results 10
```

**JMESPath for inspection and triage**

JMESPath is the query language used by the AWS CLI `--query` option. It is ideal for reducing noisy JSON into a compact triage view.

Common patterns:

- Select only the fields you need.
- Sort by severity or timestamp.
- Filter out null values.
- Safely access the first item in an array.
- Flatten nested arrays when the finding contains multiple resources or notes.

Examples:

```bash
aws securityhub get-findings \
  --region us-east-1 \
  --filters file://filters.json \
  --query 'Findings[].{Id:Id,Title:Title,Severity:Severity.Label,Account:AwsAccountId,Updated:UpdatedAt}' \
  --output table
```

```bash
aws securityhub get-findings \
  --region us-east-1 \
  --query 'Findings[?Severity.Label!=`null` && Severity.Label!=`INFORMATIONAL`].[Id,Title,Severity.Label]' \
  --output table
```

```bash
aws securityhub get-findings \
  --region us-east-1 \
  --query 'Findings[0].Resources[0].Id'
```

```bash
aws securityhub get-findings \
  --region us-east-1 \
  --query 'sort_by(Findings,&UpdatedAt)[-5:].{Id:Id,Updated:UpdatedAt,Severity:Severity.Label}'
```

A few practical notes:

- `--query` is generally applied after the CLI has collected paginated results, unless you disable pagination.
- Use `table` for human inspection and `json` when you need full fidelity or plan to pipe into another tool.
- Use `text` when you want a simple list for shell scripting, but be careful with nested structures.

**Output formats and when to use them**

Choose the output format based on the task:

| Format | Best for | Notes |
|---|---|---|
| `json` | Full inspection, piping to `jq`, preserving structure | Best default for exam troubleshooting |
| `table` | Quick human review | Good for comparing a few fields |
| `text` | Shell pipelines and simple extraction | Can be awkward with nested arrays |

For Security Hub findings, `json` is usually the safest choice because findings are deeply nested and often contain arrays such as `Resources`, `ProductFields`, `Remediation`, and `Note`.

**Cross-account and organization setup**

Cross-account querying in Security Hub is not automatic. A delegated administrator can centrally view findings only after Security Hub organization configuration is enabled and regional aggregation is configured correctly.

The typical setup flow is:

1. Enable Security Hub in the organization.
2. Designate a delegated administrator account.
3. Enable organization configuration so member accounts can send findings.
4. Configure aggregation in the desired region if you want centralized visibility.
5. Query findings from the delegated administrator or aggregation region.

Representative commands:

```bash
aws securityhub enable-organization-admin-account \
  --admin-account-id 111122223333 \
  --region us-east-1
```

```bash
aws securityhub update-organization-configuration \
  --auto-enable true \
  --region us-east-1
```

```bash
aws securityhub create-finding-aggregator \
  --region us-east-1 \
  --region-linking-mode ALL_REGIONS
```

> Cross-account visibility depends on the organization and aggregation configuration being in place. If aggregation is not enabled, you may only see findings in the local account and region.

In practice, you often query from the delegated admin account and then filter by `AwsAccountId` to isolate a member account:

```bash
aws securityhub get-findings \
  --region us-east-1 \
  --filters '{
    "AwsAccountId":[{"Value":"444455556666","Comparison":"EQUALS"}],
    "SeverityLabel":[{"Value":"CRITICAL","Comparison":"EQUALS"}]
  }' \
  --query 'Findings[].{Id:Id,Title:Title,Account:AwsAccountId,Region:Region}' \
  --output table
```

**Real-world triage workflow**

A common incident-response workflow combines filters, pagination, and JMESPath.

1. Narrow to active, high-priority findings.
2. Sort or project the most relevant fields.
3. Pull the full record for a specific finding.
4. Review resources, remediation, and notes.

Example:

```bash
aws securityhub get-findings \
  --region us-east-1 \
  --filters '{
    "SeverityLabel":[
      {"Value":"HIGH","Comparison":"EQUALS"},
      {"Value":"CRITICAL","Comparison":"EQUALS"}
    ],
    "WorkflowStatus":[
      {"Value":"NEW","Comparison":"EQUALS"},
      {"Value":"NOTIFIED","Comparison":"EQUALS"}
    ],
    "RecordState":[
      {"Value":"ACTIVE","Comparison":"EQUALS"}
    ]
  }' \
  --query 'sort_by(Findings,&UpdatedAt)[].{Id:Id,Title:Title,Severity:Severity.Label,Workflow:Workflow.Status,Updated:UpdatedAt}' \
  --output table
```

Then inspect one finding in detail:

```bash
aws securityhub batch-get-findings \
  --region us-east-1 \
  --finding-identifiers '[
    {
      "Id":"arn:aws:securityhub:us-east-1:111122223333:subscription/aws-foundational-security-best-practices/v/1.0.0/EC2.1/finding-id",
      "ProductArn":"arn:aws:securityhub:us-east-1::product/aws/securityhub"
    }
  ]' \
  --query 'Findings[0].{Id:Id,Title:Title,Description:Description,Resources:Resources,Remediation:Remediation,Note:Note}' \
  --output json
```

This pattern is useful because it separates discovery from deep inspection.

**Practical JMESPath patterns for nested findings**

Security Hub findings often contain nested arrays and optional fields. These patterns help you avoid brittle queries:

- Select the first resource safely:
  - `Findings[0].Resources[0].Id`
- Return only findings that have a note:
  - `Findings[?Note.Text!=\`null\`].[Id,Note.Text,Note.UpdatedAt]`
- Flatten resource IDs across findings:
  - `Findings[].Resources[].Id`
- Handle missing fields by projecting only what exists:
  - `Findings[].{Id:Id,SourceUrl:ProductFields.SourceUrl,Title:Title}`

If a field may be absent, prefer a projection that tolerates nulls rather than assuming every finding has the same shape.

**CLI troubleshooting tips**

- Use `--output json` first when a query fails; it is easier to inspect the raw structure.
- Confirm the region, because Security Hub findings are regional.
- Verify that the account has Security Hub enabled and, for organization use, that aggregation is configured.
- Check the exact filter field names and casing; Security Hub filter keys are case-sensitive.
- If a query returns too much data, narrow with server-side filters before using `--query`.

**Exam focus summary**

- **`get-findings`** is the main search command for Security Hub findings.
- **`batch-get-findings`** is the best choice when you already know the finding identifiers.
- **`--filters`** uses JSON with field-specific comparison objects.
- **Multiple values in one field** are ORed; **different fields** are ANDed.
- **Pagination** has both service-side and CLI-side behavior.
- **`--page-size`**, **`--max-results`**, and **`--starting-token`** serve different pagination roles.
- **`--query`** is for shaping output, not for replacing server-side filtering.
- **Cross-account visibility** requires organization setup and aggregation configuration.
- **`json` output** is usually best for deep inspection; `table` is best for quick review.
