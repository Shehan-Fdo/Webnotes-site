## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**GuardDuty organization model**

Amazon GuardDuty is a managed threat detection service that analyzes signals such as <abbr title="Amazon Elastic Compute Cloud">EC2</abbr> activity, <abbr title="AWS CloudTrail">CloudTrail</abbr> events, DNS logs, and VPC Flow Logs to identify suspicious behavior. In an AWS Organizations environment, you typically centralize administration in one account while still collecting findings from every member account and Region.

The exam-relevant distinction is simple:

- **Delegated administrator** manages GuardDuty for the organization.
- **Member accounts** still need GuardDuty enabled in each Region where you want detection.
- **Findings** are aggregated centrally in the delegated admin view, but detection happens where the detector exists.

> A single delegated admin account does not replace detectors in member accounts. Central management and local detection are related, but they are not the same thing.

**Core concepts and terminology**

<dl>
<dt><strong>Detector</strong></dt>
<dd>The regional GuardDuty resource that performs detection in an account.</dd>
<dt><strong>Delegated administrator</strong></dt>
<dd>An account authorized by AWS Organizations to administer GuardDuty for the organization.</dd>
<dt><strong>Member account</strong></dt>
<dd>An account enrolled under organization-managed GuardDuty whose findings are visible to the delegated admin.</dd>
<dt><strong>Auto-enable</strong></dt>
<dd>A setting that controls whether new organization accounts are automatically enrolled into GuardDuty.</dd>
</dl>

**How organization-wide GuardDuty works**

GuardDuty is regional. If you enable it in `us-east-1`, that does not automatically enable it in `us-west-2`. For multi-Region coverage, you must enable GuardDuty in every Region you care about, including opt-in Regions if your organization uses them.

A practical mental model:

- One organization.
- One delegated admin for GuardDuty.
- One detector per account per Region.
- Centralized findings aggregation in the delegated admin account.

This matters for exam questions because AWS often separates:
- **management plane** actions, such as registering a delegated admin or setting auto-enable behavior
- **data plane** actions, such as creating detectors and generating findings in each Region

**Enable GuardDuty with AWS Organizations**

The usual onboarding flow is:

1. Enable GuardDuty in the management account or a chosen security account.
2. Register a delegated administrator for GuardDuty from the management account.
3. Configure organization-wide auto-enable behavior.
4. Enable GuardDuty in each Region you want covered.
5. Verify that member accounts are enrolled and detectors exist in each Region.

For the delegated admin registration, use the GuardDuty organization management APIs from the **management account**. The exact CLI command names can vary by AWS CLI version, but the flow is consistent: you designate the delegated admin account through GuardDuty organization administration, not by passing a detector ID.

A representative CLI flow looks like this:

```bash
# Run from the AWS Organizations management account
aws guardduty enable-organization-admin-account \
  --admin-account-id 111122223333 \
  --region us-east-1
```

After registration, confirm the delegated admin status:

```bash
aws guardduty list-organization-admin-accounts \
  --region us-east-1
```

You should see the delegated admin account ID in the response.

**Organization auto-enable behavior**

Auto-enable controls whether GuardDuty automatically enrolls organization accounts. It is useful for new accounts created after you standardize the service, but you should not assume it retroactively fixes every existing account without checking the current organization state.

Use auto-enable when you want:
- new accounts to be enrolled automatically
- reduced manual onboarding overhead
- consistent baseline coverage across the organization

Be careful with the scope:
- Existing accounts may need explicit enrollment depending on current settings and the state of the organization.
- Auto-enable does not remove the need to verify detectors in each Region.
- Auto-enable does not create a single shared detector for the whole organization.

A representative configuration pattern is:

```bash
# Example only: confirm the current organization configuration first
aws guardduty get-organization-configuration \
  --detector-id <detector-id> \
  --region us-east-1
```

Then update the organization configuration using the current GuardDuty API shape for your CLI version and Region. In practice, you are looking for the setting that enables automatic enrollment of organization accounts.

**Multi-Region deployment strategy**

GuardDuty is regional by design, so a complete deployment requires coverage in every active Region. In large environments, you usually automate this rather than clicking through the console one Region at a time.

A scalable pattern is:

- enumerate enabled Regions with the AWS CLI or SDK
- iterate through accounts discovered from AWS Organizations
- enable GuardDuty in each account and Region
- verify detectors and organization membership after rollout

Example Region enumeration:

```bash
aws ec2 describe-regions \
  --all-regions \
  --query 'Regions[].RegionName' \
  --output text
```

Example account discovery:

```bash
aws organizations list-accounts \
  --query 'Accounts[].Id' \
  --output text
```

A simple automation approach uses a shell loop or a pipeline in your CI/CD tooling:

```bash
for region in us-east-1 us-west-2 eu-west-1; do
  aws guardduty create-detector \
    --enable \
    --region "$region"
done
```

For organization-managed environments, you typically combine this with delegated admin registration and organization configuration so that new accounts are enrolled consistently.

**Opt-in Regions and newly launched Regions**

Opt-in Regions require explicit activation at the account level before you can use them. GuardDuty cannot protect a Region that the account has not enabled.

For exam purposes, remember:

- **Enabled Region in the account** is a prerequisite for GuardDuty coverage there.
- Newly launched Regions are not automatically covered unless your automation discovers them and enables GuardDuty there.
- A mature security program periodically re-runs Region discovery so that new Regions do not become blind spots.

A practical control pattern is to:
- maintain a Region allowlist or approved Region set
- compare it against `describe-regions`
- trigger GuardDuty enablement for any newly approved Region
- alert if an account has not opted into a required Region

**Console workflow**

If you use the console, the high-level steps are:

1. Open GuardDuty in the management account.
2. Choose the Region you want to configure.
3. Register a delegated administrator account.
4. Enable organization auto-enable settings.
5. Confirm member accounts are enrolled.
6. Repeat for each Region.

The console is useful for initial setup and validation, but automation is better for scale and repeatability.

**Migration from standalone GuardDuty to organization-managed GuardDuty**

A common exam scenario is moving from isolated, account-by-account GuardDuty to centralized organization management.

A safe migration path is:

1. Identify the current standalone detectors and Regions in each account.
2. Register the delegated administrator in the management account.
3. Enable organization management and auto-enable behavior.
4. Enroll existing accounts into the organization-managed configuration.
5. Verify that each member account still has a detector in each required Region.
6. Remove or decommission any obsolete standalone configuration only after confirming central visibility.

Important operational point:
- If an account leaves AWS Organizations, organization-managed GuardDuty no longer applies to it.
- You may need to reconfigure that account as standalone or re-onboard it later.
- Do not assume findings continue to flow centrally after the account is removed from the organization.

**Verification and troubleshooting**

Use verification commands after every major change. Good validation checks include delegated admin status, organization configuration, member enrollment, and detector presence per Region.

Representative checks:

```bash
aws guardduty list-organization-admin-accounts \
  --region us-east-1
```

```bash
aws guardduty list-members \
  --detector-id <detector-id> \
  --region us-east-1
```

```bash
aws guardduty list-detectors \
  --region us-east-1
```

```bash
aws guardduty get-member-detectors \
  --detector-id <detector-id> \
  --account-ids 111122223333 444455556666 \
  --region us-east-1
```

What to look for:
- the delegated admin account appears in the admin list
- member accounts appear as associated or invited/enrolled, depending on the configuration model
- each required Region has a detector in each account that should be monitored
- findings appear in the delegated admin account for the Region where they were generated

Common troubleshooting patterns:
- If a member account is missing from the organization view, confirm it is still in AWS Organizations.
- If a Region has no findings, confirm GuardDuty is enabled in that Region for the member account.
- If a newly created account is not enrolled, check whether auto-enable is active and whether the account was created before the setting was applied.
- If an account leaves the organization, expect the central relationship to break and plan a reconfiguration path.

**End-to-end example: onboarding a new organization**

Suppose you have a management account, a security delegated admin account, and three member accounts. You want coverage in `us-east-1` and `us-west-2`.

A practical rollout looks like this:

1. In the management account, register the security account as delegated admin.
2. In each target Region, enable GuardDuty organization management.
3. Turn on auto-enable for new organization accounts.
4. Enable GuardDuty in `us-east-1` and `us-west-2` for the organization.
5. Confirm each member account has a detector in both Regions.
6. Validate that findings from any member account appear in the delegated admin account for the same Region.

Example validation sequence:

```bash
# Check detectors in the delegated admin account
aws guardduty list-detectors --region us-east-1
aws guardduty list-detectors --region us-west-2

# Check organization members
aws guardduty list-members \
  --detector-id <detector-id-us-east-1> \
  --region us-east-1

# Check delegated admin registration
aws guardduty list-organization-admin-accounts \
  --region us-east-1
```

**Automation at scale**

For large environments, use infrastructure automation rather than manual setup. Common patterns include:
- AWS CLI scripts in a secure admin pipeline
- AWS SDK automation in Python or Go
- AWS CloudFormation StackSets for account-level baseline deployment
- AWS Organizations account discovery plus Region enumeration
- Event-driven onboarding when new accounts are created

A simple Python-style automation outline is:

```python
regions = ["us-east-1", "us-west-2", "eu-west-1"]
accounts = list_organization_accounts()

for account in accounts:
    assume_role(account, "SecurityAutomationRole")
    for region in regions:
        enable_guardduty(region)
        ensure_detector(region)
```

This pattern is useful because it makes the two dimensions explicit:
- account coverage
- Region coverage

**Exam takeaways**

- **Delegated admin** centralizes GuardDuty administration, but it does not eliminate regional detectors.
- **Member accounts** still need GuardDuty enabled in each Region you want monitored.
- **Auto-enable** helps standardize onboarding, but verify how it applies to existing accounts in the current organization state.
- **Opt-in Regions** must be enabled before GuardDuty can protect them.
- **New Regions** require ongoing automation or governance so coverage stays current.
- **Migration** from standalone to organization-managed GuardDuty requires validation of both membership and detector presence.

**Quick comparison**

| Scenario | Delegated admin | Member detectors | Central findings view |
|---|---:|---:|---:|
| Standalone account | No | Yes, per Region | No |
| Organization-managed account | Yes, via delegated admin | Yes, per Region | Yes |
| Account removed from organization | No longer applies | Must be reconfigured if needed | No |
