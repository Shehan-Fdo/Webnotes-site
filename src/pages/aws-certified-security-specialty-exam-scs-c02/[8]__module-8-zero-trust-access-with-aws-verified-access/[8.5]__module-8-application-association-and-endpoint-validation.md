## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Overview**

Application association and endpoint validation is about proving that a workload can reach the intended AWS service or private application through the intended network path, with the intended identity and policy controls. In exam scenarios, you usually need to confirm three things at once:

- The application is using the correct endpoint type.
- DNS resolves to the expected private or public target.
- IAM, resource, endpoint, and service policies all allow the request.

This topic appears often in private connectivity designs, especially when you must keep traffic off the public internet, restrict access by VPC, or validate that a service is reachable only through AWS PrivateLink or a gateway endpoint.

**Core concepts**

An **endpoint** is the network entry point used to reach a service. In AWS security questions, the important distinction is usually between:

- **Gateway endpoints** for Amazon S3 and DynamoDB.
- **Interface endpoints** for AWS PrivateLink-enabled services and many AWS APIs.
- **Gateway Load Balancer endpoints** for transparent insertion of network appliances.
- **Public service endpoints** that are reachable over the internet unless you add private connectivity.

An **application association** is the relationship between a workload and the endpoint or service it is supposed to use. Validation means proving that the association is correct in practice, not just in design documents.

**Endpoint types and when to use them**

Use the endpoint type that matches the service and the security goal.

| Endpoint type | Typical services | Network behavior | Policy control |
|---|---|---|---|
| Gateway endpoint | S3, DynamoDB | Adds routes in route tables; traffic stays on AWS network path to the service | Endpoint policy plus IAM/resource policy |
| Interface endpoint | Many AWS services, PrivateLink services | Creates ENIs in subnets; traffic uses private IPs and DNS | Endpoint policy plus IAM/resource policy/service policy |
| Gateway Load Balancer endpoint | Third-party or custom network appliances | Inserts traffic through an appliance chain | Security groups, route tables, appliance policy |
| Public endpoint | Most AWS service public APIs | Reaches the public service endpoint unless private connectivity is configured | IAM, resource policy, service-specific controls |

Gateway endpoints are route-table based. Interface endpoints are ENI-based. That difference matters in validation:

- For a gateway endpoint, you verify route table entries and service access.
- For an interface endpoint, you verify ENIs, security groups, private DNS, and DNS resolution.

> For exam purposes, do not assume a public AWS service endpoint is automatically private just because the traffic stays within AWS at some point. If you need private access from a VPC, use a VPC endpoint or another service-specific private connectivity feature.

**Gateway endpoint validation**

Gateway endpoints are the simplest private path for S3 and DynamoDB. They do not create ENIs. Instead, AWS inserts a prefix list target into the route table associated with the subnet.

What you validate:

- The route table contains the endpoint route.
- The subnet uses the correct route table.
- The IAM principal and any bucket/table policy allow the action.
- The endpoint policy does not block the request.

Example route table association check:

```bash
aws ec2 describe-route-tables \
  --filters Name=route.vpc-endpoint-id,Values=vpce-0123456789abcdef0
```

Example endpoint description:

```bash
aws ec2 describe-vpc-endpoints \
  --vpc-endpoint-ids vpce-0123456789abcdef0
```

For S3, a common validation pattern is to confirm that the request uses the gateway endpoint and not the public internet path. You can test from an EC2 instance in the subnet:

```bash
aws s3 ls s3://my-private-bucket --region us-east-1
```

If access fails, check:

- The bucket policy.
- The endpoint policy.
- The IAM permissions on the role or user.
- Whether the route table is associated with the subnet.

For DynamoDB, the same logic applies, but the route target is the DynamoDB prefix list rather than a public IP route.

**Interface endpoint validation**

Interface endpoints are used for AWS PrivateLink and many AWS service APIs. They create one or more ENIs in your subnets and expose private IP addresses.

What you validate:

- The endpoint exists in the correct VPC and subnets.
- The endpoint ENIs are present and in `available` state.
- Security groups allow the required ports, usually TCP 443.
- Private DNS is enabled when you want the standard service hostname to resolve to the private IP.
- The service policy, IAM policy, and endpoint policy allow the action.

Example:

```bash
aws ec2 describe-vpc-endpoints \
  --filters Name=vpc-id,Values=vpc-0123456789abcdef0
```

To inspect the ENIs created by the endpoint:

```bash
aws ec2 describe-network-interfaces \
  --filters Name=vpc-endpoint-id,Values=vpce-0123456789abcdef0
```

Typical validation checks:

- `nslookup logs.us-east-1.amazonaws.com` resolves to a private IP when private DNS is enabled.
- `curl` or the AWS CLI can reach the service from the instance.
- The security group on the endpoint allows inbound from the client subnet or client security group.

Example security group rule for an interface endpoint:

```bash
aws ec2 authorize-security-group-ingress \
  --group-id sg-0abc1234def567890 \
  --protocol tcp \
  --port 443 \
  --source-group sg-0123456789abcdef0
```

**PrivateLink provider-side validation**

When you publish a service through AWS PrivateLink, you act as the service provider. Validation is not just about the consumer endpoint; it also includes who can connect to your endpoint service.

Key provider controls:

- **Allowed principals** define which AWS principals can create interface endpoints to your service.
- **Acceptance required** determines whether you must manually approve each connection.
- **Endpoint service policy** can further restrict who may connect and under what conditions.

Example provider-side configuration:

```bash
aws ec2 modify-vpc-endpoint-service-configuration \
  --service-id vpce-svc-0123456789abcdef0 \
  --acceptance-required
```

Add allowed principals:

```bash
aws ec2 modify-vpc-endpoint-service-permissions \
  --service-id vpce-svc-0123456789abcdef0 \
  --add-allowed-principals arn:aws:iam::111122223333:root
```

Provider validation steps:

1. Confirm the service name is published correctly.
2. Confirm the allowed principals list includes the consumer account or role.
3. Confirm acceptance is either automatic or manually approved.
4. Confirm the backend targets are healthy behind the Network Load Balancer.
5. Confirm the consumer endpoint state becomes `available`.

This is important in exam questions where a consumer says the endpoint exists but cannot connect. The issue may be on the provider side, not the consumer side.

**Policy evaluation layers**

Endpoint validation is often really policy validation. AWS evaluates multiple layers, and the effective result is the intersection of all applicable controls.

Common layers include:

- **IAM identity policy** on the user, role, or session.
- **Resource policy** on the target resource, such as an S3 bucket policy, Secrets Manager secret policy, or KMS key policy.
- **Endpoint policy** on the VPC endpoint.
- **Service-specific conditions** such as source VPC, source VPC endpoint, principal tags, or encryption context.
- **Trust policy** for role assumption when STS is involved.

A useful mental model is:

- If any required layer denies the request, the request fails.
- A permissive endpoint policy does not override a restrictive resource policy.
- A key policy or resource policy can still block access even when IAM allows it.

**IAM and STS validation**

For role-based access, the role trust policy controls who can assume the role, but it is not the only requirement. The caller also needs permission to call `sts:AssumeRole`, and some federation flows add extra conditions or service-specific trust relationships.

Example trust policy fragment:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": { "AWS": "arn:aws:iam::111122223333:root" },
      "Action": "sts:AssumeRole",
      "Condition": {
        "StringEquals": {
          "aws:PrincipalArn": "arn:aws:iam::111122223333:role/AppRole"
        }
      }
    }
  ]
}
```

Validation checklist:

- The caller has `sts:AssumeRole` permission.
- The role trust policy trusts the caller.
- Any session policy or permission boundary does not block the action.
- The assumed role has the downstream permissions needed for the service call.

**KMS validation**

AWS Key Management Service (KMS) access is governed primarily by IAM, the key policy, grants, and service conditions. Endpoint policy can matter when you use a KMS interface endpoint, but it is not the main authorization layer to focus on.

What to validate:

- The caller can reach the KMS endpoint privately if required.
- The IAM policy allows the KMS action.
- The key policy allows the principal or the delegated AWS service.
- Any grants required for the workload exist.
- The encryption context or condition keys match the request.

Example KMS CLI test:

```bash
aws kms describe-key --key-id arn:aws:kms:us-east-1:111122223333:key/abcd1234-5678-90ab-cdef-1234567890ab
```

If access fails, inspect:

- `kms:ViaService`
- `kms:CallerAccount`
- `kms:EncryptionContext`
- The key policy statements that delegate to IAM

**Secrets Manager validation**

Secrets Manager often appears in private access questions because workloads should retrieve secrets without using the public internet. A common pattern is to use an interface endpoint for `secretsmanager` and a secret resource policy that restricts access to a specific VPC endpoint.

Important distinction:

- A **secret resource policy** can use `aws:SourceVpce` or `aws:SourceVpc`.
- An **IAM identity policy** does not use `Principal` in the same way a resource policy does.

Example secret resource policy pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowReadOnlyFromSpecificVpce",
      "Effect": "Allow",
      "Principal": "*",
      "Action": [
        "secretsmanager:GetSecretValue",
        "secretsmanager:DescribeSecret"
      ],
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "aws:SourceVpce": "vpce-0123456789abcdef0"
        }
      }
    }
  ]
}
```

Validation steps:

- Confirm the secret policy is attached to the secret.
- Confirm the endpoint ID in the condition matches the actual interface endpoint.
- Confirm the caller has IAM permission to read the secret.
- Confirm the KMS key used by the secret also allows decrypt.

**CloudWatch Logs validation**

CloudWatch Logs is a common exam service for interface endpoint questions. The workload may need to send logs privately from EC2, ECS, EKS, Lambda, or on-premises-connected systems.

What to validate:

- The `logs` interface endpoint exists in the VPC.
- Private DNS is enabled if the application uses the standard service hostname.
- The endpoint security group allows TCP 443.
- The IAM role allows `logs:CreateLogStream` and `logs:PutLogEvents`.

Example CLI test:

```bash
aws logs describe-log-groups --region us-east-1
```

If the application cannot publish logs:

- Check the endpoint DNS resolution.
- Check the instance or task role permissions.
- Check whether the application is using the correct region.
- Check whether the endpoint is in the same VPC as the workload.

**EC2, ECR, and SSM validation**

These services often appear together in secure build or management architectures.

For **EC2** API access:

- Use the EC2 interface endpoint when instances or automation need private access to EC2 APIs.
- Validate that the client can call `ec2:DescribeInstances` or similar actions through the endpoint.

For **ECR**:

- Use both `ecr.api` and `ecr.dkr` interface endpoints for private image pulls in many architectures.
- Validate that the node or task role can authenticate and pull images.
- Confirm S3 access if image layers depend on S3-backed retrieval in the architecture.

For **SSM**:

- Use `ssm`, `ssmmessages`, and `ec2messages` interface endpoints for Systems Manager managed instances.
- Validate that the instance profile has the required SSM permissions.
- Confirm the SSM agent is running and the instance can register.

Example validation commands:

```bash
aws ssm describe-instance-information
aws ecr describe-repositories
aws ec2 describe-instances --region us-east-1
```

Common endpoint combinations:

- **SSM managed instance access**: `ssm`, `ssmmessages`, `ec2messages`
- **Private container pulls**: `ecr.api`, `ecr.dkr`, often plus `s3`
- **Private logging**: `logs`
- **Private secrets retrieval**: `secretsmanager`, often plus `kms`

**DNS and name resolution**

DNS is a frequent failure point in endpoint validation. Private connectivity can exist, but the application may still resolve the public hostname if private DNS is disabled or misconfigured.

What to check:

- Private DNS is enabled on the interface endpoint.
- The VPC has DNS resolution and DNS hostnames enabled.
- The client uses the VPC resolver.
- Custom DNS servers forward the relevant AWS zones correctly.

Example checks from an EC2 instance:

```bash
nslookup secretsmanager.us-east-1.amazonaws.com
dig logs.us-east-1.amazonaws.com
```

If you use a custom private service behind PrivateLink, the certificate and hostname must match the service design. Do not assume the `vpce.amazonaws.com` name is always the certificate name. The certificate presented depends on how the service is configured and what hostname the client uses.

**Certificate validation for private services**

For custom services behind PrivateLink, TLS validation matters. The client must trust the certificate chain and the hostname must match the certificate subject or subject alternative name.

Best practices:

- Use a certificate issued by a trusted CA.
- Use **ACM Private CA** when you need an internal trust anchor for private services.
- Rotate certificates before expiration and validate that clients trust the new chain.
- Ensure the service name and DNS name align with the certificate SANs.

Example OpenSSL check:

```bash
openssl s_client -connect service.internal.example.com:443 -servername service.internal.example.com
```

Validation focus:

- The certificate chain is complete.
- The hostname matches the certificate.
- The client trusts the issuing CA.
- The backend service presents the expected certificate after rotation.

**Shared VPC and AWS RAM validation**

In shared networking designs, the endpoint may be created in one account and consumed by workloads in another, or the subnet may be shared through AWS Resource Access Manager (AWS RAM).

What to validate:

- The subnet is actually shared to the consuming account.
- The endpoint is created in the correct owning account and VPC.
- Security groups and route tables are associated with the shared subnet as expected.
- The consuming workload can resolve and reach the endpoint.

Useful checks:

```bash
aws ec2 describe-subnets --subnet-ids subnet-0123456789abcdef0
aws ec2 describe-vpc-endpoints --filters Name=vpc-id,Values=vpc-0123456789abcdef0
```

In multi-account environments, a common failure is assuming that a shared subnet automatically grants endpoint access. It does not. You still need the correct endpoint, DNS, security group, and policy configuration.

**Troubleshooting workflow**

Use a layered approach so you can isolate the failure quickly.

1. Confirm the intended endpoint type.
2. Confirm DNS resolves to the expected target.
3. Confirm network reachability to the endpoint IP or route target.
4. Confirm the endpoint policy allows the action.
5. Confirm the resource policy or service policy allows the principal.
6. Confirm IAM permissions on the caller.
7. Confirm service-specific prerequisites such as KMS, SSM agent, or ECR auth.

Useful commands:

```bash
aws ec2 describe-vpc-endpoints --vpc-endpoint-ids vpce-0123456789abcdef0
aws ec2 describe-network-interfaces --filters Name=vpc-endpoint-id,Values=vpce-0123456789abcdef0
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::111122223333:role/AppRole \
  --action-names secretsmanager:GetSecretValue \
  --resource-arns arn:aws:secretsmanager:us-east-1:111122223333:secret:MySecret-abc123
```

For service-specific diagnostics:

- Check CloudTrail for `AccessDenied` or `UnauthorizedOperation`.
- Check VPC Flow Logs for rejected traffic.
- Check the service’s own access logs or error messages.
- Check whether the request is reaching the intended endpoint at all.

**Common exam patterns**

Expect questions that ask you to choose the minimum secure change.

Typical patterns include:

- An application in a private subnet cannot reach S3 without internet access.
  - Use a gateway endpoint and verify the route table association.
- A workload must call AWS APIs privately.
  - Use an interface endpoint and enable private DNS.
- A secret must be readable only from one VPC endpoint.
  - Use a secret resource policy with `aws:SourceVpce`.
- A custom service must be exposed only to approved accounts.
  - Use PrivateLink, allowed principals, and optional acceptance required.
- A managed instance cannot register with Systems Manager.
  - Add the `ssm`, `ssmmessages`, and `ec2messages` endpoints and verify the instance profile.

**Practical validation examples**

S3 private access from an EC2 instance:

```bash
aws sts get-caller-identity
aws s3api list-objects-v2 --bucket my-private-bucket --max-items 5
```

Secrets Manager private access:

```bash
aws secretsmanager get-secret-value \
  --secret-id MySecret \
  --region us-east-1
```

SSM connectivity:

```bash
aws ssm describe-instance-information --region us-east-1
```

ECR image pull validation:

```bash
aws ecr get-login-password --region us-east-1 | \
docker login --username AWS --password-stdin 111122223333.dkr.ecr.us-east-1.amazonaws.com
```

If the command fails, determine whether the issue is:

- DNS
- Security group
- Route table
- Endpoint policy
- IAM policy
- Resource policy
- Service-side prerequisite

**Key takeaways**

- Gateway endpoints change route tables; interface endpoints create ENIs.
- Private DNS is often the difference between a working private path and a silent public path.
- Endpoint policy is only one layer; IAM and resource policies still matter.
- PrivateLink providers must validate allowed principals and acceptance settings.
- Service-specific endpoint combinations matter for SSM, ECR, Logs, and similar workloads.
- Certificate trust and hostname validation are essential for custom private services.
