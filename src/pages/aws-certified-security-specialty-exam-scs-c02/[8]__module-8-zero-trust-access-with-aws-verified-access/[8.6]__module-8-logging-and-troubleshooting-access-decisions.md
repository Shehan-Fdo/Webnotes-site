## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Why access decisions fail**

AWS authorization problems usually come from one of a few layers: identity-based policy, resource-based policy, permissions boundary, session policy, service control policy (SCP), VPC endpoint policy, or a service-specific control such as AWS Key Management Service (KMS) key policy. The exam often tests whether you can identify the *first* place to look rather than memorizing a single universal order.

A useful mental model is:

- The request must be authenticated.
- The relevant policy types must collectively allow the action.
- Any explicit deny in an applicable policy type wins.
- Some services add their own authorization layer or require a second permission path, such as KMS decrypt permissions for encrypted data.

For troubleshooting, you want to answer three questions quickly:

- Who made the request?
- What exact API action was attempted?
- Which policy or service layer blocked it?

**CloudTrail as the primary evidence source**

AWS CloudTrail records management events for most control-plane API calls and is usually the first place to inspect for access failures. The event record tells you what was attempted, by whom, from where, and what error occurred.

Key fields to inspect:

- **eventName**: the API action, such as `AssumeRole`, `PutObject`, or `Decrypt`
- **errorCode**: often `AccessDenied`, `Client.UnauthorizedOperation`, or a service-specific error
- **errorMessage**: sometimes includes the denied action or missing condition
- **userIdentity**: the caller type, ARN, account, session context, and assumed-role details
- **sourceIPAddress**: useful for spotting unexpected networks, NAT paths, or service calls
- **requestParameters**: shows the target resource, bucket name, key ID, role ARN, or condition-related inputs
- **responseElements**: may include request IDs, temporary credentials, or service-specific response data
- **recipientAccountId**: helps in cross-account troubleshooting
- **additionalEventData**: can include useful context such as MFA usage, TLS details, or service integration hints

A typical CloudTrail event for a denied API call may look like this:

```json
{
  "eventTime": "2026-04-26T12:34:56Z",
  "eventSource": "s3.amazonaws.com",
  "eventName": "GetObject",
  "awsRegion": "us-east-1",
  "sourceIPAddress": "10.0.2.15",
  "userIdentity": {
    "type": "AssumedRole",
    "arn": "arn:aws:sts::111122223333:assumed-role/AppRole/session-123"
  },
  "requestParameters": {
    "bucketName": "example-bucket",
    "key": "reports/q1.pdf"
  },
  "responseElements": null,
  "errorCode": "AccessDenied",
  "errorMessage": "Access Denied"
}
```

When you see a denial, do not stop at `AccessDenied`. Use the surrounding fields to infer the layer:

- `AssumedRole` often means you must inspect the role trust policy, identity policy, permissions boundary, and session policy.
- A cross-account `recipientAccountId` often means a resource policy or KMS key policy is involved.
- A private IP in `sourceIPAddress` can indicate a VPC endpoint path, but it does not prove an endpoint policy issue by itself.

**CloudTrail lookup and query techniques**

For quick triage, use the CloudTrail console event history or the AWS CLI. Filter by action, principal, and error code to reduce noise.

Example CLI lookup:

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=PutObject \
  --max-results 20
```

Useful filters:

- `eventName` for the API action
- `userIdentity.arn` or `userIdentity.principalId` for the caller
- `errorCode` for denied or failed requests
- `sourceIPAddress` for network-origin clues
- `eventSource` for the service namespace

If you use CloudTrail Lake, SQL-style queries are often faster for investigations:

```sql
SELECT
  eventTime,
  eventSource,
  eventName,
  userIdentity.arn,
  sourceIPAddress,
  errorCode,
  errorMessage,
  requestParameters,
  responseElements,
  recipientAccountId,
  additionalEventData
FROM aws_cloudtrail_events
WHERE eventTime > TIMESTAMP '2026-04-26 00:00:00'
  AND errorCode IS NOT NULL
  AND eventName = 'AssumeRole'
ORDER BY eventTime DESC;
```

A practical workflow is:

1. Find the denied event.
2. Identify the exact principal and action.
3. Check whether the target is cross-account.
4. Inspect the relevant policy layers in order of effect, not in a simplistic linear sequence.
5. Correlate with service logs if the API call alone does not explain the failure.

**IAM policy evaluation nuances**

IAM authorization is not just “allow if any policy says yes.” The result depends on the policy types that apply to the request.

Important rules:

- **Explicit deny** in any applicable policy type wins.
- Identity-based policies can allow actions for a principal.
- Resource-based policies can allow access directly to a resource.
- Permissions boundaries limit the maximum permissions a principal can use.
- Session policies further restrict temporary credentials.
- SCPs limit what identities in an account can do, but they do not grant permissions.
- Some services require both identity-side and resource-side permission paths to be satisfied.

Cross-account access is a common exam trap:

- In same-account access, an identity-based policy may be enough if the service supports it.
- In cross-account access, you often need a resource-based policy on the target resource, and the caller may also need an identity-based allow.
- For some services, a resource policy can grant access directly to a principal in another account without a matching identity policy in the caller account, but this depends on the service and action.

A practical example is S3 cross-account access:

- The bucket policy can allow a principal from another account to read an object.
- The caller may still need IAM permissions in its own account depending on the service and the exact access path.
- If the object is encrypted with KMS, the KMS key policy and KMS permissions must also allow decrypt.

**Permissions boundaries and session policies**

A common troubleshooting mistake is to verify that the role or user has an allow and stop there. Permissions boundaries and session policies can still block the request.

Use cases:

- A developer role has `s3:*` in its identity policy, but the permissions boundary only allows `s3:GetObject`.
- An assumed role session is created with a restrictive session policy that removes `iam:PassRole`.
- A federated session includes session tags or a session policy that narrows access to a subset of resources.

Example of a permissions boundary effect:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowReadOnlyS3",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": "*"
    }
  ]
}
```

Even if the identity policy allows `s3:PutObject`, the boundary prevents it.

For assumed roles, inspect:

- The role’s identity policy
- The role trust policy
- The permissions boundary on the role or user
- The session policy passed to `AssumeRole`
- Any SCPs affecting the account or OU

**SCP troubleshooting and Organizations nuance**

Service control policies define the maximum permissions available in member accounts. They do not grant access by themselves.

Exam-relevant points:

- SCPs apply to accounts in an organization, not to standalone accounts.
- SCPs do not apply to the management account in the same way they apply to member accounts.
- An allow in IAM can still fail if an SCP blocks the action.
- An SCP deny is often the reason a request fails even when the identity policy looks correct.

To verify effective SCPs:

- Check the account’s OU placement.
- Review attached SCPs at the root, OU, and account level.
- Use AWS Organizations tools to inspect policy inheritance.
- Compare the denied action against the SCP’s allowed actions and explicit denies.

A useful troubleshooting habit is to ask whether the action is even possible in that account under the current organization structure before debugging the identity policy.

**Resource-based policies versus identity-based policies**

Resource-based policies are especially important for cross-account access and service integrations.

Examples:

- S3 bucket policies
- KMS key policies
- Secrets Manager resource policies
- SNS topic policies
- SQS queue policies
- Lambda function resource policies

Key distinctions:

- Identity-based policies attach to users, groups, or roles.
- Resource-based policies attach to the resource itself.
- A resource policy can sometimes grant access without a matching identity policy in the caller account, depending on the service.
- Resource policies are often the deciding factor in cross-account scenarios.

For exam prep, remember that the resource policy must name the correct principal and resource, and the action must match the service’s authorization model.

**S3 access-denied troubleshooting**

S3 is a frequent source of access confusion because bucket-level and object-level permissions are different.

Important action-to-resource mapping:

- `s3:ListBucket` uses the bucket ARN: `arn:aws:s3:::example-bucket`
- `s3:GetObject`, `s3:PutObject`, and similar object actions use the object ARN: `arn:aws:s3:::example-bucket/*`

A correct least-privilege example:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ListBucket",
      "Effect": "Allow",
      "Action": "s3:ListBucket",
      "Resource": "arn:aws:s3:::example-bucket"
    },
    {
      "Sid": "ReadObjects",
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::example-bucket/*"
    }
  ]
}
```

Common S3 troubleshooting patterns:

- A user can list the bucket but cannot read objects because the object ARN is missing.
- A user can read some objects but not others because a prefix condition or object tag condition blocks access.
- A cross-account principal can reach the bucket policy but still fail because the object is encrypted with KMS.
- A request returns `403 AccessDenied` or `404 NoSuchKey` depending on whether the caller has `s3:ListBucket` and whether the object exists.

The 403 versus 404 behavior matters:

- If the object does not exist and the caller has `s3:ListBucket`, S3 can return `404 NoSuchKey`.
- If the caller lacks `s3:ListBucket`, S3 may return `403 AccessDenied` even when the object is missing, to avoid revealing whether the key exists.
- Therefore, a 403 does not prove the object exists, and a 404 does not prove the caller has access.

Useful supporting logs:

- **S3 server access logs** for request-level access patterns
- **CloudTrail data events** for object-level API calls
- **AWS Config** for bucket policy drift or public access changes

**Cross-account S3 plus KMS scenario**

A classic exam scenario is a principal in Account A reading an object in Account B that is encrypted with a customer managed KMS key.

You typically need:

- A bucket policy in Account B allowing the principal or source account
- An identity policy in Account A allowing S3 access, depending on the access path
- A KMS key policy in Account B that allows the account to use IAM policies for the key, or directly allows the principal
- A KMS permission such as `kms:Decrypt` for the caller path
- If using AWS services like S3 with SSE-KMS, the service integration must be able to call KMS on behalf of the request

Troubleshooting order:

1. Confirm the S3 bucket policy allows the principal.
2. Confirm the object ARN and prefix are correct.
3. Confirm the KMS key policy permits the account or principal path.
4. Confirm the caller has `kms:Decrypt` if required.
5. Check CloudTrail for both S3 and KMS events.

**KMS authorization and encryption context**

KMS access is often misunderstood because multiple layers can be involved.

Important concepts:

- The **key policy** controls who can use the KMS key.
- IAM policies can grant KMS permissions only if the key policy allows the account to use IAM policies for that key, or if the principal is explicitly allowed in the key policy.
- **Grants** can authorize temporary or service-specific use of a key without editing the key policy.
- **Encryption context** can be required by policy conditions and must match at decrypt time.

A common failure pattern is an encryption context mismatch:

- Data was encrypted with context `{"App":"Payroll","Env":"Prod"}`
- Decrypt is attempted with a different context or no context
- KMS denies the request even though the principal has `kms:Decrypt`

Example KMS policy condition using encryption context:

```json
{
  "Sid": "AllowDecryptForPayrollOnly",
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::111122223333:role/AppRole"
  },
  "Action": "kms:Decrypt",
  "Resource": "*",
  "Condition": {
    "StringEquals": {
      "kms:EncryptionContext:App": "Payroll"
    }
  }
}
```

Service-integrated KMS patterns to remember:

- **S3 SSE-KMS**: S3 calls KMS on behalf of the request.
- **EBS encryption**: EC2 and EBS use KMS for volume and snapshot operations.
- **RDS encryption**: database creation, snapshot restore, and cross-account snapshot sharing can involve KMS.
- **Lambda environment variables**: Lambda may need KMS permissions to decrypt encrypted environment variables.

Useful CloudTrail clues for KMS:

- `eventName` such as `Decrypt`, `GenerateDataKey`, `Encrypt`, `CreateGrant`
- `requestParameters.encryptionContext`
- `userIdentity` showing whether the caller is a service principal or an assumed role
- `errorCode` and `errorMessage` for denied decrypts

**Secrets Manager troubleshooting**

Secrets Manager access can fail for two separate reasons: authorization to the secret and authorization to the KMS key used to encrypt it.

What to check:

- The secret’s resource policy, if cross-account access is involved
- The caller’s identity policy
- The KMS key policy and `kms:Decrypt` permission
- Any condition keys on the secret policy, such as VPC endpoint restrictions or principal tags

A resource policy usually applies to the specific secret ARN rather than using a broad wildcard in practice. For exam purposes, think in terms of the secret resource and the principal that needs access.

Example pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCrossAccountRead",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::444455556666:role/ReadSecretsRole"
      },
      "Action": [
        "secretsmanager:GetSecretValue",
        "secretsmanager:DescribeSecret"
      ],
      "Resource": "arn:aws:secretsmanager:us-east-1:111122223333:secret:prod/db-abc123"
    }
  ]
}
```

If the secret uses a customer managed KMS key, you still need KMS access for decryption.

**STS and assumed-role failures**

AWS Security Token Service (STS) issues temporary credentials, so failures around `AssumeRole` often involve more than one policy.

Possible causes:

- The caller lacks `sts:AssumeRole` in its identity policy
- The role trust policy does not trust the caller principal
- The trust policy has conditions that fail, such as MFA, external ID, source identity, or principal tags
- A permissions boundary blocks the caller from assuming the role
- An SCP denies the action
- A session policy or role session duration constraint causes the request to fail

CloudTrail clues:

- `eventName = AssumeRole`
- `userIdentity` shows the caller principal
- `requestParameters.roleArn` shows the target role
- `requestParameters.externalId` or MFA-related fields may explain a condition failure
- `errorMessage` may mention not authorized or invalid identity token

A practical troubleshooting checklist:

1. Verify the caller can call `sts:AssumeRole`.
2. Verify the role trust policy names the caller or federated identity correctly.
3. Check trust policy conditions such as `aws:MultiFactorAuthPresent`, `sts:ExternalId`, or `aws:PrincipalTag`.
4. Check SCPs and permissions boundaries.
5. Confirm the session duration and session policy are not overly restrictive.

**VPC endpoint policy and private network access**

VPC endpoint policies are a common source of “works from public subnet, fails from private subnet” problems, but they are not the only cause. Route tables, security groups, network ACLs, DNS resolution, and missing endpoint associations can also break access.

Endpoint types matter:

- **Gateway endpoints** are used for S3 and DynamoDB.
- **Interface endpoints** are used for many other AWS services and use AWS PrivateLink.

Gateway endpoint policy example for S3:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowOnlyFinanceBucket",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::finance-data",
        "arn:aws:s3:::finance-data/*"
      ]
    }
  ]
}
```

For a gateway endpoint policy, you do not use `Principal` because the policy is attached to the endpoint and controls traffic through that endpoint.

Interface endpoint policies can be more service-specific and may include conditions that restrict actions or resources. A deny can occur even when IAM allows the action.

Practical troubleshooting steps:

1. Confirm the instance or subnet routes traffic to the endpoint.
2. Confirm DNS resolves the service name to the endpoint path when expected.
3. Confirm the endpoint is associated with the correct route tables or subnets.
4. Review the endpoint policy for action or resource restrictions.
5. Check whether the service request is actually using the endpoint path.

CloudTrail may show the service API failure, but the endpoint policy itself is not always surfaced as a distinct denial reason. Use network path evidence alongside API logs.

**Service-specific logs that support CloudTrail**

CloudTrail tells you what API was attempted. Other logs often explain why the service rejected it.

Useful sources:

- **S3 server access logs**: request-level access patterns, requester, status code, and error code
- **ALB access logs**: target response codes, client IPs, and load balancer behavior
- **CloudWatch Logs**: application-level authorization failures, Lambda execution errors, and custom app logs
- **AWS Config**: resource configuration drift, policy changes, and compliance history
- **VPC Flow Logs**: network path confirmation for private subnet issues
- **KMS CloudTrail events**: decrypt, encrypt, and grant operations
- **RDS logs**: database authentication or integration issues when the service is involved

Use these logs together rather than in isolation. For example, a denied S3 request may show up in CloudTrail, while the application log shows a missing prefix, and the server access log confirms the 403.

**High-value exam scenarios**

Cross-account S3 read with KMS:

- Bucket policy allows the external role
- Identity policy in the caller account allows S3 access
- KMS key policy allows the account or role path
- `kms:Decrypt` is present
- CloudTrail shows both S3 and KMS events

Assumed role with MFA condition:

- Trust policy requires MFA
- Caller has `sts:AssumeRole`
- CloudTrail shows `AssumeRole` denied
- `additionalEventData` or request context indicates MFA was absent

Session tags or source identity mismatch:

- Trust policy requires a tag such as `Department=Finance`
- The session is created without the tag or with the wrong value
- The role assumption fails even though the identity policy looks correct

Private subnet access failure:

- Route table missing the endpoint route
- Security group blocks egress
- DNS does not resolve to the endpoint
- Endpoint policy denies the action
- IAM allows the action, but the network path is wrong

KMS decrypt failure for an AWS service:

- S3, EBS, RDS, or Lambda is calling KMS on behalf of the user
- The service principal or grant is missing
- The key policy does not allow the account to use IAM policies
- Encryption context does not match

**Practical troubleshooting checklist**

1. Identify the exact API call in CloudTrail.
2. Note the caller identity, account, and session type.
3. Check whether the request is cross-account.
4. Inspect identity policy, resource policy, permissions boundary, session policy, and SCPs.
5. Verify service-specific requirements such as KMS, endpoint policy, or trust policy conditions.
6. Correlate with service logs, network logs, and configuration history.
7. Confirm the resource ARN, action name, and condition keys are correct.

**Quick reference: likely denial layer and first evidence source**

| Symptom | Likely layer to inspect first | First source to check |
|---|---|---|
| `AccessDenied` on `AssumeRole` | Trust policy, caller policy, MFA/external ID conditions, SCP, boundary | CloudTrail `AssumeRole` event |
| S3 object read denied | Bucket policy, object ARN, KMS key policy | CloudTrail data event, S3 server access logs |
| `403` or `404` confusion on S3 | `s3:ListBucket` and object existence | CloudTrail + S3 server access logs |
| KMS decrypt failure | Key policy, grants, encryption context, service integration | CloudTrail KMS event |
| Cross-account secret read fails | Secret resource policy and KMS permissions | CloudTrail + Secrets Manager event |
| Private subnet only failure | Route table, endpoint association, endpoint policy, SG, NACL, DNS | VPC Flow Logs + CloudTrail |
| IAM allow but request still fails | Permissions boundary, session policy, SCP | IAM policy simulator, Organizations, CloudTrail |
| Service works in console but not app | Session policy, role assumption, endpoint path, app credentials | CloudTrail + app logs |

**CLI commands you should know**

Inspect a role trust policy:

```bash
aws iam get-role --role-name AppRole
```

Inspect attached identity policies:

```bash
aws iam list-attached-role-policies --role-name AppRole
aws iam list-role-policies --role-name AppRole
```

Inspect a bucket policy:

```bash
aws s3api get-bucket-policy --bucket example-bucket
```

Inspect a KMS key policy:

```bash
aws kms get-key-policy \
  --key-id arn:aws:kms:us-east-1:111122223333:key/1234abcd-12ab-34cd-56ef-1234567890ab \
  --policy-name default
```

Inspect a VPC endpoint policy:

```bash
aws ec2 describe-vpc-endpoints \
  --vpc-endpoint-ids vpce-0123456789abcdef0
```

Look up recent denied events:

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=ErrorCode,AttributeValue=AccessDenied \
  --max-results 10
```

**How to think on the exam**

When you see an access failure, avoid assuming a single “IAM problem.” Instead, map the request to the full authorization path:

- Identity policy
- Resource policy
- Permissions boundary
- Session policy
- SCP
- Service-specific policy or grant
- Network path and endpoint policy
- Supporting logs

The best answer is usually the one that identifies the narrowest missing permission or the exact policy layer that can explain the observed symptom. If multiple layers could apply, choose the one that matches the service, account relationship, and evidence in CloudTrail.
