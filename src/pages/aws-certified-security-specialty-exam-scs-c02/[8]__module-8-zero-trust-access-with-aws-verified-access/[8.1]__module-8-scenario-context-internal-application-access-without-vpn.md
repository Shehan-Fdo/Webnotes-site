## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Scenario overview**

This scenario is about giving users secure access to internal applications without exposing the network through a traditional VPN. In exam questions, the key decision is usually whether you need **application-level access** or **network-level access**.

- Use **AWS Verified Access** when users should reach specific applications without joining the corporate network.
- Use **AWS Client VPN** when users need broader network access to private subnets, internal hosts, or multiple protocols that behave like being on the network.
- Use **ALB authentication** with an identity provider when you only need sign-in at the application edge and do not need device posture checks.
- Use **Zero Trust** principles when access should depend on identity, device trust, and policy rather than network location.

A common exam trap is assuming “no VPN” always means “browser only.” That is not accurate. Verified Access can support browser-based access and, depending on configuration and endpoint type, non-browser TCP applications as well.

**When to choose Verified Access**

Verified Access is designed to provide secure access to corporate applications without requiring a VPN. It is a strong fit when you want to avoid exposing the internal network and instead authorize access per application.

- **Identity-aware** access decisions are based on user identity from a trust provider.
- **Device-aware** access can include posture signals from a device trust provider.
- **Per-application** access reduces lateral movement compared with network-level connectivity.
- **No full tunnel** is needed, which simplifies access for users and reduces the blast radius of compromise.

Use it when the requirement says:
- “Users must access internal apps from unmanaged or managed devices.”
- “Do not allow direct network access to the VPC.”
- “Enforce identity and device posture before app access.”
- “Avoid VPN client installation if possible.”

**Verified Access versus Client VPN**

These services solve different problems.

| Capability | AWS Verified Access | AWS Client VPN |
|---|---|---|
| Access model | Application-level | Network-level |
| User experience | Access to specific apps | Access to private network resources |
| Device posture | Supported through device trust integrations | Not the primary control model |
| Browser access | Common use case | Possible, but not the main pattern |
| Non-browser TCP apps | Supported in some configurations | Supported as network access |
| Lateral movement risk | Lower | Higher, because users join the network |
| Best fit | Zero Trust app access | Remote access to private subnets |

For exam questions, if the user needs access to a whole subnet, file shares, SSH, or many internal services, Client VPN is often the better answer. If the user needs access to a specific internal web app or a limited set of applications with identity and posture checks, Verified Access is usually the better answer.

**Verified Access resource model**

The service model is important for exam prep because the resource names are easy to confuse.

- **Verified Access instance**
  - The top-level service container in your account and Region.
  - It is associated with one or more trust providers.
- **Trust provider**
  - Supplies identity or device trust signals.
  - Commonly includes an identity provider such as an OIDC (OpenID Connect) provider.
  - Can also include a device trust provider for posture-based decisions.
- **Verified Access group**
  - A logical grouping used to organize access policies and endpoints.
  - Helps you manage multiple applications with similar policy requirements.
- **Verified Access endpoint**
  - The access entry point for an application.
  - Associated with a VPC and configured to reach the application target.
- **Application attachment / target**
  - The backend application resource that the endpoint forwards to.
  - Commonly an Application Load Balancer (ALB); some scenarios may use a Network Load Balancer (NLB) depending on the application type.

A useful mental model is:

1. The user authenticates through the trust provider.
2. Verified Access evaluates the access policy.
3. The endpoint forwards allowed traffic to the application target.
4. The backend application responds through the configured path.

**How the traffic flow works**

Verified Access sits in front of the application. It is not a general-purpose network tunnel.

- The user connects to the Verified Access endpoint.
- The endpoint consults the configured trust provider for identity and, if enabled, device trust.
- The policy engine evaluates the request context.
- If allowed, traffic is forwarded to the backend application target.
- The backend application remains private and is not directly exposed to the internet.

This design is useful when you want to keep the application private while still avoiding a VPN.

> Do not describe Verified Access as “browser-only” in exam answers. That framing is too narrow and can lead to the wrong service choice.

**Device posture and trust signals**

Device posture is one of the main differentiators between Verified Access and simpler identity-only patterns.

In practice, posture signals usually come from a **device trust provider** or endpoint management integration rather than from a generic identity provider claim alone.

Common posture examples include:
- Device is managed by the organization.
- Device has a compliant security agent installed.
- Device uses disk encryption.
- Device has current antivirus or endpoint detection and response status.
- Device meets OS version or patch-level requirements.

In exam scenarios, posture checks are often described as:
- “Only managed devices can access the app.”
- “Block access from jailbroken or rooted devices.”
- “Require compliant endpoints.”
- “Allow access only from devices enrolled in MDM.”

Here, **MDM** means Mobile Device Management. The posture source is typically an external device management or trust integration, not a simple IAM (Identity and Access Management) attribute.

**Policy evaluation concepts**

Verified Access policies are not written like generic allow/deny JSON for arbitrary application paths. The policy model is service-specific and evaluates request context such as identity, device trust, and other attributes exposed by the service.

Think in terms of conditions, not raw network rules.

- Identity attributes can include user, group, or claims from the identity provider.
- Device attributes can include managed/compliant status.
- Context can include application-specific metadata and request details.
- The result is an allow or deny decision for the request.

A practical policy intent might be:
- Allow members of the finance group from compliant managed devices.
- Deny access if the device is unmanaged.
- Allow contractors only to a limited application group.
- Require stronger controls for sensitive applications.

**Example architecture for an internal web app**

A realistic deployment often looks like this:

1. Users access a private HR portal from the internet.
2. The portal is behind an ALB in a private VPC.
3. Verified Access endpoint fronts the application.
4. An OIDC identity provider authenticates the user.
5. A device trust provider checks whether the laptop is managed and compliant.
6. Verified Access policy allows only approved users and devices.
7. The ALB forwards traffic to the portal targets in private subnets.

This pattern keeps the application private while avoiding a VPN client for every user.

```text
User -> Verified Access endpoint -> policy evaluation -> ALB -> private app targets
```

**Practical configuration example**

The exact AWS CLI syntax can vary by service version and resource type, so treat this as a conceptual example of the workflow rather than a copy-paste production script.

```bash
# Example workflow: create a Verified Access instance and attach trust providers
aws ec2 create-verified-access-instance \
  --description "Internal app access"

aws ec2 create-verified-access-trust-provider \
  --trust-provider-type user-identity \
  --user-trust-provider-type oidc \
  --policy-reference-name corp-idp \
  --description "Corporate OIDC provider"

aws ec2 create-verified-access-group \
  --verified-access-instance-id vai-0123456789abcdef0 \
  --description "Finance apps"

aws ec2 create-verified-access-endpoint \
  --verified-access-group-id vag-0123456789abcdef0 \
  --endpoint-type load-balancer \
  --attachment-type application-load-balancer \
  --application-network-interface-options '{
    "NetworkInterfaceId": "eni-0123456789abcdef0"
  }'
```

Use the AWS console or current AWS CLI documentation to confirm the exact parameter names for your Region and endpoint type. For exam purposes, focus on the resource relationships and traffic flow rather than memorizing brittle syntax.

**Example access policy intent**

The following example is intentionally simplified to show the logic, not the exact service syntax.

```text
Allow if:
- user is authenticated by the corporate IdP
- user belongs to the approved group
- device is managed and compliant
- request is for the internal application endpoint
Otherwise deny
```

If you need to express this in a real deployment, use the Verified Access policy language and the attributes exposed by the configured trust providers.

**Common exam scenarios**

- **Internal web app for employees only**
  - Best answer is often Verified Access if the goal is no VPN and per-app access.
- **Private admin console for managed laptops**
  - Verified Access is a strong fit because device posture matters.
- **SSH to many EC2 instances**
  - Client VPN or Session Manager is usually more appropriate than Verified Access.
- **Need access to multiple private services across subnets**
  - Client VPN is usually the better network-level solution.
- **Need only sign-in at the app edge**
  - ALB authentication with OIDC or Cognito may be sufficient if device posture is not required.

**Verified Access versus ALB authentication**

ALB authentication and Verified Access can look similar at first glance because both can sit in front of an application and use an identity provider. The difference is in the control depth.

- **ALB authentication**
  - Good for identity-based access at the load balancer.
  - Useful when the app only needs user sign-in.
  - Does not provide the same device trust model as Verified Access.
- **Verified Access**
  - Adds device trust and Zero Trust access decisions.
  - Better when the exam mentions managed devices, posture, or stronger access governance.
  - Can be used to keep the application private without exposing the network.

If the question says “authenticate users with OIDC” and nothing about device posture or no-VPN app access, ALB authentication may be enough. If the question adds “only compliant corporate devices” or “no network access to the VPC,” Verified Access is the more precise answer.

**Security design considerations**

- Keep the application in private subnets.
- Expose only the Verified Access entry point and the load balancer path required for the app.
- Use least privilege in the access policy.
- Separate applications into groups when policy requirements differ.
- Log and monitor access decisions for audit and troubleshooting.

> Security groups do not allow a rule like “from Verified Access-managed traffic path.” Use valid sources such as CIDRs, prefix lists, or security group references where applicable.

**Troubleshooting checklist**

When access fails, check the failure point in order from the edge inward.

1. Verify the endpoint is associated with the correct Verified Access group and application target.
2. Confirm the trust provider is healthy and correctly configured.
3. Check whether the policy evaluates to allow for the user and device context.
4. Validate the ALB listener, target group, and target health.
5. Confirm backend security groups and network ACLs allow the required traffic between the load balancer and targets.
6. Review application logs for authentication or authorization failures.

Useful observability signals include:
- Verified Access logs for policy decisions.
- CloudTrail for configuration changes.
- ALB access logs for request patterns and status codes.
- Application logs for backend authorization or session issues.

If users can reach the endpoint but are denied, the issue is often policy or trust-provider related. If users are allowed but the app still fails, the issue is more likely in the ALB listener, target group health, or backend connectivity.

**Exam traps to avoid**

- Do not confuse **application-level** access with **network-level** access.
- Do not assume Verified Access is only for browsers.
- Do not treat device posture as a generic IAM claim.
- Do not use Client VPN when the requirement is specifically to avoid network access.
- Do not choose ALB authentication when the question explicitly requires device trust or managed-device enforcement.
- Do not overstate the role of security groups in Verified Access troubleshooting; the access decision happens before backend connectivity.

**Quick decision guide**

- Need access to one internal app without VPN: choose **Verified Access**.
- Need access to many private resources or subnets: choose **Client VPN**.
- Need only user sign-in at the load balancer: choose **ALB authentication**.
- Need device compliance checks: choose **Verified Access**.
- Need SSH or broad TCP access like being on the network: choose **Client VPN** or another network/session-based approach.

**Key takeaways**

Verified Access is the exam-friendly answer for secure, per-application access to internal resources without a VPN, especially when identity and device posture both matter. Client VPN is broader and network-oriented. ALB authentication is simpler and identity-focused. The best answer depends on whether the question asks for application-level Zero Trust access or full network connectivity.
