## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Overview**

AWS private connectivity choices solve different problems. A VPN gives you network-level reachability across encrypted tunnels, while AWS PrivateLink and VPC endpoints give you service-level private access without exposing traffic to the public internet. For exam purposes, you need to compare them by scope, routing, authentication, DNS behavior, and operational overhead.

The key distinction is simple: use a VPN when you need broad network access; use PrivateLink or VPC endpoints when you need private access to a specific AWS service or application.

**Core access patterns**

- **AWS Site-to-Site VPN** connects an on-premises network to a VPC over IPsec tunnels.
- **AWS Client VPN** connects individual users or devices to AWS using an OpenVPN-based managed client VPN service.
- **AWS Direct Connect** provides dedicated private network connectivity and often complements VPN for resilience or hybrid routing.
- **AWS PrivateLink** exposes a specific service privately through interface endpoints and endpoint services.
- **Gateway VPC endpoints** provide private access to supported AWS services through route tables, mainly Amazon S3 and DynamoDB.

**VPN-style access vs service-private access**

VPN-style access creates a path into a network. PrivateLink creates a path to a service. That difference affects everything from routing to authorization.

| Capability | Site-to-Site VPN | Client VPN | PrivateLink / Interface Endpoint | Gateway Endpoint |
|---|---|---|---|---|
| Primary use | Network-to-network connectivity | User-to-VPC connectivity | Private access to a specific service | Private access to S3/DynamoDB |
| Scope | Broad network reachability | Broad network reachability for authenticated clients | Narrow, service-specific | Narrow, service-specific |
| Protocol model | IP routing over IPsec | IP routing over TLS-based client tunnel | Service-specific TCP/HTTPS access | Route-table based to supported services |
| Security groups | Not for the VPN tunnel itself | Not for the tunnel itself; apply to associated resources and endpoint ENIs | Yes, for interface endpoints | No security groups on gateway endpoints |
| DNS behavior | Uses normal network DNS design | Often needs split DNS or route-aware DNS planning | Private DNS can map service names to endpoint IPs | Uses route-table path, not private ENIs |
| User authentication | Network device authentication, tunnel auth | User auth with certificates, SAML, or federated identity | IAM, endpoint policies, resource policies | IAM and resource policies where supported |
| Best fit | Hybrid network access | Remote workforce access | Private service consumption | S3/DynamoDB private routing |

**AWS Site-to-Site VPN**

Site-to-Site VPN is the classic hybrid connectivity option. You use it when an on-premises data center, branch office, or another network needs routed access to AWS resources inside a VPC.

It is built around IPsec tunnels between a customer gateway and a virtual private gateway or transit gateway. In practice, you care about route propagation, tunnel redundancy, and BGP (Border Gateway Protocol) if you want dynamic routing.

- **Use it when** you need network-to-network connectivity.
- **Use it when** multiple internal hosts or subnets must reach AWS resources.
- **Use it when** you need a standard hybrid design without exposing services publicly.
- **Do not use it** as a substitute for a single application endpoint if PrivateLink is enough.

Operationally, Site-to-Site VPN is about routing and resilience:

- You typically configure two tunnels for high availability.
- You can use static routes or BGP.
- Route propagation can simplify updates when the remote network changes.
- Latency and throughput depend on the internet path, so it is not the same as dedicated private transport.

Example AWS CLI workflow:

```bash
aws ec2 create-vpn-gateway --type ipsec.1
aws ec2 create-customer-gateway \
  --type ipsec.1 \
  --public-ip 203.0.113.10 \
  --bgp-asn 65010

aws ec2 create-vpn-connection \
  --type ipsec.1 \
  --customer-gateway-id cgw-1234567890abcdef0 \
  --vpn-gateway-id vgw-1234567890abcdef0
```

**AWS Client VPN**

Client VPN is for remote users or devices that need secure access into AWS networks. It is managed, scalable, and user-centric. Unlike Site-to-Site VPN, it is not for connecting two networks; it is for connecting a client endpoint to a VPC or multiple VPCs.

Client VPN commonly uses:

- **Mutual certificate authentication** with client certificates and a server certificate.
- **SAML** for federated identity and single sign-on.
- **Federated authentication** through identity providers that integrate with AWS-supported auth flows.

Operationally, Client VPN depends on several separate controls:

- **Authentication** determines who can connect.
- **Authorization rules** determine which destination CIDRs the connected client may reach.
- **Routes** determine where the endpoint sends traffic after the client is connected.
- **Association** binds the endpoint to one or more subnets in a VPC.
- **Split-tunnel** behavior determines whether only VPC-bound traffic goes through the tunnel or all client traffic does.

A common exam trap is confusing authorization with routing. Authorization rules allow access to a destination network; routes define how traffic is forwarded through the endpoint. They are related but not the same.

Example configuration concepts:

```bash
aws ec2 create-client-vpn-endpoint \
  --client-cidr-block 10.250.0.0/22 \
  --server-certificate-arn arn:aws:acm:us-east-1:123456789012:certificate/server-cert-id \
  --authentication-options Type=certificate-authentication,MutualAuthentication={ClientRootCertificateChainArn=arn:aws:acm:us-east-1:123456789012:certificate/root-cert-id} \
  --connection-log-options Enabled=false \
  --split-tunnel
```

Then you associate subnets and add routes and authorization rules separately:

```bash
aws ec2 associate-client-vpn-target-network \
  --client-vpn-endpoint-id cvpn-endpoint-1234567890abcdef0 \
  --subnet-id subnet-0123456789abcdef0

aws ec2 authorize-client-vpn-ingress \
  --client-vpn-endpoint-id cvpn-endpoint-1234567890abcdef0 \
  --target-network-cidr 10.0.0.0/16 \
  --authorize-all-groups

aws ec2 authorize-client-vpn-route \
  --client-vpn-endpoint-id cvpn-endpoint-1234567890abcdef0 \
  --destination-cidr-block 10.0.0.0/16 \
  --target-vpc-subnet-id subnet-0123456789abcdef0
```

> `authorize-client-vpn-ingress` is about allowing clients to reach the associated network; it is not a generic subnet firewall rule and it does not replace route configuration.

Split-tunnel matters operationally:

- **Split-tunnel enabled** sends only traffic for authorized destinations through the VPN.
- **Full-tunnel** sends all client traffic through AWS, which increases control but also increases bandwidth use and dependency on the VPN path.
- Split-tunnel is usually preferred for remote access unless policy requires full inspection.

Troubleshooting Client VPN often involves:

- Client route conflicts with local networks.
- Missing authorization rules for the destination CIDR.
- Missing route association to the target subnet.
- DNS resolution issues when private hosted zones are not reachable through the tunnel.
- Certificate trust problems or SAML assertion failures.
- Security group rules on the associated resources blocking traffic after the tunnel is established.

**AWS PrivateLink and interface endpoints**

PrivateLink gives you private access to a specific service without creating a general-purpose network tunnel. In AWS, this usually means an interface VPC endpoint backed by elastic network interfaces in your subnets.

This is not a VPN replacement. It does not provide broad network access, and it does not let a client browse arbitrary RFC 1918 (private) networks. It is best when you want a private path to one AWS service, a SaaS provider, or a service exposed through an endpoint service.

Use PrivateLink when:

- You want to keep traffic off the public internet.
- You only need access to a specific service, not an entire network.
- You want to reduce exposure by avoiding inbound internet-facing endpoints.
- You want to control access with IAM, endpoint policies, and resource policies.

Interface endpoints support security groups because they create ENIs in your subnets. That means you can restrict which sources may connect to the endpoint. Gateway endpoints do not use security groups because they are route-table based and do not create ENIs.

Example interface endpoint creation:

```bash
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0123456789abcdef0 \
  --vpc-endpoint-type Interface \
  --service-name com.amazonaws.us-east-1.logs \
  --subnet-ids subnet-11111111 subnet-22222222 \
  --security-group-ids sg-0123456789abcdef0 \
  --private-dns-enabled
```

Private DNS is useful because it lets standard service names resolve to the endpoint inside the VPC. In practice, private DNS depends on VPC DNS support and DNS hostnames being enabled, and some services have additional prerequisites. If name resolution fails, check the VPC DNS settings before assuming the endpoint is broken.

**Gateway endpoints**

Gateway endpoints are the simplest private access option for supported services such as Amazon S3 and DynamoDB. They are not tunnel-based and not ENI-based. Instead, you add route table entries that direct traffic to the endpoint.

Operationally, gateway endpoints are useful when:

- You want private access to S3 or DynamoDB from instances in a VPC.
- You want to avoid NAT gateway charges for that traffic.
- You want a simple, scalable path without managing endpoint ENIs.

Example:

```bash
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0123456789abcdef0 \
  --vpc-endpoint-type Gateway \
  --service-name com.amazonaws.us-east-1.s3 \
  --route-table-ids rtb-0123456789abcdef0
```

Important differences:

- Gateway endpoints are controlled through route tables.
- They do not use security groups.
- They are not a general private network path.
- They are limited to supported services.

**Direct Connect as a decision factor**

Direct Connect is often part of the same design conversation because it changes the tradeoff between VPN and private connectivity. It provides a dedicated network connection from your environment to AWS and is commonly used with VPN for backup or encryption.

Use Direct Connect when:

- You need more predictable latency or bandwidth than internet-based VPN.
- You have sustained hybrid traffic.
- You want a private transport path and can justify the operational cost.

Direct Connect does not replace authorization design. You still need routing, segmentation, and service-level controls. Many architectures use Direct Connect plus Site-to-Site VPN as failover, or Direct Connect plus PrivateLink for private service access.

**When to choose each option**

Choose based on the access problem, not the technology name.

- **Site-to-Site VPN**
  - Best for network-to-network connectivity.
  - Good for hybrid access to many resources.
  - Common when you need encrypted connectivity over the internet.
- **Client VPN**
  - Best for remote users who need access to VPC resources.
  - Good when you need centralized authentication and managed client access.
  - Useful for temporary or distributed workforce access.
- **PrivateLink / interface endpoint**
  - Best for private access to one service.
  - Good when you want to avoid exposing a service publicly.
  - Not suitable for arbitrary network browsing.
- **Gateway endpoint**
  - Best for S3 and DynamoDB private routing.
  - Good for reducing NAT usage and simplifying access.
- **Direct Connect**
  - Best for predictable hybrid connectivity.
  - Good as a complement to VPN or PrivateLink.

**Operational comparison scenarios**

1. A developer laptop needs access to private EC2 instances, internal APIs, and RDS in a VPC.
   - Use **Client VPN**.
   - Add authorization rules for the required CIDRs.
   - Use split-tunnel if you only want VPC traffic to traverse the tunnel.

2. An on-premises data center needs access to multiple subnets in AWS.
   - Use **Site-to-Site VPN** or **Direct Connect**.
   - Prefer BGP if you need dynamic route exchange.
   - Use VPN as backup if Direct Connect is primary.

3. An application in one VPC needs to call Amazon S3 privately.
   - Use a **gateway endpoint**.
   - Update route tables for the relevant subnets.
   - No client tunnel is required.

4. A workload needs private access to an AWS service or SaaS endpoint without exposing a public IP.
   - Use an **interface endpoint** or endpoint service.
   - Restrict access with security groups and endpoint policies.
   - Enable private DNS if the service supports it and DNS is configured correctly.

5. A partner network needs access to a small set of internal services, but not the whole VPC.
   - Consider **PrivateLink** if you can expose only the service.
   - Use VPN only if the partner truly needs routed network access.

**Security and authorization model**

VPNs and PrivateLink solve different layers of the stack.

- **VPNs** primarily solve transport and routing.
- **Client VPN** adds user authentication and authorization rules.
- **PrivateLink** adds service-level exposure control.
- **IAM** and resource policies still matter for both service access and data-plane authorization.

Do not assume private connectivity equals trust.

- A private path reduces exposure to the public internet.
- It does not automatically grant access.
- You still need least privilege at the network, identity, and resource layers.
- Endpoint policies can restrict what the endpoint may access, but they do not replace application authorization.

**Troubleshooting checklist**

- Verify the tunnel is established before checking application access.
- Confirm routes exist on both sides of the connection.
- Check authorization rules for Client VPN destination CIDRs.
- Check subnet associations for Client VPN target networks.
- Verify security groups and network ACLs on the destination resources.
- Confirm DNS resolution for private names and split-horizon behavior.
- Check whether split-tunnel or full-tunnel matches the intended traffic flow.
- For interface endpoints, confirm private DNS, endpoint policy, and security group rules.
- For gateway endpoints, confirm the route table association and supported service.
- For Site-to-Site VPN, confirm tunnel status, BGP route exchange, and customer gateway configuration.

**Exam focus**

- Know the difference between **network access** and **service access**.
- Know that **Site-to-Site VPN** is for network-to-network connectivity.
- Know that **Client VPN** is for remote user access and uses authentication plus authorization rules.
- Know that **PrivateLink** is not a general VPN replacement.
- Know that **gateway endpoints** use route tables and do not use security groups.
- Know that **Direct Connect** is a common alternative or complement when hybrid connectivity needs more predictable performance.

**Quick memory aid**

- **VPN** = connect networks or users to networks.
- **PrivateLink** = connect to a service privately.
- **Gateway endpoint** = route to S3/DynamoDB privately.
- **Direct Connect** = dedicated private transport, often paired with VPN or PrivateLink.
