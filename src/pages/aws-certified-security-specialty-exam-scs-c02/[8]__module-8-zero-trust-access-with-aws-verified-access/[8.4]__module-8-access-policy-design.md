## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Access policy design overview**

Access policy design is the process of deciding *who* can do *what* to *which resource* under *which conditions*. In AWS, authorization is usually a combination of identity-based policies, resource-based policies, organizational guardrails, and service-specific controls. For exam scenarios, you need to reason about the final decision, not just identify a single policy document.

A secure design usually starts with least privilege, then adds explicit denies, conditions, and cross-account patterns only where needed. The most common mistakes are overusing `*`, confusing trust policies with permissions policies, and assuming one policy type overrides all others.

- **Identity-based policy**: attached to a user, group, or role; grants permissions to call APIs.
- **Resource-based policy**: attached to a resource; grants access to principals, often across accounts.
- **Trust policy**: attached to an IAM role; defines who can assume the role.
- **Permission boundary**: caps the maximum permissions an identity can receive.
- **SCP**: service control policy; sets the maximum permissions allowed in an AWS Organizations account or OU.
- **Session policy**: further restricts permissions for a temporary session.

**IAM policy types and where they fit**

IAM policies are not all operationally equivalent. Exam questions often hinge on whether a policy is attached to an identity, embedded inline, managed centrally, or attached to a resource.

- **Managed policies** are reusable and versioned separately from the identity.
  - AWS managed policies are maintained by AWS.
  - Customer managed policies are reusable across identities and easier to audit.
- **Inline policies** are embedded directly in one identity.
  - They are harder to reuse.
  - They are useful for tightly scoped, one-off permissions.
- **Resource policies** live on services such as S3, SQS, SNS, KMS, ECR, Secrets Manager, and EFS.
  - They are the main tool for direct cross-account access.
- **Trust policies** are not permission grants.
  - They only control who can call `sts:AssumeRole`.

A practical exam heuristic is:
- Use a **managed policy** when multiple identities need the same permissions.
- Use an **inline policy** when the permission is unique and tightly coupled to one identity.
- Use a **resource policy** when the resource owner should directly authorize external principals.
- Use a **role assumption pattern** when you want centralized control, temporary credentials, and easier revocation.

**How AWS evaluates authorization**

AWS authorization is usually a combination of multiple policy layers. The final decision depends on the service and the request context, but the general model is consistent:

1. AWS checks whether the request is explicitly denied anywhere applicable.
2. AWS checks whether the request is allowed by the relevant identity-based and/or resource-based policies.
3. AWS applies higher-level guardrails such as permission boundaries and SCPs.
4. AWS applies session policies and service-specific policy logic.
5. If no applicable allow remains, the request is denied by default.

The important nuance is that some policy types combine by **union** and others by **intersection**:

- **Identity policies + resource policies** often combine as a union for the request, meaning either can contribute an allow if the service supports it.
- **Permission boundaries** intersect with identity permissions, so the effective permissions cannot exceed the boundary.
- **SCPs** intersect with the account’s permissions, so the account cannot exceed the organization guardrail.
- **Session policies** intersect with the role’s permissions, so the session is narrower than the role.
- **KMS key policies** are special: they must allow use of the key, and IAM permissions alone are not enough unless the key policy delegates to IAM.

> An explicit `Deny` always wins over `Allow`, regardless of where it appears, unless the service does not evaluate that policy type for the request.

A useful mental model is:
- **Union** expands what is possible.
- **Intersection** narrows what is possible.
- **Explicit deny** blocks everything below it.

**Policy evaluation order and common exam traps**

The exact evaluation path varies by service, but the exam expects you to understand the interaction of policy layers.

- **Identity policies** grant permissions to the principal.
- **Resource policies** grant access to the resource.
- **Permission boundaries** limit the principal’s maximum permissions.
- **SCPs** limit the account or OU.
- **Session policies** limit the temporary session.
- **KMS key policies** and **grants** control cryptographic use separately from IAM.

Common traps:
- A role may have `s3:GetObject`, but an SCP can still block it.
- A user may be allowed by a bucket policy, but a permission boundary can still prevent the action if the principal is an IAM identity.
- A Lambda execution role may have permissions, but the service may also need a resource policy or service-linked permission depending on the integration.
- A KMS permission in IAM is not sufficient if the key policy does not allow the principal or delegate to IAM.

**Policy variables and attribute-based access control**

Policy variables let you write dynamic policies that adapt to the caller or request. This is especially useful for attribute-based access control (ABAC), where tags drive authorization.

Common variables and condition keys:
- `${aws:username}` for user-scoped paths or names.
- `${aws:PrincipalTag/Department}` for principal tags.
- `${aws:ResourceTag/Department}` for resource tags.
- `aws:RequestTag/Project` for tags supplied in the request.
- `aws:TagKeys` to restrict which tag keys can be set.

Example: allow users to manage only resources tagged with their department.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowDepartmentMatch",
      "Effect": "Allow",
      "Action": [
        "ec2:StartInstances",
        "ec2:StopInstances",
        "ec2:DescribeInstances"
      ],
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "aws:ResourceTag/Department": "${aws:PrincipalTag/Department}"
        }
      }
    }
  ]
}
```

Example: require request tags and restrict tag keys during resource creation.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCreateWithTags",
      "Effect": "Allow",
      "Action": "ec2:RunInstances",
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "aws:RequestTag/Environment": "prod"
        },
        "ForAllValues:StringEquals": {
          "aws:TagKeys": [
            "Environment",
            "Owner"
          ]
        }
      }
    }
  ]
}
```

ABAC is useful when:
- You have many teams or projects.
- You want to avoid per-resource policy sprawl.
- You need scalable least privilege across accounts.

**Condition operators and why they matter**

Condition operators are often the difference between a secure policy and a brittle one. Choose operators carefully to avoid accidental denial or over-permission.

- **StringEquals**: exact match; use when the value must be precise.
- **StringLike**: wildcard matching; use carefully for patterns.
- **StringEqualsIfExists**: only evaluates if the key is present; useful for optional context.
- **ArnLike**: pattern match for ARNs; useful for service principals or resource patterns.
- **IpAddress**: restricts by source IP.
- **BoolIfExists**: useful when a boolean context key may not always be present.
- **Null**: checks whether a key exists; often used to require tags or MFA.
- **ForAllValues** / **ForAnyValue**: useful for multi-value keys such as `aws:TagKeys`.

Examples:
- Use `StringEqualsIfExists` when a service may not always send a context key, but you do not want the policy to fail solely because the key is absent.
- Use `ArnLike` when matching a family of ARNs, such as all roles in an account.
- Use `IpAddress` for network-based guardrails, but do not treat it as a substitute for identity controls.
- Use `BoolIfExists` for conditions like MFA-related checks where the key may not be present in all request paths.

Example: require MFA for sensitive actions.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyWithoutMFA",
      "Effect": "Deny",
      "Action": [
        "iam:DeleteUser",
        "iam:PutUserPolicy"
      ],
      "Resource": "*",
      "Condition": {
        "BoolIfExists": {
          "aws:MultiFactorAuthPresent": "false"
        }
      }
    }
  ]
}
```

**Cross-account access patterns**

Cross-account access usually follows one of two patterns:

- **Assume a role in the target account**
- **Access the resource directly through a resource policy**

Choose based on control, auditability, and operational simplicity.

**Role assumption pattern**

This is the most common and usually the safest pattern for human or workload access across accounts.

Flow:
1. A principal in Account A calls `sts:AssumeRole` on a role in Account B.
2. The role trust policy in Account B allows the principal from Account A.
3. The role permissions policy in Account B grants access to the target resources.
4. Temporary credentials are issued and used in Account B.

Example trust policy allowing a role from another account to assume the role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/ExternalAppRole"
      },
      "Action": "sts:AssumeRole",
      "Condition": {
        "StringEquals": {
          "sts:ExternalId": "example-external-id"
        }
      }
    }
  ]
}
```

Use this pattern when:
- You want centralized control in the target account.
- You need temporary credentials.
- You want easy revocation by changing the trust policy.
- You need to avoid broad resource policies on sensitive assets.

**Direct resource policy pattern**

This pattern is common for services that support resource policies, such as S3, SQS, SNS, KMS, Secrets Manager, ECR, and EFS.

Use it when:
- The resource owner wants to grant access without role assumption.
- The service supports direct cross-account authorization.
- You need service-to-service access or simple sharing.

Example S3 bucket policy allowing a role from another account to read objects:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCrossAccountRead",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/AnalyticsRole"
      },
      "Action": [
        "s3:GetObject"
      ],
      "Resource": "arn:aws:s3:::example-bucket/*"
    }
  ]
}
```

Exam pitfall:
- A resource policy does not automatically grant the caller permission in its own account if the service still requires identity permissions.
- Some services require both the caller’s identity policy and the resource policy to align.

**Service examples that commonly appear in access design questions**

**S3**

S3 supports bucket policies, access point policies, ACLs, and object ownership controls. For exam purposes, bucket policies and object ownership are the most important.

- Use bucket policies for cross-account access and explicit deny guardrails.
- Use access points when you need separate policy boundaries for different applications.
- Use ACLs only when required by legacy patterns.
- Use Object Ownership with `BucketOwnerEnforced` to disable ACLs and make the bucket owner own all objects.

Example bucket policy denying non-TLS requests:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyInsecureTransport",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:*",
      "Resource": [
        "arn:aws:s3:::example-bucket",
        "arn:aws:s3:::example-bucket/*"
      ],
      "Condition": {
        "Bool": {
          "aws:SecureTransport": "false"
        }
      }
    }
  ]
}
```

**SQS**

SQS uses queue policies for cross-account access and service integrations.

- Grant `sqs:SendMessage` to a producer principal or service.
- Grant `sqs:ReceiveMessage` and `sqs:DeleteMessage` to consumers.
- Use conditions to restrict source ARN or source account for service integrations.

Example queue policy allowing SNS to publish to a queue:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowSNSPublish",
      "Effect": "Allow",
      "Principal": {
        "Service": "sns.amazonaws.com"
      },
      "Action": "sqs:SendMessage",
      "Resource": "arn:aws:sqs:us-east-1:123456789012:example-queue",
      "Condition": {
        "ArnEquals": {
          "aws:SourceArn": "arn:aws:sns:us-east-1:123456789012:example-topic"
        }
      }
    }
  ]
}
```

**SNS**

SNS topic policies control who can publish, subscribe, or manage the topic. For subscriptions, the principal that calls `Subscribe` is typically the IAM principal, not the endpoint itself.

- Use topic policies to allow cross-account publish or subscribe.
- Use `aws:SourceArn` and `aws:SourceAccount` to reduce confused deputy risk.
- For Lambda subscriptions, the Lambda service integration and permissions must align with the topic policy and function permission.

**Lambda**

Lambda uses resource-based policies for invocation permissions.

- `lambda:AddPermission` adds a statement to the function policy.
- Use it to allow services like S3, SNS, EventBridge, or cross-account principals to invoke the function.
- The execution role is separate; it controls what the function can do after invocation.

Example allowing SNS to invoke a function:

```bash
aws lambda add-permission \
  --function-name process-topic-event \
  --statement-id AllowSNSInvoke \
  --action lambda:InvokeFunction \
  --principal sns.amazonaws.com \
  --source-arn arn:aws:sns:us-east-1:123456789012:example-topic
```

**ECR**

Amazon Elastic Container Registry (ECR) uses repository policies for cross-account image pull or push.

- Grant `ecr:BatchGetImage` and `ecr:GetDownloadUrlForLayer` for pulls.
- Grant `ecr:PutImage` and upload actions for pushes.
- Use repository policies for shared build or deployment accounts.

**Secrets Manager**

Secrets Manager supports resource policies for cross-account secret access.

- Use resource policies to share secrets with a role in another account.
- Combine with KMS permissions if the secret uses a customer managed key.
- Be careful: access to the secret and access to the KMS key are separate checks.

**EFS**

Amazon Elastic File System (EFS) uses file system policies and POSIX permissions.

- Use file system policies for IAM-based authorization.
- Use access points to simplify application-specific paths and POSIX identities.
- Remember that network controls and mount targets also matter.

**KMS authorization nuances**

AWS Key Management Service (KMS) is a frequent exam topic because authorization is layered and easy to misunderstand.

KMS access requires:
- An IAM policy that allows the action, such as `kms:Encrypt` or `kms:Decrypt`.
- A key policy that allows the principal or delegates to IAM.
- In some cases, a grant for temporary or service-specific access.

Important points:
- The key policy is authoritative for the key.
- IAM permissions alone are not enough unless the key policy allows IAM delegation.
- Cross-account KMS access usually needs explicit key policy support or a grant.
- Many AWS services use KMS on your behalf and may require service-linked permissions or grants.

A common delegation pattern is to allow the account root principal in the key policy so IAM policies in that account can be evaluated. This does **not** automatically grant every IAM principal access by itself; the IAM principal still needs an IAM allow, and cross-account access still needs explicit key policy or grant support.

Example key policy statement that enables IAM policy evaluation within the account:

```json
{
  "Sid": "EnableIAMPermissions",
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::123456789012:root"
  },
  "Action": "kms:*",
  "Resource": "*"
}
```

Use grants when:
- A service needs temporary access to a key.
- You want to avoid editing the key policy for every short-lived use case.
- You need fine-grained, revocable access for a specific principal or AWS service.

Example grant creation:

```bash
aws kms create-grant \
  --key-id arn:aws:kms:us-east-1:123456789012:key/abcd-1234 \
  --grantee-principal arn:aws:iam::123456789012:role/AppRole \
  --operations Encrypt Decrypt GenerateDataKey
```

Cross-account KMS pattern:
- The external account principal must be allowed in the key policy or via a grant.
- The caller also needs IAM permission in its own account.
- If the encrypted resource is in another service, that service may also need permission to use the key.

**ACLs, Object Ownership, and when ACLs still matter**

ACLs are legacy in many designs, but they still appear in exam questions.

- S3 ACLs can grant object or bucket-level access.
- ACLs are less expressive than policies.
- ACLs are often replaced by bucket policies and Object Ownership.

With `BucketOwnerEnforced`:
- ACLs are disabled for the bucket.
- The bucket owner owns all objects.
- This simplifies cross-account uploads and reduces ownership confusion.

Use ACLs only when:
- You are dealing with legacy integrations.
- A service or workflow still depends on them.
- The exam scenario explicitly mentions ACL-based sharing.

**Guardrail patterns for least privilege**

Deny-by-default guardrails are a core security design pattern.

Common guardrails:
- Deny non-TLS access using `aws:SecureTransport`.
- Deny actions outside approved regions using `aws:RequestedRegion`.
- Deny missing tags on create using `aws:RequestTag` and `aws:TagKeys`.
- Deny public access in S3 using bucket policies and account-level public access blocks.
- Deny privilege escalation paths in IAM by restricting `iam:PassRole`, `iam:AttachRolePolicy`, and `iam:PutRolePolicy`.

Example SCP-style region guardrail:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyOutsideApprovedRegions",
      "Effect": "Deny",
      "Action": "*",
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": [
            "us-east-1",
            "us-west-2"
          ]
        }
      }
    }
  ]
}
```

Example deny for missing required tags:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyCreateWithoutOwnerTag",
      "Effect": "Deny",
      "Action": [
        "ec2:RunInstances",
        "s3:CreateBucket"
      ],
      "Resource": "*",
      "Condition": {
        "Null": {
          "aws:RequestTag/Owner": "true"
        }
      }
    }
  ]
}
```

**Policy validation and troubleshooting tools**

You should validate policies before deployment and troubleshoot with evidence, not guesswork.

Useful tools:
- **IAM Access Analyzer** to identify unintended public or cross-account access.
- **IAM policy simulator** to test identity policy decisions.
- **Access Advisor** to see which services a principal has used.
- **CloudTrail** to inspect the exact API call, principal, and error.
- **AWS CLI `simulate-principal-policy`** for repeatable checks.

Example CLI simulation:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/AppRole \
  --action-names s3:GetObject kms:Decrypt \
  --resource-arns arn:aws:s3:::example-bucket/private/file.txt \
  --context-entries ContextKeyName=aws:PrincipalTag/Department,ContextKeyType=string,ContextKeyValues=Finance
```

Practical troubleshooting flow:
1. Confirm the principal, action, and resource from CloudTrail.
2. Check identity policies attached to the principal.
3. Check resource policies on the target service.
4. Check permission boundaries, session policies, and SCPs.
5. For KMS, check the key policy and any grants.
6. Validate conditions such as tags, source IP, region, or MFA.

> The simulator is useful, but it does not replace service-specific testing. Always verify the full request path, especially for KMS, SCPs, and service integrations.

**Design patterns you should recognize on the exam**

- **Centralized access via roles**
  - Best for human access and many workload integrations.
  - Easier to revoke and audit.
- **Direct resource sharing**
  - Best for simple cross-account service access.
  - Common for S3, SQS, SNS, ECR, Secrets Manager, and KMS.
- **ABAC at scale**
  - Best when tags can represent ownership, environment, or department.
  - Reduces policy sprawl.
- **Explicit deny guardrails**
  - Best for organization-wide controls.
  - Use SCPs and resource policies to enforce boundaries.
- **Temporary cryptographic access**
  - Best for KMS grants and service integrations.
  - Avoids overbroad key policy changes.

**Quick comparison of policy layers**

| Layer | Scope | Typical use | Key exam point |
|---|---|---|---|
| Identity policy | Principal | Grant API permissions | Can be inline or managed |
| Resource policy | Resource | Cross-account or service access | Can directly name principals |
| Trust policy | Role assumption | Who can assume a role | Not a permission grant |
| Permission boundary | Principal max | Limit delegated admin | Intersects with identity permissions |
| SCP | Account/OU max | Organization guardrail | Does not grant permissions |
| Session policy | Temporary session | Narrow assumed-role access | Intersects with role permissions |
| KMS key policy | Key | Cryptographic authorization | IAM alone is not enough |

**CLI and policy snippets to remember**

List attached managed policies:
```bash
aws iam list-attached-role-policies --role-name AppRole
```

Inspect inline policies:
```bash
aws iam list-role-policies --role-name AppRole
aws iam get-role-policy --role-name AppRole --policy-name InlineAccess
```

Check a bucket policy:
```bash
aws s3api get-bucket-policy --bucket example-bucket
```

Check a KMS key policy:
```bash
aws kms get-key-policy \
  --key-id arn:aws:kms:us-east-1:123456789012:key/abcd-1234 \
  --policy-name default
```

**Exam-focused takeaways**

- Start with the narrowest policy type that solves the problem.
- Use role assumption for controlled cross-account access.
- Use resource policies for direct sharing when the service supports it.
- Remember that explicit deny overrides allow.
- Treat KMS as a special case: key policy, IAM policy, and sometimes grants all matter.
- Use tags and conditions for scalable ABAC.
- Prefer `BucketOwnerEnforced` over ACL-based S3 designs when possible.
- Validate with Access Analyzer, CloudTrail, and the IAM policy simulator.
