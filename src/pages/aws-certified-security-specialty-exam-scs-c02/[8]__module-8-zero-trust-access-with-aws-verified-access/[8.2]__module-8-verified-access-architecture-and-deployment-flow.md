## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Verified Access overview**

AWS Verified Access provides application access based on identity and device posture without requiring a traditional VPN. It is designed for private web applications and other supported application endpoints that you want to expose through policy-based access control.

The service evaluates access requests against trust context before allowing traffic to the target application. In practice, you use it when you want to reduce network-level access and move toward application-centric, zero-trust access decisions.

- **Identity trust** comes from a supported identity source, typically IAM Identity Center.
- **Device trust** comes from a device trust provider that supplies posture or compliance signals.
- **Policy evaluation** decides whether the request is allowed, denied, or conditionally permitted.

> Verified Access is not a generic network tunnel. It is an application access service with policy enforcement at the edge of the access path.

**Core architecture**

A typical deployment includes these logical parts:

- **Verified Access instance**: the top-level container for the service in a Region.
- **Trust provider**: supplies identity trust, device trust, or both depending on the integration.
- **Verified Access group**: a policy boundary that groups endpoints with similar access rules.
- **Verified Access endpoint**: the externally reachable entry point for a specific application target.
- **Application target**: usually an internal Application Load Balancer (ALB) or other supported private web application endpoint.

A simplified flow looks like this:

```text
User -> Verified Access endpoint -> policy evaluation -> application target
                 ^                     ^
                 |                     |
          identity trust         device trust
          provider context       provider context
```

The important exam idea is that Verified Access does not replace the application. It brokers access to the application based on trust signals and policy.

**Identity and device trust**

Verified Access distinguishes between identity trust and device trust.

- **Identity trust provider**
  - Supplies user identity context.
  - Commonly integrates with IAM Identity Center.
  - Lets you evaluate who the user is and, in some cases, attributes associated with that identity.
- **Device trust provider**
  - Supplies device posture context.
  - Used to evaluate whether the device meets your security requirements.
  - Common examples include managed-device or endpoint-security integrations that report compliance signals.

The service combines these signals during authorization. A user may be authenticated, but still denied if the device does not satisfy posture requirements.

**Policy model and condition keys**

Verified Access policies are evaluated against context keys, not arbitrary JSON fields. For exam prep, focus on the idea that policies use AWS-style condition logic with supported claims and context attributes.

A policy commonly references:

- **User identity attributes**
  - User name or subject-like identity claims
  - Group membership or directory attributes, when available through the identity integration
- **Device attributes**
  - Managed status
  - Compliance or posture state
  - Device ownership or trust state, depending on the integration
- **Request attributes**
  - Application-specific or endpoint-related context that the service exposes

A realistic policy shape is closer to AWS condition syntax than to custom pseudo-JSON. For example:

```json
{
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "verifiedaccess:CreateClientVpnEndpoint",
      "Resource": "*"
    }
  ]
}
```

That example is only to show AWS policy structure style, not a real Verified Access authorization policy. A Verified Access policy is evaluated with service-specific condition keys and statements, so you should recognize the pattern rather than memorize invented fields.

A more relevant conceptual example is:

```json
{
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "verifiedaccess:access",
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "identity.user": "alice@example.com"
        },
        "Bool": {
          "device.isManaged": true,
          "device.isCompliant": true
        }
      }
    }
  ]
}
```

This is still illustrative, not exact AWS syntax. The exam-relevant point is that access decisions are based on identity and device context, and the policy language follows AWS condition-based evaluation rather than custom application logic.

**Deployment flow**

The deployment sequence matters because resources have dependencies.

1. Create the **Verified Access instance** first.
   - This is the regional container for the service.
   - It is the anchor for trust providers, groups, and endpoints.
2. Configure the **trust provider**.
   - Add the identity trust provider and, if needed, the device trust provider.
   - The exact integration depends on your identity and endpoint management stack.
3. Create a **Verified Access group**.
   - Use the group to hold endpoints that share a policy.
   - Attach the policy at the group or endpoint level depending on your design.
4. Create the **Verified Access endpoint**.
   - Point it at the supported application target.
   - Associate it with the instance and the relevant group.
5. Attach or reference the **trust context** required by the policy.
   - Identity and device signals must be available before the policy can evaluate as intended.
6. Test and validate access.
   - Confirm that allowed users and compliant devices can reach the app.
   - Confirm that noncompliant devices or unauthorized identities are denied.

A practical mental model is:

```text
Instance
  ├─ Trust provider(s)
  ├─ Group
  │   └─ Policy
  └─ Endpoint
      └─ Application target
```

**Supported target patterns**

Verified Access is primarily used for private web applications, especially internal applications behind an ALB. That is the most common exam scenario.

Typical use cases include:

- Internal HR or finance portals
- Contractor portals with device compliance requirements
- Admin consoles that should not be reachable from unmanaged devices
- Private SaaS-like web apps hosted in a VPC

Be careful with scope:

- It is not a general-purpose replacement for all VPN use cases.
- It is not the best fit for arbitrary non-web protocols.
- It is best understood as application access with policy enforcement, not broad subnet access.

**CLI inspection and validation**

Use the AWS CLI to inspect the resources and confirm relationships. The exact command names and output fields can vary by API version, but the operational pattern is consistent: describe the instance, groups, endpoints, and trust providers, then verify associations.

```bash
aws ec2 describe-verified-access-instances
aws ec2 describe-verified-access-groups
aws ec2 describe-verified-access-endpoints
aws ec2 describe-verified-access-trust-providers
```

Useful validation checks:

- Confirm the endpoint is associated with the correct instance.
- Confirm the group policy is attached where you expect it.
- Confirm the trust provider ARN or ID matches the intended identity or device source.
- Confirm the endpoint target points to the correct internal application.

If you need to inspect a specific resource, use filters and IDs:

```bash
aws ec2 describe-verified-access-endpoints \
  --verified-access-endpoint-ids vae-0123456789abcdef0
```

**Example deployment topology**

A common architecture is an internal ALB in private subnets, with Verified Access in front of it.

```text
Internet user
   |
   v
Verified Access endpoint
   |
   v
Policy evaluation
   |---- identity trust provider
   |---- device trust provider
   |
   v
Internal ALB
   |
   v
Private web app in VPC
```

This design keeps the application private while still allowing controlled external access. The ALB remains internal, and Verified Access becomes the controlled entry point.

**Practical configuration example**

A simplified CloudFormation-style sketch helps you remember the resource relationships:

```yaml
Resources:
  VerifiedAccessInstance:
    Type: AWS::EC2::VerifiedAccessInstance
    Properties: {}

  VerifiedAccessGroup:
    Type: AWS::EC2::VerifiedAccessGroup
    Properties:
      VerifiedAccessInstanceId: !Ref VerifiedAccessInstance
      PolicyDocument: |
        {
          "Statement": [
            {
              "Effect": "Allow",
              "Action": "verifiedaccess:access",
              "Resource": "*"
            }
          ]
        }

  VerifiedAccessEndpoint:
    Type: AWS::EC2::VerifiedAccessEndpoint
    Properties:
      VerifiedAccessInstanceId: !Ref VerifiedAccessInstance
      VerifiedAccessGroupId: !Ref VerifiedAccessGroup
      ApplicationDomain: app.internal.example.com
```

This is a structural example only. In real deployments, you must use the exact properties supported by the current AWS resource type and the target integration you choose.

**Observability and troubleshooting**

You should verify access decisions with logs and service metadata, not only by testing in a browser.

Key operational checks:

- **CloudTrail**
  - Confirms control-plane actions such as create, update, and delete operations.
  - Useful for auditing who changed policies or endpoints.
- **CloudWatch metrics and logs**
  - Help you observe request volume, failures, and policy-related behavior where supported.
- **Application logs**
  - Confirm whether requests reached the target application.
  - Help distinguish policy denial from application-side errors.
- **Endpoint and group descriptions**
  - Confirm the resource state and associations.

Troubleshooting approach:

1. Check whether the user is authenticated through the expected identity source.
2. Check whether the device trust signal is present and compliant.
3. Check the policy conditions for mismatched claims or attributes.
4. Check whether the endpoint is associated with the correct group and target.
5. Check whether the application target is healthy and reachable from the VPC side.

> If the request is denied before reaching the app, focus on identity, device posture, and policy. If the request reaches the app but fails there, focus on the target application and its own logs.

**Common exam distinctions**

Verified Access is often confused with other access patterns.

| Capability | Verified Access | VPN | ALB authentication |
|---|---|---|---|
| Application-centric access | Yes | No | Yes |
| Device posture evaluation | Yes | Sometimes via separate tooling | No |
| Broad network access | No | Yes | No |
| Private web app access | Yes | Yes, indirectly | Yes |
| Zero-trust style policy evaluation | Yes | Limited | Limited |

The key distinction is that Verified Access evaluates identity and device trust before granting access to the application. A VPN gives network reachability, while ALB authentication handles user sign-in but does not provide the same device-trust model.

**Second scenario: contractor access with device denial**

A useful real-world pattern is contractor access to an internal portal.

- Contractors authenticate with the approved identity source.
- Their laptops must be managed and compliant.
- The policy allows access only when both identity and device checks pass.

Example outcome:

- Contractor on a managed laptop: allowed.
- Contractor on a personal laptop: denied.
- Employee on a compliant device: allowed.
- Employee with expired or noncompliant endpoint posture: denied.

This scenario is common in security architecture questions because it demonstrates why Verified Access exists: you want to reduce implicit trust even when the user is valid.

**What to remember for the exam**

- **Verified Access** is for application access, not broad network access.
- **Identity trust** and **device trust** are separate concepts.
- **IAM Identity Center** is the common identity integration pattern.
- **Policies** use AWS-style condition logic and service-specific context keys.
- **Internal ALB-backed web apps** are the most common target pattern.
- **Resource relationships** matter: instance, trust provider, group, endpoint, and target must line up correctly.
- **Logging and CloudTrail** help you validate both configuration and access decisions.
