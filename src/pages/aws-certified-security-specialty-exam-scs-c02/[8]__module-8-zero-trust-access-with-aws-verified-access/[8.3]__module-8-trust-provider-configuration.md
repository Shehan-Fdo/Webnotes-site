## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Trust provider configuration overview**

Trust provider configuration is the set of AWS-side and identity-provider-side settings that let AWS accept identities from an external source and turn them into AWS permissions. In practice, you configure one of three common patterns:

- **IAM Identity Center** for workforce access and centralized account access management.
- **SAML 2.0 federation** for browser-based enterprise single sign-on.
- **OIDC** federation for modern web and workload identity, including Kubernetes and CI/CD systems.

At a high level, the flow is:

1. A user or workload authenticates to an identity provider.
2. The provider issues a SAML assertion or OIDC token.
3. AWS validates the trust configuration and token/assertion.
4. AWS STS returns temporary credentials or a console sign-in session.
5. IAM policies and role trust policies determine what the identity can do.

```text
User/Workload -> IdP authentication -> SAML assertion or OIDC token

```

**Core AWS concepts you must separate**

- **Trust policy**: attached to an IAM role; defines who can assume the role.
- **Permissions policy**: attached to the role or user; defines what actions are allowed after assumption.
- **Identity provider object**: the AWS resource that represents the external IdP.
- **Session**: the temporary identity created after successful federation.
- **Permission set**: in IAM Identity Center, a template that provisions IAM roles in target accounts; it is not a permissions boundary.

A common exam trap is mixing up trust and permissions. The trust policy answers, “Who may enter this role?” The permissions policy answers, “What can they do once inside?”

**IAM Identity Center architecture**

IAM Identity Center is AWS’s recommended workforce access service for managing access to multiple AWS accounts and some business applications from a central portal. It is not the same thing as classic IAM federation, although both can use external identity sources.

**Key terms**

- **Identity source**: where users and groups come from.
  - Built-in directory managed by IAM Identity Center.
  - External identity provider such as Microsoft Entra ID, Okta, or another SAML/OIDC-capable directory.
- **Instance**: the IAM Identity Center deployment in a region.
- **Portal**: the user-facing access portal where users launch assigned AWS accounts and applications.
- **Permission set**: a reusable template that defines session duration, managed policies, inline policy, permissions boundary, and optional tags for the generated role.
- **Assigned account access**: the mapping between a user/group and an AWS account plus permission set.

IAM Identity Center differs from direct IAM federation in a few important ways:

- It centralizes access across many accounts.
- It provisions IAM roles into target accounts automatically.
- It gives users a portal experience instead of requiring each account to manage its own federation setup.
- It is best for workforce access, not for application-to-application workload federation.

**Identity source options**

- **Built-in directory**
  - Useful for small environments or when you want AWS-managed identities.
  - Users and groups are created in IAM Identity Center.
- **External identity provider**
  - Useful when your organization already has a corporate directory.
  - You typically sync users and groups from the external source.
  - Authentication happens at the external IdP, while authorization is managed in IAM Identity Center.

**Permission set behavior**

Permission sets are often tested because they look like IAM policies but behave differently.

- A permission set is a template.
- AWS uses it to create an IAM role in each assigned account.
- The generated role name is account-specific and derived from the permission set and assignment.
- The permission set can include:
  - AWS managed policies
  - Customer managed policies
  - Inline policy
  - Session duration
  - Permissions boundary
  - Tags
- The permission set does not directly grant access until it is assigned to a user or group for a target account.

**Session duration and relay state**

- **Session duration** controls how long the IAM Identity Center session lasts.
- The actual usable duration may be constrained by the target role session settings and the federation path.
- **Relay state** is the destination or application target after sign-in.
- In practice, relay state helps send the user to the right AWS account, application, or landing page after authentication.

**When to choose IAM Identity Center**

Use IAM Identity Center when:

- You manage many AWS accounts.
- You want centralized workforce access.
- You need a portal for users.
- You want to reduce per-account role sprawl.
- You need easier assignment of groups to accounts and applications.

Do not use it as the first choice for:

- Machine-to-machine access.
- External customer identity.
- Fine-grained application federation where the application itself is the relying party.

**SAML federation**

SAML 2.0 (Security Assertion Markup Language) federation is a classic enterprise sign-in pattern for browser-based access to AWS. The identity provider authenticates the user and sends a signed assertion to AWS, which then exchanges it for an AWS console session or temporary credentials.

**Required AWS-side objects**

To configure SAML federation, you typically need:

- A **SAML identity provider** object in IAM.
- An IAM role with a trust policy that trusts the SAML provider.
- A permissions policy attached to the role.
- IdP metadata imported into AWS.
- AWS metadata or sign-in details configured in the IdP.

**Typical setup steps**

1. Export metadata from the IdP.
2. Create the SAML provider in IAM using the metadata document.
3. Create or update an IAM role with a trust policy for `sts:AssumeRoleWithSAML`.
4. Attach permissions to the role.
5. Configure the IdP with AWS metadata, ACS endpoint, and role mapping.
6. Test sign-in and verify the role appears in the AWS console or federation flow.

**Example IAM SAML provider creation**

```bash
aws iam create-saml-provider \
  --saml-metadata-document file://idp-metadata.xml \
  --name CorpSAML
```

**Example trust policy for SAML**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Federated": "arn:aws:iam::123456789012:saml-provider/CorpSAML"
      },
      "Action": "sts:AssumeRoleWithSAML",
      "Condition": {
        "StringEquals": {
          "SAML:aud": "https://signin.aws.amazon.com/saml"
        }
      }
    }
  ]
}
```

**How the SAML flow works**

- The user signs in to the IdP.
- The IdP issues a signed SAML assertion containing the user identity and role mapping.
- The browser posts the assertion to the AWS sign-in endpoint for console federation, or a client calls `AssumeRoleWithSAML` directly for API access.
- AWS validates the signature, audience, and trust policy.
- AWS returns temporary credentials or a console session.

**Important SAML details**

- The assertion must be signed by the IdP certificate trusted by AWS.
- The `Audience` or `SAML:aud` value must match the AWS sign-in endpoint.
- The role mapping in the assertion must reference the correct IAM role and SAML provider ARN.
- The role trust policy must allow the SAML provider and the correct STS action.
- The role permissions policy still determines what the user can do after sign-in.

**SAML role mapping**

A SAML assertion commonly includes role attributes that map the user to one or more AWS roles. The IdP often presents a role chooser if multiple roles are available.

Example conceptual role attribute pair:

```text
arn:aws:iam::123456789012:role/SAMLAdmin,arn:aws:iam::123456789012:saml-provider/CorpSAML
```

**SAML troubleshooting**

- If sign-in fails, check:
  - Certificate validity and signing algorithm support.
  - Metadata freshness.
  - Role ARN and provider ARN formatting.
  - Audience mismatch.
  - Clock skew between systems.
- If the IdP rotates certificates, update the AWS SAML provider metadata.
- If AWS metadata changes, update the IdP configuration.

> A common exam scenario is a working IdP but a broken AWS role trust policy. Remember that both the IdP assertion and the IAM role trust policy must align.

**OIDC federation**

OpenID Connect (OIDC) is an identity layer on top of OAuth 2.0. In AWS, OIDC is commonly used for workload identity and modern application federation, especially when you want short-lived credentials without storing long-term AWS keys.

**Common OIDC use cases**

- Kubernetes workloads using IAM Roles for Service Accounts (IRSA).
- CI/CD systems such as GitHub Actions or other build pipelines.
- Custom web applications that need to exchange tokens for AWS access.
- Microservices that assume AWS roles based on workload identity.

**AWS OIDC provider configuration**

An IAM OIDC provider stores the external issuer and the validation material AWS needs to trust tokens from that issuer.

Typical configuration fields include:

- **Provider URL**: the issuer URL.
- **Client ID list**: allowed audiences.
- **Thumbprint list**: certificate thumbprints used to validate the TLS chain for the provider endpoint.
- **ARN**: the IAM resource identifier for the provider.

Example creation:

```bash
aws iam create-open-id-connect-provider \
  --url https://oidc.eks.us-east-1.amazonaws.com/id/EXAMPLED539D4633E53DE1B71EXAMPLE \
  --client-id-list sts.amazonaws.com \
  --thumbprint-list 9e99a48a9960b14926bb7f3b02e22da0afd10b8f
```

**OIDC validation behavior**

AWS validates OIDC tokens by checking:

- The token signature against the provider’s JSON Web Key Set (JWKS).
- The issuer claim against the configured provider URL.
- The audience claim against the configured client IDs.
- The token expiration and not-before timing.
- The trust policy conditions on the role.

**JWKS and key rotation**

- OIDC providers publish signing keys through JWKS.
- AWS uses the provider’s keys to validate token signatures.
- If the provider rotates keys, AWS must be able to retrieve the updated JWKS.
- If the provider changes TLS certificates, the thumbprint may need updating depending on the provider configuration and certificate chain.

**OIDC trust policy example**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Federated": "arn:aws:iam::123456789012:oidc-provider/oidc.eks.us-east-1.amazonaws.com/id/EXAMPLED539D4633E53DE1B71EXAMPLE"
      },
      "Action": "sts:AssumeRoleWithWebIdentity",
      "Condition": {
        "StringEquals": {
          "oidc.eks.us-east-1.amazonaws.com/id/EXAMPLED539D4633E53DE1B71EXAMPLE:aud": "sts.amazonaws.com",
          "oidc.eks.us-east-1.amazonaws.com/id/EXAMPLED539D4633E53DE1B71EXAMPLE:sub": "system:serviceaccount:default:myapp"
        }
      }
    }
  ]
}
```

**OIDC condition keys**

Common condition keys include:

- `sub`: subject, often the workload or user identity.
- `aud`: audience, often `sts.amazonaws.com` for AWS federation.
- `iss`: issuer, which must match the provider identity.
- Provider-specific claim namespaces: claims are usually prefixed with the issuer host/path in IAM condition keys.

For Kubernetes IRSA, the `sub` claim is especially important because it identifies the service account.

**IRSA example**

IAM Roles for Service Accounts lets a Kubernetes pod assume an IAM role without node-level credentials.

Typical flow:

1. The pod gets a projected service account token.
2. The token is signed by the cluster’s OIDC issuer.
3. The pod calls STS `AssumeRoleWithWebIdentity`.
4. AWS validates the token and returns temporary credentials.

Example trust policy for IRSA:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Federated": "arn:aws:iam::123456789012:oidc-provider/oidc.eks.us-east-1.amazonaws.com/id/EXAMPLE"
      },
      "Action": "sts:AssumeRoleWithWebIdentity",
      "Condition": {
        "StringEquals": {
          "oidc.eks.us-east-1.amazonaws.com/id/EXAMPLE:sub": "system:serviceaccount:payments:api",
          "oidc.eks.us-east-1.amazonaws.com/id/EXAMPLE:aud": "sts.amazonaws.com"
        }
      }
    }
  ]
}
```

**IRSA Kubernetes service account annotation**

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: api
  namespace: payments
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::123456789012:role/PaymentsApiRole
```

**OIDC troubleshooting**

- Check that the issuer URL exactly matches the provider URL in IAM.
- Verify the audience claim matches the client ID list.
- Confirm the token is not expired.
- Confirm the trust policy uses the correct provider-specific claim namespace.
- Refresh the provider configuration if the TLS certificate chain changes.
- Verify the JWKS endpoint is reachable and current.

> For exam questions, remember that OIDC trust is not just “token present equals access.” AWS validates the issuer, audience, signature, and role trust policy together.

**STS actions and when to use them**

AWS Security Token Service (STS) is the service that issues temporary credentials. The action you choose depends on the federation type.

| STS action | Typical use | Identity type |
|---|---|---|
| `AssumeRole` | Cross-account or same-account role assumption | IAM user, IAM role, federated principal |
| `AssumeRoleWithSAML` | SAML federation | SAML principal |
| `AssumeRoleWithWebIdentity` | OIDC federation | OIDC token bearer |
| `GetFederationToken` | Temporary credentials for IAM users in some legacy patterns | IAM user |
| `GetSessionToken` | Temporary session for IAM users or MFA workflows | IAM user |

**Role trust policy vs permissions policy**

Both policies must be correct for federation to work.

- **Trust policy**
  - Defines the external principal that can assume the role.
  - Uses `Principal` and STS actions.
  - Often includes conditions on audience, subject, or SAML audience.
- **Permissions policy**
  - Defines what the assumed role can do.
  - Can be managed, inline, or attached through permission sets.
  - Does not control who may assume the role.

Example of a role that trusts a federated principal but still needs permissions:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:ListBucket",
      "Resource": "arn:aws:s3:::example-bucket"
    }
  ]
}
```

**Cross-account access patterns**

Cross-account access often uses `AssumeRole` rather than SAML or OIDC directly.

**Typical pattern**

- Source principal authenticates in the source account or via federation.
- Source principal has an identity-based policy allowing `sts:AssumeRole`.
- Target role trust policy allows the source principal, account, or federated role.
- STS returns temporary credentials for the target role.

**Important nuance**

- Trusting an entire account root in the target role does not automatically grant access.
- The source principal still needs permission to call `sts:AssumeRole`.
- If the source principal is itself an assumed role, role chaining rules and session limits apply.

**Role chaining considerations**

- A role session can assume another role if allowed by trust and identity policies.
- Session duration may be reduced in chained scenarios.
- Some session attributes may not propagate unless explicitly supported.
- This matters when a federated user assumes a role and then assumes a second role in another account.

**Session tags and attribute-based access control**

Session tags let you pass identity attributes into the AWS session and use them in authorization decisions. This supports attribute-based access control (ABAC).

**Why session tags matter**

- They reduce the need for many static roles.
- They let you authorize based on user department, project, environment, or cost center.
- They are useful in both SAML and OIDC federation when supported by the IdP and trust policy.

**SAML principal tags**

SAML assertions can carry attributes that AWS maps to session tags or principal tags.

Example conceptual attributes:

```text
https://aws.amazon.com/SAML/Attributes/PrincipalTag:Department = Finance
https://aws.amazon.com/SAML/Attributes/PrincipalTag:CostCenter = 1234
```

**OIDC session tags**

OIDC-based federation can also support session tags when the token and trust policy allow it. The exact claim mapping depends on the IdP and AWS configuration.

**ABAC example**

A role policy can allow access only when the session tag matches the resource tag.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "ec2:StartInstances",
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "aws:PrincipalTag/Department": "${aws:ResourceTag/Department}"
        }
      }
    }
  ]
}
```

**When to use session tags**

- You want one role to serve many teams.
- You need dynamic authorization based on identity attributes.
- You want to reduce role proliferation.
- You need stronger alignment between corporate directory attributes and AWS access.

**Common exam comparisons**

**IAM Identity Center vs SAML**

- Use **IAM Identity Center** for centralized workforce access across many AWS accounts.
- Use **SAML** when an enterprise IdP must directly federate users into AWS.
- IAM Identity Center gives a portal and account assignments.
- SAML gives a more direct federation path and is often used with existing enterprise SSO.

**IAM Identity Center vs OIDC**

- Use **IAM Identity Center** for human workforce access.
- Use **OIDC** for applications, workloads, and Kubernetes service accounts.
- IAM Identity Center is account-centric and portal-driven.
- OIDC is token-centric and workload-friendly.

**SAML vs OIDC**

- Use **SAML** for browser-based enterprise SSO and legacy corporate IdPs.
- Use **OIDC** for modern apps, APIs, and workload identity.
- SAML assertions are XML-based.
- OIDC tokens are JWTs (JSON Web Tokens).

**AssumeRole vs federation**

- Use **AssumeRole** when the caller already has AWS credentials or an AWS role session.
- Use **AssumeRoleWithSAML** when the caller authenticates through SAML.
- Use **AssumeRoleWithWebIdentity** when the caller authenticates through OIDC.
- Use **IAM Identity Center** when you want centralized workforce access management rather than direct role federation.

**Practical decision scenarios**

1. If employees need access to 20 AWS accounts from a single portal, choose IAM Identity Center.
2. If a corporate IdP already issues SAML assertions for employees, use SAML federation or connect that IdP to IAM Identity Center.
3. If a Kubernetes pod needs to call DynamoDB without static keys, use OIDC with IRSA.
4. If a CI/CD pipeline needs temporary AWS credentials, use OIDC federation or a dedicated role assumption pattern.
5. If one AWS role needs to access another account’s resources, use `AssumeRole` with a cross-account trust policy.

**Troubleshooting checklist**

- Verify the trust policy principal and STS action.
- Verify the permissions policy on the role.
- Confirm the IdP metadata or OIDC provider configuration is current.
- Check audience, issuer, and subject claims.
- Confirm certificate thumbprints or signing keys are valid.
- Check session duration limits and role chaining behavior.
- Review CloudTrail for `AssumeRole`, `AssumeRoleWithSAML`, or `AssumeRoleWithWebIdentity` events.

**CloudTrail clues**

CloudTrail is often the fastest way to diagnose federation issues.

- `AccessDenied` on STS usually means trust policy or condition mismatch.
- `InvalidIdentityToken` often points to OIDC issuer, audience, signature, or expiration problems.
- SAML failures often show audience or provider metadata mismatches.
- Missing permissions after successful federation usually means the role permissions policy is too restrictive.

**Quick reference commands**

```bash
aws iam list-saml-providers
aws iam get-saml-provider --saml-provider-arn arn:aws:iam::123456789012:saml-provider/CorpSAML
aws iam list-open-id-connect-providers
aws iam get-open-id-connect-provider --open-id-connect-provider-arn arn:aws:iam::123456789012:oidc-provider/oidc.eks.us-east-1.amazonaws.com/id/EXAMPLE

```

**Exam-focused takeaways**

- **Identity Center** is for centralized workforce access and account assignments.
- **SAML** is for enterprise browser federation and role mapping.
- **OIDC** is for modern token-based federation and workload identity.
- **Trust policies** control who can assume a role.
- **Permissions policies** control what the role can do.
- **Session tags** enable ABAC and reduce role sprawl.
- **Metadata, thumbprints, JWKS, issuer, audience, and subject** are the most common validation points.
