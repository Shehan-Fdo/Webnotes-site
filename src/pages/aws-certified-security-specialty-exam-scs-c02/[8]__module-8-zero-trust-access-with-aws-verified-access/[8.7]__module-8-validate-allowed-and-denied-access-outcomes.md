## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**What you are validating**

This topic is about proving whether an AWS request should succeed or fail, and then identifying *why* the result occurred. In exam scenarios, you often need to distinguish between identity-based permissions, resource-based permissions, permissions boundaries, service control policies (SCPs), session policies, and service-specific constraints.

A correct validation workflow answers three questions:

- Is the principal authenticated and using the expected credentials?
- Which policy layer allowed or denied the request?
- Is the failure actually authorization, or is it a different service error?

**Authorization evaluation model**

AWS evaluates access by combining multiple policy types and guardrails. The exact order can vary by service, but the practical exam model is:

- **Explicit deny wins** over any allow.
- Identity-based policies grant permissions to a principal.
- Resource-based policies grant permissions on the target resource.
- Permissions boundaries cap the maximum permissions a principal can receive.
- SCPs cap what accounts and principals in an AWS Organizations hierarchy can do.
- Session policies further restrict temporary credentials from STS (AWS Security Token Service).

A request can be allowed only if the effective permissions across all applicable layers permit it.

**Core policy layers to validate**

- **Identity-based policy**
  - Attached to IAM users, groups, or roles.
  - Answers: “Does this principal have permission in principle?”
- **Resource-based policy**
  - Attached to the resource, such as an S3 bucket policy, KMS key policy, SNS topic policy, SQS queue policy, Lambda resource policy, or ECR repository policy.
  - Answers: “Does the resource trust this principal?”
- **Permissions boundary**
  - Sets the maximum permissions a user or role can ever get from identity policies.
  - Useful for delegated administration and privilege containment.
- **SCP**
  - Applies at the AWS Organizations account, OU, or root level.
  - Restricts what identities in the account can do, even if IAM allows it.
- **Session policy**
  - Passed when assuming a role with STS.
  - Narrows the permissions of that specific session.

**How to validate with IAM policy simulation**

The IAM policy simulator helps you test identity-based policies and some resource-based policy interactions, but it has important limitations for exam scenarios.

Use it when you want to answer: “Would this IAM policy allow the action on this resource?”

Example CLI:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/AppRole \
  --action-names s3:GetObject ec2:StartInstances \
  --resource-arns arn:aws:s3:::my-bucket/example.txt arn:aws:ec2:us-east-1:123456789012:instance/i-0123456789abcdef0
```

Useful output fields include:

- `EvalDecision`
- `EvalDecisionDetails`
- `MatchedStatements`
- `MissingContextValues`

Important limitations:

- It does **not** fully evaluate **SCPs**.
- It does **not** fully evaluate **permissions boundaries** unless you explicitly include the boundary context in a way the simulator can model.
- It does **not** simulate every service-specific condition or downstream dependency.
- It may not reflect real-world behavior for cross-account access that depends on both identity and resource policies.

> If an exam question includes Organizations, boundaries, or STS session restrictions, do not rely on the simulator alone. Validate the full effective permission path.

**Validating permissions boundaries**

A permissions boundary limits the maximum permissions a principal can receive, even if an identity policy allows the action.

Practical validation approach:

- Check the principal’s attached identity policies.
- Check the permissions boundary policy document.
- Compare the action against both policies.
- Confirm whether the boundary contains an explicit deny or simply omits the action.

Example boundary that allows only read-only S3 access:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadOnlyS3",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": "*"
    }
  ]
}
```

If the identity policy allows `s3:PutObject` but the boundary does not, the effective result is denied.

Validation tip:

- Use `aws iam get-user`, `aws iam get-role`, or `aws iam list-attached-role-policies` to identify the boundary and attached policies.
- Use policy simulation only as a partial check.
- Confirm the final result by attempting the action with the same principal and observing the error and CloudTrail event.

**Validating SCP restrictions**

SCPs are common in exam questions because they create “mysterious” denies even when IAM looks correct.

What to check:

- The account’s OU placement in AWS Organizations.
- SCPs attached to the root, OU, or account.
- Whether the SCP uses an explicit deny or a deny-by-omission model.
- Whether the action is blocked for all principals, including administrators.

Example SCP that denies IAM role creation:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyRoleCreation",
      "Effect": "Deny",
      "Action": "iam:CreateRole",
      "Resource": "*"
    }
  ]
}
```

How an SCP-denied request typically appears:

- The API call fails even though the IAM policy allows it.
- The error often looks like an authorization failure such as `AccessDenied` or `UnauthorizedOperation`.
- CloudTrail may show the event with an error code indicating denial, but the root cause is the organization guardrail.

How an IAM-denied request differs:

- The principal lacks permission in its identity policy or resource policy.
- The same action fails, but there is no Organizations guardrail involved.
- The fix is usually to update IAM or the resource policy, not the SCP.

Validation methods for SCPs:

1. Check AWS Organizations for the account’s OU and attached SCPs.
2. Review whether the SCP explicitly denies the action or omits it.
3. Test the action with a principal that should otherwise be allowed.
4. Compare the result with a known action that is not restricted.
5. Use CloudTrail to confirm the denied API call and the principal identity.

**Validating STS session policies and assumed roles**

Temporary credentials can be more restrictive than the role itself because STS session policies narrow the effective permissions.

Common exam pattern:

- A role allows broad access.
- The caller assumes the role with a session policy that limits access to one bucket, one table, or one region.
- The request fails even though the role policy looks correct.

Example assume-role call with a session policy:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/AnalystRole \
  --role-session-name exam-session \
  --policy '{
    "Version":"2012-10-17",
    "Statement":[
      {
        "Effect":"Allow",
        "Action":["s3:GetObject"],
        "Resource":["arn:aws:s3:::reports-bucket/*"]
      }
    ]
  }'
```

Validation approach:

- Confirm the caller used `AssumeRole`, `AssumeRoleWithSAML`, or `AssumeRoleWithWebIdentity`.
- Inspect the session policy if one was supplied.
- Test the same action with and without the session policy.
- Use `sts get-caller-identity` to confirm the active principal.

Example:

```bash
aws sts get-caller-identity
```

If the role policy allows `s3:GetObject` on all buckets but the session policy allows only `reports-bucket`, then access to another bucket is denied.

**Using CloudTrail to confirm allowed and denied outcomes**

CloudTrail is one of the best tools for validating authorization outcomes because it records the API call, the principal, the event name, and the error details.

Key concepts:

- **Management events** record control-plane actions such as `CreateRole`, `RunInstances`, `PutBucketPolicy`, or `StartFunction`.
- **Data events** record data-plane actions such as `GetObject`, `PutObject`, Lambda function invocation, or some DynamoDB item-level activity depending on configuration.
- Denied events often still appear in CloudTrail, which helps you identify the exact API call that failed.

What to inspect in a denied event:

- `eventName`
- `eventSource`
- `userIdentity`
- `errorCode`
- `errorMessage`
- `requestParameters`
- `resources`

Common denial indicators:

- `AccessDenied`
- `UnauthorizedOperation`
- `Client.UnauthorizedOperation`
- `AccessDeniedException`
- `NotAuthorizedException`

Example CloudTrail event pattern for a denied EC2 action:

```json
{
  "eventName": "StartInstances",
  "eventSource": "ec2.amazonaws.com",
  "errorCode": "UnauthorizedOperation",
  "errorMessage": "You are not authorized to perform this operation.",
  "userIdentity": {
    "type": "AssumedRole",
    "arn": "arn:aws:sts::123456789012:assumed-role/AppRole/exam-session"
  }
}
```

Practical interpretation:

- `eventName` tells you the exact API action.
- `errorCode` tells you the failure class.
- `userIdentity.arn` tells you which role or user made the request.
- If the event exists with an authorization error, the request reached the service but was rejected by policy or guardrail evaluation.

**How to distinguish authorization failures from other errors**

Not every failure is a permissions problem. In exam questions, you should separate access denials from service or request issues.

Decision flow:

1. Check whether the request was authenticated.
2. Look for `AccessDenied`-style errors.
3. If the error is not authorization-related, inspect service-specific causes.
4. Confirm whether the resource exists and the ARN is correct.
5. Check throttling, quotas, and request syntax.

Common non-authorization errors:

- **Resource not found**
  - Example: wrong ARN, wrong region, deleted resource.
  - Often returns `NotFound`, `NoSuchBucket`, `ResourceNotFoundException`, or `InvalidInstanceID.NotFound`.
- **Validation error**
  - Example: malformed request, invalid parameter, unsupported combination.
  - Often returns `ValidationError` or `InvalidParameter`.
- **Throttling**
  - Example: too many requests in a short period.
  - Often returns `ThrottlingException` or `RequestLimitExceeded`.
- **Service-specific dependency failure**
  - Example: Lambda execution role missing permissions to write logs.
  - The primary action may succeed or fail depending on the downstream service call.

A useful troubleshooting rule:

- If CloudTrail shows the event with an authorization error, focus on policies.
- If CloudTrail shows no event, the request may not have reached the service, or you may be looking in the wrong region or account.
- If the event exists but the error is not access-related, investigate the service behavior instead of IAM.

**Resource-based policies and cross-account access**

Resource-based policies are often sufficient for granting access in cross-account scenarios, but not always.

When a resource policy alone can be enough:

- The resource policy directly names the external principal.
- The service supports resource-based authorization for that action.
- The principal does not need a separate identity-based allow if the resource policy grants access and no explicit deny exists.

Examples:

- S3 bucket policy granting `s3:GetObject` to a role in another account.
- SNS topic policy allowing `sns:Publish` from a specific principal.
- SQS queue policy allowing `sqs:SendMessage`.
- Lambda resource policy allowing `lambda:InvokeFunction`.
- ECR repository policy allowing image pull actions.

When identity-based permission is also required:

- The principal must call an action that the service evaluates through both identity and resource policy.
- The service or action requires the caller’s identity policy to allow the operation.
- The request is cross-account and the principal must be allowed by both sides.

Practical exam guidance:

- For cross-account access, check both the caller’s identity policy and the target resource policy.
- Confirm whether the resource policy uses the correct principal ARN, account ID, or organization condition.
- Remember that an explicit deny in either policy blocks access.

Example S3 bucket policy allowing a role from another account:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCrossAccountRead",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::210987654321:role/ExternalReader"
      },
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-shared-bucket/*"
    }
  ]
}
```

**Service examples you should recognize**

Different services surface authorization failures in slightly different ways.

- **EC2**
  - Common denied actions: `StartInstances`, `StopInstances`, `DescribeInstances`, `CreateTags`.
  - Often returns `UnauthorizedOperation` or `AuthFailure`.
  - Use CloudTrail management events to confirm the exact API call.
- **Lambda**
  - Common denied actions: `InvokeFunction`, `CreateFunction`, `UpdateFunctionCode`.
  - Resource-based policies are important for cross-account invocation.
  - Execution role permissions matter for downstream actions like writing logs.
- **SNS**
  - Common denied actions: `Publish`, `Subscribe`.
  - Topic policies often control cross-account publishing.
- **SQS**
  - Common denied actions: `SendMessage`, `ReceiveMessage`, `DeleteMessage`.
  - Queue policies can grant cross-account access, but the caller’s identity policy may still matter depending on the scenario.
- **ECR**
  - Common denied actions: `BatchGetImage`, `GetAuthorizationToken`, `PutImage`.
  - Repository policies control cross-account image pull or push.
  - Authentication to ECR is separate from repository authorization.

**Practical CLI validation patterns**

Use the AWS CLI to reproduce the exact request with the same principal and region.

Check identity first:

```bash
aws sts get-caller-identity
```

Test a specific action:

```bash
aws ec2 start-instances --instance-ids i-0123456789abcdef0
```

Test S3 access:

```bash
aws s3api get-object \
  --bucket my-bucket \
  --key example.txt \
  /tmp/example.txt
```

Test Lambda invocation:

```bash
aws lambda invoke \
  --function-name my-function \
  --payload '{}' \
  response.json
```

Test SNS publish:

```bash
aws sns publish \
  --topic-arn arn:aws:sns:us-east-1:123456789012:my-topic \
  --message "test"
```

Test SQS send:

```bash
aws sqs send-message \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/my-queue \
  --message-body "test"
```

Test ECR authorization token retrieval:

```bash
aws ecr get-authorization-token
```

Validation tips:

- Use the same region as the target resource.
- Use the same principal or assumed role session.
- Compare a known allowed action with the denied action.
- Capture the exact error text for later policy analysis.

**A practical troubleshooting flow**

Use this flow when you need to determine why access was allowed or denied:

1. Confirm the principal with `sts get-caller-identity`.
2. Verify the target resource ARN, region, and account.
3. Check whether the action is identity-based, resource-based, or both.
4. Review explicit denies in IAM policies, resource policies, permissions boundaries, SCPs, and session policies.
5. If the principal is assumed, inspect the STS session policy and role trust policy.
6. Check CloudTrail for the exact `eventName`, `errorCode`, and `userIdentity`.
7. Distinguish authorization errors from not-found, validation, or throttling errors.
8. Re-test with a minimal known-good policy or a controlled test principal.

**Quick comparison of common denial sources**

| Source | What it limits | Typical exam clue | Validation method |
|---|---|---|---|
| IAM identity policy | Principal permissions | User/role lacks allow | Simulate principal policy, inspect attached policies |
| Resource policy | Access to the resource | Cross-account access fails | Review bucket/topic/queue/key/function policy |
| Permissions boundary | Maximum permissions | Admin-like role still denied | Inspect boundary and compare against identity policy |
| SCP | Account-level guardrail | Everything in account is blocked | Check Organizations OU/root/account SCPs |
| Session policy | Temporary session permissions | Assumed role has less access than role | Inspect STS assume-role call and session policy |

**Exam mindset**

When the question asks you to validate allowed or denied access outcomes, choose the method that matches the policy layer involved:

- Use IAM simulation for identity-policy reasoning.
- Use Organizations and policy inspection for SCPs.
- Use STS context for assumed-role sessions.
- Use CloudTrail to confirm the real request and error.
- Use service-specific knowledge to interpret the denial correctly.

The safest exam answer is usually the one that validates the full effective permission path, not just the obvious IAM policy.
