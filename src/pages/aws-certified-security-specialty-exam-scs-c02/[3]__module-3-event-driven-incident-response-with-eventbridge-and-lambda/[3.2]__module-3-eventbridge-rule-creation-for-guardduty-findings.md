## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Overview**

Amazon EventBridge is the standard event routing service for Amazon GuardDuty findings. GuardDuty emits findings as events on the EventBridge default event bus in the same AWS Region where the finding is generated. You create rules to match those findings and send them to one or more targets for alerting, automation, or incident response.

For exam purposes, focus on three ideas:

- **GuardDuty** generates findings in the account and Region where it is enabled.
- **EventBridge rules** filter those findings using event patterns.
- **Targets** receive the matched events, such as Amazon SNS, AWS Lambda, Amazon SQS, AWS Step Functions, or another event bus.

A common design is to centralize security operations in a delegated administrator account for GuardDuty, then use EventBridge rules and cross-account event buses to route findings to a security tooling account or incident response workflow.

**How GuardDuty findings appear in EventBridge**

GuardDuty findings are delivered as JSON events. The event contains fields such as the finding type, severity, account ID, Region, resource details, and service-specific context. EventBridge evaluates the event against the rule pattern and invokes targets only when the pattern matches.

A typical GuardDuty event includes:

- `source`: `aws.guardduty`
- `detail-type`: `GuardDuty Finding`
- `detail.severity`: numeric severity value
- `detail.type`: finding type string
- `detail.resource.resourceType`: resource category such as `Instance`, `AccessKey`, or `S3Bucket`
- `detail.service.action`: optional nested context for the activity that triggered the finding

Because severity is numeric, EventBridge numeric matching uses an array of operator/value pairs. This is important when you want to alert only on high-severity findings.

```json
{
  "source": ["aws.guardduty"],
  "detail-type": ["GuardDuty Finding"],
  "detail": {
    "severity": [
      { "numeric": [">=", 7] }
    ]
  }
}
```

This pattern matches findings with severity 7 or higher. You can also filter on finding type prefixes or resource types to reduce noise.

**Creating the rule in the AWS Console**

Use the console when you want to validate the pattern visually or when you are building a quick alerting workflow.

1. Open the Amazon EventBridge console in the same Region where GuardDuty findings are generated.
2. Choose the default event bus unless you are intentionally forwarding events to a custom bus.
3. Choose **Create rule**.
4. Enter a rule name and description.
5. Select **Rule with an event pattern**.
6. Choose **AWS events** as the event source.
7. Select **GuardDuty** as the service source if the console offers the guided path, or build a custom pattern.
8. Define the event pattern.
9. Choose one or more targets.
10. Review and create the rule.

A rule must be **enabled** to process events. Disabled rules remain defined but do not invoke targets. This matters in troubleshooting because a valid pattern with a disabled rule produces no action.

> GuardDuty findings are emitted in the Region where GuardDuty is enabled. If you want centralized processing, you usually forward events from the source Region to a central event bus rather than expecting a different Region’s default bus to receive them automatically.

**Event pattern design**

EventBridge patterns are exacting. Small differences in field names or values can prevent matches. For GuardDuty, the most useful filters usually involve severity, finding type, resource type, and sometimes service action details.

Common pattern strategies:

- **Severity bands**
  - Match `>= 7` for high severity.
  - Match `>= 4` and `< 7` for medium severity if you want broader triage.
- **Finding type prefixes**
  - Match on a family of findings when you want a category-level alert.
  - Example: findings related to credential compromise or reconnaissance.
- **Resource type**
  - Match only EC2, IAM access keys, S3 buckets, or Kubernetes-related findings.
- **Service action**
  - Match on nested activity details when you need precision, such as API calls or network behavior.

Example pattern for high-severity findings from EC2-related resources:

```json
{
  "source": ["aws.guardduty"],
  "detail-type": ["GuardDuty Finding"],
  "detail": {
    "severity": [
      { "numeric": [">=", 7] }
    ],
    "resource": {
      "resourceType": ["Instance"]
    }
  }
}
```

Example pattern for a specific finding family using a prefix match:

```json
{
  "source": ["aws.guardduty"],
  "detail-type": ["GuardDuty Finding"],
  "detail": {
    "type": [
      { "prefix": "UnauthorizedAccess:" }
    ]
  }
}
```

Use precise patterns when you want to reduce alert fatigue. Use broader patterns when you want to ensure that security operations never miss a potentially important signal.

**Targets and delivery options**

EventBridge supports several target types. For GuardDuty alerting, the most common direct targets are:

- **Amazon SNS** for human notifications and fan-out
- **AWS Lambda** for custom enrichment or ticket creation
- **Amazon SQS** for buffering and downstream processing
- **AWS Step Functions** for multi-step response workflows
- **Another EventBridge event bus** for centralization or cross-account routing

Amazon Kinesis Data Firehose is supported as a target, but it is less common for direct GuardDuty alerting workflows than the options above.

When you target SNS, remember that the SNS topic must allow EventBridge to publish to it. That means the topic resource policy must permit `events.amazonaws.com` to call `sns:Publish`.

Example SNS topic policy statement:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowEventBridgePublish",
      "Effect": "Allow",
      "Principal": {
        "Service": "events.amazonaws.com"
      },
      "Action": "sns:Publish",
      "Resource": "arn:aws:sns:us-east-1:111122223333:SecurityAlerts"
    }
  ]
}
```

For Lambda targets, EventBridge needs permission to invoke the function. You usually add a resource-based permission statement to the function.

```bash
aws lambda add-permission \
  --function-name GuardDutyProcessor \
  --statement-id AllowEventBridgeInvoke \
  --action lambda:InvokeFunction \
  --principal events.amazonaws.com \
  --source-arn arn:aws:events:us-east-1:111122223333:rule/GuardDutyHighSeverity
```

For SQS targets, the queue policy must allow EventBridge to send messages. For Step Functions, the EventBridge rule needs permission to start the state machine execution.

**Input transformation and event shaping**

You do not always want to forward the full GuardDuty event. EventBridge input transformation lets you extract only the fields you need and format a smaller payload for the target.

This is useful when:

- You want concise SNS notifications.
- You want Lambda to receive a simplified schema.
- You want to enrich or normalize data before downstream processing.

Example input transformer for a notification target:

```json
{
  "InputPathsMap": {
    "account": "$.detail.accountId",
    "region": "$.region",
    "severity": "$.detail.severity",
    "type": "$.detail.type",
    "resource": "$.detail.resource.resourceType"
  },
  "InputTemplate": "{\"account\":\"<account>\",\"region\":\"<region>\",\"severity\":<severity>,\"type\":\"<type>\",\"resource\":\"<resource>\"}"
}
```

This reduces noise and makes alerts easier to read. It also helps when the target system expects a specific schema.

**Retry behavior and dead-letter queues**

EventBridge retries failed target deliveries automatically for a period of time. If the target remains unavailable or rejects the event, delivery can still fail. For critical security workflows, configure a dead-letter queue (DLQ) where supported so you can inspect undelivered events.

Use retry and DLQ settings when:

- The target is a Lambda function that may throttle.
- The target is an SNS topic with restrictive policy issues.
- The downstream system is not always available.
- You need auditability for missed alerts.

A DLQ does not replace correct permissions. It helps you recover from transient failures and diagnose delivery issues.

**Cross-account and centralized security patterns**

In enterprise environments, GuardDuty is often enabled across multiple accounts under AWS Organizations. A delegated administrator account can manage GuardDuty centrally, but findings still originate in the member account and Region where the activity occurred.

There are two common routing patterns:

- **Local processing in each account**
  - Each account/Region has its own EventBridge rule.
  - Useful for account-specific automation.
- **Central forwarding to a security account**
  - A rule in the source account sends matched events to a custom event bus in a central account.
  - The destination event bus policy must allow `events:PutEvents`.
  - The source rule must target the destination bus ARN correctly.

Example destination event bus policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowSourceAccountPutEvents",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:root"
      },
      "Action": "events:PutEvents",
      "Resource": "arn:aws:events:us-east-1:444455556666:event-bus/SecurityBus"
    }
  ]
}
```

Example rule target that forwards to another event bus:

```json
{
  "Arn": "arn:aws:events:us-east-1:444455556666:event-bus/SecurityBus",
  "Id": "CentralSecurityBus"
}
```

In some setups, you may also need an IAM role that allows the source rule to call `events:PutEvents` on the destination bus. The exact permission model depends on how the rule and bus are configured, so verify both the bus policy and the rule target configuration.

> Do not assume GuardDuty findings “move” across accounts by themselves. EventBridge forwarding is an explicit rule-based action, and the destination bus must accept the event.

**Organization and delegated administrator considerations**

When GuardDuty is managed centrally in AWS Organizations, the delegated administrator account is the control plane for GuardDuty configuration. That does not mean all EventBridge rules live only there.

Practical deployment model:

- GuardDuty is enabled in member accounts and Regions.
- Findings are generated where the activity occurs.
- EventBridge rules are deployed where the events are emitted, unless you are forwarding them.
- A central security account receives forwarded events for aggregation, enrichment, or ticketing.

This distinction matters on the exam because centralized GuardDuty administration and centralized event routing are related but separate design choices.

**AWS CLI examples**

You can create and manage rules with the AWS CLI. This is useful for automation and for understanding the underlying API behavior.

Create a rule:

```bash
aws events put-rule \
  --name GuardDutyHighSeverity \
  --event-pattern '{
    "source": ["aws.guardduty"],
    "detail-type": ["GuardDuty Finding"],
    "detail": {
      "severity": [{ "numeric": [">=", 7] }]
    }
  }' \
  --state ENABLED
```

Add an SNS target:

```bash
aws events put-targets \
  --rule GuardDutyHighSeverity \
  --targets '[
    {
      "Id": "SecurityAlertsTopic",
      "Arn": "arn:aws:sns:us-east-1:111122223333:SecurityAlerts"
    }
  ]'
```

Add a Lambda target:

```bash
aws events put-targets \
  --rule GuardDutyHighSeverity \
  --targets '[
    {
      "Id": "GuardDutyProcessor",
      "Arn": "arn:aws:lambda:us-east-1:111122223333:function:GuardDutyProcessor"
    }
  ]'
```

Describe the rule:

```bash
aws events describe-rule --name GuardDutyHighSeverity
```

List targets:

```bash
aws events list-targets-by-rule --rule GuardDutyHighSeverity
```

Disable the rule during maintenance:

```bash
aws events disable-rule --name GuardDutyHighSeverity
```

**Troubleshooting checklist**

If a GuardDuty finding does not trigger your workflow, check the following:

- The rule is **enabled**.
- The rule exists in the same Region as the GuardDuty finding, or the event is being forwarded intentionally.
- The event pattern matches the actual event structure.
- The target policy allows EventBridge to invoke or publish.
- The destination event bus policy allows `PutEvents` for cross-account forwarding.
- The Lambda function, SNS topic, SQS queue, or Step Functions state machine exists and is reachable.
- The input transformer does not reference missing fields.
- The rule is attached to the correct event bus, usually the default bus for native GuardDuty events.

If you are testing with sample events, make sure the sample matches the real GuardDuty schema. A pattern that works for one finding type may not match another because nested fields differ.

**Exam focus**

For AWS Certified Security – Specialty, expect questions that test your ability to design secure, scalable alerting for GuardDuty findings.

Key takeaways:

- **EventBridge** is the native routing service for GuardDuty findings.
- **Rules** filter on `source`, `detail-type`, severity, finding type, and resource details.
- **Targets** commonly include SNS, Lambda, SQS, Step Functions, and event buses.
- **SNS** requires a topic policy allowing `events.amazonaws.com` to publish.
- **Lambda** requires an invoke permission for EventBridge.
- **Cross-account forwarding** requires a destination bus policy and correct target configuration.
- **Delegated administrator** centralizes GuardDuty management, but event routing still depends on where findings are generated and where rules are deployed.

A strong exam answer usually distinguishes between centralized GuardDuty administration and centralized event delivery. That separation is often the difference between a correct architecture and an incomplete one.
