## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Event pattern matching overview**

Amazon EventBridge routes events by comparing an incoming event to a rule pattern. For exam purposes, you need to know how to match specific finding types from security services without overmatching unrelated alerts.

EventBridge patterns are JSON documents that match on fields in the event payload. You typically filter by:
- `source`
- `detail-type`
- fields inside `detail`

EventBridge supports more than simple equality. Common operators include:
- `prefix`
- `anything-but`
- `numeric`
- `exists`
- `null`
- `wildcard`
- `cidr`

Use these when you need to route only a subset of findings, such as high-severity alerts, a specific product, or a particular compliance state.

> EventBridge pattern matching is schema-sensitive. Your sample event must reflect the real service payload, or `test-event-pattern` can give misleading results.

**Core matching model**

A rule pattern matches when all specified conditions are satisfied. You usually combine top-level service metadata with detail fields to narrow the event stream.

Key points:
- Match on `source` and `detail-type` first to reduce noise.
- Match on `detail` fields for service-specific filtering.
- Arrays are matched by element value, not by array index notation.
- Nested objects are matched by their field paths, but you do not write `[0]` indexes in patterns.
- Use `exists` when you need to detect whether a field is present.
- Use `null` when the service emits explicit null values and you need to route them separately.

A practical rule often looks like this:
```json
{
  "source": ["aws.securityhub"],
  "detail-type": ["Security Hub Findings - Imported"],
  "detail": {
    "Severity": {
      "Label": ["HIGH", "CRITICAL"]
    }
  }
}
```

**EventBridge operators you should know**

These operators are common exam targets because they let you express intent precisely.

- **`prefix`** matches values that start with a string.
  - Useful for resource names, ARNs, or event type codes with a shared prefix.
- **`anything-but`** excludes one or more values.
  - Useful when you want all findings except a known noisy category.
- **`numeric`** compares numbers using operators such as `>`, `>=`, `<`, `<=`, and ranges.
  - Useful for severity thresholds.
- **`exists`** checks whether a field is present.
  - Useful when a service emits optional fields.
- **`null`** matches explicit null values.
  - Useful when a field is intentionally unset.
- **`wildcard`** matches string patterns with wildcard characters.
  - Useful for flexible matching on ARNs or event type codes.
- **`cidr`** matches IP addresses in a CIDR block.
  - Useful for network-related findings.

Example of multiple operators in one pattern:
```json
{
  "source": ["aws.guardduty"],
  "detail-type": ["GuardDuty Finding"],
  "detail": {
    "severity": [
      { "numeric": [">=", 7] }
    ],
    "type": [
      { "prefix": "UnauthorizedAccess:" }
    ]
  }
}
```

**Security Hub findings**

Security Hub findings delivered to EventBridge use the `aws.securityhub` source and commonly the `Security Hub Findings - Imported` detail type. The finding payload is a single finding object under `detail`, not a wrapper like `detail.findings`.

A simplified real-world event shape looks like this:
```json
{
  "version": "0",
  "id": "abcd-1234",
  "detail-type": "Security Hub Findings - Imported",
  "source": "aws.securityhub",
  "account": "111122223333",
  "time": "2026-04-26T12:00:00Z",
  "region": "us-east-1",
  "resources": [],
  "detail": {
    "findings": [
      {
        "AwsAccountId": "111122223333",
        "Types": [
          "Software and Configuration Checks/AWS Security Best Practices"
        ],
        "Severity": {
          "Label": "HIGH",
          "Normalized": 70
        },
        "ProductArn": "arn:aws:securityhub:us-east-1::product/aws/securityhub",
        "Title": "S3 bucket allows public read access"
      }
    ]
  }
}
```

In practice, EventBridge patterns match the actual event structure emitted by the service. For Security Hub, the finding is commonly represented as a finding object in `detail.findings` for imported findings delivered through EventBridge. When writing patterns, match the fields exactly as they appear in the event you receive.

A correct pattern for high-severity Security Hub findings is:
```json
{
  "source": ["aws.securityhub"],
  "detail-type": ["Security Hub Findings - Imported"],
  "detail": {
    "findings": {
      "Severity": {
        "Label": ["HIGH", "CRITICAL"]
      }
    }
  }
}
```

To route only specific finding types, match the `Types` field:
```json
{
  "source": ["aws.securityhub"],
  "detail-type": ["Security Hub Findings - Imported"],
  "detail": {
    "findings": {
      "Types": ["Software and Configuration Checks/AWS Security Best Practices"]
    }
  }
}
```

You can also combine product and type filters:
```json
{
  "source": ["aws.securityhub"],
  "detail-type": ["Security Hub Findings - Imported"],
  "detail": {
    "findings": {
      "ProductArn": [
        "arn:aws:securityhub:us-east-1::product/aws/securityhub"
      ],
      "Types": [
        "TTPs/Defense Evasion"
      ]
    }
  }
}
```

Use `ProductArn` when you want to scope findings to a specific provider or integration. It is a valid discriminator, but it may not be sufficient by itself if multiple products emit similar finding categories.

Common Security Hub use cases:
- Route `CRITICAL` findings to an incident response queue.
- Send compliance-related findings to a ticketing system.
- Separate AWS-native findings from partner findings using `ProductArn`.
- Filter by `Types` to isolate best-practice, malware, or privilege escalation findings.

**GuardDuty findings**

GuardDuty findings are a common exam topic because they are high-signal security alerts and often routed to automated response workflows.

A typical GuardDuty EventBridge event uses:
- `source`: `aws.guardduty`
- `detail-type`: `GuardDuty Finding`
- `detail.type`: the finding category
- `detail.severity`: a numeric severity score

Example pattern for high-severity credential or access findings:
```json
{
  "source": ["aws.guardduty"],
  "detail-type": ["GuardDuty Finding"],
  "detail": {
    "severity": [
      { "numeric": [">=", 7] }
    ],
    "type": [
      "UnauthorizedAccess:EC2/SSHBruteForce",
      "UnauthorizedAccess:IAMUser/ConsoleLogin"
    ]
  }
}
```

Use `prefix` when you want a family of related findings:
```json
{
  "source": ["aws.guardduty"],
  "detail-type": ["GuardDuty Finding"],
  "detail": {
    "type": [
      { "prefix": "UnauthorizedAccess:" }
    ]
  }
}
```

Use `anything-but` to exclude noisy categories:
```json
{
  "source": ["aws.guardduty"],
  "detail-type": ["GuardDuty Finding"],
  "detail": {
    "type": [
      {
        "anything-but": [
          "Recon:EC2/PortProbeUnprotectedPort",
          "Recon:EC2/Portscan"
        ]
      }
    ]
  }
}
```

GuardDuty severity is numeric and commonly described on a 0.0 to 8.9 scale. For exam prep, focus on threshold logic rather than memorizing every category.

**AWS Config compliance and configuration events**

AWS Config emits different event shapes depending on the event type. Do not assume one generic schema for all Config notifications.

Common scenarios include:
- Configuration item changes
- Compliance change notifications
- Rule evaluation results

A compliance change event often includes fields such as:
- `configRuleName`
- `newEvaluationResult`
- `newEvaluationResult.complianceType`
- `resourceType`
- `resourceId`

Example pattern for noncompliant evaluations from a specific rule:
```json
{
  "source": ["aws.config"],
  "detail-type": ["Config Rules Compliance Change"],
  "detail": {
    "configRuleName": ["s3-bucket-public-read-prohibited"],
    "newEvaluationResult": {
      "complianceType": ["NON_COMPLIANT"]
    },
    "resourceType": ["AWS::S3::Bucket"]
  }
}
```

Example pattern for configuration item changes on a resource type:
```json
{
  "source": ["aws.config"],
  "detail-type": ["Config Configuration Item Change"],
  "detail": {
    "resourceType": ["AWS::EC2::SecurityGroup"]
  }
}
```

Use `exists` when you want to detect whether a compliance result is present:
```json
{
  "source": ["aws.config"],
  "detail-type": ["Config Rules Compliance Change"],
  "detail": {
    "newEvaluationResult": {
      "complianceType": [
        { "exists": true }
      ]
    }
  }
}
```

Practical routing examples:
- Send `NON_COMPLIANT` S3 bucket findings to a remediation Lambda.
- Alert on security group changes affecting production subnets.
- Create tickets only for specific managed rules, not every Config event.

**AWS Health events**

AWS Health events help you route service issues, scheduled changes, and account-specific notifications. The exact schema varies by event type, so match on the fields that are stable for the event class you are using.

Common fields include:
- `source`: `aws.health`
- `detail-type`: a Health event category
- `detail.eventTypeCode`
- `detail.eventTypeCategory`
- `detail.service`
- `detail.affectedEntities`

Example pattern for service issues affecting EC2:
```json
{
  "source": ["aws.health"],
  "detail-type": ["AWS Health Event"],
  "detail": {
    "eventTypeCategory": ["issue"],
    "service": ["EC2"]
  }
}
```

A more precise pattern often uses `eventTypeCode`:
```json
{
  "source": ["aws.health"],
  "detail-type": ["AWS Health Event"],
  "detail": {
    "eventTypeCategory": ["issue"],
    "eventTypeCode": ["AWS_EC2_INSTANCE_REBOOT_SCHEDULED"]
  }
}
```

If the event includes structured service data, match the actual payload shape from a sample event rather than assuming `service` is always a plain string. For exam questions, the safest approach is to filter on `eventTypeCode` and `eventTypeCategory` first, then add entity or region constraints if needed.

**End-to-end rule example**

A realistic EventBridge rule usually includes:
- an event pattern
- a target
- optional input transformation
- a bus selection

Example rule that sends high-severity Security Hub findings to an SNS topic:
```json
{
  "Name": "securityhub-high-severity-to-sns",
  "EventPattern": {
    "source": ["aws.securityhub"],
    "detail-type": ["Security Hub Findings - Imported"],
    "detail": {
      "findings": {
        "Severity": {
          "Label": ["HIGH", "CRITICAL"]
        }
      }
    }
  },
  "State": "ENABLED",
  "Targets": [
    {
      "Id": "sns-target",
      "Arn": "arn:aws:sns:us-east-1:111122223333:security-alerts"
    }
  ]
}
```

CLI example to create the rule:
```bash
aws events put-rule \
  --name securityhub-high-severity-to-sns \
  --event-pattern file://pattern.json \
  --state ENABLED
```

Add a target:
```bash
aws events put-targets \
  --rule securityhub-high-severity-to-sns \
  --targets "Id"="sns-target","Arn"="arn:aws:sns:us-east-1:111122223333:security-alerts"
```

If you need to reshape the event before delivery, use an input transformer:
```json
{
  "Id": "lambda-target",
  "Arn": "arn:aws:lambda:us-east-1:111122223333:function:remediate-finding",
  "InputTransformer": {
    "InputPathsMap": {
      "title": "$.detail.findings[0].Title",
      "severity": "$.detail.findings[0].Severity.Label"
    },
    "InputTemplate": "{\"findingTitle\": <title>, \"severity\": <severity>}"
  }
}
```

> Verify the input paths against a real sample event. If the service emits an array, your transformer path must match the actual structure exactly.

**Testing and validation**

Use `aws events test-event-pattern` to validate a pattern against a sample event:
```bash
aws events test-event-pattern \
  --event-pattern file://pattern.json \
  --event file://sample-event.json
```

Best practice:
- Capture a real event from the EventBridge console, archive, or CloudWatch Logs if available.
- Test against the exact service payload, especially for Security Hub and GuardDuty.
- Confirm that the sample event includes the same nesting and field names as production events.
- Re-test after changing operators such as `prefix`, `anything-but`, or `numeric`.

A useful workflow is:
1. Capture a real event from the service.
2. Save it as `sample-event.json`.
3. Write the smallest possible pattern that matches one known finding.
4. Add filters one at a time.
5. Re-run `test-event-pattern` after each change.

**Troubleshooting common failures**

If a rule does not match, check these first:
- `source` is correct and exact.
- `detail-type` matches the service event type.
- Field names use the exact case from the payload.
- Arrays are matched as arrays, not with index notation.
- The sample event reflects the real schema for that service.
- The pattern is not too restrictive after adding multiple operators.

Common exam traps:
- Assuming every service uses the same `detail` shape.
- Using a field path that exists in one event type but not another.
- Overusing `ProductArn` when `Types` or severity is the better discriminator.
- Matching AWS Health with a generic `service` string when `eventTypeCode` is the more reliable filter.
- Writing a pattern that only works for one sample event but not the actual service payload.

**Quick comparison**

| Service | Common source | Useful fields for specific finding types | Typical use |
|---|---|---|---|
| Security Hub | `aws.securityhub` | `findings.Severity.Label`, `findings.Types`, `findings.ProductArn` | Route security findings by severity or category |
| GuardDuty | `aws.guardduty` | `severity`, `type` | Detect high-risk threats and specific attack families |
| AWS Config | `aws.config` | `configRuleName`, `newEvaluationResult.complianceType`, `resourceType` | Route compliance drift and configuration changes |
| AWS Health | `aws.health` | `eventTypeCode`, `eventTypeCategory`, `service`, `affectedEntities` | Route service issues and scheduled changes |

**Exam focus summary**

For SCS-C02, you should be able to:
- Recognize the correct EventBridge source and detail type for each service.
- Match specific finding types using exact values, prefixes, exclusions, and numeric thresholds.
- Choose the right field for the service instead of assuming a generic schema.
- Validate patterns with real sample events.
- Design a rule that routes only the findings you care about, not the entire alert stream.
