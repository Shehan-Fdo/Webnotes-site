## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Lambda remediation workflow design**

Lambda remediation workflows automate security response after a detection signal identifies a risky condition. In SCS-C02 scenarios, you usually design a pipeline that detects an issue, routes the event, enriches context if needed, and executes a controlled fix. The key exam skill is choosing the right orchestration and permissions model for the failure mode, blast radius, and operational constraints.

A strong design separates:
- **Detection** from remediation logic
- **Event routing** from execution
- **Idempotent** actions from one-time actions
- **Fast containment** from slower, approval-based remediation

**Core architecture patterns**

A common pattern is:

1. A security service emits an event or finding.
2. Amazon EventBridge matches the event.
3. A target invokes AWS Lambda, AWS Systems Manager Automation, or Step Functions.
4. The remediation action updates the resource, records the outcome, and optionally notifies operators.

Example flow:

```text
Security Hub / GuardDuty / CloudTrail / Config
                ↓
           EventBridge rule
                ↓
     Lambda or Step Functions
                ↓
   Remediation API call / SSM Automation
                ↓
   Tag, quarantine, rotate, disable, or notify
```

Use this pattern when you need near-real-time response and the action can be expressed as code. Prefer managed automation when the fix is mostly declarative or already available as an SSM document.

**Event sources and what they mean**

Different AWS services produce different event shapes, so you must match the remediation design to the source.

- **AWS Security Hub** aggregates findings from AWS services and partner products.
  - Security Hub integrates with EventBridge and emits events for findings and other state changes.
  - Do not assume every Security Hub event is a finding event.
  - Match the actual event schema for the event type you use.
- **Amazon GuardDuty** emits findings that often drive containment actions.
  - Common responses include isolating an EC2 instance, revoking credentials, or blocking suspicious network paths.
- **AWS CloudTrail** records API activity.
  - Use it for detection of risky actions such as disabling logging, creating access keys, or changing security groups.
- **AWS Config** evaluates resource configuration against rules.
  - Use it for drift detection and compliance-driven remediation.
  - Config remediation often uses AWS Systems Manager Automation documents or Lambda-backed custom rules.
- **Amazon EventBridge** can also receive custom events from your own applications.
  - This is useful when internal systems detect abuse, policy violations, or suspicious behavior.

**Choosing Lambda, Step Functions, or SSM Automation**

Use the right tool for the remediation shape.

| Tool | Best for | Strengths | Tradeoffs |
|---|---|---|---|
| Lambda | Small, fast, code-driven fixes | Low latency, flexible logic, easy event integration | Limited runtime, custom error handling required |
| Step Functions | Multi-step workflows | Branching, retries, approvals, auditability | More setup, higher workflow complexity |
| SSM Automation | Infrastructure and operational runbooks | Native AWS operations, document-based control, good for repeatable fixes | Less flexible for custom logic |

Use **Lambda** when:
- You need to inspect event details and make a quick decision.
- The action is a small number of AWS API calls.
- You want to enrich or normalize the event before remediation.

Use **Step Functions** when:
- You need multiple steps, branching, or human approval.
- You want explicit retry and compensation logic.
- You need to coordinate several services safely.

Use **SSM Automation** when:
- The remediation is a standard operational runbook.
- You want a managed document with clear execution steps.
- You need to remediate EC2, EBS, IAM-adjacent, or network settings in a controlled way.

**End-to-end remediation design**

A practical design should include detection, filtering, invocation permissions, execution, and feedback.

1. **Detect**
   - Use Security Hub, GuardDuty, CloudTrail, Config, or custom events.
   - Filter for the exact condition you want to remediate.
2. **Route**
   - Use EventBridge rules to match the event pattern.
   - Keep the rule narrow to avoid accidental remediation.
3. **Authorize**
   - Grant the event source permission to invoke the target.
   - Use resource-based policies where required.
4. **Execute**
   - Run Lambda, Step Functions, or SSM Automation.
   - Make the action idempotent.
5. **Record**
   - Log the action, affected resource, and outcome.
   - Emit a metric or finding update for auditability.
6. **Notify**
   - Send an alert to SNS, Slack, or ticketing if the action fails or needs review.

**EventBridge rule design**

EventBridge is the usual routing layer for security automation. It supports pattern matching on event source, detail type, and selected fields in the event payload.

A simple rule for a custom security event might look like this:

```json
{
  "source": ["custom.security"],
  "detail-type": ["UnauthorizedPublicS3Change"],
  "detail": {
    "bucketName": [{ "exists": true }],
    "accountId": [{ "exists": true }]
  }
}
```

For AWS service events, match the actual schema published by that service. Be careful with nested arrays and object fields. A pattern that looks correct in prose may not match the real event structure if the field path or data type differs.

> EventBridge pattern matching is exacting. If the event schema changes or the field is nested differently than expected, the rule may never fire.

**Lambda invocation and permissions**

When EventBridge invokes Lambda, you need both the rule and the Lambda permission. The Lambda function must allow the EventBridge rule ARN to invoke it.

Example permission:

```bash
aws lambda add-permission \
  --function-name security-remediate \
  --statement-id allow-eventbridge-invoke \
  --action lambda:InvokeFunction \
  --principal events.amazonaws.com \
  --source-arn arn:aws:events:us-east-1:123456789012:rule/security-remediation-rule
```

Example EventBridge target:

```bash
aws events put-targets \
  --rule security-remediation-rule \
  --targets "Id"="1","Arn"="arn:aws:lambda:us-east-1:123456789012:function:security-remediate"
```

Important permission points:
- **EventBridge to Lambda** uses a Lambda resource-based policy.
- **Lambda to AWS APIs** uses the function execution role.
- **Cross-account** invocation usually requires explicit resource policies and careful trust boundaries.
- **Cross-region** designs often use EventBridge event buses, forwarding rules, or centralized security accounts.

**Lambda execution role design**

The Lambda execution role should have only the permissions needed for the remediation action.

Example for a function that disables a public S3 bucket policy and tags the bucket:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "S3Remediation",
      "Effect": "Allow",
      "Action": [
        "s3:GetBucketPolicy",
        "s3:PutBucketPolicy",
        "s3:GetPublicAccessBlock",
        "s3:PutPublicAccessBlock",
        "s3:GetBucketTagging",
        "s3:PutBucketTagging"
      ],
      "Resource": [
        "arn:aws:s3:::example-bucket"
      ]
    }
  ]
}
```

For account-level public access block, the permissions and resource scope differ from bucket-level actions. Do not assume a single bucket ARN covers every public access block use case.

**Idempotency and duplicate events**

Security automation must tolerate duplicate delivery and replay.

Why duplicates happen:
- EventBridge retries on delivery failure.
- Security services may emit multiple related events.
- Operators may replay archived events for testing or recovery.

Design for idempotency by:
- Checking current state before changing anything.
- Using tags or markers such as `Remediated=true`.
- Recording a processed event ID in DynamoDB or another store.
- Making repeated API calls safe when possible.

Example idempotent logic:
- If an S3 bucket already has public access blocked, exit without changing it.
- If an EC2 instance is already isolated, do not reapply the same security group change.
- If an IAM access key is already inactive, skip deactivation.

**Failure handling, retries, and dead-letter paths**

EventBridge and Lambda have different retry behaviors, so do not treat them as the same system.

- **EventBridge** retries target delivery according to its own policy.
- **Lambda asynchronous invocation** has its own retry model when Lambda itself is the async invoker.
- For EventBridge-to-Lambda, focus on EventBridge target retries and Lambda error handling together.

For Lambda asynchronous invocations, you can configure:
- **On-failure destinations** to send failed events to SQS, SNS, EventBridge, or another Lambda.
- **Dead-letter queues (DLQs)** to capture events that could not be processed after retries.

Use a destination when:
- You want richer failure context.
- You need to route failures into a workflow or audit stream.

Use a DLQ when:
- You want a simple queue for later inspection.
- You need a durable fallback for failed async events.

Example async configuration:

```bash
aws lambda put-function-event-invoke-config \
  --function-name security-remediate \
  --maximum-retry-attempts 2 \
  --maximum-event-age-in-seconds 3600 \
  --destination-config '{
    "OnFailure": {
      "Destination": "arn:aws:sqs:us-east-1:123456789012:security-remediation-failures"
    }
  }'
```

**Safe deployment patterns for remediation functions**

Remediation functions are operationally sensitive. A bad deployment can disable the wrong resource or create outages.

Use:
- **Lambda aliases** to point EventBridge at a stable version.
- **Published versions** for immutable deployments.
- **Canary or linear traffic shifting** with AWS CodeDeploy for safer releases.
- **Rollback alarms** to revert if error rates or failed remediations increase.

Recommended pattern:
- Deploy a new version.
- Shift a small percentage of traffic to the new alias.
- Monitor logs, metrics, and remediation outcomes.
- Roll back if the function misclassifies events or touches the wrong resources.

Example alias update:

```bash
aws lambda update-alias \
  --function-name security-remediate \
  --name prod \
  --function-version 12
```

**Common remediation scenarios**

**1. Public S3 exposure**
- Detect a bucket policy or ACL that makes data public.
- Apply S3 Block Public Access where appropriate.
- Remove public bucket policy statements.
- Add a restrictive bucket policy and verify access.

Example CLI check:

```bash
aws s3api get-public-access-block --bucket example-bucket
aws s3api get-bucket-policy --bucket example-bucket
```

**2. IAM access key compromise**
- Detect suspicious key use from CloudTrail or GuardDuty.
- Disable the access key.
- Rotate credentials and notify the account owner.
- Consider forcing MFA or session revocation where applicable.

Example:

```bash
aws iam update-access-key \
  --user-name alice \
  --access-key-id AKIAEXAMPLE123456 \
  --status Inactive
```

**3. EC2 instance containment**
- Detect malware, beaconing, or suspicious outbound traffic.
- Quarantine the instance by changing its security group or NACL.
- Preserve evidence before termination if forensics are required.
- Use rule-ID-based security group changes when possible to avoid removing the wrong rule.

Example containment action:

```bash
aws ec2 modify-instance-attribute \
  --instance-id i-0123456789abcdef0 \
  --groups sg-quarantine123
```

**4. KMS or S3 encryption enforcement**
- Detect unencrypted storage or missing encryption controls.
- Enable default encryption or apply a compliant configuration.
- For S3, verify bucket encryption and public access settings together.
- For EBS, enforce encryption at creation time through policy and guardrails.

Example:

```bash
aws s3api put-bucket-encryption \
  --bucket example-bucket \
  --server-side-encryption-configuration '{
    "Rules": [{
      "ApplyServerSideEncryptionByDefault": {
        "SSEAlgorithm": "AES256"
      }
    }]
  }'
```

**5. CloudTrail or GuardDuty-triggered containment**
- Detect disabled logging, unusual API calls, or credential misuse.
- Re-enable logging if it was turned off.
- Quarantine the principal or resource.
- Escalate to human review if the action could affect production.

**6. Security group exposure**
- Detect an overly permissive inbound rule.
- Remove the exact offending rule.
- Prefer rule IDs or full `IpPermissions` matching for precision.

Example safer revoke pattern:

```bash
aws ec2 revoke-security-group-ingress \
  --group-id sg-0123456789abcdef0 \
  --ip-permissions '[
    {
      "IpProtocol": "tcp",
      "FromPort": 22,
      "ToPort": 22,
      "IpRanges": [{"CidrIp": "0.0.0.0/0"}]
    }
  ]'
```

**Cross-account and cross-region remediation**

Security operations often centralize detection in a security account and remediate resources in member accounts.

Design options:
- **Cross-account EventBridge bus** forwarding from member accounts to a central security account.
- **AssumeRole** from the remediation Lambda into the target account.
- **SSM Automation** executed in the target account with delegated permissions.
- **Step Functions** coordinating actions across accounts with explicit state and approvals.

Cross-region considerations:
- EventBridge rules are regional.
- Security Hub findings and GuardDuty findings are regional.
- You may need regional forwarding or a centralized aggregation strategy.
- Remediation should occur in the same region as the affected resource unless the action is global.

Practical guidance:
- Keep the detection close to the source region.
- Forward only the necessary event data.
- Use a target-account role with least privilege.
- Log the source account, target account, and region for every action.

**Resource-based policies and trust boundaries**

For end-to-end automation, permissions must line up across services.

Common requirements:
- EventBridge must be allowed to invoke Lambda.
- Lambda must be allowed to call the target AWS APIs.
- If Lambda assumes a role in another account, that role must trust the Lambda execution role.
- If SNS or SQS is used as a failure destination, the destination policy must allow the sender.

Example trust policy for cross-account remediation:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:role/security-remediation-lambda-role"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

**Operational logging and auditability**

Security remediation must be explainable after the fact.

Log:
- Event ID
- Source service
- Affected resource ARN
- Decision taken
- API calls made
- Success or failure
- Correlation ID for tracing

Useful outputs:
- CloudWatch Logs for function execution
- CloudWatch Metrics for success/failure counts
- Security Hub notes or updated findings
- SNS notifications for human review

Example structured log entry:

```json
{
  "eventId": "abc-123",
  "resource": "arn:aws:s3:::example-bucket",
  "action": "PutPublicAccessBlock",
  "result": "success",
  "timestamp": "2026-04-26T12:00:00Z"
}
```

**Lambda handler example**

A minimal remediation handler should validate the event, check current state, and act only when needed.

```python
import json
import boto3

s3 = boto3.client("s3")

def lambda_handler(event, context):
    detail = event.get("detail", {})
    bucket = detail.get("bucketName")

    if not bucket:
        return {"status": "ignored", "reason": "missing bucketName"}

    current = s3.get_public_access_block(Bucket=bucket)
    config = current.get("PublicAccessBlockConfiguration", {})

    if all(config.get(k) for k in [
        "BlockPublicAcls",
        "IgnorePublicAcls",
        "BlockPublicPolicy",
        "RestrictPublicBuckets"
    ]):
        return {"status": "already_compliant", "bucket": bucket}

    s3.put_public_access_block(
        Bucket=bucket,
        PublicAccessBlockConfiguration={
            "BlockPublicAcls": True,
            "IgnorePublicAcls": True,
            "BlockPublicPolicy": True,
            "RestrictPublicBuckets": True
        }
    )

    return {"status": "remediated", "bucket": bucket}
```

This example is intentionally simple. In production, you should:
- Validate the event schema.
- Handle multiple offending resources.
- Catch AWS API exceptions.
- Emit structured logs.
- Make the action safe to repeat.

**When to use human approval**

Not every security event should be auto-remediated.

Use approval or manual review when:
- The action could interrupt production traffic.
- The event confidence is low.
- The remediation may destroy evidence.
- The resource is shared across teams or accounts.

Examples:
- Terminating a suspected compromised instance in a critical workload
- Rotating a shared IAM credential without coordination
- Changing a network path that may affect customer traffic

**Exam-focused decision cues**

Look for these cues in SCS-C02 questions:
- **Fast containment** suggests Lambda or SSM Automation.
- **Multi-step branching** suggests Step Functions.
- **Standard AWS runbook** suggests SSM Automation.
- **Cross-account security operations** suggest centralized EventBridge plus AssumeRole or delegated automation.
- **Need for audit trail and approvals** suggests Step Functions or SSM Automation over plain Lambda.
- **Need for idempotent, low-latency fixes** suggests Lambda with strict checks.

**Quick comparison of common remediation actions**

- **Disable IAM access key**
  - Good for suspected credential compromise.
  - Low complexity, fast response.
- **Quarantine EC2 instance**
  - Good for malware or beaconing.
  - Requires careful network and evidence handling.
- **Block public S3 access**
  - Good for accidental exposure.
  - Often safe if the bucket should never be public.
- **Enable encryption**
  - Good for compliance drift.
  - May require coordination if data already exists.
- **Invoke SSM Automation**
  - Good for repeatable infrastructure fixes.
  - Better when the runbook is already defined.

**Practical design checklist**

- Match the exact event schema.
- Keep EventBridge rules narrow.
- Grant only the permissions needed.
- Make the remediation idempotent.
- Plan for retries, duplicates, and replay.
- Use DLQs or on-failure destinations where appropriate.
- Prefer aliases and versioned deployments for Lambda.
- Use cross-account roles deliberately.
- Log every action for audit and troubleshooting.
- Escalate to humans when the blast radius is high.
