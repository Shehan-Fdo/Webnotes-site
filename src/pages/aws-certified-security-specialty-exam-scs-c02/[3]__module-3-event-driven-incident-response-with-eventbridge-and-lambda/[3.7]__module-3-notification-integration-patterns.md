## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Why notification integration matters**

Security notifications are only useful when the right signal reaches the right responder quickly, with enough context to act. In AWS, that usually means combining event sources, routing services, filtering, and response targets into a design that balances speed, durability, and operational simplicity.

For exam scenarios, focus on three questions:

- Where does the event originate?
- How do you route and filter it?
- What target best matches the response workflow?

**Core AWS notification services**

AWS offers several services that overlap but solve different problems. The exam often tests whether you choose the simplest service that meets the requirement.

- **Amazon SNS** is a publish/subscribe service for fan-out notifications to multiple subscribers.
- **Amazon EventBridge** is an event routing service for matching structured events to targets.
- **Amazon SQS** is a queue for decoupling producers and consumers, not a notification broadcast service.
- **AWS Chatbot** delivers notifications into chat channels such as Slack and Microsoft Teams.
- **AWS Lambda** can transform, enrich, or respond to events.
- **AWS Systems Manager Incident Manager** and **OpsCenter** support operational response workflows.
- **CloudWatch Alarms** provide direct metric-based alerting from monitored resources.

A useful mental model is:

- Use **SNS** when you want simple fan-out and broad subscription support.
- Use **EventBridge** when you want content-based routing from many AWS services or SaaS partners.
- Use **SQS** when you need buffering, retry isolation, or a consumer pull model.
- Use **Lambda** when you need custom logic before notification or remediation.

**Amazon SNS patterns**

SNS is often the fastest way to broadcast a security alert to multiple endpoints. It works well for “tell everyone now” scenarios such as critical findings, operational alerts, or escalation messages.

Common SNS characteristics:

- **Fan-out** to multiple subscribers from one topic.
- **Topic policies** control who can publish and subscribe.
- **Subscription filter policies** reduce noise by matching message attributes.
- **At-least-once delivery** means duplicates are possible.
- **Ordering** is not guaranteed for standard topics.

A practical SNS topic setup for security alerts might look like this:

```bash
aws sns create-topic --name security-alerts

aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:security-alerts \
  --protocol email \
  --notification-endpoint security-team@example.com

aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:security-alerts \
  --protocol lambda \
  --notification-endpoint arn:aws:lambda:us-east-1:123456789012:function:ProcessSecurityAlert
```

Subscription filter policies are useful when one topic carries multiple severities or categories:

```json
{
  "severity": ["CRITICAL", "HIGH"],
  "service": ["guardduty", "securityhub"]
}
```

You attach the filter policy to the subscription, not the topic. This is a common exam detail.

> SNS dead-letter queue support is subscription-protocol dependent and not universal. Do not assume every endpoint type supports the same retry or DLQ behavior.

When to prefer SNS:

- You need simple broadcast to email, SMS, HTTP/S, Lambda, or other supported endpoints.
- You want multiple teams to receive the same alert.
- You want lightweight filtering on message attributes.

When SNS is not enough:

- You need rich event pattern matching on nested fields.
- You need to route based on event source, account, region, or detail-type.
- You need a central event bus for many AWS services.

**Amazon EventBridge patterns**

EventBridge is the primary AWS service for event-driven routing. It is especially important for security architectures because many AWS security services emit events that can be matched and routed without custom polling.

EventBridge concepts to know:

- **Event bus** is the entry point for events.
- **Rule** matches events using an event pattern.
- **Target** receives matched events.
- **Archive** stores events for replay.
- **Schema registry** helps discover and model event structure.

Event buses are a frequent exam topic:

- **Default bus** receives events from AWS services and your applications.
- **Custom bus** is used for application events, cross-account aggregation, or domain-specific routing.
- **Partner bus** receives events from SaaS partners that integrate with EventBridge.

For security use cases:

- Use the **default bus** for native AWS service events such as Security Hub, GuardDuty, or IAM-related service events where supported.
- Use a **custom bus** in a central security account when you want to aggregate events from multiple accounts or applications.
- Use a **partner bus** when a third-party security product publishes directly into EventBridge.

EventBridge event matching is structured and exact in important fields:

- `source` identifies the emitting service.
- `detail-type` identifies the event category.
- `account` helps scope events to a specific account.
- `region` helps scope events to a specific region.
- `detail` contains service-specific fields.

A rule for Security Hub findings might look like this:

```json
{
  "source": ["aws.securityhub"],
  "detail-type": ["Security Hub Findings - Imported"],
  "detail": {
    "findings": {
      "Severity": {
        "Label": ["CRITICAL", "HIGH"]
      },
      "RecordState": ["ACTIVE"]
    }
  }
}
```

This pattern is useful, but the exact event shape depends on the service and event type. For Security Hub, imported findings and custom action events are different event categories, so always match the correct `detail-type`.

A rule that routes only from a specific account and region can be useful in centralized designs:

```json
{
  "source": ["aws.guardduty"],
  "account": ["123456789012"],
  "region": ["us-east-1"]
}
```

EventBridge target examples:

- SNS topic for broad notification fan-out.
- Lambda function for enrichment or automated triage.
- SQS queue for buffering and downstream processing.
- Step Functions state machine for workflow orchestration.
- Systems Manager Incident Manager for incident response.
- Chatbot via SNS for chat-based alerting.

A simple EventBridge rule creation example:

```bash
aws events put-rule \
  --name high-severity-security-findings \
  --event-pattern file://pattern.json \
  --state ENABLED
```

Attach a target:

```bash
aws events put-targets \
  --rule high-severity-security-findings \
  --targets "Id"="1","Arn"="arn:aws:sns:us-east-1:123456789012:security-alerts"
```

EventBridge is usually the better choice when:

- You need content-based filtering on structured events.
- You want to route different event types to different targets.
- You need cross-account event aggregation.
- You want to integrate many AWS services without custom code.

> EventBridge delivery is at least once. Design downstream consumers to handle duplicates and idempotency.

**SNS vs EventBridge**

This comparison is frequently tested because both can trigger notifications, but they solve different problems.

| Capability | SNS | EventBridge |
|---|---|---|
| Primary model | Publish/subscribe | Event routing |
| Filtering | Subscription filter policies | Event patterns |
| Best for | Fan-out notifications | Structured event-driven workflows |
| Event structure awareness | Limited | Strong |
| Multiple targets | Yes | Yes |
| Cross-account patterns | Yes | Yes |
| Replay/archive | No native event archive | Archive and replay available |
| Typical security use | Alert distribution | Security event orchestration |

Use SNS when the message is already in the right shape and you mainly need distribution. Use EventBridge when you need to inspect event fields and route based on context.

A common exam pattern is:

- Security service emits an event.
- EventBridge rule matches severity or type.
- EventBridge sends to SNS for human notification.
- SNS fans out to email, Chatbot, or Lambda.

That pattern combines routing and distribution cleanly.

**CloudWatch Alarms as a direct notification source**

CloudWatch Alarms are a direct alerting mechanism based on metrics. They are not event routers like EventBridge, but they are often the right answer when the question is about threshold-based alerting.

Common alarm actions:

- Send notification to an SNS topic.
- Trigger Auto Scaling actions for scaling-related metrics.
- Invoke EC2 stop/terminate/recover actions in specific supported scenarios.
- Integrate with operational workflows through SNS and downstream targets.

Security-relevant examples:

- Alarm on `UnauthorizedOperations` or custom metric spikes from logs.
- Alarm on unusual API error rates.
- Alarm on network or application metrics that indicate abuse or outage.

A CloudWatch alarm example:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name HighUnauthorizedApiCalls \
  --metric-name UnauthorizedApiCalls \
  --namespace Security/Custom \
  --statistic Sum \
  --period 300 \
  --evaluation-periods 1 \
  --threshold 5 \
  --comparison-operator GreaterThanOrEqualToThreshold \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:security-alerts
```

CloudWatch Logs metric filters are often used to create custom metrics from log patterns. The filter syntax is not the same as EventBridge event patterns, so do not mix the two.

Example metric filter:

```bash
aws logs put-metric-filter \
  --log-group-name /aws/cloudtrail/security \
  --filter-name AccessDeniedCount \
  --filter-pattern '{ $.errorCode = "AccessDenied" || $.errorCode = "UnauthorizedOperation" }' \
  --metric-transformations metricName=UnauthorizedApiCalls,metricNamespace=Security/Custom,metricValue=1
```

Use CloudWatch Alarms when:

- You need threshold-based alerting.
- You already have a metric that represents the condition.
- You want a direct alarm action to SNS or another supported target.

Use EventBridge when:

- You need to react to discrete events rather than metric thresholds.
- You want to route based on event content and metadata.

**AWS Chatbot integration**

AWS Chatbot is useful when teams want alerts in chat channels without building custom webhook integrations. It commonly receives notifications from SNS topics.

Typical flow:

- EventBridge or CloudWatch Alarm publishes to SNS.
- SNS notifies AWS Chatbot.
- Chatbot posts into Slack or Microsoft Teams.

A basic Chatbot configuration uses an IAM role and a channel configuration. Example CLI pattern:

```bash
aws chatbot create-slack-channel-configuration \
  --configuration-name security-slack-channel \
  --iam-role-arn arn:aws:iam::123456789012:role/AWSChatbotRole \
  --slack-channel-id C0123456789 \
  --slack-workspace-id T0123456789 \
  --sns-topic-arns arn:aws:sns:us-east-1:123456789012:security-alerts
```

Key points:

- Chatbot is a notification delivery and command interface for chat tools.
- It is commonly used for security operations visibility.
- It is not a replacement for incident management or automated remediation.

**AWS Systems Manager Incident Manager and OpsCenter**

These services are common in security operations workflows and are worth recognizing in exam scenarios.

- **Incident Manager** helps coordinate response to incidents with escalation plans, contacts, and runbooks.
- **OpsCenter** centralizes operational issues as OpsItems for tracking and remediation.

Typical integrations:

- EventBridge rule routes critical findings to Incident Manager.
- Lambda creates an OpsItem for a security issue.
- SNS notifies responders while Incident Manager tracks the response lifecycle.

Use these when the question is not just “notify someone” but “manage the incident end to end.”

**Security Hub, GuardDuty, and centralized security workflows**

A common exam architecture is:

1. GuardDuty detects suspicious activity.
2. Security Hub aggregates and normalizes findings.
3. EventBridge routes findings based on severity or type.
4. SNS, Chatbot, Lambda, or Incident Manager handles notification and response.

This pattern is powerful because Security Hub acts as a central findings aggregator across multiple AWS security services and accounts.

Example architecture choices:

- **GuardDuty -> Security Hub -> EventBridge -> SNS** for human alerting.
- **GuardDuty -> Security Hub -> EventBridge -> Lambda** for enrichment or auto-remediation.
- **Security Hub -> EventBridge -> Incident Manager** for formal incident handling.
- **Security Hub -> EventBridge -> Chatbot** for team visibility.

Why this pattern is common:

- Security Hub normalizes findings from multiple sources.
- EventBridge provides flexible routing.
- SNS or Chatbot provides broad notification.
- Lambda or Step Functions provides automation.

**Cross-account and cross-region designs**

Security notification architectures often centralize in a dedicated security account. This is a common AWS Organizations pattern.

Typical centralized design:

- Member accounts generate findings.
- Findings are forwarded or aggregated into a central security account.
- The central account hosts EventBridge rules, SNS topics, and response workflows.
- Security teams receive notifications from the central account only.

Benefits:

- Centralized visibility.
- Simplified permissions and auditing.
- Easier incident response and reporting.
- Reduced duplication of notification logic across accounts.

Cross-account considerations:

- EventBridge rules and targets require resource policies or permissions that allow cross-account delivery.
- SNS topic policies must allow publishing from the source account or service principal.
- Lambda targets need invoke permissions.
- Chatbot and Incident Manager are usually configured in the central account.

Cross-region considerations:

- Many AWS services are regional.
- You may need separate regional event collection and a central aggregation layer.
- Use region-aware routing when the response team or compliance boundary depends on location.

A common multi-account pattern:

- Each workload account sends findings to a central EventBridge bus.
- The central bus routes to an SNS topic.
- SNS fans out to email, Chatbot, and a Lambda enrichment function.

**AWS Organizations and delegated administration**

AWS Organizations is often part of the answer when the question involves many accounts.

Relevant patterns:

- Enable security services organization-wide.
- Use delegated administration for services such as Security Hub or GuardDuty where applicable.
- Aggregate findings centrally.
- Apply consistent notification rules from the management or security account.

Exam takeaway:

- Organizations helps you scale security notification patterns across accounts.
- Delegated administration reduces the need to operate everything from the management account.
- Centralization improves governance, but you still need service-specific permissions and regional planning.

**SQS, Lambda, and workflow buffering**

SQS is not a notification service, but it is important in notification pipelines because it absorbs bursts and decouples producers from consumers.

Use SQS when:

- You need durable buffering.
- You want a worker model with controlled concurrency.
- You need to protect downstream systems from spikes.

A common pattern is:

- EventBridge sends matched events to SQS.
- Lambda polls SQS and processes messages.
- Lambda sends enriched notifications or remediation actions.

This is useful when you need retry isolation or when downstream processing may be slow.

Important tradeoffs:

- SQS is at-least-once, so duplicates can occur.
- FIFO queues provide ordering within a message group, but throughput and design constraints differ from standard queues.
- Lambda polling SQS is not the same as direct event push; it changes latency and failure handling.

**Delivery guarantees, ordering, and duplicates**

The exam often expects you to understand reliability tradeoffs.

- **SNS standard topics** provide at-least-once delivery and no ordering guarantee.
- **SQS standard queues** provide at-least-once delivery and no ordering guarantee.
- **SQS FIFO queues** provide ordering within a message group and deduplication features.
- **EventBridge** delivers events at least once and does not guarantee ordering.
- **Lambda** event sources vary; when triggered by SQS or EventBridge, duplicates and retries must be handled.

Practical implications:

- Design idempotent consumers.
- Use message IDs or finding IDs to deduplicate.
- Avoid assuming one alert equals one delivery.
- Preserve event context so responders can trace the original finding.

**Filtering and matching nuances**

Filtering is a major exam theme because it determines noise reduction and routing precision.

SNS filter policies match message attributes, not arbitrary nested JSON event fields in the same way EventBridge does.

EventBridge patterns match event fields such as:

- `source`
- `detail-type`
- `account`
- `region`
- nested `detail` values

This means:

- Use SNS filter policies for simple attribute-based fan-out.
- Use EventBridge for richer event structure matching.

A practical EventBridge pattern for high-severity findings:

```json
{
  "source": ["aws.securityhub"],
  "detail-type": ["Security Hub Findings - Imported"],
  "detail": {
    "findings": {
      "Severity": {
        "Label": ["CRITICAL", "HIGH"]
      }
    }
  }
}
```

A practical SNS filter policy for the same alert stream:

```json
{
  "severity": ["CRITICAL", "HIGH"]
}
```

The SNS message must include those attributes for the filter to work.

**Practical end-to-end architectures**

Common exam scenario 1: critical security finding to responders

1. GuardDuty generates a finding.
2. Security Hub aggregates the finding.
3. EventBridge matches `CRITICAL` severity.
4. SNS fans out to email and Chatbot.
5. Lambda optionally enriches the finding with account metadata.

Common exam scenario 2: automated incident creation

1. Security Hub emits a finding.
2. EventBridge rule matches a specific product or severity.
3. Step Functions starts a workflow.
4. The workflow creates an OpsItem or Incident Manager incident.
5. SNS notifies the on-call team.

Common exam scenario 3: centralized multi-account security monitoring

1. Member accounts send findings to a central security account.
2. Central EventBridge bus receives events.
3. Rules route by account, region, and severity.
4. SNS topic notifies the security operations team.
5. Lambda writes enriched records to a ticketing system or SIEM.

Common exam scenario 4: metric-based alerting from logs

1. CloudWatch Logs metric filter counts denied API calls.
2. CloudWatch Alarm triggers when the threshold is exceeded.
3. Alarm publishes to SNS.
4. SNS notifies Chatbot and email recipients.

**Configuration examples you should recognize**

SNS topic policy allowing a service or account to publish:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowEventBridgePublish",
      "Effect": "Allow",
      "Principal": {
        "Service": "events.amazonaws.com"
      },
      "Action": "sns:Publish",
      "Resource": "arn:aws:sns:us-east-1:123456789012:security-alerts"
    }
  ]
}
```

EventBridge target permission for Lambda:

```bash
aws lambda add-permission \
  --function-name ProcessSecurityAlert \
  --statement-id AllowEventBridgeInvoke \
  --action lambda:InvokeFunction \
  --principal events.amazonaws.com \
  --source-arn arn:aws:events:us-east-1:123456789012:rule/high-severity-security-findings
```

EventBridge rule targeting an SQS queue:

```bash
aws events put-targets \
  --rule high-severity-security-findings \
  --targets "Id"="1","Arn"="arn:aws:sqs:us-east-1:123456789012:security-queue"
```

CloudWatch alarm to SNS topic:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name RootLoginDetected \
  --metric-name RootLoginCount \
  --namespace Security/Custom \
  --statistic Sum \
  --period 300 \
  --evaluation-periods 1 \
  --threshold 1 \
  --comparison-operator GreaterThanOrEqualToThreshold \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:security-alerts
```

**Exam decision guide**

Choose the service based on the problem statement:

- Use **SNS** when the question emphasizes broadcast, subscriptions, or simple alert distribution.
- Use **EventBridge** when the question emphasizes event patterns, routing, or multiple AWS service integrations.
- Use **CloudWatch Alarms** when the question is threshold-based and metric-driven.
- Use **SQS** when the question emphasizes buffering, decoupling, or worker processing.
- Use **Chatbot** when the question asks for alerts in Slack or Teams.
- Use **Incident Manager** when the question asks for incident coordination and escalation.
- Use **OpsCenter** when the question asks for operational issue tracking.

**Common pitfalls**

- Confusing SNS filter policies with EventBridge event patterns.
- Assuming every target supports the same retry or DLQ behavior.
- Treating CloudWatch Logs metric filters like EventBridge JSON patterns.
- Assuming EventBridge or SNS guarantees exactly-once delivery.
- Using SQS when the requirement is immediate fan-out notification.
- Using SNS when the requirement is structured routing across many event types.
- Forgetting that cross-account and cross-region designs require explicit permissions and regional planning.

**High-value exam summary**

Security notification patterns in AWS usually combine one routing layer and one delivery layer:

- EventBridge routes structured events.
- SNS distributes notifications.
- CloudWatch Alarms detect threshold conditions.
- SQS buffers work.
- Lambda enriches or remediates.
- Chatbot, Incident Manager, and OpsCenter connect alerts to human response.

If you remember one thing, remember this:

- **EventBridge** is for event routing.
- **SNS** is for notification fan-out.
- **CloudWatch Alarms** are for metric thresholds.
- **SQS** is for durable buffering.
- **Chatbot** and **Incident Manager** turn alerts into operational action.
