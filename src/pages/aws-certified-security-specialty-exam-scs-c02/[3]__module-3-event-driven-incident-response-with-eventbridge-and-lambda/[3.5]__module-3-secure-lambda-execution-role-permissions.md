## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Lambda permission model overview**

AWS Lambda security has two distinct permission planes:

- **Execution role permissions** control what the function can do after it starts running.
- **Invocation permissions** control who or what can call the function.

This separation is a common exam theme. You secure Lambda correctly only when you design both sides together. A function may be invokable by a service, but still fail at runtime if its execution role lacks access to logs, KMS, VPC networking, or downstream services.

**Execution role vs resource-based policy**

Lambda uses an IAM role as the function’s **execution role**. The role is assumed by the Lambda service on behalf of your code. The role should contain only the permissions the code needs.

Lambda also supports a **resource-based policy** on the function itself. This policy grants other AWS principals or services permission to invoke the function. This is separate from the execution role.

- Use the **execution role** for runtime access such as CloudWatch Logs, Amazon S3, DynamoDB, Secrets Manager, KMS, and VPC networking.
- Use the **resource policy** for invocation access from services such as API Gateway, Application Load Balancer (ALB), Amazon EventBridge, Amazon Simple Notification Service (SNS), Amazon Simple Queue Service (SQS), and AWS Step Functions.
- Do not confuse the two: granting `lambda:InvokeFunction` in the execution role does not let another service call the function.

A secure design keeps the execution role narrow and the resource policy tightly scoped with source conditions.

**Invocation permissions and common sources**

Different invocation sources use different authorization patterns.

- **API Gateway** typically invokes Lambda through a resource policy statement that allows the API Gateway service principal and restricts the source ARN.
- **ALB** uses a Lambda target integration and requires a resource policy allowing the load balancer service to invoke the function.
- **Step Functions** can invoke Lambda directly from a task state, but the state machine’s execution role needs permission to call Lambda, and the Lambda resource policy may also be relevant depending on the integration pattern and account boundaries.
- **EventBridge**, **SNS**, and **S3** commonly rely on Lambda resource policies with `aws:SourceArn` and sometimes `aws:SourceAccount` to prevent confused deputy issues.

A good exam answer usually mentions both:
1. the service principal or caller identity, and
2. the condition keys that scope the permission.

Example Lambda resource policy statement for a service invocation:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowApiGatewayInvoke",
      "Effect": "Allow",
      "Principal": {
        "Service": "apigateway.amazonaws.com"
      },
      "Action": "lambda:InvokeFunction",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:orders-api",
      "Condition": {
        "ArnLike": {
          "AWS:SourceArn": "arn:aws:execute-api:us-east-1:123456789012:abc123/*/POST/orders"
        },
        "StringEquals": {
          "AWS:SourceAccount": "123456789012"
        }
      }
    }
  ]
}
```

**Function URLs and their separate authorization model**

Lambda function URLs are a distinct invocation path. They do not use API Gateway or ALB. Instead, the function URL itself has an authorization mode:

- **`AWS_IAM`**: callers must sign requests with SigV4 and have `lambda:InvokeFunctionUrl` permission.
- **`NONE`**: the URL is public; no IAM authentication is required.

This is a frequent exam trap because function URLs are not controlled the same way as standard Lambda invocation permissions.

Key points:

- `AWS_IAM` protects the URL with IAM authentication.
- `NONE` makes the endpoint publicly callable unless network controls or application logic add protection.
- Function URL permissions are separate from `lambda:InvokeFunction` used for standard invocation.
- You should still scope access with IAM policies and, where possible, condition keys such as `lambda:FunctionUrlAuthType`.

Example IAM policy for a caller using a function URL with IAM auth:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "lambda:InvokeFunctionUrl",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:orders-api",
      "Condition": {
        "StringEquals": {
          "lambda:FunctionUrlAuthType": "AWS_IAM"
        }
      }
    }
  ]
}
```

> If the function URL uses `NONE`, IAM does not protect the request path. Treat it as internet-facing unless another control blocks access.

**Least-privilege execution role design**

The execution role should contain only the permissions required by the function code and its runtime environment. Start with the minimum baseline, then add narrowly scoped permissions for each dependency.

Common baseline permissions:

- **CloudWatch Logs** for writing function logs.
- **KMS** for decrypting encrypted environment variables or application data.
- **VPC networking** if the function runs in a VPC.
- **Downstream service access** such as S3, DynamoDB, SQS, SNS, Secrets Manager, or Parameter Store.

A typical logging policy looks like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogGroup",
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "arn:aws:logs:us-east-1:123456789012:log-group:/aws/lambda/orders-api:*"
    }
  ]
}
```

In practice, many teams allow `logs:CreateLogGroup` on `*` because the log group may not exist at deployment time, but you should still scope the stream and event permissions as tightly as possible.

**KMS permissions for encrypted data and environment variables**

If you encrypt Lambda environment variables with a customer managed KMS key, the execution role needs permission to decrypt that key. The same applies when the function reads encrypted data from services that rely on KMS, such as Secrets Manager or S3 objects encrypted with SSE-KMS.

Important points:

- Use **`kms:Decrypt`** for runtime decryption.
- Add `kms:Encrypt` or `kms:GenerateDataKey` only if the function writes encrypted data or creates new encrypted artifacts.
- Scope the key policy and IAM policy together; KMS authorization requires both to allow the action.
- Prefer a dedicated CMK (customer managed key) for sensitive workloads so you can audit and rotate access more precisely.

Example execution role statement for decrypting a specific key:

```json
{
  "Effect": "Allow",
  "Action": [
    "kms:Decrypt"
  ],
  "Resource": "arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555"
}
```

If the function reads a secret from Secrets Manager, the role often needs both `secretsmanager:GetSecretValue` and `kms:Decrypt` for the key that protects the secret.

**VPC-related permissions and ENI management**

When a Lambda function is attached to a VPC, Lambda creates and manages elastic network interfaces (ENIs) in your subnets. The execution role must allow the Lambda service to perform the required EC2 network operations.

Common permissions include:

- `ec2:CreateNetworkInterface`
- `ec2:DescribeNetworkInterfaces`
- `ec2:DeleteNetworkInterface`
- `ec2:DescribeSubnets`
- `ec2:DescribeSecurityGroups`
- `ec2:DescribeVpcs`

These permissions are often attached through the AWS managed policy `AWSLambdaVPCAccessExecutionRole`, but exam questions may ask you to reason about what the policy does and why it is needed.

Example of a minimal custom VPC access statement:

```json
{
  "Effect": "Allow",
  "Action": [
    "ec2:CreateNetworkInterface",
    "ec2:DescribeNetworkInterfaces",
    "ec2:DeleteNetworkInterface",
    "ec2:DescribeSubnets",
    "ec2:DescribeSecurityGroups",
    "ec2:DescribeVpcs"
  ],
  "Resource": "*"
}
```

Why this matters:

- Without these permissions, the function may fail to attach to the VPC or scale correctly.
- VPC access is often needed for private RDS, ElastiCache, internal services, or private endpoints.
- Least privilege still applies, but EC2 network interface actions are commonly resource-scoped to `*` because the API does not always support fine-grained resource ARNs for these operations.

**Condition keys and policy scoping**

Condition keys help you reduce blast radius and prevent confused deputy problems.

Useful scoping patterns include:

- **`aws:SourceArn`** to restrict which specific resource can invoke the function.
- **`aws:SourceAccount`** to ensure the source belongs to the expected account.
- **`lambda:FunctionUrlAuthType`** to restrict function URL usage to `AWS_IAM` or `NONE`.
- **`lambda:SourceFunctionArn`** to scope permissions in scenarios where a function or service needs to call another Lambda and you want to tie access to a specific source function.

Use these conditions when a service integration supports them. They are especially important for event-driven architectures where multiple resources could otherwise invoke the same function.

Example of a tightly scoped S3 notification permission:

```json
{
  "Effect": "Allow",
  "Principal": {
    "Service": "s3.amazonaws.com"
  },
  "Action": "lambda:InvokeFunction",
  "Resource": "arn:aws:lambda:us-east-1:123456789012:function:image-processor",
  "Condition": {
    "StringEquals": {
      "AWS:SourceAccount": "123456789012"
    },
    "ArnLike": {
      "AWS:SourceArn": "arn:aws:s3:::my-upload-bucket"
    }
  }
}
```

**Common exam scenarios**

You should be able to identify which permission plane is missing from a broken design.

- A Lambda function cannot write logs:
  - The execution role is missing CloudWatch Logs permissions.
- A Lambda function in a VPC cannot reach a private database:
  - The execution role may be missing ENI permissions, or the subnet/security group routing is wrong.
- API Gateway receives `403` when invoking Lambda:
  - The Lambda resource policy may not allow the API Gateway service principal or the source ARN is too narrow.
- A function URL returns unauthorized:
  - The caller lacks `lambda:InvokeFunctionUrl`, or the URL is set to `AWS_IAM` and the request is unsigned.
- A function can be invoked from one account but not another:
  - The resource policy may need cross-account permissions and tighter source conditions.
- Secrets fail to decrypt at runtime:
  - The execution role or KMS key policy is missing `kms:Decrypt`.

**End-to-end secure Lambda setup example**

A realistic secure design includes both the execution role and the invocation policy.

Scenario:
- API Gateway invokes a Lambda function named `orders-api`.
- The function runs in a VPC to reach a private database.
- The function reads a secret from Secrets Manager.
- The secret is encrypted with a customer managed KMS key.
- The function writes logs to CloudWatch Logs.

Execution role policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "Logs",
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogGroup",
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "arn:aws:logs:us-east-1:123456789012:log-group:/aws/lambda/orders-api:*"
    },
    {
      "Sid": "VpcAccess",
      "Effect": "Allow",
      "Action": [
        "ec2:CreateNetworkInterface",
        "ec2:DescribeNetworkInterfaces",
        "ec2:DeleteNetworkInterface",
        "ec2:DescribeSubnets",
        "ec2:DescribeSecurityGroups",
        "ec2:DescribeVpcs"
      ],
      "Resource": "*"
    },
    {
      "Sid": "ReadSecret",
      "Effect": "Allow",
      "Action": [
        "secretsmanager:GetSecretValue"
      ],
      "Resource": "arn:aws:secretsmanager:us-east-1:123456789012:secret:orders/db-??????"
    },
    {
      "Sid": "DecryptSecret",
      "Effect": "Allow",
      "Action": [
        "kms:Decrypt"
      ],
      "Resource": "arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555"
    }
  ]
}
```

Lambda resource policy for API Gateway invocation:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowApiGatewayInvoke",
      "Effect": "Allow",
      "Principal": {
        "Service": "apigateway.amazonaws.com"
      },
      "Action": "lambda:InvokeFunction",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:orders-api",
      "Condition": {
        "ArnLike": {
          "AWS:SourceArn": "arn:aws:execute-api:us-east-1:123456789012:abc123/*/POST/orders"
        },
        "StringEquals": {
          "AWS:SourceAccount": "123456789012"
        }
      }
    }
  ]
}
```

This design is secure because:

- The function can only do the runtime actions it needs.
- API Gateway can invoke only the intended function and route.
- The source account and source ARN reduce the risk of unintended invocation.
- The KMS permission is limited to the specific key used for secrets.
- VPC access is granted only for the network operations Lambda requires.

**CLI commands you should know**

Use the AWS CLI to inspect and manage Lambda permissions.

View the function policy:

```bash
aws lambda get-policy --function-name orders-api
```

Add a resource-based permission for a service:

```bash
aws lambda add-permission \
  --function-name orders-api \
  --statement-id AllowApiGatewayInvoke \
  --action lambda:InvokeFunction \
  --principal apigateway.amazonaws.com \
  --source-arn arn:aws:execute-api:us-east-1:123456789012:abc123/*/POST/orders \
  --source-account 123456789012
```

Check the execution role attached to the function:

```bash
aws lambda get-function-configuration \
  --function-name orders-api \
  --query 'Role'
```

Update a function’s VPC configuration:

```bash
aws lambda update-function-configuration \
  --function-name orders-api \
  --vpc-config SubnetIds=subnet-11111111,subnet-22222222,SecurityGroupIds=sg-33333333
```

Invoke a function URL with IAM auth:

```bash
aws lambda invoke-function-url \
  --function-url https://abcde.lambda-url.us-east-1.on.aws/ \
  --auth-type AWS_IAM \
  --payload '{"orderId":"123"}' \
  response.json
```

**Troubleshooting and hardening tips**

- Prefer customer managed KMS keys when you need explicit control over decrypt permissions and auditability.
- Use separate roles for separate functions instead of reusing one broad role across many workloads.
- Review CloudTrail for `AccessDenied` events to identify whether the failure is in the execution role, KMS key policy, or Lambda resource policy.
- Use IAM Access Analyzer to detect unintended public or cross-account access in resource policies.
- Keep function URLs on `AWS_IAM` unless you intentionally need public access.
- Avoid wildcard principals and broad `SourceArn` patterns unless the service integration requires them.
- Remember that a successful invocation does not imply successful runtime access; both planes must be correct.

**Quick exam checklist**

- **Execution role**: runtime permissions for logs, KMS, VPC, and downstream services.
- **Resource policy**: who can invoke the function.
- **Function URL**: `AWS_IAM` or `NONE`, with separate authorization semantics.
- **VPC**: include ENI management permissions when attached to private subnets.
- **KMS**: include `kms:Decrypt` for encrypted environment variables and secrets.
- **Conditions**: use `aws:SourceArn`, `aws:SourceAccount`, and other relevant keys to scope access.
- **End-to-end design**: verify both invocation and runtime permissions in the same scenario.
