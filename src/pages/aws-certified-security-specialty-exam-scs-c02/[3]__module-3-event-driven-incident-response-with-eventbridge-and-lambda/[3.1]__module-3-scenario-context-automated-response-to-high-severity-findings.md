## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Scenario overview**

Automated response to high-severity findings is a common security operations pattern in AWS. You use event-driven services to detect a finding, enrich or classify it, notify responders, and trigger a containment action when the risk justifies it. The goal is to reduce mean time to respond while keeping the workflow auditable and least-privilege.

In exam scenarios, the key question is usually not “can you send an alert?” but “how do you safely automate the right response for the right finding source?” That means you must understand the event source, the event pattern, the remediation target, and the permissions boundary.

**Typical response pipeline**

A practical architecture often looks like this:

- **Detection source** emits a finding or compliance event.
- **Amazon EventBridge** matches the event and routes it.
- **AWS Lambda** or **AWS Step Functions** performs enrichment and decision logic.
- **Amazon SNS** or another notification channel alerts responders.
- **Containment action** changes network access, disables credentials, or isolates a workload.
- **Audit trail** records the action in CloudTrail, logs, and ticketing.

A simple mental model is: detect, decide, notify, contain, verify.

**Common event sources**

Different AWS services produce different security-relevant events, and the event pattern must match the source accurately.

- **Amazon GuardDuty**
  - Emits findings for suspicious activity such as credential exfiltration, reconnaissance, or malware.
  - EventBridge is a natural fit because GuardDuty findings are event-driven and near real time.
  - High-severity findings often trigger automated triage or containment.

- **AWS Security Hub**
  - Aggregates findings from AWS services and partner products.
  - EventBridge can route Security Hub findings based on severity, product name, compliance status, or workflow status.
  - Use it when you want a centralized security posture view and consistent remediation routing.

- **AWS Config**
  - Produces configuration and compliance change notifications.
  - Useful for drift detection, such as public S3 buckets, overly permissive security groups, or disabled encryption.
  - Often used for preventive or detective controls rather than incident containment.

- **Amazon Inspector**
  - Produces vulnerability findings for EC2 instances, container images, and Lambda functions.
  - Commonly used to trigger patching workflows, ticket creation, or quarantine of exposed workloads.

**EventBridge pattern design**

EventBridge rules should be specific enough to avoid noisy automation, but broad enough to catch the intended class of findings. Match on the fields that are stable for the source.

For example, a Security Hub rule often filters on the finding source and severity:

```json
{
  "source": ["aws.securityhub"],
  "detail-type": ["Security Hub Findings - Imported"],
  "detail": {
    "findings": {
      "Severity": {
        "Label": ["CRITICAL", "HIGH"]
      },
      "RecordState": ["ACTIVE"]
    }
  }
}
```

For GuardDuty, you typically filter on the GuardDuty source and severity label:

```json
{
  "source": ["aws.guardduty"],
  "detail-type": ["GuardDuty Finding"],
  "detail": {
    "severity": [7, 8, 9, 10]
  }
}
```

For AWS Config, the exact event shape depends on the rule and delivery path, so validate the event pattern against the actual event payload in your account before relying on automation. In exam terms, the important idea is that you match on the service source and the compliance or configuration state that indicates risk.

**Why EventBridge is the usual entry point**

EventBridge is the preferred integration layer because it is:

- **Serverless** and low maintenance.
- **Event-driven**, so it reacts quickly to new findings.
- **Fan-out capable**, allowing multiple targets such as Lambda, SNS, SQS, and Step Functions.
- **Filterable**, so you can route only high-severity or specific-resource findings.
- **Auditable**, because rules and targets are visible and manageable as infrastructure.

Use EventBridge when you need routing and filtering. Use Lambda or Step Functions when you need decision logic, branching, retries, or multi-step remediation.

**Notification patterns**

Notifications should be immediate but not overwhelming. A common pattern is to send a concise alert to responders and include enough context to triage quickly.

- **Amazon SNS**
  - Good for email, SMS, HTTPS, or fan-out to multiple subscribers.
  - Simple and fast for alerting.
  - Often used as the first notification target from Lambda or Step Functions.

- **AWS Chatbot**
  - Useful for Slack or Amazon Chime notifications.
  - Helps security teams receive alerts in collaboration channels.

- **Ticketing integration**
  - Often implemented through Lambda, API Gateway, or a third-party integration.
  - Useful when findings require human approval or tracking.

A notification should include:

- Finding ID or event ID
- Severity
- Resource identifier
- Account and Region
- Summary of the issue
- Link to the console or ticket

**Containment strategies**

Containment should reduce exposure without destroying evidence unless the scenario explicitly requires it. The best action depends on the resource type and the blast radius.

- **Security group isolation**
  - Replace the instance’s security group with a quarantine security group.
  - This is a common and exam-friendly containment pattern for EC2.
  - Prefer modifying the attached network interface when you need precise control.

- **Network ACL restriction**
  - Use a restrictive network ACL for subnet-level containment.
  - This is broader and can affect multiple workloads, so use carefully.

- **IAM credential revocation**
  - Disable or rotate access keys.
  - Detach risky policies or apply a permissions boundary.
  - Useful when the finding indicates credential compromise.

- **Instance stop or isolate**
  - Stop an instance if the business impact is acceptable.
  - Better for severe compromise when preserving the instance is less important than stopping spread.

- **Quarantine via automation**
  - Use a managed runbook or Lambda to move the workload into a restricted network posture.
  - Prefer AWS API-level controls over in-guest firewall changes for reliable containment.

> In exam scenarios, changing iptables inside the guest is usually not considered sufficient AWS-level containment. Prefer security groups, network ACLs, ENIs, or IAM controls.

**EC2 quarantine example with Lambda**

A common pattern is to move an instance’s primary network interface to a quarantine security group. The Lambda function needs permission to describe the instance, identify the network interface, and modify the interface security groups.

```python
import boto3

ec2 = boto3.client("ec2")

QUARANTINE_SG = "sg-0123456789abcdef0"

def lambda_handler(event, context):
    finding = event["detail"]["findings"][0]
    instance_id = finding["Resources"][0]["Id"]

    resp = ec2.describe_instances(InstanceIds=[instance_id])
    reservations = resp["Reservations"]

    if not reservations:
        raise Exception(f"Instance not found: {instance_id}")

    instance = reservations[0]["Instances"][0]
    eni_id = instance["NetworkInterfaces"][0]["NetworkInterfaceId"]

    ec2.modify_network_interface_attribute(
        NetworkInterfaceId=eni_id,
        Groups=[QUARANTINE_SG]
    )

    return {
        "status": "quarantined",
        "instance_id": instance_id,
        "network_interface_id": eni_id
    }
```

This pattern is useful because it changes the network boundary at the AWS control plane. It is more reliable than trying to alter the guest OS firewall, and it is easier to audit.

**Least-privilege IAM for remediation**

The remediation role should allow only the actions needed for the specific containment path. If you quarantine by modifying a network interface, grant the EC2 permissions required for describing instances and changing ENI security groups.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DescribeForQuarantine",
      "Effect": "Allow",
      "Action": [
        "ec2:DescribeInstances",
        "ec2:DescribeNetworkInterfaces"
      ],
      "Resource": "*"
    },
    {
      "Sid": "ModifySpecificNetworkInterfaces",
      "Effect": "Allow",
      "Action": [
        "ec2:ModifyNetworkInterfaceAttribute"
      ],
      "Resource": "*"
    }
  ]
}
```

If you also tag resources during remediation, scope the tagging permissions as tightly as possible.

```json
{
  "Sid": "TagRemediatedResources",
  "Effect": "Allow",
  "Action": [
    "ec2:CreateTags"
  ],
  "Resource": [
    "arn:aws:ec2:us-east-1:123456789012:instance/*",
    "arn:aws:ec2:us-east-1:123456789012:network-interface/*"
  ]
}
```

**Step Functions for multi-step remediation**

Use AWS Step Functions when the response needs branching, retries, approvals, or multiple actions. For example, you might notify first, wait for approval, then quarantine if the finding remains unresolved.

A valid pattern is to invoke Lambda from Step Functions, and let Lambda publish to SNS or update a ticket.

```json
{
  "Comment": "High severity finding response",
  "StartAt": "Notify",
  "States": {
    "Notify": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:NotifyFinding",
      "Next": "Decide"
    },
    "Decide": {
      "Type": "Choice",
      "Choices": [
        {
          "Variable": "$.severity",
          "NumericGreaterThanEquals": 8,
          "Next": "Quarantine"
        }
      ],
      "Default": "Archive"
    },
    "Quarantine": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:QuarantineInstance",
      "End": true
    },
    "Archive": {
      "Type": "Succeed"
    }
  }
}
```

This is preferable to trying to call SNS directly as a `Task` resource. Step Functions needs a supported service integration or a Lambda function.

**SSM Automation and Run Command**

AWS Systems Manager can help with response actions, but you should choose the right feature.

- **Run Command**
  - Executes commands on managed instances.
  - Good for in-guest actions such as collecting logs or stopping a service.
  - Not the best primary control for network quarantine.

- **Automation**
  - Orchestrates multi-step operational workflows.
  - Better for repeatable remediation such as snapshotting, tagging, isolating, or stopping instances.
  - Can call AWS APIs and integrate with approvals.

A more realistic quarantine automation might:

1. Create an EBS snapshot for evidence.
2. Tag the instance as quarantined.
3. Replace the security group on the primary ENI.
4. Notify the security team.
5. Open a ticket.

**Cross-account response considerations**

In organizations with multiple accounts, findings may arrive in a central security account while the affected resource lives in a workload account. In that case, the remediation role must trust the central automation principal and should include protections against confused deputy problems.

A safer trust policy often includes conditions such as `aws:PrincipalArn`, `aws:SourceArn`, or an external ID pattern where appropriate.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCentralAutomation",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/SecurityAutomationRole"
      },
      "Action": "sts:AssumeRole",
      "Condition": {
        "StringEquals": {
          "aws:PrincipalArn": "arn:aws:iam::111122223333:role/SecurityAutomationRole"
        }
      }
    }
  ]
}
```

In practice, you may also use AWS Organizations SCPs, permission boundaries, or service control patterns to limit what the assumed role can do after it is trusted.

**Operational safeguards**

Automation should not blindly quarantine every finding. Add guardrails so the system behaves predictably.

- **Severity threshold**
  - Trigger containment only for `HIGH` or `CRITICAL` findings.
  - Use lower severities for notification only.

- **Resource allowlist**
  - Exempt critical production systems if they require manual approval.
  - Route them to a human review path instead.

- **Idempotency**
  - Ensure repeated events do not repeatedly quarantine the same resource.
  - Use tags, DynamoDB state, or finding IDs to track processed events.

- **Approval workflow**
  - Require a human approval step for destructive actions.
  - Step Functions is a strong fit here.

- **Evidence preservation**
  - Snapshot volumes, preserve logs, and record timestamps before making changes.
  - This matters for forensics and incident response.

**Troubleshooting and validation**

Before relying on automation in production, test the full path with sample events and controlled findings.

- Verify the EventBridge rule matches the actual event payload.
- Confirm the Lambda execution role can read the finding and modify the target resource.
- Check that the quarantine security group is truly restrictive.
- Confirm notifications arrive with enough context to act.
- Review CloudTrail to ensure the remediation action is logged.

Useful CLI commands:

```bash
aws events test-event-pattern \
  --event-pattern file://pattern.json \
  --event file://sample-event.json
```

```bash
aws ec2 describe-instances \
  --instance-ids i-0123456789abcdef0
```

```bash
aws ec2 describe-network-interfaces \
  --filters Name=attachment.instance-id,Values=i-0123456789abcdef0
```

```bash
aws sns publish \
  --topic-arn arn:aws:sns:us-east-1:123456789012:SecurityAlerts \
  --subject "High severity finding" \
  --message "Instance i-0123456789abcdef0 has been quarantined."
```

**Exam focus**

For SCS-C02, expect questions that ask you to choose the best automated response design for a finding source and severity level.

- Use **EventBridge** for event routing and filtering.
- Use **Lambda** for lightweight decision logic and API calls.
- Use **Step Functions** for multi-step workflows and approvals.
- Use **SNS** for notifications.
- Use **security groups, ENIs, IAM, or NACLs** for actual containment.
- Prefer AWS control plane actions over guest OS-only changes.
- Preserve evidence when the scenario mentions incident response or forensics.

**Quick decision guide**

- If the question asks for alerting only, choose SNS or Chatbot.
- If the question asks for automatic containment, choose Lambda or Step Functions plus AWS API-based isolation.
- If the question asks for multi-account remediation, include cross-account roles and trust controls.
- If the question asks for repeatable operational remediation, consider Systems Manager Automation.
- If the question asks for event-driven filtering, choose EventBridge.

**Key takeaway**

The best automated response design is the one that matches the finding source, limits blast radius, and uses AWS-native controls for containment. In exam scenarios, the safest answer is usually an event-driven pipeline with EventBridge at the front, Lambda or Step Functions for logic, SNS for notification, and network or IAM controls for quarantine.
