## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Overview**

VPC Flow Logs capture metadata about IP traffic to and from network interfaces in a <abbr title="Amazon Virtual Private Cloud">VPC</abbr>. You use them for security investigations, threat hunting, and incident response when you need to answer questions such as which source IP contacted a workload, whether traffic was accepted or rejected, and which ports were involved.

Athena is a good fit when Flow Logs are delivered to Amazon S3 because you can query large volumes of logs with SQL without loading data into a database. This makes Athena useful for ad hoc investigations, periodic reporting, and cross-account analysis. It is not a streaming detector; it is a query engine over data already delivered to S3.

**What VPC Flow Logs record**

Flow Logs are metadata, not packet captures. They do not show payloads, application-layer content, or full session reconstruction. Instead, they record fields that help you reason about network behavior.

- **Source and destination** information such as IP addresses and ports
- **Protocol** and packet/byte counts
- **Action** values such as `ACCEPT` or `REJECT`
- **Log status** values that explain whether a record was delivered successfully
- Optional fields depending on the log format and version, such as VPC, subnet, instance, traffic path, and packet-level addresses

A single flow log record usually represents an aggregation window for traffic sharing common attributes. For many use cases, this is enough to identify scanning, lateral movement, denied access, or unusual egress patterns.

**Delivery destinations and why they matter**

You can send Flow Logs to CloudWatch Logs or Amazon S3. The destination changes how you investigate.

- **CloudWatch Logs**
  - Better for near-real-time visibility and log metric filters
  - Useful when you want alarms or quick operational triage
  - Less convenient for large-scale SQL analysis than Athena
- **Amazon S3**
  - Best for Athena queries and long-term retention
  - Better for large historical investigations and joins with other datasets
  - Requires careful table design and partition strategy for performance

> Athena queries S3 objects; it does not query CloudWatch Logs directly. If you need SQL over CloudWatch Logs, you typically export or route data into S3 first.

**Core concepts for exam questions**

- **Aggregation interval** controls how often records are summarized.
  - For many VPC Flow Logs configurations, the interval is `60` seconds or `600` seconds.
  - Some newer resource types or configurations may support only `60` seconds.
  - Do not assume every log source supports both values.
- **Log status** is not the same as traffic outcome.
  - `OK` means the record was delivered successfully.
  - `NODATA` means no network traffic was recorded for that interface during the aggregation interval.
  - `SKIPDATA` means some records were skipped because of internal capacity or processing limits.
- **Accepted vs rejected traffic**
  - `ACCEPT` indicates the traffic matched a security group or network ACL path that allowed it.
  - `REJECT` indicates the traffic was denied by a network control.
  - A `REJECT` record is often the first clue for blocked reconnaissance or misconfigured applications.

**S3 delivery setup**

To query Flow Logs in Athena, you first need the logs in S3. The command creates the flow log, but S3 delivery also requires the bucket policy and correct destination ARN.

```bash
aws ec2 create-flow-logs \
  --resource-type VPC \
  --resource-ids vpc-0123456789abcdef0 \
  --traffic-type ALL \
  --log-destination-type s3 \
  --log-destination arn:aws:s3:::my-flow-log-bucket \
  --deliver-logs-permission-arn arn:aws:iam::123456789012:role/FlowLogsDeliveryRole
```

For S3 delivery, make sure the bucket policy allows the Flow Logs service to write objects. A typical policy grants `s3:PutObject` to the service principal for the bucket prefix you use.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AWSLogDeliveryWrite",
      "Effect": "Allow",
      "Principal": {
        "Service": "delivery.logs.amazonaws.com"
      },
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::my-flow-log-bucket/AWSLogs/123456789012/*",
      "Condition": {
        "StringEquals": {
          "s3:x-amz-acl": "bucket-owner-full-control",
          "aws:SourceAccount": "123456789012"
        },
        "ArnLike": {
          "aws:SourceArn": "arn:aws:logs:us-east-1:123456789012:*"
        }
      }
    }
  ]
}
```

The exact policy can vary by region, account, and delivery path, but the key idea is consistent: the service must be allowed to write to the bucket.

**Table design in Athena**

Athena works best when the table schema matches the actual log format. VPC Flow Logs have evolved over time, and the field set depends on the version and selected format. Do not assume a minimal schema if your logs include additional fields.

A practical approach is to define a table that matches the fields you expect from your log format and then validate with sample records.

```sql
CREATE EXTERNAL TABLE IF NOT EXISTS vpc_flow_logs (
  version int,
  account_id string,
  interface_id string,
  srcaddr string,
  dstaddr string,
  srcport int,
  dstport int,
  protocol int,
  packets bigint,
  bytes bigint,
  start bigint,
  end bigint,
  action string,
  log_status string
)
PARTITIONED BY (
  region string,
  dt string
)
ROW FORMAT DELIMITED
FIELDS TERMINATED BY ' '
LOCATION 's3://my-flow-log-bucket/AWSLogs/123456789012/vpcflowlogs/'
TBLPROPERTIES ('skip.header.line.count'='0');
```

This example is intentionally simple. In real environments, you may need a wider schema that includes fields such as `vpc_id`, `subnet_id`, `instance_id`, `tcp_flags`, `type`, `pkt_srcaddr`, `pkt_dstaddr`, `flow_direction`, and `traffic_path`.

> Do not hardcode a partition layout unless you control the S3 prefix structure. Flow Logs in S3 do not have one universal canonical folder pattern across all setups.

**Partitioning and performance**

Athena charges by data scanned, so partitioning matters. The most common partitioning strategy is by date and sometimes region. If your S3 layout supports it, partition projection can reduce the need to run `MSCK REPAIR TABLE` repeatedly.

- **Partition by date** to limit scans for incident windows
- **Partition by region** if logs from multiple regions land in the same bucket
- **Use projection** when you want Athena to infer partitions from predictable naming patterns
- **Avoid full-bucket scans** for routine investigations

Example partition repair:

```sql
MSCK REPAIR TABLE vpc_flow_logs;
```

Example partition projection properties:

```sql
ALTER TABLE vpc_flow_logs SET TBLPROPERTIES (
  'projection.enabled'='true',
  'projection.dt.type'='date',
  'projection.dt.range'='2025/01/01,NOW',
  'projection.dt.format'='yyyy/MM/dd',
  'storage.location.template'='s3://my-flow-log-bucket/AWSLogs/123456789012/vpcflowlogs/${region}/${dt}/'
);
```

Use projection only when the S3 layout is predictable and under your control. If the prefix structure differs, adjust the template accordingly.

**Common Athena queries for investigations**

Use SQL to answer security questions quickly. Start with a narrow time window and then expand if needed.

Find rejected traffic to a specific host:

```sql
SELECT srcaddr, srcport, dstaddr, dstport, protocol, action, log_status, start, end
FROM vpc_flow_logs
WHERE dstaddr = '10.0.2.15'
  AND action = 'REJECT'
ORDER BY start DESC
LIMIT 100;
```

Identify top talkers by bytes:

```sql
SELECT srcaddr, dstaddr, SUM(bytes) AS total_bytes
FROM vpc_flow_logs
WHERE dt = '2026/04/26'
  AND action = 'ACCEPT'
GROUP BY srcaddr, dstaddr
ORDER BY total_bytes DESC
LIMIT 20;
```

Look for repeated denied connections to common ports:

```sql
SELECT srcaddr, dstport, COUNT(*) AS rejects
FROM vpc_flow_logs
WHERE action = 'REJECT'
  AND dstport IN (22, 3389, 445, 1433, 3306)
GROUP BY srcaddr, dstport
ORDER BY rejects DESC;
```

Detect possible scanning behavior by counting unique destination ports:

```sql
SELECT srcaddr, COUNT(DISTINCT dstport) AS unique_ports
FROM vpc_flow_logs
WHERE action = 'REJECT'
  AND dt = '2026/04/26'
GROUP BY srcaddr
HAVING COUNT(DISTINCT dstport) > 20
ORDER BY unique_ports DESC;
```

Find traffic from a suspicious source IP:

```sql
SELECT *
FROM vpc_flow_logs
WHERE srcaddr = '198.51.100.10'
  AND dt BETWEEN '2026/04/25' AND '2026/04/26'
ORDER BY start DESC;
```

**Interpreting results during an investigation**

Athena results are only as good as your schema and filters. Always validate the time window, partition values, and field meanings before drawing conclusions.

- A spike in `REJECT` records can indicate:
  - Port scanning
  - Misconfigured security groups or network ACLs
  - A workload trying to reach a blocked service
- A spike in `ACCEPT` records can indicate:
  - Normal batch processing
  - Data exfiltration
  - Lateral movement
  - A noisy application or load balancer
- `NODATA` should be read carefully:
  - It means no traffic was recorded for that interface during the interval
  - It does not prove that no traffic existed anywhere else in the environment
- `SKIPDATA` deserves attention:
  - It can indicate incomplete visibility
  - It may require checking log volume, delivery health, or resource limits

**Practical investigation workflow**

1. Confirm the log source, destination, and time range.
2. Verify the Athena table schema matches the actual Flow Logs format.
3. Restrict the query to the smallest useful partition set.
4. Search for `REJECT` first when investigating blocked access or scanning.
5. Expand to `ACCEPT` traffic when you need context around a suspicious host.
6. Correlate with security groups, network ACLs, Route 53 Resolver logs, GuardDuty findings, or instance logs.
7. Export query results if you need to preserve evidence or share with responders.

**Example incident pattern**

Suppose you see repeated `REJECT` traffic from one source IP to many destination ports on a single instance. That pattern can suggest a scan or a misconfigured client repeatedly retrying blocked connections.

A useful follow-up query is to group by source, destination, and port range:

```sql
SELECT srcaddr,
       dstaddr,
       dstport,
       COUNT(*) AS attempts,
       MIN(start) AS first_seen,
       MAX(end) AS last_seen
FROM vpc_flow_logs
WHERE action = 'REJECT'
  AND dt = '2026/04/26'
GROUP BY srcaddr, dstaddr, dstport
ORDER BY attempts DESC;
```

Then compare the findings with the instance’s security group rules and network ACL entries. If the traffic is expected, you may need to adjust controls. If it is unexpected, you may need to isolate the host or investigate the source.

**Operational tips**

- Use a dedicated Athena workgroup for security investigations.
- Set query result encryption and an S3 results bucket.
- Restrict access to the log bucket and query results with least privilege.
- Use lifecycle policies to manage retention costs.
- Consider Glue Data Catalog permissions if multiple teams query the same logs.
- Validate timestamps carefully because Flow Logs use epoch values in many schemas, while Athena queries often filter by partition date strings.

**Security and governance considerations**

Flow Logs can contain sensitive network metadata. Even without payloads, they reveal internal IP ranges, instance behavior, and communication patterns.

- Protect the S3 bucket with encryption at rest, preferably <abbr title="Server-Side Encryption with AWS Key Management Service">SSE-KMS</abbr> when governance requires key control
- Restrict who can query the data in Athena
- Use separate buckets or prefixes for different environments when possible
- Treat query results as sensitive evidence
- Preserve logs according to your incident response and compliance requirements

**Exam focus**

For AWS Certified Security – Specialty, expect questions that test whether you can choose the right logging destination and analysis tool.

- Choose **Athena** when you need SQL analysis over Flow Logs stored in S3
- Choose **CloudWatch Logs** when you need near-real-time operational visibility
- Remember that Athena depends on S3 delivery and table definition
- Remember that Flow Logs are metadata, not packet payloads
- Remember that `REJECT`, `NODATA`, and `SKIPDATA` have distinct meanings
