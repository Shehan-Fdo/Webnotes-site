## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Recovery objectives and operating model**

Recovery is not just “bring systems back.” In AWS incident response, you restore service while preserving evidence, preventing reinfection, and controlling risk. The recovery phase starts after containment and eradication decisions are in motion, and it should be governed by a clear approval process before production traffic returns.

A strong recovery plan defines:
- **Recovery priorities** by business impact, not by technical convenience.
- **Decision gates** for when a workload can re-enter production.
- **Communication paths** for stakeholders, security, operations, legal, and business owners.
- **Rollback criteria** if validation fails or suspicious activity reappears.

Use change control even during emergencies. Emergency changes should still be recorded, approved by the incident commander or delegated authority, and reviewed afterward. This matters in exam scenarios where “fast” recovery is not enough if it bypasses governance.

**Incident governance and communications**

Recovery should include explicit stakeholder notification and approval gates. You typically coordinate:
- Incident commander and technical leads
- Application owners and service owners
- Security operations and detection teams
- Compliance, legal, and privacy stakeholders when data exposure is possible
- Business stakeholders who approve service restoration

A practical recovery workflow often looks like this:
1. Confirm containment and scope.
2. Validate that the recovery source is clean.
3. Restore into an isolated or controlled environment first.
4. Test functionality, security controls, and logging.
5. Obtain approval to reintroduce traffic.
6. Monitor closely after cutover.

> Do not return a workload to production until you have validated the restore source, the configuration, and the monitoring path. A fast restore from a compromised snapshot can reintroduce the incident.

**Recovery sources and integrity validation**

Backups, snapshots, AMIs, container images, and artifacts are only useful if they are trustworthy. Before using them, verify that they are:
- **Recent enough** to meet recovery point objectives.
- **Complete** and not corrupted.
- **Clean** of malicious changes or persistence mechanisms.
- **Compatible** with the current target environment.

For AWS backups and snapshots, test restores regularly. Restore testing is how you prove that recovery actually works, not just that backups exist. In exam terms, “backup success” is not the same as “restore success.”

Useful validation patterns include:
- Restoring to a quarantined VPC or isolated account.
- Comparing file hashes or package manifests against known-good baselines.
- Checking startup logs, application health checks, and security telemetry.
- Verifying that IAM roles, security groups, and network ACLs match the intended hardened state.

For EBS-backed workloads, you can create a snapshot and restore a volume for validation:
```bash
aws ec2 create-snapshot \
  --volume-id vol-0123456789abcdef0 \
  --description "Pre-incident recovery snapshot"

aws ec2 create-volume \
  --availability-zone us-east-1a \
  --snapshot-id snap-0123456789abcdef0 \
  --volume-type gp3
```

For S3, versioning helps recovery, but you must account for delete markers and version state. A “restore” may mean copying a previous version into place, removing a delete marker, or re-pointing consumers to the correct object version.

```bash
aws s3api list-object-versions \
  --bucket my-bucket \
  --prefix app/config.json

aws s3api copy-object \
  --bucket my-bucket \
  --copy-source "my-bucket/app/config.json?versionId=3HL4kqtJlcpXroDTDmjVBH40Nrjfkd" \
  --key app/config.json
```

**Multi-account and AWS Organizations recovery**

Large environments usually recover across multiple accounts. Plan for:
- A **management account** that governs Organizations and SCPs.
- A **log archive account** that stores immutable or tightly controlled logs.
- A **security tooling account** for GuardDuty, Security Hub, detective controls, and automation.
- Separate **workload accounts** for blast-radius reduction.

Recovery coordination across accounts often requires:
- Delegated administration for security services.
- Cross-account roles for incident responders.
- Centralized logging that remains available even if a workload account is compromised.
- Careful handling of SCPs so you do not block your own recovery actions.

If an SCP blocks recovery, you may need a controlled exception path. Use temporary, narrowly scoped permissions rather than broad permanent relaxations. The same applies to permission boundaries and session policies.

**Restoring IAM access safely**

IAM recovery is often the hardest part of an incident because you need access to fix access. Restore access in a controlled way:
- Use a **break-glass role** with strong controls, MFA, and monitoring.
- Keep emergency access separate from day-to-day admin access.
- Review and temporarily relax SCPs only when necessary.
- Check permission boundaries, session policies, and trust policies before assuming a role.

Important nuance:
- Changing a role trust policy does **not** revoke already-issued STS sessions.
- Existing sessions usually remain valid until they expire unless you use a supported revocation mechanism or invalidate the underlying credentials.
- If you suspect credential compromise, prefer disabling or rotating the source credentials, reducing permissions, and forcing reauthentication where possible.

Example break-glass role trust policy pattern:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "EmergencyAccess",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:root"
      },
      "Action": "sts:AssumeRole",
      "Condition": {
        "Bool": {
          "aws:MultiFactorAuthPresent": "true"
        }
      }
    }
  ]
}
```

At scale, validate:
- Which roles are still needed.
- Whether access keys should be deactivated.
- Whether federated access through IAM Identity Center needs reconfiguration.
- Whether temporary elevated access is logged and time-bound.

**Shared services recovery**

Some services are shared across many workloads and should be recovered early because they affect visibility and control.

Key shared services include:
- **CloudTrail** for audit logging
- **AWS Config** for configuration history and compliance state
- **AWS KMS** for encryption key availability and access control
- **IAM Identity Center** for workforce access
- **Directory services** or identity providers used by applications
- Centralized DNS and routing services

For CloudTrail, ensure trails are still delivering logs to the intended log archive account and that log file validation remains enabled where used. For organization-wide coverage, manage organization trails centrally rather than treating them like ordinary single-account trails.

For KMS, recovery usually means ensuring keys are enabled, policies are correct, grants are valid, and dependent services can still decrypt data. If a key is suspected compromised, do not rely on rotation alone as a security fix. Instead:
- Restrict access immediately.
- Create a new key if needed.
- Re-encrypt sensitive data under the new key.
- Review all grants, key policies, and usage logs.

**Automation for repeatable recovery**

Use automation to reduce manual error and speed up repeatable tasks. AWS Systems Manager is especially useful:
- **Automation** for multi-step recovery runbooks.
- **Run Command** for targeted remediation on managed instances.
- **Incident Manager** for coordinating response workflows and escalation.

Example Systems Manager Automation document pattern:
```yaml
schemaVersion: '0.3'
description: "Quarantine an EC2 instance during recovery"
assumeRole: arn:aws:iam::111122223333:role/SSMAutomationRole
mainSteps:
  - name: isolateInstance
    action: aws:executeAwsApi
    inputs:
      Service: ec2
      Api: ModifyInstanceAttribute
      InstanceId: i-0123456789abcdef0
      Groups:
        - sg-0abc123def4567890
```

Run Command is useful when you need to:
- Stop suspicious processes.
- Collect triage data.
- Apply configuration fixes across a fleet.
- Verify package versions or file integrity.

Example:
```bash
aws ssm send-command \
  --document-name "AWS-RunShellScript" \
  --targets "Key=InstanceIds,Values=i-0123456789abcdef0" \
  --parameters 'commands=["sudo systemctl restart app","sudo journalctl -u app --since \"1 hour ago\""]'
```

Incident Manager helps you standardize escalation, paging, and response timelines. In exam scenarios, automation is preferred when the action is repeatable, auditable, and safe to codify.

**Detection-driven containment and hardening**

Recovery often begins with signals from detective services. Use findings to trigger containment and hardening actions.

Common patterns:
- **GuardDuty** finding triggers an EventBridge rule.
- EventBridge invokes a Lambda function or Systems Manager Automation document.
- The automation quarantines the instance, disables a key, or revokes a risky permission path.
- Security Hub aggregates findings and tracks response status.

Example EventBridge rule target pattern:
```json
{
  "source": ["aws.guardduty"],
  "detail-type": ["GuardDuty Finding"],
  "detail": {
    "severity": [{ "numeric": [">=", 7] }]
  }
}
```

Practical automated actions include:
- Moving an EC2 instance to a quarantine security group.
- Detaching an instance profile if compromise is suspected.
- Disabling a compromised access key.
- Adding a temporary deny statement through a managed policy or SCP.
- Opening an incident ticket and paging responders.

AWS Config can also drive remediation. For example, if an S3 bucket becomes public, a Config rule can detect the drift and an SSM Automation document can restore the intended bucket policy or block public access settings.

**Network recovery and controlled traffic shifting**

Do not restore all traffic at once unless the blast radius is small and the service is low risk. Controlled traffic shifting reduces the chance of reintroducing a bad build or hidden compromise.

Common patterns:
- **Blue/green** deployment: bring up a clean environment, validate it, then switch traffic.
- **Canary** release: send a small percentage of traffic to the recovered environment first.
- **Weighted routing** in Route 53: gradually increase traffic to the restored target.
- **Health-check-based failover** for active/passive designs.

Route 53 weighted routing example:
```text
Record A -> blue environment weight 90
Record A -> green environment weight 10
```

Then increase the recovered environment gradually after validation. This is especially useful when:
- You need to compare behavior under real traffic.
- You want to monitor for latent errors.
- You need a fast rollback path.

For DNS recovery, verify:
- Hosted zone changes are correct.
- TTL values are appropriate for the cutover.
- Health checks reflect the real service state.
- CNAME or alias targets point to the intended load balancer or endpoint.

**Container and serverless hardening after recovery**

Container and serverless workloads need post-incident hardening beyond simply redeploying the same artifacts.

For containers:
- Rebuild images from trusted source code and dependencies.
- Scan images for vulnerabilities before deployment.
- Review ECS task roles, execution roles, and task definitions.
- Check EKS cluster access, RBAC, and admission controls.
- Consider image signing and verification.

Example image scan and deployment gate:
```bash
aws ecr describe-image-scan-findings \
  --repository-name my-app \
  --image-id imageTag=prod
```

For EKS, review:
- Cluster role mappings
- Namespace isolation
- Network policies
- Pod security controls
- Access to the container registry and secrets

For Lambda:
- Review function code, layers, environment variables, and resource-based policies.
- Recreate functions from trusted artifacts if tampering is suspected.
- Check execution roles for overbroad permissions.
- Validate event source mappings and triggers.

For both containers and serverless:
- Reissue secrets if exposure is possible.
- Rotate or replace credentials used by the workload.
- Confirm that logging and tracing are enabled before reactivation.

**Infrastructure and configuration hardening**

After recovery, assume the environment may still contain weak points. Harden the baseline before full production return.

Typical hardening actions:
- Remove unnecessary inbound access.
- Tighten security groups and network ACLs.
- Enforce least privilege on IAM roles and policies.
- Reapply secure parameter values and configuration baselines.
- Patch vulnerable packages and AMIs.
- Rebuild from golden images rather than “cleaning” a suspect host when feasible.

AWS Config is useful for detecting drift from approved baselines. Security Hub helps prioritize findings across services. Use both to confirm that the recovered environment matches your intended posture.

Example of a simple least-privilege review checklist:
- Does the role still need `*` actions?
- Are resource ARNs narrowed?
- Are conditions used for MFA, source IP, or VPC endpoint restrictions?
- Are temporary permissions time-bound and documented?

**Validation before production re-entry**

Validation should be both functional and security-focused. A service that “works” may still be unsafe.

Validate:
- Application health checks and synthetic transactions
- Authentication and authorization flows
- Logging to CloudWatch Logs, S3, or a SIEM
- Metrics and alarms
- Backup and restore paths
- Security controls such as WAF, GuardDuty, Config, and KMS access

A practical validation sequence:
1. Restore into a staging or isolated environment.
2. Run smoke tests and security checks.
3. Confirm logs and alerts are flowing.
4. Verify that no unexpected outbound connections occur.
5. Approve controlled traffic shift.
6. Monitor for anomalies after cutover.

> If validation fails, stop the cutover and roll back. Do not “fix forward” in production unless the change is low risk and explicitly approved.

**Post-incident hardening and lessons learned**

The recovery phase should end with durable improvements. Feed root cause analysis into preventive controls so the same incident is less likely to recur.

Common post-incident actions:
- Update the incident response plan and runbooks.
- Add or tune detections in GuardDuty, Security Hub, CloudTrail, and Config.
- Improve backup frequency, retention, and restore testing.
- Tighten IAM policies, SCPs, and permission boundaries.
- Add automation for quarantine, evidence collection, and rollback.
- Improve patching, image scanning, and artifact signing.
- Review third-party integrations and trust relationships.

Document:
- What happened
- What was restored
- What was changed during recovery
- Which controls failed or were missing
- Which approvals were required
- Which actions should be automated next time

A lessons-learned review should produce concrete outputs:
- Updated runbooks
- New detection rules
- Revised access model
- Better backup and restore tests
- A tracked remediation backlog with owners and due dates

**Practical recovery examples**

Example: isolate a compromised EC2 instance and preserve evidence
```bash
aws ec2 modify-instance-attribute \
  --instance-id i-0123456789abcdef0 \
  --groups sg-quarantine-only
```

Example: disable a suspected compromised access key
```bash
aws iam update-access-key \
  --user-name suspicious-user \
  --access-key-id AKIAEXAMPLE123456 \
  --status Inactive
```

Example: inspect AWS Config compliance for a resource before reintroducing it
```bash
aws configservice get-compliance-details-by-resource \
  --resource-type AWS::EC2::Instance \
  --resource-id i-0123456789abcdef0
```

Example: use Route 53 weighted routing for gradual cutover
```text
blue.example.com  -> 90%
green.example.com -> 10%
```

Example: trigger a Systems Manager Automation document from a finding
```bash
aws ssm start-automation-execution \
  --document-name "QuarantineInstance" \
  --parameters InstanceId=i-0123456789abcdef0
```

**Exam focus reminders**

- Recovery is a controlled process, not a simple restore.
- Validate backup integrity and restore behavior before production use.
- Use governance, approvals, and communication gates.
- Prefer isolated validation and gradual traffic shifting.
- Restore IAM access carefully with break-glass patterns and least privilege.
- Use automation and detective services to standardize containment and hardening.
- Rebuild from trusted sources when compromise is suspected.
- Update controls, runbooks, and detections after the incident.
