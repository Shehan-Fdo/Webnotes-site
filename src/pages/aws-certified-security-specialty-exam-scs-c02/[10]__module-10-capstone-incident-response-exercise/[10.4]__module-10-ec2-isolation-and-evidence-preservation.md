## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**EC2 isolation and evidence preservation overview**

When an Amazon Elastic Compute Cloud (EC2) instance is suspected of compromise, your first goal is containment, not cleanup. You want to stop further attacker activity, preserve volatile and non-volatile evidence, and avoid actions that destroy forensic value. In AWS, that usually means isolating the instance at the network layer, preserving disk state with snapshots, and documenting every change.

A good incident response sequence is:

1. Contain the instance.
2. Preserve evidence.
3. Collect only the data you need for analysis.
4. Restore service from a known-good replacement.

This topic is exam-relevant because AWS gives you multiple ways to isolate an instance without immediately terminating it, and each choice affects evidence quality, downtime, and operational risk.

**Core containment goals**

Containment in EC2 is about preventing the instance from communicating while keeping it available for investigation if needed. You usually want to block inbound and outbound traffic, remove it from production paths, and prevent automatic replacement from undoing your work.

- **Security groups** can block traffic quickly, but they are not always enough if the instance has multiple network paths.
- **Network ACLs** can help at the subnet boundary, but they affect all resources in that subnet.
- **ENI isolation** is often the cleanest exam answer because you can detach the instance’s Elastic Network Interface (ENI) and move it to a quarantine subnet or attach a replacement ENI with restrictive rules.
- **Load balancer deregistration** prevents the compromised node from receiving new client traffic.
- **Auto Scaling** may replace the instance unless you suspend processes or remove it from the group first.

> If you need to preserve evidence, avoid terminating the instance until you have captured the data you need. Termination can destroy instance store data and may complicate later analysis.

**Isolation patterns you should know**

The best containment pattern depends on how much access you still need.

- **Security group lockdown**
  - Replace the instance’s security group with a restrictive group.
  - Allow only your investigation source IP or a bastion/quarantine host.
  - Useful when you need quick containment with minimal topology changes.
- **ENI detach and quarantine**
  - Detach the primary ENI from the compromised instance.
  - Attach it to a quarantine instance or move the instance to a quarantine subnet by reattaching a different ENI.
  - Useful when you want to preserve the network identity for analysis or keep the instance reachable only through controlled paths.
- **Subnet quarantine**
  - Move the instance into a subnet with no route to the internet, on-premises networks, or production services.
  - Pair with restrictive security groups and NACLs.
  - Useful for live response when you still need access to the host.
- **Load balancer removal**
  - Deregister the target from an Application Load Balancer (ALB) or Network Load Balancer (NLB).
  - Prevents customer traffic from reaching the instance.
  - Often combined with security group changes.
- **Auto Scaling containment**
  - Suspend scaling processes or detach the instance from the Auto Scaling group.
  - Otherwise the group may terminate and replace the instance, which can destroy evidence and reintroduce risk.

**ENI-based isolation and quarantine subnet design**

The Elastic Network Interface is the instance’s network attachment. For exam scenarios, ENI handling is a common containment move because it gives you control over the instance’s network identity and routing.

A practical quarantine design looks like this:

- Production subnet:
  - Routes to internet gateway, NAT gateway, or internal services.
  - Normal security groups and monitoring.
- Quarantine subnet:
  - No route to the internet gateway.
  - No route to production VPC peering, Transit Gateway, or VPN unless explicitly required.
  - Tight security group rules allowing only forensic access.
- Investigation host:
  - Separate hardened instance used for analysis.
  - Accessed through AWS Systems Manager Session Manager or tightly controlled SSH/RDP if absolutely necessary.

Example containment flow:

1. Remove the instance from the load balancer.
2. Suspend or detach it from Auto Scaling.
3. Detach the ENI or replace it with a quarantined network path.
4. Restrict security groups and NACLs.
5. Preserve disks and collect evidence.

Example CLI commands:

```bash
aws elbv2 deregister-targets \
  --target-group-arn arn:aws:elasticloadbalancing:us-east-1:111122223333:targetgroup/prod-tg/1234567890abcdef \
  --targets Id=i-0123456789abcdef0

aws autoscaling suspend-processes \
  --auto-scaling-group-name prod-asg \
  --scaling-processes Launch Terminate HealthCheck ReplaceUnhealthy AZRebalance

aws ec2 modify-instance-attribute \
  --instance-id i-0123456789abcdef0 \
  --groups sg-0abc1234def567890
```

If you need to move the ENI, you typically detach it first and then attach it to a quarantine instance or another controlled network path:

```bash
aws ec2 detach-network-interface \
  --attachment-id eni-attach-0abc1234def567890

aws ec2 attach-network-interface \
  --network-interface-id eni-0abc1234def567890 \
  --instance-id i-0fedcba9876543210 \
  --device-index 1
```

**Accessing a live instance safely**

If the instance is still running and you need to inspect it, prefer AWS Systems Manager Session Manager over SSH or RDP. Session Manager reduces exposure because you do not need inbound ports open and you can log sessions centrally.

Use Session Manager when:

- The instance has the SSM Agent installed and running.
- The instance profile allows `ssm:StartSession` and related permissions.
- You want to avoid opening port 22 or 3389 during an incident.

Example Session Manager access:

```bash
aws ssm start-session --target i-0123456789abcdef0
```

Why this matters:

- You can keep inbound security groups closed.
- You can record session activity in CloudTrail and optionally to S3 or CloudWatch Logs.
- You reduce the chance that an attacker can reuse exposed remote access services.

If Session Manager is unavailable, SSH or RDP may still be used, but only through a tightly controlled jump host or quarantine network. Open the minimum access needed and close it immediately after use.

**Evidence preservation principles**

Evidence preservation means keeping data in a state that supports later analysis and, if needed, chain-of-custody requirements. In AWS, you usually preserve evidence by snapshotting EBS volumes, capturing logs, and documenting metadata before making destructive changes.

Preserve these categories:

- **Volatile evidence**
  - Running processes
  - Network connections
  - Logged-in users
  - In-memory malware or decrypted secrets
- **Non-volatile evidence**
  - EBS volumes
  - Instance metadata
  - CloudTrail events
  - VPC Flow Logs
  - Application and OS logs
- **Context evidence**
  - Security group rules
  - Route tables
  - IAM role attached to the instance
  - Load balancer registration state
  - Auto Scaling group membership

> Instance store volumes are ephemeral. Data on instance store is lost on stop or terminate, so preserve it immediately if it matters. If the instance uses instance store for logs, caches, or malware staging, treat that data as time-sensitive.

**EBS snapshots and root volume timing**

Amazon Elastic Block Store (EBS) snapshots are the primary way to preserve disk evidence from EC2. A snapshot is crash-consistent by default, and you can create it while the instance is running. That is often the best choice when you need speed and cannot afford downtime.

Key points:

- **Snapshot while running**
  - Fast and usually acceptable for initial preservation.
  - Captures a point-in-time view of the volume.
  - May not be application-consistent unless you quiesce the filesystem or application first.
- **Snapshot after stop**
  - Can improve consistency because the volume is no longer changing.
  - Requires downtime and may alter evidence if the stop action itself matters.
- **Root volume**
  - Often contains the operating system, logs, and malware artifacts.
  - Snapshot it early, before remediation or rebooting.
- **Data volumes**
  - Snapshot separately if they contain application data, logs, or staging areas.
  - Preserve all attached EBS volumes, not just the root disk.

A practical approach is to snapshot the root volume while the instance is still running, then decide whether a controlled stop is justified for a second, more consistent capture. If you stop the instance first, remember that you may lose volatile evidence and any instance store data.

Example snapshot workflow:

```bash
aws ec2 describe-instances \
  --instance-ids i-0123456789abcdef0 \
  --query 'Reservations[0].Instances[0].BlockDeviceMappings'
```

Identify the volume IDs, then snapshot them:

```bash
aws ec2 create-snapshot \
  --volume-id vol-0abc1234def567890 \
  --description "IR snapshot - root volume - i-0123456789abcdef0"

aws ec2 create-snapshot \
  --volume-id vol-0def1234abc567890 \
  --description "IR snapshot - data volume - i-0123456789abcdef0"
```

If you need to reduce the chance of later tampering, tag snapshots immediately:

```bash
aws ec2 create-tags \
  --resources snap-0123456789abcdef0 \
  --tags Key=CaseId,Value=IR-2026-041 \
         Key=EvidenceType,Value=EBS-Snapshot \
         Key=Owner,Value=SecurityTeam
```

**Cross-account and cross-region snapshot handling**

Forensic isolation often means moving evidence out of the compromised account or region. AWS supports snapshot copy, which is useful for evidence handling, disaster separation, and limiting access to the original environment.

Use snapshot copy when you want to:

- Preserve evidence in a separate account with restricted access.
- Store a copy in another region for resilience or legal separation.
- Analyze data without giving investigators access to the production account.

Important considerations:

- **Cross-account copy**
  - Share the snapshot or use AWS Key Management Service (KMS) permissions appropriately.
  - Re-encrypt with a key in the destination account when needed.
- **Cross-region copy**
  - Useful if the source region is compromised or unavailable.
  - Adds time and cost, but improves isolation.
- **Encrypted snapshots**
  - Require KMS permissions for both source and destination.
  - You may need to copy and re-encrypt rather than share directly.
- **Chain of custody**
  - Record snapshot IDs, timestamps, source volume IDs, and destination account/region.
  - Restrict who can restore or mount the copied snapshot.

Example copy command:

```bash
aws ec2 copy-snapshot \
  --source-region us-east-1 \
  --source-snapshot-id snap-0123456789abcdef0 \
  --description "IR copy - isolated analysis account" \
  --encrypted \
  --kms-key-id arn:aws:kms:us-west-2:444455556666:key/abcd1234-5678-90ab-cdef-1234567890ab
```

If the snapshot is shared across accounts, ensure the destination account has the right KMS permissions and that the snapshot is not broadly accessible. For exam purposes, the key idea is that copying snapshots can separate evidence from the compromised environment before analysis.

**Preserving instance store data**

Instance store volumes are physically attached to the host and are not durable. They are useful for temporary scratch space, caches, and high-performance local storage, but they are not preserved by EBS snapshots.

Remember:

- Stopping or terminating the instance destroys instance store data.
- You cannot snapshot instance store the way you snapshot EBS.
- If instance store contains evidence, you must collect it before stop/terminate.
- If the workload uses instance store for logs or temporary files, those files may be the only copy.

Practical implications:

- Prefer memory and disk capture before any stop action if instance store matters.
- If you must preserve instance store content, copy it out while the instance is still running.
- Document that the data is volatile and may already be incomplete.

**Volatile evidence and memory forensics**

Memory is the most volatile evidence source on an EC2 instance. It can contain running malware, injected code, decrypted credentials, open network sessions, and process memory that never touches disk. AWS does not provide a simple built-in memory snapshot feature for EC2, so memory acquisition is usually a guest-level task and may be limited by permissions, tooling, and time.

Common approaches include:

- Running a memory acquisition tool inside the guest OS.
- Using a trusted live-response agent if one is already installed.
- Capturing process lists, open sockets, and loaded modules when full memory capture is not feasible.

Examples of useful live-response commands:

```bash
ps auxww
ss -plant
lsof -nP
who
last -a
```

On Windows, you might collect:

```powershell
tasklist /v
netstat -ano
quser
Get-Process
```

For memory capture, the exact tool depends on the operating system and your incident response tooling. Because memory acquisition can alter the system, use it only when the value of the evidence outweighs the risk of contamination. In exam scenarios, the important point is that volatile evidence should be collected first, before rebooting, stopping, or making major changes.

**Logs and supporting AWS evidence**

Do not focus only on the EC2 guest. AWS control-plane and network logs often explain how the compromise happened and what the instance did.

Useful sources include:

- **CloudTrail**
  - API activity such as security group changes, snapshot creation, IAM role changes, and ENI operations.
- **VPC Flow Logs**
  - Network traffic metadata for the ENI or subnet.
  - Helps identify command-and-control traffic or lateral movement.
- **ELB access logs**
  - Request patterns and suspicious client behavior.
- **SSM logs**
  - Session activity if Session Manager is used.
- **CloudWatch Logs**
  - Application and system logs if already centralized.

A strong forensic workflow correlates guest evidence with AWS logs. For example, a suspicious process on the instance may line up with a new inbound security group rule, a new IAM role attachment, or unusual outbound traffic in VPC Flow Logs.

**End-to-end incident response scenario**

A realistic exam scenario often asks what you do first and what you preserve next. Use this sequence as a mental model:

1. Detect suspicious behavior on `i-0123456789abcdef0`.
2. Deregister the instance from the load balancer.
3. Suspend Auto Scaling replacement or detach the instance from the group.
4. Restrict network access using security groups, NACLs, or ENI quarantine.
5. Use Session Manager for controlled live response if the host must remain online.
6. Capture volatile evidence such as processes, connections, and memory if feasible.
7. Snapshot the root EBS volume and any attached data volumes while the instance is running.
8. If needed, stop the instance for a second, more consistent capture, understanding that this increases downtime and loses volatile state.
9. Copy snapshots to a separate account or region for isolated analysis.
10. Restore service from a clean replacement instance, not the suspected host.

Example command set for a containment-first workflow:

```bash
# 1. Remove from load balancer
aws elbv2 deregister-targets \
  --target-group-arn arn:aws:elasticloadbalancing:us-east-1:111122223333:targetgroup/prod-tg/1234567890abcdef \
  --targets Id=i-0123456789abcdef0

# 2. Suspend Auto Scaling replacement
aws autoscaling suspend-processes \
  --auto-scaling-group-name prod-asg \
  --scaling-processes Launch Terminate HealthCheck ReplaceUnhealthy AZRebalance

# 3. Lock down security groups
aws ec2 modify-instance-attribute \
  --instance-id i-0123456789abcdef0 \
  --groups sg-quarantine123456

# 4. Snapshot the root volume
aws ec2 create-snapshot \
  --volume-id vol-0abc1234def567890 \
  --description "IR root snapshot - compromised EC2"
```

**Practical analysis workflow after isolation**

Once the instance is contained and evidence is preserved, analyze copies rather than originals whenever possible. Mount copied snapshots on a separate forensic instance in a restricted account or VPC.

Typical analysis steps:

- Create a new EBS volume from the copied snapshot.
- Attach it to a forensic workstation or analysis instance.
- Mount read-only if possible.
- Search for persistence mechanisms, suspicious binaries, and log tampering.
- Compare timestamps and hashes against known-good baselines.

Example Linux mount workflow:

```bash
lsblk
sudo mkdir -p /mnt/evidence
sudo mount -o ro,noload /dev/nvme1n1p1 /mnt/evidence
sudo grep -R "cron" /mnt/evidence/etc /mnt/evidence/var/log
```

This approach reduces the risk of contaminating evidence and keeps the original snapshot intact for later review.

**Common exam traps**

- **Terminating too early**
  - You lose instance store data and may destroy volatile evidence.
- **Forgetting Auto Scaling**
  - The group may replace the instance before you finish preserving evidence.
- **Ignoring the ENI**
  - The instance may still be reachable through another network path.
- **Using SSH/RDP by default**
  - Session Manager is usually safer for live response.
- **Snapshotting only the root volume**
  - Data volumes may contain the actual evidence.
- **Skipping snapshot copy**
  - Keeping all evidence in the compromised account or region weakens isolation.
- **Overlooking public IPs and Elastic IPs**
  - Public exposure may continue unless you remove or remap them.
- **Forgetting instance store**
  - Ephemeral data disappears on stop or terminate.

**Quick reference**

- Use **ENI quarantine** or restrictive security groups to stop traffic.
- Use **Session Manager** for safer live access.
- Snapshot **all EBS volumes**, not just the root disk.
- Snapshot while running for speed; stop first only if consistency is worth the downtime.
- Treat **instance store** as volatile and preserve it immediately if needed.
- Copy snapshots to another **account** or **region** for isolated analysis.
- Analyze copies, not originals, whenever possible.
