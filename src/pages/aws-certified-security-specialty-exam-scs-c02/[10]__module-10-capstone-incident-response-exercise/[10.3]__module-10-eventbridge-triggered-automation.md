## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**EventBridge-triggered automation overview**

Amazon EventBridge is a serverless event bus that routes events from AWS services, SaaS applications, and your own applications to targets based on rules. In security operations, you use it to detect meaningful changes and trigger automated response actions such as quarantine, notification, ticket creation, or evidence collection.

The key exam idea is that EventBridge is not the detector itself. It is the routing and orchestration layer that reacts to events emitted by services like AWS CloudTrail, AWS Security Hub, Amazon GuardDuty, AWS Config, and IAM-related services. You design rules that match event patterns, then send those events to targets such as AWS Lambda, AWS Step Functions, Amazon SNS, AWS Systems Manager Automation, Amazon SQS, or another EventBridge bus.

- **Event-driven** automation reduces response time and human error.
- **Pattern matching** lets you target only the events that matter.
- **Decoupling** keeps detection, routing, and remediation separate.

**Core EventBridge concepts**

EventBridge receives events in a standard JSON structure. A rule evaluates the event against an event pattern, and if it matches, EventBridge invokes one or more targets.

<dl>
<dt><strong>Event bus</strong></dt>
<dd>A pipeline that receives events. AWS services typically publish to the default bus, while custom buses support application or cross-account routing.</dd>
<dt><strong>Rule</strong></dt>
<dd>A filter that matches events and sends them to targets.</dd>
<dt><strong>Target</strong></dt>
<dd>The destination that performs work, such as Lambda, Step Functions, SNS, SQS, or Systems Manager Automation.</dd>
<dt><strong>Event pattern</strong></dt>
<dd>A JSON filter that matches fields like source, detail-type, account, region, and detail values.</dd>
</dl>

EventBridge is especially useful when you need near-real-time response to security-relevant changes. For example, you can trigger a Lambda function when an IAM access key is created, when a Security Hub finding changes to `CRITICAL`, or when GuardDuty detects suspicious API activity.

**Security use cases and when to use them**

Use EventBridge when the response should happen automatically after a specific event occurs. It is a strong fit for operational security workflows that are deterministic and repeatable.

- **GuardDuty findings** can trigger containment actions such as isolating an EC2 instance or disabling credentials.
- **Security Hub findings** can trigger ticketing, notification, or enrichment workflows.
- **CloudTrail API activity** can trigger alerts for sensitive actions like `PutBucketPolicy`, `CreateAccessKey`, or `AuthorizeSecurityGroupIngress`.
- **AWS Config compliance changes** can trigger remediation when a resource becomes noncompliant.
- **IAM and Organizations events** can trigger governance workflows such as approval, logging, or account-level response.

A common pattern is to use EventBridge for the first hop and then delegate the actual workflow to Step Functions or Systems Manager Automation. That keeps the rule simple and makes the remediation easier to audit and retry.

**Event sources and event types**

EventBridge can consume several event categories:

- **AWS service events**
  - Emitted directly by AWS services.
  - Examples include EC2 state changes, EBS volume notifications, and Security Hub findings.
- **CloudTrail events**
  - Represent API calls made in your account.
  - Useful for detecting administrative actions and suspicious changes.
- **Scheduled events**
  - Trigger on a cron or rate expression.
  - Useful for periodic checks, such as validating security posture or rotating artifacts.
- **Custom application events**
  - Published by your own code using `PutEvents`.
  - Useful when your application detects a condition that should drive security automation.

For exam purposes, remember that CloudTrail is the source of API activity visibility, while EventBridge is the mechanism that can react to those events.

**Event pattern matching**

Event patterns are the heart of EventBridge rules. You match on fields in the event envelope and nested `detail` fields. Matching is exact by default, but you can use arrays of acceptable values and some comparison operators.

Example rule pattern for a sensitive IAM action:

```json
{
  "source": ["aws.iam"],
  "detail-type": ["AWS API Call via CloudTrail"],
  "detail": {
    "eventName": ["CreateAccessKey", "UpdateLoginProfile", "AttachUserPolicy"]
  }
}
```

This pattern matches CloudTrail events from IAM when one of the listed API calls occurs. You can then route the event to a Lambda function that disables the key, opens a ticket, or sends an alert.

Useful matching fields include:

- `source`
- `detail-type`
- `detail.eventSource`
- `detail.eventName`
- `detail.userIdentity.type`
- `detail.requestParameters`
- `detail.responseElements`
- `account`
- `region`

> Use the narrowest pattern that still captures the security condition you care about. Overly broad rules create noisy automation and can cause unnecessary remediation.

**Practical automation targets**

EventBridge supports multiple targets, and the best target depends on the complexity of the response.

- **AWS Lambda**
  - Best for lightweight, code-driven actions.
  - Good for enrichment, tagging, notifications, and simple containment.
- **AWS Step Functions**
  - Best for multi-step workflows.
  - Good when you need branching, retries, approvals, or human-in-the-loop steps.
- **Amazon SNS**
  - Best for fan-out notifications.
  - Good for email, SMS, or downstream subscriptions.
- **Amazon SQS**
  - Best for buffering and decoupling.
  - Good when downstream processing may be bursty or delayed.
- **AWS Systems Manager Automation**
  - Best for operational remediation runbooks.
  - Good for standardized actions like instance isolation or patching.
- **Another EventBridge bus**
  - Best for cross-account or cross-domain routing.
  - Good for central security aggregation.

A common security architecture is:

1. Security service emits finding or event.
2. EventBridge rule matches the event.
3. Step Functions coordinates the response.
4. Lambda or Systems Manager performs the action.
5. SNS or Security Hub records the outcome.

**Example: GuardDuty finding to Lambda**

A GuardDuty finding can be routed to a Lambda function for enrichment and containment. The Lambda function might tag the resource, revoke temporary credentials, or notify a security team.

```json
{
  "source": ["aws.guardduty"],
  "detail-type": ["GuardDuty Finding"],
  "detail": {
    "severity": [{ "numeric": [">=", 7] }]
  }
}
```

A Lambda handler can inspect the finding and decide what to do:

```python
import json

def lambda_handler(event, context):
    finding = event["detail"]
    severity = finding.get("severity", 0)
    finding_type = finding.get("type", "unknown")

    if severity >= 7:
        # Example: send to containment workflow
        print(f"High severity finding: {finding_type}")
        return {"action": "containment-triggered"}

    return {"action": "logged"}
```

In practice, you would add error handling, idempotency, and logging. Security automation must tolerate duplicate events because event delivery is at-least-once.

**Example: CloudTrail API call to Systems Manager Automation**

You can trigger a runbook when someone changes a security group rule.

```json
{
  "source": ["aws.ec2"],
  "detail-type": ["AWS API Call via CloudTrail"],
  "detail": {
    "eventName": ["AuthorizeSecurityGroupIngress", "AuthorizeSecurityGroupEgress"]
  }
}
```

A Systems Manager Automation document can then revert the change or open a review workflow. This is useful when you want a controlled, auditable remediation path rather than custom code.

**Example: Security Hub finding to Step Functions**

Security Hub findings are often normalized and enriched, which makes them a strong trigger for automation. You can route only high-severity findings to a workflow.

```json
{
  "source": ["aws.securityhub"],
  "detail-type": ["Security Hub Findings - Imported"],
  "detail": {
    "findings": {
      "Severity": {
        "Label": ["CRITICAL", "HIGH"]
      },
      "RecordState": ["ACTIVE"]
    }
  }
}
```

Step Functions can then:

1. Enrich the finding with asset metadata.
2. Check whether the resource is production.
3. Notify the owner.
4. Quarantine or disable the affected resource if required.
5. Write the outcome to a case management system.

**Cross-account and centralized security automation**

In larger environments, you often centralize security operations in a dedicated account. EventBridge supports cross-account event routing, which lets member accounts send security events to a central bus.

This pattern is useful when:

- You want one security operations account to receive findings from many accounts.
- You need consistent remediation logic across the organization.
- You want to reduce duplication of rules and targets in each account.

A typical topology is:

- Member account publishes GuardDuty or Security Hub events.
- EventBridge rule forwards matching events to a central event bus.
- Central account runs enrichment, alerting, and remediation workflows.

For AWS Organizations, this pattern pairs well with delegated administration for security services. The security account can aggregate findings while still allowing local response where needed.

**CLI examples**

You can create rules and targets with the AWS CLI. The exact permissions depend on the target service and whether the target is in the same account.

Create a rule:

```bash
aws events put-rule \
  --name HighSeverityGuardDuty \
  --event-pattern '{
    "source": ["aws.guardduty"],
    "detail-type": ["GuardDuty Finding"],
    "detail": {
      "severity": [{ "numeric": [">=", 7] }]
    }
  }'
```

Add a Lambda target:

```bash
aws events put-targets \
  --rule HighSeverityGuardDuty \
  --targets "Id"="1","Arn"="arn:aws:lambda:us-east-1:123456789012:function:ContainmentFunction"
```

Grant EventBridge permission to invoke Lambda:

```bash
aws lambda add-permission \
  --function-name ContainmentFunction \
  --statement-id EventBridgeInvoke \
  --action lambda:InvokeFunction \
  --principal events.amazonaws.com \
  --source-arn arn:aws:events:us-east-1:123456789012:rule/HighSeverityGuardDuty
```

List rules:

```bash
aws events list-rules
```

Test event matching with a sample event:

```bash
aws events test-event-pattern \
  --event-pattern file://pattern.json \
  --event file://sample-event.json
```

**Operational design considerations**

EventBridge-triggered automation is powerful, but security workflows need careful design.

- **Idempotency**
  - EventBridge can deliver events more than once.
  - Your target should safely handle duplicates.
- **Least privilege**
  - The rule and target execution role should have only the permissions required.
  - Avoid broad permissions that let automation modify unrelated resources.
- **Blast radius**
  - Start with alerting or tagging before full containment.
  - Use staged rollout for destructive actions.
- **Observability**
  - Log rule matches, target invocations, and remediation outcomes.
  - Use CloudWatch Logs, CloudTrail, and service-specific audit trails.
- **Dead-letter handling**
  - Use an SQS dead-letter queue or retry strategy where supported.
  - Capture failed events for later analysis.
- **Filtering**
  - Filter early to reduce noise and cost.
  - Route only actionable events to expensive workflows.

> Be careful with automated containment. A false positive can disrupt production if the workflow disables access, isolates instances, or revokes credentials without validation.

**Common security response patterns**

- **Alert only**
  - Send an SNS notification or create a ticket.
  - Use this when you need human review first.
- **Enrich and alert**
  - Add context such as tags, account owner, or asset criticality.
  - Use this to improve triage quality.
- **Contain and notify**
  - Quarantine a resource and notify the security team.
  - Use this for high-confidence detections.
- **Remediate and verify**
  - Revert a configuration change and confirm the fix.
  - Use this for known-bad drift or policy violations.
- **Escalate by severity**
  - Route low severity to logging, medium to alerts, and high to automation.
  - Use this to balance speed and risk.

**Troubleshooting and validation**

When a rule does not fire, check the event shape first. Many failures come from mismatched fields, wrong `detail-type`, or using the wrong event source.

- Confirm the event actually reaches the bus.
- Verify the rule pattern against a real sample event.
- Check that the target permission allows EventBridge to invoke it.
- Inspect CloudWatch Logs for Lambda or Step Functions failures.
- Review EventBridge metrics for failed invocations and throttling.

A practical validation workflow is:

1. Capture a real event from CloudTrail, Security Hub, or GuardDuty.
2. Save it as JSON.
3. Test the event pattern locally with `test-event-pattern`.
4. Send a controlled test event to the bus.
5. Confirm the target receives and processes it.

**Exam focus points**

- EventBridge is the routing layer for event-driven automation.
- CloudTrail provides API activity; EventBridge reacts to it.
- Security Hub and GuardDuty are common security event sources.
- Lambda is best for code-based actions; Step Functions is best for multi-step orchestration.
- Systems Manager Automation is a strong choice for standardized remediation.
- Cross-account event buses support centralized security operations.
- At-least-once delivery means you must design for duplicate events.

**Quick reference**

| Need | Best fit | Why |
|---|---|---|
| Simple alerting | SNS | Fast fan-out to subscribers |
| Lightweight remediation | Lambda | Flexible code and fast execution |
| Multi-step response | Step Functions | Branching, retries, and approvals |
| Standard runbook | Systems Manager Automation | Controlled operational remediation |
| Event buffering | SQS | Decouples producers and consumers |
| Central aggregation | EventBridge bus | Cross-account and cross-service routing |

**Summary**

EventBridge-triggered automation lets you turn security events into immediate, repeatable actions. For the exam, focus on the separation between detection and response, the structure of event patterns, and the choice of target for each response type. The strongest designs are narrow, auditable, and resilient to duplicate events.
