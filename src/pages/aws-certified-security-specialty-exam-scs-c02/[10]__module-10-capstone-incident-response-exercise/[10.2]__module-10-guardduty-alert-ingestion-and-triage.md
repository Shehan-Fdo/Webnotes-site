## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**GuardDuty overview and alert lifecycle**

Amazon GuardDuty is a managed threat detection service that analyzes signals from AWS data sources and produces findings rather than raw alerts. For exam purposes, focus on how findings move from detection to routing, triage, containment, and closure.

GuardDuty findings are typically consumed in three ways:

- **Console review** for interactive investigation.
- **Event-driven automation** through Amazon EventBridge.
- **Centralized security operations** through AWS Security Hub, SIEMs, or ticketing systems.

A finding is not the same as a remediation action. You usually ingest the finding, enrich it with context, decide whether it is expected or malicious, and then respond with containment steps such as revoking credentials, isolating an instance, or blocking network paths.

**Core ingestion destinations**

GuardDuty can send findings to multiple downstream systems depending on your operating model.

- **Amazon EventBridge** is the primary native integration for real-time automation.
  - Use it when you want to trigger Lambda, Step Functions, SNS, SSM Automation, or custom workflows.
  - Best for immediate response and routing by severity, type, account, or resource.
- **AWS Security Hub** is the best choice for centralized security posture and cross-service triage.
  - Use it when you want GuardDuty findings normalized into the AWS Security Finding Format (ASFF).
  - Best for SOC dashboards, aggregation, and multi-product correlation.
- **Amazon SNS** is useful when you need simple notification fan-out.
  - Use it for email, SMS, or downstream subscriptions through Lambda or HTTPS endpoints.
  - Often paired with EventBridge rather than used directly from GuardDuty.
- **Amazon SQS** is useful when you need durable buffering for custom consumers.
  - Use it when your downstream system cannot process bursts in real time.
  - Common in SIEM ingestion pipelines or custom enrichment workers.

A practical pattern is:

1. GuardDuty emits a finding.
2. EventBridge matches the finding.
3. EventBridge routes to Lambda, SNS, SQS, or Step Functions.
4. The workflow enriches, triages, and optionally opens a ticket or starts containment.

**Finding types and what they usually mean**

GuardDuty finding names are important because exam questions often describe behavior rather than the exact service feature.

- **UnauthorizedAccess:IAMUser/ConsoleLogin** indicates suspicious console authentication activity.
- **CredentialAccess:IAMUser/AnomalousBehavior** suggests unusual API usage for an identity.
- **Trojan:EC2/BlackholeTraffic** can indicate an EC2 instance communicating with known malicious infrastructure.
- **CryptoCurrency:EC2/BitcoinTool.B!DNS** often points to mining-related activity or tooling.
- **Recon:EC2/Portscan** suggests scanning behavior from or against an instance.
- **Backdoor:EC2/Spambot** may indicate compromise and outbound abuse.
- **Behavior:EC2/NetworkPortUnusual** can indicate unexpected network activity that needs context.

Treat the finding name as a clue, not proof. You still validate whether the activity matches a known workload, a test environment, a penetration test, or an actual compromise.

**Severity, confidence, and triage priority**

GuardDuty findings include severity values that help you prioritize response. In practice, severity is numeric and often treated as low, medium, or high based on the score.

- **High severity** findings usually deserve immediate investigation.
- **Medium severity** findings often need enrichment and correlation.
- **Low severity** findings may be benign, but repeated patterns can still matter.

Do not rely on severity alone. Combine it with:

- **Finding type**
- **Affected resource**
- **Account ownership**
- **Time of day**
- **Known change windows**
- **Whether the resource is internet-facing**
- **Whether the activity matches expected automation**

> GuardDuty severity is a prioritization signal, not a verdict. A low-severity finding can still be important if it affects a production identity or a sensitive workload.

**Centralized administration with AWS Organizations**

In enterprise environments, you usually manage GuardDuty centrally.

- **Delegated administrator**: one account manages GuardDuty for the organization.
- **Member accounts**: accounts that send findings to the delegated admin.
- **Organization auto-enablement**: new accounts can be enrolled automatically.

Typical centralized workflow:

1. Enable GuardDuty in the management or delegated admin account.
2. Register a delegated administrator for GuardDuty.
3. Enable organization-wide auto-enrollment for member accounts.
4. Turn on supported data sources in each Region you care about.
5. Aggregate findings centrally for triage and response.

This matters because GuardDuty is Region-scoped. If a Region is not enabled, you may miss findings there even if the organization is otherwise configured correctly.

**Validating GuardDuty health and coverage**

A common exam trap is assuming GuardDuty is “on” everywhere when it is only enabled in one Region or one account.

Check the following:

- The detector exists in the target Region.
- The detector is enabled.
- The account is enrolled as expected.
- The relevant data sources are active.
- The Region supports the feature you expect.
- Findings are being generated and delivered to the downstream target.

Useful CLI checks:

```bash
aws guardduty list-detectors --region us-east-1
aws guardduty get-detector --detector-id a1b2c3d4e5f6 --region us-east-1
aws guardduty list-members --detector-id a1b2c3d4e5f6 --region us-east-1
```

If you do not see findings, verify:

- The detector is enabled in the same Region as the activity.
- The account is not excluded by organization settings.
- The workload actually generated the suspicious behavior.
- EventBridge rules are in the same Region as the source findings.
- Security Hub is enabled and accepting GuardDuty as a product source.

**EventBridge routing for real-time response**

EventBridge is the most important ingestion path for automation. GuardDuty publishes findings as events, and you match on fields such as source, detail type, finding type, severity, account, and resource identifiers.

A practical rule pattern should account for numeric severity values that may be represented as floating-point numbers. Matching only whole numbers can miss valid findings.

Example EventBridge rule pattern:

```json
{
  "source": ["aws.guardduty"],
  "detail-type": ["GuardDuty Finding"],
  "detail": {
    "severity": [
      { "numeric": [">=", 7] }
    ]
  }
}
```

You can also route by finding type:

```json
{
  "source": ["aws.guardduty"],
  "detail-type": ["GuardDuty Finding"],
  "detail": {
    "type": [
      "UnauthorizedAccess:IAMUser/ConsoleLogin",
      "CredentialAccess:IAMUser/AnomalousBehavior"
    ]
  }
}
```

Common targets include:

- **Lambda** for enrichment and custom logic.
- **Step Functions** for multi-step incident workflows.
- **SNS** for notifications.
- **SQS** for buffering and decoupling.
- **SSM Automation** for containment actions.

A useful pattern is to send only high-severity findings to automated containment, while sending medium and low severity to a queue or ticketing system for analyst review.

**Example: EventBridge to Lambda containment**

```json
{
  "source": ["aws.guardduty"],
  "detail-type": ["GuardDuty Finding"],
  "detail": {
    "severity": [{ "numeric": [">=", 7] }]
  }
}
```

Lambda handler sketch:

```python
import json

def lambda_handler(event, context):
    finding = event["detail"]
    finding_type = finding["type"]
    resource = finding.get("resource", {})
    account_id = event["account"]

    # Example: enrich, log, and trigger containment
    print(json.dumps({
        "account": account_id,
        "type": finding_type,
        "severity": finding.get("severity"),
        "resource": resource
    }))

    return {"status": "received"}
```

**Security Hub integration and ASFF triage**

Security Hub is the best place to normalize GuardDuty findings alongside other security products. GuardDuty findings are mapped into the AWS Security Finding Format (ASFF), which gives you consistent fields for triage.

Why use Security Hub:

- You want a single pane of glass for multiple AWS security services.
- You want standardized fields for severity, workflow status, and compliance context.
- You want to correlate GuardDuty with Inspector, Macie, IAM Access Analyzer, and partner products.

In Security Hub, you typically:

- Enable Security Hub in the Region.
- Enable GuardDuty as a product source.
- Review findings in ASFF.
- Update workflow status during triage.
- Archive findings that are accepted risk, false positive, or informational after review.

Important lifecycle distinction:

- **Workflow status** is used for triage state in Security Hub.
- **Archiving** removes a finding from active attention in Security Hub views.
- GuardDuty itself does not use a “resolved” state in the same way a ticketing system does.

Example ASFF fields you may see:

- `Severity.Label`
- `Severity.Normalized`
- `ProductName`
- `Types`
- `Resources`
- `Workflow.Status`
- `RecordState`

Example triage logic:

- Set `Workflow.Status` to `NEW` when first received.
- Move to `NOTIFIED` or `IN_PROGRESS` during investigation.
- Archive after validation if the finding is expected or no longer actionable.

**GuardDuty filters, suppression, and archive behavior**

GuardDuty supports filters that can archive findings automatically based on matching criteria. This is useful for recurring benign activity, but it must be used carefully.

Use filters when:

- A known scanner generates repeat findings.
- A trusted automation role creates expected API patterns.
- A lab or test account produces noise.
- You need to reduce alert fatigue for a specific, well-understood pattern.

Be conservative with filters because they can hide real threats if overbroad.

Common filter considerations:

- Filter on specific finding types rather than broad categories.
- Filter on account, resource, or role only when the scope is tightly controlled.
- Review filters regularly, especially after architecture changes.
- Prefer archiving only after you understand the pattern.

GuardDuty filter criteria are based on supported finding attributes, not arbitrary nested JSON paths. Use the service’s documented filter keys and avoid assuming every nested resource field is filterable.

Example of a safer filter concept:

- Match a specific finding type.
- Match a specific account ID.
- Match a known trusted role or resource tag if supported.

> Do not use filters as a substitute for investigation. If you cannot explain why a finding is benign, do not archive it.

**Programmatic retrieval at scale**

For SIEM ingestion, reporting, or custom analytics, you often need to retrieve findings programmatically rather than manually.

Typical approaches:

- **GuardDuty API/CLI** for direct retrieval.
- **EventBridge streaming** for near-real-time delivery.
- **Security Hub export** for normalized multi-product data.
- **S3-based archival pipelines** if you need long-term retention and batch processing.

Useful CLI flow:

1. List finding IDs.
2. Retrieve full finding details.
3. Enrich or forward the data.

Example:

```bash
aws guardduty list-findings \
  --detector-id a1b2c3d4e5f6 \
  --finding-criteria '{"Criterion":{"severity":{"Gte":7}}}' \
  --region us-east-1
```

Then fetch details:

```bash
aws guardduty get-findings \
  --detector-id a1b2c3d4e5f6 \
  --finding-ids finding-id-1 finding-id-2 \
  --region us-east-1
```

For large-scale ingestion, prefer EventBridge or Security Hub over polling. Polling is useful for backfill, validation, or ad hoc investigation, but it is not the best primary transport for real-time operations.

**Practical triage workflow**

A strong incident response workflow starts with context, not with immediate remediation.

1. Confirm the finding is real.
   - Check the resource, account, Region, and timestamp.
   - Determine whether the activity matches a change window or known automation.
2. Assess blast radius.
   - Identify related identities, instances, buckets, or VPCs.
   - Look for repeated findings from the same source.
3. Enrich the finding.
   - Correlate with CloudTrail, VPC Flow Logs, DNS logs, and application logs.
   - Check whether the principal is expected to perform the action.
4. Contain if needed.
   - Disable or rotate credentials.
   - Isolate the instance or security group.
   - Block outbound traffic or suspicious destinations.
5. Document and close.
   - Record the root cause.
   - Archive or close the finding in the appropriate system.
   - Add a suppression rule only if the pattern is verified and stable.

**Example: suspicious console login**

A GuardDuty finding such as `UnauthorizedAccess:IAMUser/ConsoleLogin` often means a login attempt from an unusual location or with suspicious characteristics.

Triage steps:

- Check whether the user is a human or an automation identity.
- Review CloudTrail for login source IP, MFA status, and success or failure.
- Confirm whether the user was traveling, using a VPN, or performing a test.
- If suspicious, disable the access key, reset the password, and investigate other API activity.

Useful CloudTrail lookup example:

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=ConsoleLogin \
  --region us-east-1
```

**Example: EC2 backdoor or crypto-mining behavior**

A finding such as `Backdoor:EC2/Spambot` or `CryptoCurrency:EC2/BitcoinTool.B!DNS` suggests possible compromise.

Triage steps:

- Identify the instance ID and security group.
- Check recent user data changes, package installs, and outbound connections.
- Review instance metadata access and IAM role permissions.
- Snapshot the volume if forensic preservation is required.
- Quarantine the instance by moving it to a restricted security group.

Example containment command:

```bash
aws ec2 modify-instance-attribute \
  --instance-id i-0123456789abcdef0 \
  --groups sg-quarantine123456
```

**Example: suspicious API behavior from an IAM principal**

A finding such as `CredentialAccess:IAMUser/AnomalousBehavior` may indicate credential misuse or automation drift.

Triage steps:

- Check the principal’s normal API baseline.
- Review CloudTrail for unusual services, regions, or call volume.
- Determine whether the access key is embedded in a CI/CD pipeline or used interactively.
- Rotate credentials if compromise is plausible.
- Review IAM policies for overprivilege.

**CloudTrail, VPC Flow Logs, and enrichment**

GuardDuty is strongest when paired with other telemetry.

- **CloudTrail** helps you validate API activity and identity behavior.
- **VPC Flow Logs** help you confirm network paths and suspicious destinations.
- **Route 53 Resolver query logs** can help with DNS-based indicators.
- **Application logs** help you determine whether the activity affected workloads.

Use these sources to answer:

- Who did it?
- From where?
- What was touched?
- Was the behavior successful?
- Is the activity ongoing?

**Operational response patterns**

Choose the response path based on the finding and the business impact.

- **Notify only** for low-risk, informational, or expected findings.
- **Investigate and ticket** for medium-risk findings with unclear context.
- **Automate containment** for high-confidence, high-severity compromise indicators.
- **Escalate to incident response** when multiple findings correlate across identities, hosts, and network paths.

A common automation pattern is:

- EventBridge rule matches high severity.
- Lambda enriches with tags, owner, and account metadata.
- Step Functions decides whether to isolate, disable, or notify.
- SNS sends a summary to the SOC.
- Security Hub records the triage state.

**Troubleshooting missing or delayed findings**

If findings are not appearing, check the following in order:

1. GuardDuty is enabled in the correct Region.
2. The detector is active and the account is enrolled.
3. The relevant data source is supported and enabled.
4. EventBridge rules are in the same Region.
5. Security Hub is enabled and accepting the product source.
6. Filters are not archiving the finding immediately.
7. The activity actually occurred and was not blocked before detection.

Also verify:

- Cross-account permissions are correct for centralized administration.
- The workload is generating telemetry that GuardDuty can analyze.
- You are not expecting a finding type that is not supported in that Region or account configuration.

**Exam-oriented distinctions**

Know the difference between these concepts:

- **GuardDuty finding**: native detection output from GuardDuty.
- **EventBridge rule**: routing mechanism for automation.
- **Security Hub finding**: normalized finding in ASFF.
- **Filter**: GuardDuty mechanism to archive matching findings.
- **Workflow status**: Security Hub triage state.
- **Archive**: remove from active attention in Security Hub or auto-archive in GuardDuty via filter.

Also remember:

- GuardDuty is detection, not prevention.
- EventBridge is routing, not storage.
- Security Hub is aggregation and triage, not remediation.
- Suppression should be precise and reviewed.

**End-to-end incident example**

A production EC2 instance triggers `Trojan:EC2/BlackholeTraffic`.

1. GuardDuty generates the finding in `us-east-1`.
2. EventBridge matches severity `>= 7`.
3. Lambda enriches the finding with instance tags and owner.
4. Security Hub receives the finding for analyst review.
5. The analyst checks CloudTrail and VPC Flow Logs.
6. The instance shows outbound connections to suspicious IPs.
7. The analyst quarantines the instance with a restricted security group.
8. The team snapshots the volume for forensics.
9. Credentials associated with the instance role are reviewed and rotated if needed.
10. The finding is archived or closed after remediation is documented.

This pattern is the kind of operational flow the exam expects you to understand: detect, route, enrich, triage, contain, and close.

**Quick reference**

- Use **EventBridge** for real-time automation.
- Use **Security Hub** for centralized triage and ASFF normalization.
- Use **SNS** for notifications.
- Use **SQS** for buffering and decoupling.
- Use **filters** carefully to archive known benign patterns.
- Use **CloudTrail** and **VPC Flow Logs** to validate and enrich findings.
- Use **Organizations** and delegated administration for centralized control.
