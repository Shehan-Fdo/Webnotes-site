## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**KMS policy review in an incident**

When sensitive data exposure involves AWS Key Management Service (KMS), you review both the key policy and the surrounding IAM, grant, and service-integrated access paths. The goal is to answer three questions quickly:

- Who can use the key?
- Which services can use the key on behalf of principals?
- Can the exposed data still be decrypted, re-encrypted, or copied elsewhere?

KMS is often the last control standing between encrypted data and a successful exfiltration path. During an incident, you focus on the specific key or keys protecting the affected data, then trace every policy layer that could allow `Decrypt`, `Encrypt`, `ReEncrypt*`, or `GenerateDataKey*`.

**What to review first**

Start with the key’s current metadata and policy surface. A practical triage sequence is:

1. Identify the key ARN or alias used by the affected application.
2. Inspect the key policy for broad principals, wildcard actions, and service exceptions.
3. Check IAM policies attached to users, roles, and roles assumed by workloads.
4. Review grants, because they can authorize KMS usage outside the key policy’s obvious statements.
5. Confirm whether AWS services such as Amazon S3, Amazon EBS, Amazon RDS, or AWS Lambda are using the key through service integration.

Useful CLI commands:

```bash
aws kms describe-key --key-id arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555
aws kms get-key-policy --key-id arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555 --policy-name default
aws kms list-grants --key-id arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555
aws kms list-aliases
```

To find aliases for a specific key, filter the alias list by `TargetKeyId`:

```bash
aws kms list-aliases --query "Aliases[?TargetKeyId=='11111111-2222-3333-4444-555555555555'].[AliasName,TargetKeyId]" --output table
```

**Key metadata and what it tells you**

`DescribeKey` is useful for understanding the key’s state and origin, but it does not replace policy review. The response includes `KeyMetadata`, which can show:

- `KeyId` and `Arn`
- `KeyManager` such as `AWS` or `CUSTOMER`
- `KeyState` such as `Enabled`, `Disabled`, `PendingDeletion`, `PendingImport`, or `Unavailable`
- `Origin`, such as `AWS_KMS`, `EXTERNAL`, or `AWS_CLOUDHSM`
- `MultiRegion` and replication-related fields

This matters because a key can be technically available while still being too permissive. A key in `Enabled` state can still be abused if the policy or grants allow unintended principals to call `Decrypt`.

> `DescribeKey` helps you understand the key’s condition, but it does not prove who can use it.

**Key policy fundamentals**

A KMS key policy is the primary authorization document for the key. Unlike many AWS resources, KMS does not rely only on IAM. The key policy must allow access, and IAM permissions must also align unless the policy uses the account root delegation pattern.

A typical key policy contains:

- A statement that allows the account to administer the key
- A statement that allows specific roles or services to use the key
- Optional conditions that narrow use by service, encryption context, or resource type

Example of a tightly scoped key policy statement:

```json
{
  "Sid": "AllowUseFromAppRoleOnlyViaS3",
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::123456789012:role/AppDecryptRole"
  },
  "Action": [
    "kms:Decrypt",
    "kms:Encrypt",
    "kms:GenerateDataKey",
    "kms:GenerateDataKeyWithoutPlaintext",
    "kms:DescribeKey"
  ],
  "Resource": "*",
  "Condition": {
    "StringEquals": {
      "kms:ViaService": "s3.us-east-1.amazonaws.com"
    }
  }
}
```

This pattern is useful when the workload should only use KMS through a specific AWS service. It reduces the chance that the same role can call KMS directly from a laptop, CI job, or unrelated application.

**High-risk policy patterns**

During exposure review, look for statements that are broader than intended:

- `Principal: "*"` with KMS usage actions
- `Action: "kms:*"` for non-admin principals
- Wildcard principals combined with weak or missing conditions
- Cross-account access without a clear business need
- Statements that allow `kms:Decrypt` without service or context restrictions
- Grants that were created for temporary workflows but never retired

A common incident pattern is a role that should only encrypt application data but also has `Decrypt`. That role can often read historical ciphertext if it can reach the same key and data store.

**Actions that matter most in exposure scenarios**

Focus on actions that can reveal plaintext or enable future access:

- `kms:Decrypt`
- `kms:ReEncryptFrom`
- `kms:ReEncryptTo`
- `kms:GenerateDataKey`
- `kms:GenerateDataKeyWithoutPlaintext`
- `kms:CreateGrant`
- `kms:ListGrants`
- `kms:RetireGrant`
- `kms:DescribeKey`

`ReEncrypt` is the API operation. `ReEncryptFrom` and `ReEncryptTo` are IAM action names used in permission evaluation. They matter because they can move ciphertext from one key to another without exposing plaintext to the caller, which is useful for migration but risky during containment if an attacker can pivot data between keys.

**Grants and why they are easy to miss**

KMS grants are lightweight permissions that authorize a principal or AWS service to use a key. They are common in service integrations and can outlive the original need if not cleaned up.

Review grants for:

- `GranteePrincipal`
- `Operations`
- `Constraints`
- `RetiringPrincipal`
- `IssuingAccount`

Example:

```bash
aws kms list-grants --key-id arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555
```

A grant can allow a service-linked role or AWS service principal to use the key even when the key policy looks restrictive. During incident response, this is important because the effective access path may be through a grant rather than a visible policy statement.

**Encryption context as a control**

Encryption context is additional authenticated data (AAD) that KMS binds to ciphertext. It is not secret, but it is integrity-protected. You can use it to constrain how ciphertext is decrypted.

Common uses:

- Tie ciphertext to an application name
- Bind data to a tenant identifier
- Restrict decryption to a specific resource or workflow

Example policy condition:

```json
{
  "Sid": "AllowDecryptOnlyForTenantA",
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::123456789012:role/TenantAAppRole"
  },
  "Action": "kms:Decrypt",
  "Resource": "*",
  "Condition": {
    "StringEquals": {
      "kms:EncryptionContext:tenant": "tenant-a"
    }
  }
}
```

If the application does not supply the expected encryption context, decryption fails. This is useful when you need a second factor of authorization at the data layer.

**Service integration and `kms:ViaService`**

`kms:ViaService` is one of the most useful condition keys for limiting KMS use to a specific AWS service endpoint. It helps ensure the key is used only when a service such as S3 or EBS calls KMS on behalf of the workload.

Examples of service-scoped use:

- S3 bucket encryption with SSE-KMS
- EBS volume encryption
- RDS storage encryption
- Lambda environment variable encryption
- CloudWatch Logs encryption

Example:

```json
{
  "Sid": "AllowS3Only",
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::123456789012:role/AppRole"
  },
  "Action": [
    "kms:Decrypt",
    "kms:Encrypt",
    "kms:GenerateDataKey*"
  ],
  "Resource": "*",
  "Condition": {
    "StringEquals": {
      "kms:ViaService": "s3.us-east-1.amazonaws.com"
    }
  }
}
```

This is especially valuable during exposure review because it prevents direct KMS API use from the same principal outside the intended service path.

**IAM policy review alongside the key policy**

KMS authorization is not complete until you review IAM. A principal may have permission in IAM, but still be blocked by the key policy. Conversely, a permissive key policy can make IAM permissions more dangerous than expected.

Check for:

- Roles with `kms:Decrypt` on `*`
- Managed policies attached to application roles
- Inline policies added during troubleshooting and never removed
- Permission boundaries that do not restrict KMS usage
- Session policies from `AssumeRole` that broaden access unexpectedly

Example IAM policy fragment to flag:

```json
{
  "Effect": "Allow",
  "Action": [
    "kms:Decrypt",
    "kms:ReEncryptFrom",
    "kms:ReEncryptTo"
  ],
  "Resource": "*"
}
```

In an incident, this is often too broad unless paired with strict resource ARNs and conditions.

**Cross-account access**

Cross-account KMS access is common for centralized logging, backup, analytics, and shared services. It is also a frequent source of accidental exposure.

Review:

- External account principals in the key policy
- External roles that can assume into your account
- Grants issued to external service principals
- Whether the external account still needs access after the incident

When you see cross-account access, verify the business purpose and the exact data path. If the key protects sensitive data that was exposed, you may need to revoke or disable the external path immediately.

**Multi-Region keys and replication**

Multi-Region keys can simplify disaster recovery and regional failover, but they also expand the review surface. You must check the primary key and each replica’s policy and grants.

Important points:

- Replicas have their own key policies and grants
- A change to one key does not automatically fix every replica
- Applications may use aliases that point to different regional keys
- Incident containment may require action in more than one region

If the exposed data spans regions, confirm which replica actually protected the data at rest and whether any regional service integration used a different key.

**Aliases and operational verification**

Aliases are convenient, but they are not proof of the current backing key by themselves. An alias can be repointed, so you should always verify the `TargetKeyId`.

Practical workflow:

1. List aliases.
2. Identify the alias used by the application.
3. Confirm the current `TargetKeyId`.
4. Compare that key ID with the one in the affected resource configuration.

Example:

```bash
aws kms list-aliases --query "Aliases[?AliasName=='alias/prod-data'].[AliasName,TargetKeyId]" --output table
```

If the alias is repointed during an incident, the application may begin using a different key than the one you first investigated. That can complicate containment and forensic timelines.

**CloudTrail and detective review**

CloudTrail helps you reconstruct KMS usage during exposure. Look for:

- `Decrypt`
- `Encrypt`
- `GenerateDataKey`
- `ReEncrypt`
- `CreateGrant`
- `RetireGrant`
- `DisableKey`
- `ScheduleKeyDeletion`
- `PutKeyPolicy`

Useful filters include the key ID, principal ARN, and event time window around the exposure.

Example CloudTrail lookup with the AWS CLI:

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=Decrypt \
  --start-time 2026-04-25T00:00:00Z \
  --end-time 2026-04-26T00:00:00Z
```

CloudTrail is especially useful for confirming whether a suspicious principal actually attempted decryption or only had theoretical access.

**Containment actions**

During a sensitive data exposure, you may need to reduce risk before completing a full root-cause analysis. Common containment options include:

- Disable the key if the business impact is acceptable
- Remove or narrow the offending key policy statement
- Revoke grants that are no longer needed
- Detach or reduce IAM permissions from affected principals
- Rotate application credentials that could call KMS
- Re-encrypt data under a new key if compromise is suspected

Example of disabling a key:

```bash
aws kms disable-key --key-id arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555
```

> Disabling a key can break dependent services immediately. Use it when exposure risk outweighs availability impact.

**Policy hardening patterns**

After containment, tighten the policy so the same exposure path cannot recur.

Recommended patterns:

- Use specific principals, not account-wide wildcards
- Prefer `kms:ViaService` for service-integrated workloads
- Add encryption context conditions for application-level binding
- Separate admin and data-plane roles
- Limit `CreateGrant` to trusted automation only
- Review and remove stale grants regularly
- Use aliases for operational convenience, but validate the backing key in automation

Example of a safer admin statement:

```json
{
  "Sid": "AllowKeyAdministrationToSecurityRole",
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::123456789012:role/SecurityKmsAdminRole"
  },
  "Action": [
    "kms:DescribeKey",
    "kms:EnableKey",
    "kms:DisableKey",
    "kms:PutKeyPolicy",
    "kms:UpdateKeyDescription",
    "kms:ScheduleKeyDeletion",
    "kms:CancelKeyDeletion"
  ],
  "Resource": "*"
}
```

This separates administrative control from data access, which is a core security design principle.

**Practical review checklist**

Use this checklist when a sensitive data exposure involves KMS:

- Identify the exact key or keys protecting the exposed data.
- Confirm the key state with `DescribeKey`.
- Retrieve the key policy and inspect principals, actions, and conditions.
- Review IAM policies for all relevant users, roles, and service roles.
- List grants and verify whether any are stale or overly broad.
- Check CloudTrail for recent KMS usage and policy changes.
- Validate aliases and confirm the current `TargetKeyId`.
- Review cross-account and multi-region access paths.
- Contain by narrowing policy, revoking grants, or disabling the key if needed.
- Re-encrypt or rotate data protection where appropriate.

**Exam focus**

For AWS Certified Security – Specialty, expect questions that ask you to choose the fastest safe response to a KMS-related exposure. The best answer usually balances containment with service impact.

Look for these clues:

- A role can decrypt data even though the application should only encrypt
- An alias points to a key, but the backing key may have changed
- A service integration uses KMS through a grant
- A policy is too broad because it uses wildcard principals or actions
- A condition such as `kms:ViaService` or encryption context can reduce blast radius

The key idea is to trace effective access, not just visible permissions. In KMS, the real answer is often in the combination of key policy, IAM, grants, and service context.
