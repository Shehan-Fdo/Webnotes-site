## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Incident response in AWS: the full-stack view**

A full-stack AWS security incident spans identity, network, compute, storage, data, logging, and governance. For exam scenarios, you need to decide what to contain first, what evidence to preserve, and how to recover without destroying forensic value.

The core workflow is:

- Detect suspicious activity from telemetry such as CloudTrail, GuardDuty, Security Hub, VPC Flow Logs, and application logs.
- Triage the blast radius across accounts, regions, workloads, and identities.
- Contain the incident while preserving evidence.
- Investigate root cause and attacker actions.
- Eradicate persistence and recover safely.
- Improve controls, playbooks, and detection coverage.

**Incident response governance and process artifacts**

Strong incident response is not just technical containment. It also depends on documented process artifacts that tell you who does what, when, and how.

- **Playbooks** define repeatable actions for common scenarios such as IAM credential compromise, public S3 exposure, ransomware, or container compromise.
- **Escalation paths** specify who to notify, in what order, and under what severity thresholds.
- **Severity classification** helps you prioritize response based on impact, scope, data sensitivity, and business criticality.
- **Communication plans** separate technical response from executive, legal, compliance, and customer messaging.
- **Evidence handling procedures** define how you preserve logs, snapshots, and metadata with chain-of-custody discipline.
- **AWS Support and AWS Customer Incident Response Team (CIRT)** coordination is appropriate when you need service-side guidance, account recovery help, or escalation for severe incidents.

A practical playbook usually includes:

1. Trigger conditions and detection sources.
2. Initial triage questions.
3. Containment actions and approval requirements.
4. Evidence preservation steps.
5. Investigation queries and pivot points.
6. Recovery validation checks.
7. Post-incident lessons learned.

> If you change or delete evidence too early, you may lose the ability to prove what happened, when it happened, and which identities or resources were involved.

**Initial triage: scope, impact, and trust boundaries**

Start by identifying the affected trust boundary. In AWS, that often means asking whether the issue is limited to one IAM principal, one workload, one account, one region, or an entire organization.

- Check whether the suspicious activity is tied to an IAM user, role session, federated identity, root user, or workload identity.
- Determine whether the attacker has persistence through access keys, role trust policies, Lambda triggers, event source mappings, scheduled tasks, or container orchestration objects.
- Identify whether data exposure is read-only, destructive, or includes privilege escalation.
- Confirm whether logs are still intact and whether the attacker may have tampered with them.

Useful first-pass evidence sources include:

- **CloudTrail** for API activity and identity actions.
- **GuardDuty** for threat detections and anomaly signals.
- **Security Hub** for aggregated findings and control posture.
- **AWS Config** for resource configuration history.
- **VPC Flow Logs** for network-level traffic patterns.
- **CloudWatch Logs** for application and service logs.
- **ALB/NLB access logs**, **CloudFront logs**, and **WAF logs** for edge activity.

**Containment strategy by layer**

Containment should reduce attacker freedom without destroying evidence. In many cases, you isolate first and clean up later.

**Identity containment**

- Disable or delete compromised IAM access keys.
- Remove suspicious inline policies, managed policy attachments, or trust policy changes.
- Revoke active sessions where possible by changing the underlying credentials or role trust.
- Rotate secrets in AWS Secrets Manager or Systems Manager Parameter Store if they may be exposed.
- Review and remove unexpected IAM role assumptions, especially cross-account trust.

Example CLI actions:

```bash
aws iam update-access-key \
  --user-name compromised-user \
  --access-key-id AKIAEXAMPLE \
  --status Inactive

aws iam delete-access-key \
  --user-name compromised-user \
  --access-key-id AKIAEXAMPLE
```

For federated or role-based compromise, focus on the source identity provider, session duration, and trust relationships. If the attacker can still assume a role, key rotation alone is not enough.

**Account and organization containment**

In multi-account environments, AWS Organizations and Control Tower are central to blast-radius reduction.

- Use **SCPs** (service control policies) to deny dangerous actions across an OU or account.
- Move a compromised account into a quarantine OU with restrictive SCPs.
- Use a delegated administrator account for centralized security services so you can investigate without relying on the compromised account.
- Isolate the account by removing trust relationships, cross-account roles, and shared network attachments.
- Consider detaching the account from shared services if the compromise is spreading through shared infrastructure.

Example quarantine SCP pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyMostActions",
      "Effect": "Deny",
      "NotAction": [
        "cloudtrail:*",
        "config:*",
        "guardduty:*",
        "securityhub:*",
        "iam:Get*",
        "iam:List*",
        "sts:GetCallerIdentity"
      ],
      "Resource": "*"
    }
  ]
}
```

This kind of policy is useful for containment, but you must ensure it does not block your own investigation or recovery actions.

**Network containment**

- Remove or tighten security group rules.
- Replace broad NACLs (network ACLs) with restrictive rules if you need coarse isolation.
- Detach or restrict internet gateways, NAT gateways, or route table paths if the workload is actively exfiltrating.
- Use AWS Network Firewall or WAF to block known malicious destinations or patterns.
- Isolate compromised EC2 instances in a quarantine subnet.

Example security group hardening:

```bash
aws ec2 revoke-security-group-ingress \
  --group-id sg-0123456789abcdef0 \
  --protocol tcp \
  --port 22 \
  --cidr 0.0.0.0/0
```

**Compute containment**

- Stop or isolate compromised EC2 instances if you need to preserve disk state.
- Snapshot volumes before termination if you need forensic analysis.
- Detach instance profiles if the instance role is abused.
- For Auto Scaling groups, suspend scaling processes if they would replace or reintroduce compromised nodes.
- For ECS, drain or stop tasks and review task roles and secrets.
- For EKS, cordon and drain nodes, isolate worker nodes, and inspect service account permissions.

**Storage containment**

- Restrict access to S3 buckets with bucket policies, SCPs, or explicit denies.
- Enable or verify Block Public Access.
- Remove suspicious replication rules, lifecycle rules, or bucket notifications.
- Lock down EBS snapshot sharing and AMI sharing if images or snapshots may be exposed.
- Review EFS and FSx access paths if shared file systems are involved.

**Serverless containment**

Lambda incidents often involve stolen credentials, overly permissive execution roles, or malicious event sources.

- Disable event source mappings to stop repeated invocation.
- Remove or quarantine triggers such as S3 notifications, EventBridge rules, or API Gateway routes.
- Reduce execution role permissions and inspect environment variables for secrets.
- Check for unauthorized layers, code updates, or alias changes.

Example:

```bash
aws lambda update-event-source-mapping \
  --uuid a1b2c3d4-5678-90ab-cdef-EXAMPLE \
  --enabled false
```

**Container and Kubernetes containment**

For ECS and EKS, think in terms of workload identity, node isolation, and control-plane visibility.

- In ECS, stop suspicious tasks and inspect the task role, execution role, and secrets references.
- In EKS, revoke or rotate service account tokens by restarting pods and reviewing IAM Roles for Service Accounts (IRSA).
- Cordon and drain compromised nodes to prevent new workloads from landing there.
- Review cluster role bindings, admission controllers, and image pull secrets.
- Check for malicious DaemonSets, privileged pods, or hostPath mounts.

Example EKS node isolation:

```bash
kubectl cordon ip-10-0-1-23.ec2.internal
kubectl drain ip-10-0-1-23.ec2.internal --ignore-daemonsets --delete-emptydir-data
```

**Detection and investigation workflow**

GuardDuty and Security Hub often provide the first actionable signal. The exam expects you to know how to pivot from a finding into deeper evidence.

A practical workflow is:

1. Read the finding type, affected resource, and principal.
2. Identify the time window and region.
3. Pivot to CloudTrail for API calls around the event.
4. Use AWS Config to see what changed in resource configuration.
5. Use VPC Flow Logs or application logs to confirm network or app behavior.
6. Correlate identities, resources, and timestamps in Detective or a log analytics tool.

**GuardDuty to CloudTrail example**

If GuardDuty reports `UnauthorizedAccess:IAMUser/InstanceCredentialExfiltration`, you should:

- Identify the instance profile or role involved.
- Review CloudTrail for `AssumeRole`, `GetCallerIdentity`, `ListBuckets`, `GetObject`, or unusual `iam:*` calls.
- Check whether the instance metadata service was accessed unexpectedly.
- Inspect network egress from the instance for exfiltration patterns.

**Security Hub to Config example**

If Security Hub flags an S3 bucket as public:

- Use AWS Config to see when the bucket policy changed.
- Review CloudTrail for `PutBucketPolicy`, `PutPublicAccessBlock`, or ACL changes.
- Confirm whether the exposure was intentional, temporary, or malicious.
- Check access logs for object reads during the exposure window.

**AWS Detective for graph-based correlation**

AWS Detective helps correlate identities, API calls, and network activity into a graph. It is useful when you need to answer questions like:

- Which principal first touched the resource?
- What other resources did that principal access?
- Did the same IP address or role session touch multiple accounts?
- Was the activity consistent with normal behavior?

Detective is especially helpful when you need to move from a single finding to a broader attack path.

**Log analysis choices: CloudTrail Lake, Athena, and CloudWatch Logs Insights**

Choose the right tool based on where the data lives and how you need to query it.

| Tool | Best for | Strengths | Tradeoffs |
|---|---|---|---|
| CloudTrail Lake | CloudTrail-centric investigations | Managed event store, SQL queries, long retention options | Focused on CloudTrail data |
| Athena | Large-scale queries over S3 logs | Flexible SQL over many log types | Requires data in S3 and schema management |
| CloudWatch Logs Insights | Fast queries over CloudWatch Logs | Quick operational investigation | Best for logs already in CloudWatch |

Use **CloudTrail Lake** when you need fast, centralized analysis of API activity across accounts and regions. Use **Athena** when you need to query multiple log sources stored in S3, such as CloudTrail, ALB logs, and VPC Flow Logs. Use **CloudWatch Logs Insights** when the relevant logs are already in CloudWatch and you need rapid operational filtering.

Example CloudWatch Logs Insights query:

```sql
fields @timestamp, @message
| filter @message like /AssumeRole|CreateAccessKey|PutBucketPolicy/
| sort @timestamp desc
| limit 50
```

Example Athena-style CloudTrail query pattern:

```sql
SELECT eventTime, eventName, userIdentity.arn, sourceIPAddress, requestParameters
FROM cloudtrail_logs
WHERE eventTime BETWEEN timestamp '2026-04-26 00:00:00'
  AND timestamp '2026-04-26 23:59:59'
  AND eventName IN ('CreateAccessKey', 'PutBucketPolicy', 'AssumeRole')
ORDER BY eventTime DESC;
```

**Forensic preservation and chain of custody**

Forensics in AWS is about preserving evidence in a way that remains credible and usable.

- Copy logs to a dedicated security account or evidence bucket.
- Use versioning and object lock where appropriate to support WORM-style retention.
- Restrict delete permissions and lifecycle rules on evidence stores.
- Preserve snapshots before modifying or terminating instances.
- Record who collected evidence, when, from where, and with what commands.
- Maintain timestamps, hashes, and notes for chain-of-custody integrity.

Important preservation actions include:

- **EBS snapshots** of attached volumes before remediation.
- **AMI creation** only if you understand it may alter state; snapshots are often preferred for evidence preservation.
- **S3 object versioning** and **Object Lock** for immutable retention.
- **CloudTrail log file validation** to detect tampering.
- **Centralized log archives** in a separate account with tightly controlled access.

Example evidence bucket configuration concepts:

```bash
aws s3api put-bucket-versioning \
  --bucket incident-evidence-bucket \
  --versioning-configuration Status=Enabled
```

> If the attacker has access to the same account and permissions as your evidence store, your forensic chain is weak. Separate accounts and immutable retention improve trust.

**KMS-specific incident response edge cases**

AWS Key Management Service (KMS) can complicate both containment and recovery.

- If a KMS key is disabled, encrypted data becomes unavailable until the key is re-enabled.
- If key policy changes remove access, you may block your own investigators and recovery workflows.
- If grants are abused, an attacker may retain access even after IAM cleanup.
- If you rotate or replace keys incorrectly, you can strand encrypted logs, snapshots, or databases.
- If you revoke access too aggressively, you may lose the ability to decrypt evidence.

Key response considerations:

- Review key policies, grants, and IAM permissions together.
- Check whether CloudTrail logs, EBS volumes, RDS snapshots, S3 objects, or Secrets Manager secrets depend on the key.
- Preserve a copy of the key policy and grant state before making changes.
- Remove suspicious grants after confirming they are not required for recovery.
- Be careful with disabling keys used for log encryption, because you may lose access to evidence.

Example grant review:

```bash
aws kms list-grants --key-id arn:aws:kms:us-east-1:123456789012:key/EXAMPLE
```

Example policy inspection:

```bash
aws kms get-key-policy \
  --key-id arn:aws:kms:us-east-1:123456789012:key/EXAMPLE \
  --policy-name default
```

**Service-specific investigation patterns**

**EC2 and EBS**

- Snapshot the root and data volumes before termination.
- Review user data, startup scripts, cron jobs, and systemd services.
- Check instance profile permissions and IMDS access patterns.
- Inspect `/var/log/secure`, `/var/log/auth.log`, and application logs if available.

**S3**

- Review bucket policy, ACLs, access points, replication, and public access block settings.
- Check server access logs or CloudTrail data events for object reads and writes.
- Look for mass deletion, unusual `PutObject`, or `GetObject` spikes.
- Verify whether versioning and MFA Delete were enabled.

**RDS and Aurora**

- Use snapshots for preservation before remediation.
- Review database audit logs, parameter groups, and security groups.
- Check for suspicious users, stored procedures, or privilege changes.
- Validate whether encryption keys and snapshot sharing settings are safe.

**Lambda**

- Inspect function code, layers, environment variables, and execution role.
- Disable event source mappings if the function is repeatedly triggered.
- Review CloudTrail for `UpdateFunctionCode`, `UpdateFunctionConfiguration`, and permission changes.
- Check for malicious destinations or exfiltration in outbound calls.

**ECS and EKS**

- Review task definitions, service roles, pod specs, and image sources.
- Check for privilege escalation through host networking, privileged containers, or mounted credentials.
- Isolate nodes and stop suspicious tasks or pods.
- Inspect cluster audit logs if enabled.

**Recovery and eradication**

Recovery should restore service without reintroducing the original weakness.

- Rebuild compromised hosts from known-good images.
- Recreate roles and policies from clean templates.
- Rotate secrets, certificates, and keys that may have been exposed.
- Re-enable services only after validating logging, monitoring, and access controls.
- Confirm that persistence mechanisms are removed before returning to production.

A safe recovery sequence often looks like this:

1. Preserve evidence.
2. Contain the incident.
3. Identify root cause and persistence.
4. Remove malicious changes.
5. Restore from trusted backups or clean deployments.
6. Rotate credentials and keys.
7. Validate monitoring and alerting.
8. Resume normal operations.

**Multi-account and shared-services considerations**

In AWS Organizations, incidents often cross account boundaries through shared networking, centralized logging, shared IAM roles, or delegated administration.

- A compromised workload account may still reach shared services if network and trust boundaries remain open.
- A centralized logging account should be protected from write tampering and unauthorized delete access.
- Delegated admin accounts for Security Hub, GuardDuty, or Detective can help you investigate without logging into the compromised account.
- Control Tower guardrails and SCPs help enforce baseline containment, but they do not replace workload-specific remediation.

Containment strategies in multi-account environments include:

- Quarantining the account in a restricted OU.
- Removing cross-account role trust.
- Detaching shared VPC or Transit Gateway attachments if needed.
- Blocking egress at centralized network controls.
- Preserving logs in a separate security account.

**Practical pivot examples**

If you see a suspicious `AssumeRole` event, ask:

- Which principal assumed the role?
- From which source IP and region?
- What did the session do next?
- Did it touch S3, KMS, IAM, or Secrets Manager?
- Did the same session create new access keys or modify trust policies?

If you see a suspicious S3 read spike, ask:

- Was the bucket public or accessed through a role?
- Did the access come from a known workload or an unfamiliar IP?
- Were objects encrypted with KMS, and was the key accessible?
- Did the access happen before or after a policy change?

If you see a suspicious Lambda invocation pattern, ask:

- Which event source triggered it?
- Was the event source mapping modified?
- Did the function role have permissions to exfiltrate data?
- Were environment variables or layers changed recently?

**Exam-oriented decision cues**

Use these cues to choose the best response in scenario questions:

- Choose **containment first** when the attacker is active and the blast radius is growing.
- Choose **preservation first** when you need evidence for legal, compliance, or root-cause analysis.
- Choose **CloudTrail Lake** for focused API investigation across accounts.
- Choose **Athena** for broad log analytics across S3-based logs.
- Choose **CloudWatch Logs Insights** for quick operational log searches.
- Choose **Detective** when you need relationship mapping across identities, resources, and events.
- Choose **SCPs and quarantine OUs** when you need organization-level containment.
- Choose **snapshotting and immutable retention** when evidence integrity matters.
- Choose **KMS review** when encryption may block access to logs, snapshots, or secrets.

**Common mistakes to avoid**

- Deleting compromised resources before capturing evidence.
- Rotating keys without checking for active sessions, grants, or trust policy abuse.
- Assuming one finding explains the whole incident.
- Forgetting cross-account access paths and shared services.
- Locking yourself out of logs by changing KMS or bucket policies too early.
- Treating container or serverless incidents as purely application bugs when identity and permissions are often the root cause.

**Quick reference: incident response checklist**

- Identify the affected account, region, and workload.
- Preserve logs, snapshots, and configuration state.
- Contain identity, network, compute, and data paths.
- Pivot from findings into CloudTrail, Config, and log analytics.
- Review KMS policies, grants, and encryption dependencies.
- Use Detective or equivalent correlation tools for attack-path analysis.
- Coordinate with AWS Support or CIRT when escalation is needed.
- Restore from trusted sources and validate before reopening access.
