## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Why this correlation matters**

When you investigate suspicious activity in AWS, no single data source gives the full picture. **VPC Flow Logs** show network traffic at the interface, subnet, or VPC level. **Amazon GuardDuty** turns telemetry into security findings. **EC2 metadata** and instance tags help you identify the workload behind an IP address or network interface. Correlating all three lets you answer the questions that matter during incident response:

- Which instance generated the traffic?
- Was the traffic expected for that workload?
- Did GuardDuty classify it as reconnaissance, exfiltration, or command-and-control?
- Is the source or destination internal, external, or a known AWS service?

This is a common exam theme because it tests whether you can move from raw telemetry to actionable investigation.

**Core data sources**

**VPC Flow Logs** capture information about IP traffic going to and from network interfaces in your VPC. They do not inspect packet payloads, but they are useful for identifying patterns such as repeated connections, rejected traffic, unusual ports, and unexpected destinations.

Typical fields include:

- `srcaddr` and `dstaddr` for source and destination IP addresses
- `srcport` and `dstport` for transport ports
- `protocol` for TCP, UDP, or ICMP
- `action` for `ACCEPT` or `REJECT`
- `bytes` and `packets` for volume analysis
- `start` and `end` for time correlation
- `interface-id` for mapping traffic back to an ENI

**GuardDuty** analyzes VPC Flow Logs, DNS logs, CloudTrail events, and other signals to detect threats. For this topic, the most relevant findings often involve:

- **Reconnaissance** such as port scanning or unusual connection attempts
- **Unauthorized access** attempts from suspicious IPs
- **Crypto mining** or command-and-control behavior
- **Data exfiltration** patterns, especially large outbound transfers to unusual destinations

**EC2 metadata** in this context usually means the information you use to identify the instance behind a network interface:

- Instance ID
- Private IP address
- Security groups
- Subnet and VPC
- Tags such as `Name`, `Environment`, or `Owner`
- Launch time and AMI details when you need to understand workload context

**How the correlation works**

The practical workflow is to start with a GuardDuty finding, extract the network indicators, and then map those indicators to Flow Logs and EC2 metadata.

A typical sequence looks like this:

1. Review the GuardDuty finding and note the affected IPs, ports, and timestamps.
2. Find the corresponding VPC Flow Logs entries for the same time window.
3. Map the `interface-id` or private IP to an EC2 instance or other ENI-attached resource.
4. Check instance tags, security groups, and workload purpose to determine whether the traffic is expected.
5. Decide whether the event is benign, suspicious, or clearly malicious.

This correlation is especially useful when the finding references an IP address rather than an instance name. In AWS, the IP alone is often not enough to identify the workload.

**Mapping Flow Logs to EC2 resources**

The most reliable join key is often the **network interface ID** (`eni-...`). Flow Logs can include the interface ID, and EC2 describes which instance currently uses that ENI.

Useful CLI commands:

```bash
aws ec2 describe-network-interfaces \
  --filters Name=addresses.private-ip-address,Values=10.0.1.25
```

```bash
aws ec2 describe-network-interfaces \
  --network-interface-ids eni-0123456789abcdef0
```

```bash
aws ec2 describe-instances \
  --filters Name=network-interface.addresses.private-ip-address,Values=10.0.1.25
```

If you already know the instance ID, you can inspect its interfaces and tags:

```bash
aws ec2 describe-instances \
  --instance-ids i-0123456789abcdef0 \
  --query 'Reservations[].Instances[].{InstanceId:InstanceId,PrivateIp:PrivateIpAddress,Tags:Tags,SGs:SecurityGroups}'
```

Important metadata to check:

- **Private IP address** to match Flow Logs and GuardDuty indicators
- **Security groups** to see whether the traffic should have been allowed
- **Subnet route table** to understand whether the instance has internet access
- **Tags** to identify the application owner or environment
- **Instance profile** if the workload may have been compromised and is calling AWS APIs

> Flow Logs show traffic at the network layer, but they do not tell you which process on the instance generated it. Use OS-level logs or EDR if you need process attribution.

**Using GuardDuty finding details effectively**

GuardDuty findings usually provide enough context to narrow the investigation quickly. Focus on:

- `resource.instanceDetails` or `resource.resourceType`
- `service.action.networkConnectionAction`
- `remoteIpDetails`
- `localIpDetails`
- `portProbeAction` for scan-related findings
- `threatListName` or `malware` indicators when present

A finding may show a remote IP, a local IP, and a port. You can use those values to search Flow Logs for the same time range.

Example investigation logic:

- If GuardDuty reports a port probe against `10.0.1.25:22`, search Flow Logs for `dstaddr=10.0.1.25` and `dstport=22`.
- If GuardDuty reports suspicious outbound traffic from `10.0.2.18` to an external IP, search for `srcaddr=10.0.2.18` and the destination IP.
- If the finding references a public IP, map it back to the ENI or instance with the corresponding private address and NAT path.

**Example Flow Logs analysis**

A simplified Flow Logs record might look like this:

```text
2 123456789012 eni-0abc123def4567890 10.0.1.25 203.0.113.10 44321 22 6 3 180 1714051200 1714051260 ACCEPT OK
```

Interpretation:

- `eni-0abc123def4567890` is the interface that saw the traffic
- `10.0.1.25` is the source private IP
- `203.0.113.10` is the destination IP
- `44321` is the ephemeral source port
- `22` is the destination port, commonly SSH
- `6` indicates TCP
- `ACCEPT` means the traffic was allowed by the security group and network ACL path
- The timestamps define the observation window

If GuardDuty flags SSH brute force or port probing, this record helps you confirm whether the traffic reached the instance and whether it was accepted or rejected.

**Practical correlation patterns**

**1. Reconnaissance or port scanning**

GuardDuty may detect repeated connection attempts to many ports or hosts. Flow Logs help you verify the pattern.

What to look for:

- Many `REJECT` entries from one source to one destination
- Many destination ports over a short time window
- Repeated attempts against common management ports such as `22`, `3389`, or `5985`

Example query idea in CloudWatch Logs Insights:

```sql
fields @timestamp, srcAddr, dstAddr, dstPort, action
| filter dstAddr = "10.0.1.25"
| sort @timestamp desc
| limit 50
```

If you store Flow Logs in Amazon S3 and query with Athena, you can group by source and destination to identify scan patterns.

**2. Suspicious outbound traffic**

GuardDuty may identify unusual outbound connections from an EC2 instance to an external IP. Correlate the destination with the instance’s role and tags.

Questions to ask:

- Is the destination a known software update endpoint or third-party API?
- Does the instance normally communicate with the internet?
- Is the traffic volume consistent with the workload?
- Does the security group allow outbound access broadly?

Useful checks:

- Review the instance’s `Name` and `Environment` tags
- Confirm whether the subnet uses a NAT gateway or public IP
- Check whether the destination IP belongs to a cloud provider, CDN, or known service

**3. Data exfiltration suspicion**

Large outbound byte counts in Flow Logs can support a GuardDuty exfiltration finding.

Look for:

- High `bytes` and `packets` values
- Long-lived connections
- Repeated outbound sessions to the same external destination
- Traffic that occurs outside normal business hours

Flow Logs alone do not prove exfiltration, but they can show the volume and destination pattern that supports the finding.

**4. Compromised instance making AWS API calls**

GuardDuty can correlate network behavior with CloudTrail activity. If an instance is compromised, you may see:

- Unusual outbound connections
- Suspicious IAM role usage
- API calls from the instance profile that do not match the workload

In this case, combine:

- Flow Logs for network destinations
- CloudTrail for API activity
- EC2 metadata and tags for workload identity
- IAM role permissions for blast radius assessment

**Example investigation workflow**

1. Open the GuardDuty finding and note the `instanceId`, private IP, remote IP, port, and timestamp.
2. Query Flow Logs for the same IPs and time window.
3. Confirm whether the traffic was `ACCEPT` or `REJECT`.
4. Use EC2 describe calls to map the IP or ENI to the instance.
5. Review tags, security groups, and route tables.
6. Decide whether the behavior matches the workload’s normal communication pattern.
7. If suspicious, isolate the instance, snapshot disks if needed, and preserve logs.

**Sample CLI workflow**

```bash
aws guardduty list-findings --detector-id d-1234567890abcdef0
```

```bash
aws guardduty get-findings \
  --detector-id d-1234567890abcdef0 \
  --finding-ids finding-id-1 finding-id-2
```

```bash
aws ec2 describe-network-interfaces \
  --filters Name=addresses.private-ip-address,Values=10.0.1.25
```

```bash
aws ec2 describe-instances \
  --filters Name=private-ip-address,Values=10.0.1.25
```

If Flow Logs are in CloudWatch Logs, a useful query pattern is:

```sql
fields @timestamp, interfaceId, srcAddr, dstAddr, srcPort, dstPort, action, bytes, packets
| filter srcAddr = "10.0.1.25" or dstAddr = "10.0.1.25"
| sort @timestamp desc
| limit 100
```

If Flow Logs are in Athena, a simplified SQL pattern is:

```sql
SELECT interface_id, srcaddr, dstaddr, srcport, dstport, action, bytes, packets, start, end
FROM vpc_flow_logs
WHERE srcaddr = '10.0.1.25'
   OR dstaddr = '10.0.1.25'
ORDER BY start DESC
LIMIT 100;
```

**EC2 metadata and context clues**

EC2 metadata is not just the instance ID. In investigations, context often comes from the surrounding configuration.

Check:

- **Security groups** for allowed inbound and outbound rules
- **Network ACLs** for subnet-level filtering
- **Route tables** for internet or on-premises reachability
- **Elastic IPs** if the instance has public exposure
- **Load balancers** if the traffic may be front-end related
- **Auto Scaling groups** if the instance is ephemeral and may have been replaced

A workload with a `Web` tag and inbound `443` from a load balancer is different from a database host with unexpected outbound `443` to an unknown IP. Tags help you decide whether the traffic is normal.

**Common exam distinctions**

**Flow Logs vs GuardDuty**

- Flow Logs provide raw network telemetry.
- GuardDuty provides analyzed findings and threat classification.
- Flow Logs are evidence; GuardDuty is detection.

**Flow Logs vs EC2 metadata**

- Flow Logs show traffic tied to an ENI.
- EC2 metadata identifies the compute resource behind that ENI.
- Metadata does not explain the traffic by itself, but it gives ownership and context.

**GuardDuty vs EC2 metadata**

- GuardDuty tells you what looks suspicious.
- EC2 metadata tells you where to investigate.
- Together they help you scope impact and ownership.

**Troubleshooting and limitations**

Be aware of these limitations when correlating data:

- Flow Logs do not show packet payloads, so you cannot inspect application content.
- GuardDuty findings are probabilistic detections, not absolute proof of compromise.
- Private IPs can be reused after instance termination, so always check timestamps.
- NAT gateways can obscure the original private source when you only look at public destinations.
- Some traffic may originate from managed services or AWS infrastructure rather than the instance itself.

> Always correlate by time window as well as IP address. IP reuse and ephemeral interfaces can otherwise lead to false attribution.

**A simple mental model**

Think of the investigation as three layers:

- **GuardDuty** answers: “What looks suspicious?”
- **Flow Logs** answer: “What network traffic actually occurred?”
- **EC2 metadata** answers: “Which workload owned the traffic?”

When you combine them, you can move from an alert to a defensible incident narrative.

**Quick reference checklist**

- Match the GuardDuty timestamp to the Flow Logs time window
- Map private IPs to `eni-...` IDs
- Confirm the instance ID and tags
- Review security groups and route tables
- Check whether the destination or port is expected
- Look for repeated attempts, unusual volume, or rejected traffic
- Escalate with CloudTrail and OS logs if process-level detail is needed

**Example incident summary**

A GuardDuty finding reports suspicious outbound SSH traffic from `10.0.1.25` to `203.0.113.10`. Flow Logs show repeated `ACCEPT` entries on destination port `22` from `eni-0abc123def4567890` during the same time window. EC2 metadata maps that ENI to instance `i-0123456789abcdef0`, tagged `Name=app-server-prod`. The security group allows outbound SSH to `0.0.0.0/0`, which is broader than expected for this workload. The combined evidence suggests the instance may be compromised or misconfigured, and you should isolate it and investigate further.
