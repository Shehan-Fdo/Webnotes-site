## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Network forensics workflow after an alert**

When an alert fires, your goal is to reconstruct what happened, identify the affected assets, and determine whether traffic was expected, blocked, or exfiltrated. In AWS, you usually combine control-plane evidence, data-plane telemetry, and topology validation.

A practical investigation often starts with the alert source, then moves to the instance, subnet, security group, route table, and any inspection layers such as Network Firewall, Gateway Load Balancer, or third-party appliances. You then correlate timestamps across logs to answer four questions:

- What resource generated the traffic?
- Where did it go?
- Was it allowed, denied, or modified?
- Is the path consistent with the intended architecture?

**Core evidence sources**

Use multiple telemetry sources because no single log gives the full picture. Each source answers a different part of the investigation.

- **VPC Flow Logs** show accepted or rejected IP traffic at the ENI, subnet, or VPC level.
- **CloudTrail** records API activity that may explain configuration changes before or after the alert.
- **AWS Config** helps you determine whether a security group, NACL, route table, or firewall policy changed.
- **GuardDuty** can provide the initial finding and context such as instance ID, remote IP, and finding type.
- **Traffic Mirroring** captures packet payloads when you need deeper inspection than flow metadata.
- **Reachability Analyzer** validates whether a path is possible based on current network configuration.
- **Network Firewall logs** can show traffic that was allowed, dropped, or alerted on by firewall policy.

**VPC Flow Logs: what they tell you**

VPC Flow Logs are one of the first places to look because they provide a low-friction record of network traffic metadata. They do not capture payloads, but they are useful for identifying source and destination addresses, ports, protocol, packet counts, and whether traffic was accepted or rejected.

A typical investigation uses Flow Logs to confirm whether the suspicious connection actually traversed the ENI of the workload you are investigating.

```text
2 123456789012 eni-0abc123def4567890 10.0.1.25 198.51.100.10 443 51544 6 10 840 1714060800 1714060860 ACCEPT OK
```

Key points:

- **ENI scope** matters because logs are tied to the network interface, not to the application.
- **ACCEPT** means the packet matched a rule path that allowed it; it does not prove the application processed it.
- **REJECT** can indicate a security group, network ACL, or routing issue depending on where the packet was stopped.
- **NAT Gateway traffic** is visible through the NAT Gateway’s ENI, so translated flows may appear with the NAT address rather than the original private source in downstream observations.
- **Asymmetric routing** can make one direction visible and the return path harder to correlate.

If you need to query logs at scale, use Amazon Athena over an S3 delivery bucket.

```sql
SELECT srcaddr, dstaddr, dstport, action, COUNT(*) AS hits
FROM vpc_flow_logs
WHERE action = 'ACCEPT'
  AND dstport IN (22, 3389, 443)
GROUP BY srcaddr, dstaddr, dstport, action
ORDER BY hits DESC;
```

**CloudTrail and configuration drift**

CloudTrail helps you determine whether the network behavior changed because someone modified the environment. It records API calls, the caller identity, source IP, request parameters, and response elements for supported services.

Look for changes to:

- **Security groups** such as `AuthorizeSecurityGroupIngress` or `RevokeSecurityGroupIngress`
- **Network ACLs** such as `CreateNetworkAclEntry` or `ReplaceNetworkAclEntry`
- **Route tables** such as `CreateRoute`, `ReplaceRoute`, or `DeleteRoute`
- **VPC endpoints** such as endpoint creation or policy updates
- **Firewall-related resources** such as policy or rule updates in services that support them

A useful pattern is to search for changes shortly before the alert timestamp.

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=AuthorizeSecurityGroupIngress \
  --start-time 2026-04-26T00:00:00Z \
  --end-time 2026-04-26T02:00:00Z
```

CloudTrail is especially valuable when the alert indicates a new exposure, unexpected egress, or a sudden drop in blocked traffic. It gives you the “who changed what” context that flow logs cannot provide.

**AWS Config for change history**

AWS Config is useful when you need a point-in-time view of resource configuration and compliance. It helps you answer whether a security group, route table, or NACL was already misconfigured before the alert.

Use it to:

- Compare current and previous resource states
- Identify the exact time a configuration changed
- Trace relationships between resources
- Support compliance and incident timelines

A common exam pattern is to use Config to confirm whether a security group rule was introduced before the suspicious traffic began.

**Reachability Analyzer for path validation**

Reachability Analyzer helps you determine whether a network path is possible from a source to a destination based on current configuration. It is not a packet capture tool; it is a static analysis tool that evaluates routes, security groups, NACLs, and other network controls.

Typical workflow:

1. Create a network insights path for the source and destination pair.
2. Start the analysis against that path.
3. Review the explanation for the first blocking component, if any.

Example:

```bash
aws ec2 create-network-insights-path \
  --source 10.0.1.25 \
  --destination 10.0.2.50 \
  --protocol tcp \
  --destination-port 443

aws ec2 start-network-insights-analysis \
  --network-insights-path-id nip-0123456789abcdef0
```

Use Reachability Analyzer when you need to prove whether a path should exist. It is especially helpful after an alert that suggests lateral movement, blocked exfiltration, or an unexpected open path between subnets.

**Traffic Mirroring for packet-level inspection**

Traffic Mirroring copies network traffic from an ENI to a target for deeper inspection. Use it when Flow Logs are not enough and you need packet payloads, protocol details, or application-layer clues.

Important setup considerations:

- You typically need a **traffic mirror target** first.
- You also need a **traffic mirror filter** to define what traffic is mirrored.
- The source is specified with `--network-interface-id`.
- The session description is often required in practical workflows for clarity and management.

Example session creation:

```bash
aws ec2 create-traffic-mirror-session \
  --network-interface-id eni-0abc123def4567890 \
  --traffic-mirror-target-id tmt-0123456789abcdef0 \
  --traffic-mirror-filter-id tmf-0123456789abcdef0 \
  --session-number 1 \
  --packet-length 128 \
  --virtual-network-interface-id eni-0fedcba9876543210 \
  --traffic-mirror-session-description "Mirror suspicious outbound traffic"
```

> Traffic Mirroring is most useful when you already know which ENI is suspicious and you have a collector ready. It is not a substitute for initial triage because it adds operational overhead and can generate large volumes of data.

**Network Firewall logs and inspection outcomes**

AWS Network Firewall can log traffic decisions and alerts, which is useful when the alert may involve inspection rules or policy enforcement. The logs are typically delivered in JSON and may include fields such as firewall name, availability zone, event time, and event details.

Use these logs to determine whether traffic was:

- **Allowed** by a rule
- **Dropped** by a rule
- **Alerted** on by a rule

Because log schemas can vary by delivery configuration and event type, avoid assuming a single fixed field layout. Focus on the decision outcome, rule association, and timestamps.

Example of a representative JSON-style record:

```json
{
  "firewall_name": "prod-egress-fw",
  "availability_zone": "us-east-1a",
  "event_timestamp": "2026-04-26T01:15:30Z",
  "event": {
    "type": "alert",
    "src_ip": "10.0.1.25",
    "src_port": 51544,
    "dest_ip": "198.51.100.10",
    "dest_port": 443,
    "protocol": "TCP"
  }
}
```

Use firewall logs together with Flow Logs to distinguish between traffic that reached the firewall and traffic that was later blocked or alerted on.

**NAT Gateway considerations**

NAT Gateways complicate attribution because they translate private source addresses to the NAT Gateway’s public address for outbound internet access. This means downstream logs may show the NAT Gateway as the apparent source.

Practical implications:

- Flow Logs on the workload ENI show the original private source.
- Flow Logs on the NAT Gateway ENI show the translated traffic.
- External destinations see the NAT public IP, not the private instance IP.
- Correlation requires timestamps, ports, and destination addresses.

This is important when an alert reports suspicious internet access. You should verify whether the traffic originated from a private subnet and whether the NAT path was expected.

**Security group, NACL, and route table triage**

When you investigate a suspicious connection, check the network controls in the order that best matches the symptom.

- **Security groups** are stateful and attached to ENIs.
- **Network ACLs** are stateless and evaluated at the subnet boundary.
- **Route tables** determine where traffic is sent next.
- **Internet gateways, NAT Gateways, and endpoints** affect the path and visibility of traffic.

A simple triage pattern:

- If traffic is unexpectedly allowed, inspect security group ingress and egress first.
- If traffic is intermittently blocked, inspect NACL rules and ephemeral port ranges.
- If traffic goes to the wrong destination, inspect route tables and endpoint routes.
- If traffic exits the VPC unexpectedly, inspect NAT and internet egress controls.

**Practical investigation sequence**

Use a repeatable sequence so you do not miss evidence.

1. Identify the alert source and timestamp.
2. Determine the affected resource, such as an instance, ENI, subnet, or firewall.
3. Check VPC Flow Logs for the relevant ENI and time window.
4. Review CloudTrail for recent network configuration changes.
5. Use AWS Config to confirm when the change occurred.
6. Validate the path with Reachability Analyzer.
7. If needed, enable Traffic Mirroring for packet-level inspection.
8. Review firewall and inspection logs for allow, drop, or alert outcomes.
9. Correlate all findings into a timeline.

**Example incident timeline**

A suspicious outbound connection alert might resolve like this:

- GuardDuty flags unusual outbound traffic from an EC2 instance.
- VPC Flow Logs show repeated `ACCEPT` records to a rare destination on port 443.
- CloudTrail shows a security group egress rule was widened earlier that day.
- AWS Config confirms the rule change and the exact time it occurred.
- Reachability Analyzer shows the path is now valid from the instance to the internet.
- Network Firewall logs show the traffic was allowed, not blocked.
- Traffic Mirroring is enabled to inspect the payload if deeper analysis is needed.

This sequence helps you separate expected behavior from misconfiguration, compromise, or policy bypass.

**CLI and query examples**

Use the AWS CLI to gather evidence quickly.

```bash
aws ec2 describe-network-interfaces \
  --network-interface-ids eni-0abc123def4567890
```

```bash
aws ec2 describe-security-groups \
  --group-ids sg-0123456789abcdef0
```

```bash
aws ec2 describe-route-tables \
  --filters Name=vpc-id,Values=vpc-0123456789abcdef0
```

```bash
aws ec2 describe-network-acls \
  --filters Name=vpc-id,Values=vpc-0123456789abcdef0
```

For log analysis, Athena is often the fastest way to correlate large volumes of Flow Logs.

```sql
SELECT interface_id, srcaddr, dstaddr, dstport, action, start, end
FROM vpc_flow_logs
WHERE interface_id = 'eni-0abc123def4567890'
  AND start BETWEEN 1714060800 AND 1714064400
ORDER BY start ASC;
```

**What exam questions usually test**

Expect questions that ask you to choose the best tool for the job.

- Use **Flow Logs** for traffic metadata and allow/deny visibility.
- Use **CloudTrail** for configuration and API change history.
- Use **AWS Config** for resource state and drift over time.
- Use **Reachability Analyzer** to prove whether a path exists.
- Use **Traffic Mirroring** for packet-level inspection.
- Use **Network Firewall logs** to understand inspection outcomes.
- Use **NAT Gateway context** to avoid misattributing source addresses.

**Common pitfalls**

- Do not assume Flow Logs show payloads; they do not.
- Do not assume a blocked connection means the application never retried through another path.
- Do not confuse translated NAT traffic with the original private source in downstream logs.
- Do not treat Reachability Analyzer as a live packet test.
- Do not rely on a single log source when the alert requires timeline reconstruction.

**Summary**

Network forensics in AWS is a correlation exercise. You start with the alert, identify the affected resource, and then combine flow metadata, configuration history, path analysis, and inspection logs to reconstruct the event. The strongest answers in the exam show that you know which tool answers which question and that you understand how NAT, ENIs, and inspection layers affect what you see.
