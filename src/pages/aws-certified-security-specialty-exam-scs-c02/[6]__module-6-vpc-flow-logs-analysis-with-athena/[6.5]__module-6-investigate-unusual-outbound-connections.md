## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Why outbound investigations matter**

Unusual outbound connections often indicate compromise, data theft, command-and-control (C2) activity, or misuse of credentials. In SCS-C02, you are expected to identify the right telemetry, interpret network patterns, and choose the least disruptive containment method that still stops the abuse.

A strong investigation usually answers four questions:

- What resource is sending traffic?
- Where is it going?
- What is the traffic pattern?
- What control can stop it without breaking the workload more than necessary?

**Common threat patterns to recognize**

Outbound abuse rarely looks random. It usually fits a few recognizable patterns that help you narrow the cause quickly.

- **C2 beaconing**: small, periodic connections to a stable destination, often with regular intervals and low byte counts.
- **Exfiltration**: large or sustained transfers, often to unfamiliar IPs, domains, or cloud storage endpoints.
- **Unusual DNS**: high query volume, long or random-looking subdomains, repeated NXDOMAIN responses, or lookups to suspicious domains.
- **Port probing**: one source contacting many ports or many destinations in a short time, often indicating scanning or lateral discovery.
- **Crypto-mining**: persistent connections to mining pools, often on common mining ports or over TLS to known pool infrastructure.

GuardDuty is especially useful here because it classifies suspicious behavior into findings that map to likely attacker goals.

- **Backdoor:EC2/C&CActivity** suggests a host is communicating with a command-and-control server.
- **UnauthorizedAccess:EC2/TorClient** can indicate anonymized outbound traffic that may hide malicious activity.
- **CryptoCurrency:EC2/BitcoinTool.B!DNS** or similar crypto-mining findings point to mining-related DNS or network behavior.
- **Trojan:EC2/BlackholeTraffic!DNS** and related DNS findings often indicate malicious domain resolution.
- **Recon:EC2/PortProbeUnprotectedPort** or similar port-probe findings indicate scanning behavior.
- **Exfiltration**-style findings, when present, suggest data leaving the environment in a suspicious pattern.
- **DNS-related findings** are important when the attacker uses domain lookups to locate infrastructure or tunnel data.

> GuardDuty finding names can vary by resource type and behavior family. On the exam, focus on the behavior class and the likely response, not exact punctuation.

**Start with the source of truth**

The first step is to identify the originating workload and its network path. In AWS, the same outbound connection can be observed in several places, and each source gives different context.

- **VPC Flow Logs** show accepted and rejected network flows at the ENI, subnet, or VPC level.
- **CloudWatch Logs Insights** helps you query flow logs quickly for patterns.
- **Athena** is useful when flow logs are stored in S3 and you need flexible SQL analysis.
- **GuardDuty** provides threat classification and prioritization.
- **CloudTrail** helps determine whether a configuration or credential event preceded the traffic.
- **Application logs** show whether the workload itself initiated the connection or whether the traffic is unexpected.
- **Load balancer logs** help when the workload is behind Elastic Load Balancing and the outbound traffic is tied to a specific target instance or IP.

**VPC Flow Logs for outbound triage**

VPC Flow Logs are the primary network evidence source for many exam scenarios. They record metadata, not payloads, so they are best for identifying patterns rather than content.

Useful fields include:

- Source and destination addresses and ports
- Protocol
- Packet and byte counts
- Action (`ACCEPT` or `REJECT`)
- Time window
- ENI identifier

A typical workflow is:

1. Identify the suspicious destination or source from GuardDuty, DNS logs, or an alert.
2. Query flow logs for the ENI, destination IP, or destination port.
3. Look for repetition, timing, and volume.
4. Correlate the ENI back to an EC2 instance, NAT gateway, load balancer target, or container task.

Example CloudWatch Logs Insights query for repeated outbound connections:

```sql
fields @timestamp, srcAddr, srcPort, dstAddr, dstPort, protocol, action, bytes, packets, interfaceId
| filter action = "ACCEPT"
| filter srcAddr = "10.0.1.25"
| stats count() as connections, sum(bytes) as totalBytes, sum(packets) as totalPackets by dstAddr, dstPort, protocol
| sort connections desc
| limit 20
```

Example query for beaconing-like behavior:

```sql
fields @timestamp, srcAddr, dstAddr, dstPort, bytes
| filter action = "ACCEPT"
| filter srcAddr = "10.0.1.25"
| stats count() as hits, min(@timestamp) as firstSeen, max(@timestamp) as lastSeen, avg(bytes) as avgBytes by dstAddr, dstPort, bin(5m)
| sort hits desc
```

What to look for:

- Many small connections to the same destination at regular intervals
- Repeated DNS lookups followed by short-lived TCP sessions
- A destination that appears only after a recent change or credential event
- Large byte counts to a destination that is not part of the workload’s normal dependencies

**Athena patterns for deeper analysis**

Athena is useful when you need longer retention, joins, or more complex grouping than CloudWatch Logs Insights comfortably provides.

Common analysis patterns include:

- Grouping by destination to find top talkers
- Grouping by time bucket to spot periodic traffic
- Filtering on rejected traffic to identify blocked attempts
- Joining flow logs with instance metadata or tags to identify the owning application

Example Athena query for repeated destinations:

```sql
SELECT
  dstaddr,
  dstport,
  protocol,
  COUNT(*) AS connections,
  SUM(bytes) AS total_bytes
FROM vpc_flow_logs
WHERE srcaddr = '10.0.1.25'
  AND action = 'ACCEPT'
GROUP BY dstaddr, dstport, protocol
ORDER BY connections DESC
LIMIT 20;
```

Example query for possible exfiltration:

```sql
SELECT
  srcaddr,
  dstaddr,
  dstport,
  SUM(bytes) AS total_bytes,
  COUNT(*) AS connections
FROM vpc_flow_logs
WHERE action = 'ACCEPT'
  AND bytes > 1000000
GROUP BY srcaddr, dstaddr, dstport
ORDER BY total_bytes DESC;
```

Example query for repeated destination lookups or short bursts:

```sql
SELECT
  dstaddr,
  COUNT(*) AS hits,
  COUNT(DISTINCT dstport) AS ports_seen
FROM vpc_flow_logs
WHERE action = 'ACCEPT'
GROUP BY dstaddr
HAVING COUNT(*) > 100
ORDER BY hits DESC;
```

**CloudWatch Logs and application evidence**

Flow logs tell you that traffic happened. Application logs tell you whether it was expected.

Use application logs to confirm:

- The process or service that initiated the connection
- The API call or library action that triggered it
- Whether the destination is part of a known dependency
- Whether the workload is retrying due to failure or timeout

CloudWatch Logs is often the fastest place to check:

- EC2 application logs shipped by the CloudWatch agent
- Lambda logs in `/aws/lambda/<function-name>`
- ECS task logs sent to CloudWatch Logs
- EKS container logs via Fluent Bit or similar collectors
- ALB access logs or application logs forwarded to CloudWatch for correlation

A practical exam pattern is:

- Flow logs show outbound traffic from an instance.
- Application logs show a new process or unexpected library call.
- GuardDuty flags C2 or exfiltration behavior.
- You conclude the instance is compromised rather than merely misconfigured.

**Investigating public-facing services behind Elastic Load Balancing**

When a public-facing service sits behind an Application Load Balancer (ALB) or Network Load Balancer (NLB), the outbound traffic may come from the target instances, containers, or downstream services rather than the load balancer itself.

Key points:

- ALB and NLB are usually not the source of application egress.
- The target ENI on the instance or task is often where the suspicious outbound traffic originates.
- You must map the load balancer target back to the underlying compute resource.

Useful evidence sources:

- **ALB access logs**: show client requests, target responses, and request patterns.
- **NLB access logs**: show connection-level behavior for TCP/TLS traffic.
- **Target group health and registration**: help identify which instances or IP targets were active.
- **VPC Flow Logs on target ENIs**: show the actual outbound destination traffic.
- **Application logs**: show whether the request triggered a backend call or data export.

A typical correlation flow is:

1. Identify the suspicious backend instance or IP from flow logs.
2. Check the target group to confirm the instance was serving traffic.
3. Review ALB or NLB logs for the inbound request pattern that preceded the outbound activity.
4. Inspect application logs for the code path that generated the outbound connection.
5. Determine whether the outbound traffic is a legitimate backend dependency or malicious behavior.

Example ALB access log clues:

- A spike in requests to a specific endpoint before outbound traffic begins
- Repeated requests from a single client causing backend retries
- Large response sizes or unusual status codes that may trigger data processing or export logic

Example NLB clue:

- A backend service using TCP or TLS may show connection churn, which can be correlated with flow logs and application logs to identify the exact target.

**DNS as a major signal**

DNS is often the first place suspicious outbound behavior appears. Attackers use DNS for discovery, C2, and data tunneling.

Watch for:

- High query volume to a single domain
- Random-looking subdomains
- Repeated failed lookups
- Queries for domains that are not part of the workload’s normal dependencies
- DNS requests that precede outbound connections to the resolved IPs

Practical controls and logs:

- **Route 53 Resolver query logs** capture DNS queries from VPCs.
- **VPC DNS Firewall** can block or allow DNS queries at the VPC level.
- **Route 53 Resolver DNS Firewall** is the managed control used to define domain lists and rules.
- **GuardDuty DNS findings** can highlight suspicious domain resolution patterns.

Example suspicious DNS pattern:

- `a8f3k2j9.example-malicious-domain.com`
- Many unique subdomains
- Short TTLs
- Followed by outbound HTTPS to the resolved address

Example normal DNS pattern:

- `sts.amazonaws.com`
- `s3.us-east-1.amazonaws.com`
- `ssm.us-east-1.amazonaws.com`
- Stable, expected service endpoints used by the workload

**Normal vs suspicious outbound patterns for common AWS services**

You often need to distinguish legitimate AWS service traffic from abuse.

| Service | Usually normal pattern | Suspicious pattern |
|---|---|---|
| **S3** | Access to known bucket endpoints, often for backups, artifacts, or logs | Large unexpected transfers to unknown buckets or unusual regions |
| **STS** | Short-lived calls for role assumption, token refresh, or federation | Excessive or repeated calls from an unusual host, especially after credential compromise |
| **ECR** | Pulling container images during deployment or scale-out | Repeated image pulls from unexpected hosts or outside deployment windows |
| **Systems Manager** | Connections to `ssm`, `ec2messages`, and `ssmmessages` endpoints for patching or session management | Missing endpoint access causing fallback to internet egress, or unexpected SSM use from a compromised host |

Interpretation tips:

- A workload that uses **STS** should usually show short bursts, not continuous high-volume traffic.
- **S3** traffic is often bursty and data-heavy, but it should align with known jobs, backups, or application behavior.
- **ECR** traffic is usually tied to deployment or instance startup.
- **Systems Manager** traffic should align with managed instance operations, patching, or session activity.

**Practical containment options**

Containment should stop the traffic while preserving enough access for investigation. Security groups are often the first control, but they are not always sufficient.

Options include:

- **Security groups**: remove or narrow outbound rules when you can safely isolate the host.
- **Network ACLs (NACLs)**: block traffic at the subnet boundary when you need broader containment.
- **Route table changes**: remove or redirect the default route to stop internet egress.
- **VPC endpoint restrictions**: force traffic through private endpoints and remove public internet paths.
- **AWS Network Firewall**: block suspicious domains, IPs, or protocols at the VPC edge.
- **VPC DNS Firewall / Route 53 Resolver DNS Firewall**: block suspicious domains before resolution.
- **Instance isolation**: move the host to a quarantine subnet or detach it from the application path.

> If the workload uses multiple ENIs or shared subnets, changing only one security group may not stop all egress. Consider subnet-level or route-level containment.

Example containment decision:

- If the host is clearly compromised and business impact is acceptable, isolate it with a quarantine security group and remove internet routes.
- If you need to preserve the instance for forensics, prefer NACLs, route changes, or firewall rules that block only the suspicious destination.
- If the traffic is DNS-based, block the domain with DNS Firewall before blocking the IP, because the attacker may rotate addresses.

**AWS Network Firewall and DNS firewall controls**

These controls are practical when the suspicious destination is known or can be expressed as a domain or IP set.

Use **AWS Network Firewall** when you need:

- Stateful inspection
- Domain or IP-based egress filtering
- Centralized control across multiple subnets or VPCs
- Logging of allowed and blocked traffic

Use **VPC DNS Firewall** or **Route 53 Resolver DNS Firewall** when you need:

- To stop resolution of malicious domains
- To prevent workloads from learning attacker-controlled IPs
- To enforce domain allowlists for sensitive environments

Example DNS firewall rule group concept:

```text
Block:
- *.bad-domain.example
- *.mining-pool.example
- *.dynamic-c2.example

Allow:
- sts.amazonaws.com
- s3.amazonaws.com
- ssmmessages.us-east-1.amazonaws.com
```

**Traffic Mirroring and packet-level forensics**

Traffic Mirroring copies ENI traffic to an analysis appliance. Use it when flow logs are not enough and you need packet-level visibility.

Choose Traffic Mirroring when:

- You need to inspect payloads, protocols, or TLS handshake details
- You suspect custom C2, tunneling, or encrypted exfiltration
- You want to feed traffic into IDS tools such as **Suricata** or **Zeek**
- Flow logs show the destination but not enough detail to prove malicious behavior

Use flow logs instead when:

- You only need to identify destinations, ports, and volume
- You are triaging many hosts quickly
- You need a low-overhead, broad-scope view

How they complement each other:

- Flow logs identify the suspicious ENI and destination.
- Traffic Mirroring captures packets from that ENI.
- Suricata or Zeek analyzes the mirrored traffic for signatures, protocol anomalies, or suspicious DNS and TLS behavior.

**Example investigation workflow**

1. Review GuardDuty for findings related to C2, exfiltration, DNS anomalies, port probing, or crypto-mining.
2. Identify the source ENI, instance, task, or load balancer target.
3. Query VPC Flow Logs in CloudWatch Logs Insights or Athena for destination, port, bytes, and timing.
4. Check Route 53 Resolver query logs for suspicious domain lookups.
5. Review application logs and, if relevant, ALB or NLB access logs.
6. Determine whether the traffic matches a known dependency such as S3, STS, ECR, or Systems Manager.
7. Contain the host using the least disruptive effective control.
8. Preserve evidence and document the timeline.

**Example triage commands**

Find the instance attached to an ENI:

```bash
aws ec2 describe-network-interfaces \
  --network-interface-ids eni-0123456789abcdef0 \
  --query 'NetworkInterfaces[0].Attachment.InstanceId'
```

Check security group egress rules:

```bash
aws ec2 describe-security-groups \
  --group-ids sg-0123456789abcdef0
```

Inspect a target group to map a load balancer target:

```bash
aws elbv2 describe-target-health \
  --target-group-arn arn:aws:elasticloadbalancing:us-east-1:123456789012:targetgroup/app-tg/abc123
```

Review GuardDuty findings for suspicious outbound behavior:

```bash
aws guardduty list-findings \
  --detector-id <detector-id> \
  --finding-criteria '{"Criterion":{"service.action.networkConnectionAction.connectionDirection":{"Eq":["OUTBOUND"]}}}'
```

**Preventive guardrails and automation**

Outbound abuse is easier to stop when you prevent risky network changes and credential misuse.

Useful preventive controls include:

- **AWS Organizations SCPs** to prevent disabling logging, changing firewall policies, or creating overly permissive network paths.
- **IAM guardrails** to restrict who can modify security groups, route tables, NACLs, and DNS firewall rules.
- **AWS Config** rules to detect risky changes such as open egress rules, missing flow logs, or public route exposure.
- **Security Hub** to aggregate findings and trigger automated response workflows.
- **EventBridge** to route GuardDuty or Config events to Lambda, Systems Manager Automation, or incident response tooling.

Examples of risky changes to detect:

- Security group egress set to `0.0.0.0/0` without justification
- New internet gateway route on a sensitive subnet
- DNS firewall rule group removed or bypassed
- Flow logs disabled on a critical VPC
- VPC endpoint policy widened to allow unintended access

**Exam-oriented interpretation**

For SCS-C02, the best answer is usually the one that matches the evidence source to the control objective.

- Use **GuardDuty** to identify the likely malicious behavior.
- Use **VPC Flow Logs** to confirm source, destination, and volume.
- Use **DNS logs** to validate domain-based abuse.
- Use **application and load balancer logs** to connect the traffic to the workload path.
- Use **Network Firewall** or **DNS Firewall** to block known bad destinations.
- Use **security groups, NACLs, route tables, or endpoint policies** to contain the host or subnet.
- Use **Traffic Mirroring** when you need packet-level proof or IDS analysis.

A good exam answer usually avoids overreacting. If the traffic is suspicious but the workload must stay online, prefer targeted egress controls and evidence collection over immediate full isolation. If the host is clearly compromised, isolate first and investigate second.

**Quick reference: what to use when**

- **Flow logs**: identify who talked to whom, on what port, and how much.
- **CloudWatch Logs Insights**: fast pattern hunting over recent flow logs.
- **Athena**: deeper historical analysis and joins.
- **GuardDuty**: threat classification and prioritization.
- **Route 53 Resolver query logs**: DNS investigation.
- **DNS Firewall**: block suspicious domains.
- **AWS Network Firewall**: block suspicious IPs, domains, or protocols centrally.
- **ALB/NLB logs**: correlate inbound requests with backend behavior.
- **Application logs**: confirm whether the workload initiated the traffic.
- **Traffic Mirroring**: packet-level inspection and IDS integration.
- **SCPs, IAM, Config, Security Hub**: prevent recurrence and detect risky changes.
