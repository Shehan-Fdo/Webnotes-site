## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Athena and VPC Flow Logs overview**

Amazon Athena is a serverless query service that lets you analyze data in Amazon S3 using standard SQL. For VPC Flow Logs, Athena is useful because flow logs are commonly delivered to S3 and can be queried without loading data into a database or cluster.

VPC Flow Logs capture metadata about network traffic to and from network interfaces in a VPC. They are especially valuable for security investigations because they help you answer questions such as:

- Which source and destination addresses communicated?
- Which ports and protocols were used?
- Was traffic accepted or rejected?
- Did a security group or network ACL block the flow?

Athena works best when the table definition matches the actual S3 object layout. If the table schema, delimiters, or partition structure are wrong, queries may return incomplete results or fail to parse records correctly.

**Flow log delivery formats and parsing considerations**

VPC Flow Logs are often delivered as plain text files in S3, with fields separated by spaces. The exact format depends on the flow log version and delivery configuration. In practice, you should define the table to match the log format you receive, including the correct field order and any optional fields.

A common approach is to use a text table with a space-delimited SerDe. For example:

```sql
CREATE EXTERNAL TABLE IF NOT EXISTS vpc_flow_logs (
  version int,
  account_id string,
  interface_id string,
  srcaddr string,
  dstaddr string,
  srcport int,
  dstport int,
  protocol int,
  packets bigint,
  bytes bigint,
  start bigint,
  end bigint,
  action string,
  log_status string
)
PARTITIONED BY (
  year string,
  month string,
  day string
)
ROW FORMAT DELIMITED
FIELDS TERMINATED BY ' '
STORED AS TEXTFILE
LOCATION 's3://my-flow-log-bucket/AWSLogs/123456789012/vpcflowlogs/'
TBLPROPERTIES ('skip.header.line.count'='0');
```

Use the exact field list that matches your flow log version. If your logs include additional fields such as `tcp_flags`, `type`, `pkt_srcaddr`, `pkt_dstaddr`, `region`, or `az_id`, add them in the correct order.

> Be careful with assumptions about parsing. Flow logs are space-delimited text, but the table definition must still match the delivered record structure exactly. If the format changes, update the schema.

**Choosing the right table type**

For raw flow logs, a text table is the simplest option. It is easy to create and works well for ad hoc investigations. However, text tables are less efficient than columnar formats for repeated analysis.

A common pattern is:

- Store raw flow logs in S3 as delivered.
- Query them with Athena using a text table.
- Optionally transform them into Parquet for faster and cheaper queries.

Parquet is a columnar storage format that reduces scanned data when you query only a subset of columns. This is useful for recurring security analytics, dashboards, and detection queries.

Example Parquet table pattern:

```sql
CREATE EXTERNAL TABLE IF NOT EXISTS vpc_flow_logs_parquet (
  version int,
  account_id string,
  interface_id string,
  srcaddr string,
  dstaddr string,
  srcport int,
  dstport int,
  protocol int,
  packets bigint,
  bytes bigint,
  start bigint,
  end bigint,
  action string,
  log_status string
)
PARTITIONED BY (
  year int,
  month int,
  day int
)
STORED AS PARQUET
LOCATION 's3://my-flow-log-bucket/curated/vpc-flow-logs-parquet/';
```

When you use Parquet, make sure the partition column types match how you query them. If the partitions are defined as integers, use numeric predicates in SQL.

```sql
SELECT srcaddr, dstaddr, action, bytes
FROM vpc_flow_logs_parquet
WHERE year = 2026
  AND month = 4
  AND day = 26
  AND action = 'REJECT'
LIMIT 100;
```

**Partitioning strategy**

Partitioning is one of the most important design choices for Athena performance. Without partitions, Athena scans all matching objects in the table location, which increases cost and query time.

For flow logs, common partition keys include:

- **year**
- **month**
- **day**
- sometimes **hour** if logs are delivered frequently and you need finer filtering

Partitioning by date is usually enough for most investigations. It lets you restrict scans to a small time window, which is especially important for large VPCs or long retention periods.

A typical S3 layout looks like this:

```text
s3://my-flow-log-bucket/AWSLogs/123456789012/vpcflowlogs/us-east-1/2026/04/26/
```

If you use Hive-style partitions, the folder names should align with the partition keys, such as:

```text
s3://my-flow-log-bucket/curated/vpc-flow-logs/year=2026/month=04/day=26/
```

For Athena to resolve projected or discovered partitions correctly, the storage path must match the partition naming convention exactly.

**Partition discovery and maintenance**

Athena can discover partitions in two main ways:

- **Manual partition management**
  - Add partitions explicitly with `ALTER TABLE ADD PARTITION`.
- **Partition discovery**
  - Use `MSCK REPAIR TABLE` to scan Hive-style prefixes and register missing partitions.

Example manual partition addition:

```sql
ALTER TABLE vpc_flow_logs
ADD PARTITION (year='2026', month='04', day='26')
LOCATION 's3://my-flow-log-bucket/AWSLogs/123456789012/vpcflowlogs/us-east-1/2026/04/26/';
```

Example partition repair:

```sql
MSCK REPAIR TABLE vpc_flow_logs;
```

Use `ALTER TABLE ADD PARTITION` when you know the exact partition locations. Use `MSCK REPAIR TABLE` when you have many Hive-style partitions and want Athena to discover them automatically.

> `MSCK REPAIR TABLE` only works well with Hive-style partition layouts. If your S3 prefixes do not follow that convention, prefer explicit partition adds or partition projection.

**Partition projection**

Partition projection lets Athena calculate partition values at query time instead of storing partition metadata in the Glue Data Catalog. This can reduce catalog maintenance overhead for large or frequently updated datasets.

Partition projection is useful when:

- You have many partitions.
- New partitions arrive continuously.
- You want to avoid running `MSCK REPAIR TABLE`.

Example with zero-padded month and day folders:

```sql
CREATE EXTERNAL TABLE vpc_flow_logs_projected (
  version int,
  account_id string,
  interface_id string,
  srcaddr string,
  dstaddr string,
  srcport int,
  dstport int,
  protocol int,
  packets bigint,
  bytes bigint,
  start bigint,
  end bigint,
  action string,
  log_status string
)
PARTITIONED BY (
  year int,
  month int,
  day int
)
ROW FORMAT DELIMITED
FIELDS TERMINATED BY ' '
STORED AS TEXTFILE
LOCATION 's3://my-flow-log-bucket/AWSLogs/123456789012/vpcflowlogs/'
TBLPROPERTIES (
  'projection.enabled'='true',
  'projection.year.type'='integer',
  'projection.year.range'='2024,2030',
  'projection.month.type'='integer',
  'projection.month.range'='1,12',
  'projection.month.digits'='2',
  'projection.day.type'='integer',
  'projection.day.range'='1,31',
  'projection.day.digits'='2',
  'storage.location.template'='s3://my-flow-log-bucket/AWSLogs/123456789012/vpcflowlogs/${year}/${month}/${day}/'
);
```

The `storage.location.template` must match the actual folder structure exactly. The `digits` property controls how Athena renders projected values such as `04` and `26` when the underlying folders are zero-padded.

**Querying flow logs effectively**

Flow log queries usually focus on security-relevant patterns such as denied traffic, unusual ports, or suspicious source addresses. Use predicates that reduce scanned data as early as possible.

Examples:

```sql
SELECT srcaddr, dstaddr, srcport, dstport, protocol, action
FROM vpc_flow_logs
WHERE year = '2026'
  AND month = '04'
  AND day = '26'
  AND action = 'REJECT'
LIMIT 50;
```

```sql
SELECT dstaddr, COUNT(*) AS rejected_flows
FROM vpc_flow_logs
WHERE year = '2026'
  AND month = '04'
  AND day = '26'
  AND action = 'REJECT'
GROUP BY dstaddr
ORDER BY rejected_flows DESC;
```

```sql
SELECT srcaddr, dstaddr, dstport, bytes, packets
FROM vpc_flow_logs
WHERE year = '2026'
  AND month = '04'
  AND day = '26'
  AND dstport IN (22, 3389)
  AND action = 'ACCEPT';
```

For integer partition columns, use numeric predicates:

```sql
SELECT srcaddr, dstaddr, action
FROM vpc_flow_logs_parquet
WHERE year = 2026
  AND month = 4
  AND day = 26
  AND action = 'REJECT';
```

**Interpreting log status values**

The `log_status` field helps you understand whether the record set is complete.

- **OK**
  - The flow log record was delivered successfully.
- **NODATA**
  - No flow log records were generated for the aggregation interval. This can happen when there was no traffic or no eligible traffic to record.
- **SKIPDATA**
  - Some flow log records were skipped because the service could not deliver all records for the interval due to internal limitations.

These values are important during investigations because a lack of records does not always mean a lack of network activity.

**Security analysis use cases**

Athena queries against flow logs support several common security tasks:

- Detecting rejected connections to sensitive ports.
- Identifying unexpected east-west traffic inside a VPC.
- Finding high-volume transfers that may indicate data exfiltration.
- Correlating source and destination IPs with incident timelines.
- Reviewing traffic to public IP addresses from private subnets.

A practical workflow is:

1. Narrow the time range with partitions.
2. Filter on `action = 'REJECT'` or unusual ports.
3. Aggregate by source, destination, or interface.
4. Correlate results with CloudTrail, GuardDuty, or security group changes.

**IAM and S3 permissions for Athena**

To query flow logs successfully, you typically need permissions for Athena, Glue Data Catalog, and the S3 buckets involved.

Common Athena permissions include:

- `athena:StartQueryExecution`
- `athena:GetQueryExecution`
- `athena:GetQueryResults`
- `athena:ListWorkGroups`
- `athena:ListDatabases`
- `athena:ListTableMetadata`

Common Glue permissions include:

- `glue:GetDatabase`
- `glue:GetDatabases`
- `glue:GetTable`
- `glue:GetTables`
- `glue:GetPartition`
- `glue:GetPartitions`

Common S3 permissions include:

- Read access to the flow log data bucket and prefix
- Write access to the Athena query results location
- `s3:GetObject`
- `s3:ListBucket`
- `s3:PutObject`

Example query results configuration:

```text
s3://my-athena-query-results-bucket/athena/results/
```

If the query results bucket uses SSE-KMS (Server-Side Encryption with AWS Key Management Service), the principal also needs the relevant KMS permissions such as `kms:Decrypt`, `kms:Encrypt`, and `kms:GenerateDataKey` as appropriate.

**Practical troubleshooting**

If Athena returns no rows, check the following:

- The table `LOCATION` points to the correct S3 prefix.
- The partition values match the folder names.
- The query predicates match the partition column types.
- The field order matches the actual flow log format.
- The data files are not compressed or encoded in an unexpected way.

If Athena returns malformed rows, verify:

- The delimiter is correct.
- The schema includes all fields in the right order.
- The log version matches the table definition.
- The S3 objects contain plain text flow logs rather than transformed output.

If partitioned queries are slow, consider:

- Reducing the partition range.
- Using partition projection.
- Converting raw logs to Parquet for repeated analysis.
- Avoiding `SELECT *` when you only need a few columns.

**Exam focus**

For the exam, focus on the relationship between S3, Athena, and the Glue Data Catalog. You should understand:

- Athena queries data in S3 using external tables.
- Flow logs are commonly stored as space-delimited text.
- Partitioning reduces scan cost and improves performance.
- `MSCK REPAIR TABLE` discovers Hive-style partitions, but explicit partition adds are also valid.
- Partition projection can remove the need for catalog partition maintenance.
- Parquet is more efficient than raw text for repeated analytics.
- IAM must cover Athena, Glue, S3, and often the query results bucket.

**Quick reference**

- Use **text tables** for raw flow logs.
- Use **Parquet** for optimized analytics.
- Use **date partitions** to reduce scan volume.
- Use **explicit partitions** or **partition projection** depending on your S3 layout.
- Use **numeric predicates** for integer partition columns.
- Use **`REJECT`** to find blocked traffic and **`OK`** to confirm successful delivery.
