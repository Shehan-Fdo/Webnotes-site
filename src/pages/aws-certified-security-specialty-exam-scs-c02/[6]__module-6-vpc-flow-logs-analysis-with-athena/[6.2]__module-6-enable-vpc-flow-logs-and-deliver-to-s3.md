## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Overview**

Amazon VPC Flow Logs capture metadata about IP traffic going to and from network interfaces in your VPC. You use them for security monitoring, incident response, network troubleshooting, and compliance evidence. When you deliver Flow Logs to Amazon S3, you get durable, queryable storage that works well with Athena, Glue, and lifecycle policies.

Flow Logs do not capture packet payloads. They record metadata such as source and destination addresses, ports, protocol, action, and bytes. That makes them useful for detecting patterns and validating network controls without exposing content.

**What you can create Flow Logs for**

VPC Flow Logs support multiple scopes, and exam questions often test which scope fits the use case:

- **VPC**: captures traffic for all elastic network interfaces (ENIs) in the VPC.
- **Subnet**: captures traffic for ENIs in a specific subnet.
- **Network interface (ENI)**: captures traffic for a single interface, which is useful for targeted troubleshooting or investigation.

Choose the narrowest scope that satisfies the requirement when you want to reduce noise and cost. Use VPC-level logging when you need broad visibility across an environment.

**Core Flow Log behavior**

Flow Logs are aggregated over a fixed interval before delivery. The default aggregation interval is typically 10 minutes, and you can often choose 1 minute for faster visibility at the cost of more log volume. Shorter intervals improve timeliness for detection and troubleshooting, while longer intervals reduce storage and delivery overhead.

Flow Logs can be created in different versions. Versioning matters because it affects which fields are available and how custom formats behave:

- **Version 2**: commonly used baseline format with standard fields.
- **Version 3**: adds support for additional fields and more flexible custom formatting in supported destinations.

Custom fields are not identical across all destinations and versions. For exam prep, remember that the available field set depends on the Flow Log version and the delivery target, so do not assume one universal record layout.

**Typical record fields**

A Flow Log record is a single line of space-separated values. A common default record includes fields like:

- `version`
- `account-id`
- `interface-id`
- `srcaddr`
- `dstaddr`
- `srcport`
- `dstport`
- `protocol`
- `packets`
- `bytes`
- `start`
- `end`
- `action`
- `log-status`

Example interpretation:

```text
2 123456789012 eni-0abc123def4567890 10.0.1.10 10.0.2.25 44321 443 6 12 8421 1714060800 1714060860 ACCEPT OK
```

How to read it:

- `2` indicates the record version.
- `eni-0abc123def4567890` identifies the network interface.
- `10.0.1.10` is the source address and `10.0.2.25` is the destination address.
- `44321` is the source port and `443` is the destination port.
- `6` is TCP.
- `12` packets and `8421` bytes were observed.
- `ACCEPT` means the traffic was allowed by the network path and security controls.
- `OK` means the record was successfully delivered.

> Flow Logs show network metadata, not application-layer content. If you need payload inspection, use other controls such as packet capture on approved tooling or application logs.

**S3 delivery model**

When you deliver Flow Logs to S3, AWS writes objects into a bucket you control. This is a common choice when you want:

- Long-term retention
- Centralized security analytics
- Athena queries over historical logs
- Lifecycle transitions to cheaper storage classes

For exam scenarios, the S3 bucket is usually in the same Region as the Flow Log resource. Do not assume cross-Region delivery is supported for every design; treat same-Region delivery as the safe, expected pattern unless the question explicitly states otherwise.

AWS uses a service delivery path for writing the logs. That means the bucket policy must allow the AWS log delivery service to write objects and verify bucket ownership. If the bucket uses SSE-KMS, the KMS key policy must also allow the service to use the key for S3 delivery.

**S3 bucket naming and prefix structure**

AWS Flow Logs delivery commonly writes under an `AWSLogs/` prefix. A typical layout looks like this:

```text
s3://my-flow-log-bucket/AWSLogs/123456789012/vpcflowlogs/us-east-1/2026/04/26/...
```

Important points:

- The `AWSLogs/` prefix is part of the standard delivery structure.
- The account ID often appears in the path.
- The Region and date partitioning help with organization and querying.
- Object names are service-generated, so do not hard-code a single filename pattern in verification steps.

If you use a custom prefix, keep it simple and consistent. The exact object naming and partitioning depend on the delivery format and configuration, so verify the actual prefix in the bucket rather than assuming a fixed `.gz` object name.

**Required S3 bucket policy elements**

For S3 delivery, the bucket policy must support the AWS log delivery service principal and the expected ownership controls. The canonical pattern includes permissions for:

- `s3:GetBucketAcl` so the service can verify bucket ownership
- `s3:PutObject` so the service can write log objects
- `s3:x-amz-acl = bucket-owner-full-control` so the bucket owner retains control of delivered objects

A practical bucket policy pattern looks like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AWSLogDeliveryAclCheck",
      "Effect": "Allow",
      "Principal": {
        "Service": "delivery.logs.amazonaws.com"
      },
      "Action": "s3:GetBucketAcl",
      "Resource": "arn:aws:s3:::my-flow-log-bucket",
      "Condition": {
        "StringEquals": {
          "aws:SourceAccount": "123456789012"
        }
      }
    },
    {
      "Sid": "AWSLogDeliveryWrite",
      "Effect": "Allow",
      "Principal": {
        "Service": "delivery.logs.amazonaws.com"
      },
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::my-flow-log-bucket/AWSLogs/123456789012/*",
      "Condition": {
        "StringEquals": {
          "s3:x-amz-acl": "bucket-owner-full-control",
          "aws:SourceAccount": "123456789012"
        }
      }
    }
  ]
}
```

Use this as a conceptual pattern, not a copy-paste guarantee for every environment. The exact policy may also include `aws:SourceArn` conditions or additional constraints depending on your security posture and AWS guidance.

> If the bucket policy is missing `GetBucketAcl`, Flow Logs delivery can fail even when `PutObject` is allowed.

**SSE-KMS considerations**

If the S3 bucket uses server-side encryption with AWS Key Management Service (SSE-KMS), the KMS key policy must allow the log delivery service to use the key in the context of S3 object encryption. This is a common exam trap because S3 permissions alone are not enough.

Key points:

- The KMS key policy must allow the service principal used for log delivery.
- The policy should be scoped to the S3 encryption context and the target bucket where possible.
- The bucket policy and KMS key policy both need to be correct for delivery to succeed.

A simplified example of the idea:

```json
{
  "Sid": "AllowFlowLogsUseOfKMSKeyForS3",
  "Effect": "Allow",
  "Principal": {
    "Service": "delivery.logs.amazonaws.com"
  },
  "Action": [
    "kms:Encrypt",
    "kms:GenerateDataKey",
    "kms:Decrypt",
    "kms:DescribeKey"
  ],
  "Resource": "*",
  "Condition": {
    "StringEquals": {
      "aws:SourceAccount": "123456789012"
    },
    "ArnLike": {
      "kms:EncryptionContext:aws:s3:arn": "arn:aws:s3:::my-flow-log-bucket/*"
    }
  }
}
```

The exact least-privilege policy can vary. For exam purposes, remember the principle: S3 delivery with SSE-KMS requires both S3 authorization and KMS authorization, and the KMS policy should be constrained to the intended S3 context.

**Creating Flow Logs**

You can create Flow Logs from the VPC console, EC2 console, CLI, CloudFormation, or Terraform. In practice, the destination bucket policy must already exist and be correct before delivery starts.

CLI example:

```bash
aws ec2 create-flow-logs \
  --resource-type VPC \
  --resource-ids vpc-0123456789abcdef0 \
  --traffic-type ALL \
  --log-destination-type s3 \
  --log-destination arn:aws:s3:::my-flow-log-bucket/AWSLogs/123456789012/ \
  --deliver-logs-permission-arn arn:aws:iam::123456789012:role/FlowLogsDeliveryRole
```

Notes:

- `--resource-type` can be `VPC`, `Subnet`, or `NetworkInterface`.
- `--traffic-type` can be `ALL`, `ACCEPT`, or `REJECT`.
- `--log-destination-type s3` indicates S3 delivery.
- The exact CLI parameters can vary by AWS CLI version and the delivery model you choose, so validate against the current CLI help in your environment.

A more reliable operational approach is to create the bucket policy first, then create the Flow Log, and finally verify that objects appear under the expected prefix.

**CloudFormation pattern**

CloudFormation can define the Flow Log resource, but it does not magically fix S3 authorization. You still need the bucket policy and, if applicable, the KMS key policy.

```yaml
Resources:
  FlowLogBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: my-flow-log-bucket

  VpcFlowLog:
    Type: AWS::EC2::FlowLog
    Properties:
      ResourceType: VPC
      ResourceId: vpc-0123456789abcdef0
      TrafficType: ALL
      LogDestinationType: s3
      LogDestination: !Sub arn:aws:s3:::${FlowLogBucket}/AWSLogs/${AWS::AccountId}/
      DeliverLogsPermissionArn: arn:aws:iam::123456789012:role/FlowLogsDeliveryRole
```

Use this pattern carefully:

- The bucket policy must exist separately.
- If the bucket uses SSE-KMS, the KMS key policy must also be in place.
- The resource may create successfully while delivery still fails if permissions are incomplete.

**Terraform pattern**

Terraform can manage the Flow Log and the supporting bucket policy, which is usually the safer approach for repeatable deployments.

```hcl
resource "aws_s3_bucket" "flow_logs" {
  bucket = "my-flow-log-bucket"
}

resource "aws_s3_bucket_policy" "flow_logs" {
  bucket = aws_s3_bucket.flow_logs.id
  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Sid      = "AWSLogDeliveryAclCheck"
        Effect   = "Allow"
        Principal = {
          Service = "delivery.logs.amazonaws.com"
        }
        Action   = "s3:GetBucketAcl"
        Resource = aws_s3_bucket.flow_logs.arn
      },
      {
        Sid      = "AWSLogDeliveryWrite"
        Effect   = "Allow"
        Principal = {
          Service = "delivery.logs.amazonaws.com"
        }
        Action   = "s3:PutObject"
        Resource = "${aws_s3_bucket.flow_logs.arn}/AWSLogs/123456789012/*"
        Condition = {
          StringEquals = {
            "s3:x-amz-acl" = "bucket-owner-full-control"
          }
        }
      }
    ]
  })
}

resource "aws_flow_log" "vpc" {
  log_destination_type = "s3"
  log_destination      = "${aws_s3_bucket.flow_logs.arn}/AWSLogs/123456789012/"
  traffic_type         = "ALL"
  vpc_id               = "vpc-0123456789abcdef0"
}
```

This is still only part of the solution. If you use SSE-KMS, add the KMS key and key policy. If you use a custom prefix, ensure the bucket policy resource path matches it exactly.

**Default fields, custom fields, and versions**

Flow Logs have a default record format, but the exact fields available depend on the version and destination. This matters when you build Athena tables or parse logs with downstream tools.

- **Default format**: useful for quick inspection and common security use cases.
- **Custom format**: useful when you want to reduce noise or include additional context.
- **Version differences**: newer versions expose more flexibility, but not every destination supports the same field set.

For exam questions, remember:

- Do not assume every destination supports the same custom field behavior.
- Do not assume the same record layout across all versions.
- Use the version and destination together when designing parsing or analytics.

**Aggregation interval tradeoffs**

The aggregation interval affects both timeliness and cost:

- **1 minute**
  - Faster visibility
  - Better for near-real-time detection
  - Higher object count and more S3 requests
- **10 minutes**
  - Lower delivery overhead
  - Fewer objects
  - Slower visibility for investigations

Choose 1 minute when you need rapid detection of suspicious traffic patterns. Choose 10 minutes when you prioritize lower cost and can tolerate delayed visibility.

**Interpreting common scenarios**

Use Flow Logs to answer questions like these:

- Why is an instance unable to reach a database?
  - Check `REJECT` records, destination port, and security group or NACL behavior.
- Is a workload unexpectedly talking to the internet?
  - Look for unusual `dstaddr`, `dstport`, and `ACCEPT` records.
- Did a subnet experience denied traffic spikes?
  - Filter by subnet scope and count `REJECT` actions over time.
- Is a specific ENI generating traffic to an unexpected host?
  - Use ENI-level logging for focused investigation.

Flow Logs help you confirm network behavior, but they do not tell you which IAM principal initiated the traffic. Combine them with CloudTrail, application logs, and security group analysis for full attribution.

**Sample Athena query**

If you store Flow Logs in S3, Athena is a common way to query them.

```sql
SELECT
  srcaddr,
  dstaddr,
  dstport,
  action,
  COUNT(*) AS hits
FROM vpc_flow_logs
WHERE action = 'REJECT'
  AND from_unixtime(start) >= current_timestamp - interval '1' day
GROUP BY srcaddr, dstaddr, dstport, action
ORDER BY hits DESC;
```

This kind of query helps you identify denied traffic patterns and prioritize investigation.

**Troubleshooting checklist**

When S3 delivery fails, work through the problem in this order:

1. Confirm the Flow Log is attached to the correct scope: VPC, subnet, or ENI.
2. Verify the bucket exists in the expected Region and the destination prefix matches the policy resource path.
3. Check the bucket policy for:
   - `delivery.logs.amazonaws.com` principal
   - `s3:GetBucketAcl`
   - `s3:PutObject`
   - `bucket-owner-full-control`
4. If SSE-KMS is enabled, verify the KMS key policy allows the log delivery service.
5. Confirm the bucket owner is the intended owner of the delivered objects.
6. Inspect CloudTrail for denied `PutObject`, `GetBucketAcl`, or KMS API calls.
7. Check whether the Flow Log status is `ACTIVE` and whether the destination ARN is correct.
8. Validate that the prefix in the bucket policy exactly matches the configured destination path.

Useful CLI checks:

```bash
aws ec2 describe-flow-logs --flow-log-ids fl-0123456789abcdef0
aws s3api get-bucket-policy --bucket my-flow-log-bucket
aws s3api get-bucket-encryption --bucket my-flow-log-bucket
aws kms describe-key --key-id arn:aws:kms:us-east-1:123456789012:key/abcd-1234
```

For object verification, inspect the bucket contents rather than assuming a specific filename:

```bash
aws s3 ls s3://my-flow-log-bucket/AWSLogs/123456789012/ --recursive
```

**Security and compliance considerations**

Flow Logs are often used to support detective controls and audit requirements. They are not a replacement for preventive controls such as security groups, NACLs, or firewall policies.

- Use **least privilege** in bucket and KMS policies.
- Enable **SSE-KMS** when you need stronger encryption control and key governance.
- Apply **lifecycle policies** to manage retention and cost.
- Consider **Object Lock** only if your compliance requirements demand immutability and you understand the operational impact.

Example lifecycle policy:

```json
{
  "Rules": [
    {
      "ID": "ArchiveFlowLogs",
      "Status": "Enabled",
      "Filter": {
        "Prefix": "AWSLogs/"
      },
      "Transitions": [
        {
          "Days": 30,
          "StorageClass": "GLACIER_IR"
        }
      ],
      "Expiration": {
        "Days": 365
      }
    }
  ]
}
```

Storage class choice depends on retrieval needs:

- Use **Standard** when you query frequently.
- Use **Standard-IA** or **Intelligent-Tiering** when access is unpredictable.
- Use **Glacier Instant Retrieval** when you want low-cost archival with occasional fast retrieval.
- Use deeper archive classes only when retrieval latency is acceptable.

**Exam focus points**

- Know the three Flow Log scopes: VPC, subnet, and ENI.
- Know that Flow Logs capture metadata, not payloads.
- Know that S3 delivery requires both bucket policy and, if used, KMS key policy.
- Know that `s3:GetBucketAcl` and `s3:PutObject` are both important for delivery.
- Know that `bucket-owner-full-control` is a key ownership condition.
- Know that aggregation interval affects timeliness and log volume.
- Know that version and destination affect available fields and custom formatting.
- Know that Flow Logs are useful for detection, troubleshooting, and compliance, but not for packet inspection.

**Quick mental model**

Think of the solution as four layers:

- **Capture**: choose VPC, subnet, or ENI.
- **Format**: choose version, fields, and aggregation interval.
- **Deliver**: configure S3 destination and bucket policy.
- **Protect**: add KMS, lifecycle, and access controls.

If any layer is incomplete, the Flow Log may exist but the delivery pipeline will fail.
