## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Interpretation mindset**

In incident response, you do not jump straight to “fixing” the environment. You first interpret the signal: what was detected, what is affected, what is still active, and what action reduces risk without destroying evidence or breaking recovery options. For the exam, the key skill is mapping a finding to the least disruptive containment step that actually stops further harm.

A good containment decision balances four goals:

- Stop attacker activity.
- Preserve forensic evidence.
- Limit blast radius.
- Keep recovery paths available.

**Incident response workflow in AWS**

A practical AWS response usually follows a phased workflow:

1. Triage the alert and confirm scope.
2. Contain the threat.
3. Preserve evidence.
4. Eradicate persistence.
5. Recover and monitor.

In larger organizations, you often escalate to a Security Incident Response Team (SIRT), incident commander, or equivalent formal response process before making high-impact changes. On the exam, look for clues about severity, business impact, and whether the action should be automated, coordinated, or approved.

Common decision factors include:

- Whether the compromise is active or historical.
- Whether the affected asset is production-critical.
- Whether the attacker has credentials, network access, or both.
- Whether you need immediate isolation or can first collect evidence.
- Whether the service is identity, compute, network, or data plane.

**Identity containment**

Identity compromise is one of the most common exam themes. The right action depends on the identity type: IAM user, IAM role, federated user, or root user.

For **IAM users**, containment often means:

- Deactivate or delete access keys.
- Reset the console password.
- Remove the user from privileged groups.
- Detach directly attached policies if needed.
- Review and revoke MFA devices if the device is suspected compromised.

For **root user** compromise, treat it as high severity:

- Change the root password immediately.
- Ensure root MFA is enabled and replace it if compromised.
- Remove root access keys if they exist.
- Review billing, support, and account settings for unauthorized changes.
- Open an urgent support case if account recovery or lockout is possible.

For **federated identities** and **AWS IAM Identity Center** users:

- Revoke the upstream identity session if your IdP supports it.
- Disable the user or group in the identity provider.
- Invalidate or shorten session duration where possible.
- Review permission sets and assignments for over-privilege.

For **IAM roles**, remember that role sessions are temporary. You usually cannot “log out” a role session directly in the same way as a console password reset. Instead, you reduce future and sometimes current access by:

- Updating the role trust policy to block new assumptions.
- Removing the principal from the trust relationship.
- Tightening session duration.
- Adjusting permission boundaries or session policies.
- Rotating or disabling the source credentials used to assume the role.

> A common exam trap is assuming one action revokes every kind of session. It does not. Match the containment step to the identity type and the session source.

**Access key and credential hygiene**

If the finding points to leaked credentials, containment usually starts with key rotation and access removal.

Typical actions:

- Deactivate the compromised access key.
- Create a replacement key only if the workload still needs long-term credentials.
- Rotate secrets used by applications, CI/CD systems, and third-party integrations.
- Review CloudTrail for use of the key before and after the alert time.
- Search for the same credential pattern in other accounts or environments.

Example CLI to deactivate an IAM user access key:

```bash
aws iam update-access-key \
  --user-name alice \
  --access-key-id AKIAEXAMPLE1234567890 \
  --status Inactive
```

If the key belongs to a role-based automation path, also inspect the upstream source:

- EC2 instance profile.
- Lambda execution role.
- GitHub Actions or other external OIDC federation.
- SAML or IAM Identity Center assignment.

**Compute containment: EC2, Lambda, containers**

For **EC2**, the usual containment pattern is isolation, not destruction. You often want to preserve the instance for forensics while preventing further outbound or inbound activity.

Good options include:

- Remove the instance from load balancers and auto scaling groups if needed.
- Move the instance into a quarantine security group.
- Restrict outbound traffic to only what is required for investigation.
- Preserve EBS snapshots before rebooting or terminating.
- Capture memory or volatile evidence if your process supports it.

A quarantine security group is an allow-list model: no inbound rules, and only tightly scoped outbound rules if needed.

```bash
INSTANCE_ID=i-0123456789abcdef0
QUARANTINE_SG=sg-0abc1234def567890

aws ec2 modify-instance-attribute \
  --instance-id "$INSTANCE_ID" \
  --groups "$QUARANTINE_SG"
```

If you need forensic preservation, create snapshots before making disruptive changes:

```bash
aws ec2 create-snapshot \
  --volume-id vol-0123456789abcdef0 \
  --description "Pre-containment snapshot for incident response"
```

You can also create an AMI for later analysis, but remember that an AMI is not a containment action by itself; it is a preservation and recovery artifact.

For **IMDS** (Instance Metadata Service) abuse, containment may include:

- Requiring IMDSv2 on affected instances.
- Restricting network paths that allow SSRF (server-side request forgery) to metadata endpoints.
- Reviewing instance profile permissions for overreach.
- Replacing compromised credentials obtained from metadata.

For **Lambda**, containment often means:

- Disable event sources or triggers.
- Remove or restrict the execution role.
- Roll back to a known-good version or alias.
- Review environment variables and layers for injected secrets or code.
- Check whether the function is being used as a pivot into other services.

Example: disable an EventBridge rule that triggers a suspicious function:

```bash
aws events disable-rule --name suspicious-schedule
```

For **API Gateway**, containment may involve:

- Deploying a temporary deny response or throttling.
- Restricting usage plans or API keys.
- Updating resource policies to block unwanted source IPs or VPC endpoints.
- Removing compromised authorizers or rotating authorizer secrets.

For **containers and ECS/EKS**, consider:

- Scaling down or draining suspicious tasks.
- Replacing nodes if the host is suspected compromised.
- Rotating image pull credentials and Kubernetes secrets.
- Reviewing service accounts, IAM roles for service accounts, and admission controls.

**Network containment**

Network controls are often the fastest way to reduce blast radius, but they differ in behavior.

**Security groups** are stateful allow lists:

- No inbound rules means no inbound access.
- Outbound rules should be narrowed if you need to stop exfiltration.
- They are ideal for instance-level quarantine.

**Network ACLs (NACLs)** are stateless and evaluated in rule order:

- Lower-numbered rules are processed first.
- A deny rule does not help if an earlier allow already matches.
- You often need to adjust both inbound and outbound rules.
- Return traffic must also be considered because NACLs do not track state.

Use NACLs when you need subnet-level control, but be careful not to assume a single deny rule is enough.

Example of a quarantine-oriented NACL approach:

- Deny inbound from untrusted ranges.
- Deny outbound to the internet.
- Allow only the minimum investigation traffic if required.
- Verify rule order so the deny rules actually match first.

For **VPC endpoints**, you can reduce exposure by:

- Restricting endpoint policies.
- Limiting which principals can use the endpoint.
- Removing public internet paths where possible.

For **AWS Network Firewall**, you can:

- Block known malicious destinations.
- Restrict egress by domain or IP.
- Apply centralized inspection across VPCs.

For **AWS WAF**, **AWS Shield**, and **CloudFront/ALB** controls, use them when the issue is web-layer abuse:

- WAF for malicious HTTP patterns, bots, and exploit signatures.
- Shield for DDoS resilience, especially with Shield Advanced in high-value environments.
- CloudFront or ALB rules for temporary edge blocking, geo restrictions, or rate limiting.

If the attack is at the web layer, do not jump straight to instance isolation if edge controls can absorb or filter the traffic first.

**Data and secrets containment**

When the finding involves data exposure or secret compromise, the response is usually broader than one service.

For **AWS Secrets Manager**:

- Rotate the secret if the secret may be exposed.
- Update dependent applications after rotation.
- Revoke or replace any downstream credentials that the secret unlocks.
- Treat rotation as part of remediation, not the only containment step.

For **AWS Systems Manager Parameter Store**:

- Replace compromised parameters.
- Tighten IAM access to the parameter path.
- Review application logs for retrieval of sensitive values.

For **KMS (Key Management Service)** keys:

- Disabling a key is a last-resort containment action.
- It can immediately stop decryption and encrypt operations that depend on the key.
- It may also break recovery if backups, snapshots, or archives become inaccessible.
- Use it only when the security impact outweighs the operational damage.

For **S3**:

- Block public access if exposure is public or uncertain.
- Tighten bucket policies and ACLs.
- Remove risky cross-account grants.
- Review object-level access logs and CloudTrail data events if enabled.

For **RDS** and other managed data stores:

- Restrict security groups and subnet access.
- Rotate database credentials.
- Consider snapshotting before disruptive changes.
- Review parameter groups and audit logs for tampering.

**Route 53 and traffic steering**

DNS changes can be a containment tool when you need to redirect or stop traffic, but they must be precise.

- A Route 53 change batch must include the full record set details.
- You cannot safely “revert” with only a record name.
- Use the exact prior values, TTL, and routing policy when restoring.
- DNS propagation delay means this is not always immediate containment.

Example of a valid change batch structure:

```json
{
  "Comment": "Restore previous A record",
  "Changes": [
    {
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "app.example.com",
        "Type": "A",
        "TTL": 60,
        "ResourceRecords": [
          { "Value": "203.0.113.10" }
        ]
      }
    }
  ]
}
```

Then apply it:

```bash
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch file://restore-record.json
```

**Choosing the right containment action**

When multiple actions are possible, choose the one that stops the threat with the least collateral damage.

| Scenario | Best first containment action | Why |
|---|---|---|
| Compromised IAM user access key | Deactivate the key and review usage | Stops API access quickly without breaking unrelated resources |
| Suspected root user compromise | Change root password, ensure MFA, open urgent support path | Root has account-wide impact |
| EC2 instance used for exfiltration | Quarantine security group and preserve snapshots | Stops traffic while keeping evidence |
| Suspicious Lambda behavior | Disable trigger and review execution role | Prevents repeated invocation |
| Leaked database password | Rotate secret and restrict DB access | Stops future use of the credential |
| Web exploit against public app | WAF rule, ALB/CloudFront filtering, rate limiting | Contains at the edge |
| Compromised federated user | Disable upstream identity and revoke sessions | Identity source is the control point |
| Suspected KMS abuse | Consider key disablement only as last resort | High blast radius and recovery risk |

**Scenario-based exam reasoning**

Use the alert details to infer the right control plane.

- If the alert mentions stolen credentials, think identity and secret rotation first.
- If the alert mentions outbound beaconing from an instance, think network isolation and evidence preservation.
- If the alert mentions a malicious Lambda invocation, think trigger disablement and role review.
- If the alert mentions public web exploitation, think WAF, Shield, CloudFront, or ALB controls.
- If the alert mentions data exposure, think access revocation, secret rotation, and logging review.
- If the alert mentions lateral movement, think segmentation, security groups, NACLs, and least privilege.

A useful exam habit is to ask:

1. What is the attacker using right now?
2. What control can stop that use immediately?
3. What evidence must I preserve before changing the system?
4. What recovery dependency could I break if I act too aggressively?

**Preserving forensic evidence**

Containment should not destroy evidence unless the situation is so severe that immediate disruption is required.

For EC2 and EBS:

- Snapshot volumes before termination.
- Avoid rebooting unless necessary.
- Record instance metadata, security groups, IAM role, and network interfaces.
- Preserve CloudTrail, VPC Flow Logs, and application logs.

For Lambda and API Gateway:

- Export function configuration, versions, aliases, environment variables, and trigger mappings.
- Preserve invocation logs and request IDs.
- Capture deployment artifacts and IaC (Infrastructure as Code) state.

For S3 and data stores:

- Preserve object versions and access logs.
- Export audit logs before changing retention or lifecycle settings.
- Record bucket policies, KMS key usage, and replication configuration.

**Automation and runbooks**

AWS environments often benefit from automated containment. Use automation when the response is repeatable and low-risk.

Common tools:

- AWS Systems Manager Automation runbooks.
- AWS Systems Manager Session Manager for controlled access.
- Lambda or EventBridge-driven response workflows.
- AWS Config remediation actions.
- Security Hub custom actions and integrations.

Example: use Session Manager instead of opening SSH during investigation. This reduces exposure and avoids creating new inbound paths.

```bash
aws ssm start-session --target i-0123456789abcdef0
```

Automation is especially useful for:

- Quarantining instances.
- Disabling suspicious access keys.
- Tagging resources for investigation.
- Notifying responders and opening tickets.
- Collecting snapshots or logs in a standardized way.

**Common pitfalls**

- Treating rotation as containment when the attacker still has active access.
- Assuming one deny rule in a NACL is enough without checking rule order and return traffic.
- Disabling a KMS key without considering downstream recovery impact.
- Replacing evidence-preserving isolation with immediate termination.
- Forgetting federated sessions and Identity Center users when only IAM users are discussed.
- Ignoring Lambda, API Gateway, and edge controls in web attack scenarios.
- Using a direct policy detach example when the policy is attached through a group or role instead of the user.

**Quick exam cues**

- **Identity issue**: revoke keys, reset passwords, disable federation source, review trust policies.
- **Compute issue**: isolate, snapshot, preserve, then eradicate.
- **Network issue**: security groups for instance quarantine, NACLs for subnet control, WAF/Shield for edge filtering.
- **Data issue**: rotate secrets, restrict access, preserve logs, and consider downstream dependencies.
- **High-severity account issue**: escalate immediately and protect root access.

**Practical containment checklist**

1. Confirm the alert and identify the affected service.
2. Stop the attacker’s current path of access.
3. Preserve evidence before destructive changes.
4. Rotate or revoke the credentials or trust path.
5. Validate that the containment actually blocks the observed behavior.
6. Escalate through the incident response process.
7. Document what changed and what remains at risk.

**Key takeaway**

The exam tests whether you can map a finding to the right AWS control plane. The best answer is usually the one that contains the threat quickly, preserves evidence, and avoids unnecessary disruption. Think in terms of identity, compute, network, data, and edge controls, then choose the least disruptive action that truly stops the attack.
