## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Why posture correlation matters**

Security posture issues are the configuration, control, and exposure gaps that increase incident likelihood or impact. In AWS, you rarely respond to a single signal in isolation. You correlate preventive controls, detective findings, and evidence sources to decide whether an alert is a true incident, how severe it is, and what containment action is safe.

A strong response workflow answers three questions quickly:

- What changed?
- What is affected?
- What should you contain first without destroying evidence?

**Signal types you must distinguish**

Different AWS services provide different kinds of security signals. Exam questions often test whether you know which service is best for detection, evidence, or remediation.

- **AWS CloudTrail** records API activity and is the primary evidence source for who did what, when, and from where.
- **Amazon GuardDuty** produces threat findings from analyzed telemetry such as CloudTrail, VPC Flow Logs, and DNS logs.
- **AWS Config** evaluates resource configuration against rules and shows whether a resource is compliant with a control.
- **AWS Security Hub** aggregates findings from AWS services and partner tools and helps prioritize response.
- **Amazon Detective** helps you investigate and correlate findings across CloudTrail, GuardDuty, and VPC Flow Logs.
- **AWS Audit Manager** helps collect evidence for controls and can support posture-to-incident correlation during audits and investigations.
- **AWS Security Lake** centralizes security data in an open format for broader analysis and correlation.

A useful mental model is:

- **CloudTrail** = evidence trail
- **GuardDuty** = threat detection
- **Config** = configuration compliance
- **Security Hub** = aggregation and prioritization
- **Detective** = investigation and correlation
- **Security Lake** = centralized security data store
- **Audit Manager** = control evidence collection

**How posture issues become incidents**

A posture issue is not always an incident, but it can create the conditions for one. For example, an overly permissive security group is a posture issue; if you see unusual outbound traffic from that instance, the posture issue becomes part of the incident context.

Common posture-to-incident patterns include:

- Public exposure of an S3 bucket, database, or management port
- Disabled logging or missing CloudTrail coverage
- Overly broad IAM permissions or stale access keys
- Unencrypted data at rest or in transit
- Security group or network ACL changes that open unexpected paths
- KMS key disablement or policy changes that affect access to encrypted data
- Drift from approved infrastructure baselines
- Missing patching or vulnerable AMIs that increase exploitability

When you investigate, treat posture issues as amplifiers. They may not prove compromise, but they often explain why a compromise succeeded or why blast radius increased.

**Core investigation workflow**

Use a repeatable sequence so you do not lose evidence or overreact.

1. Identify the alert source and severity.
2. Confirm whether the finding is preventive, detective, or evidentiary.
3. Check CloudTrail for the API calls that changed the posture.
4. Use Config to verify whether the resource is currently noncompliant.
5. Use GuardDuty and Security Hub to see whether the issue is linked to active threat activity.
6. Use Detective or Security Lake to correlate related activity across accounts, regions, and services.
7. Contain the threat with the least disruptive action that preserves evidence.
8. Preserve logs, snapshots, and configuration state for later analysis.

**Incident response evidence handling**

Evidence integrity matters. If you cannot trust your logs, you cannot trust your timeline.

- Centralize CloudTrail logs in a dedicated log archive account.
- Enable log file validation for CloudTrail so you can detect tampering.
- Protect log buckets with restrictive bucket policies and object lock where appropriate.
- Use separate security and log archive accounts to reduce the chance that an attacker can erase evidence.
- Monitor for changes to logging configuration, trail deletion, bucket policy changes, and KMS key policy changes.
- Keep time synchronization consistent across systems so event ordering is reliable.

A common exam theme is tamper detection. Look for:

- Trail stopped or deleted
- Log delivery failures
- S3 bucket policy changes on the log archive bucket
- KMS key disabled or policy modified
- CloudWatch Logs subscription changes
- EventBridge rules deleted or altered

> If logging is impaired, treat that as a high-priority incident condition, not just a configuration issue.

**AWS CloudTrail as the evidentiary backbone**

CloudTrail records management events and, when enabled, data events for selected resources. It is the first place to confirm whether a posture change was intentional, automated, or malicious.

Use CloudTrail to answer:

- Which principal made the change?
- Was the call made from the console, CLI, SDK, or assumed role?
- What source IP and user agent were involved?
- Was MFA present?
- Was the action part of a known automation path?

Practical examples:

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=AuthorizeSecurityGroupIngress \
  --max-results 10
```

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=Username,AttributeValue=break-glass-role \
  --start-time 2026-04-26T00:00:00Z \
  --end-time 2026-04-26T23:59:59Z
```

CloudTrail is especially useful when a posture issue is the result of:

- Security group changes
- IAM policy updates
- KMS key policy edits
- S3 bucket policy changes
- RDS parameter changes
- Lambda permission changes
- Route 53 record modifications
- CloudFront distribution updates
- API Gateway stage or authorizer changes

**AWS Config for posture verification**

AWS Config evaluates resource configuration against rules and records configuration history. It tells you whether the current state violates a rule and when the resource drifted.

Use Config when you need to verify:

- Whether a resource is currently noncompliant
- When the configuration changed
- Which resources are affected by a rule
- Whether the issue is isolated or widespread

Example commands:

```bash
aws configservice describe-compliance-by-config-rule \
  --config-rule-names cloudtrail-enabled
```

```bash
aws configservice get-resource-config-history \
  --resource-type AWS::EC2::SecurityGroup \
  --resource-id sg-0123456789abcdef0
```

Config is best for posture assessment, not for proving intent. It shows the state and history of the state, but CloudTrail is what tells you who changed it.

**GuardDuty findings and what they mean**

GuardDuty findings indicate suspicious or malicious activity inferred from telemetry. They are not the same as configuration drift.

Typical GuardDuty categories include:

- Credential-related abuse
- Reconnaissance
- Instance or container compromise indicators
- Malware or crypto-mining behavior
- Suspicious DNS or network activity
- Unusual API activity patterns

Use GuardDuty to decide whether a posture issue is actively being exploited. For example:

- A public S3 bucket plus unusual `GetObject` activity may indicate data exposure.
- A permissive security group plus port scanning findings may indicate reconnaissance.
- A compromised IAM access key plus unusual API calls may indicate account abuse.

Example CLI query:

```bash
aws guardduty list-findings --detector-id d1example
```

```bash
aws guardduty get-findings \
  --detector-id d1example \
  --finding-ids finding-id-1 finding-id-2
```

Be careful not to assume GuardDuty finding names are exact incident labels. The service surfaces categories and types that vary by detector and telemetry source.

**Security Hub for prioritization and aggregation**

Security Hub aggregates findings from AWS services and partner products. It helps you see posture and incident signals in one place.

Use Security Hub when you need to:

- Correlate findings from GuardDuty, Config, Inspector, and other sources
- Prioritize by severity and workflow status
- Track remediation ownership
- Build a single response queue

Example:

```bash
aws securityhub get-findings \
  --filters '{"SeverityLabel":[{"Value":"HIGH","Comparison":"EQUALS"}]}'
```

Security Hub is not the source of truth for raw evidence. It is the triage layer. When a finding matters, go back to the originating service for details and to CloudTrail for the change history.

**Amazon Detective for investigation**

Amazon Detective is designed to help you investigate and correlate security findings. It can connect activity across CloudTrail, GuardDuty, and VPC Flow Logs so you can understand the sequence of events.

Use Detective when you need to answer:

- What did the principal do before and after the alert?
- Which resources communicated with each other?
- Is the activity consistent with normal behavior?
- Did the issue spread across accounts or regions?

Detective is especially useful for:

- Compromised IAM credentials
- EC2 instance compromise
- Lateral movement analysis
- Suspicious API call sequences
- Network-based investigation across VPCs

A practical workflow is to start with a GuardDuty finding, then use Detective to expand the timeline and identify related principals, IPs, and resources.

**AWS Security Lake and AWS Audit Manager**

Security Lake and Audit Manager support broader correlation and evidence collection.

- **Security Lake** centralizes security data from multiple sources into a queryable lake format, which helps when you need cross-service or cross-account analysis at scale.
- **Audit Manager** helps collect evidence for controls and can support investigations by showing which controls were in place and what evidence exists for them.

Use them when:

- You need long-term, centralized security analytics
- You need to correlate posture issues with control evidence
- You need to support audit and incident review with repeatable evidence collection

**Containment strategies by service**

Containment should reduce risk quickly while preserving evidence. Choose the least disruptive action that blocks further abuse.

**EC2 and network containment**

For EC2, prefer network isolation over destructive actions when you still need the instance for forensics.

- Remove or restrict security group ingress and egress.
- Move the instance to a quarantine security group.
- Use NACLs (network ACLs) or route changes if you need broader subnet isolation.
- Snapshot attached EBS volumes before making major changes.
- Preserve instance metadata and logs if possible.

Example security group quarantine approach:

```bash
aws ec2 modify-instance-attribute \
  --instance-id i-0123456789abcdef0 \
  --groups sg-quarantine
```

Use caution with instances that have multiple ENIs (Elastic Network Interfaces). Security group changes may not fully isolate all traffic paths if you do not account for every attached interface.

**When to use Session Manager or EC2 Instance Connect**

Use safer access methods when you need to inspect a host without opening SSH or RDP broadly.

- **AWS Systems Manager Session Manager** is preferred when the instance is managed by SSM and you want shell access without inbound ports.
- **EC2 Instance Connect** is useful for temporary SSH access when you need direct login and the environment supports it.
- Avoid exposing SSH or RDP to the internet during incident response unless there is a controlled, temporary, and well-logged reason.

Typical Session Manager access:

```bash
aws ssm start-session --target i-0123456789abcdef0
```

Use Session Manager when:

- You need interactive investigation
- You want to avoid inbound firewall changes
- You need session logging and IAM-based access control

**S3 containment**

For S3 incidents, focus on access control and public exposure.

- Block public access at the account or bucket level.
- Remove overly permissive bucket policies or ACLs.
- Rotate or disable credentials that can access the bucket.
- Review access logs and CloudTrail data events if enabled.
- Consider object lock for evidence preservation.

Example:

```bash
aws s3api put-public-access-block \
  --bucket example-bucket \
  --public-access-block-configuration \
  BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true
```

**IAM containment**

IAM incidents often require rapid credential and permission containment.

- Disable or delete exposed access keys.
- Remove risky inline policies or detach managed policies.
- Revoke active sessions where possible by changing credentials or role trust conditions.
- Tighten trust policies for roles that were abused.
- Use permission boundaries or SCPs (service control policies) to limit blast radius in Organizations.

Important nuance:

- Managed policies support versioning and rollback.
- Inline policies do not have version history, so you must preserve copies before editing.

Example access key disablement:

```bash
aws iam update-access-key \
  --user-name compromised-user \
  --access-key-id AKIAEXAMPLE \
  --status Inactive
```

**KMS containment**

AWS KMS (Key Management Service) issues can affect both encryption and decryption.

- Disabling a KMS key stops new cryptographic operations and can also prevent access to data already encrypted under that key.
- Changing a key policy can block legitimate workloads or preserve access for responders.
- Use caution before disabling a key if production data depends on it.

Example:

```bash
aws kms disable-key --key-id arn:aws:kms:us-east-1:123456789012:key/example
```

Use key disablement when the key is suspected of misuse or when you need to stop further use quickly, but understand the availability impact.

**RDS containment**

For Amazon RDS, focus on network access and credentials.

- Restrict security groups attached to the DB instance.
- Rotate database credentials.
- Review parameter group changes and audit logs.
- Consider snapshotting before major changes if forensic review is needed.

**Lambda containment**

For AWS Lambda, reduce invocation paths and permissions.

- Remove event source mappings if a function is being abused.
- Tighten resource-based policies.
- Reduce IAM execution role permissions.
- Disable triggers from EventBridge, S3, SNS, or API Gateway if they are the abuse path.

Example:

```bash
aws lambda remove-permission \
  --function-name suspicious-function \
  --statement-id allow-public-invoke
```

**EKS containment**

For Amazon EKS, think in terms of cluster access, workload permissions, and network paths.

- Review Kubernetes RBAC and IAM roles for service accounts.
- Scale down or isolate suspicious workloads.
- Restrict node security groups and pod network policies.
- Rotate credentials and review kubeconfig access.
- Check for malicious DaemonSets, privileged pods, or exposed dashboards.

**CloudFront, API Gateway, and Route 53 containment**

These services often appear in internet-facing incidents.

- **CloudFront**: disable or restrict distributions, review origin access, and check for unauthorized cache behavior changes.
- **API Gateway**: tighten authorizers, usage plans, resource policies, and stage variables.
- **Route 53**: review DNS record changes, especially records that redirect traffic to attacker-controlled endpoints.

These services can be used to redirect users, expose backends, or bypass intended controls, so include them in your posture review when the incident affects public traffic.

**Account isolation and break-glass strategy**

Sometimes the safest response is to isolate an account or workload boundary.

- Use a dedicated break-glass role with strong controls and logging.
- Protect the root user with MFA and avoid routine use.
- Review Organizations SCPs to limit dangerous actions during an incident.
- Consider suspending or restricting access paths rather than deleting resources immediately.
- Use IAM Access Analyzer to understand unintended external access and to validate whether a resource policy exposes data or roles.

A break-glass role should be tightly controlled, monitored, and used only for emergency response.

**Automated remediation patterns**

Automation helps you respond consistently, but it must be safe and auditable.

Common patterns:

- EventBridge rule matches a finding or configuration change.
- Lambda function enriches the event and decides whether to remediate.
- Systems Manager Automation performs the approved containment step.
- SNS or ChatOps notifies responders.
- Human approval is required for high-impact actions.

Example EventBridge pattern for a Security Hub finding:

```json
{
  "source": ["aws.securityhub"],
  "detail-type": ["Security Hub Findings - Imported"],
  "detail": {
    "findings": {
      "Severity": {
        "Label": ["HIGH", "CRITICAL"]
      }
    }
  }
}
```

Example EventBridge target flow:

```text
Security Hub finding -> EventBridge rule -> Lambda enrichment -> SSM Automation -> quarantine action
```

When designing automation:

- Scope permissions narrowly.
- Use explicit approvals for destructive actions.
- Log every action taken by the automation role.
- Test on nonproduction resources first.
- Preserve evidence before remediation when possible.

**Practical correlation examples**

A few exam-style scenarios help connect the services.

- **Public S3 exposure**
  - Config shows the bucket is noncompliant with public access controls.
  - CloudTrail shows a bucket policy change.
  - GuardDuty or access logs show unusual reads.
  - Response: block public access, revoke risky access, preserve logs, and review object access.

- **Compromised IAM user**
  - CloudTrail shows unusual API calls from a new IP.
  - GuardDuty flags credential abuse.
  - Security Hub aggregates the finding with other alerts.
  - Response: disable keys, review role trust, investigate downstream actions in Detective.

- **EC2 compromise**
  - GuardDuty flags suspicious behavior on the instance.
  - Config shows a security group was opened recently.
  - CloudTrail identifies who changed the group.
  - Response: quarantine the instance, snapshot volumes, inspect with Session Manager, and correlate network activity in Detective.

- **Logging tampering**
  - CloudTrail delivery stops or trail settings change.
  - Config detects drift in logging-related resources.
  - Security Hub surfaces a high-severity control failure.
  - Response: isolate the account, restore logging, and preserve any remaining evidence.

**Exam focus: what to choose when**

Use the service that best matches the question intent.

| Need | Best fit | Why |
|---|---|---|
| Who changed a resource | CloudTrail | Shows API activity and identity context |
| Is a resource compliant now | AWS Config | Evaluates current configuration against rules |
| Is this likely malicious | GuardDuty | Detects suspicious activity patterns |
| What should I prioritize | Security Hub | Aggregates and ranks findings |
| How do I investigate the chain | Detective | Correlates activity across telemetry |
| How do I centralize security data | Security Lake | Stores data for broader analysis |
| How do I collect control evidence | Audit Manager | Organizes evidence for controls |

**Common pitfalls**

- Do not treat a Config finding as proof of compromise.
- Do not treat a GuardDuty finding as proof of a specific configuration change.
- Do not remediate before preserving evidence when the incident may require forensics.
- Do not assume one containment action fits every EC2 or network topology.
- Do not forget non-EC2 services such as Lambda, EKS, CloudFront, API Gateway, and Route 53.
- Do not rely on a single log source when CloudTrail, VPC Flow Logs, and DNS logs together provide a better timeline.

**Quick response checklist**

1. Confirm the alert source and severity.
2. Preserve logs and relevant resource state.
3. Check CloudTrail for the change event.
4. Check Config for current compliance.
5. Check GuardDuty and Security Hub for active threat context.
6. Use Detective or Security Lake to correlate related activity.
7. Contain with the least disruptive safe action.
8. Document what changed, what was affected, and what evidence you preserved.

**Key takeaway**

Posture issues become incident-response inputs when you can connect configuration drift, threat findings, and evidence trails into one timeline. For SCS-C02, the important skill is not just recognizing a bad configuration, but choosing the right AWS service to verify it, the right evidence source to explain it, and the right containment action to stop it without destroying proof.
