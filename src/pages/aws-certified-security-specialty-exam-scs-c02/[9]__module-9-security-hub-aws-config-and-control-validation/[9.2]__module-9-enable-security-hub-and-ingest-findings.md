## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Security Hub overview**

AWS Security Hub is a centralized security posture management and findings aggregation service. It helps you collect, normalize, prioritize, and act on security findings from AWS services, partner products, and custom sources.

Use Security Hub when you need:

- A single place to view security findings across accounts and Regions.
- Standardized findings in the <abbr title="AWS Security Finding Format">ASFF</abbr> (AWS Security Finding Format).
- Security standards checks such as AWS Foundational Security Best Practices and CIS benchmarks.
- Automated response workflows through Amazon EventBridge, AWS Lambda, AWS Systems Manager, or ticketing tools.

Security Hub is not a detector by itself for most issues. It aggregates and normalizes findings from integrated sources, then adds its own controls and standards checks.

**Core concepts**

- **Finding**: A security issue, observation, or alert represented in ASFF.
- **Insight**: A saved query or aggregation view over findings.
- **Standard**: A benchmark or best-practice set of controls.
- **Control**: A specific security check within a standard.
- **Product source**: A service or partner that publishes findings into Security Hub.
- **Workflow status**: Triage state for a finding, such as `NEW`, `NOTIFIED`, `RESOLVED`, or `SUPPRESSED`.

Security Hub operates Regionally. You enable it in each Region where you want to collect and manage findings. In an organization, you typically designate a delegated administrator and then enable organization-wide configuration.

**Enabling Security Hub in a single account**

You can enable Security Hub from the console or the CLI. The CLI is useful for automation and repeatable setup.

```bash
aws securityhub enable-security-hub \
  --enable-default-standards
```

This enables Security Hub in the current account and Region and turns on the default standards. In practice, you may choose to enable specific standards instead of all defaults, depending on your governance model.

To disable it later:

```bash
aws securityhub disable-security-hub
```

**Organization-wide enablement**

For AWS Organizations, Security Hub supports delegated administration and automatic account enrollment. The typical flow is:

1. In the management account, designate a delegated administrator for Security Hub.
2. In the delegated administrator account, enable Security Hub.
3. Configure organization settings so new and existing member accounts are automatically enabled.
4. Ensure member accounts are associated with the organization configuration in each target Region.

A common CLI sequence looks like this:

```bash
aws securityhub enable-organization-admin-account \
  --admin-account-id 111122223333
```

Then, in the delegated administrator account:

```bash
aws securityhub enable-security-hub \
  --enable-default-standards
```

To configure automatic account enrollment, use the organization configuration API with the current valid parameter structure for your CLI version and Region. The exact syntax can vary over time, so verify with `aws securityhub help` or the AWS CLI reference before using it in automation.

> For exam purposes, remember that enabling Security Hub in the management account alone does not automatically enable it in all member accounts. Organization configuration is a separate step.

**Standards and controls**

Security Hub can evaluate controls from managed standards. These controls produce findings when a resource does not meet the expected configuration.

Common exam-relevant points:

- Standards are enabled per Region.
- Controls can be enabled or disabled individually in some workflows.
- Findings from controls are usually generated by Security Hub itself, not by the underlying service.
- A control failure often indicates a configuration gap, not necessarily an active attack.

Examples of standards you may encounter include:

- AWS Foundational Security Best Practices
- CIS AWS Foundations Benchmark
- PCI DSS-related checks, where supported in your environment

**Finding sources and ingestion**

Security Hub ingests findings from multiple source types:

- **AWS native services** such as Amazon GuardDuty, Amazon Inspector, AWS Config, and IAM Access Analyzer.
- **Partner products** that integrate with Security Hub.
- **Custom findings** that you publish yourself using the Security Hub API.

The exact behavior differs by source:

- GuardDuty and Inspector generate security findings that Security Hub normalizes.
- AWS Config contributes compliance-related findings and control evaluations.
- IAM Access Analyzer contributes findings about unintended access exposure.
- Partner products may publish findings through their integration or through imported ASFF.
- Custom findings are useful when you want to push results from your own scanners, scripts, or SIEM workflows.

**ASFF basics**

ASFF is the normalized JSON structure Security Hub uses for findings. When you import custom findings, you must provide valid source metadata and a well-formed finding document.

A practical custom finding example:

```json
{
  "SchemaVersion": "2018-10-08",
  "Id": "custom-scan-12345",
  "ProductArn": "arn:aws:securityhub:us-east-1:111122223333:product/111122223333/default",
  "GeneratorId": "custom-vulnerability-scanner",
  "AwsAccountId": "111122223333",
  "Types": [
    "Software and Configuration Checks/Vulnerabilities/CVE"
  ],
  "CreatedAt": "2026-04-26T12:00:00Z",
  "UpdatedAt": "2026-04-26T12:00:00Z",
  "Severity": {
    "Label": "HIGH"
  },
  "Title": "Outdated package detected on EC2 instance",
  "Description": "The scanner detected a vulnerable package version on the instance.",
  "Resources": [
    {
      "Type": "AwsEc2Instance",
      "Id": "i-0123456789abcdef0",
      "Region": "us-east-1"
    }
  ],
  "RecordState": "ACTIVE"
}
```

Important ASFF notes:

- `ProductArn` must reference a valid source in the account and Region.
- `Id` should be stable for the same finding so updates can correlate correctly.
- `CreatedAt` and `UpdatedAt` use ISO 8601 timestamps.
- `Resources` should identify the affected asset clearly.
- `Severity` can use labels such as `INFORMATIONAL`, `LOW`, `MEDIUM`, `HIGH`, or `CRITICAL`.

To import findings, use the Security Hub API:

```bash
aws securityhub batch-import-findings \
  --findings file://findings.json
```

If you are building a custom integration, ensure the source is enabled and trusted in the target account and Region. For partner products, follow the partner’s onboarding steps and Security Hub subscription requirements.

**Workflow and triage**

Security Hub workflow status helps you track how a finding is being handled.

Common statuses include:

- `NEW`: The finding has not been triaged.
- `NOTIFIED`: Someone has been alerted or notified.
- `RESOLVED`: The issue has been addressed.
- `SUPPRESSED`: The finding is intentionally ignored or filtered from active attention.

Use workflow status to separate active issues from accepted risk or false positives. This is especially useful when you automate ticket creation or incident response.

You can also update findings programmatically:

```bash
aws securityhub batch-update-findings \
  --finding-identifiers Id=custom-scan-12345,ProductArn=arn:aws:securityhub:us-east-1:111122223333:product/111122223333/default \
  --workflow Status=SUPPRESSED
```

**Insights and filtering**

Insights are saved searches or aggregations that help you focus on what matters.

Typical use cases:

- Top findings by severity.
- Findings by account or OU.
- Open findings by resource type.
- Findings that remain unresolved for too long.

A useful mental model is:

- Findings tell you what happened.
- Insights tell you what deserves attention first.

You can filter by fields such as:

- `SeverityLabel`
- `WorkflowStatus`
- `RecordState`
- `ComplianceStatus`
- `ResourceType`
- `AwsAccountId`
- `ProductName`

**Event-driven automation**

Security Hub integrates well with Amazon EventBridge. This is the preferred way to trigger automated response when findings are created or updated.

A typical pattern is:

1. Security Hub emits an event.
2. EventBridge matches the event pattern.
3. A target such as Lambda, SNS, or Step Functions runs remediation or notification logic.

Example EventBridge rule pattern for Security Hub findings:

```json
{
  "source": ["aws.securityhub"],
  "detail-type": ["Security Hub Findings - Imported"],
  "detail": {
    "findings": {
      "Severity": {
        "Label": ["HIGH", "CRITICAL"]
      },
      "Workflow": {
        "Status": ["NEW"]
      }
    }
  }
}
```

Use this as a starting point, not as a universal pattern. Security Hub can emit multiple event types, and the exact detail structure depends on the event and finding source.

A simple Lambda target can open a ticket or send a notification:

```python
import json

def lambda_handler(event, context):
    findings = event.get("detail", {}).get("findings", [])
    for finding in findings:
        title = finding.get("Title", "Unknown finding")
        severity = finding.get("Severity", {}).get("Label", "UNKNOWN")
        print(f"Finding: {title} severity={severity}")
    return {"processed": len(findings)}
```

**Practical setup checklist**

1. Enable Security Hub in the target Region.
2. Enable the standards you need.
3. Turn on the source services that generate findings.
4. Configure organization-wide administration if you use AWS Organizations.
5. Subscribe partner products or configure custom ingestion.
6. Create EventBridge rules for high-severity or compliance-related findings.
7. Review insights and workflow statuses regularly.

**CLI and configuration examples**

Enable Security Hub with default standards:

```bash
aws securityhub enable-security-hub \
  --enable-default-standards
```

List enabled standards:

```bash
aws securityhub get-enabled-standards
```

List findings:

```bash
aws securityhub get-findings \
  --filters '{"SeverityLabel":[{"Comparison":"EQUALS","Value":"HIGH"}]}'
```

Update a finding workflow status:

```bash
aws securityhub batch-update-findings \
  --finding-identifiers Id=custom-scan-12345,ProductArn=arn:aws:securityhub:us-east-1:111122223333:product/111122223333/default \
  --workflow Status=RESOLVED
```

Example JSON filter for open critical findings:

```json
{
  "SeverityLabel": [
    {
      "Comparison": "EQUALS",
      "Value": "CRITICAL"
    }
  ],
  "WorkflowStatus": [
    {
      "Comparison": "EQUALS",
      "Value": "NEW"
    }
  ]
}
```

**IAM permissions and access control**

Security Hub setup usually requires permissions in both IAM and AWS Organizations when you manage multiple accounts.

Common permission areas include:

- Enabling and disabling Security Hub.
- Managing standards and controls.
- Importing and updating findings.
- Managing delegated administrator settings.
- Reading organization membership and account relationships.

For organization administration, you typically also need AWS Organizations permissions such as:

- `organizations:EnableAWSServiceAccess`
- `organizations:RegisterDelegatedAdministrator`
- `organizations:ListAccounts`
- `organizations:DescribeOrganization`

For Security Hub operations, permissions may include:

- `securityhub:EnableSecurityHub`
- `securityhub:DisableSecurityHub`
- `securityhub:BatchImportFindings`
- `securityhub:BatchUpdateFindings`
- `securityhub:GetFindings`
- `securityhub:UpdateOrganizationConfiguration`

Use least privilege. Separate read-only analysts from administrators and automation roles.

**Troubleshooting**

If findings do not appear, check the following:

- Security Hub is enabled in the correct Region.
- The source service or partner integration is enabled and configured correctly.
- The account is part of the expected organization configuration.
- The finding source is publishing to the same Region where you are viewing Security Hub.
- The IAM role or user has permission to publish or read findings.
- The ASFF document is valid and uses the correct `ProductArn`.
- EventBridge rules match the actual event type and detail structure.

> Some integrations depend on the source service being enabled in the same Region, but partner products and custom findings may have different onboarding and publishing requirements. Do not assume every source behaves like a native AWS detector.

If a custom finding import fails, validate:

- Required ASFF fields are present.
- Timestamps are in valid ISO 8601 format.
- Resource identifiers are accurate.
- The product source exists and is subscribed in the account.

**Exam focus**

For SCS-C02, focus on these distinctions:

- Security Hub aggregates and normalizes findings; it is not the primary detector for most issues.
- Organization-wide enablement requires delegated administration and organization configuration.
- Findings can come from AWS services, partner products, and custom imports.
- ASFF is the standard format for imported findings.
- EventBridge is the main automation path for reacting to findings.
- Workflow status helps you triage findings without deleting evidence.

**Quick memory aid**

- **Enable** Security Hub per Region.
- **Aggregate** findings from multiple sources.
- **Normalize** into ASFF.
- **Triage** with workflow status and insights.
- **Automate** with EventBridge and Lambda.
