## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**AWS Config in security operations**

AWS Config records configuration changes, evaluates resources against rules, and helps you answer two core security questions: *what changed* and *is it compliant*. For exam purposes, think of it as the service that turns resource configuration into a timeline of evidence.

It is most useful when you need:
- Continuous visibility into configuration drift
- Compliance checks against internal baselines or managed best practices
- Centralized reporting across accounts and Regions
- Automated remediation for common misconfigurations

AWS Config is not a replacement for detection services like Amazon GuardDuty or AWS Security Hub. It complements them by providing configuration state, history, and rule-based compliance evaluation.

**Core data model**

AWS Config revolves around three related concepts:

- **Configuration item (CI)**: A point-in-time record of a resource’s configuration.
- **Configuration history**: The sequence of configuration items for a resource over time.
- **Snapshot**: A point-in-time export of recorded configuration state for a Region and account.

A configuration item is the atomic record used for change tracking and rule evaluation. A snapshot is useful for bulk export and audit evidence. History helps you reconstruct the sequence of changes that led to a noncompliant state.

<dl>
<dt><strong>Configuration item</strong></dt>
<dd>Represents the state of a resource at a specific time, including relationships and metadata.</dd>
<dt><strong>Configuration history</strong></dt>
<dd>Shows how a resource changed over time by listing successive configuration items.</dd>
<dt><strong>Snapshot</strong></dt>
<dd>Captures the recorded configuration state for many resources at a point in time.</dd>
</dl>

**Recording scope and global resources**

Before rules can evaluate anything, AWS Config must record resources. Recording scope is a common exam topic because rule results depend on what is actually being recorded.

You can configure AWS Config to:
- Record all supported resource types
- Record only specific resource types
- Include or exclude global resource types depending on the recorder setting

Global resource behavior matters in multi-Region designs:
- Global resources are recorded in the Region where AWS Config is enabled for that recorder.
- The `includeGlobalResourceTypes` setting controls whether global resource types are included.
- A rule in one Region does not automatically evaluate resources in another Region.

A typical recorder setup looks like this:

```bash
aws configservice put-configuration-recorder \
  --configuration-recorder '{
    "name":"default",
    "roleARN":"arn:aws:iam::123456789012:role/AWSConfigRole",
    "recordingGroup":{
      "allSupported":true,
      "includeGlobalResourceTypes":true
    }
  }'
```

> Be careful with shell quoting when passing JSON. In many shells, single quotes around the JSON payload are the safest copy/paste pattern.

**Rule evaluation modes**

AWS Config rules evaluate resources in two main ways:

- **Configuration change-triggered**: Evaluates when AWS Config detects a change to a recorded resource.
- **Periodic**: Evaluates on a schedule, even if no configuration change occurred.

Use change-triggered rules when compliance depends on immediate reaction to resource changes, such as security group ingress or encryption settings. Use periodic rules when the condition depends on time, external state, or a broader snapshot of the environment.

Operational differences:
- Change-triggered rules are efficient for event-driven compliance.
- Periodic rules can catch drift that is not tied to a single configuration event.
- Some managed rules support only one mode.
- Periodic evaluation is common for checks that need to inspect state over time.

A rule’s evaluation result is usually one of:
- `COMPLIANT`
- `NON_COMPLIANT`
- `NOT_APPLICABLE`

Some workflows also surface `INSUFFICIENT_DATA`, but do not assume every managed rule returns it in the same way. For exam questions, focus on the standard compliance states above.

**Managed rules and custom rules**

AWS Config supports both managed rules and custom rules.

- **Managed rules** are predefined by AWS and cover common security and operational controls.
- **Custom rules** use Lambda to implement organization-specific logic.

Managed rules are faster to deploy and easier to maintain. Custom rules are useful when you need logic that is not available in a managed rule, such as checking naming conventions, cross-resource relationships, or internal policy exceptions.

Managed rules often accept **parameters** so you can tune them for your environment. For example, a rule may allow you to specify:
- Approved ports
- Allowed CIDR ranges
- Required tag keys
- Maximum age thresholds

Example managed rule with parameters:

```json
{
  "ConfigRuleName": "restricted-ssh",
  "Description": "Disallow SSH from the internet",
  "Scope": {
    "ComplianceResourceTypes": [
      "AWS::EC2::SecurityGroup"
    ]
  },
  "Source": {
    "Owner": "AWS",
    "SourceIdentifier": "INCOMING_SSH_DISABLED"
  },
  "InputParameters": "{\"blockedPort1\":\"22\"}"
}
```

The exact parameter names vary by managed rule. Always verify the rule’s documented schema before deployment.

**Rule scope and targeting**

AWS Config rule scope is defined by the resources it evaluates, not by a generic “compliance scope” concept.

Common scope controls:
- `ComplianceResourceTypes`
- `ComplianceResourceId`
- Tag-based filters where supported by the rule and evaluation model

Scope matters because it reduces noise and cost, and it prevents rules from evaluating irrelevant resources.

Examples:
- Evaluate only `AWS::EC2::SecurityGroup` resources for ingress controls
- Evaluate only S3 buckets with a specific tag
- Evaluate a single resource during troubleshooting

A practical scope example:

```json
{
  "Scope": {
    "ComplianceResourceTypes": [
      "AWS::S3::Bucket"
    ]
  }
}
```

**Custom Lambda rules**

Custom rules use AWS Lambda to evaluate configuration items. AWS Config invokes the function with event data that includes the triggering context and a `resultToken`.

Important operational detail:
- Configuration-change evaluations and periodic evaluations do not always have the same event shape.
- Your Lambda code should handle both cases.
- Do not assume `configurationItem` is always present.

A safer Python pattern:

```python
import json

def lambda_handler(event, context):
    invoking_event = json.loads(event.get("invokingEvent", "{}"))
    result_token = event["resultToken"]

    configuration_item = invoking_event.get("configurationItem")
    message_type = invoking_event.get("messageType")

    if message_type == "ScheduledNotification":
        # Periodic evaluation path
        # Query current state or use the configuration snapshot context
        compliance_type = "COMPLIANT"
    elif configuration_item:
        # Configuration change path
        resource_type = configuration_item.get("resourceType")
        resource_id = configuration_item.get("resourceId")
        compliance_type = "NON_COMPLIANT" if resource_type == "AWS::EC2::SecurityGroup" else "COMPLIANT"
    else:
        compliance_type = "NOT_APPLICABLE"

    # In real code, call put_evaluations with the result token
    return {
        "resultToken": result_token,
        "complianceType": compliance_type
    }
```

In production, the function must call `PutEvaluations` to report the result back to AWS Config.

**Rule lifecycle and compliance history**

AWS Config is valuable because it preserves compliance over time, not just current state.

Use it to answer questions like:
- When did the resource become noncompliant?
- What change caused the violation?
- Was the resource ever compliant after the change?
- Did a remediation action restore compliance?

A resource can move through multiple states:
- `COMPLIANT` after a secure change
- `NON_COMPLIANT` after drift or a risky update
- `NOT_APPLICABLE` when the resource is outside the rule’s scope

For investigations, compare:
- The current configuration item
- The previous configuration item
- The rule evaluation timeline
- The resource’s relationships to other resources

**Advanced query and search**

AWS Config advanced queries let you search recorded configuration data using SQL-like syntax. The query must include a `FROM` clause.

Example query for S3 buckets:

```sql
SELECT
  resourceId,
  resourceType,
  accountId,
  awsRegion,
  configuration.bucketName
FROM
  resources
WHERE
  resourceType = 'AWS::S3::Bucket'
```

Example query for EC2 instances:

```sql
SELECT
  resourceId,
  resourceType,
  accountId,
  awsRegion,
  configuration.instanceType
FROM
  resources
WHERE
  resourceType = 'AWS::EC2::Instance'
```

Use advanced queries when you need:
- Inventory searches
- Cross-account analysis through aggregators
- Evidence for audits
- Fast filtering by resource type, account, Region, or tags

**Aggregators and centralized compliance**

AWS Config aggregators provide a centralized view across multiple accounts and Regions.

Use an aggregator when you need:
- Organization-wide compliance reporting
- Cross-account inventory searches
- A single place to query configuration history and compliance status

Aggregator behavior:
- It collects data from source accounts and Regions.
- It supports centralized queries and dashboards.
- It is commonly paired with AWS Organizations for enterprise governance.

A typical architecture:
- Member accounts record resources with AWS Config
- A delegated administrator account hosts the aggregator
- Security operations queries compliance centrally

This is especially useful in multi-account security operations because it reduces the need to inspect each account manually.

**Organizations, delegated admin, and organization-wide deployment**

AWS Config supports organization-level governance patterns.

Key features:
- **Organization rules**: Deploy a rule across accounts in an AWS Organization.
- **Organization conformance packs**: Deploy a pack of rules and remediation actions across accounts and OUs.
- **Delegated administrator**: A designated account manages AWS Config organization features on behalf of the organization.

Why this matters:
- You can enforce baseline controls consistently.
- You can target specific organizational units (OUs).
- You can reduce manual setup in each account.

Typical deployment flow:
1. Enable AWS Config in member accounts and Regions.
2. Register a delegated administrator for AWS Config.
3. Deploy organization rules or conformance packs from the management or delegated admin account.
4. Monitor compliance centrally through aggregators and dashboards.

**Conformance packs**

A conformance pack is a packaged set of AWS Config rules and optional remediation actions defined in YAML.

Use conformance packs when you want:
- A reusable compliance baseline
- Consistent deployment across accounts and Regions
- A documented control set for audits

Example conformance pack YAML:

```yaml
Resources:
  S3BucketVersioningEnabled:
    Type: AWS::Config::ConfigRule
    Properties:
      ConfigRuleName: s3-bucket-versioning-enabled
      Description: Ensure S3 buckets have versioning enabled
      Scope:
        ComplianceResourceTypes:
          - AWS::S3::Bucket
      Source:
        Owner: AWS
        SourceIdentifier: S3_BUCKET_VERSIONING_ENABLED

  S3BucketPublicReadProhibited:
    Type: AWS::Config::ConfigRule
    Properties:
      ConfigRuleName: s3-bucket-public-read-prohibited
      Description: Prevent public read access on S3 buckets
      Scope:
        ComplianceResourceTypes:
          - AWS::S3::Bucket
      Source:
        Owner: AWS
        SourceIdentifier: S3_BUCKET_PUBLIC_READ_PROHIBITED
```

A conformance pack can also include remediation actions. In practice, remediation often uses AWS Systems Manager Automation documents or Lambda functions.

**Remediation**

AWS Config can trigger remediation when a resource is noncompliant.

Common remediation patterns:
- **SSM Automation** document execution
- **Lambda**-based custom remediation
- Manual approval workflows for sensitive changes

Important prerequisites:
- The remediation target must exist and be valid for the rule.
- The execution role must have permission to perform the change.
- The automation document must match the remediation use case.
- Some remediations are automatic; others require manual approval or invocation.

Example remediation configuration pattern:

```json
{
  "ConfigRuleName": "s3-bucket-public-read-prohibited",
  "TargetType": "SSM_DOCUMENT",
  "TargetId": "AWS-DisableS3BucketPublicReadWrite",
  "Automatic": true,
  "Parameters": {
    "AutomationAssumeRole": {
      "StaticValue": {
        "Values": [
          "arn:aws:iam::123456789012:role/ConfigRemediationRole"
        ]
      }
    },
    "BucketName": {
      "ResourceValue": {
        "Value": "RESOURCE_ID"
      }
    }
  }
}
```

Use remediation carefully:
- Test in a nonproduction account first
- Confirm the action is idempotent
- Ensure the role has least privilege
- Verify the document name and parameters for the specific rule

> Remediation is usually implemented as an automation workflow that changes configuration, not as a generic “reattach” action. Always map the remediation to a concrete API or automation document.

**Security operations use cases**

AWS Config is especially useful in security operations for:
- Detecting public exposure of storage or network resources
- Enforcing encryption and logging baselines
- Tracking IAM-related configuration drift
- Proving compliance during audits
- Investigating the sequence of changes before an incident

Examples of common controls:
- S3 bucket public access restrictions
- EBS volume encryption
- Security group ingress restrictions
- CloudTrail enabled in all Regions
- IAM password policy and MFA-related checks where applicable

**Rule examples and interpretation**

A managed rule such as `INCOMING_SSH_DISABLED` is commonly used to check that security groups do not allow SSH from overly permissive sources. The exact evaluation depends on the ingress rules, protocol, port, and CIDR conditions.

Interpretation tips:
- Do not reduce the rule to a single literal condition.
- Check the full security group ingress set.
- Remember that a rule may evaluate a resource as compliant even if one rule is safe but another is not, depending on the exact managed rule logic.

Another common pattern is encryption enforcement:
- A rule may check whether a resource is encrypted.
- The resource can be compliant only after the encryption setting is enabled and recorded.
- If the resource changes later, the rule can flip back to noncompliant.

**CLI examples for operations**

List available rules:

```bash
aws configservice describe-config-rules
```

Get compliance details for a specific rule:

```bash
aws configservice get-compliance-details-by-config-rule \
  --config-rule-name s3-bucket-public-read-prohibited
```

List compliance by resource:

```bash
aws configservice get-compliance-details-by-resource \
  --resource-type AWS::S3::Bucket \
  --resource-id my-example-bucket
```

Query an aggregator:

```bash
aws configservice select-aggregate-resource-config \
  --configuration-aggregator-name org-aggregator \
  --expression "SELECT resourceId, resourceType, accountId, awsRegion FROM resources WHERE resourceType = 'AWS::EC2::Instance'"
```

These commands are useful for incident response and audit evidence collection. Some responses are paginated, so plan to handle multiple pages in automation.

**Common exam distinctions**

AWS Config questions often test subtle differences:

- **Recorder vs rule**
  - The recorder captures configuration.
  - The rule evaluates recorded configuration.

- **Snapshot vs history**
  - A snapshot is a point-in-time export.
  - History is the sequence of changes for a resource.

- **Managed rule vs custom rule**
  - Managed rules are AWS-defined.
  - Custom rules are Lambda-defined.

- **Periodic vs change-triggered**
  - Periodic rules run on a schedule.
  - Change-triggered rules run when a resource changes.

- **Aggregator vs conformance pack**
  - Aggregators centralize visibility.
  - Conformance packs standardize controls and remediation.

- **Organization rule vs account rule**
  - Organization rules deploy broadly across accounts/OUs.
  - Account rules are local to one account.

**Limitations and practical cautions**

AWS Config is powerful, but it has boundaries:
- Not every AWS resource type is supported by every rule.
- Managed rules only work for the resource types and conditions they were designed for.
- Custom rules require Lambda and careful handling of event formats.
- Some controls are Region-specific and must be deployed in each Region you care about.
- Global resource recording behavior depends on recorder settings and Region placement.

For exam scenarios, always ask:
- Is the resource recorded?
- Is the rule supported for that resource type?
- Is the evaluation mode appropriate?
- Is the control deployed in the right account and Region?
- Is centralized visibility required through an aggregator?

**Practical deployment checklist**

1. Enable AWS Config recording in each required account and Region.
2. Decide whether to record all supported resources or a subset.
3. Confirm global resource recording behavior with `includeGlobalResourceTypes`.
4. Deploy managed rules for common controls.
5. Add custom rules only where managed rules do not fit.
6. Use parameters to tailor rules to your environment.
7. Add remediation for repeatable, low-risk fixes.
8. Centralize visibility with aggregators.
9. Use organization rules or conformance packs for consistent governance.
10. Review compliance history after changes and incidents.

**High-value exam takeaways**

- AWS Config records configuration state and evaluates compliance over time.
- Recording scope and Region placement determine what rules can see.
- Periodic and configuration-change-triggered rules solve different problems.
- Aggregators provide centralized cross-account and cross-Region visibility.
- Organization rules and conformance packs scale governance across AWS Organizations.
- Remediation usually uses SSM Automation or Lambda, with explicit permissions and parameters.
- Advanced queries require valid SQL-like syntax with a `FROM` clause.
- Managed rules are convenient, but custom rules are necessary for bespoke logic.
