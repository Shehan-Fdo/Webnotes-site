## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Remediation workflow overview**

Noncompliance remediation is the process of detecting a security or configuration drift, deciding whether to fix it automatically or with approval, and then verifying that the fix succeeded. In AWS, the most common building blocks are **AWS Config** for compliance evaluation, **Amazon EventBridge** for event routing, **AWS Systems Manager Automation** for execution, and **AWS Security Hub** for centralized finding management.

For exam scenarios, separate the problem into three questions:

- **What detected the issue?** Config evaluates resource configuration against rules; Security Hub aggregates findings from many services.
- **What should trigger action?** EventBridge can route compliance changes or findings to automation.
- **How should the fix run?** SSM Automation, Lambda, Step Functions, or a manual approval workflow can execute remediation.

A good remediation design also includes verification, retry handling, and escalation when automation cannot safely complete the fix.

**Core AWS services in the workflow**

AWS Config is the primary service for configuration compliance. It records resource configuration history, evaluates resources against managed or custom rules, and can attach remediation actions to rules. Security Hub is the primary service for security findings aggregation and prioritization. EventBridge is the event bus that connects detection to action. Systems Manager provides the automation engine, runbooks, and approval controls used to perform the actual change.

- **AWS Config**
  - Detects noncompliant resource configuration.
  - Supports managed rules and custom rules.
  - Can associate a remediation action with a rule.
- **AWS Security Hub**
  - Aggregates findings from AWS services and partner products.
  - Supports custom actions and automation rules.
  - Can archive or suppress findings after triage or remediation.
- **Amazon EventBridge**
  - Routes compliance or finding events to targets.
  - Supports filtering by event source, detail type, and key fields.
- **AWS Systems Manager Automation**
  - Runs runbooks to change resources safely and repeatably.
  - Supports approvals, branching, and parameterized actions.
- **AWS Lambda**
  - Useful for custom logic, enrichment, or lightweight remediation.
  - Best when the fix is simple or when you need event parsing and orchestration.
- **AWS Step Functions**
  - Useful when remediation requires multiple steps, retries, or human approval.
- **AWS Systems Manager Change Manager**
  - Useful for controlled remediation with change approvals and auditability.

**When to choose each service**

Scenario-based questions often test whether you understand the difference between compliance evaluation and finding aggregation.

| Need | Best fit | Why |
|---|---|---|
| Evaluate whether an S3 bucket, security group, or IAM setting violates a rule | AWS Config | It continuously evaluates resource configuration against rules |
| Centralize security findings from multiple services | Security Hub | It normalizes and aggregates findings |
| Trigger automation from compliance or finding events | EventBridge | It routes events to targets based on patterns |
| Execute a repeatable fix on AWS resources | SSM Automation | It is designed for operational runbooks |
| Add branching, retries, and approvals | Step Functions | It orchestrates complex workflows |
| Require formal change approval | Change Manager | It adds governance and approval controls |

> If the question is about whether a resource is compliant, think **Config**. If it is about a security finding that needs triage or aggregation, think **Security Hub**. If it is about moving from detection to action, think **EventBridge** and **SSM Automation**.

**AWS Config remediation workflow**

AWS Config remediation is the most direct exam topic for noncompliance response. A Config rule evaluates a resource, and a remediation action can be attached to that rule so that noncompliant resources are fixed automatically or with manual approval.

Typical flow:

1. AWS Config evaluates a resource against a rule.
2. The rule marks the resource as compliant or noncompliant.
3. A remediation configuration points to an SSM Automation document or another supported action.
4. AWS Config can trigger the remediation automatically or notify an operator for manual execution.
5. The resource is re-evaluated to confirm compliance.

Important remediation configuration concepts:

- **Automatic remediation**
  - AWS Config starts the remediation action when the rule detects noncompliance.
  - Use this for low-risk, repeatable fixes.
- **Manual remediation**
  - AWS Config records the remediation action, but a human must start it.
  - Use this when the change is risky or requires approval.
- **Maximum automatic attempts**
  - Limits how many times Config retries remediation.
  - Helps prevent loops when the fix fails repeatedly.
- **Retry window**
  - Controls how long Config keeps retrying.
  - Useful when the resource may be temporarily unavailable.
- **SSM Automation document**
  - Commonly used as the remediation action.
  - Encapsulates the steps needed to fix the resource.

A remediation configuration is attached to a specific Config rule, not to AWS Config globally. That distinction matters in exam questions: you remediate the control that detected the issue.

Example: attach a remediation action to a Config rule using the AWS CLI.

```bash
aws configservice put-remediation-configurations \
  --remediation-configurations '[
    {
      "ConfigRuleName": "restricted-ssh",
      "TargetType": "SSM_DOCUMENT",
      "TargetId": "AWS-DisablePublicAccessForSecurityGroup",
      "Automatic": true,
      "MaximumAutomaticAttempts": 3,
      "RetryAttemptSeconds": 60,
      "Parameters": {
        "GroupId": {
          "ResourceValue": {
            "Value": "RESOURCE_ID"
          }
        }
      }
    }
  ]'
```

This example shows the pattern, not a universal template. In practice, the target document and parameters must match the specific rule and resource type.

**SSM Automation as the remediation engine**

Systems Manager Automation is the most common execution layer for Config remediation. It lets you define a runbook that performs the fix in a controlled way. Automation is preferred over ad hoc scripts because it is repeatable, auditable, and parameterized.

Common Automation document actions include:

- `aws:executeAwsApi` for direct AWS API calls.
- `aws:runCommand` for running commands on managed instances.
- `aws:branch` for conditional logic.
- `aws:pause` for approval or wait steps.
- `aws:executeScript` for custom logic.

A minimal remediation runbook for a security group rule might look like this:

```yaml
schemaVersion: '0.3'
description: "Remove an overly permissive inbound rule from a security group"
parameters:
  GroupId:
    type: String
  IpProtocol:
    type: String
    default: tcp
  FromPort:
    type: String
    default: '22'
  ToPort:
    type: String
    default: '22'
  CidrIp:
    type: String
    default: 0.0.0.0/0
mainSteps:
  - name: revokeIngress
    action: aws:executeAwsApi
    inputs:
      Service: ec2
      Api: RevokeSecurityGroupIngress
      GroupId: "{{ GroupId }}"
      IpPermissions:
        - IpProtocol: "{{ IpProtocol }}"
          FromPort: "{{ FromPort }}"
          ToPort: "{{ ToPort }}"
          IpRanges:
            - CidrIp: "{{ CidrIp }}"
```

Use this pattern carefully. Real-world remediation must match the exact offending permission, and security group rules can include multiple CIDRs, IPv6 ranges, prefix lists, or source security groups.

**Approval-based remediation**

Not every fix should be automatic. High-impact changes often require approval, especially when the remediation could interrupt traffic or remove access. Approval-based remediation is common for production environments.

You can implement approval in several ways:

- **SSM Automation pause step**
  - The runbook pauses until an approver resumes execution.
  - Good for simple approval gates.
- **Step Functions manual approval**
  - A workflow can wait for a callback token or approval event.
  - Good for multi-step remediation with branching.
- **Change Manager**
  - Provides formal change requests, approvals, and audit trails.
  - Good for regulated environments.

Example of a pause step in Automation:

```yaml
mainSteps:
  - name: waitForApproval
    action: aws:pause
    inputs:
      Duration: PT1H
```

This is a simple gate, not a full governance process. For production change control, Change Manager is usually the stronger answer.

**EventBridge-driven remediation**

EventBridge is useful when you want to react to compliance or finding events without tightly coupling the detector and the fixer. It is often the glue between Config, Security Hub, Lambda, and Step Functions.

Use EventBridge when:

- You need to route events from multiple sources.
- You want to fan out to several targets.
- You need filtering before invoking automation.
- You want to trigger a workflow only for certain severities or resource types.

For AWS Config, event patterns are usually based on Config compliance change events or configuration item changes. For Security Hub, patterns are usually based on Security Hub finding events. The exact event structure varies by service and event type, so avoid memorizing a single brittle JSON filter as a universal answer.

Example EventBridge rule for Security Hub findings:

```json
{
  "source": ["aws.securityhub"],
  "detail-type": ["Security Hub Findings - Imported"],
  "detail": {
    "findings": {
      "Severity": {
        "Label": ["CRITICAL", "HIGH"]
      },
      "RecordState": ["ACTIVE"]
    }
  }
}
```

This is a common pattern, but you should still validate the actual event shape in the scenario. Security Hub findings are nested and can include multiple findings in one event.

Example EventBridge target to invoke a Lambda function:

```bash
aws events put-targets \
  --rule high-severity-securityhub-findings \
  --targets "Id"="1","Arn"="arn:aws:lambda:us-east-1:123456789012:function:remediate-finding"
```

**Security Hub remediation workflow**

Security Hub is not the remediation engine itself. It is the finding aggregator and triage layer. The workflow usually looks like this:

1. Security Hub ingests a finding from AWS or a partner product.
2. An automation rule, custom action, or EventBridge rule selects the finding.
3. A remediation workflow runs, often through Lambda, Step Functions, or SSM Automation.
4. The finding is updated, archived, or suppressed after remediation.

Security Hub-specific features to know:

- **Custom actions**
  - Let an operator trigger a workflow from the Security Hub console.
  - Useful for manual triage and controlled remediation.
- **Automation rules**
  - Automatically update findings based on conditions.
  - Can help route, archive, or suppress findings.
- **Archiving**
  - Marks a finding as no longer active in the workflow.
  - Useful after remediation or when the finding is accepted risk.
- **Suppression**
  - Reduces noise for known, accepted conditions.
  - Use carefully to avoid hiding real issues.

A practical Security Hub remediation pattern is to send high-severity active findings to EventBridge, then invoke Step Functions or Lambda to decide whether to fix, suppress, or escalate.

**Lambda remediation patterns**

Lambda is useful when remediation requires custom parsing, enrichment, or a quick API call. It is not ideal for long-running or multi-step changes.

Good Lambda use cases:

- Normalize event payloads from different sources.
- Look up tags, account metadata, or ownership.
- Decide whether a finding is safe to auto-remediate.
- Trigger an SSM Automation document or Step Functions state machine.

Be careful with Security Hub events. A finding event can contain multiple findings, so your code should iterate rather than assume a single item.

Example Lambda handler pattern:

```python
import json

def handler(event, context):
    findings = event.get("detail", {}).get("findings", [])
    if not findings:
        return {"status": "no-findings"}

    results = []
    for finding in findings:
        severity = finding.get("Severity", {}).get("Label", "UNKNOWN")
        record_state = finding.get("RecordState")
        product_arn = finding.get("ProductArn")

        if severity in ("CRITICAL", "HIGH") and record_state == "ACTIVE":
            # Trigger downstream remediation here
            results.append({
                "id": finding.get("Id"),
                "action": "remediate",
                "productArn": product_arn
            })
        else:
            results.append({
                "id": finding.get("Id"),
                "action": "ignore"
            })

    return {"results": results}
```

This is intentionally simplified. In production, validate the event source, handle idempotency, and log the remediation decision.

**Cross-account and multi-region remediation**

Large AWS environments often span multiple accounts and regions. Remediation design must account for where the finding is generated, where the automation runs, and which account owns the resource.

Key considerations:

- **AWS Organizations**
  - Centralize governance across accounts.
  - Use delegated administrator models where supported.
- **Multi-region**
  - Security Hub and Config are regional services.
  - You may need to enable them in each region you care about.
- **Cross-account automation**
  - EventBridge can route events across accounts.
  - SSM Automation can assume roles in target accounts if configured.
- **Resource ownership**
  - The account that owns the resource usually needs permission to change it.
  - Central security teams often trigger remediation in workload accounts.

A common pattern is centralized detection with decentralized execution:

- Security Hub aggregates findings centrally.
- EventBridge forwards selected findings to a remediation account or target account.
- SSM Automation assumes a role in the workload account to make the change.

**Conformance packs and standardization**

AWS Config conformance packs help you deploy a standardized set of Config rules and remediation actions across accounts and regions. They are useful when you want consistent compliance controls at scale.

Use conformance packs when:

- You need the same baseline across many accounts.
- You want to package rules and remediation together.
- You need repeatable deployment through Organizations or infrastructure as code.

Conformance packs are especially relevant in exam scenarios involving enterprise governance. They reduce drift in compliance controls and make it easier to enforce a standard remediation posture.

**Handling failures, retries, and escalation**

A robust remediation workflow must assume that some fixes fail. Failures can happen because of missing permissions, conflicting resource state, service limits, or a change that requires human judgment.

Design for failure with these controls:

- **Retries**
  - Use bounded retries for transient issues.
  - Avoid infinite loops.
- **Dead-lettering**
  - Send failed events to an SQS queue or another durable target for later analysis.
  - Useful when the downstream system is unavailable.
- **Escalation**
  - Notify an operations team through SNS, email, or chat integration.
  - Use when automation cannot safely complete the fix.
- **Idempotency**
  - Make the remediation safe to run more than once.
  - Prevent duplicate changes from causing damage.
- **Verification**
  - Re-check compliance after the action completes.
  - Confirm the resource is actually fixed.

Example escalation pattern:

```bash
aws sns publish \
  --topic-arn arn:aws:sns:us-east-1:123456789012:security-remediation-alerts \
  --subject "Remediation failed" \
  --message "Security group sg-0123456789abcdef0 could not be remediated automatically."
```

**Practical remediation examples**

Security group overly permissive inbound rule:

- Detect with AWS Config managed rule or custom rule.
- Remediate with SSM Automation or Lambda.
- Use EventBridge if you want to route only certain severities or accounts.
- Verify by re-evaluating the rule after the ingress is revoked.

Unencrypted EBS volume:

- Detect with AWS Config.
- Remediate by creating an encrypted replacement volume from an encrypted snapshot, then reattaching it.
- Do not assume this is an in-place encryption toggle; it is usually a replacement workflow.
- Plan for downtime or a controlled cutover.

Public S3 bucket:

- Detect with AWS Config or Security Hub.
- Remediate by removing public ACLs or bucket policies and enabling block public access where appropriate.
- Verify that the bucket still supports intended access patterns.

IAM policy too permissive:

- Detect with Config or Security Hub depending on the control.
- Remediate by tightening the policy or removing the risky statement.
- Use approval if the policy affects production access.

**IAM and permissions for remediation**

The remediation role needs permission to perform the fix and to read the resource state. For many EC2 security group actions, resource-level scoping is limited, so `Resource: "*"` is common and sometimes required by the service.

Example IAM policy fragment for a remediation role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowSecurityGroupRemediation",
      "Effect": "Allow",
      "Action": [
        "ec2:DescribeSecurityGroups",
        "ec2:RevokeSecurityGroupIngress"
      ],
      "Resource": "*"
    }
  ]
}
```

This is not a sign of poor design by default; it reflects how some EC2 APIs authorize access. Where resource-level permissions are supported, use them. Where they are not, scope by condition keys, role trust, account boundaries, and operational controls.

**Exam focus and decision cues**

Use these cues to answer scenario questions quickly:

- **Config rule + remediation action**
  - Think automatic or manual remediation attached to a compliance rule.
- **Security Hub finding + custom action**
  - Think triage, aggregation, and operator-driven or event-driven response.
- **EventBridge rule**
  - Think event filtering and routing.
- **SSM Automation**
  - Think repeatable fix with parameters and auditability.
- **Step Functions or Change Manager**
  - Think approval, orchestration, or controlled change.
- **Conformance pack**
  - Think standardized compliance across many accounts or regions.

A strong answer usually includes detection, execution, verification, and escalation. If the scenario mentions enterprise scale, add Organizations, cross-account roles, and multi-region enablement. If it mentions production risk, add approval and change control.

**Quick reference**

- **AWS Config** evaluates compliance.
- **Security Hub** aggregates findings.
- **EventBridge** routes events.
- **SSM Automation** performs remediation.
- **Step Functions** orchestrates complex or approved workflows.
- **Change Manager** governs controlled remediation.
- **Conformance packs** standardize controls across accounts and regions.
