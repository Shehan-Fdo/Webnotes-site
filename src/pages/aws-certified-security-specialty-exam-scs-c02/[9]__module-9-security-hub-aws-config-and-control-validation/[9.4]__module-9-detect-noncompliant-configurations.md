## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Overview**

Detecting noncompliant configurations in AWS usually starts with **AWS Config** and expands into **Security Hub**, **EventBridge**, and automated remediation. The exam expects you to know how these services differ, how they integrate, and when to use each one.

A strong design separates three concerns:

- **Inventory and drift detection** with AWS Config
- **Security posture aggregation** with Security Hub
- **Workflow and remediation** with EventBridge, Systems Manager, and Lambda

**AWS Config fundamentals**

AWS Config records configuration changes for supported resources and evaluates them against rules. It helps you answer two questions:

- What resources exist right now?
- Which of those resources violate a policy or best practice?

AWS Config stores configuration history and snapshots, which makes it useful for both compliance and forensics. You can review how a resource changed over time, not just whether it is currently compliant.

Key terms:

- **Configuration item**: A point-in-time record of a resource’s settings
- **Configuration recorder**: The component that captures resource changes
- **Delivery channel**: The path used to deliver configuration snapshots and history to S3 and optionally SNS
- **Config rule**: A policy that evaluates a resource or a set of resources for compliance

**Rule evaluation timing**

AWS Config rules can evaluate in different ways, and this matters for detection latency.

- **Configuration change-triggered** rules evaluate when a relevant resource changes
- **Periodic** rules evaluate on a schedule, such as every 24 hours
- **Hybrid** patterns are common in practice, where you use change-triggered rules for fast detection and periodic rules for broader assurance

Use change-triggered evaluation when you need near-real-time detection of drift, such as security group ingress changes. Use periodic evaluation when the rule depends on state that is not always tied to a single configuration event, or when you want a scheduled compliance sweep.

Example managed rule names you should recognize:

- `INCOMING_SSH_DISABLED` for security groups that allow SSH from the internet
- `S3_BUCKET_PUBLIC_READ_PROHIBITED`
- `S3_BUCKET_PUBLIC_WRITE_PROHIBITED`
- `CLOUD_TRAIL_ENABLED`
- `ROOT_ACCOUNT_MFA_ENABLED`

> AWS Config rule names and identifiers are not interchangeable in every context. Always verify whether a question refers to the rule name shown in the console, the managed rule identifier, or a remediation document parameter.

**Practical AWS Config setup**

A typical account-level setup includes a recorder, delivery channel, and rules.

```bash
aws configservice put-configuration-recorder \
  --configuration-recorder name=default,roleARN=arn:aws:iam::123456789012:role/AWSConfigRole,recordingGroup="{allSupported=true,includeGlobalResourceTypes=true}"

aws configservice put-delivery-channel \
  --delivery-channel name=default,s3BucketName=my-config-bucket

aws configservice start-configuration-recorder \
  --configuration-recorder-name default
```

For exam purposes, remember:

- The recorder needs an IAM role that AWS Config can assume
- The S3 bucket must allow AWS Config to write snapshots and history
- Global resource recording behavior differs by Region and service type
- You often enable Config in every account and Region that matters for security monitoring

**Organization-wide AWS Config**

For multi-account environments, AWS Config supports organization-level deployment patterns. This is important in exam scenarios because security teams usually need centralized visibility.

Common approaches:

- **Organization Config rules**: Deploy the same rule across accounts and Regions in an AWS Organization
- **Organization conformance packs**: Deploy a bundle of rules and remediations as a managed package
- **Aggregators**: Collect compliance data from multiple accounts and Regions into one place

Use organization Config rules when you want a small number of specific controls deployed consistently. Use conformance packs when you want a packaged baseline with multiple related controls and optional remediation. Do not assume conformance packs are the only organization-wide option; they are one deployment model among several.

**Conformance packs**

A conformance pack is a template-driven bundle of AWS Config rules and optional remediation actions. It is useful when you want to deploy a standard, repeatable control set such as a security baseline or industry framework mapping.

Use conformance packs when:

- You need a reusable compliance baseline
- You want to deploy multiple related controls together
- You want a single artifact that can be versioned and rolled out consistently

Use organization Config rules instead when:

- You only need a few specific rules
- You want simpler control-by-control deployment
- You do not need the packaging overhead of a full baseline

Example conformance pack template:

```yaml
Resources:
  S3BucketPublicReadProhibited:
    Type: AWS::Config::ConfigRule
    Properties:
      ConfigRuleName: s3-bucket-public-read-prohibited
      Scope:
        ComplianceResourceTypes:
          - AWS::S3::Bucket
      Source:
        Owner: AWS
        SourceIdentifier: S3_BUCKET_PUBLIC_READ_PROHIBITED

  S3BucketPublicReadRemediation:
    Type: AWS::Config::RemediationConfiguration
    Properties:
      ConfigRuleName: s3-bucket-public-read-prohibited
      TargetType: SSM_DOCUMENT
      TargetId: AWS-DisableS3BucketPublicReadWrite
      Automatic: true
      Parameters:
        AutomationAssumeRole:
          StaticValue:
            Values:
              - arn:aws:iam::123456789012:role/AWSConfigRemediationRole
        BucketName:
          ResourceValue:
            Value: RESOURCE_ID
```

Notes on remediation:

- The remediation document must match the rule and resource type
- Parameters must match the document schema exactly
- Automatic remediation is powerful, but you should test it carefully in nonproduction accounts first
- Some remediations use Systems Manager Automation documents, while others use Lambda or custom workflows

**AWS Config aggregators**

An aggregator centralizes compliance and configuration visibility across accounts and Regions. This is different from Security Hub aggregation, which focuses on findings and security standards.

Use an aggregator when you need:

- Cross-account compliance reporting
- Cross-Region visibility into resource configuration
- A central place to query resource inventory and compliance state

Important distinctions:

- A **configuration aggregator** collects AWS Config data
- It does not replace Security Hub
- It is especially useful for inventory, drift analysis, and compliance queries

Example aggregator creation:

```bash
aws configservice put-configuration-aggregator \
  --configuration-aggregator-name org-aggregator \
  --organization-aggregation-source RoleArn=arn:aws:iam::123456789012:role/AWSConfigAggregatorRole,AllAwsRegions=true
```

With an aggregator, you can query across accounts and Regions using advanced query syntax.

Example advanced query:

```sql
SELECT
  accountId,
  awsRegion,
  resourceType,
  resourceId,
  configuration.complianceType
WHERE
  resourceType = 'AWS::EC2::SecurityGroup'
```

Advanced query is useful when you need to find patterns at scale, such as:

- Security groups with broad ingress
- Publicly accessible storage resources
- Resources missing required tags
- Drift across multiple accounts

**Event-driven detection and remediation**

AWS Config can emit compliance change notifications that you route through EventBridge. This is useful when you want to trigger automation as soon as a resource becomes noncompliant.

A common pattern is:

1. AWS Config evaluates a rule
2. The compliance state changes
3. EventBridge matches the event
4. Lambda, Systems Manager, or Step Functions starts remediation or notification

Example EventBridge rule pattern for AWS Config compliance changes:

```json
{
  "source": ["aws.config"],
  "detail-type": ["Config Rules Compliance Change"],
  "detail": {
    "messageType": ["ComplianceChangeNotification"],
    "configRuleName": ["INCOMING_SSH_DISABLED"],
    "newEvaluationResult": {
      "complianceType": ["NON_COMPLIANT"]
    }
  }
}
```

Use this pattern carefully:

- Match the actual event schema for the AWS Config event type you are using
- Test the pattern in your Region and account
- Do not assume every compliance event uses the same nested fields

Example Lambda target:

```python
import json

def lambda_handler(event, context):
    print(json.dumps(event))
    rule = event["detail"]["configRuleName"]
    compliance = event["detail"]["newEvaluationResult"]["complianceType"]
    if compliance == "NON_COMPLIANT":
        # Trigger ticketing, notification, or remediation
        print(f"{rule} is noncompliant")
```

**Security Hub relationship**

Security Hub is a security posture management service. It aggregates findings from integrated AWS services and partner products, and it evaluates its own security standards controls.

Do not oversimplify the relationship between Config and Security Hub:

- Security Hub does not universally “consume” every AWS Config rule evaluation
- Some Security Hub controls are backed by AWS Config checks
- Other findings come from different services or native Security Hub logic
- Security Hub is best thought of as a findings and posture layer, not a raw configuration recorder

Use Security Hub when you want:

- Centralized security findings
- Standards-based posture assessment
- Cross-account and cross-Region security visibility
- Workflow integration for security operations

Example enabling a standard:

```bash
aws securityhub enable-security-hub

aws securityhub batch-enable-standards \
  --standards-subscription-requests '[
    {
      "StandardsArn": "arn:aws:securityhub:us-east-1::standards/aws-foundational-security-best-practices/v/1.0.0"
    }
  ]'
```

Versioned standard ARNs change over time, so verify the current version for the benchmark and Region you are using.

**Security Hub organization setup**

In multi-account environments, you usually designate a delegated administrator and then enable member accounts.

Typical flow:

1. Register a delegated administrator for Security Hub in the management account
2. Enable Security Hub in the delegated admin account
3. Invite or auto-enable member accounts
4. Enable standards centrally
5. Aggregate findings into the delegated admin account

Example organization enablement concepts:

- Use AWS Organizations integration for centralized management
- Prefer delegated administration over manual per-account setup
- Ensure Regions are enabled consistently where you need coverage

Security Hub also supports **custom actions**, which let you route findings into automation workflows.

Common uses:

- Open a ticket
- Send a finding to a chat or incident system
- Trigger a Step Functions workflow
- Start a Lambda-based enrichment or remediation process

Example custom action flow:

- Analyst selects a finding in Security Hub
- Custom action sends an EventBridge event
- EventBridge targets Lambda or Step Functions
- Workflow enriches, triages, or remediates the issue

**Comparing Config, Security Hub, and Audit Manager**

These services overlap, but they solve different problems.

| Service | Primary purpose | Best for | Not best for |
|---|---|---|---|
| AWS Config | Resource inventory, drift, compliance rules | Detecting configuration noncompliance | Human evidence collection |
| Security Hub | Security findings and standards posture | Centralized security operations | Full configuration history |
| Audit Manager | Evidence collection for audits | Control evidence and audit readiness | Real-time drift detection |

Exam tip:

- Use **AWS Config** for configuration compliance and drift
- Use **Security Hub** for security findings and standards aggregation
- Use **Audit Manager** for evidence gathering and audit workflows

**Common detection scenarios**

Security group allowing SSH from the internet:

- Use AWS Config managed rule `INCOMING_SSH_DISABLED`
- Trigger EventBridge on noncompliance
- Remediate with Systems Manager or a Lambda function that removes the offending ingress rule

S3 bucket made public:

- Use AWS Config managed rules such as `S3_BUCKET_PUBLIC_READ_PROHIBITED`
- Consider S3 Block Public Access as a preventive control
- Use Security Hub to surface the issue in a broader security dashboard

CloudTrail disabled:

- Use AWS Config managed rule `CLOUD_TRAIL_ENABLED`
- Alert immediately because logging gaps reduce forensic visibility
- Consider organization-level guardrails to prevent disabling trails

**Practical remediation patterns**

Remediation should be deterministic and scoped to the resource type.

Common patterns:

- **Systems Manager Automation** for standardized remediation steps
- **Lambda** for custom logic or enrichment
- **Step Functions** for multi-step workflows
- **SNS** or chat integrations for notification-only responses

Example Systems Manager Automation invocation:

```bash
aws ssm start-automation-execution \
  --document-name AWS-DisablePublicAccessForS3Bucket \
  --parameters BucketName=my-bucket
```

Use the exact document name and parameters required by the automation document. Do not assume a generic `GroupId` or `ResourceId` parameter will work for every remediation.

Best practices:

- Test remediation in a sandbox account
- Use least-privilege IAM roles for automation
- Add approval steps for destructive actions
- Log every remediation action for auditability

**Evaluation modes and edge cases**

AWS Config evaluation timing affects what you detect and when.

- A resource can be noncompliant before the next periodic evaluation runs
- A change-triggered rule may miss a condition if the relevant property does not emit the expected configuration event
- Some resources are global or have Region-specific behavior that affects recording and evaluation
- Deleted resources may still appear in history and can matter for investigations

Be careful with these edge cases:

- Not every AWS service is supported by AWS Config in every Region
- Not every control should be automated immediately
- Some controls are better enforced preventively with IAM, SCPs, or service-specific settings

**Exam-focused decision guide**

Choose the service based on the question wording:

- If the question asks about **resource configuration drift**, choose AWS Config
- If it asks about **central security findings**, choose Security Hub
- If it asks about **audit evidence**, choose Audit Manager
- If it asks about **cross-account compliance queries**, choose a Config aggregator
- If it asks about **automatic response to noncompliance**, use EventBridge plus Lambda or Systems Manager

**Quick CLI reference**

```bash
# Start AWS Config recording
aws configservice start-configuration-recorder \
  --configuration-recorder-name default

# Create a Config rule
aws configservice put-config-rule \
  --config-rule '{
    "ConfigRuleName":"incoming-ssh-disabled",
    "Source":{
      "Owner":"AWS",
      "SourceIdentifier":"INCOMING_SSH_DISABLED"
    }
  }'

# Query Security Hub findings
aws securityhub get-findings \
  --filters '{"ComplianceStatus":[{"Value":"FAILED","Comparison":"EQUALS"}]}'
```

**Key takeaways**

- AWS Config detects noncompliant configurations by recording resource state and evaluating rules
- Aggregators and advanced queries help you find issues across accounts and Regions
- Conformance packs package multiple controls and optional remediation into a reusable baseline
- Security Hub centralizes findings and standards posture, but it is not a universal replacement for Config
- EventBridge, Lambda, and Systems Manager turn detection into automated response
