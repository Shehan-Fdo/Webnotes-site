**Posture findings in incident context**

Posture findings are security signals that describe how a resource, account, or workload deviates from an expected security baseline. During an incident, you use these findings to separate *root cause clues* from *background hygiene issues*. The goal is not to fix every weakness immediately; it is to determine which findings are relevant to the active threat, which are pre-existing, and which increase blast radius.

In AWS, posture findings commonly come from services such as **AWS Security Hub**, **Amazon GuardDuty**, **AWS Config**, **Amazon Inspector**, and **IAM Access Analyzer**. Each service answers a different question:

- **Security Hub** aggregates and normalizes findings across services.
- **GuardDuty** detects suspicious behavior, such as credential abuse or anomalous API activity.
- **Config** evaluates resource configuration against rules and conformance packs.
- **Inspector** identifies software vulnerabilities and exposure in compute workloads.
- **Access Analyzer** identifies unintended external access in resource policies.

During an incident, you correlate these signals with logs, identity activity, and network evidence to decide whether the issue is:
- an active compromise,
- a misconfiguration that enabled compromise,
- or a pre-existing control gap unrelated to the current event.

**Why posture matters during an incident**

A posture finding can explain how an attacker gained access, how they escalated privileges, or why containment is incomplete. For example, a finding about an overly permissive security group may not be the incident itself, but it can explain lateral movement. Likewise, a public S3 bucket finding may not indicate malicious activity, but it can show data exposure that changes the severity and response priority.

Use posture findings to answer these questions:
- What control failed?
- Is the finding directly related to the incident timeline?
- Does the finding increase the attackerâ€™s reach or persistence?
- Should you remediate immediately, or preserve evidence first?

> If a finding is part of the suspected attack path, treat it as evidence until you confirm otherwise. Avoid making changes that destroy logs, snapshots, or configuration state needed for forensics.

**Common AWS posture sources and what they mean**

| Source | What it tells you | Incident value |
|---|---|---|
| Security Hub | Centralized security posture and aggregated findings | Fast triage and prioritization |
| GuardDuty | Suspicious activity and threat detection | Detects active abuse patterns |
| Config | Resource configuration drift and compliance | Shows control failures and timing |
| Inspector | Vulnerabilities in EC2, ECR, and Lambda workloads | Identifies exploitability |
| Access Analyzer | External access in policies | Reveals unintended exposure |
| CloudTrail | API activity and identity actions | Confirms who did what and when |

**Security Hub as the posture aggregator**

Security Hub is often the first place you look when an incident is reported because it consolidates findings from multiple AWS services and partner integrations. It uses a normalized finding format, which makes it easier to compare severity and status across sources.

Key Security Hub concepts:
- **Finding**: A security issue or observation.
- **Severity**: The impact level assigned to the finding.
- **Workflow status**: Tracks whether the finding is new, in progress, or resolved.
- **Product name**: Identifies the originating service.
- **Resource**: The affected AWS entity, such as an EC2 instance, IAM role, or S3 bucket.

A practical triage flow is:
1. Review high-severity findings first.
2. Check whether the finding timestamp aligns with the incident start time.
3. Identify the affected resource and its dependencies.
4. Determine whether the finding is a symptom, cause, or unrelated background issue.
5. Escalate findings that indicate exposure, privilege escalation, or persistence.

Example Security Hub finding query with the AWS CLI:

```bash
aws securityhub get-findings \
  --filters '{"SeverityLabel":[{"Value":"HIGH","Comparison":"EQUALS"}],"WorkflowStatus":[{"Value":"NEW","Comparison":"EQUALS"}]}' \
  --max-results 10
```

You can also update workflow status as you triage:

```bash
aws securityhub batch-update-findings \
  --finding-identifiers '[{"Id":"finding-id-example","ProductArn":"arn:aws:securityhub:us-east-1::product/aws/securityhub"}]' \
  --workflow '{"Status":"IN_PROGRESS"}'
```

**GuardDuty findings during active compromise**

GuardDuty is especially useful during incidents because it detects behaviors that often indicate credential theft, reconnaissance, or malware activity. It does not prove compromise by itself, but it can provide strong leads.

Common GuardDuty finding categories include:
- **Credential compromise indicators** such as unusual API calls or access from atypical geographies.
- **Reconnaissance** such as port scanning or enumeration.
- **Persistence or backdoor behavior** such as suspicious DNS activity or unusual instance communication.
- **Data exfiltration indicators** such as access to known malicious IPs or unusual S3 access patterns.

When a GuardDuty finding appears during an incident:
- Compare the finding time with CloudTrail events.
- Check whether the principal is an IAM user, assumed role, or federated identity.
- Validate whether the source IP, user agent, and region are expected.
- Look for follow-on actions such as privilege escalation, key creation, or policy changes.

Example CLI to inspect a GuardDuty finding:

```bash
aws guardduty list-findings --detector-id <detector-id>

aws guardduty get-findings \
  --detector-id <detector-id> \
  --finding-ids <finding-id-1> <finding-id-2>
```

**AWS Config findings and configuration drift**

AWS Config is critical when you need to determine whether a resource was already misconfigured before the incident or changed during the attack. Config records configuration history and evaluates resources against rules.

Use Config to:
- confirm whether a security group, bucket policy, or IAM policy changed,
- identify the exact time of a configuration drift,
- compare current state to a known-good baseline,
- support evidence preservation with historical snapshots.

Examples of incident-relevant Config findings:
- An S3 bucket became public before the incident.
- A security group opened port `22` or `3389` to `0.0.0.0/0`.
- An IAM policy gained `iam:PassRole` or wildcard permissions.
- A CloudTrail trail was disabled or altered.

Example Config queries:

```bash
aws configservice get-resource-config-history \
  --resource-type AWS::EC2::SecurityGroup \
  --resource-id sg-0123456789abcdef0
```

```bash
aws configservice describe-compliance-by-resource \
  --resource-type AWS::S3::Bucket
```

Config is especially valuable because it helps distinguish:
- **pre-existing exposure** from
- **attacker-induced change**.

That distinction affects both containment and reporting. If the weakness existed before the incident, you may need to treat it as a control failure and a broader risk issue, not just an isolated compromise.

**Amazon Inspector and exploitability**

Inspector helps you understand whether a workload had known vulnerabilities that could have been exploited. In incident response, vulnerability findings matter most when they align with the affected asset and the attack timeline.

Use Inspector to assess:
- whether the compromised instance had a known CVE,
- whether the vulnerable package was reachable,
- whether the workload was internet-facing,
- whether the vulnerability could explain initial access or privilege escalation.

Practical interpretation:
- A critical vulnerability on an isolated internal host may be less relevant than a medium vulnerability on an internet-facing service.
- A vulnerability finding becomes more important if you see matching exploit behavior in logs or GuardDuty alerts.
- Patch status alone does not prove exploitation; it only changes likelihood.

**IAM Access Analyzer and unintended access**

Access Analyzer identifies resource policies that grant access outside your intended trust boundary. During an incident, this is useful for spotting exposure in:
- S3 bucket policies,
- KMS key policies,
- IAM roles,
- Lambda permissions,
- SQS and SNS policies,
- Secrets Manager resource policies.

If a resource is unexpectedly shared with another account or the public, that can explain unauthorized access or data exposure. It can also reveal a path for persistence if an attacker modified a policy to retain access.

Example use cases:
- A bucket policy allows `Principal: "*"`.
- A KMS key policy permits an external account.
- A Lambda function resource policy allows invocation from an untrusted principal.

**CloudTrail as the timeline anchor**

Posture findings are most useful when you can place them on a timeline. CloudTrail is the primary source for API activity and identity actions, so it helps you determine whether a posture issue existed before the incident or was introduced during it.

Look for:
- `AuthorizeSecurityGroupIngress`
- `PutBucketPolicy`
- `PutRolePolicy`
- `AttachRolePolicy`
- `CreateAccessKey`
- `UpdateTrail`
- `StopLogging`
- `DeleteTrail`
- `PutBucketAcl`
- `PutPublicAccessBlock`

Example CloudTrail lookup with the AWS CLI:

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=PutBucketPolicy \
  --max-results 10
```

When you correlate CloudTrail with posture findings, you can answer:
- Who changed the control?
- Was the change intentional?
- Did the change happen before the suspicious activity?
- Did the attacker use the change to expand access?

**Incident triage workflow using posture findings**

A practical incident workflow is to combine findings, logs, and resource state in a controlled sequence.

1. **Preserve evidence**
   - Snapshot affected volumes if needed.
   - Export relevant logs and findings.
   - Avoid deleting resources until you capture state.

2. **Identify the highest-risk findings**
   - Focus on public exposure, credential compromise, privilege escalation, and persistence.
   - Prioritize findings tied to the affected account or workload.

3. **Correlate with the incident timeline**
   - Match finding timestamps with CloudTrail, VPC Flow Logs, and application logs.
   - Determine whether the finding predates the incident.

4. **Classify the finding**
   - Root cause
   - Enabler
   - Symptom
   - Unrelated hygiene issue

5. **Contain carefully**
   - Revoke compromised credentials.
   - Restrict network access.
   - Disable risky trust relationships.
   - Preserve evidence before destructive remediation when possible.

6. **Remediate and validate**
   - Fix the control gap.
   - Re-run posture checks.
   - Confirm the resource returns to compliant state.

**Containment decisions driven by posture findings**

Not every finding should trigger the same response. The right action depends on whether the finding is actively exploitable and whether remediation will destroy evidence.

Examples:
- If a security group exposes SSH to the internet and the instance is under investigation, you may first snapshot the instance and then restrict ingress.
- If a bucket policy is public and data exposure is ongoing, you may immediately block public access while preserving policy copies and access logs.
- If an IAM role has excessive permissions, you may detach the role from the workload or rotate credentials before editing the policy.

> Do not assume that â€śfixingâ€ť the finding is always the first step. In an incident, evidence preservation and timeline integrity often matter more than immediate compliance.

**Practical AWS CLI examples**

List recent Security Hub findings by severity:

```bash
aws securityhub get-findings \
  --filters '{"SeverityLabel":[{"Value":"CRITICAL","Comparison":"EQUALS"}]}' \
  --max-results 20
```

Check Config history for a bucket:

```bash
aws configservice get-resource-config-history \
  --resource-type AWS::S3::Bucket \
  --resource-id my-sensitive-bucket
```

Inspect IAM policy attachments for a role:

```bash
aws iam list-attached-role-policies --role-name IncidentRole
aws iam list-role-policies --role-name IncidentRole
```

Review recent access key creation:

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=CreateAccessKey
```

Check for public access settings on S3:

```bash
aws s3api get-public-access-block --bucket my-sensitive-bucket
aws s3api get-bucket-policy --bucket my-sensitive-bucket
```

**Configuration patterns that matter in incidents**

A few configuration patterns appear repeatedly in AWS incidents:

- **Overly permissive IAM**
  - Wildcards in actions or resources increase blast radius.
  - Privilege escalation often follows policy attachment or role assumption changes.

- **Public network exposure**
  - Security groups, NACLs, load balancers, and public IPs can expose services.
  - Exposure is more severe when paired with vulnerable software.

- **Weak resource policies**
  - S3, KMS, SNS, SQS, and Lambda policies can grant unintended access.
  - Cross-account access should be explicit and justified.

- **Logging gaps**
  - Disabled CloudTrail, missing VPC Flow Logs, or short retention can hide attacker activity.
  - Logging gaps are often themselves a finding and a response priority.

- **Unpatched workloads**
  - Vulnerabilities matter most when the workload is reachable and the exploit path is plausible.

**How to interpret severity in context**

Severity is useful, but it is not absolute. A medium finding can be more important than a high finding if it is directly tied to the incident path. For example:
- A medium misconfiguration on the compromised workload may be the actual entry point.
- A high-severity vulnerability on an unused asset may be irrelevant to the current event.
- A low-severity policy issue may become critical if it enables persistence or exfiltration.

Use severity together with:
- asset criticality,
- exposure level,
- exploitability,
- timeline alignment,
- and business impact.

**Exam focus: what you should conclude from posture findings**

For the exam, posture findings during an incident usually point to one of these conclusions:
- The environment had a pre-existing control gap.
- The attacker changed the posture to expand access or hide activity.
- The finding is a symptom of compromise rather than the cause.
- The finding is unrelated and should not distract from the active investigation.

The best response is usually to:
- correlate findings with logs,
- preserve evidence,
- prioritize findings that affect containment,
- and remediate in a way that does not erase forensic data.

**Quick reference: incident questions and likely AWS evidence**

| Question | Best evidence |
|---|---|
| Was the resource public before the incident? | AWS Config history, Access Analyzer |
| Did someone change the policy during the attack? | CloudTrail, Config |
| Is the workload vulnerable to known exploits? | Amazon Inspector |
| Is there active suspicious behavior? | GuardDuty, CloudTrail |
| What is the current security posture? | Security Hub, Config |
| Did the attacker disable logging? | CloudTrail, Config |

**Key takeaways**

- Posture findings help you understand *how* an incident became possible, not just *what* happened.
- Use Security Hub for aggregation, GuardDuty for suspicious behavior, Config for drift, Inspector for vulnerability context, and Access Analyzer for unintended exposure.
- Always correlate findings with CloudTrail and other logs before changing resources.
- Treat findings as evidence when they may be part of the attack path.
- Prioritize containment actions that preserve forensic integrity.
