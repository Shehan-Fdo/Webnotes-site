## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Evidence extraction overview**

Evidence extraction in AWS usually means collecting, preserving, and analyzing logs and configuration data in a way that supports incident response, audit, and investigation. For exam purposes, focus on two common approaches:

- **AWS CLI** for direct retrieval of service state, log objects, and metadata.
- **Amazon Athena** for SQL-based analysis of logs stored in Amazon S3.

Use the CLI when you need precise control, automation, or point-in-time collection. Use Athena when you need to search large log sets, correlate events, and summarize activity quickly.

A good forensic workflow preserves the original evidence, records timestamps in UTC, and avoids modifying source objects. When you export results, treat them as derived evidence and keep the original query, output, and hash values.

**What to collect first**

Start with the highest-value sources that support attribution and timeline reconstruction:

- **AWS CloudTrail** management events for API activity.
- **CloudTrail data events** for S3 object-level access, Lambda invocation, and other supported services when enabled.
- **AWS Config** snapshots and configuration history for resource state.
- **VPC Flow Logs** for network-level context.
- **Amazon CloudWatch Logs** for application or service logs.
- **IAM** and STS metadata for identity context.

For each artifact, capture the source, time range, region, account, and retrieval method. That metadata matters as much as the content itself.

**AWS CLI for evidence collection**

The AWS CLI is useful for fast, scriptable collection. It can pull configuration, list resources, export logs, and automate repeatable evidence capture.

Common commands include:

```bash
aws sts get-caller-identity
aws cloudtrail describe-trails
aws cloudtrail get-trail-status --name my-trail
aws configservice describe-configuration-recorders
aws ec2 describe-instances --region us-east-1
aws s3 ls s3://my-forensic-bucket/
```

Use `aws sts get-caller-identity` early to confirm the active principal, account, and ARN. That helps you avoid collecting evidence from the wrong account or role session.

**CLI patterns that matter in investigations**

- **Scope by region** because many AWS APIs are regional.
- **Use `--profile`** to separate identities and reduce mistakes.
- **Use `--query`** with JMESPath to extract only relevant fields.
- **Use `--output json`** for machine-readable evidence.
- **Redirect output to files** and hash them after collection.

Example:

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=ConsoleLogin \
  --start-time 2026-04-25T00:00:00Z \
  --end-time 2026-04-26T00:00:00Z \
  --output json > console-login-events.json

sha256sum console-login-events.json > console-login-events.json.sha256
```

`lookup-events` is useful for quick searches, but it has important limits:

- It is primarily for **management events**.
- It has a limited time window and is not a full historical archive.
- It is not the best tool for deep, long-range analysis across large datasets.
- Regional behavior matters; you may need to query the region where the event occurred or where the trail is configured.

For exam prep, remember that CloudTrail lookup is convenient but not a substitute for a queryable log lake or Athena analysis.

**CloudTrail fields you should know**

When you inspect CloudTrail records, these fields often support attribution and timeline analysis:

- **eventTime**: When AWS recorded the event.
- **eventName**: The API action, such as `RunInstances` or `PutObject`.
- **eventSource**: The AWS service namespace, such as `ec2.amazonaws.com`.
- **awsRegion**: The region where the event occurred.
- **sourceIPAddress**: The apparent source of the request.
- **userAgent**: The client or SDK used.
- **userIdentity**: The principal type and identity details.
- **recipientAccountId**: The account that received the request.
- **requestID**: Useful for correlating service-side activity.
- **eventID**: Unique identifier for the event record.
- **sessionContext** and **sessionIssuer**: Helpful for role attribution.
- **additionalEventData**: Often contains context such as MFA usage or console-specific details.

Interpret these fields together. For example, `sourceIPAddress` alone does not prove a human actor, and `userAgent` alone does not prove malicious automation. Correlation is the key.

**CLI examples for common evidence tasks**

List IAM users and attached policies:

```bash
aws iam list-users
aws iam list-attached-user-policies --user-name Alice
aws iam list-user-policies --user-name Alice
```

Inspect role trust relationships:

```bash
aws iam get-role --role-name IncidentResponseRole
aws iam get-role-policy --role-name IncidentResponseRole --policy-name InlinePolicyName
```

Review S3 bucket configuration:

```bash
aws s3api get-bucket-versioning --bucket my-sensitive-bucket
aws s3api get-bucket-encryption --bucket my-sensitive-bucket
aws s3api get-bucket-policy --bucket my-sensitive-bucket
aws s3api get-bucket-logging --bucket my-sensitive-bucket
```

Check EC2 instance metadata from the control plane:

```bash
aws ec2 describe-instances \
  --filters "Name=tag:Environment,Values=Prod" \
  --query "Reservations[].Instances[].{Id:InstanceId,State:State.Name,AZ:Placement.AvailabilityZone,PrivateIP:PrivateIpAddress}" \
  --output table
```

These commands are often enough to establish what existed, when it existed, and which identity could have changed it.

**Preserving evidence from the CLI**

When you export evidence, preserve the original output and the context around it.

- Save raw JSON whenever possible.
- Record the exact command used.
- Capture the timestamp and timezone of collection.
- Hash the file after export with `sha256sum` or an equivalent tool.
- Store originals in immutable or access-controlled storage.

Example workflow:

```bash
date -u +"%Y-%m-%dT%H:%M:%SZ"
aws ec2 describe-security-groups --output json > security-groups-2026-04-26.json
sha256sum security-groups-2026-04-26.json > security-groups-2026-04-26.json.sha256
```

> Do not edit raw evidence files in place. If you need to annotate them, create a separate working copy and keep the original intact.

**Athena for log analysis**

Amazon Athena lets you query data in Amazon S3 using SQL. It is especially useful for CloudTrail, VPC Flow Logs, ALB logs, and other structured or semi-structured datasets.

Athena is a strong fit when you need to:

- Search large volumes of logs quickly.
- Correlate events across time.
- Filter by identity, source IP, region, or API action.
- Produce summaries for incident reports.

Athena does not ingest logs by itself. You must have the data in S3 and a table definition in the AWS Glue Data Catalog or an equivalent metadata setup.

**Typical Athena setup for CloudTrail**

A practical workflow usually looks like this:

1. Store CloudTrail logs in an S3 bucket.
2. Create a Glue Data Catalog table manually or with a crawler.
3. Point the table to the correct S3 prefix.
4. Grant Athena and the querying principal permission to read the bucket and write query results.
5. Run SQL queries against the table.

Example table creation pattern:

```sql
CREATE EXTERNAL TABLE IF NOT EXISTS cloudtrail_logs (
  eventversion string,
  useridentity struct<
    type:string,
    principalid:string,
    arn:string,
    accountid:string,
    invokedby:string,
    accesskeyid:string,
    username:string,
    sessioncontext:struct<
      attributes:struct<
        mfaauthenticated:string,
        creationdate:string>,
      sessionissuer:struct<
        type:string,
        principalid:string,
        arn:string,
        accountid:string,
        username:string>>>,
  eventtime string,
  eventsource string,
  eventname string,
  awsregion string,
  sourceipaddress string,
  useragent string,
  errorcode string,
  errormessage string,
  requestparameters string,
  responseelements string,
  additionaleventdata string,
  requestid string,
  eventid string,
  readonly string,
  resources array<struct<
    arn:string,
    accountid:string,
    type:string>>,
  eventtype string,
  recipientaccountid string
)
ROW FORMAT SERDE 'org.openx.data.jsonserde.JsonSerDe'
LOCATION 's3://my-cloudtrail-bucket/AWSLogs/123456789012/CloudTrail/'
```

This is a representative pattern, not a universal schema. In practice, table definitions vary by log format, crawler output, and CloudTrail delivery structure.

**Athena permissions and setup considerations**

To query logs successfully, you need:

- Read access to the S3 log bucket.
- Permission to write Athena query results to an output bucket.
- Glue Data Catalog permissions if you manage tables there.
- KMS permissions if the logs or query results are encrypted with AWS KMS.

A common failure mode is having S3 read access but no permission to write query results. Another is pointing the table at the wrong prefix and getting empty results.

**Querying CloudTrail with Athena**

Use Athena to search for specific events, identities, or time windows.

Console login example:

```sql
SELECT eventtime,
       awsregion,
       sourceipaddress,
       useridentity.arn,
       eventname,
       errorcode,
       errormessage
FROM cloudtrail_logs
WHERE eventsource = 'signin.amazonaws.com'
  AND eventname = 'ConsoleLogin'
  AND eventtime BETWEEN '2026-04-25T00:00:00Z' AND '2026-04-26T00:00:00Z'
ORDER BY eventtime DESC;
```

To check MFA context, include the relevant field if it exists in your schema:

```sql
SELECT eventtime,
       useridentity.arn,
       json_extract_scalar(additionaleventdata, '$.MFAUsed') AS mfa_used,
       sourceipaddress,
       useragent
FROM cloudtrail_logs
WHERE eventsource = 'signin.amazonaws.com'
  AND eventname = 'ConsoleLogin';
```

The exact field path can vary by schema and log format, so verify the table structure before relying on a query.

**Finding API activity by event name**

For STS `AssumeRole`, filter on the API call itself:

```sql
SELECT eventtime,
       useridentity.type,
       useridentity.arn,
       eventsource,
       eventname,
       sourceipaddress,
       requestid,
       eventid
FROM cloudtrail_logs
WHERE eventsource = 'sts.amazonaws.com'
  AND eventname = 'AssumeRole'
ORDER BY eventtime DESC;
```

This is important: the `AssumeRole` API call is identified by `eventname = 'AssumeRole'`. Later events made with the temporary credentials may show `userIdentity.type = 'AssumedRole'`, but that is not the same thing as the original `AssumeRole` request.

**S3 object access analysis**

If S3 data events are enabled, Athena can help you identify object-level access such as `GetObject`, `PutObject`, and `DeleteObject`.

Example:

```sql
SELECT eventtime,
       eventname,
       useridentity.arn,
       sourceipaddress,
       requestparameters,
       resources[1].arn AS resource_arn
FROM cloudtrail_logs
WHERE eventsource = 's3.amazonaws.com'
  AND eventname IN ('GetObject', 'PutObject', 'DeleteObject')
  AND eventtime >= '2026-04-25T00:00:00Z'
ORDER BY eventtime DESC;
```

Remember that S3 data events must be enabled for the bucket or prefix you care about. If they were not enabled, Athena cannot query events that were never logged.

**Useful investigation patterns in Athena**

- **Timeline reconstruction**: Sort by `eventtime` and group by `eventid` or `requestid`.
- **Attribution**: Use `useridentity.arn`, `sessionissuer`, and `sourceipaddress`.
- **Scope**: Filter by `recipientaccountid`, `awsregion`, and resource ARN.
- **Suspicious automation**: Compare `useragent` values and burst patterns.
- **Privilege escalation**: Search for IAM, STS, and policy modification events.

Example privilege-related search:

```sql
SELECT eventtime,
       useridentity.arn,
       eventname,
       requestparameters,
       sourceipaddress
FROM cloudtrail_logs
WHERE eventsource IN ('iam.amazonaws.com', 'sts.amazonaws.com')
  AND eventname IN ('CreateUser', 'AttachUserPolicy', 'PutUserPolicy', 'AssumeRole', 'CreateAccessKey')
ORDER BY eventtime DESC;
```

**Partitioning and performance**

Athena performance depends heavily on how you organize data.

- Use **date-based prefixes** in S3 for CloudTrail delivery.
- Define partitions or use partition projection so Athena does not scan unnecessary data.
- Filter on partition columns such as year, month, and day when available.
- Avoid full-table scans on large log sets unless necessary.

Example partition projection idea:

```sql
ALTER TABLE cloudtrail_logs SET TBLPROPERTIES (
  'projection.enabled'='true',
  'projection.year.type'='integer',
  'projection.year.range'='2024,2026',
  'projection.month.type'='integer',
  'projection.month.range'='1,12',
  'projection.day.type'='integer',
  'projection.day.range'='1,31',
  'storage.location.template'='s3://my-cloudtrail-bucket/AWSLogs/123456789012/CloudTrail/${region}/${year}/${month}/${day}/'
);
```

Use this carefully and only when the S3 layout matches the template. Partition projection reduces manual partition management, but it does not fix a bad table definition.

**Correlating CloudTrail with other evidence**

Athena is especially powerful when you join multiple datasets. For example, you can correlate CloudTrail with VPC Flow Logs or application logs to connect API activity with network behavior.

Conceptual example:

```sql
SELECT c.eventtime,
       c.useridentity.arn,
       c.sourceipaddress,
       v.srcaddr,
       v.dstaddr,
       v.dstport,
       v.action
FROM cloudtrail_logs c
JOIN vpc_flow_logs v
  ON c.sourceipaddress = v.srcaddr
WHERE c.eventname = 'ConsoleLogin'
  AND c.eventtime BETWEEN '2026-04-25T00:00:00Z' AND '2026-04-26T00:00:00Z';
```

This kind of join is useful for investigation, but be careful with time alignment and field normalization. IP addresses, timestamps, and account boundaries may not line up perfectly.

**CloudTrail Lake versus Athena**

CloudTrail Lake is a managed, queryable event data store for CloudTrail events. Athena is a general SQL engine over data in S3.

Use this distinction:

- **CloudTrail Lake**: Best when you want native CloudTrail querying, retention management, and event data store features.
- **Athena**: Best when you already have logs in S3 or need to query multiple log sources together.

For exam prep, know that both can support investigation, but they are not interchangeable. CloudTrail Lake is purpose-built for CloudTrail; Athena is broader and depends on your table definitions.

**Lookup events limitations**

CloudTrail `lookup-events` is helpful, but it has constraints you should remember:

- It focuses on **management events**.
- It is not ideal for long-term historical analysis.
- It has limited filtering compared with SQL.
- It may not surface data events unless they are available in the queried trail context.
- Regional scope matters, so you may need to query the correct region.

If you need deeper analysis, move to Athena or CloudTrail Lake.

**Chain of custody and integrity**

Forensics is not just about finding data; it is about preserving trust in the data.

- Keep original logs in immutable storage when possible.
- Restrict write access to evidence buckets.
- Record hashes for exported files.
- Preserve original timestamps and note timezone conversions.
- Document the exact query, command, and account context used to collect evidence.

A simple evidence manifest can help:

```text
Artifact: console-login-events.json
Source: CloudTrail lookup-events
Account: 123456789012
Region: us-east-1
Collected: 2026-04-26T10:15:00Z
Hash: sha256:...
Operator: incident-response-role
```

**Exam-focused takeaways**

- Use the **AWS CLI** for targeted collection and automation.
- Use **Athena** for scalable SQL analysis over logs in S3.
- Know the difference between **management events** and **data events**.
- Remember that `AssumeRole` is identified by `eventname`, while later temporary sessions may appear as `AssumedRole`.
- Verify whether S3 data events were enabled before expecting object-level evidence.
- Treat CloudTrail lookup as a quick search tool, not a full forensic platform.
- Preserve evidence integrity with hashes, raw exports, and documented collection steps.
- Consider **CloudTrail Lake** when the question asks for a native queryable CloudTrail store.

**Quick reference**

- `aws sts get-caller-identity` confirms who you are.
- `aws cloudtrail lookup-events` finds recent management events.
- Athena queries S3-based logs with SQL.
- CloudTrail fields like `eventTime`, `eventName`, `userIdentity`, `sourceIPAddress`, and `requestID` support attribution.
- Partitioning and projection reduce Athena scan cost and improve speed.
