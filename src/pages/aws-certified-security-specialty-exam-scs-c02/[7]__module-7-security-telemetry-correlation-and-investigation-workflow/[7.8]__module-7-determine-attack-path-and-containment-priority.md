## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Attack path analysis in AWS**

Attack path analysis is the process of tracing how an adversary could move from an initial foothold to higher-value assets. In AWS, that usually means following identity permissions, network reachability, trust relationships, and data access paths across accounts, Regions, and services. Your goal is not only to find what was touched, but to determine what could be reached next.

A practical attack path view combines:
- **Identity exposure** such as overly permissive IAM policies, role trust policies, access keys, and session permissions.
- **Network exposure** such as public subnets, security groups, NACLs, VPC peering, Transit Gateway routes, and exposed load balancers.
- **Data exposure** such as S3 bucket policies, KMS key permissions, Secrets Manager access, and database credentials.
- **Privilege escalation paths** such as `iam:PassRole`, `sts:AssumeRole`, Lambda execution roles, EC2 instance profiles, and service-linked role abuse.

A useful mental model is: initial access plus reachable permissions plus reachable data equals likely attack path.

**What to look for first**

Start with the most likely pivot points. In AWS incidents, these often include:
- Compromised IAM users or access keys.
- EC2 instances with instance profiles and metadata access.
- Lambda functions with broad execution roles.
- S3 buckets with public or cross-account write access.
- Security groups that allow lateral movement inside a VPC.
- Roles that can be assumed by external principals or by services that an attacker can influence.

When you evaluate a path, ask:
- Can the attacker authenticate as a principal?
- Can that principal assume another role?
- Can that role read secrets, decrypt data, or modify infrastructure?
- Can the attacker reach another network segment or service endpoint?
- Can the attacker persist by creating new credentials, backdoors, or trust relationships?

**Identity-based attack paths**

Identity is often the shortest route to impact in AWS. A single compromised principal can become a much larger incident if it can assume roles, create access keys, or modify policies.

Common identity pivots include:
- `sts:AssumeRole` into privileged roles.
- `iam:PassRole` to launch compute with elevated permissions.
- `iam:CreateAccessKey` or `iam:UpdateAccessKey` for persistence.
- `iam:AttachUserPolicy`, `iam:PutUserPolicy`, `iam:PutRolePolicy`, or `iam:AttachRolePolicy` to expand privileges.
- `organizations:DescribeOrganization` and related read access that helps map the environment.
- `kms:Decrypt` on keys protecting secrets or data stores.

Example of a risky trust policy pattern:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "*"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

That trust policy is overly broad because it allows any AWS principal to attempt assumption if identity policy conditions permit. In practice, you should look for trust policies that include external accounts, wildcard principals, or service principals that are too permissive.

**Network-based attack paths**

Network reachability determines where a compromised workload can move next. In AWS, lateral movement often happens through private IP connectivity rather than direct internet exposure.

Key network elements to inspect:
- **Security groups** for inbound and outbound rules that allow broad internal access.
- **Network ACLs** for subnet-level filtering, especially if they are permissive.
- **Route tables** for paths to peered VPCs, Transit Gateway attachments, VPNs, or Direct Connect.
- **VPC endpoints** that expose private access to AWS services and can be abused if endpoint policies are weak.
- **Load balancers** and reverse proxies that may expose internal services indirectly.
- **DNS** records that reveal internal service names or shared endpoints.

A common pattern is:
1. Compromise an EC2 instance in a public subnet.
2. Use its security group egress to reach internal services.
3. Query instance metadata for temporary credentials.
4. Assume a role or access a secret.
5. Pivot into another account or environment.

Example security group inspection:
```bash
aws ec2 describe-security-groups \
  --group-ids sg-0123456789abcdef0 \
  --query 'SecurityGroups[0].IpPermissions'
```

Example route table inspection:
```bash
aws ec2 describe-route-tables \
  --filters Name=vpc-id,Values=vpc-0123456789abcdef0
```

**Data access paths**

Containment priority often depends on what data the attacker can reach, not just what systems they can touch. Data access paths include direct service permissions and indirect access through compute roles or application secrets.

High-value targets include:
- **S3** buckets containing logs, backups, exports, or customer data.
- **KMS** keys that protect databases, EBS volumes, S3 objects, or secrets.
- **Secrets Manager** and **SSM Parameter Store** values used by applications.
- **RDS** snapshots and database credentials.
- **ECR** repositories containing application images and embedded secrets.
- **CloudTrail** and security logs, which attackers may try to disable or tamper with.

If a principal can decrypt with KMS, it may be able to read multiple downstream data stores even if those stores are otherwise protected. If a principal can read secrets, it may gain access to databases, third-party APIs, or other cloud accounts.

**Containment priority framework**

Containment priority is the order in which you reduce risk during an incident. You usually prioritize actions that stop active abuse, prevent escalation, and protect the most sensitive assets while minimizing business disruption.

A practical priority order is:
1. Stop active credential use and session abuse.
2. Block privilege escalation and persistence.
3. Cut off lateral movement paths.
4. Protect sensitive data and encryption keys.
5. Preserve evidence and maintain visibility.

This order is not absolute. If a production outage would be worse than the incident impact, you may need to isolate more surgically. The key is to contain the attacker’s options faster than they can expand access.

**How to prioritize containment**

Use these questions to rank actions:
- Is the attacker still active?
- Is the compromised principal privileged?
- Can the attacker create new credentials or roles?
- Can the attacker decrypt data or access secrets?
- Can the attacker move into another account or Region?
- Will the containment action break critical workloads?

A high-priority containment action is one that removes many attack paths at once. For example, disabling a compromised access key may be more effective than changing a single security group rule if the attacker is operating through IAM rather than network access.

**Common containment actions**

Typical AWS containment steps include:
- Disable or delete compromised IAM access keys.
- Force password reset and revoke active sessions where applicable.
- Detach dangerous policies or remove inline policy statements.
- Tighten trust policies on roles that can be assumed externally.
- Remove `iam:PassRole` from compromised principals.
- Restrict security group ingress and egress.
- Quarantine instances by moving them to isolated security groups or subnets.
- Disable or rotate secrets in Secrets Manager and Parameter Store.
- Schedule KMS key rotation only when appropriate; for active compromise, focus on access control changes first.
- Block suspicious cross-account access in AWS Organizations SCPs (Service Control Policies).
- Suspend or restrict federation sources if identity provider compromise is suspected.

Example: disable an access key
```bash
aws iam update-access-key \
  --user-name compromised-user \
  --access-key-id AKIAEXAMPLE123456789 \
  --status Inactive
```

Example: detach a policy from a role
```bash
aws iam detach-role-policy \
  --role-name AppRole \
  --policy-arn arn:aws:iam::aws:policy/AdministratorAccess
```

Example: quarantine an EC2 instance with a restrictive security group
```bash
aws ec2 modify-instance-attribute \
  --instance-id i-0123456789abcdef0 \
  --groups sg-quarantine123456789
```

**Evidence-driven triage**

You should base containment on evidence from logs and service metadata. Useful sources include:
- **CloudTrail** for API activity, role assumptions, policy changes, and key management events.
- **VPC Flow Logs** for network connections and unusual east-west traffic.
- **GuardDuty** for findings such as credential exfiltration, anomalous API calls, and unusual data access.
- **AWS Config** for resource drift and policy changes.
- **Security Hub** for aggregated findings and prioritization.
- **CloudWatch Logs** for application and system activity.
- **ALB, CloudFront, and S3 access logs** for request patterns and source IPs.

CloudTrail queries often help identify the first malicious action and the next likely pivot. For example, if you see `AssumeRole` followed by `GetSecretValue`, the next containment step may be to revoke the role’s access to secrets and inspect any workloads using that role.

Example CloudTrail lookup:
```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=AssumeRole \
  --max-results 10
```

**Attack path examples**

A few common scenarios illustrate how to think about path and priority:

- **Compromised IAM user**
  - The user has `sts:AssumeRole` into a deployment role.
  - The deployment role has `iam:PassRole` and can launch EC2.
  - The EC2 instance profile can read Secrets Manager.
  - Priority: disable the user key, remove role assumption, then restrict secrets access.

- **Compromised EC2 instance**
  - The instance can reach internal databases through security group rules.
  - The instance profile can call `kms:Decrypt` and `secretsmanager:GetSecretValue`.
  - Priority: quarantine the instance, rotate secrets, and review the instance profile permissions.

- **Cross-account role abuse**
  - A role trust policy allows an external account.
  - The role has access to S3 and KMS in the target account.
  - Priority: tighten trust policy, revoke external access, and review all resources accessible by that role.

- **Public S3 exposure**
  - A bucket policy allows public read or cross-account write.
  - The bucket contains logs or backups with credentials.
  - Priority: block public access, inspect object history, and rotate any exposed secrets.

**Useful CLI and policy checks**

You can quickly inspect the most important paths with AWS CLI commands.

Check who can assume a role:
```bash
aws iam get-role --role-name ExampleRole
```

Inspect attached policies:
```bash
aws iam list-attached-role-policies --role-name ExampleRole
aws iam list-role-policies --role-name ExampleRole
```

Review a user’s access keys:
```bash
aws iam list-access-keys --user-name compromised-user
```

Check S3 public access settings:
```bash
aws s3api get-public-access-block --bucket example-bucket
aws s3api get-bucket-policy-status --bucket example-bucket
```

Inspect KMS key policy:
```bash
aws kms get-key-policy \
  --key-id arn:aws:kms:us-east-1:123456789012:key/abcd-1234 \
  --policy-name default
```

Check active sessions and federation context indirectly through CloudTrail and identity provider logs, since AWS does not provide a simple universal “list all sessions” command for every identity type.

**Containment tradeoffs**

Containment can disrupt production, so you need to balance speed with precision. For example:
- Disabling a role may stop an attacker but also stop an application.
- Rotating a secret may break dependent services if you do not coordinate updates.
- Blocking all outbound traffic may prevent exfiltration but also stop patching, logging, or health checks.
- Detaching a policy may remove attacker access but also remove legitimate automation.

When possible, prefer:
- **Targeted revocation** over broad shutdown.
- **Temporary isolation** over permanent deletion.
- **Policy tightening** over service-wide disablement.
- **Account-level guardrails** over one-off manual fixes.

**Practical containment workflow**

1. Identify the initial compromised principal or resource.
2. Map immediate privileges, trust relationships, and reachable data.
3. Determine whether the attacker can escalate, persist, or pivot.
4. Contain the highest-risk path first.
5. Preserve logs and snapshots before destructive changes when feasible.
6. Validate that containment did not leave an alternate path open.
7. Rotate or reissue credentials and secrets only after access paths are closed.

**Exam focus**

For the exam, expect questions that ask you to choose the best next step after detecting suspicious activity. The correct answer usually reflects:
- The fastest way to stop further compromise.
- The action that removes the broadest attack path.
- The least disruptive effective containment.
- The control that matches the observed vector, such as IAM, network, or data access.

If the scenario mentions role assumption, think IAM trust and permissions first. If it mentions lateral movement, think security groups, routes, and instance profiles. If it mentions data theft, think secrets, KMS, and S3 access. If it mentions persistence, think access keys, policy changes, and federation.

**Quick reference: path to priority mapping**

| Observed signal | Likely attack path | First containment move |
|---|---|---|
| Compromised access key | IAM privilege escalation | Disable key, review policies |
| Suspicious `AssumeRole` | Cross-role pivot | Tighten trust policy, revoke access |
| EC2 unusual outbound traffic | Lateral movement or exfiltration | Quarantine instance, inspect SGs |
| Secret access spike | Data access path | Rotate secret, restrict role permissions |
| S3 public access finding | Data exposure | Block public access, review bucket policy |
| KMS decrypt anomalies | Broad data decryption path | Restrict key policy and grants |

**Key takeaway**

Attack path analysis tells you where an attacker can go next; containment priority tells you what to stop first. In AWS, the highest-value response usually combines identity revocation, trust-policy tightening, network isolation, and secret rotation, ordered by the attacker’s ability to escalate and the sensitivity of the reachable assets.
