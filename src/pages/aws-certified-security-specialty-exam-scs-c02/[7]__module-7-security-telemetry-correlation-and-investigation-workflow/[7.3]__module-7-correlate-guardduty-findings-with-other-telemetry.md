## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**GuardDuty correlation mindset**

Amazon GuardDuty is a managed threat detection service that produces findings, not full investigations. Your job is to correlate each finding with other telemetry to confirm scope, understand attacker behavior, and decide on containment. In exam scenarios, the best answer usually combines GuardDuty with the right source of truth: AWS CloudTrail for API activity, VPC Flow Logs for network paths, Route 53 Resolver query logs for DNS behavior, and centralized security tooling for triage and response.

GuardDuty findings are most useful when you treat them as pivots. A finding gives you an entity, time window, severity, and evidence. You then use those details to search adjacent logs and determine whether the activity is expected, malicious, or part of a broader campaign.

**What GuardDuty findings contain**

A finding typically includes fields that help you pivot into other telemetry:

- **Finding ID**: stable identifier for case management and SIEM correlation.
- **Finding type**: tells you the detection family, such as credential abuse, reconnaissance, or exfiltration.
- **Severity**: helps prioritize response.
- **Account and Region**: critical in multi-account and multi-Region environments.
- **Resource details**: the affected instance, IAM principal, S3 bucket, EKS cluster, RDS database, or other resource.
- **Service action**: the action GuardDuty observed or inferred, such as network connection, API call, or DNS activity.
- **Evidence**: supporting context, often including source IP, destination IP, ports, user agent, or suspicious domain.

When you investigate, start with the finding type and the resource context. Then use the time window and entity identifiers to search CloudTrail, VPC Flow Logs, DNS logs, Security Hub, or your SIEM.

**Telemetry sources and what each one proves**

GuardDuty does not replace raw telemetry. Each source answers a different question.

| Telemetry source | Best for | Key limitation |
|---|---|---|
| CloudTrail management events | Who called AWS APIs, from where, and with what identity | Does not show data-plane payloads |
| CloudTrail data events | S3 object access, Lambda invoke, and other supported data-plane activity | Must be enabled and can be expensive at scale |
| VPC Flow Logs | Network source/destination, ports, accept/reject, traffic patterns | No packet payloads, no application content |
| Route 53 Resolver query logs | DNS queries from VPC workloads | Only covers DNS observed through Resolver logging placement |
| GuardDuty findings | Detection and enrichment | Not a complete event record |
| Security Hub | Aggregation and normalization of findings | Still requires pivoting to source logs |
| SIEM / log analytics | Cross-account, cross-service correlation and alerting | Depends on ingestion quality and parsing |

> GuardDuty findings are indicators, not proof by themselves. Use the underlying telemetry to validate the behavior before you escalate or automate containment.

**Common GuardDuty finding families and how to correlate them**

GuardDuty uses finding types that map to different telemetry patterns. The exact names can vary by service and feature set, but the investigation logic stays consistent.

**Credential and IAM anomaly findings**

These findings often involve unusual API usage, suspicious geolocation, or compromised credentials.

- Correlate with **CloudTrail management events** to identify the exact API calls.
- Check the `userIdentity` block for the principal type:
  - IAM user
  - Assumed role
  - Federated user
  - Root user
- Review `sourceIPAddress`, `userAgent`, `eventName`, and `awsRegion`.
- Look for privilege escalation patterns such as `CreateAccessKey`, `AttachUserPolicy`, `PutUserPolicy`, `AssumeRole`, or `CreateLoginProfile`.

Practical CloudTrail lookup example:

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=ConsoleLogin \
  --start-time 2026-04-26T09:00:00Z \
  --end-time 2026-04-26T11:00:00Z
```

If you know the source IP from the finding, filter by that in your SIEM or CloudTrail Lake query. For console sign-in investigations, `ConsoleLogin` plus the source IP and identity context is usually more reliable than searching only by username.

**Instance and container compromise findings**

These findings often indicate suspicious outbound connections, command-and-control behavior, or malware activity on EC2, EKS, or container workloads.

- Use **VPC Flow Logs** to confirm network connections:
  - destination IP
  - destination port
  - accept/reject status
  - bytes and packets
- Use **CloudTrail** to see whether the instance role or pod identity made AWS API calls after compromise.
- For EKS-related findings, correlate with:
  - EKS audit logs if enabled
  - CloudTrail for IAM and EKS control plane actions
  - Kubernetes logs from your cluster logging pipeline

Example VPC Flow Logs query pattern in Athena:

```sql
SELECT
  srcaddr,
  dstaddr,
  dstport,
  action,
  SUM(bytes) AS total_bytes,
  COUNT(*) AS flow_count
FROM vpc_flow_logs
WHERE from_unixtime(start) BETWEEN timestamp '2026-04-26 10:00:00' AND timestamp '2026-04-26 11:00:00'
  AND srcaddr = '10.0.1.25'
GROUP BY srcaddr, dstaddr, dstport, action
ORDER BY total_bytes DESC;
```

This kind of query helps you confirm whether the instance contacted suspicious infrastructure, but it cannot show the payload or the command that triggered the traffic.

**DNS-based findings**

GuardDuty can detect suspicious domain lookups, domain generation algorithm behavior, or communication with known malicious domains.

- Correlate with **Route 53 Resolver query logs** to confirm:
  - queried domain
  - query time
  - source VPC or resolver context
  - repeated lookups or NXDOMAIN patterns
- Use VPC Flow Logs to see whether the DNS resolution was followed by outbound traffic to the resolved IP.
- If the finding references a domain, search for the same domain across your DNS telemetry and proxy logs if available.

Example Resolver query log search concept:

```sql
fields @timestamp, srcaddr, query_name, rcode
| filter query_name = "example-malicious-domain.com"
| sort @timestamp desc
| limit 20
```

Use the exact telemetry source you actually collect. GuardDuty’s DNS correlation is strongest when Route 53 Resolver query logs are enabled for the relevant VPCs.

**S3 protection findings**

GuardDuty S3 protection findings often indicate unusual object access, suspicious API usage, or exfiltration patterns involving Amazon S3.

- Correlate with **CloudTrail data events** for S3:
  - `GetObject`
  - `PutObject`
  - `ListBucket`
  - `DeleteObject`
- Check the principal, source IP, and user agent.
- Review bucket policy, access points, and object ownership settings.
- Look for large-volume reads, unusual geographies, or access from unexpected roles.

Example CloudTrail Lake-style investigation pattern:

```sql
SELECT
  eventTime,
  eventName,
  userIdentity.arn,
  sourceIPAddress,
  requestParameters.bucketName,
  requestParameters.key
FROM <your_event_data_store>
WHERE eventSource = 's3.amazonaws.com'
  AND eventName IN ('GetObject', 'ListBucket')
  AND requestParameters.bucketName = 'sensitive-bucket'
ORDER BY eventTime DESC;
```

CloudTrail Lake queries run against your configured event data store, so the table or view name depends on your setup. The important exam concept is that S3 protection findings are validated with S3 data events, not only management events.

**RDS findings**

GuardDuty can flag suspicious activity involving Amazon RDS, such as unusual login behavior or anomalous access patterns.

- Correlate with:
  - RDS logs if enabled
  - CloudTrail for control-plane changes
  - VPC Flow Logs for source/destination network behavior
- Check whether the source is an application subnet, bastion host, or unexpected external address.
- Validate whether the database port and connection pattern match the application baseline.

RDS investigations often hinge on network context because the database payload is not visible in flow logs.

**EKS findings**

For Amazon EKS, GuardDuty can surface suspicious Kubernetes or container activity.

- Correlate with:
  - CloudTrail for EKS API actions and IAM role assumptions
  - Kubernetes audit logs for cluster-level actions
  - Container runtime or node logs if available
- Check whether the finding references:
  - a pod
  - a node
  - a service account
  - an IAM role for service accounts (IRSA)

EKS findings are especially useful when you need to determine whether the activity came from a workload identity, a node role, or an external principal.

**How to pivot from a finding**

A practical investigation flow is:

1. Read the finding type, severity, account, Region, and resource details.
2. Identify the likely telemetry source:
   - API activity → CloudTrail
   - network activity → VPC Flow Logs
   - DNS behavior → Route 53 Resolver query logs
   - S3 access → CloudTrail data events
   - Kubernetes activity → EKS audit logs and CloudTrail
3. Narrow the time window around the finding.
4. Search for the same principal, source IP, destination IP, domain, bucket, or resource ID.
5. Confirm whether the activity is expected.
6. Escalate or contain if the behavior is inconsistent with the workload baseline.

**Example correlation workflow for a suspicious EC2 instance**

Suppose GuardDuty reports suspicious outbound traffic from an EC2 instance.

1. Open the finding and note:
   - instance ID
   - source IP
   - destination IP
   - time
   - finding type
2. Search VPC Flow Logs for the instance IP and time window.
3. Search CloudTrail for API calls made by the instance profile role.
4. Check Route 53 Resolver query logs for suspicious domain lookups near the same time.
5. If the instance accessed S3 or STS, inspect CloudTrail data events and role assumption activity.
6. Use the combined evidence to decide whether the instance is compromised.

**Example correlation workflow for a suspicious console login**

If GuardDuty flags anomalous console access:

1. Use the finding’s time, source IP, and identity details.
2. Search CloudTrail for `ConsoleLogin` events in the same window.
3. Review `userIdentity`, `sourceIPAddress`, and `additionalEventData`.
4. Check whether the login was followed by privilege changes such as policy attachment or access key creation.
5. Correlate with Security Hub or your SIEM to see whether the same identity triggered other alerts.

Example CloudTrail lookup for console login activity:

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=ConsoleLogin \
  --start-time 2026-04-26T09:00:00Z \
  --end-time 2026-04-26T11:00:00Z
```

If you already know the source IP from the finding, use your log analytics tool or CloudTrail Lake to filter by that IP and the relevant identity. That is often more precise than searching by username alone.

**Multi-account and multi-Region correlation**

In AWS Organizations, GuardDuty can be enabled centrally and findings can span many accounts and Regions. Correlation becomes easier when you centralize logs.

- Use an organization trail or centralized CloudTrail Lake event data store.
- Aggregate VPC Flow Logs and Route 53 Resolver query logs into a central account or SIEM.
- Preserve account ID, Region, and resource ARN in every log pipeline.
- Use the GuardDuty finding ID as a case-management key across systems.

A common topology is:

- member accounts generate telemetry
- centralized logging account stores CloudTrail, flow logs, and DNS logs
- Security Hub aggregates findings
- SIEM or SOAR consumes Security Hub or EventBridge events

This design helps you answer cross-account questions such as whether the same source IP touched multiple workloads or whether a compromised role was reused in another Region.

**EventBridge, Security Hub, and automation**

GuardDuty integrates well with AWS-native response workflows.

- **Amazon EventBridge** can route GuardDuty findings in near real time.
- **AWS Lambda** can enrich findings, open tickets, or trigger containment.
- **AWS Security Hub** can aggregate GuardDuty with other security services.
- **AWS Systems Manager Automation** can isolate instances or apply response runbooks.

Example EventBridge pattern for GuardDuty findings:

```json
{
  "source": ["aws.guardduty"],
  "detail-type": ["GuardDuty Finding"],
  "detail": {
    "severity": [{ "numeric": [">=", 7] }]
  }
}
```

A common response pattern is to send high-severity findings to Lambda for enrichment, then forward the enriched event to a SIEM or case system.

**Practical limitations to remember**

- **CloudTrail management events** do not show object-level S3 reads or writes unless data events are enabled.
- **CloudTrail data events** can be high volume and may not be enabled everywhere.
- **VPC Flow Logs** do not show packet payloads, so you cannot see commands or file contents.
- **Route 53 Resolver query logs** only help if the relevant DNS traffic is actually logged.
- **GuardDuty** may detect behavior using AWS internal signals that are not directly visible in your logs, so you still need corroboration.

> Do not assume a missing log means no activity. It may mean the telemetry source was not enabled, not retained, or not centralized.

**Investigation fields to use in SIEM or CloudTrail Lake**

When you pivot from GuardDuty into a broader investigation, these fields are especially useful:

- `findingId`
- `type`
- `severity`
- `accountId`
- `region`
- `resource.instanceDetails.instanceId`
- `resource.s3BucketDetails.name`
- `resource.accessKeyDetails.accessKeyId`
- `service.action`
- `service.evidence`
- `service.additionalInfo`
- `service.archived`
- `service.count`
- `service.eventFirstSeen`
- `service.eventLastSeen`

These fields help you group repeated alerts, identify the first and last observed activity, and connect a finding to the underlying AWS resource or identity.

**Exam decision cues**

When the question asks how to validate or investigate a GuardDuty finding, choose the telemetry source that matches the behavior:

- API misuse or credential abuse → **CloudTrail**
- suspicious network connections → **VPC Flow Logs**
- suspicious domain lookups → **Route 53 Resolver query logs**
- S3 object access → **CloudTrail data events**
- cluster or pod activity → **EKS audit logs** plus CloudTrail
- centralized triage and escalation → **Security Hub** and **EventBridge**

If the scenario mentions multiple accounts, prefer centralized logging and organization-wide aggregation. If it mentions containment, think of automated response through EventBridge, Lambda, or Systems Manager.

**Quick reference**

- GuardDuty tells you *what looked suspicious*.
- CloudTrail tells you *who did what in AWS*.
- VPC Flow Logs tell you *what network traffic occurred*.
- Route 53 Resolver query logs tell you *what DNS names were queried*.
- Security Hub and SIEM tools tell you *how to correlate and operationalize the alert*.
