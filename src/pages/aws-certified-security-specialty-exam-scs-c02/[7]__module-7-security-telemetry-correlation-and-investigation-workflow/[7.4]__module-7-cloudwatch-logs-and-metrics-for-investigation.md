## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**CloudWatch for investigation: the core idea**

Amazon CloudWatch gives you two complementary evidence streams:

- **Logs** capture event detail, request context, and error messages.
- **Metrics** capture time-series signals that help you spot trends, anomalies, and thresholds.

For security investigations, you usually start with metrics to identify *when* something changed, then use logs to determine *what* happened and *who* did it. CloudWatch is not a full SIEM, but it is often the first place you look for operational and security evidence inside AWS.

A practical investigation workflow often looks like this:

1. Detect an unusual metric spike or alarm.
2. Narrow the time window.
3. Query logs for the affected service, account, region, or resource.
4. Correlate with CloudTrail, VPC Flow Logs, or service-specific logs.
5. Preserve evidence with export or archival workflows.

**CloudWatch Logs fundamentals**

CloudWatch Logs stores log events in **log groups** and **log streams**.

- A **log group** is the top-level container, often aligned to an application, service, or environment.
- A **log stream** is an ordered sequence of events from a single source, such as one Lambda execution environment or one EC2 instance agent.
- A **log event** is a timestamped record with a message payload.

Common sources include:

- AWS Lambda function logs
- Amazon EC2 logs via the CloudWatch Agent
- API Gateway execution/access logs
- Application Load Balancer logs when delivered elsewhere, with CloudWatch used for related metrics
- AWS service logs that are forwarded or integrated into CloudWatch

CloudWatch Logs is useful because it is searchable, centralized, and close to the workload. It is less useful when you need very long retention, very large-scale analytics, or immutable evidence storage without additional controls.

> CloudWatch Logs is near real time, but not perfectly instantaneous. In exam scenarios, remember that ingestion latency can delay visibility by seconds or more, so it may not be the best source for sub-second incident confirmation.

**Log ingestion, latency, and retention**

CloudWatch Logs is designed for near-real-time operational visibility, but there are important limits:

- Log delivery can lag behind the source.
- High-volume systems may show delayed ingestion or partial visibility during bursts.
- Retention is configurable per log group, from short operational windows to long-term retention.
- Retention expiration deletes data from CloudWatch Logs unless you archive it elsewhere.

For investigations, retention matters because:

- Short retention can erase evidence before an incident is fully understood.
- Long retention increases cost but improves forensic reach.
- You may keep hot logs in CloudWatch and archive older logs to Amazon S3 for cheaper long-term storage.

**CloudWatch Logs Insights query syntax and investigation patterns**

CloudWatch Logs Insights is the primary interactive query tool for log investigation. It supports structured filtering, parsing, aggregation, sorting, and limiting results.

A typical query pipeline uses commands such as:

- `fields` to select or compute fields
- `filter` to narrow events
- `parse` to extract values from unstructured text
- `stats` to aggregate and summarize
- `sort` to order results
- `limit` to cap output size

Example query for error investigation:

```sql
fields @timestamp, @message, @logStream
| filter @message like /ERROR|Exception/
| sort @timestamp desc
| limit 50
```

Example query for counting failures by source:

```sql
fields @timestamp, sourceIPAddress, eventName, errorCode
| filter errorCode != ""
| stats count(*) as failures by sourceIPAddress, eventName
| sort failures desc
| limit 20
```

Example query using `parse` on a text log:

```sql
fields @timestamp, @message
| parse @message "* user=* action=* status=*" as prefix, user, action, status
| filter status = "DENIED"
| stats count(*) by user, action
| sort count(*) desc
```

Example query using time bucketing:

```sql
fields @timestamp, @message
| filter @message like /timeout|throttle|5\d\d/
| stats count(*) as errors by bin(5m)
| sort bin(5m) asc
```

Common functions and patterns you should recognize:

- `count(*)` for event totals
- `min()`, `max()`, `avg()` for numeric summaries
- `sum()` for totals
- `bin()` for time grouping
- `strcontains()` for string matching in some query patterns
- `ispresent()` to test whether a field exists
- `coalesce()` to choose the first non-null value
- `tolower()` and `toupper()` for normalization
- `jsonParse()` when working with JSON-formatted logs

Example with JSON logs:

```sql
fields @timestamp, @message, requestId, userIdentity.arn, eventName
| filter eventSource = "iam.amazonaws.com"
| stats count(*) by eventName, userIdentity.arn
| sort count(*) desc
```

When you investigate, prefer structured fields over raw message text whenever possible. If your application emits JSON, CloudWatch Logs Insights can query nested fields directly, which is much more reliable than regex-heavy parsing.

**CloudWatch Logs Insights limitations and scope**

Logs Insights is powerful, but exam questions often test its boundaries:

- Queries run against log groups you select; it is not a universal cross-account search engine by default.
- Cross-account and cross-region investigation usually requires centralized logging design, such as forwarding logs into a dedicated logging account and region.
- Query performance depends on log volume, time range, and how well the data is structured.
- It is best for interactive analysis, not for permanent storage or compliance archiving.

In practice, centralized logging often uses:

- Subscription filters to stream logs to Kinesis Data Firehose, Kinesis Data Streams, Lambda, or another destination
- Cross-account log delivery into a security or logging account
- S3 archival for long-term retention and offline analysis

**CloudWatch Logs access control and data protection**

Investigation access should be tightly scoped. CloudWatch Logs supports IAM-based permissions and resource-level controls.

Important IAM actions include:

- `logs:DescribeLogGroups`
- `logs:DescribeLogStreams`
- `logs:GetLogEvents`
- `logs:FilterLogEvents`
- `logs:StartQuery`
- `logs:GetQueryResults`
- `logs:StopQuery`
- `logs:PutSubscriptionFilter`
- `logs:PutRetentionPolicy`
- `logs:CreateLogGroup`
- `logs:CreateLogStream`

For read-only investigation, you usually grant only the minimum needed actions. For example:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "InvestigateLogs",
      "Effect": "Allow",
      "Action": [
        "logs:DescribeLogGroups",
        "logs:DescribeLogStreams",
        "logs:FilterLogEvents",
        "logs:StartQuery",
        "logs:GetQueryResults"
      ],
      "Resource": "*"
    }
  ]
}
```

For tighter control, use resource ARNs where supported and restrict access to specific log groups. In security investigations, this matters because logs may contain sensitive data such as tokens, email addresses, IPs, or personal data.

CloudWatch Logs also supports **data protection** features that can detect and mask sensitive data patterns in log events. This is useful when you need to reduce exposure of secrets or regulated data while still allowing investigation. Masking helps analysts see that sensitive content exists without exposing the raw value.

Also consider **log group resource policies** when sharing logs across accounts or allowing AWS services to write to a log group. Resource policies are especially relevant for centralized logging and cross-account delivery patterns.

> Do not assume every investigator can read every log group. In exam scenarios, the correct answer is often to combine least-privilege IAM, resource policies, and retention controls rather than broad administrative access.

**Exporting logs to S3 and archival workflows**

CloudWatch Logs can export log data to Amazon S3 for long-term retention, offline analysis, and forensic preservation.

Use export tasks when you need:

- Bulk extraction of historical logs
- Long-term archival in S3
- Evidence preservation outside CloudWatch retention windows
- Offline processing with Athena, Glue, or third-party tools

Use subscription filters when you need:

- Near-real-time streaming to another service
- Ongoing forwarding to a SIEM, analytics pipeline, or security lake
- Continuous delivery rather than one-time export

A common pattern is:

- Keep recent logs in CloudWatch for fast search.
- Stream or export logs to S3 for durable storage.
- Query archived logs later with Athena if needed.

Example export workflow with the CLI:

```bash
aws logs create-export-task \
  --task-name export-incident-window \
  --log-group-name /aws/lambda/payment-api \
  --from 1714000000000 \
  --to 1714086400000 \
  --destination incident-archive-bucket \
  --destination-prefix cloudwatch-exports/payment-api
```

Remember that exports are not the same as live subscriptions. Exports are point-in-time jobs, while subscriptions are continuous pipelines.

**CloudWatch metrics fundamentals**

CloudWatch metrics are time-series data points used for monitoring, alerting, and trend analysis.

Key concepts:

- A **metric namespace** groups related metrics, such as `AWS/EC2`, `AWS/Lambda`, or `AWS/ApplicationELB`.
- A **metric name** identifies the signal, such as `CPUUtilization` or `Errors`.
- **Dimensions** are name-value pairs that scope a metric to a specific resource or logical slice, such as `InstanceId`, `FunctionName`, or `LoadBalancer`.
- A **data point** is a single measurement at a timestamp.

Dimensions are critical in investigations because they determine what you are actually measuring. For example:

- `AWS/EC2` `CPUUtilization` with `InstanceId=i-123` tells you about one instance.
- The same metric without the right dimension context can hide the affected resource.
- Alarm scope depends on the exact namespace, metric name, and dimensions.

Example CLI to inspect a metric:

```bash
aws cloudwatch get-metric-statistics \
  --namespace AWS/Lambda \
  --metric-name Errors \
  --dimensions Name=FunctionName,Value=checkout-handler \
  --start-time 2026-04-26T00:00:00Z \
  --end-time 2026-04-26T01:00:00Z \
  --period 300 \
  --statistics Sum
```

**Built-in AWS service metrics for investigation**

Built-in service metrics are often the fastest way to confirm whether a problem is service-side, resource-side, or application-side.

Common examples:

- **EC2**
  - `CPUUtilization`
  - `NetworkIn` and `NetworkOut`
  - `StatusCheckFailed`
  - `StatusCheckFailed_Instance`
  - `StatusCheckFailed_System`
- **Lambda**
  - `Invocations`
  - `Errors`
  - `Throttles`
  - `Duration`
  - `ConcurrentExecutions`
  - `IteratorAge`
- **API Gateway**
  - `Count`
  - `4XXError`
  - `5XXError`
  - `Latency`
  - `IntegrationLatency`
- **Application Load Balancer**
  - `RequestCount`
  - `HTTPCode_ELB_5XX_Count`
  - `HTTPCode_Target_5XX_Count`
  - `TargetResponseTime`
  - `HealthyHostCount`
  - `UnHealthyHostCount`

How to use them in investigations:

- A spike in `Lambda Errors` suggests application failures or downstream dependency issues.
- Rising `Throttles` suggests concurrency limits or burst pressure.
- `StatusCheckFailed` on EC2 suggests host or network impairment.
- `5XXError` on API Gateway or ALB suggests backend failures, misconfiguration, or overload.
- `HealthyHostCount` dropping on ALB suggests target health problems that may explain user-visible outages.

Example investigation sequence for a Lambda issue:

1. Check `Invocations`, `Errors`, and `Throttles`.
2. Correlate the time of the spike with log errors in CloudWatch Logs.
3. Inspect downstream service metrics if the function depends on RDS, DynamoDB, or external APIs.
4. Confirm whether the issue is code, capacity, or dependency related.

**Metric namespaces, dimensions, and alarm scoping**

Namespaces and dimensions determine the scope of both dashboards and alarms.

- A namespace separates metric families and prevents naming collisions.
- A dimension narrows the metric to a specific resource or logical grouping.
- An alarm without the correct dimension may monitor the wrong resource or aggregate too broadly.

Examples:

- `AWS/EC2` + `InstanceId` scopes to one instance.
- `AWS/Lambda` + `FunctionName` scopes to one function.
- `AWS/ApplicationELB` + `LoadBalancer` scopes to one load balancer.
- Custom metrics often use dimensions like `Environment`, `Service`, or `Tenant`.

This matters in security investigations because you want the alarm to reflect the exact asset under review. If you monitor a fleet-wide metric when you intended to monitor one instance, you may miss a targeted compromise or misread the blast radius.

**CloudWatch alarms and incident detection**

Alarms evaluate metrics against thresholds and can trigger notifications or automation.

Important alarm settings:

- **Threshold**: the value that triggers state changes.
- **Period**: the length of each evaluation interval.
- **Evaluation periods**: how many periods are considered.
- **Datapoints to alarm**: how many breaching datapoints are required within the evaluation window.
- **Missing data treatment**: how to handle gaps.

Missing data treatment options typically include:

- `breaching`
- `notBreaching`
- `ignore`
- `missing`

These settings affect incident detection significantly:

- A short period with `datapoints to alarm = 1` detects fast spikes but can be noisy.
- Requiring multiple breaching datapoints reduces false positives but may delay detection.
- Treating missing data as breaching can surface telemetry loss, but it may also create false alarms if the source is legitimately quiet.
- Treating missing data as not breaching can hide outages where the metric stops arriving.

Example alarm logic:

- Trigger if `Errors >= 5` for `3 out of 5` one-minute periods.
- This catches sustained failure while ignoring one-off blips.

Example CLI alarm creation:

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name lambda-errors-high \
  --namespace AWS/Lambda \
  --metric-name Errors \
  --dimensions Name=FunctionName,Value=checkout-handler \
  --statistic Sum \
  --period 60 \
  --evaluation-periods 5 \
  --datapoints-to-alarm 3 \
  --threshold 5 \
  --comparison-operator GreaterThanOrEqualToThreshold \
  --treat-missing-data missing \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:security-alerts
```

**Composite alarms for noise reduction**

Composite alarms combine multiple underlying alarms using Boolean logic. They are useful when you want to reduce alert noise and focus on meaningful incidents.

Use composite alarms when:

- One metric alone is too noisy.
- You want to alert only when multiple signals align.
- You need to suppress secondary alarms during a primary incident.

Examples:

- Alert only when `ALB 5XX` and `TargetResponseTime` are both high.
- Alert when `Lambda Errors` and `Throttles` are both active.
- Alert when an EC2 status check alarm and a network error alarm both indicate impact.

Composite alarms are especially useful in security operations because they reduce alert fatigue and help distinguish real incidents from isolated metric anomalies.

**CloudWatch Logs versus EventBridge terminology**

Older AWS documentation and legacy discussions may refer to **CloudWatch Events**, but the current service name is **Amazon EventBridge**.

Key distinction:

- **CloudWatch Logs** stores and queries log data.
- **CloudWatch metrics and alarms** monitor time-series signals.
- **EventBridge** routes events between AWS services and applications.

In exam questions, do not confuse event routing with log storage. If the question is about reacting to an AWS event, EventBridge is usually the right service. If the question is about searching historical request details or error messages, CloudWatch Logs is the right service.

**Practical investigation examples**

**1. EC2 instance investigation**

If an instance becomes slow or unreachable:

- Check `StatusCheckFailed`.
- Review `CPUUtilization`, `NetworkIn`, and `NetworkOut`.
- Inspect system logs and application logs in CloudWatch Logs if the agent is installed.
- Correlate with Auto Scaling activity, EBS metrics, or security group changes.

Example:

```bash
aws cloudwatch get-metric-statistics \
  --namespace AWS/EC2 \
  --metric-name StatusCheckFailed \
  --dimensions Name=InstanceId,Value=i-0123456789abcdef0 \
  --start-time 2026-04-26T00:00:00Z \
  --end-time 2026-04-26T02:00:00Z \
  --period 300 \
  --statistics Maximum
```

**2. Lambda investigation**

If a function starts failing:

- Check `Errors`, `Throttles`, and `Duration`.
- Query logs for stack traces, timeout messages, or downstream API failures.
- Look for concurrency pressure or permission errors.

Example Logs Insights query:

```sql
fields @timestamp, @message
| filter @message like /Task timed out|AccessDenied|Exception/
| sort @timestamp desc
| limit 100
```

**3. API Gateway investigation**

If clients report 5xx or latency:

- Check `5XXError`, `Latency`, and `IntegrationLatency`.
- Use access logs to identify request paths, status codes, and caller patterns.
- Determine whether the issue is in API Gateway itself or the backend integration.

Example query for access logs:

```sql
fields @timestamp, requestId, status, path, integrationStatus, latency
| filter status >= 500
| sort @timestamp desc
| limit 50
```

**4. Application Load Balancer investigation**

If users see intermittent failures:

- Check `HTTPCode_ELB_5XX_Count` and `HTTPCode_Target_5XX_Count`.
- Review `HealthyHostCount` and `TargetResponseTime`.
- Correlate with target group health checks and backend logs.

**5. Security event investigation**

If you suspect unauthorized access or abuse:

- Search logs for denied actions, unusual source IPs, or unexpected user agents.
- Correlate with CloudTrail for API activity.
- Use metrics to identify spikes in errors, throttles, or traffic volume.

Example query for suspicious access patterns:

```sql
fields @timestamp, sourceIPAddress, userAgent, eventName, errorCode
| filter errorCode = "AccessDenied" or errorCode = "UnauthorizedOperation"
| stats count(*) as denied by sourceIPAddress, userAgent, eventName
| sort denied desc
| limit 20
```

**Centralized logging patterns**

In mature environments, you often centralize logs into a dedicated security account.

Common patterns:

- Application accounts write logs to local CloudWatch Logs groups.
- Subscription filters forward logs to a central stream or processing pipeline.
- A security account stores copies in S3 or a log analytics platform.
- Analysts query the central copy rather than each workload account individually.

This design improves:

- Separation of duties
- Retention consistency
- Cross-account visibility
- Incident response speed

It also helps with cross-account investigation because CloudWatch Logs Insights is not typically used as a single global search across all accounts unless you have explicitly designed for that.

**When to use Logs, Metrics, or both**

| Need | Best fit | Why |
|---|---|---|
| Detect a threshold breach | Metrics | Fast, low-noise time-series signal |
| Find the cause of an error | Logs | Contains request and exception detail |
| Track a trend over time | Metrics | Easy aggregation and alerting |
| Investigate a specific request | Logs | Can search by request ID or timestamp |
| Preserve evidence long term | S3 export | Cheaper archival and offline analysis |
| Reduce alert noise | Composite alarms | Combines multiple conditions |

**Investigation tips for the exam**

- Use **metrics** to identify the time window and affected resource.
- Use **logs** to identify the root cause and actor.
- Remember that **dimensions** determine metric scope.
- Remember that **datapoints to alarm** and **missing data treatment** change how alarms behave.
- Prefer **subscription filters** for continuous forwarding and **export tasks** for one-time archival.
- Use **least privilege** for log access and protect sensitive data with masking and resource policies.
- Distinguish **EventBridge** from CloudWatch Logs when the question is about event routing rather than log analysis.
- Account for **ingestion latency** when comparing CloudWatch evidence to real-time user reports.

**Quick CLI reference**

```bash
# List log groups
aws logs describe-log-groups

# Search logs with Insights
aws logs start-query \
  --log-group-name '/aws/lambda/checkout-handler' \
  --start-time 1714000000 \
  --end-time 1714003600 \
  --query-string 'fields @timestamp, @message | filter @message like /ERROR/ | sort @timestamp desc | limit 20'

# Fetch query results
aws logs get-query-results --query-id <query-id>

# Export logs to S3
aws logs create-export-task \
  --task-name archive-window \
  --log-group-name '/aws/lambda/checkout-handler' \
  --from 1714000000000 \
  --to 1714086400000 \
  --destination my-archive-bucket \
  --destination-prefix cwlogs/checkout-handler
```

**Key takeaways**

CloudWatch Logs and Metrics work best together. Metrics tell you that something is wrong and help you scope the blast radius. Logs tell you what happened, who did it, and what the application or service reported at the time. For security investigations, the exam expects you to understand query syntax, metric dimensions, alarm behavior, access controls, data protection, and archival patterns well enough to choose the right evidence source for the scenario.
