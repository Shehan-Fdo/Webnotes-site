## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**CloudTrail fundamentals**

AWS CloudTrail records AWS API activity and related account events so you can answer who did what, from where, and when. For exam scenarios, treat CloudTrail as the primary source for control-plane visibility and a key evidence source during incident response.

CloudTrail event analysis usually starts with three questions:

- What action occurred?
- Which identity or service performed it?
- What resource, Region, and outcome were involved?

CloudTrail is strongest for:
- **Management events** such as creating, modifying, or deleting AWS resources.
- **Data events** such as object-level access in Amazon S3 or invocation activity in AWS Lambda.
- **Network activity events** for supported services where network-layer actions are logged separately from management and data events.

> CloudTrail does not replace service-specific logs. You often correlate it with VPC Flow Logs, GuardDuty findings, IAM Access Analyzer, S3 server access logs, or application logs to build a complete timeline.

**Event types and when to use them**

CloudTrail groups activity into event categories that matter for both detection and investigation.

- **Management events**
  - Capture control-plane operations like `CreateUser`, `AttachRolePolicy`, `StopLogging`, or `PutBucketPolicy`.
  - Are enabled by default for trails in most setups.
  - Are the first place to look for privilege changes, logging tampering, and infrastructure modifications.

- **Data events**
  - Capture high-volume resource access such as S3 object reads/writes and Lambda function invocations.
  - Are not enabled by default because they can generate significant volume and cost.
  - Are essential when you need object-level or function-level evidence.

- **Network activity events**
  - Capture network-related actions for supported services where AWS exposes them as a distinct event type.
  - Are useful when you need visibility into access patterns that are not purely management-plane or object-level data-plane actions.
  - Are not universal across all services, so always verify service support before relying on them in a detection design.

CloudTrail event records also include an `eventType` field. Common values include:
- `AwsApiCall`
- `AwsConsoleSignIn`
- `AwsServiceEvent`
- `AwsConsoleAction`
- `AwsVpceEvent`

Use the event type as a clue, not as the whole story. For example, a console-initiated API call is often still an API call event, while a sign-in event is specifically a console sign-in event.

**Core event fields you should know**

These fields are frequently tested because they help you interpret the event quickly and correctly.

| Field | Meaning | Why it matters |
|---|---|---|
| `eventTime` | When AWS recorded the event | Builds the incident timeline |
| `eventName` | API action name | Tells you what happened |
| `eventSource` | AWS service that received the request | Identifies the service boundary |
| `eventID` | Unique identifier for the event | Useful for deduplication and correlation |
| `eventVersion` | CloudTrail event schema version | Helps when parsing records |
| `awsRegion` | Region where the request was made or processed | Important for multi-Region investigations |
| `sourceIPAddress` | Source IP address seen by AWS | Useful for geolocation and anomaly checks |
| `userAgent` | Client or SDK string | Helps distinguish console, CLI, SDK, or automation |
| `userIdentity` | Identity that made the request | Central to attribution |
| `requestParameters` | Parameters supplied in the API call | Shows intent and target values |
| `responseElements` | Returned values from the API call | Can reveal created resource IDs |
| `errorCode` | Error name if the request failed | Important for denied or failed attempts |
| `errorMessage` | Human-readable failure detail | Helps explain why the action failed |
| `requestID` | Service request identifier | Useful when correlating with service support or logs |
| `resources` | Resources affected by the event | Helps map impact |
| `recipientAccountId` | Account that received the request | Useful in cross-account analysis |
| `readOnly` | Whether the event was read-only | Helps separate reads from writes |

A few interpretation notes matter for the exam:
- `recipientAccountId` is the account that received the request, which is not always the same as the resource owner in cross-account or service-linked scenarios.
- `sourceIdentity` can help preserve the original identity across role assumption chains, but only when it is explicitly set and propagated.
- `eventVersion` matters when you parse logs programmatically because field availability can vary by version and service.

**Identity and attribution analysis**

CloudTrail is often used to answer whether an action came from a human, a role, a federated session, or an AWS service.

Look closely at `userIdentity.type`:
- `Root`
- `IAMUser`
- `AssumedRole`
- `FederatedUser`
- `AWSService`
- `AWSAccount`
- `Unknown`

Useful fields inside `userIdentity` include:
- `arn`
- `accountId`
- `principalId`
- `sessionContext`
- `sessionIssuer`
- `sourceIdentity`

Practical interpretation:
- `IAMUser` often indicates direct long-term credentials.
- `AssumedRole` usually indicates temporary credentials from STS.
- `AWSService` often means an AWS service acted on your behalf.
- `Root` is a high-severity signal and should be investigated immediately.

For role sessions, `sessionContext.sessionIssuer` helps you identify the role that issued the session. This is especially useful when the session ARN itself is not enough to tell you which automation or workload used the role.

**Failed and denied API calls**

Denied or failed calls are often more valuable than successful ones during security investigations because they can show reconnaissance, privilege escalation attempts, or misconfigurations.

Watch for:
- `errorCode = AccessDenied`
- `errorCode = UnauthorizedOperation`
- `errorCode = Client.UnauthorizedOperation`
- `errorCode = ThrottlingException`
- `errorCode = InvalidClientTokenId`

What to infer:
- Repeated `AccessDenied` events may indicate brute-force probing of permissions.
- A successful `AttachRolePolicy` after earlier denied attempts can indicate privilege escalation.
- A denied `StopLogging` or `DeleteTrail` attempt is a strong indicator of tampering.

Example CloudTrail event pattern to inspect:
```json
{
  "eventTime": "2026-04-26T12:34:56Z",
  "eventSource": "iam.amazonaws.com",
  "eventName": "AttachUserPolicy",
  "awsRegion": "us-east-1",
  "sourceIPAddress": "203.0.113.10",
  "userAgent": "aws-cli/2.15.0",
  "userIdentity": {
    "type": "AssumedRole",
    "arn": "arn:aws:sts::111122223333:assumed-role/SecurityAdmin/AliceSession"
  },
  "errorCode": "AccessDenied",
  "errorMessage": "User is not authorized to perform: iam:AttachUserPolicy"
}
```

**Event history, trails, and log storage**

CloudTrail has multiple operational views, and exam questions often hinge on which one you should use.

- **Event history**
  - Shows recent management events in the console.
  - Is useful for quick lookups and initial triage.
  - Does not replace long-term retention or data event collection.

- **Trails**
  - Deliver events to Amazon S3 and optionally to CloudWatch Logs.
  - Support long-term retention and centralized auditing.
  - Can be single-Region or multi-Region.

- **CloudTrail Lake**
  - Stores events in an event data store for SQL-based analysis.
  - Is better suited for investigative queries over retained event data.
  - Is not a general-purpose data warehouse, so do not assume arbitrary joins across all external datasets.

**Single-Region vs multi-Region trails**

This distinction is important for both coverage and incident response.

- **Single-Region trail**
  - Captures events in one Region.
  - Is useful when you need scoped logging or have Region-specific compliance requirements.
  - Can miss activity in other Regions unless you create additional trails.

- **Multi-Region trail**
  - Captures management events across all Regions in the account.
  - Is the safer default for security visibility because attackers often pivot to another Region.
  - Helps you avoid blind spots during investigations.

Practical guidance:
- Use a multi-Region trail for baseline security logging.
- Add data event selectors only where the cost and investigative value justify them.
- Confirm that the S3 bucket used for trail delivery is in the same Region as the trail’s home Region.

**Trail delivery, S3 storage, and encryption**

CloudTrail delivers log files to S3, and the storage design matters for both durability and security.

Key requirements and considerations:
- The S3 bucket policy must allow CloudTrail to write logs.
- If you use SSE-KMS, the KMS key policy must allow CloudTrail to use the key.
- The KMS key must be in the same Region as the trail.
- For multi-Region trails, the trail still has a home Region, and the S3 bucket is associated with that home Region design.
- Cross-account delivery is common in centralized logging architectures, but you must explicitly grant the destination account permissions.

Typical bucket policy pattern:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AWSCloudTrailWrite",
      "Effect": "Allow",
      "Principal": { "Service": "cloudtrail.amazonaws.com" },
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::my-cloudtrail-logs/AWSLogs/111122223333/*",
      "Condition": {
        "StringEquals": {
          "s3:x-amz-acl": "bucket-owner-full-control"
        }
      }
    },
    {
      "Sid": "AWSCloudTrailAclCheck",
      "Effect": "Allow",
      "Principal": { "Service": "cloudtrail.amazonaws.com" },
      "Action": "s3:GetBucketAcl",
      "Resource": "arn:aws:s3:::my-cloudtrail-logs"
    }
  ]
}
```

If you use SSE-KMS, the key policy must allow CloudTrail to encrypt log files. A simplified pattern looks like this:
```json
{
  "Sid": "AllowCloudTrailUseOfKey",
  "Effect": "Allow",
  "Principal": { "Service": "cloudtrail.amazonaws.com" },
  "Action": [
    "kms:GenerateDataKey*",
    "kms:Decrypt",
    "kms:DescribeKey"
  ],
  "Resource": "*"
}
```

**Log file integrity validation**

CloudTrail log file integrity validation helps you verify that delivered log files have not been altered, deleted, or replaced.

What it does:
- Uses digest files and cryptographic validation to confirm log file integrity.
- Lets you prove whether a log file chain is intact.
- Supports incident response and forensic defensibility.

Why it matters:
- If an attacker tampers with logs, integrity validation can reveal the inconsistency.
- It is especially useful when you suspect log deletion, bucket compromise, or unauthorized modification.

How you use it:
1. Enable log file validation on the trail.
2. Retrieve the digest files from the S3 log delivery path.
3. Validate the log files against the digest chain using the CloudTrail validation workflow or supported tooling.
4. Investigate any missing, altered, or out-of-sequence files.

> Integrity validation confirms delivery-chain integrity, not the truthfulness of the underlying AWS API event itself. It tells you whether the log file was changed after CloudTrail wrote it.

**CloudTrail Lake for investigations**

CloudTrail Lake stores events in an event data store and lets you query them with SQL. It is useful when you need longer retention, flexible filtering, or repeated investigations over the same dataset.

Operational concepts:
- **Event data store**: The container for ingested events.
- **Retention**: How long events remain queryable in the store.
- **Ingestion**: The process of bringing events into the store from trails or organization activity.
- **Query**: SQL-based analysis over stored events.

When to use CloudTrail Lake:
- You need to search large volumes of historical events.
- You want to filter by multiple fields without exporting logs first.
- You need repeatable investigative queries for an incident.

When not to overstate it:
- Do not assume it behaves like a full relational warehouse.
- Do not assume arbitrary joins across unrelated datasets are always the right model.
- Do not rely on it as the only source for near-real-time alerting.

Example query patterns:
```sql
SELECT eventTime, eventName, eventSource, awsRegion, sourceIPAddress, userIdentity.arn
FROM $EDS_ID
WHERE eventName = 'ConsoleLogin'
ORDER BY eventTime DESC
LIMIT 20;
```

```sql
SELECT eventTime, eventName, errorCode, errorMessage, userIdentity.arn
FROM $EDS_ID
WHERE errorCode IS NOT NULL
ORDER BY eventTime DESC
LIMIT 50;
```

```sql
SELECT eventTime, eventName, resources
FROM $EDS_ID
WHERE eventSource = 's3.amazonaws.com'
  AND eventName IN ('PutObject', 'DeleteObject', 'GetObject')
ORDER BY eventTime DESC;
```

**CloudTrail lookup and filtering**

The `lookup-events` API is useful for quick searches of recent management events. It supports a limited set of lookup attributes, so know its boundaries.

Common supported lookup attributes include:
- `EventId`
- `EventName`
- `EventSource`
- `ReadOnly`
- `Username`
- `ResourceName`
- `ResourceType`
- `AccessKeyId`

Use it like this:
```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=ConsoleLogin \
  --max-results 10
```

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=Username,AttributeValue=alice \
  --max-results 20
```

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=ResourceName,AttributeValue=i-0123456789abcdef0
```

Important caution:
- `lookup-events` is not a full-text search engine.
- It does not replace CloudTrail Lake for deeper analysis.
- It is limited to management events in the event history window.

**Event selectors and data event scope**

Event selectors control what a trail records. This is a common exam topic because selector scope directly affects what you can investigate later.

Basic event selectors are used to include or exclude broad categories of events. Advanced event selectors let you filter more precisely by fields such as event category, resource type, and resource ARN patterns.

Example trail with management events and S3 data events:
```bash
aws cloudtrail put-event-selectors \
  --trail-name security-trail \
  --event-selectors '[
    {
      "ReadWriteType": "All",
      "IncludeManagementEvents": true,
      "DataResources": [
        {
          "Type": "AWS::S3::Object",
          "Values": ["arn:aws:s3:::sensitive-bucket/"]
        }
      ]
    }
  ]'
```

Advanced selector example:
```bash
aws cloudtrail put-event-selectors \
  --trail-name security-trail \
  --advanced-event-selectors '[
    {
      "Name": "S3DataEvents",
      "FieldSelectors": [
        { "Field": "eventCategory", "Equals": ["Data"] },
        { "Field": "resources.type", "Equals": ["AWS::S3::Object"] },
        { "Field": "resources.ARN", "StartsWith": ["arn:aws:s3:::sensitive-bucket/"] }
      ]
    }
  ]'
```

Selector pitfalls:
- If you do not include the right resource type, the event will not be captured.
- If you scope too narrowly, you may miss the activity you are trying to investigate.
- If you enable broad data events everywhere, cost and noise can become significant.
- Event history and trail logs are not the same thing; selector changes affect trail delivery, not the console event history window.

**Practical CLI workflow for incident response**

A common investigation flow is to start broad, then narrow down by identity, Region, and resource.

1. Identify the suspicious time window.
2. Search for the likely event name or identity.
3. Inspect the surrounding events for escalation or persistence.
4. Correlate with other logs and findings.
5. Preserve evidence and validate log integrity.

Example workflow:
```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=StopLogging \
  --start-time 2026-04-26T00:00:00Z \
  --end-time 2026-04-26T23:59:59Z
```

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=PutBucketPolicy \
  --start-time 2026-04-26T00:00:00Z \
  --end-time 2026-04-26T23:59:59Z
```

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=ResourceName,AttributeValue=my-sensitive-bucket \
  --start-time 2026-04-26T00:00:00Z \
  --end-time 2026-04-26T23:59:59Z
```

What to look for in sequence:
- `CreateAccessKey` followed by unusual API activity.
- `AttachUserPolicy` or `PutUserPolicy` followed by privilege use.
- `AssumeRole` followed by cross-account or cross-Region actions.
- `StopLogging`, `DeleteTrail`, `PutEventSelectors`, or `UpdateTrail` followed by suspicious gaps.

**Correlating CloudTrail with other AWS services**

CloudTrail is most powerful when you combine it with other telemetry.

- **Amazon CloudWatch Logs**
  - Use it for near-real-time log analysis and metric filters.
  - Useful when you want alarms on specific API patterns.
  - Common pattern: send CloudTrail to CloudWatch Logs, then create metric filters for sensitive actions.

- **Amazon EventBridge**
  - Use it for event-driven alerting and automated response.
  - Good for triggering Lambda, SNS, or Step Functions when specific API calls occur.
  - Example use case: alert on `DeleteTrail`, `PutBucketPolicy`, or root activity.

- **Amazon GuardDuty**
  - Use it to detect suspicious behavior such as anomalous API calls, credential compromise, or unusual data access.
  - Correlate GuardDuty findings with CloudTrail to confirm the underlying API sequence.

- **VPC Flow Logs**
  - Use them to correlate network source/destination patterns with API activity.
  - Helpful when CloudTrail shows an API call from an IP address and you need network context.

- **IAM Access Analyzer**
  - Use it to assess whether a policy change exposed a resource externally.
  - Pair it with CloudTrail to see who changed the policy and when.

Example EventBridge rule for sensitive CloudTrail events:
```json
{
  "source": ["aws.cloudtrail"],
  "detail-type": ["AWS API Call via CloudTrail"],
  "detail": {
    "eventSource": ["cloudtrail.amazonaws.com", "iam.amazonaws.com"],
    "eventName": ["StopLogging", "DeleteTrail", "PutBucketPolicy", "AttachRolePolicy"]
  }
}
```

**Incident response patterns**

CloudTrail investigations often follow recognizable patterns.

- **Logging tampering**
  - Look for `StopLogging`, `DeleteTrail`, `UpdateTrail`, `PutEventSelectors`, or S3 bucket policy changes.
  - Verify whether log file integrity validation still passes.
  - Check whether the attacker changed the destination bucket or KMS key.

- **Privilege escalation**
  - Look for IAM policy attachment, inline policy changes, role trust policy changes, or creation of new access keys.
  - Correlate with `AssumeRole` and subsequent high-privilege actions.

- **Credential misuse**
  - Look for unusual source IPs, user agents, Regions, or times of day.
  - Compare the activity to the normal behavior of the principal.
  - Check for failed sign-ins or repeated denied API calls before success.

- **Data access abuse**
  - Focus on S3 object-level events, Lambda invocations, and other data events.
  - Correlate object access with bucket policy changes and IAM permissions.

**Real-world command examples**

Create a multi-Region trail with log file validation and SSE-KMS:
```bash
aws cloudtrail create-trail \
  --name security-trail \
  --s3-bucket-name my-cloudtrail-logs \
  --s3-key-prefix audit \
  --is-multi-region-trail \
  --enable-log-file-validation \
  --kms-key-id arn:aws:kms:us-east-1:111122223333:key/12345678-1234-1234-1234-123456789012
```

Start logging:
```bash
aws cloudtrail start-logging --name security-trail
```

Describe trail status:
```bash
aws cloudtrail get-trail-status --name security-trail
```

Inspect a specific event by ID:
```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventId,AttributeValue=12345678-90ab-cdef-1234-567890abcdef
```

**Exam-focused distinctions**

- **Event history** is for quick recent management-event lookup.
- **Trails** are for durable delivery to S3 and optional CloudWatch Logs.
- **CloudTrail Lake** is for SQL-based investigation over retained event data.
- **Management events** are about control-plane changes.
- **Data events** are about resource-level access.
- **Network activity events** are service-specific and should not be assumed to exist for every service.
- **Log file integrity validation** proves the log chain was not tampered with after delivery.
- **Multi-Region trails** reduce blind spots and are usually the safer security choice.

**Common exam traps**

- Assuming all API activity appears in Event history forever.
- Treating `lookup-events` as a full search engine.
- Assuming every service supports data events or network activity events.
- Confusing `recipientAccountId` with resource ownership in cross-account cases.
- Assuming `sourceIdentity` is always present in role chains.
- Overstating CloudTrail Lake as a general SQL warehouse.
- Forgetting that data events can create significant cost and volume.

**Quick mental model**

When you see a CloudTrail question, ask:
- What event category is involved?
- Is the activity management, data, or network-related?
- Which identity made the request?
- Was the request successful or denied?
- Which Region and resource were affected?
- Do you need S3 logs, CloudWatch Logs, EventBridge, or CloudTrail Lake to complete the investigation?
