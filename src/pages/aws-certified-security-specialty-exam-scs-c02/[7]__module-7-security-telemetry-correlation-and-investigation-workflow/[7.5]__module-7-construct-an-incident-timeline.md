## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Why incident timelines matter**

An incident timeline turns scattered telemetry into a defensible sequence of events. In AWS, you rarely rely on one log source alone. You correlate management activity, resource configuration changes, and network evidence to answer four questions:

- What happened?
- When did it happen?
- Who or what caused it?
- What changed before, during, and after the event?

A strong timeline supports containment, scoping, root-cause analysis, and post-incident reporting. It also helps you distinguish between an actual compromise and normal automation, delayed delivery, or benign administrative activity.

**Core timeline principles**

Build the timeline around evidence quality, not just event count. The most useful sources are the ones that provide identity, time, and state change.

- **CloudTrail** shows API activity and identity context.
- **AWS Config** shows configuration state over time.
- **VPC Flow Logs** show network traffic patterns.
- **Service-specific logs** such as ALB, CloudFront, Lambda, or application logs add workload context.
- **CloudWatch Logs** can capture OS, application, and custom telemetry.

Use UTC consistently. AWS service logs are generally timestamped in UTC, but delivery latency, collection delay, and clock skew can still make events appear out of order. Normalize all sources to a single time standard before you compare them.

> When sources disagree on exact ordering, prefer the source with the strongest event semantics and the clearest identity context, then explain the discrepancy in your notes.

**CloudTrail as the primary activity record**

AWS CloudTrail is usually the backbone of an incident timeline because it records API calls made through the AWS control plane. It helps you identify who performed an action, from where, and with what result.

Important CloudTrail fields for investigations include:

- **eventTime**: When AWS recorded the event.
- **eventName**: The API action, such as `AuthorizeSecurityGroupIngress` or `CreateAccessKey`.
- **eventSource**: The service namespace, such as `ec2.amazonaws.com`.
- **userIdentity**: The actor, including IAM user, assumed role, federated user, AWS service, or root.
- **eventID**: A unique identifier for the event; useful for deduplication and cross-reference.
- **eventType**: Helps distinguish management events, data events, and other categories.
- **recipientAccountId**: The account that received the event, especially useful in cross-account activity.
- **sourceIPAddress**: The apparent origin of the request.
- **userAgent**: Often reveals console use, SDK use, automation, or third-party tooling.
- **sessionContext**: Critical for assumed roles and temporary credentials; it can show session issuer and session attributes.
- **errorCode** and **errorMessage**: Essential for failed or partial actions, such as denied access or throttling.
- **requestParameters** and **responseElements**: Show what was attempted and what changed.

CloudTrail is especially valuable for reconstructing privilege escalation, persistence, and defense evasion because it records the control-plane actions that create or modify access.

Example CloudTrail event fragment:

```json
{
  "eventTime": "2026-04-26T14:12:33Z",
  "eventSource": "iam.amazonaws.com",
  "eventName": "CreateAccessKey",
  "awsRegion": "us-east-1",
  "sourceIPAddress": "203.0.113.10",
  "userAgent": "aws-cli/2.15.0",
  "userIdentity": {
    "type": "AssumedRole",
    "arn": "arn:aws:sts::111122223333:assumed-role/SecurityAdmin/Alice",
    "sessionContext": {
      "sessionIssuer": {
        "type": "Role",
        "arn": "arn:aws:iam::111122223333:role/SecurityAdmin"
      }
    }
  },
  "recipientAccountId": "111122223333",
  "eventID": "c2f1d7e5-5d8a-4f7f-9d2c-1d2f5b8a9f11",
  "eventType": "AwsApiCall",
  "errorCode": null,
  "errorMessage": null
}
```

**CloudTrail log integrity and validation**

For defensible timelines, you need to show that logs were not altered after collection. CloudTrail supports log file validation through digest files and cryptographic verification.

- **Log files** contain the recorded events.
- **Digest files** contain hashes for log files and are used to validate integrity.
- **Validation** confirms that a log file has not been modified, deleted, or reordered after delivery.

Use validation when you need to preserve evidentiary confidence, especially for regulated environments or formal incident response.

Typical validation workflow:

1. Retrieve the CloudTrail log files and the corresponding digest files from the trail destination.
2. Verify the digest chain to confirm the files are intact.
3. Compare the expected hashes against the downloaded log files.
4. Document any missing files, hash mismatches, or delivery gaps.

Example validation command:

```bash
aws cloudtrail validate-logs \
  --trail-arn arn:aws:cloudtrail:us-east-1:111122223333:trail/OrgTrail \
  --start-time 2026-04-26T00:00:00Z \
  --end-time 2026-04-26T23:59:59Z \
  --verbose
```

> If validation fails, treat the affected interval as potentially unreliable and corroborate it with independent sources such as AWS Config, VPC Flow Logs, or application logs.

**AWS Config for state reconstruction**

AWS Config complements CloudTrail by showing how resources looked at specific points in time. CloudTrail tells you that a change occurred; Config tells you what the resource state became and how it relates to other resources.

Use AWS Config to answer questions such as:

- What security group rules existed before and after the incident?
- Which IAM role, instance profile, or network interface was attached to a resource?
- What dependencies existed between resources at the time of the event?

Key concepts:

- **Configuration item**: A point-in-time snapshot of a resource.
- **Configuration history**: The sequence of configuration items for a resource.
- **Relationships**: Links between resources, such as an EC2 instance attached to an ENI or a security group associated with an instance.
- **Advanced queries**: SQL-like queries that help you search across configuration history and relationships.

Example advanced query:

```sql
SELECT
  resourceId,
  resourceType,
  configurationItemCaptureTime,
  configuration.securityGroups,
  relationships
WHERE
  resourceType = 'AWS::EC2::Instance'
  AND configurationItemCaptureTime BETWEEN '2026-04-26T00:00:00Z' AND '2026-04-26T23:59:59Z'
```

Config is especially useful when you need to reconstruct dependency chains, such as which security group protected a workload before a suspicious inbound connection or which IAM role was attached when a malicious API call occurred.

**VPC Flow Logs for network sequencing**

VPC Flow Logs help you understand network reachability and traffic patterns, but they do not show application payloads or identify the exact application protocol beyond what can be inferred from ports and traffic behavior.

Important limitations and nuances:

- They record accepted and rejected traffic, which helps you see both successful connections and blocked attempts.
- They are aggregated records, not packet captures.
- They do not tell you what data was sent.
- They do not identify the application with certainty.
- They may not show every individual packet or short-lived connection in the way a packet capture would.

Use them to confirm whether a host could reach a destination, whether a security group or NACL blocked traffic, and whether unusual east-west or egress patterns appeared.

Example flow log interpretation:

- `ACCEPT` traffic to `443` may indicate successful outbound HTTPS communication.
- `REJECT` traffic to `22` may indicate blocked SSH probing or a denied administrative attempt.
- Repeated short bursts to an unfamiliar IP can indicate scanning, beaconing, or automation, but you need other logs to confirm intent.

Example flow log line:

```text
2 111122223333 eni-0abc123def4567890 10.0.1.25 198.51.100.77 51544 443 6 10 840 1714140753 1714140813 ACCEPT OK
```

**Time normalization and ordering**

Timeline work fails when timestamps are compared without normalization. Even if everything is in UTC, you still need to account for:

- **Delivery delay**: Some services deliver logs minutes later than the event time.
- **Collection delay**: Centralized logging pipelines can buffer or batch records.
- **Clock skew**: Workloads and appliances may have slightly incorrect local clocks.
- **Event semantics**: A control-plane action may appear before the network effect it caused.

Practical guidance:

- Convert all timestamps to UTC before analysis.
- Keep both the original timestamp and the normalized timestamp in your working notes.
- Sort by event time, then refine by delivery time or ingestion time if available.
- Use a confidence label when exact ordering is uncertain.
- Treat near-simultaneous events as a cluster when the evidence does not support a precise sequence.

A useful working format is:

```text
2026-04-26T14:12:33Z  CloudTrail  CreateAccessKey
2026-04-26T14:12:40Z  Config      Security group changed
2026-04-26T14:13:05Z  Flow Logs    Outbound connection to 198.51.100.77:443
```

**Multi-account and multi-Region timelines**

In AWS Organizations, incidents often span multiple accounts and Regions. A single compromised identity can touch shared services, workload accounts, and logging accounts.

Use a centralized logging strategy:

- Send CloudTrail from all accounts to an organization trail.
- Store logs in a dedicated logging account.
- Enable multi-Region trails when you need coverage across Regions.
- Use delegated administration where appropriate for centralized security services.
- Correlate account IDs, role ARNs, and session names across accounts.

Cross-account correlation patterns:

- Match `recipientAccountId` to the affected account.
- Match `userIdentity.arn` or `sessionContext.sessionIssuer.arn` to the originating role.
- Use `AssumeRole` events to bridge identity movement between accounts.
- Compare the same `eventID` only within the same event record; do not assume identical IDs across services.

For multi-Region work, remember that:

- A user may authenticate in one Region and act in another.
- Global services and IAM activity may appear in `us-east-1` or the home Region depending on the service and logging behavior.
- Resource state changes in one Region can trigger network effects or replication in another.

**Prioritizing evidence when sources conflict**

When logs disagree, do not force them into a false sequence. Instead, rank them by usefulness for the specific question.

Suggested priority for control-plane questions:

1. CloudTrail
2. AWS Config
3. Service logs
4. VPC Flow Logs

Suggested priority for network reachability questions:

1. VPC Flow Logs
2. Security group and NACL configuration history from Config
3. CloudTrail for the change event
4. Application logs

How to resolve discrepancies:

- Prefer the source that directly records the action you are trying to prove.
- Use Config to confirm the state that existed when the action occurred.
- Use Flow Logs to confirm whether traffic was possible or observed.
- Document gaps caused by retention, delivery delay, or missing telemetry.
- If two sources conflict, note both and explain which one is more authoritative for that question.

**End-to-end incident reconstruction example**

Scenario: A suspicious IAM role is used to create an access key, modify a security group, and then reach an external host from an EC2 instance.

Evidence fragments:

```json
{
  "eventTime": "2026-04-26T14:12:33Z",
  "eventName": "CreateAccessKey",
  "eventSource": "iam.amazonaws.com",
  "userIdentity": {
    "type": "AssumedRole",
    "arn": "arn:aws:sts::111122223333:assumed-role/SecurityAdmin/Alice"
  },
  "recipientAccountId": "111122223333",
  "eventID": "1111-aaaa"
}
```

```json
{
  "configurationItemCaptureTime": "2026-04-26T14:12:40Z",
  "resourceType": "AWS::EC2::SecurityGroup",
  "resourceId": "sg-0123456789abcdef0",
  "configuration": {
    "ipPermissions": [
      {
        "fromPort": 22,
        "toPort": 22,
        "ipRanges": [{"cidrIp": "0.0.0.0/0"}]
      }
    ]
  }
}
```

```text
2 111122223333 eni-0abc123def4567890 10.0.1.25 198.51.100.77 51544 443 6 10 840 1714140780 1714140840 ACCEPT OK
```

Reconstructed sequence:

1. An assumed role in account `111122223333` creates a new access key at `14:12:33Z`.
2. The security group is modified shortly after, exposing SSH from anywhere at `14:12:40Z`.
3. The instance associated with the affected ENI makes outbound HTTPS traffic to `198.51.100.77` at `14:13:00Z` to `14:14:00Z`.
4. The timeline suggests credential abuse followed by network-enabled activity, but you still need application logs or endpoint telemetry to confirm payload actions.

This kind of reconstruction is exam-relevant because it shows how to combine identity, configuration, and network evidence into one coherent narrative.

**Practical CLI and query workflow**

Use AWS CLI and service queries to gather evidence quickly.

CloudTrail lookup example:

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=CreateAccessKey \
  --start-time 2026-04-26T14:00:00Z \
  --end-time 2026-04-26T15:00:00Z
```

Config history example:

```bash
aws configservice get-resource-config-history \
  --resource-type AWS::EC2::SecurityGroup \
  --resource-id sg-0123456789abcdef0 \
  --earliest-time 2026-04-26T14:00:00Z \
  --later-time 2026-04-26T15:00:00Z
```

Flow Logs query example in CloudWatch Logs Insights:

```sql
fields @timestamp, srcAddr, dstAddr, srcPort, dstPort, action, bytes, packets
| filter action = "ACCEPT" and dstPort = 443
| sort @timestamp asc
| limit 50
```

**Common pitfalls**

- Assuming event order equals causal order.
- Ignoring `errorCode` and `errorMessage` when an action failed.
- Forgetting that assumed roles and federated identities require session context to identify the real actor.
- Treating VPC Flow Logs as packet captures.
- Using local time from one system without converting all sources to UTC.
- Overlooking cross-account activity in Organizations.
- Failing to validate CloudTrail log integrity before relying on the timeline for formal reporting.

**Exam-focused takeaways**

A strong AWS incident timeline combines multiple evidence types and explains their limitations.

- Use **CloudTrail** for who did what.
- Use **AWS Config** for what the resource looked like.
- Use **VPC Flow Logs** for whether network traffic occurred.
- Validate CloudTrail logs with digest files when integrity matters.
- Normalize time across sources and account for delay and skew.
- Correlate across accounts and Regions using account IDs, role ARNs, and session context.
- Resolve conflicts by choosing the most authoritative source for the question at hand.
