## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Incident investigation mindset**

In AWS, you rarely solve an incident from a single log source. You build a timeline by correlating identity, network, configuration, and data-access evidence across services, accounts, and Regions. The goal is to answer four questions:

- What happened?
- Who or what did it?
- What changed?
- What evidence proves it?

A strong investigation workflow starts with containment-safe evidence collection, then pivots through CloudTrail, AWS Config, VPC Flow Logs, DNS logs, GuardDuty, Security Hub, and service-specific logs such as S3 access logs, KMS events, and EKS audit logs.

**Core investigation principles**

- **Normalize time** to UTC before comparing events.
  - AWS services often deliver logs with different formats, delays, and ordering.
  - Use the event timestamp, not the delivery time, when building the sequence.
- **Preserve evidence** before making changes.
  - Export logs to immutable storage when possible.
  - Record hashes, source locations, and collection time for chain of custody.
- **Correlate across layers**.
  - Identity evidence explains who acted.
  - Configuration evidence explains what the resource looked like at the time.
  - Network evidence explains where traffic went.
  - Data evidence explains what was accessed.
- **Expect gaps**.
  - Some services log only control-plane activity.
  - Some logs are delayed or sampled.
  - Some evidence exists only if you enabled the right collection strategy in advance.

**CloudTrail as the primary identity and API record**

AWS CloudTrail records API activity and is usually the first place you look for control-plane actions. For exam purposes, distinguish the event categories and their investigative value:

- **Management events**
  - Capture control-plane actions such as `CreateUser`, `AuthorizeSecurityGroupIngress`, `PutBucketPolicy`, and `AssumeRole`.
  - CloudTrail event history provides management events for the last 90 days in each Region by default.
  - Trails are not automatically created in every account; long-term retention requires explicit configuration.
- **Data events**
  - Capture high-volume resource activity such as S3 object-level actions and Lambda function invocations.
  - These are not enabled by default and require explicit selection.
- **Insight events**
  - Highlight unusual API rate or error patterns.
  - Useful for detecting bursts, but not a replacement for raw event review.
- **Global service events**
  - Some AWS services are global in scope, so you may see activity recorded in a home Region even when the resource is not Region-specific.
  - Pay attention to IAM, STS, and other global control-plane activity when building a timeline.

CloudTrail also helps you understand **service-linked role** activity. Many AWS services act through service-linked roles, so the event may show an AWS service principal or a role session rather than a human user. That is normal and important: you must trace the initiating action back through the service workflow, not assume every event came directly from an IAM user.

**Practical CloudTrail collection strategy**

For high-value resources, plan collection before an incident:

- Use organization-level logging where possible.
- Centralize trails or CloudTrail Lake in a dedicated logging account.
- Enable data events for sensitive S3 buckets, Lambda functions, and other critical resources.
- Use advanced event selectors to reduce noise and cost while keeping the events you need.

Example advanced event selector for S3 object-level activity:

```json
{
  "AdvancedEventSelectors": [
    {
      "Name": "S3DataEvents",
      "FieldSelectors": [
        { "Field": "eventCategory", "Equals": ["Data"] },
        { "Field": "resources.type", "Equals": ["AWS::S3::Object"] },
        { "Field": "resources.ARN", "StartsWith": ["arn:aws:s3:::critical-bucket/"] }
      ]
    }
  ]
}
```

Example AWS CLI command to create a trail that logs to S3:

```bash
aws cloudtrail create-trail \
  --name org-security-trail \
  --s3-bucket-name central-cloudtrail-logs \
  --is-multi-region-trail
```

Then enable logging:

```bash
aws cloudtrail start-logging --name org-security-trail
```

**CloudTrail Lake and event history limits**

CloudTrail event history is useful for quick lookups, but it has limits:

- It covers only the last 90 days.
- It is Region-scoped for lookup behavior.
- It is not a substitute for long-term archival.

Use these sources when you need more history:

- **CloudTrail trails** for durable delivery to S3.
- **CloudTrail Lake** for SQL-based analysis and longer retention.
- **S3 log archives** for immutable retention and offline review.

CloudTrail Lake is especially useful when you need to query across many accounts or perform repeated investigations. You can create event data stores with selectors that focus on the resources you care about.

Example CloudTrail Lake event data store creation:

```bash
aws cloudtrail create-event-data-store \
  --name security-investigation-store \
  --multi-region-enabled \
  --organization-enabled \
  --retention-period 365
```

**AWS Config for resource state and change history**

CloudTrail tells you that an API call happened. AWS Config tells you what the resource looked like before and after that call. This is critical when you need to prove whether a security control changed.

Use AWS Config to investigate:

- Security group rule changes over time.
- IAM policy and role configuration drift.
- S3 bucket policy and public access settings.
- EBS volume, ENI, and other resource relationships.
- Resource compliance against rules and conformance packs.

A common investigation pattern is:

1. Find the suspicious API call in CloudTrail.
2. Check AWS Config for the resource’s configuration timeline.
3. Compare the before-and-after state to confirm impact.
4. Identify related resources that changed at the same time.

Example: if a security group suddenly allows `0.0.0.0/0` on port 22, CloudTrail may show `AuthorizeSecurityGroupIngress`, while AWS Config shows the exact rule set before and after the change.

**GuardDuty and Security Hub for detection and triage**

GuardDuty produces findings from threat intelligence, anomaly detection, and service telemetry. It is often the fastest way to identify suspicious activity, but it is not the final proof. Use it to prioritize investigation.

Common GuardDuty investigation uses:

- Detect unusual API calls or credential misuse.
- Identify suspicious DNS behavior.
- Flag anomalous instance behavior or exfiltration indicators.
- Surface findings related to compromised IAM credentials or unusual data access.

Security Hub acts as the **central aggregation and triage layer**. It collects findings from GuardDuty, AWS Config, and other integrated services, then normalizes them for review and automation.

Why Security Hub matters in incident response:

- It reduces the need to check each service separately.
- It helps you prioritize by severity and compliance status.
- It supports cross-service correlation for a broader incident picture.
- It can feed ticketing, automation, and response workflows.

**Network evidence: VPC Flow Logs and DNS logs**

Network logs help you answer where traffic went and whether the workload communicated with suspicious destinations.

**VPC Flow Logs** capture metadata about IP traffic to and from network interfaces, subnets, or VPCs. They do not show packet payloads, but they are useful for:

- Confirming outbound connections to unusual IPs.
- Identifying ports and protocols used during an incident.
- Correlating instance activity with exfiltration windows.
- Verifying whether traffic was accepted or rejected.

**Route 53 Resolver query logs** and other DNS logs are especially useful for command-and-control and exfiltration investigations:

- They show domain lookups made by workloads.
- They help you identify newly observed or suspicious domains.
- They can reveal beaconing patterns even when traffic is encrypted.
- They are often easier to pivot on than raw IP addresses.

A practical workflow is to start with a suspicious instance, then:

1. Check VPC Flow Logs for outbound connections.
2. Check DNS logs for the domains resolved around the same time.
3. Compare both with CloudTrail and GuardDuty findings.
4. Determine whether the traffic aligns with expected application behavior.

**KMS evidence and encryption-related investigation**

AWS Key Management Service (KMS) is often central to data-access investigations because encryption can hide the final data path while still leaving control-plane evidence.

Look for:

- `Decrypt`, `Encrypt`, `GenerateDataKey`, and `CreateGrant` activity.
- Unusual principals using a key.
- Cross-account key usage.
- Changes to key policies or grants.
- Access patterns that align with S3 reads, EBS attachment, or application decryption.

KMS evidence helps you answer whether a principal could have decrypted data, even if the data itself is not directly visible in logs. For example, a suspicious role may not show object contents, but repeated `Decrypt` calls around the same time as S3 reads can strongly support an exfiltration hypothesis.

Example CloudTrail lookup for KMS activity:

```bash
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=Decrypt \
  --max-results 20
```

**EKS and container-focused evidence**

Amazon EKS audit logs are useful for Kubernetes API activity, not for secret value disclosure. They can show requests involving Secrets resources, role bindings, pod creation, and privilege changes, but they do not reveal the secret contents themselves.

Use EKS audit logs to investigate:

- Creation or modification of pods, deployments, and daemonsets.
- Access to Kubernetes API resources.
- Changes to RBAC (Role-Based Access Control).
- Attempts to read or list Secrets resources.
- Suspicious service account usage.

Pair EKS audit logs with:

- CloudTrail for IAM and EKS control-plane actions.
- VPC Flow Logs for pod-to-external traffic.
- Container runtime or node logs when available.
- IAM role session activity for pod identity or IRSA (IAM Roles for Service Accounts).

**Cross-account and cross-Region pivoting**

Large AWS environments often centralize logging in a security account. Your investigation should assume that the suspicious activity may originate in one account and manifest in another.

Useful pivot points:

- **Organizations** to identify member accounts and delegated administrators.
- **Centralized logging accounts** to search aggregated CloudTrail, Config, and flow logs.
- **Cross-account role assumptions** to trace who accessed what.
- **Cross-Region review** to catch activity that moved between Regions.

A practical pivot workflow:

1. Start in the account where the alert fired.
2. Identify the principal, role session, and source IP.
3. Search the centralized CloudTrail logs for the same principal across all accounts.
4. Check whether the role was assumed from another account or federated identity.
5. Review AWS Config in the target account for resource changes.
6. Check network and DNS logs in the same time window.
7. Expand to adjacent Regions if the activity could have been replicated or copied.

Example CloudTrail Lake SQL for cross-account role assumptions:

```sql
SELECT eventTime, recipientAccountId, userIdentity.accountId, userIdentity.arn, sourceIPAddress, eventName
FROM security_investigation_store
WHERE eventName = 'AssumeRole'
  AND eventTime >= '2026-04-25T00:00:00Z'
ORDER BY eventTime DESC;
```

Example Athena-style query pattern for CloudTrail data stored in S3, using a timestamp cast when needed:

```sql
SELECT eventtime, useridentity.arn, eventname, sourceipaddress
FROM cloudtrail_logs
WHERE from_iso8601_timestamp(eventtime) >= TIMESTAMP '2026-04-25 00:00:00 UTC'
  AND eventname IN ('AssumeRole', 'ConsoleLogin', 'CreateAccessKey')
ORDER BY eventtime DESC;
```

> Be careful with schema differences. In many CloudTrail Athena tables, `eventtime` is stored as a string and must be cast before time comparisons.

**Time synchronization and ordering**

Incident timelines fail when timestamps are inconsistent. Always normalize before you compare sources.

Best practices:

- Convert all timestamps to UTC.
- Note the source timezone if a log is local-time based.
- Account for delivery delays in CloudTrail, Config, and flow logs.
- Use the event occurrence time, not the ingestion time, when possible.
- Preserve the original raw timestamp in your evidence notes.

A simple timeline template:

```text
2026-04-25T12:01:14Z  CloudTrail: AssumeRole by federated principal
2026-04-25T12:02:03Z  AWS Config: security group rule changed
2026-04-25T12:02:41Z  VPC Flow Logs: outbound connection to new IP
2026-04-25T12:03:10Z  DNS logs: lookup for suspicious domain
2026-04-25T12:04:22Z  CloudTrail: KMS Decrypt
```

**Evidence handling and chain of custody**

When you export logs or share them with responders, treat them as evidence.

Do this:

- Record the source account, Region, service, and log location.
- Export to a controlled bucket or evidence repository.
- Restrict write access and enable versioning or object lock where appropriate.
- Hash exported files and record the hash values.
- Keep a collection log with who collected the evidence and when.

Avoid these mistakes:

- Editing raw logs in place.
- Copying evidence into chat tools without tracking provenance.
- Relying on screenshots instead of exportable records.
- Overwriting original timestamps with local time.

Example S3 bucket policy concept for evidence retention should enforce limited deletion and controlled access. In practice, you would combine IAM, bucket policies, and object lock rather than rely on a single control.

**Practical investigation workflow**

1. Triage the alert in GuardDuty or Security Hub.
2. Identify the principal, resource, account, and Region.
3. Pull CloudTrail events for the relevant time window.
4. Check AWS Config for resource state changes.
5. Review VPC Flow Logs and DNS logs for network behavior.
6. Check KMS activity if encryption or data access is involved.
7. Review EKS audit logs if the workload is Kubernetes-based.
8. Expand to other accounts and Regions through centralized logging.
9. Preserve evidence and document the timeline.

**Common exam scenarios and how to reason about them**

- A security group was opened to the internet.
  - Use CloudTrail to find the API call.
  - Use AWS Config to confirm the exact rule change.
  - Use VPC Flow Logs to see whether traffic followed.
- A role appears to have accessed sensitive data.
  - Use CloudTrail to trace `AssumeRole` and downstream API calls.
  - Use KMS logs to confirm decrypt activity.
  - Use S3 data events or application logs to confirm object access.
- A workload contacted a suspicious domain.
  - Use DNS logs to identify the domain.
  - Use VPC Flow Logs to confirm the destination and timing.
  - Use CloudTrail and Config to see whether the workload or its permissions changed first.
- A Kubernetes workload may have been abused.
  - Use EKS audit logs to inspect API actions.
  - Use CloudTrail to trace IAM and cluster-level changes.
  - Use network logs to see whether the pod communicated externally.

**Quick reference: which source answers which question**

| Source | Best for | Limitations |
|---|---|---|
| CloudTrail | Who did what in AWS APIs | Not all data-plane activity; retention depends on configuration |
| AWS Config | What changed in resource state | Not a full activity log |
| GuardDuty | Suspicious behavior and threat detection | Finding, not proof |
| Security Hub | Central triage and aggregation | Depends on integrated sources |
| VPC Flow Logs | Network metadata and traffic patterns | No payloads |
| Route 53 Resolver query logs | DNS lookups and domain pivots | No packet content |
| KMS logs/events | Encryption and decrypt activity | Does not reveal plaintext data |
| EKS audit logs | Kubernetes API activity | Does not reveal secret values |

**What to remember for SCS-C02**

- CloudTrail event history is useful but limited to 90 days.
- Trails and CloudTrail Lake provide longer-term investigation capability.
- AWS Config is essential for proving resource state changes over time.
- Security Hub centralizes findings from GuardDuty, Config, and other services.
- DNS logs and VPC Flow Logs are often decisive for exfiltration and command-and-control cases.
- KMS activity can be the missing link in encrypted data investigations.
- Normalize timestamps and preserve evidence carefully.
- Investigations often require cross-account and cross-Region pivots through centralized logging.
