## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Why grants matter**

AWS Key Management Service (<abbr title="Key Management Service">KMS</abbr>) grants are a narrow, operational permission mechanism for using a customer managed KMS key without permanently expanding the key policy. You use them when a service, application, or automation role needs short-lived or tightly scoped access to cryptographic operations such as `Encrypt`, `Decrypt`, `GenerateDataKey`, or `DescribeKey`.

Grants are especially useful when the caller is not a human principal you want to manage directly in the key policy. In exam scenarios, think of grants as the tool for delegated, temporary, or service-driven access where you want to avoid frequent key policy edits.

- **Key policy** is the primary authorization boundary for a KMS key.
- **IAM policy** can further allow or deny a principal, but it does not replace the key policy.
- **Grant** adds a focused permission path for a specific key and use case.

**When to choose a grant**

Use a grant when a service or automation workflow needs access to a KMS key and the access pattern is operational rather than permanent. Common examples include AWS services that encrypt data on your behalf, ephemeral compute roles, and automation that must create data keys during deployment or runtime.

A grant is usually the right choice when:
- You need access for a specific key without broadening the key policy.
- The access is temporary, service-specific, or automation-driven.
- You want to delegate use of the key while keeping administrative control centralized.
- You need to support workflows where the caller may not have direct, long-term permissions in IAM.

A grant is usually not the right choice when:
- A simple key policy update is sufficient and stable.
- The principal should have ongoing, direct access that is easier to manage in IAM.
- The service already manages access through its own integration model and does not require a customer-created grant.
- You are working with an AWS managed key where AWS controls the policy model and grants are not something you normally administer.

**Grant vs key policy vs IAM policy**

KMS authorization is layered. The exam often tests whether you understand which layer is responsible for what.

| Mechanism | Purpose | Typical scope | Notes |
|---|---|---|---|
| Key policy | Primary control for the KMS key | Key-level | Must allow the use path; without it, IAM alone is not enough |
| IAM policy | Identity-based permission | Principal-level | Can allow a principal to call KMS, but key policy still matters |
| Grant | Narrow delegated permission | Key + grantee + operations | Best for service integrations and temporary automation access |

Practical interpretation:
- Use the **key policy** to define who can administer the key and who can delegate access.
- Use **IAM** to control which identities are allowed to request KMS operations.
- Use a **grant** to let a specific principal or service use a specific key for a specific purpose without rewriting the key policy.

> A grant does not magically bypass the key policy. The key policy must still allow the grant path, and cross-account use still requires the external principal path to be authorized appropriately.

**Grant lifecycle and states**

For exam purposes, know the lifecycle and who can act at each stage.

1. **Create**
   - A principal with permission to call `kms:CreateGrant` creates the grant.
   - The grant specifies the key, the grantee principal, allowed operations, and optional constraints.
   - The creator can optionally allow the grantee to retire the grant.

2. **Use**
   - The grantee uses the key operations allowed by the grant.
   - In some workflows, the caller may need to include a **grant token** immediately after creation because KMS authorization can be eventually consistent.

3. **Retire**
   - The grantee can retire the grant only if the grant was created to allow the grantee to retire it, or if the grantee is the retiring principal.
   - Retiring removes the grant from active use by the grantee.

4. **Revoke**
   - The grant creator or a principal with permission to revoke grants can revoke it.
   - Revocation is an administrative action that ends the grant even if the grantee does not cooperate.

5. **List / Describe**
   - Principals with the right permissions can list or describe grants to audit active delegated access.
   - This is important for troubleshooting and for detecting stale automation permissions.

**Grant permissions and who can do what**

The exact permission model depends on the key policy and IAM policy, but the exam expects you to understand the general control model.

- **Create grant**
  - Requires `kms:CreateGrant` on the key.
  - Often used by a service role, deployment role, or security automation role.
- **Use grant**
  - The grantee can use only the operations explicitly listed in the grant.
  - Common operations include `Encrypt`, `Decrypt`, `GenerateDataKey`, `GenerateDataKeyWithoutPlaintext`, and `DescribeKey`.
- **Retire grant**
  - The grantee can retire the grant only when the grant was created to permit that retirement path.
  - This is common in workflows where the grantee should clean up its own delegated access.
- **Revoke grant**
  - The grant creator or an authorized administrator can revoke it.
  - Use this when you need immediate termination of delegated access.
- **List/Describe grants**
  - Useful for audit and automation.
  - Often needed by operational tooling that tracks active grants.

**Grant constraints**

Grants can be narrowed with constraints so the delegated access is not overly broad.

Common constraint patterns:
- **Encryption context constraints**
  - Restrict use to requests that include specific encryption context key-value pairs.
  - This is useful when you want the key to be usable only for a particular application, tenant, or data classification.
- **Grant tokens**
  - Help the caller use a newly created grant before it is fully visible through normal eventual consistency.
- **Retiring principal**
  - Defines who can retire the grant.

Example grant creation with constraints:
```bash
aws kms create-grant \
  --key-id arn:aws:kms:us-east-1:111122223333:key/1234abcd-12ab-34cd-56ef-1234567890ab \
  --grantee-principal arn:aws:iam::111122223333:role/AppRuntimeRole \
  --operations Encrypt Decrypt GenerateDataKey \
  --constraints EncryptionContextEquals=AppName=payments,Environment=prod \
  --name payments-runtime-grant
```

This pattern is useful when you want the runtime role to use the key only for a specific application context.

**Grant tokens and eventual consistency**

KMS authorization can be eventually consistent. Right after you create a grant, the grantee may not be able to use it immediately unless the request includes the **grant token** returned by `CreateGrant`.

Typical flow:
1. Your automation calls `CreateGrant`.
2. KMS returns the grant ID and one or more grant tokens.
3. Your application or deployment script immediately calls a KMS operation and includes the token.
4. Once the grant propagates, the token is no longer needed.

Example SDK-style flow:
```python
import boto3

kms = boto3.client("kms", region_name="us-east-1")

resp = kms.create_grant(
    KeyId="arn:aws:kms:us-east-1:111122223333:key/1234abcd-12ab-34cd-56ef-1234567890ab",
    GranteePrincipal="arn:aws:iam::111122223333:role/AppRuntimeRole",
    Operations=["Decrypt", "GenerateDataKey"],
    Constraints={
        "EncryptionContextEquals": {
            "AppName": "payments"
        }
    },
    Name="payments-runtime-grant"
)

grant_token = resp["GrantToken"]

# Immediate use with token to avoid eventual consistency issues
kms.decrypt(
    CiphertextBlob=b"...",
    GrantTokens=[grant_token],
    EncryptionContext={"AppName": "payments"}
)
```

**Service integrations and automation roles**

Many AWS services use KMS on your behalf. In real deployments, the access path may involve a service principal, a service-linked role, or a customer-managed role that creates or uses grants. The exact model is service-specific, so avoid assuming every integration works the same way.

Common exam-relevant patterns:
- A service needs to encrypt data for you and uses a grant to do so.
- An automation role creates a grant for a short-lived workload.
- A deployment pipeline creates a grant for a bootstrap phase and then revokes it after setup.
- A service-linked role may need permission to create or use grants, often in combination with `kms:ViaService` conditions.

`kms:ViaService` is important when you want to restrict KMS usage to requests that come through a specific AWS service endpoint. This helps prevent direct, out-of-band use of the key by the same principal.

Example key policy fragment using `kms:ViaService`:
```json
{
  "Sid": "AllowUseThroughEBSOnly",
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::111122223333:role/EBSIntegrationRole"
  },
  "Action": [
    "kms:Encrypt",
    "kms:Decrypt",
    "kms:GenerateDataKey*",
    "kms:DescribeKey"
  ],
  "Resource": "*",
  "Condition": {
    "StringEquals": {
      "kms:ViaService": "ec2.us-east-1.amazonaws.com"
    }
  }
}
```

This is a pattern, not a universal template. Service integrations vary, and the exact principal and condition set depend on the AWS service and region.

**AWS managed keys vs customer managed keys**

This distinction matters a lot in exam questions.

- **AWS managed keys**
  - AWS controls the key policy and operational model.
  - You usually do not create grants manually for these keys.
  - The service manages access for you.
- **Customer managed keys**
  - You control the key policy.
  - You can create grants to delegate access to services, roles, and automation.
  - This is where grant design is most visible and testable.

Practical takeaway:
- If the question asks how to give a specific automation role limited access to a key you own, think **customer managed key + grant**.
- If the question asks about a service that transparently encrypts data with an AWS managed key, grants are usually not the thing you administer.

**Operational limits and grant quotas**

KMS keys have limits on the number of grants they can hold. Hitting a grant limit can break automation-heavy systems, especially when each deployment, instance, or tenant creates its own grant.

Operational implications:
- Excessive per-instance grants can accumulate quickly.
- Stale grants can prevent new automation from succeeding.
- Grant cleanup should be part of lifecycle management.
- Monitoring and periodic review matter in large environments.

If you see repeated failures in automation that creates grants, check:
- Whether the key has too many active grants.
- Whether old grants were never retired or revoked.
- Whether the workflow should reuse a grant instead of creating a new one every time.

**Troubleshooting common failures**

KMS errors often point to authorization or request-shape problems.

- **`AccessDeniedException`**
  - The principal lacks permission in IAM, the key policy, or the grant path.
  - Check all three layers.
- **`InvalidCiphertextException`**
  - Often appears when decrypting with the wrong encryption context, wrong key, or malformed ciphertext.
  - A mismatched encryption context is a common cause, but not the only one.
- **Grant not effective immediately**
  - Use the grant token in the first request after creation.
- **Grant exists but operation still fails**
  - Confirm the grant includes the exact operation you need.
  - Confirm the key policy allows the grant path.
  - Confirm the request matches any encryption context constraints.
- **Too many grants**
  - Clean up stale grants and redesign the workflow to reuse or retire them.

**Exam guidance: grant or policy update?**

Choose the least disruptive control that still meets the requirement.

Use a **grant** when:
- The access is temporary or operational.
- A service or automation role needs narrow access to one key.
- You want to avoid frequent key policy edits.
- You need a fast, revocable delegation path.

Update the **key policy** when:
- The access is stable and long-lived.
- You need to define the administrative boundary of the key.
- You are onboarding a principal that should have ongoing access.
- The workflow is simple enough that a grant adds unnecessary complexity.

Use **IAM alone** when:
- The principal already has a key policy path that allows the action.
- You are controlling which identities may call KMS, but not delegating key-specific access dynamically.

**Practical deployment pattern**

A common automation pattern is:
- A deployment role creates a grant for an application runtime role.
- The runtime role uses the grant to generate data keys during startup.
- The application includes the grant token for immediate use.
- The deployment pipeline revokes the grant after bootstrap if the access is no longer needed.

Example cleanup commands:
```bash
aws kms list-grants --key-id arn:aws:kms:us-east-1:111122223333:key/1234abcd-12ab-34cd-56ef-1234567890ab

aws kms revoke-grant \
  --key-id arn:aws:kms:us-east-1:111122223333:key/1234abcd-12ab-34cd-56ef-1234567890ab \
  --grant-id 0c1f2e3d4a5b6c7d8e9f0123456789ab
```

If the grantee is allowed to retire the grant, it may do so itself:
```bash
aws kms retire-grant \
  --grant-token <grant-token>
```

**Key takeaways**

- Grants provide **scoped, delegated access** to a specific KMS key.
- They are ideal for **services and automation roles** that need temporary or narrowly defined permissions.
- A grant does **not** replace the key policy; the key policy still governs whether the grant path is valid.
- Use **grant tokens** for immediate use after creation because KMS authorization can be eventually consistent.
- Know the lifecycle: **create, use, retire, revoke, list, describe**.
- Watch for **grant limits** in automation-heavy environments.
- Prefer grants for operational delegation; prefer key policy updates for stable, long-lived access.
