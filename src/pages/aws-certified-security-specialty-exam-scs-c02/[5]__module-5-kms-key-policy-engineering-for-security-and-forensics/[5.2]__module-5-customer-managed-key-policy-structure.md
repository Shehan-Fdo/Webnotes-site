## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Customer managed key policy fundamentals**

AWS Key Management Service (KMS) uses a **key policy** as the primary authorization document attached to a customer managed KMS key. For exam purposes, the key policy is not just a permissions list; it is the policy that determines whether IAM policies can be used with the key at all.

A customer managed key must have at least one statement that keeps the account able to administer the key. The common default pattern is the `Enable IAM User Permissions` statement, which allows the account principal to control the key through IAM. Without a safe administrative path, you can lock yourself out of `kms:PutKeyPolicy` and related management actions.

- **Key policy** is required for KMS authorization.
- **IAM policies** can grant access only when the key policy allows the account to use IAM policies for that key, or when the key policy directly allows the principal.
- **Grants** provide temporary, scoped permissions for specific use cases such as AWS services or short-lived access.

A practical mental model is:

1. KMS evaluates the key policy for the key.
2. If the key policy directly allows the principal, access can succeed.
3. If the key policy delegates to IAM for the account, IAM permissions can contribute.
4. Grants can also authorize specific operations when present.

> The exam often tests the difference between “IAM says yes” and “KMS key policy says yes.” In KMS, IAM alone is not enough unless the key policy permits IAM-based authorization for that key.

**Default key policy structure**

When you create a customer managed key in the console, AWS typically creates a default policy that includes an account-level administrative statement. The common pattern is to allow the account root principal to administer the key and to enable IAM policies in the account to control usage.

A simplified example looks like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "Enable IAM User Permissions",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:root"
      },
      "Action": "kms:*",
      "Resource": "*"
    }
  ]
}
```

This statement is often misunderstood. It does not mean the root user personally performs every action. Instead, it gives the account a trusted anchor so IAM policies in the account can be used with the key, and it preserves a principal that can update the key policy.

Key points:

- The `root` principal in a key policy is an account-level principal, not a recommendation to use the root user interactively.
- The default statement helps prevent accidental lockout.
- You should preserve a principal or role that can always call `kms:PutKeyPolicy`.

**Authorization model and delegation**

KMS authorization is stricter than many AWS services because the key policy is central to access control. A common exam rule is:

- If the key policy directly names the principal, that principal can be authorized by the key policy.
- If you want IAM policies to control access, the key policy must explicitly allow the account to use IAM policies for that key.
- If neither is true, an IAM `Allow` statement does not grant access.

This is why a key policy that omits the account-level delegation pattern can block access even when the IAM policy looks correct.

A useful way to think about it:

- **Direct allow**: the key policy names the user, role, or service principal.
- **Delegated allow**: the key policy allows the account to use IAM policies, and IAM then grants the action.
- **Grant-based allow**: a grant authorizes a specific operation, often for AWS services or constrained workflows.

**Common key policy statement types**

Customer managed key policies usually contain a mix of administrative, usage, and service-integration statements.

| Statement type | Purpose | Typical principals |
|---|---|---|
| Administration | Manage the key policy, enable/disable the key, rotate, schedule deletion | Security admins, key admins, account root delegation |
| Usage | Encrypt, decrypt, re-encrypt, generate data keys, verify signatures | Application roles, users, service roles |
| Service integration | Allow an AWS service to create grants or use the key on your behalf | Service principal such as `logs.amazonaws.com`, `s3.amazonaws.com`, `rds.amazonaws.com` |
| Cross-account access | Permit another account’s principal or account to use the key | External role, external account root delegation pattern |

Administrative actions commonly include:

- `kms:DescribeKey`
- `kms:EnableKey`
- `kms:DisableKey`
- `kms:PutKeyPolicy`
- `kms:UpdateKeyDescription`
- `kms:ScheduleKeyDeletion`
- `kms:CancelKeyDeletion`
- `kms:TagResource`
- `kms:UntagResource`
- `kms:CreateAlias`
- `kms:DeleteAlias`
- `kms:EnableKeyRotation`
- `kms:DisableKeyRotation`

Usage actions commonly include:

- `kms:Encrypt`
- `kms:Decrypt`
- `kms:ReEncrypt*`
- `kms:GenerateDataKey*`
- `kms:Sign`
- `kms:Verify`
- `kms:GenerateMac`
- `kms:VerifyMac`
- `kms:GetPublicKey`

`kms:DescribeKey` is metadata access, not a cryptographic operation.

**Lockout-safe design**

The biggest operational risk with customer managed keys is losing the ability to update the key policy. If you remove all principals that can call `kms:PutKeyPolicy`, you may create an unrecoverable access problem.

Best practices:

- Keep a dedicated security admin role in the key policy.
- Preserve the default account-level delegation statement unless you have a strong reason to replace it.
- Test policy changes in a non-production account first.
- Use change control for any policy that removes broad administrative access.
- Avoid relying on a single human user as the only key administrator.

A safe pattern is to include both:

- An account-level delegation statement for IAM-based control.
- A tightly scoped admin role that can always update the policy.

Example admin statement:

```json
{
  "Sid": "AllowKeyAdminsToManageKey",
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::123456789012:role/KMSKeyAdmins"
  },
  "Action": [
    "kms:DescribeKey",
    "kms:PutKeyPolicy",
    "kms:EnableKey",
    "kms:DisableKey",
    "kms:ScheduleKeyDeletion",
    "kms:CancelKeyDeletion",
    "kms:TagResource",
    "kms:UntagResource",
    "kms:CreateAlias",
    "kms:DeleteAlias",
    "kms:EnableKeyRotation",
    "kms:DisableKeyRotation"
  ],
  "Resource": "*"
}
```

**Cross-account access**

Cross-account KMS use is a common exam topic. The key idea is that the external principal must be authorized by the key policy, and IAM in the external account may also be required depending on how you structure the policy.

Two common patterns exist:

- **Direct principal allow**: the key policy names the external role or user directly.
- **External account delegation**: the key policy allows the external account to use IAM policies, and the external account’s IAM policy grants the action.

Direct allow example:

```json
{
  "Sid": "AllowExternalRoleUseOfKey",
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::210987654321:role/ExternalAppRole"
  },
  "Action": [
    "kms:Decrypt",
    "kms:Encrypt",
    "kms:GenerateDataKey"
  ],
  "Resource": "*"
}
```

In this model, the external role can be authorized directly by the key policy. An IAM policy in the external account may still be needed for the role to call KMS, but the key policy is the decisive resource policy for the key.

If you instead want the external account to manage access through IAM, the key policy must explicitly allow that account’s IAM policies to be used with the key. Without that, the external account’s IAM `Allow` is not enough.

**Grants and when to use them**

A **grant** is a lightweight authorization object for a KMS key. Grants are useful when you want to allow a specific principal or AWS service to use a key without editing the key policy every time.

Use grants when:

- An AWS service needs to use the key on your behalf.
- You want temporary or narrowly scoped access.
- You need to avoid frequent key policy updates.

A grant can allow actions such as `Decrypt`, `Encrypt`, or `GenerateDataKey` for a specific grantee principal.

Example grant creation:

```bash
aws kms create-grant \
  --key-id arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555 \
  --grantee-principal arn:aws:iam::123456789012:role/AppRole \
  --operations Decrypt Encrypt GenerateDataKey \
  --name app-runtime-access
```

Important grant notes:

- A grant is not a replacement for a broken key policy.
- Some workflows may need the grant to exist before the service can use the key.
- Grant tokens can help with eventual consistency in some workflows, but they are not a universal requirement for every immediate-use scenario.

To retire a grant, use the grant identifier or token context rather than `--key-id`:

```bash
aws kms retire-grant \
  --grant-id 1234abcd-56ef-78gh-90ij-klmnopqrstuv
```

**AWS service integrations**

Many AWS services use KMS keys through service principals and grants. The exact policy shape varies by service, but the general pattern is:

- Allow the service principal to use the key.
- Often allow the service to create grants.
- Scope access with conditions such as encryption context, source account, or source ARN when documented for that service.

Example pattern for a service integration:

```json
{
  "Sid": "AllowServiceUseAndGrantCreation",
  "Effect": "Allow",
  "Principal": {
    "Service": "logs.amazonaws.com"
  },
  "Action": [
    "kms:Encrypt",
    "kms:Decrypt",
    "kms:GenerateDataKey*",
    "kms:DescribeKey",
    "kms:CreateGrant"
  ],
  "Resource": "*",
  "Condition": {
    "StringEquals": {
      "kms:ViaService": "logs.us-east-1.amazonaws.com",
      "kms:CallerAccount": "123456789012"
    }
  }
}
```

Use service-specific documentation for exact condition keys and required actions. The general exam takeaway is that many AWS services need `kms:CreateGrant` and a service principal path, not just a user or role allow statement.

**Encryption context and conditions**

KMS supports conditions that bind usage to context, which helps prevent confused-deputy problems and limits key use to intended workloads.

Common condition keys include:

- `kms:ViaService`
- `kms:CallerAccount`
- `kms:EncryptionContext:<key>`
- `kms:GrantIsForAWSResource`

Example with encryption context:

```json
{
  "Sid": "AllowAppDecryptWithContext",
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::123456789012:role/AppRole"
  },
  "Action": [
    "kms:Decrypt",
    "kms:GenerateDataKey"
  ],
  "Resource": "*",
  "Condition": {
    "StringEquals": {
      "kms:EncryptionContext:AppName": "billing"
    }
  }
}
```

Use encryption context when you want the ciphertext to be usable only in a specific application or workflow. This is especially useful for application-layer controls and auditability.

**Multi-Region customer managed keys**

Multi-Region keys use related key material across Regions, but each replica still has its own key resource in its Region. Policy design is conceptually similar, but you should remember:

- Each key replica has its own key policy.
- You must manage access consistently across replicas.
- A policy change on one replica does not automatically mean every replica is updated the same way unless you apply the change to each key resource as needed.

For exam purposes, treat multi-Region keys as separate KMS resources that need consistent authorization design.

**Policy size and operational constraints**

Complex key policies can become difficult to manage. Practical constraints matter:

- Key policies have size limits, so large numbers of principals or statements can become unwieldy.
- Overly broad statements increase blast radius.
- Too many service-specific exceptions make troubleshooting harder.
- Centralized roles and grants often scale better than many per-user statements.

Design for maintainability:

- Prefer roles over individual users.
- Use conditions to narrow access.
- Use grants for service-specific or temporary access.
- Keep a small number of administrative statements.

**Exam-style end-to-end scenario**

Suppose an application in Account A must decrypt data stored in Amazon S3, and the application runs as an IAM role. You want the security team to retain control of the key.

A good design is:

1. The key policy includes the default account-level delegation statement or an equivalent account-root delegation pattern.
2. The key policy includes a security admin role that can update the policy.
3. The application role in Account A gets an IAM policy allowing `kms:Decrypt` and `kms:GenerateDataKey`.
4. The key policy allows the account to use IAM policies, so the IAM allow is effective.
5. If an AWS service such as S3 or Lambda needs to use the key on the application’s behalf, add the documented service principal or grant pattern.

Example IAM policy for the app role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowKMSUseForApp",
      "Effect": "Allow",
      "Action": [
        "kms:Decrypt",
        "kms:GenerateDataKey"
      ],
      "Resource": "arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555"
    }
  ]
}
```

This works only if the key policy permits IAM-based authorization for that key or directly allows the role.

**Troubleshooting checklist**

When KMS access fails, check these in order:

- The key policy includes a valid administrative path.
- The principal is directly allowed in the key policy, or the key policy delegates to IAM.
- The IAM policy allows the action on the correct key ARN.
- The action is valid for the principal’s use case.
- Any required grant exists.
- Service-specific conditions such as `kms:ViaService` or encryption context match the request.
- The key is enabled and not pending deletion.

**CLI commands you should know**

Inspect a key policy:

```bash
aws kms get-key-policy \
  --key-id arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555 \
  --policy-name default
```

Update a key policy:

```bash
aws kms put-key-policy \
  --key-id arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555 \
  --policy-name default \
  --policy file://key-policy.json
```

List grants:

```bash
aws kms list-grants \
  --key-id arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555
```

Create a grant:

```bash
aws kms create-grant \
  --key-id arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555 \
  --grantee-principal arn:aws:iam::123456789012:role/AppRole \
  --operations Decrypt GenerateDataKey
```

**High-value exam takeaways**

- The key policy is the core authorization document for a customer managed KMS key.
- IAM permissions do not work by themselves unless the key policy allows IAM-based control for that key.
- Preserve a safe administrative path to avoid lockout.
- Use grants for narrow, service-oriented, or temporary access.
- Cross-account access depends on how the key policy is structured.
- Service integrations often require service principals and grant creation.
- Each multi-Region key replica needs its own policy management attention.
