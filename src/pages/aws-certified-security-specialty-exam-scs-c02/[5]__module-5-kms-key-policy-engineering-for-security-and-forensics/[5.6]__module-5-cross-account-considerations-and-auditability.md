## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Cross-account security fundamentals**

Cross-account access means one AWS account assumes permissions in another account through a trust relationship. You use it to centralize administration, separate duties, and reduce long-lived credentials. The core challenge is to make access intentional, limited, and auditable.

The main control points are:

- **Identity-based policies** in the source account
- **Resource-based policies** in the target account
- **Trust policies** on IAM roles
- **Service control policies** (SCPs) in AWS Organizations
- **Permission boundaries** for delegated administration

A secure design starts by asking three questions:

- Who can request access?
- What can they do after they assume access?
- How do you prove what happened later?

**Cross-account access patterns**

The most common pattern is role assumption with AWS Security Token Service (STS). A principal in one account calls `sts:AssumeRole` into a role in another account, receives temporary credentials, and uses those credentials for the target account.

Typical use cases include:

- Centralized security operations
- Shared logging and audit accounts
- Cross-account backup and disaster recovery
- CI/CD pipelines deploying into multiple accounts
- Read-only access for auditors or support teams

A basic trust policy for a role in the target account looks like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowSecurityAuditAccount",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:root"
      },
      "Action": "sts:AssumeRole",
      "Condition": {
        "StringEquals": {
          "aws:PrincipalOrgID": "o-abcd1234"
        }
      }
    }
  ]
}
```

This pattern is stronger than trusting a single account ID alone because it can be constrained by organization membership. In practice, you should narrow the principal further to a specific role or use conditions such as `aws:PrincipalArn`, `aws:PrincipalTag`, or `sts:ExternalId` when appropriate.

**Trust policy design**

A trust policy answers the question: “Who may assume this role?” It does not grant permissions by itself; it only allows the STS call to succeed.

Important trust policy considerations:

- **Principal scope** should be as narrow as possible.
- **Conditions** should enforce context such as organization, source identity, external ID, or MFA.
- **ExternalId** helps prevent confused deputy problems in third-party access scenarios.
- **MFA** is useful for human access, but not always practical for automation.
- **Session duration** should match the task and be kept short for sensitive roles.

Example trust policy requiring MFA and a specific external ID:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::444455556666:role/PartnerAccessRole"
      },
      "Action": "sts:AssumeRole",
      "Condition": {
        "StringEquals": {
          "sts:ExternalId": "partner-12345"
        },
        "Bool": {
          "aws:MultiFactorAuthPresent": "true"
        }
      }
    }
  ]
}
```

Use `ExternalId` when a third party assumes a role in your account on behalf of multiple customers. It reduces the risk that the third party accidentally exposes one customer’s access to another.

> Be careful with `Principal: "*"`. In a trust policy, it can be acceptable only when tightly constrained by conditions. Without conditions, it is overly permissive.

**Permissions after assumption**

Once a role is assumed, the session permissions come from the role’s identity-based policies, plus any session policies, permission boundaries, and SCPs that apply. The effective permissions are the intersection of all applicable controls.

Key points:

- **Identity policy** grants what the role can do.
- **Session policy** can further restrict the temporary session.
- **Permission boundary** limits what the role can ever do.
- **SCPs** set the maximum permissions available in the account or OU.
- **Resource-based policies** can grant direct access to resources without role assumption.

A common exam trap is assuming that a trust policy alone grants access. It does not. The role also needs permissions to perform the action.

Example role permissions policy for read-only access to S3 and CloudWatch Logs:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadS3Buckets",
      "Effect": "Allow",
      "Action": [
        "s3:Get*",
        "s3:List*"
      ],
      "Resource": "*"
    },
    {
      "Sid": "ReadLogs",
      "Effect": "Allow",
      "Action": [
        "logs:Describe*",
        "logs:Get*",
        "logs:FilterLogEvents"
      ],
      "Resource": "*"
    }
  ]
}
```

**Resource-based policies**

Resource-based policies are often the simplest way to allow cross-account access to a specific resource. They are common for S3 buckets, KMS keys, SNS topics, SQS queues, Lambda functions, and Secrets Manager secrets.

Advantages:

- They can grant access directly to a resource.
- They can be easier to reason about for a single shared asset.
- They can support cross-account access without role assumption in some cases.

Examples of resource-based policy use:

- S3 bucket policy allowing a logging account to write logs
- KMS key policy allowing a security account to decrypt for audit
- SNS topic policy allowing another account to publish
- Lambda permission allowing another account or service to invoke

Example S3 bucket policy for centralized logging:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowLogDeliveryFromSecurityAccount",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:root"
      },
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::central-audit-logs/AWSLogs/444455556666/*",
      "Condition": {
        "StringEquals": {
          "s3:x-amz-acl": "bucket-owner-full-control"
        }
      }
    }
  ]
}
```

For S3, remember that object ownership matters. If another account writes objects into your bucket, you usually want the bucket owner to retain full control. Use bucket policies and ownership controls carefully to avoid access surprises.

**KMS and cross-account encryption**

AWS Key Management Service (KMS) is a frequent source of cross-account issues because access requires both IAM permission and key policy permission. For cross-account use, the key policy must allow the external principal or account, and the caller must also be allowed by IAM.

Important KMS concepts:

- **Key policy** is the primary authorization layer.
- **Grants** provide temporary or scoped permissions for services and principals.
- **Encryption context** can add additional authorization conditions.
- **Multi-Region keys** help with cross-region disaster recovery, not cross-account authorization by themselves.

Example key policy statement allowing a security account to use the key for decryption:

```json
{
  "Sid": "AllowSecurityAccountDecrypt",
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::111122223333:role/SecurityAuditRole"
  },
  "Action": [
    "kms:Decrypt",
    "kms:DescribeKey"
  ],
  "Resource": "*"
}
```

When troubleshooting KMS access, check all of these:

- The caller has IAM permission for the KMS action.
- The key policy allows the caller or its account.
- Any grant constraints are satisfied.
- The encryption context matches if used.
- The key is in the correct region.

**Organizations, SCPs, and delegated administration**

AWS Organizations helps you manage multiple accounts with centralized guardrails. SCPs do not grant permissions; they only limit the maximum permissions available to principals in member accounts.

Use SCPs to:

- Prevent disabling CloudTrail
- Block leaving the organization
- Restrict use of root credentials
- Limit regions
- Prevent changes to central logging or security services

Example SCP denying CloudTrail tampering:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyCloudTrailTampering",
      "Effect": "Deny",
      "Action": [
        "cloudtrail:DeleteTrail",
        "cloudtrail:StopLogging",
        "cloudtrail:UpdateTrail"
      ],
      "Resource": "*"
    }
  ]
}
```

Delegated administration lets a member account manage a service across the organization without giving it full management account access. This is useful for security tooling, backup, and logging. Even with delegation, you still need to verify:

- Which account is the delegated admin
- Which actions the service can perform
- Whether the service creates resources in member accounts or a central account
- How the service activity is logged

**Auditability goals**

Auditability means you can answer who did what, when, from where, and under which permissions. In AWS, this usually requires multiple telemetry sources working together.

Core audit sources:

- **AWS CloudTrail** for API activity
- **AWS Config** for resource configuration history and compliance
- **CloudWatch Logs** for application and system logs
- **VPC Flow Logs** for network traffic metadata
- **AWS Organizations** for account and policy changes
- **IAM Access Analyzer** for unintended external access findings

For cross-account activity, CloudTrail is the most important source because it records the assumed role session, the source IP, the event time, and the API call details.

A useful CloudTrail event for cross-account access includes:

- `userIdentity.type` such as `AssumedRole`
- `userIdentity.sessionContext.sessionIssuer`
- `userIdentity.arn`
- `sourceIPAddress`
- `eventSource`
- `eventName`
- `recipientAccountId`

Example CloudTrail event fragment:

```json
{
  "eventSource": "s3.amazonaws.com",
  "eventName": "GetObject",
  "userIdentity": {
    "type": "AssumedRole",
    "arn": "arn:aws:sts::444455556666:assumed-role/SecurityAuditRole/AuditSession",
    "sessionContext": {
      "sessionIssuer": {
        "type": "Role",
        "arn": "arn:aws:iam::444455556666:role/SecurityAuditRole",
        "userName": "SecurityAuditRole"
      }
    }
  },
  "recipientAccountId": "444455556666"
}
```

This tells you the action was performed using temporary credentials from an assumed role, not a long-lived IAM user.

**CloudTrail design for multi-account environments**

For strong auditability, use an organization trail that captures events from all accounts in the organization. Send logs to a dedicated log archive account and protect the bucket and trail configuration from tampering.

Recommended design:

- Enable an **organization trail**
- Log **management events** in all accounts
- Add **data events** for sensitive S3 buckets and Lambda functions where needed
- Use a dedicated **log archive account**
- Encrypt logs with KMS
- Restrict write access to the log bucket
- Enable log file validation

Example organization trail creation:

```bash
aws cloudtrail create-trail \
  --name org-security-trail \
  --s3-bucket-name central-audit-logs \
  --is-organization-trail

aws cloudtrail start-logging --name org-security-trail
```

If you need to inspect events quickly, use CloudTrail Lake or Athena over S3 logs. CloudTrail Lake is useful for managed querying, while Athena gives you flexible SQL over stored logs.

**CloudTrail hardening**

To preserve audit integrity, protect both the trail and the destination bucket.

Best practices:

- Use a dedicated account for log storage
- Enable S3 versioning on the log bucket
- Block public access
- Restrict bucket policy to the CloudTrail service and approved principals
- Use KMS encryption with a tightly scoped key policy
- Enable log file validation
- Monitor for `StopLogging`, `DeleteTrail`, and `PutEventSelectors`

Example S3 bucket policy for CloudTrail delivery:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AWSCloudTrailWrite",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudtrail.amazonaws.com"
      },
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::central-audit-logs/AWSLogs/123456789012/*",
      "Condition": {
        "StringEquals": {
          "s3:x-amz-acl": "bucket-owner-full-control",
          "aws:SourceArn": "arn:aws:cloudtrail:us-east-1:123456789012:trail/org-security-trail"
        }
      }
    }
  ]
}
```

**AWS Config for auditability**

AWS Config records resource configuration changes and can evaluate compliance against rules. It complements CloudTrail by showing the state of resources over time, not just API calls.

Use AWS Config to:

- Track configuration drift
- Detect noncompliant security settings
- Support change investigations
- Provide evidence for audits

Examples of useful managed rules:

- `s3-bucket-public-read-prohibited`
- `s3-bucket-public-write-prohibited`
- `cloudtrail-enabled`
- `encrypted-volumes`
- `iam-password-policy`

For cross-account governance, aggregate Config data into a central account. This gives you a single place to review compliance across the organization.

**IAM Access Analyzer**

IAM Access Analyzer helps identify resources that are shared with external principals. It is especially useful for cross-account review because it can detect unintended exposure in resource-based policies.

Use it to find:

- S3 buckets shared outside the account or organization
- KMS keys with broad access
- IAM roles trusted by external accounts
- Secrets or Lambda permissions exposed to other accounts

Access Analyzer is not a replacement for policy review, but it is a strong detection control for drift and misconfiguration.

**Centralized logging architecture**

A common secure topology uses separate accounts for workloads, security, and logs.

Typical layout:

- **Management account** for Organizations administration
- **Security account** for centralized security tooling
- **Log archive account** for immutable log storage
- **Workload accounts** for applications and services

Traffic and control flow:

- Workload accounts send CloudTrail, Config, and service logs to the log archive account
- Security analysts assume a read-only role into workload accounts when needed
- Security tooling uses delegated admin or cross-account roles
- SCPs prevent tampering with logging and security services

This separation improves blast-radius control and makes audit evidence easier to defend.

**Practical CLI examples**

Assume a role from a source account:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::444455556666:role/SecurityAuditRole \
  --role-session-name AuditSession
```

Export the temporary credentials:

```bash
export AWS_ACCESS_KEY_ID=ASIA...
export AWS_SECRET_ACCESS_KEY=...
export AWS_SESSION_TOKEN=...
```

Verify the identity:

```bash
aws sts get-caller-identity
```

List S3 objects in a cross-account log bucket:

```bash
aws s3 ls s3://central-audit-logs/AWSLogs/444455556666/
```

Query CloudTrail with Athena using SQL:

```sql
SELECT eventtime, eventname, useridentity.arn, sourceipaddress
FROM cloudtrail_logs
WHERE eventsource = 's3.amazonaws.com'
  AND recipientaccountid = '444455556666'
ORDER BY eventtime DESC
LIMIT 20;
```

**Common pitfalls**

Cross-account designs often fail because one layer is correct and another is missing.

Watch for these issues:

- Trust policy allows assumption, but the role has no permissions
- IAM policy allows access, but the resource policy denies it
- KMS key policy blocks decryption even though IAM allows it
- SCP blocks an action unexpectedly
- CloudTrail is enabled only in one account, not organization-wide
- Logs are stored in a bucket that the source account can modify or delete
- External access is granted broadly with `Principal: "*"` and weak conditions

> If you cannot explain the full permission path from caller to resource, the design is not auditable enough for production.

**Exam-focused distinctions**

Know the difference between these concepts:

- **Trust policy**: who can assume the role
- **Permissions policy**: what the role can do
- **Resource policy**: who can access the resource directly
- **SCP**: the maximum allowed permissions in an account or OU
- **Config**: resource state and compliance history
- **CloudTrail**: API activity and actor attribution

A strong answer on the exam usually combines prevention and detection:

- Prevent with least privilege, SCPs, and narrow trust policies
- Detect with CloudTrail, Config, and Access Analyzer
- Preserve evidence with centralized, immutable logging

**Quick reference checklist**

Before approving a cross-account design, verify:

- The trust relationship is narrowly scoped
- Temporary credentials are used instead of long-lived keys
- The target role has only required permissions
- Resource policies do not overexpose data
- KMS key policy and IAM policy both allow intended use
- CloudTrail is enabled organization-wide
- Logs are centralized, encrypted, and protected from tampering
- AWS Config and Access Analyzer are enabled for continuous review
- SCPs block destructive or logging-related actions where appropriate

**Summary**

Cross-account access is powerful, but it increases the number of policy layers you must reason about. The safest designs use temporary credentials, tightly scoped trust, centralized logging, and continuous detection. For auditability, you want a complete chain of evidence from the original principal to the final resource action, backed by immutable logs and configuration history.
