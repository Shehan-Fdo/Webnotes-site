## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**AWS KMS decrypt authorization model**

AWS Key Management Service (AWS KMS) controls decryption through a combination of the **key policy**, optional **IAM policy**, grants, and any applicable conditions. For a principal to decrypt data, the request must be authorized by the KMS key policy and, when used, IAM permissions must also allow the action. This is why a standalone IAM allow statement is not enough for most customer managed keys.

A common exam pattern is to separate:
- **Who can call KMS**
- **Which key they can use**
- **Under what conditions they can decrypt**

KMS evaluates the request against the key policy and the caller’s permissions. If you want to restrict decrypt access, you usually do it with:
- A narrow `kms:Decrypt` allow
- A key policy that explicitly permits the principal or enables IAM permissions for the key
- Conditions that bind usage to a specific application, tenant, or data classification

> Encryption context is not a standalone security boundary. It strengthens authorization and auditability, but it does not replace least-privilege access to the KMS key.

**Why restrict decrypt permissions**

Decrypt is the most sensitive KMS permission because it can reveal plaintext. If a principal can decrypt, they can often recover protected data even if they cannot re-encrypt or manage the key. In practice, you should treat `kms:Decrypt` as a high-risk permission and scope it tightly.

Use restrictive decrypt permissions when:
- Different applications share the same KMS key but must not read each other’s data
- A service should decrypt only data it created
- You need to enforce tenant isolation in a multi-tenant design
- You want to reduce blast radius if an IAM role is compromised

A good design usually combines:
- **Identity-based controls** in IAM
- **Resource-based controls** in the KMS key policy
- **Context-based controls** using encryption context
- **Operational controls** such as CloudTrail monitoring and key rotation

**Encryption context fundamentals**

Encryption context is a set of nonsecret key-value pairs that AWS KMS cryptographically binds to the ciphertext. You provide it during encryption, and you must provide the same context during decryption for the ciphertext to decrypt successfully.

Typical uses include:
- Application name
- Environment name
- Tenant identifier
- Data classification
- Record or object identifier

Example context:
```json
{
  "App": "payments",
  "Environment": "prod",
  "TenantId": "t-12345"
}
```

Important properties:
- The context is **not secret**
- It is included in KMS authorization decisions when you use condition keys
- It is useful for **integrity** and **binding**, not for hiding values
- It can appear in **CloudTrail event records** for KMS API calls, which helps with auditing and troubleshooting

**How encryption context helps**

Encryption context helps you bind ciphertext to the intended use case. If an attacker steals ciphertext but does not know the correct context, decryption fails. If a legitimate caller supplies the wrong context, KMS rejects the request because the context does not match the ciphertext’s associated context.

This is especially useful when:
- The same KMS key protects many records
- You want to prevent accidental cross-use of ciphertext
- You need to prove which application or tenant produced a ciphertext
- You want to add a second factor of authorization beyond the key itself

The context is most effective when you use it consistently in both the application and the authorization policy.

**Practical encryption and decryption examples**

Using the AWS CLI, you can encrypt with context and later decrypt with the same context.

```bash
aws kms encrypt \
  --key-id alias/payments-key \
  --plaintext fileb://secret.txt \
  --encryption-context App=payments,Environment=prod,TenantId=t-12345 \
  --output text \
  --query CiphertextBlob
```

Decrypt with the same context:
```bash
aws kms decrypt \
  --ciphertext-blob fileb://ciphertext.bin \
  --encryption-context App=payments,Environment=prod,TenantId=t-12345 \
  --output text \
  --query Plaintext
```

If the application uses the AWS SDK, pass the same context in both calls. For example, in Python with `boto3`:

```python
import boto3

kms = boto3.client("kms")

context = {
    "App": "payments",
    "Environment": "prod",
    "TenantId": "t-12345"
}

enc = kms.encrypt(
    KeyId="alias/payments-key",
    Plaintext=b"super-secret",
    EncryptionContext=context
)

dec = kms.decrypt(
    CiphertextBlob=enc["CiphertextBlob"],
    EncryptionContext=context
)

print(dec["Plaintext"].decode())
```

**Restricting decrypt with IAM and key policy**

A common pattern is to allow decrypt only for a specific role and only when the encryption context matches expected values. Remember that the IAM policy alone is not enough unless the key policy also permits the principal or allows IAM policies to control the key.

Example IAM policy statement:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowDecryptForPaymentsProd",
      "Effect": "Allow",
      "Action": "kms:Decrypt",
      "Resource": "arn:aws:kms:us-east-1:111122223333:key/12345678-1234-1234-1234-123456789012",
      "Condition": {
        "StringEquals": {
          "kms:EncryptionContext:App": "payments",
          "kms:EncryptionContext:Environment": "prod"
        }
      }
    }
  ]
}
```

This policy is only part of the solution. You still need the KMS key policy to allow the role or to delegate to IAM. A safer key policy pattern is to explicitly allow a limited admin role and enable IAM permissions for the key, then use IAM to scope decrypt access.

Example key policy fragment:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "EnableIAMPermissionsForKey",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:root"
      },
      "Action": "kms:*",
      "Resource": "*"
    }
  ]
}
```

That pattern delegates authorization to IAM, but you should still keep IAM permissions narrow and use conditions where possible.

**Using encryption context in authorization conditions**

KMS supports condition keys that let you require specific encryption context values. The most useful pattern is to match exact key-value pairs with `kms:EncryptionContext:<key>`.

Example:
```json
{
  "Effect": "Allow",
  "Action": "kms:Decrypt",
  "Resource": "*",
  "Condition": {
    "StringEquals": {
      "kms:EncryptionContext:App": "payments",
      "kms:EncryptionContext:Environment": "prod"
    }
  }
}
```

This is stronger than checking only that context keys exist. Requiring the presence of keys does not prove the values are correct, so it is not enough by itself for meaningful isolation.

Useful condition patterns:
- `StringEquals` on `kms:EncryptionContext:<key>` for exact matching
- `StringLike` when you need controlled pattern matching
- `kms:ViaService` to restrict use through a specific AWS service
- `aws:PrincipalArn` or role scoping to limit who can call KMS

Avoid relying on:
- Only `kms:EncryptionContextKeys`
- Only key presence without value checks
- Context alone without a restrictive principal and key policy

**What decrypt matching really means**

When you decrypt, the context you supply must match the context associated with the ciphertext for the keys that were used. In practice, this means:
- The same named context entries used at encryption must be present at decryption
- The values must match
- Extra keys are not a free pass; if they were not part of the original context, the request fails

This is not just a string comparison exercise. KMS uses the context as part of the cryptographic binding and authorization evaluation.

**CloudTrail visibility and auditing**

CloudTrail records KMS API activity, which is useful for investigating decrypt attempts and verifying context usage. You can often see request parameters in the event record, including encryption context values for KMS calls, depending on the API and event details.

Use CloudTrail to:
- Detect unexpected `kms:Decrypt` calls
- Confirm which principal used a key
- Review encryption context values for troubleshooting
- Correlate application activity with KMS usage

Example CloudTrail event fields you may inspect:
- `eventName`
- `userIdentity`
- `requestParameters`
- `resources`
- `errorCode`
- `sourceIPAddress`

A typical investigation flow is:
1. Find the decrypt event in CloudTrail
2. Check the caller identity and role session
3. Review the request parameters and context values
4. Compare them to the expected application or tenant
5. Confirm whether the key policy and IAM policy should have allowed the call

**Real-world design patterns**

**1. Application-level tenant isolation**

Use one KMS key for a service, but require tenant-specific encryption context values.

Example:
- `App=orders`
- `TenantId=tenant-a`
- `Environment=prod`

This lets you:
- Keep one operational key per service
- Prevent accidental cross-tenant decryption
- Improve auditability

**2. Service-bound decryption**

Restrict decrypt so only a specific AWS service path can use the key.

Example use cases:
- Amazon S3 server-side encryption with KMS
- AWS Lambda reading encrypted environment variables
- Amazon EBS volume encryption

In these cases, combine:
- `kms:ViaService`
- A narrow principal
- Context conditions where supported by the application flow

**3. Data classification enforcement**

Tag ciphertext usage with context such as:
- `Classification=confidential`
- `Owner=finance`

Then require those values in the decrypt policy. This helps prevent a role from using the key for unrelated data.

**Troubleshooting decrypt failures**

Common causes of decrypt failure include:
- **Missing context**: The application did not supply the required encryption context
- **Context mismatch**: One or more key-value pairs differ from the original encryption request
- **Authorization gap**: IAM allows the action, but the key policy does not
- **Wrong principal**: The caller role is not the one permitted by policy
- **Endpoint/key mismatch**: The client is calling the wrong regional KMS endpoint for the key being used
- **Disabled or pending-deletion key**: The key is not available for decrypt operations

A practical CLI check:
```bash
aws kms decrypt \
  --ciphertext-blob fileb://ciphertext.bin \
  --encryption-context App=payments,Environment=prod \
  --debug
```

Use `--debug` carefully in nonproduction environments because it can expose request details in logs.

**Policy design tips**

- Keep the decrypt principal as narrow as possible
- Prefer exact context matches over broad patterns
- Use separate keys when isolation requirements are strong
- Treat encryption context as a control that complements, not replaces, IAM and key policy
- Log and alert on unexpected decrypt activity
- Test both success and failure paths in a nonproduction account

A strong policy design often looks like this:
- Key policy enables IAM permissions for the key
- IAM policy allows `kms:Decrypt` only for a specific role
- Condition requires `kms:EncryptionContext:App` and `kms:EncryptionContext:Environment`
- Application always supplies the same context on encrypt and decrypt

**Quick comparison**

| Control | What it does | Best use |
|---|---|---|
| Key policy | Governs who can use the key and whether IAM can be used | Baseline authorization |
| IAM policy | Narrows which principals can call `kms:Decrypt` | Least privilege |
| Encryption context | Binds ciphertext to application-specific metadata | Tenant/app isolation and auditability |
| CloudTrail | Records KMS API activity | Monitoring and forensics |

**Exam focus points**

- `kms:Decrypt` is highly sensitive and should be tightly scoped
- IAM permission alone is not enough unless the key policy allows it
- Encryption context must be supplied consistently for encrypt and decrypt
- Use `kms:EncryptionContext:<key>` for exact value checks
- `kms:EncryptionContextKeys` by itself does not prove the values match
- CloudTrail is the place to inspect KMS API activity and request parameters
- Decrypt issues often come from policy mismatch, missing context, or the wrong KMS endpoint for the key

**Mental model**

Think of KMS decrypt as a three-part check:
- **Identity**: Who are you?
- **Authorization**: Are you allowed to use this key?
- **Context**: Is this ciphertext meant for this application, tenant, or purpose?

If any part fails, decryption fails. That is the core security value of combining restricted decrypt permissions with encryption context.
