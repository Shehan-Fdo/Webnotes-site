## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Why policy failures happen**

AWS authorization is usually an intersection of multiple controls, not a single yes/no switch. You often need to reason about identity-based policies, resource-based policies, permissions boundaries, session policies, service control policies (SCPs), and service-specific controls such as KMS key policies or S3 Block Public Access.

A request is allowed only when the effective evaluation path permits it. In practice, the most common exam traps are:

- **Implicit deny** when no applicable Allow exists.
- **Explicit deny** from any applicable policy layer.
- **Missing delegation** between IAM and a service-specific policy, especially with KMS.
- **Condition mismatch** caused by the wrong context key, wrong value, or wrong request path.
- **Service-specific behavior** that differs from the generic IAM mental model.

**Core evaluation model**

AWS evaluates permissions by combining policy types and service rules. The exact path varies by service, but the exam usually expects you to identify which layer blocks access.

- **Identity-based policies** attach to users, groups, and roles.
- **Resource-based policies** attach to resources such as S3 buckets, KMS keys, Lambda functions, SNS topics, SQS queues, and some Secrets Manager secrets.
- **Permissions boundaries** cap the maximum permissions an IAM principal can receive from identity-based policies and, for roles, also affect the effective permissions when the role is assumed.
- **Session policies** further restrict permissions for an assumed role session.
- **SCPs** set the maximum permissions available in member accounts under AWS Organizations.
- **Service-side controls** include settings such as S3 Block Public Access, KMS grants, VPC endpoint policies, and service-specific authorization rules.

A useful mental model is:

```text
Effective permission = identity policy ∩ permissions boundary ∩ session policy ∩ SCP ∩ service-specific rules
```

That model is simplified, but it helps you spot the layer that is most likely causing the denial.

**Common evaluation mistakes**

Many exam questions hinge on misunderstanding how statements combine.

- **Statements do not merge into a single logical expression.**
  - Each statement is evaluated independently.
  - An Allow in one statement does not “fix” a Deny in another statement.
- **Explicit Deny always wins.**
  - This includes Deny in identity policies, resource policies, SCPs, permissions boundaries, and session policies where applicable.
- **Conditions are statement-scoped.**
  - You cannot assume conditions from one statement combine with another statement to create a broader rule.
- **Implicit deny is the default.**
  - If no applicable Allow exists, the request fails even if nothing explicitly denies it.

Example of overlapping statements:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowReadOnly",
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::example-bucket/*"
    },
    {
      "Sid": "DenyOutsideCorpNetwork",
      "Effect": "Deny",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::example-bucket/*",
      "Condition": {
        "NotIpAddress": {
          "aws:SourceIp": "203.0.113.0/24"
        }
      }
    }
  ]
}
```

Even though the first statement allows access, the second statement denies it outside the allowed network.

**Identity-based policy mistakes**

Identity policies are the most common place to start troubleshooting, but they are rarely the whole story.

- **Wrong resource ARN**
  - A policy may allow `s3:GetObject` on the bucket ARN instead of the object ARN.
  - Example: `arn:aws:s3:::my-bucket/*` is needed for objects, while `arn:aws:s3:::my-bucket` is the bucket itself.
- **Wrong action scope**
  - Some actions require companion actions.
  - Example: listing a bucket may require `s3:ListBucket`, while reading an object requires `s3:GetObject`.
- **Missing dependent permissions**
  - Some services call other APIs behind the scenes.
  - Example: a console workflow may need `ec2:Describe*` permissions in addition to a write action.
- **Region or account mismatch**
  - The resource ARN may point to the wrong Region or account.
- **Policy variable errors**
  - Variables such as `${aws:username}` or `${aws:PrincipalTag/Department}` must resolve correctly at request time.
  - If the variable is missing or expands unexpectedly, the request may be denied or become too broad.

Example using policy variables:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "HomeFolderAccess",
      "Effect": "Allow",
      "Action": ["s3:GetObject", "s3:PutObject"],
      "Resource": "arn:aws:s3:::company-home/${aws:username}/*"
    }
  ]
}
```

Common mistakes with variables:

- **Missing context**
  - If the principal is a role session, `${aws:username}` may not behave as expected because the session is not a named IAM user.
- **Tag mismatch**
  - `${aws:PrincipalTag/Department}` only works if the principal session or role has that tag.
- **Case sensitivity**
  - Tag keys and values may not match the condition exactly.
- **Overbroad fallback**
  - A poorly designed variable pattern can accidentally grant access to the wrong path if the variable resolves to an empty or unexpected value.

**Resource-based policy nuances**

Resource policies are powerful, but their behavior is service-specific.

- **Some services allow resource policies to grant access directly**
  - This is common for cross-account access and for service principals.
- **Some services still require identity-based permission**
  - A resource policy alone is not universally sufficient.
  - The service may require both the caller’s identity policy and the resource policy.
- **KMS is a special case**
  - IAM policies can authorize KMS actions only when the key policy delegates to IAM in that account.
  - Without that delegation, an IAM Allow alone is not enough.
- **Lambda same-account vs cross-account**
  - Same-account invocation often works with identity-based permission alone.
  - Cross-account invocation commonly requires a function resource policy that allows the external principal.
- **S3 bucket policies**
  - Bucket policies can grant or deny access, but modern S3 ownership settings and public access controls can override expectations.

Example of a resource policy granting cross-account access:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCrossAccountRead",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/AnalyticsRole"
      },
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::data-bucket/reports/*"
    }
  ]
}
```

Use this pattern when the resource owner wants to grant access without creating a matching identity policy in the other account, but remember that service-specific rules may still apply.

**S3 failure modes**

S3 is a frequent source of exam questions because multiple controls can block access at once.

- **Bucket policy vs object ACL**
  - Legacy ACLs can still matter in some configurations.
  - However, modern best practice is to use **Object Ownership: Bucket owner enforced**, which disables ACLs for access control.
- **Block Public Access**
  - Account-level or bucket-level Block Public Access can override public bucket policy patterns.
  - A bucket policy that appears to allow public access may still fail if public access is blocked.
- **Object ownership**
  - If another account uploaded the object and ACLs are in use, ownership and ACL permissions can affect access.
  - With Bucket owner enforced, the bucket owner owns all objects and ACLs are disabled for authorization.
- **Prefix and object ARN mistakes**
  - A policy may allow the bucket but not the object path.
- **Missing `s3:ListBucket`**
  - Console browsing often needs bucket listing in addition to object read permissions.

Example of a bucket policy that is blocked by public access settings:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "PublicRead",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::example-public-site/*"
    }
  ]
}
```

If Block Public Access is enabled, this policy may not produce the expected public access.

**Permissions boundaries and session policies**

Permissions boundaries and session policies are common exam traps because they restrict permissions in different ways.

- **Permissions boundary**
  - Sets the maximum permissions a principal can receive from identity-based policies.
  - For a role, the boundary also constrains the permissions the role session can effectively use after assumption.
  - It does not grant permissions by itself.
- **Session policy**
  - Further restricts the permissions of an assumed role session.
  - It is useful when you want temporary, narrower access than the role normally allows.
- **Intersection behavior**
  - The effective permissions are the overlap of the role’s identity policy, the permissions boundary, and the session policy.

Example assumed-role scenario:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/DeployRole \
  --role-session-name ci-build \
  --policy '{"Version":"2012-10-17","Statement":[{"Effect":"Allow","Action":"s3:PutObject","Resource":"arn:aws:s3:::artifact-bucket/*"}]}'
```

In this case:

- The role must already allow `s3:PutObject`.
- The permissions boundary must also allow it.
- The session policy must allow it.
- Any SCP or service-side restriction must still permit it.

Common mistakes:

- **Assuming the session policy adds permissions**
  - It only narrows the session.
- **Assuming the boundary is ignored after role assumption**
  - It still matters for the role’s effective permissions.
- **Forgetting the boundary on the role itself**
  - A role can appear correctly configured in its identity policy but still fail because the boundary blocks the action.

**SCP and Organizations mistakes**

SCPs are often misunderstood because they do not grant permissions; they only limit them.

- **SCPs apply to accounts in an organization**
  - They affect principals in member accounts.
- **SCPs do not apply to the management account**
  - The management account is not restricted by SCPs in the same way.
- **SCPs do not grant access**
  - An IAM Allow still needs to exist unless a service-specific resource policy path allows access.
- **Service-linked roles**
  - In member accounts, SCPs can still affect actions taken by service-linked roles if the action is in scope.
  - In the management account, SCPs do not apply.

Example SCP deny pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyLeavingOrg",
      "Effect": "Deny",
      "Action": "organizations:LeaveOrganization",
      "Resource": "*"
    }
  ]
}
```

If a user in a member account cannot perform an action, check whether an SCP is denying it before changing the IAM policy.

**KMS-specific failure modes**

AWS Key Management Service (KMS) is one of the most exam-relevant services because its authorization model is stricter than many others.

- **Key policy is central**
  - The key policy controls who can administer the key and who can use it.
  - IAM policies can authorize KMS use only if the key policy delegates that authority to IAM in the account.
- **IAM alone is not enough**
  - If the key policy does not allow the principal or delegate to IAM, the request fails even when the IAM policy allows it.
- **Grants matter**
  - Some AWS services use KMS grants to perform cryptographic operations on your behalf.
  - A grant can allow a service to use the key without broadening the key policy.
- **Service integration failures**
  - If a service needs a grant and it is missing, encryption or decryption may fail even though the IAM policy looks correct.
- **Cross-account use**
  - Cross-account KMS access usually requires careful alignment of key policy, IAM policy, and sometimes grants.

Example of a key policy delegation pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "EnableIAMPoliciesForAccount",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:root"
      },
      "Action": "kms:*",
      "Resource": "*"
    }
  ]
}
```

This pattern delegates to IAM in the account. Without a delegation statement like this, IAM permissions may not be sufficient.

Common KMS failure symptoms:

- **`AccessDeniedException` on encrypt/decrypt**
  - Check the key policy first.
- **Service cannot use the key**
  - Check for a missing grant or missing service principal permission.
- **Cross-account access fails**
  - Verify both the caller’s IAM policy and the key policy, plus any required grant.

**Condition key mistakes**

Condition keys are a frequent source of subtle denials because the request context must match exactly.

- **`aws:SourceArn`**
  - Commonly used to prevent confused deputy problems.
  - Must match the calling resource ARN exactly enough for the service integration.
- **`aws:SourceAccount`**
  - Often paired with `aws:SourceArn` to restrict which account can invoke a service integration.
- **`aws:PrincipalOrgID`**
  - Useful for allowing only principals from your AWS Organization.
- **`aws:RequestedRegion`**
  - Useful for region restriction policies.
  - Be careful: some services call global endpoints or use service-specific regions.
- **`aws:SecureTransport`**
  - Commonly used to require HTTPS.
  - A deny on `false` is a standard pattern for S3 and other services that support it.

Example of a confused deputy protection pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowSNSPublishFromSpecificTopic",
      "Effect": "Allow",
      "Principal": {
        "Service": "events.amazonaws.com"
      },
      "Action": "sns:Publish",
      "Resource": "arn:aws:sns:us-east-1:123456789012:alerts",
      "Condition": {
        "ArnEquals": {
          "aws:SourceArn": "arn:aws:events:us-east-1:123456789012:rule/HighSeverity"
        },
        "StringEquals": {
          "aws:SourceAccount": "123456789012"
        }
      }
    }
  ]
}
```

Common mistakes:

- **Using the wrong ARN format**
  - The service may send a rule ARN, topic ARN, or function ARN, not the resource you expected.
- **Forgetting the account condition**
  - `aws:SourceArn` alone may not be enough in some designs.
- **Using a region-specific key incorrectly**
  - A global service may not populate the key the way you expect.
- **Assuming conditions across statements combine**
  - They do not.

**Policy variables and tag-based access control**

Policy variables are powerful for scalable access control, but they are easy to misuse.

- **`aws:username`**
  - Works for IAM users, but not always for assumed roles or federated sessions.
- **`aws:PrincipalTag/<TagKey>`**
  - Reads tags on the principal session or role.
  - Useful for attribute-based access control (ABAC).
- **`aws:ResourceTag/<TagKey>`**
  - Lets you compare principal attributes to resource attributes.
- **`aws:RequestTag/<TagKey>`**
  - Used when creating or tagging resources.

Example ABAC policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DepartmentMatch",
      "Effect": "Allow",
      "Action": "ec2:StartInstances",
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "aws:PrincipalTag/Department": "${aws:ResourceTag/Department}"
        }
      }
    }
  ]
}
```

Common mistakes:

- **Missing tags on the principal**
  - The condition fails if the principal session lacks the expected tag.
- **Missing tags on the resource**
  - The comparison fails if the resource is not tagged.
- **Wrong case or spelling**
  - Tag keys are often treated as exact matches in policy logic.
- **Using `aws:username` for role sessions**
  - Role sessions do not behave like IAM users.

**Policy simulator and troubleshooting limits**

The IAM policy simulator is useful, but it is not a complete substitute for real service behavior.

- **What it does well**
  - Evaluates many identity-based policies.
  - Evaluates permissions boundaries and session policies in many cases.
  - Helps you test explicit and implicit deny logic.
- **What it does not fully reproduce**
  - All service-specific authorization paths.
  - Some SCP and Organizations edge cases.
  - Endpoint policy behavior and network-path controls.
  - Service-side checks such as KMS grants, S3 Block Public Access, or other control-plane nuances.

Use the simulator to narrow the problem, then validate the service-specific layer if the result still does not match reality.

**Practical troubleshooting workflow**

When a request fails, work from the most likely blocking layer outward.

1. **Identify the exact API action**
   - Confirm the action name, resource ARN, Region, and principal.
2. **Check for explicit deny**
   - Review identity policies, resource policies, SCPs, permissions boundaries, and session policies.
3. **Verify the resource ARN**
   - Make sure the policy covers the exact resource type and path.
4. **Check conditions**
   - Validate `aws:SourceArn`, `aws:SourceAccount`, `aws:PrincipalOrgID`, `aws:RequestedRegion`, and transport conditions.
5. **Inspect service-specific controls**
   - For S3, check Block Public Access and ownership settings.
   - For KMS, check key policy and grants.
   - For VPC endpoints, check endpoint policy.
6. **Test with the policy simulator**
   - Use it to confirm the IAM-side logic, then validate the service-side path separately.

**Decision table for common symptoms**

| Symptom | Likely cause | What to check first |
|---|---|---|
| `AccessDenied` despite an Allow in IAM | Explicit deny or missing service-side permission | SCP, permissions boundary, resource policy, KMS key policy |
| S3 public access still blocked | Block Public Access | Account-level and bucket-level BPA settings |
| Cross-account Lambda invoke fails | Missing function resource policy | `lambda:AddPermission` and principal ARN |
| KMS decrypt fails with correct IAM policy | Key policy does not delegate or grant is missing | Key policy, grants, caller account |
| Role works in console but not via STS | Session policy or boundary restriction | Assumed-role session, session policy, boundary |
| Access works in one Region but not another | Region condition or service endpoint mismatch | `aws:RequestedRegion`, regional resource ARN |
| Policy looks correct but tag-based access fails | Missing or mismatched tags | `aws:PrincipalTag`, `aws:ResourceTag`, `aws:RequestTag` |

**CLI commands for investigation**

Use AWS CLI to inspect policies and simulate access.

```bash
aws iam get-user-policy --user-name Alice --policy-name HomeFolderPolicy
aws iam get-role-policy --role-name AppRole --policy-name AppPolicy
aws iam get-policy-version --policy-arn arn:aws:iam::123456789012:policy/BoundaryPolicy --version-id v1
aws organizations describe-effective-policy --policy-type SERVICE_CONTROL_POLICY --target-id ou-abc1-example
```

Simulate a request:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/AppRole \
  --action-names s3:GetObject \
  --resource-arns arn:aws:s3:::example-bucket/private/report.pdf
```

For KMS, inspect the key policy and grants:

```bash
aws kms get-key-policy \
  --key-id arn:aws:kms:us-east-1:123456789012:key/abcd-1234 \
  --policy-name default

aws kms list-grants \
  --key-id arn:aws:kms:us-east-1:123456789012:key/abcd-1234
```

**Exam patterns to remember**

- **A policy Allow is not enough if another layer denies it.**
- **Resource policies can enable cross-account access, but service rules still matter.**
- **KMS requires special attention to key policy delegation and grants.**
- **S3 access often fails because of ownership, ACL legacy behavior, or Block Public Access.**
- **Policy variables and condition keys are common sources of subtle denials.**
- **The policy simulator helps, but it does not replace service-specific validation.**

**Fast mental checklist**

Before choosing an answer, ask:

- Is there an explicit Deny anywhere?
- Is the resource ARN correct?
- Is the request in the right account and Region?
- Does the service require a resource policy, a grant, or both?
- Are conditions using the right context keys?
- Could Block Public Access, SCPs, a boundary, or a session policy be narrowing the request?
- Is the behavior service-specific rather than generic IAM behavior?

**Summary**

Most policy failures are not caused by a single missing Allow. They come from mismatched layers, service-specific authorization rules, or conditions that do not evaluate the way you expect. For SCS-C02, focus on the interaction between IAM, resource policies, SCPs, permissions boundaries, session policies, and service controls such as KMS key policies and S3 Block Public Access. If you can identify the blocking layer quickly, you can answer both troubleshooting and design questions with confidence.
