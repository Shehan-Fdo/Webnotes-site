## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Policy validation with the AWS CLI**

AWS Identity and Access Management (IAM) policy simulation helps you predict whether a request is allowed before you test it in a live environment. For exam prep, focus on what the simulator can and cannot evaluate: it is strongest for identity-based policy logic, condition keys, and effective permissions analysis, but it does not fully reproduce every service-specific authorization path.

Use the simulator when you want to answer questions like:

- Would this principal be allowed to call `s3:GetObject` on this ARN?
- Does a condition on `aws:PrincipalTag` or `aws:MultiFactorAuthPresent` change the result?
- Which statement or policy type caused the deny?

The main CLI operations are:

- `simulate-custom-policy` for testing a standalone policy document
- `simulate-principal-policy` for testing the effective permissions of a principal

**Core mental model**

IAM evaluates requests by combining policy types and context. The simulator approximates that evaluation for IAM policy logic, but you must supply the right inputs. If you omit a relevant resource ARN, context key, or policy source, the result can be incomplete or misleading.

**Key evaluation outcomes**

The simulator returns decision values in enum-style form, commonly including:

- `allowed`
- `explicitDeny`
- `implicitDeny`

A deny can come from an explicit `Deny` statement or from the absence of an applicable `Allow`. In exam questions, distinguish these carefully because the troubleshooting path differs.

**`simulate-custom-policy`**

Use `simulate-custom-policy` when you want to test a policy document that is not yet attached to a principal. This is useful for drafting a new identity policy, validating a permissions boundary statement, or checking how a condition behaves in isolation.

Typical use cases:

- Validate a new inline or managed policy before deployment
- Test a policy fragment copied from a design document
- Check how condition keys affect access decisions

Example:

```bash
aws iam simulate-custom-policy \
  --policy-input-list file://policy.json \
  --action-names s3:GetObject s3:PutObject \
  --resource-arns arn:aws:s3:::example-bucket/* \
  --context-entries ContextKeyName=aws:PrincipalTag/Department,ContextKeyType=string,ContextKeyValues=Finance
```

Important points:

- `--policy-input-list` accepts one or more policy documents.
- `--action-names` lists the API actions you want to test.
- `--resource-arns` supplies the target resource context for resource-scoped actions.
- `--context-entries` lets you simulate condition keys such as tags, MFA state, source IP, or session attributes.

If a policy uses variables like `${aws:username}` or `${aws:PrincipalTag/Team}`, you must provide matching context values. Otherwise, the simulator may return a deny because the variable cannot resolve as expected.

**`simulate-principal-policy`**

Use `simulate-principal-policy` when you want to evaluate the effective permissions of a real IAM principal, such as a user or role. This is the more exam-relevant command when the question asks, “What can this identity do right now?”

Typical use cases:

- Validate the permissions of an IAM user or role
- Check the effect of attached managed and inline policies
- Test how a permissions boundary changes the final decision
- Confirm whether a session context or tag-based condition blocks access

Example:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/AppRole \
  --action-names ec2:DescribeInstances s3:ListBucket \
  --resource-arns arn:aws:s3:::example-bucket \
  --context-entries ContextKeyName=aws:MultiFactorAuthPresent,ContextKeyType=boolean,ContextKeyValues=true
```

Important points:

- `--policy-source-arn` identifies the principal whose policies you want to evaluate.
- The simulator can use the principal’s attached identity policies, but the result still depends on the supplied context and resource ARNs.
- If the request depends on tags, session attributes, or other condition keys, you must provide them explicitly.
- For role sessions, be careful about which identity you are simulating: the role itself and the assumed-role session can differ in effective context.

> The simulator does not magically infer every real-world authorization input. If the policy depends on tags, boundaries, or session context, you must model those inputs deliberately.

**Common CLI options and when to use them**

| Option | Used with | Purpose | Exam note |
|---|---|---|---|
| `--policy-input-list` | `simulate-custom-policy` | Supplies one or more policy documents | Best for testing a draft policy in isolation |
| `--policy-source-arn` | `simulate-principal-policy` | Identifies the IAM user or role to evaluate | Use this for effective-permissions questions |
| `--action-names` | both | Lists the API actions to simulate | Multiple actions can reveal allow/deny differences |
| `--resource-arns` | both | Supplies resource context | Required for resource-scoped actions and conditions |
| `--context-entries` | both | Adds condition key values | Critical for MFA, tags, source IP, and session context |
| `--resource-owner` | both | Helps when the resource belongs to another account | Useful for cross-account scenarios, but not a full end-to-end authorization test |
| `--caller-arn` | both | Models the calling principal in some scenarios | Helpful when the request context differs from the policy source |
| `--resource-handling-option` | both | Controls how resource ARNs are interpreted | Use carefully; it affects how the simulator treats resource matching |
| `--permissions-boundary-policy-input-list` | custom simulation | Adds a boundary policy to the evaluation model | Important for understanding the final effective permissions |
| `--context-key-names` | both | Requests which context keys to include in output | Useful for troubleshooting condition-based decisions |

**Permissions boundaries**

A permissions boundary sets the maximum permissions an identity can receive, even if an identity policy allows more. In exam scenarios, think of it as a ceiling on identity-based permissions.

Practical interpretation:

- Identity policy allows an action
- Permissions boundary must also allow it
- If the boundary denies or omits the action, the final result is not allowed

Example boundary concept:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "BoundaryAllowReadOnlyS3",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": "*"
    }
  ]
}
```

If the identity policy allows `s3:PutObject` but the boundary only allows read actions, the effective decision is denied. The simulator is useful here, but only if you include the boundary in the simulation inputs or evaluate the principal in a way that reflects it.

**Conditions, tags, and session context**

Many IAM exam questions hinge on condition keys. The simulator is especially useful when a policy depends on values that are easy to overlook.

Common patterns include:

- `aws:MultiFactorAuthPresent` for MFA-based access control
- `aws:PrincipalTag/<TagKey>` for attribute-based access control
- `aws:RequestTag/<TagKey>` for tag-on-create workflows
- `aws:TagKeys` for restricting which tags can be set
- `aws:SourceIp` for IP-based restrictions
- `aws:RequestedRegion` for region-restricted access, when the service supports it

Example with a tag condition:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowOnlyFinanceTaggedPrincipals",
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::example-bucket/*",
      "Condition": {
        "StringEquals": {
          "aws:PrincipalTag/Department": "Finance"
        }
      }
    }
  ]
}
```

To test this, you must supply the matching context entry:

```bash
aws iam simulate-custom-policy \
  --policy-input-list file://policy.json \
  --action-names s3:GetObject \
  --resource-arns arn:aws:s3:::example-bucket/report.csv \
  --context-entries ContextKeyName=aws:PrincipalTag/Department,ContextKeyType=string,ContextKeyValues=Finance
```

If the tag is missing or different, the simulator should show a deny. That is often the fastest way to confirm whether a condition is the real blocker.

**Interpreting decision details**

The simulator can return additional detail fields that help explain why a request was allowed or denied. These fields are more useful than the top-level decision alone.

Look for:

- Which policy type contributed to the result
- Whether a statement matched the action and resource
- Whether a condition evaluated to true or false
- Whether the deny was explicit or implicit

Be careful with output shape:

- `MatchedStatements` may include source policy information and statement metadata, but do not assume it always maps to a simple statement ID in every case.
- `EvalDecisionDetails` can show how policy types contributed to the final result, but it is not a full service authorization trace.
- A deny in the simulator does not always mean the same thing as a service-side deny from a resource policy, trust policy, or service control policy.

**Practical troubleshooting workflow**

1. Identify the exact API action and resource ARN from the failed request.
2. Determine which policy type should allow it: identity policy, boundary, session policy, or resource policy.
3. Recreate the relevant context keys, such as MFA, tags, or source IP.
4. Run the simulator with the smallest useful set of actions and resources.
5. Inspect the decision and the matched details.
6. Adjust the policy or context and rerun until the result matches your expectation.

Example troubleshooting case:

A developer cannot upload to `s3://example-bucket/uploads/` even though the role policy appears to allow `s3:PutObject`.

Policy fragment:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowUploadsFromTaggedRole",
      "Effect": "Allow",
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::example-bucket/uploads/*",
      "Condition": {
        "StringEquals": {
          "aws:PrincipalTag/Team": "App"
        }
      }
    }
  ]
}
```

Simulation:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/AppRole \
  --action-names s3:PutObject \
  --resource-arns arn:aws:s3:::example-bucket/uploads/test.txt \
  --context-entries ContextKeyName=aws:PrincipalTag/Team,ContextKeyType=string,ContextKeyValues=Ops \
  --output json
```

Likely diagnosis:

- The action and resource match.
- The condition fails because the principal tag is `Ops`, not `App`.
- The final decision is `implicitDeny` unless another statement explicitly denies it.

This is a strong exam pattern: the policy looks correct at a glance, but the context key is wrong.

**Resource-based policy and cross-account caveats**

IAM simulation is not a full end-to-end authorization engine for every resource policy scenario. It is best at identity-based policy evaluation and condition logic. For cross-account access, you often need to reason about both sides manually:

- The caller’s identity policy must allow the action
- The target resource policy must allow the principal or account
- Any boundary, session policy, SCP (service control policy), or service-specific rule can still block access

Example scenario:

- Account A role tries to read an S3 object in Account B
- The role policy allows `s3:GetObject`
- The bucket policy in Account B must also allow the role or its account
- The simulator can help validate the caller-side policy logic, but it does not fully replace testing the bucket policy path in the service

For exam questions, do not overstate the simulator’s ability to validate trust policies, KMS key policies, or every resource-based authorization flow end to end.

**Output handling and JMESPath**

For troubleshooting, you often want only the decision and the matching statement details. Use `--output json` and a JMESPath query to reduce noise.

Example:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/AppRole \
  --action-names s3:GetObject \
  --resource-arns arn:aws:s3:::example-bucket/uploads/test.txt \
  --output json \
  --query 'EvaluationResults[0].{ActionName:EvalActionName,Decision:EvalDecision,Resource:EvalResourceName,Matched:MatchedStatements}'
```

This is useful when you need to compare several actions quickly or when you want to script policy checks in a CI/CD pipeline.

**Service-specific edge cases**

Some authorization decisions differ from what IAM simulation can model perfectly. Be cautious with:

- Services that apply additional authorization logic beyond IAM policy evaluation
- Resource policies that depend on service-specific principals or conditions
- Trust policy behavior for role assumption
- KMS and other services where key policy, grants, and IAM policy interact in service-specific ways
- SCPs and organization-level controls, which are not the same as identity policy simulation

If a lab or exam question involves a service-specific deny, use the simulator as a clue, not as the final authority.

**Quick comparison**

| Command | Best for | Strength | Limitation |
|---|---|---|---|
| `simulate-custom-policy` | Testing a draft policy document | Fast validation of policy logic and conditions | Does not represent a real principal unless you model context carefully |
| `simulate-principal-policy` | Checking a user or role’s effective permissions | Best for “what can this identity do?” questions | Requires correct source ARN, context, and resource inputs |

**Exam tips**

- Read the question carefully to identify whether you are validating a draft policy or an existing principal.
- Match the simulator command to the problem:
  - Draft policy: `simulate-custom-policy`
  - Existing identity: `simulate-principal-policy`
- Always include the relevant context keys when conditions are present.
- Treat `explicitDeny` as a hard stop.
- Treat `implicitDeny` as “no applicable allow was found.”
- Do not assume the simulator fully validates cross-account, trust policy, or service-specific authorization behavior.

**Practical command patterns**

Validate a single action:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:user/Alice \
  --action-names ec2:DescribeInstances \
  --resource-arns arn:aws:ec2:us-east-1:123456789012:instance/i-0123456789abcdef0
```

Test multiple actions at once:

```bash
aws iam simulate-custom-policy \
  --policy-input-list file://policy.json \
  --action-names s3:ListBucket s3:GetObject s3:PutObject \
  --resource-arns arn:aws:s3:::example-bucket arn:aws:s3:::example-bucket/*
```

Check a condition that depends on MFA:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:user/Alice \
  --action-names iam:ChangePassword \
  --context-entries ContextKeyName=aws:MultiFactorAuthPresent,ContextKeyType=boolean,ContextKeyValues=true
```

**What to remember**

- `simulate-custom-policy` tests a policy document.
- `simulate-principal-policy` tests a principal’s effective permissions.
- Conditions, tags, and context keys are often the real reason for allow or deny.
- The simulator is powerful, but it is not a perfect substitute for service-side authorization behavior.
- For exam questions, focus on the exact policy type, the exact context, and the exact resource.
