## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Overview**

AWS Key Management Service (KMS) lets you separate who can manage a key from who can use it for cryptographic operations. This is a common security design pattern for enforcing <mark>separation of duties</mark>, reducing blast radius, and meeting compliance requirements.

In exam scenarios, you usually need to distinguish between:

- **Key administrators** who create, rotate, enable, disable, tag, and update policies for keys.
- **Key users** who encrypt, decrypt, sign, verify, generate data keys, or otherwise consume the key in applications and services.
- **Guardrail administrators** who control the surrounding IAM, Organizations, or SCP boundaries without directly using the key.

The main design challenge is that KMS authorization is layered. A request can be allowed only when the key policy, IAM policy, grants, and any organizational controls all align.

**Core authorization model**

KMS authorization is not based on IAM alone. A principal must be allowed by the key policy, and then IAM permissions can further refine access. For many keys, the default key policy includes the account root principal, which is commonly used to delegate administration through IAM.

A typical pattern is:

- The key policy gives the account root permission to administer the key.
- IAM policies then grant specific admin or usage permissions to roles and users.
- Grants provide narrow, temporary, or service-specific access when needed.

This pattern matters because it lets you keep the key policy small while still delegating control through IAM. It also supports separation of duties: one role can manage the key, while another role can only use it.

**Administration versus usage**

Key administration and key usage are different permission sets.

- **Administration** includes actions such as `kms:CreateKey`, `kms:DescribeKey`, `kms:EnableKey`, `kms:DisableKey`, `kms:ScheduleKeyDeletion`, `kms:CancelKeyDeletion`, `kms:PutKeyPolicy`, `kms:GetKeyPolicy`, `kms:TagResource`, and `kms:UntagResource`.
- **Usage** includes actions such as `kms:Encrypt`, `kms:Decrypt`, `kms:ReEncryptFrom`, `kms:ReEncryptTo`, `kms:GenerateDataKey`, `kms:GenerateDataKeyWithoutPlaintext`, `kms:Sign`, `kms:Verify`, `kms:GenerateMac`, and `kms:VerifyMac`.

Use the right action set for the key type and workload:

- **Symmetric keys** are commonly used for `Encrypt`, `Decrypt`, and data key generation.
- **Asymmetric keys** are used for `Sign` and `Verify`, or for public-key encryption workflows where supported.
- **HMAC keys** are used for `GenerateMac` and `VerifyMac`.

> `kms:CreateKey` is typically controlled at the account level rather than scoped to an existing key ARN, so do not assume every admin action is resource-scoped in the same way.

**Practical separation pattern**

A clean design uses two IAM roles:

- `KMSKeyAdminRole` for key lifecycle and policy management.
- `KMSKeyUsageRole` for application or operator cryptographic operations.

Example admin policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AdministerSpecificKey",
      "Effect": "Allow",
      "Action": [
        "kms:DescribeKey",
        "kms:EnableKey",
        "kms:DisableKey",
        "kms:ScheduleKeyDeletion",
        "kms:CancelKeyDeletion",
        "kms:PutKeyPolicy",
        "kms:GetKeyPolicy",
        "kms:TagResource",
        "kms:UntagResource"
      ],
      "Resource": "arn:aws:kms:us-east-1:111122223333:key/12345678-1234-1234-1234-123456789012"
    },
    {
      "Sid": "CreateKeysInAccount",
      "Effect": "Allow",
      "Action": "kms:CreateKey",
      "Resource": "*"
    }
  ]
}
```

Example usage policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "UseKeyForAppCrypto",
      "Effect": "Allow",
      "Action": [
        "kms:Encrypt",
        "kms:Decrypt",
        "kms:GenerateDataKey",
        "kms:GenerateDataKeyWithoutPlaintext"
      ],
      "Resource": "arn:aws:kms:us-east-1:111122223333:key/12345678-1234-1234-1234-123456789012"
    }
  ]
}
```

**Key policy design**

The key policy is the most important control for KMS. It defines who can administer the key and who can delegate access through IAM. For exam purposes, remember these patterns:

- A key policy can directly allow a principal to use or administer the key.
- A key policy can allow the account root to delegate permissions via IAM.
- A key policy can restrict usage to specific services, accounts, or contexts.
- A key policy can deny broad access by omission, but explicit denies in IAM, SCPs, or permission boundaries can add stronger guardrails.

A common secure pattern is:

1. Allow the account root principal in the key policy.
2. Use IAM roles for admin and usage separation.
3. Add explicit conditions to narrow usage.
4. Use SCPs or permission boundaries to prevent privilege escalation.

Example key policy skeleton:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "EnableIAMPermissionsForAccount",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:root"
      },
      "Action": "kms:*",
      "Resource": "*"
    },
    {
      "Sid": "AllowAppRoleUseOfKey",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/KMSKeyUsageRole"
      },
      "Action": [
        "kms:Encrypt",
        "kms:Decrypt",
        "kms:GenerateDataKey",
        "kms:GenerateDataKeyWithoutPlaintext"
      ],
      "Resource": "*"
    }
  ]
}
```

**Grants**

KMS grants provide another way to delegate key usage without editing the key policy every time. They are especially important when AWS services need to use a key on your behalf.

Use grants when:

- An AWS service needs temporary or scoped access to a key.
- You want to avoid broad key policy changes.
- You need to delegate usage to a principal with limited permissions.

Grants can be constrained with conditions such as:

- **Encryption context** to bind usage to a specific application or dataset.
- **Retiring principal** to control who can retire the grant.
- **Operations list** to limit the exact cryptographic actions allowed.

Example grant creation with the AWS CLI:

```bash
aws kms create-grant \
  --key-id arn:aws:kms:us-east-1:111122223333:key/12345678-1234-1234-1234-123456789012 \
  --grantee-principal arn:aws:iam::111122223333:role/AppRole \
  --operations Encrypt Decrypt GenerateDataKey \
  --constraints EncryptionContextEquals=AppName=Payments \
  --name payments-app-grant
```

Grants are often the right answer when a service integration needs access but you do not want to expand the key policy for every workload.

**Encryption context**

Encryption context is a set of nonsecret key-value pairs that KMS binds to cryptographic operations. It is not encrypted secret data; it is authenticated additional data. If the context does not match during decrypt, the operation fails.

Use encryption context to:

- Tie ciphertext to a specific application, tenant, or record type.
- Prevent accidental or unauthorized reuse of ciphertext in another context.
- Add an application-level control that complements IAM and key policy.

Example application usage:

```bash
aws kms encrypt \
  --key-id arn:aws:kms:us-east-1:111122223333:key/12345678-1234-1234-1234-123456789012 \
  --plaintext fileb://secret.txt \
  --encryption-context AppName=Payments,Environment=Prod
```

Then decrypt with the same context:

```bash
aws kms decrypt \
  --ciphertext-blob fileb://ciphertext.bin \
  --encryption-context AppName=Payments,Environment=Prod
```

If you use encryption context in a policy or grant, make sure the application always supplies the same values. This is a strong exam clue for restricting key usage to a specific workload.

**Service-integrated access and `kms:ViaService`**

`kms:ViaService` is a condition key that restricts KMS requests to those made through a specific AWS service. It does not apply to direct KMS API calls from a client or application. Use it when you want the key to be usable only through a service integration.

Common examples include:

- Amazon S3 using KMS for server-side encryption.
- Amazon EBS using KMS for volume encryption.
- Amazon RDS using KMS for database encryption.
- AWS Lambda or other services that integrate with KMS for secrets or environment data.

Example policy condition:

```json
{
  "Sid": "AllowUseOnlyViaS3InThisAccount",
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::111122223333:role/S3EncryptionRole"
  },
  "Action": [
    "kms:Encrypt",
    "kms:Decrypt",
    "kms:GenerateDataKey"
  ],
  "Resource": "*",
  "Condition": {
    "StringEquals": {
      "kms:ViaService": "s3.us-east-1.amazonaws.com",
      "kms:CallerAccount": "111122223333"
    }
  }
}
```

`kms:CallerAccount` is useful when you want to ensure the request originates from your account, especially in cross-account or service-integrated scenarios.

**Multi-Region keys**

Multi-Region KMS keys are logically related keys in different Regions. They are useful for disaster recovery, regional failover, and latency-sensitive workloads.

For separation of duties, remember:

- Each Region has its own key material instance and regional key ARN.
- Administration is still controlled per key and per Region.
- Usage permissions must be considered in each Region where the replica exists.
- Policies and grants may need to be replicated or managed consistently across Regions.

A common exam scenario is a workload that fails over from one Region to another. In that case, you must ensure the replica key in the secondary Region has the correct policy, grants, and IAM access for the failover role.

**AWS managed keys versus customer managed keys**

AWS managed keys are created and managed by AWS for a service. They are convenient, but they do not give you the same level of control over key policy design as customer managed keys.

Use customer managed keys when you need:

- Explicit separation of admin and usage roles.
- Custom key policy conditions.
- Fine-grained grants.
- Stronger audit and governance requirements.

AWS managed keys can still be used with IAM permissions for cryptographic operations, but they are not the best fit when you need customer-controlled separation of duties at the key-policy level.

**Guardrails with SCPs and permission boundaries**

Even if you separate admin and usage roles, you still need guardrails to prevent privilege escalation.

Use these controls:

- **Service control policies (SCPs)** to prevent accounts from creating overly permissive KMS policies or disabling logging controls.
- **Permission boundaries** to limit what a delegated admin role can grant itself or others.
- **Explicit denies** to block dangerous actions such as `kms:PutKeyPolicy` outside a controlled pipeline.

Example permission boundary idea:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyPolicyEscalationOutsideApprovedPath",
      "Effect": "Deny",
      "Action": [
        "kms:PutKeyPolicy",
        "kms:ScheduleKeyDeletion"
      ],
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:PrincipalTag/ApprovedKMSAdmin": "true"
        }
      }
    }
  ]
}
```

This kind of boundary is useful when you want a platform team to manage keys, but only through approved roles or automation.

**Operational examples**

A secure operational model often looks like this:

1. A security team owns the key policy and guardrails.
2. A platform team manages key lifecycle through an admin role.
3. An application role uses the key only for cryptographic operations.
4. AWS services receive access through grants or service conditions.
5. CloudTrail records KMS API activity for audit and detection.

Example CloudTrail event fields to watch:

- `eventSource: kms.amazonaws.com`
- `eventName: PutKeyPolicy`
- `eventName: DisableKey`
- `eventName: ScheduleKeyDeletion`
- `eventName: Decrypt`
- `userIdentity.arn`
- `requestParameters.encryptionContext`

These events help you detect misuse, policy drift, or unexpected decrypt activity.

**Exam decision points**

When you see a scenario, ask:

- Do you need to separate who manages the key from who uses it?
- Do you need service-only access through S3, EBS, RDS, or another AWS service?
- Do you need temporary access for a service or workload? Use a grant.
- Do you need application-level binding? Use encryption context.
- Do you need regional failover? Check multi-Region key policies and grants.
- Do you need to prevent admin escalation? Add SCPs or permission boundaries.

**Quick comparison**

| Control | Best for | Notes |
|---|---|---|
| Key policy | Core authorization | Required for KMS access decisions |
| IAM policy | Delegating permissions | Works with key policy to allow access |
| Grant | Temporary or service-specific access | Common for AWS service integrations |
| Encryption context | Application binding | Helps restrict decrypt to matching context |
| `kms:ViaService` | Service-mediated access | Does not apply to direct API calls |
| SCP / permission boundary | Guardrails | Prevents privilege escalation |

**Common pitfalls**

- Assuming IAM alone can authorize KMS access.
- Giving application roles `kms:*` instead of only the actions they need.
- Forgetting that many AWS services use grants behind the scenes.
- Using a broad key policy when a grant or condition would be safer.
- Omitting encryption context in the application after designing a policy around it.
- Forgetting to replicate access controls for multi-Region keys.
- Treating AWS managed keys as equivalent to customer managed keys for separation of duties.

**Practical CLI checks**

Use these commands to inspect and validate access:

```bash
aws kms describe-key --key-id arn:aws:kms:us-east-1:111122223333:key/12345678-1234-1234-1234-123456789012
aws kms get-key-policy --key-id arn:aws:kms:us-east-1:111122223333:key/12345678-1234-1234-1234-123456789012 --policy-name default
aws kms list-grants --key-id arn:aws:kms:us-east-1:111122223333:key/12345678-1234-1234-1234-123456789012
aws kms list-resource-tags --key-id arn:aws:kms:us-east-1:111122223333:key/12345678-1234-1234-1234-123456789012
```

Use `list-grants` to confirm service or application delegation, and `get-key-policy` to verify whether the account root or specific roles are allowed.

**Summary**

Separate key administration and key usage by combining a minimal key policy, distinct IAM roles, grants for service access, and guardrails such as SCPs or permission boundaries. Use encryption context and `kms:ViaService` when you need to bind usage to a specific application or AWS service. For exam questions, the key idea is not just who can call KMS, but how authorization is layered and how you prevent one role from becoming both the administrator and the consumer of the same key.
