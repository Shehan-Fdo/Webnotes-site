## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Scenario focus**

This topic is about protecting security telemetry and evidence so you can investigate incidents without weakening the security posture of the environment. In exam scenarios, you usually need to balance three goals:

- Preserve **integrity** so logs and evidence cannot be altered or deleted.
- Preserve **availability** so investigators can access data quickly during an incident.
- Preserve **confidentiality** so only approved responders can read sensitive records.

A strong answer usually combines centralized logging, immutable storage, tightly scoped access, and a clear separation between day-to-day operations and forensic access.

**Core design pattern**

A common AWS pattern is:

- Source accounts generate logs and findings.
- A centralized log archive account stores immutable copies.
- A security tooling account analyzes telemetry.
- A separate incident-response role provides time-bound forensic access.

This separation matters because the same people who operate workloads should not automatically be able to delete evidence or change retention settings.

**Centralized logging architecture**

Use centralized collection whenever possible. It reduces the chance that a compromised member account can tamper with its own evidence and makes retention and access control easier to manage.

Typical sources include:

- **AWS CloudTrail** management events and data events
- **Amazon CloudWatch Logs** from applications and systems
- **VPC Flow Logs** for network visibility
- **AWS Config** configuration history and snapshots
- **AWS Security Hub** findings
- **Amazon GuardDuty** findings
- **AWS Backup** recovery points
- **Amazon EBS** snapshots and **Amazon Machine Images (AMIs)** for host-level evidence

A practical architecture often looks like this:

```text
Member accounts
  ├─ CloudTrail / Config / GuardDuty / Security Hub
  ├─ CloudWatch Logs
  └─ VPC Flow Logs
        ↓
Central log archive account
  ├─ S3 bucket with versioning + Object Lock
  ├─ KMS key for encryption
  └─ restricted bucket policy
        ↓
Security / IR account
  ├─ Athena / Lake Formation / OpenSearch / SIEM
  └─ read-only forensic roles
```

**CloudTrail for audit and evidence**

CloudTrail is the primary audit trail for AWS API activity. For exam questions, distinguish between:

- **Account trails**: created in a single account and scoped to that account.
- **Organization trails**: created from the AWS Organizations management account or delegated administrator and applied across the organization.

Organization trails are the preferred answer when you need centralized, consistent logging across many accounts. They reduce configuration drift and make it harder for a member account to opt out of logging.

Important points:

- Log **management events** for control-plane actions such as `CreateUser`, `PutBucketPolicy`, or `StopLogging`.
- Log **data events** when you need object-level or function-level visibility, such as S3 object access or Lambda invocation activity.
- Deliver logs to a dedicated S3 bucket in a log archive account.
- Encrypt logs with KMS and restrict who can read or delete them.

For S3 data events, the selector is bucket-scoped. A common pattern is to specify the bucket ARN with the trailing slash for all objects in that bucket:

```json
{
  "DataResources": [
    {
      "Type": "AWS::S3::Object",
      "Values": [
        "arn:aws:s3:::sensitive-evidence-bucket/"
      ]
    }
  ]
}
```

Use data events selectively because they can be high volume and increase cost.

**Protecting CloudTrail delivery and storage**

CloudTrail logs are only useful if they are protected after delivery. In exam scenarios, look for these controls:

- **S3 versioning** to preserve prior object versions.
- **S3 Object Lock** for write-once-read-many style retention when immutability is required.
- **Block Public Access** on the log archive bucket.
- A restrictive bucket policy that denies public access and limits cross-account access.
- Separate permissions for log delivery, read access, and deletion.
- KMS key policy and grants that allow the logging service to encrypt objects without giving broad decrypt rights to humans.

A hardened bucket policy often denies deletion and public ACLs while allowing only the CloudTrail service and a small set of security roles to write or read:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyPublicAccess",
      "Effect": "Deny",
      "Principal": "*",
      "Action": [
        "s3:PutBucketAcl",
        "s3:PutObjectAcl"
      ],
      "Resource": [
        "arn:aws:s3:::central-log-archive",
        "arn:aws:s3:::central-log-archive/*"
      ],
      "Condition": {
        "StringEquals": {
          "s3:x-amz-acl": [
            "public-read",
            "public-read-write"
          ]
        }
      }
    },
    {
      "Sid": "DenyDeleteExceptSecurityAdmin",
      "Effect": "Deny",
      "Principal": "*",
      "Action": [
        "s3:DeleteObject",
        "s3:DeleteObjectVersion"
      ],
      "Resource": "arn:aws:s3:::central-log-archive/*",
      "Condition": {
        "StringNotLike": {
          "aws:PrincipalArn": "arn:aws:iam::123456789012:role/SecurityAdmin"
        }
      }
    }
  ]
}
```

For S3 log archives, also consider:

- **Cross-account access controls** that allow only the log delivery service and approved forensic roles.
- **S3 Access Points** if you need separate access paths for different teams.
- **S3 Object Ownership** set to bucket owner enforced to avoid ACL complexity.

**CloudWatch Logs protection**

CloudWatch Logs often contains application, Lambda, and system telemetry that becomes evidence during an incident. Protect it with:

- Log group **retention settings** so logs do not expire too soon.
- Tight IAM permissions on `logs:DeleteLogGroup`, `logs:DeleteLogStream`, and `logs:PutRetentionPolicy`.
- Subscription filters or export jobs to copy logs to S3 for long-term retention.
- Separate roles for operators and investigators.

A useful pattern is to keep short-to-medium retention in CloudWatch Logs for searchability, then archive to S3 for immutable retention:

```bash
aws logs put-retention-policy \
  --log-group-name /aws/lambda/prod-payments \
  --retention-in-days 90
```

For export to S3, use a controlled process and preserve the export metadata as part of the evidence record.

**VPC Flow Logs and network evidence**

VPC Flow Logs provide network metadata that helps reconstruct traffic patterns, lateral movement, and exfiltration attempts. They are especially useful when paired with CloudTrail and GuardDuty.

Exam-relevant considerations:

- Deliver flow logs to CloudWatch Logs or S3 depending on the analysis and retention model.
- Use centralized storage for long-term retention.
- Protect the destination with the same controls as other security logs.
- Remember that flow logs are metadata, not packet captures.

Because CLI and API options vary by destination and current feature set, focus on the design intent rather than memorizing one brittle command. The key exam idea is that flow logs should land in a controlled, centralized location with restricted write and read access.

**AWS Organizations, SCPs, and delegated administration**

When you manage multiple accounts, AWS Organizations is a major control plane for protecting security data.

Use **service control policies (SCPs)** to prevent member accounts from disabling logging or weakening archive protections. SCPs do not grant permissions; they set the maximum available permissions in member accounts.

Common SCP goals:

- Deny `cloudtrail:StopLogging`, `cloudtrail:DeleteTrail`, and `cloudtrail:UpdateTrail` outside the security account.
- Deny `s3:DeleteBucket`, `s3:PutBucketVersioning`, and `s3:PutBucketPublicAccessBlock` on the log archive bucket except from approved admin roles.
- Deny changes to KMS key policies that protect log archives.
- Deny disabling AWS Config, GuardDuty, or Security Hub in member accounts.

A simple SCP pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyCloudTrailTampering",
      "Effect": "Deny",
      "Action": [
        "cloudtrail:StopLogging",
        "cloudtrail:DeleteTrail",
        "cloudtrail:UpdateTrail"
      ],
      "Resource": "*"
    }
  ]
}
```

For CloudTrail, know the difference between:

- **Management account control** of organization trails.
- **Delegated administrator** patterns that let a security account manage organization-wide logging without using the management account for day-to-day operations.

This is a common exam theme: centralize control, but avoid overusing the management account.

**KMS for centralized logging and forensic access**

AWS Key Management Service (KMS) is often used to encrypt log archives and evidence stores. The tricky part is authorization: the service that writes the data, the service that reads the data, and the human investigator may all need different permissions.

Good practice:

- Use a dedicated customer managed KMS key for the log archive.
- Allow the AWS service principal that delivers logs to use the key.
- Restrict decrypt permissions to approved forensic roles.
- Use key policies and, where appropriate, grants for service access.
- Add conditions such as `kms:ViaService` and encryption context constraints when they fit the access path.

For S3-delivered logs, the service often needs to use KMS on behalf of the caller. A practical key policy usually needs to account for the S3 service path, not just a human role. A simplified example:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowS3UseOfKeyForLogArchive",
      "Effect": "Allow",
      "Principal": {
        "Service": "s3.amazonaws.com"
      },
      "Action": [
        "kms:Encrypt",
        "kms:GenerateDataKey"
      ],
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "kms:ViaService": "s3.us-east-1.amazonaws.com",
          "aws:SourceAccount": "123456789012"
        },
        "StringLike": {
          "kms:EncryptionContext:aws:s3:arn": "arn:aws:s3:::central-log-archive/*"
        }
      }
    },
    {
      "Sid": "AllowForensicReadOnlyDecrypt",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:role/ForensicReadOnly"
      },
      "Action": [
        "kms:Decrypt",
        "kms:DescribeKey"
      ],
      "Resource": "*"
    }
  ]
}
```

Use grants when you need temporary or service-specific access without broadening the key policy. In exam questions, the best answer often combines a restrictive key policy with tightly scoped grants.

**Forensic access model**

Do not give investigators the same permissions as administrators. Separate roles reduce the blast radius if an investigator workstation or session is compromised.

A good model includes:

- **Read-only forensic role** for evidence review.
- **Incident-response role** for containment actions such as isolating instances or revoking credentials.
- **Break-glass role** for emergency use with strong controls and monitoring.

Use these controls:

- Short session duration with `sts:AssumeRole`.
- MFA for privileged access when feasible.
- Permission boundaries for delegated responders.
- Session tags to record case ID, ticket number, or analyst identity.
- CloudTrail logging of all role assumptions and sensitive actions.

Example of assuming a forensic role with session tags:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/ForensicReadOnly \
  --role-session-name case-2026-0412 \
  --tags CaseId=INC-1042 Analyst=jsmith
```

Remember that session policies only further restrict the resulting session; they do not create permissions by themselves. Use them to narrow access for a specific investigation, not as a substitute for role design.

**Evidence handling and chain of custody**

Forensics is not just about storage. You also need a defensible workflow that preserves provenance and access history.

Preserve:

- **Timestamps** for collection, transfer, and review.
- **Source account** and region.
- **Object version IDs** for S3 evidence.
- **Checksum or hash values** for files and exports.
- **Access records** showing who viewed or copied the evidence.
- **Case identifiers** and analyst notes.

A practical chain-of-custody workflow might be:

1. Collect evidence from the source account into a controlled staging location.
2. Record the source, time, hash, and collector identity.
3. Copy the evidence into an immutable archive bucket.
4. Restrict read access to the forensic role.
5. Log every access and export action.
6. Preserve the case record in a separate system of record.

Example metadata record:

```json
{
  "caseId": "INC-1042",
  "sourceAccount": "111122223333",
  "sourceRegion": "us-east-1",
  "artifactType": "cloudtrail-log",
  "s3Bucket": "central-log-archive",
  "s3Key": "AWSLogs/111122223333/CloudTrail/us-east-1/2026/04/26/logfile.json.gz",
  "versionId": "3HL4kqtJlcpXroDTDmjVBH40Nrjfkd",
  "sha256": "b7f3c2f1e9c4d1f2a8f0c1d9e3a4b5c6d7e8f90123456789abcdef0123456789",
  "collectedBy": "arn:aws:iam::123456789012:role/ForensicReadOnly",
  "collectedAt": "2026-04-26T14:22:10Z"
}
```

**AWS Backup, EBS snapshots, and AMIs**

For host and workload forensics, logs are not enough. You may need point-in-time copies of storage and compute state.

Use:

- **EBS snapshots** to preserve disk state.
- **AMIs** to preserve instance configuration and launch state.
- **AWS Backup** to centralize backup policies, retention, and cross-account copy.

Exam scenarios often ask for evidence preservation after compromise. In those cases:

- Snapshot the volume before remediation if you need disk artifacts.
- Create an AMI if you need a reproducible instance image.
- Copy backups to a separate account or vault with restricted access.
- Protect backup vaults with vault lock or equivalent retention controls when appropriate.

**Security Hub and GuardDuty findings**

Security telemetry from **AWS Security Hub** and **Amazon GuardDuty** is often part of the evidence set. These findings can show:

- Suspicious API activity
- Unusual network behavior
- IAM credential misuse
- Malware or reconnaissance indicators

Protect findings by:

- Centralizing them in a security account.
- Exporting or streaming them to durable storage.
- Restricting who can archive, suppress, or delete findings.
- Preserving the original finding timestamps and IDs.

In incident response, findings are often the first clue, but they should be corroborated with CloudTrail, VPC Flow Logs, and workload evidence.

**Read-only forensic access vs incident-response access**

These are not the same role.

- **Read-only forensic access** is for investigation, evidence review, and reporting.
- **Incident-response access** is for containment and recovery actions such as disabling keys, isolating instances, or rotating credentials.

Use separate roles because:

- Investigators should not accidentally change evidence.
- Responders may need broader permissions than analysts.
- Time-bound access reduces risk during and after the incident.

A practical split:

| Role | Purpose | Typical permissions |
|---|---|---|
| ForensicReadOnly | Review evidence | `s3:GetObject`, `logs:GetLogEvents`, `cloudtrail:LookupEvents`, `kms:Decrypt` on specific archive keys |
| IncidentResponse | Contain and recover | `ec2:StopInstances`, `iam:UpdateAccessKey`, `ssm:SendCommand`, `secretsmanager:RotateSecret` |
| BreakGlassAdmin | Emergency escalation | Highly restricted, monitored, short-lived |

Add permission boundaries or SCPs so even privileged responders cannot alter the archive controls.

**Practical hardening checklist**

Use this checklist when you see a scenario about protecting security data:

- Centralize logs in a dedicated account.
- Enable organization-wide logging where possible.
- Turn on S3 versioning and Object Lock for immutable archives.
- Block public access on all log buckets.
- Use restrictive bucket policies and cross-account controls.
- Encrypt logs with customer managed KMS keys.
- Allow service principals to write, not humans.
- Separate forensic read access from incident-response access.
- Use SCPs to prevent disabling logging or weakening retention.
- Preserve evidence metadata, hashes, timestamps, and version IDs.
- Archive CloudWatch Logs, CloudTrail, Config, GuardDuty, Security Hub, and backup artifacts.

**Exam decision cues**

When the question asks for the best solution, look for these signals:

- If the goal is organization-wide visibility, choose **organization trails** and centralized logging.
- If the goal is immutability, choose **S3 Object Lock** and versioning.
- If the goal is to prevent member accounts from tampering, choose **SCPs** plus centralized control.
- If the goal is controlled investigation, choose **separate forensic roles** with short sessions.
- If the goal is evidence preservation from workloads, include **snapshots**, **AMIs**, or **AWS Backup** in addition to logs.
- If the goal is secure cross-account log encryption, include **KMS key policy or grants** that support the service delivery path.

**Common pitfalls**

- Giving investigators full admin access to the log archive.
- Storing logs only in the source account.
- Forgetting to protect the archive bucket with Block Public Access and restrictive policies.
- Using KMS without considering the service path that writes the data.
- Treating session policies as a standalone permission grant.
- Relying on one telemetry source instead of correlating logs, findings, and snapshots.
- Ignoring the need to preserve evidence metadata and access history.

**Practical example: secure log archive bucket posture**

```bash
aws s3api put-public-access-block \
  --bucket central-log-archive \
  --public-access-block-configuration \
  BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true

aws s3api put-bucket-versioning \
  --bucket central-log-archive \
  --versioning-configuration Status=Enabled
```

Use these controls as part of a broader design, not as isolated fixes. The exam usually rewards answers that combine preventive controls, detective controls, and evidence-preservation controls into one coherent architecture.
