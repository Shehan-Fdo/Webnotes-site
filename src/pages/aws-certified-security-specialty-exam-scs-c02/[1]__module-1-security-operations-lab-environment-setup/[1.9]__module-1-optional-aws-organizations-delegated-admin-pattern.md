## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Overview**

AWS Organizations delegated administration lets you assign a member account to manage a supported AWS service across the organization without using the management account for day-to-day operations. This pattern is common in security and governance because it reduces blast radius, improves separation of duties, and keeps the management account reserved for organization-level control.

The exam usually tests whether you can distinguish three related ideas:

- **Trusted access**: the organization allows a service to integrate with AWS Organizations.
- **Delegated administrator**: a member account is registered to administer that service for the organization.
- **Service-side admin setup**: some services still require you to enable or configure an organization admin resource inside the service itself before org-wide management works.

**Why this pattern exists**

The management account is highly privileged and should be tightly controlled. Delegated administration gives you a safer operating model:

- You keep the management account out of routine security operations.
- You centralize service administration in a dedicated security account.
- You can apply stricter controls to the delegated admin account than to general member accounts.
- You reduce the need to share broad cross-account IAM roles for every operational task.

This pattern is especially relevant for services such as AWS Security Hub, Amazon GuardDuty, AWS Config, IAM Access Analyzer, and similar organization-aware services. Support and behavior vary by service, so you should always verify the exact service model.

**Core terminology**

<dl>
<dt><strong>Management account</strong></dt>
<dd>The AWS Organizations account that creates and controls the organization.</dd>
<dt><strong>Delegated administrator</strong></dt>
<dd>A member account authorized to administer a supported service for the organization.</dd>
<dt><strong>Trusted access</strong></dt>
<dd>A permission relationship that allows a service to integrate with AWS Organizations.</dd>
<dt><strong>Service-linked role</strong></dt>
<dd>An IAM role created and managed by AWS for a service to perform actions in your account.</dd>

**How the setup lifecycle works**

The exact sequence matters for exam questions and troubleshooting.

1. **Enable trusted access** for the service in AWS Organizations.
2. **Register the delegated administrator** account for that service.
3. **Configure the service in the delegated account** if the service requires its own organization admin setting.
4. **Onboard member accounts** using the service’s organization features, auto-enable settings, or invitations, depending on the service.
5. **Verify Region scope** because many security services are Region-specific.

For teardown, reverse the process carefully:

1. Remove or disable service-specific organization admin settings.
2. Deregister the delegated administrator.
3. Disable trusted access if no longer needed.

> Revocation in Organizations does not always clean up service-side resources immediately. Existing detectors, hubs, aggregators, findings, or member relationships may need separate cleanup in the service.

**Service-specific behavior you should know**

Not every service uses delegated administration the same way. The exam often checks whether you understand the difference between a service that supports organization-wide delegated admin and a service that only supports trusted access or a narrower organization integration.

- **Security Hub**
  - Uses an organization admin account model.
  - The delegated admin account typically must be designated as the Security Hub administrator for the organization.
  - You then configure organization settings such as auto-enable for accounts and Regions.
  - Member accounts may be auto-enabled or invited depending on configuration and service behavior.

- **GuardDuty**
  - Uses a delegated administrator model plus organization configuration.
  - The delegated admin account manages detectors, organization auto-enable, and member enrollment for supported Regions.
  - Simply creating a detector is not enough to establish organization-wide delegated administration.

- **AWS Config**
  - Supports organization-aware features such as organization aggregators and conformance packs.
  - Its delegated administration and organization management capabilities are more feature-specific than some other security services.
  - Do not assume AWS Config behaves like a fully centralized security console for every feature.

- **IAM Access Analyzer**
  - Supports organization-level analysis in supported Regions.
  - The delegated admin account can manage analyzers and findings for the organization.
  - Region scope still matters.

- **AWS Audit Manager**
  - Supports delegated administration for organization-wide assessment management in supported contexts.
  - Verify the exact Region and feature support before assuming org-wide control.

- **AWS Detective**
  - Supports organization administration in supported Regions.
  - Often used alongside GuardDuty and Security Hub for investigation workflows.

**Exam distinction: delegated admin vs cross-account role assumption**

These are not the same pattern.

| Pattern | Purpose | Typical scope | Key limitation |
|---|---|---|---|
| Delegated admin | Centralize administration of a supported AWS service | Organization-wide, service-specific | Only works for supported services and Regions |
| Cross-account IAM role | Temporarily assume permissions in another account | Account/resource-specific | Requires explicit trust policy and permissions |
| Service-linked role | Let AWS service act in your account | Service-internal | Not a general admin access pattern |

Use delegated admin when the service natively supports organization management. Use cross-account roles when you need temporary operational access to resources in another account. Use service-linked roles when AWS itself needs to perform service actions on your behalf.

**Security architecture pattern**

A common security landing zone uses a dedicated security account as the delegated administrator for multiple services.

- Management account:
  - Creates the organization.
  - Enables trusted access.
  - Registers delegated admins.
  - Rarely used for daily security operations.

- Security account:
  - Acts as delegated admin for supported services.
  - Hosts security tooling, dashboards, and automation.
  - Is tightly controlled with MFA, SCPs, and limited human access.

- Member accounts:
  - Send findings, logs, or configuration data to the delegated service.
  - Remain under account-level ownership for workloads.

**Practical example: end-to-end organization setup**

The following example shows a realistic workflow for a security account that administers a supported service across the organization.

1. In the management account, enable trusted access for the service.
2. Register the security account as the delegated administrator.
3. In the security account, enable the service in each required Region.
4. Turn on organization auto-enable or equivalent org onboarding.
5. Verify that member accounts appear as managed members.
6. Confirm findings or configuration data flow into the delegated admin account.

Example AWS CLI commands for the Organizations side:

```bash
aws organizations enable-aws-service-access \
  --service-principal securityhub.amazonaws.com

aws organizations register-delegated-administrator \
  --account-id 111122223333 \
  --service-principal securityhub.amazonaws.com

aws organizations list-delegated-administrators \
  --service-principal securityhub.amazonaws.com
```

For teardown:

```bash
aws organizations deregister-delegated-administrator \
  --account-id 111122223333 \
  --service-principal securityhub.amazonaws.com

aws organizations disable-aws-service-access \
  --service-principal securityhub.amazonaws.com
```

**Service-side setup examples**

Security Hub organization administration typically requires service-side configuration in the delegated admin account.

```bash
aws securityhub enable-security-hub --region us-east-1
aws securityhub update-organization-configuration \
  --auto-enable true \
  --region us-east-1
aws securityhub create-members \
  --account-details AccountId=444455556666,Email=user@example.com \
  --region us-east-1
```

GuardDuty organization administration also requires service-side organization configuration in the delegated admin account.

```bash
aws guardduty create-detector --enable --region us-east-1
aws guardduty update-organization-configuration \
  --detector-id a1b2c3d4e5f6 \
  --auto-enable \
  --region us-east-1
```

AWS Config organization workflows are more feature-specific. A common pattern is to create an organization aggregator in the delegated admin account and then manage organization-wide visibility from there.

```bash
aws configservice put-configuration-aggregator \
  --configuration-aggregator-name org-aggregator \
  --organization-aggregation-source RoleArn=arn:aws:iam::111122223333:role/AWSConfigRole,AllAwsRegions=true
```

**Important caveats**

- **Region scope** matters for most security services.
  - Delegated admin registration may be global at the Organizations level, but the service itself often operates per Region.
  - You may need to repeat service-side setup in each Region.

- **Service support differs**
  - Some services support delegated admin.
  - Some support only trusted access.
  - Some support both but with different feature sets.
  - Do not assume one service’s workflow applies to another.

- **Management account control remains special**
  - The management account can register and deregister delegated administrators.
  - It can also disable trusted access.
  - That does not automatically remove all service artifacts or member associations.

- **SCPs can block delegated admin actions**
  - A service control policy can prevent the delegated admin from creating detectors, hubs, analyzers, or aggregators.
  - IAM permissions boundaries and identity policies can also block setup.
  - Troubleshoot both Organizations permissions and service permissions.

**Hardening the delegated admin account**

Treat the delegated admin account as a high-value security account.

- Require MFA for human access.
- Use least privilege IAM policies for operators.
- Restrict who can assume admin roles in the account.
- Apply SCPs to prevent accidental deletion of critical service resources.
- Monitor CloudTrail for Organizations and service API calls.
- Separate break-glass access from routine operations.

A useful control pattern is to allow only a small security operations group to manage the delegated admin account, while denying broad administrative actions in member accounts.

**Common troubleshooting cases**

- **Delegated admin registration fails**
  - Trusted access is not enabled.
  - The account is not a member account.
  - The service does not support delegated admin in that Region.
  - The caller lacks Organizations permissions.

- **Service admin setup fails after registration**
  - The service has not been enabled in the delegated account.
  - The Region is unsupported or not yet configured.
  - An SCP blocks the action.
  - The service requires a separate organization admin setting.

- **Member accounts do not appear**
  - Auto-enable is off.
  - The service requires invitations or explicit member creation.
  - The account is suspended, closed, or outside the organization.
  - The service has not been enabled in the target Region.

- **Findings or data do not flow**
  - The service is not enabled in the Region where the resource exists.
  - The delegated admin is configured, but member onboarding is incomplete.
  - A service-linked role or resource policy is missing.
  - The workload account has not generated the expected service events.

**How to reason about exam questions**

Look for clues in the wording:

- If the question asks who should manage a security service across many accounts, think **delegated admin**.
- If the question asks for temporary access to a single account’s resources, think **cross-account role**.
- If the question mentions the management account performing routine security operations, consider whether that is a poor design.
- If the question includes Region-specific behavior, remember that delegated admin is often not enough by itself; the service must also be configured in each Region.

**Quick comparison with related patterns**

- **Delegated admin**
  - Best for supported AWS services that need org-wide administration.
  - Centralizes service control in a dedicated account.

- **AWS Organizations trusted access**
  - Enables the service to integrate with the organization.
  - Often required before delegated admin registration.

- **Cross-account IAM role**
  - Best for ad hoc access to resources in another account.
  - Not a replacement for service-native organization management.

- **Service-linked role**
  - Best when AWS services need to act within an account.
  - Usually invisible to end users and not used for human administration.

**Practical mental model**

Think of delegated administration as a three-layer model:

1. **Organizations layer**: authorizes the service and the delegated admin account.
2. **Service layer**: enables the service’s organization features.
3. **Account layer**: controls who can operate inside the delegated admin account.

If any layer is missing, the pattern fails.

**Key takeaways**

- Delegated admin is an AWS Organizations pattern for supported services, not a universal admin model.
- Trusted access and delegated admin registration are separate steps.
- Many services still require service-side organization configuration in the delegated account.
- Region scope and service-specific behavior are common exam traps.
- The management account retains ultimate Organizations control, but service cleanup may still be required separately.
- Harden the delegated admin account because it becomes a high-value control plane for security services.
