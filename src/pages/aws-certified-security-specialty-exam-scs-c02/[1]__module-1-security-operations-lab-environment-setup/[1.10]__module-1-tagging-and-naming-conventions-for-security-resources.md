## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Why tagging and naming matter**

Tagging and naming are core governance controls in AWS. They help you identify ownership, classify data, enforce access control, automate lifecycle actions, and support cost allocation and incident response. For the Security Specialty exam, you should understand both the operational value and the IAM implications of tags.

A strong convention reduces ambiguity when you manage security-sensitive resources such as IAM roles, KMS keys, CloudTrail trails, log buckets, Security Hub standards, GuardDuty delegated admin settings, and security groups. Naming is especially important when a resource is hard to rename in place or when tags are not consistently supported across services.

**Tagging fundamentals**

Tags are key-value metadata attached to AWS resources. They are not security controls by themselves, but they become powerful when combined with IAM, AWS Organizations, automation, and inventory tooling.

Common uses include:

- **Ownership**: identify the team, application, or account responsible for a resource.
- **Environment**: distinguish `dev`, `test`, `stage`, and `prod`.
- **Data classification**: mark resources as `public`, `internal`, `confidential`, or `restricted`.
- **Lifecycle**: support cleanup, retention, and decommissioning workflows.
- **Audit and response**: quickly find resources during investigations.

Typical tag keys include:

- `Owner`
- `Application`
- `Environment`
- `DataClassification`
- `CostCenter`
- `ManagedBy`
- `Compliance`
- `Purpose`

A practical convention is to keep keys stable and values predictable. For example, use `Environment=prod` rather than mixing `Production`, `prod`, and `prd`.

**Naming conventions**

Naming is the human-readable identifier you use when tags are unavailable, incomplete, or not visible in a given workflow. Good names are consistent, searchable, and encode only the information you truly need.

A common pattern is:

- `<org>-<app>-<env>-<resource>-<region>-<sequence>`

Examples:

- `acme-payments-prod-kms-us-east-1-01`
- `acme-shared-prod-cloudtrail-global`
- `acme-secops-prod-logbucket-us-east-1`
- `acme-identity-dev-iamrole-readonly`

Use names to support:

- **Discovery** in the console and CLI
- **Uniqueness** where the service requires it
- **Operational clarity** for responders and auditors
- **Immutable resources** that cannot be renamed in place

> Naming is often more important than tagging for resources that are difficult or impossible to rename, such as KMS keys, IAM roles, and some logging or audit resources.

**Tagging versus naming**

| Aspect | Tagging | Naming |
|---|---|---|
| Purpose | Metadata and governance | Human-readable identity |
| Searchability | Strong with inventory tools | Strong in console and logs |
| Enforcement | Can be enforced with IAM and Organizations | Usually not enforceable directly |
| Flexibility | Easy to add or change | Often constrained by service rules |
| Best for | Classification, ownership, automation | Uniqueness, readability, immutable resources |

Use both when possible. Use naming as the fallback when tags are unsupported or not reliable for a workflow.

**AWS Organizations tag policies vs SCPs**

AWS Organizations provides two different governance mechanisms that are often confused:

- **Tag policies** define allowed tag keys, allowed values, and normalization guidance across accounts in an organization.
- **Service control policies (SCPs)** define the maximum permissions available to accounts and principals in those accounts.

Use them differently:

- Use **tag policies** to standardize tag structure and improve consistency.
- Use **SCPs** to prevent actions entirely, such as creating resources without required tags when the service and action support tag-based condition keys.
- Use both together when you need organization-wide consistency and enforcement.

Tag policies do not grant permissions. They help you detect and standardize tags, but they do not replace IAM or SCPs.

**Tag limits and constraints**

AWS services impose tag limits and naming constraints that you must design around. Exact limits vary by service, but common constraints include:

- A maximum number of tags per resource, often around 50.
- Key length and value length limits.
- Reserved prefixes such as `aws:` that you cannot use for customer-defined tags.
- Service-specific restrictions on allowed characters.
- Some resources support tags only at creation time or only through specific APIs.

Practical guidance:

- Keep keys short and consistent.
- Avoid encoding too much state in tags.
- Reserve a small set of mandatory keys for governance.
- Check service documentation before assuming a tag is supported.

**ABAC with tags**

<abbr title="Attribute-Based Access Control">ABAC</abbr> uses tags as attributes in IAM policy evaluation. This is especially useful when you want permissions to scale with teams, projects, or environments instead of maintaining large static role lists.

A common ABAC pattern is:

- Tag the principal, such as an IAM user or role, with `Department=Security`.
- Tag the target resource with the same or compatible value.
- Allow access only when the principal tag matches the resource tag.

Example IAM policy for read access based on matching tags:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowReadWhenDepartmentMatches",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "kms:Decrypt"
      ],
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "aws:PrincipalTag/Department": "${aws:ResourceTag/Department}"
        }
      }
    }
  ]
}
```

This pattern is useful when:

- You have many teams and frequent onboarding/offboarding.
- You want to reduce policy sprawl.
- You need dynamic access control tied to business attributes.

Be careful with ABAC:

- Not every service supports the same tag condition keys.
- Some actions evaluate tags only on specific resources.
- Cross-service access may require service-specific condition keys.

**Create-time tag enforcement**

For many services, you can require tags at resource creation using `aws:RequestTag` and `aws:TagKeys`. These condition keys evaluate the tags included in the request, but support is service- and action-specific.

A realistic pattern is to require approved keys and specific values during creation:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyCreateWithoutRequiredTags",
      "Effect": "Deny",
      "Action": [
        "ec2:RunInstances",
        "kms:CreateKey",
        "logs:CreateLogGroup"
      ],
      "Resource": "*",
      "Condition": {
        "Null": {
          "aws:RequestTag/Environment": "true",
          "aws:RequestTag/Owner": "true"
        }
      }
    },
    {
      "Sid": "DenyCreateWithUnapprovedTagKeys",
      "Effect": "Deny",
      "Action": [
        "ec2:RunInstances",
        "kms:CreateKey",
        "logs:CreateLogGroup"
      ],
      "Resource": "*",
      "Condition": {
        "ForAnyValue:StringNotEquals": {
          "aws:TagKeys": [
            "Environment",
            "Owner",
            "Application",
            "DataClassification"
          ]
        }
      }
    }
  ]
}
```

A safer exam-oriented takeaway is:

- Use `aws:RequestTag/<key>` to require specific tag values on supported create or tag actions.
- Use `aws:TagKeys` to control which keys are allowed.
- Verify the service supports the condition keys for the action you are targeting.

> Do not assume every create action supports request tags in the same way. Tag enforcement is highly service-specific.

A more explicit allow-based example for a controlled environment is:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCreateOnlyWithApprovedEnvironment",
      "Effect": "Allow",
      "Action": "kms:CreateKey",
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "aws:RequestTag/Environment": [
            "dev",
            "test",
            "prod"
          ]
        },
        "ForAllValues:StringEquals": {
          "aws:TagKeys": [
            "Environment",
            "Owner"
          ]
        }
      }
    }
  ]
}
```

Use this style when you want to allow only a narrow set of tags at creation time. In practice, many teams combine an allow statement with a broader deny guardrail in SCPs.

**Inspecting tags with AWS CLI**

The AWS CLI helps you audit tags across services. Use the service that best matches the resource type.

For EC2 instances, `describe-instances` is often more useful than `describe-tags` when you want to inspect a single instance and its tags:

```bash
aws ec2 describe-instances \
  --instance-ids i-0123456789abcdef0 \
  --query 'Reservations[].Instances[].Tags'
```

For a broader cross-service inventory, use the Resource Groups Tagging API:

```bash
aws resourcegroupstaggingapi get-resources \
  --tag-filters Key=Environment,Values=prod \
  --query 'ResourceTagMappingList[].ResourceARN'
```

You can also list tags for a specific resource when the service supports it:

```bash
aws kms list-resource-tags --key-id arn:aws:kms:us-east-1:123456789012:key/abcd-1234
aws logs list-tags-log-group --log-group-name /aws/lambda/my-function
aws iam list-role-tags --role-name acme-secops-prod-audit-role
```

For tagging or untagging, use the service-specific API or the tagging API for that service:

```bash
aws ec2 create-tags \
  --resources i-0123456789abcdef0 \
  --tags Key=Environment,Value=prod Key=Owner,Value=secops
```

```bash
aws resourcegroupstaggingapi tag-resources \
  --resource-arn-list arn:aws:ec2:us-east-1:123456789012:instance/i-0123456789abcdef0 \
  --tags Environment=prod Owner=secops
```

**Security resource naming examples**

Security teams often benefit from names that reveal purpose, scope, and environment.

Examples:

- **IAM roles**: `acme-prod-breakglass-admin`, `acme-dev-ci-deploy-role`
- **KMS keys**: `acme-prod-secrets-kms`, `acme-shared-logs-kms`
- **CloudTrail trails**: `acme-org-cloudtrail-global`, `acme-prod-data-events`
- **Log buckets**: `acme-secops-prod-cloudtrail-logs`, `acme-central-audit-logs`
- **Security Hub standards**: `acme-prod-foundations-standard`
- **GuardDuty delegated admin**: `acme-secops-guardduty-admin`
- **Security groups**: `acme-prod-alb-sg`, `acme-prod-bastion-sg`

Good names should:

- Avoid ambiguous abbreviations.
- Include environment when the resource is environment-specific.
- Use a stable prefix for the organization or platform.
- Distinguish shared security services from application-owned resources.

**Partial and special tagging behavior**

Tag support varies significantly across AWS services. Some services support tags broadly, while others support tags only on certain resource types or only at creation time.

Important patterns to remember:

- Some services support tagging on the primary resource but not on every child object.
- Some services support tag propagation only in specific workflows.
- Some resources are effectively immutable and must be replaced rather than renamed.
- Some security services expose tags for organization or account-level settings, while others tag only the underlying resource objects.

Examples of special behavior:

- **Auto Scaling** can propagate tags to launched instances when configured to do so.
- **CloudFormation** can propagate stack tags to supported resources, but not every resource behaves the same way.
- **EBS snapshots, AMIs, and copied resources** may require explicit tag handling during copy or creation.
- **Security Hub and GuardDuty** have service-specific tagging or delegated admin behaviors that do not always mirror standard EC2-style tagging.

The exam-relevant lesson is that you should not assume generic parent-child inheritance. Always verify whether the service supports explicit propagation, copy-on-create, or no propagation at all.

**Governance workflows**

A practical governance workflow usually combines naming, tagging, and policy enforcement:

1. Define a naming standard for each resource class.
2. Define mandatory tags and allowed values.
3. Enforce tags with IAM, SCPs, or service controls where supported.
4. Use AWS Organizations tag policies to standardize keys and values.
5. Inventory resources with the Resource Groups Tagging API.
6. Remediate drift with automation such as AWS Config, Lambda, or Systems Manager.

Example AWS Config rule use cases:

- Detect resources missing `Owner` or `Environment`.
- Flag resources with noncompliant tag values.
- Trigger remediation for untagged security resources.

Example remediation logic:

```bash
aws resourcegroupstaggingapi get-resources \
  --tag-filters Key=Environment,Values=prod Key=Owner,Values=secops \
  --resource-type-filters ec2:instance kms:key logs:log-group
```

Use service-specific resource type strings that the API supports. Do not assume a generic service name is valid for every resource type.

**Practical security design patterns**

For security-sensitive environments, use these patterns:

- Tag all audit and logging resources with `DataClassification=restricted`.
- Tag break-glass roles with `Purpose=EmergencyAccess`.
- Tag KMS keys with `Rotation=enabled` or `KeyUsage=encrypt-decrypt` when useful for inventory.
- Tag CloudTrail trails and log buckets with `Compliance=sox`, `Compliance=hipaa`, or your internal control mapping.
- Tag delegated admin resources with `ManagedBy=SecurityPlatform`.

For immutable or sensitive resources:

- Prefer a naming convention that encodes purpose and environment.
- Treat rename as replacement, not mutation.
- Plan for migration when a resource name must change.
- Keep tags as the mutable metadata layer, but do not depend on them as the only identifier.

**Common exam pitfalls**

- Assuming all services support the same tag keys and condition keys.
- Treating tag policies as an enforcement mechanism instead of a standardization tool.
- Using naming as a substitute for access control.
- Assuming tags automatically propagate from parent to child resources.
- Forgetting that some resources are easier to replace than rename.
- Overlooking service-specific limits on tag count, key length, and reserved prefixes.
- Using overly complex tag schemas that are hard to maintain.

**Quick reference**

- Use **tags** for ownership, classification, automation, and ABAC.
- Use **names** for readability, uniqueness, and immutable resources.
- Use **tag policies** to standardize.
- Use **SCPs** and IAM to enforce.
- Use **Resource Groups Tagging API** for cross-service inventory.
- Use **service-specific APIs** for precise tagging and inspection.

**Exam-oriented summary**

For SCS-C02, focus on how tagging and naming support governance, not just inventory. You should know when to use ABAC, how to require tags at creation time, why Organizations tag policies differ from SCPs, and why some security resources depend more on naming than tagging. The safest design is consistent naming plus mandatory tags plus policy enforcement, with service-specific exceptions handled explicitly.
