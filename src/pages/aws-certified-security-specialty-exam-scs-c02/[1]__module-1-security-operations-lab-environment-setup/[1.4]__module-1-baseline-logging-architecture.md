## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Baseline logging architecture: purpose and design goals**

A baseline logging architecture gives you a consistent, defensible way to collect, protect, retain, and analyze security-relevant activity across AWS accounts and Regions. For exam purposes, think in layers: source logs, transport, central storage, analysis, alerting, and retention. The architecture should support incident response, compliance, and forensic reconstruction without relying on a single service or a single account.

The main design goals are:

- **Coverage** across management, data, network, and application layers.
- **Centralization** so you can investigate from one place.
- **Immutability** so logs are hard to alter or delete.
- **Retention** aligned to legal, regulatory, and operational needs.
- **Searchability** for detection engineering and incident response.
- **Separation of duties** so log administrators are not the same as workload operators.

A practical baseline usually combines AWS-native logging services with a central security account and, when needed, a SIEM (Security Information and Event Management) platform.

**Core logging sources in AWS**

You should know what each service captures and why it matters.

- **AWS CloudTrail** records AWS API activity and console actions.
  - Management events show control-plane actions such as `CreateUser`, `PutBucketPolicy`, or `AuthorizeSecurityGroupIngress`.
  - Data events capture object-level or resource-level activity such as S3 object access or Lambda function invocation.
  - Insights can help detect unusual API call rates or error rates.
- **Amazon CloudWatch Logs** stores application, system, and service logs.
  - It is often the destination for EC2, Lambda, ECS, and custom application logs.
  - It supports metric filters, subscription filters, and log insights queries.
- **Amazon VPC Flow Logs** capture network flow metadata for VPCs, subnets, or ENIs.
  - They help identify unexpected connections, exfiltration patterns, and security group issues.
- **AWS Config** records configuration history and evaluates resource compliance.
  - It is essential for drift detection and compliance evidence.
- **Amazon S3 server access logs** provide request-level visibility for S3 buckets.
  - They are useful when you need access auditing beyond CloudTrail data events.
- **AWS Route 53 Resolver query logs** capture DNS query activity.
  - They are useful for threat hunting, domain analysis, and malware detection.
- **AWS Security Lake** centralizes security data in an Open Cybersecurity Schema Framework (OCSF) format for analytics and downstream tools.

> Use the right source for the question being asked: API activity, configuration drift, network metadata, DNS, or object access each answer different security questions.

**CloudTrail as the foundation**

CloudTrail is the primary audit trail for AWS account activity. In a baseline architecture, you usually enable it in every account and Region, then centralize delivery to a dedicated log archive account.

Key points:

- **Organization trails** apply to all accounts in an AWS Organizations organization.
- A trail created in the management account can capture activity across member accounts when configured as an organization trail.
- A delegated administrator can manage CloudTrail in some organization setups, but the management account still has special control-plane responsibilities.
- You should enable multi-Region trails so you do not miss activity in newly used Regions.
- You should log both management events and the data events that matter most for risk, such as S3 buckets containing sensitive data.

CloudTrail also supports:

- **Log file validation** to detect tampering.
- Delivery to **Amazon S3** for durable storage.
- Delivery to **CloudWatch Logs** for near-real-time metric filters and alarms.
- Integration with **Amazon EventBridge** for event-driven alerting and automation.
- **CloudTrail Lake** for querying and analyzing event data without building a separate ingestion pipeline.

A common exam pattern is choosing between CloudTrail Lake, S3-based archival, and CloudWatch Logs. Use:

- **S3** for long-term, low-cost retention and evidence.
- **CloudWatch Logs** for operational alerting and short-term search.
- **CloudTrail Lake** for ad hoc queries and investigations over event history.

**CloudTrail organization trail prerequisites**

For AWS Organizations, the baseline is not just “turn on CloudTrail.” You need the right organizational setup.

- Create or use an **organization trail** in the management account or delegated admin context.
- Ensure the trail is configured to include all accounts in the organization.
- Confirm that member accounts are not independently overriding the central logging strategy.
- Use a dedicated log archive account for S3 storage and a security account for analysis.
- Apply service control policies (SCPs) or guardrails if you need to prevent disabling logging or changing trail configuration.

A typical flow is:

1. Enable AWS Organizations and designate a management account.
2. Configure CloudTrail organization trails.
3. Centralize S3 delivery in a log archive account.
4. Restrict write/delete access to the log bucket.
5. Enable validation and monitoring.

**CloudTrail log integrity and validation**

CloudTrail log file validation helps you verify that delivered log files have not been altered after delivery. This is important for forensic integrity and compliance.

Practical considerations:

- Enable **log file validation** on trails that support it.
- Store logs in a bucket with strong access controls.
- Use **S3 Object Lock** only when you need WORM (Write Once, Read Many) retention and can enable it at bucket creation time with versioning.
- Use separate roles for log ingestion and log administration.
- Monitor for unauthorized changes to bucket policies, KMS keys, or trail settings.

A useful validation workflow is:

1. Deliver CloudTrail logs to S3.
2. Verify digest files and log file hashes.
3. Alert on missing digest files or unexpected delivery gaps.
4. Investigate any tampering indicators immediately.

**CloudTrail to CloudWatch Logs and EventBridge**

CloudTrail can send events to CloudWatch Logs for filtering and alerting. This is useful when you want near-real-time detection of sensitive API activity.

Common patterns:

- Create a CloudWatch Logs metric filter for events such as `ConsoleLogin`, `DeleteTrail`, `StopLogging`, or `PutBucketPolicy`.
- Create alarms on suspicious patterns.
- Use EventBridge rules for direct routing to Lambda, SNS, SSM Automation, or Security Hub.

Example metric filter:

```bash
aws logs put-metric-filter \
  --log-group-name /aws/cloudtrail/security \
  --filter-name RootLoginFilter \
  --filter-pattern '{ ($.eventName = ConsoleLogin) && ($.userIdentity.type = Root) }' \
  --metric-transformations metricName=RootConsoleLogin,metricNamespace=Security,metricValue=1
```

Example EventBridge rule pattern:

```json
{
  "source": ["aws.cloudtrail"],
  "detail-type": ["AWS API Call via CloudTrail"],
  "detail": {
    "eventSource": ["iam.amazonaws.com"],
    "eventName": ["CreateAccessKey", "UpdateLoginProfile", "AttachUserPolicy"]
  }
}
```

Use EventBridge when you want event-driven response, and CloudWatch Logs when you want log search plus metric-based alerting.

**CloudTrail Lake**

CloudTrail Lake is relevant when you need a managed event data store and SQL-like querying over CloudTrail events.

Use it when:

- You need faster investigation than scanning raw S3 logs.
- You want to query across large event volumes.
- You need a managed retention and query model for audit data.

Be aware that CloudTrail Lake is not a replacement for all archival needs. Many architectures still keep S3 as the long-term immutable store and use CloudTrail Lake for investigation and analytics.

**AWS Config for configuration history and compliance**

AWS Config records resource configuration changes and evaluates them against rules. It complements CloudTrail because CloudTrail tells you who did what, while Config tells you what changed and whether the resulting state is compliant.

Important baseline points:

- You must enable the **configuration recorder** in each account and Region you want to monitor.
- You must create or use a **delivery channel** so Config can deliver snapshots and configuration history to S3.
- The recorder does not start automatically after `put-configuration-recorder`; you also need `start-configuration-recorder`.
- Config rules can be managed or custom.
- Aggregators centralize compliance data across accounts and Regions.
- Conformance packs bundle related rules and remediation guidance.

Example baseline setup:

```bash
aws configservice put-configuration-recorder \
  --configuration-recorder name=default,roleARN=arn:aws:iam::123456789012:role/AWSConfigRole \
  --recording-group allSupported=true,includeGlobalResourceTypes=true

aws configservice put-delivery-channel \
  --delivery-channel name=default,s3BucketName=my-config-archive-bucket

aws configservice start-configuration-recorder \
  --configuration-recorder-name default
```

Why this matters:

- **Snapshots** help you reconstruct state at a point in time.
- **History** helps you identify drift and unauthorized changes.
- **Rules** help you detect noncompliance continuously.
- **Aggregators** help central security teams monitor many accounts from one place.

**AWS Config aggregators and conformance packs**

Aggregators are especially important in multi-account environments.

- Use an **organization aggregator** to collect configuration and compliance data from all accounts.
- Use **conformance packs** to deploy a standardized set of rules across accounts and Regions.
- Pair them with remediation actions when you want automated correction for common misconfigurations.

A common exam scenario is centralized compliance reporting:

- Workloads run in many accounts.
- Each account records Config locally.
- An aggregator in the security account collects the data.
- A conformance pack enforces baseline controls such as encryption, logging, and public access restrictions.

**Amazon CloudWatch Logs and centralization patterns**

CloudWatch Logs is often the operational log layer. It is not just a destination; it is also a routing and analysis hub.

Useful features:

- **Metric filters** for alerting on patterns.
- **Subscription filters** for streaming logs to Kinesis Data Firehose, Lambda, or a destination.
- **Log Insights** for interactive queries.
- **Retention policies** for cost control and compliance.

For cross-account centralization, you usually need one of these patterns:

- **Subscription to Kinesis Data Firehose** for delivery to S3, OpenSearch Service, or a SIEM.
- **Cross-account subscription destinations** with the required resource policy or destination policy.
- **Central log account** that receives logs from source accounts through authorized subscriptions.

Example subscription filter to a destination:

```bash
aws logs put-subscription-filter \
  --log-group-name /aws/lambda/app-prod \
  --filter-name ForwardAll \
  --filter-pattern "" \
  --destination-arn arn:aws:logs:us-east-1:111122223333:destination:CentralSecurityDestination
```

Operational prerequisites matter:

- The destination must already exist.
- The destination must allow the source account or log group to publish.
- The target path often uses Kinesis Data Firehose or Lambda for transformation and forwarding.

**CloudWatch Logs to SIEM or data lake**

A baseline architecture often forwards logs to a SIEM or security data lake for normalization and correlation.

Common pipeline:

- CloudWatch Logs subscription filter
- Kinesis Data Firehose or Lambda transform
- S3 landing zone or SIEM endpoint
- Parsing and normalization
- Detection rules and dashboards

Why this is useful:

- You can normalize heterogeneous logs into a common schema.
- You can correlate CloudTrail, VPC Flow Logs, DNS logs, and application logs.
- You can retain raw logs in S3 while indexing a subset in a search platform.

**VPC Flow Logs**

VPC Flow Logs provide network metadata, not packet payloads. They are useful for detecting suspicious traffic patterns, failed connections, and unexpected exposure.

You can enable them at:

- **VPC level**
- **Subnet level**
- **Network interface level**

Destinations include:

- CloudWatch Logs
- Amazon S3

Example to create flow logs to CloudWatch Logs:

```bash
aws ec2 create-flow-logs \
  --resource-type VPC \
  --resource-ids vpc-0123456789abcdef0 \
  --traffic-type ALL \
  --log-destination-type cloud-watch-logs \
  --log-group-name /aws/vpc/flowlogs/prod \
  --deliver-logs-permission-arn arn:aws:iam::123456789012:role/FlowLogsDeliveryRole
```

Operational requirements:

- The log group should exist or be created as part of your workflow.
- The IAM role needs the correct trust relationship for the VPC Flow Logs service.
- The role needs permissions to write to the log group.
- For cross-account or centralized delivery, plan the permissions carefully.

Use Flow Logs to answer questions like:

- Which instances are talking to the internet?
- Are there rejected connections to sensitive subnets?
- Is a workload scanning internal ranges?
- Is traffic leaving a subnet unexpectedly?

**Amazon S3 logging and access visibility**

S3 is a common log archive target, but S3 itself also needs access visibility.

Relevant controls:

- **CloudTrail data events** for object-level API activity.
- **S3 server access logging** for request-level access records.
- **S3 access points** to simplify and segment access to shared buckets.
- **Bucket policies** and **IAM** to control who can read or write logs.

Use S3 server access logging when you need request metadata such as requester, bucket, key, and response status. Use CloudTrail data events when you need AWS API-level auditability and identity context.

Example lifecycle policy for log retention:

```json
{
  "Rules": [
    {
      "ID": "ArchiveAndExpireLogs",
      "Status": "Enabled",
      "Filter": { "Prefix": "cloudtrail/" },
      "Transitions": [
        {
          "Days": 30,
          "StorageClass": "STANDARD_IA"
        },
        {
          "Days": 90,
          "StorageClass": "GLACIER_IR"
        },
        {
          "Days": 180,
          "StorageClass": "DEEP_ARCHIVE"
        }
      ],
      "Expiration": {
        "Days": 2555
      }
    }
  ]
}
```

Notes on archive classes:

- `GLACIER_IR` means Glacier Instant Retrieval.
- `DEEP_ARCHIVE` is for very long-term retention.
- Avoid ambiguous use of `GLACIER` when you want to be precise about archive behavior.

**Object Lock and immutability**

Object Lock can help protect log archives from deletion or modification, but it has constraints.

- You must enable **versioning**.
- You must enable Object Lock at **bucket creation time**.
- It supports retention modes such as **governance** and **compliance**.
- It is useful for regulated log archives and forensic evidence.

Do not treat Object Lock as a universal default. It is powerful, but it adds operational complexity and requires careful planning for retention, legal hold, and access control.

**Route 53 Resolver query logging**

DNS logs are valuable for threat detection because many attacks rely on domain resolution.

Baseline considerations:

- Route 53 Resolver query logs can be sent to supported destinations such as CloudWatch Logs or S3.
- The destination must be properly configured and authorized.
- If you use CloudWatch Logs, ensure the log group policy and permissions allow delivery.
- If you use S3, ensure the bucket policy allows the service to write logs.

Use DNS logs to detect:

- Command-and-control lookups
- Domain generation algorithm behavior
- Unexpected external resolution
- Internal host discovery

**AWS Security Lake**

Security Lake is increasingly relevant in baseline logging architectures because it centralizes security data in a normalized format.

Why it matters:

- It reduces custom ingestion work.
- It supports multi-account and multi-Region collection.
- It helps downstream analytics tools consume standardized security data.
- It can ingest AWS-native and third-party security sources.

Use Security Lake when you want a broader security data lake strategy rather than only service-specific log archives.

**Retention strategy by log type**

Retention should match the value and volume of each log type.

Typical examples:

- **CloudTrail management events**: 90 days to several years, depending on compliance.
- **CloudTrail data events**: shorter retention may be acceptable due to volume and cost, but sensitive workloads may require longer.
- **AWS Config history**: often retained for audit and compliance periods.
- **VPC Flow Logs**: often 30 to 180 days for investigations, longer if required by policy.
- **Application logs**: based on operational and legal needs.
- **Security investigation logs**: retain longer when they support incident response or legal hold.

Factors that influence retention:

- Regulatory requirements
- Contractual obligations
- Incident response needs
- Storage cost
- Query frequency
- Data sensitivity

A common architecture uses hot, warm, and cold tiers:

- **Hot**: CloudWatch Logs or SIEM index for recent investigation.
- **Warm**: S3 Standard-IA or searchable data lake.
- **Cold**: Glacier Instant Retrieval or Deep Archive for long-term retention.

**KMS encryption and key policy considerations**

Logs should be encrypted at rest, usually with AWS Key Management Service (KMS).

Important points:

- Use customer managed keys when you need tighter control and auditability.
- Ensure the key policy allows the relevant AWS service principal to use the key.
- Separate log archive key administration from workload administration.
- Consider key rotation for long-lived archives.
- Review grants and policies so log delivery does not fail.

Common service principals include:

- `cloudtrail.amazonaws.com`
- `logs.amazonaws.com`
- `config.amazonaws.com`
- `delivery.logs.amazonaws.com`

A key policy should support:

- Encryption by the service
- Decryption by authorized analysts
- Administration by a limited security team
- Denial of broad workload access to archived logs

**Practical baseline architecture pattern**

A strong baseline logging architecture often looks like this:

- **Workload accounts**
  - CloudTrail enabled organization-wide
  - AWS Config enabled in every Region
  - VPC Flow Logs enabled for critical VPCs
  - Application logs sent to CloudWatch Logs
- **Security/log archive account**
  - Central S3 bucket for CloudTrail and Config archives
  - Object Lock or strict bucket policies where required
  - KMS keys dedicated to log encryption
- **Security operations account**
  - CloudWatch Logs insights and alarms
  - EventBridge rules for alerting
  - SIEM or Security Lake integration
  - AWS Config aggregators and conformance packs

This separation helps you preserve evidence, reduce blast radius, and simplify audit ownership.

**Example reference topology**

```text
[Workload Accounts] ---> CloudTrail / Config / Flow Logs / App Logs
         |                         |
         |                         v
         |------------------> [Central Log Archive Account]
                                   |
                                   +--> S3 immutable archive
                                   +--> CloudTrail Lake / Security Lake
                                   +--> SIEM / analytics
                                   +--> CloudWatch alarms / EventBridge
```

**Operational checklist**

Use this checklist to validate a baseline logging design:

1. Enable CloudTrail in all accounts and Regions.
2. Centralize CloudTrail delivery to a dedicated log archive account.
3. Turn on log file validation.
4. Enable AWS Config recorder and delivery channel in each account and Region.
5. Configure Config aggregators and conformance packs.
6. Enable VPC Flow Logs for critical networks.
7. Forward application logs to CloudWatch Logs and set retention.
8. Define CloudWatch Logs subscription filters or Firehose pipelines for centralization.
9. Protect log buckets with least privilege, encryption, and retention controls.
10. Decide whether Object Lock, Security Lake, or a SIEM is required for your compliance model.
11. Test alerting with EventBridge and CloudWatch metric filters.
12. Verify that log delivery, retention, and access review processes work during an incident.

**Common exam distinctions**

- **CloudTrail vs AWS Config**
  - CloudTrail records API activity.
  - Config records resource configuration state and compliance.
- **CloudWatch Logs vs S3**
  - CloudWatch Logs is better for near-real-time analysis.
  - S3 is better for durable, low-cost archival.
- **Flow Logs vs packet capture**
  - Flow Logs show metadata, not payloads.
  - They are not a replacement for deep packet inspection.
- **CloudTrail Lake vs S3 archive**
  - CloudTrail Lake is query-oriented.
  - S3 archive is storage-oriented.
- **Security Lake vs service-specific logs**
  - Security Lake centralizes and normalizes security data.
  - Service-specific logs remain useful as source-of-truth records.

**Practical command examples**

Enable a CloudTrail trail to S3:

```bash
aws cloudtrail create-trail \
  --name org-security-trail \
  --s3-bucket-name my-cloudtrail-archive-bucket \
  --is-multi-region-trail

aws cloudtrail start-logging \
  --name org-security-trail
```

Create an EventBridge rule for root activity:

```bash
aws events put-rule \
  --name DetectRootActivity \
  --event-pattern '{
    "source": ["aws.cloudtrail"],
    "detail-type": ["AWS API Call via CloudTrail"],
    "detail": {
      "userIdentity": {
        "type": ["Root"]
      }
    }
  }'
```

Set CloudWatch Logs retention:

```bash
aws logs put-retention-policy \
  --log-group-name /aws/lambda/app-prod \
  --retention-in-days 90
```

**What to remember for SCS-C02**

For the exam, focus on architecture decisions rather than memorizing one service in isolation.

- Use CloudTrail for account activity auditing.
- Use AWS Config for configuration drift and compliance.
- Use CloudWatch Logs for operational visibility and alerting.
- Use VPC Flow Logs for network metadata.
- Use S3 for durable archive and evidence.
- Use EventBridge and CloudWatch alarms for near-real-time detection.
- Use CloudTrail Lake or Security Lake when the question asks about centralized query and analytics.
- Use KMS, bucket policies, Object Lock, and SCPs to protect the logging pipeline itself.
