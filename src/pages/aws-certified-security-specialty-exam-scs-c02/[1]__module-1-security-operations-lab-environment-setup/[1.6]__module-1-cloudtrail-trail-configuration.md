## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**CloudTrail trail fundamentals**

AWS CloudTrail records AWS API activity and delivers log files for auditing, investigation, and compliance. A **trail** is the configuration object that tells CloudTrail where to send events, which events to capture, and how to protect the delivered logs.

For exam purposes, think in terms of three layers:

- **Event capture**: management events, data events, and insight events.
- **Delivery**: Amazon S3 for durable log storage, optionally Amazon CloudWatch Logs for near-real-time monitoring.
- **Automation**: Amazon EventBridge rules for alerting and response workflows.

A trail can be created in the console, with the AWS CLI, or through API calls. The most common configuration choices you will see in exam scenarios are:

- **Single-Region trail** for one Region only.
- **Multi-Region trail** for broader coverage across Regions.
- **Organization trail** for centralized logging across an AWS Organizations organization.

**Trail creation and update options**

When you create or update a trail, you typically choose settings such as:

- **Trail name**
- **S3 bucket** for log delivery
- **S3 prefix** for optional folder organization
- **Log file encryption** with SSE-S3 or SSE-KMS
- **CloudWatch Logs integration**
- **Event selectors** for management and data events
- **Insight selectors** for unusual API activity detection
- **Multi-Region** and **organization** scope

In the console, these appear as fields such as:

- **Trail name**
- **Storage location**
- **Log file SSE-KMS encryption**
- **CloudWatch Logs**
- **SNS notification** for trail status is historical and not a core modern delivery target; do not rely on it for current design questions
- **Multi-Region trail**
- **Organization trail**

> For exam answers, prefer S3 as the primary log destination and CloudWatch Logs or EventBridge for operational visibility.

**Event types and selector behavior**

CloudTrail supports different event categories, and the selector choice matters.

- **Management events** record control-plane actions such as `CreateUser`, `RunInstances`, or `PutBucketPolicy`.
- **Data events** record high-volume resource activity such as S3 object access or Lambda function invocation.
- **Insight events** highlight unusual API call patterns, such as spikes in `DeleteTrail` or `PutBucketPolicy`.

Basic event selectors are used for common cases, while **advanced event selectors** give you finer control over which resources and operations are logged. Use advanced selectors when you need to reduce noise or target specific resource types and API activity.

A practical rule:

- Use **basic selectors** when the exam scenario asks for standard management logging or common S3/Lambda data event capture.
- Use **advanced selectors** when the scenario requires precise filtering by resource ARN, event name, or read/write type.

**Management events and global services**

Management events are always the first thing to enable in a security-focused trail. They include IAM, CloudTrail, KMS, EC2, S3 bucket-level actions, and many other control-plane operations.

Global service events need special attention:

- For global services, CloudTrail typically delivers events in the trail’s **home Region**.
- If the trail is **multi-Region**, CloudTrail replicates the trail configuration across Regions and continues to collect events according to its multi-Region behavior.
- Do not assume global service events appear in every Region by default.

This matters in exam questions about IAM changes, AWS Organizations changes, or other global control-plane activity. If you are troubleshooting missing events, verify the trail’s home Region and whether the trail is multi-Region.

**S3 delivery and log file structure**

CloudTrail writes log files to Amazon S3 in a structured hierarchy. The exact path depends on the account, Region, and time partitioning used by CloudTrail. A typical layout includes:

- `AWSLogs/<account-id>/CloudTrail/<region>/<yyyy>/<mm>/<dd>/`
- For organization trails, member account logs still appear under the `AWSLogs` hierarchy with organization-trail delivery structure and account-specific partitions.

CloudTrail also writes digest files that help verify log integrity. These digest files are important for tamper detection and audit validation.

A common exam-safe S3 bucket design includes:

- **Versioning** enabled
- **Block Public Access** enabled
- **Least-privilege bucket policy**
- **SSE-S3 or SSE-KMS** encryption
- Optional **Object Lock** for immutable retention requirements
- Optional **MFA Delete** where operationally feasible and supported by your workflow

> Use Object Lock when the scenario requires WORM (write once, read many) retention or strong immutability controls.

**S3 bucket policy requirements**

CloudTrail needs permission to write log files and verify bucket ownership. A robust bucket policy usually includes:

- `s3:GetBucketAcl`
- `s3:PutObject`
- Conditions that restrict access to the CloudTrail service principal
- `AWS:SourceArn` and, where appropriate, `AWS:SourceAccount`

A common pattern is to allow CloudTrail to write only into the expected prefix and only from the expected trail ARN.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AWSCloudTrailAclCheck",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudtrail.amazonaws.com"
      },
      "Action": "s3:GetBucketAcl",
      "Resource": "arn:aws:s3:::my-cloudtrail-logs-bucket",
      "Condition": {
        "StringEquals": {
          "AWS:SourceArn": "arn:aws:cloudtrail:us-east-1:123456789012:trail/SecurityTrail"
        }
      }
    },
    {
      "Sid": "AWSCloudTrailWrite",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudtrail.amazonaws.com"
      },
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::my-cloudtrail-logs-bucket/AWSLogs/123456789012/*",
      "Condition": {
        "StringEquals": {
          "s3:x-amz-acl": "bucket-owner-full-control",
          "AWS:SourceArn": "arn:aws:cloudtrail:us-east-1:123456789012:trail/SecurityTrail"
        }
      }
    }
  ]
}
```

For organization trails, the policy must allow the organization trail to deliver logs for member accounts under the organization-trail delivery structure. Do not oversimplify this as a single static prefix for every case; the exact resource path and conditions must match the organization trail configuration and AWS guidance.

**Encryption choices**

CloudTrail supports two common encryption approaches for S3 log files:

- **SSE-S3**: simpler, lower operational overhead, often sufficient when the requirement is basic encryption at rest.
- **SSE-KMS**: preferred when the scenario requires customer-managed key control, key rotation governance, or tighter auditability.

If you use SSE-KMS, the KMS key policy must allow CloudTrail to use the key. In practice, that usually means permissions such as:

- `kms:GenerateDataKey*`
- `kms:Decrypt`
- Often `kms:DescribeKey`
- Correct conditions for the CloudTrail service principal and encryption context

A simplified key policy example:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCloudTrailUseOfKey",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudtrail.amazonaws.com"
      },
      "Action": [
        "kms:GenerateDataKey*",
        "kms:Decrypt",
        "kms:DescribeKey"
      ],
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "AWS:SourceArn": "arn:aws:cloudtrail:us-east-1:123456789012:trail/SecurityTrail"
        }
      }
    }
  ]
}
```

In real deployments, the exact KMS policy can be more specific and may require encryption-context conditions. For exam purposes, remember that KMS permissions must be aligned with the trail ARN and the CloudTrail service principal.

**CloudWatch Logs integration**

CloudTrail can publish events to Amazon CloudWatch Logs for faster detection and alerting. This is useful when you want near-real-time monitoring for suspicious API activity.

Prerequisites include:

- A **CloudWatch Logs log group**
- An **IAM role** that CloudTrail can assume
- A trust policy that allows the CloudTrail service principal
- Permissions to create log streams and put log events

Typical trust policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudtrail.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Typical permissions policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "arn:aws:logs:us-east-1:123456789012:log-group:/aws/cloudtrail/security:*"
    }
  ]
}
```

Use CloudWatch Logs when you need:

- Metric filters for suspicious API patterns
- Alarms on sensitive actions
- Faster operational response than waiting for S3 log file delivery

**EventBridge integration**

CloudTrail events can also flow into Amazon EventBridge, which is the modern choice for event-driven automation. This is especially useful for:

- Alerting on sensitive API calls
- Triggering Lambda-based remediation
- Sending notifications to SNS or chat integrations through downstream targets
- Building detective controls for security operations

A simple EventBridge rule pattern for a high-risk API call:

```json
{
  "source": ["aws.cloudtrail"],
  "detail-type": ["AWS API Call via CloudTrail"],
  "detail": {
    "eventSource": ["iam.amazonaws.com"],
    "eventName": ["CreateAccessKey", "PutUserPolicy", "AttachUserPolicy"]
  }
}
```

A practical response flow might be:

1. CloudTrail records the API call.
2. EventBridge matches the event pattern.
3. A Lambda function evaluates the context.
4. The function disables the key, quarantines the principal, or opens an incident ticket.

**CLI configuration examples**

Create a trail:

```bash
aws cloudtrail create-trail \
  --name SecurityTrail \
  --s3-bucket-name my-cloudtrail-logs-bucket \
  --s3-key-prefix security-logs \
  --is-multi-region-trail \
  --enable-log-file-validation \
  --kms-key-id arn:aws:kms:us-east-1:123456789012:key/11111111-2222-3333-4444-555555555555
```

Start logging:

```bash
aws cloudtrail start-logging --name SecurityTrail
```

Update a trail to send logs to CloudWatch Logs:

```bash
aws cloudtrail update-trail \
  --name SecurityTrail \
  --cloud-watch-logs-log-group-arn arn:aws:logs:us-east-1:123456789012:log-group:/aws/cloudtrail/security \
  --cloud-watch-logs-role-arn arn:aws:iam::123456789012:role/CloudTrailToCloudWatchRole
```

Add data event selectors for S3 object-level activity:

```bash
aws cloudtrail put-event-selectors \
  --trail-name SecurityTrail \
  --event-selectors '[
    {
      "ReadWriteType": "All",
      "IncludeManagementEvents": true,
      "DataResources": [
        {
          "Type": "AWS::S3::Object",
          "Values": ["arn:aws:s3:::my-sensitive-bucket/"]
        }
      ]
    }
  ]'
```

For exam accuracy, remember that S3 data event selectors are about object-level activity on the bucket, and the selector syntax must match CloudTrail’s expected resource pattern. If you are unsure, prefer the conceptual answer: “enable S3 data events for the target bucket” rather than overcommitting to a brittle ARN example.

**Advanced event selectors**

Advanced event selectors are useful when you need more precise filtering than basic selectors provide. They help reduce cost and noise by narrowing the scope of logged data events.

Use them when:

- You need to log only specific buckets, functions, or tables
- You want to filter by `eventName`, `readOnly`, or resource type
- You need to avoid collecting unnecessary high-volume data events

Example pattern:

```json
[
  {
    "Name": "SensitiveS3Access",
    "FieldSelectors": [
      {
        "Field": "eventCategory",
        "Equals": ["Data"]
      },
      {
        "Field": "resources.type",
        "Equals": ["AWS::S3::Object"]
      },
      {
        "Field": "resources.ARN",
        "StartsWith": ["arn:aws:s3:::my-sensitive-bucket/"]
      }
    ]
  }
]
```

**Trail status and verification**

Use `get-trail-status` to verify whether a trail is actively logging and whether delivery is healthy.

```bash
aws cloudtrail get-trail-status --name SecurityTrail
```

Common fields in the output include:

- `IsLogging`
- `LatestDeliveryTime`
- `LatestDeliveryError`
- `LatestNotificationTime`
- `LatestNotificationError`
- `StartLoggingTime`
- `StopLoggingTime`

These fields help you troubleshoot delivery failures, permission issues, or S3/KMS misconfiguration.

A useful verification sequence is:

1. Confirm the trail exists and is configured correctly.
2. Check `get-trail-status` for logging state and delivery errors.
3. Validate S3 bucket policy and KMS permissions.
4. Confirm CloudWatch Logs or EventBridge integration if used.

**Organization trails versus account trails**

An **account trail** is created and managed within a single AWS account. It captures activity for that account and, if multi-Region, across Regions for that account.

An **organization trail** is created by the management account in AWS Organizations and captures events from member accounts as well. This is the preferred design when you need centralized audit logging across the organization.

Key distinctions:

- **Account trail**
  - Scoped to one account
  - Managed by that account
  - Useful for isolated workloads or delegated administration

- **Organization trail**
  - Created in the management account
  - Aggregates member account activity
  - Better for centralized security, compliance, and incident response

In exam scenarios, choose an organization trail when the requirement is “one place to audit all accounts.”

**Real-world security scenario**

A security team wants to detect unauthorized IAM changes, retain logs immutably, and centralize evidence across all accounts.

A strong design is:

- Create an **organization multi-Region trail**
- Deliver logs to a dedicated security account S3 bucket
- Enable **SSE-KMS** with a customer-managed key
- Turn on **log file validation**
- Enable **CloudWatch Logs** for alerting on sensitive API calls
- Add **EventBridge** rules for immediate response
- Protect the bucket with **versioning**, **Block Public Access**, and optionally **Object Lock**

This design gives you durable evidence, centralized visibility, and automation for response.

**Common exam traps**

- Do not treat SNS as the standard CloudTrail delivery target.
- Do not confuse S3 log delivery with CloudWatch Logs publishing.
- Do not assume a single bucket policy statement is enough without the correct `GetBucketAcl` and `PutObject` permissions.
- Do not forget that KMS encryption requires both key policy alignment and CloudTrail service access.
- Do not use `get-trail-status` fields that are not part of current documented output.
- Do not assume global service events appear in every Region automatically.
- Do not overuse data events; they can be expensive and noisy.

**Quick comparison**

| Feature | Best use case | Notes |
|---|---|---|
| S3 delivery | Durable audit storage | Primary destination for trail logs |
| CloudWatch Logs | Near-real-time monitoring | Requires IAM role and log group |
| EventBridge | Automation and response | Best for event-driven workflows |
| SSE-S3 | Simple encryption at rest | Lower operational overhead |
| SSE-KMS | Customer-managed encryption | More control, more policy complexity |
| Organization trail | Centralized multi-account auditing | Created in management account |

**Exam-ready summary**

CloudTrail trail configuration is about choosing the right scope, destination, and protection controls. For most security scenarios, the safest answer is a multi-Region organization trail that writes to a hardened S3 bucket, uses log file validation, and optionally integrates with CloudWatch Logs and EventBridge for detection and response.

Focus on these decision points:

- **Scope**: single-Region, multi-Region, or organization
- **Storage**: S3 bucket with hardened policy and encryption
- **Monitoring**: CloudWatch Logs and EventBridge
- **Filtering**: basic or advanced event selectors
- **Integrity**: log file validation, versioning, and immutability controls
