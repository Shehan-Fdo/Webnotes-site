## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Why scenario context matters**

AWS Security Specialty questions often hide the real task inside a business scenario. You need to identify the security objective, the current architecture, and the constraint that makes one solution better than another. The lab topology is the map you use to reason about identity, network paths, data flow, logging, and trust boundaries.

A strong reading strategy is to separate the prompt into:
- **Goal**: what must be protected, detected, or enforced.
- **Environment**: which AWS services, accounts, Regions, and networks are involved.
- **Constraint**: what you cannot change, such as latency, cost, compliance, or operational overhead.
- **Evidence**: logs, policies, routes, endpoints, or resource relationships that reveal the failure point.

**How to parse the scenario**

Start by identifying the security domain the question targets. In SCS-C02, that usually means one or more of:
- **Identity and access management**: IAM, AWS Organizations, STS, federation, SCPs.
- **Detection and response**: CloudTrail, AWS Config, GuardDuty, Security Hub, EventBridge.
- **Data protection**: KMS, S3 bucket policies, encryption, Secrets Manager, ACM.
- **Network security**: VPCs, security groups, NACLs, endpoints, Transit Gateway, Route 53.
- **Governance and posture**: Config rules, delegated admin, service control policies, detective controls.

Then look for the “shape” of the environment:
- Single account versus multi-account.
- Single Region versus multi-Region.
- Public subnet, private subnet, or hybrid connectivity.
- Centralized security account versus distributed ownership.
- Human access versus workload-to-workload access.

**Reading the topology**

A topology diagram is not just a picture of services. It tells you where trust changes, where traffic crosses boundaries, and where logs should be collected. In exam scenarios, the most important boundaries are often:
- **Internet edge**: CloudFront, Route 53, ALB, API Gateway, WAF, Shield.
- **VPC boundary**: subnets, route tables, security groups, NACLs, NAT gateways, VPC endpoints.
- **Account boundary**: cross-account roles, resource policies, Organizations, SCPs.
- **Region boundary**: replication, failover, KMS multi-Region keys, global services.
- **Data boundary**: encryption at rest, encryption in transit, key ownership, secret storage.

Use the topology to answer three questions:
- Where does the request originate?
- Which control enforces access?
- Where is evidence recorded?

**Core AWS security building blocks**

<dl>
<dt>IAM</dt>
<dd>Identity and Access Management. Use it to define who can call which AWS API actions, under what conditions.</dd>
<dt>STS</dt>
<dd>Security Token Service. Use it for temporary credentials, role assumption, federation, and cross-account access.</dd>
<dt>KMS</dt>
<dd>Key Management Service. Use it to control encryption keys, grants, key policies, and cryptographic operations.</dd>
<dt>CloudTrail</dt>
<dd>Records AWS API activity for auditing, investigation, and detection.</dd>
<dt>Config</dt>
<dd>Tracks configuration changes and evaluates resources against rules and conformance packs.</dd>
</dl>

These services often appear together in scenario questions. For example, a workload may need encrypted S3 objects, a role in another account, and audit logs for every access attempt. The correct answer usually depends on understanding which layer enforces the control:
- **IAM policy** controls identity permissions.
- **Resource policy** controls access to the resource itself.
- **KMS key policy or grant** controls cryptographic use.
- **SCP** sets the maximum permissions in an account or OU.
- **Security group** controls stateful instance-level traffic.
- **NACL** controls stateless subnet-level traffic.

**Lab topology patterns you should recognize**

Many exam questions use a small set of recurring topologies. Learn the security implications of each.

- **Single-account application**
  - Fastest to deploy.
  - Easier to manage, but weaker isolation.
  - Security controls are usually IAM, KMS, security groups, and CloudTrail.

- **Multi-account landing zone**
  - Common in mature environments.
  - Separates security, log archive, shared services, and workload accounts.
  - Centralized governance often uses AWS Organizations, SCPs, and delegated administration.

- **Hub-and-spoke network**
  - Shared inspection or egress in a central VPC.
  - Workload VPCs connect through Transit Gateway or peering.
  - Useful when the question mentions centralized firewalling, inspection, or shared services.

- **Hybrid connectivity**
  - On-premises network connects through VPN or Direct Connect.
  - Watch for route propagation, DNS resolution, and segmentation requirements.
  - Security often depends on both AWS controls and external perimeter controls.

- **Public-facing web tier**
  - Usually includes Route 53, CloudFront, WAF, ALB, and private application tiers.
  - The key question is where to terminate TLS, where to inspect traffic, and how to keep backends private.

**Trust boundaries and control placement**

A trust boundary is where you stop assuming a component is safe and start enforcing policy. In AWS, the same architecture can be secure or insecure depending on where controls are placed.

- Put **authentication** as close to the entry point as possible.
- Put **authorization** at every boundary that can be crossed.
- Put **encryption** at rest and in transit where data changes hands.
- Put **logging** at the control points, not only at the application layer.
- Put **monitoring** where it can see both allowed and denied activity.

Practical examples:
- Use an **ALB** with ACM certificates to terminate TLS, then re-encrypt to private targets if the scenario requires end-to-end encryption.
- Use **VPC endpoints** for S3 or DynamoDB when the requirement is to avoid public internet paths.
- Use **resource policies** for S3, KMS, and Secrets Manager when cross-account access is involved.
- Use **SCPs** to prevent disabling CloudTrail or KMS key deletion in member accounts.

**Common evidence types in scenarios**

Exam prompts often include artifacts that point to the root cause. Read them carefully.

- **Policy snippets**
  - Look for explicit denies, missing actions, wrong principals, or absent conditions.
  - Check whether the policy is identity-based, resource-based, or a permissions boundary.

- **CloudTrail events**
  - `AccessDenied` suggests authorization failure.
  - `KMS.AccessDeniedException` often means key policy, grant, or IAM permission mismatch.
  - `AssumeRole` failures often indicate trust policy or external ID issues.

- **VPC flow logs**
  - Useful for confirming accepted or rejected traffic.
  - Help distinguish security group issues from route or NACL issues.

- **AWS Config findings**
  - Show drift or noncompliance.
  - Useful when the question asks for continuous evaluation rather than one-time remediation.

- **GuardDuty findings**
  - Indicate suspicious behavior, such as credential exfiltration, port scanning, or unusual API calls.

**Practical CLI commands for topology discovery**

When you need to reason about a lab or scenario, these commands help you map the environment quickly.

```bash
aws sts get-caller-identity
aws organizations describe-organization
aws ec2 describe-vpcs
aws ec2 describe-subnets
aws ec2 describe-route-tables
aws ec2 describe-security-groups
aws ec2 describe-network-acls
aws ec2 describe-vpc-endpoints
aws cloudtrail describe-trails
aws configservice describe-configuration-recorders
```

For cross-account or role-based scenarios, inspect the role trust and permissions:

```bash
aws iam get-role --role-name ExampleRole
aws iam get-role-policy --role-name ExampleRole --policy-name InlinePolicy
aws iam list-attached-role-policies --role-name ExampleRole
aws iam get-policy-version --policy-arn arn:aws:iam::123456789012:policy/ExamplePolicy --version-id v1
```

For KMS-related issues, check the key and its policy:

```bash
aws kms describe-key --key-id alias/example-key
aws kms get-key-policy --key-id alias/example-key --policy-name default
aws kms list-grants --key-id alias/example-key
```

**Example topology interpretation**

Consider a scenario with:
- A public ALB in a public subnet.
- Application instances in private subnets.
- An S3 bucket used for uploads.
- A separate security account collecting logs.
- A requirement that no data path should use the public internet except the user-facing entry point.

The likely security reasoning is:
- Users reach the ALB over the internet.
- The ALB forwards to private targets inside the VPC.
- Application instances should access S3 through a **gateway VPC endpoint** or interface endpoint, depending on the service.
- Logs should flow cross-account to a centralized log archive using resource policies and service integrations.
- CloudTrail and Config should be enabled in all accounts, with centralized aggregation if required.

If the prompt says uploads fail only from private subnets, check:
- Route tables for a NAT gateway or VPC endpoint.
- Security groups on the target and endpoint.
- S3 bucket policy conditions that restrict access by VPC endpoint or source VPC.

**Security controls by layer**

| Layer | Typical controls | What to verify |
|---|---|---|
| Identity | IAM users, roles, policies, permission boundaries, SCPs | Who can call the API and whether a deny overrides allow |
| Network | Security groups, NACLs, routes, endpoints, TGW | Whether traffic can reach the destination |
| Data | KMS, S3 encryption, Secrets Manager, TLS | Whether data is protected in transit and at rest |
| Detection | CloudTrail, Config, GuardDuty, Security Hub | Whether activity is recorded and evaluated |
| Response | EventBridge, Lambda, SNS, SSM Automation | Whether findings trigger action |

**When to choose a specific control**

Use the scenario to decide the control, not the service name alone.

- Choose **SCPs** when you must enforce guardrails across accounts.
- Choose **permissions boundaries** when you want to limit what a delegated administrator can grant.
- Choose **resource policies** when another account or service needs direct access to the resource.
- Choose **security groups** when you need stateful instance or ENI-level filtering.
- Choose **NACLs** when you need stateless subnet-level filtering or explicit deny at the subnet edge.
- Choose **VPC endpoints** when private service access is required without internet traversal.
- Choose **CloudTrail** when you need API audit history.
- Choose **Config** when you need continuous compliance evaluation.
- Choose **GuardDuty** when you need managed threat detection.

**Practical configuration examples**

A bucket policy that restricts access to a specific VPC endpoint is a common topology-aware control:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowOnlyFromVpce",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:*",
      "Resource": [
        "arn:aws:s3:::example-bucket",
        "arn:aws:s3:::example-bucket/*"
      ],
      "Condition": {
        "StringNotEquals": {
          "aws:sourceVpce": "vpce-0123456789abcdef0"
        }
      }
    }
  ]
}
```

A role trust policy for cross-account assumption should clearly define the trusted principal:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/SecurityAuditRole"
      },
      "Action": "sts:AssumeRole",
      "Condition": {
        "StringEquals": {
          "sts:ExternalId": "example-external-id"
        }
      }
    }
  ]
}
```

A simple VPC endpoint check can confirm whether private service access exists:

```bash
aws ec2 describe-vpc-endpoints \
  --filters Name=vpc-id,Values=vpc-0123456789abcdef0
```

**How to avoid common exam mistakes**

- Do not confuse **authentication** with **authorization**.
- Do not assume a security group can block all traffic if a route or endpoint issue is the real problem.
- Do not use an IAM policy when the requirement is to control access from another account to a resource.
- Do not use CloudTrail as a preventive control; it is detective.
- Do not use Config when the question asks for immediate traffic filtering.
- Do not overlook the account boundary in multi-account designs.
- Do not ignore the Region if the scenario mentions replication, failover, or global services.

**A practical reading workflow**

1. Identify the business goal and the security objective.
2. Mark the resources, accounts, Regions, and networks in the topology.
3. Find the trust boundaries and the control points.
4. Determine whether the issue is identity, network, data, or detection related.
5. Match the requirement to the least disruptive AWS control that satisfies it.
6. Verify whether the solution is preventive, detective, or responsive.

**What the exam expects from you**

The exam rewards precise mapping between scenario and control. You should be able to explain:
- Why a control belongs at a specific boundary.
- Which AWS service enforces the requirement.
- How logs or findings prove the issue.
- What changes are safe in a shared or regulated environment.

If you can read the topology, identify the trust boundaries, and connect the evidence to the right control layer, you can usually eliminate distractors quickly and choose the most secure, least operationally complex answer.
