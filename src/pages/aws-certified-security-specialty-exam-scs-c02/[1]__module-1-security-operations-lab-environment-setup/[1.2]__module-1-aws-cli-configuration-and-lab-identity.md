## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**AWS CLI identity model**

The AWS Command Line Interface (CLI) is a thin client that signs requests with credentials and region context you provide. For exam prep, focus on how the CLI resolves identity, how profiles isolate lab work, and how temporary credentials differ from long-term keys.

The CLI commonly uses these inputs:

- **Environment variables** for quick overrides in a shell session.
- **Shared config files** for persistent profiles.
- **Temporary credentials** from AWS Security Token Service (STS), AWS Identity Center, or role assumption.
- **Metadata-based credentials** from Amazon Elastic Compute Cloud (EC2) or Amazon Elastic Container Service (ECS) when running on AWS.

A secure lab workflow usually avoids hardcoding keys in scripts. Instead, you switch identities with profiles, short-lived sessions, or isolated shell environments.

**Installation and version checks**

Start by verifying that the CLI is installed and identifying the major version. AWS CLI v2 is the current standard for modern features such as `--no-cli-pager`, `AWS_PAGER`, and current Identity Center support.

```bash
aws --version
which aws
aws help
```

Typical output includes the version number and Python runtime details. For exam scenarios, note these version differences:

- **AWS CLI v2** supports modern SSO/Identity Center flows, `--no-cli-pager`, and improved credential handling.
- **AWS CLI v1** is older, has different pager behavior, and lacks some v2-only conveniences.
- Some labs and legacy systems still use v1, so you should recognize both.

If a command behaves unexpectedly, confirm that the shell is invoking the intended binary and not an older installation.

**Configuration files and where settings live**

The CLI reads from shared files in the user home directory by default:

- `~/.aws/config` for region, output format, role profiles, SSO settings, and other non-secret configuration.
- `~/.aws/credentials` for access keys and session tokens.
- `~/.aws/cli/cache/` for cached role credentials and some temporary credential retrieval data.
- `~/.aws/sso/cache/` for AWS CLI v2 Identity Center token cache.

> Do not assume all cached data lives in one directory. SSO token cache and role credential cache are separate, and that distinction matters in troubleshooting.

You can relocate these files for isolated labs or ephemeral environments:

```bash
export AWS_CONFIG_FILE=/tmp/lab/aws-config
export AWS_SHARED_CREDENTIALS_FILE=/tmp/lab/aws-credentials
```

This is useful when you want to keep a clean host profile, run multiple lab identities side by side, or avoid credential bleed between tasks.

**How the CLI resolves credentials and settings**

Credential resolution is layered. The exact behavior depends on the setting type, but the practical exam rule is:

- **Explicit command-line options** usually override other sources.
- **Environment variables** often override shared file values.
- **Shared config and credentials files** provide persistent defaults.
- **Metadata or role-based sources** can supply temporary credentials when configured.

For credentials, the CLI considers sources such as:

- `--profile`
- `AWS_PROFILE`
- `AWS_ACCESS_KEY_ID`
- `AWS_SECRET_ACCESS_KEY`
- `AWS_SESSION_TOKEN`
- Shared credentials file entries
- Shared config file role settings
- EC2 instance metadata or ECS task role credentials
- Identity Center cached sessions

For region and output, the common precedence is:

- Command-line option such as `--region`
- Environment variables such as `AWS_REGION` and `AWS_DEFAULT_REGION`
- Profile settings in `~/.aws/config`

`AWS_REGION` is recognized by AWS CLI v2 and many SDKs, and it often takes precedence over `AWS_DEFAULT_REGION`. Use `AWS_DEFAULT_REGION` as a fallback or compatibility variable, but know that `AWS_REGION` is frequently the more direct override in modern tooling.

**Profile basics**

Profiles let you separate identities, regions, and output formats. A profile can represent:

- A long-term IAM user key pair
- A role assumed from a source profile
- An Identity Center session
- A lab-specific region and output preference
- A metadata-based role on EC2 or ECS

List profiles with:

```bash
aws configure list-profiles
```

Inspect a specific setting with:

```bash
aws configure get region --profile dev
aws configure get output --profile dev
aws configure get aws_access_key_id --profile base
```

Interpretation tips:

- If `aws configure get` returns nothing, the value may come from an environment variable or another source.
- `aws configure list` shows the resolved source for each setting, which is often more useful than reading files directly.

```bash
aws configure list
aws configure list --profile dev
```

A typical `aws configure list` output helps you see whether a value came from `env`, `shared-credentials-file`, `config-file`, or `assume-role`.

**Setting values with `aws configure`**

Use `aws configure` for interactive setup:

```bash
aws configure
```

You will usually enter:

- Access key ID
- Secret access key
- Default region
- Default output format

For scripted setup, `aws configure set` is useful, but remember that it writes to different files depending on the key:

- Credential keys such as `aws_access_key_id` and `aws_secret_access_key` are stored in the shared credentials file.
- Non-credential settings such as `region`, `output`, and role configuration are stored in the config file.

Examples:

```bash
aws configure set aws_access_key_id AKIAEXAMPLE --profile base
aws configure set aws_secret_access_key 'wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY' --profile base
aws configure set region us-east-1 --profile base
aws configure set output json --profile base
```

This distinction matters in labs because you may inspect the wrong file and think a setting is missing.

**Region and output behavior**

Region selection is critical because many AWS APIs are regional. If you omit a region, the CLI may fail or use a default from your environment or profile.

Common region sources:

- `--region us-west-2`
- `AWS_REGION=us-west-2`
- `AWS_DEFAULT_REGION=us-west-2`
- `region = us-west-2` in a profile

Practical examples:

```bash
aws s3 ls --region us-east-1
export AWS_REGION=eu-west-1
aws ec2 describe-regions
```

Use `AWS_REGION` when you want a shell-wide override for modern CLI and SDK behavior. Use `AWS_DEFAULT_REGION` when you need compatibility with tools that expect that variable. In mixed environments, check both.

Output format controls readability, not authorization:

- `json` is best for automation and parsing.
- `table` is useful for human review.
- `text` is compact for shell pipelines.
- `yaml` or `yaml-stream` may help in some workflows.

```bash
aws sts get-caller-identity --output json
aws ec2 describe-instances --output table
```

**Identity verification and safe lab checks**

Before making changes, verify who you are and where requests will go.

```bash
aws sts get-caller-identity
aws configure list
aws configure list-profiles
```

`sts get-caller-identity` returns the account, user or role ARN, and user ID. It is one of the fastest ways to confirm that your lab identity is correct.

A safe lab pattern is to isolate identity in a subshell so environment variables do not leak into later tasks:

```bash
(
  export AWS_PROFILE=lab-admin
  export AWS_REGION=us-east-1
  aws sts get-caller-identity
  aws s3 ls
)
```

When the subshell exits, the exported variables disappear.

To clear accidental overrides in the current shell:

```bash
unset AWS_ACCESS_KEY_ID AWS_SECRET_ACCESS_KEY AWS_SESSION_TOKEN AWS_PROFILE AWS_REGION AWS_DEFAULT_REGION AWS_PAGER
```

**Temporary credentials and session tokens**

Temporary credentials are central to security-focused labs. They usually include:

- Access key ID
- Secret access key
- Session token
- Expiration time

If you use STS to assume a role manually, the response includes temporary credentials that you can export for the current shell:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/SecurityAuditRole \
  --role-session-name audit-session
```

You can then export the returned values:

```bash
export AWS_ACCESS_KEY_ID=ASIAEXAMPLE
export AWS_SECRET_ACCESS_KEY=exampleSecret
export AWS_SESSION_TOKEN=exampleSessionToken
```

However, manual export is only one option. In many cases, you should configure the profile to assume the role automatically.

Example profile chain:

```ini
[profile base]
region = us-east-1
output = json

[profile audit]
role_arn = arn:aws:iam::123456789012:role/SecurityAuditRole
source_profile = base
region = us-east-1
output = json
```

With this setup, the CLI uses the base profile to obtain credentials and then assumes the target role automatically.

**`source_profile` vs `credential_source`**

Use `source_profile` when the source credentials come from another named profile. Use `credential_source` when the source is an ambient AWS credential provider such as EC2 instance metadata or ECS task role credentials.

Examples:

```ini
[profile app-role]
role_arn = arn:aws:iam::123456789012:role/AppRole
source_profile = base
region = us-east-1
```

```ini
[profile ec2-role]
role_arn = arn:aws:iam::123456789012:role/AppRole
credential_source = Ec2InstanceMetadata
region = us-east-1
```

Use `credential_source` in cloud-hosted automation when you want the instance or task role to be the root identity. This avoids storing a separate source profile key pair.

**MFA and role assumption**

Multi-Factor Authentication (MFA) is common in security labs and exam scenarios. You can require MFA when assuming a role by using a trust policy condition and then supplying a token code during role assumption.

Example manual assumption:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/AdminRole \
  --role-session-name mfa-session \
  --serial-number arn:aws:iam::123456789012:mfa/alice \
  --token-code 123456
```

For profile-based workflows, you can combine MFA with role profiles and session duration controls. In practice, the CLI may prompt for MFA or use a helper process depending on how the profile is configured.

Security takeaway:

- MFA strengthens privileged access.
- Short session durations reduce blast radius.
- Role sessions are preferable to long-term keys for admin tasks.

**AWS Identity Center and modern SSO setup**

AWS CLI v2 supports AWS Identity Center, formerly AWS Single Sign-On (SSO), through profile configuration and cached browser-based login.

A modern setup often uses `aws configure sso` or explicit profile configuration with an SSO session block.

Example:

```bash
aws configure sso
```

You may be prompted for:

- SSO start URL
- SSO region
- Account
- Role
- Default region
- Output format

Modern configuration can also use `sso-session` blocks:

```ini
[sso-session corp]
sso_start_url = https://d-abc123.awsapps.com/start
sso_region = us-east-1
sso_registration_scopes = sso:account:access

[profile dev-sso]
sso_session = corp
sso_account_id = 123456789012
sso_role_name = ReadOnlyAccess
region = us-east-1
output = json
```

Then authenticate with:

```bash
aws sso login --profile dev-sso
```

Important behavior:

- The browser login creates a cached token under `~/.aws/sso/cache/`.
- The CLI uses that token to obtain role credentials.
- Expired sessions require re-login.
- If the cache is stale or corrupted, deleting the relevant cache entry can force a clean login.

Troubleshooting examples:

- `The SSO session associated with this profile has expired` usually means you need `aws sso login`.
- `Token has expired and refresh failed` often indicates the browser session or cached token is no longer valid.
- If a profile references the wrong `sso_session`, the CLI may fail before it reaches the account selection step.

**Pager behavior**

The CLI may page long output through a pager such as `less`. This is useful interactively but can break scripts or confuse lab output capture.

AWS CLI v2 supports:

- `--no-cli-pager` for a single command
- `AWS_PAGER=` to disable paging for the current shell
- `cli_pager =` in config to disable paging persistently

Examples:

```bash
aws ec2 describe-instances --no-cli-pager
export AWS_PAGER=""
aws s3 ls
```

In AWS CLI v1, pager behavior differs and `--no-cli-pager` is not available. For exam prep, remember that pager controls are a v2 feature set.

**Practical profile management**

A clean profile layout helps you avoid accidental privilege use.

Example shared config:

```ini
[default]
region = us-east-1
output = json

[profile base]
region = us-east-1
output = json

[profile read-only]
role_arn = arn:aws:iam::123456789012:role/ReadOnlyRole
source_profile = base
region = us-east-1
output = json

[profile break-glass]
role_arn = arn:aws:iam::123456789012:role/AdminRole
source_profile = base
mfa_serial = arn:aws:iam::123456789012:mfa/alice
region = us-east-1
output = json
```

Use this pattern to separate:

- Baseline identity
- Read-only inspection
- Privileged escalation
- Emergency access

If you need to inspect the active profile without guessing, use:

```bash
echo "$AWS_PROFILE"
aws configure list
aws sts get-caller-identity --profile read-only
```

**Troubleshooting common failures**

Common CLI identity problems usually fall into a few categories:

- **Expired session token**
  - Re-run `aws sso login`
  - Re-assume the role
  - Refresh the shell environment

- **Invalid profile chain**
  - Check `source_profile` names
  - Confirm the base profile has valid credentials
  - Ensure there is no circular reference

- **Wrong region**
  - Verify `AWS_REGION`, `AWS_DEFAULT_REGION`, and profile region
  - Use `--region` to test a one-off override

- **Credential bleed from another shell**
  - Run `env | grep '^AWS_'`
  - Unset conflicting variables
  - Use a subshell for isolation

- **Missing permissions**
  - Confirm the identity with `sts get-caller-identity`
  - Check whether the role trust policy allows assumption
  - Verify session policies or permission boundaries if applicable

Useful diagnostic commands:

```bash
aws configure list
aws configure get region --profile dev
aws sts get-caller-identity --debug
```

`--debug` is noisy, but it can reveal which provider the CLI selected and why a request failed.

**Metadata credentials on EC2 and ECS**

When the CLI runs on AWS compute, it can obtain credentials from the instance or task role automatically.

- On EC2, the CLI can use the Instance Metadata Service (IMDS).
- On ECS, the CLI can use the task role credential endpoint.

This is often preferable to `source_profile` in automation because no static source keys are stored on disk.

Use `credential_source = Ec2InstanceMetadata` when you want a profile to rely on the instance role. This is common in hardened environments where local key material is prohibited.

Security considerations:

- Restrict IMDS access where possible.
- Prefer IMDSv2 on EC2.
- Avoid exposing task role credentials to containers that do not need them.

**Lab identity hygiene**

Good lab hygiene prevents accidental cross-account actions and credential leakage.

Recommended practices:

- Use separate profiles for each account or role.
- Prefer temporary credentials over long-term keys.
- Keep `AWS_PROFILE` scoped to a subshell or script.
- Clear environment variables after each task.
- Verify identity before destructive commands.
- Use `--region` explicitly when the target region matters.

Example safe workflow:

```bash
(
  export AWS_PROFILE=security-lab
  export AWS_REGION=us-east-1
  aws sts get-caller-identity
  aws iam list-users --no-cli-pager
)
```

If you need to switch identities often, consider wrapper scripts or shell functions that set and clear variables predictably.

**Quick reference**

- `aws --version` confirms installation and version.
- `aws configure list-profiles` shows available profiles.
- `aws configure list` shows resolved values and sources.
- `aws sts get-caller-identity` confirms the active identity.
- `aws sso login --profile name` refreshes Identity Center sessions.
- `AWS_REGION` and `AWS_DEFAULT_REGION` influence regional behavior.
- `AWS_PAGER` and `--no-cli-pager` control paging in AWS CLI v2.
- `AWS_CONFIG_FILE` and `AWS_SHARED_CREDENTIALS_FILE` relocate config for labs.

**Exam focus**

For SCS-C02, expect questions that test whether you can:

- Distinguish long-term keys from temporary credentials.
- Choose between `source_profile`, `credential_source`, and Identity Center.
- Recognize where the CLI stores config, credentials, and caches.
- Troubleshoot expired sessions, wrong regions, and profile resolution issues.
- Use secure identity switching patterns in a lab without leaking credentials.
