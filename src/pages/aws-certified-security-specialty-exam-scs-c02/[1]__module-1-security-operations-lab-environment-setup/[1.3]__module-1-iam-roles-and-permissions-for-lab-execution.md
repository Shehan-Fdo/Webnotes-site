## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**IAM roles in lab execution**

IAM roles let you grant temporary, scoped access for hands-on labs without distributing long-term credentials. In security-focused lab environments, you usually combine roles with AWS Security Token Service (STS) to assume permissions only when needed, then let the session expire automatically.

A role has two distinct policy surfaces:

- **Trust policy**: who can assume the role.
- **Permissions policy**: what the role can do after it is assumed.

This separation is central to exam scenarios because you often need to decide whether access should come from role assumption, a resource-based policy, or both.

**Core IAM policy evaluation logic**

AWS evaluates permissions by combining multiple policy types. The result is not a simple union; explicit denies and policy boundaries can override allows.

- **Explicit deny** always wins over any allow.
- **Identity-based policies** attached to a user, group, or role grant permissions to the principal.
- **Resource-based policies** grant permissions directly on the target resource.
- **Permissions boundaries** limit the maximum permissions an identity can ever receive from identity-based policies.
- **Service control policies (SCPs)** in AWS Organizations set the outer guardrails for accounts and OUs.
- **Session policies** further restrict permissions for a specific STS session.

A practical mental model is:

1. Start with the identity’s allowed actions.
2. Intersect with permissions boundaries, SCPs, and session policies where applicable.
3. Add resource-based permissions if the resource policy grants access.
4. Remove anything explicitly denied at any layer.

> If any applicable policy contains an explicit deny, the request fails even if another policy allows it.

**How policy types interact**

The interaction matters in labs because you may see a role that appears to have permission, but the request still fails due to a boundary, SCP, or session policy.

| Policy type | Scope | Typical lab use | Can it grant access alone? |
|---|---|---|---|
| Identity-based policy | Principal | Role permissions for lab tasks | Yes, if no other restriction blocks it |
| Resource-based policy | Resource | S3 bucket, KMS key, Lambda, SNS, SQS, Secrets Manager | Yes, for supported services |
| Permissions boundary | Principal maximum | Prevent privilege escalation in delegated admin labs | No |
| SCP | Account / OU guardrail | Restrict risky actions in training accounts | No |
| Session policy | Temporary STS session | Narrow a role for a specific exercise | No |
| Explicit deny | Any applicable policy | Block sensitive actions | Yes, as a blocker |

**Example evaluation scenarios**

- A role allows `s3:GetObject`, but the bucket policy denies the object prefix: the request fails.
- A role allows `iam:PassRole`, but the permissions boundary omits it: the request fails.
- A role allows `ec2:TerminateInstances`, but the account SCP denies termination: the request fails.
- A role session policy removes `s3:DeleteObject`: the assumed role cannot delete objects even if the role policy allows it.

**Trust policies and assumption patterns**

Trust policies define which principals can call `sts:AssumeRole`, `sts:AssumeRoleWithSAML`, or `sts:AssumeRoleWithWebIdentity`. In labs, you often trust:

- An IAM user or role in the same account.
- A role in another account for cross-account exercises.
- An AWS service principal such as `lambda.amazonaws.com` or `ec2.amazonaws.com`.
- An identity provider for federated access.

A minimal trust policy for a human operator role looks like this:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowLabOperatorAssumeRole",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:user/LabAdmin"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

For cross-account access, the trusted principal is usually a role ARN from the source account, not a user ARN, because role-to-role delegation is easier to manage and audit.

**STS session controls**

STS issues temporary credentials with an access key, secret key, and session token. These credentials are time-limited and are the standard way to execute labs safely.

Common STS commands:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/LabExecutionRole \
  --role-session-name lab-session-01
```

You can export the returned credentials into your shell:

```bash
export AWS_ACCESS_KEY_ID=ASIA...
export AWS_SECRET_ACCESS_KEY=...
export AWS_SESSION_TOKEN=...
```

For a federated or web identity flow, use the matching STS API:

```bash
aws sts assume-role-with-web-identity \
  --role-arn arn:aws:iam::123456789012:role/FederatedLabRole \
  --role-session-name federated-lab \
  --web-identity-token file://token.jwt
```

**Role chaining and session duration**

Role chaining occurs when you assume one role and then use those temporary credentials to assume another role. This is common in lab workflows that jump from a base operator role into a more specialized role.

Important constraints:

- The maximum session duration for a chained role session is typically limited to one hour, even if the target role allows longer sessions.
- Chaining can complicate troubleshooting because the effective permissions are the intersection of the source session and the target role.
- If the source session is already restricted by a session policy or boundary, the second role cannot exceed those limits.

Example flow:

1. You assume `LabOperatorRole`.
2. Using those temporary credentials, you assume `TargetSecurityRole`.
3. The final session inherits the shorter chaining limit and any upstream restrictions.

This matters in exam labs because a long-running task may fail mid-way if the chained session expires.

**Session tags and transitive tags**

Session tags let you attach key-value attributes to an assumed-role session. Transitive tags persist across role chaining when marked as transitive. They are useful for attribute-based access control (ABAC), audit context, and conditional access.

Example assume-role call with tags:

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/LabExecutionRole \
  --role-session-name lab-abac \
  --tags Project=SecurityLab Environment=Dev \
  --transitive-tag-keys Project
```

Use cases:

- **ABAC**: allow access only when `aws:PrincipalTag/Project` matches `aws:ResourceTag/Project`.
- **Traceability**: carry a `TicketId` or `LabId` through chained sessions.
- **Conditional access**: require a tag before allowing sensitive actions.

Example IAM condition using principal tags:

```json
{
  "Effect": "Allow",
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::security-lab-data/*",
  "Condition": {
    "StringEquals": {
      "aws:PrincipalTag/Project": "SecurityLab"
    }
  }
}
```

**Role types you should recognize**

- **EC2 instance profile role**: attached to an EC2 instance so applications can retrieve temporary credentials from the instance metadata service.
- **Lambda execution role**: grants a function permission to write logs, read secrets, access S3, or call other services.
- **Federated access role**: assumed by SAML or OIDC identities from an external identity provider.
- **Service-linked role**: created and managed by AWS for a specific service, such as Auto Scaling or Elastic Load Balancing.
- **Cross-account lab role**: used to access resources in another account with a controlled trust relationship.

**Lambda execution role example**

A Lambda function usually needs a trust policy for the Lambda service and a permissions policy for the function’s runtime actions.

Trust policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "lambda.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Permissions policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogGroup",
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "*"
    },
    {
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::security-lab-input/*"
    }
  ]
}
```

This pattern is common in labs where a function processes objects, reads parameters, or writes findings to CloudWatch Logs.

**Federated access role example**

For SAML or OIDC federation, the trust policy references the identity provider rather than a human user. This is useful when the lab environment integrates with an external IdP and you want short-lived access without IAM users.

SAML trust example:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Federated": "arn:aws:iam::123456789012:saml-provider/CorpIdP"
      },
      "Action": "sts:AssumeRoleWithSAML",
      "Condition": {
        "StringEquals": {
          "SAML:aud": "https://signin.aws.amazon.com/saml"
        }
      }
    }
  ]
}
```

**Service-linked roles**

Service-linked roles are created by AWS services and are tightly scoped to the service’s needs. You usually do not manually attach arbitrary permissions to them.

Use them when:

- AWS requires the role for a managed service feature.
- You want AWS to manage the trust relationship and baseline permissions.
- You need to avoid over-privileging a custom service role.

In exam questions, distinguish service-linked roles from customer-managed roles because the management workflow differs.

**Resource-based policies as an alternative or complement**

In some lab scenarios, you do not need to assume a role at all. A resource-based policy can grant direct access to a resource from another principal.

Common examples:

- **S3 bucket policy** grants cross-account read access.
- **KMS key policy** grants decrypt permission to a role or account.
- **Lambda resource policy** allows invocation from another account or service.
- **SNS topic policy** allows publish or subscribe from a specific principal.
- **SQS queue policy** allows send or receive from a trusted principal.

When to prefer a resource-based policy:

- You want direct access to a single resource without broader role assumption.
- You need cross-account access to a service that supports resource policies.
- You want to avoid role chaining for a narrow use case.

When to prefer role assumption:

- You need a broader set of permissions across multiple services.
- You want temporary credentials for a human operator or automation.
- You need centralized control over session duration and tags.

**EC2 instance metadata and IMDSv2 hardening**

EC2 instances retrieve role credentials from the Instance Metadata Service (IMDS). IMDSv2 adds a session-oriented request flow that reduces the risk of SSRF (Server-Side Request Forgery) credential theft.

Hardening steps:

- Require IMDSv2 on the instance.
- Limit metadata access to trusted processes only.
- Avoid exposing metadata endpoints to untrusted containers or applications.
- Use the least-privileged instance profile possible.

Example AWS CLI hardening command:

```bash
aws ec2 modify-instance-metadata-options \
  --instance-id i-0123456789abcdef0 \
  --http-tokens required \
  --http-endpoint enabled \
  --http-put-response-hop-limit 1
```

Why it matters:

- `--http-tokens required` forces IMDSv2.
- A low hop limit reduces the chance that nested network paths or containers can reach metadata.
- In labs, metadata exposure is a common misconfiguration that leads to credential theft.

> Do not rely on IMDS as a general secret store. It is for temporary role credentials and instance identity, not application secrets.

**Common role management workflows**

During lab setup, you often need to update trust and permissions repeatedly. Know the CLI workflow for safe iteration.

Create a role:

```bash
aws iam create-role \
  --role-name LabExecutionRole \
  --assume-role-policy-document file://trust.json
```

Attach a managed policy:

```bash
aws iam attach-role-policy \
  --role-name LabExecutionRole \
  --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess
```

Put or replace an inline policy:

```bash
aws iam put-role-policy \
  --role-name LabExecutionRole \
  --policy-name LabInlineAccess \
  --policy-document file://permissions.json
```

Update the trust policy:

```bash
aws iam update-assume-role-policy \
  --role-name LabExecutionRole \
  --policy-document file://trust-updated.json
```

Detach a managed policy:

```bash
aws iam detach-role-policy \
  --role-name LabExecutionRole \
  --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess
```

Delete an inline policy:

```bash
aws iam delete-role-policy \
  --role-name LabExecutionRole \
  --policy-name LabInlineAccess
```

Delete the role:

```bash
aws iam delete-role \
  --role-name LabExecutionRole
```

List attached policies and trust details:

```bash
aws iam list-attached-role-policies --role-name LabExecutionRole
aws iam list-role-policies --role-name LabExecutionRole
aws iam get-role --role-name LabExecutionRole
```

**Rotating and tightening permissions during labs**

Lab environments often start broad and then get tightened. A safe workflow is:

1. Create the role with only the permissions needed for the first task.
2. Test the role with `aws sts assume-role`.
3. Add permissions incrementally as the lab requires them.
4. Remove temporary permissions after validation.
5. Re-test after each change to confirm the minimum access set.

This approach helps you identify whether a failure is caused by the trust policy, the permissions policy, a boundary, or an SCP.

**Troubleshooting access failures**

When a role does not work, check the layers in order:

- Confirm the principal can assume the role.
- Confirm the trust policy allows the exact principal and action.
- Confirm the role permissions include the target action and resource.
- Check for explicit denies in SCPs, boundaries, or resource policies.
- Check whether a session policy or chained session is narrowing access.
- Verify the resource ARN and region are correct.
- Confirm the session has not expired.

Useful commands:

```bash
aws sts get-caller-identity
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:role/LabExecutionRole \
  --action-names s3:GetObject \
  --resource-arns arn:aws:s3:::security-lab-data/example.txt
```

Simulation is especially helpful when you need to reason about policy precedence before running a destructive lab action.

**Practical lab topologies**

A common security lab topology includes:

- A human operator in Account A.
- A jump or operator role in Account A.
- A target lab role in Account B.
- A resource-based policy on an S3 bucket or KMS key in Account B.
- An EC2 instance profile or Lambda execution role for automation tasks.

This design lets you compare three access patterns:

- Direct role assumption for broad temporary access.
- Resource-based policy for narrow resource access.
- Service role for workload execution.

**Exam-focused distinctions**

- **Trust policy** answers: who can assume the role?
- **Permissions policy** answers: what can the role do?
- **Resource policy** answers: who can access this resource directly?
- **Boundary or SCP** answers: what is the maximum allowed?
- **Session policy** answers: what is allowed for this specific session?

If you remember only one rule, remember that an allow is never enough by itself when a higher-priority deny or boundary exists.

**Key takeaways**

- Use roles for temporary, least-privilege lab access.
- Separate trust from permissions and evaluate policy precedence carefully.
- Expect explicit denies, boundaries, SCPs, and session policies to override allows.
- Use role chaining intentionally and account for shorter session duration.
- Use session tags and transitive tags for ABAC and traceability.
- Prefer resource-based policies when you only need access to one supported resource.
- Harden EC2 metadata access with IMDSv2 and a low hop limit.
- Practice role update, detach, and delete workflows so you can iterate quickly during labs.
