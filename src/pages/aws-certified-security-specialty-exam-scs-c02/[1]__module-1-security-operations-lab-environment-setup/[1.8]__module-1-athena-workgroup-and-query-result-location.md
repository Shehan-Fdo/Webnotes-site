## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**Athena workgroups: what they control**

An <abbr title="Amazon Athena">Athena</abbr> workgroup is a governance boundary for queries. It helps you separate teams, enforce settings, and control where query results go. For exam scenarios, the key idea is that a workgroup can override or constrain client-side behavior, especially for result storage and encryption.

Workgroups are commonly used for:

- **Cost control** by tracking usage per team or application.
- **Governance** by enforcing a standard output location and encryption policy.
- **Security** by preventing ad hoc clients from writing results to an unsafe bucket or unencrypted path.
- **Operational separation** by isolating dev, test, and prod query behavior.

Athena is not a substitute for IAM, S3 bucket policies, or KMS key policies. It coordinates with them. If any one layer is misconfigured, query execution or result delivery can fail.

**Query result location: the core exam concept**

Athena writes query results to an <abbr title="Amazon Simple Storage Service">S3</abbr> output location. That location is configured per workgroup or supplied by the client, depending on the settings in effect.

Important exam points:

- The output bucket must be in the **same Region** as the Athena workgroup.
- The configured result location is where Athena stores the query output objects.
- The result bucket is not a dump for temporary processing artifacts in normal operation.
- If a workgroup enforces its result location, client-side settings are ignored.

A typical result path looks like this:

```text
s3://athena-results-prod/security/team-a/
```

Athena writes result files such as CSV or other query output artifacts into the configured prefix. The exact file naming is service-managed.

**Workgroup configuration precedence**

Athena can use a client-provided result location, but a workgroup may enforce its own configuration. This is a common exam trap.

Use this mental model:

1. The client submits a query and may include a result location.
2. The workgroup settings are evaluated.
3. If the workgroup enforces its configuration, Athena uses the workgroup-defined output location and encryption settings.
4. If enforcement is not enabled, the client setting may be used, subject to permissions and service behavior.

This matters because a secure design often relies on the workgroup to prevent users from redirecting results to an unapproved bucket.

**Where you configure the output location**

You can set the query result location in several places:

- **AWS Management Console**
  - In Athena, open the workgroup settings.
  - Set the query result location under the workgroup configuration.
  - Optionally enable enforcement so users cannot override it.
- **AWS CLI**
  - Use `create-work-group` or `update-work-group` with a configuration object.
  - Set the output location in the workgroup configuration.
- **SDKs and APIs**
  - Pass the workgroup name and query execution settings.
  - The client may also provide a result configuration, but workgroup enforcement can override it.

Example CLI pattern:

```bash
aws athena create-work-group \
  --name security-team-a \
  --configuration '{
    "ResultConfiguration": {
      "OutputLocation": "s3://athena-results-prod/security/team-a/",
      "EncryptionConfiguration": {
        "EncryptionOption": "SSE_KMS",
        "KmsKey": "arn:aws:kms:us-east-1:111122223333:key/12345678-1234-1234-1234-123456789012"
      }
    },
    "EnforceWorkGroupConfiguration": true
  }'
```

Use this as a conceptual pattern. AWS CLI parameter shapes can vary by version, but the exam-relevant idea is consistent: the workgroup can define and enforce output location and encryption.

**Console versus CLI/SDK behavior**

The console is often used to set a default result location for a workgroup. In the CLI and SDKs, the same concept appears as a workgroup configuration object.

Exam-oriented distinctions:

- **Console**
  - Easier to visualize.
  - Good for setting a default output bucket and prefix.
  - Useful when demonstrating enforcement to non-developers.
- **CLI/SDK**
  - Better for automation and repeatability.
  - Common in infrastructure-as-code and deployment pipelines.
  - Lets you standardize secure defaults across accounts and environments.

If a question asks where a user sets the default output location, the answer is usually the **Athena workgroup configuration**, not the query text itself.

**Encryption options for query results**

Athena query result encryption is separate from the encryption of the underlying data source.

Common result encryption choices include:

- **SSE-S3**: Amazon S3-managed keys.
- **SSE-KMS**: AWS Key Management Service (KMS) customer managed or AWS managed keys.
- **Client-side encryption**: In some workflows, clients encrypt before upload or handle encryption outside Athena. This is less common in exam scenarios than SSE-S3 and SSE-KMS.

Key exam distinction:

- **Data source encryption** protects the data stored in the source table location.
- **Query result encryption** protects the output written by Athena to S3.

A secure workgroup often enforces SSE-KMS for results when the output contains sensitive data.

**Permissions model: S3 and KMS both matter**

Athena query result delivery depends on more than one permission layer.

You typically need:

- **S3 permissions** for the output bucket and prefix.
- **KMS permissions** if the result location uses SSE-KMS.
- **IAM permissions** for the principal running the query.
- **Bucket policy** and, if applicable, **KMS key policy** that allow the required access.

A common exam decision point is recognizing that a query can fail even when the S3 bucket exists, because the principal lacks KMS permissions or the key policy does not allow use of the key.

Typical permission needs for SSE-KMS result writes:

- `s3:PutObject` on the result prefix.
- `s3:GetBucketLocation` on the bucket.
- `kms:GenerateDataKey` and related encrypt permissions for the KMS key.
- `kms:Decrypt` may be needed in some read or reuse scenarios, but it is not the primary write-path permission to focus on for result creation.

> If the bucket policy allows S3 writes but the KMS key policy blocks use of the key, Athena result delivery still fails.

Example IAM policy fragment for the querying role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AthenaQueryResultsWrite",
      "Effect": "Allow",
      "Action": [
        "s3:GetBucketLocation",
        "s3:PutObject"
      ],
      "Resource": [
        "arn:aws:s3:::athena-results-prod",
        "arn:aws:s3:::athena-results-prod/security/team-a/*"
      ]
    },
    {
      "Sid": "AllowKmsForAthenaResults",
      "Effect": "Allow",
      "Action": [
        "kms:GenerateDataKey",
        "kms:Encrypt",
        "kms:DescribeKey"
      ],
      "Resource": "arn:aws:kms:us-east-1:111122223333:key/12345678-1234-1234-1234-123456789012"
    }
  ]
}
```

This is a practical baseline. Real deployments may also need additional permissions for reading source data, catalog access, or workgroup management.

**Bucket policy considerations**

The S3 bucket must allow Athena-related result writes in a way that matches your security model. In many environments, the principal running the query writes results through Athena’s service behavior, so the bucket policy should be aligned with the caller’s access path and the service’s expected write pattern.

A secure bucket policy often does the following:

- Allows `s3:PutObject` only to the approved prefix.
- Restricts writes to the expected encryption method.
- Denies unencrypted uploads.
- Optionally restricts access by VPC endpoint, principal, or account.

Example bucket policy pattern:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowAthenaQueryResultsWrite",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/AthenaQueryRole"
      },
      "Action": [
        "s3:PutObject",
        "s3:GetBucketLocation"
      ],
      "Resource": [
        "arn:aws:s3:::athena-results-prod",
        "arn:aws:s3:::athena-results-prod/security/team-a/*"
      ]
    },
    {
      "Sid": "DenyUnencryptedUploads",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::athena-results-prod/security/team-a/*",
      "Condition": {
        "StringNotEquals": {
          "s3:x-amz-server-side-encryption": "aws:kms"
        }
      }
    }
  ]
}
```

Use this as a pattern, not a universal template. The exact principal and conditions depend on whether you enforce SSE-S3 or SSE-KMS and whether the bucket is shared.

**KMS key policy considerations**

For SSE-KMS, the KMS key policy must allow the relevant principal to use the key for result encryption. The key policy is often the missing piece in exam scenarios.

A practical key policy usually allows:

- `kms:Encrypt`
- `kms:GenerateDataKey`
- `kms:DescribeKey`

It may also allow `kms:Decrypt` for workflows that need to read encrypted objects later, but do not assume decrypt is required for the write path itself.

Example key policy fragment:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowAthenaQueryRoleUseOfKey",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/AthenaQueryRole"
      },
      "Action": [
        "kms:Encrypt",
        "kms:GenerateDataKey",
        "kms:DescribeKey"
      ],
      "Resource": "*"
    }
  ]
}
```

If the key policy is too restrictive, Athena may fail with access denied errors even when the IAM role appears to have permissions.

**Query result reuse and sensitive workloads**

Athena can reuse previous query results in some scenarios to reduce cost and latency. This is useful for repeated queries, but it can be a security concern for sensitive workloads.

Exam-relevant guidance:

- Enable reuse when repeated identical queries are acceptable and the data sensitivity allows it.
- Disable reuse when you need strict freshness, stronger auditability, or reduced exposure of cached results.
- Treat reuse as a governance setting, not a replacement for encryption or access control.

For sensitive environments, you may choose to disable result reuse in the workgroup or at the query level so that every query produces a fresh result path and follows the current access controls.

**Misconfiguration scenarios and fixes**

A common exam scenario is a query that fails because the workgroup and output settings are not aligned.

Typical failure patterns:

- **Wrong Region**
  - The S3 output bucket is in a different Region than the Athena workgroup.
  - Fix: move the output bucket or use a same-Region bucket and prefix.
- **Missing S3 write permission**
  - The role can run Athena but cannot write to the result prefix.
  - Fix: allow `s3:PutObject` and `s3:GetBucketLocation` for the approved path.
- **Missing KMS permission**
  - The bucket uses SSE-KMS, but the role or key policy does not allow key use.
  - Fix: update both IAM and KMS key policy.
- **Client override blocked by enforcement**
  - The user supplies a different output location, but the workgroup enforces its own configuration.
  - Fix: update the workgroup settings or use the approved location.
- **Unapproved encryption mode**
  - The bucket policy denies uploads unless SSE-KMS is used.
  - Fix: configure the workgroup to use SSE-KMS and ensure the key is accessible.

A useful troubleshooting sequence:

1. Confirm the workgroup name used by the query.
2. Check whether workgroup enforcement is enabled.
3. Verify the configured output location and Region.
4. Confirm S3 permissions on the exact prefix.
5. Confirm KMS permissions and key policy if SSE-KMS is enabled.
6. Review CloudTrail and Athena query execution errors for access denied details.

**Governance and cost control use cases**

Athena workgroups are especially useful when you need to separate trust boundaries.

Common patterns:

- **Per-team workgroups**
  - Each team gets its own output prefix and encryption settings.
- **Per-environment workgroups**
  - Dev, test, and prod use different buckets or prefixes.
- **Sensitive-data workgroups**
  - Enforce SSE-KMS, disable reuse, and restrict access tightly.
- **Shared analytics workgroups**
  - Standardize output location and logging for centralized reporting.

Workgroups help with governance, but they do not replace:

- **IAM** for identity and action authorization.
- **S3 bucket policies** for object-level and prefix-level control.
- **KMS key policies** for cryptographic access control.

**Exam cues and what to choose**

When you see a question about Athena query results, look for these clues:

- A user can run queries but results are not appearing in S3.
  - Think permissions, output location, or Region mismatch.
- A team must not override the result bucket.
  - Think workgroup enforcement.
- Sensitive results must always be encrypted with a customer-managed key.
  - Think SSE-KMS plus KMS key policy.
- Different teams need separate result paths and governance.
  - Think multiple workgroups.
- Repeated queries should not reuse prior results for compliance reasons.
  - Think disable result reuse.

**Practical secure setup pattern**

A strong baseline design looks like this:

- One Athena workgroup per security boundary.
- A dedicated S3 bucket or prefix for query results.
- Workgroup enforcement enabled.
- SSE-KMS configured for result encryption.
- IAM role permissions scoped to the exact prefix.
- KMS key policy scoped to the querying role or account.
- Query result reuse disabled for sensitive workloads.

Example architecture summary:

```text
User/Role -> Athena Workgroup (enforced settings)
          -> S3 result bucket/prefix in same Region
          -> SSE-KMS with approved KMS key
          -> CloudTrail/CloudWatch for audit and monitoring
```

**Key takeaways**

- Athena workgroups control query governance, including result location and encryption.
- The result bucket must be in the same Region as the workgroup.
- Workgroup enforcement can override client-side output settings.
- Secure result delivery usually requires both S3 and KMS permissions.
- SSE-KMS requires a valid KMS key policy, not just IAM permission.
- Query result reuse can be useful, but disable it for sensitive or compliance-heavy workloads.
- Workgroups improve control, but they do not replace IAM, bucket policies, or KMS policies.
