## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**GuardDuty enablement model**

Amazon GuardDuty is a managed threat detection service that analyzes signals from AWS data sources and produces findings. For exam purposes, the key idea is that GuardDuty is not “turned on once” for an entire organization in a single global step. You enable it **per Region**, and organization-wide administration is also **Region-specific**.

That distinction matters because many exam questions mix up:
- the **management account** in AWS Organizations,
- the **delegated administrator** for GuardDuty,
- the **detector** that exists in each Region,
- and the **organization configuration** that controls member onboarding.

A good mental model is:

1. The AWS Organizations management account authorizes a delegated GuardDuty administrator.
2. The delegated admin enables GuardDuty in each required Region.
3. The delegated admin configures organization settings for member accounts in that Region.
4. Advanced features are enabled separately and may require additional telemetry or service-specific setup.

**Core prerequisites for organization-wide GuardDuty**

Before you design org-wide GuardDuty, verify the AWS Organizations foundation. GuardDuty organization administration depends on Organizations being configured correctly.

- **AWS Organizations must be in use** for delegated administration and centralized member management.
- The management account should have **all features enabled** in AWS Organizations so delegated administration works as expected.
- You must register a **delegated administrator** for GuardDuty in the management account.
- GuardDuty must be enabled in **each Region** where you want coverage.
- Organization settings and auto-enable behavior are also **Region-scoped**, not global.

A common exam trap is assuming that enabling GuardDuty in one Region covers the whole organization. It does not. If you need coverage in `us-east-1`, `us-west-2`, and `eu-west-1`, you must repeat the relevant GuardDuty setup in each of those Regions.

**Typical prerequisite sequence**

For organization-based deployment, the practical sequence is:

1. Ensure AWS Organizations is configured with the management account and all-features mode.
2. In the management account, designate a delegated GuardDuty administrator.
3. In the delegated admin account, enable GuardDuty in the target Region.
4. Configure organization auto-enable settings for that Region.
5. Let existing and new member accounts be enrolled according to the org settings.
6. Repeat the process in every additional Region that needs GuardDuty coverage.

This sequence is important because the delegated admin can only manage organization settings after GuardDuty exists in that Region for the admin account. The detector is the regional anchor point for the service.

**GuardDuty detectors and regional scope**

A **detector** is the regional GuardDuty resource that represents GuardDuty being active in an account and Region. If you are enabling GuardDuty manually or via automation, the detector is the first concrete resource you create.

- One detector exists per account per Region.
- The detector is required before you can manage many GuardDuty settings in that Region.
- Organization-wide settings are applied around that regional detector, not instead of it.

Example CLI to create a detector in the current account and Region:

```bash
aws guardduty create-detector \
  --enable \
  --finding-publishing-frequency FIFTEEN_MINUTES
```

You can verify the detector with:

```bash
aws guardduty list-detectors
```

If you are working in multiple Regions, repeat the command in each Region or use automation that targets each Region explicitly.

**AWS Organizations delegated administration**

For centralized security operations, the management account delegates GuardDuty administration to a security account. This is the account your security team uses to manage findings, organization settings, and member onboarding.

Important exam points:
- Delegated administration is an **Organizations action**, not a detector action.
- You do **not** need to pass a detector ID from the management account to register the delegated admin.
- The delegated admin still needs GuardDuty enabled in each Region it manages.
- The management account remains the payer and governance root, but it should not be used for day-to-day security operations.

Example of registering a delegated administrator from the management account:

```bash
aws organizations register-delegated-administrator \
  --account-id 111122223333 \
  --service-principal guardduty.amazonaws.com
```

To confirm the delegated admin relationship:

```bash
aws organizations list-delegated-administrators \
  --service-principal guardduty.amazonaws.com
```

> GuardDuty delegated administration is Region-aware in practice because the service configuration you manage is regional, even though the Organizations registration itself is performed through Organizations APIs.

**Organization auto-enable and member onboarding**

After the delegated admin has GuardDuty enabled in a Region, you configure how member accounts are brought under protection in that Region. The exact API shape can vary by SDK/CLI version, but the exam-relevant concept is the same: you control whether new and existing organization accounts are automatically enabled.

Typical behaviors include:
- **Auto-enable new accounts** so future accounts are protected without manual intervention.
- **Auto-enable existing accounts** so current members are enrolled.
- **Manual account management** when you want to control onboarding account by account.

A representative CLI pattern is:

```bash
aws guardduty update-organization-configuration \
  --detector-id d1example \
  --auto-enable NEW
```

Depending on the API version and feature set, you may see parameters that distinguish new members, existing members, or both. For exam prep, focus on the concept: organization settings determine whether GuardDuty is automatically enabled for member accounts in that Region.

To inspect organization configuration:

```bash
aws guardduty describe-organization-configuration \
  --detector-id d1example
```

**Member account enablement patterns**

There are two common operational models.

- **Organization auto-enable**
  - Best for centralized security teams.
  - New accounts are protected automatically.
  - Reduces drift and missed onboarding.
- **Manual member management**
  - Useful when you need staged rollout or exception handling.
  - Requires explicit account-level enablement or invitation-style workflows where applicable.

A practical example of manual enablement in a member account is creating a detector directly in that account and Region:

```bash
aws guardduty create-detector --enable
```

In an organization-managed environment, you usually prefer org auto-enable rather than relying on ad hoc member setup.

**Region-by-Region planning**

GuardDuty is not a single global switch. You must plan coverage by Region.

- Enable GuardDuty in every Region where workloads run or where you need threat visibility.
- Configure delegated admin and org settings in each of those Regions.
- Check whether the Region supports the GuardDuty features you want.
- Remember that findings, detectors, and organization settings are all regional.

A security team often starts with the primary production Regions, then expands to:
- disaster recovery Regions,
- data residency Regions,
- and Regions used by development or sandbox accounts if those environments matter for threat detection.

**Service-linked roles and permissions**

GuardDuty and related AWS services may create or use **service-linked roles** to perform service operations. In exam scenarios, this usually means you should expect AWS-managed permissions to be involved when enabling features or integrating with other services.

Key points:
- The delegated admin and management account need the right IAM and Organizations permissions.
- Some GuardDuty features require additional service permissions or service-linked roles.
- If a feature fails to enable, check both IAM authorization and service prerequisites.

A minimal IAM policy for a security operator is usually broader than a read-only role because GuardDuty administration includes service configuration, organization management, and member account operations.

**Advanced feature prerequisites**

GuardDuty has multiple feature families, and each can have its own setup requirements. Do not assume that enabling the detector alone activates every capability.

- **S3 Protection**
  - Requires S3-related telemetry and the correct GuardDuty feature enablement.
  - Useful for detecting suspicious object-level activity patterns.
- **EKS Protection**
  - Relies on Amazon EKS telemetry, especially audit log visibility.
  - Some runtime capabilities may require additional add-ons or agents depending on the feature.
  - Make sure the cluster and Region support the specific GuardDuty capability you want.
- **Malware Protection**
  - Requires the relevant storage or workload integration and feature-specific configuration.
  - Often involves additional permissions and resource-level setup.
- **Runtime Monitoring**
  - Requires workload instrumentation or agent-based integration depending on the compute platform.
  - Plan for deployment, permissions, and operational overhead.
- **RDS-related detections**
  - GuardDuty can analyze database activity signals, but do not treat this as a generic “RDS Protection” toggle.
  - Know the exact feature name and telemetry source in exam questions.

The exam often tests whether you understand that advanced features are not just “on/off” at the detector level. They may require logs, agents, add-ons, or service integrations.

**Practical CLI workflow**

A common centralized workflow looks like this:

1. Register the delegated admin in the management account.
2. Switch to the delegated admin account.
3. Create or verify the detector in the target Region.
4. Enable organization auto-enable.
5. Confirm member accounts are enrolled.
6. Repeat for each Region.

Example sequence:

```bash
# From management account
aws organizations register-delegated-administrator \
  --account-id 111122223333 \
  --service-principal guardduty.amazonaws.com

# From delegated admin account in us-east-1
aws guardduty create-detector --enable

aws guardduty update-organization-configuration \
  --detector-id d1example \
  --auto-enable NEW
```

Then verify:

```bash
aws guardduty list-members --detector-id d1example
aws guardduty describe-organization-configuration --detector-id d1example
```

**Standalone vs organization-wide deployment**

Choose the deployment model based on governance needs.

| Model | Best for | Tradeoff |
|---|---|---|
| Standalone account | Small environments, labs, isolated workloads | More manual work, easier to miss accounts |
| Organization-wide | Central security operations, enterprise governance | Requires Organizations setup and regional administration |

A standalone deployment is acceptable when you only need one account protected. In an enterprise, organization-wide deployment is usually preferred because it reduces configuration drift and gives the security team centralized visibility.

**Common exam pitfalls**

- Assuming GuardDuty is global instead of **regional**.
- Confusing the **management account** with the **delegated administrator**.
- Thinking delegated admin registration requires a detector ID from the management account.
- Forgetting that org settings must be configured in **each Region**.
- Assuming one Region’s setup automatically protects all Regions.
- Treating advanced features as if they are enabled by the base detector alone.
- Overlooking AWS Organizations prerequisites such as all-features mode and delegated administration support.

**Quick verification checklist**

Use this checklist when validating a GuardDuty rollout:

- AWS Organizations is enabled and supports delegated administration.
- The management account has registered a GuardDuty delegated admin.
- A detector exists in the delegated admin account for each required Region.
- Organization auto-enable is configured for the intended member onboarding behavior.
- Member accounts appear in GuardDuty for the target Region.
- Advanced features have their required telemetry and service-specific prerequisites.
- IAM permissions and service-linked roles are in place.

**Troubleshooting mindset**

If GuardDuty does not behave as expected, check in this order:
- Is the Region correct?
- Is the detector present in that Region?
- Is the account actually the delegated admin?
- Is AWS Organizations configured correctly?
- Is auto-enable set the way you expect?
- Does the advanced feature require extra logs, add-ons, or permissions?

That order helps you separate service setup problems from organization governance problems, which is a common distinction in SCS-C02 scenarios.
