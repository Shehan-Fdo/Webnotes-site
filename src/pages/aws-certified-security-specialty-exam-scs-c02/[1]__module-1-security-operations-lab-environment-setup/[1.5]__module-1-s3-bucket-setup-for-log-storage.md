## **Course:** AWS Certified Security – Specialty (Exam SCS-C02)

**S3 as a log archive target**

Amazon S3 is a common destination for security, audit, and operational logs because it is durable, inexpensive, and integrates with many AWS services. For exam scenarios, focus on three goals:

- **Durability** for long-term retention.
- **Immutability** where audit requirements demand it.
- **Controlled write access** so only the logging service can deliver objects.

A good log bucket design separates ingestion from access. You usually allow a service principal to write objects, while humans and applications get read-only access through tightly scoped IAM permissions.

**Core bucket design choices**

Start by deciding whether the bucket is a general archive or a compliance-grade log vault. That choice affects encryption, retention, and whether you need object immutability.

- **Bucket naming** should be globally unique and predictable, often including account, region, and purpose.
- **Region placement** should match the source service when required by the service documentation.
- **Prefix design** should separate sources, such as `cloudtrail/`, `config/`, `elb/`, and `vpcflow/`.
- **Access control** should default to least privilege and deny public access.

A practical naming pattern looks like this:

```text
org-security-logs-us-east-1-123456789012
```

A prefix layout can help with lifecycle and troubleshooting:

```text
cloudtrail/AWSLogs/123456789012/CloudTrail/us-east-1/2026/04/26/
config/AWSLogs/123456789012/Config/us-east-1/
elb/AWSLogs/123456789012/elasticloadbalancing/us-east-1/
vpcflow/AWSLogs/123456789012/vpcflowlogs/us-east-1/
```

**Object Ownership and ACL implications**

<abbr title="Object Ownership">S3 Object Ownership</abbr> is critical for service-delivered logs. In modern setups, prefer `Bucket owner enforced`, which disables ACLs and makes the bucket owner own all objects.

- **Bucket owner enforced** is the preferred default for new buckets when the delivery pattern supports it.
- **Legacy ACL-based delivery** may still appear in older service examples or older bucket policies.
- **Do not assume** every service delivery policy still needs `bucket-owner-full-control`; that condition is relevant only when ACLs are in use.

> If a service or integration still depends on ACLs, `Bucket owner enforced` can break delivery. Check the service-specific documentation before removing ACL-related conditions.

For exam questions, distinguish between these cases:

- **Current pattern**: bucket owner enforced, no ACLs, service writes with a resource policy that allows `s3:PutObject`.
- **Legacy pattern**: ACLs enabled, service may require `x-amz-acl: bucket-owner-full-control`.
- **Mixed assumptions**: if the question says ACLs are disabled, do not choose an ACL-based answer.

**Encryption and key management**

Use server-side encryption with S3-managed keys or AWS Key Management Service (AWS KMS) depending on the compliance requirement.

- **SSE-S3** is simple and often sufficient for baseline protection.
- **SSE-KMS** gives you auditability and tighter control over key usage.
- **KMS key policies** must allow the S3 service to use the key for the specific bucket and region.

A practical SSE-KMS bucket policy and KMS policy pair usually needs careful scoping. The exact condition keys vary by service integration, but the general pattern is:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowS3UseOfKeyForLogBucket",
      "Effect": "Allow",
      "Principal": {
        "Service": "s3.amazonaws.com"
      },
      "Action": [
        "kms:Encrypt",
        "kms:Decrypt",
        "kms:GenerateDataKey*",
        "kms:DescribeKey"
      ],
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "aws:SourceAccount": "123456789012"
        }
      }
    }
  ]
}
```

This is only a pattern, not a universal template. In real deployments, you often scope the key policy more tightly to the service integration, region, and bucket ARN pattern. For exam purposes, remember that S3 and KMS must both be configured correctly or log delivery fails.

**Bucket policy fundamentals**

A log bucket policy usually does three things:

- Allows the logging service to write objects.
- Restricts writes to the expected account, source, or organization.
- Denies insecure transport and public access.

A common security control is to require TLS:

```json
{
  "Sid": "DenyInsecureTransport",
  "Effect": "Deny",
  "Principal": "*",
  "Action": "s3:*",
  "Resource": [
    "arn:aws:s3:::org-security-logs-us-east-1-123456789012",
    "arn:aws:s3:::org-security-logs-us-east-1-123456789012/*"
  ],
  "Condition": {
    "Bool": {
      "aws:SecureTransport": "false"
    }
  }
}
```

For service delivery, you often need `Allow` statements that are specific to the source service principal and bucket prefix. Depending on the service, you may also use:

- **`aws:SourceArn`** to bind delivery to a specific trail, configuration recorder, load balancer, or flow log.
- **`aws:SourceAccount`** to reduce confused-deputy risk.
- **`aws:PrincipalOrgID`** when the source is constrained to an AWS Organization.
- **`s3:x-amz-acl`** only in legacy ACL-based patterns.

> Do not use a generic deny for missing encryption headers as a blanket rule for all log delivery. Some AWS services do not send the same headers as interactive clients, and an overbroad deny can break delivery.

**Common log sources and their bucket requirements**

The major AWS log sources differ in how they write to S3 and what policy elements they need. The exact principal and policy shape can vary by service, region, and partition, so follow the service documentation for the final answer in an exam scenario.

| Log source | Typical destination pattern | Common policy elements | Notes |
|---|---|---|---|
| CloudTrail | `AWSLogs/<account-id>/CloudTrail/<region>/...` | `s3:PutObject`, `s3:GetBucketAcl` in legacy patterns, `aws:SourceArn`, `aws:SourceAccount` | Management events require a trail or organization trail; they are not delivered to S3 by default. |
| AWS Config | `AWSLogs/<account-id>/Config/<region>/...` | `s3:PutObject`, `aws:SourceAccount`, sometimes `aws:SourceArn` | Ensure the recorder and delivery channel are configured. |
| Elastic Load Balancing (ELB/ALB/NLB) access logs | `AWSLogs/<account-id>/elasticloadbalancing/<region>/...` | `s3:PutObject`, `aws:SourceAccount`, `aws:SourceArn` or load balancer-specific conditions | Bucket and prefix must match the service’s expected format. |
| VPC Flow Logs | `AWSLogs/<account-id>/vpcflowlogs/<region>/...` | Often delivered to CloudWatch Logs or Kinesis Data Firehose; S3 is supported in some scenarios | S3 is not the most common current destination, so read the question carefully. |

**CloudTrail log bucket setup**

CloudTrail is a frequent exam topic because it is central to audit logging. To store CloudTrail logs in S3, you create a trail and point it to a bucket.

Key points:

- **Management events** are not automatically sent to S3; you must create a trail.
- **Organization trails** centralize logs across accounts.
- **Log file validation** can help detect tampering.
- **Multi-Region trails** are common for security visibility.

A minimal trail creation example:

```bash
aws cloudtrail create-trail \
  --name org-audit-trail \
  --s3-bucket-name org-security-logs-us-east-1-123456789012 \
  --is-multi-region-trail
```

Then start logging:

```bash
aws cloudtrail start-logging --name org-audit-trail
```

For CloudTrail bucket policies, exam answers often include:

- `s3:GetBucketAcl` for legacy validation of bucket ownership.
- `s3:PutObject` to the `AWSLogs/<account-id>/` prefix.
- `aws:SourceArn` tied to the trail ARN.
- `aws:SourceAccount` tied to the owning account.

If the scenario says Object Ownership is `Bucket owner enforced`, avoid assuming ACL-based delivery is required.

**AWS Config log delivery**

AWS Config records configuration history and compliance snapshots. It needs an S3 bucket for snapshots and history files.

Important setup considerations:

- The bucket should be in the same region as the recorder unless the service documentation allows otherwise.
- The delivery channel must be configured before recording starts.
- The bucket policy should allow the Config service to write to the expected prefix.

A typical workflow:

1. Create the bucket.
2. Apply the bucket policy for AWS Config delivery.
3. Create or update the configuration recorder.
4. Create the delivery channel.
5. Start the recorder.

Example CLI:

```bash
aws configservice put-configuration-recorder \
  --configuration-recorder name=default,roleARN=arn:aws:iam::123456789012:role/AWSConfigRole

aws configservice put-delivery-channel \
  --delivery-channel name=default,s3BucketName=org-security-logs-us-east-1-123456789012

aws configservice start-configuration-recorder \
  --configuration-recorder-name default
```

**Elastic Load Balancing access logs**

ELB access logs are useful for traffic analysis and incident response. They are written to S3, usually under a service-specific prefix.

Design considerations:

- Use a dedicated prefix per load balancer or environment.
- Ensure the bucket policy allows the load balancer service to write.
- Keep the bucket private and restrict read access to analysts and security tooling.

A common operational mistake is using the wrong prefix or region. If the bucket is in a different region than expected by the service, delivery may fail.

**VPC Flow Logs destination choices**

VPC Flow Logs capture network metadata for ENIs, subnets, or VPCs. In practice, they are commonly delivered to CloudWatch Logs or Kinesis Data Firehose, and S3 is used in some architectures for archival or downstream analytics.

Use S3 when:

- You want low-cost long-term retention.
- You plan to query logs with Athena or another S3-native analytics tool.
- You can tolerate delivery latency and batch-oriented access.

Use CloudWatch Logs or Firehose when:

- You need near-real-time analysis.
- You want easier integration with monitoring or streaming pipelines.

**Object Lock and retention controls**

For compliance or forensic retention, S3 Object Lock can prevent deletion or overwrite for a defined period. It must be enabled when the bucket is created; you cannot turn it on later for an existing bucket.

Key terms:

- **Retention mode**: `Governance` or `Compliance`.
- **Legal hold**: an on/off flag that blocks deletion regardless of retention date.
- **WORM**: write once, read many.

Example bucket creation with Object Lock:

```bash
aws s3api create-bucket \
  --bucket org-security-logs-us-east-1-123456789012 \
  --object-lock-enabled-for-bucket \
  --create-bucket-configuration LocationConstraint=us-east-1
```

Example retention configuration:

```bash
aws s3api put-object-retention \
  --bucket org-security-logs-us-east-1-123456789012 \
  --key cloudtrail/AWSLogs/123456789012/CloudTrail/us-east-1/2026/04/26/example.json.gz \
  --retention Mode=COMPLIANCE,RetainUntilDate=2027-04-26T00:00:00Z
```

Use Object Lock only when the business requirement justifies the operational constraints. It can complicate deletion, lifecycle transitions, and incident response.

**Lifecycle and storage class strategy**

Lifecycle policies reduce cost, but log archives need careful treatment. For security and audit logs, favor durability and retrieval reliability over the cheapest storage class.

Common approach:

- Keep recent logs in `STANDARD`.
- Transition older logs to `STANDARD_IA` or `INTELLIGENT_TIERING` when access patterns are uncertain.
- Use `GLACIER` or `DEEP_ARCHIVE` for long-term retention if retrieval latency is acceptable.

> Be cautious with `ONEZONE_IA` for security logs. It has lower durability than multi-AZ classes and is usually not the right choice for audit evidence unless the scenario explicitly accepts that tradeoff.

Example lifecycle configuration:

```json
{
  "Rules": [
    {
      "ID": "ArchiveOldLogs",
      "Status": "Enabled",
      "Filter": {
        "Prefix": "cloudtrail/"
      },
      "Transitions": [
        {
          "Days": 90,
          "StorageClass": "STANDARD_IA"
        },
        {
          "Days": 365,
          "StorageClass": "DEEP_ARCHIVE"
        }
      ],
      "Expiration": {
        "Days": 2555
      }
    }
  ]
}
```

**Security hardening checklist**

A secure log bucket usually includes the following controls:

- **Block Public Access** enabled at the bucket and account level.
- **Bucket policy** that denies non-TLS requests.
- **Least privilege** IAM for readers and administrators.
- **Versioning** enabled when it supports recovery or audit needs.
- **Server access logging** or CloudTrail data events for the bucket itself if you need visibility into access.
- **KMS key rotation** when using SSE-KMS.

You can also restrict access by organization:

```json
{
  "Sid": "AllowReadOnlyFromOrg",
  "Effect": "Allow",
  "Principal": "*",
  "Action": [
    "s3:GetObject",
    "s3:ListBucket"
  ],
  "Resource": [
    "arn:aws:s3:::org-security-logs-us-east-1-123456789012",
    "arn:aws:s3:::org-security-logs-us-east-1-123456789012/*"
  ],
  "Condition": {
    "StringEquals": {
      "aws:PrincipalOrgID": "o-abc123xyz"
    }
  }
}
```

Use this carefully. It is useful for centralized security teams, but it should not replace explicit role-based access where tighter control is needed.

**Troubleshooting failed log delivery**

When logs do not arrive, check the simplest causes first.

1. Verify the source service is actually enabled.
2. Confirm the bucket name, region, and prefix match the service requirements.
3. Review the bucket policy for the correct service principal and conditions.
4. Check whether Object Ownership or ACL settings conflict with the delivery pattern.
5. Confirm KMS permissions if SSE-KMS is enabled.
6. Inspect service-specific delivery status or error messages.

Common failure causes include:

- Wrong bucket region.
- Missing `s3:PutObject` permission.
- Incorrect `aws:SourceArn` or `aws:SourceAccount`.
- ACL-based policy used against a bucket with ACLs disabled.
- KMS key policy missing S3 permissions.
- Overly broad deny statements that block service writes.
- Prefix mismatch between the service configuration and the bucket policy.

For S3-side verification, useful commands include:

```bash
aws s3api get-bucket-policy --bucket org-security-logs-us-east-1-123456789012
aws s3api get-bucket-encryption --bucket org-security-logs-us-east-1-123456789012
aws s3api get-bucket-ownership-controls --bucket org-security-logs-us-east-1-123456789012
aws s3api get-bucket-versioning --bucket org-security-logs-us-east-1-123456789012
```

If you suspect delivery is failing because of permissions, test the policy logic against the exact service principal and prefix rather than assuming a generic S3 write permission is enough.

**Exam decision cues**

Use these cues to choose the best answer quickly:

- **Audit trail to S3** usually means CloudTrail with a dedicated log bucket.
- **Centralized configuration history** usually means AWS Config with an S3 delivery bucket.
- **Load balancer traffic logs** usually means ELB access logs to S3.
- **Network flow visibility** often means CloudWatch Logs or Firehose first, not S3.
- **Immutable retention** points to Object Lock and possibly Compliance mode.
- **Modern bucket ownership** points to `Bucket owner enforced`, not ACL-based delivery.

**Practical reference architecture**

A common security logging architecture looks like this:

- One centralized log bucket per region or per organization.
- Separate prefixes for each source.
- SSE-KMS for encryption.
- Block Public Access enabled.
- Object Lock enabled when compliance requires immutability.
- Lifecycle transitions to archival storage after the active investigation window.
- Read access limited to security operations roles and analytics roles.

Example high-level flow:

```text
AWS service -> S3 log bucket prefix -> lifecycle -> archive storage
                     |
                     +-> Athena / Security analytics / incident response
```

This design keeps ingestion simple while preserving the ability to query and retain logs for investigations, audits, and compliance reviews.
