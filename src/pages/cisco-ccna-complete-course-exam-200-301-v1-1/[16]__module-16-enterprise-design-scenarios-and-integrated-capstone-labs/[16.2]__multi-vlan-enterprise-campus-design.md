## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Enterprise campus hierarchical model**

A campus network usually follows a three-layer design: **access**, **distribution**, and **core**. In CCNA terms, the access layer connects end devices, the distribution layer aggregates access switches and often performs policy and inter-VLAN routing, and the core provides fast transport between distribution blocks.

- **Access layer**: user devices, printers, phones, APs, and edge ports.
- **Distribution layer**: VLAN boundary, routing boundary, ACL enforcement, and gateway placement in many designs.
- **Core layer**: high-speed transit with minimal policy to keep forwarding simple and resilient.

Multi-VLAN design maps naturally to this hierarchy:

- VLANs are typically extended across access switches to keep users in the same logical broadcast domain.
- Trunks carry multiple VLANs between access and distribution switches.
- Layer 3 gateways live on SVIs on a multilayer switch or on a router using router-on-a-stick.
- The core usually forwards routed traffic between distribution blocks rather than carrying user VLAN policy.

In small CCNA labs, you may collapse distribution and core into one multilayer switch. The design idea still matters: keep Layer 2 domains local, route between VLANs at a controlled boundary, and avoid stretching VLANs farther than needed.

**VLAN fundamentals and range considerations**

A VLAN (Virtual Local Area Network) separates a switch into multiple logical broadcast domains. Devices in different VLANs need Layer 3 routing to communicate.

Cisco VLAN ID ranges are important for design and exam questions:

- **Normal range**: VLANs `1-1005`
  - Stored in `vlan.dat` on many Catalyst switches.
  - VLANs `1`, `1002-1005` are reserved or legacy-related.
- **Extended range**: VLANs `1006-4094`
  - Commonly used in larger environments.
  - Support depends on switch platform and VTP mode/version behavior.
- **Practical CCNA guidance**:
  - Use VLANs in the normal range for most lab and campus designs.
  - Avoid VLAN `1` for user traffic because it is the default VLAN and often associated with management/trunk defaults.
  - Use a dedicated management VLAN and separate user, voice, and server VLANs.

A simple campus VLAN plan might look like this:

| VLAN | Purpose | Example subnet |
|---|---|---|
| 10 | Users | `192.168.10.0/24` |
| 20 | Finance | `192.168.20.0/24` |
| 30 | Voice | `192.168.30.0/24` |
| 99 | Management | `192.168.99.0/24` |

**Access ports, voice VLANs, and endpoint segmentation**

An access port belongs to one data VLAN for the attached PC or printer. In a common office design, an IP phone and a PC share the same physical switch port. The phone uses a **voice VLAN** while the PC uses the access VLAN.

Typical behavior:

- The switch sends untagged frames for the data VLAN.
- The phone tags voice traffic with the voice VLAN ID.
- The phone often acts as a mini-switch, passing PC traffic through untagged.

Example access port with voice VLAN:

```cisco
interface gigabitEthernet1/0/10
 description Phone + PC
 switchport mode access
 switchport access vlan 20
 switchport voice vlan 30
 spanning-tree portfast
 spanning-tree bpduguard enable
```

Why this matters:

- Voice traffic gets separate QoS and broadcast treatment.
- The phone can receive power and network access on one port.
- The design keeps user and voice traffic logically separated without extra cabling.

**Trunks and VLAN transport**

A trunk carries traffic for multiple VLANs between switches or between a switch and a router. In Cisco campus designs, trunks are common on uplinks from access to distribution and between redundant switches.

Key trunk concepts:

- **802.1Q tagging** inserts a VLAN tag into frames.
- The **native VLAN** is the VLAN sent untagged on an 802.1Q trunk.
- The allowed VLAN list controls which VLANs can traverse the trunk.

Example trunk configuration:

```cisco
interface gigabitEthernet1/0/1
 description Uplink to Distribution
 switchport trunk encapsulation dot1q
 switchport mode trunk
 switchport trunk native vlan 99
 switchport trunk allowed vlan 10,20,30,99
```

Practical notes:

- Keep the native VLAN consistent on both ends.
- Prune unused VLANs from trunks to reduce unnecessary flooding.
- Use a dedicated native VLAN that is not used for user traffic.
- If the platform does not support `switchport trunk encapsulation dot1q`, the command may be absent because 802.1Q is the only supported encapsulation.

**Native VLAN mismatch and trunk security**

A native VLAN mismatch occurs when the two ends of a trunk use different native VLANs. This can cause untagged frames to land in the wrong VLAN and generate warnings.

Common symptoms:

- `show interfaces trunk` reveals inconsistent native VLANs.
- CDP (Cisco Discovery Protocol) or system messages report native VLAN mismatch.
- Hosts may appear to be in the wrong broadcast domain.
- Some traffic works while other traffic fails in confusing ways.

Security implications:

- Native VLAN misuse can contribute to **VLAN hopping** risk.
- Attackers may exploit trunk negotiation or double-tagging scenarios if trunks are poorly configured.
- Best practice is to disable DTP (Dynamic Trunking Protocol) on user-facing ports and explicitly set trunk mode where needed.

Safer trunk pattern:

```cisco
interface gigabitEthernet1/0/1
 switchport mode trunk
 switchport nonegotiate
 switchport trunk native vlan 999
 switchport trunk allowed vlan 10,20,30,99
```

> Use an unused native VLAN and avoid leaving access ports in an auto-trunking state.

**Inter-VLAN routing options**

VLANs isolate Layer 2 traffic, so routing is required for communication between VLANs. CCNA commonly expects you to know two approaches:

- **Router-on-a-stick**
  - One router interface connects to a trunk.
  - Subinterfaces represent VLAN gateways.
  - Useful in small labs and branch designs.
- **Layer 3 switch with SVIs**
  - Switched Virtual Interfaces (SVIs) act as VLAN gateways.
  - More scalable and common in campus distribution designs.

Router-on-a-stick example:

```cisco
interface gigabitEthernet0/0
 no shutdown

interface gigabitEthernet0/0.10
 encapsulation dot1Q 10
 ip address 192.168.10.1 255.255.255.0

interface gigabitEthernet0/0.20
 encapsulation dot1Q 20
 ip address 192.168.20.1 255.255.255.0

interface gigabitEthernet0/0.30
 encapsulation dot1Q 30
 ip address 192.168.30.1 255.255.255.0
```

Layer 3 switch SVI example:

```cisco
ip routing

vlan 10
 name Users
vlan 20
 name Finance
vlan 30
 name Voice
vlan 99
 name Management

interface vlan 10
 ip address 192.168.10.1 255.255.255.0
 no shutdown

interface vlan 20
 ip address 192.168.20.1 255.255.255.0
 no shutdown

interface vlan 30
 ip address 192.168.30.1 255.255.255.0
 no shutdown
```

**Complete end-to-end campus example**

This example ties together access ports, a trunk, an SVI gateway, and host addressing.

Topology:

- PC1 in VLAN 10
- PC2 in VLAN 20
- IP phone in VLAN 30 on the same access port as a PC
- Multilayer switch provides default gateways

Switch configuration:

```cisco
vlan 10
 name Users
vlan 20
 name Finance
vlan 30
 name Voice
vlan 99
 name Management

ip routing

interface gigabitEthernet1/0/10
 description PC1
 switchport mode access
 switchport access vlan 10
 spanning-tree portfast
 spanning-tree bpduguard enable

interface gigabitEthernet1/0/11
 description PC2
 switchport mode access
 switchport access vlan 20
 spanning-tree portfast
 spanning-tree bpduguard enable

interface gigabitEthernet1/0/12
 description Phone + PC
 switchport mode access
 switchport access vlan 20
 switchport voice vlan 30
 spanning-tree portfast
 spanning-tree bpduguard enable

interface gigabitEthernet1/0/48
 description Uplink to Access Switch
 switchport mode trunk
 switchport trunk native vlan 99
 switchport trunk allowed vlan 10,20,30,99

interface vlan 10
 ip address 192.168.10.1 255.255.255.0
 no shutdown

interface vlan 20
 ip address 192.168.20.1 255.255.255.0
 no shutdown

interface vlan 30
 ip address 192.168.30.1 255.255.255.0
 no shutdown

interface vlan 99
 ip address 192.168.99.1 255.255.255.0
 no shutdown
```

Host addressing:

- PC1: `192.168.10.10/24`, default gateway `192.168.10.1`
- PC2: `192.168.20.10/24`, default gateway `192.168.20.1`
- Phone: receives voice VLAN membership and often gets DHCP from the voice subnet
- Management host: `192.168.99.10/24`, default gateway `192.168.99.1`

Why this works:

- Each VLAN has a Layer 3 gateway.
- The trunk carries all required VLANs between switches.
- The access ports place endpoints into the correct broadcast domains.
- The switch routes between VLANs because `ip routing` is enabled.

**SVI operational state and troubleshooting**

An SVI does not always come up just because you configure it. On a multilayer switch, an SVI is typically **up/up** only when the VLAN exists and at least one port in that VLAN is active, or the VLAN is otherwise present and operational on the switch platform.

Common reasons an SVI stays down:

- The VLAN does not exist.
- No active Layer 2 ports are in that VLAN.
- The trunk does not carry the VLAN to an active downstream port.
- The SVI is administratively shut down.
- The VLAN is pruned or missing from the allowed VLAN list.
- The physical interface or port-channel carrying the VLAN is down.

Useful verification commands:

```cisco
show ip interface brief
show vlan brief
show interfaces trunk
show interfaces vlan 10
show running-config interface vlan 10
```

Troubleshooting logic:

1. Confirm the VLAN exists with `show vlan brief`.
2. Confirm at least one access port or trunk path is active for that VLAN.
3. Check whether the SVI is administratively enabled.
4. Verify the trunk allows the VLAN.
5. Verify the host is in the correct VLAN and subnet.
6. Confirm the default gateway matches the SVI IP address.

> An SVI can remain down even with a correct IP address if the VLAN has no active member ports.

**Common trunk and VLAN misconfiguration symptoms**

Misconfigurations often look like routing problems but are actually Layer 2 issues.

Typical symptoms:

- Hosts in the same VLAN cannot communicate across switches.
- Inter-VLAN routing fails for one VLAN but not others.
- The gateway ping fails from hosts in a specific VLAN.
- Only some VLANs pass over the uplink.
- Voice works but data does not, or vice versa.

Common causes:

- Wrong access VLAN on the port.
- Missing VLAN on the trunk allowed list.
- Native VLAN mismatch.
- Trunk not actually formed.
- SVI down because no active port exists in the VLAN.
- Host default gateway set incorrectly.

Useful checks:

```cisco
show interfaces switchport
show interfaces trunk
show mac address-table vlan 10
show ip route
ping 192.168.10.10 source vlan 10
```

**EtherChannel in campus trunk design**

EtherChannel bundles multiple physical links into one logical link. In campus networks, it is often used between access and distribution switches or between distribution switches to increase bandwidth and redundancy.

Benefits:

- Higher aggregate throughput.
- Redundancy if one physical link fails.
- STP (Spanning Tree Protocol) sees one logical port-channel instead of multiple parallel links.

Common configuration pattern:

```cisco
interface range gigabitEthernet1/0/49-50
 channel-group 1 mode active

interface port-channel1
 switchport mode trunk
 switchport trunk native vlan 99
 switchport trunk allowed vlan 10,20,30,99
```

Notes:

- Use the same trunk settings on all member links.
- LACP (Link Aggregation Control Protocol) is preferred in many designs because it negotiates the bundle.
- If member links disagree on speed, duplex, VLANs, or trunk mode, the bundle may not form correctly.

**First-hop redundancy for default gateways**

In enterprise campus designs, the default gateway is often made redundant with **HSRP** (Hot Standby Router Protocol), **VRRP** (Virtual Router Redundancy Protocol), or **GLBP** (Gateway Load Balancing Protocol).

Why it matters:

- A single SVI or router gateway can become a single point of failure.
- First-hop redundancy lets hosts keep the same default gateway IP while the active device changes.

Conceptual use in a distribution pair:

- Switch A and Switch B both have SVIs for VLAN 10.
- They share a virtual gateway IP, such as `192.168.10.1`.
- One switch forwards traffic while the other stands by.

Example HSRP-style idea:

```cisco
interface vlan 10
 ip address 192.168.10.2 255.255.255.0
 standby 10 ip 192.168.10.1
 standby 10 priority 110
 standby 10 preempt
```

CCNA-level takeaway:

- Know the purpose of first-hop redundancy.
- Understand that it protects the default gateway role.
- Recognize that it is common in campus distribution designs even if not deeply configured in every lab.

**ACLs on SVIs for campus segmentation**

Access control lists (ACLs) on SVIs let you filter traffic between VLANs at the routing boundary. This is a common way to enforce segmentation policy in a campus.

Typical policy examples:

- Finance VLAN can reach accounting servers but not user VLANs.
- Guest VLAN can reach the internet but not internal subnets.
- Voice VLAN can reach call manager and DHCP services only.
- Management VLAN can reach network devices via SSH and SNMP.

Example ACL applied inbound on an SVI:

```cisco
ip access-list extended FINANCE-IN
 permit tcp 192.168.20.0 0.0.0.255 host 192.168.50.10 eq 443
 permit udp 192.168.20.0 0.0.0.255 host 192.168.50.20 eq 53
 deny ip 192.168.20.0 0.0.0.255 192.168.10.0 0.0.0.255
 permit ip 192.168.20.0 0.0.0.255 any

interface vlan 20
 ip access-group FINANCE-IN in
```

Design notes:

- Apply ACLs close to the source when practical.
- Use inbound ACLs on SVIs to control traffic entering the routing boundary from a VLAN.
- Test carefully because ACLs can block return traffic if written too narrowly.

**Practical verification workflow**

When validating a multi-VLAN campus design, check Layer 1, Layer 2, and Layer 3 in order.

1. Verify physical links are up.
2. Verify access ports are in the correct VLAN.
3. Verify trunks are up and carrying the required VLANs.
4. Verify SVIs or router subinterfaces are up/up.
5. Verify hosts have correct IP addresses and gateways.
6. Verify routing and ACL policy.

Useful commands:

```cisco
show cdp neighbors
show interfaces status
show vlan brief
show interfaces trunk
show ip interface brief
show ip route
show access-lists
```

**Design best practices**

- Keep user, voice, management, and server traffic in separate VLANs.
- Use trunks only where multiple VLANs must cross a link.
- Prune unused VLANs from trunks.
- Avoid using VLAN `1` for user traffic.
- Use a dedicated native VLAN and keep it consistent.
- Enable `portfast` and `bpduguard` on edge ports.
- Prefer Layer 3 switching at the distribution layer for scalable inter-VLAN routing.
- Use EtherChannel for redundant uplinks where appropriate.
- Add ACLs at SVIs when segmentation policy is required.
- Plan for gateway redundancy with HSRP, VRRP, or GLBP in larger campus designs.

**CCNA exam focus**

For CCNA, you should be able to:

- Explain why VLANs segment broadcast domains.
- Configure access ports, trunks, and voice VLANs.
- Build inter-VLAN routing with router-on-a-stick or SVIs.
- Identify native VLAN and allowed VLAN issues.
- Troubleshoot an SVI that is down.
- Recognize the role of EtherChannel and first-hop redundancy in campus designs.
- Apply basic ACLs to control traffic between VLANs.

**Quick reference**

- **Access port**: one data VLAN for an endpoint.
- **Voice VLAN**: separate VLAN for IP phone traffic on the same port.
- **Trunk**: carries multiple VLANs between network devices.
- **Native VLAN**: untagged VLAN on an 802.1Q trunk.
- **SVI**: Layer 3 interface for a VLAN on a multilayer switch.
- **Port-channel**: logical interface formed by EtherChannel.
- **First-hop redundancy**: shared default gateway availability across devices.
