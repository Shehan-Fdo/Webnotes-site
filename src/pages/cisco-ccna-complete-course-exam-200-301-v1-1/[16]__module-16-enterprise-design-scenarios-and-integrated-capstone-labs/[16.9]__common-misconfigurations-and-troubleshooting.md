## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Troubleshooting mindset**

Troubleshooting at CCNA level is about isolating the failure domain, verifying assumptions, and checking the most likely misconfigurations first. Start with the physical layer, then move through data link, network, and services. In practice, you often find the issue in a small set of recurring mistakes: wrong cabling, interface shutdown, VLAN mismatch, missing default gateway, bad ACL direction, or incomplete routing configuration.

A useful workflow is:

- Confirm the symptom and scope.
- Check interface status and error counters.
- Verify Layer 2 adjacency, VLAN membership, and trunking.
- Verify Layer 3 addressing, routing, and next-hop reachability.
- Check policy features such as ACLs, NAT, DHCP, and wireless security.

**Layer 1 and Layer 2 cabling issues**

Physical problems are still common and often look like “network” failures. If a link is down, unstable, or slow, verify the cable type, port type, and negotiated speed/duplex before moving to higher layers.

Common issues include:

- **Wrong cable type**
  - Straight-through, crossover, rollover, fiber, and console cables serve different purposes.
  - Modern Ethernet ports often support auto-MDIX, but you should not assume it is enabled everywhere.
- **Speed/duplex mismatch**
  - One side may be forced to `100/full` while the other is set to auto or `100/half`.
  - Mismatches can cause late collisions, CRC errors, drops, and poor throughput.
- **Auto-negotiation failure**
  - If one side is forced and the other is auto, the link may come up but behave poorly.
  - Best practice is to let both sides auto-negotiate unless a design requires fixed settings.
- **Wrong port type**
  - A host connected to a trunk port will not behave as expected.
  - A switch-to-switch link configured as access will block VLANs that should traverse the link.
- **Administrative shutdown**
  - Interfaces can be down because they are manually disabled with `shutdown`.

Useful verification commands:

```bash
show interfaces status
show interfaces
show interfaces counters
show running-config interface gigabitEthernet0/1
```

Look for:

- `administratively down` versus `down/down`
- Speed and duplex values
- Input errors, CRC errors, runts, giants, and output drops
- Unexpected VLAN assignment or trunk state

> `show interfaces counters errors` exists on some platforms, but it is not universally available. Prefer `show interfaces` and `show interfaces counters` when you need broadly supported verification.

**Switching misconfigurations**

Switching problems often involve VLANs, trunks, EtherChannel, and spanning tree. These issues usually appear as “some hosts can talk, others cannot” or “the link is up but traffic does not pass.”

**VLAN and access port mistakes**

A host in the wrong VLAN cannot reach devices in the intended broadcast domain. Common causes include:

- Access port assigned to the wrong VLAN
- VLAN not created on the switch
- VLAN pruned or absent on the trunk path
- Native VLAN mismatch on a trunk
- Management SVI placed in the wrong VLAN

Verification commands:

```bash
show vlan brief
show interfaces switchport
show interfaces trunk
show running-config interface gigabitEthernet0/10
```

Practical example:

```bash
interface gigabitEthernet0/10
 switchport mode access
 switchport access vlan 20
```

If the host should be in VLAN 10 but the port is in VLAN 20, the host may obtain the wrong DHCP scope or lose access to local resources.

**Trunking and router-on-a-stick issues**

A trunk must carry the VLANs required by downstream devices. Router-on-a-stick depends on correct subinterfaces and 802.1Q encapsulation.

Common misconfigurations:

- Missing subinterface on the router
- Wrong `encapsulation dot1q` VLAN ID
- Native VLAN mismatch between switch and router
- Trunk not allowing the required VLAN
- Physical interface left in access mode instead of trunk mode

Example configuration:

```bash
interface gigabitEthernet0/0
 no shutdown

interface gigabitEthernet0/0.10
 encapsulation dot1q 10
 ip address 192.168.10.1 255.255.255.0

interface gigabitEthernet0/0.20
 encapsulation dot1q 20
 ip address 192.168.20.1 255.255.255.0
```

On the switch:

```bash
interface gigabitEthernet0/1
 switchport mode trunk
 switchport trunk allowed vlan 10,20
```

If VLAN 20 is missing from the allowed list, hosts in VLAN 20 may fail to reach the router even though the physical link is up.

**EtherChannel problems**

EtherChannel bundles multiple physical links into one logical link. All member interfaces must match on key Layer 2 settings, or the bundle will not operate correctly.

Required consistency includes:

- Speed and duplex
- Switchport mode
- Access VLAN or trunk settings
- Native VLAN
- Allowed VLAN list
- EtherChannel protocol and mode

Common symptoms:

- Port-channel does not form
- One or more member links are suspended
- Traffic is forwarded inconsistently
- STP treats links separately instead of as one logical link

Verification commands:

```bash
show etherchannel summary
show interfaces port-channel 1
show running-config interface range gigabitEthernet0/1 - 2
```

Example of a mismatch:

```bash
interface range gigabitEthernet0/1 - 2
 switchport mode trunk
 channel-group 1 mode active
```

If one member is configured as access and the other as trunk, the bundle will not function as intended. EtherChannel requires matching parameters across all member links.

**Spanning Tree Protocol troubleshooting**

Spanning Tree Protocol (STP) prevents Layer 2 loops by blocking redundant paths. Misunderstanding STP output can make a healthy network look broken.

Key concepts:

- The **root bridge** is the switch with the lowest bridge ID.
- Non-root switches choose a root port toward the root bridge.
- Redundant links become alternate or blocked ports.
- Designated ports forward traffic on a segment.

Useful commands:

```bash
show spanning-tree
show spanning-tree vlan 10
show spanning-tree root
```

What to check:

- Which switch is the root bridge
- Whether the expected uplink is the root port
- Whether a port is blocking because of a loop-prevention decision
- Whether a port is in an unexpected state due to a VLAN-specific topology

Common root causes:

- Wrong bridge priority
- Unexpected root bridge election
- Miswired redundant links
- PortFast used on a switch-to-switch link
- Edge port connected to another switch

A blocked port is not always a problem. It may be the correct STP decision. The issue is when the blocked port is not the one you expected, or when a critical uplink is blocked because of a topology or priority mistake.

**IPv4 addressing and gateway problems**

A host with the wrong IP address, subnet mask, or default gateway may reach only part of the network or nothing at all.

Common issues:

- Incorrect subnet mask causing wrong local/remote classification
- Wrong default gateway on the host
- Duplicate IP address
- Router interface in the wrong subnet
- Missing `no shutdown` on the router interface

Verification commands:

```bash
show ip interface brief
show running-config interface gigabitEthernet0/0
show ip route
ping
traceroute
```

If a host can reach devices in its own subnet but not remote networks, check the default gateway first. If it cannot reach even local peers, check the IP address, mask, VLAN, and switchport assignment.

**IPv4 DHCP troubleshooting**

DHCP (Dynamic Host Configuration Protocol) failures are a major CCNA troubleshooting topic. A client may show an APIPA address such as `169.254.x.x` when it cannot obtain a lease.

Common causes:

- DHCP server not reachable
- DHCP pool exhausted
- Wrong default gateway in the scope
- Incorrect network statement or excluded addresses
- Missing `ip helper-address` on the router interface for remote clients
- ACL blocking UDP ports 67 and 68
- Relay interface in the wrong VLAN

Server-side example:

```bash
ip dhcp excluded-address 192.168.10.1 192.168.10.20

ip dhcp pool VLAN10
 network 192.168.10.0 255.255.255.0
 default-router 192.168.10.1
 dns-server 8.8.8.8
```

Relay example:

```bash
interface gigabitEthernet0/0.10
 encapsulation dot1q 10
 ip address 192.168.10.1 255.255.255.0
 ip helper-address 10.1.1.10
```

Verification commands:

```bash
show ip dhcp binding
show ip dhcp pool
show ip interface brief
show running-config interface gigabitEthernet0/0.10
```

If remote clients fail to get addresses, verify that the router interface facing the client VLAN has the correct `ip helper-address`. Without relay, DHCP broadcasts do not cross the router.

**IPv6 troubleshooting**

IPv6 issues often involve missing global unicast addressing, missing routing enablement, or misunderstanding how hosts learn their default gateway.

Important points:

- IPv6 hosts typically learn the default gateway from Router Advertisements (RAs), and the gateway is usually the router’s **link-local address**.
- A router must have `ipv6 unicast-routing` enabled to forward IPv6 traffic.
- Static IPv6 routes must point to a valid next hop or exit interface.
- Link-local next hops are common and valid, but the interface must be correct.

Common misconfigurations:

- Missing `ipv6 unicast-routing`
- Interface missing an IPv6 address
- Wrong prefix length
- Incorrect static route next hop
- RA suppression where hosts need SLAAC (Stateless Address Autoconfiguration)
- ACL blocking ICMPv6, which can break neighbor discovery and SLAAC

Example:

```bash
ipv6 unicast-routing

interface gigabitEthernet0/0
 ipv6 address 2001:db8:10:1::1/64
 no shutdown

ipv6 route 2001:db8:20:1::/64 2001:db8:10:2::2
```

Verification commands:

```bash
show ipv6 interface brief
show ipv6 route
show ipv6 neighbors
ping ipv6 2001:db8:10:1::2
```

If hosts can form addresses but cannot reach off-link destinations, check the router’s IPv6 forwarding setting and the default gateway learned from RA. For static IPv6 routing, confirm the next hop is reachable and the route is installed.

**ACL troubleshooting**

Access Control Lists (ACLs) filter traffic based on sequence, direction, protocol, and placement. A small mistake can block more traffic than intended.

Common issues:

- ACL applied in the wrong direction
- ACL applied to the wrong interface
- Standard ACL placed too close to the source
- Extended ACL placed too far from the source
- Rule order causing an earlier match to override a later one
- Missing permit statement for required traffic
- Implicit `deny any` at the end of every ACL

Verification commands:

```bash
show access-lists
show ip interface gigabitEthernet0/0
show running-config | section access-list
```

Example of a web-filtering ACL:

```bash
ip access-list extended WEB-FILTER
 permit tcp 192.168.10.0 0.0.0.255 any eq 80
 permit tcp 192.168.10.0 0.0.0.255 any eq 443
 deny ip any any
```

This ACL blocks all traffic except HTTP and HTTPS from the source subnet. That is valid only if you truly want to deny everything else. In troubleshooting, confirm the intent before assuming the ACL is “wrong.”

Best practices:

- Use extended ACLs near the source.
- Use standard ACLs near the destination.
- Verify hit counts to see whether traffic matches the expected ACEs.
- Remember that ACLs are processed top-down, first match wins.

**Device management and console access**

Management access problems often come from line configuration, authentication settings, or reachability to the management VLAN.

Common issues:

- Incorrect console password or missing `login`
- VTY lines not configured for SSH
- `transport input` set incorrectly
- Missing local user account for SSH
- Wrong management VLAN or SVI address
- Default gateway missing on a Layer 2 switch
- SSH enabled but RSA keys not generated

Example SSH configuration:

```bash
hostname SW1
ip domain-name example.local
username admin secret Str0ngP@ss!
enable secret En@bleP@ss!
crypto key generate rsa modulus 2048
ip ssh version 2

line vty 0 4
 login local
 transport input ssh
```

Important distinction:

- SSH login uses the local user database or AAA (Authentication, Authorization, and Accounting).
- `enable secret` is only needed if you want to enter privileged EXEC mode after login.

For a Layer 2 switch management SVI:

```bash
interface vlan 99
 ip address 192.168.99.2 255.255.255.0
 no shutdown

ip default-gateway 192.168.99.1
```

If the switch cannot be reached from another subnet, verify the SVI, VLAN membership, and `ip default-gateway`. A Layer 2 switch does not route traffic.

**Wireless troubleshooting**

Wireless problems can occur at the radio, SSID, security, VLAN mapping, or DHCP layers. A client may see the SSID but fail to authenticate, associate, or obtain an IP address.

Common distinctions:

- **Authentication**: proving identity, such as with a password or credentials
- **Association**: joining the access point or WLAN
- **Authorization**: being allowed to use the network after authentication

Common issues:

- Wrong SSID name or security settings
- WPA2/WPA3 mismatch
- Incorrect pre-shared key
- SSID mapped to the wrong VLAN
- DHCP relay missing for the WLAN VLAN
- RF interference or weak signal
- Client driver or band mismatch

Practical verification ideas:

```bash
show wlan summary
show ap summary
show interface status
show vlan brief
show ip dhcp binding
```

On many platforms, wireless troubleshooting is controller- or vendor-specific. Focus on the general workflow: confirm the SSID, security policy, VLAN mapping, and DHCP path before investigating RF details.

**Routing and next-hop issues**

Routing failures often appear as “local connectivity works, remote connectivity fails.” The problem may be missing routes, wrong masks, or a next hop that is not reachable.

Common causes:

- Missing static route
- Incorrect network statement in a dynamic routing process
- Wrong next-hop IP address
- Interface down
- Overlapping or incorrect subnet mask
- Route not installed because the next hop is unreachable

Verification commands:

```bash
show ip route
show ip protocols
show running-config | include ^ip route
ping
traceroute
```

Example static route:

```bash
ip route 10.10.20.0 255.255.255.0 192.168.1.2
```

If the next hop is wrong or unreachable, the route may not work even if it appears in the configuration. Always verify both the route and the path to the next hop.

**A practical troubleshooting sequence**

Use a repeatable process instead of guessing.

1. Check the physical link and interface state.
2. Verify VLAN, trunk, or port-channel settings.
3. Confirm IP addressing, masks, and gateways.
4. Check routing tables and next hops.
5. Verify policy features such as ACLs, DHCP, and wireless security.
6. Test with `ping` and `traceroute`.
7. Compare the running configuration to the intended design.

Example workflow for a host that cannot reach the internet:

1. Confirm the host has a valid IP address, mask, and default gateway.
2. Check the switchport VLAN and the uplink trunk.
3. Verify the router subinterface or SVI.
4. Confirm the router has a default route or upstream route.
5. Check NAT and ACLs if they are part of the design.

**Scenario-based examples**

**Example 1: Host gets no DHCP address**

Symptoms:

- Client shows `169.254.x.x`
- Switch port is up
- Other VLANs work

Likely causes:

- Wrong VLAN on the access port
- Missing `ip helper-address`
- DHCP pool exhausted
- ACL blocking DHCP

Fix:

```bash
interface gigabitEthernet0/0.30
 encapsulation dot1q 30
 ip address 192.168.30.1 255.255.255.0
 ip helper-address 10.1.1.10
```

**Example 2: Inter-VLAN routing fails**

Symptoms:

- Hosts in the same VLAN communicate
- Hosts in different VLANs cannot

Likely causes:

- Missing router subinterface
- Wrong `encapsulation dot1q`
- Trunk not allowing the VLAN
- Wrong default gateway on the host

Fix:

```bash
interface gigabitEthernet0/0.20
 encapsulation dot1q 20
 ip address 192.168.20.1 255.255.255.0
```

**Example 3: SSH works but privileged commands fail**

Symptoms:

- You can log in with SSH
- `enable` fails or prompts for a missing password

Likely causes:

- No `enable secret`
- User lacks privilege level 15
- AAA policy restricts privilege escalation

Fix:

```bash
enable secret En@bleP@ss!
username admin privilege 15 secret Str0ngP@ss!
```

**Example 4: Wireless client sees SSID but cannot connect**

Symptoms:

- SSID is visible
- Authentication fails or client gets no IP

Likely causes:

- Wrong PSK
- Security mismatch
- SSID-to-VLAN mapping error
- DHCP relay missing on the WLAN VLAN

Fix the WLAN policy, then verify DHCP reachability on the mapped VLAN.

**High-value verification commands**

Use these commands often during CCNA troubleshooting:

```bash
show ip interface brief
show interfaces status
show interfaces trunk
show vlan brief
show etherchannel summary
show spanning-tree
show ip route
show ipv6 interface brief
show ipv6 route
show access-lists
show ip dhcp binding
show running-config
```

**What to remember for the exam**

- Start with the simplest likely cause and work outward.
- Match the command to the layer you are testing.
- Verify configuration on both ends of a link when possible.
- Do not assume a link is healthy just because it is up.
- Check direction, VLAN membership, and next-hop reachability before changing multiple things at once.
- Treat ACLs, DHCP, and routing as common sources of “it should work” failures.
