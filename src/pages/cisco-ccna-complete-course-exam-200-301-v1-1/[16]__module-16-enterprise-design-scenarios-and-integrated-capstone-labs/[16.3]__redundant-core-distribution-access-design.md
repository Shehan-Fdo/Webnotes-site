## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Campus redundancy design goals**

A redundant campus design keeps users connected when a link, switch, or gateway fails. In a classic hierarchical campus, the access layer connects endpoints, the distribution layer aggregates access switches and enforces policy, and the core layer provides fast transport between distribution blocks. The design choice is not just about adding more links; it is about controlling failure domains, avoiding loops, and making failover predictable.

At CCNA level, you should understand when to use a **collapsed core** versus a **three-tier design**:

- **Collapsed core** combines core and distribution into one layer.
  - Use it in smaller campuses.
  - It reduces cost and complexity.
  - It creates a larger shared failure domain than a full three-tier design.
- **Three-tier design** separates core, distribution, and access.
  - Use it when scale, policy separation, and resiliency matter more.
  - It improves modularity and makes troubleshooting easier.
  - It adds more devices and operational complexity.

A practical rule is to prefer routed links and smaller failure domains as the network grows. Use Layer 2 redundancy where you need VLAN extension and gateway continuity, but use Layer 3 redundancy where you want to eliminate spanning tree dependency and reduce broadcast propagation.

**Why redundancy needs control**

Redundant links create alternate paths, but Layer 2 does not naturally prevent loops. Without a loop-prevention mechanism, broadcast, multicast, and unknown unicast frames can circulate indefinitely and overwhelm the network. This is why **Spanning Tree Protocol** (STP) and its faster variants exist.

In redundant campus designs, you usually combine:

- **STP** to prevent Layer 2 loops.
- **EtherChannel** to bundle parallel links and reduce blocked ports.
- **First Hop Redundancy Protocols** (FHRPs) such as HSRP, VRRP, or GLBP to keep the default gateway available.
- **Layer 3 routed uplinks** where you want deterministic forwarding and no STP dependency.

**Layer 2 traffic behavior in redundant topologies**

When a switch receives a broadcast, multicast, or unknown unicast frame on a VLAN, it floods that frame out all forwarding ports in that VLAN except the ingress port. In a redundant Layer 2 topology, that flooding can reach multiple switches and return to the source if loops exist. STP blocks some ports to break the loop, but the network still relies on careful design so that the active forwarding path matches the intended topology.

Key implications:

- **Broadcast traffic** is flooded to all ports in the VLAN.
- **Multicast traffic** is flooded unless multicast control features are in place.
- **Unknown unicast traffic** is flooded until the destination MAC address is learned.
- **Asymmetric routing** can occur when Layer 2 and FHRP forwarding paths do not align, which can create suboptimal traffic flow or troubleshooting confusion.

**Spanning Tree Protocol in redundant campus designs**

STP elects a root bridge for each spanning-tree instance and then selects the best path to that root. In modern Cisco campus networks, you commonly see **Rapid PVST+**, which runs a separate Rapid Spanning Tree instance per VLAN and converges faster than classic STP.

Important STP concepts:

- The **root bridge** is the switch with the lowest bridge ID.
- The **bridge ID** combines priority and MAC address.
- The **root port** is the best port on a non-root switch toward the root bridge.
- The **designated port** is the forwarding port chosen for a segment.
- The **alternate port** is a backup path toward the root and is typically blocking in Rapid PVST+.
- The **backup port** can exist on shared segments, though it is less common in typical campus designs.

Rapid PVST+ port behavior is exam relevant because it changes how quickly the network recovers:

- Ports can transition rapidly to forwarding when the topology is stable.
- Edge ports can come up immediately when configured correctly.
- Non-edge ports still perform a handshake and may use proposal/agreement behavior to accelerate convergence.
- During a topology change, Rapid PVST+ updates forwarding state more efficiently than classic STP, but it still depends on correct root placement and consistent design.

**STP port roles and states**

For CCNA, know both the role and the state conceptually.

- **Root port**
  - Best path from a non-root switch to the root bridge.
  - One per VLAN instance on a switch, if the switch is not the root.
- **Designated port**
  - Forwards traffic away from the root on a segment.
  - One designated port per segment per VLAN instance.
- **Alternate port**
  - Provides a backup path to the root.
  - Usually blocks until needed.
- **Disabled port**
  - Administratively down or not participating.

Classic STP states are:

- Blocking
- Listening
- Learning
- Forwarding
- Disabled

Rapid PVST+ simplifies the operational view by using roles and faster transitions, but the exam may still reference the classic states. The practical takeaway is that a port must learn MAC addresses before forwarding user traffic, unless it is an edge port.

**Root bridge placement and design intent**

You should place the root bridge intentionally, usually in the distribution layer, so traffic from access switches has a predictable path. In a dual-distribution design, one distribution switch is often the primary root and the other is the secondary root.

Typical approach:

- Make the primary distribution switch the root for a set of VLANs.
- Make the secondary distribution switch the root for other VLANs if you want load sharing.
- Keep the root close to the default gateway to reduce hairpinning.

Example root placement:

```cisco
conf t
spanning-tree mode rapid-pvst
spanning-tree vlan 10,20 root primary
spanning-tree vlan 30,40 root secondary
end
```

You can also set explicit priorities for more control:

```cisco
conf t
spanning-tree vlan 10 priority 4096
spanning-tree vlan 20 priority 4096
spanning-tree vlan 30 priority 8192
spanning-tree vlan 40 priority 8192
end
```

**EtherChannel and port-channel redundancy**

**EtherChannel** bundles multiple physical links into one logical link called a **port-channel**. This is a major campus design tool because it increases bandwidth, provides link redundancy, and prevents STP from blocking individual parallel links between the same two devices.

Why it matters:

- STP sees the bundle as one logical link.
- All member links can forward traffic together.
- If one physical link fails, the port-channel stays up as long as at least one member remains active.
- It reduces the number of blocked links in redundant Layer 2 designs.

Common use cases:

- Access switch to distribution switch uplinks.
- Distribution-to-distribution links.
- Core interconnects when Layer 2 is still required.

Common negotiation protocols:

- **LACP** (Link Aggregation Control Protocol), standardized and preferred.
- **PAgP** (Port Aggregation Protocol), Cisco proprietary and less common in modern designs.
- Static EtherChannel, which does not negotiate and is more error-prone.

LACP example:

```cisco
conf t
interface range g1/0/1-2
 channel-group 1 mode active
exit

interface port-channel 1
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30
end
```

Operational notes:

- Member interfaces must match speed, duplex, trunking, and allowed VLAN settings.
- A mismatch can cause the bundle to fail or suspend ports.
- Use `show etherchannel summary` to verify status.

```cisco
show etherchannel summary
show interfaces port-channel 1
show lacp neighbor
```

**When to use EtherChannel versus separate links**

Use EtherChannel when you need multiple physical links between the same two switches and want them all to forward traffic. Use separate links when you want STP to block one path intentionally or when the links terminate on different devices and cannot be bundled.

A common exam point is that EtherChannel improves redundancy without creating additional STP blocked links between the same pair of switches. However, it does not remove the need for STP across the broader Layer 2 topology.

**Layer 3 access uplinks and routed design**

A routed access design uses Layer 3 links between access and distribution, which removes STP from those uplinks. This is often preferred when you want better scalability, faster convergence, and smaller Layer 2 domains.

Benefits:

- No STP on routed uplinks.
- No VLAN trunking across the uplink.
- Broadcast domains stay smaller.
- Failures are easier to isolate.

A simple routed uplink example:

```cisco
conf t
interface g1/0/1
 no switchport
 ip address 10.1.1.1 255.255.255.252
 no shutdown

interface g1/0/2
 no switchport
 ip address 10.1.1.5 255.255.255.252
 no shutdown
end
```

On the distribution switch, you would configure the peer addresses and run a routing protocol or static routes:

```cisco
conf t
interface g1/0/1
 no switchport
 ip address 10.1.1.2 255.255.255.252
 no shutdown

interface g1/0/2
 no switchport
 ip address 10.1.1.6 255.255.255.252
 no shutdown

ip route 0.0.0.0 0.0.0.0 10.1.1.1
end
```

Operational notes:

- Routed uplinks are common between distribution and core.
- They are also useful from access to distribution in larger designs.
- If you need VLAN extension to the access layer, you may still use Layer 2 trunks there.

**FHRP: HSRP, VRRP, and GLBP**

An FHRP provides a virtual default gateway so hosts can keep sending traffic even if a physical gateway fails.

Common protocols:

- **HSRP** (Hot Standby Router Protocol)
  - Cisco proprietary.
  - One active router and one standby router.
- **VRRP** (Virtual Router Redundancy Protocol)
  - Open standard.
  - One master and one or more backups.
- **GLBP** (Gateway Load Balancing Protocol)
  - Cisco proprietary.
  - Provides gateway redundancy and load sharing.

For CCNA, HSRP is the most commonly emphasized. You should know that the virtual IP address is what hosts use as their default gateway, while the active device forwards traffic.

HSRP example:

```cisco
conf t
interface vlan 10
 ip address 192.168.10.2 255.255.255.0
 standby 10 ip 192.168.10.1
 standby 10 priority 110
 standby 10 preempt
 standby 10 timers 1 3
 standby 10 track g1/0/1 20
end
```

What the tuning does:

- `priority` determines which router becomes active.
- `preempt` allows a higher-priority router to reclaim the active role after recovery.
- `timers` adjust hello and hold intervals for faster failover.
- `track` lowers priority if an interface fails, which helps the standby take over.

VRRP example:

```cisco
conf t
interface vlan 10
 ip address 192.168.10.3 255.255.255.0
 vrrp 10 ip 192.168.10.1
 vrrp 10 priority 110
 vrrp 10 preempt
end
```

GLBP is useful when you want multiple routers to actively forward for the same virtual gateway, but it is less central to CCNA than HSRP and VRRP.

**Tracking and failover behavior**

Tracking improves failover accuracy by tying gateway priority to real path health.

Common tracking options:

- **Interface tracking**
  - Reduce priority if an uplink fails.
- **Object tracking**
  - Track a route, interface, or IP SLA object.
  - More flexible than simple interface tracking.

Example with object tracking:

```cisco
conf t
track 1 interface g1/0/1 line-protocol
track 2 ip route 0.0.0.0/0 reachability

interface vlan 10
 standby 10 track 1 decrement 20
 standby 10 track 2 decrement 30
end
```

This matters because a router may still be physically up while losing its upstream path. Tracking prevents it from staying active when it should not.

**STP and FHRP alignment**

A strong campus design aligns the STP root with the active FHRP gateway to keep traffic paths efficient.

Why alignment matters:

- Hosts send traffic to the virtual gateway.
- The active gateway should be close to the STP root for that VLAN.
- Misalignment can cause traffic to hairpin across the distribution layer.
- Misalignment can also create asymmetric paths that complicate troubleshooting.

Example design pattern:

- Distribution switch A is STP root and HSRP active for VLAN 10.
- Distribution switch B is STP secondary and HSRP standby for VLAN 10.
- For VLAN 20, reverse the roles if you want load sharing.

This creates predictable forwarding and reduces unnecessary cross-switch traffic.

**STP protections and edge cases**

Redundant Layer 2 designs need safeguards against accidental loops and malicious or misconfigured devices.

Important protections:

- **BPDU Guard**
  - Shuts down an edge port if it receives a Bridge Protocol Data Unit (BPDU).
  - Use on access ports that should never connect to another switch.
- **BPDU Filter**
  - Suppresses BPDU transmission or reception depending on configuration.
  - Use carefully; it can hide loops and is riskier than BPDU Guard.
- **Root Guard**
  - Prevents a downstream switch from becoming root.
  - Use on ports where you never want a better root to appear.
- **Loop Guard**
  - Protects against unidirectional link failures that could cause a blocked port to incorrectly transition to forwarding.
  - Useful on non-edge redundant links.

Typical access-port hardening:

```cisco
conf t
interface range f0/1-24
 spanning-tree portfast
 spanning-tree bpduguard enable
end
```

Root guard example on a distribution-facing port:

```cisco
conf t
interface g1/0/24
 spanning-tree guard root
end
```

Loop guard example:

```cisco
conf t
conf t
spanning-tree loopguard default
end
```

Important caution:

> Do not use BPDU filter casually on ports that could ever connect to another switch. It can suppress the very signaling STP needs to prevent loops.

**PortFast and edge ports**

**PortFast** is used on access ports connected to end devices. It allows the port to move to forwarding quickly because end hosts do not participate in STP.

Use it when:

- The port connects to a PC, printer, phone, or server NIC that is not bridging traffic.
- You want faster DHCP and network access at boot.

Do not use it on switch-to-switch links unless the design explicitly requires it and you understand the consequences.

Example:

```cisco
conf t
interface range f0/1-24
 spanning-tree portfast
 spanning-tree bpduguard enable
end
```

**Topology change handling**

In Rapid PVST+, topology changes are handled more efficiently than in classic STP. When a link fails or a port changes state, the network updates forwarding information faster, which reduces disruption. Still, the design should minimize unnecessary topology changes by using stable root placement, EtherChannel for parallel links, and routed links where possible.

Practical effects of a topology change:

- MAC address tables may age out faster on affected switches.
- Traffic may briefly flood while the network relearns paths.
- A poorly designed Layer 2 domain can amplify the impact of a single failure.

**Redundancy tradeoffs: Layer 2 versus Layer 3**

Choose the redundancy model based on scale and operational goals.

| Design choice | Best when | Main benefit | Main drawback |
|---|---|---|---|
| STP-based Layer 2 redundancy | You need VLAN extension and simple gateway continuity | Familiar, flexible, supports shared VLANs | Blocked links, larger failure domains |
| Routed Layer 3 redundancy | You want scale and fast convergence | No STP on uplinks, smaller broadcast domains | Requires routing design and gateway planning |
| EtherChannel on Layer 2 links | You need parallel links between the same devices | All links forward, no STP blocking between peers | Both ends must match exactly |
| FHRP at distribution | You need a resilient default gateway | Hosts keep using one virtual IP | Must align with STP and track failures |

A common CCNA scenario is an access switch with two uplinks to two distribution switches. If the uplinks are Layer 2 trunks, STP blocks one path per VLAN unless you use EtherChannel to the same device. If the uplinks are routed, STP is removed from the uplinks, but you need routing and FHRP or dynamic routing to maintain gateway reachability.

**Practical campus design patterns**

1. **Small campus**
   - Use a collapsed core.
   - Keep VLANs limited.
   - Use HSRP or VRRP at the distribution layer.
   - Use EtherChannel for access uplinks where appropriate.

2. **Medium campus**
   - Use a three-tier design or a partial three-tier design.
   - Use routed links between distribution and core.
   - Keep Layer 2 domains at the access layer.
   - Place STP root and FHRP active roles intentionally.

3. **Large campus**
   - Prefer routed access or at least routed distribution-to-core links.
   - Minimize VLAN stretch.
   - Use tracking and clear failure-domain boundaries.
   - Standardize on Rapid PVST+ or another consistent STP strategy where Layer 2 remains.

**Verification and troubleshooting commands**

Use these commands to confirm the design is behaving as intended:

```cisco
show spanning-tree
show spanning-tree vlan 10
show spanning-tree root
show spanning-tree interface g1/0/1 detail
show etherchannel summary
show interfaces trunk
show standby brief
show vrrp brief
show ip interface brief
show ip route
```

What to look for:

- The intended switch is root for the VLAN.
- Access ports are forwarding quickly with PortFast.
- EtherChannel members are bundled and not suspended.
- The active FHRP router matches the design.
- Routed uplinks have correct IP addressing and routing adjacency.

**Exam-focused takeaways**

- STP prevents Layer 2 loops, but it also blocks redundant paths unless you use EtherChannel or redesign with Layer 3.
- Rapid PVST+ converges faster than classic STP and uses per-VLAN instances.
- Root bridge placement should be intentional, usually in distribution.
- EtherChannel is the preferred way to use multiple parallel links between the same two devices.
- HSRP, VRRP, and GLBP provide default-gateway redundancy; tracking and preemption affect failover behavior.
- Routed access or routed distribution uplinks reduce STP dependence and shrink failure domains.
- STP protections such as BPDU Guard, Root Guard, and Loop Guard are essential in redundant campus designs.
- Align STP root placement with FHRP active gateways to avoid suboptimal or asymmetric traffic paths.
