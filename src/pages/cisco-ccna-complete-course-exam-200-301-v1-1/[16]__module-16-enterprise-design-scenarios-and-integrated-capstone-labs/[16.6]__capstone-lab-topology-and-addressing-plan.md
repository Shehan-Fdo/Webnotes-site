## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Capstone lab purpose**

A capstone lab brings together switching, routing, addressing, and verification into one design. You use it to prove that your network plan is internally consistent before you start configuring devices. The topology and addressing plan is the foundation: if the IP plan, VLAN plan, and gateway placement are wrong, the rest of the lab becomes harder to troubleshoot.

In a CCNA context, you should be able to read a topology, assign subnets to each segment, map hosts to VLANs, and decide where Layer 2 ends and Layer 3 begins. This is also where you practice documenting the network clearly enough that another engineer could build it from your notes.

**What a good topology plan includes**

A complete plan usually defines:

- **Devices**: routers, switches, access points, servers, and end hosts.
- **Links**: which interfaces connect, and whether each link is access, trunk, routed, or WAN.
- **VLANs**: which user groups or services belong together at Layer 2.
- **Subnets**: one IP network per broadcast domain or routed segment.
- **Gateways**: the default gateway for each subnet, usually the SVI or router interface.
- **Management**: IP addressing for device management, DNS, and remote access.
- **Special services**: DHCP, NAT, NTP, and any server networks.

A strong plan avoids overlap, leaves room for growth, and makes troubleshooting easier because every address has a purpose.

**Topology design principles**

When you design the lab topology, start with the traffic boundaries. A broadcast domain should be small enough to manage but large enough to support the required hosts. A routed boundary is the right place to separate departments, security zones, or services.

- **Layer 2 domains** should stay local to a switch or switch stack unless you intentionally extend them with trunks.
- **Layer 3 boundaries** should separate VLANs, WAN links, and server segments.
- **Redundancy** should be visible in the topology if the lab includes it, such as dual uplinks or paired distribution switches.
- **Naming consistency** matters; use predictable names for devices, interfaces, VLANs, and subnets.

A practical design often uses a hierarchical model:

- **Access layer** for end devices
- **Distribution layer** for inter-VLAN routing and policy
- **Core layer** for fast transport, if the topology is large enough

For a CCNA capstone, the topology may be smaller, but the same logic applies.

**Reading the physical and logical topology**

The physical topology shows how devices are cabled. The logical topology shows how traffic flows after VLANs, trunks, routing, and gateways are applied. You need both views to avoid mistakes.

- A switch port connected to a PC is usually an **access port** in one VLAN.
- A switch-to-switch link is often a **trunk** carrying multiple VLANs.
- A router-to-switch link may be a trunk for router-on-a-stick or a routed link if the router has separate interfaces.
- A server may sit in a dedicated VLAN even if it is physically connected to the same switch as users.

When you document the topology, label:

- Interface names, such as `G0/0`, `G0/1`, `F0/24`
- VLAN IDs and names
- IP subnets and masks
- Default gateways
- Special roles, such as DHCP server or management host

**Addressing plan fundamentals**

An addressing plan assigns IP networks to each segment in a way that is easy to understand and scale. The plan should be deterministic, meaning you can infer the subnet from the VLAN or site name.

A common approach is to reserve blocks by function:

- `10.10.10.0/24` for user VLANs
- `10.10.20.0/24` for voice or wireless
- `10.10.30.0/24` for servers
- `10.10.99.0/24` for management

This is only an example pattern; the exact ranges depend on the lab requirements. The important part is consistency.

**Subnetting decisions**

Before assigning addresses, determine how many hosts each subnet needs. Then choose a prefix length that supports the required host count with some growth.

- A `/24` provides 254 usable addresses.
- A `/25` provides 126 usable addresses.
- A `/26` provides 62 usable addresses.
- A `/27` provides 30 usable addresses.
- A `/28` provides 14 usable addresses.

Use smaller subnets when you want tighter control and less waste. Use larger subnets when the segment must support many devices or future expansion.

> Be careful not to assign overlapping subnets to different VLANs or routed interfaces. Overlap causes routing ambiguity and breaks connectivity.

**VLAN-to-subnet mapping**

A clean lab design maps each VLAN to exactly one IP subnet. This makes troubleshooting much easier because the VLAN ID, subnet, and gateway all point to the same segment.

| VLAN | Purpose | Subnet example | Gateway example |
|---|---|---|---|
| 10 | Users | `10.10.10.0/24` | `10.10.10.1` |
| 20 | Voice | `10.10.20.0/24` | `10.10.20.1` |
| 30 | Servers | `10.10.30.0/24` | `10.10.30.1` |
| 99 | Management | `10.10.99.0/24` | `10.10.99.1` |

This pattern is easy to document and verify. If the lab uses router-on-a-stick or SVIs, the gateway address is usually the first usable address, but that is a convention, not a requirement.

**Gateway placement**

The default gateway is the next hop for hosts leaving their local subnet. In a CCNA lab, the gateway is commonly placed on:

- A router subinterface for router-on-a-stick
- A switched virtual interface, or **SVI** (Switched Virtual Interface), on a multilayer switch
- A routed interface on a router for point-to-point or WAN segments

Choose the gateway location based on the topology:

- Use **router-on-a-stick** when a router provides inter-VLAN routing over a trunk.
- Use **SVIs** when a Layer 3 switch performs inter-VLAN routing.
- Use **routed ports** for point-to-point links between Layer 3 devices.

**Example addressing plan**

The following example shows a small campus-style lab with one router, two switches, and multiple VLANs.

```text
R1
- G0/0 connected to SW1 G0/1 as trunk
- G0/1 connected to ISP or WAN

SW1
- F0/1-F0/10 VLAN 10 users
- F0/11-F0/15 VLAN 20 voice
- F0/16-F0/20 VLAN 30 servers
- G0/1 trunk to R1
- G0/2 trunk to SW2

SW2
- F0/1-F0/12 VLAN 10 users
- F0/13-F0/24 VLAN 99 management
- G0/1 trunk to SW1
```

Addressing plan:

- VLAN 10: `10.10.10.0/24`, gateway `10.10.10.1`
- VLAN 20: `10.10.20.0/24`, gateway `10.10.20.1`
- VLAN 30: `10.10.30.0/24`, gateway `10.10.30.1`
- VLAN 99: `10.10.99.0/24`, gateway `10.10.99.1`
- R1 WAN: `203.0.113.2/30`
- ISP next hop: `203.0.113.1/30`

This layout separates user traffic, voice, servers, and management while keeping the plan simple enough to troubleshoot quickly.

**Router-on-a-stick example**

If the router handles inter-VLAN routing, you create one subinterface per VLAN and assign the gateway IP to each subinterface.

```cisco
interface g0/0
 no shutdown

interface g0/0.10
 encapsulation dot1Q 10
 ip address 10.10.10.1 255.255.255.0

interface g0/0.20
 encapsulation dot1Q 20
 ip address 10.10.20.1 255.255.255.0

interface g0/0.30
 encapsulation dot1Q 30
 ip address 10.10.30.1 255.255.255.0

interface g0/0.99
 encapsulation dot1Q 99
 ip address 10.10.99.1 255.255.255.0
```

This design works well in labs because it clearly shows the relationship between VLANs and gateways. The switch port facing the router must be a trunk.

**Layer 3 switch example**

If the lab uses a multilayer switch, you can place the gateways on SVIs and enable routing.

```cisco
ip routing

vlan 10
 name USERS
vlan 20
 name VOICE
vlan 30
 name SERVERS
vlan 99
 name MGMT

interface vlan 10
 ip address 10.10.10.1 255.255.255.0
 no shutdown

interface vlan 20
 ip address 10.10.20.1 255.255.255.0
 no shutdown

interface vlan 30
 ip address 10.10.30.1 255.255.255.0
 no shutdown

interface vlan 99
 ip address 10.10.99.1 255.255.255.0
 no shutdown
```

Use SVIs when you want the switch to route between VLANs at wire speed, subject to platform capabilities. This is common in campus designs and is a frequent CCNA lab pattern.

**Management addressing**

Management addresses let you reach network devices remotely using SSH, SNMP, or web management when enabled. Keep management separate from user traffic when possible.

Typical management practices:

- Assign each switch a unique management IP in a dedicated management VLAN.
- Use a consistent naming scheme for device hostnames.
- Set the default gateway on Layer 2 switches so they can reach remote management hosts.
- Restrict management access with ACLs if the lab requires security controls.

Example on a Layer 2 switch:

```cisco
hostname SW1

interface vlan 99
 ip address 10.10.99.11 255.255.255.0
 no shutdown

ip default-gateway 10.10.99.1
```

A Layer 2 switch does not route, so it needs `ip default-gateway` for off-subnet management access. A Layer 3 switch uses routing instead of `ip default-gateway` for normal traffic.

**Host addressing strategy**

Hosts can use static addressing or DHCP (Dynamic Host Configuration Protocol), depending on the lab requirements.

- Use **static IPs** for routers, switches, servers, and infrastructure devices.
- Use **DHCP** for end-user devices when the lab includes dynamic assignment.
- Reserve a block of addresses for infrastructure and another block for clients.

A common static reservation pattern is:

- `.1` gateway
- `.2` to `.20` infrastructure
- `.21` to `.100` servers or reserved hosts
- `.101` and above for clients

This is only a convention, but it helps you remember where to look during troubleshooting.

**DHCP planning example**

If the router or server provides DHCP, define the scope to match the VLAN subnet and exclude reserved addresses.

```cisco
ip dhcp excluded-address 10.10.10.1 10.10.10.20

ip dhcp pool USERS
 network 10.10.10.0 255.255.255.0
 default-router 10.10.10.1
 dns-server 10.10.30.10
 domain-name lab.local
```

If clients are in a different VLAN from the DHCP server, you may need a DHCP relay using `ip helper-address` on the gateway interface.

```cisco
interface vlan 10
 ip helper-address 10.10.30.10
```

**WAN and point-to-point links**

Point-to-point links usually use small subnets such as `/30` or `/31` when supported. In CCNA labs, `/30` is the most common choice because it is easy to understand and widely supported.

Example:

- `203.0.113.0/30`
  - Network: `203.0.113.0`
  - Usable: `203.0.113.1` and `203.0.113.2`
  - Broadcast: `203.0.113.3`

Use point-to-point addressing for:

- Router-to-router links
- Router-to-ISP links
- Routed uplinks between Layer 3 devices

**Verification workflow**

After you build the plan, verify it before and after configuration. Verification prevents small design errors from becoming long troubleshooting sessions.

1. Confirm the topology matches the intended physical connections.
2. Verify VLAN IDs, trunk links, and access port assignments.
3. Check that each subnet has a unique network address and mask.
4. Confirm each host has the correct default gateway.
5. Test local connectivity first, then remote connectivity.
6. Verify routing tables and neighbor relationships if routing is involved.

Useful commands include:

```cisco
show ip interface brief
show vlan brief
show interfaces trunk
show running-config
show ip route
show cdp neighbors
show lldp neighbors
ping 10.10.10.1
traceroute 10.10.30.10
```

On a switch, `show interfaces trunk` confirms which VLANs traverse the uplink. `show vlan brief` confirms access port membership. `show ip route` confirms that the device knows how to reach remote subnets.

**Troubleshooting common design errors**

Many lab failures come from planning mistakes rather than syntax mistakes.

- **Overlapping subnets** cause routing conflicts and unpredictable reachability.
- **Wrong default gateway** prevents hosts from leaving their local subnet.
- **Trunk mismatch** blocks VLAN traffic between switches or to a router.
- **Missing VLAN** leaves access ports inactive for the intended traffic.
- **Incorrect mask** changes the subnet boundary and breaks host communication.
- **Management VLAN mismatch** makes switches unreachable from the admin network.

When troubleshooting, start with the simplest checks:

- Is the cable connected to the correct interface?
- Is the interface up/up?
- Is the port in the correct VLAN?
- Is the gateway reachable from the host?
- Does the routing table contain the destination network?

**Documentation best practices**

Good documentation is part of the lab deliverable. It should be clear enough that you can rebuild the network later without guessing.

Include:

- A topology diagram with interface labels
- A VLAN table with IDs and names
- An IP addressing table with subnets, gateways, and reserved ranges
- A device inventory with hostnames and management IPs
- Notes about special features such as DHCP, NAT, or ACLs

A concise addressing table might look like this:

| Segment | VLAN | Subnet | Gateway | Notes |
|---|---:|---|---|---|
| Users | 10 | `10.10.10.0/24` | `10.10.10.1` | DHCP clients |
| Voice | 20 | `10.10.20.0/24` | `10.10.20.1` | IP phones |
| Servers | 30 | `10.10.30.0/24` | `10.10.30.1` | Static IPs |
| Management | 99 | `10.10.99.0/24` | `10.10.99.1` | Network devices |

**Practical build checklist**

Use this checklist when you turn the plan into a working lab.

1. Assign hostnames to all devices.
2. Create VLANs and name them consistently.
3. Map access ports to the correct VLANs.
4. Configure trunks and allow the required VLANs.
5. Assign gateway IPs on router subinterfaces or SVIs.
6. Configure host IPs, masks, and default gateways.
7. Add DHCP, DNS, or NAT if the lab requires them.
8. Verify end-to-end connectivity with `ping` and `traceroute`.
9. Save the configuration with `copy running-config startup-config`.

**Why this topic matters for CCNA**

The topology and addressing plan tests whether you can think like a network engineer, not just type commands. You must understand how Layer 2 and Layer 3 boundaries shape the design, how subnets map to VLANs, and how gateways enable communication beyond the local network. In exam and lab settings, a correct plan saves time, reduces errors, and makes every later configuration step more predictable.
