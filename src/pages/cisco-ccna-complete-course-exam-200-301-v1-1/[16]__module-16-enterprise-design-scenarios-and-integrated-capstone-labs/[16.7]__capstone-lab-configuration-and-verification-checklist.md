## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Purpose of the capstone checklist**

A capstone lab ties together the major CCNA domains: switching, routing, IP addressing, VLANs, inter-VLAN routing, default gateways, basic security, and verification. The goal is not just to make the network work once, but to build a repeatable process you can use under exam pressure and in real deployments.

A good checklist helps you:
- Build the network in a logical order.
- Verify each layer before moving on.
- Catch common mistakes early, such as wrong VLAN assignment, missing trunking, or incorrect gateway settings.
- Troubleshoot efficiently by isolating whether the issue is Layer 1, Layer 2, or Layer 3.

**Typical capstone topology**

A common CCNA-style lab includes:
- Two or more switches connected by trunks.
- One router or multilayer switch for inter-VLAN routing.
- Multiple VLANs for user groups and management.
- End devices in different subnets.
- Optional services such as DHCP, SSH, and static routing.

A simple example topology might look like this:

```text
PC-A --- SW1 --- trunk --- SW2 --- PC-B
             \             /
              \--- R1 ----/
```

In this design:
- `SW1` and `SW2` handle VLAN membership and trunking.
- `R1` provides routing between VLANs using router-on-a-stick, or a multilayer switch provides switched virtual interfaces (SVIs).
- PCs use the router or SVI as their default gateway.

**Planning before configuration**

Before you type commands, confirm the addressing and VLAN plan. This prevents rework and makes verification easier.

A practical planning checklist:
- Identify each VLAN ID and purpose.
- Assign an IP subnet to each VLAN.
- Decide which device provides Layer 3 gateways.
- Map access ports to the correct VLANs.
- Identify trunk links and allowed VLANs.
- Reserve management addressing for switches and routers.
- Note any services such as DHCP pools, static routes, or SSH access.

Example planning table:

| VLAN | Purpose | Subnet | Gateway |
|---|---|---|---|
| 10 | Users | `192.168.10.0/24` | `192.168.10.1` |
| 20 | Sales | `192.168.20.0/24` | `192.168.20.1` |
| 99 | Management | `192.168.99.0/24` | `192.168.99.1` |

**Layer 1 and Layer 2 foundation**

Start with physical connectivity and switching basics. If interfaces are down, higher-layer checks are not useful.

Verify:
- Cables are connected to the correct ports.
- Interfaces are not administratively shut down.
- Speed and duplex are appropriate, especially on older or mixed environments.
- Switch ports are assigned to the intended VLANs.
- Trunks are up and carrying the expected VLANs.

Useful commands:
```bash
show ip interface brief
show interfaces status
show interfaces description
show vlan brief
show interfaces trunk
```

What to look for:
- `up/up` on active interfaces.
- Access ports in the correct VLAN.
- Trunk ports showing `trunking` and the expected native VLAN.
- No unexpected err-disabled or down/down interfaces.

> If a host cannot communicate at all, verify Layer 1 and Layer 2 first. A routing problem is unlikely if the interface is down or the VLAN is wrong.

**Switch configuration checklist**

A switch in a capstone lab usually needs hostname, secure access, VLANs, access ports, trunks, and management reachability.

Common configuration tasks:
- Set the hostname.
- Disable DNS lookup if you want to avoid delays from mistyped commands.
- Configure a secure enable secret.
- Create required VLANs.
- Assign access ports to the correct VLANs.
- Configure trunk links.
- Set the management SVI and default gateway if the switch is Layer 2.
- Secure remote access with SSH.

Example Layer 2 switch configuration:

```cisco
hostname SW1
no ip domain-lookup
enable secret class123

vlan 10
 name USERS
vlan 20
 name SALES
vlan 99
 name MANAGEMENT

interface range fa0/1 - 10
 switchport mode access
 switchport access vlan 10

interface range fa0/11 - 20
 switchport mode access
 switchport access vlan 20

interface gi0/1
 switchport mode trunk
 switchport trunk allowed vlan 10,20,99

interface vlan 99
 ip address 192.168.99.2 255.255.255.0
 no shutdown

ip default-gateway 192.168.99.1
```

Verification commands:
```bash
show vlan brief
show interfaces trunk
show ip interface brief
show running-config | section vlan
```

Key checks:
- Access ports are not left in VLAN 1 by accident.
- The management SVI is in the correct VLAN.
- The switch has a default gateway if you need to manage it from another subnet.
- Trunks allow only the VLANs you actually need.

**Router-on-a-stick checklist**

If you use a router for inter-VLAN routing, configure one physical interface as a trunk and create subinterfaces for each VLAN.

Core requirements:
- The physical interface must be up.
- Each subinterface needs `encapsulation dot1Q <vlan-id>`.
- Each subinterface needs the correct IP address.
- Hosts must use the subinterface IP as their default gateway.

Example:

```cisco
interface g0/0
 no shutdown

interface g0/0.10
 encapsulation dot1Q 10
 ip address 192.168.10.1 255.255.255.0

interface g0/0.20
 encapsulation dot1Q 20
 ip address 192.168.20.1 255.255.255.0

interface g0/0.99
 encapsulation dot1Q 99
 ip address 192.168.99.1 255.255.255.0
```

Verification commands:
```bash
show ip interface brief
show running-config interface g0/0.10
show interfaces g0/0
show arp
```

Common mistakes:
- Missing `no shutdown` on the physical interface.
- Wrong VLAN ID in `encapsulation dot1Q`.
- Incorrect default gateway on the host.
- Native VLAN mismatch between switch and router trunking expectations.

**Multilayer switch checklist**

If a multilayer switch performs routing, you typically use SVIs and enable Layer 3 forwarding.

Core requirements:
- Create the VLANs first.
- Configure an SVI for each routed VLAN.
- Enable `ip routing`.
- Ensure access ports and trunks place traffic into the correct VLANs.

Example:

```cisco
vlan 10
vlan 20
vlan 99

interface vlan 10
 ip address 192.168.10.1 255.255.255.0
 no shutdown

interface vlan 20
 ip address 192.168.20.1 255.255.255.0
 no shutdown

interface vlan 99
 ip address 192.168.99.1 255.255.255.0
 no shutdown

ip routing
```

Verification commands:
```bash
show ip interface brief
show ip route
show vlan brief
show interfaces vlan 10
```

Important behavior:
- An SVI stays down if the VLAN does not exist or no active port is in that VLAN.
- `ip routing` is required for inter-VLAN routing on a multilayer switch.
- The SVI IP address becomes the default gateway for hosts in that VLAN.

**IP addressing and host configuration**

Host configuration is often the source of simple but costly errors. Confirm that each device has the correct IP address, subnet mask, and default gateway.

Checklist for each host:
- IP address matches the assigned subnet.
- Subnet mask is correct.
- Default gateway points to the local VLAN gateway.
- DNS server is set if name resolution is required.

Example host settings:
- PC-A: `192.168.10.10/24`, gateway `192.168.10.1`
- PC-B: `192.168.20.10/24`, gateway `192.168.20.1`

Verification from the host:
```bash
ipconfig /all
ping 192.168.10.1
ping 192.168.20.1
tracert 192.168.20.10
```

On Linux or macOS:
```bash
ip addr
ip route
ping -c 4 192.168.10.1
traceroute 192.168.20.10
```

**Routing checklist**

Once VLANs and gateways are correct, verify routing between subnets and toward any upstream network.

If you use static routes:
- Confirm the destination network and mask are correct.
- Confirm the next hop is reachable.
- Verify return routes exist.

Example static route:
```cisco
ip route 10.10.0.0 255.255.0.0 192.168.99.254
```

Verification commands:
```bash
show ip route
show ip route static
show running-config | include ^ip route
```

What to confirm:
- Connected routes appear automatically.
- Static routes are installed in the routing table.
- The next hop is not pointing to an unreachable address.
- Return traffic can find its way back.

**DHCP checklist**

If the lab includes DHCP, verify that clients receive the correct addressing information.

On a router DHCP server:
```cisco
ip dhcp excluded-address 192.168.10.1 192.168.10.20

ip dhcp pool VLAN10
 network 192.168.10.0 255.255.255.0
 default-router 192.168.10.1
 dns-server 8.8.8.8
```

Verification commands:
```bash
show ip dhcp binding
show ip dhcp pool
show running-config | section dhcp
```

Checklist:
- Excluded addresses protect gateways and infrastructure devices.
- The pool network matches the VLAN subnet.
- The default router matches the SVI or subinterface IP.
- The client renews its lease after changes.

**Device security checklist**

A capstone lab should include basic device hardening. These settings are common in CCNA labs and help reinforce secure management practices.

Recommended items:
- Set `enable secret`.
- Configure local user authentication.
- Use SSH instead of Telnet.
- Set a domain name before generating RSA keys.
- Restrict VTY access if needed.
- Encrypt plaintext passwords where appropriate.

Example SSH configuration:
```cisco
hostname SW1
ip domain-name lab.local
username admin secret AdminP@ss123
crypto key generate rsa modulus 2048
ip ssh version 2

line vty 0 4
 login local
 transport input ssh
```

Verification commands:
```bash
show ip ssh
show running-config | section line\ vty
show users
```

Security checks:
- SSH works from a management host.
- Telnet is disabled unless explicitly required.
- Console and VTY access use the intended authentication method.
- Passwords are not left in clear text when avoidable.

**Verification workflow**

Use a layered verification process instead of random testing. Start local, then move outward.

A practical order:
1. Verify interface status.
2. Verify VLAN membership and trunking.
3. Verify SVI or subinterface addressing.
4. Verify host IP settings.
5. Verify gateway reachability.
6. Verify inter-VLAN connectivity.
7. Verify routing to remote networks.
8. Verify management access and services.

Useful commands by layer:

```bash
show ip interface brief
show vlan brief
show interfaces trunk
show cdp neighbors
show mac address-table
show ip route
show arp
```

Interpretation tips:
- `show cdp neighbors` helps confirm physical adjacency on Cisco devices.
- `show mac address-table` confirms whether frames are arriving on the expected ports.
- `show arp` confirms Layer 3 resolution after successful communication.
- `show ip route` confirms whether the device knows where to send traffic.

**Troubleshooting checklist**

When something fails, isolate the fault domain.

Common symptoms and likely causes:
- No link light: cable, port shutdown, wrong interface, or hardware issue.
- Same-VLAN hosts cannot communicate: wrong VLAN assignment, access port issue, or trunk problem.
- Hosts in different VLANs cannot communicate: missing gateway, missing SVI/subinterface, or routing disabled.
- Switch management unreachable: wrong SVI IP, missing default gateway, or VLAN 99 not active.
- DHCP fails: wrong pool, missing helper address, or excluded-address conflict.

A simple troubleshooting sequence:
1. Check physical link and interface state.
2. Check VLAN assignment on access ports.
3. Check trunk status and allowed VLANs.
4. Check gateway IP configuration.
5. Check routing table and return path.
6. Check ARP and MAC tables.
7. Test with `ping` and `traceroute`.

Example diagnostic commands:
```bash
show interfaces status
show interfaces trunk
show vlan brief
show ip interface brief
show ip route
show arp
ping 192.168.10.1
traceroute 192.168.20.10
```

> A successful ping to the local gateway proves only local Layer 3 reachability. It does not prove that routing, return paths, or ACLs are correct beyond that point.

**Verification checklist for a completed lab**

Use this final pass before you consider the lab done:
- All required devices are powered on and cabled correctly.
- All interfaces are up unless intentionally unused.
- VLANs exist and are named correctly.
- Access ports are in the correct VLANs.
- Trunks carry the required VLANs.
- Inter-VLAN routing works.
- Hosts have correct IP settings and gateways.
- Static routes or dynamic routing entries are present if required.
- DHCP works if configured.
- SSH access works for management.
- The configuration is saved.

Save the configuration:
```bash
copy running-config startup-config
```

Or:
```bash
write memory
```

**Exam-day habits**

In the exam, speed matters, but so does accuracy. Build the network in phases and verify after each phase rather than waiting until the end.

Best habits:
- Read the addressing table carefully before configuring.
- Configure and verify one device or one feature at a time.
- Use `show` commands before changing working settings.
- Keep a mental map of VLANs, subnets, and gateways.
- Save the configuration only after verification succeeds.

A strong capstone workflow is:
- Plan.
- Configure.
- Verify.
- Troubleshoot.
- Re-verify.
- Save.
