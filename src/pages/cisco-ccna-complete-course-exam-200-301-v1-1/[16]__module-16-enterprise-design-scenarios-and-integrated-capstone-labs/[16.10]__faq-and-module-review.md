## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Module review mindset**

This review ties together the most testable CCNA concepts: network access, IP connectivity, IP services, security fundamentals, automation, and basic device management. On the exam, you are rarely asked to recite a definition in isolation; you are more often asked to interpret output, choose the correct command, or identify the most likely cause of a failure.

A strong review strategy is to connect each topic to three things:
- **What it does**
- **How you verify it**
- **What breaks when it is wrong**

**Core device management and IOS access**

Cisco IOS and IOS XE devices are commonly managed through the console, auxiliary port, Telnet, or SSH. For exam purposes, you should know that SSH is preferred because it encrypts the session, while Telnet sends credentials and commands in clear text.

Typical access and management concepts:
- **Console access** is used for initial setup and recovery.
- **AUX access** is legacy and less common, but it can be used for out-of-band management.
- **SSH** requires a hostname, domain name, local user, RSA keys, and VTY line configuration.
- **Telnet** may still appear in labs, but it is insecure and should be disabled in production.

A common secure remote-access configuration looks like this:

```cisco
conf t
hostname R1
ip domain-name example.com
username admin privilege 15 secret Str0ngP@ss!
crypto key generate rsa modulus 2048
ip ssh version 2

line vty 0 4
login local
transport input ssh
exit
service password-encryption
banner motd ^CUnauthorized access prohibited.^C
end
write memory
```

Verification commands you should recognize:
```cisco
show running-config | section line vty
show ip ssh
show users
show ssh
show startup-config
```

Important management reminders:
- **`enable secret`** is preferred over `enable password`.
- **`service password-encryption`** only obscures some plaintext passwords; it is not strong security.
- **`copy running-config startup-config`** and `write memory` save the active configuration.
- **`reload`** without saving can discard changes.
- **`show version`** helps confirm uptime, IOS version, configuration register, and boot image.

**Basic device initialization and recovery**

When a device boots, it loads the IOS image, then the startup configuration from NVRAM if present. If no startup configuration exists, the device may enter the initial configuration dialog.

Common initialization tasks:
1. Set the hostname.
2. Configure passwords and local access.
3. Assign management IP addressing.
4. Save the configuration.
5. Verify reachability and remote access.

If you need to recover access, you may use ROMMON or password recovery procedures depending on platform and policy. In exam scenarios, the key idea is that startup configuration and IOS image are separate, and losing one does not necessarily mean losing the other.

**IPv4 addressing, subnetting, and routing table interpretation**

IPv4 subnetting remains a core CCNA skill because it affects addressing, VLAN design, route selection, and troubleshooting. You should be able to identify:
- Network address
- Broadcast address
- Usable host range
- Prefix length and subnet mask
- Number of hosts per subnet

The routing table is one of the most important troubleshooting tools. A route entry tells you:
- **Source code** of the route, such as connected, static, or dynamic
- **Destination prefix**
- **Next hop**
- **Outgoing interface**
- **Administrative distance** and metric, depending on route type

Example output pattern:
```cisco
show ip route
```

You should interpret common route codes:
- **C** = connected
- **L** = local
- **S** = static
- **O** = OSPF
- **D** = EIGRP
- **R** = RIP
- **S\*** = candidate default route

A default route is used when no more specific route matches. On routers, a static default route often looks like this:

```cisco
ip route 0.0.0.0 0.0.0.0 192.0.2.1
```

Or, on point-to-point links:
```cisco
ip route 0.0.0.0 0.0.0.0 GigabitEthernet0/0
```

When troubleshooting IPv4 connectivity, check in this order:
1. Interface status and IP addressing.
2. Local host default gateway.
3. Routing table on the router.
4. ACLs or NAT rules that may block traffic.
5. Layer 2 issues such as VLAN membership or trunking.

**IPv6 addressing and verification**

IPv6 introduces global unicast, link-local, multicast, and unique local addressing. For CCNA, the most important operational concept is that every IPv6-enabled interface typically has a link-local address, and many routing protocols use link-local addresses for neighbor relationships.

Key IPv6 address types:
- **Global unicast**: routable on the IPv6 internet or enterprise network.
- **Link-local**: valid only on the local link; starts with `fe80::/10`.
- **Multicast**: used for one-to-many delivery.
- **Unique local**: private addressing, similar in purpose to RFC 1918 IPv4 space.

Important operational details:
- Link-local addresses are not routed beyond the local segment.
- When using a link-local address as a next hop, you usually must specify the outgoing interface as well.
- IPv6 routing tables often show connected and local routes automatically after interface configuration.

Example interface configuration:
```cisco
conf t
interface GigabitEthernet0/0
 ipv6 address 2001:db8:10:1::1/64
 ipv6 address fe80::1 link-local
 no shutdown
end
```

Verification commands:
```cisco
show ipv6 interface GigabitEthernet0/0
show ipv6 route
show ipv6 neighbors
```

For reachability testing, use the IPv6-aware forms supported by the platform. On Cisco IOS/IOS XE, common patterns include:
```cisco
ping ipv6
traceroute ipv6
```

You can also test a specific IPv6 destination from the prompt or the IPv6 ping/traceroute workflow, depending on platform behavior:
```cisco
ping 2001:db8:10:1::2
traceroute 2001:db8:10:1::2
```

For link-local testing, include the interface zone when required:
```cisco
ping fe80::1%GigabitEthernet0/0
```

> Link-local next hops are interface-scoped. If you omit the outgoing interface context, the route or test may fail even when the address is correct.

**Switching, VLANs, and trunking**

VLANs segment a Layer 2 network into separate broadcast domains. They improve security, reduce broadcast scope, and support logical separation without requiring separate physical switches.

Core VLAN concepts:
- **Access port** carries traffic for one VLAN.
- **Trunk port** carries traffic for multiple VLANs using tagging.
- **Native VLAN** is the untagged VLAN on an 802.1Q trunk.
- **Allowed VLAN list** restricts which VLANs may traverse a trunk.

Example topology:
```text
PC1 --- SW1 access VLAN 10
PC2 --- SW1 access VLAN 20
SW1 trunk --- SW2 trunk
R1 subinterfaces --- SW1 trunk for inter-VLAN routing
```

Example switch configuration:
```cisco
conf t
vlan 10
 name USERS
vlan 20
 name VOICE
!
interface GigabitEthernet0/1
 switchport mode access
 switchport access vlan 10
!
interface GigabitEthernet0/2
 switchport mode access
 switchport access vlan 20
!
interface GigabitEthernet0/24
 switchport mode trunk
 switchport trunk allowed vlan 10,20
end
```

Verification commands:
```cisco
show vlan brief
show interfaces trunk
show interfaces switchport
show mac address-table
```

Common VLAN/trunk troubleshooting scenarios:
- A host cannot reach another host in the same VLAN because the access port is in the wrong VLAN.
- Inter-VLAN traffic fails because the trunk is down or VLANs are not allowed on the trunk.
- Native VLAN mismatch causes warnings and can lead to unexpected traffic handling.
- A VLAN exists on one switch but not on another, so traffic stops at the trunk boundary.

**Inter-VLAN routing**

Inter-VLAN routing allows devices in different VLANs to communicate through a Layer 3 device. The two common CCNA methods are:
- **Router-on-a-stick**
- **Layer 3 switch SVIs** (Switched Virtual Interfaces)

Router-on-a-stick example:
```cisco
conf t
interface GigabitEthernet0/0
 no shutdown
!
interface GigabitEthernet0/0.10
 encapsulation dot1Q 10
 ip address 192.168.10.1 255.255.255.0
!
interface GigabitEthernet0/0.20
 encapsulation dot1Q 20
 ip address 192.168.20.1 255.255.255.0
end
```

Layer 3 switch example:
```cisco
conf t
ip routing
interface vlan 10
 ip address 192.168.10.1 255.255.255.0
 no shutdown
interface vlan 20
 ip address 192.168.20.1 255.255.255.0
 no shutdown
end
```

Troubleshooting checklist:
1. Confirm the VLAN exists on the switch.
2. Confirm the host access port is in the correct VLAN.
3. Confirm the trunk carries the VLAN.
4. Confirm the router subinterface or SVI has the correct IP address.
5. Confirm the host default gateway points to the correct Layer 3 interface.

**Spanning Tree Protocol and loop prevention**

Spanning Tree Protocol (STP) prevents Layer 2 loops by blocking redundant paths. Without STP, a loop can cause broadcast storms, MAC address flapping, and unstable forwarding.

Important STP ideas:
- **Root bridge** is the reference switch for the topology.
- **Root port** is the best path toward the root on a non-root switch.
- **Designated port** forwards traffic for a segment.
- **Blocking** prevents loops while preserving redundancy.

Common exam concepts:
- Lower bridge ID wins root election.
- Bridge ID includes priority and MAC address.
- Port cost influences path selection.
- PortFast is used on edge ports connected to end devices, not switches.

Useful commands:
```cisco
show spanning-tree
show spanning-tree vlan 10
show spanning-tree root
```

Practical guidance:
- Use PortFast on access ports to speed up host connectivity.
- Use BPDU Guard on edge ports to protect against accidental switch connections.
- Do not enable PortFast on trunk links between switches.

**EtherChannel and link aggregation**

EtherChannel combines multiple physical links into one logical link for increased bandwidth and redundancy. Cisco commonly uses LACP (Link Aggregation Control Protocol) or PAgP (Port Aggregation Protocol), though LACP is the more universal standard.

Benefits:
- Higher aggregate throughput
- Redundancy if one member link fails
- STP treats the bundle as one logical link

Example LACP configuration:
```cisco
conf t
interface range GigabitEthernet0/1 - 2
 switchport mode trunk
 channel-group 1 mode active
!
interface Port-channel1
 switchport mode trunk
end
```

Verification:
```cisco
show etherchannel summary
show interfaces port-channel 1
show lacp neighbor
```

Common failure causes:
- Mismatched speed/duplex
- Different trunk settings on member links
- Inconsistent allowed VLAN lists
- One side using access mode and the other using trunk mode

**Wireless fundamentals**

Wireless LANs use radio frequency channels, access points, and client association to provide network access. For CCNA, focus on the practical differences between bands, channels, and common deployment issues.

Key wireless concepts:
- **2.4 GHz** has longer range and better wall penetration, but more interference and fewer non-overlapping channels.
- **5 GHz** offers more channels and typically less interference, but shorter range.
- **Channel selection** matters because overlapping channels increase contention and reduce performance.
- **SSID** identifies the wireless network name.
- **WPA2/WPA3** provide stronger security than open or legacy WEP networks.

Practical wireless considerations:
- Use 2.4 GHz for legacy compatibility or longer coverage needs.
- Use 5 GHz for higher throughput and denser deployments.
- Avoid channel overlap on 2.4 GHz by using non-overlapping channels where possible.
- Place access points to reduce dead zones and co-channel interference.

Common troubleshooting symptoms:
- Clients can see the SSID but cannot connect: check authentication, password, and security mode.
- Clients connect but have poor throughput: check signal strength, interference, channel overlap, and band selection.
- Clients roam poorly: check AP placement, transmit power, and overlapping coverage.

**Routing protocols and path selection**

Routing protocols exchange routes dynamically so routers can adapt to topology changes. CCNA commonly emphasizes OSPF, but you should also recognize RIP and EIGRP at a high level.

OSPF basics:
- **Link-state** protocol
- Uses areas, with area 0 as the backbone
- Forms neighbor adjacencies before exchanging routes
- Uses cost as its metric

Useful OSPF commands:
```cisco
show ip ospf neighbor
show ip route ospf
show ip protocols
```

Example OSPF configuration:
```cisco
conf t
router ospf 1
 router-id 1.1.1.1
 network 192.168.10.0 0.0.0.255 area 0
 network 192.168.20.0 0.0.0.255 area 0
end
```

Common OSPF issues:
- Mismatched area IDs prevent adjacency.
- Mismatched hello/dead timers prevent adjacency.
- Passive interfaces stop neighbor formation on that interface.
- Duplicate router IDs can cause instability.

RIP and EIGRP are often tested more for recognition than deep configuration:
- **RIP** uses hop count and has a small maximum hop limit.
- **EIGRP** is Cisco-originated and uses a composite metric.

When interpreting routing behavior, remember:
- The router chooses the route with the longest prefix match first.
- If multiple routes match the same prefix, administrative distance helps choose the best source.
- If the source is the same, the metric decides the best path.

**ACLs and traffic filtering**

Access Control Lists (ACLs) filter traffic based on criteria such as source IP, destination IP, protocol, and ports. They are used for security, traffic control, and policy enforcement.

ACL types:
- **Standard ACL** filters mainly by source IP.
- **Extended ACL** filters by source, destination, protocol, and ports.

Example standard ACL:
```cisco
access-list 10 permit 192.168.10.0 0.0.0.255
access-list 10 deny any
```

Example extended ACL:
```cisco
ip access-list extended WEB-FILTER
 deny tcp 192.168.10.0 0.0.0.255 any eq 80
 permit ip any any
```

Apply an ACL to an interface:
```cisco
interface GigabitEthernet0/0
 ip access-group WEB-FILTER in
```

ACL troubleshooting points:
- ACLs are processed top-down, first match wins.
- An implicit `deny any` exists at the end.
- Direction matters: inbound and outbound are not interchangeable.
- Placement matters: standard ACLs are usually placed close to the destination, extended ACLs close to the source.

**NAT and PAT**

Network Address Translation (NAT) translates private addresses to public addresses. PAT (Port Address Translation), also called NAT overload, allows many inside hosts to share one public IP using different source ports.

Why NAT matters:
- Conserves public IPv4 addresses
- Hides internal addressing from external networks
- Enables internet access for private networks

Example PAT configuration:
```cisco
conf t
interface GigabitEthernet0/0
 ip nat inside
!
interface GigabitEthernet0/1
 ip nat outside
!
access-list 1 permit 192.168.10.0 0.0.0.255
ip nat inside source list 1 interface GigabitEthernet0/1 overload
end
```

Verification:
```cisco
show ip nat translations
show ip nat statistics
show access-lists
```

Common NAT issues:
- Inside and outside interfaces are reversed.
- The ACL does not match the inside subnet.
- The default route to the outside network is missing.
- Return traffic is blocked by an ACL or routing problem.

**IP services: DHCP, DNS, NTP, and SNMP**

These services support network operation and are frequently referenced in troubleshooting scenarios.

DHCP (Dynamic Host Configuration Protocol):
- Automatically assigns IP address, mask, gateway, and DNS settings.
- Uses DORA: Discover, Offer, Request, Acknowledge.

Example DHCP pool:
```cisco
conf t
ip dhcp excluded-address 192.168.10.1 192.168.10.20
ip dhcp pool USERS
 network 192.168.10.0 255.255.255.0
 default-router 192.168.10.1
 dns-server 8.8.8.8
end
```

DNS (Domain Name System):
- Resolves names to IP addresses.
- If DNS fails, users may report that websites do not load by name but do load by IP.

NTP (Network Time Protocol):
- Synchronizes device clocks.
- Accurate time is important for logs, certificates, and troubleshooting.

SNMP (Simple Network Management Protocol):
- Used by monitoring systems to collect device data.
- Community strings in SNMPv2c are sensitive and should be protected.

**Security fundamentals**

CCNA security topics focus on basic device hardening and access control rather than advanced threat hunting.

Key practices:
- Use strong passwords and `enable secret`.
- Prefer SSH over Telnet.
- Restrict VTY access where possible.
- Use banners to warn unauthorized users.
- Disable unused ports or place them in an unused VLAN.
- Use Port Security on access ports when appropriate.

Example port security:
```cisco
conf t
interface GigabitEthernet0/3
 switchport mode access
 switchport access vlan 10
 switchport port-security
 switchport port-security maximum 2
 switchport port-security mac-address sticky
 switchport port-security violation restrict
end
```

Verification:
```cisco
show port-security interface GigabitEthernet0/3
show port-security address
show running-config interface GigabitEthernet0/3
```

Security troubleshooting:
- If a port shuts down unexpectedly, check for port-security violations.
- If remote access fails, confirm VTY transport settings and authentication.
- If users can reach services they should not, check ACL placement and direction.

**Automation, programmability, and APIs**

Automation reduces repetitive configuration and improves consistency. At CCNA level, you should recognize the main terms and understand what they are used for.

Core concepts:
- **API** (Application Programming Interface) allows software to communicate with software.
- **REST** (Representational State Transfer) is a common web-based API style.
- **RESTCONF** is a REST-like protocol for network configuration and monitoring.
- **NETCONF** is a protocol for network device configuration using structured data.
- **YANG** models define the data structure used by NETCONF and RESTCONF.

What to look for in scenarios:
- JSON or XML payloads often indicate API-based automation.
- A device exposing `restconf` or `netconf` services suggests programmable management.
- YANG models describe what data can be read or configured.

Example REST-style request pattern:
```bash
curl -k -u admin:Str0ngP@ss! \
  -H "Accept: application/yang-data+json" \
  https://192.0.2.10/restconf/data/ietf-interfaces:interfaces
```

Example of a JSON response shape:
```json
{
  "ietf-interfaces:interfaces": {
    "interface": [
      {
        "name": "GigabitEthernet1",
        "enabled": true
      }
    ]
  }
}
```

Practical exam guidance:
- If you see structured data and HTTP methods like `GET`, `POST`, `PUT`, or `DELETE`, think REST API.
- If you see `netconf` or XML-based configuration exchange, think NETCONF.
- If you see `yang` in a model name or path, think data modeling.

**Troubleshooting methodology**

A consistent troubleshooting process helps you avoid guessing. Start with the simplest likely cause and move outward from the endpoint.

A practical workflow:
1. Identify the symptom and scope.
2. Check physical and link status.
3. Verify addressing and default gateway.
4. Verify VLANs, trunks, and routing.
5. Check ACLs, NAT, and services.
6. Confirm the issue with a targeted test.

Useful commands by layer:
```cisco
show ip interface brief
show interfaces status
show vlan brief
show interfaces trunk
show mac address-table
show ip route
show ipv6 route
show access-lists
show ip nat translations
show cdp neighbors
show lldp neighbors
```

Scenario examples:
- **VLAN mismatch**: A PC in VLAN 10 cannot reach its gateway. Check the access port VLAN and the SVI or router subinterface.
- **Trunk issue**: Two switches can ping locally but hosts in a remote VLAN cannot communicate. Check trunk mode, allowed VLANs, and native VLAN consistency.
- **OSPF adjacency failure**: Routes are missing after a topology change. Check neighbor state, area ID, timers, and interface network type.
- **NAT failure**: Inside hosts can ping the router but not the internet. Check inside/outside markings, ACL match, and default route.
- **IPv6 link-local issue**: A route uses a link-local next hop but traffic fails. Confirm the outgoing interface is specified and the neighbor is reachable on that link.

**High-value exam reminders**

- **Longest prefix match** always wins before administrative distance.
- **Administrative distance** decides between routes from different sources.
- **STP** prevents loops, but it can also block a redundant path you expected to use.
- **Trunks** carry multiple VLANs; access ports carry one VLAN.
- **SSH** is the secure remote-access standard; Telnet is not.
- **IPv6 link-local** addresses are local to the interface and often need interface context.
- **ACL order** matters because the first match is applied.
- **NAT** requires correct inside/outside roles and routing.
- **Automation** is about structured data and APIs, not just scripts.

**Quick self-check questions**

- Can you explain why a host with a correct IP address still cannot reach another subnet?
- Can you identify whether a failure is Layer 2, Layer 3, or policy-related from command output?
- Can you distinguish between a trunk problem and an inter-VLAN routing problem?
- Can you read a routing table and determine the next hop and exit interface?
- Can you tell whether a device is using SSH, Telnet, or console access?
- Can you recognize when a link-local IPv6 address needs an interface scope?

**Final review approach**

Use the exam lens: every feature should be tied to verification and troubleshooting. If you can configure a feature but cannot validate it with `show` commands or explain its failure modes, you are not fully prepared. Focus on the relationships between switching, routing, security, services, and automation, because the exam often combines them in one scenario.
