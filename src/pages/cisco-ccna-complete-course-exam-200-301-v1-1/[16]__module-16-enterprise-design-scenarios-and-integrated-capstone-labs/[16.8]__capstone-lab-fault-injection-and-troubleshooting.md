## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Capstone troubleshooting mindset**

A good CCNA troubleshooting workflow is not random command checking. You start with the symptom, narrow the fault domain, verify the expected baseline, and then test one layer at a time. In labs and on the exam, this approach saves time and prevents you from “fixing” the wrong thing.

A practical method is to isolate the problem in this order:

1. **Layer 1**: power, cabling, speed/duplex, link state.
2. **Layer 2**: VLANs, trunking, EtherChannel, MAC learning, STP.
3. **Layer 3**: IP addressing, routing, default gateway, IPv6 neighbor discovery.
4. **Services**: DHCP, NAT/PAT, ACLs, DNS, wireless authentication.

Use `show` commands first, then `debug` only when needed, and always turn debugging off after you collect the evidence.

**Troubleshooting workflow**

A repeatable workflow helps you avoid guesswork. Start by defining the failure boundary: is the issue local to one host, one VLAN, one subnet, one site, or one service? Then compare the broken device against a known-good device or the documented baseline.

A simple decision flow looks like this:

```text
Symptom
  -> Is the link up?
     -> No: check Layer 1, speed/duplex, cabling, optics, PoE
     -> Yes: check VLAN/trunk/EtherChannel/STP
        -> L2 good: check IP, gateway, routing, ACLs, NAT
           -> L3 good: check services, DNS, DHCP, wireless auth
```

Useful habits:

- Verify the **baseline** before changing anything.
- Change one variable at a time.
- Re-test after every fix.
- Record the exact command output before and after the change.
- Prefer the simplest explanation that matches the evidence.

**Baseline topology for the capstone lab**

A realistic lab should include multiple fault domains so you can practice isolating issues instead of memorizing one command. A small but useful topology is:

```text
                Internet / ISP
                     |
                 [R1 Edge]
              Gi0/0 |  | Gi0/1
                    NAT  |
                     |   |
                203.0.113.2/30
                     |
                 [SW1 Distribution]
               ____/   |    \____
              /        trunk      \
         [SW2 Access]          [AP1 Wireless]
            |   |                   |
         PC1  PC2                WLAN clients

VLAN 10 Users      10.10.10.0/24
VLAN 20 Voice      10.10.20.0/24
VLAN 30 Servers    10.10.30.0/24
Mgmt VLAN 99       10.10.99.0/24
IPv6 prefix        2001:db8:10::/64 per VLAN
```

Before injecting faults, confirm the baseline:

- Interfaces are up/up where expected.
- VLANs exist and access ports are assigned correctly.
- Trunks carry the required VLANs.
- Inter-VLAN routing works on the L3 device or router-on-a-stick.
- DHCP pools match the client VLANs.
- NAT/PAT translates inside traffic to the outside interface.
- ACLs permit intended traffic and deny only what is required.
- Wireless clients authenticate and map to the correct VLAN.

**Core verification commands**

Use `show` commands to identify where the failure lives. The exact command set depends on the platform, but these are common CCNA-level checks:

```bash
show ip interface brief
show interfaces status
show interfaces trunk
show vlan brief
show mac address-table
show spanning-tree
show etherchannel summary
show ip route
show ipv6 route
show arp
show ipv6 neighbors
show running-config
show logging
```

For interface-specific checks, use commands that are valid on the platform you are working on. For example:

```bash
show interfaces g0/1
show interfaces vlan 20
show running-config interface g0/1
show running-config interface vlan 20
```

For IPv4 and IPv6 reachability tests, use the common privileged EXEC forms:

```bash
ping 10.10.30.10
traceroute 10.10.30.10
ping 2001:db8:10:20::10
traceroute 2001:db8:10:20::10
```

If a platform supports alternate syntax, use the form that matches that device, but do not assume `ipv6` is required in the command.

**Layer 1 troubleshooting**

Layer 1 faults usually appear as no link, intermittent link, or poor performance. In a lab, they often come from bad cabling, wrong transceiver type, disabled ports, or speed/duplex mismatch.

Check for:

- Interface administratively down.
- Physical link down.
- Incorrect speed or duplex settings.
- Bad cable, bad patch panel, or wrong optic.
- Power issues on access points, phones, or small switches.

Useful commands:

```bash
show interfaces status
show interfaces g0/1
show controllers ethernet-controller g0/1
show logging
```

What to look for:

- `administratively down` means the interface is shut.
- `notconnect` or `down/down` suggests a physical issue.
- High input errors, CRC errors, or late collisions can indicate cabling or duplex problems.
- On modern Ethernet, keep both ends consistent and preferably use auto-negotiation on both sides unless a design requires otherwise.

> Do not force one side to a fixed speed/duplex and leave the other side on auto. That can still create a mismatch and unstable behavior.

**Layer 2 troubleshooting**

Layer 2 issues are common in CCNA labs because VLANs, trunks, and EtherChannel depend on multiple settings matching exactly.

Check for:

- Wrong access VLAN on an edge port.
- Missing VLAN in the VLAN database.
- Trunk not allowing the required VLAN.
- Native VLAN mismatch.
- STP blocking a path unexpectedly.
- MAC address not learned on the expected port.
- EtherChannel members not bundling.

Useful commands:

```bash
show vlan brief
show interfaces trunk
show spanning-tree vlan 10
show mac address-table dynamic
show interfaces switchport
show etherchannel summary
show lacp neighbor
```

Practical checks:

- If a host cannot reach another host in the same VLAN, verify the access port VLAN assignment and MAC learning.
- If a host can reach local devices but not another switch, verify trunking and allowed VLANs.
- If traffic works on one path but not another, check STP state and EtherChannel consistency.

**EtherChannel and LACP troubleshooting**

EtherChannel problems often show up as links that stay up individually but do not forward correctly as a bundle. Common causes are mismatched channel mode, speed/duplex, trunk settings, allowed VLANs, or hashing expectations.

Typical issues:

- One side uses `mode active` and the other uses `mode on`.
- Member interfaces have different access/trunk settings.
- One member is in the wrong VLAN or has a different native VLAN.
- Physical interfaces are not identical in speed/duplex.
- The port-channel interface is configured differently from its members.

Verification commands:

```bash
show etherchannel summary
show interfaces port-channel 1
show run interface g0/1
show run interface g0/2
show lacp neighbor
```

Example of a consistent LACP configuration:

```cisco
interface range g0/1 - 2
 switchport
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30,99
 channel-group 1 mode active

interface port-channel 1
 switchport
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30,99
```

If the bundle does not form, compare both ends line by line. The most common fix is to make the member interfaces and the port-channel interface match on both devices.

**Layer 3 IPv4 troubleshooting**

Layer 3 faults usually involve addressing, masks, gateways, routing, or ACLs. A host may have link and VLAN connectivity but still fail to reach remote networks.

Check for:

- Wrong IP address or subnet mask.
- Missing or wrong default gateway.
- Missing route on the router or L3 switch.
- Incorrect next hop.
- Overlapping subnets.
- ACL blocking traffic.

Useful commands:

```bash
show ip interface brief
show ip route
show arp
show running-config | section interface
show running-config | include ip route
ping 10.10.30.1
traceroute 10.10.30.10
```

A common exam pattern is:

- Host can ping its gateway.
- Host cannot ping remote subnet.
- Router has no route back, or ACL blocks return traffic.

When troubleshooting routing, confirm both directions:

- The source device has a route to the destination.
- The destination device has a route back to the source.
- The default gateway on the host is correct.

**NAT and PAT troubleshooting**

Network Address Translation (NAT) and Port Address Translation (PAT) are frequent sources of “internet down” symptoms in labs. The inside network may route correctly, but traffic fails when it reaches the edge router.

Common faults:

- Inside and outside interfaces reversed.
- Missing `ip nat inside` or `ip nat outside`.
- ACL used for NAT does not match the inside subnet.
- Overly narrow ACL excludes some hosts.
- No default route toward the ISP.
- PAT overload not configured on the outside interface.
- Translation table empty because traffic never matches.

Example configuration:

```cisco
access-list 1 permit 10.10.10.0 0.0.0.255
access-list 1 permit 10.10.20.0 0.0.0.255
access-list 1 permit 10.10.30.0 0.0.0.255

interface g0/0
 ip address 10.10.10.1 255.255.255.0
 ip nat inside

interface g0/1
 ip address 203.0.113.2 255.255.255.252
 ip nat outside

ip nat inside source list 1 interface g0/1 overload
ip route 0.0.0.0 0.0.0.0 203.0.113.1
```

Verification commands:

```bash
show ip nat translations
show ip nat statistics
show access-lists
show ip route
```

Troubleshooting approach:

1. Confirm the inside host can reach the edge router.
2. Confirm the edge router has a default route.
3. Confirm the NAT ACL matches the inside source addresses.
4. Generate traffic, then check the translation table.
5. Verify the outside interface is marked correctly.

If translations do not appear, the problem is usually classification, interface role, or routing rather than NAT itself.

**ACL troubleshooting**

Access control lists (ACLs) often fail because of placement, direction, or matching logic. In CCNA labs, you should be able to explain why a packet is permitted or denied before you change the ACL.

Key rules:

- Standard ACLs match only the source IP address.
- Extended ACLs can match source, destination, protocol, and ports.
- Place extended ACLs close to the source when possible.
- Place standard ACLs close to the destination when possible.
- Apply the ACL in the correct direction on the correct interface.

Example extended ACL:

```cisco
ip access-list extended WEB-FILTER
 permit tcp 10.10.10.0 0.0.0.255 host 10.10.30.10 eq 80
 permit tcp 10.10.10.0 0.0.0.255 host 10.10.30.10 eq 443
 deny ip any any log

interface g0/1
 ip access-group WEB-FILTER in
```

The direction must match where the traffic enters the device interface you are filtering. If the traffic enters on a different interface, the ACL must be applied there instead.

Standard ACL example:

```cisco
ip access-list standard MGMT-ONLY
 permit 10.10.99.0 0.0.0.255
 deny any

line vty 0 4
 access-class MGMT-ONLY in
```

Verification commands:

```bash
show access-lists
show ip access-lists
show running-config | section access-list
show ip interface g0/1
```

What to check:

- Sequence numbers and rule order.
- Implicit deny at the end.
- Hit counts increasing on the expected entries.
- ACL applied to the intended interface and direction.
- Wildcard masks match the intended subnet.

If traffic is denied unexpectedly, compare the packet flow to the ACL line by line. Remember that ACLs are processed top-down, first match wins.

**IPv6 troubleshooting**

IPv6 problems often involve addressing, prefix length, default route, neighbor discovery, or router advertisements. A host may have a global address but still fail to reach remote networks if the default route or next hop is wrong.

Common failure cases:

- Wrong prefix length.
- Missing link-local or global address on the router interface.
- No IPv6 routing enabled on the router.
- Incorrect default gateway on the host.
- Router Advertisement (RA) suppression when SLAAC is expected.
- DHCPv6 used when the client expects SLAAC, or vice versa.
- Link-local next hop not used correctly in static routes.

Useful commands:

```bash
show ipv6 interface brief
show ipv6 route
show ipv6 neighbors
show running-config | section ipv6
ping 2001:db8:10:30::10
traceroute 2001:db8:10:30::10
```

Example static IPv6 routing with a link-local next hop:

```cisco
ipv6 unicast-routing

interface g0/0
 ipv6 address 2001:db8:10:1::1/64

interface g0/1
 ipv6 address 2001:db8:10:2::1/64

ipv6 route 2001:db8:10:30::/64 g0/1 fe80::2
```

Troubleshooting notes:

- If you use a link-local next hop, you usually must specify the exit interface as well.
- If hosts are using SLAAC, verify that router advertisements are being sent.
- If hosts are using DHCPv6, verify the DHCPv6 server or relay configuration and the M/O flags as designed.
- If neighbor discovery fails, check that the link-local addresses and multicast reachability are intact.

**DHCP troubleshooting**

Dynamic Host Configuration Protocol (DHCP) issues often look like “the client has no IP address” or “the client got the wrong network settings.” The problem may be on the server, relay, or client side.

Common server-side issues:

- Pool exhausted.
- Wrong network statement.
- Missing or wrong default router.
- Wrong DNS server.
- Excluded addresses overlap the pool.
- Interface or VLAN mismatch on the server.
- DHCP relay missing on the gateway interface.

Example DHCP server configuration:

```cisco
ip dhcp excluded-address 10.10.10.1 10.10.10.20

ip dhcp pool USERS
 network 10.10.10.0 255.255.255.0
 default-router 10.10.10.1
 dns-server 8.8.8.8
 domain-name example.local
```

Verification commands:

```bash
show ip dhcp binding
show ip dhcp pool
show ip dhcp conflict
show running-config | section dhcp
```

Troubleshooting approach:

1. Confirm the client is in the correct VLAN.
2. Confirm the gateway interface or SVI is up.
3. Confirm the DHCP pool matches the client subnet.
4. Check for exhausted leases or excluded-address overlap.
5. Verify relay configuration if the server is on another subnet.

If the client gets an address but cannot reach the network, inspect the default gateway and DNS settings in the lease.

**Wireless troubleshooting**

Wireless faults often appear as association failures, authentication failures, or clients joining the wrong VLAN. In CCNA labs, focus on the relationship between the AP, controller, SSID, security settings, and VLAN mapping.

Common issues:

- AP not powered or not joined to the controller.
- Wrong AP mode.
- SSID not broadcast or not mapped to the correct WLAN.
- Security mismatch between client and WLAN.
- RADIUS or PSK mismatch.
- VLAN mapping mismatch.
- RF interference or weak signal.
- Client association succeeds but DHCP fails.

Useful checks:

```bash
show wireless summary
show ap summary
show wlan summary
show client summary
show logging
```

On controller-based systems, verify:

- The AP has joined the controller.
- The WLAN is enabled.
- The WLAN maps to the correct VLAN or interface.
- The security policy matches the client.
- The client receives an IP address from the expected subnet.

Practical troubleshooting sequence:

1. Confirm the AP is online and joined.
2. Confirm the SSID is visible.
3. Confirm the client can authenticate.
4. Confirm the client is assigned to the correct VLAN.
5. Confirm DHCP and routing work for that VLAN.

If a client associates but cannot pass traffic, the issue is often VLAN mapping, DHCP, or ACL policy rather than RF.

**Logs and debug output**

Logs help you confirm what the device thinks is happening. Use them to correlate events with your test actions. Debug output is powerful but can be noisy, so use it carefully.

Useful commands:

```bash
show logging
terminal monitor
debug ip icmp
debug ip packet
debug dhcp detail
undebug all
```

Guidelines:

- Use `show logging` first to look for interface flaps, STP changes, DHCP failures, or authentication errors.
- Use `debug` only when you need live packet or protocol behavior.
- Stop debugging immediately after you capture the evidence.
- Avoid enabling broad debugs on busy devices unless the lab specifically requires it.

**Common fault patterns and what they usually mean**

| Symptom | Likely fault domain | First checks |
|---|---|---|
| Link down | Layer 1 | Cable, port state, speed/duplex, optics |
| Same-VLAN hosts cannot talk | Layer 2 | VLAN assignment, trunking, MAC table |
| Remote subnet unreachable | Layer 3 | IP, mask, gateway, routing |
| Internet unreachable | Services | NAT/PAT, default route, ACLs |
| DHCP failure | Services | Pool, relay, exclusions, VLAN |
| Wireless client joins but no traffic | L2/Services | VLAN mapping, DHCP, ACL, security |
| EtherChannel not bundling | Layer 2 | LACP mode, trunk consistency, member settings |

**Fault injection ideas for lab practice**

Use controlled faults to build pattern recognition. Change one item at a time and document the symptom.

Examples:

- Shut down an access port with `shutdown`.
- Assign the wrong VLAN to a user port.
- Remove a VLAN from a trunk allowed list.
- Break the EtherChannel mode on one side.
- Change the default gateway on a host.
- Remove the NAT overload statement.
- Narrow a DHCP pool too much.
- Apply an ACL in the wrong direction.
- Change the wireless security key.
- Suppress router advertisements when SLAAC clients are expected.

After each fault, ask yourself:

- What changed?
- Which layer failed first?
- Which command proves the fault?
- What is the smallest safe fix?

**End-to-end troubleshooting procedure**

Use this ordered process when you face an unknown issue in a lab or exam scenario:

1. Identify the symptom and the affected scope.
2. Check physical and interface status.
3. Verify VLANs, trunks, and EtherChannel if switching is involved.
4. Verify IP addressing, gateways, and routes.
5. Check ACLs, NAT/PAT, DHCP, and wireless policy as needed.
6. Test with `ping` and `traceroute`.
7. Confirm the fix with `show` commands and a repeat test.
8. Save the configuration if the change is intended.

**Capstone lab checklist**

Before you finish the lab, confirm that each service works from the user edge:

- PC in VLAN 10 gets DHCP and reaches the default gateway.
- PC in VLAN 10 reaches a server in VLAN 30.
- NAT/PAT translates inside traffic to the outside network.
- ACL permits only the intended web traffic.
- IPv6 hosts can reach the IPv6 server and the router advertises the correct prefix.
- Wireless clients authenticate and land in the correct VLAN.
- EtherChannel forwards traffic and all member links are bundled.

A strong CCNA troubleshooting answer is not just the fix. It is the reasoning that proves why the fault existed and why the chosen command or configuration resolves it.
