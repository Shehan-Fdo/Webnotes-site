## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Lab purpose and mindset**

This lab compares two troubleshooting styles:

- **Manual troubleshooting**: you inspect the network yourself using a structured method.
- **AI-assisted troubleshooting**: you use an assistant to suggest likely causes, commands, and next checks.

The goal is not to replace one with the other. In practice, you use AI to accelerate hypothesis generation, but you still verify with device output, packet flow, and configuration review. For CCNA-level work, the most important skill is disciplined validation.

A good troubleshooting workflow answers three questions:

- What is failing?
- Where does it fail?
- Why does the failure happen?

**Core troubleshooting method**

Use a repeatable process instead of guessing. A common approach is:

1. Identify the symptom.
2. Define the expected behavior.
3. Compare the actual behavior to the expected behavior.
4. Narrow the scope by testing one layer or segment at a time.
5. Confirm the root cause with evidence.
6. Apply the fix.
7. Re-test and document the result.

This method works for Layer 1 through Layer 3 issues, including VLAN, trunking, inter-VLAN routing, ACLs, default gateways, and IP addressing mistakes.

**Manual troubleshooting workflow**

Manual troubleshooting is strongest when you need certainty. You inspect the topology, verify interface state, and test each hop.

Typical checks include:

- **Physical layer**
  - Confirm cables, link lights, and interface status.
  - Check for `up/up`, `down/down`, or `administratively down/down`.
- **Data link layer**
  - Verify VLAN membership, trunk status, and allowed VLANs.
  - Confirm native VLAN consistency where relevant.
- **Network layer**
  - Check IP addressing, subnet masks, default gateways, and routing.
  - Verify inter-VLAN routing on router subinterfaces or Layer 3 interfaces.
- **Policy layer**
  - Review ACLs, NAT, and security features that may filter traffic.

Useful commands on Cisco IOS include:

```bash
show ip interface brief
show interfaces status
show interfaces trunk
show vlan brief
show running-config
show ip route
show access-lists
show cdp neighbors
ping
traceroute
```

When you troubleshoot manually, you should always interpret the output in context. For example:

- `show ip interface brief` tells you whether an interface is operational, but not whether the VLAN path is correct.
- `show interfaces trunk` confirms trunking and allowed VLANs, but not whether hosts have valid IP settings.
- `show access-lists` shows hit counts, but hit counts alone do not prove the ACL is the root cause.

**AI-assisted troubleshooting workflow**

AI-assisted troubleshooting is useful when you want faster pattern recognition or a second opinion. You can provide:

- Topology description
- Relevant configs
- Command output
- Observed symptoms
- What you already tested

A strong prompt is specific. For example:

```text
I have two VLANs on a router-on-a-stick topology.
PC-A in VLAN 10 can ping its gateway but not PC-B in VLAN 20.
Here is the output of show interfaces trunk, show ip interface brief, and the router subinterface config.
Help me identify the most likely cause and the next verification step.
```

AI can help you:

- Suggest likely misconfigurations
- Prioritize commands to run next
- Explain unfamiliar output
- Compare a working and broken configuration
- Summarize findings into a troubleshooting report

AI is less reliable when:

- The prompt is vague
- The topology is incomplete
- The output is partial or outdated
- You ask it to infer facts not present in the evidence

> Do not treat an AI suggestion as proof. Validate every claim against device output or packet behavior.

**Common lab scenarios**

**1. VLAN mismatch on an access port**

A host may fail to reach its gateway if the switchport is assigned to the wrong VLAN.

Symptoms:

- Host can link up, but cannot ping the default gateway.
- `show vlan brief` shows the port in the wrong VLAN.
- The switchport may be configured as access mode, but in the wrong access VLAN.

Example verification:

```bash
show interfaces fa0/1 switchport
show vlan brief
```

Example fix:

```bash
conf t
interface fa0/1
 switchport mode access
 switchport access vlan 10
end
```

**2. Trunk missing an allowed VLAN**

In a router-on-a-stick design, a host may reach its local gateway but fail to reach another VLAN if the trunk does not carry the needed VLAN.

Symptoms:

- PC-A in VLAN 10 can ping the router subinterface for VLAN 10.
- PC-A cannot reach a host in VLAN 20.
- `show interfaces trunk` shows VLAN 20 missing from the allowed list.

Example verification:

```bash
show interfaces trunk
show running-config interface g0/1
```

Example fix:

```bash
conf t
interface g0/1
 switchport mode trunk
 switchport trunk allowed vlan 10,20
end
```

**3. Router subinterface or encapsulation issue**

If the router-on-a-stick subinterface is misconfigured, inter-VLAN routing fails.

Symptoms:

- The physical router interface is up.
- Subinterfaces may be present, but traffic does not route correctly.
- The VLAN tag or IP address may be wrong.

Example configuration:

```bash
conf t
interface g0/0
 no shutdown
!
interface g0/0.10
 encapsulation dot1Q 10
 ip address 192.168.10.1 255.255.255.0
!
interface g0/0.20
 encapsulation dot1Q 20
 ip address 192.168.20.1 255.255.255.0
end
```

Important behavior:

- A configured subinterface is normally `up/up` when the parent interface is up and the VLAN tagging matches.
- If the parent interface is down, the subinterface is down.
- If the encapsulation or VLAN is wrong, the subinterface may appear operational but traffic still fails.

**4. Default gateway or IP addressing error**

A host may be correctly cabled and switched, but still fail because of an IP configuration mistake.

Symptoms:

- Host cannot reach remote networks.
- Host may not even reach the local gateway.
- `ipconfig` or `ifconfig` shows an incorrect subnet mask or gateway.

Example host checks:

```bash
ipconfig /all
```

Look for:

- Correct IP address
- Correct subnet mask
- Correct default gateway
- No duplicate address conflict

**5. ACL filtering traffic**

An access control list (ACL) can block traffic even when routing is correct.

Symptoms:

- Ping works to the gateway but fails to a remote host or service.
- `show access-lists` shows matches on a deny entry.
- The ACL direction matters: inbound and outbound placement affect which traffic is filtered.

Example ACL and interface application:

```bash
conf t
ip access-list extended BLOCK_ICMP
 deny icmp 192.168.10.0 0.0.0.255 192.168.20.0 0.0.0.255
 permit ip any any
!
interface g0/0.10
 ip access-group BLOCK_ICMP in
end
```

This blocks ICMP entering from VLAN 10 toward the router and onward to VLAN 20, but it does not automatically block all traffic in the reverse direction unless you also filter that path. Always check direction and scope.

**How to compare manual and AI-assisted results**

Use a side-by-side approach:

| Aspect | Manual method | AI-assisted method |
|---|---|---|
| Speed | Slower at first | Faster for initial ideas |
| Accuracy | High when evidence is complete | Depends on prompt quality |
| Best use | Confirming root cause | Generating hypotheses |
| Risk | Human oversight | Hallucinated or overconfident answers |
| Output | Direct device-based proof | Suggested next steps |

A practical workflow is:

1. Gather the evidence manually.
2. Ask AI to interpret the evidence.
3. Verify the AI’s conclusion with one more command or test.
4. Apply the fix only after confirmation.

**Practical lab example**

Scenario:

- PC-A in VLAN 10 can ping its gateway.
- PC-A cannot reach PC-B in VLAN 20.
- The switch trunk is missing VLAN 20 in the allowed list.

Manual investigation:

```bash
show vlan brief
show interfaces trunk
show ip interface brief
show running-config interface g0/1
```

Expected findings:

- VLAN 10 exists and the access port is assigned correctly.
- The router subinterfaces are configured.
- The trunk carries VLAN 10 but not VLAN 20.

Likely fix:

```bash
conf t
interface g0/1
 switchport mode trunk
 switchport trunk allowed vlan add 20
end
```

Re-test:

```bash
ping 192.168.20.1
ping 192.168.20.10
traceroute 192.168.20.10
```

Interpretation:

- If the gateway responds but the remote host still fails, continue checking host IP settings, ACLs, and return path configuration.
- If the remote host now responds, the trunk VLAN omission was the root cause.

**Troubleshooting commands and what they tell you**

- `show ip interface brief`
  - Confirms interface status and IP assignment.
- `show interfaces trunk`
  - Confirms trunk mode, native VLAN, and allowed VLANs.
- `show vlan brief`
  - Confirms access VLAN membership.
- `show running-config`
  - Reveals interface, ACL, and routing configuration.
- `show ip route`
  - Confirms whether the router knows the destination network.
- `show access-lists`
  - Shows ACL entries and hit counts.
- `ping`
  - Tests reachability and basic forwarding.
- `traceroute`
  - Shows the path to a destination and where forwarding stops.

**How to use hit counts correctly**

ACL hit counts are useful, but they are not proof by themselves.

Use them to:

- Confirm that traffic matches an ACL entry
- Identify which ACE is being triggered
- Compare before and after a change

Do not use them alone to conclude:

- The ACL is the only problem
- The ACL is blocking the exact traffic you care about
- The deny entry is the root cause without checking direction and match criteria

A better approach is to combine:

- ACL hit counts
- Interface direction
- Packet source and destination
- Ping or traceroute results
- Configuration review

**Best practices for AI prompts**

When asking AI for help, include:

- Topology type
- Device roles
- Relevant command output
- Exact symptom
- What changed recently
- What you already ruled out

Example prompt:

```text
Cisco router-on-a-stick lab:
- Switch port Gi0/1 connects to router Gi0/0
- VLANs 10 and 20 exist
- PC-A in VLAN 10 can ping 192.168.10.1
- PC-A cannot ping 192.168.20.10
- show interfaces trunk does not list VLAN 20
What is the most likely cause and the safest next command to verify it?
```

Good AI usage habits:

- Ask for the next verification step, not just the answer.
- Request command-by-command reasoning.
- Compare the suggestion with device output.
- Keep a record of the final confirmed cause.

**Exam and lab takeaways**

For CCNA labs, you should be able to:

- Read interface and VLAN status quickly
- Distinguish Layer 1, Layer 2, and Layer 3 failures
- Understand router-on-a-stick behavior
- Interpret ACL direction and scope
- Use AI as a helper without skipping verification

The strongest troubleshooting answer is the one you can prove with configuration and output, not the one that sounds most plausible.

**Quick reference checklist**

- Confirm the symptom with a test.
- Check the local interface state first.
- Verify VLAN membership and trunking.
- Confirm IP addressing and default gateway.
- Check routing or subinterface configuration.
- Review ACLs and their direction.
- Re-test after each change.
- Use AI to accelerate analysis, then validate manually.
