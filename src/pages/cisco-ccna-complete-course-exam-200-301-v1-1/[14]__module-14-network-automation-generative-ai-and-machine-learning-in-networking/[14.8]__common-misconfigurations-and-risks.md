## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Why misconfigurations matter**

Most network outages are not caused by broken hardware; they come from mismatched assumptions between devices. In CCNA troubleshooting, you need to recognize how a small configuration error can create a total outage, partial connectivity, asymmetric routing, or intermittent failures.

Common risks usually fall into a few categories:

- **Layer 2** issues such as VLAN, trunk, STP, and EtherChannel mismatches
- **Layer 3** issues such as missing gateways, bad masks, or incorrect static routes
- **Policy** issues such as ACLs blocking management or routing traffic
- **Services** issues such as DHCP, IPv6, and relay misconfiguration
- **Security** issues such as snooping, DAI, and port security side effects

A good troubleshooting habit is to ask: “Is the failure local to one host, one VLAN, one subnet, or the entire path?” That narrows the likely misconfiguration quickly.

**Layer 2 misconfigurations**

Layer 2 problems often look like “the link is up, but traffic does not pass.” That usually means the physical interface is fine, but the switching behavior is wrong.

Common VLAN and trunk mistakes include:

- **Access port VLAN mismatch**: the switchport is in the wrong access VLAN, so the host lands in the wrong broadcast domain.
- **Native VLAN mismatch**: untagged traffic is interpreted differently on each side of the trunk, which can cause connectivity issues and security warnings.
- **Allowed VLAN mismatch**: the trunk is up, but the needed VLAN is not permitted across the trunk.
- **Trunk negotiation mismatch**: one side expects a trunk while the other remains access or uses incompatible DTP settings.

Example verification commands:

```bash
show interfaces trunk
show vlan brief
show interfaces switchport
show cdp neighbors detail
```

EtherChannel and <abbr title="Link Aggregation Control Protocol">LACP</abbr> mismatches are a common exam trap. If the member links do not match, the bundle may not form, or only some links may be suspended.

Typical EtherChannel problems include:

- **Mode mismatch**: one side uses `active` while the other uses `on`, or one side uses `passive` with no active partner.
- **Parameter mismatch**: speed, duplex, trunking mode, native VLAN, or allowed VLANs differ across member interfaces.
- **Inconsistent channel-group config**: interfaces are not assigned to the same port-channel number.
- **Partial bundling**: one link joins the bundle while another is suspended because of a mismatch.

Example LACP configuration:

```cisco
interface range g0/1 - 2
 switchport mode trunk
 switchport trunk allowed vlan 10,20
 channel-group 1 mode active

interface port-channel 1
 switchport mode trunk
 switchport trunk allowed vlan 10,20
```

If the bundle fails, check:

```bash
show etherchannel summary
show lacp neighbor
show interfaces port-channel 1
```

Spanning Tree Protocol (<abbr title="Spanning Tree Protocol">STP</abbr>) misconfigurations can also create outages:

- **Wrong root bridge placement**: traffic takes an unexpected path, increasing latency or causing blocked links in the wrong place.
- **PortFast on the wrong port**: enabling it on switch-to-switch links can create loops.
- **BPDU Guard on an uplink**: the port may err-disable when it receives Bridge Protocol Data Units (BPDUs).
- **Loop caused by redundant links without STP protection**: broadcast storms and MAC table instability can follow.

**Inter-VLAN routing misconfigurations**

Inter-VLAN routing failures are often caused by a mismatch between the VLAN, the switchport, and the Layer 3 gateway.

On a multilayer switch, common problems include:

- **Missing Switched Virtual Interface (SVI) IP address**: the VLAN interface exists but has no IP, so hosts have no gateway.
- **SVI shutdown state**: the interface is administratively down, often because the VLAN is not active or the interface was never enabled.
- **Wrong subnet mask**: hosts and gateway disagree on the subnet boundary, causing local delivery failures.
- **VLAN not active on the switch**: the SVI stays down if the VLAN does not exist or has no active ports.
- **Incorrect default gateway on hosts**: hosts send traffic to the wrong router or SVI.

Example SVI configuration:

```cisco
vlan 10
 name USERS

interface vlan 10
 ip address 192.168.10.1 255.255.255.0
 no shutdown
```

On a router-on-a-stick design, the most common errors are:

- **Missing subinterface encapsulation**: the subinterface exists, but `encapsulation dot1Q` is absent or uses the wrong VLAN ID.
- **Wrong subinterface IP address**: the gateway address does not match the host subnet.
- **Parent interface down**: all subinterfaces fail if the physical interface is down.
- **Trunk mismatch on the switch port**: the router expects tagged VLAN traffic, but the switch port is not trunking.

Example router-on-a-stick configuration:

```cisco
interface g0/0
 no shutdown

interface g0/0.10
 encapsulation dot1Q 10
 ip address 192.168.10.1 255.255.255.0

interface g0/0.20
 encapsulation dot1Q 20
 ip address 192.168.20.1 255.255.255.0
```

If hosts in VLAN 10 can reach the gateway but not VLAN 20, verify the trunk, subinterface VLAN tags, and host default gateway first.

**Routing misconfigurations**

Routing issues often produce one-way connectivity or reachability to some networks but not others. Static routes are especially prone to simple but costly mistakes.

Common static route errors include:

- **Incorrect subnet mask or prefix length**: the route points to the wrong network size, so traffic is either too broad or too narrow.
- **Wrong next-hop address**: the router forwards to an unreachable or incorrect neighbor.
- **Wrong exit interface on a multiaccess network**: using only an exit interface on Ethernet can create ambiguity or recursive resolution issues depending on platform and topology.
- **Recursive next-hop resolution problems**: the router must resolve the next-hop through another route, which can fail if the supporting route is missing.
- **Overlapping routes**: a more specific route overrides a less specific one, sometimes unexpectedly.

Example static routes:

```cisco
ip route 10.10.20.0 255.255.255.0 192.0.2.2
ip route 10.10.30.0 255.255.255.0 g0/0
```

Verify with:

```bash
show ip route
show ip route static
show ip cef
ping
traceroute
```

A useful troubleshooting rule is to confirm the route in both directions. A host may reach a server, but the return path may fail if the server’s default gateway or static route is wrong.

Dynamic routing can also fail because of:

- **Passive-interface mistakes**: neighbors never form because the interface is passive.
- **Area or network statement mismatch**: the interface is not included in the routing process.
- **Authentication mismatch**: routing updates are rejected.
- **Administrative distance conflicts**: the wrong route wins in the routing table.

**ACL misconfigurations and control-plane impact**

Access Control Lists (<abbr title="Access Control List">ACL</abbr>s) are powerful, but a small mistake can block legitimate traffic or management access.

Standard ACLs match only source IP addresses, so placement matters:

- Place **standard ACLs close to the destination** to avoid blocking unrelated traffic from the same source.
- Use them when you only need to filter by source and the destination is the key control point.

Extended ACLs match source, destination, protocol, and ports, so placement matters differently:

- Place **extended ACLs close to the source** to stop unwanted traffic early.
- Use them when you need precise control over application traffic.

Common ACL mistakes include:

- **Wrong direction**: applying the ACL inbound instead of outbound, or vice versa.
- **Wrong interface**: the ACL is attached to the wrong interface or subinterface.
- **Implicit deny surprise**: traffic not explicitly permitted is dropped.
- **Wrong wildcard mask**: the ACL matches too much or too little.
- **Sequence order error**: a broad deny appears before a needed permit.

ACLs can also affect control-plane traffic:

- **SSH/Telnet management access** can fail if VTY access is filtered or if the source subnet is denied.
- **Routing protocols** can fail if multicast or protocol-specific traffic is blocked.
- **ICMP** may be blocked, making troubleshooting harder even when forwarding still works.

Example ACL and VTY restriction:

```cisco
ip access-list standard MGMT-SOURCES
 permit 192.168.10.0 0.0.0.255
 deny any

line vty 0 4
 access-class MGMT-SOURCES in
 transport input ssh
```

If you lock yourself out, check whether the management subnet is permitted and whether the ACL is applied to the correct VTY lines or interface.

**DHCP misconfigurations and relay issues**

Dynamic Host Configuration Protocol (<abbr title="Dynamic Host Configuration Protocol">DHCP</abbr>) failures often appear as APIPA/Auto-IP addresses, no default gateway, or intermittent lease assignment.

Common server-side issues include:

- **Missing exclusion range**: the server hands out addresses reserved for infrastructure devices.
- **Wrong scope mask**: clients receive addresses that do not match the subnet.
- **Lease exhaustion**: the pool runs out of available addresses.
- **Bad default gateway option**: clients get a usable IP but cannot reach other networks.
- **Wrong DNS option**: name resolution fails even though IP connectivity works.

Example DHCP pool:

```cisco
ip dhcp excluded-address 192.168.10.1 192.168.10.20

ip dhcp pool USERS
 network 192.168.10.0 255.255.255.0
 default-router 192.168.10.1
 dns-server 8.8.8.8
```

Relay problems are common in multi-VLAN networks:

- **Missing helper address**: broadcasts do not cross the router boundary.
- **Helper on the wrong interface**: the relay is configured on a different VLAN or physical interface than the client subnet.
- **Multiple helper addresses**: useful for redundancy, but can create confusion if one server is misconfigured or slow.
- **Relay path blocked by ACLs**: UDP ports 67 and 68 may be filtered.

Example relay configuration:

```cisco
interface vlan 10
 ip helper-address 192.0.2.50
 ip helper-address 192.0.2.51
```

Remember that DHCP relay forwards client broadcasts as unicast requests to the server. If the relay is on the wrong interface, the server never sees the request from the correct client subnet.

**IPv6 misconfigurations and neighbor discovery**

IPv6 connectivity depends on correct addressing, router advertisements, and Neighbor Discovery Protocol (<abbr title="Neighbor Discovery Protocol">NDP</abbr>) behavior.

Common IPv6 issues include:

- **Missing global unicast address**: the interface only has a link-local address, so it cannot route beyond the local link.
- **Wrong prefix length**: hosts and routers disagree on the subnet boundary.
- **Duplicate address**: Duplicate Address Detection (DAD) prevents use of the address.
- **Disabled forwarding on a router**: the device cannot act as an IPv6 router.
- **Missing default route**: the host or router cannot reach remote prefixes.

Host behavior depends on Router Advertisement (<abbr title="Router Advertisement">RA</abbr>) flags:

- **SLAAC** (<abbr title="Stateless Address Autoconfiguration">SLAAC</abbr>) uses router advertisements to build the address automatically.
- **DHCPv6** can provide addresses or only additional information.
- **M flag** indicates managed addressing, meaning the host should use DHCPv6 for an address.
- **O flag** indicates other configuration, meaning the host should use DHCPv6 for options such as DNS.

Common RA/DHCPv6 problems include:

- **Wrong M/O flags**: hosts use the wrong provisioning method.
- **No RA on the segment**: SLAAC hosts never learn a prefix or default gateway.
- **DHCPv6 server unreachable**: relay or routing is missing.
- **Link-local next-hop issues**: the router is reachable only by link-local address, but the neighbor entry is incomplete or stale.

Example IPv6 configuration:

```cisco
interface g0/0
 ipv6 address 2001:db8:10:1::1/64
 ipv6 enable
 ipv6 nd managed-config-flag
 ipv6 nd other-config-flag
```

Useful checks:

```bash
show ipv6 interface brief
show ipv6 neighbors
show ipv6 route
ping ipv6 2001:db8:10:1::2
```

If IPv6 hosts can talk to the local router but not beyond it, verify the RA flags, the default gateway learned by the host, and whether the router is advertising the correct prefix.

**Security feature misconfigurations**

Security features improve resilience, but they can also break legitimate traffic if you trust the wrong port or omit required bindings.

DHCP Snooping issues include:

- **Untrusted uplink**: the switch drops legitimate DHCP offers from the server because the server-facing port is not trusted.
- **Trusted access port**: a rogue client can send DHCP replies if the access port is trusted.
- **Missing binding table**: Dynamic ARP Inspection (DAI) cannot validate ARP packets without DHCP snooping bindings.
- **VLAN not enabled for snooping**: the feature does not protect the intended VLAN.

Example:

```cisco
ip dhcp snooping
ip dhcp snooping vlan 10,20

interface g0/1
 ip dhcp snooping trust

interface range g0/2 - 24
 ip dhcp snooping limit rate 15
```

DAI depends on DHCP snooping bindings and trusted uplinks:

- If the binding table is empty, valid ARP packets may be dropped.
- If the uplink is not trusted, ARP replies from the gateway or server may be blocked.
- If the wrong VLAN is protected, attacks may pass through the unprotected VLAN.

Port security can also cause outages:

- **Violation mode shutdown** disables the port after a MAC address violation.
- **Sticky MAC mismatch** occurs when the learned address changes after a device replacement.
- **Too few allowed MAC addresses** blocks legitimate downstream devices.

**Practical troubleshooting workflow**

Use a layered approach so you do not chase symptoms instead of causes.

1. **Identify the scope**
   - One host, one VLAN, one subnet, or the whole site?
2. **Check Layer 1 and Layer 2 first**
   - Verify link status, VLAN membership, trunking, and EtherChannel state.
3. **Confirm addressing**
   - Check IP address, mask, default gateway, and IPv6 prefix.
4. **Inspect policy**
   - Review ACLs, VTY access, DHCP snooping, and port security.
5. **Validate routing**
   - Confirm routes in both directions and test with `ping` and `traceroute`.
6. **Check service behavior**
   - Verify DHCP leases, IPv6 RA/DHCPv6 behavior, and neighbor tables.

Useful commands:

```bash
show ip interface brief
show interfaces status
show running-config
show access-lists
show ip dhcp binding
show ip dhcp snooping binding
show ipv6 neighbors
show spanning-tree
show etherchannel summary
```

**High-value exam patterns**

These are the kinds of failures you should recognize quickly:

- A trunk is up, but one VLAN is missing from the allowed list.
- An EtherChannel forms partially because one member link has a mode mismatch.
- A host in VLAN 20 has the wrong default gateway, so it cannot reach remote networks.
- A router-on-a-stick subinterface uses the wrong `dot1Q` VLAN tag.
- A static route points to the wrong prefix length, so traffic misses the destination.
- An ACL blocks SSH to the router because the management subnet is not permitted.
- DHCP works on one VLAN but not another because the helper address is on the wrong interface.
- IPv6 hosts get an address, but no default route because RA is missing or filtered.
- DHCP snooping is enabled, but the server-facing port is untrusted, so offers are dropped.
- DAI is enabled, but the binding table is empty, so valid ARP traffic is blocked.

**Quick comparison of common failure types**

| Area | Typical symptom | Common root cause |
|---|---|---|
| VLAN/trunk | Host can reach local devices only | Wrong VLAN, native VLAN mismatch, or missing allowed VLAN |
| EtherChannel | Partial connectivity or suspended links | LACP mode or parameter mismatch |
| Inter-VLAN routing | Same-VLAN works, other VLANs fail | Missing SVI IP, shutdown SVI, or bad subinterface tag |
| Static routing | Some remote networks unreachable | Wrong mask, wrong next hop, or recursive resolution issue |
| ACLs | Management or app traffic blocked | Wrong direction, placement, or implicit deny |
| DHCP | APIPA address or no gateway | Missing helper, exhausted pool, or bad scope |
| IPv6 | Link-local works, remote fails | Missing RA, wrong flags, or neighbor discovery issue |
| Security features | Legitimate traffic dropped | Untrusted uplink, missing bindings, or violation action |

**Key takeaways**

Misconfigurations are often subtle because the network may appear partially functional. The best exam strategy is to map the symptom to the layer where the failure is most likely, then verify the configuration that controls that layer.

- **Layer 2** errors usually affect local reachability, trunks, and bundles.
- **Layer 3** errors usually affect gateway access and remote networks.
- **Policy** errors often block specific protocols or management access.
- **Service** errors often show up as missing addresses, missing DNS, or missing IPv6 defaults.
- **Security** errors often look like random drops until you inspect trust, bindings, and violation logs.
