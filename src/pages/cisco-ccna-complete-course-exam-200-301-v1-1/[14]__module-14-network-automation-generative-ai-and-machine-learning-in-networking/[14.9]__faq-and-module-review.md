## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**How to use this review**

This module is a consolidation pass for the CCNA blueprint. You should use it to connect topics that often appear together on the exam: VLANs with trunks, STP with EtherChannel, routing with first-hop redundancy, and IP services with NAT, ACLs, and DHCP. The goal is not memorization alone; it is recognizing the problem, choosing the right tool, and validating the result with commands.

A good review strategy is to move from layer 1 upward, then finish with services and automation. That mirrors how many troubleshooting scenarios are built: physical and data-link issues first, then IP reachability, then policy and service behavior.

**Core exam mindset**

CCNA questions often test whether you can identify the *best* answer, not just a technically true answer. That means you should know:

- What each feature does.
- Where it is applied.
- What problem it solves.
- Which command confirms it.
- Which symptoms point to a misconfiguration.

> If two answers seem possible, choose the one that matches the network role, the OSI layer, and the least disruptive fix.

**Layered troubleshooting approach**

Use a structured method instead of random command execution. A common approach is to check from the bottom up:

1. Verify physical connectivity and interface status.
2. Verify VLAN, trunk, and switching behavior.
3. Verify IP addressing, routing, and default gateway reachability.
4. Verify policy features such as ACLs, NAT, and wireless security.
5. Verify service dependencies such as DHCP, DNS, and redundancy.

This approach reduces guesswork and helps you interpret symptoms correctly. For example, if a host cannot reach the internet, the issue may be a missing default route, a NAT overload problem, or an ACL blocking return traffic. The same symptom can have different causes, so you should isolate the layer first.

**Switching review: VLANs, trunks, STP, and EtherChannel**

VLANs segment a Layer 2 network into separate broadcast domains. You use them to improve containment, simplify policy, and reduce unnecessary broadcast traffic. Trunks carry multiple VLANs between switches or between a switch and a router/firewall.

Common verification commands include:

```bash
show vlan brief
show interfaces trunk
show interfaces switchport
```

Spanning Tree Protocol (STP) prevents Layer 2 loops by blocking redundant paths. You should know that STP is about loop prevention, not load balancing. If a link is blocked, that is often expected behavior, not a fault.

EtherChannel bundles multiple physical links into one logical port-channel. You use it to increase bandwidth and provide redundancy while keeping STP from blocking each member link individually. It is commonly configured with LACP (Link Aggregation Control Protocol) or PAgP (Port Aggregation Protocol), though LACP is the more common modern choice.

Example EtherChannel configuration:

```cisco
interface range gi0/1 - 2
 switchport mode trunk
 channel-group 1 mode active

interface port-channel1
 switchport mode trunk
```

Key EtherChannel points:

- All member links must match on speed, duplex, and switchport settings.
- The port-channel is the logical interface you troubleshoot first.
- If one member fails, the bundle can stay up if enough links remain active.
- EtherChannel and STP work together; STP sees the bundle as one link.

**Routing review: static routes, dynamic routing, and default routes**

Routing moves traffic between networks. On CCNA, you should be comfortable with static routes, default routes, and the idea of dynamic routing protocols even if the exam emphasis is conceptual.

A static route is manually configured and predictable. A default route is the route of last resort, often used on edge routers.

```cisco
ip route 10.10.20.0 255.255.255.0 10.10.10.2
ip route 0.0.0.0 0.0.0.0 192.0.2.1
```

Use static routes when:

- The network is small.
- The path should not change often.
- You want precise control.
- You need a default path to an upstream router or ISP.

Dynamic routing protocols automatically learn routes and adapt to topology changes. For CCNA, you should recognize that they reduce manual configuration and improve scalability, but they also add protocol overhead and design complexity.

Useful verification commands:

```bash
show ip route
show ip protocols
show ip interface brief
```

**First-hop redundancy: HSRP, VRRP, and GLBP**

First-hop redundancy provides a resilient default gateway for hosts. If the active gateway device fails, another device takes over so clients keep reaching off-subnet destinations.

You should recognize the purpose of each:

- **HSRP** (Hot Standby Router Protocol): Cisco proprietary gateway redundancy.
- **VRRP** (Virtual Router Redundancy Protocol): Standards-based gateway redundancy.
- **GLBP** (Gateway Load Balancing Protocol): Cisco proprietary and can load-balance gateway use across multiple routers.

At CCNA level, the main idea is that hosts use a virtual IP address as their default gateway, while one device is active and another is ready to take over. This protects against a single router or Layer 3 switch failure.

Practical interpretation:

- If the active gateway fails, hosts should not need manual reconfiguration.
- The virtual MAC address and virtual IP help preserve continuity.
- These protocols are most relevant at the distribution or edge layer.

**IP services: DHCP, DNS, NAT, and PAT**

DHCP (Dynamic Host Configuration Protocol) automates address assignment. DNS (Domain Name System) resolves names to IP addresses. NAT (Network Address Translation) changes IP addresses between inside and outside networks. PAT (Port Address Translation), also called NAT overload, maps many inside hosts to one public IP using different source ports.

NAT/PAT is a major exam topic because it is common in edge designs and easy to test with traffic-flow questions.

Typical NAT terminology:

- **Inside local**: private address assigned to an internal host.
- **Inside global**: public address representing that host on the outside.
- **Outside local**: outside host address as seen from inside, often unchanged.
- **Outside global**: real public address of the outside host.

Example PAT configuration:

```cisco
interface g0/0
 ip address 192.168.10.1 255.255.255.0
 ip nat inside

interface g0/1
 ip address 203.0.113.2 255.255.255.252
 ip nat outside

access-list 1 permit 192.168.10.0 0.0.0.255

ip nat inside source list 1 interface g0/1 overload
```

Verification commands:

```bash
show ip nat translations
show ip nat statistics
show access-lists
```

Common NAT/PAT failure causes:

- The inside and outside interfaces are reversed.
- The ACL does not match the inside subnet.
- The route to the outside network is missing.
- Return traffic is blocked by an ACL or upstream policy.
- The NAT pool or public address is exhausted.

**ACLs: purpose, placement, and common use cases**

ACLs (Access Control Lists) filter traffic based on rules. They are used for security, traffic control, and policy enforcement. On CCNA, you should know both standard and extended ACLs.

- **Standard ACLs** match only source IP address.
- **Extended ACLs** match source and destination IP, protocol, and ports.

Placement matters:

- Put **extended ACLs** close to the source of unwanted traffic.
- Put **standard ACLs** close to the destination when you want to avoid blocking legitimate traffic from other sources.

This placement rule helps you reduce unnecessary traffic and avoid overblocking. For example, if you want to stop a subnet from reaching a specific server, an extended ACL near the source is usually the better choice. If you only want to restrict which networks can reach a destination, a standard ACL near the destination may be sufficient.

Example extended ACL:

```cisco
ip access-list extended WEB-FILTER
 deny tcp 192.168.10.0 0.0.0.255 host 10.10.20.10 eq 80
 permit ip any any
```

Apply it to an interface:

```cisco
interface g0/0
 ip access-group WEB-FILTER in
```

ACL review points:

- ACLs are processed top-down, first match wins.
- There is an implicit `deny any` at the end.
- Order matters more than many students expect.
- You should place explicit permits before broader denies when needed.
- ACLs can be applied inbound or outbound depending on traffic direction and design.

Common use cases:

- Restricting management access to devices.
- Filtering user access to specific servers.
- Controlling routing protocol or service traffic.
- Supporting NAT by matching inside source networks.

**Wireless review: SSIDs, security, and enterprise authentication**

Wireless networks use SSIDs (Service Set Identifiers) to identify WLANs. In enterprise environments, security is usually stronger than simple pre-shared keys because user identity and device trust matter.

You should know the basic enterprise authentication flow:

- The client associates to the access point.
- The client starts 802.1X authentication.
- The access point acts as the authenticator.
- The RADIUS server validates credentials.
- EAP (Extensible Authentication Protocol) carries the authentication exchange.

This flow matters because the AP does not usually make the final trust decision by itself. Instead, it forwards authentication information to a RADIUS server, which centralizes policy and identity control.

Practical wireless security points:

- **WPA2/WPA3** are common security modes.
- **PSK** (Pre-Shared Key) is simpler but less scalable for enterprises.
- **802.1X** is preferred for user-based access control.
- RADIUS supports centralized authentication, authorization, and accounting.

Troubleshooting wireless failures often involves checking:

- SSID broadcast and correct network selection.
- Authentication method mismatch.
- Wrong PSK or expired credentials.
- VLAN assignment or policy mismatch after authentication.
- Signal strength, channel overlap, and interference.

**Automation and programmability**

Automation reduces repetitive configuration and improves consistency. On CCNA, you should understand the purpose of APIs and the difference between configuration retrieval and configuration push.

Common concepts:

- **REST** (Representational State Transfer) uses HTTP methods such as `GET`, `POST`, `PUT`, and `DELETE`.
- **RESTCONF** exposes network configuration and operational data through RESTful APIs.
- **NETCONF** is a network configuration protocol that uses structured data, often XML, to manage devices.
- **YANG** models define the data structure that APIs and management tools use.

Simple RESTCONF-style workflow example:

```bash
curl -k -u admin:password \
  -H "Accept: application/yang-data+json" \
  https://router.example.com/restconf/data/ietf-interfaces:interfaces
```

What this means:

- You authenticate to the device.
- You request structured interface data.
- The device returns machine-readable output that automation tools can parse.

Why this matters:

- APIs reduce manual CLI repetition.
- Structured data is easier to validate than screen-scraped text.
- Automation tools can enforce standard configurations across many devices.

**Security review: device access and management protection**

Device hardening protects the management plane. You should know the difference between securing user traffic and securing device access.

Important controls include:

- Strong passwords and local user accounts.
- `enable secret` for privileged access.
- SSH instead of Telnet.
- Banner messages for legal notice.
- Limiting management access with ACLs.
- Disabling unused services and ports.

Example SSH setup:

```cisco
hostname SW1
ip domain-name example.local
username admin privilege 15 secret Str0ngP@ss!
crypto key generate rsa modulus 2048
line vty 0 4
 transport input ssh
 login local
```

Security best practices:

- Use SSH because Telnet sends credentials in clear text.
- Restrict VTY access to trusted management subnets.
- Use unique credentials and avoid shared accounts when possible.
- Keep management traffic on a dedicated VLAN or out-of-band network when available.

**Troubleshooting commands and interpretation**

You should not just memorize commands; you should know what each one tells you.

```bash
show ip interface brief
show running-config
show cdp neighbors
show lldp neighbors
show mac address-table
show spanning-tree
show etherchannel summary
show ip route
show access-lists
show ip nat translations
show wireless summary
```

Interpretation tips:

- `show ip interface brief` quickly reveals administratively down or protocol-down interfaces.
- `show mac address-table` helps you trace where a host is connected.
- `show spanning-tree` shows root bridge status and blocked ports.
- `show etherchannel summary` confirms whether the bundle is up and which members are active.
- `show access-lists` shows hit counts, which help confirm whether traffic matches a rule.
- `show ip nat translations` confirms whether NAT is actually translating traffic.

If an ACL appears correct but traffic still fails, check:

- Interface direction of the ACL.
- Rule order and implicit deny.
- Whether the return path is blocked.
- Whether the ACL is applied to the correct interface and direction.

If NAT appears correct but internet access fails, check:

- Inside/outside interface roles.
- Matching ACL.
- Default route or upstream reachability.
- Translation table entries and hit counts.

**Common exam traps**

Many CCNA questions are designed around small details. Watch for these traps:

- Confusing a trunk with an access port.
- Assuming STP blocking means failure.
- Forgetting that ACLs are processed top-down.
- Mixing up NAT and PAT.
- Treating 802.1X as the same thing as WPA2-PSK.
- Assuming EtherChannel works when member interfaces do not match.
- Forgetting that a default route is needed for unknown destinations.
- Choosing a routing answer when the real issue is Layer 2 or policy.

**Quick comparison reference**

| Topic | Main purpose | Common clue | Verification |
|---|---|---|---|
| VLAN | Segment broadcast domains | Hosts isolated by design | `show vlan brief` |
| Trunk | Carry multiple VLANs | Switch-to-switch link | `show interfaces trunk` |
| STP | Prevent loops | One link blocked | `show spanning-tree` |
| EtherChannel | Bundle links | Port-channel interface | `show etherchannel summary` |
| ACL | Filter traffic | Permit/deny policy | `show access-lists` |
| NAT/PAT | Translate addresses | Private to public access | `show ip nat translations` |
| HSRP/VRRP/GLBP | Gateway redundancy | Virtual default gateway | Platform-specific show commands |
| 802.1X/RADIUS | Enterprise authentication | User login before access | WLAN/controller status |
| RESTCONF/NETCONF | Automation and management | Structured API access | API request/response |

**Final exam review checklist**

Before the exam, make sure you can explain each of the following in your own words:

- How VLANs, trunks, and STP work together.
- Why EtherChannel is used and what must match on member links.
- How static routes and default routes differ.
- What first-hop redundancy solves.
- How DHCP, DNS, NAT, and PAT support end-user connectivity.
- How ACL placement changes the effect of a rule.
- How 802.1X, EAP, and RADIUS cooperate in enterprise wireless.
- What RESTCONF and NETCONF are used for.
- Which commands you use to verify each feature.

If you can identify the problem layer, name the feature involved, and choose the correct verification command, you are thinking like a CCNA candidate and not just memorizing terms.
