## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Lab overview**

AI-assisted log analysis helps you turn raw device output into actionable troubleshooting clues. In a CCNA context, you use logs to identify interface flaps, authentication failures, spanning tree changes, routing instability, and service errors. AI can accelerate the first pass by summarizing patterns, grouping repeated events, and suggesting likely causes, but you still validate every conclusion against device state and network design.

The core workflow is simple: collect relevant logs, normalize the time window, identify the event sequence, correlate with configuration and topology, and confirm the root cause with direct commands. AI is most useful when the log volume is large or the symptoms are vague.

**What you analyze**

- **Syslog** messages from routers, switches, firewalls, and servers.
- **Interface counters** and state changes.
- **Authentication** and access-control events.
- **Routing** protocol adjacency and convergence messages.
- **Spanning Tree Protocol (STP)** topology changes.
- **DHCP**, DNS, and other service-related failures.

**Why AI helps**

- It reduces time spent scanning repetitive messages.
- It highlights unusual frequency, sequence, or timing.
- It can summarize long logs into a short incident narrative.
- It can suggest hypotheses you then verify with CLI output.

> AI is an assistant, not an authority. Treat its output as a hypothesis until you confirm it with device commands, timestamps, and topology context.

**Log sources and collection**

You need the right data before analysis can be useful. In Cisco environments, logs often come from the device itself, a centralized syslog server, or a monitoring platform. If you only have a partial time window, you may miss the trigger event and misdiagnose the issue.

Common collection points include:

- **Local device buffer** using `show logging`
- **Remote syslog** servers receiving UDP 514 or TCP-based logging
- **SNMP traps** for event notifications
- **AAA** logs for login and authorization events
- **NetFlow/IPFIX** for traffic pattern context, when available

Example commands on a Cisco IOS/IOS XE device:

```bash
show logging
show logging | include LINK|LINEPROTO|SPANTREE|SYS-5|SEC_LOGIN
show clock
show interfaces status
show ip interface brief
```

If you are preparing logs for AI review, include:

- The exact time range of the incident.
- The device hostname and interface names.
- The relevant configuration snippets.
- Any recent change window or maintenance activity.
- The observed user impact.

**Reading syslog effectively**

Syslog messages usually contain a severity, mnemonic, and descriptive text. The mnemonic often points to the subsystem that generated the event. Severity helps you prioritize, but the sequence of messages matters more than a single line.

Typical severity levels:

- `0` Emergency
- `1` Alert
- `2` Critical
- `3` Error
- `4` Warning
- `5` Notification
- `6` Informational
- `7` Debug

A useful pattern is to sort messages into three buckets:

- **Trigger**: the first abnormal event.
- **Cascade**: follow-on messages caused by the trigger.
- **Noise**: unrelated background events.

Example of a common interface flap sequence:

```text
%LINK-3-UPDOWN: Interface GigabitEthernet0/1, changed state to down
%LINEPROTO-5-UPDOWN: Line protocol on Interface GigabitEthernet0/1, changed state to down
%LINK-3-UPDOWN: Interface GigabitEthernet0/1, changed state to up
%LINEPROTO-5-UPDOWN: Line protocol on Interface GigabitEthernet0/1, changed state to up
```

This sequence suggests a physical or Layer 1/Layer 2 interruption. You then verify whether the cause is cabling, duplex mismatch, power loss, port security, or an upstream device issue.

**AI-assisted analysis workflow**

Use a repeatable process so the AI output stays grounded in evidence.

1. Collect the relevant logs and note the time window.
2. Identify the first abnormal event.
3. Group related messages by interface, protocol, or service.
4. Compare the log sequence with the device configuration.
5. Validate the likely cause with live CLI commands.
6. Document the fix and the verification result.

A practical prompt structure is:

```text
Analyze these Cisco syslog messages.
1. Identify the first abnormal event.
2. Group related messages into a timeline.
3. Suggest the most likely root cause.
4. List the CLI commands I should run to verify it.
5. State any assumptions or uncertainties.
```

Good AI output should include uncertainty when evidence is incomplete. If it gives a confident answer without timestamps, topology, or configuration context, treat it cautiously.

**Correlating logs with CLI verification**

Logs rarely tell the full story. You confirm the issue by checking the current state of the device and comparing it to the event history.

Useful verification commands:

```bash
show interfaces gigabitEthernet0/1
show interfaces gigabitEthernet0/1 counters errors
show running-config interface gigabitEthernet0/1
show mac address-table interface gigabitEthernet0/1
show spanning-tree interface gigabitEthernet0/1 detail
show ip route
show ip ospf neighbor
show cdp neighbors detail
```

What each command helps you confirm:

- `show interfaces` reveals line status, errors, duplex, speed, and resets.
- `show interfaces counters errors` helps identify physical or framing problems.
- `show running-config interface` shows shutdown state, port security, VLAN assignment, and speed/duplex settings.
- `show mac address-table` confirms whether the port is learning addresses.
- `show spanning-tree` shows topology changes and port role transitions.
- `show ip route` and `show ip ospf neighbor` help verify routing stability.
- `show cdp neighbors detail` helps identify the connected device and topology.

**Common log patterns and likely causes**

| Log pattern | Likely area | What to check |
|---|---|---|
| `LINK-3-UPDOWN` and `LINEPROTO-5-UPDOWN` | Physical or interface state | Cable, transceiver, speed/duplex, upstream device |
| `SPANTREE` topology change | Layer 2 topology | Redundant links, loops, port roles, STP timers |
| `SEC_LOGIN` failures | Authentication | Username, password, AAA server reachability, lockout policy |
| `DHCP` failures | Address assignment | Scope availability, relay configuration, VLAN placement |
| Routing neighbor down/up messages | Dynamic routing | Interface stability, hello/dead timers, ACLs, MTU mismatch |
| `%SYS-5-CONFIG_I` | Configuration change | Recent admin activity, change window, rollback need |

**Example: interface flap investigation**

Suppose the logs show repeated up/down events on an access port. You should not assume the switch is faulty. The issue could be a bad patch cable, a powered device rebooting, or an upstream negotiation problem.

A focused investigation might look like this:

```bash
show logging | include GigabitEthernet0/1
show interfaces gigabitEthernet0/1
show interfaces gigabitEthernet0/1 counters errors
show cdp neighbors interface gigabitEthernet0/1 detail
show running-config interface gigabitEthernet0/1
```

Interpretation guide:

- Frequent `up/down` messages with increasing CRC or input errors often indicate a Layer 1 problem.
- A port that repeatedly transitions while connected to an endpoint may indicate endpoint rebooting or power cycling.
- If the port is administratively up but never learns MAC addresses, the connected device may be absent or mispatched.
- If the port is in the wrong VLAN, users may report connectivity loss even though the link is up.

**Example: spanning tree event analysis**

STP protects against loops, but topology changes can briefly disrupt traffic. AI can help summarize whether the changes are isolated or recurring.

```bash
show spanning-tree summary
show spanning-tree detail | include ieee|occurr|from|is executing
show spanning-tree interface gigabitEthernet0/2 detail
show logging | include SPANTREE|STP
```

Look for:

- Repeated topology change notifications.
- Ports moving between forwarding and blocking.
- Unexpected root bridge changes.
- Edge ports that are not configured correctly.

If a user reports intermittent connectivity and the logs show frequent STP changes, you should inspect for loops, mispatched cables, or a non-edge device connected to an access port.

**Example: authentication failure analysis**

Authentication logs often show patterns that are easy to miss manually. AI can group repeated failures by source IP, username, or device.

```bash
show logging | include SEC_LOGIN|AAA|AUTH|FAIL
show aaa servers
show running-config | section aaa
```

Possible interpretations:

- Repeated failures from one host may indicate a stale password, misconfigured script, or brute-force attempt.
- Failures across many users may indicate AAA server reachability or policy issues.
- Successful local logins but failed remote logins may point to RADIUS or TACACS+ connectivity problems.

**Practical prompt engineering for log analysis**

The quality of AI output depends on the prompt. Include enough context to reduce guesswork.

Provide:

- Device type and OS, such as IOS, IOS XE, or Catalyst switch.
- The exact log excerpt with timestamps.
- The topology role of the device.
- The expected normal behavior.
- Any recent changes.

Example prompt:

```text
You are analyzing logs from a Cisco access switch running IOS XE.
The switch connects end users on access ports and uplinks to a distribution switch.
Recent change: VLAN 20 was added to Gi0/24.
Analyze the following logs and tell me:
- the first abnormal event
- the most likely root cause
- what commands to run next
- whether this looks like a Layer 1, Layer 2, or Layer 3 issue

Logs:
%LINK-3-UPDOWN: Interface GigabitEthernet0/24, changed state to down
%LINEPROTO-5-UPDOWN: Line protocol on Interface GigabitEthernet0/24, changed state to down
%LINK-3-UPDOWN: Interface GigabitEthernet0/24, changed state to up
%LINEPROTO-5-UPDOWN: Line protocol on Interface GigabitEthernet0/24, changed state to up
```

Better prompts produce better triage. If you omit the device role or change history, the AI may overfit to the log text and miss the operational context.

**Validation and troubleshooting discipline**

AI can suggest a likely root cause, but you still need to prove it. Use a simple rule: if the log suggests a problem, find at least one command that confirms the state and one command that rules out an alternative.

Examples:

- If logs suggest an interface issue, confirm with `show interfaces` and rule out a VLAN mismatch with `show running-config interface`.
- If logs suggest routing instability, confirm neighbor state and rule out link errors.
- If logs suggest authentication failure, confirm server reachability and rule out local credential issues.

> Do not change multiple variables at once during troubleshooting. Make one change, verify the result, and keep notes so you can correlate the next log event with the action you took.

**Mini lab workflow**

1. Capture the logs from the device or syslog server.
2. Copy the relevant excerpt into your analysis tool.
3. Ask for a timeline, root-cause hypothesis, and verification commands.
4. Run the suggested Cisco CLI checks.
5. Compare the AI hypothesis with the live output.
6. Record the confirmed cause and remediation.

Example remediation notes:

- Replace a faulty cable.
- Correct the VLAN assignment on an access port.
- Restore the correct trunk allowed VLAN list.
- Fix AAA server reachability or credentials.
- Adjust STP edge-port configuration where appropriate.

**Troubleshooting checklist**

- **Time sync**: confirm device clocks are accurate with NTP so log order is meaningful.
- **Scope**: focus on the affected device, interface, or service.
- **Sequence**: identify the first event, not just the loudest one.
- **Correlation**: compare logs with config, counters, and topology.
- **Verification**: confirm the fix with fresh output and user testing.

**Key takeaways**

AI-assisted log analysis speeds up triage, but it does not replace network fundamentals. You get the best results when you combine log patterns, device state, and topology knowledge. In CCNA-level troubleshooting, the goal is not to memorize every syslog message; it is to recognize the subsystem involved, identify the first abnormal event, and verify the cause with Cisco CLI commands.
