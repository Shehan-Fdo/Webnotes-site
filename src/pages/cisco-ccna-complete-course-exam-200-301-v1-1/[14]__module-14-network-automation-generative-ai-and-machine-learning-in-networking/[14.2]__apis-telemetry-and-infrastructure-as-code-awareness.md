## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Overview**

Modern network management increasingly uses programmatic interfaces instead of only manual CLI access. For CCNA awareness, you should understand three related ideas:

- **APIs** let software interact with network devices and controllers.
- **Telemetry** lets devices send operational data to a collector or analytics platform.
- **Infrastructure as Code (IaC)** lets you define and manage infrastructure through version-controlled code and templates.

These concepts support automation, consistency, and faster operations. In exam terms, the key is to recognize what each technology does, how it differs from traditional management, and where it fits in a network workflow.

**Traditional Management vs Programmatic Management**

Traditional network administration often uses the command-line interface (CLI) to log in to each device and configure it manually. That approach works, but it does not scale well and is prone to human error.

Programmatic management uses software interfaces and structured data so tools can read, write, and monitor network state more efficiently.

- **CLI-based management**
  - Human enters commands directly on the device.
  - Useful for troubleshooting and one-off changes.
  - Harder to automate at scale.
- **API-based management**
  - Software sends requests to a device, controller, or service.
  - Better for repeatable tasks and integration with tools.
  - Often uses structured payloads such as JSON or XML.
- **Telemetry-based monitoring**
  - Devices stream or export operational data to a collector.
  - Better for near-real-time visibility than periodic polling alone.
  - Common in modern monitoring and analytics systems.

> <mark>Exam focus:</mark> APIs are for programmatic control, telemetry is for operational data collection, and IaC is for defining infrastructure in code.

**APIs: Core Concepts**

An <abbr title="Application Programming Interface">API</abbr> is a defined way for software systems to communicate. In networking, APIs often expose configuration, operational state, or analytics data.

Common API characteristics:

- **Structured requests and responses**
  - Data is exchanged in a machine-readable format.
  - JSON and XML are the most common formats you should recognize.
- **Authentication and authorization**
  - Access is controlled with credentials, tokens, or certificates.
  - The exact method depends on the platform.
- **Transport over HTTP or HTTPS**
  - Many network APIs use web-style requests.
  - HTTPS is preferred because it encrypts traffic.
- **Resource-oriented design**
  - You request or modify a resource such as an interface, VLAN, or routing object.

A simple conceptual REST request might look like this:

```http
GET /restconf/data/ietf-interfaces:interfaces/interface=GigabitEthernet1 HTTP/1.1
Host: 192.0.2.10
Accept: application/yang-data+json
Authorization: Basic <credentials>
```

A corresponding response may return JSON:

```json
{
  "ietf-interfaces:interface": {
    "name": "GigabitEthernet1",
    "enabled": true,
    "description": "Uplink to distribution switch"
  }
}
```

**API Styles You Should Recognize**

For CCNA awareness, you do not need deep implementation detail, but you should know the common names and general purpose.

- **REST**
  - Representational State Transfer.
  - Uses standard HTTP methods such as `GET`, `POST`, `PUT`, `PATCH`, and `DELETE`.
  - Common for web and network automation.
- **RESTCONF**
  - A REST-like protocol for accessing YANG-modeled data.
  - Uses HTTP/HTTPS and structured resources.
  - Common in model-driven network management.
- **NETCONF**
  - A network configuration protocol that uses XML-based messages.
  - Often used with YANG models.
  - More configuration-oriented than REST-style APIs.
- **gRPC**
  - A high-performance remote procedure call framework.
  - Often used with streaming telemetry and modern automation systems.
  - Common in controller and analytics ecosystems.

**Northbound and Southbound Interfaces**

This distinction is important in automation awareness.

- **Northbound interface**
  - Connects a lower-layer system to a higher-layer system.
  - Example: a network controller exposing an API to an orchestration platform.
  - Used by applications, automation tools, and management systems.
- **Southbound interface**
  - Connects a controller or management system to devices below it.
  - Example: a controller pushing configuration to switches and routers.
  - Used to communicate with the infrastructure being managed.

A simple mental model:

- Applications and orchestration tools sit above the controller.
- The controller talks northbound to apps and southbound to devices.
- Devices may expose APIs directly, but controllers often simplify scale and policy enforcement.

**Model-Driven Programmability**

Model-driven programmability means the device or controller exposes data and configuration according to a defined model, often described with <abbr title="Yet Another Next Generation">YANG</abbr>.

Why this matters:

- The model defines what data exists and how it is structured.
- Tools can validate requests against the model.
- Automation becomes more predictable than screen-scraping CLI output.

Key terms:

- **YANG**
  - A data modeling language used to define configuration and state data.
  - Often paired with NETCONF, RESTCONF, and telemetry systems.
- **Schema**
  - The structure and rules for the data model.
  - Helps clients know what fields are valid.
- **Operational state**
  - Data describing what the device is currently doing.
  - Examples include interface counters, neighbor state, and CPU usage.

A model-driven workflow often looks like this:

1. The device exposes a YANG model.
2. An automation tool reads the model or uses a known schema.
3. The tool sends a structured request to configure or query the device.
4. The device returns structured data or an error if the request is invalid.

**Telemetry: What It Is and Why It Matters**

Telemetry is the automated collection of operational data from devices, usually at regular intervals or when state changes occur. It is used for monitoring, troubleshooting, and analytics.

For CCNA, focus on the difference between:

- **Polling**
  - A management system repeatedly asks devices for data.
  - Classic SNMP polling is a common example.
  - Useful, but can be less timely and less scalable at large scale.
- **Streaming telemetry**
  - Devices send data continuously or at configured intervals to a collector.
  - Often more efficient for high-frequency operational visibility.
  - Common in modern monitoring platforms.
- **SNMP traps and notifications**
  - Devices send event-based alerts to a manager.
  - These are push-based, but they are not the same as full streaming telemetry.
  - Traps are useful for alerts, while telemetry is often broader and more structured.

> Telemetry is not simply “push instead of pull.” The exam-relevant distinction is that streaming telemetry provides structured operational data at scale, while SNMP polling and SNMP traps serve different monitoring roles.

**Telemetry Data Flow**

A typical telemetry workflow includes:

- A device collects counters, status, or performance metrics.
- The device exports data to a collector or analytics platform.
- The collector stores, visualizes, or alerts on the data.
- Operations teams use the data for troubleshooting and capacity planning.

Example conceptual flow:

```text
Router/Switch -> Telemetry Collector -> Dashboard / Alerting / Analytics
```

Common telemetry use cases:

- Interface utilization monitoring
- CPU and memory trend analysis
- Routing adjacency and protocol health
- Packet loss and latency visibility
- Fault detection and performance baselining

**Telemetry vs SNMP**

| Feature | SNMP polling | SNMP traps | Streaming telemetry |
|---|---|---|---|
| Data direction | Pull | Push | Push/stream |
| Timing | Periodic | Event-driven | Continuous or periodic stream |
| Data structure | Limited, MIB-based | Event notifications | Structured, model-based |
| Scale | Moderate | Alert-focused | Better suited to large-scale monitoring |
| Typical use | Basic monitoring | Alerts | Real-time visibility and analytics |

Important exam takeaway:

- SNMP polling asks for data.
- SNMP traps notify about events.
- Streaming telemetry sends structured operational data more efficiently for modern monitoring.

**Configuration Awareness Examples**

You are not expected to memorize platform-specific syntax for every API or telemetry feature, but you should recognize what a configuration workflow looks like.

A RESTCONF-style request often uses HTTP headers that describe the payload format:

```http
PATCH /restconf/data/example-module:interfaces/interface=GigabitEthernet1 HTTP/1.1
Host: 192.0.2.10
Content-Type: application/yang-data+json
Accept: application/yang-data+json
Authorization: Basic <credentials>

{
  "example-module:interface": {
    "description": "Uplink to core"
  }
}
```

A NETCONF exchange is XML-based and typically uses an RPC-style message:

```xml
<rpc message-id="101" xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <get-config>
    <source>
      <running/>
    </source>
  </get-config>
</rpc>
```

A telemetry subscription conceptually defines:

- What data to collect
- How often to send it
- Where to send it
- What encoding or transport to use

A simplified conceptual example:

```text
Subscription:
- Source: interface counters
- Interval: 10 seconds
- Destination: telemetry collector
- Format: structured model data
```

> Avoid memorizing misleading vendor-specific syntax unless your lab or platform documentation confirms it. For CCNA, understand the workflow and purpose first.

**JSON and XML in Network Automation**

APIs commonly use structured data formats.

- **JSON**
  - JavaScript Object Notation.
  - Lightweight and easy to read.
  - Common in REST APIs and RESTCONF payloads.
- **XML**
  - Extensible Markup Language.
  - More verbose than JSON.
  - Common in NETCONF and some legacy integrations.

Example JSON payload:

```json
{
  "interface": {
    "name": "GigabitEthernet1",
    "enabled": true,
    "description": "Server uplink"
  }
}
```

Example XML payload:

```xml
<interface>
  <name>GigabitEthernet1</name>
  <enabled>true</enabled>
  <description>Server uplink</description>
</interface>
```

Recognition points:

- JSON is common in modern web-style APIs.
- XML is common in NETCONF and older structured integrations.
- Both are machine-readable and better than parsing free-form CLI output.

**Infrastructure as Code**

Infrastructure as Code means you define infrastructure using files, templates, or scripts instead of only manual configuration. The code becomes the source of truth for what should exist.

Common IaC characteristics:

- **Version control**
  - Store configuration in Git or a similar system.
  - Track changes, review diffs, and roll back when needed.
- **Repeatability**
  - Apply the same template to many devices or environments.
  - Reduce configuration drift.
- **Automation**
  - Use tools to deploy, validate, and update infrastructure.
  - Reduce manual errors.
- **Declarative intent**
  - Describe the desired end state.
  - The tool determines how to reach that state.

Common tools and ecosystems you should recognize:

- **Git**
  - Version control for templates, playbooks, and configuration files.
- **Ansible**
  - Popular automation tool for network and server configuration.
  - Often uses YAML playbooks.
- **Python**
  - Common scripting language for network automation.
  - Used with APIs, parsing, and orchestration.
- **Templates**
  - Reusable configuration patterns.
  - Useful for standardizing device builds.

A simple Ansible-style example:

```yaml
- name: Configure interface description
  hosts: routers
  gather_facts: no
  tasks:
    - name: Set description
      cisco.ios.ios_config:
        lines:
          - description Uplink to core
        parents: interface GigabitEthernet1
```

A simple Python API example:

```python
import requests

url = "https://192.0.2.10/restconf/data/ietf-interfaces:interfaces"
headers = {
    "Accept": "application/yang-data+json",
    "Content-Type": "application/yang-data+json"
}

response = requests.get(url, headers=headers, auth=("admin", "password"), verify=False)
print(response.status_code)
print(response.text)
```

**When to Use API, Telemetry, or IaC**

Use the right tool for the job.

- **Use APIs**
  - When you need to query or change device state programmatically.
  - When integrating network devices with orchestration or external applications.
  - When you need structured access to configuration or operational data.
- **Use telemetry**
  - When you need ongoing visibility into device health and performance.
  - When you want near-real-time operational data for dashboards and alerts.
  - When polling alone is too slow or too expensive.
- **Use IaC**
  - When you want repeatable deployments and standardized configurations.
  - When you manage many devices or frequent changes.
  - When version control and change review matter.

A practical workflow might look like this:

1. Store intended configuration in Git.
2. Use Ansible or Python to push changes through an API or CLI automation library.
3. Use telemetry to verify the device behaves as expected.
4. Alert or roll back if the observed state does not match the intended state.

**Operational and Security Considerations**

Automation improves efficiency, but it also introduces risk if not controlled.

- **Authentication**
  - Protect API access with strong credentials or certificates.
  - Limit access to only what is needed.
- **Authorization**
  - Use role-based access control where available.
  - Separate read-only monitoring from configuration access.
- **Transport security**
  - Prefer HTTPS, SSH, or encrypted channels.
  - Avoid sending sensitive data in clear text.
- **Change control**
  - Review code before deployment.
  - Test in a lab or staging environment first.
- **Validation**
  - Confirm the target device supports the model or API version you use.
  - Mismatched schemas can cause failures.

**Common Exam Recognition Points**

Be able to identify these ideas quickly:

- APIs allow software-to-software communication.
- RESTCONF and NETCONF are model-driven management protocols.
- YANG defines the data model.
- JSON and XML are common payload formats.
- Telemetry provides operational data for monitoring and analytics.
- SNMP polling is pull-based; SNMP traps are event notifications; streaming telemetry is structured data export.
- IaC uses code, templates, and version control to manage infrastructure.
- Northbound interfaces connect controllers to higher-level systems.
- Southbound interfaces connect controllers to devices.

**Study Summary**

For CCNA awareness, focus on the role each technology plays in modern network operations:

- APIs let tools interact with devices and controllers in a structured way.
- Telemetry gives you timely operational data for monitoring and troubleshooting.
- IaC lets you define and deploy infrastructure consistently through code.

If you remember only one workflow, remember this:

- Define intent in code.
- Push or query through an API.
- Observe behavior through telemetry.
- Manage changes through version control and automation tools.
