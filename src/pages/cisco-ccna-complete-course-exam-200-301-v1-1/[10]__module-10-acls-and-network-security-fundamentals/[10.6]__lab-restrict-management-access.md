## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Purpose of the lab**

Restricting management access means you control which hosts can reach a Cisco device for administration. In practice, you usually protect the **VTY** (virtual terminal) lines used by Telnet and SSH, and you may also restrict access to the device itself with an **access control list** (<abbr title="Access Control List">ACL</abbr>) applied to the management interface or to the VTY lines.

The goal is simple: only trusted admin systems should be able to log in, and only through secure methods such as **SSH**. This reduces the attack surface and helps prevent brute-force attempts, unauthorized configuration changes, and accidental access from user subnets.

**Core concepts**

- **VTY lines** are the remote access lines on Cisco IOS devices.
- **SSH** encrypts management traffic; Telnet does not.
- **ACLs** can limit which source IP addresses are allowed to connect.
- **Local authentication** is common in labs and small networks.
- **Privilege levels** and `enable secret` protect access to configuration mode.

A typical secure management design uses:
- A dedicated management subnet or management VLAN
- SSH only, with Telnet disabled
- An ACL that permits only admin hosts
- Strong local usernames and secrets
- Optional logging and timeout settings to reduce idle sessions

**Why you restrict management access**

If management services are reachable from every network, any compromised host can attempt login. Restricting access helps you:
- Reduce exposure to unauthorized users
- Limit brute-force password attempts
- Keep management traffic off user-facing networks where possible
- Make troubleshooting easier by clearly defining who is allowed to administer devices

> <mark>Do not rely on passwords alone.</mark> Combine authentication, encrypted transport, and source restrictions.

**Common lab topology**

A simple lab often includes:
- One Cisco switch or router
- One admin PC on a trusted subnet
- One or more user PCs on untrusted subnets

Example addressing:
- Admin PC: `192.168.10.10/24`
- Management VLAN or interface: `192.168.10.1/24`
- User subnet: `192.168.20.0/24`

Only the admin PC should be able to SSH to the device.

**Baseline SSH configuration**

Before restricting access, make sure SSH is enabled correctly. A common IOS setup looks like this:

```bash
conf t
hostname SW1
ip domain-name example.local
username admin privilege 15 secret Str0ngP@ssw0rd
crypto key generate rsa modulus 2048
ip ssh version 2
line vty 0 15
 login local
 transport input ssh
end
write memory
```

What each part does:
- `ip domain-name` is required for RSA key generation.
- `username ... secret` creates a local account with an encrypted password hash.
- `crypto key generate rsa` creates the key pair used by SSH.
- `ip ssh version 2` forces SSHv2, which is preferred.
- `login local` uses the local username database.
- `transport input ssh` disables Telnet on the VTY lines.

**Restricting VTY access with an ACL**

The most common method is to apply an ACL to the VTY lines using `access-class`. This filters which source IP addresses can open a remote management session.

Example:

```bash
conf t
ip access-list standard MGMT-ONLY
 permit 192.168.10.10
 deny any
exit
line vty 0 15
 access-class MGMT-ONLY in
 transport input ssh
 login local
end
```

How it works:
- The ACL checks the source IP of the incoming SSH connection.
- Only `192.168.10.10` is allowed.
- All other sources are denied.
- The ACL is applied inbound on the VTY lines, not on a physical interface.

**Important behavior**
- `access-class` on VTY lines controls management sessions only.
- It does not filter transit traffic through the device.
- If you need to protect the device interface itself, use an interface ACL or a management-plane feature such as Control Plane Policing in more advanced designs.

**Using a management ACL on the interface**

In some labs, you may also apply an ACL to the management interface or SVI to restrict who can reach the device IP address. This is useful when the device has a dedicated management VLAN.

Example on an SVI:

```bash
conf t
ip access-list standard MGMT-SRC
 permit 192.168.10.10
 deny any
interface vlan 10
 ip access-group MGMT-SRC in
end
```

This approach filters traffic entering the interface. It is different from `access-class`, which filters VTY sessions.

**Standard ACL design for management**

For management access, a **standard ACL** is often enough because you usually care about the source IP address only.

Example with multiple admin hosts:

```bash
ip access-list standard MGMT-ONLY
 permit 192.168.10.10
 permit 192.168.10.11
 permit 192.168.10.12
 deny any
```

Best practices:
- Place the most specific permits first.
- End with an explicit `deny any` for clarity, even though ACLs have an implicit deny.
- Keep the ACL small and easy to audit.
- Use comments or naming conventions if your IOS version supports them in your workflow.

**Verification commands**

After configuration, verify both SSH and ACL behavior.

```bash
show ip ssh
show running-config | section line vty
show access-lists
show users
show ssh
```

What to look for:
- `show ip ssh` confirms SSH is enabled and shows the version.
- `show running-config | section line vty` confirms `access-class`, `login local`, and `transport input ssh`.
- `show access-lists` shows hit counts, which help confirm whether the ACL is matching.
- `show users` shows active remote sessions.
- `show ssh` displays current SSH connections.

You can also test from an allowed host and a denied host:
- Allowed host should connect successfully with `ssh -l admin 192.168.10.1`
- Denied host should fail to establish a session

**Example client-side test**

```bash
ssh -l admin 192.168.10.1
```

If the ACL is correct, a host outside the permitted source list should not reach the login prompt.

**Troubleshooting checklist**

If SSH fails, check the configuration in this order:
1. Confirm the device has an IP address reachable from the admin host.
2. Confirm SSH is enabled with `show ip ssh`.
3. Confirm RSA keys exist.
4. Confirm the VTY lines allow SSH with `transport input ssh`.
5. Confirm the local username and password are correct.
6. Confirm the ACL permits the admin host source IP.
7. Confirm no interface ACL blocks the traffic before it reaches the VTY lines.

Common mistakes:
- Forgetting `ip domain-name` before generating RSA keys
- Leaving Telnet enabled unintentionally
- Applying the ACL in the wrong direction or on the wrong interface
- Permitting the wrong source subnet
- Locking yourself out by denying all sources without a valid permit

> If you are working remotely, <mark>always keep console access available</mark> before changing management ACLs.

**Safer operational practices**

When you change management access on a live device:
- Keep an active console or out-of-band session open
- Add new permit entries before removing old ones
- Test from a second terminal before saving
- Use a maintenance window if the device is production-critical

A cautious workflow looks like this:

```bash
conf t
ip access-list standard MGMT-ONLY
 permit 192.168.10.10
 permit 192.168.10.11
 deny any
exit
line vty 0 15
 access-class MGMT-ONLY in
end
```

Then test from each approved host before committing to the change.

**Related security settings**

Restricting management access is stronger when combined with other controls:
- `service password-encryption` obscures plain-text passwords in the configuration, though it is not strong encryption.
- `enable secret` protects privileged EXEC mode.
- `exec-timeout` logs out idle sessions.
- `login block-for` can slow repeated login attempts on supported platforms.
- `banner motd` can provide an authorized-use warning.

Example:

```bash
conf t
enable secret En@bleS3cret
line vty 0 15
 exec-timeout 5 0
 login local
 transport input ssh
end
```

**Key exam takeaways**

- Use **SSH** instead of Telnet for remote management.
- Use `access-class` on VTY lines to restrict which hosts can connect.
- Use a **standard ACL** when filtering by source IP is sufficient.
- Verify with `show ip ssh`, `show access-lists`, and `show running-config`.
- Avoid locking yourself out by testing before saving changes.

**Quick reference**

| Task | Command |
|---|---|
| Enable SSH | `ip ssh version 2` |
| Create local admin | `username admin privilege 15 secret ...` |
| Restrict VTY by source | `access-class MGMT-ONLY in` |
| Allow only SSH | `transport input ssh` |
| View ACL hits | `show access-lists` |
| View SSH status | `show ip ssh` |

**Summary**

Management access control is a foundational security practice on Cisco devices. In a CCNA lab, the expected solution is usually to enable SSH, create a local admin account, and apply a standard ACL to the VTY lines so only trusted admin hosts can connect. The main idea is to protect the management plane without disrupting normal data traffic.
