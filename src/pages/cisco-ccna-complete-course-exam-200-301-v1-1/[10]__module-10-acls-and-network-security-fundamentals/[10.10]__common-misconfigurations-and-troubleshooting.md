## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Troubleshooting mindset**

Troubleshooting on CCNA-level networks is about isolating the failure domain, not guessing the fix. You start with symptoms, map them to a layer or service, verify the most likely misconfiguration, and then confirm the correction with a repeatable test.

A practical workflow is:

1. Identify the scope: one host, one VLAN, one subnet, one site, or the whole network.
2. Check the simplest causes first: power, cabling, link state, IP settings, and default gateway.
3. Move upward through the stack: Layer 1, Layer 2, Layer 3, and then services such as DHCP, DNS, NAT, ACLs, and routing.
4. Verify with `show` commands before changing anything.
5. Make one change at a time and retest.

**Layer-by-layer troubleshooting model**

A clean mental model helps you avoid chasing symptoms. For example, a host that cannot reach a remote server may have a bad cable, a VLAN mismatch, a missing default gateway, a blocked ACL, or a broken route. The symptom is the same, but the failure point is different.

| Layer | Common symptom | Typical checks |
|---|---|---|
| 1 | Interface down, no link, errors | Cable, optics, speed/duplex, transceiver, counters |
| 2 | VLAN access failure, STP blocking, trunk mismatch | VLAN membership, trunk allowed VLANs, native VLAN, EtherChannel, port security |
| 3 | No inter-VLAN or remote network reachability | IP addressing, masks, default gateway, static routes, dynamic routing, route selection |
| 4-7 | App or service failure | ACLs, NAT, DHCP, DNS, SSH, NTP, SNMP |

**Physical and Layer 1 issues**

Layer 1 problems are often the fastest to confirm because they affect link status and error counters. A bad cable, damaged fiber, incorrect optic, or mismatched speed/duplex can prevent traffic from flowing even when the interface appears administratively enabled.

Common symptoms include:

- `down/down` on an interface, which usually means no physical link.
- High input errors, CRC errors, or runts, which often point to cabling, interference, or duplex problems.
- Late collisions or excessive collisions on older half-duplex scenarios.
- Flapping interfaces caused by loose cables, failing optics, or unstable power.

Useful commands include:

```bash
show ip interface brief
show interfaces
show interfaces status
show interfaces counters errors
show logging
```

Interpret the output carefully:

- `show ip interface brief` tells you whether the interface is administratively down, protocol down, or fully up.
- `show interfaces` gives deeper detail such as line protocol state, duplex, speed, MTU, and error counters.
- `show logging` can reveal link flaps, err-disable causes, or transceiver warnings.

> `show interfaces counters errors` is platform-dependent in exact output and availability. On some labs, `show interfaces counters` or `show interfaces` is more useful for error analysis.

**Speed, duplex, and transceiver problems**

Mismatch between speed and duplex can still appear in labs and older environments, especially when one side is manually configured and the other side negotiates automatically. Modern Ethernet usually auto-negotiates, but you should still verify both ends when you see collisions, late collisions, or poor throughput.

Transceiver and optic issues can also look like a Layer 1 failure:

- Unsupported SFP or GBIC modules may keep the port down.
- Dirty fiber ends can cause intermittent loss.
- Wrong fiber type or polarity can prevent link.
- A copper SFP used with the wrong cable type may fail to come up.

Practical checks:

```bash
show interfaces transceiver
show inventory
show controllers ethernet-controller
```

If the platform supports it, inspect DOM (digital optical monitoring) values for receive power, transmit power, and temperature. Low receive power can indicate a bad fiber path or failing optic.

**Layer 2 switching misconfigurations**

Layer 2 issues are common in CCNA scenarios because they affect VLAN membership, trunking, spanning tree, and port behavior. A host may have link but still fail to reach the gateway if the switchport is in the wrong VLAN or the trunk is not carrying the needed VLAN.

**Access port and VLAN mistakes**

A frequent error is placing a user port in the wrong VLAN or forgetting to create the VLAN on the switch. Another common issue is assigning the correct VLAN to the port but leaving the host with the wrong IP subnet.

Checks:

```bash
show vlan brief
show interfaces switchport
show mac address-table
```

What to look for:

- The access port is assigned to the expected VLAN.
- The VLAN exists and is active.
- The switch learns the host MAC address on the expected interface.
- The host IP address matches the VLAN subnet.

**Trunk misconfigurations**

Trunk problems often appear when one switch can see another switch physically, but VLAN traffic does not pass correctly. Common causes include:

- One side is an access port and the other is a trunk.
- Native VLAN mismatch.
- Allowed VLAN list missing the required VLAN.
- DTP behavior not matching expectations on older gear or labs.

Useful commands:

```bash
show interfaces trunk
show interfaces switchport
show cdp neighbors detail
```

A trunk should agree on encapsulation where relevant, trunking mode, native VLAN, and allowed VLANs. If a VLAN is missing from the allowed list, hosts in that VLAN may fail even though the trunk is up.

**Spanning Tree Protocol issues**

Spanning Tree Protocol (STP) prevents loops, but it can also block a port you expected to forward. That is normal if the topology creates a redundant path. The problem is not STP itself; the problem is when the active path is not the one you intended.

Common symptoms:

- A port is in blocking or alternate state.
- A host on one switch cannot reach another switch even though trunks are up.
- A topology change causes temporary disruption.

Checks:

```bash
show spanning-tree
show spanning-tree vlan 10
show spanning-tree interface gi0/1 detail
```

Important points:

- PortFast is intended for edge or access ports connected to end devices.
- Do not enable PortFast on trunk links unless the design specifically supports it.
- BPDU Guard can protect edge ports, but if enabled on the wrong port it can place the interface into err-disabled state.

> PortFast on a non-edge port can create loops. On trunk links, use it only when the design explicitly calls for it.

**Port security misconfigurations**

Port security is a common cause of an interface going err-disabled. If the switch sees too many MAC addresses, an unexpected MAC address, or a violation action that shuts the port, the host loses connectivity even though the cable is fine.

Typical symptoms:

- Interface enters `err-disabled`.
- The host works briefly and then stops.
- A new device plugged into the port fails immediately.

Check the configuration and status:

```bash
show port-security interface gi0/1
show port-security address
show interfaces status err-disabled
show logging
```

Common causes:

- Maximum MAC address limit is too low.
- Sticky MAC address was learned from a different device.
- Violation mode is `shutdown`.
- The port was moved to a different device without clearing the old secure MAC entry.

Example configuration:

```cisco
interface gi0/1
 switchport mode access
 switchport access vlan 10
 switchport port-security
 switchport port-security maximum 2
 switchport port-security mac-address sticky
 switchport port-security violation restrict
```

Troubleshooting approach:

1. Verify the port is in the correct access VLAN.
2. Check whether the learned MAC matches the connected device.
3. Confirm the violation mode.
4. Clear the secure MAC table or recover the port if needed.

**EtherChannel and link aggregation issues**

EtherChannel bundles multiple physical links into one logical link. If the member links do not match, the bundle may fail to form or may form incorrectly. This is a common switching troubleshooting topic because the physical links can look up while the logical port-channel does not behave as expected.

Common mismatch causes:

- Different speed or duplex settings.
- Different trunk/access mode.
- Different allowed VLAN lists.
- Different native VLANs.
- One side using LACP and the other using PAgP or static mode incorrectly.

Useful commands:

```bash
show etherchannel summary
show etherchannel port-channel
show interfaces port-channel 1
show lacp neighbor
show pagp neighbor
```

Example of a correct LACP configuration:

```cisco
interface range gi0/1 - 2
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30
 channel-group 1 mode active

interface port-channel 1
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30
```

Troubleshooting tips:

- Verify all member interfaces have identical Layer 2 settings.
- Confirm both sides use compatible negotiation modes.
- Check whether the port-channel is suspended due to a mismatch.
- Remember that the logical interface is the one that should carry the intended configuration.

**Layer 3 addressing and gateway problems**

Layer 3 issues often look like “the network is down,” but the root cause is usually a bad IP address, wrong subnet mask, missing default gateway, or incorrect route. A host can communicate locally but fail to reach remote networks if any of these are wrong.

Common host-side mistakes:

- Wrong IP address.
- Wrong subnet mask.
- Wrong default gateway.
- Duplicate IP address.
- Incorrect DNS server when the issue is name resolution rather than reachability.

Verification on a host:

```bash
ipconfig /all
ping 127.0.0.1
ping <default-gateway>
ping <same-subnet-host>
ping <remote-ip>
nslookup example.com
```

A useful pattern is:

- If the host cannot ping its own gateway, focus on Layer 1, Layer 2, VLAN, or gateway IP configuration.
- If it can ping the gateway but not remote networks, focus on routing, ACLs, or NAT.
- If it can ping by IP but not by name, focus on DNS.

**Static route mistakes**

Static routes are simple, but small syntax or design errors can break reachability. A static route must point to the correct destination network, mask, and next hop or exit interface.

Example:

```cisco
ip route 10.10.20.0 255.255.255.0 192.168.1.2
```

Common mistakes:

- Wrong destination network or mask.
- Next hop not reachable from the local router.
- Missing return route on the far side.
- Using only an exit interface on a multiaccess Ethernet network without understanding the ARP behavior.
- Forgetting a default route when the router must send unknown traffic upstream.

Useful commands:

```bash
show ip route
show running-config | include ^ip route
show ip cef
ping 192.168.1.2
traceroute 10.10.20.1
```

A default route is especially important on edge routers and small branch routers:

```cisco
ip route 0.0.0.0 0.0.0.0 203.0.113.1
```

If the default route is missing, traffic to unknown destinations is dropped. If the default route points to the wrong next hop, traffic may blackhole or loop.

**Route selection and administrative distance**

When multiple routes exist to the same destination, the router chooses the best route based on longest prefix match first, then administrative distance, then metric within the same routing protocol. This matters when a static route, a connected route, and a dynamic route overlap.

Common troubleshooting cases:

- A more specific route overrides a less specific route.
- A static route with a better administrative distance overrides a dynamic route.
- A route learned from one protocol is preferred over another because of lower administrative distance.
- A summary route hides a more specific failure until traffic hits the wrong path.

Route codes you should recognize include:

- `C` for connected
- `L` for local
- `S` for static
- `O` for OSPF
- `D` for EIGRP
- `R` for RIP

Check the routing table with:

```bash
show ip route
show ip route static
show ip route ospf
show ip protocols
```

**Dynamic routing issues**

Dynamic routing problems often come from adjacency failure, passive-interface mistakes, area mismatch, authentication mismatch, or missing network statements. In CCNA scenarios, you usually need to confirm whether the router is learning the route at all.

General checks:

```bash
show ip protocols
show ip route
show ip ospf neighbor
show ip eigrp neighbors
show ip rip database
```

Common causes:

- Interfaces are in the wrong subnet or mask.
- The routing process does not include the interface network.
- Authentication settings do not match.
- Passive interfaces prevent neighbor formation where adjacency is required.
- Summarization hides needed specifics.

**ACL troubleshooting**

Access Control Lists (ACLs) are processed top-down, first match wins, and an implicit deny exists at the end. That means a packet can be blocked by an earlier line even if a later line would permit it.

Example:

```cisco
ip access-list extended WEB-FILTER
 permit tcp 192.168.10.0 0.0.0.255 any eq 80
 permit tcp 192.168.10.0 0.0.0.255 any eq 443
 deny ip any any log
```

Troubleshooting steps:

1. Confirm the ACL is applied to the correct interface and direction.
2. Verify the source and destination addresses match the intended traffic.
3. Check the order of entries.
4. Remember the implicit deny if no permit matches.
5. Use counters to see whether the ACL is matching traffic.

Commands:

```bash
show access-lists
show ip access-lists
show running-config interface gi0/0
```

Common mistakes:

- ACL applied inbound when the design expects outbound.
- Wrong wildcard mask.
- Wrong source or destination direction in extended ACLs.
- Missing permit for return traffic in stateful-less filtering designs.
- DNS works but HTTP fails, which may indicate ACL filtering rather than general connectivity.

**NAT misconfigurations**

Network Address Translation (NAT) problems often appear when inside hosts can reach local resources but not the internet. The issue may be missing inside/outside interface roles, wrong ACL match criteria, or no route to the NAT outside next hop.

Typical checks:

```bash
show ip nat translations
show ip nat statistics
show running-config | section ip nat
show access-lists
show ip route
```

Common mistakes:

- `ip nat inside` and `ip nat outside` are applied to the wrong interfaces.
- The NAT ACL does not match the inside source subnet.
- The overload statement is missing for PAT (Port Address Translation).
- The outside interface has no default route.
- Return traffic is blocked by ACLs upstream.

Example PAT configuration:

```cisco
access-list 1 permit 192.168.10.0 0.0.0.255

interface gi0/0
 ip nat inside

interface gi0/1
 ip nat outside

ip nat inside source list 1 interface gi0/1 overload
```

If translations do not appear, verify that traffic is actually leaving the inside network and that the ACL matches the source addresses.

**DHCP problems**

Dynamic Host Configuration Protocol (DHCP) issues usually show up as clients receiving an Automatic Private IP Addressing (APIPA) address such as `169.254.x.x` or failing to obtain a lease. The cause may be a bad DHCP pool, a missing default gateway option, or a relay problem across subnets.

Common checks:

```bash
show ip dhcp binding
show ip dhcp pool
show ip dhcp server statistics
show running-config | section dhcp
```

If clients are on a remote subnet, configure `ip helper-address` on the client-facing interface or switched virtual interface (SVI), not on the DHCP server-facing interface.

Example relay configuration:

```cisco
interface vlan 10
 ip address 192.168.10.1 255.255.255.0
 ip helper-address 192.168.1.100
```

Common mistakes:

- Pool network and mask do not match the client subnet.
- Excluded addresses remove too much of the pool.
- Default router option points to the wrong gateway.
- Relay is missing on the interface that receives client broadcasts.
- ACLs block UDP ports used by DHCP relay and server communication.

**DNS, SSH, NTP, and SNMP service issues**

Service problems often look like connectivity failures even when the network path is fine. You should separate transport reachability from application behavior.

**DNS**

If IP connectivity works but names do not resolve, check DNS server configuration on the host or router.

- Verify the DNS server address.
- Confirm the server is reachable.
- Test with `nslookup` or `dig` where available.
- Check whether ACLs or NAT are blocking DNS traffic.

**SSH**

Secure Shell (SSH) failures often come from missing domain name, missing RSA keys, wrong local user, or VTY line configuration.

Example:

```cisco
hostname R1
ip domain-name example.com
username admin secret Str0ngP@ss
crypto key generate rsa modulus 2048
ip ssh version 2

line vty 0 4
 login local
 transport input ssh
```

Troubleshooting checks:

```bash
show ip ssh
show running-config | section line\ vty
show users
```

Common mistakes:

- Telnet is still allowed when SSH-only access is intended.
- RSA keys were not generated.
- The local username or password is wrong.
- The device lacks a domain name, so key generation fails.

**NTP**

Network Time Protocol (NTP) issues can affect logs and event correlation. If time is wrong, troubleshooting becomes harder because log timestamps are misleading.

Example:

```cisco
ntp server 192.168.1.50
show ntp status
show ntp associations
```

Common mistakes:

- Wrong NTP server IP.
- ACL blocks UDP port 123.
- Device clock is not synchronized.
- Time zone or daylight saving settings are inconsistent.

**SNMP**

Simple Network Management Protocol (SNMP) problems usually affect monitoring tools rather than user traffic. If a monitoring system cannot poll the device, verify community strings or user-based SNMP settings, ACLs, and reachability.

Example SNMPv2c snippet:

```cisco
snmp-server community MONITOR ro
snmp-server host 192.168.1.200 version 2c MONITOR
```

Common mistakes:

- Wrong community string.
- ACL on the SNMP community blocks the manager.
- SNMP packets are routed incorrectly or filtered by ACLs.
- The management station is using the wrong version or credentials.

**Troubleshooting by symptom**

A symptom-based approach helps you narrow the problem quickly.

- If an interface is `down/down`, check cable, optic, and physical connectivity first.
- If an interface is `up/down`, check Layer 2 issues such as encapsulation, VLAN mismatch, STP state, or port security.
- If a host can reach the gateway but not the internet, check routing, NAT, and upstream ACLs.
- If a host can reach IP addresses but not names, check DNS.
- If a switchport goes err-disabled, check port security, BPDU Guard, or other protection features.
- If a trunk is up but a VLAN is missing, check allowed VLANs and native VLAN consistency.
- If a remote subnet is unreachable, check static routes, dynamic routing neighbors, and return paths.

**Practical troubleshooting walkthroughs**

**Scenario 1: Host cannot reach the default gateway**

Symptoms:

- The host has a link light.
- The host cannot ping the gateway.
- Other hosts in the same VLAN may work.

Likely causes:

- Wrong VLAN on the access port.
- Wrong IP address or mask on the host.
- Gateway IP not configured on the router or SVI.
- Port security violation or err-disable.

Verification:

```bash
show vlan brief
show interfaces switchport
show ip interface brief
show port-security interface gi0/1
```

Fix:

- Place the port in the correct VLAN.
- Correct the host IP settings.
- Ensure the gateway interface is up and addressed correctly.

**Scenario 2: Two switches are connected, but one VLAN does not pass**

Symptoms:

- Some VLANs work across the trunk.
- One VLAN fails between switches.

Likely causes:

- VLAN missing from trunk allowed list.
- VLAN not created on one switch.
- Native VLAN mismatch causing unexpected behavior.

Verification:

```bash
show interfaces trunk
show vlan brief
show cdp neighbors detail
```

Fix:

- Add the VLAN to the allowed list on both sides.
- Create the VLAN if needed.
- Align native VLAN settings.

**Scenario 3: Users lose connectivity after a new device is connected**

Symptoms:

- The port goes err-disabled or traffic stops after a MAC change.
- The issue appears only on one access port.

Likely causes:

- Port security violation.
- Sticky MAC mismatch.
- Maximum MAC count exceeded.

Verification:

```bash
show port-security interface gi0/1
show logging
show interfaces status err-disabled
```

Fix:

- Clear the secure MAC entry if appropriate.
- Adjust the maximum MAC count.
- Change the violation mode if the design allows it.

**Scenario 4: Remote network unreachable after adding a static route**

Symptoms:

- Local networks work.
- One remote subnet is unreachable.
- `show ip route` does not show the expected path.

Likely causes:

- Wrong static route mask or next hop.
- Missing return route on the remote router.
- Default route missing on the edge device.
- More specific route overrides the intended path.

Verification:

```bash
show ip route
show running-config | include ^ip route
traceroute 10.10.20.1
```

Fix:

- Correct the static route.
- Add the return route.
- Verify route preference and prefix length.

**Verification habits**

Always verify after each change. A good verification sequence is:

1. Confirm the interface or service state.
2. Confirm the relevant configuration.
3. Confirm the forwarding table or neighbor relationship.
4. Test with a targeted ping, traceroute, or application request.
5. Recheck logs if the issue persists.

Useful commands across many scenarios:

```bash
show running-config
show ip interface brief
show interfaces
show mac address-table
show vlan brief
show interfaces trunk
show spanning-tree
show ip route
show access-lists
show logging
```

**Common exam traps**

- Assuming link up means end-to-end connectivity.
- Forgetting that ACLs are top-down with an implicit deny.
- Checking the wrong interface for `ip helper-address`.
- Forgetting that a trunk can be up while a VLAN is still blocked or missing.
- Ignoring return-path routing when only one direction is tested.
- Confusing DNS failure with general network failure.
- Overlooking port security or err-disable when a port suddenly stops working.
- Treating a summary route as if it proves reachability to all more specific networks.

**Quick reference checklist**

- Verify physical link and error counters.
- Confirm VLAN, trunk, and STP state.
- Check port security and err-disable status.
- Validate IP address, mask, and default gateway.
- Inspect routing table and route selection.
- Review ACLs, NAT, and DHCP relay.
- Test DNS, SSH, NTP, and SNMP separately from basic connectivity.

**Key takeaway**

Most CCNA troubleshooting problems are caused by a small set of misconfigurations: wrong VLANs, bad trunks, missing routes, ACL order, NAT roles, DHCP relay placement, port security violations, and service mismatches. If you work layer by layer and verify each fix, you can isolate the fault quickly and avoid unnecessary changes.
