## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**ACL purpose and filtering model**

An <abbr title="Access Control List">ACL</abbr> is a sequential set of permit and deny statements that filters traffic based on header fields. In CCNA-level design, you use ACLs to control which packets are allowed through a router or Layer 3 interface, and to restrict management access to device lines such as <abbr title="Virtual Teletype">VTY</abbr> lines.

ACLs are **stateless**. They do not track sessions the way a firewall does, so each packet is evaluated independently against the ACL entries in order. The first matching entry wins, and an implicit `deny ip any any` ends the list.

- ACLs are processed top-down, one line at a time.
- The first match determines the action.
- If no entry matches, traffic is denied by the implicit deny.
- Placement matters because ACLs can block traffic before it reaches the intended destination or before it leaves an interface.

**ACL types and when to use them**

CCNA commonly expects you to distinguish between **standard** and **extended** ACLs, and between **numbered** and **named** ACLs.

- **Standard ACLs** match only the source IPv4 address.
- **Extended ACLs** can match source and destination IP addresses, protocol, and ports.
- **Numbered ACLs** use a numeric identifier such as `1-99`, `100-199`, `2000-2699` for IPv4 classic ranges.
- **Named ACLs** use a descriptive name and are easier to read, edit, and troubleshoot.

Use numbered ACLs when you want a quick, simple configuration or when following older examples. Use named ACLs when readability, change control, or multiple entries matter. Named ACLs are especially useful in exam scenarios where you need to recognize intent quickly.

```bash
! Standard numbered ACL
access-list 10 permit 192.168.10.0 0.0.0.255

! Extended named ACL
ip access-list extended WEB-FILTER
 permit tcp 192.168.10.0 0.0.0.255 host 203.0.113.10 eq 443
 deny ip any any
```

**Where to place ACLs**

The classic rule of thumb is:

- Place **standard ACLs closest to the destination**.
- Place **extended ACLs closest to the source**.

This rule exists because standard ACLs can only identify the source, so placing them too early can block that source from reaching many destinations unnecessarily. Extended ACLs can match more specific traffic, so placing them near the source reduces unwanted traffic earlier and conserves bandwidth.

That rule is a guideline, not an absolute law. You may place an ACL elsewhere when the design goal is to protect a specific destination, simplify management, or avoid blocking other legitimate traffic that shares the same source network.

- Use a standard ACL near the destination when you want to restrict a source host or subnet from reaching one or more destination networks.
- Use an extended ACL near the source when you want to stop unwanted traffic before it crosses the network.
- Consider operational impact: a misplaced ACL can unintentionally block multiple services.

**Inbound vs outbound direction**

ACL direction is relative to the interface where you apply it.

- **Inbound** means packets are filtered as they enter the interface.
- **Outbound** means packets are filtered as they leave the interface.

For a routed interface, the packet arrives on the interface, the inbound ACL is evaluated, and then the router performs forwarding decisions if the packet is permitted. Outbound ACLs are checked after the routing decision, just before the packet exits the interface.

This distinction matters because the same ACL line can have very different effects depending on direction.

- Inbound ACLs are useful when you want to stop traffic as soon as it enters a router interface.
- Outbound ACLs are useful when multiple ingress paths converge and you want one policy on the exit interface.
- If you need to protect a destination network from many sources, outbound filtering on the destination-facing interface can be simpler in some topologies.

**Practical placement strategy**

Think about three questions before you apply an ACL:

1. What traffic do you want to block or allow?
2. Where is that traffic easiest to identify?
3. Where can you stop it with the least side effect?

A good strategy is to filter as early as possible when the ACL is extended, and as late as practical when the ACL is standard.

- If you know the protocol and port, use an extended ACL near the source.
- If you only know the source host or subnet, use a standard ACL near the destination.
- If the same source must reach some destinations but not others, prefer an extended ACL so you can be precise.
- If the ACL is meant to protect a server farm, place it where the server traffic converges, but verify that you are not blocking unrelated return paths.

**Return traffic and stateless filtering**

Because ACLs are stateless, they do not automatically “remember” that a connection was allowed earlier. However, that does not mean you always need a separate reverse rule for every flow. The need for a reverse permit depends on where the ACL is applied and which direction the packets travel.

For example:

- If a client initiates a TCP connection and an ACL on the client-side outbound path permits the client’s SYN packet, the return traffic still needs to be permitted by whatever ACLs exist on the return path.
- If you place an ACL on only one interface and only one direction, return traffic may pass if it does not match a deny on that path.
- If you filter both directions or multiple interfaces, you must account for both directions of the conversation.

In practice, you design ACLs around the actual packet path, not around a generic “allow both directions” rule.

> ACLs do not create session state. If a return packet is blocked by a different ACL entry or a different interface policy, the connection fails even if the initial packet was permitted.

**Standard ACL placement examples**

Standard ACLs are best when the requirement is source-based control.

Example requirement: deny host `192.168.10.50` from reaching the `10.10.20.0/24` network, but allow that host to reach other networks.

A standard ACL can identify the source host, but not the destination. So you place it near the destination network, on the interface that leads to `10.10.20.0/24`.

```bash
access-list 10 deny host 192.168.10.50
access-list 10 permit any

interface GigabitEthernet0/1
 description Toward 10.10.20.0/24
 ip access-group 10 out
```

Why this works:

- The ACL blocks the host only when traffic is headed toward the protected destination.
- Other destinations remain reachable because the ACL is applied on the destination-facing path.
- The `permit any` is necessary because of the implicit deny.

Another common use is restricting access to a branch network from a specific management subnet while leaving other traffic unaffected. In that case, a standard ACL on the branch-facing interface outbound can be appropriate.

**Extended ACL placement examples**

Extended ACLs are best when you need to match protocol and port.

Example requirement: allow only HTTP and HTTPS from the user VLAN to a web server, and deny all other traffic to that server.

```bash
ip access-list extended WEB-ACCESS
 permit tcp 192.168.10.0 0.0.0.255 host 203.0.113.10 eq 80
 permit tcp 192.168.10.0 0.0.0.255 host 203.0.113.10 eq 443
 deny ip any host 203.0.113.10
 permit ip any any
```

A more efficient placement is near the source VLAN interface so unwanted traffic is stopped before it crosses the network.

```bash
interface GigabitEthernet0/0.10
 description User VLAN
 ip access-group WEB-ACCESS in
```

This placement reduces unnecessary transit traffic and makes the policy easier to reason about.

**Port matching and exam traps**

Extended ACL port syntax is a common exam trap. The port number usually matches the service on the destination side for client-initiated traffic.

- For a client sending DNS queries, the client uses an ephemeral source port and the destination port is `53/udp`.
- For a web client, the destination port is `80/tcp` or `443/tcp`.
- For SSH to a server, the destination port is `22/tcp`.

Examples:

```bash
! DNS queries from a subnet to any DNS server
permit udp 192.168.10.0 0.0.0.255 any eq 53

! HTTPS to a specific server
permit tcp 192.168.10.0 0.0.0.255 host 203.0.113.10 eq 443

! SSH to a management host
permit tcp any host 198.51.100.20 eq 22
```

Be careful not to confuse source and destination ports. If you write an ACL for client traffic, the service port is usually the destination port. If you write an ACL for traffic coming from a server, the source port may be the service port instead.

**Interface ACLs vs VTY line ACLs**

ACLs are not only for routed interfaces. You also use them on VTY lines to restrict remote management access.

- **Interface ACLs** filter transit traffic passing through a router or Layer 3 switch.
- **VTY ACLs** filter management sessions such as Telnet or SSH to the device itself.
- **Control-plane filtering** is a broader concept, but at CCNA level you mainly need to recognize interface filtering versus line access control.

Example: allow only the management subnet to SSH to the router.

```bash
ip access-list standard MGMT-SSH
 permit 192.168.99.0 0.0.0.255

line vty 0 4
 transport input ssh
 access-class MGMT-SSH in
```

This does not filter transit traffic. It only controls who can open a VTY session to the device.

**IPv6 ACL placement strategy**

IPv6 ACLs follow the same general placement logic as IPv4 ACLs.

- Use extended IPv6 ACLs near the source when you can identify the protocol and service.
- Use source-based filtering near the destination when the requirement is only to restrict a host or subnet.
- Apply ACLs to IPv6-enabled interfaces with `ipv6 traffic-filter`.

Example:

```bash
ipv6 access-list V6-WEB-FILTER
 permit tcp 2001:db8:10:10::/64 host 2001:db8:20::10 eq 443
 deny ipv6 any host 2001:db8:20::10
 permit ipv6 any any

interface GigabitEthernet0/0
 ipv6 traffic-filter V6-WEB-FILTER in
```

IPv6 ACLs are often written with the same logic as IPv4 extended ACLs, but the syntax uses IPv6 prefixes and the `ipv6` keyword.

**Choosing inbound or outbound in real topologies**

Use the packet path to decide direction.

- Apply inbound on the interface where the unwanted traffic first enters the router if you want early filtering.
- Apply outbound on the interface leading to the protected destination if you want one policy for all sources.
- If a router has multiple upstream links, outbound filtering on the destination-facing interface can simplify policy enforcement.
- If a branch router has a single LAN and a single WAN, inbound filtering on the WAN may be the cleanest way to block unwanted inbound traffic.

Example topology decision:

- Users on `192.168.10.0/24` should access only a web server in `203.0.113.0/24`.
- If the router connects the user VLAN and the server VLAN directly, an extended ACL on the user VLAN inbound is efficient.
- If multiple user VLANs feed a distribution router and you want one policy before the server VLAN, an outbound ACL on the server-facing interface may be easier to maintain.

**Stateful vs stateless filtering strategy**

ACLs are stateless, so they are best for simple, deterministic rules. They are not ideal for complex application flows that require dynamic port negotiation or session tracking.

Use ACLs when:

- The traffic pattern is predictable.
- You can identify source, destination, protocol, and port.
- You want lightweight filtering on routers and Layer 3 switches.

Be cautious when:

- The application uses multiple dynamic ports.
- The return path is asymmetric.
- You need session awareness or inspection beyond basic header matching.

In those cases, a stateful firewall is usually a better fit than an ACL-only design.

**Exam-style decision guide**

Use this quick reasoning process during CCNA questions:

1. Identify whether the requirement is source-based or service-based.
2. Choose standard ACL for source-only filtering, extended ACL for protocol/port filtering.
3. Place standard ACL near the destination.
4. Place extended ACL near the source.
5. Decide inbound or outbound based on where the traffic should be stopped with the least impact.
6. Verify return traffic and implicit deny behavior.

**Common configuration patterns**

Standard ACL on a destination-facing interface:

```bash
access-list 15 deny host 192.168.1.50
access-list 15 permit any

interface GigabitEthernet0/1
 ip access-group 15 out
```

Extended ACL on a source-facing interface:

```bash
ip access-list extended BLOCK-TELNET
 deny tcp 192.168.10.0 0.0.0.255 any eq 23
 permit ip any any

interface GigabitEthernet0/0
 ip access-group BLOCK-TELNET in
```

VTY restriction:

```bash
ip access-list standard ADMIN-ONLY
 permit 192.168.99.0 0.0.0.255

line vty 0 15
 access-class ADMIN-ONLY in
 transport input ssh
```

**Verification and troubleshooting**

Always verify both the ACL content and where it is applied.

```bash
show access-lists
show ip interface gigabitEthernet0/0
show running-config | section access-list
show running-config interface gigabitEthernet0/0
show running-config | section line\ vty
```

Look for these issues:

- The ACL is applied to the wrong interface.
- The ACL direction is reversed.
- A permit statement is missing before the implicit deny.
- The port number is matched on the wrong side of the flow.
- A VTY ACL is confused with an interface ACL.
- IPv6 traffic is being filtered with IPv4 ACL syntax, or vice versa.

**Quick comparison**

| Topic | Standard ACL | Extended ACL |
|---|---|---|
| Matches | Source IP only | Source, destination, protocol, ports |
| Best placement | Closest to destination | Closest to source |
| Typical use | Restrict a host or subnet | Filter specific services |
| Precision | Lower | Higher |
| Exam focus | Source-based policy | Service-based policy |

**Key takeaways**

- ACLs are sequential, stateless filters with an implicit deny at the end.
- Standard ACLs match source only, so place them near the destination.
- Extended ACLs match more fields, so place them near the source when possible.
- Direction matters: inbound filters on entry, outbound filters on exit.
- VTY ACLs control management access, not transit traffic.
- IPv6 ACLs use the same placement logic as IPv4 ACLs.
- Always verify the actual packet path before deciding where to apply the ACL.
