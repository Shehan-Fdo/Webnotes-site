## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Module review mindset**

This review ties together the major CCNA domains: network fundamentals, switching, routing, IP services, security, wireless, and automation. For the exam, you need more than definitions. You need to recognize behavior, predict outcomes, and troubleshoot from symptoms.

A good way to study this module is to ask three questions for every topic:

- What problem does it solve?
- What does it look like in configuration or output?
- What breaks when it is misconfigured?

**Network fundamentals recap**

The CCNA expects you to understand how devices communicate at Layer 2 and Layer 3, how addresses are assigned, and how to read common outputs. You should be comfortable moving between the conceptual model and the CLI.

Key ideas:

- **OSI model**: Use it to localize faults. If a host cannot reach a gateway, check Layer 1, Layer 2, and Layer 3 in order.
- **MAC address table**: Switches learn source MAC addresses and forward frames based on destination MAC entries.
- **ARP**: Address Resolution Protocol maps IPv4 addresses to MAC addresses on the local network.
- **ICMP**: Internet Control Message Protocol supports reachability tests such as `ping` and `traceroute`.

Useful verification commands:

```bash
show ip interface brief
show interfaces status
show mac address-table
show arp
ping 192.0.2.1
traceroute 198.51.100.10
```

If a host can ping its default gateway but not remote networks, the issue is often routing, NAT, or an upstream ACL rather than local switching.

**IPv4 addressing and subnetting**

Subnetting is a core CCNA skill. You should be able to determine network, broadcast, usable host range, and the number of subnets or hosts from a prefix length. Variable Length Subnet Masking (VLSM) lets you allocate different subnet sizes efficiently.

Common exam expectations:

- Convert between dotted decimal masks and prefix lengths.
- Identify whether two hosts are in the same subnet.
- Choose the smallest subnet that fits a host requirement.
- Recognize that smaller prefixes mean larger networks.

Quick reference:

| Prefix | Mask | Usable Hosts |
|---|---|---:|
| /24 | 255.255.255.0 | 254 |
| /25 | 255.255.255.128 | 126 |
| /26 | 255.255.255.192 | 62 |
| /27 | 255.255.255.224 | 30 |
| /28 | 255.255.255.240 | 14 |
| /30 | 255.255.255.252 | 2 |

A practical VLSM example:

- You need one subnet for 100 hosts, one for 50 hosts, and one for 10 hosts.
- Start with the largest requirement.
- Allocate `/25` for 100 hosts, `/26` for 50 hosts, and `/28` for 10 hosts if the address block allows it.
- Assign subnets from largest to smallest to reduce waste.

**IPv6 essentials**

IPv6 removes the need for NAT in many designs and uses a much larger address space. You should know the difference between global unicast, link-local, unique local, multicast, and unspecified addresses.

Important behaviors:

- **Link-local** addresses begin with `fe80::/10` and are used for neighbor discovery and next-hop communication.
- **SLAAC** (Stateless Address Autoconfiguration) lets hosts build an address from router advertisements.
- **DHCPv6** can provide addressing and/or other configuration, depending on stateful or stateless mode.
- IPv6 uses Neighbor Discovery Protocol instead of ARP.

Example verification:

```bash
show ipv6 interface brief
show ipv6 neighbors
ping ipv6 2001:db8:1::1
```

A common troubleshooting pattern is to verify that the interface has a link-local address, the router is sending advertisements, and the host has a default route learned from the router.

**Switching and VLANs**

VLANs segment Layer 2 networks into separate broadcast domains. Trunks carry multiple VLANs between switches or between a switch and a router. Access ports belong to one VLAN.

Core concepts:

- **Access port**: Carries traffic for one VLAN.
- **Trunk port**: Carries traffic for multiple VLANs using 802.1Q tagging.
- **Native VLAN**: Untagged VLAN on an 802.1Q trunk; it should match on both ends.
- **Voice VLAN**: Separates voice traffic from data traffic on an access port.

Example configuration:

```bash
vlan 10
 name USERS
vlan 20
 name VOICE

interface gigabitEthernet0/1
 switchport mode access
 switchport access vlan 10

interface gigabitEthernet0/24
 switchport mode trunk
 switchport trunk allowed vlan 10,20
 switchport trunk native vlan 99
```

Troubleshooting checklist:

- Confirm the VLAN exists with `show vlan brief`.
- Confirm trunk status with `show interfaces trunk`.
- Confirm the port mode matches the design.
- Check for native VLAN mismatches if you see unexpected traffic or spanning-tree warnings.

**Inter-VLAN routing**

Inter-VLAN routing allows devices in different VLANs to communicate through a Layer 3 device. CCNA commonly expects you to recognize three approaches:

| Method | Where routing happens | Typical use |
|---|---|---|
| Router-on-a-stick | Router subinterfaces | Small or lab networks |
| Layer 3 switch SVIs | Switched virtual interfaces | Campus access/distribution |
| Routed ports | Physical Layer 3 interfaces | Point-to-point routed links |

Router-on-a-stick example:

```bash
interface gigabitEthernet0/0
 no shutdown

interface gigabitEthernet0/0.10
 encapsulation dot1Q 10
 ip address 192.168.10.1 255.255.255.0

interface gigabitEthernet0/0.20
 encapsulation dot1Q 20
 ip address 192.168.20.1 255.255.255.0
```

Layer 3 switch SVI example:

```bash
ip routing

interface vlan 10
 ip address 192.168.10.1 255.255.255.0
 no shutdown

interface vlan 20
 ip address 192.168.20.1 255.255.255.0
 no shutdown
```

If hosts in the same VLAN cannot reach the gateway, check the SVI or subinterface state, VLAN membership, and trunking before assuming a routing issue.

**Spanning Tree Protocol and loop prevention**

Spanning Tree Protocol (STP) prevents Layer 2 loops by blocking redundant paths. You should understand why loops are dangerous: they create broadcast storms, MAC table instability, and duplicate frames.

Key variants:

| Variant | Meaning | Notes |
|---|---|---|
| STP | Original 802.1D | Slow convergence |
| RSTP | Rapid Spanning Tree 802.1w | Faster convergence |
| MST | Multiple Spanning Tree | Maps VLANs to instances |

Important terms:

- **Root bridge**: The switch with the lowest bridge ID becomes the reference point.
- **Port roles**: Root, designated, alternate, and backup roles determine forwarding behavior.
- **PortFast**: Speeds up edge-port forwarding for end devices.
- **BPDU Guard**: Shuts down a PortFast port if it receives a Bridge Protocol Data Unit.

Useful commands:

```bash
show spanning-tree
show spanning-tree vlan 10
show spanning-tree interface gigabitEthernet0/1 detail
```

A common scenario is a user plugging a small unmanaged switch into an access port. If BPDU Guard is enabled, the port may err-disable to protect the network.

**EtherChannel and link aggregation**

EtherChannel bundles multiple physical links into one logical link for more bandwidth and redundancy. The bundle must match on both ends for speed, duplex, trunking, and negotiation settings.

Protocols:

- **LACP**: Link Aggregation Control Protocol, standard 802.3ad/802.1AX.
- **PAgP**: Port Aggregation Protocol, Cisco proprietary.

Example:

```bash
interface range gigabitEthernet0/1 - 2
 channel-group 1 mode active

interface port-channel1
 switchport mode trunk
 switchport trunk allowed vlan 10,20
```

If one member link is misconfigured, the bundle may form partially or fail. Always verify both the physical interfaces and the Port-Channel interface.

**Routing fundamentals and route selection**

Routers choose the best route based on longest prefix match first, then administrative distance (AD), then metric. Longest prefix match means the most specific route wins.

Route sources and AD values commonly tested:

| Route type | AD |
|---|---:|
| Connected | 0 |
| Static | 1 |
| EIGRP summary | 5 |
| eBGP | 20 |
| EIGRP internal | 90 |
| OSPF | 110 |
| RIP | 120 |
| EIGRP external | 170 |

Static route example:

```bash
ip route 203.0.113.0 255.255.255.0 192.0.2.2
ip route 0.0.0.0 0.0.0.0 192.0.2.1
```

Route selection example:

- A router has `10.10.0.0/16` via OSPF and `10.10.10.0/24` as a static route.
- Traffic to `10.10.10.25` uses the `/24` static route because it is more specific.
- Traffic to `10.10.20.25` uses the `/16` OSPF route because no more specific route exists.

Troubleshooting route issues:

- Use `show ip route` to confirm the route exists.
- Use `show ip protocols` to verify routing protocol participation.
- Check next-hop reachability.
- Confirm the exit interface is up/up.

**OSPF review**

Open Shortest Path First (OSPF) is a link-state routing protocol. It builds a topology map from Link-State Advertisements (LSAs) and runs the Shortest Path First algorithm to compute best paths.

CCNA-level points:

- OSPF uses areas, and area 0 is the backbone.
- Routers form neighbor relationships before exchanging routes.
- Hello and dead timers must match on neighbors.
- OSPF uses cost as its metric.

Basic configuration example:

```bash
router ospf 1
 router-id 1.1.1.1
 network 10.0.0.0 0.0.0.255 area 0
 network 10.0.1.0 0.0.0.255 area 0
```

Verification:

```bash
show ip ospf neighbor
show ip ospf interface brief
show ip route ospf
```

Troubleshooting checklist:

- Confirm the interfaces are in the correct area.
- Check for mismatched subnet masks.
- Verify hello/dead timer consistency.
- Ensure the router ID is unique.
- Confirm passive interfaces are intentional.

**EIGRP review**

Enhanced Interior Gateway Routing Protocol (EIGRP) is an advanced distance-vector routing protocol. It uses the Diffusing Update Algorithm (DUAL) to choose loop-free paths and maintain backups.

Key ideas:

- EIGRP forms neighbor adjacencies using hello packets.
- It uses bandwidth and delay in its composite metric by default.
- Feasible successors provide a backup path.
- Administrative distance is 90 for internal routes.

Example:

```bash
router eigrp 100
 network 10.0.0.0 0.0.0.255
 no auto-summary
```

Verification:

```bash
show ip eigrp neighbors
show ip route eigrp
show ip protocols
```

If neighbors do not form, check K-values, interface network statements, and whether the interfaces are passive.

**Static routing and default routes**

Static routes are simple, predictable, and useful for stub networks or backup paths. They do not adapt automatically to topology changes unless you configure tracking or floating static routes.

Common patterns:

- **Default route**: Sends unknown traffic to a next hop or exit interface.
- **Floating static route**: Uses a higher AD than a dynamic route so it acts as backup.
- **Summary static route**: Aggregates multiple networks into one route.

Example floating static route:

```bash
ip route 10.20.0.0 255.255.0.0 192.0.2.2 200
```

This route is only preferred if no route with a lower AD exists.

**DHCP, DNS, NAT, and NTP**

These services are common in enterprise networks and appear frequently in troubleshooting scenarios.

DHCP basics:

- **DHCP** assigns IP settings dynamically.
- The DORA process is Discover, Offer, Request, Acknowledge.
- A relay agent forwards DHCP broadcasts across subnets.

Example relay configuration:

```bash
interface vlan 10
 ip helper-address 192.0.2.50
```

DHCP snooping helps protect against rogue DHCP servers by trusting only selected ports.

```bash
ip dhcp snooping
ip dhcp snooping vlan 10
interface gigabitEthernet0/24
 ip dhcp snooping trust
```

DNS basics:

- **DNS** resolves names to IP addresses.
- If users can reach IPs but not hostnames, check DNS server settings and name resolution.
- Test with `nslookup` or `dig` where available.

Example:

```bash
nslookup www.example.com
```

NAT basics:

- **NAT** translates private addresses to public addresses.
- PAT (Port Address Translation) overloads many inside hosts onto one public IP.
- Verify inside and outside interfaces carefully.

Example:

```bash
access-list 1 permit 192.168.10.0 0.0.0.255

interface gigabitEthernet0/0
 ip nat inside

interface gigabitEthernet0/1
 ip nat outside

ip nat inside source list 1 interface gigabitEthernet0/1 overload
```

NTP basics:

- **NTP** synchronizes device clocks.
- Accurate time matters for logs, certificates, and troubleshooting.

Example:

```bash
ntp server 192.0.2.100
show ntp status
show ntp associations
```

If logs appear out of order, verify NTP before chasing unrelated faults.

**Security fundamentals**

CCNA security coverage includes device hardening, access control, and common threat awareness. You should know both the configuration and the reason behind it.

Device hardening practices:

- Use strong passwords and encrypted password storage where possible.
- Disable unused services and interfaces.
- Change default credentials.
- Restrict management access to trusted hosts.
- Use SSH instead of Telnet.

Example SSH setup:

```bash
hostname SW1
ip domain-name example.com
username admin secret Str0ngP@ssw0rd
crypto key generate rsa modulus 2048
ip ssh version 2

line vty 0 4
 login local
 transport input ssh
```

AAA basics:

- **AAA** stands for Authentication, Authorization, and Accounting.
- Authentication verifies identity.
- Authorization determines what the user can do.
- Accounting records what the user did.

Simple local AAA example:

```bash
aaa new-model
username netadmin secret Str0ngP@ssw0rd
aaa authentication login default local
aaa authorization exec default local
```

Common threats and mitigations:

- **Phishing**: Train users and use email filtering.
- **DoS/DDoS**: Rate-limit where possible and use upstream protection.
- **MAC flooding**: Use port security.
- **VLAN hopping**: Avoid unused native VLANs and disable DTP where appropriate.
- **Rogue devices**: Use DHCP snooping, port security, and 802.1X where available.

Port security example:

```bash
interface gigabitEthernet0/10
 switchport mode access
 switchport port-security
 switchport port-security maximum 2
 switchport port-security mac-address sticky
 switchport port-security violation restrict
```

Unused port hardening:

```bash
interface range gigabitEthernet0/11 - 24
 shutdown
 switchport mode access
 switchport access vlan 999
```

> If a port enters err-disabled state, check the violation reason before re-enabling it. Repeatedly bouncing the port without fixing the cause only hides the problem.

**Wireless fundamentals**

Wireless networking on the CCNA focuses on standards, bands, AP behavior, and basic security. You should understand how wireless differs from wired Ethernet in terms of interference, shared medium behavior, and roaming.

Key concepts:

- **2.4 GHz** offers longer range but more interference and fewer non-overlapping channels.
- **5 GHz** offers more channels and typically better performance.
- **6 GHz** is newer and supports modern high-throughput designs where available.
- **SSID** identifies the WLAN name broadcast to clients.
- **BSSID** identifies the access point radio interface.

Access point modes:

| Mode | Description |
|---|---|
| Autonomous | AP operates independently |
| Lightweight | AP is managed by a controller |
| Bridge | AP connects separate wired segments wirelessly |
| Repeater | AP extends coverage by relaying traffic |

Wireless security distinctions:

- **WPA2-Personal** uses a pre-shared key.
- **WPA2-Enterprise** uses 802.1X and a RADIUS server.
- **WPA3** improves security over WPA2 where supported.

Practical troubleshooting:

- Check whether the client is on the correct band.
- Verify SSID broadcast and authentication settings.
- Confirm signal strength and channel overlap.
- Look for mismatched security settings between client and AP.

Example controller-style WLAN concepts:

```text
SSID: CorpWiFi
Security: WPA2-Enterprise
Authentication: 802.1X / RADIUS
Band: 5 GHz preferred
```

**Automation and programmability**

Automation is about using software interfaces to configure, monitor, and troubleshoot networks consistently. For CCNA, you should know the purpose of APIs, JSON, YAML, NETCONF, RESTCONF, and YANG.

Definitions:

- **API**: Application Programming Interface, a way for software to communicate.
- **JSON**: JavaScript Object Notation, a common data format for API responses.
- **YANG**: A data modeling language used to describe network configuration and state.
- **NETCONF**: A protocol for network device configuration and retrieval.
- **RESTCONF**: A RESTful interface for accessing YANG-modeled data.

Comparison:

| Technology | Typical use | Notes |
|---|---|---|
| NETCONF | Structured configuration | Uses XML and RPC-style operations |
| RESTCONF | Web-style API access | Uses HTTP methods and often JSON |
| YANG | Data model | Describes what can be configured or read |

Example JSON output:

```json
{
  "interfaces": [
    {
      "name": "GigabitEthernet0/1",
      "admin-status": "up",
      "oper-status": "up"
    }
  ]
}
```

Troubleshooting with API output:

- Check whether the field names match the expected schema.
- Confirm whether the interface is administratively down or operationally down.
- Compare intended state versus reported state.
- Use the model, not just the raw text, to understand what the device is exposing.

A simple controller-based networking idea is that the controller centralizes policy while devices report state and receive configuration through APIs. This reduces manual drift and makes large networks easier to manage.

**Exam-style troubleshooting flow**

When you face a CCNA scenario, work from the symptom outward.

1. Identify the failure domain.
   - Is it one host, one VLAN, one switch, one site, or the whole network?
2. Check local connectivity.
   - Verify link lights, interface status, VLAN membership, and IP settings.
3. Verify Layer 2.
   - Check trunking, STP state, MAC learning, and port security.
4. Verify Layer 3.
   - Check default gateway, routing table, and next-hop reachability.
5. Verify services.
   - Check DHCP, DNS, NAT, and NTP if the symptom suggests them.
6. Verify security and policy.
   - Check ACLs, SSH access, AAA, and firewall-like restrictions.

**Common CCNA “why is this happening?” scenarios**

- A host gets an APIPA address.
  - DHCP may be unavailable, blocked, or misrouted.
- A trunk passes some VLANs but not others.
  - Allowed VLAN lists may not match on both ends.
- A route exists but traffic still fails.
  - An ACL, NAT issue, or incorrect return path may be blocking traffic.
- A switch port shuts down after a device is connected.
  - Port security or BPDU Guard may have triggered.
- Users can browse by IP but not by name.
  - DNS is likely misconfigured or unreachable.
- IPv6 hosts self-assign addresses but cannot reach the router.
  - Router advertisements, link-local reachability, or IPv6 forwarding may be missing.

**What to remember for the exam**

- Use the routing table and longest prefix match before assuming a protocol problem.
- Match VLANs, trunks, and native VLANs on both ends.
- Know the difference between static, OSPF, and EIGRP behavior.
- Treat DHCP, DNS, NAT, and NTP as common service dependencies.
- Harden devices with SSH, AAA basics, port security, and unused-port shutdown.
- Recognize wireless bands, AP modes, and security types.
- Understand how automation tools expose structured data and why YANG matters.

**Final review checklist**

- You can subnet IPv4 networks quickly and accurately.
- You can explain IPv6 addressing, SLAAC, and DHCPv6 at a basic level.
- You can configure and verify VLANs, trunks, and inter-VLAN routing.
- You can compare STP, RSTP, and MST at a high level.
- You can interpret static routes, OSPF, and EIGRP outputs.
- You can configure basic DHCP relay, NAT, NTP, SSH, and port security.
- You can identify common wireless and automation concepts from output or scenario text.
