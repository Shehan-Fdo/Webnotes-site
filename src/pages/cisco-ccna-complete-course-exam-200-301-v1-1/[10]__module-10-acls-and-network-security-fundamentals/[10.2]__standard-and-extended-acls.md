## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**ACL fundamentals**

An <abbr title="Access Control List">ACL</abbr> is an ordered list of permit and deny statements that a router or switch uses to filter traffic. ACLs are evaluated top-down, first match wins, and an implicit deny exists at the end of every ACL.

Use ACLs to control:
- traffic through an interface
- access to VTY lines
- some routing-related filtering features, depending on the feature used

ACLs do not “inspect” traffic deeply in the firewall sense. They match fields in the packet header and then take an action.

**Core behavior you must remember**

- ACL entries are processed in order from top to bottom.
- The first matching ACE (<abbr title="Access Control Entry">ACE</abbr>) stops evaluation.
- If no ACE matches, the packet is denied by the implicit `deny any` at the end.
- A final explicit permit is often required when you want to allow all remaining traffic.

> <mark>Important:</mark> A standard ACL can match only the source address. It cannot match destination IP, protocol, or port.

**ACL types and what they match**

| ACL type | Matches | Typical use |
|---|---|---|
| Standard ACL | Source IP only | Simple source-based filtering |
| Extended ACL | Source IP, destination IP, protocol, ports, and some flags/options | Precise traffic control |

Standard ACLs are best when you care only about *who* is sending traffic. Extended ACLs are best when you care about *who is talking to whom* and *on what service*.

**Numbered and named ACL ranges**

Cisco IOS supports both numbered and named ACLs.

- Standard numbered ACLs commonly use `1-99` and `1300-1999`
- Extended numbered ACLs commonly use `100-199` and `2000-2699`
- Named ACLs can be standard or extended and are easier to read and edit

Named ACLs are usually preferred in real networks because they are clearer and easier to maintain.

Examples:
```bash
ip access-list standard ADMIN_ONLY
ip access-list extended WEB_FILTER
```

**Wildcard masks**

ACLs use wildcard masks, not subnet masks. A wildcard mask tells IOS which bits must match and which bits can vary.

- `0` means the bit must match
- `1` means the bit can vary

Common patterns:
- `0.0.0.0` matches one exact host address
- `0.0.0.255` matches a /24 subnet
- `0.0.15.255` matches a /20-sized block
- `0.255.255.255` matches a /8-sized block

Examples:
```bash
access-list 10 permit 192.168.1.0 0.0.0.255
access-list 10 permit host 192.168.1.10
access-list 10 permit any
```

`host` and `any` are common IOS shortcuts:
- `host 192.168.1.10` means exactly one IP
- `any` means all IPv4 addresses

> <mark>Important:</mark> `any` is a Cisco keyword, not a wildcard mask. It is a shortcut for matching every source or destination address in the ACL statement.

**Non-contiguous wildcard masks**

Wildcard masks do not have to be contiguous like subnet masks. This is useful when you want to match a repeating pattern of addresses.

Example:
```bash
access-list 10 permit 192.168.0.0 0.0.15.255
```

This matches addresses where the third octet can vary across a 16-subnet block. For CCNA, you should recognize that wildcard masks can be non-contiguous, but you usually use contiguous patterns in day-to-day ACL design.

**Standard ACLs**

Standard ACLs are source-based filters. Because they only see the source IP, placement matters a lot.

Use a standard ACL when:
- you want to block or allow traffic from a source network or host
- you do not need to distinguish between different destinations or services
- you want a simple VTY access restriction

Typical placement rule:
- place standard ACLs as close to the destination as practical

Why:
- if you place a standard ACL too close to the source, you may block that source from reaching *all* destinations, not just one target

Example scenario:
- You want to block `192.168.10.0/24` from reaching a server network, but still allow it to reach other networks.
- A standard ACL cannot match the server destination, so it is not the best tool for that requirement.
- An extended ACL is the correct choice if you need to block only traffic to a specific destination.

**Standard ACL examples**

Block one host:
```bash
access-list 10 deny host 192.168.1.50
access-list 10 permit any
```

Block a subnet and allow the rest:
```bash
access-list 10 deny 192.168.1.0 0.0.0.255
access-list 10 permit any
```

Apply it to an interface:
```bash
interface g0/0
 ip access-group 10 in
```

This denies traffic entering `g0/0` from the source addresses in the ACL.

**Standard ACL placement example**

Suppose:
- PC1 is on `192.168.1.0/24`
- Server1 is on `10.10.10.0/24`
- You want to stop PC1 from reaching Server1 only

A standard ACL is not precise enough because it cannot match the destination. If you place it near PC1, it blocks PC1 from reaching everything. If you place it near Server1, it can block PC1 from reaching Server1, but only because all traffic to that destination is isolated at that point in the path. That is a placement effect, not destination matching.

For exam purposes:
- use a **standard ACL** when source-only control is enough
- use an **extended ACL** when you need destination or service precision

**Extended ACLs**

Extended ACLs match more fields:
- source IP
- destination IP
- protocol
- TCP/UDP ports
- some protocol-specific options

Use an extended ACL when you need to:
- allow or deny specific services like HTTP, HTTPS, DNS, SSH, SMTP, FTP, or Telnet
- block traffic from one subnet to one server
- permit one application while denying others between the same hosts

Extended ACLs are usually placed:
- close to the source for efficiency
- or close to the source when you want to stop unwanted traffic early

**Extended ACL examples**

Allow only web traffic from a subnet to a server:
```bash
ip access-list extended WEB_ONLY
 permit tcp 192.168.1.0 0.0.0.255 host 10.10.10.10 eq 80
 permit tcp 192.168.1.0 0.0.0.255 host 10.10.10.10 eq 443
 deny ip any any
```

Block SSH but allow ping:
```bash
ip access-list extended NO_SSH
 deny tcp any host 10.10.10.10 eq 22
 permit icmp any host 10.10.10.10
 permit ip any any
```

Allow DNS to a resolver:
```bash
ip access-list extended DNS_TO_RESOLVER
 permit udp any host 10.10.10.53 eq 53
 permit tcp any host 10.10.10.53 eq 53
```

**Common port and protocol matching**

Extended ACLs often use `eq` to match a destination port.

Examples:
- `eq 80` for HTTP
- `eq 443` for HTTPS
- `eq 22` for SSH
- `eq 23` for Telnet
- `eq 53` for DNS
- `eq 25` for SMTP
- `eq 110` for POP3
- `eq 143` for IMAP
- `eq 161` for SNMP
- `eq 69` for TFTP

You can also match protocol types:
```bash
permit icmp any any
permit tcp any any
permit udp any any
```

For TCP, the `established` keyword is sometimes used to permit return traffic for sessions that have the ACK or RST bit set. It is a legacy feature and not a substitute for stateful inspection, but you may still see it in IOS examples.

Example:
```bash
ip access-list extended RETURN_TCP
 permit tcp any any established
```

**Named ACLs and sequence numbers**

Named ACLs are configured in ACL submode and can be edited more cleanly than numbered ACLs.

Example:
```bash
ip access-list extended WEB_FILTER
 10 permit tcp any host 10.10.10.10 eq 80
 20 permit tcp any host 10.10.10.10 eq 443
 30 deny ip any any
```

You can insert, remove, or reorder entries by sequence number.

To remove one entry:
```bash
ip access-list extended WEB_FILTER
 no 20
```

To add a new entry between existing lines:
```bash
ip access-list extended WEB_FILTER
 15 permit tcp any host 10.10.10.10 eq 8080
```

Numbered ACLs are harder to edit because you often need to delete and recreate the list if you want to change the order significantly.

**Standard vs extended: when to use each**

| Requirement | Best choice | Why |
|---|---|---|
| Block a source subnet from all access | Standard ACL | Source-only match is enough |
| Block one host from reaching one server | Extended ACL | Needs source + destination |
| Allow only HTTP/HTTPS to a server | Extended ACL | Needs port matching |
| Restrict VTY access by admin subnet | Standard ACL | Source IP control is enough |
| Stop a subnet from reaching a specific service | Extended ACL | Needs protocol/port control |

A useful troubleshooting rule:
- if the question mentions only the source, think standard ACL
- if the question mentions destination, protocol, or port, think extended ACL

**Applying ACLs to interfaces**

ACLs are applied with `ip access-group` in either the inbound or outbound direction.

Example:
```bash
interface g0/1
 ip access-group 101 in
```

Direction matters:
- `in` filters packets as they enter the interface
- `out` filters packets as they leave the interface

A practical way to reason about direction:
- inbound ACLs stop traffic before the router processes and forwards it
- outbound ACLs stop traffic after routing decisions are made, just before exit

**End-to-end example**

Topology:
- PC1: `192.168.1.10`
- PC2: `192.168.1.20`
- Web server: `10.10.10.10`
- Router connects both networks

Goal:
- allow PC1 to reach the web server on HTTP and HTTPS
- deny PC2 from reaching the web server
- allow PC2 to reach everything else

Extended ACL solution:
```bash
ip access-list extended WEB_POLICY
 10 permit tcp host 192.168.1.10 host 10.10.10.10 eq 80
 20 permit tcp host 192.168.1.10 host 10.10.10.10 eq 443
 30 deny ip host 192.168.1.20 host 10.10.10.10
 40 permit ip any any
```

Apply it near the source:
```bash
interface g0/0
 ip access-group WEB_POLICY in
```

Why this works:
- PC1 gets only the allowed web services
- PC2 is blocked only from the server
- the final permit prevents the implicit deny from blocking all other traffic

**Implicit deny and explicit permit**

Every ACL ends with an implicit deny. If your intent is “block one thing, allow everything else,” you must add an explicit permit statement.

Examples:
```bash
access-list 10 deny host 192.168.1.50
access-list 10 permit any
```

```bash
ip access-list extended BLOCK_SSH
 deny tcp any any eq 22
 permit ip any any
```

Without the final permit, all unmatched traffic is denied.

**ACLs on VTY lines**

ACLs can restrict who can connect to the virtual terminal lines used for remote management.

Example:
```bash
ip access-list standard VTY_ADMIN
 permit 10.10.30.0 0.0.0.255
 deny any
line vty 0 4
 access-class VTY_ADMIN in
 transport input ssh
```

Important distinction:
- `access-class` controls which source IPs may connect to the VTY lines
- `transport input ssh` controls which remote access protocol is allowed
- the ACL does not “allow SSH” by itself; it only permits the source addresses

**Verification commands**

Use these commands to verify ACL behavior:

```bash
show ip access-lists
show access-lists
show ip interface g0/0
show running-config | section ip access-list
show running-config | include access-group|access-class
show ip interface brief
```

What to look for:
- ACL sequence order
- hit counts on each ACE
- whether the ACL is applied inbound or outbound
- the interface status and IP addressing context

`show ip access-lists` is usually the most useful command because it shows:
- ACL names or numbers
- sequence numbers
- match counters

Example output pattern:
```bash
Standard IP access list 10
    10 deny 192.168.1.0, wildcard bits 0.0.0.255 (5 matches)
    20 permit any (120 matches)
```

Hit counts help you confirm whether traffic is matching the expected line.

**Editing and operational behavior**

Numbered ACLs:
- are less flexible
- are often removed and recreated when changes are needed
- are harder to maintain in production

Named ACLs:
- support sequence numbers
- are easier to insert into
- are easier to document and troubleshoot

If you need to change a numbered ACL, be careful:
- deleting one line may require rebuilding the list
- changing order can affect traffic immediately

**Common exam traps**

- A standard ACL cannot match destination IP or port.
- `permit any` is often needed to avoid the implicit deny.
- `host` is valid and common in Cisco ACL syntax.
- `any` is not a wildcard mask; it is a keyword.
- ACL direction matters as much as ACL content.
- Extended ACLs are the correct tool when the requirement mentions a service or destination.
- VTY `access-class` filters source IPs, not SSH itself.
- Route filtering depends on the feature used; do not assume a packet ACL directly filters routes.

**Quick decision guide**

1. If you need to filter by source only, use a standard ACL.
2. If you need to filter by source and destination, use an extended ACL.
3. If you need to filter by protocol or port, use an extended ACL.
4. If you need to control remote management access, use a standard ACL with `access-class`.
5. If you need to allow everything except one item, remember the explicit permit after the deny.

**Practical lab pattern**

A simple lab workflow helps you avoid mistakes:
1. Identify the traffic you want to match.
2. Decide whether source-only or source-plus-destination/service matching is required.
3. Choose standard or extended ACL accordingly.
4. Write the ACL with the most specific entries first.
5. Add an explicit permit for remaining traffic if needed.
6. Apply the ACL in the correct direction.
7. Verify with `show ip access-lists` and interface commands.
8. Test with `ping`, `telnet`, `ssh`, or application traffic as appropriate.

**Example test commands**

```bash
ping 10.10.10.10
telnet 10.10.10.10 80
ssh -l admin 10.10.10.1
```

For web services, you may also test with a browser or `curl` from a host that supports it.

**Summary**

Standard ACLs are simple source filters. Extended ACLs provide precise control over source, destination, protocol, and ports. For CCNA, the most important skills are choosing the right ACL type, placing it correctly, understanding wildcard masks, remembering the implicit deny, and verifying the ACL with the right show commands.
