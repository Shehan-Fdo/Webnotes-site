## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**SSH-only management overview**

Secure Shell (<abbr title="Secure Shell">SSH</abbr>) is the standard remote management protocol you use to protect device administration traffic. Unlike Telnet, SSH encrypts the session, including usernames, passwords, and commands. In a CCNA lab, “SSH-only access” usually means you configure the device so that remote management is available through SSH and Telnet is disabled on the VTY lines.

This matters because management-plane security is a basic operational control. If you leave Telnet enabled, anyone who can reach the device can potentially read credentials and session content in clear text. SSH also supports stronger authentication workflows and is the expected default for modern Cisco device administration.

**What you must configure for SSH**

Cisco IOS devices need a few prerequisites before they can accept SSH connections. The device must have a hostname, a domain name, local credentials or another authentication source, and RSA keys for the SSH server process.

- **Hostname** is required because the SSH key generation process uses the device identity.
- **Domain name** is required to create the RSA key pair.
- **RSA keys** enable the SSH server.
- **Local user account** provides login credentials when you use local authentication.
- **VTY lines** control remote access methods such as Telnet and SSH.

A typical lab goal is to make the device reachable only through SSH on the VTY lines while keeping console access available for local recovery.

**Core configuration sequence**

You usually configure SSH in this order because each step depends on the previous one.

1. Set the hostname.
2. Set the IP domain name.
3. Create a local user with privilege and secret.
4. Generate RSA keys.
5. Enable SSH version 2.
6. Configure the VTY lines to accept only SSH.
7. Save and verify the configuration.

Example configuration:

```cisco
conf t
hostname R1
ip domain-name example.local
username admin privilege 15 secret Str0ngP@ssw0rd
crypto key generate rsa modulus 2048
ip ssh version 2
line vty 0 4
 login local
 transport input ssh
 exec-timeout 10 0
end
write memory
```

**Why each command matters**

- `hostname R1` gives the device a stable identity for key generation.
- `ip domain-name example.local` completes the fully qualified name used by the crypto subsystem.
- `username admin privilege 15 secret ...` creates a local account and stores the password as a secret hash.
- `crypto key generate rsa modulus 2048` creates the public/private key pair used by SSH.
- `ip ssh version 2` forces SSHv2, which is the secure and current version.
- `login local` tells the VTY lines to use the local username database.
- `transport input ssh` disables Telnet on the VTY lines.
- `exec-timeout 10 0` disconnects idle sessions after 10 minutes.

**VTY line hardening**

The virtual terminal (<abbr title="Virtual Teletype">VTY</abbr>) lines are the remote access lines on Cisco IOS. If you want SSH-only access, you must explicitly restrict the transport protocol. If you do not, Telnet may remain available depending on the device defaults and prior configuration.

Common hardening options include:

- Restricting transport to SSH only.
- Using local authentication instead of line passwords.
- Setting an idle timeout.
- Limiting the number of allowed VTY lines if your design requires it.
- Applying access control lists (<abbr title="Access Control List">ACL</abbr>s) to restrict which source IP addresses can connect.

Example with source restriction:

```cisco
conf t
ip access-list standard MGMT-SOURCES
 permit 192.0.2.10
 permit 192.0.2.11
line vty 0 4
 access-class MGMT-SOURCES in
 transport input ssh
 login local
end
```

This approach is useful when you want only approved management hosts to reach the device, even if SSH is enabled.

**SSH version and security considerations**

SSHv2 is the version you should use. SSHv1 is obsolete and should be avoided because it lacks the same security properties and is not recommended for modern deployments.

> If the device supports only weak RSA key sizes or older software defaults, regenerate keys with a stronger modulus such as `2048` bits when the platform allows it.

Practical security choices:

- Use SSHv2 only.
- Use strong local passwords or, where supported, centralized authentication.
- Use `enable secret` for privileged access.
- Restrict VTY access with ACLs.
- Disable unused management services.

**Verification commands**

After configuration, verify both the SSH service and the line settings. Verification is important because a device can appear configured but still accept Telnet if the VTY lines were not restricted correctly.

Useful commands:

```cisco
show ip ssh
show crypto key mypubkey rsa
show running-config | section line vty
show running-config | include username|ip domain-name|ip ssh|transport input
show users
```

What to look for:

- `show ip ssh` should indicate SSH is enabled and version 2 is active.
- `show crypto key mypubkey rsa` should display the RSA key pair.
- `show running-config | section line vty` should show `transport input ssh`.
- `show users` helps confirm active remote sessions.

Example output indicators:

- SSH enabled: `SSH Enabled - version 2.0`
- VTY transport: `transport input ssh`
- Authentication: `login local`

**Testing access from a client**

Once the device is configured, test from a management host using an SSH client. The exact command depends on the client operating system, but the goal is the same: connect to the device IP address on TCP port 22.

Examples:

```bash
ssh admin@192.0.2.1
```

If you need to specify a nondefault port or a key file on a workstation, the syntax varies by client. For CCNA labs, the standard test is usually a simple username-based SSH login to the device management IP.

If Telnet is disabled correctly, a Telnet attempt should fail:

```bash
telnet 192.0.2.1
```

A failed Telnet test is a good sign when your objective is SSH-only access.

**Troubleshooting common problems**

SSH setup failures usually come from missing prerequisites or line misconfiguration.

- **No RSA keys generated**
  - Check that the hostname and domain name are configured.
  - Regenerate the keys after setting both values.
- **SSH still not working**
  - Confirm `ip ssh version 2`.
  - Verify the device has an IP address and reachable management path.
- **Login fails**
  - Confirm `login local` is configured on the VTY lines.
  - Verify the local username and secret are correct.
- **Telnet still works**
  - Check `transport input ssh` on all relevant VTY lines.
  - Make sure you configured the full VTY range used by the platform, such as `line vty 0 4` or `line vty 0 15`.
- **Connection times out**
  - Check routing, VLAN, interface status, and ACLs.
  - Confirm the management host can reach the device IP.

**Typical lab validation checklist**

Use this checklist to confirm the device is truly SSH-only:

- Hostname is set.
- Domain name is set.
- RSA keys exist.
- SSH version 2 is enabled.
- A local user account exists.
- VTY lines use `login local`.
- VTY lines allow only `ssh`.
- Telnet test fails.
- SSH test succeeds.

**Practical example: full minimal configuration**

This example shows a compact baseline you can adapt in a lab.

```cisco
conf t
hostname SW1
ip domain-name lab.local
username netadmin privilege 15 secret Cisco123!
crypto key generate rsa modulus 2048
ip ssh version 2
line vty 0 15
 login local
 transport input ssh
 exec-timeout 5 0
end
copy running-config startup-config
```

This configuration is common for a small switch or router in a training environment. If the platform uses a different VTY range, adjust the line numbers accordingly.

**Operational notes**

SSH-only access is a management-plane control, not a substitute for broader network security. You still need to consider:

- Physical access to console ports.
- Strong passwords and privilege separation.
- Management VLAN or out-of-band management design.
- ACLs and device hardening.
- Logging and monitoring for administrative access.

In real networks, you often combine SSH with centralized AAA (<abbr title="Authentication, Authorization, and Accounting">AAA</abbr>) services such as RADIUS or TACACS+, but local authentication is the standard CCNA lab approach when external services are not part of the task.

**Key takeaways**

- SSH encrypts remote management and should replace Telnet.
- Cisco IOS SSH requires hostname, domain name, and RSA keys.
- `login local` plus `transport input ssh` is the core SSH-only VTY pattern.
- Verification is essential because Telnet may remain enabled if you miss the VTY configuration.
- Use SSHv2 and restrict access further with ACLs when appropriate.
