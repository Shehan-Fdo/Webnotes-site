## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Security goals and design mindset**

Network security in CCNA focuses on protecting **confidentiality**, **integrity**, and **availability**. You do not need to become a full security engineer, but you do need to recognize common threats, choose appropriate controls, and understand how basic device and network hardening reduces risk.

A practical way to think about security is in layers:

- **Physical security** protects devices from theft, tampering, and unauthorized access.
- **Administrative security** covers policies, procedures, and user access control.
- **Technical security** uses device features such as ACLs, authentication, encryption, and monitoring.

Security is never a single feature. You combine controls so that if one layer fails, another still limits damage.

**Core security principles**

The three classic security objectives are the foundation for exam questions and real-world design decisions.

- **Confidentiality** means only authorized users can view data.
- **Integrity** means data is not altered without detection.
- **Availability** means systems and services remain reachable when needed.

Other useful principles include:

- **Least privilege** gives users and devices only the access they need.
- **Defense in depth** uses multiple controls instead of relying on one.
- **Zero trust** assumes no user or device is trusted by default.
- **Separation of duties** reduces the risk of one person having too much control.

These principles show up in access control, management-plane protection, wireless authentication, and segmentation.

**Common attack types you should recognize**

CCNA expects you to identify threats and understand the basic mitigation idea. You are not usually asked for deep exploit details, but you should know what the attack targets.

- **Password attacks**
  - Brute-force attacks try many combinations.
  - Dictionary attacks use common words and leaked passwords.
  - Credential stuffing reuses stolen username/password pairs.
- **Spoofing attacks**
  - **MAC spoofing** changes a device’s MAC address to impersonate another host.
  - **IP spoofing** forges the source IP address.
  - **ARP spoofing** poisons the Address Resolution Protocol cache to redirect traffic.
- **Layer 2 attacks**
  - **VLAN hopping** attempts to send traffic into another VLAN without authorization.
  - **STP attacks** target the Spanning Tree Protocol to influence root bridge election or create instability.
  - **DHCP starvation** exhausts the DHCP pool by requesting many leases.
  - **DHCP spoofing** introduces a rogue DHCP server.
- **Wireless attacks**
  - Rogue access points provide unauthorized wireless access.
  - Evil twin attacks imitate a legitimate SSID.
  - Weak encryption or shared passwords increase exposure.
- **Service and management attacks**
  - SNMP abuse can expose device information or allow unauthorized changes.
  - Telnet interception exposes credentials in cleartext.
  - Unrestricted management access increases the attack surface.

> <mark>CCNA exam questions often ask for the best mitigation, not the attack mechanics.</mark> Match the threat to the control that reduces it.

**Secure management protocols and why they matter**

Management traffic should be protected because it often carries credentials and privileged commands. Use encrypted and authenticated protocols whenever possible.

- **SSH** encrypts remote CLI access and replaces Telnet.
- **HTTPS** protects web-based management with TLS (Transport Layer Security).
- **SNMPv3** adds authentication and encryption for monitoring and management.
- **SFTP** and **SCP** provide secure file transfer alternatives to FTP/TFTP in many environments.
- **NTP with authentication** is preferred where supported, because time integrity matters for logs and certificates.

Avoid or restrict insecure protocols:

- **Telnet** sends usernames and passwords in cleartext.
- **HTTP** exposes web management traffic.
- **SNMPv1/v2c** use community strings in cleartext and provide weak security.
- **TFTP** has no authentication and no encryption.

A common exam distinction is that **SNMPv1/v2c** are community-string based, while **SNMPv3** supports stronger security features such as authentication and privacy.

**SNMP security basics**

Simple Network Management Protocol (SNMP) is used to monitor and manage network devices. The security model matters.

- **SNMPv1/v2c**
  - Use community strings such as `public` and `private`.
  - Community strings act like shared passwords but are not encrypted.
  - They are easy to intercept if traffic is not otherwise protected.
- **SNMPv3**
  - Supports user-based authentication.
  - Can encrypt management traffic.
  - Is the preferred choice for secure monitoring.

Practical Cisco-style example:

```bash
conf t
snmp-server view SECUREVIEW iso included
snmp-server group NMSGROUP v3 priv read SECUREVIEW
snmp-server user nmsadmin NMSGROUP v3 auth sha AuthPass123 priv aes 128 PrivPass123
snmp-server host 10.10.10.50 version 3 priv nmsadmin
snmp-server enable traps
end
```

Useful points:

- Use **auth** for authentication.
- Use **priv** for privacy, meaning encryption.
- Restrict SNMP access to known management hosts with ACLs.
- Disable older SNMP versions when possible.

**Access control lists as a security control**

Access control lists (ACLs) are one of the most important CCNA security tools. An ACL filters traffic based on criteria such as source IP, destination IP, protocol, and port. You use ACLs to reduce exposure, limit lateral movement, and protect management access.

ACLs help you:

- Restrict who can reach a server or subnet.
- Limit access to device management interfaces.
- Block unwanted protocols or sources.
- Permit only required traffic and deny everything else by default.

Common ACL types:

- **Standard ACLs** match only the source IP address.
- **Extended ACLs** match source and destination IP, protocol, and ports.
- **Named ACLs** improve readability and are easier to maintain.

Where ACLs are applied:

- **Inbound** on an interface, before traffic enters the router or Layer 3 switch.
- **Outbound** on an interface, after routing decisions are made.
- On **VTY lines** to restrict remote management access.
- On **management interfaces** or SVIs to protect device administration.

Example: restrict SSH access to a management subnet.

```bash
conf t
ip access-list standard MGMT-SSH
 permit 10.10.10.0 0.0.0.255
 deny any
exit

line vty 0 4
 access-class MGMT-SSH in
 transport input ssh
 login local
end
```

Example: block a user VLAN from reaching a server except for HTTPS.

```bash
conf t
ip access-list extended USER-TO-SERVER
 permit tcp 10.20.30.0 0.0.0.255 host 10.50.50.10 eq 443
 deny ip any host 10.50.50.10
 permit ip any any
exit

interface g0/1
 ip access-group USER-TO-SERVER in
end
```

Exam tips:

- Use a **standard ACL** when only the source matters.
- Use an **extended ACL** when you need to filter by service or destination.
- Place extended ACLs close to the source when practical.
- Place standard ACLs close to the destination when practical.

**Device hardening and management-plane protection**

Hardening reduces the attack surface by disabling features you do not need and securing the ones you keep.

Common hardening actions:

- Set strong local passwords and enable secret protection.
- Use `service password-encryption` to obscure plaintext passwords in configuration files, while remembering it is not strong encryption.
- Configure `enable secret` instead of `enable password`.
- Use SSH instead of Telnet.
- Restrict VTY access with ACLs.
- Use banners to warn unauthorized users.
- Keep device software updated when operationally appropriate.
- Back up configurations regularly.

Example hardening configuration:

```bash
conf t
hostname R1
ip domain-name example.local
username admin privilege 15 secret Str0ngP@ss!
enable secret Enabl3S3cret!
service password-encryption
banner motd ^CUnauthorized access prohibited.^C

crypto key generate rsa modulus 2048
ip ssh version 2

line vty 0 4
 login local
 transport input ssh
 access-class MGMT-SSH in
end
```

Also disable unused services and interfaces where possible:

- Shut down unused switch ports with `shutdown`.
- Remove unused VLANs from trunks when appropriate.
- Disable unused Layer 2 discovery features such as **CDP** (Cisco Discovery Protocol) and **LLDP** (Link Layer Discovery Protocol) on interfaces where they are not needed.
- Turn off unnecessary services like HTTP server if web management is not used.

Example:

```bash
conf t
interface range g0/2 - 24
 shutdown
 no cdp enable
 no lldp transmit
 no lldp receive
end
```

> <mark>Unused ports and discovery protocols can leak topology information.</mark> Disable them on access ports and other interfaces that do not need them.

**Wireless security fundamentals**

Wireless networks need both encryption and authentication. CCNA commonly compares home-style security with enterprise authentication.

- **WPA2-Personal**
  - Uses a pre-shared key (PSK).
  - All users share the same passphrase.
  - Simpler to deploy, but harder to manage at scale.
- **WPA2-Enterprise**
  - Uses 802.1X authentication with a RADIUS server.
  - Each user authenticates individually.
  - Better for organizations because credentials can be managed per user.

**802.1X** is port-based network access control. In wireless, it helps ensure a client cannot join the network until it authenticates through an authentication server.

Typical enterprise wireless flow:

1. The client associates to the access point.
2. The AP acts as the authenticator.
3. The client provides credentials through EAP (Extensible Authentication Protocol).
4. The AP forwards authentication to a RADIUS server.
5. The RADIUS server approves or denies access.

Practical example of the concept:

```text
Client  <--EAP-->  Access Point  <--RADIUS-->  Authentication Server
```

RADIUS is common for network access authentication because it centralizes user validation and can return authorization attributes. TACACS+ is also used in enterprise networks, especially for device administration, because it separates authentication, authorization, and accounting more cleanly.

**RADIUS and TACACS+**

You should know the difference at a high level.

| Feature | RADIUS | TACACS+ |
|---|---|---|
| Common use | Network access, wireless, VPN | Device administration |
| Transport | UDP | TCP |
| Encryption | Encrypts only parts of the payload | Encrypts the entire payload |
| AAA model | Combines authentication and authorization more tightly | Separates authentication, authorization, accounting |

AAA means:

- **Authentication** verifies identity.
- **Authorization** determines what the user can do.
- **Accounting** records what the user did.

Example AAA configuration concept on Cisco IOS:

```bash
conf t
aaa new-model

radius server RADIUS1
 address ipv4 10.10.10.20 auth-port 1812 acct-port 1813
 key RadiusKey123

aaa authentication login default group radius local
aaa authorization exec default group radius local
aaa accounting exec default start-stop group radius
end
```

For TACACS+:

```bash
conf t
aaa new-model

tacacs server TAC1
 address ipv4 10.10.10.30
 key TacKey123

aaa authentication login default group tacacs+ local
aaa authorization exec default group tacacs+ local
aaa accounting exec default start-stop group tacacs+
end
```

Practical exam takeaway:

- Use **RADIUS** for user access to network services, especially wireless.
- Use **TACACS+** for administrative access to routers and switches.

**802.1X deployment considerations**

802.1X is not just for wireless. It can also be used on wired access ports to control who can connect.

Key components:

- **Supplicant**: the client device requesting access.
- **Authenticator**: the switch port or wireless AP controlling access.
- **Authentication server**: usually RADIUS.

Why use it:

- Prevents unauthorized devices from gaining network access.
- Supports per-user or per-device policy.
- Improves visibility and control in enterprise environments.

Basic switch-side example concept:

```bash
conf t
aaa new-model
radius server RADIUS1
 address ipv4 10.10.10.20 auth-port 1812 acct-port 1813
 key RadiusKey123

aaa authentication dot1x default group radius
dot1x system-auth-control

interface g0/1
 switchport mode access
 authentication port-control auto
 mab
 dot1x pae authenticator
end
```

Notes:

- `mab` means MAC Authentication Bypass, which can be used when a device does not support 802.1X.
- 802.1X is preferred over MAC-based access because MAC addresses are easy to spoof.
- In exam contexts, remember that 802.1X is about controlling access before full network connectivity is granted.

**Layer 2 security controls**

Layer 2 is a common attack surface because switches forward frames based on MAC addresses and VLANs.

Important protections include:

- **Port security**
  - Limits the number of MAC addresses on a port.
  - Can restrict, protect, or shut down the port on violation.
- **DHCP snooping**
  - Blocks rogue DHCP servers.
  - Builds a trusted binding table for later protections.
- **Dynamic ARP Inspection (DAI)**
  - Uses DHCP snooping bindings to validate ARP packets.
  - Helps stop ARP spoofing.
- **IP Source Guard**
  - Restricts traffic based on DHCP snooping bindings.
- **Root guard**
  - Prevents an unexpected switch from becoming STP root.
- **BPDU guard**
  - Shuts down an edge port if it receives Bridge Protocol Data Units (BPDUs).
- **Storm control**
  - Limits broadcast, multicast, or unknown unicast storms.

Example port security:

```bash
conf t
interface g0/10
 switchport mode access
 switchport port-security
 switchport port-security maximum 2
 switchport port-security mac-address sticky
 switchport port-security violation restrict
end
```

Example DHCP snooping and DAI concept:

```bash
conf t
ip dhcp snooping
ip dhcp snooping vlan 10,20

interface g0/1
 ip dhcp snooping trust

ip arp inspection vlan 10,20
end
```

**VLAN hopping and STP-related attacks**

VLAN hopping usually refers to one of two ideas:

- **Switch spoofing** tricks a switch into forming a trunk.
- **Double-tagging** exploits native VLAN handling to send frames into another VLAN.

Mitigations:

- Disable unused ports.
- Set access ports explicitly with `switchport mode access`.
- Disable DTP (Dynamic Trunking Protocol) where not needed.
- Use a non-default native VLAN and avoid using VLAN 1 for user traffic.
- Prune unnecessary VLANs from trunks.

STP-related attacks target topology control.

- A malicious device may try to become root bridge.
- A loop can be introduced to disrupt forwarding.
- BPDUs can be injected on access ports.

Mitigations:

- Enable **BPDU guard** on edge ports.
- Enable **root guard** on ports where downstream switches should never become root.
- Use PortFast only on true edge ports.

**MAC spoofing and ARP spoofing**

MAC spoofing is common because many systems allow the MAC address to be changed in software. It can be used to bypass weak port security or impersonate a trusted device.

Mitigations:

- Use 802.1X instead of MAC-only access.
- Use port security with sticky MAC addresses.
- Monitor for duplicate MAC addresses and unexpected moves.

ARP spoofing is dangerous because ARP has no built-in authentication. Attackers can redirect traffic through themselves.

Mitigations:

- Enable DHCP snooping and DAI.
- Use static ARP entries only where operationally justified.
- Segment sensitive systems and monitor for anomalies.

**Logging, monitoring, and detection**

Security is not only prevention. You also need visibility.

Useful monitoring tools and practices:

- **Syslog** centralizes device logs.
- **NetFlow** or similar flow telemetry helps identify unusual traffic patterns.
- **SNMP traps** notify a management station about events.
- **NTP** keeps timestamps aligned across devices.
- **SPAN** or packet capture can help investigate incidents.

Example syslog and NTP configuration:

```bash
conf t
logging host 10.10.10.40
logging trap informational
service timestamps log datetime msec
ntp server 10.10.10.41
end
```

Good logs help you answer:

- What happened?
- When did it happen?
- Which device was affected?
- Was the event isolated or widespread?

**Incident response basics for CCNA**

A simple incident response flow is enough for exam prep, but you should connect it to network actions.

1. **Identify** the issue using logs, alerts, and user reports.
2. **Contain** the threat by shutting down ports, blocking traffic, or isolating VLANs.
3. **Eradicate** the cause by removing rogue devices, fixing credentials, or patching misconfigurations.
4. **Recover** service by restoring configurations, re-enabling ports, and validating connectivity.
5. **Review** the event to prevent recurrence.

Examples of recovery and verification:

- After a rogue DHCP server incident:
  - Disable the rogue port.
  - Verify DHCP snooping is enabled.
  - Confirm clients receive correct addresses.
  - Check gateway and DNS reachability.
- After an ARP spoofing incident:
  - Enable DAI on the affected VLAN.
  - Verify DHCP snooping bindings exist.
  - Test access to the default gateway and critical servers.
- After a VLAN hopping attempt:
  - Confirm access ports are not trunking.
  - Verify native VLAN and trunk pruning settings.
  - Test inter-VLAN access controls.

**Practical verification commands**

Use show commands to confirm security posture and troubleshoot issues.

```bash
show ip interface brief
show running-config | section line\ vty
show access-lists
show ip access-lists
show port-security interface g0/10
show ip dhcp snooping binding
show ip arp inspection
show authentication sessions
show cdp neighbors
show lldp neighbors
show logging
show snmp user
```

What to look for:

- ACL hit counts and unexpected denies.
- Port-security violations.
- DHCP snooping bindings for valid clients.
- Authentication session state for 802.1X.
- Logs that show unauthorized access attempts or interface changes.

**Exam-focused summary**

For CCNA, focus on matching the threat to the control:

- Use **ACLs** to restrict traffic and management access.
- Use **SSH**, **HTTPS**, and **SNMPv3** for secure management.
- Prefer **RADIUS** for network access and **TACACS+** for device administration.
- Use **802.1X** for enterprise authentication, especially with WPA2-Enterprise.
- Use **port security**, **DHCP snooping**, **DAI**, **BPDU guard**, and **root guard** to protect Layer 2.
- Disable unused services, ports, **CDP**, and **LLDP** where they are not needed.
- Recognize threats such as VLAN hopping, MAC spoofing, ARP spoofing, rogue DHCP, and STP manipulation.

Security questions on the exam often test whether you understand the control plane, management plane, and data plane separately. If you can explain what is being protected, how the attack works at a high level, and which control best reduces risk, you are in strong shape.
