## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Port security overview**

Port security limits which source MAC addresses can use a switch port. You use it mainly on **access ports** to reduce the risk of unauthorized devices being plugged into the LAN. It is a Layer 2 control, so it does not replace authentication, but it is a practical first line of defense on edge ports.

Port security is most useful where you expect a small, predictable number of devices:
- User access ports
- Printer ports
- Lab or kiosk ports
- Some voice/data edge designs with an IP phone and a PC behind it

It is generally not appropriate for most **trunk** ports because trunks carry multiple VLANs and often connect to infrastructure devices that may legitimately learn many MAC addresses.

**Port security behavior and MAC learning**

When port security is enabled, the switch compares incoming source MAC addresses against the secure MAC address list for that interface. If a frame arrives from an unauthorized MAC, the switch takes the configured violation action.

A key CCNA detail is the **maximum** setting:
- If you explicitly set `switchport port-security maximum <n>`, the port can learn up to that number of secure MAC addresses.
- If you do not set a maximum, the default maximum is **1**.
- On a fresh port, the first learned MAC address becomes the secure MAC address, and that single learned address counts as the port’s secure entry.

Common secure MAC address types:
- **Static secure MAC**: manually configured with `switchport port-security mac-address <mac>`
- **Sticky secure MAC**: dynamically learned and written into the running configuration with `switchport port-security mac-address sticky`

Sticky learning is convenient for access ports because it lets the switch learn the first valid device automatically and preserve it across reloads if you save the configuration.

**Basic configuration example**

You typically configure port security on an access interface after setting the port mode.

```cisco
conf t
interface gigabitEthernet1/0/10
 switchport mode access
 switchport access vlan 10
 switchport port-security
 switchport port-security maximum 2
 switchport port-security mac-address sticky
 switchport port-security violation restrict
end
write memory
```

This example:
- Forces the interface into access mode
- Places it in VLAN 10
- Allows up to two secure MAC addresses
- Learns MAC addresses dynamically and stores them as sticky entries
- Uses `restrict` so violations are logged and counted without shutting the port down

**Violation modes**

Port security supports three common violation actions:

| Mode | Behavior | Typical use |
|---|---|---|
| `protect` | Drops frames from unauthorized MAC addresses silently | Low-noise environments where logging is not needed |
| `restrict` | Drops frames, increments violation counter, and generates logs/SNMP traps | Common CCNA-friendly choice |
| `shutdown` | Puts the port into err-disabled state | Strongest enforcement, default behavior on many platforms |

`shutdown` is the most disruptive. If an unauthorized device appears, the interface goes down until you manually recover it or use errdisable recovery.

**Aging and secure MAC expiration**

Port security can age out secure MAC addresses so the switch can relearn devices over time. This is important in environments where devices move, are replaced, or reconnect after long idle periods.

There are two aging modes:

- **Absolute aging**
  - The secure MAC entry expires after the configured time, regardless of traffic.
  - Use this when you want a hard lifetime for learned addresses.
- **Inactivity aging**
  - The secure MAC entry expires only after it has been inactive for the configured time.
  - Use this when you want active devices to remain learned but stale entries to disappear.

Example configuration:

```cisco
conf t
interface gigabitEthernet1/0/10
 switchport port-security aging time 60
 switchport port-security aging type inactivity
end
```

In this example, a secure MAC entry is removed after 60 minutes of inactivity. If you change the type to `absolute`, the entry expires after 60 minutes even if traffic continues.

Aging is useful when:
- Users swap devices on the same port
- You want sticky entries to refresh naturally
- You need to avoid stale secure MAC addresses after maintenance

> Be careful with aging on ports that connect to phones or shared devices. If the aging timer is too aggressive, legitimate devices may be removed and trigger violations.

**Voice VLAN and IP phone edge cases**

A common access-layer design uses an IP phone with a PC connected through the phone’s built-in switch. In that case, the access port may see more than one MAC address:
- The phone’s MAC address
- The PC’s MAC address

If you use port security on such a port, you must account for both devices:
- Increase the maximum to at least 2
- Consider sticky learning so the switch learns the phone and PC automatically
- Verify the voice VLAN design before enforcing strict limits

Example:

```cisco
interface gigabitEthernet1/0/20
 switchport mode access
 switchport access vlan 20
 switchport voice vlan 30
 switchport port-security
 switchport port-security maximum 2
 switchport port-security mac-address sticky
```

If the maximum is left at the default of 1, the second MAC from the PC or phone can trigger a violation depending on which device is learned first.

**Verification commands**

Use show commands to confirm the port-security state and learned addresses.

```cisco
show port-security interface gigabitEthernet1/0/10
show port-security address
show running-config interface gigabitEthernet1/0/10
show interfaces status
show mac address-table interface gigabitEthernet1/0/10
```

What to look for:
- Security status enabled or disabled
- Maximum MAC count
- Current secure MAC count
- Violation count
- Violation mode
- Aging settings
- Sticky MAC entries in the running configuration

If you are troubleshooting sticky learning, confirm that the interface is up, in access mode, and receiving traffic from the expected device.

**Clearing and resetting port security**

You often need to clear learned state during troubleshooting or when replacing a device.

Common reset actions:

```cisco
clear port-security sticky interface gigabitEthernet1/0/10
clear port-security all
```

You can also remove secure MAC entries from the interface configuration:

```cisco
conf t
interface gigabitEthernet1/0/10
 no switchport port-security mac-address sticky
 no switchport port-security mac-address 0011.2233.4455
end
```

If the port is err-disabled because of a violation, you may need to recover it:

```cisco
conf t
interface gigabitEthernet1/0/10
 shutdown
 no shutdown
end
```

For repeated violations, consider enabling **errdisable recovery** so the switch can automatically bring the port back after a timer.

```cisco
conf t
errdisable recovery cause psecure-violation
errdisable recovery interval 300
end
```

This tells the switch to attempt recovery after 300 seconds for port-security violations.

**Basic switch hardening baseline**

Port security is only one part of a broader hardening baseline. A secure access switch also reduces unnecessary services and limits management exposure.

Common hardening steps include:
- Disable unused switch ports and place them in an unused VLAN
- Disable **CDP** (Cisco Discovery Protocol) and **LLDP** (Link Layer Discovery Protocol) on untrusted ports when discovery is not needed
- Use SSH instead of Telnet for remote management
- Restrict management access with ACLs (Access Control Lists)
- Use strong passwords and encrypted password storage where supported
- Set an enable secret
- Configure a login banner
- Disable unused services where appropriate for the platform and lab scope

Example of disabling unused ports:

```cisco
conf t
interface range gigabitEthernet1/0/21-24
 shutdown
 switchport mode access
 switchport access vlan 999
end
```

Example of disabling discovery protocols on an access port:

```cisco
conf t
interface gigabitEthernet1/0/10
 no cdp enable
 no lldp transmit
 no lldp receive
end
```

If the port connects to an untrusted endpoint, disabling discovery reduces information leakage about the switch and neighbors.

**SSH management setup**

SSH is the standard secure remote management protocol for CCNA-level switch administration. To enable SSH, the switch needs a hostname, a domain name, RSA keys, local credentials, and VTY line configuration.

Typical setup:

```cisco
conf t
hostname SW1
ip domain-name example.local
username admin privilege 15 secret Str0ngP@ssw0rd
enable secret An0therStr0ngSecret
crypto key generate rsa modulus 2048
ip ssh version 2
line vty 0 15
 login local
 transport input ssh
end
write memory
```

Why each step matters:
- `hostname` and `ip domain-name` are required before RSA key generation
- `username` and `login local` provide local authentication
- `crypto key generate rsa` creates the key pair SSH uses
- `ip ssh version 2` enables the more secure SSH version
- `transport input ssh` blocks Telnet on the VTY lines

**SSH verification and troubleshooting**

After configuration, verify that SSH is active and the VTY lines are correct.

```cisco
show ip ssh
show crypto key mypubkey rsa
show running-config | section line vty
show users
```

Common SSH problems:
- Missing `ip domain-name`, which prevents RSA key generation
- No RSA keys generated, so SSH cannot start
- VTY lines still allow Telnet because `transport input` is misconfigured
- `login local` missing, so authentication fails
- Wrong username or secret
- Management IP or default gateway not configured correctly for remote access across subnets

If SSH fails, check the management SVI and reachability:

```cisco
show ip interface brief
show running-config interface vlan 1
show ip route
ping <management-host>
```

**Management-plane access control**

Hardening also means limiting who can reach the switch management plane. On Layer 2 switches, this usually means:
- Assigning a management IP address to an SVI
- Setting a default gateway for remote management
- Applying an ACL to VTY lines or the management interface where supported
- Allowing only trusted admin subnets

Example with a management SVI:

```cisco
conf t
interface vlan 99
 ip address 192.168.99.2 255.255.255.0
 no shutdown
exit
ip default-gateway 192.168.99.1
end
```

Example VTY access control:

```cisco
conf t
ip access-list standard MGMT-ONLY
 permit 192.168.99.0 0.0.0.255
 deny any
line vty 0 15
 access-class MGMT-ONLY in
 transport input ssh
end
```

This ensures only the management subnet can open remote sessions.

**Practical hardening workflow**

1. Disable and repurpose unused ports.
2. Configure access ports with the correct VLANs.
3. Enable port security on user-facing ports.
4. Use sticky MAC learning where appropriate.
5. Set a sensible maximum for each port, especially with phones or shared devices.
6. Choose a violation mode that matches the operational impact you can tolerate.
7. Disable CDP and LLDP on untrusted ports.
8. Enable SSH and block Telnet.
9. Restrict management access with ACLs.
10. Verify and save the configuration.

**Troubleshooting port-security violations**

When a port goes err-disabled or starts dropping traffic, check:
- Whether the learned MAC changed because a device was replaced
- Whether a phone and PC share the port
- Whether the maximum is too low
- Whether sticky entries are stale
- Whether aging removed an expected secure MAC

Useful commands:

```cisco
show port-security interface gigabitEthernet1/0/10
show port-security address
show interfaces status err-disabled
show logging
```

If the port is err-disabled, the log often shows the cause. After correcting the issue, clear the sticky entry or bounce the interface, then let the correct device relearn.

**Key exam takeaways**

- Port security is typically used on **access ports**, not trunks.
- The default maximum secure MAC count is **1** if you do not configure it.
- Sticky MAC addresses are learned dynamically and can be saved in the configuration.
- `protect`, `restrict`, and `shutdown` define how the switch responds to violations.
- Aging can be **absolute** or **inactivity** based.
- IP phones can require more than one secure MAC on a single access port.
- SSH requires hostname, domain name, RSA keys, and correct VTY settings.
- Basic hardening includes disabling unused ports, limiting management access, and turning off discovery protocols on untrusted interfaces.
