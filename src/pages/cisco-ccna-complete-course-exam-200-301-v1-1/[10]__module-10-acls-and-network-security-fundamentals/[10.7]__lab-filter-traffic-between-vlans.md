## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Lab goal and traffic model**

Filtering traffic between VLANs means you allow some hosts in one VLAN to reach another VLAN while blocking other flows. In a CCNA lab, this is usually done with an <abbr title="Access Control List">ACL</abbr> applied on a router-on-a-stick subinterface or on a multilayer switch routed interface. The key idea is that VLANs separate Layer 2 broadcast domains, while ACLs control Layer 3 forwarding between those domains.

A typical lab topology includes:
- Two or more VLANs with separate IP subnets
- A Layer 3 device performing inter-VLAN routing
- End hosts that need selective access to servers, printers, or management networks
- An ACL that permits required traffic and denies unwanted traffic

**Core concepts**

Inter-VLAN routing happens after a frame is switched to the default gateway for its VLAN. Once the packet reaches the router or Layer 3 switch, the device makes a forwarding decision based on the destination IP address and any applied ACLs. This is why ACL placement matters: you filter traffic where it enters or leaves the routed interface, not on the access switch ports unless you are using special features such as port ACLs.

Important terms:
- **VLAN**: A logical Layer 2 segment that isolates broadcast traffic
- **SVI**: Switched Virtual Interface, a Layer 3 interface tied to a VLAN on a multilayer switch
- **Router-on-a-stick**: A router using 802.1Q subinterfaces to route between VLANs over one trunk
- **ACL**: A rule set that permits or denies traffic based on Layer 3 and Layer 4 fields
- **Implicit deny**: The hidden `deny any` at the end of every ACL

**When to use ACLs between VLANs**

Use ACLs when you need to:
- Block user VLANs from reaching sensitive server VLANs
- Allow only specific services such as DNS, HTTP, or SSH
- Restrict management access to network devices
- Prevent one department VLAN from accessing another except for approved applications

ACLs are best when the policy is simple and based on source, destination, and protocol. If you need identity-based or application-aware policy, that is beyond CCNA scope and usually requires more advanced features.

**Lab design example**

A common lab uses three VLANs:
- VLAN 10: Users `192.168.10.0/24`
- VLAN 20: Users `192.168.20.0/24`
- VLAN 30: Servers `192.168.30.0/24`

A simple policy might be:
- VLAN 10 can reach VLAN 30 only for HTTP and HTTPS
- VLAN 20 can reach VLAN 30 only for DNS and SSH
- VLAN 10 and VLAN 20 cannot reach each other directly

This kind of policy is easier to understand if you think in terms of source subnet, destination subnet, and permitted service ports.

**Router-on-a-stick implementation**

On a router, you create one subinterface per VLAN and assign 802.1Q encapsulation plus an IP address. The trunk link carries all VLANs between the switch and router.

Example configuration:
```bash
interface g0/0
 no shutdown

interface g0/0.10
 encapsulation dot1Q 10
 ip address 192.168.10.1 255.255.255.0

interface g0/0.20
 encapsulation dot1Q 20
 ip address 192.168.20.1 255.255.255.0

interface g0/0.30
 encapsulation dot1Q 30
 ip address 192.168.30.1 255.255.255.0
```

On the switch, the uplink must be a trunk:
```bash
interface g0/1
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30
```

**ACL placement strategy**

For inter-VLAN filtering, extended ACLs are usually placed close to the source when possible. That reduces unnecessary traffic and makes the policy easier to reason about. In a lab, you often apply the ACL inbound on the source VLAN subinterface.

Example:
- Apply an ACL inbound on `g0/0.10` to control traffic sourced from VLAN 10
- Apply a different ACL inbound on `g0/0.20` to control traffic sourced from VLAN 20

This approach is common because extended ACLs can match source, destination, protocol, and ports.

**Extended ACL example**

Suppose VLAN 10 should only access the web server in VLAN 30 using HTTP and HTTPS. You can build an ACL like this:

```bash
ip access-list extended VLAN10_TO_SERVERS
 permit tcp 192.168.10.0 0.0.0.255 192.168.30.0 0.0.0.255 eq 80
 permit tcp 192.168.10.0 0.0.0.255 192.168.30.0 0.0.0.255 eq 443
 deny ip 192.168.10.0 0.0.0.255 192.168.30.0 0.0.0.255
 permit ip any any
```

Apply it inbound on the VLAN 10 subinterface:
```bash
interface g0/0.10
 ip access-group VLAN10_TO_SERVERS in
```

This ACL:
- Permits HTTP and HTTPS from VLAN 10 to VLAN 30
- Denies all other IP traffic from VLAN 10 to VLAN 30
- Permits all other traffic so VLAN 10 can still reach other allowed destinations

> Be careful with the final `permit ip any any`. Without it, the implicit deny blocks all traffic not explicitly matched, which may break unrelated connectivity.

**Why wildcard masks matter**

Cisco ACLs use wildcard masks, not subnet masks. A wildcard mask of `0.0.0.255` means “match the first three octets exactly, ignore the last octet.” For a /24 network, that is the correct wildcard.

Examples:
- `192.168.10.0 0.0.0.255` matches `192.168.10.0/24`
- `192.168.30.10 0.0.0.0` matches one host only
- `any` matches all addresses

**Common filtering patterns**

You may need to allow specific services while blocking everything else. Typical examples:
- Web access: TCP ports 80 and 443
- DNS: UDP and sometimes TCP port 53
- SSH: TCP port 22
- ICMP: useful for testing, but often restricted in policy

Example allowing DNS and SSH from VLAN 20 to a server VLAN:
```bash
ip access-list extended VLAN20_TO_SERVERS
 permit udp 192.168.20.0 0.0.0.255 192.168.30.0 0.0.0.255 eq 53
 permit tcp 192.168.20.0 0.0.0.255 192.168.30.0 0.0.0.255 eq 22
 deny ip 192.168.20.0 0.0.0.255 192.168.30.0 0.0.0.255
 permit ip any any
```

**Verification workflow**

Use a structured verification process so you can isolate whether the problem is VLAN, trunking, routing, or ACL logic.

1. Confirm VLAN membership on the switch.
   ```bash
   show vlan brief
   ```
2. Confirm the trunk is up and carrying the expected VLANs.
   ```bash
   show interfaces trunk
   ```
3. Confirm the router subinterfaces or SVI addresses are correct.
   ```bash
   show ip interface brief
   ```
4. Confirm the ACL is applied where you expect.
   ```bash
   show ip interface g0/0.10
   show access-lists
   ```
5. Test connectivity from a host using `ping`, `tracert`, or `ssh` as appropriate.
6. Check ACL hit counts to see whether packets match the intended entries.
   ```bash
   show access-lists VLAN10_TO_SERVERS
   ```

**Troubleshooting checklist**

If traffic is blocked unexpectedly, check these items:
- The ACL is applied in the correct direction, usually `in` on the source interface
- The ACL includes permits for return traffic if the policy requires it
- The wildcard masks match the intended subnet
- The VLAN trunk allows the VLANs in use
- The host default gateway points to the correct router or SVI address
- The ACL does not accidentally block ICMP needed for testing

If traffic is allowed when it should be blocked, check:
- The deny statement appears before a broader permit
- The ACL is applied to the correct interface
- Another ACL is not overriding the intended policy
- You are testing the correct source and destination addresses

**SVI-based filtering on a multilayer switch**

On a multilayer switch, you can route between VLANs using SVIs and apply ACLs to those interfaces. The configuration is similar in concept to router-on-a-stick, but the routing happens on the switch itself.

Example:
```bash
ip routing

interface vlan 10
 ip address 192.168.10.1 255.255.255.0
 ip access-group VLAN10_TO_SERVERS in

interface vlan 20
 ip address 192.168.20.1 255.255.255.0
 ip access-group VLAN20_TO_SERVERS in
```

This design is common in campus networks because it reduces dependence on an external router and improves performance.

**Practical lab example**

A simple lab objective might be:
- Host A in VLAN 10 can browse a web server in VLAN 30
- Host B in VLAN 20 can SSH to the same server
- Hosts in VLAN 10 and VLAN 20 cannot communicate with each other

A possible policy implementation:
```bash
ip access-list extended VLAN10_FILTER
 permit tcp 192.168.10.0 0.0.0.255 host 192.168.30.10 eq 80
 permit tcp 192.168.10.0 0.0.0.255 host 192.168.30.10 eq 443
 deny ip 192.168.10.0 0.0.0.255 192.168.20.0 0.0.0.255
 deny ip 192.168.10.0 0.0.0.255 192.168.30.0 0.0.0.255
 permit ip any any
```

This example uses a host-specific destination for the server and then blocks the rest of the server subnet. In a real lab, you should align the ACL with the exact policy statement so you do not overblock legitimate traffic.

**Best practices for exam and lab work**

- Write the policy first in plain language before touching the CLI
- Decide whether the ACL should be applied inbound or outbound
- Use extended ACLs for inter-VLAN filtering because they can match ports
- Keep one policy per source VLAN when possible
- Test incrementally after each change
- Remember that ACLs are processed top-down, first match wins

**Key takeaways**

Inter-VLAN filtering is about controlling routed traffic between VLANs, not blocking Layer 2 switching itself. The most important skills are choosing the correct ACL type, placing it on the correct interface and direction, and writing permit and deny statements that match the intended policy. In CCNA labs, success usually comes from careful planning, precise wildcard masks, and methodical verification.
