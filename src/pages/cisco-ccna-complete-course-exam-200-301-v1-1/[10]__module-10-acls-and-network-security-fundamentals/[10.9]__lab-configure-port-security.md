## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Port security overview**

Port security limits which source MAC addresses can use a switch port. In CCNA labs, you usually apply it to an **access port** that connects to a single end device such as a PC, printer, or IP phone access port. The goal is to reduce unauthorized access and to control what happens when an unexpected device appears.

Port security is most commonly used on access interfaces because the expected device count is simple and predictable. On some Cisco platforms, port security can also be used in other scenarios, but for exam and lab practice you should treat access ports as the primary use case.

- It helps prevent casual device swapping on edge ports.
- It can learn MAC addresses automatically or use manually configured entries.
- It can shut down, restrict, or protect a port when a violation occurs.

**How port security works**

A secure MAC address is a MAC address that the switch allows on the interface. The switch compares the source MAC address of incoming frames against the secure list. If the address is allowed, traffic passes. If not, the port reacts according to the configured violation mode.

Common secure MAC types include:

- **Static secure MAC**: You configure the MAC address manually.
- **Sticky secure MAC**: The switch learns the MAC address and can write it into the running configuration.
- **Dynamic secure MAC**: The switch learns the MAC address for the current session only.

A useful mental model is: port security defines a whitelist, and the violation mode defines the response when a device is not on that list.

**Prerequisites and interface checks**

Before enabling port security, confirm that the interface is a true access port and is not being used for a trunk or an EtherChannel member. The exact platform behavior can vary, but the CCNA expectation is that you configure port security on access ports.

Check the interface state first:

```bash
show interfaces status
show running-config interface fa0/1
show interfaces fa0/1 switchport
```

Look for these conditions:

- The port is in **access** mode.
- The port is assigned to the correct VLAN.
- The port is not part of a trunk.
- The port is not bundled into an EtherChannel.

If the interface is not an access port, fix that first. A typical access-port setup looks like this:

```bash
configure terminal
interface fa0/1
 switchport mode access
 switchport access vlan 10
 spanning-tree portfast
```

**Basic port security configuration**

A common lab uses one access port and one PC. Start with a clean access interface, then enable port security and define the allowed MAC behavior.

Example configuration:

```bash
configure terminal
interface fa0/1
 switchport mode access
 switchport access vlan 10
 switchport port-security
 switchport port-security maximum 1
 switchport port-security violation shutdown
 switchport port-security mac-address sticky
end
```

What each command does:

- `switchport port-security` enables the feature.
- `switchport port-security maximum 1` limits the number of secure MAC addresses.
- `switchport port-security violation shutdown` places the port into an error-disabled state on violation.
- `switchport port-security mac-address sticky` learns the first MAC address and stores it as sticky.

If you want to use a manually assigned secure MAC instead of sticky learning, configure the MAC explicitly:

```bash
configure terminal
interface fa0/1
 switchport mode access
 switchport access vlan 10
 switchport port-security
 switchport port-security maximum 1
 switchport port-security mac-address 0011.2233.4455
 switchport port-security violation restrict
end
```

This is useful when you already know the device MAC and want deterministic behavior.

**Violation modes**

Violation mode determines what the switch does when an unauthorized MAC address appears.

| Mode | Behavior | Typical use |
|---|---|---|
| `protect` | Drops unauthorized frames silently | Low-noise environments |
| `restrict` | Drops frames and increments violation counters | Monitoring and troubleshooting |
| `shutdown` | Puts the interface into err-disabled state | Strong enforcement, common lab default |

Key points:

- **Protect** does not generate as much visible feedback, so it can be harder to notice.
- **Restrict** is often easier to troubleshoot because counters increase.
- **Shutdown** is the most visible and common in labs because it clearly demonstrates enforcement.

> The exact interface status text can vary by IOS version and platform. Focus on the err-disabled behavior rather than memorizing one display string.

**Sticky MAC learning**

Sticky learning is a common lab topic because it combines automation with persistence in the running configuration. When sticky is enabled, the switch learns the first source MAC address seen on the port and treats it as secure.

Typical workflow:

1. Configure port security with sticky enabled.
2. Connect the end device and generate traffic.
3. Verify that the MAC address appears as sticky.
4. Save the configuration if you want the learned MAC to survive a reload.

Example:

```bash
configure terminal
interface fa0/1
 switchport port-security mac-address sticky
end
show running-config interface fa0/1
```

Important behavior:

- Sticky MAC addresses are written to the **running configuration**.
- If you save the configuration with `copy running-config startup-config`, the sticky entry can persist across reloads.
- Dynamic secure MAC addresses are not saved in the configuration and are lost on reload.

**Verification commands**

Use multiple commands to verify both the interface and the secure MAC table. `show port-security interface` is helpful, but it does not always show the full picture by itself.

Recommended verification set:

```bash
show port-security interface fa0/1
show port-security address
show mac address-table secure
show running-config interface fa0/1
show interfaces status
```

What to look for:

- Port security is enabled on the interface.
- The maximum MAC count is correct.
- The violation mode matches your design.
- The current secure MAC address list contains the expected device.
- Sticky entries appear in the running configuration if sticky learning is used.

Example output interpretation:

- If the secure MAC count is `1` and the learned address matches the PC, the port is working as intended.
- If the violation counter increases, an unauthorized device likely attempted access.
- If the interface is err-disabled, a shutdown violation probably occurred.

**Generating and observing a violation**

A good lab test is to connect a second device or change the source MAC on the same port. This forces a violation and shows how the switch reacts.

Suggested test sequence:

1. Configure the port for one secure MAC.
2. Connect the first PC and verify learning.
3. Disconnect the first PC or keep it connected.
4. Connect a second device or a device with a different MAC.
5. Generate traffic such as `ping` or ARP activity.
6. Check the violation counters and interface state.

Useful checks:

```bash
show port-security interface fa0/1
show interfaces status err-disabled
show logging | include PORT_SECURITY|err-disable
```

If the violation mode is `shutdown`, the interface may go err-disabled. If it is `restrict`, the port stays up but drops unauthorized frames and increments counters.

**Err-disabled recovery**

When a port is shut down by port security, you need to recover it before it can pass traffic again. The common workflow is to identify the cause, clear the condition, and re-enable the interface.

Typical recovery steps:

1. Confirm the port is err-disabled.
2. Remove the offending device or correct the MAC configuration.
3. Clear the port state if needed.
4. Bounce the interface.

Example:

```bash
show interfaces status err-disabled
configure terminal
interface fa0/1
 shutdown
 no shutdown
end
```

If the port remains blocked, verify that the secure MAC list still matches the intended device. On some platforms, you may also need to clear learned sticky entries or the MAC address table before retesting.

**Clearing learned entries**

Clearing port-security data depends on what you learned and how it was stored.

Common approaches:

- Remove sticky entries from the running configuration if you want to relearn them.
- Clear secure MAC entries from the table if the platform supports it.
- Clear the MAC address table entry for the interface when troubleshooting forwarding behavior.

Examples:

```bash
clear port-security sticky interface fa0/1
clear mac address-table dynamic interface fa0/1
```

If your platform supports a broader clear command, use it carefully and verify the syntax with `?` in the CLI. The exact command set can vary by IOS release and switch model.

> Do not assume one clear command works identically on every Cisco platform. Always confirm the available syntax on the device you are using.

**Removing port security**

When you are done with the lab, remove the feature cleanly and restore the interface to its prior state.

A safe removal workflow is:

1. Remove the port-security feature from the interface.
2. Remove any sticky or manually configured secure MAC entries from the running configuration if present.
3. Restore the desired access or trunk settings.
4. Verify the interface is forwarding normally.

Example:

```bash
configure terminal
interface fa0/1
 no switchport port-security
 no switchport port-security mac-address sticky
 no switchport port-security mac-address 0011.2233.4455
end
```

Depending on the platform, some specific subcommands may need to be removed by editing the running configuration or by reconfiguring the interface. The most reliable first step is usually `no switchport port-security`.

**Lab procedure**

Use this sequence for a hands-on practice lab.

1. Build the topology.
   - Connect one PC to `Fa0/1`.
   - Optionally connect a second PC for violation testing.
   - Ensure the switch port is an access port in the correct VLAN.

2. Configure the interface.
   ```bash
   configure terminal
   interface fa0/1
    switchport mode access
    switchport access vlan 10
    spanning-tree portfast
   ```

3. Enable port security.
   ```bash
   interface fa0/1
    switchport port-security
    switchport port-security maximum 1
    switchport port-security violation shutdown
    switchport port-security mac-address sticky
   end
   ```

4. Generate traffic from the first PC.
   - Use `ping` to the default gateway or another reachable host.
   - Confirm the MAC address is learned.

5. Verify the secure state.
   ```bash
   show port-security interface fa0/1
   show port-security address
   show mac address-table secure
   show running-config interface fa0/1
   ```

6. Trigger a violation.
   - Connect a second device or change the source MAC.
   - Send traffic again.

7. Check the result.
   ```bash
   show interfaces status err-disabled
   show logging | include PORT_SECURITY
   show port-security interface fa0/1
   ```

8. Recover the port.
   ```bash
   configure terminal
   interface fa0/1
    shutdown
    no shutdown
   end
   ```

9. Clean up.
   - Remove sticky entries if needed.
   - Remove port security.
   - Restore the interface to its original state.

**Troubleshooting tips**

If port security does not behave as expected, check the basics first.

- Confirm the interface is not a trunk.
- Confirm the interface is not part of an EtherChannel.
- Confirm the VLAN is correct.
- Confirm the device MAC address is the one you expect.
- Confirm the violation mode matches the test you are trying to perform.

Common mistakes:

- Enabling sticky before the correct device is connected.
- Forgetting that the first learned MAC becomes the allowed MAC.
- Testing with a device that changes its MAC address.
- Assuming the port will recover automatically after a shutdown violation.

**Exam focus**

For CCNA-level understanding, know these points well:

- Port security is primarily an **access-port** feature.
- You can allow MAC addresses by manual entry or sticky learning.
- Violation modes are `protect`, `restrict`, and `shutdown`.
- `shutdown` can place the port into err-disabled state.
- Verification requires checking both the interface and the secure MAC table.
- Sticky entries persist in the running configuration and can be saved to startup configuration.
