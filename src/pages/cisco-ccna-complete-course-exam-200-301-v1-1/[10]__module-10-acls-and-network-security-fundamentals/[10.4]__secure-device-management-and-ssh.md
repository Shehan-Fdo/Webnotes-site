## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Why secure device management matters**

Remote management is one of the first places you harden a Cisco device. Telnet sends credentials in cleartext, while SSH (Secure Shell) encrypts the management session and is the expected standard for CCNA-level secure administration. In practice, you use SSH to reach the device’s CLI over the network, then control who can connect, how they authenticate, and which VTY (virtual terminal) lines accept remote access.

A secure management design usually includes:

- A reachable management IP address on the device.
- SSH enabled with RSA keys and a domain name.
- VTY lines restricted to SSH.
- Local user accounts or AAA (Authentication, Authorization, and Accounting) for login control.
- Basic hardening such as strong passwords, `enable secret`, and limited login retries.

> SSH protects the management session, but it does not automatically make the device “secure.” You still need proper authentication, authorization, and line restrictions.

**Core SSH concepts**

SSH is a client/server protocol used for encrypted remote CLI access. On Cisco IOS, the device acts as the SSH server when you connect to it from a workstation, and it can also act as an SSH client when it initiates outbound SSH sessions to other systems.

Key terms:

- **VTY lines**: Logical terminal lines used for remote access sessions.
- **RSA keys**: Public/private key pair required for SSH on many IOS platforms.
- **Domain name**: Needed when generating RSA keys on classic IOS.
- **Privilege level**: Determines what commands you can run after login.
- **AAA**: Centralized framework for authentication, authorization, and accounting.

SSH uses encryption to protect:
- Username and password exchange
- CLI commands and output
- Session integrity

SSH does not replace:
- Strong account management
- Device reachability
- Access control lists (ACLs)
- Proper privilege configuration

**Prerequisites for SSH on Cisco IOS**

Before SSH works, the device must be reachable and properly identified.

Typical requirements:

- Configure an IP address on an interface that can be reached from the management host.
- Ensure the interface is up/up.
- Set a hostname.
- Set an IP domain name.
- Create a local user or AAA method for authentication.
- Generate RSA keys.
- Enable SSH on the VTY lines.

A common exam point is that the device needs a configured IP address on a reachable interface before remote SSH management can work. If the device has no reachable IP, you cannot SSH to it.

Example interface setup:

```cisco
conf t
interface vlan 1
 ip address 192.168.1.10 255.255.255.0
 no shutdown
exit
ip default-gateway 192.168.1.1
end
```

If the device is a Layer 3 switch or router, the management IP may be on a routed interface or SVI (Switched Virtual Interface). The exact interface type matters less than reachability.

**Basic SSH configuration**

A typical secure SSH setup on Cisco IOS looks like this:

```cisco
conf t
hostname SW1
ip domain-name example.local
username admin privilege 15 secret Str0ngP@ssw0rd
crypto key generate rsa modulus 2048
ip ssh version 2
line vty 0 15
 login local
 transport input ssh
exit
end
wr mem
```

What each command does:

- `hostname SW1` gives the device an identity used in key generation and prompts.
- `ip domain-name example.local` is required on many IOS versions before RSA key generation.
- `username admin privilege 15 secret ...` creates a local account.
- `crypto key generate rsa modulus 2048` creates the SSH key pair.
- `ip ssh version 2` enables SSHv2, which is preferred over SSHv1.
- `login local` tells the VTY lines to use the local username database.
- `transport input ssh` allows only SSH on the VTY lines.

**VTY line access control**

VTY lines control how remote users connect. The most common CCNA patterns are:

- `transport input ssh`  
  Allows SSH only.
- `transport input ssh telnet`  
  Allows both SSH and Telnet, often used during migration or troubleshooting.
- `transport input telnet`  
  Allows Telnet only.
- `transport input none`  
  Disables inbound remote access on those VTY lines.

Use `transport input ssh telnet` when you need a transition period and want to test SSH without immediately removing Telnet. Use `transport input ssh` when you want to enforce encrypted access only.

Example:

```cisco
line vty 0 4
 login local
 transport input ssh telnet
```

This configuration permits both protocols, so you can verify SSH while still allowing legacy Telnet access if needed. For secure production management, you normally remove Telnet after validation.

**Authentication methods: `login local` vs `login authentication`**

This is a common exam topic.

- `login local`  
  Uses the local username database on the device for VTY login.
- `login authentication <method-list>`  
  Uses AAA method lists, such as a local database, RADIUS, or TACACS+ depending on the configured method list.

Use `login local` when:
- You want a simple local-only setup.
- AAA is not configured.
- You need a quick secure baseline.

Use `login authentication <method-list>` when:
- AAA is enabled.
- You want centralized authentication or fallback methods.
- You need more control over authentication behavior.

Example AAA-style configuration:

```cisco
conf t
aaa new-model
aaa authentication login default local
username admin secret Str0ngP@ssw0rd
line vty 0 15
 login authentication default
 transport input ssh
end
```

In this example, the VTY lines use the AAA method list named `default`. If AAA is not enabled, `login local` is the simpler choice.

**Privilege and login are not the same**

SSH login only gets you into the device. It does not automatically grant privileged EXEC mode unless the account or AAA policy provides it.

Important distinctions:

- **Login authentication** verifies identity.
- **Privilege level** determines command access after login.
- `enable secret` protects access to privileged EXEC mode when the user is not already privileged.

Example:

```cisco
username ops secret OpsP@ss123
enable secret En@bleP@ss123
line vty 0 4
 login local
 transport input ssh
```

A user may log in successfully and still need to enter `enable` and the enable secret to reach privileged mode, depending on the account privilege level.

**SSH hardening commands**

Cisco IOS includes SSH-related commands that are often tested in exam prep.

- `ip ssh version 2`  
  Forces SSHv2.
- `ip ssh time-out <seconds>`  
  Sets the SSH authentication/login timeout, not the idle session timeout.
- `ip ssh authentication-retries <count>`  
  Limits the number of failed SSH login attempts before disconnect.
- `ip ssh source-interface <interface>`  
  Sets the source interface used by the device for its own SSH connections and for SSH client sessions initiated by the device.

Example:

```cisco
conf t
ip ssh version 2
ip ssh time-out 60
ip ssh authentication-retries 2
ip ssh source-interface loopback0
end
```

Notes:

- `ip ssh time-out 60` controls how long the device waits during SSH authentication before timing out.
- Idle session timeout is configured on the VTY line with `exec-timeout`, not with `ip ssh time-out`.
- `ip ssh source-interface` is useful when the device initiates SSH sessions and when you want a stable source address for management traffic.

Example VTY idle timeout:

```cisco
line vty 0 4
 exec-timeout 5 0
```

This disconnects idle sessions after 5 minutes. It is separate from SSH authentication timeout.

**RSA keys and host identity**

SSH depends on cryptographic keys. On Cisco IOS, you usually generate RSA keys after setting the hostname and domain name.

Typical sequence:

1. Set `hostname`.
2. Set `ip domain-name`.
3. Generate RSA keys.
4. Enable SSHv2.

Example:

```cisco
conf t
hostname R1
ip domain-name lab.local
crypto key generate rsa modulus 2048
ip ssh version 2
end
```

If you skip the hostname or domain name on many IOS platforms, key generation may fail or be incomplete. The exact prompts vary by platform and IOS release, but the exam-level concept is consistent: SSH needs keys, and keys need device identity information.

**First-time connection and host key trust**

When you connect to a device by SSH for the first time, the client usually sees a host key fingerprint prompt. This is normal.

What happens:

- The client does not yet know the server’s host key.
- You are prompted to trust and store the key.
- Future connections compare the stored key to the device’s presented key.

If the host key changes unexpectedly, the client may warn about a possible security issue. That can happen after key regeneration, device replacement, or a man-in-the-middle concern.

Practical example from a workstation:

```bash
ssh admin@192.168.1.10
```

First connection may show a fingerprint prompt similar to:

```text
The authenticity of host '192.168.1.10' can't be established.
ECDSA key fingerprint is SHA256:...
Are you sure you want to continue connecting (yes/no/[fingerprint])?
```

For exam understanding, remember:
- First connection often requires trust confirmation.
- A changed host key should be investigated.
- SSH client trust is separate from username/password authentication.

**Verification commands**

You should verify both configuration and active sessions.

Useful commands:

```cisco
show ip ssh
show running-config | section line vty
show running-config | include username|ip ssh|transport input|login
show users
show ssh
```

What to look for:

- `show ip ssh`  
  Confirms SSH version, timeout, and retry settings.
- `show users`  
  Shows active console and VTY sessions.
- `show ssh`  
  Shows SSH session status and active connections.
- `show running-config | section line vty`  
  Confirms VTY restrictions and login method.
- `show running-config | include username|ip ssh|transport input|login`  
  Quickly checks the key SSH-related lines.

Example interpretation:

```text
SW1# show ssh
Connection Version Encryption State Username
0          2.0     AES128      Session started admin
```

This indicates an active SSH session using SSHv2 and encryption. If no sessions appear, the device may still be configured correctly but currently unused.

`show users` can help you identify who is connected:

```text
SW1# show users
    Line       User       Host(s)              Idle       Location
*  0 con 0                idle                 00:00:00
   2 vty 0     admin      idle                 00:00:12 192.168.1.50
```

The asterisk marks the current line. VTY entries show remote users and their source host.

**Troubleshooting SSH**

When SSH fails, check the problem in layers: reachability, service configuration, authentication, and line access.

Common issues:

- No IP address on the device or wrong subnet.
- Interface is down or administratively shut down.
- Missing `ip domain-name` or RSA keys.
- VTY lines do not permit SSH.
- AAA or local username is missing.
- ACLs block TCP port 22.
- The client is using the wrong username or password.
- The device is using SSHv1 when the client expects SSHv2, or vice versa.

A practical troubleshooting flow:

1. Verify reachability with `ping` from the client or device.
2. Confirm the management interface has the correct IP and is up.
3. Check `show ip ssh`.
4. Check VTY configuration with `show run | section line vty`.
5. Confirm a local user or AAA method exists.
6. Verify ACLs and routing if the connection still fails.

Example client-side test:

```bash
ping 192.168.1.10
ssh admin@192.168.1.10
```

If the connection is refused or times out, the cause is often one of these:
- The device is unreachable at Layer 3.
- SSH is not enabled correctly.
- VTY lines do not accept SSH.
- An ACL or firewall blocks TCP 22.
- The device lacks valid RSA keys.

> On Cisco IOS, SSH is not usually described as a separate daemon you “start” like on a server OS. Failures are more often caused by missing keys, disabled SSH, VTY restrictions, or reachability problems.

**Common exam pitfalls**

These are easy to mix up:

- `ip ssh time-out` is not the same as VTY idle timeout.
- `login local` is not the same as `enable secret`.
- `transport input ssh` allows SSH only; it does not “turn on SSH” by itself if keys are missing.
- `service password-encryption` does not secure SSH and is weak protection for line passwords.
- `transport input none` disables inbound remote access on the VTY lines.
- `transport input ssh telnet` allows both protocols, which is useful during migration but not ideal for final hardening.
- SSH login success does not guarantee privileged EXEC access.

**`service password-encryption` in context**

`service password-encryption` obscures some plaintext passwords in the configuration using a weak reversible type 7 scheme. It is not a real security control for SSH or for strong password protection.

Example:

```cisco
conf t
service password-encryption
end
```

Use it only as a cosmetic or legacy measure. For exam purposes, do not treat it as meaningful encryption. Prefer `secret`-based passwords and SSH.

**Practical secure management example**

A compact CCNA-style secure management configuration might look like this:

```cisco
conf t
hostname SW1
ip domain-name lab.local
enable secret En@bleP@ss123
username admin privilege 15 secret Adm1nP@ss123
crypto key generate rsa modulus 2048
ip ssh version 2
ip ssh time-out 60
ip ssh authentication-retries 2
line vty 0 15
 login local
 transport input ssh
 exec-timeout 10 0
end
wr mem
```

Why this works:

- The device has an identity and RSA keys.
- SSHv2 is enabled.
- Authentication is local and encrypted in transit.
- VTY lines accept only SSH.
- Idle sessions disconnect after 10 minutes.
- Login attempts are limited.

**When to use SSH vs Telnet**

| Feature | SSH | Telnet |
|---|---:|---:|
| Encryption | Yes | No |
| Credential protection | Yes | No |
| Recommended for management | Yes | No |
| CCNA secure default | Yes | No |

Use Telnet only when a lab, legacy device, or temporary transition requires it. For secure device management, SSH is the standard.

**Quick exam checklist**

Before you answer an SSH configuration question, verify these items:

- The device has a reachable IP address.
- The hostname and domain name are set.
- RSA keys exist.
- SSHv2 is enabled.
- VTY lines allow SSH.
- A local user or AAA method exists.
- Idle timeout is set with `exec-timeout` if needed.
- You know whether the account lands in user EXEC or privileged EXEC mode.

**Summary**

Secure device management on Cisco IOS centers on SSH, VTY restrictions, and proper authentication. The most important exam ideas are simple: give the device a reachable IP, generate RSA keys, enable SSHv2, restrict VTY access to SSH, and choose the correct login method. Then verify with `show ip ssh`, `show users`, and `show ssh` so you can confirm both configuration and active sessions.
