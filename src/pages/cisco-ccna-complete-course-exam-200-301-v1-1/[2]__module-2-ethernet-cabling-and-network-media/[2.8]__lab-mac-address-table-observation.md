## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Lab purpose and switching behavior**

A MAC address table, also called a CAM (Content Addressable Memory) table, is the switch’s forwarding database. In this lab, you observe how a Layer 2 switch learns source MAC addresses, associates them with switch ports, and uses that information to forward frames only where they need to go.

The key idea is simple: a switch learns from the **source MAC address** of incoming frames, not the destination. If the destination is unknown, the switch floods the frame out all other ports in the same VLAN except the ingress port. Once the destination responds, the switch learns that MAC address too and can forward future frames unicast.

**What you should observe in the lab**

- The switch creates table entries after hosts send traffic.
- Each entry maps a **MAC address** to a specific interface and VLAN.
- Entries age out after inactivity unless refreshed by new traffic.
- Broadcast and unknown unicast traffic are flooded within the VLAN.
- Known unicast traffic is forwarded only to the correct port.

**Core concepts**

A switch operates at Layer 2 of the OSI model and makes forwarding decisions using MAC addresses. It does not inspect IP addresses for normal switching decisions. When a frame arrives, the switch checks the source MAC address and records it if it is new. Then it checks the destination MAC address against the table.

- **Learning** happens on ingress.
- **Forwarding** happens on egress based on the destination lookup.
- **Flooding** happens when the destination is unknown, broadcast, or certain multicast cases.
- **Filtering** happens when the destination is on the same ingress port, so the frame does not need to be sent back out that port.

**Typical lab topology**

A simple lab uses one switch and two end devices in the same VLAN.

```text
PC1 ---- Fa0/1
          |
        [SW1]
          |
PC2 ---- Fa0/2
```

If PC1 pings PC2, the switch first learns PC1’s MAC on `Fa0/1`. If PC2 replies, the switch learns PC2’s MAC on `Fa0/2`. After that, frames between the two hosts are forwarded as known unicast traffic.

**Useful verification commands**

Use these commands to inspect the MAC address table and related interface state:

```bash
show mac address-table
show mac address-table dynamic
show mac address-table interface fa0/1
show mac address-table vlan 10
show interfaces status
show interfaces fa0/1 switchport
```

On many Cisco switches, `show mac address-table` displays columns similar to these:

- **VLAN**: the VLAN associated with the learned MAC
- **MAC Address**: the learned Layer 2 address
- **Type**: dynamic, static, or secure
- **Ports**: the interface where the MAC was learned

**How learning works**

When a frame enters a switch port, the switch examines the source MAC address and records it in the table if it is not already present. If the same source MAC appears on a different port, the switch updates the entry to the new port. This is why the table reflects where a device is currently reachable.

This behavior is important in real networks because devices move, links fail over, and access ports may be repatched. The switch continuously refreshes its view of the network based on observed traffic.

**Example observation sequence**

1. Clear the table if needed.
2. Generate traffic from PC1 to PC2.
3. Check the MAC table.
4. Generate reply traffic from PC2 to PC1.
5. Check the table again and compare entries.

Example commands:

```bash
clear mac address-table dynamic
show mac address-table dynamic
ping 192.168.10.2
show mac address-table dynamic
```

> Be careful when clearing the MAC table on a production switch. It can temporarily increase flooding until the switch relearns active addresses.

**Aging and refresh behavior**

Dynamic MAC entries do not stay forever. They age out after a period of inactivity, commonly 300 seconds on Cisco switches unless changed by configuration. If a host remains silent long enough, its entry is removed. The next frame from that host causes the switch to relearn it.

This matters because:

- Quiet hosts may disappear from the table temporarily.
- Active hosts keep their entries refreshed.
- Aging helps the table stay accurate and prevents stale forwarding information.

You can inspect aging-related behavior with:

```bash
show mac address-table aging-time
show mac address-table count
```

**Forwarding outcomes**

A switch can handle a frame in several ways:

- **Known unicast**: destination MAC is in the table, so the frame is sent only out the correct port.
- **Unknown unicast**: destination MAC is not in the table, so the frame is flooded within the VLAN.
- **Broadcast**: destination `ff:ff:ff:ff:ff:ff`, so the frame is flooded within the VLAN.
- **Multicast**: may be flooded or selectively forwarded depending on the protocol and switch features.

This is why the first ping between two hosts often takes longer than later pings: the switch and hosts may need to resolve addresses and learn paths before steady-state forwarding begins.

**Practical lab example**

Suppose PC1 is connected to `Fa0/1` in VLAN 10 and PC2 is connected to `Fa0/2` in VLAN 10.

```text
VLAN 10
PC1 --- Fa0/1 [SW1] Fa0/2 --- PC2
```

After PC1 sends traffic:

```bash
show mac address-table dynamic
```

You might see output conceptually like this:

```text
Vlan    Mac Address       Type        Ports
----    -----------       --------    -----
10      00e0.b0aa.1111    DYNAMIC     Fa0/1
```

After PC2 replies:

```text
Vlan    Mac Address       Type        Ports
----    -----------       --------    -----
10      00e0.b0aa.1111    DYNAMIC     Fa0/1
10      00e0.b0aa.2222    DYNAMIC     Fa0/2
```

Now the switch can forward frames between the two ports without flooding.

**VLAN awareness**

MAC address tables are VLAN-specific. The same MAC address can appear in different VLAN contexts on different ports or in special designs, but forwarding always occurs within the VLAN boundary. A switch does not forward Layer 2 frames between VLANs unless a Layer 3 device or Layer 3 interface performs routing.

This is why you should always check the VLAN column when troubleshooting:

- A MAC may be learned on the correct port but the wrong VLAN.
- A host may be connected and active but isolated by VLAN mismatch.
- Trunk and access port configuration issues can prevent expected learning behavior.

**Troubleshooting checklist**

If the MAC table does not look right, verify the following:

- The interface is up/up.
- The port is in the expected VLAN.
- The host is generating traffic.
- The switchport mode matches the design.
- The MAC entry is dynamic and not stale.
- There is no port security violation or err-disable condition.

Useful commands:

```bash
show interfaces status
show vlan brief
show interfaces fa0/1 switchport
show port-security interface fa0/1
show logging
```

**Static, dynamic, and secure entries**

Cisco switches can store different MAC entry types:

- **Dynamic** entries are learned automatically from traffic.
- **Static** entries are manually configured and do not age out.
- **Secure** entries are associated with port security and may be learned or configured depending on the policy.

Example static MAC entry:

```bash
mac address-table static 00e0.b0aa.3333 vlan 10 interface fa0/3
```

Static entries are useful when you need deterministic forwarding, but they are less flexible than dynamic learning. In most access-layer environments, dynamic learning is the normal behavior.

**Port security interaction**

If port security is enabled, the switch may limit which MAC addresses are allowed on a port. This can affect what you observe in the MAC table.

- A secure MAC may be learned and allowed.
- An unexpected MAC may trigger a violation.
- A violation can restrict forwarding or shut down the port depending on the configured mode.

Example:

```bash
interface fa0/1
 switchport mode access
 switchport port-security
 switchport port-security maximum 1
 switchport port-security mac-address sticky
 switchport port-security violation shutdown
```

In a lab, if a host is connected but traffic fails, check whether port security is blocking the frame rather than assuming the MAC table is wrong.

**Why this lab matters for CCNA**

This lab reinforces several exam-relevant ideas:

- Switches learn from source MAC addresses.
- MAC tables are used for Layer 2 forwarding decisions.
- Unknown destinations cause flooding within the VLAN.
- MAC entries age out if unused.
- VLAN membership affects where frames can be forwarded.
- Troubleshooting often starts with `show mac address-table` and interface status checks.

**Common misconceptions**

- A switch does not learn the destination MAC from the first frame; it learns the source MAC.
- A MAC table entry is not the same as an ARP (Address Resolution Protocol) table entry. ARP maps IPv4 addresses to MAC addresses on hosts and routers; the switch MAC table maps MAC addresses to ports.
- A MAC table does not route traffic between VLANs.
- A device may be reachable even if its MAC entry is temporarily absent, because the next frame can relearn it quickly.

**Quick comparison: MAC table vs ARP table**

| Feature | MAC Address Table | ARP Table |
|---|---|---|
| Layer | Layer 2 | Layer 3 to Layer 2 mapping |
| Device | Switch | Host or router |
| Learns from | Source MAC in frames | ARP requests/replies |
| Purpose | Forward frames to ports | Resolve IP to MAC |
| Scope | Per VLAN on a switch | Per IP subnet/interface context |

**Suggested lab workflow**

1. Confirm the physical connections and interface status.
2. Verify VLAN assignment on the switch ports.
3. Clear the dynamic MAC table if the lab allows it.
4. Generate traffic between hosts, such as `ping`.
5. Observe the MAC table before and after traffic.
6. Note which port each MAC address is learned on.
7. Wait for the aging timer or stop traffic and observe entry removal.

**Example packet-flow interpretation**

When PC1 sends an ARP request for PC2, the switch floods the broadcast frame in VLAN 10. PC2 receives it and replies. The switch learns both MAC addresses from the source fields of those frames. After that, ICMP (Internet Control Message Protocol) echo traffic between the hosts is forwarded as known unicast.

**Key takeaways**

- The switch learns MAC addresses from incoming source addresses.
- The MAC table ties a MAC address to a VLAN and port.
- Unknown destinations are flooded within the VLAN.
- Dynamic entries age out unless refreshed.
- `show mac address-table` is the primary command for observation and troubleshooting.
