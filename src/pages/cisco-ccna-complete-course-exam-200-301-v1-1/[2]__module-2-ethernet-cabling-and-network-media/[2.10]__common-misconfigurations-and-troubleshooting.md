## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Troubleshooting mindset**

Troubleshooting is a process of isolating the fault domain, verifying assumptions, and changing one variable at a time. In CCNA-level networks, most outages come from a small set of predictable issues: wrong IP settings, missing default gateways, VLAN mismatches, trunk errors, bad routing statements, ACLs, and Layer 1 problems such as cabling or duplex/speed mismatches.

A good workflow reduces guesswork:

- Identify the symptom.
- Determine the scope.
- Compare the current state to the intended design.
- Test the most likely cause first.
- Verify the fix and document it.

**Common troubleshooting methodology**

Use a layered approach so you do not skip obvious checks. The OSI model helps you narrow the problem from physical connectivity up to application reachability.

1. Check Layer 1 for power, cabling, link lights, and interface status.
2. Check Layer 2 for VLAN membership, trunking, MAC learning, and STP behavior.
3. Check Layer 3 for IP addressing, masks, gateways, routing, and ARP.
4. Check Layer 4 and above for ACLs, DNS, NAT, and service availability.

Useful verification commands include:

```bash
show ip interface brief
show interfaces status
show interfaces
show vlan brief
show interfaces trunk
show mac address-table
show ip route
show arp
ping
traceroute
```

**Layer 1 misconfigurations and failures**

Physical issues are often the easiest to confirm but the easiest to overlook. If an interface is down, start with the cable, transceiver, port, and remote device status before changing configuration.

Common Layer 1 problems include:

- Unplugged or damaged cables.
- Incorrect fiber type or bad SFP/transceiver.
- Interface administratively shut down.
- Speed and duplex mismatches on older or manually configured links.
- Wrong patch-panel or switchport connection.

On Cisco devices, verify interface state with:

```bash
show ip interface brief
show interfaces gigabitEthernet0/1
```

If needed, enable an interface:

```bash
configure terminal
interface gigabitEthernet0/1
no shutdown
```

> A link can appear “up” while still being unusable if speed, duplex, or media settings do not match on both ends.

**Layer 2 misconfigurations**

Layer 2 issues are common in switched networks because VLANs, trunks, and STP must align across devices. A host may have link but still fail to reach other devices if it is in the wrong VLAN or if the trunk does not carry the expected VLANs.

Typical Layer 2 mistakes:

- Access port assigned to the wrong VLAN.
- VLAN not created on the switch.
- Trunk not configured on one side.
- Native VLAN mismatch.
- Allowed VLAN list missing the required VLAN.
- EtherChannel members configured inconsistently.
- STP blocking a port unexpectedly due to topology or priority.

Check VLAN and trunk state with:

```bash
show vlan brief
show interfaces trunk
show spanning-tree
show etherchannel summary
```

Example access port configuration:

```bash
configure terminal
vlan 10
 name Users
interface gigabitEthernet0/10
 switchport mode access
 switchport access vlan 10
 spanning-tree portfast
```

Example trunk configuration:

```bash
configure terminal
interface gigabitEthernet0/1
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30
 switchport trunk native vlan 99
```

If hosts in the same VLAN cannot communicate, check:

- Whether both ports are in the same VLAN.
- Whether the switch has learned the MAC addresses.
- Whether STP is blocking one of the ports.
- Whether port security is limiting traffic.

**VLAN and trunk troubleshooting clues**

A host in the wrong VLAN often shows these symptoms:

- It can reach the local switch management IP only if that IP is in the same VLAN.
- It cannot reach the default gateway.
- It can ping some devices but not others, depending on VLAN placement.

A trunk problem often shows these symptoms:

- One VLAN works while another does not.
- Inter-VLAN routing fails for only certain VLANs.
- A downstream switch cannot reach upstream resources.

Useful checks:

```bash
show interfaces gigabitEthernet0/1 switchport
show interfaces trunk
show vlan brief
```

**Layer 3 misconfigurations**

Layer 3 problems usually involve IP addressing, subnet masks, default gateways, routing, or ARP resolution. These are among the most common causes of “network down” reports.

Common Layer 3 mistakes:

- Wrong IP address or subnet mask.
- Duplicate IP address.
- Missing or incorrect default gateway on hosts.
- Incorrect static route.
- Missing route advertisement in a dynamic routing process.
- Wrong next-hop address.
- Interface IP configured in the wrong subnet.
- ACL blocking traffic unexpectedly.

Verify host and interface addressing:

```bash
show ip interface brief
show running-config interface gigabitEthernet0/0
show ip route
show arp
```

Example router interface configuration:

```bash
configure terminal
interface gigabitEthernet0/0
 ip address 192.168.10.1 255.255.255.0
 no shutdown
```

Example host settings:

```text
IP address: 192.168.10.25
Subnet mask: 255.255.255.0
Default gateway: 192.168.10.1
DNS server: 8.8.8.8
```

If a host can reach its gateway but not remote networks, the problem is often routing rather than local connectivity.

**Routing issues**

Routing failures are common when a network is partially reachable. The local subnet works, but traffic stops at the router or never returns.

Typical routing problems:

- Missing route to the destination network.
- Missing return route.
- Incorrect static route next hop.
- Overlapping or incorrect summary route.
- Dynamic routing neighbor adjacency failure.
- Passive interface or network statement mismatch.

Static route example:

```bash
configure terminal
ip route 10.10.20.0 255.255.255.0 192.168.1.2
```

Verify routes with:

```bash
show ip route
show ip protocols
show running-config | include ^ip route|router
```

When troubleshooting routing, always check both directions:

- Forward path from source to destination.
- Return path from destination back to source.

A one-way route can make ping fail even when one side appears correctly configured.

**ACL-related problems**

Access control lists (ACLs) can silently block traffic if applied in the wrong direction, on the wrong interface, or with the wrong sequence of permit and deny statements. Because ACLs are processed top-down, the first match wins.

Common ACL mistakes:

- ACL applied inbound instead of outbound, or vice versa.
- Wrong source or destination wildcard mask.
- Missing permit statement for required traffic.
- Implicit deny at the end of the ACL.
- ACL applied to the wrong interface.
- Extended ACL placed too far from the source or standard ACL placed too close to the destination, contrary to design intent.

Verify ACLs with:

```bash
show access-lists
show ip interface gigabitEthernet0/0
show running-config | section access-list
```

Example ACL:

```bash
configure terminal
ip access-list extended WEB-FILTER
 deny tcp any host 192.168.10.50 eq 80
 permit ip any any
interface gigabitEthernet0/1
 ip access-group WEB-FILTER in
```

If traffic is unexpectedly blocked, check counters on the ACL entries and confirm the interface direction.

**NAT and DHCP issues**

Network Address Translation (NAT) and Dynamic Host Configuration Protocol (DHCP) problems often look like general connectivity failures.

NAT issues may include:

- Inside and outside interfaces reversed.
- Missing NAT overload rule.
- ACL used for NAT does not match the inside subnet.
- Translation table not building because traffic never matches.

DHCP issues may include:

- DHCP server unreachable.
- Wrong helper address on the router.
- Scope exhausted.
- Incorrect default gateway or DNS options.
- VLAN or trunk issue preventing DHCP broadcast from reaching the server.

Useful commands:

```bash
show ip nat translations
show ip nat statistics
show running-config | include ip nat
show ip dhcp binding
show ip dhcp pool
```

Example DHCP relay configuration:

```bash
configure terminal
interface vlan 10
 ip address 192.168.10.1 255.255.255.0
 ip helper-address 192.168.100.10
```

**STP and EtherChannel problems**

Spanning Tree Protocol (STP) prevents loops, but a misread topology can make a healthy port look broken. EtherChannel bundles links, but all member interfaces must match in key settings.

STP-related symptoms:

- A port is in blocking or alternate state.
- Traffic fails on one path but works on another.
- A newly connected switch causes instability or topology changes.

EtherChannel-related mistakes:

- Mismatched channel mode.
- Inconsistent trunk/access settings.
- Different allowed VLAN lists.
- Speed/duplex mismatch across members.
- One side configured for LACP while the other uses a different protocol or no bundling.

Check with:

```bash
show spanning-tree
show etherchannel summary
show interfaces port-channel 1
```

**Host-side troubleshooting**

Not every issue is on the network device. Host configuration errors are common and should be checked early.

Verify:

- IP address, subnet mask, and default gateway.
- DNS server settings.
- Wi-Fi association or Ethernet link.
- Firewall rules on the endpoint.
- VPN client state if remote access is involved.

Useful host commands:

```bash
ipconfig /all
ping 127.0.0.1
ping <default-gateway>
ping <remote-ip>
tracert <remote-ip>
nslookup example.com
```

On Linux:

```bash
ip addr
ip route
ping -c 4 192.168.10.1
traceroute 8.8.8.8
```

A practical sequence is to test from the host outward:

1. Loopback or local stack.
2. Local IP address.
3. Default gateway.
4. Same-subnet peer.
5. Remote subnet.
6. DNS name resolution.

**Cisco device troubleshooting workflow**

When you troubleshoot a Cisco switch or router, use a repeatable sequence so you can isolate the fault quickly.

1. Confirm the symptom with a test such as `ping` or `traceroute`.
2. Check interface status with `show ip interface brief`.
3. Inspect Layer 2 state with `show vlan brief` and `show interfaces trunk`.
4. Verify Layer 3 with `show ip route` and `show arp`.
5. Review the running configuration for the affected interface or feature.
6. Compare the device to a known-good neighbor.
7. Make one change, then retest.

Example investigation flow:

```bash
show ip interface brief
show interfaces gigabitEthernet0/1
show vlan brief
show interfaces trunk
show ip route
ping 192.168.10.1
traceroute 8.8.8.8
```

**Common “symptom to cause” patterns**

These patterns help you narrow the issue faster:

- No link light: physical cable, port shutdown, or bad transceiver.
- Link up, no traffic: VLAN mismatch, trunk issue, ACL, or IP mismatch.
- Same VLAN works, other VLANs fail: trunk allowed VLANs, inter-VLAN routing, or ACL.
- Can ping gateway, not Internet: routing, NAT, or upstream issue.
- Can ping IP, not name: DNS problem.
- One device cannot reach others: host IP, mask, gateway, or port security.
- Intermittent connectivity: duplex mismatch, STP changes, or flapping link.

**Verification and rollback**

After a fix, verify more than one path if possible. A single successful ping does not prove the network is fully healthy.

Check:

- Local and remote reachability.
- Name resolution if applicable.
- Return traffic.
- Logs and counters for errors or drops.

Useful commands:

```bash
show logging
show interfaces counters errors
show ip route
show access-lists
```

If a change does not help, revert it before trying the next hypothesis. This keeps troubleshooting controlled and prevents multiple overlapping faults.

**Practical troubleshooting examples**

Example 1: A PC cannot reach the default gateway.

- Check the PC IP settings.
- Confirm the switchport is in the correct VLAN.
- Verify the gateway interface is up.
- Test the switchport with another known-good host.

Example 2: Two switches cannot pass traffic for VLAN 20.

- Confirm VLAN 20 exists on both switches.
- Check trunk status and allowed VLANs.
- Verify native VLAN consistency.
- Inspect STP state on the uplink.

Example 3: Users can reach internal servers but not the Internet.

- Verify the default route on the edge router.
- Check NAT overload configuration.
- Confirm the inside subnet matches the NAT ACL.
- Test upstream connectivity from the router.

**Best practices**

- Document the intended design before troubleshooting.
- Change one thing at a time.
- Start with the simplest explanation.
- Use show commands before making changes.
- Compare against a known-good device or interface.
- Save the working configuration after validation.
