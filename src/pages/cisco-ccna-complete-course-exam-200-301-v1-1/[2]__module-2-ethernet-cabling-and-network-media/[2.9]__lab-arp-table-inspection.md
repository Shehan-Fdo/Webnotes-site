## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**ARP fundamentals**

<abbr title="Address Resolution Protocol">ARP</abbr> maps an IPv4 address to a Layer 2 <abbr title="Media Access Control">MAC</abbr> address on a local broadcast domain. You inspect the ARP table when you need to confirm how a host or router learned a neighbor’s hardware address, or when you are troubleshooting why traffic is not reaching the next hop.

ARP matters most for local delivery:
- If the destination is on the same subnet, the sender resolves the destination host’s MAC address directly.
- If the destination is off subnet, the sender resolves the default gateway’s MAC address and forwards the frame to that router.
- On routers, the ARP table helps you verify which neighbors are reachable on directly connected IPv4 networks.

> ARP is an IPv4 mechanism. IPv6 uses <abbr title="Neighbor Discovery Protocol">NDP</abbr> instead of ARP.

**How ARP works in a lab**

When a device needs a MAC address and does not already have one cached, it sends an ARP request as a broadcast frame. The owner of the IPv4 address replies with its MAC address in an ARP reply. The sender then stores the mapping in its ARP cache for a period of time.

A typical sequence is:
1. The host checks its ARP cache for the target IPv4 address.
2. If no valid entry exists, it sends an ARP request.
3. The device with that IPv4 address replies with its MAC address.
4. The sender stores the mapping and uses it for subsequent frames.

This behavior is why ARP inspection is useful in labs: you can generate traffic, then verify that the expected mapping appears in the table.

**Cisco IOS ARP table inspection**

On Cisco IOS and IOS XE, the primary command to view the ARP table is:

```bash
show ip arp
```

A typical output looks like this:

```bash
Router# show ip arp
Protocol  Address          Age (min)  Hardware Addr   Type   Interface
Internet  192.168.1.1             -    aaaa.bbbb.cc01  ARPA   GigabitEthernet0/0
Internet  192.168.1.50            2    aaaa.bbbb.cc50  ARPA   GigabitEthernet0/0
Internet  192.168.1.100           0    aaaa.bbbb.cc64  ARPA   GigabitEthernet0/0
```

Key points about the columns:
- **Protocol** usually shows `Internet` for IPv4 entries.
- **Address** is the IPv4 address being resolved.
- **Age (min)** shows how long the entry has been in the table, or `-` for entries that do not age in the usual way, such as some static or local entries depending on platform behavior.
- **Hardware Addr** is the resolved MAC address.
- **Type** commonly shows the encapsulation type, such as `ARPA`, rather than a dynamic/static state.
- **Interface** identifies the local interface associated with the entry.

**What to look for in the table**

ARP inspection is not just about reading a list. You are checking whether the table matches the expected topology and traffic flow.

- **Correct IP-to-MAC mapping** confirms that the device learned the neighbor successfully.
- **Correct interface** confirms the entry belongs on the expected Layer 2 segment.
- **Recent age value** suggests active traffic or a recently learned neighbor.
- **Missing entry** may indicate no traffic has been generated, a VLAN mismatch, an IP addressing problem, or a Layer 2 reachability issue.
- **Unexpected MAC address** can indicate duplicate IP addressing, a mispatched cable, a rogue device, or a stale cache entry.

**Static and local entries**

Some ARP entries do not age normally. On Cisco platforms, a dash in the age field often indicates an entry that is not aging in the usual dynamic sense, but the exact meaning depends on the platform and entry type.

Examples of non-aging or special entries can include:
- A statically configured ARP entry.
- The router’s own interface address on a connected network.
- Platform-specific local or permanent entries.

Use caution when interpreting `-`; do not assume it always means the same thing on every device.

**Configuring a static ARP entry**

Static ARP entries are useful when you need a fixed IP-to-MAC mapping, such as in controlled lab scenarios or for devices that should not rely on dynamic learning. On Cisco IOS/IOS XE, the syntax can vary slightly by platform and version, but a common global configuration form is:

```bash
Router(config)# arp 192.168.1.50 aaaa.bbbb.cccc arpa
```

Some platforms also support an optional `alias` keyword. Always verify the exact syntax on the target device with context-sensitive help.

> Static ARP entries reduce flexibility. Use them only when you have a clear operational reason, because a changed NIC, VM migration, or replacement device can break connectivity until the entry is updated.

**Clearing ARP cache entries**

If you suspect stale mappings, you can clear the ARP cache and force relearning.

Common Cisco IOS/IOS XE approaches include:
```bash
Router# clear arp-cache
```

On some platforms or software trains, you may also see platform-specific variants such as `clear ip arp`. Use the command supported by the device you are working on.

After clearing the cache, generate traffic again so the device relearns the mapping:
```bash
Router# ping 192.168.1.50
Router# show ip arp
```

**Lab workflow for ARP inspection**

A practical lab usually follows this pattern:
1. Verify IP addressing on the host and router interfaces.
2. Confirm the devices are in the same VLAN or directly connected subnet.
3. Generate traffic with `ping` or another IPv4 packet.
4. Inspect the ARP table with `show ip arp`.
5. Compare the learned MAC address with the expected neighbor MAC.
6. Clear the cache if needed and repeat the test.

This workflow helps you separate ARP problems from routing problems. If the ARP table is correct but traffic still fails, the issue may be elsewhere, such as ACLs, interface shutdown, VLAN mismatch, or routing.

**Practical verification commands**

Use these commands together to build a complete picture:

```bash
Router# show ip interface brief
Router# show interfaces gigabitEthernet0/0
Router# show ip arp
Router# ping 192.168.1.50
Router# traceroute 8.8.8.8
```

On an end host, common checks include:
```bash
C:\> ipconfig /all
C:\> arp -a
```

On Linux:
```bash
$ ip addr show
$ ip neigh show
$ ping -c 4 192.168.1.1
```

These commands help you confirm whether the host has learned the gateway MAC, whether the router has learned the host MAC, and whether the neighbor cache is consistent with the physical topology.

**End-host ARP cache inspection**

End hosts maintain their own ARP cache, separate from router tables. The exact command depends on the operating system.

- Windows: `arp -a`
- Linux: `ip neigh show`
- macOS: `arp -a`

Example Linux output:
```bash
$ ip neigh show
192.168.1.1 dev eth0 lladdr aaaa.bbbb.cc01 REACHABLE
192.168.1.50 dev eth0 lladdr aaaa.bbbb.cc50 STALE
```

Interpretation:
- **REACHABLE** means the neighbor was recently confirmed.
- **STALE** means the entry exists but has not been confirmed recently.
- Other states may appear depending on the OS and traffic pattern.

**Troubleshooting patterns**

ARP inspection is especially useful when you see one of these symptoms:
- A host can ping the default gateway but not another host on the same subnet.
- A router can reach one neighbor but not another on the same interface.
- The ARP table shows an unexpected or duplicate MAC address.
- The ARP table is empty even after you expect traffic to have occurred.

Common causes include:
- Wrong subnet mask causing the device to ARP for the wrong destination.
- VLAN mismatch preventing ARP broadcasts from reaching the target.
- Interface down or administratively shut.
- Duplicate IP address causing conflicting replies.
- Stale cache entries after a NIC replacement or VM move.
- Security features such as dynamic ARP inspection or port security blocking traffic.

**Dynamic ARP Inspection context**

In switched networks, <abbr title="Dynamic ARP Inspection">DAI</abbr> can validate ARP packets against trusted bindings. If DAI is enabled and misconfigured, legitimate ARP replies may be dropped, which can make the ARP table appear incomplete or prevent correct learning.

Useful checks on a switch may include:
```bash
Switch# show ip arp inspection
Switch# show ip arp inspection statistics
Switch# show mac address-table
```

If ARP learning fails unexpectedly, confirm whether ARP inspection, DHCP snooping, or port trust settings are involved.

**Example lab scenario**

Suppose a router interface `G0/0` is connected to a switch VLAN with host `192.168.1.50`.

Expected behavior:
- The host ARPs for `192.168.1.1` if it needs to reach off-subnet traffic.
- The router learns `192.168.1.50` after the host sends traffic.
- `show ip arp` on the router displays the host’s IP and MAC on `GigabitEthernet0/0`.

Example verification:
```bash
Router# ping 192.168.1.50
Router# show ip arp
```

If the entry does not appear:
- Confirm the host and router are in the same subnet.
- Confirm the switch port is in the correct VLAN.
- Confirm the host firewall is not blocking ICMP if you are using ping as the trigger.
- Confirm the router interface is up/up.

**Key takeaways**

- ARP resolves IPv4 addresses to MAC addresses on a local network.
- `show ip arp` is the main Cisco IOS command for inspecting the ARP table.
- The `Type` column in standard IOS output commonly indicates encapsulation, such as `ARPA`.
- A dash in the age field usually means the entry does not age normally, but interpretation can vary by platform.
- `clear arp-cache` is the common IOS command to flush ARP entries, though platform-specific variants exist.
- ARP inspection is a fast way to separate Layer 2 neighbor issues from routing problems.
