**MTU fundamentals**

The <abbr title="Maximum Transmission Unit">MTU</abbr> is the largest packet or protocol data unit a link can carry in a single transaction without fragmentation. It is a link-level limit, so it depends on the interface, the media, and any encapsulation in use.

On Ethernet, the common MTU is 1500 bytes, but the actual value can be smaller or larger depending on the network design. Larger MTU values can reduce overhead, while smaller values can reduce delay in some cases.

- MTU applies to the largest data unit a layer can forward.
- It is often associated with a physical or logical interface.
- It may be set by standards, interface characteristics, or automatic discovery.
- It is not the same as the maximum frame size on the data-link layer.

**Fragmentation behavior**

If a packet is larger than the MTU on an outgoing interface, the device cannot forward it as-is. In IPv4, the device may fragment the packet into smaller pieces that fit the link. The destination host reassembles the fragments using fields in the IP header.

Fragmentation is useful when a packet must cross a smaller link, but it adds overhead and can hurt performance. Some protocols do not support fragmentation, so oversized packets are dropped instead.

- A router fragments only when the outgoing MTU is too small for the packet.
- Fragmented packets carry identification and offset information for reassembly.
- Reassembly happens at the endpoint host, not in transit.
- Fragmentation can increase delay and packet loss risk.

> If a packet exceeds the MTU and the protocol or path does not allow fragmentation, the packet is dropped.

**Physical media and encapsulation effects**

Physical media and encapsulation influence the usable packet size on a link. The MTU must fit the capabilities of the underlying network segment, including any added headers from tunnels or security features.

This is why MTU problems often appear in tunnel and security scenarios such as GRE and IPsec. Extra encapsulation reduces the space available for the original payload, so a packet that fits on one link may become too large after encapsulation.

- Physical interfaces can have different MTU limits.
- Encapsulation adds overhead and reduces effective payload size.
- Tunnel paths may require MTU tuning to avoid fragmentation.
- Jumbo frames are larger-than-standard frames used on some networks.

**MTU, MSS, and Path MTU Discovery**

<abbr title="Maximum Segment Size">MSS</abbr> is different from MTU. MTU limits the full IP packet on a link, while MSS limits the TCP payload negotiated during the connection setup. Both matter when you want to avoid fragmentation.

<abbr title="Path MTU Discovery">PMTUD</abbr> helps a system determine the smallest MTU along the path so it can send packets that fit without fragmentation. Some systems discover this automatically at connection time.

| Term | Scope | What it limits |
|---|---|---|
| MTU | Link or interface | Full packet size on the path segment |
| MSS | TCP | TCP payload size |
| PMTUD | End-to-end path | Packet size based on the smallest link |

**Practical troubleshooting approach**

When you troubleshoot MTU issues, start by checking the interface MTU and then consider any encapsulation in the path. If packets are being dropped or fragmented unexpectedly, compare the packet size to the smallest MTU along the route.

1. Check the interface MTU on each relevant link.
2. Identify tunnels, VPNs, or other encapsulation that add overhead.
3. Compare the packet size with the smallest MTU in the path.
4. Adjust MTU or MSS if needed to prevent fragmentation.
5. Verify that the traffic now passes without drops or excessive fragmentation.

**Key takeaways**

MTU is a size limit, fragmentation is the fallback when packets are too large, and physical media plus encapsulation determine the real usable size on a path. For CCNA troubleshooting, always think about the interface, the packet size, and the extra overhead added by the network design.
