## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Ethernet speed and duplex basics**

Ethernet links must agree on both **speed** and **duplex** to communicate cleanly. Speed is the link rate, such as `10/100/1000 Mb`, and duplex describes whether traffic can flow in one direction at a time or both directions at once.

Auto-negotiation is the normal way to establish these settings on modern Ethernet links. When both sides negotiate correctly, the interface comes up with matching values and avoids many common physical-layer problems.

- **Half-duplex** allows traffic in one direction at a time.
- **Full-duplex** allows simultaneous send and receive.
- **Auto-negotiation** helps both ends select compatible settings.
- A mismatch can happen when one side is set to auto and the other is forced to a fixed value.

> A duplex mismatch often produces symptoms that look like a cable or link problem even when the physical connection is present.

**Common mismatch symptoms**

A duplex mismatch usually shows up as unstable performance rather than a complete outage. The link may appear up, but traffic behaves poorly.

- **Late collisions** can appear in interface statistics.
- **FCS errors** may increase on one side of the link.
- Throughput can drop sharply, sometimes to a small fraction of expected speed.
- Pings may fail intermittently even though the interface stays up.
- Some systems may report flapping or a misleading `network cable unplugged` message.

These symptoms are especially useful when the interface is operational but the user experience is poor. In that case, you should suspect a speed or duplex disagreement before replacing hardware.

**Troubleshooting approach**

Use a structured check to confirm the negotiated settings and look for error counters. On Cisco switches, the goal is to verify the interface state and compare both ends of the link.

1. Check the interface for the current speed and duplex values.
2. Compare both sides of the connection for matching settings.
3. Review interface counters for collisions, errors, and drops.
4. If one side is forced, make sure the other side is configured to match.
5. Re-test connectivity after correcting the settings.

When you see auto-negotiation on one end and a forced setting on the other, treat that as a likely cause of trouble. Conservative troubleshooting means confirming the configuration first before assuming a cable fault.

**Useful verification examples**

Different platforms expose the same information in different ways. The key is to confirm the negotiated speed, duplex, and error counters.

```bash
ethtool eth0
ethtool -S eth0 | grep -E "error|collision|miss|drop"
ip -s link show
```

On Cisco devices, interface status and error counters help you identify whether the problem is physical, configuration-related, or both. If the link is up but errors continue to rise, focus on the negotiated settings and the cable path.

**What to remember for CCNA**

- Match **speed** and **duplex** on both ends of the link.
- Prefer auto-negotiation unless a specific design requires a fixed setting.
- Look for collisions, errors, and drops when troubleshooting unstable links.
- A link that is up is not necessarily healthy.
- Physical-layer issues often present as performance problems before total failure.

**Practical interpretation**

If one side is `auto/auto` and the other side is forced to `100/full`, the link can behave badly. The exact symptoms depend on the devices involved, but the safe assumption is that the settings are inconsistent until proven otherwise.

Use the interface counters and negotiated values together. That combination gives you a faster path to the root cause than checking only connectivity tests.
