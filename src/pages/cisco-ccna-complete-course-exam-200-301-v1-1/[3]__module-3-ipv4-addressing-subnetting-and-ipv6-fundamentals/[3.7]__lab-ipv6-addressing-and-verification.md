## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**IPv6 addressing fundamentals**

IPv6 uses 128-bit addresses written in hexadecimal and separated by colons. In CCNA labs, you usually work with a global unicast prefix for routed networks, a link-local address on every interface, and sometimes a default route for reachability beyond the local topology.

A useful habit is to read and write IPv6 in both expanded and compressed forms. Compression rules help you verify addresses quickly during labs:

- Remove leading zeros in each hextet.
- Replace one contiguous run of all-zero hextets with `::` only once.
- Keep the prefix length separate from the address, such as `2001:db8:acad:1::1/64`.

Examples:

```text
Expanded:   2001:0db8:0000:0000:0000:0000:0000:0001
Compressed:  2001:db8::1
```

Common lab prefixes are documentation prefixes such as `2001:db8::/32`. These are safe for practice and help you focus on configuration rather than public addressing.

**Address types you should recognize**

IPv6 labs often combine several address types on the same interface or host.

- **Global unicast** addresses are routable in the IPv6 network and are used for normal end-to-end communication.
- **Link-local** addresses begin with `fe80::/10` and are used on the local link for neighbor discovery, routing protocols, and next-hop resolution.
- **Multicast** addresses replace broadcast in IPv6 and are used for discovery and control traffic.
- **Unspecified** address `::` means “no address.”
- **Loopback** address `::1` is the IPv6 equivalent of `127.0.0.1`.

> When you configure a static IPv6 route using a link-local next hop in Cisco IOS, you must specify the directly connected exit interface as well. The interface is not optional in that case.

**IPv6 address creation methods**

CCNA labs commonly use three ways to create host or interface addresses.

- **Static addressing**: You manually assign the full IPv6 address and prefix length.
- **SLAAC**: Stateless Address Autoconfiguration lets hosts build an address from router advertisements.
- **DHCPv6**: A server assigns addresses or other configuration parameters.

On Cisco routers, interface addresses can also be formed automatically from the interface MAC address using **EUI-64** or modified EUI-64 behavior. This is common in labs where you want the host portion to be generated predictably.

**EUI-64 and modified EUI-64**

EUI-64 expands a 48-bit MAC address into a 64-bit interface identifier by inserting `ff:fe` in the middle and flipping the universal/local bit. Cisco IOS can generate an IPv6 interface ID from the MAC when you use the `eui-64` keyword.

Example:

```bash
R1(config)# interface g0/0
R1(config-if)# ipv6 address 2001:db8:acad:1::/64 eui-64
R1(config-if)# no shutdown
```

This is useful when you want the router to advertise a prefix and still have a deterministic interface address without typing the full host portion manually.

**Basic IPv6 interface configuration**

A typical router interface needs IPv6 routing enabled globally and an IPv6 address on the interface.

```bash
R1(config)# ipv6 unicast-routing
R1(config)# interface g0/0
R1(config-if)# ipv6 address 2001:db8:acad:1::1/64
R1(config-if)# ipv6 address fe80::1 link-local
R1(config-if)# no shutdown
```

Key points:

- `ipv6 unicast-routing` enables IPv6 forwarding on the router.
- The global unicast address provides routed connectivity.
- The link-local address is often used as a stable next hop in routing and neighbor discovery.
- `no shutdown` is still required on many router interfaces.

**SLAAC behavior and router advertisement flags**

SLAAC depends on **Router Advertisement** messages sent by routers through IPv6 Neighbor Discovery. Hosts listen for these advertisements and build their own address using the advertised prefix.

The RA flags matter:

- **A flag**: Address Autoconfiguration. If set, the host can use SLAAC to create its own IPv6 address.
- **M flag**: Managed Address Configuration. If set, the host should use DHCPv6 for an address.
- **O flag**: Other Configuration. If set, the host should use DHCPv6 for additional information such as DNS, even if it uses SLAAC for the address.

Practical interpretation:

- `A=1, M=0, O=0` → SLAAC only.
- `A=1, M=0, O=1` → SLAAC for the address, DHCPv6 for other settings.
- `A=0, M=1` → DHCPv6 for the address.
- `A=1, M=1` is uncommon in labs and can confuse troubleshooting; follow the lab instructions carefully.

A simple conceptual flow looks like this:

```text
Router sends RA
   |
   +-- A=1 --> Host forms address from prefix + interface ID
   |
   +-- M=1 --> Host requests DHCPv6 address
   |
   +-- O=1 --> Host may request DHCPv6 options such as DNS
```

**DHCPv6 basics**

DHCPv6 can be used in two main ways:

- **Stateful DHCPv6**: The server assigns the IPv6 address.
- **Stateless DHCPv6**: The host uses SLAAC for the address and DHCPv6 for extra parameters like DNS and domain name.

A common CCNA lab pattern is to configure the router to advertise a prefix and then choose whether hosts use SLAAC or DHCPv6 based on the RA flags.

Example router configuration for stateless DHCPv6:

```bash
R1(config)# ipv6 unicast-routing
R1(config)# ipv6 dhcp pool LAB-DNS
R1(config-dhcpv6)# dns-server 2001:db8:acad::53
R1(config-dhcpv6)# domain-name example.local

R1(config)# interface g0/0
R1(config-if)# ipv6 address 2001:db8:acad:1::1/64
R1(config-if)# ipv6 nd other-config-flag
R1(config-if)# ipv6 dhcp server LAB-DNS
R1(config-if)# no shutdown
```

Example for stateful DHCPv6:

```bash
R1(config)# ipv6 unicast-routing
R1(config)# ipv6 dhcp pool STATEFUL-LAB
R1(config-dhcpv6)# address prefix 2001:db8:acad:1::/64
R1(config-dhcpv6)# dns-server 2001:db8:acad::53
R1(config-dhcpv6)# domain-name example.local

R1(config)# interface g0/0
R1(config-if)# ipv6 address 2001:db8:acad:1::1/64
R1(config-if)# ipv6 nd managed-config-flag
R1(config-if)# ipv6 dhcp server STATEFUL-LAB
R1(config-if)# no shutdown
```

**Host setup in Packet Tracer or a lab**

When you configure a host, you usually choose one of three modes:

1. **Static**: Enter the IPv6 address, prefix length, and default gateway manually.
2. **Auto Config**: Use SLAAC to obtain an address automatically.
3. **DHCPv6**: Request an address and/or options from a DHCPv6 server.

Practical host workflow:

1. Open the host’s IPv6 settings.
2. Select the correct addressing mode.
3. Verify that the host receives an address in the expected prefix.
4. Confirm the default gateway is the router’s link-local or global address, depending on the lab design.
5. Test connectivity with `ping` to the router and remote hosts.

If the host uses SLAAC, it should learn the prefix from the router advertisement and generate its own interface ID. If it uses DHCPv6, it should receive the address or options from the server after the RA indicates the correct behavior.

**Neighbor Discovery and why it matters**

IPv6 uses **Neighbor Discovery Protocol** (NDP) instead of ARP. NDP relies on several ICMPv6 message types that are important in labs and troubleshooting.

- **RS**: Router Solicitation. A host sends this to ask for router advertisements quickly.
- **RA**: Router Advertisement. A router advertises prefixes, flags, and default gateway information.
- **NS**: Neighbor Solicitation. Used to discover a neighbor’s link-layer address or verify reachability.
- **NA**: Neighbor Advertisement. Used to reply to NS and confirm the neighbor’s address.

Why this matters:

- A host may not form an address if it never receives an RA.
- A router may not appear in the neighbor table until NS/NA exchange occurs.
- Default gateway behavior in IPv6 depends on the router advertisement and link-local reachability, not on ARP.

A simple packet flow example:

```text
Host -> RS
Router -> RA with prefix and flags
Host -> NS for neighbor resolution
Router -> NA
Host -> traffic to local or remote destination
```

**Verification commands on Cisco IOS**

Use verification commands in a sequence so you can isolate whether the problem is addressing, link state, neighbor discovery, or routing.

```bash
show ipv6 interface brief
show ipv6 interface g0/0
show ipv6 interface
show ipv6 route
show ipv6 neighbors
show ipv6 neighbors detail
show running-config | section ipv6
```

What each command helps you confirm:

- `show ipv6 interface brief` shows which interfaces have IPv6 addresses and whether they are up.
- `show ipv6 interface g0/0` shows link-local, global unicast, RA behavior, and ND settings on one interface.
- `show ipv6 interface` without an interface gives a broader view of IPv6-enabled interfaces and their status.
- `show ipv6 route` confirms connected routes, local routes, and static routes.
- `show ipv6 neighbors` shows the neighbor cache and learned link-layer addresses.
- `show ipv6 neighbors detail` gives deeper inspection, which is useful when a neighbor entry is incomplete or stale.
- `show running-config | section ipv6` helps you verify the exact configuration applied.

**Host verification commands**

On a PC or Linux host, use commands that confirm the address, gateway, and neighbor state.

Windows:

```powershell
ipconfig /all
ping 2001:db8:acad:1::1
ping fe80::1
netsh interface ipv6 show neighbors
```

Linux:

```bash
ip -6 addr
ip -6 route
ping -6 2001:db8:acad:1::1
ip -6 neigh
```

Useful checks:

- Confirm the host has a global unicast address in the correct prefix.
- Confirm the default gateway is present.
- Confirm the host can reach the router’s link-local or global address.
- Confirm the neighbor cache contains the router after a ping or other traffic.

**Static IPv6 routes and default routes**

Static routes are common in CCNA labs with two or more routers. You can route to a specific prefix or create a default route.

Examples:

```bash
R1(config)# ipv6 route 2001:db8:acad:2::/64 2001:db8:acad:12::2
R1(config)# ipv6 route 2001:db8:acad:2::/64 g0/1 fe80::2
R1(config)# ipv6 route ::/0 2001:db8:acad:12::2
```

Important details:

- A static route to a **global unicast next hop** works when the next hop is reachable through a connected network or can be recursively resolved.
- A static route using a **link-local next hop** must include the directly connected exit interface.
- A default route `::/0` sends unknown destinations toward an upstream router, but the next hop still must be valid and reachable in the topology.

If you use a link-local next hop, the correct pattern is:

```bash
ipv6 route 2001:db8:acad:2::/64 g0/1 fe80::2
```

Here, `g0/1` must be the directly connected exit interface toward the neighbor that owns `fe80::2`.

**Troubleshooting workflow**

A disciplined troubleshooting sequence helps you isolate IPv6 failures quickly.

1. Check the physical and interface state with `show ipv6 interface brief`.
2. Confirm the router has `ipv6 unicast-routing` enabled.
3. Verify the interface has the correct global and link-local addresses.
4. Check whether the host is using the expected method: static, SLAAC, or DHCPv6.
5. Inspect RA flags if the host is not receiving the right configuration.
6. Review the neighbor table with `show ipv6 neighbors detail`.
7. Confirm the routing table with `show ipv6 route`.
8. Test end-to-end connectivity with `ping` and `traceroute`.

Common issues:

- **Wrong prefix length**: In CCNA labs, `/64` is the normal choice for LANs, but other prefix lengths can still work in special designs. Verify the lab requirement rather than assuming every IPv6 network must be `/64`.
- **Missing `ipv6 unicast-routing`**: The router may have addresses but still not forward IPv6 traffic.
- **Incorrect RA flags**: Hosts may use SLAAC when DHCPv6 was expected, or vice versa.
- **Wrong default gateway**: The host may point to the wrong router address or an unreachable next hop.
- **Missing exit interface with link-local static route**: The route will fail or behave unexpectedly if the interface is omitted.
- **Neighbor discovery not completing**: The host may not learn the router’s MAC address, so pings fail even though addresses look correct.

**End-to-end lab example**

A small two-router topology is a good way to practice addressing and verification.

```text
PC1 --- R1 --- R2 --- PC2
```

Example addressing plan:

- `R1 g0/0`: `2001:db8:acad:1::1/64`
- `R1 g0/1`: `2001:db8:acad:12::1/64`
- `R2 g0/0`: `2001:db8:acad:12::2/64`
- `R2 g0/1`: `2001:db8:acad:2::1/64`
- `PC1`: `2001:db8:acad:1::10/64`, gateway `2001:db8:acad:1::1`
- `PC2`: `2001:db8:acad:2::10/64`, gateway `2001:db8:acad:2::1`

Router configuration:

```bash
R1(config)# ipv6 unicast-routing
R1(config)# interface g0/0
R1(config-if)# ipv6 address 2001:db8:acad:1::1/64
R1(config-if)# no shutdown
R1(config)# interface g0/1
R1(config-if)# ipv6 address 2001:db8:acad:12::1/64
R1(config-if)# no shutdown
R1(config)# ipv6 route 2001:db8:acad:2::/64 2001:db8:acad:12::2

R2(config)# ipv6 unicast-routing
R2(config)# interface g0/0
R2(config-if)# ipv6 address 2001:db8:acad:12::2/64
R2(config-if)# no shutdown
R2(config)# interface g0/1
R2(config-if)# ipv6 address 2001:db8:acad:2::1/64
R2(config-if)# no shutdown
R2(config)# ipv6 route 2001:db8:acad:1::/64 2001:db8:acad:12::1
```

Verification sequence:

1. Confirm each interface is up with `show ipv6 interface brief`.
2. Confirm each router has connected and local routes with `show ipv6 route`.
3. Ping the directly connected neighbor from each router.
4. Ping the remote LAN address from each router.
5. Ping from PC1 to PC2.
6. If the ping fails, check the neighbor table and default gateway on the host.

**Practical host default gateway checks**

IPv6 hosts do not use ARP for gateway discovery. They rely on NDP and the router advertisement process, so a wrong gateway often appears as a neighbor discovery problem rather than a simple address typo.

Things to verify on the host:

- The host has an address in the correct subnet.
- The host has a default route learned from the router or entered manually.
- The router interface is sending RAs if SLAAC is expected.
- The host can resolve the router’s link-local address.

If the host is static, make sure you enter the gateway exactly as the lab requires. Some labs use the router’s global unicast address, while others use the link-local address.

**Quick exam reminders**

- IPv6 uses **NDP**, not ARP.
- Routers advertise prefixes with **RA** messages.
- `A`, `M`, and `O` flags control host behavior.
- `ipv6 unicast-routing` is required for routing on Cisco routers.
- Link-local next hops require the exit interface in static routes.
- `show ipv6 interface brief`, `show ipv6 neighbors detail`, and `show ipv6 route` are core verification commands.
- `/64` is the standard LAN prefix in most CCNA labs, but always follow the design requirement.

**Mini checklist for lab completion**

- Configure IPv6 addresses on all router interfaces.
- Enable IPv6 routing globally.
- Decide whether hosts use static addressing, SLAAC, or DHCPv6.
- Verify RA flags match the intended host behavior.
- Add static routes or a default route where needed.
- Test connectivity from host to router, router to router, and host to host.
- Use neighbor and route tables to isolate failures.
