## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Dual-stack overview**

Dual-stack means you run IPv4 and IPv6 at the same time on the same network devices and links. In migration projects, this is the most common approach because it lets you keep existing IPv4 services working while you introduce IPv6 gradually.

A dual-stack lab usually focuses on three things:

- Assigning both IPv4 and IPv6 addresses to router interfaces.
- Enabling IPv6 routing so the router can forward IPv6 between interfaces.
- Verifying host connectivity with both protocols and troubleshooting gateway or neighbor issues.

In Cisco IOS, an interface can have an IPv4 address and one or more IPv6 addresses at the same time. IPv6 link-local addresses are automatically created when IPv6 is enabled on an interface, and they are often used as the next hop for routing and as the default gateway on hosts.

**Why organizations use dual-stack**

Dual-stack is practical because it avoids a hard cutover. You can support legacy IPv4 applications while testing IPv6 readiness, training staff, and validating routing, DNS, and security policies.

Common reasons to keep both stacks active include:

- Legacy applications still depend on IPv4.
- Some services or third-party systems are not yet IPv6-ready.
- You want a low-risk migration path with rollback options.
- You need to validate IPv6 behavior before decommissioning IPv4.

**Typical lab topology**

A simple CCNA dual-stack lab often uses two routers and two LANs, or one router with two routed interfaces and a switch in between. The goal is to prove end-to-end IPv4 and IPv6 reachability across the router.

Example interface map:

```text
PC-A ---- SW1 ---- R1 G0/0 ---- R1 G0/1 ---- SW2 ---- PC-B

LAN 1: 192.168.10.0/24 and 2001:db8:10:1::/64
LAN 2: 192.168.20.0/24 and 2001:db8:10:2::/64
```

A useful mental model is:

- Each LAN gets one IPv4 subnet and one IPv6 prefix.
- Each router interface gets an address in both families.
- Hosts use the router as the default gateway for both IPv4 and IPv6.
- The router must be able to route IPv6 between interfaces before IPv6 traffic will pass end to end.

**IPv4 and IPv6 addressing plan**

A clean addressing plan makes verification easier. Use predictable subnets and keep the router interface addresses consistent.

Example plan:

| Device | Interface | IPv4 Address | IPv6 Address |
|---|---|---:|---|
| R1 | G0/0 | 192.168.10.1/24 | 2001:db8:10:1::1/64 |
| R1 | G0/1 | 192.168.20.1/24 | 2001:db8:10:2::1/64 |
| PC-A | NIC | 192.168.10.10/24 | 2001:db8:10:1::10/64 |
| PC-B | NIC | 192.168.20.10/24 | 2001:db8:10:2::10/64 |

For IPv6, the `/64` prefix is the standard LAN size in most CCNA labs because Stateless Address Autoconfiguration (SLAAC) depends on it. SLAAC lets hosts build their own IPv6 address from the prefix advertised by the router and their interface identifier.

**Router interface configuration**

You configure both protocol families directly on the interface. IPv4 uses `ip address`, and IPv6 uses `ipv6 address`.

Example configuration:

```cisco
conf t
ipv6 unicast-routing

interface g0/0
 description LAN 1
 ip address 192.168.10.1 255.255.255.0
 ipv6 address 2001:db8:10:1::1/64
 no shutdown

interface g0/1
 description LAN 2
 ip address 192.168.20.1 255.255.255.0
 ipv6 address 2001:db8:10:2::1/64
 no shutdown
end
```

Key points:

- **`ipv6 unicast-routing`** enables IPv6 routing on the router so it can forward IPv6 packets between interfaces.
- Without that command, the router can still have IPv6 addresses on interfaces, but it will not route IPv6 between them.
- `no shutdown` is still required if the interface is administratively down.
- The router automatically creates a link-local address on each IPv6-enabled interface unless you configure one manually.

**Host configuration**

Hosts need both an IPv4 address and an IPv6 address, plus the correct default gateway for each protocol.

Typical host settings:

- IPv4 address and subnet mask.
- IPv4 default gateway set to the router’s IPv4 interface address.
- IPv6 address and prefix length, or SLAAC if the router advertises the prefix.
- IPv6 default gateway usually learned from router advertisements and based on the router’s link-local address.

Example host configuration on PC-A:

```text
IPv4 address:      192.168.10.10
Subnet mask:       255.255.255.0
Default gateway:   192.168.10.1

IPv6 address:      2001:db8:10:1::10/64
Default gateway:    fe80::1 (or the router's actual link-local address in the lab)
```

> In practice, IPv6 hosts commonly use the router’s **link-local** address as the default gateway. Many labs and operating systems learn that gateway automatically through router advertisements, so you often do not manually enter a global IPv6 address as the gateway.

If the lab uses SLAAC, the host may not need a manually configured IPv6 address. It can generate one automatically after receiving router advertisements.

**Router advertisements and host autoconfiguration**

Router Advertisements (RAs) are ICMPv6 messages sent by routers to announce the IPv6 prefix and default gateway information. They are central to IPv6 host setup.

What RAs do:

- Tell hosts which prefix to use.
- Indicate whether hosts should use SLAAC.
- Provide the default gateway information.
- Help hosts discover on-link neighbors.

If hosts do not receive RAs, they may fail to autoconfigure IPv6 addresses or default routes. That is a common troubleshooting point in labs.

Useful checks:

- Confirm the router interface is up/up.
- Confirm IPv6 is enabled on the interface.
- Confirm `ipv6 unicast-routing` is configured if the router must forward traffic.
- Confirm the host NIC is set to obtain IPv6 automatically if the lab expects SLAAC.

**Manual IPv6 addressing and link-local next hops**

You can manually configure a link-local address on a router interface if the lab requires a stable next hop.

Example:

```cisco
conf t
interface g0/0
 ipv6 address fe80::1 link-local
 ipv6 address 2001:db8:10:1::1/64
end
```

Why this matters:

- A fixed link-local address is useful for routing protocols and static routes.
- It gives you a predictable next hop in labs.
- It avoids relying on the automatically generated link-local address, which can vary.

For static IPv6 routes, the next hop is often a link-local address plus an exit interface:

```cisco
ipv6 route 2001:db8:10:2::/64 g0/1 fe80::2
```

This is important because a link-local next hop is only meaningful on the local link, so the outgoing interface must be specified.

**Verification commands**

Verification is where you prove the lab works. Check both address assignment and forwarding.

Common IOS commands:

```cisco
show ip interface brief
show ipv6 interface brief
show running-config interface g0/0
show ipv6 route
show ip route
show ipv6 neighbors
```

What to look for:

- Interfaces are `up/up`.
- IPv4 addresses are correct.
- IPv6 global and link-local addresses are present.
- The IPv6 routing table contains connected routes for each LAN.
- Neighbor discovery entries appear after traffic is generated.

Example output clues:

- `show ipv6 interface brief` should show both the global address and the link-local address.
- `show ipv6 route` should include `C` for connected and `L` for local routes.
- `show ipv6 neighbors` should show learned neighbors after pings or other traffic.

**Testing connectivity**

Test in layers so you can isolate failures quickly.

Recommended order:

1. Ping the local interface from the router.
2. Ping the host from the router on the same LAN.
3. Ping the router from the host.
4. Ping across the router between LANs using IPv4.
5. Repeat the cross-LAN test using IPv6.

Example tests:

```cisco
ping 192.168.10.10
ping 2001:db8:10:1::10
ping 192.168.20.10
ping 2001:db8:10:2::10
```

For IPv6 link-local testing, use the correct platform-specific syntax. On many Cisco IOS devices, you test a link-local neighbor by specifying the outgoing interface in the extended ping workflow or by using a supported extended command form. The exact syntax can vary by platform, so verify it in your lab environment.

**Troubleshooting common failures**

Dual-stack labs fail for a small set of predictable reasons. Start with the interface state, then move to addressing, then routing, then host configuration.

Common issues:

- **IPv6 routing not enabled**
  - The router has IPv6 addresses, but it does not forward IPv6 between interfaces.
  - Fix: configure `ipv6 unicast-routing`.

- **Wrong default gateway on the host**
  - IPv4 gateway points to the wrong router address.
  - IPv6 gateway is manually set incorrectly or does not match the router’s link-local behavior.
  - Fix: verify the host gateway for each protocol.

- **Missing or incorrect prefix length**
  - IPv6 host or interface is not using `/64` when the lab expects SLAAC.
  - Fix: correct the prefix length and reapply the address.

- **Interface is down**
  - The port is administratively shut or the cable/link is missing.
  - Fix: use `no shutdown` and verify physical connectivity.

- **No router advertisements**
  - Hosts do not autoconfigure IPv6 addresses or default routes.
  - Fix: confirm the router interface is up and IPv6-enabled, and that the host is set to auto-configure if required.

- **Link-local next-hop confusion**
  - A static IPv6 route uses a link-local next hop without an exit interface.
  - Fix: specify both the interface and the link-local address.

- **ACL or firewall interference**
  - ICMPv6 can be blocked, which breaks neighbor discovery and ping tests.
  - Fix: ensure ICMPv6 is permitted where needed.

**Practical troubleshooting workflow**

Use a repeatable sequence so you do not guess.

1. Check interface status with `show ip interface brief` and `show ipv6 interface brief`.
2. Confirm the router has the expected IPv4 and IPv6 addresses.
3. Verify `ipv6 unicast-routing` is present if routing is required.
4. Check the host’s IPv4 and IPv6 settings, including gateways.
5. Ping the local gateway from the host.
6. Ping the remote host from the router.
7. Inspect `show ipv6 neighbors` and `show ipv6 route` for missing entries.

**Dual-stack routing table behavior**

IPv4 and IPv6 maintain separate forwarding tables. You must verify both.

Important distinctions:

- `show ip route` displays IPv4 routes.
- `show ipv6 route` displays IPv6 routes.
- A working IPv4 route does not imply IPv6 is working.
- A connected IPv6 interface does not imply the router can forward IPv6 unless IPv6 routing is enabled.

This separation is one reason dual-stack is useful during migration: you can validate IPv6 independently while keeping IPv4 stable.

**Real-world deployment nuance**

In production, dual-stack is often a transition strategy rather than a permanent end state. It gives you time to update DNS, security policy, monitoring, and application dependencies without breaking existing users.

Operational considerations include:

- You must manage two address families in routing, ACLs, and troubleshooting.
- Logging and monitoring tools need IPv6 visibility.
- Security policies should explicitly account for ICMPv6, neighbor discovery, and router advertisements.
- Address planning matters more because IPv6 subnets are larger and more hierarchical.

**Quick lab checklist**

Use this checklist before you declare the lab complete:

- Router interfaces have both IPv4 and IPv6 addresses.
- `ipv6 unicast-routing` is configured on the router.
- Interfaces are `up/up`.
- Hosts have correct IPv4 addresses and gateways.
- Hosts have correct IPv6 addresses or SLAAC results.
- IPv6 default gateway behavior matches the lab design.
- You can ping within each LAN and across the router using both protocols.
- `show ipv6 route` and `show ipv6 neighbors` show expected entries.

**Reference configuration example**

A compact end-to-end example for a single router with two LANs:

```cisco
conf t
ipv6 unicast-routing

interface g0/0
 description To LAN 1
 ip address 192.168.10.1 255.255.255.0
 ipv6 address 2001:db8:10:1::1/64
 ipv6 address fe80::1 link-local
 no shutdown

interface g0/1
 description To LAN 2
 ip address 192.168.20.1 255.255.255.0
 ipv6 address 2001:db8:10:2::1/64
 ipv6 address fe80::2 link-local
 no shutdown
end
```

This design gives you predictable IPv6 link-local addresses for lab verification and static routing, while still supporting standard global unicast addressing on each LAN.

**Key takeaways**

Dual-stack configuration is about parallel operation, not replacement. You configure both protocol families, verify each independently, and troubleshoot them separately. In IPv6, pay special attention to router advertisements, link-local next hops, and the fact that routing must be explicitly enabled on Cisco IOS routers before IPv6 can pass between interfaces.
