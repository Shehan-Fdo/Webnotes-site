**CIDR and why it replaced classful addressing**

<abbr title="Classless Inter-Domain Routing">CIDR</abbr> lets you describe networks with a prefix length such as `/24` instead of relying on fixed class boundaries. It replaced classful addressing because class A, B, and C networks wasted address space and forced rigid network sizes that did not match real-world needs.

Classful addressing assumed:
- Class A used `/8`
- Class B used `/16`
- Class C used `/24`

That model breaks down when you need:
- Smaller networks for branch offices, VLANs, or point-to-point links
- Larger contiguous blocks for aggregation
- Efficient allocation of public and private IPv4 space

CIDR makes routing more efficient because routers can advertise and match prefixes of any length. It also enables route summarization, which reduces routing table size and improves convergence behavior.

**Classful vs classless addressing**

Classful routing uses the default mask implied by the first octet. Classless routing carries the prefix length with the route, so the mask is part of the route information.

- **Classful**:
  - Network boundaries are fixed by address class
  - Subnet masks are assumed, not always carried
  - Inefficient for modern networks
- **Classless**:
  - Prefix length is explicit
  - Supports VLSM and summarization
  - Works with modern routing protocols and IPv4 design

A classful network like `192.168.10.0` would historically be treated as a Class C network with `/24`. Under CIDR, you can also use `192.168.10.0/26`, `192.168.10.64/26`, or any other valid prefix that fits the design.

> CIDR is not just a notation change. It is the foundation for efficient subnetting, route aggregation, and flexible address planning.

**Subnet masks and prefix lengths**

A subnet mask separates the network portion from the host portion of an IPv4 address. In binary, the mask contains contiguous `1` bits followed by contiguous `0` bits.

- `1` bits identify the network prefix
- `0` bits identify host addresses
- Prefix length is the count of `1` bits in the mask

Common examples:
- `/24` = `255.255.255.0`
- `/25` = `255.255.255.128`
- `/26` = `255.255.255.192`
- `/27` = `255.255.255.224`
- `/28` = `255.255.255.240`
- `/29` = `255.255.255.248`
- `/30` = `255.255.255.252`
- `/31` = `255.255.255.254`
- `/32` = `255.255.255.255`

**Binary, decimal, and prefix conversion**

You should be able to move between all three forms quickly.

| Prefix | Mask | Usable Hosts | Block Size |
|---|---:|---:|---:|
| `/24` | `255.255.255.0` | 254 | 256 |
| `/25` | `255.255.255.128` | 126 | 128 |
| `/26` | `255.255.255.192` | 62 | 64 |
| `/27` | `255.255.255.224` | 30 | 32 |
| `/28` | `255.255.255.240` | 14 | 16 |
| `/29` | `255.255.255.248` | 6 | 8 |
| `/30` | `255.255.255.252` | 2 | 4 |
| `/31` | `255.255.255.254` | 2 addresses, point-to-point use | 2 |
| `/32` | `255.255.255.255` | 1 host route | 1 |

The block size is the number of addresses in each subnet. You can calculate it as `256 - mask value` in the interesting octet.

Examples:
- `255.255.255.192` → `256 - 192 = 64`
- `255.255.255.240` → `256 - 240 = 16`
- `255.255.255.252` → `256 - 252 = 4`

**How to determine usable hosts**

For most subnets, usable hosts are:

`2^(host bits) - 2`

Examples:
- `/24` has 8 host bits → `2^8 - 2 = 254`
- `/26` has 6 host bits → `2^6 - 2 = 62`
- `/30` has 2 host bits → `2^2 - 2 = 2`

The subtraction of 2 accounts for:
- Network address
- Broadcast address

Special cases:
- `/31` is used on point-to-point links and does not reserve network and broadcast in the traditional way
- `/32` identifies a single host route, often used for loopbacks, route entries, and management reachability

**How to determine the number of subnets created**

When you borrow bits from the host portion, you increase the number of subnets.

Formula:
- `2^(borrowed bits) = number of subnets`

Example:
- Starting from `/24` and moving to `/27`
- Borrowed bits = `27 - 24 = 3`
- Number of subnets = `2^3 = 8`

Those eight `/27` subnets inside a `/24` are:
- `192.168.1.0/27`
- `192.168.1.32/27`
- `192.168.1.64/27`
- `192.168.1.96/27`
- `192.168.1.128/27`
- `192.168.1.160/27`
- `192.168.1.192/27`
- `192.168.1.224/27`

**Finding subnet ranges quickly**

A subnet range has:
- Network address
- First usable host
- Last usable host
- Broadcast address

Use the block size to find boundaries.

Example for `192.168.10.0/26`:
- Block size = `64`
- Subnets start at:
  - `0`
  - `64`
  - `128`
  - `192`

So the first subnet is:
- Network: `192.168.10.0`
- First host: `192.168.10.1`
- Last host: `192.168.10.62`
- Broadcast: `192.168.10.63`

The next subnet is:
- Network: `192.168.10.64`
- First host: `192.168.10.65`
- Last host: `192.168.10.126`
- Broadcast: `192.168.10.127`

**Practical conversion examples**

Convert `/20` to dotted decimal:
- `/20` means 20 network bits
- First two octets are full `255`
- Third octet has 4 network bits: `11110000` = `240`
- Mask = `255.255.240.0`

Convert `255.255.248.0` to prefix:
- `255` = 8 bits
- `255` = 8 bits
- `248` = 5 bits
- `0` = 0 bits
- Prefix = `/21`

Convert `255.255.255.224` to hosts:
- `/27`
- Host bits = 5
- Usable hosts = `2^5 - 2 = 30`

**Binary practice for exam readiness**

You should be comfortable with the common mask octet values:

- `128` = `10000000`
- `192` = `11000000`
- `224` = `11100000`
- `240` = `11110000`
- `248` = `11111000`
- `252` = `11111100`
- `254` = `11111110`

A quick mental method:
- Count the number of leading `1`s in the interesting octet
- Add full `255` octets before it
- Add trailing `0` octets after it

Example:
- `255.255.255.240`
- Three full octets = 24 bits
- `240` adds 4 bits
- Total = `/28`

**VLSM: Variable Length Subnet Mask**

<abbr title="Variable Length Subnet Mask">VLSM</abbr> lets you use different subnet sizes inside the same major network. This avoids wasting addresses by assigning each subnet only as many addresses as it needs.

VLSM is useful when:
- Different VLANs need different host counts
- You want to conserve IPv4 space
- You need a hierarchical design with room for growth
- You plan to summarize routes later

Without VLSM, you might force every subnet to be the same size, which wastes addresses on small segments.

**Why VLSM matters in real networks**

A campus may need:
- A user VLAN with 120 hosts
- A voice VLAN with 50 hosts
- A printer VLAN with 20 hosts
- A point-to-point WAN link with 2 hosts

If you used `/25` everywhere, the WAN link would waste most of the subnet. VLSM lets you assign:
- `/25` for the user VLAN
- `/26` for the voice VLAN
- `/27` for the printer VLAN
- `/30` or `/31` for the WAN link

**Worked VLSM planning example**

You receive `10.10.0.0/24` and need:
- Sales: 100 hosts
- Engineering: 50 hosts
- Voice: 20 hosts
- WAN link: 2 hosts

Step 1: Sort requirements from largest to smallest.
1. Sales: 100
2. Engineering: 50
3. Voice: 20
4. WAN: 2

Step 2: Choose the smallest subnet that fits each requirement.
- 100 hosts → `/25` gives 126 usable
- 50 hosts → `/26` gives 62 usable
- 20 hosts → `/27` gives 30 usable
- 2 hosts → `/30` gives 2 usable

Step 3: Allocate from the start of the block.

```text
10.10.0.0/24
├─ Sales        10.10.0.0/25    usable 10.10.0.1 - 10.10.0.126   broadcast 10.10.0.127
├─ Engineering  10.10.0.128/26  usable 10.10.0.129 - 10.10.0.190 broadcast 10.10.0.191
├─ Voice        10.10.0.192/27  usable 10.10.0.193 - 10.10.0.222 broadcast 10.10.0.223
└─ WAN          10.10.0.224/30  usable 10.10.0.225 - 10.10.0.226 broadcast 10.10.0.227
```

Step 4: Verify that the remaining space is unused but available for future growth.
- Remaining addresses: `10.10.0.228 - 10.10.0.255`
- This space can support another small subnet later

This is the standard VLSM workflow:
1. List host requirements
2. Sort from largest to smallest
3. Pick the smallest fitting prefix
4. Assign subnets sequentially
5. Verify boundaries and unused space

**VLSM verification tips**

Check each subnet for:
- Correct network boundary
- No overlap with adjacent subnets
- Enough usable hosts
- Proper broadcast address
- Room for future expansion if required

A common exam mistake is assigning a subnet that starts on the wrong boundary. For example, a `/26` must start on multiples of 64 in the last octet: `0`, `64`, `128`, or `192`.

**Route summarization**

Route summarization, also called route aggregation, combines multiple contiguous networks into a single shorter prefix. This reduces routing table size and can improve stability.

Summarization works best when:
- Networks are contiguous
- The prefixes are the same size
- The summary boundary aligns with binary structure

Example:
- `192.168.8.0/24`
- `192.168.9.0/24`
- `192.168.10.0/24`
- `192.168.11.0/24`

These four `/24` networks can be summarized as:
- `192.168.8.0/22`

Why:
- Four contiguous `/24`s equal one `/22`
- A `/22` covers `192.168.8.0` through `192.168.11.255`

**Step-by-step method to find a summary prefix**

Use binary comparison to find the common leading bits.

Example: summarize `172.16.8.0/24` through `172.16.11.0/24`

1. Write the third octet in binary:
   - `8`  = `00001000`
   - `9`  = `00001001`
   - `10` = `00001010`
   - `11` = `00001011`

2. Compare the bits from left to right:
   - Common bits: `000010`
   - First differing bit occurs after 6 bits

3. Add the common bits to the first two octets:
   - `172.16` = 16 bits
   - Plus 6 common bits in the third octet
   - Summary prefix = `/22`

4. Zero out the remaining host bits:
   - Summary network = `172.16.8.0/22`

5. Confirm the range:
   - `172.16.8.0` to `172.16.11.255`

A quick shortcut:
- Find the largest power-of-two block that starts on a valid boundary and covers all networks
- The summary must not include unrelated networks

**Another summarization example with binary reasoning**

Summarize:
- `10.20.4.0/24`
- `10.20.5.0/24`
- `10.20.6.0/24`
- `10.20.7.0/24`

Third octet values:
- `4` = `00000100`
- `5` = `00000101`
- `6` = `00000110`
- `7` = `00000111`

Common leading bits in the third octet:
- `000001`
- That gives 6 shared bits beyond the first 16 bits
- Summary prefix = `/22`

Summary:
- `10.20.4.0/22`

**Important summarization rules**

- Summarize only contiguous blocks that align on the correct boundary
- Do not summarize across gaps
- Do not summarize if it would include unrelated routes
- Use the most specific summary that covers all target networks

**Special subnet cases: /31 and /32**

`/31` and `/32` are important exam topics because they behave differently from normal subnets.

- **/31**
  - Used on point-to-point links
  - Provides two addresses with no traditional network or broadcast address
  - Saves address space on links that need only two endpoints
  - Common in modern routed point-to-point designs

- **/32**
  - Represents a single host route
  - Often used for loopback interfaces
  - Useful for router IDs, management, and stable reachability
  - Can appear in routing tables as a specific destination

Example loopback configuration:
```cisco
interface loopback0
 ip address 10.255.255.1 255.255.255.255
```

Example point-to-point link using `/31`:
```cisco
interface gigabitEthernet0/0
 ip address 192.0.2.0 255.255.255.254
 no shutdown
```

> Support for `/31` depends on platform and protocol behavior, but it is widely used on modern routers for point-to-point links.

**Real-world design scenario tying it all together**

Imagine a branch office gets `10.50.0.0/22` and needs:
- Admin VLAN: 200 hosts
- Staff VLAN: 100 hosts
- Guest VLAN: 50 hosts
- Voice VLAN: 25 hosts
- Router-to-router link: 2 hosts

You can design it like this:
- Admin → `/24` for 254 usable
- Staff → `/25` for 126 usable
- Guest → `/26` for 62 usable
- Voice → `/27` for 30 usable
- WAN link → `/31` or `/30` depending on design and support

This approach:
- Conserves addresses
- Leaves room for growth
- Makes summarization easier at the edge
- Keeps routing tables smaller

**CLI verification commands**

Use router and switch commands to confirm subnetting and addressing.

```cisco
show ip interface brief
show running-config interface gigabitEthernet0/0
show ip route
show ip route summary
show interfaces
```

Useful checks:
- Confirm the interface IP and mask
- Verify the route appears in the routing table
- Check whether the prefix length matches the design
- Confirm connected routes and summaries

On a Cisco device, you may configure an interface like this:
```cisco
interface vlan 10
 ip address 10.10.0.1 255.255.255.128
 no shutdown
```

**Common exam traps**

- Confusing the subnet mask with the default classful mask
- Forgetting that usable hosts are `2^n - 2` for normal subnets
- Miscounting subnet boundaries when the block size is not obvious
- Summarizing networks that are not contiguous
- Using `/30` when `/31` is more appropriate for point-to-point links
- Forgetting that `/32` is a host route, not a normal LAN subnet
- Assigning VLSM subnets out of order, which can cause overlap

**Fast subnetting checklist**

1. Identify the prefix or mask.
2. Convert to block size if needed.
3. Calculate usable hosts.
4. Determine subnet boundaries.
5. For VLSM, sort requirements largest to smallest.
6. Assign the largest subnet first.
7. Verify no overlap.
8. Summarize only when prefixes align cleanly.

**Quick practice examples**

- `255.255.255.224` = `/27`
- `/29` = 6 usable hosts
- `/26` creates 4 subnets inside a `/24`
- `192.168.1.128/25` covers `192.168.1.128` to `192.168.1.255`
- `10.0.0.0/30` gives 2 usable hosts
- `10.0.0.1/32` is a single host route

**Mental model for subnet planning**

Think of IPv4 space as a set of nested blocks:
- CIDR defines the size of the block
- Subnet masks define where the block starts and ends
- VLSM lets you choose different block sizes for different needs
- Summarization combines adjacent blocks into a larger block

When you understand those relationships, subnetting becomes a design tool rather than a memorization task.
