## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**IPv6 notation and prefix basics**

IPv6 uses 128-bit addresses written as eight 16-bit hexadecimal groups separated by colons. Each group is called a hextet. You shorten IPv6 addresses by removing leading zeros in each hextet and by replacing one contiguous run of all-zero hextets with `::` once per address.

A prefix length tells you how many bits identify the network portion. In IPv6, `/64` is the standard LAN prefix for <abbr title="Stateless Address Autoconfiguration">SLAAC</abbr> because the first 64 bits are the prefix and the last 64 bits are available for the interface ID.

- `2001:db8:acad:1::/64` means:
  - `2001:db8:acad:1` is the network prefix
  - the remaining 64 bits identify the host/interface
- `/48` and `/56` are common allocation sizes from providers or enterprises
- `/128` identifies a single IPv6 host address
- `/64` is the default expectation for most Ethernet LANs and is the exam-safe answer for SLAAC

A practical way to think about IPv6 subnetting is that the prefix length determines where the network ends and the interface ID begins. If the prefix is not `/64`, SLAAC generally does not work in the normal way on Ethernet because hosts expect a 64-bit interface ID for address formation.

```text
2001:db8:acad:1:1234:5678:9abc:def0
|<------ 64-bit prefix ------>|<------ 64-bit interface ID ------>|
```

**Address types you must recognize**

IPv6 uses several address types, and the type often determines how traffic is delivered.

- **Unicast**: one-to-one communication
- **Multicast**: one-to-many communication
- **Anycast**: one-to-nearest communication using the same address on multiple devices

IPv6 does not use broadcast. Instead, it relies heavily on multicast.

Core multicast addresses to know:

- `ff02::1` = **all-nodes multicast**
  - every IPv6-enabled node on the local link joins this group
- `ff02::2` = **all-routers multicast**
  - every IPv6 router on the local link joins this group
- `ff02::1:ff00:0/104` = **solicited-node multicast range**
  - used by Neighbor Discovery for address resolution and duplicate address detection

These multicast groups matter because IPv6 control traffic is targeted more precisely than IPv4 broadcast traffic. That reduces unnecessary processing on the LAN.

**Link-local addresses and why they matter**

A link-local address is an IPv6 address in `fe80::/10`. It is valid only on the local link and is not routed. On Cisco IOS, when you enable IPv6 on an interface, the router automatically creates a link-local address if one is not manually configured.

Link-local addresses are essential because routers and hosts use them for local control-plane functions:

- routers advertise themselves in Router Advertisements
- hosts learn the default gateway from the router’s link-local address
- Neighbor Discovery messages often use link-local source addresses
- routing protocols in IPv6 commonly use link-local next hops

> On Cisco devices, you usually do not need to manually configure a link-local address unless you want a predictable value for troubleshooting or routing protocol design.

Example:

```cisco
interface g0/0
 ipv6 address 2001:db8:acad:1::1/64
 ipv6 enable
```

If you configure only `ipv6 enable`, the interface still gets a link-local address automatically.

**SLAAC: Stateless Address Autoconfiguration**

SLAAC lets a host build its own IPv6 address without a DHCP server assigning the address. The router advertises the prefix, and the host combines that prefix with a locally generated interface ID.

The process is:

1. The router sends a Router Advertisement (RA).
2. The RA includes the prefix and flags that tell the host whether SLAAC is allowed.
3. The host forms an address using the advertised prefix plus an interface ID.
4. The host performs Duplicate Address Detection (DAD) before using the address.

The key idea is that the router provides the network prefix, but the host creates the full address.

Example:

- Advertised prefix: `2001:db8:acad:1::/64`
- Host interface ID: `0211:22ff:fe33:4455`
- Resulting address: `2001:db8:acad:1:0211:22ff:fe33:4455`

The interface ID may come from:
- a modified EUI-64 process based on the MAC address
- a randomly generated stable privacy address
- an operating-system-specific method

For CCNA, the important point is that the host forms the address locally after receiving the prefix.

**Router Advertisement flags and what they mean**

RA messages include flags that influence host behavior. The two most important are the <abbr title="Managed Address Configuration">M</abbr> and <abbr title="Other Configuration">O</abbr> flags.

| Flag | Meaning | Typical effect |
|---|---|---|
| M | Managed address configuration | Use DHCPv6 for address assignment |
| O | Other configuration | Use DHCPv6 for additional information such as DNS |

The prefix advertisement also includes the **A flag** in the Prefix Information Option:

- **A flag set** means the prefix is autonomous and can be used for SLAAC address formation
- the host still combines the prefix with a host-generated interface ID
- the A flag does not mean the router assigns the full address

This distinction is important for exam wording.

A common RA scenario:

- `A=1`, `M=0`, `O=0`
  - SLAAC only
- `A=1`, `M=0`, `O=1`
  - SLAAC for address plus DHCPv6 for other info
- `A=0`, `M=1`, `O=0`
  - stateful DHCPv6 for address assignment
- `A=1`, `M=1`, `O=1`
  - mixed behavior; client behavior depends on OS and configuration

**DHCPv6: stateful, stateless, and relay**

DHCPv6 is the IPv6 version of DHCP. It can provide addresses, DNS information, and other configuration data. The behavior depends on whether it is stateful or stateless.

**Stateful DHCPv6**

In stateful DHCPv6, the server assigns an IPv6 address from a managed pool and keeps track of the lease. The client does not build its own address from SLAAC for that interface.

Use stateful DHCPv6 when you want centralized address management and tracking.

Typical characteristics:

- server provides the IPv6 address
- server can also provide DNS and domain information
- RA usually has the M flag set
- the client uses DHCPv6 to obtain the address

Cisco IOS example:

```cisco
ipv6 unicast-routing

ipv6 dhcp pool STATEFUL-POOL
 address prefix 2001:db8:acad:1::/64
 dns-server 2001:db8:acad:1::53
 domain-name example.local

interface g0/0
 ipv6 address 2001:db8:acad:1::1/64
 ipv6 nd managed-config-flag
 ipv6 dhcp server STATEFUL-POOL
```

Notes:
- `ipv6 nd managed-config-flag` sets the M flag in RA messages
- `ipv6 dhcp server STATEFUL-POOL` attaches the DHCPv6 pool to the interface
- the router must also be enabled for IPv6 routing with `ipv6 unicast-routing`

**Stateless DHCPv6**

In stateless DHCPv6, the host uses SLAAC to create its IPv6 address, but it uses DHCPv6 to learn additional settings such as DNS servers and the domain name.

Typical characteristics:

- SLAAC provides the address
- DHCPv6 provides “other” configuration
- RA usually has the O flag set
- the host still learns the default gateway from RA, not DHCPv6

Cisco IOS example:

```cisco
ipv6 unicast-routing

ipv6 dhcp pool STATELESS-POOL
 dns-server 2001:db8:acad:1::53
 domain-name example.local

interface g0/0
 ipv6 address 2001:db8:acad:1::1/64
 ipv6 nd other-config-flag
 ipv6 dhcp server STATELESS-POOL
```

**DHCPv6 relay**

If the DHCPv6 server is not on the local link, the router can relay DHCPv6 messages to a remote server.

Cisco IOS relay example:

```cisco
interface g0/0
 ipv6 address 2001:db8:acad:1::1/64
 ipv6 nd managed-config-flag
 ipv6 dhcp relay destination 2001:db8:ffff::10 g0/1
```

Relay behavior matters when:
- clients are on one subnet
- the DHCPv6 server is centralized elsewhere
- you want one server to serve multiple VLANs or subnets

The relay agent forwards client messages to the server and returns replies back to the client. This is common in enterprise designs.

**SLAAC vs stateful DHCPv6 vs stateless DHCPv6**

| Feature | SLAAC | Stateful DHCPv6 | Stateless DHCPv6 |
|---|---|---|---|
| IPv6 address | Host creates it | Server assigns it | Host creates it |
| Default gateway | Learned from RA | Learned from RA | Learned from RA |
| DNS server | Usually not provided by SLAAC alone | Can be provided by DHCPv6 | Provided by DHCPv6 |
| RA flags | A=1 | M=1 | O=1 |
| Address tracking | No server lease tracking | Yes | No address lease tracking |
| Common use | Simple LANs | Centralized control | Address autoconfig plus DNS |

A useful exam shortcut:

- if the question asks who provides the address, think **SLAAC** or **stateful DHCPv6**
- if the question asks who provides DNS only, think **stateless DHCPv6**
- if the question asks about the default gateway, think **RA and the router’s link-local address**

**Neighbor Discovery Protocol**

<abbr title="Neighbor Discovery Protocol">NDP</abbr> replaces several IPv4 functions such as ARP, router discovery, and parts of address autoconfiguration. It uses ICMPv6 messages.

Main NDP messages:

- **RS** = Router Solicitation
- **RA** = Router Advertisement
- **NS** = Neighbor Solicitation
- **NA** = Neighbor Advertisement
- **Redirect** = informs a host of a better next hop

NDP is central to IPv6 operation because it handles:
- default gateway discovery
- address resolution
- duplicate address detection
- neighbor reachability

**RS and RA behavior**

A host sends an RS when it wants routers to advertise quickly. The RS is typically sent from the host’s link-local address to the all-routers multicast address `ff02::2`.

The router replies with an RA, usually sourced from the router’s link-local address. The RA is often sent to:
- the all-nodes multicast address `ff02::1`
- or directly to the host that sent the RS

The RA tells the host:
- which prefix to use
- whether SLAAC is allowed
- whether DHCPv6 should be used
- the default gateway information

The default gateway in IPv6 is typically the router’s **link-local address**, not a global unicast address. This is why link-local addresses are so important.

**NS and NA behavior**

Neighbor Solicitation is used for:
- address resolution
- duplicate address detection
- neighbor reachability checks

The host sends an NS to the **solicited-node multicast address** derived from the target IPv6 address. The solicited-node multicast address uses the last 24 bits of the target address.

Example:

- Target unicast address: `2001:db8:acad:1::1234:5678`
- Last 24 bits: `34:56:78`
- Solicited-node multicast: `ff02::1:ff34:5678`

This multicast group is efficient because only nodes that might own the target address listen to that group. More than one node can join the same solicited-node multicast address if their last 24 bits match, but only the node that owns the target address should respond.

Neighbor Advertisement is the reply to NS. It confirms the mapping between an IPv6 address and a link-layer address.

**Duplicate Address Detection**

DAD checks whether an IPv6 address is already in use before a host starts using it.

The process is:

1. The host configures a tentative address.
2. The host sends an NS for that tentative address.
3. The NS is sent to the solicited-node multicast address derived from the tentative address.
4. If no NA is received, the address is considered unique.
5. If a response indicates a duplicate, the host must not use the address.

DAD is important for SLAAC because the host generates its own address and must verify uniqueness.

**Packet flow examples**

A typical SLAAC flow looks like this:

```text
Host -> RS to ff02::2
Router -> RA from link-local address to ff02::1
Host -> forms global address from prefix + interface ID
Host -> DAD using NS to solicited-node multicast
Host -> uses address if no duplicate is detected
```

A typical stateless DHCPv6 flow looks like this:

```text
Host -> RS/RA exchange
Host -> forms IPv6 address using SLAAC
Host -> DHCPv6 Information-Request for DNS/domain
Server -> replies with DNS/domain options
```

A typical stateful DHCPv6 flow looks like this:

```text
Host -> RS/RA exchange
Host -> DHCPv6 Solicit/Advertise/Request/Reply
Server -> assigns address from managed pool
Host -> uses DHCPv6-provided address and RA-learned gateway
```

**Cisco IOS verification commands**

Use these commands to verify IPv6 configuration and DHCPv6 behavior:

```cisco
show ipv6 interface brief
show ipv6 interface g0/0
show ipv6 neighbors
show ipv6 route
show ipv6 dhcp pool
show ipv6 dhcp binding
show ipv6 dhcp interface
```

What each command is useful for:

- `show ipv6 interface brief`
  - quick view of IPv6-enabled interfaces and addresses
- `show ipv6 interface g0/0`
  - detailed interface IPv6 status, including RA-related settings
- `show ipv6 neighbors`
  - neighbor cache entries learned through NDP
- `show ipv6 route`
  - IPv6 routing table, including connected and local routes
- `show ipv6 dhcp pool`
  - pool parameters and statistics
- `show ipv6 dhcp binding`
  - active DHCPv6 bindings
- `show ipv6 dhcp interface`
  - interface-level DHCPv6 information; output varies by platform and role

**Troubleshooting scenarios**

If a host does not get an IPv6 address, check the following:

1. Confirm IPv6 is enabled on the router interface.
2. Confirm the interface has a valid `/64` prefix for SLAAC.
3. Check whether the correct RA flags are set.
4. Verify the host is receiving multicast traffic on the LAN.
5. Confirm DAD is not failing because of a duplicate address.

If DHCPv6 is not working:

1. Verify `ipv6 unicast-routing` is enabled.
2. Check that the DHCPv6 pool exists and is attached to the interface.
3. Confirm the correct M or O flag is configured.
4. If the server is remote, verify relay configuration.
5. Use `show ipv6 dhcp binding` to confirm leases.

If the host has an address but no internet access:

- verify the default gateway learned from RA
- confirm the router has a route to upstream networks
- check DNS separately if name resolution fails
- remember that DHCPv6 does not provide the default gateway in normal CCNA scenarios

**Practical configuration example**

This example shows a router providing SLAAC plus stateless DHCPv6 for DNS:

```cisco
ipv6 unicast-routing

ipv6 dhcp pool LAN-DNS
 dns-server 2001:db8:acad:1::53
 domain-name lab.local

interface g0/0
 description LAN
 ipv6 address 2001:db8:acad:1::1/64
 ipv6 nd other-config-flag
 ipv6 dhcp server LAN-DNS
 no shutdown
```

In this design:
- the router advertises the prefix `2001:db8:acad:1::/64`
- hosts create their own global addresses
- hosts learn DNS and domain information from DHCPv6
- hosts learn the default gateway from the router’s RA

**Exam reminders**

- IPv6 uses **multicast**, not broadcast
- SLAAC needs a **/64** prefix in normal LAN designs
- The router’s **link-local address** is usually the default gateway
- The **A flag** allows SLAAC; it does not mean the router assigns the full address
- The **M flag** points to stateful DHCPv6
- The **O flag** points to stateless DHCPv6 for extra options
- `ff02::1` is all-nodes, `ff02::2` is all-routers
- `show ipv6 dhcp binding` shows leases; `show ipv6 dhcp pool` shows pool details and statistics

**Mental model for the exam**

Think of IPv6 host setup as three separate jobs:

- **Address creation**
  - done by SLAAC or stateful DHCPv6
- **Gateway discovery**
  - done by RA through NDP
- **Extra configuration**
  - often done by stateless DHCPv6

If you keep those jobs separate, the RA flags and DHCPv6 roles become much easier to interpret.
