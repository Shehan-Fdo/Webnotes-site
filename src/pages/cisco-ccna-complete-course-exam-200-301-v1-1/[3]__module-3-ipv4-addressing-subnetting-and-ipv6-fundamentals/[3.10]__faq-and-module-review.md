## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**What this review is for**

This module review ties together the CCNA topics that commonly appear across the exam: switching, routing, IPv4 and IPv6, wireless, security, and troubleshooting. The goal is not just to memorize commands, but to recognize how features interact in a real network and how to verify them quickly.

A strong CCNA candidate can explain what should happen, predict what will fail, and confirm the result with show commands. That means you should know both configuration patterns and the outputs that prove they worked.

**Core switching review**

VLANs segment a Layer 2 network into separate broadcast domains. You use them to isolate users, reduce unnecessary flooding, and support different policy requirements on the same physical switch infrastructure.

- **Access ports** carry traffic for one VLAN.
- **Trunk ports** carry multiple VLANs using IEEE 802.1Q tagging.
- **Native VLAN** traffic is untagged on an 802.1Q trunk; both ends should match.
- **Voice VLAN** separates IP phone traffic from data traffic on the same access port.

A common exam scenario is a host that can reach devices in its own VLAN but not another VLAN. That usually means the switchport is correct, but Layer 3 routing between VLANs is missing or misconfigured.

```bash
show vlan brief
show interfaces trunk
show interfaces switchport
```

Typical checks:
- Confirm the access port is in the expected VLAN.
- Confirm the trunk is up and allowing the required VLANs.
- Confirm the native VLAN matches on both ends.
- Confirm the destination VLAN exists on every switch that must carry it.

**Inter-VLAN routing**

Inter-VLAN routing lets hosts in different VLANs communicate through a Layer 3 device. CCNA commonly expects you to understand two designs:

- **Router-on-a-stick**: one router interface uses subinterfaces, each with an 802.1Q tag.
- **Switched Virtual Interface (SVI)**: a Layer 3 switch routes between VLANs using virtual interfaces.

Router-on-a-stick is common in smaller designs and labs. SVIs are more efficient on multilayer switches because routing happens on the switch itself.

```bash
interface g0/0
 no shutdown

interface g0/0.10
 encapsulation dot1Q 10
 ip address 192.168.10.1 255.255.255.0

interface g0/0.20
 encapsulation dot1Q 20
 ip address 192.168.20.1 255.255.255.0
```

For an SVI-based design:

```bash
ip routing

interface vlan 10
 ip address 192.168.10.1 255.255.255.0
 no shutdown

interface vlan 20
 ip address 192.168.20.1 255.255.255.0
 no shutdown
```

Verification commands:
- `show ip interface brief`
- `show running-config interface vlan 10`
- `show ip route`
- `show interfaces trunk`

> If an SVI stays down/down, the VLAN may not exist, no active port may be in that VLAN, or the SVI may be administratively shut.

**Spanning Tree Protocol review**

<abbr title="Spanning Tree Protocol">STP</abbr> prevents Layer 2 loops by blocking redundant paths. Without it, broadcast storms, MAC table instability, and duplicate frames can take down a network.

Key ideas:
- **Root bridge** is the switch with the lowest bridge ID.
- **Root port** is the best path toward the root on a non-root switch.
- **Designated port** forwards traffic for a segment.
- **Alternate port** provides a backup path and usually blocks.

Rapid Spanning Tree Protocol (<abbr title="Rapid Spanning Tree Protocol">RSTP</abbr>) converges faster than classic STP and is the common modern behavior you should expect in CCNA scenarios.

```bash
show spanning-tree
show spanning-tree vlan 10
show spanning-tree root
```

Troubleshooting clues:
- A port in blocking or alternate state may be normal if redundancy exists.
- If a switch unexpectedly becomes root, check bridge priority.
- If users report intermittent connectivity after a topology change, suspect a loop or a misconfigured trunk.

| Concept | STP/RSTP meaning | Why it matters |
|---|---|---|
| Root bridge | Central reference switch | Determines path selection |
| Root port | Best path to root | Forwards toward the root |
| Designated port | Best port on a segment | Forwards away from the root |
| Alternate port | Backup path | Prevents loops |

**Routing fundamentals**

Routing moves packets between different IP networks. A router or Layer 3 switch makes forwarding decisions using the routing table, which contains connected, static, and dynamic routes.

Common route codes:
- `C` = connected
- `L` = local
- `S` = static
- `D` = EIGRP
- `O` = OSPF
- `R` = RIP

```bash
show ip route
show ip route static
show ip protocols
```

A route is only useful if:
- The destination network is known.
- The next hop is reachable.
- The outgoing interface is up.
- The route has the best administrative distance and metric for that destination.

Static routes are simple and predictable. Dynamic routing protocols adapt to topology changes and reduce manual configuration, but they add protocol overhead and require correct neighbor relationships.

**Dynamic routing protocol behavior**

At CCNA level, you should understand how routing protocols exchange reachability information and how to verify that they are working.

- **RIP** uses hop count as its metric and has a maximum of 15 hops.
- **OSPF** uses cost, forms neighbor adjacencies, and builds a link-state database.
- **EIGRP** uses a composite metric and forms neighbor relationships with directly connected peers.

Verification is more important than memorizing names. You should know how to confirm neighbors, learned routes, and interface participation.

```bash
show ip ospf neighbor
show ip ospf interface brief
show ip eigrp neighbors
show ip protocols
show ip route ospf
show ip route eigrp
```

Practical interpretation:
- No neighbor adjacency usually means a mismatch in network statements, area, timers, authentication, or interface state.
- A route present in the routing table but not preferred may be losing to a lower administrative distance route.
- If a dynamic route disappears after a link failure, check whether the protocol reconverged and whether the alternate path exists.

**IPv4 addressing and subnetting**

IPv4 addresses are 32-bit addresses written in dotted decimal notation. Subnetting divides a network into smaller logical segments and determines how many hosts fit in each subnet.

You should be comfortable with:
- Network address
- Broadcast address
- Usable host range
- Subnet mask and prefix length
- Variable-length subnet masking (<abbr title="Variable-Length Subnet Masking">VLSM</abbr>)

Example:
- `192.168.10.0/24`
- `192.168.10.0` network
- `192.168.10.1` through `192.168.10.254` usable hosts
- `192.168.10.255` broadcast

Useful verification:
```bash
show ip interface brief
show running-config interface g0/0
ping 192.168.10.1
traceroute 8.8.8.8
```

When troubleshooting IPv4 connectivity, check the path in this order:
1. Interface status and IP addressing.
2. Default gateway on the host.
3. VLAN and trunk membership.
4. Routing table and next hop reachability.
5. ACLs, NAT, and firewall policy.

**IPv6 addressing and verification**

IPv6 uses 128-bit addresses written in hexadecimal and separated by colons. You should know the common address types and how to read them.

Common IPv6 address types:
- **Global unicast**: routable on the public Internet.
- **Link-local**: automatically present on every IPv6-enabled interface and used for local neighbor discovery and routing protocol adjacencies.
- **Unique local**: private addressing for internal use.
- **Multicast**: one-to-many delivery.
- **Anycast**: one-to-nearest delivery using the same address on multiple devices.

IPv6 compression rules:
- Remove leading zeros in a hextet.
- Replace one contiguous run of zero hextets with `::` only once.

Examples:
- Full: `2001:0db8:0000:0000:0000:0000:0000:0001`
- Compressed: `2001:db8::1`

```bash
show ipv6 interface brief
show ipv6 route
show ipv6 neighbors
ping ipv6 2001:db8::1
traceroute ipv6 2001:db8::1
```

Neighbor Discovery Protocol (<abbr title="Neighbor Discovery Protocol">NDP</abbr>) replaces many IPv4 ARP functions. It helps hosts discover routers, resolve neighbors, and maintain reachability information.

> If IPv6 routing fails but interfaces look correct, verify that IPv6 routing is enabled, the link-local neighbors are present, and the default route or router advertisements are correct.

**Wireless fundamentals**

Wireless design depends on frequency band, channel planning, coverage, and security. CCNA-level wireless questions often focus on why a client connects poorly or why throughput is lower than expected.

Key concepts:
- **2.4 GHz** has longer range and better wall penetration, but more interference and fewer non-overlapping channels.
- **5 GHz** offers more channels, less interference, and usually better performance, but shorter range.
- **Channel overlap** in 2.4 GHz can cause co-channel or adjacent-channel interference if access points are placed poorly.
- **RSSI** (<abbr title="Received Signal Strength Indicator">RSSI</abbr>) indicates signal strength; low RSSI often means weak coverage or too much attenuation.
- **SNR** (<abbr title="Signal-to-Noise Ratio">SNR</abbr>) affects usable throughput and stability.

Basic WLAN security:
- **PSK** (<abbr title="Pre-Shared Key">PSK</abbr>) uses a shared passphrase and is common in home or small-office networks.
- **Enterprise authentication** uses 802.1X with a RADIUS server and individual user credentials.
- Enterprise security scales better because you can revoke one user without changing the shared password for everyone.

| Topic | 2.4 GHz | 5 GHz |
|---|---|---|
| Range | Better | Shorter |
| Interference | Higher | Lower |
| Channel availability | Fewer non-overlapping channels | More channels |
| Typical use | Legacy and coverage | Performance and capacity |

Basic verification and troubleshooting:
- Confirm the SSID is broadcasting if it should be visible.
- Confirm the client has the correct security settings and passphrase.
- Check channel overlap and AP placement.
- Check whether the client is associating to the intended band.
- Verify DHCP if the client connects but gets no usable IP address.

**Security fundamentals**

Device hardening reduces the attack surface of network devices. The exam often expects you to know the difference between secure management access and insecure legacy access.

Best practices:
- Use **SSH** instead of Telnet because SSH encrypts management traffic.
- Disable unused services and interfaces.
- Use strong passwords and local or centralized authentication.
- Protect privileged access with `enable secret`.
- Restrict management access with ACLs where appropriate.
- Use banners to warn unauthorized users.

```bash
hostname R1
ip domain-name example.local
username admin privilege 15 secret Str0ngP@ss!
crypto key generate rsa modulus 2048
ip ssh version 2

line vty 0 4
 login local
 transport input ssh
```

SSH vs Telnet:
- **Telnet** sends credentials in clear text.
- **SSH** encrypts the session and is the correct choice for secure remote administration.

Management-plane protection examples:
- Restrict VTY access with an ACL.
- Use `login local` or AAA where configured.
- Disable Telnet with `transport input ssh`.
- Limit who can reach the device on the management VLAN or interface.

**ACLs and traffic filtering**

Access Control Lists (<abbr title="Access Control List">ACL</abbr>s) filter traffic based on source, destination, protocol, and ports. They are commonly used for security, traffic control, and management-plane protection.

Types:
- **Standard ACLs** match source IP only.
- **Extended ACLs** match source, destination, protocol, and ports.

Placement rule:
- Standard ACLs are usually placed close to the destination.
- Extended ACLs are usually placed close to the source.

Example:
```bash
access-list 10 permit 192.168.10.0 0.0.0.255
access-list 10 deny any

interface g0/1
 ip access-group 10 in
```

Extended ACL example:
```bash
ip access-list extended WEB-FILTER
 deny tcp 192.168.10.0 0.0.0.255 any eq 80
 permit ip any any
```

Verification:
```bash
show access-lists
show ip interface g0/1
```

Troubleshooting clues:
- If traffic is unexpectedly blocked, check ACL order because ACLs are processed top-down.
- Remember that an implicit `deny any` exists at the end.
- If counters do not increment, the ACL may be applied to the wrong interface or direction.

**NAT and PAT**

Network Address Translation (<abbr title="Network Address Translation">NAT</abbr>) translates private addresses to public addresses. Port Address Translation (<abbr title="Port Address Translation">PAT</abbr>) allows many inside hosts to share one public IP using different source ports.

Common terms:
- **Inside local**: private address of the internal host.
- **Inside global**: public address representing the inside host to the outside world.
- **Outside local**: address of the outside host as seen from inside, usually unchanged.
- **Outside global**: real public address of the outside host.

Example PAT configuration:
```bash
access-list 1 permit 192.168.10.0 0.0.0.255

interface g0/0
 ip nat inside

interface g0/1
 ip nat outside

ip nat inside source list 1 interface g0/1 overload
```

Verification:
```bash
show ip nat translations
show ip nat statistics
show running-config | include ip nat
```

Troubleshooting scenarios:
- If inside hosts cannot reach the Internet, verify inside and outside interface roles.
- If translations do not appear, verify the ACL matches the inside subnet.
- If only some traffic fails, check whether return routing exists on the outside side.

**DHCP review**

Dynamic Host Configuration Protocol (<abbr title="Dynamic Host Configuration Protocol">DHCP</abbr>) automatically assigns IP settings to clients. It reduces manual configuration and helps prevent addressing mistakes.

A DHCP lease typically provides:
- IP address
- Subnet mask
- Default gateway
- DNS server
- Lease time

Router as DHCP server example:
```bash
ip dhcp excluded-address 192.168.10.1 192.168.10.20

ip dhcp pool VLAN10
 network 192.168.10.0 255.255.255.0
 default-router 192.168.10.1
 dns-server 8.8.8.8
```

Verification:
```bash
show ip dhcp binding
show ip dhcp pool
show running-config | section dhcp
```

Common failure points:
- The client VLAN is not routed.
- The DHCP pool network does not match the client subnet.
- The default gateway is wrong.
- A relay agent is missing when the server is on another network.

**Troubleshooting by symptom**

A good exam strategy is to map symptoms to likely layers and then verify with the right command.

- **No link light**: check cable, interface shutdown state, speed/duplex, or transceiver issues.
- **Same-VLAN communication fails**: check VLAN membership, trunking, or STP blocking.
- **Inter-VLAN communication fails**: check SVI/router subinterfaces, `ip routing`, and default gateways.
- **Internet access fails but internal access works**: check NAT, default route, and ACLs.
- **Client gets AP association but no IP**: check DHCP, VLAN mapping, and wireless security settings.
- **IPv6 works locally but not beyond the LAN**: check router advertisements, default route, and IPv6 forwarding.

Useful commands by problem area:

```bash
show interfaces status
show mac address-table
show spanning-tree
show ip route
show ipv6 route
show ip nat translations
show access-lists
show ip dhcp binding
show cdp neighbors
show lldp neighbors
```

**Exam-style interpretation examples**

If `show ip route` displays only connected routes, the router may not have a default route or dynamic routing neighbor. If a host can ping its default gateway but not another subnet, the issue is often routing, ACLs, or NAT rather than basic Layer 2 connectivity.

If `show spanning-tree vlan 10` shows a port in blocking state, that is not automatically a fault. It becomes a problem only if the blocked port is the only path to a needed segment or if the topology was not designed for redundancy.

If `show access-lists` shows hit counts increasing on a deny entry, the ACL is working as written, even if the result is not what you intended. In that case, the fix is usually ACL order, wildcard mask accuracy, or direction of application.

**Quick comparison tables**

| Feature | IPv4 | IPv6 |
|---|---|---|
| Address size | 32-bit | 128-bit |
| Notation | Dotted decimal | Hexadecimal colon-separated |
| Broadcast | Yes | No broadcast; uses multicast |
| Neighbor resolution | ARP | NDP |
| Common config focus | Subnetting, NAT | Address types, routing, link-local |

| ACL type | Matches | Typical use |
|---|---|---|
| Standard | Source IP | Simple filtering near destination |
| Extended | Source, destination, ports, protocol | Precise traffic control near source |

**High-value commands to remember**

```bash
show running-config
show ip interface brief
show ipv6 interface brief
show vlan brief
show interfaces trunk
show spanning-tree
show ip route
show ipv6 route
show access-lists
show ip nat translations
show ip dhcp binding
```

These commands are valuable because they answer the most common CCNA questions: what is configured, what is active, what is being learned, and what is being blocked.

**Final review mindset**

When you study this module, focus on relationships:
- VLANs need trunks or access ports to carry traffic correctly.
- Inter-VLAN routing needs a Layer 3 gateway and correct host default gateways.
- Routing needs reachable next hops and valid routes.
- IPv6 needs correct addressing, link-local awareness, and neighbor discovery.
- Wireless needs the right band, channel plan, and security mode.
- Security depends on secure management access, ACL logic, and device hardening.
- Troubleshooting works best when you verify one layer at a time.

If you can explain the expected behavior, identify the likely failure point, and prove the fix with a command output, you are thinking like a CCNA technician.
