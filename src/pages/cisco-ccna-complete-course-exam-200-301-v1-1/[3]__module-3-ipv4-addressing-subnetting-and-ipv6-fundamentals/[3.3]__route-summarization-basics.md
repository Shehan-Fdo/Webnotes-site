**Route summarization overview**

Route summarization, also called **route aggregation** or **supernetting**, combines multiple more-specific routes into a single less-specific prefix. You use it to reduce routing table size, limit routing updates, and contain topology changes.

A summary route represents a range of addresses, not just one subnet. The tradeoff is that the summary can cover addresses that are not actually in use, so you must choose the boundary carefully.

- **Goal:** advertise fewer routes.
- **Benefit:** smaller routing tables and less update traffic.
- **Tradeoff:** less precision; the summary may include unused space.
- **Common use:** at network boundaries, between areas, or between autonomous systems.

**Why summarization matters**

Summarization improves scalability. When a router receives one summary instead of many specific routes, it processes fewer entries and sends fewer updates. This is especially useful in larger networks where many contiguous subnets exist under one site, department, or branch.

It also improves stability. If one subnet inside a summary fails, the summary can remain advertised, so upstream routers do not need to learn every internal change. That reduces route flapping outside the local domain.

**How to think about a summary**

A summary route is based on the **longest common prefix** of the component networks. You compare the binary form of the network addresses and keep the bits that match.

For example, four contiguous `/24` networks often summarize to a `/22` because:

- 4 networks = 2², so the prefix becomes 2 bits shorter than `/24`
- 8 networks = 2³, so the prefix becomes 3 bits shorter than `/24`
- 16 networks = 2⁴, so the prefix becomes 4 bits shorter than `/24`

This is a fast exam shortcut, but you still need to verify that the networks are aligned on the correct boundary.

**Quick prefix math**

If you start with `/24` subnets:

- 2 contiguous subnets → `/23`
- 4 contiguous subnets → `/22`
- 8 contiguous subnets → `/21`
- 16 contiguous subnets → `/20`

The same idea applies to other prefix lengths. You reduce the prefix length by the number of bits needed to cover the count of contiguous networks.

**Worked binary example**

Suppose you want to summarize these networks:

- `192.168.8.0/24`
- `192.168.9.0/24`
- `192.168.10.0/24`
- `192.168.11.0/24`

These are four contiguous `/24` networks, so the likely summary is `/22`.

Step through the third octet in binary:

- `8`  = `00001000`
- `9`  = `00001001`
- `10` = `00001010`
- `11` = `00001011`

The first 6 bits of the third octet match: `000010`. That gives a `/22` summary:

- Summary network: `192.168.8.0/22`
- Mask: `255.255.252.0`

That summary covers `192.168.8.0` through `192.168.11.255`.

> A summary can be configured even if not every address inside the range is used. The important question is whether you are willing to advertise the entire range.

**How to calculate the summary boundary**

Use this process when the exam asks you to find the correct summary:

1. List the networks you want to summarize.
2. Convert the changing octet to binary.
3. Find the longest common leading bits.
4. Convert those bits back to a network address.
5. Set the remaining bits to `0` for the summary network.
6. Choose the mask that matches the common prefix.

A practical shortcut is to look for a block size:

- `/24` to `/22` means a block of 4 in the third octet
- `/24` to `/21` means a block of 8
- `/24` to `/20` means a block of 16

For example, `192.168.12.0/22` is valid because `12` is a multiple of 4. But `192.168.10.0/22` is not a proper boundary for a `/22` summary because `10` is not aligned to a 4-subnet block.

**Default route as a special summary**

The default route, `0.0.0.0/0`, is the broadest possible summary. It matches all IPv4 destinations and is commonly used as a gateway of last resort.

Examples:

```bash
ip route 0.0.0.0 0.0.0.0 192.0.2.1
```

On Cisco IOS, you may also see:

```bash
ip route 0.0.0.0 0.0.0.0 GigabitEthernet0/0
```

Use the default route when you want to send unknown destinations toward an upstream router instead of maintaining many specific routes.

**Manual summarization vs auto-summarization**

Modern classless routing protocols support **manual summarization**, where you explicitly configure the summary. Older behavior included **auto-summarization**, where routes were summarized at classful boundaries automatically.

- **Manual summarization**
  - You choose the summary.
  - You can summarize across VLSM/CIDR designs.
  - It is the preferred approach in modern networks.

- **Auto-summarization**
  - The protocol summarizes at the classful boundary.
  - It can hide more-specific subnet information.
  - It is often disabled in modern Cisco designs.

Classless routing changed network design because it allowed variable-length subnet masks (VLSM) and route advertisement without forcing classful boundaries. That made manual summarization more flexible and more accurate.

**Protocol behavior at a glance**

| Protocol | Summarization style | Notes |
|---|---|---|
| RIP | Manual and legacy auto-summary behavior | RIPv2 supports VLSM; RIPv1 is classful and does not carry subnet masks |
| EIGRP | Manual summarization on interfaces | Creates a discard route to `Null0` for the summary |
| OSPF | Manual summarization at area and external boundaries | Area summarization is done by an ABR; external summarization is done by an ASBR |
| BGP | Manual aggregation | Aggregate routes are created from matching more-specific routes in the BGP table |

**RIP summarization**

RIP is a distance-vector protocol. In Cisco IOS, you can configure interface-level summarization with `ip summary-address rip`. This is a manual summary feature and is not the same as generic RIP behavior across all vendors.

Example:

```bash
interface GigabitEthernet0/0
 ip summary-address rip 10.1.4.0 255.255.252.0
```

This advertises a summary for the connected RIP-learned routes out that interface.

Important points:

- **RIPv1** is classful and does not support VLSM in the same way as RIPv2.
- **RIPv2** supports subnet masks in updates.
- Cisco IOS interface summarization is a manual feature, not automatic summarization.
- Use it when you want to reduce RIP update size on a boundary interface.

**EIGRP summarization**

Enhanced Interior Gateway Routing Protocol (**EIGRP**) supports manual summarization on an interface. When you configure a summary, EIGRP also installs a route to `Null0` for the summary prefix.

Example:

```bash
interface GigabitEthernet0/0
 ip summary-address eigrp 100 10.1.4.0 255.255.252.0
```

Why the `Null0` route matters:

- It prevents routing loops for destinations that fall inside the summary but do not match a more-specific route.
- It acts as a discard route for traffic that matches the summary only.
- It ensures the router does not forward packets back into the network when the exact subnet is missing.

You may see the summary and discard route in the routing table:

```bash
show ip route
```

Typical behavior:

- More-specific routes remain preferred when present.
- The summary route is advertised upstream.
- The `Null0` route catches unmatched traffic inside the summary range.

**OSPF summarization**

Open Shortest Path First (**OSPF**) supports summarization in two main places:

- **Inter-area summarization** on an Area Border Router (**ABR**)
- **External route summarization** on an Autonomous System Boundary Router (**ASBR**)

OSPF does not summarize routes inside a normal area by default. You summarize at the boundaries.

**Inter-area summarization on an ABR**

Use `area <area-id> range` on the ABR to summarize routes learned from that area before advertising them into another area.

Example:

```bash
router ospf 1
 area 10 range 172.16.0.0 255.255.0.0
```

This command is applied on the ABR for routes from area 10. The ABR must actually know the more-specific routes from that area for the summary to be meaningful.

Key points:

- The command summarizes routes from the specified area.
- It is configured under the OSPF process.
- It is used only on an ABR.
- The summary is advertised into other areas, not back into the same area as a replacement for internal routing.

**External summarization on an ASBR**

When you redistribute routes into OSPF, you can summarize external routes with `summary-address` on the ASBR.

Example:

```bash
router ospf 1
 redistribute static subnets
 summary-address 192.168.0.0 255.255.0.0
```

Use this when multiple redistributed routes can be represented by one external summary. The ASBR must be the router performing the redistribution or external route origination.

Important exam detail:

- `summary-address` is used for external route summarization.
- It is not the same as `area range`.
- `area range` summarizes inter-area routes.
- `summary-address` summarizes redistributed external routes.

**OSPF metric behavior**

OSPF summary routes do not preserve the exact cost of every component route. The summary is advertised with a cost chosen by the summarizing router, and the detailed internal path costs are hidden from other areas.

Practical implications:

- The summary can simplify the topology view.
- The exact best path inside the summarized area is no longer visible outside the area.
- If multiple component routes exist, the summary represents the range as a whole, not each subnet individually.

**BGP aggregation**

Border Gateway Protocol (**BGP**) supports route aggregation with `aggregate-address`. This creates a summary route from more-specific BGP routes.

Example:

```bash
router bgp 65001
 aggregate-address 203.0.112.0 255.255.252.0 summary-only
```

Important behavior:

- The aggregate is formed from matching more-specific routes in the BGP table.
- The summary is advertised only when the router has the contributing routes.
- `summary-only` suppresses the more-specific routes from being advertised.
- `as-set` can be used when you want to preserve AS path information in the aggregate.

Example with `as-set`:

```bash
router bgp 65001
 aggregate-address 203.0.112.0 255.255.252.0 summary-only as-set
```

Use BGP aggregation when you want to reduce the number of prefixes sent to neighbors, especially at edge routers or provider boundaries.

**How to choose the right summary**

A good summary should:

- Cover a contiguous block of addresses.
- Start on the correct binary boundary.
- Match the design goal, such as site-level aggregation or area-level reduction.
- Avoid hiding too much unrelated address space.

A bad summary may still be syntactically valid, but it can advertise addresses you do not own or use. That can cause blackholing if traffic is sent to the summary and no more-specific route exists.

**Practical verification commands**

Use these commands to inspect summarization behavior on Cisco IOS:

```bash
show ip route
show ip protocols
show ip ospf database
show ip bgp
show running-config | section router ospf
show running-config | section router bgp
```

What to look for:

- The summary route appears in the routing table.
- More-specific routes may still exist locally even if they are not advertised.
- In EIGRP, check for the `Null0` summary route.
- In OSPF, check whether the summary is visible in the correct area or external database.
- In BGP, confirm that the aggregate exists in the BGP table before expecting it to advertise.

**Common exam traps**

- Confusing **area summarization** with **external summarization** in OSPF.
- Assuming a summary must exactly match only used addresses.
- Forgetting that a summary route can cover unused space.
- Treating RIP auto-summary as the same thing as manual interface summarization.
- Forgetting that EIGRP installs a discard route for summaries.
- Assuming BGP aggregates appear without matching more-specific routes.
- Using a summary boundary that is not aligned to the correct block size.

**Step-by-step summary calculation example**

Suppose you need to summarize these four `/24` networks:

- `10.1.4.0/24`
- `10.1.5.0/24`
- `10.1.6.0/24`
- `10.1.7.0/24`

1. Count the networks: 4.
2. Recognize that 4 contiguous `/24`s summarize to `/22`.
3. Check the boundary: `4` is a multiple of 4, so the block is aligned.
4. Build the summary:
   - Network: `10.1.4.0`
   - Mask: `255.255.252.0`
   - Prefix: `/22`
5. Verify the range:
   - Covers `10.1.4.0` through `10.1.7.255`

Configuration example:

```bash
ip route 10.1.4.0 255.255.252.0 192.0.2.1
```

If you are advertising this summary in a routing protocol, the exact command depends on the protocol and location in the topology.

**Practical design guidance**

Use summarization at natural boundaries:

- Between access and distribution layers
- At branch routers toward the WAN
- At OSPF area borders
- At redistribution points
- At BGP edge routers

Avoid summarizing too early inside a small network if you still need visibility into individual subnets for troubleshooting or policy control.

**Key takeaways**

- Summarization reduces routing table size and update overhead.
- The summary is based on the longest common prefix.
- Four contiguous `/24`s summarize to `/22`; eight summarize to `/21`.
- `0.0.0.0/0` is the default route and a special case of summarization.
- OSPF uses `area range` for inter-area summaries and `summary-address` for external summaries.
- EIGRP summaries create a `Null0` discard route.
- BGP aggregation depends on matching more-specific routes in the BGP table.
- Manual summarization is the modern, flexible approach; auto-summary is legacy behavior.
