**Traditional WAN fundamentals**

A traditional WAN (Wide Area Network) connects branch sites to a central data center or hub using provider-managed transport. The branch router typically forwards traffic based on destination IP routing, and the WAN design is often hub-and-spoke.

Common legacy and modern WAN transport options include:

- **Leased lines**: Dedicated point-to-point circuits with predictable performance and high cost.
- **Frame Relay**: Older packet-switched WAN technology that used virtual circuits.
- **ATM (Asynchronous Transfer Mode)**: Legacy cell-based transport used for voice, video, and data.
- **MPLS (Multiprotocol Label Switching)**: Provider-managed private WAN service that forwards traffic using labels.
- **Broadband Internet**: Lower-cost transport using public Internet access, often with VPNs for security.

Traditional WANs usually rely on static or dynamic routing, manual QoS (Quality of Service), and separate security devices or tunnels. This model works, but it can be expensive, slow to change, and operationally complex.

**Traditional WAN forwarding model**

In a classic WAN, the router at each site makes forwarding decisions in the data plane using the routing table built by the control plane.

- **Control plane**: Learns routes using static routes, OSPF (Open Shortest Path First), EIGRP (Enhanced Interior Gateway Routing Protocol), BGP (Border Gateway Protocol), or other routing protocols.
- **Data plane**: Forwards packets according to the routing table and next-hop information.
- **Policy**: Often applied locally with ACLs (Access Control Lists), route maps, QoS policies, or tunnel configuration.

This is different from SD-WAN, where the control logic is more centralized and the forwarding behavior is more policy-driven across multiple transports.

**Traditional WAN limitations**

Traditional WANs are functional, but they are less flexible than modern overlay designs.

- **High cost**: Private circuits such as MPLS or leased lines can be expensive.
- **Slow provisioning**: Adding a branch or changing paths often requires provider coordination.
- **Limited path awareness**: The router usually forwards based on routing metrics, not real-time application performance.
- **Manual failover**: Backup links may exist, but failover behavior is often coarse and not application-aware.
- **Operational overhead**: Each site may need similar configuration, which increases management effort.

A common exam-level comparison is that traditional WANs focus on transport connectivity first, while SD-WAN focuses on application-aware policy across multiple transports.

**SD-WAN overview**

SD-WAN (Software-Defined WAN) separates the control plane from the data plane and uses centralized policy to steer traffic across an overlay built on top of one or more underlay networks.

- **Underlay**: The physical transport network, such as MPLS, broadband, LTE/5G, or Internet.
- **Overlay**: The logical tunnel network built over the underlay, usually encrypted and policy-controlled.
- **Centralized management**: Policies are defined once and pushed to many sites.
- **Application awareness**: Traffic can be classified and sent over the best path based on business intent.

Cisco SD-WAN commonly uses:
- **vManage**: Centralized management and policy interface.
- **vSmart**: Control plane controller that distributes policy and routing information.
- **vBond**: Orchestrator that helps devices discover controllers and join the fabric.
- **cEdge**: Cisco IOS XE SD-WAN edge device.
- **vEdge**: Cisco SD-WAN edge device running the vEdge platform software.

For CCNA recognition, remember that **cEdge** and **vEdge** are edge devices, while **vManage**, **vSmart**, and **vBond** are controller roles.

**SD-WAN control plane vs data plane**

SD-WAN makes the separation between control and forwarding more explicit than a traditional WAN.

- **Control plane**:
  - Learns and advertises routes.
  - Exchanges tunnel endpoint information.
  - Receives centralized policy.
  - Decides which paths are valid and preferred.
- **Data plane**:
  - Forwards user traffic through encrypted tunnels.
  - Uses policy decisions already learned from the control plane.
  - Can switch paths based on SLA (Service Level Agreement) conditions such as loss, latency, and jitter.

A simple way to think about it:

- Traditional WAN: the router learns routes and forwards traffic mostly by routing protocol and local configuration.
- SD-WAN: the edge device still forwards traffic, but the choice of path is guided by centralized policy and tunnel health.

**How SD-WAN works in practice**

SD-WAN usually evaluates traffic in this order:

1. **Classify** the traffic by application, source, destination, user, or DSCP (Differentiated Services Code Point).
2. **Apply policy** to decide preferred transport, backup transport, and security behavior.
3. **Measure path quality** using loss, latency, and jitter.
4. **Forward traffic** over the best available tunnel.
5. **Fail over** automatically if the preferred path no longer meets policy.

This is why SD-WAN is often described as application-aware routing. The device does not just ask, “What route exists?” It also asks, “Which path is best for this application right now?”

**OMP in Cisco SD-WAN**

OMP (Overlay Management Protocol) is the control protocol used in Cisco SD-WAN to exchange overlay information between devices and controllers.

OMP carries:
- **Routes**: Reachability information for prefixes in the overlay.
- **TLOCs** (Transport Locator objects): Information about how to reach an edge device, including transport color, system IP, and encapsulation.
- **Policies**: Centralized control and data policy information that influences route advertisement and path selection.

At a high level, OMP helps the fabric learn:
- Which prefixes exist
- Which edge device owns them
- Which transport tunnels can reach them
- Which policy should be applied to those routes

This is a major difference from traditional WAN routing, where the routing protocol alone usually handles reachability and the policy is more local and manual.

**Traditional WAN vs SD-WAN comparison**

| Aspect | Traditional WAN | SD-WAN |
|---|---|---|
| Architecture | Often hub-and-spoke | Often overlay with flexible topology |
| Transport | Usually MPLS or leased circuits, sometimes Internet | MPLS, broadband, LTE/5G, Internet together |
| Control plane | Distributed routing protocols | Centralized policy and controller-assisted control |
| Data plane | Router forwards based on routing table | Edge forwards based on policy and tunnel health |
| Path selection | Mostly metric-based | Application-aware and SLA-aware |
| Provisioning | Manual and slower | Faster, template-driven, centralized |
| Security | Often separate VPN/firewall design | Commonly integrated encryption and segmentation |
| Visibility | Limited per-path application insight | Better application and path visibility |

**Why SD-WAN is used**

SD-WAN is useful when you need more flexibility than a traditional WAN can provide.

- **Lower cost**: You can use cheaper Internet links alongside or instead of expensive private circuits.
- **Better resilience**: Multiple transports provide automatic failover.
- **Improved application performance**: Critical traffic can use the best available path.
- **Simplified operations**: Central policy reduces per-site configuration.
- **Cloud readiness**: Branches can reach SaaS (Software as a Service) and cloud apps directly instead of hairpinning through a data center.

**Real-world deployment example**

A branch has:
- MPLS for stable business traffic
- Broadband Internet for low-cost backup and cloud access
- LTE as emergency failover

A policy might say:
- Voice uses the path with the lowest latency and jitter.
- ERP traffic prefers MPLS.
- SaaS traffic goes directly to the Internet.
- If MPLS fails, ERP moves to broadband automatically.
- If broadband degrades beyond SLA thresholds, traffic shifts to LTE.

This is more granular than a traditional WAN design, where failover is often based only on route availability.

**Basic SD-WAN configuration flow**

A real deployment usually follows a template-driven workflow in vManage, then device-specific transport and policy settings are pushed to the edge.

1. **Define system and transport settings**
   - Set system IP, site ID, organization name, and controller addresses.
   - Configure WAN interfaces with transport colors such as `biz-internet`, `public-internet`, or `mpls`.

2. **Create feature templates**
   - Build interface, VPN, and routing templates.
   - Define tunnel encapsulation and routing behavior.

3. **Attach device templates**
   - Apply the templates to a cEdge or vEdge device.
   - Push the configuration from vManage.

4. **Create centralized policy**
   - Define traffic classes.
   - Match applications or prefixes.
   - Set preferred transport and fallback behavior.

5. **Monitor and verify**
   - Check tunnel status, route exchange, and application performance.
   - Confirm that failover and steering work as intended.

**Example: Cisco IOS XE SD-WAN-style transport interface**

The exact syntax varies by platform and release, but a simplified device-side example looks like this:

```bash
system
 host-name BRANCH-01
 system-ip 10.10.10.1
 site-id 101
 organization-name ACME
 vbond 203.0.113.10
!
vpn 0
 interface GigabitEthernet0/0/0
  ip address 198.51.100.2/30
  tunnel-interface
   encapsulation ipsec
   color biz-internet
   allow-service all
  no shutdown
 !
 ip route 0.0.0.0/0 198.51.100.1
!
vpn 10
 interface GigabitEthernet0/0/1
  ip address 10.1.10.1/24
  no shutdown
!
```

This example shows the idea:
- VPN 0 is the transport side.
- The WAN interface is marked as a tunnel interface.
- A transport color identifies the link type.
- A service VPN carries user traffic.

**Example: simple centralized policy intent**

A policy can classify traffic and prefer one transport over another.

```text
Traffic class: voice
  Match: application voice
  Preferred path: MPLS
  Backup path: broadband
  SLA: latency < 150 ms, jitter < 30 ms, loss < 1%

Traffic class: SaaS
  Match: destination prefix or application
  Preferred path: broadband
  Backup path: MPLS

Traffic class: default
  Match: all other traffic
  Preferred path: broadband
```

In a real vManage deployment, you would build this using centralized policy objects rather than hand-editing every router.

**Troubleshooting approach**

When a path or application fails, separate the problem into underlay, overlay, or policy.

- **Underlay issue**: The physical transport is broken or unstable.
  - Check interface status, IP addressing, ISP reachability, and default routes.
  - Example symptom: no tunnel forms because the WAN interface cannot reach the Internet gateway.

- **Overlay issue**: The tunnel or controller relationship is broken.
  - Check controller reachability, certificates, OMP adjacency, and tunnel status.
  - Example symptom: the transport works, but routes do not appear across sites.

- **Policy issue**: The network is up, but traffic is taking the wrong path.
  - Check centralized policy, application match rules, and SLA thresholds.
  - Example symptom: voice traffic is up but still using the wrong link.

A simple exam-oriented decision flow:

1. **Can the WAN interface reach the transport gateway?**
   - If no, troubleshoot the underlay.
2. **Is the SD-WAN tunnel up and are controllers reachable?**
   - If no, troubleshoot the overlay.
3. **Are routes and TLOCs present in the fabric?**
   - If no, troubleshoot control-plane exchange.
4. **Is traffic following the intended path?**
   - If no, troubleshoot policy and classification.

**Common troubleshooting commands**

On Cisco SD-WAN edge devices, useful verification commands include:

```bash
show sdwan control connections
show sdwan omp peers
show sdwan omp routes
show sdwan bfd sessions
show sdwan tunnel statistics
show ip route
show interfaces
```

What to look for:
- **Control connections**: Controller sessions should be established.
- **OMP peers**: Overlay route exchange should be active.
- **BFD** (Bidirectional Forwarding Detection): Tunnel health should be up.
- **Tunnel statistics**: Loss, latency, and jitter should match expectations.
- **IP route**: Underlay reachability should exist for transport endpoints.

**Troubleshooting scenarios**

- **Scenario 1: Branch cannot reach the controller**
  - Verify the WAN interface IP address and default route.
  - Confirm DNS or static controller IP reachability if used.
  - Check firewall or NAT rules that may block controller traffic.

- **Scenario 2: Tunnels are up, but remote prefixes are missing**
  - Check OMP adjacency and route advertisement.
  - Verify that the VPN and site IDs are correct.
  - Confirm that policy is not filtering the routes.

- **Scenario 3: Voice traffic uses broadband instead of MPLS**
  - Check application classification.
  - Review SLA thresholds and preferred path order.
  - Confirm that the MPLS transport color is available and healthy.

- **Scenario 4: Internet works, but SaaS performance is poor**
  - Check latency and jitter on the broadband underlay.
  - Verify that direct Internet breakout is enabled where appropriate.
  - Confirm that the traffic is not hairpinning through a central site.

**Exam recognition tips**

For CCNA-level questions, focus on these distinctions:

- Traditional WAN uses router-centric forwarding and often depends on provider circuits.
- SD-WAN uses an overlay on top of multiple underlays.
- SD-WAN separates control and data planes more clearly.
- Cisco SD-WAN edge devices are called **cEdge** and **vEdge**.
- **vManage**, **vSmart**, and **vBond** are controller components.
- **OMP** exchanges routes, TLOCs, and policy in the overlay.
- SD-WAN is chosen when you need centralized policy, better visibility, and application-aware path selection.

**Quick memory aid**

- **Underlay** = transport network
- **Overlay** = encrypted logical network
- **Control plane** = learns and distributes reachability
- **Data plane** = forwards user traffic
- **Policy** = decides which path the traffic should use
