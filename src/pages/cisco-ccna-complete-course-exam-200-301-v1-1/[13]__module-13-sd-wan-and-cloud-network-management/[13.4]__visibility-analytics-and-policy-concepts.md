## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Visibility, analytics, and policy overview**

Modern Cisco networks rely on three related ideas:

- **Visibility** tells you what the network is doing.
- **Analytics** turns raw data into useful insight.
- **Policy** controls what traffic is allowed and how it is treated.

At the CCNA level, you should understand the common tools used to observe devices and traffic, the kinds of data they produce, and how policy is applied through access control, QoS (Quality of Service), and segmentation. In practice, you use these capabilities together: visibility finds the problem, analytics helps explain it, and policy prevents or corrects it.

**Core monitoring sources**

Cisco networks commonly use several telemetry and event sources. Each one answers a different troubleshooting question.

- **SNMP** (Simple Network Management Protocol) is used to poll device status and counters, and to receive traps for important events.
- **Syslog** sends event messages from devices to a collector for centralized review.
- **Flow data** such as NetFlow or Flexible NetFlow shows who talked to whom, on what ports, and with how much traffic.
- **IP SLA** (Internet Protocol Service Level Agreement) measures reachability, delay, jitter, and loss by generating test traffic.
- **SPAN** (Switched Port Analyzer) copies traffic to a monitoring port for packet capture and deep inspection.
- **Model-driven telemetry** streams structured data from devices at regular intervals or on change, which is more scalable than traditional polling in many environments.

> <mark>CCNA focus:</mark> know what each tool is best for, not just its definition.

**SNMP fundamentals**

SNMP is a management protocol used by a network management station to query devices and receive alerts. It is still important because it provides a standard way to monitor interface status, CPU, memory, and other counters.

- **Manager**: the monitoring system that polls devices.
- **Agent**: the software on the network device that exposes data.
- **MIB** (Management Information Base): the structured database of managed objects.
- **OID** (Object Identifier): the numeric path to a specific object in the MIB.

SNMP uses two main operations:

- **Polling**: the manager asks the device for data at intervals.
- **Traps/informs**: the device sends an unsolicited alert when an event occurs.

SNMP versions matter:

- **SNMPv1/v2c** use community strings and are simpler but less secure.
- **SNMPv3** adds authentication and encryption and is preferred for secure management.

A basic SNMPv2c example on Cisco IOS/IOS XE might look like this:

```cisco
snmp-server community MONITOR ro
snmp-server location HQ-1F
snmp-server contact netops@example.com
snmp-server host 192.0.2.50 version 2c MONITOR
snmp-server enable traps
```

A more secure SNMPv3 example is platform-dependent and usually requires a user, group, and access level to be defined correctly:

```cisco
snmp-server group NMS-GRP v3 priv
snmp-server user nmsuser NMS-GRP v3 auth sha AuthPass123 priv aes 128 PrivPass123
snmp-server host 192.0.2.50 version 3 priv nmsuser
snmp-server enable traps
```

- **ro** means read-only access.
- **priv** means privacy, which includes encryption.
- **auth** means authentication, which verifies identity.

Typical verification commands include:

```cisco
show snmp
show snmp user
show snmp group
show snmp host
```

**Syslog and event visibility**

Syslog is one of the most important sources of operational visibility because it records device events in a centralized log system. It helps you correlate interface changes, authentication failures, routing issues, and hardware alarms.

Syslog messages have a severity level from 0 to 7:

| Level | Name | Meaning |
|---|---|---|
| 0 | Emergency | System unusable |
| 1 | Alert | Immediate action required |
| 2 | Critical | Critical condition |
| 3 | Error | Error condition |
| 4 | Warning | Warning condition |
| 5 | Notification | Normal but significant |
| 6 | Informational | Informational message |
| 7 | Debugging | Debug output |

A common Cisco syslog configuration looks like this:

```cisco
service timestamps log datetime msec
logging host 192.0.2.60
logging trap informational
logging source-interface GigabitEthernet0/0
logging buffered 16384 informational
```

- **logging host** sends logs to a remote syslog server.
- **logging trap informational** sets the minimum severity sent to the server.
- **logging buffered** keeps local logs in memory for quick checks.
- **service timestamps** makes logs easier to correlate with other events.

Example output interpretation:

```cisco
%LINK-3-UPDOWN: Interface GigabitEthernet0/1, changed state to up
%LINEPROTO-5-UPDOWN: Line protocol on Interface GigabitEthernet0/1, changed state to up
```

- The first message shows the physical link changed.
- The second message shows the Layer 2/Layer 3 line protocol became operational.

**Flow data and traffic analytics**

Flow-based monitoring helps you understand traffic patterns, top talkers, application ports, and conversation volume. This is different from SNMP because SNMP is mostly device and interface oriented, while flow data is traffic oriented.

At a high level, flow records usually include:

- Source and destination IP addresses
- Source and destination ports
- Protocol
- Input and output interfaces
- Packet and byte counts
- Timestamps

**NetFlow** is the classic Cisco flow technology. **Flexible NetFlow** improves on it by allowing more flexible record definitions and exporters. On modern Cisco platforms, the exact CLI syntax can vary, so you should understand the concept rather than memorize one platform-specific exporter command.

A conceptual Flexible NetFlow configuration might include:

```cisco
flow record RECORD-1
 match ipv4 source address
 match ipv4 destination address
 match transport source-port
 match transport destination-port
 collect counter bytes
 collect counter packets

flow exporter EXPORT-1
 destination 192.0.2.70
 transport udp 2055
 source GigabitEthernet0/0
 export-protocol netflow-v9

flow monitor MONITOR-1
 record RECORD-1
 exporter EXPORT-1
 cache timeout active 60
 cache timeout inactive 15

interface GigabitEthernet0/1
 ip flow monitor MONITOR-1 input
```

Use flow data when you need to answer questions like:

- Which hosts are generating the most traffic?
- Which applications are consuming bandwidth?
- Is a specific subnet talking to an unexpected destination?
- Is traffic volume increasing before a performance issue appears?

> **Note:** verification commands differ by platform. On many modern IOS/IOS XE devices, use Flexible NetFlow-specific `show flow ...` commands rather than legacy `show ip cache flow`.

**Model-driven telemetry and streaming telemetry**

Model-driven telemetry is a modern visibility method where the device streams structured data to a collector. Instead of waiting for periodic polling, the collector receives updates at a configured interval or when values change.

Why it matters:

- It scales better than frequent polling in large networks.
- It provides more timely data for analytics.
- It uses structured models, which are easier for automation systems to consume.

At a CCNA level, you should recognize it as a modern alternative to traditional SNMP polling, especially in intent-based and analytics-driven environments.

**IP SLA and active measurement**

IP SLA creates synthetic traffic to measure network performance from the device’s perspective. It is useful when you want to test a path, not just observe existing traffic.

Common uses include:

- Measuring round-trip delay
- Checking reachability to a next hop or remote host
- Testing DNS, HTTP, or UDP jitter in more advanced scenarios
- Supporting tracking and failover decisions

A simple ICMP echo IP SLA example:

```cisco
ip sla 1
 icmp-echo 192.0.2.1 source-interface GigabitEthernet0/0
 frequency 10
ip sla schedule 1 life forever start-time now
```

Verification:

```cisco
show ip sla statistics
show ip sla configuration
show track
```

Sample interpretation:

- If `show ip sla statistics` shows increasing timeout counts, the path may be unreachable or congested.
- If delay rises sharply, the issue may be congestion, routing change, or a degraded WAN link.

**SPAN and packet capture**

SPAN copies traffic from one or more source interfaces or VLANs to a destination port where a sniffer or analyzer is connected. It is useful when you need packet-level detail that flow records cannot provide.

Use SPAN when you need to inspect:

- TCP handshakes
- Retransmissions
- DNS queries and responses
- Application-layer behavior
- Encapsulation or malformed frames

A simple local SPAN example:

```cisco
monitor session 1 source interface GigabitEthernet0/1 both
monitor session 1 destination interface GigabitEthernet0/24
```

- **source interface** is the monitored port.
- **destination interface** is where the analyzer connects.
- **both** copies ingress and egress traffic.

SPAN is more detailed than flow data, but it is also more intrusive and less scalable. Use it for targeted troubleshooting, not broad continuous monitoring.

**Assurance in Cisco environments**

Assurance is the process of confirming that the network is behaving as intended. In Cisco Catalyst Center, assurance typically means collecting telemetry, analyzing health, and presenting actionable insights about devices, clients, applications, and paths.

Assurance differs from simple monitoring:

- **Monitoring** shows raw status and counters.
- **Analytics** correlates data and identifies patterns.
- **Assurance** evaluates whether the network is meeting expected service levels and highlights where the user experience is affected.

Typical assurance outputs include:

- Device health scores
- Client health scores
- Path trace results
- Application performance indicators
- Event correlation and anomaly detection

In practical terms, assurance helps you move from “this interface is up” to “users on this path are experiencing poor performance.”

Cisco DNA Center was rebranded as Cisco Catalyst Center. You may still see both names in study material and older documentation.

**Policy concepts**

Policy defines how the network should treat traffic and users. At the CCNA level, policy usually appears in three main forms:

- **Access control**: permit or deny traffic using ACLs (Access Control Lists).
- **Traffic treatment**: classify and mark traffic for QoS.
- **Segmentation**: separate users, devices, or applications into different logical groups.

**Access control lists**

ACLs are one of the most common policy enforcement tools. They filter traffic based on source/destination IP, protocol, and ports.

Example:

```cisco
ip access-list extended BLOCK-FTP
 deny tcp any any eq 21
 permit ip any any

interface GigabitEthernet0/1
 ip access-group BLOCK-FTP in
```

Use ACLs to:

- Restrict management access
- Block unwanted protocols
- Limit access between subnets
- Support security policy at the edge

**QoS classification and marking**

QoS helps prioritize important traffic when bandwidth is limited. Policy can classify traffic into categories and mark packets with values such as DSCP (Differentiated Services Code Point).

Common QoS actions:

- **Classify** traffic by application, source, destination, or port.
- **Mark** traffic with DSCP or CoS (Class of Service).
- **Queue** traffic differently based on priority.
- **Policing** drops or remarks traffic that exceeds a limit.
- **Shaping** buffers excess traffic and sends it later.

A conceptual class map and policy map example:

```cisco
class-map match-any VOICE
 match dscp ef

policy-map WAN-POLICY
 class VOICE
  priority 1000
 class class-default
  fair-queue

interface Serial0/0/0
 service-policy output WAN-POLICY
```

- **priority** gives strict priority to delay-sensitive traffic.
- **fair-queue** helps share remaining bandwidth.

**Segmentation**

Segmentation limits where traffic can go and reduces the blast radius of failures or attacks. Common forms include:

- **VLANs** for Layer 2 separation
- **Subnets** for Layer 3 separation
- **VRFs** (Virtual Routing and Forwarding) for separate routing tables
- **ACLs** between segments to control access

Example use case:

- Users in VLAN 10 can reach the internet.
- Voice devices in VLAN 20 get priority QoS.
- Guest devices in VLAN 30 can reach only the internet, not internal servers.

**Policy enforcement in troubleshooting**

When troubleshooting, policy can be the cause of a problem as often as the solution. Check whether the network is dropping, remarking, or redirecting traffic before assuming a link or routing failure.

Common checks include:

```cisco
show access-lists
show policy-map interface
show class-map
show run | section policy-map
show run | section access-list
```

If traffic is missing, ask:

- Is an ACL blocking it?
- Is QoS policing dropping it?
- Is the traffic in the wrong VLAN or VRF?
- Is a security policy redirecting or filtering it?

**Choosing the right tool**

Different tools answer different questions. This is a common CCNA exam theme.

| Tool | Best for | Not ideal for |
|---|---|---|
| SNMP | Device health, counters, interface status | Packet-level detail |
| Syslog | Event history and alerts | Traffic volume analysis |
| NetFlow/Flexible NetFlow | Traffic conversations and top talkers | Full packet contents |
| IP SLA | Path performance and reachability testing | Passive traffic observation |
| SPAN | Deep packet inspection | Large-scale continuous monitoring |
| Streaming telemetry | Scalable structured data collection | Ad hoc packet analysis |

A practical troubleshooting workflow often looks like this:

1. Check **syslog** for recent errors or interface changes.
2. Check **SNMP** or device health for CPU, memory, and interface counters.
3. Check **flow data** to identify top talkers or unusual conversations.
4. Use **IP SLA** to confirm path performance.
5. Use **SPAN** if you need packet-level proof.
6. Review **policy** if traffic is being filtered, shaped, or prioritized unexpectedly.

**Real-world troubleshooting examples**

A user reports slow application access.

- Start with syslog to see whether interfaces flapped or routing changed.
- Check SNMP counters for errors, drops, or high utilization.
- Review flow data to see whether a backup job or large transfer is consuming bandwidth.
- Use IP SLA to test latency to the application server.
- If the path is fine but the app is still slow, capture packets with SPAN.

A branch cannot reach a server.

- Check ACLs and segmentation first.
- Verify the route exists in the correct VRF or subnet.
- Use syslog for interface or adjacency failures.
- Use IP SLA to test reachability from the branch router.
- Use flow data to confirm whether traffic is leaving the branch at all.

**Exam reminders**

- Know the difference between **polling** and **traps**.
- Know that **SNMP** is device-oriented and **flow data** is traffic-oriented.
- Know that **syslog** records events and is often the first place to look for recent changes.
- Know that **SPAN** gives packet-level visibility, while flow data gives conversation-level visibility.
- Know that **policy** can mean ACLs, QoS, or segmentation depending on context.
- Know that **assurance** is about service health and user experience, not just raw device status.

**Quick reference commands**

```cisco
show interfaces
show ip interface brief
show logging
show snmp
show access-lists
show policy-map interface
show ip sla statistics
show flow monitor
show monitor session
```

These commands help you move from symptoms to evidence. In CCNA troubleshooting, the best answer is usually the one that matches the type of data you need, not the most advanced tool available.
