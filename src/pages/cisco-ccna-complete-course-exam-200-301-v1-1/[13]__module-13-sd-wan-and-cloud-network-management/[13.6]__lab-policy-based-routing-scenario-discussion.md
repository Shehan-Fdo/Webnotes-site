## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Policy-Based Routing overview**

Policy-based routing (**PBR**) lets you override normal destination-based routing for selected traffic. Instead of choosing the path only from the destination IP address, the router evaluates a policy and can forward packets based on source IP, destination IP, protocol, or other match criteria.

Use PBR when you need traffic engineering that the routing table alone cannot provide, such as:

- Sending one subnet to a specific ISP while keeping other traffic on the default path.
- Steering guest or voice traffic to a different next hop.
- Testing a new WAN path without changing the entire routing design.

PBR is applied on an interface with an inbound route map. The router checks packets as they enter that interface, matches them against route-map sequences, and then applies the configured action.

**How PBR changes forwarding**

Normal routing uses the routing table to choose the best path to the destination. PBR changes that decision only for traffic that matches the policy.

A simple mental model is:

- Packet enters an interface.
- Router checks the PBR policy on that interface.
- If a route-map sequence matches and contains a `set` action, the router forwards according to that policy.
- If a route-map sequence matches but has no `set` action, the packet is forwarded normally.
- If no route-map sequence matches, the packet is also forwarded normally.

That distinction matters in labs. A matching sequence without a `set` does not mean “drop” and does not mean “use the next sequence automatically.” It means the packet matched the policy logic, but the route-map did not change forwarding.

**Typical lab topology**

A common CCNA lab scenario uses one LAN and two upstream paths:

```text
PC1 --- SW1 --- R1 --- ISP-A
              \--- ISP-B
```

Example goal:

- Traffic from `192.168.10.0/24` should use ISP-B.
- All other traffic should use the normal routing table, usually the default route toward ISP-A.

In this design, PBR is useful because the destination may be the same for all users, but the source network determines the path.

**Core PBR components**

PBR usually uses these building blocks:

- **ACLs** or other match criteria to identify traffic.
- A **route map** to define policy logic.
- A **set** action to specify the forwarding decision.
- An interface command to apply the policy inbound.

Common match options include:

- `match ip address` for source/destination-based ACL matching.
- `match interface` in some designs.
- `match length` or other platform-specific options, though CCNA labs usually focus on ACL-based matching.

Common set options include:

- `set ip next-hop <ip-address>`
- `set interface <interface>`
- `set ip default next-hop <ip-address>` in some IOS variants

For CCNA study, `set ip next-hop` is the most important and predictable option to understand.

**Example configuration**

This example policy sends traffic from `192.168.10.0/24` to next hop `10.2.2.2` while leaving all other traffic to normal routing.

```cisco
ip access-list extended PBR-TRAFFIC
 permit ip 192.168.10.0 0.0.0.255 any

route-map PBR-POLICY permit 10
 match ip address PBR-TRAFFIC
 set ip next-hop 10.2.2.2

route-map PBR-POLICY permit 20
 ! No match and no set means this sequence does not change forwarding

interface g0/0
 ip policy route-map PBR-POLICY
```

Key points:

- The ACL identifies the traffic you want to steer.
- Sequence `10` matches the source subnet and sets the next hop.
- Sequence `20` is not a “catch-all forwarding rule”; it is simply a later sequence that can match additional traffic if you add match statements.
- If traffic does not match any sequence with a `set`, normal routing applies.

**Route-map logic and sequence behavior**

Route maps are processed in sequence order. The router evaluates the first matching sequence and applies its action.

Important behavior to remember:

- A route-map sequence can contain:
  - `match` only
  - `set` only
  - both `match` and `set`
  - neither, though that is rarely useful
- If a sequence matches and has a `set`, the router uses that policy action.
- If a sequence matches and has no `set`, the packet continues with normal forwarding.
- If no sequence matches, the packet also uses normal forwarding.

This is why a “permit 20” sequence with no `match` statements is not a pass-through placeholder. It is a match-all sequence. If you add a `set` under it, it will policy-route all traffic that reaches that sequence.

**Before-and-after forwarding example**

Suppose the router has these routes:

```cisco
ip route 0.0.0.0 0.0.0.0 10.1.1.1
ip route 203.0.113.0 255.255.255.0 10.1.1.1
```

And PBR is configured to send `192.168.10.0/24` to `10.2.2.2`.

Packet examples:

- `192.168.10.50 -> 203.0.113.10`
  - Matches the ACL
  - Route map sets next hop to `10.2.2.2`
  - Packet follows ISP-B path
- `192.168.20.50 -> 203.0.113.10`
  - Does not match the ACL
  - Normal routing uses the routing table
  - Packet follows ISP-A path or the best route in the table

This is the core exam idea: PBR changes the path based on policy, not destination alone.

**`set ip next-hop` behavior**

`set ip next-hop` tells the router to forward the packet to a specified next hop if the policy matches.

Practical notes:

- The next hop should be reachable from the interface where PBR is applied.
- The router must be able to resolve the next hop at Layer 2, usually through ARP on Ethernet.
- If the next hop is not usable, IOS behavior can vary by platform and feature set.
- In many lab contexts, if the policy next hop cannot be used, the router may fall back to normal routing if a valid route exists.

For CCNA, focus on the intended design and basic verification rather than platform-specific edge cases.

**`set interface` versus `set ip next-hop`**

`set interface` and `set ip next-hop` are not the same.

- `set interface` forwards traffic out a specified interface.
- `set ip next-hop` forwards traffic to a specified next-hop IP address.
- `set ip next-hop` is generally easier to reason about in routed Ethernet designs.
- `set interface` can be less predictable in some topologies and is not usually the preferred first choice for CCNA labs.

Use `set ip next-hop` when you want a clear, testable policy path.

**Verification commands**

Use verification commands to confirm both policy attachment and route-map matches.

```cisco
show ip policy
show route-map
show access-lists
show ip interface g0/0
show ip route
```

What each command helps you confirm:

- `show ip policy`
  - Shows which interfaces have PBR applied.
- `show route-map`
  - Shows route-map sequences, match conditions, and hit counts.
- `show access-lists`
  - Confirms whether the ACL is matching the expected traffic.
- `show ip interface g0/0`
  - Confirms whether the interface has an inbound policy.
- `show ip route`
  - Confirms the normal routing table for comparison.

A useful troubleshooting pattern is to compare the route table with the policy counters. If the route table is correct but traffic still takes the wrong path, the issue is often in the ACL match or route-map attachment.

**Troubleshooting workflow**

1. Confirm the policy is applied to the correct interface.
2. Confirm the ACL matches the intended source or traffic class.
3. Confirm the route-map sequence has the expected `set` action.
4. Generate traffic that should match the policy.
5. Check hit counts in `show route-map` and `show access-lists`.
6. Verify the next hop is reachable and the upstream path is operational.

Common symptoms and causes:

- Traffic ignores PBR entirely
  - The route map is not applied inbound on the correct interface.
  - The ACL does not match the traffic.
- Only some hosts are policy-routed
  - The ACL is too narrow or has the wrong wildcard mask.
- Traffic matches but still uses the normal route
  - The route-map sequence matches but has no `set`.
  - The next hop is not usable, so the router falls back to normal routing in that platform/context.
- Counters stay at zero
  - No matching traffic is entering the interface.
  - The policy is attached to the wrong interface or direction.

**Practical ACL and route-map example**

This example steers only web traffic from a specific subnet.

```cisco
ip access-list extended WEB-PBR
 permit tcp 192.168.10.0 0.0.0.255 any eq 80
 permit tcp 192.168.10.0 0.0.0.255 any eq 443

route-map WEB-POLICY permit 10
 match ip address WEB-PBR
 set ip next-hop 10.2.2.2

interface g0/0
 ip policy route-map WEB-POLICY
```

Why this is useful:

- Only HTTP and HTTPS traffic from the subnet is steered.
- Other traffic from the same subnet uses normal routing.
- This lets you test selective path control without affecting all traffic.

**Design cautions**

> PBR is powerful, but it can complicate troubleshooting if you overuse it. Apply it only where you need traffic steering, and document the policy clearly.

Keep these cautions in mind:

- PBR can override the expected path from the routing table.
- Asymmetric routing can appear if return traffic follows a different path.
- NAT, firewall rules, and upstream ACLs may interact with PBR.
- Route-map order matters, so sequence planning is important.
- Broad ACLs can unintentionally steer more traffic than intended.

**Exam-focused takeaways**

- PBR is applied inbound on an interface with `ip policy route-map <name>`.
- Route maps are evaluated in sequence order.
- A matching sequence with a `set` changes forwarding.
- A matching sequence without a `set` forwards normally.
- If no sequence matches, normal routing is used.
- `set ip next-hop` is the most important PBR action for CCNA labs.
- Verification should start with `show ip policy`, `show route-map`, and ACL counters.

**Quick lab checklist**

- Confirm the source subnet in the ACL.
- Confirm the route-map sequence number and `set` action.
- Confirm the policy is applied inbound on the correct interface.
- Generate test traffic from a matching host.
- Check counters and compare the observed path with the intended path.

**Mini reference topology and flow**

```text
Host 192.168.10.50
   |
   v
R1 g0/0 inbound PBR
   |
   +-- match ACL? yes --> set next-hop 10.2.2.2 --> ISP-B
   |
   +-- match ACL? no  --> normal routing table --> ISP-A or default route
```

This flow is the easiest way to reason about PBR in a lab: match first, then set, then forward.
