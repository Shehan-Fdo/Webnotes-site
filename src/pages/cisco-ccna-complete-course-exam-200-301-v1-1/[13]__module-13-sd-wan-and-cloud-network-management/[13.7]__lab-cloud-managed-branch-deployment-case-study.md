## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Cloud-Managed Branch Deployment Overview**

A cloud-managed branch design uses centralized policy and monitoring to simplify deployment at a remote site. In a typical CCNA-style case study, you connect a branch router, access switch, wireless access point, and end devices, then verify that users can reach local resources, the internet, and management services.

The main goal is to translate business requirements into a working topology:

- **Users** need wired and wireless access.
- **Voice or guest traffic** may need separate VLANs.
- **Management** traffic should stay reachable and controlled.
- **Internet access** usually requires NAT (Network Address Translation).
- **Cloud-managed devices** often receive configuration from a controller or dashboard rather than being configured entirely by hand.

A practical branch deployment often includes a router-on-a-stick design, where one router interface carries multiple VLANs over an 802.1Q trunk to the switch. This is a common CCNA pattern because it demonstrates inter-VLAN routing, DHCP, and NAT in one lab.

**Typical Branch Topology and Roles**

A small branch usually has these roles:

- **Router**
  - Provides inter-VLAN routing.
  - Acts as the default gateway for user VLANs.
  - Performs NAT for internet-bound traffic.
  - May provide DHCP if no server exists.
- **Layer 2 switch**
  - Connects end devices.
  - Carries VLANs on trunk links.
  - Uses an SVI (Switched Virtual Interface) only for switch management, not for user routing.
- **Access point**
  - Bridges wireless clients into the wired network.
  - May map SSIDs to VLANs.
  - May be locally managed or cloud-managed.
- **Clients**
  - Receive IP settings from DHCP or static configuration.
  - Use the router as the default gateway for their VLAN.

A cloud-managed deployment does not mean every device is controlled by the cloud in the same way. In practice, the cloud platform may manage the AP, switch, or router through templates, profiles, or site assignments, while the local branch still uses standard Layer 2 and Layer 3 fundamentals.

**Business Context for the Case Study**

A realistic branch scenario helps you reason about the design. For example, a company opens a small sales office with 20 employees, a guest Wi-Fi network, and a printer on a separate VLAN. The branch must:

- Provide internet access through a single public IP.
- Keep guest traffic isolated from internal resources.
- Allow IT staff to manage the switch and AP remotely.
- Support wired desktops and wireless laptops.
- Use a simple, repeatable configuration that can be deployed to future branches.

This is why cloud-managed branch designs are attractive: you can standardize the configuration, reduce manual errors, and push consistent settings to multiple sites.

**VLAN and IP Plan**

A clean addressing plan makes the lab easier to validate and troubleshoot. One common example is:

| Purpose | VLAN | Subnet | Gateway |
|---|---:|---|---|
| Users | 10 | `192.168.10.0/24` | `192.168.10.1` |
| Voice or IoT | 20 | `192.168.20.0/24` | `192.168.20.1` |
| Guest | 30 | `192.168.30.0/24` | `192.168.30.1` |
| Management | 99 | `192.168.99.0/24` | `192.168.99.1` |

Use VLANs to separate traffic by function, not by device type alone. The exact VLAN numbers are less important than the consistency of the design.

Key points:

- **User VLANs** carry normal employee traffic.
- **Guest VLANs** should be isolated from internal subnets.
- **Management VLANs** should be restricted to admin devices and infrastructure.
- **Native VLAN** should be chosen deliberately and kept consistent on both ends of a trunk.

> If you use a management SVI on a Layer 2 switch, it is for switch management only. The switch does not route between VLANs unless it is a Layer 3 switch with routing enabled.

**Router-on-a-Stick Configuration**

Router-on-a-stick uses one physical router interface with multiple logical subinterfaces. Each subinterface tags traffic for a specific VLAN using IEEE 802.1Q.

Example configuration:

```cisco
interface g0/0
 no shutdown

interface g0/0.10
 encapsulation dot1Q 10
 ip address 192.168.10.1 255.255.255.0

interface g0/0.20
 encapsulation dot1Q 20
 ip address 192.168.20.1 255.255.255.0

interface g0/0.30
 encapsulation dot1Q 30
 ip address 192.168.30.1 255.255.255.0

interface g0/0.99
 encapsulation dot1Q 99
 ip address 192.168.99.1 255.255.255.0
```

Why this matters:

- Each subinterface becomes the default gateway for one VLAN.
- The switch-to-router link must be a trunk.
- The router can route between VLANs because it has an IP interface in each subnet.
- If you use a native VLAN, configure it consistently on both ends.

A common trunk configuration on the switch looks like this:

```cisco
interface g0/1
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30,99
 switchport trunk native vlan 99
```

Use a native VLAN carefully. In many labs, VLAN 99 is chosen as the native and management VLAN, but you should understand that the native VLAN is simply the untagged VLAN on an 802.1Q trunk. It is not automatically a management VLAN.

**Switch Management and Access Ports**

A Layer 2 switch uses an SVI for management access. That SVI does not route user traffic; it only gives the switch an IP address so you can manage it remotely.

Example:

```cisco
vlan 99
 name MANAGEMENT

interface vlan 99
 ip address 192.168.99.2 255.255.255.0
 no shutdown

ip default-gateway 192.168.99.1
```

Important details:

- The SVI must be in an active VLAN.
- At least one port in that VLAN must be up, or the SVI may remain down/down.
- `ip default-gateway` is used on a Layer 2 switch for management reachability.
- If the switch were a Layer 3 switch with routing enabled, you would use a default route instead.

Access ports should be assigned to the correct VLAN:

```cisco
interface range f0/1-12
 switchport mode access
 switchport access vlan 10

interface range f0/13-18
 switchport mode access
 switchport access vlan 20

interface f0/19
 switchport mode access
 switchport access vlan 30
```

Practical validation:

- A PC in VLAN 10 should receive an address in `192.168.10.0/24`.
- Its default gateway should be `192.168.10.1`.
- It should reach devices in other VLANs if inter-VLAN routing is enabled.
- It should reach the internet if NAT and the default route are correct.

**DHCP Design and Configuration**

DHCP (Dynamic Host Configuration Protocol) reduces manual setup and is common in branch labs. You can run DHCP on the router or point clients to a central server.

Router-based DHCP example:

```cisco
ip dhcp excluded-address 192.168.10.1 192.168.10.20
ip dhcp excluded-address 192.168.20.1 192.168.20.20
ip dhcp excluded-address 192.168.30.1 192.168.30.20
ip dhcp excluded-address 192.168.99.1 192.168.99.20

ip dhcp pool USERS
 network 192.168.10.0 255.255.255.0
 default-router 192.168.10.1
 dns-server 8.8.8.8

ip dhcp pool VOICE
 network 192.168.20.0 255.255.255.0
 default-router 192.168.20.1
 option 150 ip 192.168.20.10

ip dhcp pool GUEST
 network 192.168.30.0 255.255.255.0
 default-router 192.168.30.1
 dns-server 8.8.8.8
```

Notes:

- Use `excluded-address` for gateways, printers, servers, and other static devices.
- `default-router` must match the gateway IP in the VLAN.
- `option 150` is commonly used for Cisco voice deployments when phones need a TFTP server address.
- If DHCP is on another subnet, use `ip helper-address` on the router interface or SVI that receives the client broadcasts.

Example relay configuration:

```cisco
interface g0/0.10
 ip helper-address 192.168.50.10
```

**NAT and Internet Access**

NAT allows private branch addresses to reach the internet using a public address. In CCNA labs, PAT (Port Address Translation), also called NAT overload, is the most common form.

Example:

```cisco
access-list 1 permit 192.168.10.0 0.0.0.255
access-list 1 permit 192.168.20.0 0.0.0.255
access-list 1 permit 192.168.30.0 0.0.0.255

interface g0/0
 ip nat inside

interface g0/1
 ip nat outside

ip nat inside source list 1 interface g0/1 overload
```

Why this works:

- Inside interfaces face the branch LAN.
- The outside interface faces the ISP or upstream network.
- The ACL identifies which inside addresses are eligible for translation.
- Overload lets many inside hosts share one public IP by using different ports or protocol identifiers.

Example verification:

```cisco
show ip nat translations
show ip nat statistics
```

A sample translation entry may look like this:

```cisco
Pro  Inside global         Inside local          Outside local         Outside global
icmp 203.0.113.10:12       192.168.10.25:12      198.51.100.20:12      198.51.100.20:12
```

The exact format varies by protocol. TCP and UDP use ports, while ICMP uses identifiers.

**Wireless and Cloud-Managed AP Considerations**

A branch AP often bridges wireless clients into one or more VLANs. In a cloud-managed design, the AP may be adopted into a controller or dashboard, then assigned to a site and given SSID-to-VLAN mappings.

Common deployment patterns:

- **Single SSID for employees**
  - Maps to the user VLAN.
  - Uses WPA2/WPA3 authentication.
- **Guest SSID**
  - Maps to a guest VLAN.
  - Uses internet-only access and isolation rules.
- **Management access**
  - May use a dedicated management VLAN or a management IP on the AP’s native network.
  - Depends on the platform and design.

Practical caution:

> Do not assume every AP management path requires a router subinterface. If the AP is cloud-managed over the internet, its management traffic may simply use the branch’s normal routed internet access. A dedicated management VLAN is only needed when the AP’s management IP is placed on that VLAN.

If the AP uses a tagged management VLAN on the switch:

- The switch port must allow that VLAN.
- The AP must be configured to tag or untag management traffic as expected.
- The router only needs a subinterface or SVI for that VLAN if the AP’s management IP must be routed locally within the branch.

Example switch port for an AP:

```cisco
interface f0/24
 switchport mode trunk
 switchport trunk allowed vlan 10,30,99
```

This allows the AP to carry multiple SSIDs mapped to different VLANs. Some APs use a native VLAN for management and tag client VLANs; others tag everything. Match the switch configuration to the AP design.

**Cloud-Managed Workflow**

Cloud-managed platforms differ, but the workflow usually follows the same logic:

1. **Claim the device**
   - Add the serial number or claim code to the cloud dashboard.
2. **Assign the device to a site**
   - Group devices by branch location.
3. **Apply a template or profile**
   - Push SSIDs, VLAN mappings, switch port profiles, and uplink settings.
4. **Verify connectivity**
   - Confirm the device reaches the cloud and receives policy.
5. **Monitor status**
   - Check alarms, client counts, and interface health.

In Cisco environments, the exact interface depends on the platform:

- **Cisco Meraki** uses the Meraki Dashboard with organization and network/site concepts.
- **Cisco Catalyst Center** uses inventory, templates, and site hierarchy concepts.
- The exam may test the general idea of centralized management more than a specific vendor workflow.

A practical example of a cloud action might be:

- Create a branch site named `Branch-01`.
- Add the switch and AP to that site.
- Apply a template that sets VLANs 10, 20, 30, and 99.
- Map `Corp-WiFi` to VLAN 10 and `Guest-WiFi` to VLAN 30.
- Push a switch port profile for the AP uplink as a trunk.

**Validation Commands and What to Look For**

Validation is where you prove the design works. Use a layered approach: physical, Layer 2, Layer 3, and services.

Useful commands:

```cisco
show ip interface brief
show interfaces trunk
show vlan brief
show running-config | section interface
show ip route
show ip dhcp binding
show ip nat translations
show cdp neighbors
show lldp neighbors
```

What each command tells you:

- `show ip interface brief`
  - Confirms interfaces are up/up and have the expected IP addresses.
- `show interfaces trunk`
  - Confirms the switch trunk is active and carrying the correct VLANs.
- `show vlan brief`
  - Confirms access ports are in the correct VLANs.
- `show ip route`
  - Confirms the router knows connected networks and has a default route if needed.
- `show ip dhcp binding`
  - Confirms clients received leases.
- `show ip nat translations`
  - Confirms inside hosts are being translated.
- `show cdp neighbors` / `show lldp neighbors`
  - Confirms device adjacency and helps identify cabling.

**Troubleshooting Workflow**

A strong troubleshooting method prevents guesswork. Start at the edge and move inward.

1. **Check physical links**
   - Verify cables, link lights, and interface status.
   - Look for administratively down interfaces.
2. **Check VLAN membership**
   - Confirm access ports are in the correct VLAN.
   - Confirm the trunk allows the required VLANs.
3. **Check gateway reachability**
   - Verify the router subinterface or SVI has the correct IP.
   - Confirm the client default gateway matches the VLAN gateway.
4. **Check DHCP**
   - Confirm the pool exists and has available addresses.
   - Verify excluded addresses do not consume the pool.
5. **Check routing**
   - Confirm the router has a default route toward the ISP.
   - Confirm return traffic can get back to the branch.
6. **Check NAT**
   - Confirm inside and outside interfaces are correctly marked.
   - Confirm the ACL matches the inside subnets.
7. **Check wireless mapping**
   - Confirm the SSID maps to the intended VLAN.
   - Confirm the AP uplink allows that VLAN.

A simple case study example:

- A laptop on the guest SSID gets an IP address but cannot browse the web.
- You verify the AP uplink trunk allows VLAN 30.
- You confirm the router has `g0/0.30` with `192.168.30.1`.
- You check `show ip nat translations` and see no entries for `192.168.30.0/24`.
- You discover the NAT ACL only permits VLAN 10 and VLAN 20.
- Fixing the ACL restores internet access.

That sequence shows why validation should follow the traffic path, not just the device list.

**IPv6 in a Branch Design**

IPv6 can be deployed alongside IPv4 in a branch without NAT in most enterprise designs. The common approach is to route IPv6 end to end and use global unicast addressing with proper prefix delegation or static assignment.

Example:

```cisco
ipv6 unicast-routing

interface g0/0.10
 ipv6 address 2001:db8:10:1::1/64

interface g0/0.20
 ipv6 address 2001:db8:20:1::1/64

interface g0/1
 ipv6 address 2001:db8:ffff:1::2/64
```

Key ideas:

- `ipv6 unicast-routing` enables IPv6 forwarding on the router.
- Each VLAN can have its own `/64`.
- Hosts can use SLAAC (Stateless Address Autoconfiguration), DHCPv6, or both.
- NAT is generally not required for IPv6 in typical enterprise branch designs, though translation technologies do exist.

If you use DHCPv6, remember that it can provide addresses, DNS, or both depending on the configuration mode. For CCNA, focus on the relationship between router advertisements, prefixes, and host addressing.

**Common Mistakes in This Lab**

- **Forgetting the trunk**
  - The router subinterfaces exist, but the switch port is still an access port.
- **Mismatched VLAN IDs**
  - The router uses VLAN 20, but the switch access port is in VLAN 30.
- **Wrong default gateway**
  - Clients point to the switch SVI instead of the router gateway.
- **Missing NAT ACL entries**
  - One VLAN reaches the internet while another does not.
- **Incorrect AP VLAN mapping**
  - The SSID is mapped to the wrong VLAN or the trunk does not allow it.
- **Assuming management equals routing**
  - A switch SVI or AP management IP does not automatically provide inter-VLAN routing.

**Exam-Focused Takeaways**

- **Router-on-a-stick** is a common CCNA design for inter-VLAN routing on a small branch.
- **SVIs on Layer 2 switches** are for management, not user routing.
- **DHCP** can be local on the router or relayed to another server.
- **NAT overload** is the standard way to share one public IP among many private hosts.
- **Cloud-managed** means centralized policy and monitoring, but the underlying VLAN, trunk, routing, and DHCP concepts still apply.
- **Validation** should confirm Layer 1 through Layer 3 before you blame the cloud platform or wireless controller.

**Quick Lab Checklist**

1. Build the VLANs on the switch.
2. Assign access ports to the correct VLANs.
3. Configure the trunk to the router and AP.
4. Create router subinterfaces with matching VLAN IDs.
5. Configure DHCP pools and exclusions.
6. Configure NAT inside and outside.
7. Add a default route toward the ISP.
8. Configure switch management SVI and default gateway.
9. Map SSIDs to the correct VLANs.
10. Test client addressing, inter-VLAN routing, and internet access.

**Practical Verification Example**

A good end-to-end test sequence is:

```bash
ping 192.168.10.1
ping 192.168.20.1
ping 192.168.99.2
ping 8.8.8.8
nslookup example.com
```

Interpretation:

- If you can ping the gateway but not another VLAN, check routing.
- If you can ping the gateway but not the internet, check NAT and the default route.
- If DNS fails but IP connectivity works, check the DHCP DNS setting or upstream DNS reachability.

**Summary**

Cloud-managed branch deployment combines standard campus networking with centralized operations. The exam value comes from understanding how VLANs, trunks, router subinterfaces, DHCP, NAT, and wireless mappings fit together into one working design. If you can trace a packet from a client to the gateway, through NAT, and out to the internet, you can usually diagnose the lab quickly and confidently.
