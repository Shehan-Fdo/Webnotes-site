## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Troubleshooting mindset**

Effective troubleshooting starts with a simple rule: verify the symptom, isolate the layer, then confirm the configuration and the path. In CCNA scenarios, many failures come from a small set of mistakes: wrong VLAN assignment, missing trunk allowance, bad IP addressing, incorrect default gateway, ACL placement errors, NAT role mismatches, or routing adjacency problems.

A practical workflow is:

1. Identify the scope: one host, one VLAN, one subnet, or the whole network.
2. Check Layer 1 and Layer 2 first: cabling, link status, speed/duplex, VLANs, trunks, and EtherChannel.
3. Verify Layer 3: IP address, mask, default gateway, routing table, and neighbor relationships.
4. Inspect policy features: ACLs, NAT, DHCP relay, STP, and wireless settings.
5. Confirm with targeted tests: `ping`, `traceroute`, `show` commands, and logs.

**Common verification commands**

These commands help you quickly narrow the fault domain:

```bash
show ip interface brief
show interfaces status
show interfaces trunk
show vlan brief
show mac address-table
show cdp neighbors
show cdp neighbors detail
show lldp neighbors
show lldp neighbors detail
show ip route
show ip arp
show logging
```

- `show cdp neighbors` and `show lldp neighbors` help you confirm what is physically connected and detect cabling or topology mismatches.
- `show cdp neighbors detail` and `show lldp neighbors detail` can reveal management IPs, platform names, and interface mappings.
- `show logging` often exposes interface flaps, STP events, DHCP failures, ACL drops, and err-disable causes.

**Layer 1 and Layer 2 issues**

Many “network down” problems are actually local switching problems. If a host cannot reach its gateway, first confirm the access port, VLAN membership, and trunk path to the gateway SVI or router-on-a-stick subinterface.

Common mistakes include:

- Wrong access VLAN on the switchport.
- Trunk not allowing the required VLAN.
- Native VLAN mismatch.
- Disabled interface or err-disabled port.
- Speed/duplex mismatch on older or mixed environments.
- Incorrect EtherChannel settings.

Useful commands:

```bash
show interfaces status
show vlan brief
show interfaces trunk
show etherchannel summary
show spanning-tree vlan 10
show interfaces gigabitEthernet0/1 switchport
```

- `show interfaces status` quickly shows whether a port is connected, disabled, or err-disabled.
- `show vlan brief` confirms whether the access port is in the expected VLAN.
- `show interfaces trunk` verifies trunking mode, native VLAN, and allowed VLAN list.
- `show etherchannel summary` confirms whether member links are bundled correctly.

> If a port is err-disabled, check the interface status, the err-disable reason if available on your platform, and the logs. Common causes include BPDU Guard, port security violations, link-flap, and UDLD events.

**Spanning Tree Protocol troubleshooting**

<abbr title="Spanning Tree Protocol">STP</abbr> prevents loops, but misconfiguration can create unexpected blocking or instability. In CCNA scenarios, you usually troubleshoot root bridge placement, blocked ports, and access ports that should not participate in topology changes.

Symptoms include:

- A host on one switch cannot reach another host even though links are up.
- A trunk is forwarding on one side but blocking on the other due to topology changes.
- An access port goes into blocking or err-disabled after receiving BPDUs.
- A non-root switch unexpectedly becomes root because of bridge priority settings.

Verification commands:

```bash
show spanning-tree
show spanning-tree vlan 10
show spanning-tree interface gigabitEthernet0/1 detail
show spanning-tree root
```

- `show spanning-tree vlan <id>` shows the root bridge, port roles, and states for that VLAN.
- `show spanning-tree interface <int> detail` helps you see whether the port is forwarding, blocking, or receiving topology changes.
- `show spanning-tree root` is useful when you want to confirm whether the intended switch is actually the root.

Common fixes:

- Set the intended root bridge with a lower bridge priority.
- Enable PortFast on access ports only.
- Use BPDU Guard on edge ports to protect against accidental switch connections.
- Verify trunk links between switches are not accidentally configured as access ports.

**EtherChannel and trunk aggregation**

EtherChannel bundles multiple physical links into one logical link. Misconfiguration often happens when the member interfaces do not match or when the port-channel mode does not align on both ends.

Common issues:

- Speed, duplex, or trunk settings differ across member links.
- One side uses LACP while the other uses static or PAgP.
- The allowed VLAN list differs between the two ends.
- The physical interfaces are not added to the same channel group.
- A routed EtherChannel is configured inconsistently with Layer 2 settings.

Typical verification:

```bash
show etherchannel summary
show interfaces port-channel 1
show lacp neighbor
show pagp neighbor
```

For a Layer 3 EtherChannel on a multilayer switch, you typically configure the logical port-channel as routed with `no switchport`, then assign the IP address to the port-channel interface. The member interfaces inherit the channel behavior.

Example:

```bash
interface range gigabitEthernet1/0/1 - 2
 channel-group 1 mode active

interface port-channel1
 no switchport
 ip address 10.1.1.1 255.255.255.252
```

- Use `mode active` for LACP when you want negotiation.
- Keep member interface settings consistent.
- Verify the port-channel itself, not just the physical links.

**IP addressing and gateway mistakes**

A large number of CCNA troubleshooting questions reduce to bad addressing. If a host can reach local devices but not remote networks, check the subnet mask and default gateway first.

Common mistakes:

- Wrong IP address or subnet mask.
- Default gateway missing or pointing to the wrong router/SVI.
- Overlapping subnets on different interfaces.
- Duplicate IP addresses.
- Incorrect SVI address or shutdown SVI.

Useful checks:

```bash
show ip interface brief
show running-config interface vlan 10
show ip route
show ip arp
ping 192.168.10.1
traceroute 8.8.8.8
```

- `show ip route` confirms whether the local device knows a route to the destination.
- `show ip arp` helps you verify whether the next hop is reachable at Layer 2.
- `traceroute` helps you identify where traffic stops.

**Routing troubleshooting**

Routing problems usually fall into one of three categories: missing routes, incorrect next hops, or failed adjacency formation. For CCNA, you should be able to interpret route codes and recognize why a route is absent.

Common route-related symptoms:

- A static route points to an unreachable next hop.
- A default route is missing.
- An OSPF neighbor never reaches full adjacency.
- An EIGRP neighbor is stuck in a non-fully-adjacent state.
- A route exists in the routing table but traffic still fails due to ACLs or NAT.

Verification commands:

```bash
show ip route
show ip route static
show ip protocols
show ip ospf neighbor
show ip ospf interface brief
show ip eigrp neighbors
show ip cef
```

- `show ip route` tells you whether the route is connected, static, or learned dynamically.
- `show ip protocols` confirms which routing protocols are active and what networks are being advertised.
- `show ip ospf neighbor` helps you see whether OSPF adjacencies are forming.
- `show ip ospf interface brief` helps you confirm the interface is participating in OSPF.

Common OSPF issues:

- Area mismatch.
- Hello/dead timer mismatch.
- Authentication mismatch.
- Passive interface on the wrong interface.
- Network type mismatch in some designs.
- MTU mismatch in some environments.

Common static route issues:

- Wrong destination network or mask.
- Wrong next-hop IP.
- Recursive route resolution failure because the next hop is not reachable.
- Administrative distance conflict with another route.

Example static route:

```bash
ip route 203.0.113.0 255.255.255.0 10.1.1.2
ip route 0.0.0.0 0.0.0.0 10.1.1.2
```

If the next hop is not reachable, the route may not install or traffic may fail even though the configuration looks correct.

**ACL misconfigurations**

<abbr title="Access Control List">ACL</abbr>s are a frequent source of “network works for some things but not others” problems. Remember that ACLs are processed top-down, first match wins, and there is an implicit deny at the end.

Common mistakes:

- ACL applied in the wrong direction.
- ACL applied to the wrong interface.
- Wrong source or destination address.
- Wildcard mask error.
- Missing permit for required traffic before the implicit deny.
- Overly broad deny statement placed too early.

Example of a common wildcard mistake:

```bash
access-list 10 permit 192.168.10.0 0.0.0.255
```

If you intended to match only one host, this permits the entire /24. For a single host, use:

```bash
access-list 10 permit 192.168.10.25 0.0.0.0
```

Extended ACL example:

```bash
ip access-list extended WEB-FILTER
 permit tcp 192.168.10.0 0.0.0.255 host 203.0.113.10 eq 80
 permit tcp 192.168.10.0 0.0.0.255 host 203.0.113.10 eq 443
 deny ip any any log
```

- Place extended ACLs close to the source when possible.
- Place standard ACLs closer to the destination when you need to limit source addresses only.
- Verify the interface and direction with `show running-config interface <int>` and `show access-lists`.

Useful commands:

```bash
show access-lists
show ip interface gigabitEthernet0/1
show running-config | section access-list
```

**DHCP troubleshooting**

<abbr title="Dynamic Host Configuration Protocol">DHCP</abbr> issues often appear as clients stuck on self-assigned addresses or unable to renew leases. The problem may be on the client, the relay path, or the server.

Common causes:

- DHCP server pool missing the correct network statement.
- Excluded addresses consume the usable range.
- Wrong default gateway option.
- Relay agent not configured on the client VLAN interface.
- ACLs blocking the actual relay traffic path.
- DHCP snooping dropping legitimate offers or replies.
- Server unreachable due to routing or interface issues.

Relay example:

```bash
interface vlan 10
 ip address 192.168.10.1 255.255.255.0
 ip helper-address 10.10.10.5
```

A client broadcast on VLAN 10 is relayed by the router or SVI to the DHCP server as unicast traffic. Troubleshooting should focus on the relay interface, the path to the server, and the server’s ability to return the offer to the relay agent.

Useful commands:

```bash
show ip dhcp binding
show ip dhcp pool
show ip dhcp conflict
show running-config | include ip helper-address
debug ip dhcp server events
debug ip dhcp server packet
```

- `show ip dhcp binding` confirms assigned leases.
- `show ip dhcp pool` shows pool utilization and excluded addresses.
- `show ip dhcp conflict` helps identify duplicate address conflicts.
- `debug ip dhcp server events` can help in a lab or controlled environment when you need to see server-side behavior.

> Use debugging carefully on production devices. It can be CPU-intensive and may affect forwarding.

**NAT troubleshooting**

<abbr title="Network Address Translation">NAT</abbr> problems usually come from incorrect inside/outside roles, ACL mismatch, or missing route reachability. For CCNA, you should connect the configuration to the symptom: inside hosts cannot reach the Internet, or return traffic never comes back.

Common mistakes:

- Inside and outside interfaces reversed.
- NAT ACL does not match the inside source subnet.
- Overload configuration missing.
- No route to the NAT outside next hop.
- Overlapping address space or incorrect translation pool.

Example PAT configuration:

```bash
access-list 1 permit 192.168.10.0 0.0.0.255

interface gigabitEthernet0/0
 ip address 192.168.10.1 255.255.255.0
 ip nat inside

interface gigabitEthernet0/1
 ip address 203.0.113.2 255.255.255.252
 ip nat outside

ip nat inside source list 1 interface gigabitEthernet0/1 overload
ip route 0.0.0.0 0.0.0.0 203.0.113.1
```

Verification commands:

```bash
show ip nat translations
show ip nat statistics
show running-config | include ip nat
show access-lists
```

- `show ip nat translations` confirms whether translations are being created.
- `show ip nat statistics` helps you see hits and misses.
- If translations do not appear, check the ACL match, interface roles, and routing.

**Wireless troubleshooting**

Wireless issues are often caused by mismatched SSID settings, VLAN mapping errors, RF problems, or controller/AP join failures. Platform-specific commands vary, so verify whether you are working on a controller-based system such as a WLC or Catalyst 9800, or on a simpler autonomous AP setup.

Common wireless misconfigurations:

- Wrong SSID-to-VLAN mapping.
- Incorrect security settings, such as WPA2/WPA3 mismatch or wrong pre-shared key.
- AP channel or transmit power causing poor coverage or co-channel interference.
- AP not joining the controller.
- Client authentication failure due to RADIUS or policy mismatch.
- DHCP failure on the wireless VLAN.

Typical controller-based verification commands may include:

```bash
show wlan summary
show ap summary
show client summary
show wireless client mac-address <mac>
show ap join stats summary
```

These commands are platform-dependent and are commonly seen on WLC or Catalyst 9800 platforms, not on every IOS device.

Practical checks:

- Confirm the SSID maps to the correct VLAN or policy profile.
- Verify the AP has joined the controller and is registered.
- Check whether the client receives an IP address from the correct subnet.
- Inspect RF conditions if clients connect but performance is poor.

**Troubleshooting with neighbor discovery**

<abbr title="Cisco Discovery Protocol">CDP</abbr> and <abbr title="Link Layer Discovery Protocol">LLDP</abbr> are useful for identifying cabling mistakes, wrong switch ports, and unexpected neighbors. They are especially helpful when a diagram and the live network do not match.

Useful commands:

```bash
show cdp neighbors
show cdp neighbors detail
show lldp neighbors
show lldp neighbors detail
```

Use these outputs to confirm:

- Which device is connected to which port.
- Whether a router is connected where you expected a switch.
- Whether a link is patched to the wrong interface.
- Whether a neighbor’s management IP matches the documentation.

**A practical troubleshooting sequence**

When a host cannot reach a destination, use a repeatable process instead of guessing.

1. Check the host IP settings with `ipconfig /all` or `ip addr`.
2. Verify link and VLAN status on the switch with `show interfaces status` and `show vlan brief`.
3. Confirm the gateway interface is up with `show ip interface brief`.
4. Check neighbor discovery with `show cdp neighbors detail` or `show lldp neighbors detail`.
5. Inspect the routing table with `show ip route`.
6. Review ACLs, NAT, DHCP relay, or STP if the basic path looks correct.
7. Test with `ping` and `traceroute` at each hop.

**Exam-style symptom patterns**

- A host can ping its default gateway but not remote networks: check routing, NAT, or ACLs.
- A host cannot get an IP address: check DHCP relay, server pool, ACLs, and snooping.
- Two switches connect but only some VLANs pass: check trunk allowed VLANs and native VLAN.
- A port goes down after connecting a switch: check BPDU Guard, port security, or err-disable causes.
- A route is missing from the table: check the routing protocol, static route next hop, or interface state.
- Wireless clients connect but cannot reach resources: check SSID mapping, VLAN assignment, DHCP, and ACLs.

**Key takeaways**

Troubleshooting is fastest when you verify the simplest layers first and use the right command for the suspected feature. Most CCNA problems are caused by a small set of configuration errors, and the output from `show` commands usually points directly to the fault domain.

Focus on these habits:

- Confirm the physical and logical topology before changing config.
- Match the symptom to the layer and feature most likely involved.
- Read command output carefully, especially interface state, VLAN membership, route codes, neighbor states, and ACL counters.
- Change one thing at a time and retest.
