## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**What this review covers**

This module review ties together the major CCNA domains you are expected to understand at a practical level: network fundamentals, switching, routing, IP services, security, wireless, and automation. The goal is not memorization alone. You need to recognize how features interact, how to verify them with CLI commands, and how to troubleshoot common failures.

A good exam answer usually depends on context:
- Is the issue Layer 2 or Layer 3?
- Is the device acting as a switch, router, or wireless controller?
- Is the problem caused by addressing, policy, or topology?
- Which verification command proves the behavior you expect?

**Core troubleshooting mindset**

Use a repeatable process:
1. Identify the symptom.
2. Check the local interface and addressing state.
3. Verify Layer 2 forwarding and VLAN membership.
4. Verify Layer 3 reachability and routing.
5. Check policy features such as ACLs, NAT, or port security.
6. Confirm the result with a test and a show command.

This approach matters because many CCNA questions are scenario-based. The correct answer is often the feature that best matches the failure domain, not the first command you remember.

**Network fundamentals and addressing**

IPv4 and IPv6 addressing are foundational because nearly every other topic depends on them. You should be able to identify network, host, and broadcast behavior in IPv4, and understand how IPv6 removes broadcast while using multicast and neighbor discovery instead.

- **Subnetting** determines which hosts are local and which require a router.
- **Default gateways** provide the next hop off the local subnet.
- **Loopback interfaces** are useful for stable router identifiers and testing.
- **Link-local addresses** in IPv6 are automatically used for local neighbor communication once IPv6 is enabled on an interface, though platform behavior can vary.

Common IPv6 address types:
- **Global unicast**: routable on the IPv6 internet or enterprise network.
- **Link-local**: used on the local link, often for next-hop and neighbor discovery.
- **Unique local**: private addressing for internal use.
- **Multicast**: replaces broadcast-style discovery in IPv6.

A practical IPv6 example on a router:

```bash
conf t
ipv6 unicast-routing
interface g0/0
 ipv6 address 2001:db8:10:1::1/64
 ipv6 address fe80::1 link-local
 no shutdown
end
```

Verification commands:
```bash
show ipv6 interface brief
show ipv6 route
show ipv6 neighbors
```

**IPv6 host configuration: SLAAC and DHCPv6**

You should know the difference between Stateless Address Autoconfiguration (SLAAC) and DHCPv6.

- **SLAAC** lets a host build its own IPv6 address using router advertisements.
- **Stateless DHCPv6** provides extra information such as DNS servers, but the host still uses SLAAC for its address.
- **Stateful DHCPv6** assigns the IPv6 address itself, similar to DHCP for IPv4.

Typical router-side examples:

```bash
conf t
ipv6 unicast-routing

ipv6 dhcp pool STATLESS-INFO
 dns-server 2001:db8:53::53
 domain-name example.local

interface g0/0
 ipv6 address 2001:db8:10:1::1/64
 ipv6 nd other-config-flag
 ipv6 dhcp server STATLESS-INFO
end
```

For stateful DHCPv6, the router advertises managed addressing and the DHCPv6 server provides the address pool. In exam scenarios, look for the flags in router advertisements:
- **M flag**: managed address configuration
- **O flag**: other configuration

**Switching, VLANs, and trunking**

VLANs segment a Layer 2 network into separate broadcast domains. This improves security, reduces unnecessary flooding, and supports logical separation without requiring separate physical switches.

Key ideas:
- **Access ports** carry traffic for one VLAN.
- **Trunk ports** carry multiple VLANs between switches or to a router or firewall.
- **802.1Q** is the standard trunk encapsulation used in modern Cisco environments.
- **Native VLAN** traffic is untagged on an 802.1Q trunk.

Example configuration:

```bash
conf t
vlan 10
 name USERS
vlan 20
 name VOICE

interface g0/1
 switchport mode access
 switchport access vlan 10

interface g0/2
 switchport mode trunk
 switchport trunk allowed vlan 10,20
end
```

Useful verification:
```bash
show vlan brief
show interfaces trunk
show interfaces switchport
```

A common exam trap is assuming a trunk is enough by itself. The VLAN must exist on both sides, the trunk must allow it, and the end device or subinterface must be configured correctly.

**Inter-VLAN routing**

Inter-VLAN routing lets devices in different VLANs communicate through a Layer 3 device. Two common designs appear in CCNA scenarios:

- **Router-on-a-stick**: one router interface uses subinterfaces with 802.1Q tagging.
- **Layer 3 switch SVIs**: switched virtual interfaces route between VLANs on the switch.

Router-on-a-stick example:

```bash
conf t
interface g0/0
 no shutdown

interface g0/0.10
 encapsulation dot1Q 10
 ip address 192.168.10.1 255.255.255.0

interface g0/0.20
 encapsulation dot1Q 20
 ip address 192.168.20.1 255.255.255.0
end
```

Layer 3 switch example:

```bash
conf t
ip routing

interface vlan 10
 ip address 192.168.10.1 255.255.255.0
 no shutdown

interface vlan 20
 ip address 192.168.20.1 255.255.255.0
 no shutdown
end
```

If hosts in different VLANs cannot communicate, check:
- VLAN membership on access ports
- trunk status and allowed VLAN list
- SVI or subinterface IP addressing
- default gateway on the host
- whether `ip routing` is enabled on a multilayer switch

**STP and loop prevention**

Spanning Tree Protocol (STP) prevents Layer 2 loops by blocking redundant paths. Without STP, a loop can cause broadcast storms, MAC address flapping, and unstable forwarding.

Important concepts:
- **Root bridge**: the switch with the lowest bridge ID becomes the STP reference point.
- **Port roles**: root, designated, and blocked/alternate depending on the STP variant.
- **PortFast**: speeds up edge-port forwarding for end devices.
- **BPDU Guard**: protects PortFast ports from accidental switch connections.

Verification commands:
```bash
show spanning-tree
show spanning-tree vlan 10
show spanning-tree interface g0/1 detail
```

A practical access-port configuration:

```bash
conf t
interface range f0/1-24
 spanning-tree portfast
 spanning-tree bpduguard enable
end
```

> Use PortFast only on ports connected to end devices. Do not enable it on links that may connect to another switch unless the design explicitly requires it.

**EtherChannel**

EtherChannel bundles multiple physical links into one logical link for increased bandwidth and redundancy. It also simplifies STP because STP sees one logical port instead of several parallel links.

Common negotiation protocols:
- **LACP**: Link Aggregation Control Protocol, standardized and widely used.
- **PAgP**: Port Aggregation Protocol, Cisco-proprietary.

Example using LACP:

```bash
conf t
interface range g0/1-2
 channel-group 1 mode active

interface port-channel 1
 switchport mode trunk
 switchport trunk allowed vlan 10,20
end
```

Verification:
```bash
show etherchannel summary
show interfaces port-channel 1
show lacp neighbor
```

For EtherChannel to form correctly, the member links must match in key settings such as speed, duplex, trunking mode, and allowed VLANs.

**Routing basics and OSPF**

Open Shortest Path First (OSPF) is a link-state routing protocol. Routers build a topology map from Link-State Advertisements (LSAs) and run the Shortest Path First algorithm to choose best paths.

Why OSPF matters:
- It scales better than simple distance-vector routing in larger networks.
- It converges quickly after a topology change.
- It supports hierarchical design with areas.

CCNA-level essentials:
- **Router ID** uniquely identifies the OSPF router.
- **Area 0** is the backbone area.
- **Neighbors** must agree on key parameters such as area, timers, and network type.
- **Passive interfaces** advertise networks without forming neighbors on user-facing links.

Basic OSPFv2 example:

```bash
conf t
router ospf 1
 router-id 1.1.1.1
 network 10.0.0.0 0.0.0.255 area 0
 network 192.168.10.0 0.0.0.255 area 0
 passive-interface g0/1
end
```

Verification:
```bash
show ip ospf neighbor
show ip route ospf
show ip protocols
show ip ospf interface brief
```

If OSPF neighbors do not form, check:
- matching area numbers
- matching subnet and mask
- interface state
- hello and dead timers
- authentication, if configured
- whether the interface is passive

**ACLs and traffic filtering**

Access Control Lists (ACLs) filter traffic based on permit and deny statements. They are used for security, traffic control, and policy enforcement.

Types to know:
- **Standard ACLs**: match source IP only.
- **Extended ACLs**: match source, destination, protocol, and ports.

Placement guidance is a rule of thumb, not an absolute law:
- Place **extended ACLs** close to the source when you want to stop unwanted traffic early.
- Place **standard ACLs** close to the destination when source-based filtering could unintentionally block other traffic.

Example extended ACL:

```bash
conf t
ip access-list extended WEB-FILTER
 permit tcp 192.168.10.0 0.0.0.255 any eq 80
 permit tcp 192.168.10.0 0.0.0.255 any eq 443
 deny ip any any log

interface g0/0
 ip access-group WEB-FILTER in
end
```

Verification:
```bash
show access-lists
show ip interface g0/0
```

Remember that ACLs are processed top-down, first match wins, and there is an implicit deny at the end. If traffic is unexpectedly blocked, check the order of entries before changing the whole policy.

**Port security**

Port security limits which MAC addresses can use a switch port. It is commonly used on access ports to reduce the risk of unauthorized devices.

Typical features:
- **Maximum MAC addresses** per port
- **Sticky MAC** learning, which saves learned addresses into the running configuration
- **Violation modes**:
  - **protect**: drops frames silently
  - **restrict**: drops frames and logs a violation
  - **shutdown**: err-disables the port

Example:

```bash
conf t
interface f0/10
 switchport mode access
 switchport port-security
 switchport port-security maximum 2
 switchport port-security mac-address sticky
 switchport port-security violation restrict
end
```

Verification:
```bash
show port-security interface f0/10
show port-security address
show interfaces status err-disabled
```

Port security is most useful on user-facing access ports, not on trunks or uplinks.

**NAT and PAT**

Network Address Translation (NAT) translates one IP address space to another. It is commonly used to let private IPv4 hosts reach external networks.

Main types:
- **Static NAT**: one private address maps to one public address.
- **Dynamic NAT**: a pool of public addresses is assigned as needed.
- **PAT**: Port Address Translation, also called NAT overload, maps many inside hosts to one public address using different source ports.

Example PAT configuration:

```bash
conf t
access-list 1 permit 192.168.10.0 0.0.0.255

interface g0/0
 ip address 192.168.10.1 255.255.255.0
 ip nat inside

interface g0/1
 ip address 203.0.113.2 255.255.255.252
 ip nat outside

ip nat inside source list 1 interface g0/1 overload
end
```

Verification:
```bash
show ip nat translations
show ip nat statistics
show ip interface brief
```

> NAT requires correct inside and outside interface roles, and the outside path must be reachable through routing. If the router has no valid route to the upstream network, NAT alone will not restore connectivity.

**Wireless fundamentals**

Wireless LANs depend on radio frequency behavior, channel planning, and client roaming. CCNA questions often focus on practical tradeoffs rather than deep RF math.

Important concepts:
- **2.4 GHz** offers longer range and better wall penetration, but more interference and fewer non-overlapping channels.
- **5 GHz** offers more channels and usually better performance, but shorter range.
- **Roaming** lets a client move between access points while maintaining connectivity.
- **SSID** identifies the wireless network name.
- **BSSID** identifies the access point radio interface.

Practical implications:
- Use 2.4 GHz when coverage matters more than throughput.
- Use 5 GHz when capacity and performance matter more than range.
- Plan channels to reduce co-channel interference.
- Place access points to avoid dead zones and excessive overlap.

A simple wireless design rule:
- Use overlapping coverage for roaming.
- Avoid excessive transmit power that causes sticky clients.
- Match security settings across APs in the same WLAN.

**Wireless security and deployment**

Wireless security commonly uses WPA2 or WPA3 with strong authentication. In enterprise environments, 802.1X with RADIUS is preferred because it supports centralized identity and policy control.

Common terms:
- **PSK**: Pre-Shared Key, a shared password model.
- **802.1X**: port-based access control using an authentication server.
- **RADIUS**: Remote Authentication Dial-In User Service, often used for AAA.
- **AAA**: Authentication, Authorization, and Accounting.

Conceptual deployment example:
- Small office: WPA2/WPA3-Personal with PSK
- Enterprise: WPA2/WPA3-Enterprise with 802.1X and RADIUS

Wireless troubleshooting checklist:
- Confirm SSID and security settings match
- Check channel overlap and signal strength
- Verify client authentication method
- Confirm DHCP and gateway reachability after association

**IP services**

CCNA expects you to understand common services that support network operation.

Key services:
- **DHCP** assigns IPv4 addresses dynamically.
- **DNS** resolves names to IP addresses.
- **NTP** synchronizes time across devices.
- **Syslog** centralizes log messages.
- **SNMP** supports monitoring and management.

DHCP relay example:

```bash
conf t
interface vlan 10
 ip helper-address 192.168.1.10
end
```

NTP example:
```bash
conf t
ntp server 192.168.1.20
end
```

Verification:
```bash
show ip dhcp binding
show ip dhcp pool
show ntp status
show logging
```

If clients receive an address but cannot reach resources, check the default gateway, DNS server, and whether the DHCP scope includes the correct options.

**Security fundamentals**

Security on the CCNA level focuses on access control, segmentation, and basic device hardening.

Good practices:
- Use strong passwords and encrypted management access.
- Disable unused ports.
- Restrict management access with ACLs.
- Use SSH instead of Telnet.
- Apply port security on access ports.
- Separate user, voice, and guest traffic with VLANs.

SSH example:

```bash
conf t
hostname SW1
ip domain-name example.local
username admin secret StrongPassword123
crypto key generate rsa modulus 2048
line vty 0 4
 login local
 transport input ssh
end
```

Verification:
```bash
show ip ssh
show running-config | section line vty
```

**Automation and programmability**

Automation reduces repetitive configuration work and improves consistency. At the CCNA level, you should understand the role of APIs, data models, and controller-based networking.

Core ideas:
- **Controller-based networking** centralizes policy and device management.
- **APIs** let software interact with network devices.
- **NETCONF** and **RESTCONF** are common management protocols.
- **YANG** models define structured data for configuration and state.
- **Telemetry** streams operational data for monitoring and analytics.

High-level comparison:

| Technology | Purpose | Typical Use |
|---|---|---|
| NETCONF | Structured configuration and state management | Programmatic device changes |
| RESTCONF | REST-style access to YANG-modeled data | Web-friendly automation |
| YANG | Data modeling language | Defines device configuration/state |
| Telemetry | Streaming operational data | Monitoring and analytics |

A simple RESTCONF-style request conceptually looks like this:

```bash
curl -u admin:password \
  -H "Accept: application/yang-data+json" \
  https://router.example.local/restconf/data/ietf-interfaces:interfaces
```

You do not need to memorize every API detail for CCNA, but you should know why automation matters:
- It improves repeatability.
- It reduces manual errors.
- It supports large-scale policy deployment.
- It helps operations teams detect issues faster.

**Common verification commands to remember**

These commands appear often in troubleshooting scenarios:

```bash
show ip interface brief
show running-config
show vlan brief
show interfaces trunk
show spanning-tree
show etherchannel summary
show ip route
show ip ospf neighbor
show access-lists
show ip nat translations
show port-security interface f0/10
show cdp neighbors detail
show lldp neighbors detail
```

CDP, or Cisco Discovery Protocol, is useful for discovering Cisco neighbors and interface mappings. LLDP, or Link Layer Discovery Protocol, is vendor-neutral and often more useful in mixed-vendor environments. In practice, use whichever discovery protocol is enabled and available on the devices you are troubleshooting.

**How to think about exam scenarios**

When you see a problem, classify it before you configure anything:
- If hosts in the same VLAN cannot communicate, check switching, VLAN membership, and port security.
- If hosts in different VLANs cannot communicate, check inter-VLAN routing and gateways.
- If a route is missing, check the routing protocol, static route, or default route.
- If internet access fails but internal access works, check NAT, ACLs, and upstream routing.
- If wireless clients connect but cannot pass traffic, check DHCP, security, and VLAN mapping.
- If management access fails, check SSH, VTY lines, ACLs, and IP reachability.

**Quick review checklist**

- Know the difference between access and trunk ports.
- Know how STP prevents loops and how PortFast and BPDU Guard protect edge ports.
- Know how EtherChannel bundles links and why matching settings matter.
- Know OSPF neighbor formation basics and how to verify routes.
- Know standard vs extended ACL behavior and placement guidance.
- Know port security modes and violation actions.
- Know NAT types and the role of inside/outside interfaces.
- Know IPv6 addressing, SLAAC, and DHCPv6 flags.
- Know wireless band tradeoffs and roaming basics.
- Know the purpose of NETCONF, RESTCONF, YANG, and telemetry.

**Final exam perspective**

CCNA questions often test whether you can connect a symptom to the correct layer and feature. The best answers are usually the ones that match the network design, the protocol behavior, and the verification output. If you can explain why a command works, not just what it does, you are ready for scenario-based questions.
