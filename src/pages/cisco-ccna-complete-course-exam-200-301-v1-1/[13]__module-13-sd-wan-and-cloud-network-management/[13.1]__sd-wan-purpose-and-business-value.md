## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**SD-WAN overview**

<abbr title="Software-Defined Wide Area Network">SD-WAN</abbr> is a modern approach to WAN (Wide Area Network) connectivity that uses centralized policy and software control to steer traffic across multiple transport types. Instead of treating every site-to-site path the same, SD-WAN lets you define how applications should use the network based on business intent.

Traditional WAN designs often rely on a hub-and-spoke model with expensive private circuits and limited flexibility. SD-WAN changes that by separating control from forwarding, making it easier to use broadband, MPLS (Multiprotocol Label Switching), LTE (Long-Term Evolution), and 5G together.

**Why organizations adopt SD-WAN**

The main business driver is to improve application experience while reducing WAN complexity and cost. As more traffic moves to cloud and SaaS (Software as a Service) platforms, sending all traffic back to a central data center can add latency and create bottlenecks.

- **Application awareness** helps the network treat voice, video, ERP (Enterprise Resource Planning), and web traffic differently.
- **Transport flexibility** lets you combine multiple links and choose the best path per application.
- **Centralized policy** reduces the need to configure every branch device manually.
- **Operational efficiency** improves because changes can be pushed consistently across many sites.
- **Cost optimization** is possible when you use lower-cost Internet links alongside or instead of private WAN circuits.

**Business value in practical terms**

SD-WAN is not just a routing technology; it is a way to align network behavior with business priorities. For example, a company may want video meetings to use the lowest-latency path, while bulk backups can use a cheaper link during off-peak hours.

This policy-driven model supports:

- Faster branch deployment
- Better user experience for cloud applications
- Easier migration from legacy WAN designs
- Improved resilience when one circuit fails or degrades
- More consistent security and segmentation across sites

**Core SD-WAN concepts**

SD-WAN platforms typically include a centralized controller, edge devices at branch or data center sites, and a management plane for policy and monitoring. The exact vendor implementation varies, but the architectural goals are similar.

- **Overlay network**: A logical network built on top of physical transports using tunnels.
- **Underlay network**: The physical transport network, such as Internet, MPLS, or LTE.
- **Policy-based forwarding**: Traffic is steered according to application, SLA (Service Level Agreement), or business rules.
- **Path monitoring**: The system measures latency, jitter, and loss to decide which path is best.
- **Dynamic failover**: Traffic can move to another path when performance drops or a link fails.

A simple conceptual topology looks like this:

```text
Branch A ---- Internet ----\
Branch B ---- MPLS ---------+---- SD-WAN Controller / Orchestrator
Branch C ---- LTE ----------/
```

In practice, branch edges build encrypted tunnels over one or more transports and use policy to decide which tunnel carries each flow.

**How SD-WAN improves application performance**

SD-WAN can inspect traffic and classify it by application or destination. Once classified, it can apply rules such as:

- Send voice over the path with the lowest jitter
- Send interactive SaaS traffic over the lowest-latency Internet path
- Send backups over the cheapest available link
- Prefer private transport for sensitive or high-priority workloads when required

This is especially useful when a single link is not enough to meet all needs. Instead of overprovisioning one expensive circuit, you can use multiple links and let policy choose the best one per flow.

**Resiliency and failover**

A major advantage of SD-WAN is faster and smarter failover. Traditional routing may react to link failure, but it does not always react well to partial degradation such as high loss or jitter. SD-WAN can detect poor path quality and move traffic before users notice severe impact.

- **Hard failure**: A circuit goes down completely.
- **Brownout**: A circuit is up but performance is poor.
- **SLA-based steering**: Traffic moves when measured metrics exceed thresholds.

Example policy logic:

```text
If application = voice
  and latency < 150 ms
  and jitter < 30 ms
  and loss < 1%
then use primary Internet path
else use MPLS path
```

This type of logic helps maintain service quality even when one transport becomes unreliable.

**Security and segmentation**

Many SD-WAN solutions include built-in encryption and segmentation features. Traffic between sites is commonly protected with IPsec (Internet Protocol Security) tunnels. Segmentation allows different business units, guest traffic, or sensitive workloads to remain logically separated on the same physical infrastructure.

- **Encryption** protects data in transit across public links.
- **Segmentation** limits traffic mixing between groups or applications.
- **Central policy** helps enforce consistent security rules across branches.

> SD-WAN does not automatically replace a full security architecture. You still need firewalls, identity controls, and secure cloud access where appropriate.

**Common deployment models**

SD-WAN can be deployed in several ways depending on the organization’s size and goals.

- **Branch-to-branch**: Sites communicate directly over the overlay when needed.
- **Branch-to-data center**: Branches reach centralized services through controlled paths.
- **Branch-to-cloud**: Traffic goes directly to SaaS or cloud services instead of hairpinning through headquarters.
- **Hybrid WAN**: SD-WAN overlays multiple transport types, often alongside existing MPLS.

A common migration approach is to keep the legacy WAN in place while introducing SD-WAN at selected sites, then gradually expand coverage.

**Example CLI and configuration concepts**

Vendor syntax varies, but the operational ideas are similar. You may configure interfaces, tunnels, and policy rules on an edge device.

```bash
show ip interface brief
show route
show vpn sessiondb
show sdwan control connections
```

A simplified policy example might look like this:

```text
application-policy
  app voice
    preferred-path internet1
    backup-path mpls1
  app saas
    preferred-path internet1
  app backup
    preferred-path mpls1
```

A tunnel or interface configuration may include:

```bash
interface GigabitEthernet0/0
 description Internet transport
 ip address dhcp
 no shutdown

interface Tunnel0
 description SD-WAN overlay tunnel
 ip address 10.10.10.1 255.255.255.252
 tunnel source GigabitEthernet0/0
 tunnel mode ipsec ipv4
```

These examples are conceptual, but they reflect the real tasks you perform: bring up the underlay, establish secure overlay tunnels, and apply traffic policy.

**Operational benefits for IT teams**

SD-WAN reduces repetitive manual work and improves visibility into WAN behavior.

- **Centralized management** simplifies rollout and policy changes.
- **Telemetry** provides insight into path quality and application usage.
- **Template-based deployment** speeds up branch provisioning.
- **Consistent policy enforcement** reduces configuration drift.
- **Troubleshooting visibility** improves because you can see which path and policy a flow used.

Useful troubleshooting commands often include:

```bash
show interface
show ip route
show ip ospf neighbor
show logging
ping 8.8.8.8 source GigabitEthernet0/0
traceroute 1.1.1.1
```

In SD-WAN environments, you also check tunnel status, control connections, and SLA metrics to determine whether the issue is with the underlay, overlay, or policy.

**Business outcomes to remember**

When you explain SD-WAN in an exam or interview, connect the technology to business goals:

- Lower WAN cost by using commodity Internet links
- Improve cloud and SaaS performance by avoiding unnecessary backhaul
- Increase uptime with multiple active transports and intelligent failover
- Simplify operations through centralized control and automation
- Support growth with faster branch turn-up and easier policy changes

**Key exam takeaways**

- SD-WAN uses software and centralized policy to manage WAN traffic.
- It builds an overlay on top of one or more underlay transports.
- It improves application performance by steering traffic based on business intent and path quality.
- It increases resilience by detecting failures and brownouts.
- It can reduce cost and operational complexity compared with traditional WAN designs.
- It commonly uses encryption, segmentation, and centralized orchestration.

**Quick comparison with traditional WAN**

| Aspect | Traditional WAN | SD-WAN |
|---|---|---|
| Path selection | Mostly static or routing-based | Policy-based and application-aware |
| Transport use | Often one primary private circuit | Multiple transports, including Internet |
| Cloud access | Often backhauled through HQ | Can go direct to cloud/SaaS |
| Management | Per-device configuration | Centralized orchestration |
| Failover | Link-down focused | Link-down and quality-aware |

**Study tip**

When you see SD-WAN in CCNA material, think in three layers:

1. The **underlay** provides physical connectivity.
2. The **overlay** creates secure logical tunnels.
3. The **policy** decides how traffic uses those tunnels.

That mental model helps you explain both the purpose and the business value clearly.
