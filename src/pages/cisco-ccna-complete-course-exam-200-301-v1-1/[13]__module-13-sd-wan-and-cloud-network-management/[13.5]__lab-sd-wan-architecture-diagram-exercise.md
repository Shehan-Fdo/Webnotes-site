## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Cisco SD-WAN architecture overview**

Cisco SD-WAN separates the network into a **control plane**, **management plane**, and **data plane**. In a diagram exercise, your goal is to identify which components belong to each plane and how traffic flows between them.

At a high level:

- **vManage** provides centralized management and monitoring.
- **vSmart** distributes control information such as routing policy and overlay intelligence.
- **vBond** assists with orchestration, discovery, and NAT traversal during onboarding.
- **WAN Edge** devices forward user traffic across the overlay and exchange routing/control information with the controllers.

A correct diagram usually shows the controllers in the cloud or data center and the WAN Edge routers at branch and hub sites.

**Core roles of SD-WAN components**

Use the following mental model when labeling a topology:

- **vManage**
  - Centralized GUI and API for configuration, monitoring, and troubleshooting.
  - Pushes templates and policy to the fabric.
  - Does not forward user data traffic.

- **vSmart**
  - Control-plane brain of the fabric.
  - Exchanges routing and policy information with WAN Edge devices.
  - Helps determine which overlay paths are allowed and preferred.

- **vBond**
  - First contact point for many new WAN Edge devices.
  - Helps devices discover the fabric and establish secure onboarding.
  - Assists with NAT traversal so devices behind private addressing can join the overlay.

- **WAN Edge**
  - Branch or hub router that participates in the SD-WAN fabric.
  - Builds secure overlay tunnels to other WAN Edge devices.
  - Uses underlay transport links such as Internet, MPLS, or LTE/5G.

> **Important:** vBond is primarily an orchestration and discovery component. Persistent control connections are normally maintained with **vSmart** and **vManage**, not with vBond as a steady-state control-plane peer.

**Planes and traffic flow**

A diagram exercise often tests whether you can separate the roles of each plane.

- **Management plane**
  - vManage to controllers and WAN Edge devices for configuration and monitoring.
  - Uses secure management sessions.
  - Supports templates, alarms, and statistics.

- **Control plane**
  - WAN Edge devices establish secure control connections to vSmart.
  - vSmart distributes overlay routing and policy.
  - vBond helps with initial discovery and reachability during onboarding.

- **Data plane**
  - User traffic flows through encrypted overlay tunnels between WAN Edge devices.
  - The overlay rides on top of the underlay transport network.
  - The tunnel mechanism is separate from liveliness monitoring.

A useful way to remember the distinction is:

- **Tunnels carry traffic**
- **Control connections exchange routing and policy**
- **BFD monitors path health**

**Overlay and underlay**

Cisco SD-WAN uses two layers:

- **Underlay**
  - The physical or service-provider network.
  - Examples include broadband Internet, MPLS, and cellular.
  - Must provide IP reachability between WAN Edge transport interfaces and controller endpoints.

- **Overlay**
  - The logical SD-WAN fabric built on top of the underlay.
  - Carries enterprise traffic in secure tunnels.
  - Abstracts the transport so policy can choose the best path.

In a diagram, label the underlay as the transport cloud and the overlay as the encrypted fabric between edges.

**Tunnel behavior and path monitoring**

WAN Edge devices form secure overlay tunnels to other WAN Edge devices. These tunnels are typically IPsec-based in Cisco SD-WAN deployments, depending on platform and release. The exact tunnel behavior can vary, but the exam-level concept is consistent: the overlay tunnel is what carries traffic.

BFD, or **Bidirectional Forwarding Detection**, is used to detect liveliness and measure path health quickly. It is not the tunnel mechanism itself.

- **IPsec/DTLS overlay tunnels**
  - Carry data and control traffic depending on design and platform behavior.
  - Provide encryption and secure transport across the underlay.

- **BFD**
  - Detects whether a path is up or degraded.
  - Helps the fabric react quickly to failures.
  - Supports fast convergence and path selection.

**Typical SD-WAN topology labels**

A common lab diagram includes these elements:

- **Branch site**
  - WAN Edge router
  - LAN switch
  - End hosts
  - Internet or MPLS access

- **Headquarters or data center**
  - WAN Edge router
  - Internal servers
  - Optional service insertion or firewall

- **Controllers**
  - vManage
  - vSmart
  - vBond

A simple conceptual diagram looks like this:

```text
                +----------------------+
                |   SD-WAN Controllers |
                | vManage vSmart vBond |
                +----------+-----------+
                           |
                           | Internet / reachable transport
                           |
      +--------------------+--------------------+
      |                                         |
+-----+------+                           +------+-----+
| Branch A   |                           | Branch B   |
| WAN Edge   |                           | WAN Edge   |
| LAN users  |                           | LAN users  |
+-----+------+                           +------+-----+
      |                                         |
      +----------- Encrypted overlay -----------+
```

**What to label on an exam-style diagram**

When you see a diagram, identify these items first:

- **Controllers**
  - vManage
  - vSmart
  - vBond

- **Edge devices**
  - Branch WAN Edge
  - Hub WAN Edge
  - Data center WAN Edge

- **Transport**
  - Internet
  - MPLS
  - LTE/5G
  - Any generic underlay cloud

- **Traffic types**
  - Management traffic
  - Control traffic
  - Data traffic

- **Security and reachability**
  - Secure onboarding
  - Overlay tunnels
  - NAT traversal support

**Practical onboarding sequence**

A simplified onboarding flow helps you understand the diagram:

1. The WAN Edge boots and obtains underlay connectivity.
2. It reaches **vBond** to discover the fabric and learn how to contact the rest of the controllers.
3. It establishes secure control connections to **vSmart** and management connectivity to **vManage**.
4. It receives policy and routing information.
5. It builds overlay tunnels to other WAN Edge devices.
6. It begins forwarding user traffic across the fabric.

This sequence is conceptual and may vary by deployment, but it is a good exam model.

**Configuration concepts you should recognize**

You do not need to memorize every SD-WAN CLI detail for a diagram exercise, but you should recognize common configuration ideas.

- **System settings**
  - Device identity
  - Site ID
  - Organization name
  - Controller addresses
  - UUID and certificate-related identity

- **Transport interfaces**
  - Interfaces connected to the underlay
  - Color-based transport classification
  - Service reachability settings

- **VPNs**
  - `vpn 0` is the transport VPN used for underlay reachability and controller access.
  - Service VPNs carry user LAN traffic.

A conceptual IOS XE SD-WAN transport interface example:

```bash
system
 host-name BRANCH-1
 system-ip 10.255.255.11
 site-id 10
 organization-name EXAMPLE-CORP
 vbond 203.0.113.10
!
vpn 0
 interface GigabitEthernet0/0/0
  ip address 198.51.100.2/30
  tunnel-interface
   color biz-internet
   allow-service all
  no shutdown
 !
!
vpn 10
 interface GigabitEthernet0/0/1
  ip address 10.10.10.1/24
  no shutdown
!
```

This example is conceptual and may differ by platform or release, but it shows the important ideas:

- `vpn 0` is the transport side.
- The tunnel interface is tied to the transport interface.
- `color` identifies the transport type.
- `allow-service` controls what services are permitted over the transport.

**Verification and troubleshooting mindset**

Verification commands can vary by platform and software release, so treat them as examples rather than universal guarantees. In practice, you often check:

- Control connections
- Routing and policy state
- Tunnel status
- BFD or path health
- Interface reachability
- Certificate and onboarding status

Common verification ideas include:

```bash
show sdwan control connections
show sdwan omp peers
show sdwan bfd sessions
show sdwan omp routes
show sdwan interface
show sdwan system status
```

On some platforms or releases, you may use classic IOS XE-style commands or different SD-WAN-specific variants. Always match the command to the device and software version.

**How to interpret common diagram cues**

- If you see a cloud between sites, it usually represents the **underlay**.
- If you see encrypted lines between WAN Edge routers, that usually represents the **overlay**.
- If a controller is shown as the first reachable point for a new edge, that is likely **vBond**.
- If a controller is shown as the central policy and routing brain, that is **vSmart**.
- If a controller is shown as the GUI or dashboard, that is **vManage**.

**Common exam traps**

- Do not confuse **vBond** with the main persistent control-plane peer.
- Do not label **BFD** as the tunnel itself.
- Do not assume every command is available on every IOS XE SD-WAN release.
- Do not place user traffic directly through the controllers.
- Do not treat the underlay and overlay as the same thing.

**Quick comparison**

| Component | Primary role | Typical traffic |
|---|---|---|
| vManage | Management and orchestration | Configuration, monitoring, API |
| vSmart | Control-plane policy and routing | Control information |
| vBond | Discovery and onboarding assistance | Initial reachability, NAT traversal |
| WAN Edge | Forwarding and overlay participation | User data, control sessions |

**Study approach for the lab**

When you practice the diagram exercise, work in this order:

1. Identify the controllers.
2. Mark the branch and hub WAN Edge devices.
3. Separate underlay links from overlay tunnels.
4. Label the management, control, and data planes.
5. Check whether the diagram implies onboarding, steady-state operation, or troubleshooting.

A strong answer explains not only what each box is, but also why the traffic belongs there.

**Key takeaways**

- Cisco SD-WAN uses a layered architecture with clear separation of management, control, and data functions.
- vManage manages, vSmart controls, and vBond assists with discovery and onboarding.
- WAN Edge devices build secure overlay tunnels across the underlay.
- BFD monitors path health; it does not create the tunnel.
- Diagram questions reward precise labeling and correct traffic-flow reasoning.
