## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Troubleshooting mindset**

Troubleshooting is a process of narrowing the fault domain, not guessing. Start with the simplest layer that can explain the symptom, verify assumptions with commands, and change one thing at a time. In CCNA scenarios, many failures come from small mismatches: VLANs, masks, gateways, trunk encapsulation, ACL direction, NAT inside/outside placement, or DHCP relay placement.

A practical workflow is:

1. Identify the symptom and scope.
2. Check physical and data-link status.
3. Verify addressing, VLANs, and default gateways.
4. Confirm routing, NAT, ACLs, and services.
5. Test with targeted commands and compare expected vs actual output.

**Layer 1 and Layer 2 faults**

Physical and switching problems are often the fastest to isolate because interface state and counters reveal a lot. A link can be up but still fail due to speed/duplex mismatch, bad optics, or intermittent flaps.

Common Layer 1 issues include:

- **Bad cabling** such as damaged patch cords, wrong cable type, or loose connectors.
- **Speed/duplex mismatch** causing late collisions, CRC errors, or poor throughput.
- **Transceiver/SFP issues** such as unsupported optics, dirty fiber, or mismatched modules.
- **Physical flaps** where the interface repeatedly transitions up and down.
- **Power or PoE problems** affecting phones, APs, or cameras.

Useful verification commands:

```bash
show interfaces status
show interfaces gigabitEthernet0/1
show controllers ethernet-controller gi0/1
show logging | include up|down|LINEPROTO|LINK
```

Interpretation tips:

- `show interfaces status` helps you see whether a switchport is connected, disabled, or err-disabled.
- `show interfaces` reveals errors such as CRC, input errors, runts, giants, and collisions.
- `show logging` helps confirm whether the interface is flapping or being shut down by a protocol or security feature.

> `show interfaces brief` is not a valid IOS command. Use `show ip interface brief` for Layer 3 interfaces or `show interfaces status` for switchport state.

**VLAN, trunk, and access-port mistakes**

Many CCNA switching failures come from VLAN membership errors rather than broken hardware. If a host cannot reach its gateway, verify the access port VLAN, the trunk path, and the SVI status.

Common misconfigurations:

- **Wrong access VLAN** on the host-facing switchport.
- **Missing VLAN** in the VLAN database or on an intermediate switch.
- **Trunk not allowing the VLAN** across the path.
- **Native VLAN mismatch** causing untagged traffic to land in the wrong VLAN.
- **Access port accidentally left in trunk mode** or vice versa.

Example verification:

```bash
show vlan brief
show interfaces trunk
show running-config interface gi0/1
show mac address-table dynamic vlan 10
```

A typical access-port configuration looks like this:

```cisco
interface gigabitEthernet0/1
 switchport mode access
 switchport access vlan 10
 spanning-tree portfast
```

A trunk should be consistent on both ends:

```cisco
interface gigabitEthernet0/24
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30
 switchport trunk native vlan 99
```

If a host in VLAN 10 cannot reach the default gateway, check:

- The host IP address and mask.
- The switchport access VLAN.
- The SVI for VLAN 10.
- The trunk path between access switch and distribution switch.
- The router or multilayer switch route to the destination.

**Inter-VLAN routing and router-on-a-stick**

Inter-VLAN routing fails when the router subinterfaces or switch trunk are inconsistent. Router-on-a-stick is especially sensitive to encapsulation, native VLAN, and IP addressing errors.

Common problems:

- **Missing subinterface** for a VLAN.
- **Wrong `encapsulation dot1Q` VLAN ID**.
- **Native VLAN mismatch** between router and switch.
- **IP address or subnet mask mismatch** on the subinterface.
- **Trunk not enabled** on the switch port facing the router.

Example router-on-a-stick configuration:

```cisco
interface gigabitEthernet0/0
 no shutdown

interface gigabitEthernet0/0.10
 encapsulation dot1Q 10
 ip address 192.168.10.1 255.255.255.0

interface gigabitEthernet0/0.20
 encapsulation dot1Q 20
 ip address 192.168.20.1 255.255.255.0
```

Switch port toward the router:

```cisco
interface gigabitEthernet0/1
 switchport mode trunk
 switchport trunk allowed vlan 10,20
```

Troubleshooting approach:

1. Confirm the physical link is up.
2. Verify the switch port is trunking.
3. Verify each subinterface has the correct VLAN tag.
4. Confirm the router has the correct IP address for each VLAN.
5. Test from a host to its gateway first, then across VLANs.

> If the native VLAN is mismatched, untagged frames may be misclassified, which can break management traffic or create hard-to-diagnose one-way connectivity.

**SVIs, multilayer switches, and management reachability**

On Cisco multilayer switches, `ip default-gateway` is used only when `ip routing` is disabled. Once `ip routing` is enabled, the switch uses the routing table for management traffic and ignores `ip default-gateway`.

Common mistakes:

- **SVI down/down** because no active port exists in the VLAN.
- **Wrong management VLAN** or missing VLAN on the trunk.
- **No default route** on a Layer 3 switch when routing is enabled.
- **Using `ip default-gateway` on a routed switch** and expecting it to work.
- **Incorrect management IP/mask** causing the switch to believe the gateway is local.

Useful commands:

```bash
show ip interface brief
show interfaces vlan 99
show ip route
show running-config | include ip routing|ip default-gateway
```

Example management SVI:

```cisco
interface vlan 99
 ip address 10.99.99.2 255.255.255.0
 no shutdown
```

If routing is disabled, configure a default gateway:

```cisco
no ip routing
ip default-gateway 10.99.99.1
```

If routing is enabled, configure a default route instead:

```cisco
ip routing
ip route 0.0.0.0 0.0.0.0 10.99.99.1
```

**Host-side troubleshooting**

You should always verify the endpoint before blaming the network. Host commands reveal whether the issue is local addressing, DNS, gateway, or ARP resolution.

Windows commands:

```powershell
ipconfig /all
route print
arp -a
ping 192.168.10.1
tracert 8.8.8.8
```

Linux commands:

```bash
ip addr show
ip route show
ip neigh show
ping -c 4 192.168.10.1
traceroute 8.8.8.8
```

What to look for:

- **IP address and mask** must match the intended subnet.
- **Default gateway** must be in the same subnet as the host.
- **DNS server** must be reachable if name resolution fails.
- **ARP table** should contain the gateway MAC after a successful local ping.
- **Routing table** should include a default route for off-subnet traffic.

A useful host test sequence is:

1. Ping `127.0.0.1` or `::1` to confirm the TCP/IP stack.
2. Ping the host’s own IP address.
3. Ping the default gateway.
4. Ping a remote IP address.
5. Test DNS with a hostname.

**Static routing and route selection**

Routing failures often come from missing routes, wrong masks, or an incorrect next hop. Static routes are simple but easy to misconfigure.

Common issues:

- **Wrong destination network or mask** in the static route.
- **Incorrect next-hop IP** or exit interface.
- **Missing return route** on the remote router.
- **Floating static route** with an administrative distance that does not behave as intended.
- **Overlapping routes** causing unexpected path selection.

Example:

```cisco
ip route 192.168.30.0 255.255.255.0 10.1.1.2
ip route 0.0.0.0 0.0.0.0 10.1.1.1
```

Verification:

```bash
show ip route
show ip route static
show running-config | include ^ip route
traceroute 192.168.30.10
```

If traffic reaches the next hop but not the destination, check the return path. Many “one-way” problems are actually missing routes on the far side.

**Dynamic routing issues: OSPF, EIGRP, and RIP**

Dynamic routing problems usually fall into adjacency, network advertisement, or metric issues. The router may be up, but routes never appear in the table.

OSPF common problems:

- **Area mismatch** between neighbors.
- **Hello/dead timer mismatch**.
- **Authentication mismatch**.
- **Passive interface** preventing neighbor formation.
- **Wrong network statement or interface not included**.
- **MTU mismatch** causing adjacency issues in some cases.

Useful commands:

```bash
show ip ospf neighbor
show ip ospf interface brief
show ip route ospf
show running-config | section router ospf
```

EIGRP in modern Cisco IOS/IOS XE is often configured in **named mode**, though classic autonomous-system mode may still appear in labs. Common issues include:

- **AS or named-process mismatch**.
- **K-value mismatch**.
- **Passive interface**.
- **Authentication mismatch**.
- **Unequal summarization or filtering** affecting route visibility.

Example named-mode EIGRP:

```cisco
router eigrp CCNA
 address-family ipv4 unicast autonomous-system 100
  network 10.0.0.0 0.255.255.255
  no passive-interface gigabitEthernet0/0
 exit-address-family
```

RIP issues are usually simpler but still exam-relevant:

- **Version mismatch** between RIP v1 and v2.
- **Auto-summary** causing classful advertisement problems.
- **Missing `no auto-summary`** in discontiguous networks.
- **Passive interface** blocking updates.

Verification:

```bash
show ip protocols
show ip route rip
show running-config | section router rip
```

**ACL troubleshooting**

Access control lists (ACLs) often block traffic because of direction, sequence, or implicit deny issues. Always verify where the ACL is applied and what traffic it matches.

Common mistakes:

- **Wrong interface or direction** (`in` vs `out`).
- **ACL applied to the wrong interface**.
- **Missing permit statement** before the implicit deny.
- **Incorrect wildcard mask**.
- **Order of ACEs** causing a broader deny to match first.
- **Expecting an ACL to filter traffic it never sees**, such as return traffic on a different interface.

Example interface application:

```cisco
interface gigabitEthernet0/1
 ip access-group WEB-FILTER in
```

Example ACL:

```cisco
ip access-list extended WEB-FILTER
 permit tcp 192.168.10.0 0.0.0.255 any eq 80
 permit tcp 192.168.10.0 0.0.0.255 any eq 443
 deny ip any any log
```

Verification:

```bash
show access-lists
show ip interface gigabitEthernet0/1
show running-config interface gigabitEthernet0/1
```

Troubleshooting method:

1. Identify the source and destination IPs and ports.
2. Check the ACL direction and interface.
3. Confirm the matching ACE appears before any deny.
4. Look at hit counts to see whether the ACL is matching.
5. Temporarily test with a more specific permit if allowed in the lab.

**NAT troubleshooting**

Network Address Translation (NAT) problems usually involve inside/outside designation, ACL matching, or route reachability. If inside hosts cannot reach the Internet, verify the translation path end to end.

Common issues:

- **Wrong inside/outside interface assignment**.
- **ACL does not match the inside subnet**.
- **No default route toward the ISP**.
- **Overlapping or missing NAT pool**.
- **PAT overload not configured correctly**.
- **Static NAT mapping points to the wrong address**.

Example PAT configuration:

```cisco
interface gigabitEthernet0/0
 ip nat inside

interface gigabitEthernet0/1
 ip nat outside

access-list 1 permit 192.168.10.0 0.0.0.255
ip nat inside source list 1 interface gigabitEthernet0/1 overload
```

Verification:

```bash
show ip nat translations
show ip nat statistics
show running-config | include ip nat|access-list 1
show ip route
```

A simple NAT decision tree:

1. Confirm the inside host can ping its default gateway.
2. Confirm the router has a route to the outside network.
3. Confirm the inside interface is marked `ip nat inside`.
4. Confirm the outside interface is marked `ip nat outside`.
5. Confirm the ACL matches the inside source addresses.
6. Check translation counters and active entries.

**DHCP troubleshooting**

DHCP failures are often caused by relay placement, pool mismatch, or exclusions. The client may show an APIPA address on Windows or no address at all.

Common issues:

- **DHCP server unreachable** from the client subnet.
- **Missing `ip helper-address`** on the correct SVI or subinterface.
- **Helper configured on the wrong interface**.
- **Wrong DHCP pool network or default-router**.
- **Excluded addresses overlapping the pool**.
- **Server scope exhausted**.
- **Firewall or ACL blocking UDP ports 67 and 68**.

Example relay on an SVI:

```cisco
interface vlan 10
 ip address 192.168.10.1 255.255.255.0
 ip helper-address 10.1.1.10
```

Example DHCP pool:

```cisco
ip dhcp excluded-address 192.168.10.1 192.168.10.20

ip dhcp pool VLAN10
 network 192.168.10.0 255.255.255.0
 default-router 192.168.10.1
 dns-server 8.8.8.8
```

Verification:

```bash
show ip dhcp binding
show ip dhcp pool
show running-config | section ip dhcp
show ip interface vlan 10
```

Packet-flow view:

- The client broadcasts a DHCP Discover.
- The relay agent forwards it as unicast to the server.
- The server replies with Offer and Ack.
- The relay forwards the response back to the client subnet.

If the relay is missing or on the wrong interface, the server never sees the request from that VLAN.

**IPv6 troubleshooting**

IPv6 issues often involve addressing, neighbor discovery, or router advertisements. Link-local addresses are especially important because they are used for next-hop resolution and routing protocols.

Common problems:

- **Wrong prefix length** on the interface.
- **Duplicate Address Detection (DAD)** failure when another device already uses the address.
- **Router Advertisement (RA)** flags not matching the intended host behavior.
- **ICMPv6 blocked**, which breaks neighbor discovery and SLAAC.
- **Incorrect link-local next hop** or missing interface scope.

Useful commands:

```bash
show ipv6 interface brief
show ipv6 neighbors
show ipv6 route
ping ipv6 2001:db8:1::1
ping ipv6 fe80::1 source gigabitEthernet0/0
```

Notes:

- Link-local addresses require an interface context or source specification.
- Neighbor Discovery Protocol (NDP) depends on ICMPv6, so blocking ICMPv6 can break basic IPv6 operation.
- If a host receives a router advertisement but no global address, check the RA flags and DHCPv6/SLAAC design.

**Wireless troubleshooting**

Wireless problems can be caused by RF issues, authentication failures, or controller/AP join problems. Distinguish between association and authentication: a client may see the SSID but still fail to join.

Common issues:

- **Wrong SSID or security settings** on the client.
- **PSK mismatch** or 802.1X credential failure.
- **AP not joined to controller** in a controller-based design.
- **Channel interference** or poor signal strength.
- **Band mismatch** such as a client supporting only 2.4 GHz while the SSID is 5 GHz only.
- **Client association succeeds but authentication fails** due to RADIUS or key issues.

Useful checks:

```bash
show wlan summary
show ap summary
show wireless client summary
show logging | include CAPWAP|RADIUS|authentication
```

Practical RF considerations:

- **Channel overlap** on 2.4 GHz can cause contention and retries.
- **Low RSSI** (received signal strength indicator) can cause roaming or disconnects.
- **High interference** from non-Wi-Fi devices can reduce throughput.
- **AP placement** affects coverage more than raw transmit power in many small networks.

**Security and service misconfigurations**

Security features can look like outages when they are actually working as configured. Verify whether a feature is intentionally blocking traffic before changing it.

Common examples:

- **Port security** placing a port into err-disabled state after a MAC violation.
- **DHCP snooping** dropping untrusted DHCP offers.
- **Dynamic ARP Inspection (DAI)** blocking spoofed ARP packets.
- **Storm control** limiting broadcast or multicast traffic.
- **Spanning Tree Protocol (STP)** blocking a port or causing topology changes.

Verification commands:

```bash
show port-security interface gi0/1
show errdisable recovery
show ip dhcp snooping
show ip arp inspection
show spanning-tree vlan 10
```

If a port is err-disabled, identify the cause before re-enabling it:

```cisco
interface gigabitEthernet0/1
 shutdown
 no shutdown
```

**Troubleshooting by packet flow**

A packet-flow approach helps you isolate where traffic stops. Use it when the symptom is “host A cannot reach host B” and you need a repeatable method.

Example: host in VLAN 10 cannot reach the Internet.

1. Verify the host has a valid IP, mask, and gateway.
2. Ping the default gateway.
3. Check the access port VLAN and SVI status.
4. Verify inter-VLAN routing or router-on-a-stick.
5. Confirm the default route toward the ISP.
6. Check NAT inside/outside and translation entries.
7. Verify ACLs are not blocking the traffic.
8. Test DNS only after IP connectivity works.

Example: DHCP client gets no address.

1. Check the client NIC and link status.
2. Verify the access port VLAN.
3. Confirm the relay helper on the correct SVI or subinterface.
4. Check the DHCP server scope, exclusions, and lease availability.
5. Verify no ACL or firewall blocks UDP 67/68.
6. Inspect server logs or packet captures if available.

Example: wireless client sees SSID but cannot connect.

1. Confirm the SSID is broadcast on the correct band.
2. Verify security settings and credentials.
3. Check AP join status and controller health.
4. Review RADIUS or authentication logs.
5. Check signal strength, interference, and client compatibility.

A compact command set helps you move quickly during labs and exams:

```bash
show ip interface brief
show interfaces status

show mac address-table
show ip route
show ip protocols
show access-lists

show ip dhcp binding
show ipv6 neighbors
show logging
```

**Exam-focused reminders**

- Always verify the simplest layer first.
- Match the command to the device role: switchport, routed interface, host, or service.
- Remember that many failures are caused by mismatched assumptions, not broken hardware.
- Use `show` commands to confirm state before making changes.
- Treat ACLs, NAT, DHCP relay, and trunking as common “silent failure” sources because they often drop traffic without obvious user-facing errors.

**Quick comparison of common symptoms**

| Symptom | Likely area | First checks |
|---|---|---|
| Link down | Layer 1 | Cable, optics, interface state, logs |
| Host has APIPA | DHCP | Relay, scope, exclusions, server reachability |
| Same VLAN works, other VLAN fails | Inter-VLAN routing | SVI, trunk, router subinterfaces |
| Internet unreachable | Routing/NAT/ACL | Default route, NAT translations, ACLs |
| SSID visible but no connection | Wireless auth/RF | Security, RADIUS, signal, AP join |
| One-way connectivity | ACL/return route | ACL direction, reverse path, routing table |

**Final troubleshooting habit**

Do not change multiple variables at once. Verify, isolate, and document each step. In CCNA scenarios, the fastest path to the answer is usually a disciplined sequence of `show` commands, host checks, and packet-flow reasoning.
