## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Module review mindset**

This topic is a consolidation pass across the CCNA blueprint. You should use it to connect technologies, not just memorize definitions. The exam often tests how features interact: Layer 2 switching, routing decisions, IPv4 and IPv6 addressing, security controls, wireless behavior, and troubleshooting workflows.

A strong review strategy is to ask three questions for every topic:

- What problem does it solve?
- What does it look like in configuration or verification?
- What failure mode would I expect if it breaks?

**Layer 2 switching and VLAN behavior**

Switches forward frames based on the destination MAC address table, also called the CAM table. If the destination MAC is known, the switch sends the frame only out the correct port. If the destination MAC is unknown, the switch floods the frame within the same VLAN. Broadcast frames are also flooded within the VLAN.

Multicast handling depends on the feature set and control protocols in use. In modern networks, IGMP snooping helps switches forward multicast only to interested ports instead of flooding it everywhere.

Key review points:

- **VLANs** separate broadcast domains at Layer 2.
- **Trunks** carry multiple VLANs between switches or to routers and APs.
- **Access ports** belong to one VLAN only.
- **Native VLAN** traffic on 802.1Q trunks is untagged by default.
- **Voice VLANs** separate voice traffic from data traffic on the same access port.

Useful verification commands:

```bash
show vlan brief
show interfaces trunk
show mac address-table
show interfaces switchport
```

Common failure patterns:

- A host cannot reach another host in the same VLAN because the port is in the wrong VLAN.
- A trunk does not pass traffic because the allowed VLAN list is missing the VLAN.
- Native VLAN mismatch causes unexpected connectivity issues and log messages.
- MAC address table entries age out, so intermittent traffic may appear to “disappear” after inactivity.

**Spanning Tree Protocol review**

Spanning Tree Protocol (STP) prevents Layer 2 loops by blocking redundant paths. Classic 802.1D STP uses the states blocking, listening, learning, and forwarding. Rapid Spanning Tree Protocol (RSTP, 802.1w) converges faster and effectively uses discarding, learning, and forwarding states.

You should know the role of the root bridge:

- The root bridge is the reference point for the spanning tree.
- Switches choose the root based on the lowest Bridge ID, which includes priority and MAC address.
- Lower bridge priority wins; if priorities tie, the lower MAC address wins.

Port roles matter in troubleshooting:

- **Root port**: best path toward the root bridge on a non-root switch.
- **Designated port**: forwarding port for a segment.
- **Alternate port**: backup path in RSTP.
- **Backup port**: less common, used in specific redundant topologies.

Verification commands:

```bash
show spanning-tree
show spanning-tree vlan 10
show spanning-tree root
```

Common failure patterns:

- A switch unexpectedly becomes root because its priority is lower than intended.
- A port stays blocking because another switch has the better path to the root.
- A topology change causes temporary disruption if the network is still using classic STP behavior.
- PortFast is missing on edge ports, causing slow host connectivity during boot.

Example root bridge tuning:

```bash
conf t
spanning-tree vlan 10 priority 4096
spanning-tree vlan 20 priority 4096
end
```

**EtherChannel and link aggregation**

EtherChannel bundles multiple physical links into one logical port-channel. This increases bandwidth and provides redundancy, but the member links must match in key parameters such as speed, duplex, trunking mode, and allowed VLANs.

Common negotiation protocols:

- **LACP** (Link Aggregation Control Protocol) is IEEE 802.3ad/802.1AX based.
- **PAgP** (Port Aggregation Protocol) is Cisco proprietary.
- Static EtherChannel uses `on` and does not negotiate.

Verification commands:

```bash
show etherchannel summary
show interfaces port-channel 1
show interfaces etherchannel
```

Example configuration with LACP:

```bash
conf t
interface range g0/1 - 2
 switchport mode trunk
 channel-group 1 mode active
exit
interface port-channel 1
 switchport mode trunk
end
```

Common failure patterns:

- One member link is configured differently, so the bundle does not form.
- The trunk settings on the physical interfaces and port-channel do not match.
- A link is suspended because of a parameter mismatch.
- The port-channel appears up, but traffic does not pass because VLANs are not allowed.

**IPv4 routing, route selection, and the routing table**

When a router forwards traffic, it uses the routing table to choose the best match. The most important selection rule is **longest-prefix match**: the most specific route wins.

If multiple routes match the same destination prefix length, the router prefers the route with the lowest **administrative distance**. If administrative distance is also equal, the router uses the lowest metric for that routing source.

Important route preference concepts:

- **Connected routes** are highly preferred because they have the lowest administrative distance.
- **Static routes** are preferred over most dynamic routes when AD is lower.
- **Default routes** match when no more specific route exists.
- **Floating static routes** use a higher administrative distance so they act as backups.

Example static route and backup route:

```bash
conf t
ip route 203.0.113.0 255.255.255.0 192.0.2.2
ip route 203.0.113.0 255.255.255.0 198.51.100.2 200
end
```

Verification commands:

```bash
show ip route
show ip route static
show ip route 0.0.0.0
show ip protocols
```

Common failure patterns:

- A route exists, but a more specific prefix overrides it.
- A static route points to the wrong next hop or exit interface.
- A default route is missing, so unknown destinations are unreachable.
- A dynamic route is present, but its administrative distance is worse than expected.

**IPv4 addressing, subnetting, and VLSM**

Variable Length Subnet Masking (VLSM) lets you allocate different subnet sizes based on need. This reduces waste and is common in enterprise designs.

You should be able to identify:

- Network address
- Broadcast address
- Usable host range
- Prefix length and dotted-decimal mask

Practical subnetting reminders:

- A `/24` has 256 total addresses and 254 usable hosts.
- A `/26` has 64 total addresses and 62 usable hosts.
- A `/30` has 4 total addresses and 2 usable hosts, often used for point-to-point links.
- A `/31` can be used on some point-to-point links when supported by the platform and design.

Common failure patterns:

- The wrong subnet mask causes hosts to believe remote devices are local.
- The default gateway is outside the host subnet.
- Overlapping subnets create ambiguous routing and ARP behavior.

**IPv6 addressing and neighbor discovery**

IPv6 uses 128-bit addresses and relies on Neighbor Discovery Protocol (NDP) instead of ARP. NDP uses ICMPv6 messages for address resolution, router discovery, and neighbor reachability.

Address assignment methods include:

- **SLAAC** (Stateless Address Autoconfiguration), where the host forms its address from router advertisements.
- **Stateful DHCPv6**, where the DHCPv6 server assigns the IPv6 address.
- **Stateless DHCPv6**, where SLAAC provides the address and DHCPv6 provides other parameters such as DNS.

This distinction matters on the exam: DHCPv6 does not always provide the address.

Useful verification commands:

```bash
show ipv6 interface brief
show ipv6 route
show ipv6 neighbors
```

Common failure patterns:

- Router advertisements are blocked, so SLAAC fails.
- The host has an IPv6 address but no default gateway because the router advertisement is missing.
- Stateless DHCPv6 is configured, but the client still needs SLAAC to obtain the address.
- Link-local addresses are present, but global unicast connectivity is not working.

Example IPv6 interface configuration:

```bash
conf t
ipv6 unicast-routing
interface g0/0
 ipv6 address 2001:db8:10:1::1/64
 ipv6 nd other-config-flag
 no shutdown
end
```

**Routing protocols and dynamic routing basics**

CCNA-level routing protocol review focuses on what each protocol does and how to verify it, rather than deep algorithm details.

Key ideas:

- Dynamic routing protocols learn routes automatically and adapt to topology changes.
- Interior Gateway Protocols (IGPs) operate within an autonomous system.
- Metrics differ by protocol; for example, RIP uses hop count, while OSPF uses cost.

OSPF review points:

- OSPF is a link-state protocol.
- It forms neighbor adjacencies before exchanging topology information.
- It uses areas, with area 0 as the backbone.
- Router ID selection matters for adjacency and troubleshooting.

Verification commands:

```bash
show ip ospf neighbor
show ip ospf interface brief
show ip route ospf
```

Common failure patterns:

- Mismatched area IDs prevent adjacency.
- Hello and dead timers do not match.
- Passive interfaces stop neighbor formation on that interface.
- Authentication mismatch prevents adjacency.

**ACLs, NAT, and PAT**

Access Control Lists (ACLs) filter traffic based on rules. They are commonly used to permit or deny traffic by source, destination, protocol, or port.

ACL review points:

- Standard ACLs match only source IPv4 address.
- Extended ACLs can match source, destination, protocol, and ports.
- ACLs are processed top-down, first match wins.
- An implicit `deny any` exists at the end of every ACL.

Example extended ACL:

```bash
conf t
ip access-list extended WEB-FILTER
 permit tcp 192.168.10.0 0.0.0.255 host 203.0.113.10 eq 443
 deny ip any any
exit
interface g0/0
 ip access-group WEB-FILTER in
end
```

Verification commands:

```bash
show access-lists
show ip interface g0/0
```

Common failure patterns:

- The ACL is applied in the wrong direction.
- The ACL order blocks traffic before the intended permit statement.
- Wildcard masks are incorrect.
- The ACL is configured but not applied to an interface.

Network Address Translation (NAT) and Port Address Translation (PAT) let private addresses communicate with public networks.

Important distinctions:

- **Static NAT** maps one inside local address to one inside global address.
- **Dynamic NAT** maps from a pool of public addresses.
- **PAT** overloads many inside local addresses onto one or a few public addresses using ports.

Example PAT configuration:

```bash
conf t
access-list 1 permit 192.168.10.0 0.0.0.255
ip nat inside source list 1 interface g0/0 overload
interface g0/0
 ip nat outside
interface g0/1
 ip nat inside
end
```

Verification commands:

```bash
show ip nat translations
show ip nat statistics
show access-lists
```

Common failure patterns:

- Inside and outside interfaces are reversed.
- The ACL does not match the inside subnet.
- The default route to the ISP is missing.
- PAT works for some hosts but not others because the ACL is incomplete.

**Security fundamentals and device hardening**

CCNA security topics emphasize basic device protection and traffic control. You should know the difference between authentication, authorization, and accounting (AAA), and understand common switch protections.

Password and access hardening:

- `enable secret` stores a hashed secret and is preferred for privileged access.
- `service password-encryption` only weakly obfuscates certain plaintext passwords in the configuration; it is not strong encryption.
- Use `login local` with local usernames for basic device access control.
- SSH is preferred over Telnet because it encrypts the session.

Example secure management setup:

```bash
conf t
hostname SW1
ip domain-name example.local
username admin secret Str0ngP@ss!
enable secret An0therStr0ngP@ss!
crypto key generate rsa modulus 2048
line vty 0 4
 login local
 transport input ssh
end
```

Port security:

- Limits which MAC addresses can use a switch port.
- Can restrict the number of learned MAC addresses.
- Can shut down, restrict, or protect on violation depending on the mode.

Example port security:

```bash
conf t
interface f0/10
 switchport mode access
 switchport port-security
 switchport port-security maximum 2
 switchport port-security mac-address sticky
 switchport port-security violation restrict
end
```

Verification commands:

```bash
show port-security interface f0/10
show port-security address
show running-config interface f0/10
```

DHCP snooping and Dynamic ARP Inspection:

- **DHCP snooping** trusts only selected ports for DHCP server messages.
- It helps stop rogue DHCP servers and builds a binding table.
- **Dynamic ARP Inspection** uses the DHCP snooping binding table to validate ARP packets and reduce ARP spoofing.

Example DHCP snooping setup:

```bash
conf t
ip dhcp snooping
ip dhcp snooping vlan 10
interface g0/1
 ip dhcp snooping trust
interface f0/10
 ip dhcp snooping limit rate 15
end
```

Common failure patterns:

- A trusted uplink is not marked trusted, so DHCP offers are dropped.
- DAI is enabled without DHCP snooping bindings, causing legitimate ARP to fail.
- Port security shuts down an access port after a MAC change.
- SSH fails because RSA keys were not generated or the domain name is missing.

**Wireless concepts, security, and controller-based design**

Wireless LANs use access points (APs) to bridge wireless clients to the wired network. In controller-based designs, lightweight APs are centrally managed by a wireless LAN controller, such as a traditional WLC or a modern Catalyst 9800-class controller.

Key wireless concepts:

- **SSID** is the network name broadcast to clients.
- **BSSID** identifies the AP radio interface.
- **RF** planning includes channel selection, transmit power, and coverage overlap.
- **2.4 GHz** has more interference and fewer non-overlapping channels than 5 GHz.

Wireless security and authentication:

- **Open** networks provide no authentication and no encryption.
- **WPA2-Personal** uses a pre-shared key (PSK).
- **WPA2-Enterprise** uses 802.1X with a RADIUS server.
- **WPA3** improves security over WPA2 and is preferred when supported.

Practical configuration concepts on a controller:

- Create a WLAN/SSID.
- Map the WLAN to a VLAN or policy.
- Choose the security method.
- Apply RF and client policies as needed.

Example verification ideas:

```bash
show wlan summary
show ap summary
show wireless client summary
show dot11 associations
```

Common failure patterns:

- The SSID is configured but not enabled.
- The WLAN is mapped to the wrong VLAN.
- Authentication fails because the PSK is wrong or the RADIUS server is unreachable.
- Clients connect but cannot reach the network because the AP uplink VLAN is incorrect.

**Automation and programmability**

Automation on the CCNA focuses on understanding APIs, controllers, and data formats. You should know the purpose of REST APIs, JSON, and YAML.

Core concepts:

- **API** (Application Programming Interface) exposes functions or data for programmatic access.
- **REST** (Representational State Transfer) commonly uses HTTP methods such as `GET`, `POST`, `PUT`, and `DELETE`.
- **JSON** is a common data format for API responses.
- **YAML** is often used for configuration and automation files.

A safer REST example uses certificate validation rather than disabling it:

```bash
curl -u admin:password https://10.0.0.10/api/v1/interfaces
```

If you must test against a lab device with a self-signed certificate, treat certificate bypass as a lab-only exception, not normal practice.

Example JSON payload:

```json
{
  "name": "GigabitEthernet0/1",
  "description": "Uplink to distribution switch",
  "enabled": true
}
```

Common failure patterns:

- The API endpoint path is wrong.
- Authentication fails because credentials or tokens are invalid.
- The device returns JSON, but the client expects XML or another format.
- TLS certificate validation fails because the device uses a self-signed certificate.

**Troubleshooting workflow**

A disciplined troubleshooting process saves time on the exam and in real networks. Start with the symptom, isolate the layer, and verify assumptions with commands.

A practical workflow:

1. Identify the scope of the problem.
2. Check physical and interface status.
3. Verify Layer 2 switching and VLAN membership.
4. Verify IP addressing, default gateway, and routing.
5. Check security features such as ACLs, port security, DHCP snooping, or NAT.
6. Test name resolution and application reachability.
7. Compare expected and actual configuration.

Useful commands across layers:

```bash
show ip interface brief
show interfaces status
show cdp neighbors detail
show lldp neighbors detail
show running-config
show logging
ping
traceroute
```

Common failure patterns by area:

- **Routing**: wrong next hop, missing default route, longest-prefix match surprises.
- **STP**: blocked uplink, wrong root bridge, missing PortFast on edge ports.
- **DHCP**: no pool, excluded range too large, helper address missing on the gateway.
- **DNS**: IP connectivity works, but hostnames fail to resolve.
- **Wireless**: SSID mismatch, security mismatch, VLAN mismatch, RF interference.
- **ACL/NAT**: rule order, direction, or interface placement is wrong.

**High-value exam reminders**

- Always check the most specific route first when evaluating forwarding decisions.
- Remember that `enable secret` is the secure privileged password option; `service password-encryption` is not strong protection.
- Distinguish between SLAAC, stateful DHCPv6, and stateless DHCPv6.
- Use `show etherchannel summary` and `show interfaces port-channel` as primary EtherChannel verification tools.
- Treat multicast forwarding carefully; it is not the same as broadcast flooding.
- For wireless, know the difference between open, PSK-based, and 802.1X enterprise authentication.
- For security, understand how port security, DHCP snooping, and DAI work together to protect access ports.

**Quick self-check questions**

- Can you explain why a host uses a default gateway?
- Can you predict which route a router chooses when multiple routes match?
- Can you identify the root bridge and the root port in a topology?
- Can you distinguish between ACL placement and ACL order?
- Can you explain how PAT allows many hosts to share one public IP?
- Can you tell the difference between stateful and stateless DHCPv6?
- Can you verify an EtherChannel, VLAN, or trunk with the correct command?
- Can you describe the most likely cause when a wireless client associates but cannot pass traffic?
