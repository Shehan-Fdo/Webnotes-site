## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Why show and debug matter**

`show` commands give you a low-risk view of device state. `debug` commands expose live process activity, which makes them powerful but potentially disruptive. In CCNA troubleshooting, you usually start with `show`, confirm the symptom, and only then use `debug` with strict controls.

A good troubleshooting workflow answers three questions:

- What is the device doing now?
- What changed from the expected state?
- Can you prove the cause without overwhelming the device?

**Show commands: the safe first step**

`show` commands are read-only and are the primary tools for verifying interface status, VLAN membership, routing, Layer 2 forwarding, and access control behavior. They are ideal for baseline checks because they do not generate extra processing beyond normal output generation.

Common high-value CCNA `show` commands include:

- `show ip interface brief` for a fast summary of Layer 3 interface status and IP addressing
- `show interfaces` for physical and error counters, duplex, speed, and encapsulation details
- `show interfaces switchport` for access/trunk mode, native VLAN, and administrative versus operational state
- `show vlan brief` for VLAN membership and active VLANs
- `show spanning-tree` for root bridge status, port roles, and blocking behavior
- `show ip route` for the routing table and next-hop decisions
- `show access-lists` for ACL entries and hit counts
- `show arp` for IPv4 address-to-MAC resolution
- `show mac address-table` for Layer 2 forwarding decisions
- `show cdp neighbors detail` or `show lldp neighbors detail` for adjacent device identification

A practical habit is to compare the output against what you expect from the design. For example, if a user cannot reach a server, you check whether the client VLAN exists, whether the switchport is in the correct VLAN, whether the SVI has an IP address, and whether the route exists.

**Core show commands and what they tell you**

| Command | What you verify | Typical troubleshooting use |
|---|---|---|
| `show ip interface brief` | Interface up/down state and IP assignment | Quickly find administratively down or unaddressed interfaces |
| `show interfaces switchport` | Access/trunk mode, VLAN assignment, trunk negotiation | Confirm switchport behavior and VLAN placement |
| `show vlan brief` | VLAN existence and access-port membership | Check whether a port belongs to the intended VLAN |
| `show spanning-tree` | Root bridge, port roles, blocking state | Diagnose loops, blocked links, and unexpected topology changes |
| `show access-lists` | ACL contents and match counters | Confirm whether traffic is being filtered |
| `show ip interface` | Layer 3 interface settings and ACL application | Verify helper addresses, ACLs, and IP-related interface features |

**Debug commands: powerful but risky**

`debug` commands display real-time events as they happen. They are useful when `show` output is not enough to explain a transient problem, such as a routing adjacency failure, DHCP exchange issue, or ACL match behavior. However, debug output can be extremely verbose and CPU-intensive.

> Use `debug` carefully on production devices. Excessive output can degrade performance, flood your terminal, and in extreme cases make remote access unstable.

Important safety principles:

- Use `debug` only after you have a focused hypothesis.
- Prefer a console session when possible, especially on busy devices.
- Avoid leaving debug enabled longer than necessary.
- Always disable debug with `undebug all` or `no debug all` when finished.
- Limit output with filters, access lists, or conditional debugging when the platform supports it.
- Direct output appropriately so you do not flood your active terminal session.

**Safe debug workflow**

1. Verify the symptom with `show` commands first.
2. Narrow the scope to one interface, one host, one protocol, or one ACL.
3. If supported, apply a filter or conditional debug to reduce noise.
4. Start the debug and watch only the relevant events.
5. Stop the debug immediately after collecting enough evidence.
6. Save or note the output before clearing the session.

**Terminal paging and output control**

During troubleshooting, you often need to see full command output without pauses. On Cisco IOS, `terminal length 0` disables paging for the current session so output scrolls continuously. This is useful for long `show` outputs and debug captures, but it can also make the session harder to read if you are not prepared.

Use it carefully:

- `terminal length 0` removes pagination for the current terminal session.
- It does not permanently change the device configuration.
- It is helpful when you need to capture long outputs quickly.
- Re-enable paging later with `terminal length 24` or another preferred value.

Example:

```bash
Router# terminal length 0
Router# show running-config
Router# show ip route
Router# terminal length 24
```

If you are troubleshooting over a remote link, disabling paging can make it easier to capture output, but it also increases the chance of losing your place in a long stream. Use it intentionally, not by habit.

**Debug output control and filtering**

A major safety practice is reducing the amount of debug data before you enable it. Depending on the platform and feature, you may be able to limit output with access lists, interface-specific conditions, or protocol-specific filters.

Examples of common control methods:

- Restrict debug to a single interface when the command supports interface scoping.
- Use an ACL to match only one host or subnet.
- Use conditional debugging where supported to trace only selected traffic or events.
- Redirect output to a logging buffer or syslog destination instead of relying only on the live terminal.
- Increase logging granularity only as needed, then reduce it again.

Example of a filtered approach for ACL-related troubleshooting:

```bash
Router# show access-lists 101
Extended IP access list 101
    10 permit tcp any host 192.0.2.10 eq 80 (24 matches)
    20 deny ip any any (153 matches)
```

If the ACL counters increase on the deny line while the user reports failure, you have evidence that the traffic is being filtered. That is often safer and faster than enabling a broad debug.

**Interpreting debug output line by line**

Debug output is most useful when you can map each line to a protocol event. Read it as a sequence, not as isolated messages.

Typical interpretation pattern:

- Timestamp or sequence information tells you when the event occurred.
- Interface or process name tells you which subsystem generated the message.
- Source and destination addresses identify the traffic flow.
- State changes show where the process succeeded or failed.
- Repeated messages often indicate a loop, retry, or negotiation problem.

Examples of what to look for:

- `line protocol up/down` messages indicate a Layer 1 or Layer 2 transition.
- `DUPLEX_MISMATCH`-style symptoms often appear as errors, collisions, or late collisions in interface counters rather than a single explicit debug line.
- DHCP debug output showing `DISCOVER`, `OFFER`, `REQUEST`, and `ACK` helps you identify where the exchange stops.
- OSPF or EIGRP neighbor messages can show adjacency formation failures due to mismatched timers, network type, or authentication.
- ACL debug output can show whether a packet matched a permit or deny entry.

**Mini case study: VLAN and switchport mismatch**

A user on a switch access port cannot reach the local gateway. You suspect the port is in the wrong VLAN.

Start with `show` commands:

```bash
Switch# show ip interface brief
Interface              IP-Address      OK? Method Status                Protocol
Vlan10                 192.168.10.1     YES manual up                    up
Vlan20                 192.168.20.1     YES manual up                    up
Gi1/0/10               unassigned       YES unset  up                    up
```

Then check the switchport:

```bash
Switch# show interfaces switchport
Name: Gi1/0/10
Switchport: Enabled
Administrative Mode: static access
Operational Mode: static access
Access Mode VLAN: 20 (Users)
Trunking Native Mode VLAN: 1 (default)
```

And confirm VLAN membership:

```bash
Switch# show vlan brief

VLAN Name                             Status    Ports
1    default                          active    Gi1/0/1, Gi1/0/2
10   Staff                             active    Vlan10
20   Users                             active    Vlan20, Gi1/0/10
```

Interpretation:

- The port is up, so the physical link is working.
- The port is in VLAN 20, so if the user expects VLAN 10, the port is misassigned.
- The SVI for VLAN 20 is up, so the VLAN exists and is active.
- The next step is to correct the access VLAN or verify the user’s intended subnet.

**Mini case study: ACL blocking traffic**

A host can ping the default gateway but cannot reach a web server. You suspect an ACL.

Check the interface and ACL application:

```bash
Router# show ip interface GigabitEthernet0/0
GigabitEthernet0/0 is up, line protocol is up
  Internet address is 192.0.2.1/24
  Inbound  access list is 101
  Outbound access list is not set
```

Check the ACL counters:

```bash
Router# show access-lists 101
Extended IP access list 101
    10 permit tcp any host 198.51.100.10 eq 80 (0 matches)
    20 deny ip any any (42 matches)
```

Interpretation:

- The ACL is applied inbound on the interface.
- The permit line has zero matches, so the web traffic is not matching the expected rule.
- The deny line is increasing, so the traffic is being dropped.
- You should verify source/destination addresses, port numbers, and rule order.

**Mini case study: spanning-tree blocking a port**

A redundant switch link is not forwarding traffic as expected. Use spanning tree output to determine whether the port is intentionally blocked.

```bash
Switch# show spanning-tree
VLAN0001
  Spanning tree enabled protocol ieee
  Root ID    Priority    32769
             Address     0011.2233.4455
  Bridge ID  Priority    32769
             Address     00aa.bbcc.ddee

Interface           Role Sts Cost      Prio.Nbr Type
Gi1/0/1             Root FWD 4         128.1    P2p
Gi1/0/2             Altn BLK 4         128.2    P2p
```

Interpretation:

- `Root FWD` means the port is forwarding toward the root bridge.
- `Altn BLK` means the alternate port is blocked to prevent a loop.
- This is normal if the topology has redundancy.
- If the blocked port is unexpected, check bridge priority, link costs, and cabling.

**Common CCNA troubleshooting commands by layer**

- Layer 1:
  - `show interfaces`
  - `show ip interface brief`
  - `show controllers` on some platforms
- Layer 2:
  - `show interfaces switchport`
  - `show vlan brief`
  - `show mac address-table`
  - `show spanning-tree`
  - `show cdp neighbors detail`
- Layer 3:
  - `show ip interface`
  - `show ip route`
  - `show arp`
  - `show access-lists`

These commands help you move from physical status to switching behavior to routing and filtering. That sequence mirrors how many CCNA troubleshooting questions are structured.

**Debug examples you should recognize**

DHCP troubleshooting:

```bash
Router# debug ip dhcp server events
Router# debug ip dhcp server packet
```

What to look for:

- `DISCOVER` received from a client
- `OFFER` sent by the server
- `REQUEST` from the client
- `ACK` confirming lease assignment

If you see `DISCOVER` but no `OFFER`, the server may not have a valid pool, may be excluded from the subnet, or may not be reachable.

Routing adjacency troubleshooting:

```bash
Router# debug ip ospf adj
Router# debug ip ospf hello
```

What to look for:

- Neighbor state transitions such as `INIT`, `2-WAY`, `EXSTART`, `EXCHANGE`, and `FULL`
- Repeated hello messages without adjacency formation
- Authentication or timer mismatch symptoms

ACL troubleshooting:

```bash
Router# debug ip packet detail 101
```

Use this only with extreme caution because packet debugging can be very noisy. If supported, pair it with an ACL so only matching traffic is displayed.

**Best practices for safe debugging**

- Use `show` first, `debug` second.
- Narrow the scope before enabling any debug.
- Prefer console access for heavy troubleshooting sessions.
- Use `terminal length 0` only when you need uninterrupted output, then restore paging.
- Watch CPU and responsiveness while debugging.
- Stop debugging as soon as you have enough evidence.
- Save the relevant output before clearing the session.
- Verify whether the issue is actually caused by configuration, not by the debug itself.

**Useful verification sequence**

1. Check interface state with `show ip interface brief`.
2. Confirm Layer 2 placement with `show interfaces switchport` and `show vlan brief`.
3. Check forwarding and filtering with `show mac address-table`, `show ip route`, and `show access-lists`.
4. Inspect topology behavior with `show spanning-tree` if loops or blocked links are possible.
5. Use a targeted debug only if the problem remains unclear.

**Exam focus reminders**

- `show` commands are safe and should be your default troubleshooting tools.
- `debug` commands can affect performance and should be used carefully.
- `terminal length 0` helps you capture long outputs, but it changes only the current session.
- ACL counters and interface status often provide enough evidence without debug.
- Spanning-tree output is essential when a redundant Layer 2 path behaves unexpectedly.
- Remote debugging is riskier than console debugging because overload can interrupt your session.
