## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Lab goal and troubleshooting mindset**

Misconfiguration remediation means you do not guess at the fix; you isolate the fault, verify the intended design, and correct only the broken control or data-plane setting. In CCNA labs, the issue is often a small mismatch: wrong VLAN, missing default gateway, bad static route, ACL blocking traffic, NAT not translating, or an interface that is administratively up but logically broken.

A reliable workflow saves time under exam pressure:

1. Identify the symptom.
2. Determine the scope.
3. Check Layer 1 through Layer 3 basics first.
4. Verify the relevant service or policy.
5. Fix one issue at a time.
6. Re-test with a known-good source and destination.

**Repeatable isolation checklist**

Use a layered approach so you do not skip obvious faults.

- **Layer 1**: cable, power, interface status, speed/duplex, err-disabled state.
- **Layer 2**: VLAN assignment, trunking, allowed VLANs, native VLAN, MAC learning.
- **Layer 3**: IP address, subnet mask, default gateway, routing table, next hop reachability.
- **Policy/services**: ACLs, NAT/PAT, DHCP, DNS, wireless join/authentication, management access.

A practical first pass on Cisco devices often starts with:

```bash
show ip interface brief
show interfaces status
show vlan brief
show interfaces trunk
show ip route
show running-config
```

If IPv6 is involved, add:

```bash
show ipv6 interface brief
show ipv6 route
show ipv6 neighbors
```

> If a command is not supported on a specific platform or image, use the closest equivalent. CCNA labs often vary slightly by device model and IOS version.

**Common remediation pattern: verify, compare, correct**

The fastest way to fix a lab is to compare the broken device against the intended design or a known-good peer.

- Check the running configuration for the affected interface or feature.
- Compare it to the addressing plan, VLAN plan, or routing plan.
- Correct the smallest possible change.
- Save the configuration only after validation.

Example workflow for an access port that should be in VLAN 20:

```bash
show running-config interface gi0/10
show vlan brief
show interfaces gi0/10 switchport
```

If the port is in the wrong VLAN, correct it:

```bash
conf t
interface gi0/10
 switchport mode access
 switchport access vlan 20
 no shutdown
end
```

Then verify host reachability and MAC learning:

```bash
show mac address-table interface gi0/10
ping 192.168.20.1
```

**Layer 1 and Layer 2 faults**

Physical and switching issues are common because they create symptoms that look like routing or server problems.

- An interface may be `administratively down` because it is shut.
- A port may be `err-disabled` after a security violation or loop event.
- A host may be connected to the wrong access VLAN and still reach devices in that VLAN, which can hide the fault.
- A trunk may be up but missing the required VLAN in the allowed list.

Useful verification commands:

```bash
show ip interface brief
show interfaces status
show interfaces gi0/1
show interfaces trunk
show vlan brief
show spanning-tree vlan 20
```

Typical fixes:

```bash
conf t
interface gi0/1
 no shutdown
 speed auto
 duplex auto
```

For a trunk that is missing VLAN 30:

```bash
conf t
interface gi0/1
 switchport mode trunk
 switchport trunk allowed vlan add 30
end
```

For an access port assigned to the wrong VLAN:

```bash
conf t
interface gi0/10
 switchport mode access
 switchport access vlan 20
end
```

**Port security remediation**

Port security violations can place an interface into an err-disabled state depending on the violation mode and recovery settings. Simply bouncing the port may not be enough if the violation condition still exists.

Check the status and configuration:

```bash
show port-security interface gi0/10
show port-security address
show interfaces status err-disabled
```

Example output interpretation:

```bash
Port Security              : Enabled
Port Status                : Secure-shutdown
Violation Mode             : Shutdown
```

This means the port was shut down because a security violation occurred. Remediation usually requires both fixing the cause and recovering the interface.

Common fixes:

```bash
conf t
interface gi0/10
 switchport port-security maximum 2
 switchport port-security mac-address sticky
 switchport port-security violation restrict
 no shutdown
end
```

If the port remains err-disabled, check whether errdisable recovery is configured:

```bash
show errdisable recovery
```

If appropriate in the lab, you can recover the interface after correcting the cause:

```bash
conf t
interface gi0/10
 shutdown
 no shutdown
end
```

**VLAN and trunk troubleshooting examples**

A host in the wrong VLAN may still communicate with devices in that same VLAN, so test against the intended gateway and a known host in the correct subnet.

Example broken access-port scenario:

```bash
interface gi0/10
 switchport mode access
 switchport access vlan 30
```

But the host should be in VLAN 20. Symptoms may include:
- DHCP address from the wrong scope
- Ping works only to devices in VLAN 30
- Default gateway for VLAN 20 is unreachable

Fix:

```bash
conf t
interface gi0/10
 switchport access vlan 20
end
```

Example broken trunk scenario:

```bash
interface gi0/1
 switchport mode trunk
 switchport trunk allowed vlan 10,20
```

If VLAN 30 is required for voice or a server segment, traffic for that VLAN will fail even though the trunk is up. Add the missing VLAN:

```bash
conf t
interface gi0/1
 switchport trunk allowed vlan add 30
end
```

Verify with:

```bash
show interfaces trunk
show vlan brief
```

**IPv4 addressing and default gateway faults**

A host with the wrong IP address, mask, or default gateway may reach local devices but fail beyond the subnet. A router interface with the wrong mask can also break adjacency and route selection.

Check host-facing interface settings:

```bash
show ip interface brief
show running-config interface g0/0
```

Common mistakes:
- Wrong subnet mask, such as `/25` instead of `/24`
- Wrong default gateway on the host
- Router interface in the wrong subnet
- Duplicate IP address
- Interface configured but not enabled

Example router interface fix:

```bash
conf t
interface g0/0
 ip address 192.168.10.1 255.255.255.0
 no shutdown
end
```

Example host-side logic:
- If the host can ping its own gateway but not remote networks, check routing and ACLs.
- If the host cannot ping the gateway, check VLAN, switchport, IP addressing, and the gateway interface.

**Static routing remediation**

Static route issues are common in CCNA labs because a single typo breaks end-to-end reachability.

Verify the routing table:

```bash
show ip route
show running-config | include ^ip route
```

Common faults:
- Wrong destination network
- Wrong mask
- Wrong next-hop IP
- Wrong exit interface
- Missing return route on the remote router

Example broken route:

```bash
ip route 192.168.30.0 255.255.255.0 10.1.1.3
```

If the next hop is wrong, the route may appear in the config but traffic will fail. Correct it:

```bash
conf t
no ip route 192.168.30.0 255.255.255.0 10.1.1.3
ip route 192.168.30.0 255.255.255.0 10.1.1.2
end
```

If you use a default route, verify both direction and reachability:

```bash
ip route 0.0.0.0 0.0.0.0 10.1.1.2
```

Remember that static routing often fails because the return path is missing, not because the forward path is wrong.

**OSPF troubleshooting and remediation**

With OSPF, missing routes can result from adjacency failure, area mismatch, passive interfaces, network type issues, authentication mismatch, summarization, stub/NSSA design, or route filtering. Do not assume the problem is only route advertisement.

Verify neighbors and routes:

```bash
show ip ospf neighbor
show ip route ospf
show ip protocols
show running-config | section router ospf
```

Example interpretation:
- No neighbors: check interface IPs, masks, area IDs, hello/dead timers, authentication, and whether the interface is passive.
- Neighbor present but route missing: check whether the network is actually included, whether the area type suppresses the route, whether summarization hides it, or whether filtering is in place.

Common OSPF fixes:

```bash
conf t
router ospf 1
 network 10.1.1.0 0.0.0.255 area 0
 no passive-interface g0/0
end
```

If the interface should participate but is passive:

```bash
conf t
router ospf 1
 no passive-interface g0/1
end
```

If the area or mask is wrong, correct the network statement or interface configuration. Also verify that the router ID is stable and unique when needed:

```bash
show ip ospf
```

For IPv6 OSPFv3-style labs, use platform-appropriate verification such as:

```bash
show ipv6 interface brief
show ipv6 ospf neighbor
show ipv6 route
```

**IPv6 addressing and routing faults**

IPv6 issues often come from prefix length mistakes, missing link-local behavior, or incomplete router advertisements. A host may have an address but still fail to reach the gateway if the prefix or default router information is wrong.

Check interface and neighbor state:

```bash
show ipv6 interface brief
show ipv6 neighbors
show ipv6 route
```

Common faults:
- Wrong prefix length, such as `/60` instead of `/64` on a LAN
- Missing `ipv6 unicast-routing` on a router
- Interface IPv6 address configured but interface still shut
- Wrong link-local next hop in a static route
- Router advertisements not advertising the correct prefix or flags

Example router setup:

```bash
conf t
ipv6 unicast-routing
interface g0/0
 ipv6 address 2001:db8:10:1::1/64
 no shutdown
end
```

Example static IPv6 route:

```bash
ipv6 route 2001:db8:20:1::/64 2001:db8:10:12::2
```

If the next hop is link-local, many IOS platforms require the exit interface as well:

```bash
ipv6 route 2001:db8:20:1::/64 g0/0 fe80::2
```

For DHCPv6, be careful about the mode. In CCNA labs, stateful DHCPv6 and stateless DHCPv6 are different:

- **Stateful DHCPv6** assigns the IPv6 address from the server.
- **Stateless DHCPv6** uses SLAAC (Stateless Address Autoconfiguration) for the address and DHCPv6 for other information like DNS.

Example stateless DHCPv6 server configuration:

```bash
ipv6 dhcp pool V6-STATLESS
 dns-server 2001:db8:53::53
 domain-name example.local

interface g0/0
 ipv6 nd other-config-flag
 ipv6 dhcp server V6-STATLESS
```

Example stateful DHCPv6 concept on Cisco IOS often uses a managed-config flag and a pool that provides addressing information in the supported platform syntax. Because syntax varies by IOS version and lab image, verify the exact platform behavior before assuming a specific command set.

**DHCPv4 troubleshooting**

DHCP problems often look like Layer 3 failures because the host cannot obtain a usable address.

Verify the server, relay, and pool:

```bash
show ip dhcp pool
show ip dhcp binding
show running-config | section dhcp
show ip interface brief
```

Common faults:
- Wrong excluded range
- Pool network does not match the client subnet
- Default router option points to the wrong gateway
- DHCP relay missing on the client VLAN interface
- Server unreachable due to routing or ACLs

Example server pool:

```bash
ip dhcp excluded-address 192.168.20.1 192.168.20.20

ip dhcp pool USERS
 network 192.168.20.0 255.255.255.0
 default-router 192.168.20.1
 dns-server 8.8.8.8
```

Example relay on a router interface serving a client VLAN:

```bash
interface vlan 20
 ip address 192.168.20.1 255.255.255.0
 ip helper-address 10.10.10.5
```

If the DHCP server is on another network, ensure routing exists in both directions and that ACLs do not block UDP ports 67 and 68.

**ACL misconfiguration remediation**

Access control lists (ACLs) can silently block traffic that otherwise looks healthy. A common lab issue is that the ACL is applied in the wrong direction, on the wrong interface, or with the wrong wildcard or sequence order.

Verify ACLs and their placement:

```bash
show access-lists
show ip interface g0/0
show running-config | include access-group
```

Common faults:
- ACL applied inbound instead of outbound, or vice versa
- Missing permit statement for return traffic
- Deny statement placed before a needed permit
- Wrong source or destination subnet
- Extended ACL placed too far from the source or on the wrong interface

Example broken ACL:

```bash
ip access-list extended BLOCK_WEB
 deny tcp 192.168.20.0 0.0.0.255 any eq 80
 permit ip any any

interface g0/0
 ip access-group BLOCK_WEB in
```

If the intent is to block only one destination, refine the ACL:

```bash
conf t
ip access-list extended BLOCK_WEB
 no deny tcp 192.168.20.0 0.0.0.255 any eq 80
 deny tcp 192.168.20.0 0.0.0.255 host 203.0.113.10 eq 80
 permit ip any any
end
```

Remember that ACLs are processed top-down, first match wins. If you add a new permit below a broad deny, it will not help.

**NAT and PAT troubleshooting**

NAT (Network Address Translation) and PAT (Port Address Translation) issues often appear as internet reachability failures even when internal routing is correct.

Verify inside/outside roles and translations:

```bash
show ip nat translations
show ip nat statistics
show running-config | section ip nat
show ip interface brief
```

Common faults:
- Inside and outside interfaces reversed
- ACL for NAT does not match the inside subnet
- Missing default route toward the ISP
- Overlapping or incorrect NAT pool
- PAT overload not configured when needed

Example PAT configuration:

```bash
access-list 1 permit 192.168.20.0 0.0.0.255

interface g0/0
 ip nat inside

interface g0/1
 ip nat outside

ip nat inside source list 1 interface g0/1 overload
```

If translation does not occur:
- Confirm the inside host can reach the NAT router.
- Confirm the ACL matches the source subnet.
- Confirm the outside interface has a route to the next hop.
- Confirm return traffic can come back from the outside network.

**Wireless remediation in controller-based WLANs**

In controller-based wireless, the access point (AP) must join the controller before client service is reliable. If the AP does not join, the WLAN may not be available at all, depending on platform and configuration. Do not assume clients will always see the SSID.

Check AP and WLAN status with platform-appropriate commands such as:

```bash
show wireless summary
show ap summary
show wlan summary
show ap join stats summary
```

Common faults:
- AP cannot reach the controller management interface
- Wrong controller IP discovery method
- VLAN or trunk issue between AP and switch
- SSID mapped to the wrong VLAN
- Authentication mismatch, such as wrong PSK or RADIUS settings
- RF coverage issue causing weak signal or unstable association

Example wired-side checks for an AP port:

```bash
show interfaces status
show interfaces trunk
show power inline
show cdp neighbors
```

Example access-port configuration for an AP in a simple lab:

```bash
conf t
interface gi0/24
 switchport mode trunk
 switchport trunk allowed vlan 10,20,99
 power inline auto
 no shutdown
end
```

If the AP uses a management VLAN, ensure the native VLAN and allowed VLANs match the design. If the controller is unreachable, verify routing and ACLs between the AP and controller management network.

**Device management and access problems**

Sometimes the network is fine, but you cannot manage the device because of console, SSH, or privilege configuration issues.

Verify management access:

```bash
show running-config | section line
show ip ssh
show users
show running-config | include username|enable secret|ip domain-name
```

Common faults:
- No local user account for SSH
- RSA keys not generated
- VTY lines not permitting SSH
- Wrong domain name or hostname before key generation
- ACL blocking management access

Example SSH setup:

```bash
conf t
hostname SW1
ip domain-name example.local
username admin secret Cisco123!
crypto key generate rsa modulus 2048
ip ssh version 2

line vty 0 4
 login local
 transport input ssh
end
```

If you cannot reach the device remotely, also check whether the management SVI has an IP address and whether the default gateway is set on a Layer 2 switch:

```bash
show ip interface brief
show running-config | include ip default-gateway
```

**A practical lab remediation workflow**

Use this procedure when you are handed a broken topology.

1. Confirm the symptom with a simple test such as `ping`, `traceroute`, or a browser request.
2. Identify the first failing hop.
3. Check interface status and addressing on both ends.
4. Verify VLANs, trunks, and gateway reachability.
5. Inspect routing, ACLs, NAT, DHCP, or wireless settings as needed.
6. Make one change.
7. Re-test immediately.
8. Save the configuration only after the path works.

Example of a structured test sequence:

```bash
ping 192.168.20.1
ping 10.10.10.1
traceroute 203.0.113.10
show ip route
show access-lists
show ip nat translations
```

**Sample broken-to-fixed scenarios**

**Scenario 1: Wrong VLAN on an access port**

Broken config:

```bash
interface gi0/10
 switchport mode access
 switchport access vlan 30
```

Expected:
- Host should be in VLAN 20
- Gateway is `192.168.20.1`

Fix:

```bash
conf t
interface gi0/10
 switchport access vlan 20
end
```

**Scenario 2: Static route typo**

Broken config:

```bash
ip route 192.168.40.0 255.255.255.0 10.1.1.9
```

Expected next hop is `10.1.1.2`.

Fix:

```bash
conf t
no ip route 192.168.40.0 255.255.255.0 10.1.1.9
ip route 192.168.40.0 255.255.255.0 10.1.1.2
end
```

**Scenario 3: ACL blocks web traffic**

Broken config:

```bash
ip access-list extended FILTER
 deny tcp any any eq 80
 permit ip any any
```

If the intent is to block only one server, narrow the match:

```bash
conf t
ip access-list extended FILTER
 no deny tcp any any eq 80
 deny tcp any host 203.0.113.10 eq 80
 permit ip any any
end
```

**Scenario 4: NAT not translating**

Broken config:

```bash
access-list 1 permit 192.168.30.0 0.0.0.255
interface g0/0
 ip nat outside
interface g0/1
 ip nat inside
```

If the interfaces are reversed, fix them:

```bash
conf t
interface g0/0
 ip nat inside
interface g0/1
 ip nat outside
end
```

**Scenario 5: DHCP client gets no address**

Broken config:

```bash
ip dhcp pool USERS
 network 192.168.20.0 255.255.0.0
 default-router 192.168.20.1
```

The mask does not match the intended `/24` subnet. Fix the pool:

```bash
conf t
ip dhcp pool USERS
 network 192.168.20.0 255.255.255.0
end
```

**How to think during the exam**

Do not chase every possible cause at once. Start with the most local and most likely fault, then move outward.

- If a host cannot reach its gateway, focus on VLAN, switchport, IP addressing, and gateway interface status.
- If a gateway is reachable but remote networks are not, focus on routing, ACLs, and NAT.
- If only one service fails, focus on the policy or service layer, such as ACLs, DHCP, DNS, or wireless authentication.
- If multiple users in one segment fail, suspect a shared access-layer or distribution-layer issue.

A good remediation habit is to document the broken setting before changing it. That helps you reverse a bad fix if needed and makes your troubleshooting more deliberate.
