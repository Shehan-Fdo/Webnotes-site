## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Operational visibility: what you monitor and why**

Network operations depend on three complementary data sources: **logs**, **counters**, and **documentation**. Logs tell you *what happened and when*. Counters tell you *how often* an event or condition occurs. Documentation tells you *what should exist* so you can compare the intended design with the actual state.

For CCNA-level troubleshooting, you should be able to decide which source is most useful for a given symptom. A link flap may appear in logs, rising error counters may indicate a physical or duplex issue, and a port map or topology diagram may reveal that the wrong switchport was patched during a move.

- **Logs** are event records with timestamps and severity.
- **Counters** are cumulative interface or protocol statistics.
- **Documentation** provides the baseline for validation and recovery.

**Logs: event history and severity**

Logs are useful when you need a timeline. On Cisco devices, the system log buffer stores recent messages, and devices can also send logs to a remote syslog server for centralized collection. Centralized logging matters because local buffers are limited and can be overwritten during busy periods or after a reboot.

Common log destinations include:
- **Console** output for direct access.
- **Buffered** logs stored in RAM.
- **Syslog** servers for centralized retention and correlation.

A typical Cisco IOS configuration for remote logging looks like this:

```cisco
conf t
logging host 192.0.2.50
logging trap informational
service timestamps log datetime msec localtime
end
write memory
```

Key points:
- `logging host` sends messages to a syslog server.
- `logging trap informational` sets the minimum severity sent remotely.
- `service timestamps log datetime msec localtime` improves troubleshooting by adding precise timestamps.
- Central logging helps correlate events across multiple devices during outages.

You should know the common severity range conceptually:
- **0** Emergency
- **1** Alert
- **2** Critical
- **3** Error
- **4** Warning
- **5** Notice
- **6** Informational
- **7** Debug

For exam purposes, interpret logs by looking for:
- Interface state changes such as up/down transitions.
- Authentication failures.
- STP (Spanning Tree Protocol) topology changes.
- Routing adjacency resets.
- Power, fan, or hardware warnings.

Example output interpretation:

```cisco
Switch# show logging
*Mar  1 10:14:22.123: %LINK-3-UPDOWN: Interface GigabitEthernet1/0/10, changed state to down
*Mar  1 10:14:24.456: %LINEPROTO-5-UPDOWN: Line protocol on Interface GigabitEthernet1/0/10, changed state to down
```

Interpretation:
- The physical link dropped first, then the line protocol followed.
- The issue may be cable, transceiver, switchport, or connected device related.
- The timestamps help you match the event to user reports or change windows.

**Counters: when statistics are enough and when they are not**

Counters are cumulative values that help you detect trends. They are especially useful when you suspect a physical-layer or duplex problem, congestion, or packet loss. Interface counters are often the first place to look when a link is technically up but traffic is failing.

Useful commands include:

```cisco
show interfaces gigabitEthernet1/0/10
show interfaces counters errors
show interfaces status
show ip interface brief
```

What to look for in `show interfaces`:
- **Input errors** may indicate corruption, cabling, or duplex mismatch.
- **CRC** errors often point to physical-layer issues.
- **Runts** and **giants** can indicate framing or MTU-related problems.
- **Drops** may indicate congestion or queueing issues.
- **Collisions** are relevant mainly on half-duplex or legacy Ethernet segments.
- **Late collisions** often suggest duplex mismatch or cabling problems.

Example output interpretation:

```cisco
Switch# show interfaces gigabitEthernet1/0/10
GigabitEthernet1/0/10 is up, line protocol is up
  5 minute input rate 1000 bits/sec, 2 packets/sec
  5 minute output rate 2000 bits/sec, 3 packets/sec
  12345 packets input, 234567 bytes
  12 input errors, 8 CRC, 0 frame, 0 overrun, 0 ignored
  4 output errors, 0 collisions, 2 interface resets
```

Interpretation:
- The interface is operational, but errors exist.
- CRC errors suggest a physical-layer problem more than a routing issue.
- Interface resets may indicate instability, renegotiation, or a device issue.
- Because counters are cumulative, compare them over time rather than relying on a single snapshot.

Counters are useful, but they do not always explain *why* traffic is failing. When counters are insufficient, you may need packet capture or traffic analysis tools:
- **Interface counters** show totals and error types.
- **Packet captures** show actual frames and headers.
- **Traffic analyzers** help identify retransmissions, malformed packets, protocol exchanges, and application behavior.

Use packet capture when:
- Counters show a problem but not the cause.
- You need to inspect protocol handshakes or malformed traffic.
- You suspect ACL, MTU, fragmentation, or application-layer issues.
- You need proof of what was actually transmitted on the wire.

A practical distinction:
- Use counters to answer “Is the interface healthy?”
- Use captures to answer “What exactly is happening in the packets?”

**SNMP and telemetry: centralized operational visibility**

Beyond syslog, networks often use **SNMP** (Simple Network Management Protocol) and telemetry to collect operational data centrally. SNMP is commonly used to poll device status and counters, while telemetry can stream data more continuously and with finer granularity.

At the CCNA level, know the purpose rather than deep implementation details:
- **SNMP polling** can collect interface status, CPU, memory, and counters.
- **SNMP traps** can notify a management station when an event occurs.
- **Telemetry** can provide near-real-time operational data to a collector.

Why this matters:
- Centralized monitoring lets you compare many devices from one console.
- Historical data helps identify trends before users notice outages.
- Alerts can be correlated with syslog messages and change windows.

Example of the operational model:
- A switch sends syslog messages to a log server.
- The same switch is polled by an NMS (Network Management System) for interface counters.
- A telemetry collector receives streaming interface utilization data.
- The operations team uses all three sources to isolate the fault faster.

**Documentation: the baseline for operations and troubleshooting**

Good documentation reduces mean time to repair because it tells you what is supposed to be connected, configured, and approved. In day-to-day operations, documentation is not static; it must be updated after changes so it remains trustworthy.

Important documentation artifacts include:
- **Network diagrams** showing physical and logical topology.
- **Port maps** showing switchport-to-device assignments.
- **Asset inventories** listing device models, serial numbers, locations, and ownership.
- **IP address plans** and VLAN assignments.
- **Configuration backups** and version history.
- **Rack elevations** and cabling records.
- **Change records** and maintenance windows.

Why each matters:
- Diagrams help you trace paths and identify upstream/downstream dependencies.
- Port maps help you find the correct switchport during troubleshooting or moves.
- Asset inventories help with lifecycle management, warranty checks, and replacement planning.
- Configuration backups support recovery after failure or accidental changes.

A simple documentation workflow might include:
1. Verify the current topology and port mapping before making a change.
2. Update the implementation plan and approval record.
3. Apply the change during the maintenance window.
4. Validate the result with `show` commands and user testing.
5. Update diagrams, inventories, and the change record after completion.

Example of a useful port map entry:

```text
Switch: SW1
Port: Gi1/0/10
Connected device: AP-3
Location: 2nd floor east hallway
VLANs: 20, 30
PoE: Yes
Notes: Ceiling access point, label matches patch panel 2A-10
```

**Change control: making changes safely and repeatably**

Change control is the process of planning, approving, implementing, and validating network changes. It reduces outages by making changes predictable and traceable. For CCNA, focus on the idea that even small changes can have broad impact if they affect routing, VLANs, trunks, ACLs, or default gateways.

Common change-management artifacts include:
- **MOP** (Method of Procedure): step-by-step implementation instructions.
- **Implementation plan**: what will be changed, by whom, and when.
- **Rollback plan**: how to return to the previous state if validation fails.
- **Approval workflow**: who must review and authorize the change.
- **Maintenance window**: the approved time period for the change.
- **Validation checklist**: tests to confirm success after implementation.
- **Emergency change record**: documentation created after urgent work.

A good MOP is specific and testable. It should include:
- Scope and objective.
- Devices and interfaces affected.
- Pre-change checks.
- Exact commands or actions.
- Validation steps.
- Rollback steps.
- Communication plan.

Example MOP outline:
1. Confirm current interface and VLAN state.
2. Back up the running configuration.
3. Apply the change.
4. Verify connectivity and logs.
5. Roll back if validation fails.

Example backup and change workflow:

```cisco
show running-config
copy running-config startup-config
copy running-config tftp:
```

If you use a TFTP server, you might save a backup before making changes:
```cisco
copy running-config tftp:
Address or name of remote host []? 192.0.2.60
Destination filename [router-confg]? R1-before-change.cfg
```

**Approval and emergency handling**

Normal changes should follow an approval workflow so stakeholders understand risk and timing. The approval process often includes peer review, operations sign-off, and business authorization for changes that may affect users.

Emergency changes are different:
- They address urgent outages or security issues.
- They may be implemented before full approval if delay would cause greater harm.
- They still require documentation after the fact.
- They should include a post-change review and root-cause analysis.

Exam-oriented distinctions:
- **Standard change**: routine, low-risk, repeatable.
- **Normal change**: planned and approved through the standard workflow.
- **Emergency change**: expedited due to urgent operational need.
- **Rollback**: restoration of the previous known-good state.

> If a change is not documented, validated, and reversible, treat it as incomplete from an operations perspective.

**Practical command interpretation: show logging, show interfaces, and neighbor detail**

You should be comfortable reading common output and extracting the operational meaning.

`show logging`:
- Confirms recent events and their timestamps.
- Helps correlate failures with user reports or maintenance actions.
- Reveals severity and subsystem source.

Example:
```cisco
Switch# show logging | include LINK|LINEPROTO|SPANTREE
```

Interpretation:
- Use filters to focus on relevant events.
- Look for repeated messages that indicate flapping or instability.
- Match the event time to a recent change or physical intervention.

`show interfaces`:
- Confirms link state, protocol state, speed, duplex, and error counters.
- Helps distinguish a down interface from a degraded one.
- Shows whether the problem is physical, congestion-related, or configuration-related.

Example:
```cisco
Switch# show interfaces gigabitEthernet1/0/10
```

Interpretation checklist:
- `is up, line protocol is up` means Layer 1 and Layer 2 are operational.
- `is administratively down` means the interface is shut.
- Rising CRC or input errors suggest a physical issue.
- Output drops may suggest congestion or queueing.

`show cdp neighbors detail` and `show lldp neighbors detail`:
- Reveal directly connected devices and their management addresses.
- Help you validate cabling and identify what is connected to a port.
- Support documentation by confirming topology and device identity.

Example:
```cisco
Switch# show cdp neighbors detail
```

Typical uses:
- Confirm that a port connects to the expected switch, router, or access point.
- Find the neighbor’s IP address for remote management.
- Detect an unexpected device on a port.

LLDP is vendor-neutral, while CDP (Cisco Discovery Protocol) is Cisco-proprietary. In mixed environments, LLDP is often preferred for interoperability, but both can be useful on Cisco devices.

**Troubleshooting workflow using logs, counters, and documentation**

A practical troubleshooting sequence is to start broad, then narrow the scope.

1. Check documentation to confirm the expected topology and port assignment.
2. Review logs for recent events, flaps, or errors.
3. Inspect interface counters for physical or congestion symptoms.
4. Check neighbor discovery to validate the connected device.
5. Use packet capture or deeper protocol analysis if counters do not explain the issue.
6. Record the findings and update documentation after resolution.

Example scenario:
- Users report intermittent connectivity on a floor switch.
- `show logging` reveals repeated interface up/down messages.
- `show interfaces` shows increasing CRC errors on the uplink.
- `show cdp neighbors detail` confirms the uplink connects to the expected distribution switch.
- The port map shows the cable was recently moved during a rack cleanup.
- The fix is to replace or reseat the cable and validate that errors stop increasing.

**What to remember for the exam**

- **Logs** provide event history and are often centralized with syslog.
- **Counters** reveal trends and error patterns, but they do not always explain root cause.
- **Packet captures** are needed when counters are not enough to inspect actual traffic.
- **SNMP** and telemetry support centralized monitoring beyond syslog.
- **Documentation** must include diagrams, port maps, and asset inventories.
- **Change control** uses MOPs, approvals, rollback plans, and validation steps.
- **Emergency changes** still require documentation and review after implementation.
- `show logging`, `show interfaces`, and `show cdp/lldp neighbors detail` are core operational commands.

**Quick comparison**

| Item | Best for | Limitation |
|---|---|---|
| Logs | Event timelines and fault correlation | May be overwritten locally |
| Counters | Detecting trends and interface health | Do not show packet contents |
| Packet captures | Inspecting actual frames and protocols | More detailed and time-consuming |
| Documentation | Knowing expected state and dependencies | Must be kept current |
| Change records | Tracking what changed and why | Only useful if completed accurately |

**Practical command set to memorize**

```cisco
show logging
show interfaces
show interfaces counters errors
show ip interface brief
show cdp neighbors detail
show lldp neighbors detail
show running-config
copy running-config startup-config
copy running-config tftp:
```

Use these commands together to answer three questions:
- What happened?
- What is the interface telling you?
- What should the network look like according to documentation?
