## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Overview**

Layer-by-layer isolation is a disciplined troubleshooting method that helps you find where communication fails by checking each layer of the network stack in order. In CCNA labs, this approach is especially useful when a host cannot reach a gateway, a remote subnet, or a service. You verify the path from the physical connection upward, stopping at the first layer that fails.

The goal is not to guess the cause. You collect evidence at each layer, eliminate what is working, and narrow the fault domain quickly. This is the same logic used in real operations when you need to distinguish between cabling issues, VLAN problems, IP addressing mistakes, routing failures, and application-layer issues.

**Troubleshooting Mindset**

A good isolation workflow starts with the symptom and ends with the smallest possible fault domain. You should ask:

- What is failing?
- What is the expected behavior?
- Which layer should be responsible for that behavior?
- What evidence proves the layer is healthy?

A practical rule is to move from the bottom up when the problem is unknown. If you already know the failure domain, you can jump directly to the likely layer, but in a lab exercise you usually want to demonstrate the full process.

**Layer-by-Layer Isolation Model**

Use the OSI (Open Systems Interconnection) model as a troubleshooting guide, not as a rigid checklist. Each layer depends on the one below it.

- **Layer 1 - Physical**
  - Check power, cabling, link lights, transceiver seating, and interface status.
  - A down interface, bad cable, or speed/duplex mismatch can stop all higher-layer communication.
- **Layer 2 - Data Link**
  - Verify VLAN membership, trunking, MAC learning, and switchport mode.
  - If a host is in the wrong VLAN, it may reach only devices in that broadcast domain.
- **Layer 3 - Network**
  - Confirm IP address, subnet mask, default gateway, routing table, and reachability with ICMP (Internet Control Message Protocol).
  - A host can have link and VLAN connectivity but still fail if the gateway or route is wrong.
- **Layer 4 - Transport**
  - Check whether TCP (Transmission Control Protocol) or UDP (User Datagram Protocol) ports are open and whether sessions establish correctly.
  - A ping can succeed while an application port is blocked.
- **Layer 5-7 - Session, Presentation, Application**
  - Validate service availability, authentication, DNS, HTTP, SSH, or other application dependencies.
  - The network may be healthy while the service itself is down.

**Common Lab Workflow**

In a lab, you often begin with a simple test from the source host and work outward.

1. Verify the local host configuration.
   - Check IP address, subnet mask, default gateway, and DNS settings.
   - Confirm the host is connected to the correct switchport or wireless network.
2. Test local connectivity.
   - Ping the host itself, then the default gateway.
   - If the gateway is unreachable, focus on Layer 1, Layer 2, or local Layer 3 settings.
3. Test same-subnet and remote-subnet reachability.
   - If same-subnet devices work but remote devices fail, suspect routing or gateway issues.
4. Test name resolution and application access.
   - If IP connectivity works but names fail, inspect DNS.
   - If ping works but the service fails, inspect ports, ACLs, or the server application.
5. Verify the network device path.
   - Check switch and router interfaces, VLANs, trunks, routes, and ACLs.
   - Compare the working path to the failing path.

**Useful CLI Commands**

These commands help you isolate faults on Cisco devices and end hosts.

```bash
# End-host checks
ipconfig /all
ping 192.168.1.1
tracert 8.8.8.8
nslookup example.com
```

```cisco
! Switch checks
show interfaces status
show interfaces
show vlan brief
show interfaces trunk
show mac address-table
show cdp neighbors
show lldp neighbors
```

```cisco
! Router checks
show ip interface brief
show running-config interface g0/0
show ip route
show arp
show access-lists
show cdp neighbors
```

These commands answer different questions:

- `show ip interface brief` tells you whether interfaces are up/up, down/down, or administratively down.
- `show vlan brief` confirms access-port VLAN assignment.
- `show interfaces trunk` verifies trunk negotiation and allowed VLANs.
- `show ip route` confirms whether the router knows how to reach the destination network.
- `show arp` helps confirm Layer 3 to Layer 2 resolution on directly connected networks.
- `show access-lists` helps identify filtering that blocks traffic after routing succeeds.

**Layer 1 Checks**

Physical problems are often the fastest to confirm and the easiest to overlook. If the interface is down, no amount of routing or ACL analysis will help.

- Check that the cable is seated correctly.
- Verify the interface is not administratively shut down.
- Confirm the correct speed and duplex settings if the environment requires manual configuration.
- Inspect for damaged cables, bad optics, or incorrect patching.
- Look for interface errors such as CRC (Cyclic Redundancy Check), input errors, or collisions.

Example interface inspection:

```cisco
show interfaces g0/1
show ip interface brief
```

If you see `administratively down`, enable the interface:

```cisco
conf t
interface g0/1
no shutdown
end
```

> If a link is physically down, do not continue to higher-layer troubleshooting until the interface is stable.

**Layer 2 Checks**

Once the physical link is up, confirm that frames are being switched in the correct broadcast domain. Many lab failures come from VLAN mismatches, trunk issues, or incorrect access-port settings.

- Verify the access port belongs to the intended VLAN.
- Confirm the trunk is up and carrying the required VLANs.
- Check whether the native VLAN matches on both ends of a trunk.
- Confirm the switch has learned the host MAC address on the expected port.
- Use CDP (Cisco Discovery Protocol) or LLDP (Link Layer Discovery Protocol) to identify neighboring devices and validate topology.

Example:

```cisco
show vlan brief
show interfaces trunk
show mac address-table dynamic
show cdp neighbors detail
```

Typical Layer 2 symptoms include:

- A host can communicate with devices in the same VLAN but not with the default gateway.
- A trunk link is up, but the needed VLAN is not allowed.
- A port is connected, but the switch learned the MAC on a different interface than expected.
- Two switches disagree on trunk parameters, causing VLAN traffic loss.

Example access-port configuration:

```cisco
conf t
interface f0/10
switchport mode access
switchport access vlan 20
spanning-tree portfast
end
```

Example trunk configuration:

```cisco
conf t
interface g0/1
switchport mode trunk
switchport trunk allowed vlan 10,20,30
switchport trunk native vlan 99
end
```

**Layer 3 Checks**

If Layer 2 is correct, verify IP addressing and routing. A host can be perfectly connected at Layer 2 and still fail because of a wrong subnet mask, missing default gateway, or absent route.

- Confirm the host IP address is in the correct subnet.
- Verify the subnet mask matches the design.
- Check the default gateway on the host.
- Confirm the router interface has the correct IP address and is up.
- Inspect the routing table for a route to the destination network.
- Use ping and traceroute to identify where packets stop.

Example host verification:

```bash
ipconfig /all
ping 192.168.10.1
tracert 10.10.10.10
```

Example router verification:

```cisco
show ip interface brief
show ip route
show arp
```

Common Layer 3 issues:

- Wrong default gateway on the host.
- Incorrect subnet mask causing the host to believe a remote device is local.
- Missing static route.
- Dynamic routing not converged or not configured.
- Interface IP address mismatch between router and host subnet.
- Overlapping or duplicate IP addresses.

Example static route:

```cisco
conf t
ip route 10.10.20.0 255.255.255.0 192.168.1.2
end
```

**Layer 4 Checks**

Transport-layer problems appear when the network path exists but the application session does not. This is common when ping works but a service such as SSH, HTTP, or DNS fails.

- Confirm the destination port is listening.
- Check whether an ACL blocks the port.
- Verify stateful firewall behavior if present.
- Test with the correct protocol and port.
- Distinguish between ICMP reachability and TCP session establishment.

Examples:

```bash
# Test TCP connectivity from a host
telnet 10.10.10.20 22
nc -vz 10.10.10.20 80
```

```cisco
show access-lists
show run | include access-group
```

A successful ping only proves ICMP reachability. It does not prove that TCP 22, TCP 80, or UDP 53 is available.

**Layer 5-7 Checks**

If lower layers are healthy, focus on the service itself. Many “network” problems are actually application or name-resolution problems.

- Verify the server process is running.
- Confirm the service is bound to the correct interface and port.
- Check DNS records if users access the service by name.
- Validate credentials, certificates, and application-specific settings.
- Confirm the server firewall allows the required traffic.

Examples:

```bash
nslookup server1.example.com
curl http://10.10.10.50
ssh admin@10.10.10.50
```

If a name lookup fails but the IP address works, the issue is likely DNS rather than routing. If the service refuses connections, the application may be stopped or filtered.

**Practical Isolation Sequence**

Use a repeatable sequence so you do not skip evidence.

1. Confirm the symptom.
   - Identify the source, destination, and type of failure.
2. Check the local host.
   - Verify IP settings, link status, and correct VLAN or wireless association.
3. Check the first-hop device.
   - Inspect the switchport, trunk, or router interface.
4. Check the path.
   - Review routes, ACLs, and intermediate device status.
5. Check the destination.
   - Confirm the server or remote interface is up and configured correctly.
6. Test the application.
   - Validate DNS, ports, authentication, and service health.

This sequence helps you avoid “fixing” the wrong device. For example, if a host cannot reach a remote subnet, the root cause may be a missing route on the router, not the host itself.

**Example Lab Scenario**

Suppose PC1 cannot ping Server1 in another subnet.

- PC1 has link lights and can ping its own gateway.
- `show vlan brief` confirms PC1 is in VLAN 20.
- `show interfaces trunk` shows the trunk is up, but VLAN 20 is missing from the allowed list.
- The router has the correct route, and Server1 is reachable from another host.

In this case, the fault is Layer 2 on the trunk. The fix is to allow VLAN 20 on the trunk.

```cisco
conf t
interface g0/1
switchport trunk allowed vlan add 20
end
```

After the change, retest from PC1:

```bash
ping 10.10.30.10
tracert 10.10.30.10
```

**Verification and Documentation**

After you fix the issue, verify that the original symptom is gone and that you did not create a new problem.

- Retest the original source and destination.
- Confirm alternate paths still work.
- Save the configuration if the change is intended to persist.
- Record the root cause, the evidence, and the fix.

Example save command:

```cisco
copy running-config startup-config
```

Good documentation should include:

- The failing symptom.
- The layer where the fault was isolated.
- The commands used to prove the issue.
- The exact configuration change made.
- The final verification result.

**Common Mistakes**

- Jumping to routing before checking physical and VLAN status.
- Assuming ping success means the application is healthy.
- Forgetting to verify the default gateway on the host.
- Overlooking trunk allowed VLAN lists.
- Ignoring ACLs when a route exists but traffic still fails.
- Changing multiple variables at once, which makes root cause analysis harder.

**Exam and Lab Tips**

- Use a bottom-up approach when the failure is unknown.
- Compare a working device or subnet to the failing one.
- Read command output carefully; the first clue is often in interface state or VLAN membership.
- Separate connectivity problems from service problems.
- Make one change at a time and verify after each change.

**Quick Reference**

- **Layer 1**: power, cable, interface state, errors
- **Layer 2**: VLANs, trunks, MAC learning, switchport mode
- **Layer 3**: IP address, mask, gateway, routes, ICMP
- **Layer 4**: TCP/UDP ports, ACLs, session establishment
- **Layer 5-7**: DNS, authentication, application service, certificates

A disciplined layer-by-layer method is one of the most reliable troubleshooting skills in networking. It reduces guesswork, speeds up isolation, and gives you a repeatable process you can use in both labs and real environments.
