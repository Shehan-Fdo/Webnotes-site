## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Failure domains: the core idea**

A **failure domain** is the portion of a network that is affected when a device, link, service, or configuration fails. The smaller the failure domain, the easier it is to contain outages and troubleshoot problems. In practice, you design and operate networks so that a single fault does not take down unrelated users, VLANs, subnets, or services.

Layer-by-layer isolation means you diagnose and design the network by separating issues into the OSI (Open Systems Interconnection) layers. You verify one layer at a time, starting with the physical path and moving upward only after the lower layer is confirmed healthy. This approach reduces guesswork and prevents you from chasing symptoms at the wrong layer.

- **Containment** limits how far a fault spreads.
- **Isolation** helps you identify the exact layer or component causing the issue.
- **Redundancy** reduces the chance that one failure becomes an outage.
- **Segmentation** keeps broadcast, routing, and security problems from affecting the whole network.

**Why failure domains matter**

A large failure domain increases blast radius. For example, if many access switches depend on one uplink, one power source, or one distribution device, a single failure can affect a large number of endpoints. Smaller domains improve availability because faults are localized.

Common examples of failure domains include:
- A single access switch serving one floor
- One VLAN spanning many endpoints
- One router or Layer 3 switch acting as the default gateway
- One WAN circuit carrying all branch traffic
- One DHCP (Dynamic Host Configuration Protocol) server serving multiple subnets
- One spanning tree root or one HSRP (Hot Standby Router Protocol) active gateway

A good design asks: if this component fails, what stops working, and how many users are affected?

**Layer-by-layer troubleshooting mindset**

When troubleshooting, verify the network from the bottom up. If a lower layer is broken, upper-layer tests can mislead you.

A practical order is:
1. Check power, cabling, and link status.
2. Verify interface configuration and speed/duplex where relevant.
3. Confirm Layer 2 connectivity, VLAN membership, and trunking.
4. Confirm Layer 3 addressing, routing, and gateway reachability.
5. Test name resolution and application access.

This method is useful because each layer depends on the one below it. For example, if a switch port is down, no amount of IP troubleshooting will fix the problem until the physical issue is resolved.

**OSI layers and typical failure domains**

| Layer | What you verify | Typical failure domain examples |
|---|---|---|
| Layer 1 | Signal, cable, optics, power, port state | Bad cable, unplugged fiber, failed transceiver, powered-off device |
| Layer 2 | MAC learning, VLANs, trunks, STP (Spanning Tree Protocol) | Wrong VLAN, trunk mismatch, loop, blocked port, port-security violation |
| Layer 3 | IP addressing, routing, default gateway, ACLs (Access Control Lists) | Wrong subnet mask, missing route, gateway failure, ACL blocking traffic |
| Layer 4 | Ports, sessions, transport behavior | Firewall blocking TCP/UDP port, session timeout, NAT issue |
| Layer 7 | DNS, HTTP, SSH, application logic | Bad DNS record, server outage, authentication failure, app misconfiguration |

**Layer 1 isolation**

Layer 1 problems are the easiest to localize because they are usually visible in interface state or physical inspection. You isolate them by checking the physical path end to end.

Useful checks:
- Verify the device has power and fans/LEDs indicate normal operation.
- Inspect patch cables, fiber cleanliness, and transceiver seating.
- Check whether the interface is administratively down or operationally down.
- Confirm the correct cable type and optic type are used.

Common CLI commands:
```bash
show interfaces status
show interfaces gigabitEthernet0/1
show ip interface brief
show logging
```

What to look for:
- `administratively down` means the interface is shut down in configuration.
- `notconnect` or `down/down` often indicates a physical issue.
- `err-disabled` suggests a protection feature disabled the port due to a fault condition.

> If a port is `err-disabled`, do not just bounce it repeatedly. First identify the cause, such as port security, BPDU Guard, link-flap, or storm control.

**Layer 2 isolation**

Layer 2 failures affect switching, VLAN membership, and loop prevention. These issues often appear as “some devices can talk, others cannot” or “broadcast traffic is strange.”

Key checks:
- Confirm the access port is in the correct VLAN.
- Confirm trunks allow the expected VLANs.
- Verify native VLAN consistency on both ends of a trunk.
- Check MAC address learning and forwarding behavior.
- Review STP state if a port is blocked or topology changes occur.

Useful commands:
```bash
show vlan brief
show interfaces trunk
show spanning-tree
show mac address-table
show cdp neighbors detail
```

Practical examples:
- If a PC is connected to the wrong access VLAN, it may receive an IP address from the wrong subnet or fail to reach its gateway.
- If a trunk does not allow a VLAN, hosts in that VLAN may be isolated even though the physical link is up.
- If STP blocks a port to prevent a loop, traffic may fail on one path while another path remains active.

Layer 2 isolation is especially important in campus networks because a loop can create a broadcast storm that affects many devices. STP reduces this risk, but misconfiguration can still create outages.

**Layer 3 isolation**

Layer 3 problems involve IP addressing and routing. Once Layer 2 is confirmed, you verify whether the source and destination are in the same subnet or require routing.

Key checks:
- Confirm the host IP address, subnet mask, and default gateway.
- Verify the router or Layer 3 switch has the correct interface IP.
- Check routing tables and route reachability.
- Confirm ACLs are not blocking traffic.
- Verify NAT (Network Address Translation) if traffic crosses an edge device.

Useful commands:
```bash
show ip interface brief
show ip route
show running-config | section interface
show access-lists
ping 192.0.2.1
traceroute 198.51.100.10
```

Practical examples:
- A host with the wrong subnet mask may think a remote host is local and ARP for it instead of sending traffic to the gateway.
- A missing default route on a branch router can break Internet access while local LAN access still works.
- An ACL applied in the wrong direction can block return traffic and make a service appear unreachable.

**Layer 4 and Layer 7 isolation**

Transport and application issues often look like “the network is down” even when connectivity exists. You isolate these by proving lower-layer reachability first, then testing the specific service.

Layer 4 checks:
- Confirm the correct TCP or UDP port is open.
- Verify firewalls and security policies allow the session.
- Check for session resets, timeouts, or asymmetric routing.

Layer 7 checks:
- Verify DNS resolution.
- Confirm the application server is reachable and healthy.
- Check authentication, certificates, and application logs.

Useful commands:
```bash
nslookup www.example.com
dig www.example.com
telnet 203.0.113.20 443
nc -vz 203.0.113.20 443
curl -I https://www.example.com
```

A host may successfully ping a server but still fail to load a web page because ICMP (Internet Control Message Protocol) is allowed while TCP 443 is blocked. That is why you test the actual service, not only basic reachability.

**Designing smaller failure domains**

You reduce risk by designing the network so that faults stay local.

Common design techniques:
- Use VLANs to segment user groups and services.
- Use routed access or Layer 3 boundaries to limit broadcast propagation.
- Use redundant uplinks and devices for critical paths.
- Use first-hop redundancy protocols such as HSRP, VRRP (Virtual Router Redundancy Protocol), or GLBP (Gateway Load Balancing Protocol) where appropriate.
- Place critical services on redundant power and network paths.
- Separate management, user, voice, and server traffic.

Examples of reduced failure domains:
- One access switch failure affects only the endpoints on that switch.
- One VLAN issue affects only that broadcast domain, not the entire campus.
- One distribution switch failure is tolerated if a second distribution device can take over gateway duties.

**Redundancy and its tradeoffs**

Redundancy improves availability, but it can also increase complexity. More links and devices create more configuration points and more opportunities for misconfiguration.

Benefits:
- Higher uptime
- Faster recovery from hardware failure
- Maintenance without full outage

Tradeoffs:
- More complex troubleshooting
- Potential for loops at Layer 2
- Need for consistent configuration across redundant devices
- Higher cost

A resilient design uses redundancy intentionally, not randomly. For example, two uplinks to two distribution switches are useful only if spanning tree, EtherChannel, or Layer 3 routing is designed correctly.

**Practical troubleshooting workflow**

1. Identify the scope of the outage.
   - One host, one VLAN, one switch, one site, or the whole network?
2. Determine the highest layer that is known good.
   - If the link is down, stop at Layer 1.
3. Verify the next dependent layer.
   - For example, after link status, check VLANs and trunks.
4. Compare against a known-good device or interface.
   - Differences often reveal the fault quickly.
5. Test the actual service.
   - Ping, traceroute, DNS lookup, or application-specific checks.
6. Confirm the fix and document the root cause.

**Example troubleshooting sequence**

Suppose a user cannot reach a file server.

1. Check the PC link and IP configuration.
```bash
ipconfig /all
```

2. Verify the default gateway is reachable.
```bash
ping 192.168.10.1
```

3. Confirm the switch port is in the correct VLAN.
```bash
show vlan brief
show interfaces status
```

4. Check the routing path to the server subnet.
```bash
show ip route
traceroute 192.168.50.20
```

5. Test the service port if ping works.
```bash
nc -vz 192.168.50.20 445
```

This sequence prevents you from assuming the server is broken when the real issue may be a VLAN mismatch or missing route.

**Useful Cisco verification commands**

```bash
show interfaces status
show interfaces counters errors
show vlan brief
show interfaces trunk
show spanning-tree
show mac address-table dynamic
show ip interface brief
show ip route
show cdp neighbors
show lldp neighbors
show logging
```

These commands help you map the fault to a specific layer and device. `CDP` (Cisco Discovery Protocol) and `LLDP` (Link Layer Discovery Protocol) are especially useful for identifying neighboring devices and confirming topology.

**Key exam takeaways**

- A **failure domain** is the area impacted by a fault.
- Smaller failure domains improve availability and simplify troubleshooting.
- Layer-by-layer isolation means you verify from Layer 1 upward.
- Do not troubleshoot higher layers until lower layers are confirmed healthy.
- VLANs, routing boundaries, and redundancy all help contain failures.
- The same symptom can have different causes at different layers, so always validate the actual layer in question.

**Quick mental model**

Think of the network as stacked dependencies:
- If Layer 1 fails, everything above it fails.
- If Layer 2 fails, Layer 3 and above may still be configured but cannot pass traffic.
- If Layer 3 fails, applications may still be running but unreachable.
- If Layer 4 or 7 fails, connectivity may exist while the service is still unusable.

This layered approach is one of the fastest ways to troubleshoot efficiently and design networks that fail gracefully.
