## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**DHCP fundamentals**

<abbr title="Dynamic Host Configuration Protocol">DHCP</abbr> automatically assigns IP configuration to clients so you do not have to statically configure every host. A DHCP server typically provides:

- IP address
- Subnet mask
- Default gateway
- DNS server addresses
- Lease time and other options

DHCP is especially useful in user access networks, wireless networks, guest networks, and any environment where devices join and leave frequently. It reduces manual errors and makes address management scalable.

The client-server exchange uses broadcast at first because the client does not yet know the server’s address. After the initial discovery, the client and server can communicate using unicast in later stages, depending on the state of the exchange.

**DHCP message flow**

The classic DHCPv4 exchange is often remembered as <mark>DORA</mark>:

1. **Discover** - the client broadcasts to find available DHCP servers.
2. **Offer** - one or more servers propose an address and options.
3. **Request** - the client selects one offer and requests that lease.
4. **Acknowledge** - the chosen server confirms the lease.

A client may receive multiple offers. It usually accepts one offer and sends a DHCPREQUEST that identifies the selected server, which tells the other servers to withdraw their offers. This is important because the request is not just “please give me an address”; it also signals which offer the client accepted.

> A DHCP client can also receive a **DHCPNAK** if the server rejects the request, usually because the address is invalid for that subnet, the lease is no longer valid, or the client is using the wrong network information.

**Lease lifecycle and timing**

DHCP leases are time-bound. The server grants an address for a lease period, and the client must renew it before expiration.

- **T1 renewal timer**: the client tries to renew with the original server, usually at about 50% of the lease time.
- **T2 rebinding timer**: if renewal fails, the client broadcasts to any available DHCP server, usually at about 87.5% of the lease time.
- **Expiration**: if the lease expires without renewal, the client must stop using the address and restart the DHCP process.

This timing matters in troubleshooting because a client may appear to “keep working” until the lease expires, then suddenly lose connectivity.

A client can also send **DHCPDECLINE** if it detects that the offered address is already in use, often through an address conflict check such as ARP probing. That tells the server to mark the address as unusable for the moment.

**Server roles and scope selection**

A DHCP server manages one or more address pools, often called scopes in general networking terms. The server chooses the correct pool based on the client’s subnet and request context.

Common selection inputs include:

- The subnet the client belongs to
- The relay information inserted by a router or Layer 3 switch
- The `giaddr` field in the DHCP packet
- Option 82, if the relay inserts relay-agent information and the server is configured to use it

The most important relay-related field for CCNA is **`giaddr`**. When a relay forwards a client broadcast to a DHCP server, it sets the gateway IP address field to the address of the relay interface that received the client request. The server uses that field to determine which subnet the client came from and which scope should be used for the reply.

In practical terms, the server does not need to be on the same VLAN as the client. The relay tells the server, “this request came from that subnet,” and the server returns the correct address for that network.

**DHCP relay and `ip helper-address`**

Routers and Layer 3 switches do not forward broadcasts by default, so DHCP requests from one subnet will not reach a server on another subnet unless you configure relay.

On Cisco IOS, you typically configure relay with `ip helper-address` on the client-facing interface or SVI. This causes the device to forward selected UDP broadcasts to the DHCP server as unicast.

Example:

```cisco
interface vlan 10
 ip address 192.168.10.1 255.255.255.0
 ip helper-address 10.10.10.5
```

In this example:

- Clients in VLAN 10 broadcast DHCPDISCOVER.
- The Layer 3 interface receives the broadcast.
- The device forwards the request to `10.10.10.5`.
- The relay inserts the local interface address into `giaddr`.
- The server uses that information to choose the correct pool for VLAN 10.

Cisco IOS helper forwarding is broader than DHCP alone. By default, `ip helper-address` forwards several UDP services, not just DHCP/BOOTP. Commonly forwarded services include:

- DHCP/BOOTP on UDP 67 and 68
- TFTP on UDP 69
- DNS on UDP 53
- Time service on UDP 37
- NetBIOS name service on UDP 137
- NetBIOS datagram service on UDP 138
- TACACS+ on UDP 49

> Do not assume `ip helper-address` is only for DHCP. On exam questions, the key idea is that it forwards multiple UDP broadcasts unless you restrict the behavior.

If you want to reduce helper forwarding, you can disable specific forwarded UDP services with `no ip forward-protocol udp <port>`. For example:

```cisco
no ip forward-protocol udp 37
no ip forward-protocol udp 49
no ip forward-protocol udp 53
no ip forward-protocol udp 69
no ip forward-protocol udp 137
no ip forward-protocol udp 138
```

Use this carefully. The exact default list can vary by platform and IOS version, so the safe exam takeaway is that helper forwarding is not limited to DHCP and can be tuned.

**Small topology example**

A common CCNA scenario is a router or multilayer switch connecting multiple VLANs:

- VLAN 10: `192.168.10.0/24`
- VLAN 20: `192.168.20.0/24`
- DHCP server: `10.10.10.5` in a server VLAN

Packet flow for a client in VLAN 10:

1. Client broadcasts DHCPDISCOVER on VLAN 10.
2. The SVI or router interface for VLAN 10 receives the broadcast.
3. `ip helper-address 10.10.10.5` forwards the packet to the server.
4. The relay sets `giaddr` to the VLAN 10 interface address.
5. The server selects the VLAN 10 scope and sends DHCPOFFER back to the relay.
6. The relay forwards the offer to the client.
7. The client sends DHCPREQUEST.
8. The server sends DHCPACK and the client configures the lease.

This is the core relay pattern you should recognize in diagrams and troubleshooting questions.

**Cisco IOS configuration examples**

A basic DHCP server on Cisco IOS can be configured with a pool:

```cisco
ip dhcp excluded-address 192.168.10.1 192.168.10.20

ip dhcp pool VLAN10
 network 192.168.10.0 255.255.255.0
 default-router 192.168.10.1
 dns-server 8.8.8.8 1.1.1.1
 lease 7
```

Notes:

- `ip dhcp excluded-address` reserves addresses for infrastructure devices such as gateways, printers, and switches.
- `network` defines the subnet served by the pool.
- `default-router` is the client’s default gateway.
- `dns-server` supplies name resolution.
- `lease` sets the lease duration in days, hours, and minutes depending on syntax support.

A host reservation on Cisco IOS is usually done with a host-specific pool. A common pattern is:

```cisco
ip dhcp pool PRINTER1
 host 192.168.10.50 255.255.255.0
 hardware-address 0011.2233.4455
 default-router 192.168.10.1
 dns-server 8.8.8.8
```

You can also use a client identifier in some designs:

```cisco
ip dhcp pool PRINTER1
 host 192.168.10.50 255.255.255.0
 client-identifier 01aa.bbcc.ddee.ff
 default-router 192.168.10.1
```

Use reservations when a device needs a predictable address but you still want centralized DHCP management.

**Verification and troubleshooting**

Useful verification commands include:

```cisco
show ip dhcp binding
show ip dhcp pool
show ip dhcp conflict
show running-config | section dhcp
show running-config interface vlan 10
show ip interface vlan 10
```

What to look for:

- **`show ip dhcp binding`** confirms active leases.
- **`show ip dhcp pool`** shows pool utilization and available addresses.
- **`show ip dhcp conflict`** helps identify duplicate-address issues.
- **`show ip interface`** or interface configuration confirms whether `ip helper-address` is applied.
- **`show running-config interface`** is often the fastest way to verify relay configuration.

Common symptoms and likely causes:

- **Client gets APIPA address (`169.254.x.x`)**
  - No DHCP server response reached the client.
  - Relay is missing or misconfigured.
  - VLAN or trunking is broken.
  - Server pool is exhausted.

- **Client gets an address from the wrong subnet**
  - Wrong DHCP pool matched.
  - Relay `giaddr` points to the wrong interface.
  - Server scope selection is incorrect.

- **Client broadcasts but server never sees it**
  - `ip helper-address` is absent on the client VLAN interface.
  - ACLs or firewall rules block UDP 67/68.
  - The relay path to the server is down.

- **Server sees the request but client never gets a lease**
  - Return path from server to relay is broken.
  - The relay interface is down.
  - The server is sending the offer to the wrong address or scope.

**Troubleshooting workflow**

1. Confirm the client VLAN and gateway interface are up.
2. Verify `ip helper-address` on the correct SVI or routed interface.
3. Check that the DHCP server is reachable from the relay device.
4. Confirm the server pool matches the client subnet.
5. Inspect bindings and conflicts on the server.
6. Test with a client renew or release/renew cycle.

Example client-side refresh commands:

```bash
ipconfig /release
ipconfig /renew
```

On Linux:

```bash
sudo dhclient -r
sudo dhclient
```

**DHCP and security considerations**

DHCP is easy to abuse if you do not protect the access layer. A rogue DHCP server can hand out incorrect gateways or DNS servers and redirect traffic.

Common protections include:

- **DHCP snooping** on switches to trust only legitimate server-facing ports
- Rate limiting DHCP messages on untrusted ports
- Binding tables that track MAC, IP, VLAN, and port associations
- Filtering unauthorized server responses

For CCNA-level understanding, the key idea is that DHCP snooping helps prevent rogue servers and supports other Layer 2 security features.

**Exam focus summary**

- DHCP uses a client-server model with the DORA exchange.
- The client may receive multiple offers and chooses one server.
- `DHCPNAK` rejects an invalid request; `DHCPDECLINE` reports an address conflict.
- Lease timers include renewal, rebinding, and expiration.
- DHCP relay is needed when the server is in another subnet.
- Cisco `ip helper-address` forwards multiple UDP broadcasts, not only DHCP.
- The relay uses `giaddr` to identify the client subnet for scope selection.
- Verification usually relies on `show ip dhcp binding`, `show ip dhcp pool`, and interface configuration checks.
