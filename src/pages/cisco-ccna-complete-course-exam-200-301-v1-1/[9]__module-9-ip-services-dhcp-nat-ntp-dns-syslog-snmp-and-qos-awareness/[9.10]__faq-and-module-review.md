## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**What this review covers**

This module review ties together the most exam-relevant CCNA topics: network fundamentals, switching, routing, IP services, security, automation, and troubleshooting. Use it to check whether you can explain a concept, configure it, and verify it on real Cisco devices.

A strong CCNA answer usually includes:
- the purpose of the feature,
- the key configuration commands,
- the verification commands,
- and the most likely failure modes.

**OSI and TCP/IP model review**

The Open Systems Interconnection (<abbr title="Open Systems Interconnection">OSI</abbr>) model is still useful for troubleshooting because it gives you a structured way to isolate problems. The lower layers handle physical connectivity and framing, while upper layers handle addressing, sessions, and application behavior.

- **Layer 1**: cabling, optics, speed/duplex, signal issues.
- **Layer 2**: MAC addresses, switching, VLANs, trunks, EtherChannel, Spanning Tree Protocol (<abbr title="Spanning Tree Protocol">STP</abbr>).
- **Layer 3**: IP addressing, routing, subnetting, <abbr title="Dynamic Host Configuration Protocol">DHCP</abbr>, <abbr title="Network Address Translation">NAT</abbr>.
- **Layer 4**: TCP/UDP ports, session reliability, congestion behavior.
- **Layers 5-7**: application behavior, DNS, HTTP(S), SSH, email, APIs.

A practical troubleshooting habit is to ask:
1. Is the problem local or remote?
2. Is it Layer 1, 2, 3, or 4+?
3. What changed most recently?

**Switching fundamentals**

Switches forward frames using the destination MAC address and build the MAC address table dynamically from source addresses. If a destination is unknown, the switch floods the frame within the VLAN.

- **Broadcast domains** are separated by VLANs and routers.
- **Collision domains** are separated by switch ports.
- **Access ports** carry one VLAN.
- **Trunk ports** carry multiple VLANs using IEEE 802.1Q tagging.

A simple VLAN and trunk example:

```bash
conf t
vlan 10
 name USERS
vlan 20
 name VOICE

interface gi0/1
 switchport mode access
 switchport access vlan 10

interface gi0/24
 switchport mode trunk
 switchport trunk allowed vlan 10,20
end
```

Verify with:

```bash
show vlan brief
show interfaces trunk
show mac address-table
```

**STP and loop prevention**

STP prevents Layer 2 loops by blocking redundant paths. Without STP, a broadcast storm can overwhelm the network and cause MAC table instability.

- The **root bridge** is the switch with the lowest bridge ID.
- The root bridge is chosen by priority first, then MAC address.
- Non-root switches select one root port toward the root bridge.
- Each segment selects one designated port to forward traffic.

Common exam points:
- PortFast is for end-device ports, not switch-to-switch links.
- BPDU Guard protects PortFast ports by err-disabling the port if a Bridge Protocol Data Unit (<abbr title="Bridge Protocol Data Unit">BPDU</abbr>) arrives.
- Root Guard helps prevent an unexpected root bridge.

A useful mental diagram:

```text
[SW1 Root]
   |\
   | \__ [SW3]
   |
 [SW2]
```

If SW2 and SW3 both connect to SW1 and each other, STP blocks one redundant path to prevent a loop.

**EtherChannel and link aggregation**

EtherChannel bundles multiple physical links into one logical link. You use it to increase bandwidth and provide redundancy while keeping STP from blocking individual links in the bundle.

- **LACP** is the IEEE standard protocol.
- **PAgP** is Cisco proprietary.
- Both sides must agree on speed, duplex, trunking, and channel parameters.

Example:

```bash
conf t
interface range gi0/1-2
 channel-group 1 mode active
exit

interface port-channel 1
 switchport mode trunk
 switchport trunk allowed vlan 10,20
end
```

Verify with:

```bash
show etherchannel summary
show interfaces port-channel 1
```

**Inter-VLAN routing and router-on-a-stick**

VLANs isolate traffic at Layer 2, but hosts in different VLANs need Layer 3 routing to communicate. Router-on-a-stick uses one router interface with subinterfaces, each tagged for a VLAN.

```bash
conf t
interface g0/0
 no shutdown

interface g0/0.10
 encapsulation dot1Q 10
 ip address 192.168.10.1 255.255.255.0

interface g0/0.20
 encapsulation dot1Q 20
 ip address 192.168.20.1 255.255.255.0
end
```

On the switch, the router-facing port must be a trunk. If hosts in VLAN 10 cannot reach VLAN 20, check:
- trunk status,
- subinterface encapsulation,
- default gateway settings,
- ACLs,
- and whether the router interface is up.

**IPv4 addressing, subnetting, and route selection**

Subnetting determines how many hosts and networks fit into an address block. In CCNA scenarios, you should quickly identify:
- network address,
- broadcast address,
- usable host range,
- prefix length,
- and whether two hosts are in the same subnet.

Routing decisions use the **longest prefix match** rule. The most specific route wins, even if a less specific route also exists.

Example route selection:
- `10.0.0.0/8`
- `10.1.0.0/16`
- `10.1.1.0/24`

Traffic to `10.1.1.50` uses `10.1.1.0/24` because it is the most specific match.

**Static routes and default routes**

Static routes are manually configured and are useful for small networks, stub networks, and predictable paths. A default route is the route of last resort.

```bash
conf t
ip route 10.10.20.0 255.255.255.0 192.0.2.2
ip route 0.0.0.0 0.0.0.0 192.0.2.1
end
```

You can also point a static route to an exit interface on point-to-point links:

```bash
ip route 10.10.30.0 255.255.255.0 g0/1
```

Verify with:

```bash
show ip route
show ip route static
show running-config | include ip route
```

If a static route does not appear in the routing table, check whether the next hop is reachable and whether the outgoing interface is valid.

**Dynamic routing with OSPF**

Open Shortest Path First (<abbr title="Open Shortest Path First">OSPF</abbr>) is a link-state routing protocol. Routers form adjacencies, exchange link-state information, and compute the best path using the shortest path first algorithm.

Key exam ideas:
- OSPF uses areas; area 0 is the backbone.
- Router IDs identify routers in OSPF.
- Hello and dead timers must match on neighbors.
- Network type, area, and authentication settings can affect adjacency.

Basic single-area configuration:

```bash
conf t
router ospf 1
 router-id 1.1.1.1
 network 10.1.1.0 0.0.0.255 area 0
 network 10.1.2.0 0.0.0.255 area 0
end
```

Verify with:

```bash
show ip ospf neighbor
show ip route ospf
show ip protocols
```

If neighbors stay in `INIT` or `EXSTART`, suspect mismatched timers, MTU issues, or duplicate router IDs.

**First Hop Redundancy Protocols**

First Hop Redundancy Protocols (<abbr title="First Hop Redundancy Protocol">FHRP</abbr>) provide a shared default gateway for hosts. They improve gateway availability when a router or Layer 3 switch fails.

- **HSRP**: Hot Standby Router Protocol, Cisco proprietary.
- **VRRP**: Virtual Router Redundancy Protocol, open standard.
- **GLBP**: Gateway Load Balancing Protocol, Cisco proprietary and load-balances across gateways.

Typical behavior:
- Hosts use a virtual IP as their default gateway.
- One device is active/master and another is standby/backup.
- If the active device fails, another device takes over the virtual IP and MAC behavior.

Example HSRP configuration:

```bash
conf t
interface g0/0
 ip address 192.168.10.2 255.255.255.0
 standby 10 ip 192.168.10.1
 standby 10 priority 110
 standby 10 preempt
end
```

Verify with:

```bash
show standby brief
show standby
```

Use FHRP when gateway redundancy matters more than simple routing. In exam questions, the virtual IP is what hosts configure as their default gateway.

**NAT and PAT**

Network Address Translation (<abbr title="Network Address Translation">NAT</abbr>) translates private addresses to public addresses. Port Address Translation (<abbr title="Port Address Translation">PAT</abbr>) is a form of NAT overload that maps many inside hosts to one public IP using different source ports.

Why you use it:
- conserve public IPv4 addresses,
- hide internal addressing,
- allow private networks to reach the Internet.

Basic PAT example:

```bash
conf t
access-list 1 permit 192.168.10.0 0.0.0.255

interface g0/0
 ip address 192.168.10.1 255.255.255.0
 ip nat inside

interface g0/1
 ip address 203.0.113.2 255.255.255.252
 ip nat outside

ip nat inside source list 1 interface g0/1 overload
end
```

Verify with:

```bash
show ip nat translations
show ip nat statistics
show access-lists
```

Common NAT troubleshooting:
- inside and outside interfaces reversed,
- ACL does not match inside addresses,
- no route to the Internet,
- overlapping address space,
- translation table not building because traffic has not been generated.

**WAN and Internet edge concepts**

A WAN connects sites over service provider infrastructure. CCNA-level WAN review focuses on how you connect to the Internet or remote sites and how you identify the role of each edge component.

Common concepts:
- A **default route** points unknown traffic toward the Internet edge.
- A **next-hop router** forwards traffic to the provider or upstream device.
- A **modem**, **ONT** (Optical Network Terminal), or provider handoff may terminate the circuit.
- A firewall often sits between the internal network and the Internet edge.

A simple edge topology:

```text
LAN -> Router/Firewall -> ISP handoff -> Internet
```

In troubleshooting, confirm:
- the edge interface is up/up,
- the default route exists,
- NAT is configured if private addresses leave the site,
- DNS works for name resolution,
- and the ISP handoff is operational.

**Wireless LAN fundamentals**

Wireless LANs (<abbr title="Wireless Local Area Networks">WLANs</abbr>) use access points (<abbr title="Access Point">AP</abbr>s) to bridge wireless clients into the wired network. The SSID identifies the wireless network name, while the security settings determine who can join.

Important wireless concepts:
- 2.4 GHz has more range but more interference and fewer non-overlapping channels.
- 5 GHz usually offers more channels and better performance.
- Channel planning reduces co-channel and adjacent-channel interference.
- Transmit power affects coverage overlap and roaming behavior.

Common AP modes:
- **Local mode**: normal client access.
- **FlexConnect**: useful for branch sites with local switching and controller connectivity.
- **Monitor mode**: dedicated to scanning and intrusion detection.
- **Sniffer mode**: captures wireless frames for analysis.

Controller-based wireless often uses a Wireless LAN Controller (<abbr title="Wireless LAN Controller">WLC</abbr>) to centralize SSIDs, security, and roaming behavior. APs join the controller, download configuration, and then advertise WLANs.

Example controller-style WLAN concepts:
- SSID maps to a VLAN or dynamic interface.
- Clients authenticate using PSK or 802.1X.
- The controller may anchor guest traffic or centralize policy.

A practical verification mindset:
- Can the AP join the controller?
- Does the SSID broadcast?
- Does the client get an IP address?
- Can the client reach the default gateway and DNS?
- Is the issue RF, authentication, or Layer 3?

**Wireless configuration and verification examples**

A basic AP or controller workflow usually includes SSID creation, security assignment, and VLAN mapping.

Example conceptual configuration:

```text
SSID: CORP-WIFI
Security: WPA2-Enterprise
VLAN: 30
Gateway: 192.168.30.1
```

On a Cisco wireless controller, you commonly verify:
- AP status,
- WLAN status,
- client association,
- and interface/VLAN mapping.

Useful verification commands vary by platform, but the general checks are:

```bash
show wlan summary
show ap summary
show client summary
show interface summary
```

If clients associate but cannot reach the network:
- check DHCP scope,
- verify VLAN tagging,
- confirm the controller or AP has a valid uplink,
- and test DNS and default gateway reachability.

**Wireless security and authentication**

Wireless security should protect both access and data in transit. CCNA-level security usually centers on encryption, authentication, and segmentation.

- **WPA2/WPA3** protect wireless traffic.
- **PSK** (pre-shared key) is simple but less scalable.
- **802.1X** uses a RADIUS server for per-user authentication.
- **Enterprise authentication** is better for organizations because it supports user identity and policy control.

802.1X flow:
1. The client connects to the AP.
2. The AP acts as an authenticator.
3. The client presents credentials through EAP (Extensible Authentication Protocol).
4. The AP forwards authentication to the RADIUS server.
5. The server approves or rejects access.

Common threats and mitigations:
- **Rogue APs**: use wireless monitoring and physical security.
- **Evil twin attacks**: validate SSIDs and certificate-based authentication.
- **Weak PSKs**: use strong passphrases and rotate them.
- **Open networks**: isolate guests and use captive portals where appropriate.

**AAA and device access control**

Authentication, Authorization, and Accounting (<abbr title="Authentication, Authorization, and Accounting">AAA</abbr>) controls who can log in, what they can do, and what actions are recorded. It is central to secure device administration and enterprise access.

- **Authentication** verifies identity.
- **Authorization** determines permitted actions.
- **Accounting** logs activity for auditing.

A common Cisco approach uses local fallback plus a RADIUS or TACACS+ server. TACACS+ is often used for device administration because it separates authentication, authorization, and accounting more cleanly than RADIUS.

Example local user and SSH hardening:

```bash
conf t
username admin privilege 15 secret Str0ngP@ssw0rd
ip domain-name example.com
crypto key generate rsa modulus 2048
ip ssh version 2

line vty 0 4
 login local
 transport input ssh
end
```

Verify with:

```bash
show ip ssh
show running-config | section line vty
show users
```

**Layer 2 and Layer 3 security controls**

Security on Cisco devices is often about reducing attack surface and limiting blast radius.

- Use **ACLs** to permit or deny traffic based on source, destination, protocol, and port.
- Use **port security** to limit MAC addresses on access ports.
- Use **DHCP snooping**, **Dynamic ARP Inspection** (<abbr title="Dynamic ARP Inspection">DAI</abbr>), and **IP Source Guard** to reduce spoofing attacks.
- Use **storm control** to limit broadcast, multicast, or unknown unicast storms.

Example ACL:

```bash
conf t
ip access-list extended WEB-ONLY
 permit tcp 192.168.10.0 0.0.0.255 any eq 80
 permit tcp 192.168.10.0 0.0.0.255 any eq 443
 deny ip any any log

interface g0/0
 ip access-group WEB-ONLY in
end
```

Example port security:

```bash
conf t
interface gi0/10
 switchport mode access
 switchport port-security
 switchport port-security maximum 2
 switchport port-security mac-address sticky
 switchport port-security violation restrict
end
```

**IP services: DHCP, DNS, and NTP**

DHCP automates address assignment, DNS resolves names to IP addresses, and Network Time Protocol (<abbr title="Network Time Protocol">NTP</abbr>) keeps device clocks synchronized.

DHCP relay is important when the DHCP server is in another subnet. The router or Layer 3 switch forwards client broadcasts using `ip helper-address`.

```bash
conf t
interface vlan 10
 ip address 192.168.10.1 255.255.255.0
 ip helper-address 192.168.100.10
end
```

Verify with:

```bash
show ip interface brief
show running-config interface vlan 10
show ip dhcp binding
```

If clients do not get addresses:
- confirm the helper address,
- check the DHCP server scope,
- verify VLAN membership,
- and ensure the client broadcast reaches the gateway interface.

**Automation and controller-based networking**

Automation reduces repetitive configuration and improves consistency. In CCNA terms, you should understand why controllers and APIs matter even if you do not build large automation workflows.

- A **controller** centralizes policy and pushes configuration to devices.
- An **API** (Application Programming Interface) lets software interact with network devices programmatically.
- **JSON** and **YAML** are common data formats for automation.
- **Ansible** is often used for agentless configuration management.

Practical example of a simple API-style payload:

```json
{
  "hostname": "sw1",
  "interfaces": [
    {"name": "Gi0/1", "description": "Uplink", "enabled": true}
  ]
}
```

Why this matters:
- You reduce manual errors.
- You can standardize VLANs, ACLs, and wireless policies.
- You can verify state at scale instead of checking devices one by one.

**Troubleshooting workflow**

A repeatable troubleshooting process is more valuable than memorizing isolated fixes. Start broad, then narrow the scope.

1. Identify the symptom.
2. Determine the affected scope.
3. Check the most likely layer.
4. Verify configuration and state.
5. Test with a known-good path.
6. Document the root cause and fix.

Useful commands:

```bash
show ip interface brief
show interfaces status
show vlan brief
show interfaces trunk
show ip route
show cdp neighbors
show lldp neighbors
ping
traceroute
```

Scenario examples:
- If two VLANs cannot communicate, check the trunk, subinterfaces, and default gateways.
- If a host cannot reach the Internet but can reach the gateway, check NAT, default route, and DNS.
- If wireless clients connect but fail authentication, check PSK, RADIUS, certificates, and WLAN-to-VLAN mapping.
- If OSPF neighbors do not form, compare area, timers, authentication, and router IDs.

**Common CCNA review questions to self-check**

Use these as a quick oral exam:
- Can you explain the difference between a switch, router, and Layer 3 switch?
- Can you configure and verify a VLAN, trunk, and inter-VLAN routing?
- Can you explain how STP prevents loops and how PortFast differs from BPDU Guard?
- Can you configure a static route, default route, and OSPF network statement?
- Can you explain NAT versus PAT and identify inside versus outside interfaces?
- Can you describe HSRP, VRRP, and GLBP at a high level?
- Can you explain how a WLAN client authenticates with WPA2-Enterprise and 802.1X?
- Can you secure device access with SSH, AAA, and ACLs?
- Can you troubleshoot a problem using a layered approach?

**High-value exam reminders**

- The most specific route wins because of longest prefix match.
- A trunk carries VLAN tags; an access port does not.
- STP blocks loops, not traffic by default.
- NAT/PAT is usually required when private addresses access the Internet.
- FHRP protects the default gateway, not the entire routing domain.
- 802.1X is identity-based access control, not just a wireless password.
- Controller-based wireless centralizes policy, but APs still need uplink, VLAN, and DHCP correctness.
- Troubleshooting is often about verifying assumptions one layer at a time.

**Final review mindset**

For CCNA exam prep, focus on how technologies interact:
- VLANs need trunks and gateways.
- Routing needs correct prefixes and next hops.
- Wireless needs RF, authentication, and IP services.
- Security needs layered controls, not a single feature.
- Automation and controllers change how you manage the network, but they do not remove the need to understand the underlying protocols.

If you can explain the purpose, configure the feature, and verify the result, you are thinking at the right level for the exam.
