## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**QoS overview**

<abbr title="Quality of Service">QoS</abbr> is the set of techniques used to manage how traffic is treated when links become congested. At CCNA level, you should understand that QoS does not create bandwidth; it helps you decide which traffic gets preferred treatment, which traffic waits, and which traffic may be dropped first.

The main reason to use QoS is to protect time-sensitive traffic such as voice and interactive video from delay, jitter, and loss. Best-effort traffic such as file transfers can usually tolerate slower delivery, but real-time applications often cannot.

- **Delay** is the time a packet takes to reach the destination.
- **Jitter** is variation in packet delay.
- **Loss** happens when packets are dropped, usually because a queue fills.

**Traffic classes and marking**

Cisco QoS commonly groups traffic into classes and then applies actions to each class. The most important marking concepts for CCNA are <abbr title="Differentiated Services Code Point">DSCP</abbr> in the IP header and <abbr title="Class of Service">CoS</abbr> in the Layer 2 header.

A practical way to think about the common service models is:

- **Best effort**: no special treatment; this is the default for most traffic.
- **Expedited forwarding**: low-latency treatment for voice-like traffic.
- **Assured forwarding**: traffic gets a defined forwarding preference and drop behavior.
- **Class selector**: legacy-style values that align with IP precedence and are often used to preserve priority across domains.

At the exam level, you should know that DSCP is the more useful end-to-end marking for routed networks because it stays in the IP header across hops. CoS is still important on Ethernet links and at Layer 2 trust boundaries, where switches and access devices may map or translate CoS to DSCP.

A common Cisco trust relationship is:

- IP phones mark voice traffic with DSCP.
- Switch ports trust the phone, not necessarily the attached PC.
- The switch may map incoming CoS values to DSCP values for consistent treatment in the network.

Example of a typical trust boundary on an access port with a phone:

```cisco
interface gigabitEthernet1/0/10
 switchport mode access
 switchport voice vlan 20
 mls qos trust cos
```

On many campus designs, the phone tags voice traffic while the PC traffic remains untagged or best effort. The switch can trust the phone’s markings and preserve them into the network.

**MQC configuration model**

Cisco commonly uses the <abbr title="Modular QoS Command-Line Interface">MQC</abbr> workflow. The standard pattern is:

1. Define traffic with a `class-map`.
2. Define actions with a `policy-map`.
3. Apply the policy with `service-policy`.

This model is important because it separates classification from behavior. You first identify traffic, then decide what to do with it.

Example:

```cisco
class-map match-any REALTIME
 match dscp ef
 match dscp cs3

policy-map WAN-POLICY
 class REALTIME
  priority 1000
 class class-default
  fair-queue

interface gigabitEthernet0/0
 service-policy output WAN-POLICY
```

In this example:

- `class-map match-any` means traffic matching either DSCP value enters the class.
- `priority 1000` creates a strict priority queue with a bandwidth limit of 1000 kbps for that class.
- Excess traffic beyond the configured priority allowance is policed, not simply queued forever.
- `class-default` handles all unmatched traffic.

> On many Cisco platforms, QoS behavior and verification output vary by hardware and software release. The concepts are consistent, but exact commands and counters may differ.

**Queuing, policing, and shaping**

These are the core QoS actions you should distinguish clearly.

- **Queuing** holds packets temporarily when an interface is busy.
- **Policing** enforces a rate by dropping or remarking traffic that exceeds a threshold.
- **Shaping** buffers excess traffic and sends it later at a controlled rate.

At CCNA level, the most important distinction is that shaping is generally an egress behavior, while policing can be used to enforce inbound or outbound rate limits depending on platform and policy design. Some platforms also support ingress queuing features, so avoid assuming ingress can only classify or drop.

A simple mental model:

- Use **queuing** when you want preferred traffic to leave first.
- Use **policing** when you want to cap traffic immediately.
- Use **shaping** when you want to smooth bursts instead of dropping them.

Example shaping on an interface:

```cisco
policy-map SHAPE-OUT
 class class-default
  shape average 2000000

interface gigabitEthernet0/1
 service-policy output SHAPE-OUT
```

This example limits average sending rate to 2 Mbps by buffering bursts and releasing them over time.

**Priority queuing and low-latency traffic**

Voice traffic is often placed into a strict priority class. Cisco commonly calls this <abbr title="Low Latency Queuing">LLQ</abbr> when a priority class is combined with other classes in a policy.

The key idea is that priority traffic is sent before other classes when congestion exists. However, it is not unlimited. The configured priority bandwidth is enforced so that voice does not starve the rest of the network.

Typical voice-oriented example:

```cisco
class-map match-any VOICE
 match dscp ef

policy-map CAMPUS-QOS
 class VOICE
  priority 2000
 class class-default
  fair-queue
```

Use this model when you need to protect latency-sensitive traffic, especially voice over IP. The exact bandwidth value should match the design and available link speed.

**Marking and trust boundaries**

Marking is the process of setting QoS values on packets or frames. Trust boundaries are the points where a device decides whether to accept or overwrite those markings.

Common campus behavior:

- IP phones mark voice traffic.
- Access switches trust the phone’s markings.
- PCs connected through the phone are usually treated as best effort.
- Routers and WAN edges may remark traffic to match policy.

A useful relationship to remember is:

- **CoS** is a Layer 2 marking used on Ethernet trunks and access segments.
- **DSCP** is a Layer 3 marking used across routed paths.
- Cisco devices often map CoS to DSCP at trust boundaries so that Layer 2 priority can be preserved as Layer 3 treatment.

Example of a common mapping idea:

| CoS | Typical use | Common DSCP mapping idea |
|---:|---|---|
| 0 | Best effort | Default/BE |
| 5 | Voice | EF |
| 3 | Signaling | CS3 |
| 1 or 2 | Scavenger or lower-priority data | Lower-priority AF/CS values |

The exact mapping can vary by design and platform, so treat this as a conceptual guide rather than a universal rule.

**QoS in wireless and campus networks**

QoS matters on Wi-Fi because the wireless medium is shared and contention-based. Voice and video can suffer quickly if the access point treats all traffic the same.

In a campus design:

- The wireless client may mark traffic.
- The access point may preserve or rewrite markings.
- The wired switch uplink should carry the markings consistently.
- Voice over Wi-Fi needs both RF quality and QoS treatment.

A practical example is a wireless voice handset using DSCP EF for voice packets. The AP can map that traffic into a higher-priority wireless access category so it competes more effectively on the air interface.

You do not need deep wireless QoS internals for CCNA, but you should know that QoS is not only a wired-network topic.

**QoS models and exam-relevant terminology**

At a high level, Cisco QoS is built around classification, marking, queuing, policing, shaping, and congestion management.

- **Classification** identifies traffic.
- **Marking** labels traffic for downstream devices.
- **Congestion management** decides which packets leave first.
- **Congestion avoidance** tries to prevent queues from filling completely.
- **Policing** enforces a rate.
- **Shaping** smooths bursts.

A concise exam-oriented summary:

- Best effort traffic gets no special treatment.
- Expedited forwarding is used for low-latency traffic.
- Assured forwarding provides differentiated treatment with drop precedence.
- Class selector values help preserve priority semantics across networks.

**Verification and troubleshooting**

You should verify both configuration and behavior. On IOS and IOS XE, `show policy-map interface` is a common command for checking applied policies and counters.

Example:

```cisco
show policy-map interface gigabitEthernet0/0
```

What to look for in the output:

- **Class matches** show how much traffic entered each class.
- **Offered rate** indicates how much traffic the class is receiving.
- **Drop counters** show whether packets are being discarded.
- **Police rate** and conform/exceed counters show policing behavior.
- **Queue depth** or queue drops indicate congestion in a queue.
- **Shape rate** or shaping statistics indicate buffered traffic being released at a controlled rate.

Example interpretation:

- If a priority class shows drops, the offered traffic may exceed the configured priority allowance.
- If the default class shows queue drops, the interface is congested and lower-priority traffic is being discarded.
- If policing counters increase, traffic is exceeding the configured rate and is being dropped or remarked.

Other useful checks include:

```cisco
show class-map
show policy-map
show interfaces gigabitEthernet0/0
show mls qos interface gigabitEthernet1/0/10
```

Not every platform supports every command. Catalyst switches, routers, and wireless devices may expose different QoS verification outputs.

**Practical design guidance**

Use QoS where congestion is possible and where application sensitivity justifies the complexity. If a link is never congested, QoS has little visible effect.

Good CCNA-level design habits:

- Mark traffic as close to the source as practical.
- Trust markings only at known boundaries.
- Give voice strict priority, but keep the priority queue bounded.
- Shape at WAN edges when you need to match provider rates.
- Police when you want to enforce a hard limit.
- Verify counters after applying policy so you know the policy is active and meaningful.

A simple campus-to-WAN flow might look like this:

1. The IP phone marks voice as EF.
2. The access switch trusts the phone and preserves the marking.
3. The distribution or WAN edge classifies voice and places it in priority treatment.
4. The WAN edge shapes traffic to the service-provider rate.
5. The router verifies queue and drop counters to confirm behavior.

**Common exam traps**

- QoS does not increase total bandwidth.
- Priority queuing is not the same as unlimited bandwidth.
- DSCP is not the same as CoS.
- Shaping is not the same as policing.
- QoS is not only for routers; switches and wireless devices also participate.
- A command that works on one Cisco platform may not be the exact verification command on another.

**Key takeaways**

QoS is about prioritizing traffic during congestion, not about making the network faster. For CCNA, focus on the relationship between traffic classes, markings, and actions. Know how Cisco MQC works, understand the difference between DSCP and CoS, and be able to explain why voice traffic often gets priority while best-effort traffic does not.
