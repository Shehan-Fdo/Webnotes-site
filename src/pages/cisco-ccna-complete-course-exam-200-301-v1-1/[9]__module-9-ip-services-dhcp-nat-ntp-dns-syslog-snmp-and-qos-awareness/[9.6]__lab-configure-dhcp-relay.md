## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**DHCP relay overview**

<abbr title="Dynamic Host Configuration Protocol">DHCP</abbr> clients use broadcasts to find a server on the local subnet. Broadcasts do not cross routers, so you configure a relay agent on the router or Layer 3 switch interface that faces the client network.

The relay agent receives the client broadcast, forwards it as a unicast to the DHCP server, and inserts relay information into the packet. On Cisco IOS, `ip helper-address` is the standard command used for this purpose.

A simple topology looks like this:

```text
PC (DHCP client) --- VLAN 10 --- R1/Gateway --- VLAN 20 --- DHCP Server
     192.168.10.0/24            192.168.10.1     192.168.20.1   192.168.20.100
```

The key idea is that the router is not acting as the DHCP server. It is only helping the client reach a server on another network.

**How DHCP relay works**

When a client starts DHCP, it sends a broadcast `DHCPDISCOVER` from UDP port 68 to UDP port 67. The relay agent hears that broadcast on the client-facing interface and forwards it toward the server.

Cisco IOS sets the <abbr title="Gateway IP Address">giaddr</abbr> field to the IP address of the relay interface that received the broadcast. That address tells the DHCP server which subnet the client came from, so the server can choose the correct address pool.

The relay agent may also add relay agent information, depending on platform and configuration. For CCNA-level understanding, focus on the `giaddr` behavior and the fact that the server uses it to identify the client subnet.

> The relay interface address is not the client's gateway information. It is the relay agent address placed in `giaddr` so the server knows where the request originated.

**Default helper behavior**

`ip helper-address` does more than DHCP relay. By default, Cisco IOS forwards a set of UDP broadcast services, commonly including:

- DHCP/BOOTP
- TFTP
- DNS
- Time
- NetBIOS name service
- NetBIOS datagram service
- TACACS

This matters because you may enable helper functionality for DHCP and unintentionally forward other broadcasts too. In most CCNA labs, DHCP is the main service you care about, but it helps to know that helper behavior is broader than DHCP alone.

You can adjust the forwarded UDP ports with `ip forward-protocol udp <port>`. For example, if you want to stop forwarding a specific service, you remove that protocol from the helper forwarding list.

**Basic configuration**

Configure the helper on the interface that receives the client broadcasts. That is usually the SVI on a multilayer switch or the router subinterface/VLAN interface serving the client subnet.

```bash
conf t
interface g0/0
 ip address 192.168.10.1 255.255.255.0
 ip helper-address 192.168.20.100
 no shutdown
end
```

If the DHCP server is on a different VLAN or subnet, the helper address should point to the server's reachable IP address.

**Lab walkthrough**

1. Identify the client subnet and the DHCP server subnet.
2. Configure the client-facing interface with an IP address and `ip helper-address`.
3. Ensure the DHCP server has a scope for the client subnet.
4. Verify routing between the relay device and the server.
5. Test client address assignment.
6. Confirm the relay is forwarding requests and the server is responding.

A practical example:

```text
Client subnet: 192.168.10.0/24
Relay interface: 192.168.10.1
DHCP server: 192.168.20.100
Server pool: 192.168.10.0/24 with default gateway 192.168.10.1
```

On the relay device:

```bash
conf t
interface vlan 10
 ip address 192.168.10.1 255.255.255.0
 ip helper-address 192.168.20.100
end
```

On the DHCP server, create a scope for `192.168.10.0/24` and set the default gateway option to `192.168.10.1`.

**Server-side requirements**

The DHCP server must know which subnet it is serving. If the relay forwards a request from `192.168.10.0/24`, the server needs a matching pool or scope for that network.

Typical server settings include:

- Network and mask for the client subnet
- Default gateway option, often called router option
- DNS server option if clients need name resolution
- Lease time appropriate for the environment

If the server is missing the correct scope, the relay will still forward packets, but clients will not receive usable addresses.

**Verification**

Use interface and configuration checks first. These are the most reliable CCNA-level verification methods.

```bash
show running-config interface g0/0
show ip interface g0/0
show ip interface brief
show running-config | include helper-address
```

What you want to confirm:

- The helper address is configured on the correct client-facing interface
- The interface is up/up
- The interface has the correct IP address and mask
- Routing exists between the relay and the DHCP server

If your platform supports debugging, you can inspect DHCP relay activity:

```bash
debug ip dhcp relay
debug ip dhcp server packet
```

Use debugging carefully on production devices because it can be noisy.

A successful client test usually looks like this:

```bash
PC> ipconfig /release
PC> ipconfig /renew
```

Expected result:

- The client receives an IP address from the correct subnet
- The default gateway matches the relay interface IP
- The client can reach the DHCP server and other local resources

**Troubleshooting checklist**

If DHCP relay fails, work through the path in order.

- Confirm the client is actually sending DHCP broadcasts.
- Confirm `ip helper-address` is on the correct interface.
- Confirm the relay interface is in the client subnet.
- Confirm the DHCP server has a matching scope for that subnet.
- Confirm routing exists between relay and server.
- Confirm no ACL or firewall blocks the relay-to-server traffic.
- Confirm the server is reachable by unicast from the relay device.

A common mistake is placing `ip helper-address` on the wrong interface. The command belongs on the interface that receives the client broadcast, not on the server-facing interface.

Another common issue is misunderstanding the traffic flow. The client broadcast uses UDP 68 to 67 on the local subnet, while the relay forwards the request to the server as unicast traffic. Troubleshooting should consider both the local broadcast phase and the relay-to-server unicast phase.

> Do not assume the relay is the default gateway for the client unless the topology is designed that way. The relay only forwards DHCP messages; the gateway is a separate Layer 3 function.

**Practical packet flow**

The relay process is easier to remember if you follow the packet path:

```text
1. Client broadcasts DHCPDISCOVER on local subnet
2. Relay interface receives the broadcast
3. Relay sets giaddr to its own interface IP
4. Relay unicasts the request to the DHCP server
5. Server selects a pool based on giaddr
6. Server sends DHCPOFFER back to the relay
7. Relay forwards the offer to the client
8. Client completes DHCPREQUEST and DHCPACK exchange
```

This flow explains why the server must have a scope for the client subnet and why the relay interface IP matters.

**Useful IOS commands**

```bash
conf t
interface vlan 10
 ip helper-address 192.168.20.100
 no ip helper-address 192.168.20.100
```

You can also tune forwarded UDP services with `ip forward-protocol`:

```bash
conf t
no ip forward-protocol udp 69
no ip forward-protocol udp 53
end
```

Use this only when you understand the impact, because it changes which broadcast services the helper forwards.

**Exam focus**

For CCNA, remember these points:

- DHCP broadcasts do not cross routers by default.
- `ip helper-address` enables DHCP relay.
- Configure the helper on the client-facing interface.
- The relay sets `giaddr` to its interface address.
- The DHCP server needs a scope for the client subnet.
- Verification starts with interface and configuration checks.
