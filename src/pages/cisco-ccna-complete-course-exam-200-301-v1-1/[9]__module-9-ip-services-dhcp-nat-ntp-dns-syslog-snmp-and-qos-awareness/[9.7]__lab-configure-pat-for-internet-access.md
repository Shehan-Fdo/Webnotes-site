## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**PAT overview**

Port Address Translation (PAT), also called **NAT overload**, lets many inside hosts share one public IPv4 address by using unique Layer 4 port numbers. You use PAT when a small or medium network needs Internet access but has only one routable address on the edge.

PAT is the most common form of Network Address Translation (NAT) in enterprise edge labs because it conserves public IPv4 space and is simple to deploy. The router tracks each translation using the inside local address, inside global address, protocol, and port number.

- **Inside local**: the private address assigned to a host on the internal network.
- **Inside global**: the public address that represents the inside host to the outside network.
- **Overload**: multiple translations share one outside-facing address by using different ports.

**How PAT works**

When an inside host opens a connection to the Internet, the router rewrites the source IP address to the configured public address and assigns a unique source port if needed. Return traffic is matched against the translation table and sent back to the correct inside host.

A typical lab uses:
- an inside LAN interface
- an outside Internet-facing interface
- a default route toward the ISP or upstream router
- an ACL that matches the inside private subnet(s)

> PAT does not create connectivity by itself. You still need correct routing, interface roles, and a valid upstream path.

**Common lab topology**

A simple CCNA-style topology often looks like this:

- `PC1` and `PC2` on `192.168.10.0/24`
- Router LAN interface on `192.168.10.1/24`
- Router WAN interface on a public or simulated ISP network
- Upstream router or cloud device acting as the Internet

Example addressing:

```text
PC1: 192.168.10.10/24  GW 192.168.10.1
PC2: 192.168.10.11/24  GW 192.168.10.1
R1 G0/0: 192.168.10.1/24   inside
R1 G0/1: 203.0.113.2/30    outside
ISP:     203.0.113.1/30
```

**Core configuration steps**

You normally configure PAT in this order:

1. Mark the inside and outside interfaces.
2. Define an ACL that matches the inside source addresses.
3. Create the NAT overload rule using the outside interface or a specific public IP.
4. Ensure routing is in place so return traffic can reach the router.

Example configuration using the outside interface address:

```cisco
conf t
interface g0/0
 ip address 192.168.10.1 255.255.255.0
 ip nat inside
 no shutdown

interface g0/1
 ip address 203.0.113.2 255.255.255.252
 ip nat outside
 no shutdown

access-list 1 permit 192.168.10.0 0.0.0.255

ip nat inside source list 1 interface g0/1 overload
ip route 0.0.0.0 0.0.0.0 203.0.113.1
end
```

This configuration tells the router to:
- translate all permitted inside addresses
- use the `g0/1` interface address as the inside global address
- overload the address so multiple sessions can share it

**Using a pool instead of an interface**

If the lab provides a block of public addresses, you can build PAT from a NAT pool. This is useful when you want to control which public address is used or when the interface address is not the desired translation address.

```cisco
conf t
ip nat pool PUBLIC-POOL 203.0.113.10 203.0.113.10 netmask 255.255.255.252
access-list 1 permit 192.168.10.0 0.0.0.255
ip nat inside source list 1 pool PUBLIC-POOL overload
end
```

Use a pool when:
- the outside interface is not the translation address
- you want a specific public IP for policy or testing
- the lab requires a reusable NAT object

**Verification commands**

After configuration, verify both the NAT state and the routing path.

```cisco
show ip interface brief
show ip route
show access-lists
show ip nat translations
show ip nat statistics
```

What to look for:
- `show ip nat translations` should display active mappings after traffic is generated.
- `show ip nat statistics` should show hits, misses, and the configured inside/outside interfaces.
- `show access-lists` should show match counters increasing when inside traffic is translated.

Example translation output:

```text
Pro  Inside global      Inside local       Outside local      Outside global
tcp  203.0.113.2:1025   192.168.10.10:1025 198.51.100.10:80   198.51.100.10:80
tcp  203.0.113.2:1026   192.168.10.11:1026 198.51.100.10:80   198.51.100.10:80
```

**Testing Internet access**

Use simple reachability tests from the inside hosts and from the router.

```bash
ping 192.168.10.1
ping 203.0.113.1
ping 8.8.8.8
traceroute 8.8.8.8
```

On Cisco IOS, you can also test from the router with a source interface or source address:

```cisco
ping 8.8.8.8 source g0/0
traceroute 8.8.8.8 source g0/0
```

If inside hosts can reach the router but not the Internet, check:
- default route on the edge router
- upstream routing back to the NAT outside address
- ACL match conditions
- correct inside/outside interface assignment

**Troubleshooting PAT**

Most PAT failures come from a small set of mistakes.

- **Wrong interface roles**: `ip nat inside` and `ip nat outside` are reversed or missing.
- **ACL mismatch**: the ACL does not match the inside subnet, so no translations are created.
- **Missing route**: the router has no default route toward the ISP.
- **Upstream return path issue**: the ISP side cannot route back to the NAT outside address.
- **No traffic generated**: NAT tables stay empty until a session starts.
- **Overlapping subnets**: inside and outside networks should not overlap in a normal lab.

Useful cleanup commands:

```cisco
clear ip nat translation *
clear ip nat statistics
```

If you change NAT rules during a lab, clear existing translations so stale entries do not hide the new behavior.

**PAT versus static NAT**

PAT is best for many-to-one Internet access. Static NAT is best when you need a fixed one-to-one mapping for a server or published service.

| Feature | PAT | Static NAT |
|---|---|---|
| Address usage | Many inside hosts share one public IP | One inside host maps to one public IP |
| Port usage | Uses ports to distinguish sessions | Does not rely on port multiplexing for basic mapping |
| Typical use | Client Internet access | Public services, inbound access |
| IPv4 conservation | High | Low |

For CCNA labs, PAT is the default choice for outbound user traffic because it scales better and matches common edge designs.

**Practical lab checklist**

Before you declare the lab complete, confirm each item:

1. Inside hosts use the router as their default gateway.
2. The router has `ip nat inside` on the LAN interface.
3. The router has `ip nat outside` on the WAN interface.
4. The ACL matches the inside subnet.
5. The PAT rule uses `overload`.
6. The router has a default route to the upstream device.
7. You can generate traffic and see entries in `show ip nat translations`.

**Key exam takeaways**

PAT is a translation method, not a routing replacement. You still need correct Layer 3 reachability on both sides of the edge router. In exam scenarios, the fastest path to success is to verify interface roles, ACL coverage, and the default route before you troubleshoot the translation table.
