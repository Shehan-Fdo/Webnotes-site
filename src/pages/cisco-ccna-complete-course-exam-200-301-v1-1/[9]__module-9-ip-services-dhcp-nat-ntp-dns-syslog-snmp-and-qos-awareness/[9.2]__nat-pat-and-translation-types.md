## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**NAT fundamentals**

<abbr title="Network Address Translation">NAT</abbr> translates IP addresses as traffic crosses a routing boundary, usually between a private internal network and a public external network. You use NAT to conserve public IPv4 addresses, hide internal addressing, and allow private hosts to reach the Internet.

NAT is not a routing protocol and does not replace routing. It rewrites packet headers, then forwards traffic based on the routing table. In practice, you configure NAT on a border router or firewall where inside and outside interfaces meet.

- **Inside local**: the private address assigned to a host on the internal network.
- **Inside global**: the public address that represents an inside host to the outside world.
- **Outside global**: the real address of an external host on the Internet.
- **Outside local**: the address of an outside host as seen from the inside network; this is often the same as the outside global address.

A simple example helps:

```text
Inside host: 192.168.1.10
NAT device public address: 203.0.113.5
Internet server: 198.51.100.20
```

When `192.168.1.10` reaches the Internet, the NAT device may translate it to `203.0.113.5`. Return traffic is mapped back to the original inside host using the translation table.

**Why NAT is used**

NAT remains important in IPv4 networks because public IPv4 space is limited. It also provides a degree of address hiding, though it is not a security feature by itself. You still need ACLs, firewalls, and proper segmentation for security.

Common use cases include:

- Internet access for private hosts using RFC 1918 addresses.
- Merging networks that use overlapping address space.
- Publishing internal services to external users.
- Temporary address migration during network redesigns.

> NAT can break end-to-end connectivity assumptions. Applications that embed IP addresses in payloads, use peer-to-peer connections, or rely on inbound reachability may need special handling.

**Translation types**

NAT behavior depends on how addresses are mapped. CCNA commonly focuses on three main translation types.

| Type | Mapping | Typical use | Key characteristic |
|---|---|---|---|
| Static NAT | One inside local to one inside global | Hosting internal servers externally | Permanent, fixed mapping |
| Dynamic NAT | One inside local to one available inside global from a pool | Outbound access for a limited number of hosts | Temporary, pool-based mapping |
| PAT / NAT overload | Many inside local addresses to one or a few inside global addresses using ports | Most common Internet access design | Uses Layer 4 port numbers to multiplex sessions |

**Static NAT**

Static NAT creates a fixed one-to-one mapping between an inside local address and an inside global address. You use it when an internal device must be reachable from the outside at a predictable public address, such as a web server or mail server.

Example:

```bash
ip nat inside source static 192.168.1.10 203.0.113.10
```

This means traffic from `192.168.1.10` is always translated to `203.0.113.10`. Return traffic to `203.0.113.10` is always sent back to `192.168.1.10`.

Use static NAT when:

- You need inbound access to a specific internal host.
- You want a stable public mapping for a service.
- You are publishing a server that must keep the same external address.

Limitations:

- It consumes one public address per mapped host.
- It does not scale well for many users.
- It is not ideal for general client Internet access.

**Dynamic NAT**

Dynamic NAT maps an inside local address to an available address from a configured pool of public addresses. The mapping is temporary and exists only while the session is active.

Example pool configuration:

```bash
ip nat pool PUBLIC_POOL 203.0.113.20 203.0.113.29 netmask 255.255.255.240
access-list 1 permit 192.168.1.0 0.0.0.255
ip nat inside source list 1 pool PUBLIC_POOL
```

In this example, hosts in `192.168.1.0/24` can be translated to one of the addresses in `PUBLIC_POOL`.

Use dynamic NAT when:

- You have more inside hosts than public addresses, but not enough traffic to require PAT.
- You want a pool of public addresses assigned on demand.
- You need a simple many-to-many translation model without port multiplexing.

Limitations:

- If the pool is exhausted, new sessions fail until an address is released.
- It still requires multiple public IP addresses.
- It is less common than PAT in modern edge designs.

**PAT and NAT overload**

<abbr title="Port Address Translation">PAT</abbr>, also called NAT overload, allows many inside hosts to share one public IP address by using unique source port numbers. This is the most common NAT type for Internet access.

PAT translates:

- Source IP address
- Source port number

The NAT device tracks each session in a translation table so return traffic can be delivered to the correct internal host.

Example:

```bash
access-list 1 permit 192.168.1.0 0.0.0.255
ip nat inside source list 1 interface GigabitEthernet0/0 overload
```

If `GigabitEthernet0/0` has public address `203.0.113.5`, multiple inside hosts can share that single address. The NAT device differentiates sessions using port numbers such as `203.0.113.5:1025`, `203.0.113.5:1026`, and so on.

Use PAT when:

- You want to conserve public IPv4 addresses.
- You need scalable outbound Internet access for many clients.
- You are building the standard small-to-medium enterprise edge.

PAT is usually the default answer for client Internet access because it is efficient and simple.

**Inside and outside roles**

You must mark interfaces correctly for NAT to work.

```bash
interface GigabitEthernet0/1
 ip address 192.168.1.1 255.255.255.0
 ip nat inside

interface GigabitEthernet0/0
 ip address 203.0.113.5 255.255.255.248
 ip nat outside
```

- `ip nat inside` identifies the internal-facing interface.
- `ip nat outside` identifies the external-facing interface.

Traffic direction matters:

- Inside to outside traffic is eligible for translation.
- Outside to inside traffic is only allowed if a translation exists or a static mapping is configured.

**Typical NAT configuration workflow**

1. Identify which interfaces face the internal and external networks.
2. Define which inside addresses should be translated, usually with an ACL.
3. Configure the translation method: static, dynamic pool, or PAT overload.
4. Verify routing so the NAT device can reach both sides.
5. Test with traffic and inspect the translation table.

Example end-to-end PAT configuration:

```bash
access-list 1 permit 192.168.1.0 0.0.0.255

interface GigabitEthernet0/1
 ip address 192.168.1.1 255.255.255.0
 ip nat inside

interface GigabitEthernet0/0
 ip address 203.0.113.5 255.255.255.248
 ip nat outside

ip nat inside source list 1 interface GigabitEthernet0/0 overload
```

**Verification commands**

Use these commands to confirm NAT behavior:

```bash
show ip nat translations
show ip nat statistics
show access-lists
show ip interface brief
```

What to look for:

- `show ip nat translations` displays active mappings.
- `show ip nat statistics` shows hits, misses, and pool usage.
- `show access-lists` confirms whether the NAT ACL matches traffic.
- `show ip interface brief` helps verify interface status and addressing.

A sample translation table may look like this:

```text
Pro  Inside global      Inside local       Outside local      Outside global
tcp  203.0.113.5:1025   192.168.1.10:1025  198.51.100.20:80   198.51.100.20:80
tcp  203.0.113.5:1026   192.168.1.11:1026  198.51.100.20:80   198.51.100.20:80
```

**How to interpret translation terms**

The four address perspectives are a common exam topic.

- **Inside local** is the address you configure on the internal host.
- **Inside global** is the translated address seen on the outside.
- **Outside global** is the real external address.
- **Outside local** is the external address as it appears to the inside network, which is often unchanged.

For most CCNA scenarios:

- Inside local and inside global differ when NAT is active.
- Outside local and outside global are usually the same.
- PAT adds port numbers to distinguish sessions.

**Static NAT vs dynamic NAT vs PAT**

| Feature | Static NAT | Dynamic NAT | PAT |
|---|---|---|---|
| Mapping | Fixed one-to-one | Temporary one-to-one from a pool | Many-to-one using ports |
| Public IP usage | One per host | One per active host | One or few for many hosts |
| Inbound access | Yes | Not predictable | Not by default |
| Scalability | Low | Medium | High |
| Common use | Servers | Limited client access | Internet access for users |

**Practical examples**

Static NAT for a web server:

```bash
ip nat inside source static 192.168.1.100 203.0.113.100
```

Dynamic NAT with a pool:

```bash
ip nat pool WEB_POOL 203.0.113.20 203.0.113.29 netmask 255.255.255.240
access-list 10 permit 192.168.1.0 0.0.0.255
ip nat inside source list 10 pool WEB_POOL
```

PAT using the outside interface address:

```bash
access-list 10 permit 192.168.1.0 0.0.0.255
ip nat inside source list 10 interface GigabitEthernet0/0 overload
```

**Troubleshooting NAT**

If NAT does not work, check the problem in layers:

- Verify the ACL matches the intended inside source addresses.
- Confirm the inside and outside interfaces are correctly marked.
- Ensure routing exists to the next hop on both sides.
- Check that the public address is reachable and not duplicated.
- Confirm the NAT table contains entries after generating traffic.

Common failure patterns:

- The ACL is too narrow and misses some hosts.
- The wrong interface is marked `ip nat inside` or `ip nat outside`.
- The router has no default route toward the Internet.
- The pool is exhausted in dynamic NAT.
- The return path bypasses the NAT device, breaking session symmetry.

> If you see outbound traffic leaving but no replies returning, suspect routing or interface role issues before assuming NAT is broken.

**NAT and security considerations**

NAT can make internal addressing less visible, but it does not inspect content or block attacks by itself. Do not treat NAT as a firewall. Use it as an address translation mechanism, then apply security controls separately.

Important points:

- NAT may complicate logging because multiple users can share one public IP.
- PAT can make per-user attribution harder without additional logs.
- Some protocols and applications need NAT traversal or application-layer gateways.
- IPv6 reduces the need for NAT, but NAT concepts still matter for IPv4 exam scenarios.

**Exam focus summary**

For CCNA, make sure you can:

- Define inside local, inside global, outside local, and outside global.
- Distinguish static NAT, dynamic NAT, and PAT.
- Recognize that PAT uses ports to multiplex many sessions.
- Configure `ip nat inside`, `ip nat outside`, and a NAT rule.
- Verify translations with `show ip nat translations`.
- Explain when to use static NAT for servers and PAT for users.

**Quick memory aid**

- **Static NAT**: one host, one public IP, always the same.
- **Dynamic NAT**: one host, one public IP from a pool, temporary.
- **PAT**: many hosts, one public IP, differentiated by ports.
