## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**DHCP on a router: what you are building**

Dynamic Host Configuration Protocol (DHCP) automatically assigns IP settings to clients so you do not configure each host manually. In a small branch, lab, or home-office network, a Cisco router often acts as the DHCP server for one or more LANs.

You use router-based DHCP when:
- You want a simple, built-in address service without a dedicated server.
- You need to support a small number of subnets.
- You want to practice core CCNA services such as addressing, default gateways, and verification.

A router DHCP service typically provides:
- IP address
- Subnet mask
- Default gateway
- Domain name
- DNS server addresses
- Lease time

**DHCP process overview**

DHCP uses the DORA sequence:
- **Discover**: the client broadcasts to find a DHCP server.
- **Offer**: the server proposes an address and options.
- **Request**: the client requests the offered address.
- **Acknowledge**: the server confirms the lease.

Because the initial messages are broadcasts, DHCP works best when the server is on the same broadcast domain as the client. If the server is on another subnet, you usually configure a DHCP relay agent with `ip helper-address`.

**Basic router DHCP configuration**

Before you create a pool, make sure the router interface and addressing plan are correct. The router must have a reachable interface address that will become the clients’ default gateway.

A typical configuration includes:
- Excluding reserved addresses
- Defining a DHCP pool
- Setting the network and mask
- Setting the default router
- Optionally setting DNS and domain name

Example for a LAN `192.168.10.0/24` with gateway `192.168.10.1`:

```cisco
conf t
!
interface g0/0
 ip address 192.168.10.1 255.255.255.0
 no shutdown
!
ip dhcp excluded-address 192.168.10.1 192.168.10.20
!
ip dhcp pool LAN-10
 network 192.168.10.0 255.255.255.0
 default-router 192.168.10.1
 dns-server 8.8.8.8 1.1.1.1
 domain-name example.local
 lease 7
end
```

**Why exclusions matter**

The excluded range reserves addresses for infrastructure devices such as:
- Router interfaces
- Switch management IPs
- Printers
- Servers
- Static host assignments

If you do not exclude these addresses, DHCP may hand them out to clients and create duplicate IP conflicts.

A common practice is to reserve the first usable addresses for network devices and let DHCP allocate from a higher range. For example:
- `192.168.10.1` router gateway
- `192.168.10.2` to `192.168.10.20` reserved
- `192.168.10.21` and above available for clients

**DHCP pool fields and what they do**

The `ip dhcp pool` command creates a named scope. Inside the pool, you define the subnet and options.

Key commands:
- `network <network> <mask>`: identifies the subnet served by the pool
- `default-router <ip>`: supplies the client default gateway
- `dns-server <ip>`: supplies DNS server addresses
- `domain-name <name>`: supplies the DNS suffix
- `lease <days [hours [minutes]]>`: sets lease duration

Example with a shorter lease for a lab:

```cisco
ip dhcp pool STUDENT-LAN
 network 10.10.10.0 255.255.255.0
 default-router 10.10.10.1
 dns-server 10.10.10.53
 lease 1 0 0
```

A shorter lease is useful in labs or guest networks where clients change often. A longer lease is better for stable office networks because it reduces renewal traffic.

**Multiple DHCP pools**

A router can serve multiple subnets. Each pool must match a different network statement.

Example:

```cisco
ip dhcp excluded-address 10.10.10.1 10.10.10.20
ip dhcp excluded-address 10.10.20.1 10.10.20.20

ip dhcp pool VLAN10
 network 10.10.10.0 255.255.255.0
 default-router 10.10.10.1

ip dhcp pool VLAN20
 network 10.10.20.0 255.255.255.0
 default-router 10.10.20.1
```

This design is common when a router or Layer 3 device routes between VLANs and also provides DHCP for each VLAN.

**DHCP relay with `ip helper-address`**

If clients are on a different subnet from the DHCP server, the router interface facing the client subnet can forward DHCP broadcasts to the server using `ip helper-address`.

Example:

```cisco
interface g0/1
 ip address 10.20.20.1 255.255.255.0
 ip helper-address 10.1.1.10
```

Use relay when:
- The DHCP server is centralized on another network.
- You have multiple VLANs but only one DHCP server.
- You want to avoid running DHCP on every router.

On Cisco devices, `ip helper-address` forwards several UDP services by default, not only DHCP. In CCNA labs, the most important use is forwarding DHCP requests to a remote server.

**Verification commands**

After configuration, verify both the service and the client lease process.

Useful commands:
- `show ip dhcp pool`
- `show ip dhcp binding`
- `show running-config | section dhcp`
- `show ip interface brief`
- `show ip dhcp conflict`

Example output checks:
- Pool utilization shows how many addresses are in use.
- Bindings show which MAC addresses received which IPs.
- Conflicts reveal duplicate address detection issues.

```cisco
show ip dhcp pool
show ip dhcp binding
show ip dhcp conflict
show running-config | section dhcp
```

If a client does not receive an address, confirm:
- The router interface is up/up.
- The client is in the correct VLAN or subnet.
- The pool network matches the interface subnet.
- Excluded addresses do not consume the entire range.
- No other DHCP server is answering first.

**Troubleshooting workflow**

When DHCP fails, isolate the problem in layers:
1. Verify Layer 1 and Layer 2 connectivity.
2. Verify the router interface IP address and status.
3. Verify the DHCP pool network and exclusions.
4. Verify the client is requesting on the correct broadcast domain.
5. Verify relay configuration if the server is remote.

Common mistakes:
- Wrong subnet mask in the pool
- Missing `default-router`
- Overlapping excluded range
- Pool network not matching the interface subnet
- Forgetting `no shutdown` on the gateway interface
- Misconfigured VLAN or trunk preventing client broadcasts from reaching the router

> If the router already has a static route or interface address conflict, DHCP may appear to work but clients still fail to reach the gateway or DNS. Always verify end-to-end connectivity, not just lease assignment.

**Practical lab topology**

A simple lab often looks like this:
- Router `R1` connected to switch `S1`
- PCs on the switch in one LAN or multiple VLANs
- Router interface `G0/0` as the default gateway
- Router DHCP pool serving the LAN subnet

Example addressing plan:
- `R1 G0/0`: `192.168.10.1/24`
- `PC1`: DHCP
- `PC2`: DHCP
- Reserved range: `192.168.10.1-192.168.10.20`

If you extend the lab to multiple VLANs, each VLAN needs:
- A Layer 3 gateway interface or subinterface
- A matching DHCP pool
- Correct trunking and VLAN membership

**Sample end-to-end configuration**

```cisco
conf t
hostname R1
!
interface g0/0
 description LAN Gateway
 ip address 192.168.10.1 255.255.255.0
 no shutdown
!
ip dhcp excluded-address 192.168.10.1 192.168.10.20
!
ip dhcp pool LAN10
 network 192.168.10.0 255.255.255.0
 default-router 192.168.10.1
 dns-server 8.8.8.8
 domain-name lab.local
 lease 3
end
write memory
```

This configuration gives clients:
- An address from `192.168.10.21` upward
- Gateway `192.168.10.1`
- DNS `8.8.8.8`
- Domain suffix `lab.local`

**Client-side validation**

After the router is configured, renew the client lease and confirm the result.

On a Windows client:
```powershell
ipconfig /release
ipconfig /renew
ipconfig /all
```

On a Linux client:
```bash
sudo dhclient -r
sudo dhclient
ip addr show
resolvectl status
```

You should confirm:
- Correct IP address and mask
- Correct default gateway
- Correct DNS server
- Lease obtained from the expected router

**Best practices for CCNA labs**

- Reserve infrastructure addresses before enabling DHCP.
- Keep pool names descriptive, such as `LAN10` or `VLAN20`.
- Match the pool network exactly to the subnet mask.
- Verify the gateway interface first, then the DHCP service.
- Use `show ip dhcp binding` to confirm real client leases.
- Document static reservations and excluded ranges to avoid overlap.

**Key takeaways**

Router-based DHCP is a simple, reliable way to automate host addressing in small networks and labs. The essential workflow is:
- Configure the gateway interface.
- Exclude reserved addresses.
- Create a DHCP pool with the correct network and options.
- Verify bindings and troubleshoot systematically.

If clients do not receive addresses, the issue is usually one of three things:
- The pool does not match the subnet.
- The gateway interface is down or misaddressed.
- The client broadcast cannot reach the DHCP server, so relay is required.
