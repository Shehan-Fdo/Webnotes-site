## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Lab purpose and scope**

This lab focuses on verifying two common Layer 3 services on Cisco devices: <abbr title="Network Address Translation">NAT</abbr> translations and <abbr title="Dynamic Host Configuration Protocol">DHCP</abbr> leases. In practice, you use these checks to confirm that hosts are receiving addresses correctly and that traffic is being translated as expected when it crosses a boundary such as a router or firewall.

The key idea is not just to memorize commands, but to understand what “healthy” output looks like. You should be able to confirm:
- A client received an IP address, default gateway, DNS server, and lease time from DHCP.
- A router is creating NAT entries for inside-to-outside traffic.
- The translation table matches the traffic pattern you expect.
- Troubleshooting starts with the right verification command before you change configuration.

**Core concepts**

<dl>
<dt><strong>NAT translation</strong></dt>
<dd>A mapping between an inside local address and an inside global address, often created dynamically when traffic leaves the inside network.</dd>
<dt><strong>DHCP lease</strong></dt>
<dd>The time-limited assignment of an IP address and related parameters to a client by a DHCP server.</dd>
<dt><strong>Inside local</strong></dt>
<dd>The private address assigned to a host on the internal network.</dd>
<dt><strong>Inside global</strong></dt>
<dd>The public or externally reachable address used to represent the inside host on the outside network.</dd>
</dl>

**Verifying NAT translations**

On Cisco IOS, the most common verification command is:

```bash
show ip nat translations
```

This command displays the active translation table. A typical output includes protocol, inside local, inside global, outside local, and outside global addresses. For example:

```bash
Router# show ip nat translations
Pro  Inside global      Inside local       Outside local      Outside global
tcp  203.0.113.10:1025  192.168.1.10:1025  198.51.100.20:80   198.51.100.20:80
udp  203.0.113.10:5000  192.168.1.10:5000  8.8.8.8:53         8.8.8.8:53
```

What to look for:
- The inside local address should match the client on your LAN.
- The inside global address should match the NAT pool, interface address, or overload address configured on the router.
- The protocol and port numbers help you distinguish multiple sessions using the same public IP.
- The table may be empty if no traffic has crossed the NAT boundary yet.

If you need more detail, use:

```bash
show ip nat statistics
```

This command helps you confirm:
- How many active translations exist.
- Whether NAT is using dynamic or static mappings.
- Hit counts and pool usage, depending on the configuration.

You can also clear entries during a lab to force fresh translations:

```bash
clear ip nat translation *
```

> Be careful when clearing translations on a shared device. Doing so can interrupt active sessions.

**Practical NAT verification workflow**

1. Generate traffic from an inside host to an outside destination, such as `ping`, `traceroute`, or `curl` if available.
2. Check the translation table with `show ip nat translations`.
3. Confirm the inside local and inside global addresses match the intended NAT design.
4. If no entry appears, verify interface roles, ACLs, and NAT configuration.
5. Use `show ip nat statistics` to confirm the router is processing translations.

**Example NAT configuration for context**

```bash
interface g0/0
 ip address 192.168.1.1 255.255.255.0
 ip nat inside

interface g0/1
 ip address 203.0.113.2 255.255.255.252
 ip nat outside

access-list 1 permit 192.168.1.0 0.0.0.255

ip nat inside source list 1 interface g0/1 overload
```

This configuration creates PAT, also called <abbr title="Port Address Translation">PAT</abbr>, which allows many inside hosts to share one outside address using different source ports.

**Verifying DHCP leases**

DHCP verification depends on whether you are checking the client side or the server/router side. On a Cisco router acting as a DHCP server, the most useful command is:

```bash
show ip dhcp binding
```

This displays the active leases the server has assigned. Typical fields include:
- Client IP address
- Client hardware address
- Lease expiration
- Type of binding

Example:

```bash
Router# show ip dhcp binding
IP address       Client-ID/Hardware address      Lease expiration        Type
192.168.1.10     0100.1c7f.8a11.22               Apr 26 2026 12:30 PM    Automatic
192.168.1.11     0100.1d2a.9b33.44               Apr 26 2026 12:45 PM    Automatic
```

Useful checks:
- The assigned address should come from the correct DHCP pool.
- The lease should not conflict with excluded addresses.
- The client hardware address should match the device you expect.
- The lease expiration should be reasonable for the lab design.

To inspect the DHCP pool itself, use:

```bash
show ip dhcp pool
```

This helps you verify:
- Pool name
- Network statement
- Number of leased addresses
- Available addresses
- Default router and DNS settings

Example:

```bash
Router# show ip dhcp pool
Pool LAB-POOL :
 Utilization mark (high/low)    : 100 / 0
 Subnet size (first/next)       : 254 / 0
 Total addresses                : 254
 Leased addresses               : 2
 Excluded addresses             : 10
 Pending event                  : none
 1 subnet is currently in the pool :
 Current index        IP address range                    Leased/Excluded/Total
 192.168.1.1          192.168.1.1     - 192.168.1.254     2/10/254
```

If the router is also acting as a DHCP client on an interface, you can verify the learned address with:

```bash
show ip interface brief
show running-config interface g0/0
```

On an end host, the exact command depends on the operating system:
- Windows: `ipconfig /all`
- Linux: `ip addr show`, `nmcli dev show`, or `resolvectl status`
- macOS: `ipconfig getpacket en0`

**Client-side DHCP checks**

A client should confirm:
- IP address
- Subnet mask
- Default gateway
- DNS server
- Lease obtained and lease expires times

Windows example:

```bash
ipconfig /all
```

Look for:
- `DHCP Enabled`
- `IPv4 Address`
- `Subnet Mask`
- `Default Gateway`
- `DHCP Server`
- `Lease Obtained`
- `Lease Expires`

Linux example:

```bash
ip addr show
nmcli dev show eth0
```

If the client does not receive a lease, you may need to renew it:
- Windows: `ipconfig /release` then `ipconfig /renew`
- Linux with NetworkManager: `nmcli dev disconnect eth0` then reconnect, or restart the connection
- Linux with dhclient: `sudo dhclient -r eth0` then `sudo dhclient eth0`

**Troubleshooting patterns**

When NAT verification fails, check these common causes:
- The inside interface is not marked with `ip nat inside`.
- The outside interface is not marked with `ip nat outside`.
- The ACL used by NAT does not match the inside subnet.
- No traffic has been generated yet, so no dynamic translation exists.
- The route to the outside network is missing.
- The return path is broken, so sessions never complete.

When DHCP verification fails, check these common causes:
- The DHCP pool network does not match the interface subnet.
- The default gateway in the pool is wrong.
- Excluded addresses overlap the pool range.
- Another DHCP server is answering first.
- The client and server are separated by a router without DHCP relay.
- The interface is down or in the wrong VLAN.

> If a client is on a different subnet from the DHCP server, you usually need DHCP relay with `ip helper-address` on the client-facing interface.

**DHCP relay example**

```bash
interface g0/0
 ip address 192.168.10.1 255.255.255.0
 ip helper-address 192.168.1.2
```

This forwards DHCP broadcast requests to a server at `192.168.1.2`. Use this when the server is not on the same broadcast domain as the client.

**Useful verification commands at a glance**

| Purpose | Command | What you confirm |
|---|---|---|
| NAT translations | `show ip nat translations` | Active mappings and address/port rewriting |
| NAT statistics | `show ip nat statistics` | Translation counts, pool usage, and NAT activity |
| DHCP bindings | `show ip dhcp binding` | Assigned leases and expiration times |
| DHCP pool status | `show ip dhcp pool` | Pool utilization and address availability |
| Interface status | `show ip interface brief` | Up/down state and assigned addresses |
| Running config | `show running-config` | NAT, DHCP, and relay configuration |

**Lab execution mindset**

In a lab, verify in this order:
1. Confirm interface status and addressing.
2. Generate traffic or request a lease.
3. Check the relevant table or binding output.
4. Compare the output to the intended design.
5. Fix only the specific mismatch you observe.

This approach reduces guesswork and helps you distinguish between a configuration problem and a traffic-generation problem. For example, an empty NAT table does not always mean NAT is broken; it may simply mean no inside host has sent traffic yet.

**Common exam-relevant distinctions**

- **Static NAT** creates a fixed mapping, so you expect a persistent translation entry or a configured static statement.
- **Dynamic NAT** creates temporary mappings from a pool, so entries appear only when traffic exists.
- **PAT** uses one public IP for many clients, so port numbers are essential for identifying sessions.
- **DHCP lease** is time-bound, while a static IP assignment is not.
- **DHCP binding** is the server-side record of a lease, while `ipconfig /all` or similar output is the client-side view.

**Quick verification examples**

```bash
Router# show ip nat translations
Router# show ip nat statistics
Router# show ip dhcp binding
Router# show ip dhcp pool
Router# show ip interface brief
```

If you need to force a fresh test, clear the NAT table, renew the client lease, and generate new traffic. That sequence gives you a clean view of the current behavior and makes it easier to validate the lab outcome.
