## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**OSPF fundamentals**

<abbr title="Open Shortest Path First">OSPF</abbr> is a **link-state** routing protocol. Instead of learning routes only from neighbors’ routing tables, each router builds a map of the network from shared topology information and then runs the <abbr title="Shortest Path First">SPF</abbr> algorithm, also called Dijkstra’s algorithm, to choose best paths.

OSPF is widely used because it converges quickly, scales better than simple distance-vector protocols, and supports hierarchical design with areas. For CCNA-level understanding, focus on how OSPF discovers neighbors, exchanges topology data, floods updates, and installs routes.

- **Link-state** means routers advertise the state of their links, not just destination networks.
- **SPF** computes the shortest path tree from the local router’s perspective.
- **Areas** reduce flooding and improve scalability.
- **Classless routing** supports <abbr title="Classless Inter-Domain Routing">CIDR</abbr> and <abbr title="Variable Length Subnet Mask">VLSM</abbr>.

**How OSPF differs from distance-vector routing**

Distance-vector protocols typically send periodic route tables and rely on hop-based metrics. OSPF instead sends topology information when changes occur, then each router independently calculates routes. This reduces slow convergence and helps avoid routing loops.

| Aspect | OSPF | Distance-vector example |
|---|---|---|
| Information shared | Link-state advertisements | Route entries |
| Calculation | Local SPF algorithm | Neighbor-provided distance |
| Convergence | Usually faster | Usually slower |
| Topology view | Full area database | Partial neighbor view |
| Metric | Cost | Often hop count or vendor-specific metric |

**OSPF router ID**

The **router ID** is a 32-bit value that uniquely identifies an OSPF router. It matters because OSPF uses it in neighbor relationships, database exchange, DR/BDR election, and troubleshooting output.

OSPF router ID selection follows this order:

1. Manually configured router ID with `router-id`.
2. Highest IP address on a loopback interface.
3. Highest IP address on an active physical interface.

If you change the router ID after OSPF starts, you usually need to restart the OSPF process for the new ID to take effect.

```cisco
router ospf 1
 router-id 1.1.1.1
```

Why it matters:

- **Neighbor formation** uses the router ID to identify peers.
- **DR/BDR election** on multiaccess networks uses the highest router ID as a tiebreaker after priority.
- **Troubleshooting** is easier when router IDs are stable and predictable.
- **LSDB entries** reference the advertising router ID.

> If two routers accidentally use the same router ID, OSPF adjacency problems and unstable behavior can occur.

**OSPF packet types and their roles**

OSPF uses five packet types to form adjacencies and synchronize the link-state database.

- **Hello**: Discovers neighbors, maintains adjacency, and carries key parameters such as area ID, timers, authentication, and network type.
- **DBD** <abbr title="Database Description">Database Description</abbr>: Summarizes the contents of the link-state database so neighbors can compare what they know.
- **LSR** <abbr title="Link-State Request">Link-State Request</abbr>: Requests specific LSAs that are missing or outdated.
- **LSU** <abbr title="Link-State Update">Link-State Update</abbr>: Carries one or more LSAs to neighbors.
- **LSAck** <abbr title="Link-State Acknowledgment">Link-State Acknowledgment</abbr>: Confirms receipt of LSAs to ensure reliable flooding.

A typical adjacency sequence is:

1. Routers exchange Hello packets and confirm matching parameters.
2. They become neighbors and then exchange DBD packets.
3. Each router identifies missing LSAs and sends LSRs.
4. The neighbor responds with LSUs containing the requested LSAs.
5. The receiving router sends LSAcks to confirm reliable delivery.

**OSPF neighbor states**

Neighbor states help you understand where adjacency is failing.

- **Down**: No Hello packets received.
- **Init**: Hello received, but bidirectional communication is not confirmed.
- **2-Way**: Bidirectional communication exists; on broadcast networks, this may be enough for non-DR/BDR neighbors.
- **ExStart**: Routers decide master/slave roles for database exchange.
- **Exchange**: DBD packets are exchanged.
- **Loading**: LSRs and LSUs are used to fill missing LSAs.
- **Full**: LSDBs are synchronized.

On broadcast and NBMA networks, not every neighbor pair becomes fully adjacent. The presence of a DR and BDR changes which routers form full adjacencies.

**OSPF network types**

OSPF network type affects Hello behavior, neighbor discovery, and DR/BDR election.

| Network type | Common use | Neighbor discovery | DR/BDR election |
|---|---|---|---|
| Broadcast | Ethernet | Automatic multicast discovery | Yes |
| Point-to-point | Serial or point-to-point links | Automatic | No |
| Nonbroadcast multiaccess <abbr title="Nonbroadcast Multiaccess">NBMA</abbr> | Legacy Frame Relay, some hub-and-spoke designs | Manual or configured neighbors may be needed | Yes |
| Point-to-multipoint | Hub-and-spoke style | Automatic per neighbor | No |

Practical impact:

- On **broadcast** networks, OSPF elects a DR and BDR to reduce adjacency overhead.
- On **point-to-point** links, there is only one neighbor relationship, so DR/BDR is unnecessary.
- On **NBMA** networks, neighbor discovery may not be automatic, so manual neighbor statements can be required.
- On **point-to-multipoint** networks, OSPF treats each neighbor like a separate point-to-point relationship.

**DR and BDR election**

The **designated router** <abbr title="Designated Router">DR</abbr> and **backup designated router** <abbr title="Backup Designated Router">BDR</abbr> reduce the number of adjacencies on multiaccess networks.

Election rules:

- Highest OSPF interface priority wins.
- Priority `0` means the router is ineligible to become DR or BDR.
- If priorities tie, the highest router ID wins.
- The DR is elected first, then the BDR.

Example interface configuration:

```cisco
interface g0/0
 ip ospf priority 100
```

Why this matters:

- Routers on a broadcast segment do not all need full adjacencies with every other router.
- Non-DR routers form full adjacency with the DR and BDR, which reduces overhead.
- If the DR fails, the BDR takes over quickly.

**OSPF areas**

Areas divide the OSPF domain into smaller sections to limit flooding and reduce SPF workload. Every OSPF domain has a **backbone area**, which is **area 0**.

Common area concepts:

- **Backbone area 0**: Central transit area that connects all other areas.
- **Normal area**: Accepts inter-area and external LSAs according to design.
- **Stub area**: Blocks external LSAs and uses a default route to reach external destinations.
- **Totally stubby area**: Cisco-specific behavior that blocks more LSA types than a stub area and relies heavily on a default route.
- **Not-so-stubby area** <abbr title="Not-So-Stubby Area">NSSA</abbr>: Allows limited external route injection while still reducing flooding.

At CCNA level, the key idea is that stub-style areas reduce routing information and simplify the LSDB. They are useful when an area does not need full external route detail.

| Area type | External LSAs | Inter-area LSAs | Default route behavior |
|---|---|---|---|
| Normal | Allowed | Allowed | Not automatic |
| Stub | Blocked | Allowed | Injected by ABR |
| Totally stubby | Blocked | Mostly blocked except default | Injected by ABR |
| NSSA | Limited external support | Allowed | Often used with a default route |

> Area 0 is required as the backbone for inter-area routing. Non-backbone areas should connect to area 0 directly or through a design that preserves backbone reachability.

**LSAs and flooding**

<abbr title="Link-State Advertisement">LSA</abbr>s are the building blocks of the OSPF topology database. Routers flood LSAs within the appropriate scope so all routers in an area can build the same view of the topology.

Flooding behavior:

- A router originates LSAs describing its interfaces, neighbors, and learned routes.
- LSAs are flooded to all OSPF routers in the same area, except where area type or LSA type limits scope.
- Each router stores received LSAs in the **link-state database** <abbr title="Link-State Database">LSDB</abbr>.
- The LSDB should be identical among routers in the same area when adjacency is fully synchronized.
- When topology changes, new LSAs are flooded and old information ages out.

Why flooding scope matters:

- **Area scope** limits how far many LSAs travel.
- **Summarization** at area boundaries can reduce route table size.
- **Stub areas** reduce external route flooding.
- **Reliable flooding** uses LSAcks to confirm receipt and prevent loss.

Common LSA concepts to know:

- **Type 1 Router LSA**: Describes a router’s links inside an area.
- **Type 2 Network LSA**: Generated by the DR on broadcast/NBMA networks.
- **Type 3 Summary LSA**: Advertises inter-area routes from an ABR.
- **Type 5 External LSA**: Advertises routes redistributed into OSPF from another routing source.
- **Type 7 NSSA LSA**: Used inside NSSA areas before translation to Type 5 at the ABR.

**OSPF metric and route preference**

OSPF uses **cost** as its metric. Cost is typically based on interface bandwidth, and lower cost is preferred.

A common Cisco-style reference formula is:

- `cost = reference bandwidth / interface bandwidth`

On many Cisco devices, the default reference bandwidth is `100 Mbps`, which can make multiple fast interfaces appear to have the same cost. In modern networks, you may need to adjust the reference bandwidth for better path selection.

Example:

```cisco
router ospf 1
 auto-cost reference-bandwidth 10000
```

This sets the reference bandwidth to `10000 Mbps` so faster links can be differentiated more accurately.

Important distinction:

- **Metric** is used by OSPF to choose the best path within OSPF.
- **Administrative distance** is used to choose between routes learned from different routing sources.

Typical Cisco administrative distance values:

- Connected: `0`
- Static: `1`
- OSPF internal: `110`
- OSPF external: `110`

If two routes to the same destination exist, the router first compares administrative distance. If the routes come from the same protocol, it then compares metric.

**Basic OSPF configuration**

A simple OSPF configuration advertises connected networks into an OSPF process.

```cisco
router ospf 1
 router-id 1.1.1.1
 network 10.1.1.0 0.0.0.255 area 0
 network 10.2.2.0 0.0.0.255 area 0
```

You can also enable OSPF directly on an interface:

```cisco
interface g0/0
 ip ospf 1 area 0
```

Useful verification commands:

```cisco
show ip ospf
show ip ospf neighbor
show ip ospf interface
show ip ospf database
show ip route ospf
```

**Troubleshooting adjacency failures**

When OSPF neighbors do not reach **Full**, check the common mismatch points first.

Typical causes:

- Area ID mismatch
- Hello and dead timer mismatch
- Subnet mask mismatch on broadcast and point-to-point links
- Authentication mismatch
- Network type mismatch
- OSPF process not enabled on the interface
- ACL or firewall blocking protocol 89
- Duplicate router ID
- DR/BDR issues on multiaccess networks

Example failed adjacency scenario:

```cisco
R1# show ip ospf neighbor

Neighbor ID     Pri   State           Dead Time   Address         Interface
2.2.2.2           1   INIT/DROTHER    00:00:31    10.1.1.2        GigabitEthernet0/0
```

What this suggests:

- You are receiving Hellos.
- The neighbor is not seeing your router in its Hello packets, or the adjacency is not progressing.
- You should check whether both routers are in the same area, use the same timers, and agree on network type and authentication.

Useful checks:

```cisco
show ip ospf interface g0/0
show run interface g0/0
show ip protocols
show ip ospf neighbor detail
```

If the state is stuck in **EXSTART** or **EXCHANGE**, suspect:

- MTU mismatch
- Duplicate router ID
- Database exchange problems
- Network type mismatch

Example MTU-related clue:

```cisco
R1# show ip ospf neighbor

Neighbor ID     Pri   State           Dead Time   Address         Interface
2.2.2.2           1   EXSTART/DR      00:00:33    10.1.1.2        GigabitEthernet0/0
```

A common fix is to verify interface MTU settings on both sides:

```cisco
show interfaces g0/0 | include MTU
```

**LSDB synchronization and route installation**

OSPF does not install routes until the LSDB is synchronized and SPF has run. Once the database is complete, the router computes the best path and places the route into the routing table.

Process summary:

1. Neighbor adjacency forms.
2. LSAs are exchanged and acknowledged.
3. The LSDB becomes synchronized.
4. SPF runs on the local router.
5. The best routes are installed in the routing table.

This is why OSPF can converge quickly after a change: only affected LSAs are flooded, and each router recalculates based on the updated topology.

**Practical verification workflow**

Use a structured approach when validating OSPF.

1. Confirm the interface is up and in the correct subnet.
2. Verify OSPF is enabled on the interface or network.
3. Check neighbor state with `show ip ospf neighbor`.
4. Confirm the router ID with `show ip ospf`.
5. Inspect the interface parameters with `show ip ospf interface`.
6. Review the LSDB with `show ip ospf database`.
7. Confirm the route appears in `show ip route ospf`.

Example output interpretation:

- **Full** neighbor state means adjacency is complete.
- **2-Way** on a broadcast network may be normal for non-DR/BDR routers.
- Missing routes often indicate LSDB or area issues, not just interface reachability.
- A stable neighbor relationship with no route installation can point to filtering, area design, or summarization problems.

**Key exam takeaways**

- OSPF is a **link-state** protocol that uses LSAs and SPF to build routes.
- The **router ID** must be unique and stable.
- **Hello**, **DBD**, **LSR**, **LSU**, and **LSAck** each have a specific role in adjacency and synchronization.
- **Network type** affects neighbor behavior and DR/BDR election.
- **Area 0** is the backbone; stub-style areas reduce flooding and route detail.
- OSPF uses **cost** as its metric, while **administrative distance** decides between different routing sources.
- Troubleshooting starts with neighbor state, timers, area settings, and router ID consistency.
