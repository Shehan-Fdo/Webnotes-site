**OSPF verification workflow**

OSPF (Open Shortest Path First) is a link-state routing protocol, so troubleshooting starts by confirming that routers can form adjacencies, exchange link-state information, and install the expected routes. In practice, you verify the neighbor relationship first, then the interface and area settings, and finally the routing table and database contents.

A reliable workflow is:

1. Check whether neighbors are up.
2. Confirm interface status, IP addressing, and area membership.
3. Validate OSPF process settings such as router ID and network statements.
4. Inspect the link-state database and route selection.
5. Compare the observed behavior with the expected topology.

**Core verification commands**

Use these commands to confirm OSPF operation and isolate faults:

```bash
show ip ospf neighbor
show ip ospf interface brief
show ip ospf interface
show ip ospf
show ip route ospf
show ip protocols
show ip ospf database
show running-config | section router ospf
show ip interface brief
```

- `show ip ospf neighbor` confirms adjacency state and neighbor router ID.
- `show ip ospf interface brief` helps you quickly identify which interfaces participate in OSPF.
- `show ip route ospf` verifies that OSPF-learned routes are installed in the routing table.
- `show ip ospf database` shows whether the router has learned the expected LSAs (Link-State Advertisements).
- `show ip protocols` confirms the routing process, advertised networks, and passive-interface settings.

**Neighbor states and what they mean**

OSPF neighbors do not always reach full adjacency immediately. The state tells you where the process is failing.

| State | Meaning | Common issue |
|---|---|---|
| Down | No Hello packets received | Interface down, wrong network, ACL, multicast blocked |
| Init | Hello received, but not bidirectional | One-way communication, mismatched parameters |
| 2-Way | Bidirectional Hello exchange | Normal on multiaccess networks for non-DR/BDR neighbors |
| ExStart | Master/slave negotiation begins | MTU mismatch, duplicate router ID, timer mismatch |
| Exchange | Database Description packets exchanged | MTU mismatch, packet loss, authentication mismatch |
| Loading | Requesting missing LSAs | Database inconsistency, packet loss |
| Full | Adjacency complete | Expected stable state |

On broadcast and multiaccess networks, not every neighbor reaches Full with every router. Non-DR/BDR routers often remain in 2-Way with each other, which is normal. On point-to-point links, you usually expect Full adjacency.

**Most common OSPF problems**

OSPF failures usually come from a small set of configuration mismatches or interface problems.

- **Area mismatch**: Neighbors must share the same area on the same link.
- **Hello/dead timer mismatch**: Hello and dead intervals must match on both sides.
- **Authentication mismatch**: Both routers must use the same authentication type and key.
- **Subnet mismatch**: Interfaces must be in the same IP subnet for adjacency.
- **MTU mismatch**: Different interface MTUs can stall adjacency in ExStart or Exchange.
- **Passive interface**: OSPF does not send Hellos on passive interfaces.
- **ACL or firewall filtering**: OSPF uses IP protocol 89 and multicast addresses `224.0.0.5` and `224.0.0.6`.
- **Duplicate router ID**: Can cause unstable or missing adjacencies.
- **Interface shutdown or Layer 1/2 failure**: OSPF cannot form neighbors on a down link.

> OSPF Hellos are sent to `224.0.0.5` and DR/BDR-related traffic uses `224.0.0.6`. If multicast is filtered, adjacency formation fails even when IP addressing looks correct.

**Interface-level checks**

Before blaming OSPF, confirm the interface itself is healthy.

```bash
show ip interface brief
show interfaces g0/0
show interfaces g0/0 | include line protocol|MTU|reliability
show arp
```

Look for:

- Interface and line protocol both up.
- Correct IP address and subnet mask.
- No unexpected shutdown state.
- No excessive errors, drops, or encapsulation mismatch.
- Correct VLAN assignment on switchports when OSPF runs over routed SVIs or router-on-a-stick designs.

If the interface is on a switch, also verify the Layer 2 path:

```bash
show vlan brief
show interfaces trunk
show spanning-tree interface g0/1 detail
```

**OSPF process and configuration verification**

The OSPF process must advertise the correct interfaces into the correct areas. On Cisco IOS, you commonly verify this with `show ip protocols` and the running configuration.

```bash
show ip protocols
show running-config | section router ospf
```

Example configuration:

```bash
router ospf 1
 router-id 1.1.1.1
 network 10.1.1.0 0.0.0.255 area 0
 network 10.1.2.0 0.0.0.255 area 0
 passive-interface g0/1
```

Key points:

- The **router ID** should be unique within the OSPF domain.
- The **network statement** must match the interface IP address and wildcard mask.
- A **passive interface** still advertises the network but does not form neighbors.
- The OSPF process ID is locally significant on Cisco routers; it does not need to match across neighbors.

**Verifying the OSPF database**

The link-state database shows what the router knows about the topology. If routes are missing, the database can tell you whether the LSAs were learned at all.

```bash
show ip ospf database
show ip ospf database router
show ip ospf database network
show ip ospf database summary
```

Use the database to confirm:

- The router has learned the expected router LSAs.
- Network LSAs exist on broadcast segments with a DR.
- Summary LSAs appear when areas are summarized by an ABR (Area Border Router).
- External LSAs appear when routes are redistributed into OSPF.

If the database is incomplete, the issue may be adjacency formation, area design, or filtering.

**Route verification**

After adjacency and database checks, confirm that OSPF routes are actually installed.

```bash
show ip route ospf
show ip route
```

In the routing table, OSPF routes are marked with:

- `O` for intra-area routes
- `O IA` for inter-area routes
- `O E1` and `O E2` for external routes
- `O N1` and `O N2` for NSSA external routes

Example output pattern:

```bash
O    10.2.2.0/24 [110/20] via 10.1.1.2, 00:00:12, GigabitEthernet0/0
O IA 10.3.3.0/24 [110/30] via 10.1.1.2, 00:00:15, GigabitEthernet0/0
```

Interpretation:

- `110` is the default OSPF administrative distance.
- The second value is the OSPF metric, also called cost.
- The next-hop and outgoing interface show how traffic will be forwarded.

If the route is missing, check whether:

- The destination network is actually advertised.
- The area type allows the route.
- Summarization is hiding the more specific prefix.
- A more preferred route exists from another protocol or static route.

**Hello, dead, and network type checks**

OSPF neighbors must agree on timing and network behavior.

```bash
show ip ospf interface g0/0
```

Verify:

- Hello interval
- Dead interval
- Network type
- Designated Router (DR) and Backup Designated Router (BDR) status
- Cost
- Authentication status

Common defaults:

- Broadcast networks often use a 10-second hello and 40-second dead interval.
- Point-to-point links often use the same defaults unless manually changed.

If timers are mismatched, neighbors may see each other briefly but never reach Full.

**Authentication troubleshooting**

If authentication is enabled, both sides must match exactly.

Example configuration:

```bash
interface g0/0
 ip ospf authentication message-digest
 ip ospf message-digest-key 1 md5 Cisco123
```

Verification:

```bash
show ip ospf interface g0/0
show running-config interface g0/0
```

Troubleshooting tips:

- Confirm both routers use the same authentication type.
- Confirm the same key ID and password.
- Check for key-chain or area-level authentication if used.
- Remember that authentication failures often appear as stuck neighbors in Init, ExStart, or repeated adjacency resets.

**MTU mismatch troubleshooting**

A mismatched Maximum Transmission Unit (MTU) can prevent adjacency from reaching Full, especially in ExStart or Exchange.

Symptoms:

- Neighbor gets stuck in ExStart or Exchange.
- Hellos are exchanged, but database synchronization fails.

Checks:

```bash
show interfaces g0/0 | include MTU
show ip ospf neighbor
```

Possible fixes:

- Make MTU values match on both ends.
- If appropriate in a lab or controlled environment, use `ip ospf mtu-ignore` on one side, but prefer correcting the underlying MTU mismatch.

> `ip ospf mtu-ignore` can help in limited scenarios, but it masks the problem rather than fixing it. Use it only when you understand the operational impact.

**Router ID verification and issues**

The OSPF router ID uniquely identifies each router in the OSPF domain.

```bash
show ip ospf
```

Check:

- The router ID is unique.
- It is stable and not changing unexpectedly.
- It is manually configured when possible for predictability.

Example:

```bash
router ospf 1
 router-id 2.2.2.2
```

If you change the router ID after OSPF has started, you may need to restart the OSPF process for the change to take effect.

```bash
clear ip ospf process
```

Use caution: clearing the process resets adjacencies and can briefly disrupt routing.

**Passive interface verification**

Passive interfaces are useful for preventing neighbor formation on user-facing or loopback interfaces while still advertising the connected network.

```bash
router ospf 1
 passive-interface default
 no passive-interface g0/0
```

Verify with:

```bash
show ip protocols
```

Look for:

- Which interfaces are passive.
- Whether the intended transit interface is exempted from passive mode.
- Whether a missing neighbor is simply caused by passive-interface configuration.

**DR and BDR behavior**

On broadcast networks, OSPF elects a Designated Router and Backup Designated Router to reduce adjacency overhead.

Important points:

- The router with the highest OSPF interface priority becomes DR.
- If priorities tie, the highest router ID wins.
- Non-DR routers form Full adjacency with the DR and BDR, not necessarily with every router.

Verification:

```bash
show ip ospf neighbor
show ip ospf interface g0/0
```

Troubleshooting considerations:

- If the expected router is not DR, check interface priority.
- If the topology changes, DR election does not always re-run unless the segment is reset.
- On point-to-point links, DR/BDR election does not apply.

**Practical troubleshooting sequence**

Use a consistent process so you do not miss simple issues.

1. Verify physical and data-link status with `show ip interface brief` and interface details.
2. Confirm IP addressing, subnet masks, and VLAN or trunk behavior.
3. Check OSPF neighbors with `show ip ospf neighbor`.
4. Inspect OSPF interface parameters with `show ip ospf interface`.
5. Validate the OSPF process and advertised networks with `show ip protocols`.
6. Review the database with `show ip ospf database`.
7. Confirm route installation with `show ip route ospf`.
8. Compare the results against the intended topology and area design.

**Example troubleshooting scenario**

Suppose two routers are connected on `G0/0`, but no OSPF route appears.

```bash
R1# show ip ospf neighbor
R1# show ip ospf interface g0/0
R1# show ip route ospf
```

Possible findings:

- No neighbor appears: check interface status, subnet, ACLs, and passive-interface settings.
- Neighbor is stuck in ExStart: check MTU and authentication.
- Neighbor is Full, but route is missing: check whether the network is actually advertised, summarized, or filtered.
- Route exists but is not preferred: check administrative distance and competing routes.

**Useful configuration examples**

Basic single-area OSPF:

```bash
router ospf 1
 router-id 1.1.1.1
 network 192.168.10.0 0.0.0.255 area 0
 network 192.168.20.0 0.0.0.255 area 0
```

Interface-based OSPF configuration:

```bash
interface g0/0
 ip ospf 1 area 0
 ip ospf cost 10
```

This style is often easier to read during troubleshooting because the OSPF relationship is tied directly to the interface.

**Exam-focused takeaways**

- Verify **neighbors first**, then routes.
- Match **area, timers, authentication, and MTU** on both sides.
- Use `show ip ospf neighbor` to identify adjacency state quickly.
- Use `show ip ospf interface` to inspect per-interface OSPF settings.
- Use `show ip route ospf` to confirm route installation.
- Remember that OSPF uses **IP protocol 89** and multicast addresses `224.0.0.5` and `224.0.0.6`.
- Distinguish normal 2-Way behavior on multiaccess networks from a real failure.

**Quick reference checklist**

- Interface up/up
- Correct IP subnet
- OSPF enabled on the interface
- Matching area number
- Matching hello/dead timers
- Matching authentication
- Matching MTU
- Unique router ID
- No unintended passive-interface setting
- Expected route present in the routing table
