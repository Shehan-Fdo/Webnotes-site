## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Neighbor relationship fundamentals**

Neighbor verification is the process of confirming that devices can discover each other and form the expected adjacency on a network. In CCNA labs, this usually means checking whether switches, routers, and end devices are connected correctly at Layer 2 and Layer 3, and whether routing or discovery protocols are exchanging information as intended.

A healthy neighbor relationship typically shows:
- The physical link is up.
- The interface is in the correct VLAN or subnet.
- The protocol-specific timers, authentication, and addressing match.
- The devices can exchange hello or discovery messages.

Common neighbor types you may verify include:
- **CDP** neighbors between Cisco devices.
- **LLDP** neighbors between multi-vendor devices.
- **Routing neighbors** such as OSPF or EIGRP adjacencies.
- **Switching neighbors** indirectly through MAC address learning and trunk status.

**Why verification matters**

Neighbor checks help you isolate whether a problem is physical, Layer 2, or Layer 3. If a device does not appear as a neighbor, you can narrow the fault to cabling, interface state, VLAN mismatch, IP addressing, routing protocol mismatch, or security features such as authentication or ACLs.

> If a neighbor is missing, do not assume the remote device is down. First confirm interface status, duplex/speed, VLAN membership, and protocol configuration on both ends.

**Layer 1 and Layer 2 checks first**

Before troubleshooting protocol neighbors, verify the link itself. A protocol cannot form adjacency over a down or misconfigured interface.

Useful commands:
```bash
show ip interface brief
show interfaces status
show interfaces description
show running-config interface g0/0
```

What to look for:
- Interface state is `up/up`.
- The correct port is used.
- The interface description matches the intended peer.
- The access or trunk mode matches the design.
- The speed and duplex settings are compatible when manually configured.

For switch ports, also confirm VLAN behavior:
```bash
show vlan brief
show interfaces trunk
show interfaces switchport
```

**Cisco Discovery Protocol verification**

<abbr title="Cisco Discovery Protocol">CDP</abbr> is a Cisco proprietary Layer 2 discovery protocol that identifies directly connected Cisco devices. It is often the fastest way to confirm whether a neighbor is present and which interface connects to it.

Common commands:
```bash
show cdp neighbors
show cdp neighbors detail
show cdp interface
show cdp traffic
```

Typical information shown by `show cdp neighbors`:
- Local interface
- Holdtime
- Capability
- Platform
- Neighbor device ID
- Neighbor port

Example output pattern:
```bash
Device ID        Local Intrfce     Holdtme    Capability  Platform  Port ID
R2               Gig 0/0           153        R S I       ISR4451   Gig 0/1
SW1              Gig 0/1           122        S I         C2960X    Gig 0/24
```

Use `show cdp neighbors detail` when you need:
- The neighbor IP address.
- The exact remote interface.
- Device platform and software hints.
- Confirmation that the neighbor is the expected device.

CDP is useful when:
- You are mapping a lab topology.
- You need to confirm cabling quickly.
- You want to identify the remote interface before configuring routing or trunking.

**LLDP verification**

<abbr title="Link Layer Discovery Protocol">LLDP</abbr> is an open standard discovery protocol used across vendors. It is especially useful in mixed environments or when CDP is disabled.

Common commands:
```bash
show lldp neighbors
show lldp neighbors detail
show lldp traffic
show lldp interface
```

LLDP is often disabled by default on Cisco devices. Enable it globally if needed:
```bash
conf t
lldp run
end
```

You can enable LLDP transmit and receive on an interface:
```bash
conf t
interface g0/1
 lldp transmit
 lldp receive
end
```

Use LLDP when:
- The neighbor is not Cisco.
- You want vendor-neutral discovery.
- You are validating access layer connections in a lab.

**Routing neighbor relationships**

Routing neighbors form when Layer 3 protocols exchange hello packets and agree on key parameters. In CCNA labs, the most common examples are OSPF and EIGRP. Neighbor formation confirms that the routers can exchange routing information and build a routing table from learned routes.

**OSPF neighbor verification**

Open Shortest Path First (<abbr title="Open Shortest Path First">OSPF</abbr>) neighbors must match on several parameters before adjacency forms:
- Area ID
- Hello and dead timers
- Network type
- Authentication, if configured
- Subnet and mask on the link
- Router ID uniqueness

Useful commands:
```bash
show ip ospf neighbor
show ip ospf interface brief
show ip ospf interface g0/0
show ip protocols
```

A healthy OSPF neighbor state often progresses through:
- `DOWN`
- `INIT`
- `2-WAY`
- `EXSTART`
- `EXCHANGE`
- `LOADING`
- `FULL`

For most broadcast and point-to-point links, `FULL` indicates the adjacency is complete. On a multiaccess network, some neighbors may remain in `2-WAY` depending on the design and DR/BDR election.

Example:
```bash
R1# show ip ospf neighbor

Neighbor ID     Pri   State           Dead Time   Address         Interface
2.2.2.2           1   FULL/DR         00:00:33    10.1.1.2        GigabitEthernet0/0
```

If OSPF does not form:
- Check that both interfaces are in the same subnet.
- Confirm the area matches.
- Verify timers and authentication.
- Ensure passive interface is not blocking hello packets.
- Confirm the interface is not shut down.

**EIGRP neighbor verification**

Enhanced Interior Gateway Routing Protocol (<abbr title="Enhanced Interior Gateway Routing Protocol">EIGRP</abbr>) neighbors form when routers exchange hello packets and agree on K-values, AS number, and authentication settings.

Useful commands:
```bash
show ip eigrp neighbors
show ip eigrp interfaces
show ip protocols
```

Example output pattern:
```bash
R1# show ip eigrp neighbors

H   Address         Interface       Hold Uptime   SRTT   RTO  Q  Seq
0   10.1.1.2        Gi0/0           12   00:15:22  10    1000  0  14
```

If EIGRP neighbors do not appear:
- Verify both routers use the same autonomous system number.
- Check interface IP addressing and subnet mask.
- Confirm no ACL blocks EIGRP multicast traffic.
- Verify authentication if configured.
- Ensure the interface is active and not passive.

**IPv6 neighbor and discovery checks**

IPv6 uses Neighbor Discovery Protocol (<abbr title="Neighbor Discovery Protocol">NDP</abbr>) for address resolution and neighbor reachability. In labs, you may verify IPv6 neighbors with:
```bash
show ipv6 neighbors
show ipv6 interface brief
show ipv6 route
```

IPv6 neighbor entries can show:
- Link-local address
- MAC address
- Reachability state
- Interface

This is useful when:
- You need to confirm that IPv6 hosts can resolve each other.
- You are troubleshooting link-local communication.
- You want to verify that router advertisements are being received.

**Switching and trunk-related neighbor checks**

Switch neighbor verification often involves confirming that the correct switchport mode and VLANs are in place. A device may be physically connected but still fail to communicate if the port is in the wrong VLAN or the trunk is not carrying the expected VLANs.

Useful commands:
```bash
show interfaces switchport
show interfaces trunk
show mac address-table dynamic
show spanning-tree interface g0/1 detail
```

What to verify:
- Access ports are assigned to the correct VLAN.
- Trunks are up and negotiating as expected.
- Native VLAN matches on both sides when required.
- Allowed VLAN list includes the needed VLANs.
- Spanning Tree Protocol (<abbr title="Spanning Tree Protocol">STP</abbr>) is not blocking the path unexpectedly.

A MAC address table is a strong indirect indicator of neighbor connectivity:
```bash
show mac address-table dynamic
```

If you see the remote host’s MAC learned on the expected port, Layer 2 forwarding is working.

**Practical lab workflow**

Use a consistent troubleshooting sequence so you do not miss simple issues.

1. Check the interface state on both devices.
2. Confirm cabling and port selection.
3. Verify VLAN, trunk, or subnet configuration.
4. Check discovery or routing protocol settings.
5. Inspect neighbor tables and protocol state.
6. Compare the local and remote configurations.
7. Fix the mismatch and recheck adjacency.

A simple verification workflow for a router-to-router OSPF lab might look like this:
```bash
show ip interface brief
show run interface g0/0
show ip ospf neighbor
show ip ospf interface g0/0
show ip route ospf
```

For a switch-to-switch trunk lab:
```bash
show interfaces status
show interfaces trunk
show cdp neighbors detail
show vlan brief
show mac address-table dynamic
```

**Common causes of missing neighbors**

Neighbor failures usually come from a small set of issues:
- Interface is administratively down.
- Wrong cable or wrong port.
- Mismatched IP subnet or mask.
- VLAN mismatch on access ports.
- Trunk mismatch or missing allowed VLAN.
- OSPF area, timer, or authentication mismatch.
- EIGRP AS number or authentication mismatch.
- CDP or LLDP disabled.
- ACL or security feature blocking protocol traffic.

**Configuration examples**

Enable CDP and LLDP globally if needed:
```bash
conf t
cdp run
lldp run
end
```

Verify a router interface for OSPF adjacency:
```bash
conf t
interface g0/0
 ip address 10.1.1.1 255.255.255.252
 ip ospf 1 area 0
 no shutdown
end
```

Verify a switch trunk:
```bash
conf t
interface g0/1
 switchport mode trunk
 switchport trunk allowed vlan 10,20
 no shutdown
end
```

Verify an access port:
```bash
conf t
interface f0/10
 switchport mode access
 switchport access vlan 10
 no shutdown
end
```

**Interpreting results**

A neighbor table alone is not enough; you must interpret it in context.

- A CDP neighbor confirms Cisco Layer 2 discovery, not end-to-end routing.
- An OSPF neighbor in `FULL` confirms routing adjacency, not application reachability.
- A MAC table entry confirms Layer 2 learning, not correct VLAN design.
- An IPv6 neighbor entry confirms link-local discovery, not global routing success.

The best lab habit is to correlate multiple outputs:
- Interface status
- Neighbor table
- VLAN or trunk state
- Routing table
- Protocol-specific interface details

**Quick reference**

| Task | Command |
|---|---|
| Check interface status | `show ip interface brief` |
| View Cisco neighbors | `show cdp neighbors detail` |
| View LLDP neighbors | `show lldp neighbors detail` |
| Check OSPF neighbors | `show ip ospf neighbor` |
| Check EIGRP neighbors | `show ip eigrp neighbors` |
| Check IPv6 neighbors | `show ipv6 neighbors` |
| Check trunks | `show interfaces trunk` |
| Check MAC learning | `show mac address-table dynamic` |

**Lab takeaway**

Neighbor verification is a fast, reliable way to validate topology and isolate faults. Start with physical and Layer 2 checks, then move to protocol-specific adjacency verification. In CCNA labs, this habit saves time and helps you identify whether the issue is cabling, switching, addressing, or routing.
