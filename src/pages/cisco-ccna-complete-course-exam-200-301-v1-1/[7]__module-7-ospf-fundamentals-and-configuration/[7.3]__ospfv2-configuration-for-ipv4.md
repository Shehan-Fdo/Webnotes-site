## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**OSPFv2 overview**

Open Shortest Path First version 2 (OSPFv2) is a link-state routing protocol for IPv4. It builds a topology map from Link-State Advertisements (LSAs) and uses the Shortest Path First (SPF) algorithm to choose best paths. On Cisco IOS, the OSPF process ID is **locally significant** only; it does not need to match on neighboring routers.

OSPF is a strong CCNA topic because it combines configuration, verification, and troubleshooting. You need to understand how neighbors form, how areas affect route exchange, and how interface settings influence adjacency and route selection.

- **Link-state** routing gives each router a view of the topology, not just a list of next hops.
- **Area 0** is the backbone area and is required for multi-area designs.
- **Cost** is the OSPF metric and is based primarily on interface bandwidth.

**Core OSPF concepts**

OSPF routers exchange Hello packets to discover neighbors and maintain adjacencies. On broadcast networks, Hellos use multicast addresses `224.0.0.5` for all OSPF routers and `224.0.0.6` for designated routers and backup designated routers. This multicast behavior helps routers find each other without manual neighbor statements in most Ethernet designs.

OSPF neighbor formation depends on several parameters matching:
- Area ID
- Hello and dead timers
- Authentication settings
- Subnet and mask on the interface
- Network type, in some cases
- Stub area type, if used

If any of these do not match, adjacency may stop before the routers reach **FULL** state.

**OSPF packet exchange and neighbor states**

OSPF uses a predictable sequence of neighbor states. These states help you identify where adjacency is failing.

- **DOWN**: No Hello packets have been received yet.
- **INIT**: A Hello has been received, but your router ID is not yet seen in the neighbor’s Hello.
- **2-WAY**: Both routers see each other in Hellos; on broadcast networks, this is often enough for non-adjacent DROTHER neighbors.
- **EXSTART**: Routers negotiate master/slave roles and start database synchronization.
- **EXCHANGE**: Routers exchange Database Description (DBD) packets.
- **LOADING**: Routers request missing LSAs with Link-State Request (LSR) packets.
- **FULL**: LSDBs are synchronized and adjacency is complete.

> If a neighbor stays in `EXSTART`, `EXCHANGE`, or `LOADING`, suspect MTU mismatch, timer mismatch, authentication mismatch, or an area/network-type problem.

**OSPF router roles on broadcast networks**

On a broadcast segment, OSPF elects a Designated Router (DR) and Backup Designated Router (BDR) to reduce adjacency overhead.

- **DR**: Central adjacency point for the segment.
- **BDR**: Standby router that takes over if the DR fails.
- **DROTHER**: Any router that is neither DR nor BDR.

A DROTHER router typically forms full adjacencies with the DR and BDR, but not with every other DROTHER on the segment. The neighbor state `2-WAY` is common between DROTHER routers on a broadcast network and is expected behavior.

**Basic OSPFv2 configuration**

You can configure OSPF in two common styles on Cisco IOS:
- **Network statement style** in router configuration mode
- **Interface-based style** with `ip ospf <process-id> area <area-id>` under the interface

Both are valid. The process ID is local to the router, so you can choose any unused number on each device.

Example using network statements:

```cisco
conf t
router ospf 10
 router-id 1.1.1.1
 network 10.1.1.0 0.0.0.255 area 0
 network 10.1.2.0 0.0.0.255 area 0
end
```

Example using interface-based configuration:

```cisco
conf t
interface g0/0
 ip ospf 10 area 0
interface g0/1
 ip ospf 10 area 0
router ospf 10
 router-id 1.1.1.1
end
```

Use interface-based configuration when you want to attach OSPF directly to an interface without matching a network statement. Use network statements when you want to enable OSPF across a range of interfaces with one command.

**Wildcard masks and network statements**

OSPF network statements use wildcard masks, not subnet masks. A wildcard mask is the inverse of a subnet mask.

Common examples:
- `255.255.255.0` subnet mask becomes `0.0.0.255` wildcard
- `255.255.255.252` subnet mask becomes `0.0.0.3` wildcard
- `255.255.255.240` subnet mask becomes `0.0.0.15` wildcard

Examples:
- `network 192.168.10.0 0.0.0.255 area 0` matches all interfaces in `192.168.10.0/24`
- `network 10.0.0.4 0.0.0.3 area 0` matches `10.0.0.4/30`
- `network 172.16.1.0 0.0.0.255 area 1` matches a /24 in area 1

A network statement does two things:
- It enables OSPF on matching interfaces.
- It places those interfaces into the specified area.

**Router ID selection and control**

The OSPF router ID is a 32-bit value written in dotted-decimal format. Cisco IOS chooses it in this order when the OSPF process starts:
1. Manually configured router ID
2. Highest loopback interface IP address
3. Highest active physical interface IP address

Best practice is to set it manually so it stays stable.

```cisco
conf t
router ospf 10
 router-id 1.1.1.1
end
```

If you change the router ID after the OSPF process has already started, the new value does not take effect until the process restarts.

```cisco
clear ip ospf process
```

> Use `clear ip ospf process` carefully because it resets OSPF adjacencies and temporarily disrupts routing.

**OSPF cost and metric calculation**

OSPF uses **cost** as its metric. On Cisco IOS, interface cost is derived from bandwidth using the formula:

`cost = reference-bandwidth / interface-bandwidth`

By default, Cisco uses a reference bandwidth of `100 Mbps`, so:
- `100 Mbps` interface cost = `1`
- `10 Mbps` interface cost = `10`
- `1 Gbps` interface cost still often shows as `1` by default, which can make multiple fast links look equal

This default is too small for modern networks because many interfaces end up with the same cost. You can change the reference bandwidth so higher-speed links are differentiated.

```cisco
conf t
router ospf 10
 auto-cost reference-bandwidth 10000
end
```

The value is in Mbps, so `10000` means `10 Gbps`.

You can also set interface cost directly:

```cisco
conf t
interface g0/0
 ip ospf cost 50
end
```

Use manual interface cost when you want precise control over path selection. Use reference bandwidth changes when you want all OSPF costs to scale more realistically across the network.

**Passive interfaces**

A passive interface advertises its connected network into OSPF but does not send Hello packets on that interface. This is useful for user LANs where you want route advertisement without forming neighbors.

```cisco
conf t
router ospf 10
 passive-interface default
 no passive-interface g0/0
end
```

In this example:
- All interfaces become passive by default
- `g0/0` remains active and can form neighbors
- Other interfaces still advertise their connected networks if they are included in OSPF

This is a common exam topic because passive interfaces reduce unnecessary adjacency formation and improve security.

**Verification commands**

Use verification commands to confirm neighbor state, interface participation, and route installation.

```cisco
show ip ospf
show ip ospf interface brief
show ip ospf interface g0/0
show ip ospf neighbor
show ip route ospf
show ip protocols
```

What to look for:
- `show ip ospf neighbor` confirms adjacency state and DR/BDR roles
- `show ip ospf interface` confirms area, timers, network type, cost, and authentication
- `show ip route ospf` shows learned OSPF routes with codes such as `O` and `O IA`
- `show ip protocols` confirms which networks are advertised and which interfaces are passive

Route codes matter:
- `O` = intra-area OSPF route
- `O IA` = inter-area OSPF route
- `O E1` / `O E2` = external OSPF route types, usually from redistribution

**Multi-area OSPF and ABR behavior**

A multi-area design uses more than one area to reduce LSDB size and limit SPF calculations. An Area Border Router (ABR) connects area 0 to another area and advertises routes between them.

Simple example topology:
- R1 in area 0
- R2 in area 0 and area 1
- R3 in area 1

Example configuration:

```cisco
! R1
conf t
router ospf 10
 router-id 1.1.1.1
 network 10.0.0.0 0.0.0.3 area 0
 network 192.168.1.0 0.0.0.255 area 0
end
```

```cisco
! R2
conf t
router ospf 10
 router-id 2.2.2.2
 network 10.0.0.0 0.0.0.3 area 0
 network 10.0.0.4 0.0.0.3 area 1
end
```

```cisco
! R3
conf t
router ospf 10
 router-id 3.3.3.3
 network 10.0.0.4 0.0.0.3 area 1
 network 192.168.3.0 0.0.0.255 area 1
end
```

In this design:
- R2 becomes the ABR because it has interfaces in both area 0 and area 1
- R1 learns area 1 networks as `O IA` routes
- R3 learns area 0 networks as `O IA` routes
- Routers inside the same area learn each other’s routes as `O`

**Route summarization**

OSPF supports summarization at area boundaries and for external routes.

Area range summarization on an ABR:

```cisco
conf t
router ospf 10
 area 1 range 192.168.0.0 255.255.252.0
end
```

This can summarize multiple contiguous area 1 networks into one advertisement toward area 0. Use summarization to reduce routing table size and limit LSA flooding.

External route summarization is configured on an ASBR (Autonomous System Boundary Router) when redistributing routes into OSPF:

```cisco
conf t
router ospf 10
 summary-address 172.16.0.0 255.255.0.0
end
```

Summarization works best when networks are contiguous and aligned to the summary boundary.

**Default route advertisement**

You can inject a default route into OSPF so other routers learn a gateway of last resort.

```cisco
conf t
ip route 0.0.0.0 0.0.0.0 203.0.113.1
router ospf 10
 default-information originate
end
```

If the router does not already have a default route, use `default-information originate always` with care. That forces advertisement even without a matching static default.

**Common troubleshooting scenarios**

OSPF troubleshooting is usually about adjacency, not just route presence. Start by checking neighbor state, interface settings, and area membership.

1. Verify the interface is up and in the correct subnet.
2. Check that both routers are in the same area.
3. Confirm Hello and dead timers match.
4. Confirm authentication matches, if enabled.
5. Check passive-interface settings.
6. Verify the network type and MTU if adjacency stalls in exchange states.
7. Confirm the router ID is unique.

Common failure patterns:
- **Area mismatch**: neighbors may not form; check `network` statements and interface area assignment.
- **Authentication mismatch**: neighbors may remain down or fail to progress.
- **Timer mismatch**: Hellos are ignored if hello/dead intervals differ.
- **Passive interface misconfiguration**: no neighbor forms on that interface because Hellos are suppressed.
- **MTU mismatch**: adjacency may get stuck in `EXSTART` or `EXCHANGE`.
- **Duplicate router ID**: can cause unstable or missing adjacencies.

Example authentication configuration:

```cisco
conf t
interface g0/0
 ip ospf authentication message-digest
 ip ospf message-digest-key 1 md5 Cisco123
end
```

Both sides must use the same authentication method and key.

**Network type and point-to-point links**

OSPF network type affects neighbor behavior and DR/BDR election. Ethernet defaults to broadcast, while serial point-to-point links often work best as point-to-point.

If the physical media or design does not match the default behavior, you may need to change the network type.

```cisco
conf t
interface s0/0/0
 ip ospf network point-to-point
end
```

Use this when a link should not perform DR/BDR election, or when a nonbroadcast design needs a more suitable OSPF behavior. On point-to-point links, adjacency formation is simpler and faster because there is no DR/BDR election.

**Hello and dead timers**

Hello packets maintain neighbor relationships. The dead interval is the time a router waits without receiving Hellos before declaring the neighbor down. These values must match on both sides of the link.

Typical defaults:
- Broadcast and point-to-point: Hello 10 seconds, Dead 40 seconds
- Some other network types may use different defaults

Verify timers with:

```cisco
show ip ospf interface g0/0
```

If timers do not match, adjacency usually fails before FULL state.

**Practical lab example**

Suppose you have two routers connected by `10.1.12.0/30`, and each router also has a LAN.

R1:
```cisco
conf t
interface g0/0
 ip address 10.1.12.1 255.255.255.252
 no shutdown
interface g0/1
 ip address 192.168.1.1 255.255.255.0
 no shutdown
router ospf 10
 router-id 1.1.1.1
 network 10.1.12.0 0.0.0.3 area 0
 network 192.168.1.0 0.0.0.255 area 0
 passive-interface g0/1
end
```

R2:
```cisco
conf t
interface g0/0
 ip address 10.1.12.2 255.255.255.252
 no shutdown
interface g0/1
 ip address 192.168.2.1 255.255.255.0
 no shutdown
router ospf 10
 router-id 2.2.2.2
 network 10.1.12.0 0.0.0.3 area 0
 network 192.168.2.0 0.0.0.255 area 0
 passive-interface g0/1
end
```

Expected result:
- R1 and R2 form an OSPF adjacency on `g0/0`
- Each router advertises its LAN into area 0
- Each router learns the other LAN as an intra-area route, shown as `O`

**Troubleshooting checklist**

- Confirm the interface is not administratively down.
- Confirm the IP address and mask match the intended subnet.
- Confirm OSPF is enabled on the interface.
- Confirm the area ID matches on both sides.
- Confirm the interface is not passive if you expect neighbors.
- Confirm authentication, timers, and network type match.
- Confirm the router ID is unique and stable.
- Confirm the route appears in `show ip route ospf`.

**Exam focus points**

- OSPFv2 is for IPv4; OSPFv3 is the IPv6 version.
- Cisco OSPF process ID is local to the router.
- Wildcard masks are used in `network` statements.
- Area 0 is the backbone.
- `O` means intra-area, `O IA` means inter-area.
- Passive interfaces advertise networks without forming neighbors.
- Cost is based on bandwidth unless you override it.
- Neighbor states help you pinpoint where adjacency fails.

**Key takeaways**

OSPFv2 configuration is about more than enabling a routing protocol. You must place interfaces into the correct area, understand how neighbors form, and verify that cost, timers, and interface behavior support adjacency. When you can read neighbor states and route codes confidently, you can troubleshoot most CCNA-level OSPF problems quickly.
