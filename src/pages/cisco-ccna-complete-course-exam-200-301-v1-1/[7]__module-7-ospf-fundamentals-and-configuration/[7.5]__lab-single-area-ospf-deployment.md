## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Lab overview**

Single-area OSPF (Open Shortest Path First) is a link-state routing protocol used to exchange routes within one OSPF area, typically area `0`. In a CCNA lab, you usually build a small multi-router topology, assign IP addresses, enable OSPF on the participating interfaces, and verify that neighbors form correctly and routes appear in the routing table.

This lab focuses on the operational workflow you need in the exam and in real deployments:

- Build a consistent IP plan.
- Enable OSPF in one area.
- Confirm neighbor adjacency.
- Verify route propagation.
- Troubleshoot common adjacency failures.

**Reference topology and addressing plan**

A simple three-router lab is the most useful for learning OSPF behavior because it shows both point-to-point adjacency and broadcast-segment DR/BDR behavior.

Example topology:

- `R1` connects to `R2` over `G0/1`
- `R2` connects to `R3` over `G0/2`
- `R1`, `R2`, and `R3` each have a LAN interface on `G0/0`
- All OSPF-enabled interfaces belong to area `0`

Example addressing:

| Router | Interface | IP address | Purpose |
|---|---:|---:|---|
| R1 | G0/0 | `192.168.1.1/24` | LAN |
| R1 | G0/1 | `10.0.12.1/30` | Link to R2 |
| R2 | G0/0 | `192.168.2.1/24` | LAN |
| R2 | G0/1 | `10.0.12.2/30` | Link to R1 |
| R2 | G0/2 | `10.0.23.1/30` | Link to R3 |
| R3 | G0/0 | `192.168.3.1/24` | LAN |
| R3 | G0/1 | `10.0.23.2/30` | Link to R2 |

> If your lab uses a shared Ethernet segment instead of point-to-point links, DR/BDR election applies on that broadcast network. On point-to-point links, DR/BDR does not apply.

**OSPF concepts you must know before configuring**

OSPF is a link-state protocol, so each router builds a map of the network from link-state advertisements (LSAs) and then runs the Shortest Path First (SPF) algorithm to calculate best paths.

Key terms:

- **Router ID (RID)**: A 32-bit identifier used to uniquely identify an OSPF router.
- **Neighbor adjacency**: The relationship between routers that exchange OSPF information.
- **Area**: A logical grouping of OSPF routers; in this lab, all routers use area `0`.
- **Hello packets**: Used to discover and maintain neighbors.
- **Dead timer**: The time before a neighbor is considered down if hellos stop arriving.
- **DR/BDR**: Designated Router and Backup Designated Router, used on broadcast and NBMA networks to reduce adjacency overhead.

OSPF neighbor formation depends on several values matching:

- Area ID
- Hello and dead timers
- Network type
- Authentication settings, if used
- Subnet and mask on the connected link
- OSPF process behavior and interface participation

**Router ID selection and why it matters**

The router ID is critical because it identifies the router in OSPF databases and neighbor tables. Cisco IOS selects the router ID in this order:

1. Manually configured OSPF router ID
2. Highest loopback IP address
3. Highest active physical interface IP address

For exam prep, remember that changing the router ID in configuration does not immediately replace the active RID. You must clear the OSPF process for the new ID to take effect.

Example:

```bash
R1(config)# router ospf 1
R1(config-router)# router-id 1.1.1.1
R1# clear ip ospf process
```

> Clearing the OSPF process resets adjacencies and forces OSPF to restart with the new router ID.

**Step-by-step lab build**

1. Configure interface IP addresses on all routers.
2. Enable OSPF process `1` on each router.
3. Advertise all connected networks into area `0`.
4. Verify neighbor adjacencies.
5. Confirm routes are learned.
6. Test end-to-end connectivity with `ping` and `traceroute`.
7. Troubleshoot any missing neighbors or routes.

**R1 configuration**

```bash
R1# configure terminal
R1(config)# hostname R1
R1(config)# interface g0/0
R1(config-if)# ip address 192.168.1.1 255.255.255.0
R1(config-if)# no shutdown
R1(config-if)# exit
R1(config)# interface g0/1
R1(config-if)# ip address 10.0.12.1 255.255.255.252
R1(config-if)# no shutdown
R1(config-if)# exit
R1(config)# router ospf 1
R1(config-router)# router-id 1.1.1.1
R1(config-router)# network 192.168.1.0 0.0.0.255 area 0
R1(config-router)# network 10.0.12.0 0.0.0.3 area 0
R1(config-router)# end
```

**Alternative interface-based OSPF configuration**

Some labs use interface mode instead of `network` statements. This is often easier to read because you enable OSPF directly on each interface.

```bash
R1(config)# router ospf 1
R1(config-router)# router-id 1.1.1.1
R1(config-router)# exit
R1(config)# interface g0/0
R1(config-if)# ip ospf 1 area 0
R1(config-if)# interface g0/1
R1(config-if)# ip ospf 1 area 0
```

Use this method when you want precise control over which interfaces participate in OSPF.

**Verification commands and what to look for**

After configuration, verify the control plane first, then the data plane.

Common commands:

```bash
show ip ospf neighbor
show ip ospf interface brief
show ip ospf interface g0/0
show ip route ospf
show ip protocols
show ip ospf
ping 192.168.3.1
traceroute 192.168.3.1
```

Expected neighbor behavior on point-to-point links:

- `R1` should see `R2` in `FULL` state on `G0/1`
- `R2` should see `R1` and `R3` in `FULL` state on the two routed links
- `R3` should see `R2` in `FULL` state on `G0/1`

Example output:

```bash
R2# show ip ospf neighbor

Neighbor ID     Pri   State           Dead Time   Address         Interface
1.1.1.1           0   FULL/-          00:00:33    10.0.12.1       GigabitEthernet0/1
3.3.3.3           0   FULL/-          00:00:36    10.0.23.2       GigabitEthernet0/2
```

On a broadcast LAN, the expected states differ:

- Routers form full adjacencies with the **DR** and **BDR**
- Routers that are neither DR nor BDR are **DROTHER**
- DROTHER routers do not form full adjacencies with each other
- DROTHER routers still exchange OSPF information through the DR/BDR

Example broadcast-segment output:

```bash
R1# show ip ospf neighbor

Neighbor ID     Pri   State           Dead Time   Address         Interface
2.2.2.2           1   FULL/DR         00:00:31    192.168.1.2     GigabitEthernet0/0
3.3.3.3           1   FULL/BDR        00:00:34    192.168.1.3     GigabitEthernet0/0
```

**Interface status and OSPF role**

`show ip ospf interface brief` helps you confirm which interfaces participate in OSPF and whether the interface is acting as DR, BDR, or DROTHER on a broadcast network.

Example:

```bash
R2# show ip ospf interface brief
Interface    PID   Area            IP Address/Mask    Cost  State  Nbrs F/C
Gi0/0        1     0               192.168.2.1/24     1     DR     0/0
Gi0/1        1     0               10.0.12.2/30       1     P2P    1/1
Gi0/2        1     0               10.0.23.1/30       1     P2P    1/1
```

Interpretation:

- `DR` on a LAN means the interface is the designated router for that broadcast segment.
- `P2P` means point-to-point, where DR/BDR election does not apply.
- `Nbrs F/C` shows full neighbors versus total neighbors seen.

**Routing table expectations**

Once OSPF converges, each router should learn remote LANs as OSPF routes marked with `O` in the routing table.

Example on `R1`:

```bash
R1# show ip route ospf
O    192.168.2.0/24 [110/2] via 10.0.12.2, 00:00:18, GigabitEthernet0/1
O    192.168.3.0/24 [110/2] via 10.0.12.2, 00:00:18, GigabitEthernet0/1
```

Example on `R3`:

```bash
R3# show ip route ospf
O    192.168.1.0/24 [110/2] via 10.0.23.1, 00:00:21, GigabitEthernet0/1
O    192.168.2.0/24 [110/1] via 10.0.23.1, 00:00:21, GigabitEthernet0/1
```

The administrative distance for OSPF is `110`. The metric is the OSPF cost, which is based on interface bandwidth unless manually changed.

**End-to-end testing**

After routes appear, test reachability between LANs.

Example:

```bash
R1# ping 192.168.3.1
R1# traceroute 192.168.3.1
```

If the ping succeeds, you have confirmed:

- Interface addressing is correct
- OSPF adjacency is up
- Routes are installed
- Return routing exists

**How OSPF chooses paths**

OSPF uses cost, not hop count. Lower cost is preferred.

Important points:

- Default cost is derived from interface bandwidth.
- You can influence path selection by changing interface bandwidth or OSPF cost.
- If multiple equal-cost paths exist, OSPF can load-balance across them.

Example manual cost adjustment:

```bash
R2(config)# interface g0/1
R2(config-if)# ip ospf cost 50
```

Use this when you want to influence the preferred path in a lab or when bandwidth values do not reflect the intended design.

**DR/BDR election in a single-area lab**

DR/BDR election occurs on broadcast and NBMA networks, not on point-to-point links. In a three-router Ethernet segment, OSPF elects:

- One **DR**
- One **BDR**
- Remaining routers as **DROTHER**

Election is based on:

1. Highest OSPF interface priority
2. Highest router ID if priorities tie

Useful commands:

```bash
show ip ospf neighbor
show ip ospf interface g0/0
show ip ospf database
```

If you need to force a different DR or BDR in a lab, set interface priority:

```bash
R1(config)# interface g0/0
R1(config-if)# ip ospf priority 100
R2(config)# interface g0/0
R2(config-if)# ip ospf priority 50
R3(config)# interface g0/0
R3(config-if)# ip ospf priority 0
```

A priority of `0` makes the router ineligible to become DR or BDR.

**Common troubleshooting cases**

OSPF problems usually fall into a few repeatable categories. Check them in a consistent order.

1. Verify the interface is up/up.
2. Verify IP addressing and masks.
3. Verify OSPF is enabled on the correct interfaces.
4. Verify area IDs match.
5. Verify hello and dead timers match.
6. Verify network type matches where required.
7. Verify authentication matches if configured.
8. Verify router IDs are unique.
9. Clear the OSPF process after changes that require reinitialization.

Common failure symptoms:

- **No neighbor appears**
  - Interface is down
  - Wrong subnet or mask
  - OSPF not enabled on the interface
  - Area mismatch
  - Hello/dead timer mismatch
  - Authentication mismatch

- **Neighbor stuck in INIT**
  - One-way hello reception
  - ACL or filtering issue
  - Mismatched timers or network type
  - Neighbor never sees your hello in return

- **Neighbor stuck in EXSTART/EXCHANGE**
  - MTU mismatch is a common cause
  - DBD exchange cannot complete
  - Less commonly, network type or other adjacency parameters interfere

- **Routes missing even though neighbors are up**
  - Network not advertised
  - Wrong area
  - Passive interface on a transit link
  - Summarization or filtering in a more advanced design

Example of a timer mismatch problem:

```bash
R1(config)# interface g0/1
R1(config-if)# ip ospf hello-interval 5
R2(config)# interface g0/1
R2(config-if)# ip ospf hello-interval 10
```

This prevents adjacency formation because the hello and dead timers must match on both sides.

Example of an area mismatch:

```bash
R1(config)# interface g0/1
R1(config-if)# ip ospf 1 area 0
R2(config)# interface g0/1
R2(config-if)# ip ospf 1 area 1
```

This prevents neighbors from forming because OSPF routers on the same link must agree on the area.

Example of an authentication mismatch:

```bash
R1(config)# interface g0/1
R1(config-if)# ip ospf authentication message-digest
R1(config-if)# ip ospf message-digest-key 1 md5 cisco123
```

If the neighbor does not use the same authentication method and key, adjacency fails.

**Resetting OSPF during the lab**

You often need to reset OSPF after changing router IDs, DR/BDR priorities, or other adjacency-sensitive settings.

Useful commands:

```bash
clear ip ospf process
clear ip ospf neighbor
```

Use `clear ip ospf process` when you need OSPF to rebuild adjacencies and recalculate the database from scratch. This is especially important after changing the router ID.

**Expected end-state for a correct three-router lab**

When the lab is correct, you should see:

- All participating interfaces are `up/up`
- OSPF neighbors are in `FULL` state on point-to-point links
- On a broadcast LAN, one router is `DR`, one is `BDR`, and others are `DROTHER`
- Each router learns the remote LANs as OSPF routes
- `ping` succeeds between all LAN interfaces
- `traceroute` follows the intended path through the transit router

Example summary of expected routing knowledge:

- `R1` knows `192.168.2.0/24` and `192.168.3.0/24`
- `R2` knows `192.168.1.0/24` and `192.168.3.0/24`
- `R3` knows `192.168.1.0/24` and `192.168.2.0/24`

**Quick review checklist**

- OSPF process number is locally significant.
- Area `0` is the backbone area.
- Router ID must be unique.
- Point-to-point links do not use DR/BDR.
- Broadcast networks do use DR/BDR.
- Hello and dead timers must match.
- MTU mismatch often breaks adjacency in `EXSTART/EXCHANGE`.
- Changing the router ID requires clearing the OSPF process.

**Practical lab workflow**

1. Build the topology and cable the routers.
2. Assign IP addresses and bring interfaces up.
3. Configure router IDs.
4. Enable OSPF in area `0`.
5. Verify neighbors with `show ip ospf neighbor`.
6. Verify routes with `show ip route ospf`.
7. Test reachability with `ping`.
8. Troubleshoot mismatches and clear OSPF if needed.
9. Recheck the final state after every change.

**Why this lab matters for CCNA**

This lab combines several exam objectives into one workflow:

- IP addressing and subnetting
- OSPF configuration
- Neighbor verification
- Route verification
- Basic troubleshooting

If you can build and validate a single-area OSPF topology from scratch, you are ready for many CCNA-style simulation questions because the same logic applies to larger designs: match parameters, verify adjacency, confirm routes, and isolate the failure by checking one layer at a time.
