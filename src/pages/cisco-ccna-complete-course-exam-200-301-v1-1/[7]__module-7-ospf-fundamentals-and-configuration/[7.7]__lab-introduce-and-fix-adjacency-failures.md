## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Lab objective and troubleshooting mindset**

Adjacency failures happen when two devices cannot form a working neighbor relationship at Layer 2 or Layer 3. In a CCNA lab, you usually introduce a fault, observe the symptoms, and then restore connectivity by checking the most likely causes first: physical link state, VLAN mismatch, trunking issues, IP addressing, masks, routing, and protocol timers or authentication.

The key skill is not memorizing one fix. You need a repeatable process that narrows the problem quickly using show commands, interface status, and protocol-specific verification.

- **Adjacency** means two devices can exchange the control traffic needed to become neighbors.
- **Layer 2** failures often involve VLANs, trunks, duplex/speed, or STP-related blocking.
- **Layer 3** failures often involve IP addressing, masks, routing, ACLs, or protocol mismatch.
- **Verification** should happen after every change so you know which fix actually solved the issue.

**Common adjacency types in CCNA labs**

Different protocols build adjacencies in different ways, but the troubleshooting pattern is similar. You first confirm the link is up, then confirm the devices are in the same logical domain, and finally confirm the protocol settings match.

| Adjacency type | Typical purpose | Common failure causes |
|---|---|---|
| Switch neighbor discovery | Identify directly connected switches | VLAN mismatch, trunk mismatch, STP blocking, wrong cabling |
| Router-to-router Layer 3 neighbor | Exchange routing information | Wrong IP/mask, interface down, ACL, authentication mismatch |
| Dynamic routing neighbor | Form routing adjacency | Hello timer mismatch, network type mismatch, passive interface, AS/area mismatch |
| First-hop gateway reachability | Reach default gateway | Wrong VLAN, wrong SVI, host IP error, gateway mismatch |

**Initial checks before changing anything**

Start with the simplest evidence. If the interface is down, protocol troubleshooting is wasted effort. If the interface is up but the devices still do not see each other, move to logical configuration.

Useful commands:

```bash
show ip interface brief
show interfaces status
show interfaces
show cdp neighbors
show lldp neighbors
show vlan brief
show interfaces trunk
show ip route
```

What to look for:

- **Interface state** should be `up/up` for a healthy routed link.
- **Administrative down** means the interface is shut.
- **Protocol down** often points to Layer 2 or encapsulation mismatch.
- **VLAN membership** must match the host or switchport design.
- **Trunk status** must agree on both ends if the link carries multiple VLANs.
- **Routing table presence** confirms whether a route exists to the remote network.

**Layer 1 and Layer 2 adjacency failures**

Physical and data-link issues are the most common starting point. A cable problem, wrong port, or disabled interface prevents any higher-layer adjacency from forming.

Typical symptoms:

- No neighbor appears in `show cdp neighbors` or `show lldp neighbors`.
- Interface shows `administratively down`, `down/down`, or `up/down`.
- A switchport is in the wrong VLAN.
- A trunk is not negotiating or is manually mismatched.
- Spanning Tree Protocol (STP) places a port in blocking state.

Common fixes:

- Enable the interface with `no shutdown`.
- Verify the correct cable and port mapping.
- Match speed and duplex when required by the lab.
- Place access ports in the correct VLAN.
- Ensure both ends agree on trunking and allowed VLANs.

Example switchport configuration:

```bash
interface gigabitEthernet0/1
 switchport mode access
 switchport access vlan 10
 no shutdown
```

Example trunk configuration:

```bash
interface gigabitEthernet0/1
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30
 no shutdown
```

If you suspect STP is preventing forwarding, check the port role and state:

```bash
show spanning-tree interface gigabitEthernet0/1 detail
show spanning-tree vlan 10
```

> Do not disable STP in a troubleshooting lab unless the instructions explicitly require it. STP protects the topology from loops, and turning it off can create a broadcast storm.

**VLAN and trunk mismatch scenarios**

A very common lab fault is a host or switchport placed in the wrong VLAN. Another is a trunk that carries the wrong VLAN list or is configured as access on one side and trunk on the other.

Symptoms of VLAN mismatch:

- Hosts in the same subnet cannot communicate.
- The switch learns MAC addresses on the wrong VLAN.
- The default gateway is unreachable even though the link is up.

Verification commands:

```bash
show vlan brief
show interfaces switchport
show interfaces trunk
show mac address-table dynamic
```

Practical checks:

- Confirm the access port VLAN matches the host subnet.
- Confirm the SVI or router subinterface for that VLAN is up.
- Confirm the trunk allows the VLAN end to end.
- Confirm native VLAN settings match if the lab uses 802.1Q trunks.

Example router-on-a-stick subinterface:

```bash
interface gigabitEthernet0/0.10
 encapsulation dot1Q 10
 ip address 192.168.10.1 255.255.255.0
```

If the VLAN is missing from the trunk allowed list, add it:

```bash
interface gigabitEthernet0/1
 switchport trunk allowed vlan add 10
```

**Layer 3 adjacency failures**

Layer 3 adjacencies depend on correct IP configuration and a reachable path between interfaces. Even if the physical link is up, a wrong subnet mask or missing route can prevent neighbor formation or end-to-end reachability.

Check these items first:

- IP addresses are in the correct subnet.
- Subnet masks match the design.
- Interfaces are enabled.
- The remote network is reachable through routing.
- ACLs are not blocking control traffic.

Useful commands:

```bash
show ip interface brief
show running-config interface gigabitEthernet0/0
show ip route
ping
traceroute
```

Example of a correct point-to-point router link:

```bash
interface gigabitEthernet0/0
 ip address 10.0.0.1 255.255.255.252
 no shutdown
```

On the peer:

```bash
interface gigabitEthernet0/0
 ip address 10.0.0.2 255.255.255.252
 no shutdown
```

If the mask is wrong on one side, the devices may believe the peer is local or remote incorrectly, which breaks routing and neighbor discovery.

**Dynamic routing neighbor problems**

When the lab uses a dynamic routing protocol, the routers must agree on protocol-specific settings. A mismatch can stop adjacency even when the interfaces are up and IP connectivity exists.

For CCNA-level troubleshooting, focus on:

- **OSPF** area number, hello/dead timers, network type, authentication, and passive interfaces.
- **EIGRP** autonomous system number, K values, authentication, and passive interfaces.
- **RIPv2** version, auto-summary behavior, and network statements.

OSPF verification commands:

```bash
show ip ospf neighbor
show ip ospf interface brief
show ip protocols
show running-config | section router ospf
```

EIGRP verification commands:

```bash
show ip eigrp neighbors
show ip protocols
show running-config | section router eigrp
```

RIP verification commands:

```bash
show ip protocols
show ip route rip
show running-config | section router rip
```

Common OSPF fixes:

- Match the area on both sides.
- Match hello and dead timers.
- Remove `passive-interface` from links that must form neighbors.
- Ensure authentication settings match if enabled.

Example OSPF configuration:

```bash
router ospf 1
 network 10.0.0.0 0.0.0.3 area 0
```

If a router is passive on the wrong interface, it will advertise the network but not form neighbors on that link:

```bash
router ospf 1
 no passive-interface gigabitEthernet0/0
```

**ACL and security-related adjacency issues**

Access control lists can block the traffic needed for adjacency. This is less common in simple labs, but it is a realistic cause when security features are present.

Possible symptoms:

- Pings fail even though interfaces are up.
- Routing neighbors do not form across a filtered link.
- Management protocols work from one side but not the other.

Check for ACLs on interfaces:

```bash
show running-config interface gigabitEthernet0/0
show access-lists
show ip interface gigabitEthernet0/0
```

If an ACL is blocking required traffic, adjust it carefully. For example, an overly restrictive ACL on a routed interface may block ICMP or routing protocol packets.

Example ACL inspection:

```bash
ip access-list extended BLOCK-TEST
 deny ip any any
```

If this is applied in the wrong direction, it can break adjacency and reachability.

**Troubleshooting workflow for the lab**

Use a consistent sequence so you do not miss the obvious issue.

1. Check interface state with `show ip interface brief` or `show interfaces status`.
2. Confirm the correct cable, port, and device pair.
3. Verify VLAN, trunk, or routed interface design.
4. Check IP addressing and subnet masks.
5. Inspect routing or dynamic routing neighbor status.
6. Review ACLs, authentication, and passive-interface settings.
7. Make one change at a time and retest immediately.

A practical verification loop:

```bash
show ip interface brief
show cdp neighbors
show interfaces trunk
show ip ospf neighbor
ping 10.0.0.2
```

If the first fix does not work, do not stack multiple changes blindly. Re-check the current state so you know whether the issue is still Layer 2, Layer 3, or protocol-specific.

**Real-world lab examples**

Example 1: A switchport is in the wrong VLAN.

- Symptom: Host cannot reach the gateway.
- Check: `show vlan brief` shows the port in VLAN 1 instead of VLAN 10.
- Fix: Move the port to VLAN 10.
- Verify: Ping the default gateway and confirm MAC learning in VLAN 10.

Example 2: A trunk does not allow the needed VLAN.

- Symptom: Two switches are connected, but hosts on the far side are unreachable.
- Check: `show interfaces trunk` shows VLAN 20 missing from the allowed list.
- Fix: Add VLAN 20 to the trunk.
- Verify: Confirm traffic passes and the remote SVI responds.

Example 3: OSPF neighbors do not form.

- Symptom: Interfaces are up, but `show ip ospf neighbor` is empty.
- Check: One router uses area 0 and the other uses area 1, or hello timers differ.
- Fix: Match the OSPF area and timers.
- Verify: Neighbor state reaches `FULL`.

Example 4: A passive interface blocks adjacency.

- Symptom: The router advertises the network, but no neighbor appears.
- Check: `show ip protocols` shows the interface as passive.
- Fix: Remove passive mode from the transit interface.
- Verify: Neighbor adjacency forms and routes appear.

**Useful show commands and what they tell you**

| Command | What it confirms |
|---|---|
| `show ip interface brief` | Interface status and IP assignment |
| `show interfaces status` | Switchport state and VLAN assignment |
| `show cdp neighbors` | Directly connected Cisco devices |
| `show lldp neighbors` | Directly connected devices using LLDP |
| `show vlan brief` | VLAN existence and port membership |
| `show interfaces trunk` | Trunk mode, native VLAN, allowed VLANs |
| `show ip route` | Whether routes exist to the destination |
| `show ip protocols` | Routing protocol settings and passive interfaces |
| `show ip ospf neighbor` | OSPF adjacency state |
| `show ip eigrp neighbors` | EIGRP adjacency state |

**Best practices during the lab**

- Change one variable at a time.
- Record the original fault before fixing it.
- Verify both local and remote sides after each change.
- Use `ping` and `traceroute` to confirm data-plane recovery.
- Recheck the configuration after the fix to ensure it matches the intended design.

**Quick reference: common fixes by symptom**

- No link light: check cable, port, and `no shutdown`.
- Link up but no neighbor: check VLAN, trunk, or IP subnet.
- Neighbor exists but traffic fails: check routing, ACLs, or gateway settings.
- Routing neighbor missing: check timers, area/AS, authentication, and passive interfaces.
- Host cannot reach gateway: check access VLAN, SVI status, and default gateway address.

**Key takeaway**

Adjacency troubleshooting is a layered process. You start with physical and interface state, then validate Layer 2 design, then confirm Layer 3 addressing and routing, and finally inspect protocol-specific settings. If you work from the bottom up and verify after each change, you can isolate the fault quickly and restore connectivity with confidence.
