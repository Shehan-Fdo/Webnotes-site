## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**OSPF overview**

Open Shortest Path First (<abbr title="Open Shortest Path First">OSPF</abbr>) is a link-state routing protocol. Each router builds a map of the topology from link-state advertisements and then runs the Shortest Path First algorithm to choose best paths. For CCNA, the most important operational ideas are:

- **Router ID** identifies the OSPF process, not a physical interface.
- **Areas** limit flooding and help scale the topology.
- **DR/BDR** reduce adjacency overhead on multiaccess networks.

OSPF behavior is often tested through troubleshooting: you may have the right IP addressing and still fail to form the expected neighbor relationship because the router ID, area, or network type is wrong.

**Router ID fundamentals**

The OSPF router ID is a 32-bit value written in dotted-decimal format, such as `1.1.1.1`. It is used to uniquely identify the router inside the OSPF domain. It is not required to match any interface IP address.

A router ID can be unrelated to any active interface. That matters in troubleshooting because you might see a router ID like `10.10.10.10` even when no interface currently uses that address. OSPF keeps using the selected router ID until the process restarts or the ID is changed and the process is reset.

**How OSPF chooses the router ID**

OSPF selects the router ID in this order:

1. Manually configured router ID with `router-id`.
2. Highest IP address on a loopback interface.
3. Highest IP address on an active physical interface.

A loopback is preferred because it is stable. Physical interface addresses can disappear if the link goes down, which can cause operational confusion if you expect the router ID to track an interface.

```bash
router ospf 1
 router-id 1.1.1.1
```

If you do not configure a router ID, OSPF may choose a loopback or physical interface address automatically. If you later add a loopback with a higher IP address, the router ID does not automatically change unless the OSPF process is restarted or the router ID is explicitly set.

> If you change the router ID after OSPF has already started, you usually need to restart the OSPF process for the new ID to take effect.

**Verifying the router ID**

Use show commands to confirm the configured and active router ID:

```bash
show ip ospf
show ip protocols
show ip ospf neighbor
```

Typical `show ip ospf` output includes the router ID and area information:

```bash
Router# show ip ospf

Routing Process "ospf 1" with ID 1.1.1.1
Supports only single TOS(TOS0) routes
It is an area border router
SPF schedule delay 5 secs, Hold time between two SPFs 10 secs
```

Interpretation tips:

- `Routing Process "ospf 1" with ID 1.1.1.1` confirms the active router ID.
- `It is an area border router` means the router has interfaces in more than one area.
- If the router ID is not what you expect, check for:
  - a manually configured `router-id`
  - a loopback with a higher IP address
  - an OSPF process that has not been restarted after changes

A useful troubleshooting sequence is:

1. Check the active router ID with `show ip ospf`.
2. Check interface IPs and loopbacks with `show ip interface brief`.
3. Confirm whether a manual `router-id` exists in the OSPF configuration.
4. If needed, restart the OSPF process to force re-selection.

```bash
clear ip ospf process
```

**Router ID troubleshooting scenarios**

- If two routers accidentally use the same router ID, OSPF neighbor formation can fail or behave unpredictably.
- If a router ID changes unexpectedly after a reboot, a loopback or interface address selection likely changed.
- If you are troubleshooting a neighbor issue, do not assume the router ID must match a live interface address; it often does not.

**Areas and why they matter**

An OSPF area is a logical grouping of routers and links. Areas reduce the size of the link-state database and limit the scope of flooding. This improves scalability and reduces SPF calculation overhead.

The backbone area is **Area 0**. All other areas must connect to Area 0 directly or through a virtual link in designs beyond CCNA scope. For exam purposes, remember:

- Area 0 is the core of the OSPF domain.
- Non-backbone areas connect to Area 0.
- Routers with interfaces in multiple areas are **ABRs** (<abbr title="Area Border Router">Area Border Routers</abbr>).

**Area membership and verification**

You can verify area membership with `show ip ospf interface brief` and `show ip ospf interface`:

```bash
show ip ospf interface brief
show ip ospf interface g0/0
show ip ospf
```

Example interpretation:

```bash
Router# show ip ospf interface brief
Interface    PID   Area            IP Address/Mask    Cost  State Nbrs F/C
Gig0/0       1     0               10.1.1.1/24        1     BDR   2/2
Gig0/1       1     1               10.2.2.1/24        1     DR    1/1
```

- `Area 0` on `Gig0/0` means that interface belongs to the backbone.
- `Area 1` on `Gig0/1` means that interface belongs to a non-backbone area.
- `State` shows the interface’s DR/BDR role on multiaccess networks.

`show ip ospf interface g0/1` gives more detail, including the area ID, network type, and timers. This is useful when the area looks correct but neighbors still do not form.

**Common area types**

CCNA-relevant area types affect which routes are advertised into the area and how much external information the area accepts.

- **Standard area**
  - Accepts intra-area, inter-area, and external routes.
  - This is the default area type.
- **Stub area**
  - Blocks external Type 5 LSAs.
  - The ABR injects a default route into the stub area instead of full external routes.
  - Use this to reduce routing table size in small branch areas.
- **Totally stubby area**
  - Cisco-specific variation that blocks external routes and also suppresses most inter-area routes.
  - The area typically receives only a default route from the ABR.
  - Useful when you want maximum route reduction.
- **NSSA** (<abbr title="Not-So-Stubby Area">Not-So-Stubby Area</abbr>)
  - Allows limited external route injection from inside the area.
  - Common when a branch area needs to redistribute a small number of routes into OSPF.
- **Totally NSSA**
  - Blocks most inter-area routes and still allows limited external redistribution inside the area.
  - The ABR usually provides a default route.

A practical way to think about area types:

- Use a **standard area** when you need full route visibility.
- Use a **stub** or **totally stubby** area when you want to reduce routing table size and control external route flooding.
- Use an **NSSA** when the area must originate some external routes but should still stay relatively small.

> Stub, totally stubby, NSSA, and totally NSSA area types must match on neighboring routers in that area. A mismatch prevents adjacency.

**Area behavior and route advertisement**

Area type changes what the router learns:

- **Intra-area routes** stay inside the same area.
- **Inter-area routes** are learned from other areas through an ABR.
- **External routes** come from redistribution into OSPF.

In a stub area, external routes are suppressed. In a totally stubby area, even more non-local routes are hidden, leaving mostly a default route. In an NSSA, the area can still originate external information, but the ABR translates it for the rest of the OSPF domain.

This matters in troubleshooting because a router may have OSPF neighbors and still miss expected routes if the area type is intentionally filtering them.

**Configuring OSPF areas**

A common CCNA-style configuration uses network statements to place interfaces into areas:

```bash
router ospf 1
 network 10.1.1.0 0.0.0.255 area 0
 network 10.2.2.0 0.0.0.255 area 1
```

You can also configure OSPF directly on an interface:

```bash
interface g0/0
 ip ospf 1 area 0
```

Interface-based configuration is often easier to read during troubleshooting because the area is visible on the interface itself.

**DR/BDR on multiaccess networks**

On broadcast and some NBMA (<abbr title="Non-Broadcast Multi-Access">Non-Broadcast Multi-Access</abbr>) networks, OSPF elects a **Designated Router** (DR) and **Backup Designated Router** (BDR). This reduces the number of adjacencies and the amount of flooding.

Without DR/BDR, every router on a multiaccess segment would need a full adjacency with every other router. That scales poorly. With DR/BDR, routers form full adjacencies only with the DR and BDR.

**Adjacency pattern on a broadcast segment**

On a segment with routers R1, R2, R3, and R4:

- All routers discover each other as neighbors.
- The DR forms full adjacencies with all other routers.
- The BDR forms full adjacencies with all other routers.
- DROTHER routers do **not** form full adjacencies with each other.
- DROTHER routers exchange full LSDB synchronization only with the DR and BDR.

This is a common exam trap: neighbor presence does not mean full adjacency with every router.

A simple view:

- R1 (DR) ↔ R2 (BDR)
- R1 (DR) ↔ R3 (DROTHER)
- R1 (DR) ↔ R4 (DROTHER)
- R2 (BDR) ↔ R3 (DROTHER)
- R2 (BDR) ↔ R4 (DROTHER)
- R3 (DROTHER) ↔ R4 is not a full adjacency

**DR/BDR election rules**

Election is based on:

1. Highest OSPF interface priority.
2. Highest router ID if priorities tie.

Default priority is `1`. A priority of `0` makes the router ineligible to become DR or BDR.

```bash
interface g0/0
 ip ospf priority 100
```

Use a higher priority when you want a router with better CPU, memory, or connectivity to become DR. Use `0` when you want to prevent a router from participating in the election.

**Election behavior when routers join or fail**

The election process is not fully preemptive:

- If the current DR fails, the BDR takes over.
- A new election occurs to choose a replacement BDR.
- If a higher-priority router joins later, it does **not** automatically replace the current DR just because it has a better priority.

This behavior is important in real networks because the DR role tends to remain stable until failure or reset. If you want a different router to become DR, you usually need to reset the OSPF process or the interface after adjusting priorities.

A practical sequence:

1. Existing DR fails.
2. BDR promotes itself to DR.
3. Remaining routers elect a new BDR.
4. A later-arriving higher-priority router joins as a DROTHER unless the election is reset.

**Network type and when DR/BDR applies**

OSPF network type affects whether DR/BDR election happens.

- **Broadcast**
  - DR/BDR election occurs.
  - Common on Ethernet.
- **Non-broadcast**
  - DR/BDR election occurs.
  - Neighbors are often configured manually because multicast discovery may not work.
- **Point-to-point**
  - No DR/BDR election.
  - Common on serial links and many routed point-to-point links.
- **Point-to-multipoint**
  - No DR/BDR election.
  - Treats each neighbor more like a separate point-to-point relationship.

You can verify the network type with:

```bash
show ip ospf interface g0/0
```

Example output clues:

```bash
Router# show ip ospf interface g0/0
GigabitEthernet0/0 is up, line protocol is up
  Internet Address 10.1.1.1/24, Area 0
  Process ID 1, Router ID 1.1.1.1, Network Type BROADCAST, Cost: 1
  State DR, Priority 1
```

If the network type is not what you expect, adjacency behavior may also be wrong. For example, a link configured as point-to-point will never elect a DR or BDR.

**Configuring OSPF network type**

You can change the OSPF network type on an interface when the default behavior does not match the medium.

```bash
interface s0/0/0
 ip ospf network point-to-point
```

Common use cases:

- Use `point-to-point` on a direct router-to-router link to avoid DR/BDR election.
- Use `non-broadcast` on some Frame Relay or NBMA-style environments where multicast discovery is not reliable.
- Leave Ethernet as `broadcast` unless a design requirement says otherwise.

**Troubleshooting DR/BDR and area issues**

When OSPF does not behave as expected, check the problem in layers:

1. Confirm the interface is up/up.
2. Confirm the interface is in the correct area.
3. Confirm the network type matches the medium.
4. Confirm the router ID is unique and stable.
5. Confirm the area type matches on all routers in the area.
6. Confirm the DR/BDR election result matches the interface priorities.

Useful commands:

```bash
show ip ospf
show ip ospf interface brief
show ip ospf interface g0/0
show ip ospf neighbor
show ip protocols
show running-config | section router ospf
show running-config interface g0/0
```

`show ip ospf neighbor` is especially important because it reveals neighbor state and role:

```bash
Router# show ip ospf neighbor

Neighbor ID     Pri   State           Dead Time   Address         Interface
2.2.2.2           1   FULL/BDR        00:00:33    10.1.1.2        Gig0/0
3.3.3.3           1   FULL/DROTHER    00:00:31    10.1.1.3        Gig0/0
4.4.4.4           1   2WAY/DROTHER    00:00:34    10.1.1.4        Gig0/0
```

Interpretation:

- `FULL/BDR` means the neighbor is fully adjacent and is the BDR.
- `FULL/DROTHER` means the neighbor is fully adjacent but is not DR or BDR.
- `2WAY/DROTHER` often indicates a router on a broadcast segment that sees the neighbor but does not form a full adjacency with it because neither side is DR/BDR.

**Scenario-based troubleshooting examples**

- If two routers on Ethernet see each other in `2WAY` state but never reach `FULL`, check whether a DR/BDR election is occurring and whether the interface is in the same area.
- If a router has the wrong router ID, check for a loopback with a higher IP address or a stale OSPF process.
- If routes are missing from a branch area, check whether the area is stub, totally stubby, or NSSA.
- If a router on a multiaccess segment is not becoming DR even with a high priority, remember that election is not preemptive; the current DR stays until failure or reset.

**Quick exam checklist**

- **Router ID**
  - Unique within the OSPF domain
  - Can be manually set
  - May not match any active interface
  - Verified with `show ip ospf`
- **Areas**
  - Area 0 is the backbone
  - ABRs connect areas
  - Stub and NSSA types change route advertisement
  - Area type must match on neighbors
- **DR/BDR**
  - Applies on broadcast and some NBMA networks
  - Highest priority wins, then highest router ID
  - DR/BDR form full adjacencies with all routers
  - DROTHER routers do not fully adjacently peer with each other
  - Election is not preemptive


These commands let you verify the router ID, area membership, network type, and DR/BDR roles quickly during lab work or exam-style troubleshooting.
