## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Passive interface fundamentals**

A passive interface is a routing-protocol interface that does not send routing updates out of that interface. You still advertise the connected network into the routing process, but you stop sending hello packets and neighbor-forming traffic on the passive link. This is useful on user LANs, server VLANs, and other segments where no routing neighbor should exist.

In CCNA labs, you usually configure passive interfaces to reduce unnecessary protocol chatter, improve security, and keep routing adjacencies only on links that actually connect routers.

- **Routing updates** are suppressed on the passive interface.
- The connected network remains part of the routing domain.
- Neighbor relationships do not form on that interface.
- The feature is common in dynamic routing protocols such as RIP, EIGRP, and OSPF.

**Why you use passive interfaces**

Passive interfaces help you control where routing protocols are allowed to speak. On a LAN with end hosts, there is no reason for routers to exchange routing hellos with PCs, printers, or access points. Making the interface passive prevents accidental adjacency attempts and reduces multicast or broadcast routing traffic.

Typical benefits include:

- Less control-plane overhead on access networks.
- Better security by limiting who can hear routing protocol messages.
- Cleaner troubleshooting because neighbors appear only on intended links.
- Reduced risk of rogue routing devices forming adjacencies.

> A passive interface does not stop route advertisement for the connected network; it only stops protocol communication on that interface.

**How passive interface behavior differs by protocol**

Different routing protocols implement passive behavior in a similar way, but the exact effect can vary slightly.

| Protocol | Effect of passive interface | Notes |
|---|---|---|
| RIP | Stops sending and receiving RIP updates on the interface | The network can still be advertised through the routing process |
| EIGRP | Stops sending and receiving EIGRP hello packets on the interface | No neighbor adjacency forms on that interface |
| OSPF | Stops sending and receiving OSPF hello packets on the interface | The interface remains in the OSPF network statement, but no neighbor forms |
| IS-IS | Passive behavior is less commonly emphasized in CCNA labs | Focus on the protocol used in the lab |

**Common lab topology**

A simple lab often uses one router connected to another router on a WAN or point-to-point link, plus one or more LAN interfaces facing end devices.

- Router-to-router link: keep active so neighbors can form.
- Router-to-switch LAN interface: make passive so no routing neighbor forms with hosts.
- Loopback interfaces: often passive by design in many labs.

Example topology:

```text
PCs --- Switch --- R1 --- R2 --- Remote network
              |
              +--- LAN interface should be passive
```

**Cisco IOS configuration pattern**

The most common CCNA lab task is to enable a routing protocol and then make all interfaces passive except the one used for the router neighbor.

Example with OSPF:

```cisco
router ospf 1
 network 192.168.10.0 0.0.0.255 area 0
 network 10.0.0.0 0.0.0.3 area 0
 passive-interface default
 no passive-interface g0/0
```

This configuration means:

- All interfaces are passive by default.
- `g0/0` is the only interface allowed to form OSPF neighbors.
- The LAN network can still be advertised if it matches the OSPF network statement.

A similar pattern works for EIGRP:

```cisco
router eigrp 100
 network 192.168.10.0 0.0.0.255
 network 10.0.0.0 0.0.0.3
 passive-interface default
 no passive-interface g0/0
```

And for RIP:

```cisco
router rip
 version 2
 network 192.168.10.0
 network 10.0.0.0
 passive-interface default
 no passive-interface g0/0
```

**Step-by-step lab workflow**

1. Identify which interface connects to another router.
2. Identify which interfaces connect to end-user networks.
3. Enable the routing protocol and advertise the required networks.
4. Set passive interface behavior on the LAN-facing interfaces.
5. Leave the router-to-router link active.
6. Verify that neighbors form only on the intended link.

**Verification commands**

Use show commands to confirm both adjacency state and interface behavior.

```cisco
show ip ospf neighbor
show ip protocols
show running-config | section router ospf
show ip route
```

For EIGRP:

```cisco
show ip eigrp neighbors
show ip protocols
show running-config | section router eigrp
```

For RIP:

```cisco
show ip protocols
show ip route rip
show running-config | section router rip
```

What to look for:

- The intended router neighbor appears on the active interface.
- No neighbor appears on passive LAN interfaces.
- The connected LAN network is still present in the routing table if it is being advertised or learned appropriately.
- `show ip protocols` often lists passive interfaces explicitly.

**Troubleshooting passive interface issues**

If a neighbor does not form, check whether the interface is passive by mistake. This is one of the most common lab errors.

- Verify the interface is not passive on the router-to-router link.
- Confirm the interface IP addressing and subnet mask match the neighbor.
- Check that both routers use the same routing protocol and compatible settings.
- Ensure OSPF area, EIGRP AS number, or RIP version matches where required.
- Confirm the interface is up/up.

Useful checks:

```cisco
show ip interface brief
show ip protocols
show running-config | include passive-interface
```

If you accidentally make the wrong interface passive, remove that setting or override it with `no passive-interface` on the correct link.

**Design considerations**

Passive interfaces are a best practice on interfaces that should never form routing adjacencies. In enterprise networks, this usually includes:

- User access VLANs
- Printer VLANs
- Server VLANs where no routing peer exists
- Loopbacks used for router IDs or stable reachability
- Management networks, if no dynamic neighbor is expected

You generally keep passive disabled only on:

- Point-to-point router links
- Transit links between routing devices
- Any interface where a routing adjacency is intentionally required

**Practical example: OSPF lab**

Suppose R1 has:

- `g0/0` connected to a switch and user LAN `192.168.10.0/24`
- `g0/1` connected to R2 on `10.0.0.0/30`

A clean OSPF configuration is:

```cisco
router ospf 1
 network 192.168.10.0 0.0.0.255 area 0
 network 10.0.0.0 0.0.0.3 area 0
 passive-interface default
 no passive-interface g0/1
```

Expected result:

- OSPF adjacency forms only on `g0/1`.
- The LAN on `g0/0` is advertised but does not send OSPF hellos.
- End hosts on the LAN do not participate in OSPF.

**Practical example: interface-specific passive configuration**

You can also make only one interface passive instead of using the default-all approach.

```cisco
router ospf 1
 network 192.168.10.0 0.0.0.255 area 0
 network 10.0.0.0 0.0.0.3 area 0
 passive-interface g0/0
```

This is useful when you have only one LAN interface to suppress and want to keep the rest active. In larger labs, `passive-interface default` is usually safer because it forces you to explicitly permit only the intended neighbor links.

**Key exam takeaways**

- Passive interfaces stop routing protocol messages on that interface.
- They do not remove the network from the routing process.
- Use them on LAN-facing interfaces to reduce noise and improve security.
- Use `passive-interface default` plus `no passive-interface` for a secure, scalable configuration.
- If neighbors fail to form, check whether the interface is passive first.

**Quick memory aid**

- **Passive** = no neighbor formation on that interface.
- **Advertise** = network can still be included in routing.
- **Active link** = only router-to-router transit interfaces should usually stay active.
