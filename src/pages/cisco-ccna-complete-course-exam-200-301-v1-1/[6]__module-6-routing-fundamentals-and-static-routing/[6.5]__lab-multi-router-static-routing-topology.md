## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Lab overview**

A multi-router static routing lab teaches you how to connect multiple LANs across routed links using manually configured routes. You use static routes when you want predictable forwarding, small-scale control, or a simple design that does not require a dynamic routing protocol.

In this lab, you typically build a topology with three routers, each with a directly connected LAN and one or more inter-router links. The goal is to make every remote network reachable by adding the correct static routes on each router and then verifying end-to-end connectivity.

- **Static routing** means you manually define the destination prefix and the path to reach it.
- You must configure routes on every router that does not already have a directly connected path to the destination.
- Verification matters as much as configuration because a single wrong next hop or mask breaks reachability.

**Topology and addressing**

A common lab design uses router-to-router links in a line or triangle, with each router serving a different LAN. For example:

- R1 connects to `192.168.10.0/24`
- R2 connects to `192.168.20.0/24`
- R3 connects to `192.168.30.0/24`
- Inter-router links use separate subnets such as `10.0.12.0/30` and `10.0.23.0/30`

A simple linear topology looks like this:

```text
PC1 --- R1 --- R2 --- R3 --- PC3
         |      |      |
      LAN10   LAN20   LAN30
```

Before you add static routes, each router knows only its directly connected networks and local interface addresses. Remote LANs remain unreachable until you define a path.

**How static routes are selected**

Cisco IOS uses route selection in this order:

1. **Longest prefix match** wins first.
2. If multiple routes match the same prefix length, the route with the lowest **administrative distance** wins.
3. If administrative distance is equal, the router uses the best metric for the routing source.

For static routing labs, the key point is that connected routes already match their own subnets, and static routes are used only for remote networks. A connected route does not “always outrank” a static route in every case; the prefix length must match first. If a more specific static route exists, it can be chosen over a less specific connected or static route.

**Static route types**

You can configure static routes in several forms. The best choice depends on the link type.

| Type | Example | Typical use |
|---|---|---|
| Next-hop only | `ip route 192.168.30.0 255.255.255.0 10.0.12.2` | Common on Ethernet multiaccess links |
| Exit-interface only | `ip route 192.168.30.0 255.255.255.0 g0/0` | Common on point-to-point links |
| Fully specified | `ip route 192.168.30.0 255.255.255.0 g0/0 10.0.12.2` | Useful when you want both interface and next hop explicit |

On Ethernet multiaccess networks, a **next-hop-only** static route is often preferred because the router forwards toward a specific neighbor and avoids unnecessary ambiguity. An **exit-interface-only** route is valid, but on a multiaccess Ethernet segment the router may need to resolve the destination with ARP behavior that is less efficient than pointing to a single next hop. A **fully specified** route is useful when you want to be explicit about both the outgoing interface and the neighbor address.

> On point-to-point links, exit-interface-only static routes are normal and efficient because there is only one possible neighbor on that link.

**Basic configuration workflow**

You usually configure interfaces first, then add static routes, then verify.

1. Assign IP addresses to all router interfaces.
2. Enable interfaces with `no shutdown`.
3. Add static routes for all remote networks.
4. Test connectivity with `ping` and `traceroute`.
5. Inspect the routing table and correct any mistakes.

Example interface setup:

```cisco
R1# configure terminal
R1(config)# interface g0/0
R1(config-if)# ip address 192.168.10.1 255.255.255.0
R1(config-if)# no shutdown
R1(config-if)# exit
R1(config)# interface g0/1
R1(config-if)# ip address 10.0.12.1 255.255.255.252
R1(config-if)# no shutdown
R1(config-if)# end
```

Example static route configuration on R1:

```cisco
R1# configure terminal
R1(config)# ip route 192.168.20.0 255.255.255.0 10.0.12.2
R1(config)# ip route 192.168.30.0 255.255.255.0 10.0.12.2
R1(config)# end
```

If R2 sits between R1 and R3, it needs routes to both remote LANs:

```cisco
R2# configure terminal
R2(config)# ip route 192.168.10.0 255.255.255.0 10.0.12.1
R2(config)# ip route 192.168.30.0 255.255.255.0 10.0.23.2
R2(config)# end
```

R3 needs routes back toward R1 and R2:

```cisco
R3# configure terminal
R3(config)# ip route 192.168.10.0 255.255.255.0 10.0.23.1
R3(config)# ip route 192.168.20.0 255.255.255.0 10.0.23.1
R3(config)# end
```

**Default routes in a lab**

A default static route is useful when a router has one obvious path to all unknown destinations. In a multi-router lab, you may use a default route on edge routers, but not usually on the middle router unless the design calls for it.

```cisco
ip route 0.0.0.0 0.0.0.0 10.0.12.2
```

Use a default route when:

- The router has a single upstream path.
- You want to reduce the number of specific static routes.
- The router acts as a stub router with one exit path.

Avoid relying on a default route when:

- You need precise control over multiple remote networks.
- The topology has more than one possible upstream path.
- You are trying to learn exact route behavior for the exam.

**Verification commands**

Cisco IOS provides several commands to confirm static routing behavior. Both `show ip route` and `show ip route static` are valid and useful.

```cisco
show ip route
show ip route static
show running-config | section ip route
show ip interface brief
```

What to look for:

- `S` in the routing table indicates a static route.
- `C` indicates a connected route.
- `L` indicates a local host route for the interface IP address.
- The next hop and outgoing interface should match the intended path.

Example routing table entry:

```text
S    192.168.30.0/24 [1/0] via 10.0.12.2
C    10.0.12.0/30 is directly connected, GigabitEthernet0/1
L    10.0.12.1/32 is directly connected, GigabitEthernet0/1
```

Use `traceroute` to confirm the path across routers:

```cisco
R1# traceroute 192.168.30.10
```

Use `ping` from both routers and end hosts:

```cisco
R1# ping 192.168.30.1
PC1> ping 192.168.30.10
```

**Troubleshooting static routes**

Most static routing failures come from a small set of mistakes.

- **Wrong next hop**: the router points to an address that is not reachable on the directly connected link.
- **Wrong mask**: the destination network is summarized incorrectly or the subnet mask does not match the real network.
- **Interface down**: the outgoing interface is administratively down or physically disconnected.
- **Missing return route**: forward path exists, but the remote router has no route back.
- **Overlapping prefixes**: a more specific route or a conflicting summary changes forwarding behavior.

A useful troubleshooting sequence is:

1. Check interface status with `show ip interface brief`.
2. Check connected networks with `show ip route`.
3. Check static routes with `show ip route static`.
4. Verify the configured commands in `show running-config`.
5. Test the next hop with `ping`.
6. Trace the path with `traceroute`.

> If one direction works and the other fails, suspect the return path first. Static routing requires routes in both directions unless the destination is on a directly connected network.

**Exit interface, next hop, and recursive lookup**

When you configure a next-hop-only static route, the router must resolve the next-hop IP to an outgoing interface using its connected routes. This is called **recursive lookup**. That is normal and usually acceptable on Ethernet multiaccess networks.

A fully specified route gives the router both the exit interface and the next hop:

```cisco
ip route 192.168.30.0 255.255.255.0 g0/1 10.0.12.2
```

Use this form when you want to be explicit, especially in designs where the outgoing interface is not obvious. On point-to-point links, an exit-interface-only route is also common:

```cisco
ip route 192.168.30.0 255.255.255.0 s0/0/0
```

**Practical lab example**

Suppose you have three routers in a line:

- R1 LAN: `192.168.10.0/24`
- R2 LAN: `192.168.20.0/24`
- R3 LAN: `192.168.30.0/24`
- R1-R2 link: `10.0.12.0/30`
- R2-R3 link: `10.0.23.0/30`

R1 configuration:

```cisco
interface g0/0
 ip address 192.168.10.1 255.255.255.0
 no shutdown
interface g0/1
 ip address 10.0.12.1 255.255.255.252
 no shutdown
ip route 192.168.20.0 255.255.255.0 10.0.12.2
ip route 192.168.30.0 255.255.255.0 10.0.12.2
```

R2 configuration:

```cisco
interface g0/0
 ip address 10.0.12.2 255.255.255.252
 no shutdown
interface g0/1
 ip address 10.0.23.1 255.255.255.252
 no shutdown
interface g0/2
 ip address 192.168.20.1 255.255.255.0
 no shutdown
ip route 192.168.10.0 255.255.255.0 10.0.12.1
ip route 192.168.30.0 255.255.255.0 10.0.23.2
```

R3 configuration:

```cisco
interface g0/0
 ip address 10.0.23.2 255.255.255.252
 no shutdown
interface g0/1
 ip address 192.168.30.1 255.255.255.0
 no shutdown
ip route 192.168.10.0 255.255.255.0 10.0.23.1
ip route 192.168.20.0 255.255.255.0 10.0.23.1
```

After configuration, verify from R1:

```cisco
show ip route static
ping 192.168.20.1
ping 192.168.30.1
traceroute 192.168.30.1
```

**Common exam points**

- Know how to read the routing table codes `C`, `L`, and `S`.
- Know that static routes are manually configured and do not advertise themselves unless redistributed into another routing process.
- Know that route selection starts with longest prefix match, not administrative distance.
- Know when to use next-hop-only, exit-interface-only, or fully specified static routes.
- Know that return routing is required for successful end-to-end communication.

**Quick memory aid**

- **Connected**: directly attached network.
- **Static**: manually entered route.
- **Default**: catch-all route for unknown destinations.
- **Recursive**: router resolves the next hop through another route.

**Lab checklist**

- Confirm all interfaces are up and addressed correctly.
- Confirm each router knows its directly connected networks.
- Add static routes for every remote LAN.
- Verify both directions of traffic.
- Use `show ip route static` and `traceroute` to confirm the path.
