## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Lab objective and routing context**

A default route sends traffic to a next hop when the router has no more specific match in its routing table. In an edge design, you use it to forward unknown destinations toward the ISP. This lab focuses on building that behavior on a customer edge router and verifying that traffic exits toward the ISP edge as expected.

In CCNA labs, the most common implementation is a static default route pointing to the ISP-facing next hop or outbound interface. You use this when the customer router has only one upstream path and does not need a dynamic routing protocol to learn the Internet route.

- **Default route**: `0.0.0.0/0`, the least specific IPv4 route.
- **Next hop**: the adjacent router IP used to forward packets.
- **Exit interface**: the local interface used to send packets out.
- **Longest prefix match**: the router chooses the most specific route before considering the default route.

**Typical lab topology**

A simple edge lab usually includes a LAN, a customer edge router, and an ISP edge router.

```text
PCs/LAN --- R1 (customer edge) --- R2 (ISP edge) --- upstream/Internet simulator
```

A practical addressing example:

- LAN: `192.168.10.0/24`
- R1 LAN interface: `192.168.10.1/24`
- R1 WAN interface: `203.0.113.2/30`
- R2 WAN interface: `203.0.113.1/30`
- ISP-side network: simulated or loopback destination behind R2

**Why you use a default route**

A default route reduces routing table size and simplifies edge configuration. Instead of learning or configuring many external routes, the customer router forwards all unknown traffic to the ISP edge. This is appropriate when the router is not expected to make detailed Internet routing decisions.

Use a default route when:

- You have a single upstream path.
- You want all unknown traffic to leave through the ISP edge.
- You are building a small branch or lab network.
- You need a simple fallback route for destinations not in the routing table.

> A default route does not replace specific routes. If a more specific route exists, the router uses that route instead of the default.

**Static default route configuration**

The most common CCNA configuration uses a static route with destination `0.0.0.0 0.0.0.0`.

```bash
R1(config)# ip route 0.0.0.0 0.0.0.0 203.0.113.1
```

You can also point the default route to an exit interface, which is more common on point-to-point links.

```bash
R1(config)# ip route 0.0.0.0 0.0.0.0 g0/1
```

For Ethernet multiaccess links, the next-hop form is usually preferred because it avoids unnecessary ARP behavior and is clearer in troubleshooting.

**Verification commands**

After configuration, verify that the route appears in the routing table and that traffic can reach the upstream side.

```bash
R1# show ip route
R1# show ip route static
R1# show running-config | include ^ip route
R1# ping 203.0.113.1
R1# traceroute 8.8.8.8
```

Expected routing table output includes a default route marked with `S*`:

```text
S* 0.0.0.0/0 [1/0] via 203.0.113.1
```

Key checks:

- `S` indicates a static route.
- `*` indicates candidate default route.
- `via 203.0.113.1` confirms the next hop.
- The route should appear only if the next hop is reachable or the interface is up, depending on the platform and configuration.

**Lab build steps**

1. Configure IP addressing on both router interfaces.
2. Bring interfaces up with `no shutdown`.
3. Add the static default route on the customer edge router.
4. Add a return route on the ISP edge router if you need end-to-end reachability to the LAN.
5. Verify with `show ip route`, `ping`, and `traceroute`.

Example interface configuration:

```bash
R1(config)# interface g0/0
R1(config-if)# ip address 192.168.10.1 255.255.255.0
R1(config-if)# no shutdown

R1(config)# interface g0/1
R1(config-if)# ip address 203.0.113.2 255.255.255.252
R1(config-if)# no shutdown
```

ISP edge example:

```bash
R2(config)# interface g0/0
R2(config-if)# ip address 203.0.113.1 255.255.255.252
R2(config-if)# no shutdown
```

If you want the ISP router to return traffic to the LAN, add a route back to the customer network:

```bash
R2(config)# ip route 192.168.10.0 255.255.255.0 203.0.113.2
```

**Return path and end-to-end testing**

A default route only handles outbound traffic from the customer edge. For successful two-way communication, the return path must also exist. In a lab, you often need a static route on the ISP edge back to the LAN or a simulated upstream route.

Test sequence:

- Ping the ISP next hop from R1.
- Ping a loopback or test address behind R2.
- Use `traceroute` to confirm the path leaves R1 through the ISP-facing interface.
- Confirm the ISP router can route replies back to the source network.

Example test commands:

```bash
R1# ping 203.0.113.1
R1# ping 198.51.100.10
R1# traceroute 198.51.100.10
R2# ping 192.168.10.1
R2# show ip route
```

**Common troubleshooting points**

Default route labs often fail because of basic Layer 1, Layer 2, or return-path issues rather than the route statement itself.

- **Interface down**: verify `no shutdown`, cable state, and correct interface selection.
- **Wrong next hop**: confirm the ISP edge IP is on the directly connected subnet.
- **Missing return route**: the outbound packet leaves, but replies cannot get back.
- **Mask mismatch**: ensure both ends use the correct subnet mask.
- **Overlapping routes**: a more specific route may override the default route.
- **NAT dependency**: if the lab includes Internet simulation, NAT may be required for private addresses to reach an upstream network.

Useful checks:

```bash
R1# show ip interface brief
R1# show cdp neighbors
R1# show arp
R1# show ip route 0.0.0.0
R1# debug ip routing
```

> Use `debug ip routing` carefully in a lab only. It can be noisy and is not appropriate on production devices.

**Default route behavior and route selection**

Routers use the most specific route first. If R1 has a route for `198.51.100.0/24`, traffic to that subnet uses the specific route instead of the default route. The default route is only used when no better match exists.

This matters in lab design because you may think the default route is broken when the router is actually using a more specific route learned from another source.

Route preference examples:

- Connected routes beat static default routes.
- Static routes beat many dynamically learned routes depending on administrative distance.
- A host route such as `/32` overrides the default route for that exact destination.

**Practical configuration patterns**

A next-hop static default route is the most common and easiest to verify:

```bash
ip route 0.0.0.0 0.0.0.0 203.0.113.1
```

An exit-interface default route can work well on point-to-point links:

```bash
ip route 0.0.0.0 0.0.0.0 serial0/0/0
```

A fully specified static route includes both exit interface and next hop, which can be useful on some multiaccess networks:

```bash
ip route 0.0.0.0 0.0.0.0 g0/1 203.0.113.1
```

**What to remember for the exam**

- A default route is `0.0.0.0/0`.
- It forwards traffic when no more specific route exists.
- Static default routes are common on edge routers.
- Verification requires both route-table checks and end-to-end testing.
- A working outbound route does not guarantee return traffic unless the upstream device also knows how to reach the source network.

**Quick lab checklist**

- Confirm interface addressing.
- Confirm interfaces are up.
- Configure the default route on the customer edge router.
- Add a return route on the ISP edge router if needed.
- Verify with `show ip route` and `ping`.
- Use `traceroute` to confirm the forwarding path.
