## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Floating static routes: the core idea**

A **floating static route** is a static route configured with an administrative distance (AD) higher than the primary route source. You use it as a backup path that only enters the routing table when the preferred route disappears.

For the same destination prefix, Cisco routers prefer the route with the lowest AD. That means:

- Connected routes usually win first.
- Static routes are preferred over most dynamic routes because static AD is typically lower.
- A floating static route must have a higher AD than the primary route it is backing up.

A common exam pattern is:

- Primary route: learned from a dynamic protocol or another static route with a lower AD.
- Backup route: static route with a higher AD, so it stays configured but inactive until needed.

**Administrative distance and route selection**

Administrative distance is the router’s trust value for the source of a route. Lower is better.

Typical Cisco defaults:

| Route source | Default AD |
|---|---:|
| Connected | 0 |
| Static | 1 |
| EIGRP internal | 90 |
| OSPF | 110 |
| RIP | 120 |

This matters because a floating static route must be set above the AD of the route it is meant to replace.

Example:

```bash
ip route 192.168.50.0 255.255.255.0 10.1.1.2 200
```

- `200` makes the route float behind most common dynamic routes.
- If an OSPF route to `192.168.50.0/24` exists, OSPF AD `110` wins.
- If OSPF disappears, the static route can be installed.

> A static route configured with the default AD of `1` is not a floating backup route. It is a primary static route unless another route source has an even lower AD, such as a connected route.

**How static routes appear in the routing table**

A static route can be:

- **Installed in the routing table** when the router can resolve the next hop or outgoing interface.
- **Configured but not active** when the route exists in the configuration but is not selected because a better route has a lower AD.
- **Removed from the active table** if its next hop becomes unreachable and the router can no longer resolve it.

You should distinguish between:

- The route being present in the running configuration.
- The route being present in the routing table (`show ip route`).
- The route being usable for forwarding.

Useful verification commands:

```bash
show running-config | include ^ip route
show ip route
show ip route static
show ip cef 192.168.50.0
```

**Static route types and forwarding behavior**

Cisco static routes are commonly configured in three forms:

| Type | Example | Forwarding behavior | Best use |
|---|---|---|---|
| Next-hop only | `ip route 192.168.50.0 255.255.255.0 10.1.1.2` | Router resolves the next hop before forwarding | Common on point-to-point links and many routed designs |
| Exit interface only | `ip route 192.168.50.0 255.255.255.0 GigabitEthernet0/0` | Router sends traffic out the interface and must still determine the destination MAC on Ethernet | Best on point-to-point links; discouraged on Ethernet multiaccess unless you understand the implications |
| Fully specified | `ip route 192.168.50.0 255.255.255.0 GigabitEthernet0/0 10.1.1.2` | Router knows both the exit interface and the next hop | Preferred on Ethernet multiaccess because it avoids ambiguity and unnecessary recursion |

A **fully specified static route** is often the safest choice on Ethernet because it tells the router exactly where to send the packet and which next hop to use. That reduces extra lookup work and avoids problems where the router may need to ARP for the destination host if only an exit interface is given on a multiaccess segment.

**Recursive lookup explained**

A **recursive lookup** happens when the router must look up the next hop of a route in the routing table before it can forward the packet.

This is common with next-hop-only static routes.

Example topology:

- R1 has a static route to `192.168.50.0/24` via next hop `10.1.1.2`
- R1 reaches `10.1.1.2` through `GigabitEthernet0/0`
- The router must first find how to reach `10.1.1.2`, then forward the packet

Step-by-step forwarding flow:

1. A packet arrives for `192.168.50.10`.
2. R1 matches the static route `192.168.50.0/24 via 10.1.1.2`.
3. R1 performs a recursive lookup for `10.1.1.2`.
4. The routing table says `10.1.1.2` is reachable out `GigabitEthernet0/0`.
5. R1 forwards the packet out `GigabitEthernet0/0` to the next-hop MAC address.

This is why the route is called recursive: the router uses one route to find another route before forwarding.

**Why recursive lookup matters on Ethernet**

On Ethernet multiaccess networks, a next-hop-only static route is usually fine, but the router still needs to resolve the next hop to a Layer 2 address. That means:

- The router may do an extra lookup in the routing table.
- The router then uses Address Resolution Protocol (ARP) to learn the next-hop MAC address.
- Forwarding is still correct, but the process is less direct than a fully specified route.

A fully specified route reduces ambiguity because the router already knows:

- The outgoing interface
- The next-hop IP address

That is especially useful when multiple next hops or multiple paths exist on a shared Ethernet segment.

**Practical packet-forwarding example**

Consider this design:

- R1 `G0/0` = `10.1.1.1/24`
- R2 `G0/0` = `10.1.1.2/24`
- R2 connects to `192.168.50.0/24`

R1 configuration:

```bash
ip route 192.168.50.0 255.255.255.0 10.1.1.2
```

What happens:

- R1 stores the route in the routing table if it can reach `10.1.1.2`.
- When a packet to `192.168.50.10` arrives, R1 looks up the destination prefix.
- R1 finds the static route and then recursively resolves `10.1.1.2`.
- R1 forwards the packet out `G0/0`.

If you instead configure:

```bash
ip route 192.168.50.0 255.255.255.0 GigabitEthernet0/0 10.1.1.2
```

R1 already knows the exit interface and next hop, so forwarding is more direct and predictable.

**Floating static route failover scenario**

A realistic use case is a branch router with:

- Primary WAN link using OSPF over MPLS
- Backup Internet VPN or secondary WAN link using a static route

Example:

```bash
! Primary route learned dynamically
! Backup route configured as floating static
ip route 10.20.30.0 255.255.255.0 172.16.1.2 200
```

If OSPF learns `10.20.30.0/24` with AD `110`, the router prefers OSPF.

If the OSPF path fails:

- The OSPF route is removed from the routing table.
- The floating static route with AD `200` becomes the best available route.
- Traffic shifts to the backup path.

This is useful when you want:

- Simple failover
- Predictable backup behavior
- No need to run a dynamic protocol on the backup link

**Why choose a floating static route over a dynamic backup**

You use a floating static route when the backup path should be:

- Simple
- Stable
- Easy to control
- Independent of dynamic routing convergence

This is common when:

- The backup link is expensive
- The backup path is only for emergencies
- You want to avoid dynamic routing overhead on the backup circuit

A dynamic backup can be more adaptive, but a floating static route is often easier to reason about in small networks and exam scenarios.

**Configuration examples**

Primary static route with backup floating static route:

```bash
! Primary static route
ip route 192.168.50.0 255.255.255.0 10.1.1.2

! Backup route with higher AD
ip route 192.168.50.0 255.255.255.0 10.2.2.2 200
```

If the primary route disappears, the backup route is installed.

Backup route using an exit interface and next hop:

```bash
ip route 192.168.50.0 255.255.255.0 GigabitEthernet0/1 10.2.2.2 200
```

This is often a better choice on Ethernet than exit-interface-only routing.

**How to verify failover**

Use these commands to confirm route behavior:

```bash
show ip route 192.168.50.0
show ip route static
show ip cef 192.168.50.10
traceroute 192.168.50.10
```

What to look for:

- The primary route appears first when available.
- The floating static route appears only when the preferred route is gone.
- `show ip cef` helps confirm the forwarding path.
- `traceroute` can show the path change after failover.

**Route installation and next-hop reachability**

A static route depends on the router being able to resolve the next hop.

Important nuance:

- The route is tied to reachability of the **next hop**, not the remote destination.
- If the next hop is reachable through a connected network, the static route can remain installed even if the far-end destination is down.
- If the next hop itself becomes unreachable, the router removes the route from the active table.

This is why a static route can still look healthy even when the remote LAN is broken. The router only knows whether it can reach the next hop, not whether the destination network behind that next hop is functioning end to end.

**Troubleshooting checklist**

When a static route does not behave as expected, check:

1. Whether the route is configured correctly.
2. Whether the next hop is reachable.
3. Whether another route has a lower AD.
4. Whether the exit interface is valid and up/up.
5. Whether the route is recursive and the recursive lookup succeeds.

Useful commands:

```bash
show ip interface brief
show ip route
show ip route <prefix>
ping <next-hop-ip>
show arp
show ip cef exact-route <source-ip> <destination-ip>
```

**Common exam traps**

- A static route with AD `1` is not floating.
- Metrics do not decide between routes from different sources the way AD does.
- An exit-interface-only route on Ethernet can cause less efficient forwarding behavior.
- A route can be installed even if the remote destination is unreachable, as long as the next hop is reachable.
- A backup route must have a higher AD than the primary route source.

**Quick mental model**

Think of static routing in three layers:

- **Selection**: lowest AD wins.
- **Resolution**: the router finds the next hop or exit interface.
- **Forwarding**: the packet is sent using the resolved path.

Recursive lookup belongs to the resolution step. Floating static routes belong to the selection step.

**Summary**

Floating static routes give you a controlled backup path by raising the AD above the primary route source. Recursive lookup explains how the router resolves a next hop before forwarding. For CCNA, focus on:

- AD determines which route is installed.
- Static routes can be next-hop-only, exit-interface-only, or fully specified.
- Recursive lookup is common with next-hop-only routes.
- Fully specified routes are often preferred on Ethernet.
- A floating static route becomes active only when the better route disappears.
