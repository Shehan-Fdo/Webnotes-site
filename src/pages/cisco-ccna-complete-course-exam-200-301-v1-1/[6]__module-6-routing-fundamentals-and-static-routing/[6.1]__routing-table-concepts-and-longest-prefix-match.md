**Routing table fundamentals**

A router forwards packets by comparing the destination IP address against routes in its routing information base (<abbr title="Routing Information Base">RIB</abbr>), then using the forwarding table for actual packet delivery. In Cisco IOS, the key exam idea is that the router does not simply pick the “best-looking” route by metric alone; it first finds all matching prefixes, then applies longest prefix match, then administrative distance (<abbr title="Administrative Distance">AD</abbr>) when multiple routing sources advertise the same prefix, and finally metric within the chosen routing source.

The routing table is the control-plane view of learned and connected routes. The forwarding table, often called the <abbr title="Forwarding Information Base">FIB</abbr> or Cisco Express Forwarding (<abbr title="Cisco Express Forwarding">CEF</abbr>) table, is the data-plane structure used to forward packets quickly. A route can appear in the RIB but not be installed in the FIB if it is not the active best route, if it is unresolved, or if the platform has not programmed it for forwarding yet.

**RIB vs FIB/CEF**

The RIB is where IOS stores candidate routes from connected networks, static routes, and dynamic routing protocols. It is the place you inspect with `show ip route`. The FIB/CEF table is optimized for packet forwarding and is what the router uses after the route decision is made.

- **RIB**:
  - Shows all known routes and the selected active route for each prefix.
  - Includes route source codes such as `C`, `L`, `S`, `O`, and `D`.
  - Is useful for troubleshooting why a route was or was not chosen.
- **FIB/CEF**:
  - Contains the forwarding decision for actual packet switching.
  - May not show every candidate route from the RIB.
  - Uses adjacency information to map next hops to outgoing interfaces efficiently.

A common troubleshooting pattern is this: you see a route in `show ip route`, but traffic still fails. That often means the route exists in the RIB, but the next hop is unreachable, the interface is down, or the FIB entry is not usable for forwarding.

> If a route is not installed in the forwarding table, packets do not get forwarded even if the prefix appears in the routing table.

**How Cisco IOS chooses a route**

Cisco IOS evaluates routes in a specific order:

1. It looks for routes whose prefixes match the destination IP address.
2. It chooses the route with the longest matching prefix.
3. If multiple routes to the same prefix exist from different sources, it chooses the one with the lowest AD.
4. If the same routing protocol offers multiple paths, it uses the protocol’s metric to choose the best path.
5. If equal-cost paths exist and the platform supports it, it may install multiple equal-cost routes for load balancing.

This order matters because AD and metric are not compared across unrelated prefixes. A `/24` route beats a `/16` route for an address inside the `/24`, even if the `/16` came from a “better” source. AD only matters after the router has already narrowed the candidates to the same prefix.

**Longest prefix match**

Longest prefix match means the router prefers the most specific route that still includes the destination address. A route with a longer mask has more network bits and therefore covers a smaller address range.

Example:

- `10.1.0.0/16` matches `10.1.5.10`
- `10.1.5.0/24` also matches `10.1.5.10`
- `10.1.5.10/32` matches only that host

The `/32` host route is the most specific and wins if it exists.

A simple visual view:

```text
Destination: 10.1.5.10
   |
   +-- matches 10.0.0.0/8
   |
   +-- matches 10.1.0.0/16
   |
   +-- matches 10.1.5.0/24   <-- chosen over /16 and /8
   |
   +-- matches 10.1.5.10/32  <-- chosen over all others
```

This is why route summarization can reduce table size but still allow more specific routes to override the summary when needed.

**Overlapping routes and exact selection order**

When multiple routes overlap in Cisco IOS, the router does not “average” them or compare all fields at once. It follows a layered decision process.

- First, it filters by destination match.
- Then it prefers the most specific prefix length.
- If several routes have the same prefix length, it compares AD.
- If the same protocol has multiple paths to that same prefix, it compares metric.
- If the metric is equal and the platform supports it, it can install multiple equal-cost paths.

Example overlap:

- `192.168.0.0/16` via OSPF
- `192.168.1.0/24` via static route
- `192.168.1.128/25` via EIGRP

For destination `192.168.1.200`:

- `/16` matches
- `/24` matches
- `/25` matches
- The `/25` wins because it is the longest prefix match, even if the `/24` has a lower AD or the `/16` has a better metric

This is a frequent exam trap: a “better” route source does not beat a more specific prefix.

**Administrative distance and metric**

Administrative distance is a trust value for the routing source. Lower is better. It is used only when multiple routes to the same prefix come from different sources.

Common Cisco IOS AD values:

| Route source | AD |
|---|---:|
| Connected | 0 |
| Static | 1 |
| EIGRP summary route | 5 |
| External BGP | 20 |
| EIGRP internal | 90 |
| OSPF | 110 |
| IS-IS | 115 |
| RIP | 120 |
| EIGRP external | 170 |
| Internal BGP | 200 |
| Unknown | 255 |

Metric is the protocol-specific cost used after AD has already selected the routing source or protocol. For example, OSPF uses cost, EIGRP uses composite metric, and RIP uses hop count.

Important CCNA caveat:

- **AD** decides between different routing sources for the same prefix.
- **Metric** decides between multiple paths within the same routing source or protocol.
- Neither one overrides longest prefix match.

**Cisco IOS route codes and `show ip route` interpretation**

The `show ip route` command is one of the most important troubleshooting tools for CCNA. It shows the route code, prefix, administrative distance, metric, next hop, and outgoing interface.

Example output:

```text
R1# show ip route 10.1.5.10
Routing entry for 10.1.5.0/24
  Known via "ospf 1", distance 110, metric 20
  Last update from 192.0.2.2 on GigabitEthernet0/0
  Routing Descriptor Blocks:
  * 192.0.2.2, from 192.0.2.2, 00:12:31 ago, via GigabitEthernet0/0
      Route metric is 20, traffic share count is 1
```

How to read it:

- `Routing entry for 10.1.5.0/24`
  - This is the prefix that matched the destination.
- `Known via "ospf 1"`
  - The route source is OSPF process 1.
- `distance 110`
  - OSPF’s AD is 110.
- `metric 20`
  - OSPF cost is 20.
- `Last update from 192.0.2.2`
  - The next-hop router that advertised the route.
- `via GigabitEthernet0/0`
  - The outgoing interface used to reach the next hop.
- `*`
  - Marks the active forwarding path.

A more complete table-style interpretation:

| Field | Meaning |
|---|---|
| Route code | Source of the route, such as `C`, `L`, `S`, `O`, `D`, `R` |
| Prefix | Network and mask being matched |
| Distance | Administrative distance |
| Metric | Protocol-specific cost |
| Next hop | IP address of the next router |
| Interface | Local exit interface |
| `*` | Candidate route installed for forwarding |

Common route codes:

- `C` = connected network
- `L` = local host route for the interface IP address
- `S` = static route
- `O` = OSPF
- `D` = EIGRP
- `R` = RIP

**Host routes and connected host entries**

Cisco IOS creates a connected network route for the subnet and a local host route for the interface address itself. This is a common CCNA detail.

Example:

If interface `GigabitEthernet0/0` has `192.168.10.1/24`, IOS typically installs:

- `C 192.168.10.0/24` as the connected network route
- `L 192.168.10.1/32` as the local host route

The `/32` local route is used to deliver traffic destined specifically to the router’s own interface IP. It is also why you may see host routes in the routing table even when you did not configure them manually.

Practical implication:

- Traffic to `192.168.10.1` matches the `L /32` entry.
- Traffic to `192.168.10.50` matches the `C /24` entry.
- The host route is more specific, so it wins for the interface IP itself.

**Worked example: reading a route entry**

Consider this output:

```text
R1# show ip route 172.16.20.77
Routing entry for 172.16.20.0/24
  Known via "static", distance 1, metric 0
  Routing Descriptor Blocks:
  * 172.16.10.2
      Route metric is 0, traffic share count is 1
```

Interpretation:

- The destination `172.16.20.77` matches `172.16.20.0/24`.
- The route source is static.
- AD is `1`, which is very trusted.
- Metric is `0` because static routes do not use a dynamic protocol metric.
- The next hop is `172.16.10.2`.
- The `*` indicates this route is active for forwarding.

If another route existed, such as `172.16.0.0/16` learned by OSPF, it would not matter for this destination because `/24` is more specific than `/16`.

**End-to-end packet lookup example**

Suppose a router has these routes:

```text
C    10.0.0.0/8 is directly connected, GigabitEthernet0/0
O    10.1.0.0/16 [110/20] via 192.0.2.2, GigabitEthernet0/1
S    10.1.5.0/24 [1/0] via 192.0.2.3
D    10.1.5.128/25 [90/30720] via 192.0.2.4, GigabitEthernet0/2
```

Now a packet arrives for `10.1.5.200`.

Selection process:

1. `10.0.0.0/8` matches.
2. `10.1.0.0/16` matches.
3. `10.1.5.0/24` matches.
4. `10.1.5.128/25` matches.
5. The `/25` is the longest prefix, so it wins.
6. AD and metric are not consulted because prefix length already decided the winner.

If the destination were `10.1.5.50`, the `/25` would not match, so the `/24` static route would win over the `/16` OSPF route because it is still the longest match.

This is the exact kind of sequence you should practice mentally during exam questions.

**Troubleshooting scenarios**

Routing problems often come from misunderstanding which table is being used or which route is actually active.

- A route appears in `show ip route`, but pings fail:
  - Check whether the next hop is reachable.
  - Check whether the outgoing interface is up/up.
  - Check whether the route is installed in CEF.
- A summary route exists, but a more specific route is missing:
  - The summary may be forwarding traffic to a different path than expected.
  - A missing specific route can cause traffic to follow the summary and black-hole.
- A static route does not appear active:
  - The next hop may be unresolved.
  - The exit interface may be down.
  - The route may be floating behind a lower AD route.
- A connected network is present, but the host IP is not reachable:
  - Verify the `/32` local route exists.
  - Confirm the interface address and mask are correct.

**Useful CLI commands**

```bash
show ip route
show ip route 192.168.1.10
show ip route connected
show ip route static
show ip cef
show ip cef 192.168.1.10
show ip interface brief
show interfaces gigabitEthernet0/0
```

What each helps you confirm:

- `show ip route` shows the RIB and route selection.
- `show ip cef` shows the forwarding decision.
- `show ip interface brief` confirms interface state.
- `show interfaces` helps verify Layer 1/2 and Layer 3 details.

**Static route example**

```bash
conf t
ip route 203.0.113.0 255.255.255.0 192.0.2.1
ip route 198.51.100.10 255.255.255.255 192.0.2.1
end
```

Notes:

- The first command creates a network static route.
- The second command creates a host route (`/32`).
- A host route is useful when you want to send traffic for one specific IP to a particular next hop.

**Route selection flow**

```text
Destination IP
   |
   v
Find all matching prefixes
   |
   v
Choose longest prefix match
   |
   v
If same prefix from different sources, choose lowest AD
   |
   v
If same protocol offers multiple paths, choose lowest metric
   |
   v
Install active route in RIB and program FIB/CEF
   |
   v
Forward packet out selected next hop/interface
```

**Exam-focused takeaways**

- **Longest prefix match** always comes before AD and metric.
- **AD** compares different routing sources only when they advertise the same prefix.
- **Metric** matters after the routing source is chosen.
- **RIB** is the routing decision table; **FIB/CEF** is the forwarding table.
- `L /32` entries represent the router’s own interface IP addresses.
- `show ip route` tells you what IOS believes is the best route; `show ip cef` tells you what it will actually forward.

**Quick mental model**

When you see a destination IP, ask yourself:

1. Which routes match it?
2. Which match is the most specific?
3. If there is a tie, which source has the lowest AD?
4. If still tied within one protocol, which path has the best metric?
5. Is the chosen route actually usable for forwarding?

That sequence is the core of routing table behavior on Cisco IOS.
