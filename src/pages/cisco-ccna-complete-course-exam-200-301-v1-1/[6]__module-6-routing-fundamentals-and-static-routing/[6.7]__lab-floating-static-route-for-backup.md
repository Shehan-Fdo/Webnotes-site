## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Floating static route overview**

A floating static route is a backup static route with a higher <abbr title="Administrative Distance">AD</abbr> than the primary route. You use it so the router prefers a dynamic route or a primary static route first, then “floats” to the backup only when the preferred route disappears from the routing table.

This lab topic matters because it tests two core routing ideas:

- **Route preference** is controlled by administrative distance, not just by the presence of a route.
- **Failover** can be achieved without a dynamic routing protocol if you design the backup route correctly.

In practice, you often use a floating static route when you want a simple, deterministic backup path to a WAN, ISP, or alternate internal link.

**Key concepts**

A static route is installed in the routing table only when the next hop is reachable. A floating static route is still a static route, but it is configured with a higher AD so it is less preferred than the primary route.

- **Administrative distance** ranks route trustworthiness; lower values win.
- **Primary route** is the route the router uses first.
- **Backup route** is the route that appears only when the primary route is unavailable.
- **Next hop** is the IP address or exit interface used to reach the destination network.

Common default administrative distances include:

- Connected: `0`
- Static: `1`
- EIGRP internal: `90`
- OSPF: `110`
- RIP: `120`

Because a normal static route has AD `1`, a floating static route must use a value greater than the competing route’s AD. For example, if an OSPF route is primary, a backup static route might use AD `120` or `200`.

**How floating static routes work**

The router compares all candidate routes for the same destination prefix. It installs the route with the lowest AD into the routing table.

If the primary route disappears because:

- the interface goes down,
- the next hop becomes unreachable,
- or the dynamic protocol withdraws the route,

then the router installs the floating static route instead.

This gives you predictable backup behavior without route redistribution or complex policy.

**Typical lab topology**

A simple lab usually includes:

- One LAN behind Router A
- Two possible paths to a remote network
- A primary path using a dynamic route or a static route with lower AD
- A backup path using a static route with higher AD

Example conceptual topology:

```text
LAN 192.168.10.0/24
        |
      R1
     /  \
    /    \
Primary  Backup
  |         |
 R2        R3
  |         |
Remote network 10.10.10.0/24
```

In this design, R1 prefers the primary route through R2. If that path fails, R1 uses the backup route through R3.

**Configuration example**

Suppose Router R1 must reach `10.10.10.0/24` through a primary next hop of `192.0.2.2`, with a backup next hop of `198.51.100.2`.

A primary static route might be:

```bash
ip route 10.10.10.0 255.255.255.0 192.0.2.2
```

A floating static backup route might be:

```bash
ip route 10.10.10.0 255.255.255.0 198.51.100.2 200
```

Here, the backup route has AD `200`, so it stays out of the routing table unless the primary route is removed.

If you want to use an exit interface instead of a next-hop IP, you can configure:

```bash
ip route 10.10.10.0 255.255.255.0 GigabitEthernet0/1 200
```

For CCNA labs, using a next-hop IP is often clearer because it avoids ambiguity on multiaccess networks.

**Verification commands**

Use these commands to confirm route selection and failover:

```bash
show ip route
show ip route 10.10.10.0
show running-config | include ^ip route
show ip interface brief
ping 10.10.10.1
traceroute 10.10.10.1
```

What to look for:

- The primary route appears in `show ip route` first.
- The backup static route does not appear while the primary route is active.
- After failure, the backup route appears with code `S` and the configured AD.
- `traceroute` changes to the alternate path after failover.

Example routing table behavior:

```text
S 10.10.10.0/24 [1/0] via 192.0.2.2
```

After the primary path fails:

```text
S 10.10.10.0/24 [200/0] via 198.51.100.2
```

The bracketed value shows `[administrative distance/metric]`. For static routes, the metric is typically `0`.

**Lab procedure**

1. Configure the primary route to the destination network.
2. Configure the backup static route with a higher AD.
3. Verify that only the primary route is installed.
4. Test reachability with `ping` and `traceroute`.
5. Shut down the primary interface or remove the primary route.
6. Confirm that the floating static route enters the routing table.
7. Restore the primary path and verify that the router prefers it again.

**Example failover test**

If the primary link is on `GigabitEthernet0/0`, you can simulate failure:

```bash
conf t
interface GigabitEthernet0/0
shutdown
end
```

Then verify:

```bash
show ip route
ping 10.10.10.1
```

Restore the interface:

```bash
conf t
interface GigabitEthernet0/0
no shutdown
end
```

The router should return to the lower-AD route automatically.

**Design considerations**

A floating static route is simple, but you should still design it carefully.

- **Choose the AD intentionally** so the backup route loses to the primary route.
- **Confirm reachability** of the backup next hop; a static route is only useful if the next hop can be reached.
- **Avoid equal-cost confusion** unless you specifically want load sharing.
- **Use tracking when needed** if you want the route to fail over based on reachability rather than only interface state.

A static route with a higher AD is not the same as a tracked route. A floating static route reacts to route preference changes, while object tracking can react to a monitored condition such as an IP SLA probe.

**Common mistakes**

- Setting the backup route AD too low, which causes it to compete with or replace the primary route too early.
- Using the wrong subnet mask, which creates a route to the wrong prefix.
- Pointing the backup route at an unreachable next hop.
- Forgetting that a connected interface route may override a static route because connected routes have AD `0`.
- Assuming the backup route will appear in the routing table before failure; it should remain hidden until needed.

> If both routes have the same prefix and the same AD, the router may not behave as a simple primary/backup design. Always set the backup route AD higher than the primary route’s AD.

**Practical example with dynamic primary route**

If OSPF learns `10.10.10.0/24` with AD `110`, a floating static backup route can be configured with AD `200`:

```bash
ip route 10.10.10.0 255.255.255.0 198.51.100.2 200
```

This route stays inactive while OSPF is present. If OSPF loses the route, the static backup takes over.

This pattern is useful when you want:

- a dynamic primary path for normal operation,
- a simple static backup for resilience,
- minimal configuration overhead.

**Troubleshooting checklist**

If failover does not work as expected, check the following:

- The destination prefix matches exactly on both routes.
- The backup route has a higher AD than the primary route.
- The next hop is reachable from the router.
- The outgoing interface is up/up.
- The primary route is actually removed from the routing table during the test.
- There is no more specific route overriding the static route.

Useful commands:

```bash
show ip route
show ip route static
show ip cef 10.10.10.0
show arp
show interfaces status
```

If the route exists but traffic still fails, verify Layer 2 and Layer 3 reachability to the next hop. A route can be present while the path still fails because of ACLs, incorrect VLANs, or missing return routes.

**Exam focus**

For CCNA-style questions, remember these points:

- A floating static route is a backup static route with a higher AD.
- Lower AD wins.
- Static routes default to AD `1`.
- The backup route is installed only when the preferred route is unavailable.
- Verification usually involves `show ip route` and interface state checks.

**Summary**

Floating static routes provide a straightforward backup mechanism for Cisco routers. You use them when you want deterministic failover without relying entirely on a dynamic routing protocol. The key is to set the backup route’s administrative distance higher than the primary route so the router keeps it in reserve until needed.
