## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Purpose of practice exams and scenario questions**

Practice exams are not just score checks. They train you to recognize question patterns, manage time, and verify network behavior under exam pressure. Scenario questions are especially valuable because CCNA expects you to interpret symptoms, identify the likely fault domain, and choose the best next step rather than memorize isolated facts.

A strong approach is to treat each question as a mini troubleshooting case. Ask yourself:

- What is the symptom?
- What changed?
- Which layer or device is most likely involved?
- What command would confirm the hypothesis?
- What is the least disruptive fix?

This mindset matters because many CCNA items are designed to test reasoning, not just recall.

**How to use practice exams effectively**

Use practice exams in stages. Early on, you are learning content and building recognition. Later, you are validating readiness and timing. The goal is not to memorize answers from a test bank, but to learn why each correct answer is correct and why the distractors are wrong.

A practical study loop looks like this:

1. Take a timed exam without notes.
2. Review every missed question.
3. Classify each miss by topic, not just by answer.
4. Revisit the related command, concept, or topology.
5. Retake similar questions after a delay.

Focus on patterns such as:

- Misreading the question stem.
- Confusing similar commands.
- Missing a keyword like `best`, `first`, `most likely`, or `least disruptive`.
- Choosing a technically true answer that does not solve the stated problem.

**Question types you should expect**

CCNA scenario questions often combine configuration knowledge with operational verification. You may see:

- Multiple-choice questions with one best answer.
- Multiple-response questions where more than one choice is valid.
- Drag-and-drop style matching.
- Exhibit-based questions using CLI output, topology diagrams, or configuration snippets.
- Troubleshooting prompts that ask for the next command or the most likely cause.

The exam often rewards careful reading. For example, a question may ask for the command that verifies a route exists, not the command that configures the route. Another may ask for the interface state, not the reason the interface is down.

**Core troubleshooting workflow**

A repeatable workflow helps you solve scenario questions quickly and consistently. Start broad, then narrow down.

1. Identify the affected scope.
   - Is the issue on one host, one VLAN, one interface, one routing domain, or the whole network?
2. Check the physical and data link layers first when connectivity is completely broken.
   - Look for link status, speed/duplex mismatches, VLAN assignment, trunking, and port security issues.
3. Verify Layer 3 addressing and routing.
   - Confirm IP address, subnet mask, default gateway, and route presence.
4. Validate name resolution and application reachability if basic IP connectivity works.
   - DNS issues can look like network failures even when routing is correct.
5. Compare working and nonworking devices.
   - Differences often reveal the fault faster than isolated inspection.

A useful mental model is: **physical → switching → routing → services**.

**Common CCNA scenario domains**

**Switching and VLANs**

VLAN and trunk problems are common because they create partial connectivity. A host may reach devices in the same VLAN but fail across VLANs, or one switch may not carry a VLAN at all.

Typical checks include:

- `show vlan brief`
- `show interfaces trunk`
- `show interfaces switchport`
- `show mac address-table`
- `show spanning-tree vlan <vlan-id>`

A sample verification sequence:

```bash
show vlan brief
show interfaces trunk
show interfaces gi0/1 switchport
show mac address-table dynamic
```

What you are looking for:

- The access port is in the correct VLAN.
- The trunk is up and allowing the expected VLANs.
- The native VLAN matches on both ends when required.
- MAC addresses are learned on the expected ports.
- Spanning Tree Protocol (STP) is not blocking the only path unexpectedly.

**Worked scenario: host cannot reach another VLAN**

Topology:

```text
PC1 --- SW1 --- R1 --- SW2 --- PC2
VLAN 10        Router-on-a-stick      VLAN 20
```

Symptoms:

- PC1 can ping other VLAN 10 hosts.
- PC1 cannot ping PC2 in VLAN 20.
- PC1 can ping its default gateway.

Likely causes:

- Missing router subinterface for VLAN 20.
- Trunk not allowing VLAN 20.
- Wrong default gateway on PC2.
- VLAN 20 not created on SW2.

Verification steps:

```bash
show interfaces trunk
show ip interface brief
show running-config interface g0/0.20
show vlan brief
```

If the router subinterface is missing, you might configure:

```bash
interface g0/0.20
 encapsulation dot1Q 20
 ip address 192.168.20.1 255.255.255.0
```

**Routing and IP connectivity**

For CCNA, focus on static routing, default routing, and OSPFv2. Scenario questions often ask you to identify why a route is missing or why two routers do not form adjacency.

Useful checks:

- `show ip route`
- `show ip interface brief`
- `show running-config | section router ospf`
- `show ip ospf neighbor`
- `show ip protocols`

A route verification example:

```bash
show ip route
show ip route static
show ip route ospf
```

What to confirm:

- The destination network exists in the routing table.
- The next hop is reachable.
- The outgoing interface is correct.
- The route has the expected administrative distance and metric.

**Worked scenario: static route is configured but traffic still fails**

Symptoms:

- Router R1 has a static route to `10.10.20.0/24`.
- Hosts behind R1 still cannot reach that network.
- The route appears in `show running-config`, but not in `show ip route`.

Likely causes:

- Next-hop IP is unreachable.
- Exit interface is down.
- The route points to the wrong subnet.
- A more specific route or better route is preferred.

Verification steps:

```bash
show ip interface brief
show ip route
show running-config | include ip route
ping <next-hop-ip>
traceroute <destination-ip>
```

If the next hop is unreachable, the static route will not install in the routing table. Fix the underlying reachability first.

**OSPFv2 troubleshooting**

OSPFv2 adjacency issues are a classic scenario topic. For CCNA, focus on the most common adjacency requirements:

- Same area on both neighbors.
- Matching hello and dead timers.
- Matching authentication settings, if used.
- Correct subnet and mask on the link.
- Proper network type and DR/BDR behavior on broadcast and NBMA links when relevant.

Useful commands:

- `show ip ospf neighbor`
- `show ip ospf interface brief`
- `show ip ospf interface <interface>`
- `show ip route ospf`
- `show running-config | section router ospf`

A basic OSPF verification sequence:

```bash
show ip ospf neighbor
show ip ospf interface brief
show ip route ospf
show running-config | section router ospf
```

**Worked scenario: OSPF neighbors stay in INIT or 2-WAY**

Symptoms:

- Two routers are connected.
- OSPF is enabled on both.
- Neighbor state never reaches FULL.

Possible causes:

- Mismatched area IDs.
- Mismatched hello/dead timers.
- Authentication mismatch.
- Interface is passive on one side.
- Layer 2 issue prevents bidirectional communication.

What to check:

```bash
show ip ospf neighbor
show ip ospf interface g0/0
show ip protocols
show ip interface brief
```

If the interface is passive, OSPF will not send hellos on that interface. If timers differ, neighbors may see each other but fail to form adjacency.

**Example OSPF configuration**

```bash
router ospf 1
 router-id 1.1.1.1
 network 192.168.10.0 0.0.0.255 area 0
 network 10.0.0.0 0.0.0.3 area 0
```

On newer IOS versions, interface-based OSPF configuration may also appear:

```bash
interface g0/0
 ip ospf 1 area 0
```

**Wireless verification**

For CCNA, wireless questions usually focus on concepts, SSIDs, security, and basic AP/controller behavior rather than obscure platform-specific commands. In scenario questions, expect to interpret a wireless client association problem, SSID mismatch, or security mismatch.

Common checks include:

- SSID broadcast and client selection.
- WPA2/WPA3 security settings.
- Authentication and passphrase correctness.
- RF interference or weak signal.
- VLAN mapping for the wireless network.

If a wireless client cannot connect, verify:

- The SSID name matches exactly.
- The security mode matches the client profile.
- The password or credentials are correct.
- The AP is reachable and broadcasting.
- The client is in range and not blocked by policy.

**Security and device hardening**

Scenario questions may ask you to identify insecure management access or the correct hardening step. Focus on SSH, local authentication, passwords, and service reduction.

Common verification commands:

```bash
show ip ssh
show running-config | include username|enable secret|transport input|login local
show running-config | section line vty
```

A typical SSH hardening sequence on many IOS devices is:

```bash
hostname R1
ip domain-name example.com
username admin secret Str0ngP@ssw0rd
crypto key generate rsa
ip ssh version 2

line vty 0 4
 login local
 transport input ssh
```

> The exact RSA key generation syntax can vary by IOS or IOS XE version, so verify platform-specific prompts if you are practicing on real hardware or a simulator.

**DHCP and DNS scenarios**

DHCP and DNS issues often appear as “the network is up, but the user still cannot work.” That usually means Layer 3 connectivity exists, but service configuration is wrong.

For DHCP, check:

- Scope or pool network statement.
- Default gateway option.
- Excluded addresses.
- Relay configuration with `ip helper-address`.
- Client lease status.

Useful commands:

```bash
show ip dhcp binding
show ip dhcp pool
show running-config | include ip helper-address
show ip interface brief
```

For DNS, check:

- Correct DNS server address on the client.
- Reachability to the DNS server.
- Correct hostname-to-IP mapping.
- Whether the issue is name resolution or actual connectivity.

**Worked scenario: client gets an IP address but cannot reach remote resources**

Symptoms:

- Client receives an address from DHCP.
- Client can ping the default gateway.
- Client cannot resolve hostnames.

Likely causes:

- DNS server option missing from the DHCP pool.
- Wrong DNS server address.
- DNS server unreachable from the client subnet.

Example DHCP pool:

```bash
ip dhcp pool USERS
 network 192.168.10.0 255.255.255.0
 default-router 192.168.10.1
 dns-server 8.8.8.8
```

**Subnetting and address planning questions**

Many scenario questions hide a subnetting problem inside a routing or VLAN issue. You may need to determine whether two hosts are truly in the same subnet, whether a mask is correct, or whether a gateway address belongs to the subnet.

Remember to verify:

- Network address.
- Broadcast address.
- Usable host range.
- Prefix length.
- Whether the default gateway is inside the subnet.

A quick example:

- `192.168.10.0/24` supports hosts `192.168.10.1` through `192.168.10.254`.
- `192.168.10.0/25` splits into two subnets, so `192.168.10.130` is not in the first half.

**Reading CLI output under pressure**

Scenario questions often include output that looks longer than it is. Train yourself to scan for the key lines first.

Look for:

- Interface status: `up/up`, `administratively down`, `down/down`
- IP address and mask
- Default gateway
- Route codes in `show ip route`
- OSPF neighbor state
- VLAN membership
- Trunk allowed VLANs
- Error messages in configuration output

Example:

```bash
show ip interface brief
```

Sample output interpretation:

```text
Interface              IP-Address      OK? Method Status                Protocol
GigabitEthernet0/0     192.168.1.1     YES manual up                    up
GigabitEthernet0/1     unassigned      YES unset  administratively down down
```

This tells you:

- `g0/0` is operational.
- `g0/1` is shut down, so it cannot pass traffic until enabled.

**Exam-day strategy for scenario questions**

Use a disciplined approach so you do not waste time.

1. Read the last line of the question first.
   - This tells you what the exam wants you to solve.
2. Highlight keywords mentally.
   - Words like `best`, `first`, `most likely`, and `not` change the answer.
3. Eliminate obviously wrong choices.
   - Remove answers that solve a different problem.
4. Match the symptom to the layer.
   - If the interface is down, do not jump to routing.
5. Choose the least disruptive valid fix.
   - CCNA often prefers the simplest correct action.

A good rule is to avoid overengineering. If a host cannot ping its gateway, the answer is usually not DNS, and if two routers do not form OSPF adjacency, the answer is usually not a static route.

**Mini case study: end-to-end troubleshooting**

Scenario:

A user in VLAN 30 can reach the local printer but cannot access a server in VLAN 40. The user can ping the default gateway. Other VLANs work normally.

Likely reasoning:

- Local switching is fine because same-VLAN traffic works.
- The access port is probably correct because the gateway is reachable.
- The issue is likely in inter-VLAN routing, trunking, or the VLAN 40 path.

Verification path:

```bash
show vlan brief
show interfaces trunk
show ip route
show ip interface brief
```

Possible findings:

- VLAN 40 is missing from the trunk allowed list.
- The router subinterface for VLAN 40 is missing.
- The server has the wrong default gateway.
- The switchport toward the server is in the wrong VLAN.

This kind of question rewards layered thinking. You do not need to guess every possible fault; you need to identify the most probable one from the symptoms.

**Mini case study: router adjacency and route propagation**

Scenario:

Two routers are connected over `10.0.0.0/30`. OSPF is configured, but one router does not learn the remote LAN.

Verification:

```bash
show ip ospf neighbor
show ip route
show running-config | section router ospf
show ip ospf interface g0/0
```

Possible outcomes:

- No neighbor entry: check area, timers, interface state, and subnet.
- Neighbor is FULL but route is missing: check network statements, passive interfaces, and whether the LAN is included in OSPF.
- Route exists but traffic still fails: check ACLs, return path, and host gateway settings.

**What to memorize versus what to reason through**

Memorize the commands you use most often, but do not rely on memorization alone. Scenario questions often require you to combine facts.

Memorize:

- `show ip interface brief`
- `show running-config`
- `show ip route`
- `show vlan brief`
- `show interfaces trunk`
- `show ip ospf neighbor`

Reason through:

- Whether the symptom is Layer 2, Layer 3, or service-related.
- Which device is the likely fault domain.
- Which command confirms the hypothesis fastest.
- Whether the proposed fix matches the problem statement.

**Practical lab habits**

If you practice in Packet Tracer, GNS3, or real gear, build the habit of verifying after every change. Do not assume a configuration worked just because the command was accepted.

A simple verification cycle:

1. Configure one change.
2. Save or note the change.
3. Verify with a show command.
4. Test connectivity.
5. Roll back if the result is not what you expected.

Useful connectivity tests:

```bash
ping 192.168.10.1
ping 8.8.8.8
traceroute 10.10.20.10
```

If ping works but traceroute fails at a certain hop, the issue may be beyond the local device. If ping to the gateway fails, stay local and inspect interface and VLAN settings first.

**Common traps in practice questions**

- Assuming the first visible error is the root cause.
- Ignoring the direction of traffic flow.
- Confusing configuration presence with operational success.
- Overlooking return-path routing.
- Missing a shutdown interface or passive interface.
- Choosing a command that displays information instead of the command that fixes the issue.

**Quick reference checklist**

- Confirm the symptom and scope.
- Check interface status and addressing.
- Verify VLANs, trunks, and switchport mode.
- Inspect routing tables and OSPF neighbors.
- Validate DHCP, DNS, and gateway settings.
- Read the question carefully for the exact task.
- Prefer the simplest correct fix.

**Final exam mindset**

Practice exams are most useful when they teach you how to think. The best candidates do not just know commands; they know how to map symptoms to layers, eliminate distractors, and verify the network step by step. If you can explain why an answer is correct and what evidence supports it, you are ready for scenario-based CCNA questions.
