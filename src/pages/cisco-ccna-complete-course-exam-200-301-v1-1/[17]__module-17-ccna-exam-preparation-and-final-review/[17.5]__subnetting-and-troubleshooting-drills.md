## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Subnetting fundamentals**

Subnetting is the process of dividing a larger network into smaller logical networks. On the CCNA, you need to move quickly between prefix length, subnet mask, network address, broadcast address, usable host range, and host count. The goal is not just memorization; you need to recognize whether an address belongs to a subnet, whether a mask is valid for the design, and whether a route or interface configuration matches the intended boundary.

For IPv4, the number of usable hosts is usually `2^h - 2`, where `<var>h</var>` is the number of host bits. The exceptions matter:

- **/31** is valid only on point-to-point links per RFC 3021, where both addresses can be used and there is no broadcast address.
- **/32** represents a single host route or host address, commonly used for loopbacks and specific host routes.
- **/30** is common for point-to-point links because it provides 2 usable addresses.
- **/24** is a common LAN size, but you should not assume it is always the right choice.

A quick mental method is to identify the block size from the interesting octet. If the mask is `255.255.255.192`, the block size is `64`, so subnets begin at `0, 64, 128, 192`. That lets you determine the network and broadcast addresses without a calculator.

**IPv4 prefix calculations**

Use prefix math to answer exam questions quickly. The prefix length tells you how many bits are fixed for the network portion.

| Prefix | Mask | Total addresses | Usable hosts | Typical use |
|---|---:|---:|---:|---|
| /24 | 255.255.255.0 | 256 | 254 | Small LAN |
| /25 | 255.255.255.128 | 128 | 126 | Split a /24 into two equal subnets |
| /26 | 255.255.255.192 | 64 | 62 | Department or VLAN segment |
| /27 | 255.255.255.224 | 32 | 30 | Small user group |
| /28 | 255.255.255.240 | 16 | 14 | Small service subnet |
| /29 | 255.255.255.248 | 8 | 6 | Small infrastructure segment |
| /30 | 255.255.255.252 | 4 | 2 | Point-to-point link |
| /31 | 255.255.255.254 | 2 | 2 | Point-to-point only |
| /32 | 255.255.255.255 | 1 | 1 | Host route / loopback |

A subnet boundary example:

- `192.168.10.130/25`
  - Mask: `255.255.255.128`
  - Block size: `128`
  - Network: `192.168.10.128`
  - Broadcast: `192.168.10.255`
  - Usable range: `192.168.10.129` to `192.168.10.254`

If a host is configured as `192.168.10.50/25`, it classifies `192.168.10.130` as remote because its local subnet is `192.168.10.0/25`. That does not automatically mean the gateway is wrong; it means the host’s mask causes it to make a different local-versus-remote decision than intended.

**Mixed IPv4 drill patterns**

Practice these edge cases until they are automatic:

- Determine whether an address is a network, broadcast, or host address.
- Identify the subnet that contains a given IP.
- Compare two hosts and decide whether they are in the same subnet.
- Recognize when a mask creates an invalid design for the link type.
- Calculate the next subnet boundary for VLSM planning.

Examples:

- `10.1.1.64/26`
  - Network address: `10.1.1.64`
  - Broadcast address: `10.1.1.127`
  - First host: `10.1.1.65`
  - Last host: `10.1.1.126`

- `172.16.5.31/27`
  - Block size: `32`
  - Subnets: `0, 32, 64, ...`
  - `31` falls in `0–31`, so:
    - Network: `172.16.5.0`
    - Broadcast: `172.16.5.31`
    - Usable hosts: `172.16.5.1` to `172.16.5.30`

- `192.0.2.10/30`
  - Network: `192.0.2.8`
  - Broadcast: `192.0.2.11`
  - Usable hosts: `192.0.2.9` and `192.0.2.10`

- `198.51.100.4/31`
  - Valid only on a point-to-point link
  - Both addresses are usable when the devices support RFC 3021 behavior

- `203.0.113.77/32`
  - Single host route
  - Often used for loopbacks, router IDs, or specific reachability entries

**VLSM planning**

<abbr title="Variable Length Subnet Masking">VLSM</abbr> lets you allocate different subnet sizes from the same address block. This is essential when one VLAN needs 120 hosts, another needs 50, and a point-to-point link needs only 2 addresses. You conserve space by assigning the largest subnet first and then fitting smaller subnets into the remaining space.

A practical planning workflow:

1. List requirements from largest to smallest.
2. Choose the smallest prefix that satisfies each requirement.
3. Allocate the largest subnet first.
4. Move to the next available boundary.
5. Verify that each subnet starts on a valid boundary.

Example allocation from `10.10.0.0/24`:

- 100-host VLAN → `/25`
- 50-host VLAN → `/26`
- 20-host VLAN → `/27`
- Point-to-point link → `/30`

Possible layout:

- `10.10.0.0/25`
- `10.10.0.128/26`
- `10.10.0.192/27`
- `10.10.0.224/30`

This leaves room for future growth and avoids wasting large blocks on small segments.

**IPv6 subnetting essentials**

IPv6 subnetting is simpler in one sense because the standard LAN subnet is usually `/64`, but you still need to understand prefix boundaries and address planning. IPv6 does not use broadcast addresses. Instead, it uses multicast and neighbor discovery.

Common prefix meanings:

- **/64** is the standard subnet size for most LANs and SLAAC.
- **/127** is commonly used on point-to-point links in modern designs.
- **/128** identifies a single interface address or host route.

A `/64` gives an enormous host space, so host-count calculations are not practical in the same way as IPv4. Instead, focus on subnet count and prefix boundaries.

Example subnet planning with `2001:db8:acad::/48`:

- `2001:db8:acad:1::/64`
- `2001:db8:acad:2::/64`
- `2001:db8:acad:3::/64`

Each `/64` changes the fourth hextet. That is the most common exam pattern.

Useful IPv6 examples:

- `2001:db8:acad:10::1/64`
  - Subnet: `2001:db8:acad:10::/64`
- `2001:db8:acad:10::1/127`
  - Point-to-point subnet with two addresses
- `2001:db8:acad:10::1/128`
  - Single host/interface route

**Subnetting verification commands**

Use device commands to confirm what the router or switch believes the subnet is. This is especially important when the addressing plan looks correct on paper but traffic still fails.

```bash
show ip interface brief
show running-config interface g0/0
show ip route
show ipv6 interface brief
show ipv6 route
```

Helpful checks:

- `show ip interface brief` confirms interface status and assigned IPv4 addresses.
- `show running-config interface` shows the configured mask, encapsulation, and helper address.
- `show ip route` reveals connected, static, and learned routes.
- `show ipv6 interface brief` confirms IPv6 addressing and interface state.
- `show ipv6 route` verifies connected and static IPv6 routes.

**Layer 3 troubleshooting mindset**

Troubleshooting works best when you isolate the failure domain. Start with the most local evidence you can trust, then expand outward. Do not assume the first symptom points to the root cause.

A practical sequence is:

1. Confirm the interface is up/up.
2. Verify the address and mask on the device.
3. Check the routing table for the destination prefix.
4. Confirm the next hop is reachable.
5. Test name resolution if the IP path works but the hostname fails.
6. Check ACLs, NAT, DHCP, and VLAN configuration if the path still fails.

This sequence is more reliable than jumping straight to a ping test because ping failure can result from many unrelated issues, including ARP problems, ACLs, missing routes, or host stack misconfiguration.

**Common IPv4 connectivity faults**

Duplicate IP addresses can create intermittent or confusing failures. One host may appear reachable for a while, then traffic shifts or ARP entries change and connectivity breaks.

Symptoms of a duplicate IP:

- Intermittent ping replies
- ARP table changes for the same IP
- One device steals traffic from another
- Log messages about duplicate addresses on some platforms

Incorrect static routes are another common issue. A route may point to the wrong next hop, use the wrong exit interface, or be missing entirely.

Example static route:

```bash
ip route 10.20.30.0 255.255.255.0 192.0.2.2
```

If the next hop is wrong, the route may appear in the configuration but traffic will still fail. Verify with:

```bash
show ip route static
show ip cef 10.20.30.1
ping 192.0.2.2
traceroute 10.20.30.1
```

Missing default routes are especially common on edge routers and small branch devices. If a router knows local networks but not the Internet or upstream path, check for a default route:

```bash
show ip route | include 0.0.0.0
ip route 0.0.0.0 0.0.0.0 192.0.2.1
```

Longest-prefix match matters when multiple routes exist for the same destination. The router chooses the most specific route, not necessarily the first one configured. For example, a `/24` route overrides a `/16` route for addresses inside that `/24`.

**Routing verification drills**

Use these commands to inspect routing behavior:

```bash
show ip route
show ip route 10.1.1.0
show ip protocols
show cdp neighbors
show lldp neighbors
```

What to look for:

- `show ip route` confirms whether the destination is connected, static, or learned by a routing protocol.
- `show ip route <prefix>` helps you see the exact route selected.
- `show ip protocols` shows routing protocol timers, networks, and redistribution details.
- `show cdp neighbors` and `show lldp neighbors` help you identify the adjacent device and interface.
- If a route is missing, check whether the interface is in the correct subnet and whether the routing protocol advertises that network.

A useful exam pattern is a mismatch between the interface subnet and the advertised network statement. If the interface is `192.168.1.1/24` but the routing process only advertises `192.168.0.0/16` incorrectly or omits the network entirely, the neighbor may never learn the route.

**Inter-VLAN routing troubleshooting**

You should know both router-on-a-stick and SVI-based inter-VLAN routing on multilayer switches.

Router-on-a-stick uses one physical router interface with multiple subinterfaces and 802.1Q tagging. Each VLAN gets a subinterface with its own IP address.

```bash
interface g0/0.10
 encapsulation dot1Q 10
 ip address 192.168.10.1 255.255.255.0

interface g0/0.20
 encapsulation dot1Q 20
 ip address 192.168.20.1 255.255.255.0
```

Common issues:

- Missing `encapsulation dot1Q <vlan-id>`
- Wrong VLAN ID on the subinterface
- Trunk not allowing the VLAN
- Wrong default gateway on the host
- Native VLAN mismatch

SVI-based inter-VLAN routing uses switched virtual interfaces on a multilayer switch.

```bash
ip routing

interface vlan 10
 ip address 192.168.10.1 255.255.255.0
 no shutdown

interface vlan 20
 ip address 192.168.20.1 255.255.255.0
 no shutdown
```

Common SVI issues:

- `ip routing` is missing, so the switch does not route between VLANs.
- The SVI is down because no active ports exist in that VLAN.
- The VLAN does not exist in the VLAN database.
- The host default gateway points to the wrong SVI address.

Verification commands:

```bash
show vlan brief
show interfaces trunk
show ip interface brief
show running-config interface vlan 10
show ip route
```

If an SVI is administratively up but line protocol is down, check whether the VLAN has at least one active access port or trunk carrying that VLAN.

**ACL troubleshooting**

<abbr title="Access Control List">ACL</abbr> issues are a frequent cause of “network is up but traffic fails” problems. An ACL can block traffic even when routing is correct.

Key ACL facts:

- ACLs are processed top-down.
- The first matching entry wins.
- There is an implicit `deny any` at the end.
- Placement matters: standard ACLs are usually placed close to the destination, while extended ACLs are usually placed close to the source.

Example extended ACL:

```bash
ip access-list extended WEB-FILTER
 permit tcp 192.168.10.0 0.0.0.255 host 203.0.113.10 eq 80
 permit tcp 192.168.10.0 0.0.0.255 host 203.0.113.10 eq 443
 deny ip any any
```

Verification:

```bash
show access-lists
show ip interface g0/0
show running-config | section access-list
```

Troubleshooting tips:

- Check whether the ACL is applied inbound or outbound on the correct interface.
- Confirm the wildcard mask matches the intended subnet.
- Remember that an ACL can block return traffic if it is placed incorrectly.
- Use sequence numbers on named ACLs to insert or adjust entries cleanly.

A common exam scenario is a host that can ping the gateway but cannot reach a server because an ACL blocks TCP port `80` or `443`. Another common issue is forgetting that ICMP may be denied while TCP still works, or vice versa.

**NAT troubleshooting**

<abbr title="Network Address Translation">NAT</abbr> problems often look like routing failures. The inside host may reach the default gateway but not the outside network.

A typical PAT (Port Address Translation) configuration is:

```bash
access-list 1 permit 192.168.10.0 0.0.0.255

interface g0/0
 ip nat inside

interface g0/1
 ip nat outside

ip nat inside source list 1 interface g0/1 overload
```

Important verification points:

- The inside interface must be marked `ip nat inside`.
- The outside interface must be marked `ip nat outside`.
- The ACL must match the inside source addresses.
- The overload statement belongs in the `ip nat inside source ... overload` command.
- The router needs a route to the outside network and usually a default route upstream.

Useful commands:

```bash
show ip nat translations
show ip nat statistics
show running-config | include ip nat
```

Common failures:

- Wrong ACL matches no inside hosts.
- Inside/outside roles are reversed.
- Missing default route to the ISP or upstream router.
- NAT configured on the wrong interface.
- Return traffic is blocked by an ACL.

**DHCP troubleshooting**

<abbr title="Dynamic Host Configuration Protocol">DHCP</abbr> issues are common in routed VLAN environments. The problem is often not the client itself but the path between the client subnet and the DHCP server.

If the DHCP server is on another network, you usually need a helper address on the SVI or router interface:

```bash
interface vlan 10
 ip helper-address 192.168.100.10
```

The helper address forwards DHCP client broadcasts to the server as unicast. In Cisco environments, this is often the real fix when clients in a VLAN fail to obtain addresses.

Verification commands:

```bash
show running-config interface vlan 10
show ip interface vlan 10
show ip dhcp binding
show ip dhcp pool
```

Common DHCP issues:

- Missing `ip helper-address` on the client VLAN interface.
- Wrong default gateway in the DHCP pool.
- Exhausted address pool.
- Server unreachable from the relay interface.
- ACLs blocking the DHCP relay path or server response path.
- Incorrect VLAN membership on the access port.

> On routed VLANs, do not assume the problem is simply UDP port blocking. The more common issue is missing relay configuration, wrong helper address, or a path problem between the relay interface and the DHCP server.

**DNS troubleshooting**

<abbr title="Domain Name System">DNS</abbr> failures can look like general connectivity problems when IP reachability is actually fine. If you can ping an IP address but not a hostname, test name resolution separately.

Useful checks:

```bash
ping 8.8.8.8
ping www.example.com
show hosts
```

Common causes:

- Wrong DNS server address from DHCP.
- Missing DNS server configuration on the host.
- ACL blocking DNS queries.
- No route to the DNS server.
- Incorrect local DNS suffix or search domain.

If IP connectivity works but DNS fails, verify the host’s configured DNS server before changing routing or NAT.

**Practical troubleshooting scenarios**

Scenario 1: Host cannot reach another subnet

- Host: `192.168.10.50/25`
- Gateway: `192.168.10.1`
- Destination: `192.168.10.130`

Likely issue:

- The host mask makes `192.168.10.130` appear remote, so the host sends traffic to the gateway.
- If the gateway or routing path is wrong, traffic fails.
- If the host mask is incorrect for the design, fix the mask first.

Scenario 2: VLAN clients cannot get DHCP addresses

- Access ports are in the correct VLAN.
- SVI is up.
- Clients still show APIPA or self-assigned addresses.

Likely checks:

1. Verify `ip helper-address` on the SVI.
2. Confirm the DHCP server is reachable from the relay interface.
3. Check the DHCP pool scope and exclusions.
4. Confirm no ACL blocks the relay or server response.

Scenario 3: Internet access fails after NAT deployment

- Inside hosts can reach the gateway.
- Outside hosts are unreachable.
- NAT translations do not appear.

Likely checks:

- Verify `ip nat inside` and `ip nat outside`.
- Confirm the ACL matches the inside subnet.
- Confirm the overload command is present.
- Verify the router has a default route.

Scenario 4: Inter-VLAN routing fails on a multilayer switch

- VLANs exist.
- Access ports are assigned correctly.
- Hosts cannot reach the SVI gateway.

Likely checks:

- `ip routing` is enabled.
- The SVI is not shutdown.
- The VLAN has active ports.
- Trunks carry the VLAN if needed.
- The host default gateway matches the SVI IP.

**Verification command set for CCNA drills**

Use this command set as a quick reference during practice:

```bash
show ip interface brief
show ipv6 interface brief
show vlan brief
show interfaces trunk
show cdp neighbors
show lldp neighbors
show ip route
show ipv6 route
show ip protocols
show access-lists
show ip nat translations
show ip dhcp binding
show running-config | section interface
```

These commands help you answer three questions quickly:

- Is the interface up and addressed correctly?
- Does the device know a route to the destination?
- Is something filtering, translating, or relaying the traffic?

**Exam-style reasoning tips**

- Start with the addressing plan before blaming routing.
- Check the mask when a host can reach some devices but not others in the same physical network.
- Use longest-prefix match when multiple routes overlap.
- Remember that ACLs can permit one protocol and block another.
- Treat DHCP and DNS as dependent services that often fail because of relay, routing, or ACL issues.
- For IPv6, verify both the address and the prefix length; a wrong prefix can break neighbor discovery and routing assumptions.
- For point-to-point links, use `/30` or `/31` in IPv4 only when the design supports it, and use `/127` commonly in IPv6 point-to-point designs.

**Quick drill checklist**

- Calculate network, broadcast, and host range for a given IPv4 prefix.
- Identify whether an address is a network, host, or broadcast address.
- Allocate subnets using VLSM from largest to smallest requirement.
- Verify router and switch interfaces with `show ip interface brief` and `show ipv6 interface brief`.
- Check routing with `show ip route` and `show ipv6 route`.
- Confirm inter-VLAN routing with router-on-a-stick and SVI configurations.
- Diagnose ACL, NAT, DHCP, and DNS failures using targeted verification commands.
- Use `show cdp neighbors` and `show lldp neighbors` to identify adjacent devices during topology troubleshooting.
