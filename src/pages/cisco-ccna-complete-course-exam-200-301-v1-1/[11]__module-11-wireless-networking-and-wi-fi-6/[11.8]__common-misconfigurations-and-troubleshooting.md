## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Troubleshooting mindset**

Troubleshooting is a process of reducing uncertainty. You start with symptoms, verify the scope, isolate the layer or service involved, and then test one change at a time. In CCNA scenarios, the fastest path is usually to confirm physical connectivity, interface state, VLAN membership, IP addressing, routing, and policy controls in that order.

A practical workflow is:

1. Identify the failure domain: one host, one VLAN, one subnet, one site, or one service.
2. Check the most basic indicators first: link lights, interface status, and addressing.
3. Compare the intended design with the actual configuration.
4. Verify the control plane and data plane separately.
5. Make one change, then retest.

Useful baseline commands include:

```bash
show interfaces status
show ip interface brief
show running-config
show vlan brief
show interfaces counters errors
show cdp neighbors detail
show lldp neighbors detail
```

**Layer 1 and Layer 2 misconfigurations**

Many CCNA troubleshooting problems begin with simple Layer 1 or Layer 2 issues. A port can appear administratively up but still fail to pass traffic because of speed/duplex mismatch, bad cabling, wrong transceiver type, or a security feature that shut the port down.

Common Layer 1 checks:

- **Physical link state**: verify the interface is not `administratively down`, `down/down`, or `err-disabled`.
- **Cable and transceiver**: inspect copper patching, fiber polarity, SFP/SFP+ seating, and whether the optic type matches the link.
- **Speed and duplex**: mismatches can cause late collisions, CRC errors, and poor throughput on older or manually configured links.
- **Interface counters**: rising CRC, input errors, frame errors, or runts often point to cabling or duplex problems.
- **Optical levels**: on supported platforms, check receive/transmit power and transceiver diagnostics.

Example verification:

```bash
show interfaces gigabitEthernet1/0/1
show interfaces gigabitEthernet1/0/1 counters errors
show interfaces transceiver detail
show interfaces status
```

A useful interpretation pattern:

- `up/up` with errors increasing suggests a data integrity problem, not a pure link failure.
- `down/down` usually points to cabling, optics, or the far end being down.
- `err-disabled` often indicates a protection feature or security violation.

> For a pure link-down issue, `show interfaces status` is often the quickest first check on a switch, while `show ip interface brief` is better when you need a fast view of Layer 3 interface state.

**VLAN and trunk misconfigurations**

VLAN problems are among the most common CCNA switching issues. The port may be physically up, but the host still cannot communicate because the access VLAN is wrong, the trunk is not carrying the expected VLAN, or the native VLAN does not match on both ends.

Typical access-port mistakes:

- The port is assigned to the wrong access VLAN.
- The VLAN does not exist on the switch.
- The host is connected to a trunk port by mistake.
- The port is in the correct VLAN, but the host has the wrong IP subnet or default gateway.

Typical trunk mistakes:

- The port is not actually trunking.
- The allowed VLAN list does not include the needed VLAN.
- The native VLAN differs on each side.
- The neighboring device expects a trunk but the port is configured as access.

On modern Catalyst platforms used in CCNA-level labs, trunking is typically fixed to IEEE 802.1Q, so you usually verify trunk mode and allowed VLANs rather than selecting an encapsulation type.

Verification commands:

```bash
show vlan brief
show interfaces trunk
show interfaces gigabitEthernet1/0/24 switchport
show running-config interface gigabitEthernet1/0/24
```

Example access-port configuration:

```bash
interface gigabitEthernet1/0/10
 switchport mode access
 switchport access vlan 20
 spanning-tree portfast
 spanning-tree bpduguard enable
```

Example trunk configuration:

```bash
interface gigabitEthernet1/0/24
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30
 switchport trunk native vlan 99
```

A common exam pattern is a host that can link up but cannot reach the gateway. In that case, check:

- The access VLAN on the switchport.
- Whether the VLAN exists and is active.
- Whether the SVI for that VLAN is up.
- Whether the trunk path carries that VLAN end to end.
- Whether the host IP settings match the VLAN subnet.

**Spanning Tree Protocol and port protection issues**

<abbr title="Spanning Tree Protocol">STP</abbr> prevents Layer 2 loops, but protection features can also block traffic or place ports into `err-disabled`. In CCNA troubleshooting, you should recognize when a port is intentionally blocked versus when it is misconfigured.

Key concepts:

- **Root bridge selection**: the switch with the lowest bridge priority wins; if priorities tie, the lowest MAC address wins.
- **PortFast**: intended for edge ports connected to end devices, not switches.
- **BPDU Guard**: shuts down a PortFast-enabled port if it receives a Bridge Protocol Data Unit (BPDU).
- **Root Guard**: prevents a downstream switch from becoming root on a designated port.
- **Loop Guard**: helps protect against unidirectional link failures that could create loops.

Common misconfigurations:

- Enabling PortFast on a port connected to another switch.
- Enabling BPDU Guard on a port where a switch or hub may be connected.
- Applying Root Guard on the wrong uplink and unexpectedly blocking a valid topology.
- Forgetting that an access port can become `err-disabled` after receiving BPDUs.

Verification commands:

```bash
show spanning-tree
show spanning-tree interface gigabitEthernet1/0/10 detail
show interfaces status err-disabled
show logging
```

Example of a protected access port:

```bash
interface range gigabitEthernet1/0/1 - 24
 switchport mode access
 spanning-tree portfast
 spanning-tree bpduguard enable
```

> If a port goes `err-disabled`, do not assume the cable is bad. Check the logs first to identify the triggering feature.

**IPv4 routing and default gateway issues**

Routing problems often look like “the network is down” when the real issue is a missing route, wrong next hop, or incorrect default gateway. On routers, verify both the routing table and the interface status. On hosts, verify the default gateway and subnet mask.

Common IPv4 issues:

- The host default gateway is wrong or missing.
- The router interface is in the wrong subnet.
- A static route points to the wrong next hop.
- The next hop is not reachable through a connected interface.
- The return path is missing, causing one-way communication.
- An ACL blocks the traffic even though routing is correct.

Useful commands:

```bash
show ip interface brief
show ip route
show running-config | section ip route
ping
traceroute
```

Example static route:

```bash
ip route 192.168.20.0 255.255.255.0 10.1.1.2
```

For static routes, the next hop must be reachable according to the router’s connected topology. If you use only a next-hop address, the router must be able to resolve that address through a connected interface or a valid recursive lookup. On point-to-point links, many engineers prefer specifying the exit interface or using an exit-interface plus next hop when appropriate.

Example with exit interface:

```bash
ip route 192.168.20.0 255.255.255.0 gigabitEthernet0/0/0 10.1.1.2
```

A useful troubleshooting pattern is to compare the route you expect with the route actually installed in the routing table. If the route is missing, the problem may be administrative distance, next-hop reachability, interface state, or a typo in the prefix or mask.

**IPv6 misconfigurations**

IPv6 troubleshooting requires you to check addressing, neighbor discovery, router advertisements, and the method used to obtain the address. A host may have an IPv6 address but still fail to communicate if the prefix, default gateway, or neighbor cache is wrong.

Common IPv6 issues:

- The interface lacks a global unicast address.
- The default gateway is missing or incorrect.
- The router advertisement is not being received.
- The host is using the wrong method: SLAAC, DHCPv6, or static.
- Duplicate Address Detection (DAD) detects a duplicate address.
- Neighbor Discovery fails because of Layer 2 or ACL issues.

Key terms:

- **SLAAC**: Stateless Address Autoconfiguration, where hosts build their own address from router advertisements.
- **DHCPv6**: Dynamic Host Configuration Protocol for IPv6, used for address assignment or additional options.
- **RA**: Router Advertisement, sent by routers to advertise prefixes and flags.
- **DAD**: Duplicate Address Detection, used to ensure an IPv6 address is unique.

Verification commands:

```bash
show ipv6 interface brief
show ipv6 route
show ipv6 neighbors
show running-config | section ipv6
```

On routers, inspect the IPv6 interface configuration and RA behavior:

```bash
interface gigabitEthernet0/0
 ipv6 address 2001:db8:10:1::1/64
 ipv6 nd other-config-flag
 ipv6 nd managed-config-flag
```

Troubleshooting guidance:

- If hosts should use **SLAAC**, confirm the router interface sends RAs and advertises the correct prefix.
- If hosts should use **DHCPv6**, confirm the correct RA flags are set and the DHCPv6 server or relay is reachable.
- If a host has a link-local address but no global address, check RA reception and prefix advertisement.
- If communication fails between IPv6 neighbors, check neighbor cache entries and Layer 2 reachability.

**Wireless troubleshooting**

Wireless issues often involve registration, authentication, RF coverage, or controller connectivity. A client may see the SSID but still fail to join because of security mismatch, DHCP failure, or AP/controller problems.

Common wireless problem areas:

- The AP is not joined to the controller.
- The AP has no CAPWAP connectivity to the wireless LAN controller.
- The SSID is broadcast but the security settings do not match the client.
- The client receives an IP address but cannot reach the network due to VLAN or ACL issues.
- RF interference, low signal strength, or channel overlap causes unstable connectivity.

CAPWAP stands for Control and Provisioning of Wireless Access Points. If an AP cannot join, verify:

- IP addressing on the AP.
- Default gateway and DNS if discovery depends on name resolution.
- Reachability to the controller.
- Controller discovery method.
- Time, certificates, and software compatibility when relevant.

Useful verification commands on wireless infrastructure vary by platform, but common checks include:

```bash
show wireless summary
show ap summary
show ap join stats detailed <ap-name>
show logging
```

For client-side checks, verify:

- SSID name and security type.
- Passphrase or 802.1X credentials.
- DHCP lease acquisition.
- Signal strength and roaming behavior.

**ACL placement and direction**

An <abbr title="Access Control List">ACL</abbr> can make a network appear broken even when routing and switching are correct. The most common mistakes are placing the ACL on the wrong interface, applying it in the wrong direction, or forgetting that ACLs are processed top-down with an implicit deny at the end.

Common ACL errors:

- Applying the ACL inbound when the traffic needs to be filtered outbound, or vice versa.
- Filtering the wrong source or destination subnet.
- Forgetting that extended ACLs are usually placed close to the source.
- Blocking return traffic unintentionally.
- Using the wrong wildcard mask.

Verification commands:

```bash
show access-lists
show ip interface gigabitEthernet0/0
show running-config | include access-group
```

Example:

```bash
ip access-list extended WEB-FILTER
 permit tcp 192.168.10.0 0.0.0.255 host 203.0.113.10 eq 443
 deny ip any any log

interface gigabitEthernet0/1
 ip access-group WEB-FILTER in
```

When troubleshooting ACLs, test with a known-good flow and inspect hit counts. If the deny line increments unexpectedly, the ACL is matching traffic you did not intend to block.

**NAT verification and common mistakes**

<abbr title="Network Address Translation">NAT</abbr> problems often show up as inside hosts reaching local resources but failing to reach the internet. The most common issues are incorrect inside/outside designation, missing route to the outside, or a NAT rule that does not match the source subnet.

Common NAT mistakes:

- The inside interface is not marked `ip nat inside`.
- The outside interface is not marked `ip nat outside`.
- The ACL used for NAT does not match the inside subnet.
- The overload rule is missing.
- The router lacks a default route toward the ISP.
- Return traffic cannot find the translated address because the outside path is wrong.

Verification commands:

```bash
show ip nat translations
show ip nat statistics
show running-config | include ip nat
show ip route
```

Example PAT configuration:

```bash
access-list 1 permit 192.168.10.0 0.0.0.255

interface gigabitEthernet0/0
 ip nat inside

interface gigabitEthernet0/1
 ip nat outside

ip nat inside source list 1 interface gigabitEthernet0/1 overload
```

Troubleshooting approach:

1. Confirm the inside host has the correct default gateway.
2. Confirm the router has a route to the outside.
3. Confirm the NAT ACL matches the inside source addresses.
4. Generate traffic and check `show ip nat translations`.
5. Verify the outside return path.

**DHCP issues**

<abbr title="Dynamic Host Configuration Protocol">DHCP</abbr> failures are often caused by relay problems, scope exhaustion, or interface/VLAN mismatches. A client may show `0.0.0.0` or an APIPA address on IPv4 if it cannot reach a DHCP server.

Common DHCP problems:

- The client is in the wrong VLAN.
- The DHCP server is not reachable.
- The relay address is missing on the SVI or router interface.
- The DHCP pool is exhausted or excluded addresses are too broad.
- The default gateway option is wrong.
- The server is blocked by an ACL.

Verification commands:

```bash
show ip dhcp binding
show ip dhcp pool
show running-config | section dhcp
show ip interface brief
```

Example relay configuration:

```bash
interface vlan 20
 ip address 192.168.20.1 255.255.255.0
 ip helper-address 10.10.10.10
```

If the client cannot obtain a lease, verify the broadcast domain first, then the relay path, then the server pool. On a switch, confirm the access port is in the correct VLAN and the SVI for that VLAN is up if the switch is acting as the default gateway.

**Security features that cause unexpected outages**

Some security features are designed to stop attacks, but they can also create outages when misconfigured. CCNA troubleshooting should include these because they often produce `err-disabled` ports or silent drops.

Important features:

- **Port security**: can shut down a port if the MAC address limit is exceeded or a different MAC appears.
- **DHCP snooping**: blocks untrusted DHCP server responses on untrusted ports.
- **Dynamic ARP Inspection (DAI)**: drops ARP packets that do not match the DHCP snooping binding table.
- **IP Source Guard**: blocks traffic from hosts with unexpected IP/MAC bindings.

Verification commands:

```bash
show port-security interface gigabitEthernet1/0/10
show ip dhcp snooping
show ip arp inspection
show ip verify source
show logging
```

Example port security:

```bash
interface gigabitEthernet1/0/10
 switchport mode access
 switchport port-security
 switchport port-security maximum 2
 switchport port-security violation shutdown
 switchport port-security mac-address sticky
```

If a host suddenly loses connectivity after a security change, check whether the port is `err-disabled`, whether the binding table is populated, and whether the port is trusted or untrusted as intended.

**Troubleshooting with show commands and outputs**

Good troubleshooting depends on reading command output, not just running commands. You should learn to map symptoms to the right verification output.

Example 1: interface is up but host cannot reach the gateway

```bash
Switch# show interfaces status
Gi1/0/10  connected  20  a-full a-100 10/100/1000BaseTX
Gi1/0/24  connected  trunk a-full a-1000 10/100/1000BaseTX

Switch# show vlan brief
VLAN Name                             Status    Ports
20   USERS                            active    Gi1/0/11
30   VOICE                            active    Gi1/0/10
```

Interpretation:

- The port is connected, so Layer 1 is fine.
- The port is in VLAN 30, but the host should be in VLAN 20.
- The fix is to correct the access VLAN.

Example 2: trunk is up but one VLAN is missing

```bash
Switch# show interfaces trunk

Port        Mode         Encapsulation  Status        Native vlan
Gi1/0/24    on           802.1q         trunking      99

Port        Vlans allowed on trunk
Gi1/0/24    10,20,30

Port        Vlans allowed and active in management domain
Gi1/0/24    10,20,30

Port        Vlans in spanning tree forwarding state and not pruned
Gi1/0/24    10,30
```

Interpretation:

- VLAN 20 is allowed and active, but it is not forwarding.
- The issue may be STP state, pruning, or a downstream problem.
- Check spanning tree and the far-end trunk configuration.

Example 3: err-disabled due to BPDU Guard

```bash
Switch# show interfaces status err-disabled
Port      Name  Status       Reason
Gi1/0/10        err-disabled  bpduguard

Switch# show logging
%SPANTREE-2-BLOCK_BPDUGUARD: Received BPDU on port Gi1/0/10 with BPDU Guard enabled. Disabling port.
```

Interpretation:

- The port received BPDUs, so it is connected to another switch or a device generating BPDUs.
- Remove PortFast/BPDU Guard from that uplink or connect the correct end device.

**A practical troubleshooting sequence**

Use a repeatable sequence so you do not skip obvious checks.

1. Verify the symptom and scope.
2. Check physical link and interface state.
3. Confirm VLAN, trunk, or SVI configuration.
4. Verify IP addressing and default gateway.
5. Check routing, ACLs, and NAT if Layer 3 is involved.
6. Inspect logs for security features or err-disable causes.
7. Test with `ping`, `traceroute`, and a known-good endpoint.
8. Change one variable at a time and retest.

**Quick reference of common commands**

| Area | Useful commands | What you learn |
|---|---|---|
| Interface state | `show interfaces status`, `show ip interface brief` | Link, admin state, VLAN, IP state |
| Errors | `show interfaces counters errors`, `show interfaces` | CRC, input errors, duplex symptoms |
| VLANs/trunks | `show vlan brief`, `show interfaces trunk` | VLAN membership and trunk forwarding |
| STP | `show spanning-tree`, `show logging` | Root role, blocking, protection events |
| Routing | `show ip route`, `show ipv6 route` | Installed routes and next hops |
| ACLs | `show access-lists`, `show ip interface` | Match counts and applied direction |
| NAT | `show ip nat translations`, `show ip nat statistics` | Translation activity and overload state |
| DHCP | `show ip dhcp binding`, `show ip dhcp pool` | Lease status and pool health |
| Wireless | `show ap summary`, `show wireless summary` | AP join and controller status |

**Exam-focused takeaways**

- Start with the simplest layer that can explain the symptom.
- A port being `up` does not mean the service is correct.
- VLAN, trunk, and SVI mismatches are frequent causes of “connected but unreachable” problems.
- STP protection features can intentionally shut down ports; always check logs and err-disabled status.
- Routing issues often come from missing return paths, wrong next hops, or incorrect default gateways.
- ACLs, NAT, DHCP snooping, and DAI can all create symptoms that look like basic connectivity failures.
- For IPv6, always consider RA behavior, DHCPv6 flags, neighbor discovery, and duplicate address checks.
- For wireless, verify AP registration, CAPWAP connectivity, and client authentication before blaming RF.
