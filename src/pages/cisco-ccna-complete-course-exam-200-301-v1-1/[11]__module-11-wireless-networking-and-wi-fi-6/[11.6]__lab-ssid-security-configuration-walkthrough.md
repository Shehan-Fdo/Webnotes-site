## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**SSID security configuration overview**

An <abbr title="Service Set Identifier">SSID</abbr> is the wireless network name clients see when they scan for Wi-Fi networks. In Cisco environments, SSID security is not just about naming the network; it is about how the SSID is broadcast, what authentication method protects it, and how client traffic is mapped after association.

For CCNA-level work, you should understand the difference between:

- **Autonomous AP** configuration, where the access point handles SSID, security, and bridging locally.
- **Controller-based WLAN** configuration, where the wireless LAN controller defines the WLAN and applies policy centrally.
- **Security mode selection**, such as open, <abbr title="Wi-Fi Protected Access 2">WPA2</abbr>-Personal, or enterprise authentication with <abbr title="Extensible Authentication Protocol">EAP</abbr> and <abbr title="Remote Authentication Dial-In User Service">RADIUS</abbr>.

A good lab workflow is to define the SSID, apply the security policy, map the SSID to the correct VLAN or interface, and then verify client association and traffic flow. That sequence is common in practice, but the exact object model changes by platform.

**Core wireless security concepts**

Wireless security starts with the authentication method. The SSID itself is only the network label; the security policy determines who can join and how traffic is protected.

- **Open authentication** allows association without a shared key, but it provides no real access protection by itself.
- **WPA2-Personal** uses a pre-shared key <abbr title="Pre-Shared Key">PSK</abbr> and is common in small environments and labs.
- **WPA2-Enterprise** uses 802.1X authentication with a RADIUS server and is preferred in managed enterprise networks.
- **Encryption** is typically <abbr title="Advanced Encryption Standard">AES</abbr>-CCMP for WPA2; avoid older <abbr title="Temporal Key Integrity Protocol">TKIP</abbr> unless you are dealing with legacy compatibility.

Hidden SSIDs are often misunderstood. Disabling SSID broadcast does not make a network secure; it only removes the name from normal beacon advertisements. Clients and management traffic can still reveal the SSID, so security must come from authentication and encryption, not concealment.

**Autonomous AP SSID configuration**

On an autonomous Cisco access point, you usually configure the SSID under global configuration, then bind it to the radio interface and apply the security settings. The radio interface also needs the correct role and band context.

A typical 2.4 GHz example looks like this:

```cisco
conf t
!
dot11 ssid CORP-WIFI
 authentication open
 authentication key-management wpa version 2
 wpa-psk ascii 0 StrongPass123
 guest-mode
!
interface Dot11Radio0
 no shutdown
 station-role root
 encryption mode ciphers aes-ccm
 ssid CORP-WIFI
!
interface FastEthernet0
 no shutdown
!
interface BVI1
 ip address 192.168.10.2 255.255.255.0
 no shutdown
end
write memory
```

Key points in that example:

- **`dot11 ssid`** creates the SSID profile.
- **`guest-mode`** enables SSID broadcast in beacon frames.
- **`station-role root`** is required on many autonomous AP platforms so the radio operates as an access point.
- **`ssid CORP-WIFI`** binds the SSID to the radio interface.
- **`encryption mode ciphers aes-ccm`** enables WPA2-compatible AES encryption.

> Hidden SSIDs are not a security control. If you disable `guest-mode`, the SSID is simply not advertised in normal beacons, but clients can still reveal it during association and probe activity.

If you want a more complete autonomous AP design, you often map the wireless network to a bridge group and a BVI <abbr title="Bridge Virtual Interface">BVI</abbr> for Layer 3 management or local routing. In many lab scenarios, the AP bridges wireless clients into the wired LAN through the BVI or a VLAN subinterface.

**Autonomous AP verification**

After configuration, verify the SSID, radio state, and security settings.

```cisco
show running-config | section dot11
show dot11 ssid
show interfaces dot11radio 0
show dot11 associations
show ip interface brief
```

What to look for:

- The SSID appears in the running configuration.
- The radio is up and not administratively shut down.
- The security mode matches the intended design.
- Associated clients appear in the association table.

If clients cannot connect, check these common issues:

- The SSID is not bound to the radio interface.
- The radio is missing `station-role root`.
- The encryption or PSK does not match the client.
- The AP interface or BVI is down.
- The client is trying to use the wrong band or unsupported security mode.

**Controller-based WLAN configuration**

On a Cisco wireless LAN controller, the SSID is usually represented as a WLAN profile. The controller handles the SSID definition, security policy, and interface mapping separately from the access point radio.

A typical AireOS-style workflow is:

```cisco
config wlan create 10 CORP-WIFI CORP-WIFI
config wlan security wpa enable 10
config wlan security wpa wpa2 enable 10
config wlan security wpa akm psk enable 10
config wlan security wpa akm psk set-key ascii 10 StrongPass123
config wlan interface 10 CORP-VLAN10
config wlan enable 10
```

Important notes:

- **WLAN ID** identifies the wireless profile on the controller.
- **WPA2** must be enabled explicitly when using WPA2-Personal.
- **PSK** is configured as part of the WPA key management settings.
- **Interface mapping** sends client traffic to the correct VLAN or dynamic interface.

The exact syntax can vary by controller software release, but the concept stays the same: create the WLAN, define security, map it to an interface, and enable it.

For Catalyst 9800-style controllers, the model is more policy-driven and may use different commands or GUI objects. The exam focus is usually on understanding the separation of SSID, security policy, and VLAN/interface mapping rather than memorizing one exact syntax family.

**Controller-based verification**

Use show commands to confirm the WLAN is active and mapped correctly.

```cisco
show wlan summary
show wlan id 10
show client summary
show client detail <client-mac>
show interface summary
```

You want to confirm:

- The WLAN is enabled.
- The security settings match the intended authentication method.
- The WLAN is tied to the correct interface or policy.
- Clients receive the expected IP addressing and can reach the network.

If the SSID appears but clients fail to pass traffic, the issue is often not the SSID itself. Check the mapped interface, VLAN assignment, DHCP availability, and upstream switching configuration.

**WPA2-Personal vs WPA2-Enterprise**

Choosing the right security model depends on the environment.

| Feature | WPA2-Personal | WPA2-Enterprise |
|---|---|---|
| Authentication | Shared passphrase | 802.1X with RADIUS |
| User control | Low | High |
| Scalability | Small to medium | Medium to large |
| Credential changes | Must change PSK on all devices | Change per-user credentials or certificates |
| Typical use | Home, lab, small office | Enterprise, campus, managed access |

Practical guidance:

- Use **WPA2-Personal** when you need a simple shared key and the environment is small.
- Use **WPA2-Enterprise** when you need per-user authentication, better auditing, and centralized access control.
- Avoid relying on SSID hiding as a substitute for proper authentication.

**Example enterprise design with RADIUS**

A WPA2-Enterprise deployment usually includes a wireless controller, an authentication server, and a VLAN or policy mapping for authorized users.

```cisco
aaa new-model
radius server RADIUS1
 address ipv4 192.168.20.10 auth-port 1812 acct-port 1813
 key RadiusSecret123
!
aaa authentication dot1x default group radius
aaa authorization network default group radius
```

In this design:

- The controller forwards authentication requests to the RADIUS server.
- The RADIUS server validates the user or device.
- The controller applies the resulting authorization and network policy.

At the client side, the user typically enters credentials or uses a certificate-based profile, depending on the organization’s 802.1X method.

**Practical lab workflow**

Use a repeatable workflow so you do not miss required pieces.

1. Define the SSID name and decide whether it should be broadcast.
2. Choose the security model: open, WPA2-Personal, or WPA2-Enterprise.
3. Configure the radio or WLAN object with the correct security settings.
4. Map the SSID to the correct bridge group, VLAN, or controller interface.
5. Enable the WLAN or radio.
6. Verify association, IP addressing, and end-to-end connectivity.

This workflow is useful, but do not treat it as universal across every Cisco platform. Autonomous APs, AireOS controllers, and Catalyst 9800 controllers use different configuration models, so the order and command names can change.

**Troubleshooting checklist**

When an SSID is visible but clients cannot connect, isolate the failure by layer.

- **Broadcast layer**
  - Confirm the SSID is enabled and, if desired, broadcast with `guest-mode`.
  - Check that the radio is up and on the correct band.
- **Authentication layer**
  - Confirm the PSK matches exactly, including case and special characters.
  - For enterprise WLANs, verify RADIUS reachability and shared secrets.
- **Association layer**
  - Check that the SSID is bound to the radio or WLAN profile.
  - Confirm the security mode matches the client profile.
- **Network access layer**
  - Verify VLAN mapping, DHCP, and upstream trunking.
  - Confirm the client receives the expected IP address and default gateway.

Useful commands for troubleshooting include:

```cisco
show logging
show dot11 associations
show wlan summary
show client detail <client-mac>
show radius statistics
show ip dhcp binding
```

**Exam-focused takeaways**

For CCNA, focus on the relationship between SSID, security, and traffic mapping rather than memorizing every platform-specific command.

- An SSID is the wireless network name.
- Security determines how clients authenticate and encrypt traffic.
- Autonomous APs configure SSIDs locally on the AP.
- Controllers define WLANs centrally and map them to interfaces or policies.
- Hidden SSIDs do not provide meaningful security.
- WPA2-Personal uses a shared key; WPA2-Enterprise uses 802.1X and RADIUS.

If you can explain how a client discovers the SSID, authenticates, associates, and receives network access, you understand the core of this lab topic.
