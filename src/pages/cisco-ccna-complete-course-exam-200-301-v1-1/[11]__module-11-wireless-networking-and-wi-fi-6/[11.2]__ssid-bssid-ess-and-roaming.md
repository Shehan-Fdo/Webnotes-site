**Wireless identity basics**

In 802.11 wireless networking, you identify the network at several layers. The **SSID** is the human-readable network name you join, while the **BSSID** identifies the specific access point (AP) radio transmitting that wireless cell. Multiple APs can advertise the same SSID so clients see one logical network, even though they may connect to different radios as they move.

A useful way to think about it is:

- **SSID** = the network name you select
- **BSSID** = the radio’s MAC address for that specific AP cell
- **BSS** = one AP radio and its associated clients
- **ESS** = multiple BSSs using the same SSID and acting like one extended network

**SSID, BSSID, BSS, and ESS**

An **SSID** (Service Set Identifier) is the broadcast name of a WLAN. It is not a security feature by itself and does not uniquely identify a single AP. In a campus or office, you often reuse the same SSID across many APs so users can move without manually changing networks.

A **BSSID** (Basic Service Set Identifier) is typically the **MAC address of the AP radio** in infrastructure mode. This is the value clients use to distinguish one AP cell from another, even when the SSID is identical. In an **ad hoc** network, the BSSID is a special case: it is not tied to a physical AP radio because there is no infrastructure AP; the network uses a locally generated identifier for the peer-to-peer cell.

A **BSS** (Basic Service Set) is the coverage area of one AP radio and the clients associated to it. An **ESS** (Extended Service Set) is a group of BSSs connected by a distribution system, usually the wired LAN and/or wireless controller infrastructure, that share the same SSID and security settings.

| Term | Meaning | Typical exam clue |
|---|---|---|
| SSID | Wireless network name | What users choose from the Wi-Fi list |
| BSSID | AP radio identifier, usually the radio MAC address | What changes when you roam to another AP |
| BSS | One AP radio plus its associated clients | One wireless cell |
| ESS | Multiple BSSs with the same SSID | One logical WLAN across many APs |

**How clients use these identifiers**

When you connect to Wi-Fi, your device scans for beacons and probe responses. It sees the SSID, then compares available BSSIDs for that SSID and chooses one based on signal quality, security, band preference, roaming policy, and vendor-specific logic.

- The client usually decides which AP to join.
- The AP does not normally “push” the client to roam in basic deployments.
- In controller-based designs, the infrastructure can influence roaming behavior with features such as band steering, load balancing, and roaming assistance.

**Roaming versus reassociation and reauthentication**

Roaming is the overall user experience of moving from one AP to another while staying on the same WLAN. At the protocol level, several related events can occur:

- **Reassociation** is the 802.11 management action where the client moves its association from one AP/BSSID to another AP/BSSID within the same ESS.
- **Reauthentication** is the security process that may happen after reassociation, depending on the authentication method and timing.
- **Roaming** is the broader term that includes the client’s decision to switch APs and the resulting reassociation, with possible reauthentication.

In practical terms:

- If the client stays on the same SSID but changes BSSID, it is roaming.
- If the client must perform a full 802.1X/EAP authentication again, roaming is slower.
- If the client can reuse cached security state, roaming is faster and voice/video quality improves.

> For exam purposes, do not treat roaming, reassociation, and reauthentication as the same thing. Roaming is the user-visible event; reassociation and reauthentication are protocol steps that may or may not both occur.

**Real-world roaming behavior**

Roaming is primarily **client-driven**. Different devices make different decisions:

- Some clients hold onto a weak AP too long, causing sticky-client behavior.
- Some clients roam aggressively and switch APs sooner.
- Voice devices often roam faster than laptops because they prioritize latency and packet loss.
- Controller-managed WLANs can improve consistency, but the client still makes the final association decision in most cases.

In a well-designed ESS, you want overlapping coverage, similar transmit power, and matching SSID/security settings so clients can roam without dropping sessions.

**Roaming optimization features**

Cisco and other vendors use optional enhancements to improve roaming. These are helpful, but they are **not core requirements** for basic roaming.

- **802.11r** Fast BSS Transition (FT) reduces roaming delay by allowing faster key negotiation.
- **802.11k** Radio Resource Management helps clients learn about nearby APs and channels more efficiently.
- **802.11v** Wireless Network Management can suggest better roaming targets or assist with network-directed behavior.

At CCNA level, remember:

- These standards are **optional enhancements**.
- They improve roaming performance.
- They do not replace the need for a properly designed ESS with overlapping coverage.

A simple memory aid:

- **11r** = faster security transition
- **11k** = neighbor information
- **11v** = network assistance and steering

**Controller-based WLANs and roaming**

In a Cisco wireless LAN controller (WLC) design, you often create one WLAN/SSID and map it to multiple APs. The controller applies the same SSID, security, and policy across the AP group, so the client sees one logical network while moving between APs.

Example topology:

- One SSID: `CorpWiFi`
- Multiple APs: `AP1`, `AP2`, `AP3`
- Same security policy: WPA2/WPA3 Enterprise or PSK, depending on design
- Same VLAN mapping or policy profile, depending on platform

This design creates an ESS. As the client moves, it associates to the strongest AP BSSID that advertises the same SSID and compatible security settings.

**Cisco WLC configuration example**

The exact commands vary by controller platform and software release, but the concept is consistent: create a WLAN/SSID, enable it, and map it to the desired APs or AP groups.

Example on a Cisco AireOS-style controller:

```text
(Cisco Controller) > config wlan create 10 CorpWiFi CorpWiFi
(Cisco Controller) > config wlan security wpa2 enable 10
(Cisco Controller) > config wlan security wpa akm psk set-key ascii 0 MySecretPass123 10
(Cisco Controller) > config wlan enable 10
(Cisco Controller) > config wlan interface 10 corp-vlan
(Cisco Controller) > show wlan summary
```

Example on a Cisco Catalyst 9800-style controller, conceptually:

```text
conf t
wlan CorpWiFi 10 CorpWiFi
 security wpa psk set-key ascii 0 MySecretPass123
 no shutdown
exit

wireless profile policy CorpPolicy
 vlan 20
 no shutdown
exit
```

On Catalyst 9800 platforms, WLANs are commonly tied to policy profiles and tags rather than the older AireOS model. The exam focus is the concept: one SSID can be broadcast by many APs, and the controller centralizes policy.

**Practical client-side verification**

When troubleshooting, you often verify which BSSID the client is using and whether roaming is occurring as expected.

Examples:

```bash
# Linux
nmcli dev wifi
iw dev wlan0 link
```

```powershell
# Windows
netsh wlan show interfaces
```

```text
# Cisco IOS AP or controller-style checks
show wireless client summary
show wireless client mac-address <client-mac>
show ap summary
show wlan summary
```

Useful things to check:

- Current SSID
- Current BSSID
- Signal strength / RSSI
- Security type
- Roam count or last roam time
- Whether the client is stuck on a distant AP

**Troubleshooting scenario**

A user reports that voice calls drop when walking from one end of the office to the other. You check the client and see it remains associated to a far AP even when a closer AP is available.

Likely causes:

- AP coverage overlap is too small or uneven
- Transmit power is too high on the distant AP
- The client is sticky and roams too late
- Security reauthentication is slowing the transition
- 802.11r is disabled in a voice-sensitive environment where the client supports it

Possible fixes:

1. Verify that adjacent APs have overlapping coverage.
2. Reduce excessive transmit power so clients do not cling to distant APs.
3. Confirm the same SSID and security settings are used across APs.
4. Enable roaming enhancements if supported and appropriate.
5. Test with a different client to determine whether the issue is device-specific.

**Quick comparison for exam review**

| Concept | What it represents | Changes during roaming? |
|---|---|---|
| SSID | Network name | Usually no |
| BSSID | Specific AP radio | Yes |
| BSS | One AP cell | Yes, if you move to another AP |
| ESS | Multiple AP cells with same SSID | No, the client stays within it |

**Key exam takeaways**

- The **SSID** is the name users see.
- The **BSSID** is typically the AP radio MAC address in infrastructure mode.
- In an **ad hoc** network, the BSSID is a special case because there is no AP radio.
- A **BSS** is one AP radio and its clients.
- An **ESS** is multiple BSSs sharing the same SSID and security so clients can roam.
- **Roaming** is the move between APs; **reassociation** and **reauthentication** are protocol steps that may occur during that move.
- **802.11r/k/v** are optional roaming enhancements, not mandatory requirements.
- Controller-based WLANs simplify consistent SSID deployment across many APs.
