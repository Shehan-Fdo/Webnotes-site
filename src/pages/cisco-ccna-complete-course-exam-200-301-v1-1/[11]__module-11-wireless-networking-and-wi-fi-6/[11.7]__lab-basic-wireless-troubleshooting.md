**Wireless troubleshooting workflow**

Wireless issues usually fall into one of four areas: radio frequency (RF), access point (AP), client, or upstream network services. You troubleshoot faster when you isolate the failure domain before changing settings. Start by confirming whether the problem is limited to one client, one SSID, one AP, or the entire WLAN.

A practical workflow is:

1. Verify the symptom and scope.
2. Check physical and RF health.
3. Validate SSID, security, and authentication.
4. Confirm IP addressing and gateway reachability.
5. Review controller, AP, and client logs.

Common wireless symptoms include no SSID visibility, association failures, authentication failures, weak signal, intermittent drops, and no network access after connection. Each symptom points to a different layer of the wireless stack.

- **Scope** matters first: one device, one room, one AP, or all users.
- **RF** issues often look like “slow Wi-Fi” or random disconnects.
- **Authentication** issues often appear after the client sees the SSID but cannot join.
- **DHCP** issues often appear after successful association and security negotiation.

**Basic wireless architecture**

In a typical enterprise WLAN, clients connect to an AP, the AP bridges traffic to the wired network, and a wireless LAN controller (WLC) may manage APs centrally. In smaller deployments, a standalone AP may handle SSID broadcasting and security locally. Understanding where the control plane lives helps you decide where to look first.

Key components include:

- **Client adapter**: wireless NIC, driver, and OS profile.
- **AP**: radio, SSID advertisement, channel, power, and uplink.
- **WLC**: centralized configuration, policy, and roaming control.
- **Infrastructure services**: DHCP, DNS, default gateway, and authentication servers.

If the AP is controller-managed, the AP may still broadcast an SSID even when the controller has policy issues, but clients can fail during authentication or IP assignment. If the AP itself is down, clients will not see the SSID at all.

**RF fundamentals you should check**

Wireless troubleshooting starts with the radio environment because Wi-Fi is sensitive to interference, attenuation, and channel overlap. A client may connect successfully but still perform poorly if the signal-to-noise ratio is low or if the channel is congested.

Important RF terms:

- **RSSI** (Received Signal Strength Indicator): how strong the signal appears to the client.
- **SNR** (Signal-to-Noise Ratio): signal quality relative to background noise.
- **Channel overlap**: multiple APs using overlapping channels, causing contention.
- **Co-channel interference**: APs on the same channel competing for airtime.
- **Adjacent-channel interference**: APs on partially overlapping channels, often worse than co-channel issues.

Practical RF checks:

- Confirm the client is within usable range.
- Check for physical obstructions such as concrete, metal, or elevator shafts.
- Look for interference from microwave ovens, cordless phones, Bluetooth devices, and neighboring WLANs.
- Verify channel planning and transmit power settings.
- Prefer 5 GHz or 6 GHz when available for less congestion; 2.4 GHz has fewer non-overlapping channels.

> <mark>Do not assume “full bars” means good Wi-Fi.</mark> A client can show strong signal but still suffer from high interference or poor SNR.

**Useful CLI and verification commands**

On a client or network device, you often begin with basic reachability and interface checks. The exact commands vary by platform, but the goal is the same: confirm association, IP configuration, and path to the gateway.

```bash
# Linux client checks
ip addr show
iw dev
nmcli dev wifi list
ping -c 4 <default-gateway>
traceroute <destination>
```

```powershell
# Windows client checks
ipconfig /all
netsh wlan show interfaces
netsh wlan show profiles
ping <default-gateway>
tracert <destination>
```

On Cisco devices, useful verification commands include:

```bash
show ip interface brief
show interfaces status
show running-config
show logging
show wireless summary
show ap summary
show wlan summary
show client summary
```

If you are troubleshooting a controller-based WLAN, `show ap summary` helps confirm AP registration, while `show client summary` helps confirm whether the client is associated and where it is failing.

**SSID and security validation**

A client must match the SSID name, security mode, and credentials exactly. Many wireless problems are caused by profile mismatches rather than RF issues. If the client sees the SSID but cannot join, verify the security settings first.

Check these items:

- SSID spelling and case sensitivity in saved profiles.
- Security type: open, WPA2-Personal, WPA2-Enterprise, WPA3, or mixed mode.
- Authentication method: pre-shared key (PSK) or 802.1X.
- Encryption compatibility: AES is common; legacy TKIP can cause issues.
- Hidden SSID behavior: some clients struggle with manually configured hidden networks.

For enterprise authentication, the client may need:

- Correct username and password or certificate.
- Reachability to the authentication server.
- Proper time synchronization for certificate validation.
- Matching EAP (Extensible Authentication Protocol) method.

A common failure pattern is: SSID visible, association succeeds, authentication fails, and the client retries repeatedly. That usually points to credentials, RADIUS, or policy mismatch rather than RF.

**DHCP and IP addressing checks**

A wireless client can connect to the WLAN and still have no usable network if DHCP fails. In that case, the client may self-assign an APIPA address on Windows or show no IPv4 address at all.

Check the following:

- DHCP server availability.
- VLAN assignment for the WLAN.
- DHCP relay configuration on the gateway or controller.
- Scope exhaustion.
- ACLs or firewall rules blocking DHCP traffic.

Typical DHCP flow is:

1. Client broadcasts DHCP Discover.
2. Server replies with DHCP Offer.
3. Client requests the offered address.
4. Server acknowledges the lease.

If this process fails, verify that the client’s VLAN can reach the DHCP server and that the wireless policy maps the SSID to the correct VLAN.

Example of a simple DHCP pool on Cisco IOS:

```cisco
ip dhcp excluded-address 192.168.10.1 192.168.10.20

ip dhcp pool WLAN_USERS
 network 192.168.10.0 255.255.255.0
 default-router 192.168.10.1
 dns-server 8.8.8.8
 domain-name example.local
```

If the AP or WLC bridges the client into the wrong VLAN, the client may authenticate successfully but fail to obtain an address. That is a strong clue that the wireless layer is working but the wired integration is not.

**Controller and AP checks**

In controller-based WLANs, the AP must join the controller and receive policy before clients can work correctly. If the AP is not registered, the SSID may not appear or may appear inconsistently depending on design.

Check:

- AP power and Ethernet uplink.
- CAPWAP (Control and Provisioning of Wireless Access Points) tunnel status, if applicable.
- AP registration state.
- WLAN-to-VLAN mapping.
- RF profile, channel, and power settings.
- Client exclusion or policy restrictions.

Useful controller-style checks often include:

```bash
show ap join stats summary
show ap config general <ap-name>
show wlan <wlan-id>
show client detail <client-mac>
show logging | include CAPWAP
```

If an AP is powered by PoE (Power over Ethernet), insufficient power can reduce radio functionality or disable one band. That can create a confusing symptom where 2.4 GHz works but 5 GHz does not, or the AP reboots under load.

**Client-side troubleshooting**

Client issues are often the fastest to verify because they are local and visible. Before changing network settings, confirm the client adapter is enabled and the correct profile is selected.

Check:

- Wireless adapter enabled in the OS.
- Airplane mode disabled.
- Correct SSID profile selected.
- Saved credentials are current.
- Driver version is compatible and not corrupted.
- MAC address randomization behavior, if it affects access control.

Useful client actions include:

```bash
# Linux
nmcli radio wifi on
nmcli dev disconnect wlan0
nmcli dev connect wlan0
sudo journalctl -u NetworkManager --since "10 min ago"
```

```powershell
# Windows
netsh wlan show interfaces
netsh wlan delete profile name="SSID_NAME"
netsh wlan add profile filename="profile.xml"
```

If the client has a stale profile, deleting and recreating it often resolves repeated authentication failures. If the adapter driver is outdated, update it before suspecting the WLAN infrastructure.

**Interference and channel planning**

Wireless networks share airtime, so channel planning is critical. In 2.4 GHz, only channels 1, 6, and 11 are typically non-overlapping in many regulatory domains. In 5 GHz, more channels are available, which reduces contention. In 6 GHz, the spectrum is cleaner still, but client support varies.

Troubleshooting steps:

- Identify nearby APs and their channels.
- Look for overlapping channel assignments.
- Reduce transmit power if APs are too close.
- Move clients to less congested bands when possible.
- Check for non-Wi-Fi interference sources.

A simple site-survey style view might look like this:

```text
AP-A: Channel 1, high utilization
AP-B: Channel 6, moderate utilization
AP-C: Channel 11, low utilization
Neighbor APs: Channel 6 and 11 overlap in coverage area
```

If users complain about poor performance only in one area, the issue may be RF design rather than a device fault. If the problem follows the client everywhere, the issue is more likely the client adapter or profile.

**Troubleshooting by symptom**

Use the symptom to narrow the likely cause.

- **SSID not visible**
  - AP down or not broadcasting.
  - Wrong band or disabled radio.
  - Hidden SSID and client profile mismatch.
  - Coverage gap or severe interference.

- **Can see SSID but cannot connect**
  - Wrong password or security type.
  - 802.1X or RADIUS failure.
  - MAC filtering or access control policy.
  - Client driver incompatibility.

- **Connects but no IP address**
  - DHCP failure.
  - Wrong VLAN.
  - Gateway or relay misconfiguration.
  - Scope exhaustion.

- **Connects but slow or unstable**
  - Low SNR.
  - Channel overlap or interference.
  - Excessive client density.
  - Power settings or roaming issues.

- **Intermittent disconnects**
  - Weak signal at cell edge.
  - Band steering or roaming thresholds.
  - Power-save behavior on the client.
  - AP reboot or PoE instability.

**Practical lab sequence**

When you perform a basic wireless troubleshooting lab, use a repeatable sequence so you do not skip layers.

1. Confirm the client can detect the SSID.
2. Verify the client profile and security settings.
3. Check association and authentication status.
4. Confirm the client receives an IP address.
5. Test gateway reachability with `ping`.
6. Test external reachability with `traceroute` or `tracert`.
7. Review AP, controller, and client logs for errors.
8. Adjust only one variable at a time and retest.

Example validation flow on a Windows client:

```powershell
netsh wlan show interfaces
ipconfig /all
ping 192.168.10.1
ping 8.8.8.8
tracert 8.8.8.8
```

If `ping` to the gateway fails but the client has an IP address, the issue may be VLAN, gateway, or ACL related. If the gateway works but public IPs fail, check routing, NAT, or DNS depending on the symptom.

**Common mistakes to avoid**

- Changing multiple settings at once and losing the root cause.
- Assuming the AP is broken when the client profile is wrong.
- Ignoring RF health because the SSID is visible.
- Forgetting to verify DHCP after successful authentication.
- Overlooking VLAN mapping between the WLAN and the wired network.
- Treating a single-user complaint as a network-wide outage.

**Quick reference checklist**

- **Client**: adapter enabled, profile correct, driver current.
- **RF**: signal, noise, channel, interference, band selection.
- **Security**: SSID, PSK, 802.1X, certificates, RADIUS.
- **IP**: DHCP, VLAN, gateway, DNS.
- **Infrastructure**: AP status, controller registration, logs, PoE, uplink.

**Key takeaway**

Wireless troubleshooting is most effective when you isolate the failure domain and verify each layer in order. Start with the client and RF environment, then move to authentication, addressing, and infrastructure. That approach reduces guesswork and helps you identify whether the issue is local, radio-related, or caused by upstream network services.
