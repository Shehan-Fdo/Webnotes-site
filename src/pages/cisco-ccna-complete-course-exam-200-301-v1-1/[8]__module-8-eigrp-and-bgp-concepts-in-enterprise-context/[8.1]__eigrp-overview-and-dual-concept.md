## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**EIGRP at a glance**

Enhanced Interior Gateway Routing Protocol (**EIGRP**) is an advanced distance-vector routing protocol that uses a neighbor table, topology table, and routing table to make fast, loop-free routing decisions. It is often described as a hybrid protocol because it combines distance-vector simplicity with link-state-like convergence behavior, but the exam focus is on how it forms neighbors, selects best paths, and reacts to failures.

EIGRP runs the **Diffusing Update Algorithm (DUAL)** to calculate the best route and maintain a backup path when one exists. That design helps EIGRP converge quickly without flooding the entire network with updates after every change.

- EIGRP is classless and supports **VLSM** (Variable Length Subnet Mask) and route summarization.
- It uses a composite metric based primarily on bandwidth and delay by default.
- It supports IPv4 and IPv6, including named mode on modern IOS.
- It sends **Hello** packets to discover and maintain neighbors.
- It can use unequal-cost load balancing with **variance**.

**Core EIGRP behavior**

EIGRP does not advertise full routing tables to all neighbors the way some older protocols do. Instead, it exchanges only the information needed to build and maintain routes. The protocol keeps a local view of the network in the topology table, then DUAL chooses the successor route for installation into the routing table.

A route can be in a stable **passive** state or an active state while EIGRP searches for a new path. This distinction matters for troubleshooting because most healthy routes remain passive, while active routes indicate the router is still trying to find a loop-free alternative.

- **Successor**: the best next hop for a destination.
- **Feasible successor**: a backup path that already satisfies the feasibility condition.
- **Feasibility condition**: a loop-prevention rule based on reported distance.
- **Feasible distance**: the router’s best known distance to a destination.

**EIGRP tables and their roles**

EIGRP uses three main tables, and each one serves a different purpose in convergence.

- **Neighbor table**: lists directly adjacent EIGRP routers.
- **Topology table**: stores all learned routes and their metrics.
- **Routing table**: contains the selected best route used for forwarding.

The neighbor table is the first place to check when EIGRP is not working. If neighbors do not form, no topology exchange occurs, and no routes are learned.

**Neighbor formation requirements**

For two routers to become EIGRP neighbors, they must be able to exchange Hello packets and agree on key EIGRP parameters.

- Layer 3 reachability must exist between the routers.
- The EIGRP autonomous system must match in classic mode, or the address-family settings must align in named mode.
- K-values must match.
- Authentication must match if configured.
- Hello and hold timers must be compatible enough for adjacency to form.
- Interfaces must not be passive for the relevant network.

> EIGRP does not require the same primary network. It requires Layer 3 reachability and matching EIGRP parameters.

**Packet types and what they do**

EIGRP uses several packet types, and the exam often asks what each one is for.

- **Hello**: discovers and maintains neighbors; sent unreliably.
- **Update**: carries routing information; sent reliably when needed.
- **Query**: asks neighbors for alternative paths; sent reliably.
- **Reply**: answers a query; sent reliably.
- **Acknowledgment**: confirms receipt of reliable packets; usually a Hello with no data.

Hello packets are the key to adjacency formation. Updates, Queries, and Replies are part of EIGRP’s reliable transport behavior, which means EIGRP tracks them and retransmits when necessary.

**Reliable Transport Protocol and DUAL**

EIGRP uses its own **Reliable Transport Protocol (RTP)**, not the real-time transport protocol. RTP helps EIGRP deliver packets that need acknowledgment, while DUAL handles route computation and loop avoidance.

These are separate concepts:

- **RTP** handles delivery reliability for EIGRP control packets.
- **DUAL** decides which route is best and whether a backup route is safe to use.

That separation is important. A reliable packet delivery mechanism does not by itself guarantee loop-free routing; DUAL provides that logic.

**DUAL operation**

DUAL keeps EIGRP stable by using a finite-state process for routes.

- In the **passive** state, the route is stable and no recomputation is in progress.
- In the **active** state, the router is searching for a new path because the current route failed and no feasible successor is available.

When a successor fails:

1. If a feasible successor exists, DUAL promotes it immediately.
2. If no feasible successor exists, the route becomes active.
3. The router sends Queries to neighbors asking for an alternate path.
4. Neighbors respond with Replies.
5. DUAL computes a new loop-free route and returns the route to passive.

This is why EIGRP often converges quickly when a backup path is already known.

**Feasible successor and feasibility condition**

The feasibility condition is the rule that prevents loops. A neighbor can be used as a feasible successor only if its reported distance to the destination is less than the local feasible distance.

- **Reported distance**: the neighbor’s advertised distance to the destination.
- **Feasible distance**: the local router’s best known distance to the destination.

If the condition is met, the route is considered loop-free as a backup. If not, the route may still be a valid path, but it cannot be used immediately as a feasible successor.

**Metric calculation**

EIGRP’s default metric uses bandwidth and delay. It can also include reliability, load, and MTU in the formula, but by default only bandwidth and delay are active.

The classic metric formula is based on K-values, which are metric weights. In most CCNA contexts, you should remember:

- Bandwidth is the minimum bandwidth along the path.
- Delay is the sum of interface delays along the path.
- Lower metric values are preferred.

Example output often shows the metric in the routing table:

```bash
Router# show ip route eigrp
D    10.10.20.0/24 [90/30720] via 10.10.12.2, 00:00:12, GigabitEthernet0/0
```

In this example:

- `D` means EIGRP.
- `90` is the administrative distance for internal EIGRP.
- `30720` is the metric.
- `10.10.12.2` is the next hop.

**Administrative distance**

Administrative distance (**AD**) is the trust value used when multiple routing sources know the same destination.

- Internal EIGRP AD is `90`.
- External EIGRP AD is `170`.

Internal routes are preferred over external EIGRP routes if both are available and otherwise comparable.

**Unequal-cost load balancing and variance**

EIGRP can install multiple paths when they are not equal cost, as long as they are within the configured **variance** and satisfy the feasibility condition.

- **Variance** multiplies the feasible distance to determine which alternate paths may be installed.
- Only feasible successors can be used for unequal-cost load balancing.
- This feature is useful when one path is faster but another path still has acceptable capacity.

Example:

```bash
Router(config)# router eigrp 100
Router(config-router)# variance 2
Router(config-router)# maximum-paths 4
```

This allows EIGRP to consider paths up to twice the feasible distance, assuming they are feasible successors.

**Stub routing**

EIGRP stub routing is commonly used on branch routers to limit query propagation and reduce unnecessary route searches.

- A stub router advertises only selected route types.
- It helps prevent large query chains in hub-and-spoke designs.
- It improves stability in small branch sites with limited connectivity.

Example:

```bash
Router(config)# router eigrp 100
Router(config-router)# eigrp stub connected summary
```

Common stub options include:

- `connected`
- `summary`
- `static`
- `redistributed`
- `receive-only`

Use stub routing when a branch router should not become a transit point for EIGRP queries.

**Passive-interface behavior**

The `passive-interface` command stops EIGRP Hello packets on an interface. That means no neighbor relationship forms on that interface, but the connected network can still be advertised if the router learns it through the EIGRP process.

This is useful on user-facing or access-layer interfaces where you do not want adjacency formation.

Example:

```bash
Router(config)# router eigrp 100
Router(config-router)# passive-interface default
Router(config-router)# no passive-interface g0/0
```

This pattern is common because it disables EIGRP on all interfaces first, then selectively enables it only where neighbors should form.

**Classic EIGRP configuration example**

A simple IPv4 EIGRP configuration might look like this:

```bash
Router(config)# router eigrp 100
Router(config-router)# network 10.10.10.0 0.0.0.255
Router(config-router)# network 10.10.12.0 0.0.0.3
Router(config-router)# no auto-summary
```

Notes:

- The `network` command matches interfaces by address range.
- `no auto-summary` is important in older-style EIGRP deployments to avoid classful summarization.
- Modern CCNA study should still understand that EIGRP is classless and supports VLSM.

**Named mode configuration example**

Named mode is the modern EIGRP configuration style in Cisco IOS. It organizes settings under a named process and address families.

Example for IPv4:

```bash
Router(config)# router eigrp CORP
Router(config-router)# address-family ipv4 unicast autonomous-system 100
Router(config-router-af)# network 10.10.10.0 0.0.0.255
Router(config-router-af)# passive-interface default
Router(config-router-af)# no passive-interface g0/0
Router(config-router-af)# exit-address-family
```

Example for IPv6:

```bash
Router(config)# ipv6 unicast-routing
Router(config)# router eigrp CORP
Router(config-router)# address-family ipv6 unicast autonomous-system 100
Router(config-router-af)# af-interface g0/0
Router(config-router-af-interface)# no shutdown
Router(config-router-af-interface)# exit-af-interface
Router(config-router-af)# exit-address-family
```

In named mode, you work inside the address family and interface submodes. This is the safest way to think about modern EIGRP configuration because IPv4 and IPv6 can be managed consistently under one process.

**Classic IPv6 EIGRP example**

Older IOS syntax may still appear in labs or legacy environments:

```bash
Router(config)# ipv6 unicast-routing
Router(config)# ipv6 router eigrp 100
Router(config-rtr)# eigrp router-id 1.1.1.1
Router(config)# interface g0/0
Router(config-if)# ipv6 eigrp 100
```

Important points:

- IPv6 EIGRP requires `ipv6 unicast-routing`.
- You should configure a router ID manually when possible.
- Interface activation uses `ipv6 eigrp 100` in classic mode.

**Router ID selection**

The EIGRP router ID is a 32-bit value written in IPv4 dotted-decimal format. It is used to identify the router in the EIGRP domain.

Selection behavior:

- If you configure it manually, that value is used.
- If not, the highest IPv4 address on a loopback interface is typically chosen.
- If no loopback exists, the highest IPv4 address on an active physical interface may be used.

For IPv6 and named mode, setting the router ID manually is best practice because it avoids ambiguity and helps with troubleshooting.

Example:

```bash
Router(config)# router eigrp 100
Router(config-router)# eigrp router-id 1.1.1.1
```

If the router ID changes, you may need to reset the EIGRP process for the new ID to take effect.

**Summarization**

EIGRP supports manual summarization at interface boundaries. Summarization reduces routing table size and can reduce the scope of queries, but it can also hide more-specific routes and create black holes if the summary is too broad.

Example:

```bash
Router(config)# interface g0/0
Router(config-if)# ip summary-address eigrp 100 10.10.0.0 255.255.0.0
```

Use summarization carefully:

- It can reduce routing overhead.
- It can improve scalability in hierarchical designs.
- It can also cause suboptimal forwarding if the summary covers unreachable space.

**Troubleshooting EIGRP neighbors**

When EIGRP adjacency fails, check the basics first.

1. Verify Layer 3 connectivity with `ping`.
2. Confirm the AS number or named-mode address-family matches.
3. Check that interfaces are up and not passive.
4. Verify K-values match.
5. Confirm authentication settings match.
6. Check for ACLs or firewalls blocking EIGRP control traffic.
7. Review timers only after the basics are confirmed.

Useful commands:

```bash
Router# show ip eigrp neighbors
Router# show ip eigrp interfaces
Router# show ip protocols
Router# show ip route eigrp
Router# show ip eigrp topology
```

For IPv6 or named mode, use the corresponding IPv6 or address-family show commands available on the platform.

> If neighbors do not form, routes do not exchange. Always start troubleshooting at adjacency, not at the routing table.

**Troubleshooting active routes and SIA**

A route that remains active too long can lead to **stuck-in-active (SIA)** behavior. This means the router sent Queries but did not receive all expected Replies in time.

Common causes include:

- A neighbor is unreachable or overloaded.
- The query scope is too large.
- A branch router is not configured as a stub.
- There is a design problem causing excessive query propagation.

Symptoms may include:

- Missing routes after a failure.
- Long convergence delays.
- EIGRP neighbor resets in larger topologies.

A simple conceptual failure scenario:

- Router A loses its successor route to a remote network.
- No feasible successor exists.
- Router A sends Queries to Router B and Router C.
- Router B replies quickly, but Router C is behind a long chain of routers and delays its Reply.
- Router A remains active until it receives all Replies or times out.

This is why stub routing and good hierarchy matter.

**Practical verification example**

A healthy EIGRP topology often shows a passive route with a successor and possibly a feasible successor.

```bash
Router# show ip eigrp topology 10.10.20.0/24
EIGRP-IPv4 Topology Entry for AS(100)/ID(1.1.1.1) for 10.10.20.0/24
  State is Passive, Query origin flag is 1, 1 Successor(s), FD is 30720
  Routing Descriptor Blocks:
  10.10.12.2 (GigabitEthernet0/0), from 10.10.12.2, Send flag is 0x0
      Composite metric is (30720/28160), Route is Internal
```

Interpretation:

- `State is Passive` means the route is stable.
- `1 Successor(s)` means the best path is installed.
- The metric pair shows feasible distance and reported distance.
- If a feasible successor exists, failover can be immediate.

**Common exam takeaways**

- EIGRP is an advanced distance-vector protocol that uses DUAL for loop-free convergence.
- Hellos form and maintain neighbors; Updates, Queries, and Replies are reliable EIGRP control packets.
- The neighbor table, topology table, and routing table each serve a distinct role.
- A route is usually passive when stable and active when DUAL is searching for a new path.
- Feasible successors are backup routes that satisfy the feasibility condition.
- Stub routing, passive interfaces, and summarization are important design tools.
- Named mode is the modern configuration style, especially for mixed IPv4 and IPv6 deployments.

**Quick comparison of key states**

| Concept | Meaning | Why it matters |
|---|---|---|
| Passive | Route is stable | Normal operating state |
| Active | Route is being recomputed | Indicates a failure or search |
| Successor | Best path | Installed in routing table |
| Feasible successor | Safe backup path | Enables fast convergence |
| SIA | Query process took too long | Points to design or reachability issues |

**Minimal lab topology to practice**

A simple three-router lab helps you see EIGRP behavior clearly.

- R1 connects to R2 over `10.10.12.0/30`.
- R2 connects to R3 over `10.10.23.0/30`.
- R1 advertises `10.10.1.0/24`.
- R3 advertises `10.10.3.0/24`.

You can then:

- Shut down the R1-R2 link and observe route changes.
- Check whether R1 has a feasible successor through R3 if an alternate path exists.
- Compare `show ip eigrp neighbors` before and after the failure.
- Observe whether the route becomes active or fails over immediately.

This kind of lab makes DUAL behavior much easier to remember than memorizing definitions alone.
