## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**FAQ mindset for CCNA review**

A strong final review is not just memorizing terms. You should connect each topic to how devices behave, how you verify that behavior, and what symptoms appear when something breaks. For CCNA, that means you can explain the purpose of a feature, recognize the relevant commands, and interpret output at a basic troubleshooting level.

<dl>
<dt>What should you focus on in a final review?</dt>
<dd>Core concepts, common verification commands, basic configuration patterns, and how to identify the most likely cause of a failure.</dd>
<dt>What does the exam usually test?</dt>
<dd>Understanding of networking behavior, not deep vendor-specific tuning. You should know what a feature does, where it is used, and how to confirm it works.</dd>
</dl>

- Review by domain: network fundamentals, switching, routing, IP services, security, automation, and wireless.
- Tie every concept to a command or observable output.
- Use troubleshooting questions to test whether you can isolate the layer where a problem occurs.

**Network fundamentals review**

You should be comfortable with the building blocks of IPv4 and IPv6, subnetting, and basic device roles. A lot of CCNA questions are really asking whether you understand how packets move and what information each protocol adds or depends on.

- **IPv4 addressing** uses a subnet mask to separate the network and host portions.
- **IPv6 addressing** uses prefixes, and hosts learn the router’s link-local address as the next-hop default router from router advertisements.
- **ARP** resolves IPv4 addresses to MAC addresses on the local segment.
- **Neighbor Discovery Protocol (NDP)** performs similar functions in IPv6, including address resolution and router discovery.

Practical verification examples:

```bash
show ip interface brief
show ipv6 interface brief
show arp
show ipv6 neighbors
ping 192.0.2.1
ping ipv6 2001:db8:1::1
traceroute 8.8.8.8
traceroute ipv6 2001:db8::1
```

- `show ip interface brief` helps you confirm interface status and assigned IPv4 addresses.
- `show ipv6 interface brief` helps you confirm IPv6 enablement and interface addresses.
- `show arp` and `show ipv6 neighbors` help you confirm local Layer 2 to Layer 3 resolution.
- `traceroute` helps you identify where traffic stops along the path.

> If two hosts are on the same IPv4 subnet, they must use matching subnet masks on that shared segment for normal adjacency and direct communication to work as expected.

**Switching and VLAN review**

Switching questions often test whether you understand segmentation, trunking, and how frames are forwarded. VLANs create separate broadcast domains, and trunks carry multiple VLANs between switches or between a switch and a router or firewall.

- **Access ports** carry traffic for one VLAN.
- **Trunk ports** carry traffic for multiple VLANs using 802.1Q tagging.
- **Native VLAN** traffic on an 802.1Q trunk is untagged by default.
- **Inter-VLAN routing** lets hosts in different VLANs communicate through a Layer 3 device.

Common verification commands:

```bash
show vlan brief
show interfaces trunk
show interfaces switchport
show mac address-table
```

- `show vlan brief` confirms VLAN membership.
- `show interfaces trunk` confirms trunk status, allowed VLANs, and native VLAN.
- `show mac address-table` shows learned MAC addresses and the ports where they were seen.

Example VLAN and trunk configuration:

```cisco
vlan 10
 name USERS
vlan 20
 name VOICE

interface gi0/1
 switchport mode access
 switchport access vlan 10

interface gi0/24
 switchport mode trunk
 switchport trunk allowed vlan 10,20
 switchport trunk native vlan 99
```

- Use a dedicated native VLAN that is not used for user traffic.
- Keep the allowed VLAN list as small as practical.
- Verify that both ends of a trunk agree on trunking behavior.

**Layer 2 security review**

CCNA-level security includes basic protections that reduce common switch attacks. You should know what each feature protects and what happens when a violation occurs.

- **Port security** limits which MAC addresses can use a switch port.
- **DHCP snooping** blocks rogue DHCP servers and builds a trusted binding table.
- **Dynamic ARP Inspection (DAI)** checks ARP packets against DHCP snooping bindings.
- **IP Source Guard** helps prevent spoofed source IP addresses on access ports.

Example configuration:

```cisco
interface gi0/10
 switchport mode access
 switchport access vlan 10
 switchport port-security
 switchport port-security maximum 2
 switchport port-security mac-address sticky
 switchport port-security violation restrict

ip dhcp snooping
ip dhcp snooping vlan 10

interface gi0/1
 ip dhcp snooping trust

ip arp inspection vlan 10
```

- **Sticky MAC** learns and saves the first seen MAC address in the running configuration.
- **Restrict** drops violating frames and increments counters.
- **Shutdown** places the port into an err-disabled state after a violation.

Troubleshooting examples:

- If a port goes err-disabled after a new device is connected, check port security counters and the learned MAC list.
- If clients cannot get DHCP addresses, confirm that the uplink toward the DHCP server is trusted for DHCP snooping.
- If ARP replies are being dropped, verify the DHCP snooping binding table and DAI trust settings.

**Routing and path selection review**

Routing questions usually focus on how routers choose paths and how you verify that a route exists. You should know the difference between connected, static, and dynamic routes, and you should be able to read a routing table.

- **Connected routes** appear when an interface is up and has an IP address.
- **Static routes** are manually configured.
- **Dynamic routes** are learned through a routing protocol.
- **Longest prefix match** determines which route is used when multiple routes match a destination.

Verification commands:

```bash
show ip route
show ip route static
show ip protocols
show ip cef
```

- `show ip route` shows the full routing table and route codes.
- `show ip protocols` shows routing protocol timers, networks, and neighbors where applicable.
- `show ip cef` can help confirm forwarding behavior on platforms that support Cisco Express Forwarding.

Static route example:

```cisco
ip route 203.0.113.0 255.255.255.0 192.0.2.2
ip route 0.0.0.0 0.0.0.0 192.0.2.1
```

- Use a default route when the router should send unknown destinations to an upstream next hop.
- Use a specific static route when you want deterministic forwarding for a known network.

**OSPF review**

Open Shortest Path First (OSPF) is a link-state routing protocol that builds a topology map and calculates best paths with the Shortest Path First algorithm. For CCNA, you should know how neighbors form, what areas do, and how to verify adjacency.

- OSPF neighbors must match key parameters such as area, subnet mask on the shared segment, hello/dead timers, and authentication if configured.
- OSPF uses multicast addresses for neighbor discovery and database exchange.
- Router IDs identify OSPF routers uniquely.
- Area 0 is the backbone area.

Basic configuration example:

```cisco
router ospf 1
 router-id 1.1.1.1
 network 10.1.1.0 0.0.0.255 area 0
 network 10.1.2.0 0.0.0.255 area 0
```

Verification commands:

```bash
show ip ospf neighbor
show ip ospf interface brief
show ip route ospf
show ip protocols
```

- `show ip ospf neighbor` should show neighbors in the `FULL` state for a healthy adjacency.
- `show ip ospf interface brief` helps you confirm which interfaces participate in OSPF.
- `show ip route ospf` shows routes learned through OSPF.

Troubleshooting checklist:

1. Confirm the interfaces are in the same subnet and use matching subnet masks on the shared segment.
2. Confirm the area number matches.
3. Confirm hello and dead timers match.
4. Confirm the interfaces are not passive if they should form neighbors.
5. Confirm the router IDs are unique.

> If neighbors are stuck in `INIT`, `EXSTART`, or `EXCHANGE`, focus on adjacency parameters and Layer 2 reachability before assuming a routing problem.

**First Hop Redundancy Protocols review**

First Hop Redundancy Protocols (FHRPs) provide a shared default gateway for hosts. The exam commonly expects you to understand the purpose of HSRP, VRRP, and GLBP, and to recognize basic configuration and verification.

- **HSRP** uses an active and standby router.
- **VRRP** uses a master and backup router.
- **GLBP** can load-balance traffic across multiple routers while presenting one virtual gateway.

HSRP example:

```cisco
interface gi0/0
 ip address 192.0.2.2 255.255.255.0
 standby 1 ip 192.0.2.1
 standby 1 priority 110
 standby 1 preempt
```

VRRP example:

```cisco
interface gi0/0
 ip address 192.0.2.3 255.255.255.0
 vrrp 1 ip 192.0.2.1
 vrrp 1 priority 110
 vrrp 1 preempt
```

GLBP example:

```cisco
interface gi0/0
 ip address 192.0.2.4 255.255.255.0
 glbp 1 ip 192.0.2.1
 glbp 1 priority 110
 glbp 1 preempt
```

Verification commands:

```bash
show standby brief
show vrrp brief
show glbp brief
```

- Use FHRPs when you need gateway redundancy for end hosts.
- Use preemption when you want a higher-priority router to reclaim the active role after recovery.
- Verify that the virtual IP address is in the same subnet as the hosts.

**IP services review**

CCNA IP services include DHCP, DNS, NAT, NTP, SNMP, and syslog. You should know what each service does and how to verify that it is working.

- **DHCP** automatically assigns IP settings to clients.
- **DNS** resolves names to IP addresses.
- **NAT** translates private addresses to public addresses.
- **NTP** synchronizes device clocks.
- **SNMP** allows monitoring and management.
- **Syslog** records events for troubleshooting and auditing.

DHCP server example:

```cisco
ip dhcp excluded-address 192.0.2.1 192.0.2.20

ip dhcp pool USERS
 network 192.0.2.0 255.255.255.0
 default-router 192.0.2.1
 dns-server 8.8.8.8
```

NAT overload example:

```cisco
access-list 1 permit 192.0.2.0 0.0.0.255

interface gi0/0
 ip nat inside

interface gi0/1
 ip nat outside

ip nat inside source list 1 interface gi0/1 overload
```

Verification commands:

```bash
show ip dhcp binding
show ip nat translations
show ip nat statistics
show ntp status
show snmp
show logging
```

- `show ip dhcp binding` confirms leased addresses.
- `show ip nat translations` confirms active translations.
- `show ntp status` confirms clock synchronization state.
- `show snmp` confirms SNMP configuration and counters on platforms that support it.
- `show logging` helps you correlate events with failures.

**Wireless review**

Wireless questions usually test basic WLAN architecture, security, and client association flow. You should know the role of the wireless LAN controller (WLC), access point (AP), SSIDs, and common authentication methods.

- **SSID** is the network name advertised to clients.
- **WLC** centrally manages APs in controller-based wireless designs.
- **CAPWAP** is the control and data protocol used between APs and controllers in many Cisco designs.
- **WPA2/WPA3** are common security standards; CCNA typically emphasizes the purpose of strong authentication and encryption rather than vendor-specific tuning.

Basic controller-based WLAN example:

```text
WLAN: CORP-WIFI
VLAN: 30
Security: WPA2-PSK
```

Example AP-to-VLAN mapping concept:

```cisco
interface vlan 30
 ip address 192.0.2.30 255.255.255.0

interface gi0/1
 switchport mode trunk
 switchport trunk allowed vlan 30
```

- Map the SSID to the correct VLAN so wireless clients land in the intended broadcast domain.
- Confirm the AP uplink carries the VLAN used by the WLAN.
- Verify that clients receive an IP address from the correct DHCP scope.

Client association flow:

1. The client discovers the SSID.
2. The client authenticates using the configured security method.
3. The client associates to the AP.
4. The client receives network settings, usually through DHCP.
5. The client can reach the default gateway and other resources.

Troubleshooting examples:

- If the SSID is visible but clients cannot connect, check security settings and credentials.
- If clients connect but cannot reach the network, check VLAN mapping, DHCP, and gateway reachability.
- If roaming is poor, confirm AP coverage overlap and consistent SSID/security settings across APs.

**Automation and programmability review**

Network automation is part of CCNA v1.1 and usually appears as a conceptual topic. You should understand why automation exists and recognize the common tools and data formats used to interact with network devices and controllers.

- **Controller-based networking** centralizes policy and management.
- **APIs** (Application Programming Interfaces) let software communicate with devices or controllers.
- **REST** (Representational State Transfer) is a common API style.
- **JSON** (JavaScript Object Notation) and **YAML** (YAML Ain’t Markup Language) are common data formats.
- **Ansible** is a popular automation tool that uses playbooks to apply configuration.

Example JSON payload:

```json
{
  "hostname": "sw1",
  "interfaces": [
    {
      "name": "GigabitEthernet0/1",
      "description": "Uplink",
      "enabled": true
    }
  ]
}
```

Example YAML playbook:

```yaml
- name: Configure switch basics
  hosts: switches
  gather_facts: no
  tasks:
    - name: Set hostname
      cisco.ios.ios_config:
        lines:
          - hostname SW1
```

Example REST concept:

```http
GET /api/v1/interfaces
Authorization: Bearer <token>
Accept: application/json
```

- Use APIs when you need programmatic access to device state or configuration.
- Use JSON and YAML to structure data for automation workflows.
- Use Ansible when you want repeatable, declarative configuration across many devices.

**Troubleshooting workflow review**

A good troubleshooting process is systematic. You should isolate the problem domain, verify the expected state, and compare it to the actual output.

1. Identify the symptom.
2. Determine the scope: one host, one VLAN, one subnet, or the whole network.
3. Check physical and interface status.
4. Verify Layer 2 forwarding and VLAN membership.
5. Verify Layer 3 addressing and routing.
6. Check services such as DHCP, DNS, NAT, and NTP.
7. Review logs and counters for clues.

Useful commands:

```bash
show interfaces status
show ip interface brief
show vlan brief
show interfaces trunk
show mac address-table
show ip route
show ip ospf neighbor
show ip dhcp binding
show ip nat translations
show logging
```

Interpretation examples:

- An interface in `administratively down` state usually means it is shut down.
- An interface in `down/down` often indicates a physical or cabling issue.
- A trunk missing the expected VLAN may cause hosts to lose connectivity even when the link is up.
- A route missing from the routing table means the router cannot forward to that destination unless another route exists.

**Exam-style FAQ**

<details>
<summary>How do you know whether a problem is Layer 2 or Layer 3?</summary>
<p>If hosts in the same VLAN cannot communicate, check switching, VLAN membership, trunking, and MAC learning first. If hosts can reach the default gateway but not remote networks, check routing, NAT, and upstream reachability.</p>
</details>

<details>
<summary>What does a healthy OSPF neighbor relationship look like?</summary>
<p>You should usually see the neighbor in the <code>FULL</code> state in <code>show ip ospf neighbor</code>. If it is stuck in an earlier state, compare area, timers, subnet mask, and interface participation.</p>
</details>

<details>
<summary>What is the fastest way to confirm a DHCP issue?</summary>
<p>Check whether the client received an address, whether the DHCP server has a binding, and whether the relay or server interface is reachable. On Cisco devices, <code>show ip dhcp binding</code> and interface status are good starting points.</p>
</details>

<details>
<summary>Why do wireless clients connect but still fail to reach resources?</summary>
<p>The SSID may be correct, but the VLAN mapping, DHCP scope, or default gateway may be wrong. Verify the AP uplink trunk, the client VLAN, and the gateway interface.</p>
</details>

**Final review checklist**

- Know the purpose of each major protocol and feature.
- Know at least one verification command for each topic.
- Know the common failure symptoms and the most likely cause.
- Know how to read interface, routing, and neighbor outputs.
- Know the difference between configuration intent and operational state.

```bash
show ip interface brief
show ipv6 interface brief
show vlan brief
show interfaces trunk
show mac address-table

show ip ospf neighbor
```

These commands appear often because they quickly reveal whether the network is healthy at the access, distribution, or service layer. If you can explain what each output should look like in a normal case and what changes when something breaks, you are in strong shape for the exam.
