## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Lab goal and mental model**

This lab compares how the router decides which route to install when the same destination prefix is learned from different sources. The key idea is that routing decisions happen in layers:

- A routing protocol may choose its own best path internally.
- The router then compares the resulting route against other routes to the same prefix in the **routing table** or **RIB** (Routing Information Base).
- **Administrative distance** decides which source is trusted most when multiple route sources compete for the same prefix.

For CCNA, focus on the practical outcome: which route appears in `show ip route`, which route is active, and what changes when you add, remove, or fail a path.

> **Important:** Connected and static routes are not selected by a protocol best-path process. If present and valid, they are installed directly, and they compete with other routes to the same prefix using administrative distance.

**Core comparison: what wins and why**

The router prefers the route source with the lowest administrative distance for the same prefix, assuming the route is valid and reachable.

| Route source | Typical default AD | Notes |
|---|---:|---|
| Connected | 0 | Most trusted; installed when an interface is up/up and addressed |
| Static | 1 | Can be used as a primary route or backup route |
| EIGRP internal | 90 | Lower than OSPF by default |
| OSPF | 110 | Common IGP for enterprise routing |
| EIGRP external | 170 | Less trusted than internal EIGRP |
| BGP eBGP | 20 | Lower than IGPs, but BGP still must have a usable next hop |
| BGP iBGP | 200 | Less trusted than most IGP routes |

A lower AD does not mean the protocol is “better” in every sense. It only means the router prefers that source when multiple routes to the same prefix compete.

**Suggested lab topology**

Use a simple three-router or four-router topology so you can compare behavior on one destination prefix.

- R1 is the source router where you inspect the RIB.
- R2 advertises the destination prefix through OSPF or EIGRP.
- R3 provides a static route or BGP-learned route to the same prefix.
- A loopback on the far side acts as the destination network, such as `10.10.10.0/24`.

Example topology idea:

- R1 connects to R2 and R3.
- R2 reaches `10.10.10.0/24` through OSPF or EIGRP.
- R3 reaches `10.10.10.0/24` through a static route or BGP.
- R1 learns the same prefix from more than one source and must choose one.

**Static route behavior**

Static routes are manually configured and are often used for default routes, stub networks, or backups. If a static route points to a valid next hop or exit interface, the router installs it immediately.

Example static route:

```bash
ip route 10.10.10.0 255.255.255.0 172.16.1.2
```

If the same prefix is also learned by OSPF or EIGRP, the static route usually wins because its AD is 1, which is lower than OSPF 110 and EIGRP 90 only if the static route is present and valid. If you want the static route to act as a backup, raise its AD:

```bash
ip route 10.10.10.0 255.255.255.0 172.16.1.2 200
```

That creates a **floating static route**. It stays in the config but is not preferred unless the primary route disappears.

Practical points:

- A static route can point to a next-hop IP, an exit interface, or both.
- A static route is only useful if the next hop is reachable.
- A floating static route is a common backup design for WAN failover.

**Connected route behavior**

Connected routes are created automatically when an interface is up/up and has an IP address in the subnet.

Example:

```bash
interface Loopback10
 ip address 10.10.10.1 255.255.255.0
```

This creates a connected route for `10.10.10.0/24` on that router. Connected routes have AD 0, so they beat static, OSPF, EIGRP, and BGP for the same prefix.

Important details:

- The route appears only when the interface is operational.
- If the interface goes down, the connected route is removed.
- A connected route is not “learned”; it is directly attached to the router.

**OSPF behavior**

OSPF (Open Shortest Path First) calculates path cost using its own SPF (Shortest Path First) process. It does not use administrative distance to choose among OSPF paths. AD matters only when the OSPF route competes with a route from another source in the RIB.

Example OSPF configuration:

```bash
router ospf 1
 router-id 1.1.1.1
 network 10.10.10.0 0.0.0.255 area 0
 network 172.16.1.0 0.0.0.3 area 0
```

OSPF behavior to observe:

- If one OSPF path exists, it is installed if no better route source exists.
- If multiple equal-cost OSPF paths exist, OSPF can install **ECMP** (Equal-Cost Multi-Path) routes when the platform and configuration allow it.
- If a static route to the same prefix exists with lower AD, the static route wins in the RIB.
- If the OSPF route is the best OSPF path but a connected route exists for the same prefix, the connected route wins because AD 0 is lower.

Example of equal-cost multipath concept:

```bash
show ip route 10.10.10.0
```

You may see multiple next hops if the router has equal-cost OSPF paths and `maximum-paths` permits them.

**EIGRP behavior**

EIGRP (Enhanced Interior Gateway Routing Protocol) uses its own metric based on bandwidth and delay by default. It selects a **successor** as the active route. A **feasible successor** is a loop-free backup route that EIGRP can use quickly if the successor fails.

Key corrections for study:

- The successor is the installed active route.
- A feasible successor exists only if the topology satisfies the feasibility condition.
- Not every topology has a feasible successor.
- EIGRP does not install a backup route unless it is loop-free and eligible.

Example EIGRP configuration:

```bash
router eigrp 100
 network 10.10.10.0 0.0.0.255
 network 172.16.1.0 0.0.0.3
 no auto-summary
```

Useful verification:

```bash
show ip eigrp neighbors
show ip eigrp topology
show ip route 10.10.10.0
```

What to look for:

- The topology table shows the successor and any feasible successors.
- The routing table shows only the successor unless multiple equal-cost routes are valid.
- If the successor fails and a feasible successor exists, EIGRP can switch quickly.
- If no feasible successor exists, EIGRP must recompute, which can cause a longer convergence event.

A simple failover test:

1. Build two paths from R1 to the same destination prefix through EIGRP.
2. Confirm one path is the successor and the other is a feasible successor, if the topology allows it.
3. Shut the primary link or remove the primary route.
4. Verify whether the backup becomes active immediately or whether EIGRP must recalculate.

**BGP behavior**

BGP (Border Gateway Protocol) chooses its best path using BGP attributes such as local preference, AS path, origin, MED (Multi-Exit Discriminator), and next-hop reachability. After BGP selects its best path, the router still compares that route against other route sources in the RIB using administrative distance.

For CCNA, keep the focus on these practical points:

- BGP must have a usable next hop before the route can be installed.
- eBGP-learned routes are usually preferred over IGP routes because default AD is 20.
- iBGP-learned routes have default AD 200, so they usually lose to OSPF and EIGRP.
- A BGP route can be the best BGP path and still fail to enter the routing table if the next hop is unreachable.

Example eBGP-style configuration on two routers:

```bash
router bgp 65001
 neighbor 172.16.12.2 remote-as 65002
 network 10.10.10.0 mask 255.255.255.0
```

On the peer router:

```bash
router bgp 65002
 neighbor 172.16.12.1 remote-as 65001
 network 192.0.2.0 mask 255.255.255.0
```

Verification:

```bash
show ip bgp summary
show ip bgp
show ip route 10.10.10.0
```

Practical BGP notes:

- `network` only advertises a prefix if that exact prefix exists in the local routing table.
- eBGP adjacency requires correct neighbor IPs, AS numbers, and reachability.
- For iBGP, the next hop often needs to be reachable through the IGP or a static route.
- BGP is often used at the edge, so in CCNA labs it is usually conceptual or simplified.

**How the router chooses between sources**

When multiple route sources advertise the same prefix, the router compares the candidates in the RIB using administrative distance.

Typical outcome order for the same prefix:

1. Connected route
2. Static route
3. EIGRP internal
4. OSPF
5. EIGRP external
6. eBGP
7. iBGP

This order is not a universal “best protocol” ranking. It is only the default trust order for competing routes to the same destination.

Example scenario for `10.10.10.0/24`:

- R1 has a connected loopback in that subnet.
- R1 also learns the same subnet through OSPF.
- R1 also has a static route to the subnet.
- R1 also receives the subnet through EIGRP and BGP.

Expected result:

- The connected route wins if the interface is up.
- If the connected route disappears, the static route wins if valid.
- If the static route is removed, EIGRP internal usually beats OSPF.
- eBGP can beat OSPF and EIGRP if it is the competing route source and the next hop is reachable.
- iBGP usually loses to IGP routes because its AD is 200.

**Expected output patterns**

Use these patterns to validate your lab.

- Connected route:
  - `C 10.10.10.0/24 is directly connected, Loopback10`
- Static route:
  - `S 10.10.10.0/24 [1/0] via 172.16.1.2`
- OSPF route:
  - `O 10.10.10.0/24 [110/...] via ...`
- EIGRP internal route:
  - `D 10.10.10.0/24 [90/...] via ...`
- eBGP route:
  - `B 10.10.10.0/24 [20/...] via ...`
- iBGP route:
  - `B 10.10.10.0/24 [200/...] via ...`

The bracketed values show administrative distance and metric. The metric is protocol-specific; the AD is what matters when comparing different route sources.

**Step-by-step lab procedure**

1. Build the topology and assign IP addresses.
2. Add the destination prefix as a connected loopback on one router.
3. Advertise the prefix with one protocol at a time.
4. Confirm the route appears in `show ip route`.
5. Add a competing route source for the same prefix.
6. Observe which route wins and why.
7. Remove or fail the preferred route.
8. Confirm the backup route becomes active if it is valid.

**Suggested reproducible test sequence**

1. Start with a connected route on the destination router.
2. Add a static route on R1 to the same prefix and confirm the connected route still wins on the destination router itself, while R1 installs whichever route it learns for that prefix.
3. Add OSPF for the same prefix and confirm OSPF loses to static but beats no route.
4. Add EIGRP for the same prefix and confirm EIGRP internal beats OSPF by default AD.
5. Add eBGP for the same prefix and confirm eBGP beats OSPF and EIGRP if it is the competing route source on R1.
6. Add iBGP for the same prefix and confirm it usually loses to OSPF and EIGRP.
7. Remove the preferred route source and verify the next valid route appears.

**Troubleshooting checklist**

- Confirm the prefix exists exactly as advertised.
- Confirm the next hop is reachable.
- Confirm interfaces are up/up.
- Confirm the routing process includes the network statement or redistribution needed to advertise the prefix.
- Confirm no ACL, passive-interface setting, or AS mismatch blocks adjacency.
- Confirm the route is not filtered by a route map, distribute list, or prefix list.

**Exam focus summary**

For CCNA, remember these high-value ideas:

- **Connected** routes are directly attached and have the lowest AD.
- **Static** routes are manually configured and can be made floating backups.
- **OSPF** chooses paths with SPF and can install equal-cost routes.
- **EIGRP** installs a successor and may have a feasible successor as a backup.
- **BGP** selects a best path using BGP attributes, then the router compares it against other route sources using AD.
- **Administrative distance** decides which route source wins in the RIB when multiple sources advertise the same prefix.

**Quick comparison matrix**

| Behavior | Connected | Static | OSPF | EIGRP | BGP |
|---|---|---|---|---|---|
| Manually configured | No | Yes | No | No | Yes |
| Uses protocol best-path logic | No | No | Yes | Yes | Yes |
| Default AD | 0 | 1 | 110 | 90 | 20 eBGP / 200 iBGP |
| Can be backup route | No | Yes | Yes | Yes | Yes |
| Common CCNA lab role | Directly attached network | Primary or floating route | IGP learning | IGP learning and failover | Edge routing concept |

**Practical takeaway**

If you see multiple routes to the same prefix, do not assume the protocol with the “best” internal metric wins. First determine whether the route is connected, static, or learned, then check administrative distance, then inspect the protocol’s own path selection rules. That sequence explains most routing-table outcomes in CCNA labs.
