## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**BGP at a glance**

Border Gateway Protocol (**BGP**) is the routing protocol used to exchange routes between autonomous systems (**ASes**). An AS is a network or group of networks under a single administrative control. In practice, BGP is the protocol that connects an enterprise to one or more Internet service providers (**ISPs**) and is also used in large internal designs where policy control matters more than fast convergence.

BGP is a **path-vector** protocol, not a classic distance-vector protocol like RIP. It carries path information, especially the **AS_PATH** attribute, so routers can make policy-based decisions and avoid routing loops across multiple organizations.

- BGP is designed for **policy control**, not rapid convergence.
- It scales well for large routing domains because it does not flood full topology information.
- It is commonly used at the edge of enterprise networks, data centers, and service provider networks.
- It exchanges routes over **TCP** port `179`, but route exchange only begins after the BGP finite state machine reaches the **Established** state.

**Why enterprises use BGP**

Enterprises usually do not deploy BGP everywhere. They use it where they need control over how traffic enters and leaves the network, especially with multiple upstream connections or complex WAN designs. BGP lets you influence path choice with attributes instead of relying only on shortest-path metrics.

Common enterprise use cases include:

- **Dual-homed Internet access** with two ISPs for redundancy.
- **Traffic engineering** to prefer one ISP for outbound or inbound traffic.
- **Mergers and acquisitions** where separate routing domains must be connected carefully.
- **Large branch or WAN designs** where route policy and summarization matter.
- **Multihoming** to improve availability and reduce dependency on a single provider.

> BGP is often chosen when you need control over route advertisement and path preference, not when you need the fastest possible reaction to link failure.

**BGP session basics**

Two BGP neighbors are called **peers** or **neighbors**. They form a TCP session and then exchange BGP messages to establish the relationship. A TCP connection alone is not enough; the peers must complete the BGP state machine before they exchange routing information.

The main BGP message types are:

- **Open**: starts the BGP relationship and negotiates parameters.
- **Keepalive**: maintains the session.
- **Update**: advertises or withdraws routes.
- **Notification**: reports errors and tears down the session.

BGP uses timers and keepalives to maintain the session. If the session fails, routes learned from that neighbor are withdrawn.

**Internal BGP and external BGP**

BGP has two common relationship types:

| Type | Meaning | Typical use |
|---|---|---|
| **eBGP** | External BGP between different ASes | Enterprise to ISP, ISP to ISP |
| **iBGP** | Internal BGP within the same AS | Large enterprise core, route distribution between edge routers |

eBGP is used to exchange routes with another organization. iBGP is used when you want BGP policy inside your own AS. In CCNA-level enterprise designs, you mainly need to recognize that eBGP is the common Internet edge protocol and iBGP is used internally when BGP is intentionally extended beyond the border.

Important behavior differences:

- eBGP neighbors are usually directly connected, though multihop is possible.
- iBGP requires additional design care because routes learned from one iBGP peer are not automatically advertised to another iBGP peer unless you use route reflectors or a full mesh.
- BGP does not replace an IGP (**Interior Gateway Protocol**) such as OSPF or EIGRP inside a typical enterprise; it often complements one.

**BGP attributes and path selection**

BGP chooses a best path using attributes. The exact algorithm is vendor-specific in details, but Cisco IOS/IOS XE follows a well-known order of preference. The key idea is that BGP compares multiple possible paths and selects one best path for the BGP table before deciding whether that route can be installed into the routing table (**RIB**, Routing Information Base).

Common attributes include:

- **Weight**: Cisco-local preference; higher is better.
- **Local Preference**: preferred exit path inside the AS; higher is better.
- **AS_PATH**: list of ASes the route traversed; shorter is usually better.
- **Origin**: route origin type; IGP is preferred over EGP and incomplete.
- **MED** (**Multi-Exit Discriminator**): suggests preferred entry point into an AS; lower is better.
- **Next hop**: must be reachable for the route to be usable.
- **Community**: tags routes for policy handling.

A practical interpretation:

- Use **weight** when you want one router to prefer a path locally.
- Use **local preference** when you want the whole AS to prefer one exit.
- Use **AS_PATH prepending** when you want to make a path less attractive to inbound traffic from the Internet.
- Use **MED** when you want to influence how a neighboring AS enters your network, usually when multiple links exist between the same two ASes.

> A route can appear in the BGP table but still not be installed in the RIB if another route is preferred or if the next hop is unreachable.

**BGP and the routing table**

BGP best-path selection happens before route installation into the RIB. That means:

- A route may be present in the BGP table but not in the routing table.
- Administrative distance affects which route wins during RIB installation, not the BGP best-path decision itself.
- If a better route from another source exists, BGP may lose the installation even if it is the best BGP path.

Useful Cisco commands:

```bash
show ip bgp
show ip route
show ip bgp summary
show ip bgp neighbors
```

On newer address-family-based syntax, you may also see:

```bash
show bgp ipv4 unicast
show bgp ipv4 unicast summary
show bgp ipv4 unicast neighbors
```

**Enterprise topology examples**

A common enterprise edge design uses two ISPs and one border router or two border routers. BGP gives you control over which ISP is preferred for outbound traffic and how your prefixes are advertised to the Internet.

Example logical topology:

```text
          ISP 1 (AS 64501)
                 |
                 | eBGP
                 |
        +-------------------+
        | Border Router R1   |
        +-------------------+
                 |
                 | iBGP or IGP
                 |
        +-------------------+
        | Enterprise Core    |
        +-------------------+
                 |
                 | eBGP
                 |
          ISP 2 (AS 64502)
```

In a dual-homed design, you might:

- Prefer ISP 1 for outbound traffic using local preference.
- Keep ISP 2 as backup.
- Advertise the same public prefix to both ISPs for redundancy.
- Use AS_PATH prepending toward one ISP to make the other path more attractive for inbound traffic.

**Basic BGP configuration example**

The following example shows a simple eBGP setup between an enterprise router and an ISP router. The exact IPs and AS numbers are examples.

Enterprise router:

```cisco
router bgp 65010
 bgp log-neighbor-changes
 neighbor 203.0.113.1 remote-as 64501
 network 198.51.100.0 mask 255.255.255.0
```

ISP-facing neighbor:

```cisco
router bgp 64501
 bgp log-neighbor-changes
 neighbor 203.0.113.2 remote-as 65010
```

What this does:

- `router bgp 65010` starts BGP in AS `65010`.
- `neighbor 203.0.113.1 remote-as 64501` defines the eBGP peer.
- `network 198.51.100.0 mask 255.255.255.0` advertises the prefix only if that exact network is present in the local routing table.

> The `network` statement does not create the route. The prefix must already exist in the routing table, such as through a connected, static, or IGP-learned route, or BGP will not originate it.

If you want to originate a default route, the default must exist in the routing table first:

```cisco
ip route 0.0.0.0 0.0.0.0 198.51.100.1

router bgp 65010
 network 0.0.0.0
```

This advertises `0.0.0.0/0` only while the static default route is active in the RIB.

**Verification and troubleshooting**

BGP troubleshooting starts with session state, then route presence, then policy.

1. Check neighbor state.
2. Check whether prefixes are received and advertised.
3. Check whether the best path is selected.
4. Check whether the route is installed in the RIB.
5. Check next-hop reachability and filtering.

Useful commands:

```bash
show ip bgp summary
show ip bgp neighbors
show ip bgp
show ip route bgp
show ip protocols
```

What to look for:

- **Idle** or **Active** state usually means the session is not established.
- **Established** means the BGP session is up and routes can be exchanged.
- Missing prefixes may indicate route filtering, incorrect `network` statements, or next-hop problems.
- A route in BGP but not in the RIB may indicate a better competing route or an unreachable next hop.

Example troubleshooting checks:

```bash
show ip bgp 198.51.100.0
show ip bgp neighbors 203.0.113.1 advertised-routes
show ip bgp neighbors 203.0.113.1 received-routes
show ip route 198.51.100.0
```

**Common enterprise policy techniques**

BGP is valuable because it lets you shape traffic with policy. Common techniques include:

- **Local preference** to choose the preferred outbound ISP for the whole AS.
- **AS_PATH prepending** to make one advertised path less attractive to remote networks.
- **Route maps** to match prefixes and set attributes.
- **Prefix lists** to control which routes are accepted or advertised.
- **Communities** to tag routes for downstream policy handling.

Example route map that increases local preference for routes learned from ISP 1:

```cisco
ip prefix-list DEFAULT-ONLY permit 0.0.0.0/0

route-map ISP1-IN permit 10
 match ip address prefix-list DEFAULT-ONLY
 set local-preference 200

router bgp 65010
 neighbor 203.0.113.1 route-map ISP1-IN in
```

This tells the enterprise to prefer routes learned from ISP 1 when the policy matches.

Example AS_PATH prepending for outbound advertisement:

```cisco
route-map PREPEND-OUT permit 10
 set as-path prepend 65010 65010 65010

router bgp 65010
 neighbor 203.0.113.1 route-map PREPEND-OUT out
```

This makes the advertised path look longer, which can reduce its attractiveness to external networks.

**When BGP is the right tool**

Use BGP when you need one or more of the following:

- Multiple upstream connections.
- Fine-grained control over route advertisement.
- Traffic engineering across WAN or Internet links.
- Scalability across many prefixes and peers.
- Policy separation between internal and external routing domains.

Do not use BGP just because it exists. For small internal networks, an IGP is usually simpler and faster to operate. BGP adds configuration complexity and requires careful policy design.

**CCNA-level takeaways**

For exam purposes, focus on these points:

- BGP is a **path-vector** routing protocol used between ASes.
- It runs over **TCP port 179**, but routes are exchanged only after the session reaches **Established**.
- eBGP connects different ASes; iBGP operates within the same AS.
- BGP selects a best path using attributes such as local preference, AS_PATH, MED, and weight.
- A route may be in the BGP table but not installed in the routing table.
- Enterprise use cases center on redundancy, policy control, and traffic engineering.
- `network` statements advertise only prefixes already present in the routing table.
