## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Troubleshooting mindset**

Most network faults fall into a small set of categories: physical connectivity, VLAN and trunking errors, IP addressing mistakes, routing failures, name resolution issues, access control blocks, and service-specific problems such as DHCP, NAT, or wireless authentication. The fastest path to a fix is to verify the path hop by hop and compare the working side with the failing side.

A practical workflow is:

1. Identify the symptom precisely.
2. Determine the scope: one host, one VLAN, one site, or the whole network.
3. Check Layer 1 and Layer 2 first, then Layer 3 and services.
4. Compare configuration, status, and counters against a known-good device or interface.
5. Test again after each change.

**Common host-side misconfigurations**

Many “network” problems are actually endpoint problems. Before changing switches or routers, verify the client’s local settings.

- **IP address**: A wrong static address can place the host in the wrong subnet or create an address conflict.
- **Subnet mask**: An incorrect mask changes what the host believes is local versus remote.
- **Default gateway**: If the gateway is missing or wrong, the host can usually reach local devices but not remote networks.
- **DNS server**: If DNS fails, users may report “the internet is down” even when IP connectivity works.
- **Duplicate IP address**: Two devices using the same address can cause intermittent connectivity, ARP instability, or session drops.

Useful checks on a host include:

```bash
ipconfig /all
ping 192.168.10.1
ping 8.8.8.8
nslookup www.example.com
arp -a
```

On Linux or macOS, use:

```bash
ip addr show
ip route
ping -c 4 192.168.10.1
dig www.example.com
```

If a host can ping an IP address but not a hostname, suspect DNS. If it can reach the local gateway but not remote networks, suspect the default gateway, routing, NAT, or ACLs.

**Layer 1 and Layer 2 faults**

Physical and switching issues often show up as link-down states, flapping interfaces, or one-way connectivity.

- **Bad cable or optics**: A damaged cable, wrong fiber type, or unsupported transceiver can prevent link establishment.
- **Speed/duplex mismatch**: Modern Ethernet usually negotiates automatically, but mismatches can still occur on legacy or manually configured links and cause errors, collisions, or poor throughput.
- **Err-disabled ports**: Security features such as port security, BPDU Guard, or storm control can shut a port down.
- **Wrong VLAN assignment**: An access port in the wrong VLAN places the host in the wrong broadcast domain.
- **Trunk problems**: Missing trunking, wrong native VLAN, or an allowed VLAN list that excludes the needed VLAN can block traffic.

Verify interface state and errors with:

```bash
show interfaces status
show interfaces g0/1
show interfaces counters errors
show logging
```

Example output clues:

```text
GigabitEthernet0/1 is up, line protocol is up
  5 minute input rate 0 bits/sec, 0 packets/sec
  12 input errors, 0 CRC, 12 frame, 0 overrun
```

Input errors and frame errors often point to cabling, duplex, or physical-layer issues. A port that is `err-disabled` usually needs both the root cause fixed and the interface re-enabled.

**VLAN, trunk, and inter-VLAN routing problems**

VLAN faults are common because traffic must match on both ends of the path. A host in VLAN 20 cannot reach another VLAN unless inter-VLAN routing is working.

Typical misconfigurations include:

- **Access port in the wrong VLAN**
- **Trunk not formed** because one side is access and the other is trunk
- **Native VLAN mismatch** on an 802.1Q trunk
- **Allowed VLAN list missing the required VLAN**
- **Missing VLAN on the switch**
- **SVI down/down** because the VLAN has no active ports or is not present
- **Routed port misconfiguration** on Layer 3 switches
- **Missing `ip routing`** on a multilayer switch

Check VLAN and trunk status with:

```bash
show vlan brief
show interfaces trunk
show interfaces g0/1 switchport
show ip interface brief
show running-config interface vlan 20
```

A healthy SVI typically appears `up/up` only when the VLAN exists and at least one port in that VLAN is active. If `interface vlan 20` is `down/down`, verify:

- VLAN 20 exists
- At least one access port in VLAN 20 is up
- The trunk carries VLAN 20 if the SVI depends on upstream connectivity
- The SVI has an IP address and is not administratively shut down

Example SVI configuration:

```cisco
vlan 20
 name USERS

interface vlan 20
 ip address 192.168.20.1 255.255.255.0
 no shutdown

ip routing
```

For a routed port on a Layer 3 switch:

```cisco
interface g0/1
 no switchport
 ip address 10.1.1.1 255.255.255.252
 no shutdown
```

If a host in VLAN 20 cannot reach its gateway, first confirm the access port VLAN, then the SVI state, then the trunk path to the SVI if applicable.

**Routing and next-hop issues**

Routing failures often appear as “local works, remote does not.” The router may know the destination network, but the next hop or exit interface is wrong.

Common problems include:

- **Missing route**: No static route or no dynamic route exists for the destination.
- **Wrong mask**: A route points to the wrong prefix length and misses the intended network.
- **Unreachable next hop**: On Cisco IOS, a static route is installed only when the next hop is resolvable through the routing table or a valid exit interface.
- **Wrong administrative distance**: A less preferred route may not be used if a better route exists.
- **Asymmetric routing**: Return traffic takes a different path and is blocked by stateful devices or ACLs.

Verify routing with:

```bash
show ip route
show ip route static
show ip cef
show running-config | include ^ip route
traceroute 203.0.113.10
```

Example static route:

```cisco
ip route 203.0.113.0 255.255.255.0 10.1.1.2
```

If you use a link-local IPv6 next hop, you must specify the exit interface as well:

```cisco
ipv6 route 2001:db8:20:1::/64 GigabitEthernet0/0 fe80::2
```

For IPv6, also verify that IPv6 routing is enabled:

```cisco
ipv6 unicast-routing
show ipv6 route
show ipv6 interface brief
```

**IPv4 and IPv6 addressing faults**

Addressing mistakes are among the most frequent exam scenarios because they are easy to overlook and have clear symptoms.

- **Wrong subnet mask**: The host may ARP for remote devices it thinks are local, or it may send local traffic to the gateway unnecessarily.
- **Wrong prefix length in IPv6**: A host may misclassify on-link versus off-link destinations.
- **Duplicate address**: Can cause intermittent reachability and ARP table instability in IPv4.
- **Missing default route**: Routers and hosts may not know where to send nonlocal traffic.

Useful verification commands:

```bash
show ip interface brief
show ipv6 interface brief
show arp
show ipv6 neighbors
ping
traceroute
```

For IPv4, `show arp` helps confirm whether the device resolves the gateway MAC address. For IPv6, `show ipv6 neighbors` shows Neighbor Discovery Protocol (NDP) entries.

**DHCP, SLAAC, and DHCPv6 issues**

Dynamic addressing problems usually show up as self-assigned or incomplete addressing, inability to reach the gateway, or missing DNS information.

For IPv4 DHCP, common causes include:

- **No DHCP server reachable**
- **Wrong VLAN or trunk path to the DHCP server**
- **Missing DHCP relay** on the router or SVI
- **Address pool exhausted**
- **Excluded addresses overlap the pool**
- **Wrong default gateway or DNS options in the pool**

For IPv6, clients may use Stateless Address Autoconfiguration (SLAAC), DHCPv6, or both depending on the network design.

- **SLAAC issue**: Router Advertisements (RAs) are missing or blocked.
- **DHCPv6 issue**: The DHCPv6 server is unreachable or the client is not configured for the correct mode.
- **RA guard or ACL issue**: Control-plane or Layer 2 filtering can block advertisements.

Verify DHCP and DHCPv6 with:

```bash
show ip dhcp binding
show ip dhcp pool
show ip dhcp server statistics
show ipv6 dhcp binding
show ipv6 neighbors
debug ip dhcp server events
debug ipv6 dhcp detail
```

Example DHCP relay on an SVI:

```cisco
interface vlan 20
 ip helper-address 10.10.10.10
```

Example DHCPv6 server configuration:

```cisco
ipv6 unicast-routing

ipv6 dhcp pool USERS-V6
 address prefix 2001:db8:20:1::/64
 dns-server 2001:db8:53::53
 domain-name example.com

interface vlan 20
 ipv6 address 2001:db8:20:1::1/64
 ipv6 nd managed-config-flag
 ipv6 dhcp server USERS-V6
```

If a client gets a link-local address in IPv6 but no global address, check whether RAs are present and whether the interface is configured for SLAAC or DHCPv6 as intended.

**ACL and firewall filtering**

Access control lists (ACLs) are a frequent source of “it pings one way but not the other” problems. A packet may be routed correctly and still be dropped by a filter.

Troubleshooting priorities:

- **ACL placement**: Confirm whether the ACL is applied on the correct interface.
- **Direction**: Verify whether it is applied `in` or `out`.
- **Order of entries**: ACLs are processed top-down, first match wins.
- **Implicit deny**: Anything not explicitly permitted is denied at the end.
- **Protocol and port match**: A rule may permit ICMP but block TCP, or permit one subnet but not another.

Verify ACLs with:

```bash
show access-lists
show ip access-lists
show running-config | include access-group
show ip interface g0/0
```

Example:

```cisco
ip access-list extended WEB-FILTER
 permit tcp 192.168.10.0 0.0.0.255 host 203.0.113.10 eq 443
 deny ip any any log

interface g0/0
 ip access-group WEB-FILTER in
```

If traffic is unexpectedly blocked, check hit counts and confirm the ACL direction matches the traffic flow you are testing. A deny entry with increasing counters confirms the ACL is matching traffic.

**NAT and PAT troubleshooting**

Network Address Translation (NAT) and Port Address Translation (PAT) problems often appear as internal hosts reaching local resources but failing to access the internet.

Key concepts:

- **Inside local**: The private address of the internal host.
- **Inside global**: The public address used after translation.
- **Outside local**: The address of the external host as seen from inside, if translated.
- **Outside global**: The real address of the external host.

Common issues include:

- **Inside/outside interfaces reversed**
- **Missing ACL that defines the inside addresses to translate**
- **Wrong NAT pool or overload configuration**
- **No default route toward the ISP**
- **Overlapping private address ranges**
- **Translation table not populating**

Verify NAT with:

```bash
show ip nat translations
show ip nat statistics
show running-config | include ip nat
show access-lists
```

Example PAT overload:

```cisco
access-list 1 permit 192.168.10.0 0.0.0.255

interface g0/0
 ip nat inside

interface g0/1
 ip nat outside

ip nat inside source list 1 interface g0/1 overload
```

Sample translation table:

```text
Pro  Inside global      Inside local       Outside local      Outside global
tcp  203.0.113.5:49152  192.168.10.25:49152 198.51.100.10:443 198.51.100.10:443
```

This tells you that host `192.168.10.25` is translated to `203.0.113.5` for a TCP session to `198.51.100.10:443`. If the table stays empty, verify the ACL, interface roles, and whether traffic is actually crossing the NAT boundary.

**Wireless troubleshooting**

Wireless problems can be caused by RF, authentication, or controller-related issues. A powered AP does not guarantee client connectivity.

Common causes include:

- **Wrong SSID** or security settings on the client
- **Authentication failure** due to incorrect passphrase, 802.1X credentials, or RADIUS issues
- **Association failure** because the client cannot join the AP
- **Channel overlap** or excessive co-channel interference
- **Low signal strength** or high attenuation
- **AP join failure** to a wireless LAN controller (WLC) in controller-based designs
- **Incorrect VLAN mapping** for the SSID
- **Client stuck on 2.4 GHz or 5 GHz due to band steering behavior**

Useful checks depend on the platform, but common verification includes:

```bash
show wireless summary
show ap summary
show wlan summary
show dot11 associations
show logging
```

On a controller, verify that the AP has joined, the WLAN is enabled, and the SSID maps to the correct VLAN or policy. On the client side, check whether the device is in the states of scanning, authenticating, associating, and obtaining an IP address.

A simple troubleshooting sequence is:

1. Confirm the AP is online and broadcasting the expected SSID.
2. Confirm the client can see the SSID and authenticate.
3. Confirm the client associates and receives an IP address.
4. Confirm the client can reach the default gateway.
5. Confirm DNS and external connectivity.

**Redundancy and first-hop gateway issues**

First-hop redundancy protocols such as HSRP (Hot Standby Router Protocol) protect the default gateway, but misconfiguration can leave hosts pointing at the wrong device or a standby router that is not forwarding.

Common problems include:

- **Virtual IP mismatch**: Hosts use a gateway address that does not match the active group.
- **Priority and preemption confusion**: The intended active router may not take over after recovery.
- **Interface tracking failure**: A router stays active even though its upstream link is down.
- **Authentication mismatch**: Peers do not form a valid redundancy relationship.
- **VLAN or trunk issue**: The virtual gateway is unreachable because the SVI or trunk is down.

Verify with:

```bash
show standby brief
show standby
show ip interface brief
show interfaces trunk
```

Example HSRP configuration:

```cisco
interface vlan 10
 ip address 192.168.10.2 255.255.255.0
 standby 10 ip 192.168.10.1
 standby 10 priority 110
 standby 10 preempt
 standby 10 track g0/1 20
```

If clients cannot reach the gateway, confirm that the virtual IP is configured on both peers, the VLAN is active, and the trunk carries the VLAN to both switches.

**Packet-flow examples**

A packet-flow view helps you isolate where traffic stops.

Example: a PC in VLAN 20 cannot reach a server on another subnet.

- The PC sends traffic to its default gateway because the destination is remote.
- The switch forwards the frame to the SVI or router interface for VLAN 20.
- The router performs a route lookup and forwards toward the next hop.
- An ACL, missing route, NAT issue, or return-path problem may drop the packet.
- If the return path fails, the server may reply but the PC never receives the response.

Example: a host can ping the gateway but not the internet.

- Local VLAN and gateway are likely correct.
- Check the default route on the router.
- Check NAT/PAT translation.
- Check ACLs on the WAN or outside interface.
- Check the ISP-facing interface status and next hop reachability.

**Troubleshooting best practices**

- Start with the simplest explanation and the closest device to the fault.
- Change one thing at a time so you can verify cause and effect.
- Save a working baseline configuration for comparison.
- Use `show` commands before `debug` commands whenever possible.
- Be careful with `debug` on production devices because it can increase CPU usage.
- Document the symptom, the test, the result, and the fix.

**Quick exam cues**

- If only remote networks fail, suspect the default gateway, routing, NAT, or ACLs.
- If only one VLAN fails, suspect access VLAN, trunk allowed VLANs, or SVI state.
- If a host gets an address but no internet, suspect DNS, NAT, or ACLs.
- If wireless clients see the SSID but cannot connect, suspect authentication, association, or RF issues.
- If a gateway fails over incorrectly, inspect HSRP state, priority, preemption, and tracking.
