## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**WAN edge lab overview**

A WAN edge topology usually combines an inside LAN, an edge router, a provider or upstream router, and one or more remote networks. The goal is to practice the full path of traffic as it leaves the LAN, crosses the WAN, and returns through routing, NAT, and filtering controls.

This kind of lab is valuable because it forces you to verify more than one feature at a time:

- **Layer 1/2** link state and interface status
- **Layer 3** addressing and routing
- **NAT** behavior for inside-to-outside traffic
- **ACLs** for traffic control
- **IPv6** dual-stack reachability
- **Serial encapsulation** and WAN interface troubleshooting

A good lab workflow is to build the topology, configure addressing, add routing, then validate each feature in isolation before combining them.

**Reference topology and addressing plan**

Use a simple three-router design with one LAN switch and one or two hosts.

- **R1**: edge router for the inside LAN
- **R2**: upstream/provider router
- **R3**: remote site router or test network
- **SW1**: access switch for the inside LAN
- **PC1**: inside host
- **PC2**: remote host or test endpoint

Example addressing plan:

| Device | Interface | IPv4 | IPv6 |
|---|---|---:|---:|
| R1 | G0/0 | 192.168.10.1/24 | 2001:db8:10:1::1/64 |
| R1 | S0/0/0 | 203.0.113.2/30 | 2001:db8:203:1::2/64 |
| R2 | S0/0/0 | 203.0.113.1/30 | 2001:db8:203:1::1/64 |
| R2 | G0/0 | 198.51.100.1/24 | 2001:db8:198:51::1/64 |
| R3 | G0/0 | 198.51.100.2/24 | 2001:db8:198:51::2/64 |
| PC1 | NIC | 192.168.10.10/24 | 2001:db8:10:1::10/64 |
| PC2 | NIC | 198.51.100.10/24 | 2001:db8:198:51::10/64 |

> Use documentation prefixes in labs such as `192.0.2.0/24`, `198.51.100.0/24`, and `2001:db8::/32` to avoid conflicts with real networks.

**Physical and logical build**

Start by connecting the devices in a way that matches the traffic path you want to test.

- Connect the inside host to the switch with a straight-through Ethernet cable.
- Connect the switch to the edge router’s LAN interface.
- Connect the edge router to the upstream router using the WAN link required by the lab.
- If the lab uses serial interfaces, make sure one side provides clocking with `clock rate` on the DCE side.
- On modern Cisco hardware and simulators, Auto-MDIX often removes the need to choose crossover manually for Ethernet links.

For serial labs, confirm the encapsulation before troubleshooting higher-layer issues:

```bash
show controllers serial 0/0/0
show interfaces serial 0/0/0
show running-config interface serial 0/0/0
```

Look for these common indicators:

- **DCE/DTE** role on the serial cable
- **Encapsulation** type, such as HDLC or PPP
- **Line protocol** state
- **Keepalive** or framing errors if the link is unstable

**Base device configuration**

Configure hostnames, disable DNS lookup on routers, and set interface descriptions so the topology is easy to read during troubleshooting.

```bash
conf t
hostname R1
no ip domain-lookup
service password-encryption
banner motd ^CUnauthorized access prohibited^C
```

Apply interface configuration on the edge router:

```bash
conf t
interface g0/0
 description Inside LAN
 ip address 192.168.10.1 255.255.255.0
 ipv6 address 2001:db8:10:1::1/64
 no shutdown

interface s0/0/0
 description WAN to R2
 ip address 203.0.113.2 255.255.255.252
 ipv6 address 2001:db8:203:1::2/64
 encapsulation ppp
 no shutdown
```

If the interface is DCE, set clocking on that side:

```bash
interface s0/0/0
 clock rate 64000
```

On the upstream router:

```bash
conf t
hostname R2
no ip domain-lookup

interface s0/0/0
 description WAN to R1
 ip address 203.0.113.1 255.255.255.252
 ipv6 address 2001:db8:203:1::1/64
 encapsulation ppp
 no shutdown

interface g0/0
 description Remote LAN
 ip address 198.51.100.1 255.255.255.0
 ipv6 address 2001:db8:198:51::1/64
 no shutdown
```

On the remote router or test endpoint router:

```bash
conf t
hostname R3
no ip domain-lookup

interface g0/0
 description Remote test LAN
 ip address 198.51.100.2 255.255.255.0
 ipv6 address 2001:db8:198:51::2/64
 no shutdown
```

**Host-side configuration**

Do not skip the PCs. A lab is incomplete until the end hosts can actually send traffic.

Configure PC1:

- IPv4 address: `192.168.10.10`
- Subnet mask: `255.255.255.0`
- Default gateway: `192.168.10.1`
- IPv6 address: `2001:db8:10:1::10/64`
- IPv6 default gateway: `2001:db8:10:1::1`

Configure PC2 or the remote test host:

- IPv4 address: `198.51.100.10`
- Subnet mask: `255.255.255.0`
- Default gateway: `198.51.100.1`
- IPv6 address: `2001:db8:198:51::10/64`
- IPv6 default gateway: `2001:db8:198:51::1`

If the lab platform uses a GUI host, verify the settings in the IP configuration panel. If it uses a command-line host, use the platform’s equivalent of `ip addr`, `route`, or the simulator’s desktop IP settings.

**Routing the topology**

Routing must exist in both directions. Forward traffic alone is not enough; return traffic must know how to get back.

For IPv4, add static routes on the edge and upstream routers:

```bash
R1(config)# ip route 198.51.100.0 255.255.255.0 203.0.113.1
R2(config)# ip route 192.168.10.0 255.255.255.0 203.0.113.2
R2(config)# ip route 198.51.100.0 255.255.255.0 198.51.100.2
R3(config)# ip route 192.168.10.0 255.255.255.0 198.51.100.1
```

For IPv6, add static routes with the next-hop address or exit interface as appropriate:

```bash
R1(config)# ipv6 route 2001:db8:198:51::/64 2001:db8:203:1::1
R2(config)# ipv6 route 2001:db8:10:1::/64 2001:db8:203:1::2
R2(config)# ipv6 route 2001:db8:198:51::/64 2001:db8:198:51::2
R3(config)# ipv6 route 2001:db8:10:1::/64 2001:db8:198:51::1
```

> A return route must point toward the router that can actually forward the packet to the destination. In a WAN edge lab, the upstream router typically needs a route back to the inside LAN via the edge router, not via its own directly connected remote side.

Verify routing with:

```bash
show ip route
show ipv6 route
show ip protocols
```

Look for:

- Connected routes for each active interface
- Static routes with the correct next hop
- No missing route to the remote LAN
- No accidental overlap or wrong mask

**NAT configuration and behavior**

NAT translates private inside addresses to a routable outside address. In CCNA labs, you usually practice **PAT** (Port Address Translation), also called NAT overload, because it allows many inside hosts to share one public address.

On R1, define inside and outside interfaces:

```bash
conf t
interface g0/0
 ip nat inside

interface s0/0/0
 ip nat outside
```

Create an ACL that matches inside source addresses:

```bash
access-list 1 permit 192.168.10.0 0.0.0.255
```

Apply PAT using the WAN interface address:

```bash
ip nat inside source list 1 interface s0/0/0 overload
```

This is the most common CCNA pattern because it is simple and realistic for a small edge router.

Validate NAT with:

```bash
show ip nat translations
show ip nat statistics
show access-lists
```

A translation table may show entries similar to this pattern:

```bash
Pro  Inside global      Inside local       Outside local      Outside global
icmp 203.0.113.2:1      192.168.10.10:1    198.51.100.10:1    198.51.100.10:1
tcp  203.0.113.2:1025   192.168.10.10:1025 198.51.100.10:80   198.51.100.10:80
```

Exact formatting varies by IOS version and protocol, so treat the output as a pattern rather than a fixed string.

Generate traffic to create translations:

```bash
ping 198.51.100.10
traceroute 198.51.100.10
```

For IPv6, NAT is usually not part of the CCNA dual-stack lab unless the task explicitly asks for it. In most cases, IPv6 uses native routing rather than translation.

**ACL placement and traffic control**

ACLs are often used on the WAN edge to filter traffic entering from the provider side or leaving the inside network. The key exam concept is that ACL direction and placement determine what is matched.

A common inbound WAN ACL on R1 might block unwanted traffic from the upstream side while allowing established return traffic and management exceptions.

Example:

```bash
ip access-list extended WAN-IN
 permit tcp any host 203.0.113.2 established
 permit icmp any any echo-reply
 permit icmp any any time-exceeded
 deny ip any 192.168.10.0 0.0.0.255
 permit ip any any
```

Apply it inbound on the WAN interface:

```bash
interface s0/0/0
 ip access-group WAN-IN in
```

Use care with ICMP rules. If you deny `echo` toward the inside subnet, you are blocking ping requests destined for inside hosts. That is not the same as blocking return traffic for pings initiated from inside, which uses `echo-reply`.

A more precise lab approach is:

- Permit the specific ICMP types you need for troubleshooting.
- Deny only the traffic you intend to block.
- Test both directions after applying the ACL.

Verify ACL behavior with:

```bash
show access-lists
show ip interface s0/0/0
```

Check the hit counts to confirm whether the ACL is matching the traffic you expect.

**IPv6 dual-stack workflow**

IPv6 troubleshooting is easier when you verify each layer in order: interface state, address assignment, neighbor discovery, then routing.

On routers, confirm IPv6 forwarding is enabled:

```bash
conf t
ipv6 unicast-routing
```

Verify interface addresses and link-local addresses:

```bash
show ipv6 interface brief
show ipv6 interface g0/0
show ipv6 interface s0/0/0
```

On hosts, confirm the IPv6 address, prefix length, and default gateway. If the host uses SLAAC (Stateless Address Autoconfiguration), verify that router advertisements are being sent. If the lab uses static addressing, ensure the host and router are in the same /64.

Test reachability in stages:

1. Ping the local default gateway from PC1.
2. Ping the remote router’s WAN or LAN address.
3. Ping the remote host.
4. Trace the path if the ping fails.

Example commands:

```bash
ping 2001:db8:10:1::1
ping 2001:db8:203:1::1
ping 2001:db8:198:51::10
traceroute ipv6 2001:db8:198:51::10
```

Check neighbor discovery when IPv6 fails unexpectedly:

```bash
show ipv6 neighbors
show ipv6 route
```

If neighbors are missing, suspect:

- Wrong prefix length
- Interface shutdown
- Missing `ipv6 unicast-routing`
- Incorrect default gateway on the host
- Layer 2 connectivity problems

**Serial WAN encapsulation and troubleshooting**

Serial links are still useful in labs because they force you to understand WAN interface behavior. The two common encapsulations are **HDLC** and **PPP**.

Use these checks when a serial link is down:

```bash
show interfaces serial 0/0/0
show controllers serial 0/0/0
show running-config interface serial 0/0/0
```

Interpret the output carefully:

- `Serial0/0/0 is up, line protocol is up` means Layer 1 and Layer 2 are working.
- `up/down` often indicates a Layer 2 mismatch, such as encapsulation or keepalive issues.
- `administratively down/down` means the interface is shut.
- `down/down` often points to cabling, clocking, or hardware issues.

If PPP is configured, you can verify authentication and negotiation details:

```bash
show ppp all
show ppp multilink
```

For CCNA-level labs, you usually only need to know that both ends must match encapsulation and that the DCE side must provide clocking when required.

**End-to-end validation sequence**

Use a repeatable validation order so you can isolate failures quickly.

1. Verify physical and interface status.
   - Check `show ip interface brief`.
   - Check `show ipv6 interface brief`.
   - Confirm interfaces are `up/up`.

2. Verify Layer 2 WAN settings.
   - Check serial encapsulation.
   - Confirm clock rate on the DCE side.
   - Confirm no encapsulation mismatch.

3. Verify addressing.
   - Check IPv4 and IPv6 addresses on routers.
   - Confirm host IP, mask, and default gateway.

4. Verify routing.
   - Check static routes on all routers.
   - Confirm the return path exists.

5. Verify NAT.
   - Generate traffic from PC1 to the remote host.
   - Check translation entries and counters.

6. Verify ACLs.
   - Confirm the ACL is applied in the correct direction.
   - Check hit counts and test both allowed and denied traffic.

7. Verify IPv6 reachability.
   - Ping the local gateway.
   - Ping the remote endpoint.
   - Check neighbor discovery and routes if needed.

**Practical troubleshooting examples**

If PC1 cannot reach PC2 over IPv4:

- Confirm PC1 default gateway is `192.168.10.1`.
- Confirm R1 has a route to `198.51.100.0/24`.
- Confirm R2 has a route back to `192.168.10.0/24`.
- Confirm NAT is not accidentally applied to the wrong interface.
- Confirm the ACL is not blocking the return path.

If IPv6 ping fails but IPv4 works:

- Confirm `ipv6 unicast-routing` is enabled on routers.
- Confirm both ends use the same /64 on each link.
- Confirm the IPv6 static routes exist in both directions.
- Check `show ipv6 neighbors` for missing neighbor discovery entries.

If the serial link is down:

- Check whether the interface is shut.
- Check DCE clocking.
- Check encapsulation on both ends.
- Check for a cable or simulator wiring issue.

**Expected results and success criteria**

A completed WAN edge lab should produce these results:

- PC1 can ping its default gateway.
- R1 can reach R2 over the WAN link.
- R1 can reach the remote IPv4 and IPv6 networks.
- PC1 can reach the remote host using IPv4 and IPv6.
- NAT translations appear when inside-to-outside IPv4 traffic is generated.
- ACL counters increase when matching traffic crosses the WAN edge.
- Serial interfaces show `up/up` with matching encapsulation.
- IPv6 neighbor and route tables contain the expected entries.

If all of those checks pass, your topology is functioning as a realistic WAN edge design and you have validated the core CCNA skills for routing, NAT, ACLs, and dual-stack operation.
