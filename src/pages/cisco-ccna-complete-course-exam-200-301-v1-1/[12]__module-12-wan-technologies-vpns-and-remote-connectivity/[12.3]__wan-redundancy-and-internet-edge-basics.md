**WAN redundancy and edge design goals**

A resilient Internet edge keeps users connected when a link, router, or upstream provider fails. At CCNA level, you should understand the difference between redundancy for the WAN path, redundancy for the default gateway, and redundancy for address translation state. These mechanisms solve different problems and do not all preserve active sessions.

- **WAN path redundancy** gives you alternate routes to remote networks or the Internet.
- **First-hop redundancy** keeps the default gateway available to hosts on the LAN.
- **NAT redundancy** is limited; translation state usually does not survive a device failover unless a stateful design is used.

A practical edge design often combines:
- Two edge routers or a dual-homed router
- Two ISP links or one primary and one backup link
- A first-hop redundancy protocol such as HSRP or VRRP
- NAT overload for inside users
- ACLs and firewall controls at the perimeter
- Tracking so the active path changes when the primary uplink fails

**WAN redundancy technologies**

At CCNA scope, redundancy can be static, dynamic, or link-level.

- **Static default routing** is simple and common for small networks.
  - You point the edge router to a primary next hop.
  - You add a floating static route with a higher administrative distance for backup.
- **Dynamic routing** can provide automatic path selection.
  - OSPF and EIGRP are common in enterprise internal designs.
  - BGP is used at the Internet edge when you connect to one or more ISPs or need policy control.
- **EtherChannel** bundles multiple physical links into one logical link.
  - It increases bandwidth and provides link redundancy between switches or between a switch and a router that supports it.
  - If one member link fails, the port-channel stays up as long as enough members remain.
- **Dual-homed edge design** connects a site to two upstream devices or two ISPs.
  - This improves availability.
  - It also introduces routing, NAT, and failover decisions that must be consistent on both paths.

> <mark>Redundancy is not the same as session continuity.</mark> A backup path can restore reachability while still breaking active TCP sessions if the source IP, NAT mapping, or return path changes.

**Static routing for Internet backup**

A floating static route is a backup route with a higher administrative distance than the primary route. The router installs the best route in the routing table and keeps the backup inactive until needed.

Example:
```bash
ip route 0.0.0.0 0.0.0.0 203.0.113.1
ip route 0.0.0.0 0.0.0.0 198.51.100.1 200
```

- The first default route is preferred.
- The second route is a backup because its administrative distance is `200`.
- If the primary next hop becomes unreachable, the router can use the backup route.

Use static defaults when:
- You have a small edge with one primary ISP and one backup ISP.
- You want predictable behavior.
- You do not need complex policy control.

Use dynamic routing when:
- You need automatic route exchange with an ISP or between edge routers.
- You need to advertise internal prefixes to an upstream device.
- You want better control over path selection and failover.

**BGP basics for the CCNA edge**

Border Gateway Protocol (BGP) is the Internet routing protocol used between autonomous systems (ASes). At CCNA level, focus on what it does rather than deep policy tuning.

- **Autonomous system**: a collection of networks under one administrative control.
- **Neighbor establishment**: BGP peers form a TCP session on port `179`.
- **Path selection**: BGP chooses routes using attributes such as local preference and AS-path.
- **Default route advertisement**: an edge router can learn or advertise a default route depending on design.

Basic conceptual behavior:
- A router forms a BGP neighbor relationship with the ISP router.
- It exchanges prefixes and attributes.
- It prefers routes based on policy and path attributes, not just shortest hop count.
- It can advertise a default route to internal routers or receive one from the ISP.

Simple example:
```bash
router bgp 65001
 neighbor 203.0.113.2 remote-as 65010
 network 192.0.2.0 mask 255.255.255.0
```

Useful CCNA-level points:
- BGP is common at the edge, not inside a small campus.
- BGP is often used when you need control over inbound and outbound Internet paths.
- A single-homed site usually does not need full BGP complexity.
- A default route from the ISP is often enough for basic Internet access.

Verification:
```bash
show ip bgp summary
show ip bgp
show ip route
```

**First-hop redundancy: HSRP and VRRP**

First-hop redundancy protocols keep the default gateway available to hosts if one router fails. Hosts point to a virtual IP address instead of a physical router interface.

- **HSRP**: Hot Standby Router Protocol, Cisco proprietary.
- **VRRP**: Virtual Router Redundancy Protocol, open standard.
- **Virtual IP**: the gateway address used by hosts.
- **Virtual MAC**: the MAC address associated with the active virtual gateway.

Operational behavior:
- One router is active or master.
- Another router is standby or backup.
- The active device answers ARP requests for the virtual IP using the virtual MAC.
- If the active router fails, the standby takes over the virtual IP and virtual MAC role.

HSRP state names to know:
- `Initial`
- `Learn`
- `Listen`
- `Speak`
- `Standby`
- `Active`

VRRP state names to know:
- `Initialize`
- `Backup`
- `Master`

Election logic:
- The router with the higher priority usually wins.
- If priorities tie, the higher interface IP address can win.
- Preemption allows a higher-priority router to reclaim the active role after recovery.

HSRP example:
```bash
interface g0/0
 ip address 192.168.10.2 255.255.255.0
 standby 1 ip 192.168.10.1
 standby 1 priority 110
 standby 1 preempt
 standby 1 track g0/1 20
```

What this does:
- `192.168.10.1` is the virtual default gateway.
- Priority `110` makes this router preferred.
- `preempt` lets it take back the active role after recovery.
- Tracking lowers priority if the uplink fails.

Verification:
```bash
show standby brief
show standby
show ip arp
```

IPv6 note:
- HSRP and VRRP can also protect IPv6 default gateways.
- Hosts still use a virtual gateway concept, but IPv6 neighbor discovery replaces ARP.
- In IPv6 designs, you may also see router advertisements used with first-hop redundancy.

**Routing failover versus first-hop failover versus NAT state**

These three failover types are often confused.

- **Routing failover** changes the path used by routers.
  - Example: a floating static route becomes active.
  - Example: BGP withdraws a failed path and selects another.
- **First-hop failover** changes which router acts as the default gateway for hosts.
  - Example: HSRP standby becomes active.
  - Hosts keep using the same virtual IP.
- **NAT failover** concerns translation state.
  - If the active NAT device fails, existing sessions often break because the translation table is lost.
  - New sessions can work after failover if the backup device has the same inside/outside reachability and NAT policy.

> Stateful failover is not automatic in basic CCNA NAT designs. If the translation table is not synchronized, return traffic for existing sessions usually fails.

Practical implication:
- A host may keep the same default gateway through HSRP.
- The router may still lose NAT state if the active edge device changes.
- The user can often open new connections, but long-lived sessions may reset.

**NAT and edge translation basics**

Network Address Translation (NAT) lets private inside addresses reach the public Internet using one or more public addresses.

Common types:
- **Static NAT**: one inside local address maps to one inside global address.
- **Dynamic NAT**: inside hosts use a pool of public addresses.
- **PAT**: Port Address Translation, also called NAT overload, maps many inside hosts to one public IP using different source ports.

Typical inside/outside interface roles:
- `ip nat inside` on the LAN-facing interface
- `ip nat outside` on the ISP-facing interface

Example:
```bash
interface g0/0
 description Inside LAN
 ip address 192.168.10.1 255.255.255.0
 ip nat inside

interface g0/1
 description ISP
 ip address 203.0.113.2 255.255.255.252
 ip nat outside

access-list 1 permit 192.168.10.0 0.0.0.255

ip nat inside source list 1 interface g0/1 overload
```

How NAT works in practice:
- The router matches inside traffic against the NAT rule.
- It translates the source address before sending traffic out the outside interface.
- Return traffic is translated back to the inside local address.
- PAT uses port numbers to keep many sessions distinct.

NAT order of operations matters:
- The router applies routing and NAT in a specific sequence.
- If the inside/outside roles are wrong, translations fail.
- If the ACL does not match the inside source network, no translation occurs.
- If the route to the destination or default route is missing, NAT alone does not fix reachability.

Verification commands:
```bash
show ip nat translations
show ip nat statistics
show running-config | section ip nat
show ip interface brief
show ip route
```

Troubleshooting clues:
- `show ip nat translations` should display active mappings.
- `show ip nat statistics` helps confirm hits, misses, and pool usage.
- `show ip interface brief` confirms the interface is up/up.
- `show running-config interface g0/0` confirms `ip nat inside` or `ip nat outside`.
- `debug ip nat` can help in a lab, but use caution on production devices.

**NAT and failover limitations**

NAT is a common weak point in edge redundancy.

- If the active router fails, the backup router may not know existing translations.
- New sessions can work if the backup has the same public address and policy.
- Existing TCP sessions often reset because sequence tracking and translation state are lost.
- Stateful NAT failover requires special synchronization features that are beyond basic CCNA design.

Design considerations:
- Keep NAT simple when possible.
- Prefer a design where the same virtual gateway and same public address remain reachable after failover.
- Use tracking so the active router changes before the upstream path fully fails.

**Tracking and object-based failover**

Interface tracking lets a router reduce priority or remove a route when a critical link fails.

Example with HSRP tracking:
```bash
track 10 interface g0/1 line-protocol
interface g0/0
 standby 1 track 10 decrement 30
```

What happens:
- The router monitors the uplink.
- If the uplink fails, the tracked object goes down.
- HSRP priority drops.
- The standby router can take over.

You can also track reachability with IP SLA in more advanced designs.

Example:
```bash
ip sla 1
 icmp-echo 8.8.8.8 source-interface g0/1
 frequency 5
ip sla schedule 1 life forever start-time now
track 1 ip sla 1 reachability
```

Use tracking when:
- The router is still physically up but the ISP path is dead.
- You want failover based on actual reachability, not just interface state.

**Internet edge security basics**

The edge is where you control traffic entering and leaving the network. At CCNA level, know the purpose of ACLs, basic firewall concepts, and control plane protection.

ACL basics:
- **Standard ACLs** match source IP only.
- **Extended ACLs** match source, destination, protocol, and ports.
- Apply ACLs close to the traffic source when filtering outbound traffic, or close to the destination when protecting a specific service.

Example extended ACL:
```bash
ip access-list extended EDGE-IN
 permit tcp any host 203.0.113.10 eq 443
 permit icmp any any echo-reply
 deny ip any any log

interface g0/1
 ip access-group EDGE-IN in
```

What to remember:
- ACLs are stateless.
- You must permit return traffic explicitly when needed.
- `log` helps with troubleshooting but can increase CPU load.

Control plane policing concept:
- The control plane is the part of the router that processes routing protocols, management traffic, and exceptions.
- Control plane policing limits how much traffic can reach the CPU.
- It helps protect the router from scans or floods.

Zone-Based Firewall basics:
- Interfaces belong to security zones.
- Traffic between zones is denied by default unless you create a policy.
- It is more stateful than ACLs and better suited for simple perimeter inspection.

**IPv6 edge considerations**

IPv6 changes some edge details but not the overall design goals.

- IPv6 uses a default route such as `::/0`.
- NAT is usually not required for IPv6 because addresses are globally routable.
- If translation is needed, you may hear about **NAT64** or **NPTv6**.
  - NAT64 translates between IPv6 and IPv4.
  - NPTv6 preserves prefix structure while translating prefixes, but it is not a common CCNA focus.
- First-hop redundancy still matters for IPv6 gateways.
- Neighbor Discovery replaces ARP for address resolution.

Example IPv6 default route:
```bash
ipv6 route ::/0 2001:db8:1::1
```

Example IPv6 HSRP-style gateway concept:
- Hosts use a virtual IPv6 gateway.
- The active router responds to neighbor discovery for that virtual address.
- Failover should preserve the same gateway address for hosts.

**End-to-end edge design example**

A common small-site design uses two routers, one LAN, one ISP, HSRP, tracking, and PAT.

Topology:
- `LAN` hosts connect to a switch.
- `R1` and `R2` connect to the LAN switch.
- `R1` and `R2` each connect to the ISP or to separate ISP links.
- HSRP provides a virtual default gateway to the LAN.
- NAT overload occurs on the active edge router.
- Tracking lowers priority if the primary WAN link fails.

Sample configuration:
```bash
! R1
interface g0/0
 description LAN
 ip address 192.168.10.2 255.255.255.0
 ip nat inside
 standby 1 ip 192.168.10.1
 standby 1 priority 110
 standby 1 preempt
 standby 1 track g0/1 30

interface g0/1
 description ISP
 ip address 203.0.113.2 255.255.255.252
 ip nat outside

access-list 1 permit 192.168.10.0 0.0.0.255
ip nat inside source list 1 interface g0/1 overload
ip route 0.0.0.0 0.0.0.0 203.0.113.1

! R2
interface g0/0
 description LAN
 ip address 192.168.10.3 255.255.255.0
 ip nat inside
 standby 1 ip 192.168.10.1
 standby 1 priority 100
 standby 1 preempt
 standby 1 track g0/1 30

interface g0/1
 description ISP
 ip address 198.51.100.2 255.255.255.252
 ip nat outside

access-list 1 permit 192.168.10.0 0.0.0.255
ip nat inside source list 1 interface g0/1 overload
ip route 0.0.0.0 0.0.0.0 198.51.100.1
```

How this design behaves:
- Hosts use `192.168.10.1` as the gateway.
- R1 starts as HSRP active because it has higher priority.
- R1 performs NAT for inside users.
- If R1’s WAN link fails, tracking reduces its priority.
- R2 becomes active and takes over the virtual gateway.
- New traffic can exit through R2, but existing NAT sessions may break.

**Verification workflow**

Use a consistent troubleshooting sequence.

1. Check physical and interface status.
   ```bash
   show ip interface brief
   show interfaces status
   ```
2. Check the routing table.
   ```bash
   show ip route
   show ipv6 route
   ```
3. Check first-hop redundancy.
   ```bash
   show standby brief
   show vrrp brief
   ```
4. Check NAT.
   ```bash
   show ip nat translations
   show ip nat statistics
   ```
5. Check ACLs and edge policy.
   ```bash
   show access-lists
   show policy-map control-plane
   ```
6. Test reachability.
   ```bash
   ping 8.8.8.8
   traceroute 8.8.8.8
   ```

**Exam focus summary**

- Know when to use static default routes versus dynamic routing.
- Understand that BGP is the Internet-edge routing protocol and uses neighbor relationships and path attributes.
- Distinguish routing failover, first-hop redundancy, and NAT state continuity.
- Know HSRP and VRRP roles, state names, virtual IP behavior, and election logic.
- Remember that NAT requires correct inside/outside roles, matching ACLs, and a valid route.
- Recognize that EtherChannel adds link redundancy and can support edge or uplink designs.
- Understand that ACLs, control plane policing, and zone-based firewalls are basic edge security tools.
- Be aware that IPv6 uses default routes and first-hop redundancy too, but usually not NAT in the same way as IPv4.
