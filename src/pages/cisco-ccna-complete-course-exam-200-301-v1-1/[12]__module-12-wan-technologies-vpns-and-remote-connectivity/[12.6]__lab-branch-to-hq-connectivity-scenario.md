## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Lab overview**

This lab models a small enterprise with a branch office, a headquarters site, and an Internet edge. Your goal is to make end-to-end connectivity work across IPv4 and IPv6 while validating routing, host settings, and NAT behavior. In a CCNA context, the important skill is not just entering commands, but understanding how each layer of the design supports reachability.

A typical branch-to-HQ scenario includes:
- A branch LAN with user hosts
- A branch router that connects to HQ over a WAN link
- An HQ router that connects to internal servers and/or an ISP edge
- A default route toward the Internet
- A routing method that exchanges internal prefixes between branch and HQ
- NAT for IPv4 Internet access when required

**Reference topology and addressing plan**

Use a complete addressing plan before configuring anything. A clear plan prevents overlap, makes verification easier, and helps you identify whether a failure is caused by Layer 1, Layer 2, Layer 3, or host misconfiguration.

| Device | Interface | IPv4 Address | IPv6 Address | Notes |
|---|---|---:|---:|---|
| Branch R1 | G0/0 | 192.168.10.1/24 | 2001:db8:10:1::1/64 | Branch LAN gateway |
| Branch R1 | S0/0/0 | 10.0.0.1/30 | 2001:db8:12:1::1/64 | WAN to HQ |
| HQ R2 | S0/0/0 | 10.0.0.2/30 | 2001:db8:12:1::2/64 | WAN to branch |
| HQ R2 | G0/0 | 192.168.20.1/24 | 2001:db8:20:1::1/64 | HQ LAN gateway |
| ISP R3 | G0/0 | 203.0.113.2/30 | — | Outside next hop for NAT |
| HQ R2 | G0/1 | 203.0.113.1/30 | — | NAT outside toward ISP |
| Branch PC1 | NIC | 192.168.10.10/24 | 2001:db8:10:1::10/64 | Default gateway to R1 |
| HQ Server | NIC | 192.168.20.10/24 | 2001:db8:20:1::10/64 | Default gateway to R2 |

> If your lab uses different prefixes, keep the design logic the same and substitute the correct values consistently.

**Core design goals**

- **Branch users** must reach HQ resources through the WAN.
- **HQ users** must reach branch resources through the WAN.
- **IPv4 Internet access** from inside networks should work through NAT overload if the lab includes an ISP.
- **IPv6 connectivity** should work end to end without NAT in the typical CCNA design.
- **Routing** should advertise internal networks on both sides so return traffic follows the correct path.

**Layer 1 and interface preparation**

Before routing works, interfaces must be enabled and correctly addressed. On Cisco routers, an interface stays down until you enable it with `no shutdown`. Serial links also require correct encapsulation if the lab specifies it, and the DCE side must provide clocking.

```bash
show ip interface brief
show ipv6 interface brief
show controllers serial 0/0/0
```

- **Interface state** should be `up/up` for active links.
- **Serial clocking** belongs on the DCE end only.
- **IPv6 routing** requires `ipv6 unicast-routing` on each router that forwards IPv6 packets.

Example router interface configuration:

```bash
conf t
hostname R1
ipv6 unicast-routing

interface g0/0
 description Branch LAN
 ip address 192.168.10.1 255.255.255.0
 ipv6 address 2001:db8:10:1::1/64
 no shutdown

interface s0/0/0
 description WAN to HQ
 ip address 10.0.0.1 255.255.255.252
 ipv6 address 2001:db8:12:1::1/64
 no shutdown
end
```

On the DCE serial interface only, you may need:

```bash
conf t
interface s0/0/0
 clock rate 64000
end
```

**Routing strategy for the lab**

For a CCNA branch-to-HQ lab, the most practical routing approach is usually **static routing** for the WAN and default routes, with optional dynamic routing if the lab explicitly asks for it. If the scenario includes OSPF, use it to exchange branch and HQ LAN prefixes. If the lab is simpler, static routes are often enough and easier to verify.

Use the routing method the lab requires:
- **Static routes** for direct control and simple topologies
- **OSPF** when the lab expects dynamic route exchange
- **RIP** only if the exercise specifically asks for it, since it is less common in modern designs

**Static routing example**

Static routing is useful when you know the exact next hop and want predictable behavior. In a branch-to-HQ design, each router needs a route to the remote LAN and often a default route toward the ISP or upstream edge.

Branch router:

```bash
conf t
ip route 192.168.20.0 255.255.255.0 10.0.0.2
ipv6 route 2001:db8:20:1::/64 2001:db8:12:1::2
end
```

HQ router:

```bash
conf t
ip route 192.168.10.0 255.255.255.0 10.0.0.1
ipv6 route 2001:db8:10:1::/64 2001:db8:12:1::1
end
```

For IPv6 static routes, the next hop must be reachable. If the next-hop address is not on a directly connected link, use an exit interface, a link-local next hop with an interface, or a valid directly connected global next hop as appropriate for the topology.

Example with an exit interface:

```bash
ipv6 route 2001:db8:20:1::/64 s0/0/0 2001:db8:12:1::2
```

**OSPF example for branch-to-HQ**

If the lab uses OSPF, advertise the internal networks on both routers. OSPF is a link-state routing protocol that converges quickly and scales better than static routes when more networks are added.

Branch router:

```bash
conf t
router ospf 1
 router-id 1.1.1.1
 network 192.168.10.0 0.0.0.255 area 0
 network 10.0.0.0 0.0.0.3 area 0
end
```

HQ router:

```bash
conf t
router ospf 1
 router-id 2.2.2.2
 network 192.168.20.0 0.0.0.255 area 0
 network 10.0.0.0 0.0.0.3 area 0
end
```

Verify adjacency and learned routes:

```bash
show ip ospf neighbor
show ip route ospf
show ip protocols
```

If IPv6 routing is included with OSPFv3, enable it per interface and process as required by the platform and lab instructions.

```bash
conf t
ipv6 unicast-routing
ipv6 router ospf 10
 router-id 10.10.10.10

interface g0/0
 ipv6 ospf 10 area 0

interface s0/0/0
 ipv6 ospf 10 area 0
end
```

**NAT for Internet access**

NAT (Network Address Translation) is commonly used on the IPv4 edge so private inside addresses can reach the public Internet. In CCNA labs, the most common form is **PAT** (Port Address Translation), also called NAT overload. It lets many inside hosts share one public IPv4 address.

A typical NAT design uses:
- `ip nat inside` on internal interfaces
- `ip nat outside` on the ISP-facing interface
- An ACL to match inside source addresses
- A NAT overload rule tied to the outside interface

Example on the HQ router:

```bash
conf t
interface g0/0
 ip nat inside

interface s0/0/0
 ip nat inside

interface g0/1
 ip nat outside

access-list 1 permit 192.168.10.0 0.0.0.255
access-list 1 permit 192.168.20.0 0.0.0.255

ip nat inside source list 1 interface g0/1 overload
end
```

Add a default route toward the ISP:

```bash
conf t
ip route 0.0.0.0 0.0.0.0 203.0.113.2
end
```

A full working NAT test flow looks like this:
1. Confirm the inside host has the correct default gateway.
2. Ping the HQ router from the branch host.
3. Ping the ISP next hop from the HQ router.
4. Ping an external address from the branch host.
5. Check translation entries on the NAT router.

Useful verification commands:

```bash
show ip nat translations
show ip nat statistics
show access-lists
show ip route
```

> NAT applies to IPv4 in this lab. IPv6 typically uses global routing without NAT, although translation technologies such as NAT64 and NPTv6 exist in other designs.

**IPv6 branch-to-HQ connectivity**

IPv6 lab success depends on correct addressing, enabled forwarding, and a route back to the source. Unlike IPv4 NAT-based designs, IPv6 usually relies on end-to-end routing.

Branch router example:

```bash
conf t
ipv6 unicast-routing

interface g0/0
 ipv6 address 2001:db8:10:1::1/64
 no shutdown

interface s0/0/0
 ipv6 address 2001:db8:12:1::1/64
 no shutdown

ipv6 route 2001:db8:20:1::/64 2001:db8:12:1::2
end
```

HQ router example:

```bash
conf t
ipv6 unicast-routing

interface g0/0
 ipv6 address 2001:db8:20:1::1/64
 no shutdown

interface s0/0/0
 ipv6 address 2001:db8:12:1::2/64
 no shutdown

ipv6 route 2001:db8:10:1::/64 2001:db8:12:1::1
end
```

Host IPv6 settings should include:
- A valid IPv6 address in the correct prefix
- A `/64` prefix length for the LAN
- The router interface as the default gateway

Example host configuration:
- **Branch PC1**
  - IPv6 address: `2001:db8:10:1::10`
  - Prefix length: `64`
  - Default gateway: `2001:db8:10:1::1`
- **HQ Server**
  - IPv6 address: `2001:db8:20:1::10`
  - Prefix length: `64`
  - Default gateway: `2001:db8:20:1::1`

Verify IPv6 reachability with:

```bash
ping ipv6 2001:db8:20:1::10
traceroute ipv6 2001:db8:20:1::10
show ipv6 route
show ipv6 neighbors
```

**Host configuration examples**

Host misconfiguration is one of the most common reasons a lab fails even when router configs are correct. Always verify the host IP, mask, gateway, and DNS if the lab includes name resolution.

Example IPv4 host settings:
- **Branch PC1**
  - IP address: `192.168.10.10`
  - Subnet mask: `255.255.255.0`
  - Default gateway: `192.168.10.1`
  - DNS server: `8.8.8.8` or the lab-provided DNS
- **HQ Server**
  - IP address: `192.168.20.10`
  - Subnet mask: `255.255.255.0`
  - Default gateway: `192.168.20.1`
  - DNS server: lab-specific

Example Windows CLI checks:

```powershell
ipconfig /all
ping 192.168.10.1
ping 192.168.20.10
tracert 192.168.20.10
```

Example Linux checks:

```bash
ip addr
ip route
ping -c 4 192.168.10.1
traceroute 192.168.20.10
```

**Verification checklist**

Use a structured validation sequence so you can isolate the fault quickly.

1. Verify physical and interface status with `show ip interface brief` and `show ipv6 interface brief`.
2. Confirm addressing on all router interfaces and hosts.
3. Check that serial interfaces are up and that the DCE side has clocking if required.
4. Verify routing tables on both routers with `show ip route` and `show ipv6 route`.
5. Confirm that branch and HQ can ping each other across the WAN.
6. Test host-to-router connectivity on each LAN.
7. If NAT is included, generate traffic to the outside and inspect translations.
8. Use traceroute to identify where packets stop.
9. Check ACLs, default routes, and next-hop reachability if traffic fails.
10. Re-test after each correction rather than changing multiple items at once.

**Troubleshooting patterns**

A methodical approach saves time in the lab. Most failures fall into a few predictable categories.

- **Wrong default gateway**
  - Hosts can reach local devices but not remote networks.
- **Missing return route**
  - One direction works, but replies never come back.
- **Interface shutdown**
  - The link stays down until you issue `no shutdown`.
- **Incorrect NAT role**
  - Inside and outside interfaces are reversed, so translations never occur.
- **ACL mismatch**
  - NAT or filtering rules do not match the source subnet.
- **Serial clocking issue**
  - The DCE side lacks `clock rate`, so the WAN never comes up.
- **IPv6 forwarding disabled**
  - The router has addresses but does not route IPv6 traffic.
- **Wrong prefix length**
  - A host or interface is placed in the wrong subnet, breaking neighbor discovery or routing.

**Practical end-to-end lab sequence**

Use this sequence when building the scenario from scratch:

1. Configure router hostnames and enable interfaces.
2. Assign IPv4 and IPv6 addresses to all router interfaces.
3. Enable `ipv6 unicast-routing` on each router.
4. Configure serial clocking on the DCE end if the lab uses serial WAN links.
5. Add static routes or OSPF, depending on the lab requirement.
6. Configure NAT on the edge router if Internet access is part of the scenario.
7. Configure host IPv4 and IPv6 settings.
8. Test local LAN connectivity first.
9. Test branch-to-HQ connectivity next.
10. Test Internet reachability last.
11. Verify routing tables, NAT translations, and neighbor tables.
12. Fix one issue at a time and retest.

**What to remember for the exam**

- **Routing must be symmetric enough** for return traffic to find its way back.
- **IPv4 private networks** need NAT only when they must reach a public IPv4 network.
- **IPv6** usually uses routed connectivity rather than NAT in CCNA labs.
- **Static routes** are simple and predictable; **OSPF** is better when the lab expects dynamic exchange.
- **Verification commands** are as important as configuration commands.
- **Host settings** are part of the solution, not an afterthought.

**Final lab mindset**

Treat the branch-to-HQ scenario as a complete system: interfaces, addressing, routing, NAT, and hosts all have to agree. If one piece is wrong, the symptom may appear far away from the real cause. Build the topology in layers, verify after each layer, and use the routing table and translation table as your primary truth sources.
