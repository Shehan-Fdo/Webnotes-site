## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**VPN fundamentals**

A <abbr title="Virtual Private Network">VPN</abbr> creates an encrypted tunnel over an untrusted network, usually the Internet. You use it to protect traffic confidentiality, integrity, and peer identity while allowing private communication between sites or users.

At the CCNA level, focus on two common VPN use cases:

- **Site-to-site VPN** connects two networks, usually branch-to-branch or branch-to-headend.
- **Remote access VPN** connects a single user device to a private network.

A VPN solution usually combines:
- **Encryption** to hide payload data.
- **Integrity** to detect tampering.
- **Authentication** to verify the peer.
- **Key exchange** to establish shared secrets securely.

**VPN types and when to use them**

Site-to-site VPNs are best when you want always-on connectivity between networks. Remote access VPNs are best when individual users need secure access from home, hotels, or other untrusted locations.

- **Site-to-site**
  - Common for branch office connectivity.
  - Usually transparent to end users.
  - Often built on IPsec between routers or firewalls.
- **Remote access**
  - Common for mobile workers and contractors.
  - Usually requires a VPN client such as Cisco AnyConnect.
  - Often uses AAA, group policy, and user profiles on the headend.

**IPsec architecture**

<abbr title="Internet Protocol Security">IPsec</abbr> is the main framework Cisco uses for secure VPN tunnels at this level. It protects IP traffic by negotiating security associations and then encrypting traffic between peers.

Two phases matter for exam understanding:

1. **IKE Phase 1** builds a secure control channel between peers.
2. **IKE Phase 2** negotiates the parameters used to protect user data.

In older Cisco terminology, Phase 1 is often called <abbr title="Internet Security Association and Key Management Protocol">ISAKMP</abbr> negotiation. On many platforms, `show crypto isakmp sa` is still seen in legacy examples, but modern Cisco documentation often emphasizes <abbr title="Internet Key Exchange version 2">IKEv2</abbr>.

Key distinctions:
- **IKE SA** protects the control-plane negotiation.
- **IPsec SA** protects the actual data traffic.
- One IKE SA can support one or more IPsec SAs depending on the design and platform.

> <mark>Do not confuse the IKE security association with the IPsec security association.</mark> The first secures negotiation; the second secures user traffic.

**Common IPsec building blocks**

A working IPsec design usually includes several selectable parameters. You should know what each one does and why it matters.

- **Encryption algorithm**
  - Examples: `aes`, `aes 256`, `3des` in older designs.
  - Stronger encryption increases confidentiality but may require more CPU.
- **Integrity algorithm**
  - Examples: `sha`, `sha256`, `md5` in legacy examples.
  - Integrity detects packet modification.
- **Diffie-Hellman (DH) group**
  - Used to create shared keying material.
  - Higher groups generally provide stronger security but may cost more processing.
- **Peer authentication**
  - Usually pre-shared keys or certificates.
  - Certificates scale better in larger deployments.
- **Lifetime**
  - Defines how long an SA remains valid before renegotiation.

A typical legacy-style IKEv1/IPsec configuration may look like this:

```cisco
crypto isakmp policy 10
 encr aes
 hash sha
 authentication pre-share
 group 14
 lifetime 86400

crypto isakmp key cisco123 address 203.0.113.2

crypto ipsec transform-set TS esp-aes esp-sha-hmac
 mode tunnel
```

This syntax is valid in many IOS examples, but it is best treated as a **legacy-style** model. In newer designs, you often see IKEv2 with proposals, policies, profiles, and transform sets or transform profiles depending on platform.

**Site-to-site VPN concepts**

A site-to-site VPN creates a secure tunnel between two gateways. End hosts usually do not know the tunnel exists; they simply send traffic to their local default gateway.

Typical topology:

- Branch LAN `192.168.10.0/24`
- Headend LAN `10.10.10.0/24`
- Public WAN addresses on both VPN peers

Traffic flow:
- A host in the branch sends traffic to the headend subnet.
- The branch router matches the traffic as “interesting” or policy-based traffic.
- The VPN peers negotiate IKE and IPsec.
- The packet is encrypted and sent across the Internet.
- The remote peer decrypts and forwards it into the destination LAN.

Common site-to-site deployment models:
- **Policy-based VPN**
  - Uses crypto maps and access lists to define interesting traffic.
  - Common in older Cisco IOS examples.
  - Simple and still useful for exam concepts.
- **Route-based VPN**
  - Uses a tunnel interface and routing over the tunnel.
  - Easier for dynamic routing and multiple subnets.
  - More flexible in modern designs.

A classic policy-based example:

```cisco
access-list 110 permit ip 192.168.10.0 0.0.0.255 10.10.10.0 0.0.0.255

crypto isakmp policy 10
 encr aes
 hash sha
 authentication pre-share
 group 14

crypto isakmp key cisco123 address 198.51.100.2

crypto ipsec transform-set TS esp-aes esp-sha-hmac

crypto map CMAP 10 ipsec-isakmp
 set peer 198.51.100.2
 set transform-set TS
 match address 110

interface GigabitEthernet0/0
 description Internet-facing
 ip address 203.0.113.1 255.255.255.252
 crypto map CMAP
```

Important exam points:
- The crypto map is applied to the **outside interface**.
- The ACL defines traffic that should be encrypted.
- NAT may need an exemption so protected traffic is not translated before encryption.

> <mark>Crypto map-based IPsec is common in CCNA examples, but it is not the only VPN model.</mark> Route-based designs and IKEv2 profiles are also widely used in real networks.

**Why GRE over IPsec exists**

<abbr title="Generic Routing Encapsulation">GRE</abbr> is often paired with IPsec when you need to carry more than simple unicast traffic between sites.

Use GRE over IPsec when you need:
- Dynamic routing protocols such as OSPF or EIGRP across the tunnel.
- Multicast or broadcast traffic.
- Multiple routed subnets without creating many crypto ACL entries.

Why GRE helps:
- IPsec protects the GRE tunnel.
- GRE provides a logical point-to-point tunnel interface.
- Routing protocols can run over the tunnel interface as if it were a real link.

Example concept:
- GRE creates a tunnel from `Tunnel0` on Router A to `Tunnel0` on Router B.
- IPsec encrypts the GRE packets between the public addresses.
- OSPF can then advertise routes through the tunnel.

A simplified GRE over IPsec example:

```cisco
interface Tunnel0
 ip address 172.16.0.1 255.255.255.252
 tunnel source GigabitEthernet0/0
 tunnel destination 198.51.100.2
 tunnel mode gre ip

crypto isakmp policy 10
 encr aes
 hash sha
 authentication pre-share
 group 14

crypto ipsec transform-set TS esp-aes esp-sha-hmac

crypto map CMAP 10 ipsec-isakmp
 set peer 198.51.100.2
 set transform-set TS
 match address 120

access-list 120 permit gre host 203.0.113.1 host 198.51.100.2
```

**Remote access VPN concepts**

Remote access VPNs let a single user connect securely to a private network from an untrusted location. Cisco deployments usually place a **VPN headend** at the edge of the enterprise network.

Common headend components:
- **AAA** for authentication, authorization, and accounting.
- **Group policy** to define user permissions and tunnel behavior.
- **Address pool** to assign client IP addresses.
- **Tunnel interface or virtual-access interface** to terminate user sessions.
- **Client software** such as Cisco AnyConnect for client-based access.

Remote access VPNs are usually more user-centric than site-to-site VPNs:
- The user authenticates with credentials, certificates, or both.
- The headend assigns policy and an IP address.
- The client receives routes, DNS settings, and access restrictions.

Common Cisco remote access model:
- A user launches the VPN client.
- The client connects to the public IP of the headend.
- The headend authenticates the user through local credentials or AAA.
- The headend applies a group policy.
- The client receives a virtual IP address and secure access to internal resources.

A conceptual configuration example:

```cisco
ip local pool RA_POOL 10.20.20.10 10.20.20.50

aaa new-model
aaa authentication login VPN_AUTH local
aaa authorization network VPN_AUTHZ local

username alice secret Str0ngP@ss

ip access-list standard VPN_SPLIT
 permit 10.10.10.0 0.0.0.255

crypto ikev2 proposal IKEV2-PROP
 encryption aes-cbc-256
 integrity sha256
 group 14

crypto ikev2 policy IKEV2-POLICY
 proposal IKEV2-PROP

crypto ikev2 keyring RA-KEYRING
 peer RA-PEER
  address 0.0.0.0 0.0.0.0
  pre-shared-key local cisco123
  pre-shared-key remote cisco123

crypto ikev2 profile RA-PROFILE
 match identity remote any
 authentication local pre-share
 authentication remote pre-share
 keyring local RA-KEYRING

! Platform-specific remote access designs may use virtual-template,
! virtual-access, or tunnel interfaces depending on IOS/IOS XE features.
```

Remote access access models:
- **Client-based VPN**
  - Uses software on the endpoint.
  - Common with AnyConnect.
  - Supports full tunnel or split tunnel.
- **Clientless VPN**
  - Uses a browser-based portal.
  - Limited to web-access use cases.
  - Less common in CCNA-level operational detail, but useful to recognize.

**SSL VPN and TLS VPN terminology**

Cisco documentation and products often use the terms **SSL VPN** and **TLS VPN** in overlapping ways. Strictly speaking, <abbr title="Transport Layer Security">TLS</abbr> is the modern successor to <abbr title="Secure Sockets Layer">SSL</abbr>, but many people still say SSL VPN as a generic label.

Practical exam takeaway:
- Treat SSL VPN as the older/common umbrella term.
- Understand that modern secure remote access often uses TLS-based tunnels.
- Cisco AnyConnect is the common client associated with this style of remote access.

**Split tunnel vs full tunnel**

Remote access clients can send all traffic through the VPN or only selected traffic.

- **Full tunnel**
  - All client traffic goes through the VPN.
  - Better for centralized inspection and policy enforcement.
  - Uses more headend bandwidth.
- **Split tunnel**
  - Only corporate traffic goes through the VPN.
  - Internet traffic exits locally from the client.
  - Reduces headend load but may reduce visibility.

Example split-tunnel ACL:

```cisco
ip access-list standard SPLIT_TUNNEL
 permit 10.10.10.0 0.0.0.255
 permit 10.20.0.0 0.0.255.255
```

**NAT exemption and VPN**

<abbr title="Network Address Translation">NAT</abbr> can break VPN traffic if protected packets are translated before encryption. In site-to-site designs, you often exempt VPN traffic from NAT.

Why this matters:
- The crypto ACL or tunnel policy expects original private addresses.
- NAT changes source or destination addresses.
- The VPN peer may no longer match the intended traffic.

Typical NAT exemption idea:

```cisco
ip access-list extended NAT-EXEMPT
 deny   ip 192.168.10.0 0.0.0.255 10.10.10.0 0.0.0.255
 permit ip 192.168.10.0 0.0.0.255 any

route-map NAT-ROUTE permit 10
 match ip address NAT-EXEMPT

ip nat inside source route-map NAT-ROUTE interface GigabitEthernet0/0 overload
```

**Verification and troubleshooting**

You verify VPNs by checking negotiation, encapsulation, routing, and interface binding. Start with the control plane, then confirm data flow.

Useful commands:
```cisco
show crypto isakmp sa
show crypto ikev2 sa
show crypto ipsec sa
show crypto session
show crypto map
show run interface gigabitEthernet0/0
show access-lists
show ip route
show ip interface brief
```

What to look for:
- **IKE SA state** should show an established or ready state.
- **IPsec SA counters** should increase when traffic flows.
- **Crypto map binding** should appear on the correct outside interface.
- **ACL hit counts** should increase for interesting traffic.
- **Routes** should point to the correct tunnel or next hop.

Troubleshooting checklist:
1. Confirm basic IP reachability to the peer public address.
2. Verify NAT exemption or NAT policy.
3. Check that the crypto map or tunnel interface is applied correctly.
4. Confirm matching IKE and IPsec parameters on both peers.
5. Generate interesting traffic to trigger negotiation.
6. Inspect SA state and packet counters.
7. Use debug commands carefully on lab systems only.

Common debug concepts:
```cisco
debug crypto ikev2 protocol
debug crypto ipsec
debug crypto isakmp
```

> Use debug commands carefully on production devices because they can be CPU-intensive.

**Common exam comparisons**

| Topic | Site-to-site VPN | Remote access VPN |
|---|---|---|
| Endpoint | Gateway to gateway | User device to gateway |
| Typical users | Branch offices | Mobile users |
| Client software | Usually not needed | Usually required |
| Policy control | Network-based | User-based |
| Common Cisco tools | Crypto map, tunnel interface, IPsec | AnyConnect, AAA, group policy, virtual-access |

**Practical design reminders**

- Use **site-to-site** when you need permanent network connectivity.
- Use **remote access** when individual users need secure access.
- Use **GRE over IPsec** when routing protocols or multicast must traverse the tunnel.
- Use **split tunneling** only when policy allows local Internet breakout.
- Use **NAT exemption** whenever protected traffic would otherwise be translated.
- Prefer **IKEv2** in modern Cisco designs when the platform supports it.

**CCNA-level takeaways**

At the exam level, you should be able to:
- Distinguish site-to-site from remote access VPNs.
- Explain the purpose of IKE Phase 1 and Phase 2.
- Recognize the role of IPsec in protecting traffic.
- Identify why GRE may be added to a VPN design.
- Understand that remote access VPNs often use AnyConnect, AAA, and group policy.
- Verify a VPN with `show crypto` commands and interface checks.
