## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**VPN concept overview**

A <abbr title="Virtual Private Network">VPN</abbr> creates a protected path across an untrusted network, usually the Internet. In CCNA-level labs, the goal is often to understand the *concept* of encapsulation, tunnel endpoints, and traffic selection rather than to build a production-grade design.

A VPN lab commonly shows three layers of thinking:

- The **inside network** that generates user traffic.
- The **underlay** network that carries the encrypted or encapsulated packets.
- The **tunnel** or security association that makes the traffic private.

In Cisco environments, you may see different VPN styles:

- **Site-to-site IPsec** protects traffic between two routers or firewalls.
- **GRE over IPsec** adds a tunnel interface for routing flexibility and then encrypts the tunnel traffic.
- **Remote-access VPN** connects a single client to a central headend.

For CCNA study, focus on the idea that the VPN does not magically move packets by itself. You still need:

- Reachability to the peer’s outer address.
- Correct tunnel or crypto configuration.
- A routing decision that sends the intended traffic toward the VPN path.

**Conceptual packet flow**

A useful mental model is:

1. A host sends traffic to a remote subnet.
2. The local router decides whether that traffic should use the VPN.
3. The router encapsulates or encrypts the packet.
4. The outer packet is forwarded across the underlay.
5. The remote device decapsulates the packet and forwards it to the destination host.

For IPsec, the protected traffic is usually selected by a crypto ACL or policy match. For GRE tunnel interfaces, the tunnel itself becomes a logical interface, and routing can send traffic into it like any other interface. These are related ideas, but they are not identical, so keep the VPN type clear when you explain the lab.

**Topology and lab intent**

A simple concept simulation often uses two routers and two LANs:

- **R1 LAN**: `192.168.10.0/24`
- **R2 LAN**: `192.168.20.0/24`
- **Underlay link**: public or transit network between routers
- **Tunnel endpoints**: router interface addresses or loopbacks

Example topology sketch:

```text
PC1 --- R1 --- Internet/Transit --- R2 --- PC2
  |       |                          |       |
192.168.10.0/24                 192.168.20.0/24

Optional logical tunnel:
R1 Tunnel0 <=====================> Tunnel0 R2
```

The lab objective is to verify that:

- The underlay can reach the peer endpoint.
- The tunnel comes up if the design uses a tunnel interface.
- The protected or tunneled traffic follows the intended path.
- End-to-end connectivity works between the inside networks.

**VPN types you may encounter in a concept lab**

| VPN style | What it does | What you validate |
|---|---|---|
| Site-to-site IPsec | Encrypts selected traffic between peers | ACL match, ISAKMP/IKE status, IPsec SAs |
| GRE tunnel | Creates a logical point-to-point link | Tunnel interface status, routing over tunnel |
| GRE over IPsec | Combines routing flexibility and encryption | Tunnel up, encryption counters, routed traffic |
| Remote access | Connects a client to a headend | Client address assignment, authentication, reachability |

**Important distinction**

> Do not assume every VPN uses a tunnel interface. IPsec can protect traffic without a tunnel interface, while GRE uses a tunnel interface but does not encrypt by itself.

**Basic configuration pattern for a GRE tunnel**

A GRE tunnel is a common way to demonstrate VPN-like behavior in a lab because it is easy to route over. The tunnel source and destination must point to reachable underlay addresses.

Example configuration:

```cisco
! R1
interface Tunnel0
 ip address 10.10.10.1 255.255.255.252
 tunnel source GigabitEthernet0/0
 tunnel destination 203.0.113.2

ip route 192.168.20.0 255.255.255.0 Tunnel0
```

```cisco
! R2
interface Tunnel0
 ip address 10.10.10.2 255.255.255.252
 tunnel source GigabitEthernet0/0
 tunnel destination 203.0.113.1

ip route 192.168.10.0 255.255.255.0 Tunnel0
```

This example is useful for learning, but remember the routing statement is specific to a tunnel-interface design. In many IPsec-only designs, traffic selection is handled differently.

**Why tunnel source and destination matter**

The tunnel source tells the router which local underlay interface or address to use as the outer packet source. The tunnel destination is the peer’s reachable underlay address.

If either value is wrong:

- The tunnel may stay down.
- The router may send packets from the wrong source address.
- The peer may never receive traffic on the expected underlay path.
- Routing over the tunnel may fail even if the tunnel interface exists.

A good troubleshooting habit is to verify underlay reachability first:

```cisco
ping 203.0.113.2 source 203.0.113.1
traceroute 203.0.113.2 source 203.0.113.1
```

If the underlay cannot reach the peer endpoint, the tunnel cannot form reliably.

**IPsec concept summary**

IPsec <abbr title="Internet Protocol Security">IPsec</abbr> is a suite used to secure IP traffic. In CCNA-level labs, you usually need to understand the purpose of the main components rather than memorize every advanced option.

Key pieces include:

- **IKE**: Internet Key Exchange, used to negotiate security parameters.
- **ISAKMP**: Internet Security Association and Key Management Protocol, often used in older Cisco terminology for Phase 1 negotiation.
- **Security Association (SA)**: the negotiated state that defines how traffic is protected.
- **Transform set**: the encryption and integrity methods used for data traffic.
- **Crypto ACL**: the traffic selector that identifies which packets should be protected.

A common site-to-site IPsec flow is:

1. A packet matches the crypto ACL.
2. The router starts IKE negotiation if needed.
3. The peers authenticate and build a secure control channel.
4. IPsec SAs are created for data traffic.
5. Matching packets are encrypted and sent across the underlay.

**Sample IPsec-style traffic selection**

```cisco
access-list 110 permit ip 192.168.10.0 0.0.0.255 192.168.20.0 0.0.0.255
```

This ACL does not route traffic. It identifies which traffic should be protected by IPsec. That distinction matters:

- **Routing** decides where packets go.
- **Crypto policy** decides whether packets are encrypted.

**Example IPsec configuration skeleton**

```cisco
crypto isakmp policy 10
 encryption aes
 hash sha
 authentication pre-share
 group 14
 lifetime 86400

crypto isakmp key cisco123 address 203.0.113.2

crypto ipsec transform-set TS esp-aes esp-sha-hmac

crypto map CMAP 10 ipsec-isakmp
 set peer 203.0.113.2
 set transform-set TS
 match address 110

interface GigabitEthernet0/0
 crypto map CMAP
```

This is a simplified example for study purposes. Exact syntax can vary by IOS version and feature set.

**Validation workflow**

A strong lab workflow is to validate in layers. Do not jump straight to the final ping.

1. Verify interface status on both routers.
2. Verify underlay reachability between peer addresses.
3. Verify tunnel interface state if using GRE.
4. Verify routing to the remote LAN.
5. Generate interesting traffic.
6. Check VPN counters and security associations.
7. Confirm end-to-end host connectivity.

Useful commands:

```cisco
show ip interface brief
show interfaces tunnel 0
show ip route
show crypto isakmp sa
show crypto ipsec sa
show access-lists
show running-config interface tunnel 0
```

**What to look for in command output**

`show ip interface brief` is a quick status check, but it does not explain *why* a tunnel is up or down. Use it to confirm the interface state, then investigate further with interface and routing commands.

Typical tunnel checks:

- `up/up` suggests the tunnel interface is operational.
- `up/down` often indicates the tunnel interface is enabled but the line protocol is not established.
- `down/down` usually points to a disabled interface or a more basic reachability problem.

For GRE, `show interfaces tunnel 0` gives more detail than the brief summary. You can inspect:

- Encapsulation type
- Tunnel source and destination
- Keepalive behavior, if configured
- Input/output packet counters

Example output pattern:

```cisco
R1# show interfaces tunnel 0
Tunnel0 is up, line protocol is up
  Internet address is 10.10.10.1/30
  Tunnel source 203.0.113.1 (GigabitEthernet0/0)
  Tunnel destination 203.0.113.2
  Tunnel protocol/transport GRE/IP
```

**End-to-end lab walkthrough**

Use this sequence when you simulate the lab from scratch.

1. Confirm the underlay addresses on both routers.
   - Check the WAN or transit interfaces.
   - Verify each router can ping the other router’s underlay IP.

2. Configure the tunnel interface if the lab uses GRE.
   - Assign a tunnel IP address.
   - Set `tunnel source` to the local underlay interface or address.
   - Set `tunnel destination` to the remote underlay address.

3. Add routing for the remote LAN.
   - For GRE, route the remote subnet to the tunnel interface or use a dynamic routing protocol over the tunnel.
   - For IPsec-only designs, ensure the protected subnets are reachable through normal routing and matched by the crypto ACL.

4. Configure IPsec if the lab includes encryption.
   - Define IKE policy and pre-shared key.
   - Create the transform set.
   - Apply the crypto map to the correct outside interface.
   - Match the correct interesting traffic.

5. Generate traffic from one LAN to the other.
   - Ping from host to host.
   - Use extended ping from the router if needed.

6. Verify the tunnel or IPsec state.
   - Check tunnel status for GRE.
   - Check SAs and packet counters for IPsec.

7. Troubleshoot any failure.
   - Fix underlay reachability first.
   - Then verify tunnel endpoints or crypto selectors.
   - Finally confirm routing and ACLs.

**Practical validation examples**

Extended ping can help you test from the router using the correct source:

```cisco
R1# ping 192.168.20.10 source 192.168.10.1
```

If the ping fails, test the underlay separately:

```cisco
R1# ping 203.0.113.2 source 203.0.113.1
```

For GRE, verify the tunnel route exists:

```cisco
R1# show ip route 192.168.20.0
```

For IPsec, verify that the ACL is matching traffic and that counters increase when you generate traffic:

```cisco
R1# show access-lists 110
R1# show crypto ipsec sa
```

Expected signs of success include:

- ACL hit counts increasing.
- IPsec encapsulation and decapsulation counters increasing.
- Hosts on opposite LANs can ping each other.
- The tunnel interface remains stable.

**Common failure cases and fixes**

A realistic troubleshooting scenario is more useful than a perfect lab. Consider this example:

- The tunnel interface is configured.
- The underlay ping works.
- The remote LAN is unreachable.

Likely causes:

- The tunnel destination is incorrect.
- The tunnel source points to the wrong interface.
- The remote route is missing.
- The crypto ACL does not match the actual source and destination subnets.
- The crypto map is applied to the wrong interface.

A practical fix sequence is:

1. Verify the peer underlay address.
2. Confirm the tunnel source and destination.
3. Check routing to the remote LAN.
4. Confirm the crypto ACL or tunnel routing matches the intended subnets.
5. Regenerate traffic and recheck counters.

**Troubleshooting checklist**

- **Underlay reachability**
  - Can each router ping the other router’s outside address?
  - Is the correct interface up and addressed?

- **Tunnel configuration**
  - Is the tunnel source correct?
  - Is the tunnel destination reachable?
  - Are tunnel IP addresses in the same subnet if required by the design?

- **Routing**
  - Does each router know how to reach the remote LAN?
  - Is the route pointing to the correct next hop or tunnel interface?

- **IPsec policy**
  - Do both peers use compatible IKE and IPsec settings?
  - Does the crypto ACL match the intended traffic in both directions?
  - Is the crypto map applied to the correct outside interface?

- **Traffic generation**
  - Are you testing with the correct source host or source interface?
  - Did you clear stale SAs if needed?

**Useful show and debug commands**

Use show commands first. Debug commands can be noisy and should be used carefully in a lab.

```cisco
show ip interface brief
show interfaces tunnel 0
show ip route
show crypto isakmp sa
show crypto ipsec sa
show crypto session
show access-lists
```

If your lab permits debugging, you may use:

```cisco
debug crypto isakmp
debug crypto ipsec
```

> Use debug commands only in a controlled lab. They can produce a large amount of output and affect device performance.

**Exam-focused takeaways**

For CCNA, the most important ideas are:

- A VPN protects traffic across an untrusted network.
- The underlay must be reachable before the VPN can work.
- GRE uses a tunnel interface; IPsec uses policy and security associations.
- Routing and encryption are separate functions.
- Validation should move from physical and underlay checks to tunnel, policy, and end-to-end tests.

**Quick memory aid**

- **Underlay**: how the outer packet moves.
- **Overlay**: the logical path created by the tunnel or VPN.
- **Selector**: what traffic is protected.
- **SA**: how traffic is protected.

**Mini lab summary**

If you are asked to explain the lab in one sentence, say:

A VPN concept simulation demonstrates how routers establish a secure or logical path between two sites, verify underlay reachability, and confirm that traffic between remote LANs is correctly routed, encapsulated, and, when applicable, encrypted.
