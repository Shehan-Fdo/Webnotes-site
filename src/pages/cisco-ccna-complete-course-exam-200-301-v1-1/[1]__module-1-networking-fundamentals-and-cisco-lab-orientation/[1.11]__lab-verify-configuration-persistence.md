## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Configuration persistence overview**

Cisco IOS stores the **running configuration** in volatile memory, so changes are lost if the device powers off or restarts unless you save them. The saved copy is the **startup configuration**, which is used when the device boots. This lab focuses on verifying that a configuration change persists after a reload.

Cisco routers and switches use the CLI (Command Line Interface) to enter and verify configuration commands. In lab environments, you should confirm both the current active settings and the saved startup file before and after a reload.

- **Running config** changes take effect immediately.
- **Startup config** is the saved boot-time configuration.
- Unsaved changes disappear after a restart or power loss.

**Saving the configuration**

To preserve changes, copy the running configuration to the startup configuration. This is the standard persistence step for Cisco IOS devices.

```bash
copy running-config startup-config
```

You can also use the abbreviated form commonly accepted in IOS:

```bash
copy run start
```

> If you do not save the running configuration, a reload returns the device to the last saved startup configuration.

**Verification workflow**

Use a simple before-and-after check to confirm persistence. First, make a small configuration change. Then save it, reload the device, and verify that the change remains present after boot.

1. Make a configuration change in the running configuration.
2. Save the configuration to startup configuration.
3. Reload the router or switch.
4. Check the configuration again after the device comes back online.

A practical verification approach is to compare the active configuration with the saved configuration state. If the device reloads and the change is still present, persistence is confirmed.

**Lab expectations**

CCNA hands-on labs often assume devices start with no startup configuration so results stay predictable. If a device already contains saved settings, your lab output may differ from the expected behavior. Device model and Cisco IOS version can also affect the exact commands and output you see.

- Start from a known state when possible.
- Expect minor output differences across IOS versions.
- Verify the saved configuration before relying on reload behavior.

**Troubleshooting reminders**

If a change disappears after reload, the most likely cause is that you did not save the running configuration. In lab work, always confirm the save step before rebooting the device. This habit helps you distinguish between temporary runtime changes and persistent configuration.
