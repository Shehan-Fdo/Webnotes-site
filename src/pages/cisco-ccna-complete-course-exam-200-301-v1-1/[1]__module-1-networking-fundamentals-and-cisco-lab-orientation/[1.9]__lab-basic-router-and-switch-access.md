## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Lab overview**

This lab focuses on the first steps of device access and initial configuration in a Cisco IOS environment. You verify the default router state, connect through the console, secure access with passwords, add login banners, and save the configuration so it persists after reload.

The main goal is to build a reliable starting point before you move into routing, switching, or interface configuration. If you cannot enter the correct CLI mode or save changes correctly, later labs become harder to troubleshoot.

- Establish a **console** connection to the router before making changes.
- Use privileged mode to inspect the current configuration and device state.
- Apply basic security to the CLI and console access paths.
- Add a warning banner for unauthorized users.
- Save the running configuration after verification.

**Default access and CLI navigation**

A basic lab often begins with checking the device in its default state. You connect to the router through the console port, open the terminal, and press `ENTER` to reach the prompt. From there, you move into privileged EXEC mode so you can inspect and configure the device.

Understanding the current mode matters because IOS commands behave differently depending on where you are in the CLI hierarchy. If a command does not work as expected, the first thing to check is whether you are in the correct mode.

- Use the console connection for initial access when remote access is not yet configured.
- Enter privileged EXEC mode before reviewing or changing the configuration.
- Confirm the router’s default settings before applying new values.
- Treat mode awareness as part of the troubleshooting process.

**Initial router and switch access tasks**

The evidence for this lab emphasizes basic device configuration, including hostnames, passwords, and bringing interfaces up. The same early-stage workflow applies to both routers and switches: gain access, secure the device, and verify that the configuration is in place.

> Use conservative, practical settings during the lab so you can verify each change clearly before moving on.

1. Connect to the device through the console port.
2. Enter privileged mode and review the current configuration.
3. Configure the initial device settings.
4. Add access protection for the console and CLI.
5. Configure a banner message that warns unauthorized users.
6. Verify the changes.
7. Save the running configuration.

**Security and access control**

This lab uses both plain text and encrypted passwords to secure access. The purpose is not just to set credentials, but to protect the management plane from casual access. A banner also reinforces that the system is restricted and that unauthorized access is prohibited.

When you configure passwords, keep the distinction clear between access methods. Console access protects local physical access, while privileged access protects administrative control. The banner does not enforce security by itself, but it provides a visible warning at login.

- Protect the **console** port with a password.
- Secure privileged access with an encrypted password.
- Use a banner to display an access warning.
- Verify that login prompts reflect the intended protection.

**Verification and saving configuration**

After configuration, you verify the running state before saving. This step matters because the running configuration exists only in memory until you store it. Saving ensures the device keeps the settings after a restart.

A practical workflow is to confirm the active configuration first, then write it to persistent storage. That sequence helps you catch mistakes before they become part of the saved startup configuration.

- Check the active configuration after each major change.
- Confirm that passwords and banners are present.
- Save the running configuration when verification is complete.
- Recheck the saved state if the lab asks you to confirm persistence.

**Key takeaways**

This lab builds the habits you need for every later Cisco configuration task. You start with access, move through secure setup, and finish by verifying and saving. Those steps create a stable baseline for more advanced labs.

- Access the device through the console first.
- Work in the correct CLI mode.
- Secure management access early.
- Verify before you save.
