## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Initial setup workflow**

When you first access a Cisco router or switch, you typically connect through the console port and move into privileged mode before making changes. The basic setup flow includes checking the default configuration, applying global settings, securing access, and then verifying the result.

- Use a **console connection** to reach the device for first-time access.
- Enter **privileged EXEC mode** to inspect and manage the device.
- Apply global configuration settings before interface-specific changes.
- Configure passwords and banners to secure local access.
- Verify the configuration after each major change.

Cisco IOS (Internetwork Operating System) uses different command modes, and each mode controls what you can change. User mode is limited, privileged mode exposes full monitoring and management, and global configuration mode affects the whole device. Interface and line modes let you target specific ports or access methods.

```text
Router> enable
Router# configure terminal
Router(config)# hostname R1
R1(config)# exit
```

**Common configuration areas**

Initial device setup usually covers identity, access control, and interface readiness. The evidence highlights hostname changes, password configuration, console security, and interface configuration as standard tasks in a basic lab or first-run setup.

- Set a **hostname** so the device is easy to identify.
- Configure an **enable secret** for privileged access protection.
- Use an **enable password** only when required by the lab or legacy context.
- Secure the **console line** for local access control.
- Configure interfaces so the device can participate in the network.

> The running configuration is active in RAM, so changes are temporary until you save them.

**Running configuration vs. startup configuration**

Cisco devices keep the active configuration in memory while the device is running. If you reboot without saving, you can lose changes. Saving copies the current running configuration into startup configuration so the device can restore it on boot.

| Item | Purpose | Persistence |
|---|---|---|
| `running-config` | Active configuration in use now | Stored in RAM |
| `startup-config` | Configuration loaded at boot | Stored for reboot use |

This distinction is central to exam prep because many lab tasks end with a save step. The evidence specifically calls out saving the running configuration file after initial setup and verification.

**Saving configuration changes**

Use the save step after you finish configuration and confirm the device behaves as expected. The common Cisco command copies the running configuration into startup configuration.

```text
copy running-config startup-config
```

You may also see the shorter form:

```text
write memory
```

- Save after you complete the intended changes.
- Verify the configuration before you save when possible.
- Remember that unsaved changes disappear after a reboot.

**Verification mindset**

Verification is part of the setup process, not an optional extra. After you configure the device, check the current settings and confirm that access, interface status, and saved state match your intent.

1. Connect to the device through the console.
2. Enter privileged mode and review the default state.
3. Apply the initial configuration.
4. Verify the changes.
5. Save the running configuration.

**Practical exam focus**

For CCNA-style tasks, focus on the sequence and the purpose of each step rather than memorizing isolated commands. You should know how to get into the device, secure it, configure basic settings, and preserve the work so it survives a restart.

- **Access** starts with the console.
- **Security** includes passwords and banners.
- **Configuration** includes hostname and interfaces.
- **Persistence** requires saving to startup configuration.
