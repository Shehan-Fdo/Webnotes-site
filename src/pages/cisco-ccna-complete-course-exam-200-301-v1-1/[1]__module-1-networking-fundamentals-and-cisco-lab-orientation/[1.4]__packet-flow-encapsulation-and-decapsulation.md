## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Packet flow overview**

Packet flow describes how data moves through network layers from sender to receiver. The sender adds control information as data moves down the stack, and the receiver removes that information as data moves back up. This wrapping and unwrapping process is central to network communication and helps you understand how devices exchange information reliably.

- **Encapsulation** is the sender-side process of adding headers and, in some layers, trailers.
- **Decapsulation** is the receiver-side process of removing those added fields.
- Each layer contributes its own control information as the data moves through the model.
- The result is a layered exchange where each protocol handles its own part of the communication.

**Encapsulation basics**

Encapsulation marks where a packet begins and ends. The beginning of the packet is the **header**, the end is the **trailer**, and the data between them is the **payload**. As data travels from source to destination, different layers add information to the packet header, and the exact format depends on the protocol in use.

> The trailer often includes an error-check value so the receiving device can confirm it received the full packet.

A useful way to think about encapsulation is that each layer wraps the data it receives and creates a new protocol data unit, or **PDU**. A PDU is the data unit defined by a layer after that layer adds its control information.

- The header identifies the packet and carries protocol-specific information.
- The trailer, when present, helps the receiver detect the end of the packet.
- The payload is the original data being transported.
- The packet becomes larger as it moves down the stack.

**Decapsulation basics**

Decapsulation is the reverse of encapsulation. On the receiving side, each layer removes the information added by its peer layer on the sender side. This same-layer interaction means the Transport layer removes the Transport layer information, the Network layer removes the Network layer information, and so on until the original data reaches the application.

- The receiver strips off headers and trailers in reverse order.
- The process continues until the data reaches the final application layer.
- The original information becomes readable again after all layer-specific data is removed.
- Encapsulation and decapsulation happen together during normal communication.

**Layer-by-layer packet handling**

The evidence shows a common flow through the model: application data moves to the Transport layer, then to the Network layer, and then to the Network Access layer. Each step adds information needed for delivery.

1. The application creates the data.
2. The Transport layer adds its header, such as port information.
3. The Network layer adds its header, including source and destination IP addresses.
4. The Network Access layer adds both a header and a trailer before transmission.
5. The receiving device reverses the process through decapsulation.

This layered approach explains why network troubleshooting often focuses on where the process breaks. If a layer adds the wrong information or a peer layer cannot remove it correctly, communication fails.

**Protocol data units and layer names**

Different layers produce differently named PDUs. The exact names vary by model, but the key idea is that each layer wraps the data in a new form before passing it down.

| Layer role | Typical action | Resulting unit |
|---|---|---|
| Application/upper layers | Create user data | Data |
| Transport | Add transport information | Segment |
| Network | Add IP information | Packet |
| Network Access | Add link-layer header and trailer | Frame |

The important exam takeaway is not memorizing every label in isolation, but understanding that each layer adds its own control information and hands the result to the next layer.

**Why this matters for CCNA**

Encapsulation and decapsulation are foundational concepts for understanding routers, switches, and end hosts. They explain how devices prepare data for transmission, how the receiving side reconstructs it, and why protocol layers must work in sequence.

- They help you trace packet movement across the stack.
- They support troubleshooting by showing where data changes form.
- They clarify why headers and trailers exist.
- They reinforce the OSI and TCP/IP model relationship to real traffic flow.

**Quick memory aid**

Think of the process as sending a letter:

- You write the message.
- You place it in an envelope.
- You add addressing information.
- The receiver opens the envelope and removes each layer of packaging.

That analogy matches the technical flow: data is wrapped on the sender side and unwrapped on the receiver side.
