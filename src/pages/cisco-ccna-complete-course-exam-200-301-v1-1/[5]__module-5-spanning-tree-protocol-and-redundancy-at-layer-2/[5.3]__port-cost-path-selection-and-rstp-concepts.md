## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Spanning Tree purpose and why path selection matters**

Spanning Tree Protocol (STP) prevents Layer 2 loops by allowing only one active logical path between switches for a given VLAN or spanning-tree instance. Without STP, redundant links can create broadcast storms, MAC table instability, and duplicate frames.

STP does not simply “turn off” links at random. It elects a root bridge, then each non-root switch chooses the best path toward that root based on BPDU (Bridge Protocol Data Unit) comparison and interface cost. Understanding the selection order is essential for troubleshooting why one uplink forwards while another blocks.

- **Loop prevention** is the primary goal.
- **Redundant links** remain available as backups.
- **Per-VLAN operation** matters in PVST+ and Rapid PVST+ environments.

**STP port cost and how it is chosen**

Port cost is a numeric value assigned to an interface based on its bandwidth. Lower cost means a better path. Cisco devices typically derive the cost from interface speed automatically, though you can also set it manually.

Common representative costs vary by platform and STP cost method, but exam-relevant values often include:

| Interface speed | Typical STP cost |
|---|---:|
| 10 Mb/s | 100 |
| 100 Mb/s | 19 |
| 1 Gb/s | 4 |
| 10 Gb/s | 2 |
| 40 Gb/s | 1 |
| 100 Gb/s | 1 |

The exact values can differ depending on whether the switch uses the short or long path-cost method, but the key idea is consistent: faster links have lower cost.

You can verify or influence cost with commands such as:

```bash
show spanning-tree vlan 10
show spanning-tree interface gigabitEthernet1/0/1 detail
```

To set a manual cost on Cisco IOS, use:

```bash
conf t
interface gigabitEthernet1/0/1
 spanning-tree vlan 10 cost 10
end
```

Manual cost is useful when you want to prefer one uplink over another even if both links have the same speed.

**BPDU comparison and exact path selection order**

When a switch receives multiple BPDUs and must decide which path is best, it compares fields in a specific order. The comparison is not based on a single value; it is a sequence of tie-breakers.

The usual order is:

1. **Root bridge ID**  
   The BPDU advertising the lowest root ID wins. The root ID includes bridge priority and MAC address.

2. **Root path cost**  
   If the root ID is the same, the switch prefers the BPDU with the lower accumulated cost to reach the root.

3. **Sender bridge ID**  
   If root ID and path cost tie, the lower bridge ID of the sending switch wins.

4. **Sender port ID**  
   If the sender bridge ID also ties, the lower port ID on the sending switch wins.

5. **Local port ID**  
   If needed in a final tie, the receiving switch uses the local port ID to break the tie between otherwise identical candidates.

This order is important for exam questions that ask why one port becomes the root port and another becomes blocked or discarded.

> The local port ID is not usually the headline tie-breaker in simplified explanations, but it can still matter in exact comparisons when all earlier fields are equal.

**Root bridge election and per-VLAN behavior**

In PVST+ and Rapid PVST+, STP runs separately per VLAN. That means you can have a different root bridge for VLAN 10 and VLAN 20, and the root port on a switch can differ by VLAN.

A common design goal is to make one distribution switch the root for some VLANs and another distribution switch the root for other VLANs to balance traffic.

Example design:

- Switch A is root for VLAN 10.
- Switch B is root for VLAN 20.
- Access switches choose their root port independently for each VLAN.

A simple verification workflow:

```bash
show spanning-tree vlan 10
show spanning-tree vlan 20
```

Look for:

- **Root ID**: identifies the elected root bridge for that VLAN.
- **This bridge**: identifies the local switch.
- **Root port**: the interface the local switch uses to reach the root.
- **Port cost**: the total cost to the root through that port.

If a switch has two uplinks of equal speed, the one with the lower total path cost to the root becomes the root port. If the costs tie, the BPDU tie-breakers decide.

**Worked path-selection example**

Consider three switches in VLAN 10:

- Switch A has the lowest bridge priority, so it becomes the root bridge.
- Switch B connects to A with a 1 Gb/s link.
- Switch C connects to A with a 1 Gb/s link.
- Switch D connects to both B and C with 1 Gb/s links.

For Switch D, both uplinks may advertise the same root ID and the same root path cost if the paths are symmetrical. Then D compares the sender bridge IDs:

- If B has a lower bridge ID than C, D selects the port toward B as the root port.
- The other port becomes non-designated and does not forward user traffic for that VLAN.

If B and C also tie on bridge ID, the sender port ID breaks the tie. If that still ties in a contrived lab, the local port ID can be the final discriminator.

This is the kind of step-by-step logic you should apply on the exam rather than guessing based on physical layout alone.

**STP port roles and states**

Classic 802.1D STP uses port states that include blocking, listening, learning, and forwarding. Rapid STP (RSTP, IEEE 802.1w) simplifies the operational behavior by using discarding, learning, and forwarding.

Port roles describe why a port is in a certain condition; states describe what the port is doing.

Common roles include:

- **Root port**: best path toward the root bridge on a non-root switch.
- **Designated port**: best port on a segment; forwards traffic away from the switch toward downstream devices.
- **Alternate port**: backup path to the root; usually discarding in RSTP.
- **Backup port**: redundant port on the same shared segment; usually discarding in RSTP.
- **Disabled port**: not participating in spanning tree because the interface is administratively shut down or otherwise unavailable.

Common states include:

- **Discarding**: the port does not forward frames or learn MAC addresses.
- **Learning**: the port learns MAC addresses but does not forward user traffic.
- **Forwarding**: the port forwards frames and learns MAC addresses.

> In Cisco output, “disabled” can appear as a valid spanning-tree port condition, and it may also correspond to an administratively down interface. Do not treat it as a separate forwarding state.

**RSTP concepts and why it converges faster**

RSTP improves convergence by reacting to topology changes more quickly than classic STP. It reduces reliance on long timer-based waiting periods and uses handshake behavior between switches.

Key reasons RSTP converges faster:

- It uses **proposal and agreement** messages to negotiate forwarding transitions.
- It can move ports to forwarding without waiting through the full classic STP timer sequence in many cases.
- It treats edge ports specially so they can come up immediately when safe.

Classic STP often waits through:

- **Max Age** to age out old information.
- **Forward Delay** while moving through listening and learning.

RSTP can often bypass those delays when the topology change is local and the neighbor supports RSTP.

**RSTP edge ports and non-edge ports**

An edge port connects to an end device, not to another switch. It is the RSTP equivalent of a PortFast-enabled port in Cisco terminology.

- **Edge port**
  - Moves to forwarding immediately.
  - Does not participate in the normal RSTP synchronization process.
  - Should be used only on access ports connected to hosts, printers, phones, or similar endpoints.

- **Non-edge port**
  - Participates in RSTP role negotiation.
  - May transition through discarding and learning as the topology stabilizes.
  - Is used on switch-to-switch links.

Example configuration:

```bash
conf t
interface gigabitEthernet1/0/10
 spanning-tree portfast
 spanning-tree bpduguard enable
end
```

On many Cisco platforms, `spanning-tree portfast` on an access port causes the port to behave as an edge port. BPDU Guard is commonly paired with it to protect against accidental switch connections.

**RSTP synchronization and proposal/agreement behavior**

RSTP uses synchronization to ensure that a new forwarding path does not create a loop. When a switch receives a proposal on a designated port, it may temporarily block or discard other non-edge ports while it confirms that the rest of the topology is safe.

A simplified sequence is:

1. A switch proposes a new designated port state to a neighbor.
2. The neighbor checks whether its other ports are synchronized.
3. If needed, the neighbor blocks or discards other ports that could create a loop.
4. The neighbor sends an agreement.
5. The proposing switch can then move the port to forwarding quickly.

This behavior is why RSTP often converges much faster than classic STP. It does not blindly wait for timers if it can prove the topology is safe.

**RSTP port roles in practical terms**

A quick mental model helps during troubleshooting:

- **Root port** points toward the root bridge.
- **Designated port** points away from the root on a segment and forwards.
- **Alternate port** is a backup path toward the root and stays ready.
- **Backup port** is a backup on the same shared segment as another port on the same switch.
- **Edge port** is for end devices and forwards immediately.

A simple role/state mapping:

| Role | Typical RSTP state behavior | Purpose |
|---|---|---|
| Root | Forwarding after sync | Best path to root |
| Designated | Forwarding after sync | Best port on a segment |
| Alternate | Discarding | Backup path to root |
| Backup | Discarding | Redundant local segment path |
| Edge | Forwarding immediately | End-device access |

**Classic STP versus RSTP**

| Feature | Classic STP (802.1D) | RSTP (802.1w) |
|---|---|---|
| Convergence | Slower | Faster |
| Transitional states | Blocking, listening, learning, forwarding | Discarding, learning, forwarding |
| Topology change handling | Timer-driven | Handshake-driven where possible |
| Edge access ports | PortFast concept | Edge port behavior |
| Backup path handling | Less dynamic | Alternate and backup roles |

RSTP still uses learning and forwarding, so it is not correct to say it only has discarding and forwarding. The main difference is that discarding replaces the old blocking/listening behavior.

**How timers relate to exam scenarios**

Classic STP timers are still important because they explain why convergence can be slow:

- **Hello time**: interval for root bridge BPDUs.
- **Max Age**: how long BPDU information is kept before aging out.
- **Forward Delay**: time spent in listening and learning.

In classic STP, a port may wait through these timers before forwarding. In RSTP, the protocol often avoids waiting for Max Age and Forward Delay when it can rapidly confirm a safe topology change.

For exam questions, remember:

- If the network is running classic STP, timer delays matter a lot.
- If the network is running RSTP or Rapid PVST+, convergence is usually faster because of synchronization and edge behavior.
- A topology change on an edge port is usually handled more quickly than a change on a switch-to-switch link.

**Cisco verification commands**

Use these commands to inspect spanning tree behavior:

```bash
show spanning-tree
show spanning-tree vlan 10
show spanning-tree root
show spanning-tree interface gigabitEthernet1/0/1 detail
show spanning-tree summary
```

Useful things to look for:

- **Root bridge** identity for the VLAN.
- **Root port** on each non-root switch.
- **Port role** and **state**.
- **Path cost** values.
- **Topology change** indicators.
- **Edge port** status if PortFast is enabled.

Example output interpretation:

```bash
Switch# show spanning-tree vlan 10

VLAN0010
  Spanning tree enabled protocol rstp
  Root ID    Priority    24586
             Address     0011.2233.4455
             Cost        4
             Port        1 (GigabitEthernet1/0/1)
  Bridge ID  Priority    32778
             Address     00aa.bbcc.ddee
             Hello Time   2 sec  Max Age 20 sec  Forward Delay 15 sec
```

Interpretation:

- The VLAN is using an RSTP-based mode.
- The root bridge has a lower bridge ID than the local switch.
- The local switch reaches the root through `GigabitEthernet1/0/1`.
- The total root path cost is `4`, which is consistent with a 1 Gb/s link in common cost tables.

Exact formatting varies by platform and IOS version, so focus on the fields, not the punctuation.

**Troubleshooting scenarios**

A few common problems show up repeatedly in labs and production networks:

- **Unexpected root bridge**
  - A lower-priority switch may have taken over as root.
  - Verify bridge priority and adjust root placement intentionally.

- **Wrong root port**
  - A link with lower speed or higher manual cost may be preferred incorrectly.
  - Check interface cost and BPDU tie-breakers.

- **Port stuck in discarding**
  - The port may be an alternate or backup port.
  - The neighbor may not support RSTP, or a loop-prevention condition may exist.

- **Access port flapping into blocking**
  - A switch may be connected where an end device was expected.
  - Use PortFast and BPDU Guard only on true edge ports.

- **Topology change storms**
  - Frequent link up/down events can trigger repeated recalculation.
  - Check cabling, duplex, and redundant links.

A practical troubleshooting workflow:

1. Identify the VLAN or spanning-tree instance affected.
2. Check the root bridge with `show spanning-tree vlan <vlan-id>`.
3. Verify the root port and path cost on each non-root switch.
4. Compare interface speeds and any manual cost settings.
5. Confirm whether the port is edge, designated, alternate, or backup.
6. Look for topology change counters or recent interface events.

**Configuration examples for exam prep**

Enable Rapid PVST+ globally:

```bash
conf t
spanning-tree mode rapid-pvst
end
```

Verify the mode:

```bash
show spanning-tree summary
```

Set a switch as the preferred root for a VLAN by lowering its priority:

```bash
conf t
spanning-tree vlan 10 root primary
spanning-tree vlan 20 root secondary
end
```

Or set priority manually:

```bash
conf t
spanning-tree vlan 10 priority 24576
end
```

Prefer one uplink by adjusting cost:

```bash
conf t
interface gigabitEthernet1/0/1
 spanning-tree vlan 10 cost 4
end
```

These commands are useful in labs because they let you observe how the root port changes when cost or priority changes.

**Exam-focused takeaways**

- STP chooses paths by comparing BPDUs in this order: **root ID, root path cost, sender bridge ID, sender port ID, then local port ID if needed**.
- Lower **path cost** is better, and cost is derived from link speed unless you override it.
- In PVST+ and Rapid PVST+, root election and root port selection happen **per VLAN**.
- RSTP converges faster because it uses **proposal/agreement** and **edge port** behavior instead of relying only on classic timer delays.
- **Alternate** and **backup** ports are ready-to-use redundant paths, but they do not forward unless the active path fails.
- Cisco output can vary, so learn to interpret the fields rather than memorize one exact display.
