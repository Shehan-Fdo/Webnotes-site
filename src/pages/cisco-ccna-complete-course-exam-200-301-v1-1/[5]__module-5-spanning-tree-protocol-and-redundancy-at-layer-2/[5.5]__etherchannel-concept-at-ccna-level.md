## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**EtherChannel overview**

EtherChannel combines multiple physical Ethernet links into one logical link called a **port-channel**. You use it to increase bandwidth, add redundancy, and simplify switch configuration.

At CCNA level, focus on three ideas:

- The switch treats the bundle as one logical interface for many Layer 2 operations.
- The physical links must match in key parameters to join the bundle.
- Load balancing happens per flow, not by splitting a single frame across links.

Common Cisco implementations include:

- **LACP**: Link Aggregation Control Protocol, an IEEE standard.
- **PAgP**: Port Aggregation Protocol, Cisco proprietary.
- **Static EtherChannel**: no negotiation protocol, usually `mode on`.

> EtherChannel does not make one conversation faster by itself. It spreads different flows across member links using a hashing algorithm.

**Why you use EtherChannel**

EtherChannel is useful when a single uplink is not enough and you want more resilience without building separate parallel topologies.

Typical benefits:

- Higher aggregate throughput across multiple links.
- Faster recovery from a single member failure.
- Cleaner topology with fewer STP (Spanning Tree Protocol) blocked links.
- Easier management because you configure the bundle as one logical path.

A practical example is a switch uplink to another switch or to a distribution device. Instead of two separate trunk links where STP blocks one, you bundle both links so both can forward traffic as one logical connection.

**How the bundle works**

The switch groups interfaces into a port-channel when the interfaces are compatible and the negotiation mode allows it.

Key requirements usually include:

- Same speed and duplex.
- Same access or trunk mode.
- Same VLAN membership or trunk allowed VLANs.
- Same native VLAN on trunks.
- Same Layer 3 or Layer 2 role.
- Compatible EtherChannel protocol and mode.

Some settings are expected to match on the physical members, while others may be inherited or applied at the port-channel level depending on platform and IOS/IOS XE behavior. Do not assume every shared setting belongs only on the port-channel.

Commonly matched items include:

- `switchport mode access` or `switchport mode trunk`
- Access VLAN or trunk VLAN settings
- Native VLAN
- Speed and duplex
- Layer 2 versus Layer 3 mode

**Negotiation protocols**

LACP is the most important protocol for CCNA study.

- **Active**: actively sends LACP packets and tries to form a bundle.
- **Passive**: listens and responds, but does not initiate negotiation.
- **On**: no protocol negotiation; the interface is forced into the bundle.

LACP mode pairing rules:

- `active` + `active` = forms a bundle
- `active` + `passive` = forms a bundle
- `passive` + `passive` = does not form a bundle
- `on` + `on` = static bundle, if the interfaces are otherwise compatible
- `on` mixed with LACP or PAgP is not a valid negotiation match

PAgP modes are similar in concept:

- `desirable` actively negotiates.
- `auto` waits for the other side.
- `desirable` + `desirable` or `desirable` + `auto` can form a bundle.
- `auto` + `auto` does not initiate formation.

For CCNA, LACP is the most exam-relevant because it is standards-based and widely used.

**Load balancing**

EtherChannel uses a hashing algorithm to choose which member link carries a given flow. The hash is based on selected header fields such as:

- Source MAC address
- Destination MAC address
- Source IP address
- Destination IP address
- Layer 4 ports on some platforms

Important limitations:

- It is typically **per-flow**, not per-packet.
- One large flow usually stays on one physical link.
- Many small flows can be distributed across links more evenly.
- The exact hash options and defaults vary by platform.

This means you should not expect a single file transfer to automatically use all links at once. EtherChannel improves aggregate capacity across multiple conversations.

**Layer 2 EtherChannel**

A Layer 2 EtherChannel is the most common CCNA scenario. You can use it as an access bundle or a trunk bundle.

Access bundle example:

```cisco
interface range gigabitEthernet1/0/1 - 2
 switchport
 switchport mode access
 switchport access vlan 10
 channel-group 1 mode active
```

Trunk bundle example:

```cisco
interface range gigabitEthernet1/0/1 - 2
 switchport
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30
 switchport trunk native vlan 99
 channel-group 1 mode active
```

Then verify the logical interface:

```cisco
show etherchannel summary
show interfaces port-channel 1
show interfaces trunk
```

A typical summary output may show flags such as:

- `S` for Layer 2 switched
- `R` for Layer 3 routed
- `U` for in use
- `P` for bundled in the port-channel

The exact formatting and flags can vary by platform and IOS family, so use the output as a clue rather than memorizing one rigid pattern.

**Layer 3 EtherChannel**

A Layer 3 EtherChannel is a routed bundle. You use it between Layer 3 devices or on multilayer switches when you want the port-channel to carry IP routing instead of switching.

Typical configuration pattern:

```cisco
interface range gigabitEthernet1/0/1 - 2
 no switchport
 channel-group 10 mode active

interface port-channel10
 no switchport
 ip address 10.10.10.1 255.255.255.252
```

Notes for lab work:

- The member interfaces must be routed-capable.
- Some platforms expect the port-channel to exist before you assign the IP address.
- If an interface was previously a switchport, you may need to clear or reconfigure it before joining the routed bundle.

Verify with:

```cisco
show ip interface brief
show etherchannel summary
show interfaces port-channel 10
```

**Configuration consistency rules**

EtherChannel fails or suspends members when the bundle parameters do not match.

Common mismatch causes:

- One side is trunk, the other is access.
- VLAN settings differ.
- Native VLAN differs.
- Speed or duplex differs.
- One side uses LACP and the other is static or uses a different protocol.
- One interface is configured as routed and the other as switched.
- One member has an extra feature that is not compatible with the bundle.

A useful troubleshooting mindset is:

- Check the physical link first.
- Check the member interface settings.
- Check the port-channel settings.
- Check the negotiation mode on both ends.

**Verification commands**

Use a combination of commands because no single command tells the whole story.

Common verification commands:

```cisco
show etherchannel summary
show etherchannel port-channel
show interfaces port-channel 1
show interfaces gigabitEthernet1/0/1 etherchannel
show lacp neighbor
show lacp neighbors
show pagp neighbor
```

Use these carefully:

- `show etherchannel summary` gives the quickest high-level view.
- `show interfaces port-channel X` shows the logical interface state.
- `show interfaces <member> etherchannel` helps confirm member participation.
- LACP neighbor commands vary by platform and IOS version, so treat them as platform-dependent.

What to look for:

- The port-channel is up and protocol is working.
- Member interfaces are bundled, not suspended.
- Both sides agree on the bundle type and mode.
- The logical interface carries the expected VLANs or IP address.

**Troubleshooting indicators**

When EtherChannel does not form, you often see one of these symptoms:

- The port-channel exists but is down.
- One or more members are suspended.
- The bundle forms on one side but not the other.
- STP treats links separately instead of as one logical link.
- Traffic uses only one link because hashing is not distributing flows as expected.

Common clues in output:

- `I` may indicate an individual interface not bundled.
- `s` may indicate suspended.
- `D` may indicate down.
- `P` indicates bundled in the port-channel.

Because flags and abbreviations can vary, always read the full command output and not just one letter.

**Minimum links and operational behavior**

Some platforms support a **minimum-links** feature that keeps the port-channel operational only if a minimum number of member links are active.

Why it matters:

- You can prevent a partially degraded bundle from carrying traffic.
- It is useful when you need a minimum bandwidth threshold.

Conceptually:

- If enough links are up, the port-channel stays operational.
- If active members drop below the configured minimum, the port-channel can go down.

This is platform-dependent, so treat it as an operational feature rather than a universal default.

**STP interaction**

EtherChannel works closely with STP.

Why it matters:

- STP sees the bundle as one logical link.
- This reduces blocked redundant links compared with separate parallel connections.
- The bundle still participates in STP as a single interface.

Practical effect:

- Two separate uplinks may cause one link to block.
- The same two links in an EtherChannel can both forward as one logical path.

This is one reason EtherChannel is preferred for redundant switch-to-switch connections.

**Real-world example scenario**

Imagine two access switches connected to a distribution switch with two parallel links.

Without EtherChannel:

- STP may block one link.
- Only one link forwards traffic.
- The second link sits idle unless the topology changes.

With EtherChannel:

- Both links join one port-channel.
- Both links can forward different flows.
- If one physical link fails, the bundle can continue on the remaining link if the bundle still meets operational requirements.

This is a common campus design pattern.

**End-to-end LACP example**

Topology:

- SW1 `Gi1/0/1-2` connects to SW2 `Gi1/0/1-2`
- Bundle uses port-channel 1
- Trunk carries VLANs 10, 20, and 99

SW1:

```cisco
conf t
interface range gigabitEthernet1/0/1 - 2
 switchport
 switchport mode trunk
 switchport trunk allowed vlan 10,20,99
 switchport trunk native vlan 99
 channel-group 1 mode active
exit

interface port-channel 1
 switchport
 switchport mode trunk
 switchport trunk allowed vlan 10,20,99
 switchport trunk native vlan 99
end
```

SW2:

```cisco
conf t
interface range gigabitEthernet1/0/1 - 2
 switchport
 switchport mode trunk
 switchport trunk allowed vlan 10,20,99
 switchport trunk native vlan 99
 channel-group 1 mode active
exit

interface port-channel 1
 switchport
 switchport mode trunk
 switchport trunk allowed vlan 10,20,99
 switchport trunk native vlan 99
end
```

Verification walkthrough:

1. Check that the port-channel is formed.
2. Confirm both member links are bundled.
3. Confirm the trunk allows the intended VLANs.
4. Test connectivity across VLANs.
5. Shut one member interface and confirm the bundle stays up if enough links remain.

Example checks:

```cisco
show etherchannel summary
show interfaces trunk
show interfaces port-channel 1
show spanning-tree interface port-channel 1
```

**Troubleshooting mini case**

A student configures two trunk links with LACP, but the bundle does not form.

Likely causes to check:

- One side uses `mode active`, the other uses `mode on`.
- Native VLAN differs between the two switches.
- One interface is still configured as an access port.
- One member has a speed mismatch.
- The allowed VLAN list differs.

A good troubleshooting sequence is:

1. Compare the physical interface configuration on both ends.
2. Compare the port-channel configuration on both ends.
3. Check `show etherchannel summary` for suspended or individual members.
4. Check `show interfaces trunk` for VLAN consistency.
5. Fix the mismatch and recheck the bundle state.

**Exam focus points**

For CCNA, remember these high-value facts:

- EtherChannel bundles multiple physical links into one logical link.
- LACP is the standard protocol and uses active/passive modes.
- Load balancing is based on a hash and is usually per-flow.
- Member interfaces must match in key parameters.
- STP sees the bundle as one link.
- Verification usually starts with `show etherchannel summary`.

**Quick memory aid**

- **Bundle**: multiple links act as one.
- **Hash**: decides which link a flow uses.
- **Match**: settings must align.
- **Negotiate**: LACP or PAgP forms the bundle.
- **Verify**: check summary, members, and logical interface.
