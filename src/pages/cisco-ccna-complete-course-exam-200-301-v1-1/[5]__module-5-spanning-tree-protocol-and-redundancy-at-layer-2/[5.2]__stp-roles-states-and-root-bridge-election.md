**Spanning Tree Protocol overview**

<abbr title="Spanning Tree Protocol">STP</abbr> prevents Layer 2 switching loops by logically blocking redundant paths while keeping them available for failover. In a switched network, physical redundancy is useful, but without loop prevention, frames can circulate indefinitely, causing broadcast storms, MAC table instability, and duplicate frames.

STP builds a loop-free tree with one switch elected as the **root bridge**. Every other switch calculates the best path to that root and places ports into roles and states based on path cost, bridge ID, and port priority.

- STP operates at Layer 2 and is transparent to end hosts.
- It uses control frames called <abbr title="Bridge Protocol Data Units">BPDUs</abbr> to exchange topology information.
- The root bridge becomes the reference point for all path calculations.
- Redundant links remain available, but some ports are blocked to prevent loops.

**Why STP matters**

Without STP, a single Layer 2 loop can destabilize the entire broadcast domain. Because Ethernet frames do not include a hop count, a loop can persist until links fail or devices are overwhelmed. STP reduces this risk by ensuring only one active logical path exists between any two Layer 2 points.

**Root bridge election**

The root bridge is the switch with the lowest <abbr title="Bridge ID">BID</abbr>. The BID is made up of:

- **Bridge priority**
- **Extended system ID** for VLAN-aware STP
- **MAC address**

The lowest BID wins. If priorities tie, the lowest MAC address wins. In modern Cisco switches, the priority is typically adjusted in increments of 4096 because the extended system ID consumes part of the priority field.

**Root bridge election process**

1. Each switch starts by assuming it is the root and sends BPDUs advertising itself.
2. Switches compare received BPDUs to their current best BPDU.
3. The switch with the lowest BID becomes the root bridge.
4. Non-root switches choose the best root path based on total path cost.
5. Ports are assigned roles and states to create a loop-free topology.

**Practical root bridge control**

You usually want the root bridge to be a central, well-connected distribution switch rather than an access-layer switch. This keeps traffic paths predictable and reduces suboptimal forwarding.

```bash
show spanning-tree
show spanning-tree vlan 10
show spanning-tree root
```

To influence root election on Cisco switches, you can set the bridge priority:

```bash
conf t
spanning-tree vlan 10 priority 4096
end
```

A lower priority makes the switch more likely to become root. Cisco also provides a shortcut:

```bash
conf t
spanning-tree vlan 10 root primary
spanning-tree vlan 10 root secondary
end
```

These commands adjust priority values automatically to bias the election.

**Bridge ID components**

The BID is the key identifier in STP decision-making.

- **Priority**: Lower is better.
- **Extended system ID**: Identifies the VLAN in PVST+ and Rapid PVST+ environments.
- **MAC address**: Used as the final tie-breaker.

Because the extended system ID is included, the effective priority value is tied to the VLAN. This is why you often see priorities like `4096`, `8192`, `24576`, and so on.

**Port roles**

Port roles describe what a port is doing in the STP topology.

- **Root port**: On a non-root switch, the port with the best path to the root bridge.
- **Designated port**: The forwarding port chosen for a segment; one per Layer 2 segment.
- **Blocked port**: A port that does not forward user traffic to prevent loops.
- **Alternate port**: In Rapid STP, a backup path toward the root.
- **Backup port**: In Rapid STP, a backup on the same shared segment as another port on the same switch.

For CCNA-level understanding, focus on root and designated ports first. The root bridge has no root port because it is the root. Every non-root switch has exactly one root port per VLAN instance.

**How root ports are chosen**

A switch selects its root port using these tie-breakers in order:

1. Lowest total path cost to the root bridge
2. Lowest upstream bridge ID
3. Lowest upstream port ID
4. Lowest local port ID if needed

Path cost is based on link speed. Faster links have lower cost, so they are preferred.

Example conceptually:

- A 1 Gbps link is preferred over a 100 Mbps link.
- If two paths have equal cost, STP uses bridge and port identifiers to break the tie.

**Port states**

Port states describe whether a port forwards traffic and learns MAC addresses. Classic 802.1D STP uses five states:

- **Blocking**: Does not forward frames or learn MAC addresses; listens for BPDUs only.
- **Listening**: Does not forward or learn; prepares to participate in the topology.
- **Learning**: Does not forward, but learns MAC addresses.
- **Forwarding**: Forwards frames and learns MAC addresses.
- **Disabled**: Administratively or operationally down.

Rapid STP simplifies this model and converges faster, but the classic states are still important for exam understanding.

**State behavior summary**

| State | Forwards user traffic | Learns MACs | Purpose |
|---|---:|---:|---|
| Blocking | No | No | Prevent loops |
| Listening | No | No | Wait for stable topology |
| Learning | No | Yes | Build MAC table before forwarding |
| Forwarding | Yes | Yes | Normal operation |
| Disabled | No | No | Port is not participating |

**Convergence timing**

Classic STP can take up to about 30 to 50 seconds to converge after a topology change because of timers such as:

- **Hello time**: BPDU interval, commonly 2 seconds
- **Max age**: How long BPDU information is kept, commonly 20 seconds
- **Forward delay**: Time spent in listening and learning, commonly 15 seconds each

These timers help prevent loops during transitions, but they also slow convergence. Rapid STP improves this significantly by using a more aggressive handshake and faster state transitions.

**BPDU basics**

BPDUs are the control messages STP uses to share topology information. They carry the information needed for root election and port decisions, including:

- Root bridge ID
- Sender bridge ID
- Path cost to the root
- Timer values

Switches compare BPDUs to determine which topology information is superior. The “best” BPDU is the one with the lowest root ID, then lowest path cost, then lowest sender bridge ID, and so on.

**Practical verification commands**

Use these commands to inspect STP behavior on Cisco switches:

```bash
show spanning-tree
show spanning-tree vlan 10
show spanning-tree interface gigabitEthernet1/0/1 detail
show spanning-tree summary
```

Useful output fields include:

- Root ID and bridge ID
- Root port
- Port roles and states
- Path cost
- Timers
- Topology change information

**Example topology reasoning**

Consider three switches in a triangle:

- Switch A has the lowest priority, so it becomes root.
- Switch B and Switch C each choose the best path to A as their root port.
- On the link between B and C, one side becomes designated and the other side blocks, depending on BID and path cost.

This is the core STP decision pattern: one root, one root port per non-root switch, and one designated port per segment.

**Common exam points**

- The **root bridge** is elected by the lowest BID, not by the most ports or highest speed.
- The **root port** is the best path *toward* the root on a non-root switch.
- The **designated port** is the best forwarding port *for a segment*.
- A **blocked port** prevents loops and does not forward user traffic.
- STP uses **BPDUs** to maintain topology information.
- Lower **path cost** is preferred over higher path cost.
- Lower **priority** is preferred over lower MAC address only when priorities tie.

**Cisco configuration examples**

Set a predictable root bridge for VLAN 10 on a distribution switch:

```bash
conf t
spanning-tree vlan 10 priority 4096
end
```

Let Cisco choose a root primary value automatically:

```bash
conf t
spanning-tree vlan 10 root primary
end
```

Make a backup root:

```bash
conf t
spanning-tree vlan 10 root secondary
end
```

Check interface participation:

```bash
show spanning-tree interface gigabitEthernet1/0/24
```

**Troubleshooting tips**

If a switch is unexpectedly becoming root, check:

- Its configured priority
- Whether another switch has a lower BID
- VLAN-specific STP settings
- Whether the network is using PVST+ or Rapid PVST+

If a port is blocking unexpectedly, verify:

- Path cost differences
- Port priority and port ID tie-breakers
- Whether the link is redundant by design
- Whether the port is connected to an access switch or a shared segment

> A blocked port is often healthy in STP. Blocking is the mechanism that keeps redundancy from becoming a loop.

**Quick mental model**

Think of STP as a decision tree:

- Elect one root bridge.
- Pick the best root port on each non-root switch.
- Pick one designated port per segment.
- Block the remaining redundant ports.
- Recalculate when the topology changes.
