## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**EtherChannel overview**

EtherChannel bundles multiple physical links into one logical link called a **Port-channel**. You use it to increase bandwidth, add redundancy, and simplify switching design. In CCNA labs, the key idea is that the switch treats the bundle as one interface for spanning tree, trunking, and many Layer 2 settings.

A successful bundle depends on consistent parameters across the member links. If the interfaces do not match on the required settings, the channel usually does not form.

- **Port-channel** is the logical interface.
- **Member interfaces** are the physical links added to the bundle.
- **Load balancing** spreads traffic across links, but a single flow usually uses one physical link.
- **Spanning Tree Protocol (STP)** sees the bundle as one link, which reduces blocked ports.

**Why EtherChannel matters in labs and on the exam**

EtherChannel is a common CCNA topic because it combines switching, trunking, and troubleshooting. You need to know both the configuration and the failure modes.

- It increases available bandwidth by using multiple links together.
- It provides link redundancy if one member fails.
- It reduces STP complexity because the bundle is one logical path.
- It is often used between access switches, distribution switches, and servers.

**EtherChannel negotiation methods**

Cisco switches support several ways to build a bundle. For CCNA, the most important are **LACP** and static EtherChannel.

| Method | Protocol | Notes |
|---|---|---|
| Static | `on` | No negotiation protocol; both sides must be configured manually |
| LACP | IEEE 802.3ad / 802.1AX | Common standard protocol; supports active/passive modes |
| PAgP | Cisco proprietary | Less emphasized in modern CCNA labs, but still worth recognizing |

LACP is usually the safest choice in mixed environments because it is standards-based. Static EtherChannel works, but it gives you no negotiation help, so mismatches are easier to create.

**Core rules for a valid bundle**

The physical interfaces in an EtherChannel must agree on the settings that affect Layer 2 behavior. If they do not, the bundle may fail to form or one interface may be suspended.

- **Speed** must match.
- **Duplex** must match.
- **Switchport mode** must match, such as access or trunk.
- **VLAN membership** must match for access ports.
- **Allowed VLAN list** must match for trunks.
- **Native VLAN** must match for trunks.
- **Channel protocol** must match on both ends if you use LACP or PAgP.
- **Channel-group number** is local to the switch and does not need to match on the far end.

> The channel-group number is only a local identifier. You can use `channel-group 1` on one switch and `channel-group 7` on the other as long as the bundle parameters and protocol settings are compatible.

**What is inherited from the Port-channel**

A common CCNA pitfall is assuming every setting must be configured on every member interface. In practice, many Layer 2 settings are best applied to the **Port-channel** so the bundle stays consistent.

- Configure trunking or access mode on the `Port-channel` when possible.
- Member interfaces inherit many operational characteristics from the logical bundle.
- Some physical settings still must match on the members, especially speed and duplex on platforms that require them.
- If you configure conflicting settings on a member interface, the switch may reject the bundle or suspend the port.

A practical rule: configure the logical interface first, then add the physical interfaces to the channel-group.

**LACP modes and behavior**

LACP uses active and passive states to decide whether a bundle forms.

- **Active** initiates LACP negotiation.
- **Passive** waits for the other side to initiate.
- Two passive interfaces do not form a bundle.
- At least one side must be active for LACP to negotiate.

Example combinations:

- active + active = forms
- active + passive = forms
- passive + passive = does not form

**Basic lab topology**

A simple lab often uses two switches with two parallel links between them.

```text
SW1 Gi0/1 ---- Gi0/1 SW2
SW1 Gi0/2 ---- Gi0/2 SW2
```

You can build the bundle as an access EtherChannel or a trunk EtherChannel depending on the lab goal.

- Use an **access EtherChannel** when the links carry one VLAN.
- Use a **trunk EtherChannel** when the links carry multiple VLANs.
- Use the same physical ports on both ends when possible to reduce mistakes.

**Example: LACP trunk EtherChannel**

This is a common CCNA lab pattern. The Port-channel carries VLANs 10, 20, and 99.

```cisco
! SW1
conf t
interface range gi0/1 - 2
 switchport
 switchport mode trunk
 switchport trunk allowed vlan 10,20,99
 channel-group 1 mode active
exit

interface port-channel 1
 switchport mode trunk
 switchport trunk allowed vlan 10,20,99
end
```

```cisco
! SW2
conf t
interface range gi0/1 - 2
 switchport
 switchport mode trunk
 switchport trunk allowed vlan 10,20,99
 channel-group 1 mode passive
exit

interface port-channel 1
 switchport mode trunk
 switchport trunk allowed vlan 10,20,99
end
```

Notes on the example:

- The member interfaces join `channel-group 1`.
- The Port-channel carries the trunk configuration.
- `active` on one side and `passive` on the other is valid.
- The channel-group number can differ on the other switch, but keeping it the same is easier to read in labs.

**Example: static EtherChannel**

Static EtherChannel uses `mode on` and does not negotiate.

```cisco
conf t
interface range gi0/1 - 2
 switchport
 switchport mode trunk
 switchport trunk allowed vlan 10,20,99
 channel-group 2 mode on
exit

interface port-channel 2
 switchport mode trunk
 switchport trunk allowed vlan 10,20,99
end
```

Use static bundles carefully:

- Both sides must be configured manually.
- A mismatch can create a nonfunctional bundle with little negotiation feedback.
- It is useful for understanding the mechanics of bundling, but LACP is usually preferred in production.

**Access EtherChannel example**

If the bundle carries only one VLAN, configure it as an access link.

```cisco
conf t
interface range gi0/1 - 2
 switchport mode access
 switchport access vlan 30
 channel-group 3 mode active
exit

interface port-channel 3
 switchport mode access
 switchport access vlan 30
end
```

This is useful when connecting to a server or another device that does not need trunking.

**Verification commands**

Use verification commands to confirm that the bundle formed and that the logical interface is carrying traffic.

```cisco
show etherchannel summary
show etherchannel port-channel
show etherchannel 1 detail
show interfaces port-channel 1
show interfaces trunk
show lacp neighbor
```

What to look for:

- `show etherchannel summary` shows whether the bundle is up and which ports are bundled.
- `show etherchannel port-channel` shows the logical interface and member status.
- `show etherchannel 1 detail` gives more detail about the bundle and protocol state.
- `show interfaces port-channel 1` confirms the logical interface status and counters.
- `show interfaces trunk` confirms trunking and allowed VLANs.
- `show lacp neighbor` helps confirm LACP adjacency.

A typical summary output may look like this:

```cisco
SW1# show etherchannel summary
Group  Port-channel  Protocol    Ports
------+-------------+-----------+-------------------------------
1      Po1(SU)         LACP      Gi0/1(P) Gi0/2(P)
```

Interpretation:

- `Po1` is the Port-channel interface.
- `S` means Layer 2 switched.
- `U` means in use.
- `LACP` shows the negotiation protocol.
- `(P)` means the physical port is bundled in the channel.

> Exact formatting can vary by IOS version. Focus on the status flags and whether the member ports show as bundled.

**How to read a failed bundle**

A failed EtherChannel usually shows one of these patterns:

- The Port-channel exists, but member ports are not bundled.
- The member ports are suspended or standalone.
- The protocol state does not match on both sides.
- The trunk or access settings differ between member links.

Example of a problem state:

```cisco
SW1# show etherchannel summary
Group  Port-channel  Protocol    Ports
------+-------------+-----------+-------------------------------
1      Po1(SD)         LACP      Gi0/1(I) Gi0/2(I)
```

Possible interpretation:

- `D` can indicate down.
- `(I)` often indicates the port is standalone or not bundled.
- The bundle did not form correctly, so you should check matching settings.

**Troubleshooting checklist**

When EtherChannel fails, verify the bundle from the bottom up.

1. Confirm the physical links are up with `show interfaces status`.
2. Check that both sides use the same EtherChannel method.
3. Verify that the switchport mode matches on both ends.
4. Confirm trunk settings match if the link is a trunk.
5. Confirm access VLANs match if the link is an access bundle.
6. Check speed and duplex consistency.
7. Verify the Port-channel itself is configured correctly.
8. Look for suspended or standalone member ports in the summary output.

Useful commands:

```cisco
show interfaces status
show run interface gi0/1
show run interface port-channel 1
show etherchannel summary
show etherchannel 1 detail
show lacp neighbor
```

**Common mismatch scenarios**

These are the most common lab mistakes.

- **Trunk/access mismatch**: the bundle usually does not form because the Layer 2 mode is inconsistent.
- **Allowed VLAN mismatch**: the bundle may fail or the ports may be suspended.
- **Native VLAN mismatch**: trunk negotiation may fail or traffic may behave unexpectedly.
- **Speed mismatch**: the physical ports cannot bundle.
- **Duplex mismatch**: the physical ports cannot bundle.
- **Protocol mismatch**: LACP on one side and static `on` on the other do not negotiate.

**Load balancing basics**

EtherChannel does not send one packet stream across all links at once. Instead, it uses a hashing algorithm to choose one member link per flow.

- A single conversation usually stays on one physical link.
- Different flows can use different links.
- The switch hashes fields such as source MAC, destination MAC, source IP, destination IP, or Layer 4 ports depending on platform and configuration.
- Better load balancing happens when the traffic has varied source and destination values.

Practical impact:

- If you have one large file transfer, it may use only one link.
- If you have many users or many sessions, the bundle distributes traffic better.
- Choosing the right hash can improve utilization in real networks.

On many Cisco switches, you can influence the load-balancing method globally.

```cisco
conf t
port-channel load-balance src-dst-ip
end
```

Common options vary by platform, but the idea is the same:

- Use MAC-based hashing in simple Layer 2 environments.
- Use IP-based hashing when traffic patterns are more Layer 3 oriented.
- Use source and destination fields to spread flows more evenly.

**Practical lab scenario**

Imagine SW1 connects to SW2 with two links. You configure one side as a trunk and forget to allow VLAN 99 on the Port-channel. The bundle may form, but management traffic in VLAN 99 does not pass.

What you should check:

- The Port-channel trunk allowed VLAN list.
- The member interfaces for inherited or conflicting settings.
- The VLAN status on both switches.
- The SVI or management interface if you are testing reachability.

This is a good reminder that a formed EtherChannel is not enough. The logical interface must also carry the correct VLANs and Layer 2 settings.

**Cleanup and reset**

When you finish a lab, remove the channel-group from the member interfaces and then clear the Port-channel configuration if needed.

```cisco
conf t
interface range gi0/1 - 2
 no channel-group 1
exit
no interface port-channel 1
end
```

If you need to return a physical interface to default settings, do it as a separate global configuration action.

```cisco
conf t
default interface gi0/1
default interface gi0/2
end
```

Use cleanup carefully:

- Remove the bundle before reusing the ports in another lab.
- Clear the Port-channel if you want to rebuild from scratch.
- Verify that no stale trunk or access settings remain.

**CCNA exam pitfalls**

- The **channel-group number** is local and does not need to match on both switches.
- The **Port-channel** is the interface you should think about first when configuring trunk or access behavior.
- Two passive LACP interfaces do not form a bundle.
- A trunk EtherChannel still needs matching trunk parameters on both sides.
- A bundle can appear up while a VLAN problem still blocks traffic.
- Verification output matters more than memorizing one exact formatting style.

**Mini case study**

You configure two links between SW1 and SW2 for a trunk EtherChannel. The summary shows one port bundled and one port suspended. You also notice that VLAN 20 traffic does not pass.

Likely causes:

- One member interface has a different trunk allowed VLAN list.
- One side uses access mode on a member interface.
- The Port-channel is configured correctly, but one physical interface still has a conflicting setting.
- The LACP state is not fully negotiated on both links.

Best response:

- Compare `show run interface gi0/1`, `show run interface gi0/2`, and `show run interface port-channel 1`.
- Make the member interfaces consistent.
- Recheck `show etherchannel summary` until both ports show as bundled.

**Key takeaways**

- EtherChannel bundles multiple physical links into one logical Port-channel.
- LACP is the most important negotiation protocol for CCNA labs.
- Member interfaces must match on the settings that affect bundling.
- Configure the logical Port-channel consistently and verify with show commands.
- Troubleshooting starts with the summary output and then moves to interface-level mismatches.
