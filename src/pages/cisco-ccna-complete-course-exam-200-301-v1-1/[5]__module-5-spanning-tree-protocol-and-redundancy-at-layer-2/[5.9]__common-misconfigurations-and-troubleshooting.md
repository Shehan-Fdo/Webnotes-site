## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Troubleshooting mindset**

Troubleshooting on Cisco networks is usually a process of elimination. You verify the symptom, isolate the scope, check the most likely failure points, and confirm the fix with repeatable commands. The CCNA expects you to recognize common misconfigurations quickly, not to guess.

A practical workflow is:

1. Identify the affected device, VLAN, subnet, or service.
2. Check physical and interface status first.
3. Verify Layer 2, then Layer 3, then services such as DHCP, NAT, and routing.
4. Compare the working side to the broken side.
5. Confirm the fix with `show` commands and end-to-end testing.

**Common verification commands**

These commands help you establish a baseline before changing anything:

```bash
show ip interface brief
show interfaces status
show vlan brief
show interfaces trunk
show ip route
show running-config
ping
traceroute
```

- `show ip interface brief` quickly reveals administratively down interfaces, wrong IPs, and unassigned interfaces.
- `show interfaces status` helps you spot err-disabled ports, speed/duplex issues, and inactive links.
- `show ip route` confirms whether the router knows the destination network.
- `ping` and `traceroute` validate reachability and path selection.

**Layer 1 and Layer 2 issues**

Many CCNA troubleshooting problems begin with a simple physical or switching mistake. If a host cannot reach its gateway, verify the cable, port state, VLAN assignment, and trunking before moving to routing.

Common Layer 2 problems include:

- **Access port VLAN mismatch**: the switchport is in the wrong VLAN, so the host lands in the wrong broadcast domain.
- **Trunk not formed**: one side is configured as access while the other expects a trunk.
- **Native VLAN mismatch**: untagged traffic lands in different VLANs on each side of the trunk.
- **Allowed VLAN mismatch**: the trunk exists, but the needed VLAN is not permitted.
- **DTP negotiation mismatch**: Dynamic Trunking Protocol (DTP) settings prevent trunk formation when one side expects negotiation and the other side does not.
- **Port security violation**: the port enters an err-disabled or restricted state after a MAC address violation.

Useful checks:

```bash
show interfaces status
show interfaces switchport
show interfaces trunk
show vlan brief
show port-security interface gi1/0/10
show errdisable recovery
```

> A trunk can be up and still fail to carry the VLAN you need. Always verify the allowed VLAN list and the native VLAN, not just link status.

**Trunking and VLAN misconfigurations**

Trunk problems are common when you connect switches or a switch to a router. A trunk may appear operational while still dropping the traffic you care about.

Watch for these issues:

- The port is left in access mode instead of trunk mode.
- The trunk is up, but the required VLAN is missing from the allowed list.
- The native VLAN differs on each end of the link.
- The encapsulation expectation does not match the platform, where applicable.
- The VLAN does not exist on the switch, so traffic has nowhere to go.

Example verification:

```bash
show interfaces trunk
show vlan brief
show running-config interface gi1/0/1
```

Example fix:

```bash
interface gi1/0/1
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30
 switchport trunk native vlan 99
```

**Switch management VLAN and SVI troubleshooting**

A Layer 2 switch is managed through a switched virtual interface (SVI), usually `interface vlan X`. If the management SVI is wrong, you may lose remote access even though user ports still work.

Common mistakes include:

- The SVI IP address is incorrect or in the wrong subnet.
- The SVI is administratively shut down.
- No active port exists in that VLAN, so the SVI stays down/down.
- The default gateway is missing on the Layer 2 switch.
- The management VLAN is not allowed on the trunk carrying management traffic.

Example:

```bash
show ip interface brief
show running-config interface vlan 10
show running-config | include ip default-gateway
```

Typical Layer 2 switch management configuration:

```bash
interface vlan 10
 ip address 192.168.10.2 255.255.255.0
 no shutdown

ip default-gateway 192.168.10.1
```

- The SVI must be up/up for remote management.
- The default gateway is required on a Layer 2 switch for off-subnet management access.
- The management VLAN must be active somewhere in the switch fabric.

**Routing and IP addressing problems**

Routing failures often come from incorrect addressing, missing routes, or bad next-hop information. Before blaming the routing protocol, verify that the interfaces themselves are correct.

Common issues include:

- Wrong subnet mask on an interface, which breaks directly connected route recognition.
- Wrong IP address on the interface.
- Interface shutdown.
- Static route points to the wrong next-hop address.
- Static route uses the wrong exit interface.
- Summarized route hides a more specific path or points traffic to the wrong place.
- Default route is missing or incorrectly advertised.

Useful checks:

```bash
show ip interface brief
show ip route
show running-config | include ip route
show cdp neighbors detail
show arp
```

Static route examples:

```bash
ip route 10.10.20.0 255.255.255.0 192.168.1.2
ip route 10.10.20.0 255.255.255.0 g0/0/1
```

- A **next-hop** static route is usually easier to troubleshoot on multiaccess networks.
- An **exit-interface** static route can work well on point-to-point links.
- If you use the wrong mask, the route may never match the destination you expect.

> On Ethernet multiaccess networks, a static route that points only to an exit interface can create ambiguity if the router must resolve the destination at Layer 2. Verify the design before choosing the route style.

**Routing protocol troubleshooting**

Routing protocols can fail because of interface issues, mismatched network statements, passive interfaces, authentication problems, or filtering. The exact symptoms depend on the protocol.

For OSPF, verify:

- The interface is in the correct area.
- The network type and timers are compatible when needed.
- Neighbor adjacency forms successfully.
- The router ID is unique.
- Authentication matches if configured.
- ACLs are not blocking OSPF packets, which use IP protocol 89 and multicast addresses `224.0.0.5` and `224.0.0.6`.

Commands:

```bash
show ip ospf neighbor
show ip ospf interface brief
show ip protocols
show ip route ospf
```

For EIGRP, verify:

- The correct networks are included.
- Interfaces are not accidentally passive.
- Authentication matches if used.
- The AS number matches on neighbors.
- K-values match if custom metrics are configured, though this is less common in basic CCNA scenarios.

Commands:

```bash
show ip eigrp neighbors
show ip protocols
show ip route eigrp
```

For RIP, verify:

- The correct version is used.
- Auto-summary behavior does not create unexpected routes in older designs.
- The network statements include the intended interfaces.

Commands:

```bash
show ip protocols
show ip route rip
```

**DHCP troubleshooting**

DHCP (Dynamic Host Configuration Protocol) issues are common in both client access and router-based labs. Problems usually fall into one of four categories: the client cannot reach the server, the server has no available addresses, the relay is missing, or the server configuration is wrong.

Common DHCP failures:

- The client interface is not set to obtain an address automatically.
- The DHCP server pool does not match the client subnet.
- The default gateway option is wrong or missing.
- The pool is exhausted.
- The server interface or service is down.
- A relay agent is missing on the router interface facing the client subnet.
- The helper address points to the wrong server.

Useful verification:

```bash
show ip dhcp binding
show ip dhcp pool
show running-config | section dhcp
show ip interface brief
show running-config interface g0/0
```

Example DHCP server configuration:

```bash
ip dhcp excluded-address 192.168.10.1 192.168.10.20

ip dhcp pool LAN10
 network 192.168.10.0 255.255.255.0
 default-router 192.168.10.1
 dns-server 8.8.8.8
```

Example relay configuration:

```bash
interface g0/0
 ip address 192.168.10.1 255.255.255.0
 ip helper-address 10.1.1.10
```

- `ip helper-address` forwards DHCP broadcasts to a remote server.
- The helper belongs on the client-facing interface, not on the server-facing interface.
- `show ip dhcp binding` confirms which addresses have been leased.
- `show ip dhcp pool` helps you detect exhaustion and pool utilization.

A practical DHCP troubleshooting sequence is:

1. Verify the client is on the correct VLAN and can reach the default gateway.
2. Confirm the relay helper exists if the server is remote.
3. Check that the DHCP pool matches the client subnet and mask.
4. Verify exclusions do not consume the entire pool.
5. Confirm the server has bindings available and the service is active.

**IPv4 NAT troubleshooting**

Network Address Translation (NAT) problems often come from interface roles, ACL matching, or missing routes. PAT (Port Address Translation) is the common overload form that maps many inside hosts to one public address using ports.

Common mistakes include:

- Inside and outside interfaces are reversed.
- The NAT ACL matches the wrong source subnet.
- The NAT ACL is written to match destination traffic, but NAT standard ACLs match source addresses.
- The inside local network is not routed toward the NAT device.
- The outside return path is missing.
- The translation table is empty because traffic never matches the NAT rule.

Example:

```bash
access-list 1 permit 192.168.10.0 0.0.0.255

interface g0/0
 ip nat inside

interface g0/1
 ip nat outside

ip nat inside source list 1 interface g0/1 overload
```

Verification commands:

```bash
show ip nat translations
show ip nat statistics
show access-lists
```

- `show ip nat translations` shows active mappings.
- `show ip nat statistics` helps confirm hits and misses.
- The NAT ACL should match the inside source addresses only.

**IPv6 troubleshooting**

IPv6 troubleshooting requires you to distinguish between global unicast addressing, link-local addressing, SLAAC (Stateless Address Autoconfiguration), and DHCPv6. A host may appear partially configured and still fail to reach the network.

Common issues include:

- IPv6 routing is disabled on the router.
- The interface lacks a global unicast address.
- The link-local address is wrong or missing.
- Router Advertisement (RA) flags do not match the intended host configuration.
- Stateless DHCPv6 is configured, but the host still needs SLAAC for the address.
- Stateful DHCPv6 is configured, but the client is not requesting DHCPv6 properly.
- The default gateway is not learned because the router advertisement is missing.
- The next hop is reachable only via link-local, and the route or neighbor information is not being used correctly.

Useful commands:

```bash
show ipv6 interface brief
show ipv6 route
show ipv6 neighbors
show running-config | section ipv6
```

Example router configuration for SLAAC:

```bash
ipv6 unicast-routing

interface g0/0
 ipv6 address 2001:db8:10:1::1/64
 ipv6 address fe80::1 link-local
 no shutdown
```

Example for stateful DHCPv6:

```bash
ipv6 unicast-routing

ipv6 dhcp pool LANv6
 address prefix 2001:db8:10:1::/64
 dns-server 2001:4860:4860::8888
 domain-name example.local

interface g0/0
 ipv6 address 2001:db8:10:1::1/64
 ipv6 nd managed-config-flag
 ipv6 dhcp server LANv6
 no shutdown
```

- `ipv6 nd managed-config-flag` tells hosts to use DHCPv6 for addressing.
- Without the correct RA flag, a stateful DHCPv6 pool alone is not enough for clients to behave as expected.
- For stateless DHCPv6, hosts use SLAAC for the address and DHCPv6 for other information such as DNS.

Example for stateless DHCPv6:

```bash
ipv6 dhcp pool LANv6-STATELESS
 dns-server 2001:4860:4860::8888
 domain-name example.local

interface g0/0
 ipv6 address 2001:db8:10:1::1/64
 ipv6 nd other-config-flag
 ipv6 dhcp server LANv6-STATELESS
```

**Wireless troubleshooting**

Wireless issues can occur at the AP, controller, or client layer. On controller-based platforms such as a Wireless LAN Controller (WLC) or Catalyst 9800, you verify AP join status, WLAN-to-VLAN mapping, and client association state.

Common problems include:

- The AP cannot join the controller because of CAPWAP (Control and Provisioning of Wireless Access Points) discovery or reachability issues.
- The WLAN is disabled or mapped to the wrong VLAN.
- The SSID is broadcast, but authentication fails.
- The client associates but does not receive an IP address.
- RF (radio frequency) coverage is weak or channel overlap is excessive.
- The controller sees the AP, but the client remains in an unauthenticated or DHCP state.

Useful controller-oriented checks:

```bash
show ap summary
show wlan summary
show wireless client summary
show wireless client mac-address <mac>
show ap join stats summary
```

Useful AP-side checks:

```bash
show capwap client config
show capwap client rcb
show ip interface brief
show logging
```

- `show ap summary` confirms whether the AP is joined and operational.
- `show wireless client summary` helps you see association and authentication states.
- `show wireless client mac-address <mac>` is useful for a specific client failure.
- If the AP cannot join, verify IP reachability, DNS, DHCP, and controller discovery.

A practical wireless verification flow is:

1. Confirm the AP has an IP address and can reach the controller.
2. Confirm the AP joins successfully and appears in the controller summary.
3. Verify the WLAN is enabled and mapped to the correct VLAN.
4. Check client association, authentication, and DHCP status.
5. Confirm the client receives an IP address from the expected subnet.

**ACL and filtering issues**

Access Control Lists (ACLs) can block traffic in ways that look like routing failures. The key is to identify whether the packet is being filtered on ingress or egress and whether the ACL is applied to the correct interface and direction.

Common mistakes include:

- ACL applied to the wrong interface.
- ACL applied in the wrong direction.
- Missing permit statement for return traffic.
- Overly broad deny statements.
- Wrong source or destination match.
- Forgotten implicit deny at the end of the ACL.

Verification:

```bash
show access-lists
show ip interface
show running-config | include access-group
```

- Standard ACLs match source IP addresses only.
- Extended ACLs can match protocol, source, destination, and ports.
- The implicit deny at the end of every ACL can block traffic if you do not add the needed permit statements.

**Port security and err-disabled recovery**

Port security protects switch ports by limiting MAC addresses and defining a violation action. A violation can shut the port down, restrict traffic, or protect silently depending on the configuration.

Common symptoms:

- The port is err-disabled after a MAC violation.
- A device was replaced and now has a different MAC address.
- The maximum MAC count is too low.
- Sticky MAC addresses were learned from the wrong device.

Useful commands:

```bash
show port-security
show port-security interface gi1/0/10
show interfaces status err-disabled
show errdisable recovery
```

Example:

```bash
interface gi1/0/10
 switchport mode access
 switchport port-security
 switchport port-security maximum 1
 switchport port-security mac-address sticky
 switchport port-security violation shutdown
```

- `shutdown` is the most disruptive violation mode.
- `restrict` drops violating frames and increments counters.
- `protect` drops violating frames without logging as much detail.
- If err-disable recovery is enabled, the port may recover automatically after the timer expires.

**Practical end-to-end scenarios**

A few realistic scenarios help you connect symptoms to likely causes.

**Scenario 1: Host cannot get a DHCP address**

- Check the host VLAN and switchport mode.
- Verify the SVI or router interface for that VLAN is up.
- Confirm the DHCP relay helper exists if the server is remote.
- Check `show ip dhcp pool` for exhaustion.
- Confirm the server pool matches the subnet and default gateway.

**Scenario 2: Remote management to a switch fails**

- Verify the management SVI has the correct IP address.
- Confirm the SVI is not shut down.
- Check that at least one port in the VLAN is active.
- Verify `ip default-gateway` on the Layer 2 switch.
- Confirm the management VLAN is allowed across trunks.

**Scenario 3: Two switches connect, but hosts in one VLAN cannot communicate**

- Check trunk status on both ends.
- Verify the VLAN is allowed on the trunk.
- Confirm the native VLAN matches.
- Ensure the VLAN exists on both switches.
- Check whether DTP negotiation prevented trunk formation.

**Scenario 4: IPv6 host gets an address but no internet or no DNS**

- Determine whether the host should use SLAAC, stateless DHCPv6, or stateful DHCPv6.
- Verify the router advertisement flags match the design.
- Confirm the router has `ipv6 unicast-routing` enabled.
- Check the IPv6 default gateway via the link-local router address.
- Verify the DHCPv6 pool includes the correct DNS server if needed.

**Troubleshooting command summary**

| Area | Primary checks | Common clue |
|---|---|---|
| Interfaces | `show ip interface brief`, `show interfaces status` | Down, shut, or wrong IP |
| VLANs/trunks | `show vlan brief`, `show interfaces trunk` | VLAN missing or trunk mismatch |
| SVI/management | `show ip interface brief`, `show running-config interface vlan X` | SVI down or wrong gateway |
| Routing | `show ip route`, `show ip protocols` | Missing route or bad next hop |
| DHCP | `show ip dhcp binding`, `show ip dhcp pool` | No lease or exhausted pool |
| NAT | `show ip nat translations`, `show ip nat statistics` | No translation or wrong ACL |
| IPv6 | `show ipv6 interface brief`, `show ipv6 route` | Missing RA, link-local, or prefix |
| Wireless | `show ap summary`, `show wireless client summary` | AP not joined or client not associated |

**Exam approach**

For CCNA troubleshooting questions, read the symptom carefully and map it to the layer most likely responsible. If a host cannot reach its gateway, start with the access port, VLAN, and SVI. If a host reaches the gateway but not a remote network, check routing, NAT, ACLs, and default routes. If a service fails only for new hosts, think DHCP or IPv6 autoconfiguration before assuming a routing issue.

The best habit is to verify one layer at a time and avoid changing multiple variables at once. That keeps your diagnosis clear and makes it easier to confirm the real cause.
