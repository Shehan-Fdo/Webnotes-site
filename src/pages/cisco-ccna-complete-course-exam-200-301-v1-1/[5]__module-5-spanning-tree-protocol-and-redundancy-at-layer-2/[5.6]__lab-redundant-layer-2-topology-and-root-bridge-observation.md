## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Lab goal and what you should observe**

This lab builds a small redundant Layer 2 topology so you can see <abbr title="Spanning Tree Protocol">STP</abbr> prevent loops while still keeping backup links available. In a Cisco CCNA environment, the key outcome is not just knowing the theory of STP, but recognizing how the root bridge is elected, which ports forward, and which ports block or alternate when redundancy exists.

The lab is typically done with three switches in a triangle or partial mesh. You connect them with trunk links so the same VLAN can span multiple switches, then verify which switch becomes the root bridge and how each non-root switch selects its root port.

**Typical lab topology**

A simple and effective topology uses three switches:

- `S1` connected to `S2`
- `S2` connected to `S3`
- `S1` connected to `S3`

This creates a Layer 2 loop if STP is not active. To make the lab more realistic, place end devices on access ports and use trunks between switches.

Example interface plan:

- `S1`
  - `Gi0/1` to `S2 Gi0/1`
  - `Gi0/2` to `S3 Gi0/1`
  - `Gi0/3` access port for a PC in VLAN 10
- `S2`
  - `Gi0/1` to `S1 Gi0/1`
  - `Gi0/2` to `S3 Gi0/2`
  - `Gi0/3` access port for a PC in VLAN 10
- `S3`
  - `Gi0/1` to `S1 Gi0/2`
  - `Gi0/2` to `S2 Gi0/2`
  - `Gi0/3` access port for a PC in VLAN 10

> Keep the links between switches as trunks. If you leave them as access ports in different VLANs, you will not create the same spanning-tree behavior across the redundant Layer 2 domain.

**Why this lab matters**

STP exists to stop broadcast storms, MAC address table instability, and duplicate frames caused by switching loops. In a redundant topology, you want backup paths for resilience, but you do not want all paths forwarding at the same time. STP solves that by electing a root bridge and placing some ports into a non-forwarding role.

For CCNA, you should be able to:

- identify the root bridge from `show spanning-tree`
- predict which port becomes the root port on a non-root switch
- recognize designated ports on each segment
- identify blocked or alternate ports
- explain why the output changes per VLAN in PVST+ and Rapid PVST+

**STP mode used in the lab**

Most Cisco CCNA labs assume <abbr title="Per-VLAN Spanning Tree Plus">PVST+</abbr> or Rapid PVST+ on Cisco switches, because Cisco IOS commonly runs per-VLAN spanning tree by default on many lab platforms. That means each VLAN can elect its own root bridge.

For a single-VLAN lab, the behavior is easy to follow:

- VLAN 10 has its own root bridge election
- `show spanning-tree vlan 10` shows the topology for that VLAN only
- if you add VLAN 20 later, it can have a different root bridge

If your lab uses classic 802.1D STP, the concepts are the same, but convergence is slower and the output emphasizes classic states more strongly. If it uses Rapid PVST+, the roles are similar, but convergence is faster and the output may show <abbr title="Rapid Spanning Tree Protocol">RSTP</abbr> terminology such as alternate ports.

**Build the redundant topology**

1. Create the VLAN you will test, such as VLAN 10.
2. Configure the inter-switch links as trunks.
3. Configure at least one access port on each switch for a host in the same VLAN.
4. Ensure all switches can see the same VLAN across the trunk links.
5. Verify that the topology contains a loop by checking that all three switches are interconnected.

Example configuration:

```cisco
! On all switches
conf t
vlan 10
 name USERS
exit

! S1
interface gi0/1
 description Trunk to S2
 switchport mode trunk
 switchport trunk allowed vlan 10
 no shutdown

interface gi0/2
 description Trunk to S3
 switchport mode trunk
 switchport trunk allowed vlan 10
 no shutdown

interface gi0/3
 description Access to PC1
 switchport mode access
 switchport access vlan 10
 spanning-tree portfast
 no shutdown

! S2
interface gi0/1
 description Trunk to S1
 switchport mode trunk
 switchport trunk allowed vlan 10
 no shutdown

interface gi0/2
 description Trunk to S3
 switchport mode trunk
 switchport trunk allowed vlan 10
 no shutdown

interface gi0/3
 description Access to PC2
 switchport mode access
 switchport access vlan 10
 spanning-tree portfast
 no shutdown

! S3
interface gi0/1
 description Trunk to S1
 switchport mode trunk
 switchport trunk allowed vlan 10
 no shutdown

interface gi0/2
 description Trunk to S2

 switchport trunk allowed vlan 10
 no shutdown

interface gi0/3
 description Access to PC3
 switchport mode access
 switchport access vlan 10

 no shutdown
end
```

**Root bridge election rules**

STP chooses the root bridge based on the lowest Bridge ID. The Bridge ID is made from:

- bridge priority
- extended system ID, which includes the VLAN in PVST+/Rapid PVST+
- MAC address as the tie-breaker

The switch with the lowest priority wins. If priorities are equal, the lowest MAC address wins.

Common lab practice is to force one switch to become root by lowering its priority:

```cisco
conf t
spanning-tree vlan 10 root primary
end
```

Or set the priority manually:

```cisco
conf t
spanning-tree vlan 10 priority 4096
end
```

If you want a backup root, use:

```cisco
conf t
spanning-tree vlan 10 root secondary
end
```

**What to expect in the output**

On the root bridge:

- all active trunk ports for that VLAN are usually designated ports
- there is no root port on the root bridge
- the root bridge advertises itself as the root in STP BPDUs

On a non-root switch:

- one port becomes the **root port**
- one or more ports become **designated ports**
- the redundant path may become **alternate** or **blocked**, depending on STP flavor and output style


How to read this:

- `Root ID` shows the elected root bridge for the VLAN
- `This bridge is the root` means the local switch is the root
- `Root Port` identifies the best path toward the root on a non-root switch
- `Desg` means designated port for that segment
- `Altn` means alternate port, usually blocking in RSTP-style output
- `BLK` means the port is not forwarding frames for that VLAN

**Classic STP states versus RSTP roles**

Cisco output can be confusing because classic STP and RSTP use different terminology.

| Concept | Classic STP | RSTP / Rapid PVST+ |
|---|---|---|
| Forwarding path toward root | Root port | Root port |
| Best port on a segment | Designated port | Designated port |
| Backup path | Blocked port | Alternate port |
| Learning/forwarding behavior | Separate states | Faster transition, fewer states |
| Common output clue | `BLK`, `LIS`, `LRN`, `FWD` | `Altn`, `Backup`, `Discarding`, `Learning`, `Forwarding` |

In classic 802.1D STP, ports move through the states:

- blocking
- listening
- learning
- forwarding
- disabled

In Rapid STP, the port roles are more important than the old state progression. You may still see state-like output, but the protocol converges faster and uses proposal/agreement behavior to move ports to forwarding more quickly.

**Per-VLAN behavior in PVST+ and Rapid PVST+**

In Cisco per-VLAN STP, each VLAN runs its own spanning-tree instance. That means:

- VLAN 10 can use `S1` as root
- VLAN 20 can use `S2` as root
- the blocked link for VLAN 10 may be different from the blocked link for VLAN 20

This is important in labs because the same physical topology can produce different forwarding paths per VLAN. If you only check one VLAN, you may miss that another VLAN is using a different root bridge and a different blocked port.

Example:

```cisco
show spanning-tree vlan 10
show spanning-tree vlan 20
```

If you intentionally set different root priorities, you can observe load sharing across VLANs:

```cisco
! S1
spanning-tree vlan 10 root primary

! S2
spanning-tree vlan 20 root primary
```

That design lets one switch forward for VLAN 10 while another switch forwards for VLAN 20, improving link utilization without creating loops.

**Verification commands you should know**

Use these commands to confirm the lab behavior:

```cisco
show spanning-tree
show spanning-tree vlan 10
show spanning-tree summary
show spanning-tree root
show spanning-tree interface gi0/1 detail
show interfaces trunk
show vlan brief
show cdp neighbors
```

What each command helps you confirm:

- `show spanning-tree vlan 10` shows the root bridge, local bridge ID, and port roles for VLAN 10
- `show spanning-tree root` quickly identifies the current root bridge and root port
- `show spanning-tree interface gi0/1 detail` helps you inspect a specific port
- `show interfaces trunk` confirms trunk negotiation and allowed VLANs
- `show cdp neighbors` helps verify physical switch-to-switch connections

**Practical lab procedure**

1. Build the triangle topology with three switches.
2. Create VLAN 10 on all switches.
3. Configure the inter-switch links as trunks and allow VLAN 10.
4. Configure access ports for end hosts in VLAN 10.
5. Set one switch as the preferred root for VLAN 10.
6. Verify the root bridge with `show spanning-tree vlan 10`.
7. Identify the root port on each non-root switch.
8. Identify the redundant link that becomes blocked or alternate.
9. Change the root priority on another switch and observe how the roles change.
10. Repeat the verification to confirm per-VLAN behavior if you add another VLAN.

**Expected lab observations**

When the topology is stable, you should see:

- one root bridge per VLAN
- all ports on the root bridge forwarding for that VLAN
- each non-root switch using one best path toward the root
- one redundant path not forwarding for that VLAN
- no Layer 2 loop symptoms such as broadcast storms or MAC flapping

If you change the root bridge, the blocked port may move to a different link. That is normal and is one of the main learning goals of the lab.

**Troubleshooting tips**

If the output does not match your expectation, check these items first:

- the trunk links are up and carrying the correct VLAN
- the VLAN exists on all switches
- the access ports are in the intended VLAN
- the switch you expect to be root has the lowest priority
- no interface is administratively down
- the lab platform is using PVST+ or Rapid PVST+ as expected

> If a port stays forwarding when you expect it to block, verify that the topology actually forms a loop in the same VLAN. STP only blocks redundant paths within the same Layer 2 broadcast domain.

**Exam-focused takeaways**

- STP prevents loops by blocking redundant Layer 2 paths.
- The root bridge is elected by the lowest Bridge ID.
- In Cisco per-VLAN STP, root election happens separately for each VLAN.
- The root bridge has no root port.
- Non-root switches choose one root port toward the root bridge.
- Redundant links become blocked or alternate depending on the STP mode and output style.
- Rapid PVST+ converges faster than classic STP, but the root election logic is the same.

**Quick recognition guide**

- `This bridge is the root` means the local switch is root for that VLAN.
- `Root Port` identifies the best path toward the root.
- `Desg FWD` means the port forwards traffic for that segment.
- `Altn BLK` or `Discarding` means the port is not forwarding and serves as a backup path.
- Different VLANs can have different roots and different blocked ports.
