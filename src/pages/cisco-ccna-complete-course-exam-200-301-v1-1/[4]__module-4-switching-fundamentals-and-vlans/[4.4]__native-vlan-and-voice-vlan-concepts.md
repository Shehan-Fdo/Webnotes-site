**Native VLAN overview**

A **VLAN** (Virtual Local Area Network) separates Layer 2 broadcast domains on a switch. On an IEEE 802.1Q trunk, most VLAN traffic is tagged so the receiving switch knows which VLAN the frame belongs to. The **native VLAN** is the VLAN whose frames are sent across the trunk **without an 802.1Q tag**.

This matters because untagged frames on a trunk are assumed to belong to the native VLAN. If the two ends of a trunk use different native VLANs, you can create traffic leaks, spanning-tree issues, and hard-to-troubleshoot connectivity problems.

- The native VLAN applies to **802.1Q trunks**.
- Untagged frames on the trunk map to the native VLAN.
- Cisco switches default the native VLAN to VLAN 1, but best practice is to change it.
- The native VLAN should match on both ends of the trunk.

**Why the native VLAN exists**

The native VLAN provides compatibility with devices or traffic that do not tag frames. In modern switched networks, you usually use it only for trunk behavior, not for user traffic. Keeping it unused for endpoints reduces the risk of accidental bridging between untagged traffic and production VLANs.

> A native VLAN mismatch can cause untagged traffic to land in the wrong VLAN and may generate Cisco CDP/LLDP warnings or spanning-tree inconsistencies.

**802.1Q trunk tagging behavior**

On a trunk, each Ethernet frame carries a VLAN tag except frames in the native VLAN. The switch uses the tag to forward the frame only to ports in the same VLAN. When a frame exits a trunk, the switch removes the tag if the frame belongs to the native VLAN.

Common trunk behavior:

- Tagged frames carry a VLAN ID from 1 to 4094.
- The native VLAN is untagged on the trunk.
- The native VLAN is still a VLAN; it is not “no VLAN.”
- You can change the native VLAN to a dedicated unused VLAN for better design.

**Practical trunk example**

```bash
Switch(config)# vlan 999
Switch(config-vlan)# name NATIVE-BLACKHOLE
Switch(config-vlan)# exit

Switch(config)# interface gigabitEthernet0/1
Switch(config-if)# switchport mode trunk
Switch(config-if)# switchport trunk native vlan 999
Switch(config-if)# switchport trunk allowed vlan 10,20,30,999
```

In this example, VLAN 999 becomes the native VLAN. Many designs use a dedicated native VLAN that is not assigned to user access ports. Some administrators choose a “blackhole” VLAN name to make accidental use obvious.

**Native VLAN best practices**

- Use a dedicated native VLAN that is not used for end-user access.
- Match the native VLAN on both trunk endpoints.
- Avoid VLAN 1 for user traffic and, where possible, for the native VLAN.
- Prune allowed VLANs on trunks so only required VLANs traverse the link.
- Verify trunk status after changes to prevent outages.

**Useful verification commands**

```bash
Switch# show interfaces trunk
Switch# show interfaces gigabitEthernet0/1 switchport
Switch# show vlan brief
Switch# show spanning-tree vlan 999
```

These commands help you confirm the trunk mode, native VLAN, allowed VLAN list, and spanning-tree state. `show interfaces trunk` is especially useful because it displays the native VLAN and which VLANs are active on the trunk.

**Voice VLAN overview**

A **voice VLAN** is a separate VLAN used for IP phones. It keeps voice traffic isolated from data traffic and allows the switch to treat voice frames differently from regular endpoint traffic. In a typical Cisco deployment, the phone connects to the switch, and the PC connects through the phone’s built-in switch port.

The switchport often carries:

- **Data traffic** for the attached PC
- **Voice traffic** for the IP phone

This design lets you place the phone and PC on different VLANs while using a single physical switch port.

**Why use a voice VLAN**

Voice traffic is sensitive to delay, jitter, and packet loss. Separating voice from data helps with:

- **QoS** (Quality of Service) marking and prioritization
- Cleaner broadcast containment
- Easier policy enforcement
- Simplified IP phone deployment

Cisco switches can automatically identify voice devices and place their traffic into the configured voice VLAN. The phone typically tags its voice frames, while the attached PC sends untagged data frames.

**How voice VLAN works on an access port**

A switchport configured for voice VLAN usually behaves like this:

- The port remains an access port for the data VLAN.
- The phone uses the voice VLAN for its own traffic.
- The PC behind the phone uses the data VLAN.
- The switch may trust or remark voice QoS markings depending on configuration.

A common Cisco configuration looks like this:

```bash
Switch(config)# vlan 10
Switch(config-vlan)# name DATA
Switch(config-vlan)# exit

Switch(config)# vlan 20
Switch(config-vlan)# name VOICE
Switch(config-vlan)# exit

Switch(config)# interface fastEthernet0/10
Switch(config-if)# switchport mode access
Switch(config-if)# switchport access vlan 10
Switch(config-if)# switchport voice vlan 20
Switch(config-if)# spanning-tree portfast
```

In this example:

- VLAN 10 carries data for the PC.
- VLAN 20 carries voice for the IP phone.
- `spanning-tree portfast` helps the phone come online faster by skipping the normal STP listening/learning delay on an edge port.

**Voice VLAN discovery and signaling**

Cisco IP phones can learn the voice VLAN through switch assistance mechanisms such as CDP (Cisco Discovery Protocol). In mixed-vendor environments, LLDP-MED (Link Layer Discovery Protocol - Media Endpoint Discovery) may be used instead. The exact behavior depends on the phone model and switch configuration.

Key ideas:

- The phone may initially connect untagged.
- The switch advertises the voice VLAN to the phone.
- The phone then tags voice traffic with the voice VLAN ID.
- The PC traffic remains in the access VLAN unless otherwise configured.

**Voice VLAN and QoS**

Voice VLANs are often paired with QoS because voice packets need priority over best-effort data. The VLAN itself does not guarantee quality; it simply separates traffic. QoS policies provide the actual prioritization.

Typical voice-related QoS goals:

- Mark voice packets with higher priority values.
- Preserve markings across the network where appropriate.
- Reduce latency and jitter on congested links.

A switch may trust the phone’s markings or apply its own policy. In Cisco environments, voice traffic is commonly marked with DSCP (Differentiated Services Code Point) values such as EF (Expedited Forwarding), though the exact policy depends on the design.

**Voice VLAN verification commands**

```bash
Switch# show interfaces fastEthernet0/10 switchport
Switch# show vlan brief
Switch# show cdp neighbors detail
Switch# show lldp neighbors detail
Switch# show power inline
```

These commands help you confirm:

- The access VLAN and voice VLAN on the port
- Whether the phone is detected
- Whether the switch is supplying power through PoE (Power over Ethernet)
- Whether the port is operating as expected

**Native VLAN vs. voice VLAN**

These two concepts are related to VLAN handling but serve different purposes.

| Concept | Purpose | Typical port type | Tagging behavior |
|---|---|---|---|
| Native VLAN | Carries untagged frames on an 802.1Q trunk | Trunk | Untagged on the trunk |
| Voice VLAN | Separates IP phone traffic from data traffic | Access port with voice VLAN | Voice traffic is usually tagged by the phone |

Important distinctions:

- The native VLAN is a trunk concept.
- The voice VLAN is an access-port concept used for phones.
- The native VLAN handles untagged trunk traffic.
- The voice VLAN handles tagged phone traffic on an access port.

**Common design patterns**

1. **Access switch to distribution switch trunk**
   - Use a trunk between switches.
   - Set a dedicated native VLAN.
   - Allow only required VLANs.

2. **Phone plus PC on one access port**
   - Set the access VLAN for the PC.
   - Set the voice VLAN for the phone.
   - Enable PortFast on the edge port.

3. **Unused native VLAN for security**
   - Create a dedicated native VLAN.
   - Do not assign it to user ports.
   - Keep it consistent across the trunk.

**Troubleshooting checklist**

If a trunk or phone does not behave as expected, check the following:

- The trunk is actually in trunk mode.
- The native VLAN matches on both ends of the trunk.
- The allowed VLAN list includes the needed VLANs.
- The access VLAN and voice VLAN are correct on the phone port.
- CDP or LLDP is enabled if the phone depends on discovery.
- PoE is available if the phone needs power.
- Spanning Tree Protocol is not blocking the edge port unexpectedly.

**Example troubleshooting workflow**

1. Confirm the trunk and native VLAN.
   ```bash
   Switch# show interfaces trunk
   ```
2. Confirm the port VLAN settings.
   ```bash
   Switch# show interfaces gigabitEthernet0/1 switchport
   ```
3. Confirm the phone is discovered.
   ```bash
   Switch# show cdp neighbors detail
   ```
4. Confirm the phone port is powered.
   ```bash
   Switch# show power inline
   ```

**Security and operational considerations**

A poorly planned native VLAN can become a security and stability risk. If the native VLAN is left at VLAN 1 and also used broadly, untagged traffic may be harder to control. A dedicated unused native VLAN reduces accidental exposure.

For voice VLANs, keep in mind:

- Voice devices often need DHCP, call control, and QoS support.
- The phone may require access to a voice services subnet or call manager.
- The PC behind the phone should remain isolated in the data VLAN.
- If the phone is not detected, the port may still pass data traffic, but voice service will fail.

**Quick configuration reference**

```bash
! Trunk with dedicated native VLAN
vlan 999
 name NATIVE-BLACKHOLE

interface gigabitEthernet0/1
 switchport mode trunk
 switchport trunk native vlan 999
 switchport trunk allowed vlan 10,20,30,999

! Access port with voice VLAN
vlan 10
 name DATA
vlan 20
 name VOICE

interface fastEthernet0/10
 switchport mode access
 switchport access vlan 10
 switchport voice vlan 20
 spanning-tree portfast
```

**Exam focus**

For CCNA-level questions, be ready to identify:

- The native VLAN as the untagged VLAN on an 802.1Q trunk
- The voice VLAN as the VLAN used by IP phones on an access port
- The importance of matching native VLANs on both ends of a trunk
- The use of `switchport voice vlan` for phone deployments
- The role of `show interfaces trunk` and `show interfaces switchport` in verification

**Key takeaways**

- The **native VLAN** carries untagged traffic on an 802.1Q trunk.
- The **voice VLAN** separates IP phone traffic from data traffic on an access port.
- Native VLAN mismatches can cause serious Layer 2 problems.
- Voice VLANs are commonly paired with QoS and PoE for IP telephony.
- Verification commands are essential for confirming trunk and phone-port behavior.
