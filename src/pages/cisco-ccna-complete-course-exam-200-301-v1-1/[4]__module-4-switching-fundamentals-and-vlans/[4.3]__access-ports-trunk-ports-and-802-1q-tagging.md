## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**VLAN port roles and frame handling**

Switch ports carry traffic in one of two main ways: as an **access port** for a single VLAN, or as a **trunk port** for multiple VLANs. The exam often tests not just the definitions, but how frames actually move across the link and how to verify the configuration.

An access port belongs to one VLAN for user traffic. A trunk port carries traffic for multiple VLANs between switches, between a switch and a router, or between a switch and a server that understands VLAN tagging. On Cisco switches, trunking is commonly done with **802.1Q**.

- **Access port**: one data VLAN for end devices that do not tag frames.
- **Trunk port**: multiple VLANs over one physical link.
- **802.1Q**: the standard VLAN tagging method used on modern Cisco trunks.
- **Native VLAN**: the VLAN whose frames are sent untagged on the trunk.

**Access ports and default behavior**

Access ports are used for PCs, printers, and other devices that expect plain Ethernet frames. The switch places untagged incoming frames into the configured access VLAN and sends frames out without VLAN tags.

On Cisco switches, many interfaces default to a non-trunking state, but the exact default can vary by platform and interface type. In practice, you should not assume a port is ready for production until you verify its mode and VLAN assignment. Some ports may start as access ports, while others may participate in Dynamic Trunking Protocol (DTP) negotiation depending on the switch model and software.

Typical access-port configuration:

```bash
conf t
interface gigabitEthernet0/1
 switchport mode access
 switchport access vlan 10
 spanning-tree portfast
```

- `switchport mode access` forces the interface to behave as an access port.
- `switchport access vlan 10` assigns the user VLAN.
- `spanning-tree portfast` is commonly used on edge ports to speed up forwarding for end devices.

> Access ports do not tag user frames with 802.1Q headers.

**Trunk ports and why they matter**

A trunk is used when you need more than one VLAN to cross the same physical link. This is common between switches, and it is also used for router-on-a-stick designs where one router interface carries multiple VLANs using subinterfaces.

A trunk preserves VLAN separation by tagging frames so the receiving switch knows which VLAN each frame belongs to. Without tagging, the receiving device would not know how to keep traffic separated.

Common trunk use cases:

- Switch-to-switch uplinks
- Switch-to-router links for inter-VLAN routing
- Switch-to-firewall links
- Switch-to-hypervisor or server links that support VLAN tagging

Example trunk configuration:

```bash
conf t
interface gigabitEthernet0/24
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30
 switchport trunk native vlan 99
```

- `switchport mode trunk` forces trunking.
- `switchport trunk allowed vlan` limits which VLANs can traverse the link.
- `switchport trunk native vlan` sets the untagged VLAN on the trunk.

**802.1Q tagging basics**

**802.1Q** inserts a small tag into the Ethernet frame header. That tag identifies the VLAN number so the receiving switch can place the frame into the correct broadcast domain.

The important exam rule is simple:

- All VLANs on a trunk are tagged except the **native VLAN**.
- Native VLAN frames are sent **untagged on both ends** of the trunk.
- The native VLAN must match on both sides to avoid VLAN mismatch issues.

A simplified view:

| Frame type | Access port | Trunk port |
|---|---:|---:|
| User data frame | Untagged | Tagged, except native VLAN |
| Native VLAN frame | N/A | Untagged |
| Other VLAN frame | N/A | Tagged |

The 802.1Q tag includes VLAN identification information and a priority field used by QoS (Quality of Service). For CCNA, focus on the VLAN ID and the fact that the tag is added only on trunk links.

**Access VLAN versus voice VLAN**

A single switch port can support both a data device and an IP phone. In that design, the PC traffic and phone traffic are handled differently.

- The **access VLAN** carries untagged frames from the attached PC.
- The **voice VLAN** carries tagged frames from the IP phone.
- The phone often tags its own voice traffic with the voice VLAN ID, while the PC behind the phone remains untagged in the access VLAN.

Example configuration:

```bash
conf t
interface fastEthernet0/10
 switchport mode access
 switchport access vlan 10
 switchport voice vlan 20
 spanning-tree portfast
```

Practical behavior:

- The PC sends untagged frames into the access VLAN.
- The phone tags voice frames for the voice VLAN.
- The switch separates the two traffic types on the same physical port.

This is a common troubleshooting area because the port is still an access port for data, but it also supports a tagged voice VLAN for the phone.

**Dynamic Trunking Protocol and negotiation outcomes**

**DTP** (Dynamic Trunking Protocol) is Cisco’s trunk negotiation protocol. It can automatically negotiate whether a link becomes a trunk, depending on the mode configured on each side.

Common DTP modes:

- `access`: forces the port to stay non-trunking.
- `trunk`: forces the port to trunk.
- `dynamic desirable`: actively tries to form a trunk.
- `dynamic auto`: passively waits for the other side to initiate trunking.
- `nonegotiate`: disables DTP frames on the interface.

Common negotiation outcomes:

| Side A | Side B | Result |
|---|---|---|
| dynamic desirable | dynamic desirable | Trunk forms |
| dynamic desirable | dynamic auto | Trunk forms |
| dynamic auto | dynamic auto | No trunk |
| trunk | dynamic desirable | Trunk forms |
| trunk | dynamic auto | Trunk forms |
| trunk | access | No trunk |
| access | dynamic desirable | No trunk |
| access | access | Access link |

Key exam points:

- `dynamic desirable` actively negotiates trunking.
- `dynamic auto` does not initiate trunking.
- `access` prevents trunk formation.
- `trunk` forces trunking regardless of the other side, unless the other side is explicitly set to access.

Example of disabling negotiation on a forced trunk:

```bash
conf t
interface gigabitEthernet0/24
 switchport mode trunk
 switchport nonegotiate
```

Use `nonegotiate` when you want a static trunk and do not want DTP frames sent.

**Default switchport behavior and platform differences**

Cisco switch ports do not all behave identically by default across platforms and software versions. For CCNA-style troubleshooting, the safest approach is to verify the current state rather than assume it.

Important practical notes:

- Many Catalyst access ports default to a non-trunking state.
- Some interfaces may default to `dynamic auto`, which means they wait for the other side to initiate trunking.
- Some platforms or interface types may not support DTP in the same way.
- You should check the running configuration and operational state before making assumptions.

Useful verification commands:

```bash
show interfaces switchport
show interfaces trunk
show dtp interface gigabitEthernet0/24
show running-config interface gigabitEthernet0/24
```

What to look for:

- Administrative mode: what you configured
- Operational mode: what the port is actually doing
- Access VLAN and native VLAN values
- Allowed VLAN list on trunks
- Whether DTP is enabled or suppressed

**Legacy trunk encapsulation and compatibility**

Older Cisco switches could support more than one trunk encapsulation method, especially **ISL** (Inter-Switch Link) and **802.1Q**. Modern Cisco access-layer gear typically uses 802.1Q, and many platforms do not support ISL at all.

For legacy-style exam scenarios:

- Verify whether the interface supports trunk encapsulation selection.
- Ensure both ends use a compatible encapsulation.
- Remember that if one side does not support ISL, the trunk cannot use ISL.
- On many modern switches, you may not see an encapsulation choice because only 802.1Q is available.

Example on older platforms:

```bash
conf t
interface gigabitEthernet0/1
 switchport trunk encapsulation dot1q
 switchport mode trunk
```

> If the platform does not support ISL, you cannot configure it there. In that case, use 802.1Q on both ends.

**Verification and troubleshooting workflow**

When a VLAN link does not behave as expected, verify the physical link, the switchport mode, and the VLAN settings in that order. Many issues come from a mismatch between the two ends of the link.

A practical troubleshooting sequence:

1. Check link status and interface errors.
2. Verify the port mode on both sides.
3. Confirm the access VLAN or trunk VLAN list.
4. Confirm the native VLAN matches on both ends.
5. Check whether DTP is causing or preventing trunk formation.
6. Verify that the VLAN exists in the VLAN database.

Useful commands:

```bash
show interfaces status
show interfaces switchport
show interfaces trunk
show vlan brief
show cdp neighbors detail
```

Common problems:

- One side is `access` and the other side expects a trunk.
- The native VLAN differs between switches.
- The allowed VLAN list omits a required VLAN.
- The VLAN does not exist on the switch.
- DTP negotiation fails because both sides are `dynamic auto`.
- A legacy trunk encapsulation mismatch prevents trunk formation.

**Router-on-a-stick context**

A trunk is often used between a switch and a router when one physical router interface must route for multiple VLANs. The switch port is a trunk, and the router uses subinterfaces with 802.1Q encapsulation.

Example switch configuration:

```bash
conf t
interface gigabitEthernet0/1
 switchport mode trunk
 switchport trunk allowed vlan 10,20
```

Example router configuration:

```bash
conf t
interface gigabitEthernet0/0
 no shutdown

interface gigabitEthernet0/0.10
 encapsulation dot1Q 10
 ip address 192.168.10.1 255.255.255.0

interface gigabitEthernet0/0.20
 encapsulation dot1Q 20
 ip address 192.168.20.1 255.255.255.0
```

- The router subinterface uses `encapsulation dot1Q <vlan-id>`.
- The switch trunk carries the VLANs to the router.
- The native VLAN, if used, must be handled consistently on both devices.

**Exam-focused summary**

- **Access ports** carry one VLAN and send frames untagged.
- **Trunk ports** carry multiple VLANs and tag frames with **802.1Q**.
- The **native VLAN** is untagged on the trunk; all other VLANs are tagged.
- A port can carry both data and voice traffic, but the PC uses the access VLAN while the phone uses the voice VLAN.
- **DTP** can negotiate trunks, and the most important outcomes are `dynamic desirable` with `dynamic auto` forming a trunk, while `dynamic auto` with `dynamic auto` does not.
- On older platforms, verify trunk encapsulation compatibility and remember that some devices do not support ISL.
