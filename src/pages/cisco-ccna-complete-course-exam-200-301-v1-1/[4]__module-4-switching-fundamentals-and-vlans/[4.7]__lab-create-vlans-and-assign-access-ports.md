## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Lab overview**

This lab focuses on two core Layer 2 switching tasks: creating <abbr title="Virtual Local Area Network">VLAN</abbr>s and assigning switch interfaces as access ports. VLANs let you segment a switch into separate broadcast domains, which improves traffic isolation, security, and manageability. Access ports carry traffic for one VLAN only, making them the normal choice for end devices such as PCs, printers, and IP phones when the phone is not using voice VLAN configuration.

In CCNA-level work, you should be able to create VLANs, place ports into the correct VLAN, and verify the result with show commands. You should also understand the difference between the VLAN database and interface configuration, because a VLAN can exist on the switch but still not be active on any port.

**VLAN fundamentals**

A VLAN is a logical grouping of devices on the same Layer 2 network. Devices in different VLANs do not communicate directly at Layer 2; they need a Layer 3 device such as a router or multilayer switch for inter-VLAN routing.

- **Broadcast containment** reduces unnecessary traffic on each segment.
- **Security separation** helps isolate departments, labs, or guest devices.
- **Operational flexibility** lets you move users without rewiring the physical topology.
- **Scalability** improves network design by dividing a large switch into smaller logical networks.

A switch port in access mode belongs to one VLAN for untagged traffic. End devices typically send and receive untagged Ethernet frames, so the switch maps that traffic to the configured access VLAN.

**Common lab topology**

A simple lab usually includes one switch and two or more PCs connected to access ports. Each PC is placed into a different VLAN to demonstrate separation.

```text
PC1 --- Fa0/1      VLAN 10
PC2 --- Fa0/2      VLAN 20
PC3 --- Fa0/3      VLAN 10
```

If the PCs are in different VLANs and there is no router or Layer 3 switch configured for inter-VLAN routing, they should not be able to ping each other across VLAN boundaries.

**Create VLANs**

You create VLANs in global configuration mode. On Cisco IOS, the basic syntax is:

```bash
Switch# configure terminal
Switch(config)# vlan 10
Switch(config-vlan)# name SALES
Switch(config-vlan)# exit
Switch(config)# vlan 20
Switch(config-vlan)# name ENGINEERING
Switch(config-vlan)# end
```

Key points:

- **VLAN ID** values are typically in the range 1 to 4094, though some are reserved or used differently depending on platform and feature set.
- **Names** are optional but strongly recommended because they make verification easier.
- **VLAN 1** exists by default on most Cisco switches and is the default access VLAN unless changed.
- **Extended-range VLANs** may have platform-specific behavior; in many CCNA labs, you work with normal-range VLANs.

You can create multiple VLANs quickly:

```bash
Switch(config)# vlan 10,20,30
Switch(config-vlan)# name USERS
```

However, naming multiple VLANs in one command is not always as clear as creating them individually, so many labs prefer one VLAN at a time for readability.

**Assign access ports**

After the VLAN exists, assign the interface to that VLAN and set the port mode to access.

```bash
Switch# configure terminal
Switch(config)# interface fastethernet 0/1
Switch(config-if)# switchport mode access
Switch(config-if)# switchport access vlan 10
Switch(config-if)# exit
Switch(config)# interface fastethernet 0/2
Switch(config-if)# switchport mode access
Switch(config-if)# switchport access vlan 20
Switch(config-if)# end
```

Why the order matters:

- `switchport mode access` forces the interface to behave as an access port.
- `switchport access vlan <id>` places untagged frames into the selected VLAN.
- If the VLAN does not exist yet, the switch may reject the assignment or place the port into an unusable state depending on platform behavior.

A practical habit is to create the VLAN first, then assign the port, then verify.

**Verification commands**

Verification is essential in switch labs because a configuration can look correct while still failing due to a typo, wrong interface, or missing VLAN.

```bash
Switch# show vlan brief
Switch# show interfaces fastethernet 0/1 switchport
Switch# show running-config interface fastethernet 0/1
Switch# show interfaces status
```

What to look for:

- `show vlan brief` confirms the VLAN exists and lists member ports.
- `show interfaces ... switchport` confirms administrative and operational mode.
- `show running-config interface ...` shows the exact interface configuration.
- `show interfaces status` often displays the VLAN column for access ports.

Example `show vlan brief` output pattern:

```text
VLAN Name                             Status    Ports
---- -------------------------------- --------- -------------------------------
1    default                          active    Fa0/4, Fa0/5
10   SALES                            active    Fa0/1, Fa0/3
20   ENGINEERING                     active    Fa0/2
```

**Step-by-step lab procedure**

1. Enter privileged EXEC mode and global configuration mode.
2. Create the required VLANs.
3. Optionally name each VLAN for clarity.
4. Configure each target interface as an access port.
5. Assign the access VLAN to each interface.
6. Verify VLAN membership and interface status.
7. Test connectivity within the same VLAN and confirm isolation between VLANs.

**Example complete configuration**

```bash
Switch# configure terminal
Switch(config)# vlan 10
Switch(config-vlan)# name SALES
Switch(config-vlan)# exit
Switch(config)# vlan 20
Switch(config-vlan)# name ENGINEERING
Switch(config-vlan)# exit

Switch(config)# interface range fastethernet 0/1 - 2
Switch(config-if-range)# switchport mode access
Switch(config-if-range)# switchport access vlan 10
Switch(config-if-range)# exit

Switch(config)# interface fastethernet 0/3
Switch(config-if)# switchport mode access
Switch(config-if)# switchport access vlan 20
Switch(config-if)# end
```

This example places Fa0/1 and Fa0/2 into VLAN 10 and Fa0/3 into VLAN 20. Using `interface range` saves time when multiple ports need the same configuration.

**Interface range usage**

`interface range` is useful when you are assigning many adjacent ports to the same VLAN.

```bash
Switch(config)# interface range fa0/10 - 12
Switch(config-if-range)# switchport mode access
Switch(config-if-range)# switchport access vlan 30
```

Use ranges carefully:

- Make sure the selected ports are physically correct.
- Avoid accidentally changing uplinks or trunk ports.
- Verify the range after applying the configuration.

**Troubleshooting common issues**

> If a port does not appear in the expected VLAN, check both the VLAN definition and the interface mode. A port in dynamic or trunk mode will not behave like a normal access port.

Common problems include:

- **Missing VLAN**: the VLAN was never created, so the port cannot join it.
- **Wrong interface**: you configured the wrong FastEthernet or GigabitEthernet port.
- **Trunk mode left in place**: the interface is still operating as a trunk or dynamic port.
- **Administrative shutdown**: the port is down because it is disabled.
- **Cable or endpoint issue**: the switch is correct, but the connected device is off or disconnected.

Useful checks:

```bash
Switch# show interfaces status
Switch# show interfaces fa0/1 switchport
Switch# show vlan brief
Switch# show running-config | section interface FastEthernet0/1
```

If needed, restore a port to a clean access configuration:

```bash
Switch(config)# interface fa0/1
Switch(config-if)# default interface fa0/1
Switch(config)# interface fa0/1
Switch(config-if)# switchport mode access
Switch(config-if)# switchport access vlan 10
```

`default interface` is a fast way to remove prior settings before reapplying the intended configuration.

**Connectivity testing**

Once VLANs and access ports are configured, test hosts in the same VLAN first. Devices in the same VLAN should communicate if IP addressing is correct and no host firewall blocks traffic.

- Ping between hosts in the same VLAN to confirm Layer 2 reachability plus Layer 3 addressing.
- Ping between hosts in different VLANs to confirm isolation when no routing exists.
- Use `arp -a` on hosts to inspect learned neighbors after successful communication.

Example expectation:

- PC1 in VLAN 10 can ping PC3 in VLAN 10.
- PC1 in VLAN 10 cannot ping PC2 in VLAN 20 unless inter-VLAN routing is configured.

**Practical design notes**

VLANs are not just a lab exercise; they are a standard enterprise design tool. You often use them to separate:

- User departments
- Voice traffic
- Guest access
- Management interfaces
- Server segments
- Lab or test environments

Access ports are the default choice for endpoints because they keep the frame handling simple. Trunks are used between switches or to routers when multiple VLANs must traverse the same physical link.

**Quick reference**

| Task | Command |
|---|---|
| Create VLAN | `vlan 10` |
| Name VLAN | `name SALES` |
| Set access mode | `switchport mode access` |
| Assign VLAN to port | `switchport access vlan 10` |
| Verify VLANs | `show vlan brief` |
| Verify port mode | `show interfaces fa0/1 switchport` |

**Key takeaways**

- VLANs create separate broadcast domains on the same physical switch.
- Access ports carry traffic for one VLAN and are used for end devices.
- Create the VLAN before assigning it to a port.
- Verify with `show vlan brief` and interface-specific show commands.
- If a port behaves unexpectedly, check mode, VLAN membership, and prior configuration.
