## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Switching fundamentals**

A Layer 2 switch forwards Ethernet frames using <abbr title="Media Access Control">MAC</abbr> addresses. It builds a forwarding database by observing source addresses on incoming frames, then uses that database to decide where to send later traffic. In CCNA terms, the switch performs three core actions on every ingress frame: it learns the source MAC, looks up the destination MAC, and then forwards, floods, or filters the frame based on the lookup result.

A switch is not “routing” between IP networks here; it is making a Layer 2 decision inside a broadcast domain. That distinction matters because the switch does not inspect the IP header to make its basic forwarding choice. It uses the destination MAC address in the Ethernet header.

- **Ingress frame**: the frame arrives on a switch port and becomes the input to the forwarding decision.
- **Source MAC learning**: the switch records the source MAC and the ingress port in its MAC address table.
- **Destination MAC lookup**: the switch checks whether the destination MAC is already known.
- **Forwarding action**: the switch sends, floods, or drops the frame depending on the lookup and frame type.

**MAC address table, CAM table, and forwarding table**

These terms are often used interchangeably in CCNA contexts, but they describe related ideas:

- **MAC address table**: the operational table you inspect with commands such as `show mac address-table`.
- **CAM table**: <abbr title="Content Addressable Memory">CAM</abbr>, the hardware memory used for fast lookup in many switches.
- **Forwarding table**: a general term for the table the switch uses to decide where to send frames.

In practice, the switch stores learned MAC-to-port mappings in hardware so it can forward at line rate. The table entry usually includes the MAC address, VLAN, associated port, and entry type such as dynamic or static.

**Switch forwarding decision process**

The classic CCNA forwarding logic is:

1. A frame enters a switch port.
2. The switch reads the source MAC address.
3. The switch learns or refreshes the source MAC on that ingress port for the frame’s VLAN.
4. The switch checks the destination MAC address in the MAC table.
5. The switch applies one of the forwarding behaviors below.

- **Known unicast**: the destination MAC is in the table.
  - The switch forwards the frame only out the mapped port.
  - If the destination port is the same as the ingress port, the switch filters the frame and does not send it back out that same port.
- **Unknown unicast**: the destination MAC is not in the table.
  - The switch floods the frame out all other ports in the same VLAN except the ingress port.
  - When the destination host replies, the switch learns that host’s MAC from the source address of the return frame.
- **Broadcast**: destination MAC is `ff:ff:ff:ff:ff:ff`.
  - The switch floods the frame out all other ports in the same VLAN except the ingress port.
- **Multicast**: destination MAC is a multicast address.
  - Basic Layer 2 switches typically flood multicast frames unless they have multicast control features such as <abbr title="Internet Group Management Protocol">IGMP</abbr> snooping or similar mechanisms enabled.
  - With IGMP snooping, the switch can learn which ports want a given multicast group and forward more selectively.

> A switch never sends a frame back out the same port it arrived on for normal Layer 2 forwarding decisions.

**Learning behavior and table updates**

The switch learns from source addresses, not destination addresses. This is a common exam point. If host A sends traffic to host B, the switch learns host A’s MAC on A’s port immediately, even if host B is unknown.

A learned entry is usually tied to a VLAN because MAC addresses are significant within a broadcast domain. The same MAC on a different VLAN is treated as a different forwarding context.

Typical learned entry fields include:

- MAC address
- VLAN ID
- Port
- Type, such as dynamic, static, or secure
- Age timer state

Dynamic entries age out after inactivity. The aging timer prevents stale mappings from remaining forever when devices move or disconnect.

**Aging, inspection, and troubleshooting**

You should know how to inspect and manage the MAC table during troubleshooting.

Common verification commands on Cisco IOS include:

```bash
show mac address-table
show mac address-table dynamic
show mac address-table interface gigabitEthernet1/0/1
show mac address-table vlan 10
show mac address-table aging-time
```

Useful troubleshooting actions include:

```bash
clear mac address-table dynamic
clear mac address-table dynamic interface gigabitEthernet1/0/1
clear mac address-table dynamic vlan 10
```

- `show mac address-table aging-time` helps you verify how long dynamic entries remain before removal.
- Clearing the dynamic table is useful when you want to force the switch to relearn current host locations after a move, outage, or cabling change.
- If a host is silent, its entry may age out and the next frame may be flooded until learning happens again.

**Frame errors and switching method**

Switches can use different internal forwarding methods. The two most important are **store-and-forward** and **cut-through**.

| Method | How it works | Error handling | Typical use |
|---|---|---|---|
| **Store-and-forward** | Receives the entire frame before forwarding | Checks the frame for errors using the FCS (Frame Check Sequence) and discards bad frames | Common default behavior on modern switches |
| **Cut-through** | Starts forwarding after reading the destination MAC address | May forward frames before the full frame is received, so it can pass errored frames | Lower latency, but less error checking |

Store-and-forward is the safer and more common model for exam prep because the switch can verify the FCS and discard frames with CRC (Cyclic Redundancy Check) errors. If the frame is corrupted, the switch does not forward it. This prevents bad frames from spreading through the network.

Cut-through reduces latency, but because it begins forwarding before the full frame is received, it may not detect errors before transmission begins. Some switches use hybrid behavior, but for CCNA you should understand the contrast clearly.

**End-to-end learning scenario**

Consider three hosts on VLAN 10:

- Host A: `00:11:22:33:44:55` on port `Gi1/0/1`
- Host B: `00:11:22:33:44:66` on port `Gi1/0/2`
- Host C: `00:11:22:33:44:77` on port `Gi1/0/3`

At the start, the MAC table is empty.

1. Host A sends a frame to Host B.
   - The switch receives the frame on `Gi1/0/1`.
   - It learns `00:11:22:33:44:55` on `Gi1/0/1` in VLAN 10.
   - It looks up destination `00:11:22:33:44:66` and does not find it.
   - The switch floods the frame out `Gi1/0/2` and `Gi1/0/3`, but not `Gi1/0/1`.

2. Host B receives the frame and replies to Host A.
   - The switch receives the reply on `Gi1/0/2`.
   - It learns `00:11:22:33:44:66` on `Gi1/0/2` in VLAN 10.
   - It looks up destination `00:11:22:33:44:55` and finds it on `Gi1/0/1`.
   - The switch forwards the reply only to `Gi1/0/1`.

3. Host C sends traffic to Host A.
   - The switch learns `00:11:22:33:44:77` on `Gi1/0/3`.
   - It already knows Host A on `Gi1/0/1`.
   - The frame is forwarded only to `Gi1/0/1`.

After these exchanges, the MAC table contains three dynamic entries, one for each host, and future traffic is forwarded efficiently without flooding.

```bash
Switch# show mac address-table dynamic

          Mac Address Table
-------------------------------------------

Vlan    Mac Address       Type        Ports
----    -----------       --------    -----
  10    0011.2233.4455    DYNAMIC     Gi1/0/1
  10    0011.2233.4466    DYNAMIC     Gi1/0/2
  10    0011.2233.4477    DYNAMIC     Gi1/0/3
```

**Edge cases you should know**

- If the destination MAC is on the same ingress port, the switch filters the frame instead of forwarding it back out.
- If the destination MAC is unknown, the switch floods only within the same VLAN, not across all VLANs.
- If the destination is a broadcast address, the switch floods to all ports in the VLAN except the ingress port.
- If the destination is multicast and no snooping or group awareness is configured, the switch usually floods it like an unknown Layer 2 destination.
- If the frame has a bad FCS and the switch uses store-and-forward, the frame is discarded.

**Practical verification workflow**

Use a simple workflow when troubleshooting MAC learning:

1. Confirm the host is connected to the expected port.
2. Check whether the switch has learned the source MAC.
3. Verify the VLAN assignment on the port.
4. Inspect the aging timer if the entry disappears unexpectedly.
5. Clear the dynamic table if you need to force relearning.
6. Generate fresh traffic and recheck the table.

Example commands:

```bash
show interfaces status
show vlan brief
show mac address-table interface gi1/0/1
show mac address-table vlan 10
show mac address-table aging-time
clear mac address-table dynamic
```

**Why this matters for CCNA**

Switch operation and MAC learning explain why Layer 2 networks are efficient after the first frame exchange. The first packet may be flooded, but the return traffic usually becomes unicast once the switch learns both endpoints. This behavior reduces unnecessary traffic and is the foundation for understanding VLANs, broadcast domains, and basic switching troubleshooting.
