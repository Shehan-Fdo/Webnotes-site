## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Troubleshooting mindset**

Troubleshooting is a process of narrowing the problem from the symptom to the exact fault. In CCNA scenarios, you usually start with the OSI layer, then verify interface state, addressing, VLANs, routing, and policy controls such as ACLs or NAT.

A practical workflow is:

1. Identify the symptom and scope.
2. Check physical and data-link status.
3. Verify Layer 3 addressing and reachability.
4. Confirm routing, filtering, and translation.
5. Compare the running configuration to the intended design.

Useful baseline commands include:

```bash
show ip interface brief
show interfaces
show running-config
show cdp neighbors detail
show lldp neighbors detail
ping
traceroute
```

**Layer 1 and Layer 2 checks**

Many CCNA problems begin with a simple interface issue. A port can be administratively down, physically down, err-disabled, or up but still misconfigured.

- **Administrative shutdown** means the interface is disabled with `shutdown`.
- **Line protocol down** often points to encapsulation, keepalive, or Layer 2 issues.
- **Err-disabled** usually indicates a protection feature triggered, such as port security, BPDU Guard, or link-flap detection.
- **Speed and duplex mismatches** can cause late collisions, CRC errors, and poor throughput on legacy or manually configured links.

Use these commands to inspect interface health:

```bash
show ip interface brief
show interfaces gigabitEthernet0/1
show interfaces status
show interfaces counters errors
```

`show interfaces status` is useful for a quick view of port state, VLAN, duplex, speed, and type. For deeper fault isolation, `show interfaces` and error counters are more reliable because they expose CRCs, input errors, runts, giants, and other indicators of physical or duplex problems.

Common Layer 1/2 symptoms include:

- No link light or `administratively down`
- `up/down` or `down/down` interface states
- High CRC, input error, or runts counters
- One-way connectivity caused by cabling, optics, or speed/duplex mismatch
- Wrong access VLAN or trunk mismatch

**Switchport and VLAN misconfigurations**

A large number of campus issues come from incorrect switchport mode or VLAN assignment. If a host cannot reach its gateway, verify the access port, VLAN membership, and trunk path.

Typical mistakes include:

- Access port assigned to the wrong VLAN
- Trunk not allowing the required VLAN
- Native VLAN mismatch on a trunk
- Dynamic trunking not forming as expected
- Voice VLAN missing on a phone port
- Port security blocking a legitimate device

Verification commands:

```bash
show vlan brief
show interfaces switchport
show interfaces trunk
show mac address-table
show spanning-tree vlan 10
```

A useful mental model is:

- If the host cannot talk to devices in the same VLAN, check the access port and VLAN membership.
- If the host can talk locally but not across the network, check the trunk path and default gateway.
- If only one VLAN fails across a trunk, check allowed VLANs and native VLAN consistency.

**Port security troubleshooting**

Port security is a common source of `err-disabled` ports in CCNA labs and real networks. It limits which MAC addresses can use a switchport.

Common misconfigurations include:

- Too many MAC addresses learned for the configured limit
- A device replacement causing a new MAC address violation
- Sticky MAC configured, but the saved configuration does not match the current device
- Violation mode set to `shutdown`, which disables the port after a violation
- Port security enabled on the wrong interface type or with the wrong access mode

Example configuration:

```bash
interface gigabitEthernet0/10
 switchport mode access
 switchport access vlan 20
 switchport port-security
 switchport port-security maximum 1
 switchport port-security mac-address sticky
 switchport port-security violation restrict
```

Verification and recovery:

```bash
show port-security interface gigabitEthernet0/10
show port-security address
show interfaces status err-disabled
```

If the port is err-disabled, you often need to correct the cause and then recover the interface:

```bash
interface gigabitEthernet0/10
 shutdown
 no shutdown
```

> If the violation mode is `shutdown`, the port stays down until you clear the cause and manually or automatically recover it.

**Spanning Tree Protocol troubleshooting**

<abbr title="Spanning Tree Protocol">STP</abbr> prevents Layer 2 loops, but a misconfiguration can block the wrong port or create instability. In classic 802.1D troubleshooting, the port states are blocking, listening, learning, and forwarding. `disabled` is an interface/admin state, not an STP state.

Common STP issues include:

- Unexpected root bridge election
- Wrong port blocked because of bridge priority or path cost
- PortFast missing on access ports
- BPDU Guard shutting down an edge port after receiving a BPDU
- Root Guard preventing an access switch from becoming root
- Loop caused by redundant links without EtherChannel or STP protection

Useful commands:

```bash
show spanning-tree
show spanning-tree vlan 10
show spanning-tree interface gigabitEthernet0/1 detail
show spanning-tree root
```

Practical interpretation:

- If a port is blocking unexpectedly, compare bridge priority and path cost.
- If an access port goes err-disabled, check for BPDU Guard.
- If the topology changes frequently, look for loops, flapping links, or miswired access switches.

**EtherChannel and trunk aggregation issues**

EtherChannel bundles multiple physical links into one logical link. If the member links do not match, the bundle may fail or only partially form.

Common misconfigurations include:

- Mismatched speed or duplex
- Different switchport mode on member links
- VLAN or trunk parameter mismatch
- LACP/PAgP mode mismatch
- One side configured as access and the other as trunk
- Native VLAN mismatch on a trunk EtherChannel

Verification commands:

```bash
show etherchannel summary
show interfaces port-channel 1
show lacp neighbors
show pagp neighbor
show interfaces trunk
```

A healthy bundle should show member ports in the correct state and the port-channel should carry the intended VLANs. If the bundle does not form, compare both sides line by line.

**IPv4 addressing and default gateway problems**

A host may have link but still fail to reach remote networks because of incorrect IP settings. In CCNA troubleshooting, always verify the host address, subnet mask, and default gateway.

Common mistakes include:

- Wrong subnet mask causing local/remote confusion
- Incorrect default gateway
- Duplicate IP address
- SVI or router interface in the wrong subnet
- Host placed in the wrong VLAN

Useful checks:

```bash
show ip interface brief
show running-config interface vlan 10
ping 192.168.10.1
traceroute 8.8.8.8
```

On the host side, verify:

- IP address
- Subnet mask
- Default gateway
- DNS server if name resolution fails

A duplicate IP often appears as intermittent connectivity, ARP instability, or a device that works only sometimes.

**Inter-VLAN routing misconfigurations**

Inter-VLAN routing fails when the Layer 3 gateway for a VLAN is missing, incorrect, or unreachable. The exact design matters: you may use a multilayer switch with SVIs or a router-on-a-stick design with subinterfaces.

Common SVI problems on multilayer switches:

- SVI not created
- SVI created but `shutdown`
- SVI IP address in the wrong subnet
- VLAN not active on the switch
- No Layer 2 access ports in that VLAN, so the SVI stays down on some platforms
- Missing `ip routing` on the multilayer switch

Example SVI configuration:

```bash
vlan 10
 name USERS

interface vlan 10
 ip address 192.168.10.1 255.255.255.0
 no shutdown

ip routing
```

Router-on-a-stick problems include:

- Missing subinterface
- Wrong encapsulation VLAN ID
- Physical interface not configured as a trunk on the switch side
- Native VLAN mismatch
- Wrong IP address on the subinterface

Example router-on-a-stick configuration:

```bash
interface gigabitEthernet0/0
 no shutdown

interface gigabitEthernet0/0.10
 encapsulation dot1Q 10
 ip address 192.168.10.1 255.255.255.0

interface gigabitEthernet0/0.20
 encapsulation dot1Q 20
 ip address 192.168.20.1 255.255.255.0
```

Verification commands:

```bash
show ip interface brief
show interfaces trunk
show running-config interface vlan 10
show running-config interface gigabitEthernet0/0.10
```

If a VLAN works locally but not across the network, check the gateway interface first.

**Routing table and dynamic routing issues**

Routing problems often come from missing advertisements, passive interfaces, or adjacency failures. The symptom is usually that directly connected networks work, but remote networks do not.

Common dynamic routing misconfigurations include:

- Missing `network` statement or prefix advertisement
- Wrong wildcard mask in a routing process
- `passive-interface` applied to the wrong interface
- Neighbor adjacency failure due to area, authentication, timer, or hello mismatch
- ACL blocking routing protocol traffic
- Summarization hiding a needed route
- Administrative distance causing the wrong route to win

Verification commands:

```bash
show ip route
show ip protocols
show ip ospf neighbor
show ip ospf interface brief
show ip eigrp neighbors
show ip rip database
```

Troubleshooting approach:

1. Confirm the route exists in the routing table.
2. Confirm the routing process is enabled on the correct interfaces.
3. Check neighbor adjacencies.
4. Verify that the route is being advertised and learned.
5. Compare metrics and administrative distance if multiple paths exist.

If a route is missing, ask whether the network is connected, advertised, filtered, or suppressed.

**ACL misconfigurations**

Access control lists <abbr title="Access Control Lists">ACLs</abbr> are a frequent cause of “network is up but traffic is blocked” problems. The most common issue is not the ACL itself, but where it is applied and in what order.

Common ACL mistakes include:

- Wrong interface or direction
- Standard ACL placed too close to the source or too far from the destination
- Extended ACL placed in the wrong location
- Incorrect wildcard mask
- Sequence/order causing an early deny to match first
- Missing permit statement for required traffic
- Implicit deny at the end of every ACL
- ACL applied to the wrong protocol or interface

Example extended ACL:

```bash
ip access-list extended WEB-FILTER
 permit tcp 192.168.10.0 0.0.0.255 host 203.0.113.10 eq 443
 deny ip any host 203.0.113.10
 permit ip any any

interface gigabitEthernet0/1
 ip access-group WEB-FILTER in
```

Verification commands:

```bash
show access-lists
show ip interface gigabitEthernet0/1
show running-config | section access-list
```

Look at hit counts to identify which line is matching traffic. If a deny line increments unexpectedly, the ACL order or wildcard is likely wrong.

**NAT and PAT troubleshooting**

<abbr title="Network Address Translation">NAT</abbr> and <abbr title="Port Address Translation">PAT</abbr> issues usually come from interface roles, ACL matching, or translation rules. The most common symptom is that inside hosts can reach local resources but not the Internet.

Common misconfigurations include:

- Inside and outside interfaces reversed
- ACL does not match the inside source addresses
- NAT overload not configured when many hosts need translation
- Wrong inside local subnet in the ACL
- Missing default route toward the ISP
- Translation table not building because traffic never matches

Example NAT overload configuration:

```bash
access-list 1 permit 192.168.10.0 0.0.0.255

interface gigabitEthernet0/0
 ip address 192.168.10.1 255.255.255.0
 ip nat inside

interface gigabitEthernet0/1
 ip address 203.0.113.2 255.255.255.252
 ip nat outside

ip nat inside source list 1 interface gigabitEthernet0/1 overload
ip route 0.0.0.0 0.0.0.0 203.0.113.1
```

Verification commands:

```bash
show ip nat translations
show ip nat statistics
show access-lists
show ip route
```

If translations do not appear, verify that traffic is actually passing through the router and that the ACL matches the source subnet.

**IPv6 troubleshooting**

IPv6 issues often involve address assignment, router advertisements, or default route discovery. A host with only a link-local address may be missing router advertisements, may not be receiving DHCPv6 information, or may not have a valid prefix configured on the router.

Common IPv6 problems include:

- Interface missing a global unicast address
- Wrong prefix length
- Router advertisements disabled or absent
- DHCPv6 server or relay misconfigured
- Default route missing on the router
- Neighbor discovery issues

Example interface configuration:

```bash
interface gigabitEthernet0/0
 ipv6 address 2001:db8:10:1::1/64
 ipv6 nd other-config-flag
 no shutdown
```

On Cisco IOS, assigning a global unicast address enables IPv6 on the interface automatically, so `ipv6 enable` is not required in that example.

Useful commands:

```bash
show ipv6 interface brief
show ipv6 route
show ipv6 neighbors
show running-config interface gigabitEthernet0/0
```

If a host has only a link-local address, check:

- Whether the router interface is up
- Whether the prefix is configured correctly
- Whether router advertisements are being sent
- Whether DHCPv6 is expected instead of SLAAC
- Whether the VLAN and trunk path are correct

**Wireless troubleshooting**

Wireless troubleshooting depends on the platform. In controller-based environments such as WLC or Catalyst 9800-style deployments, you often verify AP status, client association, and WLAN-to-VLAN mapping from the controller. These commands are not universal across all Cisco wireless platforms, so use them in the appropriate environment.

Common wireless issues include:

- AP cannot join the controller
- Wrong management IP, gateway, or DNS on the AP
- CAPWAP discovery failure
- SSID not mapped to the correct VLAN
- WPA2/WPA3 or PSK mismatch
- 802.1X or RADIUS authentication failure
- RF coverage or channel interference problems
- Client receives an IP address from the wrong scope

Controller-style verification examples:

```bash
show wlan summary
show ap summary
show ap join stats summary
show client summary
show wireless client mac-address 0011.2233.4455
```

Practical checks:

- Confirm the AP can reach the controller management address.
- Confirm the SSID is enabled and broadcast as intended.
- Confirm the WLAN maps to the correct interface or VLAN.
- Confirm authentication servers are reachable and configured correctly.
- Confirm the client receives an address from the expected DHCP scope.

**SSH and remote management issues**

Secure Shell <abbr title="Secure Shell">SSH</abbr> problems usually come from missing prerequisites rather than the SSH command itself. Modern Cisco devices should use SSH version 2.

Common misconfigurations include:

- Missing hostname or domain name
- No RSA keys generated
- `ip ssh version 2` not configured
- VTY lines not allowing SSH
- Local user missing or incorrect privilege
- Management interface not reachable

Example configuration:

```bash
hostname SW1
ip domain-name example.local
username admin privilege 15 secret Cisco123!
crypto key generate rsa modulus 2048
ip ssh version 2

line vty 0 4
 login local
 transport input ssh
```

Verification commands:

```bash
show ip ssh
show ssh
show running-config | section line\ vty
```

If SSH fails, verify reachability first, then authentication, then VTY transport settings.

**Scenario-based troubleshooting examples**

A structured scenario helps you isolate the fault faster.

1. A PC in VLAN 20 cannot reach the default gateway.
   - Check the access port VLAN assignment.
   - Verify the SVI or router subinterface for VLAN 20.
   - Confirm the gateway IP and mask.
   - Check whether the SVI is up/up.

2. A switchport goes err-disabled after a device is connected.
   - Check port security status.
   - Review the learned MAC address count.
   - Confirm the violation mode.
   - Clear the fault and recover the port.

3. Users can reach local resources but not the Internet.
   - Verify the default route.
   - Check NAT inside/outside roles.
   - Confirm the NAT ACL matches the inside subnet.
   - Inspect translation entries.

4. Two switches form a trunk, but one VLAN does not pass.
   - Check allowed VLAN lists.
   - Verify native VLAN consistency.
   - Confirm the VLAN exists on both switches.
   - Inspect MAC learning on the affected VLAN.

5. A routing protocol neighbor does not form.
   - Check interface IP addressing and masks.
   - Verify hello/dead timers, area, or authentication.
   - Confirm the interface is not passive.
   - Check for ACLs blocking protocol traffic.

**Exam-focused verification habits**

CCNA questions often ask you to identify the most likely cause from a small set of clues. Train yourself to map symptoms to likely faults.

- **No link** usually points to cabling, interface shutdown, or hardware.
- **Link up, no traffic** often points to VLAN, trunk, ACL, or gateway issues.
- **Intermittent loss** often points to duplex mismatch, loops, or duplicate IPs.
- **Local works, remote fails** often points to routing, NAT, or ACLs.
- **One VLAN fails** often points to trunk allowed VLANs or SVI/subinterface mismatch.

A good troubleshooting habit is to compare the working side with the broken side. In many CCNA labs, the fastest fix is to find the one line that differs from the intended configuration.
