## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Trunk link fundamentals**

A trunk is a switch port that carries traffic for multiple VLANs over a single physical link. In Cisco environments, trunking is most commonly implemented with IEEE 802.1Q, which inserts a VLAN tag into Ethernet frames so the receiving switch knows which VLAN the frame belongs to.

Use trunks when you need to extend multiple VLANs between switches, to a router-on-a-stick interface, or to a virtualization host carrying several segmented networks. Without a trunk, a link is typically an access link and carries traffic for only one VLAN.

- **802.1Q** is the standard trunking encapsulation used on modern Cisco switches.
- Trunks preserve VLAN separation across the Layer 2 domain.
- Native VLAN traffic is sent untagged on an 802.1Q trunk unless configured otherwise.
- Trunking is not the same as routing; it only extends Layer 2 VLAN membership.

**How 802.1Q tagging works**

When a frame enters a trunk, the switch adds a tag that identifies the VLAN. The receiving switch reads the tag, forwards the frame only within the correct VLAN, and removes the tag before sending the frame out an access port.

The trunk also has a **native VLAN**, which is the VLAN whose frames are transmitted without a tag. Both ends of the trunk should use the same native VLAN to avoid mismatches and potential security or interoperability issues.

> A native VLAN mismatch can cause unexpected traffic handling and is a common troubleshooting item in CCNA labs.

**Common trunk use cases**

- Switch-to-switch uplinks that must carry multiple VLANs.
- Switch-to-router links for inter-VLAN routing using router-on-a-stick.
- Switch-to-firewall links where multiple VLANs are passed to a security device.
- Switch-to-server links when the server NIC or hypervisor supports VLAN tagging.

**Basic trunk configuration on Cisco IOS**

You usually configure a trunk by setting the interface mode to trunk. On many Catalyst switches, the trunk uses 802.1Q by default, but older platforms may support multiple encapsulations.

```bash
enable
configure terminal
interface gigabitEthernet0/1
 switchport mode trunk
 switchport trunk native vlan 99
 switchport trunk allowed vlan 10,20,30,99
 no shutdown
end
```

This example does three important things:

- Forces the port into **trunk mode**.
- Sets VLAN 99 as the **native VLAN**.
- Limits the trunk to specific VLANs with an **allowed VLAN list**.

**Why restrict allowed VLANs**

By default, some switches allow all VLANs on a trunk. In production, you often restrict the trunk to only the VLANs that actually need to traverse the link. This reduces unnecessary flooding, improves clarity, and can help limit the impact of misconfigurations.

- Allow only required VLANs on the trunk.
- Keep the native VLAN consistent on both ends.
- Avoid using VLAN 1 as the native VLAN when possible.
- Document which VLANs should traverse each uplink.

**Matching trunk settings on both ends**

For a trunk to form correctly, both switches must agree on the trunking parameters. At minimum, the link must be configured to trunk on both sides, and the native VLAN should match.

Example of a two-switch topology:

```text
PC1 --- SW1 Fa0/10
PC2 --- SW2 Fa0/10

SW1 Gi0/1 ===== Gi0/1 SW2
        trunk carrying VLANs 10,20,30
```

A typical configuration might look like this:

```bash
! SW1
interface gigabitEthernet0/1
 switchport mode trunk
 switchport trunk native vlan 99
 switchport trunk allowed vlan 10,20,30,99

! SW2
interface gigabitEthernet0/1
 switchport mode trunk
 switchport trunk native vlan 99
 switchport trunk allowed vlan 10,20,30,99
```

**Verification commands**

After configuration, verify the trunk status and VLAN forwarding behavior. The most useful commands are `show interfaces trunk`, `show interfaces switchport`, and `show vlan brief`.

```bash
show interfaces trunk
show interfaces switchport
show vlan brief
show running-config interface gigabitEthernet0/1
```

What to look for:

- The interface is listed as an active trunk.
- The encapsulation is 802.1Q where applicable.
- The native VLAN matches on both ends.
- The allowed VLAN list includes the required VLANs.
- The interface is not administratively down.

**Example verification output clues**

- `Status: trunking` indicates the port is operating as a trunk.
- `Native vlan` should be the same on both switches.
- `Vlans allowed on trunk` should include the VLANs you expect.
- `Vlans in spanning tree forwarding state and not pruned` helps confirm forwarding behavior.

**Troubleshooting trunk problems**

Trunk issues usually come from a small set of causes. Start by checking the physical link, then confirm interface mode, native VLAN, allowed VLAN list, and VLAN existence.

Common problems include:

- One side is set to access mode while the other is set to trunk.
- The native VLAN differs between switches.
- The VLAN is missing from the allowed VLAN list.
- The VLAN does not exist in the VLAN database.
- The interface is shut down or err-disabled.
- DTP negotiation behavior does not match expectations on older gear.

> If a VLAN is not created on the switch, it may not forward correctly even if the trunk allows it.

Useful troubleshooting workflow:

1. Check link status with `show interfaces status`.
2. Confirm trunking with `show interfaces trunk`.
3. Verify interface settings with `show interfaces switchport`.
4. Confirm VLAN presence with `show vlan brief`.
5. Compare both ends of the trunk for mismatched native VLAN or allowed VLAN lists.

**Dynamic Trunking Protocol considerations**

Dynamic Trunking Protocol (DTP) is a Cisco protocol used to negotiate trunking on some interfaces. In modern designs and labs, you often disable negotiation and explicitly set trunk mode to avoid ambiguity.

```bash
interface gigabitEthernet0/1
 switchport mode trunk
 switchport nonegotiate
```

Use `switchport nonegotiate` when you want to prevent DTP frames from being sent, such as on links to non-Cisco devices or when you want deterministic trunk behavior.

**Access versus trunk comparison**

| Feature | Access Port | Trunk Port |
|---|---|---|
| VLANs carried | One VLAN | Multiple VLANs |
| Frame tagging | Untagged | Tagged with 802.1Q, except native VLAN |
| Typical use | End devices | Switches, routers, hypervisors |
| Configuration goal | Assign a single VLAN | Transport several VLANs |

**Practical lab checklist**

Before you finish the lab, confirm the following:

- The trunk is up and operational.
- The correct native VLAN is configured on both ends.
- The allowed VLAN list includes all required VLANs.
- End devices in different VLANs still remain isolated at Layer 2.
- Inter-VLAN communication, if needed, is handled by a router or Layer 3 switch.

**Example end-to-end lab sequence**

1. Create the required VLANs.
2. Assign access ports to the correct VLANs.
3. Configure the inter-switch link as a trunk.
4. Set the native VLAN consistently on both switches.
5. Restrict the allowed VLAN list to the required VLANs.
6. Verify trunk status and test connectivity between hosts in the same VLAN across switches.

```bash
vlan 10
 name SALES
vlan 20
 name ENGINEERING
vlan 30
 name HR
vlan 99
 name NATIVE

interface gigabitEthernet0/1
 switchport mode trunk
 switchport trunk native vlan 99
 switchport trunk allowed vlan 10,20,30,99
```

**Security and design notes**

Trunks expand the Layer 2 broadcast domain across switches, so you should design them carefully. Keep the native VLAN unused by end hosts when possible, and avoid leaving unnecessary VLANs on trunks.

- Use a dedicated native VLAN that is not used for user access.
- Prune unused VLANs from trunks.
- Keep trunk links between trusted devices only.
- Document trunk purpose, native VLAN, and allowed VLANs.

**Key takeaways**

A trunk link carries multiple VLANs across one physical connection using 802.1Q tagging. Correct trunk configuration depends on matching trunk mode, native VLAN, and allowed VLANs on both ends. Verification and troubleshooting are essential because most trunk issues are caused by simple mismatches rather than hardware failure.
