## **Course:** Cisco CCNA Complete Course (Exam 200-301 v1.1)

**Router-on-a-stick overview**

Router-on-a-stick is a legacy but still important inter-VLAN routing design where a single physical router interface carries traffic for multiple VLANs using 802.1Q trunking. You create one subinterface per VLAN, assign each subinterface an IP address in that VLAN’s subnet, and use the router as the default gateway for hosts in that VLAN.

This design is useful in labs and small networks because it is simple, inexpensive, and easy to understand. It also reinforces core CCNA concepts: VLANs, trunks, default gateways, and Layer 3 forwarding between broadcast domains.

- **VLAN separation** keeps user traffic logically isolated at Layer 2.
- **802.1Q trunking** tags frames so one link can carry multiple VLANs.
- **Subinterfaces** let one router port act like multiple logical interfaces.
- **Default gateway** addresses on the router provide inter-VLAN routing.

> Router-on-a-stick depends on the switch port facing the router being a trunk. If the trunk is misconfigured, hosts may still communicate within the same VLAN but will fail to reach other VLANs.

**Typical topology and traffic flow**

A common lab topology uses one router connected to one switch port. The switch access ports connect end devices in different VLANs, and the router interface is split into subinterfaces such as `G0/0.10`, `G0/0.20`, and `G0/0.30`.

Traffic flow works like this:

1. A host sends a frame to its default gateway.
2. The switch forwards the frame over the trunk, tagging it with the source VLAN.
3. The router receives the tagged frame on the matching subinterface.
4. The router routes the packet to the destination VLAN.
5. The router sends the return traffic back out the correct subinterface with the proper VLAN tag.

**Why you use subinterfaces**

Subinterfaces allow you to map one physical interface to multiple logical Layer 3 interfaces. Each subinterface can have its own encapsulation and IP address. This is the standard method for router-on-a-stick on Cisco IOS when you need one router interface to route between several VLANs.

- **Physical interface**: no IP address in most designs; it acts as the trunk carrier.
- **Subinterface**: logical interface tied to a VLAN.
- **Encapsulation**: identifies the VLAN tag used on that subinterface.

**Core configuration steps**

You usually configure the switch first, then the router, then verify end-to-end connectivity.

1. Create the VLANs on the switch.
2. Assign access ports to the correct VLANs.
3. Configure the switch port to the router as an 802.1Q trunk.
4. Create router subinterfaces.
5. Apply `encapsulation dot1Q <vlan-id>` on each subinterface.
6. Assign an IP address to each subinterface.
7. Set host default gateways to the router IP in each VLAN.
8. Verify trunking, interface status, and ping connectivity.

**Switch configuration example**

The switch needs VLANs, access ports, and a trunk toward the router.

```cisco
conf t
vlan 10
 name SALES
vlan 20
 name ENGINEERING
vlan 30
 name HR

interface range fa0/1 - 2
 switchport mode access
 switchport access vlan 10

interface range fa0/3 - 4
 switchport mode access
 switchport access vlan 20

interface fa0/5
 switchport mode access
 switchport access vlan 30

interface g0/1
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30
 no shutdown
end
```

Important switch-side points:

- **Access ports** belong to exactly one VLAN.
- **Trunk ports** carry multiple VLANs.
- **Allowed VLANs** should match the VLANs you intend to route.
- **Native VLAN** should be consistent on both ends if you use one.

**Router configuration example**

On the router, the physical interface stays up and the subinterfaces handle VLAN tagging.

```cisco
conf t
interface g0/0
 no shutdown

interface g0/0.10
 encapsulation dot1Q 10
 ip address 192.168.10.1 255.255.255.0

interface g0/0.20
 encapsulation dot1Q 20
 ip address 192.168.20.1 255.255.255.0

interface g0/0.30
 encapsulation dot1Q 30
 ip address 192.168.30.1 255.255.255.0
end
```

If you use a native VLAN on the trunk, configure it explicitly on the matching subinterface:

```cisco
interface g0/0.99
 encapsulation dot1Q 99 native
 ip address 192.168.99.1 255.255.255.0
```

**Addressing and gateway planning**

Each VLAN needs a unique subnet. Hosts in that VLAN use the router subinterface IP as their default gateway.

| VLAN | Purpose | Subnet | Router IP | Example host |
|---|---|---:|---:|---:|
| 10 | SALES | 192.168.10.0/24 | 192.168.10.1 | 192.168.10.10 |
| 20 | ENGINEERING | 192.168.20.0/24 | 192.168.20.1 | 192.168.20.10 |
| 30 | HR | 192.168.30.0/24 | 192.168.30.1 | 192.168.30.10 |

Use a consistent addressing scheme so you can troubleshoot quickly. A common pattern is to make the router `.1` address the gateway in every VLAN.

**Verification commands**

Verification is essential because router-on-a-stick failures often come from trunk mismatches, wrong VLAN IDs, or incorrect host gateways.

```cisco
show vlan brief
show interfaces trunk
show ip interface brief
show running-config interface g0/0
show running-config interface g0/0.10
show running-config interface g0/0.20
show running-config interface g0/0.30
```

Useful checks:

- `show vlan brief` confirms access port VLAN membership.
- `show interfaces trunk` confirms the router-facing port is trunking.
- `show ip interface brief` confirms subinterfaces are up/up.
- `show running-config interface ...` confirms encapsulation and IP settings.

**Ping and path validation**

After configuration, test from a host in one VLAN to the gateway in the same VLAN, then to a host in another VLAN.

```bash
ping 192.168.10.1
ping 192.168.20.10
tracert 192.168.20.10
```

Expected behavior:

- A host should ping its default gateway successfully.
- Inter-VLAN pings should succeed if routing and trunking are correct.
- A failed ping to the gateway usually indicates VLAN, trunk, or IP mismatch.
- A failed ping to another VLAN with gateway reachability often indicates routing or host gateway issues.

**Common mistakes and how to fix them**

- **Wrong encapsulation VLAN ID**: the subinterface VLAN must match the switch VLAN.
- **Missing `no shutdown`**: the physical router interface must be enabled.
- **Access port in the wrong VLAN**: the host will be isolated from its intended subnet.
- **Trunk not configured**: the router will not receive tagged frames correctly.
- **Allowed VLAN list too restrictive**: the VLAN may be blocked on the trunk.
- **Incorrect default gateway on hosts**: hosts cannot reach other VLANs even if routing works.
- **Native VLAN mismatch**: can cause connectivity issues and warnings.

> If the router subinterface shows `up/down` or `down/down`, check the switch trunk first. If the subinterface is `up/up` but hosts cannot communicate, check VLAN membership and host IP settings.

**Troubleshooting workflow**

Use a layered approach so you isolate the fault quickly.

1. Confirm the host IP address, subnet mask, and default gateway.
2. Confirm the access port is in the correct VLAN.
3. Confirm the router-facing switch port is trunking.
4. Confirm the router subinterface has the correct `encapsulation dot1Q` value.
5. Confirm the subinterface IP address is in the correct subnet.
6. Test ping to the gateway, then to another VLAN.
7. Check for ACLs or security features only after basic Layer 2 and Layer 3 checks pass.

**Practical lab checklist**

Before you declare the lab complete, verify each item:

- VLANs exist on the switch.
- End-device ports are assigned to the correct VLANs.
- The router-facing port is a trunk.
- The trunk allows the required VLANs.
- The router physical interface is enabled.
- Each subinterface has the correct VLAN tag.
- Each subinterface has the correct IP address.
- Hosts use the router IP as their default gateway.
- Inter-VLAN pings succeed in both directions.

**Exam and lab tips**

Router-on-a-stick is a common CCNA lab pattern because it tests multiple foundational skills at once. You should be able to configure it from memory and verify it with a small set of commands.

- Memorize the relationship: **VLAN = subinterface = gateway IP**.
- Remember that the physical router interface usually has no IP address.
- Use `encapsulation dot1Q <vlan-id>` on every subinterface.
- Match the switch trunk VLANs to the router subinterfaces exactly.
- Verify from both the switch and router side before changing multiple settings at once.

**Quick reference configuration**

```cisco
! Switch
vlan 10
vlan 20

interface fa0/1
 switchport mode access
 switchport access vlan 10

interface fa0/2
 switchport mode access
 switchport access vlan 20

interface g0/1
 switchport mode trunk
 switchport trunk allowed vlan 10,20

! Router
interface g0/0
 no shutdown

interface g0/0.10
 encapsulation dot1Q 10
 ip address 192.168.10.1 255.255.255.0

interface g0/0.20
 encapsulation dot1Q 20
 ip address 192.168.20.1 255.255.255.0
```

**When router-on-a-stick is appropriate**

Use this design when you need inter-VLAN routing in a small lab, a branch environment, or a training scenario where simplicity matters more than scalability. In larger networks, multilayer switches usually provide better performance and cleaner design, but router-on-a-stick remains a valuable concept for understanding how VLANs and routing interact.
