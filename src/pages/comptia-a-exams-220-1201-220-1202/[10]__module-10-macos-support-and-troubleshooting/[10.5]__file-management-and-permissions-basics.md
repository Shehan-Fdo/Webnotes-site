## **Course:** CompTIA A+ (Exams 220-1201 & 220-1202)

**File management and permissions overview**

File management on Linux and macOS centers on how files are organized, identified, and protected. For A+ purposes, focus on the practical ideas: file types tell you how a file is used, and permissions control who can read, write, or execute it.

Both operating systems use a simpler permission model than Windows NTFS (New Technology File System), but the core idea is the same: access is limited based on identity and role. On macOS and Linux, you should think in terms of ownership and group membership first.

- **Owner**: the user who owns the file, usually the creator by default.
- **Group**: users who share a common set of permissions.
- **Everyone**: all other users on the system.

**Common file types in macOS**

macOS uses file extensions to indicate file purpose, just like other operating systems. The extension helps you identify how the file should open or what process should handle it.

- `.dmg` is a **disk image** used to distribute software.
- Double-clicking a `.dmg` mounts it in Finder so you can view its contents.
- `.pkg` is an **installer package** used to launch a setup workflow.
- `.zip` is a compressed archive format commonly recognized across platforms.

A file extension does not guarantee the file’s contents, but it is the first clue you use when deciding how to open or install something.

**Permissions on Linux and macOS**

Permissions define what a user can do with a file or folder. The evidence emphasizes basic access control rather than deep platform-specific tuning, so keep the model simple and practical.

- Permissions can allow or deny **read**, **write**, and **execute** access.
- Linux and macOS both use the owner/group/everyone model.
- macOS permissions are part of a broader security model that can also include higher-level controls.
- Access can apply to files, directories, subdirectories, and applications.

> Permissions are not the same as privacy controls or application-specific security prompts. Keep the basic file permission model separate from broader system protections.

**How to reason about access**

When you evaluate a file permission problem, start with identity and then check the allowed action. This helps you decide whether the issue is with the owner, the group, or all other users.

1. Identify who owns the file.
2. Check whether the user belongs to the relevant group.
3. Determine whether the needed action is read, write, or execute.
4. Confirm whether the file type or application expects a specific access level.

This approach is useful when troubleshooting why a file opens, runs, or refuses access.

**macOS security context**

macOS permissions exist within a larger security framework. Basic file permissions are only one layer, and some access restrictions can come from other system controls.

- Permissions can be enforced at multiple levels of the operating system.
- Some restrictions apply beyond simple file ownership.
- System-level controls may limit access even when a user has elevated privileges.

For exam prep, remember the distinction between basic permissions and broader security enforcement. If a file seems inaccessible, the cause may not be the file’s simple owner/group settings alone.

**Quick comparison**

| Concept | What it tells you | Practical use |
|---|---|---|
| Owner | Who controls the file | Check the primary user relationship |
| Group | Which shared users have access | Manage team or role-based access |
| Everyone | All other users | Understand the broadest access scope |
| `.dmg` | Disk image | Distribute and mount software |
| `.pkg` | Installer package | Run an installation workflow |

**Key takeaways**

- Use the owner/group/everyone model to reason about Linux and macOS permissions.
- Remember that permissions control read, write, and execute access.
- Recognize `.dmg` and `.pkg` as common macOS file types.
- Treat file permissions as one layer of access control, not the entire security story.
