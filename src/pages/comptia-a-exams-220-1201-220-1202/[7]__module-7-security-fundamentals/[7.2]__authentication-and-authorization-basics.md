## **Course:** CompTIA A+ (Exams 220-1201 & 220-1202)

**Authentication vs. authorization**

Authentication is the step where you prove who you are. Authorization is the step where the system decides what you can access after your identity is verified. In identity and access management (IAM), these are separate checks, and both matter for secure access control.

- **Authentication** verifies a claimed identity with credentials such as a username and password.
- **Authorization** grants or denies access to resources after authentication succeeds.
- IAM aims to ensure the right people, machines, and software components access the right resources at the right time.
- Access decisions often depend on roles, tasks, or policy-based permissions.

**Centralized authentication**

Many networks do not store every user credential on each device. Instead, a device sends login credentials to a centralized authentication server, which checks them against a shared database and returns approval or denial. This model is common when you log in to devices such as VPN concentrators, firewalls, or access points.

- The local device receives the login request.
- The device forwards the credentials to the authentication server.
- The server validates the credentials against its database.
- The server returns an approval or denial message.
- If approved, the user can continue to access the requested device or service.

> Centralized authentication keeps credential checking in one place instead of duplicating user databases across many devices.

**Common authentication protocols**

Enterprise environments use specialized protocols to support centralized authentication. The evidence highlights several important examples.

- **RADIUS** stands for Remote Authentication Dial-In User Service.
- **TACACS** and **TACACS+** are authentication-related protocols used in enterprise networks.
- **Kerberos** is another centralized authentication protocol used in network environments.
- These protocols help a client or network device verify credentials through an authentication server.

**Multi-factor authentication**

Multi-factor authentication (MFA) adds more than one proof of identity. The evidence mentions passwords, biometrics, and other authentication features as examples of factors used in the authentication process.

- A password is one common factor.
- Biometrics can provide an additional factor.
- MFA strengthens login security by requiring more than a single credential.

**Authorization and access control**

Once authentication succeeds, the system determines what the user can do. Authorization is about permissions, not identity proof. In practice, this means a user may be allowed to access one resource but denied another.

- Access can be limited to specific resources.
- Permissions can be defined by policy, role, or access level.
- Least privilege helps reduce unnecessary access.
- Authorization decisions should match the user’s job or task requirements.

**IAM lifecycle view**

IAM is often described as a lifecycle that includes identification, authentication, authorization, and accounting or auditing. For exam prep, focus on the order of the first three steps and what each one does.

1. Identify the user, machine, or software component.
2. Authenticate the claimed identity.
3. Authorize access to specific resources.
4. Log or audit activity when the system supports accounting.

**Practical exam reminders**

When you see a scenario question, separate the problem into identity proof and access permission. If the question asks how a system confirms a login, think authentication. If it asks what a user can reach after login, think authorization.

- Login credentials are used for authentication.
- Resource access rules are used for authorization.
- Centralized servers often handle credential validation.
- MFA adds another layer beyond a password.
