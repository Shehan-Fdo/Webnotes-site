## **Course:** CompTIA A+ (Exams 220-1201 & 220-1202)

**Access and authorization basics**

Account and access management support focuses on whether a user has the right permissions for a task or resource. In practice, access problems often show up as messages such as `Access is denied` or `User is not authorized` when connecting to a remote system through Remote Desktop Protocol (RDP).

These issues usually point to a permissions problem rather than a network failure. When you troubleshoot, start by checking whether the account is allowed to use the target service, resource, or remote session.

- **Authorization** determines whether the account can perform the requested action.
- **Permissions** control access to files, services, applications, and remote connections.
- **User context** matters because the same device can behave differently depending on which account is signed in.

**Common access-denied symptoms**

Microsoft documentation for Remote Desktop Services notes that access-denied errors commonly appear when a user tries to connect to a Windows computer or virtual machine by using RDP. The message may be direct, or it may indicate that the user is not authorized.

A practical troubleshooting approach is to treat the message as a clue about account rights first, then verify the target system and the user’s access path.

- `Access is denied` usually means the user lacks the necessary permission.
- `User is not authorized` usually means the account is not permitted to connect in the current context.
- The issue may involve local rights, remote access rights, or group-based restrictions.

> If the user cannot connect, do not assume the account is broken. Check authorization and permissions before changing the device or network.

**Checking the current security context**

A technician may need a snapshot of the current user’s security context to understand what the account can do. That snapshot can include privileges, group memberships, logon ID, and security identifiers.

The `whoami` command is useful for this kind of verification. In exam-style troubleshooting, it helps you confirm the identity and membership details of the account currently in use.

```powershell
whoami /all
```

- `whoami /all` shows a broad view of the current security context.
- Use it when you need to confirm group membership or privilege-related access.
- Compare the output with the access requirement for the resource or service.

**Troubleshooting workflow**

1. Confirm the exact error message.
2. Identify whether the problem is authorization-related or something else.
3. Check the user’s permissions, group membership, and access rights.
4. Verify the account context with `whoami /all` if needed.
5. Re-test the connection or access attempt after confirming the required rights.

This workflow keeps the focus on the most likely cause: the account does not have the required access. It also helps you avoid unnecessary changes when the issue is simply a missing permission or an unauthorized user context.

**Exam focus**

For A+ exam prep, expect questions that ask you to interpret access-denied messages, recognize authorization failures, or choose a command that reveals the current user’s security context. The key idea is to match the account’s rights to the requested action.

- Know the difference between a permission problem and a connectivity problem.
- Recognize that RDP access errors often point to authorization.
- Remember that `whoami /all` provides a comprehensive security snapshot.
