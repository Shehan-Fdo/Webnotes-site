## **Course:** CompTIA A+ (Exams 220-1201 & 220-1202)

**What application crashes mean**

An application crash happens when a program stops working unexpectedly. It can affect productivity apps, web browsers, and system tools, so you need to identify whether the issue is isolated to one app or tied to a broader Windows problem.

Common crash symptoms include:

- The app closes without warning.
- The app stops responding before it exits.
- The app behaves incorrectly after startup or during a specific task.
- Multiple apps fail when a shared resource or conflict is involved.

**Common causes**

Application crashes often come from a small set of practical issues. Start by checking the most likely causes before moving to deeper troubleshooting.

- **Insufficient resources** can cause an app to stop responding or close unexpectedly.
  - Low RAM, CPU capacity, or disk space can trigger instability.
- **Corrupted files** can break normal program behavior.
  - Incomplete installs, failed updates, or malware can damage program or system files.
- **Software conflicts** can make apps interfere with each other.
  - Background services or multiple security tools can create instability.
- **Outdated software** can leave known bugs unresolved.
  - Older versions may not work well with current system changes.

**How to troubleshoot crashes**

Use a practical, step-by-step approach. Start with the app itself, then expand to system-level checks if the problem continues.

1. Confirm the crash pattern.
   - Note which app fails, when it fails, and whether the issue is repeatable.
2. Check resource usage.
   - Use **Task Manager** to review CPU, memory, disk, and network activity in real time.
   - Sort by usage to see whether one process is consuming unusual resources.
3. Review the app and system state.
   - Look for recent installs, updates, or configuration changes that may have introduced the problem.
4. Try a software recovery action.
   - Update the application if a newer version is available.
   - Reinstall the app if files appear damaged.
   - Adjust configuration settings if the issue is tied to a specific option.
5. If the crash affects startup or broader Windows behavior, use recovery options.
   - Try **Safe Mode** to reach the desktop with minimal drivers and services.
   - Use **System Restore** to roll back to a previous configuration.
   - Use the **last known good** login or configuration when software changes appear to be the cause.
   - Roll back a recently installed driver if the crash started after the driver change.

> If the crash appears alongside a Windows stop error, also known as a **Blue Screen of Death (BSOD)**, treat it as a broader system issue and check for hardware, driver, or bad application causes.

**Useful Windows tools**

Windows provides built-in tools that help you narrow down the cause of instability. Use them to identify whether the problem is the app, the system, or a resource bottleneck.

- **Task Manager**
  - Shows live CPU, memory, disk, and network usage.
  - Helps you identify the process that is consuming the most resources.
- **Performance tab**
  - Gives a graphical view of resource behavior over time.
  - Helps you spot trends instead of only momentary spikes.
- **Safe Mode**
  - Starts Windows with minimal components.
  - Helps you test whether a driver or service is contributing to the crash.
- **System Restore**
  - Returns the system to an earlier working state.
  - Helps when a recent change introduced instability.

**Practical troubleshooting mindset**

Application troubleshooting is about matching the fix to the scope of the problem. If one app crashes, focus on that program first. If several apps fail or the issue appears during startup, widen the investigation to Windows, drivers, and recent changes.

A good exam answer usually follows this logic:

- Check the app and its version first.
- Verify resource availability.
- Look for corruption or conflicts.
- Use Windows recovery tools when the problem extends beyond one program.
