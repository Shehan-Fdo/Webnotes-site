## **Course:** CompTIA A+ (Exams 220-1201 & 220-1202)

**Network configuration basics**

Network configuration defines the settings that let devices communicate on a network. For A+ purposes, focus on the core values a workstation needs to reach local resources and the internet: an IP address, subnet mask, default gateway, and DNS server settings.

- An **IP address** uniquely identifies a device on the subnet.
- A **subnet mask** helps the device determine what is local versus remote.
- A **default gateway** is the route used to reach networks outside the local subnet.
- **DNS** settings let the device resolve names into IP addresses.

When these values are correct, the workstation can communicate on the local network and reach external services without manual reconfiguration for every device.

**DHCP and automatic addressing**

<abbr title="Dynamic Host Configuration Protocol">DHCP</abbr> automatically assigns network settings to devices. It reduces the need to manually enter configuration values on every workstation, laptop, or mobile device.

A DHCP server typically needs these settings:

- An IP address range for assignments
- The associated subnet mask
- A lease duration that controls how long a device keeps the address
- DNS server settings
- A default gateway setting
- Optional additional configuration values, such as settings for voice over IP services

DHCP uses scopes and pools to manage the addresses it can hand out. A scope defines the range available for assignment, and the pool is the set of addresses the server can distribute from that range. DHCP can also support reservations, which keep a specific address tied to a specific device.

> If DHCP is unavailable, devices may fail to receive the addressing information they need to communicate normally on the network.

**DNS and name resolution**

<abbr title="Domain Name System">DNS</abbr> translates human-readable names into IP addresses. This matters because users usually connect to names, while network devices route traffic using numeric addresses.

DHCP commonly supplies DNS server information to clients so they know where to send name-resolution requests. Without a working DNS configuration, a device may still have network connectivity but fail when trying to reach services by name.

- Use DNS when you need name-to-address resolution.
- Verify the configured DNS server if a site works by IP address but not by hostname.
- Keep DNS settings consistent across devices when possible.

**NAT and internet access**

<abbr title="Network Address Translation">NAT</abbr> helps devices on a private network communicate with external networks by translating addresses as traffic passes through a router or similar device. In practical terms, NAT supports internet access for internal devices that use private addressing.

NAT is often part of the path between a local workstation and the wider internet. The workstation sends traffic to the default gateway, and the gateway device handles the translation needed for external communication.

**Default gateway and traffic flow**

The default gateway is the next hop a device uses when the destination is not on the local subnet. If the destination is local, the device communicates directly. If the destination is remote, the device sends traffic to the gateway.

This makes the default gateway a critical setting for internet access and for reaching other remote networks.

- A missing or incorrect gateway can block traffic beyond the local subnet.
- A correct gateway does not replace DNS; both settings serve different purposes.
- DHCP often provides the gateway automatically, which reduces manual setup errors.

**Manual versus automatic configuration**

You can configure network settings manually or let DHCP assign them automatically. Manual configuration can work well for a small number of devices with static needs, but it becomes difficult to maintain at scale.

1. Enter the IP address, subnet mask, default gateway, and DNS server values manually if the device requires static settings.
2. Use DHCP if you want the device to receive those values automatically.
3. Verify that the assigned values match the subnet and the intended network path.

**Troubleshooting focus**

When a device cannot reach the network or the internet, check the addressing chain in order:

- Confirm the device has a valid IP address.
- Confirm the subnet mask matches the network.
- Confirm the default gateway is present and correct.
- Confirm DNS is configured if name resolution fails.
- Confirm DHCP is functioning if the device should receive settings automatically.

A problem in any one of these areas can look like a general connectivity issue, so isolate the failure before changing multiple settings at once.
