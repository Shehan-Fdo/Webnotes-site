## **Course:** CompTIA A+ (Exams 220-1201 & 220-1202)

**Why these tools matter**

Network troubleshooting starts with simple indicators and then moves to targeted commands and tools. You use them to confirm whether the problem is physical, local, or somewhere beyond the device.

- A **link light** gives you a quick wired-connectivity check.
- No link light can point to a bad cable, a loose connection, or a device-side issue.
- A loopback test with `ping 127.0.0.1` checks the local TCP/IP stack before you look at the network path.
- Intermittent wireless issues, high latency, high jitter, poor VoIP quality, and port flapping are all symptoms that help narrow the fault domain.

**Physical tools for cable work**

If you build or repair cables, you need tools that support termination and validation. These tools help you confirm that the cable itself is usable before you spend time on higher-level troubleshooting.

- A **cable crimper** fastens a modular connector onto the end of a cable.
- Crimpers may support different connector types, such as six-position connectors for RJ11 and eight-position connectors for RJ45.
- A **cable tester** checks whether the cable is wired correctly and can help identify faults in the run.
- A **tone generator** helps you trace a cable through walls, ceilings, or a network closet.

> Use physical tools first when the symptom suggests a bad cable or bad termination. Software commands cannot fix a broken copper run.

**Common command-line utilities**

Command-line tools help you verify addressing, name resolution, reachability, and path behavior. They are especially useful when the network is up in some places but not others.

- `ipconfig` shows interface configuration on Windows, including the IP address, subnet mask, and default gateway.
- `ifconfig` provides similar interface information on Linux.
- `ping` tests basic connectivity by sending packets to a target.
- `tracert` or `traceroute` helps you see the path traffic takes toward a destination.
- `pathping` combines path tracing with packet-loss information.
- `nslookup` checks DNS (Domain Name System) name resolution.
- `netstat` shows network connections and related status.
- `nbtstat` is used for NetBIOS-related troubleshooting on Windows.

**What the indicators tell you**

Each tool gives you a different clue, so you should match the result to the symptom instead of treating every failure the same way.

- A valid IP address, subnet mask, and default gateway from `ipconfig` suggest the interface is configured.
- A failed `ping` to a remote host may indicate a routing, firewall, or destination issue.
- A successful `ping` to `127.0.0.1` suggests the local stack is responding.
- `nslookup` helps you separate a DNS problem from a general connectivity problem.
- `netstat` helps you inspect active connections when you suspect a session or port issue.
- `tracert` and `pathping` help you narrow where traffic stops or slows down.

**Practical troubleshooting flow**

1. Check the physical link indicator on the wired connection.
2. Verify the cable and termination if the link light is off.
3. Test the local stack with `ping 127.0.0.1`.
4. Review interface settings with `ipconfig` or `ifconfig`.
5. Test name resolution with `nslookup` if a hostname fails.
6. Trace the route with `tracert` or `pathping` if the destination is still unreachable.
7. Use `netstat` or `nbtstat` when the issue appears to involve sessions, ports, or Windows name-related behavior.

**Troubleshooting mindset**

A structured approach helps you avoid guessing. Start with the simplest indicator, confirm the local device, and then move outward toward the network path and remote service.

- Treat symptoms like high latency and poor VoIP quality as clues, not final answers.
- Use the smallest test that can confirm or eliminate a layer.
- Prefer repeatable checks over assumptions when the issue is intermittent.
