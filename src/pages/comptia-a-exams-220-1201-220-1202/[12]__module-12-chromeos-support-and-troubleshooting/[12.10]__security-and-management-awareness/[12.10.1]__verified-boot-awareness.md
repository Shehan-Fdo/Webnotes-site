## **Course:** CompTIA A+ (Exams 220-1201 & 220-1202)

**Verified boot overview**

Verified boot is a boot integrity approach used on Chrome OS devices to help ensure that the code executed during startup is trusted. The boot process begins with a **read-only** portion of firmware, which verifies the next stage before allowing it to run.

This model is designed to reduce the risk of corruption or tampering during startup. It focuses on making sure executed code comes from the expected source tree rather than from an attacker or damaged storage.

- The first stage firmware is read-only.
- The next stage is verified before execution.
- The goal is to detect boot path changes and support recovery when needed.
- Verification happens on-the-fly to avoid slowing startup.

**How the boot chain is protected**

Chrome OS uses a first-stage read-only firmware and a second-stage updatable firmware. The updatable firmware is signed and includes kernel keys and a `dm-verify` hash, which helps protect the firmware, Linux kernel, and root filesystem from corruption and attack.

This layered design means each stage depends on the integrity of the previous stage. In practical terms, verified boot is not just about loading an operating system; it is about validating the chain of trust before the system fully starts.

- Signed firmware helps confirm authenticity.
- Kernel keys support later verification steps.
- `dm-verify` protects the root filesystem integrity.
- Stored cryptographic hashes support verification during boot.

**Security purpose and limits**

Verified boot is aimed at stopping opportunistic attackers and detecting when the boot path has changed. It is not described as a guarantee against every possible attack. Instead, it acts as a significant deterrent and a recovery aid.

> Verified boot improves trust in startup integrity, but it does not replace broader system security controls.

A useful exam takeaway is that verified boot emphasizes integrity checking during startup, while recovery mode supports restoring a device when boot verification fails or the boot path changes.

- It helps detect non-volatile boot changes.
- It supports user awareness that recovery may be needed.
- It is focused on practical deterrence, not perfect prevention.

**Related boot modes**

ChromeOS devices use distinct boot modes that control what firmware and software can load. Normal mode is the default verified boot mode and is intended for secure, Google-signed ChromeOS operation.

Recovery mode is used when the device needs restoration. Understanding the difference helps you connect verified boot with real-world troubleshooting and device recovery.

| Mode | Purpose | Key idea |
|---|---|---|
| Normal / Verified Boot Mode | Secure default startup | Full firmware and OS verification is enabled |
| Recovery Mode | Restore a device | Used when boot or system recovery is needed |

**Exam focus points**

For A+ exam prep, focus on the relationship between boot integrity, signed firmware, and recovery behavior. You do not need to treat verified boot as a generic boot method; it is specifically about verifying trusted startup components.

- **Read-only firmware** starts the trust chain.
- **Signed updates** help protect the updatable firmware stage.
- **Cryptographic hashes** support integrity checks.
- **Recovery mode** is the fallback when verification or boot path changes indicate a problem.
