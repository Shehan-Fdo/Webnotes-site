## **Course:** CompTIA A+ (Exams 220-1201 & 220-1202)

**Network configuration overview**

A+ Core 1 and 2 expects you to understand networking basics and configure wired and wireless small office/home office (SOHO) networks. In practice, that means you identify how a device is connected, verify its interface settings, and confirm whether the system can communicate on the network.

The first step in troubleshooting is usually to establish a baseline. Check the current configuration before changing anything so you can compare what is working against what is not.

- **IP address** information tells you how the device is identified on the network.
- **Subnet mask** helps define the network portion of the address.
- **Interface settings** show the state and details of the network adapter.

**Checking Linux network configuration**

On Linux, the `ip` command is a common way to display and manage address settings. The `ip addr` subcommand shows the current IP address and subnet mask for interfaces on the system.

```bash
ip addr
```

You use this output to confirm whether the adapter has an address, whether the expected interface is active, and whether the system is using the configuration you expect. This is especially useful when you are validating connectivity or comparing a working system to a problem system.

- `ip` is available by default on many Linux distributions.
- `ip addr` displays address information for all interfaces.
- You can use related `ip` subcommands to display or change address settings.

**Basic troubleshooting workflow**

When a network problem appears, start with simple checks that narrow the fault domain. A wired connection should show a link light if the physical connection is good. If the light is off, the issue may be the cable or the connection itself.

A loopback test helps you verify the local TCP/IP stack before you investigate the wider network. Pinging `127.0.0.1` checks whether the device can communicate with itself.

1. Check the link light on the wired connection.
2. If the light is off, inspect the cable and physical connection.
3. Ping `127.0.0.1` to test the local stack.
4. Review interface and address information with `ip addr`.
5. Compare the current configuration with the expected settings.

> If the device cannot reach `127.0.0.1`, the problem is local to the host rather than the network path.

**What to verify first**

A practical network check focuses on the most visible and least disruptive indicators first. That approach helps you avoid unnecessary changes and makes it easier to isolate the failure.

- **Connectivity**: confirm whether the device has any network path at all.
- **Interface status**: verify that the adapter is present and active.
- **Addressing**: confirm the IP address and subnet mask.
- **Physical layer**: inspect the cable and link light on wired systems.

**Linux and file-sharing context**

Network configuration often supports other services, such as Samba file sharing between Linux and Windows systems. In that kind of setup, you still begin with the same basics: confirm the interface details, then validate that the service can be reached from another machine.

- Use command-line tools to inspect the network before testing services.
- Confirm the interface configuration before troubleshooting access to a share.
- Treat service access problems separately from basic address or link issues.

**Exam focus**

For A+ study, focus on the relationship between configuration and troubleshooting rather than memorizing isolated commands. You should be able to explain what the system’s address information means, how to check it, and how to use simple tests to narrow a connectivity problem.

- Know what `ip addr` shows.
- Know why a link light matters.
- Know why `127.0.0.1` is useful for local testing.
- Know that networking basics include wired and wireless SOHO configuration.
