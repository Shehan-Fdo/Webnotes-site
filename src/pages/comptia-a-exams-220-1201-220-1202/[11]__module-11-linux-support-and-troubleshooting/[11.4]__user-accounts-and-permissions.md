## **Course:** CompTIA A+ (Exams 220-1201 & 220-1202)

**Permission model overview**

Linux permissions use three access levels and three identities. The access levels are **read** (`r`), **write** (`w`), and **execute** (`x`). The identities are **user/owner** (`u`), **group** (`g`), and **all others** (`o`).

This model is simpler than very granular permission systems, but it still gives you practical control over who can view, change, or run a file. The same permission bits can behave differently on files and directories, so always check the object type before you interpret the result.

- `r` lets you view file contents or list directory contents.
- `w` lets you modify a file or change entries in a directory.
- `x` lets you run a file or enter a directory with `cd`.

**Ownership and access control**

Permissions are tied to ownership and group membership. When you log in, you act as a user account and usually belong to one or more groups. That relationship determines which permission set applies first.

A practical way to think about access is:

- **Owner** controls the file by default.
- **Group** shares access with members of the same group.
- **Others** covers everyone else on the system.

This is the core model you need for exam questions about who can access a resource and why a command changes that access.

**Changing permissions with `chmod`**

Use `chmod` to modify standard Linux permissions. The command works with either absolute or symbolic notation, and the general pattern is:

```bash
chmod {access-level} {resource}
```

You may need `sudo` depending on system configuration and the file you are changing. A numeric example from the evidence is:

```bash
chmod 764 /opt/projects
```

In that example, `764` is the access level and `/opt/projects` is the resource. For exam prep, focus on recognizing that `chmod` changes permissions, while the numeric or symbolic form expresses *which* permissions change.

**Files versus directories**

The same permission letter can mean slightly different things depending on whether you apply it to a file or a directory. That distinction matters most for `execute`.

- On a **file**, `x` means the system can run it.
- On a **directory**, `x` means you can enter it with `cd`.

This is a common test point because directory access is not the same as file execution. If a directory lacks `x`, you may not be able to traverse into it even if you can list it.

**Quick exam reminders**

- Know the three identities: `u`, `g`, and `o`.
- Know the three permissions: `r`, `w`, and `x`.
- Remember that `chmod` changes standard permissions.
- Remember that `sudo` may be required.
- Treat directory `x` as traversal access, not program execution.

**Practical interpretation**

When you see a permission string or numeric mode, translate it into access by identity. Start with the owner, then the group, then others. If a question asks why a user can or cannot do something, check both the permission bit and whether the target is a file or a directory.
