## **Course:** CompTIA A+ (Exams 220-1201 & 220-1202)

**Secure Boot overview**

Secure Boot is a security feature in <abbr title="Unified Extensible Firmware Interface">UEFI</abbr>-based systems. It helps stop unauthorized or malicious software from loading during startup by allowing only trusted, signed components to run.

The main idea is simple: the firmware checks the digital signatures of boot-related files before handing control to the operating system. This reduces the chance that low-level malware can take hold before the OS starts.

- It protects the boot process, not just the running operating system.
- It checks the bootloader and other startup components for valid signatures.
- It helps defend against rootkits and bootkits that try to load early.

**Why it matters**

Boot-time attacks are difficult to detect because they can run before normal security tools fully load. Secure Boot adds a trust check at the earliest practical stage of startup, which makes it harder for malicious code to persist.

This feature is especially relevant when you are dealing with systems that must maintain a trusted startup chain. It is also a common exam topic because it connects firmware security with operating system integrity.

- **Rootkits** can hide deep in the system and interfere with OS behavior.
- **Bootkits** target the boot sequence itself.
- Signed boot files help establish trust before the OS takes over.

**How Secure Boot works**

Secure Boot depends on a trusted signing authority and signed boot files. In practice, the firmware compares signatures against trusted keys stored in the UEFI environment.

> Secure Boot can limit compatibility if an operating system or boot component is not properly signed or configured.

- The system starts in firmware.
- The firmware validates boot components.
- Trusted components continue the startup chain.
- Untrusted components are blocked from executing.

**Compatibility considerations**

Secure Boot improves security, but it can create compatibility issues for some operating systems, including certain Linux distributions, if they are not configured to work with it. When troubleshooting boot problems, you should consider whether Secure Boot is preventing a legitimate bootloader from loading.

- Check whether the OS supports Secure Boot.
- Verify that the bootloader is signed.
- Review firmware settings if a system fails to boot after a change.

**Related BIOS and UEFI awareness**

Secure Boot is part of the broader firmware security area you should recognize for A+ exam purposes. Other firmware controls can also affect startup behavior and access to configuration settings.

- BIOS and UEFI settings are accessed during startup.
- Password controls can restrict boot access or BIOS changes.
- Boot options may affect whether a system starts from internal storage or another device.

**Exam focus**

For exam questions, look for clues about preventing unauthorized startup code, protecting against low-level malware, or requiring signed boot components. If the scenario mentions firmware-level startup protection, Secure Boot is usually the best match.

- Choose Secure Boot when the goal is to block unsigned boot code.
- Think of UEFI when the question mentions modern firmware.
- Associate the feature with startup integrity and trusted execution.
