## **Course:** CompTIA A+ (Exams 220-1201 & 220-1202)

**Common connectivity symptoms**

Wireless and wired problems often show up as limited connectivity, local connectivity, or intermittent connectivity. In exam scenarios, start by deciding whether the issue affects a single device or multiple nearby devices. That distinction helps you separate a local client problem from a broader network issue.

A practical first check is to try a known working address on the local network. This tells you whether the failure is in the physical connection or in the software and protocol stack. For wired systems, also inspect the link indicators to confirm the port is active.

- **Limited connectivity** often means the device reaches only part of the network.
- **Local connectivity** can still allow access to nearby resources such as shares, printers, and email.
- **Intermittent connectivity** points to an unstable link, weak signal, or cabling issue.
- **Scope** matters: one PC suggests a client-side fault; many PCs suggest a shared infrastructure problem.

**Wired troubleshooting checks**

Wired links depend on signal quality across copper or fiber. As distance increases, the signal weakens through attenuation, which is signal loss over the medium. If attenuation is too high, the receiving side may not detect the signal reliably.

Use a methodical approach when you troubleshoot a wired failure. Check the physical path first, then move to addressing and protocol validation if the link appears up.

- Inspect cable condition and seating at both ends.
- Verify link lights on the network interface and switch port.
- Compare the problem device with neighboring systems on the same segment.
- Confirm whether the issue is isolated to one host or shared across multiple hosts.

> If multiple devices fail at once, focus on the common path before you spend time on a single workstation.

**Wireless troubleshooting checks**

Wireless links also experience attenuation, but the cause is usually distance from the access point or environmental interference rather than cable damage. A weak or unstable wireless connection can look similar to a wired issue at the user level, so you still begin with scope and basic connectivity checks.

When you troubleshoot wireless failures, verify whether the device can reach a known good local resource and whether the problem changes as the device moves closer to the access point. That helps you separate signal-related issues from configuration problems.

- Check whether the device is within usable range of the access point.
- Look for symptoms that change with location or movement.
- Compare the behavior with other wireless clients nearby.
- Treat repeated drops as a sign of instability, not just a one-time failure.

**Useful command-line verification**

Command-line tools help you confirm whether the host has a valid network configuration and whether name resolution or pathing is part of the problem. On Windows, `ipconfig /all` shows IP address details, subnet mask, DNS server address, and default gateway. On Linux, `ifconfig` displays interface status and IP configuration.

Other tools help you narrow the fault after you confirm basic addressing.

```bash
ipconfig /all
ifconfig
nslookup example.com
netstat
tracert example.com
```

- `ipconfig /all` helps you verify adapter configuration on Windows.
- `ifconfig` helps you verify interface configuration on Linux.
- `nslookup` helps you test name resolution.
- `netstat` helps you view established TCP/IP sessions.
- `tracert` helps you see the Layer 3 path to a target.

**Troubleshooting flow**

Use a simple sequence so you do not skip the basics. Start with the local host, then expand outward only if the first checks do not explain the failure.

1. Test a known working local address.
2. Check whether the issue affects one device or multiple devices.
3. Inspect wired link lights or wireless signal behavior.
4. Verify host addressing with the appropriate command.
5. Use path and name-resolution tools if the basic link appears healthy.

This flow keeps you focused on the most likely cause first. It also helps you avoid confusing a physical connectivity problem with a DNS, IP addressing, or application-layer issue.

**Exam focus**

For A+ troubleshooting questions, pay attention to the wording. The exam often describes symptoms rather than naming the fault directly. Your job is to map the symptom to the most likely layer or component.

- **Local resources** can still be reachable even when broader access fails.
- **Attenuation** affects both wired and wireless links, but the cause differs by medium.
- **Scope of impact** is a major clue in deciding whether to troubleshoot one device or the shared network.
- **Indicator lights** are a fast check for wired link status.
