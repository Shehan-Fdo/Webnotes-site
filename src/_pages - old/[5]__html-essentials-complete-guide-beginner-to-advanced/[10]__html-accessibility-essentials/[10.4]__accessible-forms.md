## **Course:** HTML Essentials: Complete Guide (Beginner to Advanced)

**Why accessible forms matter**

- Forms support common tasks like sign-up, purchase, and contact flows.
- Accessible forms help users who rely on screen readers or keyboard navigation.
- Clear form structure reduces barriers for users with disabilities.
- Accessible forms support inclusive experiences and align with WCAG.

**Core labeling rules**

- Give every interactive control a descriptive **label**.
- Use `<label>` with `for` and matching `id` for native form controls.
- Keep labels concise and specific.
- Use `aria-labelledby` when you need to reference existing text as the accessible name.
- Use `aria-describedby` for extra guidance or supporting details.
- Ensure each interactive element has an accessible name.

**Grouping related fields**

- Use `<fieldset>` to group related controls.
- Use `<legend>` to name the group.
- Grouping helps users understand form sections faster.
- Keep group labels short and meaningful.

**Validation and error messages**

- Show validation feedback in a way users can perceive.
- Use clear, unambiguous error text.
- Explain what went wrong and how to fix it.
- Provide in-line errors when users need immediate feedback.
- Provide post-submission errors when the form needs a summary of issues.
- Mark invalid fields with `aria-invalid` when appropriate.
- Use a WAI-ARIA live region for error announcements when needed.

**Practical accessibility behavior**

- Make sure keyboard users can reach and operate all controls.
- Keep error messages visible and readable.
- Avoid vague messages like “invalid input.”
- Write feedback for screen reader users as well as sighted users.
- Use concise language for users with cognitive load or short attention spans.

**Advanced ARIA support**

- Use ARIA only when native HTML does not cover the need.
- Prefer semantic HTML first.
- Add `aria-describedby` to connect controls with help text or error text.
- Add `aria-labelledby` when the label comes from another element.
- Use `aria-invalid` to signal a field with an error state.

**Common accessibility outcomes**

- Users can understand what each field expects.
- Users can identify errors without guessing.
- Users can recover from mistakes more easily.
- Users can complete forms with assistive technology.

**Checklist for accessible forms**

- Add a visible label to each control.
- Group related controls with `<fieldset>` and `<legend>`.
- Provide clear instructions where needed.
- Validate input and explain errors clearly.
- Announce important errors to assistive technologies.
- Test with keyboard navigation and a screen reader.
