## **Course:** HTML Essentials: Complete Guide (Beginner to Advanced)

**What ARIA is**

- **ARIA** stands for Accessible Rich Internet Applications.
- **WAI-ARIA** expands to Web Accessibility Initiative - Accessible Rich Internet Applications.
- It adds semantics to HTML so assistive technologies can better understand UI elements.
- It helps describe roles, states, and properties for interactive content.
- It is especially useful for web apps with dynamic JavaScript-updated content.

**When to use ARIA**

- Use ARIA when native HTML does not provide the semantics you need.
- Prefer semantic HTML first.
- Use ARIA as a last resort when you must enhance a custom control.
- Use it to improve accessibility for complex widgets and dynamic interfaces.
- Use it to help screen readers understand what changes on the page.

**Core ARIA concepts**

- **Roles** define what kind of UI element something is.
- **States** describe current conditions, such as whether something is active or expanded.
- **Properties** provide additional information about an element.
- ARIA semantics are added to the accessibility API.
- ARIA does not change the DOM itself.

**First rule of ARIA**

- If a native HTML element already provides the needed semantics and behavior, use it.
- Do not replace a semantic element with a `div` plus ARIA when a native element exists.
- Native elements already include built-in keyboard accessibility and expected behavior.
- If you use ARIA, you must mimic the equivalent browser behavior in script.

**Practical guidance**

- Use `<nav>` instead of `role="navigation"` on a `div`.
- Use semantic elements for headings, tables, and other document structure when possible.
- Use ARIA for custom widgets that need extra accessibility support.
- Use ARIA to expose live content updates when content changes dynamically.
- Use ARIA to add form hints and error messages when needed.

**Common use cases**

- Accessible JavaScript widgets.
- Dynamic content updates.
- Form hints.
- Error messages.
- Landmarks and signposting.
- Keyboard accessibility support.

**Key limitations**

- ARIA should not be your first choice.
- ARIA cannot fix poor HTML structure by itself.
- ARIA only exposes extra information to accessibility tools.
- ARIA is not a substitute for correct semantic markup.
- If you choose ARIA, you are responsible for matching expected interaction behavior.

**Useful mindset**

- Start with semantic HTML.
- Add ARIA only when native elements are not enough.
- Test with assistive technologies after adding ARIA.
- Keep custom controls predictable for keyboard users.
- Keep markup simple when a native element already solves the problem.
